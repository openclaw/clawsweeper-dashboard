## Summary

Keep open: the PR targets a real append-mode gap, but the current patch can still truncate files when a present `append` value is malformed and falls through to overwrite behavior.

## What This Changes

The PR adds an optional `append?: boolean` mode to the agent `write` tool for host/workspace and sandbox writes, adds focused append/abort/workspace tests, and updates the changelog.

## Best Possible Solution

Land a maintainer-approved append-capable file primitive only after malformed append intent is rejected before any overwrite-capable fallback, with the existing workspace and sandbox guards preserved.

## Reproduction Assessment

Yes, by source inspection. Current main routes ordinary host and sandbox writes through overwrite-style `createWriteTool`, and the PR branch has a concrete malformed-append path where `append: "true"` or `append: 1` falls through to overwrite behavior.

## Solution Assessment

No, not as-is. The optional flag is a plausible narrow implementation, but present non-boolean `append` values must be rejected before the base overwrite tool can run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject malformed append values before overwriting:** `src/agents/pi-tools.read.ts:882-884`
  - body: When a model sends an `append` key with a non-boolean value like `"true"` or `1`, `doAppend` is false and this delegates to the original overwrite implementation. That can truncate the shared log this feature is meant to protect, so reject any present non-boolean `append` value before calling the base write tool.
  - confidence: 0.9
