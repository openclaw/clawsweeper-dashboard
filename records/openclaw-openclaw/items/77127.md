## Summary

Keep this PR open. The append-mode direction addresses a real general-write gap, but the latest head still lets an append call complete successfully if cancellation arrives while the active append operation is in progress.

## What This Changes

This PR adds an optional `append` boolean to the agent `write` tool for host, workspace, and sandbox writes, with regression tests and a changelog entry.

## Best Possible Solution

Land append-capable writes only after the append path preserves the existing write tool safety envelope: path policy, validation, queue serialization, parent creation, and active cancellation behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main is overwrite-only for general `write`, and the PR-specific abort gap is visible where the append wrapper awaits `ops.appendFile` without observing later aborts.

## Solution Assessment

No. The optional append flag is a plausible narrow direction, but this implementation should match upstream `write` cancellation behavior before merge, preferably through an active abort listener or signal-aware append operation plus regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Reject append calls aborted during the active write:** `src/agents/pi-tools.read.ts:882`
  - body: The append path checks `signal` before entering the queue and again at queue start, but once `ops.appendFile` begins it only awaits completion. If the run is cancelled during the sandbox read/mkdir/write or host append, this can still return success, unlike upstream `createWriteTool`, which rejects on abort during the active queued mutation. Add active-operation abort handling or pass a signal-aware append operation, plus a regression that aborts after `appendFile` starts.
  - confidence: 0.8
