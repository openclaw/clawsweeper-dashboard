## Summary

Keep open: the PR addresses a valid append gap, but the latest head still has a blocking correctness issue where an in-flight append can report success after cancellation.

## What This Changes

The PR adds an optional `append: true` write-tool parameter across host/workspace and sandbox write paths, plus regression tests and a changelog entry.

## Best Possible Solution

Land append-capable writes only after the append path preserves existing validation, path boundary, queue serialization, parent creation, and active cancellation semantics.

## Reproduction Assessment

Yes. Source inspection shows current main is overwrite-only for general `write`, and the PR-specific abort gap can be reproduced by stubbing `appendFile` to block after it starts, aborting the signal, then observing that the wrapper can still return success.

## Solution Assessment

No. The optional append flag is a plausible narrow direction, but this implementation should match upstream `write` cancellation behavior before merge, preferably through active abort handling or a signal-aware append operation plus regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Reject append calls aborted during the active write:** `src/agents/pi-tools.read.ts:882`
  - body: The append path checks `signal` before entering the queue and again at queue start, but once `ops.appendFile` begins it only awaits completion. If cancellation arrives during the sandbox read/mkdir/write or host append, this can still return success, unlike upstream `createWriteTool`, which rejects on abort during the active queued mutation. Add active-operation abort handling plus a regression that aborts after `appendFile` starts.
  - confidence: 0.8
