## Summary

This PR should stay open: it addresses a real write-tool append gap documented by #40001 and #75321, but the latest branch still does not reject append calls aborted while the underlying append operation is active. Current main remains overwrite-only outside memory flush, so the work is useful but not merge-ready.

## What This Changes

This PR adds optional `append: true` handling to the agent `write` tool for host, workspace, and sandbox writes, with regression tests and a changelog entry.

## Best Possible Solution

Land append-capable writes only after the append path preserves the existing write-tool safety envelope, including active cancellation behavior, queue serialization, validation, and path guards.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main is overwrite-only for general `write`, and the PR-specific abort gap is visible where the append wrapper awaits `ops.appendFile` without observing later aborts.

## Solution Assessment

No. The optional append flag is a plausible narrow direction, but this implementation should match upstream `createWriteTool` cancellation behavior before merge, preferably with active abort handling or a signal-aware append operation plus regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Reject append calls aborted during the active write:** `src/agents/pi-tools.read.ts:882`
  - body: The append path checks `signal` before entering the queue and again at queue start, but once `ops.appendFile` begins it only awaits completion. If the run is cancelled during the sandbox read/mkdir/write or host append, this can still return success, unlike upstream `createWriteTool`, which rejects on abort during the active queued mutation. Add active-operation abort handling or a signal-aware append operation, plus a regression that aborts after `appendFile` starts.
  - confidence: 0.8
