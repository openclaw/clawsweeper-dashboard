## Summary

The PR is still necessary because current main has overwrite-only ordinary write behavior and the canonical data-loss issue is open, but the current head is not merge-ready: `append: true` still returns the overwrite-mode identical-content no-op before reaching the append branch, so duplicate append entries can be silently dropped. The branch also changes the model-facing write tool contract and carries broad unrelated formatting/import drift, which makes this a maintainer/API decision after the concrete bug is fixed.

## What This Changes

Adds optional `append?: boolean` support to the shared and wrapped agent `write` tools, exposes that contract in tool/event types and prompt metadata, and adds focused append regression coverage alongside unrelated formatting/import drift.

## Best Possible Solution

Land a narrow append-capable write contract only after the append no-op is fixed, duplicate-append regression coverage is added, unrelated drift is trimmed, and maintainers accept the optional write-tool API.

## Reproduction Assessment

Yes. Source inspection on the PR head gives a high-confidence path: create a file whose current content equals the planned appended payload, call `write` with `append: true`, and the identical-content no-op returns before append execution.

## Solution Assessment

No as currently proposed. A first-class append path is the right direction, but this branch is not the best merge shape until the no-op is gated to overwrite mode, duplicate append behavior is tested, unrelated drift is removed, and maintainers accept the API shape.
