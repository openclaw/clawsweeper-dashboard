## Summary

Keep this PR open: it addresses a real overwrite/data-loss gap and includes sufficient terminal proof, but the append path has a concrete correctness issue because it reimplements upstream path resolution without Unicode-space normalization. The public API shape still needs maintainer approval after the narrow resolver fix.

## What This Changes

The PR adds an optional `append` boolean to the agent `write` tool for host/workspace and sandbox writes, with regression tests and a changelog entry.

## Best Possible Solution

Land an append-capable file primitive that preserves existing write path semantics, workspace/sandbox guards, same-file queueing, validation, and maintainer-approved API shape.

## Reproduction Assessment

Yes. Current main is source-reproducible as overwrite-only outside the memory-flush wrapper, and the linked issue has production cron overwrite evidence on that same surface.

## Solution Assessment

No, not as-is. The optional append flag is a narrow backwards-compatible direction, but the implementation should reuse or fully mirror upstream path resolution before maintainer API approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Reuse upstream path normalization for append writes:** `src/agents/pi-tools.read.ts:974-976`
  - body: The append branch resolves `filePath` itself and only strips `@`/expands `~`, while upstream write's `resolveToCwd` also normalizes Unicode space characters before resolving. With `append: true`, a path copied with NBSP or narrow NBSP can create or append a different filename than normal write/read would target, so append should share that resolver behavior.
  - confidence: 0.82
