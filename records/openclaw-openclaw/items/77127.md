## Label Justifications

- `P1`: The PR targets a P1 data-loss workflow where agent and cron writes can silently overwrite shared memory files.
- `merge-risk: 🚨 compatibility`: The diff changes the public write-tool input contract and currently leaves the exported base write factory/schema inconsistent with that contract.

## Summary

Keep open: this remains the active fix path for the linked write-tool data-loss issue, but it is not merge-ready because the patch widens the public write-tool contract without aligning the public base write factory/schema and can still report a completed append as aborted, risking duplicate appends on retry.

## What This Changes

The PR adds an optional `append?: boolean` path to agent write operations for host/workspace/sandbox writes, updates exported write input/event types, and adds runtime/type tests.

## Best Possible Solution

Land append only after the maintainer-approved API shape aligns the base session write schema/runtime, OpenClaw host/sandbox wrappers, exported types/events, and append result semantics.

## Reproduction Assessment

Yes. Current main and the latest release expose ordinary `write` as overwrite-only while the linked issue has production truncation evidence; the PR blockers are source-reproducible from the head code paths.

## Solution Assessment

No, not yet. A first-class append primitive is the right direction, but this patch should align the public base write tool contract and completed-append result semantics before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Align append with the public write tool factory:** `src/agents/sessions/tools/tool-contracts.ts:79`
  - body: `WriteToolInput` now advertises `append?: boolean`, and `WriteToolCallEvent` inherits that public shape, but the exported base `createWriteToolDefinition`/`createWriteTool` path still exposes a schema with only `path` and `content` and overwrite-only behavior. That leaves SDK/session consumers with a type that says append is supported while factory-created write tools still ignore or reject the same field; either move append into the base session write tool contract/runtime or keep this shared type unchanged until the public factory is aligned.
  - confidence: 0.88
- **[P2] Return success once append has already mutated:** `src/agents/agent-tools.read.ts:948-950`
  - body: If the abort signal fires while `append()` is in flight, this helper still throws after `append()` returns successfully. A caller can then retry a tool call that already appended bytes, duplicating memory/log content; match the base write tool's post-write recovery invariant by reporting success once the append operation has completed.
  - confidence: 0.86
- **[P3] Make the append contract tests fail on removal:** `src/agents/sessions/tools/tool-contracts.test.ts:6-10`
  - body: This `toMatchTypeOf` direction still passes if `WriteToolInput` only contains `path` and `content`, because a type without an optional property is assignable to one with it. Assert the `append` key or property type directly here and in the sibling event test so the coverage actually protects the new contract.
  - confidence: 0.78
