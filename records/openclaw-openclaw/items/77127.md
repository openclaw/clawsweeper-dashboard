## Label Justifications

- `P1`: The linked issue describes real agent and cron workflows silently overwriting shared memory files, and this PR changes the core tool path intended to mitigate that data-loss workflow.
- `merge-risk: 🚨 compatibility`: The PR introduces a new public write-tool input field that SDK/tool consumers may depend on while not all exported write-tool implementations honor it.
- `merge-risk: 🚨 session-state`: The abort-after-mutation path can report a failed append after changing shared memory/log files, inviting duplicate retries or corrupted agent state.

## Summary

Keep open: current main and v2026.6.6 still lack a general append-capable write tool, and this PR is the active fix path for the linked data-loss workflow, but the patch is not merge-ready because the shared SDK-visible write input now advertises `append` while the base `createWriteTool` remains overwrite-only.

## What This Changes

The PR adds optional `append?: boolean` handling to OpenClaw host/sandbox agent write tools, updates the shared write-tool input type, and adds focused append regression/type tests.

## Best Possible Solution

Make append one canonical write-tool contract across runtime schema, base `createWriteTool` behavior, OpenClaw wrappers, events, and tests, with completed appends reported as success, then land after maintainers approve the `append: true` API shape.

## Reproduction Assessment

Yes, source-level. Current main and v2026.6.6 expose ordinary `write` as path/content overwrite-only, while the linked issue supplies cron and single-agent overwrite evidence.

## Solution Assessment

No, not yet. The append primitive is the right direction, but the patch must align the public base write-tool contract and fix append abort/result semantics before this is the best maintainable solution.

Full review comments:

- **[P1] Align the shared write input with the base write tool:** `src/agents/sessions/tools/tool-contracts.ts:79`
  - body: `WriteToolInput` is exported next to `createWriteTool`, but the base write factory still exposes only `path` and `content` and calls `ops.writeFile`, so an SDK consumer can now type-check `{ append: true }` and still overwrite the file. Either implement append in the base write tool contract or keep this field out of the shared exported input until that surface supports it.
  - confidence: 0.91
- **[P2] Return success once an append has mutated the file:** `src/agents/agent-tools.read.ts:948-950`
  - body: After `append()` completes, this branch still throws `AbortError` if the signal fired during the write. That reports a failed tool call after content may already be appended, unlike base `write` recovery, and a retry can duplicate the appended content.
  - confidence: 0.86
- **[P3] Make the append contract test fail when the field is missing:** `src/agents/sessions/tools/tool-contracts.test.ts:7-11`
  - body: This `toMatchTypeOf` direction still passes if `WriteToolInput` only has `path` and `content`, because a type without an optional field is assignable to one with that optional field. Assert the key or property type directly so the test protects the new contract.
  - confidence: 0.8
