## Summary

Keep this PR open, but it is not merge-ready: the append capability addresses the canonical data-loss report, yet the current diff still lets an active aborted append release the same-file mutation queue before the underlying append settles.

## What This Changes

The branch adds an optional `append?: boolean` path to the agent `write` tool for host/workspace and sandbox writes, with focused tests and a changelog entry.

## Best Possible Solution

Land an append-capable write surface only after preserving same-file serialization through active aborts and confirming the public tool API shape with maintainers.

## Reproduction Assessment

Yes by source inspection. Current main exposes ordinary `write` as create-or-overwrite while append is limited to memory-flush runs, and the PR regression is visible in the abort wrapper plus mutation queue control flow.

## Solution Assessment

No, not yet. The optional flag is a plausible narrow shape, but an active abort must keep the mutation queue occupied until the underlying append settles before reporting `AbortError`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Hold the queue until aborted appends settle:** `src/agents/pi-tools.read.ts:747-750`
  - body: If the signal aborts after `append()` has started, this handler rejects immediately. Because that promise is awaited inside `withFileMutationQueue`, the queue releases while the sandbox append can still be doing its read-then-write, so a later append to the same file can interleave and drop content. Keep the queue held until `append()` settles, then report the abort, and update the active-abort test to prove a following same-file append waits.
  - confidence: 0.86
