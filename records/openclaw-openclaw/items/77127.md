## Summary

Keep this PR open, but do not merge it as-is. The append-mode direction matches the open data-loss issue and the supplied terminal proof is sufficient, but the current PR head still lets a present non-boolean `append` value fall through to the overwrite path.

## What This Changes

Adds optional `append?: boolean` support to the agent `write` tool for host/workspace and sandbox writes, with regression tests and a changelog entry.

## Best Possible Solution

Land a maintainer-approved append-capable file primitive that preserves workspace and sandbox guards and rejects malformed append intent before any overwrite-capable fallback runs.

## Reproduction Assessment

Yes. By source inspection, current main routes ordinary host and sandbox writes through overwrite-only `createWriteTool`, while the PR head still reproduces the review bug because malformed present `append` values delegate to the overwrite implementation.

## Solution Assessment

No, not as-is. The optional flag is a plausible narrow implementation, but a present non-boolean `append` value must be rejected before any overwrite fallback can run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject malformed append values before overwriting:** `src/agents/pi-tools.read.ts:882-884`
  - body: When a model sends an `append` key with a non-boolean value like `"true"` or `1`, `doAppend` is false and this delegates to the original overwrite implementation. That can truncate the shared log this feature is meant to protect, so reject any present non-boolean `append` value before calling the base write tool.
  - confidence: 0.9
