## Summary

Keep open: the PR targets the right append-mode gap and now has real behavior proof, but the new append wrapper can release same-file serialization before an in-flight append has actually settled after an abort.

## What This Changes

The PR adds an optional `append: true` parameter to the agent `write` tool for host/workspace and sandbox writes, plus regression tests and a changelog entry for the canonical data-loss issue.

## Best Possible Solution

Land an append-capable write surface after preserving same-file serialization through aborted sandbox appends and getting maintainer approval for the model-facing parameter.

## Reproduction Assessment

Yes by source inspection. Current main delegates ordinary `write` to upstream overwrite behavior while append is scoped to memory-flush runs; the PR finding is also source-reproducible from the abort wrapper and mutation queue control flow.

## Solution Assessment

No, not yet. The optional flag is a plausible narrow API shape, but the current abort handling should keep the mutation queue occupied until the underlying append settles, or avoid active-abort rejection for non-cancelable writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Keep aborted appends in the mutation queue:** `src/agents/pi-tools.read.ts:904`
  - body: If the signal aborts after `ops.appendFile` has started, `appendFileWithAbort` rejects immediately. Because that rejection is awaited inside `withFileMutationQueue`, the queue releases while the underlying sandbox append can still be doing its read-then-write, so a later append to the same file can interleave and drop content. Keep the queue held until the append operation settles, then report the abort.
  - confidence: 0.82
