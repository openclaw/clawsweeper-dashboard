## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass gateway auth into plugin upgrades:** `src/gateway/server-http.ts:870`
  - body: `handlePluginUpgrade` always receives `gatewayAuthSatisfied: false`, so any plugin WebSocket route declaring `auth: "gateway"` is rejected even after the gateway has authenticated the request. That makes authenticated plugin upgrade routes unreachable and can misroute matched endpoints into the generic WebSocket path.
  - confidence: 0.95
- **[P2] Reject blocked plugin upgrades at the matched route:** `src/gateway/server/plugins-http.ts:190-192`
  - body: When a matched plugin upgrade route requires gateway auth, this handler logs and returns `false`. The caller treats that as unhandled and continues into the generic gateway WebSocket path, so the matched plugin endpoint is not rejected at its auth boundary.
  - confidence: 0.91
- **[P1] Bind node capabilities to the owning route:** `src/gateway/plugin-node-capability.ts:244-250`
  - body: Capability authorization is keyed only by the plugin-provided surface string. Two routes or plugins that reuse a surface name can share a token and bypass the intended owner or TTL policy for the route being accessed.
  - confidence: 0.9
- **[P2] Continue after resolver failures:** `src/media/web-media.ts:77-78`
  - body: A thrown error from one `hostedMediaResolver` escapes `loadWebMedia`, which can break unrelated media loads. Optional plugin-owned resolvers should be isolated per resolver and skipped unless they return a handled URL.
  - confidence: 0.86
- **[P2] Keep Crabbox on standard capacity by default:** `.crabbox.yaml:3`
  - body: Changing the repo default from `standard` to `beast` makes routine Crabbox sync/test commands request the largest owned AWS class, while current guidance reserves `beast` for exceptional maintainer-approved lanes. Keep the default conservative and require explicit escalation.
  - confidence: 0.88
- **[P2] Restore proof redaction guidance:** `.github/pull_request_template.md:40`
  - body: The template drops the reminder to redact IP addresses, API keys, phone numbers, and non-public endpoints from external proof. That weakens the proof gate and conflicts with current contributor privacy guidance.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces concrete gateway auth/capability regressions and weakens proof privacy and owned-cloud capacity safeguards.

Concerns:

- **[high] Plugin WebSocket auth context is lost:** `src/gateway/server-http.ts:870`
  - body: Plugin upgrades are dispatched with `gatewayAuthSatisfied: false`, so gateway-authenticated plugin WebSocket routes cannot authorize correctly and matched routes can fall through to the generic WebSocket path.
  - confidence: 0.94
- **[high] Blocked plugin upgrades do not fail closed:** `src/gateway/server/plugins-http.ts:190`
  - body: The upgrade handler returns `false` when gateway-auth preconditions fail, allowing the outer WebSocket upgrade path to continue instead of rejecting the matched plugin endpoint.
  - confidence: 0.89
- **[high] Node capability tokens are scoped only by surface:** `src/gateway/plugin-node-capability.ts:245`
  - body: Capability lookup accepts any live token for the same surface string, so another route or plugin using that surface can piggyback on credentials not issued for it.
  - confidence: 0.9

  - body: The PR template and contributing guidance remove explicit reminders to redact private information from proof artifacts, increasing the chance that public PR evidence leaks IPs, keys, phone numbers, or private endpoints.
  - confidence: 0.84
