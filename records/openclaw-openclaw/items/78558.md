## What This Changes

The PR removes vendored/generated A2UI surfaces, moves Canvas runtime ownership into a bundled experimental plugin, and adds generic plugin HTTP, WebSocket, media, CLI, and surface URL seams with native-client and docs updates.

## Best Possible Solution

Rebase and repair the Canvas plugin refactor so the generic plugin seams preserve node refresh authorization, WebSocket auth, capability routing, CLI conflict handling, resolver isolation, and route TTL semantics, then add redacted live Canvas/A2UI proof before merge.

Yes for the review findings: PR-head source inspection shows registered/native-called refresh methods are not node-role authorized, gateway-authenticated plugin WebSocket routes cannot pass auth dispatch, and blocked upgrade routes can fall through. ## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Allow node role to invoke plugin surface refresh:** `src/gateway/method-scopes.ts:32-39`
  - body: `node.pluginSurface.refresh` and the legacy Canvas refresh alias are registered handlers and native clients call them, but neither method is in `NODE_ROLE_METHODS`. Real node sessions will be rejected with `unauthorized role: node` before they can refresh an expired Canvas/plugin surface URL.
  - confidence: 0.96
- **[P1] Pass successful gateway auth into plugin upgrade dispatch:** `src/gateway/server-http.ts:869-871`
  - body: The upgrade path always calls `handlePluginUpgrade` with `gatewayAuthSatisfied: false`, so any plugin WebSocket route declared with `auth: "gateway"` is rejected even after the gateway has authenticated the request.
  - confidence: 0.94
- **[P2] Fail closed when plugin upgrade auth blocks a route:** `src/gateway/server/plugins-http.ts:189-193`
  - body: When a matched plugin upgrade route fails gateway-auth preconditions, returning `false` lets the caller treat it as unhandled and continue into the generic gateway WebSocket path. Unauthorized plugin upgrade attempts should terminate instead of falling through.
  - confidence: 0.9
- **[P2] Isolate hosted media resolver failures per plugin:** `src/media/web-media.ts:77-79`
  - body: A thrown exception from one registered hosted media resolver bubbles out of `loadWebMedia`, so one plugin can break media loading for unrelated URLs. Catch resolver failures, log them, and continue to the next resolver.
  - confidence: 0.86
- **[P2] Preserve built-in node subcommands when loading plugin CLI:** `src/plugins/register-plugin-cli-command-groups.ts:78-83`
  - body: The `primary` branch removes command groups before overlap checks when the parent path matches, so a plugin under `openclaw nodes` can replace built-in node subcommands such as `status`. Nested plugin commands should still respect existing-command conflict handling.
  - confidence: 0.87
- **[P2] Gate capability auth on the selected route:** `src/gateway/server/plugins-http/route-capability.ts:23-27`
  - body: Capability auth is based on the first matching route with `nodeCapability`, not necessarily the route that will handle the request. Overlapping routes can force capability auth onto a more-specific route that should not require it.
  - confidence: 0.84
- **[P2] Honor route-specific TTL at connect time:** `src/gateway/server/ws-connection/message-handler.ts:1325-1329`
  - body: Connect-time plugin surface URLs are minted from `{ surface }` only, so a route-level `nodeCapability.ttlMs` is ignored until later refresh paths. First-use capability lifetimes should honor the effective route contract.
  - confidence: 0.84
- **[P2] Reset cached Canvas host init promise after startup failure:** `extensions/canvas/src/http-route.ts:26-40`
  - body: `hostHandlerPromise ??=` caches a rejected initialization promise, so a transient Canvas host startup failure makes all later requests fail until process restart. Clear the cached promise on rejection so later requests can retry.
  - confidence: 0.86
