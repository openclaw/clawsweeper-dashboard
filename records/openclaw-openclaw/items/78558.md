## Summary

Keep open: the protected maintainer-labeled PR still has source-proven blocking issues in plugin WebSocket auth, core method advertisement, upgrade fallthrough, media resolver isolation, and proxied plugin-surface URL derivation, and its proof is only a verification command list.

## What This Changes

The PR moves Canvas runtime ownership into a bundled experimental Canvas plugin, removes vendored A2UI/generated surfaces, and adds generic plugin HTTP, WebSocket, media, CLI, hosted-surface, and node-capability seams across gateway, clients, docs, and tests.

## Best Possible Solution

Keep the Canvas-to-plugin direction only if the generic seams preserve Gateway auth, core method ownership, proxy URL semantics, and plugin fault isolation, with focused regression coverage plus redacted live Canvas/A2UI proof before merge.

## Reproduction Assessment

Yes for the review findings: PR-head source inspection shows the gateway-auth upgrade flag is hard-coded false, blocked matched upgrades return unhandled, core refresh methods are omitted from `BASE_METHODS`, plugin media resolver errors bubble globally, and plugin surface URLs force a bind-host override. I did not run live Canvas/A2UI or native paired-node flows.

## Solution Assessment

No. Moving Canvas behind generic plugin seams may be maintainable, but this patch is not mergeable until it fixes the auth, method ownership, fallthrough, media isolation, and proxy URL regressions and includes real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Pass gateway auth into plugin WebSocket upgrades:** `src/gateway/server-http.ts:870`
  - body: The upgrade path still calls `handlePluginUpgrade` with `gatewayAuthSatisfied: false`, so any plugin WebSocket route declared with `auth: "gateway"` is rejected even after the request has satisfied Gateway auth. This makes gateway-authenticated plugin upgrade routes unreachable.
  - confidence: 0.95
- **[P1] Advertise plugin surface refresh RPCs as core methods:** `src/gateway/server-methods-list.ts:146-150`
  - body: `node.pluginSurface.refresh` and the legacy Canvas refresh handler are implemented and allowed for node role, but neither is listed in `BASE_METHODS`. Since startup uses this list as core method ownership and dispatch prefers plugin `extraHandlers`, a plugin can shadow these core refresh methods and clients will not see them in advertised methods.
  - confidence: 0.9
- **[P2] Fail closed when plugin upgrade auth rejects a route:** `src/gateway/server/plugins-http.ts:190-192`
  - body: When a matched plugin upgrade route requires Gateway auth but the dispatch context is not satisfied, returning `false` makes the caller treat the request as unhandled and continue into the generic Gateway WebSocket upgrade path. A matched unauthorized plugin endpoint should terminate the upgrade instead of being accepted as a different route.
  - confidence: 0.91
- **[P2] Isolate hosted media resolver failures per plugin:** `src/media/web-media.ts:78`
  - body: A thrown error from one registered hosted media resolver currently escapes `resolveHostedPluginMediaUrl`, which can break media loading for unrelated URLs. Optional plugin resolvers should be caught and skipped unless they actually produce a handled result.
  - confidence: 0.84
- **[P2] Preserve proxy host resolution for plugin surface URLs:** `src/gateway/server/ws-connection.ts:252-259`
  - body: The PR derives `pluginSurfaceHostOverride` from any concrete bind host, which makes `resolveHostedPluginSurfaceUrl` ignore the request host and proxy-derived external port. Deployments bound to an internal interface but reached through a reverse proxy or tunnel can receive an unreachable internal Canvas/plugin surface URL.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The patch introduces concrete security-sensitive regressions at the plugin WebSocket auth boundary and core refresh-method ownership boundary.

Concerns:

- **[high] Plugin WebSocket auth can be bypassed or misrouted:** `src/gateway/server-http.ts:870`
  - body: Gateway-authenticated plugin upgrade routes are dispatched with `gatewayAuthSatisfied: false`, and blocked matched routes return unhandled to the generic WebSocket path instead of failing closed. This breaks route-specific upgrade authorization at the transport boundary.
  - confidence: 0.94
- **[medium] Plugins can shadow core refresh RPCs:** `src/gateway/server-methods-list.ts:146`
  - body: The new node plugin-surface refresh RPCs are implemented as core handlers but omitted from `BASE_METHODS`, while plugin extra handlers take precedence during dispatch. That leaves the core capability refresh behavior open to plugin name collisions and omits the methods from advertised support.
  - confidence: 0.88
