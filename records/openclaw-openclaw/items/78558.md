## Summary

Keep open: the latest PR head still has source-proven gateway plugin WebSocket auth and legacy Canvas config compatibility blockers, and the maintainer-protected XL refactor still needs real behavior proof before merge.

## What This Changes

The PR removes vendored A2UI/generated Canvas surfaces, moves Canvas runtime into a bundled experimental plugin, and adds generic plugin HTTP, WebSocket, media, CLI, and surface URL seams with native-client and docs updates.

## Best Possible Solution

Rework the refactor so generic plugin seams preserve existing Gateway auth, config-upgrade, proxy URL, and media isolation behavior, then add focused regression coverage and redacted live Canvas/A2UI proof.

## Reproduction Assessment

Yes for the review findings: PR-head source inspection shows gateway-authenticated plugin WebSocket upgrades cannot receive a satisfied auth context, blocked routes can fall through, and legacy `canvasHost` configs are rejected before doctor migration. I did not run live Canvas/A2UI or native paired-node flows.

## Solution Assessment

No. Moving Canvas behind generic plugin seams may be the right long-term direction, but this patch is not the best mergeable solution until it preserves auth, migration, and proxy behavior and includes real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Pass gateway auth into plugin WebSocket upgrades:** `src/gateway/server-http.ts:869-871`
  - body: The HTTP plugin path records successful gateway auth and forwards that context, but the upgrade path still calls `handlePluginUpgrade` with `gatewayAuthSatisfied: false`. Any plugin WebSocket route declared with `auth: "gateway"` is rejected even after the request was authenticated, so gateway-authenticated plugin upgrades cannot work.
  - confidence: 0.95
- **[P1] Keep legacy canvasHost valid until migration can run:** `src/config/zod-schema.ts:795`
  - body: The PR keeps a doctor migration for old `canvasHost` configs, but removes `canvasHost` from the strict root schema. Normal config validation safe-parses `OpenClawSchema` before doctor/fix runs, so upgraded users with an existing `canvasHost` key get an unrecognized-key startup/config-validation failure instead of a repair path.
  - confidence: 0.93
- **[P2] Reject blocked plugin upgrade routes without falling through:** `src/gateway/server/plugins-http.ts:190-192`
  - body: When a matched plugin upgrade route fails the gateway-auth precondition, returning `false` makes the caller treat it as unhandled and continue into the generic Gateway WebSocket upgrade path. A matched-but-unauthorized plugin endpoint should terminate the upgrade instead of being accepted as a different WebSocket route.
  - confidence: 0.91
- **[P2] Do not advertise bind-only plugin surface URLs behind proxies:** `src/gateway/server/ws-connection.ts:252-259`
  - body: Deriving `pluginSurfaceHostOverride` from any non-wildcard bind host forces `resolveHostedPluginSurfaceUrl` to ignore `requestHost` and proxy-derived external ports. Deployments that bind to an internal interface but are reached through a reverse proxy or tunnel can receive an internal host/port for Canvas plugin surfaces.
  - confidence: 0.86
- **[P2] Continue media resolution after plugin resolver failures:** `src/media/web-media.ts:77-78`
  - body: A thrown error from one hosted media resolver currently bubbles out of `loadWebMedia`, so one optional plugin resolver can break media loading for unrelated URLs. Catch resolver failures, log them, and continue to the next resolver unless that resolver actually produced a handled result.
  - confidence: 0.84
