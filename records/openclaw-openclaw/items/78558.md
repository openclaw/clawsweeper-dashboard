## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Pass gateway auth context into plugin upgrades:** `src/gateway/server-http.ts:870-871`
  - body: The upgrade path always calls plugin WebSocket dispatch with `gatewayAuthSatisfied: false`, but the plugin upgrade handler rejects routes registered with `auth: "gateway"` unless that flag is true. Authenticated plugin WebSocket endpoints can never be handled and fall through to the generic gateway WebSocket path.
  - confidence: 0.93
- **[P2] Forward canvas snapshot sanitization limits:** `extensions/canvas/src/tool.ts:217-221`
  - body: The moved Canvas snapshot path now returns `imageResultFromFile(...)` without forwarding the configured image sanitization limits. Current main passes those limits through, so users relying on `agents.defaults.imageMaxDimensionPx` can again produce oversized provider payloads.
  - confidence: 0.94
- **[P2] Add the managed NPM plugin helper:** `src/plugins/installed-plugin-index-records.test.ts:17`
  - body: These tests import `./test-helpers/managed-npm-plugin.js`, but that helper file is absent at the PR head. The affected plugin test file will fail during module loading before assertions run.
  - confidence: 0.91
- **[P2] Advertise Canvas only after plugin routes load:** `src/gateway/server-runtime-config.ts:126-129`
  - body: `canvasHostEnabled` is derived from config before Canvas plugin route registration is known. Because gateway plugin load errors are non-fatal, clients can receive a Canvas host URL even when the plugin failed to register handlers, leaving them with a dead URL or failed upgrade.
  - confidence: 0.86
- **[P3] Add a user-facing changelog entry:** `docs/refactor/canvas.md:10`
  - body: This PR changes Canvas distribution, config migration, CLI behavior, and plugin SDK surfaces, but the PR diff does not include `CHANGELOG.md`. OpenClaw policy requires a changelog entry for user-facing behavior changes before merge.
  - confidence: 0.84
