## Summary

Keep open: the latest PR head still has blocking gateway/plugin auth, node capability isolation, and plugin resolver fault-isolation issues, plus missing after-fix real behavior proof and a protected `maintainer` label.

## What This Changes

This PR removes vendored A2UI surfaces, moves Canvas runtime ownership into a bundled experimental Canvas plugin, adds generic plugin gateway/CLI/surface seams, and updates native clients, docs, CI, and SDK baselines.

## Best Possible Solution

Keep the Canvas-to-plugin direction only after the branch preserves gateway auth, route-owned capability isolation, plugin fault isolation, native client recovery, config migration semantics, docs/API baselines, and redacted live proof.

## Reproduction Assessment

Yes for the review findings: latest-head source inspection shows the hard-coded plugin upgrade auth flag, blocked-upgrade fallthrough, surface-only capability lookup, and media resolver exception propagation. I did not establish a live Canvas/A2UI or paired-node runtime reproduction.

## Solution Assessment

No. Moving Canvas behind generic plugin seams may be a maintainable direction, but this patch is not the safest merge path until the auth, capability scoping, resolver isolation, and real-behavior-proof gaps are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Pass gateway auth into plugin upgrades:** `src/gateway/server-http.ts:869-871`
  - body: `handlePluginUpgrade` always receives `gatewayAuthSatisfied: false`, so any plugin WebSocket route declaring `auth: "gateway"` is rejected even after the gateway has authenticated the request. That makes the new authenticated plugin upgrade seam unreachable.
  - confidence: 0.95
- **[P2] Fail closed for blocked plugin upgrades:** `src/gateway/server/plugins-http.ts:189-192`
  - body: When a matched plugin upgrade route requires gateway auth, this handler logs and returns `false`. The caller treats that as unhandled and continues into the generic gateway WebSocket path, so unauthorized plugin endpoint upgrades are not rejected at the matched route boundary.
  - confidence: 0.91
- **[P1] Scope capability tokens to the route owner:** `src/gateway/plugin-node-capability.ts:244-250`
  - body: Capability authorization is keyed only by the plugin-provided surface string. Two plugins or route families that reuse a surface name can share a token and bypass the intended owner or TTL policy for the route being accessed.
  - confidence: 0.9
- **[P2] Skip failing hosted media resolvers:** `src/media/web-media.ts:77-78`
  - body: A thrown error from one `hostedMediaResolver` escapes `loadWebMedia`, which can break unrelated media loads. Optional plugin-owned resolvers should be isolated per resolver and skipped unless they return a handled URL.
  - confidence: 0.86

  - confidence: 0.94

  - confidence: 0.89

  - body: Capability lookup accepts any live token for the same surface string, so another route or plugin using that surface can piggyback on credentials not issued for it.
  - confidence: 0.9
