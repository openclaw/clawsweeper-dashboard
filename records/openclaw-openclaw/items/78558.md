## Summary

Keep this PR open: it is an active protected `maintainer`-labeled XL refactor, current main has not implemented the Canvas plugin migration, and latest-head source inspection still shows blocking gateway/plugin security and correctness issues.

## What This Changes

The PR moves Canvas runtime and A2UI hosting into a bundled experimental Canvas plugin, removes vendored A2UI/generated surfaces, adds generic plugin gateway/CLI/surface seams, updates native clients to plugin surface URLs, and refreshes docs, CI, and SDK baselines.

## Best Possible Solution

Keep the Canvas-to-plugin direction only after the final branch preserves gateway auth, route-owned node capability isolation, core method ownership, plugin fault isolation, native-client recovery behavior, config migration semantics, docs/API baselines, and redacted live proof.

## Reproduction Assessment

Yes for the review findings: latest-head source inspection shows the hard-coded plugin upgrade auth flag, blocked-upgrade fallthrough, surface-only capability lookup, missing core method reservation, and media resolver exception propagation. I did not establish a live Canvas/A2UI or paired-node runtime reproduction.

## Solution Assessment

No. Moving Canvas behind generic plugin seams may be maintainable, but this patch is not the safest merge path until the auth, capability scoping, method ownership, resolver isolation, and real-behavior-proof gaps are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Pass gateway auth into plugin upgrades:** `src/gateway/server-http.ts:869-871`
  - body: `handlePluginUpgrade` always receives `gatewayAuthSatisfied: false`, so any plugin WebSocket route that declares `auth: "gateway"` is rejected even after the gateway has authenticated the request. That makes the new authenticated plugin upgrade seam unreachable.
  - confidence: 0.95
- **[P2] Fail closed for blocked plugin upgrades:** `src/gateway/server/plugins-http.ts:189-192`
  - body: When a matched plugin upgrade route requires gateway auth, this handler logs and returns `false`. The caller treats that as unhandled and continues into the generic gateway WebSocket path, so unauthorized plugin endpoint upgrades are not rejected at the matched route boundary.
  - confidence: 0.91
- **[P1] Scope capability tokens to the route owner:** `src/gateway/plugin-node-capability.ts:238-245`
  - body: Capability authorization is keyed only by the plugin-provided surface string. Two plugins or route families that reuse a surface name can share a token and bypass the intended owner or TTL policy for the route being accessed.
  - confidence: 0.9
- **[P1] Advertise and reserve plugin surface refresh:** `src/gateway/server-methods-list.ts:149-151`
  - body: `node.pluginSurface.refresh` is implemented as a core node handler and allowed for the node role, but it is missing from `BASE_METHODS`. Clients that gate on advertised methods will not see it, and plugins can collide with the core refresh RPC during method registration.
  - confidence: 0.88
- **[P2] Skip failing hosted media resolvers:** `src/media/web-media.ts:77-78`
  - body: A thrown error from one `hostedMediaResolver` escapes `loadWebMedia`, which can break unrelated media loads. Optional plugin-owned resolvers should be isolated per resolver and skipped unless they return a handled URL.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch introduces unresolved security-sensitive regressions in plugin WebSocket auth, node capability token ownership, and core refresh-method reservation.

Concerns:

- **[high] Plugin WebSocket auth context is lost:** `src/gateway/server-http.ts:870`
  - body: Plugin upgrades are dispatched with `gatewayAuthSatisfied: false`, so gateway-authenticated plugin WebSocket routes cannot authorize correctly and blocked routes can fall through to the generic gateway WebSocket path.
  - confidence: 0.94
- **[high] Node capability tokens are scoped only by surface:** `src/gateway/plugin-node-capability.ts:245`
  - body: Capability lookup accepts any live token for the same surface string, so another route or plugin using that surface can piggyback on credentials not issued for it.
  - confidence: 0.9
- **[medium] Core refresh RPC is not reserved or advertised:** `src/gateway/server-methods-list.ts:150`
  - body: The new core `node.pluginSurface.refresh` handler is missing from `BASE_METHODS`, which weakens method ownership and advertised support for a security-sensitive capability refresh path.
  - confidence: 0.86
