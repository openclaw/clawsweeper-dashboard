## Summary

Keep open: this protected, maintainer-labeled XL refactor still has source-proven gateway/plugin auth, capability ownership, method reservation, media isolation, and proxy URL regressions, plus missing real behavior proof.

## What This Changes

The PR moves Canvas runtime ownership into a bundled experimental Canvas plugin, removes vendored A2UI/generated surfaces, and adds generic plugin HTTP, WebSocket, media, CLI, hosted-surface, and node-capability seams.

## Best Possible Solution

Keep the Canvas-to-plugin direction only after preserving gateway auth, fail-closed plugin route handling, route-owned capability tokens, core refresh RPC ownership, plugin fault isolation, proxy URL semantics, focused regression coverage, and redacted live proof.

## Reproduction Assessment

Yes for the review findings: latest-head source inspection shows the hard-coded plugin upgrade auth flag, blocked-upgrade fallthrough, surface-only capability lookup, missing core method reservation, media resolver exception propagation, and bind-host URL override. I did not establish a live Canvas/A2UI or native paired-node reproduction path.

## Solution Assessment

No. Moving Canvas behind generic plugin seams may be maintainable, but this patch is not the safest merge path until the auth, capability, method ownership, fault-isolation, and proxy regressions are fixed and real behavior proof is attached.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Scope capability tokens to the route owner:** `src/gateway/plugin-node-capability.ts:222-227`
  - body: Capability authorization indexes tokens only by the plugin-provided `surface`, so two plugins or route families using the same surface implicitly share credentials. A route can piggyback on another surface credential and bypass intended owner isolation or TTL policy.
  - confidence: 0.91
- **[P1] Pass authenticated gateway context to plugin upgrades:** `src/gateway/server-http.ts:869-871`
  - body: The upgrade dispatcher always passes `gatewayAuthSatisfied: false`, so plugin WebSocket routes that declare `auth: "gateway"` cannot succeed even after the gateway has authenticated the request. This breaks the new authenticated plugin WebSocket seam.
  - confidence: 0.95
- **[P2] Reject unauthorized matched plugin upgrades:** `src/gateway/server/plugins-http.ts:189-192`
  - body: A matched plugin upgrade route that requires gateway auth returns `false` when auth is missing, which the caller treats as unhandled and then continues toward the generic gateway WebSocket path. Matched unauthorized plugin endpoints should fail closed after the auth decision.
  - confidence: 0.91
- **[P1] Keep plugin surface refresh RPCs in BASE_METHODS:** `src/gateway/server-methods-list.ts:146-150`
  - body: The branch implements `node.pluginSurface.refresh` as a core node handler but omits it from `BASE_METHODS`, which startup uses for core method ownership and advertised method support. A plugin can collide with the refresh RPC through extra handlers, and clients that gate calls on advertised methods will not see it.
  - confidence: 0.88
- **[P2] Skip failing hosted media resolvers:** `src/media/web-media.ts:77-79`
  - body: One plugin resolver throwing from `entry.resolver(mediaUrl)` aborts hosted media resolution, which can break unrelated media loads. Optional plugin-owned resolvers should be caught and skipped unless they return a handled URL.
  - confidence: 0.86
- **[P2] Preserve proxy host resolution for plugin surfaces:** `src/gateway/server/ws-connection.ts:252-259`
  - body: Deriving `hostOverride` from every concrete bind host causes hosted plugin surface URLs to ignore request/proxy host semantics. Gateways bound to an internal interface but reached through a reverse proxy or tunnel can advertise unreachable Canvas/plugin URLs.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces concrete security-sensitive regressions in plugin WebSocket auth, node capability token ownership, and core refresh-method reservation.

Concerns:

- **[high] Plugin WebSocket auth can be bypassed or misrouted:** `src/gateway/server-http.ts:870`
  - body: Plugin upgrades are dispatched with `gatewayAuthSatisfied: false`, and matched routes that fail gateway-auth checks return unhandled instead of failing closed. That weakens route-specific authorization at the transport boundary.
  - confidence: 0.94
- **[high] Node capability tokens are scoped only by surface string:** `src/gateway/plugin-node-capability.ts:222`
  - body: Capability lookup accepts any client token for the same plugin-provided surface name, so another plugin route can reuse that surface and share credentials instead of being isolated by plugin or route owner.
  - confidence: 0.91
- **[medium] Plugins can collide with core refresh RPCs:** `src/gateway/server-methods-list.ts:146`
  - body: The new plugin-surface refresh handler is implemented as a core node handler but omitted from `BASE_METHODS`, leaving ownership and advertised support inconsistent while plugin extra handlers have dispatch precedence.
  - confidence: 0.88
