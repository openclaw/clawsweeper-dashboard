## Summary

## What This Changes

The PR wraps node-pairing read helpers in the existing pairing mutex and changes the macOS OpenClawKit snapshot wait after connect from 500 ms to 5000 ms.

## Best Possible Solution

Land a narrow fix that serializes node-pairing state access and either validates or explicitly accepts the longer snapshot grace period with redacted after-fix macOS node proof.

## Reproduction Assessment

No high-confidence live reproduction was run. The linked issue has concrete commands/logs and current source shows the implicated node-pairing reads outside the existing mutex, so the pairing-file race is source-reproducible but the full macOS flapping regression is not locally reproduced.

## Solution Assessment

Unclear as a complete fix: serializing node-pairing reads under the existing lock is the right boundary for the file race, but the 5s snapshot wait needs matching macOS node proof or an explicit maintainer decision because it changes shared connection timing.
