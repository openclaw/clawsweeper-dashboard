## Summary

Keep this open: current main has explicit reset/delete, one-shot, idle, and process-tree cleanup, but it still keys bundled MCP runtimes by session id and allows separate sessions to overlap until the idle TTL. That leaves the reported file-lock contention plausible, and the open shared-runtime PR is a candidate path but not merged.

## What This Changes

The issue reports leaked MCP server processes across OpenClaw sessions, causing stale PIDs to hold file locks and block later MCP tool calls.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/79882 or choose an equivalent lifecycle policy so lock-sensitive MCP servers are reused or reaped before a competing same-server process is spawned.

## Reproduction Assessment

Yes, source-reproducible with medium confidence: current main creates bundled MCP runtimes per session id and only reaps separate idle sessions after the configured TTL. I did not run the reporter's exact LadybugDB lock-contention setup.

## Solution Assessment

Unclear: the issue correctly points at OpenClaw-owned MCP lifecycle management, but the best permanent fix needs maintainer choice between shared runtime reuse, stricter reap-before-spawn, or an explicit per-server lifecycle policy.
