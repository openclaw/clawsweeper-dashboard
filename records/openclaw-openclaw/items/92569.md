## Label Justifications

- `P2`: This is a normal-priority MCP lifecycle bug with limited but real runtime availability impact for users with file-locking MCP servers.
- `impact:crash-loop`: Leaked MCP child processes can keep file locks and push MCP calls into repeated failure/backoff states.
- `impact:session-state`: The bug is rooted in stale session-to-runtime lifecycle state when a session key transitions to a new session id.

## Summary

Keep open: current `main` and the latest release still contain the MCP runtime lifecycle gaps, and the related live PRs show an open partial reset-path fix plus a closed unsafe rebind-disposal attempt rather than a merged complete fix.

## What This Changes

The issue reports leaked MCP server child processes across session transitions, where a stale process can hold an exclusive file lock and make the next session's MCP tool calls fail.

## Best Possible Solution

Implement the canonical MCP runtime handoff: atomically bind and lease the selected runtime, mark displaced runtimes for deferred retirement, and dispose them only after the final lease releases, with focused coverage for reset timeout and sessionKey rebind paths.

## Reproduction Assessment

Yes. Source inspection shows current `main` can rebind a session key to a new session id while leaving the prior MCP runtime alive until idle sweep, and the gateway reset path can skip MCP retirement when the abort wait times out.

## Solution Assessment

No complete fix is on `main`. The open PR covers only the reset timeout path, while the closed rebind PR was rejected as unsafe; the best remaining path is the maintainer-described atomic lease handoff plus deferred retirement.
