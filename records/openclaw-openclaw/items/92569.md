## What This Changes

The issue reports leaked MCP server processes across sessions, where stale children can keep file locks and make the next session's MCP tools fail.

## Best Possible Solution

Land a lifecycle-safe MCP cleanup fix that retires stale runtimes after the run drains, is explicitly force-cleared, or defers disposal until final lease release, with coverage for reset/delete timeout and any confirmed sessionKey-rebind path.

## Reproduction Assessment

Yes at source level: current main can return from the session reset timeout path without retiring the MCP runtime, and related code shows session-keyed runtimes can remain alive until explicit disposal or idle sweep. I did not run the live WSL2/LadybugDB file-locking scenario in this read-only review.

## Solution Assessment

No for the current linked PR as submitted: reaping in the lifecycle owner is the right direction, but immediate pre-wait disposal is unsafe without a drained, force-cleared, or deferred-lease contract. The best fix should preserve active MCP work while still guaranteeing stale process cleanup.
