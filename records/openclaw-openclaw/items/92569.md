## Summary

Keep open: current main and the latest release still have the session-reset timeout path where MCP runtime retirement is skipped, and the open closing PR is only a candidate until maintainers resolve the lifecycle-safety concerns.

## What This Changes

The issue reports leaked MCP server processes across session boundaries where stale child processes hold exclusive file locks and block the next session's MCP tools.

## Best Possible Solution

Land or replace the open candidate with a lifecycle-safe cleanup that retires stale MCP runtimes after drain, explicit force-clear, or deferred final lease release, with reset/delete timeout coverage.

## Reproduction Assessment

Yes at source level: current main skips MCP runtime retirement when waitForEmbeddedAgentRunEnd times out, and the latest release has the same path. I did not execute the WSL2/LadybugDB lock scenario in this read-only review.

## Solution Assessment

No, current main is not the best solution because the documented reset/delete cleanup contract is not consistently met. The best fix is a lease-safe cleanup path rather than pushing stale-PID self-healing into each MCP server or adding a restart-strategy config first.
