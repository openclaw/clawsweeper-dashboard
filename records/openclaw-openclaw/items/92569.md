## Summary

Keep open: current main and the latest release still show a session reset timeout path that can return before retiring the session MCP runtime, and the linked closing PR has not merged yet. ## What This Changes

Ensure MCP server processes are reused or reaped across session boundaries so stale processes cannot hold embedded database file locks on reconnect.

## Best Possible Solution

Land a focused lifecycle fix that reaps the old session MCP runtime before a replacement session can spawn, with regression coverage for the timeout path and any confirmed sessionKey-rebind path.

## Reproduction Assessment

Yes at source level. Current main shows the timeout branch can return before retireSessionMcpRuntime runs, and the sessionKey mapping path can rebind without disposing the prior runtime; I did not run a live file-locking MCP server repro.

## Solution Assessment
