## Summary

Keep open: current main and the latest release still have source-level MCP runtime cleanup gaps, and the linked open PR is a candidate fix rather than a reason to close before merge.

## What This Changes

The issue reports MCP server child processes surviving session boundaries and holding exclusive file locks on later reconnects.

## Best Possible Solution

Land or replace the open candidate PR with a lease-aware MCP runtime cleanup contract that covers reset/delete timeout behavior and the sessionKey rebind case, then close this issue after the fix merges.

## Reproduction Assessment

Yes from source inspection: current main can skip MCP retirement on the session-reset timeout path, and the manager can remap a sessionKey without disposing the prior session runtime. I did not run a live file-locking MCP server reproduction because this review is read-only.

## Solution Assessment

Not yet. The source confirms a real bug, but the safest solution should be lease-aware or deferred rather than an unconditional kill-before-wait path that can close transports under active MCP work.
