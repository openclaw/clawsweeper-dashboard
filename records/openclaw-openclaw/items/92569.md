## Label Justifications

- `P2`: The report describes a concrete MCP lifecycle bug with lock contention and backoff, but not a confirmed broad outage or data-loss emergency.
- `impact:crash-loop`: Stale MCP server processes can make the MCP subsystem repeatedly fail and enter paused/backoff behavior, which is a process-level availability problem.
- `impact:other`: The issue also concerns leaked external MCP server processes and exclusive file-lock contention, which is maintainer-visible but outside the narrower owned labels.

## Summary

Keep this open: current main and the latest release still intentionally cache session-scoped MCP runtimes for 10 minutes by default, while the reporter describes a concrete file-lock failure when a fresh session starts the same stdio MCP server before the old runtime is reaped. Existing cleanup covers explicit reset/delete and one-shot runs, but the reconnect/new-session handoff still needs maintainer review rather than an auto-close.

## What This Changes

The issue reports leaked MCP server processes across gateway sessions causing stale file locks on reconnect for file-locking MCP servers.

## Best Possible Solution

Preserve normal warm-session reuse, but add lifecycle-owned handling or operator guidance so a fresh effective session cannot start a second stdio MCP server that contends with a still-warm file-locking server from the previous session.

## Reproduction Assessment

No, not with a high-confidence live reproduction. The source path is credible because current main and v2026.6.6 keep session-scoped MCP runtimes warm by default, but the reporter's WSL2/LadybugDB reconnect flow was not executed here.

## Solution Assessment

Unclear. A per-server restart strategy would be a new product/config contract, while a narrower lifecycle fix at the reconnect or session-handoff boundary may solve the lock contention without adding config.
