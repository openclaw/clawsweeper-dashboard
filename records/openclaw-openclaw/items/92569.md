## Summary

Keep this open: current main still has source-level MCP lifecycle paths that can leave stale session runtimes alive, and the open closing PR for this issue has not merged. A related manager-level PR was closed unmerged after maintainer review found the immediate rebind-dispose approach unsafe, so the remaining work needs a safe lifecycle repair rather than cleanup closure.

## What This Changes

The issue requests that OpenClaw stop leaking stale MCP server processes across session transitions so file-locking MCP servers do not block the next session.

## Best Possible Solution

Implement one canonical MCP runtime lifecycle invariant that prevents stale runtimes from surviving session transitions, using atomic checkout/lease handoff and deferred retirement where needed, then close this issue after the merged fix is proven.

## Reproduction Assessment

Yes, at source level. Current main can rebind a sessionKey to a new sessionId while the prior runtime remains registered until idle sweep, and the reset-timeout branch can return without retiring that session's MCP runtime; I did not run the live WSL2/LadybugDB file-lock scenario.

## Solution Assessment

The requested direction is right, but the safe implementation is still unsettled. The rejected immediate manager-dispose patch is not the best fix; the best path is a manager-owned atomic handoff/deferred-retirement repair plus separate review of the open reset-timeout PR.
