## Label Justifications

- `P1`: The patch can silently attribute a later real model call to the wrong upstream caller during ordinary session reuse.
- `merge-risk: 🚨 session-state`: Exporter attribution remains cached after core has explicitly cleared the reused session’s context.
- `merge-risk: 🚨 security-boundary`: Opaque caller metadata from one run can cross the intended run boundary and be exported on another run’s trusted telemetry.
- `merge-risk: 🚨 compatibility`: The PR adds coordinated gateway protocol, SDK, Swift model, and plugin diagnostic contract surfaces requiring explicit upgrade and ownership review.

## Summary

This PR should stay open but is not merge-ready: its trusted-channel design is plausible, yet the bundled exporter retains the previous run’s context when a session is reused without context, so later model spans can be silently attributed to the wrong upstream caller; the supplied live proof also bypasses that exporter cache.

## What This Changes

The PR adds an optional gateway and SDK `clientContext` field, carries it through trusted diagnostic private data, and maps it onto bundled diagnostics-otel model-call span attributes.

## Best Possible Solution

Make lifecycle events authoritative by deleting every matching exporter cache alias when `clientContext` is absent, add seeded-then-unseeded same-session coverage, and directly prove both runs through the bundled diagnostics-otel exporter before approving the new gateway/SDK contract.

## Reproduction Assessment

Yes from source inspection: seed a session with context, then reuse that session without context and emit a model call; core clears its value, but the exporter retains and resolves the prior bag.

## Solution Assessment

No in its current form; the trusted private-data approach is maintainable in principle, but the exporter must honor explicit clearing and the bundled consumer must be proven live before this becomes the best safe implementation.
