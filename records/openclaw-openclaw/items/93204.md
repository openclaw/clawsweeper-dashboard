## Summary

Keep open: current main and the latest release still lack Mattermost server-side thread-history backfill, and multiple open linked PRs already target this issue with closing syntax.

## What This Changes

Mattermost should backfill thread history from the server when a thread mention arrives after restart or session-clear and the in-memory history window is empty.

## Best Possible Solution

Land one canonical Mattermost plugin fix that backfills supported thread context from Mattermost's thread API only on recovery, bounds entries by `historyLimit`, and proves the behavior with focused tests plus live Mattermost evidence.

## Reproduction Assessment

Yes, at source level. Current main creates a process-local Mattermost `channelHistories` map and builds thread context only from that map, with no server-thread fetch before dispatch.

## Solution Assessment

Yes, with scope guards. Mattermost-owned recovery backfill through the existing client/history path is the narrow fix, but it should land through one open candidate PR with recovery-aware gating and proof.
