## Summary

Keep open: current main and v2026.6.8 still lack server-side Mattermost thread backfill, and multiple open closing PRs already own candidate implementations; this issue should remain the canonical tracker until one lands.

## What This Changes

The issue requests Mattermost server-side thread-history backfill after restart or session clear before building inbound thread context.

## Best Possible Solution

Land one canonical Mattermost-owned fix that backfills supported thread-scoped inbound messages from the server on recovery, bounds history processing, and keeps DM-threading policy separate.

## Reproduction Assessment

Yes, at source level. Current main initializes a fresh in-memory Mattermost history map and builds thread context only from that map, so restart or session clear leaves no prior server thread context before dispatch.

## Solution Assessment

Yes for supported group/channel thread recovery: the narrow maintainable fix is a Mattermost plugin backfill before context construction. Expanding direct-message threading should not be bundled into this repair without a separate product decision.
