## Label Justifications

- `P2`: The issue is a real Mattermost channel bug with limited blast radius and active fix PRs, not an emergency runtime outage.
- `impact:session-state`: The reported failure loses thread-scoped conversation context after restart or session clear.
- `impact:message-loss`: Prior thread messages are omitted from the recovered agent turn, so the reply can be generated without the conversation history users expect.

## Summary

Keep open: current main and the latest release still build Mattermost thread context only from the process-local pending-history map, and three open closing PRs now own candidate fixes for the same recovery bug.

## What This Changes

The issue asks Mattermost to backfill an existing thread from the server after gateway restart or session clear before building inbound thread context.

## Best Possible Solution

Land one focused Mattermost PR that backfills only during restart/session-clear recovery, skips the current post, trims to `historyLimit`, and adds regression coverage for active threads whose pending window was cleared after a successful turn.

## Reproduction Assessment

Yes at source level: current main initializes a process-local Mattermost history map and builds inbound context only from that map, with no server backfill path. I did not run a live Mattermost restart scenario in this read-only review.

## Solution Assessment

Yes, a Mattermost-plugin backfill before context construction is the right boundary; the best version must be recovery-gated because the shared turn kernel clears pending history after successful dispatch.
