## Summary

Keep open: current main and v2026.6.8 still build Mattermost thread context from process-local history only, while three open PRs are active closing candidates for this exact recovery gap.

## What This Changes

Request server-side Mattermost thread history backfill when an inbound thread message arrives after restart or session clear with an empty in-memory history window.

## Best Possible Solution

Land one focused Mattermost plugin fix that backfills supported thread history from the Mattermost server only for recovery/cold-thread cases before context construction, while keeping DM-threading policy separate.

## Reproduction Assessment

Yes, at source level. A fresh/restarted Mattermost monitor has an empty `channelHistories` map, and current main builds pending context and `InboundHistory` from that map without fetching `/posts/{threadRootId}/thread`.

## Solution Assessment

Yes, with a recovery gate. The narrow maintainable solution is Mattermost-owned server-thread backfill before context construction, not a core state change or a new config option.
