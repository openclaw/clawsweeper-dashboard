## Label Justifications

- `P2`: The bug affects a real Mattermost recovery flow but is bounded to one channel plugin path.
- `impact:session-state`: The reported failure is loss of thread conversation context after restart or session clear.
- `impact:message-loss`: Prior thread messages are omitted from the inbound agent context, so relevant channel context is effectively missing for that turn.

## Summary

Keep open: current main and the latest release still build Mattermost thread context only from the in-memory history map, and several open PRs already target the missing server-side backfill. The issue remains necessary until maintainers choose and land one canonical recovery-gated fix.

## What This Changes

Mattermost should backfill server-side thread history after restart or session clear when the in-memory thread window is empty.

## Best Possible Solution

Keep this issue open until one recovery-gated Mattermost backfill PR lands, then close it as implemented with the merge commit and release or main-only provenance.

## Reproduction Assessment

Yes at source level. Current main initializes a process-local `channelHistories` map, builds the Mattermost prompt and structured inbound history only from that map, and has no server-side thread fetch before dispatch.

## Solution Assessment

Yes, the right solution boundary is a Mattermost-owned backfill before context construction. The implementation should be recovery-gated rather than triggered by every empty history window, because the shared turn kernel clears pending history after successful turns.
