## What This Changes

The issue requests Mattermost server-side thread backfill when an existing thread is mentioned after restart or session clear and the in-memory history window is empty.

## Best Possible Solution

Land one canonical Mattermost plugin fix that backfills accepted thread recovery turns from Mattermost's thread API, bounds work by historyLimit, and preserves ordinary active-thread behavior.

## Reproduction Assessment

Yes, at source level. Current main creates a process-local Mattermost history map, derives the thread history key locally, and builds pending/inbound context from that map without a server-thread fetch.

## Solution Assessment

Yes, the requested Mattermost-local backfill is the right owner boundary, but the best implementation must use a recovery-aware trigger rather than every empty pending-history window.
