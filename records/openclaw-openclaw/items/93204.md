## Label Justifications

- `P2`: The issue affects real Mattermost thread recovery but is bounded to one plugin/channel path and is not a core outage.
- `impact:session-state`: The report is about losing thread-scoped context after gateway restart or session clear.
- `impact:message-loss`: Prior thread messages are omitted from the agent's inbound context for the recovery turn, so relevant conversation content is effectively suppressed.

## Summary

Keep open: current main and v2026.6.8 still build Mattermost inbound thread context from a process-local history map without a server-side thread fetch, and open linked fix PRs are already competing to own the repair.

## What This Changes

The issue requests Mattermost thread-history backfill from the server when an inbound thread mention arrives after restart or session clear and the local history window is empty.

## Best Possible Solution

Land one Mattermost-owned fix that backfills supported channel/group thread history only on recovery, with focused client/monitor tests and live Mattermost proof; leave DM threading policy to its separate track.

## Reproduction Assessment

Yes, source-reproducible: current main starts Mattermost with an empty in-memory history map and builds thread context only from that map, with no server-thread fetch before dispatch.

## Solution Assessment

Yes for the narrowed recovery path: the best fix is a Mattermost plugin helper that seeds the existing history window from Mattermost's thread API before context construction, gated so normal post-turn empty windows do not refetch.
