## Summary

Keep open: current main and the latest release still lack Mattermost server-side thread backfill, and three open PRs already use closing references for this issue.

## What This Changes

The issue requests Mattermost server-side thread-history backfill when a thread-scoped mention arrives after restart or session clear and the in-memory history window is empty.

## Best Possible Solution

Land one canonical Mattermost plugin fix that backfills supported thread context from Mattermost’s thread API only on recovery, bounds work by historyLimit, and keeps DM-thread scope explicit or separately tracked.

## Reproduction Assessment

Yes, at source level. Current main initializes Mattermost channelHistories in memory and builds thread context only from that map, so a cold process has no prior server-side thread messages to include.

## Solution Assessment

Yes, with recovery-aware gating. A Mattermost-owned backfill through the existing client/history path is the right repair shape, but one open candidate PR should be selected and reviewed rather than opening another duplicate fix.
