## Label Justifications

- `P2`: The issue affects real Mattermost thread recovery but is bounded to one plugin/channel path and is not a core outage.
- `impact:session-state`: The report is about losing Mattermost thread context after restart or session clear.
- `impact:message-loss`: Prior thread messages are omitted from the agent's inbound context, so relevant channel messages are effectively suppressed for that recovery turn.

## Summary

Keep open: current main and the latest release still build Mattermost thread context only from an in-memory history map, so a restarted or session-cleared gateway has no server-side backfill before dispatching the inbound thread turn.

## What This Changes

The issue requests Mattermost server-side thread-history backfill when an inbound thread message arrives with an empty in-memory history window after restart or session clear.

## Best Possible Solution

Add Mattermost-owned backfill for supported thread-scoped inbound messages when `threadRootId` and `historyKey` are present and the local history window is empty, while keeping DM-threading policy changes separate.

## Reproduction Assessment

Yes. Source inspection shows a fresh gateway has an empty `channelHistories` map, and the Mattermost inbound path builds thread context only from that map without fetching the server thread.

## Solution Assessment

Yes, with a scope guard. The maintainable fix is a Mattermost plugin helper that seeds the existing history window from the Mattermost thread API before context construction, without changing direct-message threading policy or adding config.
