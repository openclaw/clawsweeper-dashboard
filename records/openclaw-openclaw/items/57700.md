## Summary

Keep open. Current main and the latest release still intentionally surface non-ok or thrown hook-agent outcomes as model-visible untrusted system events; only successful silent/already-attempted hook fallbacks were fixed, so the remaining delivery-artifact/error relay policy still needs maintainer judgment.

## What This Changes

The issue requests that hook delivery errors or artifacts stop being injected into agent-visible sessions, or that OpenClaw provide an equivalent suppression policy.

## Best Possible Solution

Define a hook relay/event-audience policy that keeps delivery artifacts out of model-visible prompts while preserving operator-visible failure diagnostics, explicit opt-in surfacing if needed, and regression coverage for returned non-ok, skipped, thrown, direct, and explicit-agent hook paths.

## Reproduction Assessment

Yes, by source inspection. A hook agent run that returns status "error" or "skipped", or throws, still reaches enqueueSystemEvent, and those queued events are drained into the next prompt as System (untrusted) lines.

## Solution Assessment

Unclear. Suppressing noisy delivery artifacts is the right direction, but the maintainable solution should be a hook dispatcher/event-audience policy rather than an outbound Discord text filter, and maintainers need to decide failure visibility semantics.
