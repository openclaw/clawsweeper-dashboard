## Summary

Keep open. Current main and v2026.5.18 still enqueue non-ok and thrown hook-agent results as prompt-visible system events, and the related waitForResult/announceToMain work is still open and needs Gateway hook policy review rather than cleanup closure.

## What This Changes

The issue requests a hook delivery relay policy or suppression option so hook error/artifact system events do not become model-visible agent session context and noisy channel replies.

## Best Possible Solution

Define an explicit hook event-audience policy or API option that keeps machine-only hook artifacts out of model-visible prompts while preserving operator diagnostics, trust metadata, idempotency, timeout/error semantics, and docs.

## Reproduction Assessment

Yes, by source inspection: a `/hooks/agent` run that returns `status: "error"` or `status: "skipped"`, or throws before completion, still reaches `enqueueSystemEvent`, and those queued entries are drained into the next prompt. I did not run a live Discord reproduction in this read-only review.

## Solution Assessment

Unclear. Suppressing prompt injection for machine-only hook artifacts is plausible, but current tests intentionally preserve non-ok announcements, so the maintainable fix needs a Gateway hook event-audience/API decision rather than a Discord-only outbound text filter.
