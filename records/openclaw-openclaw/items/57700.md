## Summary

Keep open. Current main has a partial hook-announcement fix, but it still intentionally enqueues non-ok hook results and thrown hook-agent errors as untrusted `Hook ...` system events into an agent main session, where the normal reply path can prepend them into model-visible prompt text. The broader system-event audience work is related but does not clearly supersede this hook-specific delivery-relay policy gap.

## What This Changes

The issue requests a core hook delivery policy that stops hook delivery errors or artifacts from being injected into agent-visible sessions, or provides an equivalent suppression/filter mechanism.

## Best Possible Solution

Add an explicit hook delivery relay policy in core hook handling: keep already-handled delivery artifacts out of agent-visible prompts, decide whether hook failures are internal-only by default or configurable, and align the implementation with the emerging system-event audience model so hooks do not grow a one-off visibility path.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: mock `runCronIsolatedAgentTurn` to return a non-ok result or reject, and the current hook tests already assert that `enqueueSystemEvent` receives `Hook ... (error)` entries that later drain into prompt text.

## Solution Assessment

No. Current main’s partial fix is useful but not sufficient for the requested behavior; AGENTS.md instructions and outbound string filters are weaker than a hook relay/audience policy because the artifact has already entered the agent prompt by then.
