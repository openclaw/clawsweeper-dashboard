## Summary

Keep open. Current main still deliberately routes returned non-ok and thrown hook-agent outcomes into model-visible untrusted system events; the already-merged narrow hook announcement fix only silenced successful no-delivery or already-attempted paths, so the remaining failure/artifact relay behavior needs maintainer policy/API review.

## What This Changes

The issue requests suppressing or filtering hook delivery error/artifact events so they do not leak into target agent sessions and create channel noise.

## Best Possible Solution

Add a hook relay/event-audience policy that keeps delivery artifacts and optionally failed hook diagnostics out of model-visible prompts while preserving operator-visible diagnostics and focused regression coverage.

## Reproduction Assessment

Yes, by source inspection. A hook agent run that returns `status:"error"` or `status:"skipped"`, or throws before completion, still reaches `enqueueSystemEvent`, and those queued events are drained into the next prompt.

## Solution Assessment

Unclear. Suppressing prompt injection for hook artifacts is reasonable, but an outbound Discord text filter is the wrong layer; the maintainable fix belongs in hook dispatcher/event audience policy with explicit failure-visibility semantics.
