## Summary

Keep open. Current main and v2026.5.18 still enqueue non-ok and thrown hook-agent results into prompt-visible system-event queues; the successful silent-hook fallback slice was fixed by a merged PR, but this remaining error/artifact audience policy is still unresolved and tied to open Gateway hook API work.

## What This Changes

The issue requests a way to stop hook delivery errors or delivery artifacts from being injected into agent sessions where agents may treat them as prompt-visible messages and produce noisy channel replies.

## Best Possible Solution

Define an explicit Gateway hook event-audience policy or API option that keeps machine-only hook artifacts out of model-visible prompts while preserving operator diagnostics, auth/trust metadata, idempotency, timeout/error semantics, and docs.

## Reproduction Assessment

Yes, by source inspection: a `/hooks/agent` run that returns `status: "error"` or `status: "skipped"`, or throws before completion, still reaches `enqueueSystemEvent`, and those queued entries are drained into the next prompt. I did not run a live Discord reproduction in this read-only review.

## Solution Assessment

Unclear. Suppressing prompt injection for machine-only hook artifacts is plausible, but current tests intentionally preserve non-ok announcements, so the maintainable fix needs a Gateway hook event-audience/API decision rather than a Discord-only outbound text filter.
