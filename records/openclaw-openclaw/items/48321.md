## Summary

Keep open. Current main still does not implement per-image sent-time, sender, or thread-position metadata for Slack thread history; prior file-bearing thread messages are retained as text/placeholders, the formatter ignores `historyMsg.files`, and model media attachments come only from the current message or thread starter. The directly related implementation PR #48725 is useful source context, but it was closed unmerged rather than shipped.

## What This Changes

The issue requests timestamp, sender, and thread-position metadata around images in channel thread history so models can reason about image freshness.

## Best Possible Solution

Implement this in the Slack plugin first: annotate image-bearing prior thread-history entries with sanitized sent time, sender, and message position/total when they are represented in model context, while preserving existing allowlist, bot-author filtering, and thread-history limits. Promote a shared SDK helper only after another channel needs the same contract.

## Reproduction Assessment

Yes. A code-level reproduction is clear: create a new Slack thread session where a prior history message has `files`; `resolveSlackThreadHistory` retains the files/placeholder, but `resolveSlackThreadContextData` emits only formatted text plus Slack message id and no per-image timestamp, sender, or position metadata.

## Solution Assessment

The requested direction is reasonable, but the best maintainable fix is Slack-owned first. Reusing the closed PR’s core-helper shape from extension production code would be weaker unless that helper is promoted through a documented plugin SDK seam.
