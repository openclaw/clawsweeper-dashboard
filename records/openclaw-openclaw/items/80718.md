## Summary

Keep open: the PR addresses a source-backed ACPX diagnostics gap, but latest head still redacts stderr per stream chunk and exits on child `exit` before piped stderr is guaranteed closed, so it can leak split secrets and lose the diagnostics it is trying to surface. The supplied proof also does not exercise the patched installed `sessions_spawn(runtime="acp", agentId="codex")` failure path.

## What This Changes

The PR adds per-lease Codex ACP wrapper stderr logging and diagnostic tails for generic ACP internal errors, inherits selected Codex runtime config into isolated ACP setup, and adds focused ACPX/acp-spawn tests.

## Best Possible Solution

Keep the PR open, fix stderr capture to redact across chunk boundaries and drain on `close`, then add redacted installed-path proof for the ACP spawn failure.

## Reproduction Assessment

Yes, at source level: current main has no retained Codex wrapper stderr tail and rethrows generic ACP `Internal error` failures without diagnostic context. I did not live-run the installed gateway path in this read-only review.

## Solution Assessment

No, not yet. Per-lease diagnostics are the right direction, but this implementation needs stream-aware redaction, `close`-based stderr draining, and installed-path proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Redact stderr across stream chunks:** `extensions/acpx/src/codex-auth-bridge.ts:338`
  - body: The wrapper redacts each stderr `data` chunk before appending it to the per-lease log. Because child stderr is a Readable stream, a token, bearer header, URL secret, or PEM block can cross chunk boundaries and be persisted before the regex sees the full secret; keep a small raw carry/buffer and only write redacted finalized text.
  - confidence: 0.9
- **[P2] Wait for the stderr pipe before exiting:** `extensions/acpx/src/codex-auth-bridge.ts:481-494`
  - body: This changes child stderr from inherited stdio to a pipe, but the wrapper still calls `process.exit` from the child `exit` handler. Node can emit `exit` while stdio streams are still open, so diagnostics written right before adapter shutdown can be lost from both the mirror and the per-lease log; preserve the exit code and finish from `close` after stderr drains.
  - confidence: 0.87
