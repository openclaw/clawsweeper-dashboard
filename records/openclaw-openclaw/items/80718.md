## Summary

Keep open: this is a plausible source-backed acpx diagnostic fix, but the latest patch still has blocking correctness and security issues, and the supplied proof does not show the patched OpenClaw sessions_spawn failure path surfacing a redacted stderr tail.

## What This Changes

The PR captures generated Codex ACP wrapper stderr, appends redacted tails to generic acpx internal errors, and adds focused acpx/acp-spawn tests.

## Best Possible Solution

Keep the diagnostic work in the acpx boundary, but store only bounded redacted stderr in a per-run/per-lease scope, align the spawn-order test with the canonical ACP metadata fix, and prove the patched sessions_spawn failure path with redacted terminal or log output.

## Reproduction Assessment

Source-reproducible. Current main still forwards generic acpx Internal error failures without a captured wrapper stderr tail, and the linked issue supplies a concrete OpenClaw/Codex sessions_spawn failure path, but I did not run a live gateway repro in this read-only review.

## Solution Assessment

No. The acpx wrapper/runtime boundary is the right place to preserve diagnostics, but this patch should not persist raw/shared stderr and should not add a test that locks in the stale ACP spawn ordering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Redact stderr before writing the wrapper log:** `extensions/acpx/src/codex-auth-bridge.ts:178`
  - body: The generated wrapper appends raw Codex stderr to codex-acp-wrapper.stderr.log before any redaction. Codex/acpx diagnostics can include tokens or credential-bearing request details, so a failed run can leave unredacted secrets in plugin state even though the surfaced AcpRuntimeError is redacted.
  - confidence: 0.84
- **[P2] Scope stderr logs to a single wrapper run:** `extensions/acpx/src/codex-auth-bridge.ts:166`
  - body: This fixed wrapper-root log is shared by every Codex ACP wrapper process, and each wrapper truncates then appends to it. Concurrent sessions can erase or interleave tails, causing one failed run to surface another run's stderr or lose its own diagnostic; use a per-lease or per-process log/buffer instead.
  - confidence: 0.78
- **[P2] Assert runtime initialization before session patch:** `src/agents/acp-spawn.test.ts:774-775`
  - body: The linked regression and the open fix at https://github.com/openclaw/openclaw/pull/80195 require ACP runtime initialization before sessions.patch so metadata is persisted before the first turn. This assertion requires the opposite ordering, which would block that fix or lock in the reported race.
  - confidence: 0.76
