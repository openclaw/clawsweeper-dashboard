## Summary

Keep open: the PR is aimed at a real ACPX diagnostics bug, but the patch needs changes before merge because it can surface unredacted stderr when log redaction is disabled and it uses one shared stderr log for all Codex wrapper processes.

## What This Changes

The PR updates the ACPX Codex wrapper to capture wrapper stderr and append a redacted stderr tail to generic Codex ACP internal errors, with focused runtime and ACP spawn ordering tests.

## Best Possible Solution

Keep the ACPX-owned diagnostics fix, but make stderr capture session-specific, force safety redaction for user-facing stderr tails, and add real sessions_spawn proof before merge.

## Reproduction Assessment

Yes, at source level: current main launches Codex through the shared ACPX wrapper and generic acpx Internal error failures lose the wrapper stderr, while the linked issue supplies real direct-acpx-works/sessions_spawn-fails evidence. I did not live-run the installed gateway reproduction in this read-only review.

## Solution Assessment

No, not yet. The PR is on the right owner boundary, but the implementation needs forced safety redaction and per-session stderr log ownership before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Force redaction for stderr diagnostics:** `extensions/acpx/src/runtime.ts:80`
  - body: The stderr tail is copied into `AcpRuntimeError` with `redactSensitiveText(...)`, which honors `logging.redactSensitive: "off"` and can return raw text. This new diagnostic is surfaced through ACP task/tool errors, so Codex adapter stderr containing tokens can leak; use a tool/safety redaction path that ignores the log-redaction opt-out before building the error.
  - confidence: 0.9
- **[P2] Isolate stderr logs per Codex session:** `extensions/acpx/src/codex-auth-bridge.ts:347`
  - body: Every generated Codex wrapper writes to and truncates the same `codex-acp-wrapper.stderr.log` under the shared wrapper root. Concurrent Codex ACP sessions can erase or interleave each other's stderr, causing an `Internal error` to show another run's diagnostic; include a lease/session-specific log path and read that same path for the failing handle.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete secret-exposure risk by placing raw adapter stderr into ACP errors through a redaction helper that can be disabled by config.

Concerns:

- **[high] Stderr diagnostics can bypass safety redaction:** `extensions/acpx/src/runtime.ts:80`
  - body: `readCodexWrapperStderrTail` uses `redactSensitiveText` without forcing tools/safety redaction, so `logging.redactSensitive: "off"` can leave tokens from Codex/acpx stderr in the AcpRuntimeError message shown to users or persisted in task output.
  - confidence: 0.9
- **[medium] Shared stderr log can cross-contaminate sessions:** `extensions/acpx/src/codex-auth-bridge.ts:347`
  - body: The fixed `codex-acp-wrapper.stderr.log` is shared by all Codex wrapper processes in the wrapper root, so concurrent sessions can expose another run's diagnostic text in the wrong error path even after best-effort redaction.
  - confidence: 0.82
