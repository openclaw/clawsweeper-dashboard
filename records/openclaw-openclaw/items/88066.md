## Summary

Keep open: current main still answers Telegram callbacks only after the sequentialized handler, so the latency problem is real, but this PR is not merge-ready because the new fast path can preempt plugin-owned callbacks, sends action-specific text before authorization, double-answers the same callback, and lacks exact-current-head Telegram proof.

## What This Changes

The PR adds a pre-sequential Telegram callback acknowledgement path for selected final reply-button payloads, sends a visible post-authorization acknowledgement, updates callback tests to use the shared env helper, and bumps one raw-fetch allowlist line.

## Best Possible Solution

Keep the fast acknowledgement direction, but make it plugin-ownership-aware, keep pre-auth callback toasts neutral or authorization-aware, suppress duplicate normal answers after fast-lane success, and require redacted current-head Telegram proof before merge.

## Reproduction Assessment

Yes from source inspection: current main installs Telegram sequentialization before the callback handler, so answerCallbackQuery cannot run until the per-chat queue releases. I did not establish a fresh live Telegram reproduction in this read-only review.

## Solution Assessment

No: the fast-ack approach is reasonable, but this implementation should preserve plugin callback ownership, avoid action-specific pre-auth text, and skip the normal answer after a successful fast-lane ack.

- **[P1] Defer final-button replies until plugins can claim callbacks:** `extensions/telegram/src/bot-handlers.runtime.ts:2147-2154`
