## Summary

Keep open. Current main still treats chat model directives as persistent session model changes, and the existing one-shot-like surfaces (`/btw`, cron `--model`, CLI one-shot provider runs) do not provide a normal chat reply on another model while preserving the session default and queued/background work behavior.

## What This Changes

Request a per-message chat model override that runs one response on a selected provider/model without changing the session default or retargeting future queued, heartbeat, or cron work.

## Best Possible Solution

Add a first-class one-turn model override for normal chat runs that reuses existing model resolution, allowlist, and auth-profile validation, carries the selected provider/model only through the current reply run, leaves `SessionEntry` model override fields untouched, and avoids refreshing queued followups or changing heartbeat and cron model resolution.

## Reproduction Assessment

Yes. A source-level reproduction is clear: an authorized mixed-content message like `/model openai/gpt-4o hello` is parsed, persisted through `persistInlineDirectives`, writes session model override fields, and later model selection reads those stored fields for future turns.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix, but it is not implemented on current main. A parser-only alias would be insufficient because the run path must bypass session persistence and queue retargeting while still using the existing validation contracts.
