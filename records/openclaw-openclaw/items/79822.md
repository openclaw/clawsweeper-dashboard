## Summary

Keep open. The PR addresses a real source-reproducible iMessage bug in the right channel-owned surface, but it is draft/protected, lacks after-fix real behavior proof, and the new guard misses standard media aliases so the silent-drop path remains for some attachment-bearing replies.

## What This Changes

The PR adds a runtime guard in the iMessage `reply` action that throws on selected attachment parameters and adds a regression test for four aliases.

## Best Possible Solution

Keep the channel-owned runtime rejection, but make it cover the full supported media-source alias set before dispatch and add matching regression coverage.

## Reproduction Assessment

Yes, from source inspection rather than a live run: current main's iMessage `reply` path reads text and calls text-only `sendRichMessage`, while attachment delivery is a separate `upload-file`/`sendAttachment` path. That gives a high-confidence failing path for attachment params being accepted but ignored.

## Solution Assessment

No, not as written. Rejecting unsupported attachments at the iMessage action boundary is the narrow maintainable solution, but the guard should cover all standard media aliases such as `path` and `fileUrl` before it can fully solve the silent-drop behavior.

## Review Findings
