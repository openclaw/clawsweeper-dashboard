## Summary

Keep open: source inspection confirms the bug and the reject-at-boundary direction, but this draft protected-label PR still misses shared media aliases and its live proof does not exercise the new rejection path.

## What This Changes

The PR adds a runtime guard in the native iMessage `reply` action to reject selected attachment fields and tests those aliases.

## Best Possible Solution

Keep the iMessage-owned runtime rejection before `sendRichMessage`, cover the complete shared media-source alias set plus `buffer`, and require redacted live proof of the rejection path before normal maintainer review.

## Reproduction Assessment

Yes. Current main's iMessage `reply` path reads only text and calls the text-only `sendRichMessage`/`imsg send-rich --reply-to` path, while attachment delivery is handled separately by `upload-file`/`sendAttachment`.

## Solution Assessment

No, not as written. Runtime rejection is the narrow maintainable fix, but it must use the full shared media-source alias set before it fully prevents silent drops.

## Review Findings

Overall correctness: patch is incorrect
