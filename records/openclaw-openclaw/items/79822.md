## Summary

Keep open: the owner-module fix direction is well supported by current source and the imsg contract, but the PR still needs a maintainer pass because the posted live proof does not exercise the new reply-with-attachment rejection and there is one low-priority edge case in the guard ordering.

## What This Changes

The PR adds an iMessage reply guard and regression test so attachment-bearing reply params throw an explicit error, plus a changelog entry.

## Best Possible Solution

Land the owner-module rejection after the guard covers media-only reply attempts consistently and a redacted real trace shows `reply` with an attachment returning the new explicit error; keep the true combined reply+file primitive tracked in https://github.com/openclaw/imsg/issues/113.

## Reproduction Assessment

Yes, at source level: current main routes `reply` through `sendRichMessage` after reading only text/message, while the runtime builds `imsg send-rich --reply-to` with no attachment argument. A unit harness passing `reply` with text plus media would reproduce the silent-drop path without needing a live Mac.

## Solution Assessment

Yes, rejecting unsupported iMessage reply attachments in the iMessage action adapter is the narrowest maintainable fix until imsg supports a combined reply+file primitive. The safer tweak is to run the attachment guard before the text-required check so media-only attempts receive the same actionable guidance.
