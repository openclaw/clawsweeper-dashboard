## Summary

Keep open. Current `main` still has the reported event gap: command-side goal mutations persist session metadata, but only gateway session RPC paths emit `sessions.changed`; the related `/name` PR at https://github.com/openclaw/openclaw/pull/88581 explicitly tracks this as follow-up work rather than fixing it.

## What This Changes

The issue asks OpenClaw to emit `sessions.changed` when in-chat store-mutating commands such as `/goal` and the proposed `/name` update session metadata.

## Best Possible Solution

Add a narrow core session-metadata mutation signal from command handling to the gateway context and reuse the existing `sessions.changed` payload semantics for command-driven metadata writes.

## Reproduction Assessment

Yes, from source inspection: `/goal pause`, `/goal complete`, `/goal clear`, and related mutations write the session store without a gateway event broadcaster in the command parameters. I did not run a live websocket repro because this review was read-only.

## Solution Assessment

Yes, the requested event parity is the right repair direction because `sessions.changed` is already the documented metadata-change event and `sessions.patch` already emits the desired row shape. The implementation should share the existing gateway emitter rather than adding a second payload builder in auto-reply code.
