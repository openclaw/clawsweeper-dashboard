## Summary

This should stay open: current Slack routing still ignores channel-name keys by default, while the config schema accepts arbitrary `channels.slack.channels` keys and the existing diagnostics do not specifically warn about route keys that will not match.

## What This Changes

The issue asks for a Slack diagnostic when `channels.slack.channels` contains name-keyed entries under `groupPolicy: "allowlist"` so they are not silently ignored at routing time.

## Best Possible Solution

Add a Slack-owned startup diagnostic for non-ID route keys that remain unresolved or would be ignored under allowlist, preserving ID-first routing semantics and avoiding a new config mode.

## Reproduction Assessment

Yes, source-reproducible: the focused routing test and `resolveSlackChannelConfig` path show name-keyed channel entries are blocked by default, while startup diagnostics do not emit the requested actionable warning. I did not run a live Slack workspace in this read-only review.

## Solution Assessment

Yes, with a narrower scope: a warning for non-ID keys that will remain ignored is the maintainable fix; it should not enable name matching, reject config, or change channel routing semantics.
