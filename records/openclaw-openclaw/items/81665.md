## Summary

Keep open: current main still has the Slack allowlist diagnostic gap. The code documents ID-keyed Slack routes and blocks unresolved name-keyed routes by default, but startup only emits a generic unresolved mapping and `openclaw doctor` does not inspect route keys; the only linked fix PR was closed unmerged.

## What This Changes

Request a Slack diagnostic for name-keyed `channels.slack.channels` entries under `groupPolicy: "allowlist"` so operators are warned that unsupported names will not route.

## Best Possible Solution

Add targeted Slack startup and doctor diagnostics for non-ID route keys under allowlist while preserving ID-first routing and dangerous name-matching semantics.

## Reproduction Assessment

Yes, source-reproducible: schema accepts arbitrary Slack route keys, while the resolver denies name matches when `allowNameMatching` is false and doctor does not inspect route keys. I did not run a live Slack workspace in this read-only review.

## Solution Assessment

Yes: a targeted startup and doctor warning is the narrow maintainable fix for the silent unsupported-config shape. Enabling name-keyed routing or rejecting config would change behavior and need a separate product decision.
