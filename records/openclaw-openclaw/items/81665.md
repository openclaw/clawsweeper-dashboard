## Summary

This should stay open: current main accepts arbitrary Slack channel route keys and default routing is ID-first, so name-like entries can be ignored without an actionable startup warning; no duplicate or shipped fix was found.

## What This Changes

The issue asks for an actionable Slack diagnostic when `channels.slack.channels` contains name-keyed entries under `groupPolicy: "allowlist"` instead of stable channel IDs.

## Best Possible Solution

Add a Slack-owned startup or doctor diagnostic for non-ID route keys that remain unresolved or would be ignored under allowlist, while preserving ID-first routing semantics.

## Reproduction Assessment

Yes, source-reproducible: the focused routing test and `resolveSlackChannelConfig` path show name-keyed channel entries are blocked by default, while startup diagnostics do not emit the requested actionable warning. I did not run a live Slack workspace in this read-only review.

## Solution Assessment

Yes, with a narrow scope: an actionable warning for non-ID keys that will remain ignored is the maintainable fix; it should not enable name matching, reject config, or change routing semantics.
