Keep open: current main still has the silent Slack allowlist diagnostic gap, and the exact linked fix PR was closed without merging. Source, docs, tests, and live PR context all point to a narrow Slack startup/doctor warning as the remaining work.

## What This Changes

Report that Slack `groupPolicy: "allowlist"` accepts `channels.slack.channels` entries keyed by channel name even though they never match ID-first routing and produce no operator warning.

## Best Possible Solution

Add focused Slack startup and `openclaw doctor` warnings for non-ID route keys under allowlist while preserving ID-first routing and the existing dangerous name-matching override semantics.

## Reproduction Assessment

Yes, source-reproducible: config accepts arbitrary Slack route keys, docs require IDs, and the resolver/policy path blocks name-keyed routes under allowlist unless dangerous name matching is enabled. I did not run a live Slack workspace in this read-only review.

## Solution Assessment

Yes: a startup and doctor diagnostic is the narrow maintainable fix for the silent unsupported-config shape. Enabling name routing by default or adding a strict-mode flag would be a larger product change and is not needed for this issue.
