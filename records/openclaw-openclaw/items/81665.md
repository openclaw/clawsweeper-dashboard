## Summary

This should stay open: current main still accepts arbitrary Slack channel route keys and blocks name-keyed allowlist routes by default without the requested actionable startup/doctor diagnostic. The issue is already paired with the open closing PR https://github.com/openclaw/openclaw/pull/81688, so the right path is to review and land that PR rather than close this issue early.

## What This Changes

Request a Slack diagnostic when `channels.slack.channels` allowlist entries are keyed by channel name instead of Slack channel ID.

## Best Possible Solution

Review and land the focused Slack diagnostic in https://github.com/openclaw/openclaw/pull/81688 once merge gates are green, preserving ID-first routing and warning only unresolved non-ID allowlist route keys.

## Reproduction Assessment

Yes, source-reproducible: the schema accepts name-like keys, the resolver excludes names unless dangerous name matching is enabled, and the focused test shows name-keyed routes are blocked by default. I did not run a live Slack workspace in this read-only review.

## Solution Assessment

Yes, with scope narrowed to diagnostics: warn at startup and through doctor for unresolved non-ID route keys that will not match. The fix should not enable name routing, reject config, or add a strict-mode setting as part of this issue.
