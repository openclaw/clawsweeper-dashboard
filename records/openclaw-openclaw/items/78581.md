## Summary

Keep open: current main contains the landed classifier/test partial fix, and the give-up path now has failed-status plumbing, but announce retry still uses fixed backoff and does not use the structured provider cooldown expiry that already exists in model fallback errors.

## What This Changes

Treat sub-agent announce FallbackSummaryError provider-cooldown failures as retryable, wait until cooldown expiry when available, log retries, and keep task delivery status accurate after exhaustion.

## Best Possible Solution

Finish the narrow cooldown-aware retry path by propagating structured cooldown timing to announce delivery and proving exhausted provider-cooldown announces leave task status non-delivered.

## Reproduction Assessment

Yes, by source inspection for the remaining gap: current main uses fixed 5/10/20 second direct announce retry delays and does not consume `FallbackSummaryError.soonestCooldownExpiry` or gateway `retryAfterMs`. I did not run a live provider-overload reproduction.

## Solution Assessment

Yes for the already-landed classifier portion, but it is incomplete. The maintainable fix is to keep the text classifier, add bounded cooldown-aware retry timing through existing structured error metadata, and add regression coverage for final task delivery status.
