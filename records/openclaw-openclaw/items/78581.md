## Summary

Keep open: current main still has a source-backed retry-classification gap for provider-cooldown fallback-summary errors, and the docs promise push-based sub-agent completion delivery with retry before give-up.

## What This Changes

The issue asks OpenClaw to retry sub-agent announce wake delivery when announce-summary fails with provider cooldown FallbackSummaryError messages, and to avoid reporting failed wake delivery as delivered.

## Best Possible Solution

Add a narrow regression-backed fix in the sub-agent announce delivery path that treats provider cooldown fallback-summary failures as transient, uses cooldown expiry when available with a bounded cap, and proves task delivery status reflects the final announce outcome.

## Reproduction Assessment

Yes for the primary classifier path: current-main source plus a direct pattern check show the reported FallbackSummaryError cooldown message is not classified as transient, so the retry loop would rethrow before retrying. No full live provider-cooldown run was performed.

## Solution Assessment

The one-line pattern addition is a useful partial fix, but the best maintainable solution is a structured transient classifier with text fallback, cooldown-aware bounded delay, and regression coverage for both retry logging and final delivery status.
