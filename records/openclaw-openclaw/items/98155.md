## Summary

Keep open: the PR addresses a real browser snapshot text bug, but the current patch is not merge-ready because it changes the truncation budget from UTF-16 code units to code points and still lacks real proof from a changed browser path.

## What This Changes

The PR replaces raw `.slice(0, N)` truncation in three Browser plugin snapshot paths with `Array.from(...).slice(...).join("")` and adds a colocated truncation test file.

## Best Possible Solution

Keep the narrow bug fix, but mirror `truncateUtf16Safe` semantics inside the evaluated browser scripts so the old code-unit caps remain intact while dangling surrogate halves are avoided.

## Reproduction Assessment

Yes from source inspection: current main uses raw `.slice(0, 100)` and `.slice(0, 120)` on browser snapshot labels, so an astral character at the cut boundary can leave a dangling surrogate half. I did not run a live browser snapshot reproduction in this read-only review.

## Solution Assessment

No. The direction is right, but the implementation should preserve the existing UTF-16 code-unit cap instead of changing it to a code-point count.
