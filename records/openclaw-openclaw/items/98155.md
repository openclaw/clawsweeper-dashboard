## Summary

Keep open: the bug is real, but this PR is not merge-ready because the fix changes existing 100/120 UTF-16 code-unit caps into 100/120 code-point caps, and the proof still does not exercise a changed Browser snapshot path.

## What This Changes

This PR replaces three Browser plugin snapshot/element-label raw `.slice(...)` truncations with code-point slicing and adds colocated tests for that expression.

## Best Possible Solution

Keep the narrow bug fix, but mirror `truncateUtf16Safe` semantics inside the evaluated browser scripts so the old 100/120 UTF-16 code-unit caps remain intact while dangling surrogate halves are avoided.

## Reproduction Assessment

Yes from source inspection: current main uses raw `.slice(0, 100)` and `.slice(0, 120)` on Browser snapshot labels, so an astral character at the cut boundary can leave a dangling surrogate half. I did not run a live Browser snapshot reproduction in this read-only review.

## Solution Assessment

No. The target bug is narrow and valid, but the implementation should preserve the existing UTF-16 code-unit cap instead of changing truncation to a code-point count.
