## Summary

Keep open: the PR targets a real doctor/plugin auto-enable regression, but the proposed guard is not safe to merge as-is because it treats a raw `plugins.entries.<id>.enabled=true` flag as an active alternative without honoring restrictive `plugins.allow`. Current main has later `preferOver` and channel-presence work, but that does not fully cover the PR's no-`preferOver` `openclaw-lark` scenario, so a corrected current-main replacement is still useful.

## What This Changes

The PR adds a plugin auto-enable guard and regression test so `doctor --fix` does not auto-enable bundled `feishu` when an enabled alternative plugin such as `openclaw-lark` claims the same Feishu channel.

## Best Possible Solution

Port the useful behavior to current main by suppressing bundled Feishu auto-enable only when another plugin that claims the same channel is effectively active and trusted under the same activation, deny, disable, and allowlist rules the loader uses. Preserve bundled fallback when the alternative is denied, disabled, not allowlisted, or untrusted, and add focused regression tests for both paths.

## Reproduction Assessment

Yes. The linked bug and PR test provide a concrete `channels.feishu` plus `plugins.entries.openclaw-lark.enabled=true` setup; the PR-specific regression is reproducible by adding a restrictive `plugins.allow` that excludes `openclaw-lark`, which makes the alternative inactive while the new guard still skips bundled Feishu.

## Solution Assessment

No. The direction is right, but the PR should not use a raw enabled-entry check as the source of truth; it should reuse effective activation/channel ownership policy and be adapted to the current split auto-enable files.
