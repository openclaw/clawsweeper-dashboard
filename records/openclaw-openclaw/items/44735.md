## Summary

Keep open: the Feishu/Lark auto-enable problem is still source-reproducible on current main, but this PR is stale and not correct as-is because it checks only the raw enabled flag instead of effective activation under `plugins.allow`.

## What This Changes

The PR adds an auto-enable guard and regression test so bundled `feishu` is not enabled when another enabled plugin declares the Feishu channel.

## Best Possible Solution

Implement generic effective channel-owner detection in the current auto-enable path, sharing the allowlist-aware semantics already used by doctor install repair, then add Feishu/Lark regression coverage and a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `channels.feishu` still resolves to the built-in `feishu` candidate unless `preferOver` metadata is present, and materialization then enables that channel without consulting enabled alternatives.

## Solution Assessment

No. The PR direction is right, but the implementation should be refreshed to the current split files and use effective activation/allowlist semantics rather than raw `enabled` flags.
