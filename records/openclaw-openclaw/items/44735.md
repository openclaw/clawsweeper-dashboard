## Summary

Keep open: the PR targets a real Feishu/Lark doctor auto-enable conflict and includes a useful regression test, but the submitted patch is not merge-ready. Its new alternative-plugin guard treats `plugins.entries.<id>.enabled: true` as active without checking `plugins.allow`, and the branch edits the pre-refactor monolithic auto-enable file while current `main` has moved this logic into shared/apply modules.

## What This Changes

The PR adds a plugin auto-enable guard and regression test so `doctor --fix` does not auto-enable bundled `feishu` when another enabled plugin declares the same Feishu channel.

## Best Possible Solution

Port the useful Feishu alternative-plugin protection into the current shared auto-enable flow, using existing activation/allowlist semantics so an alternative only suppresses bundled Feishu when it can actually load, and cover both unrestricted and restrictive-allowlist configs with focused tests plus a changelog entry.

## Reproduction Assessment

Yes. The PR's regression test gives a clear reproduction for `channels.feishu` plus `plugins.entries.openclaw-lark.enabled=true`, and source inspection shows the proposed patch has a second reproducible failure when `plugins.allow` excludes that alternative.

## Solution Assessment

No. The direction is good, but the submitted implementation is not the best fix because it checks only the raw enabled flag and targets an old file layout; the safer solution is to use current activation semantics in `plugin-auto-enable.shared.ts`.
