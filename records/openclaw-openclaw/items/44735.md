The PR adds a plugin auto-enable guard and regression test so a plugin is skipped when another enabled registry plugin declares the same channel.

## Best Possible Solution

Rebase onto the split auto-enable modules and implement an effective channel-owner check that respects allowlist, deny, explicit disable, and `preferOver`, with regressions for enabled alternatives and restrictive allowlist fallback.

Yes. Source inspection gives a high-confidence path: with restrictive `plugins.allow`, `plugins.entries.openclaw-lark.enabled=true` is not enough to activate that plugin, but the PR guard still skips Feishu auto-enable.

## Solution Assessment

No. The direction is useful, but the submitted solution should check effective channel ownership rather than raw `entries.enabled`, and it must be adapted to current main's split auto-enable files.
