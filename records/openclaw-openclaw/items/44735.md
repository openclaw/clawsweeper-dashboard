## Summary

Keep open, but this patch is not ready to merge: the new guard can suppress the Feishu fallback even when the supposed alternative is disabled by the effective allowlist, and the branch now conflicts with the refactored auto-enable implementation on current main.

## What This Changes

The PR adds a plugin auto-enable guard and regression test so an enabled plugin that declares the same channel prevents auto-enabling another plugin for that channel.

## Best Possible Solution

Rebase the fix onto `src/config/plugin-auto-enable.shared.ts` and reuse the effective channel-owner/activation policy so replacement plugins are honored only when they are actually allowed and active.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `channels.feishu` plus `plugins.entries.openclaw-lark.enabled=true` reaches the new guard, and adding a restrictive `plugins.allow` that omits `openclaw-lark` makes the guard skip `feishu` even though the alternative is not effectively enabled.

## Solution Assessment

No. The current PR is not the best implementation because it adds a local raw-entry check instead of using the current effective channel-owner and allowlist policy, and it needs to be rebased onto the refactored auto-enable files.

- **[P2] Require effective activation before skipping auto-enable:** `src/config/plugin-auto-enable.ts:485`
