## Summary

Keep open: the PR addresses a narrow, still-plausible Feishu/openclaw-lark auto-enable conflict, but the submitted patch is not merge-ready. Current main has split the old auto-enable implementation into shared/apply/test files, and the branch guard treats an entry flag as active without checking effective activation under restrictive `plugins.allow`.

## What This Changes

The PR adds a plugin auto-enable guard and regression test so configured Feishu does not auto-enable the bundled `feishu` plugin when `openclaw-lark` already claims the Feishu channel.

## Best Possible Solution

Implement a shared auto-enable materialization guard that suppresses a bundled channel only when another plugin claiming the same normalized channel is already effective, or will become effective in the same repair pass, while preserving bundled fallback when the alternative is blocked. Cover Feishu/openclaw-lark plus restrictive allowlist, denylist, and explicit-disable cases, and add the required user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: configured `channels.feishu` with both `feishu` and `openclaw-lark` manifest claims but no `preferOver` still produces the bundled `feishu` candidate on current main, and the materialization loop has no same-channel effective-owner guard before enabling it.

## Solution Assessment

No, not as submitted. The direction is useful, but the guard must be ported into the current shared implementation and must use effective activation semantics instead of treating `plugins.entries.<id>.enabled === true` as sufficient.
