## Summary

Keep open. The PR targets a real doctor/plugin auto-enable conflict, but the submitted guard is not merge-ready: it treats raw `plugins.entries.<id>.enabled=true` as effective ownership, conflicts with current main's split auto-enable files, and still lacks real behavior proof.

## What This Changes

The PR adds a doctor/plugin auto-enable guard and regression test so built-in Feishu is not auto-enabled when an enabled alternative plugin declares the Feishu channel.

## Best Possible Solution

Rebase onto current main's split auto-enable modules and implement a generic effective channel-owner check that respects allowlist, deny, explicit disable, and `preferOver`, with focused regressions for enabled alternatives and restrictive allowlist fallback.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: under restrictive `plugins.allow`, `plugins.entries.openclaw-lark.enabled=true` is not enough to make that plugin effective, but the PR guard still skips Feishu auto-enable.

## Solution Assessment

No. The direction is reasonable, but the submitted solution should check effective ownership instead of raw `entries.enabled`, and it must be adapted to current main's split auto-enable files.
