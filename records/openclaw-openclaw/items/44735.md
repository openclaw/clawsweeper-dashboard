## Summary

Keep open. The PR targets a real Feishu/openclaw-lark doctor conflict, but the submitted guard checks raw `plugins.entries.<id>.enabled` instead of effective activation, so it can suppress the Feishu fallback when the alternative is not actually allowed to load.

## What This Changes

The branch adds a plugin auto-enable guard and regression test to avoid enabling built-in Feishu when another enabled plugin declares the same channel.

## Best Possible Solution

Rebase onto the split auto-enable modules and implement a generic effective channel-owner check that respects allowlist, deny, explicit disable, and `preferOver`, with regressions for enabled alternatives and restrictive allowlist fallback.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: under restrictive `plugins.allow`, `plugins.entries.openclaw-lark.enabled=true` is not enough to make that plugin effective, but the PR guard still skips Feishu auto-enable.

## Solution Assessment

No. The direction is reasonable, but the submitted solution should check effective channel ownership rather than raw `entries.enabled`, and it must be adapted to current main's split auto-enable files.
