## Summary

Keep open, but do not merge as-is. The PR is aimed at a real Feishu/openclaw-lark auto-enable gap that is not fully covered by current main, but the submitted guard treats a configured plugin as effective even when policy would block it, the branch is stale against the split auto-enable files, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds a plugin auto-enable guard and regression test so Feishu is not auto-enabled when another enabled plugin declares the Feishu channel.

## Best Possible Solution

Rebase onto the split auto-enable modules and implement an effective channel-owner check that respects allowlist, deny, explicit disable, and `preferOver`, with regressions for enabled alternatives and restrictive allowlist fallback.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `plugins.entries.openclaw-lark.enabled=true` is not enough to make that plugin effective under a restrictive `plugins.allow`, but the PR guard still skips Feishu auto-enable; current main also lacks a generic enabled-alternative guard when same-channel claims have no `preferOver`.

## Solution Assessment

No. The direction is right, but the submitted solution should check effective ownership instead of raw `entries.enabled`, and it must be adapted to current main's split auto-enable files.
