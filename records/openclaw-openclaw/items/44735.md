## Summary

Keep open. The PR targets a real Feishu/openclaw-lark doctor auto-enable conflict, but the submitted branch is not merge-ready because it trusts `plugins.entries.<id>.enabled=true` without checking whether that plugin is actually effective under `plugins.allow`, and it lacks after-fix real behavior proof.

## What This Changes

The PR adds a plugin auto-enable guard that skips a candidate when another enabled plugin declares the same channel, plus Feishu/openclaw-lark regression coverage.

## Best Possible Solution

Rebase onto the current split auto-enable code and implement a generic effective channel-owner check that reuses the existing activation/allowlist and `preferOver` policy, with regression tests for both enabled alternatives and restrictive allowlist fallback.

## Reproduction Assessment

Yes. Source inspection of the PR shows a high-confidence failing case: with `plugins.allow` excluding `openclaw-lark`, `plugins.entries.openclaw-lark.enabled=true` is not enough to make that plugin effective, but the PR still skips Feishu auto-enable.

## Solution Assessment

No. The direction is right, but the submitted implementation should check effective channel ownership rather than raw `entries.enabled`, and it needs to be adapted to current main's refactored auto-enable files.
