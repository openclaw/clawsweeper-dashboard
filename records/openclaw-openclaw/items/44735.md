## Summary

Keep open for contributor/maintainer follow-up: the PR addresses a real Feishu/openclaw-lark doctor conflict, but the submitted guard treats raw `plugins.entries.<id>.enabled=true` as active ownership and can skip Feishu even when the alternative is blocked by `plugins.allow`. Current main has related channel-owner and install-repair work, but it does not make this stale branch mergeable as-is.

## What This Changes

The PR adds a plugin auto-enable guard that skips enabling a plugin when another enabled registry plugin declares one of the same channels, plus a Feishu/openclaw-lark regression test.

## Best Possible Solution

Rebase onto the split auto-enable modules and gate alternative-channel skips on effective channel ownership, including allowlist, deny, explicit disable, and `preferOver`, with regressions for both enabled alternatives and restrictive-allowlist fallback.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with restrictive `plugins.allow`, `plugins.entries.openclaw-lark.enabled=true` is not enough to activate that plugin, but the PR guard still skips Feishu auto-enable.

## Solution Assessment

No. The direction is useful, but the submitted solution should check effective channel ownership rather than raw `entries.enabled`, and it needs to be adapted to current main's split auto-enable files.
