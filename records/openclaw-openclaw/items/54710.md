## Summary

Keep open. Current main shows a shared Discord-to-reply-to-exec path and generic env pass-through, plus local shell behavior preserves a value containing `##`; however there is still no Discord-routed regression, live proof, or source change that specifically explains or fixes the reported channel-specific mutation.

## What This Changes

The issue reports that Discord-bound exec commands mangle container environment values containing `##` when referenced as `${VAR}`, while `printenv` and other channel contexts return the raw value.

## Best Possible Solution

Add focused proof for the Discord-bound runtime path. If a live or regression test shows `${SOME_PASSWORD}` and `printenv SOME_PASSWORD` both preserve `pw_pw##` from a Discord-routed exec run, close as cannot-reproduce with that evidence; if it fails, fix the specific mutation layer without changing unrelated channel or exec semantics.
