## Summary

Keep open: this is a plausible draft implementation over the current durable delivery API, but it adds a new script-visible CLI JSON contract and the external PR currently lacks after-fix real behavior proof.

## What This Changes

Adds a documented `deliveryStatus` object to `deliverAgentCommandResult` and `openclaw agent --json --deliver`, mapping `sendDurableMessageBatch` outcomes into CLI JSON/return status with tests and a changelog entry.

## Best Possible Solution

Expose one additive CLI JSON status projection from `sendDurableMessageBatch`, keep the field semantics tightly documented, and require redacted real CLI/channel output before merge.

## Reproduction Assessment

Source-reproducible: current main writes the `--json` envelope before delivery and returns only payloads/meta, so it cannot include durable delivery status. I did not run a live channel delivery in this read-only review.

## Solution Assessment

Mostly yes in direction: projecting the canonical `sendDurableMessageBatch` result is cleaner than extending deprecated delivery-result semantics. The remaining merge blockers are maintainer approval of the public JSON shape, proof from a real setup, and a small docs precision fix.
