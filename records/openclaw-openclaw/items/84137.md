## What This Changes

The issue asks whether post-tool raw assistant completion should keep the current fail-fast default with operator opt-in, or get a longer/default post-tool synthesis or terminal-state contract.

## Best Possible Solution

Keep the compatible fail-fast default until maintainers accept a semantic change backed by real workload evidence; if they approve a new default, add targeted regressions for pre-tool release, post-tool budget, and stuck-turn latency while keeping the config-knob PR separate.

## Solution Assessment

Unclear until maintainers choose the desired default. The compatible config path in https://github.com/openclaw/openclaw/pull/84974 is the conservative short-term option, while a semantic default change needs real workload proof and explicit acceptance of the stuck-turn latency tradeoff.
