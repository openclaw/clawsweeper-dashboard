## Summary

Keep open: current main still uses the short assistant idle budget for post-tool raw assistant completion, and the linked open PR only adds a compatible config knob; changing the default semantics needs maintainer agreement and workload proof.

## What This Changes

The issue asks whether Codex app-server post-tool raw assistant completion should keep the current short fail-fast default or get a longer default post-tool synthesis budget.

## Best Possible Solution

Keep the compatible fail-fast default unless maintainers approve a semantic change backed by real Codex app-server workload evidence, then update the default and regression coverage together.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a semantic/defaults decision. Source and tests give high-confidence proof of the current behavior: post-tool raw assistant progress uses the short assistant idle budget, while pre-tool raw assistant output can release final text.

## Solution Assessment

Unclear until maintainers choose the desired default. The linked config-knob PR is the conservative short-term path, while a default semantic change needs workload proof and explicit agreement on stuck-turn latency tradeoffs.
