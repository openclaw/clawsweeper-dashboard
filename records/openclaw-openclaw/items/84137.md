## What This Changes

The issue asks whether Codex app-server post-tool raw assistant completion should keep the current short fail-fast default or move to a longer/distinct post-tool synthesis or terminal-wait state.

## Best Possible Solution

Keep the compatible default and opt-in config path unless maintainers approve a default semantic change backed by real Codex app-server workload evidence, then add an explicit post-tool synthesis or terminal-wait contract with focused regression tests and docs.

## Reproduction Assessment

Yes for the current behavior, not for a failing bug path. Source, docs, and focused tests show that current main treats pre-tool raw assistant completion as releasable output while post-tool raw assistant progress uses the short assistant idle budget and waits for terminal completion.

## Solution Assessment

Unclear until maintainers choose the intended default semantics. The conservative opt-in config path in https://github.com/openclaw/openclaw/pull/84974 is narrow and compatible, but changing the shipped default needs workload proof and agreement on stuck-turn latency tradeoffs.
