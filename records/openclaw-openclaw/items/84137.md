## Summary

Keep open: current main still treats post-tool raw assistant completion as a short fail-fast terminal wait, and changing that default needs maintainer product agreement plus real workload evidence rather than an automated cleanup close.

## What This Changes

Evaluate whether Codex app-server post-tool raw assistant completion should keep the current short fail-fast budget or receive a distinct longer default post-tool synthesis budget.

## Best Possible Solution

Keep the compatible fail-fast default unless maintainers approve a longer post-tool default backed by real Codex app-server workload evidence, then update code, docs, and regression tests for both pre-tool release and post-tool waiting.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a semantic/defaults decision. Source and tests give high-confidence proof of the current behavior: post-tool raw assistant progress uses the short assistant idle budget, while pre-tool raw assistant output can release final text.

## Solution Assessment

Unclear until maintainers choose the desired default. The related compatible config PR is the conservative short-term path, while a default semantic change needs workload proof and explicit agreement on the stuck-turn latency tradeoff.
