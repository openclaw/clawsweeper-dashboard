## Summary

Keep open: current main still ties post-tool raw assistant completion to the short assistant idle guard, and the related open PR only adds an opt-in config path; changing the shipped default semantics still needs maintainer agreement and real workload proof.

## What This Changes

Evaluate whether Codex app-server should keep fail-fast post-tool raw assistant completion semantics or give post-tool synthesis a longer default budget.

## Best Possible Solution

Keep the compatible fail-fast default unless maintainers approve a semantic change backed by real Codex app-server workload evidence, then update the default budget with focused regression coverage for pre-tool release, post-tool waiting, and stuck-turn latency.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a default-semantics decision. Source and tests give high-confidence proof of the current behavior: post-tool raw assistant progress uses the short assistant idle budget, while pre-tool raw assistant output can release final text.

## Solution Assessment

Unclear until maintainers choose the desired default. The related config-knob PR is the conservative short-term path, while a default semantic change needs workload proof and explicit agreement on stuck-turn latency tradeoffs.
