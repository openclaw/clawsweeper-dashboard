## Summary

Keep open. The PR addresses a real gateway whitespace bug and has useful proof, but its branch targets an older inline final-reply assembly path; current main now routes final display/transcript text through helper functions that still trim split payload text.

## What This Changes

The branch adds a non-streaming gateway reply-part combiner, applies it to final and `/btw` replies, and adds indentation, single-newline, and CRLF regression coverage.

## Best Possible Solution

Land a rebased gateway fix that centralizes non-streaming text recombination across `/btw`, assistant display, and transcript text while leaving streaming, Markdown parsing, media handling, and channel-specific formatting unchanged.

## Reproduction Assessment

Yes. Source inspection on current main shows split non-streaming final and `/btw` payload text still flows through `.trim()` before blank-line joins, so a next chunk that starts with indentation loses that indentation.

## Solution Assessment

No, not as the branch currently stands. The shared combiner is the right narrow solution, but it must be rebased into the current helper-based final display/transcript paths as well as the current `/btw` assembly.
