## Label Justifications

- `P2`: This is a normal-priority behavior bug in agent lifecycle observability with a clear narrow surface and limited blast radius.
- `impact:session-state`: The missing terminal lifecycle metadata can make subscribers and persisted session/run state misclassify cancellation as completion.

## Summary

Keep open. Current main and the latest stable release still emit `lifecycle:end` from the pi-embedded subscribe layer without an explicit `aborted` field, while downstream lifecycle consumers already distinguish `aborted` and `stopReason`.

## What This Changes

The issue asks for pi-embedded `lifecycle:end` events to include `aborted` and `stopReason` so subscribers can distinguish cancellation from natural completion.

## Best Possible Solution

Propagate the runner's explicit abort boolean through the existing terminal lifecycle metadata contract and emit it with `stopReason` on both lifecycle event sinks, with focused coverage for cancelled and naturally completed runs.

## Reproduction Assessment

Yes by source inspection: an aborted pi-embedded attempt carries an `aborted` result, but the subscribe-layer terminal metadata state and `lifecycle:end` emitter cannot carry that boolean. I did not run a live abort scenario in this read-only sweep.

## Solution Assessment

Yes. The narrow maintainable fix is to extend the existing terminal lifecycle metadata path rather than relying on the guarded `agent-command.ts` fallback or re-inferring cancellation from `stopReason` in the emitter.
