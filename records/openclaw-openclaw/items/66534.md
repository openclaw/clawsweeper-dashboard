## Summary

Keep open. Current main partially addresses the report by forwarding `stopReason`, but the primary pi-embedded terminal lifecycle event still has no explicit `aborted` boolean, and the related fix PR #66574 is closed unmerged.

## What This Changes

The issue requests that the pi-embedded `lifecycle:end` event include `aborted` and `stopReason` so event subscribers can distinguish cancellation from natural completion.

## Best Possible Solution

Extend the typed terminal lifecycle metadata path to carry the runner’s explicit `aborted` value into both global and direct terminal lifecycle events, while preserving the existing stopReason, yielded, liveness, and replay metadata behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: the primary subscribe-layer terminal event can include `stopReason`, but there is no state field, setter parameter, or emitted payload property for `aborted`.

## Solution Assessment

Yes. The narrow maintainable fix is to propagate the explicit runner attempt metadata; deriving abort state heuristically from `stopReason` would be less reliable.
