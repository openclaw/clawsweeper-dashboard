## Summary

Keep open. Current main and the latest release still propagate `stopReason` on the pi-embedded terminal path but do not carry an explicit `aborted` boolean through the lifecycle metadata state, setter, or emitted `lifecycle:end` event.

## What This Changes

Ensure pi-embedded `lifecycle:end` events include `aborted` and `stopReason` so subscribers can distinguish cancellation from normal completion.

## Best Possible Solution

Propagate the runner's explicit abort result through the existing terminal lifecycle metadata contract and emit it with `stopReason` from the primary pi-embedded lifecycle path.

## Reproduction Assessment

Yes. Source inspection on current main shows the runner has `meta.aborted`, but the terminal lifecycle setter and emitted `lifecycle:end` payload cannot carry it; a live abort run was not executed in this read-only sweep.

## Solution Assessment

Yes. The narrow maintainable fix is to add an explicit `aborted` field to the existing terminal lifecycle metadata path, not infer cancellation heuristically from `stopReason` or rely on the guarded fallback emitter.
