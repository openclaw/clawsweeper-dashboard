## Summary

Keep open. Current main only forwards `stopReason` on the primary pi-embedded terminal lifecycle event; it still lacks the explicit `aborted` boolean even though the runner and downstream consumers treat that metadata as meaningful, and the related fix PR is closed unmerged.

## What This Changes

The issue requests that pi-embedded `lifecycle:end` events include `aborted` and `stopReason` so event subscribers can distinguish cancellation from normal completion.

## Best Possible Solution

Add typed `aborted` propagation from the embedded attempt result into terminal lifecycle metadata and cover both global and direct event payloads with regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows the primary subscribe-layer terminal lifecycle event can carry `stopReason` but has no `aborted` state field, setter parameter, or emitted payload property.

## Solution Assessment

Yes. The narrow maintainable fix is to propagate the runner's explicit `aborted` result metadata; deriving cancellation only from `stopReason` would be less reliable.
