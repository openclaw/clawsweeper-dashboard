## Summary

Keep open. Current `main` partially addresses the report by forwarding `stopReason` through the pi-embedded terminal lifecycle metadata, but it still never carries an `aborted` boolean on the primary `lifecycle:end` event, and the fallback emitter that includes `aborted` remains guarded off after the subscribe path emits the terminal lifecycle event.

## What This Changes

The issue requests that pi-embedded `lifecycle:end` event payloads include `aborted` and `stopReason` so event subscribers can distinguish cancellation from natural completion.

## Best Possible Solution

Extend the pi-embedded terminal lifecycle metadata contract with an explicit `aborted` boolean from the runner attempt, emit that boolean on both global and direct lifecycle terminal events, and add focused regression tests for natural completion, explicit abort, and timeout/fallback behavior while preserving current `stopReason`, `yielded`, liveness, replay, and gateway session status mapping.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current `main`: `handleAgentEnd` can emit `stopReason` from `ctx.state.terminalStopReason`, but there is no state field, setter parameter, or emitted payload property for `aborted`, so subscribers cannot receive the requested boolean on the primary subscribe path.

## Solution Assessment

Yes, the best fix is a narrow typed metadata propagation change rather than a heuristic at the emitter. The closed PR's approach of deriving abort state from stop reasons would be less robust than forwarding the explicit runner `aborted` metadata already present in attempt results.
