## Summary

Keep open. Current main still does not propagate the explicit `aborted` boolean through the pi-embedded terminal lifecycle path, even though `stopReason` is now carried and downstream consumers already understand both fields.

## What This Changes

The issue asks for pi-embedded `lifecycle:end` events to include `aborted` and `stopReason` so event-only subscribers can distinguish cancellation from natural completion.

## Best Possible Solution

Propagate the runner's explicit abort result through the existing terminal lifecycle metadata contract and emit it alongside `stopReason` on both lifecycle event sinks, with focused cancellation and natural-completion coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: an aborted pi-embedded run has `result.meta.aborted`, but the primary subscribe-layer terminal event cannot carry it. I did not execute a live abort flow in this read-only sweep.

## Solution Assessment

Yes. The narrow maintainable fix is to add explicit abort metadata to the existing terminal lifecycle path instead of inferring cancellation from `stopReason` or relying on the guarded fallback emitter.
