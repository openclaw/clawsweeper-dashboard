## Summary

Keep open. Current main and the latest release still have the core problem: `stopReason` can be carried on the primary pi-embedded terminal lifecycle path, but the explicit `aborted` boolean never reaches the subscribe-layer state, setter, or emitted `lifecycle:end` payload. The CoClaw plugin context is relevant because event-only observers cannot rely on the `agent()` RPC completion frame, and the related fix PR was closed unmerged after bundling unrelated changes.

## What This Changes

The issue requests that pi-embedded `lifecycle:end` events include `aborted` and `stopReason` so event subscribers can distinguish cancellation from natural completion.

## Best Possible Solution

Propagate the runner's explicit abort result through the existing terminal lifecycle metadata contract and emit it with `stopReason` on both lifecycle event sinks, with regression coverage for cancellation and natural completion.

## Reproduction Assessment

Yes. Source inspection on current main shows the runner has explicit `meta.aborted`, but the terminal lifecycle setter and emitted `lifecycle:end` payload cannot carry that boolean; a live abort run was not executed in this read-only sweep.

## Solution Assessment

Yes. The narrow maintainable fix is to add explicit abort metadata to the existing terminal lifecycle path instead of inferring cancellation from `stopReason` or relying on the guarded fallback emitter.
