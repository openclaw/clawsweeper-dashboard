## Summary

Keep open. Current main still forwards `stopReason` through the pi-embedded terminal lifecycle metadata, but the primary `lifecycle:end` event never includes an explicit `aborted` boolean, and the fallback emitter that would include it remains skipped once the subscribe path emits the terminal lifecycle event. The linked fix PR is closed unmerged, so there is no implemented-on-main basis to close this issue.

## What This Changes

The issue requests that pi-embedded `lifecycle:end` event payloads include explicit `aborted` and `stopReason` fields so subscribers can distinguish cancellation from natural completion.

## Best Possible Solution

Extend the pi-embedded terminal lifecycle metadata contract with an explicit `aborted` boolean from the runner attempt, emit it on both global and direct terminal lifecycle events, and add focused regression tests while preserving the existing `stopReason`, `yielded`, liveness, replay, and gateway session status behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: the subscribe-layer terminal event can emit `stopReason`, but there is no state field, setter parameter, or emitted payload property for `aborted`, so subscribers cannot receive the requested boolean on the primary path.

## Solution Assessment

Yes. The best fix is a narrow typed metadata propagation change from runner result to lifecycle event; the closed PR's documented heuristic of deriving abort state from stop reasons would be less robust than forwarding the explicit attempt metadata already available.
