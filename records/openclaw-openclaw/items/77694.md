## Summary

Keep open: current OpenClaw now pins a newer `acpx` than the reporter used, but published `acpx` 0.11.2 still routes flow ACP-node output through the quiet raw-text capture path, and that path can remain empty unless the dependency emits the terminating stop-reason message that flushes buffered text.

## What This Changes

The issue reports that `acpx flow run` ACP-node outputs complete as empty strings instead of returning the agent reply text.

## Best Possible Solution

Fix and prove quiet ACPX flow-output capture in the upstream `acpx` package, publish it, then update the OpenClaw ACPX plugin pin with live `acpx flow run` proof for pi or opencode.

## Reproduction Assessment

Yes for a source-level path, but not a live current-main repro: the current dependency still reads flow output from quiet-captured raw text, and that formatter can retain text internally without writing to the capture stream unless a stop-reason message arrives. The original live steps should be rerun against current `acpx` 0.11.2 with pi or opencode.

## Solution Assessment

No close-worthy solution is proven yet. A dependency bump alone is not the best fix unless the upstream `acpx` quiet-capture behavior is changed or live proof shows current agents now emit the terminating message needed by the capture path.
