## Summary

Keep open for maintainer review: the patch addresses a source-proven Gateway restart close-path gap and I found no blocking code or security findings, but it intentionally writes restart-recovery session state during shutdown so the session-state merge risk should remain visible.

## What This Changes

The PR adds a Gateway restart shutdown bridge that marks active main-session refs for restart recovery before aborting active runs after a drain timeout, with focused close-handler coverage.

## Best Possible Solution

Land the narrow close-path bridge after maintainers accept the session-identity scope and normal CI/proof gates, keeping restart-recovery filtering centralized in `src/agents/main-session-restart-recovery.ts`.

## Reproduction Assessment

Yes. Current main shows the restart close path can time out and abort active runs without a marker hook, and the PR body includes a source-level probe that exercises the same handler order after the patch.

## Solution Assessment

Yes. Reusing the existing main-session restart-recovery marker immediately before close-path abort cleanup is the narrowest maintainable fix; moving this only to the run loop would miss this server-close abort path and duplicate marker policy.
