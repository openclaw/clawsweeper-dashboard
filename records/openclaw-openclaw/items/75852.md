## Summary

Keep open: the latest shipped report remains plausible on current main. The central Telegram polling code still treats 409s as dirty restarts, but stale transport closure is fire-and-forget and existing tests do not cover slow stale-transport shutdown before replacement polling starts.

## What This Changes

The issue reports Telegram long-polling entering recurring 409 getUpdates conflicts on macOS v2026.4.29 despite one local gateway process, empty webhook state, and direct Bot API success while OpenClaw is stopped.

## Best Possible Solution

Keep the repair inside the Telegram plugin lifecycle: make dirty 409/stall restarts single-flight, abort the active fetch, stop runner/bot cleanly, and ensure stale transport close/destroy is completed or otherwise coordinated before a replacement cycle can issue getUpdates.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction path with v2026.4.29 on macOS, one local gateway, health monitor disabled, empty webhook state, direct getUpdates success while OpenClaw is stopped, and conflicts returning after restart; I did not live-replay it without a token.

## Solution Assessment

No. The current fresh-transport recovery is useful but not proven sufficient for this report; the safer narrow fix is to serialize the dirty restart teardown path and add coverage for slow stale transport closure.
