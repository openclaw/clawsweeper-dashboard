## Summary

Keep open: current main and the latest shipped tag still use a fire-and-forget stale Telegram transport close during dirty rebuilds, and the open closing PR for this issue is not merged. The issue should remain open until that PR or an equivalent Telegram-plugin fix lands.

## What This Changes

The issue asks OpenClaw to stop Telegram polling from self-conflicting with repeated 409 getUpdates errors when only one local gateway process is running.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/76153 or an equivalent Telegram-plugin fix that serializes stale transport teardown before the next dirty-rebuild polling cycle, then close this issue from the merged fix.

## Reproduction Assessment

Yes, with high confidence from source inspection and reporter evidence. Current main marks the transport dirty after getUpdates 409 conflicts, then starts the next bot with a replacement transport while the stale transport close is still fire-and-forget; I did not run a live Telegram token locally.

## Solution Assessment

Yes. The narrow maintainable fix is inside the Telegram plugin: await or otherwise serialize stale TelegramTransport.close() before the replacement polling transport can start getUpdates, without adding core policy or config.
