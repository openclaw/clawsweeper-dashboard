## Summary

Keep open: current main still has a Telegram dirty-rebuild path that can start the replacement polling transport before the stale transport close has resolved, and open closing PR #76153 already targets that lifecycle gap. Reviewed item: https://github.com/openclaw/openclaw/issues/75852.

## What This Changes

Ensure Telegram polling mode keeps exactly one active Bot API long poll per local gateway process and does not enter repeated 409 getUpdates conflicts after restart or transport rebuild.

## Best Possible Solution

Keep the fix in the Telegram plugin polling lifecycle by serializing stale transport teardown before the next dirty-rebuild poll can start, with regression coverage for slow stale closes, then close this issue via #76153 or an equivalent merged fix.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction path with macOS v2026.4.29, single-process checks, empty webhook state, and direct Bot API stop/start proof; current-main code review shows a matching dirty-rebuild race.

## Solution Assessment

Yes. The narrow maintainable solution is to await or otherwise serialize stale transport close before returning the replacement polling transport; #76153 is the appropriate implementation candidate to review first.
