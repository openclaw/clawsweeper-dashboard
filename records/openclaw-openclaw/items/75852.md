## Summary

Keep open: the v2026.4.29 macOS report remains plausible on current main because Telegram still rebuilds after 409 conflicts while stale transport shutdown is explicitly fire-and-forget, and the tests only prove eventual closure with immediately resolving mocks.

## What This Changes

The issue reports Telegram polling repeatedly self-conflicting with 409 getUpdates errors on macOS v2026.4.29 despite one local gateway process and direct Bot API success while OpenClaw is stopped.

## Best Possible Solution

Keep the fix inside the Telegram plugin lifecycle by making dirty 409, network-error, and stall restarts single-flight and coordinating stale transport close/destroy before a replacement poll can start.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction path with v2026.4.29 on macOS, one local gateway, health monitor disabled, empty webhook state, direct getUpdates success while OpenClaw is stopped, and conflicts returning after restart; I did not live-replay it without a token.

## Solution Assessment

No. The current fresh-transport recovery is useful but not sufficient evidence for this report; the safer narrow fix is to serialize dirty restart teardown and add coverage for slow stale transport closure.

In the dirty restart path after getUpdates 409 conflicts, recoverable polling network errors, and stall restarts, serialize teardown so the active fetch is aborted, runner/bot stop is coordinated, and stale Telegram transport close/destroy is completed or otherwise single-flight before the replacement cycle can issue a fresh getUpdates.
