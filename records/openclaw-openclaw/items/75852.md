## Summary

Keep open: current main still permits a dirty Telegram transport rebuild to create and use the next transport while stale transport shutdown is fire-and-forget, so the macOS 409 self-conflict report remains plausible and narrow.

## What This Changes

The issue reports Telegram long polling on macOS v2026.4.29 entering repeated 409 getUpdates conflicts despite one local gateway and direct Bot API success while OpenClaw is stopped.

## Best Possible Solution

Keep the fix inside the Telegram plugin lifecycle by making dirty 409, network-error, and stall restarts single-flight and coordinating stale transport teardown before any replacement getUpdates poll can start.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path with one macOS gateway, health monitor disabled, no webhook, direct getUpdates success while stopped, and conflicts returning after OpenClaw restarts; I did not live-replay it without a bot token.

## Solution Assessment

No. The current fresh-transport recovery is useful but not sufficient for this report; the safer narrow fix is to serialize Telegram teardown/rebuild rather than add broader core restart policy.
