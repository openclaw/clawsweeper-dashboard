## Summary

Keep open: the macOS v2026.4.29 Telegram 409 loop remains a plausible current Telegram plugin lifecycle bug, because current main still rebuilds after 409 conflicts while stale transport shutdown is asynchronous and existing tests do not prove slow stale teardown is serialized before the next long poll.

## What This Changes

The issue requests a fix for Telegram polling on macOS v2026.4.29 where one local gateway appears to self-conflict with Telegram getUpdates and loops on 409 errors.

## Best Possible Solution

Keep the fix inside the Telegram plugin by making dirty 409, recoverable-error, and stall restarts single-flight and ensuring stale runner/bot/transport teardown is coordinated before a replacement long poll can start.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction path on macOS v2026.4.29 with one local gateway, health monitor disabled, empty webhook state, direct getUpdates success while stopped, and conflicts returning after restart; I did not live-replay it without a bot token.

## Solution Assessment

No for the current implementation. Rebuilding the transport is useful, but the safer narrow fix is to serialize dirty restart teardown and add a slow-close regression test that proves the next getUpdates cannot overlap stale transport cleanup.
