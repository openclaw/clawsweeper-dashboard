## Summary

Keep open: current main has several Telegram 409 mitigations, but this report is against the latest shipped release and includes direct stop/start Bot API evidence that points to a remaining single-gateway polling lifecycle problem. The current recovery path still logs the reported signatures, and existing tests do not prove that dirty transport rebuilds wait for the old transport/socket to be fully quiesced before the next long poll starts.

## What This Changes

Fix Telegram polling so a single macOS local gateway process keeps exactly one active `getUpdates` long poll and does not enter repeated 409 conflict/rebuild loops.

## Best Possible Solution

Keep this in the Telegram plugin and make the polling restart path fully serialize teardown before the next long poll starts: old runner stopped, bot stopped, active fetch aborted, and stale transport closed or destroyed before a fresh `getUpdates` can run. Add a regression test that simulates slow stale-transport close and proves no second polling cycle starts early.

## Reproduction Assessment

Yes. The issue provides a concrete macOS v2026.4.29 reproduction with direct Telegram Bot API stop/start verification, and the current code still contains the reported log signatures; live token validation was not run in this read-only review.

## Solution Assessment

No current fix is proven sufficient. The existing in-process lease and fresh-transport-on-409 behavior are useful, but the safer fix is to close the teardown ordering gap and test slow transport shutdown before starting the replacement poller.

## Repair Work Prompt

Fix the Telegram polling self-conflict reported in https://github.com/openclaw/openclaw/issues/75852. The observable bug is repeated `409 Conflict: terminated by other getUpdates request` on macOS v2026.4.29 even with one local gateway process, empty webhook state, and health monitor disabled; direct Bot API polling succeeds while OpenClaw is stopped and conflicts return after OpenClaw restarts. Keep the fix inside the Telegram plugin. Investigate `extensions/telegram/src/polling-session.ts`, `extensions/telegram/src/polling-transport-state.ts`, `extensions/telegram/src/monitor.ts`, and `extensions/telegram/src/polling-lease.ts`. Ensure dirty restart paths after `getUpdates` 409, recoverable polling errors, or stalls fully stop the previous runner/bot, abort in-flight fetch, and close/destroy the stale transport before the next polling cycle can start a fresh `getUpdates`. Preserve multi-account/different-token behavior and the same-token lease guard; do not move Telegram-specific policy into core and do not change launchd/service behavior unless a test proves it is required. Add regression coverage with a slow `transport.close()` or equivalent teardown delay that fails on the current fire-and-forget rebuild behavior and proves the second poller does not start early. Add a `CHANGELOG.md` fix entry for the user-facing Telegram reliability fix; update `docs/channels/telegram.md` only if diagnostics or documented behavior changes.
