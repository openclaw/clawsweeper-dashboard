Keep open: older 409 fixes exist, but this v2026.4.29 macOS report adds direct OpenClaw stop/start proof and current main still starts the replacement polling transport before awaiting stale transport close.

## What This Changes

The issue reports repeated Telegram getUpdates 409 conflicts on macOS v2026.4.29 despite a single local gateway process and direct Bot API success while OpenClaw is stopped.

## Best Possible Solution

Telegram polling should keep one in-process getUpdates lifecycle by serializing dirty restart teardown before a replacement poll can start, with regression coverage for delayed stale transport close after 409 recovery.

## Reproduction Assessment

Yes. The issue provides high-confidence macOS v2026.4.29 steps with one gateway, health monitor disabled, empty webhook state, direct getUpdates success while stopped, and conflicts returning after restart.

## Solution Assessment

No. The current fresh-transport recovery is useful but not sufficient; the safer narrow fix is a Telegram-plugin single-flight teardown/rebuild path rather than a core or docs-only change.
