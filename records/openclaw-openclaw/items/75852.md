## Summary

Keep open: the macOS v2026.4.29 report remains plausible on current main because Telegram still starts the replacement polling transport before stale transport close/destroy is awaited, and current tests do not cover slow stale-transport teardown before the next getUpdates cycle.

## What This Changes

The issue asks Telegram polling mode to maintain exactly one active getUpdates long-poll on macOS v2026.4.29 when only one local gateway process is running.

## Best Possible Solution

Keep the fix inside the Telegram plugin by making dirty 409, network-error, and stall restarts single-flight, coordinating stale bot/runner/transport teardown before any replacement getUpdates can start, and adding slow-close regression coverage.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction path on macOS v2026.4.29 with direct Bot API stop/start verification; this review verified that the current code still has a plausible overlapping rebuild path but did not live-replay it with a token.

## Solution Assessment

No. The current fresh-transport recovery is useful but not enough for this report because stale transport close is intentionally fire-and-forget; the safer narrow solution is to serialize or bound teardown before the replacement poll starts.
