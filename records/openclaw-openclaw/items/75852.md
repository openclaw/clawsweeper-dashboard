## Summary

Keep open: current main and v2026.6.10 include useful Telegram 409 mitigations, but they still route isolated getUpdates conflicts through worker exit plus parent restart/backoff and keep dirty transport rebuild teardown asynchronous, so the reported same-token stale long-poll race is not proven fixed.

## What This Changes

The issue reports Telegram polling entering repeated Bot API getUpdates 409 conflicts on macOS with one local gateway, suggesting overlapping OpenClaw long-poll lifecycle state rather than an external duplicate poller.

## Best Possible Solution

Keep this issue as the canonical Telegram polling self-conflict tracker and land a Telegram-owned lifecycle fix that prevents same-token overlapping getUpdates across worker restart and transport rebuild paths, backed by redacted live Bot API proof.

## Reproduction Assessment

No high-confidence live current-main reproduction was run in this read-only review. The issue and later comments provide credible stop/start and timing evidence, and current source still contains the worker restart, dirty rebuild, and timeout paths that match the reported 409 loop.

## Solution Assessment

No current fix is proven complete. The best fix should address Telegram polling cancellation/lifecycle timing across isolated worker and transport rebuild paths, rather than only adding another retry or config escape hatch.
