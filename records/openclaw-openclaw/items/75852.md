## Summary

Keep open: current main still uses a fire-and-forget stale Telegram transport close during dirty rebuilds, and open PR #76153 already targets that lifecycle gap, so this issue should remain open until that or an equivalent fix lands.

## What This Changes

The issue reports Telegram polling 409 self-conflicts on macOS with one local gateway process and asks for the polling lifecycle to avoid overlapping `getUpdates` calls.

## Best Possible Solution

Review and land #76153 or an equivalent Telegram-plugin fix that serializes stale transport teardown before the next dirty rebuild poll cycle starts, with regression coverage for slow stale closes.

## Reproduction Assessment

Yes, from source inspection and reporter evidence. The issue provides a concrete stop/start Bot API reproduction, and current main still has the non-awaited dirty rebuild close path that can match the reported 409 loop.

## Solution Assessment

Yes. The narrow owner-module fix is to await or otherwise serialize stale Telegram transport close before starting the replacement polling cycle; no core policy or config change is needed.
