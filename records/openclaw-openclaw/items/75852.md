## Summary

Keep open: the report remains valid on current main, and open PR #76153 already targets the Telegram dirty-rebuild teardown race with closing syntax, so this issue should wait for that PR or an equivalent fix to land.

## What This Changes

A single macOS local Gateway using Telegram polling should not self-conflict with repeated Bot API `getUpdates` 409 errors after startup.

## Best Possible Solution

Keep the fix in the Telegram plugin by serializing stale transport teardown before the next dirty-rebuild poll starts, with regression coverage for slow stale closes.

## Reproduction Assessment

Yes. The issue provides a high-confidence macOS v2026.4.29 stop/start reproduction with direct Bot API proof, and current main still has a source-level race that matches the observed 409 loop.

## Solution Assessment

Yes. The narrow maintainable fix is to await or otherwise serialize dirty-rebuild transport close in `extensions/telegram/src/polling-transport-state.ts` and update the polling-session/tests around that contract.
