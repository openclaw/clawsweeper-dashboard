## Summary

Keep open: current main does not prove the v2026.4.29 macOS 409 loop is fixed, because the Telegram plugin still rebuilds after getUpdates conflicts while stale transport shutdown on the rebuild path is fire-and-forget.

## What This Changes

The issue reports Telegram polling entering repeated 409 getUpdates conflicts on macOS v2026.4.29 despite a single local gateway process.

## Best Possible Solution

Keep the fix inside the Telegram plugin lifecycle by making dirty 409, network-error, and stall restarts single-flight and coordinating stale runner, bot, fetch, and transport teardown before a replacement poll can issue getUpdates.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction path with macOS v2026.4.29, one local gateway, disabled health monitor, empty webhook state, direct getUpdates success while OpenClaw is stopped, and conflicts returning after restart.

## Solution Assessment

No. The current fresh-transport recovery is not sufficient for this report; the safer narrow fix is to serialize dirty restart teardown and add a regression test where stale transport close is deliberately slow.
