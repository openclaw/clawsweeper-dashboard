## Summary

Keep open: current main and v2026.5.12 still start the replacement Telegram polling transport before the stale dirty transport close is awaited, and the open closing PR has not merged.

## What This Changes

A macOS Telegram polling bug report asks OpenClaw to avoid self-conflicting getUpdates 409 loops when only one local gateway process is running.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/76153 or an equivalent Telegram-plugin fix that serializes stale transport teardown before the next dirty-rebuild polling cycle starts, with live Telegram proof.

## Reproduction Assessment

Yes, source-level with high confidence: the reporter gives a concrete stop/start Bot API path, and current main still routes getUpdates 409 conflicts into a dirty rebuild that closes the stale transport asynchronously without awaiting it. I did not run a live Telegram token locally.

## Solution Assessment

Yes, the best direction is a narrow Telegram plugin lifecycle fix that awaits or otherwise serializes stale TelegramTransport.close() before a replacement getUpdates poller can start. The existing open closing PR is the right review target before this issue is closed.
