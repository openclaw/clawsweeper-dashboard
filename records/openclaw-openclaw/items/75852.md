## Summary

Keep open: current main still handles Telegram 409 recovery by starting a replacement transport while stale transport shutdown is fire-and-forget, so the reported self-conflict remains plausible and is not covered by the existing immediate-close tests.

## What This Changes

The issue reports macOS v2026.4.29 Telegram polling entering repeated self-conflicting getUpdates 409 loops with only one local OpenClaw gateway process.

## Best Possible Solution

Keep the fix inside the Telegram plugin by making dirty 409, network-error, and stall restarts single-flight so stale transport close/destroy is coordinated before any replacement getUpdates begins.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path on macOS v2026.4.29 with one local gateway, health monitor disabled, empty webhook state, direct getUpdates success while OpenClaw is stopped, and conflicts returning after restart; I did not live-test a Telegram token.

## Solution Assessment

No. The current fresh-transport recovery is not the best final state for this report because it still allows fire-and-forget stale transport shutdown; the narrower maintainable fix is to serialize teardown and add slow-close coverage.
