## Label Justifications

- `P1`: A repeated Telegram `getUpdates` 409 loop can make a configured Telegram channel unreliable or unusable for real users.
- `impact:message-loss`: The conflict loop blocks Telegram inbound updates and command replies, causing dropped or delayed channel messages.

## Summary

Keep open: current main and v2026.6.10 include useful Telegram 409 mitigations, especially the merged isolated-ingress error-code handling, but they do not prove the same-token stale `getUpdates` overlap is eliminated. The current source still restarts/backoffs after conflicts and still closes dirty transports through a fire-and-forget rebuild path, while recent issue comments report newer macOS, Windows, and isolated-ingress reproductions.

## What This Changes

The issue reports Telegram `getUpdates` 409 self-conflicts in polling mode after macOS gateway startup or restart, even when only one local OpenClaw gateway process appears to be running.

## Best Possible Solution

Keep this issue as the canonical Telegram polling self-conflict tracker and land a Telegram-owned lifecycle fix that prevents same-token overlapping `getUpdates` across classic polling, isolated ingress, restart, and transport-fallback paths, backed by redacted live Bot API proof.

## Reproduction Assessment

No high-confidence live current-main reproduction was run in this read-only review. The issue and later comments provide credible stop/start and timing evidence, and current source still contains the restart/dirty-rebuild paths that match the reported logs.

## Solution Assessment

No, current main is not proven to be the complete fix. The merged 409 handling is a useful mitigation, but the best solution still needs maintainer-selected lifecycle behavior and live Telegram proof rather than only restart/backoff recovery.
