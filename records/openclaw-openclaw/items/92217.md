## Summary

Keep this PR open, but do not merge the current head: the live Fireworks turns succeed and the request-compat approach is sound, yet the plugin’s dynamic resolver still preempts the four new manifest rows and replaces their runtime metadata with the default Kimi template. The open companion PR identifies and repairs that same catalog-resolution ownership problem, so these changes need a coordinated rebase or integration.

## What This Changes

Adds four bundled Fireworks reasoning models, family-specific thinking and request compatibility for catalog and dynamic IDs, provider documentation and tests, and lowest-supported clamping for below-range thinking requests.

## Best Possible Solution

Rebase this work after, or integrate the narrow manifest-ID deferral from, the companion PR so cataloged IDs resolve from `openclaw.plugin.json`, while only genuinely dynamic IDs receive the family compat patch; retain the existing DeepSeek compat seam and the generic lowest-supported-effort clamp with resolver-level regression coverage.

## Reproduction Assessment

Yes. The successful live turns are directly demonstrated, and the metadata regression is source-reproducible because `resolveFireworksDynamicModel` clones the default template before core can select the matching manifest row.

## Solution Assessment

No for the current complete patch. The compat and clamp portions are well-shaped, but cataloged IDs must defer to the canonical manifest resolver instead of being reconstructed through the dynamic fallback.
