## Summary

The feature direction is useful, and the follow-up comment addressed same-chat fallback tracking, but the PR head still has a merge-blocking precedence bug for WhatsApp DM automatic mode and lacks the required user-facing changelog entry; the security pass found no concrete supply-chain concern.

## What This Changes

The PR adds `channels.whatsapp.dmVisibleReplies` with account overrides, routes opted-in WhatsApp DMs through message-tool-only delivery with a suppressed-final fallback, updates prompt/config/docs surfaces, and adds targeted tests.

## Best Possible Solution

Keep the feature direction, but make WhatsApp DM mode precedence explicit: direct chats should pass `"automatic"` unless the resolved WhatsApp account setting is `"message_tool"`, add a regression test with global `messages.visibleReplies: "message_tool"`, add a `CHANGELOG.md` entry with valid contributor credit, then run the targeted tests plus the normal changed gate.

## Reproduction Assessment

Yes. Configure global `messages.visibleReplies: "message_tool"`, leave WhatsApp `dmVisibleReplies` unset or `"automatic"`, and process a WhatsApp direct chat; the PR passes `undefined`, so shared dispatch resolves the DM to `message_tool_only` instead of the documented automatic final delivery.

## Solution Assessment

No, not yet. The proposed shape is narrow and maintainable, but the implementation should explicitly request `"automatic"` for WhatsApp DM automatic mode rather than allowing the global direct-chat setting to override the new WhatsApp-specific default.
