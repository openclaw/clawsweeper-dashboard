## Summary

Keep open. The PR adds a useful, narrow WhatsApp DM visible-reply mode, but the current patch still has a source-level precedence bug where documented automatic WhatsApp DMs can be overridden by global direct-chat `messages.visibleReplies`, and it is missing the required changelog entry for a user-facing config feature.

## What This Changes

The PR adds `channels.whatsapp.dmVisibleReplies` with account overrides, routes opted-in WhatsApp DMs through message-tool-only delivery with a suppressed-final fallback, updates prompt/docs/config schema surfaces, and adds targeted tests.

## Best Possible Solution

Keep the feature direction, but make WhatsApp DM visible-reply precedence explicit and covered by regression tests, add a `CHANGELOG.md` entry under the active release section with valid contributor credit, then merge after the normal changed gate while preserving automatic defaults, group behavior, and send-policy denials.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `messages.visibleReplies: "message_tool"` with a WhatsApp direct chat whose resolved `dmVisibleReplies` is unset or `"automatic"`; the PR passes `undefined`, so shared dispatch can still resolve that DM to `message_tool_only`.

## Solution Assessment

No. The PR is mostly narrow and maintainable, and the follow-up same-chat fallback tracking is useful, but the implementation does not yet enforce the documented automatic WhatsApp DM mode; the safer fix is to resolve the WhatsApp DM mode explicitly before calling shared dispatch or revise the docs/tests to state global precedence.
