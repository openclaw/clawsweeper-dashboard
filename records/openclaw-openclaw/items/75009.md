## Summary

Keep open: the feature direction is useful, and the same-chat fallback tracking appears to address the author’s follow-up comment, but the PR still lets documented automatic WhatsApp DM delivery fall through to the global direct-chat `messages.visibleReplies` setting. It also needs the required user-facing changelog entry before merge.

## What This Changes

The PR adds a WhatsApp `dmVisibleReplies` config surface with account overrides, routes opted-in WhatsApp DMs through message-tool-only reply delivery with a suppressed-final fallback, updates prompt/docs/schema surfaces, and adds targeted tests.

## Best Possible Solution

Keep the PR’s feature shape, but make the WhatsApp DM setting authoritative for direct chats, add focused coverage for global `messages.visibleReplies` precedence, document the release note, and then merge through the normal changed gate while preserving group behavior and send-policy denials.

## Reproduction Assessment

Yes. Configure `messages.visibleReplies: "message_tool"` and process a direct WhatsApp message whose resolved `dmVisibleReplies` is unset or `"automatic"`; the PR passes no requested mode, so shared direct-chat dispatch selects `message_tool_only` instead of the documented automatic WhatsApp DM behavior.

## Solution Assessment

No. The proposed design is mostly narrow and maintainable, but the implementation should pass explicit `"automatic"` for WhatsApp DMs that are not opted into `"message_tool"`, plus a regression test and changelog entry.
