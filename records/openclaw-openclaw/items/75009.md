## Summary

Keep open: the feature is not implemented on current main, and the PR still has a source-level blocker where WhatsApp DMs documented as automatic can inherit global tool-only delivery. The external PR also needs real WhatsApp behavior proof before it can merge.

## What This Changes

The PR adds `channels.whatsapp.dmVisibleReplies` with account overrides, routes opted-in WhatsApp DMs through message-tool-only replies with a fallback, and updates prompt, docs, config schema, and tests.

## Best Possible Solution

Keep the feature direction, but make WhatsApp DM visible-reply precedence explicit in implementation and tests, then require redacted real WhatsApp proof before maintainer landing.

## Reproduction Assessment

Yes. A high-confidence source-level path is to set `messages.visibleReplies: "message_tool"` while leaving WhatsApp `dmVisibleReplies` unset or `"automatic"`; the PR passes `undefined`, and shared direct-chat dispatch can resolve that to `message_tool_only`.

## Solution Assessment

No. The feature shape is plausible, but the PR should pass explicit `"automatic"` for the documented WhatsApp automatic case or intentionally document and test that the global setting wins.
