## Summary

Keep this PR open. The feature is useful and current main does not already provide `channels.whatsapp.dmVisibleReplies`, but the latest PR head still has a merge-blocking precedence gap for automatic WhatsApp DMs when global `messages.visibleReplies` is `message_tool`, plus the required user-facing changelog entry is missing.

## What This Changes

The PR adds a WhatsApp `dmVisibleReplies` config with account overrides, routes opted-in DMs through message-tool-only dispatch with a suppressed-final fallback, updates prompt/docs/schema surfaces, and adds targeted WhatsApp, dispatch, and message-tool tests.

## Best Possible Solution

Keep the feature direction, but make WhatsApp DM visible-reply precedence explicit in code, docs, and tests, add the user-facing changelog entry under `Unreleased`, then run the targeted WhatsApp/config/dispatch tests plus the normal changed gate before merge.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `messages.visibleReplies: "message_tool"` and process a WhatsApp direct chat whose resolved `dmVisibleReplies` is unset or `"automatic"`; the PR passes `undefined`, so current shared dispatch can still resolve the turn to `message_tool_only`.

## Solution Assessment

No. The proposed structure is mostly narrow, but it does not reliably implement its documented automatic-mode semantics; either pass an explicit `"automatic"` for WhatsApp automatic DMs or document and test that the global setting wins.
