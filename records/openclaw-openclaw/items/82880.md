## Summary

Keep open: current main still lacks the requested ACPX and Firecrawl hardening, but this PR is not merge-ready because it changes Firecrawl target semantics, lets non-HTTP scrape URLs bypass the old runtime guard, and adds verifier-only ACPX signing without launcher wiring.

## Best Possible Solution

Redesign this as maintainer-approved security hardening that preserves Firecrawl URL semantics and non-HTTP blocking, wires ACPX payload signing end-to-end, and includes real Firecrawl runtime proof.

Yes. Source inspection is enough to reproduce the PR defects: non-HTTP Firecrawl URLs now skip the old runtime assertion path, and the repo has no current ACPX launcher wiring for the new HMAC secret.

## Solution Assessment

No. The current implementation changes Firecrawl target semantics and adds verifier-only ACPX signing; the safer solution is an end-to-end design that preserves default behavior unless maintainers explicitly approve a breaking strict mode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep non-HTTP scrape targets blocked:** `extensions/firecrawl/src/firecrawl-client.ts:547`
  - body: The old `runFirecrawlScrape` called `assertFirecrawlScrapeTargetAllowed` before building any Firecrawl request. This branch only runs the new DNS branch for `http:`/`https:` and otherwise continues with the original `urlToScrape`, so `file:///...` and other non-HTTP targets are no longer blocked by OpenClaw before reaching Firecrawl.
  - confidence: 0.95
- **[P1] Preserve the Firecrawl scrape hostname:** `extensions/firecrawl/src/firecrawl-client.ts:555-556`
  - body: Replacing `parsed.hostname` with the first resolved IP changes the URL sent to Firecrawl, which can break normal HTTPS virtual hosts and certificates; the added comment even notes Firecrawl may reject IP-based requests. Keep the original target URL unless there is a maintainer-approved Firecrawl-compatible pinning design.
  - confidence: 0.9
- **[P1] Wire ACPX signatures through the real launcher:** `extensions/acpx/src/runtime-internals/mcp-proxy.mjs:11-13`
  - body: The proxy only enforces HMAC when `OPENCLAW_MCP_PROXY_SECRET` is set, but this PR does not add launcher code that sets the secret and signs the `--payload`. Defaults remain unsigned, while manually setting the env would make actual ACPX proxy launches reject unsigned payloads.
  - confidence: 0.88
- **[P2] Bind command policy to exact argv:** `extensions/acpx/src/runtime-internals/mcp-proxy.mjs:116-118`
  - body: `OPENCLAW_MCP_PROXY_ALLOWED_COMMAND_PREFIX` checks only `target.command.startsWith(...)`, so allowing a generic executable such as `node` still permits dangerous attacker-controlled args, and similar path prefixes can match. Bind policy to the exact executable plus expected arguments, or remove this as a claimed security boundary.
  - confidence: 0.82
