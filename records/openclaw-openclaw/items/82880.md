## Summary

Keep open: this security PR still needs contributor real behavior proof and has blocking Firecrawl and ACPX correctness/security issues in the latest head.

## What This Changes

The PR adds ACPX proxy payload signature/prefix checks, Firecrawl DNS-aware scrape validation with URL rewriting, and related unit tests.

## Best Possible Solution

Keep Firecrawl's protocol/literal guard, add only enforceable DNS protection, and make ACPX payload authentication ESM-safe and wired to the payload producer with targeted tests and real proof.

## Reproduction Assessment

Yes. Source inspection shows the latest runFirecrawlScrape path skips non-HTTP rejection and the ACPX signed path calls require inside ESM; CI also confirms a current type-check failure on the PR head.

## Solution Assessment

No. The proposed patch is not yet the best solution because it bypasses an existing Firecrawl guard, rewrites public hostnames to IP URLs without proven Firecrawl support, and leaves the ACPX signing path broken.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Reject non-HTTP scrape URLs before posting:** `extensions/firecrawl/src/firecrawl-client.ts:547`
  - body: The new runFirecrawlScrape path only validates URLs inside the http/https branch. A file: or ftp: URL now skips that branch and is still sent as the Firecrawl request body, whereas current main rejected non-HTTP(S) URLs before any request. Restore the shared guard or explicitly throw before building the Firecrawl request.
  - confidence: 0.91
- **[P1] Do not replace scrape hostnames with raw IPs:** `extensions/firecrawl/src/firecrawl-client.ts:555-556`
  - body: This rewrites every public scrape URL to the first resolved IP before sending it to Firecrawl, losing the original hostname for Host/SNI and virtual-host routing. Many normal HTTPS scrapes will fail or fetch the wrong site, while the cache key still uses the original URL. Keep the original URL unless there is a proven Firecrawl-supported way to bind the remote fetch to the validated address.
  - confidence: 0.86
- **[P2] Use an ESM crypto import in the proxy:** `extensions/acpx/src/runtime-internals/mcp-proxy.mjs:23`
  - body: mcp-proxy.mjs runs as an ES module, so require("node:crypto") is undefined when OPENCLAW_MCP_PROXY_SECRET is set. That makes both valid and invalid signed payloads fail before HMAC verification. Import crypto at module scope or use createRequire.
  - confidence: 0.92
- **[P2] Remove the unused hostname binding:** `extensions/firecrawl/src/firecrawl-client.ts:554`
  - body: originalHostname is declared but never read, and the PR's check-prod-types job is failing with TS6133 on this line. The branch cannot merge until the unused binding is removed or used.
  - confidence: 0.98

## Security Review

Status: needs_attention

Summary: The diff attempts security hardening but currently introduces a Firecrawl validation bypass and leaves ACPX signed-payload verification unusable.

Concerns:

- **[high] Firecrawl non-HTTP targets can bypass the request guard:** `extensions/firecrawl/src/firecrawl-client.ts:547`
  - body: Replacing the shared guard with an HTTP(S)-only branch lets non-HTTP URLs proceed to the Firecrawl API body instead of being rejected before request construction.
  - confidence: 0.91
- **[medium] Firecrawl IP rewriting changes the fetched origin:** `extensions/firecrawl/src/firecrawl-client.ts:555`
  - body: Sending an IP URL to the hosted API does not preserve the original hostname or TLS/SNI semantics in this code path, so it is not a safe drop-in SSRF fix without service-side support proof.
  - confidence: 0.86
- **[medium] ACPX signature verifier crashes in ESM:** `extensions/acpx/src/runtime-internals/mcp-proxy.mjs:23`
  - body: The new verifier calls CommonJS require inside an .mjs module, so enabling the secret-based signing path fails before the payload can be authenticated.
  - confidence: 0.92
