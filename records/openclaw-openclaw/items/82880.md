## Summary

Keep open: the security-hardening goal is plausible, but this PR is not mergeable because the current patch breaks signed ACPX mode, leaves ACPX authentication optional and unwired, weakens Firecrawl scrape validation for non-HTTP URLs, rewrites hosted scrape targets to IP URLs, and currently fails prod type checking.

## What This Changes

This PR adds ACPX MCP proxy prefix/signature checks and Firecrawl scrape-target DNS validation/IP rewriting with related tests.

## Best Possible Solution

Implement a maintainer-approved ACPX payload-auth contract wired at both producer and proxy, and use a Firecrawl SSRF boundary that preserves the requested hosted URL or moves pinning to an enforceable self-hosted transport path.

## Reproduction Assessment

Yes. Source inspection of the PR head gives high-confidence paths for the review findings, and CI already reproduces the unused-variable type failure; I did not run tests because this was a read-only review.

## Solution Assessment

No. The proposed implementation is not the best fix because it introduces an optional/unwired ACPX secret, uses CommonJS require in an ESM proxy, and trades Firecrawl SSRF hardening for protocol bypass and hosted URL rewriting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Use ESM crypto in signed proxy mode:** `extensions/acpx/src/runtime-internals/mcp-proxy.mjs:23`
  - body: When OPENCLAW_MCP_PROXY_SECRET is set, this .mjs file calls require("node:crypto"). Node ESM modules do not define require, so both valid and invalid signed payloads fail before the intended signature check can run.
  - confidence: 0.95
- **[P1] Make ACPX payload signing enforceable:** `extensions/acpx/src/runtime-internals/mcp-proxy.mjs:11-13`
  - body: The new signing check returns when the secret is unset, and this PR does not add a production launcher path that signs the payload. In the default path a tampered base64 payload is still accepted, while the prefix guard only checks the executable and not the arguments.
  - confidence: 0.88
- **[P1] Reject non-HTTP scrape targets in the scrape path:** `extensions/firecrawl/src/firecrawl-client.ts:547`
  - body: runFirecrawlScrape no longer calls assertFirecrawlScrapeTargetAllowed. For file: or other non-HTTP(S) URLs, this new branch skips DNS validation and then posts the original URL to Firecrawl, regressing the documented pre-forwarding target check.
  - confidence: 0.91
- **[P1] Keep the Firecrawl scrape host intact:** `extensions/firecrawl/src/firecrawl-client.ts:555-556`
  - body: Replacing the requested hostname with the first resolved IP changes what Firecrawl is asked to scrape. For normal HTTPS sites behind shared hosting or CDNs, the remote request loses the original host/SNI target and can fail or fetch the wrong virtual host.
  - confidence: 0.88
- **[P2] Fix the Firecrawl typecheck failure:** `extensions/firecrawl/src/firecrawl-client.ts:554`
  - body: The PR leaves originalHostname declared but unread, and check-prod-types is currently failing with TS6133 on this line. The branch needs to compile before the security behavior can be reviewed for merge.
  - confidence: 0.97

## Security Review

Status: needs_attention

Summary: The diff attempts security hardening but introduces concrete ACPX authentication and Firecrawl SSRF-boundary problems that need correction before merge.

Concerns:

- **[high] Signed ACPX mode crashes before verification:** `extensions/acpx/src/runtime-internals/mcp-proxy.mjs:23`
  - body: The proxy uses require inside a .mjs module, so enabling the new signature secret breaks the security mode before signatures are validated.
  - confidence: 0.95
- **[high] ACPX authentication is opt-in and producer-unwired:** `extensions/acpx/src/runtime-internals/mcp-proxy.mjs:11`
  - body: The proxy silently accepts unsigned payloads when the secret is absent, and the PR does not sign payloads at the production source, so the default tampered-payload risk is not closed.
  - confidence: 0.88
- **[high] Firecrawl scrape path can forward non-HTTP targets:** `extensions/firecrawl/src/firecrawl-client.ts:547`
  - body: runFirecrawlScrape skips the existing non-HTTP(S) assertion and only validates DNS inside an http/https branch, allowing file: and other schemes to reach the Firecrawl request body.
  - confidence: 0.91
- **[medium] Firecrawl IP rewriting changes the protected fetch target:** `extensions/firecrawl/src/firecrawl-client.ts:555`
  - body: The patch sends an IP URL to the remote scrape API, which may avoid local DNS rebinding but can break or misdirect normal HTTPS virtual-host scraping.
  - confidence: 0.88
