## Summary

Keep this PR open: the updated patch still has blocking security and compatibility issues, and the external contributor proof is only unit-test prose. Source inspection shows signed ACPX mode cannot run under ESM, ACPX authentication remains opt-in rather than tied to the launcher, and Firecrawl rewrites normal scrape URLs to IP URLs before handing them to the remote scrape API.

## What This Changes

The PR adds ACPX proxy payload signature and command-prefix checks with new ACPX tests, and changes Firecrawl scrape validation to async DNS checks plus IP-rewritten scrape URLs.

## Best Possible Solution

Implement ACPX authentication where the payload is produced and consumed with ESM-safe crypto and full payload binding, and scope Firecrawl SSRF protection to an enforceable service/self-hosted boundary without rewriting ordinary public scrape URLs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path for the PR findings: signed ACPX mode executes a CommonJS require inside .mjs, and Firecrawl rewrites a normal HTTPS hostname to an IP before posting the scrape request. I did not run tests because this was a read-only review.

## Solution Assessment

No. The proposed path is not the best fix because it leaves ACPX authentication optional and partly unwired, and it trades Firecrawl DNS-rebinding protection for a scrape URL rewrite that can break normal HTTPS virtual-hosted sites.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use ESM crypto in signed proxy mode:** `extensions/acpx/src/runtime-internals/mcp-proxy.mjs:23`
  - body: When OPENCLAW_MCP_PROXY_SECRET is set, this .mjs file calls require("node:crypto"). Node ESM modules do not define require, so both valid and invalid signed payloads fail before the intended signature check can run.
  - confidence: 0.94
- **[P1] Make ACPX payload signing enforceable:** `extensions/acpx/src/runtime-internals/mcp-proxy.mjs:12-13`
  - body: The new signing check returns when the secret is unset, and the PR does not add any production launcher path that signs the payload. In the default path, a tampered base64 payload is still accepted, while the prefix guard only checks the executable and not the arguments.
  - confidence: 0.86
- **[P1] Keep the Firecrawl scrape host intact:** `extensions/firecrawl/src/firecrawl-client.ts:555-556`
  - body: Replacing the requested hostname with the first resolved IP changes what Firecrawl is asked to scrape. For normal HTTPS sites behind shared hosting or CDNs, the remote request loses the original host/SNI target and can fail or fetch the wrong virtual host, so this breaks existing public scrape URLs.
  - confidence: 0.88
