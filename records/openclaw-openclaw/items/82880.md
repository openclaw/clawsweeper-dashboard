## Summary

Keep open: this PR touches real security-sensitive ACPX and Firecrawl surfaces, but the proposed protections are incomplete. The ACPX guard only checks an executable prefix, the Firecrawl DNS lookup is not tied to the scrape that Firecrawl performs, and the external PR proof is unit-test-only.

## What This Changes

The PR adds an ACPX proxy command-prefix guard, makes Firecrawl scrape target validation async with a shared DNS SSRF helper, and updates the Firecrawl scrape-target unit test to await the check.

## Best Possible Solution

Replace the prefix check with full-command or signed-payload validation for ACPX, and treat Firecrawl target DNS checking as enforceable only if the actual scrape request can be pinned or the service-side policy is confirmed; add targeted regression tests and real behavior proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path for the review findings: a tampered ACPX payload can preserve the allowed executable while changing arguments, and Firecrawl still sends the original target URL to the scrape API after discarding the DNS pin. I did not run live exploit proof because this was a read-only review.

## Solution Assessment

No. The current patch is not the best fix because it validates only target.command for ACPX and performs a local Firecrawl DNS precheck that is not bound to the later remote scrape; the safer direction is exact payload/command binding and an enforceable SSRF guard at the actual fetch boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Validate the full ACPX target command:** `extensions/acpx/src/runtime-internals/mcp-proxy.mjs:88`
  - body: The new guard only checks target.command with startsWith. A manipulated payload can keep the allowed executable, such as node, and replace the args with a different script or `-e` payload, so the command-injection path remains open. Validate the exact generated command or authenticate the full payload, including args, before spawning.
  - confidence: 0.88
- **[P1] Do not treat local DNS lookup as Firecrawl pinning:** `extensions/firecrawl/src/firecrawl-client.ts:121`
  - body: The resolved addresses are discarded, and runFirecrawlScrape still posts the original params.url to Firecrawl as JSON. For the hosted API, Firecrawl resolves and fetches that URL later, so rebinding or split-horizon DNS can still differ after this check, while DNS failures are allowed. The fix needs an enforceable service-side or self-hosted fetch guard, or this should be scoped as a best-effort precheck rather than rebinding protection.
  - confidence: 0.82
