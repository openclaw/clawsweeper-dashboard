- `merge-risk: 🚨 compatibility`: The Firecrawl IP rewrite can make existing hostname-based HTTPS scrape targets fail or fetch a different origin after merge.

## Summary

Keep this PR open: it is a plausible security-hardening direction, but the current patch still has merge-blocking Firecrawl compatibility and ACPX security-boundary problems, and the proof remains test-only rather than real runtime behavior.

## What This Changes

The PR adds optional ACPX MCP proxy payload HMAC verification, command-prefix checks, Firecrawl scrape URL DNS pinning by IP rewrite, and new ACPX/Firecrawl security tests.

## Best Possible Solution

Land a complete, documented security boundary: preserve normal Firecrawl target semantics unless an explicit strict mode is approved, and wire ACPX payload signing through the real producer with full command/argument binding and upgrade proof.

## Reproduction Assessment

Yes, by source inspection. The PR head shows Firecrawl replacing the scrape hostname before posting to Firecrawl, and ACPX verifying OPENCLAW_MCP_PROXY_SECRET without any production signing path in current main.

## Solution Assessment

No. The current approach is not the best fix because it changes Firecrawl target semantics and adds only the ACPX verifier half; the maintainable path is a complete producer/consumer signing contract plus a host-preserving or explicitly opt-in Firecrawl SSRF strategy.

Full review comments:

- **[P1] Preserve the Firecrawl scrape hostname:** `extensions/firecrawl/src/firecrawl-client.ts:555-556`
  - body: The patch rewrites `https://example.com/page` to an IP URL before sending it to Firecrawl. That changes HTTPS/SNI and virtual-host routing for normal scrape targets, so existing CDN or shared-host sites can fail or fetch the wrong origin; keep the original URL unless maintainers approve a documented strict mode.
  - confidence: 0.92
- **[P1] Wire ACPX signatures through the producer:** `extensions/acpx/src/runtime-internals/mcp-proxy.mjs:12-14`
  - body: `OPENCLAW_MCP_PROXY_SECRET` now makes the proxy reject unsigned payloads, but the real ACPX launch path does not sign payloads. Enabling the secret would break current launches, while leaving it unset gives no command-binding boundary; add producer-side signing for the canonical payload or remove the opt-in verifier.
  - confidence: 0.9
- **[P2] Bind command policy to arguments too:** `extensions/acpx/src/runtime-internals/mcp-proxy.mjs:118`
  - body: The allowed-prefix check only inspects `target.command`. A payload can keep an allowed interpreter or shell executable and move the dangerous behavior into `target.args`, so the policy should validate or sign the canonical full command and arguments rather than only the binary prefix.
  - confidence: 0.82
