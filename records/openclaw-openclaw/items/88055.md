## Summary

Keep this PR open, but do not merge it as-is: the fix targets the right session-store bug, yet the later same-run transcript persistence path still receives the original session id and can rewrite the store back to the stale id. The PR also still needs real behavior proof beyond tests and code inspection.

## What This Changes

The PR changes `src/agents/agent-command.ts` so the post-run session-store update uses `result.meta.agentMeta.sessionId` when present instead of always using the pre-run session id.

## Best Possible Solution

Carry one resolved post-run session identity through all subsequent session-store, transcript persistence, and compaction lifecycle calls after automatic rotation, then prove the store remains on the successor session after the whole post-run path completes.

## Reproduction Assessment

Yes from source: automatic compaction can rotate `sessionIdUsed`, but current main passes the original `sessionId` to the session-store updater. The PR's changed path is also source-reproducibly incomplete because later transcript persistence still receives the original id.

## Solution Assessment

No: using `agentMeta.sessionId` for only the first store update is too narrow because later post-run persistence still runs with the original session id. The better fix is to compute the effective post-rotation session identity once and use it consistently for the rest of the post-run persistence path.
