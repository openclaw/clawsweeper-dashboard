## Label Justifications

- `impact:message-loss`: The reported workflow loses in-flight visible progress messages from spawned subagent sessions because the child cannot call `message`.

## Summary

Keep this open: current `main` still intentionally disables the message tool for spawn-initiated native child runs, and there is no existing config/schema path that can opt it back in. This is a real source-proven request, but it asks for a new message-delivery/config contract that needs maintainer product review rather than an automatic cleanup close.

## What This Changes

The issue requests an explicit opt-in way for spawned subagent sessions to regain the `message` tool while keeping the current default disabled.

## Best Possible Solution

Keep the safe default disabled, and if maintainers accept the feature, add one explicit opt-in contract at the spawn/tool-construction boundary with schema, docs, and regression coverage for default-off plus opt-in behavior.

## Reproduction Assessment

Yes. Source inspection reproduces the issue: `src/agents/subagent-spawn.ts` passes `disableMessageTool: true`, and `src/agents/openclaw-tools.ts` skips constructing `message` when that flag is set, so allowlists cannot restore the absent tool.

## Solution Assessment

Unclear. The requested direction is plausible, but maintainers need to choose whether the public contract should be a new subagent config key, explicit `tools.subagents.tools.allow` semantics, or no opt-in at all for safety.
