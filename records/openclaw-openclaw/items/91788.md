## Label Justifications

- `P2`: This is a bounded capability request affecting long-running multi-agent status workflows, without evidence of a core outage or broad regression.
- `impact:message-loss`: Spawned agents cannot emit requested in-flight status messages because the message tool is removed before their configured tool policy runs.

## Summary

Keep open for maintainer product judgment: current main and v2026.6.5 intentionally omit the message tool from spawned subagents, and no canonical issue or PR now owns an opt-in; the narrowest viable direction is to decide whether an explicit existing subagent tool allow entry may reopen this trusted delivery capability.

## What This Changes

Request an explicit opt-in that restores the message tool for spawned subagent sessions while keeping it disabled by default.

## Best Possible Solution

Use one canonical explicit policy: if maintainers approve the trust model, let an explicit message entry in the existing subagent tool policy restore construction while preserving default disablement, documenting targeting and completion-handoff behavior, and covering both opt-in and default paths.

## Reproduction Assessment

Yes, at source level with high confidence: spawned runs set disableMessageTool, and the tool factory consequently omits createMessageTool before the existing subagent allow/deny filters can apply. This read-only review did not run a live channel scenario.

## Solution Assessment

No to adding a separate messageTool configuration key; reusing the existing explicit subagent tool policy is the narrower maintainable shape. Whether that policy should be allowed to reopen outbound messaging, and under which delivery constraints, requires maintainer approval.
