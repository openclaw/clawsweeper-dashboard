## Summary

Keep open: current main and the latest release still intentionally prevent spawned native subagent child runs from constructing the `message` tool, and the request remains a product/security decision rather than already-implemented cleanup.

## What This Changes

The issue requests an explicit opt-in configuration path that lets spawned subagent child sessions use the `message` tool again while keeping the current default disabled.

## Best Possible Solution

Maintainers should choose one supported child-progress contract: either add a narrow explicit message-tool opt-in with schema, docs, default-disabled and opt-in tests, and security review, or close this direction in favor of a parent-only session-send API.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: the spawn path passes `disableMessageTool: true`, and the tool factory does not construct `message` under that flag. I did not run a live Slack progress-card scenario in this read-only review.

## Solution Assessment

Unclear pending maintainer direction. The requested opt-in is plausible and fits trusted operator-controlled workflows, but a narrower parent-scoped communication API may be safer than direct child-to-channel messaging.
