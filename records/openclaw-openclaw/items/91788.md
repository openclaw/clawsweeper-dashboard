## Summary

Keep open: current main and the latest release still hard-disable the message tool for native spawned subagents, and the request would add or reinterpret a trusted message-delivery config surface that needs maintainer product judgment.

## What This Changes

The issue requests an explicit opt-in path to let spawned subagent sessions construct and use the `message` tool again while keeping the current default disabled.

## Best Possible Solution

Keep the safe default-off behavior unless maintainers approve a narrow trusted opt-in, then implement it through the canonical config/schema, spawn gateway call, docs, and default-off plus explicit-opt-in tests.

## Reproduction Assessment

Yes, from source inspection: spawned native child runs pass `disableMessageTool: true`, and `createOpenClawTools` skips `createMessageTool` when that flag is set. I did not run a live Slack/channel progress-card scenario.

## Solution Assessment

Unclear: an explicit opt-in knob is plausible, but repository policy sets a high bar for new config and message-delivery defaults. Maintainers should decide whether the permanent contract is a subagent config field, honoring an explicit `tools.subagents` allow entry, or keeping parent-only delivery.
