## Summary

Current main intentionally removes the message tool from spawned child runs, and the documented subagent allowlist cannot restore it. The use case is credible, but re-enabling direct child delivery would alter the deliberate orchestrator-owned completion handoff and therefore needs an explicit product and safety decision.

## What This Changes

Requests an explicit opt-in that lets spawned subagent sessions construct and use the message tool while keeping it disabled by default.

## Best Possible Solution

Preserve message suppression by default, but if maintainers approve trusted child-originated progress delivery, express it through one explicit subagent permission path, require explicit targets, keep completion delivery owned by the orchestrator, and cover both opt-in progress sends and duplicate/misroute prevention.

## Reproduction Assessment

Yes, from source with high confidence: every spawned child kickoff disables message-tool construction, and the documented subagent allowlist cannot add a tool back after it has been omitted. The exact Slack progress-card scenario was not run live, so this is source-reproducible rather than an executed current-main reproduction.

## Solution Assessment

Unclear pending product judgment. An opt-in is reasonable, but the narrowest implementation should reuse the existing subagent permission surface if possible and preserve orchestrator-owned completion delivery rather than introducing a competing delivery path.
