## Label Justifications

- `P2`: The issue reports a bounded but real loss of in-flight channel progress updates for spawned subagent workflows.
- `impact:message-loss`: The affected workflow loses child-session progress/status messages that previously could be posted directly to a channel.

## Summary

Keep open: current main still unconditionally disables the message tool for spawn-started child runs, and the config/docs surface has no opt-in path to restore it. This is a real source-level gap, but granting child agents direct message delivery is a new config and safety contract that needs maintainer product review before implementation.

## What This Changes

Request an explicit opt-in config or policy path that lets spawned subagent sessions construct and use the `message` tool while keeping the default disabled.

## Best Possible Solution

Choose one disabled-by-default opt-in contract, wire it through config schema, spawn params, tool construction, docs, and tests, and keep parent-owned completion delivery as the default.

## Reproduction Assessment

Yes, at source level: current main unconditionally starts child runs with `disableMessageTool: true`, and tool construction returns no `message` tool when that flag is set. I did not run a live Slack progress-card workflow.

## Solution Assessment

Unclear: an explicit opt-in is plausible, but maintainers need to choose whether the durable contract is a new subagent config key or changed semantics for explicit subagent tool allowlists.
