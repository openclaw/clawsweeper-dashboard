## Summary

Keep open: current main and the latest release still hard-disable the message tool for native spawned subagents, and there is no existing config/schema path that restores it. Because enabling it would add or reinterpret a trusted message-delivery config surface, this needs maintainer product judgment rather than auto-close or an automatic fix PR.

## What This Changes

The issue requests an explicit opt-in config path that lets spawned native subagent sessions receive the message tool again while keeping the current disabled default.

## Best Possible Solution

Keep the safe default-off behavior, then add a narrow documented trusted-operator opt-in only if maintainers decide child sessions may own direct progress delivery; the implementation should cover schema/types, spawn policy, docs, and default-off plus explicit-opt-in tests.

## Reproduction Assessment

Yes, from source inspection: spawned native child runs pass disableMessageTool: true, and createOpenClawTools skips createMessageTool when that flag is set. I did not run a live Slack or channel progress-card scenario.

## Solution Assessment

Unclear: an explicit opt-in knob is plausible, but repository policy sets a high bar for new config and message-delivery defaults. Maintainers should first decide whether the permanent contract is a subagent config field, honoring an explicit tools.subagents allow entry, or keeping parent-only delivery.
