## Summary

Current main still unconditionally omits the `message` tool from every spawned child kickoff, so the reported use case remains unsupported; keep this open for a maintainer decision because restoring direct child delivery is aligned with operator-controlled capability but changes an intentional parent-owned handoff boundary.

## What This Changes

Add an explicit opt-in that lets spawned subagent sessions use the `message` tool while preserving the disabled default.

## Best Possible Solution

Make an explicit `message` entry in `tools.subagents.tools.allow` or `alsoAllow` permit message-tool construction for spawned runs, keep the current disabled default, and document and test both default and opted-in delivery behavior.

## Reproduction Assessment

Yes. Current source deterministically passes `disableMessageTool: true` for every spawned child kickoff, and tool assembly omits `message` before the configured subagent allowlist is applied.

## Solution Assessment

No to the separate config-key example, but yes to the underlying opt-in direction: the narrowest maintainable solution is to make the existing explicit subagent tool policy control construction after maintainers approve the capability boundary.
