## Summary

Keep open: current main and v2026.6.8 still launch spawned native subagent child runs with the message tool disabled, and the docs describe that as the current delivery model. The request is valid source-level product work, but it needs maintainer approval because it adds config surface and changes child-session outbound messaging authority.

## What This Changes

The issue requests an explicit opt-in path that lets spawned subagent sessions construct and use the message tool again while keeping the current disabled default.

## Best Possible Solution

Keep the secure disabled default, and add a single maintainer-approved explicit opt-in only if it includes schema/types/docs plus tests for default-disabled behavior and explicit child message-tool availability.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 launch spawned native child runs with `disableMessageTool: true`, and `openclaw-tools` does not construct `message` when that flag is set. I did not run a live spawned-subagent Slack progress scenario in this read-only review.

## Solution Assessment

Unclear pending maintainer direction. The requested explicit opt-in is plausible and aligned with operator-controlled high-power workflows, but the safest shape may be a narrower parent-scoped progress API such as https://github.com/openclaw/openclaw/issues/18967 rather than direct child-to-channel messaging.
