## Summary

Keep open: current main and v2026.6.9 still keep spawned native subagents from constructing the `message` tool, and the requested opt-in remains a product/security decision rather than implemented cleanup.

## What This Changes

Request an explicit, default-disabled config path that lets spawned native subagent sessions regain the `message` tool for in-flight progress updates.

## Best Possible Solution

Maintainers should choose one supported child-progress contract: add a narrow explicit default-disabled message-tool opt-in with schema, docs, security tests, and live proof, or close this direction in favor of a parent-only session-send API.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: spawned child runs pass `disableMessageTool: true`, and the tool factory does not construct `message` under that flag; I did not run a live Slack progress-card scenario in this read-only review.

## Solution Assessment

Unclear pending maintainer direction. The requested opt-in is plausible for trusted operator-controlled workflows, but a parent-scoped communication API may be the safer permanent contract than direct child-to-channel messaging.
