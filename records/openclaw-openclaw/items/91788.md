## Summary

Keep open: current main and v2026.6.5 still hard-disable the message tool for native spawned subagents, and no existing config/schema/docs path or duplicate implementation PR restores it. The request is still necessary if maintainers want this capability, but it is a new trusted child-session message-delivery contract rather than a strict bug fix.

## What This Changes

The issue requests an explicit opt-in to make the `message` tool available again inside spawned native subagent sessions while keeping the current default disabled.

## Best Possible Solution

Keep the safe default-off behavior, and if maintainers approve the capability, add one narrow documented trusted-operator opt-in with schema/types, spawn policy, docs, and default-off plus explicit-opt-in tests.

## Reproduction Assessment

Yes, from source inspection: spawned native child runs pass `disableMessageTool: true`, and `createOpenClawTools` skips `createMessageTool` when that flag is set. I did not run a live Slack progress-card scenario.

## Solution Assessment

Unclear until maintainers choose the product contract. An explicit default-off opt-in is plausible, but repository policy sets a high bar for new config and direct message-delivery surfaces.
