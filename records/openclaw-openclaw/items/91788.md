## Label Justifications

- `P2`: The issue blocks a legitimate but specialized long-running multi-agent progress workflow while final completion delivery remains functional.
- `impact:message-loss`: Spawned sessions are structurally unable to emit operator-requested in-flight status messages.

## Summary

The issue remains necessary: current main deliberately removes the `message` tool from every spawned child, and no existing configuration can restore a tool that was never constructed. This is a valid advanced-workflow request, but enabling child-originated channel sends changes the completion-delivery and security boundary, so maintainers must choose the explicit capability contract before implementation.

## What This Changes

Add an opt-in setting that restores the `message` tool for spawned subagent sessions while keeping it disabled by default.

## Best Possible Solution

Preserve the disabled default and add one explicit spawn-level capability, with constrained target semantics and focused tests proving progress sends work without duplicating or bypassing the parent’s final handoff.

## Reproduction Assessment

Yes, at source level with high confidence: every spawned child kickoff sets `disableMessageTool: true`, tool construction then omits `message`, and focused lifecycle coverage asserts that behavior. This read-only review did not run a live Slack scenario.

## Solution Assessment

The opt-in outcome is reasonable, but the issue’s alternatives are not equivalent. A dedicated spawn capability is safer and clearer than silently changing `tools.subagents.tools.allow` from a filter into a tool-construction override.
