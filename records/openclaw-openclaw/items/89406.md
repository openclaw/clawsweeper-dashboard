## Summary

Keep open: this behavior is not on current main or the latest release, and the PR is a maintainer-authored draft with a protected label. The patch looks like a focused, plausible fix, but the per-tool quarantine policy is compatibility-sensitive and needs explicit maintainer sign-off before merge.

## What This Changes

The PR snapshots extension tool descriptors during `registerTool`, skips unreadable tool registrations with diagnostics, and forwards those diagnostics through path-based and inline extension loading.

## Best Possible Solution

Land this only after maintainers confirm that per-tool quarantine with visible diagnostics is the intended extension registration contract.

## Reproduction Assessment

Yes, from source: current main and v2026.6.6 read `tool.name` directly during `registerTool`, so a throwing getter can abort extension loading. I did not run tests because this review is read-only.

## Solution Assessment

Yes, with a policy caveat: snapshotting at registration is the narrow owner-boundary fix because registry, prompt, and wrapper paths read tool fields later. The unresolved question is whether maintainers want partial extension loading instead of fail-fast behavior for malformed descriptors.
