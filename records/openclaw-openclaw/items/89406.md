## Summary

Keep open: current main and v2026.6.6 still use fail-fast/raw extension tool registration, while this PR adds a focused per-tool quarantine path. The branch has no concrete line-level defect from this review, but it is a MEMBER-authored draft with the protected `maintainer` label and an intentional compatibility policy change, so the next step is maintainer review rather than cleanup close.

## What This Changes

The PR snapshots registered extension tool descriptors, skips unreadable tool registrations with diagnostics, and propagates those diagnostics through path-based and inline extension loading.

## Best Possible Solution

Land this only after maintainer review confirms that per-tool quarantine with visible diagnostics is the desired extension registration contract.

## Reproduction Assessment

Yes from source: current main and v2026.6.6 read `tool.name` directly during `registerTool`, so a throwing getter aborts extension loading. I did not run tests because this review was read-only.

## Solution Assessment

Yes, with a policy caveat: snapshotting at registration is the narrow owner-boundary fix because downstream registry, prompt, and wrapper paths read tool fields later. The unresolved question is whether maintainers want partial extension loading instead of fail-fast behavior for malformed tool descriptors.
