The patch looks like a focused fix, but it intentionally changes unreadable extension tool descriptors from extension-level failure to per-tool quarantine, so human maintainer review should confirm that compatibility policy before merge.

## What This Changes

The branch snapshots registered extension tools, skips unreadable descriptors with diagnostics, and adds loader coverage for preserving healthy tools and commands.

## Best Possible Solution

Land this only after maintainer review confirms per-tool quarantine with visible diagnostics is the desired long-term extension registration contract.

## Reproduction Assessment

Yes from source: on current main `registerTool` reads `tool.name` directly and `loadExtension` turns factory errors into extension load failures, so a throwing descriptor getter can fail the extension. I did not execute the test because the checkout is read-only, so this is source-reproducible rather than locally reproduced.

## Solution Assessment

Yes, with one policy caveat: snapshotting at registration is the narrow owner-boundary fix because downstream registry, prompt, and schema paths read tool definition fields later. The remaining question is whether maintainers want partial extension loading instead of fail-fast behavior for malformed tool descriptors.
