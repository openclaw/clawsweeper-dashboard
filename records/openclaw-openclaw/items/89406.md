## What This Changes

The PR snapshots extension tool descriptors during `api.registerTool(...)`, skips unreadable tool registrations with diagnostics, and keeps readable sibling tools and commands available.

## Best Possible Solution

Land this only after maintainer review confirms per-tool quarantine with visible diagnostics is the desired extension registration contract.

## Reproduction Assessment

Yes from source: current main and `v2026.6.6` read `tool.name` directly during `registerTool`, so a throwing getter aborts extension loading. I did not run tests because this was a read-only review.

## Solution Assessment

Yes, with a policy caveat: snapshotting at registration is the narrow owner-boundary fix because downstream registry, prompt, and wrapper paths read tool fields later. The remaining question is whether maintainers want partial extension loading instead of fail-fast behavior for malformed trusted extension descriptors.
