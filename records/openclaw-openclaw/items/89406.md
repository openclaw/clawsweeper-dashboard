## Summary

Keep this PR open. Current main still has the fail-fast extension `registerTool` path, and the patch appears technically sound, but it intentionally changes malformed extension tool handling from extension-level failure to per-tool quarantine, so it needs maintainer acceptance before merge.

## What This Changes

The branch snapshots extension-registered tool definitions, skips unreadable descriptors with loader diagnostics, preserves healthy sibling tools and commands, and adds focused loader coverage.

## Best Possible Solution

Land this only after maintainer review confirms per-tool quarantine with visible diagnostics is the desired extension registration contract, or consolidate it into the broader session-tool snapshot path without losing the diagnostic behavior.

## Reproduction Assessment

Yes from source: current main reads `tool.name` directly during `registerTool` and catches factory errors as extension-load failures, so a throwing descriptor getter can stop later healthy registrations. I did not execute the test in this read-only review.

## Solution Assessment

Yes, with a policy caveat: registration-time snapshotting is the narrow owner-boundary fix for delayed descriptor getter failures. The unresolved question is whether maintainers want partial extension loading as the permanent contract, especially with the broader session snapshot work still open.
