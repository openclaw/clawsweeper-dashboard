## Summary

Keep open: the PR is a focused hardening patch for extension tool registration and current main still has the direct-read failure path, but it is a maintainer-authored draft with a protected label and an intentional compatibility-sensitive change from extension-level failure to per-tool quarantine.

## What This Changes

The PR snapshots extension tool definitions during registration, skips descriptors whose fields throw, emits loader diagnostics, and adds a loader regression test.

## Best Possible Solution

Land this only after maintainers confirm that per-tool quarantine with visible diagnostics is the desired extension registration contract.

## Reproduction Assessment

Yes from source: current main and v2026.6.6 read tool.name directly inside registerTool, so a throwing name getter fails extension loading before later registrations. I did not run tests because this review is read-only.

## Solution Assessment

Yes, with a policy caveat: snapshotting at registration is the narrow owner-boundary fix because downstream registry and wrapper paths read tool fields later. The remaining question is whether maintainers want partial extension loading instead of fail-fast behavior for malformed tool descriptors.
