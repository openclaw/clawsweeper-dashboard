## Summary

Keep open: the patch addresses a real current-main extension-loader failure path, but this is a maintainer-authored draft with a protected label and an intentional compatibility-sensitive change from extension-level failure to per-tool quarantine.

## What This Changes

The PR snapshots registered extension tool descriptors, skips unreadable tool registrations with diagnostics, and propagates those diagnostics through path-based and inline extension loading.

## Best Possible Solution

Land only after maintainer review confirms per-tool quarantine with visible diagnostics is the desired extension registration contract.

## Reproduction Assessment

Yes from source: current main and v2026.6.6 read tool.name directly during registerTool, so a throwing descriptor getter can abort extension loading. I did not run tests because this was a read-only review.

## Solution Assessment

Yes, with a policy caveat: snapshotting at registration is the narrow owner-boundary fix because downstream registry, prompt, and wrapper paths read tool fields later. The unresolved question is whether maintainers want partial extension loading instead of fail-fast behavior for malformed tool descriptors.
