## Summary

Current main now covers the reported post-tool Codex app-server stall with owner-local liveness handling: after an app-server tool response, OpenClaw interrupts and releases the run if Codex never produces `turn/completed`, while same-turn activity is tracked separately under a longer terminal watchdog.

## What This Changes

Report that forced native Codex runtime can hang after a tool-call response and follow-up sampling, leading to websocket idle timeout and high host CPU.

## Best Possible Solution

Keep the current main Codex app-server watchdog/liveness implementation, and ask for a fresh current-main report with sanitized trajectory logs if the same Feishu flow still hangs after this fix reaches a release.

## Reproduction Assessment

Yes, source and regression coverage give a high-confidence reproduction for the failure class: return a dynamic `item/tool/call` response, omit `turn/completed`, and current main interrupts/releases the run instead of staying stuck. I did not run the original Feishu live path.

## Solution Assessment

Yes, this is the narrow maintainable fix in the owning Codex app-server runner: OpenClaw still does not drive OpenAI Responses `function_call_output` directly, but it now bounds the post-tool app-server gap and releases the OpenClaw session with a clear timeout.
