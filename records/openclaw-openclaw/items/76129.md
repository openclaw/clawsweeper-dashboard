## Summary

Keep open. The PR is opted into automerge, but it introduces a blocking repair-mode regression: a plain diagnostic `openclaw doctor` can now install configured plugins and stamp config metadata without `--fix`.

## What This Changes

The PR adds a one-time 2026.5.2 doctor migration for configured downloadable plugin installs, ClawHub-first install fallback behavior, upgrade-survivor coverage, docs, and a changelog entry.

## Best Possible Solution

Keep the migration, but make the install/stamp path repair-only and leave normal doctor runs as preview/diagnostic output.

## Reproduction Assessment

Yes. Current main runs doctor health contributions unconditionally, and the PR adds the installer call inside that path without a `shouldRepair` guard, so the issue is reproducible by invoking `openclaw doctor` on a pre-2026.5.2 config with configured downloadable plugin usage.

## Solution Assessment

No. The migration direction is reasonable, but the implementation should preserve the existing doctor boundary by running package-manager/plugin install side effects only under `openclaw doctor --fix` or equivalent repair mode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate release installs behind repair mode:** `src/flows/doctor-health-contributions.ts:279-285`
  - body: `openclaw doctor` runs health contributions even when the user did not pass `--fix`, but this new contribution calls the release repair unconditionally. Please skip this path unless `ctx.prompter.shouldRepair` is true, or make the non-repair path preview-only.
