## Summary

Keep open: the latest PR head still has blocking behavior because a normal doctor health run can install configured plugin packages and stamp config without repair mode, and the upgrade-survivor ClawHub override path can still fail on a missing local fixture log.

## What This Changes

This PR adds a one-time 2026.5.2 doctor migration for configured downloadable plugin installs, ClawHub/npm repair behavior, upgrade-survivor coverage, docs, and changelog updates.

## Best Possible Solution

Keep the migration direction, but make install and stamp side effects repair-only while keeping normal doctor diagnostic, then make the ClawHub request assertion conditional on owning the local fixture.

## Reproduction Assessment

Yes. The issue is reproducible from static code inspection: current main runs doctor health contributions unconditionally, and the PR adds installer and config-stamp side effects to that path without a repair-mode guard; the fixture override failure follows from the early return and unconditional grep.

## Solution Assessment

No. The migration is useful, but the implementation should preserve the existing doctor boundary by running package-manager/plugin install side effects only under openclaw doctor --fix or an explicit repair/update mode.

In override environments the file is never created, so the scenario fails even when doctor correctly attempts ClawHub through the external URL.
  - confidence: 0.9
