## Summary

Keep open. The PR targets a real Telegram adapter-level gap, but the patch is not landable as written because the early silent-token guard ignores `opts.buttons` and can drop inline keyboard payloads; it also lacks the required changelog entry for a user-facing Telegram fix. Current main already normalizes ordinary Telegram delivery, but direct low-level adapter sends can still reach the Bot API with exact `NO_REPLY`, so the useful path is a narrower Telegram-owned guard with focused tests.

## What This Changes

This PR imports `isSilentReplyText` in the Telegram sender and adds an early no-op return for exact `NO_REPLY` text when `mediaUrl` is absent.

## Best Possible Solution

Land a Telegram-owned defense-in-depth guard that suppresses exact silent text only when there is no media and no button/reply-markup payload, while keeping shared outbound normalization as the primary delivery fix. Add focused `sendMessageTelegram` regression coverage and a one-line changelog entry for the user-facing Telegram bug fix.

## Reproduction Assessment

Yes. Current main can be checked with a mocked direct `sendMessageTelegram("123", "NO_REPLY", { cfg, token, api })` call; code inspection shows it reaches the `api.sendMessage` text path, while the PR variant would return early and also suppress button-bearing sends.

## Solution Assessment

No. The adapter-level guard is a reasonable direction, but this implementation is too broad because it ignores `opts.buttons`; the safer fix should suppress only text-only silent sends and preserve inline keyboards.
