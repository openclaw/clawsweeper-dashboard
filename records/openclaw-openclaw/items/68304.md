## Summary

Keep open. The Telegram adapter-level silent-reply guard is a valid bug fix direction, but this PR should not merge as-is because the guard would also suppress inline-keyboard sends and the patch lacks the required changelog and regression coverage.

## What This Changes

The PR imports `isSilentReplyText` into `sendMessageTelegram` and suppresses exact `NO_REPLY` payloads before Telegram API resolution when `opts.mediaUrl` is absent.

## Best Possible Solution

Land a Telegram-owned adapter guard that suppresses exact silent text only when there is no media and no sendable inline keyboard payload, with focused sender tests and a single-line Unreleased changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main sends a direct `sendMessageTelegram("123", "NO_REPLY", { cfg, token, api })` call through `api.sendMessage`, while the PR would suppress the same text even when `opts.buttons` is present.

## Solution Assessment

No. The guard direction matches the existing Slack defense, but Telegram's structured payload equivalent is `buttons`/`reply_markup`, so the condition must preserve button-bearing sends instead of checking only `mediaUrl`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88
