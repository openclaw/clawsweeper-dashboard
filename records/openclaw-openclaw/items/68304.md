## Summary

Keep open. The PR targets a real Telegram adapter gap linked to #38603, but the guard would also suppress inline keyboard payloads and the user-facing fix still needs changelog coverage before merge.

## What This Changes

The PR imports `isSilentReplyText` into the Telegram sender and returns a suppressed no-op for exact silent text when no media URL is present.

## Best Possible Solution

Land a Telegram-owned guard that suppresses exact silent text only when there is no media and no sendable inline keyboard, with focused sender tests and an Unreleased Fixes changelog entry.

## Reproduction Assessment

Yes, by source inspection. On current main, a mocked `sendMessageTelegram("123", "NO_REPLY", { cfg, token, api })` follows the text path to `api.sendMessage`; with this PR, the same call carrying `buttons` would return `suppressed` before `reply_markup` is sent.

## Solution Assessment

No. The guard is in the right owner module, but the condition is too broad because it ignores `opts.buttons`; the safer fix should preserve button-bearing payloads and add tests plus changelog coverage.
