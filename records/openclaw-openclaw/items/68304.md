## Summary

Keep this PR open, but it still needs changes before merge: the adapter-level silent-token guard is useful, yet the current condition would suppress Telegram inline-keyboard sends and the user-facing fix lacks the required changelog entry.

## What This Changes

The PR imports `isSilentReplyText` into the Telegram sender and returns a suppressed no-op before resolving Telegram API context when the text is an exact silent reply token and no media URL is present.

## Best Possible Solution

Land a Telegram-owned guard that suppresses exact silent text only when there is no media, no inline keyboard, and no other sendable payload, with focused sender regression tests and a single-line Unreleased changelog entry.

## Reproduction Assessment

Yes. On current main, a mocked `sendMessageTelegram("123", "NO_REPLY", { cfg, token, api })` follows the text path to `api.sendMessage`; with this PR, the same call plus `buttons` would return `suppressed` before sending `reply_markup`.

## Solution Assessment

No. The adapter-level guard is the right direction, but the proposed condition is too broad because it ignores Telegram buttons; the safer fix should mirror Slack's structured-payload exception and test both suppression and button preservation.
