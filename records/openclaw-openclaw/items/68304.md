## Summary

Keep open. Current main still lacks a low-level `sendMessageTelegram` silent-token guard, so the PR addresses a real adapter gap linked to #38603, but the patch is not landable as written because it can suppress inline-keyboard sends and it lacks the required changelog entry for a user-facing Telegram fix.

## What This Changes

The PR imports `isSilentReplyText` into the Telegram sender and returns a suppressed no-op before Telegram API resolution when trimmed text is a silent reply token and `opts.mediaUrl` is absent.

## Best Possible Solution

Land a Telegram-owned defense-in-depth guard that suppresses exact silent text only when there is no media and no inline keyboard or other sendable payload, with focused sender regression tests and a single-line Unreleased changelog entry.

## Reproduction Assessment

Yes. On current main, a direct mocked `sendMessageTelegram("123", "NO_REPLY", { cfg, token, api })` follows the text path to `api.sendMessage`; with this PR, the same call plus `buttons` would return `suppressed` before sending `reply_markup`.

## Solution Assessment

No. The direction is reasonable, but the condition is too broad because it ignores `opts.buttons`; the safer patch should preserve button-bearing payloads and add focused tests plus changelog coverage.
