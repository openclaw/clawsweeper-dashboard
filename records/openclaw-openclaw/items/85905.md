## Label Justifications

- `P2`: The bug affects a real WebChat media workflow but is limited to HEIC/HEIF image handling rather than core runtime availability.
- `impact:message-loss`: The uploaded image can be absent from the effective agent prompt or replaced by only a media placeholder.
- `impact:session-state`: The report describes the agent using the previous image description, which is conversation context drift.

## Summary

Keep open: current main still has HEIC/HEIF gaps in the WebChat image path. The related closed work covers direct `input_image`/prompt-image and iMessage normalization, but WebChat inline images and media-understanding image descriptions can still carry raw HEIC bytes into provider calls.

## What This Changes

The issue requests reliable HEIC/HEIF normalization and `[Image]` description injection for WebChat/Control UI uploads handled by `tools.media.image`.

## Best Possible Solution

Normalize HEIC/HEIF once in OpenClaw-owned WebChat/media image handling before any provider call, then cover both inline native images and `ctx.MediaPaths` media-understanding descriptions with focused regressions.

## Reproduction Assessment

Yes from current-main source inspection, though not from a live HEIC upload run. A WebChat HEIC can remain raw in `parsedImages` for vision-capable sessions, and text-only media-understanding can pass raw `image/heic` bytes to `describeImage`.

## Solution Assessment

Yes, the maintainable fix is to reuse or centralize HEIC/HEIF normalization inside the OpenClaw media pipeline. Relying on external filesystem conversion leaves both WebChat and provider behavior timing-dependent.
