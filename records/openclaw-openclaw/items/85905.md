## Label Justifications

- `P2`: This is a normal-priority WebChat media bug with a bounded surface: image context can be lost or stale, but it is not a crash, security bypass, or global outage.
- `impact:message-loss`: The reported failure drops the current image description from the agent-visible message context.
- `impact:session-state`: The report includes the agent describing a previous image, which is session context drift for media turns.

## Summary

Keep open: current main and the latest release still pass HEIC/HEIF media-understanding image buffers and MIME types through unchanged, while the documented WebChat text-only path expects image attachments to be preserved for the configured image model and injected as an image summary.

## What This Changes

The issue asks OpenClaw to normalize HEIC/HEIF WebChat image attachments in the tools.media.image/media-understanding path so the current image description is injected into agent context.

## Best Possible Solution

Normalize HEIC/HEIF in the media-understanding attachment buffer path before describeImage, preserve existing media safety limits, and add regression coverage proving the current WebChat image becomes the injected [Image] description.

## Reproduction Assessment

Yes for a source-level reproduction path: WebChat text-only image offloads become MediaPaths, the media-understanding cache reads the original buffer/MIME, and the provider handoff sends that MIME unchanged. No live HEIC upload was replayed during this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to reuse or mirror the existing HEIC-to-JPEG normalization at the media-understanding attachment boundary before the image provider call. Relying on external filesystem conversion or symlink timing would keep the race-prone behavior.
