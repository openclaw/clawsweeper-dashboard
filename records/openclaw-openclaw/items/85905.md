## Label Justifications

- `P2`: The bug affects a real WebChat media workflow but is scoped to HEIC/HEIF image handling rather than core runtime availability.
- `impact:message-loss`: The uploaded image can be missing from the effective agent prompt or reduced to a placeholder when HEIC normalization is skipped.
- `impact:session-state`: The report describes previous-image context being used for the current turn, which is conversation context drift.

## Summary

Keep open: current main still has a source-visible HEIC/HEIF gap in the WebChat and media-understanding image paths. The related fixes at https://github.com/openclaw/openclaw/issues/50081 and https://github.com/openclaw/openclaw/issues/17670 cover other HEIC paths, but this issue still has a distinct raw `image/heic` provider handoff risk.

## What This Changes

The issue asks OpenClaw to normalize WebChat HEIC/HEIF uploads before `tools.media.image` provider calls and reliably inject the resulting image description or native image into the agent turn.

## Best Possible Solution

Normalize HEIC/HEIF to JPEG inside OpenClaw-owned WebChat/media image handling before any provider call, using a shared conversion path and focused regressions for inline images and offloaded `ctx.MediaPaths`.

## Reproduction Assessment

Yes, source-reproducible from current main: inline WebChat HEIC images can remain raw in `parsedImages`, and offloaded media-understanding can pass raw `image/heic` bytes to `describeImage`. I did not run a live HEIC upload.

## Solution Assessment

Yes. The narrow maintainable fix is OpenClaw-owned HEIC/HEIF normalization before native prompt-image delivery or image-description provider calls; relying on an external filesystem converter is timing-dependent.
