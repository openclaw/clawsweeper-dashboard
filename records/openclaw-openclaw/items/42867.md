## Summary

Keep open. Current main has shipped the heartbeat-aware daily/idle freshness behavior, but it still has no normal-session max-turn TTL config, schema, durable counter, or reuse-gate enforcement; the remaining TTL work overlaps the same author's open #45390 and needs maintainer API/persistence direction before a narrow fix is safe.

## What This Changes

Requests configurable automatic session rotation after a maximum number of conversation turns, plus heartbeat/system-event handling that does not keep abandoned sessions fresh.

## Best Possible Solution

Define one core session TTL contract across this issue and #45390, then add schema-validated and documented max-turn rotation that counts real conversation turns, excludes maintenance/system-event traffic, persists or derives enough state to enforce the limit before session reuse, and preserves the shipped heartbeat-aware daily/idle freshness behavior.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: the proposed max-turn config is absent and strict-rejected, the freshness evaluator has no turn-count input, and persisted session rows have no counter to enforce turn-count TTL.

## Solution Assessment

Unclear. The shipped heartbeat freshness implementation is the right solution for the second half, but the remaining max-turn behavior should be unified with #45390 before choosing the public config location, counter semantics, and any memory-save hook behavior.
