## Summary

Keep open: current main does not implement the suppress-hook recovery fallback, and this PR is still not merge-ready because of blocking correctness, proof, and security-scope concerns.

## What This Changes

The PR adds a one-attempt hook-suppressed context-overflow retry in the embedded PI runner, adds context-window stop-reason normalization tests, and changes Google Chat/memory-core peer dependency lockfile state.

## Best Possible Solution

Land a rebased replacement that suppresses only shrinkable hook-provided context when useful, preserves policy/system hook effects, gates retries by loop budget and hook availability, and drops unrelated dependency churn.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows current main can still reach terminal context_overflow after compaction/tool-result truncation while always resolving prompt-build hooks, but the PR does not prove a real memory-hook overflow recovery path.

## Solution Assessment

No. The fallback direction is plausible, but this implementation suppresses the full prompt-build hook result, retries without hook-availability or loop-budget guards, and mixes unrelated dependency churn into the branch.
