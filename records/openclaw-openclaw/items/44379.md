## Summary

Keep open: the hook-overflow recovery problem is still plausible on current main, but this PR needs changes before merge because it bypasses more hook behavior than intended, schedules a retry when it may not help, and carries stale package/lockfile changes.

## What This Changes

The PR adds a one-attempt prompt-hook suppression fallback after context-overflow recovery, normalizes context-window stop-reason handling, updates targeted tests, and changes plugin peer dependency plus lockfile metadata.

## Best Possible Solution

Rebase or replace the PR with a narrow runner fix that suppresses only hook-provided context for one eligible retry, preserves non-context hook effects, gates the retry on hook availability and loop budget, drops unrelated package/lockfile changes, and keeps focused regression coverage plus changelog.

## Reproduction Assessment

No high-confidence executable reproduction was established in this read-only review. Source inspection shows the plausible failure path: current main always applies prompt-build hook output before attempts, and overflow recovery can terminate after compaction/truncation without a hook-suppressed retry.

## Solution Assessment

No. The direction is plausible, but the PR is not the best current fix because it suppresses the entire prompt-build hook result, retries even when no hook or loop budget can make progress, and includes stale unrelated package-manager changes.

- **[P2] Preserve non-context hook effects during recovery:** `src/agents/pi-embedded-runner/run/attempt.ts:2314-2324`
  - body: The suppress flag skips `resolvePromptBuildHookResult` entirely, so the recovery attempt loses `systemPrompt` overrides and legacy `before_agent_start` prompt-build behavior, not only the memory/context text that may have caused overflow. That can change the agent's policy/instructions on the retry; resolve the hook result and drop only the context fields, or explicitly implement and test the broader behavior.
  - confidence: 0.86
- **[P2] Keep overflow classification when the loop budget is exhausted:** `src/agents/pi-embedded-runner/run.ts:1192-1198`
  - body: This branch always schedules another iteration after terminal overflow. If the overflow happens on the final allowed run-loop pass, the next loop hits the retry-limit guard before making the suppress attempt, so users get `retry_limit` instead of the intended context-overflow guidance and telemetry. Gate the `continue` on remaining loop budget or return the overflow immediately when exhausted.
  - confidence: 0.84

  - body: In hook-free runs, `suppressPromptHookContext` does not change the constructed prompt, so this adds one more full model call that should deterministically overflow again. Check for prompt-build/legacy hook availability or a recorded hook context contribution before spending the retry.
