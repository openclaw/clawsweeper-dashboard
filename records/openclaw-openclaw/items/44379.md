## Summary

Keep open. The central suppress-hook recovery behavior is not implemented on current main, but this PR still has blocking correctness/security concerns, lacks real after-fix behavior proof, and needs maintainer judgment on what hook effects may be suppressed.

## What This Changes

The PR adds a one-attempt prompt-hook-suppressed retry for embedded PI context overflow, normalizes context-window stop-reason strings, updates targeted tests, and changes Google Chat/memory-core peer metadata plus the lockfile.

## Best Possible Solution

Land a rebased replacement that suppresses only shrinkable hook-provided context when present and loop budget remains, preserves non-size hook effects, removes package/lockfile churn, and includes redacted real-run recovery proof.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows current main can still reach terminal context_overflow after compaction/tool-result truncation while prompt-build hooks always run, but this review did not exercise a real memory-hook overflow setup.

## Solution Assessment

No. The fallback direction is plausible, but this implementation suppresses all prompt-build hook effects, retries without loop-budget or hook-availability guards, and mixes unrelated dependency churn into the branch.

  - body: When there is no queued injection or prompt-build hook context that can shrink the prompt, suppressPromptHookContext cannot change the constructed request. This adds another full model call that should fail with the same overflow.
  - confidence: 0.84
