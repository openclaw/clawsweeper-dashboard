## Summary

The credential split is directionally sound, but this head is not merge-ready: every first-publish or trusted-publisher repair candidate is routed into a bootstrap workflow whose pinned released CLI lacks the required command, and the protected OIDC/token paths have no real workflow proof.

## What This Changes

The PR splits routine ClawHub plugin publishing from token-gated package bootstrap and adds release routing, verification, and tests.

## Best Possible Solution

Pin and directly validate a published ClawHub CLI that supports the exact trusted-publisher command and flags, then prove routine OIDC publication, first-package bootstrap, and trusted-publisher repair in real protected workflows before merging.

## Reproduction Assessment

## Solution Assessment

No, not in its current staged form. Splitting routine OIDC publication from token bootstrap is the right security direction, but production routing should not land until the bootstrap dependency is executable and all protected paths are live-proven.
