## Summary

The OIDC/token split is directionally sound, but this PR cannot merge as written because every non-dry-run bootstrap is pinned to a published CLI that lacks the command its preflight requires; the maintainer label also requires explicit release-owner handling.

## What This Changes

The PR splits plugin ClawHub publishing into a routine OIDC workflow and an environment-protected token bootstrap workflow, then updates release routing, planning, verification, and tests for both paths.

## Best Possible Solution

Keep the two-path design, pin a reviewed published ClawHub CLI that supports trusted-publisher setup, replace pin-only assertions with a capability check where practical, and require redacted GitHub Actions evidence for successful routine OIDC, first-publish bootstrap, and configure-only recovery before landing.

## Reproduction Assessment

Yes. Inspecting the published `clawhub@0.20.2` artifact reproduces the exact preflight failure: the required `package trusted-publisher set` command is absent, so every non-dry-run bootstrap stops before publishing.

## Solution Assessment

No, not at this head. Separating routine OIDC publishing from exceptional token bootstrap is the preferable architecture, but production routing must not depend on a deliberately nonfunctional pinned dependency and needs real upgrade/operation proof.
