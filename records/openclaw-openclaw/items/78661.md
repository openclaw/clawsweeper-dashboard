## Summary

Keep open. Current main still has a source-reproducible embedded-runner bypass: OpenClaw's own completions transport sends streaming usage, but stream resolution preserves non-`streamSimple` session streams, including Pi's default auth-wrapped stream path. This is distinct from https://github.com/openclaw/openclaw/issues/73990 and is a focused follow-up to the closed https://github.com/openclaw/openclaw/issues/75357 transport fix.

## What This Changes

The issue asks OpenClaw to send `stream_options.include_usage: true` for custom `api: "openai-completions"` embedded streaming sessions without requiring a per-model compat override.

## Best Possible Solution

Teach stream resolution to recognize Pi's default OpenAI-compatible fallback stream wrappers and route those through OpenClaw's boundary-aware transport, with regression coverage for custom `openai-completions` embedded sessions.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: create an embedded session whose base stream is Pi's auth-wrapped `streamSimple` for a custom `openai-completions` model with normalized streaming-usage compat false, then inspect that the resolver returns the existing stream instead of OpenClaw's boundary-aware transport. I did not run the live llama.cpp proxy probe in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is in embedded stream resolution, not a new config knob or a requirement for every custom model to set `compat.supportsUsageInStreaming`.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/78661. Custom `api: "openai-completions"` embedded sessions can keep Pi's default auth-wrapped stream function, which is not literally the imported `streamSimple`, so `resolveEmbeddedAgentStreamFn` treats it as `session-custom` and bypasses OpenClaw's boundary-aware completions transport that always sends `stream_options.include_usage: true`. Keep true provider-owned and explicit custom streams intact; do not solve this by requiring users to set `compat.supportsUsageInStreaming`, and do not implement the broader fallback-estimation request from https://github.com/openclaw/openclaw/issues/73990. Add a focused regression test in `src/agents/pi-embedded-runner/stream-resolution.test.ts` using Pi's native/default `openai-completions` stream shape or an equivalent recognized fallback wrapper, prove the resolver returns the OpenClaw boundary-aware stream and injects the resolved API key, and add a `CHANGELOG.md` fix entry under Unreleased Fixes. Likely implementation belongs in `src/agents/pi-embedded-runner/stream-resolution.ts`, with any small helper kept local unless the provider transport seam genuinely needs it.
