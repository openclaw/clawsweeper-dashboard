## Summary

Keep open: current main still lacks the requested onboard daemon-env flag, but the PR as written would bypass the existing daemon environment safety boundary by persisting arbitrary user-supplied env keys after filtering.

## What This Changes

The PR adds a repeatable `--daemon-env KEY=VALUE` option to `openclaw onboard`, threads it through onboarding options, parses entries, and merges them into gateway daemon service environments in both non-interactive and wizard install paths.

## Best Possible Solution

Add a reviewed daemon-env mechanism that normalizes portable env names, rejects dangerous host-env and override names through the existing policy helpers, gives diagnostics for malformed or blocked entries, documents the supported bootstrap use case, and adds the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to follow the PR path with `--daemon-env NODE_OPTIONS=--require=/tmp/pwn.js`; the parser accepts the key and the install paths merge it after `buildGatewayInstallPlan()` has already applied host-env filtering.

## Solution Assessment

No, not as written. The plumbing addresses the requested bootstrap gap, but the maintainable fix should validate or narrowly allow daemon env entries before they become persistent service metadata.
