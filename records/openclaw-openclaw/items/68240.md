## Summary

Keep open: the requested onboard daemon-env feature is not on current main, but this PR still needs changes because it introduces a persistent daemon environment injection path without the existing dangerous-key validation and without real install proof.

## What This Changes

The PR adds a repeatable `--daemon-env KEY=VALUE` option to `openclaw onboard --install-daemon`, parses the entries, and merges them into gateway daemon service environments for non-interactive and wizard installs.

## Best Possible Solution

Add a policy-backed daemon-env mechanism that validates portable keys, rejects dangerous host-env and generated-service overrides, preserves service-env source metadata, documents the supported bootstrap use case, and includes real generated-unit or install proof.

## Reproduction Assessment

Yes for the PR defect by source inspection: the parser accepts any non-empty key and callers merge the result after `buildGatewayInstallPlan()` filtering. I did not run a live daemon install in this read-only review.

## Solution Assessment

No, not as written. The bootstrap flag is plausible, but the maintainable solution needs policy-backed key validation, service-env provenance preservation, tests, and real install proof before persistent daemon metadata accepts user input.
