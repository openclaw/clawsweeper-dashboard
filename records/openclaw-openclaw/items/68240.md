## Summary

Keep open: current main still lacks the requested onboard daemon-env feature, but the PR bypasses existing daemon environment safety filtering and lacks contributor real behavior proof, so it is not merge-ready.

## What This Changes

The PR adds a repeatable `--daemon-env KEY=VALUE` flag to `openclaw onboard --install-daemon` and merges parsed values into gateway service install environments for non-interactive and wizard flows.

## Best Possible Solution

Add a reviewed daemon-env mechanism that normalizes portable keys, rejects dangerous host-env and override names through existing policy helpers, preserves service env provenance, documents the bootstrap use case, and includes real generated-unit or install proof.

## Reproduction Assessment

Yes for the PR defect by source inspection: `--daemon-env NODE_OPTIONS=...` would be accepted by the parser and merged after `buildGatewayInstallPlan()` filtering. I did not run a live daemon install in this read-only review.

## Solution Assessment

No, not as written. The requested bootstrap capability is useful, but the maintainable solution needs policy-backed key validation and current service-env provenance handling before user input becomes persistent daemon metadata.
