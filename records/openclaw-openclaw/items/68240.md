## Summary

Keep this open for maintainer/security review: current main does not already implement the onboard flag, the linked feature issue was closed in favor of this PR, and the branch still has a concrete persistent-env injection concern plus missing real behavior proof.

## What This Changes

The PR adds a repeatable `--daemon-env KEY=VALUE` onboard option, parses entries in a daemon-install helper, and merges them into gateway daemon service environments for non-interactive and wizard installs.

## Best Possible Solution

Implement a policy-backed daemon-env mechanism that allows the intended bootstrap use case without reintroducing blocked host-env/control overrides, with focused tests, docs, and real install proof.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `--daemon-env NODE_OPTIONS=--require=/tmp/pwn.js`: the PR parser accepts the key and both install paths merge it after `buildGatewayInstallPlan()` filtering has already run.

## Solution Assessment

No, not as written. The plumbing addresses the bootstrap gap, but the maintainable solution needs reviewed key validation or an allowlist before user input becomes persistent daemon service metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91
