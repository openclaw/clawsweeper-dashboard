## Summary

Keep open: current main still lacks the requested onboarding flag, but the PR currently bypasses the daemon service environment safety boundary and has no real install proof, so it should not merge without maintainer policy review and contributor proof.

## What This Changes

The PR adds a repeatable `--daemon-env KEY=VALUE` option to `openclaw onboard --install-daemon`, threads it through onboarding options, and merges parsed entries into non-interactive and wizard service install environments.

## Best Possible Solution

Add a reviewed daemon-env mechanism that normalizes portable keys, rejects dangerous host-env and override names through existing policy helpers, emits clear diagnostics, covers both install paths with tests/docs, and includes real generated-unit or install proof.

## Reproduction Assessment

Yes for the PR defect by source inspection: `--daemon-env NODE_OPTIONS=--require=/tmp/pwn.js` is accepted by the PR parser and merged into both install paths after `buildGatewayInstallPlan()` filtering has already run.

## Solution Assessment

No, not as written. The requested bootstrap capability is useful, but the maintainable solution needs policy-backed key validation before user input becomes persistent daemon service metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Reject unsafe daemon env keys:** `src/commands/daemon-install-helpers.ts:337`
  - body: `parseDaemonEnvEntries()` stores every non-empty key, and both callers merge the result after `buildGatewayInstallPlan()` has already filtered service env. This allows `--daemon-env NODE_OPTIONS=...` or `LD_PRELOAD=...` to persist into the daemon unit despite the host-env security policy, so the parser should normalize portable keys and reject dangerous host-env/override names before returning entries.
  - confidence: 0.92
