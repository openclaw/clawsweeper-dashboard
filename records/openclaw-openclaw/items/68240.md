## Summary

Keep open. The PR addresses a real onboarding bootstrap gap from #67565, but the proposed merge point accepts arbitrary persistent daemon environment keys after the existing host-env safety filters have already run, so it needs maintainer/security direction before merge.

## What This Changes

The PR adds a repeatable `--daemon-env KEY=VALUE` onboard flag, threads entries through `OnboardOptions`, parses them in daemon-install helpers, and merges them into gateway service environments for non-interactive and wizard installs.

## Best Possible Solution

Add a reviewed daemon-env mechanism that normalizes portable keys, rejects dangerous host-env and override names through the existing policy helpers, diagnoses malformed or blocked entries, documents the supported bootstrap use case, and includes focused tests plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `--daemon-env NODE_OPTIONS=--require=/tmp/pwn.js`: the PR parser accepts the key and both install paths merge it after `buildGatewayInstallPlan()` has already filtered blocked keys.

## Solution Assessment

No, not as written. The plumbing solves the bootstrap gap, but the narrow maintainable solution must validate daemon env entries before they become persistent service metadata.

## Review Findings

Concerns:
