## Summary

Keep open. Current main still lacks the requested onboard daemon environment flag, but the PR as proposed crosses an existing host-env security boundary by accepting arbitrary keys and merging them after the install plan has already filtered dangerous runtime-control variables.

## What This Changes

The PR adds a repeatable `--daemon-env KEY=VALUE` option to `openclaw onboard`, threads it through onboarding options, parses entries, and merges them into gateway daemon service environments for non-interactive and interactive install paths.

## Best Possible Solution

Land a reviewed daemon-env mechanism only after it preserves the existing security boundary: centralize parsing, normalize portable env names, reject dangerous host-env and override names with diagnostics, and document the supported bootstrap use case with targeted tests for the CLI, non-interactive install, and wizard install paths.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `--daemon-env NODE_OPTIONS=--require=/tmp/pwn.js` on the PR path: the parser accepts the key and the callers merge it after filtered install-plan environment generation, so it reaches persistent service metadata.

## Solution Assessment

No. The PR solves the plumbing gap, but the safer maintainable solution is to validate or narrowly allow daemon env entries before they are merged into the service environment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject dangerous daemon env keys:** `src/commands/daemon-install-helpers.ts:335`
  - body: `parseDaemonEnvEntries()` accepts any non-empty key, and the install paths merge the result after `buildGatewayInstallPlan()` has stripped blocked host-env names. Passing `--daemon-env NODE_OPTIONS=...` can therefore persist a runtime-injection variable into the daemon service; normalize and reject blocked or invalid keys before assigning them.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The diff introduces a persistent-env injection concern by accepting arbitrary daemon env keys after existing host-env filters run.

Concerns:

- **[medium] Arbitrary daemon env injection bypasses filters:** `src/commands/daemon-install-helpers.ts:335`
  - body: The new parser accepts user-supplied keys without `normalizeEnvVarKey()`, `isDangerousHostEnvVarName()`, or `isDangerousHostEnvOverrideVarName()`, then the install paths merge those keys over the filtered environment. That can persist blocked runtime-control variables such as `NODE_OPTIONS` into systemd or other service metadata.
  - confidence: 0.92
