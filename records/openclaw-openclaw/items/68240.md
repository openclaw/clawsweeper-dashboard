## Summary

Keep open. Current `main` still lacks the requested onboard daemon-env flag, but this PR should not merge as written because it adds user-controlled persistent daemon environment entries after the existing host-env security filters have already run.

## What This Changes

The PR adds a repeatable `--daemon-env KEY=VALUE` option to `openclaw onboard`, threads it through onboarding options, parses entries, and merges them into gateway daemon service environments for non-interactive and wizard installs.

## Best Possible Solution

Add a reviewed daemon-env mechanism that normalizes portable env keys, rejects dangerous host-env and override names through the existing policy helpers before the service environment is produced, reports malformed or blocked entries clearly, and documents the supported LAN/private-WS bootstrap use case if maintainers accept it.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to pass `--daemon-env NODE_OPTIONS=--require=/tmp/pwn.js` on the PR branch: the helper accepts the key and the callers merge it after `buildGatewayInstallPlan()` has applied filtering.

## Solution Assessment

No. The plumbing target is reasonable, but the current approach bypasses an existing security boundary; the safer solution is a policy-backed parser or narrow allowlist integrated before the daemon service environment is installed.
