## Summary

Keep open: current main still lacks the requested onboard daemon-env flag, but this PR needs security/product review because it accepts arbitrary persistent daemon env keys after existing filtering and has no external real behavior proof.

## What This Changes

Adds a repeatable `--daemon-env KEY=VALUE` onboard option and merges parsed entries into gateway daemon install environments for non-interactive and wizard paths.

## Best Possible Solution

Add a policy-backed daemon-env mechanism that normalizes portable keys, rejects dangerous host-env and override names, preserves service env value-source metadata, documents the bootstrap use case, and includes real generated-unit or install proof.

## Reproduction Assessment

Yes for the PR defect by source inspection: `--daemon-env NODE_OPTIONS=...` is accepted by the parser and merged after `buildGatewayInstallPlan()` filtering. I did not run a live daemon install in this read-only review.

## Solution Assessment

No, not as written. The requested bootstrap capability is useful, but the maintainable solution needs policy-backed key validation before user input becomes persistent daemon service metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Reject unsafe daemon env keys:** `src/commands/daemon-install-helpers.ts:337`
  - body: `parseDaemonEnvEntries()` stores every non-empty key, and both callers merge that result after `buildGatewayInstallPlan()` has already filtered service env.
