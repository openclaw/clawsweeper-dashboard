## Summary

Keep open. Current main still lacks the requested onboarding flag, but the PR should not merge as written because it lets arbitrary caller-supplied keys bypass the existing persistent daemon environment filtering boundary.

## What This Changes

The PR adds a repeatable `--daemon-env KEY=VALUE` onboarding option and merges parsed entries into gateway service install environments for non-interactive and wizard daemon installs.

## Best Possible Solution

Add a reviewed daemon-env mechanism that normalizes portable keys, rejects dangerous host-env and override names through the existing policy helpers, diagnoses malformed or blocked entries, documents the bootstrap use case, and covers both install paths with focused tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `--daemon-env NODE_OPTIONS=--require=/tmp/pwn.js`: the PR parser accepts the key and both install paths merge it after `buildGatewayInstallPlan()` has already filtered blocked keys.

## Solution Assessment

No. The plumbing solves the bootstrap gap, but the safer maintainable path is to validate or allowlist daemon env keys before they become persistent service metadata.

## Review Findings
