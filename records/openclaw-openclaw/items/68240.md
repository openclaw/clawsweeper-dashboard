## Summary

Keep open. Current main still lacks the requested onboarding flag, and this PR is useful source work, but it should not merge as written because it creates a persistent daemon-environment bypass around the existing host-env safety boundary.

## What This Changes

The PR adds a repeatable `--daemon-env KEY=VALUE` onboarding option, threads it through onboarding options, parses entries in a shared helper, and merges them into gateway daemon service environments for non-interactive and wizard installs.

## Best Possible Solution

Add a reviewed daemon-env mechanism that normalizes portable keys, rejects dangerous host-env and override names with diagnostics, documents the LAN bootstrap use case, and includes the required changelog and regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to follow the PR path with `--daemon-env NODE_OPTIONS=--require=/tmp/pwn.js`; the parser accepts the key and the install paths merge it after existing plan filtering has run.

## Solution Assessment

No, not as written. The plumbing addresses the bootstrap gap, but the maintainable solution should validate or narrowly allow daemon env entries before they become persistent service metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

## Security Review
