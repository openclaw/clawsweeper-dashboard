## Summary

Keep open. Current main still has the reported Discord voice abort paths: initial join waits only 15 seconds for `Ready` before destroying the connection, and the later `Disconnected` recovery path waits only 5 seconds for `Signalling` or `Connecting` before silently clearing and destroying the session. Recent issue comments independently confirm the behavior through OpenClaw 2026.4.26, so this remains an actionable Discord plugin reliability bug rather than cleanup.

## What This Changes

Discord voice join and reconnect handling should stop aborting or silently destroying sessions after the 2026.4.5 timeout split, likely by lengthening or making the relevant voice timeouts configurable and adding recovery logging.

## Best Possible Solution

Land a focused Discord plugin fix that lengthens and/or makes configurable the initial Ready wait and disconnected recovery grace, adds a warning on the destroy path with guild/channel context, and covers both initial join abort and post-join silent disconnect behavior in targeted voice manager tests. Keep the change inside the Discord plugin boundary; core should only change if a generic plugin API gap is discovered.

## Reproduction Assessment

Yes. The issue has concrete `/vc join` and auto-join reproduction steps plus multiple independent confirmations, and current source gives a high-confidence simulated path by forcing `entersState(... Ready ...)` or the disconnected recovery waits to reject.

## Solution Assessment

Yes. A focused Discord plugin repair is the narrowest maintainable path; moving this into core would cross ownership boundaries, while only changing playback/DAVE receive behavior would miss the observed join and disconnected recovery abort paths.
