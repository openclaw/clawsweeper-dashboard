## Summary

Keep open. Current main and the latest shipped tag still preserve the reported Discord voice failure boundaries: initial join waits 15 seconds for `Ready` before destroying the connection, and the later disconnected recovery path waits only 5 seconds for `Signalling` or `Connecting` before clearing the session and destroying without a warning. The thread also has independent Linux and macOS confirmations, so this remains an actionable Discord plugin reliability bug rather than cleanup.

## What This Changes

The issue asks OpenClaw to stop Discord voice `/vc join` and auto-join sessions from aborting after the 2026.4.x update by lengthening or configuring voice connect/recovery timeouts and adding retry or recovery logging.

## Best Possible Solution

Land a focused Discord plugin fix that lengthens and/or makes configurable the initial Ready wait and disconnected recovery grace, logs the destroy path with enough guild/channel context for operators, and adds regression coverage for both initial join aborts and post-join silent disconnects. Keep the behavior in the Discord plugin boundary; only update shared config/docs metadata if a new public config field is added.

## Reproduction Assessment

Yes. The issue has concrete `/vc join` and auto-join reproduction steps plus independent Linux and macOS confirmations, and current source gives a high-confidence simulated reproduction path by forcing the Ready wait or disconnected recovery waits to reject.

## Solution Assessment

Yes. A focused Discord plugin repair is the narrowest maintainable path; closing as a duplicate is not ideal because this thread carries both initial Ready abort evidence and later silent-disconnect evidence, while #65039 is more handshake-specific.
