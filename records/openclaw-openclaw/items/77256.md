## Summary

Keep open: the feature is useful but the PR is not merge-ready because the Pi runner queues hidden runtime context before the new fail-closed preflight can block, so a denied model call can still affect the next turn.

## What This Changes

Adds a typed fail-closed `before_model_call` plugin hook, conversation-access gating, Pi/CLI/Codex model-boundary wiring, docs/generated baselines, tests, and changelog coverage.

## Best Possible Solution

Land after moving Pi runtime-context queuing behind a successful preflight decision, adding regression coverage, and getting maintainer approval for the public hook API.

## Reproduction Assessment

Yes for the review finding: source inspection of the PR head shows non-runtime Pi prompts queue runtime context before `runAgentHarnessBeforeModelCallHook`, and the helper enqueues a hidden next-turn message. Reproduction is not applicable to the feature request itself.

## Solution Assessment

No, not yet: the general hook shape is coherent, but the Pi runner should make the blocking preflight decision before side effects like `queueRuntimeContextForNextTurn`. After that, maintainer API approval is the remaining non-mechanical gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Queue runtime context only after preflight passes:** `src/agents/pi-embedded-runner/run/attempt.ts:3012-3015`
  - body: In this branch, non-runtime-only Pi prompts queue `runtimeContext` for the next turn before `before_model_call` runs. If the hook blocks or times out/fails closed, no model request is sent but the hidden runtime-context message remains queued via `sendCustomMessage(..., { deliverAs: "nextTurn" })`, so a denied prompt can still influence the next turn. Move this queue after the hook returns `continue`, or undo it on block/failure, before calling `activeSession.prompt`.
  - confidence: 0.91
