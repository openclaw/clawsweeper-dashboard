## Summary

Keep open: the branch is a plausible hook implementation, but the Pi path queues hidden runtime context before the new blocking preflight can deny the model call, so the PR needs a focused fix before maintainer API approval.

## What This Changes

The PR adds a typed fail-closed `before_model_call` plugin hook, conversation-access gating, Pi/CLI/Codex model-submission wiring, docs, generated baselines, and tests.

## Best Possible Solution

Move Pi runtime-context queuing into the accepted submission path after `before_model_call` continues, add regression coverage for blocked preflight with runtime context, then continue maintainer API review of the new hook surface.

## Reproduction Assessment

Yes: source inspection of the PR head shows `queueRuntimeContextForNextTurn` runs before `runAgentHarnessBeforeModelCallHook`, and the queue helper persists hidden next-turn context. A blocking or failing preflight then skips prompt submission only after that side effect has happened.

## Solution Assessment

No, not as currently written: the generic hook design is coherent, but the Pi ordering should run the preflight before queuing hidden runtime context. The safer fix is to keep queueing, `llm_input`, trajectory submission, and `activeSession.prompt` together in the non-blocked path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Move runtime context queue after preflight approval:** `src/agents/pi-embedded-runner/run/attempt.ts:3012-3015`
  - body: `queueRuntimeContextForNextTurn` sends a hidden next-turn custom message, but this now runs before `before_model_call` can block or fail. When a policy hook denies the Pi model request, that runtime context remains in the session even though no provider call happened, so a later turn can inherit context from a blocked request. Run the preflight first and only queue the runtime context on the accepted submission path.
  - confidence: 0.84
