## Summary

Keep this PR open. The forum-topic fail-closed direction is well supported by the related reports, especially https://github.com/openclaw/openclaw/issues/78755, but the current patch also makes explicit DM-topic sends retry without their topic id and still lacks after-fix live Telegram proof.

## What This Changes

The PR narrows Telegram thread-not-found fallback for forum topics, adds tests for Telegram send/draft behavior, persists explicit cron topic routes into isolated-session delivery context, and preserves heartbeat route thread IDs for group routes.

## Best Possible Solution

Keep the scoped forum-topic and cron/heartbeat routing fixes, but keep explicit DM-topic sends fail-closed and require live Telegram proof or a maintainer override before merge.

## Reproduction Assessment

Yes, from source inspection: current main retries explicit forum sends threadless, and the PR's DM-topic regression is visible from the changed helper and test. I did not run tests or a live Telegram probe because this review is read-only.

## Solution Assessment

No, this is not merge-ready as written. The forum fail-closed direction is the right narrow fix, but direct/private topic sends should not retry without topic metadata, and the real Telegram proof gate is still unsatisfied.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep DM-topic sends fail-closed:** `extensions/telegram/src/send.ts:664`
  - body: This enables threadless fallback for direct chats, so a send to a private Telegram target with `messageThreadId` now retries after `message thread not found` and sends again without `message_thread_id`. Current main explicitly rejects that case, and the Telegram maintainer notes say failed topic sends should not be silently retried without topic metadata; otherwise an intended DM-topic message can land in the private chat root instead of surfacing the stale topic. Keep direct/DM topic sends fail-closed unless a caller can prove the thread id is only incidental plain-DM metadata.
  - confidence: 0.88
