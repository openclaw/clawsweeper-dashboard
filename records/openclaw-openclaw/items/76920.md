## Summary

Keep open: the PR addresses a real Telegram forum-topic routing bug, but it is not merge-ready because the branch still opts direct/private topic sends into threadless retry and the required live Telegram proof is missing.

## What This Changes

The PR narrows Telegram thread-not-found fallback, preserves async/heartbeat topic routes, seeds explicit cron topic delivery context, and adds focused Telegram/cron/outbound tests.

## Best Possible Solution

Keep the forum fail-closed, cron, and heartbeat route-preservation work, but preserve fail-closed behavior for explicit direct/private topic sends unless the caller can prove the thread id is incidental plain-DM metadata.

## Reproduction Assessment

Yes, from source inspection and the contributor's base-fails/head-passes test proof: current main retries explicit forum preview/sends without message_thread_id, while the PR covers the forum boundary. I did not run a live Telegram probe in this read-only review.

## Solution Assessment

No, not merge-ready as written. The forum fail-closed and async route preservation direction is right, but direct/private topic sends should not fall back threadless without stronger proof, and live Telegram proof or a maintainer override is still required.
