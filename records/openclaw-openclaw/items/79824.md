## Summary

Keep open: current main still emits Feishu schema-2.0 interactive cards with deprecated `tag: "action"` button containers on the reported approval and presentation paths, and no current code converts them to `column_set`.

## What This Changes

The issue asks the Feishu plugin to render V2-compatible button rows so exec/plugin approval cards and quick-action cards are accepted by Feishu/Lark.

## Best Possible Solution

Fix this inside `@openclaw/feishu` with a shared schema-V2 button-row renderer/sanitizer, update the Feishu card tests and changelog, and live-check at least one approval card plus one presentation-button card.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: any Feishu approval, launcher, or presentation card with buttons currently serializes `tag: "action"` into an interactive card. I did not run a live Feishu send, so the structured status is source-reproducible rather than reproduced.

## Solution Assessment

Yes, the narrowest maintainable fix is an owner-local Feishu helper that emits V2-compatible button rows and is reused by the sanitizer, presentation renderer, approval card, and launcher card. A JSON-stringify rewriter would be a weaker workaround because it would bypass the existing card construction and tests.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79824 in the Feishu plugin. Current `@openclaw/feishu` emits schema-2.0 cards that wrap buttons in `{ tag: "action", actions: [...] }`, which Feishu now rejects for exec/plugin approvals, quick actions, and shared presentation button cards. Add a shared Feishu V2-compatible button-row helper, likely returning `column_set` with one `column` per button, and use it from `extensions/feishu/src/outbound.ts`, `extensions/feishu/src/card-ux-approval.ts`, and `extensions/feishu/src/card-ux-launcher.ts`; keep existing URL/value sanitization and interaction envelopes intact. Update tests that currently assert `tag: "action"` so they assert the new V2 shape and button values. Add a user-facing changelog fix. Do not move this behavior into core, add config flags, or use a blanket `JSON.stringify` rewriter. Validate with targeted Feishu tests and, if credentials are available, a live Feishu/Lark send of one approval card and one presentation-button card.
