## Summary

Keep open. Current main fixes the basic WebChat reconnect/sessionId carry-forward and uses dedicated session RPCs for New Chat, but the issue still has unresolved reset provenance and session-not-found recovery gaps, and the latest comments add fresh evidence that missing session state can still present as apparent silent reset rather than an explicit recovery path.

## What This Changes

The issue reports that WebChat can rotate the active `agent:main:main` session after gateway restart or reconnect, hiding prior transcript and checkpoint state despite recoverable files on disk.

## Best Possible Solution

Define the WebChat continuity policy for missing or rotated sessions, then add typed reset provenance and a visible recover/create notification path while preserving the already-implemented reconnect sessionId carry-forward.

## Reproduction Assessment

No for the exact gateway-restart plus normal-message rotation on current main. Source inspection does confirm remaining gaps: `chat.history` failures only become a UI error, and reset results still do not expose client-visible old/new session ids, trigger, or archive provenance.

## Solution Assessment

Unclear for the full requested direction. The already-landed reconnect sessionId fix is the right narrow fix for refresh/reconnect sends, but the remaining recovery and reset-audit behavior should be scoped as a protocol/UI design rather than swept in by automation.
