## Summary

Keep open. Current main fixes the basic reconnect follow-up send path, but the remaining continuity problem is still valid: missing or failed history loads can leave WebChat without recovery, reset/create responses do not expose old/new reset provenance to the client, and reset archives are not a first-class recovery surface.

## What This Changes

Ensure WebChat continues the existing `agent:main:main` session after reconnect or makes any reset, rotation, archive, and recovery path explicit to the user.

## Best Possible Solution

Define the WebChat continuity policy for missing or rotated sessions, then add typed reset/rotation provenance plus a visible recover/create/restore path while preserving the already-landed `sessionId` carry-forward.

## Reproduction Assessment

No, not for the exact gateway-restart plus plain-message rotation on current main. Source inspection does confirm the remaining missing-session recovery and reset-provenance gaps, and current comments add a concrete WebSocket 1006/history-failure variant.

## Solution Assessment

Unclear for the full requested direction. The existing reconnect `sessionId` carry-forward is the right narrow fix for one failure mode, but reset provenance, archive/checkpoint discoverability, and missing-session recovery need maintainer-scoped UI/protocol behavior.
