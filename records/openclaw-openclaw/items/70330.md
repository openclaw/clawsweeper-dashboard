## Summary

Keep open: current main fixes the narrow reconnect-follow-up-send path by carrying the history-backed `sessionId`, but the remaining source path still lacks session-not-found recovery, reset old/new provenance, and reset archive/checkpoint discoverability.

## What This Changes

Request that WebChat preserve or clearly recover and audit the active `agent:main:main` session across gateway restart/reconnect instead of silently rotating or hiding prior transcript/checkpoint state.

## Best Possible Solution

Define a WebChat continuity policy for missing or rotated sessions, then add typed reset provenance plus a visible recover/create/restore path for reset archives and checkpoints.

## Reproduction Assessment

No, not for the exact gateway-restart plus plain-message rotation on current main. Source inspection confirms the remaining session-not-found/empty-history and reset-provenance gaps, and recent comments add a concrete WebSocket 1006/history-failure variant.

## Solution Assessment

Unclear as a small patch. The reconnect `sessionId` carry-forward is the right narrow fix for one failure mode, but the remaining recovery, provenance, and archive behavior should be scoped as protocol-visible UI behavior.
