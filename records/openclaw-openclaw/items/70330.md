## Summary

Keep open. Current main has improved parts of the WebChat flow, but the remaining session-control and observability gap is still real: generic `chat.send` can still consume admin `/new` or `/reset`, post-reset tails can appear as normal first messages, and reset archives are not a first-class sessions surface.

## What This Changes

The issue asks WebChat to preserve `agent:main:main` across gateway restart/reconnect, make any reset explicit and auditable, and expose prior reset archives/checkpoints instead of hiding them.

## Best Possible Solution

Add a typed WebChat session-control path with reset provenance, keep ordinary reconnect sends bound to the prior session id, and make reset archives/checkpoints discoverable or restorable through a supported sessions surface.

## Reproduction Assessment

Unclear. The issue has detailed log/disk evidence and current main has focused coverage for carrying `sessionId` after history, but this read-only review did not establish an end-to-end gateway restart/reconnect plus plain-send reproduction that creates a reset archive.

## Solution Assessment

Yes for keeping this open and queueing a focused repair. The narrowest maintainable path is additive protocol/UI/reset-service provenance plus regression coverage, while broader archive-browser UX can follow separately if needed.
