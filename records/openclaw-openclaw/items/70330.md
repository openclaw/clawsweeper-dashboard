## Summary

Keep open. Current main has partial reset-flow improvements, but the reported WebChat continuity and observability gap remains: Control UI New Session still routes through text `chat.send("/new")`, admin-scoped `chat.send` can still perform a session reset, reset provenance is not part of the `chat.send` or `sessions.reset` protocol, and reset archives are still not first-class session-list entries.

## What This Changes

WebChat should preserve the active `agent:main:main` session across gateway restart/reconnect unless an explicit reset is requested, and any reset should be auditable with old/new session ids and archive discoverability.

## Best Possible Solution

Add an explicit WebChat session-control path that separates intentional reset/new actions from ordinary chat text, records reset provenance with old/new session ids and archive paths, prevents reconnect/plain-send paths from silently rotating the active mapping, and adds a typed way for Control UI to discover prior reset archives or clearly restore them.

## Reproduction Assessment

Unclear. The issue has detailed disk/log evidence and current main still contains the generic `chat.send` reset path, but this read-only pass did not establish a high-confidence automated gateway restart/reconnect reproduction for a plain post-reconnect message.

## Solution Assessment

Yes for keeping this open and queueing a focused fix. The narrowest maintainable first step is additive protocol/UI/reset-service provenance plus regression coverage, leaving broader history-browser UX as a follow-up if needed.
