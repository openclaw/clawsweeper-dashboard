## Summary

Keep open. Current main has fixed part of the reconnect path by carrying the history-backed `sessionId`, but the issue’s remaining safety gap is still present: ordinary `chat.send` still has a reset-command path without an explicit session-control flag or reset audit payload, and recent comments add a session-not-found reconnect variant that is not covered by the current UI behavior.

## What This Changes

The issue reports that WebChat can rotate the canonical direct session after gateway restart or reconnect, making prior transcripts and checkpoints look lost.

## Best Possible Solution

Add a typed, auditable WebChat session-control path and reconnect/session-not-found recovery behavior while keeping ordinary reconnect sends bound to the prior session id; defer any larger archive browser design if needed.

## Reproduction Assessment

No, not for the exact gateway restart plus normal-message rotation on current main. Source inspection does reproduce the remaining ambiguity: reset control is still accepted through generic `chat.send` without explicit reset provenance, and the issue has detailed disk/log evidence for the observed failure.

## Solution Assessment

Yes, keeping this open and queueing a focused repair is the best path. Current main fixed the basic reconnect `sessionId` carry-forward but not reset provenance, generic send hardening, or the session-not-found recovery surface described by the latest comments.
