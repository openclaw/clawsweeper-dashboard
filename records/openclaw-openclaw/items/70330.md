## Summary

Keep open. Current main addresses the narrow reconnect send path by preserving the `chat.history` backing session id, but the remaining user problem is still valid: history-load failure has no explicit recover/create/restore path, reset/create responses do not expose old/new reset provenance to the WebChat client, and reset archives are not a first-class session recovery surface.

## What This Changes

The issue asks WebChat to avoid silent `agent:main:main` session rotation after restart or reconnect and to make any reset, archive, or recovery path explicit and discoverable.

## Best Possible Solution

Define a WebChat continuity policy for missing or rotated sessions, then add typed reset provenance plus a visible recover/create/restore path while preserving the already-landed reconnect `sessionId` carry-forward.

## Reproduction Assessment

No, not for the exact gateway-restart plus plain-message rotation on current main. Source inspection does confirm the remaining session-not-found and reset-audit gaps, and the latest comments add a concrete WebSocket 1006/history-failure variant that still has no explicit recovery path.

## Solution Assessment

Unclear for the full requested direction. The existing reconnect `sessionId` carry-forward is the right narrow fix for one failure mode, but reset provenance, archive/checkpoint discoverability, and missing-session recovery should be scoped as protocol-visible UI behavior rather than a small automated patch.
