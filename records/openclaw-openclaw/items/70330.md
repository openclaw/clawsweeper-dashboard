## Summary

Keep open. The merged reconnect fix covers carrying the history-backed sessionId through follow-up sends, but current main still lacks explicit recovery/audit behavior for missing or rotated WebChat sessions and reset archives; recent comments add a missing-session variant not closed by the merged fix.

## What This Changes

The issue asks WebChat to preserve or clearly recover and audit the active session across gateway restart/reconnect instead of silently rotating `agent:main:main` and hiding prior transcripts/checkpoints.

## Best Possible Solution

Define a WebChat continuity policy for missing or rotated sessions, then add typed reset provenance plus a visible recover/create/restore path while keeping the already-landed reconnect sessionId carry-forward.

## Reproduction Assessment

No for the exact gateway-restart plus plain-message rotation on current main. Source inspection does provide a medium-confidence current-main path for the missing-session variant: history can return empty/no sessionId, the UI clears its backing id, and the next send is no longer bound to the prior session.

## Solution Assessment

Unclear for the full requested direction. The existing reconnect sessionId carry-forward is the right narrow fix for one failure mode, but reset provenance, archive/checkpoint discoverability, and missing-session recovery should be scoped as protocol-visible UI behavior rather than a small automated patch.
