- `impact:session-state`: The report centers on session id rotation, hidden history, checkpoints, and agent context drift.

## Summary

Keep open. Current main covers the basic reconnect follow-up send path, but source and live issue context still support the remaining WebChat continuity gaps: missing history can leave blank state without recovery, reset responses still do not expose old/new reset provenance to the client, and reset archives are not a first-class recovery surface.

## What This Changes

The issue asks WebChat to preserve or explicitly recover/audit `agent:main:main` continuity after gateway restart/reconnect instead of silently rotating to a new backing session and hiding prior transcript/checkpoints.

## Best Possible Solution

Define the WebChat continuity policy for missing or rotated sessions, then add typed reset/rotation provenance plus a visible recover/create/restore path while preserving the already-landed `sessionId` carry-forward behavior.

## Reproduction Assessment

No, not for the exact gateway-restart plus plain-message rotation on current main. Source inspection does confirm remaining missing-session recovery and reset-provenance gaps, and current comments add a concrete WebSocket 1006/history-failure variant.

## Solution Assessment

Unclear for the full requested direction. The existing reconnect `sessionId` carry-forward is the right narrow fix for one failure mode, but reset provenance, archive/checkpoint discoverability, and missing-session recovery need maintainer-scoped UI/protocol behavior.
