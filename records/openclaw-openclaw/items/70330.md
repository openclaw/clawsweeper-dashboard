## Label Justifications

- `P1`: The issue describes repeated WebChat session continuity loss in released builds, hiding transcripts/checkpoints and causing agents to continue without expected context.
- `impact:session-state`: The affected behavior is session id rotation, hidden reset archives, missing checkpoints, and active-session state drift for one logical WebChat conversation.

## Summary

Keep open. Current main fixes the basic reconnect `sessionId` carry-forward and the WebChat `/new` main-session reset slice, but the remaining missing-session recovery, reset/rotation provenance, and reset-archive discoverability gaps are still unresolved and overlap active related PRs.

## What This Changes

The issue asks WebChat to preserve or clearly recover and audit the active `agent:main:main` session after gateway restart/reconnect instead of silently rotating to a new session.

## Best Possible Solution

Define a backward-compatible WebChat continuity contract that preserves the history-backed `sessionId`, exposes typed reset/rotation provenance, and provides a visible recover/create/restore path for missing active sessions and reset archives.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No, not for the exact gateway-restart plus plain-message rotation on current main; source inspection does confirm the remaining missing-session recovery and reset-provenance gaps, and comments add a concrete WebSocket 1006/history-failure variant.

## Solution Assessment

Is this the best way to solve the issue? Unclear for the full requested direction. The shipped `sessionId` carry-forward is the right narrow reconnect fix, but reset provenance, archive/checkpoint discoverability, and missing-session recovery need maintainer-scoped protocol and UI behavior.
