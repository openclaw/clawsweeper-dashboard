## Summary

Keep open: current main still has the unsafe git checkout argv, the docs promise that `git:<repo>@<ref>` checks out the requested ref, and https://github.com/openclaw/openclaw/pull/79901 is an open same-author fix PR with closing syntax.

## What This Changes

Git plugin installs can treat an option-like explicit ref as a checkout flag and continue from the default branch instead of failing.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/79901 or an equivalent focused fix that separates Git options from the requested ref and keeps regression coverage for option-like refs.

## Reproduction Assessment

Yes for a high-confidence source-level path: current main parses the option-like selector into `parsed.ref` and sends it to `git checkout --detach <ref>`. A scratch Git 2.53.0 repo confirmed the reported checkout form exits successfully while the separated `git switch` form rejects the invalid ref.

## Solution Assessment

Yes, the linked PR direction is the narrow maintainable fix: pass the user-supplied value after a Git option separator using the ref-oriented switch command and lock the argv boundary with a unit regression test.
