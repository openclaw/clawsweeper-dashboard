## Label Justifications

- `P1`: The reported race can break gateway client authentication/handshake flows in the latest release when paired-device state is rewritten concurrently.
- `impact:crash-loop`: The issue is about transient pairing-state read failures surfacing as gateway closes and reconnect amplification rather than being absorbed by bounded retry.

## Summary

Keep open: current `main` still has a source-clear path where a stable-target read race from `@openclaw/fs-safe` becomes `JsonFileReadError` without retry, and `paired.json` reads sit on gateway authentication paths.

## What This Changes

Add bounded retry behavior for transient JSON file read races such as `File changed during read` while preserving corrupt-file failures.

## Best Possible Solution

Centralize a small bounded retry for target-change read races in the JSON file helper layer, and keep parse/corruption and non-transient read errors fail-closed after retries.

## Reproduction Assessment

Yes, by source inspection: `@openclaw/fs-safe` throws `File changed during read`, OpenClaw wraps that as `JsonFileReadError`, and pairing reads do not retry. I did not run a live concurrent rewrite harness.

## Solution Assessment

Yes, the narrow maintainable fix is a central JSON-read retry for only the transient stable-target race while preserving strict parse/read failures for existing corrupt files.
