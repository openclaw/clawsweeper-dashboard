## Label Justifications

- `P1`: The source-backed race can break gateway client authentication/handshake flows in the latest release when paired-device state is rewritten concurrently.
- `impact:crash-loop`: The report is about transient pairing-state read failures surfacing as gateway closes and reconnect amplification instead of being absorbed by bounded retry.

## Summary

Keep open: current main still delegates JSON reads directly to `@openclaw/fs-safe`, whose stable-target race is wrapped as `JsonFileReadError` without retry, and `paired.json` reads sit on gateway authentication paths.

## What This Changes

JSON file helpers should retry transient stable-target read races so concurrent `paired.json` atomic rewrites do not close gateway clients.

## Best Possible Solution

Centralize a bounded retry in the JSON helper layer for only the transient stable-target race, while keeping missing-file semantics and corrupt/unreadable-file failures unchanged.

## Reproduction Assessment

Yes, by source inspection: `@openclaw/fs-safe` throws `File changed during read`, OpenClaw wraps that as `JsonFileReadError`, and current pairing reads do not retry. I did not run a live concurrent rewrite harness.

## Solution Assessment

Yes, the narrow maintainable fix is a helper-level retry for only the transient target-change read race. The safer boundary is to preserve strict parse/read failures for existing corrupt files after retries are exhausted.
