- `impact:crash-loop`: The report is about transient paired-state read failures causing gateway closes and reconnect loops instead of being absorbed by bounded retry.

## Summary

Keep open: current main and the latest stable release still use the non-retrying async JSON read path, and an open implementation PR already references this issue as its closing target.

## What This Changes

The issue requests bounded retry handling for transient JSON file read races when paired.json or other JSON state files are atomically rewritten during reads.

## Best Possible Solution

Land or replace the linked retry fix, preferably by consuming a released @openclaw/fs-safe version that retries only the typed transient path-mismatch race while preserving fail-closed parse and non-transient read errors.

## Reproduction Assessment

Yes, by source inspection: current main reads paired.json through the direct @openclaw/fs-safe JSON helper export without retry, and gateway control/auth paths depend on those paired-device reads. I did not run a live concurrent rewrite harness in this read-only review.

## Solution Assessment

Yes. The maintainable fix is a central async JSON-read retry for only the transient stable-target race, ideally in @openclaw/fs-safe with an OpenClaw dependency bump rather than permanent duplicate wrapper logic.
