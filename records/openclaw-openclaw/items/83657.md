## Label Justifications

- `P1`: The issue describes a source-backed gateway paired-state read race that can break active client/control UI authorization flows and amplify reconnects.
- `impact:crash-loop`: The reported failure surfaces as gateway closes and reconnect amplification rather than a harmless transient file-read retry.

## Summary

Keep open: current main and the latest stable release still pin `@openclaw/fs-safe@0.2.4` and re-export its non-retrying async JSON readers, while the linked OpenClaw fix PR and upstream fs-safe PR are both still open.

## What This Changes

Retry async JSON reads on transient `File changed during read` races so concurrent atomic rewrites do not fail `paired.json` gateway reads.

## Best Possible Solution

Land the typed async JSON retry in `@openclaw/fs-safe`, release it, then bump OpenClaw and close this issue when the OpenClaw PR or a replacement merges.

## Reproduction Assessment

Yes. Source inspection shows current OpenClaw delegates to fs-safe 0.2.4, fs-safe throws `File changed during read` without retry, and `paired.json` reads sit on gateway authentication paths; I did not run the live multi-client race.

## Solution Assessment

Yes for the requested bounded async retry, but the best boundary is upstream fs-safe before JSON wrapping/swallowing, followed by an OpenClaw dependency bump. The currently linked OpenClaw-only PR should be repaired or replaced before merge.
