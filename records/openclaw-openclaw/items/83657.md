## Label Justifications

- `P1`: A transient pairing-state read race can break gateway authentication/handshake flows and trigger reconnect amplification for active users.
- `impact:crash-loop`: The reported failure mode is gateway close/reconnect churn caused by surfacing a transient JSON read race instead of retrying it.

## Summary

Keep open: current main and the latest release still pin `@openclaw/fs-safe@0.2.4`, whose async JSON readers do not retry the `File changed during read` race; the right fix is active in openclaw/fs-safe, but it is not merged, released, or bumped here yet.

## What This Changes

The issue asks OpenClaw JSON reads to retry transient atomic-rename races, especially `paired.json` reads during concurrent gateway client activity.

## Best Possible Solution

Land and release the upstream fs-safe bounded async JSON retry, then bump `@openclaw/fs-safe` in OpenClaw and keep parse/corruption failures fail-closed.

## Reproduction Assessment

Yes, by source inspection and upstream real-disk proof: OpenClaw pins fs-safe `0.2.4`, the dependency throws `File changed during read` without retry, and `paired.json` reads are on the gateway handshake path. I did not run the full multi-client gateway harness in this read-only review.

## Solution Assessment

Yes, the best fix is the dependency-level typed transient sentinel plus bounded async JSON retry, followed by an OpenClaw dependency bump. A local OpenClaw wrapper would be a fallback only if the upstream fix stalls.
