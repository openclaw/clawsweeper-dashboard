## Label Justifications

- `P1`: The source-backed race can break gateway/control UI authentication or reconnect flows for users when pairing state is rewritten concurrently.
- `impact:crash-loop`: The issue reports transient pairing-state read failures surfacing as gateway closes and reconnect amplification rather than being absorbed by bounded retry.

## Summary

Keep open: current main still pins `@openclaw/fs-safe` 0.2.4 and re-exports its async JSON readers without retry, while the OpenClaw closing PR is still open and under review. Closure should wait for a merged upstream fs-safe fix plus OpenClaw version bump, or a maintainer-accepted local hotfix.

## What This Changes

The issue requests bounded retry behavior for transient JSON read races such as `File changed during read` on `paired.json`.

## Best Possible Solution

Land the retry at the fs-safe JSON-reader boundary and bump OpenClaw to that released version, while preserving fail-closed behavior for parse errors, corruption, and non-transient read failures.

## Reproduction Assessment

Yes, at source level: current main re-exports fs-safe JSON readers without retry, fs-safe 0.2.4 throws and wraps `File changed during read`, and pairing reads sit on gateway/control UI authorization paths. I did not run a live multi-client race harness in this read-only review.

## Solution Assessment

Yes, bounded retry only for the transient stable-target race is the right fix shape. The cleaner implementation is upstream fs-safe retry before JSON errors are wrapped, followed by an OpenClaw dependency bump.
