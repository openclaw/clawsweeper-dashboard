## Label Justifications

- `P1`: The reported race can surface during gateway pairing/authentication and cause gateway closes or reconnect amplification for real users.
- `impact:crash-loop`: The issue is about a transient paired-state read failure propagating as gateway close/reconnect behavior instead of being absorbed by bounded retry.

## Summary

Keep open: current main and the latest release still use the non-retrying `@openclaw/fs-safe` JSON readers, and an open fixing PR explicitly references this issue, so the issue should remain open until that fix path lands.

## What This Changes

The issue asks for bounded retries around transient JSON file read races so atomic rewrites do not surface as gateway failures.

## Best Possible Solution

Ship a central helper or dependency-level retry that handles only the transient file-identity race, preserves parse/corruption failures, and closes this issue after the linked fix lands.

## Reproduction Assessment

Yes, by source inspection: current main re-exports non-retrying async JSON readers from `@openclaw/fs-safe@0.2.4`, and paired-device state reads use those readers on gateway authentication paths. I did not run a live concurrent rewrite harness in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is bounded retry for the transient file-identity race at the shared JSON-helper or fs-safe layer, while keeping parse and real read failures fail-closed. The upstream fs-safe PR is the cleanest ownership boundary, and the linked OpenClaw PR is the active local implementation candidate.
