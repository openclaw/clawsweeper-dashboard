## Summary

Keep open: current main still validates `browser.upload` inputs only against `DEFAULT_UPLOAD_DIR`, while WebChat attachments are persisted as managed inbound media and can be staged as `media/inbound/*`, so the reported cross-tool failure remains source-reproducible.

## What This Changes

The issue requests that `browser.upload` accept OpenClaw-managed inbound WebChat media files in addition to files under the browser upload directory.

## Best Possible Solution

Add a shared browser-upload path resolver that preserves `DEFAULT_UPLOAD_DIR` behavior while explicitly accepting managed inbound media through the existing media-reference/store policy, then cover the tool wrapper, existing-session route, Playwright revalidation, and docs.

## Reproduction Assessment

Yes, at source level. Current main exposes managed WebChat media as `media://inbound` or staged `media/inbound/*`, while `browser.upload` and its upload hooks validate only paths under `DEFAULT_UPLOAD_DIR`.

## Solution Assessment

Yes, the maintainable fix is to reuse the existing managed-media resolver rather than adding arbitrary local roots or asking agents to copy files into `/tmp/openclaw/uploads`. The implementation should stop for maintainer input if supporting sandbox-relative paths requires a new browser tool context contract.
