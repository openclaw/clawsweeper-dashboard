- `P2`: The issue describes a real but bounded WebChat attachment-to-browser upload workflow break.
- `impact:security`: The affected code is a browser file-upload allowlist, and any fix changes the model-driven file selection boundary.

## Summary

Keep open: current main and the latest release still limit browser upload paths to the temp uploads root, while WebChat attachments are saved and staged as managed inbound media. The open fixing PR already closes this issue, so the right path is maintainer review and landing of that PR rather than closing the issue early.

## What This Changes

The issue asks `browser.upload` to accept OpenClaw-managed inbound WebChat attachment files and media references in addition to the temp upload directory.

## Best Possible Solution

Land one canonical fix that centralizes validation for temp uploads and verified managed inbound media across browser upload preflight, server hook, and Playwright use-time validation, with rejection tests for traversal and nested refs.

## Reproduction Assessment

Yes, at source level: browser upload paths validate only against `DEFAULT_UPLOAD_DIR`, while WebChat saves and stages attachments as `media://inbound/<id>` and `media/inbound/*`. I did not run a live WebChat/Gateway/browser flow in this read-only review.

## Solution Assessment

Yes, the maintainable fix is a shared upload resolver that accepts the existing uploads root plus verified managed inbound media references, as the open linked PR attempts. The issue should stay open until that PR is reviewed, landed, or explicitly rejected.
