## Label Justifications

- `P2`: The issue describes a real but bounded WebChat attachment-to-browser upload workflow break.
- `impact:security`: The affected code is a browser file-upload path allowlist and any fix must preserve the sandbox/file-access boundary.

## Summary

Keep open: current main still scopes browser uploads to the temp uploads root, while WebChat attachment handling can stage inbound files as `media/inbound/*` for the agent. The report is source-reproducible, but the fix broadens a browser file-upload allowlist, so it needs explicit owner review before repair.

## What This Changes

The issue asks for `browser.upload` to accept OpenClaw-managed inbound WebChat media paths as upload sources.

## Best Possible Solution

Add a browser-owned upload path resolver that accepts `DEFAULT_UPLOAD_DIR` plus verified managed inbound media references or staged `media/inbound/*` files, update both tool preflight and server-side validation, and cover escape/rejection cases with focused tests.

## Reproduction Assessment

Yes, source-level: WebChat non-image attachments can be staged as `media/inbound/report.pdf`, while `browser.upload` and the browser server only accept paths under `DEFAULT_UPLOAD_DIR`. I did not run the full live browser/WebChat flow in this read-only review.

## Solution Assessment

Yes, the requested direction is maintainable if it reuses the existing managed inbound media resolution pattern instead of adding a broad local-path exception. Owner review is still needed because this changes the browser upload security boundary.
