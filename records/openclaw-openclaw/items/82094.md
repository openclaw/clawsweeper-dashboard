## Summary

Keep open: current main still matches the reported regression path, and the dependency contract points to a real failure mode. The latest release/current code sends ffmpeg a temporary output ending in `.wav.part` without forcing WAV format, while ffmpeg guesses output format from the file extension unless `-f` is provided.

## What This Changes

The issue reports a v2026.5.12 regression where CLI-backed audio transcription via whisper-cli fails for Telegram voice messages after the WAV conversion path was wrapped in fs-safe external-output handling.

## Best Possible Solution

Preserve the fs-safe external-output hardening, but make the whisper-cli transcode explicit to ffmpeg by forcing WAV output for the temp path and covering that with a focused regression test.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, live no in this checkout: current main can be exercised with non-WAV audio and whisper-cli, and it will pass ffmpeg a `.wav.part` output path without `-f wav`; local live proof was blocked because ffmpeg is not installed.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix is to keep the fs-safe wrapper and force the ffmpeg output format, instead of reverting the broader security-hardening refactor.

Fix the STT regression reported at https://github.com/openclaw/openclaw/issues/82094: non-WAV audio sent to a `whisper-cli` media-understanding CLI entry is transcoded through `writeExternalFileWithinRoot`, which gives ffmpeg a temp output path ending in `.wav.part`; because ffmpeg guesses output format from the extension, the conversion can fail with `Command failed`.
