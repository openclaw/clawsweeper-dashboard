## Summary

Keep open. Current main now covers WebChat/Control UI upload selection, non-video file transport, and text/PDF extraction, but the requested Word-document extraction is still not implemented: Office and legacy Word MIME types are treated as binary and skipped, while the bundled document-extract plugin only advertises PDF. Related #57612 is the clearest DOCX extraction candidate, but it is still open and does not settle WebChat media-understanding routing or legacy .doc policy.

## What This Changes

The issue requests WebChat document upload support for PDF, Word, and TXT files, including extracting file text and sending it to the assistant from the web interface.

## Best Possible Solution

Keep the current upload transport, add a policy-backed DOCX extraction path through the existing document-extractor/media pipeline, wire uploaded DOCX files through WebChat media-understanding when maintainers want automatic extraction, and explicitly reject or defer legacy .doc unless a safe extractor is approved. Update docs and regression tests around Control UI uploads, Gateway staging, input_file extraction, and the document-extract plugin contract.

## Reproduction Assessment

Yes. Current main can be checked from source and tests: WebChat accepts and stages Word-shaped files, but media-understanding classifies Office/legacy Word MIME types as binary and skips them before extraction, while input_file extraction only allows text/PDF by default.

## Solution Assessment

No. The issue’s proposed new multer-style upload endpoint and direct pdf-parse/mammoth server route do not match current architecture; the maintainable path is extending the existing chat.send attachment, managed media, and plugin document-extractor pipeline.
