## Summary

Keep open. Current main has the low-level last-assistant stripping helper and provider-gated replay plumbing, but it does not implement the requested user-visible `thinkingHistory` retention policy, and current Claude 4.x behavior deliberately preserves prior thinking blocks for prompt-cache stability.

## What This Changes

The issue requests a configurable thinking-history retention policy that can keep, strip all, or keep only the latest assistant thinking blocks before API replay to reduce token usage.

## Best Possible Solution

Add a documented opt-in retention policy that composes with provider replay policy instead of silently changing provider-safe defaults. The implementation should update config schema/help/docs and cover legacy Claude, Claude 4.x, Bedrock/Anthropic-compatible replay, tool-continuation replay, and context-pruning estimates.

## Reproduction Assessment

Yes for the implementation gap: source inspection shows no `thinkingHistory` config key, and both replay-history and run-loop stripping remain gated by provider replay policy. A live provider token measurement is not needed to prove the requested config surface is absent.

## Solution Assessment

Unclear. The requested `last-only` default is not obviously the safest global solution because current main deliberately preserves prior thinking blocks for newer Claude models; a maintainer-approved opt-in or per-model policy is the narrower maintainable direction.
