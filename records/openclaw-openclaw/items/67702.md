## Summary

Keep open. The PR addresses a real git-mode update availability bug, but the current branch is conflicting, fails the check job, lacks contributor real behavior proof, and still has merge-blocking defects in service detection and restart recovery.

## What This Changes

The PR adds a git-mode update preflight that detects a same-install running Gateway service, auto-stops it before in-place dist replacement unless `--no-restart` is set, and adds update CLI tests.

## Best Possible Solution

Land a rebased git-mode update guard that handles direct and wrapper-backed service commands, restores any stopped service on all non-success paths, uses the service-control stream contract, fixes the stop/update/restart guidance, and includes focused tests plus redacted real update proof.

## Reproduction Assessment

Yes, source-level. Current main's git update path still reaches `runGitUpdate` without a same-install Gateway stop boundary, and the related stale hashed chunk logs support the failure class, though I did not replay a live git-mode update.

## Solution Assessment

No. The direction is right, but the patch is not the best mergeable form until it handles wrapper-backed services, restores stopped services after skipped/error updates, passes a real writable stream, fixes guidance, rebases, and adds real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Handle wrapper-backed gateway service commands:** `src/cli/update-cli/update-command.ts:135`
  - body: `gateway install --wrapper` can persist `ProgramArguments` as `[wrapperPath, "gateway", ...]`, so returning the token before `gateway` compares the wrapper executable to this install's dist entrypoint. That lets a same-install wrapped Gateway keep running through the in-place git update this guard is meant to prevent.
  - confidence: 0.91
- **[P1] Restart the gateway after skipped or failed git updates:** `src/cli/update-cli/update-command.ts:1128-1129`
  - body: After the new pre-update stop, `runGitUpdate` can still return `skipped` for a dirty checkout or `error` for fetch/rebase/build failures. Those paths return before using `gatewayStoppedForUpdate`, so a previously running Gateway can stay down even though no update succeeded.
  - confidence: 0.92
- **[P1] Pass a WritableStream to service.stop:** `src/cli/update-cli/update-command.ts:1128`
  - body: `GatewayServiceControlArgs.stdout` is a `NodeJS.WritableStream`, but this call passes `defaultRuntime`, which only exposes logging helpers and matches the PR's failing TS2740 check. Use `process.stdout` or another real writable stream.
  - confidence: 0.95
- **[P2] Remove restart from the pre-update stop hint:** `src/cli/update-cli/update-command.ts:1112-1117`
  - body: In the `--no-restart` block, telling users to run `openclaw gateway restart` before rerunning update leaves the service running and hits the same guard again. The hint should say stop first, rerun update, then restart after the update completes.
  - confidence: 0.89
