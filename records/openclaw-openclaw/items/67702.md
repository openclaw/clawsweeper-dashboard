Adds a git-mode update preflight that detects a same-install running Gateway service, auto-stops it before in-place dist replacement unless restart is disabled, and adds update CLI tests.

## Best Possible Solution

Land a rebased git-mode update guard that reuses the current service layout/entrypoint contracts, handles wrapper-backed services, restores an auto-stopped Gateway on every non-success path, uses the service-control stream contract, fixes the stop/update/restart guidance, and includes focused tests plus redacted real update proof.

Yes from source inspection, but not from a live replay in this read-only review. ## Solution Assessment

No. The direction is right, but this patch is not the best mergeable form until it handles wrapper-backed service commands, restores stopped services after skip/error paths, passes a real writable stream, fixes user guidance, rebases, and adds real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Handle wrapper-backed gateway service commands:** `src/cli/update-cli/update-command.ts:135`
  - body: `gateway install --wrapper` can persist `ProgramArguments` as `[wrapperPath, "gateway", ...]`, so returning the token before `gateway` compares the wrapper executable to this install's dist entrypoint. That lets a same-install wrapped Gateway keep running through the in-place git update this guard is meant to prevent.
  - confidence: 0.91
- **[P1] Restart the gateway after skipped or failed git updates:** `src/cli/update-cli/update-command.ts:1128-1129`
  - body: After the new pre-update stop, `runGitUpdate` can still return `skipped` for a dirty checkout or `error` for fetch/rebase/build failures. Those paths return without using `gatewayStoppedForUpdate`, so a previously running Gateway can stay down even though no update succeeded.
  - confidence: 0.92
- **[P1] Pass a WritableStream to service.stop:** `src/cli/update-cli/update-command.ts:1128`
  - body: `GatewayServiceControlArgs.stdout` is a `NodeJS.WritableStream`, but this call passes `defaultRuntime`, which only exposes logging helpers. The PR's `check` job reports TS2740 here; use `process.stdout` or another real writable stream.
  - confidence: 0.95
- **[P2] Remove restart from the pre-update stop hint:** `src/cli/update-cli/update-command.ts:1112-1117`
  - body: In the `--no-restart` block, telling users to run `openclaw gateway restart` before rerunning update leaves the service running and hits the same guard again. The hint should say stop first, rerun update, then restart after the update completes.
  - confidence: 0.89
