## Summary

Keep open. Current main still lacks a git-mode safe boundary for updating under a running same-install Gateway, but this PR is not merge-ready because it misses wrapper-backed service commands, can leave the Gateway stopped after skipped or failed updates, fails typecheck, and lacks real behavior proof.

## What This Changes

The PR adds git-mode update preflight logic that detects a same-install running Gateway service, stops or blocks before in-place dist replacement, and adds focused update CLI tests.

## Best Possible Solution

Land a repaired git-mode update guard that recognizes direct and wrapper-backed same-install Gateway services, restores any pre-stopped service on every non-success path, blocks cleanly with `--no-restart`, and proves the behavior with focused tests plus real update validation.

## Reproduction Assessment

Yes, source-reproducible. Current main sends git updates directly to `runGitUpdate` without the package-update service stop boundary, and the PR defects are traceable from supported service `ProgramArguments`, `runGatewayUpdate` skipped/error paths, and the `GatewayServiceControlArgs` type contract.

## Solution Assessment

No. The direction is right, but the patch needs wrapper-aware command detection, rollback restart for git-mode skipped/error results, a real writable stream for `service.stop`, and corrected stop-before-update guidance before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Detect wrapper-backed gateway commands:** `src/cli/update-cli/update-command.ts:132`
  - body: `gateway install --wrapper` can persist `ProgramArguments` as `[wrapperPath, "gateway", ...]`, so returning the token before `gateway` compares the wrapper executable to this install's dist entrypoint. That lets a same-install wrapped Gateway keep running through the in-place git update this guard is meant to prevent.
  - confidence: 0.91
- **[P1] Restart the service after skipped or failed git updates:** `src/cli/update-cli/update-command.ts:1125`
  - body: After this pre-update stop, `runGitUpdate` can still return `skipped` for a dirty checkout or `error` for fetch/rebase/build failures. Those paths only restore `prePackageServiceStop`, so a previously running Gateway can be left down even though no update succeeded.
  - confidence: 0.92
- **[P1] Pass a writable stream to service.stop:** `src/cli/update-cli/update-command.ts:1125`
  - body: `GatewayServiceControlArgs.stdout` is a `NodeJS.WritableStream`, but this call passes `defaultRuntime`. CI reports TS2740 here; use `process.stdout` or another real writable stream so service adapters can write status output.
  - confidence: 0.95
- **[P2] Remove restart from the pre-update stop hint:** `src/cli/update-cli/update-command.ts:1109-1115`
  - body: In the `--no-restart` block, telling users to run `openclaw gateway restart` before rerunning update leaves the service running and hits the same guard again. The hint should say stop first, rerun update, then restart after the update completes.
  - confidence: 0.89
