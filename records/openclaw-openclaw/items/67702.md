## Summary

Keep open: this PR targets a real git-mode stale-dist safety gap on current main, but the branch still misses wrapper-backed Gateway service commands, can leave a stopped Gateway down on skipped/error updates, and fails typecheck by passing the runtime logger where a writable stream is required.

## What This Changes

The PR adds git-mode update preflight logic that detects a running same-install Gateway service, auto-stops or blocks it before in-place dist replacement, and adds update CLI tests.

## Best Possible Solution

Land a repaired git-mode update guard that recognizes direct and wrapper-backed same-install Gateway services, restores any pre-stopped service on every non-success path, blocks cleanly with --no-restart, and proves the behavior with focused tests plus update validation.

## Reproduction Assessment

Yes, source-reproducible. Current main sends git updates directly to runGitUpdate without the package-update service stop boundary, and the PR defects can be reproduced from the service ProgramArguments and skipped/error result paths.

## Solution Assessment

No. The direction is right, but this patch needs wrapper-aware command detection, shared stop/restore bookkeeping for git-mode non-success results, a real writable stream for service.stop, and corrected stop-before-update guidance.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Handle wrapper-backed gateway commands:** `src/cli/update-cli/update-command.ts:132`
  - body: `gateway install --wrapper` can write ProgramArguments as `[wrapperPath, "gateway", ...]`, so returning the token before `gateway` compares the wrapper executable to this install's dist entrypoint. That lets a same-install wrapped Gateway keep running through the in-place git update this guard is meant to prevent.
  - confidence: 0.91
- **[P1] Restore the gateway after skipped or failed git updates:** `src/cli/update-cli/update-command.ts:1125`
  - body: After the new pre-update stop, `runGitUpdate` can still return `skipped` for a dirty checkout or `error` for fetch/rebase/build failures. Those paths only restore `prePackageServiceStop`, so a previously running Gateway can be left down even though no update succeeded.
  - confidence: 0.92
- **[P1] Pass a writable stream to service.stop:** `src/cli/update-cli/update-command.ts:1125`
  - body: `GatewayServiceControlArgs.stdout` is a `NodeJS.WritableStream`, but this call passes `defaultRuntime`. CI already reports TS2740 here; use `process.stdout` or another real writable stream so service adapters can write status output.
  - confidence: 0.95
- **[P2] Remove restart from the pre-update stop hint:** `src/cli/update-cli/update-command.ts:1109-1115`
  - body: In the `--no-restart` block, telling users to run `openclaw gateway restart` before rerunning update leaves the service running and hits the same guard again. The hint should say stop first, rerun update, then restart after the update completes.
  - confidence: 0.89
