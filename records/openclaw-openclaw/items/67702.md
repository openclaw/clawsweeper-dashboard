## Summary

Keep open. Current main has package-update lifecycle hardening, but git-mode updates still run without a same-install Gateway guard; this PR is useful source work, but its current patch has concrete correctness blockers before merge.

## What This Changes

The PR adds git-mode update detection for a running same-install Gateway service, auto-stops it when restart is enabled, blocks with `--no-restart`, and adds update CLI tests.

## Best Possible Solution

Land a current-main git-mode update guard that recognizes direct and wrapper-backed same-install Gateway services, restores the service on every non-success path, blocks cleanly with `--no-restart`, and includes focused tests plus a changelog entry.

## Reproduction Assessment

Yes. Mock a running service with ProgramArguments `[wrapperPath, "gateway", ...]` and the PR matcher compares the wrapper to `dist/index.js`; mock `runGitUpdate` returning `skipped` after the PR stop call and the existing failure handler has no git stop state to restart.

## Solution Assessment

No. The direction is right, but the safest implementation should share explicit stop/restore bookkeeping with package updates, handle wrapper-backed service commands, pass a real writable stream, and keep restart guidance to stop-before-update then restart-after-update.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Handle wrapper-backed gateway commands:** `src/cli/update-cli/update-command.ts:132`
  - body: `gateway install --wrapper` writes ProgramArguments as `[wrapperPath, ...gateway args]`, so the token before `gateway` is the wrapper. Comparing that to `dist/index.js` lets a same-install wrapped Gateway keep running through the in-place git update, missing the stale-dist crash this guard is meant to prevent.
  - confidence: 0.91
- **[P1] Restore the gateway after stopped git updates that fail or skip:** `src/cli/update-cli/update-command.ts:1125`
  - body: After this stop runs, `runGitUpdate` can return `skipped` for a dirty checkout or `error` for fetch/rebase/build failures. Those branches only restore `prePackageServiceStop`, so a previously running Gateway can be left down even though no update succeeded.
  - confidence: 0.92
- **[P1] Pass a writable stream to service.stop:** `src/cli/update-cli/update-command.ts:1125`
  - body: `GatewayServiceControlArgs.stdout` is a `NodeJS.WritableStream`, but this call passes `defaultRuntime`, which only exposes logging/json helpers. Use `process.stdout` or another real stream so the source typechecks and service adapters can write status output.
  - confidence: 0.88
- **[P2] Remove restart from the pre-update stop hint:** `src/cli/update-cli/update-command.ts:1109-1114`
  - body: In the `--no-restart` block, `openclaw gateway restart` leaves the service running, so rerunning update hits the same guard again. The hint should tell users to stop first, rerun update, and restart only after the update completes.
  - confidence: 0.89
