## Summary

Keep this PR open: current main still lacks a git-mode guard before in-place update work, so the contribution targets a real remaining problem. The PR should not merge as-is because it misses documented wrapper-backed service commands and can leave a previously running managed Gateway stopped when the git update is skipped or fails.

## What This Changes

The PR adds a git-mode update preflight that detects a running same-install Gateway service from service runtime/command metadata, stops it before in-place updates when restart is allowed, blocks with `--no-restart`, and adds update CLI tests.

## Best Possible Solution

Keep the PR as useful source work, but land a current-main implementation that protects git-mode in-place updates, recognizes both direct and wrapper-backed service commands, and guarantees Gateway restoration across success, skipped, and failed update paths with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes. The code-level reproduction is high-confidence: mock a wrapper ProgramArguments shape `[wrapperPath, 'gateway', ...]` and the PR matcher compares the wrapper path to `dist/index.js`; mock a dirty or failed git update after `service.stop` and the flow returns before restart handling.

## Solution Assessment

No. The PR is the right general direction, but the best maintainable fix should integrate with current main's service-stop/restart bookkeeping or an equivalent `try`/restore path, and it must parse wrapper-backed managed-service commands instead of assuming the token before `gateway` is always the dist entrypoint.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Handle wrapper-backed gateway commands:** `src/cli/update-cli/update-command.ts:132`
  - body: `gateway install --wrapper` writes ProgramArguments as `[wrapperPath, ...gateway args]`, so the token before `gateway` is the wrapper. Comparing that to this install's dist entrypoint lets wrapper-managed gateways continue through the git update, leaving the stale-dist crash path unguarded.
  - confidence: 0.9
- **[P1] Restore the gateway when the git update does not apply:** `src/cli/update-cli/update-command.ts:1125`
  - body: After stopping the service, `runGitUpdate` can return `skipped` for a dirty checkout or `error` for fetch/rebase/build failures. Those branches return before normal restart handling, so a previously running managed gateway can be left down even though no update succeeded.
  - confidence: 0.92
- **[P2] Remove restart from the pre-update stop hint:** `src/cli/update-cli/update-command.ts:1109-1114`
  - body: In the `--no-restart` block, `gateway restart` leaves the service running, so rerunning `openclaw update --no-restart` hits the same guard again. The hint should tell users to stop first and restart only after the update.
  - confidence: 0.88
