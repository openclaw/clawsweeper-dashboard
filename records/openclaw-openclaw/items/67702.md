## Summary

Keep open. The PR addresses a real git-mode update reliability gap that current main still does not cover, but the branch is not mergeable as-is: it misses supported wrapper-backed service commands, can leave a pre-stopped Gateway down on skipped/error updates, fails the service-control stdout type contract, and lacks external real behavior proof.

## What This Changes

The PR adds git-mode update preflight logic and tests to stop or block a same-install running Gateway before replacing the dist tree in place.

## Best Possible Solution

Land a rebased git-mode update guard that recognizes direct and wrapper-backed same-install Gateway services, restores any auto-stopped service on every non-success path, uses the service-control stream contract, and includes focused tests plus redacted real update proof.

## Reproduction Assessment

Yes, source-reproducible. Current main still sends git updates to runGitUpdate without a same-install Gateway service boundary, and the PR blockers are traceable through service ProgramArguments, runGatewayUpdate skipped/error returns, and the GatewayServiceControlArgs type contract.

## Solution Assessment

No. The direction is right, but this patch is not the best mergeable form until it handles wrapper-backed service commands, restarts on skipped/error updates, passes a real writable stream, fixes the --no-restart guidance, rebases, and adds real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Handle wrapper-backed gateway commands:** `src/cli/update-cli/update-command.ts:135`
  - body: `gateway install --wrapper` can persist `ProgramArguments` as `[wrapperPath, "gateway", ...]`, so returning the token before `gateway` compares the wrapper executable to this install's dist entrypoint. That lets a same-install wrapped Gateway keep running through the in-place git update this guard is meant to prevent.
  - confidence: 0.91
- **[P1] Restart after skipped or failed git updates:** `src/cli/update-cli/update-command.ts:1128-1129`
  - body: After the new pre-update stop, `runGitUpdate` can still return `skipped` for a dirty checkout or `error` for fetch/rebase/build failures. Those paths return without using `gatewayStoppedForUpdate`, so a previously running Gateway can stay down even though no update succeeded.
  - confidence: 0.92
- **[P1] Pass a writable stream to service.stop:** `src/cli/update-cli/update-command.ts:1128`
  - body: `GatewayServiceControlArgs.stdout` is a `NodeJS.WritableStream`, but this call passes `defaultRuntime`, which only exposes logging helpers. The PR's `check` job reports TS2740 here; use `process.stdout` or another real writable stream.
  - confidence: 0.95
- **[P2] Remove restart from the pre-update stop hint:** `src/cli/update-cli/update-command.ts:1112-1117`
  - body: In the `--no-restart` block, telling users to run `openclaw gateway restart` before rerunning update leaves the service running and hits the same guard again. The hint should say stop first, rerun update, then restart after the update completes.
  - confidence: 0.89
