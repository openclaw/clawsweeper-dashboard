## Summary

Keep open: current main and the latest shipped tag still let externalized bundled-plugin sync install the bridge npm spec and overwrite the managed npm root direct dependency, so a newer user pin can be replaced. The related peer-dependency PRs do not cover this direct dependency path.

## What This Changes

The issue asks `openclaw update --tag` to stop silently downgrading or rewriting user-pinned managed npm direct dependencies.

## Best Possible Solution

Keep this open and land a narrow update-path fix that preserves or explicitly blocks newer compatible managed npm direct dependencies during externalized bundled-plugin sync, with regression coverage for the Lossless Claw shape.

## Reproduction Assessment

Yes. The issue has a high-confidence source reproduction path: an enabled persisted bundled bridge with an older npm spec flows into `installPluginFromNpmSpec`, whose managed npm root writer overwrites the existing package dependency.

## Solution Assessment

Yes for the downgrade-prevention half: the best fix is a narrow guard in externalized bundled-plugin sync or managed npm root planning before writing the dependency. The broader respect-all-pins behavior should stay out of scope unless maintainers choose that policy explicitly.

## Repair Work Prompt

Fix the bug reported at https://github.com/openclaw/openclaw/issues/85184: `openclaw update --tag` must not silently downgrade a newer managed npm root direct dependency when externalized bundled-plugin sync installs an older bridge npm spec. Scope the fix to the post-core plugin sync / managed npm root path: detect an existing direct dependency or installed package for the same package name before `installPluginFromNpmSpec` writes the managed npm root, preserve or skip the downgrade when the existing version is newer and compatible with the target OpenClaw host, and surface a clear warning/outcome instead of rewriting the pin. Add regression coverage for a persisted bundled bridge whose npm spec is `@martian-engineering/lossless-claw@0.10.0` while the managed npm root already has `@martian-engineering/lossless-claw: 0.11.2`; keep normal migration behavior when no newer pin exists. Do not solve the broader respect-all-pins policy or add a new CLI/config flag unless maintainers explicitly choose that direction. Add a changelog fix entry because this is user-visible update behavior.
