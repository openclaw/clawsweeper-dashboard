## Summary

Keep open: the report is source-reproducible on current main because post-core plugin sync can install the persisted bundled-plugin npm spec and the managed npm root writer overwrites the direct dependency entry without checking whether that is a downgrade from the installed or pinned package version.

## What This Changes

The issue reports that `openclaw update --tag` rewrites a user-pinned direct managed-root plugin dependency to OpenClaw's bundled target version, silently downgrading the installed package and leaving package metadata inconsistent.

## Best Possible Solution

Add a post-core plugin sync guard that preserves or loudly aborts when the target bridge/install spec would downgrade an installed or direct managed-root dependency, while leaving broader pin-preference policy as a separate maintainer choice.

## Reproduction Assessment

Yes, by source inspection and the reporter's concrete update logs. A focused repro can seed a persisted bundled-plugin bridge for `lossless-claw`, set the managed npm root dependency and installed package to 0.11.2, then run the post-core plugin sync path with an older bridge npm spec and observe the direct dependency rewrite path.

## Solution Assessment

Yes for the unambiguous downgrade-prevention half: the narrow maintainable fix is to compare the installed/direct managed-root version before writing the bridge spec and skip or fail loudly on downgrade. The broader question of always respecting manual pins can be handled separately if maintainers want that policy.
