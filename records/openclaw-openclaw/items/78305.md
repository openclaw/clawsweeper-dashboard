## Summary

Keep open: the ACP opacity problem is real, but the current PR still exposes raw nested error detail, lacks live gateway ACP proof, and needs to be rebased/narrowed before it is mergeable.

## What This Changes

The branch adds ACP cause-chain formatting through lifecycle/session/gateway error surfaces and tests it, while currently carrying many unrelated release, workflow, docs, plugin, TUI, and package-version changes.

## Best Possible Solution

Keep the ACP diagnostic improvement, but rebase it into a narrow branch, redact all formatted cause chains, add the required changelog entry, and attach live gateway ACP failure proof before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a live run in this read-only review, but the source path is clear: current main preserves `cause` on `AcpRuntimeError` and then records only `acpError.message` or `String(err)` at the affected lifecycle/task/gateway surfaces.

## Solution Assessment

Is this the best way to solve the issue? No; preserving the cause detail is the right direction, but the implementation must redact the rendered chain, be rebased to a narrow ACP diff, and be backed by real gateway proof.
