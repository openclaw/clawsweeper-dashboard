## Summary

Keep open. The PR addresses a source-visible ACP error detail loss, but it is not merge-ready because the new error-chain paths bypass redaction and the external contributor has not supplied real gateway proof.

## What This Changes

The PR adds ACP error-chain formatting, routes ACP lifecycle/task/gateway failures through it, and adds regression coverage for ACP lifecycle events and gateway log formatting.

## Best Possible Solution

Preserve actionable ACP JSON-RPC details through the boundary only after redaction, add the required changelog entry, and require real gateway ACP failure proof before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a live run from this review, but the opacity problem is source-reproducible: current main preserves `cause` and then records only `acpError.message` or `String(err)` at the affected surfaces.

## Solution Assessment

Is this the best way to solve the issue? No; preserving cause details is reasonable, but the implementation should redact the formatted chain before any log, task, lifecycle, or reply surface and should be backed by live gateway proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Redact ACP error chains before surfacing them:** `src/acp/runtime/errors.ts:97`
  - body: `formatAcpErrorChain` is used for lifecycle/task/session errors that can reach chat and tool-result surfaces, but it returns raw nested `Error.message` text and raw non-Error causes. ACP subprocess/provider errors can include request details or tokens; use the existing redacted error formatting or call `redactSensitiveText` before returning this chain.
  - confidence: 0.87

  - body: The new gateway `Error` path returns the rendered cause chain directly, bypassing the redaction that `formatForLog` applies to JSON/string values. A nested cause message containing a token would now be written to gateway logs, so redact before truncating or returning the chain.
  - confidence: 0.84

  - body: This changes user-visible ACP failure text in logs, replies, and task/session status, so repo policy requires a single-line `CHANGELOG.md` entry under Unreleased before merge.
  - confidence: 0.78
