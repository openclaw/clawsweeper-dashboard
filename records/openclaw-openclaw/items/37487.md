## Summary

Keep open. Current main and v2026.6.5 have named browser profiles, per-profile CDP/userDataDir for some modes, and global Chrome launch args, but they still lack first-class per-profile proxy config, agent-bound default browser profiles, and a concurrency/ownership contract for multi-agent browser lanes.

## What This Changes

The issue requests concurrent per-agent browser isolation, per-profile proxy configuration, agent-to-browser-profile binding, diagnostics, and documentation.

## Best Possible Solution

Define a maintainer-approved browser-session ownership design, then implement safe per-profile proxy config and agent/profile binding in the browser plugin with SecretRef, redaction, SSRF, concurrency, diagnostics, docs, and upgrade tests.

## Reproduction Assessment

Yes for the missing-surface claim, but not as a strict bug reproduction. Source inspection confirms current main lacks browser.profiles.<name>.proxy and agent-level browser profile binding, while named profiles and global extraArgs only cover part of the request.

## Solution Assessment

Unclear as a single patch: the direction is useful, but PR 42806 showed that proxy credentials and SSRF policy need design first. The safer solution is staged browser-plugin/config/session work with explicit security and compatibility boundaries.

## Risks / Open Questions

- Per-profile browser proxy support changes network egress and can weaken current SSRF assumptions unless proxy-routed profiles are explicitly modeled and tested.

- Agent-bound browser defaults and inheritance can expose cookies, localStorage, SSO sessions, and tabs across agents if ownership and override semantics are implicit.
- The requested concurrency model needs resource limits and tab/process cleanup semantics; prior comments call out launch serialization and multi-Chromium memory usage.
