## Summary

Keep open. Current main has partial named-profile isolation and a global Chrome `extraArgs` proxy workaround, but it still lacks first-class per-profile proxy config, per-agent/default browser-profile binding, sub-agent inheritance, and a reviewed concurrency/resource policy; the prior implementation was closed unmerged after concrete security concerns.

## What This Changes

The issue requests first-class per-agent isolated browser instances, per-profile proxy configuration, and concurrent browser execution with predictable isolation, resource, and diagnostic behavior.

## Best Possible Solution

Design this in the bundled browser plugin and agent config contracts with secret-safe per-profile browser proxy config, agent/default profile binding, sub-agent inheritance rules, explicit SSRF behavior, tab ownership, concurrency/resource limits, diagnostics, docs, and focused regression tests coordinated with the broader proxy umbrella.

Not applicable: this is a feature/config request, not a broken existing behavior report. Source inspection gives a high-confidence missing-surface check for per-profile proxy config, agent/default browser-profile binding, and concurrency policy.

## Solution Assessment

Unclear as a ready implementation path. The requested direction is plausible, but the narrow maintainable solution needs product and security design around credentials, SSRF, profile-selection authority, sub-agent inheritance, and resource limits before code is added.

## Security Review

Status: needs_attention

Summary: This feature changes browser egress, proxy credentials, and cross-agent state isolation, so security design is required before implementation.

Concerns:

- **[medium] Design proxy credential handling before implementation:**
  - body: A first-class per-profile proxy surface must avoid leaking credentials through config snapshots, logs, status responses, Chrome command-line arguments, process listings, crash reports, or diagnostics.
