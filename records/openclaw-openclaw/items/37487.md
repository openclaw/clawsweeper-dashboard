## Summary

Keep open. Current main has named browser profiles, separate managed profile CDP/user-data directories, tracked-tab cleanup, sandbox browser controls, and a global managed HTTP/HTTPS proxy, but it still lacks the requested first-class per-profile proxy field, agent/default browser-profile binding, subagent inheritance, SOCKS support, and a reviewed concurrency/resource policy. The linked `m13v/browser-lock` and FSB comments are useful prior art, and the closed implementation attempt confirms this is a real feature surface, not something already shipped.

## What This Changes

The issue requests first-class per-agent isolated browser/profile instances with per-profile proxy configuration and reliable concurrent execution.

## Best Possible Solution

Design a staged browser/profile contract that covers secret-safe per-profile proxy config, agent/default profile binding with subagent inheritance, profile-selection authority, tab ownership and cleanup, launch serialization, resource limits, diagnostics, docs, and regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature/config request; source inspection gives a high-confidence missing-surface check for per-profile proxy config, agent browser-profile binding, subagent inheritance, and concurrency policy on current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear as a direct implementation path; the requested direction is plausible, but the maintainable fix needs product and security design around credentials, SSRF, profile-selection authority, tab ownership, and resource limits before code is added.
