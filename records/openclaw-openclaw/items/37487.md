## Summary

Keep open. Current main still lacks first-class per-profile browser proxy config, per-agent/default browser-profile binding, sub-agent inheritance, and a reviewed concurrency/resource policy; the related browser-lock and FSB comments provide useful external implementation context, and the prior matching PR was closed unmerged after security review concerns.

## What This Changes

Request first-class per-agent isolated browser profiles with per-profile proxy support and reliable concurrent execution semantics.

## Best Possible Solution

Design this as a coordinated browser plugin plus agent config contract: secret-safe per-profile proxy config, agent/default profile binding with sub-agent inheritance, explicit SSRF behavior, tab ownership, launch serialization, resource limits, diagnostics, docs, and focused regression tests, while coordinating with the broader proxy umbrella.

## Risks / Open Questions

- Proxy credentials can leak through Chrome command-line arguments, status payloads, logs, crash reports, or diagnostics if the config surface is added without a secret-safe design.
- Browser proxy routing hides the final connect target from OpenClaw, so per-profile proxies need explicit SSRF and private-network semantics before implementation.
- The authority model for agent and sub-agent selection of proxy-enabled profiles is unresolved.
- Concurrency still needs launch serialization, resource limits, queueing, and tab ownership semantics rather than only named profile storage.
