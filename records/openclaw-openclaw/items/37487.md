## Summary

Keep open. Current main partially covers the area with named browser profiles, profile-specific storage/CDP ports, documented Chrome proxy flags via `browser.extraArgs`, and top-level HTTP managed proxy routing, but it still lacks first-class per-profile proxy config, agent/default browser-profile binding, sub-agent inheritance, and a reviewed concurrency/resource policy.

## What This Changes

The issue requests first-class per-agent browser/profile isolation, per-profile proxy configuration, concurrent execution guarantees, and docs/examples for multi-agent browser workflows.

## Best Possible Solution

Design a staged browser/proxy contract covering secret-safe per-profile proxy config, agent/default profile binding with subagent inheritance, profile-selection authority, tab ownership/cleanup, launch serialization, resource limits, diagnostics, docs, and regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature/config request; source inspection gives a high-confidence missing-surface check for per-profile proxy config, agent browser-profile binding, sub-agent inheritance, and concurrency policy on current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear as a direct implementation path; the requested direction is plausible, but the maintainable fix needs product and security design around credentials, SSRF, profile-selection authority, tab ownership, and resource limits before code is added.
