## Summary

Keep open. Current main and v2026.5.22 cover named managed browser profiles, separate CDP ports/user-data dirs, explicit browser-tool profile selection, global managed-browser proxy flags via `browser.extraArgs`, strict SSRF handling for explicit browser proxy routing, and tracked tab cleanup, but they still do not provide first-class per-profile proxy config or agent/default browser-profile binding; the prior-art comments and closed attempted PR make the remaining product and security design concrete.

## What This Changes

The issue requests per-agent isolated browser instances with per-profile proxy support for concurrent agent and subagent browser workflows.

## Best Possible Solution

Design a staged browser ownership contract in the browser plugin and agent config covering SecretRef-safe per-profile proxy config, agent/default profile binding with subagent inheritance, tab ownership, launch serialization, resource limits, SSRF semantics, diagnostics, docs, and regression coverage, coordinated with the narrower open proxy and subagent-session issues.

## Reproduction Assessment

Yes for the source-level missing feature surface: current strict schemas lack `browser.profiles.<name>.proxy` and `agents.defaults/list` browser-profile binding, and the browser tool only reads an explicit profile argument. This is not a runtime bug reproduction; it is a feature/config gap visible from current main and the latest release tag.

## Solution Assessment

Unclear as a direct implementation because the requested direction is plausible but security-sensitive. The maintainable solution should define proxy credentials, SSRF semantics, profile-selection authority, tab ownership, launch concurrency, and diagnostics rather than adding a narrow config field alone.

## Risks / Open Questions

- A narrow `browser.profiles.*.proxy` field could expose proxy credentials through config snapshots, status payloads, logs, Chrome argv, process listings, crash diagnostics, or docs without a SecretRef and redaction design.
- Browser proxy routing changes SSRF enforcement because OpenClaw cannot inspect the proxy's final connect target; current strict mode fails closed for explicit browser proxy routing unless private-network access is intentionally allowed.
- Agent-bound default profiles decide which cookies, tabs, CDP endpoint, and egress path an agent or subagent can use, so inheritance and override rules need product/security review before public config keys are added.
- Concurrency support needs launch serialization, resource limits, tab ownership, and cleanup semantics under load; the external `browser-lock` and FSB comments show these are operational requirements, not just config naming.
