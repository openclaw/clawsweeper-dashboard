## Summary

Keep open. Current main still supports named browser profiles and a global `browser.extraArgs` proxy workaround, but it does not provide first-class per-profile proxy config, agent/default browser-profile binding, sub-agent inheritance, or concurrency/resource policy; the matching implementation pull request was closed unmerged and raised concrete security concerns.

## What This Changes

Request first-class per-agent browser profile isolation with per-profile outbound proxy configuration and concurrent execution guidance.

## Best Possible Solution

Design this in the bundled browser plugin and agent config contracts with secret-safe per-profile proxy config, agent-scoped profile defaults, sub-agent inheritance rules, explicit SSRF behavior, concurrency/resource limits, diagnostics, docs, and focused regression tests.

## Risks / Open Questions

- Proxy credentials, Chrome argv exposure, logs/status redaction, and secret-reference semantics need maintainer security design before implementation.
- Proxy-routed browser traffic changes SSRF/private-network assumptions because OpenClaw may not see the final network destination.
- Agent default-profile and sub-agent inheritance semantics could break existing shared-profile workflows or create surprising resource usage without an explicit concurrency policy.
- https://github.com/openclaw/openclaw/issues/60035 overlaps the broader proxy umbrella, so this browser-specific design should coordinate with that thread without losing the per-agent isolation requirement.
