## Label Justifications

- `P2`: This is a real multi-agent browser workflow improvement with meaningful user impact, but it is a feature/design item rather than an urgent regression or outage.
- `impact:session-state`: The request concerns cross-agent browser tabs, cookies, storage, profile state, cleanup ownership, and subagent inheritance.
- `impact:security`: The requested proxy and profile routing affects browser egress, proxy credentials, SSRF boundaries, and agent access to browser state.

## Summary

Keep open. Current main partially supports named managed browser profiles and global Chrome proxy flags, but it still lacks first-class per-profile proxy configuration and agent/default browser-profile binding; the linked prior-art comments and closed unmerged implementation PR show the need is real and security-sensitive.

## What This Changes

Request first-class per-agent isolated browser instances with per-profile proxy support, concurrent execution guarantees, agent/default profile binding, diagnostics, and docs.

## Best Possible Solution

Design a staged browser/profile contract covering SecretRef-safe per-profile proxy config, agent/default profile binding with subagent inheritance, launch serialization, resource limits, tab ownership, diagnostics, docs, and regression coverage, aligned with the broader proxy and built-in browser trackers where their scope overlaps.

## Reproduction Assessment

Yes for the source-level missing feature surface: current strict schemas lack browser.profiles.<name>.proxy and agents.*.browser.profile, and browser tool profile selection is not agent-scoped. This is not a runtime bug reproduction; it is a feature gap visible from current main.

## Solution Assessment

Unclear as a direct implementation because the requested direction is plausible but security-sensitive. The maintainable solution should define proxy credentials, SSRF semantics, profile-selection authority, tab ownership, and concurrency limits before adding public config fields.
