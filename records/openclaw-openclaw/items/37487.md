## Summary

Keep open. Current main still has named isolated browser profiles and an explicit `profile` browser-tool argument, but it does not implement first-class per-profile proxy config or agent/sub-agent default browser profile routing; the related implementation PR is closed unmerged, and the linked `m13v/browser-lock` project is useful prior art rather than shipped OpenClaw behavior.

## What This Changes

The issue requests first-class per-agent isolated browser instances with per-profile proxy support for concurrent agent and sub-agent browser workflows.

## Best Possible Solution

Define a maintainer-reviewed, secret-aware per-profile proxy contract plus agent/sub-agent browser profile inheritance rules, redacted diagnostics, docs/examples, and focused browser/config/agent regression coverage.

## Reproduction Assessment

Yes. This is reproducible as a source-level configuration-contract gap: strict schemas omit `browser.profiles.<name>.proxy` and `agents.*.browser.profile`, while managed browser proxy routing is only global through `browser.extraArgs`.

## Solution Assessment

No, current main is not the best final solution for the requested workflow. Explicit tool `profile` arguments, remote CDP/Browserless, multiple Gateways, or global Chrome proxy flags are workarounds rather than first-class per-agent/per-profile routing.
