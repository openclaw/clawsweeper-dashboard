## Summary

Keep open. Current main only partially covers the use case with named browser profiles, per-profile user-data directories, sandbox browser isolation, and a process-level HTTP proxy; it still lacks first-class per-profile browser proxy config, agent/default browser-profile binding, sub-agent inheritance, and a reviewed concurrency/resource policy. The browser-lock and FSB comments add useful outside implementation context, and the matching implementation PR was closed unmerged after security review concerns.

## What This Changes

The issue requests per-agent isolated browser/profile instances, per-profile proxy support, concurrent execution guarantees, and docs/examples.

## Best Possible Solution

Design this as a coordinated browser plugin plus agent config contract: secret-safe per-profile proxy config, agent/default profile binding with sub-agent inheritance, explicit SSRF behavior, tab ownership, launch serialization, resource limits, diagnostics, docs, and focused regression tests, while coordinating with the broader proxy umbrella.

## Risks / Open Questions

- Proxy credentials could leak through Chrome command-line arguments, config/status snapshots, logs, process listings, or diagnostics if per-profile proxy support is added without a secret-safe design.
- Browser proxy routing hides the final connect target from OpenClaw, so per-profile proxies need explicit SSRF/private-network semantics rather than bypassing current navigation guardrails.
- Agent and sub-agent authority to select proxy-enabled browser profiles is unresolved, including inheritance, least privilege, and cross-agent state isolation.
- Concurrency still needs launch serialization, resource limits, queueing, tab ownership, and cleanup semantics beyond named profile storage.
