## Summary

Keep open. Current main still provides named isolated browser profiles and a global `browser.extraArgs` proxy workaround, but it does not implement the requested first-class `browser.profiles.<name>.proxy` contract or `agents.defaults.browser.profile` / `agents.list[].browser.profile` routing for concurrent agent browser runs. The related implementation PRs are closed unmerged, and the linked `m13v/browser-lock` project is useful prior art rather than shipped OpenClaw behavior.

## What This Changes

The issue requests first-class per-agent browser profile binding, concurrent isolated browser instances, and per-profile proxy configuration with docs and diagnostics.

## Best Possible Solution

Define a maintainer-reviewed, secret-aware browser profile proxy contract for locally managed profiles, add agent and sub-agent browser profile inheritance rules, expose redacted status/diagnostics, document examples, and cover the behavior with focused browser/config/agent tests while preserving existing remote CDP, existing-session, sandbox, and node-host semantics.

## Reproduction Assessment

Yes. This is reproducible as a configuration-contract gap: current strict schemas reject `browser.profiles.<name>.proxy` and `agents.*.browser.profile`, while browser tool calls only use an explicit `profile` argument and managed Chrome proxy routing is global via `browser.extraArgs`.

## Solution Assessment

Unclear for implementation, but not solved by current main. The requested direction is plausible, yet the best fix needs maintainer design around secret refs, redaction, SSRF/private-network behavior, sub-agent inheritance, and resource/concurrency limits before an automated implementation should proceed.
