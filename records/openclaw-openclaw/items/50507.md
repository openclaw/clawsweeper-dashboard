## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks native outbound WhatsApp location pins through the message tool, so the contribution is not obsolete, but the proposed patch misses required discovery, production outbound, and Gateway wiring, and it logs precise coordinates in structured logs.

## What This Changes

The PR adds a `location` message-tool action and WhatsApp/Baileys send path by adding coordinate schema fields, outbound plumbing, channel adapter types, and WhatsApp send API tests.

## Best Possible Solution

Revise the feature against current main: add a supported location-send contract or plugin action seam, advertise it in WhatsApp message-tool discovery, wire the current production WhatsApp outbound path to Baileys, add a validated Gateway location RPC or use the existing plugin-action Gateway path, cover message-tool/core/Gateway/WhatsApp tests, update user-facing docs/changelog if shipped, and keep precise coordinates out of logs.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current main lacks a `location` action and coordinate schema, while the PR branch's non-dry-run path calls an unregistered Gateway `location` method. No live WhatsApp session is needed to prove those routing failures.

## Solution Assessment

No. The PR is directionally useful, but it is not the best/safe implementation as written because it adds only part of the end-to-end contract and introduces coordinate logging; the safer path is a current-main revision that treats this as a complete action/Gateway/plugin contract change.
