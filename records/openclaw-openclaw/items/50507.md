## Summary

Keep open. Current main still does not implement native WhatsApp location sends, and this PR is not mergeable as-is because the proposed action is not discoverable, calls an unregistered Gateway method, is not wired into the active WhatsApp outbound surface, and logs raw coordinates.

## What This Changes

The PR adds a WhatsApp `location` message action through the message tool, core outbound path, WhatsApp Baileys send API, and related unit-test mocks.

## Best Possible Solution

Implement native WhatsApp location pins on current main by adding a discovered location action, a typed Gateway/protocol or plugin-action send route, production WhatsApp outbound support, privacy-safe logging, regression tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. Source inspection is enough: the PR's real send path calls Gateway method `location`, while both current main and the PR head expose no matching send-server method and WhatsApp discovery does not advertise `location`.

## Solution Assessment

No. The feature direction is reasonable, but this PR is not the best implementation yet because it misses discovery, Gateway/protocol, active outbound wiring, and privacy requirements on current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Expose location in WhatsApp discovery:** `src/agents/tools/message-tool.ts:125-130`
  - body: Configured message tools derive the valid `action` enum from channel discovery. The PR adds `latitude`/`longitude` fields, but WhatsApp discovery still advertises only `react` and `poll`, so normal configured tools reject `action: "location"` before these fields can be used.
  - confidence: 0.91
- **[P1] Wire location into the active WhatsApp outbound:** `extensions/whatsapp/src/outbound-adapter.ts:91-109`
  - body: This adds `sendLocation` to `whatsappOutbound`, but the shipped WhatsApp plugin path is built from the channel outbound surface, which still has no `sendLocation`. The core `sendLocation()` check will therefore throw `Unsupported location channel: whatsapp` before delivery.
  - confidence: 0.9
- **[P1] Register the gateway location method before calling it:** `src/infra/outbound/message.ts:424-426`
  - body: Every non-dry-run location send calls Gateway method `location`, but the Gateway send server registers `send` and `poll` only. Real WhatsApp sends run through the gateway path, so this will fail with an unknown method before reaching Baileys.
  - confidence: 0.93
- **[P2] Stop logging exact coordinates:** `extensions/whatsapp/src/send.ts:184-186`
  - body: Latitude and longitude are precise personal location data. Logging them in structured outbound logs can expose sensitive places in log sinks; keep the redacted JID and omit or coarsen the coordinates.
  - confidence: 0.92
- **[P3] Add the required changelog entry:** `src/agents/tools/message-tool.ts:125-130`
  - body: This is a user-facing WhatsApp/message-tool feature, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires a changelog entry for user-facing feature changes.
  - confidence: 0.86
