## Summary

Keep open: native WhatsApp location pins remain a plausible feature, but this PR is not mergeable because configured discovery, production outbound routing, and Gateway handling still do not deliver the action, and the patch logs precise coordinates.

## What This Changes

The PR adds a WhatsApp `location` message action, core outbound location types and call path, WhatsApp Baileys location send support, and related unit-test updates.

## Best Possible Solution

A current-main implementation should expose `location` through WhatsApp discovery, register an additive Gateway or plugin-action send route, wire the production WhatsApp outbound path, redact location data in logs, and update docs/changelog coverage.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: the PR's real-send path calls Gateway method `location`, while both current main and the PR head expose no matching send-server method and configured WhatsApp discovery does not advertise `location`.

## Solution Assessment

No. The feature direction is reasonable, but this PR is not the best implementation yet because it misses current discovery, Gateway, and production outbound contracts and introduces location PII logging.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Expose location through WhatsApp discovery:** `src/agents/tools/message-tool.ts:125-130`
  - body: Configured message tools derive valid actions from channel discovery, not only from the extra fields added here. WhatsApp discovery still advertises only `react` and `poll`, so normal configured tools will reject `action: "location"` before these coordinates can be used.
  - confidence: 0.91
- **[P1] Wire location into the shipped WhatsApp outbound:** `extensions/whatsapp/src/outbound-adapter.ts:91-109`
  - body: This adds `sendLocation` to `whatsappOutbound`, but the active WhatsApp plugin registers the outbound surface built from `createWhatsAppOutboundBase()`. That production adapter still has no location sender, so the core fallback will throw `Unsupported location channel: whatsapp` before delivery.
  - confidence: 0.9
- **[P1] Register the gateway location method before calling it:** `src/infra/outbound/message.ts:424-426`
  - body: Every non-dry-run location send calls Gateway method `location`, but the Gateway send server methods only register `send` and `poll`. Real WhatsApp sends run in gateway mode and will fail with an unknown method before reaching the WhatsApp sender.
  - confidence: 0.93
- **[P2] Stop logging exact coordinates:** `extensions/whatsapp/src/send.ts:184-186`
  - body: Latitude and longitude are precise personal location data. Logging them in structured outbound logs can expose sensitive places in log sinks; keep the redacted JID and omit or coarsen the coordinates.
  - confidence: 0.92
- **[P3] Add the required changelog entry:** `src/agents/tools/message-tool.ts:125-130`
  - body: This is a user-facing `feat` change to the message tool and WhatsApp channel, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires a changelog entry for user-facing feature changes.
  - confidence: 0.86
