## Summary

Keep this PR open: current main still lacks native WhatsApp location sends, so the contribution is not obsolete, but the branch is not merge-ready because it does not expose the action through current WhatsApp discovery, does not wire the production outbound surface, calls a missing Gateway method, and logs raw coordinates.

## What This Changes

The PR adds a `location` message-tool action with latitude/longitude schema, outbound/Gateway plumbing, channel adapter types, and WhatsApp Baileys send API tests for native location pins.

## Best Possible Solution

Land a current-main implementation that advertises `location` through WhatsApp discovery, adds an additive Gateway/plugin send contract or uses the existing plugin-action path, sends Baileys-native location payloads through the production WhatsApp outbound surface, keeps coordinates out of logs, and covers schema, Gateway, and WhatsApp tests with a changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current main has no `location` action/schema/Gateway handler, while the PR's non-dry-run path calls a `location` Gateway RPC that is not registered. No live WhatsApp session is needed to prove those routing failures.

## Solution Assessment

No. The direction is useful, but this PR is not the safest implementation as written because it threads only part of the current end-to-end contract and introduces coordinate logging; the safer path is a current-main revision across discovery, protocol, production outbound, tests, and logging.
