## Summary

Keep open: current main still lacks native WhatsApp location sends, and this PR is useful source work but remains incorrect because it does not expose the action through WhatsApp discovery, does not wire the production outbound surface, calls an unregistered Gateway method, and logs precise coordinates.

## What This Changes

The PR adds a WhatsApp `location` message-tool action with core outbound plumbing, new channel adapter types, WhatsApp Baileys send support, and send API tests.

## Best Possible Solution

Land a current-main implementation that advertises `location` through WhatsApp discovery, adds an additive Gateway/plugin send contract or equivalent plugin-action route, wires the production WhatsApp outbound adapter, omits precise coordinates from logs, and covers schema, Gateway, WhatsApp send, docs, and changelog updates.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current main has no `location` action/schema/Gateway handler, while the PR's non-dry-run path calls a `location` Gateway RPC that is not registered and does not reach the production WhatsApp outbound surface.

## Solution Assessment

No. The feature direction is reasonable, but this PR is not the best implementation yet because it extends only part of the current discovery, Gateway, and production outbound contracts and introduces location PII logging.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Expose location through WhatsApp action discovery:** `src/agents/tools/message-tool.ts:125-130`
  - body: Configured message tools derive their `action` enum from channel discovery. Adding coordinate fields here is not enough while WhatsApp discovery still advertises only `react` and `poll`, so normal configured tools will reject `action: "location"`.
  - confidence: 0.91
- **[P1] Wire location into the production WhatsApp outbound:** `extensions/whatsapp/src/outbound-adapter.ts:88-109`
  - body: This adds `sendLocation` to `whatsappOutbound`, but the shipped plugin registers `whatsappChannelOutbound` from `createWhatsAppOutboundBase()`. That production adapter still has no `sendLocation`, so location sends will fail as unsupported before delivery.
  - confidence: 0.9
- **[P1] Add the matching Gateway location method:** `src/infra/outbound/message.ts:424-426`
  - body: Every non-dry-run location send calls Gateway method `location`, but the Gateway send server methods only register `send` and `poll`. Real sends will fail with an unknown or unhandled method before reaching WhatsApp.
  - confidence: 0.93
- **[P2] Stop logging raw location coordinates:** `extensions/whatsapp/src/send.ts:184-186`
  - body: Latitude and longitude are precise personal location data. Logging them in structured outbound logs can expose sensitive places in log sinks; keep the redacted JID and omit or coarsen the coordinates.
  - confidence: 0.92
