## Summary

Keep open: current main has not implemented provider-isolated model loading, and this PR is useful but not mergeable as-is because it imports a non-exported schema, validates before the PI failure boundary, and emits unsanitized config-derived warnings.

## What This Changes

The PR adds a provider-sanitization pre-pass to raw config validation plus tests for mixed valid and invalid `models.providers` entries.

## Best Possible Solution

Implement provider isolation at the generated `models.json` or PI discovery boundary with PI-equivalent semantic validation, preservation of valid provider overrides, sanitized diagnostics, and regression coverage for mixed, all-invalid, missing-apiKey, invalid API enum, and generated-provider cases.

## Reproduction Assessment

Yes, source-reproducible. A valid custom provider with a `baseUrl` plus a second non-built-in provider with `models` but no `apiKey` can still flow into PI via `models.json`, while OpenClaw's current schema marks `apiKey` optional.

## Solution Assessment

No. The PR validates too early against a weaker schema and logs raw config text; the narrower maintainable fix belongs where OpenClaw generates or loads PI `models.json`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Export the provider schema before importing it:** `src/config/validation.ts:31`
  - body: The new named import of `ModelProviderSchema` cannot typecheck against current main because `src/config/zod-schema.core.ts` declares that schema as a local `const`. Export the schema intentionally or move the helper next to it before using it here.
  - confidence: 0.94
- **[P2] Recover at the models.json boundary:** `src/config/validation.ts:251`
  - body: This pre-pass only calls OpenClaw's zod schema, where `apiKey` is optional, before generated or merged providers are handed to PI. The reported failure happens later when PI semantically rejects a non-built-in provider with models but no `apiKey`, so the bad provider can still make PI drop unrelated overrides.
  - confidence: 0.9
- **[P2] Sanitize provider skip warnings:** `src/config/validation.ts:264`
  - body: The warning interpolates provider ids and validation issue text from user-editable config directly into `console.warn`. Existing config diagnostics sanitize path and message text first, so this can inject newlines or terminal control characters into logs.
  - confidence: 0.88
