## Summary

Keep open: this is a real-looking models/config bug and the linked discussion plus later user comment confirm the same symptom, but this PR misses the PI `models.json` validation boundary and adds an unsanitized warning path. It also still lacks after-fix real behavior proof.

## What This Changes

The PR adds a `sanitizeInvalidProviders` pre-validation pass to `src/config/validation.ts` plus a new Vitest file for provider-level config validation cases.

## Best Possible Solution

Implement provider isolation at the `models.json` generation or PI discovery boundary with PI-equivalent semantic validation, preserve valid provider overrides, skip only invalid provider entries, and emit sanitized diagnostics with focused regression coverage.

## Reproduction Assessment

Yes from source inspection. A non-built-in provider with custom models and no `apiKey` passes the OpenClaw schema but is rejected by PI `validateConfig`, which makes `loadCustomModels` return an empty custom result for the whole file.

## Solution Assessment

No. The PR validates the wrong boundary and logs untrusted config text directly; the safer fix is per-provider recovery where OpenClaw generates or loads PI `models.json`, using sanitized config diagnostics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Recover at the models.json boundary:** `src/config/validation.ts:251`
  - body: This pre-pass only runs `ModelProviderSchema.safeParse`, where `apiKey` is optional, before `models.json` generation. The reported failure happens later when PI rejects a non-built-in provider with custom models and no `apiKey`, so the bad provider can still poison the whole `models.json` load and drop unrelated `baseUrl` overrides.
  - confidence: 0.91

  - body: The warning interpolates the provider key and Zod reason directly from user config into `console.warn`. Existing config diagnostics sanitize path and message text before terminal output, so this can inject newlines or terminal control characters into logs.
  - confidence: 0.88
