## Summary

Keep open. This is a plausible bug-fix PR tied to the closed report and a follow-up user confirmation, but the current patch misses the PI semantic validation failure and adds unsafe warning output, so it should not merge as-is.

## What This Changes

The PR adds a `sanitizeInvalidProviders` pre-pass in raw config validation and a new Vitest file for mixed valid and invalid `models.providers` entries.

## Best Possible Solution

Implement provider isolation at the models.json generation or PI discovery boundary with PI-equivalent validation, preserve valid provider overrides, skip only invalid providers, and emit sanitized diagnostics with focused regression coverage.

## Reproduction Assessment

Yes, from source inspection. A non-built-in provider with custom models and no `apiKey` can pass OpenClaw schema validation, then PI `validateConfig` rejects the generated `models.json` and drops unrelated provider overrides.

## Solution Assessment

No. The PR validates only OpenClaw Zod schema before models.json generation, while the reported failure is later PI semantic validation; the fix should live at the generated models.json or PI discovery boundary and use sanitized diagnostics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Recover from PI semantic provider failures:** `src/config/validation.ts:251`
  - body: This pre-pass only calls `ModelProviderSchema.safeParse`, where `apiKey` is optional, and it runs before generated providers are written to `models.json`. The reported missing-apiKey custom-provider case still reaches PI `validateConfig`, so the bad provider can still make the whole custom models load drop valid overrides.
  - confidence: 0.91
- **[P2] Sanitize provider warning text:** `src/config/validation.ts:264`
  - body: This warning interpolates the provider key and validation reason directly from user config into `console.warn`. Existing config diagnostics sanitize path and message before terminal output; route this through that path or sanitize the fields to prevent crafted ids/messages from injecting newlines or ANSI escapes into logs.
  - confidence: 0.88
