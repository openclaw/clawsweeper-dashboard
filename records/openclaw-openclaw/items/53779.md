## Summary

Keep open: the underlying provider-routing bug is real, but this PR validates the wrong boundary and adds unsanitized warning output, so it needs rework and real behavior proof before merge.

## What This Changes

The PR adds a `models.providers` pre-validation pass in `src/config/validation.ts` plus focused Vitest coverage for valid, mixed-invalid, warning, and non-provider-invalid configs.

## Best Possible Solution

Implement provider isolation at the generated `models.json` or PI discovery boundary with PI-equivalent validation, sanitized diagnostics, focused regression tests, and real gateway proof that valid provider overrides survive one bad provider.

## Reproduction Assessment

Yes, by source and dependency inspection: current main writes all final providers into one PI `models.json`, and the pinned PI registry drops all custom models when one non-built-in custom provider with models lacks `apiKey`. I did not run a live gateway reproduction in this read-only sweep.

## Solution Assessment

No. The PR validates raw OpenClaw provider shape before generation, but the reported failure happens later in PI semantic validation of generated `models.json`; the maintainable fix belongs at that boundary with sanitized diagnostics.
