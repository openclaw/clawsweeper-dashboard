## Summary

Keep open: this is a real-looking bug, but the patch does not catch the reported PI semantic failure and adds an unsanitized warning path. The contributor also has not provided after-fix real behavior proof for the mixed-provider gateway flow.

## What This Changes

The PR adds a pre-validation pass in `src/config/validation.ts` that drops providers failing `ModelProviderSchema.safeParse` and adds Vitest coverage for mixed valid/invalid `models.providers`.

## Best Possible Solution

Implement provider isolation where OpenClaw generates or loads `models.json`, using PI-equivalent per-provider semantic validation and sanitized diagnostics while preserving valid providers' baseUrl, auth, headers, models, overrides, and metadata.

## Reproduction Assessment

Yes at source level. A non-built-in provider with `models` but no `apiKey` passes OpenClaw's schema, then PI `validateConfig` throws during `models.json` loading and returns an empty custom models result; I did not run a live gateway because this sweep is read-only.

## Solution Assessment

No. The current PR validates too early and against the wrong contract, so it does not reliably recover from the reported PI failure and should move the isolation to the generated/runtime `models.json` boundary with sanitized warning output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Recover from PI semantic provider failures:** `src/config/validation.ts:251`
  - body: The pre-pass only calls `ModelProviderSchema.safeParse`, where `apiKey` is optional, and it runs before generated or merged providers are written to `models.json`. The reported failure occurs later in PI when a non-built-in provider has custom models but no `apiKey`, so this patch can leave the bad provider in place and PI will still reject the whole custom models file, dropping unrelated overrides.
  - confidence: 0.91
- **[P2] Sanitize provider warning text:** `src/config/validation.ts:264`
  - body: This warning interpolates the provider key and Zod issue text directly from user config into `console.warn`. Other config diagnostics sanitize path and message text before terminal output, so a crafted provider id or validation message could inject newlines or terminal control characters into logs.
  - confidence: 0.88
