## Summary

Keep open: this targets a real provider-validation bug, but the patch recovers at the wrong layer and adds unsanitized terminal output, so it should not merge as-is.

## What This Changes

The PR adds a `sanitizeInvalidProviders` pre-pass to config validation plus a new Vitest file for mixed valid/invalid model provider configs.

## Best Possible Solution

Implement provider isolation at the generated `models.json` or PI discovery boundary with PI-equivalent per-provider semantic validation, preserve valid overrides, skip only invalid entries, and emit sanitized diagnostics with focused regression coverage.

## Reproduction Assessment

Yes, from source inspection. A non-built-in provider with custom models and no `apiKey` passes OpenClaw schema validation but is rejected by PI `validateConfig`, which makes PI return an empty custom models result for the whole file.

## Solution Assessment

No. The PR validates the OpenClaw config schema before models.json generation, but the reported failure is a later PI semantic validation failure, and the warning path should use sanitized config diagnostics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Recover from PI provider semantic failures:** `src/config/validation.ts:251`
  - body: This pre-pass only calls `ModelProviderSchema.safeParse`, where `apiKey` is optional, before generated providers are written to `models.json`. The reported failure happens later when PI rejects a non-built-in provider with custom models and no `apiKey`, so the bad provider can still poison the whole custom models load and drop unrelated overrides.
  - confidence: 0.91
- **[P2] Sanitize provider warning output:** `src/config/validation.ts:264`
  - body: The warning interpolates the provider key and Zod reason directly from user-controlled config into `console.warn`. Existing config diagnostics sanitize path and message text before terminal output, so this can inject newlines or terminal control characters into logs.
  - confidence: 0.88
