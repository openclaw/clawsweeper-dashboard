## Summary

Keep open: the underlying provider-poisoning bug is real and source-reproducible, but this branch validates the raw OpenClaw config instead of the generated PI-facing `models.json` boundary where the reported failure happens, lacks real behavior proof, and adds an unsanitized warning path.

## What This Changes

The branch adds a `models.providers` pre-validation pass in `src/config/validation.ts` that skips invalid provider entries and adds Vitest coverage for mixed valid and invalid provider configs.

## Best Possible Solution

Implement provider isolation at the generated `models.json` or PI discovery boundary with PI-equivalent validation, sanitized diagnostics, regression tests for mixed and all-invalid providers, and real gateway proof that valid baseUrl overrides survive one bad provider.

## Reproduction Assessment

Yes. Source and dependency inspection show current main writes all final providers into one PI `models.json`, and the pinned PI registry drops all custom models when one non-built-in custom provider with models lacks `apiKey`; I did not run a live gateway reproduction.

## Solution Assessment

No. The PR validates raw OpenClaw provider shape before generation, but the reported failure happens later in PI semantic validation of generated `models.json`; the maintainable fix belongs at that boundary with sanitized diagnostics.
