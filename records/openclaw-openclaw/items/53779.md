## Summary

Keep open: the report is a real-looking bug, but this PR does not fix the reported PI semantic validation failure and introduces an unsanitized warning path; it also lacks after-fix real behavior proof and is currently conflicting.

## What This Changes

The PR adds a raw config pre-pass that drops OpenClaw-Zod-invalid model providers with a warning, plus Vitest coverage for mixed valid and invalid providers.

## Best Possible Solution

Implement provider isolation at the models.json generation or PI discovery boundary with PI-equivalent validation, sanitized diagnostics, regression coverage, and real gateway proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a non-built-in provider with custom models and no `apiKey` makes PI `validateConfig` throw, and `loadCustomModels` then returns an empty result that drops unrelated overrides.

## Solution Assessment

No. The PR validates only the OpenClaw Zod shape before generated `models.json`, while the reported failure is PI semantic validation after generation; the warning also needs existing terminal sanitization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Recover at the PI models.json boundary:** `src/config/validation.ts:251`
  - body: This pre-pass only calls `ModelProviderSchema.safeParse`, where `apiKey` is optional, before generated/merged providers are written to `models.json`. The reported missing-apiKey custom-provider case still reaches PI validation, so one bad provider can still make the whole custom models file fail and drop unrelated overrides.

Existing config diagnostics sanitize path and message text before terminal output, so crafted provider ids or issue text could inject newlines or terminal control characters into logs.
  - confidence: 0.88
