## Summary

Keep open: this is a real provider/model loading bug, but the PR does not fix the PI semantic validation boundary that drops valid providers, and it adds an unsanitized warning path for user-controlled config text.

## What This Changes

The branch adds a `sanitizeInvalidProviders` pre-validation pass in `src/config/validation.ts` plus a new Vitest file for mixed valid and invalid `models.providers` entries.

## Best Possible Solution

Implement provider isolation at the generated `models.json` or PI discovery boundary with PI-equivalent semantic validation, preserve valid provider overrides, and emit sanitized diagnostics with focused regression and gateway proof.

## Reproduction Assessment

Yes, from source inspection rather than an executed run: current main writes all providers into one `models.json`, and PI 0.75.1 drops all custom models when one non-built-in provider with models lacks `apiKey`. The PR discussion also includes a user report on OpenClaw 2026.4.10 with the same disappearance symptom.

## Solution Assessment

No. The current PR validates the OpenClaw Zod shape before generation, but the reported failure is PI semantic validation after generation; the safer fix belongs at the generated `models.json` or PI discovery boundary with sanitized diagnostics.
