## Summary

Keep open: the PR targets a real model-provider failure mode, but the proposed pre-validation runs at the wrong boundary and does not catch the reported PI semantic error; it also adds an unsanitized warning path and lacks contributor after-fix runtime proof.

## What This Changes

The PR adds a `sanitizeInvalidProviders` pre-pass in config validation plus a new Vitest file for mixed valid and invalid `models.providers` entries.

## Best Possible Solution

Implement provider isolation at the generated `models.json` or PI discovery boundary with PI-equivalent semantic validation, preserve valid overrides, skip only invalid providers, and emit sanitized diagnostics with focused regression coverage.

## Reproduction Assessment

Yes, from source inspection. A non-built-in provider with custom models and no `apiKey` can pass OpenClaw schema validation, then PI `validateConfig` rejects the whole generated `models.json` and drops unrelated provider overrides.

## Solution Assessment

No. The PR validates the OpenClaw schema too early, while the reported failure is a later PI semantic validation failure, and the new warning should use the existing sanitized config diagnostic path.

## Review Findings

- **[P2] Recover from PI semantic provider failures:** `src/config/validation.ts:251`
  - body: This pre-pass only calls `ModelProviderSchema.safeParse`, where `apiKey` is optional, before generated providers are written to `models.json`. Existing config diagnostics sanitize path and message text before terminal output; without that, crafted provider ids or issue text can inject newlines or terminal control characters into logs.
  - confidence: 0.88
