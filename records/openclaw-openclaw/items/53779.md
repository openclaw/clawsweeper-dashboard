## Summary

Keep open: the PR targets a real source-reproducible models.json failure, but the current patch recovers at the wrong boundary, misses the reported missing-apiKey semantic failure, emits unsanitized config-derived warning text, and lacks after-fix real behavior proof.

## What This Changes

The branch adds a config-validation pre-pass that drops schema-invalid `models.providers` entries and logs warnings, plus Vitest coverage for mixed provider validation cases.

## Best Possible Solution

Implement provider isolation at the models.json generation or runtime discovery boundary, preserve valid provider overrides, skip only PI-semantically invalid entries, and route warnings through sanitized diagnostics with regression coverage.

## Reproduction Assessment

Yes, from source inspection. A non-built-in remote provider with custom models and no `apiKey` passes OpenClaw schema, is written into models.json, then PI validation returns an empty custom-model result for the whole file; I did not execute it in this read-only sweep.

## Solution Assessment

No. The current PR validates raw OpenClaw provider schema before generated/merged models.json reaches PI, but the reported failure happens in PI semantic validation and should be handled there or immediately before that handoff with sanitized diagnostics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Recover at the PI models.json boundary:** `src/config/validation.ts:251`
  - body: Move the recovery out of raw config validation. This pre-pass only runs `ModelProviderSchema.safeParse`, where `apiKey` is optional, before generated/merged providers are written to models.json; the reported failure happens later when PI rejects a non-built-in provider with custom models and no `apiKey`, so the bad provider can still poison the whole custom models load and drop unrelated overrides.
  - confidence: 0.9
- **[P2] Route provider warnings through sanitized diagnostics:** `src/config/validation.ts:264`
  - body: Route this warning through the existing sanitized config diagnostic path. `key` and `reason` are user-controlled config-derived text; writing them directly to `console.warn` allows newline or terminal-control injection in logs, unlike current config issue output that sanitizes path and message.
  - confidence: 0.88
