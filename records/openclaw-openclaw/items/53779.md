## Summary

Keep open. The PR targets a real models.json failure confirmed by #21584 and a later OpenClaw 2026.4.10 reproduction, but the branch is not safe to merge because it imports a non-exported schema, validates too early to catch the reported PI semantic failure, and logs unsanitized config-derived warning text.

## What This Changes

The PR adds a `sanitizeInvalidProviders` pre-validation pass in `src/config/validation.ts` and a new Vitest file covering mixed valid/invalid provider config cases.

## Best Possible Solution

Implement provider isolation at the generated `models.json` or PI discovery boundary with PI-equivalent semantic validation, sanitized diagnostics, and regression coverage for mixed, all-invalid, missing-apiKey, invalid API enum, and generated-provider cases.

## Reproduction Assessment

Yes. A source-level reproduction is clear: include one valid provider with a custom baseUrl and one non-built-in custom-model provider without `apiKey` in `models.json`; PI 0.71.1 rejects the whole custom models result, while this PR’s OpenClaw schema pre-pass would not reject that missing-apiKey shape.

## Solution Assessment

No. The proposed pre-pass is earlier and weaker than the failing PI contract, and it bypasses existing sanitized config diagnostics; the safer solution is recovery at the models.json generation/discovery boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Export the schema before importing it:** `src/config/validation.ts:31`
  - body: `ModelProviderSchema` is declared as a local const in `src/config/zod-schema.core.ts`, and this PR does not change that file. The new named import therefore cannot typecheck on current main until the schema is intentionally exported or the helper is moved next to the schema.
  - confidence: 0.92
- **[P2] Recover at the models.json boundary:** `src/config/validation.ts:251`
  - body: This pre-pass only runs `ModelProviderSchema.safeParse`, where `apiKey` is optional, before generated or merged providers are loaded from `models.json`. A non-built-in provider with custom models but no `apiKey` can still reach PI, which rejects the whole custom models file and drops unrelated valid overrides.
  - confidence: 0.9
- **[P2] Sanitize provider skip warnings:** `src/config/validation.ts:264`
  - body: The warning interpolates the provider key and Zod issue text directly from user-editable config into `console.warn`. Existing config diagnostics sanitize path and message text first, so this can allow log forging or terminal control-character injection.
  - confidence: 0.88
