## Summary

Keep this PR open as a plausible bug-fix attempt, but the current patch should not merge: the reported models.json failure path is real, while this branch validates the wrong layer, emits unsanitized user-controlled warning text, lacks real behavior proof, and is currently conflicting with main.

## What This Changes

The PR adds a `sanitizeInvalidProviders` pre-pass to raw config validation and a Vitest file for mixed valid/invalid `models.providers` entries.

## Best Possible Solution

Implement provider isolation at the generated models.json or PI discovery boundary using PI-equivalent validation, preserve valid provider overrides, skip only invalid providers, and emit sanitized diagnostics with regression and real gateway proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: generate one valid provider override plus one non-built-in custom provider with models and no `apiKey`, then PI 0.74.0 validation rejects the whole models.json load and drops unrelated overrides.

## Solution Assessment

No. The PR validates only the OpenClaw Zod shape before models.json generation, while the reported failure is PI semantic validation after generation; the warning path also needs sanitized diagnostics.
