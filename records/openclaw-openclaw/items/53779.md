## Summary

Keep open: the underlying provider-validation bug is real, but this PR does not fix the PI runtime validation boundary, adds an unsanitized warning path, and still lacks real after-fix behavior proof.

## What This Changes

The PR adds a `sanitizeInvalidProviders` pre-validation pass for `models.providers` in `src/config/validation.ts` plus a new Vitest file for mixed valid and invalid provider entries.

## Best Possible Solution

Implement provider isolation at the generated `models.json` or PI discovery boundary with PI-equivalent validation, sanitized diagnostics, regression tests for mixed/all-invalid providers, and real gateway proof that valid baseUrl overrides survive one bad provider.

## Reproduction Assessment

Yes. Source inspection shows current main still writes all providers into one generated `models.json`, and the pinned PI registry drops all custom models when one non-built-in custom provider lacks `apiKey`; the PR discussion also includes a matching user report on OpenClaw 2026.4.10.

## Solution Assessment

No. The PR validates the OpenClaw Zod shape before generation, but the reported failure happens later in PI semantic validation of generated `models.json`; the maintainable fix belongs at that runtime boundary with sanitized diagnostics.
