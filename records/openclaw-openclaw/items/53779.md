## Summary

Keep open. The PR addresses a real models.json/provider-isolation bug, but the current patch is not correct against current main: it imports a non-exported schema, validates before the PI semantic failure boundary, and logs config-derived text without terminal sanitization.

## What This Changes

The PR adds a pre-validation pass that drops invalid `models.providers` entries during raw config validation and adds unit tests for mixed provider validation.

## Best Possible Solution

Implement provider isolation at the generated models.json or runtime discovery boundary with PI-equivalent semantic validation, sanitized diagnostics, and regression coverage for mixed valid/invalid providers while preserving valid baseUrl and auth overrides.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a valid custom provider plus a non-built-in provider with custom models and no apiKey reaches PI models.json validation, where PI rejects the whole custom models result and OpenClaw loses unrelated overrides.

## Solution Assessment

No. The PR validates too early against a weaker OpenClaw schema, misses the reported PI semantic failure, and adds unsanitized warning output; the safer fix belongs at the models.json generation or PI discovery boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Export the provider schema before importing it:** `src/config/validation.ts:31`
  - body: The new named import cannot typecheck against current main because `src/config/zod-schema.core.ts` declares `ModelProviderSchema` as a local `const`, not an export. Export it intentionally or keep the helper next to the schema before using it here.
  - confidence: 0.94
- **[P2] Recover from PI semantic provider failures:** `src/config/validation.ts:251`
  - body: This pre-pass only runs OpenClaw's zod schema, where `apiKey` is optional, before generated or merged providers are handed to PI. The reported failure occurs later when PI rejects a non-built-in provider with models but no `apiKey`, so the bad provider can still make PI drop unrelated overrides.
  - confidence: 0.9
- **[P2] Sanitize provider skip warnings:** `src/config/validation.ts:265`
  - body: The warning interpolates provider ids and validation issue text from user-editable config directly into `console.warn`. Existing config diagnostics sanitize path and message text first, so crafted config can inject newlines or terminal control characters into logs.
  - confidence: 0.88
