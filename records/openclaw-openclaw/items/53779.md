## Summary

Keep open: the linked issue and follow-up user comment describe a real current-main bug, but this PR does not recover from the PI semantic validation failure that drops valid provider overrides and it adds an unsanitized warning path.

## What This Changes

The PR adds a raw config pre-validation pass that drops `models.providers` entries failing `ModelProviderSchema` and adds Vitest coverage for mixed valid/invalid providers.

## Best Possible Solution

Implement provider isolation at the generated `models.json` or PI discovery boundary with PI-equivalent validation, skip only invalid providers, preserve valid overrides, and emit sanitized diagnostics with regression and real gateway proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main writes providers into one `models.json`, and PI 0.74.0 drops the whole custom models result when one non-built-in provider with models lacks `apiKey`.

## Solution Assessment

No. This PR validates only the OpenClaw Zod shape before generation, while the reported failure is PI semantic validation after generation; the warning path also needs sanitized diagnostics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Existing config diagnostics sanitize path and message text before terminal output; crafted provider ids or issue text could inject newlines or terminal control characters into logs.
  - confidence: 0.88
