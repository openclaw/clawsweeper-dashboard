## Summary

Keep open, but block merge: the underlying models.json failure is credible and was reinforced by a user hitting it on OpenClaw 2026.4.10, but this patch validates the wrong boundary, misses the PI semantic missing-apiKey failure, introduces unsanitized warning output, and still lacks real behavior proof.

## What This Changes

The PR adds a `sanitizeInvalidProviders` pre-validation pass to config validation plus a new Vitest file for mixed provider validation cases.

## Best Possible Solution

Move recovery to the models.json generation or runtime discovery boundary, apply PI-equivalent per-provider semantic validation, preserve valid provider overrides, skip only invalid provider entries, and emit sanitized diagnostics with regression coverage.

## Reproduction Assessment

Yes from source inspection. A models.json containing a valid provider override plus a non-built-in provider with custom models and no apiKey reaches PI via `discoverModels`, where PI validation returns an empty custom-model result for the whole file; I did not execute it in this read-only sweep.

## Solution Assessment

No. The PR validates OpenClaw config schema before models.json generation, but the reported failure happens later in PI semantic validation, so the narrow maintainable fix belongs at models.json generation/runtime discovery and should use sanitized diagnostics.
