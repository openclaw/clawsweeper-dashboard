## Summary

Keep open for maintainer review: the patch appears correct and well-proven for default DeepInfra behavior, but the video endpoint migration can affect custom inference-only DeepInfra proxies and needs maintainer acceptance before merge.

## What This Changes

The PR moves DeepInfra video generation from native inference URLs to the OpenAI-compatible async videos endpoint, tags DeepInfra DeepSeek-family models with `thinkingFormat: "deepseek"`, removes unsupported video guidance options, and updates focused docs/tests.

## Best Possible Solution

Land the provider-local endpoint and DeepSeek dialect fixes after a maintainer accepts the custom-proxy compatibility posture, or add a compatibility path first if inference-only DeepInfra proxies are a supported upgrade contract.

## Reproduction Assessment

Yes, at source level: current main still posts DeepInfra video requests to native `/v1/inference/<model>` and lacks DeepInfra DeepSeek-family `thinkingFormat: "deepseek"` tagging. I did not rerun a live current-main failure, but the PR body's after-fix live proof was inspected and matches the changed source path.

## Solution Assessment

Yes for default DeepInfra behavior: the patch stays inside the DeepInfra plugin, uses the OpenAI-compatible video job surface, and feeds existing core DeepSeek compat handling. The safer alternative only matters if maintainers require old custom native-video proxies to keep working.
