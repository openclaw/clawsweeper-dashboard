## Best Possible Solution

Land the provider-local endpoint and DeepSeek dialect fix after a maintainer explicitly accepts the DeepInfra video compatibility tradeoff and required checks finish.

## Reproduction Assessment

Yes at source level: current main still posts DeepInfra video requests to native `/v1/inference/<model>` and does not tag DeepInfra DeepSeek-family rows as `thinkingFormat: "deepseek"`. I did not rerun a live current-main failure; the PR provides after-fix live proof.

## Solution Assessment

Yes for default DeepInfra behavior: the patch stays in the provider plugin, reuses the shared async polling helper, and feeds existing core DeepSeek parsing via model compat metadata. The safer alternative only applies if maintainers require backward compatibility for custom native-video proxies.
