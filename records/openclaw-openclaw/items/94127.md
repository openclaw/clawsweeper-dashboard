## Summary

Keep open: current main still prefers a supplied gateway method registry in `handleGatewayRequest`, the latest release shipped the same dispatch path, and an open linked implementation candidate already owns the fix path.

## What This Changes

The issue requests fixing plugin-registered gateway RPC methods so `openclaw gateway call <method>` dispatches through the live plugin registry instead of returning an unknown-method error.

## Best Possible Solution

Review and land one canonical fix such as https://github.com/openclaw/openclaw/pull/94154, with stale-registry regression coverage and live CLI proof, then close this issue after merge.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main passes a supplied startup `methodRegistry` into `handleGatewayRequest`, while plugin registration populates the active registry; a stale supplied registry can therefore produce the reported unknown-method dispatch. I did not run a live gateway install/restart/CLI repro in this read-only review.

## Solution Assessment

Yes. The best fix belongs in gateway dispatch: use the live request registry for plugin RPC handler lookup, preserve scope authorization, and prove it with the stale-registry regression plus a live `openclaw gateway call` check on the landing PR.
