- `impact:other`: The issue breaks externally invoked plugin RPC methods, a maintainer-visible gateway/plugin contract failure outside the specific owned impact labels.

## Summary

Keep this issue open: current main still lets the WebSocket gateway path pass a supplied attached method registry into dispatch, and `handleGatewayRequest` still uses that supplied registry for handler lookup before it can consult the live active plugin registry. The latest release is also still affected, and the linked candidate PRs are open but currently conflicting against main.

## What This Changes

Fix plugin-registered gateway RPC methods so `openclaw gateway call <method>` dispatches methods shown by `plugins inspect --runtime` instead of returning `unknown method`.

## Best Possible Solution

Land one canonical gateway-dispatch fix that uses the live plugin registry for request dispatch, with a stale-registry regression test and live CLI proof.

## Reproduction Assessment

Yes, at source level. Current main can pass a stale attached `methodRegistry` into `handleGatewayRequest`, and handler lookup then misses plugin methods registered only in the live active registry.

## Solution Assessment

Yes, the best fix belongs in gateway dispatch rather than plugin workarounds or docs-only guidance. The landing branch should consolidate the open candidates, keep the plugin SDK contract intact, and prove stale-registry dispatch with tests and live CLI output.
