## Summary

Keep this issue open: current main and the latest release still prefer a supplied gateway method registry before rebuilding from live plugin registration state, so the reported plugin RPC path remains broken. The open linked PR is the right candidate fix and should be reviewed or landed before this issue closes.

## What This Changes

The issue asks for plugin-registered gateway RPC methods shown by runtime inspection to be callable through `openclaw gateway call <method>`, or for the unsupported boundary to be documented clearly.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/94154, or an equivalent attached-registry-first/live-registry-fallback fix with end-user CLI proof, then close this issue as implemented.

## Reproduction Assessment

Yes. Source inspection shows current main receives an attached startup registry in the WebSocket request path and selects it before rebuilding from the live plugin registry that `registerGatewayMethod` populates.

## Solution Assessment

Yes, the best fix belongs at the gateway dispatch chokepoint: preserve attached registries when they own the method, and fall back to the live plugin registry for stale misses. The open linked PR implements that shape but still needs end-user-path proof before landing.
