## Summary

Keep open: current main and the latest v2026.6.8 release still prefer the WebSocket-supplied gateway method registry snapshot before rebuilding from the live plugin registry, while `registerGatewayMethod` records plugin handlers in the live registry. Two open fix PRs already target the same bug, so this issue should remain open until maintainers land one canonical fix.

## What This Changes

The issue reports that plugin-registered gateway RPC methods appear in runtime plugin inspection but fail through `openclaw gateway call` with `unknown method`.

## Best Possible Solution

Keep this issue open until one canonical fix lands; dispatch should resolve plugin gateway RPC calls from the live registry that `registerGatewayMethod` populates and cover the stale-registry case with regression proof.

## Reproduction Assessment

Yes, at source level: current main passes a startup `methodRegistry` snapshot into `handleGatewayRequest`, which chooses it over rebuilding from the active plugin registry that holds plugin-registered handlers. I did not run a live plugin install plus gateway call in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to make the gateway-call dispatcher use the live plugin method registry or explicitly document a different supported boundary. The open candidate fix with a stale-registry regression test is the right direction, with live CLI proof still needed before landing.
