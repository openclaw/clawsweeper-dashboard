## Summary

Keep this issue open: current main and the latest stable release still select a caller-supplied Gateway method registry before rebuilding from the live plugin registry, so plugin-registered methods can still miss dispatch. The open linked PR https://github.com/openclaw/openclaw/pull/94154 is the canonical fix candidate and should be reviewed, landed, or rejected before this issue closes.

## What This Changes

The issue asks for `openclaw gateway call <plugin-method>` to dispatch plugin-registered RPC methods that are visible in runtime plugin inspection instead of returning `unknown method`.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/94154 as the canonical attached-registry-first live-fallback dispatch fix, then close this issue after the merged fix is proven on the user path.

## Reproduction Assessment

Yes. Source inspection shows the WebSocket path passes an attached method registry into `handleGatewayRequest`, and current main chooses that snapshot before the live plugin registry that `registerGatewayMethod` updates; I did not run the live install/restart repro.

## Solution Assessment

Yes for the linked fix direction. The best fix belongs at the gateway dispatch chokepoint: use the attached registry when it owns the method and fall back to the live plugin registry for stale misses, rather than documenting a workaround or replacing the attached registry unconditionally.
