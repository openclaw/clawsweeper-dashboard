## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested provider/gateway-specific stripping, and the PR branch only adds an opt-in flag that no live provider/tool assembly path sets, so the reported strict-schema failure would still occur in normal operation.

## What This Changes

The PR adds an opt-in `simplifiedSchemaForGateway` flag that builds a stripped `sessions_spawn` schema and forwards that flag from `createOpenClawTools` into `createSessionsSpawnTool`.

## Best Possible Solution

Keep a provider/deployment-aware `sessions_spawn` schema filter, but wire it through a real model/provider compatibility or explicit config path so strict OpenAI-compatible gateways receive the reduced schema while default native ACP/subagent users keep the full schema. Add focused schema-shape tests for the default, ACP-unavailable, and strict-gateway cases.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: inspect the `sessions_spawn` tool produced by the current live tool assembly path and it still includes the strict-provider-unfriendly subagent fields, while the PR branch leaves the new strip flag unset by non-test callers.

## Solution Assessment

No. The branch is a plausible starting point, but an internal opt-in flag without provider/config activation is not the best fix because it does not change the failing runtime path; the safer solution is to activate it through a deliberate compatibility surface with tests.
