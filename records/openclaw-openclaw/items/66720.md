## Summary

Keep open: the provider-compatibility problem is still real, but this PR is not mergeable as-is because the new schema-stripping flag only passes through an option that no live provider path sets, and the PR still lacks real strict-provider proof.

## What This Changes

The branch adds a `simplifiedSchemaForGateway` option that filters selected `sessions_spawn` schema properties and forwards that option from `createOpenClawTools` to `createSessionsSpawnTool`.

## Best Possible Solution

Use a deliberate provider/model-compat or config-backed schema-property filter in the live tool-preparation path, preserving ACP/subagent affordances for native runtimes and covering it with focused schema tests plus redacted strict-provider proof.

## Reproduction Assessment

Yes. Source reproduction is high-confidence: the PR's option only has an effect if an upstream caller supplies it, while the current `pi-tools` and gateway tool-resolution paths do not set any `sessions_spawn` property-stripping option.

## Solution Assessment

No. The schema filtering helper is useful, but the proposed wiring does not activate in a real provider path, so ordinary strict OpenAI-compatible requests can still emit the unstripped `sessions_spawn` schema.
