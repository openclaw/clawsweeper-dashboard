## What This Changes

The PR threads a Feishu bot `union_id` from probe/startup/state into message parsing so `@mention` detection can fall back from `open_id` in multi-bot groups.

## Best Possible Solution

Rewrite the fix on current main around a Feishu-documented or live-proven stable bot identity source, then cover text, post, mention-forward, slash-command stripping, debounce behavior, regression tests, and redacted live Feishu proof before merge.

Full review comments:

- **[P1] Prove the union_id source before using it:** `extensions/feishu/src/probe.ts:137`
  - body: The patch reads `bot?.union_id` from `bot/v3/info`, but the branch's response type only declares `bot_name` and `open_id`, the linked issue says that field is absent, and the official Feishu response contract I checked does not list it.   - confidence: 0.96
- **[P2] Strip the matched fallback mention before command parsing:** `extensions/feishu/src/bot.ts:797`
  - body: `checkBotMentioned` can return true via `botUnionId`, but `normalizeMentions` still strips only `botOpenId`. In the mismatch case, `@Bot /help` keeps the bot mention in content and can still break slash-command detection in the same scenario the PR is trying to fix.
  - confidence: 0.88
- **[P2] Carry fallback identity into rich-text post matching:** `extensions/feishu/src/bot.ts:479`
  - body: The fallback only applies when top-level `message.mentions` is populated; the rich-text post branch still compares parsed mention IDs to `botOpenId` only. Post messages with the same app-scoped mismatch can still be treated as unmentioned.
  - confidence: 0.84
- **[P2] Pass fallback identity through debounce merging:** `extensions/feishu/src/monitor.account.ts:251`
  - body: This handoff passes `botUnionId` to the final message handler, but debounce text and retained bot mentions are still derived from open-id-only checks before that handoff. Batched messages can therefore still lose the bot mention under `requireMention`.
  - confidence: 0.83
