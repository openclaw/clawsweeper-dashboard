## Summary

This PR addresses a real Feishu multi-bot delivery bug, but it cannot fix the reported setup because the official bot-info endpoint does not provide the proposed `union_id`; it also leaves mention stripping and debounce aggregation on open-id-only logic. The branch now conflicts across a substantially refactored current-main message path, so it needs a current-main redesign rather than a mechanical rebase.

## What This Changes

The PR threads a Feishu bot `union_id` from startup probing into message parsing as a fallback for multi-bot group mention detection.

## Best Possible Solution

Replace the stale patch with a current-main implementation that obtains or derives a documented stable bot identity, centralizes matching and own-mention stripping in one resolver, and proves direct, debounced, and slash-command flows in a real multi-bot group.

## Reproduction Assessment

Yes at source level: open-id-only matching rejects a mention when the event carries another app's scoped identifier, consistent with the reporter's cross-token evidence. No live current-main reproduction was run during this read-only review.

## Solution Assessment

No; the proposed startup field is absent from the official endpoint contract, and the patch does not establish one identity invariant across all current message-processing paths.
