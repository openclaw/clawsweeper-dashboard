Full review comments:

- **[P1] Use an identity source that actually returns `union_id`:** `extensions/feishu/src/probe.ts:137`
  - body: `/open-apis/bot/v3/info` does not expose `union_id` in the documented response (and this file's `FeishuBotInfoResponse` likewise declares only `bot_name` and `open_id`). Consequently `botUnionId` is never populated and the new fallback cannot fix the reported multi-bot delivery failure. Replace this with a supported identity source or mapping before threading the value through runtime state.
  - confidence: 0.98
- **[P2] Resolve the bot mention before normalizing message text:** `extensions/feishu/src/bot.ts:787`
  - body: A message may now be accepted because its mention matches `botUnionId`, but `normalizeMentions` still strips only the stored `botOpenId`. In the exact cross-app case, the bot's `<at>` token remains in the parsed content and can prevent `/model`, `/reset`, and similar group commands from being recognized. Find the matched mention by either identifier and strip its event `open_id`.
  - confidence: 0.94
- **[P2] Carry union-id matching through the debounce path:** `extensions/feishu/src/monitor.account.ts:251`
  - body: The added `botUnionId` reaches final dispatch, but debounce parsing and `resolveFeishuDebounceMentions` still identify the bot only by `open_id`. When the mention occurs on an earlier event in a debounced batch, the merged event can lose the only union-id match and be dropped by `requireMention`. Pass the resolved bot identity through debounce text parsing and merged-mention selection too.
  - confidence: 0.93
