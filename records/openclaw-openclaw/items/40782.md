## Summary

Keep open: this PR targets a real Feishu multi-bot mention bug, but it is not merge-ready because the proposed union-id source is unverified/likely absent, several mention-sensitive paths remain open-id-only, the branch is stale against current main’s Feishu refactor, and the contributor has not provided after-fix live behavior proof.

## What This Changes

The PR threads a Feishu bot union ID through probe, startup, monitor state, and message parsing so mention detection can fall back from open_id to union_id in multi-bot groups.

## Best Possible Solution

Land a rebased Feishu fix that obtains a verified stable bot identity or cross-app mention mapping, applies the same matcher to detection, command stripping, debounce merging, and mention-forward handling, and includes focused regression tests plus redacted live multi-bot proof.

## Reproduction Assessment

Yes, at source level. A Feishu group event whose mention metadata has an open_id different from the receiving account’s botOpenId will set `mentionedBot=false`, and the existing requireMention group path will drop it.

## Solution Assessment

No, not as written. The Feishu mention identity plumbing is the right surface, but this branch relies on an unverified union_id source and leaves command stripping, debounce merging, and mention-forward handling on open-id-only logic.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Fetch union IDs from a source that returns them:** `extensions/feishu/src/probe.ts:134`
  - body: This stores `botUnionId` from `bot?.union_id`, but the linked Feishu investigation says the bot-info source does not return that field, and current main’s probe contract only exposes botID/botName. When `botUnionId` is absent, the fallback never runs and the reported multi-bot drop remains.
  - confidence: 0.9
- **[P2] Strip the union-id matched bot mention before parsing commands:** `extensions/feishu/src/bot.ts:791`
  - body: After `checkBotMentioned` can succeed by `union_id`, the parser still normalizes content using only `botOpenId`. In the mismatched-open-id case, the bot mention remains in `ctx.content`, so messages like `@Bot /model` are not reduced to `/model` before command handling.
  - confidence: 0.86
- **[P2] Carry the fallback through debounce and mention forwarding:** `extensions/feishu/src/monitor.account.ts:248`
  - body: The direct receive wiring adds `botUnionId`, but debounce text, merged mention selection, and mention-forward helpers still use open-id-only logic. If the bot mention is in an earlier debounced entry or part of a forward request, the flushed event can still fail the require-mention gate or misclassify targets.
  - confidence: 0.82
