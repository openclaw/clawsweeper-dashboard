## Summary

This PR should stay open, but it is not merge-ready. Current main still has the reported Feishu multi-bot mention bug, while this PR’s union_id fallback depends on an identity source the linked investigation says is unavailable and leaves related parsing paths inconsistent.

## What This Changes

The PR threads a Feishu botUnionId from probe/startup/monitor state into message parsing and uses it as a fallback for @mention detection when open_id does not match.

## Best Possible Solution

Repair or replace the branch with a verified Feishu bot identity strategy that covers text, rich-text posts, command stripping, debounce merging, and mention-forward paths, then land it with focused tests and redacted live multi-bot proof.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose mention metadata has an open_id different from the stored botOpenId will set mentionedBot=false on current main, and the requireMention group path then logs and drops the event. I did not perform live Feishu validation.

## Solution Assessment

No, not as written. The PR addresses the right identity-mismatch surface, but it uses an unverified union_id source and leaves rich-text, debounce, and command-stripping paths inconsistent with the fallback matcher.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Fetch fallback identity from a verified source:** `extensions/feishu/src/probe.ts:137`
  - body: This stores bot?.union_id from the bot info response as the fallback identity, but the linked live investigation and closed sibling PR say bot/v3/info does not return union_id. When that field is absent, botUnionId remains undefined and the multi-bot @mention drop is not fixed.
  - confidence: 0.9
- **[P2] Apply fallback matching to rich-text posts:** `extensions/feishu/src/bot.ts:477-479`
  - body: The new matcher accepts botUnionId for top-level mentions, but the post-message fallback still checks mentionedOpenIds only against botOpenId. Rich-text @mentions in the same app-scoped mismatch case can still be treated as not mentioning the bot.
  - confidence: 0.82
- **[P2] Strip union-matched bot mentions before parsing:** `extensions/feishu/src/bot.ts:797`
  - body: After checkBotMentioned can succeed through botUnionId, normalizeMentions still strips only botOpenId. In the mismatched-open_id case, messages such as @Bot /model keep the bot mention in ctx.content, so command probing no longer sees a clean leading slash command.
  - confidence: 0.88
- **[P2] Carry fallback identity through debounce parsing:** `extensions/feishu/src/monitor.account.ts:251`
  - body: The direct handler receives botUnionId, but the debounce text path still parses each entry with only botOpenId and botName. A burst whose earlier entry mentions the bot through the fallback identity can still merge content and mentions using open-id-only behavior.
  - confidence: 0.82
