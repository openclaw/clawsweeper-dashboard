## Summary

Keep this PR open: the underlying Feishu multi-bot @mention bug is still source-reproducible on current main, but this branch is incomplete and needs human repair before merge.

## What This Changes

The PR threads a Feishu `botUnionId` from probing/state into message parsing and uses it as a fallback when checking whether a bot was @mentioned.

## Best Possible Solution

Repair this PR or replace it with a Feishu-owned fix that obtains a verified stable bot identifier and carries it through mention detection, stripping, debounce/forward handling, state cleanup, regression tests, and a changelog entry.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced: current main compares Feishu @mentions to `botOpenId` only, while the linked report provides concrete multi-token evidence that pushed mention `open_id` values can differ by app. That path reaches the existing `requireMention` drop in `handleFeishuMessage`.

## Solution Assessment

No, not as written. A union_id fallback is only maintainable if the bot union_id comes from a verified source and every mention-sensitive path uses it; this patch leaves identity source, command stripping, and debounce handling incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Fetch union ids from a verified Feishu source:** `extensions/feishu/src/probe.ts:134`
  - body: The PR stores `botUnionId` from the startup probe, but the linked Feishu investigation says the attempted bot-info endpoint does not return `union_id`, and current probe contracts only expose botID/botName. When this value is absent, the new fallback never runs and multi-bot mention detection still fails exactly as reported.
  - confidence: 0.88
- **[P2] Strip the matched union-id mention before command parsing:** `extensions/feishu/src/bot.ts:797`
  - body: After `checkBotMentioned` succeeds by `union_id`, `parseFeishuMessageEvent` still normalizes content using only `botOpenId`. In the mismatched-open_id case, the bot mention stays in `ctx.content`, so `@Bot /model` is not reduced to `/model` before command authorization.
  - confidence: 0.86
- **[P2] Carry union ids through the debounce merge path:** `extensions/feishu/src/monitor.account.ts:248`
  - body: The receive handler still computes debounced text and merged mention metadata with open-id-only helpers. A burst where the bot mention is in an earlier entry can flush without preserving the union-id match, and the final dispatch can still be dropped under `requireMention`.
  - confidence: 0.8
