  - body: The patch reads bot?.union_id from bot/v3/info, but the linked issue says that field is absent and the official Feishu/Lark response contract lists open_id without union_id. If this stays undefined, the fallback never fixes the multi-bot drop.
  - confidence: 0.96
- **[P2] Strip fallback-matched bot mentions before command parsing:** `extensions/feishu/src/bot.ts:790`
  - body: This change can make mentionedBot true via botUnionId, but normalizeMentions still strips only botOpenId. In the mismatch case, @Bot /help keeps the bot mention in content and can still break slash-command detection in the scenario the PR is trying to fix.
  - confidence: 0.88
- **[P2] Carry fallback identity into rich-text post matching:** `extensions/feishu/src/bot.ts:473`
  - body: The fallback only applies when top-level message.mentions is populated; the rich-text post branch still compares parsed mention IDs to botOpenId only. Post messages with the same app-scoped mismatch can still be treated as unmentioned.
  - confidence: 0.84
- **[P2] Pass fallback identity through debounce merging:** `extensions/feishu/src/monitor.account.ts:251`
  - body: This handoff passes botUnionId to the final message handler, but debounce text and retained bot mentions are still derived from open-id-only checks before that handoff. Batched messages can therefore still lose the bot mention under requireMention.
  - confidence: 0.83
