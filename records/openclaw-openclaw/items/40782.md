## What This Changes

The PR threads a Feishu bot union ID from probe/startup/state into group-message parsing so mention detection can fall back from app-scoped open IDs to union IDs.

## Best Possible Solution

Rewrite the fix on current main around a proven stable Feishu bot identity source, then carry that identity through text, post, mention stripping, mention-forward, debounce, tests, and redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level: when a Feishu event mention open_id differs from botOpenId, current main sets mentionedBot false and the group requireMention gate can drop the message. I did not run a live multi-bot Feishu tenant in this read-only review.

## Solution Assessment

No. This PR is a plausible old-base mitigation, but it is not the best fix because the identity source is unproven and the fallback does not cover all current-main Feishu mention paths.
