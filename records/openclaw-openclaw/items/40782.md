## What This Changes

The PR adds botUnionId plumbing through the Feishu probe/startup/monitor state and uses union_id as a fallback when checking whether a Feishu group message mentioned the bot.

## Best Possible Solution

Keep this PR open only as the canonical repair path while maintainers prove a supported stable Feishu bot identity or cross-app open_id mapping, then rebase and carry that identity consistently through detection, stripping, post matching, mention-forward, and debounce paths.

## Reproduction Assessment

Yes, at source level: current main compares Feishu group mentions against botOpenId only, so a WebSocket event whose mention open_id is scoped differently will make mentionedBot false and the require-mention path can return without dispatch. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Unclear as submitted: using a stable cross-app identity is the right shape, but this PR must first prove the Feishu identity source and apply the fallback consistently across current main's parsing and debounce paths.
