  - body: The new fallback is populated only from `bot?.union_id`, but this PR's response type for `bot/v3/info` lists only `bot_name` and `open_id`, and the linked issue says that endpoint does not return `union_id`.   - confidence: 0.92
- **[P2] Pass the matched identity into mention stripping:** `extensions/feishu/src/bot.ts:797`
  - body: When `open_id` differs but `union_id` matches, `mentionedBot` can become true while `normalizeMentions` still strips only `botOpenId`. That leaves the bot mention in content, so mention-prefixed slash commands can still miss the command parsing path this code is preserving.
  - confidence: 0.88
- **[P2] Propagate union-id matching through debounce:** `extensions/feishu/src/monitor.account.ts:251`
  - body: This passes `botUnionId` to the final handler, but the debounce path still builds combined text and merged mentions with open-id-only parsing/filtering. A debounced multi-message burst can still lose the bot mention and be dropped under `requireMention`.
  - confidence: 0.9
- **[P2] Handle rich-text mentions with the same fallback:** `extensions/feishu/src/bot.ts:477-479`
  - body: The PR adds `union_id` matching only for the top-level `mentions` array, while the rich-text `post` branch still compares parsed mention ids to `botOpenId` only. Rich-text group messages with the same app-scoped mismatch can still be treated as unmentioned.
  - confidence: 0.82
