## Summary

Keep open. This is a valid Feishu multi-bot mention bug and current main still has an open_id-only matcher, but the PR is not merge-ready: the proposed union_id source is disputed by the paired report, and the patch leaves several mention consumers open_id-only while also needing a refresh against current main behavior.

## What This Changes

The PR threads a Feishu bot union_id from probe results through monitor state and account handling into message parsing so @mention detection can fall back from open_id to union_id.

## Best Possible Solution

Land a refreshed Feishu-owned fix that first verifies a stable bot identity or cross-app open_id mapping strategy, then applies that identity consistently across the central bot-content helpers and all Feishu mention consumers, with focused regression tests and a user-facing changelog entry.

## Reproduction Assessment

Yes. The paired report gives concrete multi-bot Feishu group steps, and current main has a source-level synthetic path: a group event whose mention.open_id differs from botOpenId leaves ctx.mentionedBot false and is dropped under requireMention.

## Solution Assessment

No. The union_id fallback is plausible only after proving a reliable source for the bot's union_id, and this patch leaves important Feishu mention paths open_id-only; the safer fix is to verify the Feishu API contract first and update all consumers together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Verify a stable Feishu bot identity source:** `extensions/feishu/src/probe.ts:137`
  - body: The patch reads bot?.union_id from the bot-info response, but the paired report and closed sibling PR say that endpoint does not return union_id. If this field is absent, botUnionId is never populated and the fallback cannot fix the reported multi-bot group.
  - confidence: 0.9
- **[P2] Thread fallback identity through startup preflight:** `extensions/feishu/src/monitor.account.ts:510`
  - body: The prefetched path accepts botUnionId here, but the normal multi-account startup still calls monitorSingleAccount with only botOpenId and botName. That leaves botUnionIds empty on the primary startup path.
  - confidence: 0.88
- **[P2] Strip the union-id matched bot mention:** `extensions/feishu/src/bot.ts:797`
  - body: After mentionedBot can become true through union_id, normalizeMentions still strips only a mention whose open_id equals botOpenId. In the mismatch case, @Bot /model can keep the bot placeholder and miss slash-command parsing.
  - confidence: 0.9
- **[P2] Carry fallback identity through debounce:** `extensions/feishu/src/monitor.account.ts:251`
  - body: The final dispatch receives botUnionId, but debounce still builds merged text and preserved mentions with botOpenId-only parsing and filtering. Batched messages can still lose the bot mention and be dropped under requireMention.
  - confidence: 0.86
- **[P2] Preserve broadcast-only at-all filtering:** `extensions/feishu/src/bot.ts:462`
  - body: The branch treats raw @_all as mentioning every bot, but current main deliberately excludes broadcast-only @all/@_all and has docs/tests for that behavior. A refreshed fix must preserve the broadcast filtering.
  - confidence: 0.84
