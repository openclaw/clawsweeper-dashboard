## Summary

Keep open: current main still has the Feishu multi-bot mention bug, and this PR is not merge-ready because its fallback identity source is disputed and the branch leaves rich-text, mention stripping, and debounce paths open_id-only.

## What This Changes

The PR threads a Feishu bot union_id from probe/startup state into message parsing and uses it as a fallback when mention open_id does not match.

## Best Possible Solution

Repair or replace the branch with a Feishu-validated identity matcher that uses a real stable bot identity source and covers text, rich-text posts, mention stripping, debounce merging, tests, and redacted live proof.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose mention metadata has an open_id different from the stored botOpenId makes current main's matcher return false, and requireMention group handling then drops the event. I did not see after-fix live Feishu validation.

## Solution Assessment

No, not as written. The PR targets the right identity-mismatch surface, but it relies on a union_id source that the paired discussion says is unavailable and leaves several message paths inconsistent.

  - body: This stores bot?.union_id as the fallback identity, but the paired issue and closed sibling PR say bot/v3/info does not return union_id. When that field is absent, botUnionId stays undefined and the multi-bot mention drop is not fixed.
  - confidence: 0.9
- **[P2] Apply fallback matching to rich-text posts:** `extensions/feishu/src/bot.ts:477-479`
  - body: The new matcher can use botUnionId for normal mention metadata, but the post-message branch still compares parsed mentionedOpenIds only to botOpenId. Rich-text @mentions in the same app-scoped mismatch case can still be treated as not mentioning the bot.
  - confidence: 0.84
- **[P2] Strip union-matched bot mentions before parsing:** `extensions/feishu/src/bot.ts:797`
  - body: After checkBotMentioned can succeed through botUnionId, normalizeMentions still strips only botOpenId. In the mismatched-open_id case, messages like @Bot /model keep the bot mention in content, so slash-command probing no longer sees a clean leading slash command.
  - confidence: 0.88
- **[P2] Carry fallback identity through debounce parsing:** `extensions/feishu/src/monitor.account.ts:263-266`
  - body: The direct dispatch path receives botUnionId, but resolveDebounceText and merged mention selection still parse/filter with botOpenId only. A burst whose earlier entry mentions the bot through the fallback identity can still debounce using open-id-only content and mention state.
  - confidence: 0.8
