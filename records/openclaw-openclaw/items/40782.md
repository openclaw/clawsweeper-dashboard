## Summary

Keep open: current main still has the Feishu multi-bot mention drop, and this PR is the linked repair path, but the branch is not merge-ready because the fallback identity source is disputed, fallback handling is incomplete, real Feishu proof is missing, and the repair loop is explicitly paused for human review.

## What This Changes

The PR threads a Feishu `botUnionId` from probe/startup state into message parsing and uses it as a fallback when detecting bot @mentions.

## Best Possible Solution

Repair or replace the branch with a Feishu-validated stable identity matcher that is applied consistently to text, rich text, mention stripping, debounce, and current-main file boundaries, with focused regression tests and redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level: a group event whose mention `open_id` differs from the receiving `botOpenId` makes the current matcher return false, and the mention-gated group path then drops it. I did not establish a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

No, not as written. The PR targets the right identity-mismatch surface, but it relies on a disputed `union_id` source and does not carry fallback matching consistently through rich-text posts, mention stripping, or debounce handling.

  - body: This stores `bot?.union_id` as the fallback identity, but the linked issue and closed sibling PR report that Feishu `bot/v3/info` does not return `union_id` in this setup. In that case `botUnionId` stays undefined, so the new matcher never fixes the multi-bot mention drop.
  - confidence: 0.9

  - body: The new matcher can use `botUnionId` for normal mention metadata, but the post-message branch still compares parsed mention IDs only to `botOpenId`. Rich-text @mentions in the same app-scoped mismatch case can still be treated as not mentioning the bot.
  - confidence: 0.84
- **[P2] Strip union-matched bot mentions before parsing:** `extensions/feishu/src/bot.ts:797`
  - body: After `checkBotMentioned` can succeed through `botUnionId`, `normalizeMentions` still strips only `botOpenId`. In the mismatched-open_id case, messages like `@Bot /model` keep the bot mention in content, so slash-command probing can miss the command.
  - confidence: 0.88
- **[P2] Carry fallback identity through debounce parsing:** `extensions/feishu/src/monitor.account.ts:251`
  - body: This passes `botUnionId` to final dispatch, but debounce text resolution and merged mention filtering still parse/filter with `botOpenId` only. A debounced batch whose earlier entry mentions the bot through the fallback identity can still lose the mention state before dispatch.
  - confidence: 0.8
