## Summary

Keep open: this is the linked implementation path for a real Feishu message-loss bug, but the branch is not merge-ready because its fallback identity source is disputed, several parsing paths remain open_id-only, and no after-fix live Feishu proof is attached.

## What This Changes

The PR threads a Feishu bot union_id through probe, monitor state, and message parsing so @mention detection can fall back from open_id in multi-bot groups.

## Best Possible Solution

Repair or replace the branch with a Feishu-validated stable identity matcher applied consistently to text, rich text, mention stripping, debounce, and current-main file boundaries, with focused regression tests and redacted live proof.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose non-broadcast mention open_id differs from botOpenId makes checkBotMentioned return false, and the requireMention group path then drops it. I did not establish a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

No, not as written. The PR targets the right identity mismatch, but it uses an unverified fallback source and does not carry fallback matching through rich-text posts, mention stripping, or debounce handling.
