## Summary

Keep open: this PR targets a real Feishu multi-bot mention bug, but the current patch is not merge-ready because the proposed `union_id` source is unproven on current main and the fallback is not propagated through all mention-consuming paths.

## What This Changes

The PR threads a Feishu `botUnionId` from startup probing through monitor state into message parsing so mention detection can fall back from app-scoped `open_id` to `union_id`.

## Best Possible Solution

Update the Feishu plugin on current main to obtain a verified stable bot identity or cross-app mention mapping, apply it consistently through mention detection, stripping, debounce, forwarding, and startup state, then add focused regression tests and a changelog entry.

## Reproduction Assessment

Yes, from source inspection and the linked report: current main compares Feishu mentions only by `open_id`, then drops group messages under `requireMention` when that comparison fails. I did not run a live Feishu multi-bot workspace, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No. A stable-identity fallback is a plausible direction, but this PR depends on an unverified `union_id` source and does not propagate the fallback into mention stripping or debounce merging.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Verify the union_id source before threading it:** `extensions/feishu/src/probe.ts:137`
  - body: The branch stores `bot?.union_id` from startup bot info, but current main's Feishu probe response only models `pingBotInfo.botID` and `botName`, and the related discussion says the earlier `bot/v3/info` route does not actually return `union_id`. If this value is always undefined, the fallback never fires and the reported multi-bot mention miss remains. Please use a verified Feishu identity source or a different cross-app mapping before wiring this through state.
  - confidence: 0.88
- **[P2] Resolve fallback mentions before stripping:** `extensions/feishu/src/bot.ts:788`
  - body: After this change `mentionedBot` can become true via `union_id`, but the message content is still normalized with only `botOpenId`. In the multi-bot case, `@Bot /model` can keep the bot mention placeholder in the content, which preserves the command-detection regression that the current comment calls out. Resolve the matching mention by `union_id` and strip that mention too, or teach `normalizeMentions` to match the fallback id.
  - confidence: 0.9
- **[P2] Carry botUnionId through debounce merging:** `extensions/feishu/src/monitor.account.ts:248`
  - body: The direct dispatch path gets `botUnionId`, but inbound debounce still derives merged text and merged mentions through open-id-only helpers. A batch where the bot mention is identified only by `union_id` can lose the mention before `handleFeishuMessage`, then get dropped under `requireMention`. Add a `getBotUnionId` path and apply the same fallback in debounce text parsing and mention filtering.
  - confidence: 0.87
