## Summary

The reported Feishu multi-bot delivery bug remains valid, but this branch is not mergeable: it reads an identifier absent from the declared and official bot-info response, and it updates only the final gate while leaving command normalization and debounce mention handling on app-scoped `open_id`.

## What This Changes

The PR fetches and stores a proposed bot `union_id` and uses it as a fallback when Feishu mention `open_id` values do not match in multi-bot groups.

## Best Possible Solution

Obtain the receiving bot's stable identity through a documented Feishu contract, then centralize target-mention resolution in the current `bot-content.ts` boundary and reuse it for detection, mention stripping, debounce merging, and sequential parsing.

## Reproduction Assessment

Yes at the source level: Feishu mention gating compares the event's app-scoped `open_id` with the receiving bot's `open_id`, and the report provides concrete cross-app mismatch observations. No after-fix live reproduction has been shown.

## Solution Assessment

No. A union-based direction may be useful only if a documented bot identity source exists, and the final fix must replace the parallel open-id-only behavior across every mention consumer.
