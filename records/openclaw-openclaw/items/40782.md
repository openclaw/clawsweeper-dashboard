## Summary

Keep open, but this PR is not merge-ready. The Feishu multi-bot mention bug is real, but the branch depends on a `bot/v3/info` `union_id` value that the API/current response contract does not provide, misses debounce and message-normalization paths, lacks after-fix real behavior proof, and currently conflicts with `main`.

## What This Changes

The branch threads a Feishu bot `union_id` from probing/startup state into group @mention detection as a fallback when Feishu `open_id` differs across bot apps.

## Best Possible Solution

Rewrite the Feishu mention identity path on current main around a verified stable identifier or explicit cross-app open_id mapping, cover text/post/debounce/mention-forward behavior with focused tests, and attach redacted real Feishu multi-bot proof before merge.

## Reproduction Assessment

Yes at source level, but not with a live Feishu run in this review. Current main compares Feishu mention `open_id` to the bot's `open_id`, so an app-scoped mismatch under `requireMention` reaches the existing `did not mention bot` drop path.

## Solution Assessment

No. The branch is only a plausible direction: it reads a `union_id` from an endpoint whose documented/current contract does not expose it, and it does not cover the sibling debounce, rich-text, and mention-normalization paths.

Full review comments:

- **[P1] Use a real source for botUnionId:** `extensions/feishu/src/probe.ts:137`
  - body: This line reads `bot?.union_id` from `/bot/v3/info`, but the response type in this file only declares `bot_name` and `open_id`, and the linked Feishu investigation says that endpoint does not return `union_id`. In that state `botUnionId` stays undefined, so the new fallback never fixes the multi-bot `requireMention` drop; use a verified identity source or cross-app open_id mapping instead.
  - confidence: 0.93
- **[P2] Propagate union matching through debounce:** `extensions/feishu/src/monitor.account.ts:251`
  - body: The final dispatch receives `botUnionId`, but the debounce path still parses text and filters merged mentions with `botOpenId` only. If the mention is on an earlier message in a debounced batch and Feishu reports a different app-scoped `open_id`, the merged event can still lose the bot mention and be dropped under `requireMention`.
  - confidence: 0.88
- **[P2] Normalize self mentions using the matched identity:** `extensions/feishu/src/bot.ts:797`
  - body: `parseFeishuMessageEvent` now allows `mentionedBot` to become true via `botUnionId`, but the rest of the same parse still strips the bot mention and detects mention-forward targets by `botOpenId` only. A union-id match can therefore leave this bot's own mention in the content or fail to populate forwarding targets for `@Bot @User` messages.
  - confidence: 0.82
- **[P2] Cover rich-text mention matching:** `extensions/feishu/src/bot.ts:477-480`
  - body: The `post` fallback still checks only `mentionedOpenIds` against `botOpenId`, so rich-text Feishu messages keep the same app-scoped mismatch that this PR fixes for ordinary mention metadata. The identity decision should be shared across text and post paths, or the PR should explicitly prove why rich text cannot carry the affected mismatch.
  - confidence: 0.76
