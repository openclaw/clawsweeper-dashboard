## Summary

Keep open: this is a real Feishu mention-gating bug and the PR is the paired implementation path, but the branch is conflicting, missing real Feishu proof, and the proposed union_id plumbing is not merge-ready.

## What This Changes

The branch attempts to thread a Feishu bot union_id from probe/startup state into message parsing and use it as a fallback for multi-bot @mention matching.

## Best Possible Solution

Repair or replace the branch with a Feishu-validated identity matcher that covers text, rich-text posts, command stripping, debounce merging, and current refactored files, then land with focused tests and redacted live multi-bot proof.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose mention metadata has an open_id different from the stored botOpenId makes the current matcher return false, and the documented requireMention group path then drops the event. I did not run a live Feishu multi-bot workspace.

## Solution Assessment

No, not as written. The PR targets the right identity-mismatch surface, but it relies on an unverified union_id source and leaves rich-text, debounce, and mention-stripping paths open_id-only.

  - body: This stores bot?.union_id as the fallback identity, but the response type in this branch only models bot_name/open_id and the linked investigation says bot/v3/info does not return union_id. When that field is absent, botUnionId stays undefined and the multi-bot mention drop is not fixed.
  - confidence: 0.9

  - body: The new matcher accepts botUnionId for normal mention metadata, but the post-message branch still compares mentionedOpenIds only to botOpenId. Rich-text @mentions in the same app-scoped mismatch case can still be treated as not mentioning the bot.
  - confidence: 0.84
- **[P2] Strip union-matched bot mentions before parsing:** `extensions/feishu/src/bot.ts:797`
  - body: After checkBotMentioned can succeed through botUnionId, normalizeMentions still strips only botOpenId. In the mismatched-open_id case, messages like @Bot /model keep the bot mention in content, so slash-command probing no longer sees a clean leading slash command.
  - confidence: 0.88
- **[P2] Carry fallback identity through debounce parsing:** `extensions/feishu/src/monitor.account.ts:263-266`
  - body: The direct dispatch path receives botUnionId, but resolveDebounceText and merged mention selection still parse/filter with botOpenId only. A burst whose earlier entry mentions the bot through the fallback identity can still debounce using open-id-only content and mention state.
  - confidence: 0.8
