## Summary

Keep open: current main still has the Feishu open_id-only mention failure, while this PR’s union_id source is disputed by the paired issue and the branch leaves rich-text, mention-stripping, and debounce paths inconsistent.

## What This Changes

The PR threads a Feishu bot `union_id` from startup probing into monitor state and message parsing, then falls back to matching mention `union_id` when `open_id` differs.

## Best Possible Solution

Repair or replace the branch with a Feishu-validated identity matcher that uses a real stable bot identity source and keeps text, rich-text, mention stripping, and debounce paths consistent.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose mention metadata has an `open_id` different from the stored `botOpenId` makes current main’s matcher return false, and the `requireMention` group path then drops it. I did not establish a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

No, not as written. The PR targets the right identity-mismatch surface, but it relies on a disputed `union_id` source and does not carry the fallback consistently through rich-text posts, mention stripping, or debounce parsing.
