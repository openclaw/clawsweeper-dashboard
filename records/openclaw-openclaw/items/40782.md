## Summary

Keep open: this targets a real Feishu multi-bot mention bug, but the patch is not merge-ready because the proposed `union_id` source is unverified, mention stripping and debounce/post paths remain open-id-only, real Feishu proof is missing, and the PR is explicitly paused for human review.

## What This Changes

The PR adds Feishu bot `union_id` plumbing from probe/startup/state into message parsing and uses it as a fallback for group @mention detection.

## Best Possible Solution

Repair this PR or replace it with a Feishu-validated identity matcher that covers text and rich-text mentions, command stripping, debounce/forward handling, identity recovery/state cleanup, regression tests, and real multi-bot proof.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose mention metadata has a different app-scoped `open_id` than the receiving account's `botOpenId` produces `mentionedBot=false` and reaches the existing require-mention drop path. I did not live-test against Feishu.

## Solution Assessment

No, not as written. A union-id fallback is only maintainable if the bot union ID comes from a verified Feishu source and every mention-sensitive path uses it; this patch leaves identity-source, command stripping, debounce, rich-text, and proof gaps.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Fetch union IDs from a verified Feishu identity source:** `extensions/feishu/src/probe.ts:134`
  - body: This stores `botUnionId` from `bot?.union_id`, but the linked investigation, closed sibling PR, and current dependency/source contract show the bot-info path only supplies bot open ID/name. When `botUnionId` is undefined, the fallback never runs and the reported multi-bot mention drop remains.
  - confidence: 0.9
- **[P2] Strip union-id matched bot mentions before command parsing:** `extensions/feishu/src/bot.ts:797`
  - body: After `checkBotMentioned` can succeed by `union_id`, this path still normalizes content using only `botOpenId`. In the mismatched-open-id case, `@Bot /model` keeps the bot mention in `ctx.content`, so the command parser no longer sees `/model` as the leading command.
  - confidence: 0.87
- **[P2] Carry union-id matching through debounced mention merges:** `extensions/feishu/src/monitor.account.ts:248`
  - body: The direct receive wiring passes `botUnionId`, but debounce text and merged mention selection still use open-id-only helpers. If the bot mention appears in an earlier debounced entry with a mismatched app-scoped open ID, the flushed event can still be treated as unmentioned and dropped.
  - confidence: 0.82
- **[P2] Handle rich-text post mentions with the same fallback:** `extensions/feishu/src/bot.ts:477-480`
  - body: The new fallback applies to `event.message.mentions`, but the post-message branch still parses open/user IDs from rich text and compares them only to `botOpenId`. Rich-text bot mentions with the same app-scoped open ID mismatch will still return `mentionedBot=false`.
  - confidence: 0.78
