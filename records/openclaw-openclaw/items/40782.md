The PR threads a Feishu botUnionId from startup probing into group mention parsing so multi-bot groups can match mentions when open_id differs by app context.

## Best Possible Solution

Repair or replace this PR on current main with a Feishu-validated identity matcher that covers text, rich-text, mention stripping, mention-forward, and debounce paths, backed by focused regression tests and redacted live Feishu proof.

Yes at source level: a Feishu group event whose delivered mention open_id differs from `botOpenId` makes current `checkBotMentioned` return false, and the requireMention path returns with the reported did-not-mention log. I did not perform live Feishu validation.

## Solution Assessment
