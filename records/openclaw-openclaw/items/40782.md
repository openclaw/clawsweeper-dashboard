## Summary

Keep open: this is a plausible Feishu message-loss bug fix, but the branch is not merge-ready because the fallback identity source is disputed, several mention paths remain open_id-only, real Feishu proof is missing, and the PR is explicitly paused for human review.

## What This Changes

The branch threads a Feishu botUnionId from probe/startup state into monitor state and message parsing so bot mention detection can fall back from open_id to union_id.

## Best Possible Solution

Repair or replace the branch with a Feishu-validated stable identity matcher applied consistently to text, rich text, mention stripping, debounce, and current-main file boundaries, with focused regression tests and redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose non-broadcast mention open_id differs from botOpenId makes checkBotMentioned return false, and the requireMention group path then drops it. I did not establish a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

No, not as written. The PR targets the right identity mismatch, but it uses an unverified fallback source and does not carry fallback matching through rich-text posts, mention stripping, or debounce handling.
