## Summary

Keep open: this PR targets a real Feishu multi-bot mention bug, but the branch is not merge-ready because the proposed fallback identity source is disputed, the implementation leaves several mention-sensitive paths open-id-only, the branch conflicts with current main, and contributor real Feishu proof is still missing.

## What This Changes

The PR adds Feishu bot union_id plumbing through probe/startup/monitor state and uses it as a fallback for bot @mention detection in multi-bot groups.

## Best Possible Solution

Repair or replace the branch with a current-main Feishu identity matcher that uses a verified stable bot mapping across text, rich-text, command stripping, debounce merging, and mention-forward handling, with focused tests and redacted live multi-bot proof.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose mention metadata has an open_id different from the receiving account's botOpenId sets mentionedBot=false, and the documented requireMention group path then drops it. I did not perform live Feishu validation.

## Solution Assessment

No, not as written. The fix targets the right Feishu mention identity surface, but it relies on an unresolved botUnionId source and leaves post messages, command stripping, and debounce parsing inconsistent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Fetch fallback identity from a verified source:** `extensions/feishu/src/probe.ts:137`
  - body: This stores bot?.union_id as the only fallback identity, but the linked live investigation and closed sibling PR say the bot info response used for this approach does not return union_id. When that field is absent, botUnionId stays undefined and the reported multi-bot @mention drop is not fixed.
  - confidence: 0.9
- **[P2] Apply fallback matching to rich-text posts:** `extensions/feishu/src/bot.ts:477-479`
  - body: The new matcher accepts botUnionId for top-level mentions, but the post-message fallback still checks mentionedOpenIds only against botOpenId. Rich-text @mentions in the same app-scoped mismatch case can still be treated as not mentioning the bot.
  - confidence: 0.82
- **[P2] Strip union-matched bot mentions before parsing:** `extensions/feishu/src/bot.ts:797`
  - body: After checkBotMentioned can succeed through botUnionId, normalizeMentions still strips only botOpenId. In the mismatched-open-id case, messages such as @Bot /model keep the bot mention in ctx.content, so command probing no longer sees a clean leading slash command.
  - confidence: 0.88
- **[P2] Carry fallback identity through debounce parsing:** `extensions/feishu/src/monitor.account.ts:251`
  - body: The direct handler receives botUnionId, but the debounce text path still parses each entry with only botOpenId and botName. A burst whose earlier entry mentions the bot through the fallback identity can still merge content and mentions using open-id-only behavior.
  - confidence: 0.82
