## Summary

Keep open. Current main still uses Feishu mention open_id equality for requireMention, but this PR is not merge-ready because the proposed union_id source is unproven and the patch leaves startup, debounce, mention stripping, and @all policy gaps.

## What This Changes

The PR attempts to add Feishu bot union_id capture and pass it through monitor state and message parsing so @mention detection can fall back from open_id to union_id in multi-bot groups.

## Best Possible Solution

Refresh the Feishu-owned fix around a proven cross-app identity or open_id variant mapping, centralize mention matching, carry it through startup, debounce, stripping, and post handling, preserve @all policy, and add regression tests plus a changelog entry.

## Reproduction Assessment

Yes, source-level. A Feishu group event whose mention.id.open_id differs from botOpenId leaves ctx.mentionedBot false in current main, then requireMention drops the message in handleFeishuMessage.

## Solution Assessment

No. The union_id fallback is plausible only after proving a reliable Feishu/Lark bot identity source, and this patch leaves several mention consumers open_id-only while changing broadcast @all semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Verify a stable Feishu bot identity source:** `extensions/feishu/src/probe.ts:137`
  - body: The patch reads botUnionId from the bot-info response, but the paired report and closed sibling PR say that endpoint does not return union_id. If this field is absent, the fallback is never populated and the multi-bot mention failure remains.
  - confidence: 0.9
- **[P2] Carry fallback identity through startup preflight:** `extensions/feishu/src/monitor.account.ts:510`
  - body: The patch makes the prefetched identity capable of carrying botUnionId, but the normal multi-account startup preflight still passes only botOpenId and botName. That leaves the union-id map empty on the primary multi-account startup path.
  - confidence: 0.87
- **[P2] Strip the union-id matched bot mention:** `extensions/feishu/src/bot.ts:797`
  - body: After mentionedBot can become true through union_id, normalizeMentions still strips only a mention whose open_id equals botOpenId. In the mismatch case, @Bot /model keeps the bot placeholder and can miss slash-command parsing.
  - confidence: 0.9
- **[P2] Carry fallback identity through debounce:** `extensions/feishu/src/monitor.account.ts:251`
  - body: This adds botUnionId to the final message dispatch, but debounce still builds combined text and preserved mentions through open_id-only parsing and filtering. Batched multi-bot messages can still lose the bot mention and be dropped under requireMention.
  - confidence: 0.86
- **[P2] Preserve broadcast-only at-all filtering:** `extensions/feishu/src/bot.ts:462`
  - body: The new raw @_all shortcut treats broadcast-only messages as mentioning every bot, but current docs and tests deliberately exclude broadcast-only @all/@_all from bot mention detection. A refreshed fix must keep that behavior unless the policy is intentionally changed.
  - confidence: 0.84
