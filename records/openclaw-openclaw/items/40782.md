- **[P1] Prove the union_id source before wiring it:** `extensions/feishu/src/probe.ts:137`
  - body: The patch stores `botUnionId` from `bot?.union_id`, but the branch's response type does not declare that field and https://github.com/openclaw/openclaw/issues/40768 plus https://github.com/openclaw/openclaw/pull/40783 say the bot info source does not return it.   - confidence: 0.93
- **[P2] Resolve the bot mention before stripping content:** `extensions/feishu/src/bot.ts:790`
  - body: This can mark the bot as mentioned through `botUnionId`, but content normalization still strips only by `botOpenId`. In the same app-scoped mismatch, an addressed `@Bot /help` can keep the mention tag and miss the slash-command path that the nearby comment protects.
  - confidence: 0.88
- **[P2] Cover rich-text post mentions with the same identity rule:** `extensions/feishu/src/bot.ts:473`
  - body: The new fallback only applies when top-level `message.mentions` is populated, while the rich-text post branch still compares parsed mention IDs to `botOpenId` only. Post messages with the same app-scoped mismatch can still be treated as unmentioned.
  - confidence: 0.84
- **[P2] Pass the identity fallback through debounce merging:** `extensions/feishu/src/monitor.account.ts:251`
  - body: The final handler receives `botUnionId`, but debounce text and retained bot mentions are still derived from open_id-only checks before that handoff. Batched messages can therefore still lose the bot mention under `requireMention`.
  - confidence: 0.83
