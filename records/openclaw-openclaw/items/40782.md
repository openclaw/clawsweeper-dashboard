## Summary

Keep open: this PR targets a real Feishu multi-bot mention bug, but it is not merge-ready because the proposed fallback does not establish a reliable bot union_id source and still leaves open-id-only mention stripping and debounce paths in place.

## What This Changes

The PR threads a new Feishu `botUnionId` from startup probing through monitor state and uses it as a fallback for Feishu @mention detection.

## Best Possible Solution

Repair this PR or replace it with a Feishu-owned fix that first obtains a verified stable bot identity, then propagates it through mention detection, mention stripping, mention-forward/debounce handling, cleanup state, and focused Feishu regression tests with a user-facing changelog entry.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced: current main compares Feishu @mentions to `botOpenId` only, while the linked issue provides concrete multi-token evidence that the pushed mention open_id can differ by app. That path reaches the existing `requireMention` drop in `handleFeishuMessage`.

## Solution Assessment

No, not as written. A union_id fallback is only a good fix if the bot union_id is fetched from a verified source and carried through every mention-sensitive path; this PR leaves the identity source, content stripping, and debounce merge incomplete.
