## Summary

Keep open. The PR targets a valid Feishu multi-bot mention bug tracked by #40768, but the patch is not merge-ready because the proposed union_id source is disputed, current main still has open_id-only mention gates, and the PR leaves startup, debounce, mention stripping, and broadcast-only @all behavior incomplete.

## What This Changes

The PR threads a Feishu bot union_id from probe results through monitor state and account handling into message parsing so @mention detection can fall back from open_id to union_id.

## Best Possible Solution

Land a refreshed Feishu-owned fix that verifies a stable cross-app identity or open_id mapping strategy, centralizes matching in the current bot-content helpers, carries the identity through startup/debounce/stripping, preserves @all behavior, and adds focused regression tests plus a changelog entry.

## Reproduction Assessment

Yes. The paired report gives concrete multi-bot Feishu group steps, and current main has a source-level reproduction: a group event whose mention.id.open_id differs from botOpenId leaves ctx.mentionedBot false and is dropped under requireMention.

## Solution Assessment

No. The union_id fallback is plausible only after proving a reliable bot identity source, and this patch leaves several Feishu mention consumers open_id-only; the safer path is to verify the API contract first and update all mention paths together.

## Review Findings

  - body: The patch reads botUnionId from the bot-info response, but the paired report and closed sibling PR say that endpoint does not return union_id. If this field is absent, the fallback is never populated and the multi-bot mention failure remains.
  - confidence: 0.9

  - body: The prefetched source can carry botUnionId here, but the normal multi-account startup preflight still passes only botOpenId and botName. That leaves the union-id map empty on the primary startup path.
  - confidence: 0.88
- **[P2] Strip the union-id matched bot mention:** `extensions/feishu/src/bot.ts:797`
  - body: After mentionedBot can become true through union_id, normalizeMentions still strips only a mention whose open_id equals botOpenId. In the mismatch case, @Bot /model keeps the bot placeholder and can miss slash-command parsing.
  - confidence: 0.9
- **[P2] Carry fallback identity through debounce:** `extensions/feishu/src/monitor.account.ts:251`
  - body: The final dispatch receives botUnionId, but debounce still builds combined text and preserved mentions with botOpenId-only parsing and filtering. Batched multi-bot messages can still lose the bot mention and be dropped under requireMention.
  - confidence: 0.86
- **[P2] Preserve broadcast-only at-all filtering:** `extensions/feishu/src/bot.ts:462`
  - body: This branch treats raw @_all as mentioning every bot, but current main deliberately excludes broadcast-only @all/@_all and has docs/tests for that behavior. A refreshed fix must not wake every bot for broadcast-only group messages.
  - confidence: 0.84
