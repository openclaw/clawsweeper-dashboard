  - body: This reads `botUnionId` from `bot?.union_id`, but the typed bot-info response above only defines `bot_name` and `open_id`, the linked issue says this endpoint does not return `union_id`, and current main now uses a different `pingBotInfo.botID` probe. If `botUnionId` stays undefined, the new fallback never runs and the reported multi-bot group mentions are still dropped.
  - confidence: 0.9
- **[P2] Strip the union-id-matched bot mention:** `extensions/feishu/src/bot.ts:797`
  - body: After `checkBotMentioned()` succeeds via `botUnionId`, this still strips the bot mention using only `botOpenId`. In the exact open-id mismatch this PR targets, `@Bot /help` keeps the mention tag in `ctx.content`, so group slash-command detection can still miss the leading `/`.
  - confidence: 0.88
- **[P2] Carry union-id matching through debounce:** `extensions/feishu/src/monitor.account.ts:251`
  - body: This only passes `botUnionId` to the final `handleFeishuMessage()` task; the debounce text and merged-mention logic still parse and filter with `botOpenId` only. A burst where the mention appears in an earlier debounced message can still flush without the bot mention and be dropped under `requireMention`.
  - confidence: 0.84
- **[P2] Apply the fallback to rich-text mentions too:** `extensions/feishu/src/bot.ts:473`
  - body: The new union-id fallback only covers `event.message.mentions`; the `post` fallback still checks parsed `mentionedOpenIds` against `botOpenId`. Rich-text Feishu group messages can therefore retain the same app-scoped open-id mismatch and still be treated as unmentioned.
  - confidence: 0.78
