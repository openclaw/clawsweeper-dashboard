## Summary

Keep open. The PR targets a real source-reproducible Feishu mention bug, but current main and related discussion show the branch is stale and incomplete: the union_id source is not backed by the probe contract, command stripping and debounce merge remain open_id-only, and a human-review pause is active.

## What This Changes

The PR threads a Feishu botUnionId from probing and monitor state into message parsing, then uses it as a fallback for @mention detection.

## Best Possible Solution

Repair or replace this PR with a Feishu-owned fix that obtains a verified stable bot identity and carries it through mention detection, mention stripping, debounce/forward handling, regression tests, and a changelog entry without regressing current broadcast-only @all behavior.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced: current main compares Feishu @mentions to botOpenId only, while #40768 provides concrete multi-token evidence that pushed mention open_id values can differ by app. That path reaches the existing requireMention drop in handleFeishuMessage.

## Solution Assessment

No, not as written. A union_id fallback is only maintainable if the bot union_id comes from a verified source and every mention-sensitive path uses it; this patch leaves identity source, command stripping, and debounce handling incomplete.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Fetch union ids from a verified Feishu source:** `extensions/feishu/src/probe.ts:134`
  - body: The patch stores botUnionId from bot?.union_id, but the related Feishu investigation says the attempted bot-info endpoint does not return union_id and current probe contracts expose only botID/botName.   - confidence: 0.88

  - body: After checkBotMentioned succeeds by union_id, parseFeishuMessageEvent still normalizes content using only botOpenId. In the mismatched-open_id case, the bot mention stays in ctx.content, so @Bot /model is not reduced to /model before command authorization.
  - confidence: 0.86
- **[P2] Carry union ids through the debounce merge path:** `extensions/feishu/src/monitor.account.ts:248`
  - body: This wiring adds botUnionId to the final handleFeishuMessage call, but the debounce path still derives merged text and mention metadata with open-id-only helpers. A batch where the bot mention is in an earlier entry can flush without preserving the union-id match and still be dropped under requireMention.
  - confidence: 0.8
