## Summary

Keep open: current main still has open-id-only Feishu mention detection, and this PR is not merge-ready because the proposed union_id source is unverified while command stripping and debounce handling remain incomplete.

## What This Changes

The PR threads a Feishu `botUnionId` from probing through monitor state into message parsing and uses it as a fallback for @mention matching when `open_id` differs.

## Best Possible Solution

Repair this PR or replace it with a Feishu-owned fix that uses a verified stable bot identifier across detection, mention stripping, debounce/forward handling, state cleanup, regression tests, and a changelog entry.

## Reproduction Assessment

Yes, source-reproducible rather than live-reproduced: current main compares Feishu @mentions to `botOpenId` only, while the linked issue provides concrete multi-token evidence that pushed mention `open_id` values can differ by app. That path reaches the existing `requireMention` drop in `handleFeishuMessage`.

## Solution Assessment

No, not as written. A union_id fallback is only maintainable if the bot union_id comes from a verified Feishu source and every mention-sensitive path uses it; this patch leaves identity source, command stripping, and debounce handling incomplete.

  - body: The patch stores `botUnionId` from the startup probe, but the linked Feishu investigation says the attempted bot-info endpoint does not return `union_id`, and current probe contracts only expose botID/botName.   - confidence: 0.86

  - body: This wiring adds `botUnionId` to the final message call, but the debounce path still derives merged text and mention metadata with open-id-only helpers. A burst where the bot mention is in an earlier entry can flush without preserving the union-id match and still be dropped under `requireMention`.
  - confidence: 0.8
