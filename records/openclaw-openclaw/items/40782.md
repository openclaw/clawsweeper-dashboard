## Summary

Keep open: the PR targets a real Feishu mention-gating bug, but it is not merge-ready because the proposed fallback depends on an unverified union-id source, leaves command stripping and debounce paths open-id-only, lacks after-fix Feishu proof, and is explicitly paused for human review.

## What This Changes

The PR adds Feishu `botUnionId` plumbing from startup probing into monitor state and message parsing so group @mention detection can fall back from `open_id` to `union_id`.

## Best Possible Solution

Repair this PR or replace it with a Feishu-validated identity matcher that covers mention detection, mention stripping, debounce/forward paths, state cleanup, regression tests, and real multi-bot proof.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose mention metadata has an `open_id` different from the receiving account's `botOpenId` makes `mentionedBot=false`, and the existing `requireMention` group path then drops it. I did not live-test against Feishu.

## Solution Assessment

No, not as written. A union-id fallback is maintainable only if the bot union id comes from a verified Feishu source and every mention-sensitive path uses it; this patch leaves source verification, command stripping, and debounce handling incomplete.
