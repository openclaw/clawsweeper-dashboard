## What This Changes

The PR adds `botUnionId` plumbing from Feishu probe/startup state into message parsing and uses `union_id` as a fallback when detecting bot @mentions.

## Best Possible Solution

Rebase or replace this branch with a Feishu-validated stable identity matcher applied consistently to text, rich text, mention stripping, and debounce paths, with focused regression tests and redacted live proof.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose mention `open_id` differs from the receiving bot's `botOpenId` makes current `checkBotMentioned` return false, and the require-mention group path drops it. I did not establish a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

No, not as written. The PR targets the right identity mismatch, but its `union_id` source is disputed and fallback matching is not carried through rich-text posts, mention stripping, or debounce handling.
