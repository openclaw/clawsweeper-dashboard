## What This Changes

The PR adds a Feishu botUnionId field from probe/startup state and uses it as a fallback when detecting group @mentions.

## Best Possible Solution

Land a Feishu-validated stable identity matcher or cross-app open_id mapping across detection, stripping, rich-text posts, mention-forward, and debounce, with focused tests and redacted live Feishu/Lark proof.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose delivered mention open_id differs from the receiving bot's botOpenId makes `checkBotMentioned` return false and the require-mention group path returns before dispatch. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

No, not as submitted: the PR depends on a union_id source that the linked issue and response type do not prove, and it does not carry fallback identity through all current Feishu mention paths. A safer fix needs a supported identity source or cross-app mapping first, then consistent propagation and proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use a proven Feishu bot identity source:** `extensions/feishu/src/probe.ts:137`
  - body: The patch reads `bot?.union_id`, but the response type only declares `bot_name` and `open_id`, and the linked investigation says this endpoint did not return `union_id` in the affected setup. If this field stays undefined, the fallback never runs and the reported multi-bot messages remain dropped.
  - confidence: 0.9
- **[P2] Strip fallback-matched bot mentions:** `extensions/feishu/src/bot.ts:797`
  - body: After `checkBotMentioned` can return true via `botUnionId`, parsed content still strips the bot mention only by `botOpenId`. In the mismatch case this leaves the bot mention in `@Bot /help`, so command probing can still miss the leading slash even though the message was admitted.
  - confidence: 0.86
- **[P2] Propagate fallback identity through debounce:** `extensions/feishu/src/monitor.account.ts:251`
  - body: This added field reaches final dispatch, but debounce text and merged mention selection still parse/filter with `botOpenId` only. A multi-message debounce window can therefore keep the wrong mention text or drop the bot mention before the fallback-aware handler runs.
  - confidence: 0.83
- **[P2] Cover post messages in fallback matching:** `extensions/feishu/src/bot.ts:477-479`
  - body: The new `botUnionId` parameter is only used for top-level `message.mentions`; the rich-text `post` branch still compares parsed mention IDs only with `botOpenId`. Post messages in the same app-scoped mismatch scenario can still be treated as unmentioned and skipped.
  - confidence: 0.78
