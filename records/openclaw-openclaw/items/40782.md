## What This Changes

The PR threads a Feishu botUnionId from startup probe/state into message parsing so mention detection can fall back from open_id to union_id in multi-bot group chats.

## Best Possible Solution

Repair or replace this branch with a Feishu-validated stable identity matcher that works across text/post detection, mention stripping, debounce, and startup identity on current main, then add focused regression tests and redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level: current main matches Feishu bot mentions only by botOpenId and the group requireMention path returns with the reported did-not-mention log when mentionedBot is false. I did not perform live Feishu validation.

## Solution Assessment

No. The PR targets the right surface, but the proposed union_id source is disputed, the branch misses command stripping and debounce propagation, and it needs a current-main rebase plus live Feishu proof.

  - body: The only producer for `botUnionId` is `bot?.union_id` from `/bot/v3/info`, but this branch's response contract only declares `bot_name`/`open_id`, and the linked issue follow-up says that endpoint did not return `union_id` in the affected setup.   - confidence: 0.9

  - body: `parseFeishuMessageEvent` now detects the bot through `botUnionId`, but it still calls `normalizeMentions(..., botOpenId)`. In the fallback case the delivered mention `open_id` does not equal `botOpenId`, so `@Bot /help` can keep the bot mention tag in `content` and miss the command path that this parser comment says depends on stripping the bot mention.
  - confidence: 0.88
- **[P2] Propagate fallback matching through debounce mentions:** `extensions/feishu/src/monitor.account.ts:251`
  - body: The final dispatch receives `botUnionId`, but the debounce path still builds text with `parseFeishuMessageEvent(event, botOpenId, ...)` and filters merged mentions by `mention.id.open_id`. A burst where only an earlier entry contains the union-id matched mention can still flush a merged event that looks unmentioned and gets dropped under `requireMention`.
  - confidence: 0.86
