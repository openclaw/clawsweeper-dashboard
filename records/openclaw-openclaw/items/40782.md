## Summary

Keep open: the PR targets a real Feishu multi-bot mention bug, but the branch is not merge-ready because the union_id source is disputed, fallback handling is incomplete, real Feishu proof is missing, and the live PR is conflicting.

## What This Changes

The PR adds Feishu bot union_id plumbing from probe/startup state into message parsing and uses union_id as a fallback for bot @mention matching.

## Best Possible Solution

Repair or replace this branch with a Feishu-validated identity matcher that uses a real stable bot identity source and applies consistently to text, rich text, mention stripping, and debounce handling.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose mention metadata has an open_id different from the stored botOpenId makes current main's matcher return false, and the requireMention group path then drops it. I did not establish a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

No, not as written. The PR targets the right identity-mismatch surface, but it relies on a disputed union_id source and does not carry fallback matching consistently through rich-text posts, mention stripping, or debounce parsing.

  - confidence: 0.84

  - confidence: 0.88

  - body: This passes `botUnionId` to the final handler, but debounce text resolution and merged mention filtering still parse/filter with `botOpenId` only. A debounced batch whose earlier entry mentions the bot through the fallback identity can still lose the mention state before dispatch.
  - confidence: 0.8
