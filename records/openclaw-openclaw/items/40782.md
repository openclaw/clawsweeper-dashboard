## Summary

Keep open: this targets a real source-reproducible Feishu message-delivery bug, but the branch is not merge-ready because its fallback identity source is disputed and the fallback is not carried through all current message paths.

## What This Changes

The PR threads a Feishu `botUnionId` from probe/startup/monitor state into message parsing and uses it as a fallback for group @mention detection when `open_id` mismatches.

## Best Possible Solution

Use a Feishu-supported stable identity or cross-app mapping consistently across detection, post messages, mention stripping, mention-forward, and debounce, then prove it in a redacted real multi-bot Feishu/Lark group before merge.

## Reproduction Assessment

Yes, at source level: current main compares Feishu bot mentions by `open_id` only, so a group event whose delivered mention `open_id` differs from `botOpenId` makes `mentionedBot` false and the require-mention path returns without dispatch. I did not perform live Feishu validation.

## Solution Assessment

No, not as-is: the PR depends on a `union_id` source that the linked issue says is unavailable, and it does not carry the fallback through current main's stripping, rich-text, mention-forward, and debounce surfaces.

  - confidence: 0.83
