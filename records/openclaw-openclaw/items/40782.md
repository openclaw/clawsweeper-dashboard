## Summary

Keep open: this PR targets a real Feishu multi-bot message-delivery bug, but the proposed union_id source is disputed, the branch is stale against current main's Feishu file split, real Feishu proof is missing, and open_id-only parsing paths remain.

## What This Changes

The PR adds a Feishu botUnionId probe/state path and uses it as a fallback for bot @mention detection when mention open_id does not match the receiving bot's open_id.

## Best Possible Solution

Repair or replace this branch on current main with a Feishu-validated stable identity matcher applied consistently across text, rich-text, mention stripping, mention-forward, and debounce paths, backed by focused regression tests and redacted real Feishu proof.

## Reproduction Assessment

Yes, at source level: current main compares Feishu bot mentions only by open_id, so a group event whose delivered mention open_id differs from botOpenId makes mentionedBot false and the requireMention path returns without dispatch. I did not perform live Feishu validation.

## Solution Assessment

No: the PR targets the right bug family, but the union_id source is disputed by the linked issue, the branch is stale against current main, and several parsing paths remain open_id-only. The safer direction is to prove a supported stable identity source and wire it through every mention decision point before merge.
