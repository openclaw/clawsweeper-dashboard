## Summary

Keep open: current main and the latest shipped release still use open_id-only Feishu mention matching, but this PR is not merge-ready because its union_id source is disputed, command stripping/debounce paths remain open_id-only, and no after-fix live Feishu proof is present.

## What This Changes

The PR threads a Feishu bot union ID from probe/startup state into message parsing so @mention detection can fall back from open_id in multi-bot groups.

## Best Possible Solution

Land a rebased Feishu fix that uses a verified stable bot identity or cross-app mention mapping consistently across detection, command stripping, debounce merging, and mention-forward handling, with focused regression tests and redacted live multi-bot proof.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose mention metadata has an open_id different from the receiving account's botOpenId will set mentionedBot=false, and the documented requireMention group path then drops it. I did not perform live Feishu validation.

## Solution Assessment

No, not as written. The Feishu mention identity surface is the right place to fix this, but this branch relies on an unverified union_id source and leaves command stripping, debounce merging, and mention-forward handling tied to open_id-only logic.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Fetch union IDs from a verified source:** `extensions/feishu/src/probe.ts:137`
  - body: This reads `bot?.union_id`, but the PR branch's `FeishuBotInfoResponse` only declares `bot_name`/`open_id`, and the linked Feishu investigation says `bot/v3/info` does not return `union_id`. When that field is absent, `botUnionId` stays undefined and the new fallback never fixes the reported multi-bot drop.
  - confidence: 0.9
- **[P2] Strip the union-id matched bot mention before parsing:** `extensions/feishu/src/bot.ts:790`
  - body: After `checkBotMentioned` can succeed by `union_id`, the parser still normalizes content using only `botOpenId`. In the mismatched-open-id case, the bot mention stays in `ctx.content`, so messages such as `@Bot /model` are not reduced to `/model` before command handling.
  - confidence: 0.86
- **[P2] Pass the fallback through debounce handling:** `extensions/feishu/src/monitor.account.ts:251`
  - body: The direct receive wiring passes `botUnionId`, but debounce text and merged mention selection still use open-id-only logic. If the bot mention appears in an earlier debounced entry, the flushed event can still fail the require-mention gate or keep the bot mention in the merged command text.
  - confidence: 0.82
