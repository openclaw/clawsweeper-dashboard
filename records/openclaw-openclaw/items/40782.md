## Summary

Keep open: current main still has open_id-only Feishu mention recognition, and this PR targets a real linked bug, but the branch is not merge-ready because its union_id source is disputed, related mention paths remain open_id-only, it lacks after-fix Feishu proof, and GitHub reports it as conflicting.

## What This Changes

The PR adds Feishu bot union ID plumbing through probe/startup/monitor state and uses it as a fallback when detecting bot mentions in multi-bot Feishu groups.

## Best Possible Solution

Land a rebased Feishu fix that uses a verified stable bot identity or cross-app mention mapping consistently across detection, command stripping, debounce merging, and mention-forward handling, with focused regression tests and redacted live multi-bot proof.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose mention metadata has an open_id different from the receiving account's botOpenId will set mentionedBot=false, and the documented requireMention group path then drops it. I did not perform live Feishu validation.

## Solution Assessment

No, not as written. Feishu mention identity matching is the right surface, but this branch relies on an unverified union_id source and leaves command stripping, debounce merging, and mention-forward handling tied to open_id-only logic.

Full review comments:

- **[P2] Fetch union IDs from a verified source:** `extensions/feishu/src/probe.ts:137`
  - body: This reads bot?.union_id as the only source for the fallback, but the linked live investigation and closed sibling PR state that bot/v3/info does not return union_id. When that field is absent, botUnionId stays undefined and the reported multi-bot drop is not fixed.
  - confidence: 0.9
- **[P2] Strip the union-id matched bot mention before parsing:** `extensions/feishu/src/bot.ts:790`

  - confidence: 0.86

  - body: The direct handler gets botUnionId, but debounce text parsing and merged mention selection remain open-id-only. If only an earlier entry in a debounce window mentions the bot with a mismatched open_id, the flushed event can still be dropped or keep the bot mention in the merged command text.
  - confidence: 0.82
