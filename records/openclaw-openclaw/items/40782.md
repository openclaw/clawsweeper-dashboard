## Summary

Keep open. Current main still has the Feishu multi-bot mention gap, but this PR is not ready to merge: the proposed fallback identity source is disputed by the paired report, several Feishu mention paths remain open_id-only, and the branch is dirty against current main.

## What This Changes

The PR threads a Feishu bot union_id from probe results through monitor state and account handling into Feishu message parsing so @mention detection can fall back from open_id to union_id.

## Best Possible Solution

Land a refreshed Feishu-owned fix that first verifies a stable bot identity or cross-app open_id mapping strategy, then applies that identity consistently across all Feishu mention consumers and adds focused regression coverage plus a user-facing changelog entry.

## Reproduction Assessment

Yes. The paired report gives concrete multi-bot Feishu group steps, and current main has a synthetic reproduction path: send a group event whose mention.open_id differs from botOpenId under requireMention and the message is dropped as not mentioning the bot.

## Solution Assessment

No. The union_id fallback is plausible only if the source is proven, and this patch leaves important Feishu mention paths open_id-only; the safer direction is to verify the Feishu API contract first and update all consumers together on a refreshed branch.

  - body: The patch reads bot?.union_id from the bot-info response, but the paired report and closed sibling PR say that endpoint does not return union_id. If this field is absent, the fallback never runs for the reported multi-bot case.

  - confidence: 0.88
- **[P2] Strip the union-id matched bot mention:** `extensions/feishu/src/bot.ts:797`
  - body: After mentionedBot can become true through union_id, normalizeMentions still strips only a mention whose open_id equals botOpenId. In the mismatch case, @Bot /model can keep the bot placeholder and miss slash-command parsing.
  - confidence: 0.9
- **[P2] Carry fallback identity through debounce:** `extensions/feishu/src/monitor.account.ts:251`
  - body: The final dispatch receives botUnionId, but debounce still builds merged text and preserved mentions with botOpenId-only parsing and filtering. Batched messages can still lose the bot mention and be dropped under requireMention.
  - confidence: 0.86

  - body: The branch still treats raw @_all as mentioning every bot, but current main deliberately excludes broadcast-only @all/@_all and has docs/tests for that behavior. A refreshed fix must preserve the broadcast filtering.
