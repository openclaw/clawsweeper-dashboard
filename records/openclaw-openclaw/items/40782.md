The PR threads a Feishu bot union_id through probe/startup state into message parsing and uses it as a fallback for group @mention detection when open_id does not match.

## Best Possible Solution

Rebase or replace the branch with a Feishu-validated stable identity matcher applied consistently to text, rich-text, mention stripping, and debounce paths, with focused regression coverage and redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose delivered mention open_id differs from botOpenId makes current checkBotMentioned return false, and the requireMention group path returns with the reported did-not-mention log. I did not establish a live Feishu tenant reproduction.

## Solution Assessment

No, not as written: union_id fallback is only safe if the bot identity source is verified, and this branch still misses rich-text post, mention stripping, and debounce paths. A safer fix should first prove the Feishu identity contract and then apply it consistently across the message-admission pipeline.
