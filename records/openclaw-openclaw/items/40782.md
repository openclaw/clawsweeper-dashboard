## Summary

Keep open: this is a useful implementation attempt for https://github.com/openclaw/openclaw/issues/40768, but it is still conflicting, lacks after-fix Feishu multi-bot proof, and depends on an unproven `union_id` source that the linked issue and dependency inspection do not support.

## What This Changes

The PR threads a Feishu bot `union_id` from probe/startup/state into message parsing so `@mention` detection can fall back from `open_id` in multi-bot groups.

## Best Possible Solution

Rewrite the fix on current main around a proven stable Feishu bot identity source, then cover text, post, mention-forward, slash-command stripping, debounce behavior, regression tests, and redacted live Feishu proof before merge.

## Reproduction Assessment

Yes, at source level: when a Feishu event mention `open_id` differs from `botOpenId`, current main sets `mentionedBot` false and the group `requireMention` gate can drop the message. I did not run a live multi-bot Feishu tenant in this read-only review.

## Solution Assessment

No. This PR is a plausible old-base mitigation, but it is not the best fix because the identity source is unproven, the branch conflicts with current main, and the fallback does not cover all Feishu mention paths.

Full review comments:

- **[P1] Prove the union_id source before using it:** `extensions/feishu/src/probe.ts:137`
  - body: The patch reads `bot?.union_id` from `bot/v3/info`, but the branch's response type only declares `bot_name` and `open_id`, the linked issue investigation says that field is absent, and the inspected SDK/current probe contract does not expose bot `union_id`. If this stays undefined, the fallback never fixes the multi-bot drop.
  - confidence: 0.95
- **[P2] Strip the matched fallback mention before command parsing:** `extensions/feishu/src/bot.ts:797`
  - body: `checkBotMentioned` can return true via `botUnionId`, but `normalizeMentions` still strips only `botOpenId`. In the mismatch case, `@Bot /help` keeps the bot mention in content and can still break slash-command detection in the same scenario the PR is trying to fix.
  - confidence: 0.88
- **[P2] Carry fallback identity into rich-text post matching:** `extensions/feishu/src/bot.ts:479`
  - body: The fallback only applies when top-level `message.mentions` is populated; the rich-text post branch still compares parsed mention IDs to `botOpenId` only. Post messages with the same app-scoped mismatch can still be treated as unmentioned.
  - confidence: 0.84
- **[P2] Pass fallback identity through debounce merging:** `extensions/feishu/src/monitor.account.ts:251`
  - body: This handoff passes `botUnionId` to the final message handler, but debounce text and retained bot mentions are still derived from open-id-only checks before that handoff. Batched messages can therefore still lose the bot mention under `requireMention`.
  - confidence: 0.83
