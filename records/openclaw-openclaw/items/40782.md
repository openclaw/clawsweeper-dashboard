## Summary

Keep open: this PR targets a real Feishu multi-bot message-loss bug, but the branch is not merge-ready because the fallback identity source is disputed, current main has moved the affected code paths, and there is no after-fix real Feishu/Lark proof.

## What This Changes

The PR threads a Feishu `botUnionId` from probe/startup/monitor state into message parsing so `union_id` can be used as a fallback when mention `open_id` mismatches.

## Best Possible Solution

Use a Feishu-supported stable identity or cross-app `open_id` mapping consistently across current-main detection, rich-text posts, mention stripping, mention-forward targeting, and debounce merging, with focused regression tests and redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level: current main compares Feishu bot mentions by `open_id` only, so a group event whose delivered mention `open_id` differs from `botOpenId` makes `mentionedBot` false and the require-mention path returns without dispatch. I did not perform live Feishu validation.

## Solution Assessment

No, not as-is: the PR uses a `union_id` source that the linked issue says is unavailable, and it does not carry the fallback through current main's stripping, rich-text, mention-forward, and debounce surfaces.
