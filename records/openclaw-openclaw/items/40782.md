Overall confidence: 0.86

Full review comments:

- **[P1] Prove the union_id source before relying on it:** `extensions/feishu/src/probe.ts:137`
  - body: This adds `botUnionId: bot?.union_id`, but the linked issue, closed same-author sibling, and official bot-info docs do not establish that Feishu bot info returns `union_id`. If that remains true, `botUnionId` is always undefined and the new fallback never fixes the reported multi-bot mention loss.
  - confidence: 0.91
- **[P2] Apply fallback matching outside text mentions:** `extensions/feishu/src/bot.ts:473`
  - body: The new fallback only participates in the `message.mentions` branch. Rich-text post mentions still flow through the open_id-only post branch, so the same app-scoped mismatch can still make a post @mention look unmentioned.
  - confidence: 0.82
- **[P2] Strip fallback-matched bot mentions before command parsing:** `extensions/feishu/src/bot.ts:797`
  - body: `parseFeishuMessageEvent` accepts `botUnionId`, but content normalization still strips only by `botOpenId`. A multi-bot `@Bot /help` that matched by union_id can keep the raw mention in front of `/help`, regressing the slash-command behavior this parser comment protects.
  - confidence: 0.86
- **[P2] Carry fallback identity through debounced batches:** `extensions/feishu/src/monitor.account.ts:251`
  - body: The final handler receives `botUnionId`, but the debounced batch path still derives merged text and bot mentions from open_id-only parsing. A burst where the target mention is only in an earlier event can still be merged into a message that fails the requireMention gate.
  - confidence: 0.8
