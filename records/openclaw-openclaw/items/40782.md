## Summary

Keep open. Current main still has the reported Feishu multi-bot @mention failure, but this PR is not merge-ready because the proposed union_id source is unproven, several mention paths remain open_id-only, and the patch would conflict with current @all mention policy.

## What This Changes

The PR adds a Feishu botUnionId field from bot info and threads it through monitor state and message parsing so bot mention matching can fall back from open_id to union_id.

## Best Possible Solution

Refresh this as a Feishu-owned fix against current main after verifying the identity contract, then update all mention consumers consistently while preserving broadcast-only @all behavior and adding focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: a Feishu group event whose mention.id.open_id differs from botOpenId leaves mentionedBot false and is dropped when requireMention is enabled; #40768 adds concrete multi-bot steps.

## Solution Assessment

No. The union_id fallback is plausible only after a verified source exists, and this patch leaves startup preflight, debounce preservation, and mention stripping inconsistent with the new identity path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Verify a stable bot identity source:** `extensions/feishu/src/probe.ts:137`
  - body: The patch reads botUnionId from the bot-info response, but the paired report and closed sibling PR say that endpoint does not return union_id. If this field is absent, the fallback is never populated and the multi-bot mention failure remains.
  - confidence: 0.9
- **[P2] Pass fallback identity through startup preflight:** `extensions/feishu/src/monitor.startup.ts:40`
  - body: Returning botUnionId from startup is not enough because the current multi-account startup caller still destructures and passes only botOpenId and botName. Normal startup would leave the new union-id state empty.
  - confidence: 0.88
- **[P2] Strip the union-id matched bot mention:** `extensions/feishu/src/bot.ts:797`
  - body: After mentionedBot can become true through union_id, normalizeMentions still strips only a mention whose open_id equals botOpenId. In the mismatch case, @Bot /model keeps the bot placeholder and can miss slash-command parsing.
  - confidence: 0.9
- **[P2] Carry fallback identity through debounce:** `extensions/feishu/src/monitor.account.ts:251`
  - body: This only passes botUnionId to final dispatch, while debounce text and merged mention preservation still parse and filter by botOpenId. Batched multi-bot messages can still lose the bot mention and be dropped under requireMention.
  - confidence: 0.86
- **[P2] Preserve broadcast-only at-all filtering:** `extensions/feishu/src/bot.ts:459-460`
  - body: The raw @_all shortcut makes broadcast-only group messages count as bot mentions, but current docs and tests require @all/@_all alone not to wake every Feishu bot.
  - confidence: 0.84
