## Summary

Keep open: the PR targets a real Feishu multi-bot mention bug, but it is not merge-ready because the proposed union-id source is unresolved and the fallback is not carried through mention stripping or debounce handling.

## What This Changes

The PR threads a Feishu `union_id` from startup probe state into group @mention detection as a fallback when `open_id` mismatches.

## Best Possible Solution

Keep this PR as the canonical repair path for #40768, but update it against current main with a verified Feishu-supported stable bot identity source and cover detection, stripping, debounce, and regression tests before merge.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence path: a group event whose mention has a different app-scoped `open_id` from the receiving bot is treated as unmentioned and then dropped by `requireMention`.

## Solution Assessment

No. A union-id fallback is only maintainable if OpenClaw can reliably obtain the receiving bot's union id, and this patch also leaves command stripping and debounce mention merging on the old open-id-only path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Fetch a stable bot identity before matching union_id:** `extensions/feishu/src/probe.ts:134`
  - body: This PR reads `bot?.union_id` from startup bot info and relies on it to make the new fallback work, but the linked issue and the closed duplicate say that `bot/v3/info` does not actually return `union_id`. When `botUnionId` stays undefined, `checkBotMentioned` still falls back to the same app-scoped `open_id` comparison, so the multi-bot bug remains.
  - confidence: 0.87
- **[P2] Strip the union-id matched bot mention:** `extensions/feishu/src/bot.ts:797`
  - body: After a union-id fallback match, `parseFeishuMessageEvent` still strips mentions using only the stored bot `open_id`. In the same mismatch scenario, `@Bot /model` keeps the Feishu mention tag in `ctx.content`, so the command probe no longer sees the leading slash that the existing #35994 handling depends on.
  - confidence: 0.84
- **[P2] Carry union-id matching through debounce merging:** `extensions/feishu/src/monitor.account.ts:248`
  - body: The PR passes `botUnionId` to the final message handler, but the inbound debounce path still resolves text and merged mentions with `botOpenId` only. A burst where only an earlier message mentions the bot by matching `union_id` can flush without the bot mention metadata and still be dropped under `requireMention`.
  - confidence: 0.82
