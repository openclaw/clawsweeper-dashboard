## Summary

Keep open: this PR targets a real Feishu multi-bot @mention bug, but the branch is not merge-ready because the proposed union_id source is disputed, several mention-sensitive paths remain open_id-only, and after-fix real Feishu proof is missing.

## What This Changes

The branch threads a Feishu bot union ID through probe, startup, monitor state, and message parsing so mention detection can fall back from open_id to union_id.

## Best Possible Solution

Land a repaired or replacement Feishu fix that obtains a verified stable bot identity or cross-app mention mapping, applies the same matcher to detection, command stripping, debounce merging, and mention-forward handling, and includes focused regression tests plus redacted live multi-bot proof.

## Reproduction Assessment

Yes, at source level. A Feishu group event whose mention metadata has an open_id different from the receiving account's botOpenId will keep mentionedBot=false, and the documented requireMention group path can drop it.

## Solution Assessment

No, not as written. The affected surface is Feishu mention identity matching, but this branch depends on an unverified union_id source and leaves command stripping, debounce merging, and mention-forward handling on open-id-only logic.

## Review Findings

Full review comments:

- **[P2] Fetch union IDs from a source that returns them:** `extensions/feishu/src/probe.ts:134`
  - body: This stores botUnionId from bot?.union_id, but the linked issue says this bot-info source does not return union_id, and the generated bot/v3 BotInfo contract does not expose that field. When botUnionId is absent, the fallback never runs and the reported multi-bot drop remains. ([pkg.go.dev](https://pkg.go.dev/github.com/larksuite/oapi-sdk-go/service/bot/v3))
  - confidence: 0.9
- **[P2] Strip the union-id matched bot mention before parsing commands:** `extensions/feishu/src/bot.ts:797`
  - body: After checkBotMentioned can succeed by union_id, the parser still normalizes content using only botOpenId. In the mismatched-open-id case, the bot mention remains in ctx.content, so messages like @Bot /model are not reduced to /model before command handling.
  - confidence: 0.86
- **[P2] Carry the fallback through debounce and mention forwarding:** `extensions/feishu/src/monitor.account.ts:248`
  - body: The direct receive wiring adds botUnionId, but debounce text, merged mention selection, and mention-forward helpers still use open-id-only logic. If the bot mention is in an earlier debounced entry or part of a forward request, the flushed event can still fail the require-mention gate or misclassify targets.
  - confidence: 0.82
