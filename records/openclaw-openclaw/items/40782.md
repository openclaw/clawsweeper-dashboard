## Summary

Keep open: this PR targets a real Feishu mention-gating bug, but the patch is not merge-ready because its proposed union_id source is disputed, several mention-sensitive paths remain open-id-only, and contributor real behavior proof is still missing.

## What This Changes

The PR threads a Feishu bot union ID from probe/startup state into message parsing so @mention detection can fall back from open_id to union_id in multi-bot groups.

## Best Possible Solution

Land a rebased Feishu fix that uses a verified stable bot identity or cross-app mention mapping consistently across detection, command stripping, debounce merging, and mention-forward handling, with focused regression tests and redacted live multi-bot proof.

## Reproduction Assessment

Yes, at source level. A Feishu group event whose mention metadata has an open_id different from the receiving account’s botOpenId will set `mentionedBot=false`, and the documented require-mention group path then drops the message.

## Solution Assessment

No, not as written. The Feishu mention identity surface is the right place to fix this, but this branch relies on an unverified union_id source and leaves command stripping, debounce merging, and mention-forward handling tied to open_id-only logic.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Fetch union IDs from a verified source:** `extensions/feishu/src/probe.ts:134`
  - body: This stores `botUnionId` from `bot?.union_id`, but the linked Feishu investigation and the closed sibling PR say the bot-info source does not return that field. When `botUnionId` is absent, the fallback never runs and the reported multi-bot drop remains.
  - confidence: 0.9
- **[P2] Strip the union-id matched bot mention before parsing:** `extensions/feishu/src/bot.ts:790`
  - body: After `checkBotMentioned` can succeed by `union_id`, the parser still normalizes content using only `botOpenId`. In the mismatched-open-id case, the bot mention stays in `ctx.content`, so messages such as `@Bot /model` are not reduced to `/model` before command handling.
  - confidence: 0.86
- **[P2] Thread the fallback through debounce and forward handling:** `extensions/feishu/src/monitor.account.ts:248`
  - body: The direct receive wiring passes `botUnionId`, but debounce text, merged mention selection, and mention-forward helpers still use open-id-only logic. If the bot mention appears in an earlier debounced entry or in a forward request, the flushed event can still fail the require-mention gate or misclassify targets.
  - confidence: 0.82
