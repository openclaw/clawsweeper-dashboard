## Summary

Keep open: the underlying Feishu multi-bot mention bug is valid and still present on main, but this PR is not merge-ready because the fallback identifier is not reliably populated and the parser/debounce paths remain open_id-only; the existing human-review pause should stand.

## What This Changes

This PR attempts to thread a Feishu bot union_id from startup probing into monitor state and use it as a fallback when detecting @mentions in multi-bot groups.

## Best Possible Solution

Repair the PR around a verified Feishu bot identity source, then thread that identity through mention detection, stripping, mention-forward/debounce handling, startup recovery state, and focused regression tests before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: construct a Feishu group text event whose mention has a mismatching open_id and a matching union_id; current main's checkBotMentioned only compares open_id, so requireMention drops it.

## Solution Assessment

No. Stable-identifier matching is the right direction, but this implementation depends on an unpopulated botUnionId and leaves command stripping plus debounce mention preservation on open_id-only logic.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Do not depend on an unpopulated botUnionId:** `extensions/feishu/src/probe.ts:134`
  - body: This reads `bot?.union_id` from the startup probe, but the current probe response shape only provides bot name/open_id and the related closed PR notes that `bot/v3/info` did not return `union_id`. With `botUnionId` never populated, the fallback path never runs and the multi-bot `requireMention` failure remains.
  - confidence: 0.88
- **[P2] Strip union-matched mentions before command parsing:** `extensions/feishu/src/bot.ts:791`
  - body: After this line can mark the bot as mentioned via `union_id`, the content normalization below still strips only `botOpenId`. In the exact open_id-mismatch case, `@Bot /model` leaves the Feishu mention tag in the text and breaks the command path that the existing comment ties to #35994.
  - confidence: 0.86
- **[P2] Carry botUnionId through the debounce path:** `extensions/feishu/src/monitor.account.ts:251`
  - body: Adding `botUnionId` to the final message handler is not enough because the receive/debounce path still derives text and filters merged mentions with only `botOpenId`. A debounced batch can therefore drop the union_id-matched bot mention before `handleFeishuMessage` checks `requireMention`.
  - confidence: 0.82
