## Summary

Keep open: this is a plausible Feishu message-delivery fix path, but the PR is protected for human review, conflicts with current main, lacks real Feishu proof, and still has blocking identity and propagation gaps.

## What This Changes

The branch adds a botUnionId field to Feishu probe/startup/monitor state and uses it as a fallback identifier in Feishu @mention detection.

## Best Possible Solution

Use a Feishu-supported stable identity or cross-app open_id mapping and apply it consistently across current-main mention detection, rich-text posts, mention stripping, mention-forward targeting, and debounce merging, with focused regression tests and redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level: current main compares Feishu bot mentions by open_id only, so a group event whose delivered mention open_id differs from botOpenId makes mentionedBot false and the requireMention path returns without dispatch. I did not perform live Feishu validation.

## Solution Assessment

No, not as-is: the PR uses a union_id source that the linked issue and sibling PR say is unavailable, and it does not carry the fallback through current main's stripping, rich-text, mention-forward, and debounce surfaces.

Full review comments:

- **[P1] Use a proven bot identity source:** `extensions/feishu/src/probe.ts:137`
  - body: The patch reads `bot?.union_id`, but the response type declared in this file only contains `bot_name` and `open_id`, and the linked Feishu investigation says this endpoint did not return `union_id` in the affected setup. As written, the fallback identity is type-incomplete and may stay undefined, leaving the reported multi-bot messages dropped.
  - confidence: 0.9
- **[P2] Strip the mention matched by fallback identity:** `extensions/feishu/src/bot.ts:790`
  - body: After `checkBotMentioned` can return true via `botUnionId`, the parsed content still strips the bot mention only by `botOpenId`. In the mismatch case this leaves the bot's mention tag in `@Bot /help`, so command probing can still miss the leading slash even though the message was admitted.
  - confidence: 0.86
- **[P2] Carry fallback identity through debounce:** `extensions/feishu/src/monitor.account.ts:251`
  - body: The final dispatch receives `botUnionId`, but debounce text and merged mention selection still parse/filter with `botOpenId` only. A multi-message debounce window can therefore keep the wrong mention text or drop the bot mention before the fallback-aware handler runs.
  - confidence: 0.83
- **[P2] Cover rich-text mention matching:** `extensions/feishu/src/bot.ts:470-473`
  - body: The new fallback is only applied to top-level `message.mentions`; the `post` branch still compares parsed rich-text mention IDs only with `botOpenId`. Rich-text posts in the same app-scoped mismatch scenario can still be treated as unmentioned and skipped.
  - confidence: 0.78
