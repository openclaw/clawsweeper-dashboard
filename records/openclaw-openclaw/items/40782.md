## Summary

Keep open: the linked Feishu bug is source-reproducible on current main, but this PR is not merge-ready because the union-id source is unverified, downstream mention paths remain open-id-only, real after-fix proof is missing, and the repair loop is explicitly paused for human review.

## What This Changes

The PR threads a Feishu `botUnionId` from probing through monitor state into message parsing and uses it as a fallback for group @mention detection when `open_id` differs.

## Best Possible Solution

Repair this PR or replace it with a Feishu-validated identity fix that covers mention detection, mention stripping, debounce/forward handling, state cleanup, regression tests, and real multi-bot proof.

## Reproduction Assessment

Yes, at source level: current main compares Feishu @mention metadata to `botOpenId` only, so a pushed mention with a different app-scoped `open_id` produces `mentionedBot=false` and reaches the existing group require-mention drop path. I did not live-test against Feishu.

## Solution Assessment

No, not as written. A union-id fallback is only maintainable if the bot union id comes from a verified Feishu source and every mention-sensitive path uses it; this patch leaves source verification, command stripping, and debounce handling incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Fetch union IDs from a verified Feishu identity source:** `extensions/feishu/src/probe.ts:134`
  - body: This stores `botUnionId` from `bot?.union_id`, but the linked Feishu investigation and closed follow-up say the attempted bot-info source does not return that field. When it is absent, the fallback never runs and the reported multi-bot mention drop remains.
  - confidence: 0.88
- **[P2] Strip the matched union-id mention before command parsing:** `extensions/feishu/src/bot.ts:797`
  - body: After `checkBotMentioned` can succeed by `union_id`, `parseFeishuMessageEvent` still normalizes content using only `botOpenId`. In the mismatched-open_id case, the bot mention stays in `ctx.content`, so `@Bot /model` is not reduced to `/model` before command handling.
  - confidence: 0.86
- **[P2] Carry union IDs through the debounce merge path:** `extensions/feishu/src/monitor.account.ts:248`
  - body: This direct receive wiring adds `botUnionId`, but debounce text and merged mention selection still use open-id-only helpers. If the bot mention is in an earlier debounced entry, the flushed event can still fail the require-mention gate.
  - confidence: 0.8
