## Summary

This PR should not merge as written: the diagnosis is credible, but the documented Feishu endpoint cannot populate the fallback, the implementation misses related mention-processing paths, and the branch predates the current probe architecture.

## What This Changes

The PR propagates a proposed Feishu bot `union_id` from startup probing into direct group-mention detection as a fallback for app-scoped `open_id` mismatches.

## Best Possible Solution

Choose a documented way to map the receiving Feishu bot to event mentions, then implement one canonical resolver across detection, command-preserving stripping, and debounce merging on current main.

## Reproduction Assessment

Yes at source level: the reported app-scoped `open_id` mismatch reaches an equality check that rejects the addressed bot, and the official API confirms the proposed fallback is unavailable from the chosen endpoint. A live multi-bot Feishu run was not performed.

## Solution Assessment

No; the patch cannot populate `botUnionId` from the documented endpoint and does not cover command stripping or debounce mention merging, so maintainers must first choose a supported identity-mapping design.
