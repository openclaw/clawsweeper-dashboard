## Summary

Keep this PR open: the reported Feishu multi-bot mention bug is still real on current main, but this branch is stale and incomplete, with blockers in the union-id source, debounce path, and command-stripping path.

## What This Changes

The PR adds a Feishu botUnionId identity field and threads it into mention detection so union_id can be used when open_id matching fails.

## Best Possible Solution

Rework the Feishu plugin fix on current main with a verified stable bot identifier, carry it through parser, debounce, and mention-forward paths, strip the resolved bot mention before command parsing, and add regression coverage for text, post/rich-text, and debounced messages.

## Reproduction Assessment

Yes. Source inspection gives a concrete path: a group event whose mention open_id differs from botOpenId but whose union_id matches would be dropped on current main, and this PR still fails related command-stripping and debounce cases.

## Solution Assessment

No. Stable-ID matching is the right direction, but this patch is not the best complete fix because the union_id source is unproven and several open_id-only paths remain in the message pipeline.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Resolve the bot mention before stripping content:** `extensions/feishu/src/bot.ts:791`
  - body: After this change, `mentionedBot` can be true because `union_id` matched, but `normalizeMentions` still strips only `botOpenId`. In the reported multi-bot case the mention's open_id is the mismatched app-scoped value, so content like `@Bot /model` keeps a mention tag and the command parser will not see `/model`.
  - confidence: 0.92
- **[P2] Carry union IDs through debounce mention merging:** `extensions/feishu/src/monitor.account.ts:251`
  - body: This only passes `botUnionId` to the final message handler, while the debounce receiver and `resolveFeishuDebounceMentions` still know only `botOpenId`. If a debounced batch has the bot mention on an earlier message and Feishu supplied only the foreign open_id plus matching union_id, the merged dispatch can lose the bot mention and be dropped under `requireMention`.
  - confidence: 0.86
- **[P2] Verify a real union-id source before using it:** `extensions/feishu/src/probe.ts:137`
  - body: The fallback depends on `botUnionId` coming from probe, but the related issue says `bot/v3/info` did not return `union_id`, and current main's probe shape exposes only botID/botName. If this field is always undefined, the new fallback never activates for the reported setup.
  - confidence: 0.8
