## What This Changes

The PR threads a Feishu bot union ID through probe, startup, monitor state, and message parsing so @mention detection can fall back from open_id to union_id.

## Best Possible Solution

Land a rebased Feishu fix that obtains a verified cross-app bot identity, applies it consistently to mention detection, mention stripping, debounce merging, and mention-forward detection, and includes regression tests plus live multi-bot proof.

## Reproduction Assessment

Yes, at source level. Current main still gates group messages on an open-id-only matcher, so a Feishu event whose mention open_id differs from the receiving botOpenId will be treated as not mentioning the bot when requireMention is enabled.

## Solution Assessment

No, not as written. A union-id fallback could be maintainable only if the bot union ID comes from a verified Feishu source and every mention-sensitive path uses the same matcher; this patch leaves those pieces incomplete.

## Review Findings

Overall confidence: 0.87

Full review comments:

- **[P2] Fetch union IDs from a source that returns them:** `extensions/feishu/src/probe.ts:134`
  - body: This stores botUnionId from bot?.union_id, but the linked Feishu investigation says the attempted bot-info source does not return that field, and current main's probe contract only exposes botID and botName. When union_id is absent, the fallback never runs and the reported multi-bot drop remains.
  - confidence: 0.88
- **[P2] Strip the union-id matched bot mention before parsing commands:** `extensions/feishu/src/bot.ts:797`
  - body: After checkBotMentioned can succeed by union_id, parseFeishuMessageEvent still normalizes content using only botOpenId. In the mismatched-open-id case, the bot mention remains in ctx.content, so messages like @Bot /model are not reduced to /model before command handling.
  - confidence: 0.86
- **[P2] Carry the fallback through debounce and mention forwarding:** `extensions/feishu/src/monitor.account.ts:248`
  - body: The direct receive wiring adds botUnionId, but debounce text, merged mention selection, and mention-forward helpers still use open-id-only logic. If the bot mention is in an earlier debounced entry or part of a forward request, the flushed event can still fail the require-mention gate or misclassify targets.
  - confidence: 0.8
