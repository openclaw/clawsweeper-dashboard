## Summary

Keep open: this is still the linked repair path for a source-reproducible Feishu message-loss bug, but the branch is not merge-ready because the proposed union_id source is disputed, fallback matching is incomplete, real Feishu proof is missing, and the PR is currently conflicting and paused for human review.

## What This Changes

The PR threads a Feishu botUnionId from probe/startup state into monitor state and message parsing so @mention detection can fall back from open_id to union_id.

## Best Possible Solution

Repair or replace this branch with a current-main Feishu identity strategy that is validated against Feishu, applied consistently to text, rich text, mention stripping, debounce, and mention-forward paths, with focused regression tests and redacted live proof.

## Reproduction Assessment

Yes, at source level: a Feishu group event whose non-broadcast mention open_id differs from botOpenId makes current checkBotMentioned return false, and the requireMention group path then drops it. I did not establish a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

No, not as written. The PR targets the right open_id mismatch, but depends on an unavailable/untyped union_id source and misses rich-text, mention stripping, and debounce paths; the safer direction is a Feishu-validated identity matcher rebased onto current file boundaries.

Full review comments:

- **[P2] Use a verified bot identity source:** `extensions/feishu/src/probe.ts:137`
  - body: This adds `bot?.union_id`, but `FeishuBotInfoResponse` in this branch only models `bot_name` and `open_id`, and the linked investigation reports that `bot/v3/info` did not return `union_id`. If that field is absent, `botUnionId` stays undefined and the fallback never fixes the multi-bot mention drop.
  - confidence: 0.9
- **[P2] Match fallback identities in rich-text posts:** `extensions/feishu/src/bot.ts:477-480`
  - body: The new matcher can use `botUnionId` for normal mention metadata, but the post-message branch still compares parsed rich-text mention IDs only to `botOpenId`. Rich-text @mentions in the same app-scoped mismatch case can still be treated as not mentioning the bot.
  - confidence: 0.84
- **[P2] Strip fallback-matched bot mentions before parsing:** `extensions/feishu/src/bot.ts:797`
  - body: After `checkBotMentioned` succeeds through `botUnionId`, `normalizeMentions` still strips only `botOpenId`. In the mismatched-open_id case, messages like `@Bot /model` keep the bot mention tag in content, so command probing can miss the slash command.
  - confidence: 0.88
- **[P2] Carry fallback identity through debounce parsing:** `extensions/feishu/src/monitor.account.ts:251`
  - body: Adding `botUnionId` to final dispatch is not enough because debounce text resolution and merged mention filtering still parse/filter with `botOpenId` only. A debounced batch whose earlier entry mentions the bot through the fallback identity can still lose the mention before dispatch.
  - confidence: 0.8
