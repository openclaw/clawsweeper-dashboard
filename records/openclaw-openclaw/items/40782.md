## Summary

Keep open. Current main still has open_id-only Feishu bot mention matching and the paired bug remains unresolved, but this PR is not merge-ready because it relies on an unverified union_id source, is stale against current Feishu refactors, and leaves debounce, mention stripping, and broadcast mention behavior inconsistent.

## What This Changes

The PR threads a Feishu bot union_id from probe results into monitor state and message parsing so @mention detection can fall back from open_id to union_id in multi-bot groups.

## Best Possible Solution

Refresh the Feishu fix after verifying a stable cross-app bot identity or open_id mapping strategy, then update the current bot-content, debounce, startup/retry state, and mention-stripping paths together with focused regression tests and a changelog entry.

## Reproduction Assessment

Yes. The paired report gives concrete multi-bot Feishu group steps, and current main has a source-level reproduction: a group event whose mention.id.open_id differs from botOpenId leaves mentionedBot false and is dropped under requireMention.

## Solution Assessment

No. The union_id fallback is plausible only after proving a reliable bot identity source, and this branch does not update all current-main mention consumers that would need the fallback.
