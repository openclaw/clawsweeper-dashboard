## Summary

Keep open: current main still lets Claude live active-tool heartbeat events refresh diagnostic progress, and an open linked PR is the right fix path but has not merged yet.

## What This Changes

Report that Claude CLI active-tool heartbeat progress can keep a wedged native tool looking active when no matching tool_result arrives, preventing stuck-session detection from aging normally.

## Best Possible Solution

Review and land or explicitly reject https://github.com/openclaw/openclaw/pull/96198; the desired end state is heartbeat observability without treating synthetic ticks as real diagnostic progress.

## Reproduction Assessment

Yes. The source path is high-confidence: a Claude live tool_use without a matching tool_result leaves activeTools populated, the heartbeat emits cli_live:tool_running, and current diagnostic activity records that as fresh progress.

## Solution Assessment

Yes, the narrowest maintainable fix is to treat synthetic Claude live active-tool heartbeats as passive diagnostic events rather than adding a new timeout/config policy. The open linked PR attempts that boundary fix while preserving heartbeat observability.
