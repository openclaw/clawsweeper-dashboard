## Label Justifications

- `P1`: The issue reports real scheduled Claude CLI cron runs staying wedged far beyond their intended recovery window, affecting live user automation.
- `impact:crash-loop`: A wedged subprocess can keep cron agent work alive until a much longer lane timeout or operator intervention, which is an availability/hang failure.
- `impact:session-state`: The failure is caused by diagnostic active-tool and last-progress state drifting from actual subprocess progress.

## Summary

Keep open. Current main and the latest release still let the Claude live active-tool heartbeat refresh diagnostic progress while an active tool remains unmatched, so the stuck-session recovery predicates cannot age out the wedged-tool state; the older merged heartbeat fix solved the opposite false-abort case, not this wedged-subprocess case.

## What This Changes

The issue asks OpenClaw to stop Claude CLI active-tool heartbeat events from masking wedged native subprocesses and defeating stuck-session recovery.

## Best Possible Solution

Preserve the valid long-running-tool protection from https://github.com/openclaw/openclaw/pull/87546, but add a bounded stale-tool policy for Claude live tools whose only freshness is the synthetic heartbeat and whose `tool_result` never arrives.

## Reproduction Assessment

Yes, at source level: a Claude live `tool_use` without a matching `tool_result` leaves `activeTools` non-empty, the heartbeat refreshes `lastProgressAt`, and blocked-tool recovery requires that freshness age to exceed the abort threshold. I did not run the live wedged subprocess scenario in this read-only review.

## Solution Assessment

Unclear as a final product policy, but the narrow owner boundary is the Claude live diagnostic/recovery path rather than provider routing or Anthropic API failover. The safest repair should reuse existing stuck-session thresholds or a maintainer-approved explicit limit, and must test both wedged tools and legitimate long-running silent tools.
