## Summary

Keep open: current main and the latest release still let Claude live `cli_live:tool_running` heartbeat events refresh diagnostic session progress, and the linked closing PR has not merged.

## What This Changes

The issue asks OpenClaw to stop Claude CLI active-tool heartbeat events from masking wedged subprocesses in stuck-session detection.

## Best Possible Solution

Review and land or explicitly reject https://github.com/openclaw/openclaw/pull/96198; the desired end state is heartbeat observability without treating synthetic ticks as real diagnostic progress.

## Reproduction Assessment

Yes, source-reproducible: a Claude live `tool_use` without a matching `tool_result` leaves `activeTools` populated, the heartbeat emits `cli_live:tool_running`, and current diagnostic activity records that as fresh progress.

## Solution Assessment

Yes, the narrowest maintainable fix is the linked diagnostic-boundary approach: keep heartbeat observability but stop synthetic heartbeat ticks from refreshing last real progress. Adding a new timeout/config policy would be broader than needed unless maintainers reject that approach.
