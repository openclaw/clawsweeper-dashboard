## Summary

Keep open. Current main still creates a fresh `GatewayClient` for every `probeGateway()` call and has no per-URL `device-required` suppression, while the exact source-level fix remains in closed, unmerged PR #63446; adjacent PR #68280 addresses missing local probe auth, not repeated device-required cache dampening.

## What This Changes

The issue asks for CLI gateway WebSocket probes rejected with `device-required` to back off or short-circuit across repeated short-lived probe invocations instead of opening a new rejected WebSocket each time.

## Best Possible Solution

Land a narrow `probeGateway()`-boundary fix that records recent `DEVICE_IDENTITY_REQUIRED` closes per gateway URL, short-circuits after a small threshold within a TTL without constructing another `GatewayClient`, clears on successful connection, and proves the behavior with focused unit tests. Keep the unauthenticated HTTP health endpoint as a separate product/security discussion.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock `GatewayClient` so repeated `probeGateway()` calls close with code 1008 and reason `device identity required`; current main constructs a new mock client on each call and has no fourth-call short-circuit test.

## Solution Assessment

Yes. A per-URL cache in `src/gateway/probe.ts` is the narrowest maintainable fix because the storm comes from fresh probe clients across short-lived invocations; changing only `GatewayClient` reconnect backoff cannot persist across instances, and adding `/health` would broaden the security/product surface.

## Repair Work Prompt

Fix openclaw/openclaw#63427 with a narrow gateway probe change. Current main creates a new `GatewayClient` inside every `probeGateway()` call; when an unpaired/local probe is rejected with code 1008 / `device identity required` / `DEVICE_IDENTITY_REQUIRED`, later short-lived CLI/status/cron probes have no memory of the failure and keep opening WebSockets. Add a module-level per-URL cache in `src/gateway/probe.ts` that tracks recent device-required probe rejections, allows the first small number of real attempts, then short-circuits subsequent probes for that URL within a TTL without constructing or starting `GatewayClient`. Add focused tests in `src/gateway/probe.test.ts` for: first threshold attempts still create/start a client, the next call short-circuits without client construction, non-device-required closes do not populate the cache, success clears the cache, and different URLs are independent. Use PR #63446 as prior art but review the bot comment there about threshold/cache-clear behavior before copying. Do not add an unauthenticated `/health` endpoint, do not change gateway auth/security policy, and do not duplicate this policy in individual status callers.
