## Summary

Keep open. Current main still creates a fresh probe WebSocket client for each `probeGateway()` call with no per-URL `device-required` suppression; the exact cache fix is only in closed unmerged https://github.com/openclaw/openclaw/pull/63446, while open https://github.com/openclaw/openclaw/pull/68280 covers adjacent local probe auth fail-fast behavior rather than repeated rejection dampening.

## What This Changes

Add dampening for repeated CLI gateway WebSocket probes rejected with `device-required`, avoiding repeated handshake failures from short-lived status/cron sessions.

## Best Possible Solution

Land a narrow `probeGateway()`-boundary cache that records recent `DEVICE_IDENTITY_REQUIRED` closes per gateway URL, short-circuits after a small threshold within a TTL, clears on successful connection, and leaves an unauthenticated HTTP health endpoint for separate product/security discussion.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: mock repeated `probeGateway()` calls that close with code 1008 and reason `device identity required`, and current main will construct a new `GatewayClient` for each call because no cache or fourth-call short-circuit exists before client construction.

## Solution Assessment

Yes. A per-URL cache in `src/gateway/probe.ts` is the narrowest maintainable fix because GatewayClient reconnect backoff is per instance, while the reported storm comes from fresh short-lived probe clients; adding `/health` would broaden the surface.
