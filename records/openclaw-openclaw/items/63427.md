## Summary

Keep open. Current main still builds a fresh `GatewayClient` inside every `probeGateway()` call with no per-URL `device-required` suppression, the exact cache fix remains only in closed unmerged https://github.com/openclaw/openclaw/pull/63446, and open https://github.com/openclaw/openclaw/pull/68280 addresses adjacent missing local probe auth rather than repeated rejection dampening.

## What This Changes

The issue asks for CLI gateway WebSocket probes that receive `device-required` rejections to stop repeatedly opening unauthenticated probe connections without backoff.

## Best Possible Solution

Land a narrow `probeGateway()`-boundary cache that records recent `DEVICE_IDENTITY_REQUIRED` closes per gateway URL, short-circuits after a small threshold within a TTL, clears on successful connection, and leaves an unauthenticated HTTP health endpoint for separate product/security discussion.

## Reproduction Assessment

Yes, source-reproducible. Mocking `GatewayClient` so repeated `probeGateway()` calls close with code 1008 and reason `device identity required` would still construct a client for each call on current main because there is no cache or fourth-call short-circuit path.

## Solution Assessment

Yes. A per-URL cache in `src/gateway/probe.ts` is the narrowest maintainable fix because `GatewayClient` reconnect backoff is per instance, while the reported storm comes from fresh short-lived probe clients; adding `/health` would broaden the surface.
