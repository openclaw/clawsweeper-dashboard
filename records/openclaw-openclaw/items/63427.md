## Label Justifications

- `P2`: The report describes a valid, limited-blast-radius CLI/gateway probe bug that creates noisy repeated WebSocket auth rejections without making the core runtime unusable.
- `impact:auth-provider`: The issue is centered on device-identity/auth rejection handling for CLI gateway probes.

## Summary

Keep open. Current main and v2026.5.18 still create a fresh `GatewayClient` for each `probeGateway()` call with no per-URL `device-required` suppression; the exact cache-shaped fix exists only in the closed, unmerged https://github.com/openclaw/openclaw/pull/63446, while https://github.com/openclaw/openclaw/pull/68280 covers adjacent probe-auth fail-fast behavior rather than repeated rejection dampening.

## What This Changes

The issue asks OpenClaw to dampen repeated CLI WebSocket probe rejections when an unpaired gateway returns `device-required`, with the HTTP health endpoint left as a separate optional idea.

## Best Possible Solution

Add a narrow `probeGateway()`-boundary cache that records recent `DEVICE_IDENTITY_REQUIRED` closes per gateway URL, short-circuits after a small threshold within a TTL, clears on successful connection, and leaves an unauthenticated HTTP health endpoint for separate product/security discussion.

## Reproduction Assessment

Yes, at source level. Mock repeated `probeGateway()` calls that close with code 1008 and reason `device identity required`; current main will construct a new `GatewayClient` for each call because no cache or fourth-call short-circuit exists before client construction.

## Solution Assessment

Yes. A per-URL cache in `src/gateway/probe.ts` is the narrowest maintainable fix because `GatewayClient` reconnect backoff is per instance, while the reported storm comes from fresh short-lived probe clients.
