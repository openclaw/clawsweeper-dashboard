## Label Justifications

- `P2`: The report describes a valid, limited-blast-radius CLI/gateway probe bug that creates noisy repeated WebSocket auth rejections without making the core runtime unusable.
- `impact:auth-provider`: The issue centers on device-identity/auth rejection handling for CLI gateway probes.

## Summary

Keep open. Current main and v2026.5.18 still create a fresh `GatewayClient` for each `probeGateway()` call, and the exact device-required dampening fix only exists in closed, unmerged https://github.com/openclaw/openclaw/pull/63446; open https://github.com/openclaw/openclaw/pull/68280 covers adjacent local probe auth fail-fast behavior, not repeated device-required suppression.

## What This Changes

Add backoff or per-URL suppression so repeated CLI gateway probes that receive `device-required` do not keep opening WebSockets indefinitely.

## Best Possible Solution

Add a narrow `probeGateway()`-boundary cache for recent 1008 `device identity required` closes per gateway URL, short-circuit after a small threshold within a TTL, clear on successful connection, and leave `/health` for separate product/security discussion.

## Reproduction Assessment

Yes, at source level. Repeated `probeGateway()` calls against a mocked 1008 `device identity required` close will construct a new `GatewayClient` each time because current main has no per-URL cache or fourth-call short-circuit before client construction.

## Solution Assessment

Yes. A per-URL cache in `src/gateway/probe.ts` is the narrowest maintainable fix because `GatewayClient` reconnect backoff is per instance, while the reported storm comes from fresh short-lived probe clients.
