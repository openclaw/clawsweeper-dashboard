## Summary

Keep open. Current main and the latest stable tag still create a fresh `GatewayClient` for each `probeGateway()` call with no per-URL `device-required` suppression; the exact cache fix only exists in closed, unmerged https://github.com/openclaw/openclaw/pull/63446, while open https://github.com/openclaw/openclaw/pull/68280 covers adjacent local probe auth fail-fast behavior rather than repeated rejection dampening.

## What This Changes

The issue requests dampening/backoff for repeated CLI gateway WebSocket probes that hit `device-required` on unpaired local gateways.

## Best Possible Solution

Add a narrow `probeGateway()`-boundary cache that records recent `DEVICE_IDENTITY_REQUIRED` closes per gateway URL, short-circuits after a small threshold within a TTL, clears on successful connection, and leaves an unauthenticated HTTP health endpoint for separate product/security discussion.

## Reproduction Assessment

Yes, at source level. Mock repeated `probeGateway()` calls that close with code 1008 and reason `device identity required`; current main will construct a new `GatewayClient` for each call because no cache or fourth-call short-circuit exists before client construction.

## Solution Assessment

Yes. A per-URL cache in `src/gateway/probe.ts` is the narrowest maintainable fix because `GatewayClient` reconnect backoff is per instance, while the reported storm comes from fresh short-lived probe clients.
