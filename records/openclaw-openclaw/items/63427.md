## Summary

Keep open. Current main still creates a fresh `GatewayClient` for each `probeGateway()` call and lacks per-URL `device-required` suppression; the exact cache fix remains only in closed, unmerged PR #63446, while open PR #68280 addresses adjacent missing local probe auth rather than repeated rejection dampening.

## What This Changes

Add backoff or short-circuit suppression for repeated CLI gateway WebSocket probes rejected as `device-required`.

## Best Possible Solution

Land a narrow `probeGateway()`-boundary cache that records recent `DEVICE_IDENTITY_REQUIRED` closes per gateway URL, short-circuits after a small threshold within a TTL, clears on successful connection, and leaves the unauthenticated HTTP health endpoint as a separate product/security discussion.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock `GatewayClient` so repeated `probeGateway()` calls close with code 1008 and reason `device identity required`; current main constructs a new mock client on every call and has no fourth-call short-circuit path.

## Solution Assessment

Yes. A per-URL cache in `src/gateway/probe.ts` is the narrowest maintainable fix because `GatewayClient` reconnect backoff is per instance, while the reporter's storm comes from fresh short-lived probe clients; adding `/health` would broaden the surface area.
