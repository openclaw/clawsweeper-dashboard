## What This Changes

This PR adds an optional `agentId` field to plugin command handler context, populates it from the existing bound-agent resolver in `executePluginCommand`, and adds focused command tests.

## Best Possible Solution

If maintainers accept `ctx.agentId`, land this resolver-backed plumbing with command-author docs aligned; otherwise decline the public context expansion and keep plugins on existing session and runtime helper surfaces.

## Reproduction Assessment

Not applicable; this is an additive Plugin SDK feature rather than a bug with an established current contract. Source inspection confirms current main resolves command agent authority for `runtimeContext` but does not expose top-level `ctx.agentId`.

## Solution Assessment

Yes for the code shape if maintainers accept the API: it reuses `resolveBoundAgentIdForSession` and keeps the exposed field aligned with runtime authority. The safer alternative is to decline the public field and keep authors on existing session/runtime APIs if maintainers do not want this contract.
