## Summary

Keep open: the patch is a narrow, correct-looking Plugin SDK expansion with sufficient live proof, but `ctx.agentId` becomes public command-context API and still needs maintainer acceptance plus a short command-author docs note before merge.

## What This Changes

The PR adds an optional resolved `agentId` field to `PluginCommandContext`, populates it in `executePluginCommand`, and adds command-path tests for agent-scoped and host-provided resolution.

## Best Possible Solution

If maintainers accept `ctx.agentId`, land the resolver-backed plumbing with command-author docs aligned; otherwise decline the public context expansion and keep plugin authors on existing session and runtime helper surfaces.

## Reproduction Assessment

Not applicable; this is an additive Plugin SDK feature rather than a bug with an established current behavior contract. Source inspection confirms current main does not expose top-level `ctx.agentId`, and the PR body provides after-fix live output for the proposed behavior.

## Solution Assessment

Yes for the code shape if maintainers accept the API: it reuses `resolveBoundAgentIdForSession` and keeps the exposed field aligned with runtime authority. The safer alternative is to decline or reshape the public SDK contract before release if maintainers do not want this field.
