## Summary

Keep open: current main and the latest shipped tag still start the overview secondary `loadUsage` call before the stale-overview guard runs, and `loadUsage` sends `usage.cost`. The fix should be narrow because the active overview dashboard currently uses usage data for its cost card, so blocking all overview usage loads would regress intended UI data.

## What This Changes

The issue requests that Control UI overview refreshes avoid firing `usage.cost` after the active overview refresh has become stale.

## Best Possible Solution

Land a focused Control UI fix that rechecks `isCurrentOverviewRefresh()` before starting secondary overview loads, preserves usage loading while the overview tab is still active, and adds regression coverage for navigating away before the primary overview load resolves.

## Reproduction Assessment

Yes from source: make the primary overview loaders pending, switch `host.tab` away from `overview`, then resolve the primary load and current code will still start `loadUsage`. I did not run a live Control UI session or tests in this read-only review.

## Solution Assessment

Yes, with a narrower scope than the log wording implies: guard the secondary overview load after the primary phase becomes stale, while preserving `loadUsage` on the active overview and usage tabs because overview cards consume usage data.
