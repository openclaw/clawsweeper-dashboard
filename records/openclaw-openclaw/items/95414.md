## Summary

Keep open: the linked cron bug is real, but this PR fixes it at the shared JSON parser layer, which would silently rewrite arbitrary provider and plugin tool-call payload keys before validation.

## What This Changes

The PR adds recursive trailing-whitespace key trimming to `parseJsonWithRepair()` and `parseStreamingJson()` and adds parser tests for whitespace-padded keys.

## Best Possible Solution

Keep strict shared JSON parsing semantics and add the observed whitespace-key recovery inside the cron tool canonicalization boundary, with cron regression coverage and redacted live or terminal proof for the reported local-model flow.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current cron canonicalization does not recover `schedule `, `sessionTarget `, `payload `, or `enabled `, while the gateway schema requires the exact keys and rejects extras. I did not run a live llama.cpp/qwen35b scenario in this read-only review.

## Solution Assessment

No. The PR is a plausible mitigation, but the best fix is cron-owned key recovery because the changed parser is shared across providers, transports, and plugin SDK consumers.
