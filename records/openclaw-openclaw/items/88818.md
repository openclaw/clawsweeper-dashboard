## Summary

Keep open. The reporter found the correct trusted-private-network workaround, so the default block is expected, but current main and the latest release still surface CDP SSRF blocks as the generic `browser endpoint blocked by policy`, leaving the requested diagnostic improvement unresolved.

## What This Changes

The issue asks OpenClaw to make remote Browserless/private CDP endpoint policy failures explain the SSRF/private-network cause more clearly.

## Best Possible Solution

Keep the strict SSRF default, but add a sanitized browser-CDP endpoint diagnostic that names the SSRF/private-network policy cause and points operators to `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork` or narrow host allowlists without leaking secrets.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: default browser SSRF policy is strict, remote CDP endpoint checks rethrow SSRF failures as `BrowserCdpEndpointBlockedError`, and the current message remains generic. I did not rerun the full Browserless Docker setup, but the reporter supplied live logs and confirmed the workaround.

## Solution Assessment

No, closing is not the best solution because the operational workaround does not fix the diagnostic gap. The best fix is a small, security-reviewed message/test/docs change that keeps details redacted while making the SSRF cause actionable.
