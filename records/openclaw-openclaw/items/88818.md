## Summary

Keep open. The reporter found the correct trusted-private-network workaround, but current main and v2026.6.6 still map the CDP SSRF/private-network block to the generic `browser endpoint blocked by policy` message, so the requested diagnostic improvement remains unresolved and security-sensitive.

## What This Changes

The issue asks for remote CDP Browserless/private-network endpoint blocks to produce an actionable browser SSRF-policy diagnostic instead of the current generic policy-block error.

## Best Possible Solution

Keep the strict SSRF default and add a sanitized browser-CDP endpoint diagnostic that names the private-network SSRF policy cause and points to `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork` or narrow host allowlists without leaking secrets.

## Reproduction Assessment

Yes for the remaining diagnostic gap. Source inspection shows strict default browser SSRF policy, CDP endpoint checks rethrow blocked SSRF failures as `BrowserCdpEndpointBlockedError`, and current tests still assert the generic message; I did not rerun Browserless Docker live.

## Solution Assessment

Yes, a narrow sanitized diagnostic is the best direction. Relaxing SSRF defaults or exposing raw resolved endpoint details would be the wrong fix; maintainers should choose the safe disclosure boundary first.
