Keep open. ## What This Changes

The issue asks OpenClaw to make remote-CDP browser policy failures, especially Browserless private-network SSRF blocks, actionable instead of surfacing the generic endpoint-policy error.

## Best Possible Solution

Keep strict SSRF defaults, then add a sanitized browser-CDP endpoint diagnostic that names the SSRF/private-network policy cause, points to safe operator choices, and proves tokens and internal endpoints stay redacted.

Yes. Source-level reproduction is high confidence: strict/default browser SSRF policy reaches CDP endpoint guards, and current main still maps blocked endpoint policy failures to the generic BrowserCdpEndpointBlockedError message.

## Solution Assessment

No, closing on the workaround is not the best solution because the diagnostic remains unclear. The narrow fix is security-reviewed, redacted error text plus focused tests and docs, not a broader SSRF policy change.
