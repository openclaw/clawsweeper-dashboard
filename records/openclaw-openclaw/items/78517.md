## Summary

Keep open: current main still bypasses before-tool-call hooks for inline skill tool dispatch, and this PR targets the right seam, but the patch has a target-session context bug, lacks a required changelog entry, and only provides mock/unit-test proof. The protected `maintainer` label also requires maintainer handling.

## What This Changes

The PR adds a before-tool-call hook check to inline skill command tool dispatch, uses adjusted hook parameters for execution, and adds focused inline dispatch regression coverage.

## Best Possible Solution

Thread the same target session context used by the rest of inline actions into the hook call, add the changelog entry, and require after-fix live or terminal/log proof before maintainer review and merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: current main's inline skill tool dispatch calls `tool.execute` directly, while the existing hook path is `runBeforeToolCallHook`; I did not establish a live-channel failing run.

## Solution Assessment

Is this the best way to solve the issue? No as written: adding the hook is the right seam, but the hook context should use the target session entry and the PR still needs changelog and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use the target session entry for hook context:** `src/auto-reply/reply/get-reply-inline-actions.ts:319`
  - body: This hook context is built before the path computes the target entry and reads `sessionEntry` directly. In callers that pass a wrapper entry plus `sessionStore[sessionKey]`, the before_tool_call hook and loop detection receive the wrong ephemeral `sessionId`, unlike the existing inline status, command, and abort-cutoff paths. Compute the target entry before tool dispatch and use it here.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/auto-reply/reply/get-reply-inline-actions.ts:314`
  - body: This changes user-visible security behavior for inline skill tool commands, but the PR only touches source and tests. OpenClaw policy requires user-facing fixes and security hardening to add a `CHANGELOG.md` entry under the active release section.
  - confidence: 0.9
