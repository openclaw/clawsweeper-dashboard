## Summary

Keep open: current main and v2026.6.10 still have no `trustedWebhooks` or webhook-to-user mapping contract, and Discord preflight still drops non-PluralKit bot/webhook messages when `allowBots` is off. The request is source-reproducible and useful, but it is a new security-sensitive Discord config surface that needs maintainer product/security direction before implementation.

## What This Changes

The issue requests a guild-level Discord `trustedWebhooks` allowlist so selected Discord webhook messages can trigger agent responses without enabling all bot messages.

## Best Possible Solution

Have maintainers define the Discord trusted-webhook security model first; if accepted, add a schema-backed Discord allowlist with explicit mapping rules, docs, and focused preflight tests without broadening the default bot-message policy.

## Reproduction Assessment

Yes at source level: current `preflightDiscordMessage` resolves a Discord webhook id and then drops non-PluralKit bot-authored messages when `allowBots` is off. I did not run a live Discord webhook scenario in this read-only review.

## Solution Assessment

Unclear. The proposed trusted webhook allowlist is plausible, but `mapToUser` crosses authorization and session boundaries, so maintainers should approve the trust semantics before implementation.
