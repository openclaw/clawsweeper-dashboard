## Label Justifications

- `P3`: This is a useful Discord feature request with limited immediate blast radius, but it needs product/security design before implementation.
- `impact:security`: The request would decide which external Discord webhook payloads may enter agent sessions and optionally map to user authority.
- `impact:message-loss`: Discord webhook-authored messages are currently dropped unless broader bot-message settings allow them.

## Summary

Keep open: current main and latest v2026.6.6 still do not provide a Discord trusted-webhook allowlist or webhook-to-user mapping, and the existing `allowBots` modes plus the generic Webhooks plugin are adjacent but not equivalent to this in-channel Discord trust boundary.

## What This Changes

Request to add a Discord guild-level trusted webhook allowlist that can allow selected webhook-authored messages and optionally map them to a Discord user identity.

## Best Possible Solution

Define the Discord-plugin trust model first; if accepted, implement a schema-backed webhook allowlist with explicit user-mapping authorization/session rules, focused tests, and docs without broadening the default bot-message policy.

## Reproduction Assessment

Yes at source level: current Discord preflight resolves a webhook ID, then drops non-PluralKit bot-authored messages when `allowBots` is off. I did not run a live Discord webhook scenario.

## Solution Assessment

Unclear. The requested allowlist is plausible, but `mapToUser` crosses authorization and session boundaries, so maintainers need to approve the trust semantics before code is queued.
