## Summary

Keep open. Current main still does not provide a Discord guild-level trusted webhook allowlist or mapped-user ingress path; webhook-authored messages remain governed by broad `allowBots` modes or PluralKit, and the config contract has no `trustedWebhooks` or `mapToUser` fields. The prior ClawSweeper keep-open comment remains consistent with current code, and related webhook mention work only covers `allowBots: "mentions"`, not this narrower trust model.

## What This Changes

Request to add Discord guild-level `trustedWebhooks` configuration so selected Discord webhooks can trigger agent responses, optionally mapped to a configured user identity.

## Best Possible Solution

If maintainers accept the trust model, implement this inside the Discord plugin with typed and zod-validated guild `trustedWebhooks` config, a webhook allowlist check before the default bot-message drop, mapped-user validation against configured guild users, clear logging/audit metadata, docs/config UI metadata, and focused preflight tests for allowed, denied, mention-only, mapped-user, and bound-thread echo cases.

## Reproduction Assessment

Yes. A Discord guild message with `author.bot=true`, a `webhook_id`, default `allowBots` behavior, and no PluralKit identity follows the current preflight path to the bot-message drop; the proposed config key would also fail the strict Discord guild schema.

## Solution Assessment

Unclear. A guild-level webhook allowlist is a narrow likely boundary, but `mapToUser` changes sender identity semantics, so maintainer and Discord owner review should settle validation and audit requirements before implementation.
