## Summary

Keep open: current main and latest v2026.6.6 still drop non-PluralKit Discord webhook messages when `allowBots` is off, and there is no `trustedWebhooks` or webhook-to-user mapping contract in the Discord config schema.

## What This Changes

Request a Discord `trustedWebhooks` allowlist that lets selected Discord webhook messages trigger agent responses without enabling all bot-authored messages.

## Best Possible Solution

Have maintainers define the Discord trusted-webhook security model first; if accepted, add a schema-backed Discord-plugin allowlist with explicit mapping rules, docs, and focused preflight tests without broadening the default bot-message policy.

## Reproduction Assessment

Yes at source level: current Discord preflight resolves the webhook id, then drops non-PluralKit bot-authored messages when `allowBots` is off. I did not run a live Discord webhook scenario because this was a read-only review.

## Solution Assessment

Unclear. The trusted webhook allowlist is plausible, but `mapToUser` crosses authorization and session boundaries, so maintainers should approve the trust semantics before implementation.
