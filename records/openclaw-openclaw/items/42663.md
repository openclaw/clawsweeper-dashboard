## Summary

Keep open: current main still drops bot-authored Discord webhook messages unless global bot handling permits them, and the requested allowlist plus `mapToUser` policy is a new security-sensitive feature/config surface needing maintainer decision.

## What This Changes

Add a guild-level Discord `trustedWebhooks` allowlist, optionally mapping webhook messages to a configured user, so selected webhook-origin messages can trigger agent responses without enabling all bot messages.

## Best Possible Solution

If accepted, implement this inside the Discord plugin with typed config/schema/docs/tests, keep bot-authored messages ignored by default, and constrain webhook trust plus user mapping to existing access policy and loop guards.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection shows the current Discord preflight drops non-PluralKit bot-authored webhook messages unless `allowBots` permits them, and no `trustedWebhooks` config exists.

## Solution Assessment

Unclear until maintainers choose the identity model. A Discord-plugin allowlist is plausible, but `mapToUser` and bot-loop/echo interactions need explicit guardrails first.
