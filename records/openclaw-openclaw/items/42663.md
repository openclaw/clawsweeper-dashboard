## Label Justifications

- `P3`: This is a useful but speculative Discord feature that needs product/security design rather than an urgent regression fix.
- `impact:security`: The request changes which external Discord webhook payloads may enter agent sessions and may map them to user authority.
- `impact:message-loss`: Webhook-authored Discord messages are currently dropped unless broader bot-message settings allow them.

## Summary

Keep open: current main and latest v2026.6.6 still lack the requested trusted Discord webhook allowlist/user mapping, while adjacent `allowBots="mentions"`, bot-loop protection, and the generic webhooks plugin do not provide the same Discord-channel trust boundary.

## What This Changes

Add a Discord guild-level `trustedWebhooks` allowlist with optional `mapToUser` so selected webhook-authored Discord messages can trigger agent responses without enabling all bot traffic.

## Best Possible Solution

Define the Discord-plugin trust semantics first; if accepted, add a schema-backed allowlist with explicit map-to-user authorization/session rules, tests, docs, and no broad bot-message default change.

## Reproduction Assessment

Yes, at source level: Discord preflight resolves a webhook ID but drops bot-authored messages when `allowBots` is off and the sender is not PluralKit. I did not run a live Discord webhook scenario.

## Solution Assessment

Unclear. The requested shape is plausible, but `mapToUser` crosses authorization and session boundaries, so the best fix needs maintainer-approved Discord-plugin trust semantics first.
