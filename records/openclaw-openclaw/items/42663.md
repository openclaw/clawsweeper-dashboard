## Summary

Keep open: current main still lacks a Discord trusted-webhook allowlist, and the request adds a new security-sensitive config and identity-mapping policy rather than fixing a broken existing contract.

## What This Changes

The issue requests a Discord guild-level `trustedWebhooks` allowlist that can optionally map webhook-authored messages to a configured user.

## Best Possible Solution

If accepted, implement this inside the Discord plugin with schema/types/docs/tests, preserve the default bot-message drop, and make webhook trust plus identity mapping explicit, auditable, and constrained by existing access policy.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high confidence that current main drops non-PluralKit bot-authored Discord webhook messages unless global `allowBots` permits them, and no `trustedWebhooks` config exists.

## Solution Assessment

Unclear until maintainer product/security review settles the identity model. A Discord plugin-local allowlist is plausible, but `mapToUser` and loop/echo interactions need approval before this is a safe implementation task.
