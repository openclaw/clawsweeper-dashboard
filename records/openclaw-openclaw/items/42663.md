## Summary

Keep open: current main still has no Discord trusted-webhook allowlist or webhook-to-user mapping, and the existing Discord bot gate only supports global bot allowance plus PluralKit-style exemptions. The request is useful but security-sensitive because it would authorize webhook-origin messages and may map them onto a user identity, so it needs maintainer product/security review before implementation.

## What This Changes

Request adding a Discord guild-level `trustedWebhooks` config that allowlists selected webhook IDs and optionally maps webhook messages to an existing user identity.

## Best Possible Solution

If maintainers accept the feature, implement it inside the Discord plugin with typed config/schema/docs, default-deny scoped trust, mapped-user validation against existing access policy, and focused preflight tests preserving current bot safeguards.

## Reproduction Assessment

No strict bug reproduction applies because this is a new trusted inbound Discord feature. Source inspection gives a high-confidence behavior path: webhook IDs are resolved, but non-PluralKit bot-authored messages are still governed only by `allowBots` and existing bot/mention gates.

## Solution Assessment

Unclear until maintainers choose the identity and trust model. A Discord-plugin allowlist is plausible, but `mapToUser` needs explicit product and security guardrails before implementation.
