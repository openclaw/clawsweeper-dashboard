## Summary

OpenClaw already supports the central workflow without a separate `trustedWebhooks` surface: enable Discord bot-authored input, then restrict guild or channel `users` to the webhook ID and any human IDs that should remain authorized. Discord identifies webhook messages with the webhook ID as `author.id`, and OpenClaw applies its stable sender-ID allowlist after bot admission. The proposed `mapToUser` remainder should not be added because it would conflate an automation with a human authorization and session identity.

## What This Changes

The issue requests a guild-level Discord webhook allowlist, optionally mapping trusted webhook messages to a configured user identity so they can trigger agent responses without enabling all bot messages.

## Best Possible Solution

Keep one generic Discord trust model: use `allowBots` for bot admission, guild or channel stable-ID `users` allowlists for authorization, and bot-loop protection; document a webhook example without mapping webhook identities onto human users.

## Reproduction Assessment

Yes at source level: webhook messages follow the bot-authored drop path when `allowBots` is off, and Discord's documented webhook author ID makes the existing narrow allowlist configuration deterministic. A live Discord run was not needed to establish the code and platform contract.

## Solution Assessment

No. A separate `trustedWebhooks` option would duplicate the existing stable sender-ID allowlist, while `mapToUser` would introduce an unsafe identity-delegation contract; documenting the existing composition is narrower and more maintainable.
