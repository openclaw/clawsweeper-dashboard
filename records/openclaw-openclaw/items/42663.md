## Summary

Keep open: current main and v2026.6.10 still lack a Discord `trustedWebhooks` or webhook-to-user mapping contract, and the current preflight still drops non-PluralKit bot/webhook messages when `allowBots` is off. This remains a useful but security-sensitive feature request that needs maintainer product/security direction before implementation.

## What This Changes

The issue requests a Discord guild-level trusted webhook allowlist that can let selected Discord webhook messages trigger agent responses without enabling all bot messages.

## Best Possible Solution

Have maintainers define the Discord trusted-webhook security model first; if accepted, implement a schema-backed allowlist in the Discord plugin with explicit mapping rules, docs, and focused preflight tests without broadening the default bot-message policy.

## Reproduction Assessment

Yes at source level: current `preflightDiscordMessage` returns `null` for bot-authored non-PluralKit messages when `allowBots` resolves to off, and the local Discord message model carries webhook IDs. I did not run a live Discord webhook scenario in this read-only review.

## Solution Assessment

Unclear. A trusted webhook allowlist is plausible, but the `mapToUser` authority and session semantics need maintainer product/security approval before implementation.
