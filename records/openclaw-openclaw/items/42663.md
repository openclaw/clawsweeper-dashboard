## Summary

Keep open: current main and the latest release still lack a Discord trusted-webhook allowlist or webhook-to-user mapping, and the existing `allowBots` paths do not cover the requested unmentioned allowlisted webhook automation. The remaining work is a security-sensitive Discord plugin feature that needs maintainer product and trust-model review.

## What This Changes

Add a guild-level Discord `trustedWebhooks` config surface so selected webhook-authored Discord messages can trigger agent responses without enabling all bot-authored messages.

## Best Possible Solution

If maintainers accept the feature, implement it inside the Discord plugin with typed config/schema/docs, default-deny per-account/guild/channel trust, mapped-user validation against existing access policy, and focused preflight regression coverage.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this requests a new trusted inbound Discord feature. Source inspection confirms current behavior: webhook IDs are resolved, but non-PluralKit bot-authored messages are governed only by `allowBots` and mention policy.

## Solution Assessment

Unclear until maintainers choose the identity and trust model. A Discord-plugin allowlist is plausible, but `mapToUser` needs explicit product and security guardrails before implementation.
