## Summary

Keep open: current main still has no trusted Discord webhook allowlist or webhook-to-user mapping, and the prior discussion correctly frames this as a security-sensitive Discord-plugin feature needing maintainer product/security review.

## What This Changes

Request adds a guild-level Discord `trustedWebhooks` allowlist with optional webhook-to-user mapping so selected webhook-authored messages can trigger agent responses without enabling all bots.

## Best Possible Solution

If maintainers accept the feature, implement it inside the Discord plugin with typed config/schema/docs, default-deny scoped trust, mapped-user validation against existing access policy, and focused preflight tests for PluralKit, bound-thread webhook echoes, `allowBots`, and bot-loop protection.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this requests a new trusted inbound Discord feature. Source inspection does confirm the current behavior: webhook IDs are resolved, but non-PluralKit bot-authored webhook messages are governed only by `allowBots` and mention policy.

## Solution Assessment

Unclear until maintainers choose the identity and trust model. A Discord-plugin allowlist is plausible, but `mapToUser` needs explicit product and security guardrails before implementation.
