## Summary

Keep open: current main still lacks a trusted Discord webhook allowlist or user-mapping path, and the requested behavior adds a security-sensitive inbound identity/config surface that needs maintainer product and security review before implementation.

## What This Changes

The issue requests a guild-scoped Discord `trustedWebhooks` configuration option that lets selected webhook-origin messages trigger agents, optionally mapped to a configured user.

## Best Possible Solution

If maintainers accept the feature, implement it in the Discord plugin with typed config, strict schema, docs, and focused preflight tests that keep trust default-deny and validate mapped users against existing access policy.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection shows webhook IDs are available in preflight, but webhook-origin bot messages are still governed only by existing bot policy and PluralKit handling.

## Solution Assessment

Unclear until maintainers choose the identity model. A Discord-plugin allowlist is plausible, but `mapToUser` and its interaction with access checks and loop guards need explicit product and security guardrails first.
