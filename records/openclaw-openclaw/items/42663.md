## Summary

Keep open: current main still has only global Discord `allowBots` handling plus the PluralKit exception, while this request adds a new per-webhook trust and identity-policy surface that needs maintainer product/security approval.

## What This Changes

Add guild-scoped Discord `trustedWebhooks` config so allowlisted webhook IDs can trigger agent replies without enabling all bot-authored messages.

## Best Possible Solution

If accepted, implement the allowlist entirely inside the Discord plugin with schema/types/docs/tests, preserve the default bot-message drop, and make any user mapping explicit, auditable, and constrained by guild access policy.

## Reproduction Assessment

Not applicable as a strict bug reproduction. Source inspection gives a high-confidence current-main path: non-PluralKit webhook messages are bot-authored and are dropped unless global `allowBots` permits them, and no `trustedWebhooks` config exists.

## Solution Assessment

Unclear until maintainer product/security review settles the identity model. A Discord plugin-local allowlist is plausible, but `mapToUser` needs explicit semantics before this is a safe implementation task.
