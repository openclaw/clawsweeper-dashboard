## Summary

Keep open: current `main` and v2026.6.8 still have no `trustedWebhooks` or webhook-to-user mapping contract, and Discord preflight still drops non-PluralKit bot/webhook messages when `allowBots` is off. This is a useful but security-sensitive feature request that needs maintainer product/security direction before implementation.

## What This Changes

Requested Discord guild-level `trustedWebhooks` configuration would allow selected webhook messages to bypass the default bot-message drop and optionally map webhook input to a user.

## Best Possible Solution

Have maintainers define the Discord trusted-webhook security model first; if accepted, add a schema-backed Discord allowlist with explicit mapping rules, docs, and focused preflight tests without broadening the default bot-message policy.

## Reproduction Assessment

Yes at source level: current preflight resolves `webhookId` but still drops non-PluralKit bot-authored messages when `allowBots` is off. I did not run a live Discord webhook scenario because this review is read-only.

## Solution Assessment

Unclear. The proposed trusted webhook allowlist is plausible, but `mapToUser` crosses authorization and session boundaries, so maintainers should approve the trust semantics before implementation.
