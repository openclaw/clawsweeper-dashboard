## Label Justifications

- `P3`: This is a useful but speculative Discord feature that needs product/security design rather than an urgent regression fix.
- `impact:security`: The request changes who can inject Discord webhook content into agent sessions and may map that input to user authority.
- `impact:message-loss`: Webhook-authored Discord messages are currently dropped unless broader bot-message settings allow them.

## Summary

Keep open: current main still lacks a trusted Discord webhook allowlist or map-to-user behavior, and the existing `allowBots` modes plus the generic webhooks plugin are related but not equivalent to the requested Discord-channel trust boundary.

## What This Changes

The issue requests a Discord guild-level `trustedWebhooks` allowlist so selected webhook-authored messages can trigger agent responses without enabling all bot messages.

## Best Possible Solution

Keep this open for a Discord-plugin product and security decision; if accepted, add a narrow schema-backed trusted-webhook design with explicit auth/session semantics, focused preflight tests, and docs while keeping global bot-message defaults conservative.

## Reproduction Assessment

Yes at source level for the current behavior: Discord preflight resolves the webhook ID but drops bot-authored messages when `allowBots` is off and the sender is not PluralKit. I did not run a live Discord webhook scenario.

## Solution Assessment

Unclear. The proposed shape is plausible, but `mapToUser` crosses auth and session boundaries, so the best fix needs maintainer-approved Discord-plugin trust semantics before implementation.
