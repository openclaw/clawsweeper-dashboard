## Summary

Keep open: current main still lacks a Discord `trustedWebhooks` trust path, and the requested `mapToUser` behavior is a new config and security-sensitive identity surface that needs maintainer product/security review.

## What This Changes

Request to add a guild-level Discord `trustedWebhooks` allowlist with optional `mapToUser` so selected webhook-origin bot messages can trigger agent responses without enabling all bot messages.

## Best Possible Solution

If accepted, implement this inside the Discord plugin with typed config, schema, docs, and tests; keep webhook trust default-deny, scope it to guild/channel/account context, verify user mapping against existing access policy, and preserve loop/echo safeguards.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection on current main shows webhook IDs are resolved, but webhook-origin bot messages are still governed only by `allowBots` and PluralKit handling, with no `trustedWebhooks` config.

## Solution Assessment

Unclear until maintainers choose the identity model. A Discord-plugin allowlist is plausible, but `mapToUser` and its interaction with loop and echo guards need explicit product/security guardrails first.
