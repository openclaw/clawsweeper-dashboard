## Summary

Keep open: current main still has no Discord trusted webhook allowlist or user-mapping path, and the request adds a new security-sensitive identity/config surface that needs maintainer product and security review before implementation.

## What This Changes

Request adds guild-level Discord `trustedWebhooks` configuration so selected webhook-origin messages can trigger agent responses, optionally mapped to an existing Discord user.

## Best Possible Solution

If accepted, implement this inside the Discord plugin with typed config, strict schema, docs, and preflight tests; keep trust default-deny, scope it to guild/channel/account context, and validate any mapped user against existing access policy.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection shows webhook IDs are available in preflight, but webhook-origin bot messages are still governed only by `allowBots` and PluralKit handling.

## Solution Assessment

Unclear until maintainers choose the identity model. A Discord-plugin allowlist is plausible, but `mapToUser` and its interaction with access checks and loop guards need explicit product/security guardrails first.
