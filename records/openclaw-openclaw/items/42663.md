## Summary

Keep open: current main still has no trusted Discord webhook allowlist or webhook-to-user mapping, and the requested behavior would add a security-sensitive inbound identity/config surface that needs maintainer product and security review before implementation.

## What This Changes

The issue requests a Discord guild-level `trustedWebhooks` allowlist with optional `mapToUser` support so selected Discord webhook messages can trigger agent responses without enabling all bot-authored messages.

## Best Possible Solution

If maintainers accept the feature, implement it in the Discord plugin with typed config/schema/docs, default-deny scoped trust, mapped-user validation against existing access policy, and focused preflight tests covering PluralKit, bound-thread webhook echoes, `allowBots`, and bot-loop protection.

Not applicable as a strict bug reproduction because this is a new feature request. Source inspection confirms the current behavior: webhook ids are resolved, but non-PluralKit bot-authored webhook messages are still governed only by `allowBots` and mention policy.

## Solution Assessment

Unclear until maintainers choose the identity and trust model. A Discord-plugin allowlist is plausible, but `mapToUser` needs explicit product and security guardrails before implementation.
