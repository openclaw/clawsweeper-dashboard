## Summary

Keep open: current main still lacks per-webhook Discord trust, and the request adds a new config and identity-policy surface that needs maintainer product/security approval before implementation.

## What This Changes

Request adding a Discord `trustedWebhooks` guild config so selected webhook-authored messages can trigger responses, optionally mapped to a user identity.

## Best Possible Solution

If accepted, implement the allowlist inside the Discord plugin with schema/types/docs/tests, preserve the default bot-message drop, and make identity mapping explicit, auditable, and constrained by existing access policy.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high confidence that current main drops non-PluralKit bot-authored Discord webhook messages unless global `allowBots` permits them, and no `trustedWebhooks` config exists.

## Solution Assessment

Unclear until maintainer product/security review settles the identity model. A Discord plugin-local allowlist is plausible, but `mapToUser` and loop-suppression semantics need approval before this is a safe implementation task.
