## Summary

Keep open: current `main` and latest release `v2026.6.10` still have no Discord `trustedWebhooks` or webhook-to-user mapping contract, and Discord preflight still drops non-PluralKit webhook/bot messages when `allowBots` is off. The request is useful but security-sensitive because it would define a new trusted inbound Discord webhook authority surface.

## What This Changes

The issue requests a Discord `trustedWebhooks` allowlist, optionally mapping a webhook to a user, so specific webhook-authored messages can trigger agent replies without enabling all bot-authored messages.

## Best Possible Solution

Have maintainers define the Discord trusted-webhook security model first; if accepted, add a schema-backed allowlist with explicit mapping rules, docs, and focused preflight tests without broadening the default bot-message policy.

## Reproduction Assessment

Yes, at source level: current preflight resolves `webhookId` but still drops non-PluralKit bot-authored messages when `allowBots` is off. I did not run a live Discord webhook scenario because this review is read-only.

## Solution Assessment

Unclear. The proposed trusted webhook allowlist is plausible, but `mapToUser` crosses authorization and session boundaries, so maintainers should approve the trust semantics before implementation.
