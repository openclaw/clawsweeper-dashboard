## Summary

Keep open: this maintainer-labeled, MEMBER-authored PR intentionally removes an existing system-event trust boundary, and the current head still drops the non-owner downgrade for externally sourced queued events. The security concern is concrete because current main uses that signal to force non-owner runs before owner-only tools are applied.

## What This Changes

The branch removes `trusted` from the system-event contract and enqueue call sites, renders drained event context with a single `System:` prefix, deletes untrusted-event owner-downgrade logic, and updates tests plus changelog.

## Best Possible Solution

Keep a machine-readable external/untrusted authority signal through enqueue, drain, heartbeat, and reply-agent setup, and only simplify the visible prompt prefix after a security-reviewed replacement preserves non-owner execution.

## Reproduction Assessment

Yes for the review findings from source: current main tests expect untrusted hook:wake and interval queued events to set `ForceSenderIsOwnerFalse`, and the PR changes those expectations while also deleting the reply-run downgrade. I did not run branch tests in this read-only review.

## Solution Assessment

No. Removing display metadata and authorization metadata together is not the narrowest maintainable solution; the safer path is to decouple any visible `System:` prefix simplification from a preserved non-owner execution guard.
