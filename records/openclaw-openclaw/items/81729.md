## Summary

Keep open. The latest head appears to address the broad heartbeat downgrade concern by forcing system-event provider runs non-owner, but it still removes the direct-reply downgrade, the AI-visible untrusted event marker, and the security scanner that currently enforces explicit trust metadata; the protected `security` and `maintainer` labels also require human handling.

## What This Changes

The PR removes queued system-event `trusted` storage and call-site options, formats all drained event context as `System:`, deletes the OpenGrep rule for explicit trust metadata, and updates channel/core tests plus changelog.

## Best Possible Solution

Keep a machine-readable untrusted/external event authority signal through enqueue, drain, heartbeat, and reply-run setup, and only simplify the visible prefix after replacement enforcement and regression tests cover direct replies plus system-event runs.

## Reproduction Assessment

Yes for the review findings from source. Current main stores `trusted`, formats untrusted events distinctly, downgrades direct replies that drain those events, and tests that behavior; the supplied PR diff removes those guards and updates the tests to the new behavior.

## Solution Assessment

No. The latest head narrows one prior concern by forcing heartbeat/system-event provider runs non-owner, but removing direct drained-event trust handling and the scanner together is not the safest maintainable path.
