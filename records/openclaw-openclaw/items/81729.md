## Summary

Keep open. This MEMBER-authored PR has protected `security` and `maintainer` labels and still changes the system-event trust boundary: the latest head now downgrades runs that drain queued events, but it still collapses external queued event context to `System:` and deletes the GHSA-backed scanner without a visible replacement.

## What This Changes

The PR removes queued system-event trust metadata, collapses drained runtime event context to a single `System:` prefix, drops related OpenGrep enforcement, and updates affected tests/changelog across core, gateway, agents, and plugins.

## Best Possible Solution

Keep an explicit machine-readable or AI-visible external-event authority signal through enqueue, drain, heartbeat, and reply-run setup, then simplify user-facing wording only after replacement enforcement and regression coverage are in place.

## Reproduction Assessment

Yes for the review findings from source inspection. Current main stores trust, formats `trusted: false` as `System (untrusted)`, downgrades owner runs from that marker, and enforces the GHSA rule; the PR diff removes the marker and rule while rendering all drained events as `System:`.

## Solution Assessment

No. The latest head fixes the owner-status part by downgrading runs with any drained queued event, but removing the visible trust marker and static enforcement is not the safest maintainable solution without a replacement boundary.
