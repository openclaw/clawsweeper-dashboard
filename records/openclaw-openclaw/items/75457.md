## Summary

Keep open: current main still has a concrete mismatch between the documented browser-direct tweakcn importer and the Control UI CSP served by the Gateway. The duplicate report #75535 is already closed back to this issue, so this should remain the canonical open item until maintainers choose the CSP boundary fix.

## What This Changes

Request to let the Control UI's browser-local tweakcn theme import fetch tweakcn registry payloads without being blocked by the Control UI Content-Security-Policy.

## Best Possible Solution

Align the security policy with the approved import architecture: either add a narrowly scoped `https://tweakcn.com` `connect-src` allowance for the existing browser-direct design, or implement a Gateway proxy if third-party browser fetches should remain disallowed. The chosen path should update CSP/import coverage and preserve the existing tweakcn schema, redirect, timeout, and payload-size guards.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A served Control UI page applies a CSP whose `connect-src` omits tweakcn, while the Appearance importer performs a browser fetch to `https://tweakcn.com/r/themes/<id>`, so the page policy blocks the documented import path.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the CSP boundary. Given the current docs and implementation specify direct browser fetching, the smallest maintainable fix is a host-specific tweakcn `connect-src` allowance; a Gateway proxy is the safer alternative if external browser fetches should stay forbidden.
