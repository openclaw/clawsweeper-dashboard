## Summary

Keep open: current main still serves a Control UI CSP that excludes tweakcn while the Appearance importer performs a direct browser fetch to the tweakcn registry, so the reported latest-release failure remains actionable.

## What This Changes

The issue asks for Control UI tweakcn theme imports to succeed by allowing or proxying the registry fetch currently blocked by the Control UI CSP.

## Best Possible Solution

Choose the approved egress boundary, then add either a host-specific tweakcn `connect-src` allowance or a Gateway import proxy with URL validation, redirect rejection, size limits, schema checks, regression tests, docs, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a deterministic reproduction path: the Control UI is served with a CSP that lacks `https://tweakcn.com`, while the Appearance importer calls browser `fetch()` for `https://tweakcn.com/r/themes/<id>`; the live endpoint itself returns CORS-friendly JSON.

## Solution Assessment

Unclear until maintainers choose the egress boundary. A host-specific CSP allowance is the narrowest fix for the current direct-fetch design; a Gateway proxy is safer if third-party browser egress should remain blocked.
