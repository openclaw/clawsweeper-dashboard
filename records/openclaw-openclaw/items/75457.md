## Summary

Keep open: current main still serves the Control UI with a `connect-src 'self' ws: wss:` CSP while the Appearance importer fetches the normalized tweakcn registry URL directly from browser code. The closed related report #75535 points to the same user-visible failure, but the remaining fix changes a Control UI CSP boundary, so maintainers should choose a narrow tweakcn host allowlist versus a Gateway proxy before implementation.

## What This Changes

The issue asks for Control UI tweakcn theme imports to work when the browser fetches `https://tweakcn.com/r/themes/<id>` under the current Control UI CSP.

## Best Possible Solution

Align the approved tweakcn import architecture with the Control UI security policy: either add a host-specific `connect-src https://tweakcn.com` allowance for the existing browser-direct fetch path, or route the import through a Gateway endpoint if third-party browser fetches should remain disallowed. The fix should update CSP/import tests and include a user-facing changelog entry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A served Control UI page applies a CSP that only permits same-origin and WebSocket connections, while the Appearance importer calls browser `fetch()` for `https://tweakcn.com/r/themes/<id>`, so the browser will block the request before CORS is relevant.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the CSP boundary. The narrowest code change is adding `https://tweakcn.com` to `connect-src`; the safer policy-preserving alternative is a Gateway proxy for the tweakcn registry fetch.
