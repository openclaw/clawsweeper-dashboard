## Summary

Keep open. Current main still has the reported mismatch: the Control UI CSP only allows same-origin and WebSocket connections, while the Appearance tweakcn importer still performs a browser fetch to `https://tweakcn.com/r/themes/<id>`. The related closed duplicate confirms this affected v2026.4.29 users, and no current-main source or shipped-release evidence shows the import path has been fixed.

## What This Changes

The issue asks OpenClaw to make the Control UI tweakcn theme import work when the served Content-Security-Policy blocks browser fetches to tweakcn.

## Best Possible Solution

Align the tweakcn import architecture with the Control UI security policy. If direct browser import is intended, add a narrow tweakcn `connect-src` allowance with regression coverage; if third-party browser egress should remain blocked, proxy the fetch through the Gateway while preserving the current validation and payload limits.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection gives a deterministic path: the served Control UI applies `connect-src 'self' ws: wss:`, then the Appearance importer calls browser `fetch()` for `https://tweakcn.com/r/themes/<id>`, which is outside that CSP allowlist.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the CSP boundary. The suggested tweakcn `connect-src` allowance is the narrowest match for the current direct-fetch design, while a Gateway proxy is safer if Control UI should not contact third-party hosts directly.
