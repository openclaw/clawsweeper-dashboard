## Summary

Keep open. Current main still has the same policy mismatch: the served Control UI CSP excludes `https://tweakcn.com`, while the Appearance importer calls browser `fetch()` against the tweakcn registry URL. The linked duplicate #75535 supports that this is affecting v2026.4.29 users, but the remaining fix crosses a CSP boundary and needs maintainer choice between a narrow host allowlist and a Gateway proxy.

## What This Changes

Fix the Control UI tweakcn theme importer so valid tweakcn share links no longer fail because the browser CSP `connect-src` directive blocks `https://tweakcn.com`.

## Best Possible Solution

Align the approved tweakcn import architecture with the Control UI security policy: either add a narrowly scoped tweakcn `connect-src` allowance for the existing browser-direct design, or implement a Gateway proxy if third-party browser fetches should remain disallowed. The chosen fix should add CSP/import regression coverage and preserve existing payload validation, redirect blocking, timeout, and size limits.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection gives a deterministic reproduction path: a served Control UI page gets a CSP that allows only same-origin plus WebSockets for `connect-src`, then the Appearance importer calls browser `fetch()` for `https://tweakcn.com/r/themes/<id>`, which the browser must block under that policy.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the CSP boundary. Given the current docs and implementation already specify a direct browser fetch, the narrowest code change is a host-specific tweakcn `connect-src` allowance; a Gateway proxy is safer if Control UI should not connect to third-party hosts directly.
