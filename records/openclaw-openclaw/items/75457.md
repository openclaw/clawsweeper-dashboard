## Summary

Keep open. Current main still has the reported policy mismatch: the served Control UI CSP allows only same-origin plus WebSocket connections, while the Appearance theme importer still performs a browser `fetch()` to the tweakcn registry. The related closed duplicate #75535 reinforces that this affected v2026.4.29 users, and there is no current-main implementation proving the requested behavior works yet.

## What This Changes

The issue requests that tweakcn theme imports work from the Control UI despite the current Content-Security-Policy blocking browser fetches to the tweakcn registry.

## Best Possible Solution

Align the approved tweakcn import architecture with the Control UI security policy. If direct browser import is intended, add a narrow tweakcn `connect-src` allowance and regression coverage; if third-party browser egress should remain blocked, move the fetch behind a Gateway proxy while preserving the existing URL validation, redirect rejection, timeout, size limit, and payload schema checks.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection gives a deterministic path: the served Control UI applies `connect-src 'self' ws: wss:`, then the Appearance importer calls browser `fetch()` for `https://tweakcn.com/r/themes/<id>`, which the browser must block under that CSP.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the CSP boundary. The narrowest implementation is a host-specific tweakcn `connect-src` allowance consistent with the current direct-fetch design, while a Gateway proxy is safer if Control UI should not contact third-party hosts directly.
