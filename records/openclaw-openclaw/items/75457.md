## Summary

Current main still has the reported Control UI mismatch: the tweakcn importer performs a browser fetch to tweakcn, while the served CSP only permits same-origin and WebSocket connections. The issue is valid and should remain open for a maintainer decision on whether to use a narrow CSP allowlist or a Gateway proxy.

## What This Changes

The issue requests fixing tweakcn theme import failures in the Control UI caused by `connect-src` blocking browser fetches to `https://tweakcn.com`.

## Best Possible Solution

Align the Control UI CSP/security design with the supported tweakcn import path. If browser-direct import is acceptable, add a host-specific `connect-src https://tweakcn.com` allowance and tests; if not, move the external fetch through a Gateway endpoint while keeping the browser CSP strict.

## Reproduction Assessment

Yes. A served Control UI page invokes a browser fetch to `https://tweakcn.com/r/themes/<id>`, while the response CSP allows only `'self'`, `ws:`, and `wss:` for `connect-src`, so the browser blocks the request before CORS matters.

## Solution Assessment

Unclear until maintainers choose the security boundary. Given current docs and design already specify browser-direct fetching, a narrow tweakcn allowlist is the smallest code change; a Gateway proxy is the safer boundary if external browser connections should remain disallowed.
