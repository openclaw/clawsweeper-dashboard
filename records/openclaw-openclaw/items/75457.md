## Summary

Keep open: current main still serves Control UI with a `connect-src` policy that excludes `https://tweakcn.com`, while the Appearance theme importer fetches the tweakcn registry URL directly in browser code. The report is valid and already has a closed duplicate pointing back to it, but the repair changes a CSP boundary, so maintainers should choose the narrow host allowlist versus a Gateway proxy before implementation.

## What This Changes

The issue requests that Control UI tweakcn theme imports stop failing under the current CSP when the browser fetches `https://tweakcn.com/r/themes/<id>`.

## Best Possible Solution

Align the Control UI security policy with the approved tweakcn import architecture: either add a host-specific `connect-src https://tweakcn.com` allowance for the existing browser-direct design, or add a Gateway proxy if external browser fetches should remain disallowed. The chosen path should update CSP/import tests and keep the tweakcn payload validation and size limits intact.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A served Control UI page uses browser `fetch()` for `https://tweakcn.com/r/themes/<id>`, and the current response CSP only allows same-origin plus WebSocket connections, so the browser blocks the fetch before CORS can help.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the CSP boundary. Given the current docs and implementation already specify a direct browser fetch, the smallest fix is a narrow tweakcn `connect-src` allowance; a Gateway proxy is the safer alternative if Control UI must not fetch third-party hosts directly.
