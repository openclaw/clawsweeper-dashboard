## Summary

Keep open. Current main still has the reported mismatch: the Control UI CSP only allows same-origin and WebSocket connections, while the Appearance theme importer still performs a browser fetch to the tweakcn registry. The related closed duplicate #75535 supports that this affects v2026.4.29 users, and no current-main source, tests, or release evidence shows the import can succeed under the served CSP.

## What This Changes

The issue requests that Control UI tweakcn theme imports succeed instead of being blocked by the current `connect-src` CSP restriction.

## Best Possible Solution

Align the approved tweakcn import architecture with the Control UI security policy. If direct browser import is intended, add a host-specific tweakcn `connect-src` allowance and regression coverage; if third-party browser egress should remain blocked, move the fetch behind a Gateway proxy while preserving URL validation, redirect rejection, timeout, size limit, and payload schema checks.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection gives a deterministic path: the served Control UI applies `connect-src 'self' ws: wss:`, then the Appearance importer calls browser `fetch()` for `https://tweakcn.com/r/themes/<id>`, which the browser blocks under that CSP.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the CSP boundary. The narrowest code change is a host-specific tweakcn `connect-src` allowance, but a Gateway proxy is safer if the intended policy is no third-party browser fetches.
