## Summary

Keep open: current main and v2026.4.29 still serve the Control UI with a `connect-src` policy that excludes tweakcn while the Appearance importer fetches the tweakcn registry URL directly. The report is well supported, but the fix changes CSP behavior, so maintainers should choose between a narrow allowlist and a Gateway proxy.

## What This Changes

The issue asks for the Control UI tweakcn theme importer to succeed by allowing or proxying the `https://tweakcn.com/r/themes/<id>` fetch that the current CSP blocks.

## Best Possible Solution

Align the Control UI security policy with the approved browser-direct tweakcn import path, preferably with the smallest host-specific CSP change if maintainers accept the existing design; otherwise add a Gateway proxy that preserves the stricter browser `connect-src` boundary. Cover the chosen path in `control-ui-csp` and custom-theme import tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A served Control UI page will execute `fetch('https://tweakcn.com/r/themes/<id>')` from the browser while the response CSP only allows same-origin and WebSocket connections, which is enough to block the request before CORS is evaluated.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the CSP boundary. Given the docs and design already specify a direct browser fetch, a narrow `connect-src https://tweakcn.com` allowance plus tests is the smallest maintainable fix; a Gateway proxy is safer only if external browser connections must remain disallowed.
