## Summary

Keep open. Current main still activates the documented top-level managed proxy through `global-agent`, and the Discord-specific proxy setting is only a workaround for the reported process-wide TLS hostname failure.

## What This Changes

Fix `proxy.enabled` so Discord and other HTTPS/WebSocket traffic routed through a local HTTP forward proxy preserves target-host TLS verification.

## Best Possible Solution

Keep the managed proxy contract, but replace or narrow the unsafe Node core hook with an SNI-safe HTTP CONNECT strategy and update docs/tests for any intentional coverage change.

## Reproduction Assessment

Yes. The issue provides exact config, logs, and a minimal `global-agent` reproduction; current main still activates that same top-level `global-agent` path.

## Solution Assessment

No current fix is present. Discord and Telegram explicit proxy settings are useful workarounds, but the maintainable fix needs to address the documented top-level managed proxy behavior.
