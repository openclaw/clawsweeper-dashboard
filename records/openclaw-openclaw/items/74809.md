## Summary

Keep open. Current main still routes the top-level managed proxy through forced global-agent hooks for Node http/https, while the working Discord and Telegram paths are owner-specific explicit proxy workarounds. The report is concrete, current-main source still matches the failure mode, and the repair touches a security-sensitive process-wide egress guardrail rather than a narrow Discord-only bug.

## What This Changes

The issue asks for top-level `proxy.enabled` routing to proxy Discord and other HTTPS/WSS traffic through a local HTTP forward proxy without changing TLS hostname verification to the proxy host.

## Best Possible Solution

Keep the managed proxy contract, but replace or narrow the unsafe global-agent portion so HTTPS and WSS traffic uses an HTTP CONNECT-capable proxy route while TLS server name verification remains tied to the original destination. Add regression coverage for that behavior and update the network proxy docs if the supported public surface changes.

## Reproduction Assessment

Yes. The issue includes config, logs, and a minimal global-agent reproduction, and current main still injects the same GLOBAL_AGENT variables and bootstraps global-agent during managed proxy startup.

## Solution Assessment

No current fix is present. The Discord and Telegram explicit proxy configurations are valid shipped workarounds for owner-specific transports, but they do not solve the documented top-level `proxy.enabled` failure path.
