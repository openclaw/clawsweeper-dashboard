## Summary

Keep open. Current main still activates the top-level managed proxy through `global-agent`, the docs/tests preserve that process-wide contract, and Discord-specific proxy settings remain a workaround rather than a fix for the documented managed proxy behavior.

## What This Changes

Fix top-level `proxy.enabled` routing so a local HTTP forward proxy does not make Discord TLS validate against the proxy hostname.

## Best Possible Solution

Preserve managed proxy egress coverage while replacing or narrowing the unsafe `global-agent` portion with an SNI-safe HTTP CONNECT strategy, plus docs for any supported-coverage changes.

## Reproduction Assessment

Yes. The issue provides exact config, logs, and a minimal `global-agent` reproduction; current main still activates that same top-level `global-agent` path for protected runtime commands.

## Solution Assessment

No current fix is present. The Discord and Telegram explicit proxy paths are useful shipped workarounds, but the maintainable fix needs to address the documented top-level managed proxy behavior.
