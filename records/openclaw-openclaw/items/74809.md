## Summary

Keep open. Current main still matches the reported failure surface: the managed top-level proxy injects `GLOBAL_AGENT_*` variables, forces `global-agent`, and documents Node HTTPS/WebSocket routing through that path, while the working Discord and Telegram routes are owner-specific explicit proxy workarounds rather than a fix for the documented top-level proxy contract.

## What This Changes

The issue asks OpenClaw to make top-level `proxy.enabled=true` route Discord and other runtime traffic through a local HTTP forward proxy without breaking TLS hostname verification against the original destination.

## Best Possible Solution

Keep the managed proxy contract, but replace or narrow the unsafe global-agent portion with an SNI-safe, HTTP CONNECT-capable Node http/https/WebSocket routing strategy. Add regression coverage that verifies HTTPS certificate hostname validation remains tied to the target host, then update the public network proxy docs if the supported coverage changes.

## Reproduction Assessment

Yes. The issue includes exact config, logs, a minimal global-agent reproduction, and working explicit-proxy counterexamples; current main still activates the same global-agent path and lacks a TLS-hostname regression test for it.

## Solution Assessment

No current fix is present. The Discord and Telegram explicit proxy paths are valid shipped workarounds, but they do not solve the documented top-level managed proxy behavior that operators are told to prefer for centralized egress control.
