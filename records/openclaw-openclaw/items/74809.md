## Summary

Keep open. Current main still routes the top-level managed proxy through `global-agent` for Node HTTPS, while the Discord-specific proxy path remains an owner-level workaround and the docs still promise top-level `node:https` and WebSocket coverage.

## What This Changes

The issue asks OpenClaw to make top-level `proxy.enabled=true` route Discord and other HTTPS/WebSocket traffic through a local HTTP forward proxy without changing TLS hostname verification to the proxy host.

## Best Possible Solution

Keep the managed proxy contract, but replace or narrow the unsafe `global-agent` portion with an SNI-safe HTTP CONNECT strategy for Node HTTP/HTTPS/WebSocket routing. Add regression coverage that verifies TLS validation uses the destination hostname, and update the network proxy docs if the supported coverage changes.

## Reproduction Assessment

Yes. The issue includes exact config, logs, and a minimal `global-agent` reproduction; current main still starts the same `global-agent` path and lacks a target-host TLS regression test for it.

## Solution Assessment

No current fix is present. The Discord and Telegram explicit proxy paths are useful workarounds, but they do not solve the documented top-level managed proxy behavior operators use for centralized egress control.
