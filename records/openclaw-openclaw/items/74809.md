## Summary

Keep open. Current main still routes the top-level managed proxy through `global-agent`, the docs and tests preserve that contract, and the Discord-specific proxy setting is only a workaround for the documented process-wide proxy behavior.

## What This Changes

Fix `proxy.enabled=true` so a local HTTP forward proxy routes Discord and other HTTPS/WebSocket traffic without causing TLS hostname verification to use the proxy host.

## Best Possible Solution

Keep the managed proxy contract, but replace or narrow the unsafe `global-agent` portion with an SNI-safe HTTP CONNECT strategy and update docs/tests for any supported-coverage changes.

## Reproduction Assessment

Yes. The issue provides exact config, logs, and a minimal `global-agent` reproduction; current main still activates that same top-level `global-agent` path.

## Solution Assessment

No current fix is present. Discord and Telegram explicit proxy paths are useful workarounds, but the maintainable fix needs to address the documented top-level managed proxy behavior.
