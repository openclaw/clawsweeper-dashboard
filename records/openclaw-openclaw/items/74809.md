## Summary

Keep open. Current main still forces the top-level managed proxy through `global-agent`, the docs/tests still preserve that contract, and Discord-specific proxy settings remain only a workaround for the documented process-wide proxy behavior.

## What This Changes

Report that top-level managed proxy routing can break Discord TLS with a local HTTP forward proxy because process-wide `global-agent` routing validates against the proxy hostname instead of the target host.

## Best Possible Solution

Keep the managed proxy contract, but replace or narrow the unsafe `global-agent` portion with an SNI-safe, HTTP CONNECT-capable routing strategy and document any supported-coverage changes.

## Reproduction Assessment

Yes. The issue provides exact config, logs, and a minimal `global-agent` reproduction; current main still activates that same top-level `global-agent` path and lacks a target-host TLS regression test.

## Solution Assessment

No current fix is present. The Discord and Telegram explicit proxy paths are useful shipped workarounds, but the maintainable fix needs to address the documented top-level managed proxy behavior.
