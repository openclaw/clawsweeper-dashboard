## Summary

Keep open: current main still routes the documented top-level managed proxy through global-agent, while Discord's explicit proxy support is only a workaround for the failing process-wide path.

## What This Changes

The issue asks for `proxy.enabled=true` with a local HTTP forward proxy to route Discord and other runtime traffic without TLS hostname validation switching to the proxy host.

## Best Possible Solution

Keep the managed proxy contract, but replace or narrow the unsafe Node core hook with SNI-safe HTTP CONNECT coverage and update docs/tests for any supported-coverage change.

## Reproduction Assessment

Yes. The report gives exact config, logs, and a minimal global-agent reproduction, and current main still activates that same top-level bootstrap path; I did not need to mutate the checkout to prove the implicated source path remains present.

## Solution Assessment

No current fix is present. Owner-specific Discord and Telegram proxy settings are valid workarounds, but the narrow durable fix should address the documented managed proxy behavior without weakening egress coverage.
