## Summary

Keep open. Current main still enables the documented managed proxy path with forced `global-agent` routing for Node core HTTP/HTTPS, while the working Discord proxy route is a separate account/channel-specific workaround. The report is concrete and source-backed, but the fix needs maintainer design because this path is a process-wide managed egress guardrail.

## What This Changes

The issue requests that top-level `proxy.enabled` route Discord gateway/API traffic through a configured local HTTP forward proxy without causing TLS hostname verification to use the proxy hostname.

## Best Possible Solution

Keep the top-level managed proxy contract, but replace or narrow the unsafe `global-agent` path so HTTPS and WSS traffic use an HTTP CONNECT-capable proxy route while TLS hostname verification remains tied to the original destination. Add regression coverage with a local CONNECT proxy and real hostname verification, then update the network proxy docs if the supported surface changes.

## Reproduction Assessment

Yes. The issue includes config, logs, and a minimal `global-agent` reproduction, and current main still injects the same `GLOBAL_AGENT_*` variables and bootstraps `global-agent` during managed proxy startup.

## Solution Assessment

No current fix is present. The Discord-specific proxy configuration is a valid shipped workaround for Discord transports, but it does not solve the documented top-level `proxy.enabled` failure path.
