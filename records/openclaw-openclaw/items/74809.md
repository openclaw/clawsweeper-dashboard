## Summary

Keep open: current main still uses the top-level `global-agent` managed proxy path and lacks a merged TLS target-host fix; open draft PR #76442 already targets this issue with closing syntax, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Top-level `proxy.enabled` with a local HTTP forward proxy should preserve the original Discord HTTPS/WebSocket target host during TLS validation instead of validating against the proxy hostname.

## Best Possible Solution

Land PR #76442 or an equivalent managed `global-agent` HTTPS CONNECT fix that preserves the original target host while retaining documented managed proxy coverage.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence. The report provides exact config, logs, and a minimal `global-agent` repro, and current main still enables that same managed proxy bootstrap path without target-host TLS coverage.

## Solution Assessment

Yes, the linked repair direction is the narrowest maintainable fix. Fixing the managed HTTPS CONNECT boundary is preferable to a Discord-only workaround or weakening TLS verification.
