## Summary

Keep open. Current main still matches the reported failure surface: top-level managed proxy startup injects and bootstraps `global-agent` for process-wide Node HTTPS routing, while the shipped Discord-specific explicit proxy path is only a workaround and does not satisfy the documented top-level proxy contract.

## What This Changes

The issue requests that `proxy.enabled=true` route Discord gateway/API traffic through a local HTTP forward proxy without TLS hostname validation being performed against the proxy host.

## Best Possible Solution

Keep the managed proxy contract, but replace or narrow the unsafe `global-agent` portion with an SNI-safe, HTTP CONNECT-capable Node HTTP/HTTPS/WebSocket routing strategy. Add regression coverage for target-host TLS validation and update the network proxy docs if supported coverage changes.

## Reproduction Assessment

Yes. The issue includes exact config, logs, a minimal `global-agent` reproduction, and working explicit-proxy counterexamples; current main still activates the same `global-agent` path and lacks a TLS-hostname regression test for it.

## Solution Assessment

No current fix is present. The Discord and Telegram explicit proxy paths are useful shipped workarounds, but they do not solve the documented top-level managed proxy behavior that operators use for centralized egress control.
