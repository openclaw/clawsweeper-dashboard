## Summary

Keep open. Current main still has the reported failure surface: the managed top-level proxy forces `global-agent` for Node core HTTPS while docs still advertise that contract, and the existing Discord/Telegram explicit proxy paths are workarounds rather than a fix for the managed proxy path.

## What This Changes

The issue asks for top-level `proxy.enabled` to route Discord gateway/API traffic through a local HTTP forward proxy without changing TLS hostname verification to the proxy host.

## Best Possible Solution

Keep the managed proxy contract, but replace or narrow the unsafe `global-agent` portion with an SNI-safe, HTTP CONNECT-capable routing strategy for Node core HTTP/HTTPS and WebSocket clients. Add a regression test for target-host certificate validation, then update the public network proxy docs if supported coverage changes.

## Reproduction Assessment

Yes. The issue includes exact config, observed Discord TLS logs, a minimal `global-agent` reproduction, and working explicit-proxy counterexamples; current main still activates the same managed `global-agent` path.

## Solution Assessment

No current fix is present. The Discord and Telegram explicit proxy paths are valid shipped workarounds, but they do not solve the documented top-level managed proxy behavior operators are told to prefer for centralized egress control.
