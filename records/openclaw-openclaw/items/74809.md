## Summary

Keep open. Current main still activates the documented top-level managed proxy by injecting `GLOBAL_AGENT_*` variables and bootstrapping `global-agent`, while the working Discord route described in the report is a separate `channels.discord.proxy` / account proxy path. The related later Discord TLS report overlaps this root cause, but this issue is the earlier, more detailed canonical report and is not superseded by a merged fix.

## What This Changes

The issue requests that top-level `proxy.enabled` route Discord gateway and API traffic through a local HTTP forward proxy without causing TLS hostname verification to use the proxy hostname.

## Best Possible Solution

Preserve the top-level managed proxy contract, but replace or narrow the unsafe Node core/global-agent routing so HTTPS and WSS traffic use an HTTP CONNECT-capable proxy path while TLS verification remains tied to the original destination. Add a regression with a local CONNECT proxy and destination hostname validation, then update network proxy docs if the supported surface changes.

## Reproduction Assessment

Yes. The issue includes exact config, logs, and a minimal `global-agent` reproduction, and current main still injects the same `GLOBAL_AGENT_*` variables and bootstraps `global-agent` during managed proxy startup.

## Solution Assessment

No current fix is present. The Discord-specific `channels.discord.proxy` workaround is valid shipped behavior, but it does not solve the documented top-level `proxy.enabled` failure path.
