## Summary

Keep open. Current main still activates the top-level managed proxy by setting forced global-agent variables and bootstrapping global-agent for Node http/https, while the working Discord and Telegram paths are owner-specific explicit proxy workarounds rather than a fix for the documented process-wide proxy contract.

## What This Changes

The issue reports that top-level `proxy.enabled=true` with a local HTTP forward proxy causes Discord TLS hostname verification to use the proxy host instead of the Discord destination.

## Best Possible Solution

Preserve the managed proxy contract, but replace or narrow the unsafe global-agent portion so HTTPS and WSS traffic uses an HTTP CONNECT-capable proxy route while TLS server name verification remains tied to the original destination. Add regression coverage for that behavior and update the network proxy docs if the supported public surface changes.

## Reproduction Assessment

Yes. The issue includes config, logs, and a minimal global-agent reproduction, and current main still injects the same GLOBAL_AGENT variables and bootstraps global-agent during managed proxy startup.

## Solution Assessment

No current fix is present. The Discord and Telegram explicit proxy configurations are useful shipped workarounds, but they do not solve the documented top-level `proxy.enabled` TLS failure path.
