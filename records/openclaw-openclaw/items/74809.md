## Summary

Keep open. Current main still preserves the reported failure surface: managed proxy startup injects and bootstraps `global-agent` for Node core HTTP/HTTPS, the public docs still promise top-level Node HTTPS and WebSocket routing, and the existing external proxy e2e coverage does not verify target-host TLS/SNI safety. The Discord and Telegram explicit proxy paths are useful workarounds, but they do not fix the documented top-level managed proxy contract.

## What This Changes

Fix top-level `proxy.enabled` routing so Discord gateway/API traffic through a local HTTP forward proxy preserves target-host TLS verification instead of validating against the proxy host.

## Best Possible Solution

Preserve the managed proxy contract, but replace or narrow the unsafe `global-agent` portion with an SNI-safe HTTP CONNECT-capable routing strategy for Node HTTP/HTTPS and WebSocket clients. Add regression coverage proving certificate hostname checks stay anchored to the target host, then update the public network proxy docs if supported coverage changes.

## Reproduction Assessment

Yes. The issue includes exact config, observed logs, working explicit-proxy counterexamples, and a minimal `global-agent` reproduction; current main still activates the same `global-agent` path for `proxy.enabled=true`.

## Solution Assessment

No current fix is present. The Discord and Telegram explicit proxy paths are valid workarounds, but they do not solve the documented top-level managed proxy behavior that operators use for centralized egress control.
