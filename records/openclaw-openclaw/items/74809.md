## Summary

Keep open. Current main still uses the documented top-level managed proxy path through `global-agent`, so the Discord-specific proxy settings remain a workaround rather than a fix for process-wide proxy TLS behavior.

## What This Changes

Fix top-level `proxy.enabled` HTTP forward-proxy routing so Discord gateway/API traffic keeps TLS verification scoped to the original target host.

## Best Possible Solution

Keep the managed proxy contract, but replace or narrow the unsafe `global-agent` portion with an SNI-safe HTTP CONNECT strategy and update docs/tests for any supported-coverage changes.

## Reproduction Assessment

Yes. The issue provides exact config, logs, and a minimal `global-agent` reproduction, and current main still activates that same top-level proxy path.

## Solution Assessment

No current fix is present. Discord and Telegram explicit proxy settings are useful workarounds, but the maintainable solution needs to address the documented process-wide proxy behavior.
