## Summary

Keep open: current main and the latest release still route the top-level managed proxy through the same forced `global-agent` Node core hook, while the Discord-specific proxy path is only a workaround for per-channel configuration.

## What This Changes

The issue asks OpenClaw's documented top-level `proxy.enabled` mode to route Discord gateway/API traffic through a local HTTP forward proxy without breaking TLS hostname validation.

## Best Possible Solution

Keep the managed proxy contract, but replace or narrow the unsafe Node core hook with SNI-safe HTTP CONNECT coverage and update docs/tests for any supported-coverage change.

## Reproduction Assessment

Yes. The report gives exact config, logs, and a minimal `global-agent` reproduction, and source inspection shows current main still activates that same top-level bootstrap path.

## Solution Assessment

No current fix is present. The owner-specific Discord and Telegram proxy settings are useful workarounds, but the durable fix needs to address the documented managed proxy behavior without reducing egress coverage.
