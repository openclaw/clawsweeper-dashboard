## Label Justifications

- `P2`: This is a focused QQBot duplicate-reply bug fix with limited channel blast radius, but it is not an emergency or confirmed core runtime outage.
- `merge-risk: 🚨 message-delivery`: Merging this PR intentionally suppresses inbound QQBot messages with repeated peer/messageId pairs, which can change whether a message is processed or replied to.
- `merge-risk: 🚨 availability`: The new dedupe state bounds each peer set but not the peer map, creating long-running memory-growth risk for bots that see many peers.

## Summary

Keep open for normal PR review, but the current patch is not ready to merge: it adds a second same-message-id dedupe path without proving the linked QQBot duplicate-send bug escapes the shared inbound dedupe, and its new peer cache is not globally bounded.

## What This Changes

This PR adds an in-memory per-peer `messageId` cache to QQBot `createMessageQueue().enqueue()` and drops repeated inbound messages before queueing.

## Best Possible Solution

Prove the failing QQBot duplicate boundary first, then keep one bounded idempotency mechanism at that boundary with regression coverage and real QQBot proof.

## Reproduction Assessment

Unclear. The linked issue has real duplicate-reply reports on QQBot, including v2026.5.18, but current evidence does not prove that same-messageId queue replay is the current-main failing path.

## Solution Assessment

No. A queue-local same-messageId cache is plausible only for one hypothesis; current main already has shared `MessageSid` inbound dedupe and the issue discussion also raises distinct-messageId/outbound retry behavior.
