## Summary

Keep open: current main and v2026.6.6 still have lane-wide suspension for provider cooldown paths, while the fallback loop itself advances to the next candidate. The remaining issue is real but needs maintainer direction because the fix overlaps provider cooldown scope, quota suspension, session state, and open cooldown/circuit-breaker PRs.

## What This Changes

The issue reports that provider rate-limit or quota suspension can freeze an entire command lane, preventing healthy configured fallback models from handling work.

## Best Possible Solution

Define one suspension contract where transient provider/model cooldowns skip or probe only the affected provider/model, while whole-lane suspension is reserved for explicit circuit-breaker cases with no healthy fallback path.

## Reproduction Assessment

Yes at source level: current main can call suspendSession with a command lane and set that lane's concurrency to 0 after provider cooldown failover. No live MiniMax/OpenCode repro was run, and the claim that continue prevents the next candidate is not supported by the loop code.

## Solution Assessment

Unclear as a final fix shape. Provider/model-scoped suspension is likely the right direction, but maintainers should first decide when lane suspension remains valid and how it interacts with quota suspension and open circuit-breaker work.
