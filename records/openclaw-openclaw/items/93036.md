- `impact:auth-provider`: The issue is about provider/model rate-limit classification, cooldown, and fallback routing.
- `impact:other`: The lane-level suspension creates a maintainer-visible temporary availability outage that does not fit the narrower owned impact labels.

## Summary

Keep open: current main and v2026.6.6 still use lane-wide suspension for rate-limit/billing failover paths, even though the fallback loop itself does advance to later candidates. The remaining fix needs maintainer direction because it overlaps provider cooldown, quota suspension, and open circuit-breaker/model-exhaustion work.

## What This Changes

The issue asks OpenClaw to scope rate-limit suspension to the failed provider/model instead of freezing the whole lane and blocking healthy fallback candidates.

## Best Possible Solution

Define one suspension contract where transient provider/model cooldowns skip or probe only the affected provider/model while whole-lane suspension remains reserved for explicit circuit-breaker cases with no healthy fallback path.

## Reproduction Assessment

Yes at source level for the lane-wide suspension: current code can call `suspendSession` with a command lane and set that lane's concurrency to 0 after rate-limit/billing failover. No live MiniMax/OpenCode repro was run, and the report's claim that `continue` prevents the next candidate is not supported by the loop code.

## Solution Assessment

No final fix shape is proven yet. Provider/model-scoped suspension is likely the right direction, but maintainers should first decide when lane suspension remains valid and how it should interact with existing auth cooldown, quota suspension, and open circuit-breaker work.
