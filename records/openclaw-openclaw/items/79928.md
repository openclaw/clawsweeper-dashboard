## Summary

This PR is a duplicate of the already-open canonical fix and its command shape is not mergeable: `git checkout --detach -- <ref>` makes normal explicit refs fail, while the canonical PR uses `git switch --detach -- <ref>` for the same bug.

## What This Changes

The PR changes git plugin ref checkout to pass `--` before the user-supplied ref and adds a unit test for option-like refs.

## Best Possible Solution

Close this PR and continue review on https://github.com/openclaw/openclaw/pull/79901 or an equivalent `git switch --detach -- <ref>` fix that preserves normal refs while rejecting option-like invalid refs.

## Reproduction Assessment

Yes. Current main has a source-level path to `git checkout --detach <ref>`, and a scratch Git 2.53.0 repository reproduced both the original option-like-ref behavior and this PR's valid-ref regression.

## Solution Assessment

No. `git checkout --detach -- <ref>` is not the right ref separator for checkout; the safer existing path is the canonical PR at https://github.com/openclaw/openclaw/pull/79901 using `git switch --detach -- <ref>`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.97

Full review comments:

- **[P1] Use a command form that preserves valid refs:** `src/plugins/git-install.ts:290`
  - body: `git checkout --detach -- <ref>` treats `<ref>` as a path argument. In Git 2.53.0, `git checkout --detach -- v1.2.3` exits 128, so this would make every explicit `git:<repo>@<ref>` install fail even for valid tags or branches. Use a command form such as `git switch --detach -- <ref>` that still accepts normal refs.
  - confidence: 0.98

## Security Review

Status: needs_attention

Summary: The diff touches a security-sensitive plugin install pinning path and would break pinned git installs for valid refs.

  - body: The proposed checkout form rejects normal refs as path arguments, so users could no longer install a plugin from an explicit tag, branch, or commit through the documented git ref path.
  - confidence: 0.95
