## Summary

Current main still classifies host-less IRC `nick!user` allowlist entries as stable, so this PR remains useful; the patch looks correct, with the main remaining decision being the intentional upgrade compatibility impact for operators relying on that undocumented shape.

## What This Changes

The PR reclassifies host-less IRC `nick!user` allowlist entries as mutable, updates the IRC doctor warning detector, and adds focused inbound and doctor regression tests.

## Best Possible Solution

Land the hardening if maintainers accept the compatibility impact, keeping `nick!user@host` as the stable default and routing host-less `nick!user` through the existing dangerous name-matching opt-in and doctor warning.

## Reproduction Assessment

Yes at source level: current main builds a host-less `nick!user` candidate and classifies any non-bare entry as stable, while mutable policy only strips dangerous entries. I did not run tests locally in this read-only review, but the PR body includes before/after dispatcher output.

## Solution Assessment

Yes, this is the best fix shape: classify only entries with `@host` as stable, route host-less `nick!user` through the existing dangerous field/policy, and align doctor warnings. Changing the shared mutable policy or adding new config would be broader than needed.
