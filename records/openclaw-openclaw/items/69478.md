## Summary

Keep open: current main contains the narrowed system-event queue dedupe, but the reported approval cascade still has a source-level path because approval creation, gateway registration, and approval forwarding remain keyed by fresh approval/request IDs rather than a stable exec intent. The merged related pull request explicitly scoped itself to the queue seam and said the forwarder/fresh-ID cascade still needed follow-up.

## What This Changes

The issue asks OpenClaw to coalesce or cancel duplicate exec approval retries so heartbeat failover cannot generate repeated approval prompts for the same exec intent.

## Best Possible Solution

Define and test a collision-safe exec approval intent identity or stale-approval cancellation path in approval registration, forwarding, or failover cleanup while keeping the shipped queue-side dedupe intact.

## Reproduction Assessment

Yes, at source level. Current main still creates random approval IDs/context keys and manager/forwarder state is keyed by those IDs, so an identical retried exec request with a fresh ID is represented as a separate pending approval.

## Solution Assessment

No. The merged system-events fix is useful but only partial; the remaining fix belongs in approval registration/forwarding or failover cleanup with a stable, collision-safe approval intent identity.
