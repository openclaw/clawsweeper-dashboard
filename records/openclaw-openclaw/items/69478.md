## Summary

Keep open: current main includes queue-side system-event dedupe and a stale-follow-up session-rebind guard, but exec approval creation, forwarding, and native delivery still use fresh approval IDs as the dedupe key, so the reported fresh-ID approval cascade is not proven fixed.

## What This Changes

The issue asks OpenClaw to coalesce or cancel duplicate exec approval prompts when heartbeat or failover retries requeue the same command under fresh run and approval identifiers.

## Best Possible Solution

Add a focused, tested stable-intent or cancellation rule across exec approval creation, forwarder/native delivery, and follow-up cleanup, then prove it with a live Telegram-style cascade repro.

## Reproduction Assessment

We have source-level reproduction evidence, not live proof: current main still creates fresh approval IDs/context keys and keys forwarded pending approvals by request.id while linked PRs cover only queue-side and rebind slices.

## Solution Assessment

The current partial fixes are good narrowed repairs but not the complete best solution; the best fix needs one owner-approved stable identity or cancellation contract for the whole exec approval cascade.
