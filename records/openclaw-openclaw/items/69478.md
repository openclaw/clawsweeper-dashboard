## Summary

Keep open: current main contains the narrowed system-events queue dedupe, but the reported approval-prompt cascade still has a source-level path because approval creation, gateway registration, approval manager state, and approval forwarding are keyed by fresh approval/request IDs instead of a stable exec intent. The merged related pull request and maintainer comments explicitly scoped the landed fix to the queue seam and left the forwarder/fresh-ID cascade as follow-up.

## What This Changes

Deduplicate or cancel duplicate exec approval prompts across heartbeat failover retries so the same exec intent does not surface repeatedly under fresh approval IDs.

## Best Possible Solution

Keep this issue as the canonical forwarder-side approval cascade bug and design a collision-safe exec approval intent identity or stale-approval cancellation path while preserving the shipped queue-side dedupe.

## Reproduction Assessment

Yes, at source level. Current main still creates random approval IDs/context keys and stores/forwards pending approvals by those IDs, so an identical retried exec request with a fresh ID can be represented as a separate pending approval.

## Solution Assessment

No. The merged system-events fix is useful but only partial; the remaining fix belongs in approval registration, forwarding, or failover cleanup with a stable collision-safe approval intent identity.
