## Label Justifications

- `impact:session-state`: The failure is tied to heartbeat failover and retry state creating new approval IDs for the same command intent.

## Summary

Keep open: the shipped queue-side system-event dedupe fixes part of the reported area, but current main still creates and forwards exec approval prompts under fresh approval IDs, so the fresh-ID prompt cascade is not proven fixed.

## What This Changes

The issue reports duplicate exec approval prompts for the same command intent after heartbeat failover/retry creates new approval IDs.

## Best Possible Solution

Define and implement a security-reviewed stable approval-intent identity or explicit cancellation path at the approval request/forwarder boundary, then prove retries/failover produce one prompt per command intent in a real channel.

## Reproduction Assessment

No high-confidence live reproduction was run against current main. The original report has concrete logs/screenshots, and source inspection still shows the approval request and forwarder paths keyed by fresh approval IDs rather than stable command intent.

## Solution Assessment

No: closing as fixed would be too broad because current main only proves queue-side and follow-up-continuation fixes. The maintainable fix should bind or cancel duplicate approval prompts at the approval request/forwarder layer with explicit security constraints.
