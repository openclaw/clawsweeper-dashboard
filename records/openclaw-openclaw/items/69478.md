## Summary

Keep open: the queue-side system-event dedupe fix has landed and shipped, but current main still creates, stores, forwards, and renders exec approvals by fresh approval IDs, so the reported repeated approval prompts under new IDs remain a source-level path. The merged related pull request and maintainer discussion explicitly scoped that work to `system-events` and left the approval forwarder/manager path as follow-up.

## What This Changes

Deduplicate or cancel retried exec approval prompts across failover so identical pending commands do not reappear under fresh approval IDs.

## Best Possible Solution

Keep this issue as the canonical fresh-ID exec approval cascade bug and design a collision-safe approval intent identity or stale-approval cancellation path while preserving the shipped queue-side dedupe.

## Reproduction Assessment

Yes, at source level. Current main still creates random approval IDs/context keys and stores/forwards pending approvals by those IDs, so an identical retried exec request with a fresh ID can be represented as a separate pending approval.

## Solution Assessment

The landed queue-side fix is useful but not the complete solution. The best remaining fix belongs in approval registration, forwarding, or failover cleanup with a stable collision-safe approval intent identity or explicit stale-approval cancellation.
