## Summary

Keep open: current main has the narrowed queue-side `enqueueSystemEvent` dedupe fix, but the reported duplicate approval cascade still has a source-level path because exec approval creation, registration, and forwarding remain keyed by fresh approval IDs.

## What This Changes

The issue requests deduplication or cancellation for repeated exec approval prompts created by heartbeat/failover retries of the same command intent.

## Best Possible Solution

Keep this as the canonical tracker for a security-preserving stable exec approval intent key or stale-approval cancellation path, while retaining the merged queue-side dedupe as a partial fix.

## Reproduction Assessment

Yes, at source level. Creating two approval requests with the same command/session intent but different generated IDs still reaches ID-keyed registration and forwarding maps, so the second request is treated as a distinct pending approval.

## Solution Assessment

No for closing this issue. The current queue-side implementation is useful, but the remaining maintainable fix belongs in approval registration/forwarding or failover cancellation with an intent identity that cannot collapse distinct approval decisions.
