## Summary

Keep open. Current main still only suppresses immediately consecutive duplicate system-event text and still builds exec approval/system-event identity from fresh run or approval IDs, so the reported retry/failover fanout is not fixed on main or in the latest release. The already-linked open PR #73040 is the right implementation candidate to review or land before this issue closes.

## What This Changes

The issue reports repeated Telegram exec approval prompts during heartbeat failover because identical approval intent can be re-enqueued under fresh run or approval IDs instead of being deduplicated by a stable context/run key.

## Best Possible Solution

Keep this issue open until a merged fix coalesces approval retry intent rather than only adjacent text. The durable endpoint should either carry a stable command/session-level dedupe key through exec approval/system-event enqueueing or cancel stale pending approval events before replacement runs can enqueue a new prompt, with regression coverage for interleaved retries and fresh approval/run IDs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to enqueue an approval-facing event, enqueue any different event, then enqueue the same command intent again under a fresh run or approval ID; current `enqueueSystemEvent` only compares `lastText`, so the later duplicate is accepted.

## Solution Assessment

No for current main. The linked PR direction is plausible, but the best maintainable fix should be reviewed against the true failover path because current main still keys both node exec events and approval prompts on fresh IDs.
