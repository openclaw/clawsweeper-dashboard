## Summary

Keep open: current main has the merged queue-side system-event dedupe and adjacent follow-up dedupe, but the initial exec approval prompt path still creates and forwards prompts under fresh approval IDs, so the reported fresh-ID cascade is not proven fixed.

## What This Changes

The issue reports duplicate exec approval prompts being enqueued under fresh IDs during heartbeat failover because system events are not deduplicated by stable run/context identity.

## Best Possible Solution

Define and implement a security-reviewed stable approval-intent identity or explicit cancellation path at the approval request/forwarder boundary, then prove retries/failover produce one prompt per command intent in a real channel.

No high-confidence live reproduction was run against current main. Source inspection still shows queue-side dedupe is fixed while prompt forwarding and approval management are keyed by fresh approval IDs, so the remaining cascade is source-reproducible but not live-reproduced here.
