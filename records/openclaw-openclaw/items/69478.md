## Summary

Keep open: current main contains the narrowed system-event queue dedupe from the merged follow-up, but the reported duplicate approval cascade still has a source-reproducible path because exec approval creation, registration, forwarding, and follow-up delivery remain approval-ID scoped.

## What This Changes

The issue requests deduplication or cancellation for duplicate exec approval prompts generated under fresh IDs during heartbeat failover retries.

## Best Possible Solution

Keep this as the canonical tracker for a security-preserving stable exec approval intent key or stale-approval cancellation path, while retaining the merged queue-side dedupe as a partial fix.

## Reproduction Assessment

Yes, at source level. Creating two approval requests with the same command/session intent but different generated IDs still reaches ID-keyed registration and forwarding maps, so the second request is treated as a distinct pending approval.

## Solution Assessment

No for closing this issue. The merged queue-side implementation is a good partial fix, but the remaining maintainable fix belongs in approval registration/forwarding with a stable intent key or cancellation path that cannot hide a different command.
