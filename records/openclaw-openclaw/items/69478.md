## Label Justifications

- `P1`: The report describes a real approval-gated heartbeat workflow that can flood a user channel and force operators to disable scheduled agent work.
- `impact:security`: The remaining fix touches the host exec approval boundary and must not collapse distinct commands into one approval.
- `impact:message-loss`: The user-visible failure is duplicate approval prompts saturating the channel and drowning other approval messages.
- `impact:session-state`: The cascade is tied to failover retries, pending approval IDs, and session/run state drifting across attempts.

## Summary

Keep open: the shipped system-events queue dedupe fixed a real narrowed leak, but current main still creates, registers, stores, and forwards exec approvals by fresh approval/request IDs, so the reported forwarder-side duplicate prompt cascade remains source-reproducible.

## What This Changes

The issue requests deduplicating or cancelling retried exec approval prompts so failover retries of the same exec intent do not create fresh channel prompts.

## Best Possible Solution

Keep this as the canonical forwarder/fresh-ID cascade bug and design a collision-safe exec approval intent identity or stale-approval cancellation path while preserving the shipped queue-side dedupe.

## Reproduction Assessment

Yes, at source level. Current main still creates fresh approval IDs/context keys and keeps approval manager plus forwarder pending state keyed by those IDs, so an identical retry with a new approval ID can become a separate pending prompt.

## Solution Assessment

No. The shipped system-events fix is useful but partial; the remaining solution needs maintainer-approved stable intent or cancellation semantics at the exec approval boundary.
