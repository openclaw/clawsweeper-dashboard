## Summary

Keep open: current main has the merged queue-side system-event dedupe, but the reported approval cascade still has a source-reproducible path because exec approval registration and forwarding are still keyed by fresh approval IDs, and maintainer discussion explicitly kept this issue open after the narrowed queue fix.

## What This Changes

The issue asks OpenClaw to coalesce or cancel duplicate exec approval prompts that are regenerated under new IDs during heartbeat failover for the same command intent.

## Best Possible Solution

Keep this issue as the canonical tracker for a security-preserving stable exec approval intent key or stale-approval cancellation path, while retaining the merged queue-side dedupe as a partial fix.

## Reproduction Assessment

Yes, at source level. Creating two approval requests with the same command/session intent but different IDs still reaches ID-keyed registration and forwarder maps, so the second request is treated as a distinct pending approval; I did not run a live Telegram cascade.

## Solution Assessment

No for closing this issue. The current queue-side implementation is a good partial fix, but the remaining maintainable fix belongs in approval registration/forwarding with a stable intent key or cancellation path that cannot hide a different command.
