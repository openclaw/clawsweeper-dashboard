## Summary

Keep open: current main has shipped related queue-side and follow-up-continuation fixes, but the initial exec approval request path still creates, registers, forwards, and renders approval prompts by fresh approval IDs rather than a stable exec intent key. The reporter and related pull request discussion already narrowed this distinction, and source inspection still supports the fresh-ID approval-prompt cascade path.

## What This Changes

The issue reports duplicate exec approval prompts when heartbeat or failover retries requeue the same command under fresh approval IDs or context keys.

## Best Possible Solution

Keep this as the canonical fresh-ID exec approval prompt cascade and design a collision-safe approval intent identity or stale-approval cancellation path in approval registration/forwarding while preserving the shipped system-events and follow-up-continuation fixes.

## Reproduction Assessment

Yes, at source level: current main still creates fresh approval IDs/context keys and stores/forwards pending approvals by those IDs. I did not run the full Telegram heartbeat-failover scenario live in this read-only review.

## Solution Assessment

No, the landed queue-side and follow-up-continuation fixes are useful but not sufficient for the reported initial approval-prompt cascade. The best remaining fix belongs in approval registration, forwarding, or failover cleanup with a stable collision-safe exec intent identity.
