## Summary

Keep open: current main still has the reported source-level path where an empty wiki page is summarized by path, passed through the related-block refresher, and overwritten with a managed Related block. The linked #64696 is complementary persistence hardening, and #75491 fixed a different duplicate-block symptom in the same helper.

## What This Changes

Prevent memory-wiki compile/related-block refresh from rewriting empty or whitespace-only source pages into managed Related-block stubs.

## Best Possible Solution

Add an empty/whitespace guard before memory-wiki writes related blocks to existing pages, log or report the skipped page, and cover the behavior with a regression test; consider helper-level defense only with explicit create-from-scratch handling.

## Reproduction Assessment

Yes from source inspection: a zero-byte `sources/*.md` page is accepted as a summary, `compileMemoryWikiVault` calls `refreshPageRelatedBlocks`, and the helper returns a managed-only Related block that is written back. I did not run a live temp-vault repro because this review is read-only.

## Solution Assessment

Yes: the narrowest maintainable fix is to guard the write path in `refreshPageRelatedBlocks` so existing empty or whitespace-only pages are not converted into generated-only stubs. Helper-level hardening is useful defense-in-depth but should preserve intentional create-from-scratch callers.
