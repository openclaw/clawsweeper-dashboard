## Summary

Current main and the latest stable tag still contain the reported stdout write followed by an immediate synchronous exit, and the consuming live CLI backend script expects parseable JSON from that command, so this should stay open as a narrow script correctness bug.

## What This Changes

The issue asks for the live CLI backend metadata script to avoid calling process.exit immediately after writing JSON to stdout so piped consumers do not receive truncated output.

## Best Possible Solution

Refactor the unsupported-provider path so it emits the JSON once and lets stdout drain naturally before exit, then add focused regression coverage for parseable piped output.

## Reproduction Assessment

Yes from source inspection: current main and v2026.5.18 still write JSON to stdout and immediately call process.exit(0) on the unsupported codex-cli path, matching Node's documented stdout truncation hazard. I did not establish a live failing pipe run.

## Solution Assessment

Yes, a drain-safe early return or main-function branch is the right fix. A bare replacement with process.exitCode at the current top level is not enough unless the rest of the script is skipped, because it would otherwise fall through and emit a second payload.
