## Summary

Keep open: current main still has the reported stdout-write followed immediately by `process.exit(0)`, and Node's documented/runtime behavior makes that a real pipe-truncation hazard rather than a stale or duplicate report.

## What This Changes

The issue asks OpenClaw to let `scripts/print-cli-backend-live-metadata.ts` drain its `codex-cli` unsupported JSON output instead of forcing a synchronous exit immediately after writing it.

## Best Possible Solution

Keep the helper's unsupported `codex-cli` diagnostic, but replace the successful explicit exit with natural process draining, then add a focused regression that validates the JSON remains parseable through a pipe.

## Reproduction Assessment

Yes, at the source and runtime-contract level. Current main has the exact stdout-write plus immediate-exit pattern, and a piped Node runtime proof showed explicit exit truncating buffered stdout while natural exit did not.

## Solution Assessment

Yes. Setting `process.exitCode = 0` and returning or otherwise letting the process end naturally is the narrowest maintainable fix and matches the helper's non-early-exit path.

Fix the stdout truncation hazard in `scripts/print-cli-backend-live-metadata.ts` for the `codex-cli` unsupported branch. Preserve the unsupported JSON payload and usage-error behavior, but do not call `process.exit(0)` immediately after `process.stdout.write`; set `process.exitCode = 0` and return/allow natural exit, or otherwise wait for the write callback before exiting. Add a focused regression under `test/scripts/` that exercises the helper through a pipe or child-process capture and verifies the `codex-cli` output parses as complete JSON. No changelog is expected for this internal tooling fix unless maintainers prefer one.
