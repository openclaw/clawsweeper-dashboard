## Label Justifications

- `P2`: This is a real provider/runtime compatibility bug for tool-using custom OpenAI-compatible models, with limited blast radius and no crash or data-loss evidence.
- `impact:message-loss`: The failed path suppresses the intended tool-use continuation and can leave the assistant turn with no useful visible answer.
- `impact:auth-provider`: The issue is about a configured custom OpenAI-compatible provider/model route failing to execute valid streamed tool calls.

## Summary

Keep open: current main and the latest release still deliberately strip parsed OpenAI-compatible streamed tool calls when the stream ends without a final finish reason, so the reported behavior remains real but needs maintainer direction on the safe completion contract.

## What This Changes

The issue requests preserving OpenAI-compatible streaming tool calls when a provider emits `delta.tool_calls` but ends with `[DONE]` and no final `finish_reason`.

## Best Possible Solution

Define and implement a narrow clean-completion signal for OpenAI-compatible streams that preserves complete parsed no-finish tool calls while still dropping partial, truncated, or visible-text-spurious tool calls.

## Reproduction Assessment

Yes, at source level: current main has both a finalizer branch and a named regression test that drop no-finish streamed tool-call blocks. I did not run the reporter's Evolink route in this read-only review.

## Solution Assessment

Unclear as a direct one-line promotion: preserving complete no-finish tool calls is likely right for a clean `[DONE]` stream, but the implementation must also preserve current truncation protection.
