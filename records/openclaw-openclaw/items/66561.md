## Summary

Keep open. Current main still has the reported classification gap: an attempt-level timeout can abort an active embedded stream and then surface the outcome as timeout/408 without carrying request-started or response-started observations into failover classification.

## What This Changes

Distinguish no-response timeouts from local aborts or stream failures after an openai-codex SSE request or response has already started.

## Best Possible Solution

Add model-call lifecycle state at the embedded-runner boundary and preserve it through failover classification and logging so post-request and post-response-start aborts are surfaced differently from no-response timeouts.

Yes. The issue provides concrete openai-codex/gpt-5.4 SSE setup details, correlated mitmproxy first-byte/client-close timings, and matching OpenClaw timeout logs; current-main source inspection shows the same classification path remains present.

## Solution Assessment

Yes. A narrow embedded-runner and failover-policy change is the maintainable direction; treating this only as an AbortError string-classifier or OpenAI base URL issue would miss the reported post-start timeout scope.
