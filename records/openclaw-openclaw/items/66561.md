## Summary

Keep open. Current main still has the reported timeout-shaped local abort path: model-call first-byte data is recorded only as diagnostics, while the embedded run result and assistant failover boundary still carry only `timedOut`/`aborted` and normalize that to timeout.

## What This Changes

The issue requests that OpenClaw distinguish post-start Codex SSE local aborts from true no-response timeouts instead of surfacing them as generic 408 timeout failover.

## Best Possible Solution

Carry model-call lifecycle observations into the embedded attempt result and failover/logging boundary so post-start local aborts are reported separately from no-response timeouts while preserving legitimate provider-timeout fallback.

## Reproduction Assessment

Yes, at source level with strong external trace evidence. The report gives correlated mitmproxy first-byte/client-close timing and OpenClaw timeout logs, and current main still lacks response-start state at the failover boundary.

## Solution Assessment

Yes. The narrow maintainable fix is to thread model-call lifecycle observations into `EmbeddedRunAttemptResult` and assistant failover observation; changing generic `AbortError` or timeout text matching alone would not distinguish post-start stream aborts from true no-response timeouts.
