## Summary

Keep open. Current main still has the reported timeout-shaped abort surface: response-start/first-byte observations are diagnostic-only, while the embedded attempt result and assistant failover boundary still normalize timed-out local aborts to timeout/408 without knowing whether the upstream SSE response had already started. The closed unmerged fix proposal does not address that boundary, and the open related PRs cover adjacent timeout or SSE issues rather than this issue’s post-start classification gap.

## What This Changes

The issue asks OpenClaw to distinguish post-start OpenAI Codex SSE aborts from no-response timeouts instead of surfacing them as generic 408 timeout failover.

## Best Possible Solution

Thread model-call or transport lifecycle observations into the embedded attempt result and failover/logging boundary so post-start local aborts or stream handling failures are surfaced separately from no-response timeouts while preserving legitimate timeout fallback behavior.

## Reproduction Assessment

Yes, at source level with strong external trace evidence. The reporter supplied correlated mitmproxy first-byte/client-close timing and OpenClaw timeout logs, and current main still lacks response-start state at the failover boundary.

## Solution Assessment

Yes in direction, but it needs maintainer coordination. The narrow maintainable fix is to carry lifecycle observations into `EmbeddedRunAttemptResult` and failover observation; the closed AbortError text-matching PR and the SSE/base URL workaround do not distinguish post-start stream aborts from no-response timeouts.
