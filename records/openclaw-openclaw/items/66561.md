## Summary

Keep open. Current main still has the reported classification gap: model-call diagnostics can observe first streamed bytes, but the embedded attempt result and failover path still normalize attempt-level timeout aborts as timeout without carrying request-started or response-started state into the surfaced failure.

## What This Changes

The issue asks OpenClaw to distinguish no-response timeouts from post-request or post-response-start SSE aborts in embedded Codex runs instead of surfacing all such attempts as timeout/408.

## Best Possible Solution

Carry model-call lifecycle observations from the embedded model-call wrapper into the attempt/failover result, then log and surface post-request and post-response-start aborts differently from no-response timeouts while preserving existing timeout behavior for true no-response runs.

## Reproduction Assessment

Yes from source inspection, without a live Codex rerun. The issue provides correlated transport and OpenClaw log timings, and current main still has the attempt-timeout abort path plus timeout-only failover normalization without model-call lifecycle state.

## Solution Assessment

Yes. A narrow embedded-runner/failover observation fix is the maintainable path; treating this only as an AbortError string-classification bug or as another openai-codex baseUrl/SSE transport issue would miss the reported post-start abort scope.
