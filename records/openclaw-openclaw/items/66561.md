## Summary

Keep open. Current main still preserves only timeout/abort booleans through the embedded-runner result and failover path, so a post-response-start local abort can still be logged and surfaced as a generic timeout/408; the related transport workaround and closed unmerged #66599 proposal do not cover the reporter's root classification case.

## What This Changes

The issue asks OpenClaw to distinguish post-start Codex SSE local aborts from plain provider silence timeouts in embedded-run failover reporting.

## Best Possible Solution

Add provider-call lifecycle state at the embedded-runner boundary and preserve it through attempt results, failover decisions, and logs so OpenClaw can distinguish no-response timeout, post-request abort, post-response-start abort, parser/stream failure, and upstream explicit errors without changing Codex routing.

## Reproduction Assessment

Yes. The issue provides a concrete Codex SSE setup, correlated mitmproxy first-byte/client-close timings, and matching OpenClaw timeout logs; current-main source inspection shows the same timeout-to-generic-failover path remains available.

## Solution Assessment

Yes. A narrow, provider-generic embedded-runner/failover lifecycle change is the maintainable fix; only changing OpenAI base URL/transport or generic AbortError string matching would not preserve the post-start stream evidence.
