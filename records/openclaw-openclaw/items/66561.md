## Summary

Keep open. Current main still has the reported classification gap: an attempt-level timeout abort can cancel the active stream and later surface the assistant outcome as `timeout`/408 without carrying provider request-started or response-started observations into the attempt result or failover decision. The related #66599 proposal was closed unmerged and the #41282 transport workaround does not address the reporter's root post-start classification case.

## What This Changes

The issue requests better classification and observability when an `openai-codex` SSE response has started but the embedded run is locally aborted and reported as a generic timeout.

## Best Possible Solution

Add model-call lifecycle state at the embedded-runner boundary and preserve it through failover classification and logging so OpenClaw can distinguish no-response timeout, post-request local abort, post-response-start local abort, parser/stream failure, and upstream explicit errors without changing unrelated OpenAI transport routing.

## Reproduction Assessment

Yes. The report provides concrete setup, correlated mitmproxy first-byte/client-close timings, and matching OpenClaw timeout logs; current-main source inspection shows the same timeout-to-failover classification path remains available.

## Solution Assessment

Yes. A narrow embedded-runner/failover change is the best fit: carry lifecycle observations from the stream wrapper into `EmbeddedRunAttemptResult` and failover policy, instead of treating this as only an AbortError string-classifier or OpenAI base URL problem.
