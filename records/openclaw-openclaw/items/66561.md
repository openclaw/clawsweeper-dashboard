## Summary

Keep open. Current main still has the timeout-shaped abort path described by the report: model-call first-byte/progress data exists only as diagnostics, while the embedded attempt result and assistant failover boundary still classify bare timed-out aborts as timeout/408 without knowing whether the upstream stream had already started.

## What This Changes

The issue requests that embedded Codex SSE aborts after upstream bytes have begun be distinguished from no-response timeouts instead of surfacing as status 408 timeout/failover.

## Best Possible Solution

Thread model-call lifecycle observations into the embedded attempt result and failover/logging boundary so post-start local aborts or stream handling failures are surfaced separately from no-response timeouts while preserving legitimate timeout fallback behavior.

## Reproduction Assessment

Yes, at source level with strong external trace evidence. The reporter supplied correlated mitmproxy first-byte/client-close timing and OpenClaw timeout logs, and current main still lacks response-start state at the failover boundary.

## Solution Assessment

Yes. The narrow maintainable fix is to carry model-call lifecycle observations into `EmbeddedRunAttemptResult` and failover observation; changing generic `AbortError` text matching or relying on the old SSE/base URL workaround would not distinguish post-start stream aborts from no-response timeouts.
