## Summary

Keep open. Current main still matches the reported gap: an embedded attempt timeout can abort an active stream and downstream failover/logging still collapses that into timeout/408 without preserving whether the provider request or response had already started.

## What This Changes

The issue asks OpenClaw to distinguish openai-codex SSE streams that have already received upstream bytes before a local embedded-run timeout or abort from plain provider silence.

## Best Possible Solution

Add model-call lifecycle state at the embedded-runner boundary and preserve it through failover classification, logging, and status mapping so post-request and post-response-start local aborts are distinct from no-response timeouts.

## Reproduction Assessment

Yes, source-level. The issue provides concrete Codex SSE timing and OpenClaw logs, and current main still has the matching timeout-abort path plus timeout/408 normalization without response-start state.

## Solution Assessment

Yes. The narrow maintainable fix is in the embedded-runner observation and failover path; an AbortError string-only change or the earlier baseUrl/SSE workaround would miss the reported post-response-start timeout scope.
