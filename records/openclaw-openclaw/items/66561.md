## Summary

Keep open. Current main still lacks a response-start or first-byte signal in the embedded attempt/failover contract, and the related fix path is still an open, conflicting PR rather than merged behavior.

## What This Changes

The issue asks OpenClaw to distinguish a locally aborted Codex SSE attempt after upstream response bytes have started from a plain no-response timeout/failover.

## Best Possible Solution

Carry model-call lifecycle or terminal abort reason into the embedded attempt result and failover observation path, or land a rebased https://github.com/openclaw/openclaw/pull/62682 if it covers this exact post-start local-abort classification while preserving legitimate provider timeout fallback.

## Reproduction Assessment

Yes, at source level with strong external trace evidence, though I did not run a live Codex SSE repro in this read-only pass. The reporter supplied correlated mitmproxy first-byte/client-close timing and OpenClaw timeout logs, and current main still lacks response-start state at the failover boundary.

## Solution Assessment

Unclear for the linked open PR until it is rebased and reviewed. The maintainable direction is to distinguish terminal or post-start local aborts from no-response provider timeouts in the attempt/failover contract, not only in diagnostic telemetry.
