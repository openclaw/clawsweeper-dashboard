## Summary

Keep open. Current main still has the timeout-shaped abort path and failover normalization described in the report, while the available first-byte and response-byte observations remain diagnostic side-channel data rather than inputs to the embedded-run result or failover classification.

## What This Changes

The issue asks OpenClaw to distinguish a Codex SSE stream that already started upstream and was then locally aborted from a plain no-response timeout.

## Best Possible Solution

Carry model-call lifecycle observations into the embedded-run result and failover/logging boundary so post-start aborts are surfaced separately from no-response timeouts.

## Reproduction Assessment

Yes, at source level with strong external trace evidence. The issue provides correlated mitmproxy first-byte/client-close timing and OpenClaw timeout logs, and current main still lacks response-started state at the failover boundary.

## Solution Assessment

Yes. The narrow maintainable fix is to thread model-call lifecycle observations from the diagnostic wrapper into attempt/failover classification; only changing generic AbortError or timeout text matching would miss the reported post-start abort case.
