## Summary

Keep this issue open. Current main still has an attempt-level timeout path that aborts the embedded run and later normalizes the assistant-side outcome to `timeout`/408 without carrying model-call request-started or response-started state into the attempt result or failover decision; the related comments also say the #41282 transport workaround did not solve the reporter's root classification problem, and #66599 was closed unmerged.

## What This Changes

The issue requests distinct handling for an openai-codex SSE response that has already started but is locally aborted and currently surfaced as a generic timeout/408.

## Best Possible Solution

Add model-call lifecycle state to the embedded runner boundary and use it in failover classification: distinguish no-response attempt timeout from post-request local abort, post-response-start local abort, parser/stream failure, and upstream explicit error, then preserve that reason through failover logs/status/payloads without changing unrelated OpenAI transport routing.

## Reproduction Assessment

Yes at the report level: the issue includes concrete OpenAI Codex SSE setup, mitmproxy timestamps showing first response bytes before client close, and matching OpenClaw timeout/failover logs. This review did not run that live setup, but current source inspection shows the reported classification path still exists.

## Solution Assessment

Yes, the best fix is a narrow embedded-runner/failover change that carries model-call lifecycle observations into `EmbeddedRunAttemptResult` and failover policy. The prior #66599 AbortError-only proposal and the #41282 transport workaround are not sufficient because they do not distinguish a started SSE stream from a true no-response timeout.
