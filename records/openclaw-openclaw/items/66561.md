## Label Justifications

- `P1`: The report describes real Codex-backed assistant turns aborting or falling through misleading timeout failover, with fresh user comments that the workflow still fails on recent builds.
- `impact:message-loss`: A local abort or unwanted fallback after a started SSE response can suppress or replace the assistant reply the user was waiting for.
- `impact:session-state`: The bug is about embedded run, timeout, abort, and failover state losing the distinction between provider silence and an already-started stream.

## Summary

Keep open. Current main still has the reported timeout-shaped local-abort surface: the embedded run watchdog aborts the local attempt, the attempt/failover contracts do not carry response-start or first-byte state, and timed-out attempts still normalize to timeout in failover logs. The related terminal-abort PR may address part of this path, but it is still open and does not prove the post-start SSE classification gap is fixed on main.

## What This Changes

The issue asks OpenClaw to distinguish OpenAI Codex SSE streams that have already started before a local embedded-run abort from provider-silent timeouts.

## Best Possible Solution

Carry model-call lifecycle state, especially response-start or first-byte observation, into the embedded attempt result and failover/logging boundary, then decide whether https://github.com/openclaw/openclaw/pull/62682 fully closes this case or needs a follow-up.

## Reproduction Assessment

Yes, source-reproducible with strong external trace evidence. The reporter supplied correlated first-byte/client-close and gateway timeout logs, and current main still lacks a response-start signal at the failover boundary.

## Solution Assessment

No, current main is not yet the best solution. The maintainable direction is to carry lifecycle observations into failover decisions; the linked terminal-abort PR may be the right base, but it still needs maintainer review against this post-start SSE case.
