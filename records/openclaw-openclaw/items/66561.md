## Summary

Keep open. Current main still has the reported timeout-shaped abort and failover classification path: response-start observations are available only as diagnostics, while assistant failover still consumes `timedOut`/`aborted` and reports timeout/408 without knowing whether the stream had already started.

## What This Changes

The issue asks OpenClaw to distinguish an OpenAI Codex SSE stream that has already started from a no-response timeout when an embedded run is locally aborted.

## Best Possible Solution

Thread model-call lifecycle observations into the embedded attempt result and failover/logging boundary so post-start local aborts are reported separately from no-response timeouts while preserving legitimate provider-timeout fallback.

## Reproduction Assessment

Yes, at source level with strong external trace evidence. The issue provides correlated mitmproxy first-byte/client-close timing and OpenClaw timeout logs, and current main still lacks response-start state at the failover boundary.

## Solution Assessment

Yes. The narrow maintainable fix is to carry model-call lifecycle observations into `EmbeddedRunAttemptResult` and assistant failover observation; changing generic AbortError text matching alone would not distinguish post-start stream aborts from no-response timeouts.
