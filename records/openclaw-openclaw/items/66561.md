## Summary

Keep open. Current main still has the reported timeout-shaped local abort surface: attempt-level timeout state and assistant failover still collapse a timed-out local abort into timeout/408 without carrying whether the upstream SSE/model stream had already produced bytes, and the latest issue discussion reports the symptom on v2026.5.12.

## What This Changes

The issue requests that OpenClaw distinguish post-start Codex SSE local aborts from provider-silent timeouts instead of surfacing them as generic timeout/408 failover.

## Best Possible Solution

Thread model-call or transport lifecycle observations into the embedded attempt result and failover/logging boundary so post-start local aborts or stream handling failures are surfaced separately from no-response timeouts while preserving legitimate timeout fallback behavior.

## Reproduction Assessment

Yes, at source level with strong external trace evidence. The reporter supplied correlated first-byte/client-close timing and OpenClaw timeout logs, and current main still lacks response-start state at the failover boundary; I did not run a live credentialed Codex SSE session in this read-only review.

## Solution Assessment

Yes on direction, but it needs maintainer coordination. The narrow maintainable fix is to carry response-start or stream-error lifecycle facts into attempt results and failover logging, rather than relying only on AbortError message text or base-URL workarounds.
