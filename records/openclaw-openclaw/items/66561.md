## Summary

Keep open. Current main still has the timeout-shaped local abort path and still lacks a response-start or first-byte signal at the embedded attempt/failover boundary, so a post-start SSE abort can still be surfaced as a plain timeout; the latest user comment also reports the class on v2026.5.12.

## What This Changes

The issue asks OpenClaw to distinguish openai-codex SSE streams that had already started before a local abort from no-response provider timeouts.

## Best Possible Solution

Carry model-call lifecycle observations into the embedded attempt result and failover/logging policy so post-start local aborts are surfaced separately from no-response timeouts while preserving legitimate timeout fallback behavior.

## Reproduction Assessment

Yes, at source level with high confidence. The reporter supplied correlated mitmproxy first-byte/client-close timing and OpenClaw timeout logs, and current main still lacks response-start state at the failover boundary.

## Solution Assessment

Yes in direction, but not as the closed AbortError text-match workaround. The maintainable fix is to carry lifecycle state from the model-call wrapper into EmbeddedRunAttemptResult and the failover observation/policy boundary, coordinated with adjacent timeout work.
