## Summary

Keep open. Current main and the latest release still preserve the reported timeout-shaped abort surface: model-call first-byte/stream-byte observations are diagnostic-only, while the embedded attempt result and assistant failover boundary still normalize timed-out local aborts to timeout/408 without knowing whether the upstream SSE response had already started.

## What This Changes

The issue asks OpenClaw to distinguish post-start Codex SSE local aborts or stream/parser failures from no-response timeouts in embedded-run failover and logging.

## Best Possible Solution

Thread model-call or transport lifecycle observations into EmbeddedRunAttemptResult and the failover/logging boundary so post-start local aborts or stream handling failures are surfaced separately from no-response timeouts while preserving legitimate timeout fallback behavior.

## Reproduction Assessment

Yes, at source level with high confidence, but not as a live current-main run in this pass. The reporter supplied correlated first-byte/client-close timing and OpenClaw timeout logs, and current main still lacks response-start state at the failover boundary.

## Solution Assessment

Yes in direction, but not as the closed AbortError-only patch. The maintainable fix is to carry lifecycle observations into the attempt result and failover observation; the Codex base-URL fix and the run-budget fallback PR address adjacent problems, not this response-start classification gap.
