## Summary

Keep open. Current main still has the reported timeout-shaped abort/classification gap: it can observe model-call first-byte data diagnostically, but the embedded attempt result and assistant failover path still make timeout decisions from `timedOut`/`aborted` without carrying whether the upstream SSE response had already started. The fresh 2026-05-17 report against 2026.5.12 and the open related terminal-abort work support maintainer follow-up rather than cleanup closure.

## What This Changes

The issue reports that OpenAI Codex SSE responses can begin receiving upstream bytes, then get locally aborted and surfaced as a generic embedded-run timeout/408.

## Best Possible Solution

Carry model-call/transport lifecycle observations into the embedded attempt result and failover/logging decision so post-start local aborts or stream handling failures are surfaced separately from no-response timeouts while preserving legitimate timeout fallback behavior.

## Reproduction Assessment

Yes, at source level with strong external trace evidence. The reporter supplied correlated first-byte/client-close timing and OpenClaw timeout logs, and current main still lacks response-start state at the failover boundary.

## Solution Assessment

Yes in direction, but it needs maintainer coordination. The maintainable fix is to thread lifecycle observations into the embedded attempt/failover contract, while the merged stream-timeout refresh and the open terminal-abort PR only address adjacent timeout fallout.
