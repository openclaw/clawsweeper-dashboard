## Summary

Keep open. Current main still has the reported classification gap: the embedded-run timeout path can abort an active model stream and downstream failover/logging still normalizes timeout-shaped assistant failures to `timeout`/408 without carrying request-started or first-byte observations into the decision.

## What This Changes

The issue asks OpenClaw to distinguish post-start OpenAI Codex SSE aborts from plain no-response timeouts instead of surfacing them as generic 408 failover events.

## Best Possible Solution

Carry model-call lifecycle observations from the diagnostic wrapper into the embedded-run attempt and failover/logging boundary so post-request and post-first-byte aborts are surfaced separately from no-response timeouts, while coordinating with the open terminal-abort and stuck-session recovery threads.

## Reproduction Assessment

Yes, source-level with strong external trace evidence. The issue provides a concrete Codex SSE setup and correlated first-byte/client-close/OpenClaw timeout logs, and current main still has the same timeout-to-408 classification path without response-started state.

## Solution Assessment

Yes. The narrow maintainable fix is at the embedded-runner model-call lifecycle and failover-observation boundary; treating this only as an AbortError string classifier or baseUrl/SSE config problem would miss the reported post-start abort case.
