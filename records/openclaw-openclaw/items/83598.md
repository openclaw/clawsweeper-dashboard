## Label Justifications

- `P1`: The report describes a current-release auth regression that can break all main-lane Anthropic traffic for Claude CLI OAuth users after token expiry.
- `impact:auth-provider`: The failure is in provider auth profile resolution, OAuth expiry handling, and Secret/Auth profile routing for `anthropic:claude-cli`.

## Summary

Keep open: the current main/release code still has a plausible auth-provider failure for macOS Claude CLI users because provider-auth resolution runs external CLI discovery with `allowKeychainPrompt: false`, while current tests explicitly keep no-prompt Claude reads on the file credential path even after a successful Keychain read.

## What This Changes

The issue reports that expired `anthropic:claude-cli` OAuth profiles can still dead-end the main agent lane despite fresh macOS Keychain credentials after the earlier cron-lane cache fix.

## Best Possible Solution

Define the supported no-prompt Keychain policy, then add a focused fix that lets expired managed Claude CLI profiles adopt or persist fresh external CLI credentials without triggering a new Keychain prompt, with diagnostics for skipped bootstrap decisions.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: a macOS Keychain-only setup with an expired `anthropic:claude-cli` profile enters provider auth with `allowKeychainPrompt: false`, cannot use the Keychain-backed credential, and can reach the explicit-profile throw path. I did not run the live macOS Keychain flow in this Linux checkout.

## Solution Assessment

Unclear until maintainers choose the no-prompt Keychain policy; reusing or persisting already-read fresh Keychain credentials looks narrower than loosening background prompt behavior. The fix should include regression coverage for both cache direction cases and the explicit-profile main-lane failure.
