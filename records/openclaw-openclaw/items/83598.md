## Label Justifications

- `P1`: The report describes a current-release auth regression that can break all main-lane Anthropic traffic for Claude CLI OAuth users after token expiry.
- `impact:auth-provider`: The failure is in provider auth profile resolution, OAuth expiry handling, and `anthropic:claude-cli` credential routing.

## Summary

Keep open: current `main` still has a credible source-level auth-provider failure for macOS Claude CLI users because runtime/provider auth paths use no-prompt external CLI discovery while current tests codify that no-prompt Claude reads stay on the file credential path, even after a Keychain read.

## What This Changes

The issue reports that expired persisted `anthropic:claude-cli` OAuth profiles can still fail main agent traffic on macOS despite fresh Claude CLI Keychain tokens.

## Best Possible Solution

Define the no-prompt Claude CLI Keychain reuse policy, then add a focused fix and regression coverage so expired same-identity `anthropic:claude-cli` profiles adopt or persist fresh external credentials, with actionable info diagnostics when bootstrap is skipped.

## Reproduction Assessment

Yes, at source level: a Keychain-only macOS setup with an expired stored `anthropic:claude-cli` profile enters provider auth through no-prompt discovery, and current tests show no-prompt Claude reads do not reuse Keychain credentials. I did not run a live macOS Keychain repro in this Linux checkout.

## Solution Assessment

Unclear until maintainers choose the no-prompt Keychain policy; the expiry gate itself can bootstrap from a fresh imported credential, but the runtime still needs a safe way to obtain or persist that credential without interactive prompting. The narrow path is credential-source policy plus targeted diagnostics and regression tests, not a broad auth rewrite.
