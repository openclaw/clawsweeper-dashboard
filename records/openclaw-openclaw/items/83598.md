## Summary

Keep open: current main still has a source-visible Claude CLI OAuth dead-end for expired `anthropic:claude-cli` credentials, and the new Linux/file-credential report broadens this beyond the earlier macOS Keychain-only theory. The open snapshot fix at https://github.com/openclaw/openclaw/pull/85558 covers a related in-memory clobber trigger, but not the expired external-CLI access-token refresh path described here.

## What This Changes

The issue asks OpenClaw to diagnose and fix `anthropic:claude-cli` OAuth refresh so expired persisted profiles adopt or refresh usable Claude CLI credentials instead of failing all Anthropic agent traffic.

## Best Possible Solution

Define the supported no-prompt Claude CLI refresh policy, then add a focused auth-provider fix that handles expired persisted and external Claude CLI OAuth access tokens on macOS and Linux, with diagnostics for skipped bootstrap/refresh decisions.

## Reproduction Assessment

Yes for source-level reproduction: current main can leave an expired `anthropic:claude-cli` profile on the explicit-provider path where external CLI overlay requires a usable imported access token and Claude CLI OAuth refresh can return null. I did not run a live macOS Keychain or Linux container gateway repro in this read-only checkout.

## Solution Assessment

Unclear until maintainers choose the no-prompt Keychain and external-CLI refresh policy. The narrowest maintainable solution is likely a provider-owned Claude CLI refresh/adoption path plus clearer bootstrap-skip diagnostics, while keeping background prompts disabled by default.
