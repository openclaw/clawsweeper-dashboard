## Summary

Keep open: current `main` still has a source-reproducible auth-provider gap around `anthropic:claude-cli` Keychain-only refreshes in no-prompt runtime/status paths, and the remaining behavior needs an explicit maintainer policy decision before an automated repair.

## What This Changes

The issue reports that expired `anthropic:claude-cli` OAuth profiles can still dead-end main-lane Anthropic traffic despite fresh macOS Keychain credentials.

## Best Possible Solution

Define the supported no-prompt Keychain policy, then add a focused fix and regression coverage so expired managed Claude CLI profiles adopt or persist fresh external credentials without surprise prompts, with info-level diagnostics for skipped bootstrap decisions.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: provider auth enters external CLI discovery with `allowKeychainPrompt: false`, no-prompt Claude reads are file-only, and current tests lock in that file-only behavior after a Keychain read. I did not run the live macOS Keychain flow in this checkout.

## Solution Assessment

Unclear until maintainers choose the no-prompt Keychain policy; reusing or persisting already-available fresh Keychain credentials looks narrower than allowing background prompts. The fix should cover the main-lane explicit-profile failure and both cache/persistence directions.
