## Label Justifications

- `P1`: The report describes a current-release auth regression that can break all main-lane Anthropic traffic for Claude CLI OAuth users after token expiry.
- `impact:auth-provider`: The failure is in provider auth profile resolution, OAuth expiry handling, and `anthropic:claude-cli` credential routing.

## Summary

Keep open: current main and the latest release still preserve the reported auth-provider failure shape for macOS Keychain-only Claude CLI users. Provider auth scopes external CLI discovery with `allowKeychainPrompt: false`, no-prompt Claude reads stay on the file path even after a Keychain read, and the explicit-profile throw path still matches the reported main-lane failure.

## What This Changes

The issue asks OpenClaw to stop `anthropic:claude-cli` OAuth refresh from dead-ending main agent traffic after the persisted profile expires, and to add diagnostics for skipped external CLI bootstrap decisions.

## Best Possible Solution

Define the supported no-prompt Keychain policy, then add a focused fix that lets expired managed Claude CLI profiles adopt or persist already-available fresh external CLI credentials without interactive prompts, with info-level diagnostics for skipped bootstrap decisions.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: a macOS Keychain-only setup with an expired `anthropic:claude-cli` profile enters provider auth with `allowKeychainPrompt: false`, cannot use the Keychain-backed credential, and can reach the explicit-profile throw path. I did not run the live macOS Keychain flow in this Linux checkout.

## Solution Assessment

Unclear until maintainers choose the no-prompt Keychain policy; reusing or persisting already-read fresh Keychain credentials looks narrower than loosening background prompt behavior. The fix should include regression coverage for both cache direction cases and the explicit-profile main-lane failure.
