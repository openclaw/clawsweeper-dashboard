## Label Justifications

- `P1`: The issue reports a security policy hook bypass on the common CLI plugin install path.
- `impact:security`: The affected surface is install-time plugin policy enforcement and fail-closed blocking.

## Summary

Keep open: current main still leaves `openclaw plugins install` on the no-plugin-bootstrap path, so a process-local `before_install` runner is not initialized before plugin install scans. The docs and tests establish that `before_install` is supposed to block installs fail-closed once the runner exists, so this is a source-reproducible security bug and a good focused fix candidate.

## What This Changes

The issue reports that CLI plugin installs skip the `before_install` hook even though Gateway chat installs invoke it.

## Best Possible Solution

Make `openclaw plugins install` initialize the plugin hook runner before install security scans, while preserving command laziness for unrelated plugin subcommands.

## Reproduction Assessment

Yes for source-level reproduction: current CLI policy keeps `plugins install` from loading plugins, and the install hook runner returns early when no process-local runner exists. I did not execute the install flow because this review is read-only.

## Solution Assessment

Yes, a CLI bootstrap/registry initialization fix is the narrow maintainable path for the existing hook contract. Gateway RPC delegation would be a larger architecture change and is not needed to honor the documented install-blocking behavior.
