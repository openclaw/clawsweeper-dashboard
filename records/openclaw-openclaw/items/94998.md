## Summary

Keep open: current main and the reported v2026.6.8 release already classify `This operation was aborted` through the shared timeout matcher, so the regex-only diagnosis is not supported, but the no-fallback symptom remains plausible until logs identify whether the abort came from provider-local timeout, an outer caller abort, or a stream/body boundary.

## What This Changes

The issue reports that provider transport AbortError timeouts are not classified as timeouts, so configured model fallbacks are not attempted.

## Best Possible Solution

Keep this open until logs identify the abort owner, then fix the provider-local timeout or stream/body boundary without weakening the terminal-abort guard.

## Reproduction Assessment

## Solution Assessment
