## Summary

Keep open: current main and v2026.6.8 already classify the reported `This operation was aborted` text as timeout-shaped, so the regex-only diagnosis is likely wrong, but the reported no-fallback symptom remains plausible until provider transport and model-fallback decision logs identify which abort signal fired.

## What This Changes

Report says provider transport AbortError timeouts are treated as user aborts, preventing configured model fallbacks after the primary model stalls.

## Best Possible Solution

Keep this open until logs identify the abort owner, then fix the provider-local timeout normalization if confirmed while preserving terminal caller-abort behavior.

## Reproduction Assessment

Unclear: the configuration steps are plausible, but current main and v2026.6.8 already classify the reported message as timeout-shaped. A high-confidence reproduction needs provider transport logs and `model_fallback_decision` lines showing fallback is skipped on current main.
