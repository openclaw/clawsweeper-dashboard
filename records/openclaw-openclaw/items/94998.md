## Summary

Keep open: current main and the latest stable release do classify `This operation was aborted` as timeout text, but the reporter's v2026.6.8 already had that matcher, so the live custom-provider fallback failure is not proven fixed by main or v2026.6.9.

## What This Changes

The issue reports that a provider transport `AbortError` after about 365 seconds is treated as cancellation, so configured model fallbacks are not attempted.

## Best Possible Solution

Keep the issue open until logs or a focused regression identify the abort owner, then land a boundary-level fix that treats provider-originated transport timeouts as failover-worthy while preserving terminal caller aborts.

## Reproduction Assessment

Unclear as a live current-main repro, but source-reproducible at the boundary level: the configured fallback path exists, and terminal caller aborts still intentionally skip fallback. The missing data is a current log proving whether this custom-provider abort is provider-originated or caller/run-originated.

## Solution Assessment

No for the suggested regex-only fix: v2026.6.8, v2026.6.9, and current main already match `operation was aborted` through the shared timeout matcher. The better fix is to prove and repair the provider-local versus terminal caller abort boundary.
