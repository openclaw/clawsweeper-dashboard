## Summary

Keep open: current main and v2026.6.8 already classify the reported `This operation was aborted` text through the shared timeout matcher, so the regex-only diagnosis is likely wrong, but the reported no-fallback symptom still needs logs to distinguish provider-local timeout from terminal caller/run abort.

## What This Changes

The issue asks OpenClaw to treat provider transport AbortError timeouts as model-fallback-eligible so configured fallback models are tried.

## Best Possible Solution

Keep this open until logs identify the abort owner, then fix the provider-local timeout/fallback boundary or confirm that a terminal run timeout is expected to stop fallback.

## Reproduction Assessment

Unclear: the configuration and symptom are plausible, but current main already matches the reported AbortError text. A high-confidence reproduction needs redacted provider transport logs and fallback decision logs showing fallback is skipped on current main.

## Solution Assessment

No: the suggested regex edit is not the best fix because the message is already covered. The maintainable path is to identify whether the abort came from provider transport, stream-stall handling, or a terminal caller/run timeout and preserve that distinction.
