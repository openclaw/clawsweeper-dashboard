## Summary

Keep open: the false Control UI WebChat busy/Stop bug is still valid on current main and the latest release, but this PR is not merge-ready because the narrowed gate can hide active-run controls, the branch is dirty against main, and the supplied proof is still mock-only.

## What This Changes

The PR narrows Control UI WebChat abort/busy UI to `sending || stream !== null` and adds unit plus mocked Chromium regressions for stale active session rows after completion.

## Best Possible Solution

Rebase and repair the Control UI split so stale session-list rows do not drive composer busy UI, while a real local run id still keeps Stop, Queue, and Steer available; then add real browser/gateway proof.

## Reproduction Assessment

Yes, at source level: current main routes combined `hasAbortableSessionRun(state)` into visible chat controls, and current source shows `chatRunId` can remain active while `chatStream` is null. I did not run the exact Windows Chrome plus WSL2 live reproduction in this read-only review.

## Solution Assessment

No. Splitting session-list-only abortability from visible composer state is the right direction, but `sending || stream !== null` is too narrow; the fix should preserve a local active-run signal while excluding session-list-only stale rows.
