## Summary

Keep this PR open: it targets the right stale-session UI bug, but the new `sending || stream !== null` gate is too narrow and can hide Stop/Steer while a local run is still active during tool execution. The supplied proof is also mocked and references a different head SHA than the current PR head.

## What This Changes

The branch narrows Control UI chat busy controls to local send/stream state and adds unit plus mocked Chromium E2E coverage for stale active session rows after completion.

## Best Possible Solution

Keep the stale-session split, but pass an explicit authoritative local active-run signal such as `chatRunId` into `renderChat` and use that for composer abort/steer controls while excluding session-list-only stale rows.

## Reproduction Assessment

Yes, at source level. Current main can have `chatRunId` active while `app-tool-stream.ts` sets `chatStream` to null during tool rendering, and the PR's new gate would then hide abort/steer UI.

## Solution Assessment

No. The stale-session direction is right, but `sending || stream !== null` is not the best boundary because it drops legitimate local active-run states; the fix should distinguish current `chatRunId` from session-list-only abortability.
