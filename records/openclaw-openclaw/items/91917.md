## Summary

Keep open for maintainer diagnosis: the report is credible and current, but OpenClaw stores cron history in the shared state database rather than the cited zero-byte files, and the available evidence does not isolate whether the failure is migration, partition-key, schema, permissions, or database initialization.

## What This Changes

Investigate missing cron run history after upgrading to 2026.6.x, where legacy JSONL logs were archived but `openclaw cron runs` returns no rows.

## Best Possible Solution

Keep the shared SQLite design, obtain active-database and gateway-log diagnostics from the affected upgrade, then repair the narrow migration, store-key, schema, or database-open defect identified by that evidence with an upgrade regression test.

## Reproduction Assessment

No. The report is concrete and source identifies the relevant migration and append paths, but current main uses `state/openclaw.sqlite`; its row/store-key evidence and the append-failure log are missing.

## Solution Assessment

Unclear. Shared SQLite plus doctor migration is the intended architecture, so reviving `runs/history.db` or `runs/index.db` would not be the best fix; the narrow repair depends on active-database diagnostics.
