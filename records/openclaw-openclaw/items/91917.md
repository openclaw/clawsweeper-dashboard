## Summary

Keep open: current and shipped code store cron run history in the shared OpenClaw state database, not `runs/history.db` or `runs/index.db`, but the reporter’s empty `openclaw cron runs` result still indicates a plausible upgrade regression that is not fixed or superseded on current main.

## What This Changes

Report that after a 2026.6.x upgrade, legacy cron JSONL files are archived while migrated and subsequent run history is absent from the cron runs CLI.

## Best Possible Solution

Add an upgraded-state regression fixture that imports representative legacy run logs into `state/openclaw.sqlite`, verifies archival only after row persistence, restarts the cron service, records a new run, and confirms `openclaw cron runs --id` returns both migrated and new entries.

## Reproduction Assessment

No. The report provides a concrete upgrade signature and source identifies the relevant migration path, but the zero-byte files are not canonical and no failing current-main upgraded-state fixture or shared-database inspection is available.

## Solution Assessment

Yes in direction, but not yet in diagnosis. The maintainable solution is to repair and test the single shared SQLite migration/write path, not restore JSONL or create separate history databases.
