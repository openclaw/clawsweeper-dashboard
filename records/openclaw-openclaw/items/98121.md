## Label Justifications

- `P1`: A long-running scheduled agent turn can make a real Telegram chat appear silent and leave a running cron task uncleared.
- `impact:message-loss`: The reported user-visible impact is delayed or missing Telegram replies while cron work occupies or contaminates the live chat lane.
- `impact:session-state`: The core failure involves cron and user messages sharing or corrupting the same persistent session/transcript state.
- `maturity:stable`: The issue affects M4 stable Gateway runtime behavior and the M4 stable CLI task-cancel surface.

## Summary

Keep open: current main still has source-level paths for a non-isolated cron `agentTurn` to bind to a named live session, and cron task cancellation still refuses to clear a running cron task when there is no active process-local cancellation handle. Related open issues cover adjacent cron/session starvation and delivery races, but none fully owns this combined `session:<live-chat>` plus childless-running-task failure.

## What This Changes

The issue requests safer cron `agentTurn` behavior when jobs target live chat sessions and a watchdog/cancel path for cron runs stuck without a child session.

## Best Possible Solution

Define the cron-session invariant first, then implement a narrow fix that keeps background cron work off live reply lanes by default and makes childless running cron records fail or clear safely through the task cancel path.

## Reproduction Assessment

Yes, at source level: current main permits persistent `session:<id>` cron targets, does not force a new run session outside `isolated`, and refuses cron cancellation when no active handle exists. I did not run a live Telegram race in this read-only review.

## Solution Assessment

Unclear. The childless cancel/watchdog piece has a narrow repair path, but the live-session cron collision needs a maintainer product decision about whether explicit session-targeted cron jobs should be rejected, warned, isolated, or made preemptible.
