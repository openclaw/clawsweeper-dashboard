## Summary

Keep open: current main still supports explicit persistent `session:<id>` cron agent turns that can reuse a live chat transcript lane, and the cron task cancel path still refuses to clear a cron task when no active process-local handle exists. Related open issues and PRs cover adjacent main/systemEvent stamping, routed-peer delivery mirrors, gateway saturation, and stale task cleanup, but none fully owns this combined explicit-session cron collision plus childless cron-task failure.

## What This Changes

The issue reports that cron `agentTurn` jobs can bind to a live Telegram chat session, block or contaminate replies, and leave childless running cron tasks that cannot be cancelled.

## Best Possible Solution

Define the cron-session invariant first, then implement a bounded fix that keeps background cron work off live reply lanes by default and makes childless running cron records fail or clear safely through the task cancel path.

## Reproduction Assessment

Yes at source level: current main permits persistent `session:<id>` cron targets, only forces fresh sessions for `isolated`, and refuses cron cancellation when no active handle exists. I did not run a live Telegram race in this read-only review.

## Solution Assessment

Unclear. The childless cancel/watchdog part has a narrow repair path, but the live-session cron collision needs a maintainer product decision about whether explicit session-targeted cron jobs should be rejected, warned, isolated, or made preemptible.
