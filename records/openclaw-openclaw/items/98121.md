## Summary

Keep open: current main still permits explicit persistent cron session targets to run on the requested session key, and cron task cancellation still refuses to clear rows when the process-local cron cancellation handle is missing. The newer watchdog and task-maintenance paths improve adjacent recovery, but they do not settle the live-chat cron policy or guarantee operator-clearable childless/stale cron rows.

## What This Changes

The issue asks OpenClaw to prevent cron agent turns from monopolizing live Telegram chat sessions and to make stalled childless cron runs fail or cancel cleanly.

## Best Possible Solution

Define a cron-session invariant that keeps background cron work off live reply lanes by default, then add a bounded cancellation/watchdog path that can fail or clear stale no-handle cron task rows safely.

## Reproduction Assessment

Yes at source level: current main accepts explicit `session:<id>` cron targets, keeps them on the requested session key, and refuses cron task cancellation when no active cancellation handle exists. I did not run a live Telegram race in this read-only review.

## Solution Assessment

Unclear. The childless/no-handle cron cancel path is a narrow repair, but the live-session cron collision needs a maintainer decision about whether explicit live chat session targets should be rejected, warned, isolated, or made preemptible.
