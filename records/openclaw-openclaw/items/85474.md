## Summary

Keep open: the warn-only diagnostic is useful, but the patch checks fallback-resolved assistant text, so it can miss a real `NO_REPLY` after an earlier message-tool send, and the PR has no after-fix real behavior proof yet.

## What This Changes

The PR adds a warn-only log in `handleMessageEnd` when `lastToolError` is set and the assistant's final reply is `NO_REPLY`, `REPLY_SKIP`, or `ANNOUNCE_SKIP`.

## Best Possible Solution

Keep the warn-only behavior, but detect the ack from the raw stripped assistant text before NO_REPLY mirror fallback and add focused coverage for the prior-message-tool-send plus later-tool-error case.

## Reproduction Assessment

No high-confidence live reproduction was provided or run. Source inspection gives a clear focused reproduction path: set `lastToolError`, keep a prior `messagingToolSentTexts` value, end with raw `NO_REPLY`, and the proposed check sees the mirrored text instead of the ack.

## Solution Assessment

No. The warn-only direction is maintainable, but the check should use the raw stripped assistant reply before `resolveSilentReplyFallbackText` can replace `NO_REPLY`.
