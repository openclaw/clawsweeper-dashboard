## Summary

Keep open for maintainer/security review. Current main and v2026.6.10 still carry the deliberate Host-header origin fallback; the exact arbitrary attacker-origin POST is not proven, but the remaining DNS-rebinding/operator-policy risk is real and compatibility-sensitive.

## What This Changes

The issue asks OpenClaw to remove or harden `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback` for network-exposed Gateway Control UI deployments.

## Best Possible Solution

Maintainers should choose a security policy for this dangerous mode: keep it with additional tested defenses, or remove it with explicit upgrade, docs, and migration handling while preserving `allowedOrigins` as the recommended path.

## Reproduction Assessment

No for the exact arbitrary attacker-origin POST: current source and tests reject mismatched `Origin`/`Host` values. Source does confirm the dangerous same-host fallback branch, so DNS-rebinding or same-host variants remain plausible security review material.

## Solution Assessment

Unclear. Removing the flag is the safest-looking remediation, but it changes an intentional documented break-glass config path; a maintainer-approved hardening plan with focused origin tests may be the better compatibility-aware solution.
