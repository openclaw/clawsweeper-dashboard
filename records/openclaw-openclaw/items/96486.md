## Summary

Keep open for maintainer/security review. Current main and v2026.6.10 do not appear to accept the exact arbitrary attacker-origin POST described, and exposed Host-header fallback is already audited as critical, but the documented break-glass fallback remains a security-sensitive DNS-rebinding/operator-policy surface with no canonical duplicate.

## What This Changes

The issue asks OpenClaw to remove or harden `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback` because Host-header fallback may weaken CSRF protection on network-exposed Gateway Control UI deployments.

## Best Possible Solution

Maintainers should either explicitly accept the current dangerous-mode model or approve a narrower hardening plan, such as `Sec-Fetch-Site` and focused origin tests, or removal with upgrade/migration handling.

## Reproduction Assessment

No. Source and existing tests show `Origin: https://evil.example` does not pass Host-header fallback unless the Origin host matches the request Host, so the exact simple cross-site POST described is not a high-confidence current-main reproduction.

## Solution Assessment

Unclear. Removing the flag may be safer, but it changes an intentional documented break-glass config path; maintainers should decide whether to preserve it with additional defenses or remove it through a compatibility-aware change.
