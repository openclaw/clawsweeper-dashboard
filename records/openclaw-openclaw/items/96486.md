## Summary

Keep open for maintainer/security review: the exact arbitrary-origin CSRF reproduction is not proven, but current main and v2026.6.10 still ship the documented dangerous Host-header fallback, and deciding whether to remove or further constrain it is a security/product compatibility call.

## What This Changes

The issue requests hardening or removal of `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback` because Host-header origin fallback may weaken CSRF protection on network-exposed gateways.

## Best Possible Solution

Maintainers should either explicitly accept the documented dangerous-mode model or approve a compatibility-aware hardening plan, with focused tests for default origins, explicit `allowedOrigins`, Host-header fallback, and non-loopback audit behavior.

## Reproduction Assessment

No for the exact arbitrary attacker-origin POST as written: current source and tests reject mismatched Origin and Host values. Yes from source for the narrower fallback condition: with the dangerous flag enabled, matching Origin host and request Host are accepted without an explicit `allowedOrigins` entry.

## Solution Assessment

Unclear. Removing the flag, requiring explicit origins only, adding Fetch Metadata checks, or adding CSRF tokens are all plausible, but each changes the documented break-glass model and needs maintainer/security approval.
