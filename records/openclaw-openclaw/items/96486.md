## Summary

Keep open for maintainer security/product review. Current main and v2026.6.10 do not confirm the exact arbitrary attacker-origin POST as written, but the documented Host-header fallback still accepts same-host Origin/Host matches when enabled and remains a deliberate security downgrade.

## What This Changes

The issue asks OpenClaw to remove or harden `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback` because it can weaken Control UI CSRF/origin protection on network-exposed gateways.

## Best Possible Solution

Maintainers should explicitly accept the current dangerous-mode contract or approve a compatibility-aware hardening plan, such as removal, stricter same-host authority, or additional fetch-metadata/CSRF defenses.

## Reproduction Assessment

No for the exact attacker-domain POST as written: current source and tests reject mismatched Origin/Host values. Yes at source level for the remaining concern: same-host Origin/Host matches are accepted when the dangerous flag is enabled.

## Solution Assessment

Unclear. Removing the flag is the strongest security answer, but it changes a documented operator break-glass path, so maintainers should choose the security/compatibility contract before implementation.
