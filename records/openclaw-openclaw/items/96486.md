## Summary

Keep open for maintainer/security review: current main does not confirm the exact arbitrary attacker-origin POST described, but the documented dangerous fallback still accepts same-host Origin/Host matches and is audited critical on exposed binds.

## What This Changes

The issue asks to remove or harden `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback` because it may weaken CSRF/origin protection for network-exposed gateways.

## Best Possible Solution

Maintainers should decide whether to keep the break-glass fallback with explicit residual-risk docs/tests or replace it with compatibility-aware hardening/removal plus any required migration guidance.

## Reproduction Assessment

No for the exact arbitrary attacker-domain POST as written: current source/tests reject mismatched Origins. Source does show the flagged fallback still accepts same-host Origin/Host matches when enabled, which is the remaining security-review surface.

## Solution Assessment

Unclear. Removing the flag is the cleanest security posture, but it changes an intentional documented break-glass config path; maintainers should decide between removal and narrower hardening such as additional browser-signal checks and regression tests.
