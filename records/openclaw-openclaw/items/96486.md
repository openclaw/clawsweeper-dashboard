## Summary

Keep open for maintainer security review: the exact arbitrary attacker-origin POST is not confirmed on current main, but the dangerous Host-header fallback still exists in current main and v2026.6.10 as an operator-selected DNS-rebinding/CSRF hardening tradeoff.

## What This Changes

The issue requests removing or hardening `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback` because it can weaken Control UI browser-origin protection on network-exposed gateways.

## Best Possible Solution

Maintainers should decide whether to preserve the dangerous mode with additional defenses such as `Sec-Fetch-Site` or CSRF-token checks, or remove it through a compatibility-aware config/docs/audit change.

## Reproduction Assessment

No. Source and existing tests show `Origin: https://attacker.example.com` does not pass Host-header fallback unless the Origin host matches the request Host, so the exact simple reproduction is not high-confidence on current main.

## Solution Assessment

Unclear. Removing the flag may be the safest security posture, but it changes an intentional documented break-glass path; the maintainable fix needs maintainer-approved security and compatibility direction.
