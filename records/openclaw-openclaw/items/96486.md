## Summary

Keep open for maintainer/security review: current source and tests do not confirm the exact arbitrary-origin POST reproduction, but the dangerous Host-header fallback remains a documented security downgrade and a compatibility-sensitive policy decision.

## What This Changes

The issue requests hardening or removal of `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback` because it can weaken browser-origin protection on exposed Gateway Control UI deployments.

## Best Possible Solution

Maintainers should decide whether to remove the dangerous fallback with compatibility handling or preserve it with narrower defenses such as canonical-origin validation, Fetch Metadata checks, CSRF-token coverage, and focused tests/docs.

## Reproduction Assessment

No for the exact arbitrary-origin POST as written: current source and tests reject mismatched Origin/Host pairs. Source does show a same-host Host-header fallback path when the dangerous flag is enabled, so the remaining concern is source-reproducible as a hardening/policy issue.

## Solution Assessment

Unclear. Removing the flag is plausible but not automatically the best fix because the flag is documented as an operator-selected break-glass path; maintainers should choose the compatibility-aware hardening direction.
