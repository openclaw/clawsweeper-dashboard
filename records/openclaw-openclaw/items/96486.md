## Summary

Keep open for maintainer/security review. Current main rejects the simple mismatched attacker-origin POST described, but the documented Host-header fallback remains an intentional DNS-rebinding hardening downgrade, and removing or adding defenses around it is a security and compatibility decision.

## What This Changes

The issue requests hardening or removal of `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback` because it may weaken browser-origin CSRF protection on exposed gateways.

## Best Possible Solution

Maintainers should either explicitly accept the documented dangerous-mode model or approve a compatibility-aware hardening/removal plan with focused HTTP, WebSocket, audit, and docs coverage.

## Reproduction Assessment

Partly. Source and tests show the dangerous flag accepts same-host Origin/Host fallback, but the exact arbitrary attacker-origin POST described is not a high-confidence current-main reproduction because mismatched Origins are rejected.

## Solution Assessment

Unclear. Removing the flag may be safer, but it changes a documented break-glass config path; maintainers should decide whether to remove it or preserve it with additional defenses and upgrade guidance.
