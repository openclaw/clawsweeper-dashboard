## Summary

Keep open for maintainer security/product review. Current main and v2026.6.10 reject the simple mismatched attacker-origin POST as written, but they still intentionally support the documented Host-header fallback path and audit it as a DNS-rebinding downgrade, with no merged canonical PR that removes or hardens it.

## What This Changes

The issue asks OpenClaw to remove or harden the Gateway Control UI Host-header origin fallback because it may weaken CSRF/DNS-rebinding protection on network-exposed gateways.

## Best Possible Solution

Maintainers should choose a security-reviewed policy: remove the dangerous fallback with compatibility handling, or keep it only with narrower defenses and regression tests for DNS-rebinding and loopback exceptions.

## Reproduction Assessment

Partly. Current source and tests do not reproduce the exact arbitrary attacker-origin POST, but they do show a source-level same-host Origin/Host fallback path when the dangerous flag is enabled.

## Solution Assessment

Unclear. Removing the flag is the cleanest security posture, but it changes a documented break-glass config path; a compatibility-aware maintainer decision is needed before choosing removal versus narrower hardening.
