## Summary

Keep this PR open for a focused revision: the reported corruption is real and the supplied live proof is convincing, but the proposed delimiter whitelist also stops valid home-path shortening in common diagnostic forms such as `path=/data/file`, comma-separated text, pipes, and backtick-wrapped paths. ## What This Changes

The PR adds boundary-aware configured-home replacement to the core and terminal display helpers, adds regression tests, and changes one MCP runtime test to wait for expected log output.

## Best Possible Solution

Use path-token boundary semantics that reject matches embedded in longer filenames or identifiers while continuing to shorten valid home paths next to arbitrary display punctuation, and apply that invariant consistently to core and terminal-core.

## Reproduction Assessment

Yes from source and supplied live evidence: current main still unconditionally replaces `/data` inside `/data_sources/...`, and the deployed release output demonstrates the same failure. I did not execute a new current-main runtime reproduction during this read-only review.

## Solution Assessment

No, not yet. Boundary-aware replacement is the right direction, but a fixed delimiter whitelist is narrower than the helper's arbitrary diagnostic-text contract; path-token semantics are the safer maintainable fix.
