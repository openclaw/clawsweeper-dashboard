## Summary

Keep open: this is a focused and plausible bug fix, but the proposed boundary logic still shortens a home-looking path inside another path, and a sibling formatter keeps the same unbounded replacement. The external proof is also only a Vitest run, so it is not ready for merge under the real-behavior proof gate.

## What This Changes

The PR replaces `shortenHomeInString()`'s global home-string split with boundary-aware replacement logic and adds a regression test for a non-home path segment.

## Best Possible Solution

Use one path-token-aware shortening rule across the core and terminal display surfaces, add regression coverage for segment suffixes and nested absolute path strings, and include redacted real terminal output showing the changed behavior outside Vitest.

## Reproduction Assessment

Yes, source-level: current `main` uses an unbounded split for `shortenHomeInString()`, so any occurrence of the home string inside a longer segment is rewritten. The PR's remaining nested-path case is also source-reproducible from its new separator-before-home condition.

## Solution Assessment

No. The patch should model path-token boundaries without accepting path separators before the home token, and it should either update the terminal-core sibling formatter or document why that surface is unaffected.

Full review comments:

- **[P2] Reject path separators before the home token:** `src/utils.ts:202`
  - body: The new boundary treats `/` or `\` before `display.home` as valid, so a non-home path like `/backup/srv/openclaw-home/.openclaw/openclaw.json` still becomes `/backup$OPENCLAW_HOME/...`. A home match inside arbitrary strings should start at the string or after a text delimiter, not after a path separator that keeps it inside another path.
  - confidence: 0.86

  - body: `packages/terminal-core/src/display-string.ts` still uses `input.split(display.home).join(display.prefix)`, so table rendering keeps the exact partial-path corruption this PR fixes in `src/utils.ts`. Please share the bounded replacement helper or add equivalent coverage for the sibling display surface.
  - confidence: 0.78
