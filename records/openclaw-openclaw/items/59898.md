## Summary

Keep open: this PR addresses a real explicit no-tools prompt/report bug with strong live proof, but the current head still embeds the skills catalog, reports raw skills data, and can reintroduce provider tool-discipline text in no-tool sessions. ## What This Changes

This branch updates agent system prompt/report generation, embedded-runner hosted-tool plumbing, and OpenResponses client-tool validation for explicit empty-tool and hosted-client-tool sessions.

## Best Possible Solution

Land a rebased, narrow no-tools prompt repair that treats explicit empty effective tools as chat-only, reports only rendered prompt content, preserves current main prompt contracts, and leaves hosted client-tool API validation compatible unless maintainers explicitly approve a stricter contract.

Yes. The PR body gives a concrete live CLI before/after repro for an isolated no-tools agent, and source inspection shows the current-main fallback path plus PR-head prompt/report paths that can still disagree; I did not rerun tests in this read-only review.

## Solution Assessment

No. The intended direction is right, but this head still renders and reports skills in no-tool sessions and leaves provider stable-prefix tool guidance active; the safer solution is a narrower rebased prompt/report fix with explicit compatibility handling for hosted tool validation.
