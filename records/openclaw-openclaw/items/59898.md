## Summary

Keep this PR open. Current main still lacks the requested explicit-empty-tool prompt/report behavior, and the PR branch is useful but not yet correct because it still renders the skills catalog for explicit no-tool prompts and still reports skills from the raw `skillsPrompt` input instead of the final rendered prompt.

## What This Changes

The PR adds explicit empty-tool prompt gating, hosted/client-tool handling in embedded prompt/report paths, OpenResponses client-tool validation, and prompt/report/gateway regression tests.

## Best Possible Solution

Land a focused prompt-layer fix that treats an explicit empty effective tool set as a true no-tool prompt: no fallback tool catalog, no skills catalog, no tool-only guidance, rendered-prompt-based report stats, and provider overlay/transform context aligned with the final effective tool set. Keep payload stripping coordinated with #70790 rather than treating it as a replacement.

## Reproduction Assessment

Yes. The PR body provides a concrete `openclaw agent --local` repro with `tools: { deny: ["*"] }`, and source inspection gives a high-confidence local code path: `buildAgentSystemPrompt({ toolNames: [], explicitEmptyToolListMeansNoTools: true, skillsPrompt })` still renders skills on the PR head.

## Solution Assessment

No. The PR is aimed at the right product boundary, but the current branch still leaves the skills catalog/report mismatch in place; the safer solution is to derive no-tool prompt content and report stats from the final effective prompt state and then coordinate separately with the payload-level #70790 work.
