## Summary

This PR should stay open because the reported empty-tool prompt bug is valid, but the current PR head still leaves two prompt/report blockers that contradict the intended no-tools behavior.

## What This Changes

The PR changes agent system prompt/report generation, embedded runner plumbing, OpenResponses client-tool normalization, and tests to distinguish explicit empty tool lists from omitted/default or hosted-tool sessions.

## Best Possible Solution

Keep the no-tools prompt/report direction, but make explicit empty effective tool sets suppress skill catalogs and compute skills diagnostics from the final rendered prompt state; leave payload stripping to the separate payload-focused work.

## Reproduction Assessment

Yes. The PR body gives a concrete live `openclaw agent --local` repro with `tools: { deny: ["*"] }`, and source inspection shows current main and PR head still have the relevant prompt/report paths.

## Solution Assessment

No. The PR targets the right boundary, but the current branch is not ready until no-tool prompt content and report stats are derived from the same final effective prompt state.
