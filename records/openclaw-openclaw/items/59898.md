## Summary

Keep open: this is a real prompt/report bug and the PR is aimed at the right boundary, but the current head still leaks skills into explicit no-tool prompts and reports raw skills data that may not be rendered.

## What This Changes

The PR changes agent system-prompt/report generation and OpenResponses hosted-tool plumbing so explicit empty effective tool lists and hosted-client-tool-only sessions are described consistently.

## Best Possible Solution

Keep the PR direction, but make the final rendered prompt and prompt report derive no-tools skill visibility from the same effective prompt state before merge.

## Reproduction Assessment

Yes. The PR body gives a concrete live `openclaw agent --local` no-tools repro, and current-main source inspection shows the empty-list fallback and raw skills-reporting paths still exist.

## Solution Assessment

No. The proposed direction is narrow and maintainable, but this head is not the best complete fix until it suppresses skills for explicit no-tool sessions and reports only rendered skills.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Suppress skills in explicit no-tool prompts:** `src/agents/system-prompt.ts:837-846`
  - body: When `toolNames: []` and `explicitEmptyToolListMeansNoTools` are active, this still builds `skillsSection` from a non-empty `skillsPrompt` and later spreads it into the final prompt. That leaves the skills catalog in a session that also says no tools are available.
  - confidence: 0.91
- **[P2] Count skills from the rendered prompt:** `src/agents/system-prompt-report.ts:149-174`
  - body: `buildSystemPromptReport` still parses and counts `params.skillsPrompt` directly. If the prompt builder or provider transform removes the Skills section, `/context` can report skill bytes and entries the model did not receive.
  - confidence: 0.9
