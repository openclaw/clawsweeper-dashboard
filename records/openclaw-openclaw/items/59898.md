## Summary

Keep this PR open: the reported empty-tools prompt bug is real and not fixed on current main, but the current PR head still contradicts the no-tools contract by rendering a skills catalog and reporting raw skill stats that may not match the final prompt.

## What This Changes

The PR changes agent system prompt/report construction and OpenResponses hosted-client-tool plumbing to distinguish explicit empty effective tool lists, hosted client tools, and legacy omitted tool lists.

## Best Possible Solution

Keep the no-tools prompt/report direction, but suppress skill catalogs for explicit empty effective tool sets and derive skills diagnostics from the final rendered prompt state; leave payload stripping to the separate payload-focused work.

## Reproduction Assessment

Yes. The PR body gives a concrete live CLI repro using a tools-deny-all agent, and current-main source inspection shows the empty-list fallback and raw skills-report paths still exist.

## Solution Assessment

No. The PR targets the right boundary, but the current branch is not ready until no-tool prompt content and report stats are derived from the same final effective prompt state.

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Suppress skill catalogs in explicit no-tool prompts:** `src/agents/system-prompt.ts:837-846`
  - body: When `toolNames: []` and `explicitEmptyToolListMeansNoTools` are active, this still builds `skillsSection` in `catalog-only` mode and later spreads it into the prompt.   - confidence: 0.9
