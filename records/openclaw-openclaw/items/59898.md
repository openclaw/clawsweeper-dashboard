## Summary

Keep open: the PR targets a real prompt/report bug and includes useful live proof, but the current head still leaves the explicit no-tool prompt and `/context` skills report internally inconsistent.

## What This Changes

The PR changes agent system-prompt construction, prompt reporting, embedded-runner prompt plumbing, and OpenResponses client-tool validation so explicit empty tool sets and hosted-client-tool-only sessions are represented more truthfully.

## Best Possible Solution

Keep the focused no-tools prompt/report fix, but make empty effective tool sets suppress all skill catalog/tool-only prompt content and derive skills diagnostics from the final rendered prompt; leave payload stripping to the separate payload-focused work.

## Reproduction Assessment

Yes. The PR body gives a concrete live CLI repro using a deny-all tools agent and `/context detail`, and current-main source inspection shows the fallback tool catalog plus raw skills-report paths still exist.

## Solution Assessment

No. The direction is the right maintainable boundary, but this branch still needs narrow source/test repairs so the final prompt and prompt report agree for explicit no-tool sessions.

- **[P2] Suppress skill catalogs in no-tool prompts:** `src/agents/system-prompt.ts:837-846`
  - body: When `toolNames: []` and `explicitEmptyToolListMeansNoTools` are active, this still builds `skillsSection` in `catalog-only` mode and later spreads it into the prompt.   - confidence: 0.9
