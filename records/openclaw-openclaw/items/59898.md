## Summary

Keep open: current main still has the empty-tool prompt/report bug, and this PR is directionally useful, but the current head still contradicts its no-tools contract by rendering the skills catalog and reporting raw skills bytes.

## What This Changes

The PR adds explicit empty-tool prompt gating, hosted/client-tool prompt/report handling, OpenResponses client-tool validation, and regression tests across agents, embedded runner, OpenAI, and gateway paths.

## Best Possible Solution

Keep the prompt-layer fix, but make explicit empty effective tool sets suppress skills catalogs and derive skills report stats from the final rendered prompt; handle payload stripping separately with #70790.

## Reproduction Assessment

Yes. The PR body gives a concrete `openclaw agent --local` repro with `tools: { deny: ["*"] }`, and source inspection gives a direct unit-level path: `buildAgentSystemPrompt({ toolNames: [], explicitEmptyToolListMeansNoTools: true, skillsPrompt })` still renders skills on the PR head.

## Solution Assessment

No. The PR targets the right product boundary, but the current branch is not the best fix until no-tool prompt content and report stats are derived from the same final effective prompt state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Drop skills catalog from explicit no-tool prompts:** `src/agents/system-prompt.ts:184-191`
  - body: The explicit no-tools path still routes through `catalog-only` mode, which renders `## Skills (mandatory)` and appends the full skills catalog. That contradicts the PR's stated no-tools contract and still exposes skill names/descriptions to sessions whose effective tool set is empty.
  - confidence: 0.91

  - confidence: 0.88
