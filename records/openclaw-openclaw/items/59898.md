Keep this PR open: current main still has the underlying explicit-empty-tools prompt mismatch, but the current PR head still leaves no-tool sessions with skill/report/provider prompt inconsistencies that should block merge.

## What This Changes

The PR updates agent system-prompt/report construction, embedded-runner prompt plumbing, and OpenResponses client-tool normalization for explicit no-tool and hosted-client-tool sessions.

## Best Possible Solution

Repair this branch or replace it with a narrow current-main branch that makes explicit no-tool prompts omit tool-only skills/provider guidance and derives prompt reports from rendered prompt contents.

## Reproduction Assessment

Yes. The PR body gives a concrete live CLI no-tools repro with before/after output, and source inspection of current main plus PR head explains the same prompt/report mismatch.

## Solution Assessment

No, not at the current PR head. The direction is maintainable, but the patch still leaks skills/provider tool guidance and reports raw skill input instead of rendered prompt contents.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Suppress skills in explicit no-tool prompts:** `src/agents/system-prompt.ts:837-846`
  - body: When toolNames is explicitly empty, this still builds a Skills section from any non-empty skillsPrompt and inserts it into the final prompt. That leaves a skills catalog in a prompt that says no tools are available, which is the leakage this PR is meant to remove.
  - confidence: 0.94
- **[P2] Report only rendered skills content:** `src/agents/system-prompt-report.ts:149`
  - body: buildSystemPromptReport still parses and counts the raw params.skillsPrompt. If prompt construction or a provider transform omits the Skills section, /context can report skill bytes and entries that were never sent to the model.
  - confidence: 0.93
- **[P2] Skip provider stable prefixes without tools:** `src/agents/system-prompt.ts:965-967`
  - body: The explicit no-tools path still injects providerStablePrefix unconditionally. The built-in GPT-5 stable prefix contains tool-discipline instructions, so no-tool sessions can still be told to continue with tool calls that are unavailable.
  - confidence: 0.88
