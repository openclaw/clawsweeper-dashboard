## Summary

Keep open: current main still has the empty-tools prompt/report bug, and this PR is useful, but the branch still leaves no-tool sessions with contradictory skills/provider prompt content and is conflict-marked against main.

## What This Changes

The PR changes agent system-prompt construction, prompt reporting, embedded-runner plumbing, and OpenResponses client-tool validation so explicit no-tool and hosted-client-tool sessions advertise a truthful tool context.

## Best Possible Solution

Land a focused prompt/report fix that suppresses all skill catalog content and tool-driving provider guidance for true no-tool sessions, derives skills diagnostics from the rendered prompt, and leaves broader effective-tool visibility to https://github.com/openclaw/openclaw/issues/44484 and payload stripping to https://github.com/openclaw/openclaw/pull/70790.

## Reproduction Assessment

Yes. The PR body gives a concrete live CLI repro using a deny-all tools agent, and source inspection of current main shows the fallback tool catalog and raw skills-reporting paths that explain the failure.

## Solution Assessment

No. The direction is the right maintainable boundary, but the current branch still needs narrow repairs so explicit no-tool prompts contain no skills catalog, no tool-driving provider prefix, and diagnostics reflect the rendered prompt.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Suppress skills for explicit no-tool prompts:** `src/agents/system-prompt.ts:837-846`
  - body: When `toolNames: []` and `explicitEmptyToolListMeansNoTools` are active, this still builds `skillsSection` in `catalog-only` mode and later inserts it into the prompt. That leaves a Skills section in sessions that also say no tools are available, so the model still sees tool-only skill catalog content the PR is meant to remove.
  - confidence: 0.93
- **[P2] Derive skill stats from the rendered prompt:** `src/agents/system-prompt-report.ts:149`
  - body: `buildSystemPromptReport` still parses `params.skillsPrompt` and reports the raw input length. If the prompt builder or provider transform removes the Skills section, `/context` can report skill bytes and entries the model did not receive, leaving the diagnostics bug in place.
  - confidence: 0.92
- **[P2] Skip tool-oriented provider prefixes without tools:** `src/agents/system-prompt.ts:966`
  - body: The no-tools path still injects `providerStablePrefix` unconditionally. For OpenAI GPT-5, that stable prefix includes tool-discipline instructions such as doing another tool call before replying, so explicit no-tool sessions can still receive contradictory provider-owned tool guidance.
  - confidence: 0.84
