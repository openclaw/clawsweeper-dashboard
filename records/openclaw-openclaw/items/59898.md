## Summary

Keep open: current main still does not implement the explicit no-tool prompt/report contract, and the PR direction is useful, but the current PR head still has blocking prompt/report defects that leave the no-tools path internally inconsistent.

## What This Changes

The PR updates agent system-prompt construction, prompt-report metadata, embedded-runner client-tool plumbing, OpenResponses client-tool validation, and focused tests for explicit no-tool and hosted-client-tool sessions.

## Best Possible Solution

Repair and rebase the branch so explicit no-tool sessions omit skills and all tool-only provider guidance, prompt reports measure only rendered prompt content, and hosted-client-tool sessions remain truthful.

## Reproduction Assessment

Yes. The PR body gives a concrete `openclaw agent --local` no-tools reproduction with before/after prompt-report output, and current-main source still has the fallback tool catalog plus raw-skills report paths.

## Solution Assessment

No, not at the current PR head. The approach is the right narrow fix, but the patch still renders skills, reports raw skills input, and applies GPT-5 provider stable-prefix tool guidance in explicit no-tool sessions.

  - body: When `toolNames` is explicitly empty, this still calls `buildSkillsSection` with a non-empty `skillsPrompt`, so the final prompt can contain the skills catalog while also saying no tools are available. That leaves the core prompt leak this PR is meant to remove.

  - confidence: 0.93

  - body: The explicit no-tools path gates the execution-bias override, but still injects `providerStablePrefix` unconditionally. The GPT-5 stable prefix contains tool-discipline instructions, so no-tool sessions can still receive guidance to prefer tool evidence and continue tool calls.
  - confidence: 0.88
