## Summary

This should stay open because the reported bug is still present on current main, and this PR is a plausible fix path, but the current PR head still leaves explicit no-tool sessions internally inconsistent in three source-level places.

## What This Changes

The PR changes agent system-prompt/report construction, embedded-runner prompt plumbing, OpenResponses client-tool validation, and tests so explicit no-tool and hosted-client-tool sessions advertise tool availability more truthfully.

## Best Possible Solution

Repair this PR so explicit no-tool sessions omit skills and provider tool guidance and /context derives skills from the rendered prompt, while leaving payload stripping to https://github.com/openclaw/openclaw/pull/70790 and broader tool diagnostics to https://github.com/openclaw/openclaw/issues/44484.

## Reproduction Assessment

Yes, source-reproducible with high confidence. The PR body gives a concrete no-tools CLI repro, and current main still has the fallback tool catalog plus raw skills-reporting path that explain the failure.

## Solution Assessment

No, not as currently implemented. The direction is maintainable, but the PR head still leaves three narrow prompt/report inconsistencies that should be fixed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Suppress skills for explicit no-tool prompts:** `src/agents/system-prompt.ts:837-846`
  - body: When `toolNames: []` and `explicitEmptyToolListMeansNoTools` are active, this still builds `skillsSection` from a non-empty `skillsPrompt` and later inserts it into the prompt. That leaves `## Skills (mandatory)` and the raw catalog in a session that says no tools are available, so the model still sees skill-selection content this PR is meant to remove.
  - confidence: 0.94
- **[P2] Derive skills stats from the rendered prompt:** `src/agents/system-prompt-report.ts:149`
  - body: `buildSystemPromptReport` still parses and counts `params.skillsPrompt` directly. If prompt construction or provider transforms remove the Skills section, `/context` can still report skill bytes and entries that were never sent to the model.
  - confidence: 0.93
- **[P2] Skip provider stable prefixes without tools:** `src/agents/system-prompt.ts:965-967`
  - body: The explicit no-tools path still injects `providerStablePrefix` unconditionally. For GPT-5/OpenAI sessions that prefix includes tool-discipline instructions, so a prompt can still combine `No tools are available` with provider-owned guidance to prefer additional tool work.
  - confidence: 0.88
