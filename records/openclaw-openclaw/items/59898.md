## Summary

Keep this PR open: current main still has the explicit empty-tool prompt/report bug, but the current PR head is not mergeable as-is because it still leaks skills/report/provider tool guidance in the no-tools path.

## What This Changes

The PR updates agent system-prompt/report construction, embedded-runner prompt plumbing, and OpenResponses client-tool handling for explicit no-tool and hosted-client-tool sessions.

## Best Possible Solution

Repair or replace this branch on current main so explicit empty-tool sessions suppress all tool-only prompt sections, derive prompt-report skills from rendered prompt content, and keep hosted-client-tool sessions truthful without a broad capability redesign.

## Reproduction Assessment

Yes, source-reproducible. The PR body supplies a concrete live CLI no-tools repro with before/after output, and current main source still falls back to standard tool guidance and raw skills reporting when the effective tool list is empty.

## Solution Assessment

No, not at the current PR head. The overall direction is maintainable, but the patch still leaks skills/provider tool guidance and reports raw skill input, so the safer path is a focused repair or current-main replacement branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Suppress skills for explicit no-tool prompts:** `src/agents/system-prompt.ts:837-846`
  - body: When `toolNames` is explicitly empty, `buildSkillsSection` still runs for any non-empty `skillsPrompt` and its result is inserted later. That leaves a Skills section in a prompt that also says no tools are available, so the no-tool path still leaks the catalog this PR is meant to remove.
  - confidence: 0.94

  - confidence: 0.93

  - body: The explicit no-tools path still injects `providerStablePrefix` unconditionally. The GPT-5 stable prefix contains tool-discipline text, so no-tool sessions can still receive instructions to keep making tool calls.
  - confidence: 0.88
