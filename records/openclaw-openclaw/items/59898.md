## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep hosted tool names out of native alias policy:** `src/agents/command/shared-types.ts:50-52`
  - body: `normalizeToolName` is the native OpenClaw tool-policy normalizer, so it folds `bash` into `exec` and lowercases names. Applying it to OpenResponses hosted/client tools makes this endpoint reject client-defined tools like `bash` plus `exec` or `Foo` plus `foo` that current main passes through; keep hosted tool names in an exact sanitized namespace or split this policy change for explicit approval.
  - confidence: 0.89
- **[P2] Omit skills from explicit no-tool prompts:** `src/agents/system-prompt.ts:837-846`
  - body: This branch still builds and inserts `skillsSection` even when `hasExplicitEmptyToolList` is true, and the added test expects `## Skills (mandatory)` to remain in a no-tool prompt. That leaves the skills catalog in sessions whose tools were explicitly disabled, contradicting the stated no-leaked-skills behavior and keeping tool-only skill guidance in a chat-only run.
  - confidence: 0.94
- **[P2] Report skills from the rendered prompt:** `src/agents/system-prompt-report.ts:149`
  - body: The report still parses `params.skillsPrompt` directly, so once the no-tool renderer correctly omits skills, `/context` can continue showing skill bytes and entries that were not present in the final prompt. Derive skill stats from the rendered prompt or pass structured rendered-skill metadata instead of the raw catalog input.
  - confidence: 0.84
- **[P2] Skip provider stable prefixes for no-tool sessions:** `src/agents/system-prompt.ts:965-967`
  - body: The no-tools path gates `execution_bias` overrides, but still injects `providerStablePrefix`. The built-in GPT-5 stable prefix contains `<tool_discipline>` instructions that prefer tool evidence and more tool work before replying, so explicit no-tool sessions can still receive provider-level tool-driving text.
  - confidence: 0.86
