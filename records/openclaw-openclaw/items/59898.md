## Summary

Keep this PR open because it addresses a real prompt/report bug, but the current head still has blocking correctness issues in the no-tools path. The useful direction is valid; it needs a narrow repair before merge.

## What This Changes

The PR updates agent system-prompt construction, prompt-report metadata, embedded-runner hosted-tool plumbing, and OpenResponses client-tool validation for explicit no-tool and hosted-client-tool sessions.

## Best Possible Solution

Land a narrow prompt/report repair that treats explicit empty effective tools as chat-only, suppresses tool-only provider guidance, and preserves hosted-client-tool behavior.

## Reproduction Assessment

Yes. The PR body gives concrete `openclaw agent --local` no-tools steps and before/after prompt-report metadata, and source inspection confirms the current main prompt/report paths can count and render skills independently of effective no-tool state.

## Solution Assessment

No, not at the current PR head. The direction is right, but the implementation still needs to suppress rendered skills, derive report skills from the rendered prompt, and avoid tool-driving provider stable prefixes when no first-class tools exist.
