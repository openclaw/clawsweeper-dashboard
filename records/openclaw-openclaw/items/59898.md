## Summary

Keep this PR open: the underlying no-tools prompt/report bug is valid and not fixed on current main, but the current PR head still has two concrete prompt/report correctness blockers. The contributor supplied useful live before/after output, so the remaining issue is repairability of the patch rather than missing proof.

## What This Changes

The PR changes agent system prompt/report generation and hosted-client-tool plumbing to distinguish explicit empty tool lists from omitted/default or client-tool-only sessions.

## Best Possible Solution

Keep the no-tools prompt/report direction, but suppress skill catalogs for explicit empty effective tool sets and derive skills diagnostics from the final rendered prompt state; leave payload stripping to the separate payload-focused work.

## Reproduction Assessment

Yes. The PR body gives a concrete live `openclaw agent --local` repro with `tools: { deny: ["*"] }`, and source inspection of current main shows the empty-list fallback and raw skills-reporting paths still exist.

## Solution Assessment

No. The PR targets the right boundary, but the current branch is not ready until no-tool prompt content and report stats are derived from the same final effective prompt state.
