## Summary

Current main fixes the reported Brave provider discovery path without changing the Brave manifest: configured web-search providers are now included in the Gateway startup plan and the agent runtime can still fall through to a scoped provider load when the active registry lacks web providers. This is current-main-only; the latest release mentioned in the report still predates the startup-plan part of the fix.

## What This Changes

The issue asks for first-class agent `web_search` to discover the configured Brave provider without manually changing the Brave plugin manifest to `activation.onStartup=true`.

## Best Possible Solution

Keep Brave’s manifest on the contract-based `onStartup:false` path, rely on the current main startup/runtime discovery fixes, and ship that behavior in the next release.

## Reproduction Assessment

Yes, at source level: the reporter supplied the v2026.5.12 Brave CLI-vs-agent failure, and the release tag lacks the current configured web-search startup path. I did not run a live Brave API/Gateway reproduction in this read-only review.

## Solution Assessment

Yes. The main-branch fix is the maintainable path because it makes provider contracts drive startup/runtime loading instead of forcing Brave to become a broad startup plugin.
