## Summary

Keep open: current main and the latest release still set the progress draft gate to started before the async start hook completes, and the timer path suppresses startup rejection while leaving later calls believing streaming is active. This is a narrow, source-proven SDK state-machine bug with no exact duplicate found.

## What This Changes

The issue reports that channel progress streaming marks a stream as started before an asynchronous onStart hook has completed.

## Best Possible Solution

Fix the shared SDK gate so hasStarted only reflects a completed start hook, failed starts can retry, and the existing delayed-start behavior remains covered by focused regression tests.

## Reproduction Assessment

Yes, from source inspection: first work schedules the timer, the timer calls a rejecting onStart through a swallowed catch, and later noteWork returns true because started stayed true. I did not execute the repro because this review was constrained to read-only inspection.

## Solution Assessment

Yes, the suggested direction is the narrow maintainable fix: mark started only after onStart resolves or reset the gate on rejection while preserving delayed-start and single-flight behavior.
