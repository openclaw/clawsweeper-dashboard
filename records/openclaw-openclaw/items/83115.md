## Label Justifications

- `P2`: The bug is a normal-priority SDK state-machine defect with limited but real channel progress-draft impact.
- `impact:session-state`: The helper can keep `hasStarted` true after failed startup, leaving channel progress state stale.
- `impact:message-loss`: Callers can believe a progress draft stream is active when startup failed, which can suppress or misroute progress/final-message handling.

## Summary

Keep open: current main and v2026.5.18 still have the stale progress-draft started-state path, and https://github.com/openclaw/openclaw/pull/83128 is an open closing PR rather than a merged fix.

## What This Changes

The issue reports that `createChannelProgressDraftGate` marks progress draft startup as active before async `onStart` completes, so failed startup can leave stale active state.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83128, or replace it with the same narrow SDK reset plus regression tests, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, source-level: first work schedules the timer, the timer starts and swallows a rejecting `onStart`, and later `noteWork()` returns true because `started` remains true. I did not execute the repro because this review was read-only.

## Solution Assessment

Yes, the narrow reset-on-rejection approach in the linked PR is the right shape because it preserves the single-flight startup guard while clearing stale active state and allowing retry.
