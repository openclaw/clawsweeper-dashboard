## Label Justifications

- `impact:message-loss`: The stale active flag can make channel code believe a progress draft is live when startup failed, suppressing or misrouting follow-up progress delivery.

## Summary

Keep open: current main and the latest release still leave the progress draft gate marked as started before async startup settles, and the linked fix PR is still open with a closing reference. This issue should stay open until that PR or an equivalent fix merges.

## What This Changes

Fix `createChannelProgressDraftGate` so rejected async `onStart` calls do not leave progress-draft streaming marked as started.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/83128, or an equivalent reset-after-rejection fix with the same regression coverage, then close this issue as implemented.

## Reproduction Assessment

Yes. The source path is high-confidence: first work schedules the timer, the timer calls `start()`, `start()` sets `started = true` before `onStart`, the timer catches rejection, and later `noteWork()` returns true from the stale flag.

## Solution Assessment

Yes. The linked PR’s approach of resetting `started` and `startPromise` on `onStart` rejection is the narrow maintainable fix while preserving the existing single-flight startup behavior.
