## Summary

Keep open: current main and the latest release still mark the progress draft gate as started before async startup completes, and the timer path swallows startup rejection while leaving later calls believing the draft is active. This is a narrow SDK state-machine bug with clear source proof and no canonical duplicate found.

## What This Changes

The issue reports that channel progress streaming sets its started flag before async onStart has completed, allowing failed starts to look active.

## Best Possible Solution

Fix createChannelProgressDraftGate so failed onStart attempts do not leave the gate permanently active, with regression coverage for timer rejection, retry, and successful single-flight startup.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction: first work schedules the timer, the timer calls a rejecting onStart through a swallowed catch, and later noteWork returns true because started stayed true. I did not execute the repro because this review was constrained to read-only inspection.

## Solution Assessment

Yes, the suggested direction is the right narrow fix: either mark started only after onStart resolves or reset/clear the gate on rejection while preserving the existing delayed-start and single-flight semantics.
