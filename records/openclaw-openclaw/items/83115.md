## Label Justifications

- `P2`: The bug is a narrow shared SDK state bug with plausible multi-channel impact and a focused fix path, but it is not an emergency runtime outage.
- `impact:message-loss`: A failed progress draft startup can make callers believe a stream is live when it is not, which can suppress or mis-handle progress/message delivery paths.
- `impact:session-state`: The stale `started` flag leaves the progress draft gate in an incorrect state after failed async startup.

## Summary

Keep open: current main still has the source-level progress draft gate bug, and the linked open fix PR uses closing syntax for this issue, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue reports that the shared channel progress draft gate marks streaming as started before async `onStart` finishes, leaving later calls to believe a failed stream is active.

## Best Possible Solution

Review and land or reject the linked focused PR; if it lands, close this issue as implemented by the merged SDK reset/retry behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: delayed work calls `start()`, `onStart` rejects through a swallowed timer catch, and later `noteWork()` still returns true because `started` was never reset.

## Solution Assessment

Yes. The narrow maintainable fix is to preserve single-flight startup while resetting the gate on `onStart` rejection, with focused tests for timer rejection, propagated rejection, and retry; the linked PR appears to target exactly that path.
