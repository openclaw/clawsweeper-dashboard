## Label Justifications

- `P1`: The report describes WebChat losing active conversation continuity after a user Stop action, which breaks a real agent workflow for long-running sessions.
- `impact:session-state`: The core failure is drift between the logical WebChat session mapping and persisted session id/restart-recovery state.

## Summary

Keep open: current main has guardrails showing RPC abort should be non-restart cleanup, but I could not prove the reported abort-plus-compaction restart-recovery marker race is fixed end to end. The closest prior compaction fix overlaps but does not cover this exact `restartRecoveryRuns` path.

## What This Changes

The issue reports that clicking Stop in WebChat while compaction is active can remap the next message to a new session and lose conversation context.

## Best Possible Solution

Add a focused fix and regression test proving user-initiated WebChat aborts cannot leave or recreate restart-recovery state for the same run, and cannot remap `agent:<id>:main` unless a real gateway restart is in progress.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not live; source inspection shows the plausible race between RPC abort cleanup and restart-recovery candidate creation, but I did not reproduce the full WebChat/compaction timing on current main.

## Solution Assessment

Is this the best way to solve the issue? Yes for triage: keep it open and repair the abort/lifecycle/restart-recovery invariant directly, rather than adding config or treating the prior compaction fix as sufficient.
