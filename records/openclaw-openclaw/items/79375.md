## Summary

Keep open: current main still has the canonical user-systemd plus system-systemd gateway gap, so a stale user unit can remain loaded/restartable while the system-unit blocker and extra-service scan miss the duplicate pair.

## What This Changes

During Linux upgrades, detect stale user-level systemd gateway units left beside newer system-level units and retire them safely to stop restart cascades.

## Best Possible Solution

Add canonical user+system systemd duplicate detection in doctor/status, route cleanup through existing confirmed service-repair policy, and harden stale-pid killing only if the managed-peer loop remains after detection.

## Reproduction Assessment

Yes, source-reproducible with high confidence: create a canonical user unit plus a canonical system unit for the same profile/port, then inspect the current doctor/status paths. The loaded user unit skips the system-unit blocker, and the deep extra-service scan filters both canonical managed units.

## Solution Assessment

No, the default solution should not silently archive and remove user units. The narrower maintainable fix is explicit duplicate detection plus confirmed cleanup through the existing service repair policy.
