## Summary

Keep open: current main still has a source-reproducible duplicate-supervisor failure. Service-mode gateway startup still runs stale PID cleanup before bind, and that cleanup can kill a peer OpenClaw gateway PID on the same port; the existing system-service guard only blocks installing a new user service when no user service is installed, so it does not retire an already-present stale user unit.

## What This Changes

The issue requests Linux upgrade/setup cleanup so stale user-level systemd gateway units are disabled or archived when a system-level gateway unit owns the same gateway port.

## Best Possible Solution

Make Linux gateway supervision treat duplicate user/system units as a conflict: skip or fail stale PID killing against peer managed gateways, surface clear doctor/status guidance, and offer prompt-gated archive-and-disable cleanup for the stale unit.

## Reproduction Assessment

Yes, from source inspection: two separate service-marked OpenClaw gateway processes on the same port can each run pre-bind stale PID cleanup, and the cleaner excludes only self/ancestors, not a sibling systemd-managed gateway. I did not execute a live WSL2/systemd repro in this read-only pass.

## Solution Assessment

The reported archive/remove direction is useful, but the narrowest robust fix should also stop the stale-PID cleanup from SIGTERMing a peer managed gateway and then use doctor/status to guide safe duplicate-unit retirement.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79375. On Linux, duplicate user-level and system-level OpenClaw gateway systemd units for the same profile/port can SIGTERM each other because service-mode startup calls `cleanStaleGatewayProcessesSync(port)` before bind, and `src/infra/restart-stale-pids.ts` treats any peer OpenClaw gateway listener outside the current process/ancestor chain as stale. Implement a conservative core fix: do not kill a peer OpenClaw gateway that appears to be another active supervisor-managed gateway on Linux; fail or skip cleanup with clear diagnostics instead. Also improve doctor/status duplicate-service handling so an existing user unit plus system unit for the same profile/port is surfaced, and in repair mode offer safe prompt/policy-gated cleanup such as disable/archive/remove of the stale user unit. Do not add a config option, do not remove support for intentional multiple gateways on isolated ports, and do not mutate system services without the existing doctor repair policy/prompt. Add a user-facing changelog entry because this changes Linux gateway service behavior.
