## Summary

Keep open: the report matches a real source-level gap in Linux gateway duplicate-service handling. Current main has safeguards for system-level services only when the user service is missing, but a default user unit plus a default system unit can bypass the duplicate scan.

## What This Changes

Request cleanup for stale Linux user-level systemd gateway units when a newer or externally managed system-level gateway unit is present.

## Best Possible Solution

Teach doctor/status to detect the canonical user+system systemd pair for the same profile/port, surface an explicit cleanup path, and only stop/disable the losing unit through confirmed repair policy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a live WSL2 run in this read-only review, but the source path is clear: a loaded user unit skips the system-unit blocker, and the deep extra-service scanner filters canonical `openclaw-gateway.service` units.

## Solution Assessment

Is this the best way to solve the issue? The exact suggested upgrade-time auto-removal is too aggressive; the maintainable fix is duplicate detection plus confirmed doctor cleanup, with stale-process behavior made aware of a peer managed instance if needed.
