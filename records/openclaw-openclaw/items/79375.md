## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79375. On Linux/WSL2, when a canonical user-level `openclaw-gateway.service` and a canonical system-level `openclaw-gateway.service` exist for the same profile/port, current doctor/status scanning can miss the pair and service-mode stale-PID cleanup can kill the peer gateway. Add conservative detection for the canonical user+system systemd pair, surface it in doctor/status with explicit cleanup guidance, and use existing service repair policy/confirmation before stopping/disabling/removing or archiving any unit. Do not change the default Linux install model from systemd user services, do not add a new config option, and do not silently delete units. If touching stale-PID cleanup, make it avoid terminating an actively managed sibling systemd instance or fail closed with a clear lock/exit-78 path. Add focused regression coverage and a user-facing changelog entry.
