## Summary

Keep open: current main still has the reported Linux duplicate-service gap. Doctor blocks installing a user service only when the user unit is missing, while the extra-service scan still filters the canonical user `openclaw-gateway.service`, so an existing canonical user unit plus system unit can remain invisible to cleanup.

## What This Changes

The issue asks OpenClaw to handle upgrades where a legacy user-level `openclaw-gateway.service` remains alongside a newer system-level systemd gateway unit and causes restart loops.

## Best Possible Solution

Detect the canonical user+system systemd gateway pair for the same profile/port, show an explicit diagnostic, and route cleanup through prompted doctor repair that archives or disables only the losing stale user unit.

## Reproduction Assessment

Yes at source level: a loaded canonical user unit keeps doctor out of the system-unit guard, and the extra-service scanner filters that same canonical user unit. I did not run a live WSL2/systemd setup in this read-only review.

## Solution Assessment

No for unconditional upgrade-time deletion; the safer fix is explicit duplicate detection plus confirmed doctor cleanup, with stale-PID handling made aware of managed peer services if needed.
