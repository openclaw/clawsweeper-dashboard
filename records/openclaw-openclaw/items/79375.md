## Summary

Keep open: current main still has a source-level gap for the reported Linux user-systemd plus system-systemd gateway pair, and the reporter confirmed that removing the stale user unit stopped the stale-process loop.

## What This Changes

The issue asks OpenClaw to detect and retire stale user-level systemd gateway units left behind when a system-level unit is installed during upgrade.

## Best Possible Solution

Add explicit canonical user+system systemd duplicate detection in doctor/status, route cleanup through existing confirmed service-repair policy, and harden stale-pid startup only if the managed-peer loop can still occur.

## Reproduction Assessment

Yes, source-reproducible with high confidence: create both canonical user and system systemd gateway units for the same profile/port, then inspect current doctor/status paths. The loaded user unit skips the system-unit blocker, and the extra-service scan filters canonical OpenClaw units.

## Solution Assessment

No, the reporter's archive-and-remove default is too aggressive by itself. The narrower maintainable fix is duplicate detection plus confirmed cleanup through the existing repair policy, with stale-process behavior hardened around sibling managed instances if needed.
