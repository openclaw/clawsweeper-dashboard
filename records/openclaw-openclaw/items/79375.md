## Summary

Keep open: current main still has a source-level gap for the reported Linux user+system systemd duplicate. The canonical user unit is filtered out of extra-service scans, while the system-unit blocker only runs when the user service is not loaded, so the reported dual-supervisor state can still escape doctor/status handling.

## What This Changes

The issue asks OpenClaw upgrade/doctor handling to retire or clearly flag a stale user-level systemd gateway unit when a newer system-level unit is installed.

## Best Possible Solution

Teach doctor/status to detect the canonical user+system systemd pair for the same profile/port, report the conflict clearly, and only stop/disable/archive the losing unit through confirmed service-repair policy.

## Reproduction Assessment

Yes at source level: a canonical user unit is ignored by the extra-service scanner, and the system-unit blocker only runs when the user service is not loaded. The OP's follow-up confirms the loop stopped after the user unit was disabled and removed.

## Solution Assessment

Yes for a focused repair: duplicate-scope detection plus explicit cleanup guidance is the narrow maintainable path. The suggested automatic archive/remove behavior should be kept behind confirmed doctor/service-repair policy rather than silently applied during normal startup.

## Repair Work Prompt

Fix the source-reproducible Linux gateway duplicate-service bug reported in https://github.com/openclaw/openclaw/issues/79375. Current main ignores the canonical `~/.config/systemd/user/openclaw-gateway.service` in `findExtraGatewayServices`, and `doctor-gateway-daemon-flow` checks system-level units only when the user service is not loaded. Add a focused doctor/status detection path for a canonical user+system OpenClaw gateway pair for the same profile/port, emit clear cleanup guidance, and only perform stop/disable/archive/removal through existing confirmed doctor/service-repair policy. Do not silently delete user units on startup, do not add a new config flag, and do not take over broad system-scoped systemd lifecycle support that is being discussed in https://github.com/openclaw/openclaw/pull/57276. Add regression coverage for both scopes existing at once, including a loaded user unit plus `/etc/systemd/system/openclaw-gateway.service`. Include a user-facing changelog fix entry if behavior changes.
