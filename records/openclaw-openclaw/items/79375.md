## Summary

Keep open: current main still has a source-level gap for canonical user and system systemd gateway units coexisting on the same port, and no open PR was found that already resolves it.

## What This Changes

Handle upgrades where a stale user-level systemd gateway unit coexists with a newer system-level unit so the two supervisors do not fight over the same gateway port.

## Best Possible Solution

Detect the canonical user-plus-system systemd gateway pair for the same profile/port in doctor/status, show an explicit cleanup path, and only stop or disable the losing unit through the existing confirmed repair policy.

## Reproduction Assessment

Yes for a source-level reproduction: current main filters the canonical user unit from extra scans and only runs the system-unit blocker when the user service is missing. I did not establish a live WSL2 reproduction in this read-only pass.

## Solution Assessment

No, not exactly as suggested: automatic upgrade-time removal is too aggressive for externally managed supervisors. The narrower maintainable fix is duplicate detection plus prompted doctor cleanup, with stale-process cleanup made peer-aware if needed.
