## Summary

Keep open: current main still has a source-level gap for the both-scopes Linux systemd case. The existing guard only blocks installing a user service when no user service is loaded, while the duplicate-service scanner filters the canonical user and system unit names that this report describes.

## What This Changes

Detect and retire or warn about stale user-scoped systemd gateway units when a system-scoped gateway unit owns the same profile and port.

## Best Possible Solution

Add canonical user-plus-system systemd duplicate detection to doctor/status with a confirmed cleanup path, and make service-mode stale-PID cleanup fail clearly when a peer managed unit owns the port.

## Reproduction Assessment

Yes, source-level. A loaded canonical user unit plus a canonical system unit on the same port bypasses the existing `!loaded` system-unit blocker and is filtered out of the extra-service scanner; the stale-PID cleanup then has a path to kill the peer listener.

## Solution Assessment

Partly. The requested automatic archive/remove flow is too aggressive by itself; the maintainable fix is explicit duplicate detection plus confirmed cleanup under the existing service repair policy, with safer stale-process behavior for peer managed units.
