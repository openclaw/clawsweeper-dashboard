## Summary

Keep open: current main still has the source-level gap for a canonical user-systemd plus system-systemd gateway pair. The shipped guard only blocks installing a second user service when the user service is missing, while the extra-service scan intentionally filters canonical OpenClaw gateway units, so the reporter's loaded/restartable duplicate can still escape detection.

## What This Changes

The issue requests upgrade/setup cleanup for stale user-level systemd gateway units when a system-level unit is present, plus safer stale-process handling for that duplicate-service loop.

## Best Possible Solution

Add canonical user+system systemd duplicate detection in doctor/status, route any retirement through the existing confirmed service-repair policy with backup output, and harden stale-pid cleanup only as needed after duplicate supervision is surfaced clearly.

## Reproduction Assessment

Yes. Source-reproducible with high confidence: create a canonical user gateway unit and a canonical system gateway unit for the same profile/port, then inspect doctor/status paths; the loaded user unit skips the system-unit blocker and the extra scan filters both canonical managed units.

## Solution Assessment

No. The reporter's archive-and-remove default is too aggressive as a first repair; the narrower maintainable fix is explicit duplicate detection plus confirmed cleanup through existing service repair policy, with stale-process hardening only if the peer-managed loop remains.
