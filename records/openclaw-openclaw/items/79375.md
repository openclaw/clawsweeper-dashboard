## Summary

Keep open: current main still has a source-level gap for a canonical user and system systemd gateway unit coexisting after upgrade. The earlier shipped guard only blocks installing a user unit when the user service is missing; it does not catch the reported case where the stale user unit is still loaded or restartable.

## What This Changes

Detect and retire or clearly warn about stale user-scoped Linux systemd gateway units when a system-scoped unit owns the same gateway after upgrade.

## Best Possible Solution

Add canonical user+system systemd duplicate detection to doctor/status, then offer a confirmed repair that stops/disables and optionally backs up the losing unit while making startup avoid silently killing a peer managed instance.

## Reproduction Assessment

Yes, source-reproducible: a loaded canonical user unit skips the system-unit blocker, and the extra-service scanner filters the same canonical user unit. I did not run a live WSL2/systemd reproduction in this read-only review.

## Solution Assessment

No for the exact automatic archive-and-remove suggestion as written; the safer maintainable fix is explicit duplicate detection plus confirmed doctor cleanup under the existing repair policy. The stale-process path should also fail clearly or spare a peer managed instance rather than silently SIGTERMing it.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79375 with a narrow Linux systemd service-lifecycle PR. Current main misses the case where both the canonical user unit `~/.config/systemd/user/openclaw-gateway.service` and a system unit such as `/etc/systemd/system/openclaw-gateway.service` exist: `maybeRepairGatewayDaemon` only blocks system units when the user service is not loaded, and `findExtraGatewayServices` filters the canonical user unit. Add detection for the canonical user+system pair for the same profile/port in doctor/status or the existing service scan, surface a clear warning, and under existing confirmed service-repair policy offer cleanup that stops/disables and optionally backs up/removes the losing stale user unit. Do not silently delete or disable units without confirmation/policy, do not add a new config flag, and do not broaden into the separate update restart-scope issue unless a tiny shared helper is needed. Add regression coverage for both scopes existing at once and for the stale-process cleanup behavior if the fix touches it. Add a single-line user-facing CHANGELOG.md entry if behavior changes. Validate with the focused tests and formatter below, and prefer WSL2/Linux systemd live proof or Testbox proof showing the duplicate is detected and the cleanup path resolves the loop.
