## Summary

Keep open: current main and v2026.6.8 still prefer the user-scope systemd unit when same-name user and system units coexist, and the remaining implementation path is the open closing PR rather than code already on main.

## What This Changes

The issue asks OpenClaw setup/doctor to detect stale user-level Linux systemd gateway units after upgrade and retire them safely so user- and system-scope services do not restart-kill each other.

## Best Possible Solution

Land one focused systemd/doctor repair that detects same-name user+system gateway units, retires only the stale redundant user unit when system ownership is authoritative, and leaves intentional multi-gateway setups untouched.

## Reproduction Assessment

Yes, source-reproducible: current main has both source and a regression test showing the resolver returns the user scope when same-name user and system units exist. I did not live-run systemd in this read-only review.

## Solution Assessment

Yes in direction: the fix belongs in Linux systemd service ownership and doctor cleanup, not in Telegram or stale-process detection alone. The exact cleanup guard still needs maintainer review through the open PR.
