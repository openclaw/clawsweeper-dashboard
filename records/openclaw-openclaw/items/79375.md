## Summary

Keep open. Current main still has a source-level path where supervised gateway startup kills gateway PIDs on the port before a health or sibling-systemd check, and the existing doctor guard only prevents installing a new user unit when no user unit is loaded; the reported stale user unit plus system unit state is not fully covered by the earlier closed report or the open systemd-scope PRs.

## What This Changes

The issue asks OpenClaw upgrade/setup to retire a stale Linux user-level `openclaw-gateway` systemd unit when a system-level unit owns the gateway, avoiding a dueling-service restart cascade.

## Best Possible Solution

Implement an explicit Linux service migration/repair path that detects canonical user and system `openclaw-gateway` units for the same profile/port, disables and archives the stale user unit with clear output and appropriate confirmation, and hardens service-mode startup so a healthy supervised gateway is not killed as stale.

## Reproduction Assessment

Yes, at source level: with both services carrying OpenClaw service markers, current main enters service-mode stale cleanup before bind and the cleanup kills gateway PIDs on the configured port without checking sibling systemd ownership. I did not run a live WSL2 reproduction, but the reporter supplied concrete logs and a verified cleanup workaround.

## Solution Assessment

Unclear as written. The archive-and-remove direction is likely right, but the safest maintainable fix needs owner-approved doctor/setup migration semantics plus coordination with the open systemd-scope PRs and a startup guard for healthy supervised listeners.
