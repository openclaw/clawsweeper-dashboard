## Summary

Keep open: current main still has a source-level gap for the reported Linux user+system systemd split-brain. The system-service guard only runs when the user service is missing, while the deep duplicate-service scanner filters out the canonical user unit name, so an old `openclaw-gateway.service` user unit can coexist with the system unit and stay a restart footgun.

## What This Changes

Detect and retire or report a stale user-level systemd gateway unit after upgrade to a system-level unit so duplicate supervisors do not fight over the same port.

## Best Possible Solution

Add canonical user+system systemd duplicate detection for the same profile/port, surface it in doctor/status, and provide a confirmed `doctor --fix` cleanup path that archives or disables the losing user unit without changing normal installer defaults.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: create both canonical user and system `openclaw-gateway.service` units and current main will not report the canonical user unit through the extra-service scan while the system-service blocker is skipped once the user service is loaded.

## Solution Assessment

Is this the best way to solve the issue? Partly: the reporter's archive-then-remove suggestion is user-friendly, but the maintainable version should run through doctor/status detection and confirmed repair policy rather than unconditional upgrade-time deletion.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79375. Current main misses the upgrade state where both canonical Linux systemd units exist: `~/.config/systemd/user/openclaw-gateway.service` and `/etc/systemd/system/openclaw-gateway.service`. Add detection for the canonical user+system pair for the same profile/port, surface it in `openclaw doctor --deep` / gateway status as appropriate, and provide a confirmed `doctor --fix` cleanup path that stops/disables and preferably archives the stale user unit when the system unit clearly owns the gateway. Do not silently delete custom/external units, do not change the default installer model, and only touch stale-PID cleanup if duplicate-service detection cannot prevent the peer-kill loop. Add regression tests for both scopes existing at once, including the case where the user unit is loaded so the existing `!loaded` system-service blocker is bypassed. Add a user-facing changelog entry if behavior changes.
