## Summary

Keep open: current main still has a real Linux systemd duplicate-service gap. The code blocks duplicate user-service installation only when the user service is missing, while the extra-service scanner filters canonical `openclaw-gateway.service` units, so a stale canonical user unit plus a system unit can still evade diagnostics.

## What This Changes

The issue requests upgrade cleanup and detection for stale user-level systemd gateway units when a newer install uses a system-level unit.

## Best Possible Solution

Add canonical user+system systemd duplicate detection, surface a clear doctor/status cleanup path, and only stop/disable or archive the losing user unit through confirmed repair policy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes as a source-level reproduction: current main can miss a canonical user+system unit pair, and the reporter provided concrete WSL2 steps and post-workaround evidence; I did not run a live WSL2 systemd reproduction.

## Solution Assessment

Is this the best way to solve the issue? Keeping this open is the right outcome. The narrow maintainable repair is duplicate detection plus confirmed doctor cleanup, not unconditional upgrade-time deletion of user units.
