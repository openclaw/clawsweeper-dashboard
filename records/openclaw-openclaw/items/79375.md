## Summary

Keep open: current main still has the reported Linux systemd split-brain gap. The default user unit and system unit can coexist without being reported as an extra service, and the startup stale-process cleanup can still kill a sibling supervised gateway on the same port.

## What This Changes

The issue asks OpenClaw to detect and retire or clearly report a stale user-level Linux systemd gateway unit after migration to a system-level unit.

## Best Possible Solution

Add canonical user+system systemd duplicate detection for the same gateway profile/port, report it in doctor/status, and offer confirmed doctor cleanup that backs up/removes the losing user unit without changing the broader system-scoped lifecycle policy.

## Reproduction Assessment

Yes, source-reproducible: a loaded canonical user unit bypasses the system-unit blocker, the extra-service scanner filters canonical units, and service-mode stale cleanup can kill the sibling gateway PID. I did not perform a live WSL2 systemd run in this read-only review.

## Solution Assessment

The exact archive-and-remove-on-upgrade suggestion is too aggressive by itself. The maintainable fix is duplicate detection plus explicit, confirmed doctor cleanup, with stale-process handling aware enough not to silently fight another managed instance.
