## Summary

Keep open: the report matches a real Linux gateway split-brain gap on current main. Doctor can block a system unit only when the user service is missing, while the extra-service scanner intentionally filters the canonical user unit, so the stale user+system pair described by the OP is not reliably surfaced or cleaned up.

## What This Changes

Detect and retire or warn about a stale user-level systemd gateway unit left behind when a system-level unit now owns the same gateway profile and port.

## Best Possible Solution

Teach doctor/status to detect the canonical user+system systemd pair for the same profile/port, surface a clear warning, and offer confirmed backup plus disable/remove through the existing service repair policy.

## Reproduction Assessment

Yes, as a source-level reproduction: create both canonical user and system `openclaw-gateway.service` units for the same port, then restart the user unit. The current code paths explain why doctor misses the pair and why stale-PID cleanup can produce the reported SIGTERM cascade.

## Solution Assessment

No for unconditional upgrade-time removal as proposed; the safer maintainable fix is explicit duplicate detection with confirmed doctor cleanup and regression coverage, using existing repair policy rather than adding a new migration mode.
