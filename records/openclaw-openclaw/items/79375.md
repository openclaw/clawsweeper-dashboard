## Summary

Keep open: current main still has a source-level gap for a canonical Linux user systemd gateway unit coexisting with a canonical system unit, matching the OP's dueling-service diagnosis and workaround.

## What This Changes

The issue asks OpenClaw to detect and retire a stale user-level Linux systemd gateway unit during upgrade so it cannot race a newer system-level gateway unit on the same port.

## Best Possible Solution

Teach doctor/status to detect the canonical user+system systemd gateway pair for the same profile/port, surface an explicit cleanup path, and only retire the losing unit through existing confirmed repair policy with backup output.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: create both canonical Linux systemd units for the same profile/port while the user unit is loaded, then doctor skips the system-unit blocker and the extra-service scanner ignores the default user unit; I did not run a live WSL2 systemd reproduction in this read-only pass.

## Solution Assessment

Is this the best way to solve the issue? The exact upgrade-time auto-archive/remove request is more aggressive than necessary; the narrower maintainable fix is duplicate detection plus confirmed cleanup, with stale-process peer handling only if the startup path is touched.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79375. On Linux, a canonical user `~/.config/systemd/user/openclaw-gateway.service` and canonical system `/etc/systemd/system/openclaw-gateway.service` for the same profile/port can coexist; current doctor/status paths miss the pair when the user service is loaded, and service-mode stale-PID cleanup can amplify the conflict. Keep OpenClaw's default user-service model. Add canonical user+system duplicate detection in the daemon inspection/doctor/status flow, surface explicit cleanup guidance, and if cleanup is implemented use existing confirmed service repair policy with backup/output rather than silent startup deletion. Do not add a config option and do not silently remove operator-managed system units. Add regression coverage for loaded user+system canonical units, the existing missing-user-service blocker, and status/doctor deep reporting if touched. Add a user-facing CHANGELOG.md fix entry.
