## Summary

Keep open: current main still has a source-level gap for the canonical user-systemd plus system-systemd gateway pair, and the stale-pid cleanup path can still kill a sibling managed gateway before the normal bind/lock guard can stop the duplicate cleanly.

## What This Changes

The issue reports that upgrading from a legacy user-level systemd gateway unit to a system-level unit can leave both supervisors active and create a restart loop.

## Best Possible Solution

Add canonical user+system systemd duplicate detection in doctor/status, route cleanup through the existing confirmed service-repair policy, and harden stale-pid cleanup only as needed to avoid killing sibling managed gateways.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to model canonical user and system `openclaw-gateway.service` units for the same profile/port: the loaded user-service path skips the system-unit blocker, the deep extra-service scan filters canonical units, and service-mode startup can kill the sibling gateway PID before bind.

## Solution Assessment

Partly. Detecting and retiring duplicate user/system systemd supervision is the right boundary, but unconditional archive-and-remove should be replaced with explicit detection plus confirmed cleanup that respects `OPENCLAW_SERVICE_REPAIR_POLICY=external`.
