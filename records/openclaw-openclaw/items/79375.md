## Label Justifications

- `P1`: The report describes an upgrade-triggered gateway restart cascade that makes the Telegram channel stop responding for real Linux/WSL2 users.
- `impact:crash-loop`: Both systemd units can continuously terminate and restart gateway processes on the same port.
- `impact:message-loss`: The gateway churn makes Telegram stop responding and can hit Telegram API rate limits during the loop.

## Summary

Keep open: current main still does not retire the stale user-level `openclaw-gateway.service` when a system-level OpenClaw unit exists. Doctor only blocks creating a new user unit when the user unit is missing, and the extra-service/legacy cleanup paths ignore canonical OpenClaw units, so the reported dueling-supervisor crash loop remains source-reproducible.

## What This Changes

Requests Linux/WSL2 upgrade cleanup that detects and retires a stale user-level systemd gateway unit when a system-level OpenClaw gateway unit takes over.

## Best Possible Solution

Extend doctor/setup service repair to detect matching canonical user and system systemd gateway units, back up and stop/disable/remove the stale user unit under the existing repair policy, reload systemd, and leave one intended supervisor with WSL2/systemd proof.

## Reproduction Assessment

Yes. The issue gives a concrete WSL2 upgrade sequence, and current source inspection shows the both-units-present state is neither cleaned up nor reported by the existing doctor paths; I did not run live systemd proof in this read-only pass.

## Solution Assessment

Yes. Retiring the duplicate user-level unit during the existing doctor/setup service repair flow is the narrowest maintainable fix, with a secondary guard so stale PID cleanup does not kill a sibling managed gateway.
