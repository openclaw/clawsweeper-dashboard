## Summary

Keep open: current main still has a source-level gap for the canonical stale user-systemd plus system-systemd gateway pair, and the issue has recent reporter confirmation that removing the old user unit stops the loop.

## What This Changes

The issue asks OpenClaw to detect and retire stale user-level Linux systemd gateway units after an upgrade to a system-level unit so duplicate supervisors cannot restart and kill each other.

## Best Possible Solution

Add canonical user+system systemd duplicate detection in doctor/status and prevent service-mode stale-pid cleanup from killing a peer systemd-managed OpenClaw gateway; retire old units only through confirmed repair policy.

## Reproduction Assessment

Yes, source-reproducible with high confidence: configure a canonical user systemd unit plus a canonical system unit for the same profile/port, then follow the current doctor/status and service-mode startup paths. The loaded user unit bypasses the system-unit blocker, the extra-service scan filters both canonical managed units, and service-mode startup still kills same-port gateway PIDs before lock recovery.

## Solution Assessment

No, the reporter's default archive-and-remove suggestion is likely too aggressive as an unattended behavior. The safer fix is explicit duplicate detection plus confirmed cleanup through the existing service repair policy, paired with a stale-pid guard that does not kill a managed peer service.
