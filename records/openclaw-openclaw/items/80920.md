## Summary

Keep open: current main and the latest release still have an unbounded IPv6 loopback bind probe, while the linked timeout fix remains open and unmerged. The stale label is not enough to close a source-reproducible gateway startup hang with an active paired PR.

## What This Changes

The issue asks for `canBindToHost("::1")` to time out so gateway startup can continue with IPv4-only binding when IPv6 loopback probing stalls on macOS Big Sur.

## Best Possible Solution

Review, update if needed, and land the linked timeout fix or an equivalent focused gateway patch with regression coverage for a stalled bind probe.

## Reproduction Assessment

Yes, source-reproducible: if `net.createServer().listen(0, "::1")` never emits `listening` or `error`, `canBindToHost` never resolves and gateway startup remains blocked while resolving listen hosts. I did not live-reproduce the macOS Big Sur networking behavior in this environment.

## Solution Assessment

Yes, a bounded timeout inside `canBindToHost` is the narrow maintainable fix because it preserves successful IPv6 probing and falls back to IPv4-only binding only when the probe stalls.
