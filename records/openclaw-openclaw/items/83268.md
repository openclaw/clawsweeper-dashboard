## Summary

Keep open: current main and the latest release have the new Anthropic API-key setup path, but source inspection still shows no `claude` to `anthropic` auth alias or doctor repair for already-stored `claude:manual` profiles.

## What This Changes

The issue asks OpenClaw to stop or repair Anthropic API keys being stored as `claude:manual` so `anthropic/*` models use the pasted key instead of falling back to Claude CLI OAuth.

## Best Possible Solution

Add a maintainer-approved doctor repair or diagnostic for shipped `claude:manual` Anthropic profiles while keeping new API-key setup canonical under `anthropic`.

## Reproduction Assessment

Yes, at source level for existing stored state: a `claude:manual` profile with `provider: "claude"` will not be returned for `anthropic` profile lookup without an alias or migration. I did not run the live wizard with a real Anthropic key.

## Solution Assessment

Unclear as a final implementation direction: current new-key setup is canonical under `anthropic`, but existing orphaned state still needs a maintainer-approved compatibility repair. A doctor repair or explicit diagnostic looks safer than silently adding a broad runtime alias unless maintainers choose that behavior.
