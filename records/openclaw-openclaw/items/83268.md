## Summary

Keep open: current main and v2026.6.8 have the canonical Anthropic API-key path, but source and release inspection still show no `claude -> anthropic` auth alias or doctor repair for already-stored `claude:manual` profiles.

## What This Changes

The issue asks OpenClaw to stop orphaning pasted Anthropic credentials under `claude` and make them usable for `anthropic/*` model auth.

## Best Possible Solution

Add a maintainer-approved doctor repair or diagnostic for shipped `claude:manual` Anthropic profiles while keeping new API-key setup canonical under `anthropic`.

## Reproduction Assessment

Yes at source level: a stored `claude:manual` profile with `provider: "claude"` will not be returned for `anthropic` profile lookup unless an alias or migration exists. I did not run a live wizard flow with a real Anthropic API key.

## Solution Assessment

Unclear as an implementation direction: the current new-key path is canonical for `anthropic`, but it does not repair existing orphaned `claude:manual` state. A doctor repair or explicit diagnostic looks safer than silently teaching runtime a broad `claude` alias unless maintainers choose that compatibility behavior.
