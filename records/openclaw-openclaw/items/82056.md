## Summary

Keep open: this PR is an active plausible fix for the open MiniMax OAuth refresh bug, but it has blocking CI/type/lint issues, repeats the legacy CN-region refresh risk called out on the earlier unmerged fix PR, and still needs stronger real refresh proof before merge.

## What This Changes

The branch migrates MiniMax OAuth to account/oauth2 endpoints, adds a MiniMax refresh helper and provider refresh hook, and adds refresh tests.

## Best Possible Solution

Keep this PR open and revise it to pass CI, preserve legacy CN/global region safety, use the approved network wrapper, update user-facing endpoint docs, and include redacted real MiniMax refresh proof before landing.

## Reproduction Assessment

Yes, source-level. Current main promises MiniMax OAuth auto-refresh in provider notes but registers no MiniMax refreshOAuth hook, so expired minimax-portal credentials fall through the generic refresh path and can resolve as missing.

## Solution Assessment

No, not yet. A MiniMax-local refreshOAuth hook is the right seam, but this patch needs legacy-region handling, guarded network calls, CI fixes, docs alignment, and real refresh proof before it is the safest implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Preserve legacy MiniMax OAuth regions:** `extensions/minimax/provider-registration.ts:208-211`
  - body: Existing MiniMax OAuth profiles from current releases do not store `enterpriseUrl`; their CN/Global choice is only in config. This fallback treats missing metadata as Global, so upgraded CN profiles can submit refresh tokens to `account.minimax.io` and still fail. Derive legacy region from a reliable source or fail safely with reauth instead of choosing the wrong host.
  - confidence: 0.92
- **[P2] Remove the new raw fetch callsite:** `extensions/minimax/oauth.ts:254`
  - body: The added refresh request calls raw `fetch(...)` from plugin runtime code, and the raw-fetch guard fails on this line. Use the plugin SDK guarded fetch/runtime wrapper for this network call; the same change should keep the shifted legacy OAuth fetch allowlist from breaking CI.
  - confidence: 0.95
- **[P2] Fix the fetch mock tuple cast:** `extensions/minimax/oauth.test.ts:72`
  - body: The CN test's `vi.fn(async () => ...)` callback takes no arguments, so tsgo infers `mock.calls[0]` as `[]` and CI fails TS2352 here. Give the mock a fetch-compatible signature or cast through `unknown` before indexing the URL.
  - confidence: 0.97
- **[P2] Avoid stringifying a possibly object body:** `extensions/minimax/oauth.test.ts:27`
  - body: Oxlint rejects `String(init?.body ?? "")` because it can hide `[object Object]` if the body is not already a string. Narrow the body type before asserting on the form payload so lint passes and the test checks the real request body.
  - confidence: 0.94
- **[P3] Update the MiniMax OAuth docs for account endpoints:** `extensions/minimax/oauth.ts:9-13`
  - body: Changing OAuth login to `account.*` leaves the provider docs saying OAuth authenticates against `api.*` for both regions. Update the user-facing OAuth docs to name the canonical account host while leaving inference/API base URL guidance on `api.*`.
  - confidence: 0.82
