## Summary

Keep open: current main still has a credible poisoned-resume loop because resumed Claude live-session watchdog timeouts surface as timeout FailoverErrors, while persisted CLI session clearing only handles missing transcripts and session_expired errors.

## What This Changes

Request discarding a saved Claude CLI resume session id after the first resumed watchdog FailoverError so the next attempt starts cold instead of reusing poisoned state.

## Best Possible Solution

Add targeted recovery that clears the saved claude-cli session binding after a resumed no-output timeout FailoverError, then let the next turn or retry start cold and persist a clean new session id.

## Reproduction Assessment

No live repro was run, but the source path is high-confidence: a resumed Claude live session can hit the no-output watchdog, throw a timeout FailoverError, and leave the persisted resume binding untouched because only missing-transcript and session_expired paths clear it.

## Solution Assessment

Yes, the narrow maintainable fix is to clear the stored Claude CLI binding for the resumed timeout path; raising the watchdog cap or adding model fallback would not remove the poisoned resume id.
