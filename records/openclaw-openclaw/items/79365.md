## Summary

Keep this issue open: current main still only retries or clears stored CLI sessions for `session_expired` and missing Claude transcript cases, not for a resumed `claude-cli` watchdog timeout. The direct closing PR is still open and unmerged, so the issue should remain as the canonical bug until that fix or an equivalent one lands.

## What This Changes

Discard the stored `claude-cli` resume id after the first resumed watchdog `FailoverError` timeout so the next attempt starts cold instead of reusing poisoned state.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79386 or an equivalent focused fix that clears the poisoned resumed `claude-cli` binding and prevents reuse of the same timed-out transcript, with regression coverage and redacted real-runtime proof.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to seed a stored `claude-cli` binding with a present transcript, have the resumed attempt throw `FailoverError` reason `timeout`, and observe that current main only clears or retries for `session_expired`.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix for the cascade. Lifting watchdog caps or broad model fallback would not stop OpenClaw from reusing the same poisoned resume id.
