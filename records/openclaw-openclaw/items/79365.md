## Summary

Keep open: current main still only clears and retries persisted CLI sessions for `session_expired`, while resumed `claude-cli` watchdog timeouts surface as `FailoverError reason=timeout` and would keep the same stored resume id. An open closing PR targets this exact fix, so the issue should remain open until that PR merges or is rejected.

## What This Changes

Discard a persisted `claude-cli` resume session id after a resumed no-output watchdog `FailoverError` so the next attempt starts cold instead of reusing poisoned state.

## Best Possible Solution

Land a narrow runtime fix that clears the `claude-cli` binding and cold-retries only resumed timeout `FailoverError`s, with regression coverage for resumed timeout, cold timeout, and non-Claude providers.

## Reproduction Assessment

Yes, at source level: seed a stored `claude-cli` session binding and make the resumed attempt fail with `FailoverError reason=timeout`; current main only clears `session_expired`, so the same binding remains available for the next turn. I did not run a live poisoned-transcript gateway repro in this read-only pass.

## Solution Assessment

Yes. The linked PR direction is the narrowest maintainable fix: extend the existing stale-session recovery to resumed Claude timeout failures while preserving cold-timeout and non-Claude behavior.
