## Summary

Keep open: current main still maps the no-output watchdog to a timeout FailoverError while the resume recovery paths only handle expired or missing Claude CLI sessions. A dedicated open fix PR, https://github.com/openclaw/openclaw/pull/79386, is linked with closing syntax and should be reviewed rather than closing this issue early.

## What This Changes

Discard persisted claude-cli resume state after a watchdog FailoverError on a resumed session so the next attempt starts cold instead of replaying poisoned state.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79386 or an equivalent narrow fix that cold-retries and clears persisted claude-cli resume state only for resumed timeout failures.

## Reproduction Assessment

Yes, source-reproducible: a resumed Claude CLI run whose supervisor exits with no-output-timeout becomes FailoverError reason "timeout", while current main's cold-retry and binding-clear logic only handles session_expired or missing transcripts. I did not recreate the real poisoned MCP state live during this read-only pass.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable fix: discard the resumed claude-cli binding after the first resumed timeout, while preserving ordinary cold-start timeout behavior and non-claude-cli providers. The open linked PR is the right implementation path once it has review and real behavior proof.
