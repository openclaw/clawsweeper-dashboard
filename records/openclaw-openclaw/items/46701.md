## Label Justifications

- `P2`: This is a useful Telegram onboarding and authorization-boundary improvement with limited channel scope, but it is not an urgent regression or outage.
- `impact:security`: The requested denial response sits on a Telegram group authorization boundary and could disclose sender identifiers or amplify unauthorized traffic if designed poorly.

## Summary

Keep open: current main and v2026.5.19 still silently skip allowlist-denied Telegram group messages, while safe implementation needs opt-in behavior, cooldowns, admin-contact/template design, and user-ID disclosure decisions.

## What This Changes

Request a Telegram group reply to allowlist-denied senders with admin-contact instructions and the sender’s Telegram user ID.

## Best Possible Solution

Add a Telegram-owned, explicit opt-in unauthorized-group-sender notice that is topic-aware, rate-limited or deduplicated, configurable for admin contact, template, and ID disclosure, documented, and tested separately from DM pairing, audit alerts, and native command denial replies.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. Source inspection gives high confidence in the current behavior: allowlist-denied Telegram group messages return before visible reply dispatch, and existing tests expect zero replies.

## Solution Assessment

No. The unconditional public auto-reply is useful for onboarding, but the safer maintainable shape is an explicit opt-in notice with cooldowns, topic-safe delivery, configurable wording/admin contact, and documented privacy behavior.
