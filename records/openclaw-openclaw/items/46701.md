## Summary

Keep open: current main and the latest release still drop Telegram group messages from senders rejected by groupPolicy/groupAllowFrom, and the requested visible onboarding reply remains a product/security decision rather than an implemented behavior.

## What This Changes

Request to have Telegram reply to non-allowlisted group senders with authorization/admin-contact instructions instead of silently dropping them.

## Best Possible Solution

Add a Telegram-owned, explicit opt-in unauthorized-group-sender notice that is thread-aware, rate-limited, privacy-conscious about sender IDs, documented, and tested while keeping silent fail-closed behavior as the default.

## Reproduction Assessment

Yes, source-level: configure Telegram with groupPolicy allowlist and a non-matching groupAllowFrom/allowFrom sender, then the current handler returns the group-policy-allowlist-unauthorized skip path and tests expect zero replies. I did not run a live Telegram scenario in this read-only review.

## Solution Assessment

Unclear as requested: a default auto-reply is not the safest shape because it can amplify spam and expose identifiers. The better solution is an explicit opt-in, rate-limited, thread-aware notice that preserves the existing authorization boundary.
