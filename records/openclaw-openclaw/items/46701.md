## Summary

Keep open: current main and v2026.6.6 still silently skip Telegram group messages from senders rejected by groupPolicy/groupAllowFrom, and the requested visible onboarding reply remains a security-sensitive feature decision rather than an implemented behavior.

## What This Changes

Request that Telegram send an admin-contact onboarding reply with the blocked sender's Telegram user ID when groupPolicy/groupAllowFrom denies a group sender.

## Best Possible Solution

Add an explicit Telegram-owned opt-in unauthorized-group-sender notice with per-user/chat/topic rate limiting, safe thread-aware delivery, minimal raw-ID disclosure controls, docs, and regression tests while keeping silent fail-closed behavior as the default.

## Reproduction Assessment

Yes for the current behavior: configure Telegram groupPolicy as allowlist, send a group message from a sender not in allowFrom/groupAllowFrom, and current source returns group-policy-allowlist-unauthorized with expectedReplyCount: 0. No live Telegram run was performed in this read-only review.

## Solution Assessment

Unclear as requested: a default auto-reply is not the best safe shape because it can amplify spam and expose identifiers. The maintainable path is an explicit opt-in, rate-limited, thread-aware notice that preserves the existing authorization boundary.
