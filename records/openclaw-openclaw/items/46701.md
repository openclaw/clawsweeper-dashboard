## Summary

Keep open. Current main still silently drops ordinary Telegram group messages from senders rejected by `groupPolicy`/`groupAllowFrom`, while visible authorization replies are limited to native command flows. The request is valid and localized, but the latest discussion reinforces that public user-ID disclosure, opt-in behavior, and rate limiting need maintainer product/security direction before this should become an automated fix PR.

## What This Changes

The issue requests a Telegram group auto-reply for senders rejected by `groupAllowFrom`/`groupPolicy`, telling them to contact a group admin and showing their Telegram user ID.

## Best Possible Solution

Add a Telegram-owned, explicit opt-in unauthorized-group-sender notice with per-group/topic configuration, thread-aware delivery, privacy-safe templating for the sender ID, an admin-contact/message setting, sender/chat cooldowns, docs, and focused tests for ordinary group messages separate from native command auth.

## Reproduction Assessment

Yes. A Telegram group message from sender `999999` under `groupPolicy: "allowlist"` with only `123456789` allowed reaches `group-policy-allowlist-unauthorized`, `shouldSkipGroupMessage` returns true, and the existing table test expects zero replies.

## Solution Assessment

No, not as the unconditional sample behavior. The maintainable solution is an opt-in, rate-limited Telegram config surface with privacy-safe defaults rather than always posting the sender's numeric ID into the group.

## Security Review

Status: needs_attention

Summary: The requested behavior itself needs privacy and spam-control design before implementation, even though no PR patch is under review.

Concerns:

- **[medium] Avoid public Telegram user ID disclosure by default:**
  - body: The sample reply would post the sender's numeric Telegram user ID into a group chat. That can be useful when deliberately configured, but it should not become an unconditional public default.
  - confidence: 0.88
- **[medium] Rate-limit unauthorized group notices:**
  - body: Replying to every rejected group message can spam group chats and reveal allowlist behavior to probing. A safe implementation should be opt-in and deduped or rate-limited.
  - confidence: 0.86
