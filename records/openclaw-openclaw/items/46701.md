## Summary

Keep open. Current main and v2026.6.10 still silently drop non-allowlisted Telegram group senders before reply dispatch; the requested sender-facing onboarding reply remains distinct from the related audit and passive-hook issues, but it needs maintainer product/security choices before implementation.

## What This Changes

The issue requests a Telegram group auto-reply that tells unauthorized group senders they need admin approval and shows their Telegram user ID.

## Best Possible Solution

Add a Telegram-owned, explicit opt-in unauthorized-group-sender notice with bounded rate limiting, topic-aware delivery, safe sender-ID disclosure, admin-contact/template docs, and regression tests while preserving the current silent fail-closed default unless maintainers approve a default change.

## Reproduction Assessment

Yes, at source level. Current main returns before reply dispatch for allowlist-denied Telegram group senders, and existing Telegram tests assert no reply; I did not run a live Telegram group scenario in this read-only review.

## Solution Assessment

No, not as an unconditional default auto-reply. The maintainable path is an explicit, rate-limited Telegram notice or audit/onboarding mode that preserves the current fail-closed authorization boundary and is documented as a privacy/security tradeoff.
