## Summary

Keep open: current main and the latest release still intentionally block Telegram group senders rejected by groupPolicy/groupAllowFrom without the requested onboarding reply, and the remaining work needs product/security decisions rather than cleanup closure.

## What This Changes

Request that Telegram reply to group senders blocked by groupAllowFrom/groupPolicy with authorization instructions and their Telegram user ID instead of silently dropping the message.

## Best Possible Solution

Define an explicit opt-in Telegram-owned unauthorized-group-sender notice that is thread-aware, rate-limited, privacy-conscious, documented, and tested separately from the existing fail-closed authorization path.

## Reproduction Assessment

Yes for the current behavior at source level: unmatched Telegram group senders hit group-policy-allowlist-unauthorized and the handler returns before normal reply dispatch. This is a feature request rather than a broken documented reply contract.

## Solution Assessment

Unclear as requested: replacing silent drops with a default visible reply is not the safest shape. The better path is an explicit, rate-limited opt-in notice with privacy and admin-contact behavior decided by maintainers.
