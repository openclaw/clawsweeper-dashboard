## Summary

Keep open. Current `main` still silently drops ordinary Telegram group messages from senders rejected by `groupPolicy`/`groupAllowFrom`; the request is valid, but the default reply, public user-ID disclosure, admin-contact wording, and cooldown behavior need maintainer product/security direction before implementation.

## What This Changes

Request that Telegram reply to group senders rejected by `groupAllowFrom` or `groupPolicy` with authorization instructions and the sender's Telegram user ID.

## Best Possible Solution

Add a Telegram-owned, explicit opt-in unauthorized-group-sender notice with thread-aware delivery, privacy-safe templating, admin-contact configuration, cooldowns, docs, and focused tests for ordinary group messages.

## Reproduction Assessment

Yes. A Telegram group message from sender `999999` under `groupPolicy: "allowlist"` with only `123456789` allowed follows the current unauthorized skip branch, and the existing group-policy table test expects zero replies.

## Solution Assessment

No, not as the unconditional sample behavior. The maintainable solution is an opt-in, rate-limited Telegram config surface with privacy-safe defaults rather than always posting a sender's numeric ID into the group.

## Security Review

Status: needs_attention

Summary: The requested behavior is security/privacy-sensitive even without a PR because it can expose user IDs and amplify unauthorized traffic.

Concerns:

- **[medium] Avoid public Telegram user ID disclosure by default:**
  - body: The sample reply would post the sender's numeric Telegram user ID into a group chat. That can be useful when deliberately configured, but it should not become an unconditional public default.
  - confidence: 0.88
- **[medium] Rate-limit unauthorized group notices:**
  - body: Replying to every rejected group message can spam group chats and reveal allowlist behavior to probing. A safe implementation should be opt-in and deduped or rate-limited.
  - confidence: 0.9
