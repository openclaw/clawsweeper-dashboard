## Summary

Keep open: current main and v2026.6.8 still use symlink-blind sandbox script lookup, and the open linked PR is the candidate fix that should be reviewed or replaced before this issue closes.

## What This Changes

The issue reports that `openclaw doctor` cannot find `scripts/sandbox-setup.sh` when the CLI is launched through a symlinked bin because sandbox script lookup resolves the symlink path literally.

## Best Possible Solution

Review and land an equivalent shared-root-resolver fix through https://github.com/openclaw/openclaw/pull/90942, then close this issue while keeping package-only script availability tracked in https://github.com/openclaw/openclaw/issues/90951.

## Reproduction Assessment

Yes, as a source-level reproduction. Current main resolves `process.argv[1]` with `path.resolve` and never adds a realpath candidate, so a symlinked launcher searches beside the symlink rather than the real checkout.

## Solution Assessment

Yes for keeping this issue open until the linked fix lands. The best implementation should use the shared package-root resolver and leave npm-package script shipping to the separate follow-up.
