## Summary

Keep open: current `main` and `v2026.6.8` still use the symlink-blind sandbox script resolver, and the open linked PR is the right candidate fix path.

## What This Changes

The issue reports that `openclaw doctor` cannot find `scripts/sandbox-setup.sh` when the CLI is launched through a symlinked bin because the sandbox script resolver does not follow the real launcher path.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/90942 for the symlinked launcher resolver bug, while keeping package-only sandbox auto-build as the separate follow-up.

## Reproduction Assessment

Yes. Source inspection shows current main derives launcher roots from `path.resolve(process.argv[1])` without following symlinks, and the linked PR supplies live before/after output for the exact resolver behavior under a symlinked launcher.

## Solution Assessment

Yes for the scoped issue. Adding realpath-derived launcher candidates while preserving cwd and literal launcher candidates matches existing sibling resolver patterns; broader npm-package script availability is separate work.
