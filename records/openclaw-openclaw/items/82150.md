## Summary

Keep open: current `main` still contains the reported OpenRouter DeepSeek V4 replay path, the existing tests still lock in `reasoning_content: ""` on an assistant tool-call message, and https://github.com/openclaw/openclaw/pull/82158 is open but unmerged as the attached fix path.

## What This Changes

The issue reports that OpenRouter DeepSeek V4 sessions fail after a tool call because empty `reasoning_content` is replayed on assistant tool-call messages.

## Best Possible Solution

Land a narrow fix that prevents the OpenRouter DeepSeek V4 wrapper from sending empty `reasoning_content` on assistant tool-call replay while preserving direct DeepSeek and unowned proxy replay behavior that requires valid reasoning content.

## Reproduction Assessment

Yes, at source level: current `main` composes the OpenRouter DeepSeek V4 wrapper and the OpenRouter test suite expects `reasoning_content: ""` on an assistant message with `tool_calls`. I did not perform a live OpenRouter request to reproduce the exact HTTP 500.

## Solution Assessment

The reporter's suggested direction is valid, but the maintainable fix should cover the OpenRouter wrapper or a shared wrapper option rather than a blanket change that could regress providers relying on DeepSeek-style replay placeholders.
