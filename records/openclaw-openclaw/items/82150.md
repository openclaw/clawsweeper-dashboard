## Summary

Keep open: current main still applies the DeepSeek V4 replay wrapper to OpenRouter and the current OpenRouter test suite explicitly expects `reasoning_content: ""` to be added to assistant messages that contain `tool_calls`, matching the reported failing payload shape.

## What This Changes

The issue reports a DeepSeek V4 regression where OpenRouter follow-up turns after tool calls fail because `reasoning_content` is injected into assistant tool-call messages.

## Best Possible Solution

Add a narrow OpenRouter-specific replay guard or wrapper option that avoids adding DeepSeek-native `reasoning_content` to assistant tool-call messages for verified OpenRouter DeepSeek V4 routes while preserving the direct DeepSeek and unowned proxy behavior covered by the prior fix.

## Reproduction Assessment

Yes at source level: current main composes the OpenRouter DeepSeek V4 wrapper and the existing OpenRouter test expects `reasoning_content` on an assistant `tool_calls` message. I did not perform a live OpenRouter request.

## Solution Assessment

The reporter's suggested blanket skip is directionally useful, but the safest fix is scoped to OpenRouter or an explicit shared-wrapper option so direct DeepSeek and unowned proxy replay behavior from https://github.com/openclaw/openclaw/issues/79608 is not regressed.
