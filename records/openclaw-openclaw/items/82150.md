## Summary

Keep open: current main still has the source path that can add blank `reasoning_content` to OpenRouter DeepSeek V4 assistant tool-call replay, and the linked fix PR remains open rather than merged.

## What This Changes

This issue reports a regression where OpenRouter-routed DeepSeek V4 follow-up turns after tool calls fail because OpenClaw sends blank `reasoning_content` on assistant replay.

## Best Possible Solution

Fix the OpenRouter DeepSeek V4 wrapper path, or add a shared wrapper option, so OpenRouter does not send blank `reasoning_content` on assistant tool-call replay while preserving direct DeepSeek and unowned proxy behavior.

## Reproduction Assessment

Yes, at source level: current main composes the OpenRouter DeepSeek V4 wrapper and the existing OpenRouter test expects `reasoning_content: ""` on an assistant message with `tool_calls`. I did not perform a live OpenRouter request.

## Solution Assessment

No, the linked sanitizer-only PR is not the best complete fix because the later wrapper can re-add the empty field. The safer fix should cover `ensureDeepSeekV4AssistantReasoningContent` or the OpenRouter wrapper path directly.
