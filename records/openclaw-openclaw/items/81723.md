## Summary

Keep open: current main still has a source-level path where a full WeChat OpenClaw session key can be treated as a group target and mirrored under a nested `group:agent:...:direct:...` session key. The remaining decision is the target contract: parse same-channel session keys intentionally, or fail closed and direct users to provider-native targets or `sessions_send`.

## What This Changes

The issue reports that `message(action=send)` with a full WeChat direct-session key is accepted but routed as a nested group session instead of delivering to the WeChat direct chat.

## Best Possible Solution

Define the `message.target` contract for session-key-shaped inputs, then add a generic parser or fail-closed guard so full OpenClaw session keys cannot become provider peer IDs; native WeChat `<peer>@im.wechat` targets should keep routing as direct messages.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: the latest WeChat plugin marks the full session-key-shaped string as id-like because it ends in `@im.wechat`, core defaults that raw input to group kind, and the outbound fallback then builds a group route from the full key. I did not run a live WeChat delivery.

## Solution Assessment

Is this the best way to solve the issue? Unclear because there is no PR and the intended contract is not yet explicit. The safest implementation is either a documented same-channel session-key parser or an explicit rejection with guidance to use the native WeChat id or `sessions_send`.
