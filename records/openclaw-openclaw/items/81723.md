## Summary

Keep open: current main still has a source-reproducible path where a full WeChat direct session key can be normalized as a group target and mirrored as a nested `group:agent:...:direct:...` session instead of failing closed or resolving to the direct peer.

## What This Changes

The issue requests that proactive `message(action=send)` calls to a WeChat direct session key deliver to the direct WeChat peer instead of creating a nested group session.

## Best Possible Solution

Define the `message.target` contract for full OpenClaw session keys, then either parse same-channel direct session keys into the intended native peer before routing or reject them with guidance to use the native WeChat id or `sessions_send`.

## Reproduction Assessment

Yes, at source level: the current core and latest WeChat plugin source show how a full direct session key ending in `@im.wechat` can be treated as an id-like group target and mirrored as a nested group session. This was not live-reproduced with a WeChat account.

## Solution Assessment

Unclear: there is no PR, and maintainers first need to choose whether `message.target` should support full OpenClaw session keys or reject them in favor of provider-native targets and `sessions_send`.
