## Summary

Keep open: current main still has an unresolved session-key-shaped message target path for WeChat, and the source shows that full `agent:<id>:...:direct:<peer>` targets are not parsed before outbound routing can treat the whole key as a peer id. A related open issue covers an adjacent Gateway `to`-as-session-key bug, but this message-tool path is distinct enough that it should not be closed as a duplicate yet.

## What This Changes

The issue requests that `message(action=send)` targeting a WeChat direct OpenClaw session key deliver to that direct chat instead of creating a nested group session.

## Best Possible Solution

Define one generic contract for session-key-shaped message targets and enforce it before provider target resolution: either resolve same-channel direct session keys intentionally or fail closed with guidance to use provider-native targets or `sessions_send`.

## Reproduction Assessment

No high-confidence live reproduction is available from this review. The issue gives concrete steps, and source inspection shows a clear malformed-route path if a full `agent:<id>:...:direct:<peer>` target reaches outbound routing as a group-like resolved target.

## Solution Assessment

Unclear: there is no PR, and the maintainable fix depends on the `message.target` session-key contract. The safer direction is a generic parse-or-reject guard before provider target resolution, not WeChat-specific routing in core.
