## Summary

Keep open: current main still has a plausible core path where a same-channel OpenClaw session key can be treated as a channel/group target, and the issue is fresh with concrete WeChat direct-session details rather than stale cleanup noise.

## What This Changes

The issue asks for `message(action=send)` with a WeChat direct session key to deliver to that direct WeChat session instead of creating a nested group-session route.

## Best Possible Solution

Define one contract for full OpenClaw session keys in `message.target`, then ensure they either parse to the intended same-channel direct route or fail closed with clear guidance instead of becoming nested group peer ids.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No: the issue gives concrete steps and source inspection shows a plausible malformed-route path, but this review did not exercise a live WeChat account or the external plugin resolver.

## Solution Assessment
