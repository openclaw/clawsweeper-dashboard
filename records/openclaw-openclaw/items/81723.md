## Summary

Keep open: the report is fresh, unprotected, and the current outbound routing source still has a plausible path to the exact malformed nested `group:agent:...:direct:...` session key described in the issue. This is not implemented on main, and the related closed Telegram/WeChat routing reports do not cover this direct proactive WeChat target case.

## What This Changes

The issue requests that `message(action=send)` with a WeChat direct-session target deliver to that direct session instead of being routed as a group session.

## Best Possible Solution

Choose one canonical contract for `message.target`: either parse same-channel agent session keys into the channel-native direct/group target before outbound routing, or reject session-key-shaped targets with clear guidance so they cannot become nested group sessions.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a medium-confidence reproduction path for the malformed nested route, but I did not run a live WeChat setup or the external plugin resolver in this checkout.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers decide whether the shared message tool should accept OpenClaw session keys as outbound targets; the safe invariant is that full session keys must not be silently converted into group peer ids.
