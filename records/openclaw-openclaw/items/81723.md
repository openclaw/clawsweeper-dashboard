## Summary

Keep open: current main still has a plausible generic outbound path where a full OpenClaw session key used as `message.target` can be treated as a provider target and mirrored under the wrong session shape. The issue is fresh, concrete, and not fully covered by the older Gateway `to` issue because this one is specifically the message-tool outbound route path.

## What This Changes

The issue asks for `message(action=send)` to handle a WeChat direct OpenClaw session key without routing it as a nested group target.

## Best Possible Solution

Define a single contract for full OpenClaw session keys in `message.target`, then either parse same-channel direct keys safely or fail closed with clear guidance instead of mirroring them as provider group ids.

## Reproduction Assessment

No high-confidence live reproduction is available from this review. The report gives concrete WeChat steps and current source shows a plausible nested-route path, but the external WeChat resolver and live delivery were not exercised.

## Solution Assessment

Unclear: there is no patch, and maintainers first need to choose whether full session-key targets are supported input for `message(action=send)` or should be rejected in favor of provider-native targets or `sessions_send`.
