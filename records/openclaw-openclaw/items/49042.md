## What This Changes

The branch adds an optional `agent_end.finalLlmOutcome` hook payload, embedded-runner derivation/tests/docs, and several unrelated test-stabilization updates.

## Best Possible Solution

If maintainers want this API, rebase it onto `src/plugins/hook-types.ts` and `docs/plugins/hooks.md`, fix provider-started failure classification, preserve import guardrails, and require redacted real plugin-output proof.

Yes for source inspection: current main's hook contract and embedded-runner `runAgentEnd` payload omit `finalLlmOutcome`, so plugins cannot read that field today. I did not run a live provider-failure plugin scenario in this read-only review.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Move the hook event contract to hook-types:** `src/plugins/types.ts:1532-1548`
  - body: Current main's hook event contracts live in `src/plugins/hook-types.ts`, with `src/plugins/types.ts` re-exporting that surface. Adding `finalLlmOutcome` only to this stale in-file type will not update the actual `agent_end` handler contract after rebase, so plugin authors still would not get the typed payload.
  - confidence: 0.9
- **[P2] Classify provider-started timeouts as provider outcomes:** `src/agents/pi-embedded-runner/run/attempt.ts:1417-1424`
  - body: When the provider call has started, an SDK or network `TimeoutError` still hits the `runnerFailure` branch before `providerCallStarted` is considered. That emits `source: "runner"` for provider-call timeouts, so plugins cannot reliably distinguish provider failures from local runner aborts/timeouts.
  - confidence: 0.84
- **[P2] Preserve the channel import guardrails:** `src/plugin-sdk/channel-import-guardrails.test.ts:16`
  - body: This removes the same-channel public SDK barrel checks and leaves only `plugin-sdk-internal` checks, so a channel helper could regress to importing its own `openclaw/plugin-sdk/<channel>` barrel and this guard would still pass. Keep this unrelated test weakening out of the hook API change.
  - confidence: 0.83
- **[P3] Document the field in the hook reference:** `docs/tools/plugin.md:1667-1689`
  - body: The canonical public hook reference is `docs/plugins/hooks.md`, while this PR documents the new payload only under `docs/tools/plugin.md`. Plugin authors reading the hook contract would miss the new field and its `success`/`ok` caveat.
  - confidence: 0.78
