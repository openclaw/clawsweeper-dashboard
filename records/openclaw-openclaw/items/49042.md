## Summary

Keep open: current main still lacks a structured `agent_end.finalLlmOutcome` surface, but this PR is not merge-ready because it targets stale hook/docs locations, misclassifies provider-started timeouts, includes unrelated test churn, and lacks real behavior proof.

## What This Changes

The branch adds optional `finalLlmOutcome` metadata to `agent_end`, computes it in the Pi embedded runner, documents it, and adds helper tests plus unrelated test-stabilization edits.

## Best Possible Solution

Define this outcome surface in `src/plugins/hook-types.ts` and `docs/plugins/hooks.md` only if maintainers want `agent_end` to carry final provider outcome data; otherwise direct plugins to `model_call_*` telemetry.

## Reproduction Assessment

Yes for source inspection: current main's `PluginHookAgentEndEvent` and runner emit omit `finalLlmOutcome`, so plugins cannot read that requested field. I did not run a live provider-failure plugin scenario in this read-only review.

## Solution Assessment

No, not as submitted. The additive API may be useful, but the patch should be rebased onto `src/plugins/hook-types.ts` and `docs/plugins/hooks.md`, classify provider-started timeouts correctly, and drop unrelated test churn.
