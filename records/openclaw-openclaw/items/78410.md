## Summary

This PR should not merge as-is: the diff changes non-object tool execution params beyond loop detection, while the claimed no-hook null-param mismatch is not established on current main and the supplied proof is source inspection only.

## What This Changes

The PR normalizes non-object `before_tool_call` params before loop detection and trusted-policy handling, and adds an e2e case for repeated null-param calls hitting the global circuit breaker.

## Best Possible Solution

Preserve raw tool execution params; if a real mismatch exists, normalize only the loop-signature inputs for the specific call/outcome hashing path and cover that exact hook or policy case.

## Reproduction Assessment

No. Current main’s no-hook path uses raw params consistently for detection, recording, execution, and outcome tracking, and the PR’s added null-param scenario appears to match existing behavior rather than reproduce a failure.

## Solution Assessment

No. The safer solution would keep raw or explicitly adjusted execution params intact and normalize only a dedicated loop-detection signature where call and outcome hashes can actually diverge.
