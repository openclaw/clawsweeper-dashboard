## Summary

Keep open for contributor and human follow-up: the patch broadens normalized params into execution and approval paths, which conflicts with the current raw-param contract, and the supplied proof is source inspection rather than executed behavior.

## What This Changes

Adds normalization of non-object `before_tool_call` params before loop detection/trusted policy handling and adds e2e coverage for repeated null-param tool calls hitting the global circuit breaker.

## Best Possible Solution

Preserve raw execution and approval params unless a hook or policy explicitly changes them; if a real null-param hash mismatch is reproduced, normalize only a dedicated loop-signature input and cover it with regression and real behavior proof.

## Reproduction Assessment

No. On current main, the no-hook null-param path uses raw `params` consistently for loop detection, call recording, execution, and outcome tracking; the missing evidence is an executed path that actually diverges.

## Solution Assessment

No. The patch changes runtime parameter delivery more broadly than the claimed loop-signature issue, so the safer fix is to preserve raw params and normalize only a reproduced loop-signature boundary if needed.
