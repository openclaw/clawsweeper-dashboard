## Summary

Keep open: the patch targets a plausible loop-detection bug, but current source does not establish the stated null/no-hook hash mismatch, the diff introduces a non-object parameter pass-through regression, and external real behavior proof is missing.

## What This Changes

The PR normalizes non-object `before_tool_call` params earlier, uses that normalized value in loop detection and trusted-policy defaults, and adds an e2e test for repeated null-param no-progress calls.

## Best Possible Solution

Limit any fix to the loop detector's call/outcome hashing path while preserving the existing raw or explicitly policy-adjusted params delivered to tools, then add regression coverage and real behavior proof.

## Reproduction Assessment

No. The PR's added no-hook null-param scenario appears to keep params consistent for both call recording and outcome recording on current main, so the claimed mismatch is not established without a failing current-main case or live log.

## Solution Assessment

No. The safest fix would normalize only the loop-detection hash inputs while preserving effective tool params; this diff also changes runtime params and lacks after-fix real behavior proof.
