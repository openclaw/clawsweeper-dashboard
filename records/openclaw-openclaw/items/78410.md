## Summary

Keep this PR open for revision rather than cleanup-close: the claimed current-main null-param loop bug is not demonstrated on the no-policy path, and the patch introduces a broader parameter-shape regression by executing non-object params as `{}` when no policy rewrites them.

## What This Changes

The PR normalizes non-object `before_tool_call` params before loop detection and policy/approval handling, and adds e2e coverage for repeated null-param no-progress calls.

## Best Possible Solution

Preserve raw execution and approval params unless a hook or trusted policy explicitly changes them; if a real adjusted-param hash mismatch is reproduced, normalize only the loop signature or recording input and add executed regression proof.

## Reproduction Assessment

No for the claimed current-main no-policy bug: source inspection shows raw params are used consistently for loop detection, recording, execution, and outcome tracking. Yes for the PR regression: source inspection shows the branch would execute `{}` instead of raw non-object params when no policy changes them.

## Solution Assessment

No. The patch changes runtime parameter delivery more broadly than needed; the safer repair is signature-only normalization or adjusted-param recording after a concrete failing path is shown.
