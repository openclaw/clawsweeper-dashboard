## Summary

Keep open for contributor revision: the patch is not safe because it changes raw non-object execution and approval params to `{}`, while current main already keeps the no-hook null-param loop path hash-consistent and the related parameterless-null validation case is handled elsewhere.

## What This Changes

The PR normalizes non-object `before_tool_call` params before loop detection and approval handling, and adds e2e coverage for repeated null-param calls.

## Best Possible Solution

Preserve raw execution and approval params unless a hook or policy explicitly changes them; if a real null-param hash mismatch is reproduced, normalize only a dedicated loop-signature input and cover it with executed regression proof.

## Reproduction Assessment

No for the claimed current-main bug: source inspection shows the no-hook null-param path uses raw params consistently for loop detection, call recording, execution, and outcome tracking. Yes for the PR regression: the changed default would deliver `{}` instead of raw non-object params to execution in the no-policy path.

## Solution Assessment

No. The patch changes runtime parameter delivery more broadly than needed, so the safer solution is a signature-only normalization after a concrete failing path is demonstrated.
