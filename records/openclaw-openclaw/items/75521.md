## Summary

Keep open. The latest PR revision fixes the previously reported gateway-bindable/default-mode miss and adds focused tests for that path, but the new scoped fallback now uses the active registry's runtime mode for every unscoped request. That can let an explicit subagent-runtime request reuse a scoped default or gateway-bindable registry instead of missing the cache, weakening the loader's runtime-mode compatibility boundary.

## What This Changes

The PR adds a scoped-active to unscoped-request compatibility path in `getCompatibleActivePluginRegistry()`, regression tests for scoped gateway-bindable registry reuse, and a changelog entry for the embedded runner tool-resolution performance fix.

## Best Possible Solution

Keep the scoped-registry reuse in loader compatibility, but compose it with the existing exact-match and default-request-to-gateway-bindable relaxation paths so explicit subagent requests still miss unless the active registry was loaded for the same runtime mode. Add a regression test for the explicit subagent unscoped request against a scoped active registry.

## Reproduction Assessment

Yes. A code-level reproduction is a scoped active registry keyed with `onlyPluginIds` and gateway-bindable mode, followed by an unscoped `getCompatibleActivePluginRegistry()` request that supplies `runtimeOptions.subagent`; the PR's fallback can return the active registry even though the request runtime mode differs.

## Solution Assessment

No for the current patch. The product direction is narrow and maintainable, but the implementation should compare scoped variants using the request mode except for the already-intentional default-to-gateway-bindable relaxation.
