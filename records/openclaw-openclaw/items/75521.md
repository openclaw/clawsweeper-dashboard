## Summary

Keep this PR open: the linked performance problem in #75520 is plausible, but this diff is not merge-ready because it bypasses the plugin registry compatibility checks that current main uses to protect workspace/config/scope/runtime-mode correctness.

## What This Changes

This PR adds an unconditional active-registry fast path in `src/plugins/tools.ts` so plugin tool resolution returns `getActivePluginRegistry()` before falling back to `resolveRuntimePluginRegistry()`.

## Best Possible Solution

Keep the compatibility contract and add a narrower reuse path that fixes the scoped-startup versus unscoped embedded-tool case only after preserving config, workspace, activation, and runtime-mode compatibility; add regression coverage for both the performance case and the existing default-mode gateway-binding guard.

## Reproduction Assessment

Yes for the PR regression: `src/plugins/tools.optional.test.ts` already constructs a default-mode active registry and a gateway-bindable tool request, and the proposed unconditional active return would skip the required reload. The original performance report has concrete timing/log evidence, but I did not live-run that scenario in this read-only review.

## Solution Assessment

No. Unconditionally reusing any active registry is broader than the existing loader contract; the safer fix is to relax only the scoped/unscoped equivalence that causes #75520 while preserving workspace/config/runtime-mode checks.
