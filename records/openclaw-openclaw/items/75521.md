## Summary

Keep this PR open but block merge for now: it targets a real current-main plugin registry cache gap and correctly accounts for gateway method cache keys, but the new scoped fallback still lets a startup-scoped registry satisfy every unscoped tool request, which can drop tools from enabled startup-lazy plugins.

## What This Changes

The PR adds loader compatibility logic to reuse scoped gateway-bindable active plugin registries for unscoped tool-resolution requests, stores core gateway method names on plugin registries and snapshots, adds regression tests, and updates the changelog.

## Best Possible Solution

Keep the loader-level reuse direction, but make the scoped fallback preserve unscoped tool semantics: reuse the active startup registry only when it covers the requested tool plugin scope, or derive and cache a dedicated tool-resolution scope that includes enabled startup-lazy tool plugins.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a config that allows a startup plugin plus an enabled startup-lazy tool plugin such as Tavily; the PR fallback can return the active startup subset for the unscoped tool request, leaving the lazy plugin's tools out of `resolvePluginTools()`.

## Solution Assessment

No. The PR is close, but substituting `activeRegistry.plugins` for every unscoped request is too broad; the safer fix is a coverage guard or a manifest-derived tool scope, with a regression test for a startup subset plus a non-startup tool plugin.

## Review Findings

Gate reuse on coverage of the requested tool scope, or load/cache a dedicated tool scope instead.
