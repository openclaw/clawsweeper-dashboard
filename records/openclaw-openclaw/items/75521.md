## Summary

Keep this PR open, but not merge-ready. The latest head appears to address the earlier runtime-mode blocker, yet the new scoped reuse path still reconstructs the cache key from the tool request's empty gateway-method list, while production gateway startup keys the active registry with `baseMethods`; that means the reported embedded-message path can still miss the active registry.

## What This Changes

The PR adds scoped active plugin-registry reuse in loader compatibility, regression tests for that reuse, and a changelog entry for the embedded plugin-tool lookup performance fix.

## Best Possible Solution

Keep the loader-level scoped-registry reuse, but make the compatibility comparison production-shaped: either carry the startup gateway method names into the tool-resolution request or intentionally allow an active registry with the gateway-method superset to satisfy this tool lookup. Add a regression that includes `coreGatewayMethodNames` on the scoped gateway startup key and an unscoped default embedded-tool request without method names.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a scoped gateway-bindable active registry keyed with `onlyPluginIds` plus startup `coreGatewayMethodNames`, followed by an unscoped default `resolvePluginTools()` request that supplies no method names; the PR fallback compares different gateway-method key segments and returns undefined.

## Solution Assessment

No. The PR is close, and the runtime-mode correction is the right direction, but the current implementation still misses the production cache-key shape for the reported path unless gateway method names are propagated or safely relaxed.
