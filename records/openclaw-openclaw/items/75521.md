## Summary

Keep this PR open, but it is not merge-ready. The performance problem remains plausible on current main, yet the proposed fast path bypasses plugin registry compatibility checks for non-gateway tool resolution and can reuse a stale, differently scoped, or different-workspace active registry; the PR also lacks the required changelog entry for a user-facing fix.

## What This Changes

This PR adds a non-gateway active-registry fast path in `src/plugins/tools.ts` so default plugin tool resolution can return `getActivePluginRegistry()` before falling back to `resolveRuntimePluginRegistry()`.

## Best Possible Solution

Keep the loader compatibility contract and add a narrower reuse path for the scoped gateway-startup registry versus unscoped embedded-tool request only when workspace, config, activation metadata, runtime mode, and loaded-plugin coverage remain compatible. Add regression coverage for that performance case, keep the existing reverse gateway-binding guard, and document the user-facing fix in the changelog.

## Reproduction Assessment

Yes for the PR regression: existing tests and loader code show that workspace and plugin-scope mismatches are deliberate cache-key guards, and this patch returns the active registry before those guards run for default tool loads. The original performance report has concrete timing and log evidence, but I did not live-run that scenario in this read-only pass.

## Solution Assessment

No. Reusing any active registry for all non-gateway tool resolution is broader than the bug requires; the maintainable fix belongs in the compatibility/reuse logic so only the scoped-startup versus unscoped embedded-tool case is relaxed safely.
