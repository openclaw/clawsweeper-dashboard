## Summary

Keep this PR open. Current main still has the scoped gateway startup registry versus unscoped embedded tool-resolution mismatch, and the PR is a targeted implementation for the linked per-turn plugin reload regression; I did not find a blocking correctness or security issue in the diff.

## What This Changes

The PR stores core gateway method names on plugin registries, scopes plugin-tool runtime loads to the active plus enabled plugin plan, relaxes compatible active-registry reuse for omitted gateway method cache keys, and adds loader/tool regression tests plus a changelog entry.

## Best Possible Solution

Land this scoped registry/tool-resolution fix after exact-head CI is green or the unrelated failures are explicitly cleared; then close the linked performance report as fixed and carry the changelog entry into the next release.

## Reproduction Assessment

Yes. The source-level reproduction is clear on current main: gateway startup loads a scoped, gateway-bindable registry with core gateway method names, while embedded tool resolution builds an unscoped request that misses the active cache key; the linked reports add repeated per-turn timing and reload traces.

## Solution Assessment

Yes. Moving reuse through loader compatibility while making tool resolution request the active plus enabled plugin scope is the narrowest maintainable direction I found, and the latest diff covers the coreGatewayMethodNames gap raised in review.
