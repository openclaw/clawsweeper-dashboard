## Summary

Keep open. The PR is aimed at a real plugin-tool hot path, but the current patch still misses the reported gateway-startup case because its scoped fallback rebuilds the cache key with the request's default runtime mode instead of the active gateway-bindable mode.

## What This Changes

The PR adds a changelog entry and a `getCompatibleActivePluginRegistry()` fallback that tries to reuse a scoped active plugin registry for an unscoped tool-resolution request by rebuilding a cache key from the active registry's plugin IDs.

## Best Possible Solution

Keep the compatibility relaxation in `src/plugins/loader.ts`, but make the scoped-active comparison compose with the existing gateway-bindable/default-mode relaxation and cover that exact embedded-run path in loader/tool tests.

## Reproduction Assessment

Yes. A code-level reproduction is an active registry keyed with `onlyPluginIds` plus `allowGatewaySubagentBinding: true`, followed by an unscoped default-mode tool/runtime request; the PR still rebuilds a scoped default-mode key and misses the active key.

## Solution Assessment

No for the current patch. The direction is maintainable, but the implementation needs to compare against the active gateway-bindable scoped variant when the active registry was loaded that way.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Match the gateway-bindable scoped cache key:** `src/plugins/loader.ts:963`

Build the scoped comparison against the gateway-bindable variant when the active registry is gateway-bindable, and add a regression test for that path.
