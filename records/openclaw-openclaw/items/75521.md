## Summary

Keep this PR open. The cache-miss regression is real and the branch addresses the earlier gateway-method compatibility gap, but the current fallback can make an unscoped plugin-tool request return only the scoped gateway-startup registry, dropping enabled non-startup tool plugins.

## What This Changes

The PR reuses an active scoped gateway-bindable plugin registry for compatible unscoped/default tool-resolution requests, persists core gateway method names on plugin registries, adds loader regression tests, and adds a changelog entry for #75520.

## Best Possible Solution

Keep the loader-level cache reuse direction, but preserve unscoped tool semantics: reuse the active scoped registry only when it covers the plugin-tool scope that the unscoped request would otherwise load, or introduce a dedicated manifest-derived tool scope that is cached separately from the gateway startup subset.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a config with a startup plugin active in the gateway registry plus an enabled non-startup tool plugin such as Tavily; the PR fallback returns the active startup subset for the unscoped tool request instead of loading the non-startup tool plugin.

## Solution Assessment

No. The current PR is close, but the scoped fallback is too broad; the safer fix is to gate reuse on coverage of the requested tool scope or load/cache a tool-specific scope rather than substituting `activeRegistry.plugins` for every unscoped request.
