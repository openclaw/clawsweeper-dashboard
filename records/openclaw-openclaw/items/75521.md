## Summary

Keep this PR open. The revised direction is much safer than the earlier `tools.ts` shortcut, but the patch is still not merge-ready because the new scoped-cache comparison does not compose with the gateway-bindable runtime-mode difference that exists in the reported regular embedded-run path.

## What This Changes

The PR adds a changelog entry and a `src/plugins/loader.ts` compatibility branch that reuses a scoped active plugin registry for unscoped requests when a cache key rebuilt from the active registry's plugin IDs matches.

## Best Possible Solution

Keep the compatibility logic in the loader, but make the scoped-active reuse path explicitly compose with the existing gateway-bindable/default-mode relaxation and prove it with focused loader/tool tests for the reported embedded-run case.

## Reproduction Assessment

Yes for the PR defect: the code-level reproduction is a gateway startup active registry keyed with `onlyPluginIds` plus `gateway-bindable`, followed by a default-mode unscoped tool/runtime request. Because the PR rebuilds the scoped key with the request's default runtime mode, the cache keys still cannot match.

## Solution Assessment

No. The patch preserves more of the loader contract than the earlier shortcut, but the narrow maintainable fix needs to compare the scoped active IDs under the active gateway-bindable runtime variant when the current request is default-mode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Match the gateway-bindable scoped cache key:** `src/plugins/loader.ts:963`
  - body: The new scoped fallback rebuilds `scopedCacheKey` with `loadContext.runtimeSubagentMode`. Build the scoped comparison against the gateway-bindable variant when the active registry is gateway-bindable, and add a regression test for that path.
