## Summary

Keep open. Current main still has the scoped active-registry versus unscoped tool-resolution mismatch, and the latest PR targets the relevant cache-key dimensions without an obvious blocking bug; it should proceed through normal maintainer review and exact-head CI rather than be closed or replaced.

## What This Changes

The PR scopes plugin-tool runtime requests to the active plus enabled plugin plan, stores core Gateway method names on plugin registries and snapshots, relaxes compatible active-registry reuse when only the request omits those method keys, and adds loader/tool regression tests plus a changelog entry.

## Best Possible Solution

Land this scoped registry/tool-resolution fix after exact-head CI completes cleanly or any unrelated check noise is explicitly cleared, then let the linked performance report close through the merged fix and keep the changelog credit in the next release.

## Reproduction Assessment

Yes. The source-level reproduction is clear on current main: Gateway startup creates a scoped, gateway-bindable active registry with core Gateway method keys, while normal embedded plugin-tool resolution requests a tool-discovery registry without those key dimensions; the linked reports add timing and repeated reload traces.

## Solution Assessment

Yes. The current PR is the narrowest maintainable direction I found: it makes tool resolution request the intended plugin scope, preserves the active registry compatibility checks, and carries coreGatewayMethodNames through the registry instead of hard-coding a bundled-plugin exception.
