Full review comments:

- **[P1] Preserve the missing-unit activation retry:** `src/daemon/systemd.ts:735-740`
  - body: Current main retries the failed `enable` or `restart` after another `daemon-reload` when systemd reports `Unit file ... does not exist`; this branch calls each action once. Refreshing this branch without that retry can reintroduce the install failure tracked by https://github.com/openclaw/openclaw/issues/65184.
  - confidence: 0.94
- **[P1] Honor custom systemd units during uninstall:** `src/daemon/systemd.ts:794-796`
  - body: The PR head computes the unit to disable with `resolveGatewaySystemdServiceName`, so `OPENCLAW_SYSTEMD_UNIT=openclaw-node` still disables `openclaw-gateway.service` while deleting the node unit file. Current main uses `resolveSystemdServiceName(env)` to keep uninstall aligned with the override from https://github.com/openclaw/openclaw/issues/68287.
  - confidence: 0.95
- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:55-63`
  - body: The new validator still allows names that begin with `-`, and the later `systemctl` calls pass the unit operand without a `--` terminator. A value such as `OPENCLAW_SYSTEMD_UNIT=-Hfoo` can be parsed as a systemctl option instead of a unit name.
  - confidence: 0.88
- **[P1] Avoid direct writes for secret-bearing drop-ins:** `src/daemon/systemd.ts:627-637`
  - body: The managed drop-in can contain gateway tokens, but the writer uses direct `copyFile` and `writeFile` on predictable HOME-derived paths before or around chmod. A pre-existing link under the unit directory can redirect the backup or drop-in write, so this needs a safe-write path or an explicit maintainer threat-model decision.
  - confidence: 0.82
- **[P1] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:270-272`
  - body: When one assignment is stripped from a mixed `Environment=` line, the kept entries are re-rendered through `renderEnvLines`, which drops empty values and trims whitespace. That changes valid systemd settings such as `Environment=LD_PRELOAD=` or values with intentional spaces during an upgrade migration.
  - confidence: 0.9
- **[P2] Remove the release-owned changelog edit:** `CHANGELOG.md:20`
  - body: Normal PRs should not edit `CHANGELOG.md`; release generation owns that file. This diff also removes unrelated existing entries, so keep the release-note context in the PR body or squash message instead.
  - confidence: 0.96

## Security Review

Status: needs_attention

Summary: The diff changes security-sensitive systemd token storage and command execution paths, and it leaves concrete unit-name parsing, link-following write, and environment-rewrite concerns unresolved.

Concerns:

- **[high] Option-like unit names can reach systemctl:** `src/daemon/systemd.ts:62`
  - body: The validator allows leading `-` and systemctl calls do not terminate options before the unit operand, so a custom unit name can still be interpreted as a systemctl flag.
  - confidence: 0.88
- **[high] Managed drop-in writes can follow attacker-placed links:** `src/daemon/systemd.ts:637`
  - body: The secret-bearing drop-in and backup paths are HOME-derived and written with direct `copyFile`/`writeFile`, leaving overwrite or leakage risk unless the threat model explicitly excludes it.
  - confidence: 0.82
- **[medium] Environment rewrite can remove clearing assignments:** `src/daemon/systemd-unit.ts:270`
  - body: Partial `Environment=` rewrites pass kept entries through a renderer that drops empty values, so clearing assignments like `LD_PRELOAD=` can disappear during migration and change inherited runtime behavior.
  - confidence: 0.9
