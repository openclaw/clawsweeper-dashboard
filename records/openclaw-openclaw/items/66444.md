- **[P1] Preserve the missing-unit activation retry:** `src/daemon/systemd.ts:735-740`
  - body: `activateSystemdService` in this branch calls `systemctl enable` and `restart` only once after the first daemon reload.   - confidence: 0.94
- **[P1] Honor custom systemd units during uninstall:** `src/daemon/systemd.ts:794-796`
  - body: `uninstallSystemdService` resolves the gateway service name instead of the effective `OPENCLAW_SYSTEMD_UNIT`, so an `openclaw-node` uninstall would disable/remove the gateway unit path again. Current main uses the override-aware resolver and has coverage for the node-only VM failure at https://github.com/openclaw/openclaw/issues/68287.
  - confidence: 0.95
- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:53-62`
  - body: The new validation still accepts names beginning with `-`, and the branch passes the unit operand to `systemctl` without a `--` terminator. That leaves `OPENCLAW_SYSTEMD_UNIT` values such as `--user` or `-Hhost` able to be parsed as options instead of unit names, so the hardening is incomplete.
  - confidence: 0.88
- **[P1] Avoid direct writes for secret-bearing drop-ins:** `src/daemon/systemd.ts:637-638`
  - body: The managed drop-in can contain gateway tokens, but this path writes it directly to a predictable HOME-derived pathname and backs up the main unit with `copyFile`. Before adding a new secret-bearing file, use a symlink-safe write strategy or document and test the accepted local threat model.
  - confidence: 0.83
- **[P2] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:271-273`
  - body: When only part of a multi-assignment `Environment=` line is stripped, the kept assignments are re-rendered through `renderEnvLines`, which filters empty values and trims whitespace. Existing operator settings such as `Environment=LD_PRELOAD=` can be silently removed during migration, changing the effective service environment.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces or leaves unresolved concrete systemd security-boundary concerns around unit-name option parsing and secret-bearing file writes.

Concerns:

- **[high] Option-like unit names remain accepted:** `src/daemon/systemd.ts:53`
  - body: `SYSTEMD_SERVICE_NAME_PATTERN` permits leading `-`, and subsequent `systemctl` calls pass the unit name without `--`, so malicious or misconfigured `OPENCLAW_SYSTEMD_UNIT` values can be interpreted as command options.

- **[high] Secret drop-in writes follow predictable paths:** `src/daemon/systemd.ts:637`
  - body: The new managed drop-in can contain tokens and is written directly under a HOME-derived systemd directory without symlink-safe open/rename behavior, leaving local overwrite or leakage concerns unresolved.
  - confidence: 0.83

  - body: When the managed-env sentinel is absent, the PR writes remaining environment values into the main unit with default file permissions, which can make secret-like values more readable than the locked-down drop-in path.
