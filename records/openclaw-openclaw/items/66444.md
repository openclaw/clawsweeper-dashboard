- **[P1] Preserve the missing-unit activation retry:** `src/daemon/systemd.ts:735-740`
  - body: Current main retries daemon-reload plus the failed enable/restart once when systemd says the freshly written unit does not exist, which protects migrated user services.   - confidence: 0.9
- **[P1] Honor custom systemd units during uninstall:** `src/daemon/systemd.ts:794-796`
  - body: Current main disables the unit resolved from OPENCLAW_SYSTEMD_UNIT, so node/custom services uninstall the unit they installed. This PR head still derives the disable target from the gateway profile, which can leave openclaw-node enabled while removing the node unit file.
  - confidence: 0.9
- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:53-63`
  - body: The new validation still accepts names beginning with a dash, and systemctl calls do not consistently put -- before the unit operand. Reject leading-dash service names or terminate options before every unit operand so OPENCLAW_SYSTEMD_UNIT cannot be parsed as a systemctl option.
  - confidence: 0.88
- **[P1] Avoid direct writes for secret-bearing drop-ins:** `src/daemon/systemd.ts:637-638`
  - body: The managed drop-in can contain token values, but the branch writes the backup, main unit, and drop-in with direct copyFile/writeFile on predictable HOME-derived paths. Use a no-symlink/atomic write path or add a reviewed threat-model note and regression coverage before introducing another secret-bearing service file.
  - confidence: 0.82
- **[P1] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:270-272`
  - body: When only some assignments are stripped, the kept entries are re-rendered through renderEnvLines, which drops empty or whitespace-only values and trims the rest. That changes valid systemd semantics such as FOO= clearing an inherited variable, so preserve exact values in partial rewrites.
  - confidence: 0.86
- **[P2] Remove the release-owned changelog edit:** `CHANGELOG.md:20`
  - body: OpenClaw release generation owns CHANGELOG.md for normal PRs, and this branch also removes unrelated release entries. Keep the release-note context in the PR body or squash message and leave CHANGELOG.md out of the branch.

## Security Review

Status: needs_attention

Summary: The diff has concrete systemd security-boundary concerns around option-like unit names, secret-bearing file writes, and Environment= rewrite semantics.

Concerns:

- **[high] Option-like unit names can reach systemctl:** `src/daemon/systemd.ts:53`
  - body: OPENCLAW_SYSTEMD_UNIT validation accepts leading dashes and systemctl invocations do not consistently terminate options before the unit operand, so a custom unit name can be interpreted as a systemctl option.
  - confidence: 0.88
- **[high] Secret drop-in writes follow predictable paths:** `src/daemon/systemd.ts:637`
  - body: The managed drop-in can contain token values, but the branch writes it with direct writeFile under a HOME-derived path and backs up the main unit with direct copyFile, leaving symlink/hardlink overwrite concerns unresolved.
  - confidence: 0.82
- **[medium] Sentinel-missing env can fall back into the main unit:** `src/daemon/systemd-unit.ts:102`
  - body: When the managed-env sentinel is absent, splitSystemdManagedEnvironment treats every value as user-owned, so caller mistakes can put sensitive service env into the main unit rather than the locked-down drop-in.
  - confidence: 0.76
- **[medium] Environment rewrite can remove clearing assignments:** `src/daemon/systemd-unit.ts:270`
  - body: Dropping FOO= or trimming whitespace while rewriting mixed Environment= lines can change inherited environment behavior for the service, which can be security-relevant for loader or runtime flags.
  - confidence: 0.78
