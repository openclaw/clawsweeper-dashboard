## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Reject option-like unit names before systemctl:** `src/daemon/systemd.ts:55-63`
  - body: `validateSystemdServiceName` still allows names beginning with `-`, and the service name is later passed to `systemctl` as a unit operand without an end-of-options marker. An `OPENCLAW_SYSTEMD_UNIT` such as `-Hhost` can be interpreted as a systemctl option instead of a unit, so the PR needs to reject leading dashes and/or terminate options consistently.
  - confidence: 0.93
- **[P1] Preserve the activation retry on current main:** `src/daemon/systemd.ts:727-743`
  - body: This activation path goes straight from `daemon-reload` to one `enable`/`restart` attempt. Current main retries `daemon-reload` plus the failed action when systemd reports the unit file does not exist, with regression coverage for that install race; dropping it would bring back the failure tracked by https://github.com/openclaw/openclaw/issues/65184.
  - confidence: 0.9
- **[P1] Honor custom unit names during uninstall:** `src/daemon/systemd.ts:793-796`
  - body: The uninstall path still disables `resolveGatewaySystemdServiceName(...)`, so an install using `OPENCLAW_SYSTEMD_UNIT=openclaw-node` removes the node unit file but asks systemctl to disable `openclaw-gateway.service`. Current main uses `resolveSystemdServiceName(env)` here and has regression coverage for the node-only VM case in https://github.com/openclaw/openclaw/issues/68287.
  - confidence: 0.9
- **[P1] Harden managed unit and drop-in writes:** `src/daemon/systemd.ts:627-638`
  - body: The migration writes backups and the managed drop-in to predictable HOME-derived paths with direct `copyFile`/`writeFile`, while the new drop-in can contain service environment secrets. Before this changes the persistent service layout, the writer needs explicit no-symlink/permission handling or a documented maintainer threat-model decision with tests.
  - confidence: 0.86
- **[P1] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:269-273`
  - body: When a mixed `Environment=` line is partially stripped, the kept assignments are re-rendered through `renderEnvLines`, which drops empty values and trims whitespace. That changes valid systemd semantics such as `Environment=LD_PRELOAD=` or values with intentional surrounding spaces during an upgrade migration.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff still has concrete systemd security-boundary concerns around option-like unit names, HOME-derived file writes for managed env, and mutation of kept environment semantics.

Concerns:

- **[high] Option-like unit names can reach systemctl:** `src/daemon/systemd.ts:55`
  - body: The service-name validator accepts leading dashes and the systemctl calls do not add `--` before the unit operand, so an environment override can be parsed as a systemctl option rather than a unit name.
  - confidence: 0.92
- **[high] Managed env writes use predictable HOME-derived paths:** `src/daemon/systemd.ts:627`
  - body: The PR writes backups and a managed drop-in containing service environment values with direct filesystem calls under a user-controlled HOME-derived directory, leaving symlink/overwrite and permission-hardening questions unresolved.
  - confidence: 0.86
- **[medium] Environment migration can remove protective empty assignments:** `src/daemon/systemd-unit.ts:270`
  - body: Re-rendering kept `Environment=` assignments through a trimming renderer can drop empty assignments such as `LD_PRELOAD=`, potentially changing the process environment security posture after migration.
  - confidence: 0.82
