## Security Review

Status: needs_attention

Summary: The diff touches systemd command execution and token-bearing service-file writes, with concrete operand validation, write-path, secret-permission, and environment-rewrite concerns still open.

Concerns:

- **[high] Option-like systemd unit names still pass validation:** `src/daemon/systemd.ts:55`
  - body: The added service-name validator rejects paths but not leading dashes, while unit operands are still passed to systemctl without an end-of-options marker.
  - confidence: 0.91
- **[high] Token-bearing drop-in uses direct HOME-derived writes:** `src/daemon/systemd.ts:637`
  - body: The managed environment drop-in and backups are written through predictable paths without no-follow or atomic protections, so symlink or hardlink redirection remains a plausible local overwrite or disclosure risk.
  - confidence: 0.86
- **[medium] Missing sentinel can put secrets in the main unit:** `src/daemon/systemd-unit.ts:102`
  - body: When the managed-env sentinel is absent, the splitter treats all environment entries as user-owned and the fresh main unit write uses the default string write path, which can expose secrets if a caller omits the sentinel.
  - confidence: 0.78
- **[medium] Environment rewrite can drop explicit empty values:** `src/daemon/systemd-unit.ts:270`
  - body: Partial Environment= rewrites use a renderer that filters empty values and trims whitespace, which can remove security-relevant environment clearing assignments.
  - confidence: 0.84
