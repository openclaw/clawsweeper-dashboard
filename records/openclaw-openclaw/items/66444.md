- **[P1] Preserve the activation retry path:** `src/daemon/systemd.ts:735-740`
  - body: Current main retries `daemon-reload` plus the failed `enable`/`restart` once when systemd says the unit file does not exist, with coverage in `src/daemon/systemd.test.ts`. This PR head calls `enable` and `restart` directly after one reload, so carrying it forward would bring back the migrated/node install failure where systemd has not noticed the freshly written unit yet.
  - confidence: 0.93
- **[P1] Use the override service name during uninstall:** `src/daemon/systemd.ts:794-796`
  - body: This line still derives the disabled unit from the gateway profile instead of `resolveSystemdServiceName(env)`. On node-only or custom-unit installs, `OPENCLAW_SYSTEMD_UNIT=openclaw-node` can remove the node unit path while disabling `openclaw-gateway.service`, which is exactly the current-main behavior covered by the custom-unit uninstall test.
  - confidence: 0.92
- **[P1] Carry environmentValueSources into the writer:** `src/daemon/systemd.ts:527-533`
  - body: The changed writer ignores the current `GatewayServiceInstallArgs.environmentValueSources` contract, which main uses to distinguish inline, file-only, and inline-and-file values. Without that provenance, a refresh can put file-backed secrets into the wrong place or leave stale generated env-file values overriding fresh inline values during systemd restaging.
  - confidence: 0.89
- **[P1] Reject option-like unit names before systemctl:** `src/daemon/systemd.ts:53-63`
  - body: The new validator still permits service names beginning with `-`, and the systemctl calls pass the unit operand without a `--` terminator. An `OPENCLAW_SYSTEMD_UNIT` such as `--user` or `-Hhost` can still be interpreted as a systemctl option rather than a unit name, so the claimed unsafe-name hardening is incomplete.
  - confidence: 0.9
- **[P1] Harden managed unit and drop-in writes:** `src/daemon/systemd.ts:627-638`
  - body: This migration introduces a secret-bearing managed drop-in but writes and backs up predictable HOME-derived paths with direct `copyFile`/`writeFile` calls. Before moving managed env into a separate file, the writer needs a clear no-symlink/no-follow or threat-model decision, plus explicit main-unit permissions where secrets can still be rendered.
  - confidence: 0.84
- **[P2] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:270-273`
  - body: When stripping one key from a multi-assignment `Environment=` line, this re-renders the kept values through `renderEnvLines`, which drops empty values and trims whitespace. Valid systemd assignments such as `Environment=LD_PRELOAD=` can disappear during migration, changing service semantics while touching unrelated user-owned entries.
  - confidence: 0.87
- **[P2] Remove release-owned changelog edits:** `CHANGELOG.md:17`
  - body: Root policy makes `CHANGELOG.md` release-owned, but this PR adds an unreleased entry and removes unrelated existing entries. Keep the release-note context in the PR body or squash message instead of landing normal PR edits to the generated changelog surface.
