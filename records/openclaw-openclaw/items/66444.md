Full review comments:

- **[P1] Preserve the released activation retry:** `src/daemon/systemd.ts:735-740`
  - body: This branch enables and restarts the unit only once after `daemon-reload`. Current `v2026.6.9` retries `daemon-reload` plus the failed action when systemd reports the written unit is not visible, so merging this stale path can reopen service install recovery failures.
  - confidence: 0.91
- **[P1] Honor custom unit names during uninstall:** `src/daemon/systemd.ts:794-796`
  - body: `uninstallSystemdService` still disables the gateway-derived unit name even when `OPENCLAW_SYSTEMD_UNIT` points at a node/custom unit. Current main disables the resolved unit override and has a regression test for that path.
  - confidence: 0.94
- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:55-63`
  - body: The new validator still allows service names that start with `-`, and the `systemctl` calls pass the unit name without a `--` operand terminator. An option-like `OPENCLAW_SYSTEMD_UNIT` can still be parsed by `systemctl` as a flag.
  - confidence: 0.89
- **[P1] Harden managed unit and drop-in writes:** `src/daemon/systemd.ts:637`
  - body: The PR adds a secret-bearing managed drop-in under a HOME-derived systemd directory but writes the backup, main unit, and drop-in through predictable direct `copyFile`/`writeFile` paths. This needs safe-write/no-symlink handling or an explicit maintainer threat-model decision before merge.
  - confidence: 0.82
- **[P1] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:270-274`
  - body: When only some assignments are stripped, this path re-renders kept values through `renderEnvLines`, which drops empty values and trims whitespace. That can remove valid operator settings such as `Environment=LD_PRELOAD=` during migration.
  - confidence: 0.87
