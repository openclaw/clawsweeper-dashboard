- **[P1] Reject option-like unit names before systemctl:** `src/daemon/systemd.ts:55-63`
  - body: `validateSystemdServiceName` still accepts names that begin with `-`, and this branch passes the resulting unit name to `systemctl` without a `--` operand terminator.   - confidence: 0.94
- **[P1] Preserve the activation retry from current main:** `src/daemon/systemd.ts:735-740`
  - body: This edited activation path fails after the first `enable` or `restart` error. Current main retries `daemon-reload` plus the failed action once when systemd reports `Unit file ... does not exist`; carry that behavior forward so migrated user managers do not hit the freshly-written-unit race again.
  - confidence: 0.9
- **[P1] Honor custom unit names during uninstall:** `src/daemon/systemd.ts:794-796`
  - body: `uninstallSystemdService` disables the gateway-derived unit even when `OPENCLAW_SYSTEMD_UNIT` points at a custom or node service. That can disable the wrong service and leave the real custom unit enabled, so use the same `resolveSystemdServiceName(env)` path as current main.
  - confidence: 0.91
- **[P1] Harden managed unit and drop-in writes:** `src/daemon/systemd.ts:593-638`
  - body: The migration writes and backs up predictable HOME-derived unit/drop-in paths with direct `writeFile`/`copyFile`. Because the new drop-in can carry service secrets, use a no-follow/atomic or explicitly threat-modeled write path before landing.
  - confidence: 0.86
- **[P1] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:269-274`
  - body: When only some assignments are stripped, the code re-renders kept values through `renderEnvLines`, which drops empty or whitespace-only values and trims the rest. That changes valid systemd semantics such as `Environment=LD_PRELOAD=` during migration; reconstruct kept assignments without dropping or trimming values and add focused tests.
  - confidence: 0.88
