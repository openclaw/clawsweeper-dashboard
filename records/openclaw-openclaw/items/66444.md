Full review comments:

- **[P1] Preserve the activation retry:** `src/daemon/systemd.ts:735-740`
  - body: Current main retries daemon-reload plus the failed enable/restart when systemd says the unit file does not exist, and current tests cover that race. This PR head still calls enable and restart once, so carrying it forward without the retry can reintroduce install failures on migrated/root user-systemd setups.
  - confidence: 0.95
- **[P1] Honor custom unit names during uninstall:** `src/daemon/systemd.ts:794-796`
  - body: The uninstall path still derives the unit from OPENCLAW_PROFILE instead of resolveSystemdServiceName(env). With OPENCLAW_SYSTEMD_UNIT=openclaw-node, uninstall would disable the gateway unit while removing the node unit file, regressing the current-main node-only fix.
  - confidence: 0.96
- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:55-63`
  - body: validateSystemdServiceName permits names beginning with '-', and the systemctl calls pass the unit operand without a '--' terminator. An OPENCLAW_SYSTEMD_UNIT override such as --machine=... can be parsed as a systemctl option instead of a service name, so the branch needs a leading-dash rejection and/or operand terminator coverage.
  - confidence: 0.91
- **[P1] Preserve empty Environment assignments:** `src/daemon/systemd-unit.ts:270-273`
  - body: When stripping only some assignments, this path re-renders the kept values through renderEnvLines, which filters out empty or whitespace-only values and trims the rest. Valid operator settings such as Environment=LD_PRELOAD= can disappear during migration, changing the effective service environment.
  - confidence: 0.9
- **[P1] Harden HOME-derived unit writes:** `src/daemon/systemd.ts:627-638`
  - body: The new writer copies the backup and writes the main unit/drop-in through predictable HOME-derived paths. Because the drop-in can contain managed secrets, the branch needs either no-symlink/permission hardening or an explicit owner-approved threat model before this file-write surface is safe to merge.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff changes systemd unit names, systemctl operands, and secret-bearing unit/drop-in writes, and those paths still have concrete hardening gaps.

Concerns:

- **[high] Option-like unit names can reach systemctl:** `src/daemon/systemd.ts:55`
  - body: The PR validation allows leading '-' names and systemctl calls do not terminate operands with '--', so an environment-provided unit name can be interpreted as a systemctl option.
  - confidence: 0.91
- **[high] HOME-derived writes can follow attacker-controlled links:** `src/daemon/systemd.ts:627`
  - body: The PR writes backups, main units, and a managed drop-in under HOME-derived paths with copyFile/writeFile; the drop-in may contain secrets and should not follow symlink or hardlink redirection without an approved threat model.
  - confidence: 0.84
- **[medium] Environment rewrite can remove clearing assignments:** `src/daemon/systemd-unit.ts:270`
  - body: Dropping Environment=FOO= while stripping managed keys can re-enable inherited environment state such as loader-related variables, which is both compatibility and security-sensitive.
  - confidence: 0.86
