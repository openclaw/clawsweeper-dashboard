  - body: Current main retries `daemon-reload` and the failed `enable`/`restart` once when systemd reports that the freshly written unit does not exist, with coverage in `src/daemon/systemd.test.ts`.   - confidence: 0.91

  - body: The branch resolves the gateway service name for uninstall, so `OPENCLAW_SYSTEMD_UNIT=openclaw-node` disables `openclaw-gateway.service` while removing the node unit file. Current main uses `resolveSystemdServiceName(env)` here and has a node uninstall regression test; the refresh needs to preserve that behavior.
  - confidence: 0.9
- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:55-63`
  - body: The new validation rejects path-like overrides but still allows names beginning with `-`, and the `systemctl` calls pass the unit operand without a `--` terminator. An `OPENCLAW_SYSTEMD_UNIT` value such as `--user` or `-Hhost` can still be parsed as a systemctl option rather than a service operand.
  - confidence: 0.84
- **[P2] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:270-272`
  - body: When stripping only some assignments from a mixed `Environment=` line, the code re-renders kept entries through `renderEnvLines`, which filters empty or whitespace-only values and trims the rest. That changes valid systemd semantics such as `Environment=LD_PRELOAD=` or values with intentional surrounding spaces during migration.
  - confidence: 0.83
- **[P1] Harden managed unit and drop-in writes:** `src/daemon/systemd.ts:627-638`
  - body: The migration writes the main unit, backup, and managed drop-in directly to predictable HOME-derived paths. Because the drop-in can contain token-bearing environment values, the PR needs either no-symlink/atomic write handling or an explicit threat-model decision before this new writer shape lands.
  - confidence: 0.76

## Security Review

Status: needs_attention

Summary: The diff changes token-bearing systemd storage and direct file writes under HOME-derived paths, with unresolved option-like unit names and symlink/hardlink overwrite concerns.

Concerns:

- **[high] Option-like unit names can reach systemctl:** `src/daemon/systemd.ts:55`
  - body: `validateSystemdServiceName` accepts leading `-`, and `systemctl` invocations pass the unit name without a `--` separator, so a hostile or malformed `OPENCLAW_SYSTEMD_UNIT` can be interpreted as an option.
  - confidence: 0.84
- **[high] Predictable unit/drop-in writes can follow links:** `src/daemon/systemd.ts:627`
  - body: The PR writes and backs up service files with direct `copyFile`/`writeFile` calls on HOME-derived paths; the new managed drop-in may contain secrets, so link traversal or overwrite behavior needs an explicit fix or threat-model decision.
  - confidence: 0.76
- **[medium] Mixed Environment rewrite can drop clearing assignments:** `src/daemon/systemd-unit.ts:270`
  - body: Partially stripping `Environment=` lines through `renderEnvLines` drops empty values such as `LD_PRELOAD=`, which can change inherited environment behavior in a security-sensitive service context.
  - confidence: 0.78
