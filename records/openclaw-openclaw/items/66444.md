Overall correctness: patch is incorrect

Full review comments:

- **[P1] Preserve the activation retry:** `src/daemon/systemd.ts:735-740`
  - body: This branch's `activateSystemdService` does a single reload, enable, and restart. Current main retries `daemon-reload` plus the failed action when systemd reports `Unit file ... does not exist`; dropping that behavior can bring back the migrated-install race where a freshly written unit is not visible yet.
  - confidence: 0.92
- **[P1] Honor custom unit names during uninstall:** `src/daemon/systemd.ts:794-796`
  - body: `uninstallSystemdService` disables the gateway-derived unit but removes the path resolved from `OPENCLAW_SYSTEMD_UNIT`. With `OPENCLAW_SYSTEMD_UNIT=openclaw-node`, uninstall can disable the wrong unit while deleting the node unit/drop-in, reintroducing the custom-unit mismatch fixed on current main.
  - confidence: 0.91
- **[P1] Reject option-like unit names:** `src/daemon/systemd.ts:53-63`
  - body: The validator still allows names beginning with `-`, and the resulting unit name is passed to `systemctl` as an operand without a `--` terminator. A value like `OPENCLAW_SYSTEMD_UNIT=-Hfoo` can be parsed as a `systemctl` option instead of a unit name, so the validator should reject leading dashes or every operand call should terminate options consistently.
  - confidence: 0.88
- **[P1] Preserve kept Environment values when stripping:** `src/daemon/systemd-unit.ts:270-272`
  - body: When only some assignments are stripped from an `Environment=` line, the kept values are re-rendered through `renderEnvLines`, which filters `value.trim()` and trims the value. That silently drops valid assignments like `FOO=` or changes whitespace-only values during migration, changing systemd semantics for existing units.
  - confidence: 0.86
- **[P1] Harden systemd unit and drop-in writes:** `src/daemon/systemd.ts:637-638`
  - body: The new migration writes the main unit, predictable backup, and managed drop-in using direct `copyFile`/`writeFile` under a HOME-derived systemd directory. Because the drop-in can contain service environment data, this needs no-follow/atomic write handling or an explicit accepted threat model before merge.
  - confidence: 0.84
- **[P3] Remove the release changelog edit:** `CHANGELOG.md:17`
  - body: Normal PRs should not edit `CHANGELOG.md`; release generation owns that file. Keep the release-note context in the PR body or squash message instead, especially because this stale diff also touches unrelated changelog entries.
  - confidence: 0.78
