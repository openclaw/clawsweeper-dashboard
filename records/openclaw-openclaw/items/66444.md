## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:55-63`
  - body: The new validator still accepts names starting with `-`, and the systemctl calls pass the unit operand without a `--` terminator, so `OPENCLAW_SYSTEMD_UNIT=--user` or similar option-like values can still be interpreted by systemctl rather than treated as a unit name. Reject leading dashes and/or terminate systemctl options consistently.
  - confidence: 0.91
- **[P1] Avoid direct writes for secret-bearing drop-ins:** `src/daemon/systemd.ts:627-638`
  - body: The new migration writes backups, rewritten main units, and the managed token-bearing drop-in with direct HOME-derived `copyFile`/`writeFile` paths. A symlink or hardlink planted under the user systemd directory can redirect those writes or leak managed environment values, so the writer needs a no-follow/atomic path or an explicit maintainer-approved threat model before merge.
  - confidence: 0.86
- **[P1] Preserve the missing-unit activation retry:** `src/daemon/systemd.ts:735`
  - body: Current main retries `daemon-reload` plus the failed enable/restart when systemd says the unit file does not exist, but this PR head still enables and restarts directly after a single reload. Carrying the branch forward without this retry reopens the migrated-systemd install failure that current main already fixed.
  - confidence: 0.9
- **[P1] Honor custom systemd units during uninstall:** `src/daemon/systemd.ts:794-795`
  - body: The PR head disables `resolveGatewaySystemdServiceName(...)` during uninstall even though the unit path uses the `OPENCLAW_SYSTEMD_UNIT` override. For node/custom-unit installs, uninstall can disable the gateway service name while removing a different file, leaving the actual custom unit enabled.
  - confidence: 0.9
- **[P1] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:270-272`
  - body: When stripping one assignment from a multi-assignment `Environment=` line, the rewrite re-renders the remaining keys through `renderEnvLines`, which filters empty values and trims whitespace. That changes valid systemd semantics such as `Environment=LD_PRELOAD=` and can silently alter an operator's existing unit during migration.
  - confidence: 0.84
- **[P2] Remove the release-owned changelog edit:** `CHANGELOG.md:18`
  - body: Normal PRs should not edit CHANGELOG.md, and this branch also removes unrelated existing release entries from the file. Keep release-note context in the PR body or squash message and leave changelog generation to the release workflow.
  - confidence: 0.96

## Security Review

Status: needs_attention

Summary: The diff touches systemd command execution and token-bearing service-file writes, and concrete option-injection, symlink/write-path, secret-permission, and environment-rewrite concerns remain before merge.

Concerns:

- **[high] Option-like systemd unit names still pass validation:** `src/daemon/systemd.ts:55`
  - body: The new service-name validator rejects paths but not leading dashes, while systemctl operands are still passed without an end-of-options marker; a hostile or accidental OPENCLAW_SYSTEMD_UNIT can be parsed as a systemctl option.
  - confidence: 0.91
- **[high] Token-bearing drop-in uses direct HOME-derived writes:** `src/daemon/systemd.ts:637`
  - body: The PR writes backups and the managed drop-in through predictable HOME-derived paths without no-follow or atomic protections, so symlink/hardlink redirection can overwrite files or leak managed environment values in local threat models.
  - confidence: 0.86
- **[medium] Missing sentinel can put secrets in the main unit:** `src/daemon/systemd-unit.ts:102`
  - body: When OPENCLAW_SERVICE_MANAGED_ENV_KEYS is absent, the splitter treats all env as user-owned, and fresh main-unit writes use the default string write path; callers that pass secrets without the sentinel can still create a group/world-readable main unit.
  - confidence: 0.78
- **[medium] Environment rewrite can drop explicit empty values:** `src/daemon/systemd-unit.ts:270`
  - body: Partial Environment= rewrites use renderEnvLines, which drops empty values and trims whitespace; removing an explicit empty assignment can re-enable inherited process environment in security-sensitive service contexts.
  - confidence: 0.84
