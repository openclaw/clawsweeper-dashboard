## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:55-63`
  - body: `validateSystemdServiceName` still accepts names beginning with `-`, and the resulting unit name is passed to `systemctl` as an operand without a `--` terminator. An `OPENCLAW_SYSTEMD_UNIT` override like `--machine=...` can be parsed as a systemctl option rather than a unit name, so reject leading dashes and/or terminate operands consistently.
  - confidence: 0.91
- **[P1] Preserve the activation retry:** `src/daemon/systemd.ts:735`
  - body: Current main retries `enable` and `restart` after another `daemon-reload` when systemd reports the unit file does not exist, which protects the migrated root/systemd-user path from https://github.com/openclaw/openclaw/issues/65184. This branch falls back to a single `systemctl enable`, so a refresh must carry the retry before merge.
  - confidence: 0.9
- **[P1] Honor custom systemd units during uninstall:** `src/daemon/systemd.ts:794-796`
  - body: Current main disables `${resolveSystemdServiceName(env)}.service`, but this branch still computes uninstall from `resolveGatewaySystemdServiceName(env.OPENCLAW_PROFILE)`. With `OPENCLAW_SYSTEMD_UNIT=openclaw-node`, uninstall can disable the gateway unit and leave the node unit behind, reintroducing the node-only failure tracked by https://github.com/openclaw/openclaw/issues/68287.
  - confidence: 0.9
- **[P1] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:270-273`
  - body: When a mixed `Environment=` line is partially stripped, the kept assignments are re-rendered through `renderEnvLines`, which filters on `value.trim()` and emits `rawValue.trim()`. That drops valid empty assignments such as `LD_PRELOAD=` and trims intentional whitespace, changing operator-authored systemd semantics during migration.
  - confidence: 0.88
- **[P1] Avoid direct writes for token-bearing drop-ins:** `src/daemon/systemd.ts:637-638`
  - body: The new managed drop-in can contain gateway tokens but is written directly to a predictable HOME-derived path with `fs.writeFile`, while backups use a predictable `.bak` path. Before landing, the branch needs no-symlink/permission hardening or an explicit owner-approved threat model for these token-bearing service files.
  - confidence: 0.82
- **[P2] Remove the release-owned changelog edit:** `CHANGELOG.md:17`
  - body: Root policy says `CHANGELOG.md` is release-owned for normal PRs, and this diff also deletes unrelated existing entries. Keep the release-note context in the PR body or squash message, but remove the changelog modification from the branch.
  - confidence: 0.96
