## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:62`
  - body: `validateSystemdServiceName` still accepts names that start with `-`, and this branch passes `unitName` directly to `systemctl` calls. An `OPENCLAW_SYSTEMD_UNIT` value such as `--user` or `-Hhost` can be parsed as a systemctl option, so reject leading dashes and/or pass `--` before unit operands consistently.
  - confidence: 0.94
- **[P1] Preserve the missing-unit activation retry:** `src/daemon/systemd.ts:735-737`
  - body: Current main retries `daemon-reload` plus the failed `enable` or `restart` once when systemd reports `Unit file ... does not exist`; this branch calls `enable` once and throws. That reintroduces the migrated root/user-systemd install race that current main already fixed.
  - confidence: 0.96
- **[P1] Honor custom systemd unit names on uninstall:** `src/daemon/systemd.ts:794-796`
  - body: Uninstall computes the unit name from `resolveGatewaySystemdServiceName(env.OPENCLAW_PROFILE)` instead of `resolveSystemdServiceName(env)`, so `OPENCLAW_SYSTEMD_UNIT=openclaw-node` disables `openclaw-gateway.service` while removing the node unit file. Current main uses the override-aware resolver here.
  - confidence: 0.96
- **[P1] Harden managed unit and drop-in writes:** `src/daemon/systemd.ts:628-630`
  - body: The migration writes predictable HOME-derived unit, backup, and drop-in paths with `copyFile`/`writeFile`. Because the drop-in can contain managed secrets, this path needs no-follow or atomic safe-write handling, or an explicit maintainer-approved threat model, before it is safe to ship.
  - confidence: 0.86
- **[P2] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:271-273`
  - body: When stripping one key from a mixed `Environment=` directive, the kept assignments are re-rendered through `renderEnvLines`, which filters empty values and trims whitespace. That changes valid operator unit semantics such as `Environment=LD_PRELOAD=` after migration.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces security-sensitive systemd unit-name and file-write behavior that needs hardening or explicit maintainer threat-model approval before merge.

Concerns:

- **[high] Option-like systemctl operands remain possible:** `src/daemon/systemd.ts:62`
  - body: The service-name validator permits leading `-` and systemctl calls do not terminate options before the unit operand, so an override can be interpreted as a systemctl flag.
  - confidence: 0.94
- **[high] Managed secret drop-in uses direct HOME-derived writes:** `src/daemon/systemd.ts:628`
  - body: The new drop-in and backup paths are predictable under `HOME`; direct `copyFile` and `writeFile` can follow attacker-prepared links if the local threat model includes hostile writable systemd directories or elevated execution with attacker-controlled HOME.
  - confidence: 0.84
- **[medium] Sentinel-absent env can stay in the main unit:** `src/daemon/systemd-unit.ts:102`
  - body: When the managed-key sentinel is absent, the split helper treats all environment values as user-owned, so sensitive OpenClaw values can be rendered into the main unit instead of the locked-down drop-in.
