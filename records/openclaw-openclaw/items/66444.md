  - body: The new validation still allows names beginning with `-`, and service actions pass the unit operand without a `--` terminator. A custom `OPENCLAW_SYSTEMD_UNIT` such as `--machine=...` can be parsed by `systemctl` as an option instead of a unit, so reject leading dashes and/or terminate options consistently.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff still has concrete systemd security-boundary concerns around option-like unit names, secret-bearing file writes, and environment rewrite semantics.

Concerns:

- **[high] Option-like unit names can reach systemctl:** `src/daemon/systemd.ts:53`
  - body: `OPENCLAW_SYSTEMD_UNIT` validation accepts leading dashes and systemctl invocations do not consistently terminate options before the unit operand, so a custom unit name can be interpreted as a systemctl option.
  - confidence: 0.88
- **[high] Secret drop-in writes follow predictable paths:** `src/daemon/systemd.ts:637`
  - body: The managed drop-in can contain token values, but the PR writes it with direct `writeFile` under a HOME-derived path and backs up the main unit with direct `copyFile`, leaving symlink/hardlink overwrite concerns unresolved.
  - confidence: 0.82
- **[medium] Sentinel-missing env can fall back into the main unit:** `src/daemon/systemd-unit.ts:102`
  - body: When the managed-env sentinel is absent, `splitSystemdManagedEnvironment` treats every value as user-owned and the main unit write has no explicit hardened mode, so caller mistakes can put sensitive service env into the main unit rather than the locked-down drop-in.
  - confidence: 0.76
- **[medium] Environment rewrite can remove clearing assignments:** `src/daemon/systemd-unit.ts:270`
  - body: Dropping `FOO=` or trimming whitespace while rewriting mixed `Environment=` lines can change inherited environment behavior for the service, which can be security-relevant for loader or runtime flags.
  - confidence: 0.78
