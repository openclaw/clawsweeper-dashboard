- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:55-63`
  - body: The new validator still accepts names beginning with `-`, and the systemctl calls pass the unit operand without a `--` terminator. An `OPENCLAW_SYSTEMD_UNIT` value like `--user` or `-Hhost` can still be parsed as a systemctl option instead of a unit name.
  - confidence: 0.91
- **[P1] Avoid direct writes for secret-bearing drop-ins:** `src/daemon/systemd.ts:627-638`
  - body: The migration writes backups, rewritten main units, and the managed token-bearing drop-in through predictable HOME-derived `copyFile`/`writeFile` paths. Please use a no-follow/atomic write path or document an explicit accepted threat model before merging this security-sensitive storage change.
  - confidence: 0.86
- **[P1] Preserve the missing-unit activation retry:** `src/daemon/systemd.ts:735`
  - body: Current main retries `daemon-reload` plus the failed enable/restart when systemd reports the unit file does not exist, but this PR head enables directly after a single reload. Carrying this forward would reopen the migrated-install race that current main already fixed.
  - confidence: 0.9
- **[P1] Honor custom systemd units during uninstall:** `src/daemon/systemd.ts:794-795`
  - body: The PR head disables `resolveGatewaySystemdServiceName(...)` even though the file path uses the `OPENCLAW_SYSTEMD_UNIT` override. For node or custom-unit installs, uninstall can disable the gateway unit while removing a different service file.
  - confidence: 0.9
- **[P1] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:270-272`
  - body: When stripping one assignment from a multi-assignment `Environment=` line, the kept entries are re-rendered through `renderEnvLines`, which drops empty values and trims whitespace. That changes valid systemd semantics such as `Environment=LD_PRELOAD=` during migration.
  - confidence: 0.84
- **[P2] Remove the release-owned changelog edit:** `CHANGELOG.md:20`
  - body: This normal PR edits `CHANGELOG.md` and removes unrelated existing entries. Keep the release-note context in the PR body or squash message and leave changelog generation to the release workflow.
  - confidence: 0.96

## Security Review

Status: needs_attention

Summary: The diff touches systemd command execution and token-bearing service-file writes, with concrete option-injection, symlink/write-path, secret-permission, and environment-rewrite concerns still open.

Concerns:

- **[high] Option-like systemd unit names still pass validation:** `src/daemon/systemd.ts:55`
  - body: The added service-name validator rejects paths but not leading dashes, while unit operands are still passed to systemctl without an end-of-options marker.
  - confidence: 0.91
- **[high] Token-bearing drop-in uses direct HOME-derived writes:** `src/daemon/systemd.ts:637`
  - body: The managed environment drop-in and backups are written through predictable paths without no-follow or atomic protections, so symlink/hardlink redirection remains a plausible local overwrite or disclosure risk.
  - confidence: 0.86
- **[medium] Missing sentinel can put secrets in the main unit:** `src/daemon/systemd-unit.ts:102`
  - body: When the managed-env sentinel is absent, the splitter treats all environment entries as user-owned and the fresh main unit write uses the default string write path, which can expose secrets if a caller omits the sentinel.
  - confidence: 0.78
- **[medium] Environment rewrite can drop explicit empty values:** `src/daemon/systemd-unit.ts:270`
  - body: Partial Environment= rewrites use a renderer that filters empty values and trims whitespace, which can remove security-relevant environment clearing assignments.
  - confidence: 0.84
