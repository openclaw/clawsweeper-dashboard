  - body: Current `main` retries `daemon-reload` plus the failed `enable`/`restart` when systemd says the just-written unit does not exist. This branch calls `enable` once, so the migrated root/user-systemd failure fixed by `e4e69c5bc6` can come back after a refresh or merge.
  - confidence: 0.91

  - body: `uninstallSystemdService` disables `resolveGatewaySystemdServiceName(...)` even though it removes `resolveSystemdUnitPath(env)`. With `OPENCLAW_SYSTEMD_UNIT=openclaw-node`, uninstall targets `openclaw-gateway.service` and leaves or fails the node unit, regressing the custom-unit fix from current `main`.
  - confidence: 0.93
- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:55-63`
  - body: The new validator still accepts names beginning with `-`, and the `systemctl` calls pass the unit operand without `--`. A custom unit such as `--user.service` can still be parsed as an option instead of a unit name, so the hardening is incomplete.
  - confidence: 0.86
- **[P1] Harden managed unit and drop-in writes:** `src/daemon/systemd.ts:627-638`
  - body: The migration writes predictable HOME-derived unit, backup, and drop-in paths with `writeFile`/`copyFile`. Because the drop-in carries managed environment values, the branch needs no-follow or equivalent symlink/hardlink handling and explicit permission behavior before this can safely rewrite existing service files.
  - confidence: 0.8
- **[P2] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:270-272`
  - body: When stripping one assignment from a multi-assignment `Environment=` line, this rebuilds the kept values through `renderEnvLines`, which filters empty values and trims whitespace. Valid operator-owned settings such as `Environment=LD_PRELOAD=` can disappear during migration.
  - confidence: 0.87
- **[P2] Remove the changelog edit:** `CHANGELOG.md:17`
  - body: `CHANGELOG.md` is release-owned in this repo, and this normal PR should keep release-note context in the PR body or final commit message instead of editing generated release notes.
  - confidence: 0.95

## Security Review

Status: needs_attention

Summary: The diff changes secret-bearing systemd file writes and does not yet settle unit-name option injection, HOME-derived file overwrite, or environment rewrite semantics.

Concerns:

- **[high] Option-like systemd unit names remain accepted:** `src/daemon/systemd.ts:55`
  - body: `validateSystemdServiceName` allows a leading dash, and `systemctl` calls do not terminate options before the unit operand, so a custom unit can still be interpreted as a flag.
  - confidence: 0.86
- **[medium] Managed drop-in writes use predictable HOME-derived paths:** `src/daemon/systemd.ts:637`
  - body: The drop-in can contain managed tokens, but the branch writes and backs up files directly under a HOME-derived systemd directory without link-hardening or a documented threat-model decision.
  - confidence: 0.8
- **[medium] Environment rewrite can remove explicit empty values:** `src/daemon/systemd-unit.ts:270`
  - body: Dropping `Environment=FOO=` while stripping another key can change inherited environment behavior for the service, which can be security-sensitive for loader or runtime variables.
  - confidence: 0.82
