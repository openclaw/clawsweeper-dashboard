## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve the activation retry:** `src/daemon/systemd.ts:735-740`
  - body: This branch still runs `enable` and `restart` only once after `daemon-reload`. Current main retries a second `daemon-reload` plus the failed action when systemd reports the just-written unit does not exist; dropping that path reintroduces the migrated root/systemd-user install failure covered by the current tests.
  - confidence: 0.91
- **[P1] Honor custom unit names during uninstall:** `src/daemon/systemd.ts:794-796`
  - body: `uninstallSystemdService` resolves `openclaw-gateway` here even when `OPENCLAW_SYSTEMD_UNIT` points at `openclaw-node` or another custom unit. On node-only hosts this disables the wrong unit and can fail before removing the intended unit, which current main fixed by using the override-aware resolver.
  - confidence: 0.9
- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:55-62`
  - body: The new validator still accepts names beginning with `-`, and later calls pass the unit name to `systemctl` without a `--` operand terminator. A custom `OPENCLAW_SYSTEMD_UNIT` can therefore be parsed as a systemctl option instead of a unit operand, so reject leading dashes or terminate operands consistently.
  - confidence: 0.88
- **[P1] Preserve Environment assignment values:** `src/daemon/systemd-unit.ts:270-272`
  - body: When only some assignments on an `Environment=` line are stripped, this code rebuilds the remaining values through `renderEnvLines`, which filters empty values and trims whitespace. That changes valid systemd semantics like `Environment=LD_PRELOAD=` and can silently alter an operator-owned unit during migration.
  - confidence: 0.87
- **[P1] Harden the managed drop-in write:** `src/daemon/systemd.ts:637`
  - body: The new drop-in can contain managed service secrets, but it is written directly to a predictable HOME-derived path, so an existing symlink in the unit drop-in directory can redirect the write before chmod. Please use a no-follow/atomic write strategy or document an explicit threat-model decision before adding this new secret-bearing file path.
  - confidence: 0.77

## Security Review

Status: needs_attention

Summary: The diff touches systemd service control, secret-bearing unit/drop-in files, and custom unit-name validation, and it has concrete security-boundary concerns before merge.

Concerns:

- **[high] Option-like unit names can reach systemctl:** `src/daemon/systemd.ts:55`
  - body: The PR validation allows leading '-' and systemctl operands are not terminated with '--', so a custom unit name can be parsed as an option instead of a unit operand.
  - confidence: 0.88
- **[high] Secret drop-in write follows HOME-derived paths:** `src/daemon/systemd.ts:637`
  - body: The new managed drop-in write uses direct writeFile/chmod on a predictable path under HOME, which can follow pre-existing symlinks and redirect secret-bearing content unless explicitly hardened or threat-modeled.
  - confidence: 0.77
- **[medium] Environment rewrites can change security-sensitive semantics:** `src/daemon/systemd-unit.ts:271`
  - body: Dropping empty Environment= assignments during migration can re-enable inherited variables such as loader-related settings that an operator intentionally cleared.
