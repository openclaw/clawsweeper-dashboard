  - body: Current main retries `daemon-reload` plus the failed `enable` or `restart` once when systemd says the freshly written unit does not exist.   - confidence: 0.93

  - body: The PR head disables `${resolveGatewaySystemdServiceName(...)}.service` while deleting the path resolved from `OPENCLAW_SYSTEMD_UNIT`. For node or custom units, that disables the gateway unit and removes the custom unit file, reintroducing the wrong-unit behavior current main fixed.
  - confidence: 0.94
- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:53-62`
  - body: `SYSTEMD_SERVICE_NAME_PATTERN` still allows names that start with `-`, and the validator does not reject them before those names are passed as systemctl operands. `OPENCLAW_SYSTEMD_UNIT=-Hhost` can therefore be interpreted as a systemctl option rather than a unit name.
  - confidence: 0.9
- **[P1] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:270-273`
  - body: When only some assignments are stripped from an `Environment=` line, the kept assignments are re-rendered through `renderEnvLines`, which drops empty values and trims whitespace. That changes valid operator systemd semantics such as `LD_PRELOAD=` during an upgrade migration.
  - confidence: 0.88
- **[P1] Harden managed unit and drop-in writes:** `src/daemon/systemd.ts:627-638`
  - body: The new managed drop-in stores service credentials under predictable HOME-derived paths and is written with direct `writeFile`, while backups use direct `copyFile` to a predictable `.bak` path. Before this new secret-bearing file becomes an upgrade path, the branch needs a safe-write strategy or explicit owner threat-model acceptance.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a new secret-bearing systemd drop-in path and still has concrete unit-name, path-write, and environment-rewrite security-boundary concerns.

- **[high] Option-like systemd unit names still pass validation:** `src/daemon/systemd.ts:53`
  - body: The PR accepts leading-dash `OPENCLAW_SYSTEMD_UNIT` values and passes them to systemctl as operands without an end-of-options terminator, so systemctl can parse them as options.
