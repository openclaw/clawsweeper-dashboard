## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:53-63`
  - body: `validateSystemdServiceName` still accepts service names that start with `-`, and this PR passes the resulting unit name to `systemctl enable`/`restart` without a `--` terminator. A value like `OPENCLAW_SYSTEMD_UNIT=--user` can be parsed as a systemctl option instead of a unit operand, so reject leading dashes or terminate systemctl options consistently.
  - confidence: 0.86
- **[P1] Use the override service name for uninstall:** `src/daemon/systemd.ts:794-796`
  - body: Install activation now honors `OPENCLAW_SYSTEMD_UNIT`, but uninstall still disables `resolveGatewaySystemdServiceName(env.OPENCLAW_PROFILE)`. For node or custom units, uninstall can disable `openclaw-gateway.service` while removing the custom unit path, leaving the actual service enabled or targeting the wrong unit.
  - confidence: 0.9
- **[P1] Harden managed drop-in writes before storing secrets:** `src/daemon/systemd.ts:637-638`
  - body: The new managed drop-in stores secret-bearing environment values, but the writer uses predictable HOME-derived paths and direct `writeFile`/`copyFile` operations. Before moving tokens into this new file, the branch needs either no-symlink/permission hardening or an explicit maintainer threat-model decision that this write pattern is acceptable.
  - confidence: 0.82
- **[P2] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:270-273`
  - body: When a mixed `Environment=` line is partially stripped, the kept assignments are re-rendered through `renderEnvLines`, which drops empty or whitespace-only values and trims all values. That changes valid systemd semantics such as `Environment=LD_PRELOAD=` during migration, so preserve kept operator-owned assignments instead of normalizing them.
  - confidence: 0.84
- **[P2] Remove release-owned changelog edits:** `CHANGELOG.md:17`
  - body: This normal PR edits `CHANGELOG.md` and also removes unrelated release entries from older sections. The repository policy keeps the changelog release-owned; put release-note context in the PR body or squash message and leave the changelog to release generation.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff changes where secret-bearing service environment values are written and still needs hardening or explicit maintainer threat-model acceptance.

Concerns:

- **[high] Option-like unit names can cross the systemctl option boundary:** `src/daemon/systemd.ts:53`
  - body: Service-name validation permits leading dashes while systemctl invocations pass the unit name without an option terminator, leaving an avoidable command-boundary risk for OPENCLAW_SYSTEMD_UNIT overrides.
  - confidence: 0.86
- **[medium] Managed drop-in write path stores secrets under HOME-derived paths:** `src/daemon/systemd.ts:637`
  - body: The PR writes the main unit, backup, and managed drop-in with direct filesystem operations under HOME-derived systemd paths; because the drop-in carries secret values, the symlink/write and permission threat model needs to be settled before merge.
  - confidence: 0.82
- **[medium] Environment rewrite can remove clearing assignments:** `src/daemon/systemd-unit.ts:270`
  - body: Dropping `Environment=KEY=` while stripping another assignment can re-enable inherited process environment unexpectedly, including loader-related variables such as `LD_PRELOAD`.
  - confidence: 0.76
