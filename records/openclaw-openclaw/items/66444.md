## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:56-63`
  - body: `SYSTEMD_SERVICE_NAME_PATTERN` still accepts names beginning with `-`, and the new code later passes that unit name directly to `systemctl` as a positional operand. A value such as `OPENCLAW_SYSTEMD_UNIT=-Hhost` can be parsed as a systemctl option rather than a unit name, so the validator should reject leading dashes and/or every systemctl unit call should terminate options with `--`.
  - confidence: 0.94
- **[P1] Preserve the activation reload retry:** `src/daemon/systemd.ts:735-740`
  - body: This head calls `systemctl enable` and `restart` once after `daemon-reload`, but current main added a retry for missing-unit responses in `e4e69c5bc6` to fix the migrated systemd race. Refreshing this branch without carrying that logic would bring back the `Unit file ... does not exist` install failure.
  - confidence: 0.93
- **[P1] Use the resolved unit name during uninstall:** `src/daemon/systemd.ts:794-796`
  - body: `uninstallSystemdService` still derives the target from `resolveGatewaySystemdServiceName(env.OPENCLAW_PROFILE)`, so `OPENCLAW_SYSTEMD_UNIT=openclaw-node` disables `openclaw-gateway.service` while deleting the node unit file. Current main fixed this in `eca9f468246e`, so the carry needs to use `resolveSystemdServiceName(env)` here too.
  - confidence: 0.95
- **[P1] Harden managed unit and drop-in writes:** `src/daemon/systemd.ts:627-630`
  - body: The migration adds a new managed drop-in that can contain service secrets, but writes the main unit, backup, and drop-in through predictable HOME-derived paths with direct `copyFile`/`writeFile`. Before merging a new secret-bearing path, this needs no-symlink/safe-write handling or an explicit maintainer-approved threat model.
  - confidence: 0.88
- **[P1] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:270-274`
  - body: When only some assignments are stripped from an `Environment=` line, the remaining entries are re-rendered through `renderEnvLines`, which filters out empty or whitespace-only values and trims the rest. Existing units can intentionally use assignments such as `LD_PRELOAD=` to clear inherited environment, so migration must preserve those values rather than silently deleting or changing them.
  - confidence: 0.9
- **[P2] Remove release-owned changelog edits:** `CHANGELOG.md:17`
  - body: This PR adds a new `CHANGELOG.md` entry and removes unrelated existing entries, but the repository policy makes `CHANGELOG.md` release-owned. Keep the release-note context in the PR body or squash message and leave changelog generation to the release flow.
  - confidence: 0.92
