Full review comments:

- **[P1] Preserve the missing-unit activation retry:** `src/daemon/systemd.ts:735-740`
  - body: Current main retries `daemon-reload` plus the failed enable/restart once when systemd says the freshly written unit does not exist, but this PR head goes straight from one `enable` to an error. Carrying this forward would re-open the install recovery failure tracked by https://github.com/openclaw/openclaw/issues/65184.
  - confidence: 0.93
- **[P1] Honor custom systemd units during uninstall:** `src/daemon/systemd.ts:794-796`
  - body: This path resolves the gateway service name instead of the full environment override, so `OPENCLAW_SYSTEMD_UNIT=openclaw-node` disables `openclaw-gateway.service` while deleting the node unit path. Current main uses the override-aware resolver and has regression coverage for this behavior.
  - confidence: 0.94
- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:53-62`
  - body: The new validation still permits names beginning with `-`, and the changed systemctl calls pass the unit operand without `--`. An `OPENCLAW_SYSTEMD_UNIT` value such as `--user` can still be parsed by systemctl as an option rather than a unit name.
  - confidence: 0.9
- **[P1] Avoid direct writes for secret-bearing drop-ins:** `src/daemon/systemd.ts:627-638`
  - body: The new managed drop-in can contain tokens, but the writer uses predictable HOME-derived paths with direct `copyFile`/`writeFile` calls and no no-follow or atomic-write guard. This needs either a safe-write helper or an explicit threat-model decision before moving secrets into that path.
  - confidence: 0.86
- **[P1] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:269-272`
  - body: When stripping one assignment from a mixed `Environment=` line, the code re-renders the kept values through `renderEnvLines`, which filters empty values and trims whitespace. That changes valid systemd semantics such as `Environment=LD_PRELOAD=` after a managed key is removed.
  - confidence: 0.92
- **[P2] Remove the release-owned changelog edit:** `CHANGELOG.md:20`
  - body: Root policy says CHANGELOG.md is release-generated and should not be edited by ordinary PRs. Move this release-note context to the PR body or squash message and leave the changelog to release generation.
  - confidence: 0.95
