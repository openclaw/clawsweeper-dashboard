  - body: The PR head still runs `enable` and `restart` only once after `daemon-reload`.   - confidence: 0.91
- **[P1] Honor custom unit names during uninstall:** `src/daemon/systemd.ts:794`
  - body: `uninstallSystemdService` still disables `resolveGatewaySystemdServiceName(...)` instead of the resolved unit override, so an install using `OPENCLAW_SYSTEMD_UNIT=openclaw-node` can disable/remove the wrong unit. Current main uses `resolveSystemdServiceName(env)` and has a regression test for the node-unit path.
  - confidence: 0.94
- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:55-63`
  - body: The new validator still accepts values that start with `-`, and systemctl calls pass the unit operand without a `--` terminator. That leaves `OPENCLAW_SYSTEMD_UNIT=--user` or similar option-like names in the command surface despite the PR claiming unsafe custom unit names are rejected.
  - confidence: 0.89
- **[P1] Harden managed unit and drop-in writes:** `src/daemon/systemd.ts:637-638`
  - body: The PR adds a secret-bearing managed drop-in under a HOME-derived systemd directory but writes the unit, backup, and drop-in with direct `copyFile`/`writeFile` calls. Before landing this security-sensitive file split, the branch needs no-symlink/safe-write handling or an explicit maintainer threat-model decision.
  - confidence: 0.82
- **[P1] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:270-274`
  - body: When only some assignments are stripped, this path re-renders the kept assignments through `renderEnvLines`, which filters empty values and trims whitespace. That can silently remove valid operator settings such as `Environment=LD_PRELOAD=` during migration and change the effective service environment.
  - confidence: 0.87
