Overall confidence: 0.91

Full review comments:

- **[P1] Preserve the missing-unit activation retry:** `src/daemon/systemd.ts:735-740`
  - body: This branch still calls `systemctl enable` and `restart` only once after `daemon-reload`. Current main retries `daemon-reload` plus the failed action when systemd reports `Unit file ... does not exist`; carrying this head forward without that behavior reopens the install/update race that current main already fixed.
  - confidence: 0.93
- **[P1] Honor custom unit names during uninstall:** `src/daemon/systemd.ts:794-796`
  - body: `uninstallSystemdService` computes the unit with `resolveGatewaySystemdServiceName`, so `OPENCLAW_SYSTEMD_UNIT=openclaw-node` still disables the gateway unit while deleting the node unit path. Current main uses `resolveSystemdServiceName(env)` here; the carry needs to preserve that behavior.
  - confidence: 0.94
- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:55-63`
  - body: The new validation still accepts unit names beginning with `-`, and the later `systemctl` calls pass the unit operand without `--`. A value like `--machine=...` or `-H...` can be parsed as a systemctl option rather than a unit name, so the hardening is incomplete.
  - confidence: 0.88
- **[P1] Harden managed unit and drop-in writes:** `src/daemon/systemd.ts:637`
  - body: The PR writes the backup, main unit, and managed drop-in through predictable HOME-derived paths using direct `copyFile`/`writeFile`. Because the new drop-in can carry managed service secrets, this needs nofollow/atomic-write handling or explicit maintainer-approved threat-model coverage before merge.
  - confidence: 0.84
- **[P2] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:270-272`
  - body: When only some assignments are stripped, the remaining entries are re-rendered through `renderEnvLines`, which drops empty/whitespace-only values and trims values. That changes valid systemd settings such as `Environment=LD_PRELOAD=` after a managed key is removed from the same line.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff still needs security attention for option-like unit names, secret-bearing HOME-derived file writes, and environment rewrite semantics.

Concerns:

- **[high] Option-like systemd unit names still pass validation:** `src/daemon/systemd.ts:55`
  - body: `OPENCLAW_SYSTEMD_UNIT` can still begin with `-`, while systemctl operands are passed without an end-of-options marker; this can change how systemctl interprets the argument.
  - confidence: 0.88
- **[high] Managed drop-in write can follow attacker-controlled links:** `src/daemon/systemd.ts:637`
  - body: The new managed drop-in can contain service secrets and is written to a predictable HOME-derived path with direct `writeFile`, without nofollow/atomic write protection or a documented accepted threat model.
  - confidence: 0.84
- **[medium] Partial Environment rewrites can remove safety-clearing values:** `src/daemon/systemd-unit.ts:270`
  - body: Re-rendering kept assignments via `renderEnvLines` can drop `KEY=` entries that intentionally clear inherited environment, which can alter service execution semantics.
