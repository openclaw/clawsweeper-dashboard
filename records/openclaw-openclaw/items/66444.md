- **[P1] Preserve the activation retry from current main:** `src/daemon/systemd.ts:735-740`
  - body: Current main retries `daemon-reload` plus the failed `enable`/`restart` once when systemd reports the written unit does not exist. This PR head calls `enable` and `restart` only once, so refreshing/merging it as-is would reintroduce the install/update race fixed for migrated root and node installs.
  - confidence: 0.93
- **[P1] Honor custom unit names during uninstall:** `src/daemon/systemd.ts:794-796`
  - body: `uninstallSystemdService` still disables `${resolveGatewaySystemdServiceName(...)}.service` instead of the resolved `OPENCLAW_SYSTEMD_UNIT`. For node/custom units this can disable the gateway unit while deleting the custom unit file/drop-in, regressing current-main custom-unit uninstall behavior.
  - confidence: 0.94
- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:55-63`
  - body: The new validator still allows names that start with `-`, and the systemctl calls pass the unit operand without a `--` terminator. An `OPENCLAW_SYSTEMD_UNIT` override like `--user` or `-Hhost` can still be parsed as a systemctl option rather than a unit name.
  - confidence: 0.9
- **[P1] Harden managed unit and drop-in writes:** `src/daemon/systemd.ts:627-638`
  - body: The new migration writes and backs up predictable HOME-derived unit/drop-in paths with direct `copyFile`/`writeFile`. Because the drop-in can contain managed service secrets, this needs safe no-symlink/atomic handling or an explicit approved threat model before moving secrets into the new path.
  - confidence: 0.86
- **[P2] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:269-273`
  - body: When only some assignments are stripped from an `Environment=` line, the kept assignments are re-rendered through `renderEnvLines`, which filters out empty values and trims whitespace. That changes valid operator unit semantics such as `Environment=LD_PRELOAD=` during migration.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff changes service-secret storage and systemctl operand handling, and the current head still has concrete security-boundary concerns before merge.

Concerns:

- **[high] Option-like systemd unit names remain accepted:** `src/daemon/systemd.ts:62`
  - body: The validator permits leading `-` while systemctl calls do not terminate options before the unit operand, so a custom unit override can be interpreted as a systemctl option.
  - confidence: 0.9
- **[high] Managed drop-in writes need symlink-safe handling:** `src/daemon/systemd.ts:637`
  - body: The PR introduces a secret-bearing managed drop-in under a HOME-derived path and writes it directly, leaving a concrete symlink/hardlink overwrite concern unless maintainers approve that threat model.
  - confidence: 0.84
- **[medium] Environment rewrite can remove clearing assignments:** `src/daemon/systemd-unit.ts:271`
  - body: Dropping `Environment=FOO=` during migration can unintentionally re-enable inherited process environment, which is security-sensitive for variables such as dynamic-loader settings.
  - confidence: 0.82
