## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve the activation retry from current main:** `src/daemon/systemd.ts:735-740`
  - body: PR head runs `enable` and `restart` once after `daemon-reload`, but current main retries the failed action after a second reload when systemd says the freshly written unit does not exist. Refreshing without that path reopens the migrated/root systemd install failure that was fixed on main.
  - confidence: 0.94
- **[P1] Honor custom unit names during uninstall:** `src/daemon/systemd.ts:794-796`
  - body: `uninstallSystemdService` resolves the disable target from the gateway profile instead of `resolveSystemdServiceName(env)`, so `OPENCLAW_SYSTEMD_UNIT=openclaw-node` can remove the node unit file while disabling the gateway unit. Current main already fixes this by targeting the override-aware unit name.
  - confidence: 0.96
- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:62`
  - body: The new validator still accepts names that start with `-`, and the unit name is later passed to `systemctl` without an end-of-options marker. A custom `OPENCLAW_SYSTEMD_UNIT` that begins with `-` can be parsed as a systemctl option instead of a unit operand, so reject leading dashes or consistently pass `--` before unit names.
  - confidence: 0.9
- **[P1] Preserve kept Environment assignments exactly:** `src/daemon/systemd-unit.ts:270-273`
  - body: When only part of an `Environment=` line is stripped, the kept assignments are re-rendered through `renderEnvLines`, which filters on `value.trim()` and trims values. That can drop valid assignments such as `FOO=` or change whitespace-only values while migrating an operator-edited unit.
  - confidence: 0.88
- **[P1] Preserve existing env-file entries when rewriting:** `src/daemon/systemd.ts:677-680`
  - body: This writer rebuilds `gateway.systemd.env` only from current state-dir dotenv values and does not read/merge existing operator-only entries. Current main preserves existing env-file secrets and node env-file migrations; without that carry, a restage can silently drop operator-provided env values from the service file path.
  - confidence: 0.86
- **[P1] Harden managed unit and drop-in writes:** `src/daemon/systemd.ts:628-637`
  - body: The migration writes predictable HOME-derived unit, backup, and drop-in paths with direct `copyFile`/`writeFile`; the drop-in can contain managed secrets. Before shipping this security-sensitive path, use a no-symlink/atomic write strategy or add an explicit threat-model decision plus permissions coverage for the main unit, backup, and drop-in.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff changes credential-bearing systemd file writes and still needs concrete unit-name, symlink/write-path, and Environment rewrite hardening before merge.

Concerns:

- **[high] Systemctl option injection through unit names:** `src/daemon/systemd.ts:62`
  - body: `validateSystemdServiceName` permits leading `-`, and systemctl unit operands are passed without `--`; an override-like unit name could be parsed as a systemctl option.
  - confidence: 0.9
- **[high] Credential-bearing files use predictable write paths:** `src/daemon/systemd.ts:628`
  - body: The PR copies and writes HOME-derived unit, backup, and drop-in paths directly; if those paths are link-controlled, managed env values or unit rewrites can be redirected.
  - confidence: 0.82
- **[medium] Environment rewrite can change inherited process state:** `src/daemon/systemd-unit.ts:270`
  - body: Dropping `FOO=` or trimming values while stripping a managed key changes systemd environment semantics and can re-enable inherited variables that the unit intentionally cleared.
  - confidence: 0.84
