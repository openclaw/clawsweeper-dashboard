Full review comments:

- **[P1] Preserve the activation retry:** `src/daemon/systemd.ts:735-740`
  - body: Current main retries `enable` and `restart` once after a second `daemon-reload` when systemd says the newly written unit does not exist, with a regression test for the node service path. This branch reverts that to a single enable/restart attempt, so updating from this PR can bring back the migrated-install failure current main already fixed.
  - confidence: 0.92
- **[P1] Honor custom unit names during uninstall:** `src/daemon/systemd.ts:794-796`
  - body: The uninstall path disables `resolveGatewaySystemdServiceName(env.OPENCLAW_PROFILE)` while deleting the override-resolved unit path. With `OPENCLAW_SYSTEMD_UNIT=openclaw-node`, that disables `openclaw-gateway.service` instead of the node service, regressing the current-main custom-unit uninstall fix.
  - confidence: 0.93
- **[P1] Reject option-like unit names:** `src/daemon/systemd.ts:55-63`
  - body: `validateSystemdServiceName` permits names that start with `-`, and the affected `systemctl` calls pass the unit operand without a `--` terminator. Reject leading dashes or terminate operands consistently so `OPENCLAW_SYSTEMD_UNIT` cannot be parsed as a systemctl option.
  - confidence: 0.86
- **[P1] Harden managed unit and drop-in writes:** `src/daemon/systemd.ts:627-638`
  - body: The migration backs up and writes predictable HOME-derived unit/drop-in paths with direct `copyFile` and `writeFile`, including secret-bearing managed env data. Before landing the new layout, use a no-follow/atomic write strategy or get an explicit accepted threat model for symlink, hardlink, and permission behavior.
  - confidence: 0.84
- **[P1] Preserve kept Environment values when stripping:** `src/daemon/systemd-unit.ts:270-273`
  - body: When a mixed `Environment=` line is partially stripped, the kept assignments are re-rendered through `renderEnvLines`, which filters empty values and trims values. That changes valid systemd semantics such as `Environment=LD_PRELOAD=`, so reconstruct kept assignments without dropping or trimming operator values.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff touches token-bearing systemd environment storage and HOME-derived unit/drop-in writes, with concrete option-injection, file-write, and environment-semantics concerns still unresolved.

Concerns:

- **[high] Option-like unit names can reach systemctl:** `src/daemon/systemd.ts:55`
  - body: `validateSystemdServiceName` permits leading `-`, and subsequent `systemctl` calls pass unit names without `--`, so an environment-provided unit override can be interpreted as an option rather than a unit operand.
  - confidence: 0.86
- **[high] Predictable service-file writes are not link-hardened:** `src/daemon/systemd.ts:627`
  - body: The new drop-in writer and backup path use direct copy/write calls on HOME-derived paths, which can overwrite or leak secret-bearing unit content through symlink or hardlink redirection in a hostile local environment.
  - confidence: 0.83
- **[medium] Migration can remove empty security-sensitive env assignments:** `src/daemon/systemd-unit.ts:270`
  - body: The rewrite path can drop empty kept assignments such as `LD_PRELOAD=`, changing the effective service environment during upgrade and potentially re-enabling inherited values an operator intentionally cleared.
  - confidence: 0.88
