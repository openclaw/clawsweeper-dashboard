## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve the activation retry:** `src/daemon/systemd.ts:735-740`
  - body: The PR head calls `enable` and `restart` once after `daemon-reload`. Current main retries the failed action after another reload when systemd reports the just-written unit is missing, and dropping that behavior can bring back install failures on migrated user managers.
  - confidence: 0.9
- **[P1] Honor custom unit names during uninstall:** `src/daemon/systemd.ts:794-796`
  - body: This uninstall path disables the gateway-derived unit but deletes the unit path resolved from `OPENCLAW_SYSTEMD_UNIT`. For node/custom installs that means the wrong service can be disabled while the target unit remains active or mismatched.
  - confidence: 0.9
- **[P1] Reject option-like unit names:** `src/daemon/systemd.ts:62`
  - body: The new validator still accepts service names beginning with `-`, and the systemctl calls pass the unit operand without a `--` terminator. A custom `OPENCLAW_SYSTEMD_UNIT` value such as an option-like name can be parsed by systemctl as a flag instead of a unit.
  - confidence: 0.84
- **[P1] Harden managed unit and drop-in writes:** `src/daemon/systemd.ts:627-638`
  - body: The migration copies and writes predictable HOME-derived unit and drop-in paths directly. Because the drop-in can contain managed tokens, the PR needs a no-follow/atomic-write strategy or an explicit accepted threat model before this file-writing path lands.
  - confidence: 0.86
- **[P1] Preserve kept Environment values when stripping:** `src/daemon/systemd-unit.ts:270-272`
  - body: When only some assignments on an `Environment=` line are stripped, the kept assignments are re-rendered through `renderEnvLines`, which filters empty values and trims whitespace. That changes valid systemd semantics such as `Environment=LD_PRELOAD=` during migration.
  - confidence: 0.88
