Full review comments:

- **[P1] Preserve the activation retry from current main:** `src/daemon/systemd.ts:735-740`
  - body: Current main retries daemon-reload plus the failed enable/restart when systemd reports the unit file does not exist. This PR head goes back to one-shot enable/restart calls, so a refresh that keeps this code can reintroduce the migrated root/user-systemd install failure covered by the current systemd tests.
  - confidence: 0.92
- **[P1] Honor custom unit names during uninstall:** `src/daemon/systemd.ts:794-796`
  - body: The install path resolves `OPENCLAW_SYSTEMD_UNIT`, but uninstall still derives the gateway service name. With `OPENCLAW_SYSTEMD_UNIT=openclaw-node`, this can disable `openclaw-gateway.service` while unlinking the node unit file, leaving the actual node service enabled.
  - confidence: 0.94
- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:53-63`
  - body: The new validation still accepts names beginning with `-`, and the later `systemctl` calls pass the unit operand without a `--` terminator. A custom unit name can therefore still be parsed as a systemctl option, so the hardening is incomplete for this security-sensitive env override.
  - confidence: 0.9
- **[P1] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:271-272`
  - body: When only some assignments are stripped, the remaining assignments are re-rendered through `renderEnvLines`, which filters empty values and trims values. Existing valid lines such as `Environment="LD_PRELOAD=" "OPENCLAW_GATEWAY_TOKEN=..."` can lose the empty assignment during migration and change the service environment.
  - confidence: 0.88
- **[P1] Harden managed unit and drop-in writes:** `src/daemon/systemd.ts:593`
  - body: The new managed drop-in can contain service secrets, but the writer uses direct writes and predictable backup/drop-in paths under HOME-derived directories. Before landing, this needs no-symlink/safe-write handling or an explicit threat-model decision for elevated or attacker-influenced HOME cases.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff touches secret-bearing systemd storage and command operands, and it still has concrete unit-name, file-write, and environment-preservation security concerns.

Concerns:

- **[high] Option-like systemd unit names remain accepted:** `src/daemon/systemd.ts:53`
  - body: `SYSTEMD_SERVICE_NAME_PATTERN` permits leading `-`, and systemctl calls do not insert `--` before the unit operand, so `OPENCLAW_SYSTEMD_UNIT` can still be interpreted as an option.
  - confidence: 0.9
- **[high] Secret-bearing drop-in writes follow predictable paths:** `src/daemon/systemd.ts:593`
  - body: The new managed drop-in may contain service tokens and is written under a HOME-derived path with direct `writeFile`/`chmod`; this needs no-symlink safe-write handling or an explicit local threat-model acceptance.
  - confidence: 0.82
- **[medium] Environment rewrite can remove security-clearing assignments:** `src/daemon/systemd-unit.ts:271`
  - body: Partial stripping re-renders kept assignments through a renderer that drops empty values, so a unit line intended to clear inherited variables can be changed during migration.
