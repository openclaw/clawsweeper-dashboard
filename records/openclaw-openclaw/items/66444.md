  - body: PR head does a single `daemon-reload`, then one `enable` and one `restart`. Current main retries `daemon-reload` plus the failed action when systemd reports the freshly written unit is missing, so this carry needs to preserve that behavior and its test before merge.
  - confidence: 0.94
- **[P1] Honor custom unit names during uninstall:** `src/daemon/systemd.ts:794-795`
  - body: `uninstallSystemdService` still disables the profile-derived gateway unit even when `OPENCLAW_SYSTEMD_UNIT` points at another service such as `openclaw-node`, while the file removal uses the override-derived path. That can leave the installed custom unit enabled and remove the wrong target.
  - confidence: 0.93
- **[P1] Reject option-like service names:** `src/daemon/systemd.ts:55-63`
  - body: The new validator still accepts names beginning with `-`, and later `systemctl` calls pass the unit name as an operand without `--`. A value such as `OPENCLAW_SYSTEMD_UNIT=-Hfoo` can still be interpreted as a systemctl option rather than a unit name.
  - confidence: 0.91
- **[P1] Preserve kept Environment assignments:** `src/daemon/systemd-unit.ts:270-274`
  - body: When stripping a managed key from a mixed `Environment=` line, the kept assignments are re-rendered through `renderEnvLines`, which drops empty values and trims whitespace. That changes valid systemd semantics like `Environment=LD_PRELOAD=` or intentional trailing-space values during migration.
  - confidence: 0.86
- **[P1] Harden managed unit and drop-in writes:** `src/daemon/systemd.ts:627-638`
  - body: The migration writes predictable HOME-derived unit, backup, and managed drop-in paths with direct copy/write calls. Because the drop-in can contain service tokens, this needs a threat-model decision or no-symlink/permission-safe write strategy before shipping.
  - confidence: 0.83
- **[P2] Drop the release-owned changelog edits:** `CHANGELOG.md:17`
  - body: This normal PR edits `CHANGELOG.md` and removes unrelated entries while adding its own release note. The repository policy makes changelog generation release-owned, so this context should move to the PR body or squash message instead.

## Security Review

Status: needs_attention

Summary: The diff touches secret-bearing systemd service files and still has concrete option-injection, write-path, and Environment= preservation concerns.

- **[high] Option-like unit names still pass validation:** `src/daemon/systemd.ts:55`
  - body: `SYSTEMD_SERVICE_NAME_PATTERN` allows leading dashes and the systemctl calls do not consistently terminate options before the unit operand, so an environment-controlled unit name can alter systemctl parsing.
