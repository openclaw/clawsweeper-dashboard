## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:55-63`
  - body: `validateSystemdServiceName` still accepts names beginning with `-`, and the later `systemctl` calls pass the unit operand without an end-of-options marker. A value like `OPENCLAW_SYSTEMD_UNIT=-Hhost` can be parsed as a systemctl option rather than a unit name; reject leading dashes or terminate options consistently before this path lands.
  - confidence: 0.94
- **[P1] Preserve the activation retry:** `src/daemon/systemd.ts:735-740`
  - body: The PR head calls `systemctl enable`/`restart` directly after the first daemon reload. Current main already retries the failed action after a second `daemon-reload` when systemd says the freshly written unit does not exist; dropping that behavior would reintroduce the migrated-root install failure already fixed on main.
  - confidence: 0.96
- **[P1] Honor custom systemd units during uninstall:** `src/daemon/systemd.ts:794-796`
  - body: `uninstallSystemdService` disables `resolveGatewaySystemdServiceName(...)`, ignoring `OPENCLAW_SYSTEMD_UNIT`, even though it unlinks the resolved override path below. That means node/custom units can leave the actual service enabled or try to disable the wrong gateway unit; current main uses `resolveSystemdServiceName(env)` here.
  - confidence: 0.95
- **[P1] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:270-273`
  - body: When a mixed `Environment=` line has only some assignments stripped, the kept assignments are re-rendered through `renderEnvLines`, which drops empty string values and trims whitespace. systemd treats `Environment=FOO=` as a valid assignment/reset input, so migration can silently change operator-authored environment semantics.
  - confidence: 0.91
- **[P1] Avoid direct writes for token-bearing drop-ins:** `src/daemon/systemd.ts:627-638`
  - body: This migration writes backups, rewritten main units, and the managed drop-in directly under HOME-derived systemd paths. Because the drop-in can contain managed tokens, this needs a no-symlink/atomic-write strategy or an explicit owner-approved threat model before the branch changes where secrets are written.
  - confidence: 0.88
- **[P2] Remove the release-owned changelog edit:** `CHANGELOG.md:17`
  - body: Normal PRs should not edit `CHANGELOG.md`, and this branch also deletes unrelated existing entries while adding its own. Put release-note context in the PR body or squash message and leave changelog generation to the release flow.
  - confidence: 0.92
