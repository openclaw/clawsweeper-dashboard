Full review comments:

- **[P1] Preserve the activation retry from current main:** `src/daemon/systemd.ts:735-741`
  - body: Current main retries the failed `enable` or `restart` once after a second `daemon-reload` when systemd reports the written unit is not visible yet. This head goes straight from one reload to one `enable` and one `restart`, so refreshed installs can regress the missing-unit recovery that fixed https://github.com/openclaw/openclaw/issues/65184.
  - confidence: 0.92
- **[P1] Honor custom unit names during uninstall:** `src/daemon/systemd.ts:794-796`
  - body: `uninstallSystemdService` still disables `${resolveGatewaySystemdServiceName(...)}.service`, while the unit file and node-service wrapper use `OPENCLAW_SYSTEMD_UNIT`. On node-only or custom-unit installs this disables/removes the wrong target and regresses the current-main fix for https://github.com/openclaw/openclaw/issues/68287.
  - confidence: 0.95
- **[P1] Carry the environment-source contract forward:** `src/daemon/systemd.ts:527-533`
  - body: The refreshed writer needs to keep current main's `environmentValueSources` contract for inline, file, and inline-and-file values. This implementation only receives `environment`, so it cannot preserve the current file-backed secret handling, stale env-file cleanup, and backup sanitization behavior that now protects operator service env state.
  - confidence: 0.84
- **[P1] Reject option-like systemd unit names:** `src/daemon/systemd.ts:55-63`
  - body: The validator still accepts names beginning with `-`, and the later `systemctl` calls pass the unit operand without a `--` terminator. A value like `--user.service` can be parsed as an option by `systemctl`, so custom unit names need a leading-dash rejection and/or consistent operand termination.
  - confidence: 0.89
- **[P1] Harden managed unit and drop-in writes:** `src/daemon/systemd.ts:624-638`
  - body: The migration writes predictable HOME-derived unit, backup, and drop-in paths with direct `copyFile`/`writeFile`. Because this PR adds a new file that can contain service tokens, the writer should either use no-follow/atomic write handling or carry an explicit maintainer threat-model decision before landing.
  - confidence: 0.82
- **[P2] Preserve kept Environment assignment values:** `src/daemon/systemd-unit.ts:269-272`
  - body: When stripping only some assignments from a mixed `Environment=` line, the kept entries are re-rendered through `renderEnvLines`, which filters empty values and trims whitespace. That changes valid systemd unit semantics such as `Environment=LD_PRELOAD=` after removing a managed key.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The PR changes service secret storage and command execution surfaces, and the current head still has concrete unit-name, file-write, and environment-rewrite security concerns.

- **[high] Option-like systemd unit names remain accepted:** `src/daemon/systemd.ts:55`
  - body: `OPENCLAW_SYSTEMD_UNIT` validation permits leading `-`, and unit names are later passed to `systemctl` without a `--` operand terminator, allowing option-like values to alter the command interpretation.

  - confidence: 0.82
- **[medium] Partial Environment rewrites can remove security-clearing values:** `src/daemon/systemd-unit.ts:271`
