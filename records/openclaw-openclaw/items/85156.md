## Summary

Keep open. Current source still has a plausible cron-only gap: cron validates and downgrades thinking through the generic catalog path, while Google’s `low` support lives in provider hooks that may not be available to the startup-scoped cron validation path when cached model metadata says `reasoning: false`.

## What This Changes

The issue reports that cron sessions still downgrade `thinkingDefault: "low"` to `off` for `google/gemini-3-flash-preview` even though the interactive path was fixed.

## Best Possible Solution

Make provider-owned thinking profiles authoritative for Google cron validation even when cached catalog metadata is stale, then add a regression that preserves `low` for `google/gemini-3-flash-preview` without emitting the downgrade warning.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a cron run with `thinkingDefault: "low"` and catalog metadata marking the Google model `reasoning: false` reaches the generic off-only profile and downgrade warning path. I did not execute a real cron run in this read-only review.

## Solution Assessment

No, the current behavior is not the best solution because provider-owned Google thinking support can be bypassed by stale generic catalog metadata. The maintainable fix is to route this validation through the Google provider profile or an equivalent lightweight provider-policy surface, not to add another cron-only special case.
