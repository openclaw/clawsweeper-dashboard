## Summary

Keep open: the underlying `/hooks/agent` thread routing request is valid, but the PR is not correct as submitted because it routes `threadId` through the cron payload instead of the current `delivery.threadId` contract, misses replay scoping, commits a backup source file, and lacks after-fix real behavior proof.

## What This Changes

The branch adds `threadId` handling to `/hooks/agent` and cron delivery plumbing, includes an unrelated cron model-selection fix, and adds `src/media/web-media.ts.backup`.

## Best Possible Solution

Rework this as a narrow hook-threading change that parses `threadId`, sets `job.delivery.threadId`, includes it in replay scope, adds gateway/cron docs and regression coverage, removes the backup file, and splits model-selection behavior out.

## Reproduction Assessment

Yes. Source inspection of current main shows `/hooks/agent` normalization omits `threadId` and dispatch builds delivery with only `mode`, `channel`, and `to`, while cron delivery already reads `delivery.threadId`.

## Solution Assessment

No. The PR targets the right endpoint, but the maintainable fix is to feed the existing `delivery.threadId` contract and update replay scope/docs/tests rather than adding payload-level cron fields and unrelated files.
