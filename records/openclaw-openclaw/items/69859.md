## Summary

Keep open: current main still drops `threadId` from direct `/hooks/agent` requests, and this PR targets the right feature, but the patch is not mergeable as submitted because it feeds the value through the wrong cron path, misses hook replay scoping, includes an accidental backup file, and lacks real behavior proof.

## What This Changes

The branch adds optional `threadId` fields to `/hooks/agent` payload normalization and cron delivery plumbing, bundles an unrelated cron model-selection change, and adds `src/media/web-media.ts.backup`.

## Best Possible Solution

Rework this as a narrow hook-threading change that parses `threadId`, sets `job.delivery.threadId`, scopes hook replay by `threadId`, adds gateway/cron regression coverage and public docs, removes the backup file, and leaves model-selection behavior for a separate PR.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: on current main, `POST /hooks/agent` with `threadId` is normalized without that field, and dispatch builds `job.delivery` without `threadId` even though downstream cron delivery already honors `delivery.threadId`.

## Solution Assessment

No. The PR targets the right user problem, but the maintainable fix is to feed the existing `job.delivery.threadId` contract and replay/docs/tests, not add stale cron payload fields or unrelated files.
