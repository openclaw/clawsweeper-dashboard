## Summary

Keep open. The requested `/hooks/agent` threaded-delivery feature is still missing from current main, and this PR is the canonical implementation thread for the related request, but the submitted branch is not mergeable without focused rework because it has a no-op propagation gap on the branch, omits replay scoping, bundles an unrelated model-selection fix, and adds an accidental backup source file.

## What This Changes

This PR adds optional `threadId` propagation for `POST /hooks/agent` into cron delivery, while also changing cron model selection and adding a backup copy of `web-media`.

## Best Possible Solution

Rework this PR or replace it with a narrow branch that uses current main's delivery contract: normalize optional `threadId` on the hook payload, include it in `/hooks/agent` idempotency replay scope, set `job.delivery.threadId` during hook dispatch, add focused gateway/cron regression coverage, update public hook docs, remove the backup artifact, and split the model-selection change.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: on current main, `normalizeAgentPayload({ message: "x", threadId: "123" })` cannot return `threadId`, and `dispatchAgentHook` cannot place it into `job.delivery`, while downstream cron delivery already honors `delivery.threadId`.

## Solution Assessment

No. The PR is aimed at the right missing feature, but the submitted implementation is not the best path because current main already has a lower-level `delivery.threadId` seam; the hook should feed that seam and update replay/docs/tests instead of adding unrelated payload/model-selection changes.
