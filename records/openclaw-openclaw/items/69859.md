## Summary

Keep open. The requested feature is valid and still missing at the direct `/hooks/agent` boundary on current `main`, but this PR is not mergeable as-is because it sends `threadId` through the wrong/stale cron path, omits hook replay scoping, adds an unrelated model-selection change, and commits a backup artifact.

## What This Changes

The PR adds optional `threadId` fields from `POST /hooks/agent` toward cron delivery, while also changing cron model-selection behavior and adding `src/media/web-media.ts.backup`.

## Best Possible Solution

Rework this into a narrow hook-threading change that feeds current `job.delivery.threadId`, scopes hook idempotency replay by `threadId`, adds gateway and cron regression coverage, updates the public hook docs and changelog, removes the backup file, and leaves the model-selection fix for a separate PR.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: send `POST /hooks/agent` with `threadId` on current `main`; `normalizeAgentPayload` cannot return it and `dispatchAgentHook` cannot place it into `job.delivery`, while downstream cron already honors `delivery.threadId`.

## Solution Assessment

No. The PR targets the right user problem, but the implementation is not the best path because current `main` already has a lower-level `delivery.threadId` contract; the hook endpoint should feed that contract and replay/docs/tests instead of adding stale cron payload fields and unrelated code.
