## Summary

Keep open with a blocking finding: current main and the linked MiniMax upstream issue agree these fields are remaining quota, so the PR reverses the dependency contract and would misreport MiniMax usage. It also lacks real behavior proof beyond a unit-test command.

## What This Changes

The PR reclassifies MiniMax `current_interval_usage_count` and weekly count aliases from remaining quota to consumed usage, then updates tests to expect higher used percentages.

## Best Possible Solution

Keep the current remaining-count parsing unless MiniMax provides new live/API evidence that the contract changed, then update code, docs, and tests together.

## Reproduction Assessment

Yes. From source, a payload with `current_interval_total_count: 100` and `current_interval_usage_count: 98` is expected to report 2% used on current main, while the PR would report 98% used; I did not run tests because this review was read-only.

## Solution Assessment

No. The PR reverses a documented and dependency-backed workaround; the safer path is to keep the remaining-count interpretation unless live upstream evidence proves MiniMax changed the field contract.
