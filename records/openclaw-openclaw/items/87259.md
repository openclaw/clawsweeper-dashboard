## Summary

Keep open, but the PR should not merge as-is: it reverses the shipped MiniMax remaining-quota interpretation without real MiniMax API/status/dashboard proof, while the open canonical issue still owns the disputed endpoint semantics.

## What This Changes

The PR moves MiniMax current-interval/current-weekly usage-count aliases from remaining-count parsing to used-count parsing and updates MiniMax usage test expectations.

## Best Possible Solution

Keep the aliases under remaining-count handling unless redacted live MiniMax evidence proves a specific endpoint returns consumed counts, then add endpoint-specific parser tests through the canonical MiniMax issue.

## Reproduction Assessment

Yes. From source, a payload with current_interval_total_count 100 and current_interval_usage_count 98 reports 2% used on current main, while this PR would report 98%; I did not run tests because this review is read-only.

## Solution Assessment

No. A global alias move is not the best fix while current source, release tests, upstream discussion, and linked live reports support remaining-quota semantics; any change should be endpoint-specific and backed by live evidence.
