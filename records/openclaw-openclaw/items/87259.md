## Summary

Keep this PR open for maintainer handling, but it is not merge-ready: the branch reverses MiniMax count-field semantics that current source, tests, shipped release code, the linked upstream MiniMax issue, and the canonical OpenClaw issue all currently treat as remaining quota.

## What This Changes

The PR moves four MiniMax `current_*_usage_count` aliases from remaining-quota parsing to consumed-usage parsing and updates the MiniMax usage tests to expect the reversed arithmetic.

## Best Possible Solution

Preserve the shipped remaining-count handling unless redacted live MiniMax evidence proves a specific endpoint now returns consumed counts; if that evidence appears, add endpoint-specific parsing and tests rather than globally moving the aliases.

## Reproduction Assessment

Yes for the PR-introduced regression from source: with `current_interval_total_count: 100` and `current_interval_usage_count: 98`, current main reports 2% used while the PR would report 98% used. I did not run live MiniMax verification in this read-only review.

## Solution Assessment

No. The PR is not the best fix because it globally reverses shipped remaining-count handling; the safer path is live endpoint proof first, then endpoint-specific parsing only if the contract really differs.
