## Summary

Keep open: the rich-message direction is useful and matches the linked Telegram table request, but the current patch still misses Telegram's 500-block limit for table rows and list items, so over-limit rich replies can be rejected instead of chunked or safely delivered.

## What This Changes

This PR rewires Telegram text send, edit, preview, outbound chunking, docs, config schema, and doctor migration paths to use Bot API rich messages while retiring the native draft preview config.

## Best Possible Solution

Keep the rich-message rollout, but count Telegram's full rich block model before send/edit, add regression coverage for over-500-row tables and over-500-item lists, and make the native-draft config retirement an explicit accepted upgrade behavior.

## Reproduction Assessment

Yes at source level: a rich markdown table or list with more than 500 rows/items passes through as one rich message even though Telegram counts those as rich blocks. I did not run a live Telegram rejection test in this read-only review.

## Solution Assessment

No, not yet. The rich-message direction is appropriate, but the implementation must count Telegram's full block model and preserve upgrade-safe behavior before it is the best fix.
