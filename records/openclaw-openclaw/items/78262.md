## Summary

Keep open: current main has Feishu topic routing coverage from #29788, but the reported starter/reply mismatch is still plausible from the source when the first native topic event lacks the later `thread_id` or carries the starter `om_*` id while replies carry `omt_*`. The issue is narrow enough for a Feishu-local repair, but I did not have live Feishu payload proof in this read-only review.

## What This Changes

The issue reports that Feishu `groupSessionScope: "group_topic"` splits one topic into separate sessions because the starter routes by `messageId` while replies route by `thread_id`/topic id.

## Best Possible Solution

Keep the fix in the Feishu plugin: normalize native topic starter and reply events to one canonical conversation id, add a regression test for the reported `om_*` starter to `omt_*` reply shape, and keep docs/schema aligned with the `thread_id` contract.

## Reproduction Assessment

No high-confidence live reproduction is available from this review. Source inspection gives a clear synthetic reproduction for the reported field shape: a native topic starter resolved from `messageId`/`root_id` can differ from a later reply resolved from `thread_id`.

## Solution Assessment

Yes, the best direction is a Feishu-local routing fix plus a focused regression test. Core session routing already accepts the plugin-provided peer id, so changing core would be broader than needed.

The reported bug is that with `groupSessionScope: "group_topic"`, the first native topic message can route by `messageId`/`om_*` while later replies route by `thread_id`/`omt_*`, splitting one Feishu topic into two OpenClaw sessions. Do not touch unrelated Feishu channel behavior or core session routing.
