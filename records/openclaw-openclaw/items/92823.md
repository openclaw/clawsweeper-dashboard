## Summary

This PR remains useful and reviewable: current main still converts QQBot outbound errors or missing platform IDs into successful empty receipts, while the branch turns those cases into send failures with focused tests and sufficient bounded runtime proof. It should stay open for maintainer acceptance of the shared QQBot text/media fail-fast behavior, not close as stale or superseded.

## What This Changes

The branch changes QQBot channel send result normalization and adapter tests so outbound errors or empty platform message receipts reject instead of returning an empty successful receipt.

## Best Possible Solution

Land the bounded adapter error-propagation fix after maintainer acceptance of the shared text/media fail-fast semantics, and keep the broader QQ C2C delivery route work in the canonical issue.

## Reproduction Assessment

Yes at source level: current main stores QQBot lower-layer errors in meta and can return an empty successful receipt when no platform id exists. I did not run a credentialed live QQBot send, but the PR body includes before/after adapter-path proof for the bounded failure-propagation behavior.

## Solution Assessment

Yes for the bounded error-propagation problem: the adapter normalizer is the narrowest maintainable layer to reject lower-level send errors and empty receipts. It is not the full solution for QQ C2C media delivery, which remains tracked separately.
