## Summary

Keep open. Current main still measures code-mode Markdown table widths and padding with JavaScript `.length`, so CJK cells remain under-padded, and this issue is paired with the open draft fix PR at https://github.com/openclaw/openclaw/pull/55596.

## What This Changes

Fix Markdown table fallback rendering so Telegram and Discord align CJK table cells by display width instead of JavaScript string length.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/55596 or a narrow replacement that uses display-width and grapheme-aware measurement in `renderTableAsCode`, with focused CJK and emoji mixed-width regression coverage.

## Reproduction Assessment

Yes by source inspection. Passing a CJK table through `markdownToIR(..., { tableMode: "code" })` still reaches `renderTableAsCode`, where `类型` is counted as length 2 even though it occupies four monospace columns.

## Solution Assessment

Yes, the narrow maintainable fix belongs in the shared `renderTableAsCode` path used by both Telegram and Discord fallback rendering. Channel-specific patches would duplicate renderer logic; the open draft PR has the right boundary but is not ready to close this issue until it lands or is replaced.
