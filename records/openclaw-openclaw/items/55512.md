## Summary

Keep open. Current main still computes code-mode Markdown table column widths and padding with JavaScript string length, so CJK cells remain under-padded; the Discord raw-table part is already routed through the shared converter, and an open draft PR with closing syntax is already paired with this issue.

## What This Changes

The issue asks for CJK-aware Markdown table alignment in Telegram code/pre output and Discord table fallback output.

## Best Possible Solution

Land PR #55596 or an equivalent replacement that fixes display-width calculation in the shared Markdown code-table renderer, preferably reusing the existing `visibleWidth`/grapheme-aware terminal helper or an approved equivalent, and adds CJK plus emoji/grapheme regression tests for the rendered table text.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence reproduction path: render a table with a CJK cell such as `类型` in `tableMode: "code"`, and the renderer will size and pad it as length 2 even though it occupies four monospace columns.

## Solution Assessment

Yes. A shared renderer fix is the narrowest maintainable solution because both Telegram and Discord now flow through the same code-mode Markdown table conversion path; the open paired PR should be reviewed, landed, or replaced rather than dispatching a competing fix.
