## Summary

Keep open. Current main and the latest release still let an explicit agent header select the agent, then reject a generic request-body model before dispatch; the related open PR at https://github.com/openclaw/openclaw/pull/45344 targets unknown agent IDs, not this valid-header precedence bug.

## What This Changes

The issue asks for OpenAI-compatible Chat Completions requests with a valid `x-openclaw-agent-id` header to ignore the request body `model` and use the selected agent's configured model.

## Best Possible Solution

Make a valid explicit agent header authoritative for agent selection, treat the body `model` as inert OpenAI SDK compatibility input in that path, and keep `x-openclaw-model` as the only backend model override.

## Reproduction Assessment

Yes, source-reproducible on current main. A valid `x-openclaw-agent-id` selects the agent first, but a generic body `model` such as `gpt-4` is still passed into the override helper and rejected before dispatch; I did not run a live gateway request in this read-only review.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix if the header is validated first. The fix should not broaden the body `model` into a raw provider-model override, and invalid or unconfigured explicit agent headers must still be rejected by the relevant validation path.
