## Summary

Keep open. Current main and the latest release still reject a generic body `model` after `x-openclaw-agent-id` selects an agent, while the public docs currently define the body `model` as an OpenClaw agent target, so this remains a maintainer product-contract decision rather than a cleanup close.

## What This Changes

Request that `/v1/chat/completions` ignore a generic request-body `model` when `x-openclaw-agent-id` selects a valid agent, using that agent's configured model instead.

## Best Possible Solution

Define one shared precedence rule in `src/gateway/http-utils.ts`: a valid agent header selects the agent and makes body `model` compatibility metadata, while body-model agent aliases remain authoritative when no header is present; cover Chat Completions, Responses, and docs together.

## Reproduction Assessment

Yes, source-level. Current main selects the header agent and then rejects a generic body model in `resolveOpenAiCompatModelOverride`; I did not run a live gateway in this read-only review.

## Solution Assessment

Unclear until maintainers decide the public precedence contract. If accepted, the maintainable fix is shared helper behavior plus Chat/Responses regression coverage and docs, not a handler-local exception.
