## Summary

Current main still rejects generic OpenAI model identifiers even when an explicit agent header has selected the agent, so this issue remains necessary. The requested change is small but compatibility-sensitive: maintainers need to decide whether the body `model` is a routing assertion or ignorable SDK metadata when an agent header is present, and that rule must apply consistently to both Chat Completions and Responses.

## What This Changes

Treat the OpenAI-compatible request body model as non-authoritative when a valid `x-openclaw-agent-id` header explicitly selects an agent, using that agent’s configured model instead.

## Best Possible Solution

Define one public precedence rule in the shared gateway layer: an explicit valid agent header selects the agent and makes the body `model` compatibility metadata, while body-model agent aliases remain authoritative only when no agent header is present; cover both generation endpoints and document the rule.

## Reproduction Assessment

Yes. Current main deterministically selects the header agent and then rejects a generic body model in `resolveOpenAiCompatModelOverride`, matching the reported request path without needing speculative state.

## Solution Assessment

Yes in principle, but the best fix is a shared and documented precedence rule for both Chat Completions and Responses rather than a handler-local exception.
