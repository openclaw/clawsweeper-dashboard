## Summary

Keep open. Current main and the latest release still reject generic request-body `model` values after a valid `x-openclaw-agent-id` selects an agent, and the public docs still define `model` as an OpenClaw agent target, so this remains a real OpenAI-compatibility/product-contract decision rather than cleanup-close material.

## What This Changes

The issue requests that `/v1/chat/completions` ignore the request body `model` when a valid `x-openclaw-agent-id` header selects the target agent.

## Best Possible Solution

Maintainers should decide the shared OpenAI-compatible precedence rule, then update the shared helper, Chat/Responses regression coverage, and docs together if header-selected agents should make the body `model` compatibility metadata.

## Reproduction Assessment

Yes, source-level: a request with a valid `x-openclaw-agent-id` and generic body `model` selects the header agent, then the shared model-override helper rejects the original body model before dispatch. I did not run a live gateway in this read-only review.

## Solution Assessment

Unclear until maintainers decide the public precedence contract. If accepted, the maintainable fix is shared helper behavior plus Chat/Responses tests and docs, not a handler-local exception.
