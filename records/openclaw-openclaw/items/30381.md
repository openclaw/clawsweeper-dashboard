## Summary

Keep open. Current main still rejects a generic OpenAI request-body model before dispatch even when an agent header has selected the agent, and the recent discussion confirms this remains an active OpenAI-SDK compatibility problem. The remaining fix is narrow and belongs in the Gateway HTTP compatibility path.

## What This Changes

The issue requests that `/v1/chat/completions` ignore the request body `model` when `x-openclaw-agent-id` selects an agent, using the selected agent's configured model unless `x-openclaw-model` explicitly overrides it.

## Best Possible Solution

Keep the agent-first contract, but make the agent-header path compatible with standard OpenAI SDKs: after a valid agent header selects an agent, treat the body `model` as client compatibility noise rather than a backend override, keep `x-openclaw-model` as the only explicit backend model override, and document/test the matching `/v1/responses` choice.

## Reproduction Assessment

Yes. A static current-main reproduction is clear: send `/v1/chat/completions` with `x-openclaw-agent-id: beta` and body `model: "gpt-4"`; the header selects `beta`, then `resolveOpenAiCompatModelOverride` returns the invalid-model error before dispatch.

## Solution Assessment

Yes. The narrowest maintainable solution is a header-aware adjustment in the shared Gateway model-override/request-context path, with regression tests and docs, rather than loosening all request-body model validation or treating generic body models as backend overrides.
