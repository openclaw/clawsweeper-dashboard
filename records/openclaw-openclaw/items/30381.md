## Summary

Keep open. Current main still selects the header agent first and then rejects a generic request-body `model` through the shared OpenAI compatibility validator, so the reported OpenAI SDK compatibility bug remains valid and narrow.

## What This Changes

Allow `/v1/chat/completions` requests with a valid `x-openclaw-agent-id` header to use the selected agent configuration while ignoring generic body `model` values.

## Best Possible Solution

Preserve agent-first routing, reserve `x-openclaw-model` for backend overrides, and add regression coverage around chat completions plus the shared Responses path.

## Reproduction Assessment

Yes, source-reproducible on current main. Send `/v1/chat/completions` with a valid `x-openclaw-agent-id` and a generic body `model` such as `gpt-4`; the code selects the header agent and then returns the invalid-model error before dispatch.

## Solution Assessment

Yes. The narrowest fix is to make the OpenAI compatibility model-override path aware that a valid agent header already selected the agent, while preserving body-model rejection when no valid header selected the agent.

## Repair Work Prompt

Fix #30381 in the OpenAI-compatible Gateway HTTP paths. When a valid `x-openclaw-agent-id` or `x-openclaw-agent` header selects an agent for `/v1/chat/completions`, ignore generic OpenAI request-body `model` values such as `gpt-4` instead of returning the invalid-model error; keep `x-openclaw-model` as the only backend provider/model override. Preserve invalid body-model rejection when no valid agent header selected the agent, and coordinate semantics with open PR #45344 so unknown headers are still rejected rather than falling back. Add focused regression coverage for chat completions and the shared `/v1/responses` helper behavior if the helper changes. Add a user-facing `CHANGELOG.md` fix entry; update docs only if the final behavior needs clarification. Do not reinterpret the OpenAI body `model` as a provider model.
