## Summary

Keep open. Current main still routes by `x-openclaw-agent-id` first, but then the shared OpenAI-compatible model override helper rejects a generic request body `model` before chat dispatch; the related open PR addresses unknown agent IDs and does not supersede this compatibility bug.

## What This Changes

Request that `/v1/chat/completions` ignore a generic request body `model` when a valid `x-openclaw-agent-id` header selects the agent.

## Best Possible Solution

Preserve agent-header routing precedence, skip body-model agent-selector validation only after a valid explicit agent header selected the agent, keep `x-openclaw-model` as the backend override path, and add focused regression coverage.

## Reproduction Assessment

Yes, from source inspection. Send `/v1/chat/completions` with a valid `x-openclaw-agent-id` and a generic body `model` such as `gpt-4`; current main selects the header agent, then returns the invalid-model 400 before dispatch.

## Solution Assessment

Yes. The narrow maintainable fix is to make the OpenAI compatibility model-validation path aware of a valid explicit agent-header selection while preserving existing rejection when no valid header selected the agent.
