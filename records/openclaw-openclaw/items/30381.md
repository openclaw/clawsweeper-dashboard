## Summary

Keep open. Current main and the latest release still reject a generic request body `model` after a valid `x-openclaw-agent-id`, but the public docs intentionally define the OpenAI-compatible `model` field as an OpenClaw agent target, so the remaining work is a maintainer API-contract decision rather than cleanup-close material.

## What This Changes

The issue asks `/v1/chat/completions` to ignore a generic request body `model` when a valid `x-openclaw-agent-id` header selects the target agent.

## Best Possible Solution

Have maintainers define one shared OpenAI-compatible precedence rule; if header-selected agents should make body `model` compatibility metadata, update the shared helper, chat/responses/embeddings coverage, and public docs together.

## Reproduction Assessment

Yes, source-level. A valid `x-openclaw-agent-id` can select the agent, then `resolveOpenAiCompatModelOverride` still rejects a generic body `model` before dispatch; no live gateway was run because this review is read-only.

## Solution Assessment

Unclear. The requested behavior is plausible for OpenAI SDK compatibility, but it changes the documented agent-first model contract and should be decided across all OpenAI-compatible endpoints.
