## Summary

Keep open. Current main and the latest release still reject a generic request-body `model` after a valid `x-openclaw-agent-id` selects an agent, but current docs intentionally define `model` as an OpenClaw agent target, so the remaining decision is API/product contract rather than cleanup-close material.

## What This Changes

The issue requests changing OpenAI-compatible chat completions routing so a valid `x-openclaw-agent-id` header makes the gateway ignore the request-body `model` and use the configured agent model.

## Best Possible Solution

Define one shared precedence rule for OpenAI-compatible endpoints, then update the shared helper, Chat/Responses/Embeddings coverage, and public docs together if maintainers approve header-selected agents making body `model` compatibility metadata.

## Reproduction Assessment

Yes. Source inspection shows a valid `x-openclaw-agent-id` can select the agent, then `resolveOpenAiCompatModelOverride` still rejects a generic body `model` before dispatch; I did not run a live gateway because this review is read-only.

## Solution Assessment

Unclear. The requested behavior is plausible for OpenAI SDK compatibility, but current docs intentionally define `model` as an OpenClaw agent selector and use `x-openclaw-model` for backend overrides.
