## Summary

Keep open. Current main still selects the explicit header agent first, then rejects a generic request-body `model` through the shared OpenAI compatibility validator before dispatch; the related open PR handles unknown agent IDs and the earlier closed test-only PR was not merged.

## What This Changes

The issue asks the OpenAI-compatible chat completions path to ignore a generic request-body model when a valid `x-openclaw-agent-id` header selects an agent.

## Best Possible Solution

Preserve agent-first routing, skip body-model validation only after a valid explicit agent header selects a known agent, keep `x-openclaw-model` as the backend override path, and add focused chat plus Responses regression coverage.

## Reproduction Assessment

Yes, from source inspection. Send `/v1/chat/completions` with a valid `x-openclaw-agent-id` and a generic body `model` such as `gpt-4`; the header selects the agent, then `resolveOpenAiCompatModelOverride` returns a 400 invalid-model error before dispatch.

## Solution Assessment

Yes, if implemented narrowly. The maintainable fix is to make the OpenAI compatibility model-validation path aware of a valid explicit agent-header selection while preserving rejection for invalid models when no valid header selected the agent.
