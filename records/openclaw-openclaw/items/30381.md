## Summary

Keep open. Current main still selects the header agent first, then rejects a generic request-body `model` through the shared OpenAI compatibility validator before dispatching the chat completion; the related open PR covers invalid agent IDs and does not supersede this issue.

## What This Changes

The issue asks the OpenAI-compatible chat completions endpoint to ignore the request body `model` when a valid `x-openclaw-agent-id` header selects an agent.

## Best Possible Solution

Preserve agent-first routing, skip body-model agent-selector validation only after a valid explicit agent header has selected the agent, keep `x-openclaw-model` as the backend override path, and add focused regression coverage across chat completions plus the shared Responses/helper path.

## Reproduction Assessment

Yes from source inspection. Send `/v1/chat/completions` with a valid `x-openclaw-agent-id` and a generic body `model` such as `gpt-4`; the header selects the agent, then `resolveOpenAiCompatModelOverride` returns a 400 invalid-model error before dispatch.

## Solution Assessment

Yes. The narrow maintainable fix is to make the OpenAI compatibility model-validation path aware of a valid explicit agent-header selection while preserving existing rejection when no valid header selected the agent.
