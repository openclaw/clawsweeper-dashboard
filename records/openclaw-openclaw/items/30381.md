## Summary

Keep open. Current main still lets the explicit agent header pick the agent, then rejects a generic request-body `model` through the shared OpenAI compatibility model validator before dispatch; recent comments also confirm this is still an active production compatibility problem, and the related open PR addresses unknown agent IDs rather than this valid-header precedence bug.

## What This Changes

Request that `/v1/chat/completions` use the configured agent model and ignore the body `model` when `x-openclaw-agent-id` selects an agent.

## Best Possible Solution

Make an explicit accepted agent header authoritative for agent selection, treat the request body `model` as SDK compatibility input on that path, keep `x-openclaw-model` as the backend model override, and add focused Chat Completions plus Responses regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows a request with `x-openclaw-agent-id: beta` and `model: gpt-4` selects `beta`, then returns the shared invalid-model error before dispatch because the body model is still validated as an agent selector.

## Solution Assessment

Yes. The narrow maintainable fix is to skip body-model agent-selector validation only when the explicit agent header path has already selected the agent, while preserving `x-openclaw-model`, default-agent aliases, and invalid-selector behavior.
