## Summary

Keep open. Current main and `v2026.6.6` still reject a generic request-body `model` after a valid `x-openclaw-agent-id` selects an agent, and the docs currently define `model` as an OpenClaw agent target, so this remains a maintainer product-contract decision rather than cleanup-close material.

## What This Changes

The issue asks Chat Completions to ignore the request-body `model` and use the header-selected agent's configured model when `x-openclaw-agent-id` is present.

## Best Possible Solution

Define one shared precedence rule for OpenAI-compatible endpoints, then update the shared helper, Chat/Responses coverage, and public docs together if maintainers approve header-selected agents making body `model` compatibility metadata.

## Reproduction Assessment

Yes, source-level: a request with a valid `x-openclaw-agent-id` and generic body `model` selects the header agent, then `resolveOpenAiCompatModelOverride` rejects the original body model before dispatch. I did not run a live gateway in this read-only review.

## Solution Assessment

Unclear until maintainers decide the public precedence contract. If accepted, the maintainable fix is shared helper behavior plus Chat/Responses regression coverage and docs, not a handler-local exception.
