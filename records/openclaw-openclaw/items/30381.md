## Summary

Keep open. Current main still lets the explicit agent header win routing, then rejects a generic request-body `model` through the shared OpenAI compatibility validator before dispatch; the related open PR covers unknown agent IDs, not this valid-header precedence bug.

## What This Changes

The issue asks the OpenAI-compatible Chat Completions gateway to ignore the request body `model` when a valid `x-openclaw-agent-id` header selects the agent.

## Best Possible Solution

Make a valid explicit agent header authoritative for agent selection, treat the body `model` as inert SDK compatibility input in that path, and keep `x-openclaw-model` as the only backend model override.

## Reproduction Assessment

Yes, source-reproducible on current main. A valid `x-openclaw-agent-id` selects the agent first, but a generic body `model` such as `gpt-4` is still passed into the override helper and rejected before dispatch; I did not run a live gateway request in this read-only review.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix if validity is checked first. A broader raw-provider-model compatibility path would conflict with the documented agent-first contract.
