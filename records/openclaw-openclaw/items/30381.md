## Summary

Keep open. Current main still selects the explicit agent header first, then rejects a generic request-body `model` through the shared OpenAI compatibility validator before dispatch; the related open PR at https://github.com/openclaw/openclaw/pull/45344 handles unknown agent IDs, not this valid-header precedence bug.

## What This Changes

The issue requests that OpenAI-compatible chat completions ignore a generic request body `model` after a valid `x-openclaw-agent-id` header selects an agent.

## Best Possible Solution

Preserve agent-first routing, skip body-model validation only after a valid explicit agent header selects a known agent, keep `x-openclaw-model` as the backend override path, and add focused Chat Completions plus Responses regression coverage.

## Reproduction Assessment

Yes, source-reproducible. A `/v1/chat/completions` request with a valid `x-openclaw-agent-id` and a generic body `model` such as `gpt-4` will select the header agent, then `resolveOpenAiCompatModelOverride` returns the invalid-model error before dispatch; I did not run a live gateway command in this read-only review.

## Solution Assessment

Yes, if implemented narrowly. The maintainable fix is to make the model-validation helper aware of a valid explicit agent-header selection while preserving current rejection for invalid body models when no valid header selected the agent.
