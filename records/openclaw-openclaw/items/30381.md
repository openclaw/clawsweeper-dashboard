## Summary

Keep open. Current main and v2026.6.10 still reject a generic request body model after a valid x-openclaw-agent-id, but the public docs intentionally define the OpenAI-compatible model field as an OpenClaw agent target, so this needs maintainer API-contract judgment rather than cleanup closure.

## What This Changes

The issue requests that /v1/chat/completions ignore the request body model when x-openclaw-agent-id selects a valid agent, so OpenAI SDK clients can send generic model values while using the agent's configured model.

## Best Possible Solution

Maintainers should define one shared OpenAI-compatible precedence rule; if header-selected agents should make the body model compatibility metadata, update the shared helper, chat/responses/embeddings coverage, and public docs together.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a valid x-openclaw-agent-id can select the agent, then resolveOpenAiCompatModelOverride still rejects a generic body model before dispatch.

## Solution Assessment

Unclear. The requested behavior is plausible for OpenAI SDK compatibility, but it changes the documented agent-first model contract and should be decided across all OpenAI-compatible endpoints.
