## Summary

Keep open. Current main still rejects a generic request-body `model` after `x-openclaw-agent-id` has selected the agent, so the reported OpenAI SDK compatibility bug remains valid and narrow.

## What This Changes

The issue asks the Gateway OpenAI-compatible chat completions endpoint to ignore the request body `model` when a valid `x-openclaw-agent-id` header selects the agent.

## Best Possible Solution

Preserve agent-first routing, ignore a generic body `model` only after a valid agent header selects the agent, keep `x-openclaw-model` as the explicit backend override path, and add regression coverage for chat completions plus the shared Responses behavior.

## Reproduction Assessment

Yes, by source inspection. Send `/v1/chat/completions` with a valid `x-openclaw-agent-id` and a generic body `model` such as `gpt-4`; current main selects the header agent and then rejects the body model before dispatch.

## Solution Assessment

Yes. The narrowest maintainable fix is a header-aware adjustment in the shared Gateway OpenAI compatibility model-override path, with tests that keep body-model rejection for requests without a valid agent header.

## Repair Work Prompt

Fix the OpenAI-compatible Gateway model validation bug tracked by https://github.com/openclaw/openclaw/issues/30381. On current main, `/v1/chat/completions` with a valid `x-openclaw-agent-id` still rejects generic OpenAI SDK body models such as `gpt-4` because `resolveOpenAiCompatModelOverride` validates the body `model` after the header has selected the agent. Keep the fix scoped to Gateway OpenAI compatibility: ignore the body `model` as compatibility noise only after a valid agent header selects the agent; keep `x-openclaw-model` as the only backend provider/model override path; preserve invalid body-model rejection when no valid agent header selected the agent. Check the shared `/v1/responses` path because it uses the same helper, and coordinate with adjacent unknown-agent work in #45344 without broad refactors. Add regression coverage for chat completions, shared helper behavior, and Responses if behavior changes there. Add a user-facing changelog entry and docs clarification if the behavior contract changes in docs.
