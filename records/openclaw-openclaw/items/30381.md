## Summary

Keep open. Current main and v2026.6.10 still reject a generic body `model` after a valid `x-openclaw-agent-id`, while the docs intentionally define `model` as an OpenClaw agent target, so this is a maintainer API-contract decision rather than cleanup-close material.

## What This Changes

Request to make `/v1/chat/completions` ignore a generic request body `model` when a valid `x-openclaw-agent-id` selects the agent.

## Best Possible Solution

Have maintainers define one shared OpenAI-compatible precedence rule; if header-selected agents should make body `model` compatibility metadata, update the shared helper, chat/responses/embeddings coverage, and public docs together.

## Reproduction Assessment

Yes, source-level. A valid `x-openclaw-agent-id` can select the agent, then `resolveOpenAiCompatModelOverride` still rejects a generic body `model` before dispatch; no live gateway was run in this read-only review.

## Solution Assessment

Unclear. The requested behavior is plausible for OpenAI SDK compatibility, but it changes the documented agent-first model contract and should be decided across all OpenAI-compatible endpoints.
