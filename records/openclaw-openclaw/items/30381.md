## Summary

Keep open. Current main still resolves a valid `x-openclaw-agent-id` first, then validates the request-body `model` through the OpenAI-compatible model override path, so a generic SDK model can still be rejected before the selected agent runs.

## What This Changes

The issue asks the gateway Chat Completions path to treat a valid `x-openclaw-agent-id` header as authoritative and ignore a generic OpenAI-compatible request-body `model` for agent selection.

## Best Possible Solution

Make a valid explicit agent header authoritative for agent routing, treat the body `model` as OpenAI SDK compatibility input on that path, preserve `x-openclaw-model` for backend model overrides, and cover both Chat Completions and Responses with focused regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows a valid `x-openclaw-agent-id` can select the agent, but the same request still validates a generic body `model` through `resolveOpenAiCompatModelOverride` and can return HTTP 400 before dispatch.

## Solution Assessment

Yes. The narrow maintainable fix is to skip body-model agent-selector validation only after a valid explicit agent header selects the agent, while preserving default-agent aliases, `x-openclaw-model`, and invalid explicit selector behavior.
