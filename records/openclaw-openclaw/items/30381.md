## Summary

Keep open. Current main and the latest release still choose the explicit agent header first, then validate and reject a generic body `model` before dispatch; the issue discussion also has repeated production-use confirmations, and the related open PR covers unknown-agent rejection rather than this header-precedence compatibility bug.

## What This Changes

Request that Chat Completions ignore a generic request-body `model` when a valid `x-openclaw-agent-id` header selects an agent, using that agent's configured model instead.

## Best Possible Solution

Make a valid explicit agent-header selection authoritative while preserving `x-openclaw-model` backend overrides and unknown-agent validation across Chat Completions and Responses.

## Reproduction Assessment

Yes, source-level. A request with `x-openclaw-agent-id` selecting an agent and body `model: "gpt-4"` selects the header agent first, then `resolveOpenAiCompatModelOverride` rejects `gpt-4` before dispatch.

## Solution Assessment

Yes. The narrow maintainable fix is to skip body-model agent-selector validation only after a valid explicit agent header selects the agent, while preserving default-agent aliases, `x-openclaw-model`, and invalid explicit selector behavior.

## Repair Work Prompt

Fix the OpenAI-compatible gateway behavior reported in https://github.com/openclaw/openclaw/issues/30381. When a valid explicit `x-openclaw-agent-id` or `x-openclaw-agent` header selects an agent, treat the request body `model` as OpenAI SDK compatibility input and do not reject generic provider model IDs from that field; continue using the selected agent's configured model unless `x-openclaw-model` is present. Preserve the documented body-model agent selectors (`openclaw`, `openclaw/default`, `openclaw/<agentId>`, `openclaw:<agentId>`, `agent:<agentId>`), preserve `x-openclaw-model` validation/allowlist behavior, and coordinate with https://github.com/openclaw/openclaw/pull/45344 so explicit unknown agent selectors still fail once that validation lands. Add focused regression coverage for Chat Completions and Responses; do not add a new config option or change provider/model allowlist policy. Add a changelog entry if the repair is landed as a user-visible fix.
