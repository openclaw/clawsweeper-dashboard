## Summary

Keep open. Current main and v2026.6.8 still reject a generic body `model` after a valid agent header, but the docs intentionally define `model` as an agent target, so this is a maintainer API-contract decision rather than cleanup-close material.

## What This Changes

The issue requests that `/v1/chat/completions` ignore the request body `model` when a valid `x-openclaw-agent-id` header targets an agent.

## Best Possible Solution

Define one shared precedence rule for OpenAI-compatible endpoints, then update the shared helper, endpoint coverage, and public docs together if maintainers approve header-selected agents making body `model` compatibility metadata.

## Reproduction Assessment

Yes. Source inspection shows a valid `x-openclaw-agent-id` can select the agent, then `resolveOpenAiCompatModelOverride` still rejects a generic body `model` before dispatch; I did not run a live gateway because this review is read-only.

## Solution Assessment

Unclear. The requested behavior is plausible for OpenAI SDK compatibility, but current docs intentionally define `model` as an OpenClaw agent selector and use `x-openclaw-model` for backend overrides.
