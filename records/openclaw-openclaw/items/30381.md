## Summary

Keep open. Current main still rejects a non-OpenClaw request-body model before dispatch even after `x-openclaw-agent-id` has selected the agent, and the issue discussion confirms this remains an active OpenAI SDK compatibility problem.

## What This Changes

The issue asks the OpenAI-compatible chat completions path to ignore the request-body `model` when a valid `x-openclaw-agent-id` header selects the agent.

## Best Possible Solution

Preserve the agent-first contract, but after a valid agent header selects an agent, treat the body `model` as OpenAI-client compatibility noise unless `x-openclaw-model` explicitly requests a backend override, with matching regression coverage and docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send `/v1/chat/completions` with `x-openclaw-agent-id: beta` and body `model: "gpt-4"`; current main selects `beta` but then `resolveOpenAiCompatModelOverride` returns the invalid-model error before dispatch.

## Solution Assessment

Yes. The narrowest maintainable fix is a header-aware adjustment in the shared Gateway OpenAI compatibility request-context/model-override path, rather than loosening all body-model validation or treating generic body models as backend overrides.
