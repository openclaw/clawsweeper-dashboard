## Summary

Keep open. Current main still validates a generic request-body `model` after the agent header has selected the agent, so the reported OpenAI SDK compatibility failure remains valid and narrow.

## What This Changes

The issue asks the Gateway OpenAI chat-completions endpoint to ignore the request-body `model` when `x-openclaw-agent-id` selects a configured agent.

## Best Possible Solution

Preserve agent-first routing, treat a generic body `model` as compatibility noise only after a valid agent header has selected the agent, keep `x-openclaw-model` as the explicit backend override path, and retain invalid body-model rejection when no valid agent header is present.

## Reproduction Assessment

Yes, by source inspection. Send `/v1/chat/completions` with a valid `x-openclaw-agent-id` and `model: "gpt-4"`; current main selects the header agent and then rejects the body model before dispatch.

## Solution Assessment

Yes. The narrowest maintainable fix is a header-aware adjustment in the shared Gateway OpenAI compatibility model-override path, with regression coverage for chat completions and intentional handling for the Responses path that shares the helper.

Do not loosen backend model allowlists or treat raw OpenAI body model values as backend overrides.
