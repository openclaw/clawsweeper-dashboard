## Summary

Keep this open: current main and the latest release still reject non-OpenClaw body models before dispatch, but the requested relaxation changes the documented agent-first gateway contract and needs maintainer product/API review.

## What This Changes

The issue requests that `/v1/chat/completions` ignore the request-body `model` when `x-openclaw-agent-id` targets an agent, so the agent's configured model is used instead.

## Best Possible Solution

Make an explicit maintainer product/API decision; if accepted, implement a narrow contract change that treats a valid explicit agent header as authoritative, keeps `x-openclaw-model` as the backend override path, and updates sibling OpenResponses behavior, tests, and docs together.

## Reproduction Assessment

Yes. Source inspection shows a generic body model such as `gpt-4` still reaches `resolveOpenAiCompatModelOverride`, which rejects it before the agent run even when `x-openclaw-agent-id` already selected an agent.

## Solution Assessment

Unclear without maintainer direction. Ignoring the body model for valid header-targeted requests would solve the reported SDK compatibility pain, but it changes the documented agent-first contract and should be decided for both OpenAI-compatible HTTP surfaces.
