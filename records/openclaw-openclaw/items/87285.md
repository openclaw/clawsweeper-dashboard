## Label Justifications

- `P1`: The original report described frequent gateway restarts and event-loop stalls interrupting live sessions and health checks.
- `impact:session-state`: The reported restarts interrupted active webchat sessions and surfaced previous-turn interruption messages.
- `impact:auth-provider`: Provider auth pre-warm and SecretRef resolution were central to the reported failure mode.
- `impact:crash-loop`: The report included launchd restart loops and repeated SIGTERM restarts.

## Summary

Current main and the shipped v2026.6.6 release cover the central gateway reliability problem: config reload planning is selective, no-op/self writes are skipped, provider auth pre-warm is delayed and worker-backed, and inactive SecretRefs no longer poison command-time secret resolution.

## What This Changes

The issue requests less aggressive gateway config reloads, non-blocking provider auth pre-warm, and clearer/non-fatal handling for inactive SecretRefs in `secrets.resolve`.

## Best Possible Solution

Keep the shipped v2026.6.6 implementation and use https://github.com/openclaw/openclaw/issues/90865 only for the narrower Discord post-start subscription follow-up if that remains reproducible.

## Reproduction Assessment

No current-main failing repro is available; the original v2026.5.22 failure is supported by concrete logs, and current source/tests show the repaired reload, provider-auth, and SecretRef paths.

## Solution Assessment

Yes. The shipped solution addresses the right owner boundaries: Gateway owns reload planning, provider auth warming runs through a bounded worker path, and inactive SecretRefs are handled by the secret-resolution layer rather than the caller failing globally.
