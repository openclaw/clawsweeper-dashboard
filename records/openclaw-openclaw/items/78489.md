## Summary

This PR should stay open: the follow-up fixed the optional-state parse issue, but the latest patch still gives Codex a copy of the OpenClaw OAuth refresh token while Codex refreshes outside OpenClaw's per-profile refresh lock, and the latest rotation/adoption behavior does not have real-environment proof.

## What This Changes

The PR adds an ACPX-generated Codex auth sync script and wrapper hooks that copy OpenClaw `openai-codex` OAuth credentials into isolated `CODEX_HOME/auth.json` and adopt rotated Codex auth back into the main auth profile.

## Best Possible Solution

Keep one coordinated refresh-token ownership path for Codex ACP, or add an explicitly approved bridge contract that prevents out-of-lock Codex refresh races and documents the behavior with regression coverage and a changelog entry.

## Reproduction Assessment

Yes, source-reproducible: current main creates an isolated Codex ACP `CODEX_HOME` and the existing test asserts that OpenClaw OAuth auth is not synthesized there. I did not run a live Docker ACP session during this read-only review.

## Solution Assessment

No. Post-exit adoption improves recovery, but the proposed bridge still lets Codex refresh the same OAuth token outside OpenClaw's per-profile lock during the ACP run.
