## Summary

This PR still fixes a real current-main cron failure-notification bug: current dedupe ignores explicit primary `delivery.threadId`, while the PR keeps the fallback destination and prevents session-derived topic carry for configured failure sends. I found no new blocking correctness issue in the diff, but the branch still needs rebase/maintainer handling before merge.

## What This Changes

The PR makes cron failure-destination dedupe thread-aware, adds a `suppressSessionThreadId` failure-send path, and covers it with focused cron delivery tests plus a changelog entry.

## Best Possible Solution

Rebase and land the narrow failure-destination fix while preserving current-main session routing and keeping the broader Telegram topic-routing work separate.

## Reproduction Assessment

Yes, at source level. On current main, a cron job with explicit `delivery.threadId` and a same chat/account failure destination can hit `isSameDeliveryTarget()` and have the failure destination suppressed because thread identity is not compared.

## Solution Assessment

Yes, after rebase. The PR targets the narrow dedupe and failure-target resolution seams without expanding cron schema, transport APIs, or plugin contracts.
