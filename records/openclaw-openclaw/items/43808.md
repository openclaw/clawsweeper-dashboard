## What This Changes

The branch updates cron failure-destination dedupe and failure notification target resolution so threadless same-chat Telegram failure destinations are retained and do not inherit the primary topic thread, with regression tests and a changelog entry.

## Best Possible Solution

Keep the narrow cron failure-destination/thread-carry fix, rebase it onto main, and require a live Telegram proof artifact before merge.

## Reproduction Assessment

Yes, by source inspection: current main dedupes a same channel/to/account failure destination without checking delivery.threadId, so a threaded primary delivery can suppress a distinct threadless failure route. I did not run local tests because this review is read-only.

## Solution Assessment

Yes for the code direction: preserving the failure destination when the primary has an explicit thread and suppressing session-derived thread carry on failure sends is the narrowest maintainable fix. The remaining blocker is proof/rebase, not a different implementation.
