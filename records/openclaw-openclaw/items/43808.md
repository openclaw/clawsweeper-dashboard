## Summary

Keep open: this targets a real current-main cron/Telegram bug and the updated PR body now includes acceptable live helper output, but the patch still lets an unthreaded same-chat failure destination inherit the cron delivery session thread at send time.

## What This Changes

The PR changes cron failure-destination dedupe to keep same-chat announce failure destinations when the primary cron delivery has an explicit threadId, adds regression coverage, and adds a changelog entry.

## Best Possible Solution

Resolve failure destinations as an explicit threadless route or compare effective resolved targets including thread context, then add contributor changelog credit and rerun the targeted cron/gateway checks.

## Reproduction Assessment

Yes, by source inspection: current main compares failure destinations against primary delivery by mode/channel/to/accountId while ignoring `delivery.threadId`, so a topic-routed primary and same-chat failure destination can be treated as the same target. I did not run tests because this review was read-only.

## Solution Assessment

No, the current PR fixes the pre-resolution dedupe check but not the later delivery-target resolution that can reattach the session thread to an unthreaded failure destination. The narrower maintainable fix should make the failure route explicitly threadless or dedupe after effective target resolution.
