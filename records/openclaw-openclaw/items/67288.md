## Summary

Keep this open. Current main and v2026.6.10 include the explicit Bedrock Mantle `discovery.enabled` opt-out, but the unset/no-credentials path still calls IAM token generation and can emit the reported diagnostic; the remaining fix is already represented by an open linked PR.

## What This Changes

The issue requests a Bedrock Mantle discovery gate so IAM bearer-token discovery does not run noisily when Mantle is not in use.

## Best Possible Solution

Keep the shipped explicit opt-out, review and land or reject https://github.com/openclaw/openclaw/pull/81208's provider-local log-spam mitigation, then close this issue after the remaining behavior is resolved.

## Reproduction Assessment

Yes. Source inspection shows the current-main path: with discovery unset and no explicit bearer token, `resolveImplicitMantleProvider` still calls IAM token generation, whose failure path logs the reported diagnostic.

## Solution Assessment

Yes for triage. The provider-local boundary is the right place to solve this, and the open linked PR's diagnostic dedupe is safer than a broad no-env skip because it preserves AWS default credential-chain support.
