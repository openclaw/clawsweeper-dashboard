## Summary

Keep open. Current main still lacks the Bedrock Mantle discovery opt-out: the manifest rejects `discovery.enabled`, registration does not pass plugin config into discovery, and the resolver still attempts IAM token generation when no explicit token is present. The issue is also paired with open fix PR #67341 using closing syntax, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests a `plugins.entries.amazon-bedrock-mantle.config.discovery.enabled` opt-out so Mantle implicit discovery does not run IAM token generation when disabled.

## Best Possible Solution

Land PR #67341 if it still applies, or replace it with an equivalent focused Mantle-provider fix: add `discovery.enabled` to the plugin manifest schema and UI hints, pass validated plugin config into discovery, short-circuit before IAM token generation and `/v1/models` when disabled, document the option, and add regression coverage for disabled/default/enabled discovery behavior.

## Reproduction Assessment

Yes. The reporter provided concrete steps and logs, and current-main source inspection gives a high-confidence static reproduction path: the requested config key is rejected, no plugin config reaches the resolver, and IAM token discovery is still attempted.

## Solution Assessment

Yes. A plugin-owned `discovery.enabled === false` gate that mirrors `amazon-bedrock` is the narrowest maintainable solution, provided it includes schema, runtime wiring, docs, and tests.
