## Summary

Keep this open. Current main and v2026.6.8 now include the explicit Bedrock Mantle `discovery.enabled: false` opt-out, but the default no-credentials path still reaches IAM token generation and the reported diagnostic; the remaining mitigation is already paired with https://github.com/openclaw/openclaw/pull/81208 using closing syntax.

## What This Changes

The issue asks for Amazon Bedrock Mantle discovery to be suppressible so IAM token discovery does not run on every request when Mantle is not in use.

## Best Possible Solution

Keep the shipped explicit opt-out, resolve https://github.com/openclaw/openclaw/pull/81208 by landing the accepted Mantle discovery/log-spam mitigation or closing it as not planned, then close this issue.

## Reproduction Assessment

Yes. Source inspection shows the remaining current-main path: with discovery unset and no explicit bearer token, `resolveImplicitMantleProvider` still calls IAM token generation, whose failure path logs the reported diagnostic.

## Solution Assessment

Yes for triage: the provider-local discovery boundary is the right place to solve this. The exact linked PR shape still needs PR review because current head uses log dedupe rather than the no-credentials short-circuit described in the issue.
