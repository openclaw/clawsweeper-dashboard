## Summary

Keep this open. Current main and v2026.6.10 include the explicit Bedrock Mantle `discovery.enabled` opt-out, but the default no-credentials path still calls IAM token generation and can emit the reported diagnostic; the remaining work is already owned by https://github.com/openclaw/openclaw/pull/81208.

## What This Changes

The issue asks Bedrock Mantle to suppress unsolicited IAM/model discovery noise by supporting a `discovery.enabled` gate and avoiding repeated discovery when Bedrock Mantle is unused.

## Best Possible Solution

Keep the shipped explicit opt-out, review and land or reject https://github.com/openclaw/openclaw/pull/81208's provider-local log-spam mitigation, then close this issue after the remaining behavior is merged.

## Reproduction Assessment

Yes. Source inspection shows the current-main path: with discovery unset and no explicit bearer token, `resolveImplicitMantleProvider` still calls IAM token generation, whose failure path logs the reported diagnostic.

## Solution Assessment

Yes for triage: the provider-local discovery boundary is the right place to solve this. The open linked PR's log-dedupe shape is safer than a broad no-env skip because it preserves AWS default credential-chain support, but it still needs normal PR review before this issue closes.
