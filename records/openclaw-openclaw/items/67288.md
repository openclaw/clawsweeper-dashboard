## Summary

Keep open. Current main partially addresses the report by adding the Bedrock Mantle `discovery.enabled` config surface, but the default no-credential path still reaches IAM bearer-token generation and logs the reported diagnostic on each failed discovery attempt; the remaining work is already paired with open PR https://github.com/openclaw/openclaw/pull/81208 using closing syntax.

## What This Changes

The issue requests a Bedrock Mantle discovery opt-out so Mantle discovery does not repeatedly attempt IAM token generation and emit log noise when Bedrock Mantle is not in use.

## Best Possible Solution

Review and land, replace, or close PR https://github.com/openclaw/openclaw/pull/81208; the end state should keep the plugin-owned config gate while suppressing repeated no-credential Mantle discovery diagnostics without breaking AWS default-chain IAM discovery.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: repeated `resolveImplicitMantleProvider` calls without an explicit bearer token still fall through to `generateBearerTokenFromIam`, whose catch block logs the reported Mantle IAM-token diagnostic unless `discovery.enabled` is explicitly false.

## Solution Assessment

Yes. A focused Mantle-owned discovery fix is the narrowest maintainable path; the active linked PR is the right place to finish that work once its review blockers are addressed.
