## Summary

Keep open. Current main now supports the explicit Bedrock Mantle `discovery.enabled=false` opt-out, but the default no-auth path still reaches IAM bearer-token generation and can emit the reported diagnostic; https://github.com/openclaw/openclaw/pull/81208 is open with closing syntax for the remaining focused fix.

## What This Changes

The issue asks Bedrock Mantle discovery to respect a plugin-owned discovery gate and stop repeated IAM-discovery diagnostics when the provider is not in use.

## Best Possible Solution

Review and land, revise, or close https://github.com/openclaw/openclaw/pull/81208; the final state should keep the explicit opt-out and suppress repeated Mantle IAM-unavailable diagnostics without breaking AWS default credential-chain discovery.

## Reproduction Assessment

Yes, by source inspection rather than a live run: repeated `resolveImplicitMantleProvider` calls without `discovery.enabled=false` or `AWS_BEARER_TOKEN_BEDROCK` still call `generateBearerTokenFromIam`, whose failure path logs the reported diagnostic.

## Solution Assessment

Yes. A Bedrock Mantle-local fix is the right boundary, and the open closing PR is the appropriate path to finish the remaining log-spam behavior before this issue closes.
