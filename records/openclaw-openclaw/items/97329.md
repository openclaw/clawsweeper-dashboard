## Label Justifications

- `P2`: The issue describes recurring token/context overhead across non-webchat channel turns, but it is a bounded product improvement rather than an urgent outage.
- `impact:session-state`: The affected metadata is injected into active prompt/session context and can consume context budget over long channel conversations.
- `maturity:stable`: The generic non-webchat metadata path applies to Discord channel turns, and the maturity shortlist reports discord as M4 Stable.

## Summary

Keep open: current main and v2026.6.10 still inject inbound channel metadata with no supported config opt-out, and the linked WeChat plugin discussion supports that this is core-owned behavior. The request is a feature/config/product policy decision, not a stale or already-implemented item.

## What This Changes

The issue asks for a configurable way to disable or narrow non-webchat inbound metadata injection into system prompts and user-message prefixes.

## Best Possible Solution

Choose one core inbound metadata policy that preserves routing/dedupe data for runtime, keeps current defaults, and uses hidden runtime context or a narrow documented opt-out only after maintainers accept the metadata-loss tradeoff.

## Reproduction Assessment

Yes, at source level. Current main still unconditionally builds the trusted inbound metadata block and includes direct non-webchat Conversation info; I did not run a live channel token-measurement scenario.

## Solution Assessment

Unclear. A config opt-out is plausible, but maintainers should first choose whether hidden runtime context, compact metadata, per-channel policy, or a strict opt-out is the intended product shape.
