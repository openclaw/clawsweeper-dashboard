- `P2`: The request targets recurring token/context overhead across channel conversations, but it is a bounded product improvement rather than an outage.
- `impact:session-state`: The affected data is injected into the active system/user prompt context and can pressure session context over long conversations.

## Summary

Keep open: current main and v2026.6.10 still inject inbound channel metadata with no supported config opt-out, and the linked WeChat plugin discussion supports that this is core-owned behavior. The request adds new configuration/product surface, so it needs maintainer direction rather than an automatic fix PR.

## What This Changes

Request to add a backward-compatible `openclaw.json` or channel-level option to disable or reduce non-webchat inbound metadata injected into agent prompts.

## Best Possible Solution

Settle one core inbound metadata policy that keeps routing/dedupe fields available to core, preserves current defaults, and adds a narrow documented opt-out or compact mode only if hidden-runtime-context work does not solve the token pressure.

## Reproduction Assessment

Yes, at source level. Current main unconditionally builds the trusted inbound metadata block and includes direct non-webchat conversation info; I did not run a live channel token-measurement scenario.

## Solution Assessment

Unclear. A config opt-out is plausible, but maintainers should first choose whether the best shape is a compact metadata mode, hidden runtime context, per-channel policy, or a strict opt-out.
