## Summary

Keep open: current main and v2026.6.10 still inject trusted inbound metadata and non-webchat Conversation info without a supported config opt-out. The linked WeChat plugin discussion supports that this is core-owned behavior, but adding a new config surface needs maintainer product direction and coordination with overlapping hidden-runtime-context work.

## What This Changes

The issue requests an openclaw.json control for trusted inbound metadata and Conversation info prompt injection, globally or per channel, while preserving default behavior.

## Best Possible Solution

Choose one core inbound metadata policy that preserves runtime routing/dedupe data, keeps current defaults, and adds a narrow documented opt-out or hidden-runtime-context design only after maintainers accept the metadata-loss tradeoff.

## Reproduction Assessment

Yes, at source level. Current main still unconditionally builds the trusted inbound metadata block and includes direct non-webchat Conversation info; I did not run a live channel token-measurement scenario.

## Solution Assessment

Unclear. A backwards-compatible opt-out is plausible, but maintainers should first choose whether hidden runtime context, compact metadata, per-channel policy, or a strict opt-out is the intended product shape.
