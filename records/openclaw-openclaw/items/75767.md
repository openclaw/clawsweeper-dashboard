## Summary

Keep open: current main still reaches `lsof` on the reported macOS gateway `--force`, restart-health, and stale-cleanup paths when the port is busy, so the SMB stat-timeout report remains a narrow valid runtime bug.

## What This Changes

Report that `openclaw gateway restart` and `openclaw gateway --force` can hang on macOS because port inspection invokes `lsof`, which stalls on unresponsive SMB/Time Machine mounts.

## Best Possible Solution

Make routine gateway busy/free checks socket-probe-first, and keep any remaining process attribution bounded, timeout-protected, and limited to verified OpenClaw-owned listener cleanup.

## Reproduction Assessment

Yes for source-level reproduction: current main still invokes `lsof` from busy `gateway --force`, restart diagnostics, and Unix stale cleanup. A live timing reproduction requires macOS with a slow or unresponsive SMB/Time Machine mount.

## Solution Assessment

Yes: the socket-probe-first direction is the narrowest maintainable fix for availability checks, while process-attribution code should remain only where OpenClaw must identify or terminate a listener.
