## Summary

Keep open: current main and the latest release still use `lsof` in the Unix gateway port-inspection paths that the report identifies, including async diagnostics, `--force`, stale restart cleanup, and restart-health/status callers. The open related mitigation at https://github.com/openclaw/openclaw/pull/90622 is useful but explicitly partial, so this issue still tracks real remaining availability work.

## What This Changes

The issue asks OpenClaw to avoid macOS gateway restart/status/force hangs caused by `lsof` blocking on SMB-mounted volumes during gateway port checks.

## Best Possible Solution

Keep this issue open until the gateway hot paths either avoid `lsof` for port-in-use checks or tightly bound/batch `lsof` behind a maintainer-approved diagnostics path, with macOS-focused proof for restart, force, and status behavior.

## Reproduction Assessment

Yes for the source-level path: current main still routes Unix gateway port checks through the reported per-port `lsof` commands in restart, force, status, and health surfaces. No for the full SMB-mounted-volume delay in this review because that needs a macOS host with a slow or busy SMB mount.

## Solution Assessment

No, the issue is not solved yet. The requested pure Node port probe is directionally useful for the hot port-in-use check, but the best fix should preserve optional owner diagnostics while removing or bounding `lsof` from latency-sensitive restart/force/status paths.
