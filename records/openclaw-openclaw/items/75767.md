## Summary

Keep open. Current `main` still routes the reported macOS `gateway --force` and restart-health paths through `lsof`, including an unbounded synchronous listener scan for forced startup, so the SMB-mounted-volume hang surface remains present. The related reports cover macOS LaunchAgent restart recovery, not this distinct port-inspection stall.

## What This Changes

The issue asks OpenClaw to stop using `lsof` for macOS gateway restart and forced-start port checks because SMB-mounted volumes can make `lsof` block or emit repeated `stat()` warnings.

## Best Possible Solution

Implement a socket-probe-first path for gateway restart health and routine busy/free checks, while keeping any remaining PID discovery narrowly scoped, bounded, and macOS-aware only where OpenClaw must terminate a listener.

## Reproduction Assessment

Yes for source-level reproduction: current `main` still calls the reported `lsof` commands from `gateway --force`, restart health, and stale gateway cleanup. A full end-to-end reproduction still needs macOS with a slow or unresponsive SMB/Time Machine mount.

## Solution Assessment

Yes, the issue's socket-probe direction is the best default for port busy/free checks. The safer complete fix is not a pure replacement everywhere: `--force` still needs bounded PID attribution or a safe fallback before killing listeners.
