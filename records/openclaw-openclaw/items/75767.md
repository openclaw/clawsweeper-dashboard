## Summary

Keep open. Current main still routes the reported macOS gateway `--force` and restart-health paths through `lsof`, including an unbounded synchronous `execFileSync` path for forced startup, so the SMB-mounted-volume hang surface is still present. The related reports in the provided context cover LaunchAgent restart recovery, not this distinct port-inspection stall.

## What This Changes

The issue asks OpenClaw to stop using `lsof` for macOS gateway restart and forced-start port checks because SMB-mounted volumes can make `lsof` hang or emit repeated stat warnings.

## Best Possible Solution

Keep the issue open and implement a socket-probe-first path for gateway restart health and routine busy/free checks, while limiting any remaining process-attribution work to bounded, macOS-aware PID discovery only where OpenClaw actually needs to terminate a listener.

## Reproduction Assessment

Yes for source-level reproduction: current main still calls the reported `lsof` commands from `gateway --force`, restart health, and stale gateway cleanup. A full end-to-end reproduction still needs macOS with a slow or unresponsive SMB/Time Machine mount.

## Solution Assessment

Yes, the issue's socket-probe direction is the best default for port busy/free checks. The safer complete fix is not a pure replacement everywhere: `--force` still needs bounded PID attribution or a safe fallback before killing listeners.
