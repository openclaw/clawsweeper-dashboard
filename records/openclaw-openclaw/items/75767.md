## Summary

Keep open: current main and v2026.6.10 still route macOS-relevant restart, force, status, and restart-health port checks through targeted Unix `lsof` calls, while the open related batching PR is explicitly partial and does not cover the central restart/force SMB stat-timeout path.

## What This Changes

The issue asks OpenClaw to stop latency-sensitive gateway port checks from hanging on macOS systems with slow SMB-mounted volumes by avoiding or tightly bounding `lsof`.

## Best Possible Solution

Keep this open while maintainers choose a shared port-inspection design that removes or tightly bounds `lsof` from restart, force, status, and restart-health hot paths while preserving optional owner diagnostics only where needed.

## Reproduction Assessment

Partly yes: current-main source still reaches the reported per-port `lsof` commands in restart, force, status, and restart-health paths. I did not live-reproduce the macOS SMB stat timeout in this read-only review.

## Solution Assessment

No, this is not solved yet. A pure Node bind probe is a plausible direction for busy/free checks, but the maintainable fix should separate latency-sensitive port health from optional owner diagnostics and account for the open partial batching PR.
