## Summary

Keep open: current main still reaches `lsof` on the reported macOS gateway `--force`, restart-health, and stale-cleanup paths, so the SMB stat-timeout report remains a narrow valid runtime bug.

## What This Changes

The issue reports macOS gateway restart and force paths hanging because port inspection invokes `lsof` while SMB or Time Machine volumes are slow or unresponsive.

## Best Possible Solution

Make routine gateway busy/free checks socket-probe-first, and keep any remaining PID attribution bounded, timeout-protected, and limited to verified OpenClaw-owned listener cleanup.

## Reproduction Assessment

Yes for source-level reproduction: current main still invokes `lsof` from `gateway --force`, restart diagnostics, and Unix stale cleanup. A live timing reproduction requires macOS with a slow or unresponsive SMB or Time Machine mount.

## Solution Assessment

Yes: the proposed socket-probe-first direction is the narrowest maintainable fix for availability checks. Process-attribution code should remain only where OpenClaw must identify or terminate a listener.

## Repair Work Prompt

Fix the macOS SMB/Time Machine `lsof` stall reported in https://github.com/openclaw/openclaw/issues/75767. Treat this as a narrow runtime bug: routine gateway busy/free checks should be socket-probe-first and must not call `lsof` just to decide whether a port is available. Keep PID attribution or process killing only where OpenClaw must identify verified OpenClaw-owned listeners, and keep that path bounded by timeouts. Likely files: `src/cli/ports.ts`, `src/infra/ports-inspect.ts`, `src/infra/restart-stale-pids.ts`, `src/cli/daemon-cli/restart-health.ts`, plus focused tests and `CHANGELOG.md`. Add regression coverage that proves the `gateway --force` no-listener/busy-free polling and restart-health availability paths can avoid `lsof` when socket probing is enough, while preserving safe kill behavior for known OpenClaw listeners. Do not add a config flag, do not broaden kill behavior to arbitrary processes, and do not move extension-specific behavior into core. Add a single-line `CHANGELOG.md` fix entry if the change is user-facing.
