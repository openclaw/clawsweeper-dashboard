## Label Justifications

- `P2`: The reported gateway restart/status hang has a concrete macOS/network-mount trigger and limited platform blast radius.
- `impact:crash-loop`: The issue is about process-level availability: gateway restart/status can hang for long periods while polling port state.

## Summary

Keep open: the reporter supplied a May 30 follow-up that the macOS SMB hang still reproduces, and current main plus latest release still run Unix/macOS gateway port inspection through `lsof` on restart, force, and status paths. The related open PR https://github.com/openclaw/openclaw/pull/90622 is useful partial mitigation, but it is not merged and explicitly leaves restart-health/single-port surfaces out of scope.

## What This Changes

The issue asks OpenClaw to stop using macOS `lsof` as the blocking gateway port-health/restart check because SMB-mounted volumes can make `lsof` hang or fail.

## Best Possible Solution

Move busy/free gateway port decisions onto filesystem-independent bind or reachability probes, and keep `lsof` only as bounded, best-effort process attribution that cannot block restart/status workflows.

## Reproduction Assessment

No live macOS SMB reproduction was run here, but source inspection gives a high-confidence path to the risky behavior: current Unix/macOS restart/status/force code still invokes `lsof` before or during port-state decisions. The full SMB stall duration remains reporter-provided rather than locally reproduced.

## Solution Assessment

Yes, using Node bind or reachability probes for hot-path busy/free decisions is the best direction. Batching `lsof` is a useful partial mitigation for multi-port diagnostics, but it does not solve single-port restart/force hangs when `lsof` itself stalls.
