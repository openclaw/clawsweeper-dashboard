## Summary

Keep open. Current main still uses `lsof` on the reported macOS gateway restart and `gateway --force` paths, including one synchronous `execFileSync` call without a timeout, and the existing Node bind probe does not replace those listener-enumeration calls. The related closed restart issues addressed LaunchAgent loading and handoff behavior, not the SMB-mounted-volume `lsof stat()` hang described here.

## What This Changes

Avoid macOS `openclaw gateway restart` and `openclaw gateway --force` hangs caused by `lsof` traversing slow or unavailable SMB-mounted volumes during port inspection.

## Best Possible Solution

Refactor the gateway port-inspection paths so macOS restart and force flows do not depend on unbounded or repeated `lsof` calls for busy/free checks, while preserving the existing safety rules around which processes may be signaled or killed.

## Reproduction Assessment

Yes. The issue gives a concrete macOS SMB setup and warning text, and current-main source still invokes the same `lsof -nP -iTCP:<port> -sTCP:LISTEN` command on the reported command paths. I did not live-reproduce the SMB mount hang, but the static path matches the symptoms closely.

## Solution Assessment

Partly. The proposed Node bind probe is the right primitive for busy/free checks, but it is not sufficient by itself for `--force` PID discovery and safe process termination; the maintainable fix should combine bind probes with bounded, safe listener identification only where needed.
