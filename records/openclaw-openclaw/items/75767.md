## Summary

Current `main` still routes the reported macOS gateway restart and forced-start paths through `lsof`. The issue is fresh, specific, and source inspection shows a real remaining hang surface: `gateway --force` uses synchronous `execFileSync` without a timeout, and restart health can repeatedly call the async port inspector that tries `lsof` before socket probing.

## What This Changes

The issue asks OpenClaw to avoid `lsof`-based gateway port inspection on macOS because SMB-mounted volumes can make `openclaw gateway restart` and `openclaw gateway --force` hang or time out.

## Best Possible Solution

Move gateway restart health and startup availability checks to a socket-probe-first path that does not need process attribution on every poll, and make any remaining PID discovery for `--force` or stale gateway cleanup bounded, macOS-aware, and covered by regression tests for slow or warning-emitting `lsof`.

## Reproduction Assessment

Yes for source-level reproduction: current `main` still calls the reported `lsof` commands from `gateway --force`, restart health, and stale-PID cleanup. A full end-to-end reproduction needs macOS plus a slow or unresponsive SMB/Time Machine mount.

## Solution Assessment

Mostly yes: the issue's socket-probe direction is the right default for busy/free checks, but `--force` still needs a bounded PID-discovery path so OpenClaw does not lose its ability to terminate the existing listener.
