## Summary

Keep open: current main and v2026.6.10 still route Unix gateway restart, force, status, and restart-health port checks through the reported per-port `lsof` paths, and the related batching PR is open, conflicting, and explicitly partial.

## What This Changes

The issue reports that macOS gateway restart and force/status paths hang when OpenClaw probes ports with `lsof` while SMB-mounted volumes are slow or blocked.

## Best Possible Solution

Keep this issue open while maintainers decide the shared port-inspection design: remove or tightly bound `lsof` from latency-sensitive restart, force, status, and restart-health paths, while preserving owner diagnostics only where they are actually needed.

## Reproduction Assessment

Partly yes: current-main source still reaches the reported per-port `lsof` commands in restart, force, status, and restart-health paths. No live macOS SMB-mounted-volume timeout was reproduced in this read-only review.

## Solution Assessment

No, the issue is not solved yet. A pure Node bind probe is a plausible direction for busy/free checks, but the maintainable fix should separate latency-sensitive port health from optional owner diagnostics and account for the open partial batching PR.
