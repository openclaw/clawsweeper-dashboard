## Summary

Keep open: current main still reaches `lsof` before socket probes on the reported macOS `gateway --force`, restart diagnostics, and Unix stale-cleanup paths, so the SMB stat-timeout report remains a narrow source-confirmed runtime bug. The latest changelog entry only removes one redundant `--force` scan after a no-listener result; it does not make the affected paths socket-probe-first.

## What This Changes

The issue requests replacing or de-prioritizing `lsof`-based gateway port inspection so macOS restarts do not hang on slow SMB or Time Machine mounts.

## Best Possible Solution

Make routine gateway busy/free checks socket-probe-first, and keep any remaining process attribution bounded, timeout-protected, and limited to verified OpenClaw-owned listener cleanup.

## Solution Assessment

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75767. On macOS/Unix, `openclaw gateway --force` and restart diagnostics should not invoke unbounded or lsof-first scans for routine gateway port availability because macOS `lsof` can hang on slow SMB/Time Machine mounts. In `src/cli/ports.ts`, make `forceFreePortAndWait` and availability polling socket-probe-first where no PID attribution is required, and avoid repeated `lsof` scans after kill attempts. In `src/infra/ports-inspect.ts`, make `inspectPortUsage` determine free/busy via `tryListenOnPort` before process attribution, and keep any attribution fallback bounded. In `src/infra/restart-stale-pids.ts`, preserve conservative verified-OpenClaw stale process cleanup, but avoid or tightly bound repeated Darwin `lsof` polling and use socket probing for port-free waits where possible. Update focused tests for the SMB/lsof-stall regression and add a single-line `CHANGELOG.md` fix entry with `Fixes #75767` and `Thanks @Enominera`. Do not change Windows port helpers, LaunchAgent restart policy, or unrelated gateway startup behavior.
