## Summary

Keep open. Current main still routes the reported macOS gateway `--force` and restart-health paths through `lsof`, including an unbounded synchronous forced-start listener scan, so the SMB-mounted-volume hang remains a real source-level bug rather than a cleanup candidate.

## What This Changes

The issue reports that macOS gateway restart and `--force` can hang when `lsof` blocks on SMB-mounted volumes, and asks for socket-probe-first port checks or bounded fallbacks instead of routine `lsof` inspection.

## Best Possible Solution

Implement a socket-probe-first path for gateway restart health and routine busy/free checks, and reserve process attribution for the narrow cases where OpenClaw must terminate a listener. Any remaining `lsof` use on macOS should be bounded and off the common health path.

## Reproduction Assessment

Yes for source-level reproduction: current main still invokes the reported `lsof` commands from `gateway --force`, restart health, and stale gateway cleanup. Full end-to-end timing reproduction requires macOS plus a slow or unresponsive SMB/Time Machine mount.

## Solution Assessment

Yes, the issue's socket-probe direction is the best default for busy/free checks. The complete fix should not blindly remove all PID attribution, because `--force` and stale cleanup still need a bounded, safe way to identify gateway listeners before killing processes.
