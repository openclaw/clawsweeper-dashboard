## Label Justifications

- `P2`: This is a focused Feishu document-write correctness bugfix with limited blast radius to the bundled Feishu plugin, but the current patch has a merge-blocking error-handling regression.
- `merge-risk: 🚨 compatibility`: The diff changes existing failure semantics from explicit `documentBlock.list` code checking to an iterator path that can hide failures and continue with a write.
- `merge-risk: 🚨 availability`: The diff can make one write issue multiple Feishu list requests and the SDK iterator hides request failures, so rate-limit and transient API behavior remain merge-relevant.

## Summary

Keep open: the Feishu clear-document pagination bug is real, but this PR is not merge-ready because it replaces checked `documentBlock.list` handling with `listWithIterator`, whose SDK implementation can hide list failures and let `writeDoc` insert replacement content without clearing stale blocks first.

## What This Changes

This PR changes Feishu `clearDocumentContent` to collect document blocks through `documentBlock.listWithIterator` and adds a mocked two-page write regression test.

## Best Possible Solution

Use an explicit checked Feishu docx pagination helper that calls `documentBlock.list` with `page_token`, preserves response-code failures, and either lands narrowly for clearing or is shared across `read`, `list_blocks`, and clearing after proof or maintainer acceptance.

## Reproduction Assessment

Yes, from source and dependency inspection: current main and `v2026.6.8` use a single-page clear path, while this PR's iterator path can treat list failures as empty pagination. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

No. Paginating the clear path is necessary, but `listWithIterator` is the wrong primitive here because it does not preserve the existing response-code/error contract; an explicit checked pagination helper is safer and can also cover sibling block-list consumers if maintainers choose the broader fix.
