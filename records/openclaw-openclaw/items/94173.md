## Label Justifications

- `P2`: This is a focused Feishu document-write correctness bugfix with limited blast radius to the bundled Feishu plugin, but the current patch has a merge-blocking error-handling regression.
- `merge-risk: 🚨 compatibility`: The diff changes existing failure semantics from explicit `documentBlock.list` code checking to an iterator path that can hide failures and continue with a write.
- `merge-risk: 🚨 availability`: The diff can make one write issue multiple Feishu list requests and the SDK iterator hides request failures, so rate-limit and transient API behavior remain merge-relevant.

## Summary

Keep open: the Feishu clear-pagination bug is real, but this PR is not merge-ready because it replaces the explicit `documentBlock.list` response-code check with an SDK iterator that catches request failures and yields `null`, so `write` can skip clearing and then insert replacement content into an uncleared document. The PR also still lacks after-fix proof from a real Feishu tenant/API path, and the broader overlapping PR is not a safe supersession yet.

## What This Changes

This PR changes Feishu `clearDocumentContent` to gather document blocks through `documentBlock.listWithIterator` and adds a mocked two-page write regression test.

## Best Possible Solution

Use an explicit Feishu docx pagination helper that calls `documentBlock.list` with `page_token`, preserves response-code failures, and either lands narrowly for clearing or is shared across `read`, `list_blocks`, and clearing after real Feishu proof or explicit maintainer acceptance.

## Reproduction Assessment

Yes, from source and dependency inspection: current main and the latest release use a single-page clear path, while this PR's iterator path can yield `null` on SDK request failure and then skip deletion. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

No. Paginating the clear path is necessary, but `listWithIterator` is the wrong primitive here because it does not preserve the existing response-code/error contract; an explicit checked pagination helper is safer and can also cover sibling block-list consumers if maintainers choose the broader fix.
