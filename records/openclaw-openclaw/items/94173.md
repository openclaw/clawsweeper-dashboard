## Summary

Keep open: the PR addresses a real Feishu document-clear pagination bug, but the proposed `listWithIterator` path does not preserve the old checked failure behavior before a destructive write inserts replacement content.

## What This Changes

The PR changes Feishu document clearing to collect blocks with `documentBlock.listWithIterator` before computing the batch delete range and adds a two-page regression test.

## Best Possible Solution

Land a Feishu docx clear pagination fix that explicitly paginates `documentBlock.list` with per-page response checks and regression coverage for multi-page clear plus page-list failure; broaden to a shared helper only if maintainers want `read` and `list_blocks` covered now.

## Reproduction Assessment

Yes from source and dependency inspection, not live tenant replay: current main and `v2026.6.8` call `documentBlock.list` once before delete, while the pinned SDK declares the endpoint paginated and the PR test demonstrates the undercounted delete range for multi-page blocks.

## Solution Assessment

No as written. Paginating the clear path is necessary, but the safest fix preserves checked list-page failures before insert; a shared helper is cleaner if maintainers want sibling docx consumers fixed too.
