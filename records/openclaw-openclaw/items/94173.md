## Summary

Keep open: the Feishu clear-document pagination bug is real, but this PR is not merge-ready because the SDK iterator path can suppress list failures and the supplied proof is only mocked. The broader overlapping PR at https://github.com/openclaw/openclaw/pull/94632 should be reconciled before either branch lands.

## What This Changes

The PR replaces `clearDocumentContent`'s single Feishu `documentBlock.list` call with the SDK `listWithIterator` and adds a mocked multi-page write regression test.

## Best Possible Solution

Use a checked Feishu docx block-list pagination helper that propagates list failures and is shared by clear, read, and list_blocks; then land either this repaired PR or https://github.com/openclaw/openclaw/pull/94632 after acceptable live proof or explicit maintainer acceptance of local contract proof.

## Reproduction Assessment

Yes from source, not from a live tenant: current main and `v2026.6.8` call `documentBlock.list` once in `clearDocumentContent` while the locked Feishu SDK marks the endpoint paginated. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

No. The PR fixes the first-page truncation shape, but using the SDK iterator drops explicit error handling, and a checked shared pagination helper for all docx block-list consumers is the narrower maintainable fix.
