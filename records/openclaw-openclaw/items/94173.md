## Summary

Keep open: the Feishu clear-document bug is valid, but this PR is not merge-ready because the SDK iterator path weakens the existing checked failure behavior and the PR still has only mocked test evidence, not real Feishu proof.

## What This Changes

The PR changes Feishu document clearing to gather blocks through `documentBlock.listWithIterator` and adds a mocked two-page regression test for `write` clearing top-level blocks.

## Best Possible Solution

Use explicit checked Feishu document-block pagination, preferably via a small owner-module helper if maintainers want all block-list consumers fixed, and require redacted live Feishu proof or explicit maintainer acceptance before merge.

## Reproduction Assessment

Yes from source and dependency inspection, not from a live tenant: current main and `v2026.6.8` call `documentBlock.list` once, while the SDK exposes paginated `page_token`/`has_more` behavior. The PR test mocks the multi-page clear path but does not exercise live Feishu.

## Solution Assessment

No as written. Paginating the clear path is necessary, but `listWithIterator` is not the safest primitive here because it does not preserve the existing checked failure contract; an explicit checked pagination helper is safer.
