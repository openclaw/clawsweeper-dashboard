## Summary

Keep open: this is a valid Feishu clear-document pagination fix candidate, but the patch should not merge as written because the SDK iterator path can hide list failures and the after-fix proof is still fake-client proof rather than a live Feishu setup.

## What This Changes

The PR changes Feishu document clearing to gather document blocks with `documentBlock.listWithIterator` and adds a mocked two-page write regression test.

## Best Possible Solution

Use explicit checked Feishu document-block pagination for the clear path, preferably through a small owner-module helper if maintainers want to cover `read` and `list_blocks`, and require live Feishu proof or explicit maintainer acceptance before merge.

## Reproduction Assessment

Yes from source and dependency inspection, not from a live tenant: current main and v2026.6.8 call `documentBlock.list` once before write, while the locked SDK exposes paginated block listing. The PR test models two pages with a fake client.

## Solution Assessment

No as written. Paginating the clear path is necessary, but the safest fix is explicit checked pagination that preserves list failures before insert; a shared helper is cleaner if maintainers want to fix sibling docx block-list consumers too.
