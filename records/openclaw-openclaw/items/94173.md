## Summary

Keep open: the Feishu large-document clear bug is real and this PR targets the right code path, but the patch changes the existing failure contract and still lacks after-fix proof from a live Feishu/Lark setup.

## What This Changes

The PR changes Feishu document clearing to gather document blocks with `documentBlock.listWithIterator` and adds a mocked two-page `write` regression test.

## Best Possible Solution

Use explicit checked Feishu document-block pagination, preferably through a small owner-module helper if maintainers want all docx block-list consumers fixed, and require redacted live Feishu proof or explicit maintainer acceptance before merge.

## Reproduction Assessment

Yes from source and dependency inspection, not from a live tenant: current main and `v2026.6.8` call `documentBlock.list` once, while the SDK describes docx block listing as paginated. The PR test and body proof use a fake Lark client rather than live Feishu.

## Solution Assessment

No as written. Paginating the clear path is necessary, but the safest fix is explicit checked pagination that preserves `res.code` failures before insert; a shared helper is the cleaner option if maintainers want to fix sibling docx block-list consumers too.
