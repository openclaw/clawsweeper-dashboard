## Summary

Keep open: current main and the latest release still have the single-page Feishu clear bug, but this PR is not merge-ready because the SDK iterator path can suppress list failures before `writeDoc` inserts replacement content, and the supplied proof still uses a fake Lark client rather than a live Feishu/Lark document.

## What This Changes

This PR changes Feishu `clearDocumentContent` to accumulate document blocks with `documentBlock.listWithIterator` before deleting top-level children and adds a two-page regression test.

## Best Possible Solution

Land a Feishu docx clear pagination fix that preserves checked page failure semantics before any replacement insert; if maintainers want the whole docx block-list surface covered, factor a shared owner-module helper for clear, read, and list_blocks.

## Reproduction Assessment

Yes from source and dependency inspection, not live tenant replay: current main and `v2026.6.8` call `documentBlock.list` once before delete, while the pinned SDK declares the endpoint paginated and the PR regression demonstrates the undercounted delete range for multi-page blocks.

## Solution Assessment

No as written. Paginating the clear path is necessary, but the safest fix preserves checked list-page failures before insert; a shared helper is cleaner if maintainers want sibling docx consumers fixed too.

- **[P1] Preserve list failures before inserting replacement content:** `extensions/feishu/src/docx.ts:459-462`
  - body: `listWithIterator` logs request failures without rethrowing and can yield an empty/null page; this loop then treats `page?.items ?? []` as a successful clear. Use explicit checked pagination, or otherwise make every list/page failure abort before insert, and cover that failure path.
