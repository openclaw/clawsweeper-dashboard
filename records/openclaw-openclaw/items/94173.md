## Summary

Keep open: current main and the latest release still have the Feishu clear pagination bug, but this PR is not merge-ready because it swaps the checked `documentBlock.list` response for an SDK iterator that can suppress list failures before the destructive write continues.

## What This Changes

The PR changes Feishu document clearing to collect document blocks with `documentBlock.listWithIterator` before computing the delete range, and adds a regression test for two block-list pages.

## Best Possible Solution

Land a Feishu docx pagination fix that explicitly checks each list response before any replacement insert; if maintainers want the whole docx block-list surface covered, factor that into one owner-module helper for clear, read, and list_blocks.

## Reproduction Assessment

Yes from source and dependency inspection, not from a live tenant: current main and `v2026.6.8` call `documentBlock.list` once, while the SDK declares document block listing as paginated. The PR test and body proof use a fake Lark client rather than live Feishu.

## Solution Assessment

No as written. Paginating the clear path is necessary, but the safest fix is explicit checked pagination that preserves list failures before insert; a shared helper is cleaner if maintainers want sibling docx consumers fixed too.
