## Summary

Keep open: current main and the latest release still use a single-page Feishu docx clear path, and this PR is useful, but it is not merge-ready because the SDK iterator path can suppress block-list failures on a destructive write and the proof is still fake-client only.

## What This Changes

The branch replaces the Feishu doc clear helper's single block-list call with SDK iterator accumulation and adds a two-page write regression test.

## Best Possible Solution

Land a checked Feishu docx pagination fix that preserves list-page failures before insertion; use a shared helper if maintainers choose to cover read/list_blocks too.

## Reproduction Assessment

Yes from source inspection: current main and `v2026.6.8` call `documentBlock.list` once before delete, while the SDK marks the endpoint paginated and the PR test demonstrates the multi-page undercount. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

No as written. Paginating the clear path is necessary, but using `listWithIterator` is not the safest fix because the pinned SDK can suppress request failures; explicit checked `page_token` pagination is the narrower maintainable repair.
