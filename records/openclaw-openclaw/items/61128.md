## Summary

Keep open. Current main still does not implement the requested bounded, metadata-validated skill-loader cache: the normal loader paths still scan roots and read/parse `SKILL.md` on each call, while existing reuse is limited to higher-level session snapshots and a bundled skill-name cache. The author-linked PR #59547 is closed and unmerged, so there is no shipped implementation to close against.

## What This Changes

The issue requests a conservative in-memory cache for direct validated skill loads, with metadata invalidation, containment rechecks, stale pruning, bounded eviction, and unchanged external behavior.

## Best Possible Solution

Add the cache inside the skill-loading layer: cache only direct validated skill loads, validate hits with `SKILL.md` metadata, rerun containment checks before returning cached records, prune scanned-root entries, bound process growth, and keep nested/grouped fallback loads uncached unless their dependencies are explicit.

## Reproduction Assessment

Yes for the structural behavior: repeated calls to `loadWorkspaceSkillEntries` or `buildWorkspaceSkillSnapshot` against unchanged roots still execute the synchronous scan/stat/read/parse path on current main. No high-confidence latency reproduction is available because the report does not include benchmark data.

## Solution Assessment

Unclear pending maintainer acceptance of the performance tradeoff, but the requested direct-load cache is a reasonable narrow approach if it remains bounded and safety-revalidating. Relying only on session snapshots is not enough because status, command, and runtime paths can still call the loader directly.
