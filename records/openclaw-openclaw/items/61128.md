## Summary

Keep open. Current main still lacks the requested bounded, metadata-validated skill-loader cache: direct workspace and local skill loading still scan directories and read/parse `SKILL.md` on each call, while the author-linked PR #59547 is closed and unmerged.

## What This Changes

The issue requests a conservative in-memory cache for repeated skill loads, with metadata validation, path safety rechecks, stale pruning, and bounded growth.

## Best Possible Solution

Add the cache in the skill-loading layer with direct-load-only entries, metadata validation on hits, root/symlink rechecks, scanned-root pruning, bounded process growth, and regression tests for invalidation and safety.

## Reproduction Assessment

Yes for the structural behavior: repeated calls to `loadWorkspaceSkillEntries` or `buildWorkspaceSkillSnapshot` against unchanged roots still execute the synchronous scan/stat/read/parse path on current main. No high-confidence latency reproduction is available because the report does not include benchmark data.

## Solution Assessment

Yes, if kept conservative: a direct-load-only cache with metadata and containment revalidation is the narrow maintainable fix. Higher-level session snapshot reuse is not enough because direct loader entry points remain uncached.
