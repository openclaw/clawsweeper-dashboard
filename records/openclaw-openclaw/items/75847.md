## Summary

Keep open. Current main has the iMessage reply dispatch path and upstream tool-event callback surface, but the iMessage monitor does not install any per-turn tool-activity watchdog or pass tool-progress callbacks into the reply turn, so the reported silent-busy behavior remains valid. The linked claude-cli event bridge issue is also a real prerequisite for that backend because the current JSONL streaming parser only surfaces assistant text deltas.

## What This Changes

The issue requests a per-turn iMessage monitor watchdog that sends one non-spammy progress ping when a turn is silent past a threshold after tool activity has occurred.

## Best Possible Solution

Add an iMessage-owned per-turn watchdog around the existing reply dispatch path: track last visible outbound/streaming activity, record tool/item/command activity through replyOptions callbacks, send exactly one sanitized progress ping after the silence threshold, and clear all timers on final reply, idle, abort, or dispatch failure. If the repair is promoted, handle the claude-cli tool_use bridge in the same focused cluster or land it first so the watchdog has events to observe.

## Reproduction Assessment

Yes. A focused source-level reproduction can drive an iMessage inbound turn with a reply resolver that calls onToolStart and then stays silent; current main passes no onToolStart callback and has no per-turn timer, so no iMessage progress message can be emitted.

## Solution Assessment

Yes, the requested direction is narrow and maintainable if implemented through existing per-run replyOptions callbacks rather than a global onAgentEvent listener. The safer shape is one channel-local watchdog with sanitized messages and tests proving one ping per turn, reset between turns, and no ping after visible output or final delivery.

## Repair Work Prompt

Fix the iMessage silent-busy turn gap reported in https://github.com/openclaw/openclaw/issues/75847, preserving plugin/channel ownership. In `extensions/imessage/src/monitor/monitor-provider.ts`, add a per-turn watchdog around the existing `dispatchInboundMessage` call that tracks the last visible outbound/progress activity, observes tool activity through existing replyOptions callbacks such as `onToolStart`, `onItemEvent`, and `onCommandOutput`, and sends at most one sanitized iMessage progress ping after the silence threshold when tool activity has occurred but no visible output has been delivered. Reset/clear the watchdog for every turn and on final reply, idle, abort, or dispatch failure; do not spam, do not leak tool args/output/secrets, and do not change unrelated channel behavior. Add focused iMessage monitor tests proving: no ping without tool activity, one ping after silent tool activity, no duplicate ping in the same turn, reset between turns, and no ping after visible block/final output. Because this is gated for claude-cli by https://github.com/openclaw/openclaw/issues/75846, either include the narrow `tool_use` to `stream:tool` bridge in `src/agents/cli-output.ts`, `src/agents/cli-runner/execute.ts`, and `src/agents/cli-runner/claude-live-session.ts` with tests, or explicitly keep the iMessage tests backend-agnostic and leave #75846 as a prerequisite. Add a user-facing `CHANGELOG.md` fix entry if runtime behavior changes. Do not add dependencies, do not move iMessage behavior into core unless a tiny generic callback helper is genuinely needed, and do not change BlueBubbles behavior unless a shared helper requires a small no-op-compatible adjustment.
