## Summary

Keep open. Current main still lacks an iMessage-owned per-turn watchdog or progress callback wiring, and the linked claude-cli tool-event bridge remains separate work needed for full coverage.

## What This Changes

The issue requests an iMessage monitor watchdog that sends one progress ping when a turn is silent past a threshold after tool activity has occurred.

## Best Possible Solution

Implement an iMessage-local per-turn watchdog on the existing GetReplyOptions progress callbacks, reset it on each inbound turn and visible output, and sequence claude-cli coverage with #75846.

## Reproduction Assessment

Yes. Source inspection is high-confidence: iMessage does not register the progress callbacks, while shared reply execution emits them and Telegram consumes the same seam.

## Solution Assessment

Yes. A channel-local callback/watchdog is the narrowest maintainable direction, but the threshold/copy and #75846 sequencing need maintainer judgment before treating this as an automated repair.
