## Summary

Keep open. Current main still has no iMessage-owned per-turn watchdog for tool/progress activity during agent silence, even though the shared reply pipeline exposes callbacks that other channels already consume; the linked claude-cli event bridge remains a related dependency for full coverage.

## What This Changes

Request an iMessage monitor per-turn progress watchdog that sends one status ping when a turn is silent but tool activity has occurred.

## Best Possible Solution

Implement the watchdog inside the iMessage monitor using existing reply progress callbacks, reset it per inbound turn and visible output, and coordinate it with #75846 for claude-cli tool event coverage.

## Reproduction Assessment

Yes. Static current-main inspection is high-confidence: iMessage dispatch does not register the progress callbacks, while shared reply execution emits them and Telegram already consumes the same seam.

## Solution Assessment

Yes. A channel-local callback/watchdog is the narrowest maintainable path because it reuses GetReplyOptions instead of adding a global event subscription API; #75846 should be fixed or sequenced for claude-cli coverage.

## Repair Work Prompt

Implement the iMessage monitor tool-activity watchdog requested in #75847, and coordinate with #75846 if the claude-cli bridge is included. In extensions/imessage/src/monitor/monitor-provider.ts, wire existing replyOptions progress callbacks such as onToolStart, onItemEvent, onCommandOutput, onPlanUpdate, onApprovalEvent, and onPatchSummary into per-turn iMessage state. Track last visible output and whether tool/progress activity occurred for the current inbound turn; after a conservative silent threshold, send at most one sanitized progress ping to the current iMessage target, then disarm. Reset state on every inbound turn and on visible reply output. Prefer the existing plugin-sdk reply callback seam and iMessage delivery helpers; do not add a new core event API or new config option unless maintainers explicitly ask. Add focused regression coverage for one ping, no spam, per-turn reset, final-output cancellation, and no ping when no tool/progress event occurs. If fixing #75846 in the same PR, extend the claude-cli stream-json parser/live-session path to surface tool_use as stream: tool events with the same shape used by pi-embedded, with regression tests. Add a user-facing CHANGELOG.md entry because this changes iMessage channel behavior.
