## Summary

Keep open. Current main still lacks an iMessage-owned per-turn tool-activity watchdog; shared reply execution already exposes the right progress callbacks and other channels consume them, so this remains a narrow channel-owned enhancement with #75846 as the backend event-source dependency.

## What This Changes

The issue asks for an iMessage monitor per-turn watchdog that sends one progress ping when a turn is silent but tool activity has occurred.

## Best Possible Solution

Implement an iMessage-local watchdog using existing GetReplyOptions progress callbacks and shared progress formatting/gating helpers, reset it per inbound turn and visible output, and keep #75846 as the separate event-source bridge unless fixed in the same PR.

## Reproduction Assessment

Yes. Static current-main source inspection is high-confidence: iMessage does not register the existing progress callbacks, while shared reply execution emits them and Telegram consumes the same seam.

## Solution Assessment

Yes. A channel-local callback/watchdog is the narrowest maintainable solution because it reuses the existing reply callback contract and avoids adding a new global agent-event subscription API.
