## Summary

Keep open. Current main still lacks iMessage-owned per-turn watchdog/progress wiring: the iMessage monitor passes only block-streaming and model-selection options into reply dispatch, while the shared reply layer and other channel monitors already expose and consume tool/progress callbacks; the linked #75846 dependency remains the Claude CLI input gap.

## What This Changes

The issue requests an iMessage monitor per-turn watchdog that sends one progress ping when a turn is silent but tool activity has occurred, with #75846 providing Claude CLI tool-event input.

## Best Possible Solution

Add an iMessage-owned per-turn watchdog fed by existing GetReplyOptions progress callbacks, reset it on visible output and new inbound turns, and bridge Claude CLI tool_use events under #75846 for the reported path end-to-end.

## Reproduction Assessment

Yes. Static current-main verification is high-confidence: the iMessage dispatch path has no progress callbacks or watchdog, while the shared runner and tests already exercise those callbacks for other consumers.

## Solution Assessment

Yes. A channel-local watchdog using the existing reply callback seam is the narrowest maintainable fix; it should avoid global event subscriptions and be clustered with #75846 for Claude CLI coverage.
