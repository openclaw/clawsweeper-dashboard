## Summary

Keep open. Current main still lacks an iMessage-owned per-turn tool-activity watchdog: iMessage only passes block-streaming and model-selection options into reply dispatch, while shared reply execution exposes tool/progress callbacks and Telegram already consumes the same seam. The linked claude-cli bridge gap remains a separate dependency for full coverage.

## What This Changes

Add an iMessage monitor per-turn tool-activity watchdog that sends one progress ping after silent tool activity and resets each turn.

## Best Possible Solution

Implement an iMessage-local watchdog using existing GetReplyOptions progress callbacks, reset it on each inbound turn and visible output, and sequence it with #75846 before claiming claude-cli coverage.

## Reproduction Assessment

Yes. Static current-main source inspection is high-confidence: iMessage does not register the progress callbacks, while shared reply execution emits them and another channel consumes them.

## Solution Assessment

Yes. A channel-local callback/watchdog is the narrowest maintainable path because it reuses the existing reply callback seam and avoids a new global event subscription API.
