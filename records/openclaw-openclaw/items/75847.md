## Summary

Keep open. Current main still lacks an iMessage-owned per-turn watchdog: the iMessage monitor only passes block-streaming and model-selection options into reply dispatch, while shared reply execution already forwards tool/progress callbacks and other channel monitors consume them. The linked #75846 remains a separate prerequisite for claude-cli tool_use events.

## What This Changes

Add a per-turn iMessage monitor watchdog that sends one progress ping after agent silence when tool activity has occurred.

## Best Possible Solution

Implement an iMessage-local per-turn watchdog using existing GetReplyOptions progress callbacks, reset it on each inbound turn and visible output, and land or sequence the #75846 claude-cli bridge before claiming full coverage.

## Reproduction Assessment

Yes. Static current-main verification is high-confidence: iMessage does not register the progress callbacks, while shared reply execution emits them and other channels consume them.

## Solution Assessment

Yes. The requested channel-local watchdog is the narrowest maintainable fix because it reuses the existing reply callback seam and avoids adding a global event subscription path.
