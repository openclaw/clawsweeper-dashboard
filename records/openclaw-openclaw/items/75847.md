## Summary

Keep open. Current main still lacks an iMessage per-turn tool-activity watchdog: the iMessage dispatch path does not pass the existing progress callbacks, while shared reply execution and other channel monitors already expose and consume those events. The linked prerequisite #75846 remains relevant for claude-cli-backed turns.

## What This Changes

Add an iMessage monitor watchdog that sends one non-spammy progress ping during a silent turn after tool activity has occurred.

## Best Possible Solution

Add an iMessage-owned, per-turn, single-shot watchdog using existing reply progress callbacks, reset it on each turn and visible output, and coordinate with #75846 for claude-cli tool-event delivery.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75847. Current main's iMessage monitor dispatches replies without any tool-progress callbacks, so a silent-but-busy turn is indistinguishable from a stalled turn. Add an iMessage-owned, per-turn, single-shot tool-activity watchdog using existing GetReplyOptions callbacks such as onToolStart, onItemEvent, onCommandOutput, onPlanUpdate, onApprovalEvent, and onPatchSummary. Keep it scoped to the active inbound iMessage turn, reset on each turn and on visible assistant/final output, send at most one sanitized progress ping after the silence threshold only when tool/progress activity occurred, and avoid global onAgentEvent subscriptions unless a typed seam is genuinely needed. Coordinate with https://github.com/openclaw/openclaw/issues/75846: if the claude-cli stream:tool bridge has not landed, either include a focused bridge for Claude stream-json tool_use events or leave the watchdog fix clearly compatible with that follow-up. Do not move iMessage-owned behavior into core or add bundled channel ids to generic core policy. Add focused tests and a user-facing changelog entry if behavior changes are shipped.
