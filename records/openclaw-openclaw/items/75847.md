## Summary

Keep open. Current main still lacks an iMessage-owned per-turn watchdog: the iMessage monitor does not pass tool/progress callbacks into reply dispatch, while shared reply execution exposes those callbacks and other channel monitors consume them for visible progress.

## What This Changes

The issue requests an iMessage monitor per-turn watchdog that emits one progress ping when an agent is silent but has tool activity in the current turn.

## Best Possible Solution

Implement the watchdog inside the iMessage monitor using existing `GetReplyOptions` progress callbacks, reset it per inbound turn and visible output, and sequence or cluster the claude-cli event bridge from #75846 for full backend coverage.

## Reproduction Assessment

Yes. Static current-main verification is high-confidence: the iMessage dispatch path has no tool/progress callback consumption, while shared reply execution and other channels already expose and consume those signals.

## Solution Assessment

Yes. A channel-local watchdog that reuses `GetReplyOptions` is the narrowest maintainable fix; a global event subscription would be broader than needed.
