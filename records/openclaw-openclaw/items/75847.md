## Summary

Keep open. Current main still lacks the iMessage-owned per-turn tool-activity watchdog: the iMessage dispatch path does not pass the existing progress callbacks, while shared reply execution and Telegram/Slack/Discord already consume those signals. The report’s local workaround and the linked #75846 prerequisite are useful context for a clustered fix.

## What This Changes

Add an iMessage monitor watchdog that sends one non-spammy progress ping during a silent turn after tool activity has occurred.

## Best Possible Solution

Add an iMessage-owned, per-turn, single-shot progress watchdog using existing reply progress callbacks, reset it on each turn and visible output, and keep any Claude CLI event bridge in the generic runner/parser layer tracked by #75846.

## Solution Assessment

Yes. The narrowest maintainable fix is to consume GetReplyOptions callbacks such as onToolStart/onItemEvent/onCommandOutput inside the iMessage monitor rather than adding a broad global event subscription.

## Repair Work Prompt

Fix the iMessage monitor tool-activity watchdog gap described by this issue. Current main dispatches iMessage replies without tool-progress callbacks, so a silent-but-busy turn is indistinguishable from a stalled turn. Add an iMessage-owned, per-turn, single-shot watchdog using existing GetReplyOptions callbacks such as onToolStart, onItemEvent, onCommandOutput, onPlanUpdate, onApprovalEvent, and onPatchSummary. Keep it scoped to the active inbound iMessage turn, reset on each turn and on visible assistant/final output, send at most one sanitized progress ping after the silence threshold only when tool/progress activity occurred, and avoid global onAgentEvent subscriptions unless a typed seam is genuinely needed. Coordinate with https://github.com/openclaw/openclaw/issues/75846: if the claude-cli stream:tool bridge has not landed, either include a focused bridge for Claude stream-json tool_use events or leave the watchdog fix clearly compatible with that follow-up. Do not move iMessage-owned behavior into core, do not add bundled channel ids to generic core policy, and do not edit post-build dist artifacts. Add focused tests and a single-line CHANGELOG.md fix entry because this changes user-visible channel behavior.
