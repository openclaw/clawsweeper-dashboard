## Summary

Keep open. Current main still lacks an iMessage-owned per-turn tool-activity watchdog: the monitor dispatch path does not subscribe to existing reply progress callbacks, while shared reply execution and other channels already expose the events needed to build this narrowly.

## What This Changes

Request an iMessage monitor watchdog that sends one progress ping when a turn is silent but tool activity has occurred.

## Best Possible Solution

Add a channel-local iMessage watchdog through the existing GetReplyOptions progress callbacks, reset it per inbound turn and visible output, and sequence or cluster the #75846 claude-cli tool-event bridge before claiming full runtime coverage.

## Reproduction Assessment

Yes. Source inspection on current main is high-confidence: iMessage does not register the progress callbacks, while shared reply execution emits them and Telegram consumes the same seam.

## Solution Assessment

Yes. The requested channel-local watchdog is the narrowest maintainable direction because it reuses existing reply callbacks and avoids a new global event subscription path; #75846 should remain the separate CLI event prerequisite unless fixed in the same PR.
