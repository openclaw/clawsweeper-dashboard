## Summary

Keep open. Current main still lacks an iMessage-owned per-turn tool-activity watchdog: the monitor dispatches iMessage turns without tool/item/command progress callbacks, while the shared reply runtime already exposes those callbacks and other chat plugins wire them into visible progress surfaces.

## What This Changes

The issue requests a single per-turn iMessage progress ping when a turn has been silent past a threshold but tool activity has occurred.

## Best Possible Solution

Add an iMessage-owned, per-turn, single-shot watchdog wired through the existing reply progress callbacks. Reset it for each inbound turn and on visible assistant/final output, use sanitized short labels, and coordinate with #75846 so claude-cli emits the same tool activity signal.

## Reproduction Assessment

Yes. Static current-main verification is high-confidence: the iMessage monitor has no tool-progress callback or watchdog, while the shared runtime and other channels already expose and consume those events.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix if implemented through existing GetReplyOptions callbacks rather than a broad global event subscription. That keeps the behavior scoped to the active iMessage turn and avoids a new core channel-specific seam.
