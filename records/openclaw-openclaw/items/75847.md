## Summary

Keep open. Current main still has no iMessage-owned per-turn watchdog or visible progress ping for tool activity during agent silence; it only threads generic typing lifecycle options into the iMessage dispatch path, while the shared reply layer exposes tool/progress callbacks and Telegram consumes them.

## What This Changes

The issue requests an iMessage monitor watchdog that sends one descriptive progress message when a turn is silent but tool activity has occurred.

## Best Possible Solution

Add an iMessage-local progress watchdog on the existing GetReplyOptions callback seam, reset it per inbound turn and on visible output, and coordinate final claude-cli coverage with the linked event-bridge issue.

## Reproduction Assessment

Yes. Static current-main inspection is high-confidence: iMessage does not register tool/progress callbacks or a watchdog, while shared reply execution emits those callbacks and Telegram already consumes them.

## Solution Assessment

Yes, with maintainer UX confirmation. A channel-local watchdog using existing reply callbacks is narrower than adding a new global event subscription API, but the timeout/message policy and claude-cli dependency should be reviewed together.
