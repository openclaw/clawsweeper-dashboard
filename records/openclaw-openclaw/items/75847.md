## Summary

Keep open. Current main still lacks an iMessage per-turn tool-activity watchdog: the iMessage monitor creates a normal reply dispatcher and dispatches with only model/block-streaming options, while the shared reply path already exposes tool/item/command progress callbacks that other channel monitors wire into visible progress surfaces. The linked prerequisite #75846 also remains open in the provided context, so claude-cli routed turns still need the tool-event bridge before this fully helps that backend.

## What This Changes

The issue requests a per-turn iMessage monitor watchdog that sends one progress ping when an agent is silent but has emitted tool activity during the turn.

## Best Possible Solution

Add an iMessage-owned, per-turn, single-shot progress watchdog wired through the existing reply progress callbacks, resetting on each turn and on visible assistant/final output. Coordinate with #75846 so claude-cli emits the same tool activity signal, and keep the behavior inside the iMessage plugin with focused tests and a changelog entry.

## Reproduction Assessment

Yes. Static current-main verification is high-confidence: the iMessage dispatch path has no tool-progress callback or watchdog, while the shared runtime and other channels already expose and consume those events.

## Solution Assessment

Yes, with a safer implementation detail: use the existing `GetReplyOptions` callbacks such as `onToolStart`/`onItemEvent` rather than a broad global event subscription where possible. That keeps the behavior scoped to the active iMessage turn and matches the established channel-monitor pattern.

## Repair Work Prompt

Fix the iMessage per-turn tool-activity silence gap. In the iMessage plugin, add a per-turn watchdog around inbound reply dispatch that records visible output timestamps and tool/item/command progress for the active turn, sends at most one short, redacted progress ping when the turn has been silent past a conservative threshold while tool activity has occurred, and resets/cleans up on turn completion, failure, abort, and visible assistant/final output. Prefer existing `GetReplyOptions` callbacks (`onToolStart`, `onItemEvent`, `onCommandOutput`, possibly `onPlanUpdate`/`onApprovalEvent`) over a broad global `onAgentEvent` subscription unless a local dispatch-scoped seam is not sufficient. Keep the behavior in `extensions/imessage`; do not add iMessage-specific policy to core. Coordinate with #75846: if claude-cli tool events are still not bridged, either fix that narrow bridge in the same cluster or document that this watchdog works for event-producing runtimes and is unblocked by the bridge follow-up. Add focused Vitest coverage proving no ping without tool activity, one ping after silence with tool activity, no repeat pings in the same turn, reset across turns, cleanup on final/error, and no sensitive raw args in the progress text. Add a user-facing `CHANGELOG.md` fix entry; do not add docs unless new config is introduced.
