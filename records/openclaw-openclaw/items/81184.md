## Summary

Keep open: the bug and delivery-boundary fix are plausible, but the patch currently leaves delivered-cache entries unbounded and lacks required real Slack/subagent behavior proof.

## What This Changes

The PR adds process-local subagent announcement delivery deduplication, focused tests, and a changelog entry to avoid duplicate requester-channel completion posts for the same direct idempotency key.

## Best Possible Solution

Keep the shared delivery-boundary dedupe, but bound or prune the delivered cache and land it with focused regression tests plus redacted real Slack/subagent proof.

## Reproduction Assessment

Unclear for the live Slack race; source-level reproduction is clear by invoking current-main `deliverSubagentAnnouncement` twice with the same `directIdempotencyKey`, which re-enters dispatch both times.

## Solution Assessment

No as written: the central delivery boundary is the right place to coalesce duplicate announces, but the delivered cache needs a bounded/pruned implementation before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Bound the delivered announce cache:** `src/agents/subagent-announce-dedupe.ts:83`
  - body: Every successful announce with a unique direct idempotency key is inserted into `state.delivered`, but entries are only removed when that same key is looked up later. These keys are normally one-per child run or media task, so a long-running gateway will retain stale delivered results indefinitely despite the intended five-minute TTL. Prune expired entries on insert/lookups or use a size/timer bounded cache before shipping.
  - confidence: 0.91
