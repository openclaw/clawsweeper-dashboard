## Summary

Keep open: the direction is useful and the supplied proof shows same-key dispatch coalescing, but the new process-local cache is unbounded for one-off announce keys and still misses the delayed retry case after the gateway TTL rolls over.

## What This Changes

Adds a process-local dedupe/coalescing wrapper around subagent announcement delivery keyed by `directIdempotencyKey`, with unit and delivery tests for repeat, failed, and concurrent dispatches.

## Best Possible Solution

Keep the delivery-boundary dedupe, but make the delivered-result cache bounded and align suppression duration with the completion retry window while preserving retry semantics for failed dispatches.

## Reproduction Assessment

Yes. Source inspection shows current main builds a stable announce idempotency key and calls the dispatch boundary each time; the PR body also supplies before/after terminal output showing two dispatches before the wrapper and one after.

## Solution Assessment

No, not in its current form. The delivery boundary is the right place to coalesce same-key races, but the cache needs bounded eviction and coverage beyond the gateway TTL before it is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bound the delivered announce cache:** `src/agents/subagent-announce-dedupe.ts:88`
  - body: Delivered entries are only checked for expiry when the same key is looked up again. A normal announce key is unique per run, so most entries will never be touched again and will remain in `state.delivered` for the process lifetime despite the TTL. Add a small sweep, scheduled cleanup, or size bound when inserting so the process-local cache cannot grow without bound.
  - confidence: 0.91
- **[P2] Cover retries after the gateway TTL rolls over:** `src/agents/subagent-announce-dedupe.ts:16`
  - body: The new cache expires delivered results after the same 5 minutes as the gateway idempotency window. A retry for the same `directIdempotencyKey` after that point misses both guards and re-enters `runSubagentAnnounceDispatch`, so the gateway-TTL-rollover case described in the PR body can still post a second Slack message. Keep delivered suppression through the full completion retry window or use another durable delivered marker, and add a post-TTL regression test.
  - confidence: 0.86
