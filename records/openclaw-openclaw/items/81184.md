## Summary

Keep open: the delivery-boundary dedupe addresses a plausible source-level duplicate Slack completion path and now has terminal proof, but the PR introduces an unbounded delivered-result cache in a long-lived process, with a smaller retry-hardening gap for synchronous throws.

## What This Changes

The branch adds a process-local `deliverSubagentAnnouncement` dedupe/coalesce helper keyed by `directIdempotencyKey`, wires it into delivery, and adds tests, logging, and a changelog entry.

## Best Possible Solution

Keep the dedupe at the `deliverSubagentAnnouncement` boundary, but make the success cache bounded or actively pruned and harden retry cleanup before merge.

## Reproduction Assessment

Yes for the delivery-boundary duplicate path: current main calls the dispatch function directly for the same stable key, and the PR proof shows the same boundary produces two gateway dispatches before the wrapper and one after. I did not establish a live Slack workspace reproduction, so the user-visible Slack duplicate remains inferred from dispatch count.

## Solution Assessment

No, not yet. The boundary is a maintainable place to dedupe, but the new delivered-result cache needs bounded expiry behavior and the helper should handle synchronous throws before this is the best mergeable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Prune expired announce dedupe entries:** `src/agents/subagent-announce-dedupe.ts:83`
  - body: Every successful unique announce key is inserted into `state.delivered`, but expired entries are only removed when that same key is looked up again. In a long-lived process, normal one-off subagent, cron, and media completions will accumulate after the 5-minute window; prune expired entries during insert or add a bounded sweep/size cap.
  - confidence: 0.9
- **[P3] Store inflight before running dispatch:** `src/agents/subagent-announce-dedupe.ts:79-90`
  - body: The promise starts running before it is stored in `state.inflight`. If `run()` throws synchronously, the `finally` deletes a missing key and the rejected promise is then stored forever, so later calls with the same key keep returning the stale rejection instead of retrying; store before execution or otherwise delete after synchronous failures and add a sync-throw test.
  - confidence: 0.82
