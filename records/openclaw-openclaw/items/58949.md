## Summary

Keep this PR open, but do not merge it as-is. Current main still has the invalid-input fast return, so the hardening target is real; however, the PR replaces the recently CodeQL-remediated padded byte comparison with fixed-key HMAC digest comparison, and GitHub Advanced Security is actively flagging that hash path.

## What This Changes

The PR changes `safeEqualSecret` to normalize null/undefined inputs to empty strings, compare fixed-key HMAC-SHA256 digests with `timingSafeEqual`, and add null/undefined-versus-empty regression cases.

## Best Possible Solution

Keep the padded timing-safe byte comparison as the equality primitive, add a reviewed invalid-input path that performs comparable dummy buffer work before returning false, and retain the new null/undefined-versus-empty regression coverage.

## Reproduction Assessment

Yes. Current main visibly returns at `src/security/secret-equal.ts:16` for non-string inputs, while valid strings allocate buffers and call `timingSafeEqual`, so the timing-shape concern has a high-confidence source-level reproduction path.

## Solution Assessment

No. The PR addresses a real hardening target, but fixed-key HMAC digest comparison is not the safest maintainable replacement for the CodeQL-remediated padded byte comparison; a dummy-work invalid-input path on the existing primitive is safer.
