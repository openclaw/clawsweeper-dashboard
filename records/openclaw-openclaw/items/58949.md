## Summary

Keep this PR open, but do not merge it as-is: the invalid-input timing concern is source-reproducible, while the current branch reintroduces a CodeQL-flagged HMAC/hash comparison path and still lacks real after-fix behavior proof.

## What This Changes

The PR replaces `safeEqualSecret`'s padded byte comparison with fixed-key HMAC digests for normalized null/undefined inputs and adds null/undefined/empty-string regression cases.

## Best Possible Solution

Keep the padded timing-safe byte comparison, add reviewed comparable work for invalid inputs before returning false, and retain the new null/undefined-versus-empty regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current main returns early for non-string inputs at `src/security/secret-equal.ts:16`, while valid string inputs run the padded `timingSafeEqual` path at `src/security/secret-equal.ts:25`. I did not run a timing benchmark in the read-only checkout.

## Solution Assessment

No. The regression tests are useful, but the production fix should harden the existing padded-buffer primitive instead of switching candidate credentials back to fixed-key HMAC digests.
