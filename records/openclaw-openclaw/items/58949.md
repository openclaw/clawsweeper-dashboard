## Summary

Keep open. The timing-shape concern is real, but this PR is not merge-ready because it replaces current main's CodeQL-remediated padded byte comparison with fixed-key HMAC digest comparison in a shared auth helper, and GitHub Advanced Security has already flagged that new hash path.

## What This Changes

The PR changes `safeEqualSecret` to normalize null/undefined inputs to empty strings, compare fixed-key HMAC-SHA256 digests with `timingSafeEqual`, and add null/undefined-versus-empty regression cases.

## Best Possible Solution

Keep the padded timing-safe byte comparison, add a reviewed invalid-input path that performs comparable dummy buffer work before returning false, and retain the new null/undefined-versus-empty regression coverage.

## Reproduction Assessment

Yes. Current main visibly returns early for non-string inputs at `src/security/secret-equal.ts:16`, while valid string inputs allocate buffers and call `timingSafeEqual`, so the reported timing-shape issue has a high-confidence source-level reproduction path.

## Solution Assessment

No. The PR preserves return semantics and adds useful tests, but fixed-key HMAC digest comparison is not the safest maintainable replacement for the CodeQL-remediated padded byte primitive.
