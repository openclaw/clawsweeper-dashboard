## Summary

Keep open: the reported invalid-input timing asymmetry is source-reproducible, but the current PR should not merge because it replaces the padded byte comparison with fixed-key HMAC digests that CodeQL flags. The added edge-case tests are useful and should be carried into a reviewed security-area fix.

## What This Changes

The PR changes `safeEqualSecret` to normalize null/undefined inputs through fixed-key HMAC digest comparison and adds null/undefined/empty-string regression cases.

## Best Possible Solution

Keep the padded timing-safe byte comparison, add reviewed comparable dummy work for invalid inputs, and retain the new null/undefined-versus-empty regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current main returns early for non-string inputs at `src/security/secret-equal.ts:16`, while valid string inputs run the padded timing-safe comparison path at `src/security/secret-equal.ts:25`. I did not run a timing benchmark in the read-only checkout.

## Solution Assessment

No. The regression tests are useful, but the production fix should harden the existing padded-buffer primitive instead of switching candidate credentials back to fixed-key HMAC digests.
