## Summary

Keep open: the invalid-input timing concern is source-reproducible, but the current patch is not safe to merge because it replaces the repo’s padded byte comparison with fixed-key HMAC hashing, which conflicts with the current CodeQL remediation direction and is failing CodeQL.

## What This Changes

The PR changes `safeEqualSecret` to normalize null/undefined to empty strings, compare fixed-key HMAC-SHA256 digests, and add null/undefined/empty-string edge-case tests.

## Best Possible Solution

Keep the padded timing-safe byte comparison, add reviewed comparable dummy work for invalid inputs, and retain the new null/undefined-versus-empty regression coverage.

## Reproduction Assessment

Yes. Source inspection of current main shows non-string inputs return at `src/security/secret-equal.ts:16` before buffer or `timingSafeEqual` work, while string inputs run the padded comparison path at `src/security/secret-equal.ts:19`.

## Solution Assessment

No. The regression tests are useful, but the production fix should harden the existing padded-buffer primitive instead of switching candidate credentials back to fixed-key HMAC digests.
