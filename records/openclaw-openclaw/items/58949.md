## Summary

Keep open for secops review: the PR addresses a real timing-shape gap, but its current implementation conflicts with current main's CodeQL remediation by replacing padded byte comparison with fixed-key HMAC/SHA-256.

## What This Changes

The PR changes `safeEqualSecret` to normalize null/undefined inputs, compare fixed-key HMAC-SHA256 digests with `timingSafeEqual`, and add null/undefined-versus-empty regression cases.

## Best Possible Solution

Keep the padded timing-safe byte comparison, add a reviewed invalid-input path with comparable dummy buffer work before returning false, and retain the new null/undefined-versus-empty regression coverage.

## Reproduction Assessment

Yes. On current main, `src/security/secret-equal.ts:16` returns before buffer allocation or `timingSafeEqual` for non-string inputs, while string inputs take the padded-buffer comparison path.

## Solution Assessment

No. The regression tests are useful, but the code fix should harden the existing padded-buffer primitive instead of reintroducing fixed-key HMAC comparison.
