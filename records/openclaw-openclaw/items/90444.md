## Summary

Keep open: current main still has subagent task finalization as a best-effort lifecycle side effect, while maintenance only reconciles subagent tasks from backing child-session presence or wedged recovery state, so the reported terminal-run/still-running-task state remains a valid core bug.

## What This Changes

The issue asks OpenClaw to reconcile `task_runs` rows and Task Flow state when a subagent run has already reached terminal state.

## Best Possible Solution

Add a bounded task-maintenance and lifecycle repair that treats terminal subagent run state as authoritative for matching parent subagent and child CLI task rows, normalizes monotonic times, and lets existing flow sync settle cancellation.

## Reproduction Assessment

No high-confidence live repro is available yet; the issue body explicitly says there is no tiny public repro script. Source inspection does show the reported state is plausible because task maintenance does not consult terminal subagent run state and cancel returns early when the subagent is already gone.

## Solution Assessment

Yes, the best fix is a narrow core repair rather than a new operator command: reconcile task rows from authoritative subagent terminal state and preserve existing flow-sync behavior. A manual SQLite workaround should remain unnecessary.
