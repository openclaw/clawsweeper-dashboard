## Summary

Keep open: current main now handles explicit `tools.exec.host=node` for Codex app-server sessions by disabling native Codex execution and keeping OpenClaw `exec`/`process` tools, but the reported `tools.exec.host=auto` plus bound node setup still resolves to gateway/native execution without a node route or clear diagnostic.

## What This Changes

Expose or clearly diagnose node-host exec in WebChat/OpenAI Codex sessions when a connected node and exec node configuration are present.

## Best Possible Solution

Keep the explicit `host=node` Codex dynamic-tool path, then decide and document whether `auto` with a bound node should expose node exec as a requestable tool or emit a clear diagnostic telling users to set `tools.exec.host=node` or a session-level node host.

## Reproduction Assessment

Yes at source level for the policy split, but no live Windows WebChat reproduction was run. Current main resolves `auto` without sandbox to gateway/native execution, while explicit node host disables native Codex execution and preserves OpenClaw `exec`/`process`.

## Solution Assessment

Unclear. Current main is the right shape for explicit `host=node`, but the `auto + bound node` behavior needs maintainer direction before broadening node execution or diagnostics.
