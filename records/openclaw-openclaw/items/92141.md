## Label Justifications

- `P1`: The issue blocks a real WebChat/Codex workflow for approved Windows node automation, but it is not a crash, data-loss, or security-bypass emergency.
- `impact:security`: The reported behavior sits on the shell execution host boundary between Codex native execution, gateway execution, sandbox execution, and node execution.
- `impact:session-state`: The active session reports and exposes direct execution even though the configured node binding is present, creating session tool-surface drift.

## Summary

Keep open: current main and v2026.6.5 still leave `tools.exec.host=auto` plus `tools.exec.node` on the Codex native/direct path because OpenClaw only re-adds node-aware dynamic `exec`/`process` when the effective host is already `node`. The merged explicit `host=node` fix does not cover this report's auto-plus-node WebChat/Codex path.

## What This Changes

Expose or clearly diagnose node-host exec for WebChat/OpenAI Codex sessions when a connected Windows node is configured.

## Best Possible Solution

Define the intended Codex/WebChat node-exec policy, then either require and surface explicit `host=node` with a clear diagnostic for `auto` plus node, or safely make that configuration expose OpenClaw dynamic exec without weakening native Codex or sandbox boundaries.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: with `tools.exec.host=auto` and `tools.exec.node` set, Codex policy resolves `auto` to gateway or sandbox and does not re-add OpenClaw dynamic `exec`/`process`. I did not run a live Windows node WebChat session.

## Solution Assessment

Unclear. The existing explicit `host=node` path is the right narrow implementation for that mode, but the remaining `auto` plus node behavior needs a maintainer decision between stricter diagnostics and changed routing semantics.
