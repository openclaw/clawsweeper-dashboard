## Summary

The central `host=auto` failure remains on current main: Codex keeps its native local execution surface because `auto` resolves to gateway, while OpenClaw only restores node-routed `exec` for an effective `host=node`. The corroborating comment also establishes a distinct mixed-session requirement—native local Codex tools plus an explicit remote node tool—that current main does not implement and that needs a deliberate security/product contract rather than closure as fixed.

## What This Changes

Expose policy-preserving remote-node shell execution in WebChat sessions using the OpenAI Codex runtime, including a usable mixed local-plus-remote session path.

## Best Possible Solution

Add a distinctly named Codex dynamic tool for remote node execution that internally pins `host: "node"`, optionally exposes only the allowed node selector shape, and delegates execution to the canonical OpenClaw exec implementation; keep Codex native local shell/files intact, and resolve general `auto` override semantics in the existing canonical issue.

## Reproduction Assessment

Yes at source level. With `tools.exec.host=auto`, current policy deterministically selects gateway/native Codex execution even when `tools.exec.node` is configured, and the builder restores OpenClaw node exec only for an effective `host=node`; two independent setups report the same observable result.

## Solution Assessment

No, making the whole Codex session node-bound is not the best complete solution because it removes the native local shell/files needed by the corroborated workflow. The narrow maintainable direction is a separate pinned remote-node tool over canonical OpenClaw exec, subject to maintainer approval of its model-visible security contract.

## Risks / Open Questions

- A mixed-session remote shell tool expands a security-sensitive model-visible surface; its schema must pin routing, preserve configured node binding, and reuse existing approval, audit, timeout, environment, and command-policy enforcement.
- Changing `host=auto` globally to prefer a configured node could silently move existing local commands off the gateway, so the mixed-session behavior should not be implemented as an implicit default change.
- The related `auto` per-call override contract is already tracked separately; combining that behavior change with the Codex mixed-session surface could blur ownership and upgrade semantics.
