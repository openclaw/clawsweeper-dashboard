## Summary

Keep open: current main appears to keep WebChat/Codex on the native local execution path when `tools.exec.host=auto`, even with a configured node, and only restores OpenClaw node shell tools when the effective host is `node`; that supports the reported gap but leaves the intended `auto` plus node policy for maintainers to decide.

## What This Changes

The issue requests that WebChat/OpenAI Codex sessions expose node-host exec, or give an actionable diagnostic, when a connected node is configured with `tools.exec.host=auto` and `tools.exec.node`.

## Best Possible Solution

Keep this open until maintainers choose the intended Codex behavior for `auto` plus node binding, then implement either node-host tool exposure or a clear diagnostic that points users to the explicit `tools.exec.host=node` route.

## Reproduction Assessment

Yes from source, but not from a live node run: current Codex policy and dynamic-tool code show the reported `auto` plus node-binding case does not expose node-host OpenClaw exec unless the effective host is `node`.

## Solution Assessment

Unclear until maintainers decide the intended `auto` semantics; the safer fix is either explicit node-host exposure for this configured case or an actionable diagnostic, rather than silently changing all Codex `auto` routing.
