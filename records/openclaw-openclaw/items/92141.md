## Summary

Keep open. Current main and the latest release handle explicit `host=node`, but the reported `host=auto` plus configured node case still resolves Codex native execution to direct gateway/native ownership, excludes OpenClaw `exec`/`process`, and only re-adds those tools when the effective host is already `node`.

## What This Changes

The issue asks WebChat/Codex sessions with `tools.exec.host=auto` and a configured node to expose usable node-host exec or an actionable diagnostic instead of silently leaving only direct local execution.

## Best Possible Solution

Codex WebChat should make the node-exec path explicit: keep the existing `host=node` routing through OpenClaw, and for `auto` plus a configured node either expose a safe per-call OpenClaw `exec host=node` route or show a clear diagnostic telling users to switch the session/config to `host=node`.

## Reproduction Assessment

Yes at source level, but not as a live Windows-node run. Current main resolves `auto` to gateway/direct for Codex native execution and filters OpenClaw `exec`/`process` unless effective host is already `node`, matching the reported missing tool path.

## Solution Assessment

Unclear until maintainers choose the product boundary. The fix belongs in the Codex app-server execution-policy/dynamic-tool surface, but maintainers should choose between a safe per-call node tool under `auto` and an explicit diagnostic that directs users to `host=node`.
