## Summary

Keep this open: current `main` intentionally supports either Codex-native local execution or a node-bound OpenClaw exec surface, but not the corroborated mixed local-plus-node workflow. The `host=auto` expectation overlaps an existing docs/runtime issue, while adding a separate remote-node tool requires an explicit product and security-contract decision.

## What This Changes

Add an approved remote-node execution surface to Codex-backed WebChat sessions that can coexist with Codex-native local shell and file tools.

## Best Possible Solution

Keep `auto` routing canonical, resolve its documented override contract in https://github.com/openclaw/openclaw/issues/61009, and—if maintainers approve mixed sessions—add a distinctly named node-only Codex dynamic tool that delegates entirely to canonical OpenClaw exec policy.

## Reproduction Assessment

Yes. Current-main source deterministically resolves `auto` to sandbox or gateway and disables native Codex tools for explicit node-bound sessions, leaving no mixed local-plus-node execution surface.

## Solution Assessment

No, silently redefining `auto` is not the best solution. A separately named, node-pinned dynamic tool that reuses canonical exec is the narrowest maintainable direction, but its permanent exposure and security contract require maintainer approval.
