## Summary

Current main already routes streaming chat markdown through a streaming-specific renderer, and that fix shipped in v2026.6.6 with focused unit and browser E2E coverage.

## What This Changes

The issue asks for Control UI streaming markdown to avoid replacing the full rendered message DOM on every streamed token.

## Best Possible Solution

Keep the shipped stable streaming markdown renderer and track only any newly reproduced, narrower rendering flicker as a fresh follow-up.

## Reproduction Assessment

Yes from source for the old behavior: the reported regular sanitizer/unsafeHTML streaming path existed, and current main now routes streaming text through toStreamingMarkdownHtml with tests for the unstable cases. I did not run a live browser repro in this read-only review.

## Solution Assessment

Yes. Current main's stable-block renderer is a better end state than a full plain-text fallback because it preserves completed markdown formatting while keeping the in-progress tail escaped until it is stable.
