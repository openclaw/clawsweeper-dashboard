## Summary

Keep open. Current main still does not parse an empty `MEDIA:` directive with the path on the next line, and this PR is a narrow implementation with no blocking functional or security finding from the read-only review. The remaining mechanical issue is the required user-facing changelog entry, while the related trailing-JSON parser hardening remains separate in #75182.

## What This Changes

The PR adds continuation parsing for empty `MEDIA:` lines so the immediate next line can provide a media source, plus parser tests for ordered segments and single-line consumption.

## Best Possible Solution

Merge a current-main-compatible version of the narrow parser enhancement that consumes only one validated continuation line, keeps existing media validation and fenced-block behavior, adds the required Unreleased changelog entry, and leaves #75182 as a separate hardening item unless maintainers intentionally combine the parser work.

## Reproduction Assessment

Yes. A focused parser case like `splitMediaFromOutput("Before\nMEDIA:\n/tmp/screenshot.png\nAfter")` reproduces the gap: current main keeps the bare `MEDIA:` line because no regex match exists, while the PR adds coverage for extracting `/tmp/screenshot.png`.

## Solution Assessment

Yes, with one mechanical follow-up. The PR’s final parser approach is narrow, uses the existing validation path, and avoids the earlier greedy-continuation bug by consuming only the immediate next line; it still needs the required changelog entry and maintainer review against current parser hardening.
