Full review comments:

- **[P2] Keep Usage hour labels tied to the selected time basis:** `ui/src/ui/views/usage-metrics.ts:31`
  - body: Usage still buckets and filters hours with its own `local`/`utc` selector, but this label now formats the synthetic hour through the global Control UI timezone. With the Usage filter set to UTC and `gateway.controlUi.timezone: "America/New_York"`, a bucket stored and selected as hour 10 is displayed around 6 AM, so the labels no longer match the filter or hour cells. Keep these labels tied to the Usage filter, or update the bucket math to the same configured zone.
  - confidence: 0.88
- **[P2] Avoid applying timestamp timezone to date-only Usage keys:** `ui/src/ui/views/usage-metrics.ts:465-473`
  - body: The daily aggregate values here are `YYYY-MM-DD` calendar keys, but `parseYmdDate` constructs UTC midnight and the new wrapper applies the configured timestamp timezone. For `gateway.controlUi.timezone: "America/New_York"`, `2026-06-05T00:00Z` renders as Jun 4, so daily charts and insights can show the previous day. Format date-only keys without the configured timestamp timezone.
  - confidence: 0.9
- **[P2] Document the new Control UI timezone config:** `src/config/zod-schema.ts:1018-1035`
  - body: This line adds a new public `gateway.controlUi.timezone` config key, but the public configuration reference still has no entry for its allowed values, `host` behavior, or default. The repo config contract expects schema/help/baseline/docs to stay aligned, so operators reading the docs will not know this setting exists or how it behaves.
  - confidence: 0.84
