## Summary

Keep open: the payload-level fix is the right direction, but the latest head still leaves provider-owned embedded streams outside the empty-tools guard and it lacks the required user-facing changelog entry.

## What This Changes

The PR adds an embedded-runner payload wrapper that removes empty `tools` arrays and orphaned `tool_choice` fields for session fallback/custom stream paths, plus a stream-resolution test update.

## Best Possible Solution

Land one egress sanitizer that composes across provider-owned, boundary-aware, session HTTP fallback, and custom embedded streams, with focused regression tests and a changelog entry.

## Reproduction Assessment

Yes, at source level. Current main can build `tools: []` for OpenAI-compatible payloads after tool filtering/history, and Active Memory uses the embedded runner with a restricted `toolsAllow` list; I did not run a live strict-provider call.

## Solution Assessment

No, not as currently proposed. A payload-level guard is a maintainable fix, but it must also wrap provider-owned streams and include the required changelog coverage before merge.
