## Summary

Keep open. The PR addresses a real empty-tools provider failure mode, but the current branch still leaves a reachable provider-owned embedded stream path unguarded and lacks inspectable after-fix real behavior proof.

## What This Changes

The PR adds an embedded-runner stream wrapper intended to strip empty `tools` arrays and orphaned `tool_choice` fields from outgoing provider payloads, plus updates a stream-resolution test expectation.

## Best Possible Solution

Land a rebased payload sanitizer that covers provider-owned, PI-native, boundary-aware, and custom embedded stream paths, with focused payload assertions and redacted live proof from the contributor's strict-provider setup.

## Reproduction Assessment

Yes, at source level: current main can build `params.tools = []` for tool-history calls with no current tools, and Active Memory reaches the embedded runner with a restricted `toolsAllow`. I did not establish the live strict-provider HTTP 400 in this read-only review.

## Solution Assessment

No, not yet. A payload-level guard is the right shape, but this PR applies it after only some stream-resolution branches and leaves provider-owned streams unguarded; it also needs inspectable after-fix proof.
