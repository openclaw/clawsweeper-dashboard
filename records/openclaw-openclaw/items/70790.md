## Summary

Keep this PR open, but do not merge it as-is. The reported empty-tools provider failure is still a valid current-main problem and the PR is directionally useful, but the head patch only wraps fallback/custom stream paths and still leaves provider-owned embedded streams uncovered; it also needs to be composed with the current credential/signal wrapper and needs the required changelog entry.

## What This Changes

The PR adds an embedded-runner stream wrapper that deletes empty `tools` arrays and orphaned `tool_choice` fields from outgoing provider payloads, plus a test expectation update for custom stream wrapping.

## Best Possible Solution

Land a narrow current-main sanitizer at the embedded-runner egress boundary that composes with `wrapEmbeddedAgentStreamFn`, covers provider-owned, boundary-aware, session HTTP fallback, and custom streams, preserves non-empty tools and caller payload hooks, and adds focused regression tests plus a changelog entry.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: Active Memory calls the embedded runner with a restricted `toolsAllow`, current main can build OpenAI-compatible payloads with `tools: []`, and strict OpenAI-compatible providers reject that payload shape.

## Solution Assessment

No, not as currently proposed. A payload-level guard is the right direction, but this patch must be rebased onto the current stream wrapper, cover provider-owned streams, and include the required changelog/test coverage before it is the narrow maintainable fix.
