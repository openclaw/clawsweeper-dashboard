## Summary

Keep open. Current main still lacks an embedded-runner empty-tools payload guard, and the PR is directionally useful for the strict-provider failures discussed by the author, Greptile, and Joey Krug, but the branch still leaves provider-owned embedded streams outside the sanitizer and is missing the required changelog entry.

## What This Changes

The PR adds a stream-resolution wrapper that deletes empty `tools` arrays and orphaned `tool_choice` from embedded runner payloads, then updates a resolver test to expect custom stream functions to be wrapped.

## Best Possible Solution

Land a narrow embedded-runner egress sanitizer on current main that composes with the existing `wrapEmbeddedAgentStreamFn` path, covers provider-owned, boundary-aware, session HTTP fallback, and custom streams, preserves non-empty tools and caller `onPayload` behavior, and adds focused regression tests plus a changelog entry.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: Active Memory calls `runEmbeddedPiAgent` with `toolsAllow`, current main can build OpenAI-compatible payloads with `tools: []`, and strict providers reject that shape before the embedded runner has any current sanitizer.

## Solution Assessment

No, not as currently proposed. A payload-level egress guard is the right direction, but the PR must also wrap provider-owned streams and be rebased onto the current credential/signal wrapper instead of adding a parallel partial composition.
