## Summary

Keep open: the PR is aimed at a real embedded-runner provider compatibility bug, but the latest head is still incomplete because provider-owned stream functions can bypass the new empty-tools sanitizer and the user-facing fix lacks a changelog entry.

## What This Changes

The PR adds an embedded Pi runner stream wrapper that deletes empty `tools` arrays and orphaned `tool_choice` fields from outgoing payloads, plus a test expectation update for custom stream wrapping.

## Best Possible Solution

Land a narrow embedded-runner egress sanitizer that composes across provider-owned, session HTTP fallback, boundary-aware, and custom streams, preserves existing auth/signal/cache-boundary behavior, and includes focused regression tests plus a changelog entry.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: the OpenAI-compatible transport can build `tools: []` after tool-history replay when no current tools remain, and Active Memory calls the embedded runner with a restricted `toolsAllow` list that reaches this stream-resolution path.

## Solution Assessment

No, not as currently proposed. A payload-level guard is the right maintainable direction, but the PR must wrap provider-owned streams too and add the required release-note surface before merge.
