## Summary

Keep open: the strict-provider empty-tools failure is source-reproducible and the payload guard is a reasonable direction, but the latest PR head still bypasses provider-owned embedded streams and lacks inspectable real behavior proof.

## What This Changes

The branch adds an embedded-runner stream wrapper that deletes empty `tools` arrays and orphaned `tool_choice` fields from payloads, and updates a stream-resolution test expectation for wrapped custom streams.

## Best Possible Solution

Land one final embedded-stream payload sanitizer that covers provider-owned, boundary-aware, session fallback, and custom streams without changing the tool filtering policy.

## Reproduction Assessment

Yes, at source level: current main can build `tools: []` for OpenAI-compatible payloads after tool history, and Active Memory reaches the embedded runner with a restricted `toolsAllow`. I did not run a live strict-provider call.

## Solution Assessment

No, not yet. A payload-level guard is the right shape, but this branch must also cover provider-owned streams and should include focused payload assertions for that path.
