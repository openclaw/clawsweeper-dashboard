## Summary

Keep this PR open, but the current branch is not ready as-is. Current main still lacks an embedded-runner egress guard for empty `tools` arrays and orphaned `tool_choice`, so the problem remains real; however the proposed wrapper misses the provider-owned stream path and needs to be rebased/composed with the newer boundary-aware OAuth/signal wrapper on main.

## What This Changes

The PR adds `wrapStreamFnStripEmptyTools` in `src/agents/pi-embedded-runner/stream-resolution.ts` to remove `tools: []` and orphaned `tool_choice` from embedded stream payloads, and updates a resolver test expectation.

## Best Possible Solution

Land a narrow embedded-runner egress sanitizer on current main that composes with `wrapEmbeddedAgentStreamFn`, covers provider-owned, boundary-aware, session HTTP fallback, and custom stream paths, preserves non-empty tools and existing `onPayload` callbacks, and adds focused regression tests in the stream-resolution/embedded-runner surface. Do not redo the already-present `toolsAllow` forwarding work.

## Reproduction Assessment

Yes. The source-level path is clear: Active Memory invokes `runEmbeddedPiAgent` with `toolsAllow`, the attempt layer can produce no current callable tools, OpenAI-compatible completions can still serialize `tools: []`, and current stream resolution has no guard before the provider request leaves the embedded runner.

## Solution Assessment

No, not as currently proposed. A payload-level egress guard is the right direction, but the exact PR needs to cover provider-owned streams and be rebased so it composes with the current credential/signal wrapper on main.
