## Summary

Keep open: current main and v2026.6.9 still lack null-content normalization at the OpenAI Responses send boundaries, and the only direct sanitizer PR closed unmerged over live strict-provider proof scope.

## What This Changes

The issue asks OpenClaw to prevent `api: "openai-responses"` requests from sending `input` message items with `content: null` to strict OpenAI-compatible providers.

## Best Possible Solution

Accept or reject the narrow Responses send-boundary sanitizer with an explicit maintainer proof-scope decision; if accepted, cover shared OpenAI, managed OpenAI, and Azure Responses send paths with regression proof.

## Reproduction Assessment

Partially: yes for a source-level send-boundary check, but not for the full live weelinking flow on current main. Current code can pass post-hook Responses payloads to the SDK without null-content normalization, while the exact reporter session-history shape is still unconfirmed.

## Solution Assessment

Unclear until maintainers decide proof scope. The serializer-level sanitizer appears to be the narrow repair shape, but the previous branch closed because strict-mock and lifecycle proof did not satisfy the live-provider gate.
