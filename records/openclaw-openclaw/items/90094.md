## Summary

Keep open: current main still has OpenAI Responses send boundaries without null-content normalization, the targeted sanitizer PR closed unmerged, and the merged related work fixed storeless item-id replay rather than this null-content schema failure.

## What This Changes

The issue asks the OpenAI Responses transport to normalize or drop `input` messages with `content: null` before sending strict OpenAI-compatible provider requests.

## Best Possible Solution

Normalize invalid Responses message content at both shared and managed send boundaries, with regression coverage and maintainer-accepted strict-provider proof scope.

## Reproduction Assessment

Yes for the source boundary, but not for the full live weelinking flow. Current main can pass post-`onPayload` Responses input directly to the SDK without null-content normalization, while the exact reporter session shape still needs live confirmation.

## Solution Assessment

Yes, the narrow serializer-level normalization proposed by the closed PR is the right repair shape. Provider-specific fallbacks would be broader than needed, but maintainers should settle proof scope before another branch is queued.
