## Summary

Keep open: the performance bug and proposed direction are valid, and the supplied real behavior proof is sufficient, but PR head still caches setup-manifest CLI-provider answers above the resolver's plugin metadata/workspace/env freshness boundary, so it can return stale status after same-process control-plane changes.

## What This Changes

The PR memoizes the setup-manifest branch of `isCliProvider`, adds regression coverage, and updates the changelog for `openclaw status` / `openclaw doctor` session-summary scaling.

## Best Possible Solution

Keep the O(unique providers) speedup, but make the cache honor the setup resolver's control-plane freshness boundary or scope it to a single status-summary scan.

## Reproduction Assessment

Yes. Current main source maps every status session row through `resolveSessionRuntimeLabel` to `isCliProvider` and then to setup CLI backend discovery; the PR body also supplies live ECS output and local timing for the before/after behavior.

## Solution Assessment

No, not yet. Memoizing only the setup branch is the right performance direction, but the cache needs resolver freshness or scan scope before this is the narrow maintainable fix.
