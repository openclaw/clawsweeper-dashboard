## Summary

Keep open: the performance problem and proof are credible, but the current patch still caches setup-manifest results only by config/provider even though that resolver is sensitive to plugin metadata, workspace, and env freshness.

## What This Changes

The PR memoizes setup-manifest `isCliProvider` lookups for status/doctor session rows, adds regression coverage, and adds a changelog entry.

## Best Possible Solution

Preserve the speedup, but scope the cache to the plugin control-plane freshness boundary or precompute CLI-provider facts for a single status summary scan.

## Reproduction Assessment

Yes, source-reproducible: current main resolves a runtime label for each session row and can re-enter setup CLI backend discovery through `isCliProvider`. The PR body also supplies before/after timings and live output, but I did not rerun them in this read-only pass.

## Solution Assessment

No. The narrowed cache fixes the active runtime registry stale-result issue, but the setup branch is still cached across metadata/workspace/env freshness changes that the resolver contract currently observes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Key setup cache by metadata freshness:** `src/agents/model-selection-cli.ts:47-49`
  - body: The setup cache returns a saved `(cfg, provider)` result before calling `resolvePluginSetupCliBackendRuntime` again. That resolver is sensitive to current plugin metadata snapshots plus workspace/env context, and existing tests expect CLI backend lookup results to change after `setCurrentPluginMetadataSnapshot` changes. A provider checked before a plugin inventory reload can stay cached false or true until the config object changes, so key this cache to the plugin control-plane/snapshot boundary or scope it to one status-summary run.
  - confidence: 0.88
