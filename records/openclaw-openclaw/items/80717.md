## Summary

Keep open: the reported status/doctor scaling problem is plausible and the PR targets the right hot path, but the patch caches the whole `isCliProvider` result by config/provider even though current main also derives that answer from active plugin runtime state. That can leave later callers with stale CLI-provider classification after plugin state changes.

## What This Changes

This PR adds a WeakMap-backed cache in `isCliProvider`, adds a repeated-lookup regression test, and adds a changelog entry for status/doctor performance.

## Best Possible Solution

Keep the performance win, but either precompute CLI-provider facts inside the status summary flow or memoize only the stable setup-lookup branch with a cache key/clear path that accounts for plugin runtime, workspace, and metadata state.

## Reproduction Assessment

Yes, source-reproducible: current main resolves a runtime label for every session row and can reach setup CLI backend discovery through `isCliProvider`. The PR body also provides a synthetic sessions benchmark, but I did not run it in this read-only pass.

## Solution Assessment

No. The proposed cache is too broad because `isCliProvider` also depends on active runtime plugin state and setup lookup context, so the maintainable fix should narrow or move the cache rather than memoize the whole function by config/provider only.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep runtime-registry results out of the cache:** `src/agents/model-selection-cli.ts:48-53`
  - body: `isCliProvider` currently checks `resolveRuntimeCliBackends()`, which reads the active plugin runtime registry, before falling back to setup lookup. Caching the entire result by `(config, provider)` means a call made before a runtime CLI backend is active can cache `false`, and later execution/session callers using the same config will still treat that backend as non-CLI after the registry is loaded. Please keep the active-runtime branch fresh, or include/clear the cache on the registry state, and only memoize inputs that are actually stable.
  - confidence: 0.9
