## Summary

Keep open: the performance bug and after-fix proof are credible, but the PR still caches setup-manifest CLI-provider answers across metadata/workspace/env freshness boundaries, so it needs a narrow repair before merge.

## What This Changes

The PR memoizes setup-manifest CLI provider detection in `isCliProvider`, adds regression coverage, and updates the changelog for `openclaw status` / `openclaw doctor` scaling.

## Best Possible Solution

Preserve O(unique-providers) status/doctor behavior by either scoping the cache to one status-summary scan or keying it to plugin control-plane freshness, with a regression covering metadata snapshot changes.

## Reproduction Assessment

Yes, source-reproducible: current main builds one runtime label per status row and can enter setup CLI backend discovery through `isCliProvider`. The PR body also supplies live ECS output and local timing output showing the before/after performance behavior.

## Solution Assessment

No. Memoizing the setup branch is the right performance direction, but this implementation needs metadata/workspace/env freshness in its cache boundary or a scan-scoped cache to avoid stale setup-manifest answers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Key setup cache to metadata freshness:** `src/agents/model-selection-cli.ts:47-49`
  - body: The cached setup-manifest answer is returned for the same config/provider before `resolvePluginSetupCliBackendRuntime` can re-check the current plugin metadata snapshot, workspace, or env. Include the control-plane freshness in the cache key or scope this memo to one status-summary scan.
