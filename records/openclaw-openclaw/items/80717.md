## Summary

Keep open: this PR targets a real status/doctor performance bug and includes sufficient live proof, but the current head still caches setup-manifest CLI backend results across the plugin metadata freshness boundary, which can leave same-config lookups stale after a snapshot refresh.

## What This Changes

The PR memoizes the setup-manifest branch of `isCliProvider`, adds regression tests, and records the status/doctor performance fix in `CHANGELOG.md`.

## Best Possible Solution

Preserve the speedup, but key the setup cache to the plugin control-plane freshness boundary or precompute CLI-provider facts for a single status summary scan.

## Reproduction Assessment

Yes. Current main resolves a runtime label for each status session row and can enter setup CLI backend discovery through `isCliProvider`; the PR body also supplies before/after timings and live status output.

## Solution Assessment

No. Caching only the setup branch is closer than the first version, but the cache still omits metadata/workspace/env freshness that the resolver contract currently observes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Key setup cache by metadata freshness:** `src/agents/model-selection-cli.ts:47-49`
  - body: `isMemoizedSetupCliBackend` returns a cached setup-manifest answer before `resolvePluginSetupCliBackendRuntime` can re-read the current metadata snapshot. That resolver intentionally observes current snapshot, env, and active workspace changes, and existing tests expect results to flip after `setCurrentPluginMetadataSnapshot`; with this cache, a same-config lookup can keep reporting the old CLI/provider label until the config object changes or the process restarts. Include the plugin control-plane freshness in the cache key or scope this cache to one status-summary scan.
  - confidence: 0.88
