## Summary

Keep open: the latest head fixes the original whole-function cache problem, but it still memoizes setup-manifest answers without plugin metadata, workspace, or env freshness, so same-process setup backend changes that current main supports can be hidden.

## What This Changes

The PR memoizes setup-manifest CLI provider detection in `isCliProvider`, adds regression coverage, and documents the `openclaw status` / `openclaw doctor` scaling improvement in the changelog.

## Best Possible Solution

Keep the scaling improvement, but scope the setup lookup cache to one status/doctor scan or key it by plugin control-plane freshness, workspace, and env inputs.

## Reproduction Assessment

Yes. Current main shows the status summary resolving a runtime label for every session row, and that label path calls `isCliProvider` before setup CLI backend resolution; the PR body adds live ECS and local timing proof for the performance symptom.

## Solution Assessment

No. Caching the setup branch is the right performance direction, but the proposed cache key is broader than the resolver contract and should include freshness or be scan-scoped.
