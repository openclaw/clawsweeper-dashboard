## Summary

Keep open: this PR fixes a real `openclaw status` / `openclaw doctor` scaling path and supplies sufficient live proof, but the current cache boundary can still return stale setup-manifest answers after same-process plugin metadata, workspace, or env changes.

## What This Changes

The PR memoizes setup-manifest CLI backend detection in `isCliProvider`, adds regression tests, and adds a changelog entry for `openclaw status` / `openclaw doctor` scaling.

## Best Possible Solution

Keep the performance improvement, but make the setup lookup cache scan-scoped or keyed by plugin control-plane freshness so metadata/workspace/env updates remain visible.

## Reproduction Assessment

Yes. Current main is source-reproducible because the status summary resolves one runtime label per session row and can enter setup CLI backend discovery through `isCliProvider`; the PR body also supplies live ECS and local timing proof.

## Solution Assessment

No. Caching the setup branch is the right performance direction, but the cache must include plugin control-plane freshness or be scoped to one status-summary scan to avoid stale setup-manifest answers.
