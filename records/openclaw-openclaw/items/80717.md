## Summary

Keep this PR open: the performance problem is real and the branch is useful, but the current patch still caches setup-manifest CLI backend results across plugin metadata/workspace freshness boundaries that the resolver treats as live.

## What This Changes

The PR memoizes the setup-manifest branch of `isCliProvider`, adds focused model-selection regression tests, and updates the changelog for `openclaw status` / `openclaw doctor` session-summary performance.

## Best Possible Solution

Keep the speedup, but key the setup lookup cache to plugin control-plane freshness or precompute CLI-provider facts inside the status summary flow where the metadata snapshot is stable.

## Reproduction Assessment

Yes, source-reproducible: current main resolves a runtime label for each status session row and can re-enter setup CLI backend discovery through `isCliProvider`. I did not rerun the synthetic benchmark, but the PR body and follow-up include copied before/after timing output.

## Solution Assessment

No. Memoizing only by config reference and provider is still broader than `resolvePluginSetupCliBackendRuntime`'s metadata/workspace/env freshness contract; the cache needs that boundary or a narrower per-summary scope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Key the setup cache by metadata freshness:** `src/agents/model-selection-cli.ts:47-49`
  - body: `isMemoizedSetupCliBackend` returns a cached value for the same config/provider before calling `resolvePluginSetupCliBackendRuntime` again. That resolver intentionally changes when the current plugin metadata snapshot, workspace, or env-derived discovery context changes, and existing setup-registry tests cover same-process snapshot refreshes. A provider checked before a plugin inventory reload can stay stale until the config object changes, so include the plugin control-plane freshness boundary in the cache key or scope the cache to one status-summary pass.
  - confidence: 0.88
