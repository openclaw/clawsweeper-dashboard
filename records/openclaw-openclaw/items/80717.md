## Summary

Keep open: the PR addresses a real status/doctor scaling bug and the author narrowed the cache after review feedback, but the setup-manifest cache still bypasses same-process plugin metadata, workspace, or env freshness for the same config/provider key.

## What This Changes

This PR memoizes setup-manifest CLI provider detection in `isCliProvider`, adds regression coverage, and updates the changelog for `openclaw status` / `openclaw doctor` scaling.

## Best Possible Solution

Preserve O(unique-providers) status/doctor behavior by scoping the memo to one status-summary scan or keying it to the existing plugin control-plane freshness inputs, with a regression for metadata snapshot changes.

## Reproduction Assessment

Yes. Current main resolves one runtime label per status row and each label can call `isCliProvider` into setup CLI backend discovery; the PR body also supplies live ECS output and local timing output for the before/after performance behavior.

## Solution Assessment

No. Memoizing only the setup branch is the right performance direction, but the cache boundary still needs metadata/workspace/env freshness or scan-local lifetime to avoid stale setup-manifest answers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Key setup cache to metadata freshness:** `src/agents/model-selection-cli.ts:47-49`

Include the control-plane freshness in the cache key or scope this memo to one status-summary scan.
