## Summary

Keep open: the reported status/doctor hot path is valid, but this PR still caches setup lookup results across plugin metadata/workspace/env freshness boundaries, so it needs one narrow repair before merge.

## What This Changes

The PR memoizes setup-manifest CLI-provider lookups in `isCliProvider`, adds regression coverage for config/runtime rechecks, and adds a changelog entry.

## Best Possible Solution

Preserve the performance win, but key or scope the setup cache to the plugin control-plane freshness boundary, or precompute CLI-provider facts for a single status summary scan.

## Reproduction Assessment

Yes. Current main maps every status session row through `resolveSessionRuntimeLabel` and `isCliProvider`, which can fall through to setup CLI backend discovery; the PR body also supplies live ECS output and local before/after timings.

## Solution Assessment

No. The proposed cache is close, but caching setup results only by config/provider bypasses resolver freshness for metadata snapshot, workspace, and env changes.
