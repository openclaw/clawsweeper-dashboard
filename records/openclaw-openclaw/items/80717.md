## Summary

Keep open: this PR targets a real, source-backed `openclaw status` / `openclaw doctor` scaling bug and supplies convincing live-output proof, but the current patch still introduces a process-wide setup-manifest cache that can bypass plugin metadata/workspace/env freshness for the same config/provider pair. Current main does not already implement this narrow fix, and the related default-status thread is still open rather than merged.

## What This Changes

The PR memoizes setup-manifest CLI provider detection in `isCliProvider`, adds regression coverage for config/runtime-registry freshness, and adds a changelog entry for `openclaw status` / `openclaw doctor` scaling.

## Best Possible Solution

Keep the O(unique-providers) status/doctor improvement, but either scope the cache to one status-summary scan or include plugin control-plane freshness in the setup cache boundary with regression coverage for metadata snapshot changes.

## Reproduction Assessment

Yes. Current main builds one runtime label per status session row and can enter setup CLI backend discovery through `isCliProvider`; the PR body also supplies live ECS output and local timing output showing before/after behavior.

## Solution Assessment

No. Memoizing only the setup branch is the right performance direction, but a process-wide `(config, provider)` cache is not the safest boundary for freshness-sensitive setup metadata; scan-scoped or freshness-keyed caching is the narrower maintainable fix.
