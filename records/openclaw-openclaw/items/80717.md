## Summary

Keep open: the PR addresses a real status/doctor scaling bug and has sufficient live proof, but the current patch still caches setup-manifest CLI backend results across plugin metadata/workspace/env freshness boundaries that the resolver contract observes.

## What This Changes

The PR memoizes the setup-manifest branch of `isCliProvider`, adds regression tests, and adds a changelog entry for faster status/doctor session summaries.

## Best Possible Solution

Preserve the speedup, but key the setup cache to the plugin control-plane freshness boundary or scope it to a single status-summary scan, with regression coverage for metadata snapshot changes.

## Reproduction Assessment

Yes, source-reproducible: current main resolves a runtime label for each session row and can re-enter setup CLI backend discovery through `isCliProvider`. The PR body also supplies before/after timings and live output, but I did not rerun them in this read-only pass.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Key setup cache by metadata freshness:** `src/agents/model-selection-cli.ts:47-49`
  - body: The setup cache returns a saved `(config, provider)` result before calling `resolvePluginSetupCliBackendRuntime` again. That resolver is sensitive to plugin metadata snapshots plus workspace/env context, and existing tests expect CLI backend lookup results to change after the current metadata snapshot changes, so a provider checked before a plugin inventory reload can stay cached false or true until the config object changes. Key this cache to the plugin control-plane/snapshot boundary or scope it to one status-summary run.
