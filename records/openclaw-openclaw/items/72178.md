## What This Changes

## Best Possible Solution

Revise this branch or replace it with a focused implementation that windows raw history before projection/capping, regenerates protocol artifacts, reconciles the overlapping `NO_REPLY` work, and includes redacted real WebChat/Gateway proof.

## Reproduction Assessment

Yes. Source inspection of PR head gives a high-confidence path: use a transcript longer than the page limit, request the latest `chat.history` page, then request `before` with the returned cursor; the handler still projects `rawMessages`, so it can return the newest page again.

## Solution Assessment

No. Cursor pagination is a reasonable additive direction, but this implementation should apply the cursor window before display projection/capping and keep generated protocol clients synchronized.
