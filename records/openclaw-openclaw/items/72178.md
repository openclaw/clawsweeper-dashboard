## Summary

Keep this PR open, but it is not merge-ready: the cursor pagination still returns the newest projected page for older-page requests, generated protocol artifacts are out of sync, and the external PR still lacks real after-fix behavior proof.

## What This Changes

The PR adds `before`/`cursor`/`hasMore` support to Gateway `chat.history`, WebChat load-older state and rendering, partial `NO_REPLY` suppression, docs, and tests.

## Best Possible Solution

Revise this branch or open a focused replacement PR that windows raw history before projection/capping, regenerates protocol artifacts, reconciles the `NO_REPLY` scope with https://github.com/openclaw/openclaw/pull/39350, and includes redacted real WebChat/Gateway proof.

## Reproduction Assessment

Yes. Source inspection of PR head gives a high-confidence path: load a transcript longer than the page limit, request the latest `chat.history` page, then request `before` with the returned cursor; the handler still projects `rawMessages` and can return the newest page again.

## Solution Assessment

No. Cursor pagination is a reasonable additive direction, but this implementation should apply the cursor window before display projection/capping and keep generated protocol clients synchronized.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Apply pagination before projecting history:** `src/gateway/server-methods/chat.ts:1527-1530`

That helper keeps the newest `maxMessages`, so an older-page request can return the latest page again and make WebChat prepend duplicates repeatedly; slice or otherwise window the raw transcript before projection/capping.
  - confidence: 0.95

  - body: Adding `before` changes the public Gateway schema, but the generated Swift models and protocol schema outputs are not included. Typed clients remain out of sync, and `protocol:check` is expected to fail until those generated artifacts are refreshed.
  - confidence: 0.92
