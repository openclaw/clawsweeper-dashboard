## Summary

Keep open: current main still lacks `chat.history` cursor pagination, and this PR is useful source work, but the current head is not merge-ready because the server still ignores the computed `before` window during projection, generated protocol clients are stale, and the required changelog entry is missing.

## What This Changes

The PR adds a `before` cursor to `chat.history`, returns cursor metadata, adds load-older WebChat UI state, suppresses partial `NO_REPLY` stream fallback commits, updates protocol docs/tests, and includes a small unrelated plugin test formatting change.

## Best Possible Solution

Repair the PR at the Gateway handler boundary so `before` pages operate on the requested raw transcript window or carry stable raw indices through projection/capping, then regenerate protocol artifacts, add the active-version changelog entry, and keep the UI/NO_REPLY coverage focused.

## Reproduction Assessment

Yes. On the PR head, a transcript with more than one page can request the latest `chat.history` page and then request `before=cursor`; source inspection shows the handler computes `start/end` but still projects the full `rawMessages` array, whose limiter keeps the newest messages.

## Solution Assessment

No. Cursor pagination is a reasonable additive protocol direction, but the current implementation is not the best fix until the server applies the cursor window before projection/capping and generated protocol clients stay in sync.
