## Summary

Keep open: current main still lacks `chat.history` cursor pagination, and the PR head has blocking correctness/protocol issues plus no real behavior proof.

## What This Changes

The branch adds `before`/`cursor`/`hasMore` pagination for `chat.history`, WebChat load-older UI/state/scroll handling, partial `NO_REPLY` suppression, docs, and tests.

## Best Possible Solution

Land a focused additive pagination implementation with stable window/cursor semantics, synchronized schema/docs/generated clients, regression coverage, and real WebChat/Gateway proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence PR-head path: with a transcript longer than the page limit, request latest `chat.history`, then request `before` with the returned cursor; the handler still projects the full transcript and can return the newest page again.

## Solution Assessment

No. Cursor pagination is a reasonable additive direction, but this implementation should apply pagination before projection/capping or align with the existing session-history cursor helper, then regenerate protocol artifacts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Apply the cursor window before projecting history:** `src/gateway/server-methods/chat.ts:1527-1530`
  - body: The handler computes `start` and `end` from `before`, but still calls `projectRecentChatDisplayMessages(rawMessages, ...)` on the full transcript. Since that helper keeps the newest `maxMessages`, an older-page request can return the latest page again and make the UI prepend duplicates. Project/sanitize the requested window, or reuse the existing session-history pagination flow, before capping.
  - confidence: 0.94
- **[P2] Regenerate protocol artifacts:** `src/gateway/protocol/schema/logs-chat.ts:31`
  - body: Adding `before` changes the public Gateway schema, but the generated protocol outputs are not included. `protocol:check` covers `dist/protocol.schema.json` and Swift models, and the generated `ChatHistoryParams` still lacks `before`, leaving typed clients out of sync.
  - confidence: 0.9
