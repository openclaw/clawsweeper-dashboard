## Summary

Keep open: current main does not implement the pagination API, but the latest PR head still has blocking cursor/window behavior, generated protocol drift, and missing real behavior proof.

## What This Changes

The PR adds cursor-based `chat.history` pagination with WebChat load-older UI/state plus partial `NO_REPLY` suppression, docs, and tests.

## Best Possible Solution

Revise this branch or a focused replacement to apply pagination before projection/capping, regenerate protocol clients, reconcile the `NO_REPLY` scope, and include redacted real WebChat/Gateway proof.

## Reproduction Assessment

Yes. Source inspection of PR head gives a high-confidence path: with a transcript longer than the page limit, request the latest `chat.history` page, then request `before` with the returned cursor; the handler still projects the full transcript and can return the newest page again.

## Solution Assessment

No. Cursor pagination is a reasonable additive direction, but this implementation should apply the cursor window before projection and keep generated protocol artifacts synchronized.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Apply the cursor window before projecting history:** `src/gateway/server-methods/chat.ts:1527-1530`
  - body: The handler computes `end`/`start` from `before`, but still passes `rawMessages` into `projectRecentChatDisplayMessages`. That helper keeps the newest `maxMessages`, so an older-page request can return the latest page again and make WebChat prepend duplicates; slice/window the raw transcript before projection or reuse the session-history pagination flow.
  - confidence: 0.94
- **[P2] Regenerate generated protocol models:** `src/gateway/protocol/schema/logs-chat.ts:31`
  - body: Adding `before` changes the public Gateway schema, but the generated Swift `ChatHistoryParams` model still lacks that field and the PR file list omits generated protocol outputs. Typed clients remain out of sync and `protocol:check` should produce a diff until those artifacts are regenerated.
  - confidence: 0.9
- **[P2] Preserve legitimate uppercase NO replies:** `ui/src/ui/controllers/chat.ts:50-52`
  - body: `isPartialSilentReply` treats every non-empty prefix of `NO_REPLY`, including a complete streamed `NO`, as silent when final or abort payloads lack a normalized message. That can drop a legitimate short uppercase answer; carry stream-suppression provenance or use the narrower finalization approach being tracked in the related Gateway fix.
  - confidence: 0.78
