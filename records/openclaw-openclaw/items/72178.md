## Summary

Keep open. Current main still lacks cursor-based chat.history pagination, so the PR is not obsolete, but the proposed branch is not merge-ready: the server computes a cursor window without applying it to history projection, generated protocol clients are stale, and the required changelog entry is missing.

## What This Changes

The PR adds a before cursor and cursor/hasMore metadata to chat.history, load-older state and rendering in the web UI, partial NO_REPLY stream suppression, protocol docs, and focused gateway/UI tests.

## Best Possible Solution

Keep the PR open and repair the gateway contract at the handler boundary: page the raw transcript window before projection or carry stable raw indices through projection/sanitization/capping, then update generated Swift protocol artifacts, docs, changelog, and handler-level tests while preserving current main's bounded history-read behavior.

## Reproduction Assessment

Yes. On the PR branch, create a session with more than 200 raw messages, request the latest chat.history page, then request chat.history with before equal to the returned cursor; because the handler still projects the full raw transcript, the older-page request can return the latest page again.

## Solution Assessment

No. Cursor-based pagination is a reasonable direction, but this implementation is not the best fix because it computes start/end without using them for the server-side message window and leaves typed protocol outputs and release notes out of sync.
