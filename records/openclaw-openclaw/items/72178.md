## Summary

Keep open. Current main still lacks the requested `chat.history` cursor contract, and this PR remains useful but not mergeable because the server pagination window is not actually applied and the protocol/changelog surfaces are incomplete.

## What This Changes

The PR adds `before`/`cursor`/`hasMore` pagination for `chat.history`, load-older WebChat UI behavior, partial `NO_REPLY` suppression, protocol docs, and focused Gateway/UI tests.

## Best Possible Solution

Repair the Gateway handler so pagination operates on the requested raw transcript window or carries stable raw indices through projection and byte capping, then regenerate protocol artifacts, add changelog coverage, and keep the UI prepend behavior covered by focused tests.

## Reproduction Assessment

Yes. On the PR head, create more than one page of raw chat messages, request the latest `chat.history`, then request again with `before` set to the returned cursor; because the handler still projects the full raw transcript, the older request can return overlapping latest messages.

## Solution Assessment

No. Cursor pagination is the right direction, but this implementation is not the best fix until the server applies the cursor window before projection/capping and keeps generated client protocol models in sync.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Apply the before window before projecting history:** `src/gateway/server-methods/chat.ts:1523-1527`
  - body: The handler computes `end` and `start` from `before`, but then still passes the full `rawMessages` array into `projectRecentChatDisplayMessages`, whose limiter keeps the latest messages. A request for an older page can therefore return the newest page again and the UI will prepend duplicates; slice/read the raw `[start,end)` window or carry stable indices through projection before capping.
  - confidence: 0.93

  - body: Adding `before` to `ChatHistoryParamsSchema` changes the public Gateway request shape, but the generated Swift `ChatHistoryParams` structs still expose only `sessionKey`, `limit`, and `maxChars`. Regenerate protocol outputs so typed clients and `protocol:check` match the schema.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `docs/gateway/protocol.md:358`
  - body: This PR adds user-facing `chat.history` pagination and fixes visible partial `NO_REPLY` leakage, but the PR head has no matching `CHANGELOG.md` entry. Add a single-line entry under the active version before merge.
  - confidence: 0.8
