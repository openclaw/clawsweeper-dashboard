## Summary

Keep open: the PR addresses a real WebChat/Gateway gap, but the current head still has blocking pagination/protocol issues and lacks contributor-provided real behavior proof.

## What This Changes

The branch adds `before`/`cursor`/`hasMore` pagination to Gateway `chat.history`, WebChat load-older UI/state, partial `NO_REPLY` stream suppression, docs, and tests.

## Best Possible Solution

Revise this branch or open a focused replacement that windows raw history before projection/capping, regenerates protocol artifacts, reconciles the `NO_REPLY` scope, and includes redacted real WebChat/Gateway proof.

## Reproduction Assessment

Yes. Source inspection of PR head gives a high-confidence path: load a transcript longer than the page limit, request the latest `chat.history` page, then request `before` with the returned cursor; the handler still projects `rawMessages` and can return the newest page again.

## Solution Assessment

No. Cursor pagination is a reasonable additive direction, but this implementation should apply the cursor window before display projection/capping and keep generated protocol clients synchronized.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Apply pagination before projecting history:** `src/gateway/server-methods/chat.ts:1527-1530`
  - body: The handler computes `end` and `start` from `before`, but still calls `projectRecentChatDisplayMessages(rawMessages, ...)`. That helper keeps the newest `maxMessages`, so an older-page request can return the latest page again and make WebChat prepend duplicates; window the raw transcript before projection/capping.
  - confidence: 0.95

  - body: Adding `before` changes the public Gateway schema, but the generated Swift models and protocol schema outputs are not included. Typed clients remain out of sync, and the repository's `protocol:check` gate is designed to catch that drift.
  - confidence: 0.92
