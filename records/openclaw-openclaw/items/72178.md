## Summary

Keep open: current main still lacks WebChat load-older pagination, but this PR is not merge-ready because it duplicates an existing cursor history surface, still computes pages from the full raw transcript, leaves generated protocol outputs stale, and has no real behavior proof.

## What This Changes

This PR adds `before`/`cursor`/`hasMore` support to Gateway `chat.history`, WebChat load-older state/rendering/scroll compensation, partial `NO_REPLY` suppression, docs, and unit tests.

## Best Possible Solution

Rework the pagination path around the existing cursor-capable session-history endpoint or get explicit maintainer approval for a new `chat.history` protocol cursor, then prove WebChat paging and `NO_REPLY` behavior in a real setup.

## Reproduction Assessment

Yes, at source level for the PR defect: the latest diff computes `start`/`end` from `before` but still projects the full `rawMessages`, and the projection helper keeps the newest `maxMessages`. I did not run a browser or Gateway instance in this read-only review.

## Solution Assessment

No. Cursor pagination is a useful direction, but this implementation should either reuse the existing session-history cursor endpoint or get maintainer approval for a new `chat.history` cursor contract, then window history before projection and keep generated protocol clients synchronized.
