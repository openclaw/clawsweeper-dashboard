## Summary

Keep open. Current main does not implement the requested cursor pagination, and the PR head still has merge-blocking protocol and Gateway handler issues; the partial NO_REPLY fix is useful but should land with a corrected pagination implementation.

## What This Changes

The PR adds a `before` cursor to `chat.history`, UI load-older state/rendering, partial `NO_REPLY` fallback suppression, protocol docs/schema changes, and focused tests.

## Best Possible Solution

Repair the PR so the Gateway handler pages over the requested raw transcript window or carries stable raw indices through projection/capping, then regenerate protocol artifacts, add the active-version changelog entry, rebase onto current main, and validate the WebChat flow.

## Reproduction Assessment

Yes. Source inspection of the PR head is enough: a transcript longer than one page can call `chat.history` once, then call it again with `before=cursor`, but the handler still projects the full transcript so the second call can overlap the newest page.

## Solution Assessment

No. Cursor pagination is a reasonable additive protocol direction, but this implementation is not the best solution until the server applies the cursor window before projection/capping and keeps generated protocol clients in sync.

## Review Findings
