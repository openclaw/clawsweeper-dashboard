## Summary

Keep open. This PR targets a real current-main Telegram DM session fragmentation bug and is paired with the open reporter issue it closes, but it needs repair before merge because current main moved Telegram startup wiring into `bot-core.ts` while the PR patches the old `bot.ts` body, and the public Telegram docs still describe the old default behavior.

## What This Changes

The PR adds a Telegram `dmThreadReplies` config at account and per-DM levels, defaults DM session-key derivation to ignore DM `message_thread_id`, preserves thread metadata, updates schema/SDK exports/tests, and adds a changelog entry.

## Best Possible Solution

Land a rebased, documented version that changes only Telegram DM session-key derivation by default, preserves outbound thread metadata, keeps forum group topic isolation unchanged, and gives users explicit account/per-DM opt-in for legacy thread-scoped DM sessions.

## Reproduction Assessment

Yes. Current main has a unit-level reproduction in the Telegram DM thread tests: a private chat message with `message_thread_id` currently maps to `agent:main:main:thread:1234:42`, matching the reporter's context-loss path.

## Solution Assessment

Not yet. The session-key policy is the right narrow direction, but the PR must be ported to current `bot-core.ts` and documented before it is the maintainable fix.
