## Summary

Keep open: current `main` and the latest shipped tag still reject hardlinked files during the staged npm global swap, so the reported failure path is real and not superseded by the broader external-updater proposal.

## What This Changes

The issue reports that `openclaw update` fails during the npm global install swap on macOS/fnm when npm creates hardlinked dependency binaries in the staging install tree.

## Best Possible Solution

Allow or copy hardlinked files only for the owned npm staging package tree during the global install swap, while keeping hardlink rejection for untrusted source, backup, credential, hook, and workspace boundary paths.

## Reproduction Assessment

Yes, source-reproducible: if any file under `params.stage.packageRoot` has `nlink > 1`, the staged swap calls the rejecting hardlink guard and returns the reported error. I did not run a live macOS/fnm update in this read-only pass.

## Solution Assessment

Yes: a narrow updater fix is better than waiting for the larger external-updater work, and it should be scoped to the trusted staged npm tree with regression coverage.
