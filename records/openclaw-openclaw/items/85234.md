## Summary

Keep open: current main and the latest shipped release still route the npm staged package swap through the hardlink-rejecting copy path, so a hardlinked dependency binary in the staging tree can abort the update with the reported error.

## What This Changes

The issue requests that `openclaw update` handle npm-created hardlinked dependency binaries during the global install swap on macOS/fnm instead of rolling back.

## Best Possible Solution

Allow or safely copy hardlinked files only for the trusted npm staging package tree during the global install swap, while keeping hardlink rejection for untrusted boundary paths.

## Reproduction Assessment

Yes, source-reproducible: if a regular file under `params.stage.packageRoot` has `nlink > 1`, the staged swap calls the rejecting hardlink guard and returns the reported error. I did not run a live macOS/fnm update in this read-only pass.

## Solution Assessment

Yes: a narrow updater fix is the best immediate path, scoped to the owned staging tree rather than weakening hardlink rejection across credential, hook, workspace, or other untrusted boundary reads.
