## Summary

Keep open: current main and the latest release still use the explicit-only heartbeat policy, so the reported mixed defaults-plus-opt-out behavior remains unresolved. Because current docs and tests now codify that policy, changing it needs maintainer compatibility/product judgment rather than stale cleanup.

## What This Changes

The issue requests per-agent heartbeat config to override `agents.defaults.heartbeat` per agent instead of making all non-explicit agents heartbeat-disabled.

## Best Possible Solution

Maintainers should choose the intended heartbeat scope contract; if mixed defaults should work, centralize that policy across the summary helper, scheduler selection, runtime skip checks, docs, and regression tests with explicit upgrade notes.

## Reproduction Assessment

Yes. Source inspection shows the reported config path still disables the default-inheriting agent via `isHeartbeatEnabledForAgent` and `resolveHeartbeatAgents`; I did not run tests because this review is read-only.

Unclear. Per-agent override semantics would solve the user workflow, but it changes currently documented/tested config behavior and should only land after maintainer compatibility approval.
