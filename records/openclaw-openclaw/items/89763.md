## Summary

Keep open: current main still has the unguarded plugin session-action authorization/dispatch path, and this PR is a useful partial hardening, but the head still leaves `requiredScopes` array metadata reads unguarded in method-scope authorization. The maintainer-labeled draft is also conflicting and overlaps https://github.com/openclaw/openclaw/pull/89852 and https://github.com/openclaw/openclaw/pull/90034, so a maintainer should choose one canonical branch and require the shared guarded-reader repair before merge.

## What This Changes

The branch adds guarded plugin session-action lookup and dispatch code in gateway method-scope and plugin host-hook paths, switches handler invocation to `Reflect.apply`, and adds plugin contract regressions for malformed session-action metadata.

## Best Possible Solution

Land one canonical guarded session-action registration and scope reader shared by gateway authorization and dispatch, with array-proxy and sparse-array regressions, then retire or rebase the overlapping drafts.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: plugin session-action authorization runs before dispatch, and PR head still reads `.length` and `.filter` on plugin-owned array metadata; a Node proxy probe showed that an Array can pass `Array.isArray` while those reads throw.

## Solution Assessment

No. The dispatch guard is useful, but the best fix is one shared guarded registration/scope reader used by both method-scope authorization and dispatch, with focused regressions for throwing array metadata.
