The dispatch-side guard is useful, current main still has the underlying unguarded session-action metadata path, and the protected `maintainer` label rules out cleanup close; however the branch leaves the authorization/scope helper able to throw on plugin-owned `requiredScopes` array traps before dispatch can fail closed.

## What This Changes

The PR guards plugin session-action registry traversal in gateway scope resolution and dispatch, and adds contract tests for unreadable metadata plus handler receiver preservation.

## Best Possible Solution

Land one canonical guarded session-action lookup path only after all plugin-owned array length and element reads are guarded in both authorization and dispatch and covered by focused regressions.

## Reproduction Assessment

Yes, source-reproducible: a Node probe confirms Array proxies pass `Array.isArray` while throwing on `length`, and the PR's method-scopes helper still reads `requiredScopes.value.length` and `.filter(...)` directly.

## Solution Assessment

No, not yet; dispatch guarding is the right direction, but the authorization/scope helper needs the same guarded array iteration or a shared helper before this is the best complete fix.
