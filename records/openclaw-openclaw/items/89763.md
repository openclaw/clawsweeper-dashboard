## Summary

Keep open: current main still has the unguarded plugin session-action metadata path, and this protected maintainer draft is not merge-ready because its authorization helper can still throw before the new guarded dispatch response runs.

## What This Changes

The PR guards plugin session-action registry traversal in gateway scope resolution and dispatch, fails closed on unreadable dispatch metadata, preserves handler receivers with `Reflect.apply`, and adds session-action contract tests.

## Best Possible Solution

Land one canonical branch only after both dynamic scope resolution and dispatch use a fully guarded session-action metadata reader with regressions for throwing array traps.

## Reproduction Assessment

Yes, source-reproducible: current main and this PR's auth helper read plugin-owned `requiredScopes` array metadata before dispatch, and a Node proxy check confirms `Array.isArray` does not make `length` safe.

## Solution Assessment

No, not yet. Guarding dispatch is the right direction, but the best fix is a single or mirrored guarded metadata reader used consistently by method-scopes authorization and dispatch.
