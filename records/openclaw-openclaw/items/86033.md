## Summary

Keep open: current main still has a source-backed Codex app-server tool-surface gap where restricted `toolsAllow` disables native Codex file/code mode, then the dynamic-tool filter removes OpenClaw-owned `read`, `write`, and `edit` before the allowlist can preserve them.

## What This Changes

The issue reports that isolated cron sessions using the openai-codex Responses route omit allowlisted `read`, `write`, and `edit` tools.

## Best Possible Solution

Keep native Codex code mode disabled for narrow allowlists, but preserve explicitly allowlisted OpenClaw-owned file tools through the dynamic-tool bridge with regression coverage for `read/write/edit` and sandbox access modes.

## Reproduction Assessment

Yes. Source inspection shows a narrow `toolsAllow` disables the native Codex file surface and then the dynamic-tool filter removes `read/write/edit` before the allowlist is applied.

## Solution Assessment

Yes. The maintainable fix is to preserve explicitly allowlisted OpenClaw-owned file tools when native Codex tool mode is disabled, rather than re-enabling the broad native file/code surface for restricted policies.
