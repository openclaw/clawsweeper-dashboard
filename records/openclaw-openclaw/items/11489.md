## Summary

Keep open: current main and v2026.5.20 still have DM-only pairing while group access remains allowlist/open/disabled with block/drop behavior. The related Telegram, WhatsApp, and listen-only items cover adjacent slices, but none implements cross-channel owner notification, pending group approval state, approve/reject controls, or a settled pending read-only mode.

## What This Changes

The issue requests a new group pairing flow with owner notification, group approve/reject controls, and optional read-only pending group routing for unconfigured groups.

## Best Possible Solution

Define an additive cross-channel group approval contract with explicit owner authority, persisted pending state keyed by channel/account/group, owner-visible group-add notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; treat pending read-only observation as a separately approved mode or a clearly bounded part of that contract.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing contract. Source, docs, latest-release inspection, and related live issue state show group behavior remains allowlist/open/disabled while pairing remains DM-only.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable solution is a deliberate group approval contract, not simply reusing DM pairing state or treating ambient/listen-only observation as the whole approval flow.
