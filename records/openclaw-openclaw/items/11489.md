## Summary

Keep open: current main and the latest release still keep pairing on the DM path only; group admission remains allowlist/open/disabled plus ambient-room options, with no cross-channel owner notification, pending approval state, approve/reject controls, or approved read-only pending-group contract. Related work such as https://github.com/openclaw/openclaw/issues/8495, https://github.com/openclaw/openclaw/pull/50682, and https://github.com/openclaw/openclaw/pull/42400 is useful context but does not supersede the requested group pairing flow.

## What This Changes

The issue requests group pairing with owner notification, approve/reject controls, and optional read-only pending-group routing for unconfigured Telegram and WhatsApp groups.

## Best Possible Solution

Define an additive cross-channel group approval contract with explicit owner authority, persisted pending state keyed by channel/account/group, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; keep read-only pending observation as an explicitly approved mode within or beside that contract.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken current contract. Source, docs, latest-release inspection, maintainer notes, and related live item state show group behavior remains allowlist/open/disabled while pairing remains DM-only.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable path is a deliberate group approval contract, not reusing DM pairing state or treating ambient/listen-only observation as the whole approval flow.
