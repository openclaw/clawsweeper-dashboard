## Summary

Keep open: current main and the latest release still provide DM-only pairing plus group open/disabled/allowlist behavior, while the requested group approval state, owner notification, approve/reject flow, and pending read-only handling remain unimplemented and security-sensitive.

## What This Changes

The issue requests a new group-pairing approval flow with owner notification and optional read-only pending-group handling for Telegram/WhatsApp-style group chats.

## Best Possible Solution

Define one additive cross-channel group-pairing contract first: persist pending group approvals by channel/account/group, deliver redacted owner-visible group-add notifications, add authorized approve/reject operations, and wire Telegram/WhatsApp through shared policy state while keeping reader-agent observation as a separately approved mode.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new group-pairing and pending-approval feature request. Source inspection confirms current main only supports DM pairing plus group open/disabled/allowlist behavior.

## Solution Assessment

Unclear until maintainers settle the product/security contract. The requested direction is plausible, but the safest solution needs a shared approval-state design first and should keep reader-agent or listen-only observation separate unless explicitly approved.
