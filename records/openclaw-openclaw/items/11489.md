## Summary

Keep open: current main and the latest release still keep pairing on the DM path while group access remains open, disabled, or allowlist-based. The requested owner notification, pending group approval state, group approve/reject controls, and pending read-only routing are useful but need product and security design before implementation.

## What This Changes

The issue requests a cross-channel group pairing flow with owner notification, approve/reject controls, and optional read-only pending-group routing for Telegram, WhatsApp, and similar group-capable channels.

## Best Possible Solution

Define an additive cross-channel group approval contract with persisted pending state, owner-visible group-add notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; split pending read-only observation into a separate mode unless maintainers decide it belongs in the same approval contract.

## Reproduction Assessment

Not applicable: this is a feature request, not broken existing behavior. Current-main source, docs, and the v2026.5.12 spot check give a high-confidence verification path that group pairing is absent and current group behavior is allow, block, or allowlist only.

## Solution Assessment

Unclear until maintainers settle the product and security model. A deliberate cross-channel approval contract is the maintainable direction; silently reusing DM pairing or treating listen-only/read-only observation as a complete replacement would leave authorization and data-exposure questions unresolved.
