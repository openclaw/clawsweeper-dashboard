## Summary

Keep open. Current main and the latest release still support DM-only pairing while group admission remains open, disabled, or allowlist-based; no replacement issue or viable PR implements owner-approved group pairing. The request fits OpenClaw’s security direction, but its authorization state, notification routing, and reader-agent behavior require explicit maintainer product and security decisions.

## What This Changes

Add owner-approved group pairing, group-join notifications, and optional restricted observation for groups awaiting approval.

## Best Possible Solution

Define an additive cross-channel group-authorization contract with account-scoped pending approvals, an explicit trusted owner-notification destination, and authorized approve/reject operations; scope reader-agent or silent-observation behavior separately unless maintainers approve one shared security model.

## Reproduction Assessment

Not applicable: this is a feature request rather than broken documented behavior. Current source, docs, and the latest release consistently confirm that pairing is DM-only and group access uses explicit policies and allowlists.

## Solution Assessment

No, not as one combined implementation. Shared group approval and owner notification are the maintainable core direction, while reader-agent routing should be a separately reviewed security capability unless maintainers intentionally unify the contracts.
