## Summary

Keep open: current main and v2026.5.22 still limit group admission to open, disabled, or allowlist behavior while pairing remains DM-only. The related open work covers adjacent group notifications, auto-leave, or listen-only behavior, but it does not supersede the requested owner-approved group pairing contract.

## What This Changes

Requests a cross-channel group pairing mode that notifies the owner, stores pending group approval, offers approve/reject controls, and optionally routes unapproved groups through a read-only reader agent.

## Best Possible Solution

Define an additive group approval contract with persisted pending state, owner-visible join notifications, authorized approve/reject operations, and channel-specific Telegram/WhatsApp wiring; treat pending read-only observation as an explicitly approved mode within or beside that contract.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken current contract. Source, docs, latest release inspection, and related live issue state show groups remain allowlist/open/disabled while pairing remains DM-only.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable path is a deliberate group approval contract, not reusing DM pairing state or treating ambient/listen-only observation as the whole approval flow.
