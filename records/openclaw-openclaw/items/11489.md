## Summary

Keep open: current main and v2026.6.6 still only support group access as open, disabled, or allowlist, while pairing remains DM-scoped. The related open issues cover adjacent unknown-group hardening and WhatsApp join handling, and the closed neverReply PR was not merged, so no canonical item or shipped implementation supersedes this cross-channel group-approval request.

## What This Changes

The issue requests a cross-channel group pairing flow with owner notification, approve/reject controls, and optional read-only pending-group observation for unconfigured Telegram or WhatsApp groups.

## Best Possible Solution

Define a shared additive group-approval contract with config/schema/docs, persisted pending approvals keyed by channel/account/group, owner-visible notifications, authorized approve/reject operations, and Telegram/WhatsApp integration; keep reader-agent observation separate unless maintainers explicitly include it.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source, docs, tests, current main, and v2026.6.6 show the requested group-pairing behavior is absent while current behavior remains DM pairing plus group open/disabled/allowlist handling.

## Solution Assessment

No, the issue's direction is useful but should not be solved by directly extending the DM pairing path. The best path is a maintainer-approved shared product/security contract for group approval, notification, persistence, and any read-only observation mode.
