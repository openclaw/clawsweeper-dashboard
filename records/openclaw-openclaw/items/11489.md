## Summary

Keep open: current main still treats pairing as a DM-only access flow, while group access remains allowlist/open/disabled with no pending group approval state, owner notification, group approve/reject command, or pending read-only group route. The related WhatsApp auto-leave, Telegram unknown-group, unauthorized-hook, and listen-only items are adjacent but do not supersede this cross-channel approval-flow request.

## What This Changes

Request a cross-channel group pairing flow with owner notification, approve/reject controls, and optional read-only handling for unconfigured groups.

## Best Possible Solution

Define an additive cross-channel group-pairing contract with persisted pending group state, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; split reader-agent observation unless maintainers explicitly include it.

## Reproduction Assessment

Not applicable as a feature request. Current-main source, docs, and tests give high confidence the requested capability is absent: schemas reject group pairing, pairing remains DM-only, and Telegram/WhatsApp group paths allow or block without pending approval.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable direction is a deliberate group approval contract, not silently reusing DM pairing or the closed listen-only core-config PR path.
