## Summary

Keep open. Current main still implements DM-only pairing while group access remains allowlist/open/disabled, and the requested owner notification, pending group approval state, group approve/reject flow, and pending read-only routing are not implemented or superseded by the adjacent open work.

## What This Changes

Request a group pairing flow that notifies the owner when a bot joins a group, blocks or restricts unapproved groups, and optionally routes pending group messages through a read-only reader agent.

## Best Possible Solution

Define an additive cross-channel group-pairing contract with schema/types, pending approvals keyed by channel/account/group, owner-visible group-add events, authorized approve/reject operations, and Telegram/WhatsApp implementations; keep reader-agent observation separately scoped unless maintainers explicitly include it.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Static verification is high-confidence because current main rejects `groupPolicy: "pairing"`, keeps pairing state sender-code based, and group runtime paths directly allow or block messages without pending approval.

## Solution Assessment

Unclear as written. Owner notification plus approve/reject group state is a maintainable direction, but bundling pending approval with read-only reader-agent ingestion is not clearly the narrowest safe first change until authorization and isolation semantics are settled.
