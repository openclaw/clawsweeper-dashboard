## Summary

Keep open: current main still has DM-only pairing, while group access remains open/disabled/allowlist with no pending group approval state, owner notification, group approve/reject CLI, or pending read-only routing. The related open WhatsApp item at https://github.com/openclaw/openclaw/issues/8495 and the closed listen-only PR at https://github.com/openclaw/openclaw/pull/42400 are adjacent but do not supersede this approval-flow request.

## What This Changes

The issue requests group pairing approval, owner notifications on group joins, and optional read-only pending-group handling for Telegram/WhatsApp-style groups.

## Best Possible Solution

Define an additive cross-channel group-pairing contract with persisted pending group state, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; split reader-agent observation unless maintainers explicitly include it.

## Reproduction Assessment

Not applicable as a feature request. Current-main source and docs give high confidence the capability is absent: schemas reject group pairing, pairing docs keep approvals DM-only, and Telegram/WhatsApp group paths allow or block without pending approval.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable direction is a deliberate group approval contract; the existing listen-only plugin direction does not solve owner notification or group approval semantics.
