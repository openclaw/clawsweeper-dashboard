## Summary

Keep open: current main and v2026.6.8 still keep pairing scoped to DMs while group access is only open, disabled, or allowlist; related Telegram/WhatsApp access-request and unknown-group work only partially overlaps and does not settle this cross-channel group approval flow.

## What This Changes

The issue requests a group-pairing flow with owner notification, approve/reject operations, and optional read-only pending-group routing for unconfigured Telegram and WhatsApp groups.

## Best Possible Solution

Define an additive shared group-approval contract with config/schema/docs, durable pending approvals, owner-visible notifications, authorized approve/reject operations, and a separately scoped read-only observation mode if maintainers accept that security boundary.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source, docs, tests, and v2026.6.8 inspection show current behavior is DM pairing plus group open/disabled/allowlist handling, not group pairing approval.

## Solution Assessment

No, directly copying DM pairing into groupPolicy is not enough. The maintainable solution needs a maintainer-approved group identity, notification, persistence, authorization, and observation contract across Telegram and WhatsApp.
