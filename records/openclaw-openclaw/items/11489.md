## Summary

Keep open: current main still has DM-only pairing and group access remains open/disabled/allowlist, with no pending group approval state, owner notification path, group approve/reject command, or pending reader-agent route. The adjacent WhatsApp auto-leave, raw inbound hook, listen-only, and closed neverReply/plugin-path discussions do not supersede this cross-channel group approval request.

## What This Changes

The issue requests a cross-channel group pairing flow with owner notification, approve/reject controls, and optional read-only pending-group observation.

## Best Possible Solution

Define an additive cross-channel group-pairing contract with persisted pending group state, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; coordinate or split read-only observation with the existing listen-only/raw-inbound work.

## Reproduction Assessment

Not applicable as a feature request. Current-main source, docs, and tests give a high-confidence verification path that the requested capability is absent: schemas reject group pairing, pairing remains DM-only, and Telegram/WhatsApp group paths allow or block without pending approval.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable direction is a deliberate group approval contract rather than silently reusing DM pairing or treating the closed neverReply/plugin-path PR as a replacement.
