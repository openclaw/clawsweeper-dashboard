## Summary

Keep open. Current main still implements pairing as a DM/node approval flow, while group access remains open, disabled, or allowlist-based with no pending group approval state, owner notification on group add, group approve/reject CLI, or pending reader-agent routing. The related open work is adjacent and does not cover the requested cross-channel approval contract.

## What This Changes

The issue requests a group pairing flow with owner notification for new group additions and optional read-only pending-group routing for Telegram, WhatsApp, and similar channels.

## Best Possible Solution

Define an additive cross-channel group-pairing contract: config/schema support for a pending group policy, persisted approval state keyed by channel/account/group, owner-visible group-add notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations. Treat reader-agent observation as a separately designed mode unless maintainers explicitly include it in the first approval contract.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Static verification is high-confidence because current main rejects groupPolicy:"pairing", keeps pairing state and CLI sender-code based, and group runtime paths directly allow or block messages without pending approval.

## Solution Assessment

Unclear as written. Owner notification plus approve/reject group state is a maintainable direction, but combining pending approval with read-only reader-agent ingestion is not clearly the narrowest safe first change until authorization and isolation semantics are designed.
