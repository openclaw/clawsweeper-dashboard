## Summary

Keep open: current main and the latest shipped release still have DM-only pairing while group access remains open, disabled, or allowlist-based. The adjacent WhatsApp auto-leave, Telegram audit, listen-only, and denied-inbound hook requests are useful context but do not supersede this cross-channel group approval flow.

## What This Changes

Request to add `groupPolicy: "pairing"` with owner notifications, approve/reject controls, and optional read-only pending group routing for Telegram/WhatsApp groups.

## Best Possible Solution

Define an additive cross-channel group approval contract with persisted pending state, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; split or explicitly coordinate pending read-only observation with listen-only and hook work.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing contract. Current-main source, docs, and tests give a high-confidence verification path that the requested capability is absent: schemas reject group pairing, pairing remains DM-only, and Telegram/WhatsApp group paths allow or block without pending approval.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable direction is a deliberate group approval contract rather than silently reusing DM pairing or treating listen-only/plugin hooks as a replacement.
