## Summary

Current main still supports pairing only for direct messages; group access remains open, disabled, or allowlisted, and no viable related PR implements the requested approval flow. This should remain open for an explicit cross-channel product and security decision.

## What This Changes

Requests cross-channel group pairing with owner notifications, approval controls, and optional read-only handling for unconfigured groups.

## Best Possible Solution

Define an additive shared pending-group authorization contract with durable channel, account, and group identity plus authorized owner notifications; scope silent observation as a separate opt-in security mode only if maintainers approve untrusted content entering agent state.

## Reproduction Assessment

Not applicable: this is a feature request, and current source, schemas, CLI behavior, and documentation consistently confirm that group pairing is not an existing contract.

## Solution Assessment

No, not as one combined feature. Owner approval and notification form a coherent authorization capability, but reader-agent observation needs a separate security and product decision because it changes the session-state boundary.
