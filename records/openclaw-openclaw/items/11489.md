## Summary

Keep open: current main and v2026.5.12 still scope pairing to DMs, while group ingress remains open/disabled/allowlist with explicit allowlists. The requested owner notification, pending group approval state, approve/reject controls, and read-only pending routing remain product/security design work rather than an implemented or superseded cleanup item.

## What This Changes

Request adding cross-channel group pairing approval, owner notifications on group add, and optional read-only pending-group handling for Telegram and WhatsApp groups.

## Best Possible Solution

Define an additive cross-channel group approval contract with explicit owner authority, persisted pending state, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; keep pending read-only observation as an explicitly approved mode within or beside that contract.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken current contract. Current-main source, docs, live related-item state, and the v2026.5.12 tag show group behavior is allowlist/open/disabled while pairing remains DM-only.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable path is a deliberate group approval contract, not reusing DM pairing state or treating ambient/read-only observation as a complete approval flow.
