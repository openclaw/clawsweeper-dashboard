## Summary

Keep open: current main and v2026.5.12 still scope pairing to DMs/devices while group access remains allowlist/open/disabled plus ambient observation for already-approved rooms; owner notification, pending group approval state, group approve/reject controls, and pending read-only routing are still missing and need product/security design.

## What This Changes

Request a cross-channel group pairing flow that notifies the owner, lets groups be approved or rejected, and optionally routes unapproved group traffic through a read-only reader agent.

## Best Possible Solution

Define an additive cross-channel group approval contract with persisted pending state, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; scope pending read-only observation as a separately approved mode or an explicit part of that contract.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken current contract. Current-main and v2026.5.12 source/docs/tests give a high-confidence verification path that group pairing is absent and current group behavior is allow, block, allowlist, or ambient observation for already-approved rooms.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable path is a deliberate group approval contract, not reusing DM pairing state or treating ambient room events as a complete owner-approval flow.
