## Summary

Keep open: current main still has DM-only pairing and group access limited to open, disabled, or allowlist behavior; there is no pending group approval state, owner join notification, group approve/reject CLI, or pending read-only routing. Related #42400 and #8495 are adjacent but do not supersede the full group-pairing request.

## What This Changes

Request adds a cross-channel group pairing mode with owner notifications, approve/reject controls, and optional read-only pending-group routing.

## Best Possible Solution

Define an additive group-pairing contract with persisted pending group state, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; split reader-agent observation unless maintainers explicitly choose to include it in the same contract.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence the requested capability is absent: schemas reject `groupPolicy: "pairing"`, docs keep DM pairing separate from group authorization, and Telegram/WhatsApp group paths block or allow without pending approval.

## Solution Assessment

Yes for keeping the issue open, but not as an autonomous fix yet. The narrow maintainable path is a maintainer-approved cross-channel approval contract first, with silent observation handled as a separate or explicitly bundled mode.
