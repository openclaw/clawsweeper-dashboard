## Summary

Keep open: current main and v2026.5.12 still implement DM-only pairing and group access remains open, disabled, or allowlist. Ambient room events and related unknown-group work cover adjacent quiet-context or auto-leave behavior, but they do not provide cross-channel owner notification, pending group approval state, group approve/reject controls, or pre-approval routing semantics.

## What This Changes

Request an additive group-pairing flow that notifies the owner when a bot joins a new group, supports group approve/reject controls, and optionally routes pending groups through read-only observation.

## Best Possible Solution

Define an additive cross-channel group approval contract with explicit owner authority, persisted pending state, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; keep pending read-only observation as an explicitly approved mode within or beside that contract.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken current contract. Current-main source, latest-release inspection, docs, tests, and related live issue state show group behavior remains allowlist/open/disabled while pairing remains DM-only.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable path is a deliberate group approval contract, not reusing DM pairing state or treating ambient/read-only observation as the whole approval flow.
