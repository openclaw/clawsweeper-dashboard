## Summary

Keep open: current main and v2026.5.12 still keep pairing on the DM/device paths, while group authorization remains allowlist/open/disabled with no pending group approval state, owner join notification, group approve/reject controls, or approved pending-reader mode. The related ambient room events, before-dispatch hook, and listen-only plugin path cover some quiet observation use cases, but they do not supersede the requested owner approval and group-join notification contract.

## What This Changes

Request to add a group pairing policy with owner notification, group approve/reject controls, and optional read-only pending-group routing for Telegram/WhatsApp-style groups.

## Best Possible Solution

Define an additive cross-channel group approval contract with persisted pending state, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; treat pending read-only observation as a separately approved mode or explicitly scoped part of the same contract.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken current contract. Current-main and v2026.5.12 source/docs/tests give high-confidence verification that group pairing is absent and current group behavior is allow, block, allowlist, or ambient observation for already-allowed rooms.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable path is a deliberate group approval contract, not reusing DM pairing state or treating before_dispatch/listen-only observation as a complete owner-approval flow.
