## Label Justifications

- `P2`: This is a normal-priority security and control enhancement for group chat authorization, not an active outage or regression.
- `impact:security`: The request changes authorization for unconfigured shared rooms and proposes owner approval before a bot can engage in them.

## Summary

Keep open: current main still keeps pairing on the DM path and handles groups through allowlist/open/disabled admission, with no owner-visible group-add notification, pending group approval state, group approve/reject CLI, or approved read-only pending-group contract. Related work covers adjacent auto-leave, unknown-group warning, or listen-only behavior but does not supersede this cross-channel group-pairing flow.

## What This Changes

The issue requests a new group pairing flow with owner notification, group approve/reject controls, and optional read-only pending-group handling for unconfigured groups.

## Best Possible Solution

Define an additive cross-channel group approval contract with explicit owner authority, persisted pending state keyed by channel/account/group, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; keep read-only pending observation as an explicitly approved mode within or beside that contract.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken current contract. Source, docs, maintainer discussion, and related live item state show group behavior remains allowlist/open/disabled while pairing remains DM-only.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable path is a deliberate group approval contract, not reusing DM pairing state or treating ambient/listen-only observation as the whole approval flow.
