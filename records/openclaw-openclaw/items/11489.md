## Summary

Keep open: current main still treats pairing as a DM-only flow, while group access remains open, disabled, or allowlist; related #8495 and #42400 cover adjacent group-leave/listen-only behavior rather than owner-approved group pairing.

## What This Changes

Request an additive group pairing flow with owner notification, approve/reject controls, and optional read-only pending-group routing for Telegram, WhatsApp, and similar group-capable channels.

## Best Possible Solution

Define an additive cross-channel group-pairing contract with persisted pending group state, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; split reader-agent observation unless maintainers explicitly include it in the same contract.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main gives high confidence the capability is absent: schemas reject group pairing, docs keep pairing DM-only, and Telegram/WhatsApp group paths allow or block without pending approval.

## Solution Assessment

Yes, keeping this open is the best current action. The requested behavior needs a maintainer-approved group approval, notification, persistence, and read-only ingestion contract rather than a narrow bug fix.
