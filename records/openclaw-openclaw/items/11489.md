## Summary

Keep open. Current main still has DM-only pairing while group access remains open, disabled, or allowlist-only; the requested group approval state, owner notification, group approve/reject flow, and pending read-only routing are not implemented.

## What This Changes

Request adds a cross-channel group pairing mode with owner notifications, approve/reject controls, and optional read-only routing for unconfigured groups.

## Best Possible Solution

Define an additive cross-channel group-pairing contract with schema/types, persisted pending approvals keyed by channel/account/group, owner-visible group-add events, authorized approve/reject operations, and Telegram/WhatsApp implementations; keep reader-agent observation separately scoped unless maintainers explicitly include it.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Static verification is high-confidence because current main rejects groupPolicy pairing and group runtime paths directly allow or block messages without pending approval.

## Solution Assessment

Unclear as written. Owner notification plus approve/reject group state is a maintainable direction, but bundling pending approval with read-only reader-agent ingestion is not clearly the narrowest safe first change until authorization and isolation semantics are settled.
