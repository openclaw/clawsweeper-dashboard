## Summary

Keep open: current main and v2026.6.8 still have DM-only pairing while group handling remains open, disabled, or allowlist-based. The related Telegram/WhatsApp items cover adjacent unknown-group warning, auto-leave, or observation behavior, but none implements this cross-channel owner-notified group approval and pending read-only contract.

## What This Changes

Request a cross-channel group pairing flow with owner notification, approve/reject operations, and optional read-only pending-group routing.

## Best Possible Solution

Define an additive shared group-approval contract with config/schema/docs, durable pending approvals, owner-visible notifications, authorized approve/reject operations, and a separately scoped read-only observation mode if maintainers accept that security boundary.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source, docs, live related-item state, and v2026.6.8 inspection show the current behavior is DM pairing plus group open/disabled/allowlist handling, not group pairing approval.

## Solution Assessment

Unclear until maintainers settle the product and security shape. Copying dmPolicy pairing into groupPolicy is not enough; the maintainable solution needs owner notification, durable approval state, authorization semantics, and an explicit decision on pending read-only routing.
