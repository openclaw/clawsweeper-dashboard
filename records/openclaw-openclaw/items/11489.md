## Summary

Current main and release v2026.6.5 still provide pairing only for direct messages; group admission remains open, disabled, or allowlist-based, with no pending group approval, owner notification, group approve/reject operation, or restricted pending-group agent path. The request remains useful but needs explicit product and security decisions before implementation.

## What This Changes

Add owner-approved group pairing, group-join notifications, and optional restricted observation for groups awaiting approval.

## Best Possible Solution

Define an additive shared group-authorization contract keyed by channel, account, and group, with authenticated owner notifications and approve/reject operations; treat restricted observation as a separate opt-in capability unless maintainers approve one unified threat model.

## Reproduction Assessment

Not applicable: source and the latest release confirm that these group modes do not exist, but the item requests a new capability rather than reporting broken established behavior.

## Solution Assessment

No, directly extending the DM pairing workflow is not yet the best solution because group identity, lifecycle events, owner routing, authorization persistence, and untrusted-content handling require a distinct shared contract; reader-agent routing should initially be scoped separately.
