## Label Justifications

- `impact:security`: The request changes authorization for unconfigured shared rooms and proposes owner approval before a bot can engage in them.

## Summary

Keep open: current main still keeps pairing on the DM path and handles groups through allowlist/open/disabled admission, with no owner-visible group-add notification, pending group approval state, group approve/reject CLI, or approved read-only pending-group contract. Related work such as https://github.com/openclaw/openclaw/issues/8495, https://github.com/openclaw/openclaw/issues/50681, https://github.com/openclaw/openclaw/pull/50682, and the closed https://github.com/openclaw/openclaw/pull/42400 covers adjacent auto-leave or listen-only behavior but does not supersede this cross-channel group-pairing flow.

## What This Changes

Requested a group pairing flow that notifies an owner when a bot joins an unconfigured group and optionally keeps pending groups read-only until approval.

## Best Possible Solution

Define an additive cross-channel group approval contract with explicit owner authority, persisted pending state keyed by channel/account/group, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; keep read-only pending observation as an explicitly approved mode within or beside that contract.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken current contract. Source, docs, maintainer notes, and related live item state show group behavior remains allowlist/open/disabled while pairing remains DM-only.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable path is a deliberate group approval contract, not reusing DM pairing state or treating ambient/listen-only observation as the whole approval flow.
