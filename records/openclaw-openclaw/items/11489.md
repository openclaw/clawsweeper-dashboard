## Summary

Keep open: current main and the latest release still have DM-only pairing, while group admission remains allowlist/open/disabled with drop/log behavior. The related listen-only plugin path and adjacent group-security issues are useful context, but they do not provide owner notification, pending group approval state, group approve/reject controls, or a safe read-only pending-group contract.

## What This Changes

The issue requests group pairing for unconfigured Telegram/WhatsApp-style groups, including owner notification, approve/reject controls, and optional read-only pending-group routing.

## Best Possible Solution

Define an additive cross-channel group approval contract with explicit owner authority, persisted pending state, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; keep pending read-only observation as an explicitly approved mode within or beside that contract.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken current contract. Source, docs, latest-release inspection, and related live issue state show group behavior remains allowlist/open/disabled while pairing remains DM-only.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable path is a deliberate group approval contract, not reusing DM pairing state or treating the listen-only plugin path as the whole approval flow.
