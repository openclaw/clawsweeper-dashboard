## Summary

Keep open: current main and the latest release still implement DM-only pairing, while group admission remains allowlist/open/disabled with drop/log behavior. Related ambient/listen-only and Telegram/WhatsApp group-security items are useful context, but none provides cross-channel owner notification, pending group approval state, approve/reject controls, or a settled read-only pending-group contract.

## What This Changes

The issue requests group pairing for Telegram/WhatsApp-style groups, including owner notification, pending approval, approve/reject controls, and optional read-only observation for unconfigured groups.

## Best Possible Solution

Define an additive cross-channel group approval contract with explicit owner authority, persisted pending state keyed by channel/account/group, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; keep pending read-only observation as an explicitly approved mode within or beside that contract.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken current contract. Source, docs, latest-release inspection, and related live issue state show group behavior remains allowlist/open/disabled while pairing remains DM-only.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable path is a deliberate group approval contract, not reusing DM pairing state or treating ambient/listen-only observation as the whole approval flow.
