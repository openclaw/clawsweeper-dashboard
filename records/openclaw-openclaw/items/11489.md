## Summary

Keep open: current main and v2026.6.8 still support DM pairing plus group open/disabled/allowlist handling, but not cross-channel group pairing, owner notification, approve/reject state, or pending read-only group routing. Related Telegram, WhatsApp, alert/audit, and listen-only work is useful context but only partially overlaps this broader product/security contract.

## What This Changes

The issue requests a group pairing flow with owner notification, approve/reject operations, and optional read-only pending-group handling for Telegram and WhatsApp groups.

## Best Possible Solution

Define an additive shared group-approval contract with config/schema/docs, durable pending approvals, owner-visible notifications, authorized approve/reject operations, and a separately scoped read-only observation mode if maintainers accept that security boundary.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source, docs, and v2026.6.8 inspection show current behavior is DM pairing plus group open/disabled/allowlist handling, not group pairing approval.

## Solution Assessment

No, directly copying DM pairing into `groupPolicy` is not enough. The maintainable solution needs a maintainer-approved group identity, notification, persistence, authorization, and observation contract across Telegram and WhatsApp.
