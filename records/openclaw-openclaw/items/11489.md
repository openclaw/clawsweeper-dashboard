## Summary

Keep open: current main and the latest shipped release still keep pairing scoped to DMs/devices, while group access remains allowlist/open/disabled plus ambient observation for already-allowed rooms. The requested owner notification, pending group approval state, group approve/reject controls, and pending read-only routing are still missing and need maintainer product/security design.

## What This Changes

Request an additive group pairing flow with owner notification, approve/reject controls, and optional read-only handling for unconfigured Telegram/WhatsApp groups.

## Best Possible Solution

Define an additive cross-channel group approval contract with persisted pending state, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; scope pending read-only observation as a separately approved mode or an explicit part of that contract.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken current contract. Current-main and v2026.5.12 source/docs/tests give high-confidence verification that group pairing is absent and current group behavior is allow, block, allowlist, or ambient observation for already-allowed rooms.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable path is a deliberate group approval contract, not reusing DM pairing state or treating ambient room events/listen-only observation as a complete owner-approval flow.
