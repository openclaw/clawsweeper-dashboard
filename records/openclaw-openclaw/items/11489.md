## Summary

Keep open. Current main still has DM-only pairing and group access is limited to open, disabled, or allowlist behavior; there is no pending group approval state, owner notification on group add, group approve/reject command, or pending read-only group routing.

## What This Changes

Request group pairing for newly added groups, owner notification/approval controls, and optional read-only reader-agent handling while groups are pending approval.

## Best Possible Solution

Define an additive cross-channel group-pairing contract with schema/types, persisted pending approvals keyed by channel/account/group, owner-visible group-add events, authorized approve/reject operations, and Telegram/WhatsApp implementations; keep reader-agent observation separately scoped unless maintainers explicitly include it.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Static verification is high-confidence because current main rejects `groupPolicy: "pairing"`, keeps pairing state sender-code based, and group runtime paths directly allow or block messages without pending approval.

## Solution Assessment

Unclear as written. Owner notification plus approve/reject group state is a maintainable direction, but bundling pending approval with read-only reader-agent ingestion is not clearly the narrowest safe first change until authorization and isolation semantics are settled.
