## Summary

Keep open: current main and the latest shipped release still treat pairing as DM-only while group access is open, disabled, or allowlist-based. The related WhatsApp auto-leave issue and the closed neverReply PR cover adjacent group-safety ideas, but they do not supersede owner notification, pending approval state, group approve/reject controls, or pending read-only routing.

## What This Changes

Add a cross-channel group pairing flow that notifies the owner when a bot is added to an unconfigured group, supports approve/reject, and can optionally route pending groups through a read-only reader agent.

## Best Possible Solution

Define an additive cross-channel group approval contract with persisted pending state, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; split or explicitly coordinate pending read-only observation with listen-only and hook work.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken current contract. Current-main source, docs, tests, and the v2026.5.12 spot check give a high-confidence verification path that group pairing is absent and current group behavior is allow, block, or allowlist only.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable direction is a deliberate cross-channel group approval contract, not silently reusing DM pairing or treating listen-only/group hooks as a complete replacement.
