## Summary

Keep open: current main and v2026.6.8 still implement DM-only pairing plus group open/disabled/allowlist behavior, while the requested owner notification, pending group approval state, group approve/reject commands, and read-only pending-group routing are not implemented. Related Telegram, WhatsApp, and listen-only/plugin work only partially overlaps and does not replace the cross-channel group-pairing design.

## What This Changes

The issue requests a cross-channel group pairing flow with owner notification, approve/reject handling, and optional read-only pending-group routing for unconfigured group chats.

## Best Possible Solution

Define one additive group-admission contract first, then implement channel-owned Telegram/WhatsApp behavior around shared policy state, owner-visible notifications, authorized approve/reject operations, and a separately scoped reader-agent or ambient observation mode if maintainers approve it.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source, tests, docs, and v2026.6.8 release inspection confirm the current behavior is DM pairing plus group open/disabled/allowlist, without group approval state or owner notification.

## Solution Assessment

Unclear until maintainers settle the product/security contract. The requested direction is reasonable, but the safest solution should separate group admission approval, owner notification, destructive leave/reject behavior, and reader-agent observation rather than merging them as one ad hoc channel patch.
