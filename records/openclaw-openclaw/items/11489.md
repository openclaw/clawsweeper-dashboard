## Summary

Keep open: current main and v2026.6.8 still keep pairing DM-only while group access is allowlist/open/disabled, with no pending group approval state, owner notification flow, group approve/reject CLI, or read-only pending-group routing. Related open work covers adjacent Telegram unknown-group actions and WhatsApp auto-leave/listen-only pieces, but no single item or merged PR supersedes this cross-channel group-pairing request.

## What This Changes

Request an additive group-pairing flow for Telegram/WhatsApp-style groups with owner notification, approve/reject controls, and optional read-only handling for unconfigured groups.

## Best Possible Solution

Define a deliberate cross-channel group admission contract: current group access stays fail-closed by default, pending group approvals are persisted by channel/account/group, owner notifications are redacted and rate-limited, approve/reject operations are authorized, and any read-only observation mode is separately scoped with tests and live Telegram/WhatsApp proof.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source and release inspection give high-confidence proof that current behavior is DM-only pairing plus group allowlist/open/disabled handling, without group approval state or read-only pending-group routing.

## Solution Assessment

No current implementation is the best solution yet. The maintainable path is a product/security-reviewed group admission contract rather than stretching DM pairing or merging the narrower Telegram/WhatsApp adjacent proposals independently.
