## Summary

Keep open: current main still treats pairing as DM-only, while group access remains open/disabled/allowlist-based with no pending approval state, owner notification, group approve/reject controls, or pending reader routing. The related listen-only/plugin work covers silent observation use cases, but it does not supersede group approval or owner-notification semantics.

## What This Changes

Request to add cross-channel group pairing with owner notifications, group approve/reject controls, and optional pending read-only routing for unconfigured Telegram/WhatsApp groups.

## Best Possible Solution

Define an additive cross-channel group approval contract with persisted pending state, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; coordinate or split pending read-only observation from the approval flow.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken current contract. Current-main source, docs, live issue context, and a v2026.5.12 spot check give a high-confidence verification path that group pairing is absent and current group behavior is allow, block, or allowlist only.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable direction is a deliberate cross-channel group approval contract, not silently reusing DM pairing or treating listen-only hooks as a complete replacement.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it changes shared-room approval authority and may process unapproved group content.
