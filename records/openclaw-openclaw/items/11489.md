## Summary

Keep open: current main and v2026.5.12 still keep pairing scoped to DMs, while group access remains allowlist/open/disabled plus mention/ambient behavior. The adjacent open and closed items cover group join hooks, auto-leave, or listen-only suppression, but they do not implement owner-notified pending group approval or group approve/reject state.

## What This Changes

Request adds group pairing with owner notifications, approve/reject controls, and optional read-only pending-group routing for unconfigured Telegram and WhatsApp groups.

## Best Possible Solution

Define an additive cross-channel group approval contract with persisted pending state, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; treat pending read-only observation as an explicitly approved mode within or beside that contract.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken current contract. Source, docs, tests, live issue state, and the v2026.5.12 tag give high-confidence verification that group pairing is absent and current group behavior is allowlist/open/disabled rather than pending approval.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable path is a deliberate group approval contract, not reusing DM pairing state or treating ambient/listen-only group observation as a complete owner-approval flow.
