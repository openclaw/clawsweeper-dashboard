## Summary

Keep open: current main and v2026.6.6 still do not implement group-policy pairing, owner notification on group add, persisted pending group approval, group approve/reject CLI, or pending read-only agent routing. Related work is partial, open, or closed unmerged, so it does not supersede the cross-channel group approval request.

## What This Changes

Request an additive group pairing flow for Telegram/WhatsApp-style groups with owner notification, approve/reject controls, and optional read-only observation for unconfigured groups.

## Best Possible Solution

Define a shared, additive group-approval contract with config/schema/docs, persisted pending approvals keyed by channel/account/group, owner-visible notifications, authorized approve/reject operations, and Telegram/WhatsApp integration; keep reader-agent observation separate unless maintainers explicitly include it.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source, docs, tests, and v2026.6.6 show the requested group-pairing behavior is absent while current behavior remains DM pairing plus group open/disabled/allowlist handling.

## Solution Assessment

No, the proposed direction is useful but should not be implemented by simply extending DM pairing. The best path is a maintainer-approved shared product/security contract for group approval, notification, persistence, and any read-only observation mode.
