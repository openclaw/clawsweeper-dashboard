## Summary

Keep open: current main still keeps pairing on the DM path only, while group access is open/disabled/allowlist-based with no pending group approval state, owner notification, group approve/reject controls, or pending read-only routing. The related silent-observation and WhatsApp group-join work does not supersede the requested approval and notification contract.

## What This Changes

Request an additive cross-channel group pairing flow with owner notification, approval/rejection controls, and optional read-only handling for unconfigured groups.

## Best Possible Solution

Define an additive cross-channel group approval contract with persisted pending state, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; treat pending read-only observation as a separately approved mode or explicitly scoped part of the same contract.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken current contract. Current-main source, docs, tests, live issue state, and v2026.5.12 source all verify that group pairing is absent and current group behavior is allow, block, or allowlist only.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable direction is a deliberate cross-channel group approval contract, not silently reusing DM pairing or treating listen-only hooks as a complete replacement.
