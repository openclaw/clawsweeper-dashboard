## Summary

Keep open: current main and v2026.5.12 still implement DM-only pairing while group access remains open, disabled, or allowlist. The related WhatsApp auto-leave issue and Telegram unknown-group PR are useful adjacent work, but they do not provide cross-channel owner notification, pending group approval state, approve/reject controls, or pending-group read-only routing.

## What This Changes

The issue requests group pairing for Telegram/WhatsApp-style groups, including owner notification, approve/reject controls, and optional read-only pending-group routing.

## Best Possible Solution

Define an additive cross-channel group approval contract with explicit owner authority, persisted pending state, owner-visible join notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations; keep pending read-only observation as an explicitly approved mode within or beside that contract.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken current contract. Source, docs, latest-release inspection, and related live issue state show group behavior remains allowlist/open/disabled while pairing remains DM-only.

## Solution Assessment

Unclear until maintainers settle the product and security model. The maintainable path is a deliberate group approval contract, not reusing DM pairing state or treating ambient/read-only observation as the whole approval flow.
