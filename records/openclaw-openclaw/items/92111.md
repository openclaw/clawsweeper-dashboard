## Summary

This PR remains necessary: current main marks failed detached updates but does not recover the managed gateway, while the branch adds recovery in the only surviving process and supplies convincing live systemd proof. The implementation is focused and no blocking correctness or security defect was found; maintainers should explicitly accept the source-and-unit-tested macOS and Windows coverage gap before merge.

## What This Changes

The PR makes the detached managed-update handoff start the prior systemd, launchd, or Windows gateway service after command failure and adds focused recovery regression tests.

## Best Possible Solution

Land the recovery in the detached handoff owner with canonical service identities and non-bouncing start semantics, retaining focused coverage for failed spawn, nonzero exit, signal termination, successful updates, and supervisor descriptors.

## Reproduction Assessment

Yes. The supplied real systemd scenario reproduces the current-main outage after an exit-127 update handoff and demonstrates service and HTTP recovery with this branch.

## Solution Assessment

Yes. Recovery belongs in the detached helper because it is the only process guaranteed to survive the failed update, and start-only service commands are narrower and safer than unconditional restarts when failure may occur before or after the gateway stop.
