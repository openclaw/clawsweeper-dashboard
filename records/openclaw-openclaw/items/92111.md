## Summary

## What This Changes

The PR adds service-recovery descriptors to the managed update handoff helper, starts the matching gateway service when the detached update command fails, and adds regression coverage for the helper behavior.

## Best Possible Solution

Land the handoff-level recovery after CI is resolved and the stale PR-body claim is corrected, keeping the recovery tied to the same service identity conventions used by the existing daemon restart paths.

## Reproduction Assessment

Yes, by source and supplied live proof rather than a local rerun. The linked issue shows systemd exit 127 after the gateway stop, current main exits the helper without service recovery, and the PR body shows before inactive/HTTP 000 versus after active/HTTP 200.

## Solution Assessment

Yes, this is the best layer for the reported failure: the detached helper is the surviving process after the gateway exits, while the existing CLI restart path already covers failures that remain inside the update command.
