## Summary

Keep open: current main has structured failed-tool state, hook follow-up primitives, final-answer revision hooks, and loop guards, but it still lacks a native tool-failure policy that classifies a failed tool result and forces a bounded same-run recovery continuation before accepting the final answer.

## What This Changes

The issue requests a native, configurable OpenClaw run-loop recovery step that gives the same agent bounded structured context after tool-call failure instead of relying on plugin-triggered follow-up turns.

## Best Possible Solution

Define an opt-in core recovery contract that classifies structured failed tool results, bounds same-run continuations, preserves side-effect metadata, and documents embedded/Codex/native capability limits.

## Reproduction Assessment

Not applicable as a feature request; source review confirms current main has observation, warning, loop guard, and generic final-answer revision primitives but no native in-loop tool-failure recovery policy to reproduce as a broken existing contract.

## Solution Assessment

Unclear pending product direction; native run-loop ownership is plausible for this gap, but the best implementation needs maintainer decisions on failure classes, mutating tools, attempt budgets, finalization policy, and runtime-specific capability limits.
