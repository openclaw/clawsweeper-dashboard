## Summary

Keep open: current main and the latest release provide hook observation, follow-up-turn primitives, warning finalization, loop guards, and retry/failover pieces, but they still do not provide the requested native in-loop tool-failure recovery continuation before final answer selection.

## What This Changes

The issue requests a native, configurable run-loop recovery step that gives the same agent one bounded continuation after a failed tool call with structured failure context.

## Best Possible Solution

Define an opt-in core-owned recovery contract that re-enters the same agent turn after structured failed tool results, preserves side-effect metadata, honors loop/replay guards, and treats Codex-native transcript boundaries as a first-class constraint.

## Reproduction Assessment

Not applicable as a feature request. Source review confirms current main has observation and follow-up primitives, but no native in-loop recovery continuation that is already an expected contract to reproduce as broken.

## Solution Assessment

Unclear until maintainers choose the product contract. Native run-loop ownership is plausible, but the implementation needs approved policy for finalization, mutation safety, attempt budgets, and native runtime boundaries.
