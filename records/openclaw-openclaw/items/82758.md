## Summary

Keep open: the report is fresh, concrete, and source inspection strongly supports the current-main bug. The Codex app-server client is still a single process-global slot keyed by agent/auth/start options, and a second agent with a different key clears and closes the first client, matching the reported cross-agent abort/wedge.

## What This Changes

Ensure concurrent native Codex app-server turns for different agents do not abort or wedge each other through the shared app-server client.

## Best Possible Solution

Replace the single shared-client slot with keyed client state, clear only the matching key/client, and add a regression test proving overlapping distinct agentDir/auth turns do not close each other.

## Reproduction Assessment

Yes, source-reproducible with high confidence: start overlapping native Codex app-server turns for two agents with different agentDir/auth keys, and current code clears the first shared client when the second key arrives. I did not live-run Telegram/Codex in this read-only review.

## Solution Assessment

Yes, the requested keyed-client map is the narrowest maintainable repair if multiple app-server processes are supported; explicit serialization or rejection is the fallback only if upstream runtime constraints require it.

## Repair Work Prompt

Fix the native Codex app-server multi-agent concurrency bug reported at https://github.com/openclaw/openclaw/issues/82758. Current behavior uses one process-global shared Codex app-server client slot, so a second agent with a different agentDir/auth/start key clears and closes the first agent's in-flight client. Implement a keyed shared-client registry so clients/promises are reused per resolved key and clearing/retry/timeout/dispose only closes the matching key or all keys when the whole harness is disposed. Preserve existing auth-token rotation behavior for the same endpoint/key family, do not add a config option, and do not change Telegram channel behavior directly. Add focused regression coverage in the Codex app-server shared-client tests showing two distinct agentDir/auth keys can initialize/remain alive without closing each other while same-key calls still share. If the fix changes user-visible reliability, add a concise CHANGELOG.md fixes entry under the active version. Validate with git diff --check and focused Vitest through the repo wrapper; stop and report if the implementation would require an upstream Codex product/API decision instead of OpenClaw client-state isolation.
