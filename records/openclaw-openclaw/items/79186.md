## Summary

Keep open: this is a plausible focused Slack mitigation, but it still needs contributor real-environment proof plus maintainer judgment, and the current diff leaves user-facing docs/help stale with one reconnect rationale inconsistent with current main.

## What This Changes

The PR raises Slack Socket Mode default `clientPingTimeout` to 30s, adds a 60s default `serverPingTimeout`, and updates the receiver-options interop expectation.

## Best Possible Solution

Ship a global Slack Socket Mode timeout-default change only after docs/help/comment alignment, focused interop coverage, redacted real Slack proof, and separate tracking for the broader no-event liveness issue at https://github.com/openclaw/openclaw/issues/77249.

## Reproduction Assessment

No. Source and upstream Slack SDK inspection show the missed-pong timeout/disconnect mechanism and OpenClaw wiring, but the macOS CPU-starvation Slack message-loss scenario has not been reproduced against current main or proven after the patch.

## Solution Assessment

Unclear. Raising ping timeouts is a narrow mitigation for event-loop starvation, but it needs docs/help updates, accurate reconnect rationale, and maintainer judgment because it can delay stale-socket detection and does not solve the no-event zombie path.
