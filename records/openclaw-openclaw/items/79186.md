## Summary

Keep open: this is a focused and plausible Slack reliability mitigation, but it is not merge-ready because it changes Socket Mode defaults for all users without real behavior proof, leaves public docs/help/type comments stale, and includes a misleading reconnect rationale.

## What This Changes

The PR raises Slack Socket Mode `clientPingTimeout` from 15s to 30s, adds a 60s default `serverPingTimeout`, and updates the receiver-options interop expectation.

## Best Possible Solution

Land this only after docs/help/types and the reconnect rationale are aligned, the contributor adds redacted real Slack Socket Mode proof from the affected setup, and maintainers explicitly accept the global timeout-default tradeoff while keeping https://github.com/openclaw/openclaw/issues/77249 as the broader liveness issue.

No high-confidence live reproduction was established in this review. Source and dependency inspection show that these timeouts control Slack SDK disconnect timing, but the after-fix CPU-starvation message-loss improvement has not been proven.

## Solution Assessment

Unclear. Raising the defaults is a plausible mitigation, but existing `channels.slack.socketMode` config already lets affected users opt into these values, so a global default change needs proof, documentation alignment, and maintainer acceptance.
