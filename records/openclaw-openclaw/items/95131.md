## Summary

Keep open: current main still auto-ensures the Codex runtime plugin for official `openai/*` model selection before considering explicit provider/model `agentRuntime.id: "openclaw"`, while docs, doctor guidance, and runtime selection treat that policy as the supported direct-OpenAI opt-out. The remaining work needs maintainer product direction because the correct contract could be a hard Codex requirement, a real OpenClaw-runtime opt-out, or a new explicit opt-out surface.

## What This Changes

The issue asks maintainers to decide whether official `openai/*` model selection may automatically install or re-enable the Codex plugin when an operator has disabled it or configured an OpenClaw runtime opt-out.

## Best Possible Solution

Align setup, doctor, docs, and runtime so explicit OpenClaw runtime policy either reliably prevents Codex auto-install/enable for that model/provider or the product clearly fails and documents that official `openai/*` requires the Codex plugin.

## Reproduction Assessment

Yes, at source level. The current model-selection path still decides to ensure Codex for official OpenAI refs from provider/base URL only, then repairs/enables or installs the Codex plugin without checking explicit `agentRuntime.id: "openclaw"`; no live gateway run was performed in this read-only review.

## Solution Assessment

Unclear until maintainers choose the intended contract. The narrowest maintainable fix is to make model-selection auto-install follow the same provider/model runtime policy used by doctor and runtime selection, or explicitly document and fail early if official OpenAI is intentionally Codex-coupled.
