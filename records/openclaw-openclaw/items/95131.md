## Summary

Keep open. Current `main` and `v2026.6.9` still auto-ensure the Codex plugin for official `openai/*` model selection before checking whether the effective runtime policy is an explicit OpenClaw opt-out, so the reported control-boundary problem remains real and needs a maintainer contract decision before repair.

## What This Changes

The issue asks OpenClaw to stop auto-installing or re-enabling the Codex plugin for official `openai/*` model selection when the operator has disabled Codex or configured an OpenClaw runtime opt-out.

## Best Possible Solution

Confirm the runtime contract, then align setup, doctor, docs, and tests so implicit official OpenAI routes can use Codex by default while explicit OpenClaw/API-key opt-out and disabled-plugin intent are either honored or fail early with clear diagnostics.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: select an official `openai/gpt-5.5` model and the setup predicate still returns true for Codex ensure, which then enables or installs Codex. I did not run a live Telegram/OpenAI OAuth reproduction in this read-only review.

## Solution Assessment

No, the current behavior is not the best solution: runtime selection can honor explicit OpenClaw policy, but setup/repair bypasses that policy and can resurrect Codex. The best fix should make setup and doctor respect the same effective runtime contract maintainers choose for the runtime path.
