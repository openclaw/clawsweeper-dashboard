- `impact:auth-provider`: The affected behavior is model/provider runtime routing and auth-provider setup around official OpenAI and the Codex plugin.

## Summary

Keep open: current main and the latest release still auto-ensure the Codex plugin for official `openai/*` model selection based only on the model route, while adjacent runtime and diagnostics code recognize explicit OpenClaw runtime policy and disabled Codex entries as meaningful operator intent. This is a real provider/runtime contract mismatch that needs maintainer direction rather than conservative cleanup closure.

## What This Changes

Requests clarification or a fix so official OpenAI model selection does not silently install or re-enable the Codex plugin when an operator intended to use the OpenClaw runtime or disable Codex.

## Best Possible Solution

Maintainers should choose the official OpenAI runtime contract and make setup, doctor, docs, and tests match it; if OpenClaw runtime is supported, model-selection ensure should honor provider/model runtime policy and disabled Codex entries before installing or enabling Codex.

## Reproduction Assessment

Yes, at source level: selecting official `openai/gpt-5.5` makes `modelSelectionShouldEnsureCodexPlugin` true without checking `agentRuntime.id` or disabled Codex config, and the ensure helper then repairs/enables or installs Codex. I did not run a live gateway scenario in this read-only review.

## Solution Assessment

Unclear as an implementation path until maintainers decide the intended contract. The narrowest maintainable fix is likely to reuse existing provider/model runtime policy rather than add a new broad config flag, unless maintainers intentionally declare official OpenAI as Codex-only.
