## Summary

Keep open: current main and v2026.6.9 still have a model-selection path that treats official `openai/*` as requiring Codex plugin install/repair without consulting explicit `agentRuntime.id: "openclaw"` or `"pi"`, while runtime policy, docs, and tests already recognize that runtime policy as an opt-out boundary.

## What This Changes

Requests that official `openai/*` model selection respect explicit OpenClaw runtime or Codex-disabled intent instead of auto-installing or re-enabling the Codex plugin.

## Best Possible Solution

Align model-selection Codex install/repair with the same resolved provider/model runtime policy used by execution and diagnostics, or explicitly document and fail early if maintainers decide official OpenAI is hard-coupled to Codex.

## Reproduction Assessment

Yes, from source inspection: selecting an official `openai/*` model reaches the Codex install/repair helper whose predicate does not consult explicit OpenClaw runtime policy. I did not run a live setup scenario in this read-only review.

## Solution Assessment

Unclear until maintainers confirm the contract. The narrow technical fix is to reuse resolved runtime policy in model-selection setup; the alternative is to hard-couple official OpenAI to Codex with clearer docs and fail-early behavior.
