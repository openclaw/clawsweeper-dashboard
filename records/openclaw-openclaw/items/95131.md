## Summary

Keep open: current main and the latest stable release still have a model-selection setup path that treats official `openai/*` as requiring the Codex plugin based on provider/base URL, while explicit `agentRuntime.id: "openclaw"` is documented and honored elsewhere as a runtime-policy boundary. This remains a real product-contract question, not a stale duplicate.

## What This Changes

The issue asks OpenClaw to clarify or fix whether official `openai/*` model selection may auto-install or re-enable the Codex plugin despite a disabled plugin entry or explicit OpenClaw runtime intent.

## Best Possible Solution

Make one runtime-policy contract authoritative across setup, model selection, doctor, docs, and execution: explicit OpenClaw runtime should either reliably prevent Codex plugin install/enable or be rejected early with clear documentation.

## Reproduction Assessment

Yes from source inspection: official `openai/*` model selection still reaches a Codex install/repair helper whose predicate does not consult explicit OpenClaw runtime policy. I did not run a live setup flow because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the intended contract. The narrowest maintainable fix appears to be reusing the existing resolved provider/model runtime policy in model-selection install/repair, rather than adding a separate opt-out knob, if `agentRuntime.id: "openclaw"` is meant to be sufficient.
