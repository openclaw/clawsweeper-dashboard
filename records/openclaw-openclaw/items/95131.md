## Summary

Keep open: current main and v2026.6.9 still route official `openai/*` model-selection setup through the Codex install/repair helper without consulting explicit provider/model `agentRuntime.id: "openclaw"`, while execution, diagnostics, and docs already treat that runtime policy as an opt-out boundary.

## What This Changes

The issue asks OpenClaw to respect explicit OpenClaw/PI runtime or disabled-Codex intent when official OpenAI model selection would otherwise auto-install, repair, or enable the Codex plugin.

## Best Possible Solution

Make model-selection install and repair reuse the same resolved provider/model runtime policy as execution and diagnostics, so explicit OpenClaw routes do not install or re-enable Codex while default Codex routes still do, with aligned docs and regression coverage.

## Reproduction Assessment

Yes, source-reproducible: selecting an official `openai/*` route reaches a Codex install/repair predicate that does not consult explicit OpenClaw runtime policy. I did not run a live stable setup scenario in this read-only review.

## Solution Assessment

Unclear until maintainer decision: the narrow technical fix is to reuse the existing resolved-runtime boundary in model-selection setup, but maintainers must confirm that `agentRuntime.id: "openclaw"` is the intended opt-out for this path.
