## Summary

Keep open: current main still has a source-reproducible mismatch where official `openai/*` model selection can require and enable/install Codex based only on provider/base URL, while current docs and runtime selection tests recognize model-scoped `agentRuntime.id: "openclaw"` as an explicit runtime policy.

## What This Changes

The issue asks OpenClaw to stop official `openai/*` model selection from auto-installing or re-enabling the Codex plugin when the operator has explicitly selected the OpenClaw runtime or disabled Codex, or to define that contract.

## Best Possible Solution

Align Codex install/repair with resolved model runtime policy so explicit OpenClaw/PI opt-out and disabled Codex intent are respected, or explicitly document and fail early if maintainers choose hard coupling.

## Reproduction Assessment

Yes, from source rather than live execution: selecting an official `openai/*` model makes the Codex model-selection predicate true regardless of explicit OpenClaw runtime policy, and the install helper can then enable or install Codex.

## Solution Assessment

Unclear until the maintainer runtime contract is confirmed. If the existing model-scoped `agentRuntime.id: "openclaw"` surface is the intended opt-out, the narrow fix is to derive Codex install/repair from resolved runtime policy and add focused setup/doctor regression tests.
