- `impact:auth-provider`: The issue concerns OpenAI provider routing, auth setup, runtime selection, and automatic Codex plugin activation.

## Summary

Keep open: current main and latest release still have a setup/model-selection path that treats official `openai/*` as requiring the Codex plugin before consulting the resolved provider/model runtime policy, while runtime execution, diagnostics, tests, and docs already recognize explicit `agentRuntime.id: "openclaw"` as an opt-out boundary.

## What This Changes

The issue asks OpenClaw to clarify and enforce whether official `openai/*` model selection may auto-install, repair, or re-enable the Codex plugin when the operator configured OpenClaw runtime policy or disabled the Codex plugin.

## Best Possible Solution

Align model-selection install and repair with the same resolved provider/model runtime policy used by execution and diagnostics, so explicit OpenClaw routes do not install or re-enable Codex while default or explicit Codex routes still do, with docs and regression coverage for both paths.

## Reproduction Assessment

Yes from source inspection: selecting an official `openai/*` model reaches a Codex install/repair helper whose predicate does not consult explicit provider/model `agentRuntime.id: "openclaw"`. I did not run a live stable setup scenario in this read-only review.

## Solution Assessment

Unclear until maintainer decision: reusing the existing resolved-runtime boundary in model-selection setup appears to be the narrowest maintainable fix, but maintainers must confirm whether official OpenAI is hard-coupled to Codex or supports a real opt-out.
