## Summary

Keep open: current main still has a setup/model-selection path that treats official `openai/*` as requiring the Codex runtime plugin without checking explicit `agentRuntime.id: "openclaw"`, while execution, doctor diagnostics, tests, and docs already recognize explicit OpenClaw runtime policy as an opt-out boundary.

## What This Changes

The issue asks maintainers to clarify and fix whether selecting official `openai/*` models should auto-install or re-enable the Codex plugin despite explicit disabled or OpenClaw-runtime opt-out intent.

## Best Possible Solution

Make setup/model-selection reuse the same resolved provider/model runtime policy as execution and doctor, then align docs and tests so explicit OpenClaw routes do not install or re-enable Codex while default Codex routes still do.

## Reproduction Assessment

Yes from source inspection: selecting an official `openai/*` route still reaches a Codex install/repair helper whose predicate does not consult explicit OpenClaw runtime policy. I did not run a live stable setup flow in this read-only review.

## Solution Assessment

Unclear until maintainers confirm the product contract. If the current docs and execution policy are intended, the best fix is to reuse resolved runtime policy in the setup/model-selection install path rather than adding a new global opt-out option.
