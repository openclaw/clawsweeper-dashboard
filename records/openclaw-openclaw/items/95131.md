## Summary

Keep open: current main and the latest release still have a setup/model-selection path that treats official `openai/*` as requiring the Codex plugin without checking explicit `agentRuntime.id: "openclaw"`, while execution, diagnostics, docs, and adjacent tests already recognize that runtime policy as an opt-out boundary.

## What This Changes

The issue asks OpenClaw to stop or clearly define automatic Codex plugin install/enablement when official `openai/*` models are selected and the operator intends to use the OpenClaw runtime.

## Best Possible Solution

Align model-selection Codex install/repair with the resolved provider/model runtime policy: explicit `agentRuntime.id: "openclaw"` should avoid Codex install or re-enablement, while the default official OpenAI route can remain Codex unless maintainers decide to document a hard coupling instead.

## Reproduction Assessment

Yes, from source inspection: selecting an official `openai/*` model reaches a Codex install/repair helper whose predicate does not consult explicit OpenClaw runtime policy. I did not run a live setup scenario in this read-only review.

## Solution Assessment

Unclear until maintainers confirm the contract. The narrow technical fix is to reuse the existing resolved-runtime boundary in model-selection setup, but maintainers could instead choose a hard-coupled Codex contract with clearer docs and fail-early behavior.
