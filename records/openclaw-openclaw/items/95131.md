## Summary

Keep open: current main and v2026.6.8 still have setup/model-selection paths that require or install Codex for official `openai/*` selections based only on provider/base URL, while execution, doctor, and docs already recognize explicit provider/model `agentRuntime.id: "openclaw"` as a non-Codex route. This is a real remaining provider/runtime contract issue, not conservative cleanup-close material.

## What This Changes

The issue asks OpenClaw to stop or clearly document setup/model-selection paths that install or re-enable the Codex plugin for official `openai/*` selections when config indicates a disabled or OpenClaw-runtime opt-out.

## Best Possible Solution

Make setup/model-selection reuse the resolved provider/model runtime policy before installing or repairing Codex; if official OpenAI is intentionally Codex-only, document that and fail early for incompatible OpenClaw opt-out configs.

## Reproduction Assessment

Yes, source-reproducible: current main still routes official `openai/*` model selection into the Codex install helper without consulting explicit OpenClaw runtime policy, while execution and doctor code do consult that policy. I did not run a live stable setup because this was a read-only review.

## Solution Assessment

Unclear until maintainers settle the official OpenAI/Codex contract. If opt-out is intended, reusing existing resolved runtime policy in the model-selection install helper is the narrowest maintainable fix; a new standalone config knob should be a product decision, not the default repair.
