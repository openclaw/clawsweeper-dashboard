- `impact:auth-provider`: The issue is about provider routing, model runtime choice, Codex OAuth/runtime coupling, and plugin enablement during OpenAI model setup.

## Summary

Keep open: current main and v2026.6.9 still auto-ensure Codex from an official `openai/*` model selection before honoring an explicit OpenClaw runtime opt-out, while the runtime policy, doctor diagnostics, and docs already treat provider/model `agentRuntime.id: "openclaw"` as a valid route selector.

## What This Changes

The issue asks OpenClaw to stop auto-installing or re-enabling the Codex plugin for official `openai/*` model setup when the operator has configured an OpenClaw runtime opt-out, or to clarify that no opt-out is supported.

## Best Possible Solution

Make the model-selection Codex ensure predicate share the same provider/model runtime-policy semantics as runtime and doctor: implicit official OpenAI can still ensure Codex, but explicit `agentRuntime.id: "openclaw"` leaves Codex disabled and uninstalled.

## Reproduction Assessment

Yes, at source level: `modelSelectionShouldEnsureCodexPlugin` ignores existing provider/model runtime policy and `ensureRuntimePluginForModelSelection` then enables or installs Codex when the selected model is official `openai/*`. I did not run a live setup flow because this review is read-only.

## Solution Assessment

Yes for the narrow bug: the best fix is to reuse the existing runtime-policy decision before ensuring Codex, not add a new opt-out config surface. If maintainers want to remove the OpenClaw opt-out entirely, that would be a separate product decision and docs/doctor change.
