## Summary

Keep open: current main and latest release still have a setup/model-selection path that treats official `openai/*` as requiring the Codex plugin without consulting explicit provider/model `agentRuntime.id: "openclaw"`, while execution and diagnostics already recognize that runtime policy as an opt-out boundary.

## What This Changes

The issue asks OpenClaw to clarify or fix official `openai/*` model selection so Codex is not auto-installed or re-enabled when the operator explicitly intends the OpenClaw runtime or has disabled the Codex plugin.

## Best Possible Solution

Make model-selection install and repair reuse the resolved provider/model runtime policy so explicit OpenClaw routes do not install or re-enable Codex, while default and explicit Codex routes still do, with aligned docs and regression coverage.

## Reproduction Assessment

Yes. Source inspection shows selected official `openai/*` models still reach a Codex install/repair helper whose predicate ignores explicit OpenClaw runtime policy; I did not run a live stable setup scenario.

## Solution Assessment

Unclear until maintainer decision. The narrow technical fix is to reuse the existing resolved-runtime boundary in model-selection setup, but maintainers must confirm that `agentRuntime.id: "openclaw"` is the intended opt-out for this path.
