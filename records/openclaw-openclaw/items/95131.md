## Summary

Keep open: current main and v2026.6.8 still have a model-selection setup path that treats official `openai/*` as requiring the Codex plugin without consulting explicit provider/model `agentRuntime.id: "openclaw"`, while execution and diagnostics already recognize that runtime policy as an opt-out boundary.

## What This Changes

The issue requests a clear OpenAI/Codex runtime contract and a fix so official `openai/*` model selection does not auto-install, repair, or re-enable the Codex plugin when the operator intended the OpenClaw runtime opt-out.

## Best Possible Solution

Make model-selection install and repair reuse the same resolved provider/model runtime policy as execution and doctor, so explicit OpenClaw routes do not install or re-enable Codex while default Codex routes still do, with aligned docs and regression tests.

## Reproduction Assessment

Yes, from source inspection: selecting an official `openai/*` route reaches a Codex install/repair helper whose predicate does not consult explicit OpenClaw runtime policy. I did not run a live v2026.6.8 setup scenario in this read-only review.

## Solution Assessment

Unclear until maintainer decision: the narrow technical fix is to reuse the existing resolved-runtime boundary in model-selection setup, but maintainers must confirm that `agentRuntime.id: "openclaw"` is the intended opt-out for this path.
