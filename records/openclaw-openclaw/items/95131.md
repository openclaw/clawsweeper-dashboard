- `impact:auth-provider`: The issue concerns OpenAI model routing, runtime selection, and automatic Codex plugin enablement during provider/auth setup.

## Summary

Keep open: current main still has a model-selection install path that treats official `openai/*` selections as requiring the Codex runtime plugin without checking an explicit `agentRuntime.id: "openclaw"` policy, while execution and doctor paths already recognize that policy as an opt-out boundary.

## What This Changes

Request that official `openai/*` model selection either respect an operator's OpenClaw-runtime opt-out before auto-installing or auto-enabling the Codex plugin, or document that Codex is hard-coupled to that route.

## Best Possible Solution

Make setup/model-selection reuse the same resolved provider/model runtime policy as execution and doctor, then align docs and tests so explicit OpenClaw routes do not install or re-enable Codex while default Codex routes still do.

## Reproduction Assessment

Yes from source inspection: selecting an official `openai/*` route still reaches a Codex install/repair helper whose predicate does not consult explicit OpenClaw runtime policy; no live stable run was performed in this read-only review.

## Solution Assessment

The best fix appears to be reusing the existing resolved-runtime boundary in model-selection install/repair rather than adding a new global opt-out, unless maintainers intentionally want a broader policy surface.
