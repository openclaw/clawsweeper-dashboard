## Summary

Current main still appears to derive the Control UI webchat busy badge and composer stop affordance from session-level `canAbort` without requiring a local active send/stream state, so the false busy/abortable state reported from 2026.5.26 remains a valid, narrow UI bug rather than cleanup-close material.

## What This Changes

The issue requests that Control UI webchat return its bottom status badge and composer control to idle after an assistant response completes when no local stream is active.

## Best Possible Solution

Gate the main webchat badge and composer Stop state on authoritative local live state, while preserving session-level abortability only where remote/session abort affordances intentionally need it.

Yes, with medium confidence: the issue gives concrete release/environment steps and current source still has the same stale `canAbort` path, but I did not live-run the Windows Chrome/WSL2 UI scenario during this read-only review.

## Solution Assessment

Yes: the suggested local-live-state gate is the narrowest maintainable fix because it separates visible main composer/badge busy state from broader session abortability instead of changing backend task/session semantics.
