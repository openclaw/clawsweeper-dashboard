## Summary

Keep this open: current main and the latest release still wire the visible webchat busy/Stop UI to session-level abortability, so the reported stale Control UI state is not cleanup-close material.

## What This Changes

The issue asks for Control UI webchat to return the composer button and status badge to idle after a completed assistant response when no local stream is active.

## Best Possible Solution

Separate visible local webchat busy state from session-scoped abortability: the main badge and Stop button should require authoritative local live state or a current active run, while broader session abort affordances stay available where they are intentionally needed.

## Reproduction Assessment

Yes at source level, but not by live run: current code can still render `In progress` and `Stop` from `hasAbortableSessionRun(state)` when local stream/send state is idle unless terminal `runStatus` is present. The report supplies a concrete 2026.5.26 live environment, but this review did not replay it in Windows Chrome/WSL2.

## Solution Assessment

Yes, the best path is a narrow Control UI fix that splits visible local busy state from session-level abortability and adds a regression test for stale active session rows with no local live state. A broader gateway/session rewrite is not needed for the reported UI symptom.
