## Label Justifications

- `P1`: The issue affects real long-running agent/channel workflows where model switches can appear unresponsive or silently strand sessions on fallback behavior.
- `impact:session-state`: The failure depends on accumulated carried-over session context and can leave long-running sessions without clear recovery guidance.
- `impact:auth-provider`: The affected paths include model selection, provider routing, and fallback behavior during live model switches.

## Summary

Keep open: current main and the latest release still have source-backed gaps for oversized carried-over context during model switches, including switch acknowledgements without a target-budget warning and preflight compaction paths that can still miss configured threshold behavior. The open reserve-token PR covers only one slice, so this still needs coordinated maintainer review across model-switch UX, compaction, and fallback behavior.

## What This Changes

The issue asks OpenClaw to surface and recover from oversized carried-over session context when switching models or continuing long-running sessions.

## Best Possible Solution

Add one shared model-switch context-risk path for Control UI, WebChat slash commands, and messaging directives, then align it with compaction and fallback handling so oversized carried-over context is compacted, refused with a recovery hint, or reported with the real cause.

## Reproduction Assessment

Yes, at source level. Current main still accepts and acknowledges model switches without target-budget checks, and preflight compaction still has source-visible paths that can bypass or miscompute the configured threshold behavior.

## Solution Assessment

Unclear for the full automatic behavior because warning thresholds, auto-compaction policy, and fallback-cause surfacing need maintainer judgment. The maintainable direction is a shared model-switch risk check plus coordinated compaction/fallback fixes, not isolated one-off messages.
