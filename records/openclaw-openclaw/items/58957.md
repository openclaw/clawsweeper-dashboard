## Label Justifications

- `P1`: The issue affects real long-running chat and heartbeat workflows where model switching or fallback can leave users with ambiguous failures or stranded sessions.
- `impact:session-state`: The failure depends on carried-over transcript/context size, compaction state, session mapping, and whether the session can safely continue after a model switch.
- `impact:auth-provider`: The reported behavior crosses provider/model selection, fallback routing, Codex/OpenAI runtime handling, and target model context budgets.

## Summary

Keep open: current main has improved overflow diagnostics, auto-compaction/retry, explicit recovery copy, and context visibility, but it still does not fully solve the reported model-switch UX because `/model` and `sessions.patch` apply the new model without a target-window/session-size warning and related heartbeat/fallback behavior still needs maintainer policy.

## What This Changes

Issue requests clear context-overflow messaging, automatic or prompted compaction, safer model-switch handling, and context-size observability when long sessions carry too much history into another model.

## Best Possible Solution

Add a shared model-switch context-risk warning and recovery path for WebChat, messaging directives, and session patch callers, then let maintainers explicitly choose any automatic compaction, heartbeat, or overflow-fallback policy.

## Reproduction Assessment

Yes at source level for the remaining behavior: current switch paths apply and acknowledge a new model without comparing current session context to the target model budget, while live reports and duplicates show the same carried-context failure family; I did not run a live gateway reproduction.

## Solution Assessment

No, current main is only a partial solution: overflow recovery and context visibility improved, but switch-time warning, heartbeat policy, and automatic compaction/fallback choices remain unresolved maintainer decisions.
