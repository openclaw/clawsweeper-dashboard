## Label Justifications

- `P1`: The issue describes real long-running agent/channel sessions becoming unresponsive or silently moving to fallback behavior during model switches.
- `impact:session-state`: The failure depends on accumulated session context and can strand long-running session state without clear recovery guidance.
- `impact:auth-provider`: The affected path includes model selection, provider routing, and fallback behavior during live model switches.

## Summary

Current main still has source-backed gaps for the reported oversized-context model-switch failure: model switch paths accept or acknowledge the new model without a target-budget warning, and preflight compaction can still return before threshold evaluation for heartbeat or fresh-token sessions. The open reserve-token PR covers only one compaction-threshold slice, so this issue still needs maintainer coordination across model switch UX, compaction, and fallback behavior.

## What This Changes

The issue asks OpenClaw to make oversized carried-over session context visible and recoverable when switching or continuing models, ideally through clearer errors, compaction, warning, and observability.

## Best Possible Solution

Add one shared model-switch context-risk path for Control UI, WebChat slash commands, and messaging directives, then align it with compaction/fallback handling so oversized carried-over context is compacted, refused with a recovery hint, or reported with the real cause.

## Reproduction Assessment

Yes, at source level. Current main still accepts model switches without target-budget checks, and runPreflightCompactionIfNeeded still bypasses token-threshold evaluation for heartbeat and fresh-token sessions.

## Solution Assessment

Unclear for the full automatic behavior because warning thresholds, auto-compaction policy, and fallback-cause surfacing need maintainer judgment. The maintainable direction is a shared model-switch risk check plus coordinated compaction/fallback fixes, not separate one-off messages.
