## Label Justifications

- `P1`: The issue affects real long-running agent/channel workflows where model switches or fallbacks can become ambiguous or silently move to a different model path.
- `impact:session-state`: The reported failure depends on accumulated session history, token totals, compaction behavior, and long-running transcript state.

## Summary

Keep open. Current main has partially improved observability and fresh-token preflight behavior, but it still lacks a shared target-context risk check for model switches, still bypasses preflight compaction for heartbeat runs, and the effective-reserve threshold fix is still an open narrower PR rather than merged main.

## What This Changes

The issue requests clear user-visible handling and recovery when model switching or continuing a long-running session carries too much context for the selected model/provider path.

## Best Possible Solution

Add a shared model-switch context-risk path used by Control UI, WebChat slash commands, and messaging directives, then coordinate it with compaction/fallback handling so oversized carried-over context is compacted, refused with a specific recovery hint, or reported with the real cause.

## Reproduction Assessment

Yes, at source level. Current `main` still has model-switch entry points that accept the new model without a target-budget warning, and heartbeat sessions still bypass preflight compaction before threshold evaluation.

## Solution Assessment

Unclear for the full automatic behavior because warning thresholds, auto-compaction timing, and fallback-cause surfacing need maintainer policy. The maintainable direction is one shared risk/recovery path rather than separate one-off messages per UI or channel.
