## Label Justifications

- `P1`: The issue describes a real long-running agent/channel workflow where model switches can silently strand or unclearly fail active sessions.
- `impact:session-state`: The failure depends on carried transcript size, token accounting, compaction state, and whether the existing session can safely continue.
- `impact:auth-provider`: The symptom involves model choice, provider fallback, Codex/OpenAI routing, and provider/account limits during model switches.

## Summary

Keep open. Current main has useful partial mitigations for context overflow, fallback visibility, and context-budget display, but the model-switch entry points still apply and acknowledge the selected model without a carried-context risk warning, recovery choice, or maintainer-approved compaction/fallback policy.

## What This Changes

The issue requests clearer user-visible handling when a long-running session carries too much context across a model switch, plus optional compaction, fallback, and context-size observability improvements.

## Best Possible Solution

Add a shared model-switch context-risk assessment used by sessions.patch, WebChat, and messaging directives; compare the freshest session usage with the selected model's effective budget and have maintainers choose the default warning, compaction, and fallback behavior.

## Reproduction Assessment

Yes, source-level. Current WebChat, messaging directive, and sessions.patch model-switch paths still apply and acknowledge a new model without comparing carried session usage to the selected model budget; the original live Telegram/Codex failure was not rerun.

## Solution Assessment

No, current main is only a partial solution. The fallback and context-window work reduce silent failure modes, but the best fix still needs a shared switch-time risk result and a maintainer decision on warnings, auto-compaction, and fallback policy.
