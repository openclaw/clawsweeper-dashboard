## Label Justifications

- `P1`: The issue describes a real long-running agent/channel workflow where model switches can silently strand or unclearly fail active sessions.
- `impact:session-state`: The failure depends on carried-over transcript size, session token accounting, compaction state, and whether the existing session can safely continue.
- `impact:auth-provider`: The symptom involves model choice, provider fallback, Codex/OpenAI routing, and provider/account limits during model switches.

## Summary

Keep open. Current main has a useful partial fix for context-window metadata, but the central model-switch UX still persists and acknowledges the requested model without returning a carried-context risk warning or recovery choice.

## What This Changes

The issue requests clearer user-visible handling when a long session is switched or continued on a model whose effective context budget cannot safely carry the existing session history.

## Best Possible Solution

Add a shared model-switch context-risk assessment used by sessions.patch, WebChat, and messaging directives, then have maintainers choose whether warnings, prompted compaction, automatic compaction, or fallback-on-overflow should be default behavior.

## Reproduction Assessment

Yes, source-level. Current WebChat, directive, and sessions.patch model-switch paths apply and acknowledge a new model without comparing current session usage to the selected model budget; the original live Telegram/Codex failure was not rerun.

## Solution Assessment

No. Current main's context-cap work is a useful partial fix, but the best solution needs a shared switch-time risk result and a maintainer decision on warning thresholds, auto-compaction, and Codex/native compaction ownership.
