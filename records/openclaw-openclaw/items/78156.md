## Summary

Close: current main and the shipped cron parser already resolve offset-less one-shot `--at` values in the requested IANA timezone, including the reported America/Chicago case and DST boundary handling.

## What This Changes

The issue requests fixing one-shot `openclaw cron add --at ... --tz ...` scheduling so offset-less datetimes are stored as the correct UTC instant.

## Best Possible Solution

Keep the shipped parser and tests; users still seeing this should update to `v2026.3.24` or newer and file a new report with the stored job schedule if a specific timezone still miscomputes.

## Reproduction Assessment

No: the reported current-main path does not reproduce. Source inspection plus a read-only parser check show the reported `2026-05-06T10:00` / `America/Chicago` case resolves to `2026-05-06T15:00:00.000Z`.

## Solution Assessment

Yes: the existing two-pass `Intl.DateTimeFormat` parts validation is a narrow implementation that preserves explicit-offset behavior and rejects nonexistent DST wall-clock times without adding a dependency or new config.
