## Summary

Keep open: current main and the latest release still pass explicit `sandbox.docker.env` through the sensitive-name sanitizer, so the reported partial injection is valid and needs a maintainer/security decision before changing Docker credential exposure.

## What This Changes

The issue requests consistent injection of configured sandbox Docker env keys, including secret-like names, into fresh Docker containers when allowed by policy.

## Best Possible Solution

Define the sandbox credential exposure contract, then update Docker create/setup/exec handling, docs, and focused tests so explicit sandbox env works consistently without unintended metadata leakage.

## Reproduction Assessment

Yes from source inspection: configured Docker env reaches `buildSandboxCreateArgs`, then the sanitizer blocks the reported `*_SECRET`, `*_PASSWORD`, and `*_API_KEY` names before Docker `--env` flags are emitted. I did not run a live Docker container in this read-only review.

## Solution Assessment

No for a blind sanitizer bypass; the narrow fix still needs a maintainer-approved split between explicit configured sandbox env, inherited host env sanitization, setup-time env, and per-exec env.
