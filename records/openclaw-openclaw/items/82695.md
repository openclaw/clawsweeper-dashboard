## Summary

Keep open. Current main and the latest release still route explicit Docker sandbox env through a credential-name blocklist, while docs tell users to put sandboxed skill API keys in Docker sandbox env; an open PR already targets the same bug but is not landed.

## What This Changes

The issue reports credential-shaped keys from `agents.defaults.sandbox.docker.env` being omitted from fresh Docker sandbox containers.

## Best Possible Solution

Review, refresh, or replace https://github.com/openclaw/openclaw/pull/40290 with a narrow fix that allows explicit Docker config env keys while preserving dangerous runtime blocks, sanitizer options, tests, docs, and redacted Docker proof.

## Reproduction Assessment

Yes at source level: `buildSandboxCreateArgs` sanitizes explicit Docker env and `sanitizeEnvVars` blocks the reported credential-shaped names. I did not establish a live Docker container reproduction in this read-only review.

## Solution Assessment

Unclear until the existing open PR is refreshed and reviewed. The maintainable direction is an explicit Docker config env exception with hard dangerous-name blocks, not broad host env inheritance or an unscoped sanitizer bypass.
