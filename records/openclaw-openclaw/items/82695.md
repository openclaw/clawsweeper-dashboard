## Summary

Keep open: current main still drops the reported secret-like names from Docker container creation because explicit `sandbox.docker.env` is passed through the sensitive-name sanitizer, while docs tell users to use that config for sandboxed skill API keys. The remaining decision is how to fix this without accidentally increasing Docker metadata exposure for credentials.

## What This Changes

The issue reports that configured Docker sandbox environment keys with secret-like names are omitted from fresh Docker sandbox containers.

## Best Possible Solution

Define the intended credential exposure contract for configured sandbox env, then update Docker create/setup/exec handling and docs/tests so explicit sandbox keys work consistently without unintended metadata leakage.

## Reproduction Assessment

Yes from source inspection: `sandbox.docker.env` is accepted and documented, but `buildSandboxCreateArgs` sanitizes it before Docker `--env` flags and the sanitizer blocks the reported names. I did not run a live Docker container in this read-only review.

## Solution Assessment

No, blindly bypassing the sanitizer is not obviously the best fix because Docker create-time env is visible through container metadata. The safer direction is a maintainer-approved split between explicit configured sandbox env, inherited host env sanitization, setup-time env, and per-exec env.
