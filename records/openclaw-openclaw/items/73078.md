## Summary

Keep open. Current main still treats `sandbox.mode: "non-main"` as a per-agent main-session comparison, so a dynamically created peer agent's primary DM session can remain unsandboxed; docs now warn about the session-based meaning, but no schema-backed main-agent-only policy exists.

## What This Changes

Request a clearer sandbox policy for the use case where only the `main` agent bypasses sandboxing while dynamic or peer agents remain sandboxed, plus matching docs or aliases for `non-main` semantics.

## Best Possible Solution

Add a backward-compatible, schema-backed main-agent-only sandbox policy or equivalent selector, then update `sandbox explain`, generated schema/docs, and focused sandbox/routing tests while preserving existing `non-main` behavior unless maintainers choose a migration.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: with defaults set to `sandbox.mode: "non-main"`, `agent:feishu-ou_peer:main` resolves to agent `feishu-ou_peer`, compares against that same agent's main session key, and returns `sandboxed: false`.

## Solution Assessment

Unclear until maintainers choose the public security contract. An additive mode looks safer than redefining `non-main`, while the current docs-only clarification does not provide the requested single-config main-agent-only isolation policy.
