## Summary

Keep open. Current main still interprets `sandbox.mode: "non-main"` by comparing the current session with that same resolved agent's main session key, the public schema lacks an agent-scoped mode, and the docs warning does not make the main-agent-only isolation policy expressible.

## What This Changes

The issue requests an agent-scoped sandbox policy or clearer config semantics so only the `main` agent bypasses sandbox while dynamic peer agents remain sandboxed.

## Best Possible Solution

Add a backward-compatible, schema-backed main-agent-only sandbox policy or equivalent selector, then update `sandbox explain`, generated schema/docs, and focused sandbox plus dynamic-agent tests while preserving existing `non-main` semantics unless maintainers choose a migration.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with defaults set to `sandbox.mode: "non-main"`, `agent:feishu-ou_peer:main` resolves to agent `feishu-ou_peer`, compares against that same agent's main session key, and returns unsandboxed.

## Solution Assessment

Unclear until maintainers choose the public security and compatibility contract. An additive mode or selector looks safer than redefining `non-main`, while documentation alone does not remove the reported isolation gap.
