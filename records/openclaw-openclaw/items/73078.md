## Summary

Keep open. Current main still exposes `non-main` as a per-session policy with no schema-backed main-agent-only mode; there is a newer direct-chat runtime policy key that may mitigate some channel DM execution paths, but it does not resolve the public sandbox mode semantics or the requested operator contract.

## What This Changes

Requests a sandbox configuration mode or documentation path so only the main agent bypasses sandbox while dynamic peer agents' main sessions remain sandboxed.

## Best Possible Solution

Add a backward-compatible, schema-backed main-agent-only sandbox policy or equivalent selector, update `sandbox explain` and public docs, and preserve existing `non-main` session semantics unless maintainers explicitly choose a migration.

## Reproduction Assessment

Yes for the reported config semantics. With `agents.defaults.sandbox.mode: "non-main"` and session key `agent:feishu-ou_peer:main`, current source resolves the agent as `feishu-ou_peer`, resolves that same agent's main session key, and returns `sandboxed: false`.

## Solution Assessment

Unclear until maintainers choose the public security and compatibility contract. An additive mode or selector is safer than redefining `non-main`, while docs-only clarification leaves the main-agent-only operator intent unexpressed.
