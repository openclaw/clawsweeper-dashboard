## Summary

Keep open. Current main still implements `sandbox.mode: "non-main"` as a session comparison against each resolved agent's own main session, the schema and runtime types still expose only `off | non-main | all`, and dynamic Feishu agents still inherit defaults while direct DMs default to their per-agent main session. Because the remaining change affects sandbox isolation semantics for untrusted peer agents, this needs maintainer/secops product-contract review rather than cleanup closure.

## What This Changes

The issue requests a clearer or new sandbox policy so only the trusted main agent can bypass sandboxing while dynamic peer agents remain sandboxed without verbose per-agent overrides.

## Best Possible Solution

Add a backward-compatible, schema-backed way to express the main-agent-only isolation policy, likely an additive mode or equivalent selector that preserves existing `non-main` behavior. Update sandbox explain output, generated config docs, sandbox docs, multi-agent sandbox docs, and focused tests for dynamic peer-agent main sessions.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with `agents.defaults.sandbox.mode: "non-main"` and a dynamic peer session such as `agent:feishu-ou_peer:main`, current main resolves `feishu-ou_peer` as the agent, resolves that agent's own main session key, and returns `sandboxed: false`.

## Solution Assessment

Unclear until maintainers choose the security/product contract. An additive schema-backed mode appears safer than redefining `non-main`, while documentation-only clarification does not remove the misconfiguration hazard described here.

- **[medium] Dynamic peer main sessions bypass non-main sandboxing:** `src/agents/sandbox/runtime-status.ts:23`
