## Summary

Keep open. Current main still implements `sandbox.mode: "non-main"` as a comparison against the resolved agent's own main session key, and the schema still lacks a concise main-agent-only policy. The newer docs warning clarifies part of the surprise, but it does not remove the security-sensitive configuration gap for dynamic peer agents inheriting defaults.

## What This Changes

Request a clearer sandbox policy for the main-agent-only use case because `agents.defaults.sandbox.mode: "non-main"` is evaluated per session, not per agent, causing dynamic peer agents' main sessions to bypass sandbox unless overridden.

## Best Possible Solution

Add a backward-compatible, schema-backed way to express the main-agent-only policy, likely an additive sandbox mode or equivalent selector. Preserve current `non-main` behavior unless maintainers explicitly choose a migration, then update `sandbox explain`, generated config schema/docs, and focused tests for dynamic peer-agent main sessions.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with defaults set to `sandbox.mode: "non-main"`, `agent:feishu-ou_peer:main` resolves to agent `feishu-ou_peer`, compares against that same agent's main session key, and returns `sandboxed: false`.

## Solution Assessment

Unclear until maintainers choose the security and compatibility contract. An additive mode appears safer than redefining `non-main`; the current documentation-only clarification does not remove the reported misconfiguration hazard.
