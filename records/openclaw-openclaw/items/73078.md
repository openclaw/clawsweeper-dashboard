## Summary

Keep open. Current `main` still implements `non-main` as a per-session comparison against the resolved agent's own main session; docs now clarify that behavior and show per-agent overrides, but there is still no main-agent-only sandbox policy.

## What This Changes

The issue requests clearer or new sandbox policy semantics so only the main agent can bypass sandbox while dynamically created peer agents remain sandboxed.

## Best Possible Solution

Add a backward-compatible, schema-backed main-agent-only sandbox policy or equivalent selector, update `sandbox explain` and public docs, and preserve existing `non-main` behavior unless maintainers choose a migration.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with `agents.defaults.sandbox.mode: "non-main"` and session `agent:feishu-ou_peer:main`, runtime resolves the agent as `feishu-ou_peer`, computes that same agent's main key, and returns `sandboxed: false`.

## Solution Assessment

Unclear until maintainers choose the public security and compatibility contract. An additive main-agent-only mode appears safer than redefining `non-main`, while documentation-only clarification leaves the reported isolation policy inexpressible without per-agent overrides.
