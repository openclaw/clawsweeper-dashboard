## Summary

Keep open. Current main still implements `non-main` as a comparison against the resolved agent's own main session, so docs now clarify the session-scoped behavior but the requested main-agent-only sandbox policy remains unimplemented and needs an explicit security/compatibility decision.

## What This Changes

Request a sandbox mode, alias, or documentation path that makes main-agent-only host bypass expressible without verbose per-agent overrides.

## Best Possible Solution

Add a backward-compatible, schema-backed main-agent-only sandbox policy or equivalent selector, update `sandbox explain` and public docs, and preserve existing `non-main` semantics unless maintainers explicitly choose a migration.

## Reproduction Assessment

Yes, as a source-level reproduction. With `agents.defaults.sandbox.mode: "non-main"` and `agent:feishu-ou_peer:main`, current main resolves the agent as `feishu-ou_peer`, computes that same agent's main key, and returns `sandboxed: false`.

## Solution Assessment

Unclear until maintainers choose the public security and compatibility contract. An additive main-agent-only mode appears safer than redefining `non-main`, while documentation-only clarification does not remove the reported misconfiguration hazard.
