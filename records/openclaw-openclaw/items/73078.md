## Summary

Keep open. Current main still implements `non-main` as a comparison against each resolved agent's own main session, so dynamic peer-agent primary sessions can inherit the default and run unsandboxed; docs clarify the pitfall and per-agent override, but the requested compact main-agent-only policy is still missing.

## What This Changes

Request a sandbox policy or docs/config path that lets only the `main` agent bypass sandboxing while dynamic peer agents stay sandboxed on their primary sessions.

## Best Possible Solution

Add a backward-compatible, schema-backed main-agent-only sandbox selector or equivalent policy, wire it through runtime status and `sandbox explain`, and document it while preserving existing `non-main` behavior unless maintainers choose a migration.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: with `agents.defaults.sandbox.mode: "non-main"` and session `agent:feishu-ou_peer:main`, the resolved agent is `feishu-ou_peer`, its resolved main key is the same session, and `shouldSandboxSession` returns `false`.

## Solution Assessment

Unclear until maintainers choose the public security and compatibility contract. An additive mode looks safer than redefining `non-main`, while documentation-only clarification leaves the reported misconfiguration hazard in place.
