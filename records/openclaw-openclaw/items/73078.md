## Summary

Keep open. Current main and the latest release still define `non-main` as a session-key comparison against each agent's own main session, so the requested main-agent-only sandbox policy is not implemented and needs a security/compatibility decision.

## What This Changes

Request a clearer or new sandbox mode so the global `main` agent can run unsandboxed while dynamic peer agents remain sandboxed on their primary sessions.

## Best Possible Solution

Add a backward-compatible, schema-backed main-agent-only sandbox policy or equivalent selector, update `sandbox explain` and public docs, and preserve existing `non-main` session semantics unless security owners choose a migration.

## Reproduction Assessment

Yes, as a source-level reproduction. With `agents.defaults.sandbox.mode: "non-main"` and `agent:feishu-ou_peer:main`, current main resolves the agent as `feishu-ou_peer`, computes that same agent's main key, and returns `sandboxed: false`.

## Solution Assessment

Unclear until maintainers choose the public security and compatibility contract. An additive main-agent-only mode appears safer than redefining `non-main`, while documentation-only clarification leaves the reported main-agent-only policy unexpressed.
