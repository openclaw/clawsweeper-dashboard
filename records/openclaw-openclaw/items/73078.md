## Summary

Keep open. Current main still implements `sandbox.mode: "non-main"` as a comparison against the resolved agent's own main session, so the requested main-agent-only policy is not available; the docs clarification helps but does not remove the security-sensitive configuration gap.

## What This Changes

The issue requests clearer or stricter sandbox semantics so inherited `sandbox.mode: "non-main"` does not leave dynamic peer agents' main sessions unsandboxed.

## Best Possible Solution

Add a backward-compatible, schema-backed way to express the main-agent-only policy, likely an additive sandbox mode or equivalent selector. Preserve current `non-main` behavior unless maintainers explicitly choose a migration, then update `sandbox explain`, generated schema/docs, and focused tests for dynamic peer-agent main sessions.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with defaults set to `sandbox.mode: "non-main"`, `agent:feishu-ou_peer:main` resolves to agent `feishu-ou_peer`, compares against that same agent's main session key, and returns `sandboxed: false`.

## Solution Assessment

Unclear until maintainers and secops choose the public compatibility contract. An additive mode looks safer than redefining `non-main`, while documentation alone does not remove the reported misconfiguration hazard.
