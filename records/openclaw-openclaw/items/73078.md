## Summary

Keep open. Current main still implements `sandbox.mode: "non-main"` as a session-key comparison against the resolved agent's own main session, the schema still has no `main-agent-only` mode or alias, and the docs only warn about the pitfall. Because this affects sandbox isolation for dynamic peer agents, the remaining decision should be made by maintainers/secops before implementation.

## What This Changes

Request a sandbox policy, schema, or documentation change so only the main agent can bypass sandboxing while dynamic peer agents remain sandboxed by default.

## Best Possible Solution

Add a backward-compatible, schema-backed way to express this policy, likely an additive `main-agent-only` mode or equivalent default-agent-aware selector. Preserve existing `non-main` semantics, update `sandbox explain`, generated config docs, sandbox docs, and focused tests for dynamic peer-agent main sessions.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with `agents.defaults.sandbox.mode: "non-main"` and a dynamic peer session like `agent:feishu-ou_peer:main`, the runtime resolves `feishu-ou_peer` as the agent, resolves that agent's own main session key, and returns `sandboxed: false`.

## Solution Assessment

Unclear until maintainers choose the security/product contract. An additive mode appears safer than changing `non-main`, while documentation-only clarification does not remove the reported misconfiguration hazard.
