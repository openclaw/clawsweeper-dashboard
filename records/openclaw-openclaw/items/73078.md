## Summary

Keep open. Current main still implements `sandbox.mode: "non-main"` as a per-session comparison against the resolved agent's own main session, the config schema still has no main-agent-only mode or alias, and the docs now warn about the pitfall without solving the reported dynamic-peer isolation use case. Because this is sandbox/security-sensitive, maintainer and secops product judgment should choose the contract before a fix is attempted.

## What This Changes

Request a sandbox policy or documentation/schema change so only the main agent can bypass sandboxing while dynamic peer agents remain sandboxed by default.

## Best Possible Solution

Add a backward-compatible, schema-backed way to express this policy, likely an additive `main-agent-only` mode or equivalent default-agent-aware selector. The implementation should preserve existing `non-main` semantics, update `sandbox explain`, generated config docs, sandbox docs, and add focused tests for dynamic peer-agent main sessions.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with `agents.defaults.sandbox.mode: "non-main"` and a dynamic peer session like `agent:feishu-ou_peer:main`, the runtime resolves `feishu-ou_peer` as the agent, resolves that agent's own main session key, and returns `sandboxed: false`.

## Solution Assessment

Unclear until maintainers choose the security/product contract. An additive mode appears safer than changing `non-main`, while docs-only clarification does not remove the verbose and easy-to-misconfigure workaround for untrusted dynamic peers.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because plausible multi-user dynamic-agent configurations can leave peer-agent main sessions outside the sandbox under `non-main` defaults.

Concerns:

- **[medium] Dynamic peer main sessions bypass non-main sandboxing:** `src/agents/sandbox/runtime-status.ts:23`
  - body: Under `agents.defaults.sandbox.mode: "non-main"`, the runtime compares the current session to that same agent's resolved main session key. Dynamic peer agents that inherit defaults and route direct DMs to their main session can therefore run outside the sandbox unless each agent is explicitly overridden.
