## Summary

Keep open. Current main still treats `sandbox.mode: "non-main"` as a per-agent main-session comparison, the public schema still has no main-agent-only mode, and the docs warning clarifies the surprise without closing the dynamic peer-agent isolation gap.

## What This Changes

The issue requests clearer or safer sandbox semantics so the trusted `main` agent can bypass sandboxing while dynamic peer agents remain sandboxed on their primary sessions.

## Best Possible Solution

Add a backward-compatible, schema-backed main-agent-only sandbox policy or equivalent selector, then update `sandbox explain`, generated schema/docs, and focused tests while preserving existing `non-main` behavior unless maintainers choose a migration.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: with defaults set to `sandbox.mode: "non-main"`, `agent:feishu-ou_peer:main` resolves to agent `feishu-ou_peer`, compares against that same agent's main session key, and returns `sandboxed: false`.

## Solution Assessment

Unclear until maintainers choose the public security and compatibility contract. An additive mode appears safer than redefining `non-main`, and a documentation-only clarification does not remove the reported misconfiguration hazard.
