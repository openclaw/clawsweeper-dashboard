## Summary

Keep open. Current main still implements `non-main` as a comparison against the resolved agent's own main session key, so a dynamically created peer agent's primary DM can remain unsandboxed; docs now clarify the session-scoped semantics, but the requested main-agent-only policy still needs maintainer/secops product judgment.

## What This Changes

Request a sandbox policy or documentation path that lets only the `main` agent bypass sandboxing while dynamic peer agents remain sandboxed on their main sessions.

## Best Possible Solution

Add a backward-compatible, schema-backed main-agent-only sandbox policy or equivalent selector, update `sandbox explain` and public docs, and preserve existing `non-main` semantics unless maintainers explicitly choose a migration.

## Reproduction Assessment

Yes, as a source-level reproduction. With `agents.defaults.sandbox.mode: "non-main"` and session `agent:feishu-ou_peer:main`, current main resolves the agent as `feishu-ou_peer`, computes that same agent's main key, and returns `sandboxed: false`.

## Solution Assessment

Unclear until maintainers choose the public security and compatibility contract. An additive main-agent-only mode appears safer than redefining `non-main`, while documentation-only clarification does not remove the reported misconfiguration hazard.
