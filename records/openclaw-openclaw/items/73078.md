## Summary

Keep open. Current main still implements `sandbox.mode: "non-main"` as a session-key comparison against the resolved agent's own main session, so a dynamic peer agent's `agent:<peer>:main` path remains unsandboxed unless that agent has an explicit override. The existing docs warning helps, but the runtime/schema still lack the requested concise main-agent-only policy and the remaining behavior is security-sensitive.

## What This Changes

The issue requests clearer sandbox mode semantics or a new mode so the trusted main agent can bypass sandboxing while dynamic or peer agents' main sessions remain sandboxed without verbose per-agent overrides.

## Best Possible Solution

Add a backward-compatible, schema-backed way to express the main-agent-only policy, likely an additive mode or equivalent selector. Preserve current `non-main` behavior unless maintainers explicitly choose a migration, update `sandbox explain`, generated config schema/docs, and add focused tests for dynamic peer-agent main sessions.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with defaults set to `sandbox.mode: "non-main"`, `agent:feishu-ou_peer:main` resolves to agent `feishu-ou_peer`, compares against that agent's own main session key, and returns `sandboxed: false`.

## Solution Assessment

Unclear until maintainers choose the security/product contract. An additive mode appears safer than redefining `non-main`, while the current documentation-only clarification does not remove the reported misconfiguration hazard.
