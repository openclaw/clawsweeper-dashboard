## Summary

Keep open. Current main still treats `sandbox.mode: "non-main"` as a per-agent main-session comparison, so a dynamic peer agent's own `agent:<peer>:main` session remains direct unless that agent gets an explicit override. The docs now warn about the pitfall, but the schema and runtime still lack a concise main-agent-only policy, and the security-sensitive semantics need maintainer/secops judgment.

## What This Changes

The issue requests clearer sandbox mode semantics or a new mode so the trusted main agent can bypass sandboxing while dynamic or peer agents' main sessions remain sandboxed without verbose per-agent overrides.

## Best Possible Solution

Add a backward-compatible, schema-backed way to express the main-agent-only policy, likely an additive mode or equivalent selector. Preserve current `non-main` behavior, update `sandbox explain`, config docs/generated schema docs, and add focused tests for dynamic peer-agent main sessions.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with defaults set to `sandbox.mode: "non-main"`, a session key like `agent:feishu-ou_peer:main` resolves to agent `feishu-ou_peer`, compares against that agent's own main session key, and returns `sandboxed: false`.

## Solution Assessment

Unclear until maintainers choose the security/product contract. An additive mode appears safer than redefining `non-main`, while documentation-only clarification does not remove the reported misconfiguration hazard.
