## Summary

Keep open. Current main still implements `sandbox.mode: "non-main"` as a per-session comparison against the resolved agent's own main session key, and the config contract still has no concise main-agent-only sandbox policy; the newer docs warning helps but does not satisfy the reported isolation use case for dynamic peer agents.

## What This Changes

The issue requests a clearer sandbox policy, such as a main-agent-only mode or explicit renamed/docs semantics, so dynamic peer agents can remain sandboxed while the trusted main agent runs on the host.

## Best Possible Solution

Add a backward-compatible, schema-backed way to express the main-agent-only policy, likely an additive sandbox mode or equivalent selector, while preserving existing `non-main` semantics unless maintainers explicitly approve a migration. Update runtime status, config schema/generated metadata, sandbox docs, `sandbox explain`, and focused tests for dynamic peer-agent main sessions.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with defaults set to `sandbox.mode: "non-main"`, a dynamic peer direct route can resolve to `agent:<peer-agent>:main`, and the runtime compares that key to the same peer agent's main session key, producing `sandboxed: false`.

## Solution Assessment

Unclear until maintainers choose the public security and compatibility contract. An additive main-agent-only mode appears safer than changing `non-main`; the current documentation-only clarification does not remove the reported misconfiguration hazard.
