## Summary

Keep open. Current main still implements `sandbox.mode: "non-main"` as a per-agent main-session comparison, the schema still lacks a main-agent-only mode or alias, and the current docs clarify the naming surprise without closing the dynamic peer-agent isolation gap.

## What This Changes

The issue requests clearer or safer sandbox semantics so the trusted `main` agent can bypass sandboxing while dynamic peer agents remain sandboxed on their primary sessions.

## Best Possible Solution

Add a backward-compatible, schema-backed main-agent-only sandbox policy or equivalent selector, then update `sandbox explain`, generated schema/docs, and focused tests while preserving existing `non-main` behavior unless maintainers choose a migration.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection is enough: with defaults set to `sandbox.mode: "non-main"`, `agent:feishu-ou_peer:main` resolves to agent `feishu-ou_peer`, compares against that same agent's main session key, and returns `sandboxed: false`.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the public security and compatibility contract. An additive mode appears safer than redefining `non-main`, and a documentation-only clarification does not remove the reported misconfiguration hazard.
