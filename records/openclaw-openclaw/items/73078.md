## Summary

Keep open. Current main still evaluates `non-main` against the resolved agent's own main session key, so inherited dynamic peer agents can leave their primary DM session unsandboxed; docs clarify the session-scoped semantics, but the requested main-agent-only policy is still missing and needs maintainer/security judgment.

## What This Changes

Request a clearer or new sandbox policy so the main admin agent can run on host while dynamic peer agents remain sandboxed by default.

## Best Possible Solution

Add a backward-compatible, schema-backed main-agent-only sandbox policy or equivalent selector, update `sandbox explain` and public docs, and preserve existing `non-main` semantics unless maintainers choose an explicit migration.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with `agents.defaults.sandbox.mode: "non-main"` and session `agent:feishu-ou_peer:main`, the runtime resolves the peer agent, computes that same peer agent's main key, and returns `sandboxed: false`.

## Solution Assessment

Unclear until maintainers choose the public security and compatibility contract. An additive main-agent-only mode or explicit selector appears safer than redefining `non-main`, while docs-only clarification leaves the reported isolation policy unexpressed.
