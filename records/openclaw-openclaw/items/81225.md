## Summary

Keep open: the reported Telegram/plugin-bound topic bug is source-backed and the PR direction looks appropriate, but the branch currently fails test typechecking because a new fixture uses an invalid `targetKind`.

## What This Changes

The PR marks plugin-owned runtime conversation routes as explicit binding routes, bypasses Telegram mention gating for those bound group routes, and lets plugin-bound claims run under `message_tool_only` delivery with regression coverage.

## Best Possible Solution

Keep the narrow runtime-binding route and dispatch behavior, but fix the invalid test fixture target kind before merge and re-run the focused type/test gates.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main leaves plugin-owned runtime routes unmarked and then skips plugin-bound claims when `message_tool_only` sets `suppressDelivery`. I did not rerun a live Telegram topic, but the PR body includes redacted after-fix runtime logs.

## Solution Assessment

The product approach appears to be the narrow maintainable fix, because it marks the existing explicit binding route and distinguishes `message_tool_only` from `sendPolicy: deny`. The current patch is not mergeable until the test fixture uses a valid `BindingTargetKind`.
