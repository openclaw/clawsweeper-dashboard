## Summary

Keep open; the bug and fix direction are valid with sufficient live log proof, but the current patch still regresses named-account Telegram DM fallback by rewriting route match metadata without changing the session target.

## What This Changes

The PR changes Telegram runtime binding routing, mention gating, and auto-reply dispatch so plugin-owned Telegram topic bindings can claim message-tool-only inbound turns, with focused tests.

## Best Possible Solution

Keep the plugin claim behavior under message_tool_only, but carry an explicit plugin-owned binding signal to Telegram and leave matchedBy/session unchanged for plugin-owned runtime binding routes.

## Reproduction Assessment

Yes, by source inspection: current main resolves the plugin-owned binding but can drop the turn at mention gating and skips the plugin claim when message_tool_only sets suppressDelivery. The PR body also includes live after-fix Telegram gateway logs, though I did not re-run the live lane.

## Solution Assessment

No. The dispatch change is narrow, but the route change should use a separate plugin-owned binding indicator instead of changing matchedBy while keeping the original session key.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve named-account DM fallback for plugin bindings:** `src/channels/plugins/binding-routing.ts:143-145`
  - body: This branch marks plugin-owned runtime bindings as `binding.channel` while keeping the original route session. Keep the route tag/session unchanged and pass a separate plugin-owned binding flag to the mention-bypass path.
