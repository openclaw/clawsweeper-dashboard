## Summary

Keep open: the PR addresses a real current-main Telegram plugin-bound topic routing bug and includes sufficient live log proof, but the latest patch still disables Telegram mention gating before the plugin claim succeeds, so missing-plugin or no-handler fallback can process unmentioned group/topic traffic through the ordinary agent path.

## What This Changes

The PR adds a plugin-owned runtime-binding signal for Telegram conversation routes, bypasses mention gating for those routes, allows plugin-owned claims under `message_tool_only`, and adds focused regression tests.

## Best Possible Solution

Keep the `message_tool_only` plugin-claim fix, but carry enough binding and mention-state through dispatch so handled plugin claims can run while plugin fallback paths still honor the configured Telegram activation policy or terminate safely.

## Reproduction Assessment

Yes. Source inspection shows current main skips plugin-owned binding claims under `message_tool_only`, and the latest PR head source-reproduces the remaining fallback regression through `requireMention=false` plus `missing_plugin`/`no_handler` fallthrough.

## Solution Assessment

No. The dispatch fix is in the right area, but disabling Telegram mention gating during context construction is too early; the fallback path should preserve the original mention decision or avoid ordinary agent fallback for unmentioned bound group turns.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve mention gating on plugin fallback:** `extensions/telegram/src/bot-message-context.ts:431-433`

Keep the configured mention decision for fallback, or stop ordinary fallback processing for unmentioned bound turns.
