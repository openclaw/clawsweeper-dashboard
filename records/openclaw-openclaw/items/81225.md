## Summary

Keep open: the PR fixes a real Telegram plugin-bound topic delivery bug and includes sufficient live log proof, but the latest head still broadens mention-gated group processing when the plugin claim falls back, so it should not merge as-is.

## What This Changes

The PR marks plugin-owned Telegram runtime bindings, bypasses Telegram mention gating for those routes, allows plugin claims under `message_tool_only`, and adds focused routing/dispatch tests.

## Best Possible Solution

Keep plugin claims enabled under `message_tool_only`, but carry enough binding and mention-state through the turn so handled plugin claims bypass Telegram mention gating while missing/no-handler fallback remains gated or stops before ordinary agent processing.

## Reproduction Assessment

Yes by source inspection. Current main can skip a plugin-owned binding claim under `message_tool_only`, and the latest PR head source-reproduces the remaining fallback regression through `requireMention=false` plus `missing_plugin`/`no_handler` fallthrough; I did not execute a live repro in this read-only review.

## Solution Assessment

No. The dispatch-side allowance for `message_tool_only` is in the right area, but the Telegram mention bypass is too broad because it runs before the plugin claim succeeds and does not protect fallback processing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep mention gating on plugin fallback:** `extensions/telegram/src/bot-message-context.ts:431-433`
