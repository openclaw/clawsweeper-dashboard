## Summary

Keep this PR open: it addresses a valid Telegram plugin-bound topic routing bug and includes credible live log proof, but the latest diff still has a blocking routing regression already noted in review. The PR uses `matchedBy` as a plugin-binding signal while leaving the original session key intact, which breaks the existing named-account Telegram DM fallback path when a plugin-bound conversation falls back to normal processing.

## What This Changes

The PR marks plugin-owned runtime bindings as explicit channel bindings, bypasses Telegram mention gating for those bound routes, and lets plugin-owned inbound claims run when source reply delivery is `message_tool_only`, with regression tests.

## Best Possible Solution

Keep the plugin claim behavior under `message_tool_only`, but carry an explicit plugin-owned binding signal to Telegram instead of reusing `matchedBy` without a matching session rewrite.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main resolves the plugin-owned binding but skips the claim when `message_tool_only` makes `suppressDelivery` true. The PR body also supplies live AlphaClaw Telegram logs for the after-fix path, though I did not run the live Telegram lane in this read-only review.

## Solution Assessment

No, not as currently written. The dispatch change is directionally right, but the route change should use a separate plugin-binding indicator instead of changing `matchedBy` while leaving the original session key in place.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve named-account DM fallback when marking plugin bindings:** `src/channels/plugins/binding-routing.ts:143`
  - body: This branch returns the original route session but changes `matchedBy` to `binding.channel` for every plugin-owned runtime binding. Telegram named-account DMs rely on `matchedBy === "default"` to replace the shared main route with the isolated per-account DM session; if the plugin claim later falls back with `missing_plugin` or `no_handler`, normal processing will now use the shared route session instead of the named-account DM session. Keep the original route tag/session here and pass a separate plugin-bound flag to the Telegram mention-bypass path.
  - confidence: 0.91
