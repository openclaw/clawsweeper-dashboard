## Summary

Keep open. The published release image reproduces the parent-directory ownership problem, and current main still has the same Dockerfile ownership gap, so this is a valid narrow Docker image bug rather than stale cleanup.

## What This Changes

The issue reports that the `2026.5.22-slim` Docker image leaves `/home/node/.config` owned by `root:root`, blocking non-root workflows that need to create config entries there.

## Best Possible Solution

Update the runtime Dockerfile to make `/home/node/.config` writable by the `node` user while preserving the stricter `/home/node/.config/openclaw` mode, and add a focused Dockerfile regression assertion for the parent directory.

## Reproduction Assessment

Yes, with high confidence from the released image and current source: the release image denies `node` writes directly under `/home/node/.config`, and current main still lacks a parent ownership assertion or fix. I did not build a fresh current-main image during the read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is in the Docker image layer that pre-creates the named-volume mount points, with a matching regression test in `src/dockerfile.test.ts`; runtime startup repair or broader setup-script changes are not needed for the reported image bug.
