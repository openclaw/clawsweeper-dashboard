## Summary

Keep open. Current main still has the Discord shared message-tool gap this PR is trying to fix, but the latest branch is not merge-ready because its component-edit support targets the pre-split messaging runtime while current main routes edits through the split runtime files, and the user-facing fix still lacks a changelog entry. The schema should remain the documented OpenClaw Discord component-spec object; the earlier raw Discord array concern is not the blocker.

## What This Changes

The branch exposes a Discord `components` schema on the shared message tool, forwards top-level `params.components` through Discord send/edit actions, and adds focused tests for component sends and edits.

## Best Possible Solution

Land a rebased version that keeps `components` as the documented OpenClaw Discord component-spec object, exposes it in the shared Discord message-tool schema, forwards top-level send components through `handle-action.ts`, wires any retained edit support into `runtime.messaging.messages.ts` and `runtime.messaging.runtime.ts`, and adds focused regression tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to call the Discord shared `message` tool with `action: "send"` and a top-level `components` object on current main; discovery does not advertise the field and the bridge drops it before the lower runtime can send components.

## Solution Assessment

No, not as currently proposed. The send-side direction and object-shaped schema are appropriate, but the component-edit changes need to move into the current split runtime files and the PR needs the required changelog entry before merge.
