## Summary

Keep open: current main already contains the send forwarding part, but it still does not advertise the Discord component schema, and this PR's latest edit-support hunk needs to be adapted to the split Discord messaging runtime before merge.

## What This Changes

This PR exposes a Discord `components` property in shared message-tool discovery, forwards send/edit component payloads through Discord message actions, and adds focused Discord tests.

## Best Possible Solution

A rebased PR should expose the documented OpenClaw Discord component-spec object only where runtime support exists, route component edits through the current split runtime, preserve existing send behavior, and add the required changelog entry.

## Reproduction Assessment

Yes. Static inspection against current main shows the original send-forwarding gap is fixed, but discovery still returns no schema and a components-only edit still fails before reaching `editDiscordComponentMessage`.

## Solution Assessment

No, not as currently proposed. The object-shaped schema matches the docs and parser, but the edit implementation targets stale monolithic runtime wiring and needs current split-runtime integration instead of replacing the OpenClaw spec with raw Discord arrays.
