## Summary

Keep this PR open: it addresses a real Discord message-tool gap, but the current branch is not merge-ready because the useful schema work is still missing on current main, send forwarding is already present, component-edit handling targets the pre-split runtime layout, and the latest edit support lacks real behavior proof.

## What This Changes

This PR adds a Discord shared message-tool `components` schema, forwards top-level components through send/edit actions, and adds focused Discord tests.

## Best Possible Solution

Add the Discord `components` schema for the documented message-tool payload, keep current main's send forwarding, move component edits into the split messaging runtime if edit support is kept, and require real Discord proof for the changed runtime path.

## Reproduction Assessment

Yes. Source inspection on current main shows docs advertise a Discord `components` payload, but configured Discord discovery still contributes no `components` schema; component edits are also still content-only in the split runtime.

## Solution Assessment

No. The object-shaped schema matches the documented OpenClaw Discord component spec, but the component-edit implementation must move from the stale monolithic runtime hunk into `runtime.messaging.messages.ts` and `runtime.messaging.runtime.ts`, or be left out of this PR.
