## Summary

Keep open. Current main still does not satisfy the reported Feishu workflow: broadcast-only `@all` is intentionally not a bot mention, mention-gated group handling returns before broadcast dispatch, observer agents process with no visible Feishu reply, and direct/group conversations are intentionally isolated session spaces. The related open Feishu broadcast parity issue covers adjacent work but does not fully cover the direct/group continuity request.

## What This Changes

Request that Feishu multi-agent groups reliably deliver `@all` messages to every configured OpenClaw agent and preserve same-agent context across direct and group sessions.

## Best Possible Solution

Define an opt-in Feishu multi-agent fan-out model that can intentionally handle group-wide broadcasts with loop control and visible-reply rules, and separately define a same-agent direct/group continuity mechanism through explicit session linking or memory tooling with clear privacy boundaries.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: in a Feishu group with `requireMention=true`, an `@all`-only event has `mentionedBot=false` and returns before broadcast dispatch, while direct and group routes build distinct session keys.

## Solution Assessment

Unclear. Treating `@all` as a normal bot mention would conflict with the current docs and the fixed false-positive behavior, so the safer solution is an explicit Feishu fan-out/continuity design rather than a narrow mention check reversal.
