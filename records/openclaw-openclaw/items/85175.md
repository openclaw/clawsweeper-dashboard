## Summary

Keep open: current main and the latest release still expose only static `sendPolicy` matching, while this issue asks for a new turn-scoped public policy predicate. The linked implementation PR was closed unmerged, so it is useful context but not an implemented or canonical fix.

## What This Changes

The issue requests a new `session.sendPolicy.rules[].match.peerEquals: "inboundPeer"` predicate that constrains outbound delivery to the peer that triggered the current turn.

## Best Possible Solution

Have maintainers approve the turn-scoped send-policy API shape, then implement the chosen core predicate or equivalent enforcement point with schema/types/help/docs and focused policy plus delivery-path tests.

## Reproduction Assessment

No high-confidence runtime reproduction of a wrong-peer send was established here. Yes for the capability gap: current main's schema, types, docs, and evaluator source cannot express `peerEquals: "inboundPeer"`.

## Solution Assessment

Unclear pending maintainer design approval. A native policy predicate is a plausible best layer because it composes with sendPolicy and avoids channel-plugin hook asymmetry, but the exact API and no-inbound/docking/thread semantics are product decisions.
