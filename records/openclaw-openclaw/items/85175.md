## Summary

Keep open: current main and the latest release still expose only static `sendPolicy` matching, while this issue asks for a new turn-scoped public policy predicate; the matching implementation PR was closed unmerged, so it remains useful evidence but not an implemented fix.

## What This Changes

Add `session.sendPolicy.rules[].match.peerEquals: "inboundPeer"` so automatic outbound delivery can be constrained to the peer that triggered the current turn.

## Best Possible Solution

Have maintainers approve the turn-scoped send-policy API shape, then implement the chosen core predicate or equivalent enforcement point with schema, types, help/docs, and focused policy plus delivery-path coverage.

## Reproduction Assessment

Yes for the source-level capability gap: current schema, types, docs, tests, and evaluator source cannot express `peerEquals: "inboundPeer"`. No live current-main wrong-peer send was reproduced in this read-only pass.

## Solution Assessment

Unclear pending maintainer design approval. A native policy predicate is a plausible best layer because it composes with `sendPolicy` and avoids channel-plugin hook asymmetry, but the exact public API and edge-case semantics need owner review.
