## Summary

Keep open: current main still exposes only static `sendPolicy` matching, while this request adds a new public, turn-scoped policy predicate with privacy-sensitive routing semantics that need maintainer API judgment. The userland workaround remains useful context, but it is not an equivalent close path while https://github.com/openclaw/openclaw/issues/77285 remains open and the workaround cannot compose inside `sendPolicy`.

## What This Changes

The issue requests a new `SessionSendPolicy` predicate that can deny outbound delivery when the destination peer differs from the inbound peer that triggered the current turn.

## Best Possible Solution

Approve a core policy design for turn-scoped outbound routing, then implement the chosen predicate or enforcement point with schema/types/help/docs and focused tests proving the inbound peer is carried to policy evaluation without trusting outbound channel context.

## Reproduction Assessment

No high-confidence runtime reproduction was established for a wrong-peer send on current main. Source inspection does confirm the current schema, types, evaluator, and delivery call sites cannot express or evaluate the requested relational peer predicate.

## Solution Assessment

Unclear: a native relational predicate is plausible and better integrated than a userland plugin workaround, but the exact public API, no-inbound behavior, docking interaction, and group/thread semantics need maintainer design approval first.
