## Summary

Keep open: current main still only supports static `sendPolicy` predicates, while this asks for a new relational outbound-to-inbound peer guard. The request is not already implemented, not a ClawHub-only plugin bundle, and the related reports at https://github.com/openclaw/openclaw/issues/85174 and https://github.com/openclaw/openclaw/issues/79308 are useful context rather than superseding the core policy-design question.

## What This Changes

Requests an additive `SessionSendPolicy` predicate that can deny outbound delivery when the target peer does not match the inbound peer for the current turn.

## Best Possible Solution

Design a native send-policy relational predicate or equivalent core policy seam with explicit semantics for outbound target comparison, no-inbound turns, composition, audit reasons, and migration from hook-based workarounds.

## Reproduction Assessment

Yes for the missing capability: source inspection shows current main only accepts and evaluates static `sendPolicy` fields, so `peerEquals: "inboundPeer"` is not available. This is not a failing bug reproduction of a shipped contract; it is a confirmed feature gap.

## Solution Assessment

Unclear pending maintainer design. A native predicate is plausible because the current policy evaluator is the right enforcement layer, but the exact schema, composition model, and no-inbound behavior should be settled before implementation.

## Risks / Open Questions

- A fail-closed peer lock could break scheduler, heartbeat, docked-channel, or other agent-initiated sends unless no-inbound and cross-target cases are explicitly designed.
- The related Telegram wrong-chat report is still open, so this predicate may be defense in depth rather than the direct fix for that channel-specific routing path.
