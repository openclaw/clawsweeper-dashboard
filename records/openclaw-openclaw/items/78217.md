## Summary

Keep open: current main still lets a noninteractive doctor repair convert a short Gateway status timeout into a restart prompt for an already-running Gateway service, which violates the documented non-service-only repair boundary.

## What This Changes

The issue asks for `openclaw doctor --fix --non-interactive` to complete safely or skip cleanly when a supervised Gateway is already running instead of terminating mid-run.

## Best Possible Solution

Harden core doctor so an inconclusive Gateway status RPC cannot trigger a noninteractive restart of an already-running local service; keep expected-listener suppression and add focused regression plus Linux/systemd proof.

## Reproduction Assessment

No, not for the full SIGTERM symptom because this read-only review did not run a live systemd Gateway. Source inspection gives a clear current-main path for the 3000 ms timeout, `healthOk=false`, and auto-approved running-service restart sequence.

## Solution Assessment

Yes, the narrow maintainable fix is in the existing doctor Gateway health/daemon path: avoid brittle short liveness classification and make running-service restarts interactive-only when doctor is noninteractive. The broader diagnostic/lint drafts are useful adjacent work but are not the direct fix.
