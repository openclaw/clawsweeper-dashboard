## Summary

Keep open: current main partially reduces the busy-port false positive for expected Gateway listeners, but the central `doctor --fix --non-interactive` path can still turn a 3s status timeout into `healthOk=false` and auto-approve a restart of an already-running Gateway service.

## What This Changes

The issue reports `openclaw doctor --fix --non-interactive` terminating with SIGTERM instead of cleanly handling a supervised Gateway that is already running.

## Best Possible Solution

Keep this in core and make noninteractive doctor treat a live Gateway as a boundary: avoid brittle short status timeouts, preserve the expected-listener suppression already on main, and require interactive confirmation before restarting an already-running service.

## Reproduction Assessment

No, not for the full SIGTERM symptom because this read-only pass did not run a live systemd Gateway. Source inspection does give a clear reproduction path for the 3000ms timeout, `healthOk=false`, and auto-approved running-service restart sequence on current main.

## Solution Assessment

Yes, the narrow maintainable fix is to harden Gateway liveness classification and mark running-service restarts as interactive-only; the broader lint/diagnostic PRs are useful adjacent work but are not the direct fix for this issue.
