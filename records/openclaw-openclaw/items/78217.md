## Summary

Keep open: current main still has a plausible unsafe flow where a 3s non-interactive Gateway WS timeout marks the Gateway unhealthy, then daemon repair diagnoses the same local listener as a port conflict and can auto-approve a systemd restart in `--fix` mode.

## What This Changes

Make `openclaw doctor --fix --non-interactive` handle a running healthy Gateway without timing out, misreporting its own listener as a port conflict, or triggering an unclean SIGTERM.

## Best Possible Solution

Keep this in core and make non-interactive doctor treat an already-owned running Gateway as a live/safe boundary: use a less brittle health fallback, suppress same-Gateway port-conflict diagnostics, and skip service restarts unless an interactive user explicitly confirms.

## Reproduction Assessment

No, not for the full SIGTERM symptom because this read-only review did not run a live systemd Gateway. Source inspection does give a high-signal reproduction path for the reported sequence: 3s WS timeout, `healthOk=false`, busy Gateway port diagnostics, and an auto-approved systemd restart in normal `--fix --non-interactive` mode.

## Solution Assessment

Yes, the best fix is narrow: harden the Gateway liveness classification and make live Gateway service restarts an explicit interactive action, rather than adding a new flag or broad service-manager policy.
