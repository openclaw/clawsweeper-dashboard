## Summary

Keep open: current main still has a source-backed path where a 3000 ms Gateway status timeout marks health as failed, enters daemon repair, and can auto-approve restarting a running Gateway service under `doctor --fix --non-interactive`, which conflicts with the documented non-service repair boundary.

## What This Changes

The issue asks for `openclaw doctor --fix --non-interactive` to avoid abruptly terminating or restarting a healthy running Gateway after a short status timeout and same-process port detection.

## Best Possible Solution

Harden the existing doctor Gateway health and daemon path so an inconclusive status RPC cannot trigger a noninteractive restart of an already-running local service, while preserving safe non-service repairs and expected-listener suppression.

## Reproduction Assessment

No, not for the full SIGTERM symptom because this read-only review did not run a live Linux systemd Gateway. Source inspection gives a clear current-main path from 3000 ms status timeout to `healthOk=false` and an auto-approved running-service restart.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the existing doctor Gateway health/daemon flow: avoid brittle short liveness classification and make running-service restarts skip or require interactive confirmation during noninteractive doctor repair.
