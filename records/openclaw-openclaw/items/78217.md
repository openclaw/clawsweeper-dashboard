## Summary

Keep open: current main still has the source path where a short Gateway status timeout can make noninteractive doctor repair treat an already-running Gateway service as a restart target.

## What This Changes

The issue asks for `openclaw doctor --fix --non-interactive` to handle a live Gateway cleanly instead of partially repairing and terminating.

## Best Possible Solution

Harden doctor so a Gateway status timeout cannot trigger noninteractive restart of an already-running local service, while preserving explicit interactive or Gateway-command service repairs.

## Reproduction Assessment

No, not for the full SIGTERM symptom because this read-only review did not run a live Linux systemd Gateway. The current source clearly reproduces the timeout-to-failed-health-to-auto-approved-restart path.

## Solution Assessment

Yes, the narrow maintainable fix is in the existing doctor Gateway health/daemon repair path: avoid treating inconclusive liveness as permission to restart a running service in noninteractive repair mode.
