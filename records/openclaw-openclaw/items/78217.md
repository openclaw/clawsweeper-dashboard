## Summary

Keep open: current main has fixed the false busy-port diagnostic for expected Gateway listeners and the recent-restart prompt loop, but the specific `doctor --fix --non-interactive` path can still auto-approve a running Gateway service restart after a transient health miss, which is the reported SIGTERM risk.

## What This Changes

The issue asks for `openclaw doctor --fix --non-interactive` to complete safe repairs or cleanly skip/refuse live-gateway repairs instead of restarting/terminating a supervised running Gateway mid-run.

## Best Possible Solution

Keep the issue open and patch the doctor Gateway daemon service-lifecycle path so `--fix --non-interactive` reports or skips live service start/restart actions unless an explicit operator command or interactive confirmation is used.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: call `maybeRepairGatewayDaemon` with `healthOk: false`, a loaded running service, no recent restart handoff, and a `repair + nonInteractive` doctor prompter; current code auto-approves and reaches `service.restart()`. I did not run a live supervised Gateway in this read-only review.

## Solution Assessment

No, current main is only a partial solution: it fixed expected Gateway listener diagnostics and recent-restart handoffs, but it still lets noninteractive `--fix` auto-approve the running-service restart branch. The narrow fix is to mark start/restart service lifecycle prompts as requiring interactive confirmation or otherwise skip them in noninteractive repair mode.
