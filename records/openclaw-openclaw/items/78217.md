## What This Changes

The issue reports that `openclaw doctor --fix --non-interactive` can terminate or restart a supervised Gateway while it is already running instead of safely completing or skipping live-service repairs.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/94148; once a merged main commit preserves stopped-service auto-start while blocking non-interactive restarts of already-running Gateway services, this issue can close as implemented.

## Reproduction Assessment

Yes, by source inspection: current main can reach `service.restart()` for a running Gateway after a failed health/probe path, and `doctor --fix --non-interactive` auto-approves runtime repair prompts unless they require interactive confirmation. I did not run a live supervised Gateway reproduction in this read-only review.

## Solution Assessment

Yes, the narrow candidate fix is the best current shape: keep stopped-service start repair under normal non-interactive `--fix` while requiring interactive confirmation for restarting an already-running service.
