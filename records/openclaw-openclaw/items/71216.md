## Summary

Keep open. Current main already covers the sandbox part through agent-scoped sandbox config, and it has a scoped `file-transfer` plugin `denyPaths` policy, but it still does not implement the requested top-level `routing.rules`, single-config `instances`, or gateway-wide `gateway.nodes.denyPaths` contract. The remaining asks are security-sensitive and broad enough that they need maintainer product/security design before a repair PR is safe.

## What This Changes

Request to add config schema/runtime support for sandbox policy, tag-based provider routing, single-file multi-instance declarations, and gateway-wide node path denial.

## Best Possible Solution

Keep this issue open and split the design: preserve the existing `agents.defaults.sandbox` / `agents.list[].sandbox` model, decide whether tag-based provider routing needs trusted runtime metadata or a narrower plugin/API seam, define whether `instances.*` is compatible with the documented isolated-profile gateway model, and either promote the plugin-scoped file-transfer `denyPaths` pattern into a clearly scoped core contract or document why path denial remains plugin-owned.

## Reproduction Assessment

Not applicable as a feature/config-schema request. The high-confidence verification path is source/docs inspection: current main has agent-scoped sandboxing and plugin-scoped file-transfer `denyPaths`, while top-level `routing.rules`, `instances`, and gateway-wide `gateway.nodes.denyPaths` are absent from the strict schema and runtime node policy.

## Solution Assessment

No, not as one bundled schema addition. The safer path is maintainer design that separates the already-supported sandbox concern from routing, instance isolation, and path denial, then lands each enforcement boundary with targeted schema/runtime/docs tests.

## Security Review

Status: needs_attention

Summary: This issue requests security policy controls; no patch is under review, but the remaining design needs explicit security review before implementation.

Concerns:

- **[medium] Define gateway-wide path-deny coverage:** `src/gateway/node-command-policy.ts:200`
  - body: Current gateway node policy only resolves command IDs, while `denyPaths` exists only inside the file-transfer plugin. A core `gateway.nodes.denyPaths` field must state whether it covers node.invoke, file-transfer, gateway filesystem tools, sandbox bridges, elevated exec, and plugin-owned tools.
  - confidence: 0.88
- **[medium] Define trusted routing tag provenance:**
  - body: Tag-based provider routing can leak sensitive work to the wrong provider if tags are optional, user-forgeable, dropped during subagent/delegation flows, or ambiguous when several rules match.
  - confidence: 0.8
- **[medium] Keep multi-instance trust boundaries explicit:** `docs/cli/security.md:28`
  - body: The docs currently steer mutually untrusted/shared-user setups toward separate gateways or OS users/hosts. A single-config `instances.*` model needs equivalent guarantees around credentials, workspaces, ports, service lifecycle, and validation.
  - confidence: 0.78

## Risks / Open Questions

- Bundling all four asks into one schema/runtime change could blur distinct enforcement boundaries and make the security posture hard to audit.
- Tag-based provider routing needs trusted tag provenance, delegation propagation rules, conflict ordering, and fail-closed behavior before it can safely protect sensitive work.
- A gateway-wide `denyPaths` field could overpromise protection unless maintainers define exactly which surfaces it covers: node.invoke, file-transfer, gateway filesystem tools, sandbox bridges, elevated exec, and plugin-owned tools.
- Single-config `instances.*` could weaken the current isolated-profile/separate-gateway model if workspace, credentials, ports, service lifecycle, and OS-user boundaries are not explicit.
