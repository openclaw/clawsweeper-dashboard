## Summary

Keep open. Current main only partially overlaps the request: sandboxing is agent-scoped, routing is binding-based rather than tag/provider-rule based, multi-gateway operation is documented as separate profiles/configs, and path denial is limited to narrower filesystem/file-transfer policies rather than a gateway-wide `gateway.nodes.denyPaths` contract.

## What This Changes

Request to add top-level `sandbox`, tag-based `routing.rules`, `instances.*`, and gateway-wide `gateway.nodes.denyPaths` config surfaces.

## Best Possible Solution

Split this into narrower maintainer-approved designs: top-level sandbox semantics, provider routing rule provenance, multi-instance lifecycle/isolation, and filesystem/path-deny policy should each get its own accepted contract and tests before implementation.

## Reproduction Assessment

Not applicable as a feature/config-schema request. The high-confidence verification path is source and docs inspection, which shows the requested four config contracts are still absent or only partially covered by adjacent mechanisms.

## Solution Assessment

No, not as one bundled schema addition. The safer path is to split the work into separate security/product designs with schema, runtime enforcement, docs, generated metadata, and focused tests for each accepted surface.
