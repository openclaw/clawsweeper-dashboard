## Summary

Keep open. Current main only partially overlaps the request: sandboxing already exists as agent-scoped config and file-transfer has a plugin-local denyPaths policy, but the requested top-level sandbox alias, tag-based provider routing rules, single-config instances, and gateway-wide gateway.nodes.denyPaths contract are not implemented. The remaining work crosses security and product boundaries, so it needs maintainer design before any small fix PR.

## What This Changes

Add config schema and runtime support for sandbox mode, tag-based provider routing rules, multi-instance gateway declarations, and gateway-wide node path denial.

## Best Possible Solution

Keep the issue open and split the design into narrower changes: keep or explicitly alias the existing agent-scoped sandbox model, design provider-routing metadata separately, decide whether instances.* should complement the profile-per-gateway model, and only promote path denial into core once the enforcement boundary is precise.

## Reproduction Assessment

Not applicable as a feature/config-schema request. The high-confidence verification path is source and docs inspection: current main shows agent-scoped sandboxing and plugin-local file-transfer denyPaths, while routing.rules, instances.*, and gateway-wide gateway.nodes.denyPaths are absent.

## Solution Assessment

No, not as one bundled schema addition. The safer path is separate maintainer-approved designs with schema, runtime enforcement, docs, generated metadata, and targeted tests for each security-sensitive surface.

Summary: No patch is under review, but the requested config surfaces are security-sensitive and need explicit maintainer security design before implementation.
