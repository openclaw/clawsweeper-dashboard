## Summary

Keep open. Current main only partially overlaps the request: sandboxing is already supported under agent-scoped config and file-transfer has a plugin-local `denyPaths` policy, but `routing.rules`, single-config `instances`, and gateway-wide `gateway.nodes.denyPaths` are still absent. The remaining work changes security and product boundaries, so it should stay with maintainer design rather than be swept into a small autonomous fix.

## What This Changes

The issue requests config schema and enforcement surfaces for global sandbox mode, tag-based provider routing, multi-instance declarations, and gateway-wide path denial.

## Best Possible Solution

Keep this issue open and split the design: leave sandboxing on the existing agent-scoped model unless maintainers want an alias, define provider-routing metadata and conflict behavior separately, decide whether `instances.*` should complement or replace the documented profile-per-gateway pattern, and only promote path denial into core once its enforcement boundary is explicit.

## Reproduction Assessment

Not applicable as a feature/config-schema request. The high-confidence verification path is source and docs inspection: current main shows agent-scoped sandboxing and plugin-scoped file-transfer `denyPaths`, while the requested `routing.rules`, `instances`, and gateway-wide `gateway.nodes.denyPaths` contract is absent.

## Solution Assessment

No, not as one bundled schema addition. The maintainable path is to split the security-sensitive asks and land each with schema, runtime enforcement, docs, generated metadata, and targeted tests after maintainer design approval.
