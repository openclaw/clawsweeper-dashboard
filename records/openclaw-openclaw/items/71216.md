## Summary

Keep open. Current main still only partially overlaps the request: sandboxing is configured under agent scope and file-transfer has plugin-local `denyPaths`, but top-level `sandbox`, `routing.rules`, single-config `instances`, and gateway-wide `gateway.nodes.denyPaths` are not implemented. The remaining asks cross config, runtime enforcement, provider routing, and security boundaries, so they need maintainer design before implementation.

## What This Changes

Request to add config/schema support for top-level sandbox mode, tag-based provider routing rules, multi-instance declarations, and gateway-wide node path denial.

## Best Possible Solution

Keep the issue open and split the design into smaller surfaces: decide whether top-level sandbox should be an alias or remain agent-scoped, specify provider-routing metadata and rule precedence separately, decide whether `instances.*` complements the documented profile model, and only add gateway-wide path denial after the enforcement boundary is explicit.

## Reproduction Assessment

Not applicable as a feature/config-schema request. The high-confidence verification path is source and docs inspection: current main shows agent-scoped sandboxing and plugin-scoped file-transfer `denyPaths`, while the requested top-level/provider/multi-instance/gateway-wide contracts are absent.

## Solution Assessment

No, not as one bundled schema addition. The maintainable path is to split the security-sensitive asks and land each with schema, runtime enforcement, docs, generated metadata, and targeted tests after maintainer design approval.

## Risks / Open Questions

- `instances.*` could blur the current profile-per-gateway isolation model unless maintainers define state, workspace, credentials, service lifecycle, and port ownership precisely.
