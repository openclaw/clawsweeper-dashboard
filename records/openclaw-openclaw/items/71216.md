## Summary

Keep open. Current main still only partially overlaps the request: sandboxing is agent-scoped, routing is binding/session based rather than tag/provider-rule based, multi-gateway operation is documented as separate profiles/config/state, and path denial exists only in narrower command or plugin-specific policy surfaces rather than a gateway-wide `gateway.nodes.denyPaths` contract.

## What This Changes

Requested config schema additions for global sandbox mode, tag-based provider routing rules, multi-instance declarations, and gateway-wide path denial.

## Best Possible Solution

Split the request into maintainer-approved contracts for sandbox precedence, trusted routing tags, multi-instance lifecycle/isolation, and filesystem path-deny enforcement, then implement accepted pieces with schema, runtime enforcement, doctor/security-audit checks, docs, and tests.

## Reproduction Assessment

Not applicable as a feature/config-schema request. The high-confidence verification path is source and docs inspection, which shows the four requested config contracts are absent or only covered by adjacent mechanisms.

## Solution Assessment

No, not as one bundled schema-only change. The safer path is separate maintainer design for each security/product boundary before adding config keys that imply runtime enforcement.
