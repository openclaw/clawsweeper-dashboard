## Summary

Keep open. Current main and v2026.6.6 only partially cover the request via `agents.defaults.sandbox.mode`; the remaining provider-routing, per-path enforcement, and multi-instance config contracts still need maintainer product/security decisions.

## What This Changes

The issue requests first-class config schema support for sandbox mode, tag-based provider routing rules, multi-instance declarations, and gateway path deny rules.

## Best Possible Solution

Split the remaining work into maintainer-approved contracts: keep sandbox activation on `agents.defaults.sandbox`, route filesystem ACLs through the canonical filesystem-policy tracker, route multi-instance lifecycle through the Docker/multi-gateway tracker, and design provider tag-routing only after proving existing model/provider config is insufficient.

## Reproduction Assessment

Yes for the source-level schema gap: current main accepts `agents.defaults.sandbox` and `gateway.nodes.allowCommands`/`denyCommands`, but not `routing.rules`, `instances`, or `gateway.nodes.denyPaths`. This is a feature/config design request, not a broken existing behavior reproduction.

## Solution Assessment

No, not as one broad schema addition. The best path is to split filesystem enforcement, provider routing, and multi-instance lifecycle so each gets the right owner, migration story, and security review.
