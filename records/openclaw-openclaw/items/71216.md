## Summary

Keep open. Current main only partially overlaps the request: sandboxing is agent-scoped and file-transfer has plugin-local `denyPaths`, but the requested top-level `sandbox`, tag-based `routing.rules`, `instances.*`, and gateway-wide `gateway.nodes.denyPaths` config contracts remain absent and need maintainer product/security design.

## What This Changes

The issue requests new config schema surfaces for sandbox defaults, tag-based provider routing rules, multi-instance declarations, and gateway-wide denied filesystem paths.

## Best Possible Solution

Split this into narrower maintainer-approved designs: clarify whether top-level sandbox should be an alias or a new policy root, design provider-routing metadata separately, decide how `instances.*` fits the profile-per-gateway model, and only promote path denial into core once the chokepoints and non-coverage are explicit.

## Reproduction Assessment

Not applicable as a feature/config-schema request. The high-confidence verification path is source and docs inspection, which shows partial adjacent support but no implementation of the requested four config contracts.

## Solution Assessment

No, not as one bundled schema addition. The safer solution is separate schema, runtime-enforcement, docs, metadata-generation, and test work for each security-sensitive surface after maintainer design approval.
