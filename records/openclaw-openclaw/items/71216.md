## Summary

Keep this open. Current main already covers the sandbox ask through agent-scoped sandbox config, but the requested `routing.rules`, single-config `instances`, and `gateway.nodes.denyPaths` schema/runtime contracts are still absent and security-sensitive enough to need maintainer design before implementation.

## What This Changes

The issue requests config schema and gateway policy surfaces for global sandbox mode, tag-based provider routing, multi-instance declarations, and gateway node path denies.

## Best Possible Solution

Keep the existing `agents.defaults.sandbox` and per-agent sandbox model, then split the remaining work into maintainer-designed tracks for provider-routing metadata, multi-instance declaration versus the documented separate-profile model, and a gateway-wide path-deny contract with explicit enforcement coverage and tests.

## Reproduction Assessment

Not applicable as a feature/config-schema request. The high-confidence verification path is source and docs inspection: current main has agent-scoped sandboxing, while `routing.rules`, `instances`, and `gateway.nodes.denyPaths` are not present in the strict schema or node policy.

## Solution Assessment

No, not as one bundled schema addition. The safer maintainable solution is to separate the already-supported sandbox concern from routing, instance validation, and path denial so each gets a clear enforcement boundary and targeted tests.
