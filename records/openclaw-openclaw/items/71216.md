## Summary

Keep open. Current main still only partially overlaps the request: sandboxing exists under agent-scoped config and file-transfer has plugin-local path denial, but the requested top-level sandbox alias, tag-based provider routing rules, single-config multi-instance model, and gateway-wide path denial contract are not implemented and need maintainer design.

## What This Changes

Request to add four OpenClaw config schema surfaces: top-level `sandbox`, tag-based `routing.rules`, multi-instance `instances.*`, and `gateway.nodes.denyPaths`.

## Best Possible Solution

Keep the issue open and split it into maintainer-approved designs before implementation: decide whether a top-level sandbox alias should exist or the agent-scoped model should remain canonical, design provider-routing metadata and tag trust separately, decide whether `instances.*` complements or replaces the profile-per-gateway model, and only promote path denial into core after the gateway-wide enforcement boundary is explicit.

## Reproduction Assessment

Not applicable as a feature/config-schema request. The high-confidence verification path is source and docs inspection, which shows the requested roots and `gateway.nodes.denyPaths` are absent while adjacent agent-scoped sandbox and plugin-local denyPaths behavior exists.

## Solution Assessment

No, not as one bundled schema addition. The maintainable path is separate security/product design and then narrow schema, runtime enforcement, generated metadata, docs, and tests for each approved surface.

## Risks / Open Questions

- `gateway.nodes.denyPaths` needs a precise enforcement boundary before implementation: node.invoke commands, file-transfer, gateway-local filesystem tools, sandbox file bridges, elevated exec, and plugin-owned tools do not share one obvious path chokepoint today.
- Tag-based provider routing needs trusted tag provenance, rule ordering, conflict behavior, and propagation semantics across skills, subagents, and delegated sessions before it can safely protect sensitive prompts.
- A single-config `instances.*` model could weaken the current documented isolation model unless it preserves separate config/state/workspace/port and service lifecycle boundaries.
