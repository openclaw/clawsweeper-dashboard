## Summary

Keep open. Current main covers the sandbox ask through the existing agent-scoped sandbox model and has a narrower file-transfer plugin `denyPaths` policy, but it still does not provide the requested top-level `routing.rules`, single-config `instances`, or gateway-wide `gateway.nodes.denyPaths` contract. The remaining work is security-sensitive and broad enough to need maintainer product/security design before implementation.

## What This Changes

Requests schema and runtime support for top-level sandbox mode, tag-based model routing rules, multi-gateway instance declarations, and gateway-wide node path denial.

## Best Possible Solution

Split the design into focused surfaces: keep sandbox under the existing agent-scoped model unless maintainers choose a migration path, define trusted runtime metadata before adding provider routing rules, decide whether multi-gateway remains profile-based or gains a validated `instances.*` layer, and only then add gateway-wide path denial with schema, runtime enforcement, docs, security-audit coverage, and targeted tests.

## Reproduction Assessment

Not applicable as a feature/config-schema request. The high-confidence verification path is source and docs inspection: current main supports agent-scoped sandboxing and plugin-scoped file-transfer `denyPaths`, while the requested top-level routing, instances, and gateway-wide deny-path contracts are absent.

## Solution Assessment

No, not as one bundled schema addition. The maintainable path is to separate the already-supported sandbox concern from the broader routing, multi-instance, and path-denial enforcement boundaries, then land each with explicit security semantics.

## Security Review

Status: needs_attention

Summary: This issue itself is security-sensitive; no patch is under review, but the requested controls need explicit security design before implementation.

Concerns:

- **[medium] Define gateway-wide path-deny coverage:** `src/gateway/node-command-policy.ts:200`
  - body: Current gateway node policy resolves command IDs, while path denial is implemented only in the file-transfer plugin. A core `gateway.nodes.denyPaths` field must define which tool and node surfaces it actually covers before it can be trusted as a chokepoint.
  - confidence: 0.88
- **[medium] Define trusted routing tag provenance:**
  - body: Tag-based provider routing can leak sensitive work if tags are optional, user-forgeable, dropped across subagent/delegation flows, or ambiguous when several rules match.
  - confidence: 0.8
- **[medium] Preserve multi-instance trust boundaries:** `docs/gateway/multiple-gateways.md:117`
  - body: The docs currently recommend separate profiles, config files, state directories, workspaces, and ports for multiple Gateways. A single-config `instances.*` model needs equivalent guarantees around credentials, service lifecycle, and isolation.
  - confidence: 0.78
