## Summary

Keep this PR open. Current main still lacks Slack ack-reaction run-start timing, and the PR remains a useful implementation candidate, but the patch is not ready as-is because the new dispatch test still mocks pre-extension paths and the config-generated artifacts are missing. The earlier Greptile scope-gate concern appears addressed in the PR with `ackReactionAllowed`.

## What This Changes

The PR adds Slack-scoped `ackReactionTiming` config/type/schema/help/docs/context/provider plumbing plus prepare/dispatch changes and tests to optionally delay Slack ack reactions until `onAgentRunStart`.

## Best Possible Solution

Land a rebased Slack implementation that defaults to receipt-time behavior, accepts `received | run-start` through strict Slack config/schema/docs and generated metadata, defers ack/status reaction startup from `onAgentRunStart` only when configured, preserves `ackReactionScope`, `message_tool_only`, silent-success cleanup, status-reaction lifecycle, and `removeAckAfterReply`, and validates the Slack path with targeted tests plus config generation checks.

## Reproduction Assessment

Yes. The missing current-main behavior is reproducible by static config/runtime inspection: the strict Slack schema lacks the key, the context/provider do not carry timing, and the runtime starts ack/status reaction work before any run-start callback.

## Solution Assessment

No, not as currently patched. The product direction is reasonable and the scope-gate fix is the right shape, but the PR should update stale extension-layout tests, regenerate config artifacts, and rebase around current Slack status-reaction code before it is the maintainable solution.
