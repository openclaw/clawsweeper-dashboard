## Summary

Keep open. Current main still charges `toolResult.details` in the embedded-runner guard estimate, while this PR narrowly removes that non-model-visible budget pressure and adds focused regression coverage plus a changelog entry. I found no discrete code-review finding, but exact-head CI for the latest head is still unstable/failing and should be green or triaged before merge.

## What This Changes

The PR removes tool-result `details` from embedded-runner context-guard character estimation, adds regression tests for large details payloads, and records the user-facing fix in `CHANGELOG.md`.

## Best Possible Solution

Land this scoped estimator fix after exact-head CI is green or any unrelated failures are clearly triaged, while preserving the existing persistence, truncation, and session JSONL behavior boundaries.

## Reproduction Assessment

Yes. On current main, a tool-result message with small `content` and very large `details` reproduces the false budget pressure because the estimator charges `details` and the guard uses that estimate for truncation and overflow decisions.

## Solution Assessment

Yes. Excluding `details` at this estimator is the narrowest maintainable fix because model-facing compaction already strips `toolResult.details`; the guard budget should reflect visible prompt content, not internal metadata.
