## Summary

Keep open. Current main still has the estimator mismatch, and this PR appears to fix it in the right owner path with focused coverage and green exact-head checks; the remaining step is maintainer review and landing, not ClawSweeper closure.

## What This Changes

The PR removes `toolResult.details` from tool-result context-budget estimation, adds focused context-guard regression tests, and records the fix in `CHANGELOG.md`.

## Best Possible Solution

Land the narrow estimator change with its regression tests and changelog after maintainer review, preserving the contract that `details` remains runtime/UI/persistence metadata while only model-visible tool-result `content` affects prompt context accounting.

## Reproduction Assessment

Yes. On current main, a tool-result message with small `content` and large `details` flows through `estimateMessageChars`, which charges `details`; the guard then consumes that inflated estimate even though provider replay strips `details`.

## Solution Assessment

Yes. Excluding `details` in the shared tool-result estimator is the narrowest maintainable fix; changing persistence, session JSONL, provider replay, or truncation policy would be broader than the observed mismatch.
