## Summary

Keep open. Current main still counts `toolResult.details` in the tool-result guard estimator while the model and compaction boundaries strip those details, and the PR is a narrow, well-scoped fix with focused regression coverage and changelog text. I did not find a functional or security finding in the diff, but the exact head is still not merge-ready because CI is unstable with a generic preflight failure and some checks were still running when inspected.

## What This Changes

This PR removes `toolResult.details` from tool-result character estimation, adds regression tests for details-heavy tool results, and records the user-facing fix in `CHANGELOG.md`.

## Best Possible Solution

Land the estimator change with its regression tests and changelog after exact-head CI is green, preserving the contract that `details` remains runtime/UI/persistence metadata while only model-visible tool-result content affects prompt context accounting.

## Reproduction Assessment

Yes. On current main, a tool-result message with small `content` and large `details` exercises `estimateMessageChars`, which charges `details`; the guard then consumes that inflated estimate even though provider replay and compaction strip `details`.

## Solution Assessment

Yes. Excluding `details` in the shared tool-result estimator is the narrowest maintainable fix; changing persistence, session JSONL, or truncation policy would be broader than the observed mismatch.
