## Summary

Keep open. Current main still counts internal `toolResult.details` in the tool-result context-guard estimate even though model and compaction boundaries strip those details, and the PR is a narrow fix with focused regression coverage; it should proceed through normal maintainer review and exact-head CI rather than cleanup closure.

## What This Changes

The PR removes `toolResult.details` from tool-result context-guard character estimation, adds two regression tests for large details payloads, and adds a changelog fix entry.

## Best Possible Solution

Land the scoped estimator change with the regression tests and changelog once exact-head CI is green or the unrelated CI failure is triaged, preserving the existing boundary where `details` stays out of model-facing prompts and compaction accounting.

## Reproduction Assessment

Yes. On current main, a tool-result message with small visible `content` and a large `details` payload exercises `estimateMessageChars`, which charges `details`; the guard then uses that inflated estimate for truncation and aggregate overflow decisions.

## Solution Assessment

Yes. Excluding `details` in the guard estimator is the narrowest maintainable fix because provider conversion and compaction already strip `toolResult.details`; changing persistence, session JSONL, or tool-output truncation policy would be broader than necessary.
