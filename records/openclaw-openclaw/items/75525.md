## Summary

Keep open. Current main still counts `toolResult.details` in the guard estimator, while this PR is a narrow, relevant fix that aligns the guard with the existing model-boundary contract; no blocking review finding was found, but exact-head CI was still running during review.

## What This Changes

The PR removes `toolResult.details` from tool-result context-budget estimation, adds focused guard regression tests, and records the user-facing agent fix in `CHANGELOG.md`.

## Best Possible Solution

Land this narrow estimator change with the regression tests and changelog once exact-head CI is green, preserving the existing contract that `details` remains runtime/UI/persistence metadata while only model-visible tool-result content affects prompt context accounting.

## Reproduction Assessment

Yes. On current main, a tool-result message with small `content` and large `details` exercises `estimateMessageChars`, which charges `details`; the installed guard then uses that inflated estimate for truncation and overflow pressure even though provider-boundary normalization strips `details`.

## Solution Assessment

Yes. Excluding `details` in the shared tool-result estimator is the narrowest maintainable fix; changing persistence, session JSONL, provider replay, or truncation policy would be broader than the observed mismatch.
