## Summary

Keep open. Current main still counts tool-result `details` in the guard estimator even though provider-boundary normalization strips those details, and this PR is a narrow fix with focused tests and no actionable review findings.

## What This Changes

The PR removes `toolResult.details` from tool-result context-budget estimation and adds focused guard regression tests plus a changelog entry.

## Best Possible Solution

Rebase or refresh the PR if needed, keep the estimator-only behavior change with its regression tests and changelog, then land after the targeted test and changed gate pass.

## Reproduction Assessment

Yes. On current main, a tool-result message with small `content` and large `details` reaches `estimateMessageChars`, which charges `details`; the guard then consumes that inflated estimate while the provider boundary strips `details` before model conversion.

## Solution Assessment

Yes. Excluding `details` in the shared estimator is the narrowest maintainable fix because persistence, UI metadata, provider replay stripping, and truncation policy remain unchanged.
