## Summary

Keep open. Current main still counts internal `toolResult.details` in the tool-result context guard estimate even though the model-facing and compaction paths strip those details, and this PR is a narrow, well-aligned fix with regression coverage and a changelog entry. I found no blocking code-review findings, but exact-head CI was still unstable at review time with two unrelated-looking failures and several in-progress checks.

## What This Changes

The PR removes `toolResult.details` from tool-result context-guard character estimation, adds focused regression tests, and records the user-facing fix in `CHANGELOG.md`.

## Best Possible Solution

Land the scoped estimator, regression test, and changelog changes after exact-head CI is green or the unrelated-looking failures are triaged, while preserving the existing boundary that keeps `details` out of model-facing prompts and compaction token accounting.

## Reproduction Assessment

Yes. On current main, a tool-result message with small visible `content` and a large `details` payload reproduces false guard pressure because the estimator charges `details` and the guard uses that estimate for truncation and overflow decisions.

## Solution Assessment

Yes. Excluding `details` at the estimator is the narrowest maintainable fix because provider conversion and compaction already strip `toolResult.details`; changing persistence, session JSONL, or tool-output truncation policy would be broader than this bug requires.
