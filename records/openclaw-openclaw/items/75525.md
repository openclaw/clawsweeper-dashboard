## Summary

Keep open. Current main still counts tool-result `details` in the embedded-runner guard estimate, while this PR narrowly removes that non-model-visible metadata from the guard budget and adds focused regression coverage. I found no discrete PR-introduced correctness or security finding; the remaining blocker is exact-head CI validation or maintainer triage of current failing checks.

## What This Changes

This PR removes tool-result `details` from embedded-runner context-guard character estimates, adds regression tests for large details payloads, and adds an Unreleased changelog fix entry.

## Best Possible Solution

Land the scoped estimator and regression-test fix after exact-head CI is green or the current failures are documented as unrelated, while keeping persistence, truncation policy, and session JSONL behavior unchanged.

## Reproduction Assessment

Yes. On current main, a tool-result message with small visible `content` and large `details` reproduces the false pressure because the estimator charges `details` and the guard uses that estimate for truncation and overflow decisions.

## Solution Assessment

Yes. Excluding `details` only from the guard estimator is the narrow maintainable fix because the LLM-boundary and compaction paths already strip tool-result details before model-facing token accounting.
