## Summary

Keep open. Current main still counts `toolResult.details` in the tool-result context-guard estimator, while this PR removes that non-model-visible payload from the guard estimate and adds focused regression coverage; the diff appears scoped with no functional review findings, but it is not merge-ready until exact-head CI is green.

## What This Changes

The PR removes `toolResult.details` from tool-result guard character estimation, adds two context-guard regression tests, and records the user-facing fix in `CHANGELOG.md`.

## Best Possible Solution

Land the estimator change with its regression tests and changelog once exact-head CI is green, preserving the existing contract that `details` remains runtime/UI/persistence metadata while only model-visible `content` affects prompt context accounting.

## Reproduction Assessment

Yes. On current main, a `role: "toolResult"` message with small `content` and large `details` exercises `estimateMessageChars`, which charges `details`; the guard then uses that inflated estimate even though the LLM boundary strips details before provider conversion.

## Solution Assessment

Yes. Excluding `details` in the shared tool-result estimator is the narrowest maintainable fix because compaction, docs, and provider-boundary normalization already define `details` as non-model-visible; changing persistence or tool-result truncation policy would be broader than needed.
