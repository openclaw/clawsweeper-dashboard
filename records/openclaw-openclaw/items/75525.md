## Summary

Keep open. Current main still counts internal tool-result details in the guard estimator, while this PR makes a narrow, well-scoped fix with regression coverage; I did not find a blocking correctness issue, but it should remain open until maintainer review and exact-head CI finish.

## What This Changes

The PR removes `toolResult.details` from tool-result context-guard character estimates, adds focused guard regression tests, and records the user-facing fix in the changelog.

## Best Possible Solution

Land the estimator fix with its regression tests and changelog once maintainer review and final CI are green, preserving the existing contract where internal details remain available to runtime/UI/persistence paths but do not affect model-visible context accounting.

## Reproduction Assessment

Yes. On current main, a tool-result message with small visible `content` and large `details` exercises `estimateMessageChars`, which charges `details`; the guard then uses that inflated estimate even though provider conversion strips the details before the model call.

## Solution Assessment

Yes, pending CI. Excluding `details` at the shared estimator is the narrowest maintainable fix because the model-boundary and compaction paths already treat details as non-model-visible; changing persistence or tool-result truncation policy would be broader than needed.
