## Summary

Keep open. The PR addresses a real current-main mismatch: tool-result `details` are charged by the guard estimator even though model-facing conversion and compaction strip them, and the diff is scoped with focused regression coverage. I found no actionable PR bug, but exact-head CI was not fully green at review time, so this should continue through maintainer review and CI triage before merge.

## What This Changes

The PR removes internal tool-result `details` from context-guard character estimates, adds regression tests for large details payloads, and adds a changelog fix entry.

## Best Possible Solution

Land the scoped estimator change with its regression tests and changelog after maintainer review and exact-head CI triage, preserving the existing contract where internal `details` remain available to runtime/UI/persistence paths but do not affect model-visible context accounting.

## Reproduction Assessment

Yes. On current `origin/main`, a tool-result message with small visible `content` and large `details` exercises `estimateMessageChars`, which charges `details`; the guard then uses that inflated estimate even though the model-boundary path strips those details.

## Solution Assessment

Yes. Excluding `details` from this estimator is the narrowest maintainable fix because provider conversion and compaction already remove `toolResult.details`; changing persistence, truncation policy, or session JSONL behavior would be broader than necessary.
