## Summary

Keep open. Current main still counts internal tool-result details in the context-guard estimate even though the model-facing and compaction paths strip those details, and this PR is a scoped fix with focused regression tests and a changelog entry. I found no blocking code-review finding in the diff, but exact-head CI still had unrelated failures and one running check at inspection time, so the remaining action is maintainer review and CI triage rather than cleanup closure.

## What This Changes

This PR removes `toolResult.details` from tool-result guard character estimates, adds regression coverage for large details payloads in the context guard, and adds a changelog fix entry.

## Best Possible Solution

Land the scoped estimator change with the regression tests and changelog after maintainer review and exact-head CI triage, preserving the existing boundary where internal `details` remain available for runtime/UI/persistence paths but do not affect model-visible context accounting.

## Reproduction Assessment

Yes. On current main, a tool-result message with small visible `content` and large `details` exercises `estimateMessageChars`, which charges `details`; the guard then uses that inflated estimate for truncation and overflow routing even though the model-boundary path strips those details.

## Solution Assessment

Yes. Excluding `details` from the guard estimator is the narrowest maintainable fix because provider conversion and compaction already remove `toolResult.details`; changing persistence, truncation policy, or session JSONL behavior would be broader than necessary.
