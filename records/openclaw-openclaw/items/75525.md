## Summary

Keep open. Current main still counts internal tool-result `details` in the embedded-runner guard estimate, while this PR is a narrow code/test/changelog fix that aligns the guard budget with the LLM-boundary stripping path; I found no blocking correctness or security issue in the diff, but exact-head CI was still unstable at review time.

## What This Changes

The PR removes `toolResult.details` from embedded-runner context-guard character estimation, adds focused regression tests for large details payloads, and records the user-facing fix in `CHANGELOG.md`.

## Best Possible Solution

Land the scoped estimator, regression test, and changelog changes after exact-head CI is green or the current unrelated-looking failures are documented, while preserving the existing persistence/UI metadata boundary for `details`.

## Reproduction Assessment

Yes. On current main, a tool-result message with small visible `content` and a large `details` payload reproduces false guard pressure because the estimator charges `details` and the guard uses that estimate for truncation and overflow decisions.

## Solution Assessment

Yes. Excluding `details` at this estimator is the narrowest maintainable fix because provider conversion and compaction already strip `toolResult.details`; changing persistence, session JSONL, or tool-output truncation policy would be broader than the observed bug requires.
