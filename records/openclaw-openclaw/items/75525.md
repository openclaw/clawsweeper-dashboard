## Summary

Keep open. The PR addresses a real current-main mismatch between guard estimation and model-visible tool-result payloads, and the diff appears narrow and correct, but the latest head still has a generic exact-head `check-preflight-guards` failure that needs maintainer CI inspection or rerun before merge.

## What This Changes

This PR removes `toolResult.details` from tool-result guard character estimation, adds two focused regression tests for details-heavy tool results, and records the user-facing fix in `CHANGELOG.md`.

## Best Possible Solution

Land the estimator change with its regression tests and changelog after exact-head CI is green, preserving the existing contract that `details` remains runtime/UI/persistence metadata while only model-visible `content` affects prompt context accounting.

## Reproduction Assessment

Yes. On current main, a `toolResult` message with small `content` and large `details` exercises `estimateMessageChars`, which charges `details`; the guard then uses that inflated value even though the provider and compaction paths strip details before model input.

## Solution Assessment

Yes. Excluding `details` in the shared tool-result estimator is the narrowest maintainable fix; changing persistence, session JSONL, or truncation policy would be broader than the observed mismatch. The remaining blocker is CI, not the chosen code path.
