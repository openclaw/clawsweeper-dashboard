## Summary

Keep open. Current main still has a source-backed claude-cli/token auth status mismatch: token credentials are supported and can be usable, but the model-auth provider rollup can still report `missing` when its effective profile set is empty, which matches the reported TUI warning path.

## What This Changes

The issue requests that token-mode claude-cli/Anthropic setup-token credentials be counted as effective auth so the TUI does not show a false expired-auth warning while runtime routing is usable.

## Best Possible Solution

Align model auth status with the effective runtime auth route for token-only claude-cli/Anthropic setup-token setups, while preserving explicit auth-order semantics and adding focused regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not from a live current-main run; the issue includes concrete status JSON and source inspection shows how `effectiveProfiles: []` becomes provider `missing`, but a focused test still needs to pin the exact token-only claude-cli setup.

## Solution Assessment

Is this the best way to solve the issue? Unclear as proposed: counting healthy token profiles is likely necessary, but the maintainable fix should align the shared auth-status rollup with runtime effective auth instead of only suppressing the TUI warning.
