## Summary

Keep open: the bug is source-reproducible and the direction is useful, but this PR cannot merge as-is because it patches the stale registry-local cleanup path, lacks after-fix real channel proof, and still needs the fallback delivery contract resolved.

## What This Changes

The PR adds a `notifyAnnounceGiveUp` fallback in `src/agents/subagent-registry.ts` to send a best-effort gateway `agent` notification with recovered subagent output after announce retry-limit give-up.

## Best Possible Solution

Rebase the fallback into `src/agents/subagent-registry-lifecycle.ts`, cover live and resumed retry-limit give-up, and make the fallback explicitly use either the documented agent-mediated contract or a separately reviewed raw outbound path.

## Reproduction Assessment

Yes, at source level: current main reaches retry-limit give-up through `resolveDeferredCleanupDecision`, then lifecycle cleanup marks delivery failed and completes without a fallback notification. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not as-is. The fallback direction is reasonable, but the patch must move into the active lifecycle controller and align with either the existing agent-mediated delivery contract or a reviewed raw outbound path.
