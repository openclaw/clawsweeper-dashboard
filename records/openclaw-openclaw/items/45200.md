## Summary

Keep open: the reported source-level bug still exists on current main and the fallback-notification direction is reasonable, but this PR should not merge as-is because it patches the stale registry cleanup surface instead of the active lifecycle controller and still lacks real behavior proof.

## What This Changes

The PR adds a best-effort `notifyAnnounceGiveUp` helper in `src/agents/subagent-registry.ts` that calls gateway `agent` delivery after announce retry-limit give-up using stored requester delivery context.

## Best Possible Solution

Move the fallback into `src/agents/subagent-registry-lifecycle.ts` as part of the give-up finalization flow, use a stable idempotency key, cover live and resumed retry-limit give-up, and prove the visible channel result exactly once.

## Reproduction Assessment

Yes, at source level. `resolveDeferredCleanupDecision` reaches retry-limit give-up, and the current lifecycle controller then marks delivery failed and completes cleanup without a fallback user notification.

## Solution Assessment

No, not as-is. A fallback notification is a reasonable narrow fix, but this branch must be rebased into the lifecycle controller and backed by focused regression coverage plus real channel proof.
