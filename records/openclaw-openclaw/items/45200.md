## Summary

Keep open: this PR targets a real subagent completion-loss bug, but the proposed fallback still re-enters the gateway agent/model delivery path after that path has already exhausted retries, and the external PR has no after-fix real behavior proof.

## What This Changes

The PR adds a `notifyAnnounceGiveUp` fallback in `src/agents/subagent-registry.ts` to notify the requester when subagent announce retries are exhausted.

## Best Possible Solution

Port a narrow retry-limit fallback onto the current lifecycle using a durable outbound or explicit delivery-state path that does not require another requester model turn, then cover it with focused lifecycle tests and real channel proof.

## Reproduction Assessment

Yes, at source level. Current main can still enter the retry-limit give-up path, mark delivery failed, and complete cleanup without a last-resort notification; I did not run a live Telegram or gateway reproduction in this read-only review.

## Solution Assessment

No. The intent is right, but another `method: "agent"` call is not the narrowest maintainable recovery path after the agent-mediated announce pipeline has already exhausted retries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use a real fallback delivery path:** `src/agents/subagent-registry.ts:193-210`
  - body: This fallback is meant to recover after announce delivery gives up, but it calls the gateway `agent` method again with `deliver: true`. That starts another requester model run and then delivers that run's output, so model/auth/session failures or the same channel delivery failure can still drop the recovered result. Move the fallback into a durable outbound/send path or the existing delivery-state machinery instead of re-entering the failed announce pipeline.
  - confidence: 0.86
