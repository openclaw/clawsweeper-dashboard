## Summary

Keep open. Current main still has source-proven terminal announce give-up paths with no final requester-channel notification, but this PR should not merge as-is because it routes recovery through gateway `agent` model mediation and lacks after-fix real behavior proof.

## What This Changes

The PR adds `notifyAnnounceGiveUp` in `src/agents/subagent-registry.ts`, firing from retry-limit/live give-up paths and calling gateway `agent` with `deliver: true` to notify the requester channel with recovered result text.

## Best Possible Solution

Implement terminal give-up recovery on current main in the lifecycle/delivery modules using one raw outbound delivery path with preserved requester origin, per-run idempotency, delivery status/error recording, regression tests, and real behavior proof.

## Reproduction Assessment

Yes, from source inspection. On current main, both `finalizeResumedAnnounceGiveUp` and the live `deferredDecision.kind === "give-up"` path mark delivery failed and clean up without a final requester-channel notifier; I did not run a live two-agent or Telegram scenario in this read-only review.

## Solution Assessment

No. A last-resort notification is the right direction, but this PR sends the notice through gateway `agent`, so the requester model mediates the recovered text; a direct outbound delivery helper with idempotency is the narrower maintainable fix.
