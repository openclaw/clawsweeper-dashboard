## Summary

Keep open: the reported silent subagent-result loss is valid at source level, but this PR still patches the old registry cleanup path while current main finalizes retry-limit give-up in the lifecycle controller, and the external PR still lacks real behavior proof.

## What This Changes

The PR adds a best-effort `notifyAnnounceGiveUp` fallback in `src/agents/subagent-registry.ts` that calls gateway `agent` after subagent announce retry-limit give-up.

## Best Possible Solution

Rebase the fallback into `src/agents/subagent-registry-lifecycle.ts`, cover live and resumed retry-limit give-up, and provide redacted real channel proof of exactly one useful requester notification.

## Reproduction Assessment

Yes, at source level: retry-limit resolves to give-up, then current main marks delivery failed and completes cleanup without a fallback notification. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not as-is. The fallback direction is reasonable, but it needs to move into the lifecycle controller and settle the raw-versus-agent-mediated delivery contract before it is the narrow maintainable fix.
