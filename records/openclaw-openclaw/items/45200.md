## Summary

Keep open: the underlying silent subagent give-up bug still exists on current main, but this PR is not merge-ready because it sends the last-resort notice through the gateway `agent` path, which starts another requester agent/model run instead of directly delivering the recovered text. The branch is also dirty against current main after the subagent lifecycle and announce-delivery refactors, so a narrow replacement is safer than merging this patch as-is.

## What This Changes

The PR adds a `notifyAnnounceGiveUp` helper to `src/agents/subagent-registry.ts` and calls it from resumed and live announce retry-exhaustion paths to send a recovered subagent result notice via gateway `agent` delivery.

## Best Possible Solution

Implement the give-up notice in the current lifecycle and announce-delivery layers by reusing or factoring the direct outbound `sendMessage` fallback path when the normalized requester origin has a deliverable channel and target. The end state should avoid an extra requester model turn, use per-run idempotency, skip intentionally killed runs, preserve cleanup semantics, log failed last-resort sends, and cover both live and resumed retry exhaustion.

## Reproduction Assessment

Yes. Force `runSubagentAnnounceFlow` to fail until `resolveDeferredCleanupDecision` returns `give-up`, or resume a run with `announceRetryCount >= MAX_ANNOUNCE_RETRY_COUNT`; current main then logs give-up and completes cleanup without a final requester send.

## Solution Assessment

No. A last-resort notification is the right product direction, but this PR's gateway `agent` approach is not the narrowest maintainable fix because it creates another requester agent/model turn instead of sending the recovered notice through the existing outbound delivery seam.
