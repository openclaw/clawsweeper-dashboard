## Summary

Keep open. Current main still has a retry-limit give-up path that records failed delivery without user-visible notification, and this PR is not merge-ready because its fallback still goes through the gateway agent/model path, lacks real behavior proof, and is outdated against the current lifecycle refactor.

## What This Changes

Adds a `notifyAnnounceGiveUp` fallback in `src/agents/subagent-registry.ts` that tries to notify the requester when subagent announce retries are exhausted.

## Best Possible Solution

Port a narrow give-up fallback onto the current subagent lifecycle using a true outbound or durable delivery path that avoids another requester model run, preserves exactly-once semantics, and includes focused lifecycle coverage plus real channel proof.

## Reproduction Assessment

Yes, at source level. Current main’s cleanup give-up path marks delivery failed and completes cleanup without a user-visible last-resort notification; I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No. The intent is right, but routing the fallback through `method: "agent"` is not the narrowest maintainable fix because it still requires another requester agent run before channel delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use a real outbound delivery path for give-up notifications:** `src/agents/subagent-registry.ts:193-198`
  - body: The fallback is documented as bypassing the session/model announce pipeline, but this call still enters the gateway `agent` RPC; the gateway then runs `agentCommandFromIngress` and only delivers that agent result. When announce give-up is caused by requester model/auth/session failure, this last-resort notification can fail through the same path instead of sending the recovered text directly. Use the existing outbound/send or durable announce delivery seam for the final notification.
  - confidence: 0.86
