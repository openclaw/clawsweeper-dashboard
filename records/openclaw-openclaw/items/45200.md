## Summary

Keep open: the silent-drop bug still exists on current main, but this PR should not merge as-is because its last-resort path still routes recovered result text through a requester-agent turn instead of the existing direct outbound fallback, and the diff is stale against the current lifecycle split.

## What This Changes

The PR adds `notifyAnnounceGiveUp` in `src/agents/subagent-registry.ts` and calls it from retry-limit cleanup paths to forward exhausted subagent announce results through gateway `agent` delivery.

## Best Possible Solution

Implement terminal announce give-up recovery in the current lifecycle and announce-delivery modules by reusing or factoring direct `sendMessage` fallback delivery so recovered result text is delivered verbatim with per-run idempotency, then add regression tests and a changelog entry.

## Reproduction Assessment

Yes. On current main, a run with `announceRetryCount >= MAX_ANNOUNCE_RETRY_COUNT` reaches resumed give-up cleanup, and a live `deferredDecision.kind === "give-up"` reaches cleanup completion; neither path invokes a final outbound send.

## Solution Assessment

No. A last-resort user notification is the right direction, but routing it through gateway `agent` is not the narrowest maintainable fix because it turns recovered text into requester-agent input instead of direct channel content.
