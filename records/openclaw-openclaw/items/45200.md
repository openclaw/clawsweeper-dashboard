## Summary

Keep open. The user-visible result-loss bug still exists on current main, but this PR should not merge as-is because its last-resort path starts another requester agent/model turn instead of using the direct outbound delivery seam that already exists for failed completion delivery.

## What This Changes

The PR adds a `notifyAnnounceGiveUp` helper in `src/agents/subagent-registry.ts` and calls it from retry-exhaustion cleanup paths to send a recovered subagent result notice through gateway `agent` delivery.

## Best Possible Solution

Implement the terminal give-up notice in the current lifecycle and announce-delivery layers by reusing or factoring the direct `sendMessage` fallback. It should deliver recovered result text without a requester model turn, use per-run idempotency, skip intentionally killed runs, preserve cleanup and ended-hook semantics, log failed last-resort sends, and cover both live and resumed retry exhaustion.

## Reproduction Assessment

Yes. Force `runSubagentAnnounceFlow` to return false until cleanup resolves to `give-up`, or restore a run with `announceRetryCount >= 3`; current main then records failed delivery and completes cleanup without a final requester-channel send.

## Solution Assessment

No. A last-resort notification is the right direction, but this PR’s gateway `agent` call is not the narrowest maintainable fix because it turns the recovered notice into requester-agent input instead of delivering the recovered result through the existing outbound channel seam.

## Review Findings
