## Summary

Keep open. This PR targets a real remaining subagent delivery bug, but the proposed fallback still runs through the gateway `agent` path after that path has already exhausted, lacks real behavior proof, and needs adaptation to current `main`'s split lifecycle controller before it can be considered merge-ready.

## What This Changes

The branch adds a best-effort `notifyAnnounceGiveUp` fallback in `src/agents/subagent-registry.ts` that calls the gateway `agent` method after subagent announce retry exhaustion.

## Best Possible Solution

Use the current lifecycle and task-delivery contracts to surface exactly one durable, user-visible completion failure/result notification, with regression coverage and live proof for the fast subagent give-up path.

## Reproduction Assessment

Yes, source-reproducible. Current main still has the finite retry-limit give-up path, and https://github.com/openclaw/openclaw/issues/44925 includes a concrete fast `sessions_spawn` reproduction and recent delivery_failed evidence, though I did not run a live Telegram/two-agent reproduction in this read-only review.

## Solution Assessment

No. A last-resort notification is the right symptom target, but reusing the gateway `agent` path after announce retries have exhausted is not the safest fallback; the fix should use the current durable delivery/task notification path or an explicit outbound send contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Use a fallback that does not re-run the failed agent path:** `src/agents/subagent-registry.ts:184-188`
  - body: This fallback is meant to notify the user after the requester announce path has already exhausted, but it calls the same gateway `agent` method on the requester session. The gateway still runs `agentCommandFromIngress` before delivering output, so model/auth/session-lock failures can fail this last-resort attempt the same way and the recovered result is still not guaranteed to surface. Route this through the current task/outbound delivery contract or another path that actually bypasses the failed requester-agent announce run.
  - confidence: 0.82
