## Summary

Keep open: the PR targets a real subagent delivery-loss bug, but it still patches the stale registry path while current main finalizes retry-limit give-up in the lifecycle controller, and the external PR has no after-fix real behavior proof.

## What This Changes

The PR adds a best-effort `notifyAnnounceGiveUp` path in `src/agents/subagent-registry.ts` that calls gateway `agent` with requester delivery context after subagent announce retry-limit give-up.

## Best Possible Solution

Rebase the fallback into `src/agents/subagent-registry-lifecycle.ts`, cover live and resumed retry-limit give-up with focused tests, and attach real channel proof showing exactly one useful notification reaches the requester.

## Reproduction Assessment

Yes, at source level. Current main reaches retry-limit give-up through `resolveDeferredCleanupDecision`, then the lifecycle controller marks delivery failed and completes cleanup without a fallback user notification.

## Solution Assessment

No, not as-is. The fallback direction is reasonable, but the patch needs to move into the active lifecycle controller and make the delivery contract explicit before it can solve the reported path safely.
