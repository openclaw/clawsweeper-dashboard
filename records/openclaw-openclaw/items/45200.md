## Summary

## What This Changes

The PR adds a `notifyAnnounceGiveUp` fallback in `src/agents/subagent-registry.ts` that calls gateway `agent` delivery when subagent announce retries reach give-up.

## Best Possible Solution

Move the fallback into the active lifecycle controller, cover both live and resumed retry-limit give-up, and prove the chosen agent-mediated or owner-approved raw-delivery contract with focused tests and redacted real channel evidence.

## Reproduction Assessment

Yes, at source level. Current main reaches retry-limit give-up through the cleanup decision and lifecycle controller, then records failed delivery without a requester fallback notification; I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not as-is. The fallback direction is reasonable, but it must be rebased into the active lifecycle controller and aligned with the documented delivery contract before merge.
