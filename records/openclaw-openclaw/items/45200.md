## Summary

Keep open: the PR targets a real current-main gap, but the patch is not mergeable as the best fix because it routes the recovered notice through another requester-agent/model turn instead of the existing direct outbound delivery seam, and the branch is stale against the current lifecycle split.

## What This Changes

The PR adds a `notifyAnnounceGiveUp` helper and calls it from retry-exhaustion cleanup paths to send recovered subagent results through gateway `agent` delivery.

## Best Possible Solution

Implement terminal announce give-up recovery in the current lifecycle/delivery modules by reusing or factoring direct `sendMessage` fallback delivery, preserving cleanup and ended-hook semantics while delivering raw recovered result text with per-run idempotency.

## Reproduction Assessment

Yes. On current main, force `runSubagentAnnounceFlow` to keep returning false until cleanup resolves to `give-up`, or restore a run with `announceRetryCount >= MAX_ANNOUNCE_RETRY_COUNT`; both paths mark delivery failed and complete cleanup without a final requester-channel send.

## Solution Assessment

No. A last-resort user notification is the right product direction, but routing it through gateway `agent` is not the narrowest maintainable fix because it turns recovered text into requester-agent input instead of direct outbound channel content.
