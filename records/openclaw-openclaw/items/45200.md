## Summary

Keep open: the bug is real and current main still has terminal give-up paths, but this PR routes the fallback through another requester agent run, lacks real behavior proof, and overlaps newer maintainer queue-first delivery work.

## What This Changes

The PR adds a `notifyAnnounceGiveUp` fallback in `src/agents/subagent-registry.ts` that fires when subagent announce retries are exhausted.

## Best Possible Solution

Preserve completed child output through the existing queue/steer or durable outbound delivery machinery, avoid another requester model turn for final failure notification, and prove exactly-once visible delivery with focused lifecycle tests plus real channel evidence.

## Reproduction Assessment

Yes, at source level. Current main still has finite retry-limit give-up paths that mark delivery failed without a last-resort user-visible notification, and the linked issue includes recent production logs matching that path.

## Solution Assessment

No. The PR intent is valid, but `method: "agent"` is not the best delivery path for a give-up fallback because it starts another requester agent run instead of sending through a durable or lower-level outbound channel path.

## Review Findings
