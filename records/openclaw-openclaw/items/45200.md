## Summary

Keep open. The reported bug is still source-reproducible on current main, but this PR should not merge as-is because its fallback repeats the same gateway agent/model delivery path that already exhausted retries, and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a best-effort `notifyAnnounceGiveUp` path that tries to notify the requester after subagent announce retry-limit give-up by calling the gateway `agent` method with the recovered result text.

## Best Possible Solution

Rework the fix to deliver a self-contained frozen-result or failure notice through a durable direct outbound/task-delivery path that does not require another requester-agent model turn, with deduplication and focused regression coverage for retry-limit give-up.

## Reproduction Assessment

Yes, at source level: current main still reaches a retry-limit give-up path that records failed delivery and cleanup without a user notification, while subagent terminal task updates are suppressed. I did not run a live Telegram reproduction.

## Solution Assessment

No. The PR’s last-resort notification calls the same gateway `agent` path used by normal announce delivery, so it remains vulnerable to the model/session/gateway-agent failures that caused retry exhaustion; a lower-level durable outbound send is the safer fix direction.
