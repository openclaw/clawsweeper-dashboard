## What This Changes

The PR adds a last-resort `notifyAnnounceGiveUp` fallback in `src/agents/subagent-registry.ts` that calls gateway `agent` with `deliver: true` after subagent announce retry-limit give-up.

## Best Possible Solution

Rebase the fallback into `src/agents/subagent-registry-lifecycle.ts`, cover both live and resumed retry-limit give-up, and make the fallback delivery contract match the documented agent-mediated handoff or explicitly add a reviewed lower-level outbound path.

## Reproduction Assessment

Yes, at source level: on current main retry-limit resolves to give-up, then lifecycle cleanup marks delivery failed, logs give-up, completes cleanup, and emits the ended hook with no fallback notification. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not as-is. The fallback direction is reasonable, but the patch must move into the lifecycle controller, add focused coverage, and resolve whether the fallback is raw outbound delivery or an agent-mediated response.
