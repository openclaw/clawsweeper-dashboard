## Summary

Keep open. The underlying silent subagent completion-loss bug is real, but this branch is not mergeable as-is because current main owns retry-limit give-up cleanup in the lifecycle controller, the PR still patches the older registry-local path, and external after-fix channel proof is missing.

## What This Changes

The PR adds a `notifyAnnounceGiveUp` fallback in `src/agents/subagent-registry.ts` that attempts a best-effort gateway `agent` delivery when subagent announce retries reach the retry-limit.

## Best Possible Solution

Move the fallback into `src/agents/subagent-registry-lifecycle.ts`, cover both live and resumed retry-limit give-up, and make the delivery contract explicit before merge.

## Reproduction Assessment

Yes, at source level. Current main reaches retry-limit give-up through the cleanup decision and lifecycle controller, then completes cleanup without a requester fallback notification; I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not as-is. The fallback direction is reasonable, but the implementation must move to the active lifecycle controller and align with the documented agent-mediated delivery contract or a reviewed raw-delivery path.
