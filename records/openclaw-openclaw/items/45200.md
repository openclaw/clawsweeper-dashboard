## Summary

Keep this PR open as useful source work, but it should not merge as-is: current main still drops terminal announce give-up results, and the PR’s fallback still routes the recovered text through a gateway `agent` turn instead of a direct outbound delivery path.

## What This Changes

This PR adds a best-effort subagent announce give-up notifier that calls gateway `agent` delivery from the requester origin after retry exhaustion.

## Best Possible Solution

Implement terminal announce give-up recovery in the current lifecycle and delivery modules by reusing direct `sendMessage` fallback delivery for raw recovered result text with per-run idempotency.

## Reproduction Assessment

Yes, from source inspection. A restored run with `announceRetryCount >= MAX_ANNOUNCE_RETRY_COUNT` or a live cleanup decision of `give-up` reaches terminal cleanup on current main without any final requester-channel notification.

## Solution Assessment

No. A last-resort notification is the right direction, but routing it through gateway `agent` creates a requester-agent/model turn; the narrower maintainable fix is direct outbound fallback delivery of the recovered text.
