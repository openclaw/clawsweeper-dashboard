## Summary

Keep open: the silent subagent announce give-up bug still exists on current main, but this PR is not safe as-is because its fallback sends the recovered notice through the gateway `agent` path instead of the existing direct outbound delivery seam.

## What This Changes

The PR adds `notifyAnnounceGiveUp` in the subagent registry and invokes it after announce retry exhaustion to send a best-effort gateway `agent` notification to the requester channel.

## Best Possible Solution

Implement terminal announce give-up recovery in the current lifecycle and delivery modules by factoring or reusing direct `sendMessage` fallback delivery for raw recovered result text with per-run idempotency, while preserving cleanup and ended-hook semantics.

## Reproduction Assessment

Yes, from source inspection. On current main, either a restored run with `announceRetryCount >= MAX_ANNOUNCE_RETRY_COUNT` or a live cleanup decision of `give-up` reaches cleanup without any final requester-channel send.

## Solution Assessment

No. A last-resort user notification is the right direction, but this PR routes it through gateway `agent`, which creates a requester-agent/model turn; the narrower maintainable fix is direct outbound fallback delivery of the recovered text.

## Review Findings
