## Summary

Keep open: the PR targets a real subagent delivery bug, but it is not mergeable as-is because current main moved retry-limit cleanup into the lifecycle controller, the PR still lacks after-fix real channel proof, and the raw-vs-agent-mediated delivery contract remains unsettled.

## What This Changes

The PR adds a `notifyAnnounceGiveUp` fallback in `src/agents/subagent-registry.ts` to send a best-effort gateway `agent` notification with recovered subagent output after announce retry-limit give-up.

## Best Possible Solution

Rebase the fallback into `src/agents/subagent-registry-lifecycle.ts`, cover live and resumed retry-limit give-up, and make the fallback explicitly match either the existing agent-mediated contract or a reviewed raw outbound path.

## Reproduction Assessment

Yes, at source level: current main reaches retry-limit give-up through `resolveDeferredCleanupDecision`, then lifecycle cleanup marks delivery failed and completes without a fallback notification. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No, not as-is. The fallback direction is reasonable, but the patch must move into the active lifecycle controller, add focused coverage, and resolve whether delivery is raw outbound text or an agent-mediated response.
