## Summary

Keep open: current main still has the terminal announce-give-up silent-drop path, but this PR is not safe as-is because its fallback goes through a requester agent/model turn instead of the existing direct outbound completion-send seam.

## What This Changes

The PR adds `notifyAnnounceGiveUp` in `src/agents/subagent-registry.ts` and calls it on resumed/live announce retry exhaustion to send recovered subagent results through gateway `agent` delivery.

## Best Possible Solution

Implement the terminal give-up notice in the current lifecycle and announce-delivery layers by reusing or factoring the direct `sendMessage` completion fallback, preserving per-run idempotency, killed-run suppression, cleanup semantics, logging, tests, and changelog coverage.

## Reproduction Assessment

Yes. Force `runSubagentAnnounceFlow` to return false until cleanup resolves to `give-up`, or restore a run with `announceRetryCount >= 3`; current main then records failed delivery and completes cleanup without a final requester-channel send.

## Solution Assessment

No. A last-resort user notice is the right direction, but this PR’s gateway `agent` call is not the narrowest maintainable fix because it turns the recovered notice into requester-agent input instead of delivering the recovered result through the existing outbound channel seam.
