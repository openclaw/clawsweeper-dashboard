## Summary

Keep open. The retry-limit give-up problem is still source-visible on current main, but this stale/conflicting PR should not merge as-is because its fallback uses gateway `agent` delivery, which runs another requester-agent/model turn instead of directly delivering the recovered result, and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a best-effort `notifyAnnounceGiveUp` fallback that tries to notify the requester channel with recovered subagent result text when announce retries are exhausted.

## Best Possible Solution

Implement terminal give-up recovery on current main in the lifecycle/delivery modules using one direct outbound delivery path with preserved requester origin, per-run idempotency, delivery status/error recording, focused regression tests, and redacted real behavior proof.

## Reproduction Assessment

Yes, from source inspection. Current main's resumed and live retry-limit give-up paths mark delivery failed and complete cleanup without a requester-channel recovery notifier; I did not run a live two-agent or Telegram scenario.

## Solution Assessment

No. A last-resort user notification is the right direction, but this PR routes recovery through gateway `agent`, so the requester model mediates the recovered text; a direct outbound delivery helper on current lifecycle modules is narrower and more predictable.
