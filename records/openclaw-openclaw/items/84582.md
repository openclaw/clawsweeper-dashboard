## Summary

Keep open: the patch is aimed at a real Telegram SOCKS proxy failure, but the current diff is incomplete and risks regressing existing plain HTTP proxy behavior while still lacking inspectable real-environment proof.

## What This Changes

The PR changes `createHttp1ProxyAgent` so OpenClaw only applies its `proxyTls` target when the proxy URI starts with `https:`.

## Best Possible Solution

Fix proxy-scheme handling in the shared helper layer so SOCKS proxies do not receive a truthy `proxyTls`, HTTP/HTTPS proxy endpoint connection policy is preserved, and both explicit and env Telegram proxy paths have regression coverage plus real proof.

## Reproduction Assessment

Yes, by source inspection: Telegram explicit SOCKS proxy transport reaches `createHttp1ProxyAgent`, and Undici 8.3.0 treats truthy `proxyTls` as TLS-to-SOCKS-proxy. I did not run a live SOCKS proxy repro in this read-only review.

## Solution Assessment

No. The direction is right for SOCKS, but the patch should cover the env proxy path and preserve HTTP proxy endpoint connect options instead of only gating `proxyTls` on `https:`.
