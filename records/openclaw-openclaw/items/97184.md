## Summary

Keep open: current main and v2026.6.10 still lack a plugin-owned emoji/icon field that reaches Discord status reactions, and adding one is a public plugin SDK/display contract decision rather than conservative cleanup.

## What This Changes

The issue requests plugin tool metadata that lets Discord and other status displays use custom per-tool emoji instead of the generic tool reaction.

## Best Possible Solution

Define one generic plugin-owned tool display metadata contract and thread it through status/progress display paths, with manifest, defineToolPlugin, docs, and Discord reaction coverage aligned.

## Reproduction Assessment

Not applicable as a runtime bug, but source inspection gives a high-confidence current-main check: plugin tool metadata cannot provide emoji/icon data consumed by Discord status reactions.

## Solution Assessment

Unclear until maintainers choose the public API shape. A generic plugin tool display metadata contract is safer than a Discord-only field or a hardcoded plugin tool table.
