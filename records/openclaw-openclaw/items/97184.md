## Summary

Keep open: current main and v2026.6.10 still resolve status reaction emojis from static core tool display metadata/category tokens, while plugin manifest, registerToolMetadata, and defineToolPlugin surfaces do not accept an emoji/icon field. This is a valid plugin-SDK product decision, not a narrow bug cleanup.

## What This Changes

Allow plugin-owned tool declarations or manifests to provide per-tool emoji/icon metadata for Discord status reactions and similar channel displays.

## Best Possible Solution

Define one generic plugin-owned tool display metadata contract, thread it through tool ownership into status/progress projections, and cover manifest, defineToolPlugin, docs, and Discord reaction tests together.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that plugin tools cannot currently declare emoji/icon metadata consumed by Discord status reactions.

## Solution Assessment

Unclear until maintainers choose the public API shape. A generic plugin tool display metadata contract is likely better than a Discord-specific field or hardcoded plugin tool table.
