## Summary

Keep open: current main and the latest stable release still route Discord tool status reactions through static core display metadata, while plugin manifest, registerToolMetadata, and defineToolPlugin surfaces do not expose a custom emoji/icon contract.

## What This Changes

The issue requests a plugin-facing emoji/icon metadata field for plugin tools so Discord and other status displays can show custom tool reactions.

## Best Possible Solution

Define one generic plugin-owned tool display metadata contract and thread it through tool ownership into status/progress projections, with manifest, defineToolPlugin, docs, and Discord status reaction coverage updated together.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that plugin tools cannot currently declare emoji/icon metadata consumed by Discord status reactions.

## Solution Assessment

Unclear until maintainers choose the public API shape. A generic plugin tool display metadata contract is likely better than a Discord-specific field or a hardcoded plugin tool table.
