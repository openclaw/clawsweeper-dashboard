## Summary

This should stay open: current main has generic status-reaction emoji selection for known tools, but plugin manifests, runtime tool metadata, and `defineToolPlugin` do not expose an emoji/icon field that Discord can consume.

## What This Changes

The issue requests a plugin tool metadata/API field for custom Discord status reaction emoji instead of the generic tool emoji.

## Best Possible Solution

Add one generic plugin-owned tool display metadata seam, then have shared status reactions consume it so Discord and other channels do not need plugin-specific hardcoded emoji tables.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a bug; source inspection shows the requested plugin-facing emoji metadata cannot currently flow from plugin manifests or `defineToolPlugin` into Discord status reactions.

## Solution Assessment

Is this the best way to solve the issue? The direction is plausible, but maintainers should pick a generic plugin tool display metadata contract rather than a Discord-only special case or another hardcoded table.
