## Summary

Keep open: current main and v2026.6.10 still route Discord tool status reactions through static core display metadata and category fallbacks, while plugin manifest, host-hook, and defineToolPlugin metadata surfaces have no emoji/icon field. This remains a plugin-SDK product decision, not a stale cleanup candidate.

## What This Changes

Request plugin-provided tools to declare per-tool emoji or icon metadata that Discord status reactions and other status displays can use instead of the generic tool emoji.

## Best Possible Solution

Define one generic plugin-owned tool display metadata contract and thread it through tool ownership into status/progress projections, with manifest, defineToolPlugin, docs, and Discord status reaction tests updated together.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that plugin tools cannot currently declare emoji/icon metadata consumed by Discord status reactions.

## Solution Assessment

Unclear until maintainers choose the public API shape. A generic plugin tool display metadata contract is likely better than a Discord-specific field or hardcoded plugin tool table.
