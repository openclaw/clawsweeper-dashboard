## Summary

Keep open: current main and v2026.6.10 still have Discord status reactions driven by static tool-name/category display metadata, while plugin manifest/API metadata has no emoji/icon field for plugin-owned tools.

## What This Changes

Allow plugin-provided tools to declare per-tool emoji/icon display metadata that Discord and other status/progress surfaces can use instead of the generic tool reaction.

## Best Possible Solution

Define one additive shared tool display metadata contract for plugin-owned tools, project it from manifest/runtime SDK metadata, and have Discord/status/progress surfaces consume it with generic fallbacks and focused docs/tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API gap. Source inspection confirms the current status reaction and plugin metadata paths have no per-plugin-tool emoji/icon contract.

## Solution Assessment

Unclear until maintainers choose the API shape. The maintainable direction is a shared plugin tool display metadata contract, not a Discord-only hardcoded table expansion.
