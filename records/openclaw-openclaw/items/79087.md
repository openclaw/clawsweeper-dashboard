## Summary

Keep open: current main still appears to allow this stale-prefix npm update failure because npm target-root resolution only trusts the package-root-derived prefix when a prefix-local npm binary exists, and otherwise falls back to `npm root -g`.

## What This Changes

The issue asks `openclaw update` to prefer the invoked OpenClaw package root over a stale npm global prefix when choosing the package install target.

## Best Possible Solution

Make npm target resolution use the trusted invoked package root for standard npm global layouts, keep the fallback to `npm root -g` when that root cannot be proven, and surface any root disagreement in dry-run/JSON output.

## Reproduction Assessment

Yes, at source level. A focused test can pass a trusted package root under `<prefix>/lib/node_modules/openclaw`, make `npm root -g` return `/usr/lib/node_modules`, omit `<prefix>/bin/npm`, and observe current `resolveGlobalInstallTarget` choose the stale npm root.

## Solution Assessment

Yes: the narrow fix belongs in the update install-target resolver and update diagnostics, not in a docs-only workaround or a new config option. The implementation should avoid weakening generic path-shape-only manager detection for untrusted roots.
