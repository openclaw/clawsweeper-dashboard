## Summary

Keep open: current main still routes npm target resolution through `npm root -g` unless a prefix-local npm binary exists, so the reported user-prefix install can still stage under a stale `/usr/lib/node_modules` root.

## What This Changes

The issue asks `openclaw update` to prefer the invoked CLI package root over a stale npm global prefix and report root mismatches in dry-run/JSON output.

## Best Possible Solution

Update npm install-target resolution to use the invoked/current OpenClaw package root's npm global layout as the mutation target when npm's reported root disagrees, surface the mismatch in dry-run/JSON, and keep regression coverage around the stale-prefix case.

## Reproduction Assessment

Yes, at source level: set `pkgRoot` to a user npm prefix package root, make `npm root -g` return `/usr/lib/node_modules`, and leave the prefix-local npm binary absent; current `resolveGlobalInstallTarget` will select the stale npm root. I did not run the mutating updater live in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable fix is in the install-target resolver and update diagnostics, using the already-discovered current package root instead of requiring operators to set `NPM_CONFIG_PREFIX`. A docs-only workaround would leave the documented update path broken.
