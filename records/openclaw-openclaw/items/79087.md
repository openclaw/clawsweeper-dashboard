## Summary

Keep open: current main still appears able to select the stale `npm root -g` target for npm package updates even after resolving the invoked OpenClaw package root, which matches the reported production failure.

## What This Changes

Request that `openclaw update` prefer the install root resolved from the invoked binary over a stale npm global prefix and expose root mismatches in dry-run/JSON output.

## Best Possible Solution

Teach npm target-root resolution to derive the npm prefix/global root from the invoked package root when that root has a normal npm global layout, report both roots when npm config disagrees, and cover the stale-prefix case with focused regression tests.

## Reproduction Assessment

Yes, at source level: a focused test can set the invoked package root to a user npm prefix while `npm root -g` returns `/usr/lib/node_modules` and observe `resolveGlobalInstallTarget` choose the stale npm root. I did not run a live mutating updater reproduction in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix belongs in the install-target resolver and dry-run/update result diagnostics, preserving the staged npm swap behavior rather than requiring operators to set `NPM_CONFIG_PREFIX`. A docs-only workaround would not satisfy the documented update path.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79087. `openclaw update` currently resolves the invoked package root from `process.argv[1]`, but npm install-target resolution can still choose stale `npm root -g` when the OpenClaw package lives under a user npm prefix and that prefix does not contain its own `bin/npm`. Update the npm target-root logic so a normal package root like `<prefix>/lib/node_modules/openclaw` is preferred as the mutation target over a disagreeing npm-config root when it came from the invoked OpenClaw binary/package root. Preserve the existing staged npm install and package verification flow. Surface mismatched invoked-root vs npm-config-root information in dry-run/JSON output where practical. Add focused regression coverage for stale `npm root -g` returning `/usr/lib/node_modules` while the invoked package root is under a user prefix. Add a single-line user-facing changelog entry under `CHANGELOG.md` because this fixes the documented update path. Do not add a new config option or require operators to set `NPM_CONFIG_PREFIX`.
