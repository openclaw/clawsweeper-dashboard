## What This Changes

The PR changes plugin registry snapshot validation to accept persisted bundled plugin records under a legacy `extensions` root and adds a regression test for that path.

## Best Possible Solution

Accept legacy source roots only when they match the discovery-valid source-checkout/source-only or mounted-overlay cases, preserve stale-root rejection, and prove it with a current-index fixture plus current-head refresh/list CLI output.

## Reproduction Assessment

Yes, source-reproducible: the linked issue and current source show that a source checkout can persist bundled plugin roots under both `dist/extensions` and `extensions`, then current validation rejects entries outside the single resolved bundled root. I did not run the CLI sequence in this read-only review.

## Solution Assessment

No. The compatibility direction is right, but this implementation is too broad because it trusts the whole legacy root instead of the narrower roots current discovery can produce or intentionally overlay.

Full review comments:

- **[P1] Gate legacy-root reuse to discovery-valid roots:** `src/plugins/plugin-registry-snapshot.ts:337`
  - body: The new predicate trusts any persisted bundled plugin under the computed legacy `extensions` root. Discovery is narrower: it accepts source-checkout roots only after source-checkout checks, skips children already present in the active bundled root, and treats copied source dirs as inert unless they are mounted overlays. As written, a stale persisted registry can stay on the fast path for a root current discovery would reject.
  - confidence: 0.88
- **[P2] Build the regression fixture with the current index contract:** `src/plugins/plugin-registry-snapshot.test.ts:841-851`
  - body: The added test writes an `InstalledPluginIndex` with retired or incomplete fields like `meta`, `id`, and `commitIsh`, and it sets the active bundled root to `<tmp>/bundled`, which does not exercise `buildLegacyBundledRootPath`. Please build the fixture from `loadInstalledPluginIndex` using a packaged `dist/extensions` root so the test proves the actual persisted schema and alias path.
  - confidence: 0.9
