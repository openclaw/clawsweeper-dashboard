## Summary

Keep open: the target source-checkout bug is real, but this latest predicate accepts every persisted bundled record under the legacy source `extensions` root, including copied legacy roots that current discovery intentionally ignores, so it can trust stale bundled paths instead of rebuilding.

## What This Changes

The PR updates plugin registry snapshot validation so persisted bundled plugin records under the legacy source `extensions` root are not treated as stale when the active bundled root is a packaged bundled root.

## Best Possible Solution

Narrow legacy-root acceptance to the same source-checkout or mounted-overlay paths that discovery can produce, keep copied or stale legacy roots rejected, and add focused registry tests for accepted and rejected cases.

## Reproduction Assessment

Yes. Source inspection plus the linked issue gives a high-confidence path: a persisted bundled root under source `extensions/<id>` is outside current `resolveBundledPluginsDir` and current main rejects it; I did not run the CLI because this review is read-only.

## Solution Assessment

No. The fix should align the stale-root validator with discovery-valid source checkout or mounted-overlay roots instead of accepting every legacy-root persisted bundled path.
