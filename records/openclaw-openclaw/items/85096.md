## Summary

Keep open for maintainer review. The PR has sufficient real behavior proof and is in the right Policy plugin area, but the patch can report read-only/no-write workspace conformance for `sandbox.mode: "non-main"` even though main sessions stay unsandboxed, and it ignores the supported sandbox-specific tool deny policy path.

## What This Changes

The branch adds Policy plugin `agents.workspace.allowedAccess` and `agents.workspace.denyTools` checks, agent workspace evidence, focused tests, and policy documentation.

## Best Possible Solution

Keep the Policy-owned conformance surface, but align it with effective sandbox runtime and sandbox tool-policy semantics, add tests for `off`/`non-main`/`all` plus sandbox deny sources, then merge after maintainer approval.

## Reproduction Assessment

Not applicable as a user bug report, but the review findings are source-reproducible: the PR treats `non-main` as sandbox-enabled while current runtime code and tests show main sessions remain direct in that mode.

## Solution Assessment

No, not yet: the Policy plugin is the right owner, but the implementation should use the effective sandbox mode/tool-policy contract rather than a non-off mode check and top-level-only deny scanning.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Require a sandbox mode that covers main sessions:** `extensions/policy/src/policy-state.ts:722`
  - body: `sandboxEnabled` is true for `mode: "non-main"`, so a config with `workspaceAccess: "ro"` can pass this read-only/no-write policy even though current runtime keeps the agent main session unsandboxed. That gives operators a clean Policy attestation for a host-running main session; require `mode: "all"` for this rule or add an explicit policy mode contract with tests for `off`, `non-main`, and `all`.
  - confidence: 0.9
- **[P2] Honor sandbox-specific deny policy:** `extensions/policy/src/policy-state.ts:727-728`
  - body: The scanner only checks `tools.deny` and `agents.list[].tools.deny`, but sandboxed runtime denials are also configured through `tools.sandbox.tools.deny` and per-agent `agents.list[].tools.sandbox.tools.deny`. A workspace that already denies `group:runtime`/`group:fs` through the supported sandbox policy path will still fail `agents.workspace.denyTools`, so this creates a conflicting conformance rule; read the effective sandbox tool policy or include those sources.
  - confidence: 0.86
