## Summary

Keep open. Current main still routes Google/Gemini LLM calls through the shared guarded provider fetch path without opting into trusted env-proxy mode for ambient HTTP_PROXY/HTTPS_PROXY, and the focused fix PR is still an open implementation candidate rather than shipped code.

## What This Changes

The issue asks OpenClaw provider LLM requests, especially Google/Gemini, to honor HTTP(S)_PROXY and surface clear failures instead of hanging silently in proxy-required environments.

## Best Possible Solution

Land #72480 or an equivalent narrow shared provider-fetch change with regression coverage for ambient HTTPS_PROXY, explicit dispatcher preservation, and strict-mode SSRF behavior.

## Reproduction Assessment

Yes for the source-level routing gap: Google LLM requests reach buildGuardedModelFetch, which calls fetchWithSsrFGuard without trusted env-proxy mode while strict mode ignores ambient proxy env. The exact UI hang remains a live proxy/Gemini confirmation step for the fix.

## Solution Assessment

Yes, the best maintainable direction is a shared guarded-fetch seam change rather than Google-specific proxy handling. The safe shape is to select trusted env-proxy mode only when shouldUseEnvHttpProxyForUrl matches and no explicit provider dispatcher policy exists.
