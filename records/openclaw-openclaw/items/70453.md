## Summary

Keep open. Current main still leaves Google LLM requests on the shared guarded provider fetch path without selecting trusted env-proxy mode for ambient HTTP_PROXY/HTTPS_PROXY, and the related open PR is an implementation candidate rather than a merged fix.

## What This Changes

The issue asks OpenClaw to route Gemini LLM requests through configured HTTP(S) proxy environment variables and surface clear CLI/WebUI errors instead of hanging silently.

## Best Possible Solution

Keep this issue open until #72480 or an equivalent narrow provider-fetch change lands with regression coverage for ambient HTTPS_PROXY use, explicit dispatcher preservation, and strict-mode SSRF behavior.

## Reproduction Assessment

Yes for the source-level routing gap: a Google LLM request reaches buildGuardedModelFetch, which calls fetchWithSsrFGuard without trusted env-proxy mode, while strict mode ignores ambient proxy env by design. The exact UI hang still needs a live proxy/Gemini confirmation during the fix.

## Solution Assessment

Yes, the best maintainable direction is a shared guarded-fetch seam change, not Google-specific proxy handling. The safe shape is to select trusted env-proxy mode only when shouldUseEnvHttpProxyForUrl matches and no explicit provider dispatcher policy exists.
