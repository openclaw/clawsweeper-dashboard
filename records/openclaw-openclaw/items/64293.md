- `P2`: This is a normal-priority behavior bug with real token-cost impact but a bounded heartbeat configuration surface.
- `impact:auth-provider`: The issue concerns unintended scheduled model/provider calls and token spend caused by heartbeat configuration resolution.

## Summary

Keep open. Current main and the latest release still accept `agents.defaults.heartbeat: {}`, treat it as enabled, and resolve the missing cadence to the default interval; the open closing PR remains unmerged/conflicting and related heartbeat work leaves the empty-object contract unsettled.

## What This Changes

The issue asks OpenClaw to treat an explicit empty heartbeat config as disabling automatic heartbeat polling instead of resolving it to the default cadence.

## Best Possible Solution

Define one heartbeat disable/absence contract, then implement it through a shared resolver used by scheduler selection, interval resolution, status, prompt/bootstrap, ACP spawn, provider defaulting, docs, and regression tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: configure `agents.defaults.heartbeat: {}` and current main treats the object as enabled while resolving the missing cadence to the default interval. I did not run a live gateway or remeasure provider usage in this read-only review.

## Solution Assessment

Unclear until maintainers choose the empty-object config contract. The maintainable path is one shared heartbeat policy resolver, not another isolated check in only one helper.
