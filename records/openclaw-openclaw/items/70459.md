## Summary

This is a focused docs PR, but the current draft is still incorrect against current main: the added prerequisite snippets fail to report missing tools, the installer wording overstates what the installers can bootstrap, and the branch carries an outdated Node 22.14 floor while main now requires 22.16+.

## What This Changes

The PR changes docs/start/getting-started.md to add prerequisite checks, install hints, and a pnpm/source-build note to the Getting started guide.

## Best Possible Solution

Keep the Step 0 docs direction, but update the draft to match the installer contract and current Node floor, use reliable command-existence/status checks in both shells, and validate the rendered docs plus snippets before merge.

Yes for the blocking review findings: the Bash snippet reproduced locally with a deliberately missing command, and the PowerShell wrapper path returned a nonzero external status without entering catch. The PR itself is a docs change, so there is no current-main product regression to reproduce.

## Solution Assessment

No, the PR is not ready as written. The maintainable path is a narrow docs correction that preserves the prerequisite guidance while fixing the snippets, installer-scope wording, and Node version floor.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Keep the Node floor at 22.16+:** `docs/start/getting-started.md:21`
  - body: This new prerequisite line says Node 22.14+ is supported, but current main now requires `>=22.16.0` and the install docs also say Node 22.16+. Merging this branch would reintroduce outdated setup guidance for first-time users.
  - confidence: 0.92
- **[P2] Scope installer auto-install wording:** `docs/start/getting-started.md:38`
  - body: The new text says the installer auto-installs anything missing, but the installers only remediate specific setup pieces such as Node, Git, and source-install pnpm; bootstrap tools like a downloader and shell still need to be present. This overpromises the first-run path.
  - confidence: 0.94
- **[P2] Check status before piping to head:** `docs/start/getting-started.md:46`
  - body: The `|| echo "MISSING"` fallback is attached to `head`, so a missing command prints the shell error and never emits the promised `MISSING` marker. Capture or test the command status before formatting the first output line.
  - confidence: 0.97
- **[P2] Probe executables before calling cmd:** `docs/start/getting-started.md:65`
  - body: `cmd /c` can return a nonzero status for a missing tool without entering the PowerShell `catch` block, so Windows users also do not get the promised `MISSING` marker. Use `Get-Command` or explicit exit-status handling before running each command.
  - confidence: 0.9
