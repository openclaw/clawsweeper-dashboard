## Summary

Keep this issue open. Current main still has a source-reproducible WebChat TTS delivery gap: trusted local TTS audio becomes a gateway `type: "audio"` base64 block, the chat display projection redacts the playable bytes, and the Control UI only renders audio through attachment or `MEDIA:` paths. The related `MEDIA:` inline-player issue is closed, and the telemetry request remains separate at https://github.com/openclaw/openclaw/issues/57297.

## What This Changes

WebChat TTS should surface generated audio as audible playback or downloadable media when generation succeeds, matching the documented audio-attachment behavior outside voice-note channels.

## Best Possible Solution

Route trusted generated WebChat TTS audio through the existing safe assistant-media or attachment surface so final broadcasts and history carry a playable/downloadable attachment while raw base64 audio remains redacted from display history.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a successful trusted local TTS media payload becomes a base64 gateway audio block, `chat.final`/history projection removes the bytes, and the Control UI only plays normalized attachments or `MEDIA:` audio; I did not run a live browser session.

## Solution Assessment

Yes. Keeping this open and fixing the media routing path is the narrow maintainable solution; the separate logging/status request at https://github.com/openclaw/openclaw/issues/57297 should not be used as a substitute for playable delivery.
