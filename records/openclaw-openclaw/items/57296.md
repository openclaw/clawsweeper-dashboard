## Summary

Keep this issue open. Current main partially improves the related WebChat audio-attachment surface, but the TTS path still converts trusted local audio into gateway `type: "audio"` blocks that are projected/redacted before the Control UI can render them as playable or downloadable audio.

## What This Changes

WebChat TTS should surface generated speech as audible playback and/or a downloadable audio item in the Control UI instead of only reporting tool success.

## Best Possible Solution

Fix WebChat TTS delivery end to end by turning trusted generated audio into a Control UI-renderable attachment or authenticated media URL, or by safely normalizing gateway audio blocks before redaction, with regression coverage for `chat.send`, `chat.final`, `chat.history`, and Control UI rendering.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a successful trusted local TTS media payload becomes a base64 gateway audio block, `chat.final`/history projection removes the bytes, and Control UI only renders audio for attachment/MEDIA paths. I did not run a live browser session.

## Solution Assessment

Yes, keeping this open is the best path because current main only solves the related `MEDIA:` attachment rendering gap, not the TTS audio-block delivery path. The narrow maintainable fix is to route TTS audio through the existing safe attachment/media surface or add an equivalent safe UI normalization path.
