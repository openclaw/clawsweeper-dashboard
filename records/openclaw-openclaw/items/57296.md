## Summary

Keep this issue open. Current main has improved WebChat audio attachment support for MEDIA/path-style attachments, but the TTS tool path still produces backend audio content blocks that the Control UI does not normalize into playable/downloadable attachments, and chat display projection strips base64 audio data from projected events/history.

## What This Changes

WebChat TTS should provide audible playback or downloadable audio in the Control UI when the TTS tool reports success.

## Best Possible Solution

Keep this issue open and fix WebChat TTS delivery end to end. The preferred product/code direction is to turn trusted local TTS output into a durable renderable WebChat attachment or authenticated media URL, or otherwise add a safe live-only audio-block normalization path, with regression coverage for chat.send, chat.final/chat.history projection, and Control UI rendering while preserving local media trust, root checks, and size limits.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: emit a trusted local TTS media payload through WebChat, observe backend audio content blocks in the gateway tests, then trace that projected chat events/history redact source.data and the UI normalizer does not convert audio content blocks into playable attachments.

## Solution Assessment

No current implementation is not sufficient. The existing MEDIA attachment audio player is useful, but the TTS tool path needs a durable attachment/media URL or explicit UI support for gateway audio content blocks instead of relying on base64 content that is stripped before rendering.
