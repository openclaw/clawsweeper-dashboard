## Summary

Keep this open. Current main still builds WebChat TTS media as gateway audio content blocks that are redacted before display and are not normalized into the Control UI attachment path, so the documented audio-attachment behavior is not satisfied for WebChat.

## What This Changes

The issue requests audible or downloadable WebChat delivery for successful TTS output that already works through Telegram.

## Best Possible Solution

Route trusted generated TTS audio through the existing safe WebChat attachment/media surface, or add an equivalent authenticated media URL normalization path, while keeping byte redaction for history.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: generated TTS becomes a trusted local audio payload, WebChat turns it into a base64 audio block, final/history projection redacts the bytes, and Control UI does not normalize that block into a playable attachment; a user also confirmed the behavior on v2026.5.3.

## Solution Assessment

Yes. Keeping this open and fixing the WebChat media routing is the narrow maintainable solution; the related inline-player work solves `MEDIA:` audio attachments but not this TTS audio-block delivery path.
