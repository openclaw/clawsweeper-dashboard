## Summary

Keep open: this protected maintainer-labeled PR is addressing a real trajectory cleanup stall, but the current head still has a blocking path where runtime capture treats a queued write as accepted even when the queued writer later skips the append at the file cap.

## What This Changes

The PR bounds trajectory runtime payloads, live capture size, queued sidecar writes, and trajectory docs/tests/changelog for the flush stall reported in #77124.

## Best Possible Solution

Keep the bounded recorder/writer direction, but make the recorder or writer account for existing sidecar bytes and max-file skips before landing so the linked bug closes through this PR.

## Reproduction Assessment

Yes, source-level. Current main serializes runtime events before queueing and lets cleanup continue after the 10s timeout, and the PR blocker is also source-reproducible from the writer max-file no-op path.

## Solution Assessment

No, not as written. The bounded capture approach is the right maintainable direction, but it needs to treat max-file skips or pre-existing sidecar size as dropped capture before merge.
