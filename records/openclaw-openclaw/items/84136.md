## Summary

Keep open: the DNS-label/path validation is useful and the latest commit fixes the `dns setup` explicit-domain diagnostic, but the shared resolver still collapses invalid configured domains to `null` for gateway/status/discovery callers and the external PR proof is test-only.

## What This Changes

The PR adds DNS-label validation and path containment for wide-area DNS zone generation, plus CLI/tests for invalid `dns setup --domain` inputs.

## Best Possible Solution

Keep the validation and path containment, but preserve a distinct invalid-domain result for configured values across shared callers, add caller coverage, and require redacted runtime CLI or gateway proof before merge.

## Reproduction Assessment

Yes by source inspection: current main normalizes domain strings without DNS-label validation and derives zone filenames from that text; the PR diff also shows the remaining invalid-config-to-null path. I did not run tests because this review is read-only.

## Solution Assessment

No as written. The validation boundary is right, and the CLI-specific fix is useful, but the shared resolver should not make configured invalid values indistinguishable from missing configuration unless maintainers explicitly accept that behavior.
