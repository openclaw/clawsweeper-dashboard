## Summary

Keep open: the PR addresses a real wide-area DNS zone path hardening gap, but the new validation can escape through the best-effort gateway discovery startup path before the existing write-error catch. It also has only unit/mock proof, so real behavior proof is still needed before merge.

## What This Changes

The PR adds DNS-label validation and path containment for wide-area DNS zone filenames and expands widearea-dns tests for invalid domains.

## Best Possible Solution

Keep the DNS-label and path containment hardening in the zone helper, but handle invalid optional wide-area domains at the gateway discovery boundary by warning and skipping publishing, with a startup regression test and redacted real runtime proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: call gateway discovery with `wideAreaDiscoveryEnabled: true` and an invalid `wideAreaDiscoveryDomain`, and the PR's new normalize throw occurs before the existing write try/catch.

## Solution Assessment

No, not as written. The zone helper validation is the right hardening direction, but invalid optional discovery config should be caught at the gateway discovery boundary unless maintainers intentionally choose a fail-closed startup policy.
