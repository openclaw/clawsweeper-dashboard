## Summary

Keep open: the PR targets the right wide-area DNS file-write boundary, but the current patch still collapses invalid configured domains into the same `null` value used for missing config, so CLI and gateway callers can show misleading no-domain behavior instead of an invalid-domain diagnostic.

## What This Changes

The PR adds DNS-label validation and DNS config-directory path containment for wide-area DNS zone generation, plus regression tests for invalid domain strings.

## Best Possible Solution

Keep the DNS-label validation and path containment, but preserve an explicit invalid-domain result through callers, add user-facing CLI/gateway coverage, and require redacted runtime proof of the chosen diagnostic.

## Reproduction Assessment

Yes by source inspection. Current main derives zone filenames from normalized-but-unvalidated domain text, and the PR diff shows invalid configured domains become `null` in the shared resolver instead of a distinct invalid result.

## Solution Assessment

No as written. The validation and path-containment boundary is appropriate, but swallowing invalid configured values is not the narrowest maintainable behavior for user-facing CLI and gateway callers.
