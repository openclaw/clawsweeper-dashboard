## Label Justifications

- `P2`: This is a focused gateway DNS hardening PR with limited surface area, but it has a merge-blocking compatibility/availability concern.
- `merge-risk: 🚨 compatibility`: The PR changes previously accepted configured domain strings into validation errors without an upgrade path or documented maintainer decision.
- `merge-risk: 🚨 availability`: Because validation now throws before gateway discovery's existing writer catch, an invalid wide-area domain can reject startup/discovery work instead of only skipping the zone update.

## Summary

Keep open: the hardening is useful, but the patch currently moves strict validation into shared domain resolution, which can make existing invalid wide-area configs throw before gateway discovery's handled zone-write warning path; the external PR also has unit-test-only behavior proof.

## What This Changes

The PR adds DNS-label validation and DNS-directory path containment checks for wide-area discovery domains, plus regression tests for invalid domain strings and valid zone path placement.

## Best Possible Solution

Keep the DNS-label validation at the wide-area zone filename/rendering boundary while preserving handled gateway startup and discovery behavior, with regression coverage for invalid configs and redacted CLI or gateway runtime proof.

## Reproduction Assessment

Yes by source inspection: current main derives the zone filename from normalized-but-unvalidated domain text, and the PR-introduced startup concern is traceable because the resolver is called before the discovery writer try/catch. I did not run tests because this review is read-only.

## Solution Assessment

No as written. The validation itself is the right direction, but the patch should either keep invalid-domain failures inside the handled zone-writing path or add explicit caller handling/tests for gateway startup, discovery, and status before merge.
