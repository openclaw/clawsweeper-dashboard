## Summary

Keep this PR open but not merge-ready: the DNS validation and path containment are useful, but the shared resolver still collapses malformed configured domains to “not configured,” and the external proof is only targeted tests.

## What This Changes

The PR validates wide-area DNS domains before zone file path and zone text generation, and adds `dns setup` CLI/test coverage for invalid `--domain` inputs.

## Best Possible Solution

Keep the DNS-label validation and path containment, but change the resolver contract to distinguish missing from invalid configured values across callers, add focused caller coverage, and merge after redacted runtime CLI or gateway proof.

## Reproduction Assessment

Yes by source inspection: current main normalizes `foo/bar` and derives a DNS zone path from that text, while the PR adds tests for invalid-domain rejection. I did not run tests because this review is read-only.

## Solution Assessment

No as written. The validation boundary is right, but configured invalid values should not be collapsed into the same `null` state as missing config unless maintainers explicitly accept that compatibility behavior.
