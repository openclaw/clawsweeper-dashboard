## Summary

Keep open: the PR fixes the right DNS zone path-safety boundary and now supplies real CLI proof, but the shared resolver still collapses invalid configured domains into “missing,” so background callers can silently disable wide-area discovery with a misleading operator warning.

## What This Changes

Adds DNS-label validation and DNS-directory path containment for wide-area DNS helpers, validates `dns setup` input, and adds helper/CLI/gateway regression tests for invalid domain strings.

## Best Possible Solution

Keep the DNS-label validation and zone path containment, but preserve a distinct invalid-domain state through the resolver and affected callers so operators get explicit invalid-config diagnostics without reintroducing unsafe writes.

## Reproduction Assessment

Yes by source inspection: current main normalizes domain strings without DNS-label validation and derives zone filenames from that text. I did not run tests because this review is read-only.

## Solution Assessment

No as written. The validation boundary is right, but collapsing invalid configured domains into `null` is not the narrowest maintainable solution for user-facing and operator-facing callers.
