## Summary

Keep open: current main still has the successful `compact_only` overflow-recovery path, but I did not find a post-recovery retained-context ratio, degraded-context flag, or user-visible low-retention warning.

## What This Changes

The issue asks OpenClaw to surface an operator/user-visible signal when overflow recovery compacts a turn down to a tiny fraction of the expected context.

## Best Possible Solution

Add a generic overflow-recovery degradation signal at the runner/session boundary, compute retained context from before/after recovery facts, and render it in operator-visible status/session surfaces before considering agent-facing prompt text.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection plus the reporter's logs show the compact-only precheck and successful retry path, but I did not replay the external context-engine scenario on current main.

## Solution Assessment

Unclear until maintainers choose the product surface. A generic runner/session metadata signal with status and dashboard rendering looks safer than a context-engine-specific workaround or a first-pass agent prompt change.
