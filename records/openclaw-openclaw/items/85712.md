## Summary

Keep open: current main still has the compact-only overflow recovery path and generic compaction/status indicators, but no retained-context ratio, degraded-context event, assistant message flag, or user-visible warning for the low-retention recovery described here.

## What This Changes

Request a user-visible degraded-context signal when `compact_only` overflow recovery trims the model-visible context down to a small fraction of the pre-recovery context.

## Best Possible Solution

Add a generic degraded-context metadata/event at the overflow recovery boundary that compares pre-recovery pressure with recovered assembled context, then surface it in `/status` and session UI without making the fix context-engine-specific.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection shows the reported `compact_only` precheck and successful overflow compaction retry path, and the issue provides detailed logs, but I did not replay the lossless-claw scenario on current main.

## Solution Assessment

Unclear until maintainers choose the product surface. The narrowest maintainable direction appears to be a generic degraded-context metadata/event signal from the runner, with dashboard and `/status` rendering layered on top rather than a context-engine-specific workaround.
