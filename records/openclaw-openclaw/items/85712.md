## Summary

Keep open: current main still has the compact_only overflow recovery path and generic compaction/status signals, but I did not find retained-context-ratio metadata or a user-visible degraded-context warning for the severe post-recovery shrink described here. The broader context visibility issue at https://github.com/openclaw/openclaw/issues/67690 is related but does not fully own this compact_only recovery signal.

## What This Changes

Request a user-visible warning, metadata, or status signal when compact_only overflow recovery sends a much smaller recovered context than the pre-recovery assembled context.

## Best Possible Solution

Define a generic degraded-context recovery signal in the runner/session metadata, based on pre-recovery pressure and recovered assembled size, then render it in /status and Control UI without making it context-engine-specific.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live reproduction was run, but the source path is reproducible: compact_only precheck leads to overflow recovery compaction and retry without a retained-context warning field or surfaced degraded-context signal.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the product surface and threshold semantics; the maintainable direction is a generic runner/session signal with UI/status rendering, not a lossless-claw-specific workaround.
