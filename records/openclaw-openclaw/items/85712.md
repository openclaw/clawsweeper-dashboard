## Summary

Keep open: current main has the compact-only overflow recovery path and generic compaction/status surfaces, but I did not find a user-visible warning or retained-context ratio for the specific low-retention recovery described here. The report is also more specific than the broader context-pruning visibility issue at https://github.com/openclaw/openclaw/issues/67690.

## What This Changes

Request to warn or annotate turns when compact-only overflow recovery trims the delivered model context to a small fraction of the originally assembled context.

## Best Possible Solution

Add a bounded degraded-context signal at the overflow recovery boundary that compares pre-recovery pressure with the recovered assembled context, then surface that signal in operator-visible status/session UI without changing normal compaction semantics.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does show the reported compact_only precheck and successful overflow compaction retry path, and the issue provides detailed logs, but I did not replay the lossless-claw scenario on current main.

## Solution Assessment

Unclear until maintainers choose the product surface. The narrowest maintainable direction appears to be a generic degraded-context metadata/event signal from the runner, with dashboard and /status rendering layered on top rather than a context-engine-specific workaround.
