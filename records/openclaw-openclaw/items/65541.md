## Summary

Keep this PR open. Current main and the latest release still implement and document disabled block streaming as final-reply-only for text, so the PR is not obsolete; however, the proposed change intentionally alters that contract and expands direct-send dedupe to ordinary text while same-channel block delivery is still queued, so it needs maintainer review before merge.

## What This Changes

The PR changes the disabled block-streaming fallback in `createBlockReplyDeliveryHandler` to send text and media block payloads immediately and updates reply-delivery tests to expect direct delivery.

## Best Possible Solution

Keep the PR open until maintainers decide whether disabled block streaming should remain final-only or become progressive for naturally bounded text. If progressive disabled-streaming delivery is desired, update the handler, docs, tests, final-payload dedupe, and the same-channel dispatcher await/fallback contract together; otherwise close the PR as a product decision and leave #32868 as the canonical same-channel delivery-ordering bug.

## Reproduction Assessment

Yes for the current-main behavior: the unit path is `createBlockReplyDeliveryHandler` with `blockStreamingEnabled: false` and a text-only payload, and current tests assert `onBlockReply` is not called. The live Discord claim was not re-run in this read-only review, but the source and tests reproduce the buffering behavior with high confidence.

## Solution Assessment

No, not as-is. The code change is narrow, but it changes documented disabled-streaming semantics and relies on direct-send dedupe before same-channel transport success is known; the safer solution is a maintainer-approved contract change that also handles dispatcher delivery confirmation and docs.
