## Summary

Current main covers the symlink-target use case with the safer explicit `skills.load.allowSymlinkTargets` contract, including docs and regression coverage. This branch should not be merged as-is because its automatic cross-root allowlist includes bundled skills and can bypass `skills.allowBundled`.

## What This Changes

The PR adds a cross-root allowed-realpath list to skill discovery so symlinked skill folders resolving into other configured roots load without warning, plus two symlink regression tests.

## Best Possible Solution

Keep the explicit trusted-target contract on main and direct any future automatic-root-trust proposal through a separate product/security review.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: pre-fix containment accepted only realpaths under the source root, while the PR and current-main tests model symlinked skill roots resolving elsewhere. I did not run the test command because this review is read-only.

## Solution Assessment

No. The PR branch is not the best solution because it trusts all configured roots, including bundled skills, whereas current main uses a narrower explicit `allowSymlinkTargets` contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep bundled skills gated by bundled source checks:** `src/agents/skills/workspace.ts:476`
  - body: Adding `bundledSkillsDir` to the cross-root allowlist lets a symlink under a workspace, managed, or personal root resolve into a bundled skill, but that skill is loaded with the caller's source. Since `skills.allowBundled` only filters entries whose source is `openclaw-bundled`, a bundled skill excluded by config can be reintroduced through the symlink.
