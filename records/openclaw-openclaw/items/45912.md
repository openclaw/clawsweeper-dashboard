## Summary

Keep this PR open, but do not merge it as-is. Current main still uses strict same-root realpath containment, while this PR broadens a security-sensitive skills boundary and can bypass bundled-skill allowlist semantics when a non-bundled symlink targets bundled skills.

## What This Changes

The PR threads an allowed configured-root realpath list through skill containment checks and adds regression tests for workspace and managed symlinks that point into other configured skill roots.

## Best Possible Solution

Implement one source-preserving symlink policy for configured skill roots: allow safe cross-root links among explicitly trusted non-bundled roots, keep external escapes rejected, and either exclude bundled targets from non-bundled aliases or relabel bundled targets so bundled allowlist checks still apply. Cover cross-root success, external rejection, bundled allowlist enforcement, grouped skills, and the docs contract.

## Reproduction Assessment

Yes. On the PR head, configure `skills.allowBundled` to exclude a bundled skill, then create a workspace or managed symlink to that bundled skill; the new allowed-root check accepts the realpath while the loaded entry keeps a non-bundled source. I did not run tests because this review was required to stay read-only.

## Solution Assessment

No. Allowing cross-root symlinks among configured non-bundled roots may be viable, but treating bundled roots as ordinary trusted targets without preserving bundled source semantics is not the narrowest maintainable fix.
