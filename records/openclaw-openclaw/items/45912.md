## Summary

Close as mostly implemented: current main supersedes this stale branch with an explicit trusted symlink target contract, while the branch's remaining automatic cross-root trust is riskier because it can include bundled skills and bypass bundled-skill allowlist policy.

## What This Changes

The PR threads a configured skill-root allowlist through skill path containment and adds regression coverage for symlinked skills pointing from workspace or managed roots into other configured roots.

## Best Possible Solution

Keep the current explicit `skills.load.allowSymlinkTargets` contract and close this branch rather than rebasing an older automatic cross-root trust policy.

## Reproduction Assessment

Yes, at source level: create a symlinked skill directory whose real path leaves its source root, then load workspace skill entries and observe whether containment accepts it or warns. Current main has focused regression coverage for the explicitly trusted-target variant, but I did not run tests in this read-only review.

## Solution Assessment

No for this PR branch: automatically trusting every configured root is broader than needed and creates a bundled allowlist bypass. The current main solution is narrower because operators must explicitly list trusted symlink target roots.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep bundled skills out of the symlink target allowlist:** `src/agents/skills/workspace.ts:476`
  - body: This includes `bundledSkillsDir` in the trusted roots used by non-bundled sources. A symlink under a workspace or managed root that resolves into bundled skills would load that skill as a non-bundled source, so `skills.allowBundled` no longer filters it.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch widens the skill trust boundary and can bypass bundled-skill allowlist policy.

Concerns:

- **[medium] Bundled allowlist bypass:** `src/agents/skills/workspace.ts:476`
  - body: Because bundled allowlist enforcement depends on the skill source being `openclaw-bundled`, automatically allowing symlinks into `bundledSkillsDir` lets a bundled skill be loaded through a higher-precedence non-bundled source and avoid operator `skills.allowBundled` policy.
  - confidence: 0.82
