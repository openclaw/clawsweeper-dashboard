## Summary

Keep open, but do not merge as-is: the branch is a focused skills symlink proposal, yet its implicit trusted-root list introduces a bundled-skill allowlist bypass, and current main has moved toward an explicit `skills.load.allowSymlinkTargets` policy that still needs maintainer reconciliation with the open symlink threads.

## What This Changes

The PR threads allowed realpath roots through skill containment checks and adds tests for workspace/managed skill symlinks resolving into project or extra skill roots.

## Best Possible Solution

Pick one canonical symlink policy for trusted skill roots, preferably source-preserving and explicit, then land it through the current loader with bundled targets either excluded or still classified as bundled.

## Reproduction Assessment

Yes for the blocking PR defect: source and diff inspection show a non-bundled root symlink into `bundledSkillsDir` would pass the new allowed-root check, then load under the caller source and skip `skills.allowBundled`. I did not execute it because this review is read-only.

## Solution Assessment

No. Allowing selected cross-root symlinks may be useful, but this implementation trusts bundled targets without preserving bundled-source semantics and is no longer aligned with current main's explicit `allowSymlinkTargets` contract.
