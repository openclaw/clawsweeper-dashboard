## Summary

Keep open: current main and the latest shipped release still have the Feishu streaming-card overwrite path, while the linked closing PRs target ACP `final_only` rather than the Feishu dispatcher surface. The reporter's live compiled-artifact workaround is useful supporting context, but the source fix has not landed upstream.

## What This Changes

Feishu streaming-card replies should retain assistant text produced before a tool call when later final assistant text updates the same card.

## Best Possible Solution

Add a Feishu dispatcher regression for partial pre-tool text followed by a later final payload on the same active streaming card, preserve or merge the accumulated visible text without double-concatenation, and then reconcile the ACP-only PRs against that corrected scope.

## Reproduction Assessment

Yes, at source level: Feishu partial callbacks accumulate text into `streamText`, and a later active-card final delivery assigns `streamText = text`, so earlier pre-tool text can be replaced. I did not run a live Feishu credentialed repro in this read-only review.

## Solution Assessment

Yes for the issue's corrected Feishu-dispatcher direction, but no for treating the linked ACP-only PRs as sufficient. The narrow maintainable fix belongs in `extensions/feishu/src/reply-dispatcher.ts` with regression coverage for partial-plus-final tool-call turns.
