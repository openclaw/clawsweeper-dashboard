## Summary

Keep open: current main and v2026.6.8 still route Feishu streaming-card final payloads through a final-text replacement path, so the pre-tool message-loss bug remains source-reproducible; related duplicate reports already point back to this issue as the canonical specific tracker.

## What This Changes

Feishu streaming card mode loses assistant text emitted before a tool call when the later final payload replaces the card content.

## Best Possible Solution

Keep this issue as the canonical Feishu pre-tool text-loss tracker and fix the Feishu dispatcher/card close path with regression coverage for partial pre-tool text followed by a separate final payload.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: pre-tool partial text updates `streamText`, a later independent `final` payload resets `streamText`, and card close uses only that later value. I did not run a live Feishu session in this read-only review.

## Solution Assessment

Unclear as an implementation choice until maintainer coordination finishes; the right fix should preserve prior visible assistant text across Feishu streaming-card invocation boundaries without mixing in the separate ACP final_only path or broader tool-progress policy.
