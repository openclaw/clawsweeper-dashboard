## Summary

Keep open: current main and v2026.6.10 still have the Feishu streaming-card final path that can replace earlier streamed assistant text with the later final payload, and the linked larksuite/openclaw-lark report adds live compiled-artifact workaround evidence rather than an upstream fix. This issue remains the canonical narrow tracker for Feishu pre-tool assistant text loss.

## What This Changes

The issue reports that Feishu streaming-card replies discard assistant text emitted before a tool call when a later model invocation overwrites the same card.

## Best Possible Solution

Keep this as the canonical Feishu pre-tool text-loss tracker and land a Feishu-owned dispatcher fix that preserves earlier visible assistant text across tool-use boundaries without duplicating transient tool progress or stale card content.

## Reproduction Assessment

Yes, at source level: Feishu can accumulate pre-tool partial text in streamText, then an active-card final delivery replaces streamText with only the later final payload before close. I did not run a live credentialed Feishu repro in this read-only review.

## Solution Assessment

Unclear for an autonomous branch: the fix belongs in the Feishu streaming-card dispatcher, but the precise state contract should be coordinated with the broader Feishu card delivery trackers to avoid introducing duplication or stale-content regressions.
