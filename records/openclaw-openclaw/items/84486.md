## Summary

Keep open: current main still has a Feishu streaming-card path where a later final payload can replace the accumulated visible card text, and the open closing PRs are aimed at ACP `final_only` rather than this Feishu dispatcher surface.

## What This Changes

The issue reports that Feishu streaming-card replies lose assistant text emitted before a tool call when the later post-tool assistant response updates the same card.

## Best Possible Solution

Fix the Feishu dispatcher/streaming-card close path so accumulated pre-tool visible text is preserved through the final card close, add a regression for partial pre-tool text followed by a final payload, and then re-scope or close the ACP-only PRs as needed.

## Reproduction Assessment

Yes, at source level: Feishu partial callbacks accumulate text into `streamText`, a later final payload assigns `streamText = text`, and streaming close can replace the card with that final-only text. I did not run a live Feishu credentialed repro in this read-only review.

## Solution Assessment

Yes for the issue's Feishu-dispatcher direction, but not for treating the linked ACP-only PRs as sufficient. The narrow maintainable fix belongs around `extensions/feishu/src/reply-dispatcher.ts` and `extensions/feishu/src/streaming-card.ts` with regression coverage for partial-plus-final tool-call turns.
