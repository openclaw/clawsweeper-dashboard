## Summary

Keep open: current main and the latest release still have the Feishu streaming-card final path replacing accumulated pre-tool partial text with the later final payload, while related cleanup has treated this issue as the canonical tracker for this specific Feishu message-loss case.

## What This Changes

The issue asks Feishu streaming-card replies to preserve assistant text emitted before tool calls instead of showing only the later post-tool final answer.

## Best Possible Solution

Patch the Feishu dispatcher so final streaming-card updates preserve the accumulated visible stream text across tool-use boundaries, and add a regression for pre-tool partial text followed by a later final payload.

## Reproduction Assessment

Yes, at source level: simulate an active Feishu streaming session with pre-tool `onPartialReply` text and then a post-tool final payload; the current final handler overwrites `streamText` with the final payload. I did not run a live credentialed Feishu repro in this read-only review.

## Solution Assessment

Yes at the ownership boundary: this should be fixed in the Feishu plugin streaming-card dispatcher, not in ACP dispatch. The exact patch should preserve accumulated visible text when a later final payload arrives and add focused regression coverage.
