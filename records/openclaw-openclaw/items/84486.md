## Summary

Keep open: current main still has a source-reproducible Feishu streaming-card path where pre-tool streamed text can be replaced by a later final payload, and the open linked fixes target ACP `final_only` rather than the Feishu dispatcher surface.

## What This Changes

Fix Feishu streaming card replies so text streamed before a tool call remains visible when the later final reply updates the same card.

## Best Possible Solution

Add a Feishu dispatcher regression for partial pre-tool text followed by a later final payload on the same streaming card, preserve or merge the accumulated visible text across that boundary, and then reconcile the ACP-only PRs against the corrected scope.

## Reproduction Assessment

Yes, at source level: Feishu partial callbacks accumulate text in `streamText`, and the later active-card final delivery replaces `streamText` with only the final payload. I did not run a live Feishu credentialed repro in this read-only review.

## Solution Assessment

No for the linked ACP-only fixes: they address `final_only` projector buffering, not the Feishu streaming-card overwrite. The maintainable fix belongs in `extensions/feishu/src/reply-dispatcher.ts` with regression coverage for the partial-plus-final tool-call turn.
