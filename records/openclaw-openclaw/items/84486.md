## Summary

Keep open: current `main` and the shipped `v2026.5.18` source still have a source-reproducible Feishu streaming-card path where a later final payload can replace visible pre-tool assistant text, and the linked open PRs target ACP `final_only` rather than the Feishu dispatcher path.

## What This Changes

The issue asks Feishu streaming-card replies to preserve assistant text emitted before tool calls instead of losing it when the later final reply arrives.

## Best Possible Solution

Add Feishu dispatcher coverage for `onPartialReply` pre-tool text followed by a later `{ kind: "final" }` payload on the same streaming card, then preserve or merge that user-visible text through idle close while keeping transient status cleanup intact.

## Reproduction Assessment

Yes, at source level: Feishu partial callbacks accumulate `streamText`, the active-card final delivery overwrites it with only the final payload, and idle close can replace the card content with that final-only text. I did not run a live Feishu credentialed repro in this read-only review.

## Solution Assessment

No for the linked ACP-only PRs: they address `final_only` projector buffering, not the Feishu streaming-card close path. The narrow maintainable fix belongs in the Feishu reply dispatcher/streaming-card tests with explicit coverage for pre-tool assistant text versus transient status text.
