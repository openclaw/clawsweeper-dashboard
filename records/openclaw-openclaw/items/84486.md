## Summary

Keep open: current main and the latest shipped release still have the Feishu streaming-card final-delivery overwrite path, and the open closing PRs target ACP `final_only` rather than the Feishu dispatcher path described by the reporter.

## What This Changes

The issue reports Feishu streaming-card replies losing assistant text emitted before a tool call when a later model invocation updates the same card.

## Best Possible Solution

Add a Feishu dispatcher regression for pre-tool partial text followed by a post-tool final payload on one streaming card, preserve accumulated visible text without duplication, and retarget or close the ACP-only linked PRs separately.

## Reproduction Assessment

Yes, at source level: Feishu partial callbacks build `streamText`, but an active-card final delivery assigns `streamText = text` before close, so a post-tool-only final can replace pre-tool text. I did not run a live Feishu repro in this read-only review.

## Solution Assessment

Yes for a narrow Feishu dispatcher fix with regression coverage; no for the current ACP-only linked PRs as a complete fix for this issue, because Feishu uses a direct card dispatcher path.
