## Label Justifications

- `P2`: The issue reported repeatable loss of user-visible Feishu reply content in tool-using turns, with channel-limited blast radius.
- `impact:message-loss`: Assistant text produced before a tool call was silently absent from the final Feishu streaming card.

## Summary

Current main and the shipped v2026.6.5 source preserve completed Feishu generation blocks when streaming snapshots reset after a tool call, with a focused regression test matching the reported loss; the newer duplication/status behavior in https://github.com/openclaw/openclaw/issues/91956 is a distinct remaining problem.

## What This Changes

Fix Feishu streaming-card replies so assistant text emitted before tool calls remains visible with the later final response.

## Best Possible Solution

Keep the shipped single-card snapshot accumulator and regression coverage; track any repeated reasoning or tool-status interleaving separately in https://github.com/openclaw/openclaw/issues/91956 because it is not the original pre-tool text-loss behavior.

## Reproduction Assessment

Yes. The v2026.5.18 prompt and session sequence are concrete, the reporter verified the symptom and a live local artifact repair, and the shipped regression test exercises the same partial-snapshot reset after tools.

## Solution Assessment

Yes. Retaining completed generation blocks inside the existing Feishu single-card snapshot accumulator is narrower and more maintainable than creating one card per invocation or delaying all output until turn completion.
