## Label Justifications

- `P1`: The report describes default ACP final-only delivery silently dropping user-visible assistant text in real channel workflows.
- `impact:message-loss`: The affected behavior is lost assistant message content across a tool-call boundary before channel delivery.

## Summary

Keep this open: current main still has ACP final-only buffering scoped inside the projector, with reset behavior and no regression coverage for preserving pre-tool assistant text across tool-use boundaries. The related closed final-only issues and the open spawn-child ACP PR do not cover this exact Feishu/Telegram pre-tool text loss.

## What This Changes

The issue asks OpenClaw to preserve assistant text emitted before tool calls when ACP delivery mode is `final_only` so channels receive the complete final turn output.

## Best Possible Solution

Preserve final-only assistant output at the logical ACP user-turn level, deliver it once at the true terminal assistant stop, and add a regression that reproduces pre-tool text plus post-tool final text without changing live-mode delivery.

## Reproduction Assessment

Yes, at source level with medium confidence: final-only ACP output is buffered in projector-local state and reset without a regression that preserves text across a tool-use model boundary. A high-confidence live reproduction still needs a Feishu/Telegram or ACP harness run that emits pre-tool assistant text and then a final assistant reply.

## Solution Assessment

Yes, carrying final-only output across internal tool-use boundaries is the narrowest maintainable direction. Flushing the pre-tool text early would weaken `final_only` semantics, while a turn-level accumulator preserves the intended UX.
