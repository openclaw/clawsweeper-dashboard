- `impact:message-loss`: The issue can silently drop assistant text emitted before a tool call from the final Feishu-visible reply.

## Summary

Keep open: current main and the latest shipped release still overwrite the active Feishu streaming card with the final payload, while the open closing PRs only change ACP `final_only` projector behavior. The reporter's linked larksuite/openclaw-lark issue and live compiled-artifact workaround are useful supporting evidence, but the upstream Feishu source fix has not landed.

## What This Changes

Preserve Feishu streaming-card text emitted before tool calls so the final visible card includes assistant text across the whole turn.

## Best Possible Solution

Land a Feishu dispatcher source fix that carries forward or safely merges pre-tool partial text into later final streaming-card content, with a regression covering partial text followed by a final payload on the same active card.

## Reproduction Assessment

Yes, at source level: Feishu partial callbacks accumulate text into `streamText`, and a later active-card final delivery assigns `streamText = text`, so earlier pre-tool text can be replaced. I did not run a live Feishu credentialed repro in this read-only review.

## Solution Assessment

Yes for the issue's corrected Feishu-dispatcher direction, but no for treating the linked ACP-only PRs as sufficient. The narrow maintainable fix belongs in `extensions/feishu/src/reply-dispatcher.ts` with regression coverage for partial-plus-final tool-call turns.
