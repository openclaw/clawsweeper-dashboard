- `impact:message-loss`: The reported failure can silently drop assistant text produced before a tool call from the final Feishu-visible reply.

## Summary

Keep open: current main still has the Feishu streaming-card path that can replace earlier partial text with a later final payload, and the open closing PRs are ACP `final_only` fixes rather than the Feishu dispatcher source fix the reporter narrowed this to.

## What This Changes

Feishu streaming card replies should preserve assistant text emitted before tool calls when a later model invocation produces the final reply.

## Best Possible Solution

Preserve prior Feishu `streamText` across assistant-message/tool-call boundaries, add a regression for partial pre-tool text followed by a later final payload, and reconcile the ACP-only PRs separately.

## Reproduction Assessment

Yes, at source level: Feishu partial callbacks accumulate visible text into `streamText`, and a later active-card final delivery assigns `streamText = text`, which can replace earlier pre-tool text. I did not run a live Feishu credentialed repro in this read-only review.

## Solution Assessment

Yes for the corrected Feishu-dispatcher direction, but no for closing this based on the linked ACP-only fixes. The narrow maintainable fix belongs in `extensions/feishu/src/reply-dispatcher.ts` with regression coverage for partial-plus-final tool-call turns.
