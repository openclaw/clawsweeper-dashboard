## Label Justifications

- `impact:message-loss`: The reported failure drops user-visible assistant text produced before a tool call from the final Feishu reply.

## Summary

Keep open: current main still has a source-reproducible Feishu streaming-card path where a later final payload can replace the accumulated pre-tool partial text, and the two linked open closing PRs target ACP `final_only` instead of the renamed Feishu dispatcher path.

## What This Changes

This issue reports that Feishu streaming-card replies lose assistant text emitted before a tool call when the later post-tool assistant message renders.

## Best Possible Solution

Add a Feishu dispatcher regression for pre-tool partial text followed by a post-tool final payload on one streaming card, preserve accumulated visible text across that final delivery, and reconcile the ACP-only linked PRs against the corrected issue scope.

## Reproduction Assessment

Yes, at source level: Feishu partial snapshots accumulate in `streamText`, but a later active-card final delivery assigns `streamText = text`, which can discard pre-tool text when the final payload only contains the post-tool answer. A live Feishu repro was not run in this read-only review.

## Solution Assessment

No for the linked ACP-only fixes: they do not address the renamed Feishu streaming-card overwrite path. The narrow maintainable fix belongs in `extensions/feishu/src/reply-dispatcher.ts` with regression coverage for partial-plus-final tool-call turns.
