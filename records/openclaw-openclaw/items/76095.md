## Summary

Keep open. Current main still emits the missing channelConfigs diagnostic without considering explicit plugin disablement and can replay it on same-path replacement; the related fix PR is closed unmerged and the focused regression tests are still absent.

## What This Changes

The issue requests suppressing duplicate channelConfigs startup warnings for openclaw-weixin when the external channel plugin is disabled or removed from the allowlist.

## Best Possible Solution

Add a generic manifest-registry repair that skips missing channelConfigs diagnostics for explicitly disabled plugins, dedupes same-physical-plugin replay, preserves active-plugin warnings, and covers both paths with focused tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: disabled installed plugins are included in cold metadata, the missing channelConfigs diagnostic does not check enablement, same-path replacement can push it again, and config validation surfaces it as a startup warning.

## Solution Assessment

Yes. A generic core fix in src/plugins/manifest-registry.ts is the narrow maintainable solution; a WeChat-specific exception or external-plugin-only metadata change would not address duplicate and inactive warning noise in the core diagnostic path.

## Repair Work Prompt

Fix the duplicate/inactive missing channelConfigs warning reported by this issue. In src/plugins/manifest-registry.ts, make the missing channelConfigs compatibility diagnostic respect explicit plugins.entries.<id>.enabled === false for the manifest/plugin id, and avoid replaying the same compatibility diagnostic when a same-physical-plugin candidate replaces an existing record. Preserve warnings for active non-bundled channel plugins that declare channels without channelConfigs; do not special-case openclaw-weixin or weaken the channelConfigs manifest contract. Add focused regression tests in src/plugins/manifest-registry.test.ts for an explicitly disabled external channel plugin and for a same-path duplicate emitting the channelConfigs warning exactly once. Use PR #76110 only as source context because it is closed unmerged. Add a single-line CHANGELOG.md Fixes entry for the user-facing startup warning cleanup, without bot attribution.
