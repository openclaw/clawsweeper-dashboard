## Summary

Keep open. Current main still has the generic plugin-registry path that can warn for a disabled installed external channel plugin and can replay the same missing-channelConfigs diagnostic during same-physical-plugin replacement; the linked fix PR is closed unmerged.

## What This Changes

The issue asks OpenClaw to suppress missing channelConfigs warnings for disabled or disallowed external channel plugins and avoid duplicate startup warnings.

## Best Possible Solution

Add a generic manifest-registry fix that skips missing channelConfigs diagnostics for explicitly disabled plugins, dedupes same-physical-plugin diagnostic replay, preserves active-plugin warnings, and adds regression tests.

Yes. Source inspection gives a high-confidence reproduction path: disabled installed plugins are included in cold metadata, the missing channelConfigs diagnostic does not check enablement, same-path replacement can push it again, and config validation surfaces it as a warning.

## Solution Assessment

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76095 with a generic core plugin-registry repair. Current main emits the missing channelConfigs warning for non-bundled channel plugins even when `plugins.entries.<id>.enabled === false`, and can emit the same warning twice when the same physical plugin is encountered through a higher-precedence replacement path. Keep the fix generic in `src/plugins/manifest-registry.ts`; do not add an openclaw-weixin special case and do not treat external plugin docs as the fix. Thread normalized plugin config or equivalent enablement knowledge into the compatibility diagnostic path, skip the missing channelConfigs warning for explicitly disabled plugins, and dedupe same-physical-plugin diagnostic replay while preserving warnings for active external channel plugins and existing duplicate-plugin diagnostics. Add focused regression tests for disabled-plugin suppression and same-path exactly-once emission. Add a single-line `CHANGELOG.md` Fixes entry if the behavior change is user-facing. Stop and ask if the implementation would require a new config option or product policy change.
