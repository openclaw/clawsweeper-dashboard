## Summary

Keep open for maintainer review: current main still treats an already paired `operator.read` device requesting `operator.admin` as a pending `scope-upgrade` approval, so the reported post-restart write failure is real rather than obsolete. The related #67799/subagent cleanup covers backend loopback callers, but ordinary CLI/Control UI admin-scope operations still need an explicit approval UX/policy decision.

## What This Changes

The issue requests a recovery path for gateway restart scenarios where read-scoped calls work but admin-scoped writes such as `cron run`, `sessions_send`, and `config.patch` fail with `PAIRING_REQUIRED` due to an unhandled `scope-upgrade`.

## Best Possible Solution

Keep explicit approval for broader scopes, but add a maintainer-approved recovery flow that makes pending local scope upgrades obvious and directly approvable from the failing CLI/Control UI path after a gateway restart.

## Reproduction Assessment

Yes. The issue gives concrete steps, and current tests reproduce the central condition by pairing a device for `operator.read`, reconnecting with `operator.admin`, and asserting `PAIRING_REQUIRED` with reason `scope-upgrade`.

## Solution Assessment

Unclear. Auto-completing the upgrade or lowering loopback pairing requirements is not obviously the best solution because current code and docs deliberately require explicit approval for wider authority; a clearer approval UX is the safer default direction.
