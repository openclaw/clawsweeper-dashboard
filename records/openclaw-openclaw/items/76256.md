## Summary

Keep open for maintainer review: current main still deliberately converts an under-scoped paired operator device requesting `operator.admin` into a pending `scope-upgrade`, so the remaining question is the approved recovery UX and security policy rather than stale cleanup.

## What This Changes

The issue asks OpenClaw to recover from post-Gateway-restart `scope-upgrade` failures where read calls work but admin/write calls such as `cron run`, `sessions_send`, and `config.patch` fail with `pairing required`.

## Best Possible Solution

Preserve explicit approval for wider scopes, but provide a maintainer-approved recovery path that makes post-restart local upgrades obvious and completes the exact pending request without silent admin escalation.

## Reproduction Assessment

Yes, at source level. Current tests pair a device for `operator.read`, reconnect it with `operator.admin`, and assert the same `PAIRING_REQUIRED`/`scope-upgrade` pending-request path.

## Solution Assessment

Unclear. Silent auto-upgrade or lowering loopback pairing requirements is not clearly the best fix because current code and docs deliberately require explicit approval for broader authority; the safer direction is an explicit one-step approval UX.
