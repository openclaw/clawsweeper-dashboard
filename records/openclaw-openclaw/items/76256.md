## Summary

Close as implemented on main: current main has an explicit scope-upgrade approval path for read-scoped devices requesting admin/write access, and the macOS app plus Control UI now surface and approve those pending requests instead of leaving a generic 1008 failure.

## What This Changes

The issue requests handling Gateway restart scope-upgrade pairing so write operations such as `cron.run`, `sessions.send`, and `config.patch` can regain the required operator scopes instead of failing with `pairing required`.

## Best Possible Solution

Keep the explicit approval model for scope upgrades and rely on the current main app/Control UI/CLI approval flow rather than silently widening local devices to admin.

## Reproduction Assessment

Yes. The report’s read-works/write-fails case maps directly to an already paired `operator.read` device reconnecting with `operator.admin`, and current tests cover that as a pending `scope-upgrade` request.

## Solution Assessment

Yes. The current implementation is the safer maintainable path: it surfaces the request ID and approval UI/command while preventing silent loopback escalation to admin.
