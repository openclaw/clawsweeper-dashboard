## Summary

Keep open for maintainer/product-security follow-up: current main intentionally rejects a read-scoped paired device that requests `operator.admin` with a pending `scope-upgrade`, so the reported failure mode is still present and the recovery path needs an approved CLI/Control UI policy decision.

## What This Changes

The issue asks OpenClaw to recover from post-Gateway-restart `scope-upgrade` pairing-required failures that block write/admin operations such as `cron run` while read operations still work.

## Best Possible Solution

Keep explicit approval for broader scopes, but make the post-restart local recovery path obvious and one-step in the failing CLI and Control UI/desktop surfaces without silently granting admin.

## Reproduction Assessment

Yes, at source level. Current gateway tests pair a device for `operator.read`, reconnect it with `operator.admin`, and assert the same `PAIRING_REQUIRED`/`scope-upgrade` pending-request path.

## Solution Assessment

Unclear. Silent auto-upgrade or lower loopback pairing requirements are not clearly the best fix because current code deliberately requires explicit approval for broader authority; the safer direction is an actionable approval UX.
