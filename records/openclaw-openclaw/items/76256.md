## Summary

Close as implemented on current main: OpenClaw still requires explicit approval for admin scope upgrades, but the unhandled post-restart dead end is now covered by structured scope-upgrade details, Control UI guidance, and a CLI approval path.

## What This Changes

The issue asks for gateway scope-upgrade recovery after restart so read-scoped operations can continue while write/admin operations get a usable approval flow instead of a dead `pairing required` failure.

## Best Possible Solution

Keep explicit approval for wider operator scopes and use the current exact-request `openclaw devices approve <requestId>` recovery path instead of silently granting admin after restart.

## Reproduction Assessment

Yes, at source/test level. Current tests and gateway code show a paired `operator.read` device requesting `operator.admin` produces the same `PAIRING_REQUIRED` `scope-upgrade` pending request, and current main then exposes an actionable approval path.

## Solution Assessment

Yes. The current implementation is the safer fix because it preserves the documented explicit-approval policy for broader scopes while making the pending upgrade visible and approvable from Control UI/CLI.
