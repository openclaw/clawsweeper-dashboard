## Summary

Keep open for maintainer review: current main intentionally turns a paired `operator.read` device asking for `operator.admin` into a pending `scope-upgrade` approval, so the reported `pairing required` path is real rather than obsolete. The related #67799 cleanup addressed subagent scope selection, but it does not cover this CLI/Control UI post-restart upgrade UX.

## What This Changes

The issue asks OpenClaw to handle post-restart `operator.read` to `operator.admin` scope upgrades so write commands such as `cron run`, `sessions_send`, and `config.patch` do not fail with `pairing required`.

## Best Possible Solution

Preserve explicit approval for wider scopes, but add a maintainer-approved recovery path that makes pending local scope upgrades obvious and actionable from the failing CLI/Control UI flow.

## Reproduction Assessment

Yes. Current tests reproduce the core condition by pairing a device for `operator.read`, reconnecting with `operator.admin`, and asserting `PAIRING_REQUIRED` with reason `scope-upgrade`.

## Solution Assessment

Unclear. Silent auto-completion or lowering the loopback pairing threshold is not obviously the best fix because current code and docs deliberately require explicit approval for broader authority; a clearer one-step approval UX is the safer default direction.
