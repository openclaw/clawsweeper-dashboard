## Summary

Keep open for maintainer review: current main still intentionally turns a paired `operator.read` device asking for `operator.admin` into a pending `scope-upgrade`, so the reported write-operation failure is real rather than obsolete. Existing docs/UI/CLI surfaces describe manual approval, but they do not implement the reporter’s requested app prompt or post-restart recovery flow.

## What This Changes

The issue requests a recovery path for gateway restart cases where read calls keep working but admin-scoped writes such as `cron run`, `sessions_send`, and `config.patch` fail with `PAIRING_REQUIRED` due to an unhandled `scope-upgrade`.

## Best Possible Solution

Preserve explicit approval for broader scopes, but add a maintainer-approved recovery flow that makes pending local scope upgrades obvious and directly approvable from the failing CLI or Control UI path after gateway restart.

## Reproduction Assessment

Yes. The issue provides concrete steps, and current tests reproduce the central condition by pairing a device for `operator.read`, reconnecting with `operator.admin`, and asserting `PAIRING_REQUIRED` with reason `scope-upgrade`; I did not execute tests because this review is read-only.

## Solution Assessment

Unclear. Auto-completing the upgrade or lowering loopback pairing requirements is not obviously the best fix because current code and docs deliberately require explicit approval for wider authority; a clearer one-step approval UX is the safer direction.
