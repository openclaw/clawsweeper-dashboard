## Summary

Keep open for maintainer review: current main still treats an `operator.read` paired device requesting `operator.admin` as an explicit pending scope upgrade, so the reported `pairing required` path is real and the remaining question is the approved CLI/Control UI recovery UX rather than an obsolete bug.

## What This Changes

The issue asks for post-restart gateway scope upgrades for write/admin operations to become actionable from the app or CLI instead of leaving `cron run`, `sessions_send`, and `config.patch` blocked by `pairing required`.

## Best Possible Solution

Preserve explicit approval for wider scopes, but make post-restart local scope upgrades obvious and actionable in the failing CLI and Control UI flow, preferably with a maintainer-approved one-step approval path rather than silent admin escalation.

## Reproduction Assessment

Yes, at source level. Current tests pair a device for `operator.read`, reconnect it with `operator.admin`, and assert the same `PAIRING_REQUIRED`/`scope-upgrade` path with a pending request.

## Solution Assessment

Unclear. Auto-completing the upgrade or lowering loopback pairing requirements is not clearly the best fix because current code and docs deliberately require explicit approval for broader authority; the safer direction is an explicit, actionable approval UX.
