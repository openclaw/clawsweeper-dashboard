## Summary

Keep open. Current main still has the reported scope-upgrade path for local CLI/app write calls: read-scoped loopback access can work, but admin-scoped methods remain blocked until an explicit device approval happens.

## What This Changes

The issue asks OpenClaw to handle gateway-restart scope upgrades so local write operations such as `cron run`, `sessions_send`, and `config.patch` do not fail with `1008 pairing required` after a read-scoped reconnect.

## Best Possible Solution

Add a maintainer-approved scope-upgrade flow for same-host CLI/app write operations that preserves the admin boundary, then cover it with gateway auth, CLI, UI, and docs tests.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: first connect with a read-scoped local CLI method such as `cron.list`, then run an admin-scoped method such as `cron.run`; the gateway currently creates a `scope-upgrade` pending request and closes the connect path until approval.

## Solution Assessment

Unclear. Automatic local admin upgrades may be too broad; a safer path is either explicit native/Control UI approval with clear CLI remediation or a deliberately scoped same-host trust rule reviewed as an auth-policy change.
