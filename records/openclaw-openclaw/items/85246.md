## Summary

Keep open: current main has the adjacent macOS update hardening stack, but the exact npm-global LaunchAgent UI Update/latest-version flow still lacks direct macOS proof and current source still intentionally schedules a restart after a managed handoff with a skipped response.

## What This Changes

The issue reports that the Control UI Update button can leave an npm-global, launchd-managed Gateway stopped or hung when the target version is already current.

## Best Possible Solution

Run direct macOS npm-global LaunchAgent proof for Control UI Update or `gateway call update.run` at the latest version; close if it passes, or fix the update contract around durable handoff versus explicit no-op/refusal if it still fails.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection does reproduce the `update.run` handoff plus SIGUSR1 restart path, but the macOS launchd collision and latest-version behavior still need direct proof.

## Solution Assessment

Unclear. A no-restart-on-latest change may be too narrow and would conflict with the documented refresh contract; the safer path is live proof first, then either durable launchd-owned handoff or an explicit fail-safe refusal before stopping the Gateway.
