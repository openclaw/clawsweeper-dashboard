## Label Justifications

- `impact:message-loss`: The live comments report unanswered Telegram/provider ingress and duplicate replies during the Gateway outage window.

## Summary

Keep open: current main and the latest release still route supervised npm-global `update.run` through the managed-service handoff and schedule an `update.run` restart before proving the update is a no-op. The issue has later live macOS launchd evidence after the earlier fix stack, and the remaining decision is the product/update contract for no-op or failed handoffs rather than a duplicate already fixed on main.

## What This Changes

The issue asks OpenClaw to stop the Control UI `update.run` path from breaking a launchd-managed npm-global Gateway when the update is already current or the handoff cannot complete healthily.

## Best Possible Solution

Define the managed `update.run` contract for supervised npm-global same-version updates, then implement either a pre-handoff no-op/refusal path or a health-verified durable handoff with macOS launchd proof.

## Reproduction Assessment

No, not for a fresh current-main live macOS crash/hang reproduction. Source inspection does show the same supervised global `update.run` handoff and restart path, and the issue comments provide live macOS proof on recent releases after the earlier fix stack.

## Solution Assessment

Unclear. Short-circuiting same-version updates before SIGUSR1 is plausible, but current docs intentionally refresh same-version package installs, so maintainers should first decide whether the best contract is no-op, explicit refusal, or durable out-of-process handoff.
