## Summary

Keep open: current main still sends supervised global `update.run` through a managed-service handoff and schedules a SIGUSR1 restart even when the immediate RPC result is `skipped`; the issue discussion includes live macOS npm-global LaunchAgent failures after the adjacent merged fixes, so this remains the canonical tracker for the no-op/update contract and launchd proof.

## What This Changes

The issue reports that the Control UI Update button on an npm-global, launchd-managed macOS Gateway can trigger managed handoff/SIGUSR1 restart on an already-latest update and leave the Gateway unhealthy instead of no-oping or restarting safely.

## Best Possible Solution

Keep this as the canonical tracker, decide the `update.run` no-op and launchd handoff contract, then close only after live macOS npm-global LaunchAgent proof covers the Control UI/update.run latest/no-op and service recovery paths.

## Reproduction Assessment

No high-confidence current-main live reproduction was run. Source inspection shows the same supervised global `update.run` handoff and SIGUSR1 restart path still exists, and the issue discussion includes live macOS evidence on recent releases.

## Solution Assessment

Unclear: short-circuiting latest/no-op updates is plausible, but maintainers still need to choose between no-op return, explicit refusal, and a more durable out-of-process handoff contract.
