## Summary

Keep open: current main still routes supervised global `update.run` calls into the managed handoff plus Gateway restart path, and the report describes an availability failure that is not fully closed by adjacent update-handoff work.

## What This Changes

The issue reports that the Control UI update action can leave an npm-global, launchd-managed Gateway stopped or hung when the update path ends up skipped/no-op.

## Best Possible Solution

Keep the documented control-plane update contract, but make the macOS launchd package handoff durable outside the dying Gateway process or explicitly refuse/skip before signaling a restart when no safe handoff exists.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection does show the reported `update.run` handoff plus SIGUSR1 restart path on current main, but the macOS launchd collision itself needs live proof.

## Solution Assessment

Unclear. A no-restart-on-latest fix may be too narrow because docs currently say package-manager updates refresh and restart even when the version already matches; the safer maintainable direction is a durable launchd-owned handoff or an explicit fail-safe refusal before stopping the Gateway.
