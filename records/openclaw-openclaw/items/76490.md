## Summary

Keep open: the PR targets a real `sessions_send` A2A routing area, but the patch is not mergeable as-is because it drops the existing channel-delivery fallback on requester-delivery failures and changes the announce destination without updating the model prompt contract or changelog.

## What This Changes

The PR changes `sessions_send` A2A announce delivery to call the requester session with `deliver: true`, keeps channel delivery only when no requester session exists, and updates related tests.

## Best Possible Solution

Keep the current requester result path separate from announce delivery unless maintainers explicitly choose requester-session announces; if they do, update the prompt contract, preserve target-channel fallback semantics, add regression coverage, and include a changelog entry.

## Reproduction Assessment

Unclear for the timeout claim: current source and tests show announce replies are sent to the target channel, but I did not establish a current-main path where `sessions_send` times out specifically because the announce reply misses the requester session.

## Solution Assessment

No: this patch changes the delivery destination in a way that conflicts with the existing prompt contract and can drop announces when requester-session delivery fails. A safer fix needs explicit destination semantics plus fallback behavior and tests.

Before this patch, the same non-skip announce would still be sent to the resolved target channel, so requester-session delivery needs to fall back to the existing `send` path after failure.

That mismatched contract can make the model choose wording or skip behavior for the wrong audience, so the destination change needs a matching prompt and test update.
