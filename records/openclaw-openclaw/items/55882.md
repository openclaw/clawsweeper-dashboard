## Summary

Keep open. Current main has durable final-delivery and heartbeat-specific fixes, but it still does not durably record arbitrary promised-but-unfinished outputs before a non-restart interrupt or task switch.

## What This Changes

The issue requests a durable per-session pending-deliverables queue or auto-resume mechanism so promised outputs are not dropped after higher-priority interruptions.

## Best Possible Solution

Define a typed session-continuity contract that records pending deliverables before preemptive switches and resolves them only by send, explicit deferral, cancellation, or supersession, with regression and live channel proof.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a credible current-main path: interrupt mode aborts active work, while durable pending delivery is recorded only after final payloads exist and heartbeat replay is the only automatic pending-delivery replay path.

## Solution Assessment

Unclear until maintainers define the product contract. A durable pending-deliverables direction is plausible, but the safe implementation needs explicit promise detection, resume/defer/cancel semantics, and duplicate-delivery safeguards.
