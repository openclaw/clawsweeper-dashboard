## Summary

Keep this issue open. Current main has adjacent safeguards for inbound follow-up queues, restart recovery, outbound delivery retries, background task tracking, BTW side questions, and optional inferred commitments, but none provides a durable per-session record for arbitrary unfinished promised outputs after non-restart task switching.

## What This Changes

The issue asks for a durable pending-deliverables or task-stack mechanism so promised agent outputs survive higher-priority task switches without requiring a gateway restart.

## Best Possible Solution

Add a core session-continuity mechanism that persists typed pending deliverables before preemptive task switches and clears them only when the promised output is sent, explicitly deferred, cancelled, or superseded, with regression coverage for a long-running direct-chat turn interrupted without a gateway restart.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Unclear for the exact live Telegram/autonomy-guard sequence, but the code gives a focused regression shape: start a long-running direct chat run, inject unrelated higher-priority work without restart, and verify no durable pending-deliverable record is created or resumed today.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers define the product contract, but the narrow maintainable direction is a typed durable pending-deliverable/session-continuity record rather than relying on inbound follow-up queues, optional inferred commitments, or restart-only recovery.
