## Summary

Keep open. Current main has several adjacent protections, but it still does not define or persist a promised unfinished deliverable before interruption, nor does it specify how heartbeat/system turns should automatically resume or clear that work.

## What This Changes

Request a durable per-session pending-deliverables or task-resume contract so promised user-visible outputs are delivered, deferred, cancelled, or resumed after task switching.

## Best Possible Solution

Define a typed session-continuity contract that records promised deliverables before preemptive switches and resolves them only by send, explicit deferral, cancellation, or supersession while reusing existing heartbeat, commitment, and pending-final safeguards.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows a credible current-main path where capture starts only after final payloads exist and interruption/heartbeat policy does not provide a durable promised-work stack.

## Solution Assessment

Unclear until maintainers define the product contract. A durable pending-deliverables stack may be the right direction, but it needs explicit cancellation, supersession, replay, and duplicate-prevention semantics before implementation.
