## Summary

Keep open. Current main has useful adjacent safeguards for heartbeat deferral, active reply-run skips, and post-payload pending-final replay, but it still does not define or persist unfinished promised deliverables before a preemptive task switch; the latest WeChat report on v2026.5.20 keeps this relevant and product-scoped.

## What This Changes

The issue requests durable per-session pending-deliverable or active-task handling so promised long-running outputs survive heartbeat, system, and operational interruptions and remain visibly delivered to external chats.

## Best Possible Solution

Define a typed session-continuity contract that records promised deliverables before preemptive switches and resolves them only by send, explicit deferral, cancellation, or supersession while reusing the shipped heartbeat, active-run, commitment, and pending-final safeguards.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows a credible current-main gap: interrupt-mode paths can abort active work while durable pending-final capture starts only after final payloads exist, and the shipped v2026.5.20 WeChat comment reports a related visible-delivery failure.

## Solution Assessment

Unclear until maintainers define the product contract. A durable pending-deliverables stack may be the right direction, but it needs explicit semantics for cancellation, supersession, replay, and duplicate prevention before implementation.
