## Summary

Keep open. Current main has useful adjacent fixes for produced final-payload retry, heartbeat ack replay, and active heartbeat deferral, but it still does not define a durable pre-interrupt pending-deliverable/task-continuity contract; the fresh WeChat v2026.5.20 report also keeps this current rather than stale.

## What This Changes

The issue requests durable session/task continuity so promised long-running deliverables are not silently lost after heartbeat, hook, operational, or other task-switching interrupts.

## Best Possible Solution

Define one typed session-continuity and visible-delivery contract that records active task obligations before preemptive switches and resolves them only by send, explicit deferral, cancellation, or supersession while reusing the existing heartbeat, pending-final, message-tool, and commitments pieces.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only pass. Source inspection shows a credible current-main path where interrupt mode can abort active work while durable pending-final capture only starts after final payloads already exist.

## Solution Assessment

Unclear until maintainers define the product contract. A durable pending-deliverables stack may be the right direction, but it needs explicit cancellation, supersession, replay, duplicate-prevention, and visible-delivery semantics first.
