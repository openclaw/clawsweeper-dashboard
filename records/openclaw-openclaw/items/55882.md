## Summary

Keep open. Current main has several adjacent delivery-recovery and heartbeat protections, but it still does not define a durable active-task or promised-deliverable contract that records unfinished user work before interruption and later resumes, delivers, defers, or cancels it explicitly.

## What This Changes

The issue asks OpenClaw to prevent long-running promised outputs from being silently lost after heartbeat, hook, or operational task switches by tracking pending deliverables or active tasks durably per session.

## Best Possible Solution

Define a core-owned session-continuity contract with explicit active-task, pending-deliverable, send, defer, cancel, supersede, and resume states, then prove it with interruption and channel-visible delivery regressions.

## Reproduction Assessment

No high-confidence live reproduction was established against current main. Source inspection does show the current mechanisms are post-output delivery recovery, queue/drop/interrupt handling, and opt-in commitments rather than a durable promised-work stack.

## Solution Assessment

Unclear as a narrow fix. A durable pending-deliverables queue is directionally plausible, but maintainers need to define cancellation, supersession, replay, duplicate-prevention, and channel-delivery semantics before implementation.
