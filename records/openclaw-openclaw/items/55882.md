## Summary

Keep this open. Current main has adjacent fixes, including the heartbeat pending-final-delivery fix in https://github.com/openclaw/openclaw/pull/80357, but it still does not provide a durable per-session record for arbitrary unfinished promised outputs before a non-restart task switch or interrupt.

## What This Changes

Request a durable per-session pending-deliverables or task-stack mechanism so long-running chat promises resume, defer, or clear after unrelated high-priority interruptions without requiring a gateway restart.

## Best Possible Solution

Define a typed session-continuity contract that records pending deliverables before preemptive switches and clears them only when sent, explicitly deferred, cancelled, or superseded, with regression coverage for non-restart interruption.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a live Telegram/autonomy-guard run, but source inspection gives a credible current-main shape: interrupts can abort active work, while durable pending delivery is only written after final payloads exist.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers define the product contract; the maintainable direction is a typed durable pending-deliverable mechanism rather than stretching heartbeat replay, optional commitments, or in-process follow-up queues.
