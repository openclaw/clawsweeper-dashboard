- `P1`: The issue reports a real LINE channel workflow where users may receive no agent reply after reply-token expiry or Push quota exhaustion.
- `impact:message-loss`: The affected behavior is dropped or undelivered LINE replies after the runtime falls back from Reply API to Push API.

## Summary

Keep open: current main and latest release still lack LINE Push quota monitoring or 429/retry safeguards for fallback sends; a loading keepalive is already present, and the open related PR only covers one focused table/Flex quota-loss branch.

## What This Changes

Request safeguards for LINE reply-token expiry by improving Push fallback quota/retry handling and long-running user feedback.

## Best Possible Solution

Review and land or reject the focused Reply-slot batching PR for the table/Flex quota branch, then decide a narrow LINE push-fallback policy for quota checks, retry keys/backoff, and user-visible failure reporting.

## Reproduction Assessment

Yes at source level for the missing safeguards: current Push sends rethrow API errors without quota consumption checks, retry keys, or 429 handling. Full end-to-end confirmation still needs a live LINE account in the relevant quota state.

## Solution Assessment

Unclear as a single fix: immediate acknowledgement plus always-push final answers would trade reply-token expiry for higher Push quota use. The safer direction is to finish the focused Reply-slot batching review, then make a maintainer-approved quota/retry policy change.
