## Summary

Keep open. Current main still has a Telegram text-only streaming finalization path that marks a preview as delivered and returns `preview-finalized` without entering the normal `deliverReplies` path where `message_sending` hooks run; related items cover broader streaming-hook design or truncation fixes, not this narrow final-hook bypass.

## What This Changes

The issue reports that Telegram streamed final replies bypass `message_sending` hooks when `streaming` is `partial` or `full`.

## Best Possible Solution

Apply the existing `message_sending` rewrite/cancel contract exactly once before a Telegram streamed final preview is marked delivered, while preserving one visible preview and leaving full streaming-edit hook design to the broader follow-up.

## Reproduction Assessment

Yes, from source inspection. Current main can return `preview-finalized` for an active Telegram streamed final without `sendPayload` or `deliverReplies`, while normal Telegram delivery is the inspected path that runs `message_sending`.

## Solution Assessment

Yes for this narrow issue. The maintainable fix is to apply the existing outbound hook contract before final preview finalization exactly once, not to add a second final message or solve the broader streaming-edit API here.
