## Label Justifications

- `P2`: This is a bounded Telegram/plugin hook-coverage bug with real guardrail impact, but not a core outage or data-loss emergency.
- `impact:security`: Plugins can use `message_sending` as an outbound filter or cancellation guard, and streamed Telegram finals can currently bypass that guardrail.

## Summary

Keep open. Current main still has a source-reproducible Telegram streaming path where a visible final preview can be marked delivered and mirrored without routing the final text through the `deliverReplies` path that applies `message_sending` rewrite/cancel hooks.

## What This Changes

The issue asks for Telegram streamed final replies to honor the documented `message_sending` hook contract instead of bypassing it during preview finalization.

## Best Possible Solution

Patch Telegram preview finalization to run the existing `message_sending` rewrite/cancel contract exactly once before a visible final preview is retained or edited, then keep broader streaming-edit lifecycle design in the separate open feature request.

## Reproduction Assessment

Yes, by source inspection. A text-only Telegram streaming final with an active draft stream can return `preview-finalized` without `sendPayload` or `deliverReplies`, while the normal Telegram send path is where `message_sending` runs.

## Solution Assessment

Yes for this narrow bug. The maintainable fix is to apply the existing outbound hook contract at the preview-finalization seam exactly once, not to add a second final bubble or solve every streaming edit hook use case here.
