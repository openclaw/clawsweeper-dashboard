## Summary

Keep open. Current main still has a Telegram streaming preview-finalization path that can edit or retain the final preview without going through `deliverReplies`, while `message_sending` is only applied inside `deliverReplies`; related Discord and Slack items support the hook-coverage expectation but do not fix this Telegram path.

## What This Changes

The issue asks Telegram streaming final deliveries to fire `message_sending` hooks for rewrites and cancellation the same way non-streaming sends do.

## Best Possible Solution

Telegram final preview edits should apply the same outbound hook contract as normal final sends before any visible final edit, honoring rewrites and cancellation inside the Telegram plugin boundary.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: with streaming preview active, text-only final delivery can edit the preview and skip `sendPayload`, while `message_sending` only runs inside `deliverReplies`.

## Solution Assessment

Yes. The narrowest maintainable solution is a Telegram-owned hook application around final preview edits, with regression tests for edit, rewrite, cancel, fallback, and no-hook paths.
