## Summary

Keep open. Current main still has a Telegram streaming final-preview path that can edit or retain the final visible message without entering `deliverReplies`, while `message_sending` is only applied in the normal reply-send path.

## What This Changes

The issue asks Telegram streaming final delivery to invoke `message_sending` hooks before every final outbound message, including preview edits.

## Best Possible Solution

Keep this as a Telegram plugin fix: apply the outbound `message_sending` contract before final preview edits, honoring rewrite and cancel semantics without broad core changes.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: with streaming preview active, a text-only final reply can edit or retain a preview and skip `sendPayload`, while `message_sending` only runs inside `deliverReplies`.

## Solution Assessment

Yes. The narrowest maintainable solution is Telegram-owned hook handling around final preview edits before `editMessageTelegram`, with regression tests for rewrite, cancel, fallback, retained-preview, and no-hook paths.

## Repair Work Prompt

Fix Telegram streaming final-preview delivery so `message_sending` hooks run before the final visible message is delivered by preview edit. Keep the fix inside `extensions/telegram`; do not add bundled-plugin IDs or broad core behavior. The hook application should match Telegram `deliverReplies` semantics as closely as practical: call the global hook runner only when `message_sending` hooks exist, pass Telegram channel/account/conversation/thread/reply metadata, honor `{ cancel: true }` by suppressing the final visible edit, and apply rewritten `content` before `editMessageTelegram`. Preserve existing duplicate-avoidance behavior for post-connect edit errors, long-lived previews, archived previews, and fallback sends. Add focused regression tests for successful rewrite before final edit, cancel before final edit, no-hook fast path, fallback send still using `deliverReplies`, and retained-preview/error behavior. Add a single-line user-facing `CHANGELOG.md` fix entry without bot attribution.
