## Summary

Keep open. Current main still lets Telegram streaming finalization edit or retain an existing preview without passing through `deliverReplies`, while Telegram’s `message_sending` hook handling is only wired in the normal reply-delivery path.

## What This Changes

The issue asks OpenClaw to fire Telegram `message_sending` plugin hooks for final outgoing messages even when Telegram streaming is set to `partial` or `full`.

## Best Possible Solution

Telegram final preview edits should apply the same outbound hook contract as normal final sends before any visible final edit, including rewrite and cancellation semantics inside the Telegram plugin boundary.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: with a transient streaming preview, text-only final delivery can edit or retain the preview and return before `sendPayload`, while `message_sending` only runs in `deliverReplies`.

## Solution Assessment

Yes. The narrowest maintainable fix is Telegram-owned hook application around final preview edits, with regression coverage for edit, rewrite, cancel, fallback, no-hook, and retained-preview paths.

## Repair Work Prompt

Fix issue #61550: Telegram streaming final delivery currently bypasses `message_sending` hooks when final text is delivered by editing or retaining a streaming preview instead of going through `deliverReplies`. Keep the fix inside the Telegram plugin boundary unless a tiny generic helper is clearly reusable. Apply the outbound `message_sending` contract before final preview edits so hooks can rewrite content or cancel delivery; preserve existing duplicate-avoidance behavior, long-lived-preview fresh-final fallback, archived-preview cleanup, edit-error retention/fallback decisions, and `message_sent` behavior. Add focused tests for successful preview edit with hook rewrite, hook cancel, no-hook behavior, fallback send behavior, and retained-preview/error paths. Do not broaden this into a multi-channel hook refactor. Add a user-facing changelog fix entry if the implementation changes runtime behavior.
