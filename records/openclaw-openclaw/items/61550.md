## Summary

Keep open. Current main still has a Telegram preview-finalization path that can deliver the final text by editing the existing preview message directly, while the `message_sending` hook is only wired through the normal `deliverReplies` send path. The related Discord and Slack hook reports support the expected behavior, but they do not implement this Telegram streaming-preview path.

## What This Changes

The issue reports that Telegram preview streaming final delivery bypasses `message_sending` plugin hooks when `channels.telegram.streaming` is enabled.

## Best Possible Solution

Telegram preview final delivery should run the same outbound hook contract as normal final delivery before any visible final preview edit, applying content rewrites and honoring cancellation without moving this channel-owned behavior into core or changing unrelated channel semantics.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: text-only preview finals edit the preview and avoid `sendPayload`, while `message_sending` is invoked only inside `deliverReplies`.

## Solution Assessment

Yes. The narrowest maintainable fix is Telegram-owned hook application around final preview edits, with focused regression coverage for edit, rewrite, cancel, and existing fallback delivery behavior.

## Repair Work Prompt

Fix the Telegram streaming `message_sending` hook bypass reported in https://github.com/openclaw/openclaw/issues/61550. On current main, final preview delivery can edit an existing Telegram preview through `createLaneTextDeliverer` / `editPreview` without going through `deliverReplies`, where `runMessageSending` currently lives. Keep the fix Telegram-owned and core-agnostic. Ensure final preview edits run the same `message_sending` contract before a visible final edit: pass useful routing context matching normal Telegram delivery (`channelId: "telegram"`, accountId, conversation/chat id, thread id, and reply target where available), apply rewritten content to the edited final text, and honor `{ cancel: true }` by skipping edit/send. Preserve existing duplicate-avoidance, long-lived-preview fresh-final fallback, archived-preview cleanup, and Telegram edit-error behavior. Add focused regression tests for an existing preview finalizing by edit: hook fires once, rewrite edits modified text, cancel suppresses edit/send, and the normal `deliverReplies` hook path still passes. Add a user-facing changelog fix entry if repo policy requires it, crediting the reporter as `Thanks @Postmanplayx`; do not change unrelated channels or broaden unbypassable outbound policy semantics.
