## Summary

Keep open: current main and the latest release still rebuild `tools.web.search` in a way that drops unknown record-valued entries during legacy web search migration. The bug is narrow, source-supported, and suitable for a focused fix with regression coverage.

## What This Changes

Preserve unrelated record-valued keys in legacy web search config migration while still migrating known legacy provider config.

## Best Possible Solution

Update the migration to copy unrelated record-valued search config while removing only positively migrated legacy provider records, known migrated globals, and dangerous prototype keys.

## Reproduction Assessment

Yes. Passing a config with `tools.web.search.apiKey` plus an unrelated record-valued key into `migrateLegacyWebSearchConfig` reaches the loop at lines 203-212, where that unrelated record is not copied into `nextSearch`.

## Solution Assessment

Yes. The narrow maintainable fix is to change the omission predicate, mirror the safer web-fetch preservation pattern, and add a regression test for unknown records plus prototype-key safety.
