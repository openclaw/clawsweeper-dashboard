## Summary

Keep open: current main and the latest release still rebuild `tools.web.search` in a way that preserves unknown scalars but drops unknown record-valued entries when any mapped legacy web-search key triggers migration. The bug is narrow, source-supported, and suitable for a focused config-preservation fix.

## What This Changes

The issue reports that legacy web search migration can drop unrelated record-valued configuration under `tools.web.search`.

## Best Possible Solution

Update the doctor legacy web-search migration to preserve unrelated record-valued search config while removing only positively migrated legacy provider records, the migrated top-level `apiKey`, and prototype-pollution-sensitive keys.

## Reproduction Assessment

Yes. By source inspection, a config with `tools.web.search.apiKey` plus an unrelated record-valued key reaches the loop that omits that record from `nextSearch`.

## Solution Assessment

Yes. The narrow maintainable fix is to adjust the omission predicate, mirror the safer web-fetch migration pattern, and add regression coverage for unknown records plus prototype-key safety.
