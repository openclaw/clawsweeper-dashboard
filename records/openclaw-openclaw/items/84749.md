- `P1`: The PR addresses a user-facing hosted installer regression that can block OpenClaw installation for users with npm freshness policy enabled.

## Summary

Keep open: the installer bug is valid and the PR targets the right files, but the patch has a blocking config-scope bug for global npm installs and the supplied proof still does not show the patched hosted installer running with real npm.

## What This Changes

The PR adds raw npmrc scanning to the hosted shell and PowerShell installers plus fixture tests so installer npm calls keep `--min-release-age=0` when a raw `min-release-age` setting is present.

## Best Possible Solution

Restrict raw npmrc probing to the config scopes npm actually uses for each installer command, add mixed project/user/global regression coverage, then supply redacted real hosted-installer proof.

## Reproduction Assessment

Yes. Current `main` has a clear source-level reproduction for the linked installer bug, and an isolated npm 11.12.1 probe confirms the before/min-release-age conflict and the global-mode project `.npmrc` boundary.

## Solution Assessment

No as written. Raw npmrc scanning is the right direction, but the hosted global installer path must not unconditionally read project `.npmrc` because npm ignores that file for `npm install -g`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Ignore project npmrc in global installer probes:** `scripts/install.sh:750`
  - body: The new raw-key probe always reads `$(pwd)/.npmrc`, but this helper chooses flags for `npm install -g`. npm does not read project `.npmrc` in global mode, so a project `min-release-age` can make this patch select `--min-release-age=0` while the effective user/global config still has `before`; npm then rejects the mutually exclusive config. Limit the hosted installer probe to user/global scopes for global installs and add mixed-config coverage, mirrored in `install-cli.sh` and `install.ps1`.
  - confidence: 0.91
