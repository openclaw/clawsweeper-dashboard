## Summary

Keep open: the branch is a plausible central fix for https://github.com/openclaw/openclaw/issues/93568 and current main plus v2026.6.9 still only auto-allowlist ClickClack, but merging would widen a compatibility- and trust-sensitive plugin allowlist gesture that needs maintainer acceptance.

## What This Changes

The branch generalizes `enableExplicitlySelectedPluginInConfig` to auto-append any explicitly selected plugin to a restrictive `plugins.allow` after an allowlist block, with updated enable tests for a channel plugin and denylist preservation.

## Best Possible Solution

Merge the helper-level fix only if maintainers confirm explicit plugin selection should update restrictive `plugins.allow` across setup, onboarding, and CLI enablement; otherwise narrow the mutation to channel setup or add an explicit prompt before editing the allowlist.

## Reproduction Assessment

Yes, from source inspection: current main and v2026.6.9 only auto-allowlist `clickclack` after an allowlist block, while the linked Weixin setup path selects a different plugin through the same helper. I did not run a live interactive Weixin setup in this read-only review.

## Solution Assessment

Likely yes if maintainers accept explicit selection as the allowlist trust gesture, because the helper already centralizes the affected setup, onboarding, and CLI paths. If that policy is too broad, the safer solution is to scope auto-append to channel setup or prompt before mutation.
