## What This Changes

The PR updates `enableExplicitlySelectedPluginInConfig` and its tests so any explicitly selected plugin can be appended to a restrictive `plugins.allow`, rather than only ClickClack.

## Best Possible Solution

Merge a single helper-level fix only if maintainers accept explicit plugin selection as the allowlist trust gesture across setup, CLI, and onboarding; otherwise narrow the mutation to the channel setup path.

## Reproduction Assessment

Yes from source: current main calls the explicit-enable helper from setup paths, and that helper still returns `blocked by allowlist` for non-ClickClack ids under a restrictive `plugins.allow`. I did not run a live Weixin setup in this read-only review.

## Solution Assessment

Likely yes as the clean owner boundary if maintainers accept the broader trust gesture, because the helper already represents explicit user selection across setup, CLI, and onboarding. If that policy is too broad, the safer alternative is a narrower channel-setup-only mutation.
