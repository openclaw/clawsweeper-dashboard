## Summary

Keep this PR open for maintainer review: current main and v2026.6.11 still have the linked bug, and the patch is a plausible central fix, but it broadens plugin allowlist trust policy across all explicit-enable callers.

## What This Changes

The PR removes the ClickClack-only allowlist retry in plugin enablement and updates tests to expect auto-allowlisting for another explicitly selected channel plugin.

## Best Possible Solution

Merge or revise this only after maintainers confirm where the explicit-selection trust gesture belongs; if helper-level policy is too broad, scope the mutation to channel setup or add an explicit prompt before editing plugins.allow.

## Reproduction Assessment

Yes from source inspection: current main rejects non-ClickClack explicit selections under a restrictive allowlist, while the linked Weixin setup path routes selected channel plugins through that helper. I did not run the live interactive Weixin flow in this read-only review.

## Solution Assessment

Likely yes if maintainers accept explicit selection as the allowlist trust gesture across helper callers. If that policy is too broad, the safer solution is to scope auto-append to guided channel setup or prompt before mutating plugins.allow.
