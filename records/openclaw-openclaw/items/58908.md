## Summary

Keep open. Current main still lacks the requested human-readable cron timestamp fields, and this PR is a useful implementation candidate, but it is not ready as-is because the new formatter can throw from best-effort log/event paths for finite out-of-range timestamps and the user-facing enhancement is missing the required changelog entry.

## What This Changes

The PR adds a `formatEpochMs()` helper, optional ISO-8601 cron timestamp fields in timer/backoff logs and finished events, and unit tests for normal, undefined, NaN, and Infinity inputs.

## Best Possible Solution

Revise this PR or a narrow replacement so the human-readable fields stay additive and best-effort: preserve the raw epoch-ms fields, return `undefined` for invalid or out-of-range dates, cover that edge case in tests, and add the required changelog credit before merge.

## Reproduction Assessment

Yes. The proposed helper can be exercised directly with `9000000000000000`: it passes the finite-number guard but `toISOString()` throws `RangeError`, and current cron parsing accepts the same value when provided as a positive digit string.

## Solution Assessment

No, not as written. Adding optional ISO-8601 companion fields is the right maintainable direction for #58574, but the formatter must be range-safe and the PR needs the repository-required changelog entry before it is the best version of the fix.
