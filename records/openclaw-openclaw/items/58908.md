## Summary

Keep open: current main still lacks readable cron timestamp fields, but this PR cannot merge as-is because the formatter can throw for finite out-of-range epoch values and real behavior proof is missing.

## What This Changes

The PR adds ISO-8601 companion fields to cron timer/backoff logs and finished events, plus a `formatEpochMs` helper and unit tests.

## Best Possible Solution

Land or replace a narrow additive implementation that preserves numeric fields, uses range-safe ISO formatting, covers out-of-range values, and includes redacted runtime cron log proof.

## Reproduction Assessment

Yes. Source inspection on current main shows only numeric `nextAt`/`nextRunAtMs` fields, and the PR-specific formatter failure is reproducible with Node throwing `RangeError` for a finite out-of-range epoch value.

## Solution Assessment

No, not as written. Additive companion fields are the right direction, but the formatter must be range-safe and the PR needs real after-fix behavior proof before merge.
