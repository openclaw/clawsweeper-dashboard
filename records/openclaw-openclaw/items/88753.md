## Summary

Keep open: current main and v2026.6.8 still expose context-engine assembly as provider messages plus optional system-prompt/projection fields, so the requested lower-authority historical/reference channel remains missing and needs maintainer API/security design.

## What This Changes

The issue requests a host-owned context-engine output channel for lower-authority historical/reference context instead of forcing engines to serialize compacted summaries as synthetic user messages.

## Best Possible Solution

Design a narrow plugin SDK reference-context contract, render it below system/developer instructions and above the active user request across supported harnesses/providers, and add regression coverage plus lossless-claw coordination.

## Reproduction Assessment

Yes, from source inspection: current main's AssembleResult has no reference/historical context field, and the embedded runner consumes only messages plus systemPromptAddition. I did not run a live prompt-injection exploit because the source-level contract gap is enough to verify the missing capability.

## Solution Assessment

Yes, the host/plugin SDK boundary is the right layer for the remaining role-authority gap. The exact field shape and provider rendering semantics are still a maintainer security/product decision.
