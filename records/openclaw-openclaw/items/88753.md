## Summary

Keep open: current main and v2026.6.8 still lack a public context-engine reference/historical data lane. Main has Codex-specific safety-labeled projection for assembled messages, but the SDK contract still returns messages, systemPromptAddition, and contextProjection, and the embedded runner consumes assembled messages as model history, so the central request remains necessary.

## What This Changes

Add a host-owned context-engine output channel for non-instruction historical/reference data so plugins do not serialize untrusted summaries as fresh user messages.

## Best Possible Solution

Design a narrow plugin SDK reference-context lane and host rendering path that preserves lower authority across embedded and Codex/native provider flows, with docs and regression coverage for lossless-claw.

## Reproduction Assessment

Yes. Source inspection shows current AssembleResult has no reference/historical context field, and both embedded and Codex paths consume assembled messages plus optional system/developer additions rather than a separate lower-authority lane.

## Solution Assessment

Yes, the host/plugin SDK boundary is the right layer for the remaining authority gap. The exact field shape and provider/Codex rendering semantics still need maintainer security/product design.
