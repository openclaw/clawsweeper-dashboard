- `impact:security`: The issue is about preventing untrusted compacted history from regaining instruction authority during later model turns.
- `impact:session-state`: The affected data is compacted historical session context that can be replayed into future turns with the wrong authority.

## Summary

Keep open: current main still has the reported host-contract gap, and the downstream lossless-claw mitigation explicitly leaves the role-authority boundary unresolved until OpenClaw provides a provider-safe non-instruction context lane.

## What This Changes

The issue requests a host-owned context-engine output lane for untrusted historical/reference context so downstream engines do not have to serialize compacted summaries as fresh user messages.

## Best Possible Solution

Design a narrow plugin SDK/host context-reference contract that renders untrusted historical context below system/developer instructions and outside fresh user authority, with provider/harness tests and downstream lossless-claw coordination.

## Reproduction Assessment

Yes, from source inspection: current main exposes only messages/systemPromptAddition/contextProjection from context-engine assembly, and the runner only consumes those fields. I did not run a live exploit reproduction, but the linked downstream PR/comment shows lossless-claw still has to keep summaries as user-role messages after its mitigation.

## Solution Assessment

Yes, the requested OpenClaw host contract is the right layer for the remaining gap. The exact API shape is still a maintainer security/product decision because role-specific local workarounds can either increase authority or break provider/tool-call validity.
