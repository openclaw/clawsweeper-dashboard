## Summary

Keep open: current main and the latest release still lack the requested SDK/host contract for lower-authority historical reference context. Codex has a harness-specific projection that labels assembled context as reference data, but the generic context-engine contract still returns model messages plus systemPromptAddition, so the downstream lossless-claw authority-boundary gap remains.

## What This Changes

Request a provider-safe context-engine output lane for untrusted historical/reference context so compacted summaries do not need to be replayed as fresh user messages.

## Best Possible Solution

Design a narrow plugin SDK/host reference-context contract with explicit trust semantics, cross-provider rendering tests, Codex harness coverage, docs, and downstream lossless-claw coordination.

## Reproduction Assessment

Yes, from source inspection: current main and v2026.6.6 expose only assembled messages plus systemPromptAddition/contextProjection, and the generic runner consumes assembled.messages as model messages. I did not run a live prompt-injection exploit, but the linked downstream issue documents the remaining user-role summary path after its merged mitigation.

## Solution Assessment

Yes, the host-contract direction is the right layer because local role swaps either raise authority or risk provider/tool-call validity. The exact API shape is still a maintainer security/product decision, especially with Codex's partial projection and generic provider paths to reconcile.
