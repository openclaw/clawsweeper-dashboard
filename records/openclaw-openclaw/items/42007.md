## Summary

Keep this PR open. Current main still lacks Feishu text keyword mention triggers, so the contribution is not obsolete, but the submitted patch is not merge-ready because keyword-triggered broadcast groups would still have no active Feishu reply dispatcher and blank trigger entries can bypass `requireMention`.

## What This Changes

The PR adds a Feishu `mentionTriggers` config field and uses it to let configured text keywords trigger group handling when `requireMention` is enabled and there is no native Feishu bot mention.

## Best Possible Solution

Keep the PR open and request a Feishu plugin revision that defines the final keyword-trigger config contract, normalizes or rejects blank entries, treats a keyword match as an effective mention through both single-agent and broadcast dispatch, updates `WasMentioned`, and adds focused schema, policy/bot, broadcast, docs, and config-metadata coverage.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure a Feishu broadcast group with `requireMention: true` and a keyword trigger, send a group text containing the keyword without a native bot mention, and the PR gate would admit the message while `activeAgentId` remains `null`; a config with `mentionTriggers: [""]` also reproduces the blank-trigger bypass because `includes("")` is always true.

## Solution Assessment

No. The proposed patch is a useful start, but it only bypasses the first mention gate; the maintainable fix should resolve an effective mention decision once, validate/normalize trigger entries, and carry that result through broadcast activation and inbound context instead of leaving later paths tied to `ctx.mentionedBot`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mark keyword-triggered broadcasts as active:** `extensions/feishu/src/bot.ts:1057`
  - body: When `requireMention` is true and a keyword match admits a message without a native bot mention, the later broadcast path still computes `activeAgentId` from `ctx.mentionedBot || !requireMention`. That leaves `activeAgentId` null, so broadcast agents run as passive observers and no Feishu reply dispatcher is created for the keyword-triggered message.
  - confidence: 0.9
- **[P2] Reject blank keyword triggers:** `extensions/feishu/src/config-schema.ts:160`
  - body: `mentionTriggers` accepts plain strings and the runtime checks `ctx.content.includes(t)`. An empty string is valid input and always matches, so `mentionTriggers: [""]` would make every group message pass the `requireMention` gate.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: No dependency, workflow, or supply-chain changes were found, but the diff introduces a config-controlled mention-gate bypass for blank trigger strings.

Concerns:

- **[medium] Blank triggers bypass group mention gating:** `extensions/feishu/src/config-schema.ts:160`
  - body: Because `z.array(z.string())` permits empty strings and `includes("")` is always true, an accidental blank trigger can make every Feishu group message eligible even when `requireMention` is intended to limit processing to addressed messages.
  - confidence: 0.88
