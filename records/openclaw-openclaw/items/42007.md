## Summary

## What This Changes

The PR adds a top-level Feishu `mentionTriggers` config field and treats substring keyword matches as satisfying `requireMention` for group text messages.

## Best Possible Solution

Implement validated non-empty Feishu keyword triggers with settled global/account/group inheritance, carry one effective addressed state through Feishu activation and broadcast dispatch, and prove it with focused tests plus redacted Feishu runtime evidence.

## Reproduction Assessment

Yes for the PR defects by source inspection: keyword-only messages can pass the new gate while broadcast `activeAgentId` and `WasMentioned` still use `ctx.mentionedBot`, and empty strings satisfy `ctx.content.includes(t)`. I did not run a live Feishu setup in this read-only review.

## Solution Assessment

No. The patch is a useful start, but the maintainable solution needs validated trigger values, settled config scope, and a single effective addressed decision shared across Feishu activation and dispatch paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Propagate keyword-triggered state into broadcast dispatch:** `extensions/feishu/src/bot.ts:1053-1057`
  - body: When this gate lets a keyword-only group message through, `ctx.mentionedBot` remains false. The broadcast path later computes `activeAgentId` and `WasMentioned` from `ctx.mentionedBot`, so broadcast groups dispatch every agent as an observer and create no Feishu reply dispatcher for the triggered turn.
  - confidence: 0.91
- **[P2] Reject blank mention triggers before matching:** `extensions/feishu/src/config-schema.ts:160`
  - body: The schema accepts `""`, and the runtime uses `ctx.content.includes(t)`. A blank trigger therefore matches every group message and bypasses `requireMention`, so empty or whitespace-only entries need to be rejected or normalized before matching.
  - confidence: 0.92
