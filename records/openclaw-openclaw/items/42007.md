## Summary

## What This Changes

The PR adds a top-level Feishu `mentionTriggers` array and uses substring keyword matching so group messages can satisfy `requireMention` without an actual Feishu `@mention`.

## Best Possible Solution

Implement validated non-empty Feishu keyword triggers through the current activation resolver and broadcast dispatch, settle account/group inheritance, and prove keyword, non-keyword, and broadcast behavior with focused tests plus redacted Feishu runtime evidence.

## Reproduction Assessment

Yes for the PR defects by source inspection: keyword-only messages can pass the new gate while broadcast dispatch still keys off `ctx.mentionedBot`, and empty strings satisfy `ctx.content.includes(t)`. I did not run a live Feishu setup in this read-only review.

## Solution Assessment

No. The patch is a useful start, but the maintainable solution needs validated trigger values, a settled public config scope, and one effective addressed decision shared across Feishu activation and dispatch paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Propagate keyword triggers into broadcast dispatch:** `extensions/feishu/src/bot.ts:1053-1057`
  - body: When this gate lets a keyword-only group message through, `ctx.mentionedBot` remains false. Broadcast dispatch later computes `activeAgentId` and `WasMentioned` from `ctx.mentionedBot`, so broadcast groups dispatch every agent as an observer and create no Feishu reply dispatcher for the triggered turn.
  - confidence: 0.91
- **[P2] Reject empty mention triggers before matching:** `extensions/feishu/src/config-schema.ts:160`
  - body: The schema accepts `""`, and runtime matching uses `ctx.content.includes(t)`. A blank trigger therefore matches every group message and bypasses `requireMention`, so empty or whitespace-only entries need to be rejected or normalized before matching.
  - confidence: 0.92
