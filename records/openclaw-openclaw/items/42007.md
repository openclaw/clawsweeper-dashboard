## Summary

Keep open: current main still lacks Feishu keyword-trigger support, but the proposed patch is not mergeable as-is because it loses keyword-triggered turns in broadcast dispatch, allows blank triggers to bypass mention gating, lacks real Feishu proof, and conflicts with the current ingress-policy shape.

## What This Changes

The PR adds a Feishu `mentionTriggers` config option and keyword check so group text containing configured words can satisfy `requireMention`.

## Best Possible Solution

Define validated non-empty Feishu keyword triggers with clear account/group inheritance, feed one effective addressed state through ingress and single/broadcast dispatch, document the contract, and add focused tests plus redacted real Feishu proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: keyword-only messages can pass the new gate with `ctx.mentionedBot` false, and blank strings satisfy substring matching. I did not run a live Feishu setup.

## Solution Assessment

No. The patch is a useful start, but current main routes mention gating through Feishu ingress policy and broadcast active-agent state, so the maintainable fix needs validated triggers, settled config scope, and one effective addressed decision shared across dispatch paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Propagate keyword-triggered state into broadcast dispatch:** `extensions/feishu/src/bot.ts:1057`
  - body: When this new gate lets a keyword-only group message through, `ctx.mentionedBot` stays false. The broadcast path still computes `activeAgentId` and `WasMentioned` from `ctx.mentionedBot`, so broadcast groups treat every agent as an observer and create no Feishu reply dispatcher for the triggered turn.
  - confidence: 0.91
- **[P2] Reject empty mention triggers before matching:** `extensions/feishu/src/config-schema.ts:160`
  - body: The schema accepts `""`, and the runtime uses `ctx.content.includes(t)`. A blank trigger therefore matches every group message and bypasses `requireMention`, so the config should reject or normalize empty trigger entries before matching.
  - confidence: 0.92

## Security Review
