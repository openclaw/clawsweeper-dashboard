The issue reports Discord embedded runs that reach `embedded_run:started` but stall before strict-agentic startup, with known-session lanes eventually aborting and `sessionId=unknown` lanes remaining unrecovered until gateway restart.

## Best Possible Solution

Register a cancellable session-key-owned embedded reply operation before prep and make stuck recovery abort that operation by session key after the threshold, while preserving legitimate active lane work and adding live Discord or equivalent transport proof.

## Solution Assessment

No, closing is not the best way to solve this issue because current main still has an intentional guard that preserves active unregistered lane work. The maintainable fix is targeted session-key-owned cancellation/recovery for embedded-run prep, not a broad lane reset.
