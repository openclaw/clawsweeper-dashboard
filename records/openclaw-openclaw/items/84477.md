## Summary

Keep open. The report is fresh, detailed, reproduced on the latest release, and current main still has a source-visible recovery gap for active session-key lanes that do not have a registered recoverable session id.

## What This Changes

The issue reports Discord embedded runs wedging after `embedded_run:started` and before model execution, with `sessionId=unknown` lanes not being recovered automatically.

## Best Possible Solution

Register a cancellable session-key-owned embedded reply/prep operation before long-running prep, then make stuck recovery abort that operation by session key after the threshold or emit a clear nonrecoverable diagnostic, with focused recovery tests and live Discord proof.

## Reproduction Assessment

Partly: current source gives a high-confidence reproduction path for the recovery gap because session-key-only processing can reach recovery and active unregistered lane work is kept. The full live Discord prep wedge has detailed release logs, but I did not reproduce that hang on current main.

## Solution Assessment

No, current behavior is not the best end state because unknown-session active lanes can remain unrecovered. The maintainable fix should be targeted session-key-aware cancellation or diagnostics, not a broad lane reset.
