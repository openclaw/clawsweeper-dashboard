## Summary

Keep open. Current main still lacks a safe targeted recovery for pre-runtime session-key-only Discord lanes: recovery deliberately keeps active unregistered lane work instead of aborting it, and the fresh live report is already assigned for owner review.

## What This Changes

The issue reports Discord embedded runs wedging after dispatch prep and before strict-agentic execution, with sessionId-unknown lanes not recovering automatically.

## Best Possible Solution

Register cancellable session-key-owned embedded reply work before prep, then have stuck recovery either abort that operation by session key after the threshold or emit a precise nonrecoverable diagnostic without clearing unrelated active work.

## Reproduction Assessment

Partly. There is a high-confidence source-level reproduction for the recovery gap: current main can track a queued/processing session by session key before a session id exists, and recovery keeps active unregistered lane work; I did not reproduce the full live Discord prep wedge.

## Solution Assessment

No, the current behavior is not the best end state because session-key-only prep wedges can remain unrecovered. A broad lane reset would be risky; the maintainable fix is explicit operation ownership before prep plus targeted abort or clearer diagnostics.
