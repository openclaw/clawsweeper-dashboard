## Summary

Keep open. The report is fresh, detailed, and current main still has a source-level recovery gap for active session-key lanes that do not have a registered session id or reply-run handle, so there is no conservative close reason.

## What This Changes

The issue reports Discord embedded reply runs wedging before strict-agentic execution and leaving session-key-only lanes unrecovered until a gateway restart.

## Best Possible Solution

Register a cancellable session-key-owned embedded reply/prep operation before any blocking prep work, then make stuck recovery abort that operation by session key after the configured threshold or emit a clear nonrecoverable diagnostic, with live Discord proof.

## Reproduction Assessment

Yes for the recovery gap at source level: session-key-only processing can reach recovery while active unregistered lane work is intentionally kept. No for the full live Discord prep wedge; the issue provides strong logs, but this review did not live-reproduce it on current main.

## Solution Assessment

No, current behavior is not the best end state because unknown-session active lanes can remain unrecovered. The safer fix is a targeted cancellable prep/reply operation rather than a broad lane reset.
