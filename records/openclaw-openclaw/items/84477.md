## Summary

Keep open. The report is fresh, assigned, and current main still has an intentional active-unregistered-lane guard that can leave a session-key-only embedded prep stall unrecovered, so there is no safe cleanup close reason.

## What This Changes

The issue reports Discord embedded runs wedging after `embedded_run:started` and before strict-agentic activation, with `sessionId=unknown` lanes not recovering automatically.

## Best Possible Solution

Register a cancellable session-key-owned embedded reply operation before prep that can wedge, then recover it by session key after the abort threshold or emit a clear nonrecoverable diagnostic without broadly resetting unrelated active lane work.

## Reproduction Assessment

Yes for the recovery gap at source level: session-key-only processing can reach recovery while active unregistered lane work is intentionally kept. No for the full live Discord prep wedge; the issue has strong logs, but I did not reproduce the live hang against current main.

## Solution Assessment

No, current behavior is not the best end state because unknown-session lanes can remain unrecovered. The safer fix is a targeted session-key-owned cancellable prep or operation path, not a broad lane reset.
