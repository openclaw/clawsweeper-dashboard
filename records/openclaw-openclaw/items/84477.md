## Summary

Keep open. The report is fresh, detailed, and current source still leaves a meaningful recovery-policy gap for active session-key lanes that cannot be tied to a registered embedded run, so there is no conservative close reason yet.

## What This Changes

The issue reports Discord embedded agent runs wedging after dispatch but before strict agent execution, with known-session recovery eventually aborting while session-key-only lanes appear to remain stuck until gateway restart.

## Best Possible Solution

Register or carry a cancellable session-key-owned reply operation before all embedded prep, then make stuck recovery abort that operation by session key at the abort threshold or emit a clear nonrecoverable diagnostic without clearing unrelated active lane work.

## Reproduction Assessment

Partly. The source-level recovery gap is reproducible from current code and tests for active unregistered session-key lanes, but I did not reproduce the full live Discord prep wedge on current main.

## Solution Assessment

No, the current behavior is not the best end state because some active session-key lanes can remain unrecovered. The safer fix is targeted session-key-owned cancellation or an explicit nonrecoverable diagnostic, not a blanket lane reset.
