## What This Changes

The issue reports Discord embedded runs reaching `embedded_run:started` but wedging before strict-agentic execution, with unknown-session lanes left unrecovered until gateway restart.

## Best Possible Solution

Register a cancellable session-key-owned embedded reply operation before prep, then let stuck recovery abort that operation by session key after the threshold or emit a clear nonrecoverable diagnostic without clearing unrelated active work.

## Reproduction Assessment

Yes for the recovery gap at source level: session-key-only processing can enter stuck recovery without a session id, and current tests intentionally keep active unregistered lane work. No for the full live Discord prep wedge; the issue provides strong logs, but I did not reproduce the live hang.

## Solution Assessment

No, current behavior is not the best end state because unknown-session lanes can remain unrecovered. The safer fix is a targeted session-key-owned cancellable prep/operation path with regression coverage, not a broad active-lane reset.
