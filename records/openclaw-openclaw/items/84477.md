## Summary

Keep open. Current main still has a source-visible recovery gap for session-key-owned active lane work before a registered session id is available, and the Discord live wedge is fresh with strong logs but no deterministic current-main repro yet.

## What This Changes

The issue reports Discord embedded reply runs stalling after `before_dispatch`/`embedded_run:started` and before model execution, with known-session recovery but `sessionId=unknown` lanes remaining stuck until gateway restart.

## Best Possible Solution

Register a cancellable session-key-owned reply operation before prep, then let stuck recovery abort that specific operation by session key after the abort threshold or emit a clear nonrecoverable diagnostic, with focused tests and a Discord/live proof run.

## Reproduction Assessment

No high-confidence live reproduction of the full Discord prep wedge is available from current main. The recovery gap is source-reproducible: session-key-only active lane work can be observed by diagnostics while recovery intentionally keeps that lane active instead of aborting it.

## Solution Assessment

No, the current behavior is not the best end state because `sessionId=unknown` lanes can remain unrecovered. A broad lane reset would be unsafe; the maintainable path is a targeted session-key-owned operation/timeout boundary with regression coverage.
