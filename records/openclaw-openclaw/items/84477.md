## Summary

Keep open. The report is fresh and detailed, and current main still has a session-key-only diagnostic path plus an explicit guard that keeps active unregistered lane work instead of recovering it, so there is no conservative cleanup close reason.

## What This Changes

The issue reports Discord embedded-run dispatches wedging after `embedded_run:started`, with known-session lanes eventually aborted but session-key-only lanes left unrecovered until gateway restart.

## Best Possible Solution

Register a cancellable session-key-owned embedded reply/prep operation before runtime dispatch, then let stuck recovery abort that operation by session key after the policy threshold or emit a clear nonrecoverable diagnostic without clearing unrelated active work.

## Reproduction Assessment

Yes for the recovery gap at source level: current main can track processing work by session key only and intentionally keeps active unregistered lane work. No for the full live Discord prep wedge; the issue provides strong logs, but this read-only review did not reproduce the live hang.

## Solution Assessment

No, current behavior is not the best end state because unknown-session lanes can remain unrecovered. The safer fix is a targeted session-key-owned cancellable prep/reply operation with focused recovery tests and live Discord proof, not an unconditional lane reset.
