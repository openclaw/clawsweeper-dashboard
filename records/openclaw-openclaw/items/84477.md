## Summary

## What This Changes

The issue reports Discord embedded runs wedging after `embedded_run:started` but before strict-agentic/model execution, with recovery covering known-session lanes but skipping `sessionId=unknown` lanes.

## Best Possible Solution

Register a cancellable session-key-owned embedded reply operation before prep, then make stuck recovery either abort that operation by session key after the abort threshold or emit a clear nonrecoverable diagnostic without clearing unrelated active lane work.

## Reproduction Assessment

Yes for the recovery gap at source level: session-key-only processing can enter recovery without a session id, and active unregistered lane work is intentionally kept. No for the full live Discord prep wedge; the issue provides strong logs, but I did not reproduce the live hang on current main.

## Solution Assessment

No, current behavior is not the best end state because unknown-session lanes can remain unrecovered. The safer fix is not a broad lane reset, but a session-key-owned cancellable prep/operation path with targeted recovery and tests.
