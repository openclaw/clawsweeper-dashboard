## Summary

Keep open: the PR targets a valid source-reproducible message-routing bug, but the branch currently infers the target by taking the first non-marker segment after the channel, which breaks supported per-account session keys by treating the account id as the peer id. The supplied proof is also a standalone copied parser probe, not patched production-path or live channel behavior.

## What This Changes

Adds a guarded message-tool fallback that replaces stale webchat current-channel context with channel and target values inferred from the agent session key, then threads those effective values through schema, description, secret scope, and runtime tool context.

## Best Possible Solution

Keep the narrow stale-webchat fallback direction, but parse the canonical session-key grammar explicitly, including `channel:account:direct:peer`, and add regression coverage for normal and per-account DM sessions before merge.

## Reproduction Assessment

Yes, at source level: current main forwards stale `webchat` context into `toolContext`, and the outbound runner can choose the internal UI sink before normal channel resolution. I did not run a live Signal or Telegram gateway reproduction in this read-only pass.

## Solution Assessment

No, not in its current form: the guarded fallback is the right narrow approach, but the parser must follow the supported session-key grammar instead of taking the first non-marker token as the peer. The safer fix is a canonical parser with targeted regression tests and production-path proof.
