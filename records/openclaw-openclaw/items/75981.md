## Summary

Keep open. Current main still has the normalizing startup path, and the proposed option matches the existing model-list performance seam, but the PR needs the required changelog entry and its new comment overstates the normalization contract because provider hooks can mutate context metadata.

## What This Changes

The PR changes `ensureContextWindowCacheLoaded()` to call `discoverModels(..., { normalizeModels: false })` and adds comments about avoiding per-model provider normalization during TUI/chat startup.

## Best Possible Solution

Land a context-warmup-only raw-discovery path with focused coverage for preserved context metadata exceptions and an `Unreleased` / `Fixes` changelog entry.

## Reproduction Assessment

Unclear: I did not live-run the TUI with a large registry, so the full user-visible hang is not reproduced here. Source inspection does give a high-confidence hot path: context warmup currently calls `discoverModels()` with default normalization, which normalizes every available registry entry.

## Solution Assessment

Mostly yes. Passing `normalizeModels: false` is the narrow startup-path change that matches the models-list seam, but the patch should correct the overbroad comment and add the required changelog before merge.
