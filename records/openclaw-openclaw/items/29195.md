## Summary

Keep open. Current main has partially addressed the report with ACPX permission docs, task status/error plumbing, parent-owned session visibility rules, and configurable ACP turn timeouts, but the central reliability gaps remain: no safe default/preflight for non-interactive Codex writes, no automatic stalled-session remediation beyond long/default timeout and maintenance cleanup paths, and changing the default to `approve-all` is explicitly security-sensitive.

## What This Changes

The issue requests more reliable non-interactive ACP/Codex sessions by surfacing permission failures, exposing owned ACP progress/status, cleaning up stalled sessions, documenting permission configuration, and adding a spawn-time health/preflight check.

## Best Possible Solution

Keep this as the canonical ACPX/Codex non-interactive reliability umbrella unless maintainers split it. The maintainable end state is a maintainer-approved permission profile or preflight with actionable remediation, parent-owned ACP child history/status access that does not require broad global visibility, and automatic timeout/cleanup/reporting for stalled ACP runs so operators do not need a manual process-kill runbook.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the permission-default failure class from current config, tests, and docs; unclear for the full 80-minute Codex stall because this read-only review did not run live ACP/Codex processes, but current docs still acknowledge that stalled-session symptom.

## Solution Assessment

Is this the best way to solve the issue? No single current fix is sufficient: docs and optional timeouts are useful partial mitigations, while defaulting to `approve-all` alone is security-sensitive. The safer path is maintainer-approved preflight/profile plus scoped visibility and cleanup changes.
