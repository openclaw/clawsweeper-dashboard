## Summary

Keep open: current main still depends on the legacy iMessage `watch.subscribe` notification path, and I did not find a shipped/current fix for the reported silent inbound failure.

## What This Changes

The issue asks to restore automatic iMessage inbound detection and replies after upgrading to 2026.5.3-1 when Messages DB rows arrive but the gateway stays silent.

## Best Possible Solution

Validate on a real macOS iMessage setup with current `imsg`, then repair the legacy monitor against the supported event stream or a documented deduped cursor fallback while keeping the BlueBubbles recommendation intact.

## Reproduction Assessment

No high-confidence live reproduction yet. Source inspection shows the reported silent-DB-row scenario is plausible because current main depends on `watch.subscribe` notifications, but the review did not run a macOS iMessage gateway and the report has contradictory text.

## Solution Assessment

Unclear. The narrow maintainable fix should be chosen after checking the current `imsg` contract; an ad hoc polling knob is not obviously better than restoring or migrating the event stream with cursor/dedupe semantics.
