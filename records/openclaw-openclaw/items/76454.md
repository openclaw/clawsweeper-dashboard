## Summary

Keep open. The prior #56450 fix is present, but this new report gives fresh v2026.5.2 evidence for a remaining doctor-specific/cross-process duplicate `.clobbered.*` path, and current main does not have regression coverage proving that `doctor --fix` avoids it across subprocess-style reads.

## What This Changes

The issue reports that `openclaw doctor --fix --yes` on v2026.5.2 can create dozens of identical `openclaw.json.clobbered.*` snapshots during one repair run.

## Best Possible Solution

Add doctor-level regression coverage for multiple repair/read passes across process boundaries, then ensure a sanctioned `doctor --fix` migration creates only the intended doctor backup and no redundant identical `.clobbered.*` files.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only pass. Source inspection shows a plausible current-main failure path when the persisted health signature is not shared or written, and the report supplies concrete v2026.5.2 artifacts.

## Solution Assessment

Unclear. The narrowest maintainable fix should first add a failing doctor/cross-process regression; after that, either repair the persisted-signature handoff or suppress forensic clobber snapshots only inside the sanctioned doctor repair flow.
