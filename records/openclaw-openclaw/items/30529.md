## Summary

Keep open. Current main has automatic Discord typing plumbing for inbound replies, component replies, and heartbeat delivery targets, but the requested `typing` or `typing-indicator` message-tool action is still absent from the shared action enum, agent schema discovery, CLI normalization, Discord action discovery, and Discord action dispatch. The related implementation PR #30535 was closed unmerged, so there is no merged candidate that resolves the remaining action surface.

## What This Changes

The issue requests native Discord typing-indicator support through the message tool, preferably as automatic lifecycle behavior or as an explicit `typing`/`typing-indicator` action.

## Best Possible Solution

Either add a backward-compatible shared `typing` or `typing-indicator` message action with Discord support and focused lifecycle tests, or explicitly document that OpenClaw only supports automatic run/heartbeat typing for Discord. If implemented, the change should wire the shared action enum, target-mode map, agent schema/description, CLI docs, Discord discovery, Discord dispatch to `sendTypingDiscord`, and tests for action acceptance, rejection, routing, endpoint invocation, and failure behavior.

## Reproduction Assessment

Yes. The missing feature is reproducible on current main by inspecting the action enum, message-tool schema construction, CLI parser, Discord discovery, and Discord dispatcher; all omit `typing`, while the CLI has a test proving unknown actions are rejected before dispatch.

## Solution Assessment

Unclear. A manual action is technically straightforward, but the discussion points to a broader cross-provider and lifecycle question, so maintainers should decide the action contract before a repair PR changes the public message-tool surface.
