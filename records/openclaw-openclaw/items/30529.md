## Summary

Keep open. Current main has automatic Discord typing and heartbeat typing plumbing, but the shared message-tool contract still has no supported `typing` action, and the generic-vs-Discord API/lifecycle choice needs maintainer direction.

## What This Changes

Request to expose native Discord typing indicators through the shared `message` tool, either automatically during processing or as an explicit typing-style action.

## Best Possible Solution

Define the public typing/busy control contract first, then implement Discord through the existing `sendTypingDiscord` and shared typing lifecycle with focused message-tool discovery, execution, docs, and regression coverage.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main can send automatic Discord typing signals but cannot reproduce the requested manual message-tool action because it is not in the advertised action contract.

## Solution Assessment

Unclear until maintainers choose the API. A generic channel-owned typing/busy action that reuses the existing heartbeat and typing lifecycle seams looks safer than adding an ad hoc Discord-only alias.
