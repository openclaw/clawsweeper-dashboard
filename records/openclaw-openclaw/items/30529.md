## Summary

Keep open: current main has Discord typing primitives and automatic reply/heartbeat typing, but it still does not expose a public `message(action="typing")` or generic message-tool typing action. The remaining work is a public channel API and lifecycle design decision, not conservative cleanup.

## What This Changes

The issue requests native Discord typing indicator support through the message tool, either automatically or through an explicit typing action.

## Best Possible Solution

Define an additive channel-owned typing or busy-control contract, then wire Discord discovery and execution to the existing typing primitive with lifecycle safeguards, docs, focused tests, and real Discord proof.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main has automatic Discord typing paths but no public message-tool `typing` or generic typing action.

## Solution Assessment

Unclear until maintainers choose the API shape. A generic channel-owned typing or busy action that reuses existing lifecycle safeguards looks safer than a Discord-only alias.
