## Summary

Keep open: current main still has adjacent memory search, promotion, Memory Wiki, and LanceDB memory surfaces, but it does not provide first-class review, edit, non-destructive forget/tombstone, or conflict-resolution operator flows. The KinthAI production comment and open sibling MVP issues are useful design evidence for sequencing, not reasons to close this issue.

## What This Changes

The issue requests operator flows for reviewing, editing, non-destructively forgetting, tombstoning/superseding, and resolving conflicts in the memory MVP.

## Best Possible Solution

Define the durable memory record/status/revision/provenance model first, then add small operator CLI/RPC or UI flows for batched review, edit-by-supersede, soft tombstone/forget, and explicit conflict resolution.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection on current main gives a high-confidence verification path: the requested first-class operator flows are absent, and the adjacent LanceDB forget path is still destructive deletion rather than tombstoning.

## Solution Assessment

Unclear as an immediate implementation because the schema, write pipeline, conflict policy, and retention model are still unsettled. The requested direction is maintainable if it builds on durable statuses, revisions, provenance, and review surfaces instead of wrapping current delete/append behavior.
