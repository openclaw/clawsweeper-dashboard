## Summary

This remains necessary: current main supports memory storage, retrieval, listing, and destructive forgetting, but not review queues, revision-backed edits, tombstones, supersession, or conflict resolution. It should remain open for coordinated product design with the linked Memory MVP schema, write-pipeline, and operator-surface issues.

## What This Changes

Requests first-class operator workflows to review, edit, forget, and resolve conflicts in durable memory without destructive deletion.

## Best Possible Solution

Define one plugin-owned canonical memory lifecycle with provenance, revisions, tombstones, supersession, and conflict state, then expose consistent review/edit/forget commands through the active memory slot while retaining a separate explicit hard-delete path for required erasure.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives a high-confidence current-main baseline: the available lifecycle surface physically deletes memories and lacks review, revision, tombstone, supersession, and conflict states.

## Solution Assessment

Yes at the product level, but not as an isolated implementation item. The canonical schema and write/conflict semantics in the linked Memory MVP cluster should be decided first, then these operator flows should target that single plugin-owned contract.
