## Summary

Keep open. Current main still has memory search, promotion/dreaming, and wiki-adjacent provenance/lint surfaces, but it does not provide first-class operator review, edit, soft-forget, or conflict-resolution flows; the KinthAI production comment adds useful design evidence rather than making this stale.

## What This Changes

Add operator-facing memory review, edit, soft-forget/tombstone, and conflict-resolution flows for the Memory MVP.

## Best Possible Solution

Define the durable record/status/revision/provenance model first, then add small operator CLI/RPC or UI flows for batched review, edit-by-supersede, tombstone/forget, and explicit conflict resolution in coordination with #42646, #42648, and #42651.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection on current main gives a high-confidence verification path: the requested operator flows are absent, and the existing LanceDB forget path still hard-deletes records.

## Solution Assessment

Unclear as a final implementation because the schema and write pipeline decisions are still open. The requested direction is maintainable if it lands on durable statuses, revisions, provenance, and review surfaces instead of wrapping the current delete behavior.
