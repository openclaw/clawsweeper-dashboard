## Summary

Keep open. Current main still lacks canonical durable memory maintenance flows; the existing surfaces are recall, promotion, and wiki-adjacent tooling, while the remaining work depends on open schema, write-pipeline, operator-surface, and retention-policy decisions.

## What This Changes

Add first-class Memory MVP operator flows for reviewing, editing, forgetting via non-destructive status changes, and resolving conflicting durable memory records.

## Best Possible Solution

Sequence this behind #42646, #42648, and #42651: define durable memory records with revision/status/provenance, route writes through dedupe and conflict handling, then expose small operator review, edit, tombstone/forget, supersede, and conflict-resolution flows through CLI/RPC or UI.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: source and docs inspection show the requested operator flows are absent and the existing LanceDB forget path is a hard delete.

## Solution Assessment

Unclear as a final implementation because the canonical schema and write pipeline are not settled yet. The requested direction is maintainable only if it lands on a durable record/status/revision model, not as a wrapper around existing deletion or wiki metadata edits.
