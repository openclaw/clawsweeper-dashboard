## Summary

Keep open. Current main still lacks first-class operator review/list, edit, non-destructive forget/tombstone/supersede, and conflict-resolution flows for canonical durable memory records; the existing surfaces are search, promotion/dreaming preview, memory-wiki companion metadata, and a LanceDB forget tool that hard-deletes records.

## What This Changes

Add operator-facing memory review/listing, edit, non-destructive forget/tombstone or supersede, and conflict-resolution flows for durable memory maintenance.

## Best Possible Solution

Sequence this with #42646, #42648, and #42651: first establish a canonical durable memory record, revision, provenance, and status model; then route writes through dedupe/conflict handling; then expose small operator CLI/RPC or UI flows for review, edit, tombstone/forget, supersede, and conflict resolution. Keep memory-wiki as a provenance/dashboard companion unless maintainers explicitly promote it into the canonical active-memory record path.

## Reproduction Assessment

Not applicable as a feature request, but the current-main verification path is high confidence: source and docs inspection show the requested operator flows are absent, while the existing LanceDB forget path is explicitly hard-delete behavior.

## Solution Assessment

Unclear as a final implementation until the related schema and write-pipeline issues are settled. The requested direction is maintainable only if it lands on a canonical durable memory model with revisions/statuses, not as an isolated wrapper around current LanceDB deletion or memory-wiki metadata edits.
