## Summary

Keep open. Current main still does not implement MCP readiness gating for advertised skills, so the PR is not obsolete, but the PR head still has a blocking precedence bug where a skill-level non-callable readiness record can fall through to a callable server record and advertise the skill anyway.

## What This Changes

The PR adds MCP metadata parsing and mcp-readiness.json prompt gating for MCP-backed skills, adds focused skills coverage and an opt-in mcporter readiness smoke script, and also touches qa-channel, package-boundary, gateway startup, cache-trace, and test plumbing.

## Best Possible Solution

Keep the PR under review only if maintainers want this as core safety behavior: make skill-level readiness records authoritative, add a regression test for denial precedence, document or enforce freshness semantics, and rebase or split out the unrelated/stale gateway and qa-channel/package-boundary changes before merge.

## Reproduction Assessment

Yes. On the PR head, create an MCP-backed skill with readiness containing skills[skill].callable:false and servers[server].callable:true; the code at src/agents/skills/workspace.ts:175-180 will still return true through the server fallback.

## Solution Assessment

No. The core gate is a plausible direction, but this patch is not the best merge shape until explicit skill-level denials win and the readiness contract is separated from stale or unrelated changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Short-circuit explicit skill readiness denials:** `src/agents/skills/workspace.ts:176-180`
  - body: When a readiness file has a skill-level record that is not callable, this code falls through to the server record and can still return true. That lets servers[server].callable:true advertise a skill whose own readiness entry explicitly says it is not callable; treat an existing skill record as authoritative before checking the server fallback.
  - confidence: 0.94
- **[P3] Use the user home for global mcporter checks:** `scripts/mcp-readiness-smoke.mjs:59-62`
  - body: For the global ~/.mcporter/mcporter.json row, workspaceDir falls back to root, so --verify-callable runs mcporter config list from the OpenClaw state directory instead of the user's home. That can fail or inspect the wrong workspace when only the global config exists; use os.homedir() for the non-workspace config path.
  - confidence: 0.78
