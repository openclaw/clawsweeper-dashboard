## Summary

Keep open. Current main prevents the older bare `/reset` and `/new` command fallthrough, but the reset lifecycle path can still emit empty `before_reset` payloads and zero-message `session_end` events, so the reported provider 400 path remains plausible and unresolved.

## What This Changes

The issue asks OpenClaw to avoid triggering model-backed reset/session lifecycle work with an empty transcript when `/reset` or `/new` is sent on an already-fresh Telegram session.

## Best Possible Solution

Define the empty-transcript lifecycle contract in core: keep useful reset/session observability, expose a reliable no-op or `hasConversationHistory` signal, and ensure model-backed lifecycle work is skipped when there are no prior conversation messages.

## Reproduction Assessment

Yes for triage. The issue includes concrete steps and provider logs, and current source still emits empty `before_reset` plus zero-message `session_end` lifecycle payloads; I did not perform a live Telegram run because this review is read-only.

## Solution Assessment

Unclear. Skipping empty-history model work is the right outcome, but the safest implementation is likely a contract-level no-op signal or narrow model-backed-work guard rather than blanket suppression of reset/session lifecycle hooks.
