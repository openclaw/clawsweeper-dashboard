## Summary

Keep open. Current main acknowledges bare `/reset` and `/new` without falling through as the user’s normal prompt, but the reset lifecycle still emits `before_reset` with empty transcript arrays and `session_end` payloads whose `messageCount` defaults to 0, so the reported plugin/lifecycle 400 path is still plausible and not proven fixed.

## What This Changes

Triggering `/reset` or `/new` on an already-fresh Telegram session with lossless-claw should avoid empty-history lifecycle model work so no Anthropic 400 or user-facing error is produced.

## Best Possible Solution

Define and test the empty-transcript lifecycle contract: already-fresh resets should not trigger model-backed wrap-up work, while lifecycle plugins still get a clear cleanup signal or an explicit empty/no-op marker they can handle safely.

## Reproduction Assessment

Yes for triage: the issue provides concrete Telegram steps and provider/cooldown logs, and current source still exposes empty reset/session lifecycle payloads. I did not perform a live Telegram/lossless-claw run because this review is read-only.

## Solution Assessment

Unclear. Skipping the empty-history model work is the right outcome, but blanket suppression of `session_end` or `before_reset` may be too broad; a safer solution should make empty transcript state explicit or guard only the model-backed lifecycle work.
