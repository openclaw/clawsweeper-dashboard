## Summary

Keep open. Current main acknowledges bare `/reset` and `/new`, but the lifecycle paths still emit `before_reset` payloads with `messages: []` and `session_end` payloads with `messageCount: 0`, so the reported lossless-claw/provider 400 path remains plausible and is not fixed on main.

## What This Changes

The issue requests that already-fresh `/reset` and `/new` sessions skip the empty-history lifecycle LLM work so users do not see an Anthropic 400 or auth-profile cooldown.

## Best Possible Solution

Define and test the empty-transcript lifecycle contract so already-fresh resets do not run model-backed wrap-up work, while preserving cleanup/observation hooks or exposing an explicit no-op signal that plugins can handle safely.

## Reproduction Assessment

Yes for triage: the issue provides concrete Telegram steps and provider/cooldown logs, and current main still emits empty reset/session lifecycle inputs. I did not run a live Telegram plus lossless-claw reproduction in this read-only review.

## Solution Assessment

Unclear. Skipping the empty-history model work is the right outcome, but blanket suppression of `before_reset` or `session_end` hooks may break plugin lifecycle cleanup; a narrower contract or explicit empty-transcript guard is safer.
