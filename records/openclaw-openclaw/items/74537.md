## Summary

Keep open. Current main acknowledges bare `/reset` and `/new`, but the reset lifecycle still sends empty-history `before_reset` payloads and emits `session_end` with `messageCount: 0`, so the reported lossless-claw/provider 400 path is still plausible and not implemented away.

## What This Changes

Already-fresh `/reset` and `/new` should avoid model-backed lifecycle work with an empty transcript so no provider 400 is surfaced to the user.

## Best Possible Solution

Define and test the empty-transcript reset lifecycle contract so already-fresh resets do not run model-backed wrap-up work, while preserving lifecycle notifications or exposing an explicit empty/no-op signal that plugins can consume safely.

## Reproduction Assessment

Yes for triage: the issue includes concrete Telegram steps and provider/cooldown logs, and current source still emits empty reset/session lifecycle inputs. I did not run a live Telegram plus lossless-claw reproduction during this read-only review.

## Solution Assessment

Unclear. Skipping the failing empty-history model work is the right outcome, but blanket suppression of lifecycle hooks may be too broad; a safer fix should make the empty-transcript state explicit or guard only the model-backed work.
