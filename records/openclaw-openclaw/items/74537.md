## Summary

Keep open. Current main acknowledges bare `/new` and `/reset`, but it still emits empty reset/session lifecycle payloads and still permits runtime-only blank prompt submissions, so the reported provider 400 path is not implemented away.

## What This Changes

The issue asks OpenClaw to avoid model-backed reset/session-end lifecycle work when `/reset` or `/new` runs against an already-fresh session with no prior messages.

## Best Possible Solution

Define and test the empty-transcript lifecycle contract: preserve useful lifecycle notifications, but expose an explicit no-op state or guard model-backed lifecycle work so empty-history resets cannot submit provider calls with no messages.

## Reproduction Assessment

Yes for source-level triage: the issue includes concrete steps/logs, and current main still emits empty `before_reset` plus zero-count `session_end` payloads. I did not run a live Telegram/lossless-claw reproduction because this review is read-only.

## Solution Assessment

Unclear. Skipping empty-history model work is the right outcome, but the maintainable fix should distinguish observation hooks from model-backed wrap-up work rather than blindly suppressing every lifecycle notification.
