## Summary

Keep open: this is a real Slack feature PR, but it is not merge-ready because a matched pre-router hit bypasses the current Slack turn/session/status delivery path, lacks real Slack proof, and adds a raw-prompt HTTP egress surface that needs maintainer/security direction.

## What This Changes

Adds an opt-in Slack monitor HTTP pre-router controlled by `OPENCLAW_PRE_ROUTER_URL` and timeout env vars, posts matched responses directly to Slack, and adds hook/unit plus local HTTP fixture tests.

## Best Possible Solution

If maintainers want this feature, keep it opt-in but route matched responses through the existing Slack inbound-turn and delivery pipeline, document the trust boundary/env behavior, and require dispatch coverage plus redacted real Slack proof.

## Reproduction Assessment

Yes. Source inspection of the PR head shows a matched pre-router hit returns at `extensions/slack/src/monitor/message-handler/dispatch.ts:496` before current main's `dispatchChannelInboundReply` path records the turn and applies bot-loop/session/status handling.

## Solution Assessment

No. The early direct-send return is a plausible shortcut, but the best maintainable shape is to integrate matched responses with the existing Slack turn/delivery pipeline or an explicit plugin/API design.
