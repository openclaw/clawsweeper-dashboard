## Summary

## What This Changes

Adds an opt-in Slack monitor HTTP pre-router controlled by `OPENCLAW_PRE_ROUTER_URL` and timeout env vars, plus unit and local HTTP fixture tests, to post matched responses before LLM dispatch.

## Best Possible Solution

If maintainers want this feature, keep it opt-in but route matched responses through the existing Slack inbound-turn and delivery pipeline, document the trust boundary, and require redacted real Slack proof before merge.

## Reproduction Assessment

Yes for the blocking review finding: source inspection shows a pre-router hit returns from the PR's `dispatch.ts` branch before current main's `dispatchChannelInboundReply` path records session state and applies bot-loop/delivery hooks. I did not run the PR in Slack.

## Solution Assessment

No. The direct early-send branch is a plausible shortcut, but the maintainable solution is to integrate the hook with the existing Slack turn/delivery pipeline or settle an explicit plugin/API design first.
