## Summary

Keep this issue open for maintainer product and security design. Current main and v2026.6.5 have adjacent provenance, review guidance, source filtering, temporal decay, and untrusted recall framing, but they do not implement durable source-trust metadata or trust-aware retrieval policy across the shared memory paths; the related open issues cover prerequisites or narrower defenses rather than superseding this request.

## What This Changes

Add durable, source-derived trust metadata and policies for memory ingestion, storage, review, expiration, retrieval filtering, and influence.

## Best Possible Solution

Extend the canonical provenance model with non-forgeable, framework-assigned origin facts, then define a shared trust-policy contract that supported memory backends preserve through ingestion, promotion, compaction, and retrieval, using secure defaults and explicit operator-controlled exceptions.

## Reproduction Assessment

Yes at source level: current main clearly lacks durable trust fields and trust-aware retrieval in the shared result contract, builtin schema, and configuration. This is a requested capability gap rather than a regression in an established runtime contract.

## Solution Assessment

No, the proposed source-to-tier config is useful direction but is not yet the best complete design. Trust must be assigned by framework-owned provenance, and maintainers must define consistent cross-backend review, quarantine, expiration, filtering, and operator-override semantics before implementation.

## Risks / Open Questions

- Trust assignment would be forgeable if an agent or retrieved text can self-report its source or trust level instead of the framework deriving it from the ingestion and tool-call chain.
- A memory-core-only or Active-Memory-only implementation would produce inconsistent guarantees across builtin search, QMD, session memory, Memory Wiki, promotion, compaction, and dreaming flows.
- Default expiration, review gates, or downweighting could silently change existing recall behavior, so fresh-install and upgrade semantics need an explicit operator policy.
- Numeric influence weighting may imply model-control guarantees that retrieval scoring alone cannot reliably provide; filtering, quarantine, and approval semantics need clearer security definitions.
