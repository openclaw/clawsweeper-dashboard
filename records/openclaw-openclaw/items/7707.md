## Summary

Keep open: current main and the latest release still have adjacent source labels, session visibility filtering, and Active Memory untrusted framing, but not persistent memory trust metadata, trust-policy config, review/expiration gates, or trust-weighted retrieval for the default memory path.

## What This Changes

The issue requests durable memory trust tagging by source, with policy controls for expiration, review/isolation, and trust-weighted retrieval to reduce memory poisoning risk.

## Best Possible Solution

Keep this as a maintainer-led memory security design item: define one canonical provenance/trust data model, persist it through default and QMD memory paths, expose documented policy config only after the model is settled, and enforce review/expiration/filtering/weighting at retrieval without agent-forgeable trust assignment.

## Reproduction Assessment

Not applicable as a runnable bug reproduction, but the source-level gap is high-confidence: current memory contracts, schema, config, retrieval, docs, and tests expose source/corpus labels but no trust metadata or policy enforcement.

## Solution Assessment

Unclear as an implementation shape because the requested direction is valid but broad; the best path is a maintainer security/product design across memory-core, QMD, SDK/config contracts, and retrieval rather than an isolated config patch.
