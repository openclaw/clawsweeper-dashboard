## Summary

Keep open: current main still has coarse memory source labels and adjacent prompt/provenance defenses, but not durable memory trust metadata, trust-policy config, review/expiration gates, or trust-weighted retrieval; the live discussion and related open issues frame this as an active product/security design gap.

## What This Changes

Request source-based trust tags and policies for persistent memory so untrusted content can be reviewed, expired, filtered, or downweighted during retrieval.

## Best Possible Solution

Define a framework-owned durable memory trust/provenance contract and enforce it through ingestion, storage, compaction/dreaming, Memory Wiki bridging, SDK/config/docs, retrieval filtering or weighting, and review/expiration policy.

## Reproduction Assessment

Not applicable. This is a feature/security-hardening request, and current-main source inspection shows the requested trust tags, policy config, and retrieval gates are absent rather than an existing behavior failing.

## Solution Assessment

Yes as a product/security direction, but not as a narrow automated patch. The maintainable solution is one framework-owned cross-backend trust/provenance contract rather than an isolated metadata field, prompt-only wrapper, or agent-reported trust flag.
