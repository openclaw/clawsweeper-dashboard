## Summary

Keep open. Current main has adjacent safeguards around session visibility, Active Memory untrusted framing, Memory Wiki provenance, and the trusted-operator security model, but it still does not implement durable source trust metadata, trust-policy config, review/expiration gates, or trust-weighted retrieval for the default memory path. The existing discussion also keeps the gap live, with HMAKT99's AKF comment as useful prior art rather than shipped behavior.

## What This Changes

Add source-based trust tagging and policy enforcement for persistent memory so untrusted content can be reviewed, expired, filtered, or downweighted before recall.

## Best Possible Solution

Define a durable memory trust contract first, then implement it end to end: ingestion should classify provenance, storage and SDK types should persist structured trust metadata, retrieval should enforce filtering/downranking/review and expiration policy, compaction/dreaming should preserve provenance without laundering untrusted content, and docs/UI should expose the effective policy.

## Reproduction Assessment

Not applicable as a feature/security hardening request. The current gap is high-confidence from static inspection: the default memory schema, search result contract, config schema, and retrieval path have no trust metadata or trust-policy enforcement.

## Solution Assessment

Unclear for the exact proposed config, but the overall direction is sound. A config-only field would not be enough; the maintainable solution needs a typed contract carried through storage, retrieval, compaction, optional backends, and plugin SDK surfaces.
