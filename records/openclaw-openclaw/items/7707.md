## Summary

Keep open: current main still has coarse memory/session source labels, session visibility checks, Active Memory framing, and Memory Wiki provenance, but not durable memory trust metadata, trust-policy config, review/expiration gates, or trust-weighted retrieval. The related open issues are complementary provenance and channel-wrapping work rather than a complete replacement for this security design gap.

## What This Changes

Request source-derived trust metadata and policy enforcement for memory ingestion, persistence, retrieval weighting, review, and expiration.

## Best Possible Solution

Define a versioned, framework-assigned memory trust/provenance contract and enforce it consistently across ingestion, persistence, retrieval, compaction, SDK/config/docs, UI, and optional memory backends.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug; this is a security feature/design request, and source inspection verifies the requested trust metadata, trust config, review gates, expiration policy, and trust-weighted retrieval are absent on current main.

## Solution Assessment

Is this the best way to solve the issue? Yes for triage: keeping it open for maintainer product/security design is the right path, because the implementation needs framework-assigned trust and cross-backend enforcement rather than a self-tagged field in one storage path.
