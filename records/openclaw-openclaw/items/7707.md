## Summary

Keep this open: current main and the latest release still have adjacent source/provenance and untrusted-framing pieces, but not persistent memory trust metadata, trust-policy config, review/expiration gates, or trust-weighted retrieval.

## What This Changes

The issue requests source-based trust metadata and policy enforcement for persisted agent memory to reduce memory-poisoning risk.

## Best Possible Solution

Keep this as a maintainer-led memory security design item: define one durable trust/provenance data model, persist it through memory backends and SDK contracts, and enforce retrieval/review/expiration policy without JSON sidecars or runtime legacy shims.

## Reproduction Assessment

Yes for the source-level gap, not as a failing runtime bug: current main can be inspected to show memory results, schemas, config, and retrieval paths do not carry or enforce trust metadata.

## Solution Assessment

No single proposed config snippet is clearly the best full solution yet; the maintainable path needs a product/security design that coordinates source provenance, review state, policy modes, SQLite-backed persistence, and memory backend contracts.
