## Summary

Keep this open: current main still lacks durable memory trust metadata, trust-policy config, review/expiration gates, and trust-weighted retrieval, and the recent discussion refines live product/security requirements rather than making the request stale or implemented.

## What This Changes

The issue requests source-based trust metadata for memory entries, with policy controls for expiration, review, isolation, and trust-weighted retrieval to reduce memory-poisoning risk.

## Best Possible Solution

Define a framework-owned durable memory trust/provenance model, persist it through the canonical memory record path, expose a documented policy contract, and enforce retrieval-time filtering, weighting, review, and expiration consistently across default and optional memory backends.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: current memory contracts, schemas, config, tool execution, and builtin/QMD retrieval paths have no durable trust metadata or trust-policy enforcement.

## Solution Assessment

Is this the best way to solve the issue? No as a concrete implementation yet: source trust tagging is the right security direction, but the proposed config alone is insufficient until maintainers decide source authority, source/review axes, audit-only versus enforcement modes, decay semantics, and backend/API compatibility.
