## Summary

Current main and v2026.6.5 still lack a general durable trust-metadata and enforcement contract for memory. Existing source labels, provenance-rich specialized paths, Active Memory untrusted framing, and narrower related issues are useful adjacent defenses, but none supplies cross-backend trust assignment, persistence, review/expiry gates, or trust-aware retrieval; this issue remains necessary and needs maintainer product/security design.

## What This Changes

Add framework-derived source trust metadata to durable memories and enforce configurable review, retention, filtering, and retrieval-weighting policies.

## Best Possible Solution

Define a small framework-owned provenance record at ingestion, persist it through every durable transformation and backend, then add explicit operator-reviewed trust classification and backend-neutral filtering, review, and retention policy without allowing agents to self-assert trust.

## Solution Assessment

No, not exactly as proposed. Source tagging is the right direction, but the maintainable solution must establish non-forgeable framework provenance and canonical backend storage before adding trust scores, expiry defaults, or retrieval weights.

## Risks / Open Questions

- Trust assignment must be derived by the framework/tool chain or explicitly controlled by the operator; model-supplied tags could launder hostile content into a trusted class.
- The design affects storage migrations, builtin and QMD retrieval, Active Memory, session and dreaming provenance, plugin SDK contracts, and UI/review surfaces, so a one-backend implementation would create inconsistent security behavior.
- Default expiration, filtering, or downweighting could silently suppress valid operator-curated or long-lived research memory unless fresh-install and upgrade semantics are explicitly designed.
- The suggested per-memory sidecar approach conflicts with repository policy that OpenClaw-owned persistent state belongs in canonical SQLite-backed stores rather than parallel JSON files.
