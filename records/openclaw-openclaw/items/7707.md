## Summary

Keep open: the current memory stack still has coarse source labels and adjacent provenance/prompt-framing mitigations, but it does not implement durable trust metadata, trust-policy config, review/expiration gates, or trust-weighted retrieval. Recent comments add useful AKF/tool-chain provenance context and do not identify a merged replacement.

## What This Changes

The issue requests source-derived trust metadata for persistent memory, plus trust-policy config for review, expiration, isolation, and retrieval weighting.

## Best Possible Solution

Design a first-class cross-backend memory trust/provenance contract, then implement durable metadata and retrieval enforcement consistently across memory-core, QMD/session memory, Active Memory, Memory Wiki bridge, SDK/config, docs, UI, compaction/dreaming, and regression tests.

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug repro; this is a feature/security design request, and source inspection verifies the requested trust metadata, trust config, review gates, expiration policy, and trust-weighted retrieval are absent.

## Solution Assessment

Is this the best way to solve the issue? Keeping this open for maintainer/security design is the best current outcome; a narrow one-file metadata or config addition would not cover ingestion, persistence, compaction, retrieval, SDK, and UI enforcement.
