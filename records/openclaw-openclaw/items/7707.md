## Summary

Current main and v2026.6.5 still do not provide persistent source-derived trust metadata or trust-policy enforcement in the shared memory path. Recent short-term promotion provenance and Active Memory’s untrusted prompt framing are useful adjacent defenses, while the open provenance MVP is a prerequisite, but none covers framework-controlled trust assignment, audit/enforcement modes, review or expiry gates, or trust-aware retrieval; the active discussion also shows unresolved product and security choices rather than a safely delegable implementation.

## What This Changes

Add framework-assigned source trust metadata to durable memories and enforce configurable review, expiration, isolation, and retrieval-weight policies.

## Best Possible Solution

Define one framework-owned provenance contract with causal source kept separate from endorsement or review state, persist it through the shared memory plugin interface and canonical store, then add explicit audit-only and enforcement policies for filtering, weighting, review, expiry, compaction, and UI visibility across supported backends.

## Reproduction Assessment

Yes, at source level: current main’s shared search result, storage schema, and retrieval options have no persistent trust metadata or trust-policy controls. This demonstrates a feature gap rather than broken behavior under an existing contract.

## Solution Assessment

No, the sample trust-level config is useful design input but is not yet the best complete solution. The maintainable path first separates causal provenance from endorsement, assigns trust outside model control, and establishes one cross-backend contract before adding policy knobs.

## Risks / Open Questions

- The canonical provenance schema and trust-assignment authority are undecided; allowing the model or memory writer to self-assert trust would undermine the security boundary.
- The requested policy spans persisted schemas, shared plugin contracts, QMD and builtin retrieval, Active Memory, compaction/dreaming, and operator UI, so a backend-local implementation would create incompatible sources of truth.
- Enforcement defaults could add false-positive confirmation friction for operator-curated memory, while permissive defaults could leave mixed-source deployments exposed; audit-only and enforced modes need an explicit product decision.
- Any persisted data-model change needs a defined migration and upgrade path rather than runtime sidecars or parallel fallback readers.
