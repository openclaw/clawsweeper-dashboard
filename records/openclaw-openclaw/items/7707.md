## Summary

Current main and v2026.6.5 still lack a durable, non-forgeable memory trust contract. Existing corpus labels, citations, temporal decay, Active Memory untrusted framing, and narrower provenance/capture issues are useful adjacent defenses, but none persists trust classifications or enforces review, expiry, filtering, or trust-weighted retrieval across memory backends; this issue remains necessary and needs product/security design.

## What This Changes

Add source-derived trust metadata and trust-aware policy enforcement across memory ingestion, persistence, retrieval, expiration, review, and context use.

## Best Possible Solution

Define provenance and trust as layered canonical memory contracts: derive origin at framework-owned ingestion boundaries, persist it in the memory owner’s SQLite record and revision model, preserve it through promotion and compaction, and apply explicit operator-configured review, expiry, filtering, and weighting policies consistently across supported backends.

## Reproduction Assessment

Not applicable. This requests a new security and memory-policy capability; current-main source inspection confirms the requested durable trust fields and enforcement are absent rather than an existing documented contract failing.

## Solution Assessment

No, not exactly as the example config alone. Source-based policy is directionally correct, but the maintainable solution must first define a framework-assigned, non-forgeable provenance authority and one canonical persistence model before adding operator policy and retrieval enforcement.

## Risks / Open Questions

- Trust assignment must come from a non-forgeable framework or operator boundary; allowing the agent or retrieved content to self-declare trust would preserve the poisoning path.
- Provenance, trust classification, review state, and retrieval policy could become competing sources of truth unless they share one canonical SQLite-backed record and revision model.
- The request spans builtin memory, QMD, Active Memory, session and dreaming promotion, SDK contracts, configuration, migrations, and possibly UI, so an isolated backend implementation would leave inconsistent security behavior.
- Operator-curated memory needs an explicit trusted override without weakening the secure default for mixed or externally sourced content.
