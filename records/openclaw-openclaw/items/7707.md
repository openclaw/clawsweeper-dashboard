## Summary

Keep open: current main still has coarse memory source labels and adjacent mitigations, but not durable trust metadata, trust-policy config, review/expiration gates, or trust-weighted retrieval. The live discussion and related searches show this remains a product/security design gap rather than a duplicate of the narrower provenance and wrapping issues.

## What This Changes

The issue requests durable memory trust tagging by source, with trust-policy config, expiration/review gates, and retrieval-time filtering or weighting to reduce memory-poisoning risk.

## Best Possible Solution

Define a framework-owned durable memory trust/provenance contract and enforce it through ingestion, storage, compaction/dreaming, Memory Wiki bridging, SDK/config/docs, retrieval filtering or weighting, and review/expiration policy.

## Reproduction Assessment

Not applicable. This is a feature/security-hardening request, and current-main source inspection shows the requested trust tags, policy config, and retrieval gates are absent rather than an existing behavior failing.

## Solution Assessment

Yes as a product direction, but not as a narrow automated patch. The maintainable solution is one framework-assigned cross-backend trust/provenance contract rather than an isolated metadata field, prompt-only wrapper, or agent-reported trust flag.
