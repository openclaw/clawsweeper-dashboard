## Summary

Current main and v2026.6.5 still lack persistent source-derived trust metadata and enforcement; the open provenance MVP covers a necessary foundation but explicitly leaves trust policy, review, expiry, and retrieval influence as separate work requiring maintainer product/security design.

## What This Changes

Add framework-derived trust and provenance metadata to durable memories, with configurable review, expiration, isolation, and retrieval-weighting policies.

## Best Possible Solution

Define one framework-assigned provenance contract across the memory plugin slot, preserve origin and endorsement as separate fields, then add explicit audit-only and enforcement policies for filtering, review, expiry, and ranking across every supported backend.

## Reproduction Assessment

Yes at high confidence from source inspection: current main and v2026.6.5 persist corpus source and relevance data but not source-derived trust, review, expiry, or enforcement metadata. This is a capability-gap reproduction rather than a failing regression scenario.

## Solution Assessment

No, the proposed source-to-label config alone is not the best complete solution. Trust must be assigned from non-forgeable framework provenance, endorsement must remain orthogonal to origin, and maintainers must choose audit-only and enforcement semantics across memory backends.
