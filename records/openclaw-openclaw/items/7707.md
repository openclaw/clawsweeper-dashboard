## Summary

Keep open: current main still has only coarse memory/session source labels plus adjacent provenance/framing work, not durable memory trust metadata, trust-policy config, review/expiration gates, or trust-weighted retrieval. The newer discussion and related open issues are complementary design inputs, not a shipped fix or a clear duplicate.

## What This Changes

Request to tag durable memory entries by source trust, configure expiration/review/isolation policy, and use those trust signals during retrieval/context building.

## Best Possible Solution

Define a versioned, framework-assigned memory trust/provenance contract and enforce it consistently across ingestion, persistence, retrieval, compaction, SDK/config/docs, UI, and optional memory paths.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug; this is a feature/security design request, and current-main source inspection verifies the requested trust metadata, trust config, review gates, expiration policy, and trust-weighted retrieval are absent.

## Solution Assessment

Is this the best way to solve the issue? Yes: keeping it open for maintainer product/security design is the best current path, because a narrow metadata field or one-backend filter would not cover the cross-backend trust and persistence contract.
