## Summary

Keep open: current main still has coarse source labels and adjacent provenance/prompt-framing mitigations, but it does not implement durable memory trust metadata, trust-policy config, review/expiration gates, or trust-weighted retrieval. The related provenance and untrusted-content issues remain open, while prior trust/continuity PRs were closed unmerged rather than replacing this issue.

## What This Changes

Request source-derived trust tags for memory entries, plus expiration, review/isolation policy, and trust-weighted retrieval to reduce memory-poisoning risk.

## Best Possible Solution

Design a first-class cross-backend memory trust/provenance contract, then implement durable metadata and retrieval enforcement consistently across memory-core, QMD/session memory, Active Memory, Memory Wiki bridge, SDK/config, docs, UI, compaction/dreaming, and regression tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug; this is a feature/security design request, and source inspection verifies the requested trust metadata, trust config, review gates, expiration policy, and trust-weighted retrieval are absent.

## Solution Assessment

Is this the best way to solve the issue? Yes: keeping it open for maintainer/security design is the best current path, because a narrow one-file metadata or config change would not cover ingestion, persistence, compaction, retrieval, SDK, and UI enforcement.
