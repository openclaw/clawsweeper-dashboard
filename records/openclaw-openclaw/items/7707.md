## Summary

Keep this issue open: current main and v2026.6.10 still have adjacent source buckets, session visibility filtering, untrusted prompt framing, and provenance comments, but not durable memory trust tiers, review/expiration gates, or trust-weighted retrieval enforcement. Related open issues cover prerequisites or broader umbrellas, but none supersedes this specific memory trust-policy tracker.

## What This Changes

The issue requests source-based trust metadata, trust-policy configuration, expiration/review gates, and trust-weighted retrieval for persistent memory.

## Best Possible Solution

Define one maintainer-approved, host-authored durable memory trust contract and enforce it consistently across ingestion, storage, retrieval, review/expiration, UI/docs, and plugin SDK boundaries.

## Reproduction Assessment

Yes, at source level: current main can be checked by reading the memory result type, SQLite schema, config schema, tool schema, and retrieval paths, all of which lack durable trust metadata and trust-policy enforcement. This is a feature/security gap rather than a failing behavior reproduction.

## Solution Assessment

Unclear as an implementation plan: the requested direction is appropriate, but the best fix needs maintainer product/security design for trust authority, persistence, migration, plugin SDK compatibility, and retrieval policy before a code PR should be queued.
