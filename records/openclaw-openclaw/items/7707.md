## Summary

The central capability remains absent on current main, and the newer provenance issue deliberately treats trust policy as a separate layer rather than superseding it. This issue should stay open for maintainer product and security design because recent operational comments sharpen unresolved policy choices instead of removing the need.

## What This Changes

Requests persistent source-based trust metadata and policy enforcement for memory ingestion, storage, retrieval, review, expiration, and influence.

## Best Possible Solution

Define framework-derived provenance and trust in the canonical memory-plugin record contract, then offer explicit audit-only and enforcement policies that preserve trust through writes, merges, compaction, and every retrieval backend.

## Reproduction Assessment

Yes at the source-contract level: current main’s shared result, persistence, and retrieval paths have no durable trust metadata or trust-policy gates. This is a missing feature rather than broken documented behavior.

## Solution Assessment

Unclear in exact policy shape. Source tagging is the correct foundation, but the best implementation must make assignment framework-owned, separate provenance from policy, and enforce one contract across all memory backends.

## Risks / Open Questions

- If the agent or an untrusted tool can assign its own trust tier, poisoned content can relabel itself and defeat the boundary.
- Implementing trust only in one backend or recall path would create a misleading security guarantee across builtin, QMD, session, compaction, Active Memory, and optional memory-plugin paths.
- Defaults require product judgment: strict enforcement can add false-negative friction for operator-curated memory, while audit-only policy may be insufficient for mixed-source agents.
- The proposed config and persisted metadata are compatibility-sensitive and need a canonical schema plus migration strategy rather than sidecar state or parallel metadata paths.
