## Summary

Keep this open: current main and the latest release still have adjacent source labels, provenance docs, session visibility filtering, Active Memory untrusted framing, and Memory Wiki provenance, but not persistent memory trust metadata, trust-policy config, review/expiration gates, or trust-weighted retrieval. Recent discussion adds useful design constraints around framework-owned trust assignment, audit-only modes, reviewed-by metadata, and access/citation-based decay, so this remains a real product/security design item rather than cleanup-closeable stale work.

## What This Changes

Request to tag durable agent memory by source trust level and enforce trust-aware retention, human review, isolation, and retrieval weighting policies.

## Best Possible Solution

Keep this open for maintainer product/security design: define a framework-owned durable provenance and trust model, persist it through memory-core/QMD/session-memory/dreaming/wiki bridge surfaces, add explicit policy config only after the default security posture is chosen, and enforce filtering/downweighting/review/expiration at retrieval and promotion boundaries.

## Reproduction Assessment

Yes, at source level. The current memory contracts, SQLite schema, config schema, and retrieval paths expose source/corpus and relevance scoring but no persisted trust tier, review state, trust threshold, or trust-weighted retrieval gate.

## Solution Assessment

Unclear as an implementation sketch. The requested direction is valid, but the proposed config is not enough by itself because maintainers still need to choose source-authority, reviewed-by semantics, audit-only versus enforcement modes, migrations, and cross-backend SDK contracts.
