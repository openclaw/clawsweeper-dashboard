## Summary

Current main and v2026.6.5 still lack durable source-derived trust metadata and retrieval enforcement. The open provenance MVP at https://github.com/openclaw/openclaw/issues/42647 is a necessary foundation, but it explicitly does not decide trust assignment, review/endorsement semantics, audit-versus-enforcement modes, expiry, or trust-weighted retrieval; recent operator discussion also shows those policy choices remain active and materially different across deployment models.

## What This Changes

Add framework-assigned source trust metadata to durable memories and enforce configurable review, expiry, filtering, and retrieval-influence policies.

## Best Possible Solution

Define one canonical durable provenance model shared by builtin memory, QMD, session memory, Active Memory, Memory Wiki, and SDK consumers; keep causal origin, reviewer/endorsement, and policy outcome as separate fields, assign origin from framework-controlled context, and then add explicit operator-selected audit and enforcement modes with tested migration, expiry, review, filtering, and ranking behavior.

## Reproduction Assessment

Yes, at high confidence from current source: the public result type, SQLite schema, strict config schema, and tool input contract all lack the requested trust and enforcement fields. This establishes a capability gap rather than a regression in an existing documented contract.

## Solution Assessment

No, the original fixed source-to-trust mapping is not sufficient by itself. The maintainable design must separate causal provenance from endorsement and policy, prevent agent-controlled trust promotion, and share one contract across all memory backends before adding enforcement.

## Risks / Open Questions

- The request spans persisted data, shared SDK contracts, multiple memory backends, compaction/dreaming flows, config, and retrieval ranking; implementing one backend or ingestion route would create inconsistent trust semantics.
- Allowing the model or memory-writing tool to self-assert trust would let prompt-injected content promote itself, so source authority must come from framework-owned execution context or explicit operator review.
- A single hardcoded policy would mis-handle both mixed-source defensive deployments and operator-curated trusted-memory deployments described in the discussion; audit-only and enforcement behavior require an explicit product decision.
- Adding persistent metadata requires a canonical SQLite migration and backend compatibility plan; runtime sidecars or parallel fallback readers would conflict with repository storage policy.
