## Summary

Keep open: current main still has coarse memory/session source labels and adjacent provenance or untrusted-context framing, but not durable memory trust metadata, trust-policy config, review/expiration gates, or trust-weighted retrieval. The latest discussion adds useful design constraints around framework-side trust assignment rather than changing the implementation status.

## What This Changes

The issue requests memory entries tagged by source trust level, with configurable expiration, review isolation, and trust-weighted retrieval to reduce memory-poisoning risk.

## Best Possible Solution

Design a first-class memory trust/provenance contract, then implement durable metadata and retrieval enforcement across memory-core, QMD/session memory, Active Memory, Memory Wiki bridge, SDK/config/docs/UI, compaction/dreaming, and regression tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug; this is a feature/security design request, and current-main source inspection verifies the requested trust metadata, trust config, review gates, expiration policy, and trust-weighted retrieval are absent.

## Solution Assessment

Is this the best way to solve the issue? Yes: keeping it open for maintainer/security design is the best current path, because a narrow metadata field or one-backend filter would not cover ingestion provenance, persistence, retrieval, compaction, SDK contracts, config, and UI/docs.
