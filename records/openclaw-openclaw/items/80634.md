## Summary

Keep open: current main already fixes the Gateway `/models` and picker behavior, but the CLI `openclaw models list` path still treats `provider/*` as a configured model row, so the reported literal-wildcard output can still happen there.

## What This Changes

Wildcard entries in `agents.defaults.models` should expand to provider catalog rows and not appear as literal models in `/model`, `/models`, or `openclaw models list`.

## Best Possible Solution

Keep the current Gateway wildcard implementation, and make `openclaw models list` interpret `provider/*` through the same visibility contract so wildcard entries expand to real catalog rows and are never printed as literal models.

## Reproduction Assessment

Yes, by source inspection: `resolveConfiguredEntries` still parses `opencode-go/*` as a configured entry and `appendConfiguredRows` emits `entry.key` directly. I did not run the CLI because this was a read-only review.

## Solution Assessment

Yes, the narrow fix is to reuse the existing provider-wildcard visibility parsing in the CLI configured-list path, skip wildcard keys as literal configured rows, and add focused command regression coverage.
