## Summary

Keep open. Current main has the WebSocket keepalive ping from related reconnect work, but it still does not expose the requested per-connection `connected`, `rttMs`, or `lastHeartbeatAt` data on `health`, and open PR #74783 is already the linked implementation candidate.

## What This Changes

Request adding per-connection gateway reachability telemetry to the existing `health` RPC or to a new lightweight ping RPC.

## Best Possible Solution

Review and either land or revise #74783 so authenticated `health` responses include a per-client connection overlay derived from a single WebSocket ping/pong owner without mutating cached health snapshots.

## Reproduction Assessment

Yes for source-level verification: current main lacks the requested fields in the health type, handler, snapshot builder, method list, and docs. This is a feature request rather than a runtime bug reproduction.

## Solution Assessment

Unclear until #74783 is reviewed. The narrow maintainable direction appears to be an additive per-response `health.connection` overlay that reuses the authenticated WebSocket ping/pong path and keeps cached service health global.
