## Summary

Keep open. Current main still lacks the requested connection-level gateway health telemetry, and open PR #74783 is already linked as the implementation candidate, so this issue should remain open until that PR lands or maintainers choose a different protocol shape.

## What This Changes

Request connection-level gateway reachability telemetry on the existing health response, or a lightweight ping-style RPC, with `connected`, `rttMs`, and `lastHeartbeatAt` fields.

## Best Possible Solution

Review #74783 as the focused implementation candidate. The desired end state is an additive, documented per-client connection telemetry surface for authenticated gateway clients, with cached health snapshots kept global and connection data computed per requesting client.

## Reproduction Assessment

Yes for the missing behavior: current-main source and docs inspection show no `health.connection` fields and no `gateway.ping` method. This is a feature request rather than a runtime bug reproduction.

## Solution Assessment

Unclear until #74783 is reviewed, but the proposed per-response `health.connection` overlay appears to be the narrowest compatible direction because the authenticated health handler already has client context while the cached service snapshot can remain shared.
