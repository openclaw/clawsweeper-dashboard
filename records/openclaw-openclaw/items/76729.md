## Label Justifications

- `P1`: The issue reports user-visible channel/WebChat replies disappearing and later turns being combined after compaction, which is a broken agent/channel workflow for affected users.
- `impact:message-loss`: Assistant replies can disappear from the active transcript history shown to users after compaction rotation.
- `impact:session-state`: Dropping assistant turns leaves future model context with effectively unanswered consecutive user messages.

## Summary

Keep open: current main still builds the rotated successor transcript by removing summarized message entries without a role-aware preservation step, so the reported assistant-turn loss remains a valid source-level bug. The same-author implementation PR is closed unmerged, and the related open WebChat rendering issue describes intact transcript data rather than this compaction-rotation data loss path.

## What This Changes

The issue asks OpenClaw to preserve assistant replies when `truncateAfterCompaction` rotates an active transcript so channel/WebChat history does not collapse into consecutive user turns.

## Best Possible Solution

Patch successor transcript construction to preserve the assistant turn(s) required for surviving user-message context, while keeping summarized history compact and duplicate user pruning intact.

## Reproduction Assessment

Yes, this is source-reproducible: enable `truncateAfterCompaction`, create a transcript whose successor tail needs an assistant boundary before a surviving user turn, and the current removal set has no role-aware preservation step. I did not run a live channel replay in this read-only pass.

## Solution Assessment

Yes, a bounded fix in `buildSuccessorEntries` plus a regression test is the right shape. A channel-specific workaround would miss the shared transcript rotation path used by Feishu, Telegram, WebChat, and other embedded sessions.
