## Summary

Keep open: current main still contains the role-agnostic successor-rotation removal path described in the report, and existing coverage does not guard preserved assistant reply context across rotation.

## What This Changes

The issue asks OpenClaw to keep assistant replies visible and semantically intact when compaction with transcript rotation creates a successor session transcript.

## Best Possible Solution

Fix generic successor transcript rotation so the active rotated transcript preserves the assistant reply context needed by unsummarized user turns, with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes, at source level: construct a transcript with `user -> assistant -> user(firstKept) -> assistant -> compaction`, rotate it, and current `buildSuccessorEntries` removes the pre-`firstKeptEntryId` assistant because it removes every summarized message. I did not run a live Feishu or Control UI reproduction in this read-only review.

## Solution Assessment

Yes, the best fix belongs in the generic successor transcript construction or boundary helper, not in Feishu or WebChat. The safer path is a narrow regression test around rotated entries and session context before changing the retention rule.
