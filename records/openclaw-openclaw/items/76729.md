## Summary

Keep open: current main still has the role-agnostic successor-rotation removal path described in the report, and PR #76485 is open and unmerged with closing syntax for this issue.

## What This Changes

The issue reports that successor transcript rotation with `truncateAfterCompaction` drops assistant replies before surviving user turns, making WebChat/Control UI history show consecutive user messages.

## Best Possible Solution

Review and land PR #76485 if its generic retention rule and tests are correct, or replace it with a narrow successor transcript fix that preserves assistant reply context for surviving unsummarized user turns.

## Reproduction Assessment

Yes, at source level: a transcript whose `firstKeptEntryId` is a surviving user message will send earlier assistant `message` entries through the current role-agnostic `removedIds` path. I did not run a live Feishu, Telegram, or WebChat reproduction in this read-only review.

## Solution Assessment

Yes, the best fix belongs in generic successor transcript construction with focused regression coverage, not in Feishu, Telegram, or WebChat. Because an open PR already references this issue with closing syntax, the issue should stay open until that implementation is reviewed and merged or superseded.
