## Summary

Keep open: current main still has the role-agnostic successor transcript removal path, and this issue is now explicitly paired with open PR #76485 rather than something ClawSweeper should close ahead of merge.

## What This Changes

The issue asks OpenClaw to preserve assistant reply context when compaction rotates an active transcript with `truncateAfterCompaction: true`.

## Best Possible Solution

Fix the generic successor transcript builder with a focused regression test, then review whether PR #76485 should land as-is or be split from its separate queued-message compaction change.

## Reproduction Assessment

Yes. At source level, a transcript shaped as user -> assistant -> first kept user -> assistant -> compaction will mark the pre-boundary assistant message as removed because `buildSuccessorEntries` removes summarized messages by type only, not by conversational pairing.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in the generic successor transcript construction and its tests, not in Feishu or WebChat; the open PR is the current implementation candidate but needs normal review because it also covers #76467.
