## Summary

Keep open. Current main still lacks `.m2a` support across the shared media classifier, Telegram inbound handling, WebChat/UI maps, and memory multimodal audio lists; the related draft PR is closed and unmerged.

## What This Changes

Add `.m2a` to OpenClaw audio/media classification so Telegram and other media surfaces treat `.m2a` files like supported audio formats.

## Best Possible Solution

Add verified `.m2a` support through shared media helpers, then propagate it consistently through Telegram inbound document handling, WebChat/UI display, media understanding, memory multimodal selection, tests, and changelog.

## Reproduction Assessment

Yes. The source-level path is high-confidence: `isAudioFileName("voice.m2a")` follows a set that lacks `.m2a`, Telegram documents keep a document placeholder, and inbound buffer MIME detection does not use the original filename.

## Solution Assessment

Yes. A shared media-handling update is the narrowest maintainable solution; a Telegram-only patch would leave WebChat, UI normalization, memory multimodal indexing, and generic media helpers inconsistent.
