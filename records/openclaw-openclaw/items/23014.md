## Summary

Keep open: current main still has no `.m2a` support hits, the shared audio classifier and MIME map omit it, Telegram filename-based documents still start as `<media:document>`, and secondary WebChat/UI/memory audio maps also omit the format. The only related implementation at https://github.com/openclaw/openclaw/pull/23260 is closed unmerged and only covered part of the needed surface, while the existing maintainer review already scoped this as follow-up work.

## What This Changes

Recognize `.m2a` files as audio across Telegram and shared media handling so they follow the existing audio placeholder, MIME, transcription, UI, and memory paths.

## Best Possible Solution

Add a verified `.m2a` mapping to the shared MIME/audio helpers and align Telegram document reclassification, WebChat/UI, media-note, memory multimodal, and regression tests without adding a config knob or broad media-policy change.

## Reproduction Assessment

Yes, by source inspection: current main has no `.m2a` support hits, the shared audio extension set omits it, and Telegram filename-based documents still take the document placeholder path. I did not run a live Telegram upload in this read-only review.

## Solution Assessment

Yes. A focused shared-media plus Telegram/UI/memory update is the narrowest maintainable direction; the closed draft at https://github.com/openclaw/openclaw/pull/23260 is useful background but would leave several user paths uncovered.
