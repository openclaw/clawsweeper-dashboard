## Summary

Keep open: current main still lacks `.m2a` support in the shared media classifier, Telegram placeholder/preflight path, WebChat/UI audio maps, auto-reply/media-understanding checks, and memory multimodal audio list; the related implementation PR is closed unmerged and covered only part of the needed surface.

## What This Changes

Request support for `.m2a` audio files so Telegram and other media paths classify them as audio rather than documents.

## Best Possible Solution

Add `.m2a` consistently to the shared MIME/filename contract and every user-facing audio path, including Telegram document reclassification and focused regression coverage; no config knob is needed.

## Reproduction Assessment

Yes, by source inspection: current main has no `.m2a` support hits, `isAudioFileName()` is backed by an extension set without `.m2a`, and Telegram filename-based documents still take the document placeholder path. I did not run a live Telegram upload in this read-only review.

## Solution Assessment

Yes. A focused shared-media plus Telegram/UI/memory update is the narrowest maintainable solution; it should verify the `.m2a` MIME mapping and avoid broad media-policy or config changes.
