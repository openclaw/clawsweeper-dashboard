## Summary

Keep this issue open. Current main still has no `.m2a` support in the shared media classifiers, Telegram document placeholder path, inbound buffer MIME detection fallback, WebChat/UI audio maps, or memory multimodal audio extension list; the related draft PR #23260 is closed and unmerged, so it does not supersede the remaining work.

## What This Changes

The issue requests recognizing `.m2a` files as audio across Telegram inbound media handling and shared OpenClaw media classification.

## Best Possible Solution

Add `.m2a` support through the shared media classification path, then carry that classification through Telegram inbound document handling, WebChat/UI display, media understanding, memory multimodal selection, tests, and a user-facing changelog entry without creating a Telegram-only workaround.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: `isAudioFileName("voice.m2a")` follows a set that lacks `.m2a`, Telegram document messages keep `<media:document>`, and `saveMediaBuffer` does not use the original filename for MIME detection.

## Solution Assessment

Yes, a focused shared media-handling fix is the best path. A Telegram-only patch would leave WebChat, UI normalization, memory multimodal indexing, and generic media helpers inconsistent.
