## Summary

Keep this issue open. Current main still does not recognize `.m2a` as audio across the shared media helpers, Telegram document placeholder/content-type path, WebChat/UI MIME maps, or memory multimodal extension list, and the related PR #23260 was closed without merge.

## What This Changes

Request `.m2a` audio files to be classified and handled like existing audio formats, especially for Telegram inbound media and shared media processing.

## Best Possible Solution

Add `.m2a` support in the shared media MIME/filename contract and propagate that classification through Telegram inbound placeholders/preflight, WebChat/UI audio maps, memory multimodal indexing, targeted regression tests, and a user-facing changelog entry.
