## Summary

Current main still leaves the reported safety gap open: the generic/ping-pong/no-progress detectors exist, but the default remains opt-in and post-compaction retries have no dedicated loop guard. This should stay open for maintainer review rather than be swept closed.

## What This Changes

The issue asks OpenClaw to prevent repeated tool-call loops by enabling existing loop detection by default and/or adding post-compaction and disabled-detector diagnostics.

## Best Possible Solution

Land a maintainer-approved narrow safeguard that preserves progress-aware blocking semantics, updates every config/docs/schema surface, and keeps broader post-compaction diagnostics separate if they remain useful.

## Reproduction Assessment

Yes, source-reproducible for the core disabled-detector condition: current main defaults `tools.loopDetection.enabled` to false and tests assert repeated no-progress polling is ignored when disabled. I did not live-replay the reporter's private Telegram/vLLM incident.

## Solution Assessment

Unclear as a single package: the minimal default flip that preserves `generic_repeat` as warning-only looks like the safest code direction, but the global default, heartbeat-only scope, post-compaction guard, and disabled-detector diagnostics need maintainer choice before implementation.
