## Summary

Keep open: current main still defaults `tools.loopDetection.enabled` to `false`, still returns before detection when disabled, and does not include the proposed post-compaction guard or disabled-detector diagnostics; linked implementation PRs are open, so closing this issue would be premature.

## What This Changes

Request to make tool-loop protection effective by default and add post-compaction and diagnostic safeguards for repeated identical tool calls.

## Best Possible Solution

Review the linked PRs as separate changes: land a narrow default-policy update only if maintainers accept it, keep generic repeats warning-only, and treat the post-compaction guard and disabled diagnostics as independently documented/tested follow-ups.

## Reproduction Assessment

No high-confidence in-repo reproduction exists for the full Telegram/vLLM loop across compaction. Source inspection and existing tests do show the narrower current-main behavior: when `tools.loopDetection.enabled` is false, repeated no-progress calls are not blocked by the detector.

## Solution Assessment

Unclear as a complete plan: the minimal default flip is narrow if generic repeats stay warning-only, but changing the default and adding a post-compaction guard are product/reliability policy decisions rather than a mechanical bug fix.
