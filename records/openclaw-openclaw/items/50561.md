## Summary

Keep open. Current main and the latest release still keep gateway startup repair read-only/manual, while the existing doctor repair path is broad enough that unattended startup mutation needs an explicit safety policy before implementation.

## What This Changes

The issue requests automatic safe doctor fixes during gateway startup, optionally via a `doctor.autoFix` config option, with logs of any applied changes.

## Best Possible Solution

Define a narrow startup-safe repair contract first: explicit opt-in or allowlisting, protected-config semantics, backups, per-path audit logs, and no security, channel-availability, plugin-disablement, package-manager, or exec-policy mutations without maintainer approval.

## Reproduction Assessment

Not applicable as a bug reproduction because this asks for a new startup repair capability. Source, tests, docs, and the latest release show current behavior is intentionally manual-hint/read-only at startup while doctor repairs are gated behind explicit repair modes.

## Solution Assessment

No, auto-running the existing broad doctor repair sequence during ordinary startup is not the best solution. The maintainable path is a maintainer-defined startup-safe subset or explicit opt-in policy with protected values, backups, and visible audit logs.

Status: needs_attention

Summary: Unattended startup repair is security- and availability-sensitive because the existing doctor repair path can mutate plugin, channel, allowlist, package-install, OAuth-adjacent, and exec-policy state.
