## Summary

Keep open. Current main and the latest release still keep gateway startup read-only for config repair, while the broad doctor repair path is explicitly gated behind doctor repair modes; the discussion also shows why unattended startup repair needs a maintainer-defined safety policy first.

## What This Changes

Request automatic application of safe doctor fixes during gateway startup, or an opt-in `doctor.autoFix` setting with logs of applied changes.

## Best Possible Solution

Define a narrow startup-safe repair contract first: explicit opt-in or allowlisting, protected-config semantics, backups, per-path audit logs, and no security, channel-availability, plugin-disablement, package-manager, or exec-policy mutations unless maintainers approve them.

## Reproduction Assessment

Not applicable as a bug reproduction because this asks for a new startup repair capability. Source, tests, docs, and the latest release show current behavior is intentionally manual-hint/read-only at startup while doctor repairs are gated behind explicit repair modes.

## Solution Assessment

No, auto-running the existing broad doctor repair sequence during ordinary startup is not the best solution. The maintainable path is a maintainer-defined startup-safe subset or explicit opt-in policy with protected values, backups, and visible audit logs.

## Security Review

Status: needs_attention

Summary: Unattended startup repair is security- and availability-sensitive because the existing doctor repair path can mutate plugin, channel, allowlist, package-install, OAuth-adjacent, and exec-policy state.

Concerns:

- **[medium] Define startup-safe repair policy first:** `src/commands/doctor/repair-sequencing.ts:65`
  - body: `runDoctorRepairSequence` applies many mutation classes, including channel repair, plugin/package repair, invalid plugin cleanup, open-policy/allowlist repair, exec safe-bin repair, and OAuth cleanup; ordinary startup should not run this broad sequence without a narrower unattended-safe policy.
  - confidence: 0.92
- **[medium] Avoid unattended plugin disablement:** `src/commands/doctor/shared/invalid-plugin-config.ts:59`
  - body: Invalid plugin repair can remove plugin config payloads and set plugin entries to disabled, which may be correct during explicit repair but risky as an automatic startup mutation without protected-config semantics and operator-visible logs.
  - confidence: 0.9
- **[medium] Keep package repair out of default startup:** `src/commands/doctor/repair-sequencing.ts:73`
  - body: The repair sequence can invoke managed package/plugin recovery work, while docs state gateway startup and config reload do not run package managers.
  - confidence: 0.86
