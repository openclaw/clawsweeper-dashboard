## Summary

Keep open. Current main still keeps ordinary gateway startup read-only for config repair and gates the broad doctor repair sequence behind explicit doctor repair modes; the KeepMyClaw drift comment and the v2026.4.2 safety counterexamples are useful context that this needs a maintainer-defined startup-safe policy rather than cleanup closure.

## What This Changes

The issue requests auto-applying safe or non-destructive doctor fixes during gateway startup, optionally behind a `doctor.autoFix`-style setting.

## Best Possible Solution

Define a narrow startup-safe repair contract first: explicit opt-in or allowlisting, protected-config semantics, backups, per-path audit logs, and no security, channel-availability, plugin-disablement, or package-manager mutations unless maintainers explicitly approve them.

Not applicable as a bug reproduction because this asks for a new startup repair capability. The absence is high-confidence from source and docs: startup validates and hints at doctor repair while the repair sequence remains gated behind explicit doctor options.

## Solution Assessment

No, auto-running the existing broad doctor repair sequence during ordinary startup is not the best solution. The safer path is a maintainer-defined startup-safe subset or explicit opt-in policy with visible logs, backups, and protected values.

## Security Review

Status: needs_attention

Summary: Unattended startup repair is security- and availability-sensitive because the existing repair path can mutate plugin, channel, allowlist, package-install, OAuth, and exec-policy state.

Concerns:

- **[medium] Define startup-safe repair policy first:** `src/commands/doctor/repair-sequencing.ts:64`
  - body: `runDoctorRepairSequence` applies many mutation classes, including channel repair, package-manager plugin repair, invalid plugin quarantine, allowlist/open-policy repair, and exec safe-bin repair; ordinary startup should not run this broad sequence without a narrower unattended-safe policy.
  - confidence: 0.92
- **[medium] Avoid unattended plugin disablement:** `src/commands/doctor/shared/invalid-plugin-config.ts:59`
  - body: Invalid plugin repair deletes plugin config payloads and disables plugin entries, which can be appropriate during explicit repair but risky as a startup mutation without protected-config semantics and operator-visible audit logs.
  - confidence: 0.9
- **[medium] Keep package repair out of default startup:** `src/commands/doctor/repair-sequencing.ts:77`
  - body: The repair sequence can invoke managed npm peer-link and configured-plugin install recovery, while docs explicitly state Gateway startup and config reload do not run package managers.
  - confidence: 0.86
