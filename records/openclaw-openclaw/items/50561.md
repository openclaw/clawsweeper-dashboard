## Summary

Keep open. Current main and the latest release still do not auto-apply safe doctor repairs during ordinary Gateway startup, and the issue discussion shows broad unattended doctor writes need maintainer policy because they can affect persisted config, sandbox posture, and channel availability.

## What This Changes

The issue requests automatic application of safe doctor fixes during Gateway startup, or an opt-in `doctor.autoFix` setting, to prevent repeated startup warnings and config drift.

## Best Possible Solution

Define a maintainer-approved startup repair contract first: classify which doctor mutations are safe unattended, exclude security/sandbox/channel/provider/plugin changes unless explicitly opted in, log every redacted changed path with backup/recovery details, and keep broader repair behind manual `openclaw doctor --fix`.

## Reproduction Assessment

Yes by source inspection for the missing behavior: ordinary Gateway startup validates startup config and emits `openclaw doctor --fix` guidance instead of applying doctor repairs automatically. I did not run a live Gateway because this review was read-only.

## Solution Assessment

No, auto-running the whole existing `doctor --fix` during every startup is not the best solution. The maintainable path is a bounded safe-repair subset or explicit opt-in with audit logs and no-clobber boundaries for security, provider, and channel settings.
