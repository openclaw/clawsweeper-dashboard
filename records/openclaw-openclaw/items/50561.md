## Summary

Keep open. Current main still has no ordinary Gateway startup path that auto-applies safe doctor fixes; the ask is valid but needs a maintainer-defined unattended repair policy because the existing repair sequence can mutate plugin, channel, allowlist, and exec-related config.

## What This Changes

Request automatic safe doctor repair on Gateway startup, or an opt-in `doctor.autoFix` setting, so repeated startup repair hints do not leave config drift unresolved.

## Best Possible Solution

Define a narrow startup repair contract: allow only explicitly classified idempotent low-risk repairs, keep sandbox/auth/channel-disablement/plugin-install/exec-policy changes manual or clearly opt-in, log every changed config path with backup details, and document/test protected intentional config behavior.

## Reproduction Assessment

Yes for the current behavior, by source and regression-test inspection: startup reads and validates config, emits the manual doctor hint on invalid config, and does not call the repair sequence. This is not a strict bug reproduction because automatic startup repair is a new product capability.

## Solution Assessment

No, broad unattended execution of the existing doctor repair sequence is not the best way to solve it. The safer maintainable path is a startup-safe repair allowlist or explicit opt-in policy with visible logs and protected-config semantics.

## Security Review

Status: needs_attention

Summary: Unattended startup repair is security- and availability-sensitive because the current doctor repair path can mutate plugin, channel, allowlist, package-install, and exec-policy related configuration.

Concerns:

- **[medium] Define startup-safe repair policy first:** `src/commands/doctor/repair-sequencing.ts:62`
  - body: `runDoctorRepairSequence` spans channel repair, allowlist/open-policy repair, plugin install/config cleanup, and exec safe-bin profile repair. Gateway startup should not call this path until maintainers define an unattended-safe mutation allowlist, logging, backup, and opt-in policy.
  - confidence: 0.9
- **[medium] Avoid silent plugin or channel disablement:** `src/commands/doctor/shared/invalid-plugin-config.ts:59`
  - body: Invalid plugin repair removes the invalid plugin config payload and disables the plugin entry. That is appropriate for explicit repair, but risky as an unattended startup mutation without protected-config semantics and visible operator logs.
  - confidence: 0.86
- **[medium] Keep package-manager repair out of normal startup:** `src/commands/doctor/repair-sequencing.ts:76`
  - body: The repair sequence can repair missing configured plugin installs, while the docs explicitly keep plugin dependency installation out of Gateway startup/config reload/runtime inspection. Auto-running this during startup would broaden startup into package resolution.
  - confidence: 0.82
