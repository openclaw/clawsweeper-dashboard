## Summary

Keep open. Current main and v2026.5.20 still keep ordinary gateway startup read-only for doctor-style config repair, while the broad repair sequence remains gated behind explicit doctor repair modes; the issue discussion also contains concrete safety counterexamples, so this needs maintainer policy before implementation.

## What This Changes

The issue requests automatic safe doctor repair during gateway startup, or a config option to enable it, so repeated startup warnings do not leave configuration drift unresolved.

## Best Possible Solution

Define a narrow startup-safe repair contract first: explicit opt-in or allowlisting, protected-config semantics, backups, per-path audit logs, and no security, channel-availability, plugin-disablement, package-manager, OAuth, or exec-policy mutations unless maintainers approve them.

## Reproduction Assessment

Not applicable as a bug reproduction because this asks for a new startup repair capability. Source, tests, docs, and the latest release show current behavior is intentionally manual-hint/read-only at startup while doctor repairs are gated behind explicit repair modes.

## Solution Assessment

No, auto-running the existing broad doctor repair sequence during ordinary startup is not the best solution. The maintainable path is a maintainer-defined startup-safe subset or explicit opt-in policy with protected values, backups, and visible audit logs.
