## Summary

Keep open. Current main and v2026.6.10 still keep ordinary Gateway startup repair manual, and the issue discussion shows unattended broad doctor writes can change sandbox and channel availability, so this needs maintainer product/security policy rather than cleanup closure.

## What This Changes

Request to have Gateway startup automatically apply safe doctor repairs, or add an opt-in doctor auto-fix setting, so repeated doctor warnings do not leave persisted config drift.

## Best Possible Solution

Define a narrow startup repair contract first: classify unattended-safe doctor mutations, exclude security/sandbox/channel/provider/plugin/auth changes unless explicitly opted in, log every redacted changed path with backup details, and keep broad repair behind manual `openclaw doctor --fix`.

## Reproduction Assessment

Yes, by source inspection: Gateway startup validates the snapshot and emits doctor guidance instead of invoking the doctor repair sequence. I did not run a live Gateway because this review is read-only.

## Solution Assessment

No, auto-running the whole existing doctor repair sequence during every startup is not the best fix. The maintainable path is a bounded safe-repair subset or explicit opt-in with audit logs and no-clobber boundaries for security, provider, plugin, auth, and channel settings.
