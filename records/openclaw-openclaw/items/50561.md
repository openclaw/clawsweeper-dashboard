## Summary

Keep open. Current main still keeps ordinary Gateway startup on targeted recovery/degraded loading plus manual doctor hints, while the broad doctor repair path remains explicit; the discussion also contains credible safety counterexamples showing that unattended `doctor --fix` can silently change intentional config.

## What This Changes

Request auto-applying safe, non-destructive doctor repairs during Gateway startup or behind an opt-in setting, with logging of applied fixes.

## Best Possible Solution

Define a narrow startup repair contract first: allow only explicitly classified idempotent repairs, keep sandbox/auth/channel/plugin-disablement/package-manager/exec-policy changes manual or clearly opt-in, log every changed config path, and document protected intentional config behavior.

## Reproduction Assessment

Yes. A high-confidence static and test-backed path exists: Gateway startup loads `loadGatewayStartupConfigSnapshot`, the repair sequence is only called from explicit doctor repair mode, and the startup invalidity test asserts no config write plus a manual doctor warning.

## Solution Assessment

No, broad unattended execution of the existing `doctor --fix` sequence is not the best solution. The safer path is a startup-safe allowlist or explicit opt-in policy with visible logs and protected-config semantics.

## Security Review

Status: needs_attention

Summary: Unattended startup repair is security-sensitive because the current doctor repair path can mutate access, plugin, channel, package-install, and command-execution related configuration.

Concerns:

- **[medium] Define startup-safe repair policy first:** `src/commands/doctor/repair-sequencing.ts:54`
  - body: `runDoctorRepairSequence` covers channel repair, access allowlist/open-policy repair, plugin cleanup/quarantine, missing plugin install repair, and exec safe-bin repair. Gateway startup should not call this path until maintainers define an unattended-safe mutation allowlist, logging, backup, and opt-in policy.
  - confidence: 0.9
- **[medium] Avoid silent plugin or channel disablement:** `src/commands/doctor/shared/invalid-plugin-config.ts:59`
  - body: Invalid-plugin repair removes the invalid plugin config payload and sets the entry disabled. That is appropriate for explicit repair, but risky as an unattended startup mutation without protected-config semantics and visible operator logs.
  - confidence: 0.86
- **[medium] Keep package-manager repair out of normal startup:** `src/commands/doctor/shared/missing-configured-plugin-install.ts:117`
  - body: The repair sequence can reinstall configured downloadable plugins, while the docs state Gateway startup and config reload do not run package managers. Auto-running this during startup would broaden startup into package resolution and needs explicit policy approval.
  - confidence: 0.84
