## Summary

Keep open. Current main and the latest release still keep ordinary Gateway startup on validation/manual doctor guidance rather than an unattended safe-repair path, and the issue discussion includes credible safety counterexamples showing that broad startup `doctor --fix` needs maintainer policy before implementation.

## What This Changes

The issue requests auto-applying safe `doctor --fix` repairs during Gateway startup, or adding an opt-in `doctor.autoFix` style setting.

## Best Possible Solution

Define a narrow startup repair contract: only classified safe migrations may run unattended, security/channel/plugin-disabling changes stay manual or explicitly opted in, and every applied mutation is logged with backup/recovery details.

## Reproduction Assessment

Yes at source level. Current startup code validates and hints at doctor repair but does not run a safe auto-repair classifier or broad `doctor --fix` sequence on ordinary Gateway startup.

## Solution Assessment

No for the broad suggested shape. Full `doctor --fix` on Gateway start is too broad because current repair sequencing crosses channel, plugin, auth, and security policy surfaces; a narrower classified startup-repair contract is the safer direction.
