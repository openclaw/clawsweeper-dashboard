## Summary

Keep open. Current main and the latest release still leave ordinary Gateway startup on strict validation plus manual doctor guidance, while the only automatic doctor run I found is the dev `gateway:watch` post-exit retry; the discussion also includes credible examples where broad startup `doctor --fix` would need product and security policy before mutating config unattended.

## What This Changes

Request to auto-apply safe/non-destructive doctor repairs during Gateway startup, or add an opt-in `doctor.autoFix` style setting with logs of applied fixes.

## Best Possible Solution

Define a maintainer-approved startup repair contract that is bounded to proven safe migrations or explicit opt-in, logs a redacted before/after summary, avoids security/channel/provider clobbering, and has regression coverage for direct Gateway startup plus watched/supervised startup.

## Reproduction Assessment

Yes, source inspection reproduces the current behavior: direct Gateway startup throws invalid-config/manual-doctor guidance and the regression test pins no automatic recovery. I did not run a live Gateway because this review is read-only.

## Solution Assessment

No, auto-running the whole existing `doctor --fix` on every startup is not yet the best solution. The maintainable path is a bounded safe-repair contract, probably opt-in or narrowly defaulted, with audit logs and explicit no-clobber boundaries for security and channel settings.
