## Summary

Keep open. Current main and the latest release still keep ordinary gateway startup read-only/manual for doctor repairs, while the broad `doctor --fix` repair runner can mutate security, plugin, channel, package, and auth-adjacent state, so the remaining work needs maintainer policy before implementation.

## What This Changes

The issue asks OpenClaw to auto-apply safe doctor repairs during gateway startup, or expose an opt-in config option for that behavior.

## Best Possible Solution

Define a narrow startup-safe repair contract first: explicit opt-in or allowlisting, protected-config semantics, backups, per-path audit logs, and no security, channel-availability, plugin-disablement, package-manager, or exec-policy mutations without maintainer approval.

## Reproduction Assessment

Not applicable as a bug reproduction because this asks for a new startup repair capability. Source, tests, and the v2026.5.22 release show current behavior is manual-hint/read-only at startup while doctor repairs are gated behind explicit repair modes.

## Solution Assessment

No, auto-running the existing broad doctor repair sequence during ordinary startup is not the best solution. The maintainable path is a maintainer-defined startup-safe subset or explicit opt-in policy with protected values, backups, and visible audit logs.
