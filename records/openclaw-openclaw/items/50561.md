## Summary

Keep open. Current main still does not auto-apply ordinary doctor repairs during Gateway startup, and the issue now has credible safety context showing that broad startup `doctor --fix` can silently alter intentional config, so this needs maintainer policy before implementation.

## What This Changes

Request Gateway startup to auto-apply safe doctor repairs, or add an opt-in `doctor.autoFix`-style setting, so repeated startup doctor warnings do not leave config drift unresolved.

## Best Possible Solution

Keep broad repair manual until maintainers define a startup-safe repair contract with an explicit allowlist or opt-in setting, protected-config semantics, backups, and per-path audit logging.

## Reproduction Assessment

Yes for the current behavior: source and regression-test inspection show startup validates config, emits the manual doctor hint, and does not call the doctor repair sequence. This is not a strict bug reproduction because the requested auto-fix behavior is a new capability.

## Solution Assessment

No, auto-running the existing broad doctor repair sequence on startup is not the best fix. The safer path is a maintainer-defined startup-safe subset or explicit opt-in policy with visible logs and preserved intentional config.

## Security Review

Status: needs_attention

Summary: Unattended startup repair is security- and availability-sensitive because the current doctor repair path can mutate plugin, channel, allowlist, package-install, sandbox-adjacent, and exec-policy related state.

Concerns:

- **[medium] Define startup-safe repair policy first:** `src/commands/doctor/repair-sequencing.ts:62`
  - body: `runDoctorRepairSequence` applies multiple classes of mutation, including channel repair, allowlist/open-policy repair, plugin cleanup/install repair, invalid plugin quarantine, and exec safe-bin profile repair. Gateway startup should not run this whole sequence without a narrower unattended-safe policy.
  - confidence: 0.91
- **[medium] Avoid silent plugin or channel disablement:** `src/commands/doctor/shared/invalid-plugin-config.ts:59`
  - body: Invalid plugin repair deletes the plugin config payload and sets the plugin entry to disabled. That is acceptable in explicit repair mode, but risky as an unattended startup mutation without protected-config semantics and operator-visible logs.
  - confidence: 0.88
- **[medium] Keep package-manager repair out of ordinary startup:** `src/commands/doctor/repair-sequencing.ts:76`
  - body: The repair sequence can repair missing configured plugin installs, while current docs keep plugin dependency installation out of Gateway startup and config reload. Startup auto-repair must preserve that package-manager boundary unless maintainers explicitly change it.
  - confidence: 0.84
