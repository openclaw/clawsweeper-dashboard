## Summary

Keep open. Current main still does not auto-apply safe doctor repairs during ordinary Gateway startup: startup performs narrow recovery/degraded loading and emits manual doctor hints, while the broad doctor repair sequence remains gated behind explicit repair/yes modes. The discussion also contains useful safety context, including a KeepMyClaw drift-check angle and a v2026.4.2 report where a startup `doctor --fix` workaround silently changed intentional sandbox and Telegram plugin settings, so this needs maintainer policy before an automated fix PR is safe.

## What This Changes

Request to have Gateway startup automatically apply safe, non-destructive doctor repairs or expose a `doctor.autoFix` option, while logging what changed instead of repeatedly warning operators to run doctor manually.

## Best Possible Solution

Define a narrow startup repair contract first: allow only explicitly classified idempotent, low-risk repairs during Gateway startup, keep sandbox/auth/channel/plugin-disablement/exec-policy/package-manager changes manual or clearly opt-in, log every changed config path with backup/recovery details, and document/test protected intentional config behavior.

## Reproduction Assessment

Yes. A high-confidence static and test-backed path exists: `startGatewayServer` loads `loadGatewayStartupConfigSnapshot`, and the startup invalidity regression test expects no config write plus a manual doctor warning.

## Solution Assessment

No, broad unattended execution of the existing doctor repair sequence is not the best solution. The maintainable path is a startup-safe allowlist or explicit opt-in policy with protected-config semantics and visible logs.

## Security Review

Status: needs_attention

Summary: Unattended startup repair is security-sensitive because the current doctor repair path can mutate access, plugin, channel, package-install, and command-execution related configuration.

Concerns:

- **[medium] Define startup-safe repair policy first:** `src/commands/doctor/repair-sequencing.ts:54`
  - body: `runDoctorRepairSequence` spans channel repair, open-policy/allowlist repair, plugin cleanup/quarantine, missing configured plugin repair, and exec safe-bin profile repair. Gateway startup should not call this path until maintainers define an unattended-safe mutation allowlist, logging, backup, and opt-in policy.
  - confidence: 0.9
- **[medium] Avoid silent plugin or channel disablement:** `src/commands/doctor/shared/invalid-plugin-config.ts:59`
  - body: Invalid-plugin repair removes the invalid plugin config payload and sets the entry disabled. That is appropriate for explicit repair, but risky as an unattended startup mutation without protected-config semantics and visible operator logs.
  - confidence: 0.86
- **[medium] Keep package-manager repair out of normal startup:** `src/commands/doctor/repair-sequencing.ts:63`
  - body: The repair sequence can repair missing configured plugin installs, while the docs explicitly keep plugin installs out of Gateway startup/config reload. Auto-running this during startup would broaden startup into package resolution and should need explicit policy approval.
  - confidence: 0.82
