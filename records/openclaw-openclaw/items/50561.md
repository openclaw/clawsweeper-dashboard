## Summary

Keep open. Current main and v2026.5.22 still keep ordinary gateway startup manual/read-only for doctor repairs, while the existing repair sequence can mutate config, plugin/channel availability, package state, allowlists, exec policy, and OAuth-adjacent state, so this needs a maintainer-approved startup-safe repair policy before implementation.

## What This Changes

Request to auto-apply safe doctor fixes during gateway startup, or add a `doctor.autoFix` option that logs any applied repairs.

## Best Possible Solution

Define a narrow startup-safe repair contract first: explicit opt-in or allowlisting, protected-config semantics, backups, per-path audit logs, and no security, channel-availability, plugin-disablement, package-manager, or exec-policy mutations without maintainer approval.

## Reproduction Assessment

Not applicable as a bug reproduction because this asks for a new startup repair capability. Source, docs, tests, and the v2026.5.22 release show current behavior is manual-hint/read-only at startup while doctor repairs are gated behind explicit repair modes.

## Solution Assessment

No, auto-running the existing broad doctor repair sequence during ordinary startup is not the best solution. The maintainable path is a maintainer-defined startup-safe subset or explicit opt-in policy with protected values, backups, and visible audit logs.
