## Summary

Keep open. Current main and v2026.6.10 still keep ordinary Gateway startup doctor repair manual, while the issue discussion shows unattended broad doctor writes can affect sandbox and channel availability, so this needs maintainer product and security policy rather than cleanup closure.

## What This Changes

Request to automatically apply safe doctor repairs during Gateway startup, or add an opt-in `doctor.autoFix` setting with logs of applied changes.

## Best Possible Solution

Define a maintainer-approved startup repair contract first: classify safe unattended repairs, exclude security/sandbox/channel/provider/plugin/auth changes unless explicitly opted in, log every redacted changed path with backup/recovery details, and keep broad repair behind manual doctor.

## Reproduction Assessment

Yes, by source inspection: Gateway startup validates the snapshot and emits doctor guidance instead of invoking the doctor repair sequence. I did not run a live Gateway because this is a read-only review.

## Solution Assessment

No, auto-running the whole existing doctor repair sequence during every startup is not the best fix. The maintainable path is a bounded safe-repair subset or explicit opt-in with audit logs and no-clobber boundaries for security, provider, plugin, auth, and channel settings.
