## Summary

Keep open. Current main and the latest release still do not auto-apply safe doctor repairs during ordinary Gateway startup, and the issue discussion shows this cannot be treated as a narrow bug fix because unattended doctor writes can affect persisted config, sandbox posture, and channel availability.

## What This Changes

The issue requests automatic safe doctor repair during Gateway startup, or an opt-in `doctor.autoFix` setting, with logs for any applied changes.

## Best Possible Solution

Define a maintainer-approved startup repair contract that is limited to proven safe migrations or explicit opt-in, logs a redacted before/after summary, preserves security and channel settings unless intentionally changed, and leaves broader repairs behind manual `openclaw doctor --fix`.

## Reproduction Assessment

Yes for the current missing behavior by source inspection: ordinary Gateway startup validates and throws/manual-hints instead of running doctor repair, while only dev watch and update flows run doctor automatically. I did not run a live Gateway because this review is read-only.

## Solution Assessment

No, auto-running the whole existing `doctor --fix` during every startup is not the best solution. The maintainable path is a bounded safe-repair subset or explicit opt-in with audit logs and no-clobber boundaries for security, provider, and channel settings.
