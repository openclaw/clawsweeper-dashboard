## Summary

Keep open. Current main still keeps ordinary Gateway startup read-only for config repair: it validates, emits manual doctor guidance, and does not invoke the explicit doctor repair sequence; the discussion also contains credible safety counterexamples showing why unattended startup mutation needs maintainer policy design first.

## What This Changes

The issue requests auto-applying safe doctor fixes on Gateway startup, or an opt-in `doctor.autoFix` setting with logs of applied changes.

## Best Possible Solution

Define a narrow startup-safe repair contract with explicit opt-in or allowlisting, protected-config semantics, backups, and per-path audit logging while keeping broad repair behind manual `openclaw doctor --fix`.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a new startup repair capability request. The absence is high-confidence from source and tests: current startup validates and hints at manual doctor repair, while repair remains gated behind explicit doctor options.

## Solution Assessment

No, auto-running the existing broad repair sequence during ordinary startup is not the best solution. The maintainable path is a maintainer-defined startup-safe subset or explicit opt-in policy with visible logs, backups, and protected values.

## Security Review

Status: needs_attention

Summary: Unattended startup repair is security- and availability-sensitive because the existing repair path can mutate plugin, channel, allowlist, package-install, and exec-policy related state.

Concerns:

- **[medium] Define startup-safe repair policy first:** `src/commands/doctor/repair-sequencing.ts:62`
  - body: `runDoctorRepairSequence` applies many mutation classes, including channel repair, allowlist/open-policy repair, package-manager plugin repair, invalid plugin quarantine, and exec safe-bin repair; ordinary startup should not run this whole sequence without a narrower unattended-safe policy.
  - confidence: 0.92
- **[medium] Avoid unattended plugin disablement:** `src/commands/doctor/shared/invalid-plugin-config.ts:59`
  - body: Invalid plugin repair deletes the plugin config payload and disables the plugin entry. That can be appropriate during explicit repair, but it is risky as an unattended startup mutation without protected-config semantics and operator-visible audit logs.
  - confidence: 0.88
- **[medium] Keep package repair out of default startup:** `src/commands/doctor/repair-sequencing.ts:76`
  - body: The repair sequence can invoke configured plugin install recovery and managed npm peer-link repair; startup auto-repair needs an explicit policy boundary before running package-manager repair during normal Gateway boot.
  - confidence: 0.84
