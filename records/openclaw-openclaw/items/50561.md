## Summary

Keep open. Current main still keeps the broad doctor repair sequence behind explicit repair/yes modes while Gateway startup performs only targeted recovery/degraded loading plus manual doctor hints; the discussion also contains credible safety counterexamples from a startup `doctor --fix` workaround, and related PR #64684 is open, unmerged, and intentionally narrower than unattended startup repair.

## What This Changes

Request that Gateway startup automatically apply safe/non-destructive doctor fixes, or expose a `doctor.autoFix` option, while logging what changed.

## Best Possible Solution

Define a narrow startup repair contract first: allow only explicitly classified idempotent low-risk repairs during Gateway startup, keep sandbox/auth/channel availability/plugin disablement/exec policy/package-manager changes manual or opt-in, and log every changed config path with backup/recovery details.

## Reproduction Assessment

Yes. A high-confidence static and test-backed path exists: `startGatewayServer` calls `loadGatewayStartupConfigSnapshot`, and the plugin-local invalidity test expects no config write plus a manual doctor warning.

## Solution Assessment

No, broad unattended execution of the existing doctor repair sequence is not the best solution. The safer path is a startup-safe allowlist or explicit opt-in policy with protected-config semantics and visible logs.

## Security Review

Status: needs_attention

Summary: Unattended startup repair is security-sensitive because the existing doctor repair path can mutate access, plugin, channel, package-install, and command-execution related configuration.

Concerns:

- **[medium] Define startup-safe repair policy first:** `src/commands/doctor/repair-sequencing.ts:54`
  - body: `runDoctorRepairSequence` spans channel repair, open-policy/allowlist repair, plugin cleanup/quarantine, missing configured plugin repair, and exec safe-bin profile repair. Gateway startup should not call this path until maintainers define an unattended-safe mutation allowlist, logging, backup, and opt-in policy.
  - confidence: 0.9
- **[medium] Avoid silent plugin or channel disablement:** `src/commands/doctor/shared/invalid-plugin-config.ts:59`
  - body: Invalid-plugin repair removes the invalid plugin config payload and sets the entry disabled. That is appropriate for explicit repair, but risky as an unattended startup mutation without protected-config semantics and visible operator logs.
  - confidence: 0.86
- **[medium] Keep package-manager repair out of normal startup:** `src/commands/doctor/repair-sequencing.ts:63`
  - body: The repair sequence can repair missing configured plugin installs, while the docs explicitly keep plugin installs out of Gateway startup/config reload. Auto-running this during startup would broaden startup into package resolution and should need explicit policy approval.
  - confidence: 0.84
