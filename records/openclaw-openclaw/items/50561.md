## Summary

Keep open. Current main still does not apply doctor repairs during ordinary Gateway startup: startup uses narrow recovery/degraded loading plus manual doctor hints, while the broader repair sequence remains explicit-only. The discussion and related open PR context make this a valid but safety-sensitive product decision, not stale cleanup.

## What This Changes

The issue asks Gateway startup to auto-apply safe doctor fixes or expose a `doctor.autoFix` option, with logs for applied changes.

## Best Possible Solution

Define and document a narrow startup-safe repair contract first: allow only explicitly classified idempotent low-risk repairs, keep sandbox/auth/channel/plugin-disablement/package-manager changes manual or opt-in, log every changed config path, and preserve intentional protected config.

## Reproduction Assessment

Yes. A high-confidence static and test-backed path exists: `startGatewayServer` loads `loadGatewayStartupConfigSnapshot`, and the startup recovery tests assert degraded no-write behavior plus manual `openclaw doctor --fix` warnings.

## Solution Assessment

No. Auto-running the existing broad `doctor --fix` sequence on every startup is not the best safe solution; a maintainable fix needs a startup-safe allowlist or explicit opt-in policy with protected-config semantics and visible logs.

## Security Review

Status: needs_attention

Summary: Unattended startup repair is security-sensitive because the current doctor repair path can mutate plugin, channel, package-install, and execution-policy state.

Concerns:

- **[medium] Define startup-safe repair policy first:** `src/commands/doctor/repair-sequencing.ts:54`
  - body: `runDoctorRepairSequence` spans channel repair, open-policy/allowlist repair, plugin cleanup/quarantine, missing configured plugin repair, and exec safe-bin profile repair. Gateway startup should not call this path until maintainers define an unattended-safe mutation allowlist, logging, backup, and opt-in policy.
  - confidence: 0.9
- **[medium] Avoid silent plugin or channel disablement:** `src/commands/doctor/shared/invalid-plugin-config.ts:59`
  - body: Invalid-plugin repair removes the plugin config payload and disables the entry. That is appropriate for explicit repair, but risky as an unattended startup mutation without protected-config semantics and visible operator logs.
  - confidence: 0.86
- **[medium] Keep package repair out of ordinary startup:** `src/commands/doctor/shared/missing-configured-plugin-install.ts:151`
  - body: Missing configured plugin repair can update or install downloadable plugins and write install records, while the docs keep package-manager work out of Gateway startup and config reload. Auto-running this during startup would broaden the runtime boundary.
  - confidence: 0.84
