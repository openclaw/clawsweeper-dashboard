## Summary

Keep open. Current main and the latest release still do not auto-apply ordinary doctor repairs during Gateway startup; the only automatic doctor path found is a gateway-watch retry after a child exits nonzero, while startup itself still blocks invalid config with manual repair guidance. The discussion also shows this needs maintainer policy because unattended `doctor --fix` can change security- and channel-affecting config.

## What This Changes

Request to auto-apply safe/non-destructive `openclaw doctor --fix` repairs on Gateway start, or add an opt-in `doctor.autoFix` setting with logs of applied changes.

## Best Possible Solution

Define a maintainer-approved startup repair policy that keeps ordinary Gateway startup explicit by default, permits only a narrow audited safe-migration subset if approved, and excludes or opt-gates security, service, channel, plugin-install, and destructive config writes.

## Reproduction Assessment

Yes, at source level: current startup validation still rejects invalid config with manual `openclaw doctor --fix` guidance, and tests explicitly pin no automatic startup recovery. I did not run a live managed-service fixture in this read-only review.

## Solution Assessment

No, not as stated. Auto-applying all safe-looking doctor fixes on every startup is broader than the current repair contract; the maintainable solution needs a named safe subset, audit logging, security exclusions, and a maintainer decision on opt-in versus default behavior.
