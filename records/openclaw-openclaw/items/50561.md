## Summary

Keep open. Current main still does not auto-apply safe doctor repairs during ordinary Gateway startup; it only has a dev `gateway:watch` one-shot repair path after a watched child exits, and the issue discussion raises credible safety concerns around unattended config mutation.

## What This Changes

Request auto-applying safe, non-destructive `openclaw doctor --fix` repairs during Gateway startup, or adding an explicit auto-fix config option.

## Best Possible Solution

Define a narrow startup-safe doctor repair contract with explicit opt-in or allowlisting, protected-config semantics, backups, and per-path audit logging while keeping broad repair behind manual `openclaw doctor --fix`.

## Reproduction Assessment

Yes for the current behavior: source and tests show ordinary Gateway startup validates config, emits the manual doctor hint, and does not call the repair sequence. This is not a strict bug reproduction because the requested unattended repair is a new capability.

## Solution Assessment

No, auto-running the existing broad doctor repair sequence on startup is not the best solution. A safer design is a maintainer-defined startup-safe subset or explicit opt-in policy with visible logs, backups, and protected values.

## Security Review

Status: needs_attention

Summary: Unattended startup repair is security- and availability-sensitive because the current doctor repair path can mutate plugin, channel, allowlist, package-install, and exec-policy related state.

Concerns:

- **[medium] Define startup-safe repair policy first:** `src/commands/doctor/repair-sequencing.ts:63`
  - body: `runDoctorRepairSequence` applies many mutation classes, including channel repair, allowlist/open-policy repair, package-manager plugin repair, invalid plugin quarantine, and exec safe-bin repair; ordinary startup should not run this whole sequence without a narrower unattended-safe policy.
  - confidence: 0.92
- **[medium] Avoid unattended plugin disablement:** `src/commands/doctor/shared/invalid-plugin-config.ts:59`
  - body: Invalid plugin repair deletes a plugin config payload and sets the plugin entry disabled. That can be appropriate during explicit repair, but it is risky as an unattended startup mutation without protected-config semantics and operator-visible audit logs.
  - confidence: 0.88
- **[medium] Keep package repair out of ordinary startup by default:** `src/commands/doctor/repair-sequencing.ts:87`
  - body: The repair sequence can invoke configured plugin install recovery; startup auto-repair needs an explicit policy boundary before running dependency or package-manager repair during normal Gateway boot.
  - confidence: 0.84
