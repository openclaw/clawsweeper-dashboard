## Summary

Keep open. Current main still keeps ordinary Gateway startup separate from broad doctor repair: startup performs targeted recovery/degraded loading and manual repair hints, while `doctor --fix` repair remains gated behind explicit repair/yes modes. The issue discussion includes credible safety counterexamples from a startup `doctor --fix` workaround, so this needs maintainer policy before an automated fix PR is safe.

## What This Changes

The issue asks for Gateway startup to auto-apply safe, non-destructive doctor fixes or expose a `doctor.autoFix` option, while logging any applied changes.

## Best Possible Solution

Define a narrow startup repair contract first: allow only explicitly classified idempotent, low-risk repairs to run unattended; keep sandbox, auth, allowlist/open-policy, exec policy, channel availability, and plugin disablement changes manual or clearly opt-in; log every changed config path with backup/recovery details; and document/test protected intentional config behavior.

## Reproduction Assessment

Yes. A high-confidence static and test-backed path exists: `startGatewayServer` calls `loadGatewayStartupConfigSnapshot`, and the plugin-local startup invalidity regression test expects no config write plus a manual `openclaw doctor --fix` warning.

## Solution Assessment

No, not as broad startup execution of the existing doctor repair sequence. The maintainable direction is a startup-safe allowlist or explicit opt-in policy, not unattended application of all current `doctor --fix` mutations.
