## Summary

Keep open. Current main still does not auto-apply doctor repairs during ordinary Gateway startup, and the discussion adds credible safety evidence that broad unattended `doctor --fix` can alter intentional config, so this needs maintainer policy before implementation.

## What This Changes

The issue requests unattended safe doctor repair during Gateway startup to reduce repeated manual config-fix warnings.

## Best Possible Solution

Define a narrow startup-safe doctor repair contract with an explicit allowlist or opt-in setting, protected-config semantics, backups, and per-path audit logging while keeping broad repair behind manual `openclaw doctor --fix`.

## Reproduction Assessment

Yes for the current behavior: source and regression-test inspection show Gateway startup validates config, emits the manual doctor hint, and does not call the doctor repair sequence. This is not a strict bug reproduction because the requested unattended repair is a new capability.

## Solution Assessment

No, auto-running the existing broad doctor repair sequence on startup is not the best fix. The safer path is a maintainer-defined startup-safe subset or explicit opt-in policy with visible logs and preserved intentional config.

## Security Review

Status: needs_attention

Summary: Unattended startup repair is security- and availability-sensitive because the current doctor repair path can mutate plugin, channel, allowlist, package-install, and exec-policy related state.

Concerns:

- **[medium] Define startup-safe repair policy first:** `src/commands/doctor/repair-sequencing.ts:63`
  - body: `runDoctorRepairSequence` applies multiple classes of mutation, including channel repair, allowlist/open-policy repair, package-manager plugin install repair, invalid plugin quarantine, and exec safe-bin profile repair. Gateway startup should not run this whole sequence without a narrower unattended-safe policy.
  - confidence: 0.91
- **[medium] Avoid silent plugin or channel disablement:** `src/commands/doctor/shared/invalid-plugin-config.ts:59`
  - body: Invalid plugin repair deletes the plugin config payload and disables the plugin entry. That can be reasonable in explicit repair mode, but it is risky as an unattended startup mutation without protected-config semantics and operator-visible logs.
  - confidence: 0.88
- **[medium] Keep package-manager repair out of ordinary startup:** `src/commands/doctor/repair-sequencing.ts:87`
  - body: Doctor can repair missing configured plugin installs, which may invoke package-manager recovery. Startup auto-repair needs an explicit policy boundary before running dependency repair during ordinary Gateway startup.
  - confidence: 0.84
