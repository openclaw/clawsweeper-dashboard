## Summary

Keep open. Current main still keeps ordinary Gateway startup separate from the broad doctor repair sequence: startup performs targeted recovery/degraded loading and manual repair hints, while `doctor --fix` repairs remain explicitly gated. The issue discussion also includes credible counterexamples where a startup `doctor --fix` workaround silently changed intentional sandbox and Telegram settings, so this needs maintainer policy before an automated fix is safe.

## What This Changes

The issue requests automatic safe/non-destructive doctor repairs on Gateway startup, or an opt-in `doctor.autoFix` setting with logs for applied fixes.

## Best Possible Solution

Define a narrow startup repair contract first: only explicitly classified idempotent low-risk repairs should run unattended; sandbox, auth, allowlist/open-policy, exec policy, channel availability, and plugin disablement changes should remain manual or clearly opt-in; every changed config path should be logged with backup/recovery details and protected intentional config behavior should be documented and tested.

## Reproduction Assessment

Yes. A high-confidence static and test-backed path exists: `startGatewayServer` calls `loadGatewayStartupConfigSnapshot`, and the plugin-local startup invalidity regression test expects no config write plus a manual `openclaw doctor --fix` warning.

## Solution Assessment

No, not as broad startup execution of the existing doctor repair sequence. The safer direction is a startup-safe allowlist or explicit opt-in policy rather than applying all current `doctor --fix` mutations unattended.

## Security Review

Status: needs_attention

Summary: Unattended startup repair is security-sensitive because the existing doctor repair path can mutate access, plugin/channel availability, and command-execution related configuration.

Concerns:

- **[medium] Define startup-safe repair policy first:** `src/commands/doctor/repair-sequencing.ts:54`
  - body: `runDoctorRepairSequence` includes channel repair, open-policy/allowlist repair, plugin cleanup/quarantine, plugin installs, and exec safe-bin profile repair. Gateway startup should not call this broad path until maintainers define an unattended-safe mutation allowlist, logging, backup, and opt-in policy.
  - confidence: 0.9
- **[medium] Avoid silent plugin or channel disablement:** `src/commands/doctor/shared/invalid-plugin-config.ts:59`
  - body: Invalid-plugin repair deletes the invalid `config` payload and sets the plugin entry to disabled. That is appropriate for explicit repair, but risky as an unattended startup mutation without protected-config semantics and visible operator logs.
  - confidence: 0.86
- **[medium] Treat exec safe-bin repair as policy-sensitive:** `src/commands/doctor/shared/exec-safe-bins.ts:246`
  - body: Doctor repair can scaffold safe-bin profiles for configured command execution. Running that automatically at startup would mutate execution policy surfaces and should be excluded or explicitly approved by the startup-safe repair contract.
  - confidence: 0.78
