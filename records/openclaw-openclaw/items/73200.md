## Summary

Keep open. Current main still has the old lost-task audit severity policy, and the PR’s code change is not complete until the operator docs and changelog are aligned with the new warning-only behavior.

## What This Changes

The PR changes task audit so lost tasks always report warnings, tolerates sub-second `startedAt < createdAt` skew, and updates the colocated audit tests.

## Best Possible Solution

Land one coherent audit-policy update: either keep the existing expired-lost error contract, or intentionally make lost findings warning-only with docs and changelog updated in the same change.

## Reproduction Assessment

Yes. A direct `listTaskAuditFindings` input with a `lost` task whose `cleanupAfter` is expired or missing reproduces the severity change, and a raw task with `startedAt` slightly before `createdAt` exercises the timestamp-tolerance branch.

## Solution Assessment

No, not yet. The code change is narrow, but warning-only lost findings alter operator-visible audit semantics while the docs and changelog still describe the previous contract.
