## Summary

Keep open. The PR is a small, relevant task-audit change, but it is not ready as-is because the warning-only lost-task behavior would make the CLI/status output contradict the current user docs, and the user-facing behavior change is missing the required changelog entry.

## What This Changes

The PR changes task audit so lost terminal tasks always report `warn`, adds a 1-second tolerance for tiny `startedAt < createdAt` ordering races, and updates the colocated audit tests for the new severity/order and timestamp cases.

## Best Possible Solution

Land a complete, maintainer-approved audit policy: either update docs and changelog to make warning-only lost findings deliberate and consistent, or keep the existing lost warn/error split and limit this PR to the timestamp-race tolerance.

## Reproduction Assessment

Yes. A direct `listTaskAuditFindings` input with `status: "lost"` and expired or missing `cleanupAfter` exercises the current error path, and a raw task with `startedAt < createdAt` exercises the timestamp audit branch; registry-created records already have separate normalization coverage.

## Solution Assessment

No, not yet. The timestamp tolerance is narrow and plausible, but hard-coding all lost findings to `warn` is incomplete while the user-facing docs and changelog still describe the old operator contract.
