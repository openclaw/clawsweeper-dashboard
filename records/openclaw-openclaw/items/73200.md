## Summary

Keep open. The code change is narrow, but merging it as-is would leave operator-facing task audit docs and the changelog inconsistent with the new warning-only lost-task policy.

## What This Changes

The PR changes task audit logic so all lost task findings are warnings, adds a one-second tolerance for tiny `startedAt < createdAt` races, and updates colocated audit tests.

## Best Possible Solution

Land one coherent audit-policy change: either document and ship warning-only lost findings with a changelog entry, or keep the existing expired/unstamped lost-task errors and limit this PR to timestamp-race tolerance.

## Reproduction Assessment

Yes. Direct `listTaskAuditFindings` inputs reproduce both changed behaviors: an expired lost task changes from `error` to `warn`, and a task with `startedAt` one millisecond before `createdAt` is suppressed by the new tolerance.

## Solution Assessment

No, not yet. The implementation is small, but the docs and changelog must be reconciled with the new operator-visible severity contract before this is the maintainable fix.
