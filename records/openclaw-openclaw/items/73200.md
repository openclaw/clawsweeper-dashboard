## Summary

Keep open. The PR makes a narrow task-audit behavior change, but it still changes operator-visible severity semantics while the current docs and changelog continue to describe the previous expired-lost error contract.

## What This Changes

The PR changes task audit so all `lost` findings report `warn`, adds a one-second tolerance for `startedAt < createdAt` audit races, and updates the colocated audit tests.

## Best Possible Solution

Land one coherent audit-policy update: either preserve expired-lost errors, or intentionally make lost findings warning-only with docs and changelog updated in the same PR.

## Reproduction Assessment

Yes. A direct `listTaskAuditFindings` input with a `lost` task whose `cleanupAfter` is expired or missing reproduces the severity change, and a raw task with slight `startedAt < createdAt` skew exercises the tolerance branch.

## Solution Assessment

No, not yet. The code change is narrow, but warning-only lost findings are not the best complete fix until operator docs and the active changelog describe the same contract; preserving the current warn/error split is the safer alternative if docs should remain true.
