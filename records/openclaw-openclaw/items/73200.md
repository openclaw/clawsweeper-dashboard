## Summary

Keep open. The PR targets a real task-audit noise path, but the warning-only lost-task severity policy is incomplete as proposed because current docs and changelog still describe the existing warn/error operator contract.

## What This Changes

The PR branch changes task audit so lost tasks always report as warnings, adds a 1-second tolerance for small `startedAt < createdAt` races, and updates the colocated audit tests for the new severity and ordering expectations.

## Best Possible Solution

Land a complete audit-policy change: either update docs and changelog so warning-only lost findings are deliberate and user-visible, or preserve the existing lost warn/error split and keep only the timestamp-race tolerance.

## Reproduction Assessment

Yes. A direct `listTaskAuditFindings` input with `status: "lost"` and expired or missing `cleanupAfter` exercises the current error path, and a raw task with `startedAt < createdAt` exercises the timestamp audit branch.

## Solution Assessment

No, not yet. The timestamp tolerance is narrow and plausible, but changing all lost findings to `warn` is not complete while user-facing docs and changelog still describe the old operator contract.
