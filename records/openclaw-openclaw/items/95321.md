## Summary

Keep this PR open for maintainer decision: it is a focused attempt to address the linked WebChat history-cap issue, but it adds a new persisted Gateway config key after recent current-main docs and code moved WebChat toward bounded history plus an on-demand full-message reader, and the advertised config key is not registered in the strict runtime schema.

## What This Changes

The PR adds `gateway.chatHistoryTextMaxChars` to the Gateway config type, reads it in chat-history projection when no per-request override is provided, wires the HTTP session-history endpoint through that resolver, and adds focused resolver tests.

## Best Possible Solution

Maintainers should first choose the product contract: either keep bounded history plus `chat.message.get`, or sponsor a bounded config knob with schema/help/docs, validation tests, upgrade expectations, and real WebChat reload proof.

## Reproduction Assessment

Yes. Source inspection shows current main still uses the 8,000-character history projection default unless a per-request `maxChars` is supplied, and the linked issue gives concrete long-reply reload steps.

## Solution Assessment

No as submitted. The resolver change is plausible, but the PR is missing the strict config contract and needs maintainer confirmation that a persisted history-payload key is preferred over the existing bounded-history plus full-message-reader design.
