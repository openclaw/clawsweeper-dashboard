## Summary

Keep open, but this PR is not merge-ready: the proposed user-facing config key is read at runtime but is not registered in the strict config schema, and the new config surface also needs maintainer direction because current WebChat history uses bounded previews plus the `chat.message.get` full-message reader.

## What This Changes

The PR adds a `gateway.chatHistoryTextMaxChars` setting, reads it in chat history projection when no RPC override is supplied, wires the HTTP history endpoint to that resolver, and adds resolver unit tests.

## Best Possible Solution

Either keep the current bounded history plus `chat.message.get` path, or land a maintainer-sponsored config knob only after schema, help, docs, validation tests, bounds, and real WebChat proof are aligned.

## Reproduction Assessment

Yes, from source inspection: history projection is bounded by `DEFAULT_CHAT_HISTORY_TEXT_MAX_CHARS`, while Control UI now relies on `chat.message.get` for full content. I did not run a live WebChat repro in this read-only review.

## Solution Assessment

No as submitted. The runtime resolver change is plausible, but the branch misses the strict config contract and needs maintainer confirmation that a new history-payload config key is preferred over the existing full-message reader path.
