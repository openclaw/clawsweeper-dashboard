## Summary

Keep open: current main still has the reported mismatch between restart timeout/drain behavior and marker-based main-session recovery. The ignored-SIGUSR1 in-flight token branch appears covered on main, but active sessions can still be left unmarked for immediate continuation after a forced restart.

## What This Changes

Ensure gateway restart timeout and drain-timeout paths mark affected active main sessions as interrupted/resumable before recovery runs.

## Best Possible Solution

Extend the existing restart-aborted main-session recovery contract so forced restart timeout and drain-timeout paths durably mark known affected main sessions before shutdown, then have startup recovery consume that marker once.

## Reproduction Assessment

Yes, source-reproducible. A persisted main session with status `running` and `abortedLastRun` false or unset is skipped by `recoverRestartAbortedMainSessions`, while current timeout/drain paths can proceed without setting that marker.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing `abortedLastRun` restart-recovery contract by marking affected sessions before forced restart or drain-timeout shutdown; broad manifest replay can remain in the larger recovery issues.
