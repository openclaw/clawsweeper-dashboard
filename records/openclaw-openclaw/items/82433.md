## Summary

Keep open. Current main fixes the secondary unauthorized SIGUSR1 in-flight token path, but the central timeout/drain-timeout path still can cross the restart boundary without durably marking affected main sessions for the existing restart-aborted recovery selector.

## What This Changes

The issue requests durable restart-interruption marking so active main sessions interrupted by gateway restart timeout or drain timeout are eligible for continuation or explicit recovery handling.

## Best Possible Solution

Extend the existing restart-aborted main-session marker contract so forced restart timeout and drain-timeout code marks known affected main sessions before shutdown, then let startup recovery consume that marker according to the current recovery policy.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source: a persisted main session with `status: "running"` and no `abortedLastRun: true` is skipped by `recoverRestartAbortedMainSessions`, while timeout/drain paths still do not write that marker before restart.

## Solution Assessment

Is this the best way to solve the issue? Yes for the narrow bug: reuse the existing `abortedLastRun` recovery contract and add missing restart-time marking, rather than adding a new config surface or broad manifest system in this issue.
