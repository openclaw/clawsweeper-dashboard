## Summary

Keep open: current main has marker-based main-session recovery, but the timeout/drain paths can still force restart while active work remains without marking affected main sessions for that selector. The secondary ignored-SIGUSR1 in-flight token path appears fixed on main.

## What This Changes

The issue asks for forced gateway restart timeout and drain-timeout paths to mark interrupted active main sessions as resumable and to clear failed SIGUSR1 restart state.

## Best Possible Solution

Extend the existing restart-aborted main-session recovery contract so forced restart timeout and drain-timeout paths durably mark known affected main sessions before shutdown, then let startup recovery consume that marker.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a persisted main session with status `running` and `abortedLastRun` unset or false is skipped by `recoverRestartAbortedMainSessions`, while current timeout/drain paths can proceed without setting that marker.

## Solution Assessment

Yes. The best narrow repair is to reuse the existing `abortedLastRun` recovery contract by marking affected main sessions before forced restart or drain-timeout shutdown; the already-fixed SIGUSR1 token branch should stay covered by regression tests.
