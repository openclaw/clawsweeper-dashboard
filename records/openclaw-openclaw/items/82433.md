## Summary

Keep open: current main still has a source-visible gap where restart deferral/drain timeouts can proceed across the restart boundary without durably marking the affected main sessions for existing restart recovery. The secondary unauthorized SIGUSR1 token path appears fixed on main, and the broader restart-recovery/continuation issues remain adjacent rather than superseding this narrower bug.

## What This Changes

The issue asks the gateway to mark active main sessions as interrupted/resumable when restart deferral or drain timeouts force a restart, and to avoid stale in-flight restart tokens after an ignored SIGUSR1.

## Best Possible Solution

Add an internal restart-interruption marking path that snapshots known-active main session keys before forced restart/drain-timeout shutdown, sets the existing resumable marker, and lets current main-session recovery consume it without adding new config or broad manifest policy.

## Reproduction Assessment

Yes, source-reproducible: force the deferral or drain timeout with an active main-session row that remains `running` and `abortedLastRun: false`, then current recovery will skip it at the `abortedLastRun === true` guard. I did not run a live systemd/Telegram restart in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable fix is to mark actual active main sessions before the forced restart boundary and reuse the existing restart recovery marker. The broader manifest/replay/readiness-gate design belongs with https://github.com/openclaw/openclaw/issues/57425, and the already-fixed unauthorized SIGUSR1 token path should be preserved rather than reworked.
