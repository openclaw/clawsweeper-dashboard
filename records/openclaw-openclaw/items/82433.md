## Summary

Keep this open. Current main has a real recovery contract for restart-aborted main sessions, and the SIGUSR1 ignored-listener token path appears to have been fixed, but the central timeout path still can force a restart without durably marking active main sessions for that recovery selector.

## What This Changes

The issue asks OpenClaw to mark active main sessions as interrupted before forced gateway restart/drain timeout so restart recovery can continue affected work.

## Best Possible Solution

Extend the existing restart-aborted main-session recovery contract so forced restart timeout/drain-timeout code marks known affected main sessions before shutdown and recovery consumes those markers after startup.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a credible reproduction path: leave a main session `running` without `abortedLastRun: true`, then `recoverRestartAbortedMainSessions` skips it; I did not run the live timed restart scenario.

## Solution Assessment

Is this the best way to solve the issue? Yes for the remaining bug: reuse the existing marker-based recovery path and add the missing restart-time marking rather than adding a new config surface or broad checkpointing system.

## Repair Work Prompt

Fix the gateway restart timeout recovery bug reported at https://github.com/openclaw/openclaw/issues/82433. Current main only resumes main sessions whose persisted entry is `status: "running"` and `abortedLastRun: true`; forced config-reload restart timeout and restart drain timeout can still proceed while active work remains without marking affected main sessions for that recovery path. Keep the change narrow: before a forced restart timeout or drain-timeout restart crosses the process boundary, durably mark known affected main sessions as interrupted using the existing `abortedLastRun` recovery contract or a backward-compatible marker consumed by `recoverRestartAbortedMainSessions`; preserve the current unauthorized SIGUSR1 token-clearing behavior. Do not add a new config option or broad checkpointing system. Add focused regression coverage for timeout/drain marking and recovery selection.
