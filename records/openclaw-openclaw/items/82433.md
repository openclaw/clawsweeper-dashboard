## Summary

Keep this open. Current main still has the reported mismatch: restart recovery only resumes main sessions already marked `abortedLastRun: true`, while the restart timeout and drain-timeout paths can proceed without writing that marker; the secondary SIGUSR1 in-flight token path appears fixed.

## What This Changes

Ensure gateway restart timeout/drain-timeout paths durably mark active main sessions as interrupted/resumable before startup recovery runs.

## Best Possible Solution

Extend the existing restart-aborted main-session recovery contract so forced restart timeout and drain-timeout paths mark affected active main sessions before shutdown, then recovery consumes and clears those markers after startup.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for a source-level reproduction: a `running` main session without `abortedLastRun: true` is skipped by recovery, and the timeout paths inspected on current main can cross restart boundaries without setting that marker.

## Solution Assessment

Is this the best way to solve the issue? Yes: reusing the existing marker-based recovery path is narrower than adding a new restart manifest or config surface, while the already-fixed SIGUSR token behavior should be left intact.
