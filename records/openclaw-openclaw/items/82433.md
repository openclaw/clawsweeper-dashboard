## Summary

Keep this open. Current main still has the core mismatch: main-session restart recovery only considers `running` sessions with `abortedLastRun: true`, while the config-reload timeout and restart drain-timeout paths can still force or proceed with a restart without marking affected main sessions; the secondary SIGUSR1 in-flight token issue appears addressed.

## What This Changes

The issue requests durable restart-interruption marking for active main sessions when gateway restart deferral or drain times out.

## Best Possible Solution

Extend the existing restart-aborted main-session recovery contract so forced reload timeouts and restart drain timeouts mark affected active main sessions before shutdown, then let the existing recovery path consume and clear those markers.

## Reproduction Assessment

Yes, at source level: a `running` main session without `abortedLastRun: true` is skipped by recovery, and the inspected timeout paths can proceed to restart without setting that marker. I did not run a live Linux/systemd restart reproduction in this read-only pass.

## Solution Assessment

Yes: reusing the existing marker-based recovery path is the narrowest maintainable fix, while the already-fixed SIGUSR1 token handling should be left intact. A broader pending-inbound store or restart manifest is not required for this specific bug.
