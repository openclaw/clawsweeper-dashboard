## Summary

Keep open: the underlying Windows memory atomic-reindex bug is still present on current main, and this PR is a plausible narrow repair, but the latest PR head is not mergeable because it leaves an invalid `thror err;` in the rollback path and still lacks the required changelog/test coverage.

## What This Changes

The PR adds a memory-core `renameWithRetry` helper for atomic reindex file moves, retrying `EBUSY` renames and falling back to `copyFile` plus `unlink` for `EPERM` or `EEXIST`.

## Best Possible Solution

Revise this PR or replace it with a narrow memory-core patch that restores the rollback `throw`, preserves sidecar and UUID behavior, ports the retry/fallback pattern with focused regression tests, and adds the required user-facing changelog entry; any broader gateway-coordinated reindex work should stay separate unless validation proves the narrow fix is insufficient.

## Reproduction Assessment

Yes. The PR defect reproduces by static inspection of the latest branch file: line 55 contains invalid TypeScript. The underlying Windows bug also has concrete commands and EBUSY logs in #64187, while current main still direct-renames the SQLite files in the active safe-reindex path.

## Solution Assessment

No as submitted. The retry/fallback direction is a plausible narrow solution, but the current patch cannot compile and needs memory-specific regression tests, changelog coverage, and Windows-oriented validation before it is the best maintainable fix.
