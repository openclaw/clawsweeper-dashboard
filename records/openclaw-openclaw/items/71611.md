## Summary

Keep this PR open. Current main still uses direct `fs.rename()` in the memory atomic-reindex swap path, so the linked Windows EBUSY problem is not implemented on main; however, the latest PR head still has a compile-blocking rollback typo and lacks the required changelog and focused memory-core retry/fallback tests.

## What This Changes

The PR adds a memory-core `renameWithRetry` helper for atomic reindex file moves, retrying `EBUSY` renames and falling back to `copyFile` plus `unlink` on Windows-style `EPERM`/`EEXIST` failures.

## Best Possible Solution

Repair this PR or replace it with a narrow memory-core patch that preserves sidecar and UUID behavior, restores the rollback `throw`, ports the cron retry/fallback pattern with focused regression tests, and adds the required changelog entry; keep broader gateway coordination and unsafe-reindex fixes separate unless validation proves this patch cannot solve the reported lock path.

## Reproduction Assessment

Yes. The PR defect reproduces by static inspection of the latest branch diff because `thror err;` cannot compile, and the underlying Windows report has concrete EBUSY logs while current main still direct-renames the live memory SQLite files.

## Solution Assessment

No as submitted. The retry/fallback direction is a plausible narrow fix based on the existing cron-store pattern, but the branch must compile, prove memory-core retry/fallback behavior with tests, and document the user-facing fix before it is mergeable.
