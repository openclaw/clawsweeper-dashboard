## Summary

Keep open. Current main still fails the central Claude ACP `sessions_spawn` case because `timeoutSeconds` becomes a backend config control; newer capability checks may fail before the delegate instead of inside Claude ACP, but the spawn still does not succeed with the existing timeout option.

## What This Changes

Fix ACP Claude handling so `sessions_spawn` `timeoutSeconds` remains an OpenClaw-side timeout instead of being sent as an unsupported ACP config option.

## Best Possible Solution

Preserve `timeoutSeconds` as the OpenClaw-side run timeout, but suppress backend timeout config forwarding for Claude and for backends whose advertised config keys do not include a timeout alias.

## Reproduction Assessment

Yes. The source path is clear from `sessions_spawn` through ACP runtime options to config-control application, and the current Claude ACP package rejects unknown config option ids; I did not run a live macOS Claude ACP spawn.

## Solution Assessment

Yes. The narrow maintainable fix is to skip unsupported timeout config controls while keeping OpenClaw's in-process run timeout and preserving timeout forwarding only for backends that advertise it.
