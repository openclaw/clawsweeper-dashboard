## Summary

Keep open. Current main still forwards ACP `timeoutSeconds` as a `session/set_config_option` timeout control for Claude, while the packaged Claude ACP backend does not advertise or accept a timeout config option.

## What This Changes

Fix ACP `sessions_spawn` so `timeoutSeconds` does not make the Claude ACP backend reject session startup or turn execution.

## Best Possible Solution

Keep `timeoutSeconds` as an OpenClaw-side turn timeout, but stop sending timeout config options to Claude ACP when the backend does not advertise them, with regression coverage for the Claude path.

## Reproduction Assessment

Yes, source-reproducible with high confidence: `sessions_spawn` stores the timeout in ACP runtime options, current runtime control code translates it into `timeout`, and Claude ACP rejects config ids it does not advertise. I did not run a live Claude ACP spawn in this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is to suppress unsupported timeout config-option forwarding for Claude or for backends whose advertised config keys omit timeout, while preserving OpenClaw's existing in-process timeout enforcement. A general typed unsupported-option error can be a follow-up, but it is not required to fix this report.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/81127. Current `sessions_spawn` with `runtime: "acp"`, `agentId: "claude"`, and `timeoutSeconds` stores `runtimeOptions.timeoutSeconds` and then forwards it as a `session/set_config_option` timeout control to `@agentclientprotocol/claude-agent-acp`, which only supports mode/model/effort config options and rejects unknown ids. Preserve `timeoutSeconds` as OpenClaw-side turn timeout enforcement via `src/acp/control-plane/manager.core.ts`, but stop sending `timeout`/`timeout_seconds` to Claude ACP or to any backend whose advertised `configOptionKeys` omit timeout.
