## Summary

Keep this PR open. Current main still sends non-strict OpenAI tool parameters through the generic normalizer without adding empty `required` arrays, while this PR is a focused attempt to add that compatibility layer and addresses the earlier review concern about mutating literal objects under `enum`, `const`, `default`, and `examples`.

## What This Changes

The PR adds a non-strict OpenAI tool-parameter post-normalizer that inserts empty `required` arrays into object schemas through schema-bearing keywords, and updates OpenAI tool-schema and WebSocket regression tests.

## Best Possible Solution

Integrate the non-strict OpenAI required-array normalization in `src/agents/openai-tool-schema.ts`, keep strict-mode behavior unchanged, preserve literal JSON values under non-schema keywords, and land it with targeted regression coverage for the WebSocket conversion path.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to call `normalizeOpenAIStrictToolParameters({ type: "object", properties: { text: { type: "string" } } }, false)` on current main and observe that the non-strict branch returns no `required: []`; the PR adds that exact regression coverage.

## Solution Assessment

Yes, directionally. Adding the rewrite after generic schema normalization and limiting recursion to schema-bearing keywords is the narrowest maintainable fix for this compatibility case, but it should be rebased against current main's overlapping OpenAI schema test and strict-normalizer changes before merge.
