## What This Changes

The branch changes scripts/setup-auth-system.sh to render the auth-monitor systemd user unit from a copied template instead of editing the tracked template in place.

## Best Possible Solution

Render and validate a materialized auth-monitor service in a temporary file, replace notification environment lines by key, atomically install after success, and add redacted real setup proof.

## Reproduction Assessment

Yes, by source inspection: current main still edits the tracked service template with sed -i and installs a service whose ExecStart is hard-coded to /home/admin/openclaw/scripts/auth-monitor.sh. I did not execute the interactive setup script in this read-only review.

## Solution Assessment

No. Copying and rendering a materialized unit is the right direction, but the current PR still needs render-before-install behavior and key-based replacement for previously customized notification environment lines.

  - body: This copies the raw template over SERVICE_TARGET before checking python3 availability or validating placeholders.
