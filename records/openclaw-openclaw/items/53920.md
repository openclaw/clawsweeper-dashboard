## What This Changes

The PR updates `scripts/setup-auth-system.sh` to materialize the auth-monitor user systemd service from a copied template with a Python renderer for `ExecStart` and notification environment values.

## Best Possible Solution

Render and validate a temporary materialized auth-monitor service, replace notification environment lines by key before overwriting the installed unit, and attach redacted real setup proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main edits the tracked service template with `sed -i` and copies a template whose `ExecStart` is hard-coded to `/home/admin/openclaw/scripts/auth-monitor.sh`. I did not execute the interactive setup script in this read-only review.

## Solution Assessment

No. Copying and rendering a materialized unit is the right direction, but this PR should render before install and match prior notification environment lines by key before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Render before overwriting the installed service:** `scripts/setup-auth-system.sh:72`
  - body: This copies the raw template over `SERVICE_TARGET` before checking `python3` or validating placeholders. If the renderer is unavailable or exits on a mismatch, setup has already replaced a working user unit with the hard-coded template, so render to a temporary file and install it only after validation succeeds.
  - confidence: 0.93
- **[P1] Match previously rendered notification lines by key:** `scripts/setup-auth-system.sh:113-116`
  - body: Older setup runs could leave the tracked template with `Environment=NOTIFY_NTFY=<user topic>` or `Environment=NOTIFY_PHONE=<user phone>`. This helper only accepts the default placeholder values, so reruns from those checkouts still fail with `placeholder not found`; match the `Environment=<key>=...` line regardless of the old value.
  - confidence: 0.92
