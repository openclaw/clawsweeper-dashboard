## Summary

Keep open: current main still mutates the tracked auth-monitor service template, and the PR is a useful fix direction, but the latest head still has narrow correctness blockers before merge.

## What This Changes

The PR changes `scripts/setup-auth-system.sh` to copy the auth-monitor systemd template to the user unit path, render `ExecStart` and notification environment values with an inline Python helper, and leave the tracked template unmodified.

## Best Possible Solution

Keep the copy-then-render direction, but render and validate a temporary materialized unit before replacing the installed user unit, accept arbitrary legacy `Environment=NOTIFY_*=` lines, and add the required active changelog entry.

## Reproduction Assessment

Yes. Current main is source-reproducible: it edits the tracked systemd template with `sed -i` before copying it, and the template still contains a hard-coded checkout path.

## Solution Assessment

No. The PR’s direction is right, but the latest patch still overwrites the installed unit before validation and does not tolerate arbitrary previously rendered notification environment values.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Render before replacing the installed unit:** `scripts/setup-auth-system.sh:72`
  - body: The PR copies the unrendered template over `SERVICE_TARGET` before checking `python3` or placeholder validity. If either guard fails, setup exits after replacing the user's working unit with the hard-coded template, so render to a temp file and move it into place only after success.
  - confidence: 0.88
