## Summary

Keep this PR open: it targets a real setup-script bug, but the current branch is still not merge-ready because render failures can clobber an existing installed service, previously mutated custom notification lines are not fully tolerated, and the required real behavior proof is still absent.

## What This Changes

The PR changes `scripts/setup-auth-system.sh` to copy the auth-monitor systemd service template into the user unit path and render that copy with the checkout-specific script path and notification environment values.

## Best Possible Solution

Render and validate a materialized service in a temporary file, replace notification environment lines by key regardless of old value, atomically install the validated unit, and attach redacted real setup proof.

## Reproduction Assessment

Yes, by source inspection: current main mutates the tracked service template before copying it, and the template still contains the hard-coded `/home/admin/openclaw` ExecStart. I did not run the interactive systemd setup in this read-only review.

## Solution Assessment

No. The copy-then-render direction is the right boundary, but the safest implementation renders and validates off to the side before replacing the installed unit and rewrites existing notification environment lines by key.

## Review Findings

Full review comments:

- **[P1] Render before replacing the installed service:** `scripts/setup-auth-system.sh:72`
  - body: The branch copies the raw template over `$SERVICE_TARGET` before the `python3` preflight and placeholder validation run. If `python3` is missing or any replacement fails, setup exits after replacing the user's working service with the unrendered template, so the timer can later run the hard-coded `/home/admin/openclaw` path. Render into a temporary file and move it into place only after validation succeeds.
  - confidence: 0.88
- **[P1] Replace notification env lines by key:** `scripts/setup-auth-system.sh:113-115`
  - body: Current main can mutate the tracked template to `Environment=NOTIFY_NTFY=<user topic>` or `Environment=NOTIFY_PHONE=<user phone>`, but this code only accepts the original commented defaults or already-uncommented default values. Rerunning setup from a previously dirtied checkout with custom values still errors after copying the stale unit; match existing `Environment=<key>=...` lines by key instead.
  - confidence: 0.86
