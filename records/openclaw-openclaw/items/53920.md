## Summary

Keep open: current main still mutates the tracked auth-monitor service template, and this PR is aimed at the right bug, but the latest patch still has two concrete correctness issues before merge. The renderer does not actually tolerate arbitrary previously rendered notification values, and it can overwrite an existing user unit before preflight/render validation succeeds.

## What This Changes

The PR updates `scripts/setup-auth-system.sh` so auth-monitor setup copies the systemd service template to the user unit path and renders `ExecStart` plus optional notification environment lines in that copy using a Python helper.

## Best Possible Solution

Keep the copy-then-render direction, but make rendering atomic and idempotent: preflight dependencies before touching the installed unit, render into a temporary file, validate placeholders there, then move it into place only after success. The notification helper should normalize any existing `Environment=NOTIFY_NTFY=...` and `Environment=NOTIFY_PHONE=...` line, with focused coverage for commented placeholders, default uncommented placeholders, custom previously rendered values, and render failure paths.

## Reproduction Assessment

Yes. Current main reproduces by inspection: non-empty notification input runs `sed -i` against the tracked service template before copying it. The remaining PR bugs reproduce by using PR head with a source template already changed to `Environment=NOTIFY_NTFY=<user topic>` or by making `python3`/placeholder validation fail after the direct copy to `$SERVICE_TARGET`.

## Solution Assessment

No. The PR uses the right high-level approach, but the current implementation is not the narrowest safe fix because it validates after overwriting the installed unit and only tolerates default already-uncommented notification placeholders, not arbitrary values produced by older setup runs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Match previously rendered notification lines:** `scripts/setup-auth-system.sh:113-116`
  - body: Older setup runs can leave the tracked template with `Environment=NOTIFY_NTFY=<user topic>` or `Environment=NOTIFY_PHONE=<user phone>`. This helper only matches the default values, so rerunning setup from those dirty checkouts still exits with `placeholder not found`; match any existing `Environment=<key>=...` line before replacing it.
  - confidence: 0.93
- **[P2] Render before overwriting the user unit:** `scripts/setup-auth-system.sh:72-75`
  - body: The script copies the template over `$SERVICE_TARGET` before checking `python3` or validating/rendering placeholders. If that preflight or render step fails, a previously working unit is replaced with the hard-coded template while setup exits before reload/enable; render to a temporary file or preflight before replacing the installed unit.
  - confidence: 0.85
