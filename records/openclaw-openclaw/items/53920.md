## Summary

Keep open: this PR is aimed at a real current-main and shipped setup bug, but the latest head still has two blocking setup-flow issues and lacks inspectable real behavior proof.

## What This Changes

The PR changes `scripts/setup-auth-system.sh` to render the auth-monitor user systemd unit from the tracked template without editing the tracked service file.

## Best Possible Solution

Render and validate the service into a temporary materialized unit, install it only after success, and replace notification environment lines by key while keeping the tracked template unchanged.

## Reproduction Assessment

Yes, by source inspection: current main still edits `scripts/systemd/openclaw-auth-monitor.service` with `sed -i` when notification values are provided, then installs a template whose `ExecStart` is hard-coded to `/home/admin/openclaw/scripts/auth-monitor.sh`. I did not run the interactive setup because this review is read-only.

## Solution Assessment

No. Copy-then-render is the right direction, but this branch still overwrites the installed unit before validation and only matches prior notification environment lines when their old value equals the default placeholder.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Render before overwriting the installed service:** `scripts/setup-auth-system.sh:72`
  - body: This copies the unrendered template over `SERVICE_TARGET` before the `python3` preflight or placeholder validation runs. If the renderer is unavailable or exits on a mismatch, setup has already replaced a working user unit with the hard-coded template, so render to a temporary file and install it only after validation succeeds.
  - confidence: 0.93
- **[P2] Replace prior NOTIFY lines by key:** `scripts/setup-auth-system.sh:113-115`
  - body: The helper only accepts the original placeholder value or that same value already uncommented. Existing dirty checkouts can contain `Environment=NOTIFY_NTFY=<user topic>` or `Environment=NOTIFY_PHONE=<user phone>`, so reruns still fail with "placeholder not found" instead of repairing the unit; match the key regardless of the old value.
  - confidence: 0.92
