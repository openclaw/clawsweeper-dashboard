## Summary

Keep open: current main still has the reported setup bug, and this PR is directionally useful, but latest head still has two correctness blockers and lacks required real behavior proof for an external setup-script change.

## What This Changes

This PR changes `scripts/setup-auth-system.sh` so auth-monitor systemd service materialization copies the template to the user unit path and renders `ExecStart` plus optional notification `Environment` values there.

## Best Possible Solution

Render and validate a temporary user service from the tracked template, match notification `Environment` lines by key regardless of old value, then install only after successful rendering with redacted real setup proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `sed -i` mutates the tracked service template before copying it, while the template still hard-codes `/home/admin/openclaw/scripts/auth-monitor.sh`; I did not run setup because this review is read-only.

## Solution Assessment

No. Copy-then-render is the right direction, but latest PR head still copies over the installed unit before renderer checks and only matches prior notification lines when they contain the default placeholder value.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Render before replacing the installed unit:** `scripts/setup-auth-system.sh:72`
  - body: This copies the unrendered template over `SERVICE_TARGET` before the `python3` check or placeholder validation runs. If the renderer is unavailable or exits on a mismatch, setup has already replaced a working user unit file with the hard-coded template; render to a temporary file and install it only after validation succeeds.
  - confidence: 0.92
- **[P2] Match existing notification lines by key:** `scripts/setup-auth-system.sh:114-115`
  - body: Older setup runs can leave `Environment=NOTIFY_NTFY=<user topic>` or `Environment=NOTIFY_PHONE=<user phone>` in the tracked template. This helper only accepts the default placeholder value, so reruns from those dirty checkouts still fail with `placeholder not found`; match any existing line for the key instead.
  - confidence: 0.9
