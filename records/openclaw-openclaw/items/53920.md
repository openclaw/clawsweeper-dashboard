## Summary

Keep open: current main still has the tracked-template mutation bug, and this PR is in the right direction but still has narrow script-local blockers before merge.

## What This Changes

The PR changes `scripts/setup-auth-system.sh` to copy the auth-monitor systemd service to the user unit path, render the copied unit with the current checkout path and notification values, and leave the tracked template untouched.

## Best Possible Solution

Render and validate a temporary materialized unit, atomically install it only after success, accept any legacy `Environment=NOTIFY_*` line, and record the setup fix in the active changelog.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main runs `sed -i` on the tracked service template before copying it, and that template still contains the hard-coded `/home/admin/openclaw/scripts/auth-monitor.sh` path. I did not run the interactive setup because this review is read-only and that command would mutate user systemd files.

## Solution Assessment

No. Copying before rendering is the right direction, but the current patch validates after overwriting the installed unit and still fails on arbitrary legacy notification environment values from previously dirtied templates.
