## Summary

This is a focused, still-useful PR: current main still mutates the tracked auth-monitor systemd template during setup, while the PR renders a temporary unit, validates required lines, avoids the earlier Python dependency, and supplies credible terminal proof. I found no blocking patch defect; the remaining merge decision is maintainer validation of CI state and whether stubbed systemctl proof is enough for this optional setup path.

## What This Changes

The PR updates `scripts/setup-auth-system.sh` so auth-monitor service rendering happens through temporary materialized files instead of editing the tracked systemd template in place.

## Best Possible Solution

Land the temp-rendered service install after required checks are green or waived, keeping the fix limited to setup-unit materialization and not changing auth-monitor runtime behavior.

## Reproduction Assessment

Yes. Current-main source shows `setup-auth-system.sh` edits `scripts/systemd/openclaw-auth-monitor.service` with `sed -i` before copying it, so entering notification values would mutate the tracked template and preserve the template’s hardcoded ExecStart path.

## Solution Assessment

Yes. Rendering a temporary copy, validating required markers, accepting previously rendered `NOTIFY_*` lines, quoting ExecStart, and avoiding a Python dependency is the narrow maintainable fix for the reported script behavior.
