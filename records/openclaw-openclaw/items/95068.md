## Summary

Keep open: current main and the latest release still render envelope and system-event date portions as YYYY-MM-DD, and adding `dateFormat` would be a new user-facing config surface that needs maintainer product direction rather than an automated bug-fix lane.

## What This Changes

## Best Possible Solution

Maintainers should first decide the date-display contract; if accepted, implement it once in the shared timestamp formatter, thread it through envelope and system-event callers, update config schema/help/docs, and keep machine-readable timestamps and filenames ISO.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers confirm the product contract; a shared date-format option is a plausible clean path, but ISO-only docs or OS-locale behavior may be safer than adding another config key.
