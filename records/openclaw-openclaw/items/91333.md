## Summary

Keep this PR open: it targets the right message-action/payload boundary and now has branch-backed terminal proof for the happy path, but the patch has a blocking delivery-evidence bug that can suppress a final reply after an unproven or failed send.

## What This Changes

The PR annotates current-source `message(action="send")` results as delivered source replies and uses that evidence to suppress later assistant final payloads in automatic mode, with focused regression tests.

## Best Possible Solution

Keep the source-reply evidence path, but only mark a current-source message send as delivered when the send/tool result contains positive delivery evidence, with a regression for failed or ambiguous current-source sends preserving the final reply.

## Reproduction Assessment

Yes at source level: current main only suppresses later finals from message-tool source replies in `message_tool_only`, while the linked incident and PR probe exercise automatic-mode current-source sends followed by private final text. I did not run a live Telegram replay in this read-only review.

## Solution Assessment

No as submitted: the owner boundary is right, but defaulting missing delivery evidence to `sent` is not the safest implementation. The best fix is to annotate only proven successful current-source sends and preserve the final reply when delivery is failed or ambiguous.
