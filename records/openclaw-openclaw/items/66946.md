## Summary

Keep open: current `main` and `v2026.5.7` still compose the duplicated Control UI bootstrap config URL for the reported `/__openclaw__/` entry path. The directly linked fix PRs were closed unmerged, and the only merged related PR hardened auth for the same endpoint without fixing this path contract.

## What This Changes

The issue reports that opening Control UI under `/__openclaw__/` can request `/__openclaw__/__openclaw/control-ui-config.json`, returning 404 and preventing chat startup.

## Best Possible Solution

Land a focused Control UI/gateway fix that makes the bootstrap config endpoint consistently base-path-aware across the frontend loader, gateway route, dev stub, tests, docs, and changelog.

## Reproduction Assessment

Yes, by current-main source inspection: `/__openclaw__/` can infer `/__openclaw__`, then the frontend fetches `/__openclaw__/__openclaw/control-ui-config.json`, which the gateway does not serve as the bootstrap endpoint. I did not run a live gateway in this read-only review.

## Solution Assessment

Yes. The narrow maintainable solution is to align the shared bootstrap path contract with base-path handling while preserving the existing Control UI auth behavior.
