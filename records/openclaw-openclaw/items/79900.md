## Summary

Keep open: the PR fixes the main HTML 401 formatter bug, but it introduces a new auth HTML failure kind without adding it to the existing raw-HTML console suppression paths. The related plain-401 PR at https://github.com/openclaw/openclaw/pull/77394 is adjacent but does not supersede this HTML-specific fix.

## What This Changes

The PR adds `auth_html_401` classification and re-authentication copy for provider HTTP 401 responses with HTML bodies, plus formatter regression tests and a changelog entry.

## Best Possible Solution

Land a narrow version that treats HTML 401 like auth HTML everywhere: classifier, formatter, lifecycle/failover console suppression, and focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current main routes `401 <!DOCTYPE html>...</html>` through the HTML branch but only maps status 403 to `auth_html_403`, so 401 falls to `upstream_html`. I did not run tests because this review is read-only and tests may write artifacts.

## Solution Assessment

No as submitted: adding `auth_html_401` is the right narrow user-facing fix, but the maintainable patch must also mirror existing `auth_html_403` raw-error console suppression and tests.
