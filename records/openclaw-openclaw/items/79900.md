## Summary

Keep open: the bug is source-reproducible and the core classifier change is plausible, but the PR still leaves the new `auth_html_401` kind out of raw HTML console suppression and lacks after-fix real behavior proof.

## What This Changes

The PR adds an `auth_html_401` provider-runtime failure kind, maps HTML 401 responses to re-authentication copy, adds formatter tests, and updates the changelog.

## Best Possible Solution

Treat `auth_html_401` consistently with `auth_html_403` across classification, user-facing copy, lifecycle/failover console suppression, and regression coverage, then add redacted real-run proof for an HTML 401 provider response.

## Reproduction Assessment

Yes, source-reproducible: current main routes `401 <!DOCTYPE html>...</html>` through the HTML branch to `upstream_html` because only status 403 gets `auth_html_403`. I did not run tests because this review is read-only and tests may write artifacts.

## Solution Assessment

No as submitted: adding `auth_html_401` is the narrow maintainable direction for user-facing copy, but the new auth kind also needs to be wired into raw HTML log suppression and proven in a real setup.
