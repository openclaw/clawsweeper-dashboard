## Summary

Keep open: the current-main classifier bug is real and the PR is pointed at the right user-facing path, but the patch does not carry the new auth HTML kind through existing raw-error console suppression and it lacks required real behavior proof.

## What This Changes

The PR adds an `auth_html_401` provider runtime failure kind, routes HTML 401 responses to re-authentication copy, adds formatter regression tests, and updates the changelog.

## Best Possible Solution

Treat `auth_html_401` consistently with `auth_html_403` across classification, user-facing copy, lifecycle/failover observation suppression, and regression coverage, then add redacted real-run proof for an HTML 401 provider response.

## Reproduction Assessment

Source-reproducible, yes: current main routes `401 <!DOCTYPE html>...</html>` through the HTML branch to `upstream_html` because only status 403 gets `auth_html_403`. I did not run tests in this read-only review because the task forbids commands that may write artifacts.

## Solution Assessment

No as submitted: adding `auth_html_401` is the narrow maintainable direction for the user-facing copy, but the new auth kind also needs to be wired into the existing raw HTML log-suppression paths and proven in a real setup.
