## Summary

Keep open: the underlying HTML 401 misclassification is source-reproducible, but the PR still misses raw HTML console suppression for the new auth kind and lacks external after-fix real behavior proof.

## What This Changes

The PR adds an `auth_html_401` provider runtime failure kind, routes HTML 401 responses to re-authentication copy, adds formatter regression tests, and updates the changelog.

## Best Possible Solution

Treat `auth_html_401` consistently with `auth_html_403` across classification, user-facing copy, lifecycle/failover raw HTML suppression, and regression coverage, then add redacted real-run proof for an HTML 401 provider response.

## Reproduction Assessment

Yes, source-reproducible: current main routes `401 <!DOCTYPE html>...</html>` through the HTML branch to `upstream_html` because only status 403 gets `auth_html_403`. I did not run tests because this read-only review forbids commands that may write artifacts.

## Solution Assessment

No as submitted: adding `auth_html_401` is the narrow maintainable direction for the user-facing copy, but the new kind also needs existing raw HTML log-suppression handling and real behavior proof.
