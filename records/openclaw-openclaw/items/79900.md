## Summary

Keep open: current main still misclassifies HTML 401 provider responses as upstream HTML, and this PR fixes the central user-facing copy, but the patch is incomplete because the new `auth_html_401` kind is not included in existing raw HTML console-message suppression.

## What This Changes

The PR adds an `auth_html_401` provider runtime failure kind, maps HTML 401 responses to re-authentication copy, adds formatter tests, and updates the changelog.

## Best Possible Solution

Land a narrow patch that treats HTML 401 like auth HTML everywhere: classifier, formatter, lifecycle/failover console suppression, and focused tests, while leaving non-401/403 `upstream_html` behavior unchanged.

## Reproduction Assessment

Yes, source-reproducible: current main's HTML branch maps only 403 to `auth_html_403` and all other HTML responses to `upstream_html`, so a 401 HTML body gets CDN/gateway retry copy. I did not execute tests in this read-only review.

## Solution Assessment

No as submitted: adding `auth_html_401` is the right narrow fix for the user-facing copy, but the maintainable solution must also mirror the existing raw HTML console suppression and regression coverage for the new auth kind.
