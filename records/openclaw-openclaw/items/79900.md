## Summary

Keep open: the bug is real and the PR takes the right classifier direction, but it introduces `auth_html_401` without updating the existing auth HTML raw-error console suppression paths, so 401 login HTML can still be appended to lifecycle/failover `consoleMessage` output.

## What This Changes

The PR adds an `auth_html_401` provider-runtime failure kind so HTML 401 provider responses get re-authentication copy instead of CDN-blocked retry copy, with formatter tests and a changelog entry.

## Best Possible Solution

Land a narrow version that treats HTML 401 responses consistently as authentication HTML across classifier, formatter, raw-error suppression, and regression coverage.

## Reproduction Assessment

Yes, source-reproducible: current main routes `401 <!DOCTYPE html>...</html>` through the HTML branch but only maps status 403 to `auth_html_403`, so 401 falls to `upstream_html`. I did not run tests because this review is read-only.

## Solution Assessment

No as submitted: adding `auth_html_401` is the right narrow fix for user-facing copy, but the maintainable solution also mirrors the existing `auth_html_403` raw-error console suppression and tests.
