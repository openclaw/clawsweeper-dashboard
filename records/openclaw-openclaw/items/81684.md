## Summary

Keep open: this PR addresses a real source-proven web-search credential-selection bug, includes useful terminal proof, and has no blocking security or correctness issue. I found one low-priority diagnostic cleanup edge around inactive-provider warnings for the shared legacy key.

## What This Changes

The branch makes web-search runtime provider selection fall back from plugin-scoped credential reads to provider `getCredentialValue(searchConfig)` and adds regression coverage for legacy top-level search API keys in runtime and isolated cron paths.

## Best Possible Solution

Land a focused credential-selection fix that honors legacy `tools.web.search.apiKey` for compatible providers while keeping inactive-secret diagnostics tied to paths the user actually configured.

## Reproduction Assessment

Yes at source/test level: current main’s runtime selection ignores `getCredentialValue(searchConfig)` while docs and provider execution still support legacy `tools.web.search.apiKey`. I did not establish a live Docker cron reproduction.

## Solution Assessment

Mostly yes: aligning runtime selection with provider execution is the narrow fix. The patch should also avoid emitting inactive-provider warnings for unrelated plugin-scoped paths when the only configured SecretRef is the shared legacy top-level key.

## Review Findings

Overall correctness: patch is correct
