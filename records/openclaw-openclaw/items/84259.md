## Summary

Keep open: source inspection confirms the reported Codex-runtime parameter miss, but this PR is not merge-ready because the new lookup can override explicit `openai-codex/*` params and the supplied proof still does not show the real Codex `/status` or runtime-param route.

## What This Changes

The PR changes Codex runtime extra-param resolution to look up canonical `openai/*` model params for `openai-codex` sessions and adds resolver/status tests for `textVerbosity`.

## Best Possible Solution

Use canonical `openai/*` params as the Codex-runtime fallback only when provider-specific `openai-codex/*` params are absent, add a both-configured regression test, and require redacted real-route proof before merge.

## Reproduction Assessment

Yes, as a source-level reproduction. Current main resolves params by active provider/model, so an `openai-codex` runtime session misses canonical `openai/gpt-5.5` params and then applies the GPT-5 low verbosity default; I did not run live Codex OAuth.

## Solution Assessment

No, not as written. Falling back to canonical `openai/*` matches the documented route, but making it win over explicit `openai-codex/*` params is not the safest compatibility-preserving fix.
