## Summary

Keep open: the linked bug is source-reproducible and this PR targets the right resolver/status surface, but the patch globally makes canonical `openai/*` params outrank explicit legacy `openai-codex/*` params and still lacks real Codex/status-route proof.

## What This Changes

The PR changes `resolveExtraParams` to look up canonical `openai/*` model params before `openai-codex/*` for Codex runtime sessions and adds resolver/status tests for `textVerbosity`.

## Best Possible Solution

Pass the selected/configured model ref into param and status lookup so canonical `openai/*` config is used for canonical Codex-routed sessions, explicit legacy `openai-codex/*` selections keep their own params, and the real route is verified after the patch.

## Reproduction Assessment

Yes. Source inspection shows current main resolves params/status through the active `openai-codex/gpt-5.5` provider/model, misses the configured canonical `openai/gpt-5.5` params, and then applies the GPT-5 `low` default; I did not run a live OAuth route in this read-only review.

## Solution Assessment

No, not as proposed. The safe fix should distinguish canonical selected-model lookup from explicit legacy `openai-codex/*` lookup instead of always preferring `openai/*` whenever the runtime provider is `openai-codex`.
