## Summary

Keep open: current main still lets local `openclaw infer model run --thinking adaptive` reach pi-ai's native Google simple-completion path, where `gemini-flash-latest` is not recognized as Gemini 3 Flash and can fall back to `thinkingBudget: -1`. OpenClaw's provider-owned Google sanitizer already knows the alias, but this local CLI path can bypass that wrapper.

## What This Changes

The issue asks OpenClaw to stop sending legacy `thinkingBudget: -1` for `google/gemini-flash-latest` adaptive CLI inference runs and use Gemini 3 thinking-level/dynamic semantics instead.

## Best Possible Solution

Make local Google simple-completion runs reuse the provider-owned Google thinking normalization before the request leaves OpenClaw, with regression coverage for `gemini-flash-latest` and unchanged Gemini 2.5 adaptive behavior.

## Reproduction Assessment

Yes, from source inspection: local `infer model run` reaches pi-ai simple completion, and locked pi-ai 0.75.3 misses the `gemini-flash-latest` alias before falling back to a negative thinking budget. I did not run the live Google command because this review is read-only and credential-dependent.

## Solution Assessment

Yes: the narrow maintainable fix is to route local Google simple-completion payloads through the existing provider-owned Google thinking normalization instead of adding a CLI-only alias special case or changing Gemini 2.5 semantics.
