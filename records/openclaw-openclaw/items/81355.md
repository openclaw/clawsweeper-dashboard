## Summary

Keep this issue open. The plugin auto-enable fanout half is fixed on current main and in v2026.6.6, but current `tts.status` still performs the reported synchronous prefs reads and provider configured-state scans before responding.

## What This Changes

The issue reports two cold-start Gateway fanout bottlenecks: synchronous `tts.status` work and repeated `applyPluginAutoEnable` computation.

## Best Possible Solution

Keep the auto-enable cache as shipped and make one focused TTS status hot-path repair that avoids sync prefs/provider recomputation while preserving the existing RPC response shape.

## Reproduction Assessment

Yes, by source inspection rather than a fresh live run: current `tts.status` still executes the prefs reads and provider configured scans synchronously before responding. I did not reproduce the exact reported latency numbers on current main.

## Solution Assessment

Yes for the remaining direction: the auto-enable side should stay untouched, and the best fix is a bounded TTS-owned status-path repair. A safer implementation should preserve existing sync SDK helpers or add a compatible async/status helper instead of breaking exported speech-core contracts.
