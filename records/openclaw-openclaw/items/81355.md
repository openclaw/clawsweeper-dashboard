## Repair Work Prompt

Fix Bug (A) from https://github.com/openclaw/openclaw/issues/81355: first-load RPC fanout stalls because `tts.status` performs synchronous TTS config, prefs, provider-order, and provider-state work without yielding. Do not duplicate the `applyPluginAutoEnable` cache work; https://github.com/openclaw/openclaw/pull/81389 tracks that half, and https://github.com/openclaw/openclaw/pull/78639 is related broader async plugin-discovery work. Keep core/plugin boundaries intact and do not change the public `SpeechProviderPlugin.isConfigured` contract unless the existing code already has a compatible seam. Prefer a narrow internal fix in `tts.status`/speech-core that carries already-read prefs/provider/config facts forward, avoids duplicate `isConfigured` calls for the same provider within one status response, and yields between expensive sync chunks so sibling gateway RPCs can interleave. Add focused regression coverage for the handler behavior and provider-configured call reuse. Add a single changelog entry if the repair is user-visible performance work.
