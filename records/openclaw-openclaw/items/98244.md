## Label Justifications

- `P1`: The report describes OpenAI Responses streaming turns taking the SDK timeout after successful semantic completion, making the provider workflow effectively unusable when reproduced.
- `impact:auth-provider`: The affected surface is OpenAI provider transport/model streaming behavior for the Responses API.
- `impact:crash-loop`: The user-visible failure mode is a process that remains alive but stalls until a timeout rather than completing promptly.

## Summary

Keep open: current main still processes `response.completed` but then waits for the underlying async stream to finish, and the related sanitizer/first-event fixes are adjacent rather than a complete fix for this post-completion timeout report.

## What This Changes

The issue requests fixing OpenAI Responses streaming so a terminal `response.completed` event does not leave OpenClaw waiting for the SDK/request timeout.

## Best Possible Solution

Add cleanup-aware terminal-event handling for OpenAI Responses streams in the transport and shared provider paths, with regression tests for a stream that yields `response.completed` and then stalls.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level, but not with a live OpenAI request: both current loops process `response.completed` and then keep awaiting the async iterable, so a mocked completed-then-never-resolving stream should reproduce the hang path.

## Solution Assessment

Is this the best way to solve the issue? Not exactly as proposed: extending `withResponsesFirstEventTimeout()` with a broader control API is one option, but the narrower maintainable fix is terminal-event handling plus cleanup inside the Responses stream processors.
