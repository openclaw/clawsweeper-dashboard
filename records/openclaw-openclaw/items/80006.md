## Summary

Keep open. The PR targets a real ClawHub archive body-read timeout bug, but the patch cancels `response.body` after `arrayBuffer()` has locked the stream and the PR still lacks contributor-provided real behavior proof, so it should not merge as-is.

## What This Changes

The branch adds a shared timeout wrapper around ClawHub archive `Response.arrayBuffer()` reads for skill, package, and ClawPack downloads, plus mocked stalled-body regression tests.

## Best Possible Solution

Keep the PR open and rework archive body consumption so the timeout owns the active reader or abort signal through the full body read, with real-stream regression coverage and posted real behavior proof.

## Reproduction Assessment

Yes, at source level. A fetch implementation can return headers and then a body whose `arrayBuffer()` never resolves, and current main awaits that body after clearing the request timeout.

## Solution Assessment

No. The direction is right, but this patch should use a reader or abort-controller path that owns the in-flight body read instead of cancelling `response.body` after `arrayBuffer()` has locked it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Cancel the in-flight body read, not the locked stream:** `src/infra/clawhub.ts:652`
  - body: After `response.arrayBuffer()` starts, the response body is locked; Node/Web Streams reject `response.body.cancel(...)` with `TypeError: Invalid state: ReadableStream is locked`. This timeout rejects the caller but does not abort the stalled transfer, so the CLI can still be kept alive by the hanging body read; use a reader or abort path that owns the body read.
  - confidence: 0.91
