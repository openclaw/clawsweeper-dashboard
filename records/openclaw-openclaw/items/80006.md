## Summary

Keep open. The PR targets a real current-main ClawHub stalled-body gap and now has sufficient local-stub behavior proof, but the proposed timeout cancels `Response.body` after `response.arrayBuffer()` has locked it, so the patch should not merge as-is.

## What This Changes

The PR adds a shared timeout wrapper around ClawHub archive body reads and regression tests for stalled skill, package, and ClawPack downloads.

## Best Possible Solution

Use a ClawHub body-read helper that owns the active reader or abort signal through archive consumption, with stalled-body coverage for skill, package, and ClawPack downloads.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main clears the request timeout before archive `arrayBuffer()` reads, and a headers-then-stalled body can keep those reads pending; the Node stream probe also confirms the PR's cancellation target is locked.

## Solution Assessment

No. The PR targets the right ClawHub boundary, but the timeout should cancel the active reader or abort-controlled read instead of calling `Response.body.cancel()` after `arrayBuffer()` has locked the stream.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Cancel the active body read:** `src/infra/clawhub.ts:652`
  - body: After `response.arrayBuffer()` starts, `response.body` is locked, so this timeout path's `response.body.cancel(...)` rejects with `TypeError` and is ignored. The caller sees the timeout, but the stalled transfer can keep running in the background; make the timeout own a reader or abort signal for the active body read.
  - confidence: 0.91
