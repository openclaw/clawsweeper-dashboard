## Summary

Keep open. The PR targets a real current ClawHub archive download hang, and the contributor's local-stub proof shows the caller now receives a timeout, but the patch still tries to cancel a `Response.body` after `arrayBuffer()` has locked it, so the stalled transfer can remain alive in the background.

## What This Changes

The PR adds a shared ClawHub archive body-read timeout helper and stalled-body regression tests for skill zip, package zip, and ClawPack artifact downloads.

## Best Possible Solution

Implement ClawHub archive body reads with a reader- or abort-owned timeout across skill, package, and ClawPack downloads, keeping stalled-body regression coverage.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main clears the fetch timeout before archive bodies are consumed, and a headers-then-stalled body can leave `response.arrayBuffer()` pending indefinitely. I also verified in Node that `arrayBuffer()` locks the body and makes `response.body.cancel()` reject.

## Solution Assessment

No. The direction is right, but the helper should consume through an owned reader or keep an abort signal active through body consumption instead of cancelling `response.body` after `arrayBuffer()` has locked it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Abort the active body read:** `src/infra/clawhub.ts:652`
  - body: After `response.arrayBuffer()` starts, `response.body` is locked, so `response.body.cancel(...)` rejects with `TypeError: Invalid state: ReadableStream is locked` and this code ignores it. The caller sees the timeout, but the stalled transfer can keep running in the background; use a reader- or abort-owned body read that can cancel the in-flight operation.
  - confidence: 0.91
