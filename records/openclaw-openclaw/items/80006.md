## Summary

Keep open. This is a valid narrow fix path for the ClawHub install hang, but the current patch times out the caller without aborting the active body read because it cancels a stream already locked by `arrayBuffer()`.

## What This Changes

The PR adds a ClawHub archive body-read timeout helper and stalled-body regression tests for skill, package, and ClawPack artifact downloads.

## Best Possible Solution

Use reader-owned or abort-controller-owned body consumption so the timeout both rejects the caller and terminates the in-flight transfer for every ClawHub archive download path.

## Reproduction Assessment

Yes. Source inspection shows current main clears the request timeout before archive `arrayBuffer()` calls, and a focused Node Web Streams probe confirms the PR's timeout cancellation path targets a locked body.

## Solution Assessment

No. The PR direction is right, but the helper should own the active reader or carry an abort signal through body consumption instead of calling `response.body.cancel()` after `arrayBuffer()` has locked the stream.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Abort the active body read:** `src/infra/clawhub.ts:652`
  - body: After `response.arrayBuffer()` starts, `response.body` is locked; `response.body.cancel(...)` rejects with `TypeError: Invalid state: ReadableStream is locked` and this code ignores it. The caller sees the timeout, but the stalled transfer can keep running in the background, so use a reader- or abort-owned body read that can cancel the in-flight operation.
  - confidence: 0.91
