## Summary

Keep open. The PR targets a real ClawHub archive download hang and now has sufficient terminal proof, but the proposed timeout path rejects the caller without cancelling the active locked body read.

## What This Changes

The PR adds a shared ClawHub archive body-read timeout helper and stalled-body regression tests for skill, package, and ClawPack downloads.

## Best Possible Solution

Rework ClawHub archive body consumption so the timeout owns the active reader or abort signal through the body read, with real stalled-body coverage for skill, package, and ClawPack archives.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main clears the request timeout before archive body reads, and a fetch that returns headers then never completes `arrayBuffer()` can hang; the direct Node stream check also verifies the PR's cancel path hits a locked body.

## Solution Assessment

No. The direction is right, but the helper should consume via a reader or carry an abort controller through body consumption instead of cancelling `response.body` after `arrayBuffer()` has locked it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Abort the active body read:** `src/infra/clawhub.ts:652`
  - body: After `response.arrayBuffer()` starts, `response.body` is locked; `response.body.cancel(...)` rejects with `TypeError: Invalid state: ReadableStream is locked` and this code ignores it. The caller sees the timeout, but the stalled transfer can keep running in the background, so use a reader- or abort-owned body read that can cancel the in-flight operation.
  - confidence: 0.91
