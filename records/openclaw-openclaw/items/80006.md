## Summary

Keep open. The PR targets a valid current-main ClawHub archive hang, but the new timeout does not actually cancel a locked `arrayBuffer()` body read and the external PR still lacks after-fix real behavior proof.

## What This Changes

The branch adds a shared ClawHub archive body-read timeout helper, switches skill/package/ClawPack downloads to it, and adds mocked regression tests for stalled body reads.

## Best Possible Solution

Keep the PR open and rework the archive body consumption so the timeout owns the active reader or abort signal through the full body read, with real-stream regression coverage and posted real behavior proof.

## Reproduction Assessment

Yes. Source inspection shows current main awaits archive `response.arrayBuffer()` after clearing the request timeout, and a direct Node check confirms the PR’s proposed `response.body.cancel(...)` path fails once `arrayBuffer()` locks the stream.

## Solution Assessment

No. The requested direction is right, but this implementation should use a body reader or abort-controller path that owns the in-flight body read rather than cancelling the already-locked `Response.body`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Cancel the in-flight body read, not the locked stream:** `src/infra/clawhub.ts:652`
  - body: After `response.arrayBuffer()` starts, the response body is locked; Node/Web Streams reject `response.body.cancel(...)` with `TypeError: Invalid state: ReadableStream is locked`. This timeout rejects the caller but does not abort the stalled transfer, so the CLI can still be kept alive by the hanging body read. Use a reader or abort path that owns the body read, and cover it with a real stream in the regression test.
  - confidence: 0.91
