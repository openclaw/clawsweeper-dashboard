## Summary

Keep this PR open, but it still needs a code repair before merge: it targets a real ClawHub archive hang on current main, yet the proposed timeout cancels `Response.body` after `arrayBuffer()` has already locked the stream.

## What This Changes

The PR adds a shared ClawHub archive body-read timeout for skill, package, and ClawPack downloads, plus stalled-body regression tests.

## Best Possible Solution

Use a shared ClawHub body-consumption helper that owns the active reader or abort signal through the full archive read, with real `ReadableStream` stalled-body coverage for skill, package, and ClawPack downloads.

## Reproduction Assessment

Yes. Current main clears the request timeout before `response.arrayBuffer()` reads the archive body, and a headers-then-never-finishing body can reproduce the hang path; I also verified that `arrayBuffer()` locks the stream before the PR's cancellation call.

## Solution Assessment

No. The direction is right, but the helper should cancel the active reader or abort the in-flight body read instead of calling `response.body.cancel()` after `arrayBuffer()` has locked the stream.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Abort the active body read:** `src/infra/clawhub.ts:652`
  - body: After `response.arrayBuffer()` starts, the response stream is locked; when the timeout fires, `response.body.cancel(err)` rejects with `TypeError: Invalid state: ReadableStream is locked` and is ignored. The caller sees the timeout, but the stalled body read can keep running in the background; use a reader-owned loop or abort signal that owns the in-flight body consumption.
  - confidence: 0.91
