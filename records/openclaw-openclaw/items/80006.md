## Summary

Keep open. The bug and fix direction are valid, but the current patch times out the caller while leaving the active `arrayBuffer()` body read locked and uncancelled, so it needs a narrow repair before merge.

## What This Changes

The PR adds a shared ClawHub archive body-read timeout helper and stalled-body regression tests for skill, package, and ClawPack downloads.

## Best Possible Solution

Rework ClawHub archive body consumption so the timeout owns the active reader or abort signal through the body read, with real stalled-body coverage for skill, package, and ClawPack archives.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main clears the request timeout before archive body reads, and a fetch that returns headers then never completes `arrayBuffer()` can hang. A direct Node Web Streams check also confirms the PR's cancellation path rejects once `arrayBuffer()` locks the body.

## Solution Assessment

No. The direction is right, but the helper should consume via a reader or carry an abort controller through body consumption instead of cancelling `response.body` after `arrayBuffer()` has locked it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Abort the active body read:** `src/infra/clawhub.ts:652`
  - body: After `response.arrayBuffer()` starts, `response.body` is locked; `response.body.cancel(...)` rejects with `TypeError: Invalid state: ReadableStream is locked` and this code ignores it. The caller sees the timeout, but the stalled transfer can keep running in the background, so use a reader- or abort-owned body read that can cancel the in-flight operation.

## Repair Work Prompt

Repair https://github.com/openclaw/openclaw/pull/80006. The underlying bug is valid: ClawHub skill, package, and ClawPack archive downloads need a body-read timeout after headers arrive. Replace the proposed `readResponseArrayBufferWithTimeout` implementation so the timeout owns the active body read, for example by using `response.body.getReader()` and cancelling that reader on timeout, or by carrying an `AbortController` through body consumption. Do not leave `response.arrayBuffer()` running after the timeout rejects. Update the stalled-body tests to use a real `ReadableStream`/reader cancellation path rather than a fake `Response` with an independent `cancel` mock. Keep the scope to ClawHub archive download behavior and adjacent tests; do not change Telegram, Feishu, install UX, dependency versions, workflows, or package publishing metadata. Changelog handling is maintainer-owned at landing and should not block this repair PR by itself.
