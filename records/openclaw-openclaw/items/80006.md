## Summary

This PR found the right ClawHub stalled-body bug and supplied useful proof, but its implementation leaves the active locked body read uncancelled; the same fix is now owned by the newer open replacement at https://github.com/openclaw/openclaw/pull/81601.

## What This Changes

This PR adds a ClawHub archive body-read timeout helper and stalled-body regression tests for skill, package, and ClawPack downloads.

## Best Possible Solution

Review and land the reader-owned cancellation implementation in https://github.com/openclaw/openclaw/pull/81601, then resolve the linked user bug when that fix reaches main.

## Reproduction Assessment

Yes, at source level: current main clears the fetch timeout before archive bodies are consumed, and the affected paths await response.arrayBuffer() directly. A focused Node Web Streams check confirms that response.body is locked after arrayBuffer() starts, so this PR's cancellation path cannot cancel the active read.

## Solution Assessment

No. The timeout direction is correct, but the patch cancels response.body after arrayBuffer() owns the stream; the maintainable fix is the reader-owned cancellation approach now in https://github.com/openclaw/openclaw/pull/81601.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Cancel the active body reader:** `src/infra/clawhub.ts:652`
  - body: After response.arrayBuffer() starts, the body stream is locked, so response.body.cancel(...) rejects and is ignored. The caller sees the timeout, but the stalled transfer can keep running in the background; read through an owned reader or carry an abort signal through body consumption so the timeout can cancel the in-flight read.
  - confidence: 0.91
