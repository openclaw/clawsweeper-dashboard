## What This Changes

This PR adds a guard in `resolveMcpTransportConfig` to skip MCP server configs that set both a non-empty `command` and `url`, plus focused resolver tests.

## Best Possible Solution

Land a resolver-level mixed-config rejection, or an equivalent same-layer guard, after real behavior proof and explicit maintainer acceptance of the skip-on-upgrade behavior.

## Reproduction Assessment

Yes, source-level: current main resolves a non-empty `command` before considering `url` or `transport`, so the reported mixed streamable-http config takes the stdio path. I did not run the external OpenSpace setup.

## Solution Assessment

Yes, with gates: resolver-level rejection is the narrowest owner-layer fix because it sees both `command` and `url` before transport construction. Maintainers should still accept the fail-closed compatibility change and decide whether schema or doctor alignment is required.
