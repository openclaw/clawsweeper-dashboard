## Summary

Keep open: current main and the latest stable release still resolve mixed MCP `command` plus `url` configs through the stdio path, and this PR is the focused candidate for the linked bug. The patch looks source-correct, but it still needs real behavior proof and maintainer acceptance of the compatibility change from silent stdio precedence to fail-closed server skipping.

## What This Changes

The PR adds an early MCP transport resolver guard that skips configs with both non-empty `command` and `url`, plus focused tests for mixed configs, whitespace-only URLs, and sanitized diagnostics.

## Best Possible Solution

Merge a proof-backed resolver-level rejection for mixed MCP transport configs, or have maintainers explicitly choose a more upgrade-safe path such as doctor/schema diagnostics before changing runtime behavior.

## Reproduction Assessment

Yes, source-level: current main resolves a non-empty `command` before considering `url` or `transport`, so the reported mixed streamable-http config takes the stdio path. I did not run the external OpenSpace setup.

## Solution Assessment

Mostly yes: the resolver is the narrowest layer that sees both fields before SDK transport construction and it matches the current docs. It is not merge-ready until real behavior proof is supplied and maintainers accept the compatibility change.
