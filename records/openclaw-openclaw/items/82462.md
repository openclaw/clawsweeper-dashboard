## Summary

Keep open. Current main has the related split-frame sanitizer fix, but the broader model-stream starvation report is still credible: the OpenAI-compatible stream loops still process events synchronously without an in-loop abort check or macrotask yield, and the newer LINE/Windows comments add concrete channel-impact evidence rather than resolving the issue.

## What This Changes

The issue reports model-fetch streaming response processing monopolizing the event loop during large OpenAI-compatible streams, delaying channel traffic, heartbeats, and stop handling.

## Best Possible Solution

Confirm the bursty-stream failure with profiling, then add bounded cooperative yielding and abort checks at the OpenAI-compatible stream boundary with regressions proving timers, LINE-style webhook ack, and stop signals are serviced during large streams.

## Reproduction Assessment

No, not as a high-confidence live reproduction. The issue includes concrete logs and current source shows plausible starvation points, but this review did not reproduce the opencode-go large-context SocketError on current main.

## Solution Assessment

Unclear. Yielding and signal checks between processed stream events look like the narrow first fix, but the exact cadence and whether the sanitizer, OpenClaw stream processor, or worker isolation should own it needs profiling.
