## Summary

Keep open: current main still routes the reported OpenCode Go DeepSeek model through the OpenAI-compatible streaming loop, and that loop processes chunks synchronously with abort checked only after stream completion. Related sanitizer and gateway-control-plane trackers are relevant context, but they do not close this specific model-fetch streaming path.

## What This Changes

The issue asks OpenClaw to keep the gateway event loop responsive while processing large opencode-go/OpenAI-compatible streaming model responses.

## Best Possible Solution

Add bounded cooperative scheduling and prompt abort checks to the central OpenAI-compatible streaming path, with regression coverage that proves timers/abort handlers run during a high-volume synthetic stream.

## Reproduction Assessment

No high-confidence live repro was established in this review. Source inspection confirms the reported provider/model uses the OpenAI-compatible stream path and that the current chunk-processing loop has no cooperative yield or in-loop abort check.

## Solution Assessment

Yes for the first fix: a bounded yield and signal check in the shared streaming loop is the narrowest maintainable change. Worker-thread isolation should remain a follow-up only if targeted proof shows SDK-internal parsing still monopolizes the event loop.
