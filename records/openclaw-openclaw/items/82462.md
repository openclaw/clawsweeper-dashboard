## Summary

Keep open. Current main still processes OpenAI-compatible Responses and Completions stream events synchronously without an in-loop abort check or cooperative macrotask yield, and the issue comments add credible LINE webhook and Windows liveness impact rather than proof that the model-stream starvation report is fixed.

## What This Changes

The issue asks OpenClaw to prevent large OpenAI-compatible model streaming responses from monopolizing the gateway event loop.

## Best Possible Solution

Profile the burst-stream failure, then add bounded cooperative yielding and abort checks at the OpenAI-compatible stream boundary with regressions for timers, stop signals, and webhook-style channel responsiveness.

## Reproduction Assessment

No, not as a high-confidence live reproduction. The issue includes concrete logs and current source shows plausible starvation points, but this review did not reproduce the large-context opencode-go SocketError on current main.

## Solution Assessment

Unclear. Yielding and signal checks between processed stream events look like the narrow first fix, but the exact cadence and whether the OpenAI SDK wrapper, OpenClaw stream processor, or worker isolation should own it needs profiling.
