## Summary

Keep open. Current main has already fixed the split-frame sanitizer stall from the related provider-stream work, but this report is broader: the OpenAI-compatible stream processing loops still process provider events/chunks without an in-loop abort check or explicit macrotask yield, so the reported event-loop starvation and delayed `/stop` behavior are not proved fixed.

## What This Changes

The issue reports that OpenAI-compatible model-fetch streaming can monopolize the Node event loop during large opencode-go runs, delaying channel traffic, heartbeats, and `/stop`.

## Best Possible Solution

Confirm the current-main large-stream failure with focused profiling, then add bounded cooperative yielding and abort checks at the OpenAI-compatible stream processing boundary, with regressions proving timers and aborts are serviced during bursty streams.

## Reproduction Assessment

No, not a high-confidence live reproduction. The report has concrete logs and current source shows plausible starvation points in the model stream loops, but I did not reproduce the opencode-go large-context SocketError on current main.

## Solution Assessment

Unclear. Yielding and signal checks between processed events look like the narrowest first fix, but maintainers should profile current main because the already-merged split-frame fix covers only one related failure mode.
