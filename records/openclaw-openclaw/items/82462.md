## Summary

Keep open. Current main still has synchronous OpenAI-compatible stream-processing loops without an in-loop abort check or cooperative macrotask yield, and the merged split-frame sanitizer fix only covers parser-visible frame delivery, not timer, webhook, heartbeat, or stop-signal service during bursty streams.

## What This Changes

The issue reports event-loop starvation during large OpenAI-compatible model streams, causing SocketError, delayed channel work, and delayed stop or heartbeat handling.

## Best Possible Solution

Profile or synthesize a bursty OpenAI-compatible stream, then add bounded cooperative yielding and abort checks at the model stream boundary with regressions for timer service, stop handling, and webhook-style acknowledgement.

## Reproduction Assessment

No high-confidence live reproduction is established. The issue includes concrete logs and current-main source shows plausible starvation points, but this review did not reproduce the opencode-go large-context SocketError path.

## Solution Assessment

Unclear. Bounded yields and AbortSignal checks at the OpenAI-compatible stream boundary look like the narrow first fix, but the exact cadence and whether the sanitizer or transport processor should own it needs profiling.
