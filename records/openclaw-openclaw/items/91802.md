## Label Justifications

- `P1`: The linked report describes a real stuck session lane that suppresses WhatsApp replies and leaves queued user messages behind a dead run.
- `merge-risk: 🚨 message-delivery`: The new reset condition can free a lane while a queued user turn is already running, which can reorder or overlap subsequent message handling.
- `merge-risk: 🚨 session-state`: resetCommandLane clears active task accounting and bumps generation, so applying it to a fresh active turn can desynchronize session lane state.

## Summary

Keep open for author/maintainer follow-up: current main still has the queuedCount-only recovery gate, so this PR addresses the linked P1 stuck-session bug, but the new queueDepth branch can also reset the lane after a clean abort has already let the next queued turn start. That should be fixed before merge.

## What This Changes

The PR adds session queue depth to the stuck-session recovery lane reset gate and updates runtime/integration tests so clean aborts with queued session work release a wedged lane.

## Best Possible Solution

Keep the queueDepth-based wedged-lane recovery, but guard it so it only resets a still-wedged aborted lane and add a regression for the already-unwedged clean-abort path.

## Reproduction Assessment

Yes, source inspection provides a high-confidence reproduction shape: current main skips reset when queueDepth > 0 but lane queuedCount is 0 after a clean abort. The PR's new integration test also encodes that shape, though I did not run it because this review is read-only.

## Solution Assessment

No, not yet. The direction is right, but the current gate uses stale diagnostic queueDepth without proving the lane still belongs to the aborted slot, so the best fix needs one more ownership/race guard.
