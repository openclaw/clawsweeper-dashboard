## Summary

Keep open: current main partially supersedes the branch, but the PR still contains unshipped defaultChannels/catch-all behavior and has a concrete defaultChannels routing defect plus an unresolved product/security decision around implicit acpx activation.

## What This Changes

The PR extends ACP persistent binding validation, resolution, Feishu integration, SDK exports, docs, and acpx auto-enable behavior for Feishu, QQ Bot, and acp.defaultChannels.

## Best Possible Solution

Rebase or split onto the current generic configured-binding architecture: preserve shipped exact Feishu binding behavior, add QQ Bot through a plugin-owned channel provider/runtime seam, and decide defaultChannels/acpx auto-enable with explicit defaultAgent semantics.

## Reproduction Assessment

Yes for the PR defect: static inspection of the PR diff shows synthesized defaultChannels bindings omit cfg.acp.defaultAgent, while the current lifecycle initializes ACP with spec.acpAgentId ?? spec.agentId. Current-main source inspection also confirms defaultChannels and catch-all ACP bindings are absent.

## Solution Assessment

No. The branch contains useful source work, but as written it is not the best merge path because current main already has generic exact configured bindings, the documented quick start can misroute, and implicit external runtime activation needs maintainer approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor acp.defaultAgent in defaultChannels bindings:** `src/acp/persistent-bindings.resolve.ts:454-461`
  - body: The quick start sets acp.defaultAgent, but synthesized defaultChannels specs only set acpAgentId from per-agent runtime defaults. ensureConfiguredAcpBindingSession initializes with spec.acpAgentId ?? spec.agentId, and initializeSession normalizes that argument without consulting cfg.acp.defaultAgent, so defaultChannels can route to main instead of the requested harness when no per-agent override exists.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch does not add dependencies, workflows, lockfiles, or scripts, but it broadens external ACP CLI runtime activation.

Concerns:

- **[medium] Review implicit acpx activation from bindings:** `src/config/plugin-auto-enable.ts:368`
  - body: Auto-enabling acpx when acp.defaultChannels or any ACP binding exists can activate an external CLI harness from broad routing config instead of an explicit runtime gate, so this behavior needs approval with the default-channel and catch-all semantics.
