## Summary

Keep open for maintainer review: current main already has a newer generic configured-binding path, but this PR still proposes a new `acp.defaultChannels` config surface and its branch would regress current plugin-channel ACP bindings while also ignoring the advertised `acp.defaultAgent` quick start.

## What This Changes

The PR extends ACP persistent bindings to Feishu and QQ bot, adds `acp.defaultChannels`, wires Feishu route setup, auto-enables `acpx`, exports SDK helpers, and updates docs/tests.

## Best Possible Solution

Rebase or replace the PR on top of the current generic configured-binding registry, preserve plugin-channel ACP support, and only add `acp.defaultChannels` if maintainers want that config shortcut and it correctly honors `acp.defaultAgent`.

## Reproduction Assessment

Yes. The problems are source-reproducible on the PR head: the schema guard rejects current generic plugin-channel ACP bindings, and `defaultChannels` synthesis never reads `cfg.acp.defaultAgent`.

## Solution Assessment

No. The older channel-specific approach is not the best current-main solution; it should be adapted to the generic binding registry and the default-agent behavior needs to match the documented quick start.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

- **[P1] Keep ACP binding validation generic:** `src/config/zod-schema.agents.ts:66-76`
  - body: This guard rejects any ACP binding whose `match.channel` is not one of four literals. Current main’s configured-binding contract uses generic `ChannelId` values and the docs list BlueBubbles/iMessage ACP bindings, so merging this would break documented plugin-channel ACP configurations. Remove the hard-coded channel allowlist or delegate capability checks to the configured binding registry.
  - confidence: 0.9
- **[P2] Honor acp.defaultAgent for defaultChannels:** `src/acp/persistent-bindings.resolve.ts:449-460`
  - body: The quick start sets `acp.defaultAgent=claude`, but synthesized `defaultChannels` specs only take `acpAgentId` from per-agent runtime defaults. Since session initialization uses `spec.acpAgentId ?? spec.agentId`, configs without per-agent runtime defaults route to the owner/default agent instead of the requested ACP harness.
  - confidence: 0.93
