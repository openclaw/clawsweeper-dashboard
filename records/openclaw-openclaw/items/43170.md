## Summary

Keep open: current main has already shipped the central explicit Feishu/plugin ACP binding path, but this PR still proposes a separate channel-wide `acp.defaultChannels`/catch-all config surface that is not on main, has source-level correctness issues, and lacks real Feishu or qqbot behavior proof.

## What This Changes

This PR extends ACP persistent bindings toward Feishu and qqbot, adds channel-wide `acp.defaultChannels` catch-all synthesis, exports ACP route helpers, auto-enables `acpx`, and documents the flow.

## Best Possible Solution

Keep the shipped explicit peer-based generic configured-binding path, and separately decide whether channel-wide `defaultChannels`/catch-all routing belongs on top of the current registry with real Feishu and qqbot proof.

## Reproduction Assessment

Yes for the review findings by source inspection: the documented `defaultAgent` quick start cannot affect synthesized bindings, and the Feishu path continues after ACP setup failure in the PR head. No real channel run was provided.

## Solution Assessment

No as-is. The maintainable path is to rebuild any remaining catch-all/default-channel behavior on the current generic configured-binding seam, honor ACP agent selection, and preserve current failure handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

- **[P2] Honor defaultAgent in synthesized channel defaults:** `src/acp/persistent-bindings.resolve.ts:460`
  - body: The documented quick start sets `acp.defaultAgent`, but synthesized `defaultChannels` specs only set `acpAgentId` from per-agent runtime defaults and otherwise initialize with `agentId` (`main`). With the PR's own example, sessions run under the wrong ACP harness unless a separate runtime override exists.
  - confidence: 0.9

Current main's configured-binding path treats this as unavailable and returns, which avoids falling through with an uninitialized ACP session.
  - confidence: 0.82
