## Summary

Keep open: current main already supersedes much of the Feishu/plugin-channel ACP binding work with the generic configured-binding registry, but QQBot ACP binding support and `acp.defaultChannels` are not implemented on main. This PR cannot merge as-is because the advertised quick start ignores `acp.defaultAgent`, Feishu ACP init failures continue dispatch, and the contributor has not provided live behavior proof.

## What This Changes

The PR widens ACP persistent bindings to Feishu/QQBot, adds `acp.defaultChannels`, exports ACP route helpers, wires Feishu ACP routing, updates acpx auto-enable logic, and adds docs/tests.

## Best Possible Solution

Rework this on current main as a narrow generic-binding change: add any desired QQBot binding provider and a reviewed `acp.defaultChannels` design on top of the registry, keep Feishu's current fail-closed ACP behavior, and include live Feishu/QQBot proof.

## Reproduction Assessment

Yes for the blocking patch behavior: the PR head's quick-start `acp.defaultAgent` plus `acp.defaultChannels` path synthesizes a spec with owner `main` and no `acpAgentId`, while session initialization uses `spec.acpAgentId ?? spec.agentId`. I did not establish a live Feishu or QQBot reproduction.

## Solution Assessment

No. The branch is based on the older channel-specific ACP resolver; the maintainable path is to adapt any remaining QQBot/defaultChannels work to current main's generic configured-binding registry and prove it in a real setup.
