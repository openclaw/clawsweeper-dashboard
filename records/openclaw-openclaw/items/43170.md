## Summary

Keep open. Current main already covers part of the direction through generic configured bindings and Feishu explicit ACP routing, but it does not implement the PR’s full requested behavior, and the branch still has a defaultChannels routing bug plus a security-sensitive acpx auto-enable change that needs maintainer review before merge.

## What This Changes

The PR extends ACP persistent binding validation/resolution/docs to Feishu and QQ Bot, adds acp.defaultChannels, wires Feishu configured ACP route readiness, exports ACP route helpers, and auto-enables acpx from ACP binding/default-channel config.

## Best Possible Solution

Rebase or split the work onto the current generic configured-binding architecture: preserve the explicit Feishu path already on main, add QQ Bot through a channel-owned configured-binding provider plus inbound route readiness handling, and treat catch-all/defaultChannels plus acpx auto-enable as an explicit product/security decision with tests that honor acp.defaultAgent.

## Reproduction Assessment

Yes for the PR defect: static inspection of the PR head shows defaultChannels synthesis omits cfg.acp.defaultAgent, while lifecycle initialization uses spec.acpAgentId ?? spec.agentId. For current-main behavior, source inspection shows Feishu explicit ACP bindings exist, while defaultChannels, catch-all bindings, and QQ Bot configured-binding hooks are absent.

## Solution Assessment

No. The PR contains useful source work, but as written it is not the best merge path because current main now has a generic configured-binding registry, the defaultChannels implementation misroutes the documented quick start, and the implicit acpx activation requires maintainer security/product approval.
