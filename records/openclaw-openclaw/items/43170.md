## Summary

Keep open: current main covers much of the explicit configured ACP binding path through the generic binding registry and Feishu integration, but this PR still proposes a new implicit `acp.defaultChannels` catch-all mode that is not on main, and the PR implementation has a blocking default-agent routing bug.

## What This Changes

The PR widens ACP persistent binding support toward Feishu and QQ bot channels, adds implicit `acp.defaultChannels` binding synthesis, wires Feishu ACP route readiness, exports ACP helpers, and updates docs/tests.

## Best Possible Solution

Port any still-desired implicit default-channel behavior onto the current generic configured-binding architecture, with explicit maintainer approval for the new config semantics and correct `acp.defaultAgent` handling.

## Reproduction Assessment

Yes for the review finding: the PR source shows `acp.defaultChannels` synthesis selecting `main` without `acp.defaultAgent`, and the lifecycle path then initializes ACP with that synthesized agent. Feature validation itself is not a current-main bug reproduction.

## Solution Assessment

No. The PR is not the best merge shape as written because current main already has a newer generic binding architecture, and the remaining implicit channel mode needs to be reworked there while honoring `acp.defaultAgent`.

  - body: The quick-start config uses `acp.defaultAgent` with `acp.defaultChannels`, but this synthesized spec sets only `agentId` from `main` and `acpAgentId` from per-agent runtime defaults. `ensureConfiguredAcpBindingSession` then initializes ACP with `spec.acpAgentId ?? spec.agentId`, and the manager does not read `cfg.acp.defaultAgent`, so users without a per-agent runtime default are routed to `main` instead of the configured ACP harness.
