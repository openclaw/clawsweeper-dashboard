## Summary

Keep open: current main already covers exact Feishu and plugin-channel ACP bindings through the generic configured-binding registry, but this PR still proposes a unique `acp.defaultChannels` catch-all config surface that is not separately tracked. It also needs real behavior proof and correctness fixes before it is mergeable.

## What This Changes

The PR extends ACP persistent bindings to Feishu and QQ Bot, adds `acp.defaultChannels`, wires Feishu route readiness, exports ACP route helpers, and updates docs/tests.

## Best Possible Solution

Keep the shipped generic plugin-channel ACP binding path, and only carry this PR forward if maintainers want the new catch-all/defaultChannels config surface with corrected agent selection, failure handling, docs, tests, and live channel proof.

## Reproduction Assessment

Not applicable as a user bug; this is a feature PR. The blocking findings are source-reproducible from PR head: synthesized `defaultChannels` ignores `acp.defaultAgent`, and Feishu continues after ACP setup failure.

## Solution Assessment

No, not as-is. Current main has the cleaner generic exact-peer binding seam, while the remaining catch-all/defaultChannels design needs maintainer policy approval plus correctness fixes and live proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor acp.defaultAgent in defaultChannels sessions:** `src/acp/persistent-bindings.resolve.ts:449-460`
  - body: The documented quick start sets `acp.defaultAgent`, but synthesized `defaultChannels` specs only read per-agent runtime defaults and otherwise initialize with the owner agent, typically `main`. `ensureConfiguredAcpBindingSession` uses `spec.acpAgentId ?? spec.agentId`, so this can start the wrong ACP harness or fail when the user followed the documented config.
  - confidence: 0.92
- **[P2] Stop Feishu dispatch after ACP setup failure:** `extensions/feishu/src/bot.ts:1226-1228`
  - body: After replacing the route with a configured ACP route, this path only logs readiness failure and continues dispatching. Current main sends a Feishu error reply and returns, so merging this would regress configured ACP bindings to fail open when the ACP backend or runtime is unavailable.
  - confidence: 0.9
