## Summary

Keep open. Current main has superseded the PR's Feishu/type-widening foundation with the generic configured-binding registry and Feishu ACP route readiness, but it still lacks the PR's remaining central behavior for `acp.defaultChannels`, catch-all ACP bindings, QQ Bot configured ACP routing, and acpx auto-enable from bindings; the PR branch also still needs rebase/split and security review before it is mergeable.

## What This Changes

The PR extends ACP persistent binding validation/resolution/docs to Feishu and QQ Bot, adds `acp.defaultChannels`, wires Feishu configured ACP route readiness, exports ACP route helpers, and auto-enables acpx from ACP binding/default-channel config.

## Best Possible Solution

Keep this PR open for maintainer follow-up as a rebase or split: preserve the current generic configured-binding registry and shipped Feishu path, port QQ Bot through a QQ Bot configured-binding provider plus inbound `resolveConfiguredBindingRoute` / `ensureConfiguredBindingRouteReady` handling via `openclaw/plugin-sdk/conversation-runtime`, and make `acp.defaultChannels`, catch-all bindings, and acpx auto-enable an explicit product/security decision. If defaultChannels remains in scope, the synthesized binding must honor the documented ACP default-agent semantics.

## Reproduction Assessment

Yes for the PR bug: static inspection of the PR patch shows synthesized `acp.defaultChannels` specs only set `acpAgentId` from per-agent runtime defaults while existing lifecycle initialization falls back to `spec.agentId`, so the documented `acp.defaultAgent` quick start can choose the wrong harness. For current-main coverage, source and tests show Feishu explicit ACP bindings work while QQ Bot/defaultChannels/catch-all paths are absent.

## Solution Assessment

No. The PR direction partially matches real remaining work, but the branch is no longer the best direct merge because current main has a generic configured-binding architecture, the PR's broad default/catch-all behavior needs maintainer security/product review, and at least one defaultChannels routing bug remains.
