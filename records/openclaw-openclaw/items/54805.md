The branch adds a typed `plugin_updated` plugin lifecycle hook, dispatches it after `openclaw plugins update`, documents it, adds updater/CLI/hook tests, and increases a formatter script buffer.

## Best Possible Solution

Rebase onto current plugin hook/update infrastructure, add a git-aware typed event in `hook-types.ts`, dispatch only for actual updated plugin IDs with scoped runtime loading, and land after maintainer API/security review plus real update proof.

## Solution Assessment

No, not as written. The feature should be implemented through current `hook-types.ts`, git-aware event typing, managed install-record update flow, and scoped runtime loading.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Load only updated plugin runtimes:** `src/cli/plugins-update-command.ts:197`
  - body: The new dispatch path calls `ensurePluginRegistryLoaded({ scope: "all" })` before invoking targeted `plugin_updated` handlers. During a single-plugin update this can load and execute unrelated plugin runtime code; pass the updated plugin IDs through the scoped loader before invoking handlers.
  - confidence: 0.84
- **[P2] Register plugin_updated in hook-types:** `src/plugins/types.ts:1425-1455`
  - body: Current main exports hook names, event/context types, and handler maps from `src/plugins/hook-types.ts`, with `types.ts` re-exporting that surface. Adding `plugin_updated` only to the old `types.ts` hook block will not register the hook correctly after rebase.
  - confidence: 0.82
- **[P2] Include git in update hook events:** `src/plugins/types.ts:1951`
  - body: Current `updateNpmInstalledPlugins` supports `record.source === "git"`, and this PR passes `record.source` into the lifecycle event, but the public event type excludes `"git"`. Add git to the source union and cover git updates so valid git-installed plugin updates can dispatch a truthful event.
  - confidence: 0.78
