## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Load only the updated plugin before dispatch:** `src/cli/plugins-update-command.ts:197`
  - body: `ensurePluginRegistryLoaded({ scope: "all" })` makes `openclaw plugins update alpha` initialize every effective plugin runtime before dispatching one plugin's `plugin_updated` hook. Current main resolves `scope: "all"` to all effective plugin ids and calls `loadOpenClawPlugins`, so an unrelated installed plugin can run or fail during another plugin's update. Load only the updated plugin ids or add a registry seam that resolves the updated plugin's handler without materializing every runtime.
  - confidence: 0.88
- **[P1] Port the hook contract to hook-types.ts:** `src/plugins/types.ts:1429-1458`
  - body: Current main no longer owns `PluginHookName`, `PLUGIN_HOOK_NAMES`, or `PluginHookHandlerMap` in `src/plugins/types.ts`; `types.ts` imports them from `src/plugins/hook-types.ts` and says hook contracts live there. Adding `plugin_updated` to the old surface will not update the current validator/registry contract, so `api.on("plugin_updated", ...)` will not be accepted after a real rebase unless the event, name list, and handler map move to `hook-types.ts`.
  - confidence: 0.9
- **[P2] Cover current git plugin updates in the event contract:** `src/plugins/types.ts:1948`
  - body: The new event source union only allows `npm`, `marketplace`, and `clawhub`, but current main's updater also handles `record.source === "git"` and can update git-backed plugin installs. A current-main port that forwards `record.source` will either fail typing or skip lifecycle hooks for git plugins, so include `git` with appropriate version/commit fields or explicitly gate it with tests and docs.
  - confidence: 0.78
