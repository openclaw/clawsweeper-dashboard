## Summary

Keep open: current main still lacks `plugin_updated`, but this PR is not merge-ready because it is stale against the active hook contract surface, omits current git update support, broadens plugin runtime loading, and lacks real behavior proof.

## What This Changes

The PR adds a `plugin_updated` lifecycle hook across plugin contracts, hook runner, plugin update pipeline, CLI dispatch, tests/docs, plus a formatter `maxBuffer` adjustment.

## Best Possible Solution

Rebase onto current `hook-types.ts` and plugin update surfaces, emit targeted events for actual npm/marketplace/ClawHub/git updates, load only updated plugin runtimes, and add focused tests/docs plus redacted real behavior proof.

## Reproduction Assessment

Yes for review purposes. Source and PR-diff inspection show current main lacks `plugin_updated`, while the PR diff shows stale hook placement, missing git-source coverage, and broad runtime loading without needing a live plugin install.

## Solution Assessment

No, not as written. The feature direction is plausible, but the implementation should be rebased onto `hook-types.ts`, include git update events, preserve current update persistence, and use scoped runtime loading.

  - body: The new dispatch path loads the full plugin registry before targeted `plugin_updated` handlers. During a single-plugin update this can execute unrelated plugin runtime code, so load only the updated plugin IDs before dispatching.
  - confidence: 0.84

  - confidence: 0.82
