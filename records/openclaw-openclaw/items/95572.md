## Summary

Keep open for merge consideration: current main still silently truncates extra colon segments in bind specs, while this PR rejects that malformed input at the shared parser with focused regression coverage. The patch looks correct; the only notable pre-merge concern is the intentional CLI compatibility change from silent truncation to fail-closed validation.

## What This Changes

The PR makes `parseBindingSpecs` reject bind specs with extra colon-delimited account segments and adds Matrix parser regression coverage.

## Best Possible Solution

Land the shared parser rejection and regression coverage once maintainers accept the malformed-spec fail-closed behavior and exact-head gates are green.

## Reproduction Assessment

Yes. Source inspection of current main shows `parseBindingSpecs` using `split(":", 2)`, which truncates `matrix:work:extra` to account `work`; the PR body also includes copied before/after terminal output from a harness importing the real parser.

## Solution Assessment

Yes. The shared `parseBindingSpecs` function is the right owner because `agents bind`, `agents unbind`, and `agents add --bind` all consume it before config writes; validating later would duplicate command logic or rely on account-id canonicalization that intentionally rewrites invalid characters.
