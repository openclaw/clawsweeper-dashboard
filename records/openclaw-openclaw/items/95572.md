## Summary

Keep open: current main and the latest release still silently truncate extra-colon agent bind specs, while this PR implements the narrow shared-parser rejection with focused regression coverage and no blocking code findings.

## What This Changes

The PR changes the agent binding parser to reject `<channel>:<account>` specs containing extra colon-separated segments and adds Matrix parser regression coverage.

## Best Possible Solution

Land the shared parser rejection with its regression tests once required checks pass and maintainers accept the invalid-input compatibility tightening.

## Reproduction Assessment

Yes, source-reproducible without running the CLI: current main splits bind specs with `split(":", 2)`, which drops a third segment before binding construction. The PR body also includes before/after terminal output from a parser harness.

## Solution Assessment

Yes. Rejecting extra segments inside shared `parseBindingSpecs` is the best fix because `agents bind`, `agents unbind`, and `agents add --bind` all consume that parser before config mutation, and account ids do not support literal colons.
