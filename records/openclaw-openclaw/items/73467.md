## Summary

Keep open. Current main has useful mitigations for readiness diagnostics, restart deferral, task reconciliation, plugin metadata caching, and logs-follow fallback, but it does not prove the original CPU-hot Gateway event-loop starvation under orchestration load is gone.

## What This Changes

The issue reports CPU-hot Gateway main-thread stalls on OpenClaw 2026.4.26, with loopback /readyz timeouts, delayed Discord/WebUI control-plane traffic, and reloads deferred behind stale active run accounting.

## Best Possible Solution

Keep this as the current Gateway reliability umbrella, use #42026 for the long-term control-plane/runtime split, and close only after loaded current-main profiling or narrower fixes prove the stall and stale task accounting are gone.

## Reproduction Assessment

Unclear. The production report is detailed and credible, but a high-confidence current-main reproduction still needs a loaded Gateway with Discord/WebUI traffic, memory-lancedb or equivalent plugin hooks, frequent /readyz probes, and CPU/event-loop-delay profiles during a stall.

## Solution Assessment

Not applicable as a patch review. The best path is profile-driven reduction or offloading of hot paths plus validation of reload/task accounting, not closing based on diagnostics and fallback behavior alone.
