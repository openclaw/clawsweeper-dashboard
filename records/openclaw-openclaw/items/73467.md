## Summary

Keep open. Current main has important mitigations for readiness diagnostics, bounded restart deferral, active task blocker logging, log-follow fallback, plugin-tool caching, and session-list hot paths, but I did not find evidence that the reported CPU-hot Gateway event-loop starvation is fixed or reproducible on current main.

## What This Changes

The issue reports v2026.4.26 Gateway main-thread stalls under orchestration load, causing loopback `/readyz` timeouts, delayed Discord/WebUI control traffic, and reload/restart deferral behind stale active-run accounting.

## Best Possible Solution

Keep this as a Gateway reliability umbrella until maintainers profile current main under the reported orchestration load, then split confirmed CPU hot paths or lifecycle-accounting defects into narrow follow-ups while leaving the full control-plane/runtime split to #42026.

## Reproduction Assessment

No, not at high confidence. The v2026.4.26 production packets are credible and current source shows `/readyz` cannot answer while the Gateway event loop is CPU-blocked, but I did not establish a current-main workload that reproduces the same CPU-hot stall.

## Solution Assessment

Not applicable as a patch review. The current mitigations are useful but not terminal; the maintainable path is profile-driven removal, bounding, or offloading of confirmed hot paths plus validation of restart/task accounting.
