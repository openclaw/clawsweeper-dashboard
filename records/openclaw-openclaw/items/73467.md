## Summary

Keep open. Current main has meaningful mitigations for readiness diagnostics, restart deferral, task reconciliation, `logs --follow` fallback, and plugin/tool hot-path caching, but the central report still is not proven fixed because `/readyz` is served on the same Gateway event loop and no current-main CPU profile shows the reported starvation is gone.

## What This Changes

The issue reports Gateway main-thread stalls under orchestration load, causing loopback `/readyz` timeouts, delayed Discord/WebUI control traffic, `logs --follow` drops, and reload deferral behind active run accounting.

## Best Possible Solution

Keep this as a Gateway reliability umbrella until maintainers capture current-main event-loop and V8 CPU profiles under the reported Discord/WebUI/memory workload, then split confirmed hot paths or lifecycle-accounting defects into narrower follow-ups.

## Reproduction Assessment

Unclear for current main. The report gives a credible production reproduction shape, but high-confidence reproduction still needs a loaded current-main Gateway with Discord/WebUI traffic, memory/plugin hooks, frequent `/readyz` probes, and CPU or event-loop-delay profiles during a stall.

## Solution Assessment

Not applicable as a patch review. The current mitigations are useful but not terminal; the maintainable path is profile-driven bounding, caching, yielding, or offloading of confirmed hot paths plus validation of restart/task accounting.
