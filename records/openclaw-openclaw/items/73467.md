## Summary

Keep open. Current main has meaningful mitigations for readiness diagnostics, restart deferral, task reconciliation, logs fallback, hook timeouts, and plugin-tool hot paths, but those do not prove the original CPU-hot Gateway event-loop starvation under orchestration load is gone.

## What This Changes

The issue asks OpenClaw to keep Gateway control-plane probes, Discord/WebUI operations, reload/restart accounting, plugin hooks, and live logs responsive under multi-agent orchestration load.

## Best Possible Solution

Keep this as the Gateway reliability umbrella until maintainers have profile-backed proof on current main, then split or close narrower follow-ups for hot paths, task/reload accounting, plugin hook behavior, operator diagnostics, and control-plane isolation.

Unclear for current main. The production evidence is detailed and credible, but high-confidence reproduction still needs a loaded current-main Gateway with Discord/WebUI traffic, memory/plugin hooks, frequent /readyz probes, and CPU or event-loop-delay profiles during a stall.
