## Summary

Keep open. Current main has useful mitigations for several reported symptoms, including event-loop readiness diagnostics, bounded default restart deferral, inspectable task-registry reconciliation, and a local log fallback for loopback `logs --follow` failures, but the core CPU-hot Gateway event-loop starvation is not proven fixed and the profile-backed related hot-path report #74860 is still open.

## What This Changes

The issue requests Gateway control-plane responsiveness under orchestration load, bounded reload/restart blockers, inspectable active run accounting, and diagnostics/offloading for CPU-hot plugin or agent work.

## Best Possible Solution

Keep this as the umbrella Gateway reliability tracker until maintainers either land and validate the profile-backed hot-path work from #74860 against a loaded Discord/WebUI/memory-style setup, or split the reload/task-accounting and control-plane observability leftovers into narrower canonical issues. The desired end state is a responsive control plane under heavy agent/plugin work, bounded and inspectable active blockers, and live event-loop/CPU-profile proof.

## Reproduction Assessment

Unclear for current main. The report has strong production packet/log evidence and #74860 adds a later profile-backed fingerprint, but a high-confidence reproduction still needs a loaded multi-agent Gateway with Discord/WebUI traffic, memory/plugin hooks, 1-2s `/readyz` probes, and captured event-loop delay or CPU profiles.

## Solution Assessment

Not applicable as a patch review. The current-main mitigations are useful but partial; the maintainable fix path is targeted hot-path reduction or offloading plus reload/task-accounting validation, not closing the umbrella before live-load proof exists.
