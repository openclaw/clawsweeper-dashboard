## Summary

Keep open. Current main has several real mitigations for the reported 2026.4.26 failure shape, including `/readyz` event-loop diagnostics, bounded restart deferral, task-accounting reconciliation, and a local `logs` fallback, but I did not find proof that the CPU-hot Gateway event-loop starvation under loaded orchestration is fixed on current main.

## What This Changes

The issue asks OpenClaw to prevent Gateway main-thread stalls under orchestration load that caused loopback `/readyz` timeouts, delayed Discord/WebUI/control-plane traffic, and reloads deferred behind active-run accounting.

## Best Possible Solution

Keep this as the Gateway reliability umbrella until maintainers validate current main under a loaded Discord/WebUI/plugin setup with `/readyz` probes and event-loop/CPU profiles, then split any remaining failures into narrower trackers for reload/task accounting, plugin hot paths, ACP maintenance, and longer-term control-plane/runtime isolation.

Unclear for current main. The report has strong production evidence for 2026.4.26, but a high-confidence current-main reproduction still needs a loaded multi-agent Gateway with Discord/WebUI traffic, memory-style hooks, frequent `/readyz` probes, and captured event-loop delay plus CPU profiles.
