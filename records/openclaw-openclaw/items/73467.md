## Summary

Keep open. Current main has meaningful mitigations for readiness diagnostics, restart deferral, task reconciliation, logs-follow fallback, plugin-tool hot paths, and liveness recording, but that does not prove the original CPU-hot Gateway event-loop starvation under orchestration load is gone.

## What This Changes

The issue asks OpenClaw to keep Gateway probes, Discord/WebUI control traffic, reload/restart accounting, plugin hooks, and live logs responsive during multi-agent orchestration load.

## Best Possible Solution

Keep this as the Gateway reliability umbrella until maintainers have profile-backed current-main proof, then split or close narrower follow-ups for hot paths, task/reload accounting, plugin hook behavior, operator diagnostics, and control-plane isolation.

## Reproduction Assessment

Unclear. The production evidence is detailed and credible, but high-confidence reproduction on current main still needs a loaded Gateway with Discord/WebUI traffic, memory/plugin hooks, frequent /readyz probes, and CPU or event-loop-delay profiles during any stall.

## Solution Assessment

Not applicable as a patch review. The maintainable path is profile-driven reduction or offloading of confirmed hot paths plus validation of reload/task accounting, not closing based on partial diagnostics and fallback behavior alone.
