## Summary

Keep open. Current main has meaningful mitigations for the reported symptoms, including event-loop readiness diagnostics, bounded default restart deferral, inspectable task reconciliation, and local log fallback, but I did not find high-confidence evidence that the CPU-hot Gateway event-loop starvation itself is fixed. The profile-backed related hot-path report #74860 is still open in the provided context, so closing this umbrella would be premature.

## What This Changes

The issue requests Gateway reliability hardening so `/readyz`, Discord/WebUI control traffic, reload deferral, plugin hooks, and `openclaw logs --follow` remain responsive or inspectable under orchestration load.

## Best Possible Solution

Keep this as the Gateway reliability umbrella until maintainers either land and validate the profile-backed hot-path work from #74860 against a loaded Discord/WebUI/memory-style setup, or split the remaining reload/task-accounting and observability gaps into narrower canonical issues. The end state should be a responsive control plane under heavy agent/plugin work, bounded and inspectable active blockers, and profile-backed proof that CPU-hot paths no longer starve probes or operator RPCs.

## Reproduction Assessment

Unclear for current main. The report has strong production packet/log evidence and #74860 adds a later profile-backed fingerprint, but a high-confidence reproduction still needs a loaded multi-agent Gateway with Discord/WebUI traffic, memory/plugin hooks, 1-2s `/readyz` probes, and captured event-loop delay or CPU profiles on current main.

## Solution Assessment

Not applicable as a patch review. The current mitigations are useful but partial; the maintainable fix path is profile-driven hot-path reduction or offloading plus reload/task-accounting validation, not closing the umbrella before live-load proof exists.
