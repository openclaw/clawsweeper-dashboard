## Summary

Keep open. Current main has meaningful mitigations for readiness diagnostics, restart deferral, stale task reconciliation, client startup waits, and local log fallback, but it does not prove the reported CPU-hot Gateway event-loop starvation is gone; `/readyz` still runs on the same Gateway event loop and the broader control-plane/runtime split remains separately tracked.

## What This Changes

Issue reports Gateway main-thread stalls under orchestration load, causing loopback `/readyz` timeouts, delayed Discord/WebUI control-plane traffic, `logs --follow` drops, and reload deferral behind active run accounting.

## Best Possible Solution

Keep this as a Gateway reliability umbrella until maintainers capture current-main CPU/event-loop profiles under the reported load, then split confirmed hot paths or lifecycle-accounting defects into narrow follow-ups.

## Reproduction Assessment

Unclear for current main. The report gives a credible production reproduction shape, but high-confidence reproduction still needs a loaded current-main Gateway with Discord/WebUI traffic, memory/plugin hooks, frequent `/readyz` probes, and CPU or event-loop-delay profiles during a stall.

## Solution Assessment

Not applicable as a patch review. The current mitigations are useful but not terminal; the maintainable path is profile-driven removal, bounding, or offloading of confirmed hot paths plus validation of restart/task accounting.
