## Summary

Keep open. Current main contains meaningful mitigations for several reported symptoms, but the central CPU-hot Gateway event-loop starvation under orchestration load is not proven fixed: readiness diagnostics are still served in-process, plugin/tool and memory paths are reduced or bounded rather than isolated, and there is no current-main loaded reproduction or profile showing the stall is gone.

## What This Changes

The issue requests investigation and hardening for Gateway event-loop stalls under orchestration load, including responsive readiness probes, bounded reload/restart deferral, active-run inspection/reaping, plugin hook limits, and resilient live log monitoring.

## Best Possible Solution

Keep this as the Gateway reliability umbrella until maintainers either validate current main under a loaded multi-agent setup with CPU/event-loop profiles, or split the remaining work into narrower canonical issues for event-loop hot paths, task/reload accounting, plugin hook behavior, operator diagnostics, and longer-term control-plane isolation.

## Reproduction Assessment

Unclear for current main. The production report is detailed and credible, but a high-confidence reproduction still needs a loaded current-main Gateway with Discord/WebUI traffic, memory/plugin hooks, frequent /readyz probes, and captured CPU or event-loop-delay profiles during the stall window.

## Solution Assessment

Not applicable as a patch review. The current mitigations are useful but partial; the maintainable fix path is profile-driven hot-path reduction or offloading plus reload/task-accounting validation, not closing based on diagnostics and fallbacks alone.
