## Summary

Keep open. Current main has meaningful mitigations for diagnostics, bounded reload deferral, task-blocker logging, `logs --follow` recovery, and several related plugin/tool hot paths, but the central CPU-hot Gateway event-loop starvation under orchestration load is not proven fixed on current main.

## What This Changes

The issue reports Gateway event-loop stalls on 2026.4.26 where loopback `/readyz`, Discord/status traffic, WebUI RPCs, log following, and reload application were delayed behind active orchestration work.

## Best Possible Solution

Keep this as a Gateway reliability umbrella until maintainers capture current-main profiles under the reported multi-agent Discord/WebUI/memory load, then split confirmed hot paths or lifecycle-accounting defects into narrow follow-ups.

## Reproduction Assessment

No high-confidence exact current-main reproduction is established. The report has credible production evidence, and source inspection confirms in-process `/readyz` cannot answer while the Gateway event loop is blocked, but the current-main workload that still causes a CPU-hot loop is not identified.

## Solution Assessment

Not applicable as a patch review. The current mitigations are useful but not terminal; the maintainable path is profile-driven removal, bounding, or offloading of confirmed hot paths plus validation of restart/task accounting.
