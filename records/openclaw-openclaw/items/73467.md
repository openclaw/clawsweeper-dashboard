## Summary

Keep open. Current main has meaningful mitigations for the reported symptoms, including in-process event-loop readiness diagnostics, a default five-minute restart deferral cap, and local log-file fallback for loopback `logs --follow` failures, but the central CPU-hot Gateway event-loop starvation is not proven fixed and the later profile-backed hot-path report #74860 remains open.

## What This Changes

The issue reports CPU-hot Gateway event-loop stalls on OpenClaw 2026.4.26 causing local `/readyz` timeouts, delayed Discord/WebUI control traffic, `logs --follow` disconnects, and reload/restart deferral behind active run accounting.

## Best Possible Solution

Keep this as the umbrella Gateway reliability tracker until maintainers either land and validate #74860 against a loaded Discord/WebUI/memory-style setup or split the remaining reload/task-accounting observability work into a dedicated canonical issue. The end state should include a responsive control plane under loaded agent/plugin work, inspectable active blockers, bounded stale accounting, and live CPU-profile/event-loop-delay proof.

## Reproduction Assessment

Unclear for current main. The issue has strong production-style packet/log evidence and #74860 adds a later profile-backed reproduction fingerprint, but a high-confidence reproduction still requires a loaded multi-agent Gateway with Discord/WebUI traffic, memory/plugin hooks, 1-2s `/readyz` probes, and captured event-loop delay or CPU profiles.

## Solution Assessment

Not applicable as a patch review. The current-main mitigations are useful but partial; the maintainable fix path is targeted hot-path reduction/offloading plus reload/task-accounting validation, not closing the umbrella before live-load proof exists.
