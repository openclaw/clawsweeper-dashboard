## Summary

Keep open. Current main has useful mitigations, but it does not prove the reported CPU-hot Gateway event-loop starvation is gone; `/readyz` is still served by the same Gateway process, and the broader control-plane/runtime isolation work remains open.

## What This Changes

The issue reports v2026.4.26 Gateway stalls under multi-agent Discord/WebUI load, loopback `/readyz` timeouts, `logs --follow` disconnects, and reload/restart deferral behind active run accounting.

## Best Possible Solution

Keep this as a Gateway reliability umbrella until maintainers capture current-main profiles under the reported load, then split confirmed hot paths or lifecycle-accounting defects into narrow follow-ups.
