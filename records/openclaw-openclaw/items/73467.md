## Summary

Keep open. Current main has meaningful mitigations for readiness diagnostics, bounded restart deferral, task reconciliation, logs fallback, and plugin hot paths, but the original CPU-hot Gateway event-loop starvation under loaded orchestration is not proven fixed.

## What This Changes

The issue reports Gateway main-thread stalls on 2026.4.26 where loopback `/readyz`, Discord/WebUI control traffic, and config reload were delayed behind active orchestration work.

## Best Possible Solution

Keep this as the Gateway reliability umbrella until maintainers profile a loaded current-main deployment, then split or close narrower follow-ups for any confirmed hot path, task-accounting gap, or control-plane isolation work.

## Reproduction Assessment

Unclear. The production evidence is detailed, but high-confidence reproduction still requires a current-main Gateway under similar Discord/WebUI/memory-lancedb orchestration load with frequent `/readyz` probes and CPU/event-loop profiles during a stall.

## Solution Assessment

Not applicable as a patch review. The maintainable path is profile-driven removal or offloading of confirmed hot paths plus validation of reload/task accounting, not closing from diagnostics and fallback behavior alone.
