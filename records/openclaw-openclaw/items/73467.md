## Summary

Keep open. Current main has useful mitigations for event-loop diagnostics, bounded restart deferral, active task blocker logging, client startup waits, and local log fallback, but it does not prove the reported CPU-hot Gateway event-loop starvation is gone; `/readyz` is still served in the same Gateway process and the broader control-plane/runtime isolation work remains open in #42026.

## What This Changes

The issue reports v2026.4.26 Gateway event-loop stalls under multi-agent Discord/WebUI load, loopback `/readyz` timeouts, `logs --follow` disconnects, and reload deferral behind stale active task accounting.

## Best Possible Solution

Keep this as a Gateway reliability umbrella until maintainers capture current-main profiles under the reported load, then split confirmed hot paths or lifecycle-accounting defects into narrow follow-ups.

## Reproduction Assessment

Unclear. The report gives a credible v2026.4.26 production reproduction shape, but I did not establish a current-main workload that drives the same CPU-hot loop; source inspection only proves `/readyz` cannot respond while the Node event loop is blocked.

## Solution Assessment

Not applicable as a patch review. The current mitigations are useful but not terminal; the maintainable path is profile-driven removal, bounding, or offloading of confirmed hot paths plus validation of restart/task accounting.
