## Summary

Keep open. The requested static attribution feature is plausible, but the PR is not mergeable as written because it forwards `params.user` to every provider payload, lacks focused coverage, and still has no contributor real-behavior proof.

## What This Changes

The PR adds a stream wrapper that normalizes configured `params.user` and injects it into provider request payloads through `streamWithPayloadPatch`.

## Best Possible Solution

Land a narrow `params.user` passthrough only for explicitly compatible request APIs, with supported/unsupported payload-shape tests, and keep dynamic session metadata design in https://github.com/openclaw/openclaw/issues/63901.

## Reproduction Assessment

Yes for the PR defect by source inspection: the added wrapper is applied whenever `effectiveExtraParams.user` is present, and `streamWithPayloadPatch` mutates any object payload. No live provider call was run in this read-only review.

## Solution Assessment

No. The static attribution capability is reasonable, but this implementation needs a supported-API guard, focused payload tests, and contributor real behavior proof before it is mergeable.
