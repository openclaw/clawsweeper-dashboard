## Summary

Keep open. The PR targets a real missing static `params.user` passthrough on current main, but the proposed wrapper is not mergeable as-is because it patches every provider payload, lacks after-fix real request proof, and still needs maintainer confirmation of the supported API/privacy scope.

## What This Changes

This PR adds a post-plugin stream wrapper that reads configured `params.user` and patches provider request payloads with a top-level `user` field.

## Best Possible Solution

Land a narrow static `params.user` passthrough only for explicitly compatible request APIs, with supported and unsupported payload-shape tests, while leaving session-derived `trace`/`user` metadata design to https://github.com/openclaw/openclaw/issues/63901.

## Reproduction Assessment

Yes for the PR defect by source inspection: the added wrapper runs whenever configured `params.user` normalizes to a string, and the payload helper mutates any object payload. No live provider call was run in this read-only review.

## Solution Assessment

No. Static attribution is reasonable, but this implementation needs a supported-API guard, focused payload tests, contributor real behavior proof, and maintainer API/privacy confirmation before merge.
