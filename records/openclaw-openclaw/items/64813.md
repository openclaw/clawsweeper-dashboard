## Summary

This PR should stay open: current main still lacks the requested `params.user` passthrough, but the branch has a blocking provider-compatibility/privacy issue and no real behavior proof. The broader provider metadata design remains open at https://github.com/openclaw/openclaw/issues/63901.

## What This Changes

The PR adds a post-plugin embedded-runner wrapper that reads configured `params.user` and writes it into the top-level provider request payload.

## Best Possible Solution

Land a narrow `params.user` passthrough only for explicitly compatible request APIs, with supported and unsupported payload tests, contributor-provided real request proof, and alignment with the broader metadata decision in https://github.com/openclaw/openclaw/issues/63901.

## Reproduction Assessment

Yes for the PR defect by source inspection: when `params.user` normalizes to a string, the added wrapper patches object payloads through `streamWithPayloadPatch` without checking `model.api`. No live provider request proof is present.

## Solution Assessment

No. The useful part should be implemented as a supported-API-scoped payload wrapper with tests and real provider proof, not as an unconditional cross-provider body mutation.
