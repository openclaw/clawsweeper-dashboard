## Summary

Keep open. The PR is not merge-ready because it adds an unconditional request-body mutation for `params.user`, while current main already has a scoped static `extra_body` pass-through and the broader metadata/privacy contract remains unsettled.

## What This Changes

The PR adds a post-plugin embedded-runner wrapper that forwards configured `params.user` into provider request payloads.

## Best Possible Solution

Land a direct `params.user` shortcut only if maintainers choose that config surface; otherwise rely on `extra_body` for static OpenAI-compatible payload fields and keep dynamic attribution in the canonical metadata issue.

## Reproduction Assessment

Yes for the PR defect by source inspection: when `params.user` normalizes to a string, the added wrapper patches object payloads through `streamWithPayloadPatch` without checking `model.api`. No live provider request proof is present.

## Solution Assessment

No. The useful behavior should either use the existing `extra_body` static pass-through or land as a supported-API-scoped shortcut after maintainer API/privacy agreement, not as an unconditional cross-provider body mutation.
