## Summary

Keep open: the change is not mergeable as-is because it lacks the provider/API guard used by adjacent request-body wrappers, has no contributor real-request proof, and overlaps an unresolved provider metadata/privacy design.

## What This Changes

The branch adds a post-plugin embedded-runner wrapper that reads configured `params.user` and injects it as a top-level `user` field in request payloads.

## Best Possible Solution

Land a narrow, explicitly supported `params.user` passthrough only after the provider/API contract is confirmed, with supported and unsupported payload tests, docs, and privacy-safe alignment with the broader metadata request in https://github.com/openclaw/openclaw/issues/63901.

## Reproduction Assessment

Yes for the PR defect by source inspection: configuring `params.user` makes the added wrapper patch every object payload through `streamWithPayloadPatch` without checking `model.api`. I did not run a live provider request in this read-only review.

## Solution Assessment

No. Static attribution can be useful, but this implementation needs a supported-API guard, focused payload tests, contributor real behavior proof, and maintainer API/privacy confirmation before merge.

## Review Findings

Overall correctness: patch is incorrect
