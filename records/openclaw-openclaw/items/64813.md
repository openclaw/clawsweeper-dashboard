## Summary

Keep open: the PR adds a useful static attribution path, but the branch still forwards `params.user` into every provider payload without an API-family guard, and it has no after-fix real provider proof.

## What This Changes

The PR adds a stream wrapper that normalizes configured `params.user` and injects it into provider request payloads through `streamWithPayloadPatch`.

## Best Possible Solution

Land a narrow `params.user` passthrough only for explicitly compatible request APIs, with supported and unsupported payload-shape tests, while keeping dynamic session metadata design in https://github.com/openclaw/openclaw/issues/63901.

## Reproduction Assessment

Yes for the PR defect by source inspection: the added wrapper applies whenever `effectiveExtraParams.user` is present, and `streamWithPayloadPatch` mutates any object payload. No live provider call was run in this read-only review.

## Solution Assessment

No. Static attribution is reasonable, but this implementation needs a supported-API guard, focused payload tests, and contributor real behavior proof before it is mergeable.
