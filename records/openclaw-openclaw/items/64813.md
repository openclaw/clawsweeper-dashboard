## Summary

Keep open. The requested static attribution feature is plausible, but this PR patches `user` into every provider payload without an API guard, lacks real request proof, and overlaps the broader provider metadata discussion at https://github.com/openclaw/openclaw/issues/63901.

## What This Changes

The PR adds a post-plugin embedded-runner wrapper that reads configured `params.user` and writes it as a top-level `user` field into API request payloads.

## Best Possible Solution

Land a narrow `params.user` passthrough only for explicitly compatible request APIs, with supported and unsupported payload tests, docs, and privacy-safe separation from the broader session metadata work in https://github.com/openclaw/openclaw/issues/63901.

## Reproduction Assessment

Yes for the PR defect by source inspection: when configured `params.user` normalizes to a string, the added wrapper would patch any object payload through `streamWithPayloadPatch` without checking `model.api`. No live provider call was run in this read-only review.

## Solution Assessment

No. Static attribution can be useful, but this diff needs a supported-API guard, focused payload tests, real behavior proof, and maintainer API/privacy confirmation before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93
