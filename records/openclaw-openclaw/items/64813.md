## Summary

Keep open. The static `user` passthrough is a reasonable OpenAI/OpenRouter feature, but this PR applies it to every provider payload once configured, while the adjacent payload-patch pattern is API-family guarded; that can break unsupported providers and leak an attribution identifier.

## What This Changes

The PR adds a Pi embedded runner wrapper that reads static `params.user` and injects it into outgoing provider request payloads through `streamWithPayloadPatch`.

## Best Possible Solution

Land a narrow static `params.user` passthrough only for supported OpenAI/OpenRouter-compatible request families, with regression tests for supported injection and unsupported-provider no-op behavior. Keep dynamic trace/session metadata design in #63901.

## Reproduction Assessment

Yes. A focused unit case can configure `agents.defaults.params.user`, run the PR wrapper with an `anthropic-messages` model, and observe the unexpected top-level `user` key; this review did not execute tests because the task is read-only.

## Solution Assessment

No. The capability is valid, but the current implementation is not the best fix because it mutates all provider payloads; the safer path is a user-specific API-family guard plus tests modeled on the existing `parallel_tool_calls` coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Gate user payload injection by supported API:** `src/agents/pi-embedded-runner/extra-params.ts:329-348`
  - body: With a global `agents.defaults.params.user`, this wrapper patches every provider payload. Non-OpenAI-family routes such as `anthropic-messages` and native Gemini can receive an unexpected top-level `user` key, which can leak an attribution identifier or cause provider rejection. Return the underlying stream unchanged unless the model API supports this field.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff can leak a configured attribution identifier to unsupported third-party provider payloads unless the new wrapper is API-guarded.

Concerns:

  - body: `params.user` may identify a tenant, project, or operator. The PR forwards it to every object payload via `streamWithPayloadPatch`, so unsupported provider routes receive data they did not opt into; gate the wrapper before payload mutation.
