## Summary

Keep open. The PR is focused, but the proposed wrapper mutates every object payload when `params.user` is set, while current main’s comparable request-body wrappers are API-scoped; it also still lacks contributor real behavior proof and now conflicts with main.

## What This Changes

The PR adds a post-plugin embedded-runner wrapper that forwards configured `params.user` into provider request payloads.

## Best Possible Solution

Land a narrow attribution path only after maintainer API/privacy agreement, with supported-provider guards, tests for compatible and unsupported payloads, docs for the chosen config shape, and redacted real provider proof.

## Reproduction Assessment

Yes for the PR defect by source inspection: when `params.user` normalizes to a string, the added wrapper patches object payloads through `streamWithPayloadPatch` without checking `model.api`. No live provider request proof is present.

## Solution Assessment

No. The useful behavior should be implemented as a supported-API-scoped payload wrapper or folded into the broader metadata design, not as an unconditional cross-provider body mutation.
