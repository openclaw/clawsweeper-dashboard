## Summary

Keep open. The static attribution feature is useful, but the PR is not mergeable because the new payload wrapper applies `user` to every provider request and the external contributor has not provided real after-fix behavior proof.

## What This Changes

The PR adds a Pi embedded runner stream wrapper that reads configured `params.user` and injects it into request payloads via `streamWithPayloadPatch`.

## Best Possible Solution

Land a narrow static `params.user` passthrough only for verified compatible request APIs, with supported and unsupported payload-shape tests; keep dynamic session metadata design in https://github.com/openclaw/openclaw/issues/63901.

## Reproduction Assessment

Yes for the PR defect via source inspection: the added wrapper applies whenever `effectiveExtraParams.user` is present and `streamWithPayloadPatch` mutates any object payload. I did not execute tests because this review is read-only.

## Solution Assessment

No. The requested static attribution capability is reasonable, but this implementation needs a supported-API guard, focused payload tests, and contributor real behavior proof before it is mergeable.
