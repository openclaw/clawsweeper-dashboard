## Summary

Keep open. The feature is still useful, but this PR is not mergeable because it patches configured `user` into every provider payload without a supported-API guard and it has no after-fix real behavior proof.

## What This Changes

The PR adds a Pi embedded runner stream wrapper that reads `ctx.effectiveExtraParams.user` and patches `user` into outgoing provider request payloads.

## Best Possible Solution

Land a narrow static `params.user` passthrough only for verified compatible request APIs, with supported and unsupported payload-shape tests; keep dynamic trace/session metadata design in the related open metadata issue.

## Reproduction Assessment

Yes for the PR defect. Source inspection shows the new wrapper applies whenever `effectiveExtraParams.user` is present and `streamWithPayloadPatch` mutates any object payload, including unsupported APIs such as `anthropic-messages`; I did not execute tests because this review is read-only.

## Solution Assessment

No. The requested static attribution capability is reasonable, but this implementation needs a supported-API guard, focused payload tests, and contributor real behavior proof before it is the best mergeable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Gate user payload injection by supported API:** `src/agents/pi-embedded-runner/extra-params.ts:329-348`

Non-OpenAI-family routes such as `anthropic-messages` and native Google payloads will receive an unexpected top-level `user` key, which can leak an attribution identifier and can make strict providers reject the request; return the underlying stream unchanged unless the request API supports this field.
