## Summary

Keep open: current main still lacks direct `params.user` passthrough and this PR is the active implementation, but it should not merge as-is because the wrapper mutates every provider payload and the external PR has no real behavior proof.

## What This Changes

The PR adds a Pi embedded runner stream wrapper that reads `ctx.effectiveExtraParams.user` and patches `user` into outgoing provider request payloads.

## Best Possible Solution

Land a narrow static `params.user` passthrough only for verified OpenAI/OpenRouter-compatible request APIs, with payload-shape tests; keep dynamic trace/session metadata design in https://github.com/openclaw/openclaw/issues/63901 and https://github.com/openclaw/openclaw/pull/54725.

## Reproduction Assessment

Yes for the PR defect. Source inspection shows the new wrapper applies whenever `effectiveExtraParams.user` is present and `streamWithPayloadPatch` then mutates any object payload, including unsupported APIs such as `anthropic-messages`; I did not execute tests because this review is read-only.

## Solution Assessment

No. The requested static attribution capability is reasonable, but this implementation is not the best mergeable fix because it needs a supported-API guard and focused supported/unsupported payload tests before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Gate user payload injection by supported API:** `src/agents/pi-embedded-runner/extra-params.ts:329-348`
  - body: With a global `agents.defaults.params.user`, this wrapper patches every provider payload. Non-OpenAI-family routes such as `anthropic-messages` and native Google payloads will receive an unexpected top-level `user` key, which can leak an attribution identifier and can make strict providers reject the request; return the underlying stream unchanged unless the request API supports this field.
  - confidence: 0.91
