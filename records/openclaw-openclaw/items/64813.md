## Summary

Keep open. Current main still lacks direct `params.user` request-body passthrough, but this PR is not mergeable as written because it injects the configured identifier into every provider payload, lacks real request proof, and overlaps the broader open provider-metadata design.

## What This Changes

The PR adds a post-plugin stream wrapper in `src/agents/pi-embedded-runner/extra-params.ts` that normalizes `params.user` and patches request payloads with a top-level `user` field.

## Best Possible Solution

Land a narrow static `params.user` passthrough only for explicitly compatible request APIs, with supported and unsupported payload-shape tests, while leaving session-derived `trace`/`user` metadata to the broader provider-metadata issue.

## Reproduction Assessment

Yes for the PR defect by source inspection: the added wrapper runs whenever configured `params.user` normalizes to a string, and the payload helper mutates any object payload. No live provider call was run in this read-only review.

## Solution Assessment

No. Static attribution is a plausible feature, but this implementation needs a supported-API guard, focused payload tests, contributor real behavior proof, and maintainer API/privacy confirmation before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92
