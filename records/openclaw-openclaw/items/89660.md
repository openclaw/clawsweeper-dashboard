## Summary

Keep open. Current main still has a strict config-schema/runtime mismatch: the OpenAI completions runtime reads the requested compat flag, but `ModelCompatSchema` and `ModelCompatConfig` do not accept it for user config, so the reported custom-provider startup failure remains actionable.

## What This Changes

The issue requests making the existing `requiresReasoningContentOnAssistantMessages` model compat flag schema-valid for custom OpenAI-compatible providers so DeepSeek-style replay placeholders can be enabled.

## Best Possible Solution

Align `ModelCompatConfig` and `ModelCompatSchema` with the existing runtime-consumed compat flag, add focused schema coverage and docs for the DeepSeek replay opt-in, and avoid introducing a competing `reasoningFormat` key.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: placing `requiresReasoningContentOnAssistantMessages` under `models.providers.*.models[].compat` reaches a strict schema that does not declare the key. I did not run the gateway or a live proxy because this review is read-only.

## Solution Assessment

Yes, the best fix is the narrow schema/type alignment for the existing runtime flag plus regression coverage. Adding a new public `reasoningFormat` key would duplicate the current compat direction and is not needed for this report.
