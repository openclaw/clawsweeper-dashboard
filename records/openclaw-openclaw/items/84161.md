## Summary

Keep open: the voice-call transcript bug is real on current main, but this PR is not merge-ready because it can duplicate assistant transcript entries, mixes in boot/session-memory behavior, and still lacks inspectable real behavior proof.

## What This Changes

The PR appends bot transcript entries for text-bearing `call.speaking`, buffers Google realtime transcript deltas, treats final user realtime text as authoritative, and changes boot-run session classification/reaping.

## Best Possible Solution

Land a narrow voice-call fix that persists exactly one assistant transcript entry per final realtime turn, preserves existing classic TTS transcript behavior, and moves boot/session-memory changes to a separately reviewed PR.

## Reproduction Assessment

Yes, from source inspection: current main forwards final assistant realtime text as `call.speaking`, while the manager only changes state for that event. I did not run a live call in this read-only review.

## Solution Assessment

No. The right fix is narrower: persist one final assistant transcript entry for realtime `call.speaking`, avoid classic TTS and Google final-callback duplicates, and split unrelated boot/session-memory behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid duplicating Telnyx TTS transcript entries:** `extensions/voice-call/src/manager/events.ts:280-281`
  - body: `manager.speak()` already appends the bot text after `provider.playTts()`, and Telnyx normalizes `call.speak.started` into a text-bearing `call.speaking` event. Appending every non-empty `call.speaking` text here will record the same classic Telnyx TTS utterance twice; gate this to realtime-origin events or dedupe against the already-persisted bot text.
  - confidence: 0.9
- **[P2] Emit one final Google transcript per turn:** `extensions/google/realtime-voice-provider.ts:770-771`
  - body: When Google sends `outputTranscription.finished`, this block forwards that chunk as final and then flushes the buffered aggregate as another final. The realtime handler will persist both as assistant turns, often one partial and one full aggregate, so the bridge should emit only the aggregate final or otherwise suppress the duplicate final callback.
  - confidence: 0.86
