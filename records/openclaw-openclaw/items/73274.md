## Summary

Keep open. Current main still lacks a durable transcript append helper on `api.runtime.agent.session`; the linked `openclaw-plugin-cron-context` workaround uses the available in-memory system-event seam, and durable next-turn injections are single-use context rather than permanent transcript history. The request is valid API work, but it needs maintainer SDK/security design before implementation.

## What This Changes

The issue requests exposing a persistent cross-session transcript append API to plugins so cron output can be written into channel session history.

## Best Possible Solution

Define a constrained, documented Plugin SDK contract for assistant-attributed persistent transcript appends, or solve the cron delivery persistence path in core if that is the intended product behavior; arbitrary user-role writes should stay out until separately designed.

## Reproduction Assessment

Not applicable: this is a feature/API request rather than a broken documented behavior. The high-confidence static check is that current main exposes only session store/path helpers and durable next-turn injection, while the requested transcript append helper is absent from plugin runtime types, wiring, mocks, and docs.

## Solution Assessment

Unclear. Exposing the raw internal helper, especially with a `role` option, is not proven to be the safest design; a constrained assistant-only, authority-gated runtime helper or a core cron persistence fix would be more maintainable.

## Security Review

Status: needs_attention

Summary: The requested API is security-sensitive because it would let plugin code persist content into arbitrary session transcripts.

Concerns:

- **[medium] Require authority for cross-session transcript writes:** `src/plugins/runtime/types-core.ts:156`
  - body: Adding an append helper under `api.runtime.agent.session` would let plugin code persist content into arbitrary session transcripts, so ordinary runtime access should not be enough without explicit trusted capability or operator opt-in.
  - confidence: 0.9
- **[medium] Keep public writes assistant-attributed by default:** `src/config/sessions/transcript.ts:229`
  - body: The existing exact helper rejects non-assistant messages, while the sample API includes a `role` option; user-role writes need separate attribution, replay, and audit semantics before becoming public API.
  - confidence: 0.86
- **[medium] Preserve idempotency and update semantics:** `src/config/sessions/transcript.ts:262`
  - body: The internal path checks idempotency, appends through the transcript manager, and emits transcript updates; a public API should preserve those semantics and avoid encouraging manual JSONL writes.
  - confidence: 0.84
