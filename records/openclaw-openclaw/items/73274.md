## Summary

Keep open. Current main still has the internal durable assistant transcript append helper, but it is not exposed on the plugin runtime session API or documented SDK surface; the existing durable `enqueueNextTurnInjection` seam is next-turn prompt context, not permanent transcript history. The reporter's linked `openclaw-plugin-cron-context` workaround and open cron mirroring PR #74913 are useful context, but they do not resolve the requested Plugin Runtime API decision.

## What This Changes

The issue requests a plugin runtime helper for durable cross-session transcript appends so plugins can persist cron output or similar assistant messages into another session's transcript.

## Best Possible Solution

Keep this open for maintainer API design. The near-term cron workflow should be evaluated through the cron-owned destination-session mirror in #74913/#74743, while any public plugin runtime transcript API should be narrow, capability-gated, assistant-only by default, idempotent, auditable, documented, and implemented through the existing `SessionManager` append path rather than encouraging manual JSONL writes.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence static check is that current main has the internal durable helper and ephemeral system-event queue, but no plugin runtime or SDK-documented transcript append helper.

## Solution Assessment

Unclear. Exposing the internal helper directly is not proven to be the safest solution; the narrower maintainable alternatives are the existing durable next-turn injection API for one-shot prompt context and cron-owned transcript mirroring for the reported cron delivery workflow.

## Security Review

Status: needs_attention

Summary: The requested API is security-sensitive because it would expose durable cross-session transcript mutation to plugin code.

Concerns:

- **[medium] Gate durable transcript writes with explicit authority:** `src/plugins/runtime/types-core.ts:146`
  - body: Adding a helper under `api.runtime.agent.session` would let plugin code persist content into other sessions. That should require an explicit trusted-plugin or operator opt-in capability rather than ordinary hook access alone.
  - confidence: 0.9
- **[medium] Keep public writes assistant-only unless redesigned:** `src/config/sessions/transcript.ts:224`
  - body: The internal exact helper rejects non-assistant messages, while the proposal includes a `role` option. User-role writes need attribution, audit, and replay semantics before exposure.
  - confidence: 0.88
- **[medium] Preserve append and transcript update semantics:** `src/config/sessions/transcript.ts:278`
  - body: The safe internal path appends through `SessionManager` and emits transcript update events. A public API should retain that path and define redaction and audit behavior.
  - confidence: 0.82

## Risks / Open Questions

- A plugin-facing durable cross-session transcript write path can let plugins persist forged or untrusted content into unrelated session history unless it has explicit authority, audit, and role constraints.
- The cron-specific symptom may be better solved by cron-owned destination-session mirroring, so maintainers should settle that scope before expanding the public SDK.
- The proposal includes an optional `user` role, while the internal safe helper is assistant-only; user-role writes need a separate attribution and replay design.
