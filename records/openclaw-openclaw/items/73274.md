## Summary

Keep open. Current main still lacks a durable transcript append helper on the plugin runtime surface, and exposing cross-session transcript writes is a Plugin SDK/security boundary rather than conservative cleanup. The linked workaround plugin is useful evidence of demand, while the related cron PR is narrower and does not implement the requested public API.

## What This Changes

Request to expose a durable plugin runtime API that lets plugins append idempotent assistant transcript messages to arbitrary session transcripts.

## Best Possible Solution

Keep this open for maintainer SDK design. Use the existing next-turn injection seam for bounded prompt context today, review the cron-owned mirroring path separately, and only add a public transcript append API if maintainers approve a narrow assistant-only, idempotent, auditable, explicitly authorized contract with docs and contract tests.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence static check is that current main lacks the requested runtime helper while the internal durable helper and the ephemeral system-event queue both exist.

## Solution Assessment

Unclear. Exposing the internal helper directly, especially with a `role` option, is not proven to be the safest solution; a narrower assistant-only authorized API or cron-owned mirror may be more maintainable.

## Security Review

Status: needs_attention

Summary: The requested API is security-sensitive because it would create a durable cross-session transcript write path for plugin code.

Concerns:

- **[medium] Require authority for cross-session writes:** `src/plugins/runtime/types-core.ts:146`
  - body: A helper under `api.runtime.agent.session` would let plugin code persist content into sessions other than the active one, so it should require an explicit trusted-plugin or operator opt-in capability rather than ordinary hook access alone.
  - confidence: 0.9
- **[medium] Keep public writes assistant-only unless redesigned:** `src/config/sessions/transcript.ts:224`
  - body: The internal exact helper rejects non-assistant messages, while the proposal includes a `role` option. User-role transcript writes need attribution, audit, and replay semantics before becoming public API.
  - confidence: 0.88
- **[medium] Define update and audit semantics:** `src/config/sessions/transcript.ts:282`
  - body: The safe internal path appends through `SessionManager` and emits transcript update events. A public plugin API should define event, redaction, idempotency, and audit behavior instead of encouraging manual JSONL appends.
  - confidence: 0.82
