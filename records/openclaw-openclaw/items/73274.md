## Summary

Keep open. Current main still lacks a durable transcript append helper on `api.runtime.agent.session` or a public `openclaw/plugin-sdk/transcript.runtime` subpath; the linked workaround plugin uses the available in-memory system-event seam, and the related open PR at https://github.com/openclaw/openclaw/pull/80208 is not merged and still raises SDK/security design questions.

## What This Changes

Expose a durable plugin runtime API for appending messages to another session transcript, motivated by cron output injection into channel session context.

## Best Possible Solution

Define a constrained, documented Plugin SDK transcript-append contract with explicit operator authority, assistant-attributed writes, idempotency/update-event semantics, docs, API/export checks, and focused contract tests.

## Reproduction Assessment

Not applicable: this is a feature/API request rather than broken documented behavior. The high-confidence static check is that current main omits a durable transcript append helper from runtime types, runtime wiring, SDK exports, docs, and plugin runtime mocks while the internal helper exists.

## Solution Assessment

Unclear. Exposing the raw internal helper, especially with a `role` option or exact-message writes, is not proven to be the safest solution; a constrained authority-gated assistant-write contract is the more maintainable direction.

## Security Review

Status: needs_attention

Summary: The requested API would grant durable cross-session transcript write authority to plugins, so it needs explicit capability, opt-in, and audit semantics.

Concerns:

- **[medium] Require authority for cross-session transcript writes:** `src/plugins/runtime/types-core.ts:204`
  - body: Adding an append helper under `api.runtime.agent.session` would let plugin code persist content into caller-selected session transcripts, so ordinary runtime access should not be enough without an explicit trusted capability or operator opt-in.
  - confidence: 0.9
- **[medium] Avoid public user-role or exact-message spoofing by default:** `src/config/sessions/transcript.ts:250`
  - body: The internal exact helper rejects non-assistant roles, while the issue's sample API includes a role option; user-role or exact metadata writes need attribution and replay semantics before becoming public API.
  - confidence: 0.86
- **[medium] Preserve idempotency and transcript update semantics:** `src/config/sessions/transcript.ts:283`
  - body: The internal safe path checks idempotency, appends through the session manager, and emits transcript update events; any public API should preserve those semantics instead of encouraging manual JSONL writes.
  - confidence: 0.84
