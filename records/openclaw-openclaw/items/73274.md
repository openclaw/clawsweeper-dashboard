## Summary

Keep open. Current main still does not expose a durable transcript append helper on `api.runtime.agent.session`; the linked `openclaw-plugin-cron-context` workaround usefully proves demand, but `enqueueSystemEvent` and durable next-turn injections do not create permanent session transcript history, and the new API needs Plugin SDK/security design.

## What This Changes

Expose a durable plugin runtime API for appending assistant messages to another session's persistent transcript.

## Best Possible Solution

Define a constrained, documented Plugin SDK transcript-append contract with explicit capability or operator opt-in, assistant-only default semantics, idempotency, transcript update behavior, audit/redaction expectations, and contract tests.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence static check is that current main exposes only session store/path helpers on `api.runtime.agent.session` while the durable append helper remains internal and tested.

## Solution Assessment

Unclear. A public append API is plausible, but exposing the raw helper or user-role writes is not proven to be the safest design; a constrained assistant-only API with explicit authority would be safer.

## Security Review

Status: needs_attention

Summary: Needs attention because the requested API would grant plugins durable cross-session transcript write authority.

Concerns:

- **[medium] Require authority for cross-session transcript writes:** `src/plugins/runtime/types-core.ts:156`
  - body: Adding an append helper under `api.runtime.agent.session` would let plugin code persist content into arbitrary session transcripts, so it should require an explicit trusted capability or operator opt-in instead of ordinary hook access alone.
  - confidence: 0.9
- **[medium] Keep public writes assistant-only unless redesigned:** `src/config/sessions/transcript.ts:229`
  - body: The existing exact helper rejects non-assistant messages, while the proposal includes a `role` option. User-role transcript writes need clear attribution, replay, and audit semantics before becoming public API.
  - confidence: 0.88
- **[low] Preserve append idempotency and update events:** `src/config/sessions/transcript.ts:262`
  - body: The safe internal path deduplicates idempotency keys, appends through the session transcript writer, and emits transcript update events; any public API should preserve those semantics and avoid encouraging manual JSONL writes.
  - confidence: 0.86
