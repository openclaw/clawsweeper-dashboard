## Summary

Keep open. Current main still does not expose a durable transcript append helper on `api.runtime.agent.session`; the reporter's `openclaw-plugin-cron-context` workaround uses the available system-event seam, but that does not create permanent transcript history, and the requested API needs explicit Plugin SDK/security design before implementation.

## What This Changes

Expose a plugin runtime session API that lets plugins append assistant output to another session's persistent transcript instead of relying on in-memory system events.

## Best Possible Solution

Define a constrained, documented Plugin SDK transcript-append contract with explicit trusted capability or operator opt-in, assistant-only default semantics, idempotency and transcript-update guarantees, and contract tests.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence static check is that current main's runtime types, runtime wiring, docs, and plugin runtime mocks omit a durable transcript append helper while the internal helper and tests exist.

## Solution Assessment

Unclear. Exposing the raw internal helper, especially with a `role` option, is not proven to be the safest design; a constrained assistant-only API or cron/delivery-owned mirror path behind explicit authority would be safer.

## Security Review

Status: needs_attention

Summary: The requested API would grant plugins durable cross-session transcript write authority, so it needs explicit capability, opt-in, and audit semantics before implementation.

Concerns:

- **[medium] Require authority for cross-session transcript writes:** `src/plugins/runtime/types-core.ts:156`
  - body: Adding an append helper under `api.runtime.agent.session` would let plugin code persist content into arbitrary session transcripts, so it should require an explicit trusted capability or operator opt-in rather than ordinary runtime access alone.
  - confidence: 0.9
- **[medium] Keep public transcript writes assistant-attributed by default:** `src/config/sessions/transcript.ts:229`
  - body: The existing exact helper rejects non-assistant messages, while the proposal includes a `role` option; user-role writes need separate attribution, replay, and audit semantics before becoming public API.
  - confidence: 0.86
- **[medium] Preserve idempotency and transcript update semantics:** `src/config/sessions/transcript.ts:262`
  - body: The safe internal path checks idempotency, writes through `appendSessionTranscriptMessage`, and emits transcript update events; a public API should preserve those behaviors rather than encouraging manual JSONL appends.
  - confidence: 0.84
