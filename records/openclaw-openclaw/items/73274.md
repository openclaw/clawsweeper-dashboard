## Summary

Keep open. Current main still lacks a durable transcript append helper on `api.runtime.agent.session`; the internal helper exists and outbound mirroring uses it, but making it public crosses a Plugin SDK/security boundary. The linked `openclaw-plugin-cron-context` workaround is useful demand evidence, but it still relies on ephemeral `enqueueSystemEvent` rather than permanent transcript history.

## What This Changes

Request to expose a durable cross-session transcript append helper in the plugin runtime API so cron-context plugins can persist assistant output into target session history.

## Best Possible Solution

Define a constrained Plugin SDK transcript-append contract with explicit operator capability, assistant-only default semantics, idempotency, update/audit behavior, redaction expectations, docs, and contract tests.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence static check is that current main's runtime types, wiring, and docs expose only session store/path helpers while the internal durable helper remains private.

## Solution Assessment

Unclear. Exposing the internal helper directly, especially with a `role` option, is not proven to be the safest design; a constrained authorized assistant-only API or a cron-owned mirror path would be safer.

## Security Review

Status: needs_attention

Summary: The requested API is security-sensitive because it would let plugin code durably write into sessions other than the active one.

Concerns:

- **[medium] Require authority for cross-session transcript writes:** `src/plugins/runtime/types-core.ts:146`
  - body: A helper under `api.runtime.agent.session` would let plugin code persist content into arbitrary session transcripts, so it should require an explicit trusted capability or operator opt-in rather than ordinary hook access alone.
  - confidence: 0.9
- **[medium] Keep public writes assistant-only unless redesigned:** `src/config/sessions/transcript.ts:225`
  - body: The internal exact helper rejects non-assistant messages, while the proposal includes a `role` option. User-role transcript writes need clear attribution, audit, and replay semantics before becoming public API.
  - confidence: 0.88
- **[medium] Define update, idempotency, and audit semantics:** `src/config/sessions/transcript.ts:283`
  - body: The internal path appends through `SessionManager` and emits transcript updates. A public plugin API should preserve those behaviors and define redaction/audit expectations instead of encouraging manual JSONL appends.
  - confidence: 0.84
