## Summary

Keep open. Current main still does not expose a durable transcript append helper on the plugin runtime surface; the newer `api.enqueueNextTurnInjection(...)` seam covers restart-safe one-shot prompt context but not permanent session history. The cron-specific symptom is now also represented by open PR #74913, but that PR is not merged and does not expose the requested Plugin Runtime API.

## What This Changes

The issue requests a plugin runtime API that lets plugins append assistant or user messages durably to another session transcript by session key.

## Best Possible Solution

Keep the existing `enqueueNextTurnInjection` seam as the supported path for restart-safe, bounded next-turn context. For permanent cron history, review the cron-owner destination-session mirroring path in #74913/#74743 first; only add a plugin transcript append API if maintainers approve a narrow, assistant-only, auditable, explicitly authorized contract.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence static check is that current main lacks `api.runtime.agent.session` transcript append support while the internal durable helper, ephemeral system-event queue, and durable one-shot next-turn injection seam are all present.

## Solution Assessment

Unclear. Exposing the internal helper directly is not proven to be the safest solution; the narrower maintainable alternatives are the existing next-turn injection API for prompt context and a cron-owned destination transcript mirror for the reported cron delivery workflow.

## Security Review

Status: needs_attention

Summary: The requested API would create a plugin-facing durable cross-session transcript write path, so it needs explicit security design before implementation.

Concerns:

- **[medium] Gate durable transcript writes by explicit authority:** `src/plugins/runtime/types-core.ts:146`
  - body: Adding a helper under `api.runtime.agent.session` would let plugin code persist content into other sessions. That should require an explicit trusted-plugin or operator opt-in capability, not ordinary hook access alone.
  - confidence: 0.9
- **[medium] Keep public writes assistant-only unless redesigned:** `src/config/sessions/transcript.ts:224`
  - body: The internal exact helper rejects non-assistant messages, while the proposal includes a `role` option. User-role writes would need attribution, audit, and replay semantics before being exposed.
  - confidence: 0.88
- **[medium] Preserve transcript event and redaction semantics:** `src/config/sessions/transcript.ts:278`
  - body: The safe internal path appends through `SessionManager` and emits transcript update events. A plugin-facing API should define redaction/audit behavior and avoid encouraging manual JSONL appends.
  - confidence: 0.82
