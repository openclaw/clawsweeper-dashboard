## Summary

Keep open. Current main still lacks a durable transcript append helper on the plugin runtime session API; the linked cron-context plugin workaround uses the available in-memory system-event seam, but the requested cross-session persistent write API needs explicit Plugin SDK and security design.

## What This Changes

Expose a durable plugin runtime API for appending assistant messages to another session transcript from hooks such as `agent_end`.

## Best Possible Solution

Define a constrained Plugin SDK transcript-append contract with explicit operator authority, assistant attribution, idempotency guarantees, transcript-update behavior, docs, and contract tests.

## Reproduction Assessment

Not applicable: this is a feature/API request rather than broken documented behavior. The high-confidence static check is that current main omits a durable append helper from runtime types, runtime wiring, docs, and plugin runtime mocks while the internal helper exists.

## Solution Assessment

Unclear. Exposing the raw helper, especially with a `role` option, is not proven to be the safest design; a narrow assistant-only, authority-gated runtime helper that preserves idempotency and update semantics would be more maintainable.

## Security Review

Status: needs_attention

Summary: The requested API would grant durable cross-session transcript write authority to plugins, so it needs explicit capability, opt-in, and audit semantics.

Concerns:

- **[medium] Require authority for cross-session transcript writes:** `src/plugins/runtime/types-core.ts:156`
  - body: Adding a helper under `api.runtime.agent.session` would let plugin code persist content into arbitrary session transcripts, so ordinary runtime access should not be enough without an explicit trusted capability or operator opt-in.
  - confidence: 0.9
- **[medium] Keep public writes assistant-attributed by default:** `src/config/sessions/transcript.ts:229`
  - body: The existing exact helper rejects non-assistant messages, while the issue's sample API includes a `role` option; user-role writes need separate attribution, replay, and audit semantics before becoming public API.
  - confidence: 0.86
- **[medium] Preserve idempotency and transcript update semantics:** `src/config/sessions/transcript.ts:262`
  - body: The internal safe path checks idempotency, appends through the session writer, and emits transcript update events; a public API should preserve those semantics and avoid encouraging manual JSONL writes.
  - confidence: 0.84
