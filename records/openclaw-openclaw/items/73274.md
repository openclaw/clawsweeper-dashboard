## Summary

Keep open. Current main still does not expose the requested durable transcript append helper on the plugin runtime surface, and the remaining decision is a Plugin SDK/security boundary rather than conservative cleanup. The linked cron-context plugin workaround is useful evidence, while the related open cron PR is narrower and does not add the requested Plugin Runtime API.

## What This Changes

Request to expose a Plugin Runtime API that lets plugins append durable messages to another session transcript instead of relying on ephemeral system events.

## Best Possible Solution

Keep this open for maintainer design. Use `api.enqueueNextTurnInjection(...)` for bounded next-turn prompt context today, review the cron-owned destination mirroring path in #74913 for the reported cron workflow, and only add a public transcript append API if maintainers approve a narrow, assistant-only, idempotent, auditable, explicitly authorized contract with docs and contract tests.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence static check is that current main lacks the requested runtime helper while the internal durable helper, ephemeral system-event queue, and durable next-turn injection seam are all present.

## Solution Assessment

Unclear. Exposing the internal helper directly, especially with a `role` option, is not proven to be the safest solution; a cron-owned mirror or a narrower assistant-only authorized API would be safer maintainable directions.

## Security Review

Status: needs_attention

Summary: The requested API would create a durable cross-session transcript write path for plugin code, so it needs explicit security design before implementation.

Concerns:

- **[medium] Require authority for durable cross-session writes:** `src/plugins/runtime/types-core.ts:146`
  - body: A helper under `api.runtime.agent.session` would let plugin code persist content into sessions other than the active one. That should require an explicit trusted-plugin or operator opt-in capability, not ordinary hook access alone.
  - confidence: 0.9
- **[medium] Keep public writes assistant-only unless redesigned:** `src/config/sessions/transcript.ts:224`
  - body: The internal exact helper rejects non-assistant messages, while the proposal includes a `role` option. User-role transcript writes need attribution, audit, and replay semantics before becoming public API.
  - confidence: 0.88
- **[medium] Preserve transcript update and audit semantics:** `src/config/sessions/transcript.ts:282`
  - body: The safe internal path appends through `SessionManager` and emits transcript update events. A public plugin API should define event, redaction, idempotency, and audit behavior instead of encouraging manual JSONL appends.
  - confidence: 0.82
