## Summary

Keep open. Current main still does not expose a durable transcript append helper through `api.runtime.agent.session`; the linked `openclaw-plugin-cron-context` workaround is useful demand evidence, but the remaining work crosses a Plugin SDK and cross-session write-security boundary.

## What This Changes

The issue requests a plugin runtime API for durable cross-session transcript appends so cron/channel plugins can persist assistant output into another session history.

## Best Possible Solution

Define a narrow Plugin SDK contract with explicit operator authorization, assistant-only semantics unless deliberately expanded, idempotency, transcript update/audit behavior, docs, and contract tests; otherwise keep cron mirroring in a cron-owned/core delivery path.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence static check is that current main lacks the requested runtime helper while internal durable append and prompt-context alternatives are present.

## Solution Assessment

Unclear. Directly exposing the internal helper, especially with a `role` option, is not proven to be the safest shape; a constrained authorized assistant-only API or cron-owned mirror path would be safer to review.

## Security Review

Status: needs_attention

Summary: The request is security-sensitive because it would let plugin code durably write content into other sessions' transcripts.

Concerns:

- **[medium] Require authority for cross-session transcript writes:** `src/plugins/runtime/types-core.ts:146`
  - body: A helper under `api.runtime.agent.session` would let plugin code persist content into sessions other than the active one, so it should require an explicit trusted-plugin or operator opt-in capability rather than ordinary hook access alone.
  - confidence: 0.9
- **[medium] Keep public writes assistant-only unless redesigned:** `src/config/sessions/transcript.ts:225`
  - body: The internal exact helper rejects non-assistant messages, while the proposal includes a `role` option. User-role transcript writes need attribution, audit, and replay semantics before becoming public API.
  - confidence: 0.88
- **[medium] Define update and audit semantics:** `src/config/sessions/transcript.ts:283`
  - body: The safe internal path appends through `SessionManager` and emits transcript update events. A public plugin API should define event, redaction, idempotency, and audit behavior instead of encouraging manual JSONL appends.
  - confidence: 0.84
