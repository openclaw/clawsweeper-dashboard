## Label Justifications

- `P2`: The item was a normal-priority plugin API gap with clear external-plugin value but no current core outage.
- `impact:session-state`: The requested capability affects persistent session transcripts that later agent turns consume as conversation history.
- `impact:security`: The requested capability lets trusted plugin code write durable cross-session transcript content, so the plugin trust boundary is central.

## Summary

Current main now implements the requested durable plugin transcript-write surface through the documented `openclaw/plugin-sdk/session-transcript-runtime` subpath, and the fix has shipped in the v2026.6.10 release train.

## What This Changes

The issue requests a public plugin runtime API that lets plugins append messages to another session transcript persistently instead of using the in-memory `enqueueSystemEvent` workaround.

## Best Possible Solution

Keep the shipped `openclaw/plugin-sdk/session-transcript-runtime` contract as the supported plugin path for durable scoped transcript reads and writes.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source and release inspection confirm the requested durable public transcript-write surface now exists.

## Solution Assessment

Yes. The merged SDK subpath is the maintainable solution because it exposes a scoped identity/target/read/write/publish/lock contract instead of asking plugins to import internal transcript files or append JSONL manually.
