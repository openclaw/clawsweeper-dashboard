## Summary

Keep open. Current main still lacks a durable transcript append helper on `api.runtime.agent.session`; the internal helper exists and is used for core outbound mirroring, but exposing cross-session transcript writes is a Plugin SDK/security boundary that needs maintainer design. The linked `openclaw-plugin-cron-context` workaround is useful demand evidence, while existing host-hook next-turn injections provide durable prompt context rather than permanent transcript history.

## What This Changes

Request to expose a plugin runtime helper for appending messages to another session's persistent transcript, mainly so isolated cron announce output can be preserved in channel session history.

## Best Possible Solution

Keep this open for a narrow Plugin SDK design: define authorization, assistant-only semantics unless explicitly expanded, idempotency, transcript update/audit behavior, docs, and contract tests, then either expose a constrained runtime helper or choose a cron-owned mirroring path if that better matches the product boundary.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence static check is that current main lacks the requested runtime helper while the internal durable append helper and existing ephemeral/durable prompt-context alternatives are present.

## Solution Assessment

Unclear. Exposing the internal helper directly, especially with a `role` option, is not proven to be the safest solution; a narrower authorized assistant-only API or cron-owned mirror may be more maintainable.

## Security Review

Status: needs_attention

Summary: The requested API is security-sensitive because it would create a durable cross-session transcript write path for plugin code.

Concerns:

- **[medium] Require authority for cross-session transcript writes:** `src/plugins/runtime/types-core.ts:146`
  - body: A helper under `api.runtime.agent.session` would let plugin code persist content into sessions other than the active one, so it should require an explicit trusted-plugin or operator opt-in capability rather than ordinary hook access alone.
  - confidence: 0.9
- **[medium] Keep public writes assistant-only unless redesigned:** `src/config/sessions/transcript.ts:224`
  - body: The internal exact helper rejects non-assistant messages, while the proposal includes a `role` option. User-role transcript writes need attribution, audit, and replay semantics before becoming public API.
  - confidence: 0.88
- **[medium] Define update and audit semantics:** `src/config/sessions/transcript.ts:282`
  - body: The safe internal path appends through `SessionManager` and emits transcript update events. A public plugin API should define event, redaction, idempotency, and audit behavior instead of encouraging manual JSONL appends.
  - confidence: 0.84

## Risks / Open Questions

- A public cross-session transcript write API could let plugin code persist content into sessions other than the active one without an explicit capability, operator opt-in, or audit contract.
- The proposed `role` option is broader than the internal helper, which is assistant-only; user-role writes would need attribution and replay semantics before becoming public API.
- Open PR #53607 overlaps transcript mirroring and SDK export questions, so maintainers should reconcile API shape before adding another public path.
