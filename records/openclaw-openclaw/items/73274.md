## Summary

Keep open. Current main still does not expose a durable transcript append helper on `api.runtime.agent.session`, and cron announce delivery still does not mirror isolated cron output into the target channel transcript. Main now has the safer `api.enqueueNextTurnInjection(...)` seam, which addresses the reporter's restart-safe next-turn context use case better than the old in-memory system-event workaround, but it is explicitly one-shot prompt context rather than permanent transcript history.

## What This Changes

Request to expose a plugin runtime API for durable cross-session transcript appends so cron/channel plugins can persist assistant output into target sessions.

## Best Possible Solution

Treat the shipped `api.enqueueNextTurnInjection(...)` seam as the recommended path for restart-safe, bounded, exactly-once plugin context. If permanent transcript history is still required, design an additive assistant-only delivery-mirror API with explicit operator authorization and the same idempotency/session-event behavior as the internal helper, or implement cron announce transcript mirroring in the cron delivery owner path.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence static check is that current main lacks `api.runtime.agent.session` transcript append support while the internal durable helper, the ephemeral system-event queue, and the newer one-shot `enqueueNextTurnInjection` seam are all present.

## Solution Assessment

Unclear. Exposing the internal helper directly is not yet proven to be the safest API shape; the existing next-turn injection API may solve the durable prompt-context use case, while a permanent transcript append helper would need narrower assistant-only, opt-in semantics.

## Security Review

Status: needs_attention

Summary: The requested API would grant plugins durable cross-session transcript write authority, so the design needs explicit security review before implementation.

Concerns:

- **[medium] Gate durable transcript writes by explicit authority:** `src/plugins/runtime/types-core.ts:142`
  - body: Adding a write helper under `api.runtime.agent.session` would let plugin code persist content into other sessions. That should require an explicit trusted-plugin or operator opt-in capability beyond ordinary hook access.
  - confidence: 0.9
- **[medium] Preserve assistant-only delivery-mirror semantics:** `src/config/sessions/transcript.ts:224`
  - body: The internal exact helper rejects non-assistant messages, while the proposal includes a `role` option. Public exposure should stay assistant-only unless maintainers define attribution, auditing, and user-role write semantics.
  - confidence: 0.87
- **[medium] Require redaction and event semantics for persistent writes:** `src/config/sessions/transcript.ts:278`
  - body: The safe internal path writes through `SessionManager` and emits transcript update events; a plugin-facing version needs explicit guard/redaction policy and should not encourage manual JSONL appends.
  - confidence: 0.82

## Risks / Open Questions

- Maintainers still need to decide whether plugins should ever get permanent cross-session transcript write authority, or whether `api.enqueueNextTurnInjection(...)` is the intended supported seam for this class of cron/context plugins.
- Direct transcript writes need explicit decisions on capability gating, attribution, redaction, idempotency, assistant-only semantics, and cross-session abuse controls.
- Cron announce mirroring may belong in core delivery rather than a public arbitrary transcript-write API, but current cron delivery code does not do that mirroring.
