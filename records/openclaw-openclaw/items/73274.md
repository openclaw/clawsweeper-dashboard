## Summary

Keep open. Current main still lacks a durable transcript append helper on `api.runtime.agent.session`; the internal helper exists for trusted core outbound mirroring, but exposing cross-session transcript writes is a Plugin SDK and security boundary that needs maintainer design. The linked `openclaw-plugin-cron-context` workaround is useful demand evidence, while `enqueueSystemEvent` and next-turn injections still do not provide permanent transcript history.

## What This Changes

The issue requests a public Plugin Runtime API for durable cross-session transcript appends from plugins, mainly for cron-context output into target session history.

## Best Possible Solution

Define a constrained Plugin SDK transcript-append contract with explicit operator capability, assistant-only default semantics, idempotency, transcript update/audit behavior, redaction expectations, docs, and contract tests.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence static check is that current main's runtime types, runtime wiring, docs, and source search expose only store/path helpers while the durable transcript helper remains internal.

## Solution Assessment

Unclear. Exposing the internal helper directly, especially with a `role` option, is not proven to be the safest design; a constrained authorized assistant-only API or cron-owned mirror path would be safer.

## Security Review

Status: needs_attention

Summary: The requested API is security-sensitive because it would let plugin code durably write into sessions other than the active one.

Concerns:

- **[medium] Require authority for cross-session transcript writes:** `src/plugins/runtime/types-core.ts:146`
  - body: A helper under `api.runtime.agent.session` would let plugin code persist content into arbitrary session transcripts, so it should require an explicit trusted capability or operator opt-in rather than ordinary hook access alone.
  - confidence: 0.9
- **[medium] Keep public writes assistant-only unless redesigned:** `src/config/sessions/transcript.ts:225`
  - body: The internal exact helper rejects non-assistant messages, while the proposal includes a `role` option. User-role transcript writes need clear attribution, audit, and replay semantics before becoming public API.
  - confidence: 0.88
- **[medium] Define transcript update and audit semantics:** `src/config/sessions/transcript.ts:283`
  - body: The safe internal path appends through `SessionManager` and emits transcript update events. A public plugin API should preserve those behaviors and define redaction/audit expectations instead of encouraging manual JSONL appends.
  - confidence: 0.84
