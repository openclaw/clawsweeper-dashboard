## Summary

Keep open. Current main still does not expose a durable transcript append helper through the plugin runtime API, and the related cron-owner PR remains open and unmerged; the remaining decision is whether to add a broad plugin-facing transcript write authority at all.

## What This Changes

The issue requests a plugin runtime API for appending durable assistant or user messages into an arbitrary session transcript, mainly so cron-output plugins can persist context into channel sessions.

## Best Possible Solution

Review and resolve the cron-owner mirroring path in #74913/#74743 first. Add a plugin transcript append API only if maintainers approve a narrow, assistant-only, auditable, explicitly authorized SDK contract with docs and contract tests.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence static check is that current main lacks the requested runtime surface while the internal append helper, ephemeral system-event queue, and durable next-turn injection seam are all present.

## Solution Assessment

Unclear. Directly exposing the internal helper is not proven to be the safest fix; the safer alternatives are the existing next-turn injection seam for one-shot prompt context and a cron-owned destination transcript mirror for the reported cron workflow.

## Security Review

Status: needs_attention

Summary: The requested API is security-sensitive because it would expose durable cross-session transcript writes to plugin code.

Concerns:

- **[medium] Gate durable transcript writes by explicit authority:** `src/plugins/runtime/types-core.ts:146`
  - body: A helper under api.runtime.agent.session would let plugin code persist content into other sessions. That should require an explicit trusted-plugin or operator opt-in capability, not ordinary hook access alone.
  - confidence: 0.9
- **[medium] Keep public writes assistant-only unless redesigned:** `src/config/sessions/transcript.ts:224`
  - body: The internal exact helper rejects non-assistant messages, while the proposal includes a user role option. User-role writes need attribution, audit, and replay semantics before exposure.
  - confidence: 0.88
- **[medium] Preserve transcript event and audit semantics:** `src/config/sessions/transcript.ts:278`
  - body: The safe internal path appends through SessionManager and emits transcript update events. A plugin API needs to define event, redaction, and audit behavior instead of encouraging manual JSONL writes.
  - confidence: 0.82
