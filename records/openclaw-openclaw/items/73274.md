## Summary

Keep open. Current main still does not expose a durable transcript append helper on `api.runtime.agent.session`, and isolated cron announce delivery still sends outbound messages without mirroring them into the target channel transcript. The newer `api.enqueueNextTurnInjection(...)` seam is relevant and safer for restart-safe next-turn context, but it is explicitly one-shot prompt context rather than permanent transcript history.

## What This Changes

The issue requests exposing a plugin runtime API for durable cross-session transcript appends so plugins can persist assistant messages into other sessions.

## Best Possible Solution

Treat `api.enqueueNextTurnInjection(...)` as the supported path for restart-safe, bounded next-turn plugin context. If permanent history is still needed, design an additive, assistant-only, operator-authorized delivery-mirror API or implement cron announce transcript mirroring in the cron delivery owner path.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence static check is that current main lacks a plugin runtime transcript append surface while the internal durable helper, ephemeral system-event queue, and durable one-shot next-turn injection seam are all present.

## Solution Assessment

Unclear. Directly exposing the internal helper, especially with a `role` option, is not proven to be the safest solution; a narrower assistant-only mirror API or cron-owned mirroring path would better preserve attribution and security boundaries.

## Security Review

Status: needs_attention

Summary: No patch is under review, but the requested API would add durable cross-session transcript write authority and needs explicit security design before implementation.

Concerns:

- **[medium] Gate durable transcript writes by explicit authority:** `src/plugins/runtime/types-core.ts:146`
  - body: Adding a public helper under `api.runtime.agent.session` would let plugin code persist content into arbitrary sessions. That should require explicit operator authorization or a trusted-plugin capability, not ordinary runtime access.
  - confidence: 0.9
- **[medium] Preserve assistant-only mirror semantics:** `src/config/sessions/transcript.ts:224`
  - body: The internal exact append helper rejects non-assistant messages, while the proposal includes a `role` option. Public exposure should stay assistant-only unless maintainers define user-role attribution, auditing, and trust semantics.
  - confidence: 0.88
- **[medium] Require safe append and event semantics:** `src/config/sessions/transcript.ts:278`
  - body: The internal path writes through `SessionManager` and emits transcript update events. Any plugin-facing API should preserve that path, include idempotency, and avoid encouraging manual JSONL appends.
  - confidence: 0.84

## Risks / Open Questions

- Exposing durable cross-session transcript writes to plugins is a security-sensitive API decision because it would let plugin code persist content into sessions other than the active run.
- The existing next-turn injection seam may satisfy the reporter's prompt-context goal without adding permanent transcript history, so maintainers should decide whether transcript persistence is truly required.
- If a transcript append API is added, it needs explicit attribution, idempotency, update-event, cleanup, and operator-authorization semantics rather than a raw role-configurable wrapper around the internal helper.
