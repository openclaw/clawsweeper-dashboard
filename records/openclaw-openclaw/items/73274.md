- `impact:session-state`: The requested capability changes how plugins persist cross-session transcript context that later agent turns read as session history.

## Summary

Keep open: current main and v2026.6.6 still lack the requested public plugin transcript-append API. Core cron delivery mirroring and durable next-turn injections mitigate some cases, but the general cross-session persistent transcript write remains a product and security-boundary decision, and the related implementation PR is still open and conflicting.

## What This Changes

The issue requests exposing a supported plugin API for appending assistant/user messages to another session transcript so cron/channel plugins can persist cross-session context instead of relying on ephemeral system events.

## Best Possible Solution

Keep this open until maintainers decide whether to expose a narrow, documented, trusted plugin transcript-write API and either rework/land the related PR or define a different runtime-session method.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source review on current main and v2026.6.6 shows the requested public append API is absent while the internal helper exists.

## Solution Assessment

Unclear: a public transcript-write seam may be the right plugin API, but exposing the raw internal helper is not automatically the best boundary until maintainers settle trust, audit, idempotency, and message-shape semantics.
