## Summary

Keep open. Current main still does not expose a durable transcript append helper on `api.runtime.agent.session`; the linked `openclaw-plugin-cron-context` workaround uses the available system-event seam, but that seam is intentionally ephemeral and the requested durable cross-session writer needs Plugin SDK and security design.

## What This Changes

Request to expose a durable cross-session transcript append helper through `api.runtime.agent.session` so plugins can persist cron output into target channel session history.

## Best Possible Solution

Define a constrained, documented Plugin SDK transcript-append contract with explicit capability or operator opt-in, assistant-attributed writes by default, idempotency and transcript-update guarantees, and focused contract coverage.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence static check is that current main omits a durable plugin runtime writer while the internal helper exists and the documented system-event workaround is ephemeral.

## Solution Assessment

Unclear. Exposing the raw internal helper, especially with a `role` option, is not yet proven to be the safest API shape; a constrained assistant-only writer behind explicit authority is the safer design direction.

## Security Review

Status: needs_attention

Summary: The requested API would grant third-party plugin code durable cross-session transcript write authority and needs explicit capability, opt-in, and attribution design before implementation.

Concerns:

- **[medium] Require authority for durable cross-session writes:** `src/plugins/runtime/types-core.ts:156`
  - body: Adding a writer under `api.runtime.agent.session` would let plugin code persist content into arbitrary session transcripts, so it should require an explicit trusted capability or operator opt-in rather than ordinary runtime access alone.
  - confidence: 0.9
- **[medium] Keep public writes assistant-attributed by default:** `src/config/sessions/transcript.ts:229`
  - body: The existing exact helper rejects non-assistant messages, while the proposed API includes a user role option; user-role writes need separate attribution, replay, and audit semantics before becoming public SDK.
  - confidence: 0.86
- **[medium] Preserve idempotency and update events:** `src/config/sessions/transcript.ts:262`
  - body: The safe internal path checks idempotency/dedupe, appends through the transcript writer, and emits transcript update events; a public API should preserve those semantics instead of encouraging manual JSONL writes.
  - confidence: 0.84
