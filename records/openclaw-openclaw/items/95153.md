## Summary

Keep open: this PR targets a real reconnect/session identity bug, but the current head is not merge-ready because it implements reconnect adoption as a global empty-create heuristic that regresses the Gateway/SDK create contract and still lacks real Control UI proof.

## What This Changes

The branch adds a Gateway helper and regression tests that make no-key, no-intent `sessions.create` return the latest active dashboard session instead of creating a new dashboard key.

## Best Possible Solution

Keep default `sessions.create` fresh, then add a reconnect-only adoption signal at the Control UI/session-restore boundary or another documented internal Gateway contract with coverage for both reconnect reuse and SDK empty-create freshness.

## Reproduction Assessment

Yes. Source inspection reproduces the patch defect: on PR head a no-key/no-intent `sessions.create({ agentId })` adopts an existing dashboard key, while current main and v2026.6.10 create a fresh dashboard key.

## Solution Assessment

No. The latest implicit heuristic is not the best fix because it moves reconnect semantics into the general Gateway create API; the safer path is a reconnect-specific adoption boundary plus focused tests and real browser proof.
