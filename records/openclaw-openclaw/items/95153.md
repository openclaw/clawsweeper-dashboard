## Summary

Keep open: the latest branch fixed the earlier parent/hook regression, but it still silently changes the public no-key `sessions.create` contract from create to adopt and has only test-harness proof, so it needs maintainer API direction and real Control UI proof before merge.

## What This Changes

The PR changes Gateway `sessions.create` so passive no-key dashboard creates can return an existing active dashboard session and adds focused session-create tests.

## Best Possible Solution

Keep the Gateway create contract fresh by default; add explicit adopt/reconnect semantics or restrict reuse to a proven Control UI reconnect path, then demonstrate refresh reuse and New Chat separation in a real setup.

## Reproduction Assessment

Yes for the contract change: current main builds a fresh random dashboard key for no-key `sessions.create`, while the PR returns an existing active dashboard key when present. No real browser reconnect reproduction was supplied.

## Solution Assessment

No. Gateway-side reconnect reuse may be useful, but implicit adoption inside a public create method is not the best shape without an explicit adopt/reconnect contract or maintainer-approved API direction.
