## What This Changes

The PR adds bootstrap-token rate limiting with deferred failure accounting in WebSocket auth and also tightens trusted official npm plugin installs to require exact-version specs plus expected integrity.

## Best Possible Solution

Yes. Source inspection on current main shows bogus bootstrap tokens reach `verifyDeviceBootstrapToken` before any bootstrap-specific limiter, and that verifier runs through module-level `withLock` over disk-backed bootstrap state.

## Solution Assessment

Mostly yes. A bootstrap-token limiter before verification is the narrow maintainable fix, and deferred failure accounting preserves device-token fallback; the separate plugin-install hardening should be called out explicitly or split before merge.
