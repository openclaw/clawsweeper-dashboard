## Summary

Keep open. Current main still lacks bootstrap-token rate limiting, while this PR applies a narrow pre-verify limiter with focused coverage; it needs the required changelog entry and maintainer-reviewed validation before merge.

## What This Changes

The PR adds a bootstrap-token auth rate-limit scope, wraps WebSocket bootstrap verification with check/reset/deferred failure accounting, and expands auth-context tests.

## Best Possible Solution

Land the narrow bootstrap-token limiter with deferred failure accounting, focused tests, a single-line Unreleased changelog entry, and maintainer-reviewed validation.

## Reproduction Assessment

Yes. Current main reaches `verifyDeviceBootstrapToken` for bogus bootstrap-token WebSocket connects before any bootstrap-specific limiter, and that verifier serializes through module-level `withLock` over disk-backed bootstrap state.

## Solution Assessment

Mostly yes. A dedicated bootstrap-token limiter before verification is the narrow maintainable fix, and deferred failure accounting preserves valid device-token fallback; the missing changelog entry is the remaining mechanical blocker.
