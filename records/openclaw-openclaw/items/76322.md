## Summary

Keep open. The PR is useful and targets a real pre-auth gateway DoS surface, but it is not merge-ready because the deferred bootstrap failure accounting misses the device-token-rate-limited fallback path, leaving a source-level bypass; it also still lacks the required changelog entry.

## What This Changes

The PR adds a bootstrap-token auth rate-limit scope, applies check/reset/deferred failure accounting in WebSocket connect auth, and expands auth-context tests.

## Best Possible Solution

Keep the dedicated bootstrap-token limiter design, but flush pending bootstrap failures on every final failed handshake including device-token rate-limit results, add regression coverage and a single-line Unreleased changelog entry, then put the security-sensitive auth change through maintainer review.

## Reproduction Assessment

Yes. Current main reaches the mutex-backed verifyDeviceBootstrapToken path without a bootstrap-token limiter, and the PR-specific bypass is source-reproducible by first locking the device-token scope and then sending bogus bootstrap plus device tokens.

## Solution Assessment

No, not as written. The dedicated limiter is the right narrow direction, but the deferred failure record must also run when device-token fallback is rate-limited; otherwise the fix leaves a reachable pre-auth verifier path unthrottled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Record bootstrap failures when fallback is rate-limited:** `src/gateway/server/ws-connection/auth-context.ts:231`
  - body: When the device-token scope is already locked, this branch skips device-token verification and falls through to the final return without flushing `pendingBootstrapFailure`. An attacker can first lock their device-token scope, then keep sending bogus bootstrap+device tokens; every request still reaches `verifyBootstrapToken` while the bootstrap-token counter is never charged. Record the pending bootstrap failure whenever the overall handshake remains unauthenticated, including this rate-limited fallback path.
  - confidence: 0.92

  - body: This changes user-facing gateway auth behavior, but the PR file list does not include `CHANGELOG.md`. Add a single-line Unreleased fix entry for the bootstrap-token rate limit so release notes cover the security hardening.
  - confidence: 0.93
