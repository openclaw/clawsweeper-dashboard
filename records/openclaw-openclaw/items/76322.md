## Summary

Keep open: this is an active security PR for a real current-main gap, but the branch still drops deferred bootstrap failure accounting when device-token fallback is already rate-limited; the overlapping narrower fix PR and umbrella issue are still open.

## What This Changes

The branch adds a bootstrap-token rate-limit scope, gates bootstrap-token verification before the mutex-backed verifier, defers bootstrap failure accounting across device-token fallback, and expands auth-context tests.

## Best Possible Solution

Land one gateway bootstrap-token limiter fix that short-circuits before the mutex-backed verifier and charges invalid bootstrap attempts on every final unauthenticated handshake while preserving valid device-token fallback.

## Reproduction Assessment

Yes. Source inspection on current main shows bootstrap-token handshakes reach verifyDeviceBootstrapToken without a bootstrap-token limiter, and that verifier is protected by a module-level withLock; the PR body also supplies after-fix terminal proof for the changed decision path.

## Solution Assessment

No as written. The pre-verifier gate is the right narrow fix shape, but deferred bootstrap failure accounting must also flush when the final unauthenticated result comes from device-token rate-limit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Record pending bootstrap failures after rate-limited fallback:** `src/gateway/server/ws-connection/auth-context.ts:256-258`
  - body: The deferred bootstrap failure is only recorded inside the device-token mismatch branch. If an attacker first locks the device-token bucket and then sends an invalid bootstrapToken with any deviceToken, this path skips the branch and never charges the bootstrap-token bucket, so the new limiter can be bypassed.
  - confidence: 0.9
