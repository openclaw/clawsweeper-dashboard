## Summary

Keep open: the branch addresses a real bootstrap-token mutex DoS path, but the current PR head still leaves a bootstrap-token limiter bypass when device-token fallback is already rate-limited. Current main remains ungated, and the related per-surface fix at https://github.com/openclaw/openclaw/pull/77527 is still open, so this needs security/owner review rather than cleanup close.

## What This Changes

The PR adds a bootstrap-token auth rate-limit scope, gates bootstrap-token verification before the verifier, defers bootstrap failure accounting across device-token fallback, and adds auth-context tests.

## Best Possible Solution

Land one narrow gateway fix that gates bootstrap-token verification before the mutex and charges invalid bootstrap attempts on every final unauthenticated handshake while preserving valid device-token fallback.

## Reproduction Assessment

Yes. Source inspection on current main shows resolveConnectAuthDecision calls verifyBootstrapToken before any bootstrap-token limiter, and verifyDeviceBootstrapToken serializes through withLock; the PR body also supplies after-fix terminal proof for the decision path.

## Solution Assessment

No as written. The pre-verifier gate is the right shape, but pending bootstrap failures must also flush when the final unauthenticated result is a device-token rate-limit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Record bootstrap failures when fallback is rate-limited:** `src/gateway/server/ws-connection/auth-context.ts:256-258`
  - body: pendingBootstrapFailure only flushes inside the device-token mismatch branch. If an attacker first locks the device-token bucket and then sends an invalid bootstrapToken with any deviceToken, this path skips that branch and never charges the bootstrap-token bucket, so the new limiter can be bypassed.
  - confidence: 0.9

Status: needs_attention
