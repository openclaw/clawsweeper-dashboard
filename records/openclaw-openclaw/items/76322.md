## Summary

Keep open: the PR targets a real pre-auth Gateway DoS surface and supplies usable terminal proof, but the current head still leaves a bootstrap limiter bypass when device-token fallback is already rate-limited.

## What This Changes

The PR adds a bootstrap-token auth-rate-limit scope, checks it before bootstrap-token verification, defers failed-bootstrap accounting across device-token fallback, and expands auth-context tests.

## Best Possible Solution

Land one bootstrap-token limiter fix that short-circuits before the mutex-backed verifier and records invalid bootstrap attempts on every failed final handshake while preserving valid device-token fallback.

## Reproduction Assessment

Yes, from source inspection. Current main reaches `verifyDeviceBootstrapToken` without a bootstrap-token limiter, and that verifier enters `withLock`; I did not run the timing PoC in this read-only review.

## Solution Assessment

No as written. The pre-verifier limiter is the right narrow fix, but deferred failure accounting must also flush when the final handshake fails because the device-token fallback is rate-limited.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Record pending bootstrap failures after rate-limited fallback:** `src/gateway/server/ws-connection/auth-context.ts:256-258`
  - body: The deferred bootstrap failure is only recorded inside the device-token mismatch branch. If an attacker first locks the device-token bucket and then sends an invalid `bootstrapToken` with any `deviceToken`, this code skips the branch and never charges the bootstrap bucket, so the new limiter can be bypassed.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff narrows pre-auth work but leaves a concrete bypass of the new bootstrap-token limiter after device-token lockout.

Concerns:

- **[high] Bootstrap limiter bypass after device-token lockout:** `src/gateway/server/ws-connection/auth-context.ts:256`
  - body: Because `pendingBootstrapFailure` is not flushed when `deviceTokenRateLimited` is true, an unauthenticated client can keep invalid bootstrap attempts from incrementing the bootstrap scope by pairing them with a device token after priming the device-token limiter.
  - confidence: 0.88
