## Summary

Keep open: the PR addresses the right gateway auth surface, but the current branch still leaves a bootstrap-token limiter bypass when device-token fallback is already rate-limited.

## What This Changes

The PR adds a bootstrap-token auth-rate-limit scope, gates bootstrap-token verification before the mutex-backed verifier, defers bootstrap failure accounting across device-token fallback, and expands auth-context tests.

## Best Possible Solution

Land one corrected bootstrap-token limiter fix that short-circuits before the mutex-backed verifier and charges invalid bootstrap attempts on every final unauthenticated handshake while preserving valid device-token fallback.

## Reproduction Assessment

Yes. Current main source shows a bogus bootstrap token can reach verifyDeviceBootstrapToken before any bootstrap-specific limiter, and that verifier runs under the shared bootstrap lock; I did not rerun the timing PoC in this read-only review.

## Solution Assessment

No as written. The pre-verifier rate-limit gate is the right narrow shape, but deferred failure accounting must also flush when the final handshake fails because the device-token fallback is rate-limited.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Record bootstrap failures after rate-limited fallback:** `src/gateway/server/ws-connection/auth-context.ts:231`
  - body: The deferred bootstrap failure is only recorded inside the device-token mismatch branch. If a client first exhausts the device-token bucket, then sends an invalid bootstrapToken with any deviceToken, deviceTokenRateLimited skips this branch, so the bootstrap-token bucket is never charged and the next request can enter verifyBootstrapToken again.
  - confidence: 0.9

## Security Review

Status: needs_attention
