## Summary

Keep open. Source inspection confirms current main has the reported pre-auth bootstrap-token rate-limit gap and the patch is directionally correct, but this external security PR still lacks after-fix real Gateway proof and a required changelog entry; maintainers should also reconcile it with the narrower open fix at https://github.com/openclaw/openclaw/pull/77527.

## What This Changes

Adds a bootstrap-token auth rate-limit scope, gates WebSocket bootstrap-token verification before the mutex-backed verifier, defers bootstrap failure accounting across device-token fallback, and expands auth-context coverage.

## Best Possible Solution

Land one maintained bootstrap-token rate-limit fix with changelog, focused auth-context and real Gateway coverage, after-fix runtime proof, and security-owner review while preserving valid device-token fallback behavior if that path is chosen.

## Reproduction Assessment

Yes. Current main calls `verifyBootstrapToken` directly while only the device-token path is rate-limited, and the verifier enters `withLock` before reading bootstrap state; I did not live-run the WebSocket timing PoC.

## Solution Assessment

Mostly yes. Reusing the existing per-IP `AuthRateLimiter` before the mutex-backed verifier is the narrow maintainable fix, and deferred bootstrap failure accounting avoids penalizing valid device-token fallback, but the branch still needs changelog and after-fix real behavior proof.
