## Summary

Keep open. Current main still sends bootstrap-token handshakes straight into the mutex-backed verifier without a bootstrap-specific limiter, and the PR’s runtime change is a plausible narrow fix, but the branch needs the required changelog entry before it is merge-ready.

## What This Changes

The PR adds standalone bootstrap-token DoS PoC scripts and wraps WebSocket bootstrap-token authentication with its own auth-rate-limit scope before calling `verifyDeviceBootstrapToken`.

## Best Possible Solution

Land a small gateway auth hardening that rate-limits bootstrap-token verification before the mutex, with focused regression tests and an `Unreleased` changelog entry; maintainer review should decide whether the standalone PoC scripts belong in-tree.

## Reproduction Assessment

Yes. The current source gives a high-confidence reproduction path: concurrent bogus bootstrap-token handshakes reach `verifyDeviceBootstrapToken`, which serializes through `withLock` before invalid-token rejection; the PR body also provides a concrete WebSocket flood PoC, although I did not execute it in this read-only review.

## Solution Assessment

Mostly yes. A separate bootstrap-token limiter scope checked before verification is the narrowest maintainable runtime fix and mirrors the device-token path, but the PR should add changelog/regression coverage and get maintainer judgment on the PoC scripts.
