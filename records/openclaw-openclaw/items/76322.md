## Summary

Keep open: the PR targets a real current-main gateway auth bug, but the current head still leaves a bootstrap-token limiter bypass when device-token fallback is already rate-limited.

## What This Changes

Adds a bootstrap-token rate-limit scope, gates bootstrap-token verification before the mutex-backed verifier, defers bootstrap failure accounting across device-token fallback, and expands auth-context tests.

## Best Possible Solution

Land one maintained gateway fix that rate-limits bootstrap-token verification before the mutex and records bootstrap failures on every unauthenticated final handshake while preserving valid device-token fallback.

## Reproduction Assessment

Yes from source inspection. Current main reaches verifyDeviceBootstrapToken without a bootstrap-token limiter, and that verifier enters a process-wide withLock before reading bootstrap state; I did not run the WebSocket timing PoC locally.

## Solution Assessment

No as written. The pre-verifier limiter gate is the right narrow fix, but pending bootstrap failures must also be flushed when device-token fallback is rate-limited and the final handshake remains unauthenticated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Flush bootstrap failures when fallback is rate-limited:** `src/gateway/server/ws-connection/auth-context.ts:256-258`
  - body: The deferred bootstrap failure is only recorded inside the device-token mismatch branch. If a client first locks the device-token bucket and then sends an invalid bootstrap token plus any device token, this path skips the mismatch branch, returns unauthenticated, and never increments the bootstrap bucket, so the bootstrap verifier remains reachable on repeated attempts.
  - confidence: 0.9
