## Summary

Keep open: current main still lacks bootstrap-token rate limiting and this PR targets the right Gateway auth surface with supplied runtime proof, but the proposed deferred accounting leaves a bypass when device-token fallback is already rate-limited.

## What This Changes

The PR adds a bootstrap-token auth-rate-limit scope, gates bootstrap-token verification before the mutex-backed verifier, defers bootstrap failure accounting across device-token fallback, and expands auth-context tests.

## Best Possible Solution

Land one maintained bootstrap-token limiter fix that records deferred bootstrap failures on every failed final handshake, keeps valid device-token fallback unpenalized, and leaves umbrella alerting/device-signature follow-ups on their canonical issues or PRs.

## Reproduction Assessment

Yes from source inspection. Current main calls `verifyBootstrapToken` before any bootstrap-specific limiter, and `verifyDeviceBootstrapToken` enters `withLock` before reading bootstrap state; I did not run the WebSocket timing PoC.

## Solution Assessment

No as written. The limiter placement is the right narrow owner-boundary fix, but the PR must also flush deferred bootstrap failures when fallback fails by rate-limit rather than only when device-token verification runs and mismatches.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Flush pending bootstrap failures after rate-limited fallback:** `src/gateway/server/ws-connection/auth-context.ts:256-258`
  - body: The deferred bootstrap failure is recorded only inside the device-token mismatch branch. If an attacker first locks the device-token bucket and then sends an invalid `bootstrapToken` plus any `deviceToken`, the request still enters `verifyBootstrapToken`, skips this branch because the fallback is rate-limited, and never increments the bootstrap bucket. Record the pending bootstrap failure whenever the final decision remains unauthenticated, including the device-token rate-limited path.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: Needs attention: the diff narrows pre-auth Gateway work but leaves a concrete accounting gap that can keep invalid bootstrap attempts from locking out after device-token fallback is already limited.

Concerns:

  - body: Because the pending bootstrap failure is not flushed when `deviceTokenRateLimited` is true, an unauthenticated client can keep the bootstrap verifier reachable by pairing invalid bootstrap tokens with a device token after priming the device-token bucket.
