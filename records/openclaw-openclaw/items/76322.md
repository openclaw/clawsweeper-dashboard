## Summary

Keep open: the security issue is valid on current main, but this PR still has blocking gaps in the proposed mitigation and overlaps with a newer focused bootstrap-token fix thread.

## What This Changes

The PR adds a bootstrap-token auth-rate-limit scope, gates bootstrap verification before the mutex-backed verifier, defers bootstrap failure accounting across device-token fallback, and expands auth-context tests.

## Best Possible Solution

Land one canonical bootstrap-token security fix that gates verifyDeviceBootstrapToken before the mutex with complete failure accounting, preserves valid device-token fallback, and leaves broader alerting policy on the umbrella security issue.

## Reproduction Assessment

Yes. Source inspection on current main shows a device identity plus bogus bootstrap token reaches verifyDeviceBootstrapToken before any bootstrap-token limiter, and that verifier is serialized under the bootstrap state lock.

## Solution Assessment

No as written. The pre-verifier gate is the right direction, but the branch must not rely only on the optional/default auth limiter and must record deferred bootstrap failures when the final handshake fails through a rate-limited device-token fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use an always-present limiter for bootstrap verifies:** `src/gateway/server/ws-connection/auth-context.ts:168-172`
  - body: This gate only runs when `params.rateLimiter` exists. Current server construction leaves non-browser clients without a limiter unless `gateway.auth.rateLimit` is configured, and default limiter settings exempt loopback, so those clients still reach the mutex-backed bootstrap verifier. Add a bootstrap-specific limiter that is constructed for this path and cover the no-config/default-loopback cases.
  - confidence: 0.9
- **[P1] Flush bootstrap failures after rate-limited fallback:** `src/gateway/server/ws-connection/auth-context.ts:256-258`
  - body: The pending bootstrap failure is recorded only when device-token verification runs and mismatches. If the device-token bucket is already locked, the fallback returns `rate_limited`, skips this block, and never increments the bootstrap bucket, letting invalid bootstrapToken+deviceToken attempts keep entering the verifier after priming the device-token lockout.
  - confidence: 0.9

## Security Review

Status: needs_attention

Concerns:

- **[high] Default bootstrap-token verifier remains reachable:** `src/gateway/server/ws-connection/auth-context.ts:168`
  - body: Because the new check depends on the optional/default auth limiter, non-browser deployments without gateway.auth.rateLimit and loopback-exempt cases can still enter verifyDeviceBootstrapToken.
  - confidence: 0.9

  - body: A pending bootstrap failure is not flushed when device-token fallback is rate-limited, so an attacker can avoid charging the bootstrap bucket after priming the device-token bucket.
