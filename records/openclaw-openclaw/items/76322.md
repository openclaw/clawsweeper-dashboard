## Summary

Keep open: this is an active security PR with a real current-main bug and useful proof, but the proposed deferred failure accounting still leaves a concrete bypass of the new bootstrap-token limiter when the device-token fallback is already rate-limited.

## What This Changes

The PR adds a bootstrap-token auth-rate-limit scope, gates bootstrap-token verification before the mutex-backed verifier, defers bootstrap failure accounting across device-token fallback, and expands auth-context tests.

## Best Possible Solution

Land one bootstrap-token limiter fix that short-circuits before the mutex-backed verifier and charges invalid bootstrap attempts on every final unauthenticated handshake while preserving valid device-token fallback.

## Reproduction Assessment

Yes, from source inspection: current main reaches verifyDeviceBootstrapToken without a bootstrap-token limiter, and that verifier is protected by a shared withLock. I did not run the timing PoC in this read-only review.

## Solution Assessment

No as written. The pre-verifier rate-limit gate is the right narrow shape, but deferred failure accounting must also flush when the final handshake fails because the device-token fallback is rate-limited.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Record pending bootstrap failures after rate-limited fallback:** `src/gateway/server/ws-connection/auth-context.ts:256-258`
  - body: The deferred bootstrap failure is only recorded in the device-token mismatch branch. If an attacker first locks the device-token bucket and then sends an invalid bootstrapToken with any deviceToken, this code skips the branch and never charges the bootstrap bucket, so the new limiter can be bypassed.
  - confidence: 0.9
