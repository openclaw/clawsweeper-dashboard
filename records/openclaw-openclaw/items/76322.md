## Summary

Keep open: current main still has the reported source-reproducible bootstrap-token DoS surface, and this PR is an active security fix candidate, but the patch still leaves concrete limiter bypass/default-coverage gaps that should block merge.

## What This Changes

The PR adds a bootstrap-token auth-rate-limit scope, gates bootstrap-token verification before the verifier, defers bootstrap failure accounting across device-token fallback, and expands auth-context tests.

## Best Possible Solution

Land one reviewed bootstrap-token limiter fix that blocks before verifyDeviceBootstrapToken in the intended deployment envelope and records bootstrap failures for every final unauthenticated handshake while preserving valid device-token fallback.

## Reproduction Assessment

Yes. Source inspection on current main shows a signed handshake with auth.bootstrapToken reaches verifyBootstrapToken before any bootstrap limiter, and verifyDeviceBootstrapToken runs under withLock; I did not run the live timing PoC in this read-only review.

## Solution Assessment

No. The pre-verifier gate is the right direction, but this patch still depends on the optional rateLimiter and misses pending bootstrap failure accounting when device-token fallback is already rate-limited.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Record pending bootstrap failures after rate-limited fallback:** `src/gateway/server/ws-connection/auth-context.ts:256-258`
  - body: The deferred bootstrap failure is flushed only in the device-token mismatch branch. If the device-token bucket is already locked, that branch is skipped and an unauthenticated request with an invalid bootstrapToken plus any deviceToken never charges the bootstrap bucket, so the new limiter can be bypassed.
  - confidence: 0.9
- **[P1] Use an always-on limiter for bootstrap-token verifies:** `src/gateway/server/ws-connection/auth-context.ts:168-172`
  - body: The new gate only runs when params.rateLimiter is provided. Gateways without gateway.auth.rateLimit, and default loopback-exempt paths, can still enter the mutex-backed verifier with invalid bootstrap tokens unless maintainers explicitly accept that policy.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff narrows pre-auth work but still leaves concrete bootstrap-token rate-limit bypass/default-coverage concerns.

Concerns:

- **[high] Bootstrap limiter bypass after device-token lockout:** `src/gateway/server/ws-connection/auth-context.ts:256`
  - body: Because pendingBootstrapFailure is not flushed when deviceTokenRateLimited is true, an attacker can pair invalid bootstrap tokens with a request that skips device-token verification and avoid incrementing the bootstrap-token bucket.
  - confidence: 0.9
- **[high] Default bootstrap verifier path remains unthrottled:** `src/gateway/server/ws-connection/auth-context.ts:168`
  - body: The added bootstrap-token check is guarded by params.rateLimiter, which is absent unless gateway.auth.rateLimit is configured and may exempt loopback by default, leaving some mutex-backed verifier paths reachable.
  - confidence: 0.83
