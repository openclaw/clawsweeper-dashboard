## Summary

Keep open: current main still has the source-reproducible bootstrap-token mutex-stall gap, and this PR adds the right pre-verifier gate, but the branch still leaves a bootstrap limiter bypass when device-token fallback is already rate-limited.

## What This Changes

The branch adds a bootstrap-token auth-rate-limit scope, gates bootstrap-token verification before the mutex-backed verifier, defers bootstrap failure accounting across device-token fallback, and expands auth-context tests.

## Best Possible Solution

Repair this PR or use the narrower implementation in https://github.com/openclaw/openclaw/pull/77527 so bootstrap-token verification is gated before the mutex and invalid bootstrap attempts are charged on every final unauthenticated outcome while valid device-token fallback remains clean.

## Reproduction Assessment

Yes. Current main reaches verifyDeviceBootstrapToken without a bootstrap-token limiter, and that verifier is serialized under withLock; the PR body also supplies terminal proof for the patched decision path.

## Solution Assessment

No as written. The pre-verifier gate is the right narrow shape, but deferred failure accounting must also flush when the final unauthenticated outcome is caused by device-token rate limiting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Record pending bootstrap failures after rate-limited fallback:** `src/gateway/server/ws-connection/auth-context.ts:256-258`
  - body: The deferred bootstrap failure is only recorded inside the device-token mismatch branch. If the device-token bucket is already locked, an invalid `bootstrapToken` plus any `deviceToken` skips this branch and returns `rate_limited` without charging the bootstrap bucket, so the new limiter can be bypassed after the attacker primes the device-token limiter.
  - confidence: 0.9
