## Summary

Keep open. The current main path is still vulnerable by source inspection and this PR is directionally correct, but it still needs release/docs notes, exact-head CI cleanup, and maintainer security review before merge.

## What This Changes

The PR adds bootstrap-token rate limiting before WebSocket bootstrap verification, defers bootstrap failure accounting through device-token fallback, and adds focused unit/integration coverage.

## Best Possible Solution

Land the narrow bootstrap-token limiter with deferred failure accounting, add release/docs notes, require green exact-head validation, and have a gateway/security maintainer review the pre-auth flow.

## Reproduction Assessment

Yes, by source inspection. Current main reaches `verifyDeviceBootstrapToken` for bogus bootstrap-token WebSocket connects before any bootstrap-specific limiter, and that verifier serializes through `withLock` over disk-backed bootstrap state.

## Solution Assessment

Mostly yes. A dedicated bootstrap-token scope checked before verification is the narrow maintainable fix, and deferred failure accounting preserves valid device-token fallback; the remaining issues are release/docs coverage and exact-head validation.
