## Summary

Keep open: current main still lacks a bootstrap-token limiter, and the PR is a useful security fix, but the branch still has a concrete failure-accounting bypass when device-token fallback is already rate-limited.

## What This Changes

The PR adds a bootstrap-token rate-limit scope, gates bootstrap-token verification before calling the verifier, defers failed-bootstrap accounting until auth fallback fails, and expands gateway auth-context tests.

## Best Possible Solution

Land one narrow gateway auth fix that gates bootstrap-token verification before the mutex and records pending bootstrap failures whenever the final decision is unauthenticated, while still discarding the pending failure on valid device-token success.

## Reproduction Assessment

Yes. Source inspection on current main shows bootstrap-token verification runs before any bootstrap limiter, and the PR body supplies after-fix terminal proof for the intended decision path.

## Solution Assessment

No as written. The pre-verifier gate is the right shape, but pending bootstrap failures must also flush when the final unauthenticated result is a device-token rate-limit.
