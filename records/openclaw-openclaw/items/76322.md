## Summary

Keep open: current main still has the bootstrap-token rate-limit gap, and this branch is useful security work, but it lacks after-fix real behavior proof and now overlaps the narrower open fix in #77527.

## What This Changes

Adds a bootstrap-token auth rate-limit scope, gates WebSocket bootstrap-token verification before the mutex-backed verifier, defers bootstrap failure accounting across device-token fallback, and expands auth-context tests.

## Best Possible Solution

Land one maintained bootstrap-token rate-limit fix with changelog, targeted auth-context coverage, WebSocket integration proof, and explicit security-owner review; preserve the valid-device-token fallback behavior if maintainers choose the narrower #77527 path.

## Reproduction Assessment

Yes from source inspection: current main calls `verifyBootstrapToken` directly while only the device-token path is rate-limited, and the verifier enters `withLock` before reading bootstrap state. I did not live-run the WebSocket timing PoC.

## Solution Assessment

Mostly yes: reusing the existing per-IP `AuthRateLimiter` before the mutex-backed verifier is the narrow maintainable fix, and deferred bootstrap failure accounting avoids penalizing valid device-token fallback. The branch still needs changelog and after-fix real behavior proof, and maintainers should reconcile it with #77527.
