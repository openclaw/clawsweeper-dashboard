Full review comments:

- **[P1] Store handoff packets in SQLite-backed state:** `src/auto-reply/reply/commands-handoff.ts:316-318`
  - body: This writes the handoff packet, latest packet, and metadata as Markdown/JSON files under the OpenClaw state dir. The packet includes transcript-derived context and optional operator notes, so this creates a parallel retention/privacy path outside the SQLite/plugin-state lifecycle required for runtime state. Please move this persistence into SQLite-backed state or make it an explicit export artifact with retention controls.
  - confidence: 0.92
- **[P1] Store inflight records in plugin state:** `extensions/telegram/src/telegram-reliability-store.ts:234-235`
  - body: The inflight store persists prompt previews and error text to a JSON sidecar under `state/telegram`. That is OpenClaw-owned runtime/cache state, so it should use the SQLite/plugin-state path instead of adding another file-backed store that doctor and retention tooling do not own.
  - confidence: 0.91
- **[P1] Preserve current recovery safeguards when rebasing:** `src/logging/diagnostic-stuck-session-recovery.runtime.ts:166-174`
  - body: This recovery path is based on older main logic. Current main now avoids resetting a lane that has started fresh work and handles stale active reply work separately; carrying this hunk forward without those safeguards can reset a live lane and double-run or drop queued work. Please rebase the `recoverable_stale_lane` behavior on top of the current safeguards.
  - confidence: 0.84
- **[P2] Notify every interrupted request or keep it pending:** `extensions/telegram/src/bot-core.ts:204`
  - body: `markStaleTelegramInflightInterrupted` marks every stale dispatched record as interrupted, but startup only sends notices for the first five records. Any later record becomes terminal without a user-visible notice, which preserves the silent-loss behavior this PR is meant to fix; either batch/continue notifications or leave unnotified records pending for a later retry.
  - confidence: 0.88
