  - body: `/handoff` persists transcript-derived Markdown plus JSON metadata directly under the state dir.   - confidence: 0.94
- **[P1] Store inflight records in plugin state:** `extensions/telegram/src/telegram-reliability-store.ts:234-235`
  - body: The new Telegram reliability store writes `reliability-inflight-*.json` sidecars from the dispatch hot path. Telegram runtime state was recently moved to SQLite plugin state, and adding a new JSON store reintroduces the upgrade, validation, and concurrency problems that migration removed.
  - confidence: 0.95
- **[P1] Preserve the current recovery freshness guard:** `src/logging/diagnostic-stuck-session-recovery.runtime.ts:166-180`
  - body: This recovery path fabricates a released outcome from queued diagnostic work after `resetCommandLane` returns 0, but the current main implementation also checks for a fresh active task id before resetting or clearing queued state. Conflict resolution must keep that guard or a fresh turn can be mis-cleared or the queue can be marked recovered without releasing the actual lane.
  - confidence: 0.88
- **[P2] Notify all interrupted records or leave them pending:** `extensions/telegram/src/bot-core.ts:204-216`
  - body: `markStaleTelegramInflightInterrupted` terminalizes every stale dispatched record, but startup only sends notices for `interrupted.slice(0, 5)`. Any later records are no longer pending and still get no Telegram-visible interruption notice, which preserves the silent-loss behavior this PR is trying to fix.
  - confidence: 0.91
- **[P3] Remove release-owned changelog edits:** `CHANGELOG.md:8`
  - body: Normal PRs should carry release-note context in the PR body or commit message; `CHANGELOG.md` is release-generated in this repo. These entries should be removed so release generation owns the final changelog text.
  - confidence: 0.86
