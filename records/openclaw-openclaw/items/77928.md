- **[P1] Rebase recovery on the current lane reset guard:** `src/logging/diagnostic-stuck-session-recovery.runtime.ts:166-180`
  - body: This recovery hunk predates the merged fix in https://github.com/openclaw/openclaw/pull/91802: it reports a `recoverable_stale_lane` release after `resetCommandLane` returned 0 and lacks the current `getCommandLaneActiveTaskIds` fresh-turn guard.   - confidence: 0.89
- **[P1] Store handoff packets in SQLite:** `src/auto-reply/reply/commands-handoff.ts:316-318`
  - body: `writeHandoff` persists redacted transcript excerpts and metadata as `*.md` plus `latest.json` under `resolveStateDir`. Handoff packets are session state rather than import/export artifacts, so this bypasses the shared SQLite/plugin-state migration, TTL, validation, and concurrency controls.
  - confidence: 0.87
- **[P1] Store inflight records in plugin state:** `extensions/telegram/src/telegram-reliability-store.ts:234-235`
  - body: `storePath` writes Telegram delivery state to a JSON sidecar while the record includes prompt preview and error text. This should use the Telegram plugin-state/shared SQLite path so message-delivery state gets the existing validation, limits, TTL, and migration behavior.
  - confidence: 0.86
- **[P2] Notify every interrupted request or keep it pending:** `extensions/telegram/src/bot-core.ts:204`
  - body: `markStaleTelegramInflightInterrupted` terminalizes every stale dispatched record, but startup only sends notices for the first five. Any later records are skipped forever on later startups because they are no longer `dispatched`, recreating silent Telegram turn loss for those users.
  - confidence: 0.9
