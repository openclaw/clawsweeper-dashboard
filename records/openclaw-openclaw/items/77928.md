Keep open: the PR is plausible and actively maintained, but current PR head still has two concrete Telegram topic/scope regressions and the supplied proof does not exercise the visible Telegram chat behavior being added.

## What This Changes

The PR adds Telegram `/handoff` and `/resume latest` commands, inflight request tracking, failure/interruption notices, high-context long-input guarding, stale diagnostic-lane recovery, docs, changelog entries, and tests.

## Best Possible Solution

Fix the two topic/thread regressions, then land only after trusted live Telegram proof covers the visible handoff, resume, failure-notice, and interrupted-notice behavior.

## Reproduction Assessment

Yes for the review findings: source inspection at PR head shows interrupted notices bypass the central Telegram thread helper and handoff scope ignores `MessageThreadId`. The broader feature behavior is not fully reproduced because the proof does not exercise a live Telegram chat path.

## Solution Assessment

No, not yet. The maintainable path is to reuse `buildTelegramThreadParams` for interrupted notices, include `MessageThreadId` in handoff scope, and require trusted Telegram proof before merge.
