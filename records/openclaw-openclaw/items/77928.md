## Summary

Keep open. The PR is useful in direction, but it is not mergeable as-is because the new handoff command treats async transcript loaders as synchronous arrays, and the PR body explicitly says real Telegram runtime proof is still pending.

## What This Changes

Adds core `/handoff` and `/resume latest` commands, Telegram inflight reliability tracking/failure notices/long-input guard/startup interruption notices, tests, and a rollout runbook.

## Best Possible Solution

Fix the async transcript loading defect, add the required user-facing release note/docs updates, then validate the behavior in a shadow or live Telegram setup before maintainer review.

## Reproduction Assessment

Not applicable for the original feature request as a user bug. Yes for the review finding: source inspection shows the PR calls async transcript loaders without `await`, which makes `/handoff` fail whenever a real `sessionFile` is available.

## Solution Assessment

No. The product direction may be reasonable, but this patch needs the async loader fix, live Telegram proof, changelog coverage, and maintainer rollout judgment before it is the right merge path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Await the handoff transcript loaders:** `src/auto-reply/reply/commands-handoff.ts:223-240`
  - body: `loadCliSessionReseedMessages` and `loadCliSessionHistoryMessages` both return promises on current main. This code treats the promise like an array, so a real `/handoff` with a `sessionFile` will either fail typecheck on `.length` or throw when `renderHistoryMessages` calls `.slice()` on the promise. Await the reseed load before checking its length, and await the history fallback before rendering.
  - confidence: 0.96
