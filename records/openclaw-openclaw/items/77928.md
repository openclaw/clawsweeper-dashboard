Full review comments:

- **[P1] Move inflight state out of JSON sidecars:** `extensions/telegram/src/telegram-reliability-store.ts:234-235`
  - body: This new store writes runtime inflight records to `state/telegram/reliability-inflight-*.json`. Inflight delivery/session state is steady-state OpenClaw runtime state, and the repo policy keeps new runtime state in SQLite/plugin KV so upgrades, doctor, retention, and migrations have one owner. Please move this to the shared state DB or plugin KV before merge.
  - confidence: 0.91
- **[P1] Keep handoff packets in governed session state:** `src/auto-reply/reply/commands-handoff.ts:316-318`
  - body: These writes persist transcript excerpts and latest resume metadata as Markdown/JSON files under `state/handoffs/...`, and `/resume latest` depends on them as app state. That creates a second persistent session/context store outside SQLite/session retention and duplicates sensitive transcript content. Please store the packet and metadata in the canonical session/plugin state path, or make it an explicit export artifact that `/resume` does not depend on.
  - confidence: 0.89
- **[P2] Do not terminalize requests that will not be notified:** `extensions/telegram/src/bot-core.ts:204-210`
  - body: `markStaleTelegramInflightInterrupted` can return every stale dispatched record, but this loop sends notices only for `interrupted.slice(0, 5)`. Any sixth or later stale Telegram request becomes terminal and will not be picked up on a later startup, so the user still gets the silent-loss behavior this PR is meant to fix. Either notify all terminalized records with bounded scheduling, or leave unnotified records pending for a later pass.
  - confidence: 0.92
- **[P3] Remove release-owned changelog edits:** `CHANGELOG.md:78`
  - body: Normal PRs should not edit `CHANGELOG.md`; release generation owns it. Keep this release-note context in the PR body or squash message and let release tooling generate the changelog entry.
  - confidence: 0.87
