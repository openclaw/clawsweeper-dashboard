Full review comments:

- **[P1] Honor configured Hugging Face request overrides:** `extensions/huggingface/image-generation-provider.ts:97-109`
  - body: This new route builds its HTTP config without forwarding `models.providers.huggingface.request`, even though sibling image providers pass sanitized provider request overrides into `resolveProviderHttpRequestConfig`. Existing users who rely on proxy/TLS/private-network/custom transport settings for Hugging Face would get different behavior for image generation than they do for the rest of the provider, so thread the configured request through and add a regression test.
  - confidence: 0.91
- **[P1] Bound Hugging Face image response reads:** `extensions/huggingface/image-generation-provider.ts:139`
  - body: The provider reads the response with `response.arrayBuffer()` directly, so a bad or compromised upstream can stream an arbitrarily large body into memory. Use the shared `readProviderBinaryResponse` path, which already applies size limits and binary content validation, before wrapping the bytes as the generated image.
  - confidence: 0.88
- **[P2] Remove the normal PR changelog entry:** `CHANGELOG.md:23`
  - body: `CHANGELOG.md` is release-generated in this repo, and normal feature PRs should keep release notes in the PR body or merge message instead. Dropping this entry avoids conflicting with the release-owned changelog flow.
  - confidence: 0.86
- **[P3] Document Hugging Face in the capability matrix:** `docs/tools/image-generation.md:132-138`
  - body: The docs add Hugging Face to the provider route tables, but the provider capability matrix still omits it. Add a Hugging Face column with the route’s limits so users can see that edits are unsupported, count is limited to one, and size handling differs from sibling providers.
  - confidence: 0.78
