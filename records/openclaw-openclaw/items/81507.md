## Summary

Keep open: current main still stores Telegram polling leases in a process-global registry, has no production restart-boundary clear for that registry, and the issue reports the exact duplicate-poller error produced by that path.

## What This Changes

The issue requests deterministic cleanup of Telegram polling leases across in-process gateway restarts so stale same-token entries do not block new pollers.

## Best Possible Solution

Add a Telegram-owned lease cleanup path at channel shutdown or the in-process restart boundary, preserving the live duplicate-poller guard and covering stale same-token leases with regression tests.

## Reproduction Assessment

Yes, at source level: a stale same-token entry in the process-global lease map makes acquireTelegramPollingLease() throw the reported duplicate-poller error until the entry is released or the process exits. I did not live-run a packaged gateway restart in this read-only pass.

## Solution Assessment

Yes: the narrow maintainable fix is to clear or release only lifecycle-owned stale Telegram leases during shutdown or in-process restart, not to remove the same-token duplicate-poller guard.
