## Summary

Keep open: current main still keeps Telegram polling leases in a process-level registry with no production lifecycle clear path, and no open PR or canonical issue specifically owns the in-process restart stale-lease leak. The report is recent, concrete, and actionable as a Telegram plugin lifecycle bug.

## What This Changes

The issue asks for Telegram polling leases to be released across in-process gateway restarts so restarted Telegram accounts do not hit duplicate-poller errors.

## Best Possible Solution

Add Telegram-owned lifecycle cleanup for aborted or abandoned polling leases at the gateway restart boundary while preserving the per-token duplicate-poller guard, with regression coverage for stale process-level entries across an in-process restart.

## Reproduction Assessment

Yes from source inspection, but not from a live Telegram run: a stale process-level lease with a non-aborted signal still throws the reported duplicate-poller error, and current main has no production registry cleanup outside normal monitor completion.

## Solution Assessment

No to blind process-map clearing; the best fix is lifecycle-aware lease teardown or invalidation tied to Telegram account shutdown/restart so active old pollers are not accidentally hidden.

Current main stores Telegram polling leases in a process-level Symbol.for Map and releases them only when monitorTelegramProvider reaches its finally block.
