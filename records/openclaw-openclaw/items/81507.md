## Summary

Keep open: the report matches current `main` source, where Telegram polling leases live in a process-global registry and production lifecycle code has no explicit restart-boundary clear for stale entries. The useful fix is narrow and should preserve the same-token duplicate-poller guard while making in-process restart teardown deterministic.

## What This Changes

The issue requests that Telegram polling leases be released or cleared across in-process gateway restarts so stale same-token entries cannot block new pollers indefinitely.

## Best Possible Solution

Add a Telegram-owned lease cleanup path at channel shutdown or the gateway in-process restart boundary, and cover it with a regression that proves stale same-token leases are cleared without allowing two live pollers for one bot token.

## Reproduction Assessment

Yes, at source level: a stale same-token entry in the process-global lease map makes `acquireTelegramPollingLease()` throw the reported duplicate-poller error until that entry is released or the process exits. I did not live-run a packaged gateway restart in this read-only pass.

## Solution Assessment

Yes: the maintainable fix is to clear or release only stale lifecycle-owned leases during Telegram shutdown or in-process restart, not to remove the duplicate live-poller guard.
