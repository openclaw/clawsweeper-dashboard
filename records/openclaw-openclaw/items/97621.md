## Summary

Keep open: current main and the latest release still expose the xAI-billed `code_execution` and `x_search` plugin tools from xAI credential availability rather than from the active model provider, so the reported provider/billing boundary remains real and needs maintainer product/security direction.

## What This Changes

Request to prevent xAI-billed `code_execution` and `x_search` tools from being silently available to non-xAI model runs just because xAI credentials exist.

## Best Possible Solution

Define a provider-billed tool policy that defaults to visible, safe behavior, likely gating xAI server-side tools to xAI active-model runs unless an explicit opt-in or allowlist says otherwise, and update docs/tool metadata with that billing ownership.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main registers the xAI tools when xAI auth/config/env availability passes, and the xAI factories do not gate on `activeModel.provider`. I did not run a live xAI/GLM billing repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product/security policy. The maintainable direction is to use the existing plugin active-model context and explicit tool policy/config signals rather than a docs-only warning or ad hoc billing check.
