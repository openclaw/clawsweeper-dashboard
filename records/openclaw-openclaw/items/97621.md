## Label Justifications

- `P2`: This is a concrete provider/tool safety issue with real billing impact, but the blast radius is bounded to users with xAI credentials and xAI plugin tools enabled.
- `impact:security`: The issue concerns a credential-backed provider boundary where xAI credentials can be used for remote server-side execution outside the active model provider.
- `impact:auth-provider`: The reported behavior depends on auth-profile, env, and provider routing semantics rather than only on tool schema or UI copy.

## Summary

Keep open: current main still exposes the xAI-billed `code_execution` and `x_search` tools from xAI auth/config/env availability rather than the active model provider, and the related issues track tool availability or broader server-tool support rather than this cross-provider billing boundary.

## What This Changes

The issue asks OpenClaw to stop silently offering xAI-billed `code_execution` and `x_search` tools to agents running non-xAI models merely because xAI auth is present.

## Best Possible Solution

Define a provider-billed tool policy that defaults to safe, visible behavior, likely by gating xAI server-side tools to xAI active-model runs unless an explicit opt-in or allowlist says otherwise, and surface the provider/billing ownership in docs and tool catalog metadata.

## Reproduction Assessment

Yes at source level: current main and v2026.6.10 register the xAI tools whenever xAI auth/config/env availability passes, and the xAI factories do not gate on the active model provider. I did not run a live xAI/GLM billing repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the product policy. The narrowest maintainable direction is to use the existing plugin `activeModel` context and explicit tool policy/config signals rather than adding a second ad hoc billing gate.
