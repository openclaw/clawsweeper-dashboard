- `impact:security`: The issue concerns unexpected provider-side execution and data exposure to xAI from a tool call that can be offered outside an xAI model run.
- `impact:auth-provider`: The behavior is driven by xAI auth/profile/config resolution and active provider/model routing, which is the core affected area.

## Summary

Keep open: current main still exposes the xAI-owned `code_execution` and `x_search` tools from xAI auth/config availability rather than the active model provider, so the reported cross-provider billing path remains a real product/security policy gap.

## What This Changes

The issue asks OpenClaw to stop or clearly disclose xAI server-side tool exposure and billing when non-xAI model runs inherit implicitly enabled xAI tools from configured xAI credentials.

## Best Possible Solution

Decide the provider-billed tool default policy, then implement an xAI-owned guard, warning, or explicit opt-in with regression coverage for xAI and non-xAI model tool lists.

## Reproduction Assessment

Yes, source-level: current tool availability and factory gates expose xAI tools from xAI credentials without checking the active model provider, and the reporter supplied a concrete xAI 403 billing-limit failure from a non-xAI model run. I did not run a live xAI billing repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the default policy. The safest implementation shape is likely in the xAI plugin/tool availability boundary, but changing from documented auto-enable to opt-in or model-gated behavior is a product/security decision.
