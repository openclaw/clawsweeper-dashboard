## Summary

Keep this PR open. The artifact-growth problem looks plausible, but the patch has blocking correctness issues and the supplied terminal output shows the pre-fix growth state rather than an observed after-fix cleanup result.

## What This Changes

This PR caps Dream Diary blocks, prunes old session-corpus files during dreaming sweeps, and drops old short-term recall entries during store normalization.

## Best Possible Solution

Keep the artifact-bounding work in memory-core, but normalize diary entries without duplicating separators, base recall expiry on the same recallDays/lastRecalledAt semantics used by dreaming, and add focused regression tests plus real after-fix cleanup proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a live current-main run; source inspection and the PR body's terminal output make the long-run growth plausible, but I did not establish a multi-day runtime reproduction.

## Solution Assessment

Is this the best way to solve the issue? No. The direction is reasonable, but this exact patch corrupts diary block formatting and bypasses the existing recallDays recency contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Normalize diary entries before rejoining blocks:** `extensions/memory-core/src/dreaming-narrative.ts:697`
  - body: `buildDiaryEntry()` already returns text with a leading `---` separator, but the new capped append path pushes `entry.trim()` into `joinDiaryBlocks()`, which prepends another separator. Every append to an existing marked diary would write `---\n\n---` before the new entry and leave malformed diary blocks, so build the block without its delimiter before joining.
  - confidence: 0.91
- **[P2] Use shared recall recency semantics for expiry:** `extensions/memory-core/src/short-term-promotion.ts:468-471`
  - body: The new normalization filter drops entries solely because `lastRecalledAt` is older than 30 days, before the dreaming lookback helper can consider `recallDays`. Current code and tests intentionally keep entries whose `recallDays` contains a recent day even when `lastRecalledAt` is stale, so this can delete still-live recall evidence during normalization.
  - confidence: 0.78
