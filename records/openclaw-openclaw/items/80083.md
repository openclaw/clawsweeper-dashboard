## Summary

Keep open: current main still has the reported memory-core growth vectors, but this PR is not merge-ready because it lacks after-fix real behavior proof and introduces blocking diary/recall-store correctness issues.

## What This Changes

The branch changes three memory-core files to cap Dream Diary blocks, delete old session-corpus files, and expire old short-term recall entries.

## Best Possible Solution

Land a focused memory-core retention fix that normalizes diary blocks without duplicate delimiters, follows the established recallDays/lastRecalledAt contract, adds targeted regression tests, and includes redacted after-fix live proof.

## Reproduction Assessment

Yes, at source level and with the contributor's pre-fix terminal output: current main appends diary entries without a cap, writes session-corpus files without retention cleanup, and normalizes recall stores without age expiry. I did not run a live sweep because this review is read-only.

## Solution Assessment

No. The PR targets the right owner paths, but the current patch mishandles diary separators, drops recall entries before applying current lookback semantics, lacks tests, and lacks after-fix real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Strip the built diary delimiter before joining capped blocks:** `extensions/memory-core/src/dreaming-narrative.ts:697`
  - body: `buildDiaryEntry()` already includes the `---` separator, while `joinDiaryBlocks()` adds one for every block. Pushing `entry.trim()` writes `---\n\n---` before appended entries after the first, so the Dream Diary renders extra horizontal rules and the cap path rewrites malformed diary content.
  - confidence: 0.9
- **[P2] Respect recallDays before expiring recall entries:** `extensions/memory-core/src/short-term-promotion.ts:468-472`
  - body: Current main keeps entries when any `recallDays` value is in the lookback window even if `lastRecalledAt` is stale or unparseable. Dropping stale `lastRecalledAt` entries during normalization removes those entries before `recallDays` can be considered, contradicting the current lookback contract and losing recurrence evidence.
  - confidence: 0.86
