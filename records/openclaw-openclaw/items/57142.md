## Summary

Keep open. Current main still lacks configurable custom-term autocomplete for the WebChat/Control UI chat input: the existing completion surface is slash-command-only, the visible prompt suggestions are hard-coded welcome buttons, and the WebChat config/schema/docs do not define a custom suggestions list. The issue is recent, unprotected, and still actionable, but the persistence and cross-client scope need maintainer/product direction before a safe automated repair PR.

## What This Changes

Request configurable WebChat chat-input autocomplete or suggestions for project-specific terms, especially to help typing and dictation of proper nouns.

## Best Possible Solution

Keep the issue open until WebChat/Control UI owners choose the persistence and client scope, then implement custom-term suggestions in the chat composer with keyboard and ARIA behavior; add config/schema/docs only if the selected model is gateway-backed rather than browser-local.

## Reproduction Assessment

Yes. For this feature absence, inspecting the current composer, slash-menu implementation, tests, docs, and gateway WebChat config is a high-confidence reproduction path, and those surfaces do not expose configurable custom-term suggestions.

## Solution Assessment

Unclear. Autocomplete is a reasonable direction, but the proposed storage choices are intentionally open; the narrowest maintainable solution depends on whether maintainers want browser-local UI state or a shared gateway/workspace contract.
