## Summary

Keep open: current main and the latest release still expose only light/dark TUI palette selection, while named theme support would add public CLI/config surface that needs maintainer product/API direction. Prior TUI theme PRs are closed unmerged, so there is no safe canonical implementation to supersede this issue.

## What This Changes

Request to add named TUI theme selection through an `openclaw tui --theme` override and/or a `tui.theme` config default.

## Best Possible Solution

Maintainers should first define a narrow TUI theme contract, then implement theme selection through the TUI palette layer with CLI/config schema, docs, and focused accessibility/PTTY coverage.

## Reproduction Assessment

Not applicable; this is a feature/config request rather than a bug. Source and docs show the current supported controls are only `OPENCLAW_THEME=light|dark` plus `COLORFGBG`, with no named TUI theme flag or config key.

## Solution Assessment

Unclear; a flag plus config default is plausible, but supported theme names, palette mapping, precedence, validation, and accessibility requirements need maintainer design first. The safer implementation would use the TUI palette layer rather than treating bundled Shiki themes as drop-in UI palettes.
