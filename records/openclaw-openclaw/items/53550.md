## Label Justifications

- `P1`: A live user report shows a broken cross-session recall workflow affecting agent continuity across channel/gateway sessions.
- `impact:session-state`: The issue is about session transcripts existing on disk and in QMD but not surfacing through `memory_search`, causing agent state and recall drift.
- `impact:security`: The remaining fix area touches session visibility and sandbox clamps, so broadening recall can affect privacy and authorization boundaries.

## Summary

Keep open. Current main still leaves the gateway-to-DM session recall problem unresolved: QMD session export, session-memory sources, session-tool visibility, sandbox visibility clamps, and archived QMD transcript mapping remain separate gates, and the open related PR only adds diagnostics/docs rather than changing recall behavior.

## What This Changes

The issue requests that `memory_search` surface gateway-dispatched session transcripts when session memory includes `sessions`, including QMD-backed and cross-session searches.

## Best Possible Solution

Keep conservative defaults, land or supersede the diagnostic/docs work, add raw-vs-filtered session-hit diagnostics, fix archived QMD transcript stem mapping under existing visibility rules, and make any broader same-agent gateway recall a deliberate maintainer privacy decision with regression coverage.

## Reproduction Assessment

Yes, at source level. Current main still keeps QMD session export behind `memory.qmd.sessions.enabled`, filters session hits through `tools.sessions.visibility` and sandbox clamps, and appears to mishandle QMD-exported archived transcript names before mapping them back to session keys.

## Solution Assessment

Unclear as a broad recall change. The narrow archived-path and diagnostic gaps are good fix targets, but making gateway-to-DM recall work by default needs maintainer privacy and product direction rather than an automatic broad visibility change.
