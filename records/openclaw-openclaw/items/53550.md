## Label Justifications

- `P2`: The issue affects cross-session recall reliability in a real memory workflow, but it is bounded to memory/session search policy rather than an emergency outage.
- `impact:session-state`: The report is about indexed session transcripts not surfacing across related sessions, causing agent memory/session state drift.
- `impact:message-loss`: Session transcript content that exists on disk or in the index can be silently dropped from `memory_search` results after filtering.
- `impact:security`: The fix direction touches session transcript visibility, sandbox clamps, and privacy boundaries for sensitive conversation data.

## Summary

Keep open: current main has fixed the narrower archived QMD path-mapping sub-bug, but the central gateway-to-DM/session-backed `memory_search` problem is still unresolved because QMD transcript export remains a separate opt-in gate, session hits are still visibility-filtered by the session-tools policy, and open related PRs only cover diagnostics or narrower orphan-hit variants.

## What This Changes

The issue requests that opt-in session memory and `memory_search` surface gateway-dispatched session transcript content across sessions, especially for QMD-backed gateway-to-DM recall.

## Best Possible Solution

Define the session-memory contract first, then land narrow changes that keep privacy defaults explicit while adding either clearer diagnostics/docs/doctor guidance or tested opt-in behavior for QMD export and same-agent gateway-to-DM recall.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main shows `experimental.sessionMemory` gates the session source, QMD export is a separate `memory.qmd.sessions.enabled` gate, and session hits are post-filtered by default `tree` visibility and sandbox clamps. I did not run the reporter's full live QMD/Gemini/gateway setup.

## Solution Assessment

Unclear as a single code change: the narrow sub-bugs are fixable, but the full issue needs a maintainer decision on whether this should be diagnostics-only, opt-in cross-session recall, or a changed QMD/session-memory contract. The safest path is to preserve privacy defaults and make the required gates and filtered-hit reasons explicit before widening behavior.
