## Summary

Keep open. Current main still leaves the reported gateway-to-DM recall path unresolved: `experimental.sessionMemory` enables the builtin session source, but QMD transcript export remains controlled by a separate disabled-by-default `memory.qmd.sessions.enabled` switch, and session-sourced results are filtered by session visibility whose default `tree` mode hides unrelated same-agent gateway sessions from a DM query.

## What This Changes

The issue requests that `memory_search` include gateway-dispatched session transcripts when `agents.defaults.memorySearch.experimental.sessionMemory` is enabled with `sources: ["memory", "sessions"]`, including QMD/Gemini and builtin-index behavior plus observability/docs.

## Best Possible Solution

Define one explicit transcript recall contract: either make QMD honor `experimental.sessionMemory` plus `sources: ["sessions"]` with session export, status, and logs, or keep QMD export opt-in but surface an actionable diagnostic/docs path requiring `memory.qmd.sessions.enabled: true` and intentional `tools.sessions.visibility` widening for same-agent gateway-to-DM recall. Add regression coverage for a gateway-style same-agent session queried from an unrelated DM session.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with `memory.backend: "qmd"` and only `experimental.sessionMemory: true`, QMD `sessions.enabled` remains false, so no session collection/export is created; if QMD sessions are enabled separately, default `tools.sessions.visibility: "tree"` still filters unrelated same-agent gateway hits from a DM requester.

## Solution Assessment

Unclear. The requested behavior is valid, but the best fix requires maintainer judgment on whether QMD should inherit the experimental session-memory flag or remain a separate opt-in with stronger diagnostics, docs, and visibility guidance.
