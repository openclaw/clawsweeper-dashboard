## Summary

Keep open: the concrete large-transcript Gateway hang is still source-reproducible on current main and the latest release. The existing byte-size guard is opt-in, and the broader recovery thread at https://github.com/openclaw/openclaw/issues/63998 does not make this focused report obsolete.

## What This Changes

The issue reports gateway hangs and WeChat/Feishu channel freezes after updates, and asks for stronger stability, release notes, rollout, and rollback practices.

## Best Possible Solution

Add a default load-before-open recovery guard that stats and stream-inspects active session JSONL before repair/open/compaction, rotates or quarantines pathological transcripts into a bounded successor, and caps repeated full-bootstrap marker appends.

## Reproduction Assessment

Yes, as a high-confidence source-level reproduction: current main still fully reads/parses active session JSONL before the embedded run opens, matching the reported 89 MB queue-acquired stall. I did not run the full live Gateway hang locally because this review is read-only.

## Solution Assessment

Unclear for the broad release-process asks, but the code fix direction is clear: the opt-in byte guard is not enough for default stability, and the safer repair is a default pre-open guard plus bounded recovery rather than another request-time cache.
