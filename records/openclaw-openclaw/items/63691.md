## Repair Work Prompt

Fix the active-transcript oversized JSONL Gateway hang tracked by https://github.com/openclaw/openclaw/issues/63691. The concrete path is an embedded Gateway run acquiring the bootstrap/session queue, then reading/parsing a very large active session JSONL before the Gateway can keep HTTP/RPC/channel handlers responsive. Current main still calls repairSessionFileIfNeeded and then SessionManager.open in src/agents/pi-embedded-runner/run/attempt.ts, and compact.ts has the same repair/open pattern; the pinned @mariozechner/pi-coding-agent 0.71.1 SessionManager.open path synchronously loads the full file twice. Add a bounded default guard before active transcript repair/open for embedded runs and compaction: stat/classify oversized JSONL before whole-file parsing, keep the event loop responsive, and either use existing compaction/rotation recovery where safe or fail fast with an actionable recovery message. Keep the fix generic to sessions/embedded runs, not WeChat/Feishu-specific. Do not change dependency pins, provider defaults, or channel behavior without maintainer approval. Add focused tests proving an oversized transcript does not go through whole-file repair/open before the guard, update docs if config/recovery behavior changes, and add a user-facing CHANGELOG.md fix entry.
