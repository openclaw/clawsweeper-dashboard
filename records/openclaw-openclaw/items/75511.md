## Summary

Keep open. The PR adds the public config field and correctly starts threading it into loader/startup behavior, but the opt-out is still incomplete because other automatic runtime-dependency repair paths that consume persisted config can run when the field is false, and the user-facing feature is missing the required changelog entry.

## What This Changes

The PR adds optional `plugins.installBundledRuntimeDeps` config schema/types/help/docs/generated metadata, gates bundled runtime-dependency install in plugin loading and gateway startup prestage, and adds focused config, loader, and startup tests.

## Best Possible Solution

Complete the public opt-out by making every automatic bundled runtime-deps install/repair path that uses persisted config honor `plugins.installBundledRuntimeDeps === false`, while keeping explicit manual repair commands available and leaving the broader isolated-execution-root default discussion to #75032.

## Reproduction Assessment

Yes. Code inspection gives a high-confidence reproduction path: apply the PR, run a config hot reload with `planPluginRuntimeDeps: true` or a local configure/onboard post-config materialization while `plugins.installBundledRuntimeDeps` is false, and the unchanged repair helpers can still call `repairBundledRuntimeDepsPackagePlanAsync`.

## Solution Assessment

No. The loader and startup changes are the right direction, but the PR is not the best complete solution until hot reload, post-config materialization, tests, and the changelog are aligned with the new persisted config contract.
