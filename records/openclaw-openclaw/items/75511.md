## Summary

Keep open. The PR is useful and moves the public opt-out in the right direction, but the persisted config field is still only honored by the loader and one startup repair path; current code shows hot reload, post-config materialization, and source-checkout pre-stage can still automatically repair or install bundled runtime dependencies.

## What This Changes

The PR adds a `plugins.installBundledRuntimeDeps` boolean config field with docs/generated schema updates, skips bundled runtime-dependency install in loader and gateway startup prestage when false, and adds focused config, loader, and startup tests.

## Best Possible Solution

Complete the public opt-out as a single, consistent config contract: every automatic bundled runtime-dependency install or repair path that reads persisted config should skip package-manager repair when `plugins.installBundledRuntimeDeps === false`, while explicit manual repair commands remain available and the broader isolated-execution-root default work stays with #75032.

## Reproduction Assessment

Yes. Code inspection gives high-confidence paths: apply the PR, set `plugins.installBundledRuntimeDeps: false`, then trigger a hot reload with `planPluginRuntimeDeps: true`, run local configure/onboard post-config materialization, or start from a source checkout with missing bundled deps; the remaining ungated helpers can still repair/install.

## Solution Assessment

No. The PR's loader and packaged-startup changes are the right direction, but the implementation is incomplete until source-checkout pre-stage, hot reload, post-config materialization, tests, and the changelog all honor the same persisted opt-out.
