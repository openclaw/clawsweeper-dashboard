## Summary

Keep open. The PR is a useful narrow implementation of the public opt-out, but it is not merge-ready because the config hot-reload path can still run bundled runtime-dependency repair despite the new persisted opt-out, and the required user-facing changelog entry is still missing.

## What This Changes

The PR adds `plugins.installBundledRuntimeDeps` to config schema, types, help, docs, and generated metadata, threads it into plugin loader and gateway startup prestage behavior, and adds focused config, loader, and gateway tests.

## Best Possible Solution

Land the public opt-out after applying the same skip policy to config hot reload, adding focused regression coverage and the required changelog entry, while leaving the broader isolated execution-root default discussion to #75032.

## Reproduction Assessment

Yes. Current main lacks the persisted field and still repairs during startup; on the PR head, a config hot reload with `planPluginRuntimeDeps: true` and `plugins.installBundledRuntimeDeps: false` still reaches `repairBundledRuntimeDepsPackagePlanAsync`.

## Solution Assessment

No. The startup and loader changes are the right direction, but the implementation is incomplete until every automatic runtime-deps repair path that consumes persisted config honors the same opt-out.
