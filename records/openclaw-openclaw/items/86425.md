## Summary

Keep open: current main and the latest release still support browser Realtime Talk only as audio plus the existing agent/control tool path, with no `describe_view` tool or browser camera-frame injection. The request is concrete, but it crosses camera permissions, provider-specific Realtime image payloads, and the gateway/tool contract, so it needs maintainer product and security review before implementation.

## What This Changes

Request adding a `describe_view` Realtime Talk tool that captures one browser camera frame for OpenAI WebRTC sessions and reports graceful unsupported behavior for other transports.

## Best Possible Solution

Define the camera permission UX and Realtime tool/provider contract first, then implement a one-shot browser-owned OpenAI WebRTC frame capture path with explicit unsupported responses, focused tests, and docs.

## Solution Assessment
