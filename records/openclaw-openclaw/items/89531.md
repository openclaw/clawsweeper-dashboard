## Summary

Keep open. Current main still preserves separate `final_answer` message items from the Responses stream, and channel docs say ordinary channel delivery should not emit token-delta messages when streaming is off, so the report points to an adapter/provider normalization bug rather than a solved channel issue.

## What This Changes

Deduplicate or collapse cumulative OpenAI Responses `final_answer` snapshots from the Bedrock OpenAI-compatible route so downstream channels receive one final answer for a turn.

## Best Possible Solution

Normalize cumulative duplicate `final_answer` snapshots in the OpenAI Responses adapter/provider boundary so one assistant response persists one final answer block, with regression coverage and Bedrock/Mantle live proof.

## Reproduction Assessment

No live Bedrock Mantle repro was run here; source inspection gives a medium-confidence path where several cumulative `response.output_item.done` message events with phase `final_answer` are preserved and later joined as multiple visible final-answer blocks.

## Solution Assessment

Likely yes: the best fix belongs at the Responses adapter/provider boundary rather than Telegram or Discord because the duplicate blocks already exist in assistant content before channel delivery.
