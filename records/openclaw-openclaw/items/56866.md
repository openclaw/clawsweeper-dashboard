## Summary

Keep this PR open: the WhatsApp ACP binding feature is still not present on current main, but the proposed branch is not safe to merge as-is because it reimplements ACP file loading without current main's canonical-path protections and broadens ACP failure cleanup to unbind every conversation sharing a target session.

## What This Changes

The PR adds WhatsApp configured ACP binding resolution, WhatsApp media/history threading into ACP turns, ACP prompt lifecycle hooks, WhatsApp ACP block-delivery visibility handling, and ACP binding schema validation for WhatsApp peer IDs.

## Best Possible Solution

Rework the WhatsApp ACP feature on top of current main: keep the configured binding seam in the WhatsApp plugin/on-message path, thread media through existing safe attachment abstractions rather than direct fs.readFile, avoid exposing host paths in prompts, and keep ACP binding cleanup scoped to confirmed stale sessions or the current binding record.

## Reproduction Assessment

Yes. The security findings have static reproduction paths: an allowed-root path that traverses through an intermediate symlink reaches fs.readFile without realpath containment, and any ACP_TURN_FAILED path calls unbind by targetSessionKey, which removes all bindings for that target.

## Solution Assessment

No. The feature direction is useful, but this implementation bypasses current main's hardened media cache and expands failure cleanup too broadly; the safer solution is to reuse the existing ACP attachment/cache path and only add narrow WhatsApp binding/media seams.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Recheck real paths before reading ACP attachments:** `src/auto-reply/reply/dispatch-acp.ts:117-139`
  - body: `isInboundPathAllowed` is checked against the raw path, then only the final path component is checked with `lstat` before `fs.readFile`. A path under an allowed root that crosses an intermediate symlink can still resolve outside the root and be attached to the ACP turn. Reuse `MediaAttachmentCache` or canonicalize both file and roots with `realpath` before opening the file.
  - confidence: 0.93
- **[P1] Limit ACP failure unbinding to stale bindings:** `src/auto-reply/reply/dispatch-acp.ts:593-600`
  - body: The catch path now unbinds by `targetSessionKey` for every `ACP_TURN_FAILED`. The binding service removes all bindings with that target key, so one failing inbound turn can detach other conversations that intentionally share the same ACP session. Keep the existing stale-init guard or resolve and unbind only the current conversation's binding id.
  - confidence: 0.91
- **[P2] Remove local media paths from prompt history:** `src/auto-reply/reply/inbound-meta.ts:249-255`
  - body: This writes the host `mediaPath` into the untrusted chat-history prompt sent to ACP harnesses and plugin hooks. Since the image bytes are attached separately, expose an opaque media reference or placeholder instead of leaking local filesystem layout.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR introduces concrete security and availability regressions in ACP local-file reads and session-binding cleanup.

Concerns:

- **[high] ACP attachment path traversal via intermediate symlinks:** `src/auto-reply/reply/dispatch-acp.ts:139`
  - body: The branch validates the raw path and rejects only final symlinks before fs.readFile, so an allowed-root path through a symlinked directory can read files outside the allowed roots.
  - confidence: 0.93
- **[high] Generic ACP failures can unbind shared conversations:** `src/auto-reply/reply/dispatch-acp.ts:599`
  - body: Unbinding by targetSessionKey on ACP_TURN_FAILED removes every binding for that session key, making a single failed message capable of detaching unrelated conversations that share the ACP session.
  - confidence: 0.91
- **[medium] Local media paths leak into ACP prompts:** `src/auto-reply/reply/inbound-meta.ts:255`
  - body: The history prompt includes local mediaPath values, exposing host filesystem details to the ACP harness and prompt-build hooks when an opaque placeholder would be enough.
  - confidence: 0.82
