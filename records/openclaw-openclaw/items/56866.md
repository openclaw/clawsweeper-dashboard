## Summary

Keep open: current main still lacks WhatsApp configured ACP binding support, but this PR should not merge as-is because it introduces security-sensitive ACP file-read and binding-cleanup regressions on a drifted code path.

## What This Changes

The PR adds WhatsApp configured ACP session binding, WhatsApp media/history threading into ACP turns, ACP lifecycle hook dispatch, visible block-delivery handling, and ACP binding schema validation for WhatsApp peers.

## Best Possible Solution

Rework the WhatsApp ACP feature on top of current main: keep the WhatsApp configured-binding seam, use the existing MediaAttachmentCache/opaque attachment path for media, avoid prompt-visible host paths, and keep ACP binding cleanup scoped to confirmed stale sessions or the current binding record.

## Reproduction Assessment

Yes. The findings have static reproduction paths: an allowed-root path through an intermediate symlink reaches the PR's fs.readFile path, and any ACP_TURN_FAILED path calls unbind by targetSessionKey, which removes all bindings for that target session.

## Solution Assessment

No. The feature direction is useful, but the implementation bypasses current main's hardened attachment cache and expands failure cleanup too broadly; the safer solution is a current-main rework with narrow WhatsApp binding/media seams.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Recheck real paths before reading ACP attachments:** `src/auto-reply/reply/dispatch-acp.ts:117-139`
  - body: The PR validates the raw path and rejects only a final symlink before calling fs.readFile. A path under an allowed root that crosses an intermediate symlink can still resolve outside the root and be attached to the ACP turn; reuse MediaAttachmentCache or compare canonical file and root realpaths before opening the file.
  - confidence: 0.93
- **[P1] Limit ACP failure unbinding to stale bindings:** `src/auto-reply/reply/dispatch-acp.ts:593-600`
  - body: The catch path unbinds by targetSessionKey for every ACP_TURN_FAILED. The binding service removes all bindings with that target key, so one failed inbound turn can detach other conversations sharing the same ACP session; keep cleanup to confirmed stale-session errors or unbind only the current binding id.
  - confidence: 0.91
- **[P2] Remove local media paths from prompt history:** `src/auto-reply/reply/inbound-meta.ts:249-255`
  - body: This writes the host mediaPath into the chat-history prompt sent to ACP harnesses and prompt-build hooks. Since image bytes should be attached separately, expose an opaque placeholder or media reference instead of leaking local filesystem layout.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR introduces concrete security and availability regressions in ACP local-file reads, session-binding cleanup, and prompt-visible host paths.

Concerns:

- **[high] ACP attachment path traversal via intermediate symlinks:** `src/auto-reply/reply/dispatch-acp.ts:139`
  - body: The branch validates the raw attachment path and rejects only final symlinks before fs.readFile, so an allowed-root path through a symlinked directory can read files outside configured roots.
  - confidence: 0.93

  - body: Unbinding by targetSessionKey on ACP_TURN_FAILED removes every binding for that session key, making a single failed message capable of detaching unrelated conversations that share the ACP session.
