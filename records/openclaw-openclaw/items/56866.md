Rebase and split this into maintainer-approved WhatsApp configured-binding support that reuses current hardened media attachment helpers, keeps local paths out of prompts, and includes redacted live WhatsApp ACP proof.

Yes. Source inspection of the PR head gives high-confidence paths for the review findings: raw path validation before fs.readFile, target-session unbind on generic ACP errors, and mediaPath serialization into prompt text.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Recheck real paths before reading ACP attachments:** `src/auto-reply/reply/dispatch-acp.ts:117-139`
  - body: The PR checks only the raw path with isInboundPathAllowed and rejects only the final symlink before fs.readFile. A path under an allowed root that crosses an intermediate symlink can still attach a file outside configured roots; reuse MediaAttachmentCache or compare canonical file/root realpaths before opening.
  - confidence: 0.93
- **[P1] Limit ACP failure unbinding to stale bindings:** `src/auto-reply/reply/dispatch-acp.ts:595-600`
  - body: The catch path unbinds by targetSessionKey for every ACP_SESSION_INIT_FAILED or ACP_TURN_FAILED. The generic binding store removes every binding for that target session, so one failed inbound turn can detach other conversations sharing the ACP session; limit this to confirmed stale sessions or the current binding id.
  - confidence: 0.91
- **[P2] Remove local media paths from prompt history:** `src/auto-reply/reply/inbound-meta.ts:255`
  - body: This writes host mediaPath values into the chat-history prompt sent to ACP harnesses and prompt-build hooks. Image bytes should travel through the attachment channel; prompt history should use an opaque media reference or redacted media hint instead of exposing local filesystem layout.
  - confidence: 0.82
