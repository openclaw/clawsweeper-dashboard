## Summary

Keep open: current main still lacks the run-scoped announce tracking, but this branch remains incorrect because grouped/dropped queue paths can leave source runs pending, restore cleanup double-persists, and real behavior proof is absent.

## What This Changes

The PR threads `sourceRunId` through subagent and media announce delivery, adds pending/delivered announce fields plus stale-restore cleanup, and adds focused regression tests.

## Best Possible Solution

Keep the run-scoped announce-state design, but clear or deliver every represented source run in queue batching/drop paths, make restore persistence single-owner, then require redacted real-run proof before merge.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the PR blockers: collect mode sends grouped items as `...last`, queue overflow drops old items into summaries without a source callback, and the PR persists stale cleanup in both helper and caller.

## Solution Assessment

No. The direction is reasonable, but the current patch is not mergeable until queue delivery/drop state and restore persistence ownership are corrected and real behavior proof is supplied.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Mark every collected announce source as delivered:** `src/agents/subagent-announce-delivery.ts:306-310`
  - body: `collect` mode drains several queued items by sending one aggregate built from `...last`, so this new delivered-state call receives only the last item’s `sourceRunId`. Earlier runs in the same grouped prompt keep `pendingAnnounceId` and never resume after the batch succeeds; carry all grouped source run IDs through delivery or clear the grouped pending markers together.
  - confidence: 0.9
- **[P2] Clear pending state for announces dropped by queue cap:** `src/agents/subagent-announce-delivery.ts:403-409`
  - body: After a run is marked pending, a later enqueue can make `applyQueueDropPolicy` drop that older queue item into the overflow summary. Because dropped items no longer reach `sendAnnounce`, their `pendingAnnounceId` is never cleared or resumed; return the dropped source IDs from the queue helper or clear them when the summary represents them.
  - confidence: 0.82
- **[P2] Let only one layer persist stale announce cleanup:** `src/agents/subagent-registry.ts:487-489`
  - body: `clearStaleSubagentAnnouncePendingState()` already flushes `subagentRuns` when it clears pending fields, and this restore block immediately calls `persistSubagentRuns()` again on the truthy return. Make either the helper or the restore caller own that disk write.
  - confidence: 0.93
