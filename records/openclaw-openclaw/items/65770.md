## Summary

Keep open. Current main still lacks the run-scoped announce source/run-state fields this PR adds, so the work is not obsolete; however, the patch still has a concrete duplicate-persistence bug on restore and should get a small cleanup before merge.

## What This Changes

The branch threads `sourceRunId` through subagent and media announce delivery, adds pending/delivered announce fields to subagent run records, clears stale pending announce state on restore, and adds focused registry/delivery regression tests.

## Best Possible Solution

Keep the run-scoped announce-state direction, make stale-restore persistence single-owner, add the required release note if maintainers consider the behavior user-facing, rebase onto current main, and validate the affected subagent announce/registry lanes before merge.

## Reproduction Assessment

Yes. Source inspection reproduces both sides: current main lacks the requested run-scoped state, and the PR diff shows `clearStaleSubagentAnnouncePendingState()` persisting internally while the restore caller persists again on a true return.

## Solution Assessment

No, not as currently proposed. The feature boundary is narrow and maintainable, but the restore cleanup should either self-persist or return a mutation signal for the caller to persist, not both.
