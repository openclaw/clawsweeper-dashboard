## Summary

Keep open. Current main still lacks run-scoped announce source IDs and pending/delivered announce state, so this PR is not obsolete, but the branch still needs a small correctness repair before merge because stale pending cleanup currently persists in both the helper and restore caller.

## What This Changes

The PR threads `sourceRunId` through subagent and media announce delivery, adds pending/delivered announce fields plus stale-restore cleanup for subagent run records, and adds focused regression tests.

## Best Possible Solution

Keep the run-scoped announce-state design, make stale pending cleanup persistence single-owner, add the active-version changelog entry if maintainers treat this reliability change as user-facing, then rebase and validate the affected subagent announce and registry lanes before merge.

## Solution Assessment

No, not as currently proposed. The feature boundary is narrow and maintainable, but restore cleanup should either self-persist or return a mutation signal for the caller to persist, not both.
