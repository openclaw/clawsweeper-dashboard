## Summary

Keep this PR open: the run-scoped announce-state direction is useful and not already on current main, but the branch still has correctness blockers, lacks required real-run proof, and is conflicting with current main.

## What This Changes

The branch threads `sourceRunId` through subagent and media announce delivery, adds pending/delivered announce IDs to run records, clears stale pending state on restore, and adds focused regression tests.

## Best Possible Solution

Integrate the run-scoped announce-state idea on current main with grouped-source delivery cleanup, single-owner restore persistence, focused regression coverage, and redacted real-run proof.

## Reproduction Assessment

Yes. Source inspection of the PR head shows collect mode aggregating multiple announce items into one send that marks only the last source run delivered, and restore cleanup persisting in both the helper and caller.

## Solution Assessment

No. The direction is reasonable, but the current branch should clear every represented source run on grouped delivery, make restore persistence single-owner, rebase onto current main, and provide real-run proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Mark every collected announce source as delivered:** `src/agents/subagent-announce-delivery.ts:309-313`
  - body: `collect` mode drains multiple queued announce items into one aggregate send built from the last item, but this delivery path marks only that last item's `sourceRunId` delivered. Earlier grouped runs can keep `pendingAnnounceId` set and miss the resume callback after the aggregate announce succeeds; carry every grouped source run through the send or clear all represented pending markers on success.
  - confidence: 0.86
- **[P2] Make stale announce cleanup persistence single-owner:** `src/agents/subagent-registry.ts:490-491`
  - body: `clearStaleSubagentAnnouncePendingState()` already flushes when it clears pending announce fields, and this restore block immediately calls `persistSubagentRuns()` again on a truthy return. Pick one layer to own the disk write so cold-start restore does not perform duplicate identical flushes.
  - confidence: 0.92
