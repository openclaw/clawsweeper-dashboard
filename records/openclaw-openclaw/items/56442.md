## Summary

Keep open: this is a plausible implementation path for the ACP parent-completion wake gap, and related maintainer discussion points at this direction, but the PR still routes notify-mode progress into active parent steering, needs current-main contract integration, and lacks real behavior proof.

## What This Changes

Adds a `parentUpdates: "notify"` ACP `sessions_spawn` option, relay/announce routing, event-source typing, docs, and tests so ACP run completions can be announced back to the parent session.

## Best Possible Solution

Rebase onto current main and land a maintainer-approved terminal-only ACP completion notify contract that keeps ACP progress on untrusted system events, updates the central internal event-source contract, and proves a live parent-completion flow.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head shows notify-mode progress using `queueEmbeddedPiMessage`, and current main shows that queue can inject into active parent or reply runs. I did not run a live ACP session in this read-only review.

## Solution Assessment

No, not as currently proposed. The direction is useful, but notify should own terminal completion only, the current central event contract needs to be updated on rebase, and the change needs real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep ACP progress on the system-event relay:** `src/agents/acp-spawn-parent-stream.ts:291-293`
  - body: In notify mode, progress updates call `tryNotifyParentRun` and queue child output into the active parent run before falling back to system events. That contradicts the PR contract that `streamTo` controls lightweight progress relay while `parentUpdates` controls terminal completion, and it lets untrusted ACP progress steer the parent session live.
  - confidence: 0.86
- **[P2] Add ACP to the central event-source list:** `src/agents/subagent-announce.ts:550`
  - body: Current main now derives the internal-event type and gateway schema from `AGENT_INTERNAL_EVENT_SOURCES`, but this PR only updates the old local union/schema. After rebase, emitting `source: "acp"` from the announce flow must be backed by the central source list or the new event source will fail type/schema validation.
  - confidence: 0.78
- **[P3] Cache failed parent-session lookups:** `src/agents/acp-spawn-parent-stream.ts:248-249`
  - body: `cachedParentSessionId` is only effective for truthy session IDs, so a parent entry with a missing or empty `sessionId` repeats config/store loading on every progress update. Cache the lookup attempt separately so a known miss does not keep doing disk work on the relay path.
  - confidence: 0.72
