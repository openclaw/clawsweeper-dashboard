## Summary

Keep open: current main still does not implement the opt-in ACP notify API, and related triage treats this PR as the canonical direction, but the branch is not mergeable as written because it routes ACP progress into active parent steering, predates the current internal-event contract, and lacks real behavior proof.

## What This Changes

The PR adds `parentUpdates?: "system" | "notify"` for ACP `sessions_spawn`, plus docs and tests, so ACP run completions can be announced to the parent session through subagent-style completion delivery.

## Best Possible Solution

Rebase this as a terminal-only ACP completion notify contract on current main: keep progress on untrusted system events, add `acp` to the central internal-event source contract if the notify mode is accepted, preserve current task-registry completion behavior, and require focused tests plus redacted real ACP proof.

## Reproduction Assessment

Yes for the PR review findings by source inspection: PR-head `emitProgressUpdate` calls `tryNotifyParentRun` and then `queueEmbeddedPiMessage`, while current main defaults that queue to `steeringMode: "all"`. I did not run a live ACP session in this read-only review.

## Solution Assessment

No, not as currently proposed. The terminal notify direction is plausible and canonical, but the branch should be rebased and narrowed so only terminal completion uses announce flow, progress stays on system events, and the current central event-source contract is updated.
