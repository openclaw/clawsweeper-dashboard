## Summary

Keep this PR open for maintainer review. Current main has overlapping ACP background-task delivery, but it still does not implement the PR's explicit opt-in `parentUpdates: "notify"` API, and the PR branch introduces a concrete security/design concern by steering raw ACP child progress into the active parent run.

## What This Changes

The PR adds an ACP `parentUpdates` option to `sessions_spawn`, routes notify-mode ACP completion through subagent-style announce flow, updates ACP relay behavior, expands internal-event source typing, and documents/tests the new mode.

## Best Possible Solution

Rebase or replace the PR on current main's ACP background-task delivery model, then implement only the chosen completion contract: either an explicit `parentUpdates: "notify"` parameter or a generalized task-completion path, with ACP child output preserved as sanitized internal-event data rather than raw parent-run steering.

## Reproduction Assessment

Yes. Static inspection of current main reproduces the missing API (`parentUpdates` is absent from `SpawnAcpParams` and the tool schema), and the related #70264 discussion supplies a concrete ACP parent-completion symptom; I did not run tests because this was a read-only review.

## Solution Assessment

No. The PR targets a real gap, and it fixed the duplicate-delivery regression raised in review, but the current patch is not the best merge shape because it injects raw notify-mode progress into the active parent run and overlaps with newer task-registry delivery behavior on main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Keep ACP progress out of parent run steering:** `src/agents/acp-spawn-parent-stream.ts:274`
  - body: `parentUpdates: "notify"` is documented as terminal completion routing, but this path queues every cleaned progress snippet directly into the active parent embedded run. ACP child output is untrusted, so progress should stay on the system-event/progress relay path or be wrapped as structured internal context instead of steering the parent run as plain text.
  - confidence: 0.86
- **[P3] Cache failed parent session lookups:** `src/agents/acp-spawn-parent-stream.ts:242-249`
  - body: When the parent session exists but has no usable `sessionId`, `cachedParentSessionId` remains undefined and every later notify attempt repeats config/store loading. Track that resolution was attempted so repeated progress flushes do not keep hitting disk after a stable miss.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR introduces a security-sensitive prompt-steering path for untrusted ACP child progress.

Concerns:

- **[high] Raw ACP progress can steer the parent run:** `src/agents/acp-spawn-parent-stream.ts:274`
  - body: Notify-mode progress calls `queueEmbeddedPiMessage(parentSessionId, cleaned)`, which treats ACP child output as active parent-run input rather than an untrusted system event or sanitized internal completion event. A malicious or compromised ACP harness could use progress text to influence the parent agent.
