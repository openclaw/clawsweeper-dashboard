## Summary

Keep open. Current main does not expose the proposed `parentUpdates` ACP `sessions_spawn` API, so the PR is not implemented-on-main, but the branch still needs maintainer review because it changes ACP API/delivery semantics and has source-backed correctness and security blockers around current-main task delivery and untrusted ACP progress steering.

## What This Changes

The PR adds an optional ACP `sessions_spawn` `parentUpdates` mode, routes ACP run terminal completion through subagent-style announce flow, and updates related docs, tests, and internal event source typing.

## Best Possible Solution

Rebase onto current main and implement a narrow terminal-only ACP completion mode that suppresses every other user-facing completion path, keeps ACP progress on the untrusted system-event relay, and updates docs/protocol/tests for the final contract.

## Reproduction Assessment

Yes for the review blockers: source inspection shows current main still registers non-`streamTo` ACP spawns for task-registry terminal delivery, and the PR head queues notify-mode progress through embedded parent steering. I did not run tests because this was a read-only review.

## Solution Assessment

No, not as currently proposed. The API direction may be useful, but the branch must first integrate with current-main task delivery and reserve notify/announce semantics for terminal completion rather than live progress steering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Suppress registry delivery for notify-only ACP spawns:** `src/agents/tools/sessions-spawn-tool.ts:180`
  - body: `parentUpdates: "notify"` can be used without `streamTo: "parent"`, which starts a parent completion relay. On current main, ACP `sessions_spawn` still registers background-task delivery whenever `streamTo !== "parent"`, so a rebased notify-only spawn can emit both the subagent-style announce and the existing background task completion. Gate task tracking on notify mode as well, or have `spawnAcpDirect` return an explicit signal that terminal delivery is already owned by the parent relay.
  - confidence: 0.84
- **[P2] Keep ACP progress out of embedded parent steering:** `src/agents/acp-spawn-parent-stream.ts:291-293`
  - body: In notify mode, progress updates try `queueEmbeddedPiMessage` before falling back to system events. That makes untrusted ACP child output a live steering message for the parent run, even though the PR design says `parentUpdates` controls terminal completion and `streamTo` remains the lightweight progress relay. Keep progress on the system-event path and reserve notify/announce for terminal completion.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff adds no dependency or CI execution path, but it introduces a prompt-injection-sensitive routing change for ACP child output.

Concerns:

- **[medium] Untrusted ACP progress enters parent steering:** `src/agents/acp-spawn-parent-stream.ts:291`
  - body: `parentUpdates: "notify"` plus `streamTo: "parent"` routes ACP progress snippets through `queueEmbeddedPiMessage`, whose active-run path can steer the parent session. ACP child output should remain untrusted progress/system-event text unless maintainers explicitly accept steering semantics.
