## Summary

Keep open: this PR is still the canonical open path for ACP parent completion notify work after https://github.com/openclaw/openclaw/issues/70264 was closed as superseded by it, and current main still lacks the proposed `parentUpdates` API. The branch is not mergeable as-is because it routes ACP progress into parent-run steering, predates the current central internal-event source contract, is conflict-marked, and lacks real behavior proof.

## What This Changes

The PR adds a `parentUpdates` mode to ACP `sessions_spawn`, plus docs and tests, to route terminal ACP run completion through subagent-style announce handling while keeping `streamTo` as the progress relay switch.

## Best Possible Solution

Rework this as a terminal-only ACP completion notify contract on current main: keep progress on the untrusted system-event relay, add `acp` to the central event-source contract, preserve the current task-registry completion behavior, and require focused tests plus redacted real ACP proof.

## Reproduction Assessment

Yes for the review findings via source inspection: PR-head progress calls `tryNotifyParentRun` and then `queueEmbeddedPiMessage`, while current main defaults that queue to steering-capable active-run input. I did not run a live ACP session in this read-only review.

## Solution Assessment

No, not as currently proposed. The direction remains plausible and canonical, but the branch should be rebased and narrowed so only terminal completion uses announce flow, progress stays on system events, and the current central event-source contract is updated.
