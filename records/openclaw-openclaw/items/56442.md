## Summary

Keep open: current main still does not expose the requested `parentUpdates` ACP spawn option, and related #70264 explicitly treated this PR as the canonical path. The branch is useful but not merge-ready because the notify-only mode still collides with current task-registry delivery, and the notify progress path introduces a prompt-routing/security concern.

## What This Changes

The PR adds a `parentUpdates: "system" | "notify"` ACP `sessions_spawn` option, relay/announce plumbing, event-source typing, docs, and targeted tests for ACP parent completion notification.

## Best Possible Solution

Rebase onto current main and land a terminal-only ACP completion notify contract that integrates with the task registry, updates the current event-source contract, and keeps ACP progress on the untrusted relay unless maintainers explicitly accept live steering semantics.

## Reproduction Assessment

Yes for the review blockers: source inspection of current main shows non-`streamTo` ACP spawns still enter task-registry terminal delivery, and the PR diff routes notify-mode progress through `queueEmbeddedPiMessage`. I did not run tests because this review is read-only.

## Solution Assessment

No, not as currently proposed. The API direction may be useful, but the implementation should reserve notify/announce for terminal completion, suppress other completion owners, and align with current main's task and protocol contracts.
