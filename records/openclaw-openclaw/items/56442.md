## What This Changes

The PR adds `parentUpdates?: "system" | "notify"` for ACP `sessions_spawn`, docs, and tests so ACP run completions can use subagent-style parent completion delivery.

## Best Possible Solution

Rebase this as a terminal-only ACP completion notify mode on current main: keep progress on untrusted system events, add `acp` to the central event-source contract, preserve current task-registry completion behavior, and require focused tests plus redacted real ACP proof.

## Reproduction Assessment

Yes for the PR review findings by source inspection: PR-head progress calls `tryNotifyParentRun` and `queueEmbeddedPiMessage`, while current main defaults that queue to active-run steering. I did not run a live ACP session in this read-only review.

## Solution Assessment

No, not as currently proposed. The terminal notify direction is plausible, but the branch should be rebased and narrowed so only terminal completion uses announce flow, progress stays on system events, and the current central/source contracts are updated.
