## What This Changes

The PR adds an opt-in `parentUpdates?: "system" | "notify"` ACP `sessions_spawn` mode with relay, internal-event, docs, and tests so ACP run completions can use subagent-style announcement routing.

## Best Possible Solution

Keep the feature opt-in, but rebase it so only terminal completion/error uses announce delivery, progress stays on the existing relay path, and current delivery-context/task-registry contracts are preserved.

## Reproduction Assessment

Yes for the review findings by source inspection: PR-head `tryNotifyParentRun` sends notify-mode progress through `queueEmbeddedPiMessage`, and PR-head relay creation omits current main's delivery context/task registry parameters. I did not run a live ACP session in this read-only review.

## Solution Assessment

No: the opt-in terminal notify direction is plausible, but the PR-head implementation routes progress as active parent-run steering and is stale against current main's ACP relay contracts. The safer solution is terminal-only announce delivery plus preserved system-event progress relay.
