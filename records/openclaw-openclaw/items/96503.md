## What This Changes

The PR updates OpenAI-compatible streaming response normalization to sniff JSON-content-type bodies, relabel bodies that are actually SSE, and add regression coverage for the double-prefix parser failure.

## Best Possible Solution

Land the narrow transport fix after redacted real behavior proof and required checks, while preserving the existing true-JSON fallback and avoiding a new strict-mode config option.

## Reproduction Assessment

Yes, at source level. Current main and the latest release can wrap an already-SSE `application/json` streaming body as `data: data: ...`, and the OpenAI SDK parser then tries to parse `data: {...}` as JSON; I did not run a live gateway in this read-only review.

## Solution Assessment

Yes, this is the narrow best fix shape. Sniffing and relabeling only JSON-labeled already-SSE bodies preserves the existing true-JSON fallback and is safer than adding the requested strict-mode config option.
