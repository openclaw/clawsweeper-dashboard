## Summary

Keep open. Current main still uses strict model id equality in the context-window guard, while the active agent run path forwards the selected runtime model id into that guard; this PR is a narrow implementation for the valid #76532 bug and I found no blocking functional or security findings in the supplied diff.

## What This Changes

This PR adds provider-prefix-aware model matching in `resolveContextWindowInfo`, regression tests for bare/scoped ids in both directions, and a changelog fix entry.

## Best Possible Solution

Review and land this PR, or an equivalent targeted patch, so configured context metadata is honored when runtime and config model ids differ only by the same provider prefix.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main strict-matches `m.id` to `modelId`, and the runner forwards a selected runtime `modelId`, so a `provider/model` runtime id misses a bare configured id and falls back to runtime model metadata.

## Solution Assessment

Yes. Fixing the match inside `resolveContextWindowInfo` is the narrowest maintainable solution because that is where config context metadata is selected, and the PR adds focused regression coverage for both bare/scoped directions.
