## Summary

Keep open: current main still keeps LiteLLM/proxy Responses routes from opting into persisted response items, and the remaining work is already paired with open fix PR https://github.com/openclaw/openclaw/pull/78977 using closing syntax.

## What This Changes

Allow OpenAI Responses-compatible LiteLLM/proxy models to opt into persisted response items so multi-turn continuations do not fall back after `rs_*` item errors.

## Best Possible Solution

Add a documented provider/model Responses persistence opt-in that both central payload policy and transport paths honor, then validate with a LiteLLM/proxy multi-turn run and native/proxy regression tests.

## Reproduction Assessment

No high-confidence live reproduction is available because the failing LiteLLM proxy is private. The current source gives a clear reproduction path for the OpenClaw side: proxy-like Responses routes can emit `store:false` while replaying prior Responses item IDs.

## Solution Assessment

Yes for the explicit opt-in direction proposed in the issue discussion; no for a blanket proxy default change without product approval. The narrow maintainable path is a documented persistence policy shared by the central payload policy and transport behavior.
