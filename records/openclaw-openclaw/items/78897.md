## Summary

Keep open: current main still has the source-level `store:false` path for proxy-like OpenAI Responses routes, and an open linked PR with closing syntax is already the implementation path.

## What This Changes

The issue asks OpenClaw to let OpenAI Responses proxy routes such as LiteLLM avoid `store:false` continuation failures for `gpt-5.5` multi-turn sessions.

## Best Possible Solution

Review the linked provider payload-policy PR, decide whether proxy persistence should be an explicit compat option or an approved default, document the retention tradeoff, and close this issue only after the accepted fix lands.

## Reproduction Assessment

No, there is no high-confidence live reproduction because the failing LiteLLM proxy is private. Source inspection gives a clear current-main path: the HTTP/SSE Responses builder replays Responses item ids and applies a policy that emits `store:false` for proxy-like routes that support the store field.

## Solution Assessment

Unclear until maintainer review of the linked PR. A hardcoded LiteLLM model special case is not the best fix; the maintainable outcome is either an explicit provider/model persistence opt-in or a deliberately approved default policy for proxy-like Responses endpoints.
