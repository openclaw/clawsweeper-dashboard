## Summary

Keep open: current main still has the source-level `store:false` path for proxy-like OpenAI Responses routes, and an open linked PR with closing syntax remains the active implementation path.

## What This Changes

Request provider/model-aware OpenAI Responses `store` handling so LiteLLM/OpenAI-compatible proxy continuations do not fail after replaying prior `rs_*` items.

## Best Possible Solution

Review the linked provider payload-policy PR, decide whether proxy persistence should be an explicit compat opt-in or an approved default, document the retention tradeoff, and close this issue only after the accepted fix lands.

## Reproduction Assessment

No high-confidence live reproduction is available because the failing LiteLLM proxy is private. Source inspection does show a plausible current-main path: proxy-like Responses routes can replay prior Responses item ids while applying a policy that emits `store:false`.

## Solution Assessment

Unclear until maintainer review of the linked PR. A hardcoded LiteLLM model special case is not the best fix; the maintainable outcome is either an explicit provider/model persistence opt-in or a documented maintainer-approved default policy for proxy-like Responses endpoints.
