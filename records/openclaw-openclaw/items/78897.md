## Summary

Keep open: current main intentionally keeps Responses `store` off or false for LiteLLM/proxy routes, while the report provides plausible multi-turn failure evidence for a deployment that needs persisted response items.

## What This Changes

Allow OpenAI Responses configurations routed through LiteLLM to opt into `store=true` or otherwise avoid replaying non-persisted Responses item ids during continuations.

## Best Possible Solution

Add a deliberate provider/model-level Responses store policy that both transport paths honor, document the retention tradeoff for LiteLLM-compatible proxies, and cover proxy opt-in plus native no-regression cases.

## Reproduction Assessment

No high-confidence live reproduction is available because the failing LiteLLM proxy is private. Source inspection gives a clear reproduction path for the OpenClaw side: LiteLLM/proxy Responses routes are not upgraded to `store:true`, while HTTP and WebSocket builders emit or preserve `store:false`.

## Solution Assessment

No current implementation solves this for LiteLLM. Hardcoding `litellm/gpt-5.5` would be too narrow; the maintainable direction is an explicit Responses store opt-in/policy shared by the HTTP and WebSocket paths.
