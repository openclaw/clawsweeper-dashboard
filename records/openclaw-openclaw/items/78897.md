## Summary

Keep open: current main still disables or emits `store:false` for LiteLLM/proxy Responses routes, and the reporter supplied plausible real-route evidence plus a concrete opt-in shape, but solving this adds a user-facing persisted-response policy with privacy/retention tradeoffs.

## What This Changes

Request to let OpenAI Responses-compatible LiteLLM/custom provider models opt into `store:true` so multi-turn continuations that replay `rs_*` items do not fall back to a secondary model.

## Best Possible Solution

Add a deliberate provider/model-level Responses persistence opt-in that both HTTP/SSE and WebSocket paths honor, document the retention tradeoff for LiteLLM-compatible proxies, and cover proxy opt-in plus native no-regression cases.

## Reproduction Assessment

No high-confidence live reproduction is available because the failing LiteLLM proxy is private. Source inspection gives a clear OpenClaw-side path: proxy/LiteLLM Responses routes cannot opt into `store:true`, while current HTTP and WebSocket builders emit or preserve `store:false`.

## Solution Assessment

Unclear: the proposed `responsesStore` opt-in matches the likely boundary, but it is a new public config contract and persistence policy. Maintainers should approve that shape before implementation, and the hardcoded local workaround should not be upstreamed as-is.
