## Summary

Keep open: current main still keeps LiteLLM/proxy Responses routes out of provider-managed `store:true`, and an open PR explicitly closes this issue, so the issue should stay open until that PR is resolved or replaced.

## What This Changes

Requests provider/model-aware or configurable OpenAI Responses `store` behavior for LiteLLM/OpenAI-compatible proxies so multi-turn continuations do not fail and fall back from the primary model.

## Best Possible Solution

Add a documented provider/model Responses persistence opt-in that all Responses payload paths honor, with LiteLLM proxy coverage and native OpenAI/Azure no-regression tests.

## Reproduction Assessment

No high-confidence live reproduction is available because the failing LiteLLM proxy is private. Source inspection does show the current-main path: proxy-like Responses routes can replay prior item IDs while `storeMode:"disable"` emits `store:false`.

## Solution Assessment

No. Hardcoding `litellm/gpt-5.5` or globally omitting `store:false` for every proxy-like endpoint is not the narrowest maintainable solution; the safer direction is an explicit persistence opt-in because OpenAI Responses storage defaults to true and affects retention. ([platform.openai.com](https://platform.openai.com/docs/api-reference/responses/list?api-mode=responses&lang=curl&utm_source=openai))
