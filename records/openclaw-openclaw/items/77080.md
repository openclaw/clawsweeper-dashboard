## Summary

Keep open: current main still treats WhatsApp quote stanza IDs as reply metadata, not as a session-routing primitive, and the request would add a new WhatsApp thread-binding feature with config, persistence, and UX defaults that need maintainer design review.

## What This Changes

The issue requests WhatsApp reply-quote stanza IDs to drive thread-bound session routing for spawned sessions.

## Best Possible Solution

Add a WhatsApp-owned conversation-binding layer that records outbound stanza IDs, resolves quoted inbound stanza IDs before dispatch, and documents/tests a DM-first fallback policy through the existing channel binding seams.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction; source inspection shows current main extracts WhatsApp quote stanza IDs but does not feed them into route selection.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer design review; the proposed direction fits the WhatsApp plugin and existing binding seams, but persistence, defaults, DM/group scope, and CLI surface should be decided first.
