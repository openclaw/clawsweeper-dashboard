## Summary

Keep open: current main still treats WhatsApp quote stanza IDs as inbound reply metadata, not as a session-routing key, and the request would add new WhatsApp routing semantics, persistence, config, and likely admin behavior that need maintainer design review.

## What This Changes

Request to route WhatsApp reply-quoted inbound messages to the session that originally sent the quoted outbound stanza ID.

## Best Possible Solution

Add a WhatsApp-owned conversation-binding layer that records outbound stanza IDs, resolves quoted inbound stanza IDs before dispatch, and documents/tests DM-first fallback behavior through the existing channel/plugin seams.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction; source inspection gives high confidence that current main exposes quote IDs as metadata but never uses them to select the inbound session route.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer design review; the reply-quote mapping fits the existing conversation-binding direction, but persistence, defaults, group scope, and CLI/admin surface need product decisions first.
