## Summary

Keep open: the PR is a focused Tailscale process-lifecycle fix, but it still lacks real behavior proof and the current patch removes the caller-level wall-clock bound while relying on `execFile` timeout semantics that only send a kill signal and settle after child close.

## What This Changes

The PR removes the outer 3-second `Promise.race` from Tailscale binary probing so `checkBinary` awaits `runExec(path, ["--version"], { timeoutMs: 3000 })` directly.

## Best Possible Solution

Keep the Tailscale binary check bounded while ensuring the spawned probe is killed or reaped, then merge only with redacted proof from a hanging `--version` fixture.

## Reproduction Assessment

No high-confidence live reproduction is present. Source inspection shows the current duplicate timeout path and Node's timeout behavior, but no real hanging Tailscale-compatible binary proof was provided or run.

## Solution Assessment

No, not as written. The narrow cleanup is plausible, but the implementation should preserve a bounded probe while explicitly owning child cleanup instead of relying only on `execFile` timeout settlement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Preserve the bounded binary probe:** `src/infra/tailscale.ts:40`
  - body: `execFile`'s timeout only sends the kill signal and settles from the child `close` handler. If the candidate `tailscale` binary traps `SIGTERM` or never closes, this direct await may never settle, whereas current main returns `false` from the outer timeout after 3 seconds. Keep a wall-clock bound while also owning child cleanup so the setup probe cannot hang on the slow/hung binary path this PR is hardening.
  - confidence: 0.79
