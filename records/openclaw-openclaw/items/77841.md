## Summary

Keep open: current `main` still has a source-level gap where an `alsoAllow`-only tools policy preserves implicit allow-all for normal filtering but can become a restrictive allowlist before the media generation factory constructs `image_generate`, `video_generate`, and `music_generate`. The related #77244 fix is real, but it addresses provider-plugin startup, not this later tool-policy edge case.

## What This Changes

The issue reports that media generation tools stay catalog-visible but not live when `tools.alsoAllow` is configured without `tools.allow`.

## Best Possible Solution

Preserve implicit allow-all semantics for built-in media factory availability when `tools.alsoAllow` is used without `tools.allow`, while keeping the optional-plugin discovery guard that prevents loading every optional plugin tool.

## Reproduction Assessment

Yes by source inspection: `alsoAllow` without `allow` becomes an implicit allow-all policy, but the extracted factory allowlist drops the synthetic wildcard before media tool construction. I did not run a live gateway because this sweep is read-only, but the current-main path matches the reported catalog-visible/not-effective symptom.

## Solution Assessment

Yes: the narrow maintainable fix is to carry or reconstruct the implicit allow-all semantics at the media factory boundary. Steering users to replace valid `alsoAllow` config with `tools.allow: ["*", ...]` would contradict the documented default media-tool behavior.
