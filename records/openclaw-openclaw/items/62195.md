## Summary

Keep this PR open: it adds a plausible cron/heartbeat preHook feature, but the current head still lets configured executables be resolved through PATH despite documenting an absolute-path contract.

## What This Changes

The PR adds optional cron-job and heartbeat preHook executable gates with protocol, config, tool schema, Swift model, docs, and test updates for proceed/skip/error exit-code handling.

## Best Possible Solution

Require absolute executable validation at every ingress point and at the spawn boundary, then document or enforce the intended execution-policy model before merge.

## Reproduction Assessment

Yes. On the PR head, the inspected normalizer preserves a non-empty `preHook.file`, and a read-only Node proof showed `execFile('node', ..., { shell:false })` resolves through PATH and runs.

## Solution Assessment

No. The feature direction is plausible, but the proposed patch is not the best mergeable form until it enforces the documented absolute-path contract and settles the gateway-host execution boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Require absolute preHook executables:** `src/cron/pre-hook.ts:66-70`
  - body: The docs and tool schema describe `preHook.file` as an absolute path, but the normalizer persists any non-empty string and `runPreHook` passes it straight to `execFile`. With `shell:false`, bare names such as `node` still resolve through PATH, so a relative or PATH-controlled executable can run under the gateway process. Reject non-absolute paths before storing or spawning the hook.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The patch introduces persistent gateway-host executable execution and still needs path validation plus an explicit execution-policy decision before merge.

Concerns:

- **[high] Relative/PATH preHook execution:** `src/cron/pre-hook.ts:66`
  - body: `preHook.file` is accepted as any non-empty string and then executed with `execFile`; bare executable names are resolved through PATH even with `shell:false`. This contradicts the documented absolute-path contract and can execute an unintended binary when PATH or the working directory is influenced.
  - confidence: 0.91
- **[medium] Exec policy boundary is not explicit:** `src/cron/pre-hook.ts:66`
  - body: The new runner calls `child_process.execFile` directly instead of an existing documented system.run approval path. Since cron/heartbeat configuration can create scheduled host execution, maintainers should either route hooks through the policy model or document and test preHook as an admin-only bypass.
  - confidence: 0.68
