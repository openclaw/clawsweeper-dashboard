## Summary

Keep open: current main still lacks this preHook capability, and this PR remains the active implementation candidate, but the latest head has a blocking executable-path security issue and lacks after-fix real behavior proof.

## What This Changes

The PR adds a cron-job and heartbeat `preHook` executable gate across runtime execution, protocol/config/tool schemas, Swift models, docs, and tests.

## Best Possible Solution

Land a hardened additive preHook contract only after absolute executable validation is enforced at every ingress and spawn boundary, the admin/security policy is explicit, and real cron/heartbeat behavior proof is added.

## Reproduction Assessment

Yes for the PR-introduced security issue: the PR head normalizer preserves any non-empty `preHook.file`, and a read-only Node proof showed a bare executable resolves through PATH with `shell:false`. This is not a current-main bug reproduction because current main does not contain preHook.

## Solution Assessment

No. The feature direction is plausible, but the PR is not mergeable until it enforces the documented absolute-path contract at ingress and spawn time and settles the scheduled host-execution policy boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Require absolute preHook executables:** `src/cron/pre-hook.ts:66-70`
  - body: The docs and tool schema say `preHook.file` is an absolute path, but `normalizeCronJobInput` stores any non-empty string and `runPreHook` passes it directly to `execFile`. With `shell:false`, bare names still resolve through PATH, so a configured `node` or relative path can execute an unintended binary under the gateway process; reject non-absolute files before persisting and before spawning.
  - confidence: 0.92

## Security Review

Status: needs_attention
