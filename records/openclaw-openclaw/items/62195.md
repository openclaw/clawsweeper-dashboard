## Summary

Keep open: current main still has no cron/heartbeat preHook implementation, and this PR is useful but not merge-ready because it documents an absolute executable contract while accepting and executing any non-empty file string. Earlier review concerns around the runtime import seam, update-tool clear sentinel, and gateway heartbeat override merge appear addressed in the current PR head.

## What This Changes

The PR adds optional cron-job and heartbeat preHook configuration that runs a configured executable before agent execution, with protocol/config/tool/schema/docs/tests updates and skip/error exit-code handling.

## Best Possible Solution

Keep the PR open and require changes before merge: enforce absolute executable paths at every ingress point and at the runtime spawn boundary, add regression tests for bare and relative paths, and have maintainers decide whether preHook execution must route through existing exec-policy enforcement or be documented as an admin-only cron capability.

## Reproduction Assessment

Yes. On the PR head, a `preHook.file` such as `node` is accepted by normalization and passed to `execFile`; a read-only local proof showed `execFile('node', ..., { shell:false })` executes via PATH and returns the child exit code.

## Solution Assessment

No. The feature direction is plausible, but the proposed patch is not the narrowest maintainable solution until the documented absolute-path contract is enforced and the gateway-host execution policy boundary is settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Require absolute preHook executables:** `src/cron/pre-hook.ts:66-70`
  - body: The docs and tool schema describe `preHook.file` as an absolute path, but the normalizer persists any non-empty string and `runPreHook` passes it straight to `execFile`. With `shell:false`, bare names such as `node` still resolve through PATH, so a relative/PATH-controlled executable can run under the gateway process. Reject non-absolute paths before storing or spawning the hook.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The patch introduces persistent gateway-host executable execution and still needs path validation plus an explicit execution-policy decision before merge.

Concerns:

- **[high] Relative/PATH preHook execution:** `src/cron/pre-hook.ts:66`
  - body: `preHook.file` is accepted as any non-empty string and then executed with `execFile`; bare executable names are resolved through PATH even with `shell:false`. This contradicts the documented absolute-path contract and can execute an unintended binary when PATH or the working directory is influenced.
  - confidence: 0.91
- **[medium] Exec policy boundary is not explicit:** `src/cron/pre-hook.ts:66`
  - body: The new runner calls `child_process.execFile` directly instead of the documented `system.run`/exec approval path. Since the cron tool exposes scheduled work creation to owner-only agent tooling, maintainers should either route hooks through the existing exec policy or document and test preHook as an admin-only bypass.
  - confidence: 0.68
