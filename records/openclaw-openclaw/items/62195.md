## What This Changes

The PR adds optional cron and heartbeat `preHook` executable gates across runtime execution, config/protocol/tool schemas, generated Swift models, docs, and focused tests.

## Best Possible Solution

Choose one maintainer-approved cron hook contract, enforce its host-exec trust boundary in schema and runtime, and land it with redacted real cron/heartbeat proof plus conflict-free integration against current main.

## Reproduction Assessment

Yes for the PR-introduced finding: PR-head source stores any non-empty `preHook.file` and passes it to `execFile`. This is not a current-main bug reproduction because current main does not contain the preHook feature.

## Solution Assessment

Unclear as proposed: a narrow preHook gate can solve the use case, but it should either enforce pinned executable paths and be explicitly accepted as trusted host-exec behavior or be folded into the broader lifecycle-hooks design.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reject non-absolute preHook executables:** `src/cron/normalize.ts:506-516`
  - body: The docs and tool schema describe `preHook.file` as an absolute executable path, but the normalizer accepts any non-empty string and `runPreHook` then passes it to `execFile`. A value like `check-api` or `./check-api` can resolve through the gateway process PATH/CWD, so unattended cron or heartbeat runs may execute a different binary than the operator intended. Reject non-absolute paths, or otherwise canonicalize and pin the executable, before storing/running the hook.
  - confidence: 0.9
