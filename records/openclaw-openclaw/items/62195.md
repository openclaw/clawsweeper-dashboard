## Summary

Keep open: current main does not implement cron/heartbeat preHook support, and the PR still has a blocking security issue because it documents an absolute-path executable contract while accepting and executing relative or PATH-resolved files.

## What This Changes

The PR adds an optional cron and heartbeat preHook executable gate with protocol, config, tool schema, Swift model, docs, and tests for proceed/skip/error exit-code handling.

## Best Possible Solution

Land a narrowed preHook only after executable paths are validated at ingress and spawn time, with the scheduled host-execution contract documented and tested across cron and heartbeat.

## Reproduction Assessment

Yes. On the PR head, the normalizer preserves any non-empty preHook.file, runPreHook passes it to execFile, and a read-only Node proof confirmed bare executable names resolve through PATH even with shell:false.

## Solution Assessment

No. The feature direction is plausible, but this patch is not mergeable until it enforces the documented absolute-path contract and settles the gateway-host execution policy boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Require absolute preHook executables:** `src/cron/pre-hook.ts:66-70`
  - body: The docs and tool schema describe `preHook.file` as an absolute executable path, but the normalizer stores any non-empty string and this spawn call passes it directly to `execFile`. With `shell:false`, bare names such as `node` still resolve through PATH, so a PATH-controlled or relative executable can run under the gateway process. Reject non-absolute paths before storing them and again at the spawn boundary.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The patch introduces scheduled gateway-host executable execution and still needs path validation plus an explicit execution-policy decision before merge.

Concerns:

- **[high] Relative/PATH preHook execution:** `src/cron/pre-hook.ts:66`
  - body: `preHook.file` is accepted as any non-empty string and then executed with `execFile`; bare executable names are resolved through PATH even with `shell:false`. This contradicts the documented absolute-path contract and can execute an unintended binary when PATH or the working directory is influenced.
  - confidence: 0.91
- **[medium] Exec policy boundary is not explicit:** `src/cron/pre-hook.ts:66`
  - body: The new runner calls `child_process.execFile` directly rather than an existing documented system.run approval path. Since cron/heartbeat configuration can create scheduled host execution, maintainers should either route hooks through the policy model or document and test preHook as an admin-only execution surface.
  - confidence: 0.68
