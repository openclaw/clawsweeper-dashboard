## Summary

Current main still keeps the top-level status service rows on the user-systemd path, while the docs and prior fixes recognize system-scoped gateway units as valid. Two open PRs overlap this systemd-scope work, so this issue should stay open and be resolved by a coordinated fix rather than closed as implemented.

## What This Changes

The issue asks `openclaw status --deep` to stop reporting user-systemd services as not installed when a healthy OpenClaw gateway is running under a system-scoped `systemctl` unit.

## Best Possible Solution

Land one coordinated status/systemd fix that recognizes active system-scoped OpenClaw gateway units in the top-level status service rows, keeps user-systemd lifecycle commands as the managed default, and proves both user-scope and system-scope behavior.

## Reproduction Assessment

Yes, at source level: current main reads and probes the Linux user-systemd unit only, then formats the missing user service as not installed. I did not run a live systemd-system fixture in this read-only review.

## Solution Assessment

Yes directionally: a narrow status/systemd-system fallback or externally managed runtime summary is the maintainable fix. A new `--system` flag alone would leave the default `status --deep` diagnostic misleading for a documented deployment mode.
