## Label Justifications

- `P2`: This is a normal-priority active-memory bug with limited blast radius: cron windows waste recall work and emit noisy timeouts without directly breaking ordinary conversations.
- `impact:session-state`: The issue is about non-interactive dreaming session keys being treated as eligible conversational session state for active-memory recall.

## Summary

Keep open: current main and the latest release still have the active-memory eligibility path that can classify memory-core dreaming-narrative webchat subagent sessions as direct chats, and the open closing PR is the right implementation candidate but has not merged.

## What This Changes

The issue reports that active-memory treats memory-core dreaming-narrative cron session keys as interactive direct sessions, causing active-memory recall to run and time out during cron.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/93168, or an equivalent narrow active-memory eligibility guard with regression tests, rather than adding a new operator config surface.

## Reproduction Assessment

Yes, source-reproducible: current main builds `dreaming-narrative-*` subagent session keys in memory-core, then active-memory treats webchat user sessions as direct unless a more specific guard rejects them. I did not run a live cron-window reproduction in this read-only review.

## Solution Assessment

Yes: a narrow active-memory eligibility exclusion with regression coverage is the best fix. Adding `excludeSessionKeyPatterns` would expand public plugin config and should stay a fallback only if the source guard cannot cover real system-session keys.
