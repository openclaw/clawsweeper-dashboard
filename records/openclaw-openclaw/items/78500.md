## Label Justifications

- `P2`: This is a normal-priority active-memory bug with limited blast radius around wasted cron-window recall work and noisy lane timeouts.
- `impact:session-state`: The issue is about session-key classification causing non-interactive dreaming sessions to be treated like interactive active-memory sessions.

## Summary

Keep open: current main and the latest release still lack a dreaming-narrative exclusion in active-memory eligibility, while https://github.com/openclaw/openclaw/pull/93168 is an open closing PR that appears to own the narrow fix.

## What This Changes

The issue asks active-memory to stop treating memory-core dreaming-narrative cron session keys as interactive sessions, preferably through the eligibility gate instead of new configuration.

## Best Possible Solution

Review and land the narrow active-memory eligibility/test fix in https://github.com/openclaw/openclaw/pull/93168, keeping configuration unchanged unless maintainers choose a broader policy surface later.

## Reproduction Assessment

Yes, source-reproducible: memory-core creates `dreaming-narrative-*` session keys, gateway execution marks embedded attempts as user-triggered, and current active-memory has no dreaming-key exclusion before the direct-session allowlist. I did not run a live cron reproduction because this review is read-only.

## Solution Assessment

Yes, the best current fix is the narrow eligibility guard plus regression tests in the open PR. Adding an `excludeSessionKeyPatterns` config knob would add operator-facing config surface without being necessary for the established bug.
