## Summary

Keep open: current `main` still lets memory-core dreaming subagent prompt builds look like eligible webchat/direct active-memory turns, even though the active-memory docs exclude sub-agent/internal helper execution.

## What This Changes

The issue asks for active-memory to reject non-interactive dreaming/maintenance session keys, or expose an operator exclusion pattern, so cron-created memory-core sessions do not consume the active-memory lane budget.

## Best Possible Solution

Add a narrow active-memory eligibility guard for internal/plugin helper session keys, including memory-core dreaming narrative keys, while preserving documented direct/group/channel/explicit chat behavior.

## Reproduction Assessment

Yes, from source inspection. A memory-core dreaming subagent run gets a `dreaming-narrative-*` session key but enters the normal prompt-build hook with `trigger: "user"` and webchat channel context, which current active-memory gates accept as direct.

## Solution Assessment

Yes. The best fix is to tighten the active-memory-owned eligibility path; adding an `excludeSessionKeyPatterns` config knob would be broader than needed for behavior the docs already define as out of scope.
