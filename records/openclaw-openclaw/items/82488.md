## Summary

Keep this PR open. It targets a real false-positive path, but the patch currently weakens deep audit coverage for runtime code that can be imported from test-named files, and the PR body only provides mocked/unit-test proof rather than real after-fix audit output.

## What This Changes

The PR changes `getCodeSafetySummary` to pass `excludeTestFiles: true` to the shared scanner and adds a regression test that spies scanner options during plugin code-safety collection.

## Best Possible Solution

Keep the false-positive fix direction, but make deep audit preserve declared entrypoints and local runtime imports before excluding packaged test fixtures, then add real audit proof plus focused regression coverage.

## Reproduction Assessment

Yes, source-level: current main deep audit calls the scanner without `excludeTestFiles`, and the scanner default includes test files. A plugin test file with `process.env` and nearby network-send syntax will reach `env-harvesting`; I did not run a live audit in this read-only review.

## Solution Assessment

No, not as-is. Passing `excludeTestFiles: true` fixes the noisy class, but the audit should preserve runtime graph coverage like the install scanner does, and the contributor still needs real after-fix behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve runtime imports when excluding test files:** `src/security/audit-extra.async.ts:256`
  - body: Setting `excludeTestFiles: true` here makes the deep audit skip every test-named file found by the directory walk, but this audit only forces manifest entries and does not add local imports from those entries. A plugin can still execute code from `helper.test.js` if `index.js` imports it; before this change the audit scanned that file, after this change it would be invisible. Please preserve the runtime entry/import graph, or otherwise limit the exclusion so runtime-loaded test-named files remain scanned.
  - confidence: 0.82
