## Summary

Keep open: the patch targets a valid false-positive path, but it weakens deep-audit coverage for runtime files with test-like names and lacks required real behavior proof.

## What This Changes

The PR wires `excludeTestFiles: true` into the shared deep code-safety summary helper and adds a mocked regression test asserting plugin scans pass that option.

## Best Possible Solution

Exclude non-runtime test fixtures only after preserving manifest-entry runtime graph coverage, then add focused regression coverage for both the false positive and a reachable test-named runtime import.

## Reproduction Assessment

Yes, source inspection shows current main plugin deep audits call the scanner without test exclusion while the scanner default includes test files; the linked issue provides concrete Discord false-positive examples. I did not run the audit because this review had to keep the checkout read-only.

## Solution Assessment

No, not as written: using the existing exclusion option is the right direction, but the deep audit must also keep coverage for files reachable from manifest entries before skipping test-like paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve runtime graph coverage when excluding tests:** `src/security/audit-extra.async.ts:255`
  - body: This skips every walked `*.test.*`, `*.spec.*`, and `*.mock.*` file in deep plugin audits, but the audit only force-includes manifest entries. Unlike the install-time scanner, it does not add files imported by those entries, so a plugin can import `./payload.test.ts` from its runtime entry and the dangerous code in that file will no longer be scanned. Add runtime graph coverage, or otherwise force-scan reachable files, before excluding test fixtures.
  - confidence: 0.87
