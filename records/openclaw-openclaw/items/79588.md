## Summary

Keep open. Current main still constructs the compaction quality guard's identifier set from a last-ten-message text extraction path, so identifiers that only appear in tool-call arguments or non-text tool-result blocks can still be missed; the linked closing PR is open and unmerged.

## What This Changes

Request is to make compaction summary quality checks preserve opaque identifiers from original message text, tool-call arguments, and tool results, retrying or logging when strict policy loses them.

## Best Possible Solution

Land or replace the linked compaction-safeguard fix so identifier survival checks are seeded from the full pre-compaction input, exclude stripped `toolResult.details`, and include regression coverage for text, tool-call arguments, tool results, and pruned messages.

## Reproduction Assessment

Yes, source-reproducible. Current main builds the checked identifier set from the last ten text-extracted messages, so an opaque identifier that appears only in tool-call arguments or non-text tool-result content will not be supplied to the quality audit.

## Solution Assessment

Yes. Keeping this issue open while reviewing or replacing https://github.com/openclaw/openclaw/pull/75336 is the narrow maintainable path; the fix belongs in the existing compaction safeguard extraction and retry flow, not in a new policy surface.
