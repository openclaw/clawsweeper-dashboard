## Summary

Keep open. Current main still builds the compaction quality guard's identifier set from a sampled text-only view, and the linked open implementation PR has not merged, so closing this issue would preempt the active fix path.

## What This Changes

The issue asks the compaction safeguard to extract opaque identifiers from original messages, tool-call arguments, and tool results, retry strict-policy summaries that lose them, and log identifier loss.

## Best Possible Solution

Land a reviewed compaction-safeguard fix that seeds survival checks from the full pre-compaction input, excludes stripped `toolResult.details`, and covers text, tool-call arguments, tool results, and pruned messages with regression tests.

## Reproduction Assessment

Yes. From source inspection, an opaque identifier that appears only in assistant tool-call arguments will not enter `identifierSeedText` because current main only reads string and text-block content before auditing the summary.

## Solution Assessment

Yes, keeping this issue open while reviewing https://github.com/openclaw/openclaw/pull/75336 is the best path. The fix belongs in the existing compaction safeguard extraction and retry path, not in a new config or product-policy layer.
