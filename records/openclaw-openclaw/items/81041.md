## Summary

Keep open: current main still hashes the full assembled system prompt in the Claude live-session fingerprint, so volatile chat metadata can still force live Claude process restarts. The linked fix PR is open and explicitly closes this issue, so the issue should stay open until that PR, or a safer replacement, lands.

## What This Changes

Stop hashing per-turn volatile system-prompt content in the Claude CLI live-session fingerprint so chat-channel follow-up turns reuse live process context.

## Best Possible Solution

Update the Claude live-session fingerprint to use stable session-changing prompt inputs only, then land https://github.com/openclaw/openclaw/pull/81047 or a narrow replacement that preserves real prompt invalidation.

## Reproduction Assessment

Yes, source-level: current main hashes `context.systemPrompt`, chat reply preparation injects volatile inbound metadata into that prompt, and fingerprint mismatch closes the live Claude process. I did not execute a live Telegram/Discord reproduction in this read-only review.

## Solution Assessment

Yes for the direction, but the implementation should be narrower than dropping every prompt-change signal. The maintainable fix is to exclude per-turn chat metadata while preserving invalidation for stable system-prompt, auth, MCP, model, workspace, skills, argv, and env changes.
