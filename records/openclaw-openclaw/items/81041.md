## Summary

Keep open: current main still hashes the full assembled system prompt in the Claude live-session fingerprint, so volatile chat metadata can still force claude-cli live subprocess restarts. The linked fix PR is open and uses closing syntax, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to stop including volatile per-turn chat-channel system prompt content in the claude-cli live-session fingerprint.

## Best Possible Solution

Land a narrow fix that fingerprints only stable session-changing inputs, preserves existing model/workspace/auth/MCP/skills/static-prompt invalidation, and adds regression coverage for volatile chat metadata.

## Reproduction Assessment

Yes, by source inspection: chat reply runs can put per-message inbound metadata into the assembled system prompt, and current main hashes that prompt before closing the live session on fingerprint mismatch. I did not execute a live Telegram or Discord claude-cli repro in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to remove the volatile full-system-prompt hash from the live-session fingerprint while retaining the existing static prompt hash and other stable invalidation gates. The open linked PR follows that direction and should be reviewed or replaced before this issue closes.
