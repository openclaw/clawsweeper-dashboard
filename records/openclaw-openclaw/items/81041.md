## Summary

Keep open: current main still hashes the full assembled system prompt in the Claude live-session fingerprint, while chat-channel inbound metadata is intentionally excluded from the static prompt hash and can vary on each turn.

## What This Changes

The issue requests fixing `claude-cli` live-session reuse so volatile chat-channel system-prompt metadata does not restart the live Claude process every turn.

## Best Possible Solution

Move the live-session fingerprint to stable session-changing inputs only, preserving auth/MCP/env/model/workspace/skills invalidation while excluding per-turn chat metadata below the system-prompt cache boundary.

## Reproduction Assessment

Yes, by source inspection: two chat turns with different inbound metadata can produce different `context.systemPrompt` values, and current main restarts the Claude live process when that hash changes. I did not execute a live Telegram/Discord repro in this read-only review.

## Solution Assessment

Mostly yes: the fix should remove the full volatile system-prompt hash from the live-session fingerprint, but the safest version should replace it with stable prompt material or existing static hashes rather than dropping every prompt-change signal.
