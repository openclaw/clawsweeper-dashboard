## Summary

Keep open. Current main still ships a 15 second image HEALTHCHECK grace period, and the adjacent early-bind work does not by itself prove that every cold Docker start can answer health probes inside that window.

## What This Changes

The issue requests increasing the Docker image HEALTHCHECK start period so legitimate gateway cold starts are not marked unhealthy before startup completes.

## Best Possible Solution

Set the image healthcheck startup grace to a measured cold-start budget, likely 300-600 seconds, and align the Compose healthcheck if maintainers want the same operator behavior there while leaving `/healthz` semantics unchanged.

## Reproduction Assessment

Yes. The static part reproduces on current main by inspecting `Dockerfile:297`, and the issue provides concrete Docker run and `docker ps` steps for the live health-state failure; this read-only review did not execute Docker.

## Solution Assessment

Yes. A targeted healthcheck grace-period increase is the narrowest maintainable fix for false unhealthy states from legitimate cold starts; it should not rework runtime-deps staging, probe routing, interval, timeout, or retries unless live validation shows those are also wrong.
