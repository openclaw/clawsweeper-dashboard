## Summary

Keep open: current main and the latest release still use a 15s Docker HEALTHCHECK start period, and two open PRs already target the same fix with different durations.

## What This Changes

The issue asks to increase the Docker image HEALTHCHECK start period from 15s to a realistic cold-start grace period so orchestrators do not mark legitimate startup as unhealthy.

## Best Possible Solution

Review #75766 and #75809 together, choose one Docker healthcheck grace period, decide Compose parity, and land a single canonical fix with changelog coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence because current main and `v2026.4.29` both schedule the image healthcheck after only 15s, while the issue provides a Docker run sequence and related startup reports document legitimate starts exceeding that window.

## Solution Assessment

Yes for the direction: widening the Dockerfile start period is the narrowest maintainable fix for premature OCI `unhealthy` state without changing `/healthz` semantics. The remaining decision is the exact duration and whether to align the Compose template.
