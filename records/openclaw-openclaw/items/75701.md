## Summary

Keep open: the reported Docker healthcheck timing bug is still present on current main, and two open PRs already target this issue with closing syntax. The issue should remain open until maintainers choose, adjust, land, or close the competing fix PRs.

## What This Changes

Request to increase the Docker image `HEALTHCHECK --start-period` beyond 15s so legitimate cold starts do not become `unhealthy` before `/healthz` is reachable.

## Best Possible Solution

Review the two open fixing PRs together, choose one cold-start grace period, land a single Dockerfile update with any desired Compose and changelog follow-through, and close the other PR as superseded after the canonical fix merges.

## Reproduction Assessment

Yes. Current main still schedules the image healthcheck after 15s, and source inspection shows `/healthz` is only available after pre-bind startup work; any legitimate cold start longer than 15s can reproduce the premature Docker `unhealthy` state described by the reporter.

## Solution Assessment

Yes for the direction: widening the Dockerfile `--start-period` is the narrowest maintainable fix for premature OCI health state without changing `/healthz` semantics. The remaining decision is the duration and whether related Compose defaults should be aligned in the same change.
