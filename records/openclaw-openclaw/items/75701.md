## Summary

Keep open: current main still ships the Docker image healthcheck with a 15 second start period, and the public Docker install docs state that this built-in healthcheck can drive an unhealthy container state. Recent startup/runtime-deps work reduces some of the cited cold-start paths, but it does not address the Dockerfile grace-period mismatch directly.

## What This Changes

The issue asks to raise the Dockerfile `HEALTHCHECK --start-period` from `15s` to a realistic cold-start grace period, such as `300s` or `600s`, without changing the probe command.

## Best Possible Solution

Ship a small Dockerfile change that gives the OCI healthcheck a cold-start grace period matching OpenClaw's supported container startup envelope, keep `/healthz` as the liveness probe, and leave deeper startup/runtime-deps tuning to the existing startup workstream.

## Reproduction Assessment

Yes for the released-image path: the issue provides a concrete `docker run` plus `docker ps` check, and current source confirms the 15 second healthcheck grace remains. I did not execute Docker in this read-only review, so a fix PR should include a live or targeted Dockerfile validation step.

## Solution Assessment

Yes: adjusting the Dockerfile `--start-period` is the narrowest fix for the reported OCI health state without changing gateway probe semantics. A broader gateway startup rewrite is not needed for this item because current main already moved much runtime plugin work behind HTTP bind.
