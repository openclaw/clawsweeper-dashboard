## Summary

Keep open: current main and the latest release still carry the 15s Dockerfile HEALTHCHECK start period, and two open PRs already use closing syntax for this issue. The issue should remain open until maintainers choose, adjust, land, or close the competing implementation PRs.

## What This Changes

The issue requests widening the Docker image HEALTHCHECK start grace period so legitimate gateway cold starts do not become OCI-unhealthy before the HTTP server binds.

## Best Possible Solution

Review the two open fixing PRs together, choose one cold-start grace period, land a single Dockerfile update with any desired Compose and changelog follow-through, and close the other PR as superseded after the canonical fix is merged.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence because current main and `v2026.4.29` both schedule the image healthcheck after only 15s, while the issue and linked startup reports document legitimate cold starts that can exceed that window.

## Solution Assessment

Yes for the direction: widening the Dockerfile `--start-period` is the narrowest maintainable fix for premature OCI `unhealthy` state without changing `/healthz` semantics. The unsettled part is the exact duration and whether Compose defaults should be aligned in the same PR.
