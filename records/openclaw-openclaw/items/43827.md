## Summary

Keep open: the localization goal is valid, but the branch is stale against current Chat UI structure and would leave visible English copy plus generated locale drift if merged as-is.

## What This Changes

The PR replaces selected Chat view literals with `t("chat.*")` calls and adds matching `chat.*` keys to six locale files.

## Best Possible Solution

Repair or replace the branch so all current Chat UI copy uses the existing i18n source of truth, dynamic values use interpolation, generated locale outputs and metadata are synced for every supported locale, and focused tests plus changelog cover the user-facing localization change.

## Reproduction Assessment

Yes. Static source inspection on current main shows visible Chat copy still hard-coded in `chat.ts`, `run-controls.ts`, `status-indicators.ts`, `chat-queue.ts`, `side-result-render.ts`, and `context-notice.ts`, so a locale switch cannot localize those labels today.

## Solution Assessment

No. The PR points in the right direction, but the best fix must target the current split helpers, preserve assistant-name interpolation, regenerate all supported locale outputs and metadata, and avoid unsupported fallback params.
