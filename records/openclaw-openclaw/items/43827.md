## Summary

Keep open: this is a real Control UI localization improvement, but the branch is stale against current main and is not merge-ready because current Chat copy now lives across split helper modules, the locale output set is incomplete, and the composer placeholder would lose assistant context.

## What This Changes

The branch replaces selected hard-coded Chat view strings with `t("chat.*")` calls and adds matching `chat.*` keys to six locale files.

## Best Possible Solution

Repair or replace the branch so all current Chat UI copy uses the existing i18n source of truth, dynamic values use interpolation, generated locale outputs and metadata are synced for every supported locale, and focused tests/changelog cover the user-facing localization change.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Static inspection of current main shows literal Chat copy in `chat.ts`, `run-controls.ts`, `status-indicators.ts`, `chat-queue.ts`, `side-result-render.ts`, and `context-notice.ts`, so switching locale cannot localize those labels today.

## Solution Assessment

Is this the best way to solve the issue? No. The PR moves in the right direction, but the current branch localizes an older surface, updates only part of the supported locale set, uses unsupported fallback params, and drops the assistant-name placeholder context.
