## Summary

Keep open: the PR still has blocking correctness and account-isolation concerns, and the contributor has not provided real after-fix CLI proof.

## What This Changes

The PR adds accountId binding documentation/JSDoc and changes bare multi-account CLI bind specs to resolve to accountId "*".

## Best Possible Solution

Keep the accountId docs clarification, but either keep account scope explicit-only or scope any wildcard default to creation flows after maintainer approval, with corrected JSDoc, aligned examples, regression coverage, and real CLI proof.

## Reproduction Assessment

Yes. Source inspection is enough for the PR-induced issues: parseBindingSpecs() feeds both bind and unbind, and current routing tests assert omitted accountId is default-only while "*" is any-account.

## Solution Assessment

No. The documentation clarification is useful, but changing shared parsing for bare specs is too broad; a safer path is explicit docs plus a creation-only CLI default if maintainers approve.
