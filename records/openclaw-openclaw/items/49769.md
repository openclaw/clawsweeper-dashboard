## Summary

Keep open for maintainer review: the docs intent is useful, but the branch also changes bare multi-account binding semantics through shared parsing, which can break existing unbind/config behavior and broaden account routing without approval or real proof.

## What This Changes

The PR adds accountId binding docs/JSDoc and changes bare multi-account `agents add`/`agents bind` parsing to emit `accountId: "*"`.

## Best Possible Solution

Keep the default-vs-wildcard docs precise, and only implement automatic wildcard creation in a creation-only or opt-in path after maintainer approval, upgrade tests, and real CLI/config proof.

## Reproduction Assessment

Yes for the PR-head blockers by source inspection: the changed resolver feeds shared bind-spec parsing, and both creation and unbind flows consume that parsed shape. I did not run live CLI proof because this is a read-only review and the PR lacks contributor proof.

## Solution Assessment

No. The docs clarification is directionally useful, but the behavior change should not live in shared bind/unbind parsing without a maintainer account-scope decision and upgrade proof.
