## Summary

Keep open: the docs clarification is useful, but the current PR head still has blocking correctness and account-isolation concerns, plus no after-fix real behavior proof.

## What This Changes

The PR adds accountId binding documentation/type comments and changes agent binding parsing so bare multi-account channel binds resolve to `accountId: "*"`.

## Best Possible Solution

Keep the accountId docs clarification, but scope any wildcard default to creation-only flows after maintainer approval, correct the literal default-account wording, align examples, add regression coverage, and require redacted CLI proof.

## Reproduction Assessment

Yes. Source inspection is enough: `parseBindingSpecs()` feeds both bind and unbind, and current routing tests assert omitted `accountId` is default-only while `"*"` is any-account.

## Solution Assessment

No. The docs clarification is useful, but changing shared parsing for bare specs is too broad; the safer path is explicit docs plus a creation-only CLI default if maintainers approve.

## Review Findings

Full review comments:

- **[P2] Scope wildcard defaults away from unbind:** `src/commands/agents.bindings.ts:261-266`
  - body: `resolveBindingAccountId()` is used by `parseBindingSpecs()` for both bind and unbind. On multi-account channels, bare `--bind <channel>` now parses as `accountId: "*"`, so unbind no longer matches existing channel-only/default bindings keyed without an explicit account.
  - confidence: 0.93

  - confidence: 0.9

  - body: The new tip recommends `accountId: "*"`, but the DM split and family group examples in the same page still omit it. Readers copying those examples can still hit the silent default-account behavior this PR is trying to make visible.
  - confidence: 0.82
