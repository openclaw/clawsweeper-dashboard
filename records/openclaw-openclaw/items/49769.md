## Summary

Keep open: the accountId documentation clarification is useful, but the branch still widens shared CLI binding parsing, misdocuments part of the routing contract, and lacks contributor real behavior proof for the changed CLI behavior.

## What This Changes

The branch adds accountId binding docs/JSDoc and changes bare multi-account `openclaw agents add`/`openclaw agents bind` specs to emit `accountId: "*"`.

## Best Possible Solution

Keep the docs clarification, but make account scope explicit and preserve existing unbind semantics unless maintainers deliberately approve a creation-only wildcard default with tests and proof.

## Reproduction Assessment

Yes. Source inspection is enough for the PR-induced issues: the PR adds wildcard fallback in shared account resolution, `parseBindingSpecs()` feeds both bind and unbind, and current routing tests document omitted-account versus wildcard behavior.

## Solution Assessment

No. The docs clarification is useful, but changing shared bind spec parsing is too broad; a safer path is explicit docs plus a creation-only wildcard default only if maintainers approve that product behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep wildcard defaults out of shared bind parsing:** `src/commands/agents.bindings.ts:264-266`
  - body: `resolveBindingAccountId()` is called by `parseBindingSpecs()`, which `agents unbind` also uses. On a multi-account channel, this new fallback makes `openclaw agents unbind --bind <channel>` look for `accountId: "*"` instead of the existing channel-only/default binding, so the previous shorthand cannot remove persisted bindings.
  - confidence: 0.93
- **[P2] Correct the omitted-account JSDoc:** `src/config/types.agents.ts:32-34`
  - body: The new JSDoc says omitted `accountId` follows `channels.<channel>.defaultAccount`, but routing indexes an empty account pattern under the literal `DEFAULT_ACCOUNT_ID` bucket. Configs whose default account is not literally `default` would be misdocumented and can still miss.
  - confidence: 0.9
- **[P2] Make bare bind docs match the new behavior:** `docs/cli/agents.md:61-63`
  - body: The new table says `--bind <channel>` matches the default account only, but the added resolver returns `"*"` when the channel has multiple accounts. Users copying the docs will not know bare binds can become all-account binds.
  - confidence: 0.87
- **[P3] Align the remaining binding examples:** `docs/concepts/multi-agent.md:423-427`
  - body: The new tip recommends `accountId: "*"`, but the DM split and family group examples in the same page still omit it. Readers copying those examples can still hit the silent default-account behavior this PR is trying to make visible.
  - confidence: 0.82
