Full review comments:

- **[P1] Scope wildcard defaulting away from shared parsing:** `src/commands/agents.bindings.ts:261-266`
  - body: `resolveBindingAccountId()` is used by `parseBindingSpecs()` for both bind and unbind. Returning `"*"` for omitted accountId on multi-account channels makes `agents unbind --bind <channel>` miss existing channel-only/default-account bindings and also changes bare binds to all-account routing.
  - confidence: 0.91
- **[P2] Correct the omitted-account JSDoc:** `src/config/types.agents.ts:32-34`
  - body: The new JSDoc says an omitted accountId follows `channels.<channel>.defaultAccount`, but the resolver maps an empty pattern to the literal `DEFAULT_ACCOUNT_ID` bucket and then looks up the inbound account id. Named defaultAccount values can still miss.
  - confidence: 0.86
- **[P2] Make bare bind docs match the new behavior:** `docs/cli/agents.md:63`
  - body: The docs table and warning say bare `--bind <channel>` matches the default account only, but this patch resolves it to wildcard when multiple accounts exist. Either the docs need to describe that new policy or the code should keep bare binds default-scoped.
  - confidence: 0.84
- **[P3] Align the remaining examples with the wildcard tip:** `docs/concepts/multi-agent.md:423-427`
  - body: The new tip says these examples use `accountId: "*"` as the safer default, but other WhatsApp peer/group examples in the same page still omit `accountId`, leaving mixed guidance after this patch.
  - confidence: 0.76
