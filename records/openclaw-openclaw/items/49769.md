## Summary

Keep open. Current main has not implemented the PR's intended documentation/JSDoc clarifications or the multi-account CLI fallback, and the proposed patch still has blocking correctness/docs issues around shared bind/unbind parsing and account-default wording.

## What This Changes

The PR updates agent binding docs and JSDoc, and changes binding spec resolution so omitted accountId on multi-account channels resolves to `accountId: "*"`.

## Best Possible Solution

Keep the PR open and request a focused revision that scopes any auto-wildcard behavior to bind/add creation or documents an explicit-only workflow, preserves channel-only unbind compatibility, corrects default-account wording against the actual resolver, updates examples consistently, and adds targeted tests plus the required changelog entry.

## Reproduction Assessment

Yes. The unbind regression is reproducible by static call tracing: `agents unbind` uses the same `parseBindingSpecs()` path as `agents bind`, and removal compares exact binding keys. The routing/docs mismatch is also covered by existing route tests for omitted accountId versus `accountId: "*"`.

## Solution Assessment

No. The PR addresses a real footgun, but placing the wildcard fallback in shared parsing is not the narrowest maintainable fix because it changes removal semantics; a safer solution would keep parsing neutral and apply any multi-account default only during binding creation, with docs that match the resolver.

Full review comments:

- **[P2] Preserve channel-only unbind specs:** `src/commands/agents.bindings.ts:280-287`
  - body: Returning `"*"` from the shared account resolver changes `parseBindingSpecs()` for `agents unbind` too. On a multi-account channel, an existing channel-only binding is keyed as the default-account match, but `openclaw agents unbind --bind <channel>` would now parse as wildcard and fail to remove the old binding.
  - confidence: 0.93
- **[P2] Correct the omitted-account default wording:** `src/config/types.agents.ts:31-34`
  - body: The new JSDoc says an omitted accountId follows `channels.<channel>.defaultAccount`, but the route resolver maps an empty account pattern to the literal `DEFAULT_ACCOUNT_ID` (`"default"`). For a channel whose configured default is not named `default`, this guidance tells users a channel-only binding will match when it will not.
  - confidence: 0.91
- **[P2] Align bare --bind docs with the new fallback:** `docs/cli/agents.md:62`
  - body: The new format table says `--bind <channel>` matches the default account only, but this PR also makes the same bare spec resolve to `accountId: "*"` when the channel has multiple accounts. Either the code needs to keep bare specs default-scoped or the CLI docs need to describe the auto-wildcard case.
  - confidence: 0.88
- **[P3] Update all examples covered by the wildcard tip:** `docs/concepts/multi-agent.md:425-428`
  - body: After adding guidance that `accountId: "*"` is a good default, the same multi-agent page still has peer-binding examples that omit accountId. Those examples undercut the new warning and can still teach the silent default-account footgun.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `src/commands/agents.bindings.ts:280-287`
  - body: This PR changes user-facing `openclaw agents add` / `openclaw agents bind` behavior, but the diff does not add a `CHANGELOG.md` entry. OpenClaw policy requires user-facing fixes to be recorded before merge.
  - confidence: 0.82
