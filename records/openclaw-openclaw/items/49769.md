## Summary

Keep open. Current main has not absorbed the PR's requested clarification or CLI behavior change, but the branch still needs revision because the wildcard fallback is added in shared binding parsing used by unbind, and parts of the new docs overstate or contradict current routing semantics.

## What This Changes

The branch adds accountId binding-scope docs/JSDoc and changes agent binding creation/parsing so omitted accountId on multi-account channels resolves to accountId "*".

## Best Possible Solution

Keep the PR open and request a focused revision: correct the accountId wording against the actual route resolver, keep wildcard defaults scoped to binding creation flows or document the CLI behavior accurately, preserve channel-only unbind compatibility, update examples consistently, and add targeted tests around the changed CLI/routing behavior.

## Reproduction Assessment

Yes. The unbind regression is reproducible by static call tracing: agents unbind uses parseBindingSpecs, the PR changes omitted accountId resolution there to "*" for multi-account channels, and removeAgentBindings matches by the resulting binding key. The docs mismatches are also directly reproducible by comparing the PR text with current resolve-route.ts behavior.

## Solution Assessment

No. The direction is useful, but putting the multi-account wildcard fallback in shared parsing is not the narrowest maintainable fix because it changes removal semantics; a safer version would scope auto-wildcarding to bind/add creation or require explicit :* and align docs/tests with that choice.

Full review comments:

- **[P2] Preserve channel-only unbind specs:** `src/commands/agents.bindings.ts:280-287`
  - body: Returning "*" from the shared account resolver changes parseBindingSpecs for agents unbind too. On a multi-account channel, an existing channel-only binding is keyed as the default-account match, but `openclaw agents unbind --bind <channel>` would now parse as wildcard and fail to remove the old binding.
  - confidence: 0.93
- **[P2] Correct the omitted-account default wording:** `src/config/types.agents.ts:31-34`
  - body: The new JSDoc says an omitted accountId follows `channels.<channel>.defaultAccount`, but the route resolver maps an empty account pattern to the literal DEFAULT_ACCOUNT_ID (`"default"`). For a channel whose configured default is not named `default`, this guidance tells users a channel-only binding will match when it will not.
  - confidence: 0.91
- **[P2] Align bare --bind docs with the new fallback:** `docs/cli/agents.md:62`
  - body: The new format table says `--bind <channel>` matches the default account only, but this PR also makes the same bare spec resolve to accountId "*" when the channel has multiple accounts. Either the code needs to keep bare specs default-scoped or the CLI docs need to describe the auto-wildcard case.
  - confidence: 0.88
- **[P3] Update all examples covered by the wildcard tip:** `docs/concepts/multi-agent.md:425-428`
  - body: After adding guidance that accountId "*" is a good default, the same multi-agent page still has peer-binding examples that omit accountId. Those examples now undercut the new warning and can still teach the silent default-account footgun.
  - confidence: 0.78
