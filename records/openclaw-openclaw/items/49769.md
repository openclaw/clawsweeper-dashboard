## Summary

Keep open: the documentation clarification is useful, but this PR still has blocking correctness issues in the shared bind/unbind parser, inaccurate default-account wording, missing real behavior proof, and an account-isolation decision for maintainers.

## What This Changes

The branch adds accountId binding docs/JSDoc and changes CLI bind parsing so bare multi-account channel binds resolve to accountId: "*".

## Best Possible Solution

Keep the accountId docs clarification, but either keep account scope explicit-only or scope any wildcard default to creation flows after maintainer approval, with regression coverage and real CLI proof.

## Reproduction Assessment

Yes. Source inspection is enough for the PR-induced bugs: parseBindingSpecs() feeds both bind and unbind, and current routing tests assert omitted accountId is default-only while "*" is any-account.

## Solution Assessment

No. The docs clarification is useful, but changing shared parsing for bare specs is too broad; a safer path is explicit docs plus a creation-only CLI default if maintainers approve.
