## Summary

Keep open: this PR addresses a real binding-footgun area, but it is not merge-ready because the shared parser change affects unbind, the new docs overstate `defaultAccount` behavior, and after-fix real CLI proof is missing.

## What This Changes

The PR updates agent binding docs/types and changes CLI binding creation to default omitted account IDs to `"*"` for multi-account channels.

## Best Possible Solution

Keep the docs clarification, but scope any wildcard default to creation-only flows after maintainer approval, correct the default-account wording to match routing, align examples, add focused regression coverage, and require redacted CLI proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `parseBindingSpecs()` feeds both bind and unbind, and current routing tests show omitted `accountId` only matches the literal default account while `"*"` is the any-account form.

## Solution Assessment

No. The documentation direction is useful, but the runtime change is too broad as written and should either stay docs-only or be scoped to add/bind creation paths with explicit tests and maintainer approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Scope the wildcard fallback away from unbind:** `src/commands/agents.bindings.ts:264-266`
  - body: This fallback lives in `resolveBindingAccountId()`, which `parseBindingSpecs()` calls for both `agents bind` and `agents unbind`. On a multi-account channel, `openclaw agents unbind --bind telegram` would now parse as `accountId: "*"`, so it no longer removes an existing channel-only/default binding.
  - confidence: 0.93
- **[P2] Correct the omitted-account default claim:** `src/config/types.agents.ts:32-34`
  - body: The JSDoc says omitted/empty follows `channels.<channel>.defaultAccount`, but routing indexes an empty `match.accountId` under the literal `DEFAULT_ACCOUNT_ID` and then looks up the inbound normalized account id. Configs whose default account is not named `default` will be misdocumented.
  - confidence: 0.9
- **[P3] Align the remaining binding examples:** `docs/concepts/multi-agent.md:426-429`
  - body: The new tip recommends `accountId: "*"` as a good default, but the DM split and family group examples still omit it. Readers copying those examples can still hit the silent default-account behavior this PR is trying to document.
  - confidence: 0.82
