## Summary

Keep this PR open for maintainer review: the docs intent is useful, but the behavior change broadens bare multi-account bindings in a shared parser, conflicts with the added docs, carries account-isolation risk, and still lacks real after-patch proof.

## What This Changes

The branch adds binding account-scope docs/JSDoc and changes CLI binding resolution so omitted account IDs on multi-account channels become `accountId: "*"`.

## Best Possible Solution

Keep precise default-vs-wildcard docs/JSDoc, preserve default-scoped behavior unless maintainers approve a creation-only wildcard convenience, and require focused bind/unbind tests plus real CLI proof for that policy.

## Reproduction Assessment

Yes for the PR-head blockers by source inspection: the changed helper feeds shared bind-spec parsing, and both bind/add and unbind consume that parsed shape. I did not establish live CLI proof because this read-only review found no contributor artifact showing the changed real behavior.

## Solution Assessment

No. The docs clarification is useful, but changing shared bare-bind parsing is too broad; the safer path is precise docs plus a creation-specific wildcard default only if maintainers approve that routing policy.
