## Summary

Keep open: the docs clarification is useful, but this PR cannot merge as-is because the CLI change widens bare multi-account binds to wildcard scope, affects unbind semantics through shared parsing, and still lacks real behavior proof.

## What This Changes

The PR adds accountId binding docs/JSDoc and CLI examples, and changes bare multi-account channel binds to emit `accountId: "*"`.

## Best Possible Solution

Keep the docs/JSDoc clarification, preserve explicit account scoping by default, and add any wildcard convenience only in a creation-specific flow after maintainer approval, tests, and real CLI proof.

## Reproduction Assessment

Yes. Source inspection shows the PR-head wildcard fallback flows through shared `parseBindingSpecs()` into both bind and unbind, while current main removes bindings by exact normalized keys.

## Solution Assessment

No. The docs clarification is useful, but changing shared bare-bind parsing is too broad; the safer path is explicit docs plus a creation-only wildcard default if maintainers approve that routing policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve unbind semantics for bare channel specs:** `src/commands/agents.bindings.ts:264-266`
  - body: `resolveBindingAccountId()` feeds `parseBindingSpecs()`, and both `agents bind` and `agents unbind` use that parsed shape. Returning `"*"` for a bare multi-account channel makes `openclaw agents unbind --bind <channel>` look for a wildcard binding instead of an existing omitted/default-account binding, so the previous shorthand can no longer remove persisted channel-only bindings.
  - confidence: 0.93
- **[P2] Correct the omitted-account JSDoc:** `src/config/types.agents.ts:32-34`
  - body: The JSDoc says omitted `accountId` follows `channels.<channel>.defaultAccount`, but routing indexes empty patterns under literal `DEFAULT_ACCOUNT_ID` and only looks up the inbound account bucket plus wildcard bucket. Configs whose configured default is not literally `default` would still miss.
  - confidence: 0.9
- **[P2] Make bare bind docs match the new behavior:** `docs/cli/agents.md:61-63`
  - body: The new table says bare `--bind <channel>` matches the default account only, but the PR code returns `"*"` when a plugin lists multiple accounts. The docs need to match whichever behavior is approved.
  - confidence: 0.87
- **[P3] Align the remaining accountId examples:** `docs/concepts/multi-agent.md:423-427`
  - body: The new tip recommends `accountId: "*"` as the safe default, but the same page still has the DM split and family group examples without `accountId`. Readers can still copy the pattern this PR is trying to steer them away from.
  - confidence: 0.82
