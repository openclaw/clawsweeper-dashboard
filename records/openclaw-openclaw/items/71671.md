## Summary

Keep open: current main still counts stale non-contiguous no-progress tool outcomes, and this PR is a focused, tested implementation for that bug rather than cleanup-close material.

## What This Changes

This PR changes `getNoProgressStreak` to count only the contiguous no-progress tail, adds two loop-detection regression tests, and adds a changelog entry.

## Best Possible Solution

Land this scoped tail-semantics fix, or an equivalent maintainer patch, with regression coverage for both known polling no-progress detection and the global no-progress breaker.

## Reproduction Assessment

Yes. A pure unit reproduction is to record repeated identical `process` poll outcomes to the critical threshold, record another successful tool outcome, then ask `detectToolCallLoop` about the same `process` retry; current main still counts the stale earlier poll records.

## Solution Assessment

Yes. Stopping `getNoProgressStreak` at the first non-matching or missing-result tail record is the narrowest maintainable fix and matches nearby contiguous-tail repeat helpers while preserving existing thresholds and generic-repeat policy.
