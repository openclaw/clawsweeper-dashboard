## Summary

Keep open: the stale-tail bug is real, but this PR changes the shared no-progress helper used by the documented global circuit breaker, so it would weaken global loop blocking while the related session-global breaker work is still open. It also still lacks external contributor real behavior proof.

## What This Changes

The PR changes `getNoProgressStreak` to stop at the first nonmatching or missing-result history record and adds two regression tests plus a changelog entry.

## Best Possible Solution

Split the implementation so per-pattern known-poll and generic no-progress detection uses contiguous-tail semantics while the global circuit breaker keeps or gains a session/run-scoped aggregate, then refresh the branch and validate with focused loop-detection tests.

## Reproduction Assessment

Yes. Source inspection is enough: seed identical `process` poll no-progress records, insert an intervening different tool, then evaluate the same `process` retry; current main skips past the intervening record and can count stale matches.

## Solution Assessment

No as written. The tail-semantics change is the right direction for per-pattern stale matching, but applying it to the helper shared with `globalCircuitBreakerThreshold` is not the narrowest maintainable fix.
