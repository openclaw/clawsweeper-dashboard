## Summary

Keep open. Current main still has the non-contiguous no-progress counting behavior, and this PR is a focused fix with targeted regression coverage; it is not obsolete, but the contributor branch currently needs rebase or conflict resolution before normal maintainer review and fresh validation.

## What This Changes

This PR changes no-progress streak counting to stop at the current repeated tool-call tail, adds two loop-detection regression tests, and adds a changelog entry.

## Best Possible Solution

Land this scoped tail-semantics fix, or an equivalent maintainer patch, after rebasing onto current main and preserving regression coverage for both known polling no-progress detection and the global no-progress circuit breaker.

## Reproduction Assessment

Yes. A pure unit reproduction is to record identical `process` poll outcomes to the critical threshold, record a different successful tool outcome, then ask loop detection about the same `process` retry; current main still counts stale earlier poll records because `getNoProgressStreak` skips over the intervening record.

## Solution Assessment

Yes. Stopping `getNoProgressStreak` at the first non-matching or missing-result tail record is the narrowest maintainable fix and matches nearby contiguous-tail detectors while leaving generic repeat policy unchanged.
