## Summary

Keep open: the PR targets a real stale-history bug, but the patch also changes the shared count used by the documented global circuit breaker and lacks contributor-supplied after-fix real behavior proof. The branch is also conflicting against current main, and the related session-global breaker work should be reconciled before this expectation is locked in.

## What This Changes

The PR changes no-progress loop detection to stop counting stale non-contiguous matching tool results after an intervening tool call, adds two regression tests, and updates the changelog.

## Best Possible Solution

Keep tail-scoped counting for per-pattern known-poll and generic no-progress checks, but compute the global circuit breaker from a separate session/run-scoped aggregate and validate both detector and before-tool-call paths.

## Reproduction Assessment

Yes, source-reproducible. Seed identical `process` poll no-progress records, insert another successful tool call, then evaluate the same `process` retry; current main skips the intervening record and can count stale matches.

## Solution Assessment

No as written. The tail-semantics change is the right direction for per-pattern stale matching, but applying it to the helper shared with `globalCircuitBreakerThreshold` weakens the documented global hard stop.
