## Summary

Keep open: the stale no-progress tail bug is still source-reproducible on current main, but this PR cannot merge as written because it applies tail semantics to the shared global circuit-breaker counter, lacks contributor-supplied real behavior proof, and conflicts with current main.

## What This Changes

The PR changes `getNoProgressStreak` to count only contiguous tail matches, adds loop-detection regression tests, and adds a changelog entry.

## Best Possible Solution

Split tail-scoped per-pattern no-progress detection from a session/run-scoped global breaker, then refresh the branch and prove the behavior with focused detector and before-tool-call checks.

## Reproduction Assessment

Yes. Source inspection is enough: seed identical `process` poll no-progress records, insert an intervening different tool, then evaluate the same `process` retry; current main skips past the intervening record and can count stale matches.

## Solution Assessment

No as written. Tail semantics are right for per-pattern stale matching, but the global hard stop needs a separate session/run aggregate; https://github.com/openclaw/openclaw/pull/79267 is the current canonical implementation path for that half.
