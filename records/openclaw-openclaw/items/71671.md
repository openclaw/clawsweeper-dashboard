## Summary

Keep open: current main still has the stale no-progress scan this PR targets, but the patch is not merge-ready because it applies tail semantics to the helper that also enforces the documented global circuit breaker, lacks after-fix real behavior proof, and needs reconciliation with https://github.com/openclaw/openclaw/pull/79267.

## What This Changes

The PR changes `getNoProgressStreak` to count only the contiguous trailing same tool/args/result records, adds two regression tests, and adds a changelog entry.

## Best Possible Solution

Keep tail-scoped counting for per-pattern known-poll and generic no-progress checks, but preserve the global breaker with a separate session/run-scoped aggregate and align this PR with https://github.com/openclaw/openclaw/pull/79267 before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed identical no-progress `process` poll outcomes, insert another successful tool call, then evaluate the same `process` retry; current main skips the intervening record and can count stale matches.

## Solution Assessment

No as written. Tail-scoped counting is the right direction for per-pattern detectors, but the global breaker should be computed separately instead of inheriting that tail-only helper behavior.
