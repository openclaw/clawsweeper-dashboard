## Summary

Keep open: the underlying accountId bug is still present on current main and this PR is a useful implementation attempt paired with open issue #43866, but the branch is stale against current hook request-handler and cron delivery policy work and is not merge-ready because it would need a careful rebase plus security-sensitive hook metadata/account-selection review.

## What This Changes

The PR adds accountId parsing and forwarding for /hooks/agent, routes hook dispatch through cron normalization, changes isolated cron message-tool policy, adds tests, and adds a changelog entry.

## Best Possible Solution

Land a rebased, narrow gateway fix that trims accountId in HookAgentPayload/normalizeAgentPayload, includes it in the active /hooks/agent replay scope and dispatch delivery, preserves sanitized hook names plus current untrusted completion routing, leaves the newer cron message-tool policy unchanged unless separately approved, and updates webhook docs, changelog credit, and focused hook/cron tests.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main: send /hooks/agent with accountId and the field is not parsed by normalizeAgentPayload or included in the CronJob.delivery object built by dispatchAgentHook.

## Solution Assessment

No. Forwarding accountId is the right fix direction, but this PR is not currently the best implementation because it is based on older hook handler and cron policy code, routes raw hook names into prompt-visible metadata, and omits docs/changelog policy follow-through.
