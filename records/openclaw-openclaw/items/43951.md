## Summary

Current main still drops `accountId` from `/hooks/agent`, so this PR addresses a real P1 delivery bug and should remain open. The branch is now conflicting and should be rebased as a narrower patch that preserves current hook construction and delivery ownership; it also needs real multi-account WhatsApp proof and removal of the release-owned changelog edit.

## What This Changes

The PR adds `accountId` normalization and hook delivery propagation, includes it in replay scoping, routes hook jobs through cron normalization, changes shared-run message-tool policy, surfaces normalization failures as HTTP 500, and adds focused tests.

## Best Possible Solution

Rebase as a narrow current-main fix that adds a trimmed optional `accountId` to hook input, replay scope, and the existing hand-built `CronDelivery`, while preserving current asynchronous dispatch, session lifecycle, cron defaults, HTTP semantics, and message-tool policy unless owners explicitly approve and prove a broader ownership change.

## Reproduction Assessment

Yes, from current source: `/hooks/agent` never normalizes `accountId`, and the hand-built delivery object has no field for it, making the multi-account failure path deterministic. This review did not run a real WhatsApp transport reproduction.

## Solution Assessment

No, not in its current shape; preserving `accountId` is correct, but direct propagation through the current hook contract is narrower and safer than coupling hooks to cron create defaults and changing shared message-tool ownership.
