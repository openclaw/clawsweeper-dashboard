## Summary

Keep open: current main still drops `accountId` from `/hooks/agent`, and this PR is a plausible fix path but is not mergeable as-is because it regresses hook-name sanitization/shared chat-delivery policy and still lacks real multi-account proof.

## What This Changes

The PR adds a `/hooks/agent` `accountId` field, threads it into hook dispatch and cron delivery normalization, adjusts shared cron message-tool policy, and adds tests plus a changelog entry.

## Best Possible Solution

Land a rebased or replacement fix that threads trimmed `accountId` through `/hooks/agent` normalization, replay scope, and `CronDelivery` while preserving sanitized hook names, current non-webhook message-tool policy, public docs, and real multi-account proof.

## Reproduction Assessment

Yes, at source level. Current main never reads `payload.accountId` in `normalizeAgentPayload()`, and hook-created delivery only carries `mode`, `channel`, and `to`, so a hook request with `accountId` cannot produce `job.delivery.accountId`.

## Solution Assessment

No. Carrying `accountId` through the existing hook and cron delivery contracts is the right direction, but this branch should not bypass hook-name sanitization, regress current shared chat-delivery policy, or ship without docs and real setup proof.

## Review Findings

Overall correctness: patch is incorrect

- **[P1] Use sanitized hook names for cron jobs:** `src/gateway/server/hooks.ts:60`
  - body: The patch passes raw `value.name` into `normalizeCronJobCreate()`. Current main sanitizes external hook names before using them in cron metadata and hook event text, so this should pass the sanitized name to keep spoofed system markers out of job metadata.
