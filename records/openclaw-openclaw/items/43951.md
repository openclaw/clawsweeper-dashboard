## Summary

Keep this PR open, but it is not merge-ready: current main still drops hook `accountId`, while the branch is conflicting, misses the active hook handler path, weakens hook-name sanitization, changes current cron message-tool policy, and lacks real multi-account delivery proof.

## What This Changes

The PR adds `accountId` parsing for `/hooks/agent`, routes hook-created cron jobs through cron normalization, changes shared cron message-tool policy, adds focused tests, and adds a changelog entry.

## Best Possible Solution

Land a rebased narrow fix that threads a trimmed `accountId` through `HookAgentPayload`, the active hook replay scope, and `CronDelivery` while preserving sanitized names and current cron tool policy, with docs and real multi-account proof.

## Reproduction Assessment

Yes, at source level. Current main never reads `payload.accountId` in `normalizeAgentPayload()` and hook-created delivery only carries `mode`, `channel`, and `to`, so a hook request with `accountId` cannot produce `job.delivery.accountId`.

## Solution Assessment

No. Forwarding `accountId` is the right direction, but this branch must be rebased onto the active hook handler, preserve `safeName` and current message-tool policy, document account selection, and provide real behavior proof before merge.
