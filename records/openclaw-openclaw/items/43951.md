## Summary

The underlying `accountId` loss is real and still present on current main, but this stale, conflicted PR broadens the fix by enabling a second delivery path that can duplicate some message-tool sends; it also lacks required live multi-account transport proof.

## What This Changes

The PR parses and replay-scopes hook `accountId`, routes hook jobs through cron normalization, surfaces normalization failures as HTTP 500, preserves hook sessions, and conditionally re-enables the message tool for resolved shared deliveries.

## Best Possible Solution

Rebase onto current main and implement the smallest fix: normalize and replay-scope `accountId`, preserve it in the hook delivery target, keep one delivery owner unless current committed-delivery evidence proves safe deduplication, and verify exactly-once delivery in a real multi-account setup.

## Reproduction Assessment

Yes. Current main's hook payload normalization omits `accountId`, which deterministically prevents an explicitly selected multi-account channel identity from reaching delivery.

## Solution Assessment

No. Preserving `accountId` is the correct fix boundary, but routing through a broader normalizer and changing message-tool ownership are unnecessary unless their compatibility and exactly-once behavior are separately proven.
