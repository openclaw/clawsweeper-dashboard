## Summary

Keep open: the branch addresses a real accountId-drop bug, but it is not mergeable because it updates the old hook handler, bypasses current hook-name sanitization, changes current cron message-tool policy, and lacks live after-fix proof.

## What This Changes

The PR adds `accountId` parsing and propagation for `/hooks/agent`, routes hook-created delivery through cron normalization, adjusts cron message-tool policy, and adds focused tests plus a changelog entry.

## Best Possible Solution

Land a rebased, narrow fix that parses and trims `accountId` in the active hook request handler, carries it through replay scope and `CronDelivery`, preserves sanitized hook metadata and current cron message-tool policy, documents account selection, and includes focused tests plus redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main is enough: the hook normalizer never reads `accountId`, the active replay scope omits it, and hook-created cron delivery only carries `mode`, `channel`, and `to`.

## Solution Assessment

No. Forwarding `accountId` is the right direction, but this branch should be rebased onto the active hook handler, keep `sanitizeInboundSystemTags`, preserve the current cron message-tool policy, and add docs/proof before merge.
