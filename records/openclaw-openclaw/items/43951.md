## Summary

Keep this PR open: the underlying accountId drop still exists on current main, but this branch is not merge-ready because it is based on older hook plumbing, weakens the current hook-name sanitization boundary, changes current cron message-tool policy, and lacks real behavior proof.

## What This Changes

The PR branch adds `accountId` normalization for `/hooks/agent`, threads it into hook-created cron delivery, changes shared cron message-tool policy, and adds focused tests plus a changelog entry.

## Best Possible Solution

Land a rebased, narrow fix that parses and trims accountId in the active hook handler, carries it through replay scope and CronDelivery, preserves sanitized hook metadata and current cron message-tool policy, documents account selection, and includes focused tests plus redacted real behavior proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: current main never reads payload.accountId and hook-created delivery only carries mode, channel, and to, so a hook request with accountId cannot produce job.delivery.accountId.

## Solution Assessment

Is this the best way to solve the issue? No. Forwarding accountId is the right direction, but this branch should be rebased onto the active hook handler, keep sanitizeInboundSystemTags, preserve current cron message-tool policy, and add docs/proof before merge.
