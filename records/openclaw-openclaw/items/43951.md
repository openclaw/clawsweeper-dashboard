## Summary

Keep open: current main still drops `/hooks/agent` `accountId`, so the PR targets a valid bug, but the branch is not merge-ready because it is stale against the active hook handler and mixes the accountId fix with cron policy and hook-trust changes that need correction.

## What This Changes

The PR adds `accountId` parsing and forwarding for `/hooks/agent`, routes hook dispatch through cron normalization, adjusts shared cron message-tool policy, and adds focused tests plus a changelog entry.

## Best Possible Solution

Land a rebased, narrow gateway fix that parses and trims `accountId`, includes it in replay scoping and `CronJob.delivery` through the active hook handler, preserves sanitized hook metadata and existing cron message-tool policy unless separately approved, and updates docs, changelog credit, and focused tests.

## Reproduction Assessment

Yes. Source inspection on current main is enough: `/hooks/agent` normalization ignores `accountId`, and hook dispatch builds a `CronJob.delivery` object without it.

## Solution Assessment

No. Forwarding `accountId` is the right direction, but this PR is not currently the best implementation because it is stale against the active handler and carries cron policy/security-sensitive behavior changes beyond the narrow hook fix.
