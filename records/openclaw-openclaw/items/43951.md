## Summary

Keep open: current main still drops `accountId` on the `/hooks/agent` path, but this PR is not merge-ready because the head conflicts with main, has blocking hook-name and cron policy regressions, and still lacks real behavior proof.

## What This Changes

The PR adds `accountId` normalization and propagation for `/hooks/agent`, routes hook dispatch through cron normalization, adjusts shared cron message-tool policy, and adds tests plus a changelog entry.

## Best Possible Solution

Land a narrow rebased fix or replacement that threads trimmed `accountId` through hook normalization, replay scope, and `CronDelivery`, while preserving sanitized hook names, current cron message-tool behavior, tests, public hook docs, and live proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `/hooks/agent` never normalizes `payload.accountId`, hook-created delivery carries only `mode`, `channel`, and `to`, and cron can only route by `accountId` once that field reaches `CronDelivery`.

## Solution Assessment

No. The accountId threading direction is right, but this branch should be rebased or replaced so it preserves sanitized hook names, current announce message-tool policy, docs, and real behavior proof.
