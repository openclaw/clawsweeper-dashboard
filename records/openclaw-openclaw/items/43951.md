## Summary

Keep this PR open: the underlying `/hooks/agent` accountId bug is still present on current `main`, but the PR is not merge-ready because it is stale against the current hook request split, carries a raw hook name into cron metadata, changes cron message-tool policy beyond the accountId fix, and still needs docs/changelog follow-through.

## What This Changes

The PR adds `accountId` parsing and forwarding for `/hooks/agent`, routes hook dispatch through cron normalization, changes isolated cron message-tool policy, and adds focused tests plus a changelog entry.

## Best Possible Solution

Land a rebased, narrow gateway fix that parses and trims `accountId`, includes it in replay scoping and `CronJob.delivery`, preserves sanitized/untrusted hook metadata and current cron tool policy unless separately approved, and updates docs, changelog credit, and focused tests.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists on current main: send `/hooks/agent` with `accountId`, and the field is neither parsed by `normalizeAgentPayload()` nor included in the hook-built `CronJob.delivery`.

## Solution Assessment

No. Forwarding `accountId` is the right direction, but this PR is not currently the best implementation because it is stale against the active handler and mixes the hook fix with cron policy/security-sensitive behavior changes.
