## What This Changes

The PR threads `accountId` through `/hooks/agent` normalization and cron delivery dispatch, updates replay scope/error handling, and adds focused hook/cron policy tests.

## Best Possible Solution

Land a narrow rebased fix that carries a trimmed `accountId` through hook normalization, replay scope, and `CronDelivery` while preserving current hook request-handler structure and cron message-tool policy semantics.

## Reproduction Assessment

Yes, source-level. Current main cannot carry `accountId` from `/hooks/agent` because the hook payload type and normalizer omit it and the dispatcher only builds delivery from `mode/channel/to`.

## Solution Assessment

Yes in direction, but not merge-ready as submitted. The maintainable solution is to thread `accountId` through the existing hook and cron delivery contracts, include replay-key coverage, and attach real multi-account proof.
