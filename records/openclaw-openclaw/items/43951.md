## Summary

Keep this PR open: current main still has the source-level `/hooks/agent` accountId drop, but this branch is not merge-ready because it is conflicting, lacks required real behavior proof, and still has correctness/security findings in the proposed patch.

## What This Changes

The branch adds `accountId` to `/hooks/agent`, threads it into cron delivery, changes shared cron message-tool policy, adds focused tests, and updates the changelog.

## Best Possible Solution

Land a rebased narrow fix or replacement that threads a trimmed `accountId` through hook normalization, replay scope, and `CronDelivery` while preserving sanitized hook names, current non-webhook message-tool policy, public docs, and real multi-account proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main never reads `payload.accountId`, and hook-created delivery only carries `mode`, `channel`, and `to`, so a request with `accountId` cannot produce `job.delivery.accountId`. I did not run a live WhatsApp setup in this read-only review.

## Solution Assessment

No. Carrying `accountId` through the existing hook and cron delivery contracts is the right direction, but this branch should not bypass hook-name sanitization or regress the current shared chat-delivery message-tool policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use sanitized hook names for cron jobs:** `src/gateway/server/hooks.ts:60`
  - body: Current main sanitizes external hook names before putting them in cron metadata and hook event text, but this patch passes raw `value.name` into `normalizeCronJobCreate()`. Use the sanitized name for the job and event text so spoofed system markers cannot survive into queued metadata.
  - confidence: 0.9
- **[P2] Preserve shared chat-delivery tool policy:** `src/cron/isolated-agent/run.ts:123-126`
  - body: Current main keeps the message tool enabled for non-webhook delivery, including announce delivery without an account ID. This change disables it for shared delivery unless the target has an explicit `accountId`, which regresses channels and targets that do not use account IDs.
  - confidence: 0.88
