## Summary

Keep this PR open: the underlying hook accountId bug is still source-reproducible on current main, but this branch is not mergeable as-is because it targets older hook plumbing, weakens hook-name sanitization, changes current cron message-tool policy, and lacks real behavior proof.

## What This Changes

The PR adds accountId parsing for /hooks/agent, threads it into hook-created cron delivery and replay scope, changes shared cron message-tool policy, and adds focused tests plus a changelog entry.

## Best Possible Solution

Land a rebased, narrow fix that trims accountId in the active hook handler, includes it in replay scope and CronDelivery, preserves sanitized hook metadata and current cron message-tool policy, documents account selection, and includes focused tests plus redacted real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main never reads payload.accountId and hook-created delivery only carries mode, channel, and to, so a hook request with accountId cannot produce job.delivery.accountId.

## Solution Assessment

No. Forwarding accountId is the right direction, but this branch should be rebased onto the active hook request handler, keep sanitizeInboundSystemTags, preserve current cron message-tool policy, document the hook token account-selection authority, and add real behavior proof.
