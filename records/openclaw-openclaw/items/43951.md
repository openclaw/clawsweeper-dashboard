## Summary

Keep this PR open: current main still has the reported `/hooks/agent` `accountId` gap, but the branch still needs code changes, documentation/security follow-through, and redacted real multi-account proof before it is mergeable.

## What This Changes

The PR adds `accountId` support for `/hooks/agent`, changes hook dispatch to preserve it through cron delivery, adjusts shared cron message-tool policy, and adds focused tests plus a changelog entry.

## Best Possible Solution

Land a narrow rebased fix or replacement that threads a trimmed `accountId` through hook payload normalization, replay scope, and `CronDelivery` while preserving sanitized names, current shared chat-delivery policy, public docs, and real multi-account proof.

## Reproduction Assessment

Yes, at source level. Current main never reads `payload.accountId` in `normalizeAgentPayload()`, and hook-created delivery only carries `mode`, `channel`, and `to`, so a hook request with `accountId` cannot produce `job.delivery.accountId`.

## Solution Assessment

No. Carrying `accountId` through the existing hook and cron delivery contracts is the right direction, but this branch should not also bypass hook-name sanitization or regress current shared message-tool policy.
