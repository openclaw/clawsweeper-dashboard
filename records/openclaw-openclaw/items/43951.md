## Summary

Keep this PR open, but it is not merge-ready: the underlying accountId drop still exists on current main, the PR branch is conflicting and targets an older hook handler split, prior review found real correctness/security blockers, and the contributor has not provided real multi-account WhatsApp behavior proof.

## What This Changes

The PR adds accountId parsing and propagation for /hooks/agent, changes hook-created cron dispatch and shared cron message-tool policy, and adds focused tests plus a changelog entry.

## Best Possible Solution

Land a rebased, narrow fix that carries a trimmed accountId through the active hook request handler, replay scope, and CronDelivery while preserving sanitized hook metadata and current cron message-tool policy, with docs and real setup proof.

## Reproduction Assessment

Yes, at source level. Current main never reads payload.accountId and hook-created delivery only carries mode, channel, and to, so a hook request with accountId cannot produce job.delivery.accountId.

## Solution Assessment

No. Forwarding accountId is the right direction, but this branch needs a current-main rebase, sanitized-name preservation, current message-tool policy preservation, operator docs, and real behavior proof before merge.
