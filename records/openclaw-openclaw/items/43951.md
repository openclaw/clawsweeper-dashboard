## Summary

Keep open and do not merge as-is: current main still drops `/hooks/agent` `accountId`, but this PR is behind the active hook handler, weakens hook-name sanitization, changes current cron chat-delivery tool policy, and lacks real behavior proof.

## What This Changes

The branch adds `accountId` to hook payload normalization, routes hook dispatch through cron normalization, changes shared cron message-tool policy, and adds focused tests plus a changelog entry.

## Best Possible Solution

Land a rebased, narrow fix that parses and trims `accountId` in the active hook handler, carries it through replay scope and `CronDelivery`, preserves sanitized hook metadata and current cron message-tool policy, documents account selection, and includes focused tests plus redacted real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main is enough: `normalizeAgentPayload()` never reads `accountId`, the active replay scope omits it, and hook-created cron delivery only carries `mode`, `channel`, and `to`.

## Solution Assessment

No. Forwarding `accountId` is the right direction, but this branch should be rebased onto the active hook handler, keep `sanitizeInboundSystemTags`, preserve the current cron message-tool policy, and add docs/proof before merge.
