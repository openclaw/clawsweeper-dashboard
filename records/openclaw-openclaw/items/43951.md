## Summary

Keep this PR open because it targets a real current-main bug, but it is not mergeable as-is: the branch is conflicting/stale against the active hook handler, it weakens existing hook-name sanitization, it changes cron message-tool policy beyond the narrow accountId fix, and it lacks real after-fix behavior proof.

## What This Changes

The PR adds `accountId` to `/hooks/agent` normalization and delivery, routes hook-created jobs through cron normalization, adjusts cron message-tool policy, and adds focused tests plus a changelog entry.

## Best Possible Solution

Land a rebased, narrow gateway fix that parses and trims `accountId`, carries it through the active hook request handler replay scope and cron delivery object, preserves sanitized hook metadata and current non-webhook cron tool policy, documents hook account selection, and adds focused regression tests.

## Reproduction Assessment

Yes. Source inspection on current main is enough: `/hooks/agent` normalization omits `accountId`, and the hook-created cron delivery object contains only `mode`, `channel`, and `to`.

## Solution Assessment

No. Forwarding `accountId` is the right direction, but this branch should be rebased onto `src/gateway/server/hooks-request-handler.ts`, keep the sanitized hook name, and avoid changing cron message-tool policy beyond the accountId delivery path.
