## Summary

Keep open: the linked `/hooks/agent` multi-account delivery bug still exists on current main, and this PR is aimed at the right area, but the branch is currently conflicting and should not merge until it preserves hook-name sanitization, preserves the current non-webhook message-tool policy, adds operator docs, and provides real behavior proof.

## What This Changes

The PR adds `accountId` parsing to `/hooks/agent`, threads it into cron delivery and replay handling, changes cron message-tool policy, adds focused tests, and updates the changelog.

## Best Possible Solution

Land a rebased narrow fix or replacement that threads a trimmed `accountId` through `/hooks/agent` normalization, replay scope, and `CronDelivery` while preserving sanitized hook names and current non-webhook message-tool behavior, with docs and real multi-account proof.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: current main never reads `payload.accountId`, hook-created delivery only carries `mode`, `channel`, and `to`, and the replay key lacks account ID; I did not run a live WhatsApp setup.

## Solution Assessment

No. Carrying `accountId` through the hook and cron delivery contracts is the right fix direction, but this branch should be rebased and revised so it does not bypass hook-name sanitization or regress current announce message-tool policy.

  - body: Current main sanitizes external hook names before putting them in cron metadata and hook event text, but this patch passes raw `value.name` into cron job normalization. Use the sanitized name for the job and event text so spoofed system markers cannot survive into queued metadata.
