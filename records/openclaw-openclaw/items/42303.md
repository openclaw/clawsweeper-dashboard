## Summary

Keep open. Current main still does not implement the PR's explicit Linuxbrew Docker build option, but the branch is not merge-ready: it is stale against the maintained Docker setup script, reintroduces runtime build steps, changes default non-brew images, and adds an unpinned Homebrew installer execution path. The linked #14593 problem also has competing open solution PRs, so this needs maintainer product and security direction.

## What This Changes

The branch adds an optional `OPENCLAW_INSTALL_BREW` Docker build arg/Linuxbrew install block, wires that variable through `docker-setup.sh`, and adds an `OPENCLAW_SKIP_BUILD` branch.

## Best Possible Solution

Maintainers should choose the canonical #14593 direction first. If opt-in Linuxbrew is the desired path, implement it against the current multi-stage Dockerfile and `scripts/docker/setup.sh`, keep environment changes conditional, use a pinned or verified installer policy, update docs/changelog, and add focused Docker/setup coverage; otherwise land one of the narrower alternatives and supersede this branch.

## Reproduction Assessment

Yes. Static current-main reproduction is high-confidence: `OPENCLAW_INSTALL_BREW` is not consumed by the Dockerfile or maintained setup script, and brew-kind skill installs still return `brew not installed` when no brew executable exists.

## Solution Assessment

No. Restoring Linuxbrew may be a valid product answer, but this patch is not the best implementation as-is because it is stale against current Docker setup, changes default image environments, reruns builds in the runtime stage, and lacks an approved installer integrity model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Run the installer from a downloaded file:** `Dockerfile:215`
  - body: The `su -c` string expands `$(curl ...)` before switching users, then passes the downloaded Homebrew installer as a single-quoted `bash -c` program. The installer contains single quotes, so `OPENCLAW_INSTALL_BREW=1` builds can fail while parsing the script instead of installing brew.
  - confidence: 0.86
- **[P2] Keep Homebrew environment changes conditional:** `Dockerfile:222-225`
  - body: These `ENV` instructions run for every image, even when `OPENCLAW_INSTALL_BREW` is unset. Default Docker builds would advertise a non-existent Linuxbrew prefix and prepend dead directories to `PATH`, so an opt-in feature changes non-brew containers.
  - confidence: 0.9
- **[P2] Preserve the current multi-stage runtime boundary:** `Dockerfile:227-240`
  - body: The added `COPY . .` and `pnpm build`/`pnpm ui:build` commands copy full source into the final runtime image and rerun builds after current main already copies pruned runtime assets from the build stage. That can fail with production dependencies and undoes the minimal runtime image design.
  - confidence: 0.88
- **[P2] Move setup wiring to scripts/docker/setup.sh:** `docker-setup.sh:413-426`
  - body: Current main's root `docker-setup.sh` is only a wrapper around `scripts/docker/setup.sh`. Wiring `OPENCLAW_INSTALL_BREW` and skip-build into the old root script body leaves the maintained setup path without the new env persistence and build arg after a proper rebase.
  - confidence: 0.86
- **[P2] Point skip-build recovery at an existing command:** `docker-setup.sh:418`
  - body: This error tells users to run `openclaw build`, but the current CLI command registry has no top-level `build` command. The recovery hint should point to a real path, such as rerunning without the skip flag or an explicit `docker build ...` command.
  - confidence: 0.84
