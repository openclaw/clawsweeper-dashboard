## Summary

Keep open, but do not merge as-is. Current main still lacks the requested Docker Linuxbrew build option, so the PR is not obsolete; however, the proposed patch is stale against the maintained Docker setup script, has concrete Dockerfile correctness bugs, and adds a moving remote installer execution path that needs maintainer supply-chain policy approval.

## What This Changes

The PR re-adds an optional `OPENCLAW_INSTALL_BREW` Docker build arg and Linuxbrew install block, wires that env var through Docker setup persistence/build args, and adds an `OPENCLAW_SKIP_BUILD` path.

## Best Possible Solution

Maintainers should first choose the canonical #14593 product direction. If Linuxbrew in the image is the desired path, rework it against current `Dockerfile` and `scripts/docker/setup.sh`, keep all Homebrew environment changes conditional, use a safer installer execution and integrity policy, document the new Docker option, add focused setup/Dockerfile tests, and then supersede the alternative PRs as appropriate.

## Reproduction Assessment

Yes. Current-main static reproduction is high-confidence: `OPENCLAW_INSTALL_BREW=1` is not consumed by the Dockerfile or maintained setup script, and `kind: "brew"` skill installs still return `brew not installed` when no brew executable exists.

## Solution Assessment

No. Restoring opt-in Linuxbrew may be a valid solution, but this PR is not the best implementation as-is because it targets stale setup wiring, pollutes default images, has unsafe installer quoting, points to a missing recovery command, and lacks an approved installer integrity policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Run the installer from a file instead of nested quoting:** `Dockerfile:215`
  - body: The `su -c` string expands `$(curl ...)` before switching users, then passes the downloaded installer as a single-quoted `bash -c` program. Homebrew's installer contains single quotes, so `OPENCLAW_INSTALL_BREW=1` builds can fail while parsing the script instead of installing brew.
  - confidence: 0.86
- **[P2] Keep Homebrew environment changes conditional:** `Dockerfile:222-225`
  - body: These `ENV` instructions run for every image, even when `OPENCLAW_INSTALL_BREW` is unset. Default Docker builds would advertise a non-existent Linuxbrew prefix and prepend dead directories to `PATH`, so an opt-in feature changes non-brew containers.
  - confidence: 0.9
- **[P2] Apply setup wiring to the maintained setup script:** `docker-setup.sh:413-426`
  - body: Current main's root `docker-setup.sh` is only a wrapper around `scripts/docker/setup.sh`. Adding the brew export, `.env` persistence, and build arg to the old root script body will be lost on rebase unless the wiring moves to the maintained script.
  - confidence: 0.82
- **[P2] Point skip-build recovery at an existing build path:** `docker-setup.sh:418`
  - body: When `OPENCLAW_SKIP_BUILD` is set and the image is missing, this branch tells users to run `openclaw build`, but current CLI command catalogs have no top-level `build` command. The error should point to `docker build ...` or rerunning without the skip flag.
  - confidence: 0.84

## Security Review
