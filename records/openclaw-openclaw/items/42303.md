## Summary

Keep open: the Docker/Linuxbrew problem is still not solved on current main, but this PR head should not be merged as-is because it is stale against the current Docker layout and has concrete Dockerfile/setup-script blockers already surfaced in review.

## What This Changes

The PR re-adds an opt-in `OPENCLAW_INSTALL_BREW` Docker build arg, setup-script passthrough, and skip-build path to install Linuxbrew in Docker images.

## Best Possible Solution

Resolve #14593 with one chosen Docker/skills strategy against the current Docker layout, with tests, docs, and changelog aligned.

## Reproduction Assessment

Yes, from source inspection: current main does not install or pass through `OPENCLAW_INSTALL_BREW`, and the skill installer returns a brew-missing failure when no brew executable is available. I did not run a live Docker build in this read-only review.

## Solution Assessment

No. The PR's direction may be viable, but this head has a build-breaking installer command, unscoped Homebrew env, an invalid recovery command, and stale file layout; the safer path is a current-main replacement after maintainers choose the #14593 strategy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Execute the Homebrew installer from a file:** `Dockerfile:215`
  - body: This expands `$(curl ...)` in the Docker shell and then passes the full installer as a single-quoted `bash -c` string. The Homebrew installer contains single quotes, so a brew-enabled build can fail while parsing before `brew` is installed; download to a temp file or otherwise avoid embedding the script in shell quotes.
  - confidence: 0.92
- **[P3] Do not bake Homebrew env into non-brew images:** `Dockerfile:222-225`
  - body: These `ENV` instructions run regardless of `OPENCLAW_INSTALL_BREW`, so default images get `HOMEBREW_*` values and PATH entries for directories that were never created. Keep the runtime environment scoped to brew-enabled builds, for example through a profile drop-in created inside the conditional install block.
  - confidence: 0.84
- **[P2] Point skip-build failures at a real recovery command:** `docker-setup.sh:418`
  - body: The missing-image path tells users to run `openclaw build`, but current main does not register a top-level or subcommand named `build`. This leaves users without a valid recovery step; point them to `docker build ...` or rerun without `OPENCLAW_SKIP_BUILD`.
  - confidence: 0.9
- **[P3] Document the new Docker build option:** `docker-setup.sh:426`
  - body: This adds a user-facing Docker env/build arg, but the PR only changes Dockerfile/setup script surfaces. The Docker docs env-var table and the user-facing changelog should be updated if maintainers keep this approach.
  - confidence: 0.76

## Security Review

Status: needs_attention

Summary: The diff introduces build-time download and execution of Homebrew's mutable installer without pinning or integrity verification.
