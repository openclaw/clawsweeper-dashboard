## Summary

Keep this PR open for maintainer follow-up, but do not merge it as-is. Current main still lacks the requested opt-in Linuxbrew Docker support, while this branch is stale against the maintained Docker setup script and has concrete Dockerfile, recovery-message, and supply-chain blockers.

## What This Changes

The PR re-adds optional `OPENCLAW_INSTALL_BREW` Docker build wiring in `Dockerfile` and `docker-setup.sh`, plus an `OPENCLAW_SKIP_BUILD` branch for prebuilt images.

## Best Possible Solution

Pick one canonical path for #14593. If Linuxbrew is the chosen path, implement it against current `Dockerfile` and `scripts/docker/setup.sh` with conditional runtime environment handling, Docker docs, setup-script tests, real Docker proof, and an explicit installer integrity policy; otherwise land one of the narrower alternatives such as #55940 or #57553 and mark the other PRs superseded.

## Reproduction Assessment

Yes. Current-main static reproduction is high-confidence: `OPENCLAW_INSTALL_BREW=1` is not consumed by the Dockerfile or maintained setup script, and a brew-backed skill install returns `brew not installed` when no brew executable is present.

## Solution Assessment

No. Restoring opt-in Linuxbrew may be a valid solution, but this PR is not the best implementation as-is because it targets the old root setup script layout, pollutes non-brew image environments, has unsafe installer quoting, references a missing recovery command, and lacks an installer integrity policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Run the installer from a file instead of nested quoting:** `Dockerfile:215`
  - body: The outer double-quoted `su -c` expands `$(curl ...)` before `su`, then the downloaded script is parsed as a single-quoted `bash -c` string. Homebrew's installer contains single quotes, so `OPENCLAW_INSTALL_BREW=1` builds can fail while parsing the script instead of installing brew.
  - confidence: 0.86
- **[P2] Move Homebrew ENV into the opt-in path:** `Dockerfile:222-225`
  - body: These `ENV` instructions run for every image, even when `OPENCLAW_INSTALL_BREW` is unset. Default Docker builds will advertise a non-existent Linuxbrew prefix and prepend dead directories to PATH, so the opt-in feature changes non-brew containers.
  - confidence: 0.9
- **[P2] Point skip-build recovery at an existing build path:** `docker-setup.sh:418`
  - body: When `OPENCLAW_SKIP_BUILD` is set and the image is missing, this branch tells users to run `openclaw build`, but current CLI registration has no top-level `build` command. The error should point to `docker build ...` or rerunning without the skip flag.
  - confidence: 0.84
- **[P2] Apply setup wiring to the maintained setup script:** `docker-setup.sh:413-426`
  - body: Current main's root `docker-setup.sh` is only a wrapper around `scripts/docker/setup.sh`. Adding the brew export, `.env` persistence, and build arg to the old root script body will be lost on rebase unless the wiring moves to the maintained script.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds an opt-in Docker build step that downloads and executes the Homebrew installer from a moving `HEAD` URL without an integrity policy.

Concerns:

- **[medium] Pin or verify the Homebrew installer:** `Dockerfile:215`

Because this code runs while producing a shipped container image, the installer source and verification model need explicit review before merge.
