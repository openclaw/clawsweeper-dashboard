## Summary

Keep open. The underlying Docker skill-install bug is still present on main and this PR is an active implementation candidate for the open canonical issue, but the patch is not mergeable as-is because it targets the deleted root setup script, breaks Docker smoke by re-running runtime builds, adds unhardened Homebrew installer execution, and lacks contributor real behavior proof.

## What This Changes

The branch adds an optional Linuxbrew install block to `Dockerfile` and attempts to pass `OPENCLAW_INSTALL_BREW` and `OPENCLAW_SKIP_BUILD` through the old root Docker setup script.

## Best Possible Solution

Keep the canonical Docker skill-install issue open while maintainers choose one package-manager strategy; if Linuxbrew-in-Docker is selected, implement it in `Dockerfile` plus `scripts/docker/setup.sh`, harden installer execution, update docs, add focused Docker coverage, and require real Docker proof.

## Reproduction Assessment

Yes, from source inspection: current Docker lacks `OPENCLAW_INSTALL_BREW`, the supported setup script does not pass a brew build arg, `openai-whisper` is brew-only, and `installSkill` returns the missing-brew failure when no brew executable resolves. I did not run a live Docker build in this read-only review.

## Solution Assessment

No. Optional Linuxbrew-in-Docker may be a valid strategy, but this PR is not the best mergeable form because it targets an obsolete setup script, reintroduces runtime build commands that already fail CI, and needs installer hardening plus maintainer package-manager direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Avoid rebuilding the app in the runtime image:** `Dockerfile:237-240`
  - body: The PR copies the full repo into the final runtime stage and runs `pnpm build`/`pnpm ui:build` after the multi-stage build has already produced runtime assets. The Docker smoke job failed on this added `pnpm build` step, so default image builds break before the brew option is even enabled; keep the final stage to runtime assets and add only the Linuxbrew runtime setup needed for this fix.
  - confidence: 0.96
- **[P2] Wire the supported Docker setup script:** `docker-setup.sh:426`
  - body: Current main and Docker docs use `scripts/docker/setup.sh`, and root `docker-setup.sh` is no longer present. This branch only persists and passes `OPENCLAW_INSTALL_BREW` through the obsolete root script, so the documented setup flow would still build without the brew arg after conflicts are resolved.
  - confidence: 0.94
- **[P2] Execute the Homebrew installer from a file:** `Dockerfile:215`
  - body: This expands the remote installer with `$(curl ...)` and passes the fetched script through a single-quoted `bash -c` string inside `su`. Installer script contents can contain quotes that break parsing; download to a temporary file or otherwise execute without embedding the script in shell quotes.
  - confidence: 0.9
- **[P3] Point skip-build recovery at a real command:** `docker-setup.sh:418`
  - body: The new missing-image path tells users to run `openclaw build`, but current CLI command registration does not expose a top-level `build` command. Point users to rerun setup without `OPENCLAW_SKIP_BUILD` or to an explicit Docker build command.
  - confidence: 0.86
- **[P3] Scope Homebrew environment to brew-enabled builds:** `Dockerfile:222-225`
  - body: These `ENV` instructions are applied to every image, including default builds where `/home/linuxbrew/.linuxbrew` does not exist. Keep Homebrew path and prefix setup scoped to images built with the brew option enabled.
  - confidence: 0.84
