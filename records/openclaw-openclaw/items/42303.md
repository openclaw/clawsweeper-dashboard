## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/14593 as the canonical report while maintainers choose one Docker skill-dependency strategy, then land a current-main fix with Docker proof, regression coverage, and Docker/skills docs updates.

## Solution Assessment

Is this the best way to solve the issue? No. Optional Linuxbrew-in-Docker may be viable, but this branch is stale against current Docker setup and introduces correctness plus supply-chain concerns; maintainers should choose the canonical package-manager strategy first.

Full review comments:

- **[P2] Wire the supported Docker setup script:** `docker-setup.sh:426`
  - body: Current main and the Docker docs use `scripts/docker/setup.sh`, while this branch wires `OPENCLAW_INSTALL_BREW` into root `docker-setup.sh`, which current main no longer tracks. After rebasing, the supported setup flow would still not pass the brew build arg.
  - confidence: 0.92
- **[P2] Execute the Homebrew installer from a file:** `Dockerfile:215`
  - body: This expands the installer with `$(curl ...)` and passes the full script as a single-quoted `bash -c` string. The Homebrew installer contains single quotes, so a brew-enabled build can fail during shell parsing; download and execute a temporary file or otherwise avoid embedding the script in quotes.
  - confidence: 0.93
- **[P3] Scope Homebrew env to brew-enabled builds:** `Dockerfile:222-225`
  - body: These `ENV` instructions run for every build, so default images get `HOMEBREW_*` values and PATH entries for directories that do not exist. Keep the runtime environment scoped to images built with brew enabled.
  - confidence: 0.86
- **[P2] Point skip-build recovery at a real command:** `docker-setup.sh:418`
  - body: The missing-image path tells users to run `openclaw build`, but the CLI does not register that top-level command. Use a real recovery path such as rerunning setup without `OPENCLAW_SKIP_BUILD` or an explicit `docker build` command.
  - confidence: 0.9
