## Summary

Keep this PR open: the branch is a focused candidate fix for the linked Telegram generated-HTML delivery regression, and current main still loses the trusted-HTML provenance when the file is staged into outbound media. I did not find a new line-level correctness bug, but the change intentionally expands a trusted local HTML boundary and needs explicit maintainer/security acceptance before merge.

## What This Changes

The PR records content-bound trusted-generated-HTML provenance in the shared state DB when outbound staging copies a trusted temp-root HTML file, then verifies the staged file's digest during later host-read reloads and prunes stale provenance during media cleanup.

## Best Possible Solution

Land this or an equivalent content-bound provenance repair after maintainer/security acceptance of the trusted-HTML lifecycle, then close https://github.com/openclaw/openclaw/issues/90557 as fixed.

## Reproduction Assessment

Yes, at source level: current main stages local reply media into media/outbound and Telegram reloads that staged .html through a host-read gate that only trusts temp-root realpaths. I did not run a fresh live Telegram reproduction in this read-only review.

## Solution Assessment

Yes, this is the best fix shape I found: it preserves the trusted-generated-HTML decision through staging with byte-bound provenance instead of trusting the whole outbound directory. The remaining question is maintainer/security acceptance of the persisted trust lifecycle, not a narrower mechanical code repair.
