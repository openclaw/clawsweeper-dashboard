## Label Justifications

- `P1`: This fixes repeatable Telegram attachment loss but currently introduces a merge-blocking authorization gap in a security-sensitive path.
- `merge-risk: 🚨 security-boundary`: The realpath-only marker can authorize replacement HTML that was not the validated artifact originally staged.
- `merge-risk: 🚨 message-delivery`: Marker-write errors are swallowed, allowing a staged HTML path to proceed until the later channel read rejects and drops delivery.

## Summary

The regression is real and the branch follows the intended SQLite ownership, but it is not safe to merge yet: the durable marker authenticates only a reusable pathname, and marker-write failures recreate the same later delivery drop. The existing Telegram recording does not prove the fix because both baseline and candidate bypassed the host-read gate and delivered successfully.

## What This Changes

The PR adds shared-state SQLite provenance for trusted generated HTML copied into outbound media staging, consults it on subsequent host reads, and prunes stale provenance during media cleanup.

## Best Possible Solution

Treat trusted HTML staging as one atomic operation that records and later verifies the exact staged artifact—prefer an opened-file identity or integrity binding rather than pathname alone—and surface provenance-write failure before returning a deliverable path.

## Reproduction Assessment

Yes. Current main validates the original temp-root HTML, stages it to a different path, and can re-read that path under host-read where the temp-root-only trust check rejects it; the issue's instrumented live logs show this exact sequence four times.

## Solution Assessment

No. SQLite is the correct repository-owned storage surface, but authenticating only a persistent pathname is weaker than binding trust to the exact staged artifact, and best-effort persistence leaves the original delivery failure possible.
