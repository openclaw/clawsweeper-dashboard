## Summary

Keep open: the PR targets a valid released Telegram message-delivery regression, but the current patch still gives durable trust to a pathname rather than the exact staged HTML artifact and does not yet supply after-fix proof for the host-read-gated Telegram path.

## What This Changes

The PR adds SQLite-backed trusted-generated-HTML provenance for outbound-staged media, marks trusted HTML during staging, consults the marker during host-read HTML loading, prunes stale rows during media cleanup, and adds regression tests.

## Best Possible Solution

Bind trusted HTML provenance to the exact staged artifact, fail visibly when provenance cannot be recorded, and then add host-read-enabled Telegram or redacted runtime proof for the second staged-path load.

## Reproduction Assessment

Yes. Source inspection and the linked live issue logs show the two-load path: temp-root HTML is accepted, the staged outbound copy is later re-read under host-read, and current main rejects that staged path.

## Solution Assessment

No. SQLite provenance is the right storage surface, but pathname-only authorization and best-effort marker writes are not the safest fix for a host-read HTML security boundary.

  - body: The marker lookup authenticates only `realpath`, and the prune pass keeps the row as long as that path is any regular file.   - confidence: 0.9
