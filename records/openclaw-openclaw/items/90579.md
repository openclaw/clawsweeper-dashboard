## Summary

Keep open: the PR targets a real Telegram delivery regression, but the current branch still makes trusted HTML authorization durable by path only, swallows marker-write failures, and lacks after-fix proof of the host-read-enabled Telegram second-load path.

## What This Changes

The PR adds SQLite-backed provenance for trusted generated HTML staged into outbound media, checks that marker during host-read reloads, prunes stale markers, and adds media/schema regression tests.

## Best Possible Solution

Bind trusted generated HTML staging to the exact staged artifact, fail visibly when provenance cannot be persisted, and verify the host-read-enabled Telegram second-load path before merge.

## Reproduction Assessment

Yes, from source and reporter logs, but I did not execute a live Telegram run. Current main can stage temp-root trusted HTML into outbound media and then Telegram re-reads it under host-read, where the temp-root-only trust check rejects the staged path.

## Solution Assessment

No. The PR is in the right owner area and SQLite is the right storage surface, but path-only trust and best-effort persistence are not the safest permanent trust contract.
