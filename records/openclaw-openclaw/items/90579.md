## What This Changes

The PR adds content-bound SQLite provenance for trusted generated HTML copied into outbound media, then accepts the staged copy only when the provenance row and already-read bytes match.

## Best Possible Solution

Land this or an equivalent content-bound provenance repair after maintainer/security acceptance of the outbound-staged HTML trust lifecycle.

## Reproduction Assessment

Yes, at source level: current main stages the trusted temp-root HTML into media/outbound and later reloads that staged path through a host-read gate that only trusts temp-root realpaths. The reporter also provided production Telegram logs, but I did not rerun live Telegram in this read-only review.

## Solution Assessment

Yes, this is the best reviewed fix shape if maintainers accept the lifecycle: it preserves arbitrary HTML rejection while carrying content-bound provenance through the staging boundary that drops the original trust signal. A blanket outbound-dir trust root would be broader, and the previous sidecar idea would violate the repository's SQLite-state policy.
