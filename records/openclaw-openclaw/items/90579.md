## Summary

Keep open: this is the right general repair shape for the staged generated-HTML regression, but the current patch still has blocking security-boundary and delivery-fallback defects, and the supplied Telegram recording does not prove the host-read-enabled failing path.

## What This Changes

The PR carries trusted generated HTML provenance through outbound media staging by adding a shared SQLite provenance table, marking staged files, checking that marker during host-read reloads, pruning stale rows, and adding media regression coverage.

## Best Possible Solution

Keep the SQLite provenance approach, but bind the marker to the exact staged artifact or content, fail staging when provenance cannot be recorded, and add host-read-enabled Telegram proof before merge.

## Reproduction Assessment

Yes. The issue logs plus current source show local reply media is staged into media/outbound and Telegram later reloads that staged path through the host-read HTML gate, which current main trusts only under the temp root.

## Solution Assessment

No, not as submitted. The SQLite provenance direction is the right owner-boundary fix, but the marker must be bound to the exact staged artifact and marker write failures must not be hidden before this is safe to merge.

Full review comments:

- **[P1] Bind the marker to the staged artifact:** `src/media/web-media.ts:389-401`
  - body: The provenance row records only `realpath`, `kind`, `version`, and time, and the reader accepts by `kind`/`version`. If the file at that path is replaced before the second host-read load, the replacement inherits trust that belonged to the original generated report. Store and verify an artifact identity/content token, or reject on mismatch, and add a replacement-file regression test.
  - confidence: 0.87
- **[P1] Fail staging when the trust marker cannot be recorded:** `src/media/outbound-attachment.ts:40-47`
  - body: This catches every `markTrustedGeneratedHtmlPath` error and still returns the staged `.html` path. `createReplyMediaPathNormalizer` only adds the user-visible media-failure fallback when staging throws, so a DB/schema/permission failure here returns an untrusted path that the later Telegram `loadWebMedia` rejects, recreating the dropped attachment. Let marker write failures fail staging, or drop/warn before returning the path.
  - confidence: 0.91
