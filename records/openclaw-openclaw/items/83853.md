## Label Justifications

- `P1`: A swallowed Twitch auth setup failure can leave the channel with a broken client and block real chat workflows.
- `impact:message-loss`: The broken auth setup can prevent Twitch chat connection or outbound replies from succeeding.
- `impact:auth-provider`: The failure is in Twurple `RefreshingAuthProvider` token binding for the Twitch plugin.

## Summary

Keep open: current `main` partially fixed the related Twitch chat-intent problem, but this issue's failure-propagation bug remains. `addUserForToken` can still reject, get logged, and then be swallowed before `createAuthProvider` returns a `RefreshingAuthProvider`.

## What This Changes

The issue asks OpenClaw to stop swallowing Twitch `addUserForToken` failures so refresh-auth setup rejects instead of returning an unbound auth provider.

## Best Possible Solution

Make Twitch refresh-auth setup atomic: preserve the existing `['chat']` intent registration, rethrow `addUserForToken` failures after logging, and only create/cache a `ChatClient` after token binding succeeds.

## Reproduction Assessment

Yes from source inspection: configure a Twitch account with `clientSecret` and make `addUserForToken` reject; current `main` logs the rejection and still returns the provider. I did not run a live Twitch repro because this review is read-only.

## Solution Assessment

Yes; rethrowing after the existing log, or rewriting this block as `try/catch` that rethrows, is the narrow maintainable fix. The related chat-intent fix should be preserved, not replaced.
