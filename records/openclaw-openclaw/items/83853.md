## Summary

This should stay open: current main and the latest stable release still catch and log a failing Twitch addUserForToken call without rethrowing, then return the RefreshingAuthProvider as if setup succeeded. The failure mode is source-proven, narrow, and distinct from the already-linked chat-intent issue.

## What This Changes

The issue asks OpenClaw to propagate Twitch addUserForToken failures instead of returning a RefreshingAuthProvider without a bound user.

## Best Possible Solution

Make Twitch provider setup fail fast when addUserForToken rejects, keep the fix inside the Twitch plugin, and add regression coverage that getClient rejects without caching a ChatClient.

## Reproduction Assessment

Yes, by source inspection: a rejecting RefreshingAuthProvider.addUserForToken call is caught and not rethrown, after which createAuthProvider returns the provider to getClient. I did not run a live invalid-token Twitch repro in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to rethrow, or rewrite this block as try/catch that logs and propagates the setup failure before getClient constructs and caches ChatClient.
