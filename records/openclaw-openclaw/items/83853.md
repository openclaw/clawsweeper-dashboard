## Summary

Keep open: current main and the latest release still log and suppress a rejecting `addUserForToken` call, then continue constructing and caching a Twitch `ChatClient`. The related chat-intent bug at https://github.com/openclaw/openclaw/issues/83727 was fixed by https://github.com/openclaw/openclaw/pull/83750, but that PR did not make refresh-auth setup fail atomically when user binding rejects.

## What This Changes

The issue asks for Twitch refresh-auth setup to propagate `addUserForToken` failures instead of returning a `RefreshingAuthProvider` without a bound user.

## Best Possible Solution

Make Twitch refresh-auth setup atomic: keep the chat intent registration, log and rethrow `addUserForToken` failures, and only create or cache a `ChatClient` after auth binding succeeds.

## Reproduction Assessment

Yes at source level: make `addUserForToken` reject while `clientSecret` is present, and current `createAuthProvider` will log the error but still return the provider. I did not run a live Twitch credential repro in this read-only review.

## Solution Assessment

Yes; rethrowing after logging, with a regression test that the client is not created or cached, is the narrow maintainable fix. The related chat-intent repair is already on main and should be preserved, not reopened here.
