## Summary

Keep open: current main still logs and suppresses the `addUserForToken` rejection, then continues with a `RefreshingAuthProvider` that may not have a bound user. The related Twitch chat-intent report at https://github.com/openclaw/openclaw/issues/83727 is useful context on the same call site, but it is not a duplicate of this failure-propagation bug.

## What This Changes

The issue requests that Twitch `RefreshingAuthProvider.addUserForToken` failures propagate instead of returning a broken auth provider and cached chat client.

## Best Possible Solution

Make the Twitch refresh-auth setup atomic: register the token with the `chat` intent, log and rethrow setup failures, and only create or cache a `ChatClient` after auth binding succeeds.

## Reproduction Assessment

Yes from source inspection: make `addUserForToken` reject while `clientSecret` is present, and current `getClient` will still proceed past the logging-only catch. I did not run the repro because this review was explicitly read-only.

## Solution Assessment

Yes; propagating the setup failure is the narrow fix for this issue. Because the same call also lacks Twurple's documented `chat` intent, the cleanest small PR should cover both the rejection path and intent registration with focused tests.
