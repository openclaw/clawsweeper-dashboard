## Summary

Keep open: this is a useful candidate fix for the Nextcloud Talk non-message webhook 400 bug, but it is not merge-ready because the PR still routes valid reaction-added hooks to 400 and the supplied proof does not exercise the real signed webhook path.

## What This Changes

The PR splits Nextcloud Talk webhook JSON parsing from schema validation so signed valid-JSON non-Note events return 200/no dispatch while malformed JSON and malformed Note-like payloads still return 400, with replay tests for those cases.

## Best Possible Solution

Keep the JSON-before-schema split, classify supported message activities by top-level ActivityStreams type, return 200/no dispatch for known non-message hooks, preserve 400 for malformed supported messages, and require real signed webhook-path proof.

## Reproduction Assessment

Yes, source-level. Current main and v2026.6.8 parse signed webhook bodies through the strict Note schema and write HTTP 400 for schema misses before any ignore branch; I did not run a live Nextcloud server in this read-only review.

## Solution Assessment

No, not yet. JSON-before-schema is the right layer, but object.type alone is too broad for malformed-message detection because official reaction hooks embed Note objects inside non-message activities.
