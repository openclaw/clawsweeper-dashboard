## Summary

Keep open. Current main has separate observability surfaces for status, sessions, model auth, channels, logs, and services, but it still lacks the requested unified `openclaw fleet status` command or equivalent JSON payload spanning fleet-wide agents, auth, channel health, stuck-session state, and recent-error aggregation.

## What This Changes

Request a new `openclaw fleet status` command with human and JSON output for unified multi-agent observability across agents, providers/auth, channels, stuck sessions, and recent errors.

## Best Possible Solution

Define one supported read-only fleet observability schema that reuses the existing status, sessions, models/auth, channels, logs, and service collectors with clear redaction guarantees and documented JSON fields. The final command should avoid duplicating parallel health logic and should include focused tests for human output, JSON output, and sensitive-value handling.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction, but yes for feature absence: inspecting the current command catalog, docs, status schema, JSON payload, and separate models/channels/sessions surfaces shows no unified fleet command or equivalent payload.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the product/API surface. The requested direction is coherent, but the maintainable fix should first settle command naming, schema boundaries, reuse of existing collectors, and redaction policy.
