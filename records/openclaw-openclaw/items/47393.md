## Summary

Keep open: current main still lacks the requested cross-account Discord owner guard, but this PR is not mergeable as-is because it is stale against the current Discord plugin boundary and introduces blocking routing/data-scope defects.

## What This Changes

The PR adds Discord route-owner resolution, explicit mention and bot-name bypass plumbing, a bot identity fallback, and focused Discord monitor tests.

## Best Possible Solution

Refresh or replace the branch with a current-main Discord plugin implementation that ranks only runtime-active accounts, applies owner suppression to guild/channel traffic, uses Discord mention/reply metadata for alternate targeting, preserves plugin SDK boundaries, handles wildcard routing, and adds a changelog entry plus regression coverage.

## Reproduction Assessment

Yes for the reported current behavior by source inspection: current main routes each Discord monitor through its own account route and has no cross-account owner guard. I did not run live Discord, but the two-account guild-channel path and the fetchUser failure path are clear in the source.

## Solution Assessment

No. The direction is useful, but this patch is not the best mergeable solution because it can block valid DMs, choose inactive owners, bypass plugin boundaries, miss current wildcard routing, and use plain bot-name text as an owner-bypass signal.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Exclude DMs from owner blocking:** `extensions/discord/src/monitor/message-handler.preflight.ts:437-452`
  - body: The new owner check runs for DMs, but bypassOwnerBecauseMentioned is always false for DMs. If another account has a more specific direct-peer binding for the same sender, this bot silently drops a valid DM even though the other account is not handling this DM event.
  - confidence: 0.92
- **[P2] Restrict owners to active accounts:** `extensions/discord/src/monitor/route-owner.ts:35-40`
  - body: resolveDiscordRouteOwner considers account IDs that appear only in bindings, including disabled, removed, or duplicate-token accounts. A stale binding can win ownership and make active monitors suppress messages for an account that is not running.
  - confidence: 0.88
- **[P2] Use plugin SDK routing imports:** `extensions/discord/src/monitor/route-owner.ts:1-8`
  - body: This new Discord plugin production module imports config and routing internals through ../../../../src/**. Bundled plugin code should use openclaw/plugin-sdk/* or local Discord exports, so this bypasses the supported plugin boundary.
  - confidence: 0.9
- **[P2] Handle wildcard peer priority:** `extensions/discord/src/monitor/route-owner.ts:11-19`
  - body: Current main's route resolver can return matchedBy: "binding.peer.wildcard", but this priority table has no entry for it. After rebasing, wildcard owner routes will fail the type contract or compare with an undefined priority.
  - confidence: 0.84
- **[P2] Require Discord mention metadata for bypass:** `extensions/discord/src/monitor/message-handler.preflight.ts:98-100`
  - body: buildBotNameMentionRegexes makes @ optional and is merged into the owner-bypass mention set. Ordinary text containing the bot username can therefore bypass channel ownership even though the related routing policy requires Discord mention metadata to select an alternate bot.
  - confidence: 0.78
