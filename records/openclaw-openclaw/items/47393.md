## Summary

This PR should stay open for maintainer review: it targets a real Discord multi-bot routing problem, but the patch still has blocking message-delivery, session-state, compatibility, and plugin-boundary issues, and it lacks inspectable live behavior proof.

## What This Changes

The branch adds Discord route-owner resolution across accounts, explicit mention bypass logic, bot username propagation, an application-id fallback for Discord bot identity, and focused Discord monitor tests.

## Best Possible Solution

Consolidate this under the canonical Discord routing work with a Discord-owned target-selection step that maps native Discord mention and reply metadata to active configured accounts before default-owner routing, excludes DMs from owner suppression unless explicitly designed, and stays on plugin SDK seams.

## Reproduction Assessment

Yes, at source level. Current main lacks cross-account owner selection and only compares Discord mention/reply metadata to the running botUserId; I did not run a live Discord reproduction in this read-only review.

## Solution Assessment

No. The PR targets the right problem class, but the current implementation should not merge before it excludes DMs, filters inactive accounts, avoids plain-text bot-name targeting, restores plugin-boundary compliance, and proves the upgrade path in live Discord.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Exclude DMs from owner suppression:** `extensions/discord/src/monitor/message-handler.preflight.ts:437-452`
  - body: The new owner check can run for direct messages, but `bypassOwnerBecauseMentioned` is always false for DMs. If another account has a more specific direct-peer binding for the same sender, this handler drops a legitimate DM as owned by that account even though the other account does not receive this DM event.
  - confidence: 0.9
- **[P1] Filter owner candidates to active Discord accounts:** `extensions/discord/src/monitor/route-owner.ts:36-44`
  - body: `listDiscordCandidateAccountIds` includes account IDs from bindings even if those accounts are no longer configured or running. A stale binding can win ownership and make active accounts drop channel traffic with no responder left.
  - confidence: 0.88
- **[P1] Use native Discord targets for owner bypass:** `extensions/discord/src/monitor/message-handler.preflight.ts:80-99`
  - body: The bypass treats a bot-name text regex as an explicit target, but the canonical Discord routing contract requires native mention or reply metadata and says plain text names must not count as mentions. This can let ordinary text bypass owner suppression and route to the wrong bot.
  - confidence: 0.86
- **[P2] Keep Discord plugin code on SDK seams:** `extensions/discord/src/monitor/route-owner.ts:1-7`
  - body: This new production module imports config and routing internals from `../../../../src/**`, which violates the extension boundary and will drift from the plugin SDK/local Discord seams used on current main.
  - confidence: 0.9
