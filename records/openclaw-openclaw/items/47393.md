## Summary

Keep open. Current main still lacks the cross-account Discord owner selection and bot-identity fallback that this PR is trying to add, so it is not implemented already; however the branch is not mergeable as-is because the added owner guard can drop DMs, can choose inactive accounts, bypasses the plugin boundary, is stale against wildcard routing, and treats plain bot-name text as an owner-bypass signal.

## What This Changes

The PR adds Discord cross-account route-owner resolution, explicit mention/name-based owner bypass, bot username preflight plumbing, an application-id fallback for bot user ID, and focused Discord monitor tests.

## Best Possible Solution

Refresh or replace the branch with a current-main Discord-plugin implementation that ranks only runtime-active Discord accounts, suppresses non-owner guild/channel traffic by default, preserves explicit Discord metadata mention and reply-target precedence, excludes DMs from owner blocking, handles wildcard route precedence, keeps imports behind plugin SDK/local Discord barrels, and adds a changelog entry plus focused regression coverage.

## Reproduction Assessment

Yes. The PR body provides a concrete two-account guild-channel reproduction, and current main confirms the missing cross-account owner guard plus fail-closed bot identity lookup; the PR-introduced regressions are reproducible by static inspection of the added guard and route-owner helper.

## Solution Assessment

No. The direction is useful, but this patch is not the best mergeable fix because it can block valid DMs, choose disabled or stale accounts as owners, bypass extension import boundaries, miss current wildcard routing, and allow plain bot-name text to route non-owner channel content.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Exclude DMs from owner blocking:** `extensions/discord/src/monitor/message-handler.preflight.ts:437-452`
  - body: The owner check runs for direct messages, but bypassOwnerBecauseMentioned is always false for DMs. If another account has a more specific direct-peer binding for the same sender, this bot can silently drop a valid DM even though the other bot is not handling this DM event.
  - confidence: 0.92
- **[P2] Restrict owners to active accounts:** `extensions/discord/src/monitor/route-owner.ts:35-40`
  - body: resolveDiscordRouteOwner considers account IDs that appear only in bindings, including disabled, removed, or duplicate-token accounts. A stale binding can win ownership and make active monitors suppress messages for an account that is not running.
  - confidence: 0.88
- **[P2] Import routing through the plugin SDK:** `extensions/discord/src/monitor/route-owner.ts:1-8`
  - body: This new Discord plugin production module imports config and routing internals through ../../../../src/**. Bundled plugin code should use openclaw/plugin-sdk/* or local Discord exports, so this bypasses the supported plugin boundary.
  - confidence: 0.9
- **[P2] Handle wildcard peer priority:** `extensions/discord/src/monitor/route-owner.ts:11-19`
  - body: Current main's route resolver can return matchedBy: "binding.peer.wildcard", but this priority table has no entry for it. After rebasing, wildcard owner routes will fail the type contract or compare with an undefined priority.
  - confidence: 0.84
- **[P2] Require real Discord mentions for bypass:** `extensions/discord/src/monitor/message-handler.preflight.ts:98-100`
  - body: buildBotNameMentionRegexes makes @ optional and is merged into the owner-bypass mention set. Ordinary text containing the bot username can therefore bypass channel ownership even though the related routing policy requires Discord mention metadata to select an alternate bot.
  - confidence: 0.78
