## Summary

Keep open: the Discord routing problem is still relevant, but this PR is not mergeable as-is because it can suppress valid DMs, can select inactive owners, violates current plugin SDK boundaries, misses current wildcard routing, and broadens owner bypass through plain bot-name text.

## What This Changes

The PR adds Discord route-owner resolution, explicit mention/name bypass handling, a bot identity fallback, and focused Discord monitor tests.

## Best Possible Solution

Refresh or replace the branch with a current-main Discord plugin implementation that ranks only active accounts, applies owner suppression only where another account can actually receive the event, uses Discord mention/reply metadata for alternate targeting, preserves SDK boundaries, handles wildcard routes, and adds changelog plus regression coverage.

## Reproduction Assessment

Yes, by source inspection: current main routes each Discord monitor through its own account route and has no cross-account owner guard, while the PR diff shows the DM, inactive-owner, wildcard, and plain-name bypass paths. I did not run live Discord or tests in this read-only pass.

## Solution Assessment

No. The direction is useful, but this patch is not the best mergeable solution until it is rebased onto the current plugin SDK surface and narrowed to Discord metadata-based targeting with active-account ownership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Exclude DMs from owner blocking:** `extensions/discord/src/monitor/message-handler.preflight.ts:437-452`
  - body: The owner check runs for DMs, but bypassOwnerBecauseMentioned is always false for DMs. If another account has a more specific direct-peer binding for the same sender, this bot can silently drop a valid DM even though that other account is not handling this DM event.
  - confidence: 0.92
- **[P2] Restrict owners to active accounts:** `extensions/discord/src/monitor/route-owner.ts:35-40`
  - body: resolveDiscordRouteOwner considers account IDs that appear only in bindings, including disabled, removed, or otherwise non-running accounts. A stale binding can win ownership and make active monitors suppress messages for a bot that is not available to respond.
  - confidence: 0.88
- **[P2] Use plugin SDK routing imports:** `extensions/discord/src/monitor/route-owner.ts:1-8`
  - body: This new Discord plugin production module imports config and routing internals through ../../../../src/**. Bundled plugin code should use openclaw/plugin-sdk/* or local Discord exports, so this bypasses the supported plugin boundary.
  - confidence: 0.9
- **[P2] Handle wildcard peer priority:** `extensions/discord/src/monitor/route-owner.ts:11-19`
  - body: Current main's route resolver can return matchedBy: "binding.peer.wildcard", but this priority table has no entry for it. After rebasing, wildcard owner routes will either fail the type contract or compare with an undefined priority.
  - confidence: 0.84
- **[P2] Require Discord mention metadata for owner bypass:** `extensions/discord/src/monitor/message-handler.preflight.ts:98-100`
  - body: buildBotNameMentionRegexes makes @ optional and the result is merged into owner-bypass mention matching. Ordinary text containing the bot username can therefore bypass channel ownership even though the related routing policy requires Discord mention metadata to select an alternate bot.
  - confidence: 0.78
