## Summary

Keep open: the PR targets a real Discord multi-account routing gap, but the current branch is not mergeable because it can suppress valid DMs, lets inactive/stale accounts win ownership, bypasses the plugin boundary, misses current wildcard routing, and introduces a data-scope concern around plain-name bypasses.

## What This Changes

The PR adds Discord route-owner resolution with explicit mention and bot-name bypass handling, threads bot usernames through preflight, preserves a bot id fallback during startup identity failures, and adds focused Discord monitor tests.

## Best Possible Solution

Refresh or replace the branch with a current-main Discord plugin implementation that ranks only runtime-active accounts, applies owner suppression to guild/channel traffic, uses Discord mention/reply metadata for alternate targeting, preserves plugin SDK boundaries, handles wildcard routing, and adds changelog plus regression coverage.

## Reproduction Assessment

Yes, by source inspection. Current main routes Discord messages for the running account only and has no cross-account owner guard, while the PR diff clearly shows the new owner check and the introduced DM, candidate-account, wildcard, and bypass paths.

## Solution Assessment

No. The direction is useful, but this patch is not the best mergeable solution because it can block valid DMs, choose inactive owners, bypass plugin boundaries, miss wildcard routing, and treat plain bot-name text as an owner-bypass signal.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Exclude DMs from owner blocking:** `extensions/discord/src/monitor/message-handler.preflight.ts:437-452`
  - body: The new owner check runs when there is no thread/configured binding, including DMs, but bypassOwnerBecauseMentioned is always false for direct messages. If another account has a more specific direct-peer binding for the same sender, this bot drops a valid DM even though the other account is not receiving this DM event.
  - confidence: 0.92
- **[P2] Restrict owners to active accounts:** `extensions/discord/src/monitor/route-owner.ts:35-40`
  - body: resolveDiscordRouteOwner includes account IDs that appear only in bindings. A disabled, removed, or stale bound account can win ownership and make active monitors suppress a channel that no running account will handle.
  - confidence: 0.88
- **[P2] Use plugin SDK routing imports:** `extensions/discord/src/monitor/route-owner.ts:1-8`
  - body: This new Discord plugin production module imports config, routing, and session helpers through ../../../../src/**. Bundled plugin code is expected to use openclaw/plugin-sdk/* or local barrels, so this bypasses the supported plugin boundary.
  - confidence: 0.9
- **[P2] Handle wildcard peer priority:** `extensions/discord/src/monitor/route-owner.ts:11-19`
  - body: Current main's route resolver can return matchedBy: "binding.peer.wildcard", but this priority table has no entry for it. After rebasing, owner comparison either fails the type contract or compares against an undefined priority.
  - confidence: 0.84
- **[P2] Require Discord mention metadata for bypass:** `extensions/discord/src/monitor/message-handler.preflight.ts:98-100`
  - body: buildBotNameMentionRegexes makes @ optional and merges that regex into the owner-bypass path. Ordinary text containing the bot username can therefore make a non-owner account process channel content even though the related routing policy requires Discord mention metadata.
  - confidence: 0.78
