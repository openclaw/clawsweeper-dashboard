## Summary

Keep open: this PR addresses real Discord multi-bot routing gaps, but it is not mergeable as-is because the diff can drop DMs, stale bindings can suppress active accounts, proof is not inspectable, and canonical routing issues remain open.

## What This Changes

The branch adds Discord route-owner resolution, explicit alternate-bot mention bypass, bot-name mention fallback, botUserId fallback behavior, and focused Discord monitor tests.

## Best Possible Solution

Fold the useful owner-resolution and identity-fallback pieces into one Discord-owned routing repair that selects targeted bot accounts from native mention/reference metadata before default routing and proves suppression with focused tests.

## Reproduction Assessment

Yes, at source level. Current main still routes before cross-bot target metadata and only compares mentions/replies to the running bot ID, while the PR diff exposes the DM and stale-binding blockers directly.

## Solution Assessment

No, not as-is. The direction is useful, but the patch should exclude DMs from owner blocking, ignore inactive accounts, rebase onto current provider.startup identity seams, and align with the canonical mention/reply/default-owner policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Exclude direct messages from owner blocking:** `extensions/discord/src/monitor/message-handler.preflight.ts:442`
  - body: The new owner check runs for direct messages because this guard only checks thread/configured bindings, while `bypassOwnerBecauseMentioned` is always false for DMs. If another account has a more specific direct-peer binding for the sender, this handler drops a DM event that the other account will not receive.
  - confidence: 0.91
- **[P2] Ignore inactive accounts when choosing owners:** `extensions/discord/src/monitor/route-owner.ts:36`
  - body: The resolver includes account IDs found only in bindings, even when that Discord account is no longer configured or running. A stale binding for a removed account can therefore win ownership and cause active accounts to drop the message as owned by a nonexistent handler.
  - confidence: 0.87
