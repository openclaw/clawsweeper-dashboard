## Summary

Keep open: the PR addresses a real Discord routing gap that current main still lacks, but the branch is conflicting, lacks inspectable real Discord proof, and introduces blocking message-delivery and compatibility defects around DMs, stale owner candidates, and text-name bypasses.

## What This Changes

The PR adds Discord route-owner resolution across accounts, explicit cross-bot mention bypass, botUserName propagation, botUserId fallback on identity lookup failure, and focused monitor/provider tests.

## Best Possible Solution

Consolidate this under https://github.com/openclaw/openclaw/issues/13487 with a Discord-owned target-selection step that uses native mention and reply metadata, filters to active configured accounts, excludes DMs from owner suppression, stays on plugin SDK seams, and ships with live Discord proof.

Yes, at source level. Current main routes before cross-bot target selection and only compares native mention/reply metadata to the running botUserId; I did not run a live Discord route in this read-only review.

## Solution Assessment

No. The maintainable fix is a consolidated Discord target-selection design using native metadata and active configured accounts; this PR is partial, conflicting, and introduces suppression paths that need rework before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Exclude DMs from owner suppression:** `extensions/discord/src/monitor/message-handler.preflight.ts:438`
  - body: The new owner check runs even for direct messages, while `bypassOwnerBecauseMentioned` is always false for DMs. If another account has a more specific direct-peer binding for the same sender, this handler will drop a legitimate DM as owned by another account even though that other account is not receiving this DM channel event.
  - confidence: 0.93
- **[P1] Filter owner candidates to active accounts:** `extensions/discord/src/monitor/route-owner.ts:35-39`
  - body: `resolveDiscordRouteOwner` considers account IDs referenced only by bindings, including stale or disabled accounts. A removed account with an old specific binding can win owner resolution and make every active account drop the channel traffic, leaving no responder.
  - confidence: 0.9
- **[P1] Use native Discord targets for owner bypass:** `extensions/discord/src/monitor/message-handler.preflight.ts:85-100`
  - body: The bot-name regex treats mutable plain text as an explicit target and bypasses the owner guard without requiring a real Discord mention or reply target. The canonical routing issue explicitly calls out native mentions/reply metadata and says plain text names must not count as mentions.
  - confidence: 0.89
- **[P2] Import routing through plugin SDK seams:** `extensions/discord/src/monitor/route-owner.ts:1-7`
  - body: This new Discord extension module imports core internals through `../../../../src/**`, which violates the extension boundary now used on current main. Rebase this onto `openclaw/plugin-sdk/routing` and `openclaw/plugin-sdk/config-contracts` before it can fit the current plugin architecture.
  - confidence: 0.87
