## What This Changes

The branch adds Discord route-owner resolution, explicit/native and bot-name mention bypasses, a bot-user-id fallback on identity lookup failure, and focused Discord monitor tests.

## Best Possible Solution

Land one coordinated Discord-owned routing repair that derives target bot eligibility from Discord mention/reference metadata before default routing, suppresses non-target default owners, keeps plugin code on SDK seams, ports the identity fallback into current startup code, and includes focused tests plus redacted live Discord proof.

Yes, at source level. Current main resolves Discord routing before cross-bot target metadata and compares explicit mentions/replies only with the running handler's bot id; I did not run a live Discord setup in this read-only review.

## Solution Assessment

No, not as-is. The direction is useful, but this PR needs DM-safe owner blocking, active-account filtering, SDK-boundary imports, Discord-metadata-only bypass semantics, a rebase onto current startup code, and real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Exclude direct messages from owner blocking:** `extensions/discord/src/monitor/message-handler.preflight.ts:442`
  - body: The new owner check runs for direct messages because it only checks thread/configured binding state, while `bypassOwnerBecauseMentioned` is always false for DMs. If another account has a more specific direct-peer binding for the same sender, this handler drops a legitimate DM even though the other account will not receive this DM event.
  - confidence: 0.91
- **[P2] Filter owner candidates to active accounts:** `extensions/discord/src/monitor/route-owner.ts:35-40`
  - body: `resolveDiscordRouteOwner` includes account ids found only in bindings. A stale binding for a removed or disabled Discord account can win ownership and make active accounts drop the channel message for a handler that is not running.
  - confidence: 0.88
- **[P2] Import routing through plugin SDK seams:** `extensions/discord/src/monitor/route-owner.ts:1-8`
  - body: This new Discord production module imports config, routing, and session-key internals with `../../../../src/**` paths. Extension production code should use `openclaw/plugin-sdk/*` or a documented local/public seam so bundled plugins do not depend on private core layout.
  - confidence: 0.86
- **[P2] Keep owner bypass tied to Discord target metadata:** `extensions/discord/src/monitor/message-handler.preflight.ts:85-100`
  - body: The bot-name regex treats plain text like `@Bot Name` as a mention even when Discord did not resolve a user mention. That can let a non-owner account bypass the owner guard from incidental text, while the canonical Discord routing issue calls for native mention and reply metadata to drive cross-bot targeting.
  - confidence: 0.78
