## Summary

Keep open. Current main still lacks the requested Discord multi-account owner suppression and startup identity fallback, but this PR is not mergeable as-is because the branch is stale against the current plugin boundary and routing contract, and it introduces owner-bypass regressions.

## What This Changes

The PR adds Discord cross-account route-owner resolution, explicit mention/name-based owner bypass, bot username preflight plumbing, an application-id fallback for bot user ID, and focused Discord monitor tests.

## Best Possible Solution

Refresh or replace the branch with a Discord-plugin-owned implementation on current main that ranks only active Discord accounts, suppresses non-owner guild/channel traffic by default, preserves explicit Discord metadata mention and reply-target precedence, excludes DMs from owner blocking, handles wildcard route precedence, and keeps imports behind plugin SDK or local Discord exports.

## Reproduction Assessment

Yes. The PR body gives a concrete two-account Discord guild-channel reproduction, and current main confirms the missing cross-account owner-resolution path plus fail-closed identity lookup. The PR-introduced regressions are reproducible with direct-peer DM bindings, stale/disabled accounts, wildcard route bindings, and plain bot-name text.

## Solution Assessment

No. The direction is useful, but this patch is not the best mergeable fix because it can block valid DMs, choose inactive accounts as owners, bypass plugin boundaries, miss wildcard route precedence, and treat plain bot-name text as an explicit owner bypass.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Exclude DMs from owner blocking:** `extensions/discord/src/monitor/message-handler.preflight.ts:437`
  - body: The new owner check runs for direct messages while bypassOwnerBecauseMentioned is always false for DMs. If another account has a more specific direct-peer binding for the same sender, this bot can silently drop a valid DM even though the other bot is not handling this DM event.
  - confidence: 0.92
- **[P2] Restrict owners to active accounts:** `extensions/discord/src/monitor/route-owner.ts:35-40`
  - body: resolveDiscordRouteOwner considers account IDs that appear only in bindings, including disabled, removed, or duplicate-token accounts. A stale binding can win ownership and make active monitors suppress messages for an account that is not running.
  - confidence: 0.88
- **[P2] Import routing through the plugin SDK:** `extensions/discord/src/monitor/route-owner.ts:1-8`
  - body: The new Discord plugin production module imports config and routing internals through ../../../../src/**. Bundled plugin code should use openclaw/plugin-sdk/* or local Discord exports, so this bypasses the supported plugin boundary.
  - confidence: 0.9
- **[P2] Handle wildcard peer priority:** `extensions/discord/src/monitor/route-owner.ts:11-19`
  - body: Current main's route resolver can return matchedBy: "binding.peer.wildcard", but the new priority table has no entry for it. After rebasing, wildcard owner routes will fail the type contract or compare with an undefined priority.
  - confidence: 0.84
- **[P2] Require real Discord mentions for bypass:** `extensions/discord/src/monitor/message-handler.preflight.ts:98-100`
  - body: buildBotNameMentionRegexes makes @ optional and merges that regex into the owner-bypass mention set. Ordinary text containing the bot username can therefore bypass channel ownership even though the related routing policy requires Discord mention metadata, not plain names, to select an alternate bot.
  - confidence: 0.78
