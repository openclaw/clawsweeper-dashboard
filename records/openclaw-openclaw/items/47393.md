## Summary

Keep open, but this PR is not merge-ready: current main still lacks deterministic Discord cross-bot target routing, while the branch is dirty against main, lacks inspectable live proof, and introduces concrete DM, stale-account, and text-name routing hazards.

## What This Changes

The PR adds Discord cross-account route-owner resolution, explicit owner bypass for bot mentions and bot-name text matches, bot identity fallback behavior, and focused Discord monitor tests.

## Best Possible Solution

Consolidate this under https://github.com/openclaw/openclaw/issues/13487 with a Discord-owned target-selection step that maps configured bot IDs to accounts, uses native mention/reply metadata before default routing, excludes DMs and inactive accounts from owner suppression, stays on plugin SDK seams, and ships with live Discord proof.

## Reproduction Assessment

Yes, at source level. Current main still routes before cross-bot target selection and only compares mention/reply metadata to the running botUserId, while PR-head defects are visible directly in the changed source.

## Solution Assessment

No. The maintainable solution is a consolidated Discord target-selection design using native metadata and active configured accounts; this PR is partial, conflicting, and introduces suppression paths that need rework before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Exclude DMs from owner suppression:** `extensions/discord/src/monitor/message-handler.preflight.ts:442-452`
  - body: The owner check runs whenever there is no thread/configured binding, including DMs, but `bypassOwnerBecauseMentioned` is forced false for DMs. If another account has a more specific direct-peer binding, this handler drops the DM as owned by that account even though the user has no mention bypass path and the other account may never receive this DM event.
  - confidence: 0.9
- **[P1] Filter owner candidates to active accounts:** `extensions/discord/src/monitor/route-owner.ts:35-41`
  - body: `listDiscordCandidateAccountIds` includes accounts discovered only from bindings, so a stale binding for an account that is no longer configured or running can win ownership. Active accounts would then drop the message as owned by a nonexistent account, leaving the channel with no responder.
  - confidence: 0.86
- **[P1] Use native Discord targets instead of bot-name text:** `extensions/discord/src/monitor/message-handler.preflight.ts:85-100`
  - body: Adding `botUserName` regexes makes plain text such as an `@name` string count as an explicit owner bypass and normal mention. The canonical routing issue explicitly requires native `mentions[]` and reply metadata because text-name matching can target the wrong bot and conflict with configured mention patterns.
  - confidence: 0.82
- **[P2] Move route-owner imports onto plugin SDK seams:** `extensions/discord/src/monitor/route-owner.ts:1-8`
  - body: The new production module imports config and routing internals through `../../../../src/**`, but `extensions/AGENTS.md` requires bundled plugin production code to use `openclaw/plugin-sdk/*` or local barrels. Reintroducing core-relative imports conflicts with the current extension boundary and import guards.
  - confidence: 0.84
