## Summary

Keep open: this PR targets a real Discord multi-bot routing gap, but the branch is conflicting with current main, lacks inspectable real behavior proof, and introduces owner-suppression paths that can drop DMs, stale-account traffic, or plain-name targets. The remaining work should be consolidated with https://github.com/openclaw/openclaw/issues/13487 rather than merged as-is.

## What This Changes

Adds Discord cross-account route-owner resolution, explicit mention bypass for alternate bot accounts, bot-name text matching, a bot identity fallback, and focused Discord monitor tests.

## Best Possible Solution

Build one Discord-owned target-selection step that maps native mention and reply bot IDs to configured accounts before default routing, excludes DMs and inactive accounts from owner suppression, stays on plugin SDK seams, and ships with redacted live Discord proof.

## Reproduction Assessment

Yes, at source level. Current main routes before cross-bot mention/reference target selection and only compares metadata to the running botUserId; I did not run a live Discord route in this read-only review.

## Solution Assessment

No. The direction is useful, but the maintainable fix is a consolidated metadata-driven Discord target-selection step rather than this partial owner-suppression path with text-name targeting and stale-account/DM gaps.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Exclude DMs from owner suppression:** `extensions/discord/src/monitor/message-handler.preflight.ts:442-452`
  - body: The owner check runs for direct messages, but `bypassOwnerBecauseMentioned` is always false for DMs. If another account has a more specific direct-peer binding for the same sender, this handler drops the DM as owned by that account even though the other bot will not receive this DM event.
  - confidence: 0.92
- **[P1] Filter owner candidates to active accounts:** `extensions/discord/src/monitor/route-owner.ts:35-41`
  - body: `resolveDiscordRouteOwner` considers every account ID referenced by bindings, including accounts removed from `channels.discord.accounts`. A stale binding can therefore win ownership and make the active account drop channel traffic with no responder.
  - confidence: 0.9
- **[P1] Use native Discord targets instead of bot-name text:** `extensions/discord/src/monitor/message-handler.preflight.ts:85-100`
  - body: The new bot-name regex makes plain text like a bot display name act as an explicit cross-bot target. The canonical Discord routing issue requires native mentions and reply metadata, so this can route to the wrong bot from ordinary prose or quoted text.
  - confidence: 0.88
- **[P2] Handle wildcard route priority after rebase:** `extensions/discord/src/monitor/route-owner.ts:11-20`
  - body: Current main's route resolver can return `matchedBy: "binding.peer.wildcard"`, but this new priority table has no entry for it. Add an explicit priority before merging so owner resolution typechecks and preserves the intended specificity order.
  - confidence: 0.86
- **[P2] Keep owner resolution on plugin SDK seams:** `extensions/discord/src/monitor/route-owner.ts:1-8`
  - body: This new production extension file imports core internals through `../../../../src/...`, which violates the extension boundary and conflicts with current main's plugin SDK routing exports. Use `openclaw/plugin-sdk/routing` and local Discord seams instead.
  - confidence: 0.89
