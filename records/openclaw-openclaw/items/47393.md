## Summary

Keep open: current main still lacks cross-account Discord route-owner suppression, but this PR is not mergeable as-is because the owner guard can drop DMs, stale account bindings can suppress active bots, the new module crosses the plugin boundary, wildcard routes are not handled, and plain-name bypass creates a Discord data-scope risk.

## What This Changes

The PR adds Discord route-owner resolution and tests, explicit mention/name bypass handling, and a Discord bot identity fallback during monitor startup.

## Best Possible Solution

Refresh the Discord plugin implementation around active-account ownership and Discord metadata-based targeting, keep plugin code on SDK seams, add wildcard/DM coverage, and align the final behavior with the canonical routing policy in #13487.

## Reproduction Assessment

Yes. Source inspection shows current main routes each Discord monitor through its own account route without a cross-account owner guard, and the PR body gives a concrete two-account/channel-owner reproduction path.

## Solution Assessment

No. The direction is useful, but this patch is not the best mergeable solution until it uses active account candidates, real Discord mention/reply metadata, SDK routing imports, and current wildcard route semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Exclude DMs from owner blocking:** `extensions/discord/src/monitor/message-handler.preflight.ts:437-452`
  - body: The new owner check runs for DMs, but `bypassOwnerBecauseMentioned` is always false for DMs. If another account has a more specific direct-peer binding for the same sender, this bot can silently drop a valid DM with no mention bypass path.
  - confidence: 0.92
- **[P2] Restrict owners to active accounts:** `extensions/discord/src/monitor/route-owner.ts:35-40`
  - body: `resolveDiscordRouteOwner` considers account IDs from bindings, including stale, disabled, or removed accounts. A stale binding can win ownership and make active monitors suppress messages for a bot that is not available to respond.
  - confidence: 0.88
- **[P2] Use plugin SDK routing imports:** `extensions/discord/src/monitor/route-owner.ts:1-8`
  - body: This new Discord plugin production module imports config and routing internals through `../../../../src/**`. Bundled plugin code should use `openclaw/plugin-sdk/*` or local exports so it does not bypass the supported plugin boundary.
  - confidence: 0.9
- **[P2] Handle wildcard peer priority:** `extensions/discord/src/monitor/route-owner.ts:11-19`
  - body: Current main's route resolver can return `matchedBy: "binding.peer.wildcard"`, but this priority table has no entry for it. After rebasing, wildcard owner routes will either fail the type contract or compare with an undefined priority.
  - confidence: 0.84
- **[P2] Require Discord mention metadata for owner bypass:** `extensions/discord/src/monitor/message-handler.preflight.ts:98-100`
  - body: `buildBotNameMentionRegexes` makes `@` optional and merges the result into owner-bypass mention matching. Ordinary text containing the bot username can therefore bypass channel ownership even though the related routing policy requires Discord mention metadata.
  - confidence: 0.78
