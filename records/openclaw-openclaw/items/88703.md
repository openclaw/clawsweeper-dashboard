## Label Justifications

- `impact:session-state`: The failure is about session transcript ownership and prompt-lock fence state during embedded runs.
- `impact:message-loss`: The user-visible symptom is that a generated-media completion cannot be delivered to the requesting session.

## Summary

Current main and the latest release already address the reported session-fence failure: OpenClaw now publishes same-process transcript writes across embedded lock controllers and accepts those fingerprints when the original run reacquires the prompt lock.

## What This Changes

The issue reports that image-generation completion after `sessions_yield` can trigger `EmbeddedAttemptSessionTakeoverError` because a fallback announce run writes to the same session file while the original run's prompt lock is released.

## Best Possible Solution

Keep the v2026.5.28/current-main owned-write fence behavior; open a fresh report only if image-generation completion still throws this takeover error on v2026.5.28 or newer with logs.

## Reproduction Assessment

No current-main failing reproduction was established. Source and regression-test evidence show the reported 2026.5.27 fence failure is covered by the current and v2026.5.28 owned-write implementation.

## Solution Assessment

Yes. The implemented fix is the maintainable shape because it accepts locked in-process announce writes while preserving takeover detection for unowned external edits.
