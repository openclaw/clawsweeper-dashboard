## Summary

Keep open: the PR is useful and has terminal proof for the registered outbound adapter, but it still drops the new projection on the normal Telegram channel-message delivery path that core uses before outbound adapters.

## What This Changes

The PR adds `delivery.providerAccepted` to outbound delivery results and marks successful Telegram standalone and registered outbound text/media/payload sends as provider-accepted, with tests, API hash, and changelog updates.

## Best Possible Solution

Carry provider acceptance through the Telegram message adapter result used by normal outbound delivery, add regression coverage for that path, and land only after maintainer approval of the public SDK semantics.

## Reproduction Assessment

Yes, for the review finding: source inspection shows `deliverOutboundPayloads` prefers Telegram `message.send.*`, and those sends pass through `toTelegramMessageSendResult`, which strips the new projection. This is not a current-main bug reproduction; the PR adds a new public delivery projection.

## Solution Assessment

No: adding the projection to standalone and registered outbound adapters is necessary, but incomplete while the normal channel-message path drops it. The narrow fix is to preserve the projection through `ChannelMessageSendResult` or the Telegram message-send conversion path.
