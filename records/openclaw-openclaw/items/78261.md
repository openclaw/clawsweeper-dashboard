## Summary

Keep this PR open: the direction may be useful, but the branch does not wire the receipt contract into the registered Telegram plugin outbound path, and it still needs public SDK baseline/changelog follow-through plus stronger runtime proof.

## What This Changes

The PR adds outbound receipt request/result types, advertises Telegram visible-receipt capability, returns delivery confirmation from `telegramOutbound.sendText` when requested, and adds adapter unit coverage.

## Best Possible Solution

Land a maintainer-approved, baseline-backed receipt contract on the registered Telegram plugin path, with coverage for `telegramPlugin.outbound` or `loadChannelOutboundAdapter` and contributor proof of the corrected runtime behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this PR adds a new Telegram/Plugin SDK receipt capability. Current main source shows the receipt fields and delivery projection are absent from the existing outbound contract.

## Solution Assessment

No; the requested direction is plausible, but this patch updates a standalone adapter path while the registered Telegram plugin still ignores the receipt request fields. The narrower maintainable solution is to make the registered plugin adapter advertise and honor the contract, then update SDK artifacts and changelog.
