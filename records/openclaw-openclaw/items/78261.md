## What This Changes

This PR adds a shared `delivery.providerAccepted` projection to outbound and channel-message send results, marks successful Telegram sends, preserves it through the outbound bridge, and updates tests, docs, API baseline, and changelog.

## Best Possible Solution

Land the additive provider-accepted delivery projection if maintainers accept the SDK shape, keeping it separate from receipts, read state, and user-visibility guarantees.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this PR adds an additive Telegram and Plugin SDK result contract. Source inspection shows current main lacks the projection, and the PR body/diff show the after-fix adapter and bridge behavior.

## Solution Assessment

Yes, pending maintainer SDK acceptance: an optional delivery projection set only after provider send success and copied through the bridge is narrower than overloading `MessageReceipt`. The patch also documents that fallback, dry-run, failure, read, and visibility outcomes must not use this field.
