## Summary

Keep open: the latest head still has a source-reproducible correctness issue where a local Telegram payload fallback is marked as provider-accepted even though no provider send occurred.

## What This Changes

The branch adds `delivery.providerAccepted` to outbound and channel-message send results, preserves it through the outbound bridge, annotates Telegram text/media/payload sends, and updates SDK docs, tests, changelog, and the API baseline.

## Best Possible Solution

Keep the provider-acceptance projection, but only set it on results produced by successful provider I/O and then route the additive SDK contract through maintainer API review.

## Reproduction Assessment

Yes. Source inspection shows the shared payload helper can return a fallback without calling Telegram send, while PR head still wraps that returned value with `delivery.providerAccepted: true`.

## Solution Assessment

No, not quite yet. The overall direction is narrow, but payload provider acceptance should be gated on actual Telegram provider I/O or left unset for local fallback results.
