## Summary

Keep open: the patch looks functionally coherent and addresses the earlier review findings, but it adds a public Plugin SDK/result contract and still needs maintainer acceptance plus exact-head checks to finish.

## What This Changes

This PR adds an optional `delivery.providerAccepted` projection to outbound and channel-message send results, annotates successful Telegram sends, preserves the projection through the outbound bridge, and updates tests, docs, changelog, and SDK baseline hashes.

## Best Possible Solution

Land the optional provider-accepted projection if maintainers accept the SDK shape, keeping it separate from receipts, read state, and user-visibility guarantees.

## Reproduction Assessment

Not applicable as a strict bug reproduction; this PR adds an additive Telegram and Plugin SDK result contract. Source inspection shows current main lacks the projection, and the PR body plus diff show the after-fix adapter and bridge behavior.

## Solution Assessment

Yes, pending maintainer SDK acceptance: an optional delivery projection set only after provider send success and copied through the bridge is narrower than overloading `MessageReceipt`. The patch also documents that fallback, dry-run, failure, read, and visibility outcomes must not use this field.
