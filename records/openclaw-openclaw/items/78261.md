## Summary

Keep open: the PR is a focused and mostly well-covered SDK/Telegram capability, but the payload path can still stamp `delivery.providerAccepted` on a helper fallback that did not perform provider I/O.

## What This Changes

The PR adds an optional `delivery.providerAccepted` projection to outbound and channel-message send results, preserves it through the outbound bridge, annotates Telegram sends, and updates docs, tests, changelog, and the Plugin SDK API hash.

## Best Possible Solution

Keep the provider-acceptance projection direction, but gate payload annotations on actual Telegram send results before maintainer API review and merge.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this PR adds a new public result projection. The review finding is source-reproducible from the shared payload helper returning a fallback result without provider I/O while the PR annotates that result unconditionally.

## Solution Assessment

No, not quite yet. The general approach is narrow and well-scoped, but payload provider acceptance should be applied only inside paths that actually call Telegram send or after filtering out no-send fallbacks.
