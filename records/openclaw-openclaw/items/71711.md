## Summary

Keep open: current main and the latest release still lack native expiration for recurring cron jobs, and this is a public cron API/persistence/scheduler feature that needs maintainer product direction rather than conservative cleanup.

## What This Changes

The issue requests an optional recurring-cron expiration field, expire action, CLI flag, list display, scheduler enforcement, and audit behavior for jobs created with `--every` or `--cron`.

## Best Possible Solution

Define one canonical recurring-expiration contract across cron protocol/schema, SQLite persistence, CLI/list output, scheduler enforcement, audit/run-history semantics, docs, and regression tests before implementation.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high-confidence absence proof that current main has no recurring cron expiration field, CLI flag, schema, storage column, scheduler gate, or docs.

## Solution Assessment

Unclear until maintainer product review. Core cron is the plausible owner, but field naming, timezone handling, permanent/nullable jobs, remove-vs-disable default, and interaction with the agent scheduling RFC need a decision first.
