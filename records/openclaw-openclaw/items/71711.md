## Summary

Keep open: current main and v2026.6.10 still lack a recurring-job expiration field or expire action, and this remains the narrower canonical cron lifecycle request rather than cleanup-ready stale work.

## What This Changes

Request to add an optional `activeUntil`/`expiresAt` timestamp and remove-or-disable expiration policy for recurring cron jobs.

## Best Possible Solution

Design one additive recurring-job expiration contract across cron types, SQLite projection, Gateway schema, CLI add/edit/list/show, scheduler expiration handling, audit/run history, docs, and focused tests, coordinated with https://github.com/openclaw/openclaw/issues/71712.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source inspection gives high confidence that current main lacks the proposed expiration fields, CLI flag, scheduler handling, and persisted storage.

## Solution Assessment

Unclear as proposed. `activeUntil` is a plausible direction, but maintainers should first settle naming, default remove-versus-disable behavior, permanent/null semantics, audit records, and interaction with agent-facing scheduling.
