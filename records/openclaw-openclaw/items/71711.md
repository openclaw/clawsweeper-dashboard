## Summary

Keep open. Current main still lacks recurring cron expiration fields, CLI/UI/docs support, and scheduler enforcement; the linked agent-scheduling RFC keeps the final public contract and security policy in maintainer-design territory.

## What This Changes

Add optional activeUntil/expiresAt and onExpire lifetime controls for recurring cron jobs so they stop, remove, or disable after a configured timestamp.

## Best Possible Solution

Define an additive recurring-job expiration contract in core cron, then thread it through persisted schema, scheduler enforcement, Gateway protocol, CLI, Control UI, docs, plugin scheduling contracts, and audit semantics.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence that current main lacks the requested recurring expiration field and enforcement path across the persisted model, Gateway validation, scheduler, CLI/UI, plugin scheduling contract, tests, and docs.

## Solution Assessment

Unclear until maintainers choose the public contract. The capability belongs in core scheduler state, but the exact fields, defaults, timezone behavior, audit event, and interaction with https://github.com/openclaw/openclaw/issues/71712 need design review first.
