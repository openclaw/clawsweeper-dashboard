## Summary

Keep open. Current main still has no recurring cron expiration contract, and the requested behavior belongs in the core scheduler and persisted cron model rather than a plugin because it affects job storage, Gateway RPC, CLI, agent tool, Control UI, and docs.

## What This Changes

Add an optional `activeUntil` or `expiresAt` lifetime and remove-or-disable expiration behavior for recurring `--every` and `--cron` jobs.

## Best Possible Solution

Define a backward-compatible recurring-job lifecycle contract in the core cron model, validate it only for recurring schedules, enforce expiration before due execution and during maintenance/startup recomputation, persist the chosen remove/disable result, emit an audit/log event, and expose the same semantics through Gateway RPC, CLI add/edit/list/show, the agent cron tool, Control UI, and public docs with focused scheduler/protocol/CLI/UI tests.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The absence is high-confidence from source inspection: current schemas and user surfaces accept no expiration field, and recurring jobs rearm by computing future `nextRunAtMs` values after runs.

## Solution Assessment

Unclear pending maintainer API choice. The direction belongs in core because cron lifecycle is scheduler-owned persisted behavior, but the exact public contract should be settled before implementation.
