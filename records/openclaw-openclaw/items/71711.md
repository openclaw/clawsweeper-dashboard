## Summary

Keep open. Current main still lacks a recurring cron expiration contract, and the request is concrete core scheduler/API work rather than a plugin-only idea. Because it changes the persisted cron job shape plus Gateway RPC, CLI, agent-tool, UI, docs, and the open agent scheduling RFC, maintainers should choose the public contract before implementation starts.

## What This Changes

The issue requests an optional `activeUntil` or `expiresAt` field for recurring `--every` and `--cron` jobs, with configurable remove-or-disable behavior after expiration.

## Best Possible Solution

Define a backward-compatible recurring-job lifecycle contract in the core cron model, validate it only for recurring schedules, enforce expiry before due execution and during maintenance/startup recomputation, persist the remove/disable result, emit/audit the expiration, and expose the same semantics through Gateway RPC, CLI add/edit/list/show, agent tools, Control UI, and public docs with focused scheduler/protocol/CLI/UI tests.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The absence is high-confidence because current main has no accepted cron expiration field or CLI/UI/tool surface, and enabled recurring jobs continue by computing future `nextRunAtMs` values.

## Solution Assessment

Unclear pending maintainer API choice. The direction belongs in core because cron job lifecycle is owned by the scheduler and persisted job contract, but maintainers should settle naming, default action, audit semantics, and #71712 alignment before a fix PR starts.
