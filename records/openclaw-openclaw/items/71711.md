## Summary

Keep open. Current main still lacks a recurring cron expiration contract, and this is a recent concrete core cron feature request rather than a duplicate, stale report, or plugin-only idea. The request changes persisted job schema, scheduler lifecycle, Gateway RPC, CLI, agent-tool, Control UI, docs, and related agent-scheduling semantics, so maintainer API/product review should happen before an automated repair lane implements it.

## What This Changes

Add an optional `activeUntil` or `expiresAt` lifecycle field for recurring `--every` and `--cron` jobs, with configurable remove/disable behavior after expiration.

## Best Possible Solution

Define a backward-compatible recurring-job lifecycle contract in core cron, validate it only for recurring schedules, enforce expiration before due execution and during scheduler maintenance, persist the remove/disable outcome, emit auditable lifecycle events, and expose the same semantics through Gateway RPC, CLI add/edit/list/show, the agent cron tool, Control UI, and public docs with focused regression coverage.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The absence is high-confidence from current-main inspection: the exact field names are absent, persisted types and closed Gateway schemas do not accept them, and recurring scheduler paths re-arm jobs without an expiration gate.

## Solution Assessment

Unclear pending maintainer API choice. The requested direction belongs in core because recurring job lifecycle is scheduler-owned persisted behavior, but the public field names, defaults, parsing, audit semantics, and #71712 interaction need maintainer review before implementation.
