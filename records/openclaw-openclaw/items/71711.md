## What This Changes

The issue requests an optional per-job expiration for recurring cron jobs, with remove-or-disable behavior after the configured timestamp passes.

## Best Possible Solution

Have maintainers decide the recurring-job lifecycle contract, then implement it consistently across the cron job schema/store, Gateway protocol, CLI/UI creation and edit flows, scheduler expiration handling, audit/run history, docs, and focused regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives a high-confidence absence check because current cron types, protocol schemas, CLI options, UI creation flow, docs, and scheduler execution paths do not include recurring expiration.

## Solution Assessment

Unclear until maintainer product review. The requested direction is plausible for the core cron scheduler, but the durable API shape and its relationship to the linked agent-scheduling RFC should be decided before implementation.
