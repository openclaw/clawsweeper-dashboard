## Summary

Keep open. Current main still lacks any recurring cron expiration contract, and the requested behavior belongs in the core scheduler/persisted cron model rather than a plugin. Because this would change public job shape, Gateway RPC validation, CLI/agent-tool/UI surfaces, docs, and the linked agent scheduling RFC, maintainers should settle the API contract before implementation.

## What This Changes

Add an optional expiration timestamp and expiration action for recurring cron jobs so `--every` and `--cron` schedules can stop or be removed automatically after a configured time.

## Best Possible Solution

Define a backward-compatible recurring-job lifecycle contract in the core cron model, validate it only for recurring schedules, enforce expiration before due execution and during maintenance/startup recomputation, persist the remove/disable result, emit the chosen audit/log event, and expose the same semantics through Gateway RPC, CLI add/edit/list/show, the agent cron tool, Control UI, and public docs with focused scheduler/protocol/CLI/UI tests.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The absence is high-confidence because current main has no accepted expiration field or surface, and recurring jobs continue by computing future `nextRunAtMs` values after runs.

## Solution Assessment

Unclear pending maintainer API choice. The direction belongs in core because cron job lifecycle is owned by the scheduler and persisted job contract, but maintainers should choose naming, default action, audit semantics, and #71712 alignment first.
