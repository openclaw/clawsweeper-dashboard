## Summary

Keep open: current main and v2026.5.19 still lack recurring cron expiration across the persisted job model, Gateway protocol, CLI/UI creation surfaces, scheduler enforcement, docs, and audit behavior; the related scheduling RFC keeps the public contract in maintainer product-design territory.

## What This Changes

Request an optional per-recurring-job lifetime field and CLI/list/audit behavior so `every` and `cron` jobs can expire, remove themselves, or disable themselves.

## Best Possible Solution

Define an additive recurring-job lifetime contract in core cron, then thread it through persisted state, scheduler enforcement, Gateway protocol, CLI, Control UI, docs, tests, and audit semantics alongside the scheduling RFC.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence the current main job model, protocol, CLI, scheduler, UI controller, and docs do not provide `activeUntil` or `expiresAt`.

## Solution Assessment

Unclear until maintainers choose the public contract. The capability likely belongs in core cron state, but naming, defaults, timezone behavior, audit semantics, and interaction with https://github.com/openclaw/openclaw/issues/71712 need design review first.
