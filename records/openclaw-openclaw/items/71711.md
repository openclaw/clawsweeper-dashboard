## Summary

Keep open. Current main still has no recurring cron expiration contract, and this is a recent, concrete feature request tied to core scheduler and persisted cron behavior rather than an obsolete, duplicate, or plugin-only item.

## What This Changes

Request adding optional `activeUntil`/`expiresAt` and expiration handling for recurring `--every`/`--cron` jobs, including CLI/listing support, persisted job schema changes, scheduler enforcement, and audit behavior.

## Best Possible Solution

Define a backward-compatible recurring-job lifecycle contract in the core cron model, validate it only for recurring schedules, enforce expiration before due execution and during scheduler maintenance, persist the configured remove/disable outcome, and expose the same semantics through Gateway RPC, CLI add/edit/list/show, the agent cron tool, Control UI, and public docs with focused coverage.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The absence is high-confidence from source inspection: current-main types, schemas, scheduler rearming, CLI, docs, and UI do not expose or enforce a recurring-job expiration field.

## Solution Assessment

Unclear pending maintainer API choice. The requested direction belongs in core because it changes persisted cron and scheduler-owned lifecycle behavior, but maintainers should decide naming, parsing, defaults, and audit semantics before implementation.
