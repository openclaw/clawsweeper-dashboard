## Summary

Keep open. Current main still lacks a recurring cron expiration field or enforcement path, and the linked agent-scheduling RFC makes the API/security contract a maintainer decision rather than a cleanup close.

## What This Changes

Add optional expiration semantics for recurring `--every` and `--cron` jobs, including CLI/schema/listing support and configurable remove-or-disable behavior after expiry.

## Best Possible Solution

Define a backward-compatible recurring-job expiration contract in core cron, then expose the same semantics through persisted schema, scheduler enforcement, Gateway RPC, CLI, agent tool, Control UI, docs, audit/task history, and focused regression tests.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The absence is high-confidence from source/docs inspection: current persisted models, Gateway validators, scheduler gates, CLI/UI forms, and docs do not include a recurring expiration contract.

## Solution Assessment

Unclear until maintainers choose the public contract. The direction belongs in core because recurring job lifecycle is scheduler-owned persisted behavior, but the exact field names, defaults, timezone parsing, audit semantics, and #71712 interaction need review first.
