## Summary

Keep open: current main and the latest release still lack recurring cron expiration in the job model, Gateway contract, CLI/UI creation surfaces, scheduler enforcement, docs, and audit behavior. The linked scheduling RFC keeps the naming, lifetime policy, timezone handling, and provenance interaction in maintainer product/API territory.

## What This Changes

Add an optional expiration timestamp and expiration action for recurring OpenClaw cron jobs created with `--every` or `--cron`.

## Best Possible Solution

Define an additive recurring-job lifetime contract in core cron, then thread it through persisted state, scheduler enforcement, Gateway protocol, CLI, Control UI, docs, tests, and audit semantics alongside the scheduling RFC.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence that current main and v2026.5.19 do not provide `activeUntil`, `expiresAt`, `onExpire`, or recurring expiration behavior.

## Solution Assessment

Unclear until maintainers choose the public contract. The capability likely belongs in core cron state, but naming, defaults, timezone behavior, audit semantics, and interaction with https://github.com/openclaw/openclaw/issues/71712 need design review first.
