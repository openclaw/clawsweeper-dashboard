## Summary

Keep open. Current main still lacks a recurring cron expiration contract, and this is a recent concrete core scheduler feature request rather than a stale, duplicate, or plugin-only item. The change would affect persisted job schema, scheduler lifecycle, Gateway protocol, CLI, agent tool, Control UI, docs, and the linked agent-scheduling RFC, so it needs maintainer contract review before implementation.

## What This Changes

Request optional expiration and expiration-action fields for recurring `openclaw cron` jobs so `--every` and `--cron` schedules stop after a configured deadline.

## Best Possible Solution

Define a backward-compatible recurring-job lifecycle contract in core cron, validate it only for recurring schedules, enforce expiration before dispatch and during maintenance, persist remove/disable outcomes, emit auditable lifecycle events, and expose the same semantics through Gateway RPC, CLI add/edit/list/show, the agent cron tool, Control UI, and public docs with focused regression coverage.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The absence is high-confidence from current-main inspection: the proposed fields are absent, Gateway cron schemas reject extra fields, and recurring scheduler paths re-arm jobs without an expiration gate.

## Solution Assessment

Unclear until maintainers choose the API contract. The direction belongs in core because recurring job lifecycle is scheduler-owned persisted behavior, but field names, defaults, parsing, audit semantics, UI copy, and #71712 interaction should be decided before implementation.
