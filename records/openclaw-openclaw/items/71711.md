## Summary

Keep open. Current main still lacks recurring cron expiration fields, scheduler enforcement, CLI/UI/docs support, and the linked agent-scheduling RFC keeps the final lifetime and security policy in maintainer-design territory.

## What This Changes

The issue requests an optional `activeUntil`/`expiresAt` timestamp and expiration action for recurring cron jobs.

## Best Possible Solution

Define an additive core cron expiration contract, then thread it through persisted state, scheduler enforcement, Gateway protocol, CLI, Control UI, docs, tests, and audit semantics in coordination with the related scheduling RFC.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence that current main lacks the requested recurring expiration field and enforcement path across model, protocol, scheduler, CLI/UI, tests, and docs.

## Solution Assessment

Unclear until maintainers choose the public contract. The capability belongs in core scheduler state, but the exact fields, defaults, timezone behavior, audit event, and interaction with https://github.com/openclaw/openclaw/issues/71712 need design review first.
