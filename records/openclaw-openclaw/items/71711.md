## Summary

Keep open. Current main still does not expose recurring cron expiration in the persisted cron job model, Gateway cron contract, scheduler lifecycle, CLI/UI creation paths, or cron docs, and the linked scheduling RFC keeps the permanent-versus-bounded lifetime contract in maintainer design territory.

## What This Changes

Request adding optional activeUntil/expiresAt and onExpire semantics for recurring cron jobs, plus CLI, list, cleanup, and audit behavior.

## Best Possible Solution

Define an additive recurring-job lifetime contract in core cron, then thread it through persisted state, scheduler enforcement, Gateway protocol, CLI, Control UI, docs, and audit semantics in coordination with the scheduling RFC.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection on current main gives high confidence that the requested field, flag, schema, enforcement, and docs surface do not exist today.

## Solution Assessment

Unclear until maintainers choose the public contract. The capability likely belongs in core scheduler state, but the exact fields, defaults, timezone handling, audit event, and interaction with https://github.com/openclaw/openclaw/issues/71712 need design review first.
