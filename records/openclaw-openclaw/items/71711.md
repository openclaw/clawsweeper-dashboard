## Summary

Keep open: current main and the latest release still lack native recurring cron expiration, and this is a public cron API/persistence decision rather than conservative cleanup. The related agent-scheduling RFC is still open and broader, so it should inform this design rather than supersede it.

## What This Changes

The issue requests optional `activeUntil`/`expiresAt` support for recurring `openclaw cron` jobs, including CLI creation, listing, scheduler expiry handling, and remove-or-disable behavior.

## Best Possible Solution

Keep this open for maintainer product/API review, then implement one canonical recurring-expiration contract across cron protocol schemas, SQLite persistence, CLI and UI create/edit/list surfaces, scheduler expiry enforcement, docs, and focused tests if maintainers accept the feature.

## Solution Assessment

Unclear until maintainer product review. The core cron scheduler is the plausible owner, but the public field name, default expiration action, audit semantics, and relationship to the agent-scheduling RFC should be decided before implementation.
