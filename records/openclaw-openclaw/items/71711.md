## Summary

Keep open. Current main still has no recurring cron expiration contract, and the request is a valid core scheduler/job-schema feature rather than a ClawHub plugin idea. The open related RFC #71712 depends on the same `activeUntil` concept, so maintainers should settle the public API before implementation.

## What This Changes

The issue requests optional expiration metadata for recurring `--every` and `--cron` jobs, with configurable remove-or-disable behavior once the timestamp passes.

## Best Possible Solution

Define a backward-compatible recurring-job lifecycle contract in the core cron model, validate it only for recurring schedules, enforce expiration before due execution and during scheduler maintenance/startup recomputation, persist the chosen remove/disable result, emit the agreed audit/log event, and expose the same semantics through Gateway RPC, CLI add/edit/list/show, the agent cron tool, Control UI, and public docs.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The absence is high-confidence because current main has no accepted expiration field or surface, and recurring jobs continue by recomputing future `nextRunAtMs` values after runs.

## Solution Assessment

Unclear pending maintainer API choice. The direction belongs in core because cron lifecycle and persisted job shape are scheduler-owned, but maintainers should choose naming, default action, audit semantics, timezone behavior, and #71712 alignment before implementation.

## Risks / Open Questions

- The public API choice is unresolved: `activeUntil` vs `expiresAt`, default remove vs disable, timezone handling for offset-less timestamps, and audit event naming all affect persisted data and clients.
- A partial implementation could create divergent behavior across the scheduler, Gateway RPC, CLI, agent cron tool, Control UI, and docs.
- The related open RFC #71712 depends on recurring expiration for agent-created schedules, so this should be aligned with that design rather than implemented as an isolated field addition.
