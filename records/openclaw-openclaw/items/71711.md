## Summary

Keep open: current main still lacks per-job `activeUntil`/`expiresAt` or `onExpire` support for recurring cron jobs. The related scheduling RFC and open `deleteAfterRun` PRs overlap with cron lifecycle policy, but none replaces this narrow request, and the persisted/API/CLI semantics need maintainer product review before implementation.

## What This Changes

Request to add a time-bounded expiration field and CLI/list behavior for recurring `--every` and `--cron` jobs.

## Best Possible Solution

Design one core cron lifecycle contract for time-bounded recurring jobs, including persisted/API/CLI fields, audit/list behavior, validation, docs, and reconciliation with existing `deleteAfterRun` semantics.

## Reproduction Assessment

Not applicable as a bug reproduction, but yes for source-level gap proof: current main has no `activeUntil`/`expiresAt` field in the cron job type, protocol schema, CLI add flags, scheduler next-run path, or docs.

## Solution Assessment

Unclear until maintainers choose the contract. A narrow expiration field is plausible, but it should be designed together with `onExpire`, audit/list behavior, timezone parsing, and `deleteAfterRun`/permanent-job semantics.
