## Summary

Keep open. Current main still lacks a recurring cron expiration field or scheduler enforcement path, and the related RFC at https://github.com/openclaw/openclaw/issues/71712 makes the permanent-vs-expiring and agent-scheduling semantics a maintainer product/security decision.

## What This Changes

The issue requests an optional recurring cron expiration field such as `activeUntil`/`expiresAt`, plus on-expire behavior for scheduler storage, CLI/listing, and audit surfaces.

## Best Possible Solution

Define an additive recurring-job expiration contract in core cron and thread it through persisted schema, scheduler enforcement, Gateway protocol, CLI, agent tool, Control UI, docs, and audit semantics.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Source inspection gives high confidence that current main lacks the requested recurring expiration field and enforcement path across persisted models, Gateway validation, scheduler gates, CLI/UI, and docs.

## Solution Assessment

Unclear until maintainers choose the public contract. The capability belongs in core scheduler state, but the field names, defaults, timezone parsing, audit behavior, and interaction with https://github.com/openclaw/openclaw/issues/71712 need design review first.
