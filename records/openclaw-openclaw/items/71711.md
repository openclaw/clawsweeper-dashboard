## Summary

Keep open. Current main still lacks a recurring cron expiration contract, and the request is a recent, concrete core scheduler feature rather than a stale report, duplicate, or plugin-only idea. Because it changes persisted cron schema, Gateway protocol, CLI, agent-tool, Control UI, docs, and the linked agent-facing scheduling RFC, the next step should be maintainer contract review before implementation.

## What This Changes

Add an optional expiration timestamp and expiration action for recurring `openclaw cron` jobs so `--every` and `--cron` schedules stop running after a configured deadline.

## Best Possible Solution

Define a backward-compatible recurring-job lifecycle contract in core cron, validate it only for recurring schedules, enforce expiration before dispatch and during scheduler maintenance, persist the chosen remove/disable result, emit an auditable lifecycle event, and expose the same semantics through Gateway RPC, CLI add/edit/list/show, the agent cron tool, Control UI, and public docs with focused regression coverage.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The absence is high-confidence from source inspection: the proposed field names are absent, Gateway cron schemas are closed without them, and recurring scheduler paths re-arm jobs without an expiration gate.

## Solution Assessment

Unclear pending maintainer API review. The direction belongs in core because recurring job lifecycle is scheduler-owned persisted behavior, but the exact field names, defaults, parsing, audit semantics, UI copy, and #71712 interaction should be chosen before implementation.
