## Summary

Keep open. Current main still has no recurring cron expiration contract, and this is a concrete core scheduler/persisted-job feature rather than cleanup that can be closed or moved to a plugin; the remaining blocker is maintainer API choice across cron storage, Gateway RPC, CLI, agent tool, UI, docs, and the related scheduling RFC.

## What This Changes

Add optional expiration metadata for recurring cron jobs so `every` and `cron` schedules can stop after a configured timestamp and either remove or disable the job.

## Best Possible Solution

Define a backward-compatible recurring-job lifecycle contract in the core cron model, validate it only for recurring schedules, enforce expiration before execution and during maintenance/startup recomputation, persist the chosen remove-or-disable outcome, emit the agreed audit/log event, and expose the same semantics through Gateway RPC, CLI add/edit/list/show, the agent cron tool, Control UI, and public docs with focused scheduler/protocol/CLI/UI tests.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The absence is high-confidence because current main has no accepted expiration field or surface, and recurring jobs continue by computing future `nextRunAtMs` values after runs.

## Solution Assessment

Unclear pending maintainer API choice. The direction belongs in core because cron lifecycle is owned by the scheduler and persisted job contract, but maintainers should choose naming, default action, audit semantics, and #71712 alignment before implementation.

## Risks / Open Questions

- The public field names, default expiration action, timezone handling, and audit event semantics are not settled, so an automated patch could lock in the wrong persisted schema.
- The open agent-scheduling RFC wants recurring agent-created jobs to require an expiration, which may affect validation and security semantics for this feature.
- Expiration must be enforced consistently before due execution, during startup/maintenance recomputation, and after concurrent manual or scheduled runs to avoid orphaned state or duplicate cleanup.
