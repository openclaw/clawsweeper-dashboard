## Summary

Keep open. Current main still lacks a recurring cron expiration field or enforcement path, and the linked agent-facing scheduling RFC keeps the public API and security semantics in maintainer-design territory rather than conservative cleanup.

## What This Changes

Add optional `activeUntil` or `expiresAt` semantics for recurring `--every` and `--cron` jobs, with configurable removal or disabling after expiration.

## Best Possible Solution

Define a backward-compatible core cron expiration contract, then expose and test it consistently across persisted schema, scheduler enforcement, Gateway RPC, CLI, agent tool, Control UI, docs, audit/run history, and any agent-scheduling provenance policy.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The absence is high-confidence from current source/docs inspection: persisted models, scheduler gates, validators, CLI, agent tool, UI, and docs do not include a recurring expiration contract.

## Solution Assessment

Unclear until maintainers choose the public contract. The direction belongs in core because recurring job lifecycle is scheduler-owned persisted behavior, but the exact semantics and #71712 interaction should be settled before implementation.
