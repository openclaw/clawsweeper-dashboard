## Summary

Keep open. Current main still has no recurring cron expiration contract, and this is a recent concrete core scheduler feature request rather than stale, duplicate, plugin-only, or already implemented work.

## What This Changes

Request an optional activeUntil/expiresAt plus onExpire lifecycle for recurring --every/--cron jobs, with scheduler cleanup, CLI/listing, and audit visibility.

## Best Possible Solution

Define a backward-compatible recurring-job lifecycle contract in core cron, enforce it before dispatch and re-arming, persist remove/disable outcomes, audit expiration, and expose the same semantics through Gateway RPC, CLI, agent tool, Control UI, and docs with focused regression coverage.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The absence is high-confidence from current-main inspection of the persisted job model, scheduler, Gateway schemas, CLI, Control UI, agent tool, and docs.

## Solution Assessment

Unclear until maintainers choose the public contract. The direction belongs in core because recurring job lifecycle is scheduler-owned persisted behavior, but the exact API, defaults, audit semantics, and #71712 interaction need review first.
