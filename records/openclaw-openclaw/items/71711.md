## Summary

Keep open: current main and the latest release still lack a recurring cron `activeUntil`/`expiresAt` field or expiration enforcement, and the requested behavior is a real product/API decision rather than conservative cleanup.

## What This Changes

Add an optional recurring cron expiration field plus CLI/listing/scheduler behavior to remove or disable jobs after a configured timestamp.

## Best Possible Solution

Have maintainers decide the durable recurring-job expiration contract, then implement it consistently across cron persistence, Gateway protocol, CLI/UI creation and edit flows, scheduler expiration handling, audit/run history, docs, and focused regression tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source and release inspection give a high-confidence absence check for recurring cron expiration support.

## Solution Assessment

Unclear until maintainer product review. The core cron scheduler is the plausible owner, but the public field name, default expiration action, audit semantics, and relationship to the agent-scheduling RFC should be decided before implementation.
