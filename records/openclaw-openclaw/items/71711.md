## Summary

Keep open. Current main still has no recurring cron expiration contract or enforcement path, and the linked #71712 RFC makes the agent-scheduling API/security interaction a maintainer design decision rather than cleanup-close material.

## What This Changes

Add optional activeUntil/expiresAt and on-expire handling for recurring cron jobs so every/cron schedules stop, disable, or remove themselves after a configured timestamp.

## Best Possible Solution

Define a backward-compatible core cron expiration contract, then carry it through persisted schema, scheduler enforcement, Gateway RPC, CLI, agent tool, Control UI, docs, audit/task history, and focused regression tests.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The absence is high-confidence from source/docs inspection of the persisted model, scheduler gates, Gateway schemas, CLI/UI forms, agent tool schema, and docs.

## Solution Assessment

Unclear until maintainers choose the public contract. The feature belongs in core cron because recurring job lifecycle is scheduler-owned persisted behavior, but the exact semantics should be aligned with #71712 before implementation.
