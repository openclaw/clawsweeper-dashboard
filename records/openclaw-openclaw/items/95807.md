## What This Changes

The branch changes `applyJobResult` so unresolved transient-error retries for `every` cron jobs clear `nextRunAtMs` with a warning, and adds a focused regression test.

## Best Possible Solution

Land the narrow guard after contributor proof shows the after-fix cron behavior and maintainers accept that unresolved `every` retries should clear rather than continue on backoff.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level yes, real-runtime no: current `main` falls back to `backoffNext` when an `every` retry cannot compute `normalNext`, but I did not establish a live scheduler repro.

## Solution Assessment

Is this the best way to solve the issue? Yes for behavior, with a small code-shape caveat: clearing matches the existing cron unresolved-next-run invariant, though generalizing the helper would be cleaner than adding an inline one-off branch.
