## Summary

Keep open. Current main still lacks Slack per-channel `replyToMode`, and this PR remains a focused implementation candidate, but the proposed patch is not mergeable as-is because the new routing code references `channelConfig` outside its scope.

## What This Changes

The PR adds optional Slack per-channel `replyToMode` config support across types/schema, channel-config resolution, Slack routing/threading context propagation, agent reply filtering, and focused resolver/threading tests.

## Best Possible Solution

Land a small, current-main-compatible Slack implementation that accepts optional per-channel `replyToMode`, resolves exact and wildcard channel overrides before account-level fallback, carries the resolved mode through message and threading contexts, updates public docs/generated config metadata, and preserves existing account-level behavior when the channel field is absent.

## Reproduction Assessment

Yes. Static inspection of the PR diff reproduces the PR-head failure because `channelConfig?.replyToMode` is used without an in-scope binding; static inspection of current main also reproduces the missing feature because the strict schema and resolver omit per-channel `replyToMode`.

## Solution Assessment

No, not as-is. The requested direction is narrow and maintainable, but the patch needs a rebase/current-main routing fix plus config docs/metadata updates before it is the best way to solve the issue.
