## Summary

Keep open. Current main still lacks the requested Slack per-channel `replyToMode`, and the PR is a useful feature port, but the branch is not merge-ready because the routing override is not actually in scope and the public config surface is incomplete.

## What This Changes

This PR adds an optional `replyToMode` field to Slack per-channel config entries and attempts to carry the resolved mode through Slack inbound routing and reply/threading contexts.

## Best Possible Solution

Land a rebased Slack-owned implementation that resolves the per-channel mode once during inbound channel config resolution, passes it through current main's routing and threading context seams, and aligns tests, generated metadata, docs, and changelog.

## Reproduction Assessment

Yes for the PR blocker. Static review is enough: the PR-added `channelConfig?.replyToMode` read is outside any scope that defines `channelConfig`, and current main confirms the requested config sample is still rejected by the strict per-channel Slack schema.

## Solution Assessment

No, not as currently patched. The feature direction is narrow and maintainable, but the safe fix needs to plumb the resolved channel config through current main's `prepare-routing.ts` path and refresh the public config/docs/changelog surfaces.
