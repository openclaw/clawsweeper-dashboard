## Summary

Current main and the latest release still use the literal trusted-proxy comparison, while the open linked PR explicitly references this issue and owns the narrow normalization fix. Under the issue/PR-pair rule this issue should stay open until that PR lands or is closed.

## What This Changes

The issue asks voice-call webhook signature URL reconstruction to treat plain IPv4 and IPv4-mapped IPv6 forms of trusted proxy addresses equivalently.

## Best Possible Solution

Review and land the focused linked PR if it normalizes equivalent proxy address forms without broadening forwarded-header trust.

## Reproduction Assessment

Yes. Source inspection on current main shows a literal `trustedProxyIPs.includes(remoteIP)` comparison, so `trustedProxyIPs: ["127.0.0.1"]` will not trust a socket remote address of `::ffff:127.0.0.1`; the linked PR also provides before/after local runtime proof.

## Solution Assessment

Yes. Normalizing both configured trusted proxy IPs and the socket remote address before the existing trust check is the narrow maintainable fix, and it matches the already-normalized media-stream path without adding a new option.
