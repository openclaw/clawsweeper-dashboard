## What This Changes

The issue reports that voice-call webhook signature URL reconstruction should treat equivalent plain IPv4 and IPv4-mapped IPv6 proxy addresses the same when checking configured trusted proxies.

## Best Possible Solution

Review and land the focused linked PR if it preserves the forwarding-header trust boundary, then close this issue after the PR merges.

## Reproduction Assessment

Yes. Source inspection on current main shows the literal `trustedProxyIPs.includes(remoteIP)` comparison, which will not match `127.0.0.1` against `::ffff:127.0.0.1`; the linked PR also provides focused before/after verifier proof.

## Solution Assessment

Yes. Normalizing both configured trusted proxy IPs and the socket remote address inside the existing reconstruction path is the narrow fix and matches the existing media-stream path without adding a new config surface.
