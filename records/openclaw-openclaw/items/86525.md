## Summary

Keep open: current main and latest release still literal-match the webhook verifier's trusted proxy IP, while an open linked PR already owns the focused fix path.

## What This Changes

The issue reports that voice-call webhook signature verification does not treat plain IPv4 and IPv4-mapped trusted proxy addresses as equivalent.

## Best Possible Solution

Review and land the linked focused PR if it preserves the existing forwarding-header security gates while normalizing equivalent proxy address forms.

## Reproduction Assessment

Yes. Source inspection shows `trustedProxyIPs.includes(remoteIP)` will fail for `trustedProxyIPs: ["127.0.0.1"]` when the socket remote address is `::ffff:127.0.0.1`; I did not run an executable repro because this review was read-only.

## Solution Assessment

Yes. Normalizing both configured trusted proxy IPs and the socket remote address before the existing trust check is the narrow maintainable fix, and it matches the already-normalized media-stream path without adding a new option.
