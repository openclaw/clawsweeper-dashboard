## Label Justifications

- `P1`: The PR addresses a real local-memory crash path that can take down the gateway for Apple Silicon users.
- `merge-risk: 🚨 compatibility`: The default local embedding implementation moves from in-process native execution to a forked sidecar, which can affect upgrades, package layout, and source/dev launch behavior.
- `merge-risk: 🚨 availability`: The patch changes worker failure, fallback activation, and reindex behavior in a path that controls whether memory search remains available after local embedding failures.

## Summary

Keep open for maintainer review: the latest head appears to address the prior worker-argv and fallback blockers and the central local embedding crash-isolation problem, with no new blocking findings found. The remaining question is maintainer acceptance of the default worker-sidecar runtime and fallback upgrade risk.

## What This Changes

The PR routes local GGUF memory embeddings through a bundled worker sidecar by default, adds typed local-worker degradation and fallback handling in memory-core, rejects a proposed local GPU config key, and adds focused memory/config/worker tests.

## Best Possible Solution

Land the worker-isolation fix after memory/runtime owners accept the default sidecar compatibility risk and treat CPU/GPU policy guidance as separate follow-up scope.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No: the exact native ggml-metal abort was not reproduced on current main, but the linked issue logs identify the node-llama-cpp crash path and this branch has focused tests simulating worker exits and fallback behavior.

## Solution Assessment

Is this the best way to solve the issue? Yes, mostly: isolating the native local embedding runtime behind a worker is the narrowest maintainable way to keep a native abort from killing the gateway without adding a new config key; maintainers still need to accept the default behavior change.
