## Summary

Keep open: the report describes a plausible current-main claude-cli session-state bug, and the source still invalidates a stored Claude CLI binding by checking only that exact session id for an on-disk transcript without reconciling to the id Claude actually emitted or wrote.

## What This Changes

The issue asks OpenClaw to stop false `missing-transcript` resets for persistent claude-cli sessions when the stored CLI session binding diverges from the transcript session id on disk.

## Best Possible Solution

Fix the claude-cli runner to persist the actual Claude-emitted transcript session id after live/resume turns, add a constrained reconciliation path for orphaned bindings, and prove it with diverging-id tests plus one real Claude CLI run.

## Reproduction Assessment

No, not as a high-confidence live reproduction in this review. The issue is source-reproducible from the current gate: a stored claude-cli binding whose exact transcript file is absent is invalidated even if a valid related transcript exists under another emitted id.

## Solution Assessment

Yes, the narrow direction is to rebind from Claude's emitted `session_id` after each live/resume turn and only then add a scoped fallback for orphaned bindings. A naive newest-transcript fallback alone would be riskier because it could cross conversations.
