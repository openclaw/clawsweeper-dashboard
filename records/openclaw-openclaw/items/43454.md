## Summary

Keep open. Current main and v2026.6.6 still do not expose the requested workspace/internal `subagent:ended`, turn-complete, or tool-call-threshold hook contract; adjacent plugin hooks and context-engine callbacks exist, but they are not the standard `HOOK.md` internal hook surface requested here.

## What This Changes

Request to add workspace/internal lifecycle hooks for subagent completion, agent turn completion, and tool-call-threshold handling.

## Best Possible Solution

Approve one bounded lifecycle contract: add the missing workspace/internal subagent completion event only with a minimized payload, decide whether turn completion belongs in internal hooks or a typed plugin/context-engine surface, and make any threshold behavior explicitly opt-in with documented timing and failure semantics.

## Solution Assessment

Unclear pending maintainer API design. Bridging subagent completion into internal hooks is plausible, but the broader turn-complete and threshold request needs a security and owner-boundary decision before a fix PR should be queued.

## Risks / Open Questions

- New lifecycle hooks can expose child-run output, transcript snippets, tool metadata, or sensitive session context to trusted workspace code unless the payload contract is deliberately minimized.
- A mid-turn tool-call-threshold hook could stall or perturb the active agent loop if timeout, ordering, and failure semantics are not explicitly designed.
- The request overlaps typed plugin hooks, context-engine callbacks, and diagnostic events, so maintainers need to choose one coherent owner boundary before implementation.
- The threshold phase would likely create a new interpreted config contract, even though per-hook config entries are schema-passthrough today.
