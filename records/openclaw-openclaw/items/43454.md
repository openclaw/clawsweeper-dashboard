## Summary

Current main already supports the central automation use case through the public plugin lifecycle contract: completed turns use `agent_end`, tool-call counting can be composed from `after_tool_call`, and child completion uses `subagent_ended`. Adding parallel workspace events and a threshold-specific core setting would duplicate that supported contract; richer completion-announcement mutation remains separately tracked.

## What This Changes

Requests workspace-visible lifecycle events for subagent completion, completed agent turns, and configurable tool-call thresholds.

## Best Possible Solution

Keep one public plugin lifecycle contract and implement autonomous learning, memory curation, or threshold policy in an external code plugin; use the focused related issue for any distinct requirement to enrich, suppress, or replace subagent completion announcements.

## Reproduction Assessment

Not applicable as a bug: current source directly exposes the lifecycle signals needed for the central automation use case through the plugin API.

## Solution Assessment

No; adding parallel workspace lifecycle events and a core threshold setting would create competing sources of truth. A code plugin can count `after_tool_call` events and react to `agent_end` or `subagent_ended` using the existing supported seam.
