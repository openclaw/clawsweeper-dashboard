## Summary

Keep open. Current main and the latest release still pass Gateway-visible materialized skill paths directly into Docker bind creation, and the open linked PR is still the candidate fix path rather than a merged implementation.

## What This Changes

The issue reports that Docker-outside-of-Docker sandbox bind mounts use Gateway-container paths for materialized skills instead of host-visible paths.

## Best Possible Solution

Land one security-reviewed Docker Gateway/DooD host-path mapping contract for normal and browser sandbox binds, with schema/docs coverage if a new operator setting is chosen and live proof that materialized skills are readable inside the sandbox.

## Reproduction Assessment

Yes, at source level: current main builds materialized skill paths under runtime state and passes them directly as Docker bind sources, while the documented DooD constraint says the host Docker daemon resolves those paths in the host namespace. I did not run a live Gateway-in-Docker reproduction.

## Solution Assessment

Unclear. Host-path translation is plausible, but the best fix needs a maintainer-approved config/operator contract and sandbox bind-source validation before merge.
