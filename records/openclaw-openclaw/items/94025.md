## Summary

Keep open. Current main and v2026.6.8 still build the materialized skill bind source from Gateway-visible state paths and pass it directly to Docker, while two open linked PRs are candidate fixes that still need maintainer/security review.

## What This Changes

The issue reports that Docker-outside-of-Docker sandboxes cannot mount materialized skills because OpenClaw passes Gateway-container state paths to the host Docker daemon.

## Best Possible Solution

Keep this issue open until one security-reviewed Docker host-path mapping contract lands with focused mount tests and live DooD proof for materialized skill readability.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main builds state-backed materialized skill paths and passes them directly as Docker bind sources, while DooD resolves those paths on the host. I did not run a live DooD container repro.

## Solution Assessment

Is this the best way to solve the issue? Unclear: host-path translation is plausible, but the best fix needs maintainer-approved config/operator semantics and sandbox bind-source validation before merge.
