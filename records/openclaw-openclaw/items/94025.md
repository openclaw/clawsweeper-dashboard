## Summary

Keep open. Current main and the latest release still build materialized skill bind mounts from Gateway-visible paths without host-path translation, and two open PRs are already linked as candidate fixes for this issue.

## What This Changes

The issue reports Docker-outside-of-Docker sandbox bind mounts using container-internal OpenClaw state paths instead of host-visible paths for materialized skills.

## Best Possible Solution

Land one reviewed, security-validated host-path mapping contract for Docker Gateway/DooD sandbox binds, with focused mount tests and live DooD proof for materialized skill readability.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main builds materialized skill paths under runtime state and passes them directly as Docker bind sources, while the documented DooD constraint says the host Docker daemon resolves those paths in the host namespace. I did not run a live Gateway-in-Docker reproduction.

## Solution Assessment

Is this the best way to solve the issue? Unclear: host-path translation is plausible, but the permanent fix needs a maintainer-approved config/operator contract and sandbox bind-source validation before merge.
