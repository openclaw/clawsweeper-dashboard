## Summary

Keep open. Current main and the latest release still format Gateway-visible materialized skill and workspace paths directly into Docker bind specs, and the open closing PR is the candidate fix but remains conflicting and needs maintainer/security/product review.

## What This Changes

The issue reports that Gateway-in-Docker DooD deployments pass container-internal materialized skill paths to the host Docker daemon, so sandbox skills can mount empty or fail.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/94057 with one security-reviewed, schema/docs-backed host-path mapping contract for managed workspace and materialized skill mounts, then close this issue after live DooD proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main builds materialized skill paths under runtime state and formats those paths directly as Docker bind sources, while the docs confirm DooD bind paths are resolved in the host namespace.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Host-path translation is plausible, but the durable fix needs a maintainer-approved config/operator contract and sandbox validation before it is safe to land.
