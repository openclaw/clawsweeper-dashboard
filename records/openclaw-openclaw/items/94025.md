## Summary

Keep open. Current main and v2026.6.8 still pass Gateway-visible sandbox skill paths directly to Docker bind creation, and this issue has two open closing PRs that need maintainer/security direction before the issue can close.

## What This Changes

The issue reports that materialized sandbox skills cannot be bind-mounted when the Gateway runs inside Docker with a host Docker daemon because bind sources are resolved in the daemon host namespace.

## Best Possible Solution

Choose one canonical DooD host-path mapping contract, validate rewritten bind sources before Docker receives them, cover normal and browser sandbox mounts, and land one linked PR with live or Crabbox DooD proof.

## Reproduction Assessment

Yes, at source level: materialized skills are created under Gateway-visible state and then passed into Docker bind construction without host-path translation. I did not run a live DooD Gateway reproduction in this read-only review.

## Solution Assessment

Unclear as a final product shape. Host-path translation is the likely repair boundary, but the exact env/config contract and validation rules need maintainer/security approval before a PR should land.
