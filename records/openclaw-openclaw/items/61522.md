## Summary

Keep open. The linked config IO observability problem still has real remaining gaps on current main, but this branch should not merge as-is because the PR head has bootstrap-unsafe logging, misleading `EIO` classification, incomplete backup coverage, and no real behavior proof.

## What This Changes

The PR adds a new config IO error classifier and wires selected config and backup best-effort catch paths to subsystem logging without changing write semantics.

## Best Possible Solution

Repair or replace this branch with a current-main patch that uses bootstrap-safe logging, classifies errno values correctly, logs abnormal backup/audit/health write failures, preserves best-effort semantics, and adds focused tests.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence path: the PR head maps `EIO` to `disk-full` and logs config-health reads through a logger path that can re-enter config loading on the PR base; current main still has injectable silent backup and audit IO failures.

## Solution Assessment

No. The goal is valid, but this branch is not the narrowest maintainable fix as written because it introduces misleading classification and bootstrap-unsafe logging.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Avoid config-dependent logging during config load:** `src/config/io.ts:775`
  - body: This catch can run while config loading is active. Calling `logIoError()` here goes through `configLogger`; on the PR base, file-log settings can fall back to `loadConfig().logging`, so a missing or corrupt health file can recursively re-enter config loading during startup. Use a bootstrap-safe logger or avoid read-path logging here.
  - confidence: 0.9
- **[P2] Classify EIO separately from disk-full:** `src/config/io-errors.ts:27`
  - body: `EIO` is a generic input/output failure, while `ENOSPC` is the no-space condition. Mapping both to `disk-full` sends device, filesystem, or remote-mount failures to the wrong category and severity; let `EIO` fall through or add a separate IO-error category.
  - confidence: 0.94
