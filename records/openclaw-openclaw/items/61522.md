## Summary

Keep this PR open. Current main still has silent best-effort config IO, audit, recovery, and backup-rotation catch paths, so the requested behavior is not already implemented; however, this PR is not ready as-is because the proposed subsystem logging can recurse during config/logging bootstrap in the PR base and the new classifier maps generic EIO failures to disk-full.

## What This Changes

The PR adds a config IO error-classification helper, creates subsystem loggers for config IO and backup rotation, and replaces many best-effort empty catches with level-aware log calls.

## Best Possible Solution

Rework this as a small current-main patch with a non-recursive config IO logging helper, use injected or minimal logging in config bootstrap paths, cover the current audit/recovery/backup modules, and add focused tests for errno classification plus logging-bootstrap no-recursion behavior.

## Reproduction Assessment

Yes. The unsafe path is reproducible by static trace in the PR base: a config file with no `logging` section plus a missing or unreadable `config-health.json` makes `readConfigHealthStateSync` log through `configLogger`, whose settings resolution falls back to `loadConfig()` and can re-enter the same read path.

## Solution Assessment

No. The goal is valid, but the proposed solution is not the best fix because it adds config-dependent subsystem logging inside config reads, misclassifies EIO, and no longer matches current main's refactored config audit/recovery files.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Avoid config-dependent logging in config reads:** `src/config/io.ts:763`
  - body: This catch can run while `loadConfig()` is observing a config snapshot. Calling `logIoError()` here goes through `configLogger`, and the PR base's subsystem logger resolves console/file settings by falling back to `loadConfig()` when no logging config is present. A missing or corrupt `config-health.json` can therefore recursively re-enter config loading during startup; use a non-config-dependent or injected logger for this bootstrap path.
  - confidence: 0.9
- **[P2] Classify EIO separately from disk-full:** `src/config/io-errors.ts:27`
  - body: `EIO` is a generic input/output failure, not an `ENOSPC` disk-full condition. Mapping it to `disk-full` will emit the wrong category and severity for device, filesystem, or remote-mount failures; let it fall through to `unknown` or add a separate `io-error` category.
  - confidence: 0.94
