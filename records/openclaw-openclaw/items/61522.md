## Summary

Keep open. The underlying config IO logging goal remains valid, but current main only partially covers it and this PR branch is not merge-ready: it misclassifies `EIO`, uses config-dependent logging in a config-loading path on its own head, and misses current-main audit/backup surfaces that still swallow relevant IO failures.

## What This Changes

The PR adds a config IO error-classification helper and wires subsystem logging into selected catch blocks in config IO and backup rotation.

## Best Possible Solution

Rework this on current main with a bootstrap-safe IO logging helper, classify `EIO` separately from disk-full, log abnormal backup/audit/health write failures without changing best-effort semantics, and add focused regression tests.

## Reproduction Assessment

Yes. Static review gives a high-confidence reproduction path: the PR head maps an `EIO` errno to `disk-full`, and its config health read catch can log through subsystem settings while config loading is active; current main also visibly retains silent backup and audit append catches.

## Solution Assessment

No. The goal is right, but this branch is not the best fix because it targets an older monolithic config IO layout and introduces misleading/error-prone logging behavior; a small current-main replacement is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Avoid config-dependent logging while loading config:** `src/config/io.ts:775`
  - body: This catch can run while config loading is already in progress. On the PR head, `logIoError()` goes through `configLogger`, whose subsystem path can resolve logging settings by calling back into `loadConfig()` when no logging config is present, so a missing or corrupt health file can recursively re-enter config loading during startup. Use a non-config-dependent or injected logger for this bootstrap path.
  - confidence: 0.9
- **[P2] Classify EIO separately from disk-full:** `src/config/io-errors.ts:27`
  - body: `EIO` is a generic input/output failure, not an `ENOSPC` disk-full condition. Mapping it to `disk-full` gives operators the wrong category and severity for device, filesystem, or remote-mount failures; let it fall through to `unknown` or add a separate `io-error` category.
  - confidence: 0.94
