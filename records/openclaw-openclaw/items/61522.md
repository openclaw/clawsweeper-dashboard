## Summary

Keep this PR open, but it should not merge as-is: the observability goal is still useful, yet the branch introduces bootstrap-unsafe logging and a misleading EIO category while current main only partially addresses the underlying config IO logging gap.

## What This Changes

The PR adds a config IO error-classification helper and routes selected config IO, audit, health-state, and backup-rotation best-effort catches through subsystem logging.

## Best Possible Solution

Rework this on current main with bootstrap-safe logging, accurate IO categories, focused coverage for audit/backup/health paths, and unchanged best-effort semantics.

## Reproduction Assessment

Yes. Static source review gives a high-confidence reproduction path: the PR head maps `EIO` to `disk-full`, and its config health read catch can log through a settings path that falls back to `loadConfig()`.

## Solution Assessment

No. The goal is reasonable, but this implementation is not the safest fix because it uses config-dependent subsystem logging in a bootstrap path and classifies a generic IO errno as disk-full.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Avoid config-dependent logging during config bootstrap:** `src/config/io.ts:775`
  - body: This catch can run while config loading is already in progress. On this branch it calls `logIoError()` with `configLogger`, and the PR-head subsystem logger can resolve file logging by calling back into `loadConfig()`, so a missing or corrupt health file can recursively re-enter config loading during startup. Use a bootstrap-safe logger or injected logger for this path.
  - confidence: 0.9
- **[P2] Classify EIO separately from disk-full:** `src/config/io-errors.ts:27`
  - body: `EIO` is a generic input/output failure, not an `ENOSPC` disk-full condition. Mapping it to `disk-full` gives operators the wrong category and severity for device, filesystem, or remote-mount failures; let it fall through to `unknown` or add a separate `io-error` category.
  - confidence: 0.94
