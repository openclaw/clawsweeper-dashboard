## What This Changes

The PR adds a config IO error classifier and logs selected best-effort failures in config IO and backup rotation without changing write semantics.

## Best Possible Solution

Repair or replace this branch with a current-main patch that uses bootstrap-safe logging, classifies `EIO` correctly, logs abnormal backup/audit/health write failures, preserves best-effort semantics, and adds focused tests.

## Reproduction Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Avoid config-dependent logging while loading config:** `src/config/io.ts:775`
  - body: This catch can run while config loading is active. Calling `logIoError()` here goes through `configLogger`, whose logger settings path can call back into `loadConfig()` when no logging config is cached, so a missing or corrupt health file can recursively re-enter config loading during startup. Use a bootstrap-safe logger or defer this read-path logging.
  - confidence: 0.9
- **[P2] Classify EIO separately from disk-full:** `src/config/io-errors.ts:27`
  - body: `EIO` is a generic input/output failure, not an `ENOSPC` disk-full condition. Mapping it to `disk-full` sends device, filesystem, or remote-mount failures to the wrong category and severity; let it fall through to `unknown` or add a separate IO-error category.
  - confidence: 0.94
