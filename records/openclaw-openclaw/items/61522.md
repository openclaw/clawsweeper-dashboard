Full review comments:

- **[P1] Avoid subsystem logging during config bootstrap:** `src/config/io.ts:775`
  - body: This catch runs inside config loading, but `configLogger` checks file-log enablement, which on this branch calls `readLoggingConfig()` and loads config again. Missing or corrupt `config-health.json` is a normal first-run path, so this can recurse or thrash startup instead of staying best-effort; use a bootstrap-safe logger path here.
  - confidence: 0.86
- **[P2] Do not classify EIO as disk-full:** `src/config/io-errors.ts:27`
  - body: `EIO` is a generic input/output failure, not a no-space condition. Grouping it with `ENOSPC` will send operators toward disk-space remediation and emit the wrong severity for hardware, filesystem, or remote mount errors; let it fall through or give it its own category.
  - confidence: 0.9
- **[P2] Cover the remaining backup failure catches:** `src/config/backup-rotation.ts:29-30`
  - body: The new backup logger only covers rotation and orphan cleanup, while `hardenBackupPermissions()` and the primary `copyFile()` in `maintainConfigBackups()` still swallow failures. That leaves the most important backup creation failure silent even after this PR claims backup IO is observable; log those remaining best-effort failures or narrow the stated scope.
  - confidence: 0.82
