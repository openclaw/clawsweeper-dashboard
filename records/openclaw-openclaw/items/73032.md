## Summary

Keep open. Current main still does not implement xAI for `/voiceclaw/realtime`, so this PR is not obsolete, but the draft branch has blocking xAI Realtime protocol gaps and a missing required changelog entry before it can merge.

## What This Changes

The branch adds xAI as a `/voiceclaw/realtime` provider by extending provider dispatch and types, adding an xAI Realtime WebSocket adapter with tests, and updating gateway and xAI provider docs.

## Best Possible Solution

Land a revised implementation after maintainer design agreement that keeps the existing Gemini path stable, maps xAI voice transcript and setup acknowledgement events correctly, preserves the server-side key boundary, handles parallel tool-call resume semantics, and records the user-facing feature in the changelog.

## Reproduction Assessment

Yes. Focused mocked adapter tests can inject `response.output_audio_transcript.delta`, send `session.created` before `session.updated`, and simulate multiple function calls to observe missing captions, early readiness, and premature `response.create` respectively.

## Solution Assessment

No. Provider-aware dispatch is a plausible direction, but the current PR should not merge until the documented xAI event mapping, setup ordering, parallel tool-call handling, and changelog gaps are fixed or maintainers choose a shared adapter design.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Map audio transcript events:** `src/gateway/voiceclaw-realtime/xai-realtime.ts:397-398`
  - body: xAI's Realtime reference emits assistant voice captions as `response.output_audio_transcript.delta` and `.done`, but this switch only handles text-mode `response.text.delta` / `response.output_text.delta`. In a normal voice session the adapter can forward audio while never emitting assistant transcript events, so clients lose captions and transcript history; route the audio transcript events through the assistant transcript path.
  - confidence: 0.91
- **[P2] Wait for session.updated before setup completion:** `src/gateway/voiceclaw-realtime/xai-realtime.ts:277-282`
  - body: xAI sends `session.created` automatically when the socket opens and `session.updated` after applying this adapter's `session.update`. Resolving setup on `session.created` can make the gateway emit `session.ready` and flush queued work before the configured voice, instructions, audio settings, and tools are acknowledged; treat `session.created` as informational and complete setup on `session.updated`.
  - confidence: 0.87
- **[P2] Defer response.create until all tool results are sent:** `src/gateway/voiceclaw-realtime/xai-realtime.ts:165-169`
  - body: xAI documents that parallel tool calls may arrive together and that clients must send every `function_call_output` before a single `response.create`. This method sends `response.create` after each individual tool result, so the first completed tool can resume generation without the remaining tool outputs.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `docs/gateway/index.md:194-204`
  - body: This is a user-facing feature and docs change, but the PR does not touch `CHANGELOG.md`. Repo policy requires a changelog entry for user-facing changes before landing; add a single-line entry under the active version section.
  - confidence: 0.96
