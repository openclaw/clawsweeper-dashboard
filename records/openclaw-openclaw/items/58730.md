## Summary

Keep open. Current main only partially covers this broad sandbox/permission request, and the recent Codex-runtime Docker canary is still a credible source-level containment mismatch rather than evidence that the issue is resolved.

## What This Changes

The issue requests stronger exec sandbox isolation, layered tool permissions, context/session lifecycle improvements, and hooks, with later discussion narrowing part of the risk to Codex-runtime bash not honoring Docker sandbox containment.

## Best Possible Solution

Keep this open as a security/product design item, split the concrete Codex Docker sandbox enforcement mismatch into a focused tracker or fix, and leave the broader permission model under explicit maintainer architecture review.

## Reproduction Assessment

Yes for the latest concrete mismatch: source inspection shows Codex app-server resolves OpenClaw sandbox state but starts Codex turns with a host-side cwd and Codex's own sandbox policy, not the OpenClaw Docker exec backend. I did not run the live Docker/Codex canary in this read-only review.

## Solution Assessment

No, the linked macOS `sandbox-exec` PR is not the best complete solution. The maintainable path should align Codex-native shell behavior, OpenClaw sandbox explain/list semantics, Docker/SSH/OpenShell backend contracts, and exec approvals before adding another per-call sandbox profile.
