## Summary

Keep open. Current main has optional sandbox backends, exec approvals, compaction, transcripts, and hooks, but it still does not implement the requested exec sandbox profile/per-skill permission contract, and the live thread is paired with open sandbox PRs plus a source-supported Codex containment mismatch.

## What This Changes

Requests OS-level exec sandboxing, a per-tool or per-skill permission model, and related compaction, session resume, and hook improvements inspired by Claude Code.

## Best Possible Solution

Keep this as a security architecture tracker, review or supersede the macOS and Linux exec-sandbox PRs, land or replace the Codex fail-closed/exec-server work, and split the remaining per-skill permission contract into owner-reviewed milestones.

## Reproduction Assessment

Yes for the narrower Codex/Docker mismatch at source level: current main spawns the Codex app-server from the Gateway process and maps active OpenClaw sandbox state into Codex workspaceWrite policy. The broader feature request itself is architecture work rather than a single failing command reproduction.

## Solution Assessment

No, the macOS sandbox-exec proposal alone is not the best complete solution for this issue. The maintainable path is an explicit sandbox and permission contract with Codex runtime containment, fail-closed, or warning behavior chosen by maintainers and proven with upgrade-safe Docker/Testbox evidence.
