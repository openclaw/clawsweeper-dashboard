## Summary

Keep open for maintainer review: the patch appears correct and not obsolete, with sufficient real local HTTP proof, but it intentionally moves Azure Responses onto guarded transport and caps oversized non-OK bodies, which is a compatibility and availability decision maintainers should accept before merge.

## What This Changes

The PR injects `buildGuardedModelFetch` into both Azure Responses SDK client constructors and adds a shared lazy cap for non-OK response bodies in `sanitizeOpenAISdkSseResponse`.

## Best Possible Solution

Land this shape after maintainers explicitly accept the Azure guarded-transport rollout risk, or request a redacted live Azure/APIM smoke proof if that risk needs more evidence.

## Reproduction Assessment

Yes, source-reproducible. Current main returns non-OK bodies unchanged and the OpenAI SDK reads final errors with `response.text()`; this read-only review did not run a failing current-main repro.

## Solution Assessment

Yes. The shared lazy `TransformStream` cap avoids the earlier eager-read retry/cancel regression, and using the resolved Azure base URL is the right boundary for guarded-fetch policy.
