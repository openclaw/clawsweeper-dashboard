## Summary

Keep open: this maintainer-labeled draft is a real Ollama robustness fix attempt, but it still leaves an unreadable descriptor path that can abort request setup and it locks in empty-schema fallback semantics that overlap with sibling hardening drafts.

## What This Changes

The PR wraps Ollama schema entry and array traversal in defensive readers, falls unreadable schema containers back to empty object schemas, and adds a request-construction regression test.

## Best Possible Solution

Land one canonical Ollama provider hardening path that skips unreadable descriptor or schema inputs while preserving healthy tools and finite readable schemas, then retire overlapping drafts.

## Reproduction Assessment

Yes. Current main and v2026.6.9 directly walk Ollama schema records and descriptor fields during request construction, so throwing proxies or accessors are source-reproducible.

## Solution Assessment

No. This is a plausible mitigation, but the best fix should also guard descriptor field reads and should skip unreadable tools instead of advertising empty schemas unless maintainers explicitly choose that fallback contract.
