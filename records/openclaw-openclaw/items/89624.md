## Summary

Keep open: current main still has the Ollama request-construction failure path, and this maintainer draft is useful but not ready because it preserves unreadable tools as empty schemas and overlaps with the broader skip-bad-tool draft.

## What This Changes

The PR adds guarded Ollama tool-schema traversal plus a regression that keeps request construction from erroring when schema containers throw.

## Best Possible Solution

Use one canonical Ollama hardening path that defensively reads tool fields, skips unreadable schemas, preserves healthy sibling tools, and supersedes this narrower draft if the broader approach lands.

## Reproduction Assessment

Yes. Current main is source-reproducible because Ollama request construction directly walks tool schemas and the PR regression uses throwing proxy containers against that path; I did not execute tests because this review is read-only.

## Solution Assessment

No. This is a plausible mitigation, but the safer fix is to skip unreadable tools after defensive field/schema reads rather than advertise unreadable schemas as empty callable tools.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Skip unreadable schemas instead of sending empty tools:** `extensions/ollama/src/stream.ts:849-852`
  - body: When root schema entries cannot be read, this returns `{ type: "object", properties: {} }` and still advertises the tool to Ollama. That can make a malformed tool callable with no real parameter contract; skip the bad tool and preserve healthy siblings instead.
  - confidence: 0.84
- **[P2] Catch revoked schemas before record checks:** `extensions/ollama/src/stream.ts:840`
  - body: The new entry guards run after `isRecord(schema)`, but `Array.isArray` inside that guard can throw for a revoked proxy. Wrap the whole schema normalization path per tool so revoked or hostile schema roots are skipped instead of aborting request setup.
  - confidence: 0.8
- **[P2] Read tool fields defensively before normalizing:** `extensions/ollama/src/stream.ts:1050`
  - body: `extractOllamaTools` still reads `tool.name`, `tool.description`, and `tool.parameters` directly. A throwing `parameters` accessor can still escape before the new schema-entry guards run, so read tool fields through a safe helper and skip only that bad tool.
  - confidence: 0.88
