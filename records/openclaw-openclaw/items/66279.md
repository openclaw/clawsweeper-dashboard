## Summary

Keep open: current main still has the TUI-specific local TLS trust gap, because the TUI resolves a TLS-enabled local gateway URL but never carries a `tlsFingerprint` into `GatewayClient` while status/probe paths already do.

## What This Changes

The issue reports that `openclaw tui` fails against a self-signed local WSS gateway unless Node TLS verification is disabled.

## Best Possible Solution

Add scoped local TLS fingerprint propagation for configured local WSS TUI connections, matching the existing status/probe and gateway call behavior while avoiding pins for explicit URL overrides.

## Reproduction Assessment

Yes, at source level. With `gateway.tls.enabled === true`, local connection details resolve to `wss://127.0.0.1:<port>`, but the TUI connection object cannot carry `tlsFingerprint` and the TUI `GatewayClient` never receives one.

## Solution Assessment

Yes. Scoped local fingerprint propagation is the narrow maintainable fix; disabling `NODE_TLS_REJECT_UNAUTHORIZED` or applying local pins to arbitrary URL overrides would be broader and less safe.
