## Summary

Keep open: current `origin/main` still has the TUI-specific local TLS trust gap. The related open bootstrap TLS PR is useful context, but it does not touch the TUI path or close this issue.

## What This Changes

The issue asks for `openclaw tui` to pass the local gateway TLS fingerprint to `GatewayClient` for configured local self-signed WSS gateways.

## Best Possible Solution

Add scoped local TLS fingerprint propagation for configured local WSS TUI connections, ideally using the shared resolver from the bootstrap TLS work if it lands, while avoiding pins for explicit URL overrides.

## Reproduction Assessment

Yes, at source level: current `origin/main` still resolves local TLS gateways to WSS while the TUI connection object and `GatewayClient` construction cannot carry `tlsFingerprint`. I did not run a live TUI plus self-signed gateway session during this read-only review.

## Solution Assessment

Yes: scoped local fingerprint propagation is the narrow maintainable fix, matching the existing gateway call and status/probe behavior. Disabling process-wide Node TLS verification or pinning arbitrary URL overrides would be broader and less safe.
