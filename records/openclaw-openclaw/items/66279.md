## Summary

Keep open: current main still has the reported TUI-specific local TLS trust gap. The TUI resolves TLS-enabled local gateway URLs but does not carry a `tlsFingerprint` into `GatewayClient`, while the normal gateway call and status/probe paths already do.

## What This Changes

The issue asks for the TUI local gateway connection path to load and pass the local TLS fingerprint for TLS-enabled loopback WSS targets.

## Best Possible Solution

Add scoped local TLS fingerprint propagation to `src/tui/gateway-chat.ts`, with regression tests for local TLS and URL override opt-out, preferably reusing or extracting the existing gateway-call policy.

## Reproduction Assessment

Yes, at source level. With `gateway.tls.enabled === true`, local connection details resolve to `wss://127.0.0.1:<port>`, but the TUI connection object cannot carry `tlsFingerprint` and the TUI `GatewayClient` never receives one.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix. The implementation should mirror the established status/probe and gateway-call policy while avoiding local pins for explicit URL overrides.
