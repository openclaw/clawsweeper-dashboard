## Summary

Keep open: current main and the latest release still have the reported TUI-specific local TLS trust gap. The TUI resolves local TLS gateway URLs but never carries a `tlsFingerprint` into `GatewayClient`, while the status/probe and normal gateway-call paths already do.

## What This Changes

Make `openclaw tui` use the local gateway TLS fingerprint for self-signed local WSS gateway connections instead of requiring process-wide TLS verification disablement.

## Best Possible Solution

Add scoped local TLS fingerprint propagation to `src/tui/gateway-chat.ts`, with regression tests for local TLS and explicit URL override opt-out, preferably sharing the established gateway-call precedence rules.

## Reproduction Assessment

Yes, at source level. With `gateway.tls.enabled === true`, local connection details resolve to `wss://127.0.0.1:<port>`, but the TUI connection object cannot carry `tlsFingerprint` and `GatewayChatClient` never passes one to `GatewayClient`.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix. The implementation should mirror the established status/probe and `callGateway` TLS policy while keeping explicit URL overrides and unrelated remote targets unpinned.
