## Summary

Keep open. Current main still has the TUI-specific local TLS trust gap: the TUI resolves the local WSS gateway URL and auth, but there is no `tlsFingerprint` field in the resolved connection and none is passed to `GatewayClient`, while status/probe paths already forward the local gateway TLS fingerprint.

## What This Changes

Make `openclaw tui` use the same scoped local gateway TLS fingerprint handling as status/probe when connecting to a TLS-enabled local gateway.

## Best Possible Solution

Implement a scoped local TLS fingerprint handoff in `src/tui/gateway-chat.ts`: when the configured local gateway target is WSS, `gateway.tls.enabled === true`, there is no explicit URL override, and the TUI is not using remote mode, load the local gateway TLS runtime, carry `fingerprintSha256` through the resolved connection, and pass it to `GatewayClient`. Add focused TUI tests for propagation, unavailable/disabled TLS runtime fallback, URL override, and remote mode, and coordinate the helper shape with related issue #41740 if the same pattern is needed elsewhere.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: with local gateway TLS enabled, the target becomes `wss://127.0.0.1:<port>`, but the TUI resolver/client path cannot carry `tlsFingerprint`, so `GatewayClient` cannot activate its scoped self-signed certificate pinning path.

## Solution Assessment

Yes. Scoped local fingerprint propagation matching status/probe is the narrow maintainable fix; relying on `NODE_TLS_REJECT_UNAUTHORIZED=0` would weaken TLS for the whole TUI process, and inheriting arbitrary pins for explicit remote URL overrides would be broader than the reported local trust problem.
