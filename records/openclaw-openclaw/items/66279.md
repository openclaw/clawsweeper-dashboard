## Summary

Keep open: current main and the latest release still have the TUI-specific local TLS trust gap. The TUI resolves a local TLS gateway URL but has no typed `tlsFingerprint` field, never loads the local TLS runtime, and constructs `GatewayClient` without the fingerprint that the working status/probe and `callGateway` paths already forward.

## What This Changes

The issue asks for the TUI local gateway connection path to load and pass the local TLS fingerprint for TLS-enabled loopback WSS gateways.

## Best Possible Solution

Add scoped local TLS fingerprint propagation in `src/tui/gateway-chat.ts`, with focused TUI regression tests mirroring `callGateway`/status precedence and live redacted local TLS TUI proof before landing.

## Reproduction Assessment

Yes, at source level. With `gateway.tls.enabled === true`, local connection details resolve to `wss://127.0.0.1:<port>`, but the TUI connection object cannot carry `tlsFingerprint` and `GatewayChatClient` never passes one to `GatewayClient`.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix. The implementation should mirror the established status/probe and `callGateway` TLS policy while keeping explicit URL overrides and unrelated remote targets unpinned.
