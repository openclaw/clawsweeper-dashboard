## Summary

Keep open. Current main still has the TUI-specific TLS trust gap: `openclaw tui` resolves the local WSS gateway URL and auth, but the resolved connection cannot carry a TLS fingerprint and `GatewayClient` never receives one, while status/probe paths already forward a scoped local gateway TLS fingerprint.

## What This Changes

The issue requests that `openclaw tui` load and pass the configured local gateway TLS certificate fingerprint to `GatewayClient` for local self-signed `wss://` gateways.

## Best Possible Solution

Implement the TUI trust handoff in `src/tui/gateway-chat.ts`: add an optional `tlsFingerprint` to the resolved connection, load the local gateway TLS runtime only for configured local `wss://` targets with `gateway.tls.enabled === true` and no URL override or remote target, pass it into `GatewayClient`, and add focused tests in `src/tui/gateway-chat.test.ts` for propagation, unavailable-runtime fallback, URL override, and remote cases. Prior PR #22453 is useful source work but should be adapted to current main.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: with local gateway TLS enabled, `buildGatewayConnectionDetails` resolves `wss://127.0.0.1:<port>`, but the TUI resolver/client path has no `tlsFingerprint` handoff, so the self-signed WSS trust path in `GatewayClient` cannot activate.

## Solution Assessment

Yes. The best fix is scoped local fingerprint propagation for the TUI path, matching the existing status/probe and `callGateway` behavior; documenting or relying on `NODE_TLS_REJECT_UNAUTHORIZED=0` would weaken TLS process-wide and is not an acceptable solution.
