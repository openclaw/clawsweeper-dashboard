## Summary

Keep open. Current main still has the TUI-specific local TLS trust gap: the TUI resolves the TLS-enabled loopback gateway URL and auth, but it never resolves or passes a `tlsFingerprint`, while the status/probe and shared gateway call paths already do. See https://github.com/openclaw/openclaw/issues/66279.

## What This Changes

The issue requests that `openclaw tui` use the local gateway TLS fingerprint when connecting to a TLS-enabled loopback gateway.

## Best Possible Solution

Add scoped local TLS fingerprint propagation for configured local WSS TUI connections, preferably by sharing the existing local-TLS decision shape with gateway bootstrap/call paths, and add focused regression coverage without applying pins to explicit remote URL overrides.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: with `gateway.tls.enabled === true`, the local TUI target becomes `wss://127.0.0.1:<port>`, but the resolved TUI connection cannot carry `tlsFingerprint` and the TUI `GatewayClient` never receives one.

## Solution Assessment

Yes. Scoped local fingerprint propagation matching `callGateway` and daemon status/probe is the narrow maintainable fix; disabling `NODE_TLS_REJECT_UNAUTHORIZED` or inheriting pins for arbitrary URL overrides would be broader and less safe.
