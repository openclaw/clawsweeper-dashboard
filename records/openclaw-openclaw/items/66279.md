## Summary

Keep open: current `main` and `v2026.5.18` still omit scoped local TLS fingerprint propagation in the TUI path. A prior TUI fix PR was closed stale, and the newer bootstrap-client TLS PR is adjacent but does not currently touch the TUI files.

## What This Changes

The issue asks for `openclaw tui` to trust the configured local TLS gateway using the same scoped fingerprint logic as status/probe instead of requiring `NODE_TLS_REJECT_UNAUTHORIZED=0`.

## Best Possible Solution

Add scoped local TLS fingerprint propagation in `src/tui/gateway-chat.ts`, preferably reusing the shared resolver pattern from the bootstrap TLS work when available, with focused TUI regression tests and live redacted local TLS TUI proof.

## Reproduction Assessment

Yes, at source level: configured local TLS can resolve to `wss://127.0.0.1:<port>`, but the TUI connection object cannot carry `tlsFingerprint` and `GatewayChatClient` never passes one to `GatewayClient`. I did not run a live local gateway in this read-only review.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix. The implementation should mirror the established `callGateway` local TLS policy while keeping explicit URL overrides and unrelated remote targets unpinned.
