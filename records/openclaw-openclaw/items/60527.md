## Summary

Keep open. Current main and shipped #62312 solve the reported Telegram/private-proxy failure by clearing the target allowlist only when `allowPrivateProxy` is true, but this PR still proposes broader public explicit-proxy behavior that current main does not implement. That remaining behavior changes SSRF proxy trust semantics, so it needs maintainer security/product review rather than cleanup closure.

## What This Changes

The PR strips `hostnameAllowlist` from explicit proxy hostname validation and adds SSRF regression tests for private proxies, public proxies, and the private-proxy blocking invariant.

## Best Possible Solution

Keep the shipped private-proxy fix, then make an explicit maintainer decision on the public explicit-proxy contract. If public configured proxies should ignore target allowlists, rebase this PR and document/test that contract; otherwise close this PR as superseded by #62312 while preserving the current `allowPrivateProxy` trust boundary.

## Reproduction Assessment

Yes for the remaining behavior. A `fetchWithSsrFGuard` call with `policy.hostnameAllowlist: ["api.telegram.org"]`, an explicit public proxy host outside that allowlist, and no `allowPrivateProxy` still reaches the default branch on current main and is rejected before fetch.

## Solution Assessment

Unclear. Removing target allowlists from operator-trusted proxy validation is the right shape, but applying it to every explicit proxy without the current trust bit is broader than the shipped fix and should be confirmed against the SSRF security model.
