## Summary

Keep open. Current main already shipped the Telegram/private-proxy regression fix through the narrower allowPrivateProxy path, but this PR still proposes a broader change for non-allowPrivateProxy explicit proxies. That remaining trust-contract change is security-sensitive and needs maintainer review rather than cleanup closure.

## What This Changes

The PR strips hostnameAllowlist from explicit-proxy hostname validation for all explicit proxies and adds tests for private proxy, public proxy, and private-proxy blocking behavior.

## Best Possible Solution

Keep the shipped allowPrivateProxy-scoped fix for operator-configured private proxies. If maintainers want all explicit proxies treated as operator-controlled, rebase this PR and keep the public-proxy regression test; otherwise preserve the current trust bit and document that hostnameAllowlist is only cleared for trusted explicit proxies.

## Reproduction Assessment

Yes for the remaining behavior: on current main, a public explicit proxy outside policy.hostnameAllowlist still flows through the default explicit-proxy branch and is rejected by resolveHostnamePolicyChecks before fetch runs. The original Telegram private-proxy symptom is already covered by current source and tests.

## Solution Assessment

Unclear. Stripping a target allowlist from proxy-host validation is the right shape for trusted operator proxies, but applying it to every explicit proxy without allowPrivateProxy is broader than the shipped fix and should be decided against the proxy security contract.
