## Summary

Keep open. Current main still has no native `doctor --dry-run` flag, and doctor repair paths can still write config, rewrite cron state, install plugin payloads, build UI assets, or repair gateway services; the related global dry-run issue and open `--check-config` PR cover different scopes.

## What This Changes

The issue requests a native `openclaw doctor --dry-run` mode that validates and previews doctor fixes without writing config or triggering repair side effects.

## Best Possible Solution

Add a focused doctor dry-run contract that registers and types the flag, renders the same repair plan, suppresses prompts and all writes/repairs, exits non-zero when issues are found, updates docs and changelog, and keeps #41418 and #47302 separate.

## Reproduction Assessment

Yes from source inspection. Current main does not register or type a `doctor --dry-run` option, and the existing doctor repair/write paths remain reachable without any dry-run guard.

## Solution Assessment

Yes. A doctor-scoped dry-run is the narrowest maintainable solution because the global agent dry-run request and `--check-config` PR do not prevent doctor repair side effects.
