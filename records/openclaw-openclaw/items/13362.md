## Summary

Keep open. Current main still lacks a native `openclaw doctor --dry-run` contract: the flag is not registered, the doctor option type cannot carry it, the docs do not list it, and several doctor repair paths still have direct write/install/service side effects. The related global dry-run issue and the open `--check-config` PR cover different layers, so they do not supersede this request.

## What This Changes

Add an `openclaw doctor --dry-run` mode that previews doctor validation and repair recommendations without writing config, staging dependencies, normalizing cron state, or repairing services.

## Best Possible Solution

Add a focused doctor-specific dry-run mode that is registered in the CLI, typed in doctor options, threaded through prompter/repair-mode/config/health flows, and honored by every doctor mutation path while rendering the same repair plan and updating public docs and changelog.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a missing CLI safety feature. The gap is directly verifiable on current main because `doctor --dry-run` is not registered, not represented in `DoctorOptions`, not documented, and not available to guard doctor mutation paths.

## Solution Assessment

Yes. A native doctor dry-run is the narrowest maintainable solution because doctor owns config repair, bundled plugin dependency staging, cron normalization, and gateway service repair semantics that the global dry-run issue and the open `--check-config` PR do not cover.

## Repair Work Prompt

Implement `openclaw doctor --dry-run` for issue #13362. Add a doctor-specific no-mutation preview mode, not a global agent tool dry-run and not `--check-config`. Register `--dry-run` on the doctor command, add `dryRun?: boolean` to `DoctorOptions`, thread it through doctor repair mode/prompter/config/health flows, and ensure dry-run never prompts for or performs writes, dependency installs, cron saves, gateway service install/start/restart/stage, or gateway auth persistence. It should still run validation and render the same fix recommendations/repair plan, and it should return a defined non-zero result when fixable issues are found. Update `docs/cli/doctor.md` and add a user-facing changelog entry with `Thanks @Riya-chandra`. Do not change unrelated global tool execution dry-run behavior, do not fold in PR #47302's `--check-config` scope, and do not weaken existing `--fix`/`--repair` behavior.
