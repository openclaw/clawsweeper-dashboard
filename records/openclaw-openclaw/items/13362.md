## Summary

Keep open. Current main still lacks a native `openclaw doctor --dry-run` contract, and the existing global dry-run request plus the open `--check-config` PR cover different scopes rather than superseding this doctor repair-preview feature.

## What This Changes

Add an `openclaw doctor --dry-run` mode that validates and previews doctor repairs without writing config, installing dependencies, changing cron state, or repairing services.

## Best Possible Solution

Implement a doctor-specific dry-run mode that is registered, typed, documented, and honored by every doctor mutation path while keeping #41418 and #47302 separate.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a missing CLI safety feature. The gap is directly verifiable on current main because the flag is not registered, not typed, not documented, and not available to guard doctor mutation paths.

## Solution Assessment

Yes. A native doctor dry-run is the narrowest maintainable fix because doctor owns config repair, plugin install repair, cron normalization, and gateway service repair semantics that a global agent dry-run or deep config check would not cover.

## Repair Work Prompt

Implement `openclaw doctor --dry-run` for https://github.com/openclaw/openclaw/issues/13362. Add the CLI flag in `src/cli/program/register.maintenance.ts`, add `dryRun?: boolean` to `DoctorOptions`, and thread it through doctor prompter/repair-mode/config/health flows. Dry-run must run validation and render the same repair plan, but must not prompt for or perform writes, plugin install repair, cron store saves, gateway service install/start/restart/bootstrap, or other doctor mutations. It should return a defined non-zero result when issues/repairs would be needed, without treating previewed issues as command crashes. Keep #41418 (global agent dry-run) and #47302 (`--check-config`) separate; do not implement those scopes here. Update `docs/cli/doctor.md` and add a user-facing `CHANGELOG.md` entry. Do not change unrelated config migrations, plugin install resolution, or service repair policy beyond the dry-run guard. Validate with the targeted doctor tests plus formatter, then run `pnpm check:changed` in Testbox before handoff.
