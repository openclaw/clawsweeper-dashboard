## Summary

Keep open. Current main and v2026.6.8 still reject cross-agent `sessions_send` under non-`all` visibility before A2A allow rules can approve it, and the remaining fix changes a session security boundary that needs maintainer product/security review.

## What This Changes

Allow `sessions_send` to use A2A policy as the cross-agent send gate without exposing list/history/status under broader session visibility.

## Best Possible Solution

Land or replace the linked send-only authorization change after maintainer security/product approval, with tests proving A2A-allowed sends work while list/history/status stay visibility-gated.

## Reproduction Assessment

Yes, at source level: a cross-agent `send` under effective visibility below `all` reaches the visibility denial before A2A allow checks. I did not run a live current-main gateway reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the security model. The send-only A2A exception is plausible, but the best fix must explicitly preserve read/list/status isolation and avoid fragmenting the session-visibility config model.
