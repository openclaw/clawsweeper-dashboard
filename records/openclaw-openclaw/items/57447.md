## Summary

Keep open. Current main and the latest release still reject cross-agent `sessions_send` when effective session visibility is below `all`, before A2A allow rules can approve the send, and the remaining change is a security-sensitive product decision with an unmerged conflicting linked fix candidate.

## What This Changes

Issue requests allowing A2A-approved cross-agent `sessions_send` without broadening session list, history, or status visibility.

## Best Possible Solution

Keep this issue as the canonical session-authorization tracker and land a maintainer-approved focused fix that lets A2A-authorized sends work under isolated read visibility while preserving list/history/status isolation and sandbox safeguards.

## Reproduction Assessment

Yes, at source level: a cross-agent `send` under effective visibility below `all` reaches the visibility denial before A2A allow checks. I did not run a live gateway reproduction because this review is read-only and source evidence is already direct.

## Solution Assessment

Unclear until maintainers choose the security model. A send-only A2A exception is plausible, but the best fix must explicitly preserve read/list/status isolation and avoid fragmenting the session-visibility config model.
