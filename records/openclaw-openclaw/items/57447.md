## Summary

Keep open. Current main still blocks cross-agent `sessions_send` under non-`all` session visibility before an A2A allow policy can authorize it, and an open implementation PR still uses closing syntax for this issue, so cleanup closure would be premature.

## What This Changes

Allow `sessions_send` to reach A2A-authorized cross-agent sessions without granting cross-agent list, history, or status visibility.

## Best Possible Solution

Review and update PR #57483, or replace it with a narrow current-main PR, so `createSessionVisibilityChecker` lets cross-agent `send` fall through to `tools.agentToAgent` policy while read/status/list actions remain visibility-gated. Add focused regression coverage for A2A-allowed sends under `tree`, A2A-denied sends, unchanged read/status/list denial, and the sandboxed label/session resolution case if kept in scope.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `tools.sessions.visibility="tree"` with `tools.agentToAgent.enabled=true` and an allowlist containing the requester and target agent IDs; a cross-agent `sessions_send` still hits the non-`all` visibility rejection before the A2A allow check.

## Solution Assessment

Yes for the requested authorization model: send should be gated by A2A policy rather than transcript visibility. The open PR’s direction is plausible, but it must be rebased into the active SDK helper and either handle sandboxed cross-agent resolution or leave that as an explicit follow-up.
