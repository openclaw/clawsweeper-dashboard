## Summary

Keep open. Current main and the latest release still reject cross-agent `sessions_send` when effective session visibility is below `all`, before A2A allow rules can approve the send, and the remaining change is a security-sensitive product decision rather than conservative cleanup.

## What This Changes

The issue requests allowing A2A policy to authorize cross-agent `sessions_send` without broadening cross-agent session list/history/status visibility.

## Best Possible Solution

Keep this issue open as the canonical session-authorization bug and review or replace the linked send-only fix with tests proving A2A-allowed sends work while list/history/status remain visibility-gated.

## Reproduction Assessment

Yes, at source level: a cross-agent `send` under effective visibility below `all` reaches the visibility denial before A2A allow checks. I did not run a live current-main gateway reproduction because this read-only review must keep the checkout unchanged.

## Solution Assessment

Unclear until maintainers choose the security model. The send-only A2A exception is plausible, but the best fix must explicitly preserve read/list/status isolation and avoid fragmenting the session-visibility config model.
