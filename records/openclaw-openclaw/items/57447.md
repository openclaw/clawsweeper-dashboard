## Summary

Keep open. Current main and the latest release still reject cross-agent `sessions_send` when effective session visibility is below `all`, before A2A allow rules can approve the send; the remaining change is a security-sensitive product decision with an open but conflicting linked fork PR candidate.

## What This Changes

The issue requests allowing `sessions_send` to use A2A policy for cross-agent sends without broadening session list/history/status visibility.

## Best Possible Solution

Keep this as the canonical session-authorization tracker and land a maintainer-approved focused fix that lets A2A-authorized sends work under isolated read visibility while preserving list/history/status isolation and sandbox safeguards.

## Reproduction Assessment

Yes, at source level: a cross-agent `send` under effective visibility below `all` reaches the visibility denial before A2A allow checks. I did not run a live gateway reproduction because this is a read-only review and the source path is direct.

## Solution Assessment

Unclear until maintainers choose the security model. A send-only A2A exception is plausible, but the fix must explicitly preserve read/list/status isolation and sandbox safeguards.
