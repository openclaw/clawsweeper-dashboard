## Summary

Keep open. This is a new privileged Control UI and Gateway capability, not a bug or cleanup close; current main has exec/PTY plumbing for agent tooling, but no browser terminal route, Gateway terminal RPC, or Control UI terminal surface, and direct browser-to-host shell access needs maintainer product and security design first.

## What This Changes

The issue requests an embeddable xterm.js web terminal in Control UI backed by a Gateway WebSocket PTY bridge, session tabs, auth integration, and optional Canvas embedding.

## Best Possible Solution

Design an explicit opt-in terminal capability or extension API that reuses Gateway auth scopes, exec approval/audit concepts, session cleanup, and sandbox/node boundaries before adding any PTY bridge.

## Reproduction Assessment

Not applicable: this is a feature request, not broken existing behavior. Source and docs inspection show no web terminal while existing PTY support belongs to exec/process tooling.

## Solution Assessment

Unclear as proposed: xterm.js plus a PTY bridge is a plausible implementation component, but the maintainable solution first needs an operator-controlled security model and protocol/API design.
