## Summary

Keep open. Current main still lacks timeout-triggered pending-session capture and later recovery into daily memory; the existing pre-compaction flush, visible timeout error handling, command-scoped session-memory hook, and optional transcript indexing do not implement the requested lifecycle, and the persistence design needs product and security review.

## What This Changes

Add a repeated-model-timeout recovery flow that temporarily stores session content and later converts it into durable memory notes.

## Best Possible Solution

Design a generic timeout-recovery lifecycle that stores bounded private pending transcript artifacts and drains them idempotently into daily memory with dedupe, retention, cleanup, docs, and focused tests.

## Reproduction Assessment

Yes, by source inspection: repeated memory-flush provider timeouts currently become visible error payloads and return without creating a pending artifact or later recovery drain. I did not run a live GLM-5 timeout loop.

## Solution Assessment

No, not as a mechanical pending-directory patch. The requested direction is plausible, but the maintainable solution needs an explicit memory/session contract for privacy, retention, idempotency, cleanup, recovery ordering, and docs.
