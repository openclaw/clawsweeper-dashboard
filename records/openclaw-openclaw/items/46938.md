## Label Justifications

- `P2`: The issue describes a real memory/session persistence gap with user impact, but the fix is a bounded product design rather than an urgent core outage.
- `impact:data-loss`: Repeated model timeouts can leave conversation content in transcripts without a corresponding durable memory artifact.
- `impact:session-state`: The affected behavior is session transcript preservation and later recovery across failed model turns.

## Summary

Keep open. Current main has memory flush, command-scoped session-memory capture, timeout-triggered compaction, and optional session transcript indexing, but it still lacks durable repeated-timeout pending-session capture and later recovery into memory; the remaining work needs product and security design.

## What This Changes

The issue requests a session-level temporary storage and recovery mechanism so repeated model timeouts do not leave conversation content only in raw session transcripts without memory maintenance.

## Best Possible Solution

Design one generic timeout-recovery lifecycle that stores bounded private pending transcript artifacts and drains them idempotently into memory with dedupe, retention, cleanup, docs, and focused regression coverage.

## Reproduction Assessment

Yes, by source inspection rather than a live six-timeout provider run. Current main can surface memory-flush timeout failures and run timeout compaction, but no code path creates durable pending-session artifacts or drains them later into memory files.

## Solution Assessment

No, not as a mechanical memory/pending patch. The maintainable solution needs an explicit memory/session contract for privacy, retention, idempotency, cleanup, configuration, and docs.
