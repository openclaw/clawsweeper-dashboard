## Summary

Keep open. Current main has session-memory on `/new`/`/reset`, pre-compaction memory flush, and timeout-triggered compaction, but it still lacks durable repeated-timeout pending-session capture and later recovery into memory; the remaining work needs product and security design.

## What This Changes

The issue requests durable timeout-recovery storage for session transcripts so repeated model timeouts can later be processed into memory, with suggested threshold, pending-directory, and auto-recovery settings.

## Best Possible Solution

Design one generic timeout-recovery lifecycle that stores bounded private pending transcript artifacts and drains them idempotently into memory with dedupe, retention, cleanup, docs, and focused regression coverage.

## Reproduction Assessment

Yes, by source inspection rather than a live six-timeout provider run. Current main can surface memory-flush failures and run timeout compaction, but no code path creates durable pending-session artifacts or drains them later into memory files.

## Solution Assessment

No, not as a direct `memory/pending` patch. The maintainable solution needs an explicit memory/session contract for privacy, retention, idempotency, cleanup, configuration, and docs.
