## Label Justifications

- `P2`: The issue describes a real memory/session persistence gap with user impact, but the fix is a bounded product design rather than an urgent core outage.
- `impact:data-loss`: Repeated model timeouts can leave conversation content in transcripts without a corresponding memory artifact.
- `impact:session-state`: The affected behavior is session transcript preservation and later recovery across failed model turns.
- `impact:security`: The proposed pending store would persist additional conversation-derived content and needs privacy and retention controls.

## Summary

Keep open. Current main and the latest shipped tag have timeout compaction and optional session transcript indexing, but still do not implement repeated model-timeout pending-session capture or later recovery into memory files; the remaining work needs product and security design.

## What This Changes

The issue requests a session-level pending storage and recovery mechanism so repeated model timeouts do not prevent conversation content from being preserved into memory.

## Best Possible Solution

Design one generic timeout-recovery lifecycle that stores bounded private pending transcript artifacts and drains them idempotently into memory with dedupe, retention, cleanup, docs, and focused regression coverage.

## Reproduction Assessment

Yes, by source inspection rather than a live six-timeout provider run. Current main surfaces memory-flush timeout failures and has command-scoped session-memory capture, but no durable pending-session artifact or recovery drain for repeated model timeouts.

## Solution Assessment

No, not as a mechanical `memory/pending` patch. The maintainable solution needs an explicit memory/session contract for privacy, retention, idempotency, cleanup, configuration, and docs.
