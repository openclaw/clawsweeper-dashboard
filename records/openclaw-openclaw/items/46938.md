## Summary

Keep open. Current main has adjacent memory flush, session-memory, active-memory timeout handling, session transcript indexing, timeout compaction, and idle-timeout breaker behavior, but it still does not implement repeated-timeout pending session capture followed by later recovery into daily memory; the linked #46930 remains a separate gateway log-level report.

## What This Changes

The issue requests a timeout recovery mechanism that temporarily stores unprocessed session content after repeated model timeouts and later converts it into memory entries.

## Best Possible Solution

Add a generic timeout recovery lifecycle that persists bounded, private pending transcript artifacts after repeated unprocessed timeouts, then drains them through idempotent memory maintenance with dedupe, retention, cleanup, docs, and regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to simulate repeated model timeouts around a session with user turns: current timeout paths return blocked/error outcomes and memory flush failures are logged, with no pending memory artifact or later recovery job.

## Solution Assessment

No. The current adjacent mechanisms are not the best complete solution; the maintainable path is a generic timeout lifecycle design rather than extending only the command-scoped session-memory hook.
