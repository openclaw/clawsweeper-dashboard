## Summary

Keep open. Current main has adjacent memory-flush error reporting, Active Memory timeout-partial recall, opt-in session transcript indexing, and `/new`/`/reset` session-memory capture, but it still lacks repeated model-timeout pending-session capture followed by later recovery into daily memory.

## What This Changes

Add a timeout-recovery mechanism that stores session content after repeated model timeouts and later converts it into daily memory.

## Best Possible Solution

Design a generic timeout-recovery lifecycle that stores bounded private pending transcript artifacts and drains them idempotently into memory with dedupe, retention, cleanup, docs, and focused regression coverage.

## Reproduction Assessment

Yes, by source inspection rather than a live timeout run. Current main reports memory-flush timeout failures and has command-scoped session-memory capture, but no pending-session artifact or recovery drain for repeated model timeouts.

## Solution Assessment

No, not as a mechanical one-file fix. The proposed `memory/pending` direction is plausible, but the maintainable solution needs memory/session contract design for privacy, retention, idempotency, cleanup, and docs.
