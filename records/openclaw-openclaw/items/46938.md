## Summary

Keep open. Current main has adjacent memory flush, session-memory, transcript indexing, timeout compaction, and failover behavior, but it still does not implement repeated-timeout pending session capture followed by later recovery into daily memory. The linked #46930 is a separate gateway log-level report, not a canonical tracker for this memory recovery work.

## What This Changes

The issue requests a mechanism that stages session content after repeated model timeouts and later recovers it into memory when model availability returns.

## Best Possible Solution

Add a generic timeout recovery path in the session/model-fallback lifecycle: when repeated timeout exhaustion leaves user turns unprocessed, persist a redacted pending transcript artifact with stable metadata, then process it after model availability returns through bounded memory maintenance with dedupe, retention, cleanup, docs, and regression tests.

## Reproduction Assessment

Yes. A high-confidence reproduction can be built by simulating repeated model timeouts for a session with user turns; current source and timeout-compaction tests show the run falls through to timeout handling without creating a pending memory artifact or later recovery job.

## Solution Assessment

No, the current adjacent mechanisms are not a complete solution. Extending only the command-scoped session-memory hook would be the wrong boundary; the safer path is a generic timeout lifecycle recovery design with privacy and idempotency controls.
