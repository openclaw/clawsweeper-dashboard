## Summary

## What This Changes

## Best Possible Solution

Keep the scan-time camera recovery UX, but sync the RequestPermission result into the same PermissionState used by PermissionSetupScreen and applyToViewModel, with grant-path regression coverage before merge.

Yes, at source level: current main maps QR scanner startup failure to generic copy while camera permission is modeled elsewhere, and the linked issue plus contributor emulator proof show the denied-camera setup path. I did not independently run an Android emulator in this read-only review.

## Solution Assessment

No, not as-is. The scan/error boundary is the right layer for the denied-camera UX, but the grant callback must update the same permission state that onboarding later persists.
