## Summary

Keep open. The PR targets a real IRC proactive channel-send bug and has already incorporated the identity-check feedback, but the current diff still leaves CLI-to-gateway sends on the transient unjoined path and can expose a monitor client before JOIN membership is confirmed.

## What This Changes

The PR adds a module-level IRC active-client registry, registers monitor clients, and makes `sendMessageIrc` prefer a ready registered client before falling back to a transient connection.

## Best Possible Solution

Land a focused IRC delivery fix that routes proactive channel sends through the gateway-owned joined connection, only exposes monitor clients after JOIN confirmation, preserves an explicit no-monitor fallback, and adds regression tests plus the required changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run `openclaw message send` for an IRC channel while the gateway monitor owns the joined socket; the CLI process cannot see the PR's module-local registry and falls back to the transient unjoined send path. The JOIN race is also source-reproducible because readiness resolves on `001` before server-confirmed channel membership.

## Solution Assessment

No. Reusing the persistent monitor client is the right direction for same-process sends, but this implementation is not the narrowest complete fix because it misses the CLI/gateway process boundary and JOIN-confirmed readiness.
