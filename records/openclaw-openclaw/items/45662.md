## Summary

Keep open: current main still has a source-reproducible IRC proactive channel-send problem, but this PR is not merge-ready because its in-memory registry does not cover the CLI/gateway process boundary, can publish before JOIN confirmation, and still lacks inspectable real IRC proof.

## What This Changes

The PR adds an IRC active-client registry, has the monitor register its client, and makes `sendMessageIrc` prefer that persistent client before falling back to a transient connection.

## Best Possible Solution

Route proactive IRC channel sends through gateway-owned IRC connection state that is visible to CLI-originated sends, only mark a client sendable after confirmed channel membership, preserve a tested no-monitor fallback, and require redacted live IRC proof.

## Reproduction Assessment

Yes. Source inspection shows current main falls back to a transient unjoined IRC client when no explicit monitor client is passed, while normal CLI sends run outside the gateway monitor process.

## Solution Assessment

No. The module-local registry is not the best fix because it cannot bridge the CLI/gateway process boundary and can publish a client before confirmed channel membership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route proactive IRC sends through the gateway connection:** `extensions/irc/src/send.ts:72-74`
  - body: This lookup only sees monitor clients in the current Node process. `openclaw message send` runs in a CLI process while IRC outbound remains direct, so channel sends can still fall back to the same transient unjoined connection on networks that require membership before `PRIVMSG`.
  - confidence: 0.91

  - confidence: 0.84
