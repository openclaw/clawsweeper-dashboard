## Summary

Keep open. The PR targets a real IRC proactive-send bug, but the current patch still misses the CLI/gateway process boundary, registers the monitor client before confirmed joins, and lacks inspectable after-fix real behavior proof.

## What This Changes

The PR adds an IRC active-client registry, registers the monitor client, and makes `sendMessageIrc` prefer that registered client before using the transient fallback.

## Best Possible Solution

Route proactive IRC channel sends through a gateway-owned joined connection, expose readiness only after confirmed JOINs, preserve a safe no-monitor fallback, and add focused tests plus redacted real IRC proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: CLI proactive IRC sends run locally unless the plugin opts the action into gateway execution, and current main then falls back to a transient unjoined IRC client. I did not run a live Ergo reproduction in this read-only review.

## Solution Assessment

No. Reusing the monitor client is directionally useful, but a module-local registry is not the best fix because it misses the CLI/gateway process boundary and publishes the client before confirmed JOIN membership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route proactive IRC sends through the gateway connection:** `extensions/irc/src/send.ts:72-74`
  - body: The active-client registry is module-local, but `openclaw message send` executes in its own CLI process and gateway dispatch only happens for actions whose plugin declares gateway execution. IRC send does not, so CLI proactive sends cannot see the monitor's joined socket and fall back to the same transient unjoined send path.
  - confidence: 0.9
- **[P2] Publish monitor clients only after channel joins are confirmed:** `extensions/irc/src/monitor.ts:140`
  - body: `connectIrcClient()` resolves readiness on numeric `001` and only then sends JOIN commands without waiting for server-confirmed membership. Registering the monitor client immediately means startup or restart proactive sends can still route through a socket that has not joined the configured channel yet.
  - confidence: 0.86
