## Summary

Keep open: current main still has the IRC proactive-send problem and this PR targets a real bug, but the patch is not correct enough to merge because the registry is process-local and can publish a client before JOIN confirmation; it also still lacks inspectable real IRC proof.

## What This Changes

The PR adds an IRC active-client registry and makes IRC sends prefer a monitor-owned persistent client before falling back to a transient connection.

## Best Possible Solution

Route proactive IRC channel sends through a gateway-owned IRC connection known to be joined, preserve the no-monitor fallback, and add focused regression coverage plus redacted real IRC proof.

## Reproduction Assessment

Yes, from source inspection rather than a live IRC run: `openclaw message send` runs in the CLI process, IRC outbound is direct, and current `sendMessageIrc` opens a transient client when no explicit monitor client is passed.

## Solution Assessment

No. The PR's module-local registry cannot bridge the CLI/gateway process boundary and registers before channel join confirmation; a gateway-owned joined connection is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P2] Route proactive IRC sends through the gateway connection:** `extensions/irc/src/send.ts:72-74`
  - body: The active-client map is process-local. `openclaw message send` runs in a fresh CLI process while IRC outbound remains direct, so this lookup will be empty for the user-facing proactive-send path and the code falls back to the same transient unjoined client.
