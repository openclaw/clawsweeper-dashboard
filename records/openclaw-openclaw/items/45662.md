## Summary

Keep open, but the branch should not merge as-is: the reported IRC message-loss bug is source-reproducible, while this process-local registry does not solve the CLI/gateway proactive-send path and can still publish a client before channel joins are confirmed.

## What This Changes

The PR adds an IRC active-client registry, registers monitor clients on connect/stop, and makes sendMessageIrc prefer the registered client before opening a transient IRC connection.

## Best Possible Solution

Route proactive IRC channel sends through a gateway-owned IRC connection with confirmed channel membership, preserve the no-monitor fallback, and add focused regression coverage plus redacted live IRC proof.

## Reproduction Assessment

Yes, source-reproducible: current main's CLI message send path enters runMessageAction in the CLI process, IRC outbound is direct, and sendMessageIrc falls back to a transient unjoined client when no explicit monitor client is passed.

## Solution Assessment

No. The proposed process-local registry cannot bridge the CLI/gateway process boundary, and it can register the monitor client before server-confirmed JOIN membership.
