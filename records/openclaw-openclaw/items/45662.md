## Summary

Keep open: this targets a real IRC proactive-send bug, but the submitted Map-based fix still misses the CLI/gateway process boundary and can publish a monitor client before JOIN confirmation; the external PR also lacks observable after-fix real behavior proof.

## What This Changes

The PR adds an IRC active-client registry, registers monitor clients, and makes `sendMessageIrc` prefer a ready registered client before falling back to a transient connection.

## Best Possible Solution

Land a focused IRC delivery fix that routes proactive channel sends through the gateway-owned joined connection, exposes monitor clients only after JOIN-confirmed membership, preserves a no-monitor fallback, and adds focused regression coverage.

## Reproduction Assessment

Yes, by source inspection rather than a live IRC run: `openclaw message send --channel irc --target #room` executes in the CLI process while the monitor-owned joined socket lives in the gateway process, so the PR's module-local registry is empty and the transient unjoined send path remains.

## Solution Assessment

No. Reusing the monitor client is directionally right for same-process sends, but this patch is not the best fix because it misses the CLI/gateway process boundary and JOIN-confirmed readiness.

## Review Findings

Overall correctness: patch is incorrect
