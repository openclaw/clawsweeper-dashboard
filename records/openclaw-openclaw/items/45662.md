## Summary

Keep open. The IRC message-loss bug is real and user-impacting, but this PR is not merge-ready: the proposed registry is process-local, can publish before confirmed channel membership, and still lacks inspectable real IRC proof; the earlier Greptile identity-check concern was addressed by the second commit.

## What This Changes

The PR adds an IRC active-client registry, registers monitor clients, and makes `sendMessageIrc` prefer that registered client before using a transient IRC connection.

## Best Possible Solution

Route proactive IRC channel sends through a gateway-owned IRC connection with confirmed channel membership, preserve the no-monitor fallback, and add focused regression coverage plus redacted live IRC proof.

## Reproduction Assessment

Yes, source-reproducible: current main's message command enters `runMessageAction` in the CLI process, IRC outbound is direct, and `sendMessageIrc` falls back to a transient unjoined client when no explicit monitor client is passed.

## Solution Assessment

No. The proposed process-local registry cannot bridge the CLI/gateway process boundary and can publish the monitor client before server-confirmed JOIN membership; the fix should use a gateway-owned connection path with membership readiness.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route proactive IRC sends through the gateway connection:** `extensions/irc/src/send.ts:72-74`
  - body: This lookup is module-local. `openclaw message send` runs in a fresh CLI process while IRC outbound remains direct, so the user-facing proactive-send path will not see the gateway monitor's client and will fall back to the same transient unjoined connection.
  - confidence: 0.91
- **[P2] Wait for joined membership before publishing the monitor client:** `extensions/irc/src/monitor.ts:140`
  - body: `connectIrcClient()` reports ready on numeric `001` and only then emits `JOIN` commands; it does not wait for server-confirmed channel membership. Registering the client here can still send during startup before the server has accepted the channel join.
  - confidence: 0.84
