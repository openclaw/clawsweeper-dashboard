## Summary

Keep open: this PR targets a real IRC delivery bug, but the proposed module-local registry does not cover CLI-to-gateway proactive sends and can still expose a monitor socket before channel joins are confirmed. Current main has not implemented the fix, and the recent user comment shows the problem is still relevant.

## What This Changes

The PR adds an IRC active-client registry, registers monitor clients, and makes `sendMessageIrc` prefer a ready registered client before falling back to a transient connection.

## Best Possible Solution

Land a focused IRC delivery fix that routes proactive channel sends through the gateway-owned joined connection, exposes monitor clients only after JOIN-confirmed membership, preserves a no-monitor fallback, and adds focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `openclaw message send --channel irc --target #room` runs in the CLI process while the gateway monitor owns the joined IRC socket, so the PR's module-local registry is empty and the transient unjoined send path remains.

## Solution Assessment

No. Reusing the monitor client is directionally right for same-process sends, but this patch is not the best fix because it misses the CLI/gateway process boundary and JOIN-confirmed readiness.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Route active IRC sends through the gateway connection:** `extensions/irc/src/send.ts:72-73`
  - body: The registry is process-local. `openclaw message send` runs in the CLI process while the IRC monitor lives in the gateway process, so this lookup is empty for CLI proactive sends and the code falls back to the transient unjoined `PRIVMSG` path.
  - confidence: 0.88
- **[P2] Wait for joins before publishing monitor clients:** `extensions/irc/src/monitor.ts:144`
  - body: `connectIrcClient()` resolves readiness on numeric `001` and only then emits JOIN commands. Registering the client immediately can route startup proactive sends through a socket that has not joined the target channel yet.
  - confidence: 0.84
