## Summary

Keep open: the IRC delivery bug is real and the branch is focused, but the proposed module-local registry still misses CLI/gateway proactive sends, can publish a client before JOIN confirmation, and lacks inspectable real behavior proof.

## What This Changes

This PR adds a module-local IRC active-client registry, registers monitor clients, and makes `sendMessageIrc` prefer that persistent client before falling back to a transient connection.

## Best Possible Solution

Route proactive IRC channel sends through gateway-owned IRC connection state visible to CLI-originated sends, expose clients only after confirmed channel membership, preserve a tested no-monitor fallback, and collect redacted live IRC proof.

## Reproduction Assessment

Yes, source-reproducible: current main sends without an explicit ready client through a transient IRC connection, while the normal CLI message path can run outside the gateway monitor process. I did not run a live IRC server in this read-only review.

## Solution Assessment

No. The module-local map is not the best fix because it cannot bridge the CLI/gateway process boundary and can publish a client before confirmed JOIN membership.
