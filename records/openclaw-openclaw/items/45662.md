## Summary

Keep open. The reported IRC proactive-send bug is real and the PR is directionally useful, with the earlier identity-cleanup concern addressed, but the current diff still does not fix standalone/direct sends that run outside the gateway process and can publish the monitor client before channel JOIN membership is confirmed.

## What This Changes

The PR adds an IRC active-client registry populated by monitor connections and makes `sendMessageIrc` prefer a ready registered client before using the transient IRC connection fallback.

## Best Possible Solution

Land a focused IRC delivery fix that routes proactive channel sends through a gateway-owned or otherwise process-shared joined connection, only exposes a monitor client after the needed channel JOINs are confirmed, preserves an explicit no-monitor fallback, and adds regression tests plus the required changelog entry. Keep the broader #47234 routing work separate unless maintainers intentionally bundle it.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: send an IRC channel message through the direct CLI path while the gateway monitor owns the joined socket, and the PR's module-local lookup is empty in that process so it falls back to the transient client. The JOIN race is also reproducible from the client code path that resolves readiness on `001` before confirmed membership.

## Solution Assessment

No. Reusing a persistent monitor client is the right direction for same-process replies and sends, but a module-local registry is not the narrowest complete fix for gateway/CLI process boundaries and JOIN readiness. A safer fix should route through the gateway-owned IRC connection or an explicit joined-client handoff.
