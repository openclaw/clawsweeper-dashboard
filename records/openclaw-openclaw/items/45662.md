## Summary

Keep this PR open, but do not merge it as-is: the IRC message-loss bug is valid, yet this branch uses a process-local client registry that does not fix the CLI/gateway proactive-send path and can still publish a client before channel JOIN membership is confirmed.

## What This Changes

The PR adds an IRC active-client registry, registers monitor clients, and makes `sendMessageIrc` prefer that registered client before opening a transient IRC connection.

## Best Possible Solution

Route proactive IRC channel sends through a gateway-owned IRC connection with confirmed channel membership, preserve the no-monitor fallback, and add focused regression coverage plus redacted live IRC proof.

## Reproduction Assessment

Yes, source-reproducible: current main's CLI message-send path runs in the CLI process, IRC outbound is direct, and `sendMessageIrc` falls back to a transient unjoined client when no explicit monitor client is passed.

## Solution Assessment

No. The proposed process-local registry is not the best fix because it cannot bridge the CLI/gateway process boundary and it can publish a monitor client before confirmed JOIN membership.
