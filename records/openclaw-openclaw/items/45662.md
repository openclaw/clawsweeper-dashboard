## Summary

Keep open. The reported IRC proactive-send bug is credible, but this PR does not fix the user-facing CLI/gateway path because its active-client registry is process-local and it can expose a monitor client before channel joins are confirmed; it also still lacks inspectable real IRC proof.

## What This Changes

The PR adds an IRC active-client Map, registers monitor clients, and makes `sendMessageIrc` prefer that client before creating a transient IRC connection.

## Best Possible Solution

Route proactive IRC channel sends through a gateway-owned IRC connection known to be joined to the target channel, preserve the no-monitor fallback, and add focused regression coverage plus redacted real IRC proof.

## Reproduction Assessment

Yes, source-reproducible: current main's CLI message path calls `runMessageAction` in the CLI process, IRC outbound is direct, and `sendMessageIrc` opens a transient client when no explicit monitor client is passed.

## Solution Assessment

No. The PR's process-local registry cannot bridge the CLI/gateway process boundary and is published before JOIN confirmation; a gateway-owned joined connection is the narrower maintainable fix.
