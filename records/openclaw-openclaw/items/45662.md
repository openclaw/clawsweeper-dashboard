## Summary

Keep open: this targets a real IRC delivery bug, but the process-local registry does not cover CLI/gateway proactive sends, can expose a client before confirmed channel JOIN, and still lacks inspectable real IRC proof.

## What This Changes

The PR adds an IRC active-client registry, registers monitor clients, and makes `sendMessageIrc` prefer a registered monitor client before using the transient fallback.

## Best Possible Solution

Route proactive IRC channel sends through a gateway-owned joined IRC connection with explicit channel-membership readiness, preserve a tested no-monitor fallback, and add focused regression coverage plus redacted live IRC proof.

## Reproduction Assessment

Yes, source-reproducible: current main's no-client IRC send path creates a transient connection without joining channels, and the CLI send path runs outside the gateway process that owns monitor clients.

## Solution Assessment

No. The module-local registry is not the best fix because it cannot bridge the CLI/gateway process boundary and it publishes readiness before confirmed JOIN membership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route proactive IRC sends through the gateway connection:** `extensions/irc/src/send.ts:72-74`
  - body: This lookup only sees monitor clients in the current Node process. `openclaw message send` runs in a CLI process while IRC outbound remains direct, so channel sends can still fall back to the same transient unjoined connection on networks that require membership before `PRIVMSG`.
  - confidence: 0.91
- **[P2] Publish clients only after confirmed JOIN membership:** `extensions/irc/src/monitor.ts:140`
  - body: `connectIrcClient()` reports ready on numeric `001` and only then emits JOIN commands. Registering the client here can still send during startup before the server has accepted the channel join, preserving the drop window this patch is trying to remove.
  - confidence: 0.84
