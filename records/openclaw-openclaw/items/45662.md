## Summary

Keep open: the PR addresses a real IRC channel delivery bug, but the submitted Map-based registry is process-local and still misses CLI-to-gateway proactive sends plus JOIN-confirmed readiness.

## What This Changes

The PR adds an IRC active-client registry, registers monitor clients, and makes `sendMessageIrc` prefer a ready registered client before using a transient connection.

## Best Possible Solution

Land a focused IRC delivery fix that routes channel sends through the gateway-owned joined connection, exposes readiness only after JOIN confirmation, preserves an explicit no-monitor fallback, and adds regression tests plus changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: standalone `openclaw message send` runs in a CLI process, IRC direct delivery calls `sendMessageIrc` without `opts.client`, and the transient connection has no channel join override.

## Solution Assessment

No. Reusing the persistent monitor client is the right direction for same-process replies, but this patch is not complete because it does not cross the CLI/gateway process boundary or wait for confirmed IRC channel joins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Route active IRC sends through the gateway-owned connection:** `extensions/irc/src/send.ts:72-73`
  - body: The new lookup only sees monitor clients in the current Node process. `openclaw message send` runs from the CLI process while the IRC monitor lives in the gateway process, so channel sends still fall back to the transient unjoined path.
  - confidence: 0.88
- **[P2] Publish monitor clients only after channel joins finish:** `extensions/irc/src/monitor.ts:144`
  - body: `connectIrcClient()` resolves readiness on numeric `001` and then emits JOIN commands without waiting for server-confirmed membership. Registering immediately can route startup sends through a socket that has not joined the target channel yet.
  - confidence: 0.84

  - body: This is a user-facing IRC delivery fix, but the PR only changes runtime files. Repo policy requires a `CHANGELOG.md` entry for user-facing fixes before merge.
  - confidence: 0.78
