## Summary

This PR addresses a real IRC message-loss path and has an active user impact comment, but it should stay open because the branch still cannot cover the normal CLI/gateway process boundary and can register a monitor client before JOIN membership is confirmed. The missing inspectable real IRC proof is also still a merge gate.

## What This Changes

The PR adds an IRC active-client registry, registers monitor clients from the IRC monitor, and makes sendMessageIrc prefer that registered persistent client before opening a transient connection.

## Best Possible Solution

Route proactive IRC channel sends through a gateway-owned IRC connection with confirmed channel membership, preserve a deliberate no-monitor fallback, and add focused regression coverage plus redacted live IRC proof.

## Reproduction Assessment

Yes, source-reproducible: current main's sendMessageIrc opens a transient IRC client for proactive sends when no explicit ready client is passed, and that path sends PRIVMSG without joining the channel first. I did not run a live IRC server in this read-only review.

## Solution Assessment

No. Reusing a process-local registry is too narrow for the CLI/gateway boundary and should not publish a monitor client until channel membership is confirmed.
