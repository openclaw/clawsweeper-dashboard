## Summary

Keep this PR open: it targets a real IRC message-delivery bug, but the current branch is not merge-ready because its module-local client registry does not cover process-separated CLI/gateway sends and it can publish a client before confirmed channel membership.

## What This Changes

The PR adds an IRC active-client registry, registers monitor clients, and makes `sendMessageIrc` prefer a registered monitor client before falling back to a transient IRC connection.

## Best Possible Solution

Route proactive IRC channel sends through a gateway-owned IRC connection with confirmed channel membership, preserve a tested no-monitor fallback, and add focused regression coverage plus redacted real IRC proof.

## Reproduction Assessment

Yes, source-reproducible: current main's CLI send path runs in the CLI process, IRC outbound is direct, and `sendMessageIrc` falls back to a transient unjoined client when no explicit monitor client is passed.

## Solution Assessment

No. The process-local registry is not the best fix because it cannot bridge the CLI/gateway process boundary and can publish a monitor client before confirmed JOIN membership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route proactive IRC sends through the gateway connection:** `extensions/irc/src/send.ts:72-74`
  - body: This lookup only sees monitor clients in the current Node process. `openclaw message send` runs in a CLI process while IRC outbound remains direct, so channel sends can still fall back to the same transient unjoined connection on networks that require membership before `PRIVMSG`.

  - confidence: 0.84
