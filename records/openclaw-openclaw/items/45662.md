## Summary

Keep open: the PR targets a real IRC proactive-send delivery bug, but the proposed module-local registry still misses CLI-to-gateway sends and can publish a client before channel JOIN is confirmed. The branch also still lacks inspectable real IRC proof.

## What This Changes

The PR adds a module-local IRC active-client registry, has the monitor register/deregister its client, and makes `sendMessageIrc` prefer that client before falling back to a transient connection.

## Best Possible Solution

Route proactive IRC channel sends through a gateway-owned IRC connection with confirmed channel membership, preserve a tested no-monitor fallback, and add focused regression coverage plus redacted real IRC proof.

## Reproduction Assessment

Yes, source-reproducible: current main's `sendMessageIrc` falls back to a transient unjoined client when no explicit monitor client is passed, while CLI sends run outside the gateway monitor process. I did not run a live IRC server in this read-only review.

## Solution Assessment

No. The module-local registry is not the best fix because it cannot bridge the CLI/gateway process boundary and can publish a monitor client before confirmed channel membership.

## Review Findings
