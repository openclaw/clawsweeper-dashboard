## Summary

Keep open: the IRC proactive-send bug appears real and current main still has the transient-send path, but this PR does not cover the CLI/gateway process boundary, publishes the monitor client before confirmed channel membership, and lacks inspectable real behavior proof.

## What This Changes

The PR adds an IRC active-client registry, registers monitor clients, and makes `sendMessageIrc` prefer the registered persistent client before falling back to a transient connection.

## Best Possible Solution

Route proactive IRC channel sends through a gateway-owned joined IRC connection, expose readiness only after confirmed channel membership, preserve the no-monitor fallback, and add focused tests plus redacted real IRC proof.

## Reproduction Assessment

Yes. Source inspection shows `openclaw message send` invokes `runMessageAction` in the CLI process, IRC outbound is direct, and current `sendMessageIrc` opens a transient client when no explicit client is supplied; I did not run a live IRC server.

## Solution Assessment

No. The module-local registry is useful only for same-process calls and does not prove joined channel membership before publishing the client; a gateway-owned joined connection is the narrower maintainable fix for the user-facing path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

  - body: The active-client map is process-local. `openclaw message send` runs in a fresh CLI process and IRC outbound remains direct, so this lookup will be empty for the user-facing proactive-send path and the code falls back to the same transient unjoined client.
  - confidence: 0.91

  - confidence: 0.84
