## Summary

Keep open. Current main still has the transient IRC proactive-send path this PR targets, but the proposed patch is not merge-ready because a process-local registry and early monitor registration can still leave channel sends failing in supported flows.

## What This Changes

The PR adds a module-level IRC active-client registry, registers monitor clients with identity-safe cleanup, and makes `sendMessageIrc` prefer a ready active client before falling back to a transient IRC connection.

## Best Possible Solution

Land a focused IRC delivery fix that uses a verified joined monitor or gateway-owned connection when available, preserves an explicit no-monitor fallback, and adds regression coverage for monitor reuse, identity-safe cleanup, fallback behavior, process boundaries, and JOIN timing. Keep the broader #47234 routing work separate unless maintainers intentionally bundle the two.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: call `sendMessageIrc` for an IRC channel without `opts.client` on current main and it builds a transient client without configured channel joins, sends `PRIVMSG`, and quits.

## Solution Assessment

No. Reusing a persistent monitor client is directionally right for same-process sends, and the identity-safe cleanup is useful, but a module-local Map is not the best complete fix unless the process boundary and JOIN-confirmation timing are handled explicitly.

- **[P2] Route active IRC sends through the gateway-owned connection:** `extensions/irc/src/send.ts:72-73`
  - body: The new lookup only checks a module-level Map in the sender process. `openclaw message send` and other direct-delivery callers can run in a fresh CLI process while the IRC monitor lives in the gateway process, so `getActiveClient(...)` is empty and channel sends fall back to the same transient path this patch is meant to avoid.
  - confidence: 0.87
- **[P2] Wait for channel JOINs before publishing monitor clients:** `extensions/irc/src/monitor.ts:144`
  - body: `connectIrcClient()` resolves readiness immediately after sending JOIN commands, not after the server confirms membership. Registering that client as active lets a startup proactive send reuse a socket that may not have joined the channel yet, so the `PRIVMSG` can still be dropped on the same networks affected by the bug.
  - confidence: 0.84
