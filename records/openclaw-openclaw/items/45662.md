## Summary

Keep this PR open because it targets a real IRC channel-send bug and a later user comment confirms the pain, but the current patch is not mergeable: the registry is process-local, so CLI/gateway proactive sends still miss the joined monitor connection, and the client is published before JOIN confirmation.

## What This Changes

The PR adds an IRC active-client registry, registers monitor clients, deregisters them on stop, and makes `sendMessageIrc` prefer a ready registered client before the transient send fallback.

## Best Possible Solution

Implement proactive IRC channel sends through a gateway-owned joined connection, expose send readiness only after JOIN confirmation, preserve the no-monitor fallback, and add focused tests plus redacted real IRC proof.

## Reproduction Assessment

Yes, by source inspection rather than a live IRC run: CLI proactive sends execute outside the gateway monitor process, so this branch's module-local registry remains empty and channel sends can still fall back to the transient unjoined path.

## Solution Assessment

No. Reusing the monitor client is directionally useful, but a module-level registry misses the process boundary and publishes the client before JOIN confirmation; the safer fix is gateway-owned IRC delivery with confirmed channel readiness.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route proactive IRC sends through the gateway connection:** `extensions/irc/src/send.ts:72-74`
  - body: The new active-client lookup only sees monitor clients in the current Node process. `openclaw message send` runs in the CLI process while the joined monitor socket lives in the gateway, so `getActiveClient(...)` stays empty and channel sends still fall back to the transient unjoined `PRIVMSG` path.
  - confidence: 0.88
- **[P2] Delay publishing monitor clients until joins are confirmed:** `extensions/irc/src/monitor.ts:140`
  - body: `connectIrcClient()` reports readiness on numeric `001`, then sends JOIN commands without waiting for server-confirmed membership. Registering the client immediately can route startup sends through a socket that has not joined the target channel yet.
  - confidence: 0.84
