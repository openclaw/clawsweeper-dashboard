## Summary

Keep this PR open: it targets a real IRC message-delivery bug, but the current patch is not merge-ready because the registry is process-local, can publish a client before confirmed channel JOIN membership, and still lacks inspectable real IRC proof.

## What This Changes

The PR adds an IRC active-client registry, registers monitor clients, and makes `sendMessageIrc` prefer a registered persistent client before falling back to a transient IRC connection.

## Best Possible Solution

Route proactive IRC channel sends through a gateway-owned IRC connection with confirmed channel membership, keep a tested fallback for no-monitor cases, and add focused regression coverage plus redacted real IRC proof.

## Reproduction Assessment

Yes, source-reproducible: current main's direct IRC send path falls back to a transient client when no explicit monitor client is passed, and the CLI path cannot see a monitor process's module-local map.

## Solution Assessment

No. The process-local registry is not the best fix because it cannot bridge the CLI/gateway process boundary and can expose a monitor client before confirmed JOIN membership.
