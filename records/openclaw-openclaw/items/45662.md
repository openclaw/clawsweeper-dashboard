## Summary

Keep open: the underlying IRC proactive-send bug still appears valid, and a later user comment confirms real demand, but this PR does not fully fix the CLI/gateway process boundary and still lacks inspectable after-fix real behavior proof.

## What This Changes

The branch adds an IRC active-client registry, registers the monitor client, and makes `sendMessageIrc` prefer that registered client before falling back to a transient connection.

## Best Possible Solution

Route proactive IRC channel sends through a gateway-owned joined connection, expose readiness only after confirmed channel membership, keep a safe no-monitor fallback, and add focused tests plus redacted real IRC proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `openclaw message send` runs in the CLI process, IRC outbound delivery is direct, and current `sendMessageIrc` falls back to a transient client when no explicit monitor client is passed; I did not run a live Ergo server.

## Solution Assessment

No. Reusing the monitor client is directionally useful for same-process sends, but the module-local registry misses the CLI/gateway process boundary and does not prove joined channel membership before publishing the client.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route proactive IRC sends through the gateway connection:** `extensions/irc/src/send.ts:72-74`
  - body: The new active-client map is process-local. CLI sends run in a fresh command process, IRC declares direct outbound delivery, and gateway dispatch only happens for plugin actions that opt into gateway execution, so this lookup is empty for CLI proactive sends and the code falls back to the same transient unjoined path.
  - confidence: 0.91
- **[P2] Publish IRC clients only after joined membership is known:** `extensions/irc/src/monitor.ts:140`
  - body: `connectIrcClient()` resolves readiness after numeric `001` and after sending JOIN commands, not after the server confirms channel membership. Registering the client immediately lets early proactive sends use a socket whose configured channel joins may still be pending or rejected, preserving the false-success failure on stricter IRC networks.
  - confidence: 0.84
