## Summary

Keep open: the PR targets a real IRC proactive-send bug, but the current patch is not mergeable as-is because the registry is process-local, can publish a client before channel joins are confirmed, and still lacks inspectable real IRC proof.

## What This Changes

The PR adds an IRC active-client registry, registers monitor clients, and makes `sendMessageIrc` prefer that persistent client before transient sends.

## Best Possible Solution

Route proactive IRC channel sends through a gateway-owned IRC connection that is known to have joined the target channel, preserve the no-monitor fallback, and add focused regression coverage plus redacted real IRC proof.

## Reproduction Assessment

Yes, from source inspection rather than a live IRC run: `openclaw message send` runs in the CLI process, IRC outbound is direct, and current `sendMessageIrc` opens a transient client when no explicit monitor client is passed.

## Solution Assessment

No. The PR's module-local registry only helps same-process callers and can expose clients before JOIN confirmation; a gateway-owned joined connection is the narrower maintainable fix.
