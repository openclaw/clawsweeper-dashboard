## Summary

Keep open: current main still has a source-reproducible IRC proactive channel-send gap, and this PR is focused enough to preserve, but it is not merge-ready because the registry is process-local, can be exposed before JOIN confirmation, and still lacks inspectable real IRC proof.

## What This Changes

The PR adds a module-local active IRC client registry populated by the monitor and has `sendMessageIrc` prefer that persistent client before falling back to a transient connection.

## Best Possible Solution

Keep the focused IRC fix open, but rework it so proactive channel sends use gateway-owned joined IRC state or a tested joined/drained fallback path, with regression coverage and redacted live IRC proof.

## Reproduction Assessment

Yes, by source inspection. Current main falls back to a transient IRC client when `sendMessageIrc` has no explicit monitor client, and direct CLI/outbound paths can run outside the gateway monitor process; I did not run a live IRC server in this read-only review.

## Solution Assessment

No, not as currently implemented. Reusing a monitor client is a plausible part of the fix, but the registry must account for process boundaries and confirmed channel membership before it can be the durable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Route proactive sends through joined gateway state:** `extensions/irc/src/send.ts:70-74`
  - body: The new lookup only sees monitor clients in the current Node process. `openclaw message send` and other direct outbound paths can run outside the gateway monitor process, while IRC declares direct delivery, so `getActiveClient(...)` will be empty and channel sends still use the transient unjoined connection this PR is trying to avoid. Route these sends through gateway-owned IRC state or add a tested joined-channel fallback before treating this as fixed.
  - confidence: 0.89
- **[P2] Wait for channel membership before publishing the client:** `extensions/irc/src/monitor.ts:140`
  - body: `connectIrcClient()` resolves on numeric `001` after issuing JOIN commands, not after the server confirms channel membership. Registering the monitor client immediately means a startup proactive send can still hit a socket that has not joined the channel yet. Track joined channels or delay registry publication until the target channel is confirmed joined.
  - confidence: 0.84
