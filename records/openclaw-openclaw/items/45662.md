## Summary

Keep open: the reported IRC proactive-send bug is still valid, but this PR should not merge as-is because its process-local registry cannot see the gateway monitor socket from CLI sends and it publishes the monitor client before JOIN confirmation.

## What This Changes

Adds an IRC active-client registry populated by the monitor and makes `sendMessageIrc` prefer that registered client before falling back to a transient connection.

## Best Possible Solution

Implement proactive IRC channel sends through a gateway-owned joined connection, expose send readiness only after confirmed JOINs, keep a no-monitor fallback, and add focused tests plus redacted real IRC proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main still falls back to a transient unjoined IRC client for proactive sends, and this PR’s module-local registry remains empty when sends run outside the gateway monitor process.

## Solution Assessment

No. Reusing the monitor client is directionally right, but this PR’s in-memory registry misses the CLI/gateway process boundary and registers before JOIN confirmation; gateway-owned delivery with confirmed channel readiness is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Route proactive IRC sends through the gateway connection:** `extensions/irc/src/send.ts:72-74`
  - body: The active-client lookup only sees monitor clients in the current Node process. CLI proactive sends can run outside the gateway process that owns the joined IRC socket, so `getActiveClient(...)` stays empty and channel sends still fall back to the transient unjoined `PRIVMSG` path.
  - confidence: 0.89
- **[P2] Wait for JOIN confirmation before registering clients:** `extensions/irc/src/monitor.ts:140`
  - body: `connectIrcClient()` reports readiness on numeric `001`, then sends JOIN commands without waiting for confirmed membership. Registering the client immediately can route startup sends through a socket that has not joined the target channel yet.
  - confidence: 0.84
