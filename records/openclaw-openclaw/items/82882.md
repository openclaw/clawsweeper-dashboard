## What This Changes

The issue reports that the browser Control UI still sends Gateway protocol range `4..4` while the current Gateway requires protocol `5`, causing WebSocket handshakes to fail.

## Best Possible Solution

Land or revise the linked fix so the browser Control UI derives its connect protocol range from the shared Gateway protocol constants and has a regression test for the emitted fields.

## Reproduction Assessment

Yes for source-level reproduction: current main has Gateway protocol constants at 5 while the browser Control UI still emits 4..4 in `buildConnectParams`. I did not run the live browser/Gateway loop in this read-only review.

## Solution Assessment
