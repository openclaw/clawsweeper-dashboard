- `impact:crash-loop`: The affected behavior is launchd repeatedly restarting the Gateway after fatal startup secret-resolution failures.

## Summary

Keep open: current main and the latest release still turn startup SecretRef resolution failures into a fatal Gateway startup error, while the remaining linked fork PR is stale/conflicting and not a clean landing path.

## What This Changes

This issue asks OpenClaw to prevent Gateway startup SecretRef provider failures, especially 1Password exec failures under launchd, from crash-looping and exhausting external secret-service rate limits.

## Best Possible Solution

Keep this issue as the canonical tracker for a fresh, focused Gateway startup SecretRef retry/backoff/degrade policy in `src/gateway/server-startup-config.ts`, with the broader external-provider batching work left to https://github.com/openclaw/openclaw/issues/72278.

## Reproduction Assessment

Yes, source-reproducible. Current main still throws a fatal startup error when `prepareSecretsRuntimeSnapshot` fails, and adjacent tests assert that fatal wrapping; I did not run a live launchd/1Password rate-limit repro in this read-only review.

## Solution Assessment

Unclear. Bounded startup retry/backoff is a plausible mitigation, but the best permanent behavior needs maintainer choice on fail-closed versus degraded startup semantics, and the stale linked fork PR is not the right patch as-is.
