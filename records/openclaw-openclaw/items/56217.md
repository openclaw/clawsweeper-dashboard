## Summary

Keep open: current main and v2026.6.6 still abort startup after a single SecretRef resolution failure. The open linked fork PR is conflicting and the older OpenClaw PR was closed unmerged, so this remains a maintainer/security product decision.

## What This Changes

The issue requests startup SecretRef failure handling so exec-based external secret providers, especially 1Password `op`, do not crash-loop and exhaust shared rate limits.

## Best Possible Solution

Keep this issue open until maintainers land or replace a focused startup SecretRef retry/backoff/degrade policy in `src/gateway/server-startup-config.ts`, with regression coverage and separate tracking for broader external-provider batching.

## Reproduction Assessment

Yes. Source inspection shows current main calls secret preparation during startup and then throws a fatal startup error on resolution failure; adjacent tests assert that path, though no live 1Password/launchd run was performed.

## Solution Assessment

Unclear. Bounded startup retry/backoff is plausible, but a new config surface or broad batching change is not automatically the best fix; maintainers should choose the startup failure policy and keep generic external-provider batching on the separate tracker.
