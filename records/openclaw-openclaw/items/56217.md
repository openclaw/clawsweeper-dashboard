## Summary

Keep open: current main and v2026.6.8 still abort gateway startup when required SecretRef resolution fails, while exec SecretRefs are only batched per configured provider and have no startup backoff, circuit breaker, or provider-rate-limit handling. The Varlock 1Password plugin linked in discussion is useful external batching/caching workaround context, but it does not resolve OpenClaw core's gateway startup failure policy.

## What This Changes

The issue asks OpenClaw to avoid repeated exec-backed 1Password SecretRef calls when gateway startup failures are retried by launchd or another process manager.

## Best Possible Solution

Keep this issue as the canonical tracker and have maintainers choose a generic startup SecretRef failure policy, with focused gateway/secrets tests and no 1Password-specific core hardcode unless that policy is explicitly approved.

## Reproduction Assessment

Yes, as a source-level reproduction: current main still throws a fatal startup error when required SecretRef preparation fails, and exec provider failures propagate directly into that path. I did not live-reproduce the launchd plus 1Password quota exhaustion loop.

## Solution Assessment

Unclear until maintainer policy is chosen. A bounded generic startup retry/degrade strategy is plausible, but the issue's suggested retryPolicy, circuit breaker, op run batching, and rate-limit awareness are alternative designs with different security and compatibility tradeoffs.
