## Summary

Keep open: current main and v2026.6.10 still fail gateway startup when required SecretRef activation fails, and exec provider failures still propagate without startup backoff, circuit breaking, or rate-limit-aware behavior. The Varlock 1Password plugin linked in discussion is useful external batching/caching workaround context, but it does not resolve OpenClaw core's startup failure policy.

## What This Changes

The issue requests safer handling for exec-backed SecretRef startup failures so supervised gateway restarts do not repeatedly call 1Password and exhaust account-wide rate limits.

## Best Possible Solution

Keep this as the canonical tracker while maintainers choose a generic startup SecretRef failure policy, with focused gateway/secrets tests and backend-specific batching or caching staying in plugins when possible.

## Reproduction Assessment

Yes at source level: current main still throws a fatal startup error when required SecretRef preparation fails, and exec provider failures propagate directly into that path. I did not live-reproduce the launchd plus 1Password quota exhaustion loop.

## Solution Assessment

Unclear until maintainer policy is chosen. A bounded generic startup retry or degraded-startup strategy is plausible, but retryPolicy, circuit breaker, batching, and rate-limit awareness each have different security and compatibility tradeoffs.
