## Label Justifications

- `impact:crash-loop`: The reported failure mode is a process manager repeatedly restarting the gateway after fatal startup SecretRef failures.
- `impact:auth-provider`: The affected path is SecretRef/provider credential resolution against an external secret service.

## Summary

Keep open: current main and v2026.6.6 still fail gateway startup when required SecretRef preparation fails, so a process manager can restart into the same provider path. The Varlock 1Password plugin noted in discussion is useful workaround context, but it does not resolve OpenClaw's core startup failure policy, and the open linked fork PR is conflicting/stale rather than a clean landing path.

## What This Changes

The issue requests backoff, circuit-breaker, batching, or rate-limit-aware behavior for repeated external SecretRef provider failures during gateway startup.

## Best Possible Solution

Land a fresh, focused core startup SecretRef policy that bounds repeated provider attempts and documents operator recovery, while keeping backend-specific batching and caching in resolver plugins such as Varlock or ClawHub packages.

## Reproduction Assessment

Yes, source-reproducible. Current main still wraps startup SecretRef preparation failures as fatal gateway startup errors, and adjacent tests assert that behavior; no live launchd/1Password rate-limit run was performed.

## Solution Assessment

Unclear. Bounded startup retry/backoff is a plausible mitigation, but maintainers need to choose the fail-fast versus degraded-startup policy and the linked fork PR is not the best fix as-is.
