## Summary

Keep open: current main still fail-fasts startup SecretRef activation with no retry/backoff, and the linked closing PR is still open and unmerged, so this issue should remain as the canonical reporter thread until that fix is landed, replaced, or closed.

## What This Changes

The issue requests bounded startup resilience for 1Password-backed exec SecretRef failures so gateway restarts do not repeatedly invoke `op read` and exhaust account-wide rate limits.

## Best Possible Solution

Review and either land a focused current-main startup SecretRef retry/backoff fix from https://github.com/openclaw/openclaw/pull/56499 or replace it with an equivalent narrow fix, while keeping cross-provider batching and memoization on https://github.com/openclaw/openclaw/issues/72278.

## Reproduction Assessment

Yes for the current failure path: inject a rejecting `prepareRuntimeSecretsSnapshot`, or use an active exec SecretRef provider that fails, during `createRuntimeSecretsActivator(..., { reason: "startup" })`; current main calls the resolver once and throws the fatal startup error. I did not run a live launchd plus 1Password service-account loop.

## Solution Assessment

Unclear. Startup retry/backoff is a plausible first fix, but maintainers should decide the product boundary between hardcoded bounded startup resilience, configurable retry policy, rate-limit-specific handling, and the broader batching/memoization work tracked separately.
