## Label Justifications

- `P1`: The report describes a gateway startup restart loop that can exhaust external credential-provider limits and block affected users until manual intervention.
- `impact:crash-loop`: The issue is about repeated gateway process restarts after startup SecretRef resolution failure under launchd KeepAlive.

## Summary

Keep open: current main and the latest shipped tag still fail fast on startup SecretRef provider errors, and the open closing PR has not landed or cleared its review blockers.

## What This Changes

The issue requests startup backoff, circuit breaking, batching, or rate-limit-aware handling so failing external SecretRef providers cannot drive a launchd restart loop against 1Password.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/56499 with a focused current-main fix in `createRuntimeSecretsActivator` or the provider-aware resolver boundary, while leaving broader batching to https://github.com/openclaw/openclaw/issues/72278 and 1Password CLI cache behavior to https://github.com/openclaw/openclaw/issues/55459.

## Reproduction Assessment

Yes, source-level: configure an active failing exec SecretRef provider, or inject a rejecting `prepareRuntimeSecretsSnapshot` into `createRuntimeSecretsActivator` with reason `startup`. Current main calls the resolver once and wraps the failure as a fatal startup error; I did not run a live launchd plus 1Password loop.

## Solution Assessment

Unclear. Bounded startup backoff is a plausible first repair, but maintainers still need to choose the boundary between startup resilience, provider-aware retry/circuit breaking, configurable retry policy, and broader external-provider batching.
