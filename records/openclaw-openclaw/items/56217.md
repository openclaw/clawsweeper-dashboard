## Summary

Keep open: current main and the latest release still fail fast on startup SecretRef provider failures, and the open linked fix PR has not landed or cleared its proof/rework blockers.

## What This Changes

The issue asks OpenClaw to prevent failing 1Password-backed SecretRef resolution from driving a launchd gateway restart loop that exhausts external secret-provider rate limits.

## Best Possible Solution

Land a focused current-main SecretRef startup resilience fix in `createRuntimeSecretsActivator` or a provider-aware resolver, with regression coverage and redacted launchd/1Password-equivalent proof, while keeping broader batching work in https://github.com/openclaw/openclaw/issues/72278.

## Reproduction Assessment

Yes, source-level: current main can be driven through a rejecting `prepareRuntimeSecretsSnapshot` and it throws the fatal startup error; an active failing exec SecretRef provider enters the same path. I did not live-run a launchd plus 1Password rate-limit loop in this read-only review.

## Solution Assessment

Unclear: startup backoff is a plausible fix, but maintainers still need to choose the retry granularity and failure policy so the fix reduces external calls instead of replaying successful provider groups.
