- `impact:crash-loop`: The issue centers on launchd repeatedly restarting the gateway after fatal startup secret resolution failures.

## Summary

Keep open: current main and v2026.6.6 still fail fatal on startup SecretRef resolution errors, while the linked implementation path is an open conflicting fork PR plus a closed superseded upstream PR, so maintainers still need to choose and land a focused gateway startup retry/degrade policy.

## What This Changes

This issue requests startup backoff, circuit-breaking, or batching for exec-backed SecretRef provider failures so process-manager restarts do not exhaust 1Password service-account limits.

## Best Possible Solution

Keep this issue open until maintainers land or replace a focused startup SecretRef retry/degrade policy in `src/gateway/server-startup-config.ts`, with regression coverage and separate tracking for broader external-provider batching.

## Reproduction Assessment

Yes. Source inspection shows current main calls secret preparation during startup and then throws a fatal startup error on resolution failure; adjacent tests assert that fatal path, though no live 1Password/launchd run was performed.

## Solution Assessment

Unclear. The backoff direction is plausible, but the stale PRs are not the best landing path; a focused owner-approved policy in `src/gateway/server-startup-config.ts` is safer than adding a broad config retry surface by default.
