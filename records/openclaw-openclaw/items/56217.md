## Label Justifications

- `P1`: The report describes a real gateway startup restart loop that can exhaust external credential-provider rate limits and block affected users until manual intervention.
- `impact:crash-loop`: The issue is about repeated process restarts after startup SecretRef resolution failure under launchd KeepAlive.
- `impact:auth-provider`: The affected path is SecretRef-backed credential/provider resolution for external secret providers such as 1Password.

## Summary

Keep open: current main and v2026.5.18 still fail-fast active startup SecretRef resolution without retry/backoff, and the open closing PR at https://github.com/openclaw/openclaw/pull/56499 has not merged.

## What This Changes

The issue asks OpenClaw to prevent failed 1Password-backed SecretRef resolution from causing a launchd restart loop that exhausts external secret-provider rate limits.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/56499 with a focused current-main fix in `createRuntimeSecretsActivator` or the provider-aware resolver boundary, while keeping external-provider batching on https://github.com/openclaw/openclaw/issues/72278 and `op` cache behavior on https://github.com/openclaw/openclaw/issues/55459.

## Reproduction Assessment

Yes, source-level: configure an active failing exec SecretRef provider, or inject a rejecting `prepareRuntimeSecretsSnapshot` into `createRuntimeSecretsActivator` with `reason: "startup"`; current main calls the resolver once and wraps the failure as a fatal startup error. I did not run a live launchd plus 1Password loop.

## Solution Assessment

Unclear. Bounded startup backoff is a plausible first repair, but maintainers still need to choose the boundary between startup resilience, provider-aware retry, configurable retry policy, and the broader external-provider batching issue.
