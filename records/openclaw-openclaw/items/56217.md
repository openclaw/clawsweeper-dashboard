## Summary

Keep open because current main still fail-fasts startup SecretRef activation without retry/backoff, and open PR #56499 already references this issue with closing syntax.

## What This Changes

The issue reports that gateway startup SecretRef failures can crash-loop under launchd and repeatedly invoke 1Password-backed exec secret providers until account rate limits are exhausted.

## Best Possible Solution

Review and land or replace PR #56499 for bounded startup SecretRef retry/backoff, while keeping broad provider batching and memoization tracked under #72278.

## Reproduction Assessment

Yes, via source inspection. Make prepareRuntimeSecretsSnapshot or an active exec SecretRef inside it fail during activateRuntimeSecrets with `reason: "startup"`; current main rejects through the fatal startup path before any retry/backoff behavior.

## Solution Assessment

Unclear. Bounded startup exponential backoff is a narrow and plausible fix for crash-loop amplification, but maintainers should review PR #56499 against reload last-known-good behavior and leave cross-provider batching/memoization to #72278 unless they choose to combine the work.
