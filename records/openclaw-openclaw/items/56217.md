## Summary

Keep open: current main and v2026.6.10 still fail gateway startup when required SecretRef activation fails, and exec provider failures still propagate without startup backoff, circuit breaking, or rate-limit-aware behavior. The Varlock 1Password plugin linked in discussion is useful batching/caching workaround context, but it does not resolve OpenClaw core's startup failure policy.

## What This Changes

The issue requests crash-loop and rate-limit protection for gateway SecretRef provider failures, especially 1Password exec-provider failures under launchd restart.

## Best Possible Solution

Keep this as the canonical tracker for a generic startup SecretRef failure policy, while backend-specific batching or caching remains external/plugin-owned when current seams are sufficient.

## Reproduction Assessment

Yes at source level: current main still throws a fatal startup error when required SecretRef preparation fails, and exec provider failures propagate into that path. I did not live-reproduce launchd plus 1Password quota exhaustion.

## Solution Assessment

Unclear until maintainers choose the startup policy. A bounded generic startup retry or degraded-startup strategy is plausible, but the requested retryPolicy config, circuit breaker, batching, and rate-limit awareness have different compatibility and security tradeoffs.
