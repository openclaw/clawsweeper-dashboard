## Summary

Keep open: current `main` still fail-fasts startup SecretRef activation with no retry/backoff, and the open closing PR at https://github.com/openclaw/openclaw/pull/56499 is still unmerged. The broader external-provider batching/memoization part is already tracked separately at https://github.com/openclaw/openclaw/issues/72278.

## What This Changes

The issue requests startup SecretRef crash-loop protection for failing exec-backed secret providers such as 1Password CLI.

## Best Possible Solution

Review and either land a focused current-main replacement for https://github.com/openclaw/openclaw/pull/56499 or close/replace that PR, while keeping cross-provider batching and memoization on https://github.com/openclaw/openclaw/issues/72278.

## Reproduction Assessment

Yes, source-level: inject a rejecting `prepareRuntimeSecretsSnapshot`, or configure an active failing exec SecretRef provider, during `createRuntimeSecretsActivator(..., { reason: "startup" })`; current main calls the resolver once and throws the fatal startup error. I did not run a live launchd plus 1Password service-account loop.

## Solution Assessment

Unclear. Startup retry/backoff is a plausible first fix, but maintainers should choose the boundary between bounded startup resilience, provider-aware retry, configurable retry policy, and the broader batching/memoization issue.
