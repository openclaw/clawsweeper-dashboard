## Label Justifications

- `P1`: The report describes a gateway startup restart loop that can exhaust external credential-provider capacity and block recovery for real users.
- `impact:crash-loop`: The central failure mode is repeated gateway startup aborts under a process manager restart policy.
- `impact:auth-provider`: The affected path is SecretRef credential resolution through an exec-backed external provider.

## Summary

Keep open: current main and v2026.6.10 still fail gateway startup when required SecretRef activation fails, while exec provider failures propagate immediately without startup backoff, circuit breaking, or rate-limit-aware behavior. The Varlock 1Password plugin linked in discussion is useful external batching/caching workaround context, but it does not resolve OpenClaw core's startup failure policy.

## What This Changes

The issue requests a safer gateway startup policy for repeated exec SecretRef failures so 1Password service account rate limits are not exhausted by supervisor restart loops.

## Best Possible Solution

Keep this issue as the canonical tracker and have maintainers choose a generic startup SecretRef failure policy, with focused gateway/secrets tests and no 1Password-specific core behavior unless explicitly approved.

## Reproduction Assessment

Yes, at source level: current main still throws a fatal startup error when required SecretRef preparation fails, and exec provider failures propagate directly into that path. I did not live-reproduce the launchd plus 1Password quota exhaustion loop.

## Solution Assessment

Unclear until maintainer policy is chosen. A bounded generic startup retry or degraded-startup strategy is plausible, but the issue's retryPolicy, circuit breaker, `op run` batching, and rate-limit awareness suggestions have different security and compatibility tradeoffs.
