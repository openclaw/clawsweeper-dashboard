## Summary

Keep open because current main still fail-fasts startup SecretRef activation without retry/backoff, and open PR #56499 already references this issue with closing syntax.

## What This Changes

The issue asks the gateway to avoid supervised crash-loop amplification when exec-backed SecretRef providers such as 1Password fail during startup.

## Best Possible Solution

Review and either land PR #56499 or replace it with an equivalent narrow startup SecretRef retry/backoff fix, while keeping the broader batching model on #72278.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to make prepareRuntimeSecretsSnapshot, or an active exec SecretRef inside it, fail during activateRuntimeSecrets with `reason: "startup"`; current main rejects with the fatal startup error before any retry/backoff path.

## Solution Assessment

Unclear. Startup exponential backoff is a narrow and plausible fix for crash-loop amplification, but maintainers should review PR #56499 against reload last-known-good behavior and leave cross-provider batching/memoization to #72278 unless they choose to combine the work.
