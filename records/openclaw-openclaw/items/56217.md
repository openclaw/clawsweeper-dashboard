## Label Justifications

- `impact:crash-loop`: The reported failure mode is a process manager repeatedly restarting the gateway after fatal startup secret-resolution failures.

## Summary

Keep open: current main and v2026.6.6 still treat startup SecretRef resolution failures as fatal gateway startup errors, and the available linked PRs are not clean landing paths. The broader external-provider batching problem is tracked separately, but this narrower startup crash-loop/backoff behavior remains unresolved.

## What This Changes

The issue requests startup backoff, circuit-breaking, and/or batching for exec-backed SecretRef failures so gateway restarts do not rapidly exhaust 1Password service-account limits.

## Best Possible Solution

Land a fresh, focused core fix in gateway startup that bounds repeated SecretRef provider attempts while preserving a clear operator recovery path, and keep broader external-provider batching in the separate tracker.

## Reproduction Assessment

Yes, source-reproducible. Current main still throws `Startup failed: required secrets are unavailable` when startup secret preparation fails, and adjacent tests assert that fatal wrapping; no live launchd/1Password rate-limit run was performed in this read-only review.

## Solution Assessment

Unclear. Bounded startup retry/backoff is a plausible mitigation, but the best permanent behavior needs maintainer choice on fail-closed versus degraded startup semantics, and the stale linked branch is not the right patch as-is.
