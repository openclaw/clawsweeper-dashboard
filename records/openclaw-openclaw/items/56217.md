## Summary

Keep open: current main and the latest shipped tag still fail fast on startup SecretRef provider errors, and the open closing PR has not landed or cleared its review/proof blockers.

## What This Changes

The issue asks OpenClaw to prevent startup SecretRef provider failures, especially 1Password exec providers, from causing launchd crash-loops and rate-limit exhaustion.

## Best Possible Solution

Land a focused current-main SecretRef startup resilience fix in `createRuntimeSecretsActivator` or a provider-aware resolver, with regression coverage and redacted launchd/1Password-like proof, while keeping broader batching work tracked separately.

## Reproduction Assessment

Yes, source-level: current tests can inject a rejecting `prepareRuntimeSecretsSnapshot` and observe the fatal startup error, and an active failing exec SecretRef provider would hit the same path. I did not run a live launchd plus 1Password loop in this read-only review.

## Solution Assessment

Unclear as currently proposed: the user problem is valid, but maintainers still need to choose the retry boundary and granularity so the fix reduces external calls instead of replaying successful provider groups.
