## Summary

Keep open. Current main still has fail-fast startup SecretRef activation with no gateway startup retry/backoff/circuit-breaker path, and an open PR already references this issue with closing syntax, so cleanup should wait for that PR or an equivalent maintainer-reviewed fix.

## What This Changes

The issue requests safer gateway handling for transient exec SecretRef provider failures so supervisor restarts do not amplify 1Password rate-limit exhaustion.

## Best Possible Solution

Ship or replace a narrow gateway startup SecretRef retry/backoff fix that prevents transient provider failures from becoming supervisor-amplified request loops, while keeping the broader external vault batching/memoization model on #72278 unless maintainers intentionally fold that scope together.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to make prepareRuntimeSecretsSnapshot, or an active exec SecretRef inside it, fail while createRuntimeSecretsActivator runs with reason `startup`; current source and tests show that rejects with the fatal startup message a supervisor can repeatedly restart.

## Solution Assessment

Unclear. In-process startup exponential backoff is a reasonable narrow first fix for crash-loop amplification, but circuit breaking and efficient external-provider batching need separate maintainer design unless PR #56499 is expanded deliberately.
