## Label Justifications

- `P1`: The report describes a real gateway startup crash-loop that can exhaust account-wide external secret-provider limits and block recovery.
- `impact:crash-loop`: The core symptom is a LaunchAgent restart loop after fatal startup SecretRef resolution failures.
- `impact:auth-provider`: The affected surface is SecretRef provider credential resolution, including 1Password service-account backed exec providers.

## Summary

Keep open: current main and the latest release still fail-fast startup SecretRef activation without retry/backoff, and the open closing PR remains unmerged and conflicting. The broader external-provider batching and 1Password `op` cache symptoms are already split into separate open issues.

## What This Changes

The issue asks OpenClaw to prevent launchd-driven gateway crash loops from repeatedly invoking failing 1Password-backed SecretRef exec providers.

## Best Possible Solution

Review and land or replace a focused current-main fix in `createRuntimeSecretsActivator` or the provider-aware resolver path, while keeping batching/memoization on https://github.com/openclaw/openclaw/issues/72278 and `op` cache guidance on https://github.com/openclaw/openclaw/issues/55459.

## Reproduction Assessment

Yes, source-level: inject a rejecting `prepareRuntimeSecretsSnapshot`, or configure an active failing exec SecretRef provider, during `createRuntimeSecretsActivator(..., { reason: "startup" })`; current main calls the resolver once and throws the fatal startup error. I did not run a live launchd plus 1Password service-account loop.

## Solution Assessment

Unclear. Startup retry/backoff is a plausible narrow fix, but maintainers still need to choose whether resilience belongs in startup activation, provider-aware retry/rate-limit handling, or the broader external-provider batching design.
