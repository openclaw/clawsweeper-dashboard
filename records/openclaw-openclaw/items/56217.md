- `impact:crash-loop`: The central failure mode is repeated gateway restarts after fatal startup SecretRef failures under launchd or another process manager.

## Summary

Keep open: current main and the latest release still fail gateway startup when startup SecretRef preparation fails, so a launchd/systemd restart policy can keep re-entering the same exec-provider path. The Varlock 1Password plugin mentioned in discussion is useful workaround context for backend batching/caching, but it does not resolve OpenClaw core's startup failure policy.

## What This Changes

The issue requests bounded gateway SecretRef startup handling so repeated exec secret-provider failures do not exhaust 1Password service-account limits under process-manager restarts.

## Best Possible Solution

Land a fresh, focused core startup SecretRef policy that bounds repeated provider attempts and documents operator recovery, while keeping backend-specific batching and caching in resolver plugins or ClawHub packages.

## Reproduction Assessment

Yes, source-reproducible. Current main still wraps startup SecretRef preparation failures as fatal gateway startup errors, and adjacent tests assert that behavior; no live launchd/1Password rate-limit run was performed.

## Solution Assessment

Unclear. Bounded startup retry/backoff is a plausible mitigation, but maintainers still need to choose the fail-fast versus degraded-startup policy and the linked PRs are not the best fix as-is.
