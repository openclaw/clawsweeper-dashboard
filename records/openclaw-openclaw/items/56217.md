## Summary

Keep open. Current main still has a fail-fast startup SecretRef activation path with no retry, backoff, circuit breaker, or rate-limit-aware handling, and open PR #56499 already references this issue with closing syntax, so cleanup should wait for that PR or an equivalent maintainer-reviewed fix.

## What This Changes

The issue asks OpenClaw to prevent failing 1Password-backed exec SecretRefs from repeatedly aborting gateway startup under launchd restart supervision and exhausting external rate limits.

## Best Possible Solution

Review and either land PR #56499 or replace it with an equivalent narrow gateway startup fix that retries transient SecretRef provider failures in-process without weakening reload last-known-good behavior, while leaving the broader provider batching/memoization design to #72278 unless maintainers intentionally fold the tracks together.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to make prepareSecretsRuntimeSnapshot, or an active exec SecretRef inside it, fail during activateRuntimeSecrets with `reason: "startup"`; current tests and source show that rejects with a fatal startup error that an external supervisor can repeatedly restart.

## Solution Assessment

Unclear. Startup exponential backoff is a reasonable narrow first fix for crash-loop amplification, but the full requested outcome also needs a generic external-secret-provider batching or memoization model rather than a 1Password-only workaround.
