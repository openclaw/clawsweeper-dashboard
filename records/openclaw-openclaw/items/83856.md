## Summary

Keep open: current `main` and the latest release still leave `cron show` pagination unbounded, while the sibling edit lookup already has the missing guard pattern.

## What This Changes

The issue requests bounding `openclaw cron show` pagination when resolving a job by id/name so non-advancing `cron.list` results fail instead of looping.

## Best Possible Solution

Add the same bounded-page and advancing-offset checks to `loadCronJobForShow`, with focused CLI regression tests for non-advancing and excessive `cron.list` pagination.

Yes, at source level: a mocked `cron.list` response with `hasMore: true` and unchanged `nextOffset` keeps `loadCronJobForShow` on the same offset forever. I did not run that repro because this review is read-only.

## Solution Assessment

Yes; mirroring the existing `cron edit` pagination guard in the `cron show` lookup is the narrowest maintainable fix and does not require changing the gateway contract.
