## Summary

Keep this PR open: the optimistic-send direction is the right owner path for the delayed Control UI rendering bug, but the patch leaves a session-switch rollback hole that can hide retry state after a failed or reconnecting send.

## What This Changes

This PR moves Control UI queued sends into `chatMessages` before the `chat.send` ACK, adds a submitted-send marker to avoid duplicate thread rendering, and extends unit/browser E2E coverage for delayed ACKs.

## Best Possible Solution

Keep the optimistic append in `sendQueuedChatMessage`, but make the submitted-send marker lifecycle session-scoped/cache-aware and land one canonical fix for https://github.com/openclaw/openclaw/issues/94479.

## Reproduction Assessment

Yes. Current main appends the user message only after `requestChatSend()` resolves, and the existing delayed-ACK test path expects `chatMessages` to remain empty before ACK; I did not run tests because this review was read-only.

## Solution Assessment

No, not yet. Moving the append earlier is the right source-of-truth fix, but the submitted-send rollback/cleanup must also handle the original cached session when the user switches away before failure or reconnect handling completes.
