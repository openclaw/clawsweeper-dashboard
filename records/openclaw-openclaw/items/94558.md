## Summary

Keep open. The PR fixes the common delayed-ACK rendering path, but the submitted-send marker cleanup is only visible-session aware, so a session switch before failure or reconnect can leave stale optimistic transcript state and hide the queued retry row.

## What This Changes

The PR changes Control UI chat sends to append the submitted user turn before the Gateway ACK and suppress the matching pending-send thread item while that optimistic message is present.

## Best Possible Solution

Keep the early visible send behavior, but make the optimistic marker lifecycle session-cache aware so failed or reconnecting sends restore the original session transcript and leave retry state visible.

## Reproduction Assessment

Yes. Current main appends the user message only after the delayed `chat.send` ACK, and the PR-specific session-switch edge is source-reproducible because session switching caches `chatMessages` before the ACK settles.

## Solution Assessment

No, not yet. Appending before ACK is the right owner-boundary fix, but the PR needs session-cache-aware cleanup before it is the best fix.
