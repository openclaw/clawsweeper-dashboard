## Summary

Keep open. Current main still does not provide a cron job tags/categories contract; the relevant persistence model, Gateway validation and list filtering, CLI output/options, agent cron tool schema, Control UI filters, and cron docs all remain name/description/status/schedule oriented. The prior ClawSweeper keep-open comment is still consistent with the current source pass.

## What This Changes

The issue requests optional freeform cron job tags for organization, filtering, export/upgrade preservation, and CLI/UI/API visibility.

## Best Possible Solution

Ship an additive base cron tags contract across the existing cron surfaces: persisted `tags?: string[]`, create/update normalization, Gateway schema and list filtering, CLI add/edit/list/show support, agent cron tool support, Control UI create/edit/list/filter controls, focused tests, docs, and a user-facing changelog entry. Leave bulk tag actions as a follow-up after the base contract lands.

## Reproduction Assessment

Not applicable as a runtime bug, but the feature gap is high-confidence: current source shows no persisted `tags` field, Gateway schemas reject unknown tag fields, and list/UI filtering has no tag predicate. A concrete check would be passing `tags` to `cron.add` or `cron.update`, which current validation would reject because the schemas use `additionalProperties: false`.

## Solution Assessment

Yes, the narrow maintainable solution is an additive core cron job metadata field with filtering on the existing Gateway/CLI/UI/tool surfaces. Name prefixes remain a workaround, but they do not solve structured filtering or stable metadata without renaming jobs.
