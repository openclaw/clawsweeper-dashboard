## Summary

Keep open. Current main still lacks a structured cron job tags/categories contract across persistence, Gateway validation/filtering, agent tool schema, CLI, Control UI, tests, and docs, and the related Reminders-tab issue depends on this metadata rather than superseding it.

## What This Changes

Add optional freeform `tags: string[]` metadata to cron jobs for organization, filtering, display, and eventual tag-based operations.

## Best Possible Solution

Add a backward-compatible cron tags contract end to end, with `tags?: string[]` persisted and normalized, Gateway add/update/list schemas plus tag filtering, service filtering, CLI and agent-tool support, Control UI create/edit/list/filter controls, tests, docs, and a user-facing changelog entry.

## Reproduction Assessment

Not applicable as a runtime bug, but the feature gap is high-confidence from source inspection. Current main has no persisted `tags` field or tag filter, and the closed Gateway schemas would reject `tags` on add/update/list instead of storing or filtering it.

## Solution Assessment

Yes. The requested additive metadata field is the narrow maintainable path for stable grouping and filtering; name prefixes remain a workaround but do not provide structured API/UI filtering without renaming jobs.
