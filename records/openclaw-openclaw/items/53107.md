## Summary

Keep open. Current main still lacks a persisted and public cron tags/categories contract, and the adjacent plugin-owned scheduler tag cleanup is name-encoded rather than the requested general job metadata and filtering surface.

## What This Changes

Request adds optional freeform tags/categories to cron jobs for organization, filtering, UI/CLI display, and possible bulk tag operations.

## Best Possible Solution

Add a backward-compatible, normalized `tags?: string[]` cron metadata contract across storage, Gateway validation/filtering, service list filtering, agent tool, CLI, Control UI, docs, and focused tests, while deferring bulk operations unless accepted in the design.

## Reproduction Assessment

Not applicable as a runtime bug, but the feature gap is source-reproducible: current main has no persisted cron tags or tag filter in the model, Gateway schema, service list path, agent tool, CLI, or UI.

## Solution Assessment

Yes at the direction level: optional normalized tags are safer than name prefixes, but the exact public metadata/filter/UI and bulk-operation contract needs maintainer alignment before implementation.
