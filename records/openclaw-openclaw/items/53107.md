## Summary

Keep open. Current main still does not provide a cron job tags/categories contract; the core model, Gateway schema/list API, service filtering, agent tool schema, CLI commands, Control UI filters, tests, and docs are still organized around name/description/enabled/status/schedule fields rather than structured job tags.

## What This Changes

The issue requests an optional freeform `tags` array on cron jobs, with tags surviving persistence/exports and appearing in list/filter UI, CLI, and API surfaces.

## Best Possible Solution

Add a backwards-compatible cron tags contract end to end: persisted `tags?: string[]` with normalization, Gateway add/update/list schemas and `tag` filtering, service filtering, CLI add/edit/list/show support, agent cron tool support, Control UI create/edit/list/filter controls, focused tests, docs, and a user-facing changelog entry. Leave bulk tag enable/disable as a follow-up unless it stays small and API-safe.

## Reproduction Assessment

Not applicable as a runtime bug, but the feature gap is high-confidence. Current source shows no persisted `tags` field or tag filter, and the closed Gateway schemas would reject a `tags` field on add/update/list rather than storing or filtering it.

## Solution Assessment

Yes. The requested additive metadata field is the narrow maintainable solution for stable grouping and filtering; name prefixes remain a workaround, but they do not provide structured API/UI filtering without renaming jobs.
