## Summary

Keep open. Current main still lacks a persisted, public cron tags/categories contract; the only tag-like cron path is a bundled-plugin name-prefix cleanup helper, not user-visible job metadata or filtering.

## What This Changes

The issue requests optional freeform cron job tags/categories that persist across upgrades and appear in cron list, API, and UI filtering surfaces.

## Best Possible Solution

Add a backward-compatible normalized cron metadata contract, expose typed tag filtering through Gateway, service, agent tool, CLI, UI, and docs, and keep bulk operations as a separate product decision.

## Reproduction Assessment

Not applicable as a runtime bug, but yes as a source-level feature gap: current main has no persisted cron tags, tag filter, CLI tag option, or UI tag filter. The existing bundled-plugin cleanup tag is encoded into job names and does not satisfy the requested public contract.

## Solution Assessment

Yes at the direction level: optional normalized tags are safer than name prefixes for organization and filtering. The exact metadata, query, UI, export, and bulk-operation semantics need maintainer product/protocol alignment before a repair lane.
