Full review comments:

- **[P1] Pin Rastermill through a reviewed release:** `package.json:1841`
  - body: This repins a root runtime dependency to a GitHub codeload tarball while the Rastermill package still reports version `0.2.0` and carries install-time build scripts. OpenClaw packages should not ship this image-processing dependency from an unreleased source tarball without explicit maintainer approval because installs lose normal registry provenance and can execute a different package build path.
  - confidence: 0.88
- **[P1] Preserve the backend override policy:** `src/media/image-ops.ts:48-61`
  - body: `createImageProcessor()` no longer passes the current `OPENCLAW_IMAGE_BACKEND` option or the Photon-first routing for known safe in-process formats, and the updated adapter test now asserts `env` is absent. That changes an existing operator/security control into unconditional `execution: "auto"`, so users who selected a backend or relied on avoiding native tools can get different execution and failure behavior after upgrade.
  - confidence: 0.9
- **[P2] Keep EXIF normalization best-effort:** `src/media/image-ops.ts:95-105`
  - body: `normalizeExifOrientation()` now calls `probe()`/`encode()` without preserving the current unavailable-backend catch. This helper is exported through the media runtime surface, so callers that use EXIF normalization as a best-effort cleanup can now fail hard instead of receiving the original bytes when Rastermill cannot process the image.
  - confidence: 0.86
- **[P2] Swap limits for EXIF-rotated managed images:** `src/gateway/managed-image-attachments.ts:209-215`
  - body: The PR swaps metadata for EXIF orientations 5-8, but `resizeManagedImageBufferToLimits()` still passes the original `maxWidth` and `maxHeight` axes to Rastermill. Under asymmetric managed-image limits, a rotated JPEG can be shrunk against the wrong box and still be rejected even though it could fit the displayed dimensions.
  - confidence: 0.83
