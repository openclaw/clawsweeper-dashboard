## Summary

Keep open: current main and latest release still lack INFO-level local embedding worker spawn/exit or per-call timing telemetry, and the remaining decision is the logging/privacy/measurement contract rather than whether the gap exists.

## What This Changes

The issue requests INFO-level memory-core local embedding worker spawn/exit and timing telemetry for CPU forensics.

## Best Possible Solution

Define a bounded memory-core worker telemetry contract with redacted caller attribution, duration/count fields, and clear measurement limits, then implement it inside the memory plugin without broadening core behavior.

## Reproduction Assessment

Yes. Source inspection of current main and v2026.6.10 shows the worker fork/exit path has no INFO lifecycle logs, while query and batch starts are only debug-level; I did not run a live GGUF worker in this read-only review.

## Solution Assessment

Unclear until maintainers choose the logging contract. The direction is useful, but the best implementation should avoid raw session-key exposure and should define which duration, CPU, RSS, and embedding-count fields are reliable.
