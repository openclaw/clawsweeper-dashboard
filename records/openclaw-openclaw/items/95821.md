## Summary

Keep open: current main and the latest release still fork the local embedding worker without INFO-level spawn/exit, duration, caller, or resource telemetry, so the operator forensics gap remains real but needs a maintainer logging/privacy contract first.

## What This Changes

Request INFO-level local memory embedding worker spawn, exit, per-call timing, and optional progress telemetry for CPU-forensics traceability.

## Best Possible Solution

Define bounded memory-core local-worker telemetry that records spawn, exit, duration, and embedding counts with redacted caller attribution and only well-supported resource fields.

## Reproduction Assessment

Yes. The gap is source-reproducible: current main and v2026.6.10 fork and close the worker without INFO lifecycle logs, while embedding calls only DEBUG-log starts; I did not run a live local GGUF model in this read-only review.

## Solution Assessment

Unclear as written. Worker lifecycle telemetry is the right product direction, but the final shape should avoid raw session-key logging and should define which duration, CPU, and RSS measurements are reliable.

## Risks / Open Questions

- Logging raw session keys at INFO could expose sensitive operator or workspace identifiers unless the caller field is redacted or tokenized.
- CPU time and peak RSS fields need a trustworthy measurement source; otherwise INFO logs may imply precision that the parent process cannot reliably provide.
- Default-ON INFO logging for every local embedding call could increase journal volume on heavy ingest hosts, so maintainers should choose the default and any opt-in knob deliberately.
