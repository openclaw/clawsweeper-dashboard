## Summary

Keep open: current main and v2026.6.9 still fork the local embedding worker without INFO-level spawn/exit or duration telemetry; the request is a valid memory-core observability gap, but the exact INFO payload needs a privacy and measurement contract before implementation.

## What This Changes

Request INFO-level local embedding worker lifecycle and per-call CPU/duration attribution logs for memory-core local GGUF embeddings.

## Best Possible Solution

Define and implement bounded memory-core local-worker telemetry that gives operator CPU attribution without raw session identifiers or unnecessary new config surface.

## Reproduction Assessment

Yes, source-reproducible for the reported log gap: current main and v2026.6.9 fork and close the worker without INFO lifecycle logs, while embedding calls only DEBUG-log starts. I did not run a live local GGUF model in this read-only review.

## Solution Assessment

Unclear until maintainers choose the logging contract: the requested direction is useful, but the best implementation should avoid raw session-key exposure and should define which duration, CPU, and RSS fields are trustworthy.
