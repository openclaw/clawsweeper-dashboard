## Summary

Keep open: current main still lacks a clear bounded cleanup path for Codex-owned Chrome DevTools MCP sidecars, and the report is distinct from the already-closed PI/browser MCP cleanup path. The FSB comment is useful prior art for a shared browser bridge, but it does not make the OpenClaw Codex lifecycle issue implemented.

## What This Changes

The issue reports that repeated Codex app-server runs leave `chrome-devtools-mcp` process trees alive under the gateway cgroup until a full gateway restart clears them.

## Best Possible Solution

Define and test an explicit Codex app-server thread/MCP lifecycle that preserves native resume and Computer Use while bounding or reaping Chrome DevTools MCP sidecars; the FSB shared browser bridge is useful prior art for the product direction.

## Reproduction Assessment

No, we do not have a high-confidence live current-main reproduction. The issue is source-reproducible with medium confidence from the shared app-server/thread lifecycle code, upstream Codex delayed unload behavior, and the reporter's detailed systemd/process-tree evidence.

## Solution Assessment

Unclear. Bounding Codex-native MCP sidecars is the right outcome, but maintainers need to choose the lifecycle boundary between per-thread unsubscribe, app-server retirement, upstream MCP pooling, or a shared browser bridge rather than applying a blind process sweep.
