## Summary

Keep open: current main still shares a long-lived Codex app-server connection, auto-attaches Codex threads upstream, and does not show a main-turn `thread/unsubscribe` or other bounded cleanup path for Codex-owned Chrome DevTools MCP sidecars. The Linux report and macOS follow-up provide concrete process/RSS evidence, while `@openai/codex` 0.132.0 only proves MCP process termination when the upstream MCP client/manager is actually shut down.

## What This Changes

The issue requests bounding or reaping Codex app-server Chrome DevTools MCP sidecars so repeated embedded Codex runs do not accumulate gateway child processes and memory.

## Best Possible Solution

Define an explicit Codex app-server thread/MCP lifecycle that preserves native resume while ensuring Chrome DevTools MCP sidecars are reused, unsubscribed, or reaped with focused tests and a real repeated-turn gateway proof.
