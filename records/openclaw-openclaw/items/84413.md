## Summary

Keep open: current main still runs embedded Codex turns through a shared app-server client and does not show a main-turn cleanup path that unsubscribes or retires Codex-owned Chrome DevTools MCP sidecars. The reporter supplied concrete Linux process/cgroup evidence, a later macOS confirmation, and related-but-distinct lifecycle reports remain open.

## What This Changes

The issue asks OpenClaw to stop Codex app-server Chrome DevTools MCP sidecars from accumulating across embedded Codex runs, either by bounded reuse or per-run teardown.

## Best Possible Solution

Define an explicit Codex app-server thread/MCP lifecycle for OpenClaw that preserves native resume while bounding Chrome DevTools MCP sidecar count and proving cleanup with repeated real gateway runs.

## Reproduction Assessment

No, this review did not establish a high-confidence live current-main reproduction. It is source-reproducible with medium confidence from the shared app-server/thread lifecycle plus detailed Linux and macOS process evidence in the issue discussion.

## Solution Assessment

Unclear; bounding Codex-native MCP sidecars is the right end state, but maintainers need to choose the lifecycle boundary before a safe fix. A blind gateway process reaper is riskier than an ownership-aware app-server/thread/MCP cleanup contract.
