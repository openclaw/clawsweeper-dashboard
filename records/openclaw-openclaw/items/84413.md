## Label Justifications

- `P1`: The report describes real gateway cgroup memory growth, event-loop pressure, and operational restarts on the current Codex app-server release path.
- `impact:crash-loop`: Accumulating unreaped sidecar processes can drive gateway memory and availability pressure even when the main Node heap is smaller.

## Summary

Keep open: current main still has a long-lived shared Codex app-server path with no ordinary per-run unsubscribe or sidecar retirement, and the issue now has Linux plus macOS process evidence. The prior configured-MCP cleanup in https://github.com/openclaw/openclaw/issues/62731 does not cover this Codex app-server/native MCP path.

## What This Changes

The issue asks OpenClaw to stop Codex app-server Chrome DevTools MCP sidecars from accumulating across embedded Codex runs and growing gateway cgroup memory.

## Best Possible Solution

Define and test an explicit Codex app-server thread/MCP lifecycle that bounds or reaps Chrome DevTools MCP sidecars while preserving native resume and Computer Use behavior.

## Reproduction Assessment

No high-confidence current-main live reproduction was established. Source inspection plus Linux and macOS 2026.5.18 process evidence give a plausible reproduction path through repeated embedded Codex/Computer Use turns that spawn Chrome DevTools MCP sidecars.

## Solution Assessment

Unclear. Bounding Codex-native MCP sidecars is the right outcome, but maintainers need to choose whether the boundary is per-turn unsubscribe, thread retirement, shared sidecar pooling, upstream Codex behavior, or a gateway orphan reaper.
