## Summary

Keep open: current main still has a shared Codex app-server path where ordinary completed embedded runs clean up local listeners/timers but do not explicitly unsubscribe or retire Codex-owned Chrome DevTools MCP sidecars. The Linux report plus the later macOS comment give concrete v2026.5.18 process evidence, and the related issues cover adjacent but distinct lifecycle gaps rather than a merged fix for this path.

## What This Changes

The issue asks OpenClaw to stop Codex app-server embedded runs from accumulating unreaped `chrome-devtools-mcp` sidecar process trees under the gateway.

## Best Possible Solution

Define explicit Codex app-server MCP sidecar ownership that bounds `chrome-devtools-mcp` processes while preserving native thread resume/tool continuation, then prove it with repeated real gateway/Codex runs on Linux and macOS.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection shows the main Codex app-server path lacks ordinary per-run sidecar cleanup, and the reporter plus macOS commenter provide concrete process evidence on v2026.5.18.

## Solution Assessment

Unclear; bounding the sidecars is the right outcome, but the safest implementation needs maintainer choice between Codex thread unsubscribe/retirement, app-server MCP pooling, upstream behavior, or a gateway-side reaper.
