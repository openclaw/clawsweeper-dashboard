## Summary

Keep open: current main still keeps a shared Codex app-server client and starts/resumes Codex threads with MCP fingerprints, but I did not find a normal turn-completion path that unsubscribes or bounds Codex-owned Chrome DevTools MCP sidecars. The report has concrete process/RSS evidence and remains distinct from the broader open MCP lifecycle reports.

## What This Changes

The issue reports unreaped Chrome DevTools MCP sidecars spawned by embedded Codex app-server runs causing gateway cgroup memory growth on OpenClaw 2026.5.18.

## Best Possible Solution

Define and test an explicit Codex app-server thread/MCP lifecycle that preserves native resume while bounding or reaping Chrome DevTools MCP sidecars after completed embedded runs.

## Reproduction Assessment

No high-confidence live current-main reproduction was established. Source inspection shows the shared app-server/thread lifecycle still lacks a normal per-turn cleanup path for Codex-owned MCP sidecars, and the reporter supplied detailed systemd/process/RSS evidence.

## Solution Assessment

Unclear as an implementation path. Bounding or retiring Codex-native MCP sidecars is the right end state, but the safest fix needs an explicit lifecycle decision at the Codex app-server thread/MCP boundary rather than a broad gateway process sweep.
