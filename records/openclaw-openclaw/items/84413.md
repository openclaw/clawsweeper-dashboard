## Summary

Keep open: the report has concrete Linux and macOS process evidence, and current main still does not show a normal-turn Codex app-server MCP sidecar cleanup or bounded reuse path for Chrome DevTools MCP children. The related bundled MCP issues and PRs are useful context, but they do not cover the Codex app-server-owned native MCP lifecycle described here.

## What This Changes

The issue reports that repeated `openai-codex/gpt-5.5` embedded runs leave multiple `chrome-devtools-mcp` sidecars alive under the gateway, growing service cgroup memory until manual process or gateway cleanup.

## Best Possible Solution

Define an explicit Codex app-server MCP lifecycle that either reuses a bounded keyed sidecar pool or disposes native MCP sidecars when their owning run/thread is no longer active, with live gateway proof that repeated Chrome DevTools MCP turns keep process count and RSS bounded.

## Reproduction Assessment

No high-confidence local reproduction was run in this review. The issue is source-reproducible with medium confidence from the shared app-server lifecycle plus detailed Linux and macOS process evidence in the discussion.

## Solution Assessment

Unclear as an implementation choice. Bounding Codex-native MCP sidecars is the right outcome, but maintainers need to decide whether ownership belongs in OpenClaw's app-server adapter, a keyed shared pool, an explicit unsubscribe/retire call, or upstream Codex.
