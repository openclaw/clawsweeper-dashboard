## Summary

Keep open: current main still runs Codex turns through a shared app-server client and does not show a main-turn unsubscribe, retirement, or bounded pooling path for Codex-owned Chrome DevTools MCP sidecars. The Linux report and the later macOS confirmation are concrete, and the FSB bridge comment is useful prior art for a shared-browser-service direction rather than evidence that OpenClaw core is already fixed.

## What This Changes

The issue asks OpenClaw to prevent repeated embedded Codex runs from accumulating unreaped `chrome-devtools-mcp` sidecar process trees under the gateway.

## Best Possible Solution

Define and test an explicit Codex app-server thread/MCP lifecycle that preserves native resume while bounding or reaping Chrome DevTools MCP sidecars, with real gateway proof that repeated completed turns do not grow sidecar count or RSS.

## Reproduction Assessment

No, this review did not establish a high-confidence live current-main reproduction. It is source-reproducible with medium confidence from the shared app-server/thread lifecycle code plus the reporter's Linux and macOS process evidence.

## Solution Assessment

Unclear; bounding Codex-native MCP sidecars is the right outcome, but maintainers need to choose the lifecycle boundary before a safe fix. A blind gateway process sweep is riskier than an explicit Codex app-server thread/MCP ownership contract.
