Keep open: this is a draft, maintainer-labeled bug-hardening PR with useful handler-level protection, but current main and the latest release still have the production resolver path that can read unreadable plugin tool descriptors before the MCP handler is constructed.

## What This Changes

The PR changes the plugin-tools MCP handler to skip unreadable or unwrappable plugin tool descriptors while keeping healthy sibling tools listed and callable, with focused regression coverage.

## Best Possible Solution

Land the MCP handler hardening only after the registry-origin resolver path is covered in this branch or explicitly coordinated with the companion resolver-hardening PR.

## Reproduction Assessment

Yes. Current main and v2026.6.8 directly read plugin tool descriptor fields in the MCP handler and plugin resolver, and the PR fixture demonstrates throwing descriptor accessors that would fail without guarding.

## Solution Assessment

No as a complete production fix. Handler filtering is a good mitigation, but the production MCP server resolves registry tools before constructing the handler, so resolver coverage or a named companion merge is still needed.
