## Summary

Keep this issue open. Current main still has the reported bug: session maintenance can delete persisted `status: "running"` sessions unless the session is the single caller-supplied active key, and the directly related fix PR remains open and unmerged.

## What This Changes

Session maintenance in enforce mode should skip active or running sessions when applying age, entry-count, load-time, and disk-budget pruning.

## Best Possible Solution

Keep this as the tracking bug until core session maintenance has one shared protected-session set that includes the caller active key plus currently active/running sessions, and applies that set consistently to save-time stale pruning, entry-count capping, load-time oversized-store cleanup, and disk-budget eviction, with regression tests for multiple running sessions.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to store multiple entries with `status: "running"`, provide only one `activeSessionKey`, and run stale, count, load-time, or disk-budget maintenance; current helpers will still delete non-caller running entries based on age/count/oldest-first ordering.

## Solution Assessment

Yes in direction, but not yet in main. The maintainable fix should be generic session-maintenance protection, not a subagent-only special case, and should rely on existing recovery paths to terminalize truly orphaned running rows.
