## Summary

Keep open: current main still has source-proven enforce-mode maintenance paths that can delete a non-protected running subagent session when it is not the single caller active session key, and the directly related fix PR is closed unmerged.

## What This Changes

Protect active/running subagent session entries from enforce-mode session pruning, count capping, and disk-budget eviction.

## Best Possible Solution

Land a focused core fix that derives protected active subagent child-session keys once and applies them consistently across pruning, capping, disk-budget eviction, explicit cleanup/load maintenance, tests, docs, and changelog.

## Reproduction Assessment

Yes, from source inspection: an old or over-limit non-protected subagent entry with status="running" can be removed unless it is the caller's single activeSessionKey. I did not run a live throughput reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing preserveKeys-style maintenance seam with active subagent keys from the persisted-plus-memory run snapshot, not to add a new retention config or broad policy change.

Implement a narrow core fix using the existing preserveKeys-style seam: derive active subagent child session keys from the persisted-plus-memory subagent run snapshot and/or persisted SessionEntry.status where appropriate, then protect those keys consistently across stale pruning, count capping, disk-budget eviction, and sessions cleanup/load maintenance. Keep core/plugin boundaries intact, avoid static circular imports, and do not add a new config option or broad retention policy. Add regression coverage for stale and over-limit running subagent entries, persisted-only active runs, and disk-budget eviction. Update docs to describe the active-running exception for synthetic subagent entries and add the required CHANGELOG.md fix entry. Do not revive the old in-memory-only helper shape from the closed PR without addressing persisted snapshot visibility.
