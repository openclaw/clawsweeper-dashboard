## Summary

Keep open: current main still routes short Telegram TTS replies as one voice/media send with the visible reply as the caption, and changing that default is a user-facing channel/TTS UX decision rather than an already-implemented fix.

## What This Changes

The issue requests sending auto-TTS output as a pure voice message followed by a separate plain text reply, especially for Telegram with Microsoft Edge TTS.

## Best Possible Solution

Have maintainers decide whether Telegram should match WhatsApp's separate-text policy by default or behind a config knob, then update the Telegram outbound/TTS path with regression coverage that preserves thread, reply, and button behavior.

## Reproduction Assessment

Yes, source-reproducible: current main builds a single text+audioAsVoice payload and Telegram turns short text into the voice/media caption. I did not run a live Telegram send.

## Solution Assessment

Unclear: no patch is proposed, and the requested default behavior is plausible but needs a maintainer choice between changing Telegram semantics globally or adding an opt-in policy.
