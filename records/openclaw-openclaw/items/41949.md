## Summary

Keep this issue open. The latest release and current main already cap ordinary AI snapshots, provide an opt-in efficient snapshot preset, and guard oversized tool results, but they do not implement the central request for compact-by-default, goal-shaped browser reads and small post-action change reports; that remaining behavior needs a measured live reproduction and an explicit maintainer decision about the default contract. Likely related people include Peter Steinberger for the browser snapshot surface and Ted Li and zenglingbiao for recent agent context-engine work.

## What This Changes

Make browser interactions context-budget aware by returning compact, task-relevant page state instead of repeatedly injecting large page snapshots.

## Best Possible Solution

Make compact structured browser state the normal multi-step interaction path after validating representative workflows, preserve an explicit full-snapshot or targeted-detail escape hatch, and return bounded post-action change reports with regression coverage measuring context growth.

## Reproduction Assessment

No—source inspection confirms that ordinary browser snapshots can still contribute up to 40,000 characters per call, but there is no measured current-main run demonstrating context exhaustion after the shipped context guard and efficient mode are considered.

## Solution Assessment

Unclear—the proposed compact default and action-diff direction is maintainable and supported by the linked FSB experience, but maintainers must define which page structure remains available by default and how agents explicitly request full or targeted detail.
