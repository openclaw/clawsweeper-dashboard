## Summary

Keep open. Current main has partial mitigations for browser snapshot size and generic tool-result truncation, but it still does not fully implement the requested browser-specific default of compact, goal-shaped reads/action deltas with an explicit raw escape hatch; the related open browser field-report and compaction issues are adjacent rather than a clean canonical replacement.

## What This Changes

The issue requests browser-specific context-budget handling so repeated browser snapshots and page reads do not crowd out planning context during multi-step automation.

## Best Possible Solution

Keep this as a browser-plugin design item: make compact goal-shaped reads and post-action deltas the normal path, preserve an explicit raw snapshot escape hatch, and validate it alongside existing generic tool-result truncation/session-pruning behavior.

## Reproduction Assessment

No. Source inspection shows repeated browser snapshots can still add large tool-result text despite existing caps, but the issue lacks a concrete site/prompt/log and I did not run a live browser loop on current main.

## Solution Assessment

Unclear. Existing maxChars, efficient mode, session pruning, and tool-result truncation are useful partial mitigations, but the best permanent fix needs a maintainer decision on default browser read/action-diff semantics and raw-output escape hatches.
