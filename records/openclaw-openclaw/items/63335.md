## Summary

Keep open. Current main still permits the reported same-run retry shape: timeout compaction can resubmit the same provider/model under one run, assistant timeout handling rotates profiles before model fallback, and stuck-session recovery observes active embedded runs unless active abort is explicitly enabled.

## What This Changes

The issue asks OpenClaw to prevent a single Telegram assistant turn from repeatedly reissuing near-identical model requests under the same runId while the session remains stuck in processing.

## Best Possible Solution

Add a focused embedded-runner/failover guard that tracks same-run timeout retries by candidate, auth profile, and prompt-size or compaction delta, then advances to the next fallback candidate or returns a clear timeout once the retry budget is exhausted.

## Reproduction Assessment

Yes. The existing timeout-compaction harness can simulate repeated timed-out attempts with high prompt usage under one run, and current tests already assert multiple same-run attempts before timeout handling.

## Solution Assessment

No. Current main bounds some retry paths, but it does not distinguish productive compaction/profile rotation from near-identical same-candidate retry churn; the maintainable fix is a narrow runtime guard with regression coverage.
