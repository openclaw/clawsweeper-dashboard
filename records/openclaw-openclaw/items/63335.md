## Summary

Keep open. Current main still has the same core retry shape: timeout compaction can re-submit the same provider/model under one runId, assistant timeout handling rotates auth profiles before model fallback, and stuck-session recovery observes active embedded runs by default instead of aborting them.

## What This Changes

The issue reports one assistant turn repeatedly reissuing near-identical model requests under a single runId before eventual fallback, leaving the Telegram session stuck in processing.

## Best Possible Solution

Add a narrow embedded-runner guard that tracks same-run timeout retries by provider/model, auth profile, and prompt-size or compaction delta, then advances fallback or returns a clear timeout once that budget is exhausted.

## Reproduction Assessment

Yes. The issue has concrete logs for one runId, and the existing timeout-compaction harness can simulate repeated same-run timeouts with high prompt usage and multiple embedded attempts without needing live provider access.

## Solution Assessment

No. Current main bounds some retry paths, but it does not distinguish productive compaction/profile rotation from near-identical same-candidate churn; the safer fix is a focused runtime guard plus regression coverage.
