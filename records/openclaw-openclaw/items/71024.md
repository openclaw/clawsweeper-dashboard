## Summary

Keep open. The PR is a focused fix for a real Mattermost status bug, but the current branch still has a credential-precedence regression against Mattermost’s strict runtime resolver and lacks after-fix real `openclaw status` proof.

## What This Changes

The PR adds Mattermost read-only account inspection and status metadata for unresolved SecretRef bot tokens, with regression tests for the account inspector and status snapshot path.

## Best Possible Solution

Land the Mattermost-owned read-only inspector/status fix after preserving configured SecretRef precedence over env fallback, covering the SecretRef-plus-env case, and attaching redacted real CLI proof.

## Reproduction Assessment

Yes, from source inspection. The linked issue gives a concrete Mattermost SecretRef config and stack trace, and current main still has no Mattermost `inspectAccount` before the status path falls back to strict `resolveMattermostAccount`; I did not run a live `openclaw status` command in this read-only review.

## Solution Assessment

No. The read-only inspector direction is the right maintainable fix, but this branch lets env fallback override configured-unavailable SecretRef metadata while runtime resolution remains strict; the source/status precedence should be corrected and tested.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve configured SecretRef precedence:** `extensions/mattermost/src/mattermost/accounts.ts:185-198`
  - body: When `botToken` is an unresolved configured SecretRef and `MATTERMOST_BOT_TOKEN` is set, this branch reports `botTokenSource: "env"` and `botTokenStatus: "available"`. Strict `resolveMattermostAccount` still resolves the configured SecretRef first and throws before env fallback, so status/probe can claim credentials are usable for a config runtime rejects. Keep `configured_unavailable` precedence and cover the SecretRef-plus-env case.
