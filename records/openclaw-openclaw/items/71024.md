## Summary

Keep open: the PR targets a valid Mattermost status bug and current main still lacks the read-only inspector, but the head can misreport configured-unavailable SecretRef credentials as env-available and still needs after-fix real behavior proof.

## What This Changes

Adds a Mattermost read-only account inspector/status snapshot path and regression tests for unresolved SecretRef bot tokens.

## Best Possible Solution

Land the Mattermost-owned read-only inspector/status fix after keeping configured SecretRef state ahead of env fallback, adding a SecretRef-plus-env regression test, and attaching redacted real CLI proof.

## Reproduction Assessment

Yes. The linked issue gives a concrete Mattermost SecretRef config and stack trace, and current-main source shows status falling from missing inspectAccount into strict resolveMattermostAccount; I did not run openclaw status in this read-only review.

## Solution Assessment

No. The inspector/status seam is the right maintainable direction, but this implementation should preserve configured SecretRef precedence instead of letting env fallback mask a configured-unavailable credential.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve configured SecretRef precedence:** `extensions/mattermost/src/mattermost/accounts.ts:184-198`
  - body: When a configured botToken SecretRef is unresolved and MATTERMOST_BOT_TOKEN is set, this inspector sets botToken from env and reports botTokenStatus: "available". Strict resolveMattermostAccount resolves the configured token first and throws before env fallback, so status/probe can claim credentials are usable for a runtime config that will fail. Keep the configured-unavailable state ahead of env fallback and cover that mixed-source case.
  - confidence: 0.9
