## Summary

Keep open. The PR addresses a real Mattermost `openclaw status` crash from the linked report and stays in the Mattermost plugin boundary, but the current head still diverges from strict runtime credential semantics when an unresolved config SecretRef and `MATTERMOST_BOT_TOKEN` coexist, and it is missing the required changelog entry.

## What This Changes

This PR adds a Mattermost read-only account inspector, wires it into config/status snapshots, carries `botTokenStatus`, and adds regression tests for unresolved SecretRef bot tokens.

## Best Possible Solution

Land a small Mattermost-owned fix that keeps read-only inspection from dereferencing SecretRefs, preserves credential availability metadata in status snapshots, and keeps read-only metadata aligned with the unchanged strict runtime resolver.

## Reproduction Assessment

Yes. The linked report gives a concrete Mattermost SecretRef config and stack trace, and current main still has the strict resolver/no-inspector shape that explains it. The remaining PR regression is source-reproducible by combining an unresolved config `botToken` SecretRef with `MATTERMOST_BOT_TOKEN`.

## Solution Assessment

No. The inspector/status approach is the right narrow direction, but the current PR should not report env credentials as available when the same configured SecretRef still makes runtime resolution fail; preserving `configured_unavailable` precedence is safer unless runtime fallback semantics change consistently too.
