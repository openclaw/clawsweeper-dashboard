## Summary

Keep this PR open: it addresses a valid fallback retry guard bug, but the patch still trusts a start-of-file 500-record scan and lacks the required real after-fix behavior proof for an external PR.

## What This Changes

The PR changes the attempt-execution session-history guard to track assistant records after the latest user record, handles Claude-style `type: user`/`type: assistant` JSONL records, and adds a fresh-unanswered-user regression fixture.

## Best Possible Solution

Land or replace this with a tail-aware assistant-after-latest-user guard plus focused regression coverage, keep the retry-prompt hook tracked separately, and require redacted real fallback retry proof before merge.

## Reproduction Assessment

Yes. Source inspection of current main shows `jsonlFileHasAssistantMessage` returns true at the first assistant record, so a `user -> assistant -> fresh unanswered user` JSONL transcript demonstrates the bug; a transcript beyond the 500-record cap demonstrates the PR's remaining tail-scan problem.

## Solution Assessment

No. The direction is right, but the implementation should inspect the latest/tail records or fail closed when the cap is hit, and it should not close the broader retry-prompt hook request.
