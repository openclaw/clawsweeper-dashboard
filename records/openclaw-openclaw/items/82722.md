## Summary

Keep open: current main still has a source-visible heartbeat deferral gap for webchat agent runs that are accepted and tracked by Gateway before they enter the embedded session lane; the older busy-lane fix does not cover that active-run window.

## What This Changes

The issue reports that heartbeat polling can still interleave with plain text streaming responses in webchat after the earlier busy-session-lane fix.

## Best Possible Solution

Add a canonical active-run/session-busy predicate that heartbeat can query after resolving the target session, covering Gateway active chat/agent registrations and existing embedded/reply-run state, then retain the current queue and session-lane retry path.

## Reproduction Assessment

Yes, from source: a webchat agent run can be accepted and registered in Gateway before it reaches the embedded session lane, and heartbeat deferral does not inspect that active-run map. I did not establish a live end-to-end streaming interleaving.

## Solution Assessment

Yes, a narrow runtime busy predicate is the best direction; it avoids channel-specific filtering and reuses the existing retryable `requests-in-flight` heartbeat behavior.
