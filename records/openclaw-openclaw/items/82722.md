## Summary

This should stay open: current `main` still has a plausible heartbeat deferral gap for accepted webchat runs before the embedded session lane is populated, and the earlier session-lane fix does not cover that active-run state.

## What This Changes

The issue reports that heartbeat polling can still interleave with plain text model streaming in webchat despite the earlier tool/agent busy-lane fix.

## Best Possible Solution

Teach heartbeat deferral about canonical active chat runs for the target session, then keep the existing queue and session-lane checks as the lower-level guard.

## Reproduction Assessment

Yes, partially from source: a `chat.send` run can be accepted and tracked in Gateway before the embedded session lane is active, while heartbeat deferral only checks queues/session lanes. I did not establish a live end-to-end reproduction of the exact streaming interleaving.

## Solution Assessment

Yes, a narrow fix should carry an active-run/session-busy fact from Gateway into heartbeat deferral instead of adding channel-specific heartbeat policy or relying on UI-side filtering.

## Repair Work Prompt

Fix the heartbeat/webchat active-run race reported in https://github.com/openclaw/openclaw/issues/82722. Current `runHeartbeatOnce` only defers on command queue/session lane state; `chat.send` registers an active run in Gateway before yielding and dispatching the embedded runner, so a heartbeat can start before the session lane is busy. Add a narrow, plugin-agnostic active-session/run busy predicate to heartbeat scheduling, likely by extending `HeartbeatDeps`/`startHeartbeatRunner` and passing Gateway `chatAbortControllers` or an equivalent prepared session-active checker through `activateGatewayScheduledServices`. Keep the existing main queue, cron, pending-final-delivery, and session-lane behavior. Add regression coverage for an accepted webchat/chat.send run with an idle command queue and idle session lane returning `requests-in-flight`, plus any scheduled-services wiring coverage. Do not add a config option, channel-specific heartbeat policy, or UI-only filtering. Add a single-line `CHANGELOG.md` fix entry unless maintainer landing will handle it.
