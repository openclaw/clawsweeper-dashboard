## Summary

Keep open. The Kimi no-key model picker gap is still valid on current main, but this PR should not merge as-is because it uses the deprecated runtime catalog hook, would expose legacy Kimi IDs after rebase, and duplicates Moonshot rows that current main already serves from the manifest catalog.

## What This Changes

The PR adds `augmentModelCatalog` hooks and tests for Kimi and Moonshot, and exports Moonshot's model catalog constant.

## Best Possible Solution

Add a Kimi manifest/control-plane `modelCatalog` row for `kimi-for-coding`, keep legacy Kimi ids as normalization-only compatibility, and leave Moonshot on its existing manifest catalog path.

## Reproduction Assessment

Yes. By source inspection, Kimi returns no runtime catalog without an API key and its manifest has no `modelCatalog` rows, while Moonshot no longer reproduces because current main already has static manifest rows.

## Solution Assessment

No. The PR uses deprecated runtime augmentation hooks and would publish stale or legacy picker rows; the narrower maintainable fix is a Kimi-only manifest catalog update with focused picker coverage.
