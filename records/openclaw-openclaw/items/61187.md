## Summary

Keep open: the Kimi no-key model picker bug is still source-reproducible on current main, but this PR is not the right merge as-is because current main moved bundled providers toward manifest/control-plane catalog rows and Moonshot already has that path.

## What This Changes

The PR adds runtime `augmentModelCatalog` hooks for Kimi and Moonshot, exports Moonshot catalog data, and adds provider hook tests.

## Best Possible Solution

Expose only the canonical Kimi no-key picker row through the manifest/control-plane model catalog, leave Moonshot on its existing manifest rows, and keep legacy Kimi ids as normalization-only compatibility.

## Reproduction Assessment

Yes, by source inspection: Kimi returns no runtime catalog without an API key and its manifest has no `modelCatalog` rows, so the no-key preferred-provider picker has nothing Kimi-specific to list. The Moonshot half no longer reproduces on current main because static manifest rows already feed the picker.

## Solution Assessment

No. The PR uses deprecated runtime augmentation hooks and publishes Kimi legacy aliases as picker choices; the narrower maintainable fix is a Kimi-only manifest catalog row with Moonshot left unchanged.
