## Summary

Keep open: the Kimi picker bug is valid, but the branch is not correct as-is because it uses the deprecated runtime catalog hook, exposes legacy Kimi aliases as primary picker choices, duplicates Moonshot manifest rows, and adds a stale test helper import.

## What This Changes

The PR adds Kimi and Moonshot runtime `augmentModelCatalog` hooks, exports Moonshot catalog data, and adds hook-level tests for provider model picker rows.

## Best Possible Solution

Add Kimi manifest `modelCatalog` metadata for the canonical `kimi-for-coding` row, keep legacy Kimi ids as normalization compatibility, rely on Moonshot's manifest catalog, and cover the picker path with focused tests.

## Reproduction Assessment

Yes, from source inspection rather than a live run: Kimi has no manifest `modelCatalog` and its runtime catalog returns `null` without an API key. The Moonshot half does not reproduce on current main because Moonshot already has static manifest rows.

## Solution Assessment

No. The PR uses deprecated `augmentModelCatalog`, adds obsolete Moonshot runtime catalog rows, and exposes Kimi legacy aliases as primary choices; the narrow maintainable fix is manifest-backed Kimi metadata for the canonical model only.
