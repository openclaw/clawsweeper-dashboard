## Summary

Keep open: the Kimi picker bug is still valid on current main, but this PR should not merge as-is because it uses a deprecated runtime catalog hook, advertises legacy Kimi aliases as primary picker choices after rebasing, and adds obsolete Moonshot catalog plumbing that current main already covers through the manifest.

## What This Changes

The PR adds Kimi and Moonshot `augmentModelCatalog` hooks, exports Moonshot catalog data for the hook, and adds hook-level tests.

## Best Possible Solution

Add Kimi manifest `modelCatalog` metadata for the canonical `kimi-for-coding` row, keep legacy Kimi IDs as normalization-only compatibility, rely on the existing Moonshot manifest catalog, and cover the picker path with focused tests.

## Reproduction Assessment

Yes, from source inspection rather than a live run: Kimi has no no-key manifest catalog rows and its runtime catalog returns `null` without an API key. The Moonshot half does not reproduce on current main because Moonshot already has static manifest catalog rows.

## Solution Assessment

No. The PR's runtime-hook approach is not the best fix because `augmentModelCatalog` is deprecated for this use, Moonshot is already manifest-backed, and Kimi should expose a canonical manifest row instead of legacy aliases.
