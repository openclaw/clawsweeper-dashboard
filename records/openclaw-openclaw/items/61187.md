## Summary

Keep open: the Kimi picker bug is still valid on current main, but this PR is not mergeable as-is because it exposes legacy Kimi aliases, adds an obsolete Moonshot runtime catalog path, and introduces a test helper import that is not present on current main.

## What This Changes

Adds `augmentModelCatalog` hooks for Kimi and Moonshot, exports Moonshot catalog data, and adds tests for those provider catalog hooks.

## Best Possible Solution

Keep Moonshot on the manifest/static catalog path, add a Kimi no-key picker source for only `kimi/kimi-for-coding`, and cover that behavior with current SDK-runtime tests.

## Reproduction Assessment

Yes. Source inspection plus the PR screenshots give a high-confidence path: run setup, choose Kimi before credentials are configured, and open the provider-scoped default model picker; current main returns no Kimi catalog without an API key.

## Solution Assessment

No. The PR addresses a real Kimi picker gap, but the maintainable fix is Kimi-only stable static/model-catalog exposure while leaving Moonshot on its existing manifest-backed static catalog path.
