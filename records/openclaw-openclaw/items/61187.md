## Summary

Keep open: the PR still addresses a real Kimi setup model-picker gap, but current main has superseded the Moonshot half and changed Kimi model/test contracts enough that the branch would be incorrect as-is.

## What This Changes

The branch adds Kimi and Moonshot `augmentModelCatalog` hooks plus plugin tests so setup model picker rows appear before API-key-backed catalog discovery.

## Best Possible Solution

Keep Moonshot on the manifest/static catalog path, and add Kimi no-key picker support for only the current stable `kimi-for-coding` row with SDK-surface regression coverage.

## Reproduction Assessment

Yes by source inspection: current Kimi returns no catalog without an API key and has no manifest `modelCatalog`, so the provider-scoped setup picker has no Kimi rows on current main. The Moonshot side is no longer reproducible because current main already has manifest/static Moonshot rows.

## Solution Assessment

No. The Kimi intent is the right narrow fix, but the branch should expose only the stable Kimi row, use the SDK plugin test runtime, and drop the obsolete Moonshot augment/export/test changes.
