## Summary

Keep open: the Kimi no-key model picker gap is real and the PR includes useful visual proof, but the branch currently fixes it through deprecated runtime hooks, advertises legacy Kimi aliases, adds obsolete Moonshot catalog churn, and includes a test import that does not exist on current main.

## What This Changes

The PR adds `augmentModelCatalog` hooks and hook tests for Kimi and Moonshot provider model picker rows, and exports Moonshot catalog data for the new hook.

## Best Possible Solution

Add Kimi manifest `modelCatalog` metadata for the canonical `kimi-for-coding` row, keep legacy ids as normalization compatibility, leave Moonshot on its manifest catalog, and cover the picker path with focused tests.

## Reproduction Assessment

Yes for the Kimi half: current-main source shows the preferred-provider picker needs static manifest rows before runtime fallback, while Kimi has no manifest catalog and returns no runtime catalog without an API key. The Moonshot half does not reproduce on current main because Moonshot already has static manifest catalog rows.

## Solution Assessment

No. The PR fixes the visible symptom through deprecated `augmentModelCatalog` hooks and Moonshot duplication; the narrower maintainable fix is manifest-backed Kimi metadata only.
