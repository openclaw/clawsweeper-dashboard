## Summary

Keep open with changes required: the Kimi picker problem is still source-reproducible on current main, but the branch uses deprecated runtime catalog hooks, publishes legacy Kimi aliases as primary picker choices, and adds redundant Moonshot catalog behavior that current main already handles through the manifest path.

## What This Changes

The branch adds Kimi and Moonshot runtime `augmentModelCatalog` hooks, exports Moonshot catalog data, and adds provider hook tests.

## Best Possible Solution

Expose the stable Kimi no-key picker row through manifest/control-plane `modelCatalog`, leave Moonshot on its existing static manifest rows, and keep Kimi legacy ids as normalization-only compatibility.

## Reproduction Assessment

Yes by source inspection. Current main returns no Kimi runtime catalog without an API key and has no Kimi manifest `modelCatalog` rows, while Moonshot no longer reproduces because static manifest rows already feed the picker.

## Solution Assessment

No. The PR's runtime hook approach is deprecated for bundled providers and duplicates Moonshot's manifest-backed rows; the narrower maintainable fix is a Kimi-only manifest catalog update with focused tests.
