## Summary

Keep open: the Kimi picker bug is real, but this PR is not mergeable as-is because it promotes legacy Kimi model ids, re-adds Moonshot catalog rows that current main already serves from manifest metadata, and uses a stale plugin test helper import.

## What This Changes

The branch adds Kimi and Moonshot `augmentModelCatalog` hooks, exports Moonshot catalog data, and adds provider catalog hook tests.

## Best Possible Solution

Use the current manifest/control-plane catalog path to expose only Kimi's stable no-key picker row, leave Moonshot on its existing static manifest catalog, and keep regression tests on the public SDK test runtime.

## Reproduction Assessment

Yes. Source inspection shows Kimi has no no-key catalog source on current main, and the PR screenshots exercise the setup picker path; the Moonshot half no longer reproduces on main because static manifest rows are already present.

## Solution Assessment

No. The PR points at the right Kimi symptom, but the maintainable fix is Kimi-only stable catalog exposure through the current catalog surface, with Moonshot left on manifest rows and tests updated to current SDK helpers.

Overall confidence: 0.92

Full review comments:

- **[P2] Expose only the stable Kimi picker row:** `extensions/kimi-coding/index.ts:98-108`
  - body: This hook maps every model from `buildKimiCodingProvider()`, which includes legacy `kimi-code` and `k2p5` aliases. Current main normalizes those aliases to `kimi-for-coding`, so setup should not promote the legacy ids as primary model choices.
  - confidence: 0.91

  - body: Moonshot already has static manifest catalog rows, including the current `kimi-k2.6` default, and the picker reads those rows before runtime fallback. Adding this runtime hook from an older catalog path can duplicate or drift from the manifest-backed picker rows.
  - confidence: 0.92
- **[P2] Use the SDK plugin test runtime:** `extensions/kimi-coding/index.test.ts:2`
  - body: The new test imports `registerSingleProviderPlugin` from `../../test/helpers/plugins/plugin-registration.js`, which is not present on current main. Bundled provider tests import this helper from `openclaw/plugin-sdk/plugin-test-runtime`, so this will fail after rebase unless it uses the public SDK helper.
  - confidence: 0.93
