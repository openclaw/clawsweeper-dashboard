## Summary

Keep open: Kimi still needs a no-key model-picker catalog fix on current main, but this PR is not mergeable as-is because it uses deprecated runtime catalog augmentation, exposes Kimi legacy aliases as primary picker choices, and changes Moonshot despite current manifest rows already covering that provider.

## What This Changes

The branch adds Kimi and Moonshot `augmentModelCatalog` runtime hooks, exports Moonshot catalog data for that hook, and adds hook-level tests.

## Best Possible Solution

Add a Kimi manifest `modelCatalog` row for canonical `kimi-for-coding`, keep legacy Kimi IDs as normalization-only compatibility, and leave Moonshot on its existing manifest catalog path.

## Reproduction Assessment

Yes. Source inspection shows Kimi has no no-key manifest catalog rows while its runtime catalog returns `null` without an API key; Moonshot no longer reproduces on current main because it already has static manifest rows.

## Solution Assessment

No. The PR uses a deprecated runtime augmentation hook and exposes legacy Kimi IDs; the narrower maintainable solution is Kimi-only manifest catalog metadata with focused picker/list coverage.
