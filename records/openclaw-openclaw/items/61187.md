## Summary

Keep open: the Kimi no-key picker bug is still valid on current main, but this PR should not merge as-is because it exposes legacy Kimi aliases, duplicates Moonshot's manifest-owned catalog path, and adds a test import that does not match current SDK test helpers.

## What This Changes

Adds runtime `augmentModelCatalog` hooks for Kimi and Moonshot, exports a Moonshot catalog constant, and adds provider tests for those hooks.

## Best Possible Solution

Keep Moonshot on the manifest/static catalog path, add a Kimi no-key picker source for only `kimi/kimi-for-coding`, and cover the picker behavior with current SDK-runtime tests.

## Reproduction Assessment

Yes. Source inspection plus the PR screenshots give a high-confidence path: run setup, choose Kimi or Moonshot, and open the provider-scoped default model picker before credentials are configured; current main covers Moonshot through manifest rows but Kimi still lacks a no-key picker source.

## Solution Assessment

No. The PR's Kimi goal is useful, but the narrower maintainable fix is Kimi-only static/model-catalog picker exposure for the stable model while leaving Moonshot on its manifest-backed catalog path.
