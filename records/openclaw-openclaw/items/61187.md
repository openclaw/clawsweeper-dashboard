## Summary

Keep open: the Kimi picker bug is still source-reproducible on current main, but this branch is stale and would duplicate or conflict with current Moonshot catalog ownership.

## What This Changes

The PR adds runtime model-catalog augmentation hooks and tests for the Kimi and Moonshot provider plugins so setup model pickers show provider-specific rows before API-key-backed discovery.

## Best Possible Solution

Keep Moonshot's manifest/static catalog as the source of truth and expose only the current stable Kimi picker row through the current static catalog patterns.

## Reproduction Assessment

Yes, by source inspection for the Kimi side: current main has no Kimi static manifest catalog rows, and its runtime catalog returns null without resolved Kimi auth. The Moonshot side no longer reproduces on current main because manifest/static catalog rows already exist.

## Solution Assessment

No, not as-is. The Kimi intent is useful, but the branch should be rebased to current model IDs and SDK test imports, and the stale Moonshot augment/export/test changes should be removed.
