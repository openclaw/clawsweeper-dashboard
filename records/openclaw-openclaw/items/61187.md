## Summary

Keep open: the PR targets a real Kimi setup/model-picker gap, but the branch is no longer the right fix for current main because it adds deprecated runtime catalog hooks, duplicates Moonshot’s manifest-backed catalog path, exposes Kimi legacy aliases as selectable primary models, and adds a test that uses a removed helper path.

## What This Changes

The PR adds Kimi and Moonshot `augmentModelCatalog` hooks, exports Moonshot catalog data, and adds hook-level tests for those provider plugins.

## Best Possible Solution

Add a manifest-backed Kimi `modelCatalog` row for the canonical `kimi-for-coding` model, keep legacy aliases as normalization-only compatibility, and leave Moonshot on its existing manifest catalog rows.

## Reproduction Assessment

Yes. Source inspection shows Kimi has no no-key manifest catalog rows while its runtime catalog returns `null` without an API key; the Moonshot half no longer reproduces because current main already has static manifest rows.

## Solution Assessment

No. The current PR’s runtime-hook approach is not the best fix because `augmentModelCatalog` is deprecated and the picker now has a manifest-first path that should carry Kimi’s static model row.
