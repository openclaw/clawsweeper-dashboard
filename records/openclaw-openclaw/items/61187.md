## Summary

Keep open: the PR targets a real Kimi model-picker gap, but the branch is stale against current main and the proposed fix uses deprecated runtime catalog hooks, duplicates Moonshot's existing manifest catalog, and adds a test import that will not work after rebase.

## What This Changes

The PR adds Kimi and Moonshot `augmentModelCatalog` hooks plus tests, and exports Moonshot catalog data for hook reuse.

## Best Possible Solution

Add canonical Kimi manifest catalog metadata for `kimi-for-coding`, keep Kimi legacy ids as normalization compatibility, and leave Moonshot on its existing manifest-backed catalog.

## Reproduction Assessment

Yes, source-reproducible for Kimi: current main has no Kimi manifest `modelCatalog` rows and returns no Kimi runtime catalog without an API key, while the preferred-provider picker depends on static rows before runtime fallback. The Moonshot half does not reproduce on current main because Moonshot already has static manifest catalog rows.

## Solution Assessment

No. The narrower maintainable fix is manifest-backed Kimi metadata and focused picker coverage, not new deprecated `augmentModelCatalog` hooks or duplicate Moonshot catalog data.
