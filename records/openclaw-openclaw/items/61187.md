## Summary

Keep open. The Kimi setup-picker bug is still real, but current main already moved Moonshot picker rows to the manifest catalog and changed Kimi to the stable `kimi-for-coding` model ID, so this branch needs narrow repairs before it can merge.

## What This Changes

The PR adds Kimi and Moonshot runtime model-catalog augmentation hooks, exports Moonshot catalog data, and adds provider-hook tests for the setup model picker.

## Best Possible Solution

Keep Moonshot catalog ownership in the manifest/static catalog path, add a Kimi-owned no-key picker row for stable `kimi-for-coding`, and cover that path with SDK-surface regression tests.

## Reproduction Assessment

Yes, by source inspection and PR screenshots. Current Kimi returns no runtime catalog without an API key and has no manifest rows, while the preferred-provider picker falls back when no provider-scoped static rows exist; Moonshot no longer reproduces through the same path because current main has manifest rows.

## Solution Assessment

No. The Kimi direction is right, but the branch should expose only the stable Kimi row, drop the obsolete Moonshot augment/export/test path, and use the current SDK plugin test runtime.
