## Summary

Keep open: the PR addresses a real Kimi no-key model picker gap, and the screenshots are useful proof, but the patch still routes the fix through deprecated catalog hooks, advertises Kimi legacy aliases as picker choices, and adds redundant Moonshot runtime catalog work that current main no longer needs.

## What This Changes

The PR adds `augmentModelCatalog` hooks and hook tests for Kimi and Moonshot provider model picker rows, plus an export of Moonshot's catalog constant.

## Best Possible Solution

Add Kimi manifest `modelCatalog` metadata for only the canonical `kimi-for-coding` row, keep legacy ids as normalization compatibility, leave Moonshot on its manifest catalog, and cover the picker path with focused tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence Kimi path: select Kimi before configuring an API key, where the runtime catalog returns `null` and no manifest `modelCatalog` exists; the Moonshot half does not reproduce on current main because Moonshot already has static manifest rows.

## Solution Assessment

No. The PR fixes the visible symptom through deprecated `augmentModelCatalog` hooks and duplicates Moonshot rows; the narrower maintainable fix is manifest-backed Kimi metadata with tests.
