## Summary

Keep open: the PR targets a real Kimi model-picker gap, but the branch is not mergeable as-is because Moonshot is already covered on current main, the proposed runtime hook path is deprecated, and the Kimi rows would expose legacy compatibility ids as primary choices.

## What This Changes

The PR adds runtime `augmentModelCatalog` hooks for Kimi and Moonshot, exports Moonshot catalog data, and adds hook-focused tests.

## Best Possible Solution

Add a Kimi manifest/control-plane `modelCatalog` row for canonical `kimi-for-coding`, keep legacy Kimi ids as normalization-only compatibility, and leave Moonshot on its existing manifest catalog path.

## Reproduction Assessment

Yes. Source inspection shows Kimi has no no-key manifest catalog rows while its runtime catalog returns null without an API key; Moonshot no longer reproduces on current main because static manifest rows already exist.

## Solution Assessment

No. The PR uses a deprecated runtime augmentation hook and would expose stale or legacy rows; the narrower maintainable fix is Kimi-only manifest catalog metadata with focused picker coverage.
