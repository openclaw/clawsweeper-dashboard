## Summary

Keep open: this PR targets a real provider-scoped model picker bug, but it should not merge as-is because current main has moved Moonshot to manifest/static catalog ownership and Kimi now uses `kimi-for-coding` as the stable picker row.

## What This Changes

The PR adds Kimi and Moonshot `augmentModelCatalog` hooks plus provider tests so their models appear in setup/model picker flows before API keys are configured.

## Best Possible Solution

Keep Moonshot on the manifest/static catalog path, add a Kimi no-key picker source for only `kimi-for-coding`, and cover the flow with SDK-runtime provider tests plus model-picker regression coverage.

## Reproduction Assessment

Yes. Source inspection and the PR screenshots give a high-confidence path: run setup, choose Kimi or Moonshot, and open the provider-scoped model picker before credentials are configured.

## Solution Assessment

No. The Kimi direction is useful, but the branch exposes legacy Kimi aliases, reintroduces stale Moonshot catalog ownership, and uses an obsolete test helper import instead of the current SDK test runtime.
