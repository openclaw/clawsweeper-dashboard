## Summary

Keep open: the Kimi picker problem is still valid, but the branch is not ready against current main because it promotes legacy Kimi rows, duplicates Moonshot catalog ownership that main already handles, and adds a test through a removed helper path.

## What This Changes

The branch adds provider model-catalog augmentation hooks and tests for Kimi and Moonshot so setup model pickers can show provider-specific rows before API-key-backed discovery.

## Best Possible Solution

Kimi should publish a stable no-key kimi-for-coding picker row through existing provider catalog seams while Moonshot remains manifest/static catalog-owned.

## Reproduction Assessment

Yes. Source inspection shows Kimi still has no no-key provider-scoped picker rows while the picker falls back when no preferred-provider static rows exist, and the PR screenshots show the affected setup flow.

## Solution Assessment

No. The Kimi direction is right, but the branch should expose only the stable Kimi row, drop the now-obsolete Moonshot augment/export/test path, and use the current SDK plugin test runtime.

  - confidence: 0.9
