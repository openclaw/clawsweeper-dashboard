## Summary

Keep open: the PR targets a real Kimi picker gap, but the branch should not merge as-is because it uses deprecated runtime catalog augmentation, exposes legacy Kimi IDs as picker choices, and duplicates Moonshot's current manifest-backed catalog path.

## What This Changes

The branch adds runtime `augmentModelCatalog` hooks and tests for Kimi and Moonshot, plus a Moonshot catalog export for reuse.

## Best Possible Solution

Add a Kimi manifest `modelCatalog` row for canonical `kimi-for-coding`, keep legacy Kimi IDs as normalization-only compatibility, and leave Moonshot on its existing manifest catalog path.

## Reproduction Assessment

Yes. Source inspection shows Kimi has no no-key manifest catalog rows while its runtime catalog returns `null` without an API key; Moonshot no longer reproduces because current main already has static manifest rows.

## Solution Assessment

No. The PR uses a deprecated runtime augmentation hook and would expose legacy Kimi IDs; the narrower maintainable fix is Kimi-only manifest catalog metadata with focused picker/list coverage.
