## Summary

Keep this PR open because the Kimi no-key picker problem is still valid, but the current patch should not merge as-is: it uses the deprecated runtime catalog hook, adds obsolete Moonshot catalog duplication, and has a stale test import.

## What This Changes

The PR adds runtime `augmentModelCatalog` hooks and hook tests for Kimi and Moonshot provider model picker rows.

## Best Possible Solution

Add Kimi manifest `modelCatalog` metadata for the canonical `kimi-for-coding` row, keep legacy ids as normalization compatibility, leave Moonshot on its manifest catalog, and cover the picker path with focused tests.

## Reproduction Assessment

Yes, from current-main source inspection: Kimi has no manifest `modelCatalog` and its runtime catalog returns `null` without an API key. The Moonshot half does not reproduce on current main because Moonshot already has static manifest rows.

## Solution Assessment

No. The PR fixes the visible symptom through deprecated `augmentModelCatalog` hooks and duplicates Moonshot rows; the narrower maintainable solution is manifest-backed Kimi metadata only.
