## Summary

Keep open: the Kimi setup picker bug is still valid on current main, but this branch is stale against current Kimi/Moonshot catalog ownership and needs focused repair before merge.

## What This Changes

The PR adds `augmentModelCatalog` hooks and tests for Kimi and Moonshot so setup model picker rows appear without API-key-backed discovery.

## Best Possible Solution

Keep Moonshot on the manifest/static catalog path, and add only the current Kimi no-key picker row for `kimi-for-coding` with SDK-surface regression coverage.

## Reproduction Assessment

Yes by source inspection: current Kimi returns no catalog without an API key and has no manifest model rows, so the preferred-provider picker cannot get Kimi rows from current main. The Moonshot side no longer reproduces because current main already has manifest/static catalog rows.

## Solution Assessment

No. The Kimi intent is the right narrow fix, but this PR should be rebased to current stable Kimi ids and SDK test imports, and the obsolete Moonshot augment/export/test changes should be dropped.
