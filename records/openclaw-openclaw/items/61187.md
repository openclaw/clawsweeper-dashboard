## Summary

Keep open: the Kimi setup picker gap is real and the screenshot proof is sufficient for the visible flow, but the branch should not merge as-is because current main already owns Moonshot rows through the manifest catalog, Kimi now uses `kimi-for-coding` as the stable picker id, and the added Kimi test imports a removed helper path.

## What This Changes

The PR adds Kimi and Moonshot `augmentModelCatalog` hooks, exports Moonshot catalog data, and adds provider hook tests so setup model picker rows appear before API-key-backed discovery.

## Best Possible Solution

Keep Moonshot catalog ownership in the manifest/static catalog path, add a Kimi-owned no-key picker row for stable `kimi-for-coding`, and cover that path with SDK-surface regression tests.

## Reproduction Assessment

Yes, by source inspection and PR screenshots. Current Kimi returns no runtime catalog without an API key and has no manifest rows, while the preferred-provider picker falls back when no provider-scoped static rows exist; Moonshot now has manifest rows and no longer reproduces through the PR's path.

## Solution Assessment

No. The Kimi direction is the right narrow fix, but the branch should expose only the stable Kimi row, drop the obsolete Moonshot augment/export/test path, and use the current SDK plugin test runtime.
