## Summary

Keep open with repair needed: the Kimi picker gap is real, but the branch solves it through a deprecated runtime hook and duplicates Moonshot’s already-manifest-backed catalog. Current main points to a manifest-catalog fix plus test/runtime cleanup.

## What This Changes

Adds Kimi and Moonshot `augmentModelCatalog` hooks, exports Moonshot catalog data, and adds hook-focused tests.

## Best Possible Solution

Add canonical Kimi manifest `modelCatalog` metadata for `kimi-for-coding`, keep legacy IDs as compatibility, and leave Moonshot on its existing manifest-backed catalog.

## Reproduction Assessment

Yes for Kimi by source inspection: the no-key Kimi runtime catalog returns `null`, the Kimi manifest has no static `modelCatalog`, and the picker depends on static preferred-provider rows before runtime fallback. Moonshot no longer reproduces because its manifest already has static catalog rows.

## Solution Assessment

No. The PR’s deprecated runtime augmentation hooks are not the best fix; the narrow maintainable path is Kimi manifest catalog metadata plus focused picker/catalog tests, without duplicating Moonshot catalog ownership.
