## Summary

Keep open: the Kimi no-key picker bug is still source-reproducible on current `main`, but the PR should not merge as-is because it uses deprecated runtime catalog hooks, duplicates Moonshot catalog data that is already manifest-backed, and carries a stale test helper import.

## What This Changes

The PR adds runtime `augmentModelCatalog` hooks for Kimi and Moonshot provider plugins, exports Moonshot catalog data, and adds hook-focused tests.

## Best Possible Solution

Add canonical Kimi manifest catalog metadata for `kimi-for-coding`, keep legacy ids as normalization compatibility, and leave Moonshot on its existing manifest-backed catalog.

## Reproduction Assessment

Yes. Kimi is source-reproducible on current `main`: its runtime catalog returns `null` without an API key, its manifest has no static `modelCatalog` rows, and the preferred-provider picker depends on static manifest rows before runtime fallback; Moonshot no longer reproduces because it already has static manifest rows.

## Solution Assessment

No. The narrow maintainable solution is a manifest-backed Kimi catalog row plus focused picker/catalog tests, not new deprecated `augmentModelCatalog` hooks or duplicate Moonshot catalog data.
