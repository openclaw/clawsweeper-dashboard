## Summary

Keep open. Current main has a partial mitigation that maps malformed Anthropic SSE JSON to a stable friendly sentinel, but that sentinel is still not classified as a failover signal, so the canonical Telegram/Anthropic recovery bug remains actionable.

## What This Changes

The issue asks OpenClaw to recover from recurring Anthropic streaming JSON parse failures, avoid raw parse errors in Telegram, and make retry/failover consistent.

## Best Possible Solution

Add a narrow provider-boundary failover classification for the malformed streaming fragment sentinel, with regression coverage that proves Anthropic malformed SSE errors retry/fail over and still render friendly text without hiding provider request-validation JSON diagnostics.

## Reproduction Assessment

Yes for a source-level path: malformed Anthropic SSE data currently produces stopReason error with the malformed_streaming_fragment sentinel, and the failover classifier has no matching branch. I did not run a live Telegram/Anthropic reproduction for the intermittent production trigger.

## Solution Assessment

Yes. The best fix is a narrow classifier/retry path for OpenClaw's own malformed streaming fragment sentinel, not broad sanitization of every raw JSON.parse message.
