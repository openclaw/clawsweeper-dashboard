## Summary

Keep open. Current main still normalizes malformed Anthropic SSE JSON into a stable sentinel and sanitizes that text for users, but the sentinel is not classified as a failover signal, so the reported `failoverReason=null` recovery gap remains actionable.

## What This Changes

The issue requests consistent retry/failover and non-raw user-facing handling for recurring Anthropic streaming JSON parse failures in Telegram.

## Best Possible Solution

Classify the existing provider-boundary malformed-streaming-fragment sentinel as a retryable transport-style failover signal, while preserving the current guard that arbitrary raw JSON.parse diagnostics are not broadly treated as retryable.

## Reproduction Assessment

Yes, at source level. Current tests synthesize malformed Anthropic SSE into the stable sentinel, and current runner/classifier code shows that sentinel is not a failover signal, leaving the reported `failoverReason=null` path intact.

## Solution Assessment

Yes. The narrow maintainable fix is to classify only the normalized transport sentinel for retry/failover and add regression coverage, not to broadly classify every raw JSON.parse message.
