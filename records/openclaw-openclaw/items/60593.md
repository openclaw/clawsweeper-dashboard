## Summary

Keep open. Current main still has the same core gap: Anthropic malformed SSE JSON is normalized and sanitized, but that stable sentinel is not classified as retryable, so the reported `failoverReason=null` recovery failure remains actionable.

## What This Changes

The issue requests consistent retry/failover for Anthropic streaming JSON parse failures and suppression of raw JSON.parse errors in Telegram.

## Best Possible Solution

Classify the existing malformed-streaming-fragment sentinel as a retryable transport-style failover signal while keeping arbitrary raw JSON.parse diagnostics out of failover unless normalized at the provider boundary.

## Reproduction Assessment

Yes, at source level: current fixtures synthesize malformed Anthropic SSE into the stable sentinel, and current source inspection shows that sentinel still falls through `classifyFailoverReason` without a failover reason. I did not run a live Anthropic/Telegram reproduction in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to classify the existing provider-boundary sentinel for retry/failover, not to broadly treat every raw JSON.parse diagnostic as retryable.
