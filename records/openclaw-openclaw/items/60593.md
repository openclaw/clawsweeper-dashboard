## Summary

Keep open. Current main still leaves the core failover gap: malformed Anthropic SSE JSON is normalized to a stable sentinel and sanitized for users, but that sentinel is not classified as a retryable failover reason, so the reported recovery failure remains actionable.

## What This Changes

The issue asks for recurring Anthropic streaming JSON parse failures to trigger retry/failover consistently and avoid raw parse errors surfacing in Telegram.

## Best Possible Solution

Classify the normalized malformed_streaming_fragment sentinel as a retryable transport-style failover signal while keeping arbitrary raw JSON.parse diagnostics out of failover unless normalized at the provider boundary.

## Reproduction Assessment

Yes, at source level. Current fixtures synthesize malformed Anthropic SSE into the stable sentinel, and current source inspection shows that sentinel still falls through classifyFailoverReason without a failover reason.

## Solution Assessment

Yes. The narrow maintainable fix is to classify the existing provider-boundary sentinel for retry/failover, not to broadly treat every raw JSON.parse diagnostic as retryable.
