## Summary

Keep open for review, but this draft is not merge-ready: the hook is wired into a deprecated compatibility helper instead of the shared inbound turn path, redirecting only mutates `route.sessionKey` while the finalized context still carries the original `SessionKey`, and the PR lacks real behavior proof. The remaining design also overlaps with the open broader pre-route work at https://github.com/openclaw/openclaw/pull/57510 and the paired issue at https://github.com/openclaw/openclaw/issues/81061.

## What This Changes

The PR adds a new `before_route_inbound_message` plugin hook with redirect and suppress result types, a hook runner method, and a call from `dispatchChannelMessageReplyWithBase`.

## Best Possible Solution

Reconcile this with the existing pre-route proposal, then implement one documented plugin routing seam in the shared channel-turn preflight/resolve path with context updates, focused hook/dispatch tests, and clear security/audit semantics.

## Reproduction Assessment

Not applicable as a user bug reproduction: this PR adds a new plugin API capability. Source inspection does give a high-confidence patch check: current adapters enter through `runInboundReplyTurn`/`runChannelTurn`, and downstream record/dispatch prefers `ctxPayload.SessionKey` over `routeSessionKey`.

## Solution Assessment

No. The proposed helper-level hook is too narrow and too late for the stated pre-routing contract; a shared channel-turn or dispatch-from-config route override seam is the safer maintainable direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Run the hook on the shared inbound turn path:** `src/plugin-sdk/inbound-reply-dispatch.ts:157`
  - body: Placing the new hook inside `dispatchChannelMessageReplyWithBase` means it will not run for the current channel adapters that call `runInboundReplyTurn()`/`runChannelTurn()` directly. That includes the Discord use case named in the PR, so plugins registering this hook would not see the inbound messages they need to bridge. The hook needs to live in the shared channel-turn preflight/resolve path, or every active inbound entrypoint must be wired and covered.
  - confidence: 0.91
- **[P2] Update the finalized context when redirecting:** `src/plugin-sdk/inbound-reply-dispatch.ts:201-203`
  - body: When a hook returns `redirectSessionKey`, this only replaces `params.route.sessionKey`; the finalized `ctxPayload.SessionKey` still points at the original route. The channel turn record path uses `ctxPayload.SessionKey ?? routeSessionKey`, and dispatch receives the same unchanged context, so the redirected session will be ignored for normal finalized messages. The redirect must update the effective context/session facts that downstream record and dispatch actually consume.
  - confidence: 0.93
