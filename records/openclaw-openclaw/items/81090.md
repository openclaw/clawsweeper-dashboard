## Summary

Keep open: this is an active draft feature PR paired with an open design issue, and the proposed implementation has concrete routing/context defects plus missing real behavior proof and unresolved redirect/suppress security semantics.

## What This Changes

Adds a new `before_route_inbound_message` plugin hook with redirect/suppress result types, hook-runner wiring, and a call from `dispatchChannelMessageReplyWithBase`.

## Best Possible Solution

Reconcile this with the open design issue, then implement one documented plugin routing seam in the shared channel-turn preflight/resolve path with context updates, focused tests, and explicit security/audit semantics.

## Reproduction Assessment

Not applicable as a user bug reproduction: this PR adds a new public plugin routing capability. Source inspection gives a high-confidence patch check because current adapters enter through `runInboundReplyTurn`/`runChannelTurn`, and downstream record/dispatch prefers `ctxPayload.SessionKey` over `routeSessionKey`.

## Solution Assessment

No. The proposed helper-level hook is too narrow and too late for the stated pre-routing contract, and redirects do not update the finalized session context that downstream code actually uses.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Run the hook on the shared inbound turn path:** `src/plugin-sdk/inbound-reply-dispatch.ts:157`
  - body: Placing the new hook inside `dispatchChannelMessageReplyWithBase` means it will not run for current channel adapters that call `runInboundReplyTurn()`/`runChannelTurn()` directly. That includes the Discord bridging use case named by the PR, so the hook needs to live in the shared channel-turn preflight/resolve path or every active inbound entrypoint must be wired and covered.
  - confidence: 0.91
- **[P2] Update the finalized context when redirecting:** `src/plugin-sdk/inbound-reply-dispatch.ts:201-203`
  - body: When a hook returns `redirectSessionKey`, the patch only replaces `params.route.sessionKey`; finalized `ctxPayload.SessionKey` still points at the original route. The record and dispatch paths prefer `ctxPayload.SessionKey`, so normal finalized messages would ignore the redirect unless the effective context/session facts are updated too.
  - confidence: 0.93
