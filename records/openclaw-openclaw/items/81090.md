## What This Changes

The PR adds a `before_route_inbound_message` plugin hook, wires it into shared channel-turn and legacy inbound dispatch paths, and adds channel-turn tests for redirect and suppress results.

## Best Possible Solution

Design one documented plugin routing seam in the shared channel-turn path using the resolved turn facts, update the effective finalized context on redirects, and define audit/authorization semantics before publishing the hook.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use existing turn facts in the hook:** `src/channels/turn/kernel.ts:470-482`
  - body: `runChannelTurn` receives `RunChannelTurnParams`, which has no `ctxPayload`, and `ChannelTurnResolved` has `routeSessionKey` rather than `sessionKey`. These new reads make the branch fail typechecking before the hook can run; build the event from `resolved.ctxPayload`/`resolved.routeSessionKey` or normalized input facts instead.
  - confidence: 0.96
- **[P2] Update the resolved context when redirecting:** `src/channels/turn/kernel.ts:505-510`
  - body: The redirect branch writes `params.ctxPayload.SessionKey`, but the shared turn path records and dispatches the resolved payload. Because `runPreparedChannelTurnCore` prefers `ctxPayload.SessionKey` over `routeSessionKey`, a normal resolved context can keep using the original session unless `resolved.ctxPayload.SessionKey` is updated too.
  - confidence: 0.9
- **[P3] Call the matcher and assert the redirected session:** `src/channels/turn/kernel.test.ts:952`
  - body: This line only reads the Vitest matcher instead of invoking it, and the test never checks the record request or result context for the redirected session key. The test can pass without proving redirect behavior.
  - confidence: 0.87
