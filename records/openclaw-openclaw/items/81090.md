## Summary

Keep open for review: the requested public routing hook is not on current main, and this PR has blocking implementation issues in the proposed placement and redirect behavior.

## What This Changes

The PR adds a new `before_route_inbound_message` plugin hook type and runner, then calls it from the legacy inbound reply dispatch helper to redirect or suppress inbound messages.

## Best Possible Solution

Design the hook at the channel turn preflight or route-resolution boundary, or extend conversation bindings with typed redirect/suppress semantics and focused tests for `ctxPayload.SessionKey`, `routeSessionKey`, suppression, and current channel adapters.

## Reproduction Assessment

Yes for the PR bug by source inspection: current dispatch prefers `ctxPayload.SessionKey` over `routeSessionKey`, so the patch's route-only redirect will not change the actual session when channel payloads already set `SessionKey`. I did not run a live channel proof.

## Solution Assessment

No: the PR hooks a deprecated compatibility helper after route and context assembly, so it is not the narrowest maintainable implementation for a public pre-routing API.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redirect the effective session key, not only the route:** `src/plugin-sdk/inbound-reply-dispatch.ts:203`
  - body: When a hook returns `redirectSessionKey`, this rewrites only `params.route.sessionKey`. The channel turn kernel records and dispatches with `ctxPayload.SessionKey ?? routeSessionKey`, and most channel contexts already set `ctxPayload.SessionKey`, so the original session still wins and the advertised redirect does not work on the common path.
  - confidence: 0.92
- **[P2] Wire the hook into the channel turn lifecycle:** `src/plugin-sdk/inbound-reply-dispatch.ts:160`
  - body: Adding the hook only to `dispatchChannelMessageReplyWithBase` misses current channel adapters that use `runInboundReplyTurn` or the message lifecycle directly. Since this helper is explicitly legacy compatibility code on main, the new public hook is not the central all-inbound interception point described by the PR.
  - confidence: 0.86

## Security Review

Status: needs_attention
