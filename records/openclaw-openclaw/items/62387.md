## Label Justifications

- `P2`: This is a real migration bug with security-sensitive defaults, but the repair needs cross-channel design review rather than emergency response.
- `impact:data-loss`: The migration can move persisted shared channel config keys out of the root position where named accounts still need them.
- `impact:security`: The affected key set includes access-control defaults such as `allowFrom`, `dmPolicy`, `groupPolicy`, and `groupAllowFrom`.
- `impact:message-loss`: The affected key set includes delivery defaults such as `defaultTo`, which can change outbound routing or suppress expected delivery after migration.

## Summary

Keep open. Current main and v2026.5.19 still move the full common promotion key set when a named-account channel lacks `namedAccountPromotionKeys`, while representative channel docs and account resolution show named accounts inherit root defaults, not `accounts.default`; the open Feishu PR only covers one channel-specific instance.

## What This Changes

The issue asks OpenClaw to stop doctor/account-add promotion from moving shared policy and delivery defaults into `accounts.default` for named-account channels that lack `namedAccountPromotionKeys`.

## Best Possible Solution

Define a channel-aware promotion contract or conservative default that preserves root-level policy and delivery defaults for root-inheriting channels, with regressions for representative root-inheriting and `accounts.default`-sharing channels.

## Reproduction Assessment

Yes, at source level. Current main still returns all movable keys when named accounts exist and no `namedAccountPromotionKeys` are declared, then doctor copies those keys into `accounts.default` and deletes them from the channel root.

## Solution Assessment

Unclear. The bug is real, but the best fix needs a maintainer choice between broader per-channel promotion metadata, a safer generic fallback, or a documented shared-default contract for channels that intentionally use `accounts.default`.
