## Summary

Keep open. Current main still follows the reported fallback for named-account channel configs without `namedAccountPromotionKeys`, so shared policy/default keys such as `allowFrom`, `groupPolicy`, and `defaultTo` can be moved out of the channel root into a promoted account and stop applying to existing named accounts.

## What This Changes

The issue requests preserving shared root policy and delivery defaults during multi-account channel promotion for channels that do not declare `namedAccountPromotionKeys`.

## Best Possible Solution

Preserve shared root policy and delivery defaults when named accounts already exist, while still moving true credential/auth fields into the promoted account through the shared doctor and channel-add promotion paths with focused Slack/Discord plus Telegram/Matrix regressions.

## Reproduction Assessment

Yes. A Slack or Discord config with `accounts.work` and root `allowFrom` follows a source-level path where the key is selected for promotion, deleted from the root, and then not inherited by the named account according to existing account-resolution tests.

## Solution Assessment

Unclear until maintainers choose the compatibility shape. A conservative fallback for missing `namedAccountPromotionKeys` or explicit per-channel promotion surfaces would both be maintainable, but access-control behavior should be reviewed before automation changes it.
