## Summary

Keep open. Current main still follows the reported fallback for named-account channel configs without `namedAccountPromotionKeys`: policy/default keys remain in the generic movable set and are then deleted from the channel root into `accounts.default`, which breaks channels such as Slack and Discord where named accounts inherit root defaults but not `accounts.default` defaults.

## What This Changes

Fix channel account promotion so doctor and channel-add migrations do not strip shared policy/default keys such as `allowFrom`, `groupPolicy`, and `defaultTo` from named-account channel configs that lack `namedAccountPromotionKeys`.

## Best Possible Solution

Preserve shared root policy/delivery defaults when named accounts already exist, while still moving credential/auth keys into the promoted account. The fix should cover the shared promotion helper and doctor/channel-add paths, and it should add regressions for at least one non-inheriting channel such as Slack or Discord plus guardrails for Telegram, Matrix, and any channel that intentionally inherits `accounts.default`.

## Reproduction Assessment

Yes. A Slack or Discord config with `accounts.work` plus top-level `allowFrom` or `groupPolicy` follows a source-level path where those keys are selected by `resolveSingleAccountKeysToMove()`, deleted from root by doctor/channel-add promotion, and then not inherited by the named account according to existing account-resolution tests.

## Solution Assessment

Unclear until maintainers choose the compatibility shape. A narrow core fallback that treats missing `namedAccountPromotionKeys` conservatively is likely best, but it must account for channels that deliberately use `accounts.default` as shared defaults.

## Repair Work Prompt

Fix the named-account channel promotion bug tracked by this issue. On current main, `resolveSingleAccountKeysToMove()` returns all static movable keys for named-account configs when `namedAccountPromotionKeys` is undefined, so doctor/channel-add promotion can move shared root defaults like `dmPolicy`, `allowFrom`, `groupPolicy`, `groupAllowFrom`, and `defaultTo` into `accounts.default`; Slack/Discord named accounts then stop inheriting those defaults. Implement conservative named-account promotion semantics so channels without explicit `namedAccountPromotionKeys` do not strip shared policy/delivery keys from the channel root while still moving credential/auth keys into the promoted account. Prefer a generic helper-level solution or explicit setup surfaces where that is safer; do not hardcode bundled channel ids into core beyond generic promotion categories, do not change startup/load-time migrations outside the existing doctor/channel-add/setup promotion paths, and preserve Telegram/Matrix contract behavior. Add focused regressions covering the helper and doctor/channel-add behavior for a non-inheriting channel such as Slack or Discord; keep or update WhatsApp/GoogleChat coverage if their `accounts.default` inheritance semantics are intentionally different. Add a `CHANGELOG.md` fix entry if the change is user-facing, crediting `Thanks @alexey-pelykh`. Validate with targeted channel promotion tests first, then run the changed gate in Testbox before handoff.
