## Label Justifications

- `P2`: This is a real migration bug with security-sensitive defaults, but it is bounded to channel config promotion and needs maintainer design review rather than emergency response.
- `impact:data-loss`: The migration can move persisted shared channel config keys out of the root position where named accounts still need them.
- `impact:security`: The affected promotion set includes access-control defaults such as `allowFrom`, `dmPolicy`, `groupPolicy`, and `groupAllowFrom`.
- `impact:message-loss`: The same promotion set includes delivery defaults such as `defaultTo`, which can change outbound routing or suppress expected delivery after migration.

## Summary

Keep open. Current main and the latest stable tag still move all common keys when a channel has named accounts but no `namedAccountPromotionKeys`, while Slack/Discord docs and account resolution show named accounts inherit root defaults rather than `accounts.default`; the related Signal report was already closed as a duplicate of this canonical issue.

## What This Changes

The issue reports that multi-account channel promotion can move shared top-level defaults into `accounts.default` for channels without named-account promotion metadata, breaking named-account inheritance.

## Best Possible Solution

Add a channel-aware promotion contract or conservative default that preserves root-level policy and delivery defaults for root-inheriting channels, while keeping explicit `accounts.default` shared-default behavior only for channels that intentionally support it, with focused doctor and account-add regressions across representative channels.

## Reproduction Assessment

Yes, at source level. Current main still lets Slack, Discord, Signal, Feishu, and similar channels without `namedAccountPromotionKeys` move shared root keys into `accounts.default`, while representative channels document or implement named-account inheritance from the root instead.

## Solution Assessment

Unclear. The bug is real, but the best fix needs a maintainer choice between broader per-channel promotion metadata, a safer generic fallback, or a documented `accounts.default` shared-default contract for channels that intentionally support it.
