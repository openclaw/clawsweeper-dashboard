## Label Justifications

- `P2`: This is a real migration bug with security-sensitive defaults, but it has a bounded channel-config surface and no evidence of an emergency runtime outage.
- `impact:security`: The migration can remove inherited `allowFrom`, `dmPolicy`, and group policy defaults from named accounts.
- `impact:data-loss`: The report is about persisted channel config keys being moved out of the shared root position in a way that corrupts effective configuration for named accounts.
- `impact:message-loss`: The same promotion set includes delivery defaults such as `defaultTo`, which can change whether or where named accounts send messages after migration.

## Summary

Keep open. Current main and the latest stable tag still have the reported promotion behavior, and existing Slack/Discord tests confirm the root-inheritance contract that this migration can break; the repair needs maintainer judgment because some channels intentionally use `accounts.default` as shared defaults.

## What This Changes

The issue reports that multi-account channel promotion moves shared root defaults such as access-control policy into `accounts.default` for channels without `namedAccountPromotionKeys`, so named accounts can lose inherited defaults after `doctor --fix` or `channels add`.

## Best Possible Solution

Add channel-aware promotion metadata or an equivalent conservative contract so root-inheriting channels keep shared policy and delivery defaults at the channel root, while channels that intentionally use `accounts.default` keep that behavior, with focused doctor and `channels add` regressions across representative channels.

## Reproduction Assessment

Yes. Source inspection on current main shows Slack or Discord configs with named accounts and root `allowFrom`/policy keys are promoted into `accounts.default`, while those named accounts only inherit the root keys and not `accounts.default`.

## Solution Assessment

Unclear. The bug is real, but the best fix needs a maintainer choice between per-channel promotion metadata, a safer generic fallback, or an explicit shared-default-account contract for channels that intentionally inherit `accounts.default`.
