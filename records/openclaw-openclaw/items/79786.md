## Summary

Keep open: the production change is the right narrow fix for a source-reproducible WebChat heartbeat leak, but this external PR still lacks real after-fix Gateway/WebChat proof and its new regression test does not actually exercise the `showOk: true` resolver path.

## What This Changes

The PR removes WebChat heartbeat stream suppression's dependency on `resolveHeartbeatVisibility(...webchat...)` and adds a server-chat unit test for suppressing `HEARTBEAT_OK` when channel heartbeat `showOk` is true.

## Best Possible Solution

Land a narrow Gateway fix that always hides WebChat heartbeat ACK output independently of external-channel `showOk`, with a regression test that truly covers `showOk: true` and contributor-supplied real behavior proof.

## Reproduction Assessment

Yes, from source inspection: current main resolves WebChat heartbeat visibility from `channels.defaults.heartbeat.showOk` and returns `!showOk`, so `showOk: true` can bypass live chat suppression. I did not run a live Gateway/WebChat reproduction because this review is read-only.

## Solution Assessment

Mostly yes: using heartbeat run context for WebChat suppression is the narrow maintainable production fix, while leaving external delivery gated in the heartbeat runner. The test and contributor proof need tightening before this is merge-ready.
