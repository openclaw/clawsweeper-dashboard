## Summary

Current main resolves the reported built-in `codex-cli` false-success path by removing the bundled direct Codex CLI backend and migrating legacy `codex-cli` refs to the Codex app-server route.

## What This Changes

The issue asks OpenClaw to treat Codex CLI JSONL sandbox-denied action failures as failed runs even when `codex exec` exits 0.

## Best Possible Solution

Keep the Codex app-server migration as the supported OpenAI Codex path; handle any future custom CLI-backend false-success report as a separate plugin-owned or generic CLI contract issue with current-main config evidence.

## Reproduction Assessment

No current-main reproduction remains for the reported built-in `codex-cli` path: current docs/source show that backend is no longer bundled and legacy refs migrate to the Codex app-server route. The original 2026.3.2 command is good historical evidence, but it no longer maps to the supported default path on main.

## Solution Assessment

Yes. For OpenClaw core, removing the bundled direct Codex CLI route and migrating legacy refs is the maintainable solution; Codex-specific JSONL failure parsing should not be added to generic CLI core unless a supported plugin/runtime contract needs it.
