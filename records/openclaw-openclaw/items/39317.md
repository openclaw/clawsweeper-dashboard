## Summary

Keep open: current main still treats exit-0 Codex JSONL sandbox/write denials as successful CLI output, and the focused fix was closed unmerged while the output-last-message work is complementary.

## What This Changes

Classify Codex `exec --json` sandbox or write-denied JSONL signals as failed OpenClaw runs even when Codex exits with status 0.

## Best Possible Solution

Add structured Codex JSONL sandbox-denial classification before success result construction, emit a terminal `FailoverError` reason, and cover denial-only plus false-positive cases with focused tests.

## Reproduction Assessment

Yes, source-reproducible. The reported `codex exec --json --sandbox read-only` path maps to current code where exit-0 JSONL output bypasses failover handling and can be recorded as completed success; I did not run a live Codex command in this read-only pass.

Yes. A structured-field detector like the approach in https://github.com/openclaw/openclaw/pull/39460 is the narrowest maintainable fix; https://github.com/openclaw/openclaw/pull/66081 changes final-answer sourcing and does not classify semantic failures.
