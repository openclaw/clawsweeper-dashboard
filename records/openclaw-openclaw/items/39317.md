## Summary

Keep open. Current main still only enters CLI failure handling when the subprocess exits nonzero or terminates abnormally; exit-0 `codex-cli` JSONL is parsed as ordinary output, has no semantic sandbox-denied flag, and is then recorded as a successful completed CLI run. The focused fix candidate #39460 is closed unmerged, while the related #66081 artifact-finalization work is complementary and does not address failure classification.

## What This Changes

The issue asks OpenClaw to detect sandbox-denied or failed Codex CLI JSONL action signals even when `codex exec --json` exits 0, and to propagate that run as failed instead of successful.

## Best Possible Solution

Add a focused structured JSONL failure-classification path for `codex-cli`: detect real error/status/tool/action/file-change sandbox-denied signals before building a successful run result, surface a dedicated failover reason/status, avoid matching ordinary assistant message text, and cover parser plus runner behavior with regression tests.

## Reproduction Assessment

Yes. The report gives a concrete `codex exec --json --sandbox read-only` reproduction, and current source proves an exit-0 JSONL stream bypasses failure handling and is recorded as success; I did not run the live mutation repro because this review was read-only.

## Solution Assessment

Yes. A #39460-style structured detector is the narrowest maintainable fix; relying on `--output-last-message` artifacts or stdout selection would improve final reply reliability but would not classify semantic sandbox-denied failures.
