## Summary

Keep open. Current main still returns successful exit-0 CLI JSONL output as normal parsed text, and the linked fix PR was closed unmerged rather than shipped.

## What This Changes

OpenClaw should detect structured Codex CLI sandbox/write-denied JSONL signals and fail the run instead of treating an exit-0 process as successful.

## Best Possible Solution

Add structured sandbox-denial detection at the CLI output/runner boundary, propagate a typed local failover, and cover denial-only plus false-positive JSONL cases.

## Reproduction Assessment

Yes by source inspection, not by a live command in this read-only review. Current main parses exit-0 CLI JSONL/stdout into a successful result without a sandbox-denied failure check.

## Solution Assessment

Yes. The prior PR's structured-field approach is the right direction, but a fresh fix should avoid broad assistant-text keyword matching and handle denial-only JSONL before empty-text fallback.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/39317 on current main. Codex CLI can exit 0 while structured JSONL reports a sandbox/write-denied tool or action failure, and OpenClaw currently treats the parsed text as a successful assistant reply. Add structured sandbox-denial detection at the CLI output/runner boundary, propagate it as a FailoverError with an appropriate typed reason/status, and keep auth-profile failure tracking from treating it as provider auth failure. Add regression tests for top-level/nested structured denial records, denial-only JSONL with no message item, and false-positive cases where ordinary assistant/message text merely discusses sandboxing. Do not add broad keyword scanning over ordinary assistant message text, do not change unrelated CLI finalization behavior such as output-last-message artifact handling, and add a maintainer-owned CHANGELOG.md fix entry if this is treated as user-visible.
