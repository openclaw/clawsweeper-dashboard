## Summary

Keep open: current main still has a source-level false-success path for exit-0 JSONL CLI runs, even though the original bundled `codex-cli` route has since migrated to the Codex app-server runtime. The prior focused fix was closed unmerged, and the related output-last-message PR explicitly does not change failure classification.

## What This Changes

The report asks OpenClaw to classify exit-0 Codex CLI JSONL sandbox denials as failed runs instead of successful CLI outputs.

## Best Possible Solution

Add scoped structured Codex JSONL sandbox-denial classification before CLI success construction, with terminal/local failover semantics and focused regression coverage.

## Reproduction Assessment

Yes, source-level: an exit-0 JSONL CLI run reaches `parseCliOutput` and success construction with no structured sandbox-denial channel. I did not run live Codex in this read-only pass, and the old bundled `codex-cli` route has since moved to the Codex app-server runtime.

## Solution Assessment

Yes. A structured-field detector like the closed implementation in https://github.com/openclaw/openclaw/pull/39460 remains the narrow maintainable direction, but it should be scoped to real Codex CLI JSONL signals and current runtime boundaries.

Add structured sandbox-denial detection before successful `CliOutput` construction, scoped to Codex CLI JSONL or an explicit dialect so generic JSONL backends do not gain broad keyword behavior. Add a terminal/local `sandbox_denied` failover reason with status mapping, and make sure auth-profile failure tracking does not treat it as provider auth health. Add focused tests for structured error/tool/status denial records, denial-only JSONL with no message item, and false-positive cases where the assistant merely discusses sandboxing.
