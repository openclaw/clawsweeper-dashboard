## Summary

Keep open for maintainer review: current main and the latest release still allow the hidden FTS fallback, while this PR is a focused mitigation with docs/tests but intentionally changes upgrade behavior for explicit remote embedding providers.

## What This Changes

The branch makes memory-core treat explicitly configured non-local memory embedding providers as required for search/sync, returning unavailable instead of silently using FTS-only fallback, with docs and focused tests.

## Best Possible Solution

Land the internal required-provider guard once maintainers accept the fail-closed upgrade behavior, while preserving optional FTS behavior for unset, legacy auto, provider none, and local provider paths.

## Reproduction Assessment

Yes. Current main and v2026.6.1 source show providerless memory search can return keyword-only results when FTS is available, and the linked issue includes a real active-memory timeout trace; I did not run a live repro in this read-only review.

## Solution Assessment

Yes, this is an acceptable mitigation rather than a broad rewrite: it keeps the requirement internal, guards search/sync, preserves intentional optional fallback paths, and documents the upgrade-sensitive behavior. The remaining question is maintainer acceptance of fail-closed semantics, not a narrow code defect.
