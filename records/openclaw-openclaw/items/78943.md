## Summary

Keep open: current main still has a source-level gap where the CLI turn compaction lifecycle resolves the default context engine without first registering the built-in legacy engine, while docs and nearby runtime paths establish that legacy should be available by default.

## What This Changes

Request to initialize built-in context engines in the main CLI-runtime path so main-agent CLI transcript and compaction work can resolve the default legacy context engine.

## Best Possible Solution

Add the built-in context-engine initialization step to the CLI compaction/runtime resolution path and cover it with a focused regression test mirroring the existing subagent initialization-order test.

## Reproduction Assessment

Yes, source-reproducible with high confidence: default legacy resolution throws when the registry is empty, and the CLI compaction path currently reaches resolveContextEngine without the initializer. I did not run a live gateway turn in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable fix is to call or inject ensureContextEnginesInitialized immediately before CLI context-engine resolution, matching the subagent and embedded runtime paths.
