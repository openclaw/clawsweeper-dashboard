## Summary

Keep open, but decompose the proposal. Current main already has artifact hashing, digest verification, GitHub source-resolution metadata, and a workspace ClawHub lockfile, while the requested public provenance, persisted digest/moderation facts, frozen installs, and capability enforcement remain incomplete. The sandbox portion also needs a corrected threat model because ordinary OpenClaw skills are instruction bundles rather than direct registrants of the named lifecycle hooks.

Likely related people: @vincentkoc is the strongest recent routing candidate for the consolidated ClawHub lifecycle implementation; Peter Steinberger is also connected through substantial earlier history in the central ClawHub and skill-security files.

## What This Changes

Request public source and moderation provenance for ClawHub skills, digest-pinned reproducible installs with richer lock and audit semantics, and deny-by-default skill capability sandboxing.

## Best Possible Solution

Split this into linked owner-specific work: expose authoritative source, moderation, and artifact metadata from ClawHub; extend the existing `.clawhub/lock.json` and verification path with persisted digests and frozen/audit semantics; and handle capability declarations and enforcement in a separate security RFC targeting actual agent-tool and plugin execution boundaries with a tested upgrade policy.

## Reproduction Assessment

Yes for the concrete provenance and persistence gaps: the live API lacks a source URL, and current lockfile types omit artifact digest, source, and moderation snapshots. No high-confidence reproduction supports the claim that ordinary OpenClaw skills directly execute the named lifecycle hooks.

## Solution Assessment

No, not as one bundled implementation. The narrowest maintainable path extends the existing ClawHub resolution, hashing, lock, and verification primitives while treating runtime capability enforcement as a separate compatibility-sensitive product decision.
