## Summary

Keep this open for maintainer security/product follow-up: current main has partial ClawHub skill verification and a minimal `.clawhub` install record, but it still lacks source provenance on the public skill detail shape, digest-pinned installs, a reproducible lock/audit workflow, and sandbox-by-default skill capability enforcement. Related open security trackers cover pieces of the work, but none fully replaces this issue's combined ClawHub source/provenance, lockfile, and sandbox ask.

## What This Changes

The issue requests ClawHub skills supply-chain hardening: source provenance, moderation detail, pinned installs, a workspace lockfile, and sandbox-by-default execution.

## Best Possible Solution

Keep this open until maintainers either split it into canonical owner-approved milestones or define one ClawHub skills trust plan covering provenance disclosure, authenticated artifact identity, persisted lock/audit state, and staged capability sandbox enforcement.

## Reproduction Assessment

Yes at the source/live-API level: the sampled ClawHub skill detail lacks `sourceUrl`, current OpenClaw skill detail and lock metadata lack the requested provenance and digest fields, and sandbox config defaults to `off`. This is a missing security capability rather than a regression in an established contract.

## Solution Assessment

No, not as one autonomous implementation. The direction is sound, but the best maintainable path is a maintainer-approved split between ClawHub service disclosure policy, OpenClaw install/lock/audit enforcement, and the existing skill manifest/sandbox architecture trackers.
