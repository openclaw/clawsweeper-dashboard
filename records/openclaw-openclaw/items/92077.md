## Summary

Keep this issue open because the central provenance complaint reproduces against the live ClawHub API and meaningful OpenClaw work remains. ## What This Changes

Requests public skill source provenance, detailed moderation reports, hash-pinned installations, a workspace skills lockfile, and sandbox-by-default execution with explicit trust promotion.

## Best Possible Solution

Extend the existing ClawHub integrity and verification contracts with public immutable source and artifact identity, define a user-owned frozen workspace manifest only if it has a non-duplicative contract, and handle per-skill capability isolation in a separate compatibility-sensitive RFC.

## Reproduction Assessment

Yes for the central marketplace provenance complaint: the live ClawHub API currently omits source and artifact-hash fields and publishes only summarized moderation metadata. The sandbox portion is a requested change to the current trusted-installed-code model, not a reproduced bug.

## Solution Assessment

No, not as one implementation item. The verified ClawHub service gap, core frozen-install state, and per-skill execution isolation need separate owners, threat models, and acceptance criteria while reusing the existing integrity plumbing.
