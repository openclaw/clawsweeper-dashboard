## Summary

Current main partially addresses the report with GitHub-backed source resolution and a workspace ClawHub lockfile, but the central security goals remain: persisted trusted digests, frozen or auditable installs, public moderation provenance, and per-skill execution isolation are not implemented. Related security issues overlap substantially but do not preserve every distinct requirement, so this issue remains necessary pending maintainer decomposition.

## What This Changes

Requests ClawHub source provenance, moderation transparency, content-addressed installs, a richer skills lockfile, and default sandboxing for installed skills.

## Best Possible Solution

Split the roadmap by owner while retaining cross-links: ClawHub should expose and validate source and moderation provenance; OpenClaw should extend its existing lock/origin records with trusted artifact metadata and frozen/audit flows; the existing integrity issue should own install/load verification; and sandbox-by-default should proceed through a separately approved capability and migration design.

## Reproduction Assessment

Yes for the concrete gaps: current main receives or computes source and digest data but does not persist the requested trusted state or offer frozen/audit semantics. Sandbox-by-default and capability manifests are new product behavior rather than failures of an existing documented contract.

## Solution Assessment

No as a single implementation unit; extending the existing source/lock foundation is appropriate, but hosted transparency, artifact verification, reproducibility, and sandbox policy should be delivered through separate owner-aligned changes.
