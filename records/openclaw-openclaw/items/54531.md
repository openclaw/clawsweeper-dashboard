## Summary

Keep open. Current main has adjacent route-preservation fixes, but it still intentionally blocks implicit WebChat/dashboard-to-external origin routing without an explicit delivery route, so the remaining force/sticky-origin policy needs maintainer product and privacy review.

## What This Changes

Request to make agent replies reliably return to the originating messaging channel, with proposed force-origin and fallback routing configuration for Telegram, Discord, WhatsApp, and similar channels.

## Best Possible Solution

Define a maintainer-approved origin-delivery invariant that captures the external route for the active turn, preserves WebChat/dashboard privacy boundaries, adds an explicit opt-in config only if needed, and covers dashboard-active external replies with focused regression tests.

## Reproduction Assessment

Yes, at source level. Current main still has an internal-WebChat/no-ExplicitDeliverRoute gate and tests that assert no external route is created in that case.

## Solution Assessment

Unclear as proposed. A force-origin flag or persisted-origin route addresses the observed pain, but the best fix needs maintainer policy for stale sessions, dashboard-originated turns, duplicate delivery, and privacy before implementation.
