## Summary

Keep open. Current main still blocks implicit WebChat/dashboard-originated turns from routing replies to an external originating channel unless an explicit delivery route is set, and the prior persisted-origin implementation is closed and unmerged. The report is valid, but changing the default cross-surface delivery policy is privacy-sensitive and needs maintainer product/security judgment before an automated repair PR is safe.

## What This Changes

Request a default or configurable reply-routing behavior that forces agent responses back to the originating Telegram, Discord, WhatsApp, or WebChat channel, with fallback behavior when origin is unclear.

## Best Possible Solution

Keep this open for a maintainer-approved reply-routing design. The end state should define either an explicit opt-in force-origin setting or a persisted external-origin route with freshness/privacy safeguards, then add regression coverage for WebChat/dashboard sessions, explicit-delivery exceptions, channel-scoped sessions, and stale route handling.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists in the current tests: WebChat provider/surface with an external OriginatingChannel and no ExplicitDeliverRoute asserts that routeReply is not called; I did not execute tests because this was a read-only review.

## Solution Assessment

Unclear. The requested persisted-origin/default-delivery direction matches the user problem, but it is not clearly the safest fix until maintainers choose the cross-surface delivery policy; the existing ExplicitDeliverRoute path is safer but does not solve the requested default behavior.
