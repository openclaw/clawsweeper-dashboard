## Summary

Keep open. Current main still preserves some external route metadata, but it intentionally does not implement the requested forced-origin behavior for normal WebChat/dashboard turns: implicit internal WebChat turns are blocked from routing to an external originating channel unless an explicit delivery route is set. That remaining policy is privacy-sensitive and should not be closed as implemented or queued as an automated repair without maintainer direction.

## What This Changes

The issue requests a configurable or automatic reply-routing behavior that forces agent responses back to the originating messaging channel, such as Telegram, Discord, or WhatsApp, even when the session is viewed through WebChat or a dashboard.

## Best Possible Solution

Keep this open for a maintainer-reviewed reply-routing design: either an explicit opt-in force-origin setting or an immutable first-external-origin route with stale-session, account, thread, and privacy safeguards, plus docs and regression coverage for WebChat/dashboard sessions, channel-scoped sessions, and explicit-delivery exceptions.
