## Summary

Keep open: current main and v2026.5.22 still intentionally block implicit internal WebChat/dashboard-to-external origin routing unless `ExplicitDeliverRoute` is set. Related merged fixes improved group defaults, message-tool route drift, and persisted delivery context, but the central force-origin policy remains a product/security decision.

## What This Changes

Request to add configurable or sticky origin-channel reply delivery so Telegram, Discord, WhatsApp, and similar channel responses generated from dashboard/internal contexts are sent back to the originating chat.

## Best Possible Solution

Define a maintainer-approved delivery invariant that carries the original turn route forward and delivers final user-facing replies to that route only under explicit, secure semantics, with focused regressions for internal WebChat/dashboard, Codex message-tool-only, and representative external channels.

## Reproduction Assessment

Yes, at source level. Current main has a focused routing-policy gate and regression test showing internal WebChat turns with external origin metadata do not route externally unless `ExplicitDeliverRoute` is true; I did not run a live channel repro.

## Solution Assessment

Unclear until maintainers choose the policy. A sticky-origin invariant is plausible, but a broad force-origin config/default is not automatically the safest fix because it changes privacy, stale-route, duplicate-delivery, and sendPolicy behavior.
