## Summary

Keep open. Current main has a provider-agnostic origin reply router and route-preservation work, but the routing decision still has internal-WebChat and same-surface gates, the requested fallback/force policy is not implemented, and related current reports show the delivery-loss problem is still active across channel-scoped sessions.

## What This Changes

The issue requests reliable delivery of agent replies back to the originating messaging channel, with a proposed force-origin/fallback configuration for unclear origins.

## Best Possible Solution

Define a safe turn-origin delivery invariant that preserves the captured channel/target through agent, ACP, tool, and queued follow-up paths, and add any fallback or mirroring only as an explicit maintainer-approved policy with security coverage.

## Reproduction Assessment

Yes, source-reproducible but not live-reproduced here: docs promise origin-channel replies, while current routing code can suppress origin routing for internal WebChat turns and same-surface origins. The issue discussion also includes multiple channel reports and a closed unmerged attempted fix.

## Solution Assessment

No, the proposed unconditional force/fallback config is not clearly the best fix. The safer direction is to preserve a durable turn-origin route and make any fallback or mirroring an explicit security-reviewed product policy.
