## Summary

Keep open. Current main has related route-preservation fixes, but it still intentionally blocks implicit internal WebChat/dashboard turns from routing to an external origin unless `ExplicitDeliverRoute` is set, so the requested force/sticky-origin policy is not implemented and needs maintainer product/privacy approval.

## What This Changes

Request to force or configure agent replies to return to the originating messaging channel instead of remaining only on dashboard/WebChat/internal surfaces.

## Best Possible Solution

Define a maintainer-approved origin-delivery policy, likely either an explicit opt-in force-origin setting or durable per-turn external origin capture, with privacy/stale-session safeguards and regression coverage for dashboard-active external-channel sessions plus explicit-delivery exceptions.

## Reproduction Assessment

Yes, at source level. Current main still has an internal-WebChat/no-`ExplicitDeliverRoute` gate and a regression test asserting no external route for that path; no live transport reproduction was run in this read-only review.

## Solution Assessment

Unclear as proposed. A sticky per-turn external origin or explicit opt-in force-origin setting could solve the observed loss, but maintainers need to define stale-session and dashboard privacy rules first.
