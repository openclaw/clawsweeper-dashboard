## Summary

Keep open: current main and v2026.5.22 still block implicit internal WebChat/dashboard-to-external origin routing unless explicit delivery is set. Existing route-preservation work reduced some drops, but the requested force-origin/config behavior remains a privacy-sensitive product decision, with narrower route-state bugs tracked separately.

## What This Changes

The issue requests a config or policy that forces agent replies to deliver to the originating Telegram, Discord, WhatsApp, or Web UI channel, with fallback when origin is unclear.

## Best Possible Solution

Define a maintainer-approved, default-off origin-delivery policy that preserves inbound external routes, respects ExplicitDeliverRoute/sendPolicy/stale-route checks, and leaves narrower route-state bugs to their canonical fixes.

## Reproduction Assessment

Yes, at source level. Current main and v2026.5.22 show internal WebChat with an external origin does not route externally without ExplicitDeliverRoute, and focused tests assert routeReply is not called.

## Solution Assessment

Unclear as proposed: unconditional force-origin or fallback routing is not the safest fix because it changes cross-surface privacy boundaries. The maintainable path is a default-off or otherwise explicit policy plus focused fixes for independently reproduced route-state bugs.
