## Summary

Keep open: current main has fixed some related delivery defaults and route-drift cases, but it still intentionally preserves private/internal reply modes and blocks external-origin routing from internal WebChat/dashboard turns unless explicit delivery is set. The remaining request is a product/security delivery-policy decision, not conservative cleanup closure.

## What This Changes

Request a core delivery policy or configuration that forces agent replies back to the originating messaging channel instead of remaining only in dashboard/internal surfaces.

## Best Possible Solution

Define one maintainer-approved turn-origin delivery contract that delivers normal user-request finals to the source route while preserving explicit private, ambient, and message-tool-only modes with focused Telegram/Discord-style regression coverage.

## Reproduction Assessment

Yes for a source-level reproduction, but not a live end-to-end run in this review. Current tests explicitly preserve both the internal-webchat non-routing case and message-tool-only private-final behavior.

## Solution Assessment

No, an unconditional force-origin flag is not yet proven to be the safest solution. The maintainable path is a product/security-approved delivery invariant with explicit exceptions for private, ambient, and opt-in message-tool-only turns.
