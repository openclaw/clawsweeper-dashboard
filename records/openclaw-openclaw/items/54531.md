## Summary

Keep open. Current main preserves external delivery context for some narrower paths, but the requested force/sticky-origin behavior remains absent for normal internal WebChat/dashboard turns without an explicit delivery route.

## What This Changes

The issue requests a forced or sticky origin-channel reply behavior so Telegram, Discord, WhatsApp, and Web UI responses are delivered back to the originating surface.

## Best Possible Solution

Add a maintainer-approved reply-routing design, likely durable per-turn origin metadata or an explicit opt-in force-origin setting, with stale-session and UI-initiated-reply safeguards plus regression coverage.

## Reproduction Assessment

Yes. Current source and tests show that an internal WebChat turn with an external `OriginatingChannel` and no `ExplicitDeliverRoute` does not route back to that external channel.

## Solution Assessment

Unclear. The requested blanket force-origin config is not obviously the safest fix; a sticky per-turn origin design with explicit privacy safeguards likely needs maintainer policy review first.
