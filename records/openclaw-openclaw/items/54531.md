## Label Justifications

- `P1`: The report describes recurring silent non-delivery of final agent replies in real messaging-channel workflows.
- `impact:message-loss`: The central failure is generated final text not reaching Telegram, Discord, WhatsApp, or similar originating chats.
- `impact:session-state`: The suspected failure depends on origin route/session metadata being lost, overridden, or gated when execution moves through internal/dashboard surfaces.
- `impact:security`: The requested force-origin behavior changes cross-surface delivery boundaries and could expose replies to the wrong external chat if implemented too broadly.

## Summary

Keep open: current main and the latest release still intentionally block implicit external-origin routing from internal WebChat/dashboard turns unless `ExplicitDeliverRoute` is set, while Codex/source reply policy can still keep ordinary final text private unless the message tool or explicit config delivers it. The issue remains the canonical cross-surface origin-routing policy item and needs maintainer product/security approval rather than cleanup closure.

## What This Changes

The issue requests a force/sticky origin-channel delivery policy so final agent replies generated from Telegram, Discord, WhatsApp, or similar channels return to that originating chat instead of staying on an internal/dashboard surface.

## Best Possible Solution

Define a maintainer-approved delivery invariant that carries the original turn route forward and delivers final user-facing replies to that route only under explicit, secure semantics, with focused regressions for internal WebChat/dashboard, Codex message-tool-only, and representative external channels.

## Reproduction Assessment

Yes, at source level. Current main has a focused routing-policy test and implementation gate showing internal WebChat turns with external origin metadata do not route externally unless `ExplicitDeliverRoute` is true; I did not run a live channel repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the policy. A sticky-origin invariant is plausible, but a broad unconditional force-origin switch is not automatically the safest fix because it changes privacy, stale-route, duplicate-delivery, and sendPolicy behavior.
