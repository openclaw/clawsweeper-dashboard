## Summary

Keep open. Current main and v2026.6.8 still block implicit WebChat/dashboard-to-external origin delivery unless an explicit route is set, the prior persisted-origin PR is closed unmerged, and recent comments report the same user-visible non-delivery across more channels.

## What This Changes

Request a force-origin reply behavior so final agent responses are delivered back to the originating messaging channel instead of remaining only in internal/dashboard surfaces.

## Best Possible Solution

Keep this as the canonical maintainer-reviewed origin-delivery policy item, then implement sticky per-turn external origin or explicit opt-in force-origin semantics with privacy safeguards, regression tests, and live channel proof.

## Reproduction Assessment

Yes, at source level. Current source and tests preserve non-delivery from internal WebChat/dashboard to an external origin unless ExplicitDeliverRoute is set; I did not run a live Telegram/Discord/Tlon repro in this read-only review.

## Solution Assessment

No, not as a broad config-only fallback. The best fix needs maintainer-chosen sticky per-turn origin or explicit opt-in delivery semantics with route freshness, account/thread authorization, and privacy safeguards.
