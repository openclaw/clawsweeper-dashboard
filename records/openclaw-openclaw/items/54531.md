## Summary

Keep open. Current main has related route-preservation work, but it still intentionally blocks implicit WebChat/UI-to-external delivery unless an explicit route is set, the requested force/sticky-origin config is absent, and the prior persisted-origin PR was closed unmerged.

## What This Changes

The issue requests a forced or sticky reply route so agent responses return to the originating messaging channel instead of only appearing in dashboard/UI surfaces.

## Best Possible Solution

Keep this open for a maintainer-reviewed reply-routing design that preserves durable per-turn external origin metadata only where it is safe, with stale-session and UI-initiated-reply safeguards.

## Reproduction Assessment

Yes. Current source-level reproduction is covered by existing tests: an external origin with `Provider`/`Surface` set to WebChat and no `ExplicitDeliverRoute` does not call `routeReply`, and UI clients on main sessions do not inherit external routes.

## Solution Assessment

Unclear. A forced global config flag is not the safest default; a sticky turn-origin design with explicit safeguards is likely more maintainable, but that policy needs maintainer approval before automation should change routing.
