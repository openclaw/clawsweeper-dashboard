## Summary

Keep open. Current main and v2026.6.8 still intentionally block implicit WebChat/dashboard-to-external origin delivery unless an explicit route is present, and the fresh Tlon confirmation means this is not stale cleanup.

## What This Changes

The issue requests a force-origin reply behavior, potentially via config or sticky turn-origin routing, so generated replies return to the originating Telegram, Discord, WhatsApp, Tlon, or similar channel instead of remaining internal.

## Best Possible Solution

Keep this as the canonical maintainer-reviewed origin-routing policy item and implement either sticky per-turn external origin or an explicit opt-in force-origin mode with stale-route, privacy, and live-channel proof.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 still block internal WebChat/dashboard turns from external-origin delivery without ExplicitDeliverRoute, and current tests assert both the blocked default and the explicit-delivery exception.

## Solution Assessment

No, not as an unconditional config-only fallback. The best fix needs maintainer-chosen sticky per-turn origin or explicit delivery semantics with route freshness and privacy safeguards.
