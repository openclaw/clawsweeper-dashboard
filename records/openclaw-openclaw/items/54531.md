## Summary

Keep open. Current main has narrower route-preservation work, but it still intentionally blocks implicit WebChat/dashboard-origin turns from routing replies to an external channel unless explicit delivery is enabled; changing that default needs product and privacy review.

## What This Changes

The issue requests a forced or sticky reply-routing behavior so agent replies always return to the originating Telegram, Discord, WhatsApp, or Web UI channel.

## Best Possible Solution

Define a maintainer-approved routing policy, likely an explicit opt-in force-origin setting or sticky per-turn external origin metadata with stale-session, UI-origin, and group/channel privacy safeguards plus regression coverage.

## Reproduction Assessment

Yes. Current source and tests show that an internal WebChat turn with an external origin and no explicit delivery route does not route back to that external channel.

## Solution Assessment

Unclear. The requested behavior addresses real user pain, but the safest design needs maintainer policy because blanket forced delivery can cross privacy boundaries between dashboard sessions and external chats.
