## Summary

Keep open. Current main still intentionally blocks implicit WebChat/dashboard-to-external reply routing unless an explicit delivery route is set, and the linked persisted-origin PR is closed and unmerged. The remaining change is a privacy-sensitive reply-routing policy decision, not conservative cleanup.

## What This Changes

The issue requests a configurable or automatic way to force generated agent replies back to the originating Telegram, Discord, WhatsApp, or Web UI channel instead of leaving them only in internal/dashboard surfaces.

## Best Possible Solution

Keep this open for a maintainer-approved reply-routing design. The desired end state is either an explicit opt-in force-origin setting or a persisted first-external-origin route with stale-route clearing, account/thread/session scoping, and regression coverage for WebChat/dashboard, native UI, external channel sessions, shared main sessions, and `ExplicitDeliverRoute` exceptions.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists on current main: WebChat provider/surface with an external `OriginatingChannel` and no `ExplicitDeliverRoute` is expected not to call `routeReply`; tests were not executed because this was a read-only no-artifact review.

## Solution Assessment

Unclear. The current `ExplicitDeliverRoute` mechanism is the safer existing path but does not satisfy the requested default behavior; a persisted-origin or force-origin fix needs maintainer-approved privacy safeguards before it is the best solution.
