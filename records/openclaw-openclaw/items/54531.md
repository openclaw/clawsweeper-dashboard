## Summary

Keep open. Current main still intentionally blocks implicit WebChat/dashboard replies from being delivered to an external originating channel unless an explicit delivery route is present, and the exact persisted-origin proposal in #57802 remains closed and unmerged.

## What This Changes

Request a default or durable reply-routing behavior that sends agent responses back to the messaging channel where the user originated the turn.

## Best Possible Solution

Define a maintainer-approved route model that keeps external origin metadata durable for a turn or channel-scoped session while preserving explicit opt-in/freshness checks for dashboard and WebChat delivery, then cover WebChat, subagent/tool-call, system-event, and stale-route cases with regression tests.

## Reproduction Assessment

Yes for the WebChat/dashboard routing guard: current tests construct Provider/Surface webchat with an external OriginatingChannel/OriginatingTo and assert that routeReply is not called unless ExplicitDeliverRoute is true. The newer tool-call context-dropping angle still needs a focused trace or test fixture.

## Solution Assessment

Unclear. Persisted or sticky origin metadata is the right direction for the reported failure, but forcing external delivery by default is not clearly the safest implementation until maintainers choose the privacy and stale-route policy.
