## Label Justifications

- `P1`: The issue describes real channel workflows where generated replies can be silently unavailable to users in Telegram, Discord, WhatsApp, or similar clients.
- `impact:message-loss`: The central symptom is assistant replies being generated but not delivered to the originating messaging channel.
- `impact:session-state`: The suspected causes involve origin metadata, deliveryContext, lastRoute fields, and surface drift across sessions or turns.
- `impact:security`: The requested cross-surface force-origin behavior can bridge internal/dashboard replies to external chats and needs explicit privacy safeguards.

## Summary

Keep open. Current main still intentionally blocks implicit internal WebChat/dashboard turns from routing to an external origin unless `ExplicitDeliverRoute` is set, and the requested force/sticky-origin policy is still a product/privacy decision rather than implemented behavior.

## What This Changes

The issue requests a configurable or sticky-origin delivery policy so replies from Telegram, Discord, WhatsApp, or similar inbound channels return to the originating channel even when a dashboard or internal surface is active.

## Best Possible Solution

Define a maintainer-approved cross-surface origin-delivery policy, likely default-off or explicit per-turn delivery with durable external origin capture, stale-route safeguards, docs, and regression coverage that preserves the WebChat no-inheritance boundary.

## Reproduction Assessment

Yes, at source level. Current main still has an internal-WebChat/no-`ExplicitDeliverRoute` gate and a regression test asserting no external route for that path; no live transport reproduction was run in this read-only review.

## Solution Assessment

Unclear as proposed. A sticky per-turn external origin or explicit opt-in force-origin setting could solve the observed loss, but maintainers need to define stale-session, dashboard-origin, and privacy rules first.
