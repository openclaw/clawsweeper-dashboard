## Label Justifications

- `P1`: The issue describes user-visible messaging workflows where generated replies can be silently unavailable in Telegram, Discord, WhatsApp, or similar clients.
- `impact:message-loss`: The central symptom is assistant replies being generated but not delivered to the originating messaging channel.
- `impact:session-state`: The suspected cause and current implementation involve origin metadata, deliveryContext, last-route fields, and surface drift across turns or sessions.
- `impact:security`: The requested cross-surface force-origin behavior could bridge internal/dashboard replies to external chats and needs explicit privacy safeguards.

## Summary

Keep open: current main still preserves some established external routes, but it intentionally blocks implicit internal WebChat/dashboard turns from routing to an external origin unless an explicit delivery route is set, so the requested force/sticky-origin policy remains unresolved.

## What This Changes

The issue requests forced or sticky delivery of agent replies back to the originating messaging channel across Telegram, Discord, WhatsApp, and WebChat/dashboard surfaces.

## Best Possible Solution

Keep this open for a maintainer-approved cross-surface reply-routing policy, likely an explicit opt-in force-origin mode or turn-sticky external origin with stale-route safeguards, docs, and regression coverage.

## Reproduction Assessment

Yes, at source level: current main still has an internal-WebChat/no-ExplicitDeliverRoute gate and a regression test asserting no external route for that path. I did not run a live Telegram, Discord, or WhatsApp transport reproduction in this read-only review.

## Solution Assessment

Unclear as proposed. A sticky per-turn external origin or explicit force-origin setting could solve the observed non-delivery, but maintainers need to define stale-session, dashboard-origin, and privacy rules first.
