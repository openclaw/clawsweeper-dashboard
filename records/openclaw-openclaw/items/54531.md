## Summary

Keep open: current main preserves established external route metadata better than the original closed PR, but it still intentionally blocks implicit internal WebChat/dashboard-to-external origin routing unless the turn carries an explicit deliver route. The remaining request is a cross-surface delivery policy decision, not a cleanup close.

## What This Changes

The issue asks OpenClaw to guarantee that replies generated for Telegram, Discord, WhatsApp, and similar channel-originated turns are delivered back to the originating channel, with a proposed force-origin configuration fallback.

## Best Possible Solution

Define a maintainer-approved, default-safe origin delivery invariant that reuses deliveryContext/ExplicitDeliverRoute, specifies route freshness and sendPolicy behavior, and adds focused route plus live-channel proof before changing defaults.

## Reproduction Assessment

Yes, at source level. Current main and v2026.5.22 both have a routing policy and regression test showing internal WebChat turns with an external origin do not route externally unless ExplicitDeliverRoute is true.

## Solution Assessment

Unclear as requested. A force-origin option may be part of the answer, but the maintainable fix needs a reviewed cross-surface delivery policy rather than an unconditional default route change.
