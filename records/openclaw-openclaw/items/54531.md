- `P1`: The issue describes active user-facing channel workflows where generated replies can be silently unavailable in Telegram, Discord, WhatsApp, or similar clients.

## Summary

Keep open: current main still preserves some external route metadata, but it intentionally blocks implicit internal WebChat/dashboard turns from routing replies back to an external originating channel unless `ExplicitDeliverRoute` is set, so the requested force/sticky-origin policy remains unresolved and security-sensitive.

## What This Changes

The issue requests forced or sticky delivery of agent replies back to the originating Telegram, Discord, WhatsApp, or similar channel instead of leaving responses only on internal/dashboard surfaces.

## Best Possible Solution

Keep this open for a maintainer-approved cross-surface reply-routing design, likely an explicit opt-in force-origin setting or sticky per-turn external origin with stale-route, privacy, duplicate-delivery, and regression-test safeguards.

## Reproduction Assessment

Yes, at source level: current main still has an internal-WebChat/no-`ExplicitDeliverRoute` gate plus tests asserting no implicit external route for that path. I did not run a live Telegram, Discord, or WhatsApp transport reproduction in this read-only review.

## Solution Assessment

Unclear as proposed. A sticky per-turn external origin or explicit force-origin setting could solve the observed non-delivery, but maintainers need to define stale-session, dashboard-origin, duplicate-delivery, and privacy rules before implementation.
