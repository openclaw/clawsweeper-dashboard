## Summary

Keep open. Current main has fixed several adjacent route-preservation and visible-reply regressions, but it still does not provide a maintainer-approved force-origin/sticky-origin policy for internal WebChat/dashboard turns, and the remaining behavior is explicitly covered by current tests rather than fully implemented.

## What This Changes

Request to ensure agent replies from Telegram, Discord, WhatsApp, and similar channels are delivered back to the originating channel, potentially through a force-origin configuration option and fallback route.

## Best Possible Solution

Define a maintainer-approved origin-delivery policy that captures a durable per-turn external route, preserves current WebChat privacy boundaries, adds an explicit opt-in config only if needed, and covers external-channel plus dashboard-active regressions with focused tests.

## Reproduction Assessment

Yes, at source level. Current main still has an explicit internal-WebChat/no-ExplicitDeliverRoute gate and tests that assert no external route in that case; I did not run a live transport reproduction.

## Solution Assessment

Unclear as proposed. A force-origin flag or persisted origin route addresses real pain, but the maintainable solution needs maintainer policy for stale sessions, dashboard-originated turns, and privacy before implementation.
