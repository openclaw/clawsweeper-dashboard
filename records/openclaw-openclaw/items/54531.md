## Summary

Keep open: current main has route-preservation and explicit-delivery fixes, but it still intentionally keeps ordinary WebChat/dashboard turns local unless an explicit external delivery route or a special internal handoff path is present, so the requested force-origin/fallback behavior remains a product and security decision.

## What This Changes

The issue requests a configurable or zero-config way to force agent replies back to the originating external channel, with fallback behavior when the active UI/internal surface would otherwise keep replies local.

## Best Possible Solution

Define one explicit origin-delivery policy for channel-scoped turns, with safe route freshness/account/thread checks, opt-in dashboard-to-external mirroring or fallback semantics, and tests plus live channel proof across at least one external channel.

## Reproduction Assessment

Yes at source level: current main intentionally keeps ordinary WebChat/UI turns local unless an ExplicitDeliverRoute or the narrower sessions_send inherited route is present; no live Telegram/Discord/Tlon run was performed in this read-only review.

## Solution Assessment

The broad force-origin/fallback request is useful but not ready as an unconditional implementation; the best fix needs a maintainer-approved routing and security policy that separates origin delivery, optional mirroring, fallback, and delivery reconciliation.
