## Summary

Keep open. Current main still lacks the requested force-origin or sticky-origin behavior, the closed #57802 implementation did not merge, and current routing tests intentionally keep internal WebChat/dashboard turns off external channels unless explicit delivery is set.

## What This Changes

The issue requests reliable delivery of agent replies back to the originating Telegram, Discord, WhatsApp, or Web UI channel, with a proposed force-origin or sticky-origin routing option.

## Best Possible Solution

Define a maintainer-approved reply-routing design that preserves durable per-turn external origin metadata where safe, with stale-session and UI-initiated-reply safeguards plus regression coverage across the reply dispatcher and Gateway chat send paths.

## Reproduction Assessment

Yes, at source level. Current main has explicit tests and routing code showing that an internal WebChat/dashboard turn with an external origin and no explicit delivery route stays on the internal surface instead of calling `routeReply`.

## Solution Assessment

Unclear. The requested global force-origin config is not clearly the safest solution; a sticky turn-origin design with explicit delivery, staleness, and privacy guards is likely more maintainable but needs maintainer policy approval.
