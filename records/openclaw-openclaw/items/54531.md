## Summary

Keep open. Current main and latest release still preserve the internal WebChat/dashboard gate unless an explicit external delivery route is set, while route-preservation work only covers narrower session/event cases and the earlier persisted-origin PR is closed unmerged.

## What This Changes

The issue requests reliable delivery of generated replies back to the originating messaging channel when dashboard, WebChat, or internal surfaces are also involved.

## Best Possible Solution

Keep this as the canonical maintainer-reviewed origin-routing policy item and implement a sticky turn-origin or explicit-delivery design with stale-route/privacy safeguards and focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main blocks internal WebChat/dashboard turns from routing to an external origin unless `ExplicitDeliverRoute` is true, and the current tests assert that exact behavior.

## Solution Assessment

No, not as the proposed config-only fallback. The maintainable fix needs a policy choice between sticky per-turn origin delivery and explicit opt-in delivery, with safeguards for stale or unintended external routes.
