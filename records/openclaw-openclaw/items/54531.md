## Summary

Keep open. Current main still has explicit WebChat/dashboard delivery gates and no sticky persisted-origin or force-origin implementation, while the discussion and tests show a real cross-surface reply-routing gap that needs maintainer policy review before changing default external delivery.

## What This Changes

Request to ensure agent replies are delivered back to the messaging channel that originated the turn across Telegram, Discord, WhatsApp, and WebChat/dashboard surfaces.

## Best Possible Solution

Define a durable per-turn origin/delivery context that survives agent-loop, tool, subagent, and memory-context transitions, then wire reply dispatch through it with explicit freshness and UI-delivery safeguards.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists in current tests: WebChat/provider turns with an external origin and no ExplicitDeliverRoute intentionally do not call routeReply, and Gateway chat tests assert UI clients do not inherit stored external delivery routes. I did not run tests because this was a read-only review.

## Solution Assessment

Unclear. The preferred persisted/sticky origin direction matches the user problem, but the safest implementation needs maintainer policy around cross-surface delivery, stale route expiry, and explicit UI intent before changing defaults.
