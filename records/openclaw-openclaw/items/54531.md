## Summary

Keep open. Current main has narrower route-preservation safeguards, but it still blocks implicit WebChat/dashboard turns from routing replies to an external origin unless an explicit delivery route is set, and the persisted-origin PR linked from the discussion was closed unmerged.

## What This Changes

The issue requests reliable delivery of agent replies back to the messaging channel that originated the turn, even when dashboards or other surfaces are active.

## Best Possible Solution

Keep the issue open for a maintainer-reviewed routing design, likely sticky per-turn external origin metadata with explicit-delivery and stale-session safeguards, plus regression coverage for dashboard/WebChat, subagent/tool follow-up, and external-channel delivery.

## Reproduction Assessment

Yes for the policy-level failure: current tests and source reproduce WebChat plus external origin metadata producing no external route unless `ExplicitDeliverRoute` is true. Live adapter-level reproduction remains unverified here.

## Solution Assessment

No for a blanket force-origin config as-is; the safer path is a narrow sticky-origin design that preserves the inbound external envelope without allowing stale dashboard replies to leak into old chats.
