## Summary

Keep open. Current main still intentionally prevents implicit WebChat/dashboard-originated turns from routing replies to a previous external channel unless `ExplicitDeliverRoute` is set; the linked persisted-origin PR is closed and unmerged. This is a real routing gap, but the default-delivery change crosses a privacy-sensitive boundary and needs maintainer policy review rather than conservative cleanup.

## What This Changes

Request to force agent replies back to the originating messaging channel, such as Telegram, Discord, or WhatsApp, even when a dashboard/WebChat surface is active.

## Best Possible Solution

Keep this open for a maintainer-reviewed reply-routing design. The end state should either be an explicit opt-in force-origin setting or a persisted external-origin route with privacy safeguards, plus docs and regression coverage for WebChat/dashboard sessions, channel-scoped sessions, stale routes, and explicit-delivery exceptions.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is the existing dispatch test that builds a WebChat provider/surface context with an external `OriginatingChannel`/`OriginatingTo` and no `ExplicitDeliverRoute`, then asserts `routeReply` is not called.

## Solution Assessment

Unclear. The closed persisted-origin approach is useful prior art, but automatically forcing a first external origin is not clearly the safest fix without maintainer-approved freshness and privacy rules; the current explicit-delivery route is safer but does not solve the reported default behavior.
