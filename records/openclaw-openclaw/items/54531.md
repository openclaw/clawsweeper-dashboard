## Label Justifications

- `P1`: The issue describes recurring silent non-delivery in real messaging-channel workflows where users may never see the generated answer.
- `impact:message-loss`: The central failure is generated final text not reaching Telegram, Discord, WhatsApp, Signal-like, or similar originating chats.
- `impact:session-state`: The suspected failure depends on origin route/session metadata being lost, overwritten, or gated when execution moves through internal/dashboard surfaces.
- `impact:security`: The requested force-origin behavior changes cross-surface delivery boundaries and could expose replies to the wrong external chat if implemented too broadly.

## Summary

Keep open: current main and v2026.5.22 still preserve some external route metadata, but they intentionally do not force implicit WebChat/dashboard or internal-runner final replies back to an external originating channel unless an explicit delivery route or visible-reply policy allows it. The remaining work is a cross-surface delivery policy decision, not a cleanup close.

## What This Changes

The issue requests a configurable or durable policy that ensures agent final replies are delivered back to the channel and target that originated the turn instead of remaining on an internal/dashboard surface.

## Best Possible Solution

Define a maintainer-approved delivery invariant that carries the original turn route forward and delivers user-facing final replies to that route only under explicit, secure semantics, with regression coverage for internal WebChat/dashboard, Codex/direct visible-reply defaults, and representative external channels.

## Reproduction Assessment

Yes, at source level. Current main has a focused routing-policy gate and regression test showing internal WebChat turns with external origin metadata do not route externally unless ExplicitDeliverRoute is true; I did not run a live channel repro.

## Solution Assessment

Unclear until maintainers choose the policy. A sticky-origin invariant is plausible, but a broad force-origin config/default is not automatically the safest fix because it changes privacy, stale-route, duplicate-delivery, and sendPolicy behavior.
