## Summary

Keep open: current main still has a fixed post-`turn/start` Codex app-server attempt timeout that can abort and retire the client even though current-turn progress is tracked elsewhere. The central timeout fix is already in the open implementation path at https://github.com/openclaw/openclaw/pull/81152, with a separate open attribution fix at https://github.com/openclaw/openclaw/pull/81180, so the issue should wait for review, proof, and merge rather than be closed now.

## What This Changes

The issue asks OpenClaw to make Codex app-server timeouts progress-aware so active long turns do not fall back as generic timeout failures.

## Best Possible Solution

Review and land the narrow Codex runner fix in https://github.com/openclaw/openclaw/pull/81152 and the attribution fix in https://github.com/openclaw/openclaw/pull/81180 with live or maintainer-approved proof, then close this issue once the operator timeout path is verified on current main.

## Reproduction Assessment

Yes, source-reproducible: accept `turn/start`, continue emitting current-turn notifications past `params.timeoutMs` without `turn/completed`, and current main's absolute attempt timeout still aborts and retires the Codex app-server client. I did not run a live Codex app-server session in this read-only review.

## Solution Assessment

Yes, the best direction is the narrow progress-aware Codex app-server watchdog in https://github.com/openclaw/openclaw/pull/81152, with the runtime-attribution path kept separate in https://github.com/openclaw/openclaw/pull/81180. Closing this issue before those PRs merge and prove the live path would be premature.
