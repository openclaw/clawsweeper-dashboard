## Summary

Keep open: current main still has an absolute post-`turn/start` Codex app-server attempt timeout that aborts and retires the client as a generic timeout, while the existing current-turn idle/progress watchdogs only cover a narrower completion-idle path. The related fixes in https://github.com/openclaw/openclaw/pull/79667 and https://github.com/openclaw/openclaw/issues/77984 do not fully cover this near-window progressing-turn failure mode.

## What This Changes

Requests progress-aware Codex app-server timeout handling so large near-context-window turns that are still making progress are not treated as generic model timeouts/fallbacks.

## Best Possible Solution

Land a narrow Codex plugin runtime fix that separates accepted-turn progress liveness from the fixed attempt timeout, adds regression coverage for progress past `params.timeoutMs`, and emits distinct timeout diagnostics without broad release/workflow churn.
