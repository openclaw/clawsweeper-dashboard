## Summary

Keep open: current main still has the absolute Codex app-server attempt timeout and timeout-retirement behavior this issue reports, while the existing idle watchdog work does not cover still-progressing turns that cross the overall attempt deadline.

## What This Changes

Request progress-aware Codex app-server timeout handling so near-window, still-progressing turns are extended or classified distinctly instead of falling into generic timeout model fallback.

## Best Possible Solution

Implement progress-aware timeout handling in the bundled Codex app-server runner, with separate idle/dead versus still-progressing classifications and targeted regression coverage for notifications beyond the old attempt boundary.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a fake app-server can start a turn, emit current-turn activity, and still hit the absolute `params.timeoutMs` timer, which current main handles as `codex app-server attempt timed out`. I did not run the live near-window session in this read-only review.

## Solution Assessment

Unclear as a concrete implementation because the issue names several acceptable policies: bounded extension, yield/checkpoint, distinct timeout classification, and context-pressure preflight. The maintainable direction is to avoid merely raising the timeout and instead make the timeout decision use recent app-server progress facts.
