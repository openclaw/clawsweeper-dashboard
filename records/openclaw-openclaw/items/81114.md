## Summary

Keep open: current main still has the absolute post-turn/start Codex app-server attempt timeout and generic client-retirement path, while newer main changes only partially mitigate quiet completion cases. The active progress-aware timeout and attribution PRs remain open, and fresh release logs still show the same Telegram-visible stall family.

## What This Changes

The issue asks OpenClaw to make Codex app-server timeout handling progress-aware so active near-window or slow-finalizing turns are not aborted and surfaced as generic fallback timeouts.

## Best Possible Solution

Review and land a narrow Codex app-server fix that makes accepted-turn liveness progress-aware, keeps dead-turn protection, normalizes Codex-owned attribution where needed, and verifies the result with live OAuth/Telegram or packaged-release proof before closing.

## Reproduction Assessment

Yes. A fake app-server can accept turn/start, keep emitting current-turn activity past params.timeoutMs without turn/completed, and current main still reaches the absolute attempt-timeout abort path; live comments also report matching Telegram and cron symptoms.

## Solution Assessment

Yes, the maintainable direction is a narrow Codex app-server runner fix rather than a broad runtime workaround. The current open PRs are the right workstream, but they need conflicts/proof resolved and should not close this issue until merged and verified.
