## What This Changes

The issue requests fixing config/cron model normalization so non-Google LiteLLM/custom provider refs like `litellm/gemini-3-flash` remain literal in `agents.defaults.models` and cron payload preflight.

## Best Possible Solution

Scope config model-ref normalization to Google-owned providers and explicit nested `google/*` proxy refs, keep custom LiteLLM/provider bare ids literal, and add regression coverage for both the cron rejection path and the preserved proxy normalization behavior.

Yes, source-level: current main rewrites the configured LiteLLM allowlist key through Google preview normalization, then cron compares the original payload key against that rewritten allowlist. I did not run a live cron job in this read-only review.

## Solution Assessment

Yes, a narrow provider-scoped normalization repair is the maintainable path. It should avoid reintroducing the old cycle-prone helper import and should preserve nested `google/*` proxy compatibility covered by existing tests.

## Repair Work Prompt

Fix the cron/config model normalization regression reported in https://github.com/openclaw/openclaw/issues/84745. Current main rewrites non-Google custom provider refs such as `litellm/gemini-3-flash` to `litellm/gemini-3-flash-preview` via `normalizeAgentModelRefForConfig()`, causing cron `payload.model` preflight to reject the original configured model. Keep the fix inside the config/model-ref normalization boundary, avoid reintroducing cycle-prone imports from heavy agent runtime modules, and preserve intentional normalization for Google providers plus explicit nested `google/*` proxy refs such as `openrouter/google/gemini-3-pro-preview`. Add focused regression tests that prove `agents.defaults.models` and cron payload preflight accept an exact `litellm/gemini-3-flash` custom provider model while existing Google/proxy Gemini normalization tests still pass. Add a `CHANGELOG.md` fix entry because this is user-visible.
