## Summary

Keep open: current main still lacks a typed tool-use signal on the conversation-access hook payloads, and the linked closing PR remains open and conflicting with review findings, so the issue is not implemented yet.

## What This Changes

The issue requests structured tool invocation data in `llm_output` and `agent_end` hook payloads so downstream supervisors can enforce tool-level policy.

## Best Possible Solution

Land a documented additive hook API field populated from canonical current-turn tool-call data, or replace the linked PR with a maintainer-approved narrower hook-contract design.

## Reproduction Assessment

Yes, at source level: current hook type definitions and dispatch builders expose text/raw message fields but no typed `toolUses` field for `llm_output` or `agent_end`. I did not run a live hook reproduction in this read-only review.

## Solution Assessment

Yes in direction, but the exact implementation needs PR-level hook API review because plugin payload semantics become a public contract. The best shape should use the existing `allowConversationAccess` boundary and canonical current-turn tool-call data.
