## Label Justifications

- `P2`: The missing hook signal limits downstream supervisor enforcement but does not make the core runtime unusable.
- `impact:security`: The requested hook payload data would support tool-level policy, capability caps, and audit enforcement by trusted supervisor plugins.
- `impact:other`: The issue affects plugin hook observability for per-turn tool budgets and downstream supervisor workflows.

## Summary

Keep open: current main still lacks a typed tool-use hook payload, and https://github.com/openclaw/openclaw/pull/93390 is the open implementation candidate with closing syntax but is not merged.

## What This Changes

The issue requests structured tool-use blocks in conversation-access hook payloads so downstream supervisors can enforce tool policy from hook data instead of prose heuristics.

## Best Possible Solution

Expose a documented additive hook field populated from canonical current-turn tool-call data behind the existing conversation-access boundary, either by repairing https://github.com/openclaw/openclaw/pull/93390 or replacing it with a narrower implementation.

## Reproduction Assessment

Yes, at source level: current main has no typed `toolUses` field on `llm_output` or `agent_end`, and the dispatch paths only pass assistant text/raw assistant or raw messages. I did not run a live nine-bot pod reproduction in this read-only review.

## Solution Assessment

The requested direction is a reasonable plugin API extension, but the exact field semantics need maintainer-visible review because hook payloads are public plugin contracts. The current best path is to review and repair or replace the linked PR, not to close this issue early.
