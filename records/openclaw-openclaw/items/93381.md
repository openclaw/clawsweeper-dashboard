## Summary

Keep open. Current main and the latest release still expose `llm_output` and `agent_end` without a typed `toolUses`/tool-call payload, while existing `before_tool_call`, `after_tool_call`, and trusted-policy hooks provide an adjacent but different enforcement surface; this should stay with maintainer product/API review rather than be swept closed.

## What This Changes

The issue requests exposing structured tool invocation blocks in conversation-access hook payloads so downstream supervisors can audit and enforce tool-level policy.

## Best Possible Solution

Consolidate this with the broader plugin hook design in https://github.com/openclaw/openclaw/issues/87207 or explicitly split the `llm_output` audit-snapshot requirement, then define one additive, documented contract behind `allowConversationAccess` if maintainers accept it.

## Reproduction Assessment

Yes for a source-level current-state check: current main and `v2026.6.8` lack a typed tool-use field on `llm_output` and `agent_end`. I did not run a live pod reproduction, and existing tool lifecycle hooks cover part of the policy use case through a different surface.

## Solution Assessment

Unclear as an implementation direction. A one-off `toolUses` field may be plausible, but the best solution needs maintainer agreement on whether audit snapshots belong in conversation hooks or should be handled by existing tool hooks plus the broader turn-end contract.
