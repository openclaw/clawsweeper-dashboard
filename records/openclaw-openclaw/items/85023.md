## Summary

Keep open: current main still has the reported Ollama discovery path that can return the `OLLAMA_API_KEY` marker, and the surrounding auth contract separates persisted marker values from real discovery credentials.

## What This Changes

The issue asks OpenClaw to resolve Ollama Cloud API-key discovery to the real credential instead of returning the `OLLAMA_API_KEY` marker string.

## Best Possible Solution

Update Ollama discovery to use a real resolved credential for cloud discovery/runtime auth while keeping marker values only where they are intentionally non-secret config placeholders, then add regression coverage for cloud marker plus env value and local `ollama-local` cases.

## Reproduction Assessment

Yes from source inspection: configure Ollama Cloud with `apiKey: "OLLAMA_API_KEY"` and a real `OLLAMA_API_KEY` env value, and the current helper can return the marker instead of the credential. I did not run a live cloud request because this review is read-only.

## Solution Assessment

Yes, a narrow Ollama discovery/auth fix is the right path, but the issue's suggested shape should be adjusted to respect the existing `apiKey` marker versus `discoveryApiKey` secret contract.
