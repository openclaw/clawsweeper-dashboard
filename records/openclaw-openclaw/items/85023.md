## Label Justifications

- `impact:auth-provider`: The issue is about API key resolution, auth markers, and provider authorization for Ollama Cloud.

## Summary

Keep open: current `main` still has the reported Ollama Cloud discovery auth bug. Source inspection shows the catalog path can return the `OLLAMA_API_KEY` marker instead of the resolved secret value, and that marker is then treated as non-secret so the bearer header is skipped.

## What This Changes

The issue requests fixing Ollama Cloud discovery so env-backed API keys resolve to the actual secret value instead of the `OLLAMA_API_KEY` marker string.

## Best Possible Solution

Fix the Ollama plugin discovery resolver to consume `discoveryApiKey` or the env value for cloud/public endpoints, ignore known marker strings as explicit bearer secrets, and add focused regression tests for cloud env-marker and local-marker behavior.

## Reproduction Assessment

Yes. A source-level repro is `baseUrl: "https://ollama.com"` plus `apiKey: "OLLAMA_API_KEY"` or ambient `OLLAMA_API_KEY`; current code returns the marker instead of the resolver's secret value, though I did not execute a live cloud request.

## Solution Assessment

Yes. The narrow maintainable fix is in `extensions/ollama/src/discovery-shared.ts`, using the existing resolver contract instead of adding new config or changing core marker semantics.
