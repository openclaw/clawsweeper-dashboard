## Summary

Keep open: this is a focused bug fix, but it is not merge-ready because the new guard selection still leaves private HTTPS self-hosted Responses endpoints on the trusted path and the external PR has no real behavior proof.

## What This Changes

The PR changes the shared JSON web-tools POST helper to use the self-hosted guarded fetch path for private HTTP URLs and adds unit coverage for HTTP-vs-HTTPS guard selection.

## Best Possible Solution

Classify JSON web-tool endpoints by target privacy for both HTTP and HTTPS, route private/loopback/internal targets through the self-hosted guard, keep public HTTPS on the trusted guard, and prove it against a redacted real self-hosted endpoint.

## Reproduction Assessment

Yes by source inspection: current main sends every `postTrustedWebToolsJson` call through the trusted guard, while the trusted guard lacks private-network opt-in and blocks private/internal targets. I did not run a live self-hosted endpoint in this read-only review.

## Solution Assessment

No. The shared-helper location is plausible, but the implementation is too narrow because it only handles `http:` private endpoints instead of matching the established HTTP and HTTPS private-endpoint classification used by sibling providers.
