## Summary

Keep open. Current main has a source-backed no-delivery path when a newer same-target inbound supersedes an older foreground final, but the issue does not provide enough timing/log detail to tell whether this was expected stale-reply suppression, an incomplete tool-use turn, or a separate WhatsApp delivery bug.

## What This Changes

The issue reports intermittent WhatsApp direct-chat reply text drops when a model response includes both visible text and tool calls.

## Best Possible Solution

First confirm whether dropped turns were superseded by newer inbound messages; then either improve logging/background-completion policy for expected suppression or add a WhatsApp/shared-dispatch regression for unsuppressed final text plus tool calls.

## Reproduction Assessment

Unclear. A source-level path can suppress an older final before WhatsApp delivery when a newer same-target inbound starts, but the issue does not confirm that timing and I did not establish a live failing current-main path.

## Solution Assessment

Unclear. Always delivering the text is not clearly the best fix because it may reintroduce stale tool-turn replies; the maintainable path depends on distinguishing expected supersession from an actual unsuppressed delivery drop.
