## Summary

Keep open. The reported root cause still matches current `main`: native `sessions_spawn` only threads workspace lineage, `sandbox="inherit"` is still a sandboxed/unsandboxed guard, and the child run resolves its own sandbox context instead of carrying the parent’s resolved Docker env/setup/runtime state.

## What This Changes

The issue reports that sandboxed native subagent runs do not reliably receive configured Docker env variables, current workspace state, or refreshed `setupCommand` effects after sandbox recreation.

## Best Possible Solution

Define native `sessions_spawn` sandbox inheritance explicitly: same-agent sandboxed children should observe the effective sandbox Docker env/setup and intended workspace after recreate, while cross-agent children should not receive parent Docker env/setup without a reviewed policy.

## Reproduction Assessment

Yes for a focused code-level reproduction: configure a sandboxed agent with `sandbox.docker.env` and `setupCommand`, spawn a native same-agent subagent, and assert whether the child runtime sees the effective env/setup after `openclaw sandbox recreate --agent`. I did not run a live Docker repro because this review is read-only, but the static path still lacks a parent resolved-context handoff.

## Solution Assessment

Unclear until maintainers choose the inheritance semantics. A minimal same-agent resolved-context handoff may be enough, but broad container sharing or cross-agent propagation touches the sandbox secret boundary and needs explicit review.
