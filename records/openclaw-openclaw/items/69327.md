## Summary

Keep open. Current main still has a source-level gap around native subagent sandbox state: `sessions_spawn` carries workspace and inherited tool policy, but not resolved Docker env/setup state, and subagent spawning reloads the pinned runtime config instead of using the parent run's resolved sandbox context. Because Docker env can include secrets or private endpoints, the repair needs maintainer policy on same-agent versus cross-agent inheritance before automation should patch it.

## What This Changes

The issue asks for fresh sandboxed native subagents to receive configured `sandbox.docker.env`, current workspace state, and recreated `setupCommand` effects after `openclaw sandbox recreate`.

## Best Possible Solution

Define the native sandbox inheritance and config-refresh contract, preserve cross-agent isolation, then add regression coverage plus Docker-backed proof for same-agent env/setup and recreate behavior.

## Reproduction Assessment

Yes at source level: current main has no carrier for resolved parent Docker env/setup into native `sessions_spawn`, and the child resolves sandbox context independently from a pinned runtime config. I did not establish a live Docker failing run in this read-only review.

## Solution Assessment

Unclear until maintainers choose the security boundary. A narrow same-agent resolved-context handoff or explicit runtime config refresh path is safer than sharing parent containers or forwarding Docker env/setup indiscriminately.
