## Summary

Keep open. Current main still has a real source-level gap at the native subagent sandbox boundary: workspace lineage and tool policy are threaded into spawned sessions, but resolved Docker env/setup state is not, and the recreate/bootstrap portion still needs a policy-aware runtime proof before this can be closed.

## What This Changes

The issue reports that sandboxed subagent runs do not reliably receive configured `sandbox.docker.env`, current workspace state, or recreated Docker `setupCommand` effects.

## Best Possible Solution

Define the native sandbox inheritance contract, carry resolved Docker env/setup only across approved same-agent sandboxed children, preserve cross-agent isolation, and add regression plus live Docker recreate/bootstrap proof.

## Reproduction Assessment

Yes at source level for the unresolved inheritance boundary: current main has no carrier for resolved parent Docker env/setup into native `sessions_spawn`, while the child resolves sandbox config independently. I did not establish a live Docker failing run in this read-only review.

## Solution Assessment

Unclear until maintainers choose the security policy. A narrow same-agent resolved-context handoff with negative cross-agent coverage is safer than sharing parent containers or forwarding Docker env/setup indiscriminately.
