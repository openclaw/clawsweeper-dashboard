## Summary

Keep open: current main has fixed the narrower explicit Docker env filtering problem, but the reported subagent inheritance/recreate behavior is not fully proven fixed and still crosses sandbox env, workspace state, and credential-boundary policy.

## What This Changes

The issue reports that sandboxed `sessions_spawn` children do not reliably receive configured Docker env, current workspace state, or post-recreate setup effects.

## Best Possible Solution

Define the same-agent versus cross-agent sandbox inheritance contract, preserve env/setup isolation across agent boundaries, and add Docker-backed regression coverage for env, setupCommand, workspace freshness, and recreate behavior.

## Reproduction Assessment

Partly. Source shows recreate/container and copied-workspace behavior that can leave stale workspace state, and the env/setup inheritance path lacks a parent Docker context carrier, but I did not run a live Docker subagent reproduction on current main.

## Solution Assessment

Unclear. Blindly copying the parent Docker env/setup is not the best fix; the safer path is a maintainer-approved target-agent inheritance contract with negative cross-agent leakage coverage.
