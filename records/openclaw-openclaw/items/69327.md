## Summary

Keep open. Current main still carries workspace and inherited tool policy through native subagent spawns, but not the parent resolved Docker env/setup context, so the reported sandbox inheritance gap remains and needs an explicit security policy before automation should implement it.

## What This Changes

The issue reports that sandboxed subagents do not reliably receive configured `sandbox.docker.env`, current workspace/bootstrap state, or refreshed setup effects after sandbox recreation.

## Best Possible Solution

Define the native sandbox inheritance policy, then carry effective Docker env/setup only for authorized same-agent sandboxed children with same-agent positive coverage, cross-agent negative coverage, and recreate/bootstrap proof.

## Reproduction Assessment

Yes, at source level. Current main has no carrier for the parent resolved Docker env/setup context from a sandboxed parent run into native `sessions_spawn`, and the child resolves sandbox context fresh from target config.

## Solution Assessment

Unclear until maintainers choose the inheritance policy. A narrow same-agent resolved-context handoff with explicit negative cross-agent tests looks safer than sharing the parent container or forwarding Docker env indiscriminately.
