## Summary

Keep open. Current main still matches the reported root cause: native `sessions_spawn` carries workspace lineage, but it does not carry the parent’s resolved Docker env/setup context, and the child run resolves sandbox config from its own session/agent instead.

## What This Changes

The issue reports that sandboxed native subagent runs do not receive configured Docker sandbox env/setup/workspace state after sandbox recreate.

## Best Possible Solution

Define native sandbox inheritance explicitly: same-agent sandboxed children should observe the effective Docker env/setup and intended workspace after recreate, while cross-agent children should not receive parent Docker state without an explicit reviewed policy.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: configure a sandboxed agent with Docker env/setup, spawn a native same-agent subagent, and the current path has no parent resolved-context handoff for the child to receive that Docker state. I did not run a live Docker repro in this read-only review.

## Solution Assessment

Unclear until maintainers choose the inheritance policy. A narrow same-agent resolved-context handoff may be the right fix, but cross-agent propagation and container sharing touch the sandbox secret boundary.
