## Summary

Keep open. Current main still shows a source-level gap at the native subagent sandbox boundary: workspace/tool inheritance is threaded through `sessions_spawn`, but resolved Docker env/setup state is not, and the stale recreate/bootstrap report needs a security-aware policy before a repair is safe.

## What This Changes

The issue asks for sandboxed native subagents to receive configured Docker env variables, current workspace contents, and refreshed setupCommand/bootstrap state after sandbox recreate.

## Best Possible Solution

Define the native sandbox inheritance contract, carry resolved Docker env/setup only for authorized same-agent sandboxed children, preserve cross-agent isolation, and add regression plus live Docker proof for recreate/bootstrap freshness.

## Reproduction Assessment

Yes, at source level: current main has no carrier for resolved parent Docker env/setup into native `sessions_spawn`, while the child resolves sandbox config independently. I did not establish a live Docker failing run in this read-only review.

## Solution Assessment

Unclear until maintainers choose the inheritance and security policy. A narrow same-agent resolved-context handoff with negative cross-agent coverage is safer than sharing parent containers or forwarding Docker env indiscriminately.
