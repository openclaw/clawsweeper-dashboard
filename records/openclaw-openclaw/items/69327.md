## Summary

Keep open: current main still lacks a parent Docker env/setup carrier for native subagent spawns, and the latest human follow-up confirms the remaining design question is unresolved. The narrower explicit Docker env filtering bug was fixed by https://github.com/openclaw/openclaw/pull/82763, but that does not settle subagent inheritance, workspace freshness, or recreate semantics.

## What This Changes

The issue reports that sandboxed subagent runs do not reliably receive configured Docker env, current workspace state, or recreated setupCommand effects.

## Best Possible Solution

Define the same-agent versus cross-agent sandbox inheritance contract, preserve env/setup isolation across agent boundaries, and add Docker-backed regression coverage for env, setupCommand, workspace freshness, and recreate behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not fully: current source clearly shows the missing parent Docker env/setup carrier, but the full env, setupCommand, and workspace-staleness symptoms still need a live Docker subagent repro on current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear: blindly copying parent Docker env/setup is not the best default because it can cross credential boundaries; the safer path is a maintainer-approved inheritance contract with negative cross-agent leakage coverage.
