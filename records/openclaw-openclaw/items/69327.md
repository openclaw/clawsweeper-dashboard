## Summary

Keep open. Current main has fixed the narrower explicit Docker env filtering issue and has adjacent workspace/recreate coverage, but the broader native `sessions_spawn` sandbox inheritance and config-refresh boundary still lacks live Docker proof and needs a maintainer security/product decision before resolved Docker env or setup state is propagated.

## What This Changes

The issue reports that fresh sandboxed native subagent runs do not reliably receive configured `sandbox.docker.env`, current workspace state, or refreshed `setupCommand` effects after `openclaw sandbox recreate --agent`.

## Best Possible Solution

Define the native same-agent sandbox inheritance and config-refresh contract, preserve cross-agent Docker env/setup isolation unless explicitly approved, then add Docker-backed regression proof for env, setupCommand, workspace, and recreate behavior.

## Reproduction Assessment

Unclear. Source inspection shows current main does not carry a resolved parent Docker sandbox context through native `sessions_spawn`, but the reported same-agent env/setup symptom was not reproduced live after the explicit Docker env and workspace fixes.

## Solution Assessment

Unclear until maintainers choose the sandbox inheritance boundary. A narrow same-agent resolved-context or config-refresh fix with negative cross-agent coverage is safer than sharing parent containers or forwarding Docker env/setup broadly.
