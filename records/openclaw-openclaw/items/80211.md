## Summary

Keep open: the runtime-registration direction is valid, but the latest PR head still breaks provider-backed embedding auth in the new status probe and does not include real after-fix doctor/status proof.

## What This Changes

The PR adds a MemoryPluginRuntime registration to memory-lancedb, a MemoryDB.get helper, runtime search/read/status/probe methods, and regression coverage for capability registration.

## Best Possible Solution

Keep the plugin-owned runtime registration, make the embedding probe use the same provider-adapter path or a lightweight real embed probe as recall, and add redacted after-fix doctor or memory status output before merge.

## Reproduction Assessment

Yes. Source inspection shows current main's memory-lancedb plugin does not register a memory runtime, while doctor emits the warning when resolveActiveMemoryBackendConfig returns null; the PR body also includes before-fix live doctor output.

## Solution Assessment

No as proposed. Registering a plugin-owned runtime is the right maintainable boundary, but the new readiness probe must follow provider-backed auth instead of requiring plugin-local embedding.apiKey.
