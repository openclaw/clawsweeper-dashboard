## Summary

Keep open: the runtime-registration direction is the right fix for the false doctor/status warning, but the current PR adds a probe that fails valid provider-profile authentication setups without a plugin-local API key.

## What This Changes

This PR registers a MemoryPluginRuntime for memory-lancedb, adds LanceDB-backed search/read/status/probe methods, and backfills tests for the new registration call.

## Best Possible Solution

Keep the plugin-runtime registration approach, but make readiness probing use the same provider/auth path as real memory recall/store operations before merge.

## Reproduction Assessment

Yes. Source inspection shows current main has no memory-lancedb runtime registration and doctor warns when resolveActiveMemoryBackendConfig returns null; the PR body also includes live before/after doctor output for that symptom.

## Solution Assessment

No as proposed. Registering the runtime is the narrow maintainable fix, but the new embedding probe must not reject provider-backed auth configurations that current memory-lancedb behavior already supports.
