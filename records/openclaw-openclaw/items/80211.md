## Summary

Keep this PR open: the direct memory-lancedb runtime registration is the right direction, but the current patch still introduces a provider-auth regression in the new readiness probe. The updated PR body supplies enough real behavior proof for the original doctor warning, and the remaining blocker is a narrow code/test fix.

## What This Changes

The PR adds a memory capability runtime to `extensions/memory-lancedb`, exposes search/read/status/probe hooks backed by the existing LanceDB memory objects, and adds regression coverage plus mock API backfills.

## Best Possible Solution

Keep the runtime-registration approach, but make readiness probing use the same embedding provider/auth path as real memory recall and store operations.

## Reproduction Assessment

Yes. Source inspection shows current main does not register a memory runtime for memory-lancedb, and doctor warns when `resolveActiveMemoryBackendConfig` returns null; the PR body also includes before-fix live doctor output for the same symptom.

## Solution Assessment

No as proposed. Registering the runtime is the maintainable fix, but the new embedding probe must not reject provider-backed auth configurations that existing memory-lancedb behavior already supports.
