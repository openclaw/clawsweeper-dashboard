## Summary

Keep open: the runtime-registration fix is the right owner-boundary direction, but the PR still has a blocking probe bug for provider/auth-profile backed embeddings without a plugin-local API key.

## What This Changes

The PR registers memory-lancedb as a memory capability runtime for doctor/status discovery and adds regression coverage plus mock API backfills.

## Best Possible Solution

Land the plugin-owned runtime registration after changing readiness probing to use or mirror the existing provider adapter/auth path and adding focused coverage for no-apiKey provider-backed config.

## Reproduction Assessment

Yes. Source inspection shows current main does not register a memory runtime for memory-lancedb while doctor/status require that runtime, and the PR body includes live before-fix doctor output for the same warning.

## Solution Assessment

No as proposed. Runtime registration is the narrow maintainable fix, but the new embedding probe must not reject existing provider-backed auth configurations that work without cfg.embedding.apiKey.
