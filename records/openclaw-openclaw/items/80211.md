## Summary

Keep this PR open: the runtime-registration direction is valid, but the proposed embedding readiness probe diverges from provider-backed auth that memory-lancedb already supports, and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR registers a memory-lancedb MemoryPluginRuntime with search/status/probe/read hooks, adds MemoryDB.get, and adds regression coverage for runtime discovery.

## Best Possible Solution

Keep the runtime registration approach, but make the readiness probe follow the same provider-adapter auth path as real recall and add redacted after-fix doctor/status output.

## Reproduction Assessment

Yes. Source inspection plus the PR body's before-fix live output show current main has no memory-lancedb runtime registration while doctor/status depend on active runtime resolution; I did not run local tests because this review had to keep the checkout read-only.

## Solution Assessment

No as proposed. Registering a plugin-owned runtime is the right boundary, but the embedding probe should reuse provider-adapter auth or perform a lightweight real embedding probe instead of requiring cfg.embedding.apiKey.
