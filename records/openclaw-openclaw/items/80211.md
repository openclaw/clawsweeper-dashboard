## Summary

Keep open: current main still has the memory-lancedb doctor/status runtime gap, and this PR is the canonical plugin-owned fix path, but the patch still has a blocking provider-auth probe bug before merge.

## What This Changes

The PR registers a memory-lancedb `MemoryPluginRuntime` via `api.registerMemoryCapability`, adds search/read/status/probe hooks backed by the existing LanceDB objects, and backfills regression-test mocks.

## Best Possible Solution

Repair this PR so memory-lancedb registers the runtime through the plugin capability contract while its readiness probes use the same provider adapter/auth path as real recall/store operations, then rebase and land it with focused doctor/status coverage.

## Reproduction Assessment

Yes, at source level. Current main has no memory-lancedb runtime registration, while doctor/status resolve active memory through that runtime and emit the reported warning when resolution returns null.

## Solution Assessment

No as proposed. Registering the plugin-owned runtime is the right maintainable fix, but the new embedding probe must not reject provider-backed configurations that current memory-lancedb behavior and docs already support.
