## Summary

Keep open: the runtime-registration direction fixes a real source-reproducible doctor/status false warning, but the current PR head still has a correctness bug because the new embedding probe treats provider-backed auth configs as missing an apiKey.

## What This Changes

The PR registers a memory capability runtime for the memory-lancedb plugin and adds regression coverage so doctor/status can discover it as the active memory backend.

## Best Possible Solution

Keep the plugin-side runtime registration, but make readiness probing use the same embedding provider/auth path as real recall/store operations or a lightweight embedding probe with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main has no memory-lancedb runtime registration and doctor warns when active backend config resolution returns null; the PR body also includes live before-fix doctor output for the same symptom.

## Solution Assessment

No as proposed. Registering a MemoryPluginRuntime is the maintainable fix, but the new probe must not reject existing provider/profile-backed auth configurations that memory-lancedb already supports.
