## Summary

Keep open: the PR is a useful direct fix for the memory-lancedb doctor/status visibility bug, but the latest head still introduces a provider-auth regression in the new runtime readiness probe.

## What This Changes

The PR registers a `MemoryPluginRuntime` for `extensions/memory-lancedb`, adds runtime-backed search/read/status/probe hooks, and backfills memory-lancedb tests/mocks for `registerMemoryCapability`.

## Best Possible Solution

Keep the plugin-side runtime registration, but make readiness probing use the same embedding provider/auth path as real recall/store operations, with regression coverage for apiKey-less provider-backed config.

## Reproduction Assessment

Yes, source-reproducible. Current main has no memory-lancedb runtime registration, and doctor warns when active backend config resolution returns null; the PR body also supplies live before/after doctor output for the same symptom.

## Solution Assessment

No as proposed. Runtime registration is the right maintainable direction, but the new embedding probe must not reject provider-backed auth configurations that existing memory-lancedb behavior supports.
