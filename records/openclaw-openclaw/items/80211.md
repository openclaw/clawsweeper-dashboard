## Summary

Keep open. The runtime-registration direction is the right fix for the doctor/status false warning, and the supplied real behavior proof is sufficient, but the PR still introduces a provider-auth regression in the new embedding readiness probe.

## What This Changes

The PR registers a memory capability runtime for `memory-lancedb`, adds search/read/status/probe hooks backed by the existing LanceDB objects, and adds registration coverage.

## Best Possible Solution

Keep the plugin-owned runtime registration approach, but make readiness probing use the same provider adapter/auth path as real memory operations before merge.

## Reproduction Assessment

Yes, at source level. Current main does not register a memory runtime for `memory-lancedb`, and doctor/status resolve through the active memory runtime path; the PR body also supplies before/after live doctor evidence for the reported warning.

## Solution Assessment

No as proposed. Registering the runtime is the narrow maintainable fix, but the new embedding probe must not reject valid provider-profile configurations where `embedding.apiKey` is intentionally omitted.
