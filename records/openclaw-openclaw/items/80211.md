## Summary

Keep this PR open: current main still needs the memory-lancedb runtime registration, but this patch should not merge until the new embedding readiness probe stops rejecting provider-backed auth configurations that omit a plugin-local API key.

## What This Changes

The PR registers a MemoryPluginRuntime for memory-lancedb, adds LanceDB-backed search/read/status/probe hooks, and adds regression coverage for registerMemoryCapability.

## Best Possible Solution

Keep the plugin-owned runtime registration, but make readiness probing use the same provider-auth embedding path as real recall/store operations or a real lightweight embedding probe.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main has no memory-lancedb runtime registration, while doctor resolves active memory through the memory runtime path and warns when it is missing. The PR body also includes live before-fix doctor output for the same symptom.

## Solution Assessment

No as proposed. Registering the runtime is the right maintainable fix, but the new probe must not treat missing plugin-local embedding.apiKey as unhealthy when provider-profile auth is already supported.
