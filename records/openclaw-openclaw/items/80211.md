## Summary

Keep open: the runtime-registration bug is real and this PR is close, but the patch still needs real after-fix proof for the latest probe path and appears to reintroduce a doctor false-positive path for documented plugin-local embedding configs.

## What This Changes

The PR adds a memory-lancedb MemoryPluginRuntime with LanceDB-backed search/read/status/probe hooks and regression tests for runtime registration plus provider-backed embedding probes.

## Best Possible Solution

Land a plugin-owned runtime registration that also keeps doctor/status health checks on the active memory-lancedb runtime contract, with regression coverage and real proof for provider-backed and plugin-local embedding configurations.

## Reproduction Assessment

Yes. Source inspection shows current main has no memory-lancedb memory runtime while doctor/status resolve through the memory runtime path, and the PR body includes live before-fix doctor output for that symptom.

## Solution Assessment

No. Registering the runtime in memory-lancedb is the right seam, but this patch needs to avoid routing documented lancedb plugin-local embedding configs through core memorySearch doctor checks and needs real after-fix proof for the latest probe behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep doctor on the plugin-owned health path:** `extensions/memory-lancedb/index.ts:833-836`
  - body: Returning `backend: "builtin"` makes `noteMemorySearchHealth` skip the alternate-slot suppression and continue into core `agents.*.memorySearch` provider checks. For the documented `memory-lancedb` direct endpoint config that stores `embedding.apiKey`/`baseUrl` under the plugin entry, doctor can still report no embedding provider is ready even though this runtime can embed. Adjust doctor/status to trust or probe the active plugin runtime for non-default memory slots instead of validating unrelated core memorySearch credentials.
  - confidence: 0.82
- **[P3] Report the configured embedding provider:** `extensions/memory-lancedb/index.ts:749-752`
  - body: `providerLabel` is derived from `baseUrl` and model name, so provider-backed configs such as `embedding.provider: "github-copilot"` with `text-embedding-3-small` will show `provider: "openai"` in status even though the SDK uses the GitHub Copilot adapter. Prefer the configured provider id unless this is the explicit OpenAI-compatible direct-client path.
  - confidence: 0.78
