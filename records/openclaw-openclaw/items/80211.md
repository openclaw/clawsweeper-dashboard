## Summary

Keep open. The runtime-registration direction matches the memory SDK/doctor boundary and is a useful replacement for the stale related PR, but the patch should not merge yet because the new embedding probe can false-fail supported provider-backed auth and after-fix real behavior proof is missing.

## What This Changes

The PR registers a memory capability runtime for memory-lancedb and adds regression coverage for doctor/status discovery.

## Best Possible Solution

Keep the runtime registration approach, but make the status/doctor readiness probe follow the same embedding-provider path used by real recall and add redacted after-fix doctor/status output before merge.

## Reproduction Assessment

Yes from source inspection and the PR body's before-fix live output: current main does not register a memory runtime for memory-lancedb, and doctor warns when no active memory backend config is resolved. I did not run local tests because this review was required to keep the checkout read-only.

## Solution Assessment

No as proposed. Registering a runtime is the right maintainable boundary, but the readiness probe should reuse or exercise the existing embedding/provider-adapter contract instead of checking only plugin-local apiKey presence.

## Review Findings

Overall correctness: patch is incorrect
