## Summary

Keep open: this PR is a plausible fix for https://github.com/openclaw/openclaw/issues/79284 and has real behavior proof plus green exact-head checks, but source review found one remaining scoped compaction path where active process refs can still be missed.

## What This Changes

The PR adds scoped active background exec/process session references to embedded compaction, after-turn runtime contexts, and system prompts, with regression tests and a changelog entry.

## Best Possible Solution

Complete the exact-scope active process handoff for every compaction runtime-context builder, including queued engine-owned compaction, then land after the existing green exact-head gates remain stable.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the remaining PR defect: create a background process scoped to sandboxSessionKey with a different sessionKey, then build queued context-engine compaction runtimeContext and observe that activeProcessSessions is derived from sessionKey instead of the process-tool scope.

## Solution Assessment

No, not quite yet; the proposed direction is narrow and maintainable, but the queued context-engine compaction path still needs to derive active refs from the same sandbox/session scope used by the process tool.

On https://github.com/openclaw/openclaw/pull/79307, fix the remaining active exec reference scope gap. The PR currently lets buildEmbeddedCompactionRuntimeContext derive activeProcessSessions from sessionKey only, while queued/engine-owned compaction can receive a distinct sandboxSessionKey and process tools use sandboxSessionKey/sessionId/agentId scope derivation. Update src/agents/pi-embedded-runner/compact.queued.ts or the shared helper so queued compaction passes explicit activeProcessSessions using resolveProcessToolScopeKey with sandboxSessionKey || sessionKey, sessionId, and sessionAgentId. Add a regression where a running background session is scoped to sandboxSessionKey, sessionKey is different, and the queued compaction runtimeContext includes only that sandbox-scoped session. Do not broaden unscoped lookups, do not expose unrelated sessions, and do not change process persistence policy.
