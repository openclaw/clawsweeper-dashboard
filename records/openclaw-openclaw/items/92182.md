## Summary

Current main and v2026.6.6 still keep Slack preview streaming separate from normal block replies; the speculative hard-off PR was closed, and the unexplained repeated Slack posts still need the maintainer-requested sanitizer-disabled repro data.

## What This Changes

The issue reports that after upgrading to 2026.6.5, a Slack account with account-level `streaming.mode: "off"` still produced many cumulative partial-prefix messages instead of one final/coalesced reply.

## Best Possible Solution

Preserve the separate Slack preview and block-reply controls, then fix only a proven maintained delivery path once sanitizer-disabled logs and effective config show repeated posts with account block delivery disabled.

## Reproduction Assessment

No. The report gives concrete production symptoms, but current main and v2026.6.6 source show account preview mode `off` leaves `onPartialReply` unset; the remaining burst needs the requested retest with account `block.enabled: false` and the sanitizer plugin disabled.

## Solution Assessment

Unclear for a code fix right now. The closed hard-off PR was not the best fix because it changed the preview/block contract; the next maintainable step is evidence that isolates the repeated posts to a supported Slack/Codex delivery path.
