## Summary

Keep open: current main still suppresses Telegram polling stall detection when unrelated Telegram API activity is recent, matching the reported watchdog failure. The suppression came from a prior merged delivery-loss fix, so this needs a careful Telegram liveness repair rather than closure.

## What This Changes

The issue asks OpenClaw to make Telegram polling stall recovery fire when getUpdates liveness is stale instead of being masked by unrelated Telegram API activity.

## Best Possible Solution

Separate inbound getUpdates liveness from active outbound delivery protection so stale polling restarts are not masked by unrelated API calls while long-running reply delivery remains protected.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: make getUpdates stale past the threshold, record a recent non-getUpdates API activity, and `detectStall` returns `null` because the current gate uses API activity as a suppressor. I did not run a live Telegram/network repro in this read-only review.

## Solution Assessment

Unclear for the literal suggested operator change: the source confirms the bug, but the safest fix should preserve the prior in-flight reply protection from https://github.com/openclaw/openclaw/pull/56343. A maintainable solution should add regression coverage for both the stale getUpdates case and the long-agent-reply case.
