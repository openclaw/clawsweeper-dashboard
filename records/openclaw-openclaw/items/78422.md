## Summary

Keep open: current `main` still contains the reported watchdog suppression path, while the Telegram docs and config help describe the watchdog as completed-`getUpdates` liveness. The related stall reports are useful context, but this issue has a narrower source-level reproduction and should be fixed directly rather than closed as a duplicate.

## What This Changes

The issue reports that Telegram long-polling can wedge silently because unrelated Telegram API activity suppresses the `pollingStallThresholdMs` watchdog.

## Best Possible Solution

Align the Telegram polling watchdog implementation and regression tests with the documented completed-`getUpdates` liveness contract, and leave broader 30-minute last-resort recovery policy as a separate maintainer decision if desired.

## Reproduction Assessment

Yes for a source-level reproduction: make `getUpdates` stale past the threshold, refresh or start a non-getUpdates API call, and `detectStall` returns `null`. I did not establish a live 39-minute Telegram socket wedge in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to make the stall detector key on stale `getUpdates` liveness and update the conflicting tests. The 30-minute recovery tier and extra diagnostics can be considered separately, but they are not required to fix the documented watchdog bug.
