## Summary

Keep open. Current main already contains the bounded pending-tool-task drain layer, but it does not contain this PR’s per-tool-call timeout or provider-wide synthetic-result default. The PR is not ready to merge because its new adapter-level 60s default can abort existing long-running tools before their documented per-tool timeout contract.

## What This Changes

This PR enables synthetic tool-result repair by default for all transcript policies, adds configurable per-tool-call timeouts through the Pi tool adapter and embedded runner config path, extends agent-default config schema/types, and adds timeout/session-stuck tests.

## Best Possible Solution

Keep the useful reliability work open, but narrow it before merge: preserve the already-shipped pending-drain behavior, add per-tool timeout handling without shortening existing tool-specific contracts such as exec, and make any broader synthetic-result default an explicit provider/runtime policy decision with aligned docs and tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: the linked stuck-session reports give concrete lost-tool-result and hanging-tool scenarios, and the PR-introduced regression is reproducible by running an adapted tool expected to exceed 60 seconds with no agents.defaults.toolCallTimeoutSeconds configured.

## Solution Assessment

Is this the best way to solve the issue? No as currently written: the timeout wrapper is plausible, but defaulting every adapted tool to 60 seconds is broader than the existing timeout contracts. A safer fix would disable the adapter timeout by default or make it tool-aware while keeping provider replay policy explicit.

- **[P2] Preserve existing long-running tool limits:** `src/agents/pi-tool-definition-adapter.ts:389-392`
  - body: This default applies a 60s wrapper timeout to every adapted tool when users have not configured `agents.defaults.toolCallTimeoutSeconds`. Existing tools such as `exec` already expose their own timeout contract (`tools.exec.timeoutSec` defaults to 1800s and per-call `timeout: 0` disables it), so foreground/non-backgrounded commands or other long-running tools can now be aborted by the wrapper long before their documented timeout. Preserve current behavior by disabling the adapter timeout by default or making it tool-aware.
