## Summary

Keep open. Current main still has the TaskFlow SQLite migration gap, and this PR is a plausible fix with focused regression coverage, but it needs reconciliation with the overlapping open fix PR and the required changelog/rebase work before merge.

## What This Changes

This PR backfills and trims `owner_key` from legacy `owner_session_key`, rebuilds `flow_runs` without the legacy column, and adds regression tests for legacy-only and partially migrated TaskFlow SQLite schemas.

## Best Possible Solution

Land one TaskFlow-owned SQLite migration that preserves existing flow rows, normalizes `owner_key`, removes the legacy `owner_session_key` constraint, recreates the current indexes, and keeps regression coverage for both legacy-only and hybrid upgraded schemas.

## Reproduction Assessment

Yes. The linked bug gives a concrete webhooks `create_flow` configuration and curl path, and current main’s insert path plus migration guard explain why an upgraded database retaining `owner_session_key TEXT NOT NULL` can fail new flow creation.

## Solution Assessment

Yes, the PR targets the right layer: repair the TaskFlow SQLite schema migration instead of changing the webhook request contract. The maintainable path is to choose this PR, #63893, or a combined equivalent, then add changelog coverage, rebase, and rerun focused TaskFlow validation.
