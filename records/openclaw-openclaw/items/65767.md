## Summary

Keep open: current main still has the TaskFlow SQLite migration gap, and this PR is a plausible fix with regression coverage. It needs maintainer review because open PR #63893 overlaps the same `flow_runs.owner_session_key` rebuild, and this PR also lacks the required user-facing changelog entry.

## What This Changes

This PR backfills `owner_key` from legacy `owner_session_key`, rebuilds `flow_runs` without the legacy column, and adds TaskFlow SQLite regression tests for full-legacy and partially migrated schemas.

## Best Possible Solution

Land one TaskFlow-owned SQLite migration that preserves existing flow rows, normalizes `owner_key`, removes the legacy `owner_session_key` constraint, recreates current indexes, and keeps regression coverage for both legacy-only and hybrid upgraded schemas.

## Reproduction Assessment

Yes. The linked bug gives a webhook `create_flow` path and logs, and current main’s write path plus migration guard show why a database retaining `owner_session_key TEXT NOT NULL` can fail new flow creation.

## Solution Assessment

Yes, the PR’s direction is the right layer: repair the TaskFlow SQLite schema migration rather than changing the webhook request contract. The maintainable path is to choose this PR, #63893, or a combined equivalent, then add changelog and run the TaskFlow-focused checks.
