## Summary

Keep open for contributor/maintainer action: this active PR changes the active npm install selector in a way that can pin routine plugin and hook updates to stale exact versions, and it still lacks after-fix real behavior proof.

## What This Changes

The branch rewrites npm plugin and hook install records to persist resolved `name@version` values as active `spec` fields across install, update, sync, onboarding, doctor repair, and related tests.

## Best Possible Solution

Keep `record.spec` as the requested/update selector, store immutable artifact identity in `resolvedSpec` plus integrity metadata, and adjust audit policy to recognize stable resolved metadata if the current warning is too strict.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: install or update an unversioned npm plugin under this patch, then `openclaw plugins update <id>` reuses the newly persisted exact `record.spec` and no longer follows the registry default line.

## Solution Assessment

No. Persisting the resolved artifact into the active selector overloads `record.spec`; preserving the selector and teaching audit to use `resolvedSpec`/integrity is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the update selector in install records:** `src/cli/npm-resolution.ts:12-15`
  - body: This stores `resolution.resolvedSpec` in `record.spec` even when the user did not pass `--pin`. `plugins update <id>` later reuses `record.spec`, and the update-channel resolver preserves exact versions, so an install from `@scope/pkg` becomes `@scope/pkg@oldVersion` and routine id or `--all` updates stop following the registry default line documented for upgrades.
  - confidence: 0.9

## Security Review

Status: needs_attention
