## Summary

Keep open: the patch is not mergeable as written because it stores the resolved exact version in the active install `spec`, but current update code and docs use that same field as the selector for future `plugins update` runs. It also lacks after-fix real behavior proof for the install/update paths it changes.

## What This Changes

This PR stores npm resolution `resolvedSpec` as the active plugin and hook install `spec` across install, update, sync, onboarding, doctor repair, and related tests while keeping audit checks on `record.spec`.

## Best Possible Solution

Preserve `record.spec` as the user/update selector, keep exact artifact identity in `resolvedSpec` plus integrity fields, and adjust audit or metadata handling without breaking default-line updates.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: install or update an unversioned npm plugin under this patch, then run an id-based update and it will reuse the newly persisted exact `record.spec` instead of the default registry line.

## Solution Assessment

No. Persisting the resolved artifact into the active selector overloads `record.spec`; the safer fix is to keep the requested selector separate from resolved artifact identity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the update selector in install records:** `src/cli/npm-resolution.ts:9-12`
  - body: This changes non-`--pin` installs to store the exact resolved version in `record.spec`. `plugins update <id>` later reuses `record.spec`, and the resolver preserves exact versions, so a plugin installed from `@scope/pkg` will be rewritten to `@scope/pkg@oldVersion` and future id-based updates or `--all` runs will stay pinned instead of following the registry default line documented for routine upgrades.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces a supply-chain maintenance risk by making routine npm plugin updates persist and then reuse exact old versions.

  - body: Because the active update selector becomes the resolved exact version, future update runs may not advance third-party npm plugins or hook packs to newer patched releases unless the user manually supplies an unversioned package spec again.
