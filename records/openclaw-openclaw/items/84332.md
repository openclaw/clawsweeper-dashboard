## Summary

Close. The reported fork pattern matches the old heartbeat/active-reply race, and current main plus stable v2026.5.18 now defer heartbeats while the target reply run is active and keep Claude live-session reuse scoped by session identity.

## What This Changes

The issue asks for heartbeat-spawned Claude CLI live sessions to be isolated or excluded from channel inbound routing so channel replies keep their active conversation context.

## Best Possible Solution

Keep the shipped active-reply heartbeat deferral and session-scoped Claude live reuse; reopen only with a current-main or v2026.5.18+ redacted live trace showing heartbeat sessions still receive channel user inbounds.

## Reproduction Assessment

Yes, at source/history level: the old behavior is backed by reporter logs, and the shipped regression now covers the active target reply run that heartbeats must not interrupt. I did not run a live WhatsApp plus Claude CLI reproduction in this read-only review.

## Solution Assessment

Yes. The implemented fix is narrower than adding a heartbeat-disable flag: it blocks the race by deferring heartbeats during active target replies while preserving normal heartbeat behavior.
