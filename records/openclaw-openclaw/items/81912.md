## Summary

Keep open: the PR addresses a real empty-final-after-tools bug and includes credible terminal proof, but the proposed no-tool continuation still leaves hosted/client tools available on the retry.

## What This Changes

The PR adds a one-shot agent runner continuation for empty final assistant responses after completed tools and adds focused regression coverage.

## Best Possible Solution

Keep the retry, but make the forced continuation remove every tool surface, including hosted/client tools, and cover that case with a focused regression test.

## Reproduction Assessment

Yes, at source level: current main’s empty-response/incomplete-turn path can surface the generic warning after a completed tool turn with no visible assistant text. I did not run a live repro in this read-only review.

## Solution Assessment

No as written: retrying once from the existing transcript is the right shape, but the implementation must ensure the continuation truly has no tools, including client tools.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Disable client tools during the continuation retry:** `src/agents/pi-embedded-runner/run.ts:1349`
  - body: The new retry is meant to be a no-tool continuation, but this flag only changes `disableTools`. In `run/attempt.ts`, `clientTools` are still forwarded whenever tools are supported and then converted into custom tools, so a run that has hosted/client tools can still call tools on this retry after local tools already completed. Clear `clientTools` for the forced continuation, or make the attempt layer treat `disableTools` as disabling client tools too, and add coverage for that case.
