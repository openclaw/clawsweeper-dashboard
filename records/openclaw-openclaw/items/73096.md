## Summary

Keep open. Current main still lets a pinned agent harness id bypass the config-backed fallback policy, and changing that path can turn fail-closed missing-harness errors into PI routing with billing-sensitive direct API use, so this needs maintainer policy judgment rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to either honor configured PI fallback for pinned agent harness policies or document that pinned explicit runtimes intentionally fail closed.

## Best Possible Solution

Define the runtime contract for explicit plugin runtime config and persisted session pins, then align selector code, tests, schema help, user docs, and diagnostics. The end state should either preserve fail-closed pins with clearer copy or allow explicit PI fallback with a prominent billing and credential-routing warning.

## Reproduction Assessment

Yes. A static current-main reproduction is `agentHarnessId: "codex"` or another non-auto plugin pin with no registered harness and config fallback set to `pi`; the pinned resolver returns `fallback: "none"`, so selection throws before the config-backed fallback can apply.

## Solution Assessment

Unclear. Passing the configured fallback into `resolvePinnedAgentHarnessPolicy` is mechanically narrow, but it changes billing-sensitive fail-closed behavior; the safer path depends on a maintainer decision about whether pinned sessions may opt into PI fallback.
