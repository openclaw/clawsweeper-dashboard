## Summary

Keep open. Current main still lets a pinned agent harness id bypass the config-backed fallback resolver, while the docs still say explicit runtimes can opt into PI fallback with `fallback: "pi"`; the remaining decision is policy-sensitive because honoring that fallback can change billing behavior.

## What This Changes

The issue asks OpenClaw to reconcile pinned agent harness selection with configured `agentRuntime.fallback` or legacy `embeddedHarness.fallback` so explicit runtime pins do not ignore the operator's fallback setting.

## Best Possible Solution

Define the intended contract for explicit pinned runtimes, then align selector behavior, warnings/diagnostics, docs/schema help, and focused tests around that contract.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure a non-auto runtime with `fallback: "pi"`, let `runAgentAttempt` convert it into `agentHarnessId`, and the pinned resolver still forces `fallback: "none"` before config fallback can apply.

## Solution Assessment

Unclear. Passing configured fallback into the pinned resolver is the narrow code change, but the maintainable fix depends on maintainer policy for billing-sensitive PI fallback and the warning UX.
