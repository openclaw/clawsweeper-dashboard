## Summary

Keep open. Current main still lets a pinned agent harness policy preempt the config-backed fallback resolver, so an explicit runtime pin can ignore `agentRuntime.fallback: "pi"`; the remaining decision is the billing-sensitive policy for fail-closed pins versus opt-in PI fallback.

## What This Changes

The issue asks OpenClaw to either honor configured agent harness fallback for pinned runtimes or document and warn that pinned runtime fallback stays fail-closed.

## Best Possible Solution

Define the intended contract for explicit runtime pins, then align selector code, focused tests, docs/schema/help text, and diagnostics around either strict fail-closed pins or explicit PI fallback with billing-visible warnings.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure a plugin runtime with `fallback: "pi"`, let `runAgentAttempt` convert it into an `agentHarnessId`, and the pinned resolver still forces `fallback: "none"` before config fallback can apply.

## Solution Assessment

Unclear. Passing configured fallback into the pinned resolver is the narrow code change, but the best maintainable solution depends on maintainer policy for billing-sensitive PI fallback and warning behavior.
