## Summary

Keep open. Current main still resolves a supplied/persisted plugin harness id before the config-backed fallback policy, so explicit plugin runtime pins can still ignore `fallback: "pi"`; the remaining choice is a billing-sensitive product contract, not a stale report.

## What This Changes

The issue asks OpenClaw to reconcile pinned agent harness fallback behavior so explicit plugin runtimes can either honor configured PI fallback or fail closed with clearer documentation and warnings.

## Best Possible Solution

Define the explicit-runtime fallback contract, then align selector behavior, diagnostics, billing/cost warnings, docs/schema copy, and a composed attempt-execution test around that contract.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure `agents.defaults.agentRuntime: { id: "codex", fallback: "pi" }`, let `runAgentAttempt` convert that runtime into `agentHarnessId: "codex"`, and the pinned selector still forces `fallback: "none"` before config fallback can apply.

## Solution Assessment

Unclear. Passing configured fallback into the pinned resolver is the narrow code change, but the best maintainable solution depends on whether maintainers want explicit plugin pins to opt into PI fallback and how prominently the billing behavior must be surfaced.
