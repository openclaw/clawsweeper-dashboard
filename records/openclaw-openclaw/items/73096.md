## Summary

Keep open. Current main still derives a session-pinned harness id from explicit runtime config, then the pinned harness resolver ignores the configured `fallback: "pi"` by returning `fallback: "none"`; that preserves the fail-closed safety brake but leaves the documented opt-in fallback path broken.

## What This Changes

The issue asks OpenClaw to honor or explicitly tighten the documented runtime fallback contract when an explicit agent runtime is converted into a pinned harness selection.

## Best Possible Solution

Define the intended contract for explicit runtime pins, then align selector behavior, focused tests, docs/schema/help text, and diagnostics around either strict fail-closed pins or explicit PI fallback with billing-visible warnings.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure an explicit plugin runtime with `fallback: "pi"`, let command execution convert it into an `agentHarnessId`, and the pinned resolver still forces `fallback: "none"` before config fallback can apply.

## Solution Assessment

Unclear. Passing the configured fallback into the pinned resolver is the narrow code fix, but the best maintainable solution depends on maintainer policy for billing-sensitive PI fallback and warning behavior.
