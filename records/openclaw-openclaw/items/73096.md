## Summary

Keep open. Current main still gives pinned harness ids a hardcoded fail-closed policy before the config-backed fallback resolver can run, so the reported behavior remains valid and needs maintainer policy review because honoring PI fallback can affect direct-API billing.

## What This Changes

The issue asks OpenClaw to honor or more clearly document configured agent harness fallback policy when an embedded harness id is pinned.

## Best Possible Solution

Define the intended contract for explicit runtime config and sticky session pins, then align selector code, regression tests, docs/schema help, and diagnostics around either opt-in PI fallback with clear warnings or strict fail-closed pins.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: pass `agentHarnessId: "codex"` with no registered `codex` harness and config `agentRuntime.fallback: "pi"`; the pinned resolver still forces `fallback: "none"`.

## Solution Assessment

Unclear. Passing configured fallback into the pinned resolver is mechanically narrow, but the maintainable fix depends on maintainer policy for billing-sensitive PI fallback and warning behavior.
