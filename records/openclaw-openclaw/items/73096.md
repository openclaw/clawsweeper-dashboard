## Summary

Keep open. Current main still resolves a pinned harness id before the config-backed fallback policy, so the configured opt-in fallback is ignored on pinned/session paths; the prior ClawSweeper comment remains accurate and the remaining decision is billing-sensitive policy, not stale cleanup.

## What This Changes

The issue asks OpenClaw to either honor configured PI fallback for pinned agent harness runtimes or make the strict pinned-runtime behavior clearer in docs and diagnostics.

## Best Possible Solution

Define the intended contract for explicit runtime config and sticky session pins, then align selector code, tests, docs/schema/help text, and diagnostics around either strict fail-closed pins or explicit PI fallback with a prominent billing warning.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure `agents.defaults.agentRuntime: { id: "codex", fallback: "pi" }`, let the runtime become a session harness pin, and selection still throws as if PI fallback were disabled because the pinned resolver forces `fallback: "none"`.

## Solution Assessment

Unclear. Passing a configured fallback into the pinned resolver is mechanically narrow, but the best maintainable fix depends on a maintainer decision about fail-closed behavior versus explicit, potentially billable PI fallback.
