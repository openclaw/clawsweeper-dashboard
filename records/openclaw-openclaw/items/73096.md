## Summary

Keep open. Current main still resolves a pinned harness id through a fail-closed policy before the config-backed fallback path, so the report remains valid and needs an explicit maintainer policy decision because honoring the fallback can affect billing-sensitive PI/direct-API routing.

## What This Changes

The issue asks OpenClaw to honor or more clearly document configured agent harness fallback behavior when an embedded harness id is pinned.

## Best Possible Solution

Define the intended contract for explicit runtime config and sticky session pins, then align selector code, tests, docs, schema/help text, and diagnostics around either opt-in PI fallback with clear billing warnings or strict fail-closed pins.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: provide `agentHarnessId: "codex"`, no registered `codex` harness, and config with `agentRuntime.fallback: "pi"`; the pinned resolver still forces `fallback: "none"` before config fallback is consulted.

## Solution Assessment

Unclear. Passing configured fallback into the pinned resolver is mechanically narrow, but the best maintainable fix depends on a maintainer decision about billing-sensitive PI fallback and warning behavior for pinned runtimes.
