## Summary

Keep open. Current main still resolves a pinned harness id before the config-backed runtime policy and the pinned resolver still hardcodes `fallback: "none"`; changing that would affect fail-closed behavior and potentially billing-sensitive PI/direct-API routing, so this needs maintainer policy review rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to reconcile pinned agent harness fallback handling with the public `agentRuntime.fallback` and legacy `embeddedHarness.fallback` config, either by honoring configured PI fallback for pins or documenting the fail-closed contract.

## Best Possible Solution

Define the contract for explicit runtime config and persisted session pins, then align source, tests, schema help, docs, and diagnostics. Either pinned harness ids may intentionally opt into PI fallback with a prominent billing/credential-routing warning, or pins should always fail closed and the docs/error text should state that directly.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a current-main selection path with `agentHarnessId: "codex"`, no registered `codex` harness, and config fallback set to `pi`; the pinned resolver returns `fallback: "none"` before config fallback can be consulted.

## Solution Assessment

Unclear. Passing configured fallback through the pinned resolver is mechanically narrow, but it changes billing-sensitive fail-closed routing; the best fix depends on a maintainer decision about whether pinned sessions may opt into PI fallback.
