## Summary

Keep this open. Current main still builds a pinned harness policy from `agentHarnessId` before using the config-backed runtime policy, and the pinned resolver still returns `fallback: "none"`, so a real plugin harness pin does not honor same-scope `agentRuntime.fallback`. The Claude CLI-specific angle is less current because CLI aliases are now routed away from embedded harness pins, but the core plugin-pin fallback gap remains and related PR #74341 is still open context, not a merged fix.

## What This Changes

The issue asks OpenClaw to make pinned agent harness fallback behavior consistent with runtime config or to document the fail-closed pinned behavior clearly.

## Best Possible Solution

Define the runtime contract for persisted/session harness pins and make source, tests, diagnostics, and docs agree. A maintainable end state is either to persist or re-resolve fallback policy for pins with an explicit billing warning when PI is allowed, or to keep pins fail-closed and document that pinned sessions never inherit configured fallback.

## Reproduction Assessment

Yes. A static current-main reproduction is an explicit plugin runtime such as `agentRuntime: { id: "codex", fallback: "pi" }` that becomes `agentHarnessId: "codex"`; `selectAgentHarnessDecision` then uses the pinned resolver and throws the disabled-PI-fallback path if that harness is not registered. The original `claude-cli` example no longer reproduces the same way because current main treats CLI aliases as PI/CLI runtime aliases, not embedded harness pins.

## Solution Assessment

Unclear. Passing configured fallback into pinned policy is narrow code-wise, but it changes billing-sensitive fail-closed runtime routing; the safer next step is a maintainer policy decision before choosing code versus docs/error-copy behavior.
