## Summary

Keep this open. Current main still resolves a non-auto `agentHarnessId` through the pinned policy before consulting the config-backed runtime policy, and the pinned resolver still returns `fallback: "none"`; changing that behavior is billing-sensitive and needs maintainer policy review rather than cleanup closure.

## What This Changes

The issue requests that pinned agent harness selection stop silently ignoring same-scope `agentRuntime.fallback`, either by honoring it or by clearly documenting pins as always fail-closed.

## Best Possible Solution

Define the runtime contract for explicit and persisted harness pins, then make code, docs, schema help, diagnostics, and tests agree. Either pins may opt into PI fallback with a prominent billing/credential-routing warning, or pins always fail closed and the docs/error text should say that directly.

## Reproduction Assessment

Yes. A static current-main reproduction is `selectAgentHarness` or an embedded run with `agentHarnessId: "codex"`, no registered `codex` harness, and config `agents.defaults.agentRuntime: { id: "codex", fallback: "pi" }`; the pinned resolver returns `fallback: "none"` and the selection path throws the disabled-PI-fallback error before consulting config.

## Solution Assessment

Unclear. The issue's pass-through option is mechanically narrow, but it changes fail-closed, billing-sensitive routing; the best fix depends on a maintainer decision between configurable fallback with warnings and an explicit fail-closed contract with docs updates.
