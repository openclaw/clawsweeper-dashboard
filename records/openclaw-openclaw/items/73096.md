## Summary

Keep open. Current main still resolves a pinned agent harness id before the config-backed runtime policy, and the pinned resolver still returns `fallback: "none"`; changing that behavior would alter fail-closed routing with billing-sensitive PI/direct-API implications, so this needs maintainer policy review rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to either honor the configured agent-runtime fallback for pinned harness selection or document that pinned runtime selection intentionally always fails closed.

## Best Possible Solution

Define the contract for explicit runtime config and persisted session pins, then align selector code, tests, schema/help text, docs, and diagnostics. Either pinned harness ids can intentionally opt into PI fallback with prominent billing and credential-routing warnings, or pins should always fail closed and the docs/error text should state that unambiguously.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: pass a non-auto `agentHarnessId` such as `codex` while config fallback is `pi` and no matching harness is registered; `resolvePinnedAgentHarnessPolicy` returns `fallback: "none"`, so selection throws that PI fallback is disabled before config fallback is consulted.

## Solution Assessment

Unclear. Passing the configured fallback through the pinned resolver is mechanically narrow, but it changes fail-closed billing-sensitive routing; the maintainable fix depends on an explicit maintainer decision about whether pins may opt into PI fallback and what warning or diagnostic is required.
