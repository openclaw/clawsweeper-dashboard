## Summary

Keep open. Current main still gives pinned harness ids precedence over the config-backed policy and converts the pin into `{ fallback: "none" }`, so the reported fallback gap remains. The narrow code change would alter fail-closed runtime routing and could silently move work to PI/direct-API paths, so this needs maintainer policy review rather than cleanup closure.

## What This Changes

The issue requests that pinned agent harness selection either honor configured embedded runtime fallback or clearly preserve/document fail-closed behavior for explicit runtime pins.

## Best Possible Solution

Define the runtime contract for explicit runtime config and persisted session pins, then align selector source, tests, docs, schema help, and diagnostics. Either pins can intentionally opt into PI fallback with a prominent billing/credential-routing warning, or pins always fail closed and the docs/error text should say that directly.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure a plugin runtime with `fallback: "pi"`, let `runAgentAttempt` turn that runtime into an `agentHarnessId`, then observe `resolvePinnedAgentHarnessPolicy` forcing `fallback: "none"` before selection can consult config.

## Solution Assessment

Unclear. Passing configured fallback into the pinned resolver is mechanically narrow, but it changes billing-sensitive fail-closed behavior; the safer path is a maintainer decision followed by tests and docs that make the chosen contract explicit.
