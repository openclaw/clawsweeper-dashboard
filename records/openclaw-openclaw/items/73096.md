## Summary

Keep open. Current main still gives a supplied `agentHarnessId` priority over the config-backed runtime policy, and the pinned resolver still forces `fallback: "none"`; the original `claude-cli` alias path is now treated as a CLI backend, but the pinned plugin-harness fallback asymmetry remains and is billing-sensitive.

## What This Changes

The issue asks OpenClaw to either honor configured agent-runtime fallback for pinned harness policies or make the fail-closed behavior and billing implications explicit in docs and diagnostics.

## Best Possible Solution

Define the contract for explicit runtime config and sticky harness pins, then align selector behavior, tests, docs/schema help, and gateway diagnostics around that policy.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure `agents.defaults.agentRuntime: { id: "codex", fallback: "pi" }`, let `runAgentAttempt` derive `agentHarnessId: "codex"`, and with no registered `codex` harness the pinned resolver still throws with PI fallback disabled.

## Solution Assessment

Unclear. Passing the configured fallback into `resolvePinnedAgentHarnessPolicy` is mechanically narrow, but the best maintainable fix depends on a maintainer decision about fail-closed pinned runtimes versus explicitly billable PI fallback with warnings.
