## Summary

Keep open. Current main still resolves a persisted or configured non-auto agent harness id through the pinned policy first, and that pinned resolver still hardcodes `fallback: "none"`; the remaining fix is billing-sensitive because honoring the configured fallback can change a fail-closed selection error into PI/direct-API routing.

## What This Changes

The issue asks OpenClaw to either honor configured PI fallback for pinned agent harness selection or make the fail-closed behavior explicit in the runtime contract, docs, diagnostics, and tests.

## Best Possible Solution

Define the contract for explicit and persisted harness pins, then align selector code, session pinning, docs, schema help, diagnostics, and tests. Either same-scope `fallback: "pi"` must be honored for pins with a clear billing warning, or pins must remain fail-closed and the config/docs/error text should state that pinned sessions do not opt into PI fallback.

## Reproduction Assessment

Yes. A static current-main reproduction is an embedded run whose config resolves to `agentRuntime.id: "codex", fallback: "pi"` while no `codex` harness is registered: `runAgentAttempt` passes `agentHarnessId: "codex"`, and `selectAgentHarnessDecision` throws the disabled-PI-fallback error because the pinned policy overrides the config fallback.

## Solution Assessment

Unclear. Passing configured fallback into `resolvePinnedAgentHarnessPolicy` is mechanically narrow, but it changes fail-closed, billing-sensitive routing; the best fix needs a maintainer decision between configurable fallback with warnings and an explicit fail-closed pin contract.
