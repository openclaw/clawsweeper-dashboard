## Summary

Keep open. Current main still lets a pinned agent harness policy override the config-backed fallback resolver for explicit plugin harness ids, and the remaining fix needs a maintainer decision about fail-closed safety versus opt-in PI fallback billing behavior.

## What This Changes

Honor or clearly constrain `agentRuntime`/legacy `embeddedHarness` fallback behavior when agent harness selection is pinned.

## Best Possible Solution

Define the intended contract for explicit runtime pins, then align selector behavior, diagnostics, docs/schema copy, and focused composition tests around that contract.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure a plugin runtime with `fallback: "pi"`, let `runAgentAttempt` convert that runtime into `agentHarnessId`, and the pinned resolver still forces `fallback: "none"` before config fallback can apply.

## Solution Assessment

Unclear. Passing configured fallback into the pinned resolver is the narrow code change, but the best maintainable solution depends on maintainer policy for billing-sensitive PI fallback and warning behavior.
