## Label Justifications

- `P1`: The underlying bug can block resumed or targeted session writes for users carrying migrated session metadata, and the PR has a blocking correctness gap.
- `merge-risk: 🚨 compatibility`: The PR rewrites persisted path metadata during store saves and must preserve custom `session.store` and legacy upgrade behavior.
- `merge-risk: 🚨 session-state`: An incorrect canonicalization target can leave stale transcript paths in place or split metadata from the transcript files used by later turns.

## Summary

Keep this PR open for changes. The save-time cleanup targets a real session-state bug, but the patch reuses the compatibility resolver that can preserve structurally valid old-root `agents/<id>/sessions/<file>` paths, so the central stale-path case can still survive a store save.

## What This Changes

The PR adds save-time normalization of persisted `sessionFile` and `transcriptPath` fields, promotes `transcriptPath` onto `SessionEntry`, and adds a pruning integration test.

## Best Possible Solution

Derive save-time canonical `sessionFile` and `transcriptPath` values directly from the active `sessionsDir` and validated `sessionId`, align that rule with https://github.com/openclaw/openclaw/pull/61322, and cover structurally valid old-root plus custom `session.store` cases.

## Reproduction Assessment

Yes, at source level. Current main can preserve a structurally valid old-root `sessionFile`, and the PR's save-time helper calls that same resolver, so the stale value can remain persisted.

## Solution Assessment

No as written. Save-time canonicalization is a useful boundary, but it should build the target from the active `sessionsDir` rather than call a compatibility resolver that is allowed to preserve old absolute paths.
