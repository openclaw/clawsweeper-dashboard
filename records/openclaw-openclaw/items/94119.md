## Summary

Keep open: this is a small warning-only mitigation, but current main prints the update result before managed restart and verification, and the PR still lacks real update/gateway proof. It is not merge-ready and should not close the canonical stale-gateway issue as written.

## What This Changes

Adds a warning to update result output when a successful update changes versions, telling users to restart the gateway.

## Best Possible Solution

Keep restart guidance tied to actual restart outcome: warn only when restart was skipped, failed, unmanaged, or still unhealthy after verification, and keep the canonical stale-gateway issue open until the chosen recovery path is live-proven.

## Reproduction Assessment

Yes for the PR defect: source inspection shows a successful version-changing update calls printResult before maybeRestartService, so the new warning can appear before the command restarts and verifies the Gateway. I did not run a live package-update flow in this read-only review.

## Solution Assessment

No, this is only a plausible mitigation. The warning should be emitted from restart outcome handling, or maintainers should choose a broader canonical fix for the stale-gateway cluster before closing the linked issue.
