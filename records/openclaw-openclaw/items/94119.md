## Summary

Keep open. The PR is a narrow warning-only mitigation for a real stale-gateway/message-loss regression, but the warning is emitted before current update flow knows whether the Gateway will be restarted and verified, and the contributor proof is test/static-only rather than real update behavior.

## What This Changes

Adds a terminal warning to successful version-changing update output telling users to restart the Gateway.

## Best Possible Solution

Tie restart guidance to the actual restart outcome, or choose a maintainer-approved warning boundary while keeping the canonical stale-gateway issue open until live package-swap recovery is proven.

## Reproduction Assessment

Yes for the review finding: current-main source shows successful updates call printResult before maybeRestartService, so the warning can appear before restart and verification. I did not run a live package-swap flow in this read-only review.

## Solution Assessment

No. This is a plausible mitigation, but the maintainable fix is to emit restart guidance from skipped, failed, unmanaged, or unhealthy restart handling, or to choose a broader update/runtime recovery boundary first.
