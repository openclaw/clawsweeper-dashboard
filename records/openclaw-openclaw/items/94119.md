## Summary

Keep open: the PR is a plausible warning-only mitigation, but it still lacks real update/gateway proof and the warning is emitted before current `main` knows whether it will restart and verify the managed Gateway.

## What This Changes

Adds a success-case update result warning in `src/cli/update-cli/progress.ts` when before/after versions differ, telling users to restart the gateway.

## Best Possible Solution

Keep restart guidance tied to the actual update outcome: warn only when restart was skipped, failed, unmanaged, or still unhealthy after verification, while keeping the canonical stale-gateway issue open until a live-proven fix lands.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection gives a high-confidence path for the PR defect: the warning would print in `printResult` before later restart handling decides whether the Gateway was restarted and verified.

## Solution Assessment

No, not as written. A warning can be a limited mitigation, but this placement is too broad for current managed update semantics and too narrow to close the linked raw rollback delivery bug.
