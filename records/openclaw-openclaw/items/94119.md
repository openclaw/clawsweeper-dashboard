## Summary

Keep open. The warning-only branch is not merge-ready because it can emit restart guidance before current main's managed restart and verification path runs, lacks real after-fix behavior proof, and overlaps with other open candidates for the canonical stale-gateway issue.

## What This Changes

The PR adds a successful version-change warning to update result output telling users to restart the gateway.

## Best Possible Solution

Keep restart guidance tied to the actual update outcome: warn only when restart was skipped, failed, unmanaged, or still unhealthy after verification, while leaving the canonical stale-gateway issue open unless maintainers explicitly accept a narrower warning-only fix.

## Reproduction Assessment

Yes for the PR defect from source inspection: successful non-JSON updates call printResult before maybeRestartService, so the new warning can appear before the command knows whether restart succeeded. I did not run a live package-update flow in this read-only review.

## Solution Assessment

No, not as written. A warning is a plausible mitigation, but the maintainable placement is in restart outcome handling, and the linked stale-gateway issue should remain canonical unless maintainers accept warning-only scope.
