## Summary

Keep open. Current main still preserves custom Docker bind writability in path resolution, but normal sandbox file-tool writes are rejected unless `workspaceAccess` is `"rw"`; related work covers adjacent workspace/Codex paths without settling this least-privilege custom-bind policy.

## What This Changes

Request normal file tools to write only to explicitly `:rw` custom Docker bind mounts while `workspaceAccess` remains `"none"`.

## Best Possible Solution

Define a mount-aware sandbox filesystem policy that lets normal file tools mutate only resolved writable custom binds under `workspaceAccess: "none"`, while preserving read-only bind enforcement, blocked root validation, bridge parity, docs, tests, and Docker proof.

## Reproduction Assessment

Yes, at source level. A `:rw` custom bind resolves with `writable: true`, but `ensureWriteAccess` still rejects mutation unless `workspaceAccess` is `"rw"`; I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

Yes at the design level, pending maintainer approval. Authorizing writes from the resolved mount's writability is the narrow least-privilege direction, but it must be made explicit across bridges, tools, docs, security tests, and Docker proof.
