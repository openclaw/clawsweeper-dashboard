## Summary

Keep open: current main still preserves custom bind writability metadata but rejects all sandbox file-tool mutations unless `workspaceAccess` is `"rw"`; the adjacent sandbox issues do not supersede this explicit writable-bind authorization request.

## What This Changes

Request to let normal sandbox file tools write to explicitly `:rw` Docker custom binds while `workspaceAccess` remains `"none"`.

## Best Possible Solution

Implement mount-aware write authorization in the sandbox file bridge: allow mutations only on resolved writable binds, keep `workspaceAccess: "none"` from granting broad workspace writes, preserve `:ro` and path-safety failures, and document/test write, edit, mkdir/remove/rename, and `apply_patch` behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction is `workspaceAccess: "none"` plus a `:rw` bind such as `/tmp/persona:/persona:rw`, then a sandboxed write to `/persona/file`; the resolver can mark the target writable, but `ensureWriteAccess` rejects because `allowsWrites` only accepts `"rw"`.

## Solution Assessment

Yes on direction: resolved mount writability is the narrow maintainable boundary. It still needs secops/product review because this changes host-write authorization, and `apply_patch` needs an explicit writable-bind policy.

## Risks / Open Questions

- Changing this behavior broadens when sandboxed agents can write to host-mounted paths, so the fix must preserve read-only bind rejection, blocked-source validation, and symlink/hardlink path guards.
- `apply_patch` has an additional workspace-only path policy; without explicit design it could either remain unable to patch writable binds or become broader than intended.
- The adjacent external bind-source policy remains open at https://github.com/openclaw/openclaw/issues/57271, so some real deployments may also need that contract clarified before the least-privilege workflow is fully usable.
