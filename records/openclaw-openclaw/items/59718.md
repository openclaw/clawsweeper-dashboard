## Summary

Keep this issue open. Current main still preserves custom bind `:ro`/`:rw` metadata, but both sandbox filesystem write bridges reject mutations unless `workspaceAccess` is `"rw"`, so `workspaceAccess: "none"` plus an explicit writable bind is still not supported; the related sandbox reports are adjacent rather than superseding this mount-level authorization request.

## What This Changes

The issue requests writable custom Docker bind mounts to remain usable by normal file tools while `workspaceAccess` stays `"none"` for least-privilege sandbox isolation.

## Best Possible Solution

Implement mount-aware write authorization in the sandbox file bridge: allow mutations only on resolved writable binds, keep `workspaceAccess: "none"` from granting broad workspace writes, preserve read-only bind and path-safety failures, and document/test write, edit, mkdir/remove/rename, and apply_patch behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `workspaceAccess: "none"` plus a `:rw` bind such as `/tmp/persona:/persona:rw`, then a sandboxed `write` to `/persona/file`; the resolver can mark the target writable, but `ensureWriteAccess` rejects because `allowsWrites` only accepts `"rw"`.

## Solution Assessment

Yes at the design level: making authorization depend on the resolved mount's writable flag is the narrow maintainable direction. Treating `workspaceAccess: "none"` as generally writable would be broader and would weaken the intended workspace isolation model.
