## Summary

Keep this issue open. Current main still records custom bind `:ro`/`:rw` metadata, but local and remote sandbox write bridges reject mutations unless `workspaceAccess` is `"rw"`; the adjacent open issues and PRs cover workspace writability, external bind source validation, or tool-surface gating rather than this mount-level least-privilege authorization request.

## What This Changes

Allow normal file tools to write only to explicitly writable custom Docker binds while `workspaceAccess` remains `"none"`.

## Best Possible Solution

Implement a mount-aware sandbox file authorization policy that allows mutations only on resolved writable binds, keeps broad workspace writes disabled under `workspaceAccess: "none"`, preserves read-only/path-safety failures, and documents/tests write, edit, mkdir/remove/rename, and apply_patch behavior.

## Reproduction Assessment

Yes, at source level. With `workspaceAccess: "none"` and a `:rw` bind such as `/tmp/persona:/persona:rw`, `resolveSandboxFsPathWithMounts` can return a writable bind target, but `ensureWriteAccess` still rejects because `allowsWrites` only accepts `"rw"`; no live Docker sandbox was run in this read-only review.

## Solution Assessment

Yes at the design level, but it needs maintainer/security approval. The narrow maintainable direction is to authorize writes from the resolved mount's writable flag while preserving workspace isolation and read-only bind/path-safety enforcement, rather than treating all `none` paths as writable.
