## Summary

Keep open. Current main still parses custom Docker binds with independent writable metadata, but sandbox file mutations are rejected unless `workspaceAccess` is `"rw"`, so writable binds under `"none"` remain a real least-privilege sandbox capability gap that needs maintainer security/product approval.

## What This Changes

Allow sandboxed normal file tools to write only explicit custom `:rw` Docker binds while `workspaceAccess` remains `"none"` and broad workspace writes stay denied.

## Best Possible Solution

Approve and implement one resolved-mount write policy: file tools may mutate only explicit writable custom binds under `workspaceAccess: "none"`, while `/workspace`, read-only binds, protected skill overlays, blocked roots, and path guards remain denied.

## Reproduction Assessment

Yes, by source inspection. Current main resolves custom Docker bind mounts with writable metadata, but the filesystem bridge still rejects mutations unless `workspaceAccess` is `"rw"`; no live Docker repro was run in this read-only review.

## Solution Assessment

Unclear until maintainers settle the sandbox security policy. If approved, the narrow maintainable solution is mount-aware write authorization for explicit `:rw` binds, not broad `workspaceAccess: "rw"` or an unscoped host-write fallback.
