## Summary

Keep this issue open. Current main still preserves custom bind `:rw` metadata, but sandbox write authorization and tool exposure are not mount-aware, so `workspaceAccess: "none"` cannot safely support normal file-tool writes to only selected writable binds yet; the related open items cover adjacent workspace writability, bind-source validation, or tool-surface gating rather than this exact least-privilege host-bind policy.

## What This Changes

Request support for writable custom Docker bind mounts while `workspaceAccess` remains `"none"`, so only explicitly mounted paths are writable.

## Best Possible Solution

Implement a mount-aware sandbox filesystem policy that allows mutations only on resolved writable custom binds, keeps broad workspace writes disabled under `workspaceAccess: "none"`, preserves `:ro` bind and path-safety failures, and documents/tests write, edit, mkdir/remove/rename, and apply_patch behavior.

## Reproduction Assessment

Yes, at source level. With `workspaceAccess: "none"` and a `:rw` bind such as `/persona`, `resolveSandboxFsPathWithMounts` can return a writable bind target, but `ensureWriteAccess` still rejects because `allowsWrites` accepts only `"rw"`; no live Docker sandbox was run.

## Solution Assessment

Yes at the design level, but it needs maintainer/security approval. The narrow maintainable direction is to authorize writes from the resolved mount's writable flag while preserving workspace isolation, read-only bind enforcement, and path guards.
