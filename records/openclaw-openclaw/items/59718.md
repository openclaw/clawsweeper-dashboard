## Summary

Keep this issue open. Current main still resolves custom Docker bind mounts with their `:ro`/`:rw` writability, but the local and remote sandbox write bridges still reject all mutations unless `workspaceAccess` is exactly `"rw"`, so the requested least-privilege writable-bind mode is not implemented. The adjacent open reports cover external bind sources and broader sandbox workspace permissions, but they do not supersede this mount-level authorization request.

## What This Changes

Allow normal sandbox file tools to write to explicitly `:rw` custom Docker binds while `workspaceAccess` remains `"none"`, preserving read-only binds and main workspace isolation.

## Best Possible Solution

Implement a mount-aware sandbox write authorization contract in the filesystem bridges: allow mutations only when the resolved target is on a writable mount, keep `workspaceAccess: "none"` from granting broad workspace writes, preserve `:ro` bind failures, and add focused regression coverage for `write`, `edit`, mkdir/remove/rename, and `apply_patch` on writable and read-only custom binds.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is to configure `workspaceAccess: "none"` plus a `:rw` Docker bind and call `write` or `edit` on the bind target; current resolution can mark the bind writable, but `ensureWriteAccess` rejects before mutation because `allowsWrites("none")` is false.

## Solution Assessment

Yes, the mount-aware authorization direction is the narrow maintainable fix. It should be implemented with explicit sandbox security review and `apply_patch` policy coverage rather than by broadly treating `workspaceAccess: "none"` as writable.
