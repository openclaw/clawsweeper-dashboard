## Summary

Keep open. Current main and the latest release still parse custom Docker `:rw` binds as writable mounts, but normal OpenClaw file-tool mutations are blocked unless `workspaceAccess` is `"rw"`; related issues and PRs cover adjacent read resolution, native Codex policy, tool-surface gating, or isolated `/workspace` semantics rather than this custom-bind write policy.

## What This Changes

Allow normal file tools to write to explicit Docker `:rw` custom bind mounts while `workspaceAccess` remains `"none"`.

## Best Possible Solution

Define a mount-aware sandbox filesystem policy where normal file tools may mutate only resolved `:rw` custom binds under `workspaceAccess: "none"`, with unchanged read-only and blocked-root behavior plus focused Docker proof.

## Reproduction Assessment

Yes, by source inspection. Current main resolves Docker bind mounts with writable metadata, but `ensureWriteAccess` still rejects mutation unless `workspaceAccess` is `"rw"`; I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

Yes at the direction level, pending maintainer approval. The maintainable solution is resolved-mount authorization for explicit writable binds plus docs, tests, and Docker proof, not broad `workspaceAccess: "rw"` or an unscoped host-write fallback.
