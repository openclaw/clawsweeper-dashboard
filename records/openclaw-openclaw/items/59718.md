## Summary

Keep open. Current main and latest release still record custom Docker bind writability, but normal sandbox file mutations are rejected unless `workspaceAccess` is `"rw"`; this is a real least-privilege sandbox capability gap that needs maintainer security/product approval.

## What This Changes

Issue requests least-privilege Docker sandbox support where `workspaceAccess: "none"` keeps the main workspace isolated while writable custom bind mounts remain editable by normal file tools.

## Best Possible Solution

Approve a resolved-mount write policy where file tools may mutate only explicit writable custom binds under `workspaceAccess: "none"`, while read-only binds and all non-mounted paths remain denied.

## Reproduction Assessment

Yes, by source inspection. Current main resolves custom Docker bind mounts with writable metadata, but the fs bridge still rejects mutations unless `workspaceAccess` is `"rw"`; no live Docker repro was run in this read-only review.

## Solution Assessment

Unclear until maintainers settle the security policy. If approved, the narrow maintainable solution is mount-aware write authorization for explicit `:rw` binds, not broad `workspaceAccess: "rw"` or an unscoped host-write fallback.
