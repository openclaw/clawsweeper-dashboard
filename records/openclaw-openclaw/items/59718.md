## Summary

Keep open. Current main preserves custom bind `:ro`/`:rw` metadata, but sandbox write bridges still require `workspaceAccess: "rw"`, so explicit writable binds under `workspaceAccess: "none"` remain unsupported; related sandbox items are adjacent rather than superseding this mount-level authorization request.

## What This Changes

Allow sandbox file tools to write only to explicitly writable custom Docker bind mounts while `workspaceAccess` remains `"none"`.

## Best Possible Solution

Implement mount-aware write authorization for sandbox file bridges and tool exposure: allow mutations only on resolved writable binds, keep broad workspace writes blocked under `"none"`, preserve read-only bind and path-safety failures, and document/test write, edit, mkdir/remove/rename, and apply_patch behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction is `workspaceAccess: "none"` plus a writable bind such as `/tmp/persona:/persona:rw`, then a sandboxed write to `/persona/file`; the resolver can mark the bind writable, but `ensureWriteAccess` still rejects because workspace access is not `"rw"`.

## Solution Assessment

Yes at the design level: authorizing writes from the resolved mount's writable flag is the narrow maintainable direction. The implementation still needs maintainer/secops policy review and reconciliation with the current write-tool gating work.
