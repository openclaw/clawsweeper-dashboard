## Summary

Keep open. Current main still rejects normal sandbox file-tool writes unless `workspaceAccess` is `"rw"`, even when the resolved target is an explicitly writable Docker bind; adjacent Codex bind-policy work and the open writable-`none` PR do not fully settle the core sandbox write boundary.

## What This Changes

Allow sandboxed file tools to write explicitly `:rw` Docker bind mounts while `workspaceAccess: "none"` keeps the main workspace hidden.

## Best Possible Solution

Define a mount-aware sandbox filesystem policy that lets normal file tools mutate only resolved writable custom binds under `workspaceAccess: "none"`, preserves read-only and blocked-path enforcement, and aligns local/remote bridges, tool exposure, docs, tests, and Docker proof.

## Reproduction Assessment

Yes, at source level. A `:rw` custom bind can resolve as writable, but `ensureWriteAccess` still rejects the mutation unless `workspaceAccess` is `"rw"`; I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

Yes at the design level, pending maintainer approval. Authorizing writes from the resolved mount's writability is the narrow least-privilege direction, but it must be made explicit across bridges, tools, docs, and security tests.
