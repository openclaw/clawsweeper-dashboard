## Summary

Keep this issue open. Current main still records custom bind mount writability, but sandbox file mutations are rejected unless `workspaceAccess` is `"rw"`; the related reports are adjacent sandbox/write issues and do not supersede this mount-level least-privilege request.

## What This Changes

Allow sandboxed agents to write to explicitly `:rw` custom Docker binds while keeping `workspaceAccess: "none"` from granting broad workspace writes.

## Best Possible Solution

Implement mount-aware sandbox write authorization: allow mutations only on resolved writable binds, keep `workspaceAccess: "none"` from granting workspace-wide writes, preserve RO/path-safety failures, and document/test write, edit, mkdir/remove/rename, and apply_patch behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `workspaceAccess: "none"` plus a `:rw` bind such as `/tmp/persona:/persona:rw`, followed by a sandboxed `write` to `/persona/file`; resolution marks the bind writable, but `ensureWriteAccess` rejects because the workspace access is not `"rw"`.

## Solution Assessment

Yes, the issue's mount-aware authorization direction is the narrow maintainable fix. Treating `workspaceAccess: "none"` as generally writable would weaken the intended isolation model.
