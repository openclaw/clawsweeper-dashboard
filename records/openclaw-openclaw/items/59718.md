## Summary

Keep this issue open. A merged adjacent fix now handles some Codex native Docker bind writable-root cases, but current main still does not provide mount-aware normal OpenClaw `write`/`edit`/`apply_patch` behavior for `workspaceAccess: "none"`; the remaining work is a sandbox host-write policy decision, not a stale or fully implemented request.

## What This Changes

Request least-privilege Docker sandbox behavior where `workspaceAccess: "none"` keeps the main workspace isolated while normal file tools may write only to explicit `:rw` custom binds.

## Best Possible Solution

Define and implement a mount-aware sandbox filesystem policy that allows mutations only on resolved writable custom binds, keeps the main workspace isolated under `workspaceAccess: "none"`, preserves `:ro` and path-safety failures, and documents/tests the normal file-tool contract.

## Reproduction Assessment

Yes, at source level. A `:rw` custom bind can resolve as writable, but `ensureWriteAccess` still rejects normal file-tool mutations unless `workspaceAccess` is `"rw"`.

## Solution Assessment

Yes at the design level, pending maintainer approval. The maintainable direction is to authorize writes from the resolved mount's writability while keeping broad workspace writes disabled and preserving read-only bind enforcement.
