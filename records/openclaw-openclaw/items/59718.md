## Summary

Keep open. Current main still represents custom Docker `:rw` binds as writable mount records, but the sandbox filesystem bridge rejects every mutation unless `workspaceAccess` is `"rw"`; this is a real least-privilege sandbox capability gap that needs maintainer security/product approval rather than cleanup closure.

## What This Changes

The issue requests normal file-tool writes to explicitly writable Docker custom binds while keeping `workspaceAccess: "none"` and the main workspace isolated.

## Best Possible Solution

Approve a resolved-mount write policy where normal file tools may mutate only targets that resolve to explicit `:rw` custom binds under `workspaceAccess: "none"`, while all read-only and blocked paths remain denied.

## Reproduction Assessment

Yes, by source inspection. Current main resolves custom Docker bind mounts with writable metadata, but the fs bridge still rejects mutations unless `workspaceAccess` is `"rw"`; I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

Unclear until maintainers settle the security policy. If approved, the narrow maintainable shape is mount-aware write authorization for explicit `:rw` binds, not broad `workspaceAccess: "rw"` or an unscoped host-write fallback.
