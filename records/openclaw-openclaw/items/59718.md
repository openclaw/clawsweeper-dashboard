## Summary

Keep open: current main still resolves custom Docker binds as mount-aware targets, but the local sandbox file bridge rejects every write unless `workspaceAccess` is `"rw"`, so the requested least-privilege writable-bind mode is not implemented.

## What This Changes

Allow file tools to write to explicitly `:rw` Docker custom bind mounts while `workspaceAccess` remains `"none"` and the real workspace stays hidden.

## Best Possible Solution

Define the sandbox contract so file-tool mutations are authorized by the resolved mount’s writability, preserve `:ro` bind denials and real-workspace isolation, and align tests/docs with the chosen relationship to https://github.com/openclaw/openclaw/issues/37634.

## Reproduction Assessment

Yes, at source level. With `workspaceAccess: "none"`, a path under a configured `:rw` custom bind resolves as writable, but the local file bridge still rejects the write because `allowsWrites()` only accepts `"rw"`.

## Solution Assessment

Unclear until maintainer/security review decides the contract. The likely clean fix is mount-aware write authorization, but it must be coordinated with the broader isolated-workspace writability work in https://github.com/openclaw/openclaw/issues/37634.
