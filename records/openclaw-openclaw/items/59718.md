## Summary

Keep this issue open. Current main still preserves custom bind `:ro`/`:rw` metadata, but the sandbox write bridge rejects mutations unless `workspaceAccess` is `"rw"`, so `workspaceAccess: "none"` plus an explicit writable bind is still not supported; the related sandbox reports are adjacent but do not supersede this mount-level authorization request.

## What This Changes

Allow normal sandbox file tools to write to explicitly `:rw` custom Docker bind mounts while `workspaceAccess: "none"` keeps the agent workspace isolated.

## Best Possible Solution

Implement mount-aware write authorization in the sandbox file bridge: allow mutations only on resolved writable binds, keep `workspaceAccess: "none"` from granting broad workspace writes, preserve `:ro` and path-safety failures, and document/test write, edit, mkdir/remove/rename, and apply_patch behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `workspaceAccess: "none"` plus a `:rw` bind such as `/tmp/persona:/persona:rw`, then a sandboxed `write` to `/persona/file`; the resolver can mark the target writable, but `ensureWriteAccess` rejects because `allowsWrites` only accepts `"rw"`.

## Solution Assessment

Yes, the issue's mount-aware authorization direction is the narrow maintainable path. Treating `workspaceAccess: "none"` as generally writable would be broader and would weaken the intended workspace isolation model.
