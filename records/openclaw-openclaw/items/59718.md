## Summary

Keep open. Current main still has the requested custom-bind behavior missing for normal OpenClaw file tools: Docker `:rw` binds are represented as writable, but the filesystem bridge rejects mutations unless `workspaceAccess` is `"rw"`; the merged Codex bind-policy fix covers a separate native Codex path, and the open writable-`none` workspace issue is adjacent rather than a complete replacement.

## What This Changes

Request writable normal file-tool access to explicit Docker `:rw` custom binds while `workspaceAccess: "none"` keeps the main workspace isolated.

## Best Possible Solution

Define and implement a mount-aware sandbox filesystem policy where normal file tools may mutate only resolved `:rw` custom binds under `workspaceAccess: "none"`, with unchanged read-only/blocked-root behavior and focused Docker proof.

## Reproduction Assessment

Yes, source-level. A `:rw` bind resolves as writable, but `ensureWriteAccess` still rejects mutation unless `workspaceAccess` is `"rw"`; I did not run a live Docker sandbox in this read-only review.

## Solution Assessment

Yes at the direction level, pending maintainer approval. The maintainable fix is resolved-mount authorization for explicit writable binds plus docs/tests/proof, not broad `workspaceAccess: "rw"` or an unscoped host-write fallback.
