## Summary

Keep open because current main still has the reported heartbeat scheduling behavior, the launchd plist still lacks the macOS App Nap mitigation, and the issue is paired with open same-author implementation PRs #62308 and #62310.

## What This Changes

The issue asks OpenClaw to enforce heartbeat spacing for non-interval wake reasons and make macOS launchd-installed gateways resilient to idle timer stalls.

## Best Possible Solution

Keep this issue open as the canonical report while maintainers review #62308 and #62310, or replace them with one coordinated fix that updates the heartbeat runner, launchd plist, and focused regression tests.

## Reproduction Assessment

Partially yes. The non-interval bypass has a high-confidence code-level reproduction by firing a targeted or fan-out non-interval wake before `nextDueMs`; the macOS silent-gap portion needs live macOS launchd/App Nap validation.

## Solution Assessment

Yes. The maintainable direction is to fix the generic heartbeat scheduling semantics and generated macOS launchd plist rather than throttling individual wake call sites or shelling out to `defaults`/`caffeinate` at gateway startup.
