## Summary

Keep open because current main only addresses part of the burst class with the newer typed source/intent cooldown path; the macOS silent-gap/App Nap surface is still unresolved, the launchd plist still lacks `ProcessType`, and same-author PRs remain open for the two proposed fix surfaces.

## What This Changes

The issue requests reliable heartbeat scheduling by applying interval/cooldown enforcement to non-interval wake reasons and making macOS timer suspension/App Nap less likely to create silent heartbeat gaps.

## Best Possible Solution

Keep this issue open as the broad tracking report while maintainers reconcile #62308 and #62310 with the current #76086 cooldown model, preserving explicit wake-now contracts and adding focused regression/live macOS proof for the remaining silent-gap path.

## Reproduction Assessment

Partially yes. The event-driven burst class is source-reproducible and now covered by current-main cooldown tests, but the remaining silent-gap/App Nap path needs live macOS launchd validation even though current source still omits the requested plist mitigation.

## Solution Assessment

Partially. The central typed source/intent cooldown is the right direction for event-driven bursts, but the remaining macOS scheduler/App Nap fix should be reviewed separately and should not blanket-throttle documented manual or explicit immediate wake paths.
