## Summary

Keep open. Current main still allows non-interval heartbeat wakes to bypass the due-time gate, still advances the next interval after action-driven wakes, and the generated launchd plist still lacks the macOS `ProcessType` setting discussed in the report. The issue is also paired with the reporter's open implementation PRs #62308 and #62310, so maintainer cleanup should not close the issue ahead of those PRs being reviewed, replaced, or landed.

## What This Changes

The issue requests heartbeat scheduler changes so action-driven wake reasons respect heartbeat spacing and macOS launchd installs avoid App Nap timer stalls.

## Best Possible Solution

Keep this issue open as the canonical report while maintainers review #62308 and #62310, or replace them with one coordinated fix that adds the launchd plist setting, non-interval wake spacing, overdue/drift handling, and focused regression tests.

## Reproduction Assessment

Partially yes. The scheduler bypass has a high-confidence code-level reproduction by firing a non-interval wake before `nextDueMs`; the macOS silent-gap behavior needs live launchd/App Nap validation on macOS to prove the OS-level symptom.

## Solution Assessment

Yes, the best maintainable direction is to fix the generic heartbeat runner scheduling semantics and the generated macOS launchd plist, not to add per-call-site throttles or shell out to `defaults`/`caffeinate` from gateway startup.
