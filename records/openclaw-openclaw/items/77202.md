## Summary

Keep open: current main still lacks Signal live tool-call progress via a send/delete lifecycle, and the reporter's v2 proposal reduces notification noise but still needs maintainer UX, config, and delete-semantics decisions before implementation.

## What This Changes

Add Signal live tool-call progress using an edit-free send/delete lifecycle, preferably coalesced and configurable to avoid notification noise.

## Best Possible Solution

If maintainers accept the UX, implement a Signal-owned opt-in progress lifecycle using coalesced send/delete messages, existing reply progress callbacks, a typed remoteDelete helper, config/docs/tests/changelog, and defer a shared core tombstone abstraction until another channel needs it.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main confirms Signal supports typing, read receipts, and normal replies, but has no existing send/delete tool-progress lifecycle to reproduce as a broken contract.

## Solution Assessment

Unclear until maintainers choose the product/API shape. The narrowest maintainable direction appears to be a Signal-owned, opt-in coalesced send/delete implementation rather than a shared tombstone preview abstraction at this stage.
