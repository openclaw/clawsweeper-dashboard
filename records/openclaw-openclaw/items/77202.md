## Summary

Keep open: current main still lacks Signal live tool-progress via an edit-free send/delete lifecycle. The reporter's v2 comment usefully narrows the notification-noise concern to a rolling tombstone pattern, but this remains a new Signal plugin capability that needs maintainer UX, config, and signal-cli delete-semantics review before implementation.

## What This Changes

Request to add an opt-in Signal channel mode that sends temporary live tool-call progress messages and deletes or replaces them when the final reply is queued.

## Best Possible Solution

Implement this in the Signal plugin as an opt-in, coalesced progress lifecycle using existing reply progress callbacks plus a Signal delete helper, with config schema, docs, tests, and changelog; defer any shared tombstone abstraction until another channel needs the same transport.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing Signal contract. Source and docs confirm current Signal behavior is typing/read receipts plus normal/block reply delivery, with no edit-free tool-progress lifecycle to reproduce as a bug.

## Solution Assessment

Unclear until maintainers choose the product/API shape. The narrowest maintainable path appears to be a Signal-owned opt-in implementation; a shared pipeline tombstone abstraction should wait for evidence that multiple channels need it.
