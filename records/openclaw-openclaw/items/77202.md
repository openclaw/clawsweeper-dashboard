## Summary

Keep open: current main does not implement Signal live tool-call progress with a send/delete lifecycle, and the request is a new Signal plugin capability that needs maintainer UX/API judgment rather than cleanup closure.

## What This Changes

The issue requests live Signal tool-call progress using separate progress messages that are deleted when the final reply is queued.

## Best Possible Solution

If maintainers accept the UX, implement this in the Signal plugin as an opt-in progress lifecycle using existing reply progress callbacks plus a Signal delete helper, with config schema, docs, tests, and changelog aligned.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms current main has Signal typing and normal reply delivery, but no edit-free tool-progress message lifecycle and no docs promise that behavior.

## Solution Assessment

Unclear until maintainers choose the product/API shape. The narrowest maintainable path appears to be a Signal-owned opt-in implementation; a shared tombstone preview abstraction should wait for evidence that multiple channels need it.
