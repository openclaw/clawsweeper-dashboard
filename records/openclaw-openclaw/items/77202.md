## Summary

Keep open: current main still lacks Signal edit-free live tool-call progress, and the request is a new Signal channel capability that needs maintainer UX/API judgment rather than cleanup closure.

## What This Changes

Request Signal live tool-call progress by sending transient progress messages during a turn and deleting them when the final reply is queued.

## Best Possible Solution

If maintainers accept the UX, implement this in the Signal plugin as an opt-in send/delete progress lifecycle using existing reply progress callbacks, with config schema, docs, tests, and changelog aligned.

## Reproduction Assessment

Not applicable as a feature request. Source inspection confirms current Signal wiring supports typing and final replies but has no edit-free tool-progress lifecycle to reproduce as a broken existing contract.

## Solution Assessment

Unclear until maintainers choose the UX and API shape. The narrowest maintainable path appears Signal-owned and opt-in; a shared tombstone preview abstraction should wait for evidence that multiple edit-free channels need it.
