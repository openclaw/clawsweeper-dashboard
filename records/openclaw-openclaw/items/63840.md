## Summary

Keep open. Current main still drops the Slack Agents & Assistants DM self-thread root outside `replyToMode=all`, and the PR head is not ready because its new guard depends on raw `channel_type` instead of the normalized DM classification already used by the Slack pipeline.

## What This Changes

The PR changes Slack thread-context resolution and tests to set `MessageThreadId` for DM assistant self-thread root messages outside `replyToMode=all`, and also adds `context_window_exceeded` matching to an agents overflow-compaction test harness.

## Best Possible Solution

Pass the normalized direct-message classification, or an equivalent normalized boolean, into `resolveSlackThreadContext`; preserve `MessageThreadId` only for assistant-DM self-thread roots; keep non-DM self-thread roots unset; add focused coverage for raw `im`, D-prefix wrong or missing `channel_type`, non-DM roots, and concurrent assistant-DM roots; split or drop the unrelated agents harness change.

## Reproduction Assessment

Yes. A source-level reproduction is direct: call the resolver through the Slack routing path with a D-prefixed DM, `thread_ts === ts`, and `replyToMode` of `off`, `first`, or `batched`; current main returns no `messageThreadId`, and the PR still fails when raw `channel_type` is wrong or missing.

## Solution Assessment

No. The requested Slack behavior is valid, but the PR's raw `channel_type === "im"` check is narrower than the existing normalized DM contract; using the normalized routing state is the safer and more maintainable fix.
