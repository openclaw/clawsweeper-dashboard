## Summary

Keep open. The Slack bug is valid and current main still drops the explicit thread id for DM self-thread roots in non-all reply modes, but this branch keys the fix on raw Slack `channel_type` instead of the existing normalized DM state and is missing the required changelog entry.

## What This Changes

The PR updates Slack thread-context resolution/tests to keep `MessageThreadId` for Slack Agents & Assistants DM self-thread root messages and also adds an overflow-compaction test-harness matcher.

## Best Possible Solution

Pass normalized direct-message state into the Slack thread resolver, preserve `MessageThreadId` only for assistant-DM self-thread roots, cover raw `im` plus D-prefix wrong/missing `channel_type` cases, keep non-DM roots unchanged, and add the user-facing changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is to resolve a Slack D-prefixed DM with `thread_ts === ts` and `replyToMode` of `off`, `first`, or `batched`; current main returns no `messageThreadId`, and the PR branch still misses the same case when raw `channel_type` is wrong or missing.

## Solution Assessment

No. The PR targets the right Slack behavior, but the raw event-field guard is narrower than OpenClaw's normalized DM contract; threading the normalized `isDirectMessage` signal through the resolver is the safer fix.
