## Summary

Keep open for maintainer routing. Current main has the generic embedded-result classifier and wires it into several reply/command paths, but the isolated cron executor still calls `runWithModelFallback` without result classification, and the issue discussion narrowed the original outage into a fallback-semantics feature/product decision rather than a strict reproduced bug.

## What This Changes

Request that configured `model.fallbacks` advance after reasoning-only or empty-visible terminal embedded results instead of ending with an incomplete-turn error.

## Best Possible Solution

Define the product contract for fallback-safe empty/reasoning-only terminal results, then apply the existing embedded result classifier and exhaustion merge behavior only to the approved runtime paths with tests that preserve deliberate silence, committed delivery, and side-effect guards.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: current source shows the isolated cron executor lacks the result classifier that would turn fallback-safe incomplete embedded results into failed candidates, but I did not run a live cron/model repro and the reporter later corrected the original outage trigger as prompt-induced deliberate silence.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer policy is set; extending the existing embedded result classifier into the remaining approved runtime path is the likely narrow implementation, but maintainers need to decide which empty/reasoning-only cron outcomes are fallback-safe versus deliberate silence or side-effecting progress.
