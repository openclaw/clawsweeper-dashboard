## Summary

Keep open for maintainer routing. Current main has the generic embedded-result classifier and several reply paths wired into it, but isolated cron still calls `runWithModelFallback` without a result classifier, and the issue discussion narrowed the original outage into a fallback-semantics feature/product decision rather than a strict reproduced bug.

## What This Changes

The issue asks for configured model fallbacks to engage when an embedded agent attempt ends with reasoning-only or empty visible output instead of returning a visible incomplete-turn error.

## Best Possible Solution

Define the maintainer-approved fallback semantics once, then thread the existing outcome classifier into every model-fallback entry point where safe, with explicit tests for cron, normal replies, deliberate silence, and side-effectful terminal states.

## Reproduction Assessment

No high-confidence live reproduction of the original outage; the reporter identified that incident as an intentionally silent cron prompt. Source inspection still shows a current-main path where isolated cron calls `runWithModelFallback` without the embedded result classifier, so incomplete/reasoning-only result fallback remains path-specific.

## Solution Assessment

Unclear as an automatic fix: the existing classifier seam is the likely right mechanism, but maintainers need to define which empty, reasoning-only, and deliberate-silence terminal states should advance model fallback across cron and user-facing paths.
