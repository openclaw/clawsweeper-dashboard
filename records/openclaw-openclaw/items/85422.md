## Summary

Keep open for maintainer routing. Current main has the generic embedded-result classifier and top-level reply paths wired to classify empty/incomplete terminal results for model fallback, and the related thinking-only error gap was fixed by a merged PR, but source inspection still does not show the isolated cron outer fallback wrapper applying the same result classifier before accepting an incomplete embedded run result.

## What This Changes

The issue asks for configured `model.fallbacks` recovery when a primary model exhausts reasoning-only or empty-visible retries instead of leaving the agent without a useful visible response.

## Best Possible Solution

Define the maintainer-approved fallback semantics once, then make every model-fallback entry point use the same outcome classifier or deliberately document why a path stays strict.

## Reproduction Assessment

No high-confidence live repro of the original incident: the reporter later identified that outage as an intentionally silent cron prompt. Source inspection does show a plausible current-main remaining gap where isolated cron wraps embedded runs in `runWithModelFallback` without passing the result classifier.

## Solution Assessment

Unclear as a one-line fix. The best direction is to reuse the existing outcome-classification seam consistently, but maintainers need to decide which empty/reasoning-only terminal states should advance fallback versus remain deliberate silence or side-effect-preserving terminal results.
