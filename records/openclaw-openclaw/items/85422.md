## Summary

Keep open for maintainer routing. The reporter narrowed the original outage to an intentional empty-output cron prompt, but current main still has an unresolved policy/code boundary for whether isolated cron runs should classify exhausted reasoning-only or empty-visible embedded results into model fallback without breaking deliberate silence semantics.

## What This Changes

The issue requests that configured model.fallbacks advance to another provider/model when reasoning-only or empty-visible assistant turns exhaust local retries instead of ending with an incomplete-turn result.

## Best Possible Solution

Define the no-visible-output fallback contract across normal replies, cron runs, deliberate silence, and side-effectful turns; then add focused regressions and wire the cron classifier only for the approved cases.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection does show a current-main cron path where exhausted embedded results return as successful values to runWithModelFallback without the normal classifyResult hook, while the reporter's concrete outage was prompt-induced.

## Solution Assessment

Unclear. The obvious implementation is to share the existing embedded result classifier with isolated cron fallback, but maintainers first need to decide how intentional silence and reasoning-only failures should differ in cron and group contexts.
