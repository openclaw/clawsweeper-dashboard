## Summary

Keep open for maintainer routing. Current main has structured result classification and normal reply fallback coverage, and the adjacent thinking-only provider-error gap was fixed, but the isolated cron fallback wrapper still accepts embedded incomplete/reasoning-only results without passing the same classifier; the reporter also reframed the original outage as a feature/product semantics request after finding a prompt-side trigger.

## What This Changes

The issue asks for configured `model.fallbacks` to advance after reasoning-only or empty-visible embedded replies exhaust retry recovery instead of ending with an incomplete-turn result.

## Best Possible Solution

Define the fallback semantics once, then wire isolated cron through the same structured result classifier or document why that path deliberately stays strict, with regression coverage for retry exhaustion, deliberate silence, and side-effect guards.

## Reproduction Assessment

No high-confidence live reproduction of the original incident: the reporter later identified that outage as an intentionally silent cron prompt. Source inspection does show a current-main and latest-release path where isolated cron wraps embedded runs in `runWithModelFallback` without the result classifier needed to reject fallback-safe incomplete results.

## Solution Assessment

Unclear as a direct bug fix. Reusing the existing structured classifier is the cleanest implementation seam, but maintainers need to decide exactly which cron empty/reasoning-only terminal states should advance fallback instead of preserving deliberate silence or side-effect safety.
