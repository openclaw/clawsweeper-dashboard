## Summary

Keep open. Current main still exposes only source-channel message timestamps through the shared inbound context, agent-visible metadata, channel-turn/direct-DM helpers, and hook payloads; Telegram's `receivedAtMs` is local ingress debug state, and the newer #77392 Discord latency report adds related observability context without superseding this request.

## What This Changes

Add a Gateway receipt timestamp to inbound metadata so agents and hooks can distinguish channel delivery latency from OpenClaw processing latency.

## Best Possible Solution

Add a backward-compatible receipt timestamp stamped once at channel/Gateway ingress, preserve `Timestamp`/`timestamp` as the provider source time, thread the new value through inbound context, prompt metadata, hook payloads, direct-DM and channel-turn SDK contracts, and document buffered-path semantics.

## Reproduction Assessment

Not applicable as a feature request. Current-main source inspection gives high confidence that no shared agent-visible or hook-visible Gateway receipt timestamp exists today.

## Solution Assessment

Yes, the narrow maintainable direction is an additive typed receipt-time field rather than repurposing `Timestamp`. Broader per-stage timing for #77392 can build on this but should not block the receipt timestamp itself.

## Repair Work Prompt

Implement the Gateway receipt timestamp request from https://github.com/openclaw/openclaw/issues/53678 as a focused additive feature. Stamp a receipt time once when an inbound message enters the Gateway/channel handler, preserve existing `Timestamp`/`timestamp` as the provider/source message time, add a repo-consistent typed field such as `ReceivedAt?: number` to the shared inbound context and render it as `receivedAt` in agent-visible metadata. Thread it through channel-turn params/types, direct-DM helper, hook canonical/plugin/internal event payloads, Discord and Telegram ingress paths, and tests. For debounced or buffered batches, choose and document one consistent rule for first/last/flush receipt time. Update the relevant SDK/timezone docs and add a user-facing changelog entry. Do not add a config option, rename `Timestamp`, or make core depend on bundled plugin ids. Treat https://github.com/openclaw/openclaw/issues/77392 as related observability context only; do not attempt to solve the full Discord latency report unless the receipt timestamp exposes a small direct bug.
