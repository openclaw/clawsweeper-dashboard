## Summary

Keep open: current main and the latest release still expose only the source-channel timestamp in agent-facing inbound metadata, while existing `receivedAt` fields are receive/ack internals. The request remains valid, but field naming, timing semantics, and trusted-versus-user metadata placement need maintainer API/product review.

## What This Changes

Add an agent-facing Gateway receive timestamp to inbound metadata so latency can be separated into channel delivery time and OpenClaw processing time.

## Best Possible Solution

Define one typed shared inbound receive timestamp after maintainers choose the field name, timing source, and prompt placement, then wire it through the generic channel/SDK context instead of per-channel extras.

## Reproduction Assessment

Not applicable: this is a feature request rather than a broken existing contract. Source inspection of current main verifies the requested receive-time field is absent from agent-facing inbound metadata.

## Solution Assessment

Unclear until maintainers choose the API contract: the requested direction is reasonable, but the field name, receive-time source, and trusted-versus-user prompt placement are product/API decisions. The maintainable path is a typed shared context field, not channel-specific metadata extras.
