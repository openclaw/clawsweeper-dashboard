## Summary

Keep open. Current main still exposes only the source-channel timestamp across inbound context, prompt metadata, channel-turn, direct-DM, and hook payloads; the linked Discord latency report adds useful observability context but does not supersede this narrower metadata feature.

## What This Changes

Request to add a Gateway receipt timestamp to inbound metadata so agents and hooks can separate source-channel delivery latency from OpenClaw processing latency.

## Best Possible Solution

Add a backward-compatible Gateway receipt timestamp stamped at ingress, thread it through shared inbound context, prompt metadata, channel-turn/direct-DM SDK helpers, and hook payloads, and document how buffered paths choose the receipt time.

## Reproduction Assessment

Not applicable as a feature request. Current-main source inspection is high confidence: the shared inbound surfaces only carry the source-channel Timestamp/timestamp and targeted searches found no shared receipt-time field.

## Solution Assessment

Yes. An additive typed receipt timestamp is the narrowest maintainable path because it preserves the existing source timestamp while enabling latency calculation for agents, hooks, and diagnostics.
