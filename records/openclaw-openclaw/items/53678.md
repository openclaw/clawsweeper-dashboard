## Summary

Keep open: current main and the latest release still expose only source-channel message time in the shared agent-visible inbound metadata path. The related Discord latency report at https://github.com/openclaw/openclaw/issues/77392 was later closed as resolved by upgrade/config, but it reinforces the observability gap rather than superseding this feature request.

## What This Changes

Add a Gateway receipt timestamp to agent-visible inbound metadata so latency can be split between channel delivery time and processing time.

## Best Possible Solution

Add an optional receipt-time field stamped once at channel/Gateway ingress, preserve `Timestamp` as source-channel time, and thread the new value through finalized context, prompt metadata, hooks, SDK contracts, docs, and focused tests after maintainers settle naming and buffering semantics.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main still lacks an agent-visible Gateway receipt timestamp in shared inbound metadata.

## Solution Assessment

Yes, directionally: an additive optional receipt-time field is safer than repurposing the existing source-channel `Timestamp`. The exact public name and buffered-message semantics need maintainer decision before implementation.

## Risks / Open Questions

- Maintainers still need to choose the public field name and whether the timestamp means first Gateway ingress, final buffered/coalesced receipt, or agent-dispatch start.
- Adding a volatile receipt timestamp to the wrong prompt block could reduce prompt-cache stability, so metadata placement needs an explicit decision.
- The existing `MessageReceiveContext.receivedAt` name is close to the requested field but currently belongs to receive acknowledgement lifecycle, so implementation should avoid conflating ack policy with agent-visible latency metadata.
