## Summary

Keep open. Current main has partial receive-time groundwork for ack/debug paths, but it still does not expose a shared agent-visible Gateway receipt timestamp in inbound metadata across the common context, hooks, direct-DM, or channel-turn SDK surfaces.

## What This Changes

The issue requests an additive Gateway receipt timestamp in inbound metadata so agents can separate source-channel delivery delay from OpenClaw processing latency.

## Best Possible Solution

Add a backward-compatible receipt timestamp stamped once at channel or Gateway ingress, preserve `Timestamp` as the source-channel message time, thread the new value through finalized inbound context, prompt metadata, hook payloads, direct-DM and channel-turn SDK contracts, and document buffered-message semantics.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main still lacks an agent-visible Gateway receipt timestamp in the shared inbound metadata path.

## Solution Assessment

Yes. The narrow maintainable solution is an additive optional receipt-time field rather than repurposing the existing source-channel `Timestamp`; existing receive-context and Telegram debug timing are useful groundwork but do not satisfy the request.
