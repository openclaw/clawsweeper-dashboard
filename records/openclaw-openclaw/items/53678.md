## Summary

Keep open: current main and the latest release still expose only the source-channel timestamp in agent-visible inbound metadata. The requested Gateway receipt timestamp is a real observability feature, but it crosses prompt metadata, hooks, channel turn contracts, and SDK/docs surfaces, so it needs maintainer API/product decisions before implementation.

## What This Changes

Add an agent-visible inbound metadata field recording when the Gateway received a channel message so latency can be split between channel delivery and processing.

## Best Possible Solution

Add an optional Gateway receipt timestamp stamped once at channel/Gateway ingress and thread it through finalized context, prompt conversation metadata, hooks, SDK turn contracts, docs, and focused tests without repurposing `Timestamp`.

## Reproduction Assessment

Not applicable as a feature request. Source and release inspection give high confidence that current main still lacks an agent-visible Gateway receipt timestamp in shared inbound metadata.

## Solution Assessment

Yes directionally: an additive optional receipt-time field is safer than redefining the existing source-channel `Timestamp`. The exact name, semantics, and prompt/hook placement need maintainer decision before a fix PR.
