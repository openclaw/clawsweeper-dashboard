## Summary

Keep open. Current main still lacks a shared agent-visible Gateway receipt timestamp; the only broadly threaded time field is the source-channel timestamp, while Telegram's internal `receivedAtMs` remains local buffering/debug state and does not satisfy the cross-channel metadata request.

## What This Changes

Request an additive inbound metadata field that records when the Gateway received a message so agents can separate processing latency from source-channel delivery latency.

## Best Possible Solution

Add a backward-compatible receipt timestamp stamped once at Gateway/channel ingress, preserve `Timestamp`/`timestamp` as the source-channel message time, thread the new value through finalized inbound context, prompt/user metadata, hook payloads, direct-DM and channel-turn SDK contracts, and document buffered-path semantics.

## Reproduction Assessment

Not applicable as a feature request, but current-main verification is high confidence: code inspection and targeted searches show no shared `receivedAt` or `gatewayReceivedAt` field in the inbound context, metadata renderer, hook contracts, or channel turn SDK surface.

## Solution Assessment

Yes, the narrowest maintainable solution is an additive typed field on the existing inbound context and SDK/hook payloads rather than repurposing `Timestamp`; the only design choice to settle in implementation is the buffered-message timestamp rule.
