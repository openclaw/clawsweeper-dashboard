## Summary

Keep open: current main and the latest release still expose only the source-channel `Timestamp` in agent-facing inbound metadata, while existing `receivedAt` fields are internal receive/ack state. The related Discord latency report adds useful motivation, but it was resolved separately and did not supersede this shared metadata/API request.

## What This Changes

The issue requests a gateway-side receipt timestamp in inbound metadata so agents can distinguish channel delivery time from OpenClaw processing time.

## Best Possible Solution

Define one shared receive-time contract, carry it through buffered and durable channel paths, and expose it consistently in the agreed inbound metadata or hook/runtime surface with focused tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks an agent- or hook-visible gateway receipt timestamp.

## Solution Assessment

Unclear until maintainers choose the public contract. A generic shared inbound receive timestamp is maintainable, but the exact timing point, name, and visibility need API/product review first.
