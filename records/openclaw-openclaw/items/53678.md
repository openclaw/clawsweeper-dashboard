## Summary

Keep open: current main and the latest release still expose the source-channel `Timestamp` in the agent-facing inbound context, but not a canonical Gateway receipt timestamp. There are channel/SDK receive-time primitives already, so the request is directionally aligned, but promoting that into public inbound metadata needs a maintainer API/product decision.

## What This Changes

The issue requests an agent-visible Gateway receipt timestamp in inbound metadata so processing latency can be separated from source-channel delivery latency.

## Best Possible Solution

Define one canonical received-time contract in the shared inbound context and hook/runtime metadata, preserve `Timestamp` as source-channel time, carry the value through buffered and durable channel paths, and update SDK docs plus focused regression tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main lacks the requested agent-visible receive timestamp, while only ack/durable internals currently carry `receivedAt`.

## Solution Assessment

Unclear until maintainers choose the public contract. A generic shared inbound metadata field is the maintainable direction, but field name, timestamp semantics, and prompt/runtime visibility should be decided before implementation.
