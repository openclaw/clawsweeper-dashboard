## Summary

Keep open. Current main still lacks a shared agent-visible Gateway receipt timestamp; the only broadly threaded inbound time remains the source-channel timestamp, and the newer Discord latency report makes this observability gap more relevant rather than superseded.

## What This Changes

The issue requests a Gateway-stamped receipt timestamp in inbound metadata so agents and diagnostics can separate channel delivery latency from processing latency.

## Best Possible Solution

Add a backward-compatible receipt timestamp stamped once at channel/Gateway ingress, preserve Timestamp/timestamp as the source-channel message time, thread the new value through finalized inbound context, prompt metadata, hook payloads, direct-DM and channel-turn SDK contracts, and document buffered-path semantics.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main gives high-confidence evidence that the shared inbound context, metadata renderer, hook payloads, direct-DM helper, and channel-turn SDK surface still lack a Gateway receipt timestamp.

## Solution Assessment

Yes. The narrow maintainable path is an additive typed receipt-time field rather than repurposing Timestamp, with the buffered-message timestamp rule documented in the same change.
