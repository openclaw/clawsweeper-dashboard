## Summary

Keep open: current main and the latest release still expose only the source message timestamp in generic inbound/agent-visible metadata. Internal receive paths do carry `receivedAt`, and related work fixed timestamp seconds precision, but the requested Gateway-received timing contract remains a cross-channel product/API decision.

## What This Changes

The issue requests adding a Gateway receipt timestamp, such as `receivedAt` or `gatewayReceivedAt`, to inbound metadata so agents and diagnostics can separate channel delivery delay from processing latency.

## Best Possible Solution

Define one additive generic ingress timing contract, reuse existing internal `receivedAt` where available, and expose it consistently through inbound context, agent-visible metadata, docs, and focused tests.

## Reproduction Assessment

No high-confidence runtime reproduction is applicable because this is a feature request, not broken existing behavior. Source inspection does confirm that current main lacks the requested generic agent-visible Gateway receipt timestamp.

## Solution Assessment

Unclear pending maintainer API direction. The requested field is plausible, but the best maintainable solution is a generic additive timing contract rather than a per-channel or prompt-only patch.
