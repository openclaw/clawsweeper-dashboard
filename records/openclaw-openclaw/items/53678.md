## Summary

Keep open: current main and the latest release have internal `receivedAt` plumbing in durable receive/queue paths, but the generic inbound context and agent-visible prompt metadata still expose only the source message timestamp. This remains a valid observability feature that needs a maintainer decision on the public timing contract.

## What This Changes

The issue requests a gateway-received timestamp in inbound metadata so agents can distinguish channel delivery delay from processing latency.

## Best Possible Solution

Define one additive generic ingress timing contract and expose it consistently through the inbound channel context and agent-visible metadata, reusing internal `receivedAt` where available instead of adding per-channel ad hoc fields.

## Reproduction Assessment

No high-confidence runtime reproduction is applicable because this is a feature request, not broken existing behavior. Source inspection gives medium confidence that current main and the latest release have internal receipt timestamps but still lack the requested generic agent-visible inbound metadata field.

## Solution Assessment

Unclear pending maintainer API direction. The narrowest maintainable path appears to be a generic additive inbound timing field, not a Discord-, Telegram-, or long-poll-specific metadata patch.
