## Summary

Keep this open. Current main still has no shared, agent-visible Gateway receipt timestamp: the shared context, prompt metadata renderer, channel turn builder, direct-DM helper, hook events, and SDK docs expose the source-channel `timestamp` only, while Telegram's `receivedAtMs` remains an internal ingress/debug value.

## What This Changes

Add a Gateway receipt timestamp, such as `receivedAt` or `gatewayReceivedAt`, to inbound metadata so agents can distinguish channel delivery latency from OpenClaw processing latency.

## Best Possible Solution

Add a backward-compatible receipt timestamp stamped once at channel/Gateway ingress, preserve `Timestamp`/`timestamp` as the source-channel message time, expose the new value through agent-visible inbound metadata and typed hook/SDK payloads, and document buffered-path semantics.

## Reproduction Assessment

Not applicable as a feature request, but the absence is reproducible by source inspection: pass a channel turn through `buildChannelTurnContext` or the direct-DM helper and the resulting context can only carry `Timestamp`, which renders as `timestamp` in inbound metadata.

## Solution Assessment

Yes, an additive shared metadata field is the right direction. It should not replace the existing source timestamp, and the implementation should thread the new field through the shared turn, hook, direct-DM, prompt metadata, tests, and docs surfaces rather than adding a channel-specific workaround.
