## Summary

Keep open: current main and latest release still do not expose a Gateway-received timestamp in generic inbound or agent-visible metadata. The request is useful for latency attribution, but it defines a cross-channel timing/API contract and needs maintainer product direction before implementation.

## What This Changes

The issue requests a Gateway-received timestamp in inbound metadata so agents can separate channel delivery latency from processing latency.

## Best Possible Solution

Define one additive generic ingress timing contract, reuse existing internal `receivedAt` where available, and expose it consistently through inbound context, agent-visible metadata, docs, and focused tests.

## Reproduction Assessment

Yes, by source inspection: current main lacks a first-class Gateway-received field in generic inbound context or conversation metadata, even though some channel internals track receive time. A live channel run is not required to prove this feature is absent.

## Solution Assessment

Unclear pending maintainer API direction. The requested field is plausible, but the maintainable solution should be a generic additive timing contract rather than a per-channel or prompt-only patch.
