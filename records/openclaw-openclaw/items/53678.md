## Summary

Keep open: current main and the latest shipped release still expose only the source-channel message timestamp in shared inbound metadata, while existing `receivedAt` groundwork is limited to receive-ack or Telegram debug paths and is not agent-visible across the common inbound surfaces.

## What This Changes

Request adding a Gateway receipt timestamp to inbound metadata so agents can separate source-channel delivery delay from OpenClaw processing latency.

## Best Possible Solution

Add an optional receipt-time field stamped once at channel/Gateway ingress, preserve `Timestamp` as source-channel time, and thread the new value through finalized context, prompt metadata, hooks, direct-DM, channel-turn SDK contracts, docs, and focused tests.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main and `v2026.5.18` still lack an agent-visible Gateway receipt timestamp in the shared inbound metadata path.

## Solution Assessment

Yes, directionally: an additive optional receipt-time field is safer than repurposing the existing source-channel `Timestamp`. Maintainers still need to settle the public field name and buffered-message semantics before implementation.
