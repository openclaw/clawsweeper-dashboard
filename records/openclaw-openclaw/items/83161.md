## Summary

Keep open: the Telegram ownership boundary is the right direction, but this PR returns from the new dedupe branch before the normal preview-finalized hook and delivery-mirror transcript path runs, so a visible Telegram reply can still miss session/delivery bookkeeping.

## What This Changes

The PR reverts generic core preview-streamed final suppression and adds Telegram-local preview-final dedupe helpers plus dispatch regression tests.

## Best Possible Solution

Keep dedupe in the Telegram channel layer, but route deduped preview finals through the same preview-finalized hook and delivery-mirror transcript behavior as normal in-place finalization.

## Reproduction Assessment

Yes by source inspection and linked live report context, though not by a local live run. Current main can suppress every matching final in core while Telegram cleanup still sees the preview lane as unfinalized and clears it.

## Solution Assessment

No, not quite. Moving the dedupe into Telegram is the maintainable boundary, but the proposed early return must preserve the existing preview-finalized hook and transcript-mirror side effects before merge.
