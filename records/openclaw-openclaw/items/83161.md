## Summary

Keep open: this remains a useful candidate for the Telegram preview-disappears regression, but the branch is not merge-ready because it can suppress the durable final reply before the preview stream has actually flushed the matching text to Telegram.

## What This Changes

The PR moves preview-streamed final deduplication into the Telegram dispatcher and adds Telegram tests for text-only finals, per-block finals, inline buttons, transcript mirroring, preview-message-id fallback, and error finals.

## Best Possible Solution

Keep the channel-layer ownership, but route preview-streamed final dedupe through the lane finalization contract so the stream is stopped or flushed and confirmed before suppressing durable final delivery.

## Reproduction Assessment

Yes at source/proof level: the PR branch uses pending `lastPartialText` and an existing preview message id as sufficient delivery proof, while the draft stream tracks actual delivered text separately. I did not run a fresh live Telegram repro in this read-only review.

## Solution Assessment

No, not as patched. The Telegram channel layer is the right boundary, but the PR bypasses the stream finalization contract that protects against stale or unflushed previews.
