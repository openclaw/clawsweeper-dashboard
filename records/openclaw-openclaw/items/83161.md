## Summary

Keep open: this is a useful candidate fix for the Telegram preview-disappears regression, but it is not merge-ready because the new suppression path can mark a final as delivered before the stream is stopped/flushed and delivered text is verified; GitHub also reports the branch as dirty.

## What This Changes

The PR adds Telegram preview-dedup helpers/tests and changes the Telegram dispatcher to suppress text-only final payloads that match partial-preview text while marking the answer lane delivered/finalized.

## Best Possible Solution

Keep the dedupe at the Telegram channel boundary, but route suppression through the existing stream finalization contract: stop/flush, verify delivered preview text or send the final, then mark delivered and mirror on a clean rebased branch.

## Reproduction Assessment

Yes for the PR defect from source inspection: the added dedupe branch uses lastPartialText and a numeric preview message id, then marks delivery before the existing stop/flush and delivered-text verification path. The original Telegram regression also has live log proof in the PR body and linked issue, but I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No, not yet. Moving dedupe into the Telegram channel layer is the right owner boundary, but this implementation needs to honor the existing stream finalization contract before suppressing the durable final.
