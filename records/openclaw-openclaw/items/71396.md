## Summary

Keep this PR open. Current main still has the Feishu mention-cascade bug, but the latest branch removes the open-ID-backed conversion path for explicit card/post/streaming mentions, and maintainer feedback already requested a Feishu-owner follow-up before prepare.

## What This Changes

The PR removes inbound Feishu `mentionTargets` from outbound reply dispatch, changes mentioned users to sanitized prompt context, adds regression tests for non-propagation, and updates the changelog.

## Best Possible Solution

Land a Feishu-owned revision that disables unconditional inherited mention propagation while preserving intentional mentions through allowlisted stable `{openId, name}` context and safe text/card/streaming conversion, with focused regression tests and the changelog entry.

## Reproduction Assessment

Yes, by source inspection. Current main extracts `ctx.mentionTargets`, passes them into both reply-dispatcher call sites, and applies them to first text/card/fallback chunks and streaming-card close; I did not run a live Feishu test.

## Solution Assessment

No. Removing unconditional propagation is the right direction, but the current branch removes the existing open-ID conversion path before adding the explicit-only replacement needed for card/post/streaming flows.

## Review Findings

  - body: After dispatcher-side mention injection is removed, this hint only gives the model sanitized display names. Post/rich-text messages can render mentions as `@name` while keeping open IDs internal, so intentional mention-forward replies cannot reliably notify the intended Feishu user when names collide or change.
  - confidence: 0.86
- **[P2] Keep explicit card mention conversion:** `extensions/feishu/src/reply-dispatcher.ts:595-604`

Feishu card mentions need the open-ID-backed `<at id=...></at>` form produced by `buildMentionedCardContent`, so intentional mention replies can degrade to plain text unless a safe explicit rewrite path replaces the removed auto-prefixing.
  - confidence: 0.84
