## Summary

Keep open. Current main and the latest release still expose `messages_send` as a reply-only MCP tool, but allowing MCP clients to create user-role turns is a security-sensitive product decision rather than a narrow bug fix.

## What This Changes

The issue requests an optional `role` parameter for the MCP `messages_send` tool so external MCP clients can create user-role turns instead of only assistant-role replies.

## Best Possible Solution

Keep `messages_send` reply-only and design an explicit, authorized MCP user-turn API that reuses `sessions_send`-style provenance or another maintainer-approved session boundary.

## Reproduction Assessment

Yes. Source inspection is enough: current main and `v2026.6.9` expose no `role` input on `messages_send`, route through outbound Gateway `send`, and the agent loop rejects assistant-tail continuation; no live MCP/Gateway setup was run in this read-only pass.

## Solution Assessment

No, not as the literal role passthrough sketched in the issue. The maintainable fix is an explicit user-turn authority boundary with provenance and authorization, not reusing a reply-delivery tool as a hidden injection path.
