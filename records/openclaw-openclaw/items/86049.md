## Label Justifications

- `P2`: The limitation blocks a real cross-agent MCP workflow, but it is a bounded enhancement and not an emergency regression in documented behavior.
- `impact:session-state`: The request centers on transcript role assignment and whether a message enters the target session as a user turn that wakes the agent loop.
- `impact:security`: Letting external MCP clients choose user role changes who can inject agent-triggering turns and needs authorization/scope review.

## Summary

Keep open: current main and the latest release still expose `messages_send` as a text-only channel reply path with no role selection, but allowing MCP clients to inject user-role turns is a new security-sensitive MCP/Gateway contract rather than a strict bug fix.

## What This Changes

Request to add an optional role parameter to `openclaw mcp serve` `messages_send` so external MCP clients can send user-role messages that wake a target agent while preserving the assistant default.

## Best Possible Solution

Define a maintainer-approved MCP/Gateway contract for agent-triggering sends, preferably reusing `sessions.send`/`chat.send` semantics and keeping `messages_send` as a channel reply path unless an explicit safe user-injection mode is approved.

## Reproduction Assessment

Yes, at source level: current `messages_send` only accepts `session_key` and `text`, and the bridge/Gateway path has no role field. I did not run the live Hermes/Gateway scenario.

## Solution Assessment

Unclear. The optional role parameter is plausible, but the proposed bridge-only direction is incomplete and security-sensitive; maintainers should decide whether MCP should expose `sessions.send`-style user turns instead.
