## Summary

Keep open. Current main and latest release still expose `messages_send` as a reply-only MCP tool, while making it create user-role turns would change the MCP trust boundary and needs maintainer product/security approval.

## What This Changes

The issue requests an optional role parameter for the MCP `messages_send` tool so external MCP clients can send user-role turns instead of only assistant-role messages.

## Best Possible Solution

Keep this open for a maintainer decision on an explicit authorized MCP user-turn API, likely after or alongside the broader `chat.inject` user-role contract, while preserving reply-only `messages_send` semantics by default.

## Reproduction Assessment

Yes, by source inspection: current main and latest release expose no `role` input on `messages_send`, the bridge forwards through outbound Gateway `send`, and agent continuation rejects assistant-tail transcripts. I did not run a live MCP/Gateway setup in this read-only cleanup pass.

## Solution Assessment

No, not as the literal patch sketch. The best fix needs a maintainer-approved user-turn authority boundary, because `chat.inject` is admin-only and Gateway `send` is a reply-delivery schema with no role field.
