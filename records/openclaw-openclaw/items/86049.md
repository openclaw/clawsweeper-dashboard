## Summary

Keep open. Current main and the latest release still expose `messages_send` as a reply-only MCP tool, and the requested user-role injection remains a product/security contract rather than an already-implemented cleanup close.

## What This Changes

The issue requests an optional `role` parameter for MCP `messages_send` so external clients can send user-role messages that wake the target agent.

## Best Possible Solution

Define a documented MCP user-turn injection contract that routes through the session/chat execution path with explicit authorization, rather than treating outbound reply delivery as a free-form transcript-role write.

## Reproduction Assessment

Yes, at source level: current main exposes no `role` argument on `messages_send`, forwards through outbound Gateway `send`, and agent continuation rejects assistant-tail transcripts. I did not run a live external MCP client in this read-only review.

## Solution Assessment

No for the issue's literal patch sketch as a complete solution. A maintainer-approved session/chat user-turn API or dedicated MCP tool looks safer than adding a role field to outbound Gateway `send`.
