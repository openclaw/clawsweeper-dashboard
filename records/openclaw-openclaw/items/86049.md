## Label Justifications

- `P2`: This is a concrete MCP bridge feature gap blocking a reported cross-agent workflow, but it is not an urgent regression in existing channel messaging.
- `impact:security`: The requested role parameter would let an external MCP client wake an agent loop, changing an authorization-sensitive boundary.
- `impact:session-state`: The issue is about transcript role semantics and whether a session receives a user turn that advances agent state.

## Summary

Keep open. Current main and v2026.5.22 still expose `messages_send` as a reply-only MCP bridge tool, while the requested user-role path is a new external agent-turn injection capability that needs maintainer product and security review.

## What This Changes

Add an optional `role` parameter to MCP `messages_send` so external clients can request a `user` turn for cross-agent invocation while preserving the current assistant-role default.

## Best Possible Solution

Define a documented MCP user-turn injection path that keeps `messages_send` reply compatibility, routes user turns through the session/chat execution contract, and includes security review plus focused bridge, protocol, docs, and regression tests.

## Reproduction Assessment

Yes, at source level: current main exposes no `role` argument on `messages_send`, forwards through outbound Gateway `send`, and mirrors that path as assistant-role transcript output. I did not run a live Gateway plus Hermes MCP client in this read-only review.

## Solution Assessment

Unclear. The user problem is real, but adding `role` directly to Gateway `send` is not the cleanest proven design because it conflates outbound delivery with session user-turn execution.
