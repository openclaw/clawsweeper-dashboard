## Summary

Keep open. Current main and the latest release still expose `messages_send` as a route-reply MCP tool, while the requested user-role path is a new external agent-turn injection capability that needs maintainer product and security review.

## What This Changes

Add an optional role parameter to the MCP `messages_send` bridge so external clients can send user-role messages that wake the target agent loop.

## Best Possible Solution

Define an explicit, documented MCP user-turn injection contract that preserves `messages_send` reply compatibility, routes user-role turns through the session/chat execution path, and includes security review plus focused bridge, protocol, docs, and tests.

## Reproduction Assessment

Yes, at source level: current main exposes no `role` argument on `messages_send`, forwards through outbound Gateway `send`, and mirrors that path as assistant-role transcript output. I did not run a live Gateway plus Hermes MCP client in this read-only review.

## Solution Assessment

Unclear. The requested capability is real, but a raw `role` field on outbound Gateway `send` is not the cleanest proven design because it mixes channel delivery with session user-turn execution.
