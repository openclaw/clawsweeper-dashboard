## Label Justifications

- `P2`: This is a concrete MCP bridge feature gap that blocks a reported cross-agent workflow but is not an urgent regression in existing user messaging.
- `impact:security`: Allowing external MCP clients to wake an agent loop changes an authorization-sensitive tool and agent execution boundary.
- `impact:session-state`: The issue is about transcript role semantics and whether a session receives a user turn that advances agent state.

## Summary

Keep open. Current main and the latest release still expose `messages_send` as an outbound reply-only MCP tool, while the requested user-role path would add a new external agent-turn injection capability that needs maintainer product/security review.

## What This Changes

The issue requests an optional `role` parameter for the MCP `messages_send` tool so external MCP clients can send user-role messages that wake an agent loop while preserving assistant-role defaults.

## Best Possible Solution

Define an explicit, documented MCP user-turn injection path that preserves `messages_send` reply compatibility, routes user-role turns through the session/chat execution contract, and includes security review plus focused bridge/protocol/docs/tests.

## Reproduction Assessment

Yes, at source level: current main exposes no `role` argument on `messages_send`, forwards through outbound Gateway `send`, and mirrors that path as assistant-role transcript output. I did not run a live Gateway plus Hermes MCP client in this read-only review.

## Solution Assessment

Unclear. The user problem is real, but adding `role` directly to Gateway `send` is not the cleanest proven design because it conflates channel delivery with session user-turn execution and needs a trust-boundary decision.
