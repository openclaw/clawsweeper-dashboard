## Label Justifications

- `P2`: This is a concrete MCP bridge feature gap that blocks a reported cross-agent workflow but is not an urgent regression in existing messaging.
- `impact:security`: Allowing an external MCP client to create user turns changes the authorization boundary for agent execution.
- `impact:session-state`: The request changes transcript role semantics and whether a session receives a user turn that advances agent state.

## Summary

Keep open. Current main and v2026.6.6 still expose `messages_send` as a reply-only MCP tool, and the requested user-role injection path remains a feature/security contract that needs maintainer direction before implementation.

## What This Changes

Request to add an optional `role` parameter to the `openclaw mcp serve` `messages_send` tool so external MCP clients can submit user-role turns that wake an agent while preserving assistant-role replies by default.

## Best Possible Solution

Define an explicit, documented MCP user-turn injection path that preserves `messages_send` reply semantics, routes user-role turns through the session/chat execution contract or a dedicated tool, and includes security review plus focused tests and docs.

## Reproduction Assessment

Yes, at source level: current main exposes no `role` argument on `messages_send`, forwards through outbound Gateway `send`, and agent continuation rejects assistant-tail transcripts. I did not run a live external MCP client.

## Solution Assessment

Unclear. The user problem is valid, but adding `role` directly to Gateway `send` conflates outbound channel delivery with user-turn execution; a dedicated MCP tool or sessions/chat-backed path with explicit authorization looks safer.
