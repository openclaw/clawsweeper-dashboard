## Summary

Keep open. Current main and the latest release still expose `messages_send` as a reply-only MCP bridge path, and the requested user-role agent wakeup is a real feature/security contract that needs maintainer design review before implementation.

## What This Changes

Request adds an optional role parameter to MCP `messages_send` so external clients can send user-role turns that wake an OpenClaw agent while preserving the assistant-reply default.

## Best Possible Solution

Define a documented MCP user-turn injection path that preserves `messages_send` reply compatibility, routes user-role turns through the session/chat execution contract, and includes focused bridge, protocol, docs, tests, and security review.

## Reproduction Assessment

Yes, at source level: current main exposes no `role` argument on `messages_send`, forwards through outbound Gateway `send`, and mirrors delivered output as assistant-role transcript entries. I did not run a live Gateway plus Hermes MCP client in this read-only review.

## Solution Assessment

Unclear. The user problem is real, but adding `role` directly to Gateway `send` is not the cleanest proven design because `send` is a strict outbound delivery contract while user-role turns belong to session/chat execution.
