## Label Justifications

- `impact:session-state`: The requested behavior would change how externally injected MCP messages enter session transcripts and wake an agent loop.

## Summary

Keep open. Current main and v2026.6.9 still expose `messages_send` as a reply-only MCP bridge, and turning it into a user-role agent wakeup path needs maintainer product/security approval rather than a bridge-only role parameter.

## What This Changes

Add an optional role parameter to the MCP `messages_send` tool so external MCP clients can send user-role turns that wake the target agent.

## Best Possible Solution

Define a maintainer-approved MCP user-turn API or scope that reuses the existing session/A2A authorization model, while keeping `messages_send` reply-only by default.

## Reproduction Assessment

Yes, by source inspection: current main and v2026.6.9 expose only `session_key` and `text` on `messages_send`, Gateway `send` has no role field, and the agent loop rejects assistant-tail continuation. I did not run a live MCP/Gateway repro in this read-only pass.

## Solution Assessment

No for the literal patch sketch. A safe fix needs an explicit user-turn authority boundary, likely reusing `sessions_send` semantics or a new scoped MCP tool rather than forwarding `role` through outbound `send`.
