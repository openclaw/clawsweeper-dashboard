## Summary

Keep this open. Current main still lacks the requested built-in shared atomic counter, `openclaw sessions kv incr` CLI/RPC surface, named `sessions_spawn` mutex/exclusive option, or core agent-facing lock primitive for cron-spawned sub-agents. A ClawHub plugin route is feasible with the current plugin SDK, but choosing that instead of a generic core coordination primitive is still a maintainer product/API decision.

## What This Changes

Request to add a shared atomic counter or equivalent mutex/lock primitive for concurrent cron-spawned sub-agents so daily rate-limit state cannot race.

## Best Possible Solution

Maintainers should choose the API boundary first. If this belongs in core, implement a small workspace/session-scoped atomic state primitive with CLI, RPC, and agent-tool coverage; if it belongs in ClawHub, document the supported plugin pattern using `api.registerTool`, workspace resolution, `json-store`, and `file-lock`, then close the core request once that direction is explicit.

## Reproduction Assessment

Yes for the missing-feature check: current-main source and docs have no `sessions kv incr`, shared counter method, or named `sessions_spawn` mutex. The actual race is plausibly described in the issue, but no runnable minimal concurrent cron fixture is included.

## Solution Assessment

Unclear. A core counter would directly match the proposed API, while a ClawHub plugin could solve the daily-limit use case with current SDK primitives; maintainers need to choose which boundary is intended before an automated fix PR is safe.
