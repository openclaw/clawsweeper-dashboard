## Summary

Keep open. Current main and v2026.5.18 still lack an agent-facing atomic counter, session KV increment, named `sessions_spawn` mutex, or workspace lock primitive; https://github.com/openclaw/openclaw/issues/54325 only shipped same-job cron overlap prevention, so this remains a distinct product/API decision.

## What This Changes

Request a core agent/session concurrency primitive so cron-spawned subagents can update shared daily counters without read-modify-write races.

## Best Possible Solution

Choose one durable core-owned concurrency primitive for agent/subagent callers, then land the typed protocol, CLI/tool surface, docs, and cross-process regression proof together.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/API request. Source inspection gives high confidence that the requested primitive is absent from current sessions CLI, Gateway methods/protocol, `sessions_spawn`, and subagent docs.

## Solution Assessment

Unclear. An atomic counter would address the daily-limit race directly, but maintainers need to decide whether core should expose session KV, a spawn mutex, a workspace lock helper, or a plugin-facing contract.
