## Summary

Keep open. Current main and the latest release still lack the requested agent-facing atomic counter, session KV increment, named subagent mutex, or workspace lock primitive; the related cron overlap guard only prevents a second run of the same cron job, not shared-state races between concurrent subagents. The fix needs a maintainer product/API decision before implementation.

## What This Changes

Request an agent/subagent-facing atomic shared-state primitive for cron-driven daily limits, such as session KV increment, a sessions_spawn mutex, or workspace advisory locks.

## Best Possible Solution

Choose one durable, core-owned concurrency primitive for agent/subagent callers, then land the typed protocol, CLI/tool surface, docs, and cross-process regression proof together.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/API request. Source inspection gives high confidence that the requested primitive is absent from current sessions CLI, Gateway methods/protocol, sessions_spawn, and subagent docs.

## Solution Assessment

Unclear. An atomic counter would address the daily-limit race directly, but maintainers need to decide whether core should expose session KV, a spawn mutex, a workspace lock helper, or a plugin-facing contract; the existing trusted-plugin keyed store is not an agent-facing increment primitive.
