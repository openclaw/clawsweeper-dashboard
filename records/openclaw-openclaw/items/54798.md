## Summary

Keep open. Current `main` still does not expose a shared atomic counter, `openclaw sessions kv incr` CLI/RPC surface, named `sessions_spawn` mutex, or agent-facing workspace lock primitive; the remaining question is product/API scope rather than stale cleanup.

## What This Changes

The issue requests an atomic workspace/session counter or equivalent mutex/lock primitive for concurrent cron-spawned sub-agents that share daily rate-limit state.

## Best Possible Solution

Maintainers should first choose the boundary; if this belongs in core, add one small atomic workspace/session state primitive with CLI, Gateway RPC, and agent-tool coverage, otherwise document an endorsed plugin/ClawHub pattern and the required extension API.

## Reproduction Assessment

No, not applicable as a bug reproduction; this is a feature/API request. The missing-feature check is high-confidence from current-main source inspection of sessions CLI/RPC, protocol schemas, `sessions_spawn`, and cron concurrency docs/tests.

## Solution Assessment

Unclear. A core atomic counter would directly solve the stated daily-limit race, but the repo vision pushes optional capability toward plugins unless a core extension API is missing, so the maintainable solution needs maintainer API-boundary direction first.
