- `impact:session-state`: The issue is about shared agent state counters drifting when multiple subagents perform non-atomic read-modify-write updates.

## Summary

Keep open: current main still lacks an agent-facing atomic counter, session KV increment, named `sessions_spawn` mutex, or workspace lock primitive for shared daily quota state. Related cron overlap and edit-tool locking work helps adjacent races, but it does not solve shared read-modify-write counters across concurrent sub-agents.

## What This Changes

Request a workspace/session-scoped atomic counter, named spawn mutex, or reliable workspace lock primitive so concurrent cron sub-agents cannot race daily quota state.

## Best Possible Solution

Choose one core-owned concurrency primitive for agent/sub-agent callers, then land the typed protocol, CLI/tool surface, docs, and cross-process regression proof together.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/API request. Source inspection gives high confidence that the requested primitive is absent from current sessions CLI, Gateway methods/protocol, `sessions_spawn`, and subagent docs.

## Solution Assessment

Unclear. An atomic counter would address the daily-limit race directly, but maintainers need to decide whether core should expose session KV, a spawn mutex, a workspace lock helper, or a plugin-facing contract.
