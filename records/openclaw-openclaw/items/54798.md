## Summary

Keep open. Current main still lacks the requested shared atomic counter, session KV increment, named `sessions_spawn` mutex, or agent-facing workspace lock primitive; the implemented cron overlap guard only prevents overlapping runs of the same cron job and does not serialize shared state updates across concurrent subagents.

## What This Changes

Requests a workspace or session-scoped atomic counter or lock primitive so concurrent cron-spawned subagents can enforce daily rate limits without racing shared JSON state.

## Best Possible Solution

Decide the durable API boundary first; if core owns it, add one small documented atomic state primitive reachable by subagents and covered by cross-process regression tests.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/API request. Source inspection gives high confidence that the requested primitive is absent from sessions CLI, Gateway methods, `sessions_spawn`, and subagent docs on current main.

## Solution Assessment

Unclear. An atomic counter would address the daily-limit race directly, but maintainers need to choose whether the right long-term surface is core session KV, a spawn mutex, a workspace lock helper, or a plugin-facing contract.
