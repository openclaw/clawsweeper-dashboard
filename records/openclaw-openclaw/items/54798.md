## Summary

Keep open. Current main still lacks a shared atomic counter, `sessions kv incr` CLI/RPC surface, named `sessions_spawn` mutex, or documented agent-facing workspace lock primitive; the closed cron overlap work helps adjacent scheduling but does not solve shared read-modify-write state across concurrent sub-agents.

## What This Changes

Request an OpenClaw-owned atomic shared counter or equivalent mutex/lock primitive for concurrent cron-spawned sub-agents enforcing daily limits.

## Best Possible Solution

Maintainers should decide the API boundary first; if core owns it, add one small documented atomic state primitive with CLI, Gateway/tool access, and cross-process regression coverage.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/API request. The missing primitive is high-confidence from current-main inspection of sessions CLI, Gateway methods, `sessions_spawn`, cron concurrency docs/code, and plugin-state docs.

## Solution Assessment

Unclear. An atomic counter directly addresses the daily-limit race, but maintainers need to choose whether the durable surface should be core sessions KV, spawn locking, workspace locks, or a plugin-owned pattern.
