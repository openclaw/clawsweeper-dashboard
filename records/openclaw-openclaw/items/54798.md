## Summary

Keep open. Current main still has the cron overlap guard from the related closed report, but it does not expose the requested atomic counter, session KV increment, named subagent mutex, or agent-facing workspace lock primitive for concurrent subagents that update shared state.

## What This Changes

Request a workspace/session-scoped atomic counter or equivalent mutex/lock primitive for concurrent cron-spawned subagents updating shared daily-limit state.

## Best Possible Solution

Pick one small core-owned atomic state contract for agent/subagent callers, document it, and cover it with cross-process or multi-run regression proof before implementing daily-limit workflows on top of it.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/API request. Source inspection gives high confidence that the requested primitive is absent from current sessions CLI, Gateway methods/protocol, sessions_spawn, and subagent docs.

## Solution Assessment

Unclear. An atomic counter would address the daily-limit race directly, but maintainers need to decide whether core should expose session KV, a spawn mutex, a workspace lock helper, or a plugin-facing contract.
