## Summary

Keep open: current main still lacks an agent-facing atomic counter, session KV increment, named `sessions_spawn` mutex, or workspace lock primitive for shared quota state; the existing cron overlap guard and plugin-only state/lock helpers are adjacent but do not solve concurrent sub-agent read-modify-write counters.

## What This Changes

The issue requests a core concurrency primitive for cron-spawned sub-agents, such as an atomic session counter, named spawn mutex, or reliable workspace lock.

## Best Possible Solution

Pick one core-owned concurrency primitive for agent/sub-agent callers, then land the typed protocol, CLI/tool surface, docs, and cross-process regression proof together.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature/API request. Source inspection gives high confidence that the requested primitive is absent from current sessions CLI, Gateway methods/protocol, `sessions_spawn`, and subagent docs.

## Solution Assessment

Unclear. An atomic counter would address the daily-limit race directly, but maintainers need to decide whether core should expose session KV, a spawn mutex, a workspace lock helper, or a plugin-facing contract.
