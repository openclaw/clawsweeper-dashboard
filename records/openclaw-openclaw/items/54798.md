## Summary

Keep open. Current main still lacks a core `sessions kv incr`/shared counter API, a named `sessions_spawn` mutex, or an agent-facing workspace lock primitive; the existing plugin SDK lock support makes an external plugin plausible, but the core-versus-ClawHub boundary is still a maintainer product decision.

## What This Changes

Request a workspace/session-scoped atomic counter or equivalent locking primitive for concurrent cron-spawned sub-agents to enforce daily limits.

## Best Possible Solution

Maintainers should choose the public boundary first; if this belongs in core, add a small atomic workspace/session state primitive with CLI, Gateway RPC, and agent-tool coverage, otherwise document the endorsed ClawHub/plugin pattern using the existing lock and tool-registration seams.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source review gives high confidence that current main lacks the requested built-in counter, mutex, or agent-facing lock surface.

## Solution Assessment

Unclear. A core counter directly matches the request, while the current plugin SDK can support an external lock-backed tool; maintainers need to decide which boundary OpenClaw wants before implementation.
