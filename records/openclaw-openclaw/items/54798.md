## Summary

Keep open. Current main still lacks the requested shared atomic counter, `sessions kv incr` CLI/RPC, named `sessions_spawn` mutex, or documented agent-facing workspace lock primitive; the remaining work is a product/API boundary decision, not cleanup.

## What This Changes

Request a workspace-scoped atomic counter or equivalent locking primitive for concurrent cron-spawned sub-agents sharing daily rate-limit state.

## Best Possible Solution

Maintainers should decide the API boundary first; if core owns it, add one small documented atomic state primitive with CLI, Gateway/tool access, and cross-process regression coverage.

## Reproduction Assessment

No, reproduction is not applicable as a strict bug path because this is a feature/API request. The missing primitive is high-confidence from current-main inspection of sessions CLI, Gateway methods, `sessions_spawn`, cron concurrency code, and docs.

## Solution Assessment

Unclear. An atomic counter directly addresses the daily-limit race, but the maintainable solution depends on whether maintainers want core sessions KV, spawn locking, workspace file locks, or an endorsed plugin/ClawHub pattern.
