Keep open. Current `main` and the latest release still lack the requested shared atomic counter, `sessions kv incr` command/RPC, named spawn mutex, or documented agent-facing workspace lock primitive; the remaining work is an API/product boundary decision.

## What This Changes

The issue requests an atomic workspace/session counter or equivalent concurrency primitive for cron sub-agents that share daily rate-limit state.

## Best Possible Solution

Maintainers should choose the API boundary first; if this belongs in core, add one small documented atomic workspace/session state primitive with CLI, Gateway RPC, agent-tool coverage, and regression tests.

## Solution Assessment

Unclear. An atomic counter would directly address the stated daily-limit race, but the maintainable solution depends on whether maintainers want this in core sessions, spawn locking, workspace file locks, or an endorsed plugin/ClawHub pattern.
