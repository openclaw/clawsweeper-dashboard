## Summary

Keep open. Current `main` still lacks the requested shared atomic counter, `openclaw sessions kv incr` CLI/RPC surface, named `sessions_spawn` mutex, or agent-facing workspace lock primitive; a plugin route is plausible, but choosing core API versus ClawHub/plugin scope is a maintainer product decision.

## What This Changes

Request a workspace/session-scoped atomic counter or equivalent mutex/lock primitive for cron-spawned sub-agents sharing daily rate-limit state.

## Best Possible Solution

Maintainers should first choose the API boundary; if this belongs in core, add a small atomic workspace/session state primitive with CLI, Gateway RPC, and agent-tool coverage, otherwise document and endorse the plugin pattern before closing the core request.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. The missing-feature check is high-confidence from current-main source and docs: no `sessions kv incr`, no shared counter method, and no `exclusive`/mutex parameter in `sessions_spawn`.

## Solution Assessment

Unclear. A core counter would directly match the requested API, while a ClawHub plugin can likely solve the daily-limit use case with current SDK primitives; maintainers need to choose the intended boundary before implementation.
