## Summary

## What This Changes

The issue reports that heartbeat `requests-in-flight` skips retry too aggressively and can block active Telegram, Feishu, and WebChat conversations.

## Best Possible Solution

Land or replace the linked focused heartbeat-wake fix with per-reason retry timing and live transport proof; do not close this issue until the PR merges or maintainers choose a broader heartbeat scheduling design.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.5 route `requests-in-flight` through the shared 1-second wake retry, and focused tests assert that timing. I did not run a live Telegram, Feishu, or WebChat reproduction in this read-only review.

## Solution Assessment

Yes for the current issue path: a reason-specific bounded retry delay in `heartbeat-wake.ts` is narrower than adding a new config surface. The linked PR is the right implementation candidate, but it still needs PR-side review and live behavior proof before this issue closes.
