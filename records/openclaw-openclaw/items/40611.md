## Summary

Keep open. Current main and the latest release still route `requests-in-flight` heartbeat skips through the shared 1 second wake retry, with no heartbeat-owned retry delay/backoff config, so the Telegram/WebChat/Feishu contention regression remains actionable.

## What This Changes

Requests a heartbeat-specific busy-session retry delay or backoff so skipped heartbeats do not immediately seize the session after active conversations.

## Best Possible Solution

Add a heartbeat-owned busy-session retry delay or backoff that preserves no-drift scheduling, exposes the intended config/docs surface if configurable, and leaves session lock narrowing and activity-based heartbeat semantics to their existing follow-ups.

## Reproduction Assessment

Yes for the scheduler path: current fake-timer tests and source show a `requests-in-flight` skip retries after about 1 second. The end-to-end Telegram/Feishu blocking was not live-reproduced in this read-only review.

## Solution Assessment

Yes. A narrow heartbeat busy-session retry policy is the best immediate fix because it addresses the regression without reintroducing full-interval drift; broader lock-scope and activity-heartbeat work belong in #13744 and #14051.
