## Summary

Keep open. Current main and the latest release still retry retryable heartbeat busy skips after the shared 1 second wake retry, so the reported aggressive post-conversation heartbeat behavior remains a valid narrow regression distinct from the broader session-lock and activity-heartbeat trackers.

## What This Changes

The issue requests that heartbeat skips caused by active conversations use a less aggressive retry delay or idle-aware retry behavior instead of retrying roughly every second and immediately taking the session lock.

## Best Possible Solution

Add a heartbeat-owned busy-skip retry delay or bounded backoff that preserves no-drift scheduling while keeping broader session-lock narrowing and activity-based heartbeat behavior in their existing trackers.

## Reproduction Assessment

Yes for the scheduler/wake path: current source returns retryable busy skip reasons for active lanes and tests assert the retry reaches `runOnce` after roughly 1 second. The end-to-end Telegram stall was not live-run in this read-only pass.

## Solution Assessment

Yes. A narrow heartbeat busy-skip delay or backoff is the best immediate repair because it addresses the regression without reintroducing full-interval drift or waiting on broader non-blocking heartbeat/session-lock redesigns.

## Repair Work Prompt

Fix the heartbeat busy-skip retry regression tracked by https://github.com/openclaw/openclaw/issues/40611. Current main retries `requests-in-flight`, `cron-in-progress`, and `lanes-busy` through the shared 1 second `DEFAULT_RETRY_MS` in `src/infra/heartbeat-wake.ts`, so a heartbeat can run immediately after an active conversation releases the session lane. Add a heartbeat-owned busy-skip retry delay or bounded backoff separate from generic thrown-error retry, preserving the no-drift behavior from https://github.com/openclaw/openclaw/pull/39182 and the typed wake/cooldown behavior from https://github.com/openclaw/openclaw/pull/76086. Prefer a minimal internal default delay/backoff; only add a user-facing config option if it is clearly needed, and if so update schema, exported types, docs/help/metadata, and tests consistently. Do not broaden this into session write-lock scope changes or activity-based heartbeat behavior; those belong to https://github.com/openclaw/openclaw/issues/13744 and https://github.com/openclaw/openclaw/issues/14051. Add or update focused regression tests for wake retry timing and runner scheduler behavior, and add a user-facing changelog entry if the landing workflow requires one.
