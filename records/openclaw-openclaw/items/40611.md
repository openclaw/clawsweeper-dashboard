## Label Justifications

- `P1`: The issue reports a real-user channel workflow regression where active conversations can be blocked for minutes by heartbeat retry timing.
- `impact:session-state`: The failure mode involves heartbeat runs contending with the same session/runtime state used by active conversation turns.
- `impact:message-loss`: The reported symptom is Telegram, Feishu, or WebChat messages being blocked, delayed, or effectively swallowed during heartbeat contention.

## Summary

Keep open: current main and the latest release still use the 1s heartbeat wake retry for `requests-in-flight` skips, and an open closing PR already owns the narrow fix path.

## What This Changes

Fix a regression where heartbeat skips during active conversations retry too quickly and can let a heartbeat grab the session path before follow-up channel messages are handled.

## Best Possible Solution

Land or replace the open heartbeat retry PR with a narrow per-target retry timing fix, focused scheduler coverage, and redacted live channel proof that follow-up messages are not blocked by a due heartbeat.

## Reproduction Assessment

Yes, source-level: current main returns `requests-in-flight` for active reply or busy lanes and the wake layer retries all retryable busy skips after `DEFAULT_RETRY_MS = 1_000`. I did not run a live Telegram stall reproduction in this read-only review.

## Solution Assessment

No, not exactly as originally suggested: adding a new `heartbeat.retryDelayMs` config is not required to fix the regression and would raise OpenClaw's config-surface bar. The best fix is a narrow default retry-timing change that preserves fast retries for other busy reasons, plus tests and live channel proof.
