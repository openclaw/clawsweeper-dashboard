## Summary

Keep open. Current main and the latest release still route retryable heartbeat busy skips through the shared 1 second wake retry, while the direct repair was closed unmerged and the broader session-lock and activity-heartbeat trackers do not cover this narrow retry regression.

## What This Changes

The issue reports that the no-drift heartbeat scheduling fix made active-conversation heartbeat skips retry too aggressively, causing heartbeat turns to run as soon as a channel conversation releases the session lane.

## Best Possible Solution

Add a bounded heartbeat-owned busy-skip retry delay or backoff separate from generic error retry, preserving no-drift scheduling and leaving session-lock scope and activity-based heartbeat work to their canonical trackers.

## Reproduction Assessment

Yes for the scheduler/wake path: current main returns requests-in-flight for busy main/session lanes and tests assert the retry after 1,000 ms. The full Telegram channel stall was not live-run during this read-only review.

## Solution Assessment

Yes. A narrow heartbeat busy-skip retry/backoff is the best immediate fix because it addresses the regression without reintroducing full-interval drift; the old direct PR is useful source material but should be adjusted for bounded-delay and hot-reload reset concerns if a config knob is added.
