## Summary

Keep open: current main and v2026.6.8 still retry `requests-in-flight` heartbeat skips through the shared 1s wake retry path, and the linked closing PR remains open rather than merged. The issue is still necessary as the canonical tracker until that PR, or a narrow replacement, lands with the retry fix and real channel proof.

## What This Changes

The issue asks OpenClaw to stop due heartbeats from retrying immediately after `requests-in-flight` skips and blocking active Telegram or similar channel conversations.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/89989 or a narrow replacement that preserves per-target retry timing, proves active channel messages are no longer blocked by due heartbeats, and then close this issue from the merged change.

## Reproduction Assessment

Yes, at source level: current main returns retryable `requests-in-flight` heartbeat skips and the wake layer retries them through the shared 1s retry path. I did not run a live Telegram/Feishu/WebChat stall reproduction in this read-only review.

## Solution Assessment

Yes for the direction, but not complete until the linked PR's review blockers are resolved. A narrow wake-layer retry debounce is the right owner boundary for this regression; a new config knob or the broader activity-idle feature is optional follow-up, not required to keep this issue actionable.
