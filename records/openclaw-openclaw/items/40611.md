## Summary

Keep open. Current main still routes retryable heartbeat busy skips through the shared 1 second wake retry, and the latest release has the same behavior, so the reported post-conversation heartbeat contention remains a valid narrow regression.

## What This Changes

The issue requests less aggressive heartbeat retry behavior after `requests-in-flight` skips so heartbeats do not immediately seize the session after an active conversation.

## Best Possible Solution

Add a heartbeat-owned busy-skip retry delay or bounded backoff that preserves no-drift scheduling while leaving broader session-lock narrowing and activity-based heartbeat policy to their existing trackers.

## Reproduction Assessment

Yes, source-reproducible for the scheduler and wake path: current source returns retryable busy skip reasons for active lanes, and tests assert the retry reaches `runOnce` after about 1 second. The live Telegram stall itself was not reproduced in this read-only review.

## Solution Assessment

Yes. A narrow busy-skip delay or bounded backoff is the best immediate repair because it addresses the regression without reintroducing full-interval drift or waiting on broader session-lock and activity-heartbeat redesigns.

## Repair Work Prompt

Fix the heartbeat busy-skip retry regression reported in https://github.com/openclaw/openclaw/issues/40611. Current main retries `requests-in-flight`/busy heartbeat skips through the shared 1 second wake retry, which can fire a heartbeat immediately after an active conversation releases the session. Add a heartbeat-owned busy-skip retry delay or bounded backoff in the wake/runner path so `requests-in-flight` and related busy skips retry later without reintroducing full-interval schedule drift. Keep error retry behavior separate if possible, preserve the no-drift behavior from https://github.com/openclaw/openclaw/pull/39182, preserve typed wake/cooldown semantics, and avoid broad session-lock or activity-based heartbeat redesigns; those belong to https://github.com/openclaw/openclaw/issues/13744 and https://github.com/openclaw/openclaw/issues/14051. Prefer an internal default/backoff over adding a new public config unless the existing config contract already has an approved seam. Add focused regression coverage for the new retry delay and update docs/schema tests only if a public config/doc surface is intentionally changed. Add a changelog entry if the user-visible heartbeat behavior changes.
