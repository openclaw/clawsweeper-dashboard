## Label Justifications

- `P1`: The report describes a regression where heartbeat retry timing can block active channel conversations for minutes after a busy session drains.
- `impact:session-state`: The failure is driven by heartbeat scheduling, session-lane state, reply activity, and retry timing around active conversation state.
- `impact:message-loss`: The user-visible effect is delayed, swallowed, or apparently missing Telegram, WebChat, or Feishu responses while heartbeat work occupies the session path.

## Summary

Keep open: current main and the latest release still retry retryable heartbeat busy skips through the shared 1 second wake cooldown, so the reported post-conversation heartbeat race remains source-reproducible.

## What This Changes

The issue asks for heartbeat busy-skip retries to stop racing active Telegram conversations after the drift fix made requests-in-flight skips retry through the 1 second wake cooldown.

## Best Possible Solution

Implement a heartbeat-owned typed busy-skip debounce/backoff or session-idle signal for request/session busy reasons, preserving fast retry for non-conversation transient paths and leaving the broader activity-based heartbeat policy to https://github.com/openclaw/openclaw/issues/14051.

## Reproduction Assessment

Yes. Current source returns retryable busy skip reasons and the wake layer retries them after 1 second; current tests also pin that timing path, though I did not rerun a live Telegram stall.

## Solution Assessment

Yes for keeping this open and queueing a focused repair. The maintainable fix is a typed busy-skip debounce/backoff or idle signal, not a flat user-tuned retry knob or the full activity-based heartbeat feature.
