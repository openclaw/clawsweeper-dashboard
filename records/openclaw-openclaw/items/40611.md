## Summary

Keep open: current main and v2026.5.22 still route retryable heartbeat busy skips through the shared 1 second wake retry, and the direct retry-delay PR was closed unmerged.

## What This Changes

The issue reports a regression where heartbeat busy-skip retries run about 1 second after active conversation pressure clears, letting heartbeat turns block Telegram and other channel replies.

## Best Possible Solution

Add a heartbeat-owned reason-specific debounce/backoff for retryable conversation/session busy skips, while preserving fast retry for non-busy handler errors and leaving idle-timeout or non-blocking heartbeat architecture to their canonical trackers.

Yes, source-reproducible: current main returns retryable busy skip reasons and current tests pin the wake-layer retry at 1 second. I did not rerun a live Telegram stall in this read-only review.

## Solution Assessment

Yes for a focused repair: the narrow maintainable path is a typed busy-skip debounce/backoff in the heartbeat retry path, not a full idle-timeout feature or non-blocking heartbeat redesign.

## Repair Work Prompt

Fix the heartbeat busy-skip retry regression reported in https://github.com/openclaw/openclaw/issues/40611. Current main returns retryable busy skip reasons for main/session/agent busy pressure and heartbeat-wake retries them through DEFAULT_RETRY_MS=1000, which can fire heartbeat work immediately after an active conversation drains. Add a heartbeat-owned reason-specific debounce/backoff for conversation/session busy skips, preserving PR 39182's no-drift behavior and preserving fast retry for unrelated thrown handler errors. Keep the change bounded to heartbeat wake/runner scheduling and focused tests; do not implement the broader activity-based idle-timeout feature from https://github.com/openclaw/openclaw/issues/14051 or the non-blocking heartbeat architecture from https://github.com/openclaw/openclaw/issues/52046. If a public config option is added, update exported types, strict schema, labels/help/docs, config inventory tests, and clamp timer values safely; otherwise prefer an internal default with tests proving requests-in-flight and lanes-busy use the slower busy retry while ordinary error retry remains fast. Add a changelog entry because this is user-visible behavior.
