## Summary

Keep open. Current main and v2026.5.3 still route retryable heartbeat busy skips through the shared 1 second wake retry with no heartbeat-owned retry delay, while the direct fix PR is closed unmerged and the broader lock-scope/activity-heartbeat issues do not cover this narrow regression.

## What This Changes

Users report that heartbeat busy-skip retries now fire aggressively after active conversations, blocking Telegram/WebChat/Feishu turns and lacking a heartbeat-specific retry delay.

## Best Possible Solution

Add a heartbeat-owned busy-skip retry delay or backoff that preserves no-drift scheduling, exposes the intended config/docs surface if configurable, and leaves session lock narrowing plus activity-based heartbeat semantics to their existing follow-ups.

## Reproduction Assessment

Yes for the scheduler path: current source and fake-timer tests show `requests-in-flight` busy skips retry after about 1 second. The full Telegram/Feishu/WebChat blocking symptom was not live-reproduced here.

## Solution Assessment

Yes. A narrow heartbeat busy-session retry policy is the best immediate fix because it addresses the regression without reintroducing full-interval drift; broader lock-scope and activity-heartbeat work belong in #13744 and #14051.
