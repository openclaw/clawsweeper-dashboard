## Label Justifications

- `P1`: The report describes a regression where heartbeat retry timing can block active channel conversations for minutes after a busy session drains.
- `impact:session-state`: The failure depends on heartbeat scheduling, session-lane state, active reply state, and retry timing around an active conversation.
- `impact:message-loss`: The user-visible effect is delayed, swallowed, or apparently missing Telegram, WebChat, and Feishu responses while heartbeat work occupies the session path.

## Summary

Keep open: current main and v2026.5.20 still retry retryable heartbeat busy skips through the shared 1 second wake cooldown, so the reported post-conversation heartbeat race remains source-reproducible and is not superseded by the broader idle-timeout tracker.

## What This Changes

The issue asks OpenClaw to stop retrying skipped heartbeats almost immediately after active conversation busy skips, because that can let heartbeat runs take the session path before Telegram or other channel replies drain.

## Best Possible Solution

Add a heartbeat-owned typed debounce/backoff for request/session busy retries, preserving fast retry for unrelated transient failures and leaving the broader activity-based heartbeat policy to https://github.com/openclaw/openclaw/issues/14051.

## Reproduction Assessment

Yes. Current source returns retryable busy skip reasons and the wake layer retries them after 1 second; current tests also pin that timing path, though I did not rerun a live Telegram stall.

## Solution Assessment

Yes for keeping this open and queueing a focused repair. The maintainable fix is a typed busy-skip debounce/backoff or idle signal for request/session busy reasons, not a flat user-tuned retry knob or the full activity-based heartbeat feature.
