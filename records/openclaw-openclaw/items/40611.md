## Label Justifications

- `P1`: Current released behavior can interrupt active channel conversations for minutes when an overdue heartbeat starts immediately after user work drains.
- `impact:message-loss`: The reported heartbeat contention can substantially delay or suppress expected Telegram, Feishu, and WebChat responses.
- `impact:session-state`: The failure concerns heartbeat and user runs contending around shared session execution and transcript state.

## Summary

Keep open: current main and v2026.6.5 still route `requests-in-flight` heartbeat skips through the shared 1-second retry, while https://github.com/openclaw/openclaw/pull/89989 remains the open implementation candidate for the same session and message contention. Reports from Telegram, Feishu, and WebChat support a cross-channel heartbeat scheduling regression rather than a Telegram-only problem.

## What This Changes

Prevent overdue heartbeat retries from immediately contending with active conversations and delaying channel messages.

## Best Possible Solution

Review and land one reason-specific bounded retry backoff with focused scheduler tests and live channel proof, while avoiding a new `heartbeat.retryDelayMs` configuration surface unless maintainers establish that a fixed policy cannot serve the supported workloads.

## Reproduction Assessment

Yes at source level: both current main and v2026.6.5 route `requests-in-flight` through the shared 1-second wake retry, and focused tests explicitly assert that timing. This review did not execute a live transport reproduction, but multiple issue comments report the same contention class across Telegram, Feishu, and WebChat.

## Solution Assessment

Yes, the linked PR's reason-specific fixed backoff is a narrower and more maintainable repair than adding a new user configuration option. It still needs live proof that the selected delay preserves conversation responsiveness and eventually executes the heartbeat.
