## Summary

Keep open: current main and v2026.5.22 still route retryable heartbeat busy skips through the shared 1 second wake retry, and the direct retry-delay PR is closed unmerged. Later Feishu and WebChat comments add similar active-conversation stall evidence, so the narrow regression remains actionable.

## What This Changes

Reports a heartbeat retry regression where busy skips retry after about 1 second and can start a heartbeat immediately after an active Telegram conversation, blocking subsequent channel messages.

## Best Possible Solution

Add a heartbeat-owned, reason-specific debounce/backoff for retryable conversation/session busy skips while preserving fast retry for non-busy handler errors and leaving idle-timeout or non-blocking heartbeat architecture to their canonical trackers.

## Reproduction Assessment

Yes, source-level: current main returns retryable busy skip reasons and current tests pin the wake-layer retry at 1 second. I did not rerun a live Telegram stall in this read-only review.

## Solution Assessment

Yes for a focused repair: the narrow maintainable path is a typed busy-skip debounce/backoff in the heartbeat retry path, not a full idle-timeout feature or non-blocking heartbeat redesign.
