## Summary

Keep open. Current main still treats `requests-in-flight` heartbeat skips as retryable busy skips and requeues them through the shared 1 second wake retry, while the heartbeat config schema, generated config inventory, and public docs still lack a separate busy-session retry delay. The latest release tag also shows the same retry path, and the direct fix candidate in #40629 is closed unmerged.

## What This Changes

Add a configurable heartbeat retry delay/backoff for `requests-in-flight` busy-session skips so the #39182 no-drift fix does not immediately start heartbeat runs after active conversations finish.

## Best Possible Solution

Preserve the #29762/#39182 no-drift scheduling behavior, but add a separate documented and tested heartbeat busy-session retry/backoff policy for `requests-in-flight` skips, leaving broad session-lock scope to #13744 and activity-based idle heartbeat semantics to #14051.
