## Summary

Keep open: current main and the latest release still retry `requests-in-flight` heartbeat skips through the shared 1 second wake retry, and the open linked fix candidate has not merged yet.

## What This Changes

The issue reports that the earlier heartbeat drift fix changed `requests-in-flight` skip behavior into near-immediate retries that can grab the session lane after active conversations and delay Telegram or other channel replies.

## Best Possible Solution

Land or replace the open heartbeat retry debounce PR with a narrow fix that gives `requests-in-flight` skips a longer retry delay while preserving fast retries for unrelated transient heartbeat skips.

## Reproduction Assessment

Yes, at source level: current main still returns `requests-in-flight` skips and the wake layer still retries retryable busy skips after the shared 1 second delay. I did not run a live Telegram stall reproduction in this read-only review.

## Solution Assessment

Yes, the best current path is to review and land a targeted retry debounce for only `requests-in-flight` skips; the broader activity-based heartbeat request remains separate product work.
