## Label Justifications

- `P1`: The reported regression can block active channel conversations for minutes during normal heartbeat scheduling.
- `impact:session-state`: Heartbeat retries compete with active session/runtime state and can seize the same conversation context after a busy skip.
- `impact:message-loss`: The observed Telegram, Feishu, and WebChat symptoms include delayed or missing visible replies while heartbeat work holds the lane.

## Summary

Keep open: current main still routes `requests-in-flight` heartbeat skips through the shared 1 second wake retry, and https://github.com/openclaw/openclaw/pull/89989 is an open closing PR for the narrow fix rather than a merged implementation.

## What This Changes

The issue reports a regression where heartbeat `requests-in-flight` skips retry too aggressively and can block active Telegram or other channel conversations.

## Best Possible Solution

Land or replace the linked PR with a narrow wake-layer fix that gives `requests-in-flight` a longer per-target retry while preserving other busy-skip retry cadence; keep broader idle-timeout and non-blocking heartbeat work separate.

## Reproduction Assessment

Yes, source-reproducible: current code returns `requests-in-flight` for busy main or active reply runs, and `heartbeat-wake.ts` retries that skip with `DEFAULT_RETRY_MS = 1_000`. I did not run a live Telegram, Feishu, or WebChat stall reproduction in this read-only review.

## Solution Assessment

No, current main is not the best end state. The narrow wake-layer fix is the right boundary, but the open linked PR needs review/proof before this issue can close.
