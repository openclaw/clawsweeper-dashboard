## Summary

Keep open. Current main still retries `requests-in-flight` heartbeat skips through the shared 1 second wake retry, there is no separate configurable heartbeat busy-session retry delay in schema/types/docs, the latest release still contains the same retry path, and the direct fix candidate in #40629 is closed unmerged.

## What This Changes

The issue requests a less aggressive, preferably configurable heartbeat retry delay after busy-session skips so heartbeat runs do not immediately preempt active Telegram or similar channel conversations.

## Best Possible Solution

Add a heartbeat-owned busy-session retry policy that preserves the no-drift schedule behavior, gives `requests-in-flight` skips a longer configurable delay or backoff, exposes the setting in config schema/types/help/docs, and adds regression tests for default and custom delays while leaving broad lock-scope and activity-based heartbeat semantics to their existing follow-ups.

## Reproduction Assessment

Yes for the scheduler behavior: the existing fake-timer tests already reproduce a `requests-in-flight` skip and assert the next retry after `1_000` ms. The full Telegram/Feishu user-visible stall was not live-reproduced in this read-only pass, but the code path matches the report.

## Solution Assessment

Yes, a narrow heartbeat busy-session retry delay is the best immediate fix because it addresses the regression without reintroducing full-interval drift. Broader session lock narrowing belongs in #13744, and activity-based heartbeat belongs in #14051.
