## Label Justifications

- `P1`: The report describes active messaging conversations being blocked for minutes by heartbeat retry/session contention.
- `impact:message-loss`: Telegram, Feishu, and WebChat users report delayed, blocked, or effectively swallowed channel replies during heartbeat contention.
- `impact:session-state`: The failing path is a heartbeat/session-lane interaction where retry timing and session locks affect active conversation state.

## Summary

Keep open: current main and the latest release still route `requests-in-flight` heartbeat skips through the shared 1 second wake retry, while https://github.com/openclaw/openclaw/pull/89989 is an open unmerged fix candidate for the same message/session contention. The later Feishu and WebChat comments strengthen the report because they point to the same heartbeat/session blocking class, not a Telegram-only symptom.

## What This Changes

The issue asks OpenClaw to stop heartbeat requests-in-flight skips from retrying so aggressively that a heartbeat can immediately seize the session after an active conversation turn.

## Best Possible Solution

Land or reject the narrow linked retry-delay fix, then close this issue only after current main no longer retries requests-in-flight heartbeat skips on the shared 1 second path.

## Reproduction Assessment

Yes. Current source returns `requests-in-flight` for active reply/session-lane work and current wake/scheduler code retries that skip after the shared 1 second delay; I did not run a live channel repro in this read-only pass.

## Solution Assessment

Yes, the best narrow fix is to change the wake retry timing for `requests-in-flight` without reintroducing full-interval drift. A new config knob is not required unless maintainers explicitly want that extra surface.
