## Summary

Keep open. Current main and v2026.5.7 still retry retryable heartbeat busy skips through the shared 1 second wake retry, while the direct repair PR was closed unmerged and related session-lock/activity-heartbeat trackers do not cover this narrow regression.

## What This Changes

The issue requests a fix for heartbeat busy-skip retry behavior so active conversations are not immediately preempted by a heartbeat after the no-drift scheduler change.

## Best Possible Solution

Add a heartbeat-owned bounded busy-skip retry delay or backoff separate from generic error retry, preserving no-drift scheduling and leaving session-lock scope and activity-based heartbeat work to their existing trackers.

## Reproduction Assessment

Yes for the scheduler path: current source and fake-timer tests show retryable heartbeat busy skips requeue after about 1 second. The full channel-level lock-blocking symptom was not live-run in this read-only review.

## Solution Assessment

Yes. A narrow heartbeat busy-session retry policy is the best immediate fix because it addresses the regression without reintroducing full-interval drift; broader lock-scope and activity-heartbeat changes belong in https://github.com/openclaw/openclaw/issues/13744 and https://github.com/openclaw/openclaw/issues/14051.

## Repair Work Prompt

Fix the heartbeat busy-skip retry regression tracked by https://github.com/openclaw/openclaw/issues/40611. Current main returns retryable busy skips such as requests-in-flight, cron-in-progress, and lanes-busy, and heartbeat-wake retries them all through DEFAULT_RETRY_MS = 1_000; introduce a heartbeat-owned bounded busy-skip retry delay or backoff so active conversations are not immediately preempted when the busy lane clears. Preserve the no-drift behavior from https://github.com/openclaw/openclaw/pull/39182: do not advance nextDueMs by the full heartbeat interval on busy skips and do not record skipped attempts as real runs for cooldown. Use https://github.com/openclaw/openclaw/pull/40629 as useful prior art, but address its review concerns: safe integer bounds, no timer overflow/tight-loop behavior, and correct default reset on hot reload if a config path is added. Keep broader session write-lock scope work in https://github.com/openclaw/openclaw/issues/13744 and activity-based heartbeat in https://github.com/openclaw/openclaw/issues/14051; do not attempt those designs in this repair. Add focused regression tests for the retry delay/backoff in heartbeat-wake and heartbeat-runner scheduler tests. If a public config option is introduced, also update strict schema, exported types, config help/labels/metadata/docs, config-honor inventory, and changelog; otherwise keep the change internal and narrow.
