## Label Justifications

- `P1`: The issue describes a regression that can block active Telegram and other channel conversations for minutes when heartbeat runs seize the session lane.
- `impact:session-state`: The failure is driven by heartbeat scheduling and session-lane contention during active conversation state.
- `impact:message-loss`: The observed effect is user-visible channel messages being delayed, swallowed, or not responded to while heartbeat work is running.

## Summary

Keep open. Current main and the latest release still route retryable heartbeat busy skips through the shared 1 second wake retry, so the reported post-conversation contention remains a valid narrow regression.

## What This Changes

Review whether heartbeat `requests-in-flight` skips still retry too aggressively after the no-drift fix, causing active channel conversations to be blocked by heartbeat runs.

## Best Possible Solution

Implement a heartbeat-owned busy-skip retry delay or bounded backoff that preserves no-drift scheduling and leaves broader session-lock narrowing and activity-based heartbeat policy to their separate trackers.

## Reproduction Assessment

Yes, source-reproducible: current main returns retryable busy skip reasons for active lanes and the wake layer retries them after 1 second. I did not rerun the live Telegram stall, but the scheduler test and shipped code pin the same retry path.

## Solution Assessment

Yes. A narrow busy-skip delay or bounded backoff is the best immediate repair because it addresses the regression without reintroducing full-interval drift or waiting on session-lock and activity-heartbeat redesigns.
