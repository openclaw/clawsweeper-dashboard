## Label Justifications

- `P1`: The report describes a regression where heartbeat retry timing can block active channel conversations for minutes after a busy session drains.
- `impact:session-state`: The failure is driven by heartbeat scheduling, session-lane state, reply-run activity, and retry timing around active conversation state.
- `impact:message-loss`: The user-visible effect is delayed, swallowed, or apparently missing Telegram/WebChat/Feishu responses while heartbeat work occupies the session path.

## Summary

Keep open: current main and v2026.5.18 still send retryable heartbeat busy skips through the shared 1 second wake retry, so the merged active-reply guard does not solve the post-turn heartbeat lock-grab race reported here.

## What This Changes

The issue requests a fix for heartbeat busy-skip retry timing so heartbeats do not grab the session immediately after an active conversation turn drains.

## Best Possible Solution

Implement a heartbeat-owned typed busy-skip debounce/backoff or idle/quiet-window signal for request/session busy reasons, while preserving fast retry for cron/errors and leaving the broader activity-based heartbeat policy to https://github.com/openclaw/openclaw/issues/14051.

## Reproduction Assessment

Yes, source-reproducible: current main returns retryable busy skip reasons and the wake layer retries them after 1 second. I did not rerun a live Telegram stall, but current source and tests pin the same timing path.

## Solution Assessment

Yes for keeping this open and queueing a focused repair. The narrowest maintainable fix is typed busy-skip debounce/backoff or an idle signal, not a flat user-tuned retry config or a full heartbeat-worker redesign in this issue.
