## Summary

Keep open: current main and latest release still route requests-in-flight heartbeat skips through the shared 1 second wake retry, while https://github.com/openclaw/openclaw/pull/89989 is an open, unmerged fix candidate for the same issue.

## What This Changes

The issue reports that heartbeat requests-in-flight skips now retry about every second after a prior heartbeat drift fix, letting heartbeat runs take the session lock immediately after active channel turns.

## Best Possible Solution

Land a narrow wake-layer fix that gives requests-in-flight heartbeat retries a longer delay, with focused wake/scheduler coverage and at least one live transport or runtime proof before closing this issue.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: have the wake handler return requests-in-flight and current main schedules the retry after DEFAULT_RETRY_MS=1000ms. I did not run a live Telegram reproduction in this read-only sweep.

## Solution Assessment

Yes, the best direction is a narrow wake-layer special case for requests-in-flight retries. Adding a new heartbeat.retryDelayMs config surface should not be the first fix unless maintainers decide the default cannot cover the regression.
