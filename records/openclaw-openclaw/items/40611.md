## Summary

Keep this issue open: current main still uses the shared 1 second wake retry for retryable heartbeat busy skips, and an open linked PR explicitly owns the narrow fix. The user reports and follow-up comments remain useful canonical evidence for a still-present heartbeat/session contention bug.

## What This Changes

The issue reports that the heartbeat drift fix made requests-in-flight heartbeat retries too aggressive, causing heartbeat runs to block active Telegram and related channel conversations.

## Best Possible Solution

Keep this issue as the canonical bug report and review or land https://github.com/openclaw/openclaw/pull/89989, unless maintainers choose a broader heartbeat/session-lock redesign tracked separately.

## Reproduction Assessment

Yes for source reproduction: current origin/main still routes retryable busy heartbeat skips through the shared 1 second wake retry path, and tests assert that cadence. I did not run a live Telegram/WebChat/Feishu stall reproduction in this read-only review.

## Solution Assessment

Yes for the near-term issue: a narrow requests-in-flight retry delay in the wake layer is the smallest maintainable fix for this aggressive retry regression. Broader non-blocking or activity-based heartbeat behavior remains separate product work.
