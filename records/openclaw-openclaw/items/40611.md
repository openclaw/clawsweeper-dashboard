## Summary

Keep open. Current main and the latest release still route retryable heartbeat busy skips through the shared 1 second wake retry, while the direct repair at https://github.com/openclaw/openclaw/pull/40629 was closed unmerged and the broader lock/activity trackers do not own this narrow retry regression.

## What This Changes

The issue asks OpenClaw to stop heartbeat busy-skip retries from immediately grabbing active channel conversations after the no-drift heartbeat scheduling change.

## Best Possible Solution

Add a heartbeat-owned busy-skip retry delay or bounded backoff separate from generic error retry, preserving no-drift scheduling and leaving session-lock scope and activity-based heartbeat work to their existing trackers.

## Reproduction Assessment

Yes for the scheduler path: current main source and fake-timer tests show retryable busy skips requeue after about 1 second. The full channel-level blocking symptom was not live-run during this read-only review, but multiple reporter comments provide matching Telegram, Feishu, and WebChat evidence.

## Solution Assessment

Yes. A narrow heartbeat busy-session retry policy is the best immediate fix because it addresses the regression without reintroducing full-interval drift; broader lock-scope and activity-heartbeat changes belong in https://github.com/openclaw/openclaw/issues/13744 and https://github.com/openclaw/openclaw/issues/14051.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/40611. Current main retries retryable heartbeat busy skips through DEFAULT_RETRY_MS = 1_000 in src/infra/heartbeat-wake.ts, so requests-in-flight, cron-in-progress, and lanes-busy skips can requeue too aggressively and start immediately after active conversations drain. Implement a heartbeat-owned busy-skip retry delay or bounded backoff that is longer than the generic error retry, preserves the no-drift behavior from https://github.com/openclaw/openclaw/pull/39182, and does not advance schedule bookkeeping or poison cooldowns for retryable busy skips. Prefer a narrow internal default unless a public config option is clearly needed; if adding config, clamp/bound it, reset to default on config removal or hot reload, and update schema, exported types, schema help/labels/docs, generated config metadata, config-honor inventory, and changelog. Use https://github.com/openclaw/openclaw/pull/40629 as source context but address its bounds and hot-reload concerns. Do not broaden session write-lock scope work from https://github.com/openclaw/openclaw/issues/13744 or activity-based heartbeat behavior from https://github.com/openclaw/openclaw/issues/14051.
