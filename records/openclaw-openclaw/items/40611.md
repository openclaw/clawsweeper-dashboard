## Summary

Keep this open: current main and v2026.6.10 still route requests-in-flight heartbeat busy skips through the shared 1 second wake retry path, and the linked closing PR is open and unmerged.

## What This Changes

The issue reports that the heartbeat drift fix changed requests-in-flight skips into aggressive 1 second wake retries, causing heartbeat runs to seize session state and block active Telegram, WebChat, and Feishu conversations.

## Best Possible Solution

Keep this issue as the canonical bug report until https://github.com/openclaw/openclaw/pull/89989, or an equivalent narrow wake-layer fix, is proven and merged.

## Reproduction Assessment

Yes, from source: requests-in-flight skips are retryable busy skips, and current heartbeat-wake retries them through the shared 1 second DEFAULT_RETRY_MS path. I did not run a live Telegram or Feishu reproduction in this read-only review.

## Solution Assessment

Yes for the near-term direction: separating requests-in-flight retry cadence in heartbeat-wake fixes the reported regression without reintroducing full-interval drift. Broader activity-based or non-blocking heartbeat designs should remain separate product work.
