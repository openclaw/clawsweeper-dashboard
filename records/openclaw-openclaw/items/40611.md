## Summary

Keep open. Current main and v2026.5.7 still route retryable heartbeat busy skips through the shared 1 second wake retry, while the direct repair in https://github.com/openclaw/openclaw/pull/40629 was closed unmerged and the broader session-lock/activity-heartbeat trackers do not cover this narrow retry regression.

## What This Changes

Requests a less aggressive heartbeat busy-skip retry policy so heartbeats do not run immediately after active conversations and block Telegram message handling.

## Best Possible Solution

Add a heartbeat-owned busy-skip retry delay or backoff separate from generic error retry, preserving no-drift scheduling and leaving session-lock scope and activity-based heartbeat work to their existing trackers.

## Reproduction Assessment

Yes for the scheduler path: current main source and fake-timer tests show retryable busy skips requeue after about 1 second. The full Telegram/Feishu/WebChat blocking symptom was not live-run during this read-only review.

## Solution Assessment

Yes. A narrow heartbeat busy-session retry policy is the best immediate fix because it addresses the regression without reintroducing full-interval drift; broader lock-scope and activity-heartbeat changes belong in https://github.com/openclaw/openclaw/issues/13744 and https://github.com/openclaw/openclaw/issues/14051.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/40611. Current main routes retryable heartbeat busy skips (`requests-in-flight`, and likely the same busy-skip family where appropriate) through the shared 1s `DEFAULT_RETRY_MS`, so heartbeats can fire immediately after active conversations and hold the session write lock. Add a heartbeat-owned busy-session retry delay/backoff that preserves the no-drift behavior from https://github.com/openclaw/openclaw/pull/39182: do not advance `nextDueMs` by the full interval on busy skips, do not record busy skips as real runs, and do not reintroduce tight retry loops. Prefer a bounded internal default around 60s unless maintainers explicitly want a public config key; if you add a config key, update strict schema, exported types, schema help/labels, config-honor inventory, docs, tests, changelog, and clamp/reset hot-reload behavior. Do not change session write-lock scope for https://github.com/openclaw/openclaw/issues/13744 or implement activity-based heartbeat for https://github.com/openclaw/openclaw/issues/14051 in this PR.
