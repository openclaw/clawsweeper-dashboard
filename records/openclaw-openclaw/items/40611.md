- `P1`: The issue describes a user-visible regression where heartbeat retries can block active channel conversations for minutes after a busy session drains.
- `impact:session-state`: The failure is driven by heartbeat scheduling, session-lane contention, and retry timing around active conversation state.
- `impact:message-loss`: The reported effect is delayed, swallowed, or apparently missing Telegram/WebChat responses while heartbeat work holds the session path.

## Summary

Keep open: current main and v2026.5.18 still route retryable heartbeat busy skips through the shared 1 second wake retry, so active-reply guards only defer while work is active and do not solve the post-turn heartbeat lock-grab race described here.

## What This Changes

The issue asks OpenClaw to stop heartbeat busy-skip retries from immediately racing active conversations after the heartbeat drift fix.

## Best Possible Solution

Implement a heartbeat-owned typed busy-skip debounce/backoff or idle/quiet-window signal for session/request busy reasons, leaving the broader non-blocking heartbeat worker and activity policy to their existing follow-up issues.

## Reproduction Assessment

Yes, source-reproducible: current main returns retryable busy skip reasons and the wake layer retries them after 1 second. I did not rerun a live Telegram stall, but current code and tests pin the same timing path.

## Solution Assessment

Yes for keeping this issue open and queueing a focused repair. The narrowest maintainable fix is typed busy-skip debounce/backoff or an idle signal, not a flat user-tuned retry config or a full heartbeat-worker redesign in this issue.

## Repair Work Prompt

Fix the heartbeat busy-skip retry race from https://github.com/openclaw/openclaw/issues/40611. Keep the no-drift behavior from https://github.com/openclaw/openclaw/pull/39182, but stop `requests-in-flight`/session-busy heartbeat retries from immediately firing 1 second after active conversation work drains and grabbing the session path before the next user turn. Prefer a typed busy-skip debounce/backoff or idle/quiet-window mechanism in the heartbeat wake/runner boundary; preserve fast retry for unrelated transient failures and cron/dependency skips. Do not add a broad `heartbeat.retryDelayMs` config unless maintainer feedback explicitly chooses that product surface. Leave full non-blocking heartbeat execution to https://github.com/openclaw/openclaw/issues/52046 and activity-based heartbeat policy to https://github.com/openclaw/openclaw/issues/14051. Add focused regression coverage for the new busy-skip timing and keep docs/config unchanged unless the chosen implementation introduces a documented user-facing setting.
