## Label Justifications

- `P1`: The report describes a regression where heartbeat retry timing can block active channel conversations for minutes after a busy session drains.
- `impact:session-state`: The failure depends on heartbeat scheduling, active reply/session state, and retry timing around an active conversation.
- `impact:message-loss`: The user-visible effect is delayed, swallowed, or apparently missing Telegram, WebChat, and Feishu responses while heartbeat work occupies the session path.

## Summary

Keep open: current main and v2026.5.20 still retry heartbeat busy skips through the shared 1 second wake retry, while current runner paths still return retryable busy reasons for active reply/session pressure.

## What This Changes

Fix the heartbeat busy-skip retry behavior so heartbeat work does not immediately grab the session path after an active conversation drains.

## Best Possible Solution

Add a heartbeat-owned reason-specific debounce/backoff for request/session busy retries, while preserving fast retry for unrelated handler errors and leaving activity-based heartbeat policy to the separate canonical tracker.

## Reproduction Assessment

Yes. Current source returns retryable busy skip reasons and current tests pin the wake-layer retry at 1 second; I did not rerun a live Telegram stall.

## Solution Assessment

Yes for a focused repair. The maintainable fix is a typed busy-skip debounce/backoff in the heartbeat retry path, not a full idle-timeout feature or broad non-blocking heartbeat redesign.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/40611. Current main retries retryable heartbeat busy skips through DEFAULT_RETRY_MS=1s in src/infra/heartbeat-wake.ts, so a heartbeat can run immediately after an active user conversation drains and occupy the same session path. Add a heartbeat-owned, reason-specific busy retry debounce/backoff for request/session busy skips, preserving the existing fast retry for handler errors and unrelated transient failures. Prefer a narrow internal default/backoff over adding a public config key unless the existing config contract already requires it. Do not implement the broader activity-based heartbeat idle timeout from https://github.com/openclaw/openclaw/issues/14051 and do not redesign heartbeat into a separate worker. Update focused coverage in src/infra/heartbeat-wake.test.ts and src/infra/heartbeat-runner.scheduler.test.ts, and keep src/infra/heartbeat-runner.skips-busy-session-lane.test.ts green. If docs mention the retry timing, update docs/gateway/heartbeat.md only for the changed behavior.
