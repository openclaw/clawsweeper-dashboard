## Summary

Keep open: current main and the latest release still use the shared 1s heartbeat wake retry for `requests-in-flight`, and an open closing PR already owns the proposed debounce repair. Closing this issue before that PR merges would lose the canonical user-facing tracker for the Telegram/session contention regression.

## What This Changes

The issue requests less aggressive heartbeat retry behavior after `requests-in-flight` skips so active Telegram and similar channel conversations are not immediately blocked by heartbeat runs.

## Best Possible Solution

Keep this issue open as the canonical regression tracker, then close it only after the linked heartbeat retry PR or a narrower replacement lands with source, test, and release/main provenance.

## Reproduction Assessment

Yes at source level: current main and v2026.6.6 still retry `requests-in-flight` heartbeat skips through the shared 1s wake delay, and current tests assert that timing. I did not run a fresh live Telegram/Feishu/WebChat stall reproduction in this read-only review.

## Solution Assessment

Yes, a narrow requests-in-flight retry debounce is the best near-term repair for this regression; the broader activity-based heartbeat idea remains separately tracked. The open linked PR is the current fix path, but it still needs review and proof before this issue can close.
