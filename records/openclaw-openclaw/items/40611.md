## Summary

Keep open: current main and the latest release still route `requests-in-flight` heartbeat skips through the shared 1s wake retry path, and the linked repair PR is open/unmerged with proof and scheduler-review work still outstanding.

## What This Changes

The issue asks OpenClaw to stop heartbeat retry timing from immediately running a heartbeat after an active conversation releases its session lane and blocking Telegram or other channel messages.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/89989 after fixing its scheduler concerns and adding redacted real channel proof, or replace it with a narrow equivalent; close this issue only after the repair merges.

## Reproduction Assessment

Yes, source-level: current main returns retryable `requests-in-flight` skips and the wake layer retries them through the shared 1s retry timer. I did not run a live Telegram, Feishu, or WebChat reproduction in this read-only review.

## Solution Assessment

No close is not the best resolution yet. The narrow retry-delay direction is plausible, but the linked PR or a replacement still needs scheduler-safe timing and real channel proof before this issue is resolved.
