## Summary

Keep open. Current main still routes `requests-in-flight` heartbeat skips through the shared 1 second wake retry, lacks a heartbeat-owned retry/backoff config, and the latest release contains the same behavior. The new Feishu/WebChat comments add fresh affected-surface evidence, while #40629 was a direct fix candidate but is closed unmerged.

## What This Changes

The issue requests a heartbeat busy-session retry/backoff change so active conversations are not immediately preempted by heartbeat retries after `requests-in-flight` skips.

## Best Possible Solution

Add a heartbeat-owned busy-session retry policy that preserves the no-drift schedule behavior, gives `requests-in-flight` skips a longer configurable delay or bounded backoff, documents the setting, and adds focused regression tests for default and custom delays while leaving lock-scope and activity-based heartbeat semantics to their existing follow-ups.

## Reproduction Assessment

Yes for the scheduler behavior: current tests and source show a `requests-in-flight` skip retrying after `1_000` ms through the shared wake retry. The full channel stall was not live-reproduced in this read-only review, but the code path matches the Telegram, Feishu, and WebChat reports.

## Solution Assessment

Yes, a narrow heartbeat busy-session retry delay/backoff is the best immediate fix because it addresses the regression without reintroducing full-interval drift. Broader session write-lock narrowing belongs in #13744, and activity-based heartbeat belongs in #14051.
