## Summary

Keep open. Current main and the latest release still route retryable heartbeat busy skips through the shared 1 second wake retry, so the reported post-conversation heartbeat contention remains a valid narrow regression rather than a stale or superseded report.

## What This Changes

The issue asks OpenClaw to stop retrying heartbeat busy skips almost immediately after active conversations, especially when that blocks Telegram message handling.

## Best Possible Solution

Add a heartbeat-owned busy-skip retry delay or bounded backoff that preserves no-drift scheduling while leaving broader session-lock narrowing, non-blocking heartbeat execution, and activity-based heartbeat behavior to their existing trackers.

## Reproduction Assessment

Yes for the scheduler and wake path: current source returns retryable busy skip reasons for active lanes, and tests assert the retry reaches `runOnce` after about 1 second. The live Telegram stall itself was not reproduced in this read-only review.

## Solution Assessment

Yes. A narrow busy-skip delay or bounded backoff is the best immediate repair because it addresses the regression without reintroducing full-interval drift or waiting on broader session-lock and activity-heartbeat redesigns.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/40611 with a narrow heartbeat busy-skip retry repair. Current main retries retryable busy skips, especially `requests-in-flight`, through `DEFAULT_RETRY_MS = 1_000`, which can fire a heartbeat immediately after a user turn releases the session lane. Add a heartbeat-owned bounded delay/backoff for retryable busy skips that preserves the no-drift behavior from https://github.com/openclaw/openclaw/pull/39182 and does not broaden session-lock scope, implement activity-based heartbeat, or change unrelated wake/cooldown semantics. Prefer avoiding a new public config surface unless maintainers explicitly want it; if a public config key is added, update schema/types/help/docs/generated config metadata and clamp/bound the value to avoid timer storms or overflow. Add/update focused tests in the heartbeat wake and scheduler suites so `requests-in-flight` no longer retries after 1 second while error retries keep the existing short retry behavior.
