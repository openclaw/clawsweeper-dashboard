## Label Justifications

- `P1`: The issue describes a user-visible regression where heartbeat retries can block active channel conversations for minutes after a busy session drains.
- `impact:session-state`: The failure is driven by heartbeat scheduling, session-lane contention, and retry timing around active conversation state.
- `impact:message-loss`: The reported effect is delayed, swallowed, or apparently missing Telegram and other channel responses while heartbeat work holds the session path.

## Summary

Keep open: current main and the latest release still route retryable heartbeat busy skips through the shared 1 second wake retry, so active reply/session-lane guards only defer while work is active and do not solve the post-conversation lock-grab race described here.

## What This Changes

Adjust heartbeat busy-skip retry behavior so heartbeat runs do not immediately seize session lanes after active Telegram or other channel conversations.

## Best Possible Solution

Implement a heartbeat-owned typed busy-skip debounce/backoff, preferably via an idle/quiet-window signal or a bounded delay derived from the skip reason and heartbeat interval, while leaving full non-blocking heartbeat architecture to https://github.com/openclaw/openclaw/issues/52046 and activity-based policy to https://github.com/openclaw/openclaw/issues/14051.

## Reproduction Assessment

Yes, source-reproducible: current main returns retryable busy skip reasons for active lanes/reply runs and the wake layer retries them after 1 second. I did not rerun the live Telegram stall, but current code and tests pin the same timing path.

## Solution Assessment

Yes for the immediate repair: a typed requests-in-flight/lanes-busy debounce or bounded backoff is the narrowest maintainable fix. A flat `heartbeat.retryDelayMs` config is less ideal, and a full non-blocking heartbeat worker belongs to the broader linked architecture items.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/40611. Current main retries `requests-in-flight`/busy heartbeat skips through the shared 1 second `DEFAULT_RETRY_MS`, so a heartbeat can run immediately after a user turn drains and seize the same session path. Implement a narrow heartbeat-owned typed busy-skip debounce/backoff: preserve the no-drift fix from the landed requests-in-flight change, keep fast retry for true transient errors or dependency-style skips, and avoid adding a flat `heartbeat.retryDelayMs` config unless a maintainer explicitly asks. Prefer a quiet-window/idle-signal style repair when it stays bounded and simple; otherwise map session/user-busy skip reasons to a safe bounded delay derived from the heartbeat interval or a small fixed quiet window. Add focused regression coverage for requests-in-flight retry timing and ensure active reply/session-lane guards still skip. Update docs/changelog only if the user-visible behavior or config surface changes.
