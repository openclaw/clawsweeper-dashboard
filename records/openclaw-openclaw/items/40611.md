## Summary

Keep open: current main and the latest v2026.6.6 release still retry all retryable heartbeat busy skips through the shared 1 second wake retry, and the open linked PR has not landed yet. This issue should remain the canonical tracker until https://github.com/openclaw/openclaw/pull/89989 is repaired, proven, and merged or replaced.

## What This Changes

The issue reports that requests-in-flight heartbeat skips retry too aggressively and can block active Telegram, WebChat, or Feishu conversations when the heartbeat run takes over the session path.

## Best Possible Solution

Land or replace the linked PR with a narrow heartbeat-wake fix that preserves per-target retry timing, gives requests-in-flight a longer debounce than cron/lane busy skips, updates focused tests, and adds redacted live channel proof before closing this issue.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.6.6 shows requests-in-flight still goes through the shared 1 second wake retry path; I did not run a live transport stall reproduction in this read-only review.

## Solution Assessment

Yes for the immediate bug shape: the narrow fix belongs in heartbeat-wake retry timing, not in a new channel-specific workaround. The broader activity-based heartbeat idea remains a separate product/config request.
