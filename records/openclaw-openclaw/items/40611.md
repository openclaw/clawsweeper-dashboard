## Summary

Keep open: current main and the latest release still use the shared 1-second heartbeat wake retry for requests-in-flight skips, and an open linked PR already owns the narrow fix path. This issue should remain the canonical tracker until that PR lands or maintainers choose a replacement.

## What This Changes

The issue reports that heartbeat requests-in-flight retries became too aggressive after the drift fix, causing active Telegram and related channel conversations to be blocked by heartbeat runs.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/89989 with the same narrow behavior: keep the drift fix, but debounce requests-in-flight heartbeat retries separately from other busy retries.

## Reproduction Assessment

Yes. Source inspection shows current main queues requests-in-flight skips into the shared 1-second wake retry path, and current tests assert that behavior; I did not run a live Telegram or WebChat stall reproduction in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is to keep the schedule-drift repair but give requests-in-flight skips their own longer retry delay, avoiding a new config surface unless maintainers explicitly want one.
