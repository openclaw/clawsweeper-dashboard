## Summary

Keep open. Current main has adjacent name-based tool policies, before_tool_call hooks, plugin install scanning, HTTP /tools/invoke deny-listing, and security audit warnings, but it still lacks the requested capability manifest, explicit grant/revoke state, pre-model capability filtering, cross-surface execution checks, and durable high-risk capability audit receipts.

## What This Changes

Request a core/plugin-API capability permission model for skills/tools that default-denies high-risk actions unless explicitly granted and auditable.

## Best Possible Solution

Keep this as a core/plugin-API security hardening item and split implementation into manifest schema/diagnostics, persisted grants or revocations, pre-model filtering after existing tool-policy resolution, execution-boundary enforcement, and durable capability audit receipts.
