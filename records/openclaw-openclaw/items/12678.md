## What This Changes

The issue requests a capability-based permission and audit model for skills/tools with default-deny handling for high-risk actions.

## Best Possible Solution

Keep this as a core/plugin-API security hardening item and split it into manifest schema/diagnostics, persisted grants/revocations, pre-model filtering after tool-policy resolution, execution-boundary checks, and durable capability audit receipts coordinated with related skill-manifest and MCP deny-list work.
