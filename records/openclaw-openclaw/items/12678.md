## Summary

Keep open: current main and v2026.6.8 still have adjacent name/profile tool policies, before_tool_call/trusted-policy hooks, plugin metadata, and Gateway HTTP deny lists, but not the requested manifest-declared capabilities, explicit grant/revoke state, default-deny pre-model filtering, cross-surface execution authorization, or durable capability audit trail.

## What This Changes

The issue requests a capability-based permission model for skills and tools with manifest declarations, explicit grants/revocations, default-deny handling for high-risk actions, and audit logging.

## Best Possible Solution

Keep this open as the core/plugin-API security hardening umbrella and land a maintainer-approved phased design for schema, grants, filtering, execution checks, and durable audit records coordinated with the narrower RFCs.

## Reproduction Assessment

Yes for source-level gap proof, not for a runtime bug reproduction. Current main and v2026.6.8 show name/profile tool policy, before_tool_call hooks, and surface deny lists, but no capability manifest plus grant/revoke enforcement model.

## Solution Assessment

Unclear as a final implementation shape: the requested direction addresses a real security-hardening gap, but maintainers still need to decide the taxonomy, grant storage, upgrade behavior, and enforcement boundary before code should land.
