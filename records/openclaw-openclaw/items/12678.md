## Summary

Keep open. Current main has adjacent name-based tool policies, before_tool_call hooks, plugin install scanning, HTTP /tools/invoke deny-listing, and security audit warnings, but it still lacks the requested capability manifest, explicit grant/revoke state, pre-model default-deny filtering, cross-surface execution checks, and durable high-risk capability receipts.

## What This Changes

Request a capability-based permission model for skills and tools that declares sensitive capabilities, defaults high-risk actions to deny, supports grants/revocations, and audits high-risk invocations.

## Best Possible Solution

Keep this as a core/plugin-API security hardening item and split implementation into manifest schema/diagnostics, persisted grants or revocations, pre-model filtering after existing tool-policy resolution, execution-boundary enforcement, and durable capability audit receipts coordinated with https://github.com/openclaw/openclaw/issues/12219 and https://github.com/openclaw/openclaw/issues/74594.

## Reproduction Assessment

Not applicable: this is a security/product feature request, not a runtime bug with a failing repro. Static inspection of current main gives a high-confidence absence check for the requested capability grant and audit layer.

## Solution Assessment

Unclear whether the exact capability taxonomy and grant storage proposed here are the final best contract. Keeping this open for security/API design is the right path because existing name-based policies, before_tool_call hooks, and HTTP deny-listing are not equivalent to first-class capability grants.
