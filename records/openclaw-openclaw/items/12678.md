## Summary

Keep open. Current main has adjacent tool policies, hook-time blocking, exec/plugin approvals, install scanning, and an HTTP-only dangerous-tool deny list, but it still lacks the requested manifest-declared capability grants, default-deny pre-exposure filtering, cross-surface invocation enforcement, and durable high-risk audit records.

## What This Changes

Request a capability-based permission model for skills/tools with manifest declarations, explicit grants/revocations, default-deny high-risk filtering/enforcement, and audit logging.

## Best Possible Solution

Stage this as core/plugin-API security hardening: schema diagnostics first, then persisted grants/revocations, CLI/UI/doctor visibility, pre-exposure filtering, invocation-boundary checks, and durable audit records.

## Reproduction Assessment

Not applicable; this is a security-hardening feature rather than a current-main bug. Source inspection of manifest, skill metadata, tool policy, gateway, MCP, hook, approval, and docs surfaces shows the requested grant model is still absent.

## Solution Assessment

Yes, keeping this open for staged maintainer design is the best current path. Existing name-based policies, before_tool_call hooks, approvals, install scanning, and HTTP deny lists are useful foundations but do not replace a static capability grant layer enforced before exposure and again at invocation boundaries.
