## Summary

Keep open: current main and v2026.6.10 still have adjacent name/profile tool policies, before_tool_call approvals, plugin metadata, and Gateway HTTP deny lists, but not the requested manifest-declared capabilities, explicit grants/revocations, pre-model default-deny filtering, cross-surface execution checks, or durable high-risk capability audit records.

## What This Changes

The issue requests a capability-based permission model for skills and tools with default-deny high-risk actions, grant/revoke consent, and audit logging.

## Best Possible Solution

Keep this as the core/plugin-API security-hardening umbrella and split implementation into maintainer-approved phases for schema, grants, filtering, execution checks, and durable audit records coordinated with the narrower RFCs.

## Reproduction Assessment

Yes, as a source-level feature gap rather than a runtime bug: current main exposes name/profile tool policy, hook-time governance, and surface deny lists, but no capability manifest plus grant/revoke enforcement model.

## Solution Assessment

Unclear as a final implementation shape: the direction fits the security gap, but maintainers still need to decide taxonomy, grant persistence, default-deny upgrade behavior, and how static grants compose with dynamic policy hooks.
