## Summary

Keep open. Current main has adjacent name-based tool policies, before_tool_call hooks, install scanning, and an HTTP tool deny list, but it still lacks the requested capability manifest, explicit grant/revoke state, pre-model default-deny filtering, cross-surface execution checks, and durable capability audit receipts. The linked @vainplex/openclaw-governance plugin is useful call-time prior art, but it does not replace a core/plugin-API grant layer.

## What This Changes

The issue requests capability-based permissions for skills/tools, with declared sensitive capabilities, explicit grants, default-deny high-risk actions, consent/revocation UX, and audit logging.

## Best Possible Solution

Keep this as a core/plugin-API security hardening item split into manifest schema/diagnostics, persisted grants/revocations, pre-model filtering after tool-policy resolution, execution-boundary checks, and durable capability audit receipts.

Not applicable: this is a security/product feature request, not a runtime bug with a failing repro. Static inspection of current main gives a high-confidence absence check for the requested capability grant and audit layer.

## Solution Assessment

Unclear whether the exact taxonomy and grant storage proposed here are the final contract. Keeping this open for security/API design is the right path because existing name-based policies, before_tool_call hooks, and HTTP deny-listing are not equivalent to first-class capability grants.
