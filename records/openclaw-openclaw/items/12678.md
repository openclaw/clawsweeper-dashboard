## Summary

Keep open. Current main has adjacent hardening, but it still lacks the requested manifest-declared capability grants, persisted grant/revoke state, pre-model default-deny filtering, cross-surface execution checks, and durable high-risk capability audit records.

## What This Changes

The issue requests capability-based default-deny permissions for skills and tools, with manifest declarations, operator grants, pre-exposure filtering, execution-boundary authorization, and audit logging.

## Best Possible Solution

Keep this as a core/plugin-API security hardening item and split it into reviewable stages: schema diagnostics, persisted grants, CLI/UI/doctor visibility, tool-exposure filtering, execution-boundary enforcement, and durable audit records.

## Reproduction Assessment

Not applicable as a feature/security-hardening request rather than a bug report. The high-confidence verification path is source inspection of current manifest, skill metadata, tool-policy, gateway, MCP, and audit surfaces, which shows the requested grant model is still absent.

## Solution Assessment

Yes, keeping this open for staged maintainer design is the best current path. Existing name-based tool policies, hooks, approvals, install scanning, and HTTP deny lists are useful foundations but are not a substitute for a static capability grant layer enforced before exposure and at invocation boundaries.
