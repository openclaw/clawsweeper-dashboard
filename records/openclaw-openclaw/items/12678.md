## Summary

Keep open. Current main still has adjacent name-based tool policies, plugin hooks, optional plugin tools, install scanning, and an HTTP tool deny list, but not the requested capability manifest, grant/revoke state, pre-model default-deny capability filtering, cross-surface execution checks, or durable capability audit records.

## What This Changes

The issue requests a capability-based permission model for skills and tools, with declared capabilities, explicit grants, default-deny high-risk actions, consent/revocation UX, and audit logs.

## Best Possible Solution

Keep this as a core/plugin-API security hardening item split into manifest schema/diagnostics, persisted grants/revocations, pre-model capability filtering after tool-policy resolution, execution-boundary checks, and durable capability audit receipts.
