## Summary

Keep open: current main still has adjacent name/profile tool policies, before_tool_call governance hooks, plugin metadata, and Gateway HTTP deny lists, but it does not implement the requested manifest-declared capabilities, explicit grant/revoke state, pre-model default-deny filtering, cross-surface authorization, or durable high-risk capability audit records. The linked @vainplex/openclaw-governance plugin is useful prior art for call-time policy, but commenters correctly distinguish it from the missing core/plugin-API grant layer.

## What This Changes

Introduce capability-based permissions for skills/tools with manifest declarations, explicit grants and revocations, default-deny filtering, cross-surface enforcement, and high-risk audit logs.

## Best Possible Solution

Keep this as the core/plugin-API security-hardening umbrella and land a maintainer-approved phased design for schema, grants, filtering, execution checks, and durable audit records coordinated with the narrower RFCs.

## Reproduction Assessment

Yes for source-level gap proof, not for a runtime bug reproduction: current main shows name/profile policies, call-time hooks, and surface deny lists, but no capability manifest plus grant/revoke enforcement model.

## Solution Assessment

Unclear as a final implementation shape: the direction addresses a real security-hardening gap, but maintainers still need to decide taxonomy, storage, upgrade behavior, and enforcement boundaries.
