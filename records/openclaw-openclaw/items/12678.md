## Summary

Keep open. Current main has useful adjacent controls, and the linked @vainplex/openclaw-governance plugin is relevant prior art, but OpenClaw still lacks the requested manifest-declared capability grant layer with persisted grants, pre-model default-deny filtering, cross-surface invocation enforcement, and durable high-risk audit records.

## What This Changes

Request capability-based permissions for skills/tools with manifest declarations, default-deny high-risk grants, UI/CLI consent and revocation, and immutable audit records.

## Best Possible Solution

Stage this as core/plugin-API security hardening: schema diagnostics first, then persisted grants, CLI/UI/doctor visibility, pre-exposure filtering, invocation-boundary checks, and durable audit records.

## Reproduction Assessment

Not applicable; this is a security-hardening feature rather than a bug. The high-confidence verification path is source inspection of current manifest, skill metadata, tool-policy, gateway, MCP, hook, and docs surfaces, which shows the requested grant model is still absent.

## Solution Assessment

Yes, keeping this open for staged maintainer design is the best current path. Existing name-based tool policies, before_tool_call hooks, plugin approvals, install scanning, and HTTP deny lists are useful foundations but do not replace a static capability grant layer enforced before exposure and again at invocation boundaries.
