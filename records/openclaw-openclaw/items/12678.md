## Summary

Keep open. Current main has adjacent defenses, including name/profile tool policies, before_tool_call approval/block hooks, exec approvals, security-audit warnings, and a default HTTP tool deny list, but it still lacks the requested manifest-declared sensitive capabilities, explicit grant/revoke state, pre-model default-deny filtering, cross-surface invocation enforcement, and durable high-risk capability audit records.

## What This Changes

The issue requests a capability-based permission model for skills and tools with default-deny handling of high-risk actions.

## Best Possible Solution

Stage this as core/plugin-API security hardening: schema and diagnostics first, then grant/revoke state and UX, pre-exposure filtering, invocation-boundary checks, and durable audit records while keeping dynamic governance plugins complementary.

## Reproduction Assessment

Not applicable; this is a feature/security hardening request rather than a broken current-main behavior. Static inspection against current main gives a high-confidence absence check for the requested manifest/grant/enforcement/audit layer while showing adjacent controls exist.

## Solution Assessment

Yes, keeping this open for staged security/API design is the best current path. A one-shot automated fix or ClawHub-only close would prematurely choose public capability taxonomy, grant storage, consent UX, and cross-surface enforcement contracts.
