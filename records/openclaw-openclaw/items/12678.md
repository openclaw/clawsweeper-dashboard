## Summary

Keep this open for maintainer-led security architecture work. Current main has substantial adjacent controls—tool-name/profile policies, dynamic call-time vetoes, approvals, and gateway-specific deny rules—but still lacks operator-owned scoped capability grants, consistent pre-exposure and execution-boundary enforcement, revocation, and durable capability auditing; the narrower related issues remain open and do not supersede this scope.

## What This Changes

Introduce capability declarations and operator-controlled grants for sensitive skill and tool actions, with default-deny enforcement, consent, revocation, and durable auditing.

## Best Possible Solution

Keep this issue as the canonical authorization architecture: approve a small typed capability taxonomy and trust model, store operator grants canonically, integrate capability filtering after existing tool-policy resolution, reuse the same decision at every execution boundary, and add durable audit records in phased PRs coordinated with https://github.com/openclaw/openclaw/issues/12219 and https://github.com/openclaw/openclaw/issues/48510.

## Reproduction Assessment

Yes at the source level: current main demonstrably has name-based policy and dynamic call-time hooks but no scoped operator-owned capability grant contract. A failing runtime reproduction is not applicable because the issue requests new defense-in-depth behavior rather than repair of an established contract.

## Solution Assessment

Yes in direction, but not as one large implementation PR. The narrowest maintainable solution is a phased canonical subsystem integrated with existing policy resolution and reused at every invocation boundary, after maintainers approve the taxonomy, persistence model, and upgrade posture.
