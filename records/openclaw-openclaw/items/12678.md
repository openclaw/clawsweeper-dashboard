## Summary

Keep open. Current main has adjacent controls for tool allow/deny policies, before_tool_call hooks and approvals, exec approvals, plugin install scanning, and an HTTP /tools/invoke default deny list, but it still does not implement the requested capability manifest, persisted grants/revocations, pre-model default-deny filtering, execution-boundary capability checks across surfaces, or durable high-risk capability audit records. The discussion explicitly treats @vainplex/openclaw-governance and the related tool-policy PRs as useful prior art, not as completion of this core/plugin-API layer.

## What This Changes

Request to add capability-based permissions for skills and tools, including declared capabilities, explicit grants/revocations, default-deny handling for high-risk actions, UI/CLI consent, and audit records.

## Best Possible Solution

Keep this as a core/plugin-API security hardening item and split it into reviewable stages: schema/parser diagnostics for sensitive capabilities, persisted grants/revocations, CLI/UI/doctor visibility, pre-model filtering after existing tool-policy resolution, execution-boundary checks for HTTP/MCP/non-prompt paths, and durable audit records. Treat @vainplex/openclaw-governance as complementary runtime policy rather than the canonical grant source.

## Reproduction Assessment

Not applicable as a feature/security-hardening request rather than a bug report. There is a high-confidence verification path by inspecting current manifest, skill metadata, tool-policy, gateway, MCP, hook, and docs surfaces, which shows the requested grant model is still absent.

## Solution Assessment

Yes, keeping this open for maintainer design is the best current path. Existing call-time hooks and name-based policies are useful foundations, but they are not a substitute for a static capability grant layer enforced before tool exposure and again at execution boundaries.
