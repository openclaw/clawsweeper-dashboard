## Summary

Current main has substantial adjacent controls—name/profile-based tool policies, pre-exposure filtering, call-time plugin hooks, approvals, and HTTP-specific default denies—but still lacks the requested sensitive-capability declarations, persisted operator grants/revocations, common authorization across invocation surfaces, and durable capability-decision records. The linked @vainplex/openclaw-governance plugin is useful production prior art for dynamic call-time governance, while the remaining generic core/plugin-API authorization layer still needs maintainer product and security design.

## What This Changes

Add capability-based authorization for skills and tools with sensitive capability declarations, operator grants, default-deny enforcement, and durable audit records.

## Best Possible Solution

Retain this issue as the architectural umbrella, coordinate manifest syntax and tool descriptors through the narrower related issues, and deliver the work as staged generic contracts: declarations and diagnostics, SQLite-backed grants, pre-exposure filtering, centralized execution checks, then durable audit records and consent UX.

## Reproduction Assessment

Not applicable; this issue proposes a new defense-in-depth authorization contract rather than reporting a reproducible failure of an existing documented behavior.

## Solution Assessment

Yes as a staged architectural direction, but not as one implementation PR; maintainers first need to define the permanent capability taxonomy, SQLite grant model, compatibility posture, and single enforcement owner.
