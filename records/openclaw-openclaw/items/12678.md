## Summary

Keep open. Current main has useful adjacent safeguards, and @vainplex/openclaw-governance is relevant call-time governance prior art, but OpenClaw still lacks the requested manifest-declared capability grants, explicit grant/revoke flow, pre-model default-deny filtering, cross-surface invocation checks, and durable high-risk audit records.

## What This Changes

The issue requests a capability-based permission model for skills/tools with default-deny high-risk actions, declared requirements, operator grants, and audit logging.

## Best Possible Solution

Stage this as core/plugin-API security hardening: schema diagnostics first, then persisted grants, CLI/UI/doctor visibility, pre-exposure filtering, invocation-boundary checks, and durable audit records.

## Reproduction Assessment

Not applicable; this is a security-hardening feature rather than a bug. The high-confidence verification path is source inspection of the current manifest, skill metadata, tool-policy, gateway, MCP, hook, and docs surfaces, which shows the requested grant model is still absent.

## Solution Assessment

Yes, keeping this open for staged maintainer design is the best current path. Existing name-based tool policies, before_tool_call hooks, install scanning, exec approvals, and HTTP deny lists are useful foundations but do not replace a static capability grant layer enforced before exposure and again at invocation boundaries.
