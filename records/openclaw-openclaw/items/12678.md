## Summary

Current main has strong adjacent controls—optional tool exposure, name-based tool policy, per-call plugin approvals, exec approvals, HTTP/MCP restrictions, and audit diagnostics—but it still lacks one manifest-declared, persisted capability-grant decision enforced consistently before model exposure and at every invocation boundary. Related plugin and capability-descriptor work is useful prior art, not a replacement for this broader core/plugin-API security item.

## What This Changes

Add manifest-declared sensitive capabilities for skills and tools, explicit operator grants and revocations, default-deny filtering and execution-boundary enforcement, and durable high-risk audit records.

## Best Possible Solution

Keep this as the canonical cross-surface grant and enforcement architecture item; define a small capability taxonomy and shared decision contract first, then stage declarations, persisted grants, pre-exposure filtering, execution-boundary enforcement, and SQLite-backed audit while reusing existing policy and approval primitives.

## Reproduction Assessment

Not applicable. This issue proposes a new defense-in-depth authorization contract rather than reporting a failure of already-defined current behavior.

## Solution Assessment

Yes in direction, but not as one large implementation: the best maintainable approach is a shared capability decision consumed by existing exposure, policy, approval, gateway, MCP, and direct-invocation paths, delivered through staged owner-approved PRs.
