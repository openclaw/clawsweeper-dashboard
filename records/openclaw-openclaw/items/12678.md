## Summary

Current main has substantial adjacent safeguards, but it still lacks the requested operator-owned capability declarations, persisted grants and revocations, capability-based pre-exposure filtering, unified execution-boundary authorization, and durable high-risk capability audit records. The linked governance plugin and narrower open manifest/tool-metadata proposals are useful prior art and prerequisites, not substitutes for the remaining core/plugin-API security contract.

## What This Changes

Add manifest-declared, operator-granted capabilities with default-deny enforcement and durable auditing for sensitive skill and tool actions.

## Best Possible Solution

Approve one minimal public capability taxonomy and persisted operator-grant model, then implement diagnostics, pre-model filtering after existing tool-policy resolution, execution-boundary checks across agent/HTTP/MCP paths, and durable audit records in staged, compatibility-reviewed changes.

## Reproduction Assessment

Not applicable: this is a defense-in-depth feature proposal rather than a report of broken behavior with a defined current contract. Source inspection does confirm the architectural gap between existing name-based controls and the requested capability grants.

## Solution Assessment

Yes directionally, but not yet as a single implementation change: a staged core/plugin-API design is the maintainable path, starting with a maintainer-approved vocabulary and diagnostics before compatibility-sensitive grants and enforcement.

## Risks / Open Questions

- The issue overlaps narrower open declaration proposals; implementing them independently could create incompatible skill, plugin, and tool capability vocabularies.
