## Summary

Keep open. Current main has adjacent controls, but it still lacks the requested manifest-declared capability grants, default-deny pre-exposure filtering, cross-surface authorization, and durable high-risk capability audit records.

## What This Changes

The issue requests a capability-based permission model for skills and tools with manifest declarations, explicit grants/revocations, default-deny high-risk actions, consent UI/CLI, and audit logging.

## Best Possible Solution

Stage this as core/plugin-API security hardening: schema and diagnostics first, then persisted grants/revocations, CLI/UI/doctor visibility, pre-exposure filtering, invocation-boundary checks, and durable audit records.

## Reproduction Assessment

Not applicable; this is a security-hardening feature rather than a current-main bug. Source inspection of manifest, skill metadata, tool policy, gateway HTTP, MCP, docs, tests, and discussion shows the requested grant model is still absent.

## Solution Assessment

Yes, keeping this open for staged maintainer design is the best current path. Existing name-based policies, before_tool_call hooks, approvals, install scanning, and HTTP deny lists are useful foundations but do not replace a static capability grant layer enforced before exposure and again at invocation boundaries.
