## Summary

Current main and the latest release have adjacent tool-name policies, hooks, approvals, and HTTP-specific denies, but still lack declared sensitive capabilities, operator grants, consistent execution-boundary authorization, and durable capability audit records. The related open items cover narrower manifest and tool-metadata pieces rather than the complete security boundary requested here.

## What This Changes

Requests capability-based permissions and default-deny enforcement for high-risk skill and tool actions.

## Best Possible Solution

Approve a staged design with one small capability taxonomy and one canonical authorization decision reused by manifest diagnostics, persisted grants, model exposure, direct invocation checks, approval UX, doctor, and durable SQLite audit records, while leaving contextual governance to plugins.

## Reproduction Assessment

Yes at the source-contract level: current main demonstrably lacks sensitive capability declarations and operator grants. This is a feature gap rather than a failing existing-behavior reproduction.

## Solution Assessment

The defense-in-depth objective is appropriate, but the complete proposal is not ready for autonomous implementation; the maintainable path is a staged extension of the existing tool-policy, hook, approval, manifest, and direct-invocation infrastructure.
