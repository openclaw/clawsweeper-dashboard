## Summary

Keep open. Current main has useful adjacent controls, but it still does not provide the requested declared capability manifest, explicit grant/revoke state, pre-model default-deny filtering, cross-surface execution checks, or durable high-risk capability receipts for skills/tools.

## What This Changes

Request a capability-based permission model for skills/tools with declared high-risk capabilities, explicit grants and revocation, default-deny behavior, and audit logging.

## Best Possible Solution

Keep this as a core/plugin-API security hardening item and split it into manifest schema/diagnostics, persisted grants/revocations, pre-model filtering after tool-policy resolution, execution-boundary checks, and durable capability audit receipts coordinated with related skill-manifest and MCP deny-list work.

## Solution Assessment

Unclear whether the exact taxonomy and grant storage proposed here are the final contract. Keeping this open for security/API design is the right path because existing name-based policies, before_tool_call hooks, and HTTP deny-listing are not equivalent to first-class capability grants.
