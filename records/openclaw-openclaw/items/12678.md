## Summary

Keep open: current main and v2026.6.6 still have adjacent name-based tool policy, hook-time approvals, and HTTP deny lists, but not the requested capability manifest, explicit grant/revoke state, default-deny pre-exposure filtering, cross-surface execution checks, or durable capability audit trail.

## What This Changes

The issue requests a capability-based permission and audit model for skills/tools, with high-risk actions default-denied unless explicitly granted.

## Best Possible Solution

Design one versioned authorization-capability contract, then phase it through manifest/schema diagnostics, operator grant persistence, pre-exposure filtering, shared invocation checks, and SQLite-backed audit records.

## Reproduction Assessment

Yes at source level: current manifest, skill metadata, tool-policy, hook, gateway, docs, and release-tag contracts can be inspected and show the requested scoped grants and enforcement are absent. A failing runtime reproduction is not applicable because this is a defense-in-depth feature request rather than a broken existing contract.

## Solution Assessment

Unclear as a final implementation path: the layered direction is the right end state, but it should not land as one large or taxonomy-unapproved PR. The best path is maintainer-approved capability vocabulary, grant storage, compatibility policy, enforcement boundaries, and audit design before implementation.
