## Summary

Keep open. Current main has useful adjacent controls, and the @vainplex/openclaw-governance plugin is relevant prior art, but OpenClaw still lacks the requested manifest-declared capability grant layer with persisted grants, pre-model default-deny filtering, cross-surface invocation enforcement, and durable high-risk audit records.

## What This Changes

Request to add capability-based permissions for skills/tools, including manifest declarations, operator grants/revocations, default-deny exposure filtering, execution-boundary checks, and audit logs.

## Best Possible Solution

Keep this as a staged core/plugin-API security hardening effort: schema diagnostics first, then persisted grants, CLI/UI/doctor visibility, pre-exposure filtering, execution-boundary checks, and durable audit records.

## Reproduction Assessment

Not applicable; this is a feature/security-hardening request rather than a bug report. The high-confidence verification path is source inspection of current manifest, skill metadata, tool-policy, gateway, MCP, hook, and audit surfaces, which shows the requested grant model is still absent.

## Solution Assessment

Yes, keeping this open for staged maintainer design is the best current path. Existing name-based tool policies, before_tool_call hooks, plugin approvals, plugin install scanning, security audit checks, and HTTP deny lists are useful foundations but do not replace a static capability grant layer enforced before exposure and again at invocation boundaries.
