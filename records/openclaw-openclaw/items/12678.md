## Summary

Keep open. Current main has adjacent name/profile tool policies, before_tool_call governance, plugin install scanning, exec approvals, and an HTTP-only dangerous-tool deny list, but it still lacks the requested manifest-declared sensitive capabilities, persisted grant/revoke state, pre-model default-deny filtering, cross-surface authorization, and durable high-risk capability audit records.

## What This Changes

The issue requests a capability-based permission model for skills and tools with default-deny handling for high-risk actions.

## Best Possible Solution

Stage this as core/plugin-API security hardening: schema and diagnostics first, then grant state, CLI/UI/doctor visibility, pre-exposure filtering, invocation-boundary checks, and durable audit records.

## Reproduction Assessment

Not applicable; this is a security-hardening feature request rather than a current-main bug. Source inspection gives a high-confidence path showing the requested capability grant model is still absent.

## Solution Assessment

Yes, keeping this open for staged maintainer security/API design is the best current path. Existing tool policies, hooks, approvals, install scanning, and HTTP deny lists are useful foundations but do not replace a static grant layer enforced before exposure and again at invocation boundaries.
