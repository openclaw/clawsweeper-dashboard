## Summary

Keep open. Current main has adjacent controls, including tool-name policies, before_tool_call hooks, exec approvals, plugin install/security scanning, and an HTTP tool deny list, but it still lacks the requested manifest-declared capability grant layer with persisted grants, pre-exposure default-deny filtering, cross-surface invocation enforcement, and durable high-risk capability audit records.

## What This Changes

Request capability-based permissions for skills/tools with manifest declarations, default-deny grants, consent/revocation UX, and audit logging.

## Best Possible Solution

Keep this as a staged core/plugin-API security hardening effort: schema diagnostics first, then persisted grants, CLI/UI/doctor visibility, pre-exposure filtering, invocation-boundary checks, and durable audit records.

## Reproduction Assessment

Not applicable; this is a feature/security-hardening request rather than a bug report. The high-confidence verification path is source inspection of current manifest, skill metadata, tool-policy, gateway, MCP, hook, and audit surfaces, which shows the requested grant model is still absent.

## Solution Assessment

Yes, keeping this open for staged maintainer design is the best current path. Existing name-based policies, before_tool_call governance, install scanning, approvals, and surface-specific deny lists are useful foundations but do not replace a static grant layer enforced before exposure and again at invocation boundaries.
