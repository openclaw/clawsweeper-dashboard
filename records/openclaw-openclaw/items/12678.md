## Summary

Keep open: current main and v2026.6.10 still provide adjacent controls, but not the requested manifest-declared capabilities, explicit grant/revoke state, pre-model default-deny filtering, cross-surface authorization, or durable high-risk capability audit records. The linked governance plugin and related permission-manifest issues are useful prior art, not replacements for the missing core/plugin-API grant layer.

## What This Changes

The issue requests a capability-based permission model for skills and tools with manifest declarations, explicit grants/revocations, default-deny high-risk actions, and audit records.

## Best Possible Solution

Keep this as the core/plugin-API security-hardening umbrella and land a maintainer-approved phased design for capability schema, grants, filtering, execution checks, and durable audit records coordinated with the narrower RFCs.

## Reproduction Assessment

Yes, as source-level gap proof rather than a runtime bug reproduction: current main and v2026.6.10 expose name/profile policies, call-time hooks, and surface deny-lists, but no manifest plus grant/revoke capability enforcement model.

## Solution Assessment

Unclear as a final implementation path: the direction addresses a real security-hardening gap, but maintainers still need to decide taxonomy, grant storage, upgrade behavior, and execution-boundary semantics before code should land.
