## Summary

Current main has substantial adjacent controls—tool-name/profile policies, pre-call plugin hooks with approval support, sandboxing, and gateway/MCP scoping—but still lacks a canonical sensitive-capability declaration, operator grant/revoke state, default-deny pre-exposure filter, complete execution-boundary authorization, and durable capability-decision audit trail. The governance plugin linked in discussion is valuable working prior art for contextual call-time policy, but its author and the issue discussion correctly distinguish that from the missing static core/plugin-API grant layer; the issue remains necessary and requires maintainer security/product design rather than an autonomous fix PR.

## What This Changes

Add capability-based permissions for skills and tools that default-deny high-risk actions unless explicitly granted, support revocation, enforce decisions across exposure and execution paths, and record durable audit events.

## Best Possible Solution

Keep this as the canonical end-to-end authorization design item, explicitly coordinate its manifest portion with https://github.com/openclaw/openclaw/issues/12219 and its descriptive metadata portion with https://github.com/openclaw/openclaw/issues/48510, then phase work into a generic schema/diagnostics contract, operator grant persistence and doctor support, pre-exposure filtering, shared execution-boundary enforcement, and SQLite-backed audit records.

## Reproduction Assessment

Yes at source level: current manifest, tool-policy, hook, and gateway contracts can be inspected with high confidence to show that scoped sensitive grants and their cross-surface enforcement are absent. A failing runtime reproduction is not applicable because this is a defense-in-depth capability request rather than a reported broken contract.

## Solution Assessment

The proposed layered direction is the right architectural goal, but one large implementation is not the best delivery shape. The maintainable path is a maintainer-approved generic contract followed by small phases that reuse current tool-policy and hook pipelines while enforcing the same grant at every invocation boundary.

## Risks / Open Questions

- The issue overlaps the open manifest proposal at https://github.com/openclaw/openclaw/issues/12219 and descriptive tool-capability proposal at https://github.com/openclaw/openclaw/issues/48510; without one agreed taxonomy and ownership boundary, parallel incompatible public contracts could emerge.
- Grant storage, default behavior, and revocation are compatibility-sensitive config or persisted-state decisions. Changing existing users from permissive behavior to fail-closed behavior requires an explicit upgrade policy, migration/doctor plan, and fresh-install versus upgrade proof.
- Partial enforcement would be unsafe: filtering model-visible tools without enforcing the same decision at HTTP, MCP, plugin, skill-runtime, and other direct invocation boundaries would leave bypass paths.
- The existing term “capability” already describes plugin feature-provider availability, so authorization capabilities need precise naming or a clearly separate contract to avoid API confusion.
