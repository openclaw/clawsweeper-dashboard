## Summary

Current main still lacks the requested unified capability-grant layer: existing controls are tool-name/profile policies, sandboxing, approvals, hooks, and surface-specific restrictions rather than manifest-declared capabilities with persisted grants, pre-exposure filtering, execution-boundary authorization, and durable capability audit records. The discussion correctly identifies the governance plugin and narrower related issues as useful prior art, while the remaining core/plugin-API security architecture still needs explicit product and security decisions.

## What This Changes

Define manifest-declared capabilities for skills and tools, require explicit operator grants for high-risk actions, enforce them before model exposure and at invocation boundaries, and record durable audit events.

## Best Possible Solution

Design one canonical capability decision shared by plugin and skill metadata, persisted operator grants, existing tool-policy resolution, every execution boundary, doctor/CLI/UI diagnostics, and SQLite-backed audit records, with a staged compatibility plan for existing tools.

## Solution Assessment

Unclear: the proposed capability model is directionally appropriate, but maintainers must first choose the taxonomy, grant persistence, compatibility rollout, and how it composes with existing tool policy, sandbox, approvals, hooks, HTTP, and MCP boundaries.
