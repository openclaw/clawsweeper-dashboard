## Summary

Current main has substantially stronger adjacent controls—including name/profile-based tool policies, Gateway surface denies, trusted pre-tool policy contracts, hooks, approvals, and sandboxing—but still lacks operator-owned capability declarations, persisted grants/revocations, capability-based pre-exposure filtering, unified execution-boundary authorization, and durable capability audit records. The plugin governance implementation and narrower manifest RFCs are useful prior art, not replacements for the remaining cross-surface security contract.

## What This Changes

Introduce capability-based permissions for skills and tools with declarations, explicit grants and revocations, default-deny sensitive actions, cross-surface enforcement, consent UX, and durable audit records.

## Best Possible Solution

Approve a staged generic plugin/API contract that carries one canonical capability decision through existing policy resolution, persists operator grants and audit records in the appropriate SQLite store, exposes consent and diagnostics, and rechecks the same decision at every invocation boundary.

## Reproduction Assessment

Not applicable: this issue proposes a new defense-in-depth authorization model rather than reporting a reproducible failure of an existing documented contract. Source inspection does provide high confidence that the requested grant and cross-surface enforcement layer remains absent.

## Solution Assessment

Yes at the security-invariant level, but the best implementation is not one large parallel permission framework; it should be a staged generic contract integrated with existing tool policy, trusted-policy, approval, sandbox, and invocation paths.
