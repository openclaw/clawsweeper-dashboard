## Summary

Current main and v2026.6.6 provide adjacent tool-name policies, hooks, approvals, install scanning, and HTTP-specific denies, but still do not provide sensitive capability declarations, operator grants/revocations, shared pre-exposure and execution-boundary authorization, or durable capability audit records. The related open issues cover narrower manifest and tool-metadata pieces, while the referenced MCP deny-list PR closed unmerged, so this issue remains the canonical broader security architecture item.

## What This Changes

Introduce capability-based permissions for skills and tools, including declared sensitive capabilities, explicit grants and revocations, default-deny enforcement, consent controls, and durable audit records.

## Best Possible Solution

Keep this as the canonical core/plugin-API security design item and implement it in staged contracts: capability taxonomy and manifest parsing, persisted grants with doctor/CLI/UI visibility, pre-model filtering, shared execution-boundary checks, then durable audit records and optional dynamic policy hooks.

## Reproduction Assessment

Yes at source level: current main and v2026.6.6 can be inspected to show activation metadata, name/profile policies, hooks, approvals, and surface-specific denies without a unified sensitive capability declaration, grant, enforcement, and audit contract. This is a feature-gap verification rather than a failing bug reproduction.

## Solution Assessment

Yes in architectural direction, but not as one large patch. The narrow maintainable path is staged generic contracts after maintainers choose the taxonomy, persisted grant owner, compatibility posture, and composition with existing policy and approval layers.
