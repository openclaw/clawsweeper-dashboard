## Summary

Keep open. Current main still has a source-reproducible precedence bug where persisted or caller-supplied 272k context caps can beat explicit provider/model `contextWindow` or `contextTokens` config in status and `session_status`; Copilot-wide implicit 1M promotion remains a separate provider-policy question.

## What This Changes

The issue asks OpenClaw to report 1M context for `github-copilot/claude-opus-4.x-1m` models, especially when users explicitly configure `contextWindow` or `contextTokens`, instead of showing a stale 272k cap.

## Best Possible Solution

Honor explicit provider/model `contextWindow` and `contextTokens` before stale persisted or discovered caps across resolver and status surfaces, while leaving Copilot-wide 1M auto-promotion to a separate provider-policy decision.

## Reproduction Assessment

Yes, source-reproducible: a session with `entry.contextTokens` at 272000 and explicit `github-copilot` model config at 936000/1000000 will pass the persisted value as `contextTokensOverride`, and current main returns it before config lookup. I did not run a live Copilot session in this read-only review.

## Solution Assessment

Yes for the explicit-configuration precedence repair: the narrow maintainable fix is to make explicit provider/model config beat stale persisted caps in status and `session_status`. Broad Copilot name-based 1M promotion should remain out of scope unless maintainers choose that provider policy separately.
