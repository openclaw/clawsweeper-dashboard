## Summary

Keep open. Current `origin/main` still gates Anthropic 1M context promotion to `anthropic` and `claude-cli`, tests deliberately preserve non-promotion for GitHub Copilot Opus 4.7 IDs, and status rendering can still prefer a persisted smaller runtime cap over explicit provider-model configuration. The reporter briefly said it was filed by mistake, but then reopened it, so there is no strong cleanup close reason.

## What This Changes

The issue requests that `github-copilot` Claude Opus 4.x 1M variants honor a 1M context window, either through model-name recognition, explicit `context1m`, or user-configured `contextWindow` / `contextTokens` precedence.

## Best Possible Solution

Make explicit provider-model `contextWindow` / `contextTokens` authoritative over discovered and persisted smaller caps for the same selected model in runtime resolution and status/session rendering, with focused Copilot regression tests. Treat automatic Copilot `*-1m` promotion or provider-gate removal as a separate maintainer decision backed by verified Copilot API behavior and docs.

## Reproduction Assessment

Yes for a high-confidence source-level/unit reproduction: configure a `github-copilot` Claude Opus model with a 1,000,000-token provider-model window and combine it with discovered or persisted 272k `contextTokens`; current status/runtime paths can still surface the smaller cap. I did not run a live Copilot end-to-end request in this read-only review.

## Solution Assessment

Unclear for broad automatic Copilot 1M promotion, but yes for the narrow config-precedence fix. The safest maintainable path is to honor explicit user/provider-model context configuration first, while leaving the `anthropic` / `claude-cli` promotion gate in place until Copilot long-context support is verified.
