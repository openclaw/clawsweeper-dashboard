## Summary

Keep this issue open. Current main still deliberately restricts Anthropic 1M promotion to `anthropic` and `claude-cli`, and the status path can still let a persisted 272k session cap override an explicit configured provider-model context value for the same selected GitHub Copilot model. The safe repair is the explicit-configuration precedence bug; automatic Copilot 1M promotion needs maintainer/provider confirmation first.

## What This Changes

The issue reports that `github-copilot/claude-opus-4.x-1m` models show a 272k context window despite 1M context configuration, and asks for configured context precedence or provider/model-based 1M promotion.

## Best Possible Solution

Make explicit `models.providers.github-copilot.models[]` `contextWindow` and `contextTokens` authoritative in model resolution and status/session rendering for the same selected model, including when the session already persisted a smaller discovered cap. Treat Copilot `*-1m` auto-promotion or provider-gate removal as a separate maintainer policy decision after verifying Copilot's long-context support.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a `github-copilot/claude-opus-4.x-1m` model with 1M context and render status for a session entry that already persisted `contextTokens: 272000`; current main can report the persisted 272k cap before the explicit provider-model config is consulted.

## Solution Assessment

Unclear for broad automatic Copilot 1M promotion, because docs and tests currently keep that provider conservative. Yes for the narrow repair: explicit provider-model context configuration should win over discovered or persisted caps for the same selected model.
