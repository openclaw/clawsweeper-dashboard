## Summary

Keep open. Current main still has a source-reproducible precedence bug where persisted or caller-supplied context token caps can beat explicit provider-model contextWindow/contextTokens config in status/session_status paths; Copilot-wide implicit 1M promotion remains a separate provider-policy decision.

## What This Changes

Request to have GitHub Copilot Claude Opus 1M variants report/use their 1M context window, especially when configured with per-model contextWindow/contextTokens.

## Best Possible Solution

Honor explicit provider-model contextWindow/contextTokens before stale persisted/discovered caps across resolver and status surfaces, while leaving broad Copilot 1M auto-promotion to separate provider-policy review.

## Reproduction Assessment

Yes, source-reproducible: a session with entry.contextTokens 272000 and explicit github-copilot model config at 936000/1000000 will pass the persisted value as contextTokensOverride, and the resolver returns it before config lookup. I did not run a live Copilot session in this read-only review.

## Solution Assessment

Yes for the explicit configuration precedence repair. The narrow maintainable fix is to make explicit provider-model config beat stale persisted caps in status/session_status without changing Copilot-wide 1M promotion policy.
