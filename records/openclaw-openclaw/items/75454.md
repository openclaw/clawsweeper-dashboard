## Summary

Keep this issue open. Current main still has a real source-level path where a persisted 272k session context cap can override an explicit provider-model context configuration in status rendering, while automatic GitHub Copilot 1M promotion remains intentionally provider-gated and covered by tests/docs as Anthropic/Claude CLI-only behavior.

## What This Changes

The issue requests that GitHub Copilot Claude Opus 4.x 1M models use a configured 1M context window, allow explicit 1M opt-in, or otherwise avoid reporting a 272k cap in status output.

## Best Possible Solution

Make explicit provider-model `contextWindow` or `contextTokens` configuration authoritative for the selected model through resolver and status/session rendering, including when a session already persisted a smaller runtime cap. Treat automatic GitHub Copilot `*-1m` promotion or removal of the Anthropic/Claude CLI provider gate as a separate maintainer policy decision after the Copilot long-context contract is verified.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `github-copilot/claude-opus-4.x-1m` with a 1M provider-model context value and render status for a same-model session entry that already persisted `contextTokens: 272000`; current `buildStatusMessage` can pass that persisted value as an override before config lookup.

## Solution Assessment

Unclear for the broad auto-promotion request, but clear for the explicit configuration precedence bug. The narrowest maintainable repair is to make explicit provider-model context config win over persisted or discovered caps for the same selected model, while leaving Copilot 1M auto-promotion behind maintainer approval.
