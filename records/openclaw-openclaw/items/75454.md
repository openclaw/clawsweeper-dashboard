## Summary

Keep open. Current main still hard-limits Claude 1M context promotion to `anthropic` and `claude-cli`, and the existing tests explicitly preserve non-promotion for `github-copilot` Claude Opus IDs. The report also points at a real-looking configuration precedence seam: documented provider-model context overrides should win, but merged/discovered `contextTokens` can still undercut a configured `contextWindow` unless the full config shape is handled carefully.

## What This Changes

The issue requests that `github-copilot/claude-opus-4.x-1m` models resolve to a 1M context window instead of the discovered 272k cap, either by name/opt-in recognition or by honoring configured context limits over discovered metadata.

## Best Possible Solution

Make explicit provider-model `contextWindow` / `contextTokens` authoritative across model merge, runtime context resolution, and status/session reporting. Keep automatic Copilot `*-1m` or `context1m` promotion as a separate provider policy decision unless maintainers verify that Copilot accepts those long-context aliases reliably.

## Reproduction Assessment

Partially. Current main gives a high-confidence source-level reproduction for Copilot non-promotion, and the merge/runtime precedence path can explain a discovered 272k value winning over a configured 1M window; the exact reported `session_status` output still needs the full config and session-store state.

## Solution Assessment

Unclear for the broad provider-policy request. Dropping all provider gates or trusting every `*-1m` suffix is broader than current docs/tests support; the narrowest maintainable first fix is to enforce explicit config precedence and add focused Copilot/status regression tests.
