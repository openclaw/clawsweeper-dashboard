## Summary

Keep open. Current main still excludes GitHub Copilot from the explicit and implicit Claude 1M promotion paths, and several runtime/status paths can still let discovered or persisted `contextTokens` undercut an explicit configured window. The narrow repair is explicit configured context precedence; automatic Copilot long-context promotion remains a provider-policy decision.

## What This Changes

The issue requests that GitHub Copilot Claude Opus 4.x 1M variants honor a 1M context limit, especially when provider-model config sets `contextWindow`/`contextTokens` or `context1m` is explicitly enabled.

## Best Possible Solution

Make explicit provider-model `contextWindow` / `contextTokens` authoritative across model merge, runtime context resolution, persisted session usage, and status/session reporting. Keep automatic Copilot `*-1m` or `context1m` promotion as a separate provider policy decision unless maintainers verify that Copilot reliably supports those long-context aliases.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially yes: current source and tests show Copilot 1M non-promotion and status/runtime precedence paths that can preserve 272k over a configured window, but the exact `session_status` string needs the reporter's full config and session-store state.

## Solution Assessment

Is this the best way to solve the issue? Unclear for broad provider auto-promotion; dropping provider gates or trusting every `*-1m` suffix is broader than current docs/tests support. The best first fix is the narrow explicit-config precedence repair with focused Copilot/status regression coverage.
