## Summary

Keep open. Current main has partial protection for configured context windows, but the user-visible status paths can still prefer a persisted 272k session cap before the configured `github-copilot` model window, and Copilot remains excluded from the Anthropic/Claude CLI 1M promotion gates.

## What This Changes

The issue asks OpenClaw to report/use a 1M context window for `github-copilot` Claude Opus 4.x 1M models, or at least preserve explicit `contextWindow`/`contextTokens` config over provider-discovered 272k caps.

## Best Possible Solution

Make explicit per-provider model `contextWindow` or `contextTokens` authoritative in resolver, `status`, `session_status`, and status summary rendering, while leaving Copilot automatic 1M promotion behind a separate provider-contract decision.

## Reproduction Assessment

Yes. Configure `github-copilot/claude-opus-4.x-1m` with a 1M provider-model context value and inspect a same-model session row that already persisted `contextTokens: 272000`; current main routes that persisted value through `contextTokensOverride` before config lookup.

## Solution Assessment

Yes for the explicit configuration precedence part; unclear for automatic Copilot 1M promotion. The narrow maintainable fix is to honor explicit config over persisted/discovered caps without changing the Anthropic/Claude CLI provider gate.
