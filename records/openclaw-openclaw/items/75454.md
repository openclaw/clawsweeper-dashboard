## Summary

Keep open. Current main still lets a caller or persisted `contextTokensOverride` win before explicit provider-model context configuration, so a saved 272k Copilot cap can mask a configured 1M window; the broader Copilot auto-promotion request remains an intentional provider-policy question rather than an obvious cleanup close.

## What This Changes

Honor 1M context limits for `github-copilot/claude-opus-4.x-1m` models, especially when user-configured `contextWindow` or `contextTokens` specifies 1M but status reports 272k.

## Best Possible Solution

Keep this issue open and land a narrow fix that makes explicit provider-model `contextWindow` or `contextTokens` authoritative over stale persisted status caps, while deferring Copilot-wide implicit 1M promotion to a separate maintainer decision.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a `github-copilot` Claude Opus 1M session with explicit 1M provider-model config and a persisted `entry.contextTokens: 272000`; current status paths pass that persisted value as an override before config lookup.

## Solution Assessment

Yes for the explicit configuration precedence repair; unclear for broad Copilot 1M auto-promotion. The safer maintainable fix is to honor operator config over stale persisted/discovered caps without changing the Anthropic/Claude CLI provider gate.
