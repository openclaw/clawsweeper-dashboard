## Summary

Keep open. Current main still has a source-level path where status and status summary can prefer a persisted session context cap before checking explicit provider-model context configuration, so a previously stored 272k GitHub Copilot cap can survive despite a configured 1M window. Existing source, tests, and docs also keep automatic Opus 1M promotion scoped to Anthropic and Claude CLI, so the safe repair is explicit configuration precedence rather than broad Copilot auto-promotion.

## What This Changes

The issue requests that GitHub Copilot Claude Opus 1M models report and use the configured 1M context window instead of being capped by discovered or persisted 272k metadata.

## Best Possible Solution

Make explicit provider-model `contextWindow` or `contextTokens` configuration authoritative for the selected model across resolver, `session_status`, and CLI status/session-summary rendering, including when a session already persisted a smaller runtime cap. Leave GitHub Copilot 1M auto-promotion behind a separate maintainer decision unless the Copilot long-context contract is verified and documented.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `github-copilot/claude-opus-4.x-1m` with 1M provider-model context values and render status for a same-model session entry that already persisted `contextTokens: 272000`; current status code can pass that persisted value as an override before config lookup.

## Solution Assessment

Yes for the explicit configuration precedence bug, but unclear for broad Copilot auto-promotion. The narrowest maintainable fix is to make explicit provider-model context config win over persisted/discovered caps for the selected model while leaving provider-gated 1M inference unchanged until maintainers approve it.
