## Summary

Keep open. Current main has meaningful Active Memory timeout mitigations, but it still intentionally treats `modelFallback` as model-selection fallback only and still injects no memory when the selected recall model times out without partial text; the latest discussion also includes a v2026.4.29 MiniMax-M2.7 confirmation and related timeout reports remain open.

## What This Changes

The issue reports Active Memory timing out with MiniMax-M2.7, returning no memory summary, and asks for successful completion or fallback behavior when the recall model is slow.

## Best Possible Solution

Keep this open until maintainers either make a canonical Active Memory timeout issue for the MiniMax/provider-latency cluster or deliberately implement a runtime policy for slow recall models, such as bounded fail-open, local `memory_search` summary recovery, or a new explicit timeout-fallback model setting that does not overload the current selection-only `modelFallback`.

## Reproduction Assessment

Yes. Configure Active Memory to use `minimax/MiniMax-M2.7` or `minimax-portal/MiniMax-M2.7`, enable logging, and watch for `status=timeout` or hook-timeout logs with `summaryChars=0`; multiple users provided matching logs, including on v2026.4.29.

## Solution Assessment

No. Current main is a partial mitigation: it extends the Active Memory hook budget, passes a hard timeout to the embedded run, and disables recall reasoning by default, but it does not complete recall or switch to another/local fallback when MiniMax times out without partial text.
