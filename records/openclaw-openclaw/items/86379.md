## Summary

Keep this PR open: the bug target is valid and focused, but the current head does not remove `include` when filtering leaves an empty array, so the common retry request shape remains effectively unchanged. It also still lacks after-fix real behavior proof from the reported Codex OAuth setup.

## What This Changes

The PR modifies `stripResponsesRequestEncryptedContent()` so invalid encrypted-content retries attempt to remove `reasoning.encrypted_content` from OpenAI Responses `include` arrays.

## Best Possible Solution

Fix the sanitizer to remove or undefine `include` when filtering leaves no entries, add focused regression coverage for include-only and mixed-include requests, then land with redacted live Codex Responses retry proof.

## Reproduction Assessment

Yes for the patch defect: current source generates `include: ["reasoning.encrypted_content"]`, and PR head filters that to an empty array but then keeps the original include via `...params`. The original live retry storm is supported by issue logs, but the PR still lacks after-fix live reproduction proof.

## Solution Assessment

No; removing the include value on encrypted-content retry is the right direction, but this implementation fails to delete it in the common one-item case. The safer fix is to explicitly clear `include` when empty and cover that sanitizer behavior in tests.
