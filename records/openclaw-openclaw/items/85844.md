## Label Justifications

- `impact:message-loss`: Stale Codex harness imports can prevent expected agent output from completing or being delivered after update/restart recovery.

## Summary

Keep open: current main has managed-service update handoff and restart safeguards, but the latest v2026.6.6 retest still reports interrupted Codex recovery compaction carrying an old hashed `compact` import after upgrade/restart. The source still has the same lazy Codex harness import boundary, so the exact user problem is not proved fixed.

## What This Changes

The issue asks OpenClaw auto-update and restart recovery to avoid leaving a running or recovered Codex gateway turn with stale content-hashed bundle imports.

## Best Possible Solution

Keep the managed-service handoff as the default and add packaged upgrade/restart-recovery proof plus a durable guard chosen by maintainers: preserve old chunks until exit, restart or block recovery before stale imports run, or verify and repair stale Codex dynamic import targets before resuming turns.

## Reproduction Assessment

No live current-main reproduction was run, but the source path is high-confidence: Codex harness lazy-imports `run-attempt` and `compact`, restart recovery resumes interrupted sessions, and the latest v2026.6.6 retest reports the same stale `compact` import shape.

## Solution Assessment

Unclear until maintainers choose the upgrade policy. A targeted compatibility alias may help one release, but the maintainable fix needs the update/restart recovery boundary to prevent stale hash imports generically.
