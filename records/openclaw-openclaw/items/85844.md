## Summary

Keep open: current main and v2026.5.20 have broad package-update stop/handoff protections, but the exact stale Codex `run-attempt` chunk failure is not proved fixed or superseded, and the same reporter has a paired open consolidation issue that explicitly keeps this issue as a related subcase.

## What This Changes

The issue reports that auto-update can leave a live gateway using cached Codex bundle modules that import removed content-hashed chunk files until the gateway is restarted.

## Best Possible Solution

Keep the managed-service restart/handoff path as the primary contract, then add focused Codex stale-chunk coverage or a one-time legacy alias only if maintainers confirm that this shipped upgrade path should recover without manual restart.

## Reproduction Assessment

No high-confidence current-main reproduction was established. Source inspection shows the plausible stale-import path through the Codex harness dynamic `run-attempt` import and adjacent update safeguards, but the exact old hash and auto-update path are missing.

## Solution Assessment

Unclear. Current restart/handoff safeguards are in the right area, but maintainers still need to choose whether the remaining best fix is a targeted Codex legacy alias, stronger update integrity checking, or enforcing restart/reexec for this upgrade path.
