## Label Justifications

- `impact:message-loss`: Stale Codex harness imports can prevent expected agent output from completing or being delivered after update/restart recovery.

## Summary

Keep open: current main has important managed-service update handoff safeguards, but the latest v2026.6.6 retest and current Codex lazy-import boundaries do not prove the interrupted recovery/compaction stale-hash failure is fixed.

## What This Changes

Issue reports that OpenClaw auto-update can leave a running gateway or recovered Codex turn carrying stale content-hashed bundle imports after package replacement.

## Best Possible Solution

Keep the managed-service handoff as the default and add maintainer-chosen packaged upgrade/restart-recovery hardening: preserve old chunks until exit, block stale recovery before imports run, or verify and repair stale Codex dynamic import targets before resuming turns.

## Reproduction Assessment

No live current-main reproduction was established in this read-only review. Source inspection plus the v2026.6.6 retest gives a high-confidence failure shape: Codex harness lazy imports can still be reached during interrupted recovery/compaction after package tree rotation.

## Solution Assessment

Unclear until maintainers choose the upgrade policy. A one-off chunk alias or reply-loader warning is not the best durable fix; the recovery/update boundary needs to prevent stale hashed imports generically.
