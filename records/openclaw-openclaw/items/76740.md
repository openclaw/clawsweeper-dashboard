## Summary

This should stay open: current main still has a source-reproducible WebChat reconciliation gap where a live-only assistant reply can be dropped once a later history reload includes a newer user message, and the message_tool path still needs durable exact-session transcript coverage.

## What This Changes

Keep assistant replies sent through `messages.visibleReplies: "message_tool"` visible and persistent in the WebChat/Control UI transcript after later turns, with a reporter follow-up that automatic mode may share the same symptom.

## Best Possible Solution

Persist each user-visible assistant reply into the exact active session transcript once, including successful message-tool sends, and make WebChat reconciliation tolerate temporarily stale history without deleting visible assistant rows before canonical history catches up.

## Reproduction Assessment

Yes, source-reproducible: with UI state `[user, live assistant, next user]` and a `chat.history` snapshot `[user, next user]`, `preserveOptimisticTailMessages` finds the next user as the last shared entry and drops the missing assistant. The message_tool path can create that live assistant row through the `NO_REPLY` fallback and sent-text dedupe surfaces.

## Solution Assessment

Yes, the maintainable direction is to repair exact-session transcript persistence and reconciliation rather than changing `visibleReplies` policy or relying only on UI-local optimistic state. The automatic-mode claim should be verified before broadening the persistence fix beyond the proven message_tool/live-history path.
