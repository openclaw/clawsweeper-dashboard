## Summary

Keep this open: current main has an adjacent UI reload fix after the report, but the message-tool path still appears to keep delivered assistant text in live/optimistic UI state rather than durable `chat.history` transcript state, so older message-tool replies can still fall out after a later history refresh.

## What This Changes

The issue asks for message-tool-delivered agent replies to remain visible in WebChat/Control UI transcript history after later turns.

## Best Possible Solution

Persist successful message-tool sends as transcript-visible assistant output on the active session branch, using the session manager append path and keeping existing source-delivery dedupe behavior intact.

## Reproduction Assessment

Yes, source-reproducible: a message_tool reply is live-rendered from committed message-tool text, but final tool-driven history reloads rebuild from transcript state where only a silent assistant entry may exist. I did not run a live Telegram/Discord reproduction in this read-only pass.

## Solution Assessment

Yes, the narrowest maintainable fix is transcript persistence for successful message-tool sends, not changing `visibleReplies` policy, disabling dedupe, or relying on optimistic UI preservation.
