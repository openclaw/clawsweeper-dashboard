This should stay open: current main still has a source-reproducible transcript persistence gap for at least the Telegram message-tool path, and the reporter’s automatic-mode follow-up needs coverage before the issue can close.

## What This Changes

The issue reports channel-delivered assistant replies disappearing from WebChat/Control UI history after a later turn reloads the transcript.

## Best Possible Solution

Mirror successful gateway-executed message-tool sends into the active session transcript, then add WebChat history regression coverage that also checks whether automatic-mode replies follow the same durable path.

Yes, source-reproducible for the Telegram `message_tool` path on current main: gateway-executed sends receive `sessionKey` but are not appended to the stored transcript before `chat.history` reloads. I did not perform a live channel run in this read-only pass.

## Solution Assessment

Yes for the titled path: the narrow maintainable fix is to mirror successful gateway `message.action` sends into the transcript rather than changing `visibleReplies` policy or UI filtering. The automatic-mode follow-up should be verified with a regression test and split only if it proves to be a distinct source path.
