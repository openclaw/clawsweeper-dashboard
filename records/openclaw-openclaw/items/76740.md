## Summary

This should stay open: current main still has source-reproducible paths where source replies can reach the live chat event stream but are not guaranteed to exist in the durable transcript that WebChat reloads after the next turn.

## What This Changes

Keep source-channel assistant replies visible in WebChat/Control UI history after later user turns in both `message_tool` and automatic visible-reply modes.

## Best Possible Solution

Persist successful visible source replies as assistant-visible entries on the active OpenClaw transcript branch, while keeping outbound delivery dedupe and `visibleReplies` policy unchanged.

## Reproduction Assessment

Yes, source-reproducible but not live-executed. The message-tool path stores successful sends for live output/dedupe while history reloads from persisted transcript rows, and the opencode-style CLI path emits live assistant text without a generic durable history import.

## Solution Assessment

Yes. The narrow maintainable fix is to write the already-visible assistant reply to the active session transcript using the existing SessionManager-safe path, not to change source delivery mode, duplicate suppression, or WebChat reload behavior.
