## Summary

Keep open. Current main still routes `api.runtime.subagent.run()` through the generic gateway `agent` RPC path, which tracks the run as a `cli` task and never registers a `SubagentRunRecord`; the `subagent_ended` hook remains tied to the subagent registry lifecycle used by `sessions_spawn`.

## What This Changes

The issue requests that plugin-created subagent runs via `api.runtime.subagent.run()` participate in the subagent registry lifecycle so `subagent_ended` hooks fire when those runs complete.

## Best Possible Solution

Add a narrow gateway/plugin-runtime bridge so `api.runtime.subagent.run()` registers plugin-created subagent runs with the existing subagent registry lifecycle, then add a focused regression test that proves a plugin hook registered for `subagent_ended` fires when the plugin runtime run completes. Keep the fix in the generic gateway/runtime/subagent seams rather than in any bundled plugin.

## Reproduction Assessment

Yes. The issue gives a concrete plugin-level reproduction, and current-main source inspection confirms the exact divergent paths: `api.runtime.subagent.run()` reaches generic `agent` dispatch while `sessions_spawn` alone registers a subagent run.

## Solution Assessment

Yes, the suggested direction is the maintainable shape: reuse `registerSubagentRun()` or a small wrapper around it for plugin runtime subagent runs so there is one completion lifecycle. The implementation should avoid duplicating `sessions_spawn` policy and announce behavior unless explicitly intended.
