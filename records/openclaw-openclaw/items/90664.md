## Summary

Current main and shipped release tags now implement the no-delivery/no-channel cron policy requested here, and regression coverage confirms that explicit no-delivery runs can prefer final assistant-visible text while unresolved delivery-requested runs stay conservative.

## What This Changes

The issue reports that explicit no-delivery isolated cron runs can misclassify recovered tool-warning outcomes as errors because an unresolved channel policy did not prefer final assistant-visible text.

## Best Possible Solution

Keep the shipped no-delivery/no-channel recovery implementation and track broader cron false-positive cases in the remaining open cron outcome issues.

## Reproduction Assessment

No current-main failing reproduction remains for this specific path: unresolved explicit no-delivery now returns final-text-preferred and the final/interim callers pass deliveryRequested. The historical bug is source-proven by the parent commit returning false for the same unresolved channel branch.

## Solution Assessment

Yes. The merged fix is the best narrow shape because it gates the no-channel true default on explicit no-delivery rather than making all unresolved channels prefer final assistant text.
