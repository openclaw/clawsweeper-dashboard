## Summary

Current main and the shipped 2026.6.6 release line now implement the reported no-delivery/no-channel cron classifier behavior, with focused regression tests and merged PR provenance.

## What This Changes

The issue reports that unresolved-channel `--no-deliver` isolated cron runs can misclassify recovered tool-warning completions as errors instead of using final assistant-visible text.

## Best Possible Solution

Keep the merged delivery-requested boundary for this no-channel subcase and leave broader recovered-tool-error classifier questions to their separate open reports.

## Reproduction Assessment

Yes. The source path and regression tests cover explicit no-delivery, unresolved channel, tool-warning payloads, and final assistant-visible text; no live cron dispatch was executed during this read-only review.

## Solution Assessment

Yes. The implemented fix is narrower than the originally suggested unconditional no-channel default because it prefers final text only when delivery was explicitly not requested, while keeping unresolved requested delivery conservative.
