## Summary

Keep open. Current main still has the stale-session failure mode described here: bundled MCP tool calls reuse the cached per-session SDK client and there is no catch path that detects an SSE lost-session error, recreates the server client/transport, refreshes the catalog, or retries the tool call. The adjacent PR #71545 is relevant context, but it is open, unmerged, and scoped to streamable-http rather than the SSE path in this report.

## What This Changes

OpenClaw should recover SSE-backed bundled MCP client sessions after a downstream MCP server restart instead of requiring a full gateway restart.

## Best Possible Solution

Add a bounded stale-session recovery path in the session MCP runtime: when an SSE-backed MCP tool call fails with a known lost-session response, dispose and recreate only that server's client/transport, rebuild or refresh the relevant catalog state, retry the original call once, and leave unrelated stdio behavior unchanged. The implementation should coordinate with #71545 if that PR lands first so both HTTP transports share the same recovery seam where practical.

## Reproduction Assessment

Yes. The issue gives a concrete restart-after-successful-call path, and current source confirms the runtime and client are reused across calls; a focused regression can extend the existing SSE probe harness to drop or reset the downstream session before the second call.

## Solution Assessment

No current solution is in main. The narrow maintainable fix belongs in the session MCP runtime and focused SSE runtime tests; the adjacent streamable-http PR is useful context but does not solve this SSE report as written.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/68187. Current main reuses session-scoped bundled MCP clients in src/agents/pi-bundle-mcp-runtime.ts, so an SSE-backed MCP server restart can leave the cached client pinned to stale session state and subsequent tool calls fail with errors such as `Session not found` until the gateway restarts. Add a bounded recovery path for SSE-backed bundled MCP tool calls: detect known lost-session responses from the SDK/transport, dispose and recreate only the affected server's MCP client and transport through the existing resolveMcpTransport/connectWithTimeout path, refresh or validate the affected catalog entry as needed, retry the original tool call exactly once, and preserve the original error if reconnect/retry cannot recover. Coordinate with PR #71545 if it has landed or changed shape; prefer a shared helper for lost-session detection/recreate logic when practical, but do not make this fix streamable-http-only. Do not change unrelated provider/channel behavior, do not add core owner-specific IDs, and do not hide genuine server-down errors behind repeated retries. Add focused regression coverage using the existing SSE MCP test harness, extending it if necessary to simulate a dropped/reset server-side SSE session before the second tool call. Add a user-facing CHANGELOG.md fix entry crediting the reporter with `Thanks @gwjr`.
