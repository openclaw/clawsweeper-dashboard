## Summary

Keep open. Current main still sends bundled MCP tool calls through the cached SSE session without catching stale-session POST failures, while the related open PRs cover streamable-http recovery and unexpected transport-close eviction rather than this default SSE `Session not found` path.

## What This Changes

The issue requests automatic recovery for SSE-backed MCP sessions after a downstream server restart instead of requiring an OpenClaw gateway restart.

## Best Possible Solution

Add bounded SSE stale-session recovery in the session MCP runtime: classify known lost-session POST errors, rebuild only the affected SSE client/transport, refresh catalog metadata when needed, and retry once while leaving stdio behavior unchanged.

## Reproduction Assessment

Yes, with source-level confidence: the issue gives a successful-call, downstream SSE restart, and second-call failure path, and current main confirms that path reaches a cached `session.client.callTool()` with no stale-session recovery. I did not run a live SSE MCP server in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is runtime-local SSE stale-session detection with one reconnect-and-retry, preferably sharing helper structure with the streamable-http and transport-close work if those PRs land first.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/68187. Current main caches bundled MCP SSE sessions and calls `session.client.callTool()` without stale-session recovery; after a downstream SSE MCP server restart, the pinned MCP SDK can surface `Error POSTing to endpoint ... Session not found` and OpenClaw keeps using the dead client until gateway restart. Implement a narrow SSE-only recovery path in `createSessionMcpRuntime`: classify specific lost-session POST errors, dispose/rebuild only the affected SSE client/transport with the existing custom schema validator and config resolution, refresh the affected catalog/tool metadata if needed, and retry the original call once. Leave stdio behavior unchanged, avoid broad HTTP 4xx false positives on tool-level errors, serialize reconnects if concurrent calls can fail together, and coordinate helper shape with https://github.com/openclaw/openclaw/pull/71545 and https://github.com/openclaw/openclaw/pull/73792 if either has landed. Add focused Vitest coverage for successful SSE recovery, failed reconnect preserving a useful error, and no retry for unrelated tool/application errors. Changelog is expected if the user-visible runtime fix lands; maintainers can adjust wording during landing.
