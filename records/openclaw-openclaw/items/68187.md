## Summary

Keep open: current main and the latest release still reuse cached MCP sessions for tool calls without a stale-session reconnect path, and the linked reconnect PRs are open candidate fixes rather than merged implementation. The recent root-cause comment adds useful scope beyond SSE, so this remains a live maintainer item.

## What This Changes

The issue asks OpenClaw to recreate stale MCP client/proxy sessions after an SSE-backed MCP server restart so later tool calls do not fail with `Session not found`.

## Best Possible Solution

Land one proof-backed MCP runtime recovery design that recreates only the stale per-server session, safely refreshes affected catalog state, retries at most once, and leaves broader catalog reload work to its narrower tracker.

## Reproduction Assessment

Yes. The source path is clear: establish an MCP runtime session, invalidate or restart the downstream MCP server, then `callTool()` reuses the cached session and propagates the SDK transport/session error without reconnecting.

## Solution Assessment

Unclear for the final patch shape. The agent MCP runtime is the right boundary, not the UI, but maintainers still need to choose transport scope, retry classification, catalog refresh behavior, and which open PR should become canonical.
