## Summary

Keep open. Current main still reuses the cached SSE-backed MCP client session without stale-session classification, reconnect, metadata refresh, or one-time retry; the only related open PR is scoped to streamable-http, not the default SSE path.

## What This Changes

The issue reports stale SSE-backed MCP client sessions after a downstream server restart, where calls fail with `Session not found` until the OpenClaw gateway restarts.

## Best Possible Solution

Add bounded SSE stale-session recovery in the session MCP runtime: classify known lost-session POST errors, rebuild only the affected SSE client/transport, refresh metadata if needed, and retry once while leaving stdio behavior unchanged.

## Reproduction Assessment

Yes, with source-level confidence: the issue gives a successful-call, downstream SSE restart, and second-call failure path, and current main confirms the cached SSE client session is reused without reconnect or retry. I did not run a live SSE MCP server in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is runtime-local SSE stale-session recovery with one retry and metadata refresh, preferably sharing helper structure with the streamable-http recovery path if that PR lands first.
