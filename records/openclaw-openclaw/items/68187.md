## Summary

Keep open: current main and the latest release still reuse the cached MCP client for tool calls without reconnecting stale sessions, and the related reconnect PRs are open with unresolved proof and retry-safety review blockers.

## What This Changes

The issue asks OpenClaw to recover MCP client sessions after downstream SSE-backed server restarts instead of keeping stale session state.

## Best Possible Solution

Keep this issue as the canonical MCP session-recovery tracker until maintainers land one proof-backed runtime fix, while leaving the catalog reload and retry-storm leftovers to their narrower linked issues.

## Reproduction Assessment

Yes at source level: current `callTool()` reuses the cached MCP client session and rethrows transport/session errors without reconnecting. I did not run a live SSE or streamable-HTTP server restart in this read-only review.

## Solution Assessment

Unclear for implementation shape: reconnect-on-session-loss belongs in the MCP runtime path, but maintainers need to choose safe retry semantics and whether one fix should cover SSE, streamable HTTP, and stdio together.
