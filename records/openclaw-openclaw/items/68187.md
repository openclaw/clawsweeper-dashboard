## Summary

Keep open. Current main still reuses the cached bundled MCP client session and directly calls `session.client.callTool(...)` without stale-session classification, reconnect, or a one-time retry for the SSE path; the linked open PR is explicitly scoped to `streamable-http` and does not cover the default SSE transport.

## What This Changes

Recover stale SSE-backed bundled MCP sessions after a downstream server restart instead of requiring a full OpenClaw gateway restart.

## Best Possible Solution

Add bounded SSE stale-session recovery in the session MCP runtime: classify known lost-session POST failures, recreate only the affected SSE client/transport, refresh metadata as needed, and retry the original tool call once while leaving stdio behavior unchanged.

## Reproduction Assessment

Yes, with source-level confidence. The issue gives a successful-call, downstream SSE restart, second-call failure path, and current main confirms the cached SSE client session is reused without reconnect or retry; I did not run a live SSE server in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is runtime-local SSE stale-session recovery with one retry and metadata refresh, ideally sharing helper code with the streamable-http recovery path if that PR lands first.
