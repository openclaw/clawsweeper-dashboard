## Summary

Keep open: current main and the latest release still call the cached MCP SDK session directly for `callTool()` and do not retire/recreate the stale per-server session after downstream session loss. The active reconnect PRs are implementation candidates, but they are unmerged and conflicting, so they are not provenance for closing this issue.

## What This Changes

The issue asks OpenClaw to recover SSE-backed MCP client/proxy sessions after a downstream server restart instead of continuing to fail with stale `Session not found` state.

## Best Possible Solution

Land one maintainer-approved MCP runtime recovery path that retires only the stale per-server session, refreshes affected catalog state, and performs a bounded safe retry across the intended transport set.

## Reproduction Assessment

Yes, at source level: establish an MCP runtime session, restart or invalidate the downstream server, and the current `callTool()` path reuses the cached SDK client and propagates the transport/session error without reconnecting. I did not run a live restart repro in this read-only review.

## Solution Assessment

The runtime recovery direction is the right boundary; a UI-only change would not fix the cached SDK session. The exact best patch still needs maintainer choice on transport scope, retry/idempotency classification, and catalog refresh semantics.
