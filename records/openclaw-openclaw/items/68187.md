## Summary

Keep open. Current main still reuses the cached SSE-backed MCP client session without stale-session classification, reconnect, or one-time retry, and the only related open PR is scoped to streamable-http rather than the SSE default path.

## What This Changes

The issue asks OpenClaw to recover SSE-backed MCP client sessions after a downstream server restart instead of reusing stale session state that fails with `Session not found`.

## Best Possible Solution

Add bounded SSE stale-session recovery in the session MCP runtime: classify known lost-session POST errors, rebuild only the affected SSE client/transport, refresh tool metadata when needed, and retry the original call once while leaving stdio behavior unchanged.

## Reproduction Assessment

Yes, with source-level confidence: the issue gives a successful-call, downstream SSE restart, second-call path, and current main confirms the same cached SSE client session is reused without reconnect or retry. I did not run a live SSE server, so this is source-reproducible rather than directly reproduced.

## Solution Assessment

Yes. The narrow maintainable fix is runtime-local SSE stale-session recovery with one retry and metadata refresh, ideally sharing helper code with the streamable-http recovery path if that PR lands first.
