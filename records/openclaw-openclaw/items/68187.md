## Summary

Keep open. Current main still caches the SSE-backed MCP client/session and sends the next tool call through the old SDK transport without classifying `Session not found`, rebuilding the SSE transport, or retrying; the related open PR is scoped to streamable-http and does not cover the SSE default path.

## What This Changes

The issue requests automatic recovery or invalidation of stale SSE-backed MCP client sessions after a downstream MCP server restart.

## Best Possible Solution

Add bounded SSE stale-session recovery in the session MCP runtime: classify known lost-session POST errors, rebuild only the affected SSE client/transport, refresh tool metadata if needed, and retry the original call once while leaving stdio behavior unchanged.

## Reproduction Assessment

Yes, with source-level confidence: the issue gives a successful-call, downstream SSE restart, second-call path, and current main confirms the same cached SSE client session is reused without reconnect or retry. I did not run a live SSE server, so this is source-reproducible rather than directly reproduced.

## Solution Assessment

Yes. The narrow maintainable fix is runtime-local SSE stale-session recovery with one retry and metadata refresh, ideally sharing helper code with the streamable-http recovery path if that PR lands first.
