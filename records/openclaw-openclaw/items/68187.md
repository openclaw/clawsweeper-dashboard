## Summary

Keep open: current main and the latest release still reuse the cached MCP client session for tool calls without stale-session reconnect, while the linked reconnect PRs are open and still need maintainer decisions/proof before they can own the fix.

## What This Changes

The issue requests automatic recreation of stale MCP client sessions after an SSE-backed MCP server restart instead of requiring a gateway restart.

## Best Possible Solution

Land one proof-backed MCP runtime recovery design that recreates the per-server session on stale-session transport failures without unsafe duplicate tool execution, while leaving broader catalog reload work to the narrower linked tracker.

Yes at source level: current `callTool()` reuses the cached MCP client session and rethrows transport/session errors without reconnecting. No live restart repro was run in this read-only review.
