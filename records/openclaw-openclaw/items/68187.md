- `impact:session-state`: The report is about stale cached MCP client session state persisting after the downstream SSE server restarts.

## Summary

Keep open. Current main and the latest release still route SSE-backed bundled MCP tool calls through a cached client with no stale-session catch, reconnect, or retry path, and the related open PRs cover streamable-http recovery or unexpected transport-close eviction rather than the SSE POST `Session not found` failure.

## What This Changes

The issue asks OpenClaw to invalidate and recreate stale SSE MCP client sessions after a downstream server restart instead of requiring a gateway restart.

## Best Possible Solution

Add bounded runtime-local SSE lost-session recovery: classify known SSE POST lost-session errors, rebuild only the affected SSE client/transport, refresh metadata as needed, and retry once while leaving stdio unchanged.

## Reproduction Assessment

Yes, at source level: a successful SSE tool call, downstream SSE restart, and second tool call still reach the cached `session.client.callTool()` path with no reconnect. I did not run a live SSE server restart probe in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is reconnect-and-retry for the affected SSE session only, preferably sharing structure with the streamable-http and transport-close work if either lands first.
