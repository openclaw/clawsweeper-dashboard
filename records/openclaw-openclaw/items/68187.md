## Summary

Keep open. Current main still has the reported stale-session path: bundled MCP tool calls reuse the cached per-server client/transport and call through directly, with no SSE lost-session detection, reconnect, catalog refresh, or one-shot retry. The related PR #71545 is useful context, but it is still open and scoped to streamable-http recovery rather than the SSE path reported here.

## What This Changes

The issue asks OpenClaw to recover SSE-backed MCP client sessions after a downstream MCP server restart instead of leaving callers pinned to stale state that returns `Session not found`.

## Best Possible Solution

Add bounded SSE stale-session recovery in the session MCP runtime: detect known lost-session responses, dispose and recreate only the affected server client/transport, refresh relevant catalog/tool metadata, and retry the original tool call once while leaving stdio behavior unchanged.

## Reproduction Assessment

Yes. The issue gives a concrete successful-call, downstream SSE server restart, second-call reproduction path, and current source confirms the OpenClaw runtime keeps using the cached session; a regression test can model this with an SSE probe server that invalidates the session before the second call.

## Solution Assessment

No current solution is in main. The narrow maintainable fix belongs in `src/agents/pi-bundle-mcp-runtime.ts` with SSE-focused tests, coordinated with #71545 if that streamable-http recovery lands first.
