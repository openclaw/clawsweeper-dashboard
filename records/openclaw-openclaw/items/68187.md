## Summary

Keep open: current main and the latest release still reuse the cached MCP SDK session for `callTool()` without disposing and recreating stale per-server sessions after downstream restart/session loss. The linked reconnect PRs remain open candidates with proof/merge blockers, so they are not implementation provenance yet.

## What This Changes

The issue reports stale SSE-backed MCP sessions after a downstream server restart and asks OpenClaw to reconnect or recreate the per-server MCP session instead of requiring a gateway restart.

## Best Possible Solution

Land one maintainer-approved MCP runtime recovery path that recreates only stale per-server sessions, refreshes affected catalog state, retries only where safe, and validates SSE, streamable-HTTP, and stdio behavior.

## Reproduction Assessment

Yes, at source level: create an MCP runtime session, restart or invalidate the downstream server, and the current `callTool()` path reuses the cached SDK client and propagates the transport/session error without reconnecting. I did not run a live restart repro in this read-only pass.

## Solution Assessment

The agent MCP runtime is the right boundary, and a UI-only change would not fix the cached SDK client. The exact best patch still needs maintainer choice on transport scope, retry/idempotency classification, and catalog refresh semantics.
