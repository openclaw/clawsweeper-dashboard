## Summary

Keep open: current main and the latest release still reuse the cached per-server MCP SDK session for `callTool()` and propagate stale transport/session errors without recreating the session. Open PRs target parts of the fix, but they are unmerged and still need maintainer choices on retry/idempotency and catalog refresh behavior.

## What This Changes

The issue asks OpenClaw to recover stale SSE-backed MCP client sessions after downstream server restart instead of failing repeated tool calls with `Session not found`.

## Best Possible Solution

Land one maintainer-approved MCP runtime recovery path that recreates only stale per-server sessions, refreshes affected catalog state, retries only where safe, and validates SSE, streamable-HTTP, and stdio behavior.

## Reproduction Assessment

Yes, at source level: create an MCP runtime session, restart or invalidate the downstream server, and the current `callTool()` path reuses the cached SDK client and propagates the transport/session error without reconnecting. I did not run a live restart repro in this read-only pass.

## Solution Assessment

Unclear: the agent MCP runtime is the right boundary, and a UI-only change would not fix the cached SDK client. The exact best patch still needs maintainer choice on transport scope, retry/idempotency classification, and catalog refresh semantics.
