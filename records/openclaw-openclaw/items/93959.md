## Summary

Keep open: this is a useful direction for the linked cron bug, but the patch is not merge-ready because the new pending deferral Set never expires and some timer reservation-release recomputes still bypass the new skip state. The PR also has only test/CI output, not real cron service, CLI, or Gateway behavior proof.

## What This Changes

The PR adds `pendingCatchupDeferralJobIds` to cron service state and threads it into selected maintenance recomputes so startup overflow catch-up deferrals survive read RPCs and timer maintenance.

## Best Possible Solution

Make pending startup deferrals a lifecycle-owned cron maintenance invariant: guard every recompute that can run before the deferred tick, clear IDs once due or consumed, and prove the read/status plus timer paths with focused regression coverage and real runtime output.

## Reproduction Assessment

Yes, source-reproducible. Current main shows startup overflow deferrals are protected only by an explicit skip set, while read/status and timer maintenance paths can run future-slot repair without the pending deferral context.

## Solution Assessment

No. Persisting pending IDs in state is the right direction, but this patch needs expiry and a single guarded maintenance path before it is the best fix.
