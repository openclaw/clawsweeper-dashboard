## Summary

Keep open: the fix direction is valid, but the branch is not merge-ready because the pending deferral Set never expires, two timer reservation-release recomputes remain unguarded, and external real behavior proof is still only test output.

## What This Changes

The PR adds a `pendingCatchupDeferralJobIds` Set to cron service state and threads it into selected read, manual, and timer maintenance recomputes to preserve startup overflow catch-up slots.

## Best Possible Solution

Make pending startup deferrals a lifecycle-owned cron maintenance invariant: guard every recompute that can run before the deferred tick, clear IDs once due or consumed, and prove the read/status plus timer paths with focused regression coverage and real runtime output.

## Reproduction Assessment

Yes. The linked issue and current source show startup overflow deferrals are protected only by explicit skip IDs, while read/status and timer maintenance recomputes can run future-slot repair before the deferred slot fires; I did not execute tests because this review is read-only.

## Solution Assessment

No. Persisting pending IDs in service state is the right direction, but this patch is not the best fix until it expires those IDs and guards every recompute path that can run before the pending slot.
