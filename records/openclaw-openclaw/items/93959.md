## Summary

Keep open: the PR moves the cron overflow deferral guard in the right direction, but the current head is still not merge-ready because the new pending ID set has no expiry/consume lifecycle, one timer cleanup recompute remains unguarded, and the supplied proof is only focused Vitest/CI output.

## What This Changes

The PR adds `pendingCatchupDeferralJobIds` to cron service state, stores startup overflow deferral IDs there, and passes that set into selected maintenance recomputes in cron ops and timer code.

## Best Possible Solution

Make pending startup deferrals lifecycle-owned: guard every maintenance recompute path that can run before the deferred tick, clear IDs once due or consumed, preserve ordinary stale-future repair for unrelated jobs, and add real runtime proof.

## Reproduction Assessment

Yes. Current main creates only a startup-local skip set, while read/status and timer maintenance recomputes can still run future-slot repair before the staggered deferral fires; I did not execute tests because this review is read-only.

## Solution Assessment

No. Persisting deferral IDs in service state is the right direction, but this patch is not the best fix until it expires those IDs, covers the remaining cleanup recompute, and supplies real runtime proof.
