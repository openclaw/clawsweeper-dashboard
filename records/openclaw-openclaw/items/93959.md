## Summary

Keep open: the PR targets a real cron missed-run bug, but the patch is still not merge-ready because the new pending deferral state is never cleared, one timer cleanup recompute still bypasses the guard, and the contributor proof is test/CI output rather than real cron service, CLI, or Gateway behavior.

## What This Changes

Adds an in-memory pending catch-up deferral ID set to cron service state and threads it into selected read, manual-run, and timer maintenance recomputes.

## Best Possible Solution

Make pending startup deferrals lifecycle-owned: guard every recompute path that can run before the deferred tick, clear ids once due or consumed, and add real runtime proof for read/status plus timer cleanup paths.

## Reproduction Assessment

Yes. The canonical issue and current source show startup overflow deferrals are protected only by a startup-local skip set, while read/status and timer maintenance recomputes can run future-slot repair before the deferred slot fires; I did not execute tests because this review is read-only.

## Solution Assessment

No. Persisting pending ids in service state is the right direction, but this patch is not the best fix until it expires those ids, covers the remaining cleanup recompute, and supplies real runtime proof.
