## Summary

Keep open: the reported cron bug is valid and the state-owned direction is useful, but current PR head still lacks deferral cleanup, leaves reservation-release recomputes unguarded, and has only test/CI proof rather than real runtime proof.

## What This Changes

The PR adds `pendingCatchupDeferralJobIds` to cron service state and passes it into selected read, manual-run, and timer maintenance recomputes.

## Best Possible Solution

Make pending startup deferrals a lifecycle-owned cron invariant: guard every maintenance recompute that can run before the deferred tick, clear IDs once due or consumed, and provide runtime proof for read/status plus timer paths.

## Reproduction Assessment

Yes. The linked issue and current source show startup overflow deferrals are protected only by explicit skip IDs, while read/status and timer maintenance recomputes can run future-slot repair before the deferred slot fires; this review is read-only, so I did not execute the repro test.

## Solution Assessment

No. Persisting pending IDs in service state is the right direction, but this patch is not the best fix until it expires those IDs and guards every recompute path that can run before the pending slot.
