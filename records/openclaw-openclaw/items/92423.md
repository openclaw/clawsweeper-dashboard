## Summary

This remains necessary because v2026.6.5 exposes the requested lifecycle hooks but does not define a channel-neutral, idempotent outbound correlation contract: outbound `runId` is absent, and Telegram normal replies expose less typed context than shared outbound delivery. The supported direction is an external typed plugin, but maintainers must decide whether to document these limitations or align the API first.

## What This Changes

Clarify the supported pattern and guarantees for telemetry-only inbound and outbound message lifecycle capture into local SQLite across Telegram and WhatsApp.

## Best Possible Solution

Recommend a typed external plugin, align correlation fields across Telegram and shared delivery, document normal-reply versus message-tool coverage, and show durable dedupe using plugin-owned SQLite state plus channel-native message identifiers.

## Reproduction Assessment

Yes at source level: v2026.6.5 and current main expose the exact lifecycle emitters, and the public type directly documents the outbound correlation limitation. This is primarily a contract question rather than a single failing runtime scenario.

## Solution Assessment

No, a HOOK.md-only recipe is not the best solution. Durable CRM telemetry belongs in a typed external plugin, but maintainers first need to settle and document outbound identity and cross-path parity.
