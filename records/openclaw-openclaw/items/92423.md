## Summary

Keep this documentation question open: OpenClaw exposes the requested lifecycle observations, but current documentation does not give an authoritative telemetry recipe or clearly explain outbound correlation limits and WhatsApp activation boundaries.

## What This Changes

Clarify the supported pattern for telemetry-only inbound and outbound message lifecycle capture into local SQLite across Telegram and WhatsApp.

## Best Possible Solution

Add a focused plugin-hooks recipe recommending observation-only typed hooks, transactional SQLite inserts with channel/message/direction identity and a deterministic fallback key, local hashing or redaction, per-channel delivery smoke tests, and explicit WhatsApp activation and correlation caveats.

## Reproduction Assessment

Not applicable; this issue asks for authoritative guidance about existing hook guarantees rather than reporting a failing runtime path.

## Solution Assessment

Yes, a narrow documentation recipe is the best direction, provided it uses the typed plugin boundary and accurately states the shipped correlation and channel-coverage limitations rather than adding CRM behavior to core.
