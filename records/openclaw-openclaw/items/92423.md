## Summary

This remains a useful maintainer documentation decision: v2026.6.5 ships both typed plugin hooks and internal HOOK.md lifecycle events, but outbound correlation is incomplete and channel delivery paths differ enough that the repository lacks one precise telemetry contract to cite.

## What This Changes

Clarify the supported pattern for privately capturing inbound and outbound message lifecycle telemetry into SQLite across Telegram and WhatsApp without triggering automation.

## Best Possible Solution

Add a focused plugin-hook recipe recommending a code plugin with SQLite unique constraints, canonical channel message identifiers, explicit delivery and deduplication semantics, current outbound correlation limits, and linked WhatsApp admission guidance.

## Reproduction Assessment

Not applicable. This is a supported-pattern and contract question; source and shipped-tag inspection establish the relevant behavior without a failing reproduction.

## Solution Assessment

Yes, documenting the existing typed code-plugin hook path is the narrowest maintainable solution, provided the guidance explicitly preserves current correlation and channel-coverage limitations instead of presenting HOOK.md as a stable telemetry API.
