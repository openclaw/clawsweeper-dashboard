## Summary

Current docs identify the available extension surfaces, but they do not give a complete supported pattern for raw, local message-lifecycle records: diagnostic events are aggregate observability signals, while outbound typed hooks still lack reliable per-turn run correlation. This needs the requested maintainer product clarification rather than closure as already documented.

## What This Changes

Clarify the supported pattern for privacy-preserving telemetry capture of inbound and outbound message lifecycle events across Telegram and WhatsApp.

## Best Possible Solution

Document one privacy-preserving, read-only lifecycle-capture pattern that distinguishes aggregate diagnostics from raw local records, identifies the supported hook surface, states channel and message-tool coverage, defines usable SQLite uniqueness/upsert keys and current correlation limits, and explains WhatsApp group admission versus sender authorization.

## Reproduction Assessment

Not applicable. This issue asks for a supported architecture and reliability contract rather than reporting a concrete current-main failure; the relevant behavior can be evaluated from shipped docs, source, tests, and history.

## Solution Assessment

No, recommending HOOK.md, typed hooks, or diagnostic events without qualification is not sufficient. The narrowest maintainable solution is owner-approved documentation separating aggregate observability from raw local capture and explicitly describing the outbound correlation limitation.
