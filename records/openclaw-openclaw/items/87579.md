## Summary

Keep open: the performance direction fits OpenClaw's process-stable plugin metadata policy, but the patch reuses the global metadata snapshot without the facade boundary config/default-discovery guard, so facade resolution can read a manifest registry built for different plugin load paths. The supplied proof is also test-only rather than real runtime proof.

## What This Changes

This PR changes the plugin SDK facade to return the current in-memory manifest registry before falling back to `loadPluginManifestRegistry`, and updates one facade runtime test regex for Windows path separators.

## Best Possible Solution

Keep the memory-snapshot optimization only after facade resolution validates the snapshot against the resolved facade boundary config, with regression coverage for mismatched plugin load paths.

## Reproduction Assessment

Source-reproducible: the PR calls `getCurrentPluginMetadataSnapshot` without config/default-discovery validation, while current tests document why configless callers must guard load-path snapshots. I did not run write-producing tests in this read-only review.

## Solution Assessment

No; the narrow solution is to keep the in-memory optimization but validate the snapshot with the resolved facade boundary config, or explicitly restrict the configless path to default-discovery snapshots.
