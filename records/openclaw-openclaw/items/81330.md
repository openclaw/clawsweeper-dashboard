## Summary

Keep open: this targets a real Windows node background-service bug, but the current patch breaks the scheduled-task command parsing/fallback contract and still lacks real Windows after-fix proof.

## What This Changes

The PR changes the generated Windows Scheduled Task batch script in `src/daemon/schtasks.ts` to wrap the final service command with `start "" /b`.

## Best Possible Solution

Keep the fix scoped to Windows scheduled-task launching, but preserve the parseable service-command contract and prove the node lifecycle on Windows after the launcher change.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: the linked issue gives concrete Windows 11 steps, and current main still routes `openclaw node install` through `node.cmd` plus the shared interactive Scheduled Task path. A live Windows login/start run is still needed for final behavior proof.

## Solution Assessment

No. The PR targets the right renderer, but wrapping the stored command without updating `readScheduledTaskCommand` and fallback launch handling is not the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve the parsed task command when wrapping launch:** `src/daemon/schtasks.ts:279-280`
