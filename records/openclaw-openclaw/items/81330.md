## Summary

Keep open. This is a focused Windows daemon bug fix, but the current patch breaks OpenClaw's parseable scheduled-task command contract and still lacks after-fix Windows behavior proof.

## What This Changes

The PR changes `src/daemon/schtasks.ts` so generated Windows Scheduled Task `.cmd` scripts launch the final command through `start "" /b`.

## Best Possible Solution

Keep the fix scoped to Windows scheduled-task launching, but preserve the parseable service command contract and prove install, start, status, stop, and restart on Windows after the launcher change.

## Reproduction Assessment

Yes. The linked issue gives concrete Windows 11 steps and current source still routes `openclaw node install` through `node.cmd` plus the shared interactive Scheduled Task path; I did not run a live Windows login session in this read-only review.

## Solution Assessment

No. The PR targets the right renderer, but wrapping the stored command without updating `readScheduledTaskCommand`, fallback launch, and tests is not the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve the parsed task command when wrapping launch:** `src/daemon/schtasks.ts:279-280`
  - body: Wrapping the generated command makes `readScheduledTaskCommand` return `start` and `/b` as part of the service command. `launchFallbackTaskScript` later spawns `argv[0]`, so fallback and audit paths lose the real node command; either unwrap this in the parser/fallback path or use a launcher shape that preserves the command contract.
  - confidence: 0.88
