## Summary

Keep this PR open: it addresses a real P1 cron/main-session bug, but the current head still leaves task-ledger paths inconsistent with the new cron run lane, and the protected `maintainer` label also prevents cleanup closure.

## What This Changes

The PR routes main-session cron jobs onto dedicated cron-owned run session keys, forwards delivery context, and updates cron/gateway tests plus cron documentation.

## Best Possible Solution

Centralize a single main-session cron run key and use it for execution, emitted events, and task ledger records across timer, startup catch-up, and manual runs before landing.

## Reproduction Assessment

Yes, at source level. The linked issue provides concrete timing logs, and current main clearly routes main-session cron system events and heartbeat wakes through `job.sessionKey`, the same durable lane used by human web chat.

## Solution Assessment

No. The direction is the narrow maintainable fix, but this branch should compute and pass one cron run lane through task creation and execution for manual, timer, and startup catch-up paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Use one cron run lane for manual and catch-up records:** `src/cron/service/timer.ts:1741`
  - body: `executeMainSessionCronJob` now wakes `cronRunSessionKey`, but `CronService.run` still creates task records with the human `params.job.sessionKey`, and startup catch-up can create a task child key from a later timestamp than the execution lane. Compute the lane once and pass it through both task creation and execution.
