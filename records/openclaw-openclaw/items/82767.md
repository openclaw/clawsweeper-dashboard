## Summary

Keep open: the protected `maintainer` label prevents cleanup closure, and the patch still has a blocking consistency issue around cron run-lane task records. The fix direction is sound, but the derived cron lane needs to be shared by scheduled, startup catch-up, and manual run paths before merge.

## What This Changes

The PR routes main-session cron jobs through derived cron run session keys, forwards delivery context, updates docs, and adds cron seam tests.

## Best Possible Solution

Land the cron-owned lane behavior after centralizing the run key so execution, emitted events, and task records agree across every main-session cron launch path.

## Reproduction Assessment

Yes. The linked issue gives concrete steps and logs, and current main source shows main-session cron enqueue and heartbeat calls targeting `job.sessionKey`, which is the human main-session lane.

## Solution Assessment

No. The proposed direction is the right narrow fix, but the implementation should compute one cron run session key and feed it through task creation and execution for timer, startup catch-up, and manual runs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Use the same cron run lane for task records:** `src/cron/service/timer.ts:1741`
  - body: `executeMainSessionCronJob` now moves main cron execution to `cronRunSessionKey`, but other launch paths do not share that same key. Manual `CronService.run` still creates the task record with `childSessionKey: params.job.sessionKey`, and startup catch-up can derive the task child key from a later `startedAt` than the `runningAtMs` used here. That leaves task/cleanup surfaces pointing at the human lane or a different run lane while the actual wake uses `agent:<agent>:cron:...:run:<ts>`. Compute the run lane once and use it for task creation and execution across all main-session cron paths.
  - confidence: 0.86
