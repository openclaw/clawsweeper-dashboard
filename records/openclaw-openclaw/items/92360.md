## Label Justifications

- `P1`: The reported bug can make long-lived Anthropic sessions fail every subsequent user or scheduled turn until the session is reset.
- `impact:session-state`: The failure is caused by persisted session history replaying stale thinking blocks and poisoning future turns.

## Summary

Current main now handles the reported persistent-session failure at the shared embedded-runner boundary: after an Anthropic/Bedrock thinking replay rejection, OpenClaw retries with thinking blocks stripped and repairs the persisted branch so later turns do not replay the same poisoned history. The fix is current-main-only after v2026.6.6.

## What This Changes

The issue asks OpenClaw to prevent stale Anthropic thinking blocks in persistent sessions from causing repeated provider schema rejections after thinking settings change.

## Best Possible Solution

Keep the shared embedded-runner recovery and durable transcript repair from https://github.com/openclaw/openclaw/pull/92286 as the canonical fix for this issue, while routing any broader replay-policy gaps to the separate open tracking issue.

## Reproduction Assessment

Yes, at source level: a persistent transcript containing signed thinking can hit the recovery wrapper's rejected-replay path, and current tests cover stripped retry plus durable branch repair. I did not run a live Anthropic session in this read-only review, but the merged fix PR includes live Anthropic proof.

## Solution Assessment

Yes. The current fix is at the shared embedded-runner request/recovery boundary, so cron and interactive persistent sessions get the same retry and transcript repair instead of a cron-only or provider-only shim.
