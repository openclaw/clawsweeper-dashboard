## Summary

Keep open: the reported fenced-code helper gap still exists on current main, but this PR is not mergeable as-is because the regex drops same-line triple-backtick spans, mishandles longer fences, does not affect the current iMessage send path, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a fenced-code-block replacement pass to `stripMarkdown` and a new unit test file for fenced blocks and approval-message examples.

## Best Possible Solution

Use newline-delimited matched 3+ backtick fence handling that preserves inline spans, cover longer approval fences, and either wire the current affected plain-text send path or explicitly leave the broader pipeline to https://github.com/openclaw/openclaw/pull/56283.

## Reproduction Assessment

Yes for source-level helper behavior: current main leaves fenced markers in `stripMarkdown` for a simple fenced block, and the PR regex regressions reproduce with same-line triple-backtick spans and longer approval fences. I did not establish a live iMessage send reproduction.

## Solution Assessment

No. `stripMarkdown` is a plausible helper boundary for TTS/LINE-style consumers, but the proposed helper-only regex is overbroad and does not fix the current reported iMessage path.
