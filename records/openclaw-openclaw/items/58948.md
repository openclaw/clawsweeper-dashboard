## Summary

Keep this PR open: it addresses a real current-main plain-text rendering bug, but the proposed fence regex still drops same-line triple-backtick code-span content and mishandles longer fences emitted by the approval-message helper; it also needs the required user-facing changelog entry.

## What This Changes

The PR adds a fenced-code-block stripping pass to `stripMarkdown` plus focused tests for approval-message output and adjacent-text spacing.

## Best Possible Solution

Keep the narrow shared-helper fix, but implement matching newline-delimited fenced-block handling for 3+ backticks, preserve same-line multi-backtick spans, add regression tests and a changelog entry, and leave the broader delivery-pipeline feature to #56283.

## Reproduction Assessment

Yes. Current main reproduces the original bug with `stripMarkdown("```txt\nhello\n```")`, and the PR regression reproduces by applying the proposed regex to `Run ```npm i``` now` and to a four-backtick fenced block.

## Solution Assessment

No as proposed. `stripMarkdown` is the right boundary for this focused fix, but the implementation should use a tighter parser or regex that requires real block fences and matches the opening fence length.
