Keep open: the helper change is useful, but it still does not fix the described iMessage/plain-text outbound path because current main routes iMessage text through the channel sanitizer and send formatter without calling `stripMarkdown()`. The updated proof exercises the helper directly, not a real outbound send, and the broader shared plain-text policy remains tracked in https://github.com/openclaw/openclaw/pull/56283 and https://github.com/openclaw/openclaw/issues/24943.

## What This Changes

The PR adds fenced-code-block and inline triple-backtick stripping to `stripMarkdown()` plus focused unit tests for approval-style and adjacent fenced text.

## Best Possible Solution

Apply fenced-code stripping at the actual outbound plain-text boundary, preserve code contents including dynamically sized fences, and coordinate the narrow approval-message fix with the broader policy in https://github.com/openclaw/openclaw/pull/56283.
