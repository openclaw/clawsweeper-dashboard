## Summary

Keep open: the fenced-code gap is real at the helper level, but this PR still misses the current iMessage send path and its regex introduces user-visible content loss for same-line triple-backtick spans and longer fences.

## What This Changes

The PR adds a fenced-code-block replacement to `stripMarkdown` plus tests for fenced approval-message output and adjacent text spacing.

## Best Possible Solution

Use newline-delimited 3+ backtick fence handling that preserves inline spans, add regressions, and either wire the current affected iMessage/plain-text path or explicitly defer that broader pipeline to https://github.com/openclaw/openclaw/pull/56283.

## Reproduction Assessment

Yes for source behavior: current main's helper leaves fenced markers from ` ```txt\nhello\n``` `, and the PR regex drops content from `Run ```npm i``` now`. No live iMessage send reproduction or after-fix channel proof is present.

## Solution Assessment

No. A helper-level fence stripper is a reasonable boundary for current helper consumers, but this regex is too broad and the reported iMessage path is not fixed by changing only `stripMarkdown`.
