## Summary

Keep open: the fenced-code helper bug is real, but this PR does not reach the reported current iMessage send path and the added regex introduces user-visible content loss for inline triple-backtick spans and longer approval fences.

## What This Changes

The PR adds fenced-code-block stripping to `stripMarkdown` plus unit tests for fenced approval-message output and adjacent text spacing.

## Best Possible Solution

Update the branch with newline-delimited, same-length 3+ backtick fence stripping and regression tests, then either wire the current plain-text/iMessage delivery path or explicitly scope this PR to existing helper consumers while the broader policy remains in https://github.com/openclaw/openclaw/pull/56283.

## Reproduction Assessment

Yes for source-level review: current main leaves fenced markers in `stripMarkdown("```txt\nhello\n```")`, and the PR-head regex loses same-line triple-backtick content and mishandles longer fences. I did not establish a live iMessage send reproduction.

## Solution Assessment

No. `stripMarkdown` is a plausible narrow helper boundary, but the proposed regex is too broad and the PR does not currently fix the reported iMessage delivery path.

  - confidence: 0.93

  - body: `buildFence` lengthens the delimiter when command text already contains triple backticks, but this regex always closes on three backticks. Four-backtick approval blocks can leave stray markers or lose content, so the stripper should capture the opening delimiter length and close on the same delimiter.
  - confidence: 0.9
