## Best Possible Solution

Land delimiter-aware fenced-code stripping through the actual non-Markdown outbound/iMessage path, or fold it into the capability-aware plain-text policy tracked by https://github.com/openclaw/openclaw/issues/24943 and https://github.com/openclaw/openclaw/pull/56283.

## Reproduction Assessment

Yes for source-level review: current main leaves fenced markers in `stripMarkdown()`, and the PR regex can be shown to drop same-line triple-backtick content and mishandle longer fences. I did not establish a live iMessage send reproduction in this read-only review.

## Solution Assessment

No. `stripMarkdown()` is a plausible helper, but this patch is too broad in regex semantics and is not wired into the reported iMessage/plain-text delivery path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Wire stripping into the reported send path:** `src/shared/text/strip-markdown.ts:10`
  - body: The PR says it fixes iMessage approval messages, but current `sendMessageIMessage`, the iMessage channel sanitizer, and shared outbound normalization do not call `stripMarkdown()`. Changing only this helper leaves the reported delivery path unchanged.
  - confidence: 0.9
- **[P2] Require fenced blocks to start on their own line:** `src/shared/text/strip-markdown.ts:10`
  - body: The regex also matches same-line text like `Run ```npm i``` now`, treats `npm i` as the fence info string, captures no body, and emits `Run  now`. Require a newline-delimited opening fence before removing it.
  - confidence: 0.93
- **[P2] Match the opening fence delimiter:** `src/shared/text/strip-markdown.ts:10`
  - body: `buildFence()` lengthens approval fences when command text already contains triple backticks, but this regex always closes on the first three backticks. Four-backtick blocks can leave stray markers or lose command content.
  - confidence: 0.9
