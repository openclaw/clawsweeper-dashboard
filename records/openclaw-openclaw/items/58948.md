## Summary

Keep open: the PR targets a real helper gap, but the current patch does not fix the claimed iMessage path and its regex still has user-visible loss cases for inline triple-backtick text and longer generated fences.

## What This Changes

The PR adds fenced-code-block stripping to `stripMarkdown` and adds focused tests for fenced approval-message output.

## Best Possible Solution

Repair fence stripping to require newline-delimited, same-length 3+ backtick fences with regression tests, then either wire the actual iMessage/plain-text delivery path or explicitly leave that broader policy to the existing outbound Markdown stripping work.

## Reproduction Assessment

Yes for source-level helper behavior: current main leaves fenced markers, and the PR-head regex drops same-line triple-backtick content and mishandles longer approval fences. I did not establish a live iMessage send reproduction.

## Solution Assessment

  - confidence: 0.9
