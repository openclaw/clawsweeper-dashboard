## Best Possible Solution

Land delimiter-aware fenced-code stripping through the actual non-Markdown outbound/iMessage path, or fold it into the capability-aware plain-text policy tracked by https://github.com/openclaw/openclaw/issues/24943 and https://github.com/openclaw/openclaw/pull/56283.

Yes for source-level helper behavior: current main leaves fenced markers, and the PR regex can be probed to drop same-line triple-backtick content and mishandle longer approval fences. I did not establish a live iMessage send reproduction.

## Solution Assessment

No. `stripMarkdown` is a plausible narrow helper, but this patch is too broad in regex semantics and does not connect to the reported iMessage/plain-text delivery path.

- **[P2] Wire stripping into the reported send path:** `src/shared/text/strip-markdown.ts:10`
  - body: The PR says it fixes iMessage approval messages, but the changed helper is not called by `sendMessageIMessage`, the iMessage channel sanitizer, or shared outbound normalization on current main. Changing only `stripMarkdown` leaves the reported delivery path unchanged.

  - confidence: 0.93

  - body: `buildFence` lengthens approval fences when command text already contains triple backticks, but this regex always closes on the first three backticks. Four-backtick blocks can leave stray markers or lose command content.
  - confidence: 0.9
