## Summary

Keep this PR open: current main still has the fenced-code stripping gap, but the proposed regex has a blocking edge case that drops same-line triple-backtick code span content, and the user-facing fix also needs a changelog entry before merge.

## What This Changes

The PR adds a fenced-code-block stripping pass to `stripMarkdown` and adds focused tests for fenced blocks, approval-message output, ordering, and adjacent-text spacing.

## Best Possible Solution

Keep the narrow shared-helper fix, but make the fenced-block handling distinguish real newline-delimited blocks from same-line multi-backtick code spans, add regression coverage, add the required changelog entry, and leave the broader plain-text channel pipeline to #56283.

## Reproduction Assessment

Yes. The current-main gap reproduces with `stripMarkdown("```txt\nhello\n```")`, and the PR-introduced regression reproduces with the proposed regex on `Run ```npm i``` now`.

## Solution Assessment

No as proposed. The shared `stripMarkdown` boundary is the right place for this focused fix, but the regex should be tightened or replaced with a small parser so it strips fenced blocks without deleting inline multi-backtick code content.
