## Summary

Keep open: current main and v2026.5.20 still rethrow `path-mismatch` from `writeImportedSourcePage`'s page-stat guard, and the bridge/gateway paths have no per-artifact recovery before returning `internal_error`. The related open Dreaming report is similar, but this issue adds a distinct source-proven bridge `path-mismatch` failure mode with actionable fix boundaries.

## What This Changes

The issue asks Memory Wiki bridge sync and `wiki.palace` to tolerate a transient fs-safe `path-mismatch` on one imported artifact so healthy artifacts still render.

## Best Possible Solution

Treat only recoverable generated source-page stat `path-mismatch` as a skipped/retryable bridge artifact, keep fs-safe write and source-read failures fail-closed, and coordinate the result with the related Dreaming report.

## Reproduction Assessment

Yes at source level: if `vault.stat(pagePath)` throws `FsSafeError("path-mismatch")`, current main rethrows it through bridge sync and `wiki.palace` returns `internal_error`. I did not live-replay the event-loop starvation race in this read-only review.

## Solution Assessment

Mostly yes: adding a narrow recoverable `path-mismatch` skip for the generated page stat is the maintainable first fix. I would avoid changing the `wiki.palace` response schema unless a maintainer explicitly wants a broader partial-failure protocol.
