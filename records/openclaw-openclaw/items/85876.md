## Summary

Keep open: current main and v2026.5.22 still rethrow the memory-wiki bridge path-mismatch through wiki.palace, and the safest repair needs maintainer judgment around recoverable fs-safe errors and any partial-failure gateway contract.

## What This Changes

The issue asks for memory-wiki bridge sync and wiki.palace to tolerate a recoverable per-artifact fs-safe path-mismatch without failing the whole RPC.

## Best Possible Solution

Keep fs-safe fail-closed semantics for real write-safety collisions, but make bridge source sync tolerate clearly recoverable per-artifact stat/read races with logging, skipped/error accounting, and focused tests for wiki.palace and imported-source sync.

## Reproduction Assessment

Yes for the source path, but not as a deterministic live repro. Current source shows a path-mismatch from writeImportedSourcePage can rethrow through bridge sync into wiki.palace, while the event-loop pressure trigger itself was not reproduced in this read-only pass.

## Solution Assessment

Unclear. Treating a transient stat/read path-mismatch as recoverable is plausible, but the exact error taxonomy and any client-visible partial-failure schema need maintainer judgment so the fix does not weaken fs-safe path containment.
