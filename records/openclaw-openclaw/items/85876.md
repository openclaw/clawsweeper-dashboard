## Summary

Keep open: current main and the latest release still let a `path-mismatch` from memory-wiki imported-source sync abort `wiki.palace`, but the safe repair needs maintainer judgment around fs-safe containment and any partial-failure gateway contract.

## What This Changes

The issue asks for memory-wiki bridge sync to treat recoverable per-artifact `path-mismatch` races as non-fatal so `wiki.palace` can continue returning the rest of the Memory Palace data.

## Best Possible Solution

Keep fs-safe fail-closed semantics for real containment collisions, but make memory-wiki bridge source sync tolerate clearly recoverable per-artifact stat/read races with logging, skipped/error accounting, and focused regression coverage before deciding whether `wiki.palace` needs a documented partial-failure payload.

## Reproduction Assessment

Yes for the source path, but not as a deterministic live repro. Current source shows a `path-mismatch` from `writeImportedSourcePage` can rethrow through bridge sync into `wiki.palace`, while the event-loop pressure trigger itself was not reproduced in this read-only pass.

## Solution Assessment

Unclear. Treating a transient stat/read `path-mismatch` as recoverable is plausible, but the exact fs-safe error taxonomy and any client-visible partial-failure schema need maintainer judgment so the fix does not weaken path containment.
