## Summary

Keep open: current main and v2026.5.20 still rethrow the memory-wiki bridge `path-mismatch` through `wiki.palace`, but the safest repair needs maintainer judgment around recoverable fs-safe errors and any partial-failure gateway contract.

## What This Changes

The issue reports that a transient fs-safe `path-mismatch` during memory-wiki bridge sync can abort the entire `wiki.palace` RPC instead of skipping the affected artifact or returning a partial failure.

## Best Possible Solution

Keep fs-safe fail-closed semantics for real write-safety collisions, but make bridge source sync tolerate clearly recoverable per-artifact stat/read races with logging, skipped/error accounting, and focused tests for both `wiki.palace` and the related imported-source sync path.

## Reproduction Assessment

Yes for the source path, but not as a deterministic live repro. Current source shows a `path-mismatch` from `writeImportedSourcePage` can rethrow through bridge sync into `wiki.palace`, while the event-loop pressure trigger itself was not reproduced in this read-only pass.

## Solution Assessment

Unclear. Treating a transient stat/read `path-mismatch` as recoverable is plausible, but the exact error taxonomy and any client-visible partial-failure schema need maintainer judgment so the fix does not weaken fs-safe path containment.
