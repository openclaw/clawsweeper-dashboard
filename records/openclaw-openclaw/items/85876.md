## Summary

Keep open: current main and the latest shipped tag still allow a memory-wiki bridge `path-mismatch` from one imported source to abort `wiki.palace`, but the repair needs maintainer judgment because `path-mismatch` is also a safety signal for real path races.

## What This Changes

The issue asks for memory-wiki bridge source sync to tolerate recoverable per-artifact fs-safe `path-mismatch` races so `wiki.palace` can return useful results instead of failing the whole RPC.

## Best Possible Solution

Keep fs-safe fail-closed behavior for true containment and write collisions, but make bridge source sync tolerate clearly recoverable per-artifact stat/read races with logging, skipped/error accounting, and focused gateway/UI tests.

## Reproduction Assessment

Yes for the source path, but not as a deterministic live repro. Current source shows a `path-mismatch` from `writeImportedSourcePage` can rethrow through bridge sync into `wiki.palace`, while the event-loop pressure trigger itself was not reproduced in this read-only pass.

## Solution Assessment

Unclear. Treating a transient page-stat `path-mismatch` as recoverable is plausible, but the exact error taxonomy and any client-visible partial-failure schema need maintainer judgment so the fix does not weaken fs-safe path containment.
