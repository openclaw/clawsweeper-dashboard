## Summary

Keep open: current main and the latest release still let a memory-wiki bridge `path-mismatch` thrown during imported-source sync abort `wiki.palace`, but the repair needs maintainer judgment because fs-safe path mismatch can also represent a real write-safety collision.

## What This Changes

The issue reports that `wiki.palace` aborts the whole Memory Wiki bridge batch when one imported artifact hits an fs-safe `path-mismatch`, and asks for recoverable per-artifact handling plus partial-failure reporting.

## Best Possible Solution

Keep fs-safe fail-closed semantics for true vault write-safety failures, but make bridge source sync tolerate clearly recoverable per-artifact stat/read races with skipped/error accounting, logging, and focused gateway/UI tests.

## Reproduction Assessment

Partly: the source-level path is high confidence if `vault.stat` throws `FsSafeError("path-mismatch")`, because current code rethrows it into `wiki.palace`. No deterministic live reproduction of the event-loop pressure trigger was run in this read-only review.

## Solution Assessment

Unclear. Treating transient per-artifact stat/read races as recoverable is plausible, but broad `path-mismatch` swallowing and a new `wiki.palace` partial-failure payload need maintainer contract and security-boundary judgment.
