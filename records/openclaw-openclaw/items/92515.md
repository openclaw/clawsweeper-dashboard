## Summary

Keep open: the PR targets a real memory_search regression, but the current patch is not merge-ready because it uses a timer-based retry and replaces the shared SQLite handle from the search path without a lifecycle barrier; the submitted proof shows an injected metadata miss, not the claimed concurrent rebuild/WAL failure.

## What This Changes

The PR retries memory search once after missing index metadata, reopens the SQLite handle, adds recovery counters to status, and adds metadata-recovery tests.

## Best Possible Solution

Use a manager-owned database lifecycle barrier or isolated reindex publication path, likely coordinated with https://github.com/openclaw/openclaw/pull/90453, while preserving persistent identity mismatch failures and adding real concurrent proof.

## Reproduction Assessment

Yes, at the source level: current main can expose safe-reindex temp DB state to concurrent search/status reads, and the PR adds a synthetic missing-metadata test. I did not run a live concurrent gateway reproduction in this read-only review.

## Solution Assessment

No. A bounded retry may be a defensive fallback, but a timer plus shared-handle reopen is not the best owner boundary; the safer fix is lifecycle serialization or isolated reindex publication.

- **[P1] Serialize the SQLite handle replacement:** `extensions/memory-core/src/memory/manager.ts:630-632`
  - body: `reopenDatabaseForMetadataRecovery()` closes and replaces the manager-wide `this.db` without excluding another `search()`, `status()`, or `sync()` that may already be using that handle. Overlapping memory searches are part of the reported failure scenario, so one recovery can invalidate another live query. Move refresh into a shared database lifecycle barrier or isolate handles, and add deterministic overlapping-operation coverage.
  - confidence: 0.96

  - body: The fixed 500 ms wait does not establish that an atomic rebuild, WAL/checkpoint transition, or external file swap has completed. A slower rebuild will retry too early and fail as before, while an already-published replacement is delayed unnecessarily. Wait for the owned sync/reindex operation or detect backing-file replacement instead of using elapsed time alone.
  - confidence: 0.91
