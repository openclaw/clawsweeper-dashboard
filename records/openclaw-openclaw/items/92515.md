## Summary

The PR addresses a documented stale-reader failure after atomic index swaps, but its search-local recovery closes the manager’s shared SQLite handle without coordinating concurrent searches or the active sync lifecycle. That creates a concrete availability regression risk, and the PR also lacks required after-fix proof from a real rebuild/search setup.

## What This Changes

The PR retries `memory_search` once after missing index metadata by delaying, reopening the SQLite database, resetting vector state, and adding recovery diagnostics and tests.

## Best Possible Solution

Deduplicate stale-handle recovery at the manager lifecycle boundary, await or coordinate with the active sync/reindex operation before reopening, and cover a real atomic database swap plus overlapping searches without adding config or a permanent fallback mode.

## Reproduction Assessment

Yes at the problem-class level: current source atomically replaces the database and related reports describe long-lived readers remaining on the old file. The PR test itself does not reproduce that real swap because it only mocks one metadata read.

## Solution Assessment

No. Reopening a stale reader is directionally appropriate, but the proposed per-search delay and handle replacement bypass the manager’s synchronization owner; recovery should be deduplicated and coordinated with reindex completion.
