## Summary

The proposed retry targets a real memory-search failure class, but it directly closes and replaces the manager’s shared SQLite handle from an unsynchronized search path. Current main serializes sync work but permits overlapping searches to use that handle, so this recovery can turn one transient metadata read into failures in another search or sync. The supplied runtime output is useful, but it injects the missing read rather than reproducing the reported concurrent rebuild or stale-handle path.

## What This Changes

The PR retries memory search once after a 500 ms delay by reopening SQLite, resetting vector state, exposing recovery counters in status, and adding metadata-recovery tests.

## Best Possible Solution

Use one manager-owned database lifecycle barrier or isolated atomic-reindex context so incomplete or closing SQLite handles are never exposed to concurrent searches; refresh from a concrete reindex-completion or file-identity signal and preserve hard failure for persistent identity mismatches.

## Reproduction Assessment

Yes at the source level: current searches and index lifecycle operations share a mutable SQLite handle without a search-side replacement barrier. The exact reported race was not independently run, and the submitted proof injects the missing read rather than producing it through concurrent rebuild activity.

## Solution Assessment

No. A bounded retry is plausible as a final defensive step, but directly closing the shared handle after a timer is not the safest owner boundary; lifecycle serialization or isolated reindex publication addresses the root invariant.
