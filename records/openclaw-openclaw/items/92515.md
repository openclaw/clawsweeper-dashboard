## Label Justifications

- `P1`: The PR targets an active memory_search regression that can make agent memory recall unavailable for real users.
- `merge-risk: 🚨 session-state`: The patch changes index-identity recovery in the memory search path and can leave concurrent memory reads with stale or inconsistent state.

## Summary

Keep open. The underlying memory_search failure is real and active, but this branch is not merge-ready because its recovery is a fixed-delay retry that closes and replaces the shared SQLite handle without coordinating concurrent memory operations.

## What This Changes

The PR adds metadata-recovery status counters and a one-shot 500 ms delayed SQLite reopen/retry around `MemoryIndexManager.search()` when index metadata is missing.

## Best Possible Solution

Use a manager-owned database lifecycle barrier or isolated reindex publication path, likely coordinated with https://github.com/openclaw/openclaw/pull/90453, while preserving hard failures for persistent identity mismatches.

## Reproduction Assessment

Yes, at the source level: current main can expose transient reindex DB state to search/status reads, and related reports show live `index metadata is missing` failures. I did not run a live current-main reproduction in this read-only review.

## Solution Assessment

No. This is a plausible defensive fallback, but the best fix is lifecycle serialization or isolated reindex publication rather than a timer and uncoordinated handle reopen.

Full review comments:

- **[P1] Serialize the SQLite handle replacement:** `extensions/memory-core/src/memory/manager.ts:630-632`
  - body: `reopenDatabaseForMetadataRecovery()` closes and replaces the manager-wide `this.db` without excluding another `search()`, `status()`, or `sync()` that may already be using that handle. Overlapping memory searches are part of the reported failure scenario, so one recovery can invalidate another live query; move this through a shared database lifecycle barrier or isolate handles before retrying.
  - confidence: 0.95
- **[P1] Tie retry to a reindex lifecycle signal:** `extensions/memory-core/src/memory/manager.ts:611-614`
  - body: The fixed 500 ms wait does not establish that an atomic rebuild, WAL/checkpoint transition, or external file swap has completed. A slower rebuild can retry too early and fail as before, while an already-published replacement is delayed unnecessarily; wait for the owned sync/reindex operation or detect backing-file replacement instead of using elapsed time alone.
  - confidence: 0.9
