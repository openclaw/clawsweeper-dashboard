## Label Justifications

## Summary

Keep open: this PR targets a real memory_search missing-metadata failure, but it is not merge-ready because the retry is time-based and replaces the shared SQLite handle from the search path without lifecycle serialization; the supplied proof shows an injected miss, not the claimed concurrent rebuild/WAL path.

## What This Changes

This PR adds a memory-core search wrapper that retries once after missing index metadata by sleeping, reopening SQLite state, and exposing metadata recovery counters, plus new regression tests.

## Best Possible Solution

Use one manager-level lifecycle fix, likely consolidated with https://github.com/openclaw/openclaw/pull/90453, that keeps search/status readers off half-built reindex state while preserving true identity mismatch failures.

## Reproduction Assessment

Yes, source-reproducible: current main can still expose the manager to the temp DB during full reindex and the tool fails closed on post-search status. I did not run the live race in this read-only review, but linked reports provide live concurrent evidence.

## Solution Assessment

No. A retry can be a defensive fallback, but the best fix is manager-level reindex/read serialization or an isolated publication path; the current timer plus shared-handle reopen is too fragile.
