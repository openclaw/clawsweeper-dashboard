## Summary

Keep this issue open: current main still does not accumulate ordinary non-error incremental Feishu streaming `final` chunks, the latest v2026.6.8 release still has the older final-branch overwrite, and the linked fixing PR is open, unmerged, and still review/proof-gated.

## What This Changes

The issue reports that Feishu streaming replies in OpenClaw 2026.6.1 replace the visible card with each latest token instead of accumulating streamed text.

## Best Possible Solution

Repair, prove, and land or replace the linked Feishu plugin-local final-chunk accumulation PR, then use the broader Feishu streaming tracker for any remaining long-response, stale-content, or duplicate-content behavior.

## Reproduction Assessment

Yes, at source level: ordinary non-error streaming final chunks still reduce to the latest chunk and are assigned as the whole `streamText` buffer. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Yes for the issue boundary: the correct fix belongs in the Feishu streaming-card buffer because the channel owns how final, partial, and block payloads are rendered into CardKit updates. The current linked PR is the right implementation lane but still needs repair and proof before merge.
