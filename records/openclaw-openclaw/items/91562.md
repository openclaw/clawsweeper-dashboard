## Label Justifications

- `P1`: A recent bundled-channel regression makes real Feishu users receive visibly incomplete streamed replies on current releases.
- `impact:message-loss`: Previously generated assistant text is suppressed from the visible card when each final fragment replaces the accumulated reply.

## Summary

Current main still has the reported Feishu final-fragment overwrite path, and the issue is paired with an open PR that explicitly claims it. The issue remains necessary until a correct fix is merged; related reports show adjacent CardKit semantics that should be checked during review, not used to close this report.

## What This Changes

Fix Feishu streaming replies so incremental final fragments preserve previously streamed text instead of leaving only the latest fragment.

## Best Possible Solution

Keep the issue open and land a Feishu-plugin fix that normalizes incremental final fragments while preserving distinct-final behavior, backed by live Feishu proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes at source level: successive `final` payloads repeatedly execute `streamText = text`, reproducing the reported latest-fragment-only result. A live Feishu and DeepSeek run has not yet confirmed the upstream payload sequence.

## Solution Assessment

Unclear: the Feishu dispatcher is the correct ownership boundary, but the proposed merge heuristic must be validated against live payloads and the existing distinct-final and CardKit update semantics before it is the best fix.
