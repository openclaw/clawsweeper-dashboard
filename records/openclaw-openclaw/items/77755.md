## Summary

This appears to be a valid WhatsApp command-authorization bug on current main: the LID-to-phone resolver supports account auth directories, but the group command/owner paths still drop that authDir before comparing the sender to the owner allowlist.

## What This Changes

Accept WhatsApp group control commands from the configured owner when Baileys reports the sender as a resolvable LID JID.

## Best Possible Solution

Thread the resolved WhatsApp account authDir through command authorization and group owner gating, then add a regression test for /new or /stop from a LID sender mapped back to the configured owner.

## Reproduction Assessment

Yes from source inspection, but not from a live run. A current-main group command message with senderJid set to a LID and only an authDir reverse mapping cannot be resolved by resolveWhatsAppCommandAuthorized or isOwnerSender because neither path passes authDir to the identity helpers.

## Solution Assessment

Yes, the narrow fix is to pass the existing account authDir through the WhatsApp-owned authorization and owner-gating call sites. That preserves the existing LID resolver and avoids changing core command policy or adding config.
