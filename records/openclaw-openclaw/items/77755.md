## Summary

Keep open: current main and the latest release still omit authDir-backed LID resolution in the WhatsApp group command authorization and owner-bypass paths, while the linked candidate fix at https://github.com/openclaw/openclaw/pull/93379 is open rather than merged.

## What This Changes

This issue asks for WhatsApp group control commands from a configured owner to authorize when Baileys reports the sender as a LID JID that can be mapped through the account authDir.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/93379 so account.authDir is threaded through WhatsApp command authorization and owner bypass, with focused regression coverage for LID sender mapping.

## Reproduction Assessment

Yes. Source inspection shows current main resolves WhatsApp command sender/self identities without authDir at the implicated call sites, while the existing resolver can map LID JIDs through authDir when it is provided.

## Solution Assessment

Yes. The narrowest maintainable fix is to pass the existing account authDir into the existing WhatsApp identity helpers at command authorization and owner-bypass call sites, rather than adding a new resolver, config option, or parallel authorization path.
