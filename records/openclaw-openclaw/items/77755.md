## Summary

Keep open: current main and the latest release still have the WhatsApp LID-to-phone mapping helper, but the reported group command authorization paths do not pass the account authDir needed to use it.

## What This Changes

The issue reports that WhatsApp group control commands from an owner can be ignored when Baileys reports the sender as a LID JID instead of a phone JID.

## Best Possible Solution

Thread the existing WhatsApp account authDir through command authorization and group owner-bypass identity resolution, then add regression coverage for LID sender authorization in group commands.

## Reproduction Assessment

Yes at source level: a group message whose sender is only a LID JID cannot resolve through the account authDir in the current command authorization and owner-bypass call sites. I did not run a live WhatsApp transport scenario.

## Solution Assessment

Yes: the narrowest maintainable fix is to thread the existing account authDir through the affected WhatsApp plugin identity-resolution call sites and add focused regression tests.
