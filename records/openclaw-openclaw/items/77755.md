## Summary

Keep open: current main and v2026.6.6 still omit `authDir` from the WhatsApp group command and owner-bypass identity call sites, so LID-only senders cannot use the existing account mapping before command authorization.

## What This Changes

The issue requests that WhatsApp group control commands authorize the configured owner when Baileys reports the sender as an `@lid` JID that maps back to the owner's phone identity.

## Best Possible Solution

Thread the existing WhatsApp account `authDir` through command authorization and group owner-bypass identity resolution, then add focused regression coverage for LID sender authorization in group commands.

## Reproduction Assessment

Yes at source level: a group command whose sender is only a LID JID cannot resolve through the account `authDir` because the current command authorization and owner-bypass call sites omit it. I did not establish a live WhatsApp transport repro in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable fix is to thread the existing account auth directory into the affected WhatsApp plugin identity-resolution calls and add regression tests. A new config option or broader access-policy change is unnecessary.
