## Summary

Keep open: current main still drops the WhatsApp account authDir before command authorization and group owner-bypass checks, so an owner reported only as an @lid sender can fail the existing allowlist path. The later duplicate at https://github.com/openclaw/openclaw/issues/77756 should stay linked to this earlier canonical report rather than replacing it.

## What This Changes

The issue asks WhatsApp group owner commands to authorize the configured owner when Baileys reports the sender as an @lid JID that maps back through the account auth directory.

## Best Possible Solution

Keep this issue open as the canonical bug and fix the WhatsApp plugin by threading the existing account authDir through command authorization and owner-bypass identity resolution with focused regressions.

## Reproduction Assessment

Yes, at source level. A group WebInboundMessage with senderJid set to an @lid value, no senderE164, and an authDir reverse mapping cannot be authorized by the current command and owner-bypass paths because those paths do not pass authDir into the identity helpers.

## Solution Assessment

Yes. The narrow maintainable fix is to pass the existing WhatsApp account authDir through the WhatsApp-owned authorization and group-gating call sites, reusing the current LID resolver instead of changing core policy or adding config.
