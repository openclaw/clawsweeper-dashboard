## Summary

Close: current main addresses the reported operator-facing failure by warning when a routed/channel-bound agent lacks the `message` tool, while the agent prompt carries the policy-filtered effective tool names so missing tools are not hidden from the model context.

## What This Changes

The issue asks for agents to handle missing `message` tool access truthfully and for `openclaw doctor` to warn when channel-routed agents cannot use explicit message actions.

## Best Possible Solution

Keep the shipped doctor diagnostic and existing policy-filtered tool inventory in the runtime prompt; any broader always-on self-introspection tool should be opened as a separate product/API request.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the original source path: an explicit allowlist without `message` removes that tool, and current main now warns for the routed-agent mismatch instead of silently accepting it. I did not live-run Discord/Ollama to reproduce the exact model wording.

## Solution Assessment

Is this the best way to solve the issue? Yes for this cleanup: the deterministic doctor warning catches the actionable misconfiguration, and the runtime prompt already lists the effective policy-filtered tool names. A separate `tools.list_self` style tool would be broader API work, not necessary to close the fixed operator bug.
