## Summary

Keep open: the doctor diagnostic has landed, but the issue explicitly requested a second runtime self-tool-awareness path and the merged PR deliberately left that unimplemented. Current main still has no `tools.list_self`/`whoami` style tool or explicit missing-`message` diagnostic beyond the policy-filtered available-tool list.

## What This Changes

Request to stop agents from fabricating `message` capabilities when policy omits the tool and to have `doctor` warn about channel-bound agents missing `message`.

## Best Possible Solution

Keep the shipped doctor warning, then either add compact resolved-tool-policy self-awareness to the runtime or split that product/API decision into a dedicated follow-up before closing this issue.

## Reproduction Assessment

Yes, for the source-level limitation: an explicit allowlist without `message` still filters the tool out, and current prompt/runtime surfaces do not provide a deterministic self-list or missing-tool explanation. I did not live-run Discord/Ollama to reproduce the exact model wording.

## Solution Assessment

No for the full issue as currently resolved: the merged doctor warning is the right narrow operator diagnostic, but it intentionally does not solve runtime self-tool-awareness. A maintainer should choose whether that belongs in prompt context, an always-available introspection tool, or a narrower follow-up.
