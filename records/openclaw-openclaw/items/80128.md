## Summary

Keep open: current main still has a source-level doctor gap for explicit tool allowlists that omit `message`, and the prompt-side tool list only partially addresses the model-confabulation part of the report.

## What This Changes

The issue asks OpenClaw to make agents accurately know/report when `message` is unavailable and to make `doctor` warn when channel-bound agents lack the `message` tool.

## Best Possible Solution

Add a shared policy-aware `message` availability check, use it from doctor diagnostics, and clarify the existing filtered tool prompt so absent tools are treated as unavailable; avoid a new always-available introspection tool unless maintainers choose that product direction.

## Reproduction Assessment

Yes, at source level. Runtime filtering removes `message` for explicit allowlists, while the current doctor helper ignores those allowlists even though an existing test expects that exact warning shape.

## Solution Assessment

No, current main is not the best solution because doctor and runtime resolve `message` availability differently. The narrow fix is to share or mirror the effective tool-policy logic and add focused regression coverage.
