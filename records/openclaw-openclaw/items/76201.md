## Summary

Keep open: current main still shows a credible `before_tool_call` gap for native Pi/Anthropic tool execution, with `before_tool_call` depending on the mutable global runner while `after_tool_call` can use the run-scoped runner captured by the embedded subscription path.

## What This Changes

The issue requests that plugin `before_tool_call` handlers fire for native Anthropic/Pi `exec` tool calls the same way `after_tool_call` already does.

## Best Possible Solution

Native Pi tool execution should resolve the same active plugin hook runner for `before_tool_call` that the embedded subscription path uses for `after_tool_call`, with regression coverage for gateway-bindable Anthropic/Pi `exec`.

## Reproduction Assessment

Yes. The issue supplies a concrete minimal plugin and native `exec` turn on `v2026.4.29`, and current main still shows the same before/after runner asymmetry in source.

## Solution Assessment

Unclear until the hook-runner ownership contract is confirmed. The narrow maintainable direction appears to be threading the active run-scoped hook runner into native pre-exec wrapping instead of relying only on a mutable global singleton.
