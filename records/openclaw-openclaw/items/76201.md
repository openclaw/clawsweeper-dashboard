## Summary

Keep open: current main still has a plausible `before_tool_call` dispatch gap for native Pi/OpenClaw tools, and the code matches the reporter's split where `after_tool_call` can see the run-scoped hook runner while `before_tool_call` falls back to the global runner.

## What This Changes

Ensure plugins' `before_tool_call` handlers fire before native exec/tool execution in the Anthropic embedded harness.

## Best Possible Solution

Thread the active embedded-run hook runner/context into native Pi tool definitions so `before_tool_call` and `after_tool_call` resolve the same plugin registry, with regression coverage for gateway-bindable native exec.

## Reproduction Assessment

Yes. The issue provides a concrete minimal plugin and native `exec` turn on `v2026.4.29`, and current code inspection shows a credible source-level path for the observed before/after hook asymmetry.

## Solution Assessment

Unclear until the intended runner contract is confirmed. The narrowest maintainable direction appears to be run-scoped hook context for native tool wrapping rather than relying solely on the mutable global singleton during tool execution.
