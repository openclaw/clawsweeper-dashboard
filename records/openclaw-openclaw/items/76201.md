## Label Justifications

- `P1`: A documented pre-exec plugin policy hook may be skipped for native tool execution, breaking a security-relevant workflow for real plugin users.
- `impact:security`: The issue is about plugin authorization and audit hooks not running before native `exec`, which can bypass operator policy controls.

## Summary

Keep open: current main still has the native Pi `before_tool_call` path resolved through the global hook runner only, while `after_tool_call` has a run-context fallback; the related Codex relay reports do not cover this Anthropic/Pi native `exec` path.

## What This Changes

Investigate a bug where plugin-registered `before_tool_call` hooks do not fire for native `exec` under the Anthropic/Pi harness.

## Best Possible Solution

Native Pi tool wrapping should use the active gateway or run hook runner for `before_tool_call`, with regression coverage where an external plugin observes or blocks native `exec`.

## Reproduction Assessment

Yes at source level, but not as a fresh live run on current main. The reporter supplied 2026.4.29 and 2026.5.7 gateway traces, and current main still shows the before/after hook-runner asymmetry that explains the symptom.

## Solution Assessment

Unclear until maintainers confirm the hook-runner ownership contract. The maintainable direction is to carry the active gateway/run hook runner into native before-hook dispatch rather than adding another broad global lookup fallback.
