## Summary

Keep open: current main still has a credible native Pi/Anthropic `before_tool_call` dispatch gap, and I found no merged fix or canonical replacement issue that covers this exact native `exec` path.

## What This Changes

The issue reports that plugin `before_tool_call` hooks do not fire for native Anthropic/Pi `exec` tool calls even though the same plugin's `after_tool_call` hook fires.

## Best Possible Solution

Native tool wrapping should resolve the active run or gateway plugin hook runner for `before_tool_call`, with regression coverage for a gateway-bindable Anthropic/Pi native `exec` call where a plugin before hook observes or blocks the call.

## Reproduction Assessment

Yes, source-reproducible with high confidence, though I did not live-run the gateway. The supplied minimal plugin repro is concrete, and current main still routes wrapped native tools through `runBeforeToolCallHook()` while that helper consults only the global hook runner for plugin hooks.

## Solution Assessment

Unclear until the hook-runner ownership contract is confirmed. The narrow maintainable direction appears to be threading the active run/gateway hook runner into native tool wrapping rather than relying only on the mutable global singleton.
