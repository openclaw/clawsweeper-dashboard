## Summary

Keep open: current main still has a credible native Pi/Anthropic `before_tool_call` dispatch gap, and the latest issue comment re-verifies the same behavior on v2026.5.7. Current source still routes the native-tool before hook through the global hook runner while the after-hook path can use a run-scoped runner, so this is not covered by the closed loopback MCP fix.

## What This Changes

The issue reports that plugin-registered `before_tool_call` hooks do not fire for native `exec` calls in the Anthropic/Pi harness while `after_tool_call` does.

## Best Possible Solution

Thread or preserve the active gateway/run hook runner for native tool before-hooks, then add regression coverage that proves a plugin can observe or block a Pi/Anthropic native `exec` before execution.

## Reproduction Assessment

Yes. The issue provides concrete release repros, and current main still shows a source-level path where native tools call `before_tool_call` through the global runner while `after_tool_call` can use a run-scoped runner.

## Solution Assessment

Unclear until the hook-runner ownership contract is confirmed. The narrow maintainable fix appears to be making native tool wrapping use the active gateway/run hook runner rather than relying only on the mutable global singleton.
