## Summary

Keep open: current main still matches a credible native Pi/Anthropic `before_tool_call` dispatch gap, and the only related PR covers the separate loopback MCP path rather than native `exec`. The recent 2026.5.7 recheck in the issue is useful runtime evidence that this is not only a 2026.4.29 report.

## What This Changes

Plugin `before_tool_call` hooks should fire before native Anthropic/Pi `exec` tool execution so plugins can observe, rewrite, block, or require approval.

## Best Possible Solution

Native Pi/Anthropic tool wrapping should resolve the active gateway or run hook runner for `before_tool_call`, with regression coverage where a plugin observes or blocks native `exec`.

## Reproduction Assessment

Yes, source-reproducible with high confidence, though I did not live-run the gateway. The issue includes real 2026.4.29 and 2026.5.7 gateway traces, and current main still routes wrapped native tools through a helper that only consults the global hook runner while `after_tool_call` has a run-context fallback.

## Solution Assessment

Unclear as an implementation until the hook-runner ownership contract is confirmed. The maintainable direction is to carry the active gateway/run hook runner into native before-hook dispatch rather than adding another ad hoc global lookup branch.
