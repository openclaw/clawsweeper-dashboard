## Summary

Keep open: current main still has a source-visible native Pi/Anthropic `before_tool_call` dispatch gap, and the latest issue comment reports the same failure on the current shipped v2026.5.7 release.

## What This Changes

The issue reports that plugin-registered `before_tool_call` handlers do not run before native `exec` tool execution in the Anthropic/Pi harness, even though `after_tool_call` runs for the same plugin.

## Best Possible Solution

Native Pi tool execution should resolve the active run or gateway plugin hook runner before executing wrapped native tools, with regression coverage proving a plugin `before_tool_call` handler can observe and block native `exec`.

## Reproduction Assessment

Yes, we have a high-confidence source-reproducible path: the reporter supplied a minimal plugin and repeated gateway `exec` traces, while current main still routes native tools through a wrapper whose before-hook dispatch depends only on the global hook runner. I did not live-run the gateway in this read-only pass.

## Solution Assessment

Unclear until maintainers confirm the hook-runner ownership contract. The likely narrow solution is to pass or resolve the active gateway/run hook runner in the native before-hook wrapper, rather than adding another ad hoc dispatch branch.
