## Summary

Keep open: current `main` still has a credible native Anthropic/Pi `before_tool_call` dispatch gap. The source path still routes wrapped native tools through a helper that checks only the global hook runner, while `after_tool_call` can use the active run hook runner, matching the reporter's repeated 2026.5.7 verification.

## What This Changes

The issue reports that plugin-registered `before_tool_call` hooks do not fire for native `exec` calls through the Anthropic/Pi harness.

## Best Possible Solution

Native tool wrapping should resolve the active run or gateway plugin hook runner for `before_tool_call`, with regression coverage for a gateway-bindable Anthropic/Pi native `exec` call where a plugin before hook observes or blocks execution.

## Reproduction Assessment

Yes, source-reproducible with high confidence: the issue provides a concrete plugin and real gateway turns, and current source still shows the before path using only the global runner while the after path can use run context. I did not live-run the gateway in this read-only review.

## Solution Assessment

Unclear until the intended hook-runner ownership contract is confirmed. The narrow maintainable direction appears to be threading the active run/gateway hook runner into native tool wrapping rather than adding another ad hoc singleton lookup.
