## Summary

Keep open: the queue-wide cleanup is useful, but it does not yet fix the linked duplicate approval cascade because the failing paths still carry fresh approval/run IDs in the text and context key, and the PR has no after-fix real behavior proof for that runtime path.

## What This Changes

The branch replaces consecutive-only system-event dedupe with queue-wide exact `(text, contextKey)` dedupe, preserves `lastContextKey` on skipped/contextless enqueues, removes `lastText`, and adds regression tests.

## Best Possible Solution

Carry or derive a stable exec intent key through approval creation, forwarding, node/gateway exec events, and system-event enqueueing, or cancel stale pending approval events before retrying; keep the queue-wide cleanup only if it remains correct under that identity.

## Reproduction Assessment

Yes, at source level. A retry of the same command under a fresh approval id or run id is enough to bypass this PR's exact `(text, contextKey)` comparison, and the linked issue plus reporter comment provide live logs/screenshots for that cascade path.

## Solution Assessment

No. Full-queue exact-pair dedupe is a useful helper, but it is not the narrowest complete fix for the reported duplicate approvals because the failing path changes the id-bearing fields between retries.
