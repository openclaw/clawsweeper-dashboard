## Summary

Keep open. The linked bug is about the same exec intent being retried under fresh approval/run IDs, but this PR only dedupes exact `(text, contextKey)` pairs, so the central cascade can still bypass it. The PR also needs the required user-facing changelog entry before merge.

## What This Changes

The PR changes `src/infra/system-events.ts` to scan the in-memory system-event queue for exact `(text, contextKey)` duplicates, preserve `lastContextKey` on skipped or contextless enqueues, remove `lastText`, and add focused regression tests.

## Best Possible Solution

Carry or derive a stable exec intent dedupe key from the approval/failover layer into system-event and approval-forwarding paths, while still allowing distinct context-specific events. The final patch should include a regression where the same command/session intent retries under different approval or run IDs, plus a `CHANGELOG.md` Fixes entry.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to enqueue or forward the same exec command twice with different approval/run IDs; current main embeds those IDs in approval text and exec event context, while this PR compares exact text and exact context key.

## Solution Assessment

No. Queue-wide exact `(text, contextKey)` dedupe is useful for non-consecutive exact duplicates, but it is not the narrowest complete fix for the reported fresh-ID retry cascade; a stable intent-level key or normalization of ID-bearing fields is safer.
