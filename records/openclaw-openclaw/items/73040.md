## Summary

Keep open. This PR improves the system-event queue for exact non-consecutive duplicates and fixes useful `lastContextKey` edge cases, but it does not fully fix the claimed heartbeat-failover cascade because fresh exec approval/run IDs are still embedded in the fields the new dedupe compares.

## What This Changes

The branch changes `src/infra/system-events.ts` to queue-scan duplicate `(text, contextKey)` system events, preserve `lastContextKey` only for queued events with a supplied context key, remove `lastText`, and add four regression tests in `src/infra/system-events.test.ts`.

## Best Possible Solution

Carry or derive a stable exec intent dedupe key from the approval or heartbeat-failover layer into the system-event path, while preserving session/channel context boundaries. The final patch should keep the useful `lastContextKey` fixes and add a regression where the same command/session intent retries under different approval or run IDs.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to forward or enqueue the same exec command twice for the same session with fresh approval/run IDs; the current main code embeds those IDs in approval text and event context, and this PR compares those exact fields.

## Solution Assessment

No. Queue-wide exact `(text, contextKey)` matching is a useful partial cleanup, but the claimed cascade needs a stable intent-level key or normalization of ID-bearing fields before this is the narrowest complete fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Use a stable key for approval retries:** `src/infra/system-events.ts:117-118`
  - body: `findDuplicateInQueue` only suppresses entries when both the full text and `contextKey` match. The reported cascade creates fresh approval/run IDs, and current main includes those IDs in approval text and exec event context, so the same command retried with a new ID bypasses this PR. Please dedupe on a stable command/session intent or normalize the ID-bearing fields, then add a regression with different IDs.
  - confidence: 0.86
