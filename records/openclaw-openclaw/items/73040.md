## Summary

Keep open. The PR is useful for exact non-consecutive system-event duplicates and context-key clobbering, but it does not close the linked heartbeat-failover cascade because the approval/run ID is still part of the prompt text and system-event context, so a retry with a fresh ID bypasses exact `(text, contextKey)` matching. The branch also lacks the required user-facing changelog entry.

## What This Changes

The branch changes `src/infra/system-events.ts` to scan each session queue for duplicate exact `(text, contextKey)` pairs, preserve `lastContextKey` on skipped or contextless enqueues, remove `lastText`, and add four focused regression tests.

## Best Possible Solution

Carry or derive a stable exec intent key from the approval/heartbeat-failover layer into system-event and approval-forwarding behavior, while preserving distinct context-specific events. The final patch should include a regression where the same command/session intent retries under different approval or run IDs, plus the required `CHANGELOG.md` entry.

## Reproduction Assessment

Yes. A focused code-level reproduction is to create or forward the same exec command/session intent twice while generating fresh approval/run IDs; current main embeds those IDs in prompt text and event context, so exact `(text, contextKey)` dedupe will not suppress the retry.

## Solution Assessment

No. Queue-wide exact `(text, contextKey)` dedupe is a reasonable helper for exact duplicate system events, but the linked cascade needs a stable intent-level key or normalization of ID-bearing fields to be complete.
