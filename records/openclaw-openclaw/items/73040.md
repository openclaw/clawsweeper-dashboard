## Summary

Keep open. The queue cleanup is focused, but it does not fix the linked fresh-approval-id cascade at https://github.com/openclaw/openclaw/issues/69478: current approval prompts include a new ID in the rendered text, the context key is derived from that ID, and the forwarder/followup path can bypass `enqueueSystemEvent`, so the supplied exact-pair queue proof does not prove the reported Telegram cascade is gone.

## What This Changes

The branch replaces consecutive `lastText` suppression in `src/infra/system-events.ts` with queue-wide `(text, contextKey)` dedupe, preserves `lastContextKey` only after accepted contextful events, and adds four regression tests.

## Best Possible Solution

Fix the reported cascade by carrying a stable exec intent key through approval registration, forwarding, failover, and followup delivery, while keeping the queue-wide dedupe only as a scoped infra cleanup if maintainers still want it.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main creates fresh approval IDs/context keys and includes the ID in approval prompt text, while reporter and production comments show the same fresh-ID forwarder failure pattern. I did not run a live Telegram cascade in this read-only review.

## Solution Assessment

No. Exact `(text, contextKey)` queue dedupe is a useful local cleanup, but the linked bug needs stable exec intent identity or normalization before the exec approval forwarder and followup paths emit fresh prompts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Deduplicate retries on stable exec intent:** `src/infra/system-events.ts:117-118`

Please key or normalize this around the stable exec intent, with a regression using different approval IDs for the same command.
