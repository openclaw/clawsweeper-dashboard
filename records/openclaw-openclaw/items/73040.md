## Summary

Keep this PR open but do not merge it as-is: the queue cleanup is useful, but the PR still claims to fix and close the approval cascade while current code and reporter follow-up show the visible cascade flows through ID-keyed exec approval forwarding/followups that this diff does not change.

## What This Changes

The PR replaces adjacent-only system-event duplicate suppression with queue-wide `(text, contextKey)` dedupe, preserves `lastContextKey` only after accepted contextful events, removes `lastText`, and adds four system-event regression tests.

## Best Possible Solution

Carry a stable exec intent key through approval registration, forwarding, failover, and followup delivery, and keep the queue-wide dedupe only as a scoped system-events cleanup if maintainers want it.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main creates fresh approval IDs/context keys and includes the ID in rendered approval text, while reporter and production comments show the same fresh-ID forwarder failure pattern. I did not run a live Telegram cascade in this read-only review.

## Solution Assessment

No. Exact `(text, contextKey)` queue dedupe is a useful local cleanup, but it is not the narrowest complete fix for the claimed issue because the approval forwarder/followup path needs stable exec intent identity or stale-pending cancellation.

## Review Findings

- **[P2] Deduplicate retries on stable exec intent:** `src/infra/system-events.ts:117-118`
