## Summary

Keep open. The PR is aimed at a real regression from #69478 and the queue-wide dedupe is useful for exact duplicate system events, but it is not merge-ready because the reported heartbeat-failover cascade creates fresh approval/run IDs and the proposed exact `(text, contextKey)` comparison does not collapse those retries. It also needs the required changelog entry for a user-facing approval-prompt fix.

## What This Changes

The branch changes `src/infra/system-events.ts` to scan the session queue for duplicate exact `(text, contextKey)` events, preserve `lastContextKey` on skipped or contextless enqueues, remove `lastText`, and add focused regression tests.

## Best Possible Solution

Carry or derive a stable exec intent dedupe key from the approval or heartbeat-failover layer into the system-event and approval-forwarding paths, while preserving distinct context-specific events. The final patch should include a regression where the same command/session intent retries under different approval or run IDs, plus a user-facing `CHANGELOG.md` entry.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to enqueue or forward the same exec command twice with different approval/run IDs; current main embeds those IDs in approval text and exec event text/context, while this PR compares exact text and exact context key.

## Solution Assessment

No. Queue-wide exact `(text, contextKey)` dedupe fixes non-consecutive exact duplicates, but the reported cascade needs a stable intent-level key or normalization of ID-bearing fields to be complete.
