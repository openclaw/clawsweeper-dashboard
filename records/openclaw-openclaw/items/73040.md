## Summary

Keep open: the branch is a focused queue cleanup, but it is not merge-ready for the claimed heartbeat approval cascade because fresh approval/run IDs defeat exact `(text, contextKey)` matching and the reported forwarder path bypasses `enqueueSystemEvent`.

## What This Changes

The branch changes `src/infra/system-events.ts` to dedupe queued system events by exact `(text, contextKey)`, preserve `lastContextKey` only for non-duplicate keyed enqueues, remove `lastText`, and add regression tests.

## Best Possible Solution

Carry or derive a stable exec intent identity through approval request creation, failover, forwarding, and system-event enqueueing, then cover mixed-ID retries without collapsing distinct approvals.

## Reproduction Assessment

Yes, at source level. Current main and this PR both show that fresh approval/run IDs enter approval text and context keys, while the PR only matches exact text/context pairs; I did not run a live gateway reproduction.

## Solution Assessment

No. Queue-wide exact-pair dedupe is a useful helper, but the reported cascade needs stable approval intent dedupe or forwarder-side coalescing plus a regression with fresh IDs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use a stable approval intent key:** `src/infra/system-events.ts:117`
  - body: `findDuplicateInQueue` only suppresses entries when the full event text and normalized `contextKey` both match. The reported cascade creates fresh approval/run IDs, and current main includes those IDs in approval text and exec context keys while the forwarder sends directly to channels, so the same command retried with a new ID still gets another prompt. Please dedupe on a stable command/session intent or coalesce the forwarder path, then add a mixed-ID retry regression.
  - confidence: 0.88
