## Summary

Keep this PR open: the latest head appears to repair the prior merge-break, but the exact `(text, contextKey)` queue dedupe still does not fix the fresh-ID exec-approval cascade described by the linked report, and the PR still lacks after-fix real behavior proof.

## What This Changes

The branch changes `src/infra/system-events.ts` to scan the in-memory queue for duplicate `(text, contextKey)` events, preserve context state only after accepted enqueues, remove `lastText`, and add four regression tests.

## Best Possible Solution

Keep the queue-wide dedupe only as a narrow infra cleanup if maintainers want it, but fix the linked bug by carrying a stable exec intent key through approval registration, failover, forwarding, and system-event paths before closing https://github.com/openclaw/openclaw/issues/69478.

## Reproduction Assessment

Yes. Source inspection plus reporter/production logs give a high-confidence reproduction shape: retry the same exec intent under fresh approval IDs/context keys and the PR's exact-pair queue check will not coalesce the forwarder-delivered prompts.

## Solution Assessment

No. Queue-wide exact-pair dedupe is a useful local cleanup, but the maintainable fix for the reported cascade needs stable approval intent identity or ID normalization at the exec approval and forwarder boundary.
