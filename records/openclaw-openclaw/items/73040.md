## Summary

Keep open: the PR is useful but not merge-ready because exact `(text, contextKey)` queue dedupe does not cover the reported failover cascade when approval or run IDs change, and the user-facing fix also lacks the required changelog entry.

## What This Changes

The PR changes system-event queuing to drop duplicate exact `(text, contextKey)` entries anywhere in the session queue, preserve `lastContextKey` on skipped/contextless enqueues, remove `lastText`, and add regression tests.

## Best Possible Solution

Carry or derive a stable exec intent dedupe key from the approval/failover layer into the system-event and approval-forwarding paths, preserve context separation, then add a mixed-ID retry regression plus a `CHANGELOG.md` fix entry.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is the same command/session retrying with fresh approval or run IDs; current main embeds those IDs in approval text and exec event context, while this PR only compares exact text and context.

## Solution Assessment

No. Full-queue exact-pair dedupe is a useful helper, but the narrow maintainable fix needs stable intent identity or ID normalization at the approval/failover boundary to match the reported cascade.
