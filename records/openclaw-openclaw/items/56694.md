## Summary

Keep open. Current main still has the reported diagnostic gap for `file` and `filePath`, but the larger read-tool contract is inconsistent: diagnostics tolerate `file_path`, execution validation requires canonical `path`, tests intentionally reject legacy aliases, and adjacent fingerprinting still recognizes path aliases. Closing would lose a real contract-alignment follow-up.

## What This Changes

The issue requests suppressing the embedded read-tool “without path” warning when a model supplies the read path as `file` or `filePath`.

## Best Possible Solution

Set one coherent read-tool parameter contract across schema, execution validation, diagnostics, display/sanitization, fingerprinting, docs, and tests: either restore alias support end to end or enforce canonical `path` everywhere and remove stale alias assumptions from non-execution surfaces.

## Reproduction Assessment

Yes for the warning path: a `handleToolExecutionStart` event for `read` with only `file` or `filePath` reaches the warning branch by static inspection. The separate claim that alias-only execution succeeds is not true on current main, because the read wrapper validates only `path` before dispatch.

## Solution Assessment

No: the issue’s warning-only suggested fix is not the best standalone change until maintainers decide the read-tool compatibility contract. The safer direction is a contract-alignment change with focused tests for diagnostics and execution behavior together.
