Keep open. Current main still has the narrow warning guard, but the reported alias-success premise no longer matches the current path-only execution contract, and the issue is already paired with an open closing PR at https://github.com/openclaw/openclaw/pull/56847.

## What This Changes

The issue asks that embedded read-tool start diagnostics treat `file` and `filePath` path aliases as present so they do not emit false missing-path warnings.

## Best Possible Solution

Keep this issue open while maintainers decide and land a coherent read-tool parameter contract across diagnostics, runtime validation, tests, display/fingerprint helpers, and docs.

## Reproduction Assessment

Yes for the narrow warning branch: a `read` start event with only `file` or `filePath` reaches the warning guard on current main. No for the full claimed successful-read false-positive path, because current execution validation rejects alias-only reads.

## Solution Assessment

No. The warning-only direction is not the best standalone solution on current main because it diverges from the canonical `path` execution contract; the safer fix is contract alignment first.
