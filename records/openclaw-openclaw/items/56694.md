## Summary

Keep open. Current main still has the narrow read-start warning guard, but current read execution requires canonical `path`, and this issue is already paired with the open closing PR at https://github.com/openclaw/openclaw/pull/56847.

## What This Changes

The issue requests that the embedded read-tool start warning recognize `file` and `filePath` as path aliases instead of warning when those keys are present.

## Best Possible Solution

Land one coherent read-tool parameter contract across diagnostics, validation, display/fingerprint helpers, docs, and tests, using the existing open PR or a replacement branch if maintainers choose alias compatibility.

## Reproduction Assessment

Yes for the narrow handler branch: source inspection shows `file` or `filePath` without `path` reaches the warning guard. No for the full successful-read false-positive claim on current main, because current execution validation rejects alias-only reads.

## Solution Assessment

No. A warning-only alias whitelist is not the best standalone fix on current main; the safer path is maintainer-approved contract alignment across read validation and diagnostics.
