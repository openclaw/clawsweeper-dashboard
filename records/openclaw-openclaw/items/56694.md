## Summary

Keep open. Current main still has the reported warning mismatch, but the original premise that all four aliases execute successfully is no longer true: runtime validation now requires canonical `path`, while diagnostics and adjacent display/fingerprint code still recognize some legacy aliases. Closing would lose a real contract-alignment issue.

## What This Changes

The issue asks for the embedded read-tool start warning to treat `file` and `filePath` as valid path arguments, matching the reporter's stated accepted alias set.

## Best Possible Solution

Define one coherent read-tool parameter contract across schema, runtime validation, diagnostics, display, fingerprinting, docs, and tests: either restore `path`, `file_path`, `file`, and `filePath` as accepted aliases end to end, or enforce canonical `path` everywhere and remove stale alias assumptions from non-execution surfaces.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: call `handleToolExecutionStart` for `read` with args containing only `file` or `filePath`, and the current guard reaches the warning branch because it only checks `path` and `file_path`; I did not run tests because this was a read-only review.

## Solution Assessment

No. The suggested warning-only fix is not the best standalone solution on current main because execution validation now intentionally rejects aliases; the safer fix is a maintainer-approved contract alignment across the related surfaces.
