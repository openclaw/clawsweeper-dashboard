## Summary

Keep open: current main and v2026.6.10 still finalize Discord progress drafts by editing the preview message into the final answer, and there is no preserve, append, or separate-final setting. This remains a valid Discord UX feature request, but it needs maintainer product/config direction before implementation.

## What This Changes

Request to add a Discord progress-streaming finalization option that preserves the progress draft and appends or sends the final answer separately instead of replacing it.

## Best Possible Solution

Have maintainers select a default-preserving Discord finalization policy, then implement it in the Discord plugin with schema/types/docs, regression coverage, and live Discord proof that chunking and mention safeguards still hold.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. The current behavior is nevertheless source-confirmed: Discord progress drafts can be edited into the final answer, and no preserve/append setting exists.

## Solution Assessment

Unclear. An opt-in finalization policy is plausible, but maintainers should first choose append versus separate-message behavior and account for Discord limits, notification behavior, and durable exposure of progress/debug content.

## Risks / Open Questions

- The exact UX is unsettled: same-message append must respect Discord's 2000-character limit, chunking, mention-edit safeguards, and final-edit fallback rules, while separate final delivery changes ordering and notification behavior.
- Preserving reasoning/tool progress can make transient debug content durable in Discord history; current security docs warn that reasoning, verbose, and trace output can expose tool args, URLs, plugin diagnostics, and data the model saw.
- Adding a new Discord streaming config surface needs maintainer approval because OpenClaw's config surface is intentionally kept small and compatibility-sensitive.
