## Summary

Keep open. Current main still lacks the Quick Settings TTS surface, and this draft PR is the active implementation candidate for that gap, but the patch has concrete correctness and validation defects that should be fixed before merge.

## What This Changes

The PR adds a Quick Settings Text-to-Speech card, UI TTS state/controllers, `tts.setVoice` and `tts.preview` gateway methods, `voiceByProvider` persistence, and default-voice application during synthesis.

## Best Possible Solution

Keep the feature direction, but land it only after the runtime facade is wired end to end, the new gateway methods have the same write-scope and validation behavior as existing TTS methods, preview cannot silently fall back from an explicit selection, and the changelog/tests/docs are brought in line with the new public surface.

## Reproduction Assessment

Yes. Current-main reproduction is a source check: Quick Settings has no TTS card and `tts.status` has no `voiceByProvider`. The patch defects are also reproducible by inspecting PR head: the new facade names are absent from `extensions/speech-core/runtime-api.ts`, and the new RPCs are absent from `METHOD_SCOPE_GROUPS`.

## Solution Assessment

No. A Quick Settings card plus narrow TTS gateway methods is a reasonable product direction, but this implementation is not yet the best solution because core gateway calls are not fully backed by the speech-core runtime API and the new RPCs do not match existing TTS authorization/error/fallback contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Export the new voice helpers from speech-core:** `src/plugin-sdk/tts-runtime.ts:115-120`
  - body: `src/tts/tts.ts` reaches speech-core through the lazy `tts-runtime` facade, which loads `extensions/speech-core/runtime-api.ts`. This patch adds `setTtsVoice` and `getTtsVoiceByProvider` to the facade and gateway, but the speech-core runtime API still does not export them, so `tts.setVoice` will not persist anything and `tts.status.voiceByProvider` is not backed by the runtime surface. Add the runtime-api exports and cover the facade/gateway path.
  - confidence: 0.94
- **[P2] Classify the new TTS RPCs as write methods:** `src/gateway/server-methods-list.ts:29-30`
  - body: Adding `tts.setVoice` and `tts.preview` to the method list without also adding them to `METHOD_SCOPE_GROUPS` leaves them unclassified, and `authorizeOperatorScopesForMethod` treats unclassified methods as admin-only. Write-scoped callers that can use `tts.enable`, `tts.convert`, and `tts.setProvider` would be rejected for these companion TTS actions. Add both methods to the write scope and update the method-scope test.
  - confidence: 0.9
- **[P2] Keep explicit previews on the selected provider:** `src/gateway/server-methods/tts.ts:300-306`
  - body: When the UI sends an explicit provider or voice, `tts.preview` builds overrides but then calls `textToSpeech` without `disableFallback`. If the selected provider fails, the preview can silently synthesize with a fallback provider and return audio that does not match the user's selected voice/provider. Pass `disableFallback` whenever a provider or voice override is present, matching `tts.convert`.
  - confidence: 0.88
- **[P2] Return invalid request for bad preview overrides:** `src/gateway/server-methods/tts.ts:290-298`
  - body: `resolveExplicitTtsOverrides` can throw for unknown providers or unsupported voice overrides. In `tts.convert` that error is mapped to `INVALID_REQUEST`, but here it falls into the outer catch and becomes `UNAVAILABLE`, making caller input errors look like backend outages. Wrap override resolution like `tts.convert` and return `INVALID_REQUEST`.
  - confidence: 0.86
- **[P2] Validate setVoice against the active provider registry:** `src/gateway/server-methods/tts.ts:265-275`
  - body: `tts.setProvider` canonicalizes with the runtime config and rejects ids not present in `getSpeechProvider`, but `tts.setVoice` only canonicalizes the raw string and then persists it. That lets callers save a voice for a provider id that is not registered in the current config/plugin set. Use the same `canonicalizeSpeechProviderId(provider, cfg)` plus `getSpeechProvider` validation as `tts.setProvider`.
  - confidence: 0.82
- **[P3] Replace the changelog placeholder:** `CHANGELOG.md:9`
  - body: The changelog entry would ship `Fixes #XXXX` and bot attribution. Replace the placeholder with the real related issue or PR reference, and use a valid human contributor attribution when one exists; if the only source author is automation, keep the credit in PR history instead of adding a bogus `Thanks` handle.
  - confidence: 0.95
