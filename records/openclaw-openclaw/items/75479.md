## Summary

Keep open. This PR is a useful implementation candidate for the linked TTS Settings request, but current main still lacks the feature and the patch is not merge-ready because the new runtime facade, method scopes, and preview/setVoice validation paths do not yet preserve existing TTS gateway contracts.

## What This Changes

The PR adds a Quick Settings TTS card, UI state/controller wiring, `tts.setVoice` and `tts.preview` gateway methods, persisted per-provider voice preferences in speech-core, tests, and a changelog entry.

## Best Possible Solution

Finish this feature as the implementation for #71487: keep the Quick Settings TTS card direction, but wire the speech-core runtime exports, classify the new RPCs under the correct operator scope, validate and map gateway errors like the existing TTS methods, preserve explicit preview provider selection, add focused gateway/UI coverage, and clean up docs/changelog metadata before merge.

## Reproduction Assessment

Yes. Current-main reproduction is source inspection: Quick Settings has no TTS card or TTS app wiring, and the gateway lacks `tts.setVoice`, `tts.preview`, and `voiceByProvider`; the PR defects reproduce by comparing the diff to the existing runtime facade and method-scope contracts.

## Solution Assessment

No. A Quick Settings card backed by narrow additive TTS RPCs is the right product direction, but this patch is incomplete because it leaves runtime exports, authorization scopes, request validation, preview fallback behavior, and changelog metadata inconsistent with existing patterns.

Full review comments:

- **[P2] Export the voice helpers from speech-core:** `src/plugin-sdk/tts-runtime.ts:115-120`
  - body: `src/tts/tts.ts` reaches speech-core through the lazy `tts-runtime` facade, which loads `extensions/speech-core/runtime-api.ts`. This patch adds facade entries for `setTtsVoice` and `getTtsVoiceByProvider`, but the runtime API does not export them, so the new gateway paths will fail when they try to use the real runtime surface.
  - confidence: 0.94
- **[P2] Classify the new TTS RPCs as write methods:** `src/gateway/server-methods-list.ts:29-30`
  - body: Adding `tts.setVoice` and `tts.preview` to the method list without adding them to `METHOD_SCOPE_GROUPS` leaves them unclassified. The existing method-scope tests require every listed method to be classified, and scoped callers that can use the companion TTS methods would otherwise hit the admin-only fallback.
  - confidence: 0.91
- **[P2] Validate setVoice against registered providers:** `src/gateway/server-methods/tts.ts:262-272`
  - body: `tts.setProvider` canonicalizes with the runtime config and rejects ids that are not registered, but `tts.setVoice` only canonicalizes the raw string before persisting it. Use the same registry-backed validation so callers cannot save voice preferences for providers that are not available in the active config/plugin set.
  - confidence: 0.85
- **[P2] Return invalid request for bad preview overrides:** `src/gateway/server-methods/tts.ts:289-298`
  - body: `resolveExplicitTtsOverrides` can throw for unknown providers or unsupported voice overrides. `tts.convert` maps those errors to `INVALID_REQUEST`, but `tts.preview` lets them fall into the outer catch and report `UNAVAILABLE`, making caller input errors look like backend outages.
  - confidence: 0.87
- **[P2] Keep explicit previews on the selected provider:** `src/gateway/server-methods/tts.ts:300-304`
  - body: When the UI sends an explicit provider or voice, `tts.preview` builds overrides but calls `textToSpeech` without `disableFallback`. If the selected provider fails, preview can silently play another provider instead of the selected voice/provider; match `tts.convert` and disable fallback when an explicit override is present.
  - confidence: 0.88
- **[P3] Replace the changelog placeholder:** `CHANGELOG.md:9`
  - body: The new changelog entry would ship `Fixes #XXXX` and bot attribution. Replace it with the real related issue reference, likely `#71487`, and credit a human source such as the issue reporter rather than `@copilot`.
  - confidence: 0.96
- **[P3] Show loading before the empty provider state:** `ui/src/ui/views/config-quick.ts:1120-1123`
  - body: The card renders `No TTS providers configured` whenever `configuredProviders.length` is zero, but the app tracks `ttsProvidersLoading` and this prop is not passed through. On initial load, users can see a false empty state before `tts.providers` resolves; pass a loading flag or defer the empty state until provider loading completes.
  - confidence: 0.78
