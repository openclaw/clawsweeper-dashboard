Keep open. The feature direction is valid and matches the open Control UI TTS request in #71487, but current main still lacks this UI/RPC surface and the PR branch is not merge-ready because the new TTS methods are not fully wired through the runtime facade, gateway scope contract, validation, preview semantics, and changelog metadata.

## What This Changes

The branch adds a Quick Settings TTS card, UI state/controller wiring, `voiceByProvider` preference persistence, `tts.setVoice` and `tts.preview` gateway methods, speech-core default voice application, tests, CSS, and a changelog entry.

## Best Possible Solution

Keep this PR open as the implementation path for #71487, then land it only after the new TTS RPCs are exported from the speech-core runtime surface, classified with the correct operator scope, validated consistently with existing TTS writes, covered by focused gateway/UI tests, documented where the Control UI and TTS docs describe user-facing behavior, and cleaned up in the changelog.

Yes. A source check on current main reproduces the missing feature, and the PR defects reproduce by comparing the current PR head against the runtime facade, method-scope, and existing `tts.convert` contracts.

## Solution Assessment

No. A Quick Settings card plus narrow TTS gateway methods is a reasonable direction, but this exact patch is not the best mergeable solution until it preserves runtime exports, gateway authorization, invalid-request handling, explicit preview provider behavior, and release metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Export the voice helpers from speech-core runtime:** `src/plugin-sdk/tts-runtime.ts:115-120`
  - body: `src/tts/tts.ts` reaches speech-core through the lazy `tts-runtime` facade, which loads `extensions/speech-core/runtime-api.ts`. This patch adds `setTtsVoice` and `getTtsVoiceByProvider` to the facade and types, but the runtime API still does not export them, so the real gateway paths will not persist or read voice prefs through that surface.
  - confidence: 0.94
- **[P2] Classify the new TTS RPCs as write methods:** `src/gateway/server-methods-list.ts:29-30`
  - body: Adding `tts.setVoice` and `tts.preview` to the method list without adding them to `METHOD_SCOPE_GROUPS` leaves them unclassified. Write-scoped Control UI callers that can use the companion TTS methods will be rejected as admin-only, and the gateway method classification test should fail once these methods are listed.
  - confidence: 0.92
- **[P2] Keep explicit previews on the selected provider:** `src/gateway/server-methods/tts.ts:300-306`
  - body: When the UI sends an explicit provider or voice, `tts.preview` builds overrides but calls `textToSpeech` without `disableFallback`. If the selected provider fails, preview can silently play audio from a fallback provider instead of the selected voice/provider; pass `disableFallback` when an explicit override is present, matching `tts.convert`.
  - confidence: 0.89
- **[P2] Return invalid request for bad preview overrides:** `src/gateway/server-methods/tts.ts:290-298`
  - body: `resolveExplicitTtsOverrides` throws for unknown providers or unsupported voice overrides. `tts.convert` maps that to `INVALID_REQUEST`, but `tts.preview` lets it fall into the outer catch and reports `UNAVAILABLE`, making caller input errors look like backend outages.
  - confidence: 0.87
- **[P2] Validate setVoice against the active provider registry:** `src/gateway/server-methods/tts.ts:265-275`
  - body: `tts.setProvider` canonicalizes with the runtime config and rejects ids not present in `getSpeechProvider`, but `tts.setVoice` canonicalizes the raw string and persists it. Use the same registry-backed validation so callers cannot save a voice for an unregistered provider id.
  - confidence: 0.84
- **[P3] Replace the changelog placeholder:** `CHANGELOG.md:9`
  - body: The changelog entry would ship `Fixes #XXXX` and bot attribution. Replace the placeholder with the real related issue or PR reference, and avoid a `Thanks` line to the automation account when no human contributor attribution is available.
  - confidence: 0.96
- **[P3] Show provider loading instead of an empty configured state:** `ui/src/ui/views/config-quick.ts:1003-1004`
  - body: The card renders `No TTS providers configured` whenever `configuredProviders.length` is zero, but the app has a separate `ttsProvidersLoading` state that is not passed into this view. On initial refresh, users can briefly see a false empty state before `tts.providers` resolves.
  - confidence: 0.78
