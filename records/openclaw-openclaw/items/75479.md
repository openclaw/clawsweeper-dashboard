## Summary

Keep open. Current main still lacks the requested Quick Settings TTS surface, so the PR is useful, but the proposed patch has narrow gateway/runtime-contract bugs that should block merge until fixed.

## What This Changes

The PR adds Quick Settings TTS controls, new `tts.setVoice` and `tts.preview` gateway methods, persisted per-provider voice preferences, default-voice application in speech-core, UI state/controller wiring, tests, and a changelog entry.

## Best Possible Solution

Keep the feature direction and land it as the implementation for #71487 once the new TTS RPCs are wired through the speech-core runtime API, classified with the right operator scope, validated like existing TTS writes, documented, covered by focused gateway/UI tests, and cleaned up in the changelog.

## Reproduction Assessment

Yes. Current-main reproduction is a source check: Quick Settings has no TTS card and the gateway/docs do not expose `tts.setVoice`, `tts.preview`, or `voiceByProvider`; the PR defects reproduce by inspecting the head diff against the current runtime facade and method-scope contracts.

## Solution Assessment

No. A Quick Settings card plus narrow TTS gateway methods is a reasonable solution, but this patch is not merge-ready because it does not fully preserve the existing speech-core facade, gateway authorization, validation, and fallback contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Export the new voice helpers from speech-core:** `src/plugin-sdk/tts-runtime.ts:115-120`
  - body: `src/tts/tts.ts` reaches speech-core through the lazy `tts-runtime` facade, which loads `extensions/speech-core/runtime-api.ts`. This patch adds `setTtsVoice` and `getTtsVoiceByProvider` to the facade, but the runtime API does not export them, so the new gateway paths will not persist or read voice prefs through the real runtime surface.
  - confidence: 0.94
- **[P2] Classify the new TTS RPCs as write methods:** `src/gateway/server-methods-list.ts:29-30`
  - body: Adding `tts.setVoice` and `tts.preview` to the method list without adding them to `METHOD_SCOPE_GROUPS` leaves them unclassified. Non-admin write-scoped Control UI callers that can use the companion TTS methods would be rejected, and the method-scope classification test should fail once these methods are listed.
  - confidence: 0.9
- **[P2] Keep explicit previews on the selected provider:** `src/gateway/server-methods/tts.ts:300-306`
  - body: When the UI sends an explicit provider or voice, `tts.preview` builds overrides but calls `textToSpeech` without `disableFallback`. If the selected provider fails, preview can silently play audio from a fallback provider instead of the selected voice/provider; pass `disableFallback` when an explicit override is present, matching `tts.convert`.
  - confidence: 0.88
- **[P2] Return invalid request for bad preview overrides:** `src/gateway/server-methods/tts.ts:290-298`
  - body: `resolveExplicitTtsOverrides` can throw for unknown providers or unsupported voice overrides. `tts.convert` maps that to `INVALID_REQUEST`, but `tts.preview` lets it fall into the outer catch and reports `UNAVAILABLE`, making caller input errors look like backend outages.
  - confidence: 0.86
- **[P2] Validate setVoice against the active provider registry:** `src/gateway/server-methods/tts.ts:265-275`
  - body: `tts.setProvider` canonicalizes with the runtime config and rejects ids not present in `getSpeechProvider`, but `tts.setVoice` canonicalizes the raw string and persists it. Use the same registry-backed validation so callers cannot save a voice for an unregistered provider id.
  - confidence: 0.82
- **[P3] Replace the changelog placeholder:** `CHANGELOG.md:9`
  - body: The changelog entry would ship `Fixes #XXXX` and bot attribution. Replace the placeholder with the real related issue or PR reference, and avoid forcing a `Thanks` line to an automation account when no human contributor attribution is available.
  - confidence: 0.95
