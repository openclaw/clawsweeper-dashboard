## Summary

Keep open. Current main still lacks the requested Quick Settings TTS surface and the new voice/preview gateway API, but this draft PR has concrete integration blockers in the runtime facade, gateway scope classification, preview behavior, request-error mapping, provider validation, and changelog metadata before it should merge.

## What This Changes

The PR adds a Quick Settings Text-to-Speech card, UI state/controllers/styles/tests, `tts.setVoice` and `tts.preview` gateway methods, per-provider voice preference persistence in speech-core, default voice application during synthesis, SDK facade/type exports, and a changelog entry.

## Best Possible Solution

Keep the feature direction, but land it only after the runtime facade is wired end to end, the new gateway methods have the same write-scope and validation behavior as existing TTS methods, preview cannot silently fall back from an explicit provider or voice selection, public docs/changelog metadata are corrected, and the focused gateway/UI/scope tests pass.

## Reproduction Assessment

Yes. Current-main reproduction is a source check: Quick Settings has no TTS card, `tts.status` does not return `voiceByProvider`, and the gateway has no `tts.setVoice` or `tts.preview` methods. The PR defects are source-checkable from the proposed diff and current TTS gateway contracts.

## Solution Assessment

No. A Quick Settings card plus narrow TTS gateway methods is a reasonable solution for the open feature request, but this implementation is not yet the best mergeable form because it leaves runtime exports, authorization scope classification, preview fallback/error handling, provider validation, and changelog metadata incomplete.

- **[P2] Export the new voice helpers from speech-core:** `src/plugin-sdk/tts-runtime.ts:115-120`
  - body: `src/tts/tts.ts` reaches speech-core through the lazy `tts-runtime` facade, which loads `extensions/speech-core/runtime-api.ts`. This patch adds `setTtsVoice` and `getTtsVoiceByProvider` to the facade and gateway, but the speech-core runtime API still does not export them, so the new gateway path will not be backed by the runtime surface.
  - confidence: 0.94
- **[P2] Classify the new TTS RPCs as write methods:** `src/gateway/server-methods-list.ts:29-30`
  - body: Adding `tts.setVoice` and `tts.preview` to the method list without also adding them to `METHOD_SCOPE_GROUPS` leaves them unclassified, and unclassified methods require admin scope. Write-scoped Control UI callers that can use the existing TTS methods would be rejected for these companion actions.
  - confidence: 0.9
- **[P2] Keep explicit previews on the selected provider:** `src/gateway/server-methods/tts.ts:300-306`
  - body: When the UI sends an explicit provider or voice, `tts.preview` builds overrides but calls `textToSpeech` without disabling fallback. If the selected provider fails, the preview can synthesize with a fallback provider and play audio that does not match the selected provider or voice.
  - confidence: 0.88
- **[P2] Return invalid request for bad preview overrides:** `src/gateway/server-methods/tts.ts:290-298`
  - body: `resolveExplicitTtsOverrides` can throw for unknown providers or unsupported voice overrides. `tts.convert` maps that to `INVALID_REQUEST`, but the new preview handler lets it fall into the outer catch as `UNAVAILABLE`, making caller input errors look like backend outages.
  - confidence: 0.86
- **[P2] Validate setVoice against the active provider registry:** `src/gateway/server-methods/tts.ts:265-275`
  - body: `tts.setProvider` canonicalizes with the runtime config and rejects ids missing from `getSpeechProvider`, but `tts.setVoice` only canonicalizes the raw string before persisting. That allows saving a voice for a provider id that is not registered in the current config/plugin set.
  - confidence: 0.82
- **[P3] Replace the changelog placeholder:** `CHANGELOG.md:9`
  - body: The changelog entry would ship `Fixes #XXXX` and bot attribution. Replace the placeholder with the real related issue or PR reference, and use a valid human contributor attribution when one exists; if the only source author is automation, keep credit in PR history instead.
  - confidence: 0.95
