## Summary

Keep open. The PR targets a real Control UI gap and is the right general direction for the linked TTS settings request, but the current draft is not merge-ready because the new gateway/runtime methods are not fully wired through the existing facade, scope, validation, and preview-error contracts.

## What This Changes

The PR adds a Quick Settings Text-to-Speech card, persists per-provider voice preferences, applies saved voices during synthesis, and introduces `tts.setVoice` plus `tts.preview` gateway methods with UI tests and a changelog entry.

## Best Possible Solution

Land this feature after the PR preserves the existing TTS runtime facade contract, classifies the new gateway methods with the correct operator scope, validates and maps preview/setVoice request errors like the existing TTS RPCs, updates docs/changelog metadata, and adds focused tests for the new gateway surfaces.

## Reproduction Assessment

Yes. Current-main reproduction is a source check showing no Quick Settings TTS card and no `tts.setVoice`/`tts.preview` gateway methods; PR-head defects reproduce by comparing the diff against the current runtime facade, method-scope tests, and existing `tts.convert` validation/fallback behavior.

## Solution Assessment

No. The UI card plus narrow TTS gateway methods are a maintainable solution, but this draft is not the best implementation until the runtime exports, scopes, validation, preview fallback behavior, docs, and changelog entry are corrected.

Overall confidence: 0.88

Full review comments:

- **[P2] Export the voice helpers from speech-core:** `src/plugin-sdk/tts-runtime.ts:115-120`
  - body: `src/tts/tts.ts` reaches speech-core through the lazy `tts-runtime` facade, which loads `extensions/speech-core/runtime-api.ts`. This patch adds facade entries for `setTtsVoice` and `getTtsVoiceByProvider`, but the runtime API does not export those helpers, so the new status/setVoice paths will fail through the real runtime surface.
  - confidence: 0.94
- **[P2] Classify the new TTS RPCs as write methods:** `src/gateway/server-methods-list.ts:29-30`
  - body: Adding `tts.setVoice` and `tts.preview` to the method list without adding them to `METHOD_SCOPE_GROUPS` leaves them unclassified. The method-scope classification test should fail, and write-scoped Control UI callers would be treated as missing admin scope instead of using the companion TTS write scope.
  - confidence: 0.9
- **[P2] Keep explicit previews on the selected provider:** `src/gateway/server-methods/tts.ts:300-306`
  - body: When the UI sends an explicit provider or voice, `tts.preview` builds overrides but calls `textToSpeech` without `disableFallback`. If the selected provider fails, preview can silently play audio from a fallback provider instead of the selected voice/provider; pass `disableFallback` when an explicit override is present, matching `tts.convert`.
  - confidence: 0.88
- **[P2] Return invalid request for bad preview overrides:** `src/gateway/server-methods/tts.ts:290-298`
  - body: `resolveExplicitTtsOverrides` can throw for unknown providers or unsupported voice overrides. `tts.convert` maps that to `INVALID_REQUEST`, but `tts.preview` lets it fall into the outer catch and reports `UNAVAILABLE`, making caller input errors look like backend outages.
  - confidence: 0.86
- **[P2] Validate setVoice against the active provider registry:** `src/gateway/server-methods/tts.ts:265-275`
  - body: `tts.setProvider` canonicalizes with the runtime config and rejects ids not present in `getSpeechProvider`, but `tts.setVoice` canonicalizes the raw string and persists it. Use the same registry-backed validation so callers cannot save a voice for an unregistered provider id.
  - confidence: 0.82
- **[P3] Replace the changelog placeholder:** `CHANGELOG.md:9`
  - body: The changelog entry would ship `Fixes #XXXX` and bot attribution. Replace the placeholder with the real related issue or PR reference, and avoid forcing a `Thanks` line to an automation account when no human contributor attribution is available.
  - confidence: 0.95
