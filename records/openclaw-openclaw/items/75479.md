## Summary

Keep open: this is useful source work for the linked TTS Settings feature, but the PR is not correct as submitted because the new lazy TTS facade entries are not backed by the speech-core runtime API and the new gateway RPC names are not classified for operator scope.

## What This Changes

The PR adds a Quick Settings Text-to-Speech card, per-provider default voice persistence, and new `tts.setVoice`/`tts.preview` gateway methods with UI controller wiring and tests.

## Best Possible Solution

Repair this PR or a replacement branch so the UI, gateway method list, method scopes, speech-core runtime API, tests, and public TTS docs all describe the same new TTS Settings contract before landing it for https://github.com/openclaw/openclaw/issues/71487.

## Reproduction Assessment

Yes for source-level review. Current main lacks the requested Quick Settings TTS card, and the PR-head defects are reproducible by comparing the new facade/method-list entries against the speech-core runtime API and gateway method-scope contract.

## Solution Assessment

No as submitted. The product direction is reasonable, but the mergeable solution must first export the new speech-core helpers, classify the new RPCs with write scope, and document/validate the public gateway contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Export the new TTS voice helpers from speech-core:** `src/plugin-sdk/tts-runtime.ts:115-120`
  - body: `src/tts/tts.ts` reaches speech-core through the lazy `src/plugin-sdk/tts-runtime.ts` facade, but this PR adds `setTtsVoice` and `getTtsVoiceByProvider` only to the facade/types and not to `extensions/speech-core/runtime-api.ts`. When `tts.status` or `tts.setVoice` loads the real runtime module, those lazy lookups have no exported value, so the new RPCs fail at runtime.
  - confidence: 0.94
- **[P2] Classify the new TTS RPCs as write methods:** `src/gateway/server-methods-list.ts:29-30`
  - body: The PR lists `tts.setVoice` and `tts.preview` in `server-methods-list.ts`, but does not add them to `METHOD_SCOPE_GROUPS`. Current `method-scopes` coverage requires every listed gateway method to be classified, and these active TTS operations should get the same write scope as `tts.convert` and `tts.setProvider`.
  - confidence: 0.93
