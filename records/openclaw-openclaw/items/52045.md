## Summary

Keep open. Current main still has a credible browser/gateway state-divergence path after channel setup: browser status uses the browser gateway RPC, and that RPC can fail at the browser control-service activation gate while the gateway daemon itself is healthy; the linked fix PR is closed unmerged.

## What This Changes

The issue asks for `openclaw browser --browser-profile openclaw status` to report browser status accurately after Feishu pairing instead of saying the gateway is closed or disabled.

## Best Possible Solution

Land a narrow browser/config regression fix that keeps configured browser hosting active under restrictive plugin allowlists after channel setup, and make browser status report a precise browser-plugin/config problem instead of implying daemon failure.

## Reproduction Assessment

Yes for a source-level path: configure browser hosting with a restrictive `plugins.allow` that omits `browser`, then enable Feishu/channel config; browser status reaches the control-service gate that can return disabled while gateway health remains independent. The full live Feishu pairing flow still needs a fresh run for end-to-end confirmation.

## Solution Assessment

No final patch is present. The closed endpoint-only PR is not sufficient by itself because the failure can occur before route dispatch, when `startBrowserControlServiceFromConfig` returns null.
