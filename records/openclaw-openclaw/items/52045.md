## Summary

Keep open: current main still routes browser status through the admin-scoped `browser.request` path, and source/tests show a paired lower-scope device token can produce a gateway close/pairing-required path while gateway status remains healthy. Related PR #52057 was closed dirty and did not land.

## What This Changes

The issue asks for `openclaw browser --browser-profile openclaw status` to report browser status correctly after the gateway is healthy and Feishu pairing has occurred.

## Best Possible Solution

Define the intended browser status auth boundary, then add regression coverage for paired lower-scope device tokens and disabled browser config so the CLI reports an actionable browser/auth state instead of a generic gateway-closed failure.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection gives a plausible current-main path: Feishu or other pairing can leave the CLI with a lower-scope stored operator token, while browser status still needs `operator.admin` through `browser.request`.

## Solution Assessment

Unclear. The closed PR's disabled-config guard is useful but likely incomplete; the maintainable fix should address the admin-scope browser status behavior or produce a clear pairing/approval diagnostic without weakening browser-control security.
