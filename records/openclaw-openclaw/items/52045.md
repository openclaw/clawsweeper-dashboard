## Summary

Keep open. Current main has improved browser startup activation for root browser config, but the browser status request still reaches a request-time browser control-service gate that can reject a source config with a restrictive `plugins.allow` that omits `browser`; that keeps the reporter's Feishu-pairing state-divergence path credible.

## What This Changes

The issue asks for `openclaw browser --browser-profile openclaw status` to report browser status accurately after Feishu pairing instead of saying the gateway is disabled while gateway status is healthy.

## Best Possible Solution

Align the browser-owned control-service enablement gate with the documented root `browser` activation semantics under restrictive plugin allowlists, while preserving explicit `plugins.enabled=false`, `plugins.deny`, and `plugins.entries.browser.enabled=false` behavior. Add regression tests for browser status with a root browser config plus a restrictive allowlist that omits `browser`, ideally alongside the Feishu setup path that creates the reporter's state.

## Reproduction Assessment

Yes. A high-confidence config-level reproduction is root `browser.enabled=true` plus a nonempty `plugins.allow` such as `['feishu']` that omits `browser`: current main can load the browser plugin at startup, but `openclaw browser status` can still hit `startBrowserControlServiceFromConfig()` and get `UNAVAILABLE` while the gateway process is healthy.

## Solution Assessment

No final solution is present. The current startup activation change is directionally right, but the request-time control service still applies generic allowlist semantics to the source snapshot; the safer fix is to make the browser service gate honor root browser config consistently without broadening unrelated plugin activation.
