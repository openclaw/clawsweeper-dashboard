## Label Justifications

- `P1`: The report describes a user-visible WebChat session continuity break where the UI shows history but the model loses context on the next turn.
- `impact:session-state`: The affected behavior is session/transcript state drift between the history shown by the UI and the context sent to the model.

## Summary

Keep open: current main fixed the overlapping live reconnect/sleep path, but the reported browser/computer restart path still has a source-level gap because history-loaded sends carry only `sessionId` while stale idle/daily session reuse requires the private resume flag.

## What This Changes

The issue asks for WebChat `chat.send` after gateway/browser restart to use the same persisted transcript context that `chat.history` displays.

## Best Possible Solution

Align Control UI session restore and Gateway session initialization so a history-restored or selected session can make a model-visible continuation after browser/gateway restart without letting public or stale session id assertions bypass reset policy.

## Reproduction Assessment

Yes at source level: current main has a path where `chat.history` restores `currentSessionId`, but a full restart leaves no `reconnectResumeSessionId`, and session initialization only suppresses stale rollover when `resumeRequestedSession` is true. I did not run a live MiniMax/browser restart reproduction in this read-only review.

## Solution Assessment

Unclear until maintainers choose the reset-policy boundary. The maintainable fix should extend the scoped Control UI/Gateway resume mechanism or adjust the docs, not add a MiniMax-specific provider workaround.
