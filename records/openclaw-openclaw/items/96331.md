## Summary

Keep this open: current main has a narrower reconnect-resume path, but it does not prove the broader reported restart/full page-reopen case is fixed, and the linked open pull request remains the candidate implementation for the missing session-context behavior.

## What This Changes

The issue reports that WebChat can display restored session history after restart while the next `chat.send` reaches the model without that prior transcript context.

## Best Possible Solution

Review and either land or replace https://github.com/openclaw/openclaw/pull/96391 with a focused fix that keeps restored-history WebChat sends on the same transcript while preserving explicit `/new`, `/reset`, and intentional reset-policy behavior.

## Reproduction Assessment

Yes from source inspection, but not from a live run: `chat.send` passes the restored session id, while current `initSessionState` only suppresses stale rollover for the special reconnect-resume flag. A full browser/computer restart can lose that in-memory reconnect marker, matching the reporter's broader restart case.

## Solution Assessment

Unclear until the linked PR is reviewed. The proposed caller-verified session reuse is plausible, but maintainers need to decide the safest boundary against daily/idle rollover and explicit reset behavior.
