## Summary

Keep open: current main has a scoped one-shot reconnect-resume path, but this report also covers cold restart/page-reopen behavior where visible WebChat history can still diverge from model-visible session context; the open linked PR is the active candidate and still needs maintainer review.

## What This Changes

The issue reports WebChat restoring visible `chat.history` after restart while the following `chat.send` does not provide that transcript to the model.

## Best Possible Solution

Settle the WebChat cold-restart session policy, then land or replace the linked candidate with regression coverage proving `chat.history` to `chat.send` model-visible continuity and ordinary reset semantics.

## Reproduction Assessment

Yes at source level for stale restart paths: current tests and code show ordinary WebChat sends with a matching stale `sessionId` still roll over unless the private reconnect marker is present. A live MiniMax same-day restart reproduction is still missing.

## Solution Assessment

Unclear: the current one-shot reconnect fix is narrower than the full restart expectation, and the linked candidate's broader approach needs maintainer acceptance around reset compatibility and API boundaries.
