## Summary

Keep this open. Current main still has the reported Nextcloud Talk reaction wiring gap: the plugin advertises reactions and has a sender helper, but it does not register a message action adapter, so shared `react` dispatch cannot reach the helper. Two open PRs already reference this issue with closing syntax, so cleanup should wait for one to land or be superseded.

## What This Changes

The issue reports that Nextcloud Talk emoji reactions are advertised but not wired to the existing `sendReactionNextcloudTalk` sender, so agents cannot send reactions.

## Best Possible Solution

Land or supersede #72348 and #72679 with one narrow Nextcloud Talk-owned `ChannelMessageActionAdapter` that advertises and handles `react`, calls `sendReactionNextcloudTalk`, preserves the existing SSRF/signature/credential behavior, and adds regression coverage for discovery plus dispatch.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: dispatching `react` for `nextcloud-talk` reaches `dispatchChannelMessageAction`, finds no `plugin.actions.handleAction`, and returns `null` before `sendReactionNextcloudTalk` can run.

## Solution Assessment

Yes. The narrow maintainable fix is a plugin-owned message action adapter, not core inference from `capabilities.reactions`; the linked PRs are implementation candidates for that direction.
