## Summary

Keep open. The request is a new opt-in WebChat/Gateway session-linking behavior, not a cleanup-closeable bug: current code and docs already key history/live updates by explicit `sessionKey`, while the requested token/operator-based cross-device unification needs product and trust-model design.

## What This Changes

The issue requests an opt-in mode for webchat devices sharing a gateway token or operator identity to share message history and live updates across browsers.

## Best Possible Solution

Define an explicit shared-session or device-linking mode for WebChat that preserves `sessionKey` as the canonical route and only adds cross-device discovery/sync after the trust model and UI behavior are agreed.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature request for automatic/shared WebChat session selection across devices. Source inspection shows current history and live events are keyed by `sessionKey`, with browser-local session selection, and no failing current-main path was established.

## Solution Assessment

Unclear. Token/operator-based automatic merging may not be the safest default; the maintainable path is an explicit opt-in shared-session/linking design with clear session selection and visibility rules.
