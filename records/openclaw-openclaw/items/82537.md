## Summary

Keep open. Current main and v2026.6.8 still route WebChat history and live reloads through the selected `sessionKey` plus browser-local device identity, so the requested cross-device shared WebChat session mode remains missing and needs product/security design rather than cleanup closure.

## What This Changes

The issue requests an opt-in mode where multiple WebChat browser devices sharing a gateway token or operator link see unified history and live updates for the same session.

## Best Possible Solution

Define an explicit WebChat device-linking/shared-session contract that keeps `sessionKey` routing canonical while adding opt-in visibility, trust rules, UI selection, and real-time subscription semantics.

## Reproduction Assessment

No high-confidence live reproduction was run, and reproduction is not the main blocker because this asks for a new opt-in mode. Source inspection verifies the current browser-local identity plus selected-`sessionKey` routing described in the report.

## Solution Assessment

Unclear as proposed. Token/operator-based automatic merging is not yet the best-defined solution; the safer path is a maintainer-approved shared-session or device-linking design with clear UI, visibility, and trust boundaries.
