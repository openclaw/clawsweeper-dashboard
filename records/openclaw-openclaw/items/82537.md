## Summary

Keep open. Current main and latest release still use explicit WebChat `sessionKey` routing plus browser-local session/device state, while this request asks for a new opt-in shared-session mode that needs maintainer product, UX, and trust-model design before implementation.

## What This Changes

Request an opt-in mode for WebChat sessions opened from the same gateway URL to share message history and receive real-time updates across devices.

## Best Possible Solution

Define an explicit shared-session or device-linking contract for WebChat that keeps `sessionKey` as the canonical route and adds cross-device discovery/sync only after opt-in, visibility, and trust rules are agreed.

## Reproduction Assessment

No high-confidence bug reproduction is applicable. Source inspection does show the current behavior: browser-local device/session state plus explicit `sessionKey` history/event routing, but that is current design rather than a failing documented contract.

## Solution Assessment

Unclear. Token/operator-based automatic merging is not clearly the safest or narrowest solution; the better path is an explicit opt-in shared-session or device-linking design with clear UI, visibility, and trust boundaries.
