## Summary

Keep open. Current main has session-keyed WebChat history and session-scoped live updates, but it does not provide the requested token/operator-linked cross-device session discovery or shared-session opt-in behavior; that needs product, UX, and trust-boundary design first.

## What This Changes

Request an opt-in mode for multiple WebChat browser devices using the same gateway token or operator identity to share one message history and receive real-time updates.

## Best Possible Solution

Define an explicit WebChat shared-session/device-linking contract that preserves session-key routing, requires opt-in, and specifies UI visibility plus trust rules before changing browser selection or event fanout.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature request for cross-device shared WebChat session behavior. Source inspection confirms current main is session-key based, with browser-local session selection and session-scoped history/events.

## Solution Assessment

Unclear. Automatically merging sessions by gateway token or operator identity is not clearly the safest design; a maintainable solution should first define an explicit linking and visibility model for WebChat devices.
