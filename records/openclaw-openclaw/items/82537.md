## Summary

Keep open. Current main still routes WebChat by explicit sessionKey with browser-local session selection, while the request is a new cross-device shared-session capability that needs product, UX, and trust-model design before implementation.

## What This Changes

Request an opt-in WebChat mode where devices sharing a gateway token or operator see unified session history and live updates.

## Best Possible Solution

Define an explicit opt-in shared-session or device-linking contract for WebChat that preserves sessionKey as the canonical route and makes participation, visibility, and trust rules clear before implementation.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this requests new opt-in shared WebChat session behavior, not a broken documented contract. Source inspection confirms current main is sessionKey-based with browser-local session selection.

## Solution Assessment

Unclear. Token/operator-based automatic merging is not obviously the safest or narrowest design; the maintainable path is an explicit shared-session or device-linking model with clear UI and trust boundaries.
