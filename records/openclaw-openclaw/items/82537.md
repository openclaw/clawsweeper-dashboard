## Summary

Keep open. Current main still treats WebChat continuity as explicit session-key selection stored per browser and gateway, while the requested cross-device shared-session mode would add a new product and trust model rather than fix a broken documented contract.

## What This Changes

The issue requests opt-in WebChat session history and live message synchronization across multiple paired browsers sharing a gateway token or operator identity.

## Best Possible Solution

Define an explicit WebChat shared-session or device-linking contract that preserves sessionKey as the canonical route and adds opt-in discovery, visibility, authorization, and real-time sync rules before implementation.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature request for cross-device shared WebChat sessions rather than a violation of an existing documented contract. Source inspection confirms current main is sessionKey-based with browser-local session selection.

## Solution Assessment

Unclear. Token/operator-based merging may be useful, but the safest maintainable solution is an explicit opt-in shared-session or device-linking design with clear UI and trust boundaries.
