## Summary

Keep open. Current main and v2026.6.6 still use browser-local WebChat device/session state plus explicit selected `sessionKey` history and live-event routing; this asks for a new opt-in shared WebChat session mode that needs product and trust-boundary design before implementation.

## What This Changes

Request an opt-in WebChat mode where devices sharing a gateway token or linked operator identity see unified message history and real-time updates.

## Best Possible Solution

Define an explicit WebChat device-linking/shared-session contract that keeps `sessionKey` as the canonical route and adds opt-in visibility, trust rules, UI selection, and real-time subscription semantics.

## Reproduction Assessment

Not applicable as a bug reproduction: this asks for a new opt-in mode rather than a failure of documented behavior. Source inspection verifies the current design is browser-local identity plus explicit `sessionKey` routing, but I did not run a live two-device Cloudflare tunnel scenario.

## Solution Assessment

Unclear. Token/operator-based automatic merging is not yet the best-defined solution; the maintainable path is an explicit shared-session or device-linking design with clear UI, visibility, and trust boundaries.
