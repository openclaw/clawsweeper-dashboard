## Summary

Keep open. Current main still treats WebChat continuity as browser-local session selection plus explicit sessionKey routing, while the request asks for a new opt-in shared-session/device-linking mode across separate browser identities that needs product, UX, and trust-boundary design.

## What This Changes

The issue requests opt-in WebChat history and live message synchronization across multiple devices using the same gateway token or linked operator.

## Best Possible Solution

Define an explicit shared-session or device-linking contract for WebChat that preserves sessionKey as the canonical route while adding opt-in discovery, history hydration, and live update rules with clear user visibility and trust boundaries.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature request for a new shared WebChat session mode. Source inspection confirms current behavior is browser-local session selection with sessionKey-routed history and active-session updates, but no existing contract promises token/operator-wide merging.

## Solution Assessment

Unclear. Token/operator-based automatic merging is not necessarily the safest or narrowest solution; the maintainable fix should be an explicit opt-in shared-session or device-linking model with agreed UI and trust rules.
