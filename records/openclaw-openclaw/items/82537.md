## Summary

Keep open. Current main is still WebChat session-key based with browser-local device/session selection; the requested opt-in cross-device shared history and live updates by gateway token or operator identity is a new product/security/UX feature, not implemented behavior to close.

## What This Changes

The issue requests opt-in WebChat cross-device shared session history and real-time bi-directional updates for browsers using the same gateway/operator context.

## Best Possible Solution

Define an explicit shared-session or device-linking contract for WebChat that keeps sessionKey as the canonical route and adds cross-device discovery/sync only after opt-in, visibility, attribution, and trust rules are agreed.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature request for opt-in shared WebChat session selection across browser device identities. Source inspection confirms current main is sessionKey-based with browser-local session selection, but no failing current-main behavior against an existing contract was established.

## Solution Assessment

Unclear. Token/operator-based automatic merging is not obviously the safest or narrowest solution; the maintainable path is an explicit shared-session/device-linking design with clear UI, attribution, and trust boundaries.
