## Summary

Keep open. Current main and v2026.6.6 still use browser-local WebChat device/session state plus explicit sessionKey-scoped history and live-event routing, while this asks for a new opt-in shared-session feature that needs product, UX, and trust-model design.

## What This Changes

The issue requests opt-in cross-device WebChat session synchronization so multiple browsers using the same gateway/operator context share message history and live updates.

## Best Possible Solution

Define an explicit shared-session or device-linking contract for WebChat that keeps sessionKey as the canonical route and adds cross-device discovery/sync only after opt-in, visibility, and trust rules are agreed.

## Reproduction Assessment

No: there is no high-confidence bug reproduction path because the requested behavior is a new opt-in mode. Source inspection does verify the current behavior: browser-local device/session state plus explicit sessionKey history and event routing.

## Solution Assessment

Unclear. Token/operator-based automatic merging is not clearly the safest or narrowest solution; the better path is an explicit opt-in shared-session or device-linking design with clear UI, visibility, and trust boundaries.
