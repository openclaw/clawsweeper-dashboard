## Summary

Keep open. Current main still uses explicit `sessionKey` routing with browser-local WebChat session selection, while this request asks for new cross-device shared-session behavior that needs product, UX, and trust-model design before implementation.

## What This Changes

Request an opt-in WebChat/Gateway mode where multiple paired browsers sharing a gateway token or operator identity see one unified history and live updates.

## Best Possible Solution

Define an explicit shared-session or device-linking contract for WebChat that keeps `sessionKey` as the canonical route and adds cross-device discovery/sync only after opt-in, visibility, and trust rules are agreed.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a feature request for automatic or opt-in shared WebChat session selection across devices. Source inspection confirms current main is sessionKey-based with browser-local session selection, but no current-main failing behavior against an existing contract was established.

## Solution Assessment

Unclear. Token/operator-based automatic merging is not obviously the safest or narrowest solution; a maintainable fix should be an explicit shared-session or device-linking design with clear UI and trust boundaries.
