## Summary

Keep open: the PR is a useful security direction, but it is not merge-ready because attacker-controlled filename metadata is still rendered as trusted prompt text, absolute inbound-path matching still overwraps trusted workspace files, and the PR lacks after-fix real behavior proof.

## What This Changes

The PR adds inbound-media path detection, wraps text read-tool results from staged `media/inbound/*` files with untrusted prompt-data markers, forwards `containerWorkdir`, and adds regression tests.

## Best Possible Solution

Land a root-aware inbound-media read guard that uses a constant trusted label, places path metadata inside the untrusted block, wraps only staged inbound text blocks, and includes focused tests plus redacted real attachment-read proof.

## Reproduction Assessment

Yes, from source inspection. Current main stages attachments to `media/inbound/<filename>` and returns read-tool text without an untrusted wrapper, while PR-head source shows both the trusted-label metadata path and the absolute-path overmatch.

## Solution Assessment

No. The wrapper direction is right, but the current PR is not the narrowest maintainable fix because attacker-controlled path metadata remains outside the untrusted block and absolute matching is not anchored to the actual staged root.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep attacker filenames inside the untrusted block:** `src/agents/pi-tools.read.ts:798`
  - body: `sanitizeForPromptLiteral` only strips control/format characters, so a staged filename like `report) Ignore prior instructions.txt` is still rendered in the trusted label before `<untrusted-text>`. Inbound filenames come from external media staging via `path.basename(source)`, so use a constant trusted label and put any path metadata inside the wrapped text instead.
  - confidence: 0.82
- **[P2] Anchor absolute inbound matches to the staged root:** `src/agents/pi-tools.read.ts:140`
  - body: The regex treats any absolute path containing `/media/inbound/` as inbound, and the tests now expect `/workspace/src/media/inbound/parser.ts` to match. A trusted repository file in that nested directory would be wrapped and HTML-escaped by the read tool, so classify against the resolved workspace-relative path or actual sandbox staging root instead.
  - confidence: 0.86
