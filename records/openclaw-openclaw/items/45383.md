## Summary

Keep open. Current main still lacks an inbound-media untrusted wrapper, and the PR is directionally useful, but the latest PR head still overmatches absolute paths containing `/media/inbound/`, which would wrap trusted workspace files that merely have that nested directory shape.

## What This Changes

The PR adds inbound-media path detection, wraps text read-tool results from staged `media/inbound/*` files as untrusted prompt data, threads sandbox `containerWorkdir`, and adds focused regression tests.

## Best Possible Solution

Land a root-aware read-tool guard that classifies staged inbound media from the resolved workspace-relative path, wraps only text blocks with sanitized labels, preserves image blocks, and leaves the linked security issue open until that corrected fix merges.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: channel media is staged as `media/inbound/<name>`, and current main's read tool returns text through image sanitization without an untrusted prompt-data wrapper.

## Solution Assessment

No. The wrapper approach is the right mitigation direction, but the PR's classifier should be anchored to the actual staged inbound root or resolved workspace-relative path instead of matching every absolute `/media/inbound/` segment.
