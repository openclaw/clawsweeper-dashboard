## Summary

Keep open. The PR addresses useful security work and the early Greptile findings were fixed, but the current head still has a blocking classifier false positive and the external PR only provides test proof, not real after-fix behavior proof.

## What This Changes

The PR adds inbound-media path detection and wraps text read-tool results from staged `media/inbound/*` files as untrusted prompt data, with `containerWorkdir` threading and regression tests.

## Best Possible Solution

Land a root-aware read-tool guard that classifies staged inbound media from the resolved workspace-relative path or actual sandbox staging root, wraps only text blocks with sanitized labels, and preserves trusted workspace source reads.

## Reproduction Assessment

Yes, by source inspection. Current main stages inbound attachments to `media/inbound/<name>` and returns read-tool text without an untrusted wrapper, while PR head source-reproduces the false-positive path through `/workspace/src/media/inbound/parser.ts`.

## Solution Assessment

No. Wrapping inbound file text is the right direction, but this implementation classifies absolute paths by any `media/inbound` segment instead of the resolved staging root, so it can alter trusted workspace source reads.

## Review Findings

Overall correctness: patch is incorrect
