## Summary

Keep open: current main still lacks an inbound-media untrusted wrapper and this PR is the only direct implementation candidate for the linked security issue, but the PR head is not merge-ready because it keeps sender-controlled filename text in trusted prompt metadata, overmatches absolute `media/inbound` path segments, and lacks real behavior proof.

## What This Changes

The PR adds an inbound-media path classifier, wraps text read from `media/inbound/*` in untrusted-text markers, forwards `containerWorkdir`, and adds focused inbound-guard unit tests.

## Best Possible Solution

Land a root-aware inbound-media wrapper that keeps attacker-controlled path text inside the untrusted block or omits it from trusted metadata, preserves normal read accuracy, adds focused regression tests, and includes redacted real attachment-read proof.

## Reproduction Assessment

Yes. From source inspection, current main stages sandboxed attachments at `media/inbound/*`, then the read tool returns normalized/sanitized text without an untrusted-data wrapper; I did not run a live messaging flow in this read-only review.

## Solution Assessment

No. The wrapper direction is right, but this branch should use trusted/static metadata outside the guarded block and classify inbound reads against the actual staged root instead of any absolute `media/inbound` segment.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Keep inbound filenames out of trusted labels:** `src/agents/pi-tools.read.ts:798`
  - body: `sanitizeForPromptLiteral` strips controls, but sender-controlled filename words still appear in `params.label`, which the wrapper emits before `<untrusted-text>`. A file named like an instruction can therefore put instruction-shaped text in trusted prompt metadata; use a constant label or move path metadata inside the guarded block.
  - confidence: 0.82
- **[P2] Anchor absolute inbound matching to the staged root:** `src/agents/pi-tools.read.ts:140`
  - body: The regex matches any absolute path containing `/media/inbound/`, and the new test expects `/workspace/src/media/inbound/parser.ts` to be inbound. That changes normal source reads by wrapping and escaping trusted files with that directory shape, so compare against the resolved sandbox/workspace inbound root instead.
  - confidence: 0.86
