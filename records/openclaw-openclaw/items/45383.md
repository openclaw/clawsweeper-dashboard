## Label Justifications

- `P1`: This changes a security boundary for externally supplied inbound files and can affect real agent behavior across messaging channels.
- `merge-risk: 🚨 compatibility`: The broad absolute-path classifier can change existing read-tool output for trusted files whose path contains `/media/inbound/`.

## Summary

Keep open: the security direction is useful, but the patch still needs real behavior proof and has blocking security/compatibility issues around a duplicate prompt-wrapper contract and broad absolute-path classification.

## What This Changes

The branch adds an inbound-media detector to the agent read tool, wraps text results from `media/inbound` reads as untrusted data, forwards `containerWorkdir`, and adds read-tool unit tests.

## Best Possible Solution

Land a rebased security fix that reuses the shared untrusted file-content wrapper, classifies inbound files against the real staged media root, and leaves the broader read-only enforcement question tracked on the linked security issue.

## Reproduction Assessment

Yes for the source path: current main documents and implements inbound attachment staging under `media/inbound/*`, and the current read tool returns sanitized read results without an inbound untrusted-content wrapper. I did not establish a live messaging reproduction in this read-only review.

## Solution Assessment

No, not as proposed. The narrow hardening is useful, but the maintainable fix should reuse the shared external-content wrapper and make the path classifier root-aware rather than adding a parallel wrapper with broad absolute matching.

  - body: `wrapInboundFileResult` wraps inbound reads with `wrapUntrustedPromptDataBlock`, but current main already uses `wrapUntrustedFileContent`/`wrapExternalContent` for untrusted file inputs. That silently changes read output for trusted workspace files and can HTML-escape code, so classify against the actual staged media root or pass enough root context to avoid false positives.
  - confidence: 0.84
