## Summary

## What This Changes

The PR adds inbound-media read-tool wrapping in `src/agents/pi-tools.read.ts`, threads `containerWorkdir` into sandboxed read tools, and adds inbound-guard tests.

## Best Possible Solution

Land a rebased, root-aware inbound-media guard that uses a constant trusted label or moves path metadata inside the guarded block, plus redacted real attachment-read proof.

## Reproduction Assessment

Yes, from source inspection. Current main stages sandboxed attachments under `media/inbound/*` and then returns read-tool text without an untrusted-data wrapper; I did not run a live messaging flow in this read-only review.

## Solution Assessment

No. The wrapper direction is right, but the branch should keep attacker-controlled path text out of trusted labels and classify inbound reads against the actual staged root instead of any absolute `media/inbound` segment.
