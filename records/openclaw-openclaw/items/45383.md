## Summary

Keep open: current main still lacks inbound-media read wrapping and the PR targets a real security bug, but the current patch leaves attacker-controlled filename text in trusted prompt-label space, overmatches absolute paths, and has no after-fix real behavior proof.

## What This Changes

The PR adds inbound-media path detection, wraps text read results from staged `media/inbound` files with untrusted prompt-data markers, forwards `containerWorkdir`, and adds regression tests.

## Best Possible Solution

Land a root-aware inbound-media read guard with a constant trusted label, path metadata inside wrapped text, focused regression tests, green relevant checks, and redacted real attachment-read proof.

## Reproduction Assessment

Yes, from source inspection. Current main stages attachments under `media/inbound/<filename>` and returns read-tool text without an untrusted wrapper, while PR-head source shows the proposed wrapper plus the remaining label and absolute-path issues.

## Solution Assessment

No. The wrapper direction is right, but the current patch is not the best fix until attacker-controlled path metadata is moved inside the untrusted block and inbound matching is anchored to the resolved staged root.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep inbound filenames out of trusted wrapper labels:** `src/agents/pi-tools.read.ts:798`
  - body: `sanitizeForPromptLiteral` strips control and format characters, but ordinary attacker-controlled filename text still appears in the trusted label before `<untrusted-text>`. A sender can name an upload with instruction-like text, so use a constant trusted label and put any path metadata inside the wrapped text instead.
  - confidence: 0.82
- **[P2] Anchor inbound matching to the staged workspace root:** `src/agents/pi-tools.read.ts:140`
  - body: The regex treats any absolute path containing `/media/inbound/` as inbound, and the added test expects `/workspace/src/media/inbound/parser.ts` to match. That wraps and HTML-escapes trusted repository files under that path shape; classify against the resolved workspace-relative staged path or actual sandbox staging root instead.
  - confidence: 0.86
