## Summary

Keep open: current main still stages inbound attachments under `media/inbound/*` and returns read-tool text without an untrusted wrapper, so the PR addresses a real security gap, but the current patch still has blocking prompt-boundary and path-classification issues plus missing real behavior proof.

## What This Changes

This PR adds an inbound-media read guard that detects staged `media/inbound` paths, wraps text read results as untrusted prompt data, forwards `containerWorkdir`, and adds regression tests.

## Best Possible Solution

Land a root-aware inbound-media read guard with a constant trusted label, any filename metadata inside the guarded text, focused regression tests, green checks, and redacted real attachment-read proof.

## Reproduction Assessment

Yes, from source inspection. Current main stages attachments under `media/inbound/<filename>` and returns read-tool text without an untrusted wrapper, while PR head shows the intended wrapper path and the remaining defects.

## Solution Assessment

No. The wrapper direction is right, but this patch is not the best fix until attacker-controlled path metadata is moved out of trusted label text and inbound matching is anchored to the resolved staged root.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep inbound filenames out of trusted wrapper labels:** `src/agents/pi-tools.read.ts:798`
  - body: `sanitizeForPromptLiteral` strips control and format characters, but ordinary sender-controlled filename text still appears in the trusted label before `<untrusted-text>`. A hostile filename can look like an instruction outside the guarded block, so use a constant label and put any path metadata inside the untrusted text or another guarded metadata wrapper.
  - confidence: 0.82
- **[P2] Anchor inbound matching to the staged workspace root:** `src/agents/pi-tools.read.ts:140`
  - body: The absolute-path regex treats any path containing `/media/inbound/` as inbound, and the test locks in `/workspace/src/media/inbound/parser.ts` as true. That wraps and HTML-escapes normal repository files under that path shape; classify against the resolved workspace-relative staged path or actual sandbox root instead.
  - confidence: 0.86
