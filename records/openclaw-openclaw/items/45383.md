## Summary

Keep open: current main still lacks an untrusted wrapper for staged inbound media reads, but the PR head still over-classifies any absolute path containing `/media/inbound/`, so this security trust-boundary change needs rework before merge.

## What This Changes

The PR adds inbound-media path detection, wraps read-tool text blocks from `media/inbound/*` with `wrapUntrustedPromptDataBlock`, forwards `containerWorkdir`, and adds regression tests.

## Best Possible Solution

Land a root-aware inbound-media read guard that wraps only staged inbound text blocks with sanitized labels while preserving normal trusted workspace source reads.

## Reproduction Assessment

Yes. Source inspection shows current main stages attachments as `media/inbound/<filename>` and returns read-tool text without an untrusted wrapper, while PR-head source/tests show the absolute-path false-positive through `/workspace/src/media/inbound/parser.ts`.

## Solution Assessment

No. The wrapper direction is appropriate, but the current PR classifier is not the narrowest maintainable fix because it matches every absolute `/media/inbound/` segment instead of the resolved staged root.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Anchor absolute inbound matches to the staged root:** `src/agents/pi-tools.read.ts:140`

A trusted repository file in that nested directory would be wrapped and HTML-escaped by the read tool, so classify against the resolved workspace-relative path or actual sandbox staging root instead.
