## Summary

Keep open: the PR targets a valid inbound-media prompt-injection gap, but the latest patch still over-classifies trusted absolute workspace paths that contain `/media/inbound/`, which should block merge until the classifier is anchored to the actual staged root.

## What This Changes

The PR adds inbound-media detection, wraps text read-tool results from staged `media/inbound/*` files as untrusted prompt data, threads `containerWorkdir`, and adds regression tests.

## Best Possible Solution

Land a root-aware read-tool guard that classifies staged inbound media from the resolved workspace-relative path or actual sandbox staging root, wraps only text blocks with sanitized labels, and preserves trusted workspace source reads.

## Reproduction Assessment

Yes. Source inspection shows current main stages channel media as `media/inbound/<name>` and returns read-tool text without an untrusted prompt-data wrapper; the PR diff also gives a high-confidence false-positive path through `/workspace/src/media/inbound/parser.ts`.

## Solution Assessment

No. The wrapper direction is appropriate, but matching every absolute `/media/inbound/` segment is not the narrowest maintainable classifier because it can wrap and HTML-escape trusted workspace source files.
