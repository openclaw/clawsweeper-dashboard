## Summary

Keep open: this is a plausible security fix for the open indirect prompt-injection report, but PR head is not merge-ready because it still leaves attacker-chosen filename text in trusted wrapper metadata, overmatches absolute paths, and has no real attachment-read proof.

## What This Changes

The PR adds an inbound `media/inbound` read classifier, wraps matching read-tool text results with `wrapUntrustedPromptDataBlock`, forwards `containerWorkdir`, and adds inbound-guard tests.

## Best Possible Solution

Land a root-aware inbound-media wrapper that keeps attacker-controlled metadata out of trusted prompt text, preserves normal read output, and includes redacted real attachment-read proof.

## Reproduction Assessment

Yes. Current main source shows sandboxed attachments are staged as `media/inbound/*`, while the read tool returns sanitized text without an untrusted wrapper; I did not run a live messaging flow in this read-only review.

## Solution Assessment

No. The wrapper direction is right, but the branch should use a trusted constant label or move path metadata inside the guarded block, and it should classify inbound reads against the actual staged root instead of any absolute `media/inbound` segment.
