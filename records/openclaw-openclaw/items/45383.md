## Summary

Keep open. The PR addresses a real inbound-attachment prompt-injection gap, but it is not merge-ready because real behavior proof is missing and the patch introduces a separate, weaker untrusted-content boundary instead of reusing the current shared external-content wrapper.

## What This Changes

The PR wraps read-tool text returned from paths classified as `media/inbound/*`, adds inbound-guard tests, and forwards `containerWorkdir` into the sandboxed read-tool construction path.

## Best Possible Solution

Land a rebased inbound-media guard that sends attachment text through the shared external-content boundary and classifies only the actual staged inbound media root, backed by redacted real attachment-read proof.

## Reproduction Assessment

Yes from source inspection: current main stages sandboxed inbound attachments under `media/inbound/*`, and the read tool returns text after image sanitization without an inbound untrusted-data wrapper. I did not run a live messaging flow.

## Solution Assessment

No, not as currently patched. Treating inbound attachment text as untrusted is the right direction, but the maintainable fix should reuse the shared external-content wrapper and root-aware classification instead of adding a second fixed-tag security contract.
