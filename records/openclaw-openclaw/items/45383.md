## Summary

Keep open. The PR targets a real inbound-file prompt-injection gap, but it is not merge-ready because real behavior proof is absent and the diff still has compatibility and security-boundary issues around broad absolute-path matching and a second untrusted-content wrapper contract.

## What This Changes

The PR adds an inbound-media read-tool guard that detects `media/inbound/*` paths, wraps returned text blocks as untrusted prompt data, forwards sandbox container workdir, and adds focused tests.

## Best Possible Solution

Land a narrower hardening that resolves the actual workspace-relative top-level `media/inbound/*` path, uses the current shared external-content wrapper, keeps the linked security issue open unless read-only enforcement is implemented, and includes redacted real inbound-file proof.

## Reproduction Assessment

No high-confidence live reproduction was established here. Source inspection and docs show current main can return `media/inbound` read text without a read-tool wrapper, but I did not reproduce the model-level instruction hijack in a real channel.

## Solution Assessment

No, not as proposed. Wrapping inbound read-tool text is the right direction, but the implementation should reuse the current shared external-content boundary and classify the actual workspace staging path instead of matching every absolute `/media/inbound/` segment.
