## Label Justifications

- `P1`: The PR changes a security boundary for externally supplied inbound files and affects real agent behavior across messaging channels.
- `merge-risk: 🚨 compatibility`: The broad absolute-path classifier can change read-tool output for trusted files whose path contains `/media/inbound/`.

## Summary

Keep open: the security fix is useful and current main still lacks inbound read wrapping, but this PR needs real behavior proof and changes before merge because it duplicates the shared external-content boundary and overbroadly classifies absolute paths.

## What This Changes

The PR adds an inbound-media path detector, wraps text read results from `media/inbound/*` as untrusted data, forwards `containerWorkdir`, and adds read-tool unit coverage.

## Best Possible Solution

Land a rebased security fix that uses the shared external-content wrapper for inbound file reads, classifies against the actual staged media root, keeps broader directive/tool-gating work on the linked security issue, and includes real inbound proof.

## Reproduction Assessment

Yes for the source path: current main documents and stages inbound media under `media/inbound/*`, and the current read tool returns sanitized read results without an inbound untrusted-content wrapper. I did not establish a live messaging reproduction in this read-only review.

## Solution Assessment

No, not as proposed. The narrow hardening is useful, but the maintainable fix should reuse the shared external-content wrapper and make the path classifier root-aware instead of adding a parallel wrapper with broad absolute matching.
