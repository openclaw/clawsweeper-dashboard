## Summary

Keep open: current main still lacks an inbound-media untrusted wrapper, and this PR targets a real security surface, but it is not merge-ready because it has blocking trust-boundary findings and only mocked/unit-test proof.

## What This Changes

The branch adds an inbound-media path classifier, wraps text read from `media/inbound/*` with untrusted-text markers, forwards `containerWorkdir` into sandboxed read tools, and adds inbound-guard unit tests.

## Best Possible Solution

Land a root-aware inbound-media read wrapper that keeps attacker-controlled filenames out of trusted prompt metadata, preserves normal read accuracy, adds focused regression tests, and includes redacted real attachment-read proof.

## Reproduction Assessment

Yes from source inspection: current main stages sandboxed media at `media/inbound/*`, while `createOpenClawReadTool` returns sanitized read output without an untrusted wrapper. I did not run a live messaging flow in this read-only review.

## Solution Assessment

No. The wrapper direction is right, but the branch should keep sender-controlled filenames out of trusted labels and classify inbound reads against the resolved staged root instead of any absolute `media/inbound` segment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep inbound filenames out of trusted labels:** `src/agents/pi-tools.read.ts:798`
  - body: `sanitizeForPromptLiteral` removes control characters, but the sender-controlled path is still placed before `<untrusted-text>` as trusted prompt metadata. A file named with instruction-like words can put those words outside the data boundary, so use a constant trusted label or place path metadata inside the guarded block.
  - confidence: 0.82
- **[P2] Anchor inbound matching to the staged root:** `src/agents/pi-tools.read.ts:140`
  - body: The absolute-path regex classifies any path containing `/media/inbound/` as inbound, and the tests assert that `/workspace/src/media/inbound/parser.ts` returns true. Reading a normal repo file with that path shape will wrap and HTML-escape trusted source content, so classify after resolving relative to the sandbox/workspace root instead.
  - confidence: 0.86
