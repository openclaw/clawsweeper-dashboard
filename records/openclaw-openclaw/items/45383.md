## Label Justifications

- `P1`: This touches a security boundary for externally supplied inbound files and can affect real agent behavior across messaging channels.
- `merge-risk: 🚨 compatibility`: The broad absolute-path classifier can change existing read-tool output for trusted files whose path contains `/media/inbound/`.

## Summary

Keep open: this is a plausible security fix, but it is not merge-ready because it adds a second untrusted-content wrapper, broad absolute path matching can relabel trusted workspace files, and the contributor has not provided real inbound attachment proof.

## What This Changes

The PR adds an inbound-media path detector to the agent read tool, wraps text blocks from `media/inbound/*` reads as untrusted prompt data, forwards `containerWorkdir`, and adds focused read-tool unit tests.

## Best Possible Solution

Land a rebased security fix that uses the shared external-content/file wrapper for inbound read-tool text, constrains inbound classification to the real staged media root, and keeps the broader read-only/provenance work tracked separately.

## Reproduction Assessment

Yes for the source path: current main stages inbound attachments into `media/inbound/*` and the current read tool returns text after image sanitization without an inbound untrusted-content wrapper. I did not establish a live messaging reproduction in this read-only review.

## Solution Assessment

No, not as proposed. The narrow hardening is useful, but the maintainable path is to reuse the shared external-content wrapper and make the path classifier root-aware rather than adding a parallel wrapper and broad absolute matching.

- **[P1] Use the shared external-content wrapper:** `src/agents/pi-tools.read.ts:797-800`
  - body: `wrapInboundFileResult` wraps inbound reads with `wrapUntrustedPromptDataBlock`, creating a second security boundary separate from the `wrapExternalContent` contract used by OpenResponses and media-understanding.   - confidence: 0.87

  - confidence: 0.86
