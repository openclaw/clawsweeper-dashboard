## Summary

Keep this PR open, but it is not merge-ready: current main still has the reported inbound-file prompt-injection surface, while the PR head leaves attacker-controlled filename text in trusted wrapper metadata, overmatches absolute `media/inbound` paths, and still lacks real behavior proof.

## What This Changes

The PR adds an inbound-media read guard that wraps text from staged `media/inbound` files as untrusted prompt data, threads `containerWorkdir`, and adds inbound-guard tests.

## Best Possible Solution

Land a root-aware inbound-media wrapper that keeps attacker-supplied path text out of trusted prompt metadata, preserves ordinary code reads, adds regression coverage, and includes redacted real attachment-read proof.

## Reproduction Assessment

Yes. Source inspection shows current main stages sandboxed attachments under `media/inbound/*` and then returns read-tool text without an untrusted-data wrapper; I did not run a live messaging flow in this read-only review.

## Solution Assessment

No. The wrapper direction is right, but the branch should use static trusted metadata or move path metadata inside the guarded block, and classify inbound reads against the actual staged root instead of any absolute `media/inbound` segment.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Keep inbound filenames out of trusted wrapper labels:** `src/agents/pi-tools.read.ts:798`
  - body: `sanitizeForPromptLiteral` strips controls, but sender-controlled filename words still go into `params.label`, which the wrapper emits before `<untrusted-text>`. A filename that reads like an instruction can therefore appear as trusted prompt metadata; use a constant label or put path metadata inside the guarded block.
  - confidence: 0.82
- **[P2] Anchor inbound matching to the staged root:** `src/agents/pi-tools.read.ts:140`
  - body: The regex classifies any absolute path containing `/media/inbound/` as inbound, and the new test locks in `/workspace/src/media/inbound/parser.ts` as true. That wraps and escapes normal source reads for projects with that path shape; compare the resolved read path against the sandbox/workspace inbound root instead.
  - confidence: 0.86
