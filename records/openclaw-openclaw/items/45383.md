## Summary

Keep open: current main still has the inbound-media prompt-injection gap, but this PR should not merge as-is because the patch keeps sender-controlled filename text in trusted prompt label text, overmatches absolute `media/inbound` paths, and lacks after-fix real behavior proof.

## What This Changes

The PR adds an inbound-media path classifier, wraps text blocks read from `media/inbound/*`, forwards `containerWorkdir` into sandboxed reads, and adds inbound-guard unit tests.

## Best Possible Solution

Land a root-aware inbound-media guard that uses a constant trusted label, keeps sender-controlled path metadata inside guarded data or omits it, preserves current read-tool path mapping, and includes focused tests plus redacted real attachment-read proof.

## Reproduction Assessment

Yes, source-reproducible: current main stages sandboxed attachments under `media/inbound/*`, then the read tool returns sanitized text without an untrusted wrapper. I did not run a live messaging flow in this read-only review.

## Solution Assessment

No. The wrapper direction is right, but the patch should keep attacker-controlled path text out of trusted prompt labels and classify inbound files against the resolved staged root instead of any absolute `media/inbound` segment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep inbound filenames out of trusted labels:** `src/agents/pi-tools.read.ts:798`
  - body: `sanitizeForPromptLiteral` removes control characters, but the staged filename still appears before `<untrusted-text>` as trusted prompt text. Since the basename comes from sender-controlled media, instruction-like filename words can still be presented outside the guarded data block; use a constant trusted label and put any path metadata inside guarded content or omit it.
  - confidence: 0.82
- **[P2] Anchor inbound matching to the staged root:** `src/agents/pi-tools.read.ts:140`
  - body: The absolute-path regex treats any path containing `/media/inbound/` as inbound, and the tests lock in `/workspace/src/media/inbound/parser.ts` as true. That wraps and escapes normal repository files with this path shape, so classify against the resolved workspace-relative staged path or actual sandbox root instead.
  - confidence: 0.86
