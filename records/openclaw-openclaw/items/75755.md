## Summary

Keep open. The linked sanitizer bug is valid and this PR works in the right shared helper, but the proposed paragraph matcher is too broad and can delete legitimate visible replies that begin with JSON, code fences, or similar runtime-looking headings; the user-facing fix also needs the required changelog entry before merge.

## What This Changes

The PR replaces the modern runtime-context preface stripper with a paragraph-based block sanitizer in `src/agents/internal-runtime-context.ts` and adds regression tests for async-completion and conversation-metadata wrappers.

## Best Possible Solution

Keep the fix in the shared `internal-runtime-context` sanitizer, but make the parser structure-aware: strip the generated header, exact known async prompt scaffolding, and paired metadata sections while preserving the first non-wrapper user-visible paragraph, including JSON and code-fenced answers. Add false-positive tests and a changelog entry before merge.

## Reproduction Assessment

Yes. The issue reproduces at source level by passing the modern header plus notice, an async-completion wrapper body, and a visible reply through `sanitizeUserFacingText`; current main removes only the preface and leaves the wrapper body. The PR-specific regression is also reproducible from the diff: a visible answer that starts with a JSON object or code fence after an echoed preface would be consumed by the new wrapper-paragraph regex.

## Solution Assessment

No, not as written. The direction is right, but the stop rule should be narrower than matching standalone JSON, code fences, `Current time:`, or `Runtime:` as wrapper paragraphs without structural context.
