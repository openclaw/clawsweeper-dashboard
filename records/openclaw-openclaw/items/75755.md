## Summary

Keep open. The PR addresses the earlier changelog and JSON/fenced-reply false-positive concerns, but the new parser still does not match two actual generated runtime-context section shapes, so echoed trusted metadata or exec-state wrappers can still leak into user-facing output.

## What This Changes

The PR replaces the modern runtime-context preface stripper with a section-aware sanitizer, adds regression tests for async-completion and metadata wrappers, and adds a changelog entry.

## Best Possible Solution

Finish the shared sanitizer against the actual generator outputs: strip the modern header/notice, async-completion scaffolding, trusted inbound metadata markdown section, untrusted metadata blocks, and exec-state markdown section, while preserving the first non-wrapper visible reply paragraph. Keep the changelog entry and add regression tests using the real generated headings.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: feed the modern header/notice followed by `## Inbound Context (trusted metadata)` or `## Current Exec Session State` into the PR sanitizer and it stops at those headings because its matchers expect different strings.

## Solution Assessment

No, not as written. The direction is right and narrower than a generic regex scrubber, but the parser must be derived from the actual runtime-context builders before this is the maintainable fix for the linked leak.

Please consume the actual generated section and cover it in tests.
