## Summary

Keep open. The branch is still not mergeable because the latest head remains mismatched with current generator output: generated exec-state sections are single-newline markdown, while the PR parser still expects the exec heading as its own blank-line-delimited paragraph, and the metadata heading allowlist still omits several generated untrusted-context blocks.

## What This Changes

The PR updates internal runtime-context stripping, adds sanitizer regression tests, and adds a changelog entry for echoed modern runtime-context wrappers.

## Best Possible Solution

Finish the shared sanitizer so it consumes the exact sections produced by `buildExecOverridePromptHint`, `buildInboundMetaSystemPrompt`, and `buildInboundUserContextPrefix`, ideally with tests that build fixtures from those generators instead of hand-shaped markdown.

## Reproduction Assessment

Yes. Feed `stripInternalRuntimeContext` the modern header and notice followed by the exact single-newline output of `buildExecOverridePromptHint` and then a visible reply; the PR parser rejects the exec section because the heading and details are read as one paragraph.

## Solution Assessment

No. A section-aware sanitizer is the right direction, but this implementation is still fixture-shaped and not fully aligned with current generator output; the safer fix is to parse generated sections line-by-line or centralize the generated metadata sentinels.
