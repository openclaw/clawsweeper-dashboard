## Summary

Keep open: the cron-scoped repair is the right direction, but the patch still lets the new inheritance signal apply after a stored cron session model override wins selection, so a user-selected cron model can unexpectedly fall through to global fallback providers.

## What This Changes

The PR threads a cron-local fallback inheritance flag through isolated cron preflight and execution so matching string agent/default-primary jobs can use `agents.defaults.model.fallbacks`, with focused regression tests.

## Best Possible Solution

Keep the cron-local inheritance repair, but only for runs whose resolved model source is the default/agent string shorthand; stored session, payload, hook, object-primary, and differing string selections should remain strict.

## Reproduction Assessment

Yes. Source inspection reproduces the linked bug: a string agent model returns `[]`, cron passes that through, and the fallback runner treats `[]` as a strict override; I did not run tests because this review is read-only.

## Solution Assessment

No. The cron-scoped direction is best, but this implementation computes the inheritance signal before stored session model overrides are known, so it is still too broad.
