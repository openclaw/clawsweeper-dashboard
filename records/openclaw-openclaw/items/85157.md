## Label Justifications

- `P1`: A bound Telegram/Codex conversation can trap its own detach/control command, breaking an active channel workflow and forcing manual state repair.
- `impact:message-loss`: The reported control messages are misrouted into the bound plugin thread instead of reaching the OpenClaw command dispatcher.
- `impact:session-state`: The active conversation binding can become stale or unwanted with no in-channel way to inspect or detach it.

## Summary

Keep open: current main still lets plugin-owned conversation bindings claim inbound text before the normal command path, so the reported self-locking `/codex detach` failure is plausible from source inspection and not covered by an existing merged fix.

## What This Changes

Authorized native/plugin control commands such as `/codex detach` should bypass plugin-owned conversation binding dispatch so users can inspect or detach the active binding.

## Best Possible Solution

Add a shared pre-claim escape hatch for authorized native/control/plugin commands at the plugin-owned inbound dispatch boundary, with regression coverage proving `/codex detach` and `/codex binding` bypass the Codex claim while normal text still routes to the plugin.

## Reproduction Assessment

Yes at source level: create a plugin-owned binding for the Telegram conversation and send `/codex detach`; the current dispatch path resolves the binding and calls the plugin inbound claim before command handling. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

Yes: the narrow fix is to preserve plugin-bound priority for ordinary text while routing authorized native/control/plugin commands to the existing command handlers before the binding claim. A Codex-only workaround would leave the same trap for future plugin-owned bindings.
