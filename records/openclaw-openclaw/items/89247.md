## Summary

Keep open: current main still has source-backed paths where Codex/native tool failures become user-visible failed-tool warnings, generic external-run fallback copy remains available in direct conversations, and Feishu can append an error final after a successful-looking answer. Related issues cover important slices, but no single canonical item fully replaces this Feishu direct aggregate UX/security report.

## What This Changes

The issue asks OpenClaw to suppress or translate raw Feishu direct failure banners, generic fallback replies, Codex stall copy, malformed tool labels, and false failures after successful work into a consistent user-facing status model.

## Best Possible Solution

Define one shared direct-chat failure-status policy across Codex projection, embedded-run warning generation, Feishu delivery, generic fallback copy, and reset/new recovery, then prove it with focused tests plus a live Feishu direct replay.

## Reproduction Assessment

No for a fresh full Feishu direct end-to-end replay; yes for source-reproducible components. Current main still has tests and code paths for visible mutating/exec warnings after success-like replies, Feishu appended error finals, Codex native lastToolError projection, and generic direct fallback copy.

## Solution Assessment

No single sanitizer-only or Feishu-only fix is the best solution. The durable fix needs a maintainer-approved shared status policy that preserves real failure signals while keeping internal diagnostics out of normal direct-chat copy.
