## Summary

Keep open: the renderer approach is plausible, but the current PR head still drops same-session WebChat media-only `message` sends because the new render gate only considers sent text and ignores `messagingToolSentMediaUrls`. The sanitizer concern from the earlier review appears repaired, and the related automatic-delivery alternative remains open for maintainer choice.

## What This Changes

The PR treats WebChat `message_tool_only` `message(action="send")` calls as same-session WebChat replies, renders their text through source-suppression-safe payloads, and adds focused tests for the message tool, Codex bridge, payloads, and metadata preservation.

## Best Possible Solution

If maintainers choose the renderer approach, carry both text and media from committed same-session WebChat message-tool sends into source-suppression-safe reply payloads, including media-only sends.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a WebChat `message_tool_only` `message(action="send", mediaUrl="...")` call records message-tool media delivery evidence, but the new renderer only consumes sent text and therefore creates no media payload.

## Solution Assessment

No, not yet. The renderer can be maintainable if it preserves the WebChat-only scope while rendering both sanitized text and media; otherwise the automatic-delivery PR remains the narrower policy fix.
