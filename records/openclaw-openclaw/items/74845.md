## Summary

Keep open. Current main still exposes node-host invocation as a one-shot request/result path, and the requested progress-capable command primitive remains real platform design work rather than conservative cleanup.

## What This Changes

Add an opt-in long-running or streaming node-host command path that can send progress or intermediate updates before the final result.

## Best Possible Solution

Design an additive, opt-in node invoke progress contract that preserves existing one-shot behavior while exposing request-correlated progress updates through the gateway protocol and plugin/node-host SDK.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is static inspection of the current protocol schemas, method list, gateway invoke handler, node registry, plugin node-host command contract, docs, and related items, all of which still show final-result behavior only.

## Solution Assessment

Unclear pending maintainer design. A shared versioned protocol and SDK primitive is the maintainable direction; increasing timeouts or relying on generic `node.event` alone would not satisfy progress delivery to the pending invoke caller.
