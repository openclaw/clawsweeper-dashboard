## Summary

Keep open. Current main still has one-shot `node.invoke` plus generic `node.event` and exec lifecycle events, but no request-correlated progress or streaming primitive for long-running node-host commands. This needs maintainer protocol and SDK design before implementation.

## What This Changes

Add an opt-in long-running node-host command path that can send request-correlated progress or intermediate updates before a final result.

## Best Possible Solution

Design an additive, version-aware node invoke progress contract that keeps existing one-shot behavior while updating protocol schemas, Gateway handling, node-host dispatch, plugin SDK/runtime types, docs, and focused tests together.

## Reproduction Assessment

Not applicable as a feature request. Static inspection of current main gives a high-confidence verification path: node invoke, node-host command dispatch, plugin runtime typing, method discovery, node events, and docs are still final-result oriented.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol/SDK primitive is the maintainable direction; timeout tuning or generic `node.event` alone would not satisfy request-correlated caller progress updates.
