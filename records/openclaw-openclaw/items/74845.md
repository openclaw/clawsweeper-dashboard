## Summary

Keep open. Current main still treats node-host invocation as a one-request, one-final-response flow: the registry stores one pending promise, the protocol exposes invoke/result/event but no invoke-progress frame, and plugin node-host commands return a single string payload. The request is valid platform work, but it crosses Gateway protocol, node-host lifecycle, plugin SDK, authorization, timeout, and cleanup semantics, so it needs maintainer design rather than automatic cleanup.

## What This Changes

Request a standard opt-in long-running or streaming node-host command path that can send caller-visible progress updates before returning a final result.

## Best Possible Solution

Add a versioned, additive, opt-in node invoke progress contract that preserves existing one-shot behavior while allowing node-host and plugin-owned commands to emit caller-visible progress before the final result, with protocol schemas, SDK/docs, lifecycle/security rules, and focused tests updated together.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is static inspection of the node invoke registry, protocol schemas, method list, node-host dispatch, plugin command typing, tests, and docs; those show final-result behavior rather than a progress-capable command path.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol and SDK primitive is the maintainable direction for the stated platform need; increasing timeouts, polling, or reusing generic node.event notifications alone would not satisfy caller-visible progress for a pending invoke.
