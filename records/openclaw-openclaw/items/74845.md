## Summary

Keep this issue open. Current main still treats `node.invoke` and plugin node-host commands as a one-request/one-final-response contract; existing node events, exec-finished notifications, per-call timeout knobs, and ACP/session streaming paths do not provide caller-visible progress for a pending node-host command. The request is valid platform work, but it needs maintainer protocol and SDK design before an autonomous fix PR is safe.

## What This Changes

Add an opt-in long-running or streaming node-host command path that can deliver progress or intermediate updates before returning a final result.

## Best Possible Solution

Design and add an additive, versioned node-invoke progress contract that preserves existing one-shot behavior while letting authorized node-host and plugin-owned commands emit request-id-scoped progress frames before a final result, with protocol schemas, SDK/runtime docs, node-host dispatch, caller delivery, lifecycle/security rules, and focused tests updated together.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is static contract inspection: current node invoke source, protocol schemas, plugin command typing, CLI/tool surfaces, tests, and docs all show final-result behavior rather than caller-visible progress for a pending invoke.

## Solution Assessment

Unclear until maintainers choose the design. A shared additive protocol/SDK primitive is the maintainable direction for the stated platform need; increasing timeouts or relying on generic `node.event` notifications would not satisfy request-scoped progress for the caller awaiting `node.invoke`.
