## Summary

Keep open. Current main still treats node-host invocation as a request plus one final result, while the requested request-correlated progress/update primitive remains additive Gateway protocol and SDK design work; the related timeout PR only changes the wait budget and does not solve progress delivery.

## What This Changes

Request support for long-running node-host commands that can emit progress or intermediate updates before returning a final result.

## Best Possible Solution

Design an additive, version-aware node invoke progress contract that preserves one-shot behavior while updating protocol schemas, Gateway handling, node-host dispatch, plugin SDK typing, docs, and focused tests together.

## Reproduction Assessment

Not applicable as a feature request. Static inspection gives a high-confidence verification path: node invoke, protocol schema, plugin node-host commands, plugin runtime typing, generic node events, and docs are still final-result oriented.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol/SDK primitive is the maintainable direction; a timeout config or generic best-effort `node.event` usage alone would not provide reusable request-correlated progress semantics.
