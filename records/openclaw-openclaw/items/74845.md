## Summary

Keep open. Current main and the latest release still expose node-host invocation as one final result or timeout, and the related timeout PR only overlaps with the problem rather than adding caller-correlated progress updates.

## What This Changes

Request a first-class long-running node-host command path that lets callers receive progress or intermediate updates before the final result.

## Best Possible Solution

Design an additive, permission-checked node-host progress contract that preserves one-shot node.invoke while updating protocol schemas, gateway handling, node-host dispatch, plugin runtime/SDK typing, docs, and focused tests together.

## Reproduction Assessment

Yes, at source level. Current main has no request-correlated node.invoke progress event, streaming invoke API, or plugin node-host command result stream.

## Solution Assessment

Unclear final API shape. A shared additive protocol and SDK primitive is the right layer, while timeout config, polling, or ad hoc generic node.event use would not provide reusable caller-correlated progress semantics.
