## Summary

Keep open. Current main still exposes node-host invocation as a one-request, one-final-response contract, with no invoke-id-scoped progress or intermediate-update path. The related timeout PR #68857 and ACP/session progress issue #54895 are adjacent, but neither supersedes this platform primitive.

## What This Changes

Request an opt-in long-running or streaming node-host command path that can emit progress or intermediate updates before returning a final result.

## Best Possible Solution

Add a versioned, additive node-invoke progress contract that preserves the current one-shot behavior while allowing opt-in node-host and plugin-owned commands to emit caller-visible progress before the final result, with schema, SDK/docs, lifecycle/security rules, and focused tests updated together.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is static inspection of node invoke registry, protocol schemas, method list, node-host dispatch, plugin command typing, docs, and related issue context; those all show final-result behavior rather than a progress-capable command path.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol and SDK primitive is the maintainable direction for the stated platform need; increasing timeouts or reusing generic `node.event` notifications would not provide caller-visible progress for a pending invoke.
