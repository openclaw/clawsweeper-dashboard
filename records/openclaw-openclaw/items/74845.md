## Summary

Keep open. Current main and the latest release still implement `node.invoke` as a one-shot request/result with a timeout, while plugin node-host commands return one final string; the related session-progress issue and timeout-config PR do not replace a caller-correlated node-host progress primitive.

## What This Changes

Request an opt-in node-host command path that can stream progress or intermediate updates before returning a final result.

## Best Possible Solution

Design an additive, permission-checked node-host progress contract that preserves one-shot `node.invoke` behavior while updating protocol schemas, gateway handling, node-host dispatch, plugin SDK/runtime typing, docs, and focused tests together.

## Reproduction Assessment

Yes, as a source-level feature gap rather than a bug repro: current main has no request-correlated `node.invoke` progress event, streaming invoke API, or node-host command result stream.

## Solution Assessment

Unclear final API shape. A shared additive protocol/SDK primitive is the right layer, while a timeout config or generic `node.event` workaround would not provide reusable caller-correlated progress semantics.
