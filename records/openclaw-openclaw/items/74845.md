## Summary

Keep open. Current main and the latest release still expose node-host invocation as a one-shot request/result path, so the requested progress/update primitive remains real Gateway protocol, node-host, SDK, lifecycle, and security design work.

## What This Changes

The issue requests a standard streaming or long-running node-host command path that can emit progress updates before returning a final result.

## Best Possible Solution

Design an additive, permission-checked node.invoke progress contract that preserves one-shot behavior while updating protocol schemas, Gateway handling, node-host dispatch, plugin SDK/runtime typing, docs, and focused tests together.

## Reproduction Assessment

Yes, as a source-level feature gap rather than a bug repro: current main has no request-correlated node.invoke progress event, streaming invoke API, or node-host command result stream.

## Solution Assessment

Unclear final API shape. A shared additive protocol/SDK primitive is the right layer, but a timeout config or generic node.event workaround alone would not provide reusable caller-correlated progress semantics.
