## Summary

Keep open. Current main still exposes `node.invoke` and plugin node-host commands as a one-shot request/result path, with no request-correlated progress frame; the related timeout-config PR and inter-session progress issue do not supersede this broader protocol/SDK design work.

## What This Changes

Request an additive long-running node-host command path that can deliver progress or intermediate updates before a final result.

## Best Possible Solution

Design an additive, versioned, opt-in node invoke progress contract that preserves existing one-shot behavior while giving node-host and plugin-owned commands a caller-visible progress channel before the final result.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is static inspection of current protocol schemas, method discovery, Gateway invoke handling, node-host dispatch, plugin command typing, docs, and related items, all of which still show final-result behavior only.

## Solution Assessment

Unclear pending maintainer design. A shared protocol and SDK primitive is the maintainable direction; increasing timeouts, polling, or relying on generic `node.event` alone would not satisfy progress delivery to the pending invoke caller.
