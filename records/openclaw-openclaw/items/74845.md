## Summary

Keep open. Current main still exposes node-host invocation as a one-request, one-final-result flow: the pending invoke registry resolves only on `node.invoke.result` or timeout, the Gateway protocol has no invoke-progress frame, and plugin node-host commands return a single string payload. The request is valid platform work, but it needs maintainer design across protocol, SDK, authorization, timeout, cancellation, and cleanup semantics rather than cleanup closure.

## What This Changes

Add first-class support for long-running node-host commands that can emit progress or intermediate updates before returning a final result.

## Best Possible Solution

Add a versioned, additive, opt-in node invoke progress contract that preserves existing one-shot behavior while allowing node-host and plugin-owned commands to emit caller-visible progress before the final result, with protocol schemas, SDK/docs, lifecycle/security rules, and focused tests updated together.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is static inspection of the node invoke registry, protocol schemas, method list, Gateway handler, node-host dispatch, plugin command typing, docs, and tests, which still show final-result behavior rather than a progress-capable command path.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol and SDK primitive is the maintainable direction for the stated platform need; increasing timeouts, polling, or reusing generic `node.event` notifications alone would not satisfy caller-visible progress for a pending invoke.
