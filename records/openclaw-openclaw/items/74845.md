## Summary

Keep open: current main still models `node.invoke` as a one-shot request/result flow, and neither the Gateway protocol nor the plugin node-host command contract has a request-correlated progress or streaming path.

## What This Changes

Add an opt-in long-running node-host command capability that can emit progress or intermediate updates before the final result.

## Best Possible Solution

Design an additive, versioned node invoke progress contract that preserves existing one-shot behavior while giving node-host and plugin-owned commands a caller-visible, request-correlated progress channel before the final result.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is static inspection of current protocol schemas, method discovery, Gateway invoke handling, node-host dispatch, plugin command typing, docs, and related open work.

## Solution Assessment

Unclear pending maintainer design. A shared protocol/SDK primitive is the maintainable direction; increasing timeouts or relying on generic `node.event` alone would not provide request-correlated progress to the pending invoke caller.
