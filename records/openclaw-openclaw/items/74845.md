## Summary

Keep open. Current main still implements node-host invocation as a one-request, one-final-result path: the Gateway stores a single pending invoke, the protocol exposes invoke/result/event but no invoke-progress frame, the Gateway handler responds once after `NodeRegistry.invoke`, and plugin node-host commands return a single string. The request is valid platform work, but it needs maintainer design across protocol, SDK, lifecycle, timeout, cancellation, authorization, and cleanup semantics.

## What This Changes

Add first-class support for long-running node-host commands that can emit progress or intermediate updates before returning a final result.

## Best Possible Solution

Add a versioned, additive, opt-in node invoke progress contract that preserves existing one-shot behavior while allowing node-host and plugin-owned commands to emit caller-visible progress before the final result, with protocol schemas, SDK/docs, lifecycle/security rules, and focused tests updated together.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is static inspection of the node invoke registry, protocol schema, method list, Gateway handler, node-host dispatch, plugin command typing, docs, and related reports, which still show final-result behavior rather than a progress-capable command path.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol and SDK primitive is the maintainable direction for the stated platform need; increasing timeouts, polling, or reusing generic `node.event` notifications alone would not satisfy caller-visible progress for a pending invoke.
