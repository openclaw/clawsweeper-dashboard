## Summary

Keep open. Current main still treats node-host invocation as one request and one final result, while the requested progress/streaming path is additive Gateway protocol, node-host dispatch, plugin runtime, and SDK design work.

## What This Changes

Add a shared opt-in node-host command path for long-running commands to emit progress or intermediate updates before returning a final result.

## Best Possible Solution

Design an additive, version-aware node invoke progress contract that preserves current one-shot behavior while updating protocol schemas, Gateway handling, node-host dispatch, plugin SDK/runtime typing, docs, and focused tests together.

## Reproduction Assessment

Not applicable as a feature request. Static inspection of current main gives a high-confidence verification path: node invoke, protocol schemas, method discovery, node-host dispatch, plugin runtime typing, and docs are still final-result oriented.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol/SDK primitive is the maintainable direction; extending timeouts or relying on uncorrelated generic node.event messages would not provide request-correlated invoke progress semantics.
