## Summary

Keep open. Current main still exposes node-host invocation as a one-request, one-final-result flow, so the requested progress-capable path remains real platform design work rather than cleanup closure.

## What This Changes

The issue requests an opt-in node-host command path that can emit progress or intermediate updates before returning a final result.

## Best Possible Solution

Add a versioned, additive, opt-in node invoke progress contract that preserves existing one-shot behavior while giving node-host and plugin-owned commands a caller-visible progress channel before the final result.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is static inspection of protocol schemas, method lists, Gateway invoke handling, node-host dispatch, plugin command typing, docs, and related items, which still show final-result behavior only.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol and SDK primitive is the maintainable direction; increasing timeouts, polling, or reusing generic `node.event` alone would not satisfy progress delivery for a pending invoke caller.
