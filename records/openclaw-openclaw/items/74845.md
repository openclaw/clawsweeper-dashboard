## Summary

Keep open. Current main still has a final-result-only node invoke path, and the requested progress-capable long-running node-host primitive remains real platform design work rather than cleanup closure.

## What This Changes

The issue requests an opt-in long-running or streaming node-host command path that can emit progress or intermediate updates before returning a final result.

## Best Possible Solution

Add an additive, versioned, opt-in node invoke progress contract that preserves existing one-shot behavior while allowing node-host and plugin-owned commands to emit caller-visible progress before a final result, with protocol schemas, SDK/docs, lifecycle/security rules, and focused tests updated together.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is static inspection of the node invoke registry, protocol schemas, method list, Gateway handler, node-host dispatch, plugin command typing, docs, and related issues, which still show final-result behavior rather than progress-capable invokes.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol and SDK primitive appears to be the maintainable direction; increasing timeouts, polling, or relying on generic `node.event` alone would not provide progress tied to a pending invoke caller.
