## Summary

Keep open. Current main still exposes node-host invocation as a one-request, one-final-result path; there is no progress frame, streaming invoke method, or plugin node-host callback contract for intermediate updates. The request is valid platform work, but it needs maintainer design because it crosses Gateway protocol, node-host lifecycle, Plugin SDK compatibility, authorization, timeouts, cancellation, and cleanup semantics.

## What This Changes

Add first-class support for long-running node-host commands that can emit progress or intermediate updates before returning a final result.

## Best Possible Solution

Add an additive, opt-in node invoke progress contract that preserves existing one-shot behavior while allowing node-host and plugin-owned commands to emit caller-visible progress before the final result, with protocol schemas, SDK/docs, lifecycle/security rules, and focused gateway/node-host tests updated together.

## Reproduction Assessment

Not applicable as a feature request. We do have a high-confidence verification path by static inspection of current main: the registry, schemas, method list, node-host dispatch, Plugin SDK types, tests, and docs all show final-result behavior rather than a progress-capable invoke path.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol and SDK primitive is the maintainable direction for the stated platform need; increasing timeouts, polling, or reusing generic node.event notifications alone would not satisfy caller-visible progress for a pending invoke.
