## Summary

Keep open. Current main still exposes `node.invoke` as a one-shot request/result path, while plugin node-host commands return one final payload and there is no request-correlated progress or streaming invoke contract.

## What This Changes

Add an opt-in long-running or streaming node-host command path so callers can receive progress updates before the final result.

## Best Possible Solution

Design an additive, opt-in node invoke progress contract that preserves existing one-shot behavior while giving node-host and plugin-owned commands a caller-visible progress channel before the final result.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is static inspection of protocol schemas, method discovery, Gateway invoke handling, node-host dispatch, plugin command typing, docs, and related items, which still show final-result behavior only.

## Solution Assessment

Unclear pending maintainer design. A shared versioned protocol and SDK primitive is the maintainable direction; increasing timeouts, polling, or relying on generic `node.event` alone would not provide request-correlated progress to the pending invoke caller.
