## Summary

Keep open. Current main still has a one-shot `node.invoke` / node-host command path with no request-correlated progress frame, so the requested capability remains real platform design work; the related timeout-config PR and ACP/session progress issue do not supersede it.

## What This Changes

Add first-class support for long-running node-host commands that can emit progress or intermediate updates before returning a final result.

## Best Possible Solution

Design an additive, opt-in node invoke progress contract that preserves existing one-shot behavior while giving node-host and plugin-owned commands a caller-visible progress channel before the final result.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is static inspection of protocol schemas, method discovery, Gateway invoke handling, node-host dispatch, plugin command typing, docs, and related items, which still show final-result behavior only.

## Solution Assessment

Unclear pending maintainer design. A shared versioned protocol and SDK primitive is the maintainable direction; increasing timeouts, polling, or using generic `node.event` alone would not satisfy progress delivery to the pending invoke caller.
