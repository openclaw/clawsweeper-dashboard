## Summary

Keep open. Current main still exposes node-host invocation as a one-request, one-final-response contract; the existing `node.event` path and generic Gateway `expectFinal` request behavior do not provide caller-visible progress updates for a pending `node.invoke`. The adjacent timeout PR #68857 and ACP/session progress issue #54895 are related but do not supersede this platform primitive.

## What This Changes

Request to add an opt-in long-running or streaming node-host command path that can emit progress or intermediate updates before returning a final result.

## Best Possible Solution

Add a versioned, opt-in node-invoke progress contract that preserves current one-shot behavior while allowing node-host and plugin-owned commands to emit caller-visible progress before the final result, with protocol schemas, SDK/docs, lifecycle/security rules, and focused tests updated together.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is static inspection of the node invoke registry, protocol schemas, method list, node-host dispatch, plugin command typing, tests, and docs; those all show final-result behavior rather than a progress-capable command path.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol/SDK primitive is the maintainable direction for the stated platform need; increasing timeouts or reusing generic `node.event` notifications would not satisfy caller-visible progress for a pending invoke.
