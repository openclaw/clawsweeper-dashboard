## Summary

Keep open. Current main still treats node-host invocation as a one-shot request with one final result, while the requested request-correlated progress or streaming primitive would be additive Gateway protocol, node-host, plugin runtime, and SDK design work.

## What This Changes

Add a shared opt-in node-host command path for long-running commands to emit progress or intermediate updates before returning a final result.

## Best Possible Solution

Design an additive, version-aware node invoke progress contract that preserves current one-shot behavior while updating protocol schemas, Gateway handling, node-host dispatch, plugin SDK/runtime typing, docs, and focused tests together.

## Reproduction Assessment

Not applicable as a feature request. Static inspection gives high-confidence verification of the gap: current main has one pending invoke promise, one final result path, and plugin APIs without a progress sink or subscription.

## Solution Assessment

Unclear until maintainers choose the protocol shape. A shared additive progress/event contract is the maintainable direction; extending timeouts or relying on uncorrelated generic node.event messages would not provide the requested invoke progress semantics.
