## Summary

Keep open. Current main and the latest release still expose node-host invocation as a one-shot `node.invoke` request with one final result or timeout, while the open timeout-config PR only covers a partial timeout knob and explicitly defers heartbeat/progress semantics.

## What This Changes

Request first-class support for long-running node-host commands to send progress or intermediate updates before returning a final result.

## Best Possible Solution

Design an additive, permission-checked node-host progress contract that preserves existing one-shot `node.invoke` while updating gateway protocol schemas, gateway handling, node-host dispatch, plugin SDK typing, docs, and focused tests together.

## Reproduction Assessment

Yes, at source level. Current main has no request-correlated `node.invoke` progress event, streaming invoke API, or plugin node-host command result stream; I did not run a live node-host scenario in this read-only review.

## Solution Assessment

Unclear final API shape. A shared additive protocol and SDK primitive is the right layer, while timeout config, polling, or ad hoc `node.event` usage would not provide reusable caller-correlated progress semantics.
