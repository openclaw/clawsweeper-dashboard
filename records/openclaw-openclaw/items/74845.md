## Summary

Keep open. Current main still treats node-host invocation as a one-request/one-final-result contract, while existing node events and channel preview/tool-progress streaming are separate surfaces and do not provide caller-visible progress for a pending node.invoke. The request is valid platform work, but it needs maintainer design before implementation.

## What This Changes

Request to add an opt-in long-running or streaming node-host command path that can emit progress/intermediate updates before the final result.

## Best Possible Solution

Add a versioned, opt-in node-invoke progress contract that preserves existing one-shot behavior while allowing node-host and plugin-owned commands to emit caller-visible progress before the final result, with protocol schemas, SDK/docs, lifecycle/security rules, and focused tests updated together.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is static contract inspection: node.invoke, node protocol schemas, node-host dispatch, plugin command typing, tests, and docs all show final-result behavior rather than a progress-capable command path.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol/SDK primitive is the maintainable direction for the stated platform need; increasing timeouts or reusing generic node.event notifications would not satisfy caller-visible progress for a pending invoke.
