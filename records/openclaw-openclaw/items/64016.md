## Summary

Keep open. Current main still allows session-scoped and other non-interval heartbeat wakes to bypass the interval due gate, and an open closing PR is already attached, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that external wake events arriving during or soon after a scheduled heartbeat not trigger extra heartbeat runs inside the configured interval window.

## Best Possible Solution

Land #75439 or an equivalent main-branch fix that centralizes the heartbeat due/cooldown decision for targeted and broadcast wake paths, preserves explicit manual/immediate semantics, and adds regression coverage for repeated exec-event wakes within nextDueMs.

## Reproduction Assessment

Yes. The issue has a high-confidence unit reproduction path: start the heartbeat runner with fake timers, fire an exec-event wake, then fire repeated exec-event wakes before nextDueMs and observe current main calling runOnce more than once.

## Solution Assessment

Yes, the best direction is a central cooldown gate shared by targeted and broadcast wake paths. A blanket skip of all non-interval wakes would be too broad unless it preserves documented manual and explicit immediate wake behavior.
