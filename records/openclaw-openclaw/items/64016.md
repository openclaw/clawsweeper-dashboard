## Summary

Keep open. Current main still has the reported behavior shape: wake requests that arrive during an active heartbeat are kept pending and re-drained after the run, while session-scoped exec wakes call the heartbeat runner without the interval not-due guard, so the extra LLM run/checkpoint churn remains plausible and unaddressed by the existing duplicate-output suppression.

## What This Changes

The issue reports duplicate heartbeat runs within one configured interval when external OpenClaw agent/exec wake events arrive while a scheduled heartbeat is already running.

## Best Possible Solution

Add a narrow heartbeat scheduler or wake-path cooldown/dedupe for automatic external wakes queued while an interval heartbeat is already in progress, with regression coverage for the same-agent/session overlap case and explicit preservation of manual/system/cron immediate wake semantics.

## Reproduction Assessment

Yes. A high-confidence unit reproduction can use fake timers around `startHeartbeatRunner`/`requestHeartbeatNow`: let an interval run start and hang, enqueue an `exec-event` wake for the same session, resolve the first run, and observe the second `runOnce()` before the configured interval elapses.

## Solution Assessment

Yes, a targeted scheduler/wake fix is the best direction, but the issue’s broad option to skip all non-interval wakes needs refinement. The maintainable fix should distinguish automatic external wake noise from explicitly documented immediate wake modes.
