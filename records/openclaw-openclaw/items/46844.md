## Summary

Keep open. Current main and latest release still lack a between-turn Talk Mode idle timeout after Voice Wake activates Talk, and the remaining work is a public Talk lifecycle/config/protocol decision rather than an already-implemented behavior.

## What This Changes

The issue requests a configurable idle timeout that deactivates Talk Mode after Voice Wake-triggered conversations go idle.

## Best Possible Solution

Add an additive, maintainer-approved Talk lifecycle setting across config/schema/protocol/docs and macOS Talk runtime so idle Voice Wake-triggered Talk sessions return to wake listening while silenceTimeoutMs remains transcript-finalization-only.

## Reproduction Assessment

Yes, source-reproducible for the missing capability. Enable Voice Wake with Trigger Talk Mode, complete one Talk exchange, then remain silent; current main has no accepted idle-timeout field and macOS Talk ignores empty between-turn silence while resuming listening after replies or errors.

## Solution Assessment

Unclear. The requested direction is plausible and bounded, but maintainers should decide the public Talk lifecycle contract before implementation commits to scope, naming, defaults, bounds, and reset semantics.
