## Summary

Current main still keeps Talk Mode active and resumes recognition after every completed turn; no equivalent inactivity timeout or canonical implementation PR exists. This remains a valid feature request, but implementation depends on maintainer decisions about activation scope, cross-platform behavior, defaults, and what counts as activity.

## What This Changes

Add configurable inactivity-based deactivation for continuous Talk Mode after Voice Wake activates it.

## Best Possible Solution

Define one lifecycle-owned inactivity contract with explicit activation scope and phase-safe user-activity semantics, then deactivate through the existing Talk Mode controller so audio teardown completes and Voice Wake resumes cleanly without changing `silenceTimeoutMs`.

## Reproduction Assessment

Yes at source level: enable Talk Mode, complete a turn, and current main calls the listening restart path while the runtime remains enabled; repository search shows no session-idle exit timer.

## Solution Assessment

No, the proposed fields are not yet the best settled solution. An idle timeout is a reasonable direction, but maintainers must first decide Voice Wake-only versus universal scope, shared versus platform-specific configuration, default/disabled behavior, and which lifecycle phases reset or suspend the timer.
