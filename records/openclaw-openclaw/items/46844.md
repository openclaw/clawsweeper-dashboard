## Summary

Current main still restarts macOS Talk Mode recognition after every completed or failed turn and only resumes Voice Wake after Talk Mode is explicitly disabled; no between-turn inactivity shutdown or active canonical implementation exists. The request remains useful, but maintainers must define whether auto-deactivation applies only to voice-wake activation, what resets the timer, the default/disabled behavior, and whether the setting is app-local or Gateway-owned.

## What This Changes

Requests a configurable inactivity timeout that turns off macOS Talk Mode after voice-wake conversations and resumes wake-word listening.

## Best Possible Solution

Define an app-owned Talk activation mode that distinguishes voice-wake sessions from manually enabled sessions, then implement one cancellable inactivity timer with explicit reset events and a documented disabled/default value; expose Gateway configuration only if maintainers intentionally want the same lifecycle contract across Talk clients.

## Reproduction Assessment

Yes, from source inspection: macOS Talk Mode restarts recognition after completed and failed turns and remains active until explicitly disabled. This establishes the reported missing lifecycle behavior without requiring a failing-test reproduction.

## Solution Assessment

Unclear as proposed: an inactivity timer is appropriate, but a global `talk.idleTimeoutS` option may be broader than the voice-wake-specific problem. The narrowest maintainable solution should first distinguish voice-wake activation from manual Talk Mode and define exact reset/default semantics.

## Risks / Open Questions

- An unconditional timeout could unexpectedly end manually activated Talk Mode, so activation origin may need to be represented explicitly.
- Resetting on bot replies could keep sessions alive during long playback or autonomous output, while resetting only on user speech changes the proposed conversational semantics.
- A Gateway-level `talk` option would affect a cross-platform configuration contract even though the motivating lifecycle currently exists only in the macOS Voice Wake integration.
- Changing the default from persistent Talk Mode to automatic shutdown would alter existing user behavior and requires explicit upgrade and UX consideration.
