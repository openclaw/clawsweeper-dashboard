## Summary

Keep open. Current main still lacks a between-turn Talk idle timeout: Voice Wake can activate Talk Mode, but Talk only has transcript-finalization silence handling and resumes listening after replies or errors. The related heartbeat/session cleanup items are broader lifecycle requests, not a replacement for this macOS Voice Wake-triggered Talk behavior.

## What This Changes

The issue requests a configurable Talk Mode idle timeout that auto-deactivates Voice Wake-triggered Talk after between-turn inactivity.

## Best Possible Solution

Define an additive Talk lifecycle setting, validate and expose it through talk.config, document it, and honor it in macOS Talk so idle Voice Wake-triggered sessions return to wake listening without changing silenceTimeoutMs transcript-finalization semantics.

## Reproduction Assessment

Yes. Source review gives a high-confidence path: enable Voice Wake with Trigger Talk Mode, complete one Talk exchange, then remain silent; empty between-turn silence is ignored and the runtime resumes listening.

## Solution Assessment

Unclear as proposed. The direction is narrow and maintainable, but maintainers should choose the public lifecycle contract before an implementation commits to idleTimeoutS/reset flags, defaults, or reset points.
