## Summary

Keep open. Current main still lacks a between-turn Talk idle timeout: Voice Wake can activate Talk Mode directly, but Talk only uses `silenceTimeoutMs` to finalize non-empty transcripts and then resumes listening after replies or errors.

## What This Changes

The issue requests a configurable Talk Mode idle timeout after Voice Wake activation so inactive voice sessions auto-deactivate before the hard stop.

## Best Possible Solution

Add an additive Talk lifecycle setting that is validated in config, exposed through `talk.config`, documented on the public Talk and Voice Wake pages, and honored by macOS Talk so idle sessions deactivate and Voice Wake resumes while `silenceTimeoutMs` remains intra-turn transcript finalization.

## Reproduction Assessment

Yes. Enable Voice Wake with Trigger Talk Mode, wake into Talk, complete one exchange, then remain silent; current source shows empty between-turn silence is ignored and the runtime resumes recognition after replies or failures.

## Solution Assessment

Unclear as proposed. The direction is maintainable, but maintainers should define the public lifecycle contract before implementation chooses names, defaults, limits, disabled behavior, and exact timer reset events.
