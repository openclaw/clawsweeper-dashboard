## Summary

Keep open. Current main and the latest v2026.6.6 release still lack the requested between-turn Talk idle deactivation: the macOS runtime only has intra-turn transcript silence handling, Voice Wake can still enable Talk Mode directly, and the public Talk config/protocol/docs expose no idle-timeout setting. Related heartbeat/session cleanup reports are broader lifecycle requests and do not supersede this focused Voice Wake-to-Talk gap.

## What This Changes

The issue requests a configurable Talk Mode idle timeout that deactivates voice conversation after a period without user interaction following Voice Wake activation.

## Best Possible Solution

Define a maintainer-approved Talk lifecycle contract, then implement one additive idle timer with validated config, gateway protocol/test coverage, public docs, and macOS runtime handling that returns Voice Wake-triggered sessions to wake listening without changing silenceTimeoutMs semantics.

## Reproduction Assessment

Yes, at source level. Current main shows Voice Wake can enable Talk Mode directly, while TalkModeRuntime only finalizes non-empty transcripts after intra-turn silence and then restarts listening between turns.

## Solution Assessment

Unclear as proposed. An idle timer is the right class of solution, but maintainers still need to choose the lifecycle scope, reset events, field name, units, default, limits, and disabled behavior before implementation is the best fix.

## Risks / Open Questions

- The product/API contract is unsettled: maintainers need to choose whether the idle timeout applies only to Voice Wake-triggered Talk sessions or to all Talk Mode sessions, plus field name, units, default, maximum, and disabled behavior.

- Timer reset semantics are product-sensitive: user speech, assistant reply start/end, playback errors, and long thinking pauses can each produce different hands-free behavior.
