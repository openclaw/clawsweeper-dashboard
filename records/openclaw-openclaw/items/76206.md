## Summary

Keep open: current `main` still has the reported crash path because read-only status parsing leaves persisted session fields as raw JSON while the model-selection helpers call `.trim()` on them as if they are strings.

## What This Changes

The issue asks OpenClaw to prevent `openclaw status` from crashing when persisted session model/provider fields contain non-string values.

## Best Possible Solution

Treat persisted session model/provider fields as unknown at the status/model-selection boundary, fall back on invalid non-strings, and add regression coverage for corrupted `sessions.json` entries.

## Reproduction Assessment

Yes. A non-string `model`, `modelProvider`, `providerOverride`, or `modelOverride` in a parsed session entry reaches direct `.trim()` calls on current `main`; I did not mutate local state to run the CLI repro because this review is read-only.

## Solution Assessment

Yes. A small hardening patch using the existing `normalizeOptionalString`-style helper is the narrowest maintainable fix and avoids changing session persistence or repair policy.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76206 on current `main`. `openclaw status` reads raw `sessions.json` entries through `readSessionStoreReadOnly`, then status/model-selection helpers call `.trim()` on `model`, `modelProvider`, `providerOverride`, and `modelOverride`; non-string persisted values should be ignored or treated as absent rather than crashing. Keep `status` read-only: do not switch to the mutable session loader and do not add startup/load-time migrations. Prefer reusing `normalizeOptionalString` or an equivalent unknown-to-trimmed-string helper in `src/agents/model-selection.ts`, `src/agents/model-selection-display.ts`, and/or the status boundary. Add regression tests for object and numeric persisted values in the status runtime/model-selection display paths.
