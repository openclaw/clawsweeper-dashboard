## Summary

Keep open: current main still lets raw read-only session model fields reach direct `.trim()` calls, and the linked fix PR #76216 is closed unmerged.

## What This Changes

The issue asks for `openclaw status` to tolerate non-string persisted session model/provider fields instead of crashing.

## Best Possible Solution

Normalize persisted session model/provider fields as unknown before any `.trim()` call, keep `status` read-only, and add corrupted `sessions.json` regression coverage.

## Reproduction Assessment

Yes. Source inspection shows a non-string `model`, `modelProvider`, `providerOverride`, or `modelOverride` parsed by `readSessionStoreReadOnly` reaches `.trim()` in the model-selection helpers; I did not mutate local session state to run the CLI crash path.

## Solution Assessment

Yes. A narrow hardening patch using the existing `normalizeOptionalString` pattern at the raw session/model-selection boundary is the maintainable fix; switching `status` to a mutable loader or migration path would be broader than needed.

Fix openclaw/openclaw#76206. On current main, `openclaw status` reads `sessions.json` through `readSessionStoreReadOnly`, so `model`, `modelProvider`, `providerOverride`, and `modelOverride` can be arbitrary JSON values and currently reach `.trim()` in model-selection/display helpers. Add narrow defensive normalization using the existing `normalizeOptionalString` pattern before any `.trim()` call on persisted session model/provider fields. Preserve `openclaw status` as a read-only diagnostic path; do not convert it to a mutable loader, migration, repair flow, dependency change, or broad session-store rewrite. Cover corrupted `sessions.json` entries plus direct helper object/number inputs where appropriate. Update `CHANGELOG.md` under the active Fixes section because this is a user-facing CLI crash fix.
