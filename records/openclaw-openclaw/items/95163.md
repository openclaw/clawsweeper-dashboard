## Summary

Keep open: this is a useful narrow mitigation, but it still needs real local TUI behavior proof and it leaves a command-boundary gap where other advertised shared commands can still fall through to the embedded model path.

## What This Changes

The PR adds `/status` and `/compact` cases to the TUI command handler so local mode prints an unsupported-command message and gateway mode forwards the raw command.

## Best Possible Solution

Make local TUI help, autocomplete, and dispatch agree so every advertised local slash command is handled, safely routed, or explicitly rejected before embedded send, then prove the local TUI path.

## Reproduction Assessment

Yes from source: slash input enters `handleCommand`, advertised shared commands without local cases reach `sendMessage(raw)`, and the embedded backend sends that text into the local agent path. I did not run the local PTY repro in this read-only review.

## Solution Assessment

No. The two-command guard is a plausible mitigation for the named commands, but the best fix is a registry-aware local command boundary or an explicit maintainer decision to keep the broader invariant tracked separately.
