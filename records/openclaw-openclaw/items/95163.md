## Summary

Keep open: current main still has the local TUI fallthrough, but this PR is not merge-ready because it only intercepts two advertised commands while other shared slash commands can still reach the local embedded model path, and contributor proof does not show a real `openclaw tui --local` run.

## What This Changes

The branch adds `/status` and `/compact` cases to the TUI command handler so local mode shows an unsupported system message while gateway mode forwards the raw command.

## Best Possible Solution

Make local TUI help, autocomplete, and dispatch agree so every advertised local slash command is handled, routed through a safe shared adapter, or explicitly rejected before embedded send, with local PTY proof.

## Reproduction Assessment

Yes. From source, slash input enters `handleCommand`, TUI discovery/help advertise shared commands, unhandled names call `sendMessage(raw)`, and the local embedded backend forwards that text to `agentCommandFromIngress`; I did not run the local PTY repro in this read-only review.

## Solution Assessment

No. The two-command guard is a plausible mitigation for the named examples, but the best fix is a registry-aware local command boundary or explicit unsupported responses for all advertised-but-unhandled local commands.
