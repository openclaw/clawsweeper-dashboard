## Summary

Keep open: the PR is a useful QQ Bot bugfix, but it is not merge-ready because the new sanitizer runs after the silent-reply classification, so metadata-prefixed `NO_REPLY` or `[SKIP]` can still be delivered as visible QQ text. The proof is also helper/test-only rather than a live QQ Bot conversation or redacted runtime log.

## What This Changes

The PR adds a QQ Bot-visible text sanitizer, wires it into the channel outbound adapter and direct gateway dispatch, and adds focused sanitizer/dispatch tests.

## Best Possible Solution

Normalize QQ Bot visible reply text once before silent-reply classification and delivery planning, then add regression coverage for metadata-prefixed silent replies and normal replies.

## Reproduction Assessment

Yes. Source inspection shows metadata-prefixed assistant output can reach QQ Bot outbound delivery, and the PR can sanitize a final block into `NO_REPLY` only after the existing silent-reply suppression check has been bypassed.

## Solution Assessment

No. The PR is a plausible owner-boundary fix, but the best fix is to sanitize before both silence classification and delivery planning so the metadata leak and silent-token behavior share one normalized visible text value.
