## Summary

Keep open: the report is credible on current main and latest stable because the PDF tool still uses a registry-only model lookup that can throw the reported `Unknown model`, while related open PRs only cover parts of the Google and PDF surfaces.

## What This Changes

The issue reports that image/PDF media-understanding tools reject configured first-party vision models with `Unknown model` even though agent runs and `models list` resolve the same model IDs.

## Best Possible Solution

Keep this issue open until maintainers converge the PDF tool on the canonical config-aware resolver, land or replace the Gemini 3.5 Flash provider PR, and add regression coverage for first-party image/PDF model refs on the reported release shape.

## Reproduction Assessment

No live high-confidence reproduction was run, but the PDF failure path is source-reproducible: current main and `v2026.6.5` still resolve PDF attempts through a registry-only helper that throws `Unknown model` on a miss. The image-tool portion needs a focused live or regression reproduction because current source uses a different config-aware resolver path.

## Solution Assessment

No complete fix is present yet. The best fix is not another ad hoc fallback; PDF execution should reuse the canonical config-aware model resolver, while the existing Google provider PR should handle Gemini 3.5 Flash model support.
