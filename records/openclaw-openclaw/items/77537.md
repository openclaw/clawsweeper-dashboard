## Summary

Keep open: the PR addresses a real source-visible TUI gap, but the current patch still lets assistant commentary render after a run has already finalized.

## What This Changes

The PR moves pending chat run tracking earlier and adds TUI handling/tests for Codex/OpenAI commentary-phase assistant events as transient progress.

## Best Possible Solution

Merge a narrow TUI fix that binds early commentary to pending runs, renders it only while the chat run is live, and keeps final answers/history commentary-free.

## Reproduction Assessment

Yes by source inspection. Current main can drop early commentary because the pending run id is assigned after `sendChat` resolves, and the PR's remaining finalized-run commentary edge follows directly from the changed event filter and chat-log behavior.

## Solution Assessment

No as proposed. The direction is narrow, but assistant commentary must be rejected for finalized runs while preserving the existing late-tool-event behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Block finalized commentary before rendering progress:** `src/tui/tui-event-handlers.ts:535`
  - body: Because `isKnownRun` still includes finalized runs before calling `renderAssistantProgressEvent`, an assistant commentary event that arrives after the final chat event can call `updateAssistant` for a run whose streaming entry was already removed. Keep finalized-run support for tool events, but do not render finalized assistant commentary as new visible output.
