## Summary

Keep open. The PR fixes a real TUI commentary-progress gap on current main, but the latest PR head still lets finalized-run commentary pass into progress rendering, which can append commentary as visible assistant output after the final answer.

## What This Changes

The PR renders Codex/OpenAI commentary-phase assistant agent events as transient TUI progress and moves pending chat run tracking before `chat.send` resolves.

## Best Possible Solution

Land the PR after commentary only renders for active, pending, or session-tracked runs while preserving late tool events after finalization.

## Reproduction Assessment

Yes. Source inspection shows current main can drop early commentary before `pendingChatRunId` is set, and PR-head source shows the finalized-run commentary leak path through `renderAssistantProgressEvent` and `ChatLog.updateAssistant`.

## Solution Assessment

No as proposed. The approach is narrow, but it needs a guard that rejects finalized-run assistant commentary before progress rendering while still allowing finalized-run tool events.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject finalized commentary before rendering progress:** `src/tui/tui-event-handlers.ts:539-540`
  - body: Because `isKnownRun` includes finalized runs before calling `renderAssistantProgressEvent`, an assistant commentary event that arrives after the final chat event can call `updateAssistant` after finalization removed the streaming run. That appends commentary as visible assistant output; keep finalized-run support for tool events but gate commentary to active, pending, or session-tracked runs.
