## Summary

Keep open: the PR addresses the main dropped-commentary path, but it still has a blocking edge where assistant commentary for a finalized run can be rendered as visible chat output after the final answer.

## What This Changes

The PR moves TUI pending chat-run tracking earlier and renders commentary-phase assistant agent events as live TUI progress with focused tests and a changelog entry.

## Best Possible Solution

Land a narrow TUI fix that binds early commentary to the live run, renders it only before finalization, keeps final answers/history commentary-free, and preserves late tool events.

## Reproduction Assessment

Yes, by source inspection rather than an executed local test. Current main can drop early commentary because pending-run tracking is assigned after `sendChat` resolves and assistant commentary has no TUI event path; the PR’s remaining regression is reproducible by sending assistant commentary for the same run after a final chat event.

## Solution Assessment

No as proposed. The direction is the right narrow fix, but assistant commentary should be rejected for finalized runs while retaining the existing finalized-run support for tool events.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Block finalized commentary before rendering progress:** `src/tui/tui-event-handlers.ts:535-539`
  - body: Because `isKnownRun` includes finalized runs before calling `renderAssistantProgressEvent`, an assistant commentary event that arrives after the final chat event can call `updateAssistant`. After `finalizeAssistant` removes the streaming run, that late update starts a new assistant component and leaks commentary as visible output; keep finalized-run support for tool events but skip finalized assistant commentary.
  - confidence: 0.9
