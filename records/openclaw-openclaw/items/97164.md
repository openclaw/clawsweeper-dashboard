## Label Justifications

- `P1`: The PR targets and currently risks a high-priority session-state bug that can split or orphan active transcript output under default daily reset.
- `merge-risk: 🚨 session-state`: The diff changes when session initialization reuses versus rotates a stale session, directly affecting transcript and session identity state.

## Summary

Keep open: this PR targets a real P1 session-state bug and has useful proof, but the new guard can also match the current turn's queued pre-dispatch registry entry, so it can suppress ordinary stale daily/idle rollover when no older active writer exists. The protected `maintainer` label also means this item should not be cleanup-closed by this workflow.

## What This Changes

The PR defers implicit stale session rollover when the reply-run registry reports the same session key and session id as active, and adds focused tests for same-session and different-session active runs.

## Best Possible Solution

Keep the active-writer protection, but make the guard exclude the current queued pre-dispatch placeholder or move the decision to a boundary that can distinguish a prior running writer from the current turn.

## Reproduction Assessment

Yes at source level: current main archives stale daily sessions during `initSessionState` before later admission can wait behind an older active run. The PR also provides after-fix Crabbox proof for the intended same-session active-run case, but the introduced regression is source-proven from dispatch ordering.

## Solution Assessment

No: the intended deferral is the right owner-boundary shape, but this implementation uses a registry signal that also exists for the current queued pre-dispatch turn. The fix should distinguish an older/running writer from the current admission placeholder or move the decision to a boundary with that distinction available.
