## Summary

Keep open, but not merge-ready. The requested ACP token-only path is not on current main, yet this PR exposes an auth-sensitive device-identity bypass without a scope-preserving contract, current docs, branch refresh, or real behavior proof.

## What This Changes

The PR adds a public ACP `--skip-device-identity` option and `AcpServerOptions.noDeviceIdentity` path that passes `deviceIdentity: null` to `GatewayClient`, with startup tests for null versus undefined behavior.

## Best Possible Solution

Define a maintainer-approved ACP token-only contract, likely aligned with configured token-scope policy or an explicit safe client shape, then document it and prove it with focused tests plus redacted runtime evidence.

## Reproduction Assessment

Yes, at source level. The PR's `deviceIdentity: null` path reaches the current Gateway missing-device policy, where checked-in tests expect shared-token operator scopes to be cleared and scoped calls to fail with `missing scope`.

## Solution Assessment

No. Passing explicit `null` is the right low-level opt-out, but this is not the best mergeable shape until the ACP scope contract, docs, branch refresh, and real runtime proof are settled.
