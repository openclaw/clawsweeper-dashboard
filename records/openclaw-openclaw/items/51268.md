## Summary

Keep open: the requested ACP token-only mode is not implemented on current main and this branch still contains unique work, but the proposed public auth flag is not merge-ready until the Gateway scope contract, docs, branch refresh, and real behavior proof are settled.

## What This Changes

The PR adds a public ACP device-identity opt-out flag that forwards `deviceIdentity: null` into `GatewayClient` and extends startup tests for null versus undefined behavior.

## Best Possible Solution

Define a maintainer-approved token-only ACP contract, likely through configured token scopes or an explicit safe client shape, then wire the flag, docs, tests, and redacted runtime proof together.

## Reproduction Assessment

Yes, at source level for the blocking review issue. The PR's explicit `deviceIdentity: null` path reaches current missing-device code whose tests expect device-less shared-token operator scopes to be cleared, while ACP calls scoped methods such as `sessions.list` and `chat.send`.

## Solution Assessment

No. Passing `null` is the right low-level opt-out, but a public ACP auth flag should not ship until the scope preservation or denial contract is centralized, documented, and proven.
