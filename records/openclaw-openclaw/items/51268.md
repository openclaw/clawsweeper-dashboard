## Summary

Keep this PR open, but it is not merge-ready: the central token-only ACP path is auth-sensitive and currently threads `deviceIdentity: null` through the CLI-shaped client without preserving Gateway scopes, so the bridge can connect with empty scopes and then fail normal ACP requests.

## What This Changes

The PR adds an ACP CLI/server flag that passes `deviceIdentity: null` to `GatewayClient` so ACP can attempt token-only gateway authentication.

## Best Possible Solution

Define one maintainer-approved ACP token-only contract, keep scope preservation explicit and tested for the intended local or remote cases, document the final flag on https://docs.openclaw.ai/cli/acp, and require redacted runtime proof before merge.

## Reproduction Assessment

Yes, from source inspection: on the PR path, `deviceIdentity: null` makes the ACP client device-less while current Gateway policy clears scopes for non-backend shared-token clients, so scoped ACP requests can fail with `missing scope`. I did not run a live ACP/Gateway repro in this read-only review.

## Solution Assessment

No. Passing explicit `null` is the right low-level GatewayClient opt-out, but this PR is not the best mergeable solution because it does not define a scope-preserving ACP auth shape, docs, or real behavior proof for the new trust mode.
