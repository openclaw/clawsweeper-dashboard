## Summary

## What This Changes

The branch adds a `--skip-device-identity` ACP CLI/server option, passes `deviceIdentity: null` to `GatewayClient`, and extends ACP startup tests for null versus undefined device identity behavior.

## Best Possible Solution

Define a maintainer-approved token-only ACP scope contract, likely coordinated with https://github.com/openclaw/openclaw/pull/81792, then rebase and ship one documented flag with regression coverage and redacted runtime proof.

## Reproduction Assessment

Yes, at source level for the blocking review issue. The PR's explicit `deviceIdentity: null` path reaches current missing-device policy that clears device-less token/password scopes, while ACP calls scoped methods such as `sessions.list` and `chat.send`.

## Solution Assessment

No. Passing `null` is the right low-level opt-out, but a public ACP auth flag should not ship until the scope preservation or denial contract is centralized, documented, rebased onto current main, and proven in a real run.
