## Summary

Keep open. The PR targets a real ACP token-only auth gap, but the current patch suppresses device identity on the existing CLI-shaped client where Gateway policy clears scopes, so token-only ACP can connect and then fail scoped calls; it also lacks after-fix runtime proof and needs alignment with the active Gateway token-scope policy work.

## What This Changes

The PR adds an ACP `--skip-device-identity` CLI/server option that passes `deviceIdentity: null` to `GatewayClient` and tests null versus undefined startup behavior.

## Best Possible Solution

Define one maintainer-approved ACP token-only contract, align it with https://github.com/openclaw/openclaw/pull/81792 or the constrained loopback shape in https://github.com/openclaw/openclaw/pull/77229, document the final flag/security caveat, and require redacted runtime proof before merge.

## Reproduction Assessment

Yes, at source level. The PR sets `deviceIdentity: null`, current Gateway policy clears scopes for device-less token/password CLI-shaped clients, and checked-in tests expect the follow-up scoped request to fail with `missing scope`.

## Solution Assessment

No. Passing explicit `null` is the right low-level opt-out, but the mergeable solution needs an approved scope contract, docs, and real runtime proof rather than a broad flag on the existing CLI-shaped client.
