## Summary

Keep this PR open, but not merge-ready. Current main still lacks the requested ACP device-identity opt-out, while this branch exposes an auth-sensitive token-only mode without a settled Gateway scope contract, docs for the final flag spelling, or real runtime proof; it is also currently conflicting with newer ACP startup changes.

## What This Changes

The branch adds a `--skip-device-identity` ACP CLI flag, passes `deviceIdentity: null` into `GatewayClient`, and extends ACP startup tests for null versus undefined behavior.

## Best Possible Solution

Define a maintainer-approved token-only ACP scope contract in Gateway auth, then wire one final documented flag with regression coverage and redacted runtime proof that token-only ACP initializes and completes scoped requests safely.

## Reproduction Assessment

Yes, at source level for the blocking review issue. The PR's explicit `deviceIdentity: null` path reaches current missing-device policy that clears device-less token/password scopes, while ACP calls scoped methods such as `sessions.list` and `chat.send`.

## Solution Assessment

No. Passing `null` is the right low-level opt-out, but a public ACP auth flag should not ship until the scope preservation or denial contract is centralized, documented, refreshed onto current main, and proven in a real run.
