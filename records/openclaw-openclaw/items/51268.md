## Summary

This PR should stay open, but it is not merge-ready: current main still lacks the ACP device-identity opt-out, while the proposed token-only path changes an auth boundary and appears to connect with scopes cleared under current Gateway policy.

## What This Changes

The branch adds an ACP `--skip-device-identity` CLI/server option that passes `deviceIdentity: null` to `GatewayClient` and adds startup tests for null versus undefined device identity.

## Best Possible Solution

Define a maintainer-approved ACP token-only contract, preserve or intentionally deny scopes through an explicit safe client/config shape, document the final flag and caveat, and require redacted runtime proof before merge.

## Reproduction Assessment

Yes, at source level: the PR’s `deviceIdentity: null` path reaches current Gateway missing-device policy, where checked-in tests expect shared-token operator scopes to be cleared and scoped calls to fail with `missing scope`. I did not run a live ACP/Gateway repro in this read-only review.

## Solution Assessment

No. Passing explicit `null` is the right low-level opt-out, but this is not the best mergeable shape until the ACP scope contract, docs, and real runtime proof are settled.
