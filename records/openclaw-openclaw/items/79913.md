## Summary

Keep open: this is a focused and useful candidate fix for the linked Docker/Coolify spawn allowlist regression, but it is not merge-ready because the env overlay currently replaces an explicit default allowlist and the new authorization env surface is undocumented.

## What This Changes

The PR adds config-load parsing for `OPENCLAW_SPAWN_ALLOWLIST`/`SPAWN_ALLOWLIST` and writes parsed values into `agents.defaults.subagents.allowAgents`, with focused parser tests.

## Best Possible Solution

Keep the focused config-load parser, apply env values only when no explicit default `allowAgents` is configured, document the fallback precedence, and preserve configured-agent wildcard limits.

## Reproduction Assessment

Yes, source-reproducible: current main and `v2026.6.8` have no `SPAWN_ALLOWLIST` reader, while native and ACP spawn gates receive allowlists only from config-backed `allowAgents`. I did not run the Docker Compose scenario in this read-only review.

## Solution Assessment

No as submitted. Config-load materialization is a good layer for env-only Docker deployments, but env should be a missing-default fallback unless maintainers explicitly choose and document authoritative env precedence.
