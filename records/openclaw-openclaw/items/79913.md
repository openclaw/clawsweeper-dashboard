## Summary

This PR addresses a real Docker/Coolify spawn allowlist gap, but the proposed config-load overlay is not ready because it can broaden an explicit `agents.defaults.subagents.allowAgents` value from the environment.

## What This Changes

The branch adds a config-load helper that parses `OPENCLAW_SPAWN_ALLOWLIST` or `SPAWN_ALLOWLIST`, writes it to `agents.defaults.subagents.allowAgents`, wires it from `src/config/io.ts`, and adds parser/unit coverage.

## Best Possible Solution

Implement a documented env allowlist fallback that is used only when per-agent and default config allowlists are unset, and keep `agents_list` and `sessions_spawn` policy consistent.

## Reproduction Assessment

Yes. Source inspection shows current main enforces `sessions_spawn` from config allowlists and does not read `SPAWN_ALLOWLIST`, matching the linked Docker report's `allowed: none` failure path.

## Solution Assessment

No. The env parser is useful, but writing the env value over explicit config is not the safest authorization fix; fallback-only precedence is the narrower maintainable path unless maintainers explicitly approve env override semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve explicit allowAgents config:** `src/config/spawn-allowlist-env.ts:51`
  - body: `applySpawnAllowlistEnvOverlay` always replaces `agents.defaults.subagents.allowAgents` when `SPAWN_ALLOWLIST` is set. In a deployment that already configured a narrower on-disk allowlist, setting `SPAWN_ALLOWLIST=*` in the environment would broaden which agents `sessions_spawn` can target, so this should only fill the allowlist when config did not provide one or be backed by an explicit maintainer-approved precedence decision.
  - confidence: 0.86
