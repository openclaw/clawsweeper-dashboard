## Summary

Keep open: the PR addresses a real missing env path for Docker/Coolify deployments, but it should not merge as-is because it lets an env var replace an explicit default spawn allowlist. The related open implementation for the same issue uses fallback-only precedence, which is safer for this authorization surface.

## What This Changes

The PR adds a config-load helper that parses `OPENCLAW_SPAWN_ALLOWLIST` or `SPAWN_ALLOWLIST`, writes it to `agents.defaults.subagents.allowAgents`, and adds parser/unit coverage.

## Best Possible Solution

Support the env mapping only through a documented, tested contract that preserves explicit per-agent/default config precedence unless maintainers deliberately approve env override semantics.

## Reproduction Assessment

Yes. Source inspection of current main shows `sessions_spawn` only reads config-backed allowlists and has no `SPAWN_ALLOWLIST` reader, matching the linked Docker report's `allowed: none` failure.

## Solution Assessment

No. The parser and config hook are plausible, but env-overrides-config broadens a spawn authorization control; fallback-only precedence is the safer maintainable path unless maintainers approve override policy.
