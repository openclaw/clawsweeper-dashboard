## Summary

Keep open for maintainer review: the PR targets a source-reproducible gap, but it adds a new env-driven spawn authorization surface and currently lets that env value override explicit config.

## What This Changes

The PR maps `OPENCLAW_SPAWN_ALLOWLIST`/`SPAWN_ALLOWLIST` into `agents.defaults.subagents.allowAgents` during config load and adds parser/unit coverage.

## Best Possible Solution

Implement a maintainer-approved, documented env mapping for spawn allowlists, preferably preserving explicit config unless maintainers intentionally choose env override precedence.

## Reproduction Assessment

Yes. Source inspection shows current main has config-backed allowlist enforcement for `sessions_spawn` and no `SPAWN_ALLOWLIST` reader, matching the linked Docker report's `allowed: none` symptom.

## Solution Assessment

No. The narrow overlay is plausible, but letting an env var override explicit `allowAgents` is a security-sensitive policy choice; fallback-only behavior is the safer default unless maintainers approve env precedence.
