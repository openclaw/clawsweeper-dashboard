## Summary

Keep this PR open, but it should not merge as-is because the env-derived spawn allowlist can replace an explicit configured default allowlist on a security-sensitive `sessions_spawn` authorization path.

## What This Changes

Adds a config-load helper that parses `OPENCLAW_SPAWN_ALLOWLIST`/`SPAWN_ALLOWLIST` into `agents.defaults.subagents.allowAgents`, with unit tests and a config-load hook.

## Best Possible Solution

Land one canonical env allowlist implementation that preserves explicit config precedence unless maintainers deliberately document and approve env-wins authorization semantics.

## Reproduction Assessment

Yes, at source level. Current main has no `SPAWN_ALLOWLIST` reader, and direct spawn, ACP spawn, and `agents_list` only consult config-backed `subagents.allowAgents`, matching the linked Docker report's `allowed: none` path.

## Solution Assessment

No. The parser and load hook are plausible, but the proposed env overlay overwrites explicit default config on an authorization surface; fallback-only behavior or a documented maintainer-approved env-wins policy is safer.
