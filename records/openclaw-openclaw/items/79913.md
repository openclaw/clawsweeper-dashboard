## Summary

Keep open: the PR targets a real current gap, but it should not merge as-is because the env overlay can broaden `sessions_spawn` authorization by overriding explicit config, and the remaining precedence choice needs maintainer review.

## What This Changes

The PR adds config-load parsing for `OPENCLAW_SPAWN_ALLOWLIST`/`SPAWN_ALLOWLIST`, writes the result to `agents.defaults.subagents.allowAgents`, and adds focused parser/overlay unit tests.

## Best Possible Solution

Add a documented maintainer-approved env mapping for Docker deployments that preserves explicit config unless override precedence is intentionally accepted, then cover both config-precedence and spawn behavior with regression tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes by source inspection: current main has config-backed `sessions_spawn` allowlist enforcement and no `SPAWN_ALLOWLIST` reader, matching the linked Docker report's `allowed: none` symptom.

## Solution Assessment

Is this the best way to solve the issue? No. The parser and config-load hook are plausible, but env-wins-over-config is a security-sensitive precedence choice; fallback-only behavior is the safer default unless maintainers explicitly approve override semantics.
