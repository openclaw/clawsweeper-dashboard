## Summary

Keep open: the linked Docker/Coolify spawn-allowlist bug is still not implemented on current main or the latest release, but this PR is not merge-ready because the env overlay can replace an explicitly configured default allowlist and the new env authorization input has no public precedence contract.

## What This Changes

The PR adds a config-load parser for `OPENCLAW_SPAWN_ALLOWLIST` and `SPAWN_ALLOWLIST`, then writes the parsed values into `agents.defaults.subagents.allowAgents` with focused parser tests.

## Best Possible Solution

Keep the focused config-load parser, apply env values only when no explicit default `allowAgents` is configured, document the fallback precedence, and preserve configured-agent wildcard limits.

## Reproduction Assessment

Yes, source-reproducible: current main and `v2026.6.8` have no `SPAWN_ALLOWLIST` reader, while native and ACP spawn gates receive allowlists only from config-backed `allowAgents`. I did not run the Docker Compose scenario in this read-only review.

## Solution Assessment

No as submitted. Config-load materialization is a good layer for env-only Docker deployments, but env should be a missing-default fallback rather than an override of explicit allowlist policy.
