## Summary

Keep open: the linked Docker/Coolify problem is source-reproducible on current main, but this PR still changes spawn authorization semantics by letting an env allowlist overwrite explicit default config and overlaps with another open implementation path.

## What This Changes

The PR adds a config-load helper that parses `OPENCLAW_SPAWN_ALLOWLIST`/`SPAWN_ALLOWLIST` into `agents.defaults.subagents.allowAgents` and tests the parser/overlay behavior.

## Best Possible Solution

Land one canonical implementation that supports Docker env allowlists as a documented fallback without broadening explicit config unless maintainers intentionally approve env-wins authorization semantics.

## Reproduction Assessment

Yes, at source level. Current main has no `SPAWN_ALLOWLIST` reader, while `sessions_spawn` and `agents_list` only consult config-backed `subagents.allowAgents`, matching the linked `allowed: none` path.

## Solution Assessment

No. The parser and load hook are plausible, but overwriting an explicit default spawn allowlist can broaden authorization; fallback-only precedence or an explicit maintainer-approved env-wins policy is safer.
