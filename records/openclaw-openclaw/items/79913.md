## Summary

Keep open: current main still does not read `SPAWN_ALLOWLIST`, so the PR addresses a real gap, but it should not merge as-is because the env overlay can replace an explicit default `allowAgents` value on a spawn authorization surface.

## What This Changes

The PR parses `OPENCLAW_SPAWN_ALLOWLIST`/`SPAWN_ALLOWLIST` at config load, writes it to `agents.defaults.subagents.allowAgents`, and adds focused parser/overlay tests.

## Best Possible Solution

Support the env mapping through a documented fallback-only contract that preserves explicit per-agent/default config precedence, unless maintainers deliberately approve env-wins authorization semantics.

## Reproduction Assessment

Yes, at source level. Current main has no `SPAWN_ALLOWLIST` reader, and the spawn/list authorization paths only consume config-backed `subagents.allowAgents`, matching the reported env-only `allowed: none` path if env support is accepted as intended behavior.

## Solution Assessment

No. The parser and load hook are plausible, but overwriting explicit default config is not the safest authorization contract; fallback-only precedence plus docs is the narrower maintainable path unless maintainers approve env-wins behavior.
