## Summary

Keep open: the PR targets a real current-main gap, but as written it introduces a security-sensitive workspace .env path for an unprefixed spawn allowlist and still lacks real after-fix Docker or gateway proof.

## What This Changes

The branch maps OPENCLAW_SPAWN_ALLOWLIST or SPAWN_ALLOWLIST to agents.defaults.subagents.allowAgents during config load and adds parser/overlay unit tests.

## Best Possible Solution

Resolve this through a maintainer-approved spawn allowlist env contract that cannot be sourced from workspace .env, with focused docs/tests and redacted Docker or gateway proof for the reported sessions_spawn flow.

## Reproduction Assessment

Yes for the source-level current-main gap: current code passes only config-backed subagents.allowAgents into the sessions_spawn target policy and no SPAWN_ALLOWLIST reader exists. I did not run live Docker proof in this read-only review.

## Solution Assessment

No. The direction may be useful, but this implementation should not honor an unblocked unprefixed env var from the same process env that is hydrated from workspace .env; use a trusted prefixed env contract or block the alias before reading it, then prove the Docker sessions_spawn path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Block SPAWN_ALLOWLIST from workspace dotenv:** `src/config/spawn-allowlist-env.ts:12`
  - body: `loadDotEnv` imports the current working directory `.env` before config load, and the workspace dotenv guard only fail-closes the `OPENCLAW_*` runtime namespace. By honoring the unprefixed `SPAWN_ALLOWLIST` here, a project-local `.env` can set `SPAWN_ALLOWLIST=*` and silently widen `sessions_spawn` target authorization for the gateway. Either use the prefixed env var only or block this alias from workspace dotenv before reading it.
  - confidence: 0.9
