## Summary

Keep open: the PR targets a real source-level gap and includes sufficient Docker/Compose proof, but it should not merge as-is because the env overlay can replace an explicit spawn allowlist and broaden `sessions_spawn` authorization.

## What This Changes

The branch adds `OPENCLAW_SPAWN_ALLOWLIST`/`SPAWN_ALLOWLIST` parsing during config load, writes the result to `agents.defaults.subagents.allowAgents`, and adds parser/overlay unit tests.

## Best Possible Solution

Land a maintainer-approved, documented env mapping only after preserving explicit config precedence or explicitly approving env-overrides-config behavior, with regression coverage for both config precedence and `sessions_spawn` authorization.

## Reproduction Assessment

Yes. Source inspection shows current main has config-backed `sessions_spawn` allowlist enforcement and no `SPAWN_ALLOWLIST` reader, matching the linked Docker report's `allowed: none` symptom; I did not run a live Docker reproduction in this read-only review.

## Solution Assessment

No. The narrow env mapping is plausible, but making env always win over explicit `allowAgents` is not the safest authorization contract; fallback-only behavior or explicit maintainer-approved precedence is safer.
