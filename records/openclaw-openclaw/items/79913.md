## Summary

Keep open: the linked Docker symptom is source-reproducible, but this PR is not merge-ready because the env overlay can broaden a security-sensitive spawn allowlist by replacing explicit default config.

## What This Changes

The PR adds `OPENCLAW_SPAWN_ALLOWLIST`/`SPAWN_ALLOWLIST` parsing during config load and writes the parsed value into `agents.defaults.subagents.allowAgents` with focused parser tests.

## Best Possible Solution

Support the env allowlist only as a documented fallback that preserves explicit per-agent/default config precedence, unless maintainers deliberately approve env-overrides-config semantics.

## Reproduction Assessment

Yes, by source inspection: current main has no `SPAWN_ALLOWLIST` reader and the `sessions_spawn` authorization path only consults config-backed allowlists, matching the linked Docker report's `allowed: none` failure.

## Solution Assessment

No. The parser is plausible, but applying env at config load so it overwrites explicit default config is not the safest maintainable fix for a spawn authorization control.
