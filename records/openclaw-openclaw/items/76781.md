## Summary

Keep open for review/merge: current main has the partial-registry reuse bug in source, and the PR fixes the right boundary by requiring tool coverage for every selected plugin id. The only actionable gap I found is the required changelog entry for this user-facing fix.

## What This Changes

The PR tightens plugin tool registry reuse and adds a regression test for wildcard-selected plugin tools when an existing registry only exposes a subset of selected plugin ids.

## Best Possible Solution

Merge the narrow registry-coverage fix after adding the required changelog entry and letting the normal plugin test/check gates validate the exact PR head.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: on current main, a selected multi-plugin tool request can reuse a registry once any selected plugin has tool entries, leaving another selected plugin's tools absent from `tools.effective` and invocation surfaces.

## Solution Assessment

Yes. Requiring every scoped plugin id to be represented in `registry.tools` before reuse is the narrowest maintainable fix and preserves the existing unscoped and empty-scope behavior.
