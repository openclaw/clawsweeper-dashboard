## Summary

Keep open. Current main still sends proactive WhatsApp phone-number targets through plain PN JID normalization, while the exact outbound PN-to-LID fix is an open, unmerged closing PR rather than shipped behavior.

## What This Changes

Resolve proactive WhatsApp outbound phone-number recipients to LID JIDs using Baileys auth-dir forward mappings so sender-only ghost chats are not created.

## Best Possible Solution

Review and land #74925, or an equivalent WhatsApp-plugin-only fix, so createWebSendApi resolves phone-number recipients through the active account authDir/Baileys forward LID mapping while preserving qualified JIDs and the existing PN fallback.

## Reproduction Assessment

Yes. The issue provides concrete live WhatsApp steps, and current main has a source-level reproduction: calling createWebSendApi with a phone-number target still sends to digits@s.whatsapp.net even when a forward LID mapping file would exist.

## Solution Assessment

Yes. Resolving at the active Web send boundary with the listener account authDir is the narrowest maintainable fix because text/media sends, polls, and composing presence flow through createWebSendApi; this issue should stay open until #74925 or an equivalent fix merges.
