## Summary

Keep this issue open. Current main still routes proactive WhatsApp phone-number sends through plain PN JID normalization, and the linked fix is an open, unmerged PR rather than shipped behavior. The existing LID support remains inbound/reverse-oriented, while #74925 is the active implementation candidate for the central outbound delivery bug.

## What This Changes

The issue requests outbound WhatsApp phone-number sends to resolve stored Baileys LID mappings before falling back to PN JIDs, preventing sender-only duplicate ghost chats.

## Best Possible Solution

Review and land #74925, or an equivalent WhatsApp-plugin-only fix, so the active Web send boundary resolves plain phone-number recipients against the active account authDir/Baileys LID forward mapping, preserves already-qualified JIDs, and keeps unmapped numbers on the current PN fallback. Track the re-link reverse-mapping persistence concern separately if it remains after the delivery fix.

## Reproduction Assessment

Yes. The issue provides concrete WhatsApp steps, and current main gives a high-confidence source-level reproduction: even with Baileys forward mapping available, createWebSendApi still sends +1555-style targets as 1555@s.whatsapp.net.

## Solution Assessment

Yes. Resolving at createWebSendApi with the listener authDir is the narrowest maintainable fix for the central delivery bug because text/media, polls, and composing presence all flow through that active send API; #74925 appears to implement that direction but is not merged yet.
