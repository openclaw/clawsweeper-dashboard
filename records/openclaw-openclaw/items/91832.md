## Summary

Keep open: current main has status and Control UI context-usage surfaces, but I did not find a native gateway health/diagnostic alert path that evaluates fresh saturated terminal `done` sessions. The reported fleet impact is serious, but the fix needs a maintainer decision about whether this should be a health payload field, diagnostics event, Control UI warning, automatic cleanup, or another operator alert path.

## What This Changes

The issue requests native monitoring or alerting for `done` sessions whose fresh context usage exceeds a saturation threshold.

## Best Possible Solution

Define one native core diagnostic/status path for fresh terminal-session context saturation, then add focused coverage proving `done` and `running` saturated sessions are both visible without automatically deleting or resetting sessions unless maintainers choose that policy.

## Reproduction Assessment

No, not for the full reported fleet failure. Source inspection does show that current native health/liveness diagnostics do not define a terminal `done` session context-saturation alert path, but the gateway-restart state flip and silent delivery failure still need logs or a focused scenario.

## Solution Assessment

Unclear. Treating `done` exactly like `running` may be the right alert policy, but maintainers should first decide whether the durable contract belongs in health JSON, diagnostics telemetry, status output, Control UI, or cleanup automation.
