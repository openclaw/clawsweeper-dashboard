## Summary

Keep open: current `origin/main` still routes `executeWithApiKeyRotation` through a rate-limit-only retry predicate by default, media and Google embedding call sites do not opt into transient retries, and #75219 is the unmerged implementation candidate for this issue.

## What This Changes

Add retry handling for transient 5xx, timeout, and network failures in provider API-key execution instead of retrying only 429/rate-limit errors.

## Best Possible Solution

Review and land or replace #75219 with bounded opt-in same-key transient retries for idempotent provider operations while keeping 429 alternate-key rotation separate.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `executeWithApiKeyRotation` with one key and an `execute` callback that throws `Error("HTTP 500")` or an `AbortError`; current `origin/main` makes one attempt and rethrows because the default predicate is rate-limit-only.

## Solution Assessment

Yes. The best direction is opt-in same-key transient retry for safe provider operations, not globally treating 5xx failures as alternate-key rotation.
