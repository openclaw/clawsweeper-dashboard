## Summary

Keep open: current main still defaults `executeWithApiKeyRotation` to rate-limit-only key rotation, media and Google call sites do not opt into transient retries, and open PR #75219 is the unmerged implementation candidate.

## What This Changes

Add bounded transient retry/backoff for provider operations using `executeWithApiKeyRotation`, covering 5xx, timeout, and network failures instead of only rate-limit key rotation.

## Best Possible Solution

Review and land or replace #75219 with bounded opt-in same-key transient retries for idempotent provider operations while keeping 429 alternate-key rotation separate.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `executeWithApiKeyRotation` with one key and an `execute` callback that throws `Error("HTTP 500")` or an `AbortError`; current main makes one attempt and rethrows because the default predicate is rate-limit-only.

## Solution Assessment

Yes, the best direction is a narrow opt-in same-key transient retry policy for safe provider operations, not global retries or rotating API keys on 5xx failures.
