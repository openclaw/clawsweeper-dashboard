## Summary

Keep open: current main still retries `executeWithApiKeyRotation` only for API-key rate-limit errors by default, the media and Google embedding call sites do not opt into 5xx/timeout/network retries, and open PR #75219 remains the implementation candidate.

## What This Changes

The issue asks for provider execution to retry transient 5xx, timeout, and network failures instead of only rotating keys on 429/rate-limit errors.

## Best Possible Solution

Review and land or replace #75219 with bounded opt-in same-key transient retries for idempotent provider operations while preserving separate 429 alternate-key rotation.

## Reproduction Assessment

Yes. A source-level reproduction is to call `executeWithApiKeyRotation` with one key and an `execute` callback that throws `Error("HTTP 500")` or an `AbortError`; current main makes one attempt and rethrows because the default predicate is rate-limit-only.

## Solution Assessment

Yes. The narrow maintainable direction is opt-in same-key transient retry for safe provider operations, not global retries or rotating API keys on 5xx failures.
