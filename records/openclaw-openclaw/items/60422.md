## Summary

Keep open. Current main still has a rate-limit-only API-key rotation path, so single-key 5xx, timeout, and network failures fail fast. The linked #75219 is an open implementation candidate for this exact gap, but it has not merged, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests bounded transient retry/backoff for `executeWithApiKeyRotation` on provider 5xx, timeout, and network failures instead of retrying only 429/rate-limit cases.

## Best Possible Solution

Ship a bounded, opt-in same-key transient retry policy for idempotent provider operations, keep 429/rate-limit alternate-key rotation unchanged, add focused single-key and multi-key regression tests for 5xx, timeout, network, 4xx/auth, and caller-abort cases, then close this issue once the fix is on main.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `executeWithApiKeyRotation` with one key and an `execute` callback that throws `Error("Audio transcription failed (HTTP 500)")` or an `AbortError`; current main makes one attempt and rethrows because the default retry predicate is rate-limit-only.

## Solution Assessment

Yes, the narrow maintainable direction is an explicit transient retry option for safe provider operations, not a global retry of every API-key-wrapped call. The open PR appears to target that direction and should be reviewed as the implementation path.
