## Summary

Keep open: current main and v2026.5.18 still have the cron-specific duration parser path that can return `Infinity`, and an open linked PR already targets the parser and regression-test fix.

## What This Changes

The issue reports that the cron CLI duration parser can return `Infinity` for very large duration strings after unit multiplication, allowing non-finite cron schedule values to pass caller checks.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84047 if its implementation and proof are acceptable, or close it and apply the same finite-result guard or central-parser reuse with focused regression coverage.

## Reproduction Assessment

Yes, at source level. A long pure-digit days value matches the parser, parses finitely, overflows during unit multiplication, and remains truthy for callers; the literal `1e308d` example is rejected earlier by the regex.

## Solution Assessment

Yes. The narrow maintainable fix is to reject non-finite computed milliseconds in the cron-specific parser, or reuse the central parser that already enforces that contract, with focused regression coverage.
