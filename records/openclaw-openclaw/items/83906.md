## Summary

Keep open: current main and the latest release still return `Math.floor(n * factor)` without checking the multiplied value, and the only linked fix PR is closed unmerged with no replacement open. The report's exact exponent example is rejected by the regex, but the same bug is reachable with a long pure-digit duration.

## What This Changes

The issue asks OpenClaw to reject non-finite computed millisecond durations from the cron CLI parser so huge numeric inputs cannot produce `Infinity`.

## Best Possible Solution

Land a narrow cron parser fix that rejects non-finite computed milliseconds and covers the long-digit overflow path while preserving existing invalid-duration errors.

## Reproduction Assessment

Yes, at source level. Current main accepts a long pure-digit days value in the cron parser regex, parses it to a finite number, overflows during unit multiplication to `Infinity`, and the observed cron callers only reject falsy/null results.

## Solution Assessment

Yes. The narrow maintainable fix is to reject non-finite computed milliseconds after applying the unit multiplier, or to reuse the central parser that already enforces that contract, with focused regression coverage.
