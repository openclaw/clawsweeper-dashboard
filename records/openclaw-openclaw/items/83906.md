## Summary

Keep open: current main and the latest stable release still have the cron parser path that can return `Infinity`, and https://github.com/openclaw/openclaw/pull/84047 is an open linked fix candidate with closing syntax rather than a reason to close the issue early.

## What This Changes

Reject non-finite computed durations from the cron CLI duration parser so overflow inputs cannot reach cron schedule payloads as `Infinity`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84047 if its implementation and proof are acceptable, or apply the same finite-result parser guard with focused regression coverage in a replacement fix.

## Reproduction Assessment

Yes. A source-level replay of the current parser shows a long pure-digit days value matches the regex, parses finitely, overflows after unit multiplication, and remains truthy for the `--every` caller; the reporter's literal `1e308d` example is rejected earlier by the regex.

## Solution Assessment

Yes. The narrow maintainable fix is to reject non-finite computed milliseconds in the cron-specific parser, or reuse the central parser that already enforces that contract, with focused regression coverage.
