## Summary

Keep open: current main still has a cron-specific duration parser that validates the parsed number before multiplying by the unit, so oversized inputs can become `Infinity` before `--every` validation.

## What This Changes

The issue asks cron duration parsing to reject overflowed duration values such as `1e308d` instead of accepting them as `Infinity`.

## Best Possible Solution

Make the cron CLI duration parser reject non-finite computed milliseconds, preferably by reusing the central parser or adding the same finite-result guard, with focused regression coverage for overflowed cron duration inputs.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: `1e308d` matches the cron duration regex, `1e308 * 86_400_000` overflows to `Infinity`, and `--every` accepts that truthy result. I did not run a full packaged CLI/gateway repro in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to reject non-finite computed milliseconds in the cron CLI parser or replace it with the central duration parser; no gateway schema or product decision is needed.
