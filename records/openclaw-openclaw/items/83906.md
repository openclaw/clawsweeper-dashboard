## Summary

Keep open: current main and the latest release still have the cron-specific duration parser path that can return `Infinity`, and the linked fix PR was closed unmerged with no open replacement found.

## What This Changes

Reject non-finite computed cron durations so huge `--every` or failure-alert cooldown values cannot become `Infinity` schedule payloads.

## Best Possible Solution

Add a finite-result guard after cron duration unit multiplication, or reuse the central parser, with regression coverage for the long pure-digit overflow case.

## Reproduction Assessment

Yes, at source level. Current main accepts a long pure-digit days input, parses the number finitely, overflows during unit multiplication, and leaves a truthy `Infinity` for cron callers.

## Solution Assessment

Yes. The narrow maintainable fix is to reject non-finite computed milliseconds in `src/cli/cron-cli/shared.ts`, ideally with a focused regression test; no product or config decision is needed.

## Repair Work Prompt

Fix the cron CLI duration overflow bug tracked by https://github.com/openclaw/openclaw/issues/83906. In `src/cli/cron-cli/shared.ts`, ensure `parseDurationMs` returns `null` when `Math.floor(n * factor)` is not finite, while preserving existing valid duration behavior and caller error messages. Add focused Vitest coverage in `src/cli/cron-cli/shared.test.ts` for ordinary durations, malformed/zero inputs, a long pure-digit days input that overflows only after multiplication, and an input whose parsed number is already non-finite. Do not add new config, change cron schedule semantics, or refactor unrelated CLI code. Include a concise `CHANGELOG.md` fix entry if this repair PR is intended for landing, without bot thanks.
