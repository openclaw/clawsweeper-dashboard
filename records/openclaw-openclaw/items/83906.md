## Summary

Keep open: current main and the latest stable release still have the cron-specific parser path that can return `Infinity`, and an open linked PR is already the right fix candidate rather than a reason to close the issue early.

## What This Changes

The issue asks for cron duration parsing to reject non-finite millisecond results after unit multiplication so oversized durations cannot reach cron schedule payloads as `Infinity`.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84047 if its implementation and proof are acceptable, or close that PR and repair the same parser guard in one focused follow-up.

## Reproduction Assessment

Yes. Source inspection and a small arithmetic replay show a long pure-digit days input matches the parser, parses finitely, overflows during unit multiplication, and remains truthy for the `--every` caller; the reporter's literal `1e308d` example is rejected by the regex before that path.

## Solution Assessment

Yes. The narrow maintainable fix is to reject non-finite computed milliseconds in the cron-specific parser, or reuse the central parser that already enforces that contract, with focused regression coverage.
