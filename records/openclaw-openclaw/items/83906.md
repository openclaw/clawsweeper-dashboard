## Summary

Keep open: current main and v2026.5.20 still contain the cron-specific parser path that can return a non-finite duration, and the linked fix PR was closed unmerged.

## What This Changes

The issue requests rejecting overflowed cron CLI duration results so `parseDurationMs` cannot return `Infinity` for `--every` or failure-alert cooldown inputs.

## Best Possible Solution

Land one focused fix that rejects non-finite computed milliseconds in the cron-specific parser, with regression coverage for the long-digit overflow case and existing valid/invalid durations.

## Reproduction Assessment

Yes, at source level. A long pure-digit days input reaches the cron parser, parses finitely, overflows after unit multiplication, and returns a truthy `Infinity`; the literal `1e308d` example is blocked by the regex before that path.

## Solution Assessment

Yes. The narrow maintainable fix is to reject a non-finite result after `Math.floor(n * factor)`, or to reuse the central parser while preserving the cron CLI's null-on-invalid caller contract and friendly errors.

Fix https://github.com/openclaw/openclaw/issues/83906 in one focused PR. Current main's `src/cli/cron-cli/shared.ts` cron-specific `parseDurationMs` can return `Infinity` when a long pure-digit value parses finitely but overflows after multiplying by the `d` unit factor. Add a finite-result guard after computing the floored milliseconds, or reuse the central parser only if the cron CLI null-on-invalid contract and friendly caller errors stay intact. Add focused regression coverage in `src/cli/cron-cli/shared.test.ts` for ordinary valid durations, zero/malformed inputs, the long pure-digit overflow case returning `null`, and a parseFloat-Infinity input.
