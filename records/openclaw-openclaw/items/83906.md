## Summary

Keep open: current main and v2026.5.18 still have the cron-specific parser path that can return `Infinity`, and an open linked PR already uses closing syntax for the focused parser/test fix.

## What This Changes

The issue asks OpenClaw to reject cron duration inputs whose unit-scaled millisecond value overflows to `Infinity` after parsing.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84047 if its implementation and proof are acceptable, or close that PR and make the same finite-result guard with focused regression coverage.

## Reproduction Assessment

Yes, at source level. A long pure-digit days input matches the current parser, parses finitely, overflows during unit multiplication, and stays truthy for the `--every` and cooldown callers; the literal `1e308d` example is rejected earlier by the regex.

## Solution Assessment

Yes. The narrow maintainable fix is to reject non-finite computed milliseconds in the cron-specific parser, or reuse the central parser that already enforces that contract, with regression coverage.
