## Summary

Keep open: current main still lacks the finite-result guard in the cron duration parser, and an open closing PR already tracks the focused fix, so the issue should stay open until that implementation lands or is closed.

## What This Changes

The issue asks cron CLI duration parsing to reject non-finite millisecond results after unit multiplication so oversized inputs cannot produce `Infinity` in cron schedule payloads.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84047, or an equivalent finite-result guard with regression tests, then close this issue from the merged fix.

## Reproduction Assessment

Yes at source level. A long pure-digit days input matches the cron duration regex, parses to a finite number, overflows to `Infinity` after unit multiplication, and the `--every` path only rejects falsy parser results.

## Solution Assessment

Yes. The narrow maintainable fix is to reject non-finite computed milliseconds in the cron-specific parser or reuse the central duration parser's finite-result guard, with focused regression coverage.
