## Summary

Keep open: current main and the latest release still have the cron-local duration parser returning the floored multiplication result without a finite-result guard, and the linked fix PR was closed unmerged.

## What This Changes

The issue reports that cron CLI `parseDurationMs` can return `Infinity` for very large duration inputs after unit multiplication.

## Best Possible Solution

Reject non-finite computed milliseconds in the cron-local parser, or reuse the central duration parser while preserving cron’s zero-duration rejection and existing friendly errors, with focused regression tests.

## Reproduction Assessment

Yes, at source level. A long pure-digit days input matches the current parser, parses finitely, overflows during unit multiplication to `Infinity`, and the affected callers do not reject that truthy value.

## Solution Assessment

Yes. The narrow maintainable fix is a finite-result guard after multiplication, with regression coverage; central parser reuse is also acceptable only if cron’s existing zero-duration rejection and error behavior stay intact.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/83906. In current main, `src/cli/cron-cli/shared.ts` `parseDurationMs` returns `Math.floor(n * factor)` without checking whether the computed milliseconds are finite, so a long pure-digit days input can produce `Infinity` and pass `--every` / `--failure-alert-cooldown` callers. Add a finite-result check after multiplication, or reuse `src/cli/parse-duration.ts` only if cron still rejects zero-duration `--every` values and preserves existing friendly errors. Add focused regression tests in `src/cli/cron-cli/shared.test.ts` for ordinary positive durations, zero/malformed input, a long pure-digit overflow-days input returning `null`, and a parseFloat-to-Infinity input returning `null`. Keep scope to the cron CLI parser/callers; do not change gateway schema, release workflows, or unrelated duration semantics. Include a `CHANGELOG.md` fix entry if the repair lane policy requires one for this user-visible CLI validation fix.
