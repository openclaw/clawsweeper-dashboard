## Summary

Keep open: current `main` and latest release still return `Math.floor(n * factor)` from the cron-specific duration parser without rejecting non-finite results, and the only linked fix PR is now closed unmerged.

## What This Changes

The issue asks OpenClaw to reject cron CLI durations whose millisecond value overflows to `Infinity` after unit multiplication.

## Best Possible Solution

Add a finite-result check in the cron-specific parser or switch the cron CLI to the central duration parser, with focused regression coverage for the long pure-digit overflow case.

## Reproduction Assessment

Yes, at source level. The current regex and arithmetic path reproduces a finite parse followed by an `Infinity` result for a long pure-digit days input; the reporter’s literal `1e308d` example is blocked earlier by the regex.

## Solution Assessment

Yes. The narrow maintainable fix is to reject non-finite computed milliseconds before returning from the cron parser, preferably with a regression test that uses the reachable long-digit overflow case.
