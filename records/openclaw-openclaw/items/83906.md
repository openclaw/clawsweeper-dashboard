## Summary

Keep open: current main still lets the cron-specific duration parser compute `Infinity` after multiplying an oversized finite number by the unit factor, and the open fixing PR should be reviewed or landed before this issue closes. The literal `1e308d` example is blocked by the digits-only regex, but the same source path is reachable with a long pure-digit duration.

## What This Changes

The issue asks OpenClaw to reject non-finite computed millisecond values from the cron CLI `parseDurationMs` helper.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/84047, or an equivalent finite-result guard with focused regression coverage, then close this issue after the fix merges.

## Reproduction Assessment

Yes, source inspection and a focused arithmetic probe give a high-confidence reproduction path: a long pure-digit duration ending in `d` matches the regex, remains finite after `parseFloat`, then overflows to `Infinity` after multiplication. I did not run a packaged `openclaw cron add` smoke in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to reject non-finite computed milliseconds inside the cron CLI parser and add regression tests for the overflow case. No gateway schema change or product decision is needed.
