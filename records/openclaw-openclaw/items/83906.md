## Summary

Keep open: current `main` and `v2026.5.20` still return `Math.floor(n * factor)` from the cron-specific parser without a finite-result guard, and the only linked fix PR was closed unmerged. The issue is now a narrow repair candidate again because no open duplicate PR remains.

## What This Changes

The issue asks OpenClaw to reject cron duration inputs whose unit multiplication overflows to a non-finite millisecond value.

## Best Possible Solution

Add a finite-result guard to the cron-specific duration parser, or route cron through the central parser while preserving cron's null-on-invalid API, with focused regression coverage for the long pure-digit overflow case.

## Reproduction Assessment

Yes, at source level. A long pure-digit days input matches the current regex, parses to a finite number, overflows during unit multiplication, and stays truthy for the `--every` caller; I did not run a built CLI smoke in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to reject non-finite computed milliseconds in the cron parser, or to reuse the central parser that already enforces this contract while keeping existing cron error behavior.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/83906. Current `src/cli/cron-cli/shared.ts` can return `Infinity` from `parseDurationMs` when a long pure-digit value parses finitely but overflows after multiplying by the unit factor; `schedule-options.ts` and `register.cron-edit.ts` then treat that as a valid truthy duration. Add a narrow guard so non-finite computed milliseconds are rejected as invalid, or reuse `src/cli/parse-duration.ts` while preserving the cron helper's current `number | null` invalid contract and existing friendly caller errors. Add focused regression coverage for ordinary durations, zero/malformed values, a long pure-digit overflow-on-multiplication input, and a parseFloat-Infinity input. Do not add a new config option, change cron scheduling semantics, broaden accepted cron duration syntax unless intentionally routing through the central parser with tests, or touch unrelated cron/gateway behavior. Add a changelog fix entry during landing if the repository's current release process requires user-visible CLI fixes to be listed.
