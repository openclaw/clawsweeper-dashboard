## What This Changes

The issue reports that first-class `ollama-cloud/...` models can route to the retired `https://ai.ollama.com` endpoint and fail DNS, while the older `ollama/<model>:cloud` path works.

## Best Possible Solution

Land a repaired doctor migration that rewrites retired `ai.ollama.com` Ollama Cloud provider config to `https://ollama.com`, preserves explicit custom endpoints, and then close this issue after the merge.

## Reproduction Assessment

Yes for source-level reproduction: current main reads a persisted `ollama-cloud` provider URL into the runtime path, and DNS lookup for `ai.ollama.com` fails while `ollama.com` resolves. I did not run a credentialed Ollama Cloud chat request.

## Solution Assessment

Yes, the best layer is a doctor migration rather than a runtime override because repository policy says runtime reads canonical config and old shapes are repaired by `openclaw doctor --fix`. The linked PR is the right path but needs its migration-precedence finding fixed before merge.
