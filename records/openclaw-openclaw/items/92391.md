## Summary

Keep this issue open: the reported failure is credible and affects a shipped first-class provider, but current main and v2026.6.5 already use Ollama’s correct `ollama.com` endpoint. The unresolved task is to identify what supplies `ai.ollama.com` at runtime—most likely persisted provider configuration or another runtime input—before choosing the repair boundary.

## What This Changes

Investigate why direct `ollama-cloud/...` agent turns target an unresolvable endpoint while the hybrid `ollama/...:cloud` route works.

## Best Possible Solution

Trace the runtime source of `ai.ollama.com`, repair it in the Ollama plugin’s setup or doctor path if it is persisted configuration, and add focused coverage proving direct cloud turns fetch `https://ollama.com/api/chat` while preserving the hybrid local Ollama route.

## Reproduction Assessment

Unclear. The DNS failure for `ai.ollama.com` is confirmed, but current main and v2026.6.5 construct `ollama.com/api/chat` unless configured state overrides it; a redacted provider-config dump is needed to reproduce the complete agent-turn path with high confidence.

## Solution Assessment

No, changing the default endpoint is not the best fix because that value is already correct. The narrow maintainable solution is to locate and repair the runtime or persisted source of `ai.ollama.com`, without adding a fallback or another configuration surface.
