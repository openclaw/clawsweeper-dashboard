## Summary

Current main and the affected v2026.6.5 release already define and test `https://ollama.com` for the first-class Ollama Cloud provider, while runtime only appends `/api/chat` to the model’s supplied base URL. The report remains credible because configured or persisted provider state can override that default, but the issue does not yet identify what injected `ai.ollama.com`, so neither an implemented-on-main close nor a safe automated repair is supported.

## What This Changes

Fix direct `ollama-cloud/...` agent turns that resolve the invalid `ai.ollama.com` hostname even though the hybrid `ollama/<model>:cloud` route works.

## Best Possible Solution

Capture the failing installation’s redacted effective provider/model state, trace which setup or persistence path supplied `ai.ollama.com`, then fix that owner and add direct-cloud turn plus upgrade regression coverage; use plugin-owned doctor migration if OpenClaw created shipped stale state.

## Reproduction Assessment

Partly yes: the reporter gives a concrete v2026.6.5 failure path, and source confirms that configured provider state can override the correct endpoint and reach the runtime unchanged. A high-confidence complete reproduction still needs the redacted effective `ollama-cloud` provider/model configuration that contains or produces `ai.ollama.com`.

## Solution Assessment

No endpoint-default change is justified because the affected release already uses `https://ollama.com`. The narrow maintainable solution is to identify and repair the writer or migration boundary that supplies the noncanonical effective base URL.
