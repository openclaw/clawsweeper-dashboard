## Summary

Keep open. The PR targets a valid session persistence bug, but the patch is not merge-ready because the new helper can clobber a concurrently-created transcript, the branch imports the old pi-coding-agent package name, and external real behavior proof is still missing.

## What This Changes

The PR adds an optional transcript-header creation path to `resolveAndPersistSessionFile`, enables it from `initSessionState`, and adds reset/rollover tests for immediate header creation.

## Best Possible Solution

Create the successor transcript with a non-clobbering exclusive header write before publishing the store entry, keep the focused regression tests, and coordinate broader reset/replay durability with https://github.com/openclaw/openclaw/issues/75151.

## Reproduction Assessment

Yes, from source inspection. Current main can persist a new `sessionFile` during `initSessionState()` rollover before any code guarantees the successor JSONL exists, while header creation currently happens later on append.

## Solution Assessment

No. The invariant is right, but the implementation should use exclusive creation with `EEXIST` treated as success, rebase imports to the current package name, and provide real reset/rollover proof before merge.

## Review Findings
