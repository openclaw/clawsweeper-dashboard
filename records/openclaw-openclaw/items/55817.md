## Summary

Keep open. The PR targets a real session-state gap, but it is not merge-ready because it lacks real behavior proof, is conflicting against current main, and the new header bootstrap can overwrite a concurrently-created transcript.

## What This Changes

This PR adds header-only transcript creation to `resolveAndPersistSessionFile()` and enables it from `initSessionState()`, with rollover/reset regression tests.

## Best Possible Solution

Rebase onto current main and land a focused durability fix that creates the successor transcript with exclusive or locked header creation before publishing the session mapping, with rollover/concurrency coverage and real behavior proof.

## Reproduction Assessment

Yes, source-level. Current main persists a new `sessionFile` from `initSessionState()` without creating that JSONL first, and a focused rollover test can assert the mapped file is missing before append.

## Solution Assessment

No, not as submitted. Creating the header before publishing the mapping is the right direction, but the implementation should use the existing header/version helpers and non-clobbering creation under the session write invariant.

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Create the header without overwriting existing transcripts:** `src/config/sessions/session-file.ts:30-33`
  - body: This helper first observes a missing file, then writes with default overwrite semantics. Because `initSessionState` enables it even when reusing an existing entry, two workers that both see a missing current transcript can race; once one creates or appends to the transcript through the normal path, the slower bootstrap can still truncate the same file back to header-only. Use exclusive create (`flag: "wx"`) or the existing transcript header helper and treat `EEXIST` as success.
  - confidence: 0.88
