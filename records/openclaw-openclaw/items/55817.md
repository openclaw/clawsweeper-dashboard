## Summary

Keep open. The PR targets a real session-transcript durability gap and already addressed the earlier ENOENT/test-fixture review notes, but the current patch can still clobber a transcript under a stat/write race, imports the old pi-coding-agent package name, and only provides Vitest output rather than real reset/rollover proof.

## What This Changes

The PR adds an `ensureTranscriptHeader` option to `resolveAndPersistSessionFile`, enables it from `initSessionState`, and adds reset/rollover regression tests.

## Best Possible Solution

Create the successor transcript with a non-clobbering exclusive header write before publishing the store entry, keep the focused regression tests, and coordinate with https://github.com/openclaw/openclaw/issues/75151 for reset/replay durability follow-up.

## Reproduction Assessment

Yes, from source inspection. Current main can persist a new `sessionFile` during `initSessionState()` rollover before any code guarantees the successor JSONL exists.

## Solution Assessment

No. The PR points at the right invariant, but the helper should use exclusive creation with `EEXIST` treated as success, the package imports need rebasing to current main, and real reset/rollover proof is still required before merge.
