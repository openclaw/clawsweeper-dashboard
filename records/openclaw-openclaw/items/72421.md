## Summary

Keep this open. Current main has useful timeout and stuck-session plumbing, but the reported Discord cron/heartbeat stale-lane failure is not conclusively fixed because active-looking session lanes are still intentionally preserved, the outer session enqueue lacks the embedded-run timeout wrapper, heartbeats still defer on busy session lanes, and there is no supported lane-state status surface.

## What This Changes

The issue requests verification and recovery/diagnostics for Discord session lanes that stay held after cron, heartbeat, or system-event runs appear to finish.

## Best Possible Solution

Add liveness-aware stale-lane recovery that distinguishes abandoned queue bookkeeping from real in-flight work, cover Discord/system-event/heartbeat delivery with targeted tests, and expose read-only lane snapshots through a supported status or gateway diagnostic path.

## Reproduction Assessment

No. The report gives credible long-running operational evidence and current-main source inspection confirms plausible stuck-lane preservation paths, but there is no deterministic reproduction for the exact Discord cron/heartbeat stale-session race.

## Solution Assessment

No. A fixed-duration unconditional watchdog is too risky; the maintainable path is explicit liveness detection, bounded abort/drain policy, and operator-visible diagnostics.
