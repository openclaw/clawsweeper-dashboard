## Summary

Keep this issue open. Current main has partial stuck-lane diagnostics, lane snapshots, and opt-in task timeout plumbing, but it does not conclusively fix the reported Discord cron/heartbeat stale session-lane behavior: active-looking session lanes are still preserved by default, the outer session enqueue does not receive the embedded-run timeout wrapper, and there is no supported `openclaw status --lanes` diagnostic surface.

## What This Changes

The issue asks OpenClaw to diagnose and recover Discord session lanes that intermittently remain held after a run completes, causing later channel messages to wait behind stale per-session state.

## Best Possible Solution

Add liveness-aware stale-lane recovery that can distinguish abandoned bookkeeping from real in-flight work, cover the Discord cron/heartbeat delivery shape with focused tests, and expose a read-only lane snapshot through a supported diagnostic command or gateway status endpoint.

## Reproduction Assessment

No. The report provides credible long-running operational evidence, and current source inspection confirms a plausible active-lane preservation path, but there is no deterministic current-main reproduction for the exact Discord cron/heartbeat stale-lane race.

## Solution Assessment

No. An unconditional watchdog that auto-releases lanes after a fixed interval is risky for legitimate long-running work; the maintainable direction is policy-driven liveness detection plus abort/cleanup proof and operator diagnostics.
