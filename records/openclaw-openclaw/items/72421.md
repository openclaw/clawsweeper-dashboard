## Summary

Keep this open. Current main has useful timeout and conservative stuck-session recovery plumbing, but it still preserves active-looking session lanes, leaves the outer session enqueue without the embedded-run lane timeout wrapper, defers heartbeat work behind busy target session lanes, and does not provide the requested lane-state status surface.

## What This Changes

The issue asks for reliable release and diagnostics for Discord session lanes that remain held after cron, heartbeat, or system-event runs complete, plus operator-visible lane state.

## Best Possible Solution

Add liveness-aware stale-lane recovery that distinguishes abandoned queue bookkeeping from real in-flight work, cover Discord/system-event/heartbeat delivery with focused tests, and expose read-only lane snapshots through a supported status or diagnostics path.

## Reproduction Assessment

No. The report gives credible long-running operational observations and a concrete workaround, and current-main source inspection shows plausible preservation paths, but there is no deterministic reproduction for the exact Discord cron/heartbeat race.

## Solution Assessment

No. Current main's conservative recovery is a useful partial fix, but an unconditional fixed-duration watchdog is too risky; the maintainable path is explicit liveness detection, bounded abort/drain policy, and operator-visible diagnostics.
