## Summary

Keep this issue open. Current main has partial queue timeout, stale-session recovery, cron timeout cleanup, and lane snapshot plumbing, but the reported Discord cron/heartbeat stale-lane failure is not fully resolved because the outer per-session lane still lacks the embedded-run timeout wrapper and diagnostic recovery intentionally preserves active-looking lanes by default.

## What This Changes

The issue asks OpenClaw to fix intermittent Discord session-lane stalls after cron, heartbeat, or system-event runs, add defensive stale-lane recovery, and expose lane state through a diagnostic status surface.

## Best Possible Solution

Add liveness-aware core recovery that can distinguish stale bookkeeping from real in-flight work, abort or release only under approved criteria, cover the Discord cron/heartbeat delivery shape with focused tests, and expose read-only lane snapshots through a supported diagnostic command.

## Reproduction Assessment

No. The reporter provided credible long-run observations and current source/tests show the matching active-lane preservation and heartbeat deferral behavior, but there is no deterministic current-main reproduction for the exact Discord cron/heartbeat stall.

## Solution Assessment

No. An unconditional watchdog or auto-release would be risky; the maintainable fix is a liveness-aware recovery policy plus diagnostics, with tests proving active model/tool work is not interrupted.
