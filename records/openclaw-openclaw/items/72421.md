## Summary

Keep open. Current main has partial queue timeout, lane snapshot, and stuck-session recovery plumbing, but source, tests, and docs still show active-looking session lanes are preserved by default, the outer session enqueue does not receive the embedded-run timeout wrapper, and `openclaw status` has no lane diagnostic flag. Related #48488 covers the generic hung-promise queue problem, but this Discord cron/heartbeat stale session-lane surface still has distinct remaining work.

## What This Changes

The issue requests investigation and repair for intermittent Discord session-lane stalls after cron, heartbeat, or system-event runs, plus safer stale-lane recovery and lane diagnostics.

## Best Possible Solution

Add liveness-aware stale-lane recovery that can distinguish abandoned bookkeeping from real in-flight work, cover the Discord cron/heartbeat delivery shape with focused tests, and expose a read-only lane snapshot through a supported diagnostic command or gateway status endpoint.

## Reproduction Assessment

No. The report provides credible long-running operational evidence and current source confirms a plausible active-lane preservation path, but there is no deterministic current-main reproduction for the exact Discord cron/heartbeat race.

## Solution Assessment

No. Current main is only a partial mitigation, and an unconditional fixed-interval auto-release would be risky for legitimate long-running work; the maintainable path is policy-driven liveness detection plus abort/cleanup proof and operator diagnostics.
