## Summary

Keep this issue open. Current main has several relevant mitigations, including opt-in lane task timeouts, cron timeout cleanup, and conservative stuck-session recovery, but it still does not fully cover the reported Discord cron/heartbeat stale-lane shape because active-looking session lanes are intentionally preserved and there is no `openclaw status --lanes` diagnostic surface.

## What This Changes

The issue reports intermittent Discord session-lane stalls after runs complete, with stale transcript mtimes and repeated `lane wait exceeded` diagnostics.

## Best Possible Solution

Add liveness-aware core recovery that can distinguish stale bookkeeping from real in-flight work, cover the Discord cron/heartbeat delivery shape with focused tests, and expose read-only lane snapshots through a supported diagnostic command.

## Reproduction Assessment

No. The reporter provided credible long-run observations and current source/tests show the active-lane preservation shape, but there is no high-confidence deterministic reproduction for the exact Discord cron/heartbeat stall on current main.

## Solution Assessment

No. An unconditional watchdog/auto-release would be risky; the maintainable direction is a maintainer-approved liveness policy that aborts or resets only when stale bookkeeping can be distinguished from real active work.
