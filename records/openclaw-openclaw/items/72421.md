## Summary

Keep this open. Current main has partial stuck-session recovery and heartbeat busy-lane protections, but it still preserves active-looking session lanes and does not expose a supported lane-state status surface, so the reported Discord cron/heartbeat stale-lane failure is not conclusively fixed.

## What This Changes

The issue reports intermittent Discord session-lane holds after runs complete, causing `lane wait exceeded` diagnostics and stuck channel processing.

## Best Possible Solution

Add liveness-aware stale-lane recovery that distinguishes abandoned queue bookkeeping from real in-flight work, cover Discord/system-event/heartbeat delivery with targeted tests, and expose read-only lane snapshots through a supported status or gateway diagnostic path.

## Reproduction Assessment

No. The report gives credible long-running operational evidence and current-main source inspection confirms plausible stuck-lane preservation paths, but there is no deterministic reproduction for the exact Discord cron/heartbeat stale-session race.

## Solution Assessment

No. A fixed-duration unconditional watchdog would be too risky; the maintainable path is explicit liveness detection, bounded abort/drain policy, and operator-visible diagnostics.
