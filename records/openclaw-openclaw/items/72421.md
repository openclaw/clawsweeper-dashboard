## Summary

Current main and the shipped v2026.4.29 release now cover the reported stale Discord/session-lane failure modes with bounded command-lane task timeouts, bounded embedded-run cleanup, cron timeout force-cleanup, and conservative stale-session lane recovery. The remaining `status --lanes` idea is a diagnostics enhancement, not enough to keep this incident report open after the central stuck-processing behavior was fixed and released.

## What This Changes

The issue reports intermittent Discord channel session lanes staying held after cron, heartbeat, or system-event runs complete, producing `lane wait exceeded` diagnostics until the session row is manually removed.

## Best Possible Solution

Keep the shipped v2026.4.29 lane-timeout, cron-cleanup, and stale-session recovery implementation. If operators still want a read-only lane inspector, track that as a narrower diagnostics enhancement rather than keeping this fixed incident open.

## Reproduction Assessment

No. The original Discord race is intermittent and long-running, but current-main source and focused regression tests cover the observed stale-lane mechanisms: wedged command tasks, cleanup hangs, cron timeout cleanup, and stale diagnostic session recovery.

## Solution Assessment

Yes. The current implementation avoids an unsafe unconditional watchdog by bounding known run/cleanup paths and only releasing stale session lanes when diagnostics show no active embedded, reply, or lane work.
