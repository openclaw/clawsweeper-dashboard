## Summary

Keep this open. Current main has conservative stuck-session recovery and better diagnostics, but it still intentionally preserves active-looking embedded/reply/lane work and does not expose a supported `openclaw status --lanes` surface, so the reported Discord cron/heartbeat stale-lane failure is not conclusively fixed.

## What This Changes

Investigate intermittent Discord channel session lanes remaining held after completed cron/heartbeat/system-event runs, with defensive recovery and lane-state diagnostics.

## Best Possible Solution

Add liveness-aware stale-lane recovery that distinguishes abandoned active lane bookkeeping from real in-flight model/tool work, cover Discord/system-event/heartbeat paths with targeted tests, and expose read-only lane snapshots through CLI or Gateway diagnostics.

## Reproduction Assessment

No. The report provides credible long-running operational evidence and current source inspection confirms plausible non-recoverable active-looking lane paths, but there is no deterministic reproduction for the exact Discord cron/heartbeat stale-session race.

## Solution Assessment

No. Current conservative recovery is safer than an unconditional watchdog but incomplete for the reported active-looking stale lane; the maintainable fix needs explicit liveness detection plus bounded abort/drain policy.
