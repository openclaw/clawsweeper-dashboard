## Summary

Keep this issue open. Current main has partial queue timeout, lane snapshot, and diagnostic recovery plumbing, and the current attempt cleanup does release the session write lock, but the observable Discord cron/heartbeat stale-lane problem is not fully solved: the outer session lane lacks the embedded-run timeout wrapper, active-looking lanes are intentionally preserved, heartbeat and cron still defer behind busy session lanes, and there is no `openclaw status --lanes` surface.

## What This Changes

Request diagnostic and recovery improvements for Discord session lanes that can remain busy after cron, heartbeat, or system-event runs finish, including safe stale-lane auto-release and a lane status surface.

## Best Possible Solution

Add liveness-aware core recovery that can distinguish stale bookkeeping from real in-flight work, abort or release only after maintainer-approved criteria, cover the Discord cron/heartbeat delivery shape with focused tests, and expose read-only lane snapshots through a supported diagnostic command.

## Reproduction Assessment

No. The reporter provided credible long-run observations and current source/tests show the active-lane preservation and heartbeat skip shape, but there is no deterministic current-main reproduction for the exact Discord cron/heartbeat stall.

## Solution Assessment

No. The requested unconditional watchdog/auto-release direction is risky as a final fix; the narrower maintainable solution is a liveness-aware recovery policy plus diagnostics, with tests proving active work is not interrupted.
