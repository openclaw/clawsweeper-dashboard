## Summary

Keep this issue open. Current main has partial timeout and stuck-session recovery plumbing, but the reported Discord cron/heartbeat session-lane stall is not conclusively fixed: session-lane timeouts are not applied at the outer per-session enqueue, active-looking lanes remain preserved by default, and there is still no supported lane-state status surface.

## What This Changes

The issue asks OpenClaw to fix and diagnose intermittent Discord session lanes that stay held after cron, heartbeat, or system-event runs complete, including safer stale-lane recovery and an `openclaw status --lanes` diagnostic surface.

## Best Possible Solution

Add liveness-aware stale-lane recovery that can distinguish abandoned queue bookkeeping from real in-flight work, cover Discord cron/heartbeat delivery with targeted tests, and expose read-only lane snapshots through a supported status or gateway diagnostic path.

## Reproduction Assessment

No. The report gives credible long-running operational evidence and current source inspection confirms plausible stuck-lane preservation paths, but there is no deterministic current-main reproduction for the exact Discord cron/heartbeat stale-lane race.

## Solution Assessment

No. An unconditional watchdog that releases lanes after a fixed interval is too risky; the maintainable solution is policy-driven liveness detection with abort/drain semantics and operator-visible diagnostics.
