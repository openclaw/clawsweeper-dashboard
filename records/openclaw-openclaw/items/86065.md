## Summary

Keep open. Current main and the latest release still enforce a 120000 ms Active Memory ceiling in runtime normalization and the plugin manifest, while the related outer hook timeout work is already separate and does not remove this plugin-owned cap.

## What This Changes

The issue requests raising or making configurable the Active Memory plugin's internal `timeoutMs` ceiling above 120000 ms for slower local recall models.

## Best Possible Solution

Choose a bounded Active Memory timeout policy, then align the runtime clamp, manifest schema, docs, and focused tests around that ceiling or an explicit plugin-level configuration surface.

## Reproduction Assessment

Yes from source inspection: current main rejects or clamps values above 120000 through the plugin manifest and `normalizePluginConfig()`. I did not run a live vLLM reproduction of the reported latency profile.

## Solution Assessment

Unclear until maintainers choose the policy. Raising the fixed ceiling is the smallest implementation, while a new explicit cap or strict/extended mode would be safer if maintainers want longer budgets without changing the default reply-path behavior.
