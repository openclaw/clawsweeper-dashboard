## Summary

Keep open. Current main and v2026.5.22 still enforce the 120000 ms Active Memory `timeoutMs` ceiling in runtime normalization, plugin schema, docs, and tests; changing that ceiling is a bounded but real product/safety decision for a blocking pre-reply memory plugin.

## What This Changes

The issue asks OpenClaw to raise or make configurable the Active Memory plugin’s `timeoutMs` hard cap above 120000 ms for slower local recall models.

## Best Possible Solution

Choose an owner-approved Active Memory timeout policy, then align the runtime clamp, manifest schema, docs, and focused tests around either a higher fixed ceiling or an explicit opt-in extended budget while preserving safe defaults.

## Reproduction Assessment

Yes for source reproduction: configuring a value above 120000 is rejected by the manifest and clamped by `normalizePluginConfig()` on current main. I did not run a live local-vLLM reproduction of the latency profile.

## Solution Assessment

Unclear until maintainers choose the policy. A higher fixed ceiling is the smallest code change, while an explicit extended-budget opt-in is safer if longer blocking recall should not become the normal default.
