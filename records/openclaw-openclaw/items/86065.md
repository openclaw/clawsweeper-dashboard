## Summary

Keep open. Current main and v2026.5.22 still enforce the 120000 ms Active Memory ceiling in runtime normalization, manifest validation, tests, and docs; changing that ceiling is a valid enhancement but needs a maintainer policy decision because this plugin blocks the reply path.

## What This Changes

Requests increasing or making configurable the Active Memory plugin `timeoutMs` ceiling above 120000 ms for slow local-model recall runs.

## Best Possible Solution

Choose a bounded Active Memory timeout policy, then align runtime normalization, manifest schema, docs, and focused tests around that ceiling or an explicit plugin-level configuration surface.

## Reproduction Assessment

Yes, at source level: current main rejects or clamps `timeoutMs` above 120000 through the plugin manifest and `normalizePluginConfig()`. I did not run a live vLLM reproduction of the reported latency profile.

## Solution Assessment

Unclear until maintainers choose the policy. Raising the fixed ceiling is the smallest code change, while an explicit extended-budget config surface may be safer if longer blocking recall should remain opt-in.
