## Summary

Keep this open: current main still has a separator-consuming block streaming path, and open PR #73981 already targets this issue with closing syntax, so the issue should be resolved by landing or replacing that PR rather than closing early.

## What This Changes

Preserve paragraph and newline separators when block-streamed Markdown is split across multiple visible reply deliveries.

## Best Possible Solution

Land or replace PR #73981 with a narrow shared streaming fix that preserves paragraph/newline boundary semantics through chunking, coalescing, and delivery, with regression coverage for multi-paragraph Markdown split across flushes.

## Reproduction Assessment

Yes, as a source-level reproduction: append multi-paragraph Markdown to the flush-on-paragraph chunker, let the pipeline flush between emitted payloads, and the chunker strips the separator before downstream plain-text delivery. I did not run a live channel roundtrip in this read-only sweep.

## Solution Assessment

Yes, preserving boundary semantics in the shared block streaming pipeline is the narrow maintainable fix. A downstream rule that always joins deliveries with blank lines would be unsafe because flushes can also happen for size, idle, or visibility reasons.
