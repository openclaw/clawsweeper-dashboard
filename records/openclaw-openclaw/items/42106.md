## Summary

Keep this open: current main still has the separator-consuming block streaming path, and open PR #73981 already targets this issue with closing syntax, so it should be resolved by landing or replacing that PR rather than by closing early.

## What This Changes

The issue reports that block-streamed Markdown can lose paragraph separators when a visible reply is reconstructed from multiple streamed deliveries.

## Best Possible Solution

Land or replace PR #73981 with a narrow shared streaming fix that preserves paragraph/newline boundary semantics through chunking, coalescing, and delivery, with regression coverage for multi-paragraph Markdown split across flushes.

## Reproduction Assessment

Yes, from source inspection rather than a live run: append multi-paragraph Markdown to the flush-on-paragraph chunker and let the coalescer flush between emitted payloads. The chunker drops the separator before downstream clients receive plain text deliveries, and the current test suite asserts that separator-consuming behavior.

## Solution Assessment

Yes, preserving boundary semantics in the shared block streaming pipeline is the narrow maintainable fix. A downstream rule such as always joining deliveries with blank lines would be unsafe because flushes can also happen for size, idle, or visibility reasons.
