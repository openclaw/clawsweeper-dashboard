## Summary

Keep open: current main and v2026.6.6 still have separator-consuming block chunking, and the only directly linked fix PR was closed unmerged and routed the remaining work back here.

## What This Changes

The issue asks OpenClaw to preserve paragraph/newline boundary semantics when block-streamed reply chunks are delivered across separate visible updates.

## Best Possible Solution

Fix the shared block-streaming chunker or delivery pipeline so emitted chunks carry or encode paragraph separators across delivery flushes, with focused chunker and subscribe-pipeline regression coverage instead of channel-specific joiner guesses.

## Reproduction Assessment

Yes, source-reproducible: current `EmbeddedBlockChunker` emits before paragraph separators and advances past them, and the subscribe test asserts separate visible block replies without a joiner. I did not run live channel proof in this read-only review.

## Solution Assessment

Yes, the shared chunker/pipeline boundary is the best layer because block streaming is documented as generic chunker-owned channel-message delivery. A channel-level rule that always joins deliveries with blank lines would corrupt non-paragraph flush boundaries.
