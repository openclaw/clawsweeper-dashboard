## Summary

Keep this open: current main still has the separator-consuming block-streaming path, and the linked closing PR is open but not merged.

## What This Changes

The issue reports that block-streamed replies can lose Markdown paragraph separators when a multi-paragraph response is delivered across multiple visible updates.

## Best Possible Solution

Land or replace the open shared streaming fix so boundary semantics survive chunking, subscribe delivery trimming, coalescing, and regression tests for multi-paragraph Markdown.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: stream multi-paragraph Markdown through `EmbeddedBlockChunker` with `flushOnParagraph`, then observe that emitted and delivered plain text loses the separator across flush boundaries.

## Solution Assessment

Yes. A shared block-streaming pipeline fix is the narrow maintainable solution; channel-specific joining rules would be unsafe because flushes can happen for paragraph, size, idle, assistant-block, or final-delivery reasons.
