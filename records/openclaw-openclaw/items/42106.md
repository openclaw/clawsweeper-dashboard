## Summary

Keep open: current main and v2026.5.20 still lose paragraph-boundary information in the shared block-streaming path, and the linked closing fix remains open rather than merged.

## What This Changes

The issue reports that block-streamed replies can drop paragraph separators when multi-paragraph Markdown is split across multiple streamed deliveries.

## Best Possible Solution

Repair and land https://github.com/openclaw/openclaw/pull/73981 or a narrow replacement that preserves boundary semantics through chunking, subscribe delivery, and regression coverage.

## Reproduction Assessment

Yes, at source level. Current main emits before the paragraph separator, skips leading newlines after it, and trimEnd()s delivered block reply text before downstream reconstruction.

## Solution Assessment

Yes. The shared block-streaming pipeline is the right fix surface because channel-specific joiners cannot infer paragraph boundaries after plain text deliveries have already been flushed.
