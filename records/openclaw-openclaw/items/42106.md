## Summary

Keep open. Current main still contains the reported block-streaming separator loss, and the issue is paired with open PR #73981, which should be reviewed, landed, or closed before this issue is resolved.

## What This Changes

The issue requests preserving paragraph separators when multi-paragraph Markdown is split across multiple block-streamed deliveries.

## Best Possible Solution

Keep this issue open until #73981 or an equivalent shared pipeline fix lands with regression coverage for reconstructing multi-paragraph Markdown across separately flushed block replies.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: `First block line\n\nSecond block line` is currently expected to stream as two plain text payloads, `First block line` and `Second block line`, so a concatenating client loses the paragraph break.

## Solution Assessment

Yes. The maintainable fix is in the shared block-streaming chunker/coalescer delivery path, either by retaining consumed separators in emitted chunks with normalized dedupe or by carrying explicit boundary metadata; per-channel join rules would mis-handle non-paragraph flushes.
