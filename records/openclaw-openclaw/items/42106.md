## Summary

Keep open: current main still has the separator-consuming block-streaming path, and the linked fix at https://github.com/openclaw/openclaw/pull/73981 closes this issue but remains open.

## What This Changes

Preserve Markdown paragraph separators when block streaming splits a reply across multiple streamed deliveries.

## Best Possible Solution

Land or replace the shared streaming fix so boundary semantics survive chunking, subscribe delivery, coalescing, and regression tests for multi-paragraph Markdown.

## Reproduction Assessment

Yes, at source level. Current main routes flush-on-paragraph Markdown through EmbeddedBlockChunker, emits text before the paragraph separator, skips leading newlines for the next delivery, and trims delivered block text.

## Solution Assessment

Yes. A shared block-streaming pipeline fix is the maintainable direction because channel-specific joiners cannot distinguish paragraph boundaries from size, idle, assistant-block, or final-delivery flushes after plain text has been delivered.
