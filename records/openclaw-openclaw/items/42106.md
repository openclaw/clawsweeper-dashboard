## Summary

Keep open: current main still strips block-streaming paragraph separators before delivery, and open PR #73981 is the paired closing implementation candidate that should be reviewed before resolving this issue.

## What This Changes

The issue requests preserving paragraph and newline boundary semantics when block streaming splits a Markdown reply across multiple visible deliveries.

## Best Possible Solution

Review and land PR #73981, or replace it with an equivalent shared streaming-pipeline fix that preserves boundary semantics through chunking/coalescing and updates regression coverage for reconstructed Markdown.

## Reproduction Assessment

Yes. Source inspection plus the existing paragraph-preference harness show `First block line\n\nSecond block line` delivered as two stripped `onBlockReply` payloads, so a client concatenating visible updates loses the blank line.

## Solution Assessment

Yes. The maintainable solution is a shared block-streaming boundary fix rather than per-channel join rules, and PR #73981 is the active implementation candidate to review first.
