## Summary

Keep this open: current main and the latest release still consume paragraph separators at the shared block-streaming chunk boundary, and the related repair PR was closed unmerged with the remaining work routed back here.

## What This Changes

The issue reports that block-streamed replies can lose paragraph separators when visible text is reconstructed from multiple streamed deliveries.

## Best Possible Solution

Preserve paragraph boundary semantics in the shared block-streaming pipeline so progressive visible reconstruction remains Markdown-equivalent across flushes.

## Reproduction Assessment

Yes from source inspection: paragraph chunking emits the pre-separator text and drops/skips the separator before buffering the next text, so a client that concatenates deliveries cannot recover the blank line.

## Solution Assessment

No current implemented fix was found. The shared chunker/coalescing boundary is the right layer; channel-level rules that always join deliveries with blank lines would corrupt non-paragraph flush boundaries.
