## Label Justifications

- `P2`: The bug deterministically corrupts user-visible Markdown for affected block-streaming consumers but has limited blast radius and does not disable the core runtime.
- `impact:message-loss`: The shared streaming path can silently discard paragraph-separator characters between delivered message blocks.

## Summary

Current main still consumes paragraph separators before emitting separate block-streamed deliveries, and the open linked PR has not yet merged, so this issue remains necessary as the canonical bug report.

## What This Changes

Preserve paragraph separators when block-streamed replies are reconstructed across multiple deliveries.

## Best Possible Solution

Preserve explicit boundary semantics in the shared block-streaming pipeline, with focused coverage showing that concatenating separate flushed deliveries reconstructs paragraph Markdown while hard-size and fenced-code splits remain unchanged.

## Reproduction Assessment

Yes from source inspection: the paragraph-flush branch emits text before the separator, consumes that separator, and buffers only the following text, so concatenating separate deliveries loses the blank line. A live channel reproduction was not run during this read-only review.

## Solution Assessment

Yes in direction: the invariant belongs in the shared chunking or delivery boundary rather than in channel-specific join logic. The linked PR should retain only the table-related expansion that is necessary to preserve the same boundary invariant safely.
