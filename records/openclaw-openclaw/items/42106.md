## Summary

Keep open: current main still has the separator-consuming block-streaming path, and the linked closing PR remains open and conflicting, so this issue should not be closed before that fix is landed or replaced.

## What This Changes

Preserve paragraph separators when block streaming reconstructs multi-delivery Markdown replies.

## Best Possible Solution

Land or replace the shared streaming fix so boundary semantics survive chunking, subscribe delivery, coalescing, and regression coverage for multi-paragraph Markdown.

## Reproduction Assessment

Yes, at source level. Current main drains flushOnParagraph Markdown through EmbeddedBlockChunker, emits text before the paragraph separator, skips leading newlines for the next delivery, and trimEnd()s delivered block text.

## Solution Assessment

Yes, the shared block-streaming pipeline is the right place to solve this. Channel-specific joiners cannot distinguish paragraph boundaries from size, idle, assistant-block, or final-delivery flushes once plain text has been delivered.
