## Summary

Keep this open: current main still has the separator-consuming block-streaming path, and https://github.com/openclaw/openclaw/pull/73981 closes this issue but remains open with review blockers.

## What This Changes

Preserve paragraph boundaries when block-streamed Markdown is reconstructed across multiple delivered updates.

## Best Possible Solution

Land or replace the shared streaming fix so boundary semantics survive chunking, subscribe delivery, coalescing, and regression tests for multi-paragraph Markdown.

## Reproduction Assessment

Yes, at source level. Current main routes flush-on-paragraph Markdown through `EmbeddedBlockChunker`, emits text before the paragraph separator, then skips leading newlines before the next delivery.

## Solution Assessment

Yes. A shared block-streaming pipeline fix is the narrow maintainable solution; channel-specific joiners cannot distinguish paragraph boundaries from size, idle, assistant-block, or final-delivery flushes after plain text has been delivered.
