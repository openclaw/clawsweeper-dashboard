## Summary

Keep this open: current `main` still has the separator-consuming block-streaming path, and the open closing PR at https://github.com/openclaw/openclaw/pull/73981 is the active fix path but has not merged.

## What This Changes

Preserve paragraph separators when a block-streamed Markdown reply is delivered as multiple visible updates.

## Best Possible Solution

Land or replace the open shared streaming fix so boundary semantics survive chunking, delivery trimming, coalescing, and regression tests for multi-paragraph Markdown.

## Reproduction Assessment

Yes. The source path is reproducible by streaming multi-paragraph Markdown through `EmbeddedBlockChunker` with `flushOnParagraph`, then observing that the emitted and delivered text loses the separator before downstream visible updates can reconstruct it.

## Solution Assessment

Yes. The narrow maintainable solution is a shared block-streaming pipeline fix; channel-specific joining rules would be unsafe because flushes can happen for size, idle, assistant-block, or final-delivery reasons.
