## Summary

Keep open: current main and the latest release still have the separator-consuming block-streaming path, and https://github.com/openclaw/openclaw/pull/73981 is an open, unmerged fix candidate. This issue is source-reproducible and should remain the tracker until the shared streaming fix lands or is replaced.

## What This Changes

The issue asks OpenClaw to preserve paragraph separators when block-streamed Markdown replies are reconstructed across multiple streamed deliveries.

## Best Possible Solution

Review, repair, and land or replace https://github.com/openclaw/openclaw/pull/73981 so boundary semantics survive chunking, subscribe delivery, coalescing, and focused regression coverage.

## Reproduction Assessment

Yes, at source level. Current main emits before paragraph separators, skips leading newlines after the separator, and trimEnds delivered block text before downstream reconstruction.

## Solution Assessment

Yes. The shared block-streaming pipeline is the right place to fix this because channel-specific joiners cannot infer paragraph boundaries after plain text deliveries have already been flushed.
