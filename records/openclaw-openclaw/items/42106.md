## Summary

Keep open: current main still has the separator-consuming block-streaming path, and the open closing PR at https://github.com/openclaw/openclaw/pull/73981 remains the active implementation candidate rather than a merged fix.

## What This Changes

The issue asks for block streaming to preserve paragraph/newline boundary semantics across multiple streamed deliveries so reconstructed Markdown does not collapse paragraph separators.

## Best Possible Solution

Land or replace the shared streaming fix so boundary semantics survive chunking, subscribe delivery, coalescing, and regression coverage for multi-paragraph Markdown.

## Reproduction Assessment

Yes, at source level. Current main drains flushOnParagraph Markdown through EmbeddedBlockChunker, emits text before the paragraph separator, strips leading newlines from the remainder, and trimEnd()s delivered block text.

## Solution Assessment

Yes, the shared block-streaming pipeline is the right place to solve this. Channel-specific joiners cannot distinguish paragraph boundaries from size, idle, assistant-block, or final-delivery flushes once plain text has been delivered.
