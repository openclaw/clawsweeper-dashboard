## Label Justifications

- `P2`: This is a reproducible user-visible Markdown corruption bug in block-streamed replies with limited blast radius.
- `impact:message-loss`: Paragraph separators can be silently dropped from delivered channel text across streamed updates.

## Summary

Keep open: current main still has the separator-consuming block-streaming path, and the open closing PR at https://github.com/openclaw/openclaw/pull/73981 is the active fix candidate rather than a merged implementation.

## What This Changes

Preserve paragraph separators across multiple block-streamed deliveries so Markdown structure remains intact.

## Best Possible Solution

Repair and land the shared streaming fix so boundary semantics survive chunking, subscribe delivery, coalescing, and regression coverage for multi-paragraph Markdown.

## Reproduction Assessment

Yes, at source level. Current main's flushOnParagraph path emits text before the paragraph separator, strips leading newlines from the remainder, and trimEnd()s delivered block text.

## Solution Assessment

Yes, the shared block-streaming pipeline is the right place to solve this. Channel-specific joiners cannot distinguish paragraph boundaries from size, idle, assistant-block, or final-delivery flushes once plain text has been delivered.
