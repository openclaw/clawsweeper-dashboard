## Summary

Keep this open: current main and the latest release still have the separator-consuming block-streaming path, and two open closing PRs are active candidate fixes.

## What This Changes

The issue asks OpenClaw to preserve paragraph separators when block-streamed reply chunks are delivered and reconstructed across multiple flushes.

## Best Possible Solution

Review and land one narrow shared chunker/coalescer fix that preserves paragraph separators across flushed deliveries without doubling coalesced joiners, then let the linked issue close from the merge.

## Reproduction Assessment

Yes. Source inspection shows a paragraph-split chunk can be emitted before the `\n\n` separator while the buffer start moves past that separator, leaving downstream deliveries without boundary metadata after a flush.

## Solution Assessment

Yes. The best fix belongs in the shared chunker/coalescer boundary; a channel-specific downstream join rule cannot distinguish paragraph boundaries from size or timing flushes.
