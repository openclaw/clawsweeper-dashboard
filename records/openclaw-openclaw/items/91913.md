## Summary

Current main and the latest release retain a source-confirmed split between the two OpenAI completions paths: active-memory supplies an explicit thinking level, but the legacy builder ignores the compatibility mapping used by LM Studio. The issue remains necessary and has a bounded repair path.

## What This Changes

Fix LM Studio binary-reasoning requests so active-memory’s explicit thinking-off setting and provider-native effort mapping survive the legacy OpenAI completions path.

## Best Possible Solution

Create one complete canonical LM Studio reasoning translation—disabled input to the supported disabled value and every enabled OpenClaw level to `on`—then make both OpenAI-completions builders consume the same mapping without adding plugin configuration.

## Reproduction Assessment

Yes at source level with high confidence: current main has a concrete caller-to-builder mismatch that sends an untranslated effort on the legacy path. The reporter also supplied a matching LM Studio warning, though this review did not rerun the external model.

## Solution Assessment

No, the proposed one-line fallback is incomplete because the binary LM Studio map also omits canonical enabled-level mappings. The narrowest maintainable solution is a complete shared mapping used by both builders, with coverage for explicit off and enabled requests.
