- `impact:auth-provider`: The failure is in provider/model option routing and compatibility normalization for an OpenAI-compatible backend.

## Summary

The issue remains actionable on current main, but the proposed legacy-path patch targets the wrong boundary: active-memory now uses the newer OpenClaw completions transport, where missing reasoning defaults to `high`, explicit `thinking: off` does not suppress `reasoning_effort`, and LM Studio’s binary compatibility map does not translate ordinary enabled levels to native `on`.

## What This Changes

Fix OpenAI-compatible reasoning propagation so active-memory thinking settings and LM Studio binary `off`/`on` values are honored instead of silently sending `high`.

## Best Possible Solution

Handle explicit off and provider-native binary translation in the shared current OpenAI-completions reasoning boundary, mapping disabled intent to LM Studio `off` and canonical enabled levels to `on`, with focused active-memory and LM Studio regression coverage.

## Reproduction Assessment

Yes at source level: active-memory forwards `thinkLevel: off`, while the current completions transport defaults a missing effort to `high` and the off wrapper does not suppress that field. The reporter also supplied a concrete LM Studio warning and timing trace, though this review did not run the live server.

## Solution Assessment

No, changing only the legacy `src/llm/providers/openai-completions.ts` builder is not the best solution because current embedded runs select the newer boundary-aware transport. The narrow maintainable fix belongs in the shared current reasoning resolver/wrapper and LM Studio’s provider-owned binary map.
