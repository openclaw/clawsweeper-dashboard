## Summary

Keep open: current main and the latest v2026.6.6 release still show the source-level active-memory/LM Studio reasoning path, and multiple open PRs explicitly claim to fix this issue but have not merged.

## What This Changes

The issue reports that LM Studio binary-thinking models can receive an unsupported `reasoning_effort` value because the active-memory embedded path does not reliably carry documented reasoning compatibility mapping or explicit thinking-off intent to the OpenAI-completions request.

## Best Possible Solution

Review and repair one canonical linked PR so the current active-memory embedded path sends provider-accepted reasoning values, with focused regression coverage and redacted LM Studio proof.

## Reproduction Assessment

Yes at source level: active-memory forwards `thinkLevel: params.config.thinking`, and the current OpenAI completions transport resolves missing reasoning effort to `high` before emitting `reasoning_effort`. I did not run a live LM Studio server in this read-only review.

## Solution Assessment

The issue is valid, but a legacy-builder-only fix is not clearly the best current solution because embedded runs can use the boundary-aware transport. The best repair should target the current transport or LM Studio owner hook and prove the active-memory path end to end.
