## Summary

Keep open: current main and the latest release still leave the legacy OpenAI completions builder on a different reasoning mapping path from the newer transport builder, and active-memory is already passing its `thinking` setting into the embedded run.

## What This Changes

The issue reports that active-memory calls through the legacy OpenAI completions path ignore configured reasoning-effort compatibility mapping and can send unsupported LM Studio binary-thinking values.

## Best Possible Solution

Keep this issue open and align the legacy OpenAI completions request builder with the shared reasoning compatibility resolver, preserving active-memory's existing `thinking` config without adding a new plugin knob.

## Reproduction Assessment

Yes, from source: configure an OpenAI-completions reasoning model with `compat.reasoningEffortMap` but no top-level `thinkingLevelMap`, then route a simple embedded run through active-memory; the legacy builder either omits the mapped off value or emits the raw enabled effort. I did not run a live LM Studio endpoint in this read-only review.

## Solution Assessment

Yes, the best fix is in the shared legacy OpenAI completions request-building path, not a plugin-side active-memory schema addition. Reusing the same resolver shape as the newer transport path is narrower and keeps provider compatibility in the provider/request layer.

Fix https://github.com/openclaw/openclaw/issues/91913. Current main shows active-memory passes `thinkLevel: params.config.thinking` into embedded runs, but `src/llm/providers/openai-completions.ts` maps simple `reasoning: "off"` to `undefined` and later maps `reasoning_effort` only through top-level `model.thinkingLevelMap`, ignoring `model.compat.reasoningEffortMap`. Align the legacy OpenAI completions builder with the shared reasoning compatibility behavior used by `src/agents/openai-transport-stream.ts` / `src/agents/openai-reasoning-compat.ts`, preserving existing `thinkingLevelMap` behavior and avoiding LM Studio-specific hardcoding or new config/env surfaces. Do not edit `CHANGELOG.md`.
