## Summary

Current main still has a real contract split: active-memory forwards its configured thinking level through the embedded runner, but the lower-level OpenAI completions provider maps request effort through `thinkingLevelMap` instead of the documented compat-aware resolver used by the newer transport. The related fix established provider-native mapping as supported behavior, so this distinct legacy-path regression should remain open for a focused repair.

## What This Changes

Fix active-memory’s embedded LM Studio chat-completions requests so explicit thinking-off and configured binary provider-native reasoning mappings are preserved.

## Best Possible Solution

Reuse one provider-native reasoning resolver across embedded and ordinary OpenAI-compatible calls, with focused tests proving explicit `off` maps to `off` and enabled canonical levels map to `on` for binary LM Studio models.

## Reproduction Assessment

Yes at source level: current main preserves active-memory’s setting through the plugin and embedded runner, then exposes two request builders with different reasoning-map contracts. A live LM Studio inference was not run during this read-only review.

## Solution Assessment

The suggested legacy-path change is directionally correct, but the best fix is to reuse the shared compat-aware resolver and explicitly test thinking `off`; adding active-memory config or a plugin-side request override would duplicate policy at the wrong boundary.
