## Summary

Keep open. Current main still exposes `llm_output` as identifiers, assistant output, and usage counters only; the requested model context-window, effective context-budget, and max-output-token metadata exists in resolved model/attempt data but is not threaded through the hook contract or emitters.

## What This Changes

Request to include model context-window and max-output-token metadata, plus optional utilization or effective-budget data, in the `llm_output` plugin hook event.

## Best Possible Solution

Add optional, backwards-compatible numeric fields for native context window, effective context token budget, and max output tokens to the `llm_output` hook payload, populate them consistently across Pi, CLI, and Codex app-server emitters, and leave utilization plugin-derived unless core defines a stable numerator.

## Reproduction Assessment

Yes. A static current-main check of `PluginHookLlmOutputEvent` plus the Pi, CLI, and Codex app-server `llm_output` emitters reproduces the missing fields without needing a live LLM call.

## Solution Assessment

Yes. An additive hook-contract update is the narrowest maintainable fix because plugins already observe `llm_output` and core already resolves the relevant model metadata; plugin-owned lookup tables would drift.
