## Summary

Keep open. Current main still exposes `llm_output` as an assistant-text-and-usage hook only, while the resolved model metadata needed for native context window, effective context budget, and max output-token limit exists elsewhere and is already threaded through nearby runtime paths.

## What This Changes

Add context-window and model max output-token metadata to the plugin-facing `llm_output` event so plugins can compute context health without maintaining their own model lookup table.

## Best Possible Solution

Add optional, backwards-compatible fields to the plugin hook contract for native model context window, effective context-token budget, and model max output tokens; populate them in the embedded Pi runner, CLI harness, and Codex app-server harness; update focused hook tests and public plugin docs. Leave utilization plugin-derived unless OpenClaw defines a reliable context-used numerator.

## Reproduction Assessment

Yes. A static current-main check of `PluginHookLlmOutputEvent` plus the Pi, CLI, and Codex `runLlmOutput` emitters reproduces the missing fields without needing a live LLM call.

## Solution Assessment

Yes. An additive hook-contract update is the narrowest maintainable fix because plugins already receive `llm_output` and the relevant model metadata exists; computing utilization in core should wait until the numerator semantics are explicit.
