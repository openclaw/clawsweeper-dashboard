## Summary

Keep open. Current main still exposes `llm_output` as identifiers, assistant text, and usage only; the requested context-window and model max-output-token metadata exists upstream but is not threaded through the plugin hook contract or emitters. Related #64327 is adjacent effective-budget work, not a superseding fix.

## What This Changes

Expose model context-window size and max output-token metadata on `llm_output` hook events so plugins can compute context utilization.

## Best Possible Solution

Add optional, backwards-compatible numeric fields for native context window, effective context-token budget, and model max output tokens to the hook payload, then update emitters, docs, focused tests, and changelog; leave utilization plugin-derived unless core defines an exact numerator.

## Reproduction Assessment

Yes. A static current-main check of `PluginHookLlmOutputEvent` plus the Pi, CLI, and Codex app-server `llm_output` emitters reproduces the missing fields without a live LLM call.

## Solution Assessment

Yes. An additive hook-contract update is the narrowest maintainable solution because plugins already observe `llm_output` and core already resolves the relevant model metadata; plugin-side lookup tables would drift.
