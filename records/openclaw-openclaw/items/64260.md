## Summary

Keep open. Current main still exposes `llm_output` as identifiers, assistant text, and usage only; the requested context-window and max-output-token metadata is present in resolved model/config structures but is not threaded through the plugin hook contract or emitters. The related #64327 should inform effective-budget naming, but it does not supersede this request because this issue also covers native context window and model max output tokens.

## What This Changes

Request to add model context-window and max-output-token metadata, optionally effective utilization, to `llm_output` plugin hook events.

## Best Possible Solution

Add optional, backwards-compatible numeric fields to the plugin hook event for native context window, effective context-token budget, and model max output tokens; populate them consistently in Pi, CLI, and Codex app-server emitters; update focused tests, plugin docs, and changelog. Leave utilization plugin-derived unless core defines an exact context-used numerator.

## Reproduction Assessment

Yes. A static current-main check of `PluginHookLlmOutputEvent` and the Pi, CLI, and Codex app-server `llm_output` emitters reproduces the missing fields without a live LLM call.

## Solution Assessment

Yes. An additive hook-contract update is the narrowest maintainable solution because plugins already observe `llm_output` and core already resolves the relevant model metadata; maintaining external plugin lookup tables would drift.
