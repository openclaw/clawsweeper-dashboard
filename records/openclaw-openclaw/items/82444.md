## What This Changes

This PR adds a client socket error listener in the debug proxy CONNECT tunnel and switches upstream-error cleanup from `clientSocket.end()` to `clientSocket.destroy()`.

## Best Possible Solution

Land the symmetric CONNECT socket error cleanup after redacted after-fix proof from patched code is supplied or maintainers run equivalent real proxy proof; add focused regression coverage if feasible.

## Reproduction Assessment

Yes, at source level: current main pipes both CONNECT sockets but only registers `upstreamSocket.on("error")`, and a Node error event without a listener throws. The PR body also includes a release repro showing the unpatched proxy did not record an error burst, but I did not run patched code.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable fix: handle client-side socket errors in the tunnel, record the existing capture error shape, and destroy the paired socket. The remaining blocker is after-fix proof, not a different implementation direction.
