## What This Changes

The PR adds a `WeakMap` cache to `applyPluginAutoEnable` keyed by config/env identity and adds cache behavior/timing tests.

## Best Possible Solution

Keep the optimization only after the cache key covers the effective manifest registry, with registry-varying regression coverage and redacted gateway/dashboard fanout proof.

## Reproduction Assessment

Yes for the blocking patch regression: source inspection shows the patched helper returns a cache hit for the same config/env pair before reading a different `manifestRegistry`. I did not live-run the full dashboard fanout.

## Solution Assessment

No; caching is a plausible fix direction, but config/env-only memoization is not correct for the current helper contract. The safer solution is to include the effective registry identity or skip caching when it cannot be keyed.

  - body: `applyPluginAutoEnable` still accepts `manifestRegistry`, and current main forwards it into both detection and materialization.   - confidence: 0.93

  - body: This test compares seven cached calls against one uncached call using `performance.now()`. The uncached `{}` path is small enough that CI scheduling or clock noise can invert the comparison even when caching works; assert recomputation behavior through an observable counter or registry-varying case instead of wall-clock speed.
  - confidence: 0.78
