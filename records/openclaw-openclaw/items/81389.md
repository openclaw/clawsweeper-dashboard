## Summary

Keep this PR open: the latest revision fixes the explicit `manifestRegistry` cache miss from the prior review, but calls that omit `manifestRegistry` are still cached under `undefined` even though current code resolves a mutable manifest registry behind the scenes. The PR also still lacks the promised real gateway/dashboard fanout proof.

## What This Changes

The PR adds a WeakMap cache around `applyPluginAutoEnable` keyed by config, env, and explicit manifest registry identity, plus cache behavior and timing tests.

## Best Possible Solution

Resolve the manifest registry before cache lookup and key by that resolved object, or cache only when an explicit registry is supplied; then add current-snapshot rotation coverage and redacted gateway fanout proof.

## Reproduction Assessment

Yes for the PR regression: source inspection gives a high-confidence path by calling the patched helper twice with the same config/env and no explicit registry while replacing the current metadata snapshot between calls. I did not live-reproduce the full dashboard fanout.

## Solution Assessment

No; caching is a reasonable direction, but caching unresolved registry lookups under `undefined` is not the narrowest correct fix. The safer implementation includes the resolved registry identity or skips cache use when that identity cannot be keyed.
