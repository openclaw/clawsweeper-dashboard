- `merge-risk: 🚨 compatibility`: The cache changes config resolution behavior and can preserve stale plugin auto-enable results across plugin metadata snapshot changes.

## Summary

Keep open: the PR partially addressed the earlier explicit-registry cache-key finding, but omitted-registry calls are still cached under `undefined` even though current main resolves registry state from mutable plugin metadata snapshots. The PR also still has only unit/timing output rather than real gateway/dashboard fanout proof.

## What This Changes

Adds an in-process cache to `applyPluginAutoEnable` keyed by config, env, and explicit `manifestRegistry` identity, plus cache behavior and timing tests.

## Best Possible Solution

Resolve the manifest registry before cache lookup and key by the resolved registry identity, or cache only when an explicit registry is supplied; then add snapshot-rotation coverage and real fanout proof.

## Reproduction Assessment

Yes for the PR regression: source inspection shows same config/env omitted-registry calls are cached under `undefined` even though registry identity can resolve from mutable current snapshot state. I did not live-reproduce the full dashboard fanout.

## Solution Assessment

No; caching is reasonable, but keying omitted-registry calls on `params.manifestRegistry` is not the narrowest correct fix. The safer path is to key by resolved registry identity or skip caching when no explicit registry is available.

Use deterministic call-count or registry-load coverage instead.
