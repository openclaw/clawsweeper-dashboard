## Summary

Keep open: the proposed memoization omits part of the helper input from the cache identity, so it can return stale output when explicit manifest registries vary, and the supplied after-fix proof is still only a Vitest transcript rather than real gateway/dashboard behavior.

## What This Changes

The branch adds a WeakMap cache around `applyPluginAutoEnable` keyed by config/env identity and adds focused cache behavior and timing tests.

## Best Possible Solution

Keep the cache optimization, but make the memoization identity cover `manifestRegistry` or bypass caching when an explicit registry is supplied, then prove the latency path with redacted real gateway/dashboard fanout evidence.

## Reproduction Assessment

Yes for the review finding: source inspection gives a high-confidence path by calling `applyPluginAutoEnable` twice with the same config/env objects and different explicit `manifestRegistry` objects. The full dashboard latency claim has reporter measurements but was not live-reproduced in this read-only review.

## Solution Assessment

No; caching this helper is a reasonable direction, but keying only by config/env is not the narrowest correct implementation because the current contract also depends on the manifest registry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Key the cache by manifest registry too:** `src/config/plugin-auto-enable.apply.ts:52-55`
  - body: `applyPluginAutoEnable` still accepts `manifestRegistry`, and detection/materialization use it to decide plugin ownership and materialized output. This cache returns a hit using only `config` and `env`, so a later call with the same objects but a different explicit registry can reuse results from the wrong plugin set. Include the registry identity in the key, or bypass this cache when an explicit registry is supplied.
