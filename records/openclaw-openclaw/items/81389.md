## Summary

Keep open: the performance direction is plausible, but the current PR head still caches only by config/env even though the helper output also depends on the manifest registry, and the supplied proof is unit-test-only rather than the reported gateway fanout behavior.

## What This Changes

The PR adds a WeakMap memoization layer to `applyPluginAutoEnable` keyed by config/env object identity and adds focused cache behavior/timing tests.

## Best Possible Solution

Keep the cache optimization, but make the memoization identity include the resolved manifest registry or bypass caching for explicit registry calls, replace the timing assertion with deterministic cache coverage, and then add redacted runtime fanout proof.

## Reproduction Assessment

Yes for the PR-introduced correctness bug: source inspection gives a high-confidence path by calling `applyPluginAutoEnable` twice with the same config/env objects but different explicit manifest registries. The linked dashboard latency report itself was not live-reproduced in this read-only review.

## Solution Assessment

No; caching this helper is a reasonable optimization, but keying only by config/env is not the narrowest correct solution because current behavior also depends on the manifest registry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Key cache hits by manifest registry:** `src/config/plugin-auto-enable.apply.ts:55-57`
  - body: `applyPluginAutoEnable` still accepts `manifestRegistry`, and both detection and materialization use it to decide plugin ownership and registration. With this cache, the first result for a config/env pair wins even if a later call supplies a different registry, so callers can get stale auto-enable output for the wrong plugin set. Include the registry in the memoization identity, or skip this cache when an explicit registry is supplied unless that resolved registry identity is part of the key.
  - confidence: 0.91
- **[P2] Replace timing assertion with deterministic proof:** `src/config/plugin-auto-enable.apply.test.ts:55-71`
  - body: This test compares seven cached calls against one uncached call for an empty config, where the uncached path is already a fast no-op and wall-clock timing can flip under scheduler, JIT, or machine noise. Assert cache behavior with deterministic instrumentation or a registry/load-count proof instead of a relative timing threshold.
  - confidence: 0.78
