## Summary

This PR should stay open: the performance goal is valid, but the proposed cache key is too narrow for the current helper contract, and the supplied proof only exercises unit-level behavior rather than the reported dashboard fanout.

## What This Changes

The PR adds a WeakMap cache around `applyPluginAutoEnable` keyed by `config` and `env` object identity and adds focused cache behavior tests.

## Best Possible Solution

Keep the cache optimization, but make the memoization identity match the helper contract and add a regression test that varies `manifestRegistry` for the same config/env references.

## Reproduction Assessment

Yes for the review finding: source inspection gives a high-confidence reproduction path by calling `applyPluginAutoEnable` twice with the same `config` and `env` objects but different explicit `manifestRegistry` objects. The linked dashboard performance issue itself was not reproduced live in this read-only review.

## Solution Assessment

No; caching this helper is a reasonable direction, but keying only by config/env is not the narrowest correct solution because current behavior also depends on the manifest registry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Key the cache by manifest registry too:** `src/config/plugin-auto-enable.apply.ts:52`
  - body: `applyPluginAutoEnable` still accepts `manifestRegistry`, and both detection and materialization use it to decide which plugin owns configured channels/providers and how entries are registered. With this cache, the first call for a given `config`/`env` pair wins even if a later call passes a different registry, so callers can get stale auto-enable output for the wrong plugin set. Include the explicit registry in the memoization identity, or bypass this cache when `manifestRegistry` is supplied unless the resolved registry identity is part of the key.
  - confidence: 0.91
