## Summary

Keep open: the performance problem is plausible and the PR is focused, but the current patch is not mergeable because the cache can return a result computed with a different manifest registry, and the supplied proof is unit-test-only.

## What This Changes

The PR adds a module-level WeakMap cache around `applyPluginAutoEnable` keyed by config/env object identity and adds a new Vitest file for cache identity and timing assertions.

## Best Possible Solution

Keep the focused performance direction, but make the cache preserve every semantic input or scope it to immutable runtime snapshots, then replace wall-clock timing assertions with deterministic cache coverage and real dashboard fanout proof.

## Reproduction Assessment

Yes, there is a credible source-backed reproduction path: the linked issue describes a dashboard fanout with repeated helper calls, and current main has no cache in the helper. I did not run the dashboard or instrument the gateway in this read-only review.

## Solution Assessment

No, not as written. The cache target is reasonable, but the implementation drops `manifestRegistry` from the cache identity even though current source and tests show it changes the result.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Include the manifest registry in the cache identity:** `src/config/plugin-auto-enable.apply.ts:52-65`
  - body: `applyPluginAutoEnable`'s result depends on `manifestRegistry`: current code uses it to resolve configured channel/provider ownership, and existing tests expect a supplied registry to change which plugin is enabled. With this two-level cache, a second call using the same `config` and `env` but a different registry returns the first result, so callers can silently skip or reuse the wrong plugin auto-enable decision. Include the registry in the key, or bypass this cache when a registry is supplied.
  - confidence: 0.93
- **[P3] Replace the timing assertion with deterministic cache proof:** `src/config/plugin-auto-enable.apply.test.ts:55-71`
  - body: This test compares seven cached calls against one uncached call using `performance.now()`. That is not a stable CI contract, especially when the uncached path is fast or the runner is noisy; it can fail even when the cache works. Prefer a deterministic assertion such as spying on the expensive helper path or varying inputs to prove hit/miss behavior.
  - confidence: 0.82
