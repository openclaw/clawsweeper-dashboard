## Summary

Keep open: the latest PR head still has a correctness blocker because the proposed cache ignores `manifestRegistry`, and the supplied proof remains a unit-test transcript rather than real gateway/dashboard fanout proof.

## What This Changes

The PR adds a WeakMap cache around `applyPluginAutoEnable` keyed by `config` and `env` identity, plus focused cache behavior and timing tests.

## Best Possible Solution

Keep the cache optimization only after the memoization identity includes the resolved manifest registry, or bypasses caching when that identity is not part of the key, with a registry-varying regression test.

## Reproduction Assessment

Yes for the review finding: source inspection gives a high-confidence path by calling `applyPluginAutoEnable` twice with the same config/env references and different `manifestRegistry` objects. The linked dashboard latency issue has reporter measurements, but I did not live-reproduce the full fanout in this read-only review.

## Solution Assessment

No; caching this helper is a reasonable direction, but config/env-only memoization is not the narrowest correct solution because the current helper contract also depends on `manifestRegistry`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Key the cache by manifest registry too:** `src/config/plugin-auto-enable.apply.ts:52-57`
  - body: `applyPluginAutoEnable` still accepts `manifestRegistry`, and both detection and materialization use it to decide which plugin owns configured channels/providers and how entries are registered. With this cache, the first call for a given `config`/`env` pair wins even if a later call passes a different registry, so callers can get stale auto-enable output for the wrong plugin set. Include the explicit or resolved registry identity in the key, or bypass caching when that identity is unavailable.
  - confidence: 0.91
