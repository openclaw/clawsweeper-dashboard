## Summary

Keep open: the latest PR head still has the correctness blocker from the earlier review because the cache ignores `manifestRegistry`, even though current main uses it for candidate detection and materialization. The PR also still supplies only Vitest/timing proof, so real gateway/dashboard fanout evidence is missing before merge.

## What This Changes

The branch adds an identity-based WeakMap cache around `applyPluginAutoEnable` plus cache behavior and timing tests.

## Best Possible Solution

Keep the optimization only after the cache identity includes the explicit or resolved manifest registry, or bypasses caching when that identity is not part of the key, with registry-varying regression coverage and redacted gateway/dashboard fanout proof.

## Reproduction Assessment

Yes for the blocking review finding: source inspection gives a high-confidence path by calling the patched helper twice with the same config/env references and different `manifestRegistry` objects, such as an empty registry followed by one declaring the `brave` web-search provider. I did not live-reproduce the full dashboard fanout.

## Solution Assessment

No; caching is a reasonable direction, but config/env-only memoization is not the narrowest correct solution because the helper contract also depends on `manifestRegistry`. The safer fix is to include the registry identity or skip caching when it cannot be keyed correctly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Key the cache by manifest registry too:** `src/config/plugin-auto-enable.apply.ts:52-57`
  - body: `applyPluginAutoEnable` still accepts `manifestRegistry`, and both detection and materialization use it to decide which plugin owns configured channels/providers and how entries are registered. This cache returns the first result for a given `config`/`env` pair even when a later call passes a different registry, so callers can get stale auto-enable output for the wrong plugin set. Include the explicit or resolved registry identity in the key, or bypass caching when that identity is unavailable.
  - confidence: 0.91
