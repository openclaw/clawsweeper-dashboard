## Summary

Keep open: the performance goal is plausible, but the PR head is not safe because the cache key ignores `manifestRegistry`, and the supplied proof is still unit-test-only rather than real gateway/dashboard fanout proof.

## What This Changes

The PR adds a WeakMap memoization layer around `applyPluginAutoEnable` keyed by config/env identity and adds focused cache behavior and timing tests.

## Best Possible Solution

Keep the cache optimization only after memoization includes the resolved manifest registry identity, or bypasses caching when that identity is unavailable, with registry-varying regression coverage and real runtime fanout proof.

## Reproduction Assessment

Yes for the review finding: source inspection gives a high-confidence path by calling `applyPluginAutoEnable` twice with the same config/env references and different explicit `manifestRegistry` objects. The linked dashboard latency issue has reporter measurements, but I did not live-reproduce the full fanout in this read-only review.

## Solution Assessment

No; caching this helper is a reasonable direction, but config/env-only memoization is not the narrowest correct solution because `manifestRegistry` is part of the current helper contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Key the cache by manifest registry too:** `src/config/plugin-auto-enable.apply.ts:52-57`
  - body: `applyPluginAutoEnable` output still depends on `manifestRegistry`: detection uses it to map configured channels/providers/model refs to plugin ids, and materialization uses it to register plugin entries and labels. This cache returns the first result for a config/env pair before inspecting a later registry, so the same config/env with a different explicit registry can get stale auto-enable output. Include the resolved registry identity in the key, or skip caching when that identity is not part of the key.
  - confidence: 0.91
