## Summary

Keep open: the latest branch partially fixes the explicit-registry cache key, but omitted-registry calls are still cached under `undefined` even though current `main` resolves them through mutable plugin metadata snapshots, and the proof is still Vitest-only rather than a real gateway/dashboard fanout run.

## What This Changes

The PR adds a WeakMap cache for `applyPluginAutoEnable` keyed by config/env/manifestRegistry identity and adds cache behavior plus timing tests.

## Best Possible Solution

Resolve the manifest registry before cache lookup and key by that resolved object, or skip caching when no explicit registry is supplied; add deterministic snapshot-rotation coverage and redacted gateway fanout proof.

## Reproduction Assessment

Yes for the PR regression: source inspection shows omitted-registry calls share the `undefined` cache key while current `main` resolves omitted registries from mutable plugin metadata snapshots. I did not run the full dashboard fanout.

## Solution Assessment

No; caching is a reasonable direction, but caching unresolved registry lookups is not the narrowest correct fix. The safer solution keys on resolved registry identity or avoids caching when that identity is not available.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Key omitted-registry cache entries by the resolved registry:** `src/config/plugin-auto-enable.apply.ts:56`
  - body: This uses `params.manifestRegistry` as the cache key, so calls without an explicit registry all share `undefined`. Current code resolves omitted registries from current or loaded plugin metadata snapshots, and gateway reloads can replace that snapshot; with the same config/env objects, this can return stale auto-enable output for the wrong plugin set. Resolve the registry before lookup and key on it, or skip caching omitted-registry calls.
  - confidence: 0.91
- **[P2] Use deterministic cache proof instead of timing:** `src/config/plugin-auto-enable.apply.test.ts:100`
  - body: This assertion compares seven cached calls with one empty-config uncached call, which is dominated by timer and scheduler noise rather than the reported expensive path. Replace it with deterministic coverage such as call-count, registry-load, or snapshot-rotation assertions so the test proves cache behavior without flaking.
  - confidence: 0.82
