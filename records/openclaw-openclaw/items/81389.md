## Summary

Keep open: the cache is still keyed only by config/env even though the current helper result also depends on manifestRegistry, so this PR can return stale auto-enable output for a different registry. The supplied proof is still Vitest/timing output rather than a real gateway/dashboard fanout run.

## What This Changes

The branch adds WeakMap memoization around applyPluginAutoEnable keyed by config/env object identity and adds cache behavior/timing tests.

## Best Possible Solution

Make the cache registry-aware, or bypass caching when registry identity cannot be keyed safely; add same-config/env different-registry regression coverage and then provide redacted real gateway/dashboard fanout proof.

## Reproduction Assessment

Yes for the blocking PR regression: source inspection shows a cache hit is returned before manifestRegistry participates, so two calls with the same config/env and different registries can get stale results. I did not live-run the full dashboard fanout.

## Solution Assessment

No; caching is a reasonable direction, but config/env-only memoization is not the narrowest correct solution because the helper contract also depends on the explicit or resolved registry identity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Key the cache by manifest registry too:** `src/config/plugin-auto-enable.apply.ts:52-57`
  - body: applyPluginAutoEnable still accepts manifestRegistry, and both detection and materialization use it to decide plugin ownership and registration. This cache returns the first result for a config/env pair even when a later call passes a different registry, so callers can receive auto-enable output for the wrong plugin set. Include the explicit or resolved registry identity in the key, or bypass caching when it cannot be keyed correctly.
  - confidence: 0.93
- **[P3] Remove timing from the cache behavior test:** `src/config/plugin-auto-enable.apply.test.ts:55-71`
  - body: This assertion compares seven cached calls with one uncached call on an empty config, where both paths are tiny and performance.now/JIT noise can flip the result. That makes the test flaky without proving the real fanout win; assert observable cache behavior and registry-varying coverage instead of wall-clock timing.
  - confidence: 0.8
