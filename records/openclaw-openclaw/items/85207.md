## Summary

Keep open: current main and v2026.6.8 include Codex ACP stderr-tail diagnostic enrichment and plugin-version drift diagnostics, but the requested bounded retry for generic/client-closed ACPX prompt failures before side effects is still missing. The issue remains tied to an operator-visible policy decision for stale active user-level `@openclaw/acpx` installs.

## What This Changes

The issue requests active ACPX/Codex runtime hardening so stale external ACPX installs are detected or repaired and transient pre-side-effect ACP prompt failures retry before surfacing bare `ACP_TURN_FAILED`.

## Best Possible Solution

Keep the diagnostic enrichment already on main, then add a maintainer-approved stale-ACPX compatibility path plus a bounded retry for transient Codex ACP prompt failures only before assistant output, tool calls, or session-side effects.

## Reproduction Assessment

Source-level yes for the remaining retry gap: current main can be inspected or mocked so a generic Codex ACP `Internal error` before output is enriched and returned without a bounded retry. Live reproduction of the original stale active plugin deployment was not run in this read-only review.

## Solution Assessment

Partly yes, but not fully decided: the retry belongs at the ACPX/manager pre-side-effect boundary, while stale active plugin handling needs a maintainer policy decision before code repair changes startup or update behavior.
