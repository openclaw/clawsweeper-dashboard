## Summary

Keep open: current main and the latest `v2026.6.6` release include ACPX Codex diagnostic enrichment and some plugin-version/update diagnostics, but the bounded pre-side-effect retry for generic/client-closed Codex ACP prompt failures is still not implemented. The related open consolidation issue confirms this remains part of the installed-runtime stability cluster rather than a cleanup-close candidate.

## What This Changes

The issue asks OpenClaw to prevent older active ACPX plugin runtimes from surfacing bare `ACP_TURN_FAILED` errors by checking plugin compatibility, carrying Codex ACP diagnostics into the maintained runtime, and retrying transient prompt failures before side effects.

## Best Possible Solution

Keep the shipped diagnostic enrichment, then add a maintainer-approved stale-ACPX compatibility path and a bounded retry only before any Codex ACP prompt side effects, with coverage proving no retry after output/tool/session effects.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a live current-main run; source inspection shows the requested generic/client-closed pre-side-effect retry is still absent, while the original failure depends on an installed gateway using an older active ACPX plugin.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the plugin upgrade/startup policy; the retry belongs near the ACPX/ACP manager boundary, but the compatibility behavior for intentionally pinned plugins is a product decision.
