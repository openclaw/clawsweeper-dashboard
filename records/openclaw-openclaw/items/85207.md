## Summary

Keep open: current main already has Codex ACP stderr-tail diagnostic enrichment in the ACPX plugin, but the remaining reported upgrade/compatibility path and bounded pre-side-effect retry are not fully implemented or proven obsolete. The open related issue https://github.com/openclaw/openclaw/issues/83959 covers Codex app-server startup retry exhaustion, while this issue is distinct because an older active `@openclaw/acpx` plugin can miss the runtime hardening shipped with OpenClaw.

## What This Changes

The issue asks OpenClaw to prevent older active `@openclaw/acpx` installs from shadowing bundled ACP/Codex hardening and to retry transient pre-side-effect Codex ACP turn failures with redacted diagnostics.

## Best Possible Solution

Add a narrow ACPX/plugin compatibility check for stale active `@openclaw/acpx` installs and a bounded Codex ACP retry only before any prompt-side effects, reusing the current redacted stderr-tail diagnostics and covering the no-retry-after-output/tool cases.

## Reproduction Assessment

No, not as a live current-main reproduction. It is source-reproducible that current main enriches generic Codex ACP diagnostics but lacks the requested bounded retry for generic/client-closed pre-side-effect failures, and installed plugin records can select a user-level plugin over the bundled one.

## Solution Assessment

Unclear. The diagnostic enrichment already belongs in ACPX runtime, but maintainers should choose the startup/update policy for older active `@openclaw/acpx` installs and the exact replay-safe retry predicate before implementation.
