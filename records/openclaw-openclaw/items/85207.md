## Summary

Keep open: current main and v2026.6.8 include ACPX stderr-tail diagnostic enrichment and plugin-version drift status warnings, but the requested bounded retry for generic/client-closed ACPX prompt failures before side effects is still missing, and stale active ACPX install handling still needs a maintainer policy decision.

## What This Changes

The issue asks OpenClaw to harden the active ACPX/Codex runtime path so stale user-level ACPX installs are detected or repaired, transient pre-side-effect ACP prompt failures get bounded retries, and remaining failures include sanitized diagnostics.

## Best Possible Solution

Keep the diagnostic enrichment and drift warning already on main, then add a maintainer-approved stale-ACPX compatibility path plus bounded retry for transient Codex ACP prompt failures only before assistant output, tool calls, or session-side effects.

## Reproduction Assessment

Yes at source level: current main can be inspected or mocked so a generic Codex ACP `Internal error` before output is enriched and returned without a bounded retry. I did not run a live reproduction of the original stale active plugin deployment.

## Solution Assessment

Partly no: the current diagnostic and drift-warning pieces are good, but they do not complete the requested fix. The best path still needs a bounded pre-side-effect retry and a maintainer-approved stale active plugin policy.
