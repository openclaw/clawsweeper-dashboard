## Summary

Keep open: the PR targets a real gateway/browser reload bug, but the proposed no-op rule would bypass the hot-reload path that refreshes runtime config snapshots for external browser-profile edits.

## What This Changes

Adds a `browser.profiles` no-op reload rule and a gateway reload-plan unit test so browser profile changes no longer mark the gateway for restart.

## Best Possible Solution

Classify `browser.profiles.*` as hot/no-restart so `config.patch` avoids SIGUSR1 while the gateway reloader still refreshes runtime config snapshots; keep `browser.enabled` and broader browser settings restart-backed.

## Reproduction Assessment

Yes. Source inspection shows current main routes `browser.profiles.*` through the browser plugin's broad restart prefix, while the PR's no-op rule would make the shared reloader skip the hot reload path.

## Solution Assessment

No. Avoiding SIGUSR1 for browser profile patches is the right goal, but `kind: "none"` is not the best fix because it bypasses runtime config refresh; a hot/no-restart rule is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep browser profile edits on the hot reload path:** `src/gateway/config-reload-plan.ts:118`
  - body: This makes `browser.profiles.*` a no-op. For external config-file edits, `startGatewayConfigReloader` returns before `onHotReload` for no-op plans, so the runtime config snapshot is not refreshed. Browser profile resolution then reads from the stale runtime snapshot, meaning a manual `browser.profiles.<name>.cdpUrl` edit can silently fail to take effect until restart. Use a hot/no-restart classification instead so profile edits avoid SIGUSR1 but still refresh runtime config.
  - confidence: 0.9
