## Summary

Keep open: current main and v2026.6.8 still route accepted Telegram reactions only into the ephemeral system-event queue, with no durable/queryable feedback event store or reaction hook/API for reflection workflows.

## What This Changes

The issue requests durable, queryable persistence of accepted Telegram message reaction events for later reflection, analytics, and memory workflows.

## Best Possible Solution

Define one maintainer-approved durable reaction feedback contract, backed by SQLite/shared state and exposed through the appropriate plugin or channel API, then wire Telegram into it with docs and restart-proof tests.

## Reproduction Assessment

Yes. Source inspection shows an accepted Telegram `message_reaction` update only enqueues an in-memory system event, and `src/infra/system-events.ts` explicitly avoids persistence, so restart-proof/queryable feedback is absent.

## Solution Assessment

Unclear until maintainers choose the public contract. A Telegram-only JSONL journal is not the best fix; a typed durable feedback/event surface with SQLite-backed storage is the safer direction.
