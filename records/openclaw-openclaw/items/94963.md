## Summary

Keep open: current main and the latest release still surface accepted Telegram reactions only as in-memory system events, not as durable, queryable feedback records for reflection or memory workflows.

## What This Changes

Request to durably persist accepted Telegram `message_reaction` updates as queryable feedback events with enough metadata to map reactions back to assistant messages.

## Best Possible Solution

Define a SQLite-backed Telegram or shared channel feedback event surface after existing Telegram authorization and reaction filters, with retention policy, focused tests, docs, and a narrow reader for reflection or memory workflows.

## Reproduction Assessment

Yes, at source level. Current main accepts Telegram `message_reaction` updates and enqueues in-memory system events, while the documented and tested path has no durable feedback record or reader.

## Solution Assessment

Unclear until maintainers choose the storage/API boundary. The local JSONL workaround is not the right in-repo shape; repository policy points toward a SQLite-backed plugin or shared state design with tests and docs.
