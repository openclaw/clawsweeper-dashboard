## Summary

Keep open: current `main` and v2026.6.8 still route accepted Telegram reactions only into the ephemeral system-event queue, and the related open/closed items cover wake behavior, prompt guidance, or approval-only reactions rather than this issue's durable feedback persistence contract.

## What This Changes

The issue requests durable, queryable persistence of accepted Telegram `message_reaction` updates so reflection, analytics, or memory workflows can consume reaction feedback after restarts.

## Best Possible Solution

Define a maintainer-approved durable reaction feedback contract, preferably shared through the plugin/channel SDK and backed by SQLite/plugin state, then implement Telegram persistence/query support with focused docs and restart-proof tests.

## Reproduction Assessment

Yes, at source level: the current Telegram handler enqueues accepted added reactions, while the system-event queue explicitly avoids persistence and is drained from process memory. I did not run a live Telegram account in this read-only review.

## Solution Assessment

Unclear until maintainers choose the public contract. A JSONL journal should not be the core fix; a durable SQLite/plugin-state event store plus a documented reader or hook/API boundary is the safer direction.
