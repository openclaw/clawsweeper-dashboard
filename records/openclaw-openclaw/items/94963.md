## Summary

Keep open: current main still turns accepted Telegram reactions into ephemeral system events and restart-safety ingress processing, but does not persist accepted reaction feedback as durable, queryable events for later reflection, analytics, or memory workflows.

## What This Changes

The issue requests durable, queryable persistence for accepted Telegram message reaction feedback events.

## Best Possible Solution

Define a generic durable feedback/reaction event contract through the channel/plugin SDK boundary, then have Telegram persist accepted reaction events into SQLite-backed state with tests for restart retrieval.

## Reproduction Assessment

Yes, at source level. Current main accepts Telegram added reaction updates and routes them to an explicitly in-memory system-event queue; the successful spooled update path deletes the durable ingress row.

## Solution Assessment

Unclear until maintainers choose the storage/API boundary. The local JSONL workaround is not the best in-repo shape; repository policy points toward SQLite-backed plugin or shared state with a documented reader and retention policy.
