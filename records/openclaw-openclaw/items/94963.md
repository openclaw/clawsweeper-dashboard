## Summary

Keep open: current main still turns accepted Telegram reactions into ephemeral system events plus a restart-safety ingress queue, but it does not persist accepted reaction feedback as durable, queryable events for later reflection or memory workflows.

## What This Changes

Requests durable, queryable storage and read access for accepted Telegram `message_reaction` feedback events.

## Best Possible Solution

Define a SQLite-backed Telegram or shared channel feedback event surface after existing Telegram authorization/reaction filters, with retention/redaction policy, focused tests, docs, and a narrow reader for reflection or memory workflows.

## Reproduction Assessment

Yes, at source level. Current main accepts Telegram `message_reaction` updates and enqueues ephemeral system events, while successful ingress rows are deleted and no durable feedback reader is documented or wired.

## Solution Assessment

Unclear until maintainers choose the storage/API boundary. The JSONL workaround is not the best in-repo shape; repository policy points toward SQLite-backed plugin or shared state with tests and docs.
