## Summary

Keep open: current main and v2026.6.8 only route accepted Telegram reactions into ephemeral in-memory system events plus a restart-safety ingress queue; there is still no durable, queryable feedback event store or read API. The related reaction issues cover wake-up, prompt guidance, or stale structured-hook discussion, but they do not replace this narrower persistence request.

## What This Changes

Requested durable, queryable persistence for accepted Telegram message reaction updates so later reflection or analytics workflows can consume them.

## Best Possible Solution

Define a SQLite-backed Telegram or shared channel feedback event surface that stores accepted reaction add/change/remove metadata after authorization, exposes a narrow documented reader for reflection/memory, and avoids JSONL sidecars.

## Reproduction Assessment

Yes at source level. Current main accepts Telegram `message_reaction` updates and queues ephemeral system events, while handled ingress entries are deleted and no durable reaction-feedback query surface exists.

## Solution Assessment

Unclear until maintainers choose the public storage/API boundary. The reporter's JSONL workaround is not the best in-repo solution; repository policy points toward SQLite-backed plugin/shared state with docs and tests.
