## Label Justifications

- `P2`: The gap blocks durable reaction-feedback workflows for Telegram users, but it is an enhancement with bounded channel/session-state impact rather than an urgent runtime outage.
- `impact:session-state`: The request is about preserving reaction feedback across prompt turns and restarts so later reflection, memory, or analytics workflows can recover session-linked context.

## Summary

Keep open: current main still treats accepted Telegram reactions as transient system events and restart-retry ingress updates, not as durable, queryable feedback records for later reflection, analytics, or memory workflows. The related approval-reaction PR is adjacent because it handles approval resolution and preserves the generic reaction notification path rather than adding a durable feedback store.

## What This Changes

Request durable, queryable persistence for accepted Telegram reaction feedback events with enough metadata to map reactions back to assistant messages after restarts.

## Best Possible Solution

Define a maintainer-approved SQLite-backed durable reaction feedback/event model, with retention and access rules, and expose a documented reader path for reflection or memory workflows across reaction-capable channels where practical.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: Telegram `message_reaction` updates are accepted and enqueued as ephemeral system events, while the restart spool and ingress queue do not retain accepted reactions as queryable feedback after handling.

## Solution Assessment

Unclear as a concrete implementation: durable persistence is the right product outcome, but the best fix needs maintainer choice of shared event store, plugin-owned state, hook category, retention policy, and reader API before a safe PR can be queued.
