## Summary

Keep open: current main and v2026.6.8 still route accepted Telegram reactions into the ephemeral system-event queue, while the requested durable/queryable feedback contract is not implemented and needs a maintainer boundary decision.

## What This Changes

The issue requests durable, queryable persistence for accepted Telegram message reaction feedback events so later reflection, analytics, or memory workflows can read them after restart.

## Best Possible Solution

Pick one canonical durable feedback contract, preferably a plugin/API-accessible SQLite-backed event path, then adapt Telegram and sibling reaction sources to emit into that contract without JSONL sidecars.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: accepted Telegram reactions currently enqueue in the in-memory system-event queue, whose source explicitly avoids persistence; I did not run a live Telegram gateway restart test in this read-only review.

## Solution Assessment

Unclear. Durable persistence is the right problem, but the best implementation needs maintainer choice between a durable event store, a plugin SDK seam, internal hooks, or a cross-channel reaction feedback contract.
