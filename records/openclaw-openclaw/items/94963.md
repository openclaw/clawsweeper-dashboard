## Summary

Keep open: current main and v2026.6.9 still route accepted Telegram reaction additions only into the ephemeral system-event queue, so the durable/queryable feedback capability remains missing and needs a maintainer API/storage decision.

## What This Changes

The issue requests durable, queryable persistence for accepted Telegram message reaction events so reflection, analytics, or memory workflows can use them after restarts.

## Best Possible Solution

Define one maintainer-approved durable reaction feedback contract, backed by SQLite/shared state and exposed through the appropriate plugin or channel API, then wire Telegram into it with docs and restart-proof tests.

## Reproduction Assessment

Yes for source reproduction: the accepted Telegram reaction path enqueues only an in-memory system event, and that queue explicitly avoids persistence. I did not run a live Telegram restart test in this read-only review.

## Solution Assessment

Partly. Durable persistence is the right end state, but the exact public contract needs maintainer direction; a Telegram-only JSONL journal is not the best maintainable fix.
