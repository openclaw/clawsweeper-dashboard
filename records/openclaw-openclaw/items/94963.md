## Summary

Keep open: current main and v2026.6.9 still accept Telegram reactions only as ephemeral system events, with no durable/queryable reaction feedback store or reaction hook/API for reflection workflows. The related approval-reaction PR is adjacent and does not cover generic persisted reaction feedback.

## What This Changes

Request to durably persist accepted Telegram `message_reaction` updates as queryable feedback events that survive restarts and can be consumed by reflection, analytics, or memory workflows.

## Best Possible Solution

Define a maintainer-approved SQLite-backed channel reaction feedback contract, expose it through the appropriate plugin/hook/query API, then wire Telegram into it with restart-proof tests and docs.

## Reproduction Assessment

Yes. Source inspection shows accepted Telegram reactions are enqueued only into the in-memory system-event queue, and the latest release has the same behavior, so restart-proof/queryable feedback is absent.

## Solution Assessment

Unclear until maintainers choose the public contract. A local JSONL journal is not the best fix; the maintainable direction is a typed SQLite-backed feedback/event surface that respects the plugin boundary.
