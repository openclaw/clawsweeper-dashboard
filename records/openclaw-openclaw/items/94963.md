## Summary

Keep open: current main and v2026.6.10 still route accepted Telegram reactions only into ephemeral system events, with no durable/queryable feedback event store or reaction hook/API for later reflection workflows.

## What This Changes

Request durable, queryable persistence for accepted Telegram message_reaction updates so agents can use reaction feedback after restarts and upgrades.

## Best Possible Solution

Define one maintainer-approved SQLite-backed reaction feedback/event contract, expose it through the appropriate plugin/channel API, then wire Telegram into it with docs and restart-proof tests.

## Reproduction Assessment

Yes, at source level. Current main accepts Telegram message_reaction updates but only enqueues in-memory system events, and the latest release still has the same non-durable path.

## Solution Assessment

Unclear until maintainers choose the public contract. A local JSONL journal is not the right repository fix; a typed SQLite-backed feedback/event surface is the safer direction.
