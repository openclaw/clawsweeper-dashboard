## Summary

Keep open: current main and v2026.6.10 only enqueue accepted Telegram reactions into ephemeral system events, with no durable/queryable feedback event path or reaction hook/API for reflection workflows. The linked approval-reaction PR and broader observability issues are adjacent, not replacements for this narrower persistence request.

## What This Changes

Request durable, queryable persistence for accepted Telegram message reaction feedback events so reflection or analytics can consume them after restarts.

## Best Possible Solution

Define a maintainer-approved durable reaction feedback contract, then implement it through a canonical SQLite-backed feedback, diagnostic, or plugin API instead of a Telegram-only JSONL sidecar.

## Reproduction Assessment

Yes: source inspection shows current main accepts Telegram reaction updates into the system-event queue, and that queue is explicitly in-memory. I did not run a live Telegram restart repro in this read-only review.

## Solution Assessment

Unclear: durable reaction feedback is a valid target, but maintainers still need to choose whether it belongs in diagnostics, a plugin hook/event API, transcript persistence, or another SQLite-backed feedback store.
