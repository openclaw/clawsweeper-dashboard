## Summary

Keep open: current main still has the security-facing diagnostics and docs gap. CLI backends declare always-on native tools outside OpenClaw's tool catalog, while doctor and the relevant config docs still do not warn that sandbox, workspace-only filesystem, allow/deny, and session-tool settings only govern OpenClaw-managed tools in that runtime shape.

## What This Changes

The issue requests doctor warnings and documentation callouts for CLI runtimes where sandbox, workspace-only filesystem policy, tool allow/deny, and sessions_send settings do not constrain native CLI tools.

## Best Possible Solution

Add a security-reviewed doctor/audit warning plus docs callouts that clearly distinguish OpenClaw-managed tools from native CLI tools, without trying to make OpenClaw intercept CLI-native filesystem and shell operations in this issue.

## Reproduction Assessment

Yes for the diagnostics gap: current source shows CLI native-tool mode and the doctor path lacks a matching warning, and current docs lack the requested CLI caveat. I did not run the reporter's live claude-cli cross-workspace setup.

## Solution Assessment

Yes, the requested warning and documentation path is the narrow maintainable solution. Enforcing these settings against native CLI tools would be a larger runtime interception feature and should not be bundled into this diagnostics issue.
