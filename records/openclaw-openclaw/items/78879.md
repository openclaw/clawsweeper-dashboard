## Summary

Keep open: current main partially mitigates runtime `toolsAllow` by failing closed for CLI backends, but it still does not provide the requested proactive doctor warning or targeted docs callouts for CLI-native tool/sandbox expectations.

## What This Changes

The issue asks for doctor warnings and docs callouts when CLI runtimes make sandbox, filesystem, tools allow/deny, and sessions_send-related config ineffective.

## Best Possible Solution

Add a narrow doctor/--lint diagnostic plus docs callouts for configured CLI runtimes with explicit sandbox/fs/tool/session policy expectations, and keep the runtime fail-closed behavior for unenforceable `toolsAllow`.

## Reproduction Assessment

Yes, by source inspection: configure a provider/model CLI runtime with explicit sandbox/fs/tool/session policy and current doctor only has generic security collectors, not a CLI-runtime no-op diagnostic. I did not execute a live doctor run in this read-only review.

## Solution Assessment

Unclear until maintainer/security scoping: a warning-and-docs path is likely the right level, but the exact affected runtimes and wording need an owner decision so OpenClaw does not overpromise isolation.
