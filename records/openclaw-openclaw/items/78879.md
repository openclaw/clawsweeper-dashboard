## Summary

Keep open: current main only partially mitigates CLI tool-policy mismatch by failing closed for runtime toolsAllow, but the requested doctor warning, user-facing docs callouts, and clearer allowlist explanation are still missing for CLI-native runtimes.

## What This Changes

Request doctor warnings, docs callouts, and clearer allowlist logging when CLI runtimes make sandbox, workspaceOnly, tool allow/deny, or sessions_send settings ineffective for native CLI tools.

## Best Possible Solution

Add a security-reviewed diagnostic keyed off selected CLI/nativeToolMode runtime plus explicit sandbox, fs, tool-policy, or session-tool settings, then document the PI-tool versus native-CLI boundary in the user docs and allowlist warning.

## Reproduction Assessment

Yes for the diagnostics gap by source inspection: current doctor/security code does not inspect CLI native-tool runtime selection, and the user docs still describe sandbox/workspace/session tool behavior without a CLI-runtime caveat. I did not run the live host filesystem reproduction.

## Solution Assessment

Yes, the requested warning and documentation path is the narrow maintainable direction. Enforcing these settings against native CLI tools would require a much larger runtime interception feature.
