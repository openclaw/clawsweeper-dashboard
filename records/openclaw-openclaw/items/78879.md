## Summary

Keep open: current main only partially mitigates the runtime/tool-policy mismatch by failing closed for runtime `toolsAllow` and warning about generic exec/filesystem drift, but it still lacks the requested CLI-native doctor warning, targeted docs callouts, and clearer allowlist explanation.

## What This Changes

Request warning and documentation updates when CLI-native runtimes make OpenClaw sandbox, filesystem, tool allow/deny, and session-tool settings ineffective.

## Best Possible Solution

Add a security-reviewed doctor diagnostic keyed off the selected model/provider CLI backend and `nativeToolMode: "always-on"`, then document the PI-tool versus CLI-native boundary in sandboxing, agent config, session-tool docs, and the allowlist warning path.

## Reproduction Assessment

Yes for the diagnostics gap by source inspection: current doctor/security code has no CLI-native runtime mismatch check, and the relevant user docs still describe sandbox/workspace/session-tool behavior without that caveat. I did not run the live host filesystem reproduction.

## Solution Assessment

Yes, the warning and documentation approach is the narrow maintainable direction; enforcing OpenClaw sandbox/tool policy against native CLI tools would require a much larger runtime interception feature. The implementation should target the current resolved runtime policy rather than legacy whole-agent runtime pins.
