## Summary

Keep open: current main only partially handles CLI-native tool-policy mismatch by failing closed for one runtime allowlist path, while the requested doctor warning, docs caveats, and clearer allowlist explanation are still missing for CLI runtimes.

## What This Changes

The issue asks for doctor warnings, docs callouts, and clearer tool-policy messaging when CLI runtimes make OpenClaw sandbox/workspace/session-tool settings ineffective.

## Best Possible Solution

Add a security-reviewed doctor diagnostic keyed off CLI nativeToolMode plus explicit sandbox, fs, tool-policy, or session-tool settings, then document the PI-tool versus native-CLI boundary in the user docs and allowlist warning.

## Reproduction Assessment

Yes for the diagnostics/docs gap by source inspection: current doctor-security does not check CLI-native runtime selection, and the relevant docs still describe sandbox/workspace/session tools without a CLI-runtime caveat. I did not run the reporter's live cross-workspace filesystem scenario.

## Solution Assessment

Yes, the requested warning and documentation path is the narrow maintainable direction. Enforcing OpenClaw sandbox and PI-tool policy against native CLI tools would require a much larger runtime interception feature.
