## Summary

Keep open: current main still has the security-facing diagnostics/docs gap. CLI backends declare always-on native tools, while doctor and the relevant docs do not clearly warn that these OpenClaw tool controls are PI-tool scoped.

## What This Changes

The issue asks doctor, docs, and the tool allowlist warning to explain that sandbox, workspace-only filesystem, tool allow/deny, and sessions_send settings do not constrain CLI-native tools for claude-cli, codex-cli, or gemini-cli.

## Best Possible Solution

Add a secops-reviewed doctor/audit warning, docs callouts, and clearer allowlist wording that distinguish OpenClaw PI tools from CLI-native tools without attempting to intercept CLI tool execution.

## Reproduction Assessment

Yes. Source inspection on current main shows always-on CLI-native tool capability and a doctor/docs path that does not warn about this configuration mismatch; no live host repro was run.

## Solution Assessment

Yes. A warning and documentation path is the narrow maintainable direction; enforcing these settings against native CLI tools would require a much larger runtime interception design.
