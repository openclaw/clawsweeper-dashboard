- `impact:message-loss`: The issue is about channel messages being routed to the wrong agent when no binding matches.

## Summary

Keep open. Current main and v2026.6.10 still use default-agent fallback when no route binding matches, and Feishu’s normal dispatch log does not distinguish that fallback from an intentionally bound route; the older Feishu main-agent issue only covered bound-route support, not missing-binding diagnostics.

## What This Changes

The issue requests warnings, validation, setup assistance, and hot-reload guidance when a configured channel account has no binding and dispatch falls back to the main agent.

## Best Possible Solution

Add generic diagnostics that make default-agent fallback explicit for configured-but-unbound channel accounts without changing established fallback routing semantics; treat convention-based routing and config-check CLI as follow-up product decisions.

## Reproduction Assessment

Yes from source for the central fallback path: current routing tests and `resolveAgentRoute` show no-match routes use the default agent. A full live Feishu run was not performed in this read-only review.

## Solution Assessment

Unclear for the full RFC; warning-only diagnostics are the best first fix, while fail-closed routing, convention-based binding, and a new validation CLI need maintainer product direction.
