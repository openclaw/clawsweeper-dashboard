## Summary

Keep open: current main and the latest release still route unmatched channel/account traffic to the default agent and Feishu still logs only the resolved session key, so the requested diagnostics remain missing and need maintainer scoping.

## What This Changes

The issue requests warnings or validation when a configured channel account has no matching binding and silently falls back to the main agent, plus setup and reload improvements.

## Best Possible Solution

Preserve the current default-agent fallback unless maintainers explicitly change that policy, and add scoped diagnostics that warn on suspicious unbound accounts, identify default fallback at dispatch, and explain when binding changes require restart.

## Reproduction Assessment

Yes, at source level: a channel/account with no matching binding falls through to `matchedBy: "default"`, and Feishu logs only the resolved session key. I did not run a live Feishu gateway repro in this read-only review.

## Solution Assessment

Unclear for the full RFC: dispatch/startup warnings and restart guidance are the narrowest maintainable first fix, while fail-closed routing, convention routing, auto-binding, and a new validation CLI need maintainer product decisions.
