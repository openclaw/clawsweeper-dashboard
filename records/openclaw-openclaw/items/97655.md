## Summary

Keep open: current main and the latest release still route an unmatched channel/account to the default agent, and Feishu logs only the resulting session key, so the requested warning/validation remains missing. The report is valid setup-diagnostics work, but the broader RFC needs maintainer scoping because the default-agent fallback is documented current behavior.

## What This Changes

Request warnings, validation, setup, and reload improvements when a configured channel account lacks a matching `bindings[]` route and falls back to the default agent.

## Best Possible Solution

Keep the default-agent fallback contract for compatibility, but add scoped diagnostics: warn when a configured non-default account appears unbound, make dispatch logs identify default fallback, and make `bindings` config changes clearly say whether restart is required; leave convention-based routing for a separate maintainer decision.

## Reproduction Assessment

Yes at source level: `resolveAgentRoute` returns the default agent when no binding matches, and Feishu dispatches/logs the resulting `route.sessionKey`. I did not run a live Feishu bot, so this is source-reproducible rather than an executed live repro.

## Solution Assessment

Unclear until maintainer scoping: explicit warnings and doctor/reload guidance are the likely best first fix, while fail-closed behavior or name-convention routing would be a larger product change. The safest path is a bounded diagnostics improvement that preserves documented fallback semantics.
