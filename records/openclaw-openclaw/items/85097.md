## Summary

Keep open: current main and v2026.6.8 still reject direct thread-session `sessions_send` targets, and the parent-session Mattermost relay/result path still does not expose the root/thread/account/post payload details needed to debug the reported `NO_REPLY` outcome. Related Mattermost PRs cover adjacent root and thread-routing slices, but none fully owns this session-relay debuggability request.

## What This Changes

The issue asks for Mattermost threaded `sessions_send` failures to either deliver reliably from the parent session or expose structured delivery/root-thread diagnostics instead of ending opaquely as `NO_REPLY`.

## Best Possible Solution

Preserve the parent-session coordination contract, then add Mattermost parent-relay delivery diagnostics or result fields and fail explicitly when an intended visible threaded delivery produces no reply.

## Reproduction Assessment

Partially. Source inspection reproduces the direct thread-session rejection and the missing result/debug fields on current main, but the live Mattermost `NO_REPLY` outcome still needs a real transport run.

## Solution Assessment

No complete solution is present on main. The maintainable direction is a narrow parent-relay diagnostics/result repair, not simply allowing direct thread-session orchestration targets by default.
