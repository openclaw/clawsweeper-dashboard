## Summary

Keep open: current main and the latest release still reject direct thread-session `sessions_send` targets and still do not expose enough Mattermost relay/result details to debug an accepted parent-session send that produces no visible threaded reply. The merged and open Mattermost thread-root work covers narrower routing symptoms, not the broader `sessions_send` debuggability contract.

## What This Changes

The issue asks for Mattermost threaded reply delivery to be reliable or debuggable when direct thread-session targeting is rejected and parent-channel relay can end in `NO_REPLY` without payload details.

## Best Possible Solution

Keep the documented parent-session `sessions_send` contract, then add a narrow redacted Mattermost relay diagnostic or delivery-result contract and prove the accepted-but-no-visible-reply path with live Mattermost evidence.

## Reproduction Assessment

Partially. Source inspection on current main reproduces the direct thread-session rejection and missing structured debug/result fields, but the live Mattermost `NO_REPLY` outcome still needs a real transport reproduction.

## Solution Assessment

No complete solution is present on main. Reallowing direct thread-session targets would conflict with the documented A2A contract; the better solution is a redacted parent-relay delivery/result contract plus live proof.
