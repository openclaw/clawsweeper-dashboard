## Summary

Keep open: current main and the latest release still leave the Mattermost parent-session relay path too coarse to debug when a threaded visible reply does not appear. Adjacent Mattermost fixes cover final-delivery diagnostics and message-tool root selection, but they do not define the broader `sessions_send` relay-result/debug contract requested here.

## What This Changes

The issue requests a debuggable Mattermost threaded reply path for `sessions_send`, either through first-class thread delivery or redacted delivery details for parent-session relays.

## Best Possible Solution

Keep parent-session targeting as the documented `sessions_send` contract, then add a redacted Mattermost delivery-result or diagnostic surface that proves the effective root/thread/account route when a visible relay fails.

## Reproduction Assessment

Partially yes. Source inspection reproduces the direct thread-session rejection and missing structured debug/result fields, but the live Mattermost `NO_REPLY` outcome still needs a real transport reproduction.

## Solution Assessment

Unclear as a complete solution. Reallowing direct thread-session targets would conflict with the documented A2A contract; the maintainable path is a redacted parent-relay delivery/result contract plus live Mattermost proof.
