## Summary

Keep open: current main still intentionally rejects direct thread-session `sessions_send` targets, while the parent-session Mattermost announce path can bypass session delivery-context lookup and still lacks the effective root/thread diagnostic fields needed to explain a no-visible-reply result.

## What This Changes

The issue asks OpenClaw to make Mattermost `sessions_send` parent-channel thread replies either deliver reliably or expose structured root/thread delivery diagnostics when they do not.

## Best Possible Solution

Keep the direct thread-session guard, hydrate Mattermost parent-session announce targets from stored delivery context, and extend no-visible diagnostics with redacted effective thread/root/account/post-shape fields.

## Reproduction Assessment

No, not at high confidence: I did not run a live Mattermost relay, but current source reproduces the direct thread-session rejection and shows a clear parent-session path where Mattermost can skip thread/account hydration and under-report no-visible delivery details.

## Solution Assessment

Yes: the best fix is a narrow repair around Mattermost parent-session announce target hydration and richer delivery diagnostics, while preserving the documented direct thread-session rejection.
