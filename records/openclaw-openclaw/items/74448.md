## Summary

Keep open. Current main still has a credible premature-completion path: lifecycle `end` can make `agent.wait` return `ok`, and the subagent registry then freezes the latest visible child output without accounting for compaction `willRetry` state or a resumed terminal handoff.

## What This Changes

Ensure subagent task completion only fires after a compacted/resumed worker actually reaches its final terminal result.

## Best Possible Solution

Make compaction/retry nonterminal for subagent completion so the tracked task remains pending across compaction and announces exactly once from the resumed run's real terminal output.

## Reproduction Assessment

Yes. The issue gives concrete production reproductions, and a focused regression can emit compaction `willRetry` plus lifecycle `end` events and assert that the registry does not announce until the resumed terminal result is available.

## Solution Assessment

Yes. Fixing the lifecycle contract is the narrowest maintainable direction; extending only the output-capture retry window would still allow a successful completion event before the child task is actually done.

## Repair Work Prompt

Fix #74448 so subagent completion is not emitted when an embedded child run is only ending the pre-compaction attempt and will resume after compaction. Treat compaction `willRetry`/pending retry state as nonterminal for `agent.wait` and/or the subagent registry, then freeze and announce exactly once from the resumed run's real terminal output. Add a regression that simulates a child with a mid-task assistant reply, a compaction `willRetry` event, a lifecycle `end` for the old attempt, and a later terminal result; assert no parent completion is announced until the terminal result is available. Do not solve this by only increasing the `captureSubagentCompletionReply` polling window, and preserve `sessions_yield`, lifecycle error/timeout grace, orphan recovery, descendant announce deferral, cleanup retry, and existing compaction retry aggregate timeout behavior. Add a user-facing `CHANGELOG.md` fix entry for the active version without inventing bot or maintainer attribution.
