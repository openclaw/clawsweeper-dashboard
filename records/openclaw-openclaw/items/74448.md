## Label Justifications

- `impact:message-loss`: The actual final child handoff can be lost to the parent after an earlier stale completion event is delivered.
- `impact:session-state`: The bug mis-associates child run state across compaction/resume boundaries and freezes the wrong session output.

## Summary

Keep open. Current main still has the reported false-success path: compaction retry state is tracked separately, but a later lifecycle `end` can still become `agent.wait` success and subagent completion success before the resumed child has produced its real final handoff.

## What This Changes

The issue reports that subagent completion and `agent.wait` can resolve successfully at a compaction/retry boundary, freezing intermediate child output instead of the actual final result.

## Best Possible Solution

Carry compaction/resume continuation through lifecycle, `agent.wait`, and subagent registry completion so the parent receives exactly one success event from the resumed run's real terminal assistant output.

## Reproduction Assessment

Yes, at source level. The issue includes concrete production traces, and current main still maps an ordinary lifecycle `end` to `agent.wait` success and subagent completion without proving the compaction-resumed task actually finished.

## Solution Assessment

Yes. Fixing the lifecycle/wait contract is the maintainable repair; frozen-output polling or progress-text heuristics would still allow success to be emitted from the wrong attempt boundary.
