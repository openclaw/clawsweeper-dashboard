## Summary

Keep open. Current main still has a plausible premature subagent-completion path for compaction/resume: terminal `agent.wait` results are treated as completion, result text is frozen immediately from child history, and the later same-session refresh only helps while cleanup is still pending rather than after a successful premature announce.

## What This Changes

The issue reports that subagent compaction/resume can be treated as terminal completion, causing the parent to freeze and announce a mid-task child transcript instead of the eventual final handoff.

## Best Possible Solution

Add a compaction-aware lifecycle contract so internal compaction/resume does not satisfy parent-visible subagent completion. The final behavior should announce exactly once, after the resumed child run reaches terminal state, while preserving existing `sessions_yield`, retry-grace, and deferred-announce behavior.

## Reproduction Assessment

Yes. The issue provides three timestamped compaction incidents where the parent completion preceded the worker's later final assistant message, plus a no-compaction control; current source still shows the matching wait/freeze path.

## Solution Assessment

Yes. Keeping the task lifecycle pending across compaction/resume is the narrowest maintainable solution; only extending the output-capture retry or refreshing text after cleanup would leave successful premature announces uncorrected.

## Repair Work Prompt

The observable bug is that a subagent session compaction/resume can make the parent-visible `agent.wait`/subagent registry path treat the child task as completed, freeze a mid-task assistant line into `frozenResultText`, and announce it before the resumed run produces the real final handoff. Keep the fix narrow to subagent lifecycle/wait/announce behavior: compaction must remain an internal runtime event, and the parent should announce exactly once after the resumed child run actually reaches terminal state.
