## Summary

Keep open. Current `main` still has a credible premature-completion path: compaction emits on a separate `compaction` stream, while `agent.wait`/subagent registry completion still treats a non-pending, non-yielded `ok` result as terminal and freezes the latest visible child output immediately; the related open PR #75462 is adjacent logging/recovery work, not a compaction-resume fix.

## What This Changes

The issue reports that subagent completion is announced and `frozenResultText` is captured during mid-task session compaction instead of after the resumed child run actually finishes.

## Best Possible Solution

Make compaction/resume nonterminal for subagent completion, keeping the tracked task pending or remapping the wait until the resumed run reaches its real terminal assistant output and announces exactly once.

## Reproduction Assessment

Yes. The issue gives a 3-of-3 production reproduction pattern, and current source supports a focused regression by simulating compaction/retry plus an early `agent.wait`/lifecycle `ok` before the final child assistant output.

## Solution Assessment

Yes. Fixing the lifecycle/wait contract is the narrow maintainable solution; only extending output-capture polling would still allow a successful parent completion event before the child task is actually done.
