## Summary

Keep open. Current main still has a credible premature-completion path: compaction retry state is emitted on a separate compaction stream, while ordinary lifecycle end still resolves `agent.wait` as `ok` and lets the subagent registry freeze the current child output.

## What This Changes

The issue reports that subagent completion can be announced during session compaction, before the resumed subagent run produces its real final handoff.

## Best Possible Solution

Make compaction retry nonterminal for `agent.wait` and subagent completion so the tracked run remains pending across compaction and announces exactly once with the resumed run's real terminal output.

## Reproduction Assessment

Yes. The report has concrete production reproductions, and a focused regression can emit `compaction_end` with `willRetry: true` followed by lifecycle `end`, then verify `agent.wait` and subagent completion do not resolve until the resumed terminal result exists.

## Solution Assessment

Yes. Fixing the lifecycle/wait contract is the narrow maintainable solution; only extending the frozen-output retry window would still allow an apparently successful completion event before the child task is done.

## Repair Work Prompt

Fix the current subagent compaction completion bug described in this issue. Current main can treat `compaction_end` with `willRetry: true` followed by lifecycle `end` as a successful `agent.wait`, causing `waitForSubagentCompletion` to call `completeSubagentRun`, freeze the current `chat.history` assistant text, and announce before the resumed child run produces its final handoff. Implement a generic lifecycle/wait fix so compaction retry is nonterminal for `agent.wait` and subagent completion; prefer carrying or tracking compaction retry state in the gateway wait/subagent registry path rather than relying only on a longer output-capture retry. Preserve existing yielded, aborted timeout, replay-invalid, and delivery retry behavior. Add focused regressions for `agent.wait` and subagent registry completion with `willRetry: true` followed by resumed terminal output. Update `CHANGELOG.md` under fixes because this is user-facing runtime behavior. Do not add plugin-specific ids or startup migrations.
