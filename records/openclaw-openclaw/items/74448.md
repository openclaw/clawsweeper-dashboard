## Summary

Keep open. Current main still has the reported lifecycle shape: subagent completion is driven by a terminal-looking `agent.wait` or lifecycle end event, while compaction is emitted on a separate stream and is not used to transfer the wait to a resumed run before freezing and announcing the result.

## What This Changes

The issue requests that subagent task completion and `frozenResultText` capture occur only after the real resumed child run finishes, not when session compaction interrupts the original run mid-task.

## Best Possible Solution

Add a compaction-aware subagent lifecycle contract: compaction/resume should keep the parent-visible task pending, preserve one final announce after the resumed run reaches terminal state, and leave existing `sessions_yield`, retry-grace, frozen-result cap, orphan recovery, and cleanup retry behavior intact.

## Reproduction Assessment

Yes. The issue provides three timestamped compaction incidents plus a no-compaction control, and current source still exposes the same wait/freeze/announce path. A focused regression can simulate a compaction event followed by an `agent.wait` ok for the old run and assert no parent announce until the resumed run ends.

## Solution Assessment

Yes. Keeping the lifecycle pending across compaction/resume is the narrowest maintainable direction; only extending the capture retry or refreshing `frozenResultText` after cleanup would still allow a premature successful completion event to reach the parent.
