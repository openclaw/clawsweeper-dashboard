## Summary

Keep open. Current main still has a credible premature subagent-completion path: compaction retry state is tracked separately, but lifecycle `end` events can still produce an `agent.wait` `ok` result, and the registry then freezes and announces the child session output as terminal.

## What This Changes

The issue reports that subagent completion is announced during mid-task session compaction, causing the parent to receive stale `frozenResultText` instead of the worker's final handoff.

## Best Possible Solution

Make compaction retry nonterminal for subagent completion so the tracked task remains pending across compaction and announces exactly once from the resumed run's real terminal output.

## Reproduction Assessment

Yes. The issue gives concrete production reproductions, and current source supports a focused harness reproduction: emit compaction `willRetry`, then a lifecycle `end`/`agent.wait` `ok`, and verify no parent completion is announced until the resumed terminal result exists.

## Solution Assessment

Yes. Fixing the lifecycle/wait contract is the narrowest maintainable solution; only extending output-capture retries would still allow a successful completion event before the child task is actually done.

## Repair Work Prompt

Fix openclaw/openclaw#74448. Subagent completion must not be announced while a child run is only crossing an internal compaction retry/resume boundary. Keep the fix in the subagent/embedded lifecycle boundary: make compaction `willRetry` nonterminal for `agent.wait` and subagent completion cleanup, or otherwise carry the retry/continuation state so completion freezes and announces only once from the final resumed assistant handoff. Do not solve this by merely lengthening output-capture polling, and do not add a config option. Add regression coverage that simulates compaction `willRetry` plus an intermediate lifecycle `end` before the final resumed output, asserting no parent completion event, cleanup, or stale `frozenResultText` is delivered until the true terminal result. Preserve existing sessions_yield behavior and current retry/dedupe semantics. Add a user-facing changelog fix entry crediting the reporter if policy allows.
