## Summary

Keep open. Current main still has the reported lifecycle shape: compaction is emitted on a separate stream, while `agent.wait` and the subagent registry treat a non-aborted lifecycle `end` as a successful terminal run and immediately freeze/announce the current child transcript output.

## What This Changes

Ensure subagent completion and `agent.wait` do not resolve, freeze, or announce on mid-task compaction, but wait for the resumed run's real terminal handoff.

## Best Possible Solution

Keep this issue open and add a compaction-aware subagent lifecycle contract so parent-visible completion remains pending across compaction/resume and exactly one final handoff is frozen and announced after the resumed run reaches its true terminal state.

## Reproduction Assessment

Yes. The issue provides three timestamped compaction incidents plus a no-compaction control, and current source has a deterministic in-repo simulation path: emit compaction retry/end activity, observe a terminal-looking old-run lifecycle `end`, then assert no parent announce or frozen result until the resumed final answer.

## Solution Assessment

Yes. Making compaction/resume nonterminal for `agent.wait` and the subagent registry is the narrowest maintainable fix; only extending output polling or refreshing `frozenResultText` after cleanup would still allow a premature successful completion event to reach the parent.
