## Summary

Keep open. Current main still matches the reported premature-completion path: terminal `agent.wait` results are converted into subagent completion without a compaction/resume-chain guard, result text is frozen immediately from child history, and the later refresh path only helps while cleanup remains pending rather than after a successful premature announce.

## What This Changes

The issue reports that subagent completion is announced when a child session compacts and resumes, freezing an intermediate assistant message instead of the final child handoff.

## Best Possible Solution

Add a compaction-aware subagent lifecycle contract so internal compaction/resume does not satisfy parent-visible completion. The final behavior should announce exactly once after the resumed child run reaches terminal state, while preserving existing `sessions_yield`, retry grace, frozen-result, and deferred-announce behavior.

## Reproduction Assessment

Yes. Force a long subagent run to compact, then compare the parent completion event timestamp/body with the child session transcript's later final assistant message; the issue already provides three such incidents and a no-compaction control.

## Solution Assessment

Yes. Carrying the wait/completion lifecycle across compaction or suppressing the pre-resume terminal event is the narrowest maintainable fix; only extending output-capture polling would still allow a successful premature announce to be treated as final.

Current main can treat the original run's terminal `agent.wait` result during compaction/resume as real subagent completion, freeze a mid-task assistant line, and announce it to the parent before the resumed child run finishes.
