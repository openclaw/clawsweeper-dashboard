## Summary

Keep open. The PR is aimed at a real recovery bug, but it changes a helper that current production replay does not call, and the proposed drop-all behavior can discard valid assistant text for some `incomplete-text` classifications.

## What This Changes

The PR removes the `incomplete-text` prefill branch in `sanitizeThinkingForRecovery` and updates the focused unit test to expect the last assistant message to be dropped.

## Best Possible Solution

Retarget the fix to the active runtime replay path, avoid assistant prefill for genuinely unfinished turns, preserve valid visible assistant content, and leave the broader recovery-state issue open for follow-up work.

## Reproduction Assessment

Yes for the PR defects: source search shows the changed helper is test-only, and the classifier/drop-all edge case follows directly from `assessLastAssistantMessage`. The broader aborted-run inconsistency still needs focused live or integration reproduction after a runtime-path fix.

## Solution Assessment

No. The current PR is not the best fix because it changes an inactive helper and would be too broad if wired in as-is; the safer repair belongs where provider replay messages are built and should distinguish unfinished fragments from valid assistant output.
