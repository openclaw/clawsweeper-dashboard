## Summary

Keep open. Current main still omits model compat when normalizing OpenAI Responses and Completions tool schemas, the bundled Fireworks Kimi router catalog row still lacks the requested unsupported-keyword metadata, and open PR #75476 already targets this issue with closing syntax, so this issue should stay open until that PR is reviewed and either landed or rejected.

## What This Changes

Thread model compatibility into OpenAI-compatible tool schema normalization and add Fireworks Kimi metadata so unsupported JSON Schema keywords such as `not` are stripped before provider requests.

## Best Possible Solution

Review and land a focused fix like PR #75476 that passes `ModelCompatConfig` through OpenAI Responses and Completions tool schema normalization, adds the Fireworks Kimi unsupported-keyword metadata, and includes targeted regression coverage plus the required changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction can build OpenAI Responses or Completions params with a model whose compat lists `unsupportedToolSchemaKeywords: ["not"]` and a tool schema containing `{ "not": {} }`; current main never passes that compat into the OpenAI tool normalizer, so the keyword remains.

## Solution Assessment

Yes. The narrow maintainable fix is to thread the existing compat object into the existing schema-normalization seam and keep Fireworks metadata in the Fireworks provider catalog, rather than adding Fireworks-specific branching in core.
