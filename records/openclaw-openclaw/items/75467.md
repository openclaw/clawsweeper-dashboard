## Summary

Keep open. Current main still does not pass model compat into the OpenAI Responses or Completions tool-schema normalization path, the Fireworks Kimi K2.5 Turbo catalog row still lacks the requested unsupported-keyword metadata, and open PR #75476 already targets this issue with closing syntax, so this should remain open until that PR is reviewed and either landed or rejected.

## What This Changes

The issue requests threading `compat.unsupportedToolSchemaKeywords` into OpenAI-compatible tool-schema normalization and adding Fireworks Kimi K2.5 Turbo catalog metadata so schemas containing `not` are stripped before provider calls.

## Best Possible Solution

Review and land a focused fix that passes the existing `ModelCompatConfig` into the OpenAI tool-schema normalization path, keeps Fireworks-specific unsupported-keyword metadata in the Fireworks provider catalog, adds strict and non-strict regression coverage, and preserves the existing lower-level schema-normalization contract.

## Reproduction Assessment

Yes. A source-level reproduction can use an OpenAI-compatible model with `compat.unsupportedToolSchemaKeywords: ["not"]` and a tool schema containing `{ "not": {} }`; current main never passes that compat into the OpenAI tool normalizer, so the keyword remains.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix because the lower-level normalizer already supports compat-declared keyword stripping; this should be wired through the generic OpenAI tool path rather than adding Fireworks-specific branching in core.
