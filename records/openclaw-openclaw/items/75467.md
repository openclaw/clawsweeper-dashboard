## Summary

Keep open. Current main still has the reported compatibility gap, and the issue is already paired with open PR #75476, which uses closing syntax and should be reviewed or landed before this issue is closed.

## What This Changes

The issue requests that OpenAI-compatible completions providers honor `compat.unsupportedToolSchemaKeywords` during tool-schema normalization so Fireworks `kimi-k2p5-turbo` can strip unsupported JSON Schema keywords such as `not`.

## Best Possible Solution

Land a focused fix that threads `ModelCompatConfig` into OpenAI Responses and Completions tool-schema normalization, adds provider-owned Fireworks compat metadata for the affected Kimi router, and locks the behavior with targeted unit coverage and the required changelog entry.

## Reproduction Assessment

Yes. A focused source-level reproduction can build an OpenAI-compatible model with `compat.unsupportedToolSchemaKeywords: ["not"]` and a tool schema containing `{ "not": {} }`; current main never passes that compat into the OpenAI tool normalizer, so the unsupported keyword remains.

## Solution Assessment

Yes. The narrow maintainable solution is to pass existing model compat into the existing schema-normalization contract and keep Fireworks-specific metadata in the Fireworks plugin, rather than adding Fireworks branching in core.
