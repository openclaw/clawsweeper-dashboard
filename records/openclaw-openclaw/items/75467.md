## Summary

Keep open. Current main still has the reported gap: the generic unsupported-schema-keyword stripper exists, but the OpenAI completions and responses tool conversion paths call strict tool normalization without any model compat, and the bundled Fireworks Kimi K2.5 Turbo catalog row does not declare the reported `not` keyword incompatibility.

## What This Changes

The issue requests preserving `compat.unsupportedToolSchemaKeywords` through OpenAI tool schema normalization and adding `not` as unsupported schema metadata for the Fireworks Kimi K2.5 Turbo catalog model.

## Best Possible Solution

Keep the generic schema-cleaning seam and thread raw model compat, or the full model, into OpenAI completions and responses tool parameter normalization. Add the Fireworks plugin manifest compat metadata for the confirmed affected model(s), focused regression tests for both strict and non-strict tool normalization where relevant, and the required user-facing changelog entry.

## Reproduction Assessment

Yes. A focused unit can construct an `openai-completions` model with `compat.unsupportedToolSchemaKeywords: ["not"]` and a tool schema containing `{ "not": {} }`; current main's source path shows that `buildOpenAICompletionsParams` will still emit the `not` keyword because no model compat reaches `normalizeToolParameterSchema`.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: pass model compat through the existing generic normalization seam and keep Fireworks-specific behavior in the Fireworks plugin catalog. A core Fireworks-only branch would be less appropriate because the repo already has a provider-agnostic unsupported-keyword mechanism.
