## Summary

Keep open. Current main still has the reported OpenAI-compatible tool-schema compatibility gap, and open PR #75476 already targets this issue with closing syntax, so the issue should stay open until that PR is reviewed and either landed or rejected.

## What This Changes

The issue asks OpenAI-compatible providers to honor `compat.unsupportedToolSchemaKeywords` so Fireworks tool schemas strip unsupported `not` keywords before request dispatch.

## Best Possible Solution

Review and land a focused fix that threads `ModelCompatConfig` into OpenAI Responses and Completions tool-schema normalization, adds the Fireworks provider metadata for the affected Kimi router, and keeps the regression covered by targeted tests and the required changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: construct an OpenAI-compatible model with `compat.unsupportedToolSchemaKeywords: ["not"]` and a tool schema containing `{ "not": {} }`; current main never passes that compat into the OpenAI tool normalizer, so the keyword remains.

## Solution Assessment

Yes. The narrow maintainable fix is to pass existing model compat into the existing schema-normalization contract and keep Fireworks-specific metadata in the Fireworks plugin manifest, rather than adding Fireworks-specific branching in core transport code.
