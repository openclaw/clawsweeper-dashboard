## Summary

Keep open. Current main still drops model compatibility before OpenAI tool-schema normalization, and the Fireworks Kimi K2.5 Turbo catalog entry still lacks the requested `unsupportedToolSchemaKeywords` metadata. An open closing PR, #75476, already targets the exact gap, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Honor `compat.unsupportedToolSchemaKeywords` for OpenAI-compatible tool schema normalization and add Fireworks Kimi K2.5 Turbo metadata so unsupported JSON Schema keywords such as `not` are stripped before requests.

## Best Possible Solution

Review and land #75476 or an equivalent narrow fix that threads `ModelCompatConfig` into the OpenAI Responses and Completions tool-schema normalization path, adds provider-owned Fireworks manifest compat metadata, and preserves the behavior with focused regression tests.

## Reproduction Assessment

Yes. A focused source-level reproduction can use an `openai-completions` model with `compat.unsupportedToolSchemaKeywords: ["not"]` and a tool schema containing `{ "not": {} }`; current main reaches `normalizeToolParameterSchema` without `modelCompat`, so the keyword remains.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix: pass extracted model compat into the existing schema normalizer and keep Fireworks-specific compatibility data in the Fireworks plugin manifest rather than adding provider-specific branching in core.
