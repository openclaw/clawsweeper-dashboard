## Summary

Keep open. Current `main` still has the reported OpenAI-compatible tool-schema compat gap, and the issue is already paired with open, unmerged PR #75476, which uses closing syntax and should be reviewed or landed before the issue is closed.

## What This Changes

The issue requests that OpenAI-compatible completions providers honor `model.compat.unsupportedToolSchemaKeywords` for Fireworks tool schemas so unsupported JSON Schema keywords such as `not` are stripped before dispatch.

## Best Possible Solution

Review and land PR #75476, or an equivalent focused fix, so existing model compat metadata is threaded into OpenAI Responses and Completions tool-schema normalization and Fireworks owns the Kimi router keyword metadata in its plugin manifest.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: use an OpenAI-compatible model with `compat.unsupportedToolSchemaKeywords: ["not"]` and a tool schema containing `{ "not": {} }`; current `main` never passes that compat into the OpenAI tool normalizer, so the keyword remains.

## Solution Assessment

Yes. Passing existing model compat into the existing schema-normalization contract, with Fireworks-specific metadata staying in the Fireworks plugin manifest, is the narrow maintainable fix and avoids hard-coding Fireworks behavior in core.
