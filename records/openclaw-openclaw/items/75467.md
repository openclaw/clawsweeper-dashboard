## Summary

Keep open. Current main still has the reported compat-plumbing gap, and the linked fix PR #75476 is open with closing syntax for this issue, so the issue should remain open until that PR lands or is replaced.

## What This Changes

The issue reports that OpenAI-completions providers such as Fireworks ignore `compat.unsupportedToolSchemaKeywords`, letting JSON Schema `not` reach tool payloads and trigger provider 400s.

## Best Possible Solution

Review and land PR #75476, or an equivalent narrow replacement, so model compat is threaded into OpenAI Responses and Completions tool schema normalization and Fireworks owns the affected Kimi router metadata.

## Reproduction Assessment

Yes. A source-level reproduction is clear: build an OpenAI-completions tool schema containing `not` with a model compat entry listing `unsupportedToolSchemaKeywords: ["not"]`; current main never passes that compat into `normalizeToolParameterSchema`, so the keyword remains.

## Solution Assessment

Yes. The maintainable fix is to pass existing model compat into the schema normalizer and add provider-owned Fireworks catalog metadata, rather than adding Fireworks-specific branching in core.
