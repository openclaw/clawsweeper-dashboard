## Summary

Keep open. Current `main` still has the reported OpenAI-compatible tool-schema compatibility gap, and the linked closing implementation PR is still open, so this issue should remain the tracker until that PR lands or is replaced.

## What This Changes

The issue requests that OpenAI-compatible tool schema normalization preserve provider model compatibility so Fireworks can strip unsupported JSON Schema keywords such as `not`.

## Best Possible Solution

Review and land or reject #75476; the desired end state is generic OpenAI tool-schema normalization honoring `ModelCompatConfig`, Fireworks unsupported-keyword metadata, and focused regression coverage.

## Reproduction Assessment

Yes, by source inspection. Use an OpenAI-compatible model with `compat.unsupportedToolSchemaKeywords: ["not"]` and a tool schema containing `{ "not": {} }`; current `main` drops compat before `normalizeToolParameterSchema`, so the keyword remains.

## Solution Assessment

Yes. The narrow maintainable solution is to thread existing model compatibility into OpenAI tool-schema normalization and keep Fireworks unsupported-keyword data in the provider catalog, which is what #75476 proposes.
