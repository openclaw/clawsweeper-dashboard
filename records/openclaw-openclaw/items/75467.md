## Summary

Keep open. Current main still has the reported compatibility gap: OpenAI Responses and Completions tool conversion does not pass model compat into tool-schema normalization, the reduced transport compat object still omits unsupported schema keywords, and the Fireworks Kimi K2.5 Turbo catalog row has no `compat.unsupportedToolSchemaKeywords` metadata. Open PR #75476 already targets this issue with closing syntax, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests applying `compat.unsupportedToolSchemaKeywords` to OpenAI-compatible tool-schema normalization, especially for Fireworks Kimi K2.5 Turbo rejecting schemas that contain `not`.

## Best Possible Solution

Review and land PR #75476 or an equivalent focused fix that threads `ModelCompatConfig` into OpenAI Responses and Completions tool-schema normalization, adds provider-owned Fireworks compat metadata for the affected Kimi router, and includes targeted regression coverage plus the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to use an OpenAI-compatible model with `compat.unsupportedToolSchemaKeywords: ["not"]` and a tool schema containing `{ "not": {} }`; current main never passes that compat into OpenAI tool normalization, so the keyword remains.

## Solution Assessment

Yes. The narrow maintainable fix is to pass existing model compat into the existing schema-normalization contract and keep Fireworks-specific metadata in the Fireworks plugin manifest, rather than adding Fireworks-specific branching in the core transport.
