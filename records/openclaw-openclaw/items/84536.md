## Summary

Keep open: current main has precheck and overflow-recovery code, but the fresh logs describe a remaining user-visible failure where the embedded run reaches terminal context-overflow handling while WebChat/Telegram delivery and processing-state cleanup do not complete.

## What This Changes

The issue asks for preemptive embedded context overflow to notify the user and recover or clear the session instead of silently leaving it stuck in processing.

## Best Possible Solution

Make preemptive and mid-turn overflow either compact-and-resume or deliver a marked visible channel error while clearing processing/lane state, with regression coverage for the embedded runner and at least one channel delivery path.

## Reproduction Assessment

No, not as a high-confidence end-to-end reproduction. Source inspection and reporter logs show a plausible precheck-overflow, compaction, and stale-processing path, but I did not run a live MiniMax/WebChat or Telegram scenario to reproduce the silent channel failure.

## Solution Assessment

Unclear whether the reporter's suggested threshold/config change is the best fix. The narrower maintainable path is to prove and repair terminal delivery plus liveness cleanup without changing compaction policy unless a maintainer explicitly chooses that.
