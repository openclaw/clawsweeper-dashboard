## Label Justifications

- `P2`: This is a useful user-facing ergonomics feature with limited blast radius, but it is not an urgent shipped regression.
- `merge-risk: 🚨 compatibility`: The `/m` alias would reserve a global command token and can override existing configured model alias expectations without a migration path.
- `merge-risk: 🚨 session-state`: The proposed `/m` and `/c` aliases affect model selection and compaction behavior, so incorrect alias wiring can change or fail to change session state.

## Summary

Keep open: the alias idea is focused, but the current patch is not merge-ready because two proposed aliases do not follow the execution paths of the full commands, the branch conflicts with current slash-command docs, and the PR still lacks real after-fix behavior proof.

## What This Changes

The PR adds nine built-in slash-command aliases in `src/auto-reply/commands-registry.shared.ts` and updates `docs/tools/slash-commands.md` to document them.

## Best Possible Solution

Accept the alias feature only after directive-backed and raw-body-sensitive aliases are wired through the same execution paths as their full commands, the docs are rebased to current main, and redacted real behavior proof is added.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `/m` is only a registry alias while model switching is parsed from raw `/model` or configured aliases, and `/c` dispatches compaction after normalization while instruction extraction reads only raw `/compact` text.

## Solution Assessment

No. Registry-only aliases are not the best complete solution for directive-backed or raw-body-sensitive commands; those aliases need parser support or should be omitted until maintainers accept the compatibility tradeoff.
