## Label Justifications

- `P2`: This is a useful user-facing ergonomics feature with limited blast radius, but it is not an urgent shipped regression.
- `merge-risk: 🚨 compatibility`: The proposed `/m` alias would reserve a global command token and can override existing configured model alias expectations without a migration path.

## Summary

Keep open: the alias idea is still useful and not implemented on current main, but this PR is not merge-ready because two aliases do not preserve the behavior users would expect and the external PR still lacks real behavior proof.

## What This Changes

Registers nine short text aliases for existing chat slash commands and updates the slash-command documentation.

## Best Possible Solution

Accept the alias feature only after directive-backed and raw-body-sensitive aliases are wired through the same execution paths as their full commands, risky short names are explicitly approved or omitted, docs are rebased, and redacted real behavior proof is added.

## Reproduction Assessment

Yes for the review findings. Source inspection shows `/m` is registry-only while model switching is parsed from raw `/model` or configured aliases, and `/c` dispatches compaction after normalization while instruction extraction reads only raw `/compact` text.

## Solution Assessment

No. Registry-only aliases are not the best complete solution for directive-backed or raw-body-sensitive commands; those aliases need parser support, handler support, or omission until maintainers accept the compatibility tradeoff.
