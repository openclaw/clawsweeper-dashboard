## Label Justifications

- `P2`: This is a useful user-facing ergonomics feature with limited blast radius, but it is not an urgent shipped regression.
- `merge-risk: 🚨 compatibility`: The /m alias would reserve a global command token and can override existing configured model alias expectations without a migration path.
- `merge-risk: 🚨 session-state`: The proposed /m and /c aliases affect model selection and compaction behavior, so incorrect alias wiring can change or fail to change session state.

## Summary

Keep open: the alias idea is useful, but the patch is not merge-ready because `/m` and `/c` do not fully follow the same runtime paths as their full commands, the branch now conflicts with current slash-command docs, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds short built-in text aliases for existing slash commands and updates the slash-command documentation.

## Best Possible Solution

Accept the alias feature only after directive-backed and raw-body-sensitive aliases are wired through the same execution paths as their full commands, the docs are rebased to current main, and real channel or terminal proof is added.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: /m is only a registry alias while model switching is parsed from raw directive text, and /c dispatches compaction after normalization while instruction extraction reads only the raw /compact form. I did not run tests because this review is read-only.

## Solution Assessment

No. Registry-only aliases are not the best complete solution for directive-backed or raw-body-sensitive commands; those aliases need parser support or should be omitted until maintainers accept the compatibility tradeoff.
