## Summary

Current main and the latest shipped release do not treat `compactionAttempts=0` as disabling overflow recovery; it is the before-first-attempt diagnostic counter, while the actual recovery path is bounded by a separate max-attempt limit and surfaces warnings/errors.

## What This Changes

The issue reports that `compactionAttempts=0` prevents auto-compaction from starting and leaves context overflow unrecoverable without warning or user-visible failure.

## Best Possible Solution

Keep the shipped bounded overflow-recovery behavior and treat `compactionAttempts=0` as a diagnostic counter value, not a user-facing disabled setting.

## Reproduction Assessment

No current-main failing reproduction is available from source inspection: the counter starts at 0, but that value does not gate recovery. Current source and shipped release source show the run loop logs the overflow, attempts compaction up to three times, and surfaces a visible error if recovery fails.

## Solution Assessment

Yes. The current implementation is the narrow maintainable solution: keep the diagnostic counter, use a separate bounded max-attempt guard, retry after successful compaction, and surface a blocked overflow error when recovery is exhausted.
