## Summary

Keep open: this protected, security-sensitive XL storage refactor still has source-confirmed blocking regressions and no after-fix real behavior proof.

## What This Changes

This PR moves OpenClaw runtime, session, transcript, plugin, cron, device, pairing, backup, app, docs, and test state from file/JSON stores toward global and per-agent SQLite databases.

## Best Possible Solution

Keep the database-first direction, but land only after maintainer/security review verifies recency-safe imports, protocol generation parity, bounded transcript reads, regression coverage, and redacted real behavior proof.

## Reproduction Assessment

Yes for the review findings by source inspection: PR head `d3a938996bf20108287bd44a2faf604c3bb55ba8` still contains the Swift protocol mismatch and overwrite-style legacy imports. I did not run live migration scenarios because this review is read-only and the PR lacks real behavior proof.

## Solution Assessment

No. The SQLite storage-boundary refactor is a plausible direction for https://github.com/openclaw/openclaw/issues/78096, but the current patch is not mergeable until the protocol mismatch, stale-import rollback risks, bounded read regressions, and proof gap are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep Swift protocol version aligned with the server:** `apps/shared/OpenClawKit/Sources/OpenClawProtocol/GatewayModels.swift:5`
  - body: This generated Swift model advertises protocol 3, but the gateway server still requires protocol 4 and closes handshakes whose advertised range excludes it. As changed, OpenClawKit/macOS clients will fail to connect to current gateways; regenerate or revert the Swift model so it advertises protocol 4.
  - confidence: 0.95
- **[P1] Preserve newer per-agent session rows during migration:** `src/infra/state-migrations.ts:2683-2687`
  - body: The global session migration always overwrites per-agent `session_entries` with `excluded` values and then drops the global table. If upgraded runtime traffic has already written newer per-agent rows before doctor/migrate runs, this permanently rolls session metadata backward; merge by recency or skip newer rows.
  - confidence: 0.91
- **[P1] Merge pairing imports instead of replacing access state:** `src/pairing/pairing-store.ts:731-747`
  - body: The legacy pairing import replaces current SQLite requests and account allowlists from legacy JSON before deleting the files. A delayed doctor run can re-authorize revoked senders or remove newer approvals, so this needs merge or recency-aware behavior.
  - confidence: 0.9
- **[P1] Do not restore revoked device tokens from legacy JSON:** `src/infra/device-auth-store.ts:95-97`
  - body: `importLegacyDeviceAuthFileToSqlite` writes the entire legacy token store into SQLite and then removes the source file. If tokens were rotated or revoked in SQLite before doctor runs, this can permanently reintroduce stale credentials.
  - confidence: 0.89
- **[P1] Honor recent transcript byte and line caps:** `src/gateway/session-transcript-readers.ts:253-258`
  - body: The recent-read path loads all SQLite transcript messages and slices only by `maxMessages`, ignoring the byte and line caps callers use to bound large transcript tails. Long sessions can now force full-history reads and memory spikes where a bounded tail was expected.
  - confidence: 0.9
- **[P2] Count UTF-8 bytes when pruning cron run logs:** `src/cron/run-log.ts:203-204`
  - body: `pruneCronRunLogRows` measures `maxBytes` with `row.entry_json.length`, which counts UTF-16 code units rather than stored UTF-8 bytes. Non-ASCII run entries can exceed the configured byte cap, so use UTF-8 byte length for the retention budget.
  - confidence: 0.84
