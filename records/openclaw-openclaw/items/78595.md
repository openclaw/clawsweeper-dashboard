## Summary

Keep open. Current head is an unmergeable XL storage refactor with protected `security` and `maintainer` labels, and source inspection still shows legacy import paths that can overwrite live SQLite state before deleting the legacy source.

## What This Changes

The PR moves OpenClaw runtime state, session/transcript data, plugin/channel state, auth/profile state, and related migration/export paths from JSON/JSONL/sidecar stores into global and per-agent SQLite databases.

## Best Possible Solution

Keep the database-first direction, fold or land the direct correctness fixes such as https://github.com/openclaw/openclaw/pull/79971, keep companion seams in their follow-up lane, and require exact-head populated-upgrade proof before merge.

## Reproduction Assessment

Yes by source inspection. A delayed doctor run after upgraded runtime writes to SQLite can import stale legacy files into the same rows and then delete the legacy source; I did not run a live migration in this read-only review.

## Solution Assessment

No for merging as-is. The SQLite architecture is plausible, but import idempotency/recency, protected security review, and current-head populated-upgrade proof need to land before this is the maintainable solution.
