## Summary

Keep open: current main does not contain this SQLite runtime-state refactor, and the PR remains a protected, security-sensitive architecture change with confirmed source-level migration data-loss blockers and no real behavior proof.

## What This Changes

This PR moves broad OpenClaw runtime state from JSON/JSONL and sidecar files into shared and per-agent SQLite databases across sessions, transcripts, plugin state, cron/task state, docs, tests, and UI/protocol surfaces.

## Best Possible Solution

Keep the PR in maintainer review until migration conflict semantics preserve newer SQLite state, storage/backup guarantees are settled, and a real upgrade/runtime proof artifact is added.

## Reproduction Assessment

Yes for the confirmed PR blockers: source inspection of the PR head shows delayed legacy imports can overwrite newer SQLite cron rows and drop migrated plugin metadata. I did not run a live migration because this review is read-only.

## Solution Assessment

No. The database-first direction may be right, but this patch is not yet the safest maintainable solution because migration paths can overwrite newer state and remove recovery sources.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve newer cron rows during legacy doctor repair:** `src/commands/doctor-cron.ts:282-285`
  - body: When `cron/jobs.json` exists, doctor loads that legacy file and saves it through the SQLite-backed cron store. `saveCronStore` rewrites conflicting rows and deletes rows absent from the imported file, so a delayed doctor run after upgraded runtime activity can lose newer SQLite cron jobs or state.
  - confidence: 0.9
- **[P1] Keep Telegram binding metadata during migration:** `extensions/telegram/src/state-migrations.ts:143-153`
  - body: The legacy Telegram binding importer rebuilds each record with only core routing fields, dropping optional fields such as timeouts, labels, and metadata before deleting the source file. Preserve the parsed optional fields so existing binding behavior survives upgrade.
  - confidence: 0.86
- **[P2] Store migrated Discord deploy hashes under runtime keys:** `extensions/discord/src/state-migrations.ts:87-91`
  - body: The Discord importer writes command deploy hashes as `legacy:${key}`. Runtime deploy-cache lookups use client-id-scoped keys, so migrated hashes will not be read and upgrades will discard the cache after the legacy file is removed.
  - confidence: 0.86
