## Summary

Keep open: this is a protected, broad storage/security refactor with unresolved P1/P2 correctness findings and no after-fix real behavior proof, so it is not safe to close or automate.

## What This Changes

This PR moves runtime/session/transcript/cron/pairing/task/plugin state from JSON/JSONL sidecars into shared and per-agent SQLite stores, with migration, docs, SDK, and test updates.

## Best Possible Solution

Land the SQLite runtime-state model only after maintainers/security owners review agent partitioning, migration semantics, bounded reads, cross-process writes, cleanup, cron merge behavior, and real upgrade/runtime proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection of PR head shows the scope fallbacks, configured-store migration gap, process-local append serialization, bounded-read regression, cleanup no-op, and cron overwrite path. This is a PR review, not a current-main user bug reproduction.

## Solution Assessment

No. SQLite-first runtime state may be the right architecture, but this patch is not yet the best landing shape while it can mis-scope agent data, skip migration inputs, weaken bounded reads, and overwrite scheduler state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve configured store agent scope:** `src/config/sessions/store-backend.sqlite.ts:43-44`
  - body: When `session.store` uses a custom non-canonical path, this fallback writes rows under `main`, so non-main agent session metadata can be persisted and queried from the wrong SQLite partition.
  - confidence: 0.92
- **[P1] Preserve transcript scope for custom paths:** `src/agents/transcript/session-manager.ts:58-67`
  - body: `SessionManager.open(sessionFile)` callers do not always pass an agent id, and non-canonical transcript paths fall back to `main`, mixing non-main agent transcript events into the wrong partition.
  - confidence: 0.9
- **[P1] Import configured session.store paths in doctor:** `src/infra/state-migrations.ts:763-767`
  - body: Doctor migration detection only checks the default per-agent `sessions.json` path. Installs using a configured `session.store` can be skipped before runtime JSON support is removed, hiding existing session history after upgrade.
  - confidence: 0.9
- **[P1] Make transcript appends cross-process atomic:** `src/config/sessions/transcript-append.ts:142-166`
  - body: The append queue is process-local and parent/idempotency decisions are based on a stale pre-insert event list, so a gateway plus CLI or worker can append conflicting parents or duplicate idempotent messages.
  - confidence: 0.86
- **[P1] Honor recent-read byte and line caps:** `src/gateway/session-utils.fs.ts:299-303`
  - body: Recent reads now load the entire SQLite transcript and only apply `maxMessages`, so callers that pass `maxBytes` or `maxLines` can hit full-history latency and memory spikes on long sessions.
  - confidence: 0.87
- **[P1] Preserve transcript agent scope during import:** `src/commands/doctor-session-transcripts.ts:76-84`
  - body: Doctor import maps transcripts outside the canonical `agents/<id>/sessions` shape to `main` and then removes the source file, making custom non-main transcript history disappear from its intended agent partition.
  - confidence: 0.86
- **[P2] Delete discarded checkpoint snapshots:** `src/gateway/session-compaction-checkpoints.ts:281-285`
  - body: Checkpoint capture writes full snapshot transcripts into SQLite, but cleanup is a no-op, so failed or discarded compactions leave unreferenced snapshot rows that can accumulate over time.
  - confidence: 0.84
- **[P2] Repair transcripts before deleting legacy JSONL:** `src/infra/state-migrations.ts:957-963`
  - body: The migration imports legacy transcript JSONL and removes the source before the older doctor repair flow can scan it, preserving broken prompt-rewrite branches permanently after upgrade.
  - confidence: 0.82
- **[P2] Merge legacy cron state by recency:** `src/cron/store.ts:206`
  - body: The import writes the whole legacy sidecar payload into SQLite. If SQLite already has newer schedule state and a stale sidecar remains, doctor can roll scheduler state backward.
  - confidence: 0.82
