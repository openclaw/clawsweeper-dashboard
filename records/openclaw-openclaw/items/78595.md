## Summary

Keep open: this protected `security`/`maintainer` XL PR is still not merge-ready because current-head migration paths can overwrite live SQLite state from stale legacy files, and the PR body/thread does not provide sufficient current-head real behavior proof.

## What This Changes

The PR moves runtime state from JSON/JSONL and sidecar files into SQLite-backed global and per-agent stores across sessions, transcripts, auth, plugins, channels, apps, docs, and tests.

## Best Possible Solution

Keep the SQLite direction, but fix legacy import freshness/merge semantics, resolve the Node support decision explicitly, and require current-head migration/runtime proof before merge.

## Reproduction Assessment

Yes, at source level: run the upgraded runtime so SQLite has newer rows, leave a corresponding legacy JSON file in place, then run doctor import. The current-head source shows those imports writing legacy snapshots over active SQLite state and deleting the source; I did not run the destructive migration scenario in this read-only review.

## Solution Assessment

No, not as currently implemented. The database-first design is maintainable, but migration must merge or skip stale legacy inputs when live SQLite rows already exist, and the Node runtime bump needs explicit approval rather than landing as incidental refactor fallout.
