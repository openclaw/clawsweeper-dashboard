## Summary

This PR should stay open: it has protected `security` and `maintainer` labels, current `main` does not implement the SQLite refactor, and the PR head still has source-reproducible migration bugs that can replace newer SQLite state with stale legacy files. It also needs redacted real behavior proof from an actual upgrade/runtime setup before merge.

## What This Changes

The PR moves broad OpenClaw runtime state from JSON/JSONL and sidecar files into shared and per-agent SQLite databases, updating sessions, transcripts, plugin state, cron/task state, docs, tests, and app/UI protocol surfaces around that storage boundary.

## Best Possible Solution

Keep the PR in maintainer review until migration conflict semantics preserve newer SQLite state, SQLite transaction/backup guarantees are settled, and a real upgrade/runtime proof artifact is supplied.

## Reproduction Assessment

Yes for the confirmed PR blocker: source inspection shows a delayed `openclaw doctor --fix` path can save stale legacy cron data over newer SQLite rows. I did not run a live migration because this review is read-only, but the destructive call chain is clear in the PR head.

## Solution Assessment

No. The database-first direction may be right, but this patch is not yet the safest maintainable solution because legacy imports can overwrite newer live SQLite state and delete recovery sources.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve newer cron rows during legacy doctor repair:** `src/commands/doctor-cron.ts:215`
  - body: When `cron/jobs.json` exists, doctor loads that legacy file and then saves it through the SQLite-backed cron store. `saveCronStore` rewrites conflicting rows and deletes rows absent from the imported file, so users who start the upgraded runtime before running doctor can lose newer SQLite cron jobs or state. Merge by recency or skip conflicting rows when SQLite already has newer data.
  - confidence: 0.9
- **[P1] Keep Telegram binding metadata during migration:** `extensions/telegram/src/state-migrations.ts:143-153`
  - body: The legacy Telegram binding importer rebuilds each record with only core routing fields, dropping `idleTimeoutMs`, `maxAgeMs`, `label`, and `metadata` even though those fields are part of `TelegramThreadBindingRecord` and affect expiry/display behavior. Preserve the parsed optional fields before deleting the source file.
  - confidence: 0.86
- **[P2] Store migrated Discord deploy hashes under runtime keys:** `extensions/discord/src/state-migrations.ts:87-91`
  - body: The Discord importer writes command deploy hashes as `legacy:${key}`, but runtime loading only reads keys prefixed with `${clientId}:`. Migrated hashes will never be used, and the legacy cache is removed, forcing avoidable command redeploys after upgrade.
  - confidence: 0.88
