## Summary

Keep open: protected labels plus source-verified P1/P2 storage blockers and missing real behavior proof make this a maintainer/security review item, not cleanup or automation.

## What This Changes

This PR moves runtime/session/transcript/cron/pairing/plugin state from JSON and JSONL sidecars into shared and per-agent SQLite stores with migrations, SDK/docs updates, and tests.

## Best Possible Solution

Land the SQLite runtime-state model only after maintainers resolve scoped state, migration, concurrency, cleanup, cron merge, integrity, and backup blockers and add real upgrade/runtime proof.

## Reproduction Assessment

Yes for the PR blockers: source inspection of the PR diff shows the scope fallbacks, configured-store migration gap, process-local append serialization, bounded-read regression, cleanup no-op, and cron overwrite path. This is PR review evidence, not a current-main bug reproduction.

## Solution Assessment

No. SQLite-first runtime state may be the right direction, but this patch is not yet the best landing shape while it can mis-scope agent data, skip migration inputs, weaken bounded reads, and overwrite scheduler state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve configured store agent scope:** `src/config/sessions/store-backend.sqlite.ts:38-39`
  - body: When `session.store` uses a custom non-canonical path, this fallback writes rows under `main`, so non-main agent session metadata can be persisted and queried from the wrong SQLite partition.
  - confidence: 0.92
- **[P1] Preserve transcript scope for custom paths:** `src/agents/transcript/session-manager.ts:52-60`
  - body: `SessionManager.open(sessionFile)` callers do not always pass an agent id, and non-canonical transcript paths fall back to `main`, mixing non-main agent transcript events into the wrong partition.
  - confidence: 0.9
- **[P1] Import configured session.store paths in doctor:** `src/infra/state-migrations.ts:764-767`
  - body: Doctor migration detection only checks the default per-agent `sessions.json` path. Installs using a configured `session.store` can be skipped before runtime JSON support is removed, hiding existing session history after upgrade.
  - confidence: 0.9
- **[P1] Make transcript appends cross-process atomic:** `src/config/sessions/transcript-append.ts:142-166`
  - body: The append queue is process-local and parent/idempotency decisions are based on a stale pre-insert event list, so a gateway plus CLI or worker can append conflicting parents or duplicate idempotent messages.
  - confidence: 0.86
- **[P1] Honor recent-read byte and line caps:** `src/gateway/session-utils.fs.ts:299-303`
  - body: Recent reads now load the entire SQLite transcript and only apply `maxMessages`, so callers that pass `maxBytes` or `maxLines` can hit full-history latency and memory spikes on long sessions.
  - confidence: 0.87
- **[P1] Preserve transcript agent scope during import:** `src/commands/doctor-session-transcripts.ts:76-84`
  - body: Doctor import maps transcripts outside the canonical `agents/<id>/sessions` shape to `main` and then removes the source file, making custom non-main transcript history disappear from its intended agent partition.
  - confidence: 0.86
- **[P2] Delete discarded checkpoint snapshots:** `src/gateway/session-compaction-checkpoints.ts:281-285`
  - body: Checkpoint capture writes full snapshot transcripts into SQLite, but cleanup is a no-op, so failed or discarded compactions leave unreferenced snapshot rows that can accumulate over time.
  - confidence: 0.84
- **[P2] Repair transcripts before deleting legacy JSONL:** `src/infra/state-migrations.ts:957-963`
  - body: The migration imports legacy transcript JSONL and removes the source before the older doctor repair flow can scan it, preserving broken prompt-rewrite branches permanently after upgrade.
  - confidence: 0.82
- **[P2] Merge legacy cron state by recency:** `src/cron/store.ts:206`
  - body: The import writes the whole legacy sidecar payload into SQLite. If SQLite already has newer schedule state and a stale sidecar remains, doctor can roll scheduler state backward.
  - confidence: 0.82
- **[P2] Raise the shared SQLite busy timeout:** `src/state/openclaw-state-db.ts:154`
  - body: The shared state database uses a 5 second `busy_timeout`; with this PR moving gateway, CLI, cron, transcript, pairing, and artifact writes into one DB, concurrent lanes can surface avoidable `SQLITE_BUSY` failures instead of waiting through expected contention.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff moves sensitive session, transcript, pairing, device, and queue state into SQLite, and current partition/migration defects can mix or lose scoped data.

Concerns:

- **[high] Prevent agent partition mixing:** `src/config/sessions/store-backend.sqlite.ts:38`
  - body: Custom session and transcript paths fall back to `DEFAULT_AGENT_ID`, which can mix non-main agent history and metadata into the `main` SQLite partition.
  - confidence: 0.9
- **[medium] Avoid lossy sensitive state migration:** `src/infra/state-migrations.ts:764`
  - body: Doctor detection skips configured session stores and some imports delete legacy transcript files after import, so upgrades can hide or preserve sensitive history in the wrong or unrepaired partition.
  - confidence: 0.86
- **[medium] Harden shared database contention and integrity:** `src/state/openclaw-state-db.ts:154`
  - body: The new shared state database keeps a 5 second busy timeout and does not show foreign-key enforcement in the connection setup, which is weak for a database that now owns cross-agent runtime and security-relevant state.
  - confidence: 0.78
