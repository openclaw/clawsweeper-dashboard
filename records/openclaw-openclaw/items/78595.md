## Summary

Keep open: the live PR is security/maintainer-labeled, dirty/unmergeable, lacks real behavior proof, and current head still has source-proven protocol and security-state migration blockers.

## What This Changes

The PR moves OpenClaw runtime/session/transcript/plugin/channel/scheduler/artifact state from file-backed stores into SQLite-backed control-plane and per-agent data-plane storage, updating schemas, stores, migrations, SDK/protocol/UI models, docs, and tests.

## Best Possible Solution

Continue the SQLite refactor after restoring sessions protocol compatibility, making legacy imports non-destructive and recency-aware, supplying real upgraded-run proof, and passing maintainer/security-selected gates.

## Reproduction Assessment

Yes, for the PR blockers by source inspection at current head 1eb0f7394b6cff994bfd06ed3ef23d1d44d4cb7e. The Control UI sends a now-rejected `configuredAgentsOnly` parameter, and delayed doctor imports can overwrite the same SQLite-backed security state that runtime token paths mutate.

## Solution Assessment

No. SQLite may be the right architecture, but this patch is not merge-ready while it can reject normal sessions UI calls and roll live security/runtime state backward during migration.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Re-allow configuredAgentsOnly in sessions.list:** `src/gateway/protocol/schema/sessions.ts:39-64`
  - body: `sessions.list` now rejects unknown params, but the Control UI still sends `configuredAgentsOnly` by default. Because this schema validates before the handler runs, the sessions page can get an invalid-request response instead of rows.
  - confidence: 0.95
- **[P1] Skip stale device-auth imports when SQLite state exists:** `src/commands/doctor/legacy/device-auth-store.ts:39-41`
  - body: This importer writes the whole legacy `identity/device-auth.json` snapshot into the live SQLite token store and then removes the source file. If the upgraded runtime has already rotated or revoked tokens in SQLite, a later doctor run can restore stale credentials permanently.
  - confidence: 0.92
- **[P1] Skip stale bootstrap snapshots on SQLite conflict:** `src/commands/doctor/legacy/device-bootstrap.ts:35-41`
  - body: This migration persists the legacy bootstrap-token snapshot into the live device pairing state and deletes the source file. A delayed doctor run can reintroduce tokens that runtime issue/revoke/redeem flows already removed or updated in SQLite.
  - confidence: 0.91
