## Summary

Keep this PR open: the SQLite state direction matches the architecture work tracked in https://github.com/openclaw/openclaw/issues/78096, but the latest head still has source-backed migration and access-control regressions, carries protected `security` and `maintainer` labels, and lacks after-fix real behavior proof.

## What This Changes

This PR moves runtime, session, transcript, plugin, cron, device, pairing, backup, app, docs, and test state paths from JSON/file stores toward global and per-agent SQLite databases.

## Best Possible Solution

Keep the database-first direction, but land only after maintainer/security review verifies recency-safe imports, bounded transcript reads, focused regression coverage, and redacted real behavior proof for upgrade, runtime, and backup paths.

## Reproduction Assessment

Yes for the review findings by source inspection: latest head `8c1ad6a3276be9d628de762776dd321b24fde554` still shows overwrite-style legacy/global imports and recent transcript reads that ignore byte/line caps. I did not run live migration scenarios because this review is read-only and the PR lacks real behavior proof.

## Solution Assessment

No. The SQLite storage-boundary refactor is a plausible direction for https://github.com/openclaw/openclaw/issues/78096, but the current patch is not the best mergeable solution until migration conflict handling preserves newer SQLite state and hot recent-read contracts remain bounded.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve newer per-agent session rows during migration:** `src/infra/state-migrations.ts:2660-2663`
  - body: When `openclaw doctor --fix` runs after upgraded runtime traffic has already updated per-agent `session_entries`, this conflict handler always replaces the per-agent row with the legacy global-table payload and then drops the global table. That can roll session metadata backward permanently, so the upsert needs timestamp-aware merge or skip behavior instead of always taking `excluded`.
  - confidence: 0.91
- **[P1] Merge pairing imports instead of replacing access state:** `src/pairing/pairing-store.ts:731-747`
  - body: The channel pairing migration overwrites current SQLite requests and account allowlists with legacy file contents before deleting the source files. A delayed doctor run can restore revoked senders or remove newly approved senders, so this import needs merge or recency checks.
  - confidence: 0.9
- **[P1] Do not restore revoked device tokens from legacy JSON:** `src/infra/device-auth-store.ts:95-97`
  - body: `importLegacyDeviceAuthFileToSqlite` writes the entire legacy token store into SQLite and then removes the JSON source. If tokens were rotated or revoked in SQLite before doctor runs, this can permanently reintroduce stale credentials.
  - confidence: 0.89
- **[P1] Honor recent transcript byte and line caps:** `src/gateway/session-transcript-readers.ts:253-259`
  - body: The recent-read path loads all SQLite transcript messages and slices only by `maxMessages`, ignoring the byte and line caps that callers use to bound large transcript tails. Long sessions can now force full-history reads and memory spikes where a bounded tail was expected.
  - confidence: 0.9
- **[P2] Count UTF-8 bytes when pruning cron run logs:** `src/cron/run-log.ts:203-204`
  - body: `pruneCronRunLogRows` measures `maxBytes` with `row.entry_json.length`, which counts UTF-16 code units rather than stored UTF-8 bytes. Non-ASCII run entries can exceed the documented byte cap, so use UTF-8 byte length for the retention budget.
  - confidence: 0.84
