## Summary

Keep open: this is an active, broad database-first runtime refactor with protected `security` and `maintainer` labels, missing current-head real behavior proof, and sampled PR-head blockers around runtime support and merge-safe legacy imports.

## What This Changes

This PR moves runtime state from JSON, JSONL, sidecar files, and file-backed stores into global and per-agent SQLite stores across core, apps, plugins, docs, scripts, and tests.

## Best Possible Solution

Keep the database-first architecture under maintainer/security review, preserve or explicitly sequence Node support changes, make legacy imports merge-safe, and require current-head populated-install proof before merge.

## Reproduction Assessment

Yes, for the PR-head blockers by source inspection: the branch raises the runtime floor while current main still supports Node 22, and sampled migration paths still replace or overwrite active SQLite state. I did not establish a live populated-install reproduction in this read-only pass.

## Solution Assessment

No, not as-is. The storage direction is reasonable, but merge-safe migration semantics, runtime-support sequencing, and current-head real behavior proof are still needed before this is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve Node 22 support or split the runtime bump:** `package.json:1836`
  - body: Current main and repo policy still support Node >=22.16.0, but this PR changes the package engine to Node 24 and removes the Node 22 compatibility lane inside a storage migration. That can strand supported installs before users can run the migration unless maintainers explicitly approve and sequence the breaking runtime change.
  - confidence: 0.9
- **[P1] Merge legacy transcript imports instead of replacing rows:** `src/commands/doctor/state-migrations.ts:306-311`
  - body: `importLegacyTranscriptFileToSqlite` calls the replace helper for each legacy source file. That helper deletes all SQLite transcript rows for the session before inserting, so same-session checkpoint files or newer SQLite rows can be dropped by a delayed doctor run.
  - confidence: 0.92
- **[P1] Avoid clobbering live device-auth state during migration:** `src/commands/doctor/legacy/device-auth-store.ts:39-41`
  - body: This importer writes the legacy `device-auth.json` snapshot into active SQLite state and then removes the file. Runtime token changes use the same state key, so a delayed doctor run can restore stale or revoked tokens and drop newly issued ones.
  - confidence: 0.9
- **[P1] Merge channel pairing imports with live state:** `src/commands/doctor/legacy/channel-pairing.ts:231-247`
  - body: The pairing importer replaces request and allowlist entries from legacy files before deleting those files. If SQLite already has newer approvals or revocations, migration can re-add revoked senders or remove newly approved ones.
  - confidence: 0.86
- **[P2] Preserve existing plugin state on migration conflict:** `src/plugin-sdk/migration-runtime.ts:55-60`
  - body: The plugin migration helper resolves `plugin_state_entries` conflicts by overwriting the current row with the imported value. Because plugin state is live runtime state, delayed sidecar migration can replace newer plugin data with stale legacy data.
  - confidence: 0.85
