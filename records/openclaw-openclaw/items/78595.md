## Summary

Keep open: this is an active, protected, security-sensitive SQLite runtime refactor with current-head blocking findings and insufficient latest-head real behavior proof. The direction is supported in discussion, but the PR still needs maintainer/security sequencing before merge.

## What This Changes

The PR moves runtime/session/transcript/auth/plugin state from JSON, JSONL, and sidecar files into shared and per-agent SQLite stores across core, apps, plugins, docs, and tests.

## Best Possible Solution

Keep the database-first architecture under maintainer/security review, preserve or explicitly sequence Node support policy, make legacy imports merge-safe/source-preserving, and require current-head populated-install proof before merge.

## Reproduction Assessment

Yes for the blocking findings by source inspection: current main supports Node 22.16+, while PR head raises the engine/runtime guard to Node 24 and several legacy importers still overwrite live SQLite state. I did not run a populated-install migration in this read-only review.

## Solution Assessment

No, not as-is. The SQLite storage direction is reasonable, but the current patch needs a Node support decision, merge-safe migration semantics, and latest-head real behavior proof before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve Node 22 support or split the runtime bump:** `package.json:1836`
  - body: Current main still supports and tests Node 22.16+, but this PR raises the package engine to Node 24. Supported installs can fail before users reach the SQLite migration unless maintainers explicitly approve and sequence this breaking runtime change.
  - confidence: 0.9
- **[P1] Merge legacy transcript imports instead of replacing rows:** `src/commands/doctor/state-migrations.ts:306-311`
  - body: Each legacy transcript import uses `replaceSqliteSessionTranscriptEvents`, which deletes all existing rows for that session before inserting the current file. Real installs can have multiple legacy files or newer SQLite rows for one session, so a delayed doctor run can drop transcript history.
  - confidence: 0.92
- **[P1] Skip stale exec approvals when SQLite state exists:** `src/commands/doctor/legacy/exec-approvals.ts:44-50`
  - body: This importer writes the legacy approvals snapshot over `exec.approvals/current` and then removes the file without checking for newer SQLite state. A delayed doctor run can roll back runtime approval policy and persist that rollback.
  - confidence: 0.9
- **[P1] Preserve current device identity during legacy import:** `src/commands/doctor/legacy/device-identity.ts:150-152`
  - body: The legacy device identity importer writes the old identity snapshot into SQLite and then removes the source file. If runtime has already generated or rotated identity state in SQLite, doctor can restore stale keys and invalidate newer trust relationships.
  - confidence: 0.9
- **[P1] Merge channel pairing imports with live state:** `src/commands/doctor/legacy/channel-pairing.ts:231-247`
  - body: The channel-pairing importer replaces request and allowlist entries from legacy files and then removes those files. If SQLite already contains newer approvals or revocations, delayed migration can re-add revoked senders or drop newly approved ones.
  - confidence: 0.86
- **[P1] Preserve existing plugin state on migration conflict:** `src/plugin-sdk/migration-runtime.ts:55-60`
  - body: The plugin migration helper resolves `plugin_state_entries` conflicts by replacing the existing row with the imported value. Because plugin state is live runtime state, delayed sidecar migration can overwrite newer plugin data instead of preserving or merging it.
  - confidence: 0.85
- **[P1] Avoid replacing live device auth from legacy files:** `src/commands/doctor/legacy/device-auth-store.ts:39-41`
  - body: The device auth importer writes the entire legacy token store into SQLite and then removes the file without checking for newer token state. If runtime has already rotated or revoked tokens, a delayed doctor run can restore stale credentials.
  - confidence: 0.87
- **[P1] Prevent web-push import from overwriting newer state:** `src/commands/doctor/legacy/push-web.ts:48-50`
  - body: The web-push importer persists legacy subscription state and removes the source file without checking for newer SQLite subscriptions. Delayed doctor runs can roll endpoints or VAPID state back to stale file-era values.
  - confidence: 0.84
- **[P2] Do not delete update-check state after parse failures:** `src/commands/doctor/legacy/update-check.ts:53-88`
  - body: `readLegacyStateFile` converts any read or parse failure into an empty object, after which the importer writes that empty state and removes the source file. A malformed legacy file should remain repairable instead of being treated as a successful empty import.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff moves durable auth, approval, identity, pairing, push, and plugin state into SQLite while multiple legacy import paths can still overwrite newer live state.

Concerns:

  - body: `importLegacyExecApprovalsFileToSqlite` writes a legacy approvals snapshot over the live SQLite `exec.approvals/current` key without checking for newer SQLite state, so delayed doctor runs can reintroduce stale exec decisions.
  - confidence: 0.9
- **[high] Legacy device identity can restore stale keys:** `src/commands/doctor/legacy/device-identity.ts:150`
  - body: `importLegacyDeviceIdentityFileToSqlite` writes the legacy identity snapshot into SQLite and then removes the source file. If newer identity material already exists, this can roll back trust identity state.
  - confidence: 0.9
