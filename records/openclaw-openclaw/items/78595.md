## Summary

This PR should stay open: it is a protected `security`/`maintainer`-labeled architecture refactor for https://github.com/openclaw/openclaw/issues/78096, and current PR head still has source-proven blockers in delayed legacy imports, bounded transcript reads, and security-state rollback. It also lacks the required after-fix real behavior proof for an external contributor PR.

## What This Changes

This PR moves runtime state from scattered JSON/JSONL/sidecar stores into global and per-agent SQLite databases, with migration/import paths, backup snapshot handling, docs, and broad session/cron/channel/runtime call-site updates.

## Best Possible Solution

Keep the branch open and harden migration/import semantics so legacy files never overwrite newer SQLite security or runtime state, preserve bounded transcript reads, add focused tests plus live migration/backup proof, then route through maintainer and security review.

## Reproduction Assessment

Yes for the PR review findings at source level: current PR head shows the unsafe import and unbounded read paths directly. I did not run live migrations because this review is read-only and the PR lacks real behavior proof.

## Solution Assessment

No. The SQLite direction matches the linked architecture goal, but this implementation is not the narrowest maintainable solution until delayed legacy imports merge or skip newer SQLite state, recent reads stay bounded, and security-sensitive migrations are proven with real data.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve live cron rows during legacy import:** `src/cron/store.ts:292-294`
  - body: `importLegacyCronStoreToSqlite` writes the legacy store into SQLite and then `writeCronJobsToSqlite` deletes any existing SQLite jobs missing from that stale file. If an upgraded gateway has already created or updated cron jobs before doctor runs, this migration can roll those jobs back or drop them permanently when it removes the legacy source.
  - confidence: 0.89
- **[P1] Honor recent-read byte and line limits:** `src/gateway/session-transcript-readers.ts:283-287`
  - body: The recent-read path now loads the full SQLite transcript and slices only by message count, ignoring `maxBytes` and `maxLines`. Callers that ask for a bounded tail can still materialize large transcripts, regressing latency and memory behavior for long sessions.
  - confidence: 0.9
- **[P1] Merge pairing allowlists instead of replacing them:** `src/pairing/pairing-store.ts:743-747`
  - body: During legacy channel pairing import, the SQLite allowlist for an account is replaced by the legacy file contents. A delayed doctor run can therefore re-add revoked senders or remove approvals made after upgrade, which is a security-sensitive access-control rollback.
  - confidence: 0.87
- **[P2] Pass the doctor env into plugin-binding import:** `src/commands/doctor-sqlite-state.ts:300-302`
  - body: `maybeRepairLegacyRuntimeStateFiles` probes with the supplied `env`, but plugin-binding approvals are imported without passing that environment. Programmatic doctor runs targeting a non-default state dir can import from or delete the wrong approval file.
  - confidence: 0.82
