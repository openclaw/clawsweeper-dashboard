## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep transcript appends atomic across processes:** `src/config/sessions/transcript-append.ts:142-166`
  - body: `appendSessionTranscriptMessage` still performs idempotency and parent/tail selection outside the final SQLite insert and only serializes through a process-local queue. Concurrent gateway/CLI/worker writers can read the same tail, miss the same idempotency key, and insert duplicate or conflicting parent chains.
  - confidence: 0.86
- **[P1] Run legacy cron imports when jobs need no rewrite:** `src/commands/doctor-cron.ts:264-266`
  - body: `maybeRepairLegacyCronStore` can return before importing `jobs-state.json` and cron run logs when `jobs.json` is already normalized. That leaves legacy runtime state stranded even though doctor prompted for repair.
  - confidence: 0.86
- **[P1] Enforce bounded recent transcript reads:** `src/gateway/session-utils.fs.ts:299-303`
  - body: The SQLite recent-read paths described in review comments ignore byte/line caps or discard `maxBytes`, so callers that pass caps for status and memory preflight can still trigger full-history scans on large sessions.
  - confidence: 0.85
- **[P1] Snapshot SQLite databases before archiving state:** `src/infra/backup-create.ts:372-390`
  - body: Archiving live SQLite paths directly can capture an inconsistent database, WAL, and SHM set while writes are active. Backup should create and verify SQLite-consistent snapshots before adding state databases to the archive.
  - confidence: 0.84
- **[P2] Avoid overwriting live commitments during legacy import:** `src/commands/doctor-sqlite-state.ts:1`
  - body: The review context reports that legacy commitment import deletes and replaces all commitment rows whenever the old JSON file exists. Running doctor after SQLite has newer rows can roll state back and drop live commitments.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff moves sensitive durable runtime state into SQLite and still has concrete integrity, transaction, and backup-consistency concerns.

Concerns:

- **[medium] Live SQLite backup may be inconsistent:** `src/infra/backup-create.ts:372`
  - body: Backing up live WAL-mode database files without a verified SQLite snapshot risks restoring an inconsistent state database.
  - confidence: 0.84
