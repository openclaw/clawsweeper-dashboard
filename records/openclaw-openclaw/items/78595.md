  - body: Current main still supports Node 22.16+, but this PR raises the package engine to Node 24. That can block supported installs before users reach migration unless maintainers explicitly approve and sequence the breaking runtime change.

  - body: Each legacy transcript import replaces all rows for the session before inserting the current file. Real installs can have multiple legacy transcript files or newer SQLite rows for one session, so a delayed doctor run can drop transcript history.
  - confidence: 0.92
- **[P1] Skip stale exec approvals when SQLite state exists:** `src/commands/doctor/legacy/exec-approvals.ts:44-50`
  - body: The importer writes the legacy approvals snapshot over the active SQLite key and removes the source without checking for newer SQLite policy, so a delayed doctor run can roll approval state backward.
  - confidence: 0.9
- **[P1] Preserve current device identity during legacy import:** `src/commands/doctor/legacy/device-identity.ts:150-152`
  - body: The importer writes the old identity snapshot into SQLite and then removes the source file. If runtime already generated or rotated identity state, doctor can restore stale keys and invalidate newer trust relationships.
  - confidence: 0.9
- **[P1] Merge channel pairing imports with live state:** `src/commands/doctor/legacy/channel-pairing.ts:231-247`
  - body: The pairing importer replaces request and allowlist entries from legacy files, then removes those files. If SQLite already contains newer approvals or revocations, migration can re-add revoked senders or drop newly approved ones.
  - confidence: 0.86
- **[P1] Preserve existing plugin state on migration conflict:** `src/plugin-sdk/migration-runtime.ts:55-60`
  - body: The plugin migration helper resolves `plugin_state_entries` conflicts by replacing the existing row with the imported value. Because plugin state is live runtime state, delayed sidecar migration can overwrite newer plugin data.
  - confidence: 0.85
