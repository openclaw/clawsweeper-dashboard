## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Re-allow configuredAgentsOnly in sessions.list params:** `src/gateway/protocol/schema/sessions.ts:49-64`
  - body: `sessions.list` still receives `configuredAgentsOnly` from the Control UI, but the PR-head schema no longer declares it while rejecting additional properties. Normal session loads can fail validation unless the field is restored or every client stops sending it.
  - confidence: 0.94
- **[P1] Preserve live device tokens during legacy import:** `src/infra/device-auth-store.ts:95-97`
  - body: The legacy device-auth importer writes the whole legacy token store into the live SQLite record and then removes the file. If users rotate or revoke tokens before running doctor, doctor can restore stale tokens permanently.
  - confidence: 0.9
- **[P1] Merge channel pairing imports instead of replacing state:** `src/pairing/pairing-store.ts:732-747`
  - body: The channel pairing migration replaces current SQLite requests and account allowlists with legacy file contents before deleting those files. A delayed doctor run can re-add revoked senders or remove newly approved ones.
  - confidence: 0.89
- **[P1] Treat missing legacy pairing files as absent:** `src/infra/pairing-files.ts:96-107`
  - body: The import gates on `pending !== undefined` and `paired !== undefined`, but missing sibling reads can be coerced into empty records. Partial legacy pairing state can therefore wipe the current SQLite pending or paired side.
  - confidence: 0.88
- **[P2] Honor byte caps for recent transcript reads:** `src/gateway/session-transcript-readers.ts:283-287`
  - body: Recent transcript readers materialize all SQLite events and ignore `maxBytes`/`maxLines`, so large sessions can force full-history scans where callers requested a bounded tail.
  - confidence: 0.87
