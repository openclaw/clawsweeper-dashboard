## Summary

Keep open: this protected storage/security PR is not implemented on current main and still has source-backed P1 migration rollback findings plus missing real behavior proof, so it needs maintainer/security review rather than cleanup closure.

## What This Changes

The PR moves runtime, session, transcript, cron, plugin, device, pairing, backup, UI, and protocol state from JSON/JSONL/sidecar files into global and per-agent SQLite stores with doctor migrations, docs, tests, and generated schema/protocol updates.

## Best Possible Solution

Keep the database-first direction in maintainer review, make legacy imports conflict-aware and non-destructive, preserve recovery files on malformed input, and require real upgrade, backup, startup, and channel/plugin proof before merge.

## Reproduction Assessment

Yes for the current PR blockers by source inspection: delayed doctor imports can replace newer live SQLite state or delete recovery files. I did not run a live migration because this review is read-only.

## Solution Assessment

No. The SQLite direction may be maintainable, but this patch is not the best merge candidate while legacy imports can roll back trust/runtime state and real upgraded-run proof is absent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve live device-auth state during import:** `src/commands/doctor/legacy/device-auth-store.ts:39-41`
  - body: `importLegacyDeviceAuthFileToSqlite` writes the full legacy snapshot into the SQLite-backed device-auth store and then removes the file. If the upgraded runtime has already rotated or revoked tokens, a later doctor run can restore stale tokens or drop newer approvals.
  - confidence: 0.9
- **[P1] Merge legacy commitments instead of replacing rows:** `src/commands/doctor/legacy/commitments.ts:97-98`
  - body: This import routes a legacy `commitments.json` snapshot through `saveCommitmentStore`, whose SQLite path deletes all current commitment rows before inserting the imported records. A delayed doctor run can therefore roll newer commitment status or attempts back to stale file-era data.
  - confidence: 0.88
- **[P1] Keep newer conversation bindings on conflicts:** `src/commands/doctor/state-migrations.ts:1021-1036`
  - body: The migration conflict handler always replaces existing `current_conversation_bindings` rows with the imported legacy row before deleting the source. If runtime has already updated the binding, doctor can restore stale routing metadata and misroute active delivery.
  - confidence: 0.87
- **[P1] Reject malformed cron stores before saving empty jobs:** `src/commands/doctor/legacy/cron-store.ts:129-134`
  - body: `loadLegacyCronStoreForMigration` coerces unexpected parsed payloads into an empty jobs list. The import path can then save that empty store and remove the source file, silently erasing recoverable cron definitions instead of preserving the file for operator repair.
  - confidence: 0.84
- **[P1] Merge pairing allowlists instead of replacing them:** `src/commands/doctor/legacy/channel-pairing.ts:241-247`
  - body: The channel pairing importer assigns each legacy allowlist directly onto the SQLite state account entry and then removes the JSON file. A delayed doctor run can re-add revoked senders or drop newly approved senders from access-control state.
  - confidence: 0.86
