## Summary

This PR should stay open because it is an active, protected, security-labeled database-first runtime refactor with concrete migration-safety blockers still visible at the current head.

## What This Changes

The PR moves OpenClaw runtime state from JSON/JSONL and sidecar files into typed SQLite global and per-agent databases across gateway, sessions, transcripts, auth, plugins, apps, docs, and tests.

## Best Possible Solution

Keep the branch open for explicit maintainer/security review, fix the stale-import overwrite class, preserve recoverable legacy inputs on partial failure, and require current-head real behavior proof plus broad validation before merge.

## Reproduction Assessment

Yes at source level for the blocking migration bugs: start an upgraded runtime so SQLite has newer rows, leave the corresponding legacy file in place, then run doctor import and the current code overwrites SQLite from the legacy file. I did not run that scenario end to end in this read-only review.

## Solution Assessment

No for the current patch as-is. The SQLite direction appears intentional, but migration must treat legacy files as one-time inputs that cannot clobber newer live SQLite state and should not delete recoverable sources after failed or partial import.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Skip stale device identity import when SQLite state exists:** `src/commands/doctor/legacy/device-identity.ts:150-152`
  - body: The legacy identity importer always writes `identity/device.json` into the live SQLite identity row and then removes the file. If a user starts the upgraded runtime before running doctor, SQLite can already contain a newer generated or rotated device identity; a later doctor run would roll it back to stale file-era keys and make that rollback durable.
  - confidence: 0.88
- **[P1] Prevent cron legacy import from overwriting live SQLite rows:** `src/commands/doctor/legacy/cron-store.ts:178-181`
  - body: The cron migration treats legacy job/runtime files as authoritative even when the SQLite cron store may already have newer runtime updates. A delayed doctor run can replace newer schedule state or run history with stale legacy values, causing mistimed or duplicate scheduling after the source files are removed.
  - confidence: 0.86
- **[P1] Skip stale exec approvals import when SQLite state exists:** `src/commands/doctor/legacy/exec-approvals.ts:34-35`
  - body: This importer writes the legacy `exec-approvals.json` snapshot directly into the active SQLite approvals key and then deletes the file. Because normal approval changes write the same SQLite state, running doctor later can roll back newer approvals or revocations to stale file-era policy.
  - confidence: 0.84
