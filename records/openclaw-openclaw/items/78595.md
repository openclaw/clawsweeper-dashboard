## Summary

Keep open: the protected `security` and `maintainer` labels require explicit handling, and the latest PR head still has source-backed P1/P2 storage, migration, and partitioning blockers plus missing real behavior proof.

## What This Changes

The PR moves runtime/session/transcript/cron/pairing/task/artifact state from JSON and JSONL sidecars into shared and per-agent SQLite stores, with migrations, docs, SDK surfaces, and tests updated.

## Best Possible Solution

Land the SQLite runtime-state model only after preserving agent partitioning, importing configured legacy stores safely, keeping bounded transcript reads and appends correct, cleaning discarded SQLite snapshots, merging cron state by recency, adding regression coverage, and providing redacted real migration/runtime proof with maintainer/security review.

## Reproduction Assessment

Yes for the PR blockers: source inspection of the latest PR head shows the custom-scope fallbacks, migration-detection gap, bounded-read regression, cleanup no-op, and cron overwrite path. This is a PR review, not a current-main user bug reproduction.

## Solution Assessment

No. SQLite-first runtime state may be the right architectural direction, but this patch is not yet the narrowest maintainable landing shape while it can mis-scope agent data, miss migration inputs, weaken bounded reads, and overwrite newer durable scheduler state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve configured store agent scope:** `src/config/sessions/store-backend.sqlite.ts:43-44`
  - body: When `session.store` uses a custom non-canonical path, this fallback assigns rows to `main`. After JSON runtime fallback is removed, non-main agent session metadata can be persisted and queried under the wrong SQLite partition.
  - confidence: 0.92
- **[P1] Preserve transcript agent scope for custom paths:** `src/agents/transcript/session-manager.ts:66`
  - body: `SessionManager.open(sessionFile)` callers do not always pass an agent id, and non-canonical paths fall back to `main`. Custom per-agent transcript layouts can therefore read and write SQLite events under the wrong agent partition.
  - confidence: 0.9
- **[P1] Import configured session.store paths in doctor:** `src/infra/state-migrations.ts:767`
  - body: Doctor migration detection only checks the default per-agent `sessions.json` path. Installs that used a configured `session.store` path can be skipped before runtime JSON support is removed, making visible session history disappear until manual recovery.
  - confidence: 0.9
- **[P1] Make transcript appends cross-process atomic:** `src/config/sessions/transcript-append.ts:142-166`
  - body: The idempotency and parent decisions are based on events loaded before the SQLite insert, and the append queue is process-local. A gateway plus CLI or worker process can append from the same stale tail, duplicating idempotent messages or splitting parent chains.
  - confidence: 0.86
- **[P1] Honor recent-read byte and line caps:** `src/gateway/session-utils.fs.ts:299-303`
  - body: Recent reads now load all scoped SQLite messages and slice only by `maxMessages`. Callers that pass `maxBytes` or `maxLines` to avoid full-history scans can hit avoidable latency and memory spikes on long transcripts.
  - confidence: 0.87
- **[P1] Preserve transcript agent scope during import:** `src/commands/doctor-session-transcripts.ts:84`
  - body: Doctor import maps transcripts outside the canonical `agents/<id>/sessions` shape to `main` and then removes the source file. Custom non-main transcript locations can therefore be migrated into the wrong partition and disappear from intended history.
  - confidence: 0.86
- **[P2] Delete discarded checkpoint snapshots:** `src/gateway/session-compaction-checkpoints.ts:281-285`
  - body: Compaction checkpoint capture writes full snapshot transcripts into SQLite, but cleanup is a no-op. Failed or discarded compactions can leave unreferenced snapshot rows that accumulate over time.
  - confidence: 0.84
- **[P2] Repair transcripts before deleting legacy JSONL:** `src/infra/state-migrations.ts:963`
  - body: This migration imports legacy transcript JSONL into SQLite and removes the source before the older doctor repair flow can scan it. Broken prompt-rewrite branches can therefore be preserved permanently after upgrade.
  - confidence: 0.82
- **[P2] Merge legacy cron sidecar state by recency:** `src/cron/store.ts:206`
  - body: This import writes the whole legacy sidecar payload into SQLite. If the gateway already has newer SQLite schedule state and a stale sidecar remains, doctor can roll scheduler state backward and cause duplicate or mistimed runs.
  - confidence: 0.82
