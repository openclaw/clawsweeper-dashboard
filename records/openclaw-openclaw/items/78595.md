## Summary

Keep this PR open. The SQLite direction is well scoped and the thread has useful empirical migration and follow-up seam work, but current head still has source-visible stale legacy-import paths that can overwrite live SQLite state, the PR has protected `security` and `maintainer` labels, and current-head real behavior proof is not sufficient for merge.

## What This Changes

The PR refactors OpenClaw runtime state from JSON/JSONL and sidecar stores into global and per-agent SQLite databases across sessions, transcripts, auth, plugins, channels, apps, docs, and tests.

## Best Possible Solution

Keep the database-first architecture, but require stale-safe import merge/skip semantics, explicit Node support approval, and current-head migration/runtime proof before merge.

## Reproduction Assessment

Yes, at source level: run the upgraded runtime so SQLite has newer rows, leave a corresponding legacy JSON file in place, then run doctor import. I did not execute the destructive scenario, but the current PR-head source shows legacy snapshots replacing active SQLite state and deleting the source.

## Solution Assessment

No, not as currently implemented. The SQLite storage boundary is the right direction, but legacy importers need freshness-aware merge/skip behavior and the Node runtime bump needs explicit maintainer release approval.
