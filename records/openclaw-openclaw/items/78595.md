## Summary

Keep this PR open. It is an active, unmerged, protected-label storage rewrite with concrete P1 review findings around migration safety, session-row correctness, and security-sensitive runtime state rollback.

## What This Changes

The PR moves OpenClaw runtime state from JSON/JSONL/file-backed stores into global and per-agent SQLite stores across sessions, transcripts, auth, plugin/channel state, apps, migration, backup, docs, scripts, and tests.

## Best Possible Solution

Keep the database-first direction under maintainer review, but require safe merge-or-skip semantics for legacy imports, normalized session-key writes, current-head runtime proof, and a clear decision on which public storage seams land here versus follow-up PRs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? For the broad feature, reproduction is not a single current-main bug path; for the blocking findings, yes, source inspection of the PR head shows the unsafe write/delete and raw-key update paths, and the discussion includes populated-install migration evidence for transcript clobbering.

## Solution Assessment

Is this the best way to solve the issue? Not yet. The SQLite architecture may be the right direction, but the current branch is not the narrowest maintainable solution until migration safety, normalized session updates, security rollback hazards, and validation proof are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Skip stale exec approvals before deleting legacy state:** `src/commands/doctor/legacy/exec-approvals.ts:34-35`
  - body: This import writes the legacy `exec-approvals.json` snapshot into the live SQLite approvals store and then removes the source file without checking whether SQLite already contains newer approval policy. A delayed doctor run can roll back allowlists/socket settings and make that stale policy durable.
  - confidence: 0.9
- **[P1] Reject unreadable auth state before deleting source:** `src/commands/doctor/legacy/auth-profile-state.ts:30-33`
  - body: `loadJsonFile(statePath)` can return `undefined` for malformed or unreadable legacy JSON; `coerceAuthProfileState` turns that into `{}`, and the save path deletes the SQLite state before the source file is unlinked. This can silently wipe newer auth routing state and remove the recoverable legacy input.
  - confidence: 0.88
- **[P1] Normalize the key used for optimistic session patches:** `src/config/sessions/store.ts:104-108`
  - body: `patchSessionEntry` may read an existing row through the normalized-key fallback, but it then applies the compare-and-swap using the raw `options.sessionKey`. Non-canonical key inputs can conflict forever or create/update the wrong alias row instead of the canonical session row.
  - confidence: 0.86
- **[P1] Merge legacy transcript files instead of replacing rows:** `src/commands/doctor/state-migrations.ts:307-312`
  - body: Legacy import calls `replaceSqliteSessionTranscriptEvents`, whose implementation deletes all existing transcript rows for the session before inserting the current file. Real installs can have multiple legacy transcript files for the same session, so importing each file by replacement drops events imported from earlier files.
  - confidence: 0.87
