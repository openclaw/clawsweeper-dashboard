## Summary

This PR should stay open: it is a protected, security-sensitive SQLite runtime-state refactor, is currently dirty, and current head still has source-proven migration paths that can overwrite newer SQLite state from stale legacy files.

## What This Changes

The branch moves runtime state from JSON/JSONL and sidecar stores into global and per-agent SQLite databases with Kysely schemas, doctor/migration paths, backup/export changes, API/UI renames, docs, and broad tests.

## Best Possible Solution

Keep the SQLite direction, fold or land the focused migration-clobber fixes, keep companion API seams in their linked follow-ups, resolve the dirty branch, and require current-head populated-upgrade proof before merge.

## Reproduction Assessment

Yes, by source inspection: run upgraded runtime first so SQLite rows mutate, then run openclaw doctor --fix while stale legacy files still exist. The reviewed import paths write legacy snapshots into the same SQLite rows and delete the sources; I did not run a live migration in this read-only review.

## Solution Assessment

No for merging as-is: the database-first storage model is plausible, but the migration boundary needs recency/idempotency safeguards and current-head real upgrade proof before it is the maintainable solution.
