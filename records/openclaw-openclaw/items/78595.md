## Summary

Keep open: this is an active protected storage refactor PR, and current head still has replay-unsafe migration paths, security-state rollback risks, and no sufficient after-fix real behavior proof.

## What This Changes

The branch refactors runtime, session, transcript, plugin, scheduler, channel, and artifact state from file-backed stores into shared and per-agent SQLite databases with migration, SDK, docs, UI, and test updates.

## Best Possible Solution

Keep the SQLite refactor open until legacy imports are replay-safe and idempotent, security-state rollbacks are blocked, locator-based transcript reading is supported, and redacted upgraded-runtime proof is supplied.

## Reproduction Assessment

Yes, for the current PR blockers: source inspection on head 81bd599699eb2ec212ede8ebaba4522cf03d6e05 shows destructive imports, and the populated-install comment demonstrates transcript loss. I did not run a live upgrade migration in this read-only review.

## Solution Assessment

No. The database-first direction may be valid, but the current patch is not the best mergeable form while delayed doctor migrations can overwrite newer SQLite state and legacy transcript imports can delete already-imported events.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Merge legacy transcripts instead of replacing rows:** `src/commands/doctor/state-migrations.ts:304-309`
  - body: Legacy transcript migration imports each JSONL file by calling replaceSqliteSessionTranscriptEvents, which deletes existing rows for that session before inserting the current file. Real installs can have multiple files with the same session header, so later imports discard earlier events; the discussion reports 1,424 lost events on a populated snapshot.
  - confidence: 0.94
- **[P1] Preserve live device-auth tokens during import:** `src/commands/doctor/legacy/device-auth-store.ts:39-41`
  - body: This importer writes the legacy device-auth snapshot into the live SQLite token store and removes the source file. If the upgraded runtime has already rotated or revoked tokens in SQLite, a later doctor run can permanently restore stale credentials.
  - confidence: 0.92
- **[P1] Preserve current device identity during import:** `src/commands/doctor/legacy/device-identity.ts:38-40`
  - body: This migration writes the legacy identity snapshot into the live identity store and then deletes the source. A delayed doctor run can roll back a newer SQLite identity/keypair and break existing device trust flows.
  - confidence: 0.9
- **[P1] Merge channel pairing state instead of replacing it:** `src/commands/doctor/legacy/channel-pairing.ts:229-245`
  - body: The migration replaces current pairing requests and account allowlists with legacy file contents before deleting the files. If SQLite already has newer approvals, revocations, or pending requests, a delayed doctor run can roll access control backward.
  - confidence: 0.9
- **[P1] Do not overwrite existing plugin-state rows during migration:** `src/plugin-sdk/migration-runtime.ts:55-60`
  - body: The generic migration helper updates existing plugin_state_entries on conflict, so stale legacy plugin snapshots can overwrite newer runtime plugin state. Many callers delete their legacy source after import, making that rollback hard to inspect or retry.
  - confidence: 0.88
- **[P1] Skip stale cron snapshots when SQLite cron state exists:** `src/commands/doctor/legacy/cron-store.ts:178-181`
  - body: This import writes legacy cron jobs and runtime state into SQLite and then removes the source file without checking whether runtime already saved newer cron state. Running doctor after the upgraded gateway has run can roll schedules or run metadata backward.
  - confidence: 0.86
- **[P1] Preserve newer commitments during legacy import:** `src/commands/doctor/legacy/commitments.ts:96-98`
  - body: The commitment importer saves the coerced legacy store and removes commitments.json; saveCommitmentStore replaces the commitments table. A delayed doctor run can drop newer commitments already written by SQLite runtime.
  - confidence: 0.87
- **[P2] Expose a supported locator transcript reader:** `src/plugin-sdk/session-store-runtime.ts:23-26`
  - body: Context-engine plugins receive transcript locators, but the public runtime export only exposes loadSqliteSessionTranscriptEvents by agent/session identifiers and does not export a supported locator parser or resolver. Plugins would need to parse sqlite-transcript internals to keep reading transcripts.
  - confidence: 0.83
