## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Re-allow configuredAgentsOnly in sessions.list params:** `src/gateway/protocol/schema/sessions.ts:64`
  - body: The Control UI still sends `configuredAgentsOnly` on sessions.list requests, but the schema no longer declares that field while `additionalProperties` remains false. Normal session loads from the UI can be rejected instead of returning rows unless the field is restored or the client stops sending it.
  - confidence: 0.94
- **[P1] Preserve newer per-agent session rows during migration:** `src/infra/state-migrations.ts:2709-2714`
  - body: This conflict handler always takes the legacy global row. If upgraded runtime traffic already wrote a newer per-agent `session_entries` row before doctor/migrate runs, migration can roll that metadata backward permanently; preserve the newer row or merge by recency.
  - confidence: 0.91
- **[P1] Treat missing legacy pairing files as non-imports:** `src/infra/pairing-files.ts:96-107`
  - body: `readJsonIfExists` returns null for missing files, but these checks accept anything except `undefined`, so a partial legacy pairing state writes `{}` over the missing sibling SQLite key. That can wipe current pending or paired device/node state during doctor import.
  - confidence: 0.9
- **[P1] Avoid replacing live device auth state from legacy files:** `src/infra/device-auth-store.ts:95-97`
  - body: The importer writes the entire legacy `device-auth.json` payload into the live SQLite store and removes the source file. A delayed doctor run after token rotation or revocation can restore stale credentials, including revoked tokens, with no legacy source left for recovery.
  - confidence: 0.89
- **[P1] Merge legacy pairing imports instead of replacing access state:** `src/pairing/pairing-store.ts:731-747`
  - body: The legacy channel pairing import replaces current SQLite requests and account allowlists with file-era data before deleting those files. Running doctor after approvals or revocations happened in SQLite can re-authorize stale senders or remove newer approvals.
  - confidence: 0.88
- **[P2] Honor byte caps for recent transcript reads:** `src/gateway/session-transcript-readers.ts:278-287`
  - body: The recent read and usage paths load all SQLite transcript events and ignore `maxBytes`, so large sessions can force full-history reads in hot paths where callers requested bounded tails.
  - confidence: 0.87

## Security Review

Status: needs_attention

  - body: The importer replaces SQLite pairing requests and allowlists with legacy JSON contents, then deletes the legacy files. A delayed migration can re-add revoked senders or remove newly approved senders.
