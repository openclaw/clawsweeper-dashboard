## Summary

This PR must stay open: it is protected by `security` and `maintainer` labels and remains a broad runtime-state migration with blocking agent-scope/migration concerns and no contributor-supplied real behavior proof. The SQLite direction may be right, but the current branch needs fixes and human security/maintainer review before landing.

## What This Changes

The branch moves runtime session, transcript, cron, pairing/device, commitment, TUI, artifact, and related state from JSON/JSONL sidecars into shared and per-agent SQLite stores, with doctor migration paths, docs, generated SDK baselines, changelog, and tests updated.

## Best Possible Solution

Land the database-first runtime model only after the agent-scope, migration, transcript concurrency, bounded-read, cleanup, and repair-ordering blockers have regression coverage and the contributor adds redacted real doctor-migration plus SQLite-runtime proof.

## Reproduction Assessment

Yes for the PR blockers: the failing paths are source-reproducible from the proposed diff against current file-backed contracts for custom `session.store`, transcript ownership, doctor migration, cross-process appends, and bounded recent reads. This is a broad refactor review, not a single end-user bug reproduction.

## Solution Assessment

No. SQLite-first runtime state may be the right architectural direction, but this patch is not yet the best landing shape while it can mis-scope custom state, miss legacy migration inputs, weaken transcript coordination, and lacks real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve configured store agent scope:** `src/config/sessions/store-backend.sqlite.ts:43`
  - body: When `session.store` uses a custom path that is not shaped like the canonical per-agent `sessions.json` path, this fallback assigns rows to the default agent. After runtime JSON fallback is removed, non-main agent sessions can be persisted and queried under the wrong SQLite partition.
  - confidence: 0.92
- **[P1] Require agent scope for custom transcript paths:** `src/agents/transcript/session-manager.ts:66`
  - body: `SessionManager.open(sessionFile)` callers do not always pass an agent id, and non-canonical transcript paths fall back to the default agent. Custom per-agent transcript layouts can therefore read and write SQLite events under the wrong agent partition.
  - confidence: 0.9
- **[P1] Import configured session.store paths in doctor:** `src/infra/state-migrations.ts:767`
  - body: Doctor migration detection only checks the default per-agent `sessions.json` path. Installs that used a configured `session.store` path can be skipped before runtime JSON support is removed, so visible session history is not imported unless users recover it manually.
  - confidence: 0.9
- **[P1] Make transcript appends cross-process atomic:** `src/config/sessions/transcript-append.ts:142`
  - body: The new append queue is process-local while idempotency and parent-link decisions are made before the SQLite insert. A gateway plus CLI or worker process can read the same stale event list and then both append, duplicating messages or splitting parent chains.
  - confidence: 0.88
- **[P1] Honor recent-read byte and line caps:** `src/gateway/session-utils.fs.ts:299-303`
  - body: `readSessionMessagesAsync` loads the whole SQLite transcript before slicing by message count and ignores byte/line limits. Callers that pass those caps to avoid full-history scans can now hit avoidable memory and latency spikes on long sessions.
  - confidence: 0.87
- **[P1] Preserve transcript agent scope during import:** `src/commands/doctor-session-transcripts.ts:84`
  - body: Doctor import maps transcripts outside the canonical `agents/<id>/sessions` shape to the default agent and then removes the source file. Custom non-main transcript locations can be migrated into the wrong agent partition and disappear from intended history.
  - confidence: 0.86
- **[P2] Delete discarded checkpoint snapshots:** `src/gateway/session-compaction-checkpoints.ts:284-285`
  - body: Compaction checkpoint capture writes full snapshot transcripts into SQLite, but cleanup is a no-op. Failed or discarded compactions can leave unreferenced snapshot rows that accumulate over time.
  - confidence: 0.84
- **[P2] Repair transcripts before deleting legacy JSONL:** `src/infra/state-migrations.ts:963`
  - body: The migration imports legacy transcript JSONL into SQLite and removes the source file before the doctor transcript repair helper can run. Broken prompt-rewrite branches can therefore be preserved permanently after upgrade.
  - confidence: 0.82
