## Summary

Keep this PR open: it is a recent, protected `security`/`maintainer` XL storage refactor, and current PR-head source still has blocking stale-legacy-import paths that can overwrite newer SQLite runtime/security state.

## What This Changes

The PR moves runtime state from JSON/JSONL and sidecar files into typed SQLite global and per-agent stores across gateway, sessions, transcripts, auth, plugins, apps, docs, and tests.

## Best Possible Solution

Keep the SQLite refactor direction, but require maintainer/security review to fix legacy imports so stale files cannot clobber newer SQLite state and to prove the current head with broad migration/runtime validation.

## Reproduction Assessment

Yes, at source level: run the upgraded runtime so SQLite has newer rows, leave a stale legacy file in place, then run doctor import; the sampled importers overwrite SQLite from the legacy file and remove the source. I did not run the destructive migration scenario in this read-only review.

## Solution Assessment

No, not as currently implemented. The database-first direction is maintainable, but migration must merge or skip stale legacy inputs when live SQLite rows already exist, especially for identity, approvals, and scheduler state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Skip stale device identity import when SQLite state exists:** `src/commands/doctor/legacy/device-identity.ts:150-152`
  - body: The legacy identity importer always writes `identity/device.json` into the live SQLite identity row and then removes the source file. If a user starts the upgraded runtime before running doctor, SQLite can already contain a newer generated or rotated device identity; a later doctor run would roll it back to stale file-era keys and make that rollback durable.
  - confidence: 0.88

  - body: The cron migration treats legacy job/runtime files as authoritative even when the SQLite cron store may already have newer runtime updates. A delayed doctor run can replace newer schedule state or run history with stale legacy values, causing mistimed or duplicate scheduling after the source files are removed.
  - confidence: 0.86
- **[P1] Skip stale exec approvals import when SQLite state exists:** `src/commands/doctor/legacy/exec-approvals.ts:34-35`
  - body: This importer writes the legacy `exec-approvals.json` snapshot directly into the active SQLite approvals key and then deletes the file. Because normal approval changes write the same SQLite state, running doctor later can roll back newer approvals or revocations to stale file-era policy.
  - confidence: 0.84
