## Summary

Keep open: current main has not implemented this SQLite storage refactor, and the latest PR head still has source-confirmed destructive legacy-import paths plus protected `security` and `maintainer` labels.

## What This Changes

This PR moves runtime/session/transcript/cron/plugin/device/pairing/backup/UI/protocol state from file-backed stores to global and per-agent SQLite databases, with doctor migration and docs/test updates.

## Best Possible Solution

Keep the database-first direction in maintainer/security review, make legacy imports conflict-aware and non-destructive, then require real upgrade, backup, startup, and channel/plugin proof before merge.

## Reproduction Assessment

Yes for the confirmed PR blockers by source inspection: latest head ce565a2e79f188c163dbc175a9fef87a3962d741 still has delayed doctor imports that can overwrite or delete live SQLite state. I did not run a live migration because this review is read-only.

## Solution Assessment

No. The storage direction may be maintainable, but the current patch is not the best merge candidate while legacy imports can roll back runtime/trust state and real upgraded-run proof is absent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve live transcript rows on legacy import:** `src/commands/doctor/state-migrations.ts:304-310`
  - body: `importLegacyTranscriptFileToSqlite` routes through `replaceSqliteSessionTranscriptEvents`, which deletes all existing SQLite transcript events for the session before inserting the legacy JSONL payload. A delayed doctor run after upgraded runtime activity can roll a session back to stale history and then remove the source file.
  - confidence: 0.9
- **[P1] Preserve newer device-auth tokens during import:** `src/commands/doctor/legacy/device-auth-store.ts:39-41`
  - body: This importer writes the legacy `device-auth.json` payload to the live SQLite device-auth store and then deletes the file. If tokens were added, rotated, or revoked after upgrade, a later doctor run can restore stale tokens or drop newer ones.
  - confidence: 0.9
- **[P1] Preserve newer cron rows during legacy repair:** `src/commands/doctor-cron.ts:281-285`
  - body: When a legacy cron store exists, doctor saves that snapshot through the SQLite-backed cron store. `saveCronStore` rewrites conflicting rows and deletes rows absent from the imported file, so a delayed repair can lose newer SQLite cron jobs or runtime state.
  - confidence: 0.88
- **[P1] Reject malformed cron files before empty import:** `src/commands/doctor/legacy/cron-store.ts:107-112`
  - body: Unexpected parsed cron shapes are coerced to an empty jobs list. Because the presence of a legacy store can trigger saving and source removal, malformed legacy input can erase current SQLite cron rows instead of failing safely for operator retry.
  - confidence: 0.84
- **[P1] Preserve Telegram binding semantics during import:** `extensions/telegram/src/state-migrations.ts:143-153`
  - body: The Telegram thread binding importer rebuilds each record with only core routing fields, defaults missing or unrecognized `targetKind` to `subagent`, and drops persisted fields such as idle timeout, max age, label, and metadata. Those fields affect binding expiry and routing behavior, and the source file is removed after import.
  - confidence: 0.86
