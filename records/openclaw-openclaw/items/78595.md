## Summary

Keep open: the database-first direction is useful, but this PR still has protected security/maintainer labels, concrete latest-head migration/security blockers, an explicit Node support policy decision, and insufficient current-head real upgrade proof.

## What This Changes

This PR moves OpenClaw runtime/session/transcript/auth/plugin state from JSON, JSONL, and sidecar files into global and per-agent SQLite stores across core, apps, plugins, docs, and tests.

## Best Possible Solution

Keep the SQLite architecture under maintainer/security review, preserve or explicitly sequence Node 22 support, make every legacy import merge-safe/source-preserving, and require current-head populated-install proof before merge.

## Reproduction Assessment

Yes for the blocking review findings by source inspection: current main still supports Node 22, while this PR raises the engine to Node 24 and latest-head import paths overwrite live SQLite rows from legacy sources. I did not run a populated-install migration in this read-only review.

## Solution Assessment

No, not as-is. The database-first solution is directionally sound, but the current PR needs a Node support decision, merge-safe migration semantics across state owners, and current-head real behavior proof before it is the maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve Node 22 support or split the runtime bump:** `package.json:1836`
  - body: Current main still supports Node >=22.16.0 and has Node 22 compatibility docs/CI, but this PR raises the package engine to >=24.0.0. Supported installs can fail before they run the SQLite migration unless maintainers explicitly approve and sequence this breaking runtime change.
  - confidence: 0.9
- **[P1] Merge legacy transcript imports instead of replacing rows:** `src/commands/doctor/state-migrations.ts:306-311`
  - body: Each legacy transcript file is imported through `replaceSqliteSessionTranscriptEvents`, which deletes all existing rows for that session before inserting the current file. Real installs can have multiple files or newer SQLite rows for one session, so a delayed doctor run can drop transcript history.
  - confidence: 0.91
- **[P1] Skip stale exec approvals when SQLite state exists:** `src/commands/doctor/legacy/exec-approvals.ts:44-50`
  - body: This importer writes the legacy approvals snapshot over `exec.approvals/current` and then removes the file without checking for newer SQLite state. A delayed doctor run can roll back runtime approval policy and make that rollback durable.
  - confidence: 0.9
- **[P1] Preserve current device identity during legacy import:** `src/commands/doctor/legacy/device-identity.ts:150-152`
  - body: The legacy device identity importer writes the legacy identity snapshot into SQLite and then removes the source file. If runtime has already generated or rotated identity state in SQLite, a delayed doctor run can restore stale keys and invalidate newer trust relationships.
  - confidence: 0.88
- **[P1] Merge channel pairing imports with live state:** `src/commands/doctor/legacy/channel-pairing.ts:231-247`
  - body: The channel-pairing importer replaces request and allowlist entries from legacy files and then removes those files. If SQLite already contains newer approvals or revocations, delayed migration can re-add revoked senders or drop newly approved ones.
  - confidence: 0.86
- **[P1] Preserve existing plugin state on migration conflict:** `src/plugin-sdk/migration-runtime.ts:55-60`
  - body: The migration helper resolves `plugin_state_entries` conflicts by replacing the existing row with the imported value. Because plugin state is live runtime state, delayed sidecar migration can overwrite newer plugin data instead of preserving or merging it.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff moves durable auth, approval, identity, pairing, and plugin state into SQLite while several legacy import paths can still overwrite newer authorization/runtime state.

Concerns:

- **[high] Legacy exec approvals can restore stale policy:** `src/commands/doctor/legacy/exec-approvals.ts:44`
  - body: `importLegacyExecApprovalsFileToSqlite` writes a legacy approvals snapshot over the live SQLite `exec.approvals/current` key without checking for newer SQLite state, so delayed doctor runs can reintroduce stale exec decisions.
  - confidence: 0.9
- **[high] Legacy device identity can restore stale keys:** `src/commands/doctor/legacy/device-identity.ts:150`

  - confidence: 0.88

  - confidence: 0.86

  - body: The migration helper unconditionally updates `plugin_state_entries` on conflict, allowing stale sidecar migration data to replace newer live plugin state.
