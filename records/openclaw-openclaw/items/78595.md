## Summary

Keep this PR open: it is a broad database-first storage refactor with protected `security` and `maintainer` labels, and current-head source still shows migration paths that can overwrite newer SQLite state or delete recovery sources before imports are lossless.

## What This Changes

This PR moves runtime/session/auth/plugin/channel state from JSON, JSONL, and sidecar files into global and per-agent SQLite stores with migration tooling, SDK/runtime API updates, docs, UI updates, and tests.

## Best Possible Solution

Land the database-first architecture only after migration imports merge or skip safely against newer SQLite state, source deletion is recoverable, linked follow-up seams are settled or explicitly tracked, and maintainers/security approve latest-head populated-upgrade proof.

## Reproduction Assessment

Yes for the blocker class: source inspection shows replacement imports, live-state overwrites, and immediate source deletion, and the discussion includes populated-install migration output measuring lost transcript rows. I did not execute the PR branch under the read-only review constraint.

## Solution Assessment

No for merge as-is. The database-first direction is coherent, but the current patch needs lossless or conservative import semantics, stale-state guards, recoverable source handling, security review, and latest-head real upgrade proof before it is the best landable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Merge legacy transcript imports before replacing rows:** `src/commands/doctor/state-migrations.ts:304-309`
  - body: `importLegacyTranscriptFileToSqlite` still imports every legacy JSONL file by replacing the session's SQLite transcript rows. Multiple legacy files for the same session, or newer SQLite rows created before a delayed doctor run, can be silently clobbered by the last imported file.
  - confidence: 0.88
- **[P2] Keep transcript sources until import is recoverable:** `src/commands/doctor-session-transcripts.ts:233-238`
  - body: `migrateSessionTranscriptFileToSqlite` removes the source JSONL immediately after a replacement-style write. Until the importer proves it preserved and deduped all expected rows without clobbering newer SQLite data, deleting the original makes recovery difficult.
  - confidence: 0.82
- **[P1] Skip stale exec approvals imports when SQLite state exists:** `src/commands/doctor/legacy/exec-approvals.ts:44-50`
  - body: This migration writes the legacy `exec-approvals.json` payload into the live SQLite KV row and then deletes the file without checking for newer approvals already saved by runtime paths. A delayed doctor run can roll current approval policy or allowlist decisions backward.
  - confidence: 0.87
- **[P1] Merge channel pairing imports with live SQLite state:** `src/commands/doctor/legacy/channel-pairing.ts:231-233`
  - body: When a legacy pairing file exists, the importer replaces `state.requests` with the legacy snapshot before writing SQLite state. If the upgraded runtime already recorded newer pairing requests, doctor can revert pending-pairing state to stale file-era data.
  - confidence: 0.84
- **[P1] Skip stale device-auth imports when SQLite state exists:** `src/commands/doctor/legacy/device-auth-store.ts:35-41`
  - body: The device-auth importer writes the legacy token store into SQLite and removes the source file without checking for newer token issuance or revocation. That can restore stale or revoked device tokens after users have already operated on the SQLite store.
  - confidence: 0.86
