## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reject unreadable auth state before deleting source:** `src/commands/doctor/legacy/auth-profile-state.ts:30-33`
  - body: `loadJsonFile(statePath)` can return `undefined` for malformed or unreadable JSON, which `coerceAuthProfileState` turns into `{}`. `savePersistedAuthProfileState` then deletes SQLite auth state and the legacy file is unlinked, so a repair run can wipe newer auth routing state and remove the recoverable source.
  - confidence: 0.88
- **[P1] Skip stale exec approvals before deleting legacy state:** `src/commands/doctor/legacy/exec-approvals.ts:34-35`
  - body: This importer writes the legacy `exec-approvals.json` snapshot into the live SQLite approval store and immediately deletes the source file without checking existing SQLite state. A delayed doctor run can roll allowlists or socket policy back to stale file-era values and make that rollback durable.
  - confidence: 0.9
- **[P1] Normalize session keys before optimistic patches:** `src/config/sessions/store.ts:116-120`
  - body: `patchSessionEntry` may read an existing row through the normalized-key fallback, but it applies the compare-and-swap using the raw `options.sessionKey`. Non-canonical inputs can conflict forever or update an alias row instead of the canonical session row.
  - confidence: 0.86
- **[P1] Merge legacy transcript files instead of replacing rows:** `src/commands/doctor/state-migrations.ts:307-312`
  - body: Legacy import calls `replaceSqliteSessionTranscriptEvents`, which deletes all existing transcript rows for the session before inserting the current file. Real installs can have multiple legacy transcript files for one session, so each later file can drop events imported from earlier files.
  - confidence: 0.87
