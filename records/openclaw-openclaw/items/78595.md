## Summary

This PR should stay open: it has protected `security` and `maintainer` labels, is still active, and current-head source review shows unresolved stale-legacy-import paths that can overwrite newer SQLite runtime state.

## What This Changes

The PR moves OpenClaw runtime state from JSON/JSONL and sidecar files into global and per-agent SQLite stores, with migrations, Kysely schemas, API/UI/docs updates, and broad tests.

## Best Possible Solution

Keep the database-first architecture direction, but merge only after legacy imports are idempotent or recency-safe, current-head populated-upgrade proof is posted, and protected maintainer/security review signs off.

## Reproduction Assessment

Yes, by source inspection: run upgraded runtime long enough to create newer SQLite rows, leave legacy files present, then run `openclaw doctor --fix`; the sampled import paths overwrite live SQLite state from stale files and delete those sources.

## Solution Assessment

No, not as-is; the SQLite storage boundary is maintainable, but delayed legacy imports need merge, skip, or recency guards before this is the narrowest safe implementation.
