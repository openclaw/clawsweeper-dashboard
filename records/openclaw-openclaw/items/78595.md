## Summary

Keep open. This protected, broad storage-migration PR still has P1 stale-import paths that can overwrite live SQLite auth, pairing, and plugin state, and the thread does not yet include successful current-head real upgrade proof.

## What This Changes

Moves OpenClaw runtime/session/transcript/plugin/channel/scheduler/artifact state from JSON, JSONL, and sidecar stores into shared and per-agent SQLite databases with migration, SDK, protocol, docs, app, and test updates.

## Best Possible Solution

Keep the database-first direction under maintainer/security review, make doctor and migrate idempotent and merge-preserving, deliberately sequence the linked correctness/seam follow-ups, and require redacted populated-install upgrade proof before merge.

## Reproduction Assessment

Yes for the current blocker class by source inspection: create newer SQLite auth, pairing, or plugin-state rows, leave older legacy files present, then run `openclaw doctor --fix`; the import helpers replace those rows. I did not run that live migration locally.

## Solution Assessment

No for merge as-is. The SQLite storage direction may be the right long-term solution, but the current PR needs merge-preserving migrations, security review, and successful current-head populated-install proof before it is the maintainable fix.
