## Summary

This PR should stay open: it is a protected `security`/`maintainer` XL storage refactor with active review findings around stale legacy imports clobbering live SQLite state, plus the PR body itself says it is not merge-ready until cleanup and Testbox validation are green.

## What This Changes

The PR moves OpenClaw runtime state from JSON/JSONL and sidecar stores into shared and per-agent SQLite databases across sessions, transcripts, auth, plugin/channel state, apps, docs, and tests.

## Best Possible Solution

Keep the database-first direction, but finish stale-safe legacy import semantics, public runtime seams that are already split into follow-up work, and current-head Testbox/runtime proof before maintainer/security approval.

## Reproduction Assessment

Yes, at source level: the reviewed migration paths write legacy snapshots into live SQLite stores and then delete source files, so running the upgraded runtime before `doctor --fix` can roll newer SQLite state back to stale file-era state. The provided real-install migration comment also gives empirical transcript-row loss evidence for the same replacement-style bug class.

## Solution Assessment

No, not as currently proposed for merge. The SQLite runtime-state architecture is the right direction, but the migration/import boundary still needs freshness-aware merge/skip behavior and explicit maintainer/security approval before landing.
