## Summary

Keep open: this broad storage refactor has protected `security` and `maintainer` labels, missing real behavior proof, dirty mergeability, and source-proven P1 blockers still present at the latest head.

## What This Changes

The PR moves runtime/session/transcript/plugin/channel/scheduler/artifact state from file-backed stores into SQLite-backed control-plane and per-agent data-plane storage, with related migrations, SDK/protocol/UI updates, docs, and tests.

## Best Possible Solution

Continue the SQLite refactor only after restoring protocol compatibility, making legacy imports non-destructive and recency-aware, supplying real upgraded-run proof, and passing maintainer/security-selected gates.

## Reproduction Assessment

Yes, for the PR blockers by source inspection at head 9942608fb35bf2d3af93161b65932d0e9daf5138. The Control UI sends a now-rejected `configuredAgentsOnly` parameter, and delayed doctor imports can overwrite SQLite-backed security state that runtime token paths mutate.

## Solution Assessment

No. SQLite may be the right architecture, but this patch is not merge-ready while it can reject normal sessions UI calls and roll live security/runtime state backward during migration.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Re-allow configuredAgentsOnly in sessions.list:** `src/gateway/protocol/schema/sessions.ts:39-64`
  - body: `sessions.list` validates params before the handler runs. The schema now rejects unknown params, but the Control UI still sends `configuredAgentsOnly` by default, so the sessions page can receive an invalid-request response instead of rows.
  - confidence: 0.95
- **[P1] Skip stale device-auth imports when SQLite state exists:** `src/commands/doctor/legacy/device-auth-store.ts:39-41`
  - body: This importer writes the whole legacy `identity/device-auth.json` snapshot into the live SQLite token store and then removes the source file. If the upgraded runtime has already rotated or revoked tokens in SQLite, a later doctor run can restore stale credentials permanently.
  - confidence: 0.92
- **[P1] Skip stale bootstrap snapshots on SQLite conflict:** `src/commands/doctor/legacy/device-bootstrap.ts:35-41`
  - body: This migration persists the legacy bootstrap-token snapshot into live device pairing state and deletes the source file. A delayed doctor run can reintroduce tokens that runtime issue/revoke/redeem flows already removed or updated in SQLite.
  - confidence: 0.91
