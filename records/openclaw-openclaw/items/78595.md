## Summary

Keep open: this protected, XL storage PR is not merge-ready while latest-head source still shows migration data-loss/security rollback paths and the discussion lacks after-fix real behavior proof for the current head.

## What This Changes

The PR moves runtime, session, transcript, plugin, channel, scheduler, artifact, and related state from file-backed stores into global and per-agent SQLite databases with migrations, SDK/docs/UI updates, and broad test rewrites.

## Best Possible Solution

Keep the database-first direction open for maintainer/security review, make migrations idempotent and merge-preserving before deleting legacy inputs, then require redacted latest-head upgrade proof on a populated install before merge.

## Reproduction Assessment

Yes for the PR blockers: latest-head source shows destructive transcript replacement and unconditional legacy-over-live imports, and the populated-install comment gives measured transcript-event loss. I did not run a live upgrade migration in this read-only review.

## Solution Assessment

No. The database-first architecture may be the right direction, but this diff is not the best mergeable solution until legacy imports preserve newer SQLite state, avoid deleting partially imported sources, and expose or defer the public read seams deliberately.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Merge legacy transcripts instead of replacing rows:** `src/commands/doctor/state-migrations.ts:304-309`
  - body: Legacy transcript migration imports each file through `replaceSqliteSessionTranscriptEvents`, which deletes the session's existing rows before inserting the current file. Real installs can have multiple legacy files with the same session header, so later imports discard earlier transcript events.
  - confidence: 0.94
- **[P1] Preserve live device-auth tokens during import:** `src/commands/doctor/legacy/device-auth-store.ts:39-41`
  - body: This importer writes the legacy device-auth snapshot into the live SQLite token store and removes the source file. If the upgraded runtime has already rotated or revoked tokens in SQLite, a later doctor run can permanently restore stale credentials.
  - confidence: 0.91
- **[P1] Preserve current device identity during import:** `src/commands/doctor/legacy/device-identity.ts:38-40`
  - body: This migration writes the legacy identity snapshot into the live identity store and then deletes the source file. A delayed doctor run can roll back a newer SQLite identity/keypair and break existing device trust flows.
  - confidence: 0.9
- **[P1] Merge channel pairing state instead of replacing it:** `src/commands/doctor/legacy/channel-pairing.ts:229-245`
  - body: The migration replaces current pairing requests and account allowlists with legacy file contents before deleting the files. If SQLite already has newer approvals, revocations, or pending requests, a delayed doctor run can roll access control backward.
  - confidence: 0.89
- **[P1] Skip stale plugin-state upserts when rows already exist:** `src/plugin-sdk/migration-runtime.ts:55-60`
  - body: `upsertPluginStateMigrationEntry` updates existing plugin-state rows on conflict, so delayed doctor runs can replace newer runtime plugin state with stale legacy snapshots. Many callers remove the source file afterward, making the rollback durable.
  - confidence: 0.86
