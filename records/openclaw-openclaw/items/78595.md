## Summary

Keep open: protected security/maintainer labels apply, and current PR-head source still shows P1 migration rollback/data-loss blockers in legacy import paths plus an unsupported transcript-locator SDK seam.

## What This Changes

This PR moves runtime, session, transcript, plugin, channel, scheduler, and artifact state from file-backed stores into shared and per-agent SQLite databases with migrations, SDK/protocol/UI/docs/tests updates.

## Best Possible Solution

Keep the PR open until migration imports are replay-safe, security-state rollbacks are prevented, locator-based transcript reading is supported, and real upgraded-runtime proof is supplied.

## Reproduction Assessment

Yes, for the PR blockers: source inspection shows destructive legacy imports and the populated-install comment demonstrates transcript loss on a real migration. I did not run a live upgraded-runtime repro in this read-only review.

## Solution Assessment

No. The database-first direction may be right, but the current patch is not merge-ready while delayed doctor migrations can overwrite newer SQLite state and legacy transcript imports can delete already-imported events.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Merge legacy transcripts instead of replacing rows:** `src/commands/doctor/state-migrations.ts:304-309`
  - body: Legacy transcript migration imports each JSONL file with `replaceSqliteSessionTranscriptEvents`, which deletes all existing rows for that session before inserting the current file. Real installs can have multiple files with the same session header, so later imports discard earlier events; a reviewer measured 1,424 lost events on a populated snapshot.
  - confidence: 0.94
- **[P1] Preserve live device-auth tokens on legacy import:** `src/commands/doctor/legacy/device-auth-store.ts:39-41`
  - body: This import writes the legacy device-auth snapshot into the live SQLite token store and removes the source file. If runtime has already rotated or revoked tokens in SQLite, a later doctor run can restore stale credentials permanently.
  - confidence: 0.92
- **[P1] Preserve current device identity during import:** `src/commands/doctor/legacy/device-identity.ts:38-40`
  - body: This migration writes the legacy identity snapshot into the live identity store and then deletes the source. A delayed doctor run can roll back a newer SQLite identity/keypair and break existing device trust flows.
  - confidence: 0.89
- **[P1] Merge channel pairing state instead of replacing it:** `src/commands/doctor/legacy/channel-pairing.ts:229-245`
  - body: The channel pairing migration replaces current requests and allowlists with legacy file contents before deleting the files. If SQLite already has newer approvals or revocations, a later doctor run can roll access control backward.
  - confidence: 0.9
- **[P2] Expose a supported locator transcript reader:** `src/plugin-sdk/session-store-runtime.ts:23-26`
  - body: Context-engine plugins receive transcript locators, but the exported reader still requires agent and session identifiers while the locator parser is not exported. Plugins must parse `sqlite-transcript://` internals or lose supported access to SQLite-backed transcripts.
  - confidence: 0.83
