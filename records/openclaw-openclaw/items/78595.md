## What This Changes

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Re-allow configuredAgentsOnly in sessions.list params:** `src/gateway/protocol/schema/sessions.ts:49-64`
  - body: `sessions.list` still receives `configuredAgentsOnly` from the Control UI, but the schema no longer declares it while rejecting additional properties. Normal session loads can fail validation instead of returning rows unless the field is restored or every client stops sending it.
  - confidence: 0.94
- **[P1] Preserve live device tokens during legacy import:** `src/infra/device-auth-store.ts:95-97`
  - body: The legacy device-auth importer writes the whole `device-auth.json` payload into the live SQLite store and then removes the file. If users rotate or revoke tokens after starting the upgraded runtime but before running doctor, doctor can restore stale tokens permanently.
  - confidence: 0.9
- **[P1] Merge channel pairing imports instead of replacing state:** `src/pairing/pairing-store.ts:732-747`
  - body: The channel pairing migration replaces current SQLite requests and account allowlists with legacy file contents before deleting those files. A delayed doctor run can re-add revoked senders or remove newly approved ones, so the import needs recency-aware merge or skip behavior.
  - confidence: 0.89
- **[P1] Treat missing legacy pairing files as absent:** `src/infra/pairing-files.ts:96-107`
  - body: `readJsonIfExists` can yield a missing sibling as a non-object value that is coerced to `{}`, but the import gates on `!== undefined` and writes that empty record. Partial legacy pairing state can therefore wipe the current SQLite pending or paired side.
  - confidence: 0.88
- **[P2] Honor byte caps for recent transcript reads:** `src/gateway/session-transcript-readers.ts:283-287`
  - body: Recent transcript and usage readers materialize all SQLite events and ignore `maxBytes`/`maxLines`, so large sessions can force full-history scans on hot paths where callers requested a bounded tail.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: Several legacy import paths can roll back newer SQLite auth, pairing, and access-control state during delayed doctor migrations.

Concerns:

- **[high] Device-auth import can restore stale tokens:** `src/infra/device-auth-store.ts:95`
  - body: `importLegacyDeviceAuthFileToSqlite` writes the legacy token store over the live SQLite record and removes the source, so delayed doctor runs can restore revoked or rotated credentials.
  - confidence: 0.9
- **[high] Channel pairing import can restore revoked access:** `src/pairing/pairing-store.ts:732`
  - body: The migration replaces live SQLite pairing requests and allowlists from legacy files, which can re-authorize stale senders or drop newer approvals after users have already changed SQLite state.
  - confidence: 0.89
- **[medium] Missing pairing siblings can wipe live state:** `src/infra/pairing-files.ts:96`
  - body: Missing pending/paired legacy files are treated as writable empty records, so partial legacy imports can clear the current SQLite pairing state for the missing side.
  - confidence: 0.88
