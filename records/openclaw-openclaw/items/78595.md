## Summary

Keep this PR open. It is a recent, protected-label, broad storage refactor that is not implemented on current main, and source inspection of the latest head still shows migration paths that can overwrite newer SQLite state from stale legacy files.

## What This Changes

The PR replaces file-backed runtime stores with global and per-agent SQLite databases across sessions, transcripts, auth/profile state, plugin/channel state, cron/task state, SDK/UI/docs, and doctor migration paths.

## Best Possible Solution

Finish this PR with idempotent, recency-preserving migrations, retry-safe legacy-source handling, populated-upgrade proof on the latest head, and explicit maintainer/security review before landing.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create newer SQLite runtime state, leave the matching stale legacy file in place, then run `openclaw doctor --fix` and observe the importer overwrite SQLite state and remove the source.

## Solution Assessment

No, not yet. The database-first direction is reasonable, but the current patch still needs stale-import guards or safe merge semantics plus populated-upgrade proof before it is the narrowest maintainable solution.

  - body: This importer writes the legacy `auth-state.json` snapshot into the SQLite auth-profile state key and removes the source file. If runtime has already updated profile order, last-good, or usage stats in SQLite, a delayed doctor run rolls that state back permanently.

  - body: The legacy ACP importer writes the whole snapshot into the same SQLite replay tables that live ACP sessions mutate. Because the writer updates existing session rows and deletes/reinserts events, a delayed doctor run can replace newer event history with stale legacy data.
  - confidence: 0.88
- **[P1] Do not replace device auth from stale legacy files:** `src/commands/doctor/legacy/device-auth-store.ts:35-41`
  - body: The device-auth migration unconditionally writes the parsed legacy token store into SQLite and then removes the file. If tokens have already rotated or been revoked in SQLite, this can restore stale credentials and make the rollback durable.
  - confidence: 0.9
- **[P2] Guard Voice Wake imports against stale rollback:** `src/commands/doctor/legacy/voicewake.ts:40-42`
  - body: The Voice Wake migration writes the legacy trigger config to the SQLite-backed runtime store and removes the source file without checking for existing newer SQLite settings, so doctor can silently roll user-updated wake routing back.
  - confidence: 0.83
