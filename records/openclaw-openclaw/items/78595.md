## Summary

Keep open: this is a protected, unmergeable, security-sensitive storage refactor with source-confirmed P1/P2 blockers and missing real behavior proof, so it needs maintainer-led hardening rather than cleanup close.

## What This Changes

The PR replaces JSON/JSONL and sidecar runtime stores with SQLite-backed control-plane, session, transcript, cron, pairing, device, plugin state, VFS/artifact, migration, SDK, and docs paths.

## Best Possible Solution

Land a SQLite runtime-state model only after scoped state, migration, concurrency, backup, integrity, and real upgrade/runtime proof are resolved under maintainer review.

## Reproduction Assessment

Yes for PR review blockers: direct PR-head source inspection shows concrete storage-scope, transaction, bounded-read, cleanup, cron-import, and backup paths. This is patch review evidence rather than a current-main bug reproduction.

## Solution Assessment

No. SQLite-first runtime state may be the right direction, but this patch is not yet the safest landing shape while it can mis-scope agent data, skip migration inputs, weaken bounded reads, and archive live WAL-mode databases unsafely.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve configured store agent scope:** `src/config/sessions/store-backend.sqlite.ts:43-44`
  - body: When `session.store` uses a custom non-canonical path, this fallback writes rows under `main`, so non-main agent session metadata can be persisted and queried from the wrong SQLite partition.
  - confidence: 0.92
- **[P1] Preserve transcript scope for custom paths:** `src/agents/transcript/session-manager.ts:112-119`
  - body: `SessionManager.open(sessionFile)` callers do not always pass an agent id, and non-canonical transcript paths fall back to `main`, mixing non-main transcript events into the wrong partition.
  - confidence: 0.9
- **[P1] Import configured session.store paths in doctor:** `src/infra/state-migrations.ts:763-767`
  - body: Doctor migration detection only checks the default per-agent `sessions.json` path, so installs using a configured `session.store` can be skipped before runtime JSON support is removed.
  - confidence: 0.9
- **[P1] Make transcript appends cross-process atomic:** `src/config/sessions/transcript-append.ts:142-166`
  - body: The append queue is process-local and parent/idempotency decisions are based on a stale pre-insert event list, so gateway, CLI, or worker writers can append conflicting parents or duplicate idempotent messages.
  - confidence: 0.87
- **[P1] Honor recent-read byte and line caps:** `src/gateway/session-utils.fs.ts:299-303`
  - body: Recent reads now load the full SQLite transcript and only apply `maxMessages`, ignoring `maxBytes` and `maxLines` bounds that callers use to avoid full-history latency and memory spikes.
  - confidence: 0.87
- **[P1] Preserve transcript agent scope during import:** `src/commands/doctor-session-transcripts.ts:76-84`
  - body: Doctor import maps transcripts outside the canonical `agents/<id>/sessions` shape to `main` and then removes the source file, making custom non-main history disappear from its intended agent partition.
  - confidence: 0.86
- **[P1] Snapshot SQLite databases before backup:** `src/infra/backup-create.ts:372-390`
  - body: The backup archive includes the live state directory directly; in WAL mode this can capture an inconsistent mix of database, `-wal`, and `-shm` files while writes are active.
  - confidence: 0.84
- **[P2] Delete discarded checkpoint snapshots:** `src/gateway/session-compaction-checkpoints.ts:281-285`
  - body: Checkpoint capture writes full snapshot transcripts into SQLite, but cleanup is a no-op, so failed or discarded compactions leave unreferenced snapshot rows that can accumulate over time.
  - confidence: 0.84
- **[P2] Repair transcripts before deleting legacy JSONL:** `src/infra/state-migrations.ts:957-963`
  - body: The migration imports legacy transcript JSONL and removes the source before the older doctor repair flow can scan it, preserving broken prompt-rewrite branches permanently after upgrade.
  - confidence: 0.82
- **[P2] Merge legacy cron state by recency:** `src/cron/store.ts:201-207`
  - body: The import writes the whole legacy sidecar payload into SQLite, so a stale `jobs-state.json` can roll scheduler state backward after SQLite already has newer runtime state.
  - confidence: 0.82
- **[P2] Raise the shared SQLite busy timeout:** `src/state/openclaw-state-db.ts:177-179`
  - body: The shared state database uses a 5 second busy timeout even though this PR moves gateway, CLI, cron, transcript, pairing, and artifact writes into shared SQLite paths that can contend during normal operation.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff moves sensitive runtime state into SQLite while leaving concrete partitioning, integrity, and backup-consistency concerns.

Concerns:

- **[high] Prevent agent partition mixing:** `src/config/sessions/store-backend.sqlite.ts:43`
  - body: Session and transcript fallback paths can store custom non-main agent data under the `main` partition, risking cross-agent state exposure or hidden history after migration.
  - confidence: 0.9
- **[high] Use SQLite-consistent backup snapshots:** `src/infra/backup-create.ts:390`
  - body: Archiving live WAL-mode database files directly can produce an inconsistent backup of sensitive session, pairing, device, and queue state.
  - confidence: 0.84
- **[medium] Enable relational integrity for state rows:** `src/state/openclaw-state-db.ts:63`
  - body: The new schema defines related agent, session, transcript, VFS, artifact, and cron rows without visible foreign-key enforcement, which makes orphaned or cross-scoped sensitive state easier to create and harder to clean safely.
  - confidence: 0.76
