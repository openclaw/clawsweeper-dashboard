## Summary

Keep open: current-head source inspection still shows stale legacy-import paths that can overwrite live SQLite auth/control state, and the protected `security`/`maintainer` labels plus insufficient current-head behavior proof require maintainer/security review.

## What This Changes

This PR moves OpenClaw runtime state from file-backed JSON/JSONL stores into global and per-agent SQLite databases, with migration, doctor, SDK, UI, docs, and test updates.

## Best Possible Solution

Keep the database-first direction, but make legacy imports idempotent or recency-safe, preserve recoverable malformed sources for retry, and merge only after protected maintainer/security review plus current-head populated-upgrade proof.

## Reproduction Assessment

Yes, by source inspection against PR head `66da4a2ef16ec20a4400f22b9239b832a52e1240`: run upgraded runtime long enough to write newer SQLite rows, leave legacy files present, then run `openclaw doctor --fix`; the sampled importers replace active SQLite state from stale files and remove those sources.

## Solution Assessment

No, not as-is; the SQLite storage boundary is directionally maintainable, but delayed imports need merge, skip, or recency guards before this is the narrowest safe implementation.
