## Summary

Keep open: this is protected by security and maintainer labels, is currently dirty, and exact-head source inspection still shows stale legacy import paths that can overwrite newer SQLite-backed state and then remove the legacy source.

## What This Changes

The branch moves many file-backed runtime stores into global and per-agent SQLite databases with new schemas, Kysely access, migration paths, docs, tests, and UI/API field updates.

## Best Possible Solution

Keep the SQLite direction, fix the stale legacy-import and source-deletion paths with recency/idempotency checks, land or fold the direct correctness stack such as https://github.com/openclaw/openclaw/pull/79971, and require exact-head populated-upgrade proof before merge.

## Reproduction Assessment

Yes, by source inspection: running an upgraded runtime first can create or update SQLite state, and a later doctor migration can import older legacy files into the same records and delete the source. I did not run a live migration because this review is read-only.

## Solution Assessment

No for merging as-is: the database-first design is plausible, but the migration boundary still needs recency/idempotency safeguards and exact-head real upgrade proof before it is the maintainable fix.
