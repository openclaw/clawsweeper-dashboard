## Summary

Keep open: the latest PR head still has source-proven correctness and security blockers in SQLite legacy import paths, plus the PR has protected `security` and `maintainer` labels and no after-fix real behavior proof.

## What This Changes

The PR replaces file-backed runtime, session, plugin, channel, scheduler, artifact, SDK, docs, and UI protocol state with global and per-agent SQLite storage plus migration paths.

## Best Possible Solution

Continue the SQLite refactor only after legacy imports become non-destructive and recency-aware, client/server protocol compatibility is restored, real upgraded-run proof is supplied, and maintainer/security-selected gates pass.

## Reproduction Assessment

Yes, for the PR blockers by source inspection: current-head migration paths overwrite live SQLite records with legacy snapshots and the sessions RPC schema rejects a field the UI still sends. I did not run a live migration in this read-only review.

## Solution Assessment

No. SQLite may be the right architecture, but this patch is not the best merge candidate while it can roll back security/routing/runtime state and break the sessions list RPC contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Re-allow configuredAgentsOnly in sessions.list:** `src/gateway/protocol/schema/sessions.ts:49`
  - body: `sessions.list` now rejects unknown params, but the Control UI still sends `configuredAgentsOnly` on session loads. Restore the schema field or stop sending it before this lands, otherwise normal session listing returns a validation error instead of rows.
  - confidence: 0.93
- **[P1] Skip stale device-auth imports when SQLite state exists:** `src/commands/doctor/legacy/device-auth-store.ts:39-41`
  - body: This writes the legacy `identity/device-auth.json` snapshot into the live SQLite key and then removes the file. If runtime has already rotated or revoked tokens in SQLite, a later doctor run can restore stale credentials permanently.
  - confidence: 0.91
- **[P1] Preserve live transcript rows during legacy import:** `src/commands/doctor/state-migrations.ts:304-309`
  - body: The importer calls `replaceSqliteSessionTranscriptEvents`, which deletes existing events for the session before inserting the legacy JSONL payload. A delayed doctor migration can therefore wipe newer SQLite transcript events and then delete the legacy source.
  - confidence: 0.9
- **[P1] Preserve existing plugin state on migration conflicts:** `src/plugin-sdk/migration-runtime.ts:55-60`
  - body: The shared migration helper updates existing `(plugin_id, namespace, entry_key)` rows with legacy values on conflict. Plugin migrations that run after live SQLite writes can roll back channel or plugin state once the old sidecar files are removed.
  - confidence: 0.88
- **[P1] Merge legacy commitments instead of replacing rows:** `src/commands/doctor/legacy/commitments.ts:96-98`
  - body: The importer saves the legacy commitment file through a write path that replaces the commitments table wholesale. Running doctor after new SQLite-backed reminders or status updates exist can drop those newer rows and persist the rollback.
  - confidence: 0.88
- **[P1] Keep newer channel pairing state during legacy import:** `src/commands/doctor/legacy/channel-pairing.ts:229-245`
  - body: This import replaces current SQLite requests and per-account allowlists with legacy file contents before deleting the source files. Delayed migration can re-add revoked senders or remove newly approved senders.
  - confidence: 0.88
