## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Re-allow configuredAgentsOnly in sessions.list params:** `src/gateway/protocol/schema/sessions.ts:64`
  - body: The Control UI still sends `configuredAgentsOnly` on each `sessions.list` request, but the PR-head schema no longer declares that field while `additionalProperties` remains false. Normal session loads from the UI can be rejected instead of returning rows unless the field is restored or every client stops sending it.
  - confidence: 0.94
- **[P1] Preserve newer per-agent session rows during migration:** `src/infra/state-migrations.ts:2711-2714`
  - body: This conflict handler always takes the legacy global row and then drops the global table. If upgraded runtime traffic has already written a newer per-agent `session_entries` row before doctor/migrate runs, the migration rolls that metadata backward permanently; merge by recency or skip newer target rows.
  - confidence: 0.91
- **[P1] Treat missing legacy pairing files as non-imports:** `src/infra/pairing-files.ts:96-107`
  - body: `readJsonIfExists` returns `null` for missing files, but these checks accept anything except `undefined`, so a partial legacy pairing state writes `{}` over the missing sibling SQLite key. That can wipe current pending or paired device/node state during doctor import.
  - confidence: 0.9
- **[P1] Avoid replacing live device auth state from legacy files:** `src/infra/device-auth-store.ts:95-97`
  - body: The importer writes the entire legacy `device-auth.json` payload into the live SQLite store and removes the source file. A delayed doctor run after token rotation or revocation can restore stale credentials, including revoked tokens, with no legacy source left for recovery.
  - confidence: 0.89
- **[P1] Merge legacy pairing imports instead of replacing access state:** `src/pairing/pairing-store.ts:731-747`
  - body: The legacy channel pairing import replaces current SQLite requests and account allowlists with file-era data before deleting those files. Running doctor after approvals or revocations have happened in SQLite can re-authorize stale senders or remove newer approvals, so import should be recency-aware or merge-only.
  - confidence: 0.88
- **[P1] Honor byte and line caps for recent transcript reads:** `src/gateway/session-transcript-readers.ts:283-287`
  - body: The recent read path loads all SQLite transcript messages and only slices by `maxMessages`, ignoring byte and line bounds callers use to cap large transcript tails. Long sessions can force full-history reads in hot paths where bounded tails were expected.
  - confidence: 0.88
- **[P2] Honor maxBytes in recent usage readers:** `src/gateway/session-transcript-readers.ts:732-753`
  - body: Both recent usage helpers discard `maxBytes` and load all usage events, so status, auto-reply, and token-estimation paths can parse an entire large session despite passing a byte cap. Apply bounded-tail semantics before extracting usage.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: Needs attention: several legacy import paths can roll back newer SQLite auth, pairing, and access-control state during delayed doctor migrations.

Concerns:

- **[high] Legacy pairing import can restore revoked access:** `src/pairing/pairing-store.ts:731`
  - body: The importer replaces SQLite pairing requests and allowlists with legacy JSON contents, then deletes the legacy file. A delayed migration can re-add revoked senders or remove newly approved senders.
  - confidence: 0.88
- **[high] Device-auth import can restore stale tokens:** `src/infra/device-auth-store.ts:95`
  - body: The device-auth importer writes the full legacy token store over SQLite and removes the source. If runtime already rotated or revoked tokens, doctor can restore stale credentials.
  - confidence: 0.89
- **[medium] Missing pairing sidecars can clear live state:** `src/infra/pairing-files.ts:96`
  - body: Missing pairing files are treated as importable empty records, so partial legacy state can overwrite existing pending or paired SQLite records with `{}`.
