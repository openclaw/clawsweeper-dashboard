## What This Changes

The PR moves runtime/session/transcript/plugin/cron state from file-backed stores into global and per-agent SQLite databases, with migration, backup, docs, app/UI, plugin, and test updates.

## Best Possible Solution

Keep the SQLite direction but require maintainer/security review after the branch preserves newer SQLite state during legacy imports, restores bounded transcript reads, proves migration/backup/runtime behavior with redacted real output, and lands focused regression coverage.

## Reproduction Assessment

## Solution Assessment

No. The database-first direction matches the linked architecture goal, but this implementation is not ready until migration/import semantics preserve newer SQLite state, bounded reads remain bounded, and the security-sensitive paths are proven with real data.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve newer cron rows during legacy import:** `src/cron/store.ts:292-294`
  - body: `importLegacyCronStoreToSqlite` writes the legacy store and then calls `writeCronJobsToSqlite`, which deletes existing SQLite cron jobs not present in that stale file. A delayed doctor/migrate run after the upgraded runtime has created or changed jobs can roll definitions back or drop newer jobs permanently when the legacy source is removed.
  - confidence: 0.9
- **[P1] Honor recent-read byte and line limits:** `src/gateway/session-transcript-readers.ts:283-287`
  - body: The recent-read path loads all scoped SQLite messages and only slices by `maxMessages`, ignoring `maxBytes` and `maxLines`. Callers that request a bounded tail can still materialize long transcripts, regressing latency and memory behavior for large sessions.
  - confidence: 0.91
- **[P1] Merge pairing allowlists during legacy import:** `src/pairing/pairing-store.ts:743-747`
  - body: The legacy allowlist import replaces the current SQLite account allowlist with file contents. If users approve or revoke senders in SQLite before running doctor, this can re-authorize stale senders or remove newer approvals after migration.
  - confidence: 0.88
- **[P1] Preserve newer device auth tokens during legacy import:** `src/infra/device-auth-store.ts:91-97`
  - body: `importLegacyDeviceAuthFileToSqlite` writes the entire legacy token store over the SQLite record before deleting the source file. In delayed-doctor upgrades, that can restore revoked device tokens or drop tokens issued after upgrade.
  - confidence: 0.88
