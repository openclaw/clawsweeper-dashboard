## Summary

Keep open: this broad protected PR still has source-backed blocking regressions in bounded transcript reads and security-sensitive legacy imports, and it lacks after-fix real behavior proof.

## What This Changes

The PR moves runtime, session, transcript, plugin, cron, device, pairing, backup, docs, app, UI, and test state paths from JSON/file stores toward global and per-agent SQLite databases.

## Best Possible Solution

Keep the SQLite direction, but land it only after imports preserve newer SQLite state, recent transcript reads remain bounded, focused regression coverage is added, real behavior proof is posted, and maintainer/security review clears the architecture scope.

## Reproduction Assessment

Yes for the review findings by source inspection: current PR head shows unbounded recent transcript reads and overwrite-style legacy imports. I did not run live migration scenarios because this review is read-only and the PR lacks real behavior proof.

## Solution Assessment

No. The database-first direction matches https://github.com/openclaw/openclaw/issues/78096, but this branch is not ready until migration/import conflict handling is recency-safe and bounded reads stay bounded.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor recent-read byte and line caps:** `src/gateway/session-transcript-readers.ts:283-287`
  - body: The recent-read paths still materialize every scoped SQLite message and only slice by `maxMessages`, ignoring `maxBytes` and `maxLines`. Callers pass those caps to bound large transcript tails, so long sessions can now trigger full-history DB reads and memory spikes.
  - confidence: 0.91
- **[P1] Preserve newer cron rows during legacy import:** `src/cron/store.ts:292-294`
  - body: `importLegacyCronStoreToSqlite` writes legacy state and then imports the legacy job list. Because the writer upserts legacy rows and deletes SQLite rows missing from the legacy file, a delayed doctor run can roll back or delete newer cron jobs created after upgrade.
  - confidence: 0.9
- **[P1] Merge pairing state instead of replacing SQLite rows:** `src/pairing/pairing-store.ts:731-747`
  - body: The legacy channel-pairing import replaces current requests and account allowlists with stale file contents before deleting those files. That can re-authorize revoked senders or remove newly approved senders after a delayed doctor run.
  - confidence: 0.89
- **[P1] Avoid restoring revoked device tokens from legacy files:** `src/infra/device-auth-store.ts:95-97`
  - body: The legacy device-auth import writes the whole JSON token store into SQLite and then removes the source file. If tokens were rotated or revoked in SQLite first, doctor can permanently restore stale credentials.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces security-sensitive rollback risks in auth, pairing, and approval imports.

Concerns:

- **[high] Legacy device auth can restore revoked tokens:** `src/infra/device-auth-store.ts:95`
  - body: `importLegacyDeviceAuthFileToSqlite` overwrites the SQLite token store with the legacy JSON file and then removes the source, so delayed migration can reintroduce revoked or superseded device tokens.
  - confidence: 0.88
- **[high] Legacy pairing imports can roll back access control:** `src/pairing/pairing-store.ts:746`
  - body: The channel pairing import replaces SQLite requests and allowlists with legacy file contents, which can re-authorize stale senders or remove newer approvals/revocations.
  - confidence: 0.87

  - body: `importLegacyExecApprovalsFileToSqlite` writes the legacy approvals payload into SQLite and deletes the file without checking existing SQLite decisions, so delayed migration can revive outdated command approvals or drop newer denials.
