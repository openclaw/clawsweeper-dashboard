## Summary

Keep this PR open. It is an active, protected, security-sensitive database-first refactor, and the latest reviewed head still shows blocking migration clobber risks plus an unsequenced Node 24 support break.

## What This Changes

This PR moves OpenClaw runtime/session/transcript/auth/plugin state from file-backed JSON/JSONL stores into shared and per-agent SQLite stores across core, apps, plugins, docs, and tests.

## Best Possible Solution

Keep the database-first direction under maintainer/security review, preserve or explicitly sequence Node support, make legacy imports merge-safe and source-preserving, and require latest-head real upgrade proof before merge.

## Reproduction Assessment

Yes for the blocking review findings by source inspection: current main still supports Node 22, while the PR raises the runtime floor and several latest-head legacy import paths overwrite live SQLite rows. I did not run a populated-install migration in this read-only review.

## Solution Assessment

No, not as-is. The SQLite architecture is plausible, but the current patch needs an explicit Node support decision, merge-safe migration semantics, and latest-head real behavior proof before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve Node 22 compatibility or split the runtime bump:** `package.json:1835`
  - body: Current main still supports Node >=22.16.0 and carries a Node 22 compatibility lane, but this PR raises the package engine to Node >=24.0.0. Existing supported installs can fail before they can run the SQLite migration unless maintainers explicitly approve and sequence that runtime-breaking change.
  - confidence: 0.9
- **[P1] Merge legacy transcript imports instead of replacing rows:** `src/commands/doctor/state-migrations.ts:306-311`
  - body: Each legacy transcript file is imported through `replaceSqliteSessionTranscriptEvents`, which removes existing rows for that session before inserting the current file. Real installs can have multiple legacy files or newer SQLite rows for one session, so a delayed doctor run can silently drop transcript history.
  - confidence: 0.91
- **[P1] Skip stale exec approvals when SQLite state exists:** `src/commands/doctor/legacy/exec-approvals.ts:44-50`
  - body: This importer writes the legacy approvals snapshot over `exec.approvals/current` and then removes the file without checking for newer SQLite state. A delayed doctor run can roll back runtime approval policy and make that rollback durable.
  - confidence: 0.9
- **[P1] Preserve current device identity during legacy import:** `src/commands/doctor/legacy/device-identity.ts:150-152`
  - body: The legacy device identity importer writes the legacy identity snapshot into SQLite and then removes the source file. If runtime has already generated or rotated identity state in SQLite, a delayed doctor run can restore stale keys and invalidate newer trust relationships.
  - confidence: 0.88
- **[P1] Merge channel pairing imports with live state:** `src/commands/doctor/legacy/channel-pairing.ts:231-247`
  - body: The channel-pairing importer replaces request and allowlist entries from legacy files and then removes those files. If SQLite already contains newer approvals or revocations, delayed migration can re-add revoked senders or drop newly approved ones.
  - confidence: 0.86
- **[P1] Preserve existing plugin state on migration conflict:** `src/plugin-sdk/migration-runtime.ts:55-60`
  - body: The migration helper resolves `plugin_state_entries` conflicts by replacing the existing row with the imported value. Because plugin state is live runtime state, delayed sidecar migration can overwrite newer plugin data instead of preserving or merging it.
  - confidence: 0.84
