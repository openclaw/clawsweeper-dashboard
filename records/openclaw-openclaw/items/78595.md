## Summary

Keep open: current main still uses file-backed session and cron state, while this broad SQLite refactor PR has confirmed source-level migration blockers, missing real upgrade proof, and protected security/maintainer labels.

## What This Changes

The branch moves broad OpenClaw runtime, session, transcript, plugin, cron, device, backup, docs, UI, and protocol state surfaces from file-backed stores toward shared and per-agent SQLite databases.

## Best Possible Solution

Keep this in maintainer/security review, fix conflict-aware migration semantics and recovery-source preservation, then require real upgrade/backup/runtime proof before merge.

## Reproduction Assessment

Yes for the confirmed PR blockers: source inspection of the latest PR head shows legacy cron, transcript, and device-auth imports can overwrite newer SQLite state and remove legacy recovery files. I did not run a live migration because this review is read-only.

## Solution Assessment

No. The database-first direction may be maintainable, but the current patch is not the best merge candidate while delayed migration paths can roll back live state and no real upgrade proof is supplied.

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve live transcript rows on legacy JSONL import:** `src/commands/doctor/state-migrations.ts:304-309`
  - body: `importLegacyTranscriptFileToSqlite` routes through `replaceSqliteSessionTranscriptEvents`, which deletes all existing SQLite transcript events for the session before inserting the legacy JSONL payload. A delayed doctor run after upgraded runtime activity can roll a session back to stale history and then remove the source file.
  - confidence: 0.9
- **[P1] Preserve newer device-auth tokens during legacy import:** `src/commands/doctor/legacy/device-auth-store.ts:39-41`
  - body: This importer writes the legacy `device-auth.json` payload to the live SQLite device-auth store and then deletes the file. If tokens were added, rotated, or revoked after upgrade, a later doctor run can restore stale tokens or drop newer ones.
  - confidence: 0.9
- **[P1] Preserve newer cron rows during legacy repair:** `src/commands/doctor-cron.ts:282-285`
  - body: When a legacy cron store exists, doctor saves that snapshot through the SQLite-backed cron store. `saveCronStore` rewrites conflicting rows and deletes rows absent from the imported file, so a delayed repair can lose newer SQLite cron jobs or runtime state.
  - confidence: 0.9
- **[P1] Reject malformed cron files before empty import:** `src/commands/doctor/legacy/cron-store.ts:107-112`
  - body: Unexpected parsed cron shapes are coerced to an empty jobs list. Because the presence of a legacy store can trigger saving and source removal, malformed legacy input can erase current SQLite cron rows instead of failing safely for operator retry.
  - confidence: 0.82
- **[P2] Keep Telegram binding metadata during migration:** `extensions/telegram/src/state-migrations.ts:143-153`
  - body: The Telegram thread binding importer rebuilds each record with only core routing fields and drops persisted fields such as idle timeout, max age, label, and metadata. Those fields affect binding expiry and behavior, and the source file is removed after import.
  - confidence: 0.86
