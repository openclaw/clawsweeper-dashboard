## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Honor bounded transcript reads:** `src/gateway/session-transcript-readers.ts:253-258`
  - body: `readRecentSessionMessages` accepts `maxBytes`/`maxLines`, but this path now calls `loadScopedSessionMessages(...)` and only slices by `maxMessages`. That materializes the full SQLite transcript before trimming, so callers that request a bounded tail can regress to full-history reads on large sessions.
  - confidence: 0.9
- **[P1] Honor maxBytes for usage scans:** `src/gateway/session-transcript-readers.ts:746-754`
  - body: Both recent usage readers discard `maxBytes` before loading usage events. Status/session-list callers that pass byte caps can now scan full transcripts, which loses the previous bounded-tail behavior and can create latency or memory spikes.
  - confidence: 0.88
- **[P2] Thread doctor env into plugin approval import:** `src/commands/doctor-sqlite-state.ts:300-303`
  - body: `maybeRepairLegacyRuntimeStateFiles` is parameterized by `ctx.env`, but this call imports plugin-binding approvals without passing it. The importer resolves legacy paths from `process.env`, so tests or programmatic doctor runs targeting a non-default state dir can miss the intended file or mutate the wrong state.
  - confidence: 0.9
- **[P1] Merge legacy approvals instead of replacing SQLite:** `src/plugins/conversation-binding.ts:471-477`
  - body: The importer writes the legacy approvals file directly over the SQLite KV value. If the upgraded runtime has already recorded newer allow/deny decisions, a delayed doctor run can restore stale plugin-binding approvals before deleting the source file.
  - confidence: 0.82
- **[P1] Avoid replacing same-session transcripts during import:** `src/config/sessions/transcript-store.sqlite.ts:840-844`
  - body: `importJsonlTranscriptToSqlite` delegates to a replace helper that deletes all existing rows for the session before inserting the current file. If migration sees multiple legacy files for the same session id, later imports can drop earlier transcript events depending on traversal order.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The patch moves durable trust and access-control state into SQLite, and at least one importer can overwrite newer approval state from a stale legacy file.

Concerns:

- **[high] Legacy approvals can restore stale trust decisions:** `src/plugins/conversation-binding.ts:472`
  - body: `importLegacyPluginBindingApprovalFileToSqlite` writes the entire legacy approvals payload to the SQLite KV entry. In delayed-doctor upgrades, that can re-authorize stale plugin bindings or drop newer denials/approvals.
  - confidence: 0.82
- **[medium] Plugin approval import can target the wrong state dir:** `src/commands/doctor-sqlite-state.ts:302`
  - body: The doctor flow accepts an env override but calls the plugin-binding approval importer without it, while the importer resolves paths from `process.env`; this can inspect or mutate the wrong state during non-default doctor runs.
  - confidence: 0.88
