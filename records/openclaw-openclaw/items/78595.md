## Summary

Keep open: protected `security`/`maintainer` labels plus verified P1 migration/security and bounded-read regressions mean this broad storage refactor needs maintainer/security review rather than cleanup.

## What This Changes

The PR moves runtime, session, transcript, plugin, cron, device, and pairing state from JSON/JSONL/file stores into global and per-agent SQLite databases with migration, backup, docs, app, UI, plugin, and test updates.

## Best Possible Solution

Keep the database-first direction, but land it only after newer SQLite state is preserved during legacy imports, bounded reads remain bounded, focused regression coverage is added, and redacted real runtime proof is supplied.

## Reproduction Assessment

Yes for the review findings by source inspection: PR head directly shows unbounded recent reads and overwrite-style legacy imports. I did not run live migration scenarios because this review is read-only and the PR lacks real behavior proof.

## Solution Assessment

No. The database-first direction matches https://github.com/openclaw/openclaw/issues/78096, but this branch is not ready until security-sensitive imports are merge/recency-safe and recent reads stay bounded.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor byte and line caps for recent transcript reads:** `src/gateway/session-transcript-readers.ts:283-287`
  - body: The recent-read path materializes every scoped SQLite message and only slices by `maxMessages`. Callers pass `maxBytes`/`maxLines` to bound large transcript tails, so long sessions can now trigger full-history DB reads and memory spikes.
  - confidence: 0.91
- **[P1] Preserve newer cron rows during legacy import:** `src/cron/store.ts:292-294`
  - body: `importLegacyCronStoreToSqlite` writes legacy state and then imports the legacy job list. A delayed doctor run can roll back or delete newer SQLite cron jobs that were created after upgrade but before migration.
  - confidence: 0.9
- **[P1] Merge pairing state instead of replacing SQLite rows:** `src/pairing/pairing-store.ts:731-747`
  - body: The legacy pairing import replaces current requests and account allowlists with stale file contents before deleting those files. That can re-authorize revoked senders or remove newly approved senders after a delayed doctor run.
  - confidence: 0.89
- **[P1] Prevent stale device auth overwrites during import:** `src/infra/device-auth-store.ts:95-97`
  - body: The legacy device-auth import writes the whole JSON token store into SQLite and then removes the source file. If tokens were rotated or revoked in SQLite first, doctor can permanently restore stale credentials.
  - confidence: 0.88
