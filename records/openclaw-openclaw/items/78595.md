## Summary

Keep open: protected `security` and `maintainer` labels require human handling, and current head still has source-provable migration/data-loss blockers plus no sufficient latest-head real behavior proof.

## What This Changes

The PR moves runtime/session/auth/plugin state from scattered file-backed stores into global and per-agent SQLite stores across core, apps, plugins, docs, and tests.

## Best Possible Solution

Keep the database-first direction under maintainer/security review, preserve Node 22 unless a runtime bump is explicitly sequenced, make legacy imports merge-safe/source-preserving, and require latest-head real upgrade proof before merge.

## Reproduction Assessment

Yes, for the blocking classes by source inspection: current main still supports Node 22 while the PR raises the runtime floor, and several legacy importers overwrite live SQLite rows without merge or recency checks. I did not run a live populated-install upgrade in this read-only review.

## Solution Assessment

No, not as-is. The SQLite architecture is plausible, but the current patch needs a deliberate Node support decision, merge-safe migration semantics, and latest-head real behavior proof before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve Node 22 compatibility or split the runtime bump:** `package.json:1835`
  - body: Current main supports Node >=22.16.0 and still carries a Node 22 compatibility lane, but this PR raises the package engine to Node >=24.0.0. Existing supported installs can fail before they can run the SQLite migration unless maintainers explicitly approve and sequence that runtime-breaking change.
  - confidence: 0.9
- **[P1] Merge legacy transcript imports instead of replacing rows:** `src/commands/doctor/state-migrations.ts:306-311`
  - body: Each legacy transcript file is imported through `replaceSqliteSessionTranscriptEvents`, which deletes all rows for that session before inserting the current file. Real installs can have multiple legacy files or newer SQLite rows for one session, so a delayed doctor run can silently drop transcript history.
  - confidence: 0.91
- **[P1] Skip stale exec approvals when SQLite state exists:** `src/commands/doctor/legacy/exec-approvals.ts:44-50`
  - body: This importer writes the legacy approvals snapshot over `exec.approvals/current` and then removes the file without checking for newer SQLite state. A delayed doctor run can roll back runtime approval policy and make that rollback durable.
  - confidence: 0.9
- **[P1] Preserve current device auth during legacy import:** `src/commands/doctor/legacy/device-auth-store.ts:39-41`
  - body: The legacy device-auth importer replaces the live SQLite auth snapshot and then removes the source file. If tokens were created, rotated, or revoked after upgrade but before doctor runs, this can restore stale tokens.
  - confidence: 0.88
- **[P1] Merge channel pairing imports with live state:** `src/commands/doctor/legacy/channel-pairing.ts:231-247`
  - body: The channel-pairing importer replaces request and allowlist entries from legacy files and then removes the files. If SQLite already contains newer approvals or revocations, delayed migration can re-add revoked senders or drop newly approved ones.
  - confidence: 0.86
- **[P1] Preserve existing plugin state on migration conflict:** `src/plugin-sdk/migration-runtime.ts:55-60`
  - body: The migration helper resolves `plugin_state_entries` conflicts by replacing the existing row with the imported value. Because plugin state is live runtime state, delayed sidecar migration can overwrite newer plugin data instead of preserving or merging it.
  - confidence: 0.84
