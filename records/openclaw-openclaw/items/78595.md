## Summary

Keep open: this is an active protected storage/security refactor, and latest-head source still shows blocking migration data-loss and auth rollback paths plus an unapproved Node 24 support break. The discussion adds useful populated-install evidence and follow-up PRs, but I did not find latest-head real upgrade proof for this PR.

## What This Changes

The branch moves runtime, session, transcript, auth, plugin, channel, scheduler, cache, VFS, and artifact state from JSON/JSONL/sidecars into global and per-agent SQLite databases, with SDK/API/docs/UI/test updates and a Node runtime requirement bump.

## Best Possible Solution

Keep the SQLite direction in maintainer/security review, restore or explicitly approve the runtime support change, make legacy imports merge-safe and source-preserving, and land the correctness and public seam follow-ups in a deliberate order.

## Reproduction Assessment

Yes for the blocking classes by source inspection: delayed doctor runs after SQLite writes can overwrite live state, and multiple legacy transcript files for one session can be collapsed by replacement import. I did not run a local live upgrade in this read-only review.

## Solution Assessment

No for merge as-is. The architecture direction is reasonable, but the current patch needs conservative migration semantics, a Node support decision, latest-head real upgrade proof, and maintainer/security sequencing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restore Node 22 support or split the runtime bump:** `package.json:1823`
  - body: Current main supports Node >=22.16.0 and keeps a Node 22 compatibility lane. This PR raises the package engine and related runtime gates to Node >=24, so existing supported Node 22 installs can fail before they can run the SQLite migration unless maintainers explicitly approve a release-breaking runtime bump.
  - confidence: 0.9
- **[P1] Skip stale exec approvals when SQLite state exists:** `src/commands/doctor/legacy/exec-approvals.ts:44-50`
  - body: The legacy importer writes exec-approvals.json into the live SQLite KV row and then removes the file without checking for existing state. A delayed doctor run can roll back newer approval policy already written by runtime paths and make that rollback durable.
  - confidence: 0.88
- **[P1] Merge legacy transcript imports instead of replacing rows:** `src/commands/doctor/state-migrations.ts:304-309`
  - body: Each legacy transcript file is imported through the replacement helper, which deletes all existing rows for the session before inserting that file. Real installs can have multiple legacy files for one session, so only the last import survives and earlier or newer SQLite rows are lost.
  - confidence: 0.9
- **[P2] Keep transcript sources when repaired imports drop rows:** `src/commands/doctor-session-transcripts.ts:233-238`
  - body: The repair path can persist only the selected active event list and then remove the original JSONL source. If that selection is incomplete or later proves wrong, operators cannot correct the legacy file and retry recovery.
  - confidence: 0.78
