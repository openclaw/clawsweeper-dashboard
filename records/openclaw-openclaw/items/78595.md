## Summary

This PR should stay open: current head still has merge-blocking protocol and security-state regressions, plus protected security/maintainer labels and no real upgraded-run proof.

## What This Changes

The branch moves runtime/session/transcript/plugin/channel/scheduler/artifact state from file-backed stores into SQLite-backed control-plane and per-agent storage, with doctor migrations, SDK/protocol/UI/docs/tests updates.

## Best Possible Solution

Continue the SQLite refactor after restoring sessions.list compatibility, making legacy imports preserve newer SQLite state, adding real upgraded-run proof, and passing maintainer/security-selected gates.

## Reproduction Assessment

Yes, for the PR blockers by source inspection at current head e9b012587919862e2eb85a18891622f21bb06687. The Control UI sends a now-rejected sessions.list parameter, and delayed doctor imports can overwrite SQLite-backed device security state that runtime token paths mutate.

## Solution Assessment

No. SQLite may be the right architecture, but this patch is not merge-ready while it can reject normal sessions UI calls and roll live security/runtime state backward during migration.

  - confidence: 0.92

  - body: This migration persists the legacy bootstrap-token snapshot into live device pairing state and deletes the source file. A delayed doctor run can reintroduce tokens that runtime issue/revoke/redeem flows already removed or updated in SQLite.
  - confidence: 0.91
