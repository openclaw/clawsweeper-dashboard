## Summary

Keep open: this is an active 2,325-file SQLite runtime-state refactor with protected `security` and `maintainer` labels, and current-head source inspection still shows stale legacy imports can overwrite newer SQLite runtime/auth state.

## What This Changes

This PR moves OpenClaw runtime, session, transcript, plugin, channel, scheduler, artifact, app, docs, and test surfaces from legacy JSON/JSONL/sidecar stores toward shared and per-agent SQLite databases.

## Best Possible Solution

Keep the database-first direction, harden legacy import idempotency and recency checks across auth/access/scheduler/runtime state, and merge only after maintainer/security review plus exact-head populated-install proof.

## Reproduction Assessment

Yes. Source inspection on current PR head shows delayed doctor imports can write stale legacy snapshots into the same SQLite rows later mutated by runtime code; I did not run a live migration in this read-only review.

## Solution Assessment

No for merging as-is. The SQLite architecture may be the right direction, but this patch still needs stale-import rollback hardening, security review, and current-head populated-upgrade proof before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve newer cron rows during legacy import:** `src/commands/doctor-cron.ts:284-288`
  - body: When legacy `cron/jobs.json` exists, this path saves that snapshot through `saveCronStore`. The SQLite writer then removes existing cron rows missing from the imported file, so a delayed doctor run after runtime or CLI cron edits can roll newer SQLite jobs back to stale file-era state and delete the source snapshot.
  - confidence: 0.88
- **[P1] Skip stale device-auth snapshots when SQLite has state:** `src/commands/doctor/legacy/device-auth-store.ts:39-41`
  - body: This importer writes the legacy `identity/device-auth.json` snapshot into the same SQLite record used by runtime token issue/revoke paths, then removes the file. If the upgraded runtime has already rotated or revoked tokens, a later doctor run can restore stale tokens and make that rollback durable.
  - confidence: 0.9
- **[P1] Avoid overwriting current exec approvals from legacy files:** `src/commands/doctor/legacy/exec-approvals.ts:39-45`
  - body: This import path writes the legacy exec-approvals blob into the current SQLite approvals row and then deletes the file. Since runtime approval changes also persist to that row, running doctor later can revert updated policy or allowlist decisions to stale file-era state.
  - confidence: 0.86
