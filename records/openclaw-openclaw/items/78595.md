## Summary

Keep open: current main still uses file-backed runtime/session/cron state, while this unmerged PR is broad, protected by security and maintainer labels, lacks real upgrade/runtime proof, and still has source-proven migration rollback defects.

## What This Changes

This PR moves OpenClaw runtime, session, transcript, plugin, cron, device, backup, docs, tests, UI, and protocol state surfaces from scattered file-backed stores toward shared and per-agent SQLite databases.

## Best Possible Solution

Resolve destructive migration conflict semantics, preserve recovery sources on failed or stale imports, add real upgrade/backup/runtime proof, and keep the PR in owner review before merge.

## Reproduction Assessment

Yes for the confirmed PR blockers: source inspection of PR head shows legacy cron, transcript, and device-auth imports can overwrite newer SQLite state and delete legacy recovery files. I did not run a live migration because this review is read-only.

## Solution Assessment

No. The database-first direction may be maintainable, but the current patch is not the best merge candidate while delayed migration paths can roll back live state and no real upgrade proof is supplied.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve live transcript rows on legacy JSONL import:** `src/commands/doctor/state-migrations.ts:304-309`
  - body: `importLegacyTranscriptFileToSqlite` routes through `replaceSqliteSessionTranscriptEvents`, which deletes all existing SQLite transcript events for the session before inserting the legacy JSONL payload. A delayed doctor run after upgraded runtime activity can roll a session back to stale history, then remove the source file and make the rollback persistent.
  - confidence: 0.9
- **[P1] Preserve newer device-auth tokens during legacy import:** `src/commands/doctor/legacy/device-auth-store.ts:39-41`
  - body: This importer writes the legacy `device-auth.json` payload to the live SQLite device-auth store and then deletes the file. If tokens were added, rotated, or revoked after upgrade, a later doctor run can restore stale tokens or drop newer ones.
  - confidence: 0.9
- **[P1] Preserve newer cron rows during legacy repair:** `src/commands/doctor-cron.ts:282-285`
  - body: When a legacy cron store exists, doctor saves that snapshot through the SQLite-backed cron store. `saveCronStore` rewrites conflicting rows and deletes rows absent from the imported file, so a delayed repair can lose newer SQLite cron jobs or runtime state.
  - confidence: 0.9
- **[P1] Reject malformed cron files before empty import:** `src/commands/doctor/legacy/cron-store.ts:107-112`
  - body: Unexpected parsed cron shapes are coerced to an empty jobs list. Because the presence of a legacy store can trigger saving and source removal, malformed legacy input can erase current SQLite cron rows instead of failing safely for operator retry.
  - confidence: 0.82
- **[P1] Keep Telegram binding metadata during migration:** `extensions/telegram/src/state-migrations.ts:143-153`
  - body: The Telegram thread binding importer rebuilds each record with only core routing fields and drops persisted fields such as `idleTimeoutMs`, `maxAgeMs`, `label`, and `metadata`. Those fields affect binding expiry and behavior, and the source file is removed after import.
  - confidence: 0.86
- **[P2] Store Discord deploy hashes under runtime keys:** `extensions/discord/src/state-migrations.ts:87-91`
  - body: The Discord importer stores migrated hashes as `legacy:${key}`, but runtime deploy-cache loading only reads entries prefixed with `${clientId}:`. Migrated hashes are therefore ignored after the source file is removed, forcing unnecessary command redeploy work.
  - confidence: 0.86
