## Summary

Keep https://github.com/openclaw/openclaw/pull/78595 open: current main still uses file-backed session stores, while the PR is an active protected-label SQLite storage refactor with line-verified migration rollback/data-loss blockers and insufficient latest-head real upgrade proof.

## What This Changes

This PR replaces file-backed runtime/session/auth/plugin/channel state with global and per-agent SQLite stores, migration tooling, SDK/runtime API updates, docs, UI labels, and broad tests.

## Best Possible Solution

Land the database-first architecture only after legacy migrations merge or skip against newer SQLite state safely, source deletion is recoverable, companion seams stay in their linked follow-up lanes, and exact-head populated-upgrade proof is attached.

## Reproduction Assessment

Yes for the merge-blocker class: source inspection shows replacement imports and immediate source deletion, and the discussion includes a populated-install migration that measured lost transcript rows. I did not run the branch locally under the read-only review constraint.

## Solution Assessment

No for merge as-is. The database-first direction is coherent, but the current patch still needs lossless migration semantics, stale-SQLite-state guards, maintainer/security approval, and latest-head real upgrade proof before it is the best landable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Merge legacy transcript imports instead of replacing rows:** `src/commands/doctor/state-migrations.ts:304-309`
  - body: `importLegacyTranscriptFileToSqlite` still routes every legacy JSONL file through `replaceSqliteSessionTranscriptEvents`. On installs with multiple files for the same session, or with newer SQLite rows created before a delayed doctor run, each import deletes the previous SQLite rows and can silently drop real transcript events.
  - confidence: 0.88
- **[P2] Delay source removal until transcript import is lossless:** `src/commands/doctor-session-transcripts.ts:233-238`
  - body: `migrateSessionTranscriptFileToSqlite` removes the source JSONL immediately after a replacement-style write. Until the importer proves it preserved and deduped all expected rows without clobbering newer SQLite data, deleting the original makes a bad migration hard to repair.
  - confidence: 0.82
- **[P1] Skip stale exec approvals imports when SQLite state exists:** `src/commands/doctor/legacy/exec-approvals.ts:44-50`
  - body: This migration writes the legacy `exec-approvals.json` payload into the live SQLite KV row and then deletes the file, without checking for newer approvals already saved by runtime paths. A delayed doctor run can roll back current approval policy or allowlist decisions.
  - confidence: 0.87
- **[P1] Merge channel pairing requests instead of replacing them:** `src/commands/doctor/legacy/channel-pairing.ts:231-233`
  - body: When a legacy pairing file exists, the importer replaces `state.requests` with the legacy snapshot before writing SQLite state. If the upgraded runtime already recorded newer pairing requests, doctor can revert live pending-pairing state to stale file-era data.
  - confidence: 0.84
- **[P1] Skip stale device-auth imports when SQLite state exists:** `src/commands/doctor/legacy/device-auth-store.ts:39-41`
  - body: The device-auth importer writes the legacy token store into the SQLite auth row and removes the source file without checking for newer token issuance or revocation. That can restore stale or revoked device tokens after users have already operated on the SQLite store.
  - confidence: 0.86

## Security Review
