## Summary

Keep this PR open, but it is not ready to merge. The branch fixes the default-profile case, yet the new managed-workspace guard still misses a setup-created profiled workspace when `agents add` later runs without that profile, so the same nested workspace footgun can remain. Current `main` still has the original bug, and overlapping open PRs mean maintainers should choose one canonical implementation.

## What This Changes

The PR changes agent workspace resolution to avoid nesting added agents under setup-created main workspaces, adds resolver regression tests, documents peer-level workspaces, and adds a changelog entry for #71889.

## Best Possible Solution

Land one canonical fix for #71889 that keeps brand-new `agents add` defaults peer-level, preserves explicit workspaces and intentional shared-base configs, covers both profile-transition directions, and avoids silently relocating existing agents unless maintainers explicitly choose a runtime resolver migration.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: setup writes `agents.defaults.workspace`, `agents add` uses `resolveAgentWorkspaceDir` for the prompt initial value, and current main appends the new id under that fallback; the PR head still reproduces the remaining gap after setup under `OPENCLAW_PROFILE=dev` followed by `agents add` without that profile.

## Solution Assessment

No. This PR is close for the default setup case, but the current guard is incomplete for profiled setup workspaces and the runtime-resolver scope needs maintainer choice against narrower wizard-only alternatives.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Recognize profiled setup workspaces before nesting:** `src/agents/agent-scope-config.ts:72-73`
  - body: The guard only treats `<stateDir>/workspace` and the current profile's `resolveDefaultAgentWorkspaceDir(env)` as setup-managed. If setup ran with `OPENCLAW_PROFILE=dev`, it can persist `<stateDir>/workspace-dev`; running `agents add work` later without that profile leaves the fallback unrecognized, so this branch still returns `<stateDir>/workspace-dev/work`, the nested-under-main layout this PR is meant to prevent.
  - confidence: 0.84
