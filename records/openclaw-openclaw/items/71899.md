## Summary

Keep open. Current main still has the reported workspace-prefill path, and this PR is a plausible fix candidate but remains incomplete because its managed-workspace guard misses the inverse profile-transition case while also changing runtime resolver behavior beyond the wizard default.

## What This Changes

The PR changes agent workspace resolution to avoid nesting added agents under setup-created main workspaces, adds resolver regression tests, documents peer-level workspaces, and adds a changelog entry for #71889.

## Best Possible Solution

Land one canonical #71889 fix that keeps brand-new agents add wizard defaults peer-level, preserves explicit workspaces and intentional shared-base configs, covers profile transitions directly, and avoids silently relocating existing agents unless maintainers explicitly choose a runtime resolver migration.

## Reproduction Assessment

Yes. The static path is high-confidence: setup writes agents.defaults.workspace, agents add uses resolveAgentWorkspaceDir for the prompt initialValue, and the resolver joins that fallback with the new non-default id; the PR’s guard can be checked against a profiled fallback such as <stateDir>/workspace-dev under a default-profile environment.

## Solution Assessment

No. This patch improves the default setup case, but it is not the narrowest maintainable fix yet because it changes runtime resolution and still misses a setup-created workspace from a different profile; a wizard-only suggestion helper or a more complete managed-workspace classifier would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Recognize profiled setup workspaces before nesting:** `src/agents/agent-scope-config.ts:72-73`
  - body: The guard only treats `<stateDir>/workspace` and the current profile's `resolveDefaultAgentWorkspaceDir(env)` as setup-managed. If setup ran with `OPENCLAW_PROFILE=dev`, it persists `<stateDir>/workspace-dev`; running `agents add work` later without that profile leaves the fallback unrecognized, so this branch still returns `<stateDir>/workspace-dev/work`, the same nested-under-main layout this PR is meant to prevent.
  - confidence: 0.84
