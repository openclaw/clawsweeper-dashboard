## Summary

Keep open. The PR targets a real cron delivery bug, but the current patch restores the entire pre-run delivery object and would overwrite delivery writebacks that current main deliberately preserves after force reload.

## What This Changes

The PR snapshots `CronJob.delivery` before manual, timer, and startup catch-up executions, restores it when applying outcomes, adds delivery-channel regression coverage, and changes an existing Telegram writeback expectation.

## Best Possible Solution

Preserve the user-selected delivery identity across execution without restoring the whole object, so legitimate target/account/thread writebacks and post-reload persisted edits survive.

## Reproduction Assessment

Yes for the PR blocker. Current main has a focused manual `cron.run` regression test that simulates a persisted Telegram target writeback and expects the rewritten target to survive, while this PR changes that expected result and restores delivery wholesale after reload.

## Solution Assessment

No. Whole-object restoration is broader than the reported `delivery.channel` regression; cloning execution input or reconciling only immutable delivery identity fields would be narrower and safer.
