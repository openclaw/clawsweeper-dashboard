## Summary

Keep open: the reported cron delivery-channel bug is real enough to fix, but this branch restores the entire pre-run delivery object and would clobber intentional delivery writebacks that current main explicitly preserves.

## What This Changes

The PR snapshots and restores `CronJob.delivery` across manual, timer, and startup catch-up runs, and adds/updates cron regression tests for preserved Telegram delivery.

## Best Possible Solution

Protect explicit delivery channel identity by cloning execution inputs or reconciling only the minimal immutable fields, while preserving reloaded delivery target writebacks.

## Reproduction Assessment

Yes for the PR regression: current main has a focused manual-run test that simulates a persisted Telegram target writeback and expects it to survive, while the PR changes that expectation. The original channel-revert report is source-plausible on timer/startup paths because current main passes the stored job object into execution, but I did not run live cron in this read-only review.

## Solution Assessment

No. Restoring the whole delivery object is broader than needed; cloning the execution job or reconciling only explicit channel identity would fix the reported overwrite without discarding valid delivery writebacks.
