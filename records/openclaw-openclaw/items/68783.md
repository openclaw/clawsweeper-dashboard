## Summary

Keep open: the PR targets a real cron delivery bug, but the current patch regresses existing delivery writeback behavior by restoring the entire pre-run delivery object after store reloads.

## What This Changes

The PR snapshots `CronJob.delivery` for manual, timer, and startup catch-up executions, restores it after each run, and adds/updates cron regression tests around Telegram delivery preservation.

## Best Possible Solution

Protect the explicit channel identity by cloning execution inputs or reconciling only immutable fields, while preserving persisted delivery target, account, and thread writebacks after reload.

## Reproduction Assessment

Yes for the PR regression: current main has a focused manual-run regression that writes a new Telegram target during execution and expects it to persist, while the PR changes that expectation. The original channel-revert report is source-plausible on scheduled/startup paths because those paths execute with the stored job object and later reload before persisting outcomes.

## Solution Assessment

No. Restoring the whole delivery object is broader than needed; the safer fix is to isolate the execution-time mutation or reconcile only the channel field without discarding valid persisted delivery writebacks.
