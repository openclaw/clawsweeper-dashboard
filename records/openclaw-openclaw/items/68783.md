## Summary

Keep this PR open, but do not merge it as-is. The proposed whole-object delivery restore conflicts with the current post-run reload/writeback flow and would regress persisted Telegram target normalization that current main already protects.

## What This Changes

The PR snapshots `CronJob.delivery` for manual, timer, and startup catch-up cron runs, restores it after outcomes, adds delivery-channel preservation coverage, and changes an existing manual-run writeback expectation.

## Best Possible Solution

Fix the channel-revert by cloning execution input or reconciling only immutable delivery identity fields, while preserving post-reload `delivery.to`, `accountId`, `threadId`, and other legitimate writebacks.

## Reproduction Assessment

Yes for the PR regression: current main has a focused manual-run test that simulates a persisted Telegram target writeback and expects it to survive, while this PR changes that expectation. The original channel-revert report is source-plausible on timer/startup paths, but I did not run live cron in this read-only review.

## Solution Assessment

No. Restoring the entire `delivery` object is broader than the bug requires; a narrower execution clone or identity-field reconcile would preserve explicit channel config without clobbering valid delivery writebacks.
