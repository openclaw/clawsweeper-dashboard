## Summary

Keep this PR open, but the current patch should not merge: the bug target is valid, yet restoring the whole pre-run `delivery` object discards legitimate delivery writebacks that current main intentionally preserves.

## What This Changes

The PR snapshots `CronJob.delivery` for manual, timer, and startup cron executions, restores it after run outcomes, adds channel-preservation tests, and changes an existing Telegram writeback regression expectation.

## Best Possible Solution

Replace whole-object delivery restoration with a narrower reconcile that preserves user-selected delivery identity across manual, timer, and startup runs while keeping legitimate post-reload target/account/thread writebacks.

## Reproduction Assessment

Yes. Current main has a focused manual-run regression test that simulates persisted Telegram target writeback during an isolated run and asserts the rewritten target survives; the PR changes that expectation to the stale original value.

## Solution Assessment

No. Restoring the entire `delivery` object is broader than the bug requires; cloning execution input or reconciling only immutable identity fields such as `mode` and `channel` would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve delivery writebacks after manual runs:** `src/cron/service/ops.ts:76`
  - body: `mergeManualRunSnapshotAfterReload` now writes the pre-run `delivery` snapshot onto the force-reloaded job. Manual `cron.run` reloads before final persist so isolated Telegram target normalization/writeback survives, but this assignment reverts fields like `delivery.to`, `accountId`, or `threadId` and regresses the existing writeback flow.
  - confidence: 0.93
- **[P1] Preserve timer delivery writebacks after reload:** `src/cron/service/timer.ts:614`
  - body: `applyOutcomeToStoredJob` runs after the timer path reloads the store from disk, so assigning `result.deliverySnapshot` overwrites delivery data saved during execution. Scheduled and startup catch-up runs can lose normalized targets or concurrent delivery edits when the final persist writes this restored object back.
  - confidence: 0.9
