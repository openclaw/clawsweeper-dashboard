## Summary

Keep this PR open, but do not merge it as-is. The linked cron delivery-channel regression is still plausible on current main, yet the PR restores the entire pre-run delivery object and would discard legitimate delivery writebacks that current code and tests intentionally preserve.

## What This Changes

The PR snapshots `CronJob.delivery` before manual, timer, and startup cron executions, restores that snapshot after outcomes are applied, adds a delivery-channel preservation test, and changes an existing Telegram writeback regression test to expect the original target.

## Best Possible Solution

Keep the contribution open and replace the whole-object restore with a narrower reconcile: prevent execution-time mutation from changing user-selected delivery identity such as `channel`/`mode`, while preserving legitimate post-reload target/account/thread writebacks. Coverage should exercise manual, timer-driven, and startup catch-up runs for both channel preservation and Telegram target writeback.

## Reproduction Assessment

Yes. The current regression test at `src/cron/service.issue-regressions.test.ts:318` provides a high-confidence reproduction for the PR-introduced regression by simulating an isolated run that writes back a normalized Telegram target during `cron.run`.

## Solution Assessment

No. Restoring the entire pre-run `delivery` object is broader than the bug requires; a field-wise reconcile or execution-only clone is safer because cron delivery is shared between the agent and runner and current behavior intentionally preserves persisted target writebacks.
