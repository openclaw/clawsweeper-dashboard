## Summary

Keep open: the PR targets a plausible cron delivery-channel bug, but the whole-delivery restore regresses current Telegram target writeback semantics protected by main's service flow and regression coverage.

## What This Changes

The PR snapshots and restores `CronJob.delivery` around manual, timer, and startup catch-up executions, adds delivery-channel preservation tests, and changes the existing manual Telegram writeback expectation.

## Best Possible Solution

Protect the immutable delivery channel identity by cloning execution inputs or reconciling only the fields that must not drift, while preserving persisted target/account/thread writebacks.

## Reproduction Assessment

Yes, source-backed. Current main has a focused manual-run regression that preserves persisted Telegram `delivery.to` writeback, and the PR changes that expectation while adding code that overwrites reloaded delivery state.

## Solution Assessment

No. Restoring the entire delivery object is broader than the bug requires and discards valid delivery writebacks; a narrower execution clone or field-level reconciliation is safer.
