## Summary

Keep this PR open, but not mergeable as-is: it targets a real cron delivery regression, yet the current patch discards valid delivery writebacks that current main intentionally preserves.

## What This Changes

The PR snapshots and restores `CronJob.delivery` around manual, scheduled, and startup cron executions and adds Telegram channel preservation regression coverage.

## Best Possible Solution

Clone execution inputs or reconcile only immutable channel identity so explicit `delivery.channel` survives while reloaded `delivery.to`, `accountId`, and `threadId` writebacks remain intact.

## Reproduction Assessment

Yes for the PR regression from source: current main has a focused manual-run regression that preserves persisted Telegram target writeback, while this PR changes that expectation and restores the pre-run delivery object. The original timer/startup channel-revert report is source-plausible because current main passes stored jobs into those execution paths, but I did not live-run cron in this read-only sweep.

## Solution Assessment

No. Restoring the whole delivery object is broader than needed; cloning the execution job or merging only the explicit channel identity would fix the reported overwrite without discarding valid delivery writebacks.
