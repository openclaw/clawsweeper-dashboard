## Summary

Keep open: the PR targets a real cron delivery-channel bug, but the current patch restores the entire `delivery` object after reload and regresses existing delivery writeback semantics.

## What This Changes

The PR snapshots and restores `CronJob.delivery` around manual, timer-driven, and startup catch-up executions, and adds Telegram delivery-channel preservation tests.

## Best Possible Solution

Protect the configured channel from execution-time mutation while preserving valid persisted delivery target, account, and thread writebacks after reload.

## Reproduction Assessment

Yes, source-reproducible: applying the PR would flip the existing manual-run regression that currently expects persisted Telegram target writeback to survive. Timer and startup paths have the same clobber risk because they reload the store before final outcome persistence.

## Solution Assessment

No. Restoring the whole delivery object is broader than the reported channel protection; a narrower fix should isolate execution inputs or reconcile only the channel identity without discarding persisted delivery writebacks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve manual delivery writebacks:** `src/cron/service/ops.ts:120-123`
  - body: Manual runs force-reload the cron store before final persist so isolated delivery target normalization can survive execution. Assigning the pre-run delivery snapshot over the reloaded job discards valid `delivery.to`, `accountId`, or `threadId` updates and the PR changes the existing regression to expect that loss.
  - confidence: 0.93
- **[P1] Preserve scheduled delivery writebacks:** `src/cron/service/timer.ts:742-749`
  - body: `applyOutcomeToStoredJob` runs after scheduled and startup executions have reloaded jobs from disk. Restoring the old `deliverySnapshot` here overwrites any delivery target/account/thread edits persisted during the run window before the final persist.
  - confidence: 0.9
