## Summary

Keep open. The reported cron delivery-channel bug is valid, but this PR is not mergeable as written because it restores the whole pre-run `delivery` object after reload and would clobber legitimate delivery target/account/thread writebacks that current main intentionally preserves.

## What This Changes

The PR snapshots `CronJob.delivery` before manual, timer, and startup cron executions, restores that snapshot after outcomes, and adds regression coverage for preserving `delivery.channel`.

## Best Possible Solution

Preserve the user-configured delivery channel by isolating execution-time mutation or reconciling only immutable delivery identity fields, while keeping valid persisted delivery target/account/thread writebacks.

## Reproduction Assessment

Yes for the PR regression: current main has a focused manual-run test that mutates persisted `delivery.to` during execution and expects the rewritten Telegram target to survive. I did not run tests in this read-only review because `pnpm` is unavailable on PATH.

## Solution Assessment

No. Restoring the entire `delivery` object is broader than the bug requires; a narrower execution clone or field-level reconcile would preserve `delivery.channel` without clobbering writebacks.
