## Summary

Keep this PR open, but it still needs changes before merge. The reported cron delivery regression is valid, but the current patch restores the entire pre-run delivery object after store reloads, which regresses the existing delivery writeback contract for fields such as `to`, `accountId`, and `threadId`.

## What This Changes

The PR snapshots and restores `CronJob.delivery` across manual, timer, and startup cron executions and adds regression coverage for preserving a Telegram delivery channel.

## Best Possible Solution

Keep the cron delivery identity stable across executions while reconciling only the fields that must be immutable, so legitimate persisted target/account/thread writebacks still survive reloads.

## Reproduction Assessment

Yes. Current main has a focused manual-run regression that writes a new Telegram target during execution and expects it to remain persisted, and the PR changes that expectation while adding whole-object restore code that would overwrite it.

## Solution Assessment

No. Restoring the entire delivery object is broader than the bug requires; a narrower reconciliation of immutable identity fields, or avoiding mutation of the execution input, would preserve both the configured channel and legitimate writebacks.
