## Summary

Keep open: this is a plausible xAI provider feature, but the current patch still has source-reproducible OAuth lifecycle/refresh defects and only partial real behavior proof. The protected `maintainer` label also means this PR needs explicit maintainer handling rather than cleanup closure.

## What This Changes

The PR adds an xAI Grok OAuth auth method and manifest choice, wires xAI provider OAuth refresh, and updates xAI docs, changelog, and tests.

## Best Possible Solution

Land this as provider-owned xAI OAuth after fixing refresh expiry semantics and callback cleanup, then add real OAuth completion or refresh proof while keeping the generic SDK auth seam small.

## Reproduction Assessment

Yes for the review findings by source inspection: the PR head preserves stale expiry on refresh responses without `expires_in`, and it can leave a local callback listener running after pre-callback failures. I did not reproduce a live SuperGrok OAuth completion.

## Solution Assessment

No. The provider-owned direction and generic `extraAuth` seam are reasonable, but the patch should make refresh expiry and callback lifecycle behavior safe before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Handle refresh responses without a new expiry:** `extensions/xai/xai-oauth.ts:336`
  - body: `parseXaiOAuthTokenResponse` allows `expires_in` to be absent, but refresh then spreads the old credential and only overwrites `expires` when a new value is present. If the stored credential was already expired, the refreshed access token is saved with the old expired timestamp, and current auth-profile logic will immediately treat it as unusable and refresh again. Derive or require a fresh expiry, or fail the refresh instead of preserving stale expiry metadata.
  - confidence: 0.82
- **[P2] Close the callback listener on pre-callback failures:** `extensions/xai/xai-oauth.ts:255-268`
  - body: The OAuth listener is started before note/browser-open work, but the promise is detached and there is no abort handle. If any step before awaiting `callbackPromise` throws, this function exits while `waitForLocalOAuthCallback` keeps the port bound until callback, server error, or the five-minute timeout, so a retry can fail with the callback port still in use. Add an explicit cleanup/abort path or structure the flow so the listener is always awaited and closed.
  - confidence: 0.78
