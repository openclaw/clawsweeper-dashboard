## Summary

Keep open: the branch adds a useful multimodal chat.completions feature, but the current head is not mergeable because audio inputs can still be dropped or misplaced in the prompt, and the PR body still lacks inspectable real behavior proof.

## What This Changes

The PR adds `/v1/chat/completions` handling for `input_audio`, `file`, and `video_url` blocks, new audio/files config schema/types, and focused gateway/config tests.

## Best Possible Solution

Keep the multimodal gateway direction, but revise the audio path to fail closed for malformed or untranscribable input and make successful transcripts part of the active user turn before prompt formatting.

## Reproduction Assessment

Yes. Source inspection of the PR head shows `transcribeFirstAudio` returning `undefined` leaves `audioTranscripts` empty without error, and blank `input_audio.data` is dropped before validation, so the failure paths are reproducible from code.

## Solution Assessment

No, not as currently patched. Reusing the existing STT and file-extraction pipelines is maintainable, but the PR should reject unusable audio and merge transcript text into the active user message before `buildAgentPrompt` selects the current turn.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Reject audio when transcription produces no text:** `src/gateway/openai-http.ts:647-649`
  - body: `transcribeFirstAudio` returns `undefined` for disabled, unconfigured, failed, or empty transcription, but this branch only skips adding text and lets the request continue. For an audio-only turn, the gateway can return 200 while the user's speech is absent from the prompt; return an `invalid_request_error` when supplied audio cannot produce transcript text.
  - confidence: 0.91
- **[P2] Attach audio transcript to the active turn:** `src/gateway/openai-http.ts:1071`
  - body: `buildAgentPrompt` has already formatted the current-message boundary before this prefix is added. In multi-turn audio-only requests, the active turn remains a generic placeholder while the spoken text sits outside the current user turn; merge the transcript before prompt assembly or make it the active turn content.
  - confidence: 0.86
- **[P2] Reject empty input_audio blocks instead of dropping them:** `src/gateway/openai-http.ts:357-360`
  - body: When an `input_audio` part is present but `input_audio.data` is missing or blank, this path returns `undefined` and silently skips it. If that malformed block is the newest user turn after prior history, the request can ignore the active turn and answer older context instead of returning a 400 validation error.
  - confidence: 0.88
