## Summary

Keep open. Current main still lacks audio/file/video_url handling for `/v1/chat/completions`, so the PR is useful, but the current PR head still has blocking audio validation and prompt-assembly defects. Earlier review comments were partly addressed for schema, file parsing, and base64 validation, but the remaining audio issues are still present at head `44f27d9b4416dda4db935b1232ba5b40625544d3`.

## What This Changes

The PR branch adds `input_audio`, `file`, and `video_url` content-block handling to the OpenAI-compatible `/v1/chat/completions` gateway endpoint, plus chatCompletions audio/files config schema/types and focused gateway/config tests.

## Best Possible Solution

Revise the PR so audio content is validated as request input, failed or empty transcription returns a clear `400 invalid_request_error`, and successful transcripts are assembled into the active user turn before prompt/history formatting. Then document the new chat-completions media contract, add the required changelog entry, and run targeted gateway/config tests plus the changed gate.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: make `transcribeFirstAudio` return `undefined` for a request containing `input_audio`, or send a latest user turn with blank `input_audio.data` after prior chat history; the PR code proceeds without the speech content or drops the malformed active turn instead of returning a 400.

## Solution Assessment

No. Reusing the existing STT and file-extraction pipelines is the right direction, but the current patch should fail closed for bad/untranscribed audio and merge transcripts into the active user message before prompt assembly.

- **[P1] Reject audio when transcription produces no text:** `src/gateway/openai-http.ts:647-649`
  - body: `transcribeFirstAudio` returns `undefined` for provider/config/runtime failures and no transcript, but this code only skips pushing a transcript and lets the request continue. For an audio-only turn, the gateway can dispatch a prompt containing only the generic audio placeholder and return 200 even though the user's speech was not included. Treat a supplied audio part with no transcript as invalid request input.
  - confidence: 0.91
- **[P2] Merge transcripts into the active user turn:** `src/gateway/openai-http.ts:1071`
  - body: At this point `buildAgentPrompt` has already formatted history and selected the current message as the media-only placeholder. Prefixing the transcript onto the whole prompt leaves the actual speech outside the active user turn in multi-turn requests. Add the transcript before prompt assembly or replace the active turn content with it.
  - confidence: 0.86

  - body: When `input_audio.data` is missing or blank, `resolveInputAudioPart` returns `undefined` and the block is silently dropped.   - confidence: 0.88
