## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reject audio when transcription produces no text:** `src/gateway/openai-http.ts:647-649`
  - body: `transcribeFirstAudio` returns `undefined` when STT is disabled, fails, or produces no transcript, but this branch only appends truthy transcripts and then lets the request continue. An audio-only turn can return 200 with only the synthetic placeholder, so the user's speech is silently omitted instead of producing a clear `invalid_request_error`.
  - confidence: 0.93
- **[P2] Reject empty input_audio blocks instead of dropping them:** `src/gateway/openai-http.ts:357-360`
  - body: Returning `undefined` for an `input_audio` part with missing or blank `data` makes `extractAudioParts` skip a malformed client content block. For a latest media-only user turn this can erase the supplied input and fall through to stale prompt text or the generic missing-message path rather than returning a targeted 400.
  - confidence: 0.9
- **[P2] Reject unsupported file_id file blocks instead of dropping them:** `src/gateway/openai-http.ts:396-400`
  - body: A `file` block with `file_id` but no `file_data` takes this early return and is silently ignored. Since this endpoint does not resolve OpenAI file IDs, it should reject that unsupported shape with 400 or implement the lookup; accepting the block type while dropping the attachment causes message loss.
  - confidence: 0.86
- **[P2] Attach audio transcripts to the active user turn:** `src/gateway/openai-http.ts:1058-1071`
  - body: The transcript is prefixed after `buildAgentPrompt` has already selected and formatted the current message. In multi-turn chats the active user message remains the generic audio placeholder while the actual spoken text sits outside the current-turn structure, so the agent can underweight or mis-associate the speech.
  - confidence: 0.84
- **[P2] Document the new chat media config keys:** `src/config/types.gateway.ts:240-243`
  - body: The PR adds operator-facing `chatCompletions.audio` and `chatCompletions.files` config, but the public gateway docs and schema help/labels still only describe the existing chat-completions controls. Gateway config additions need documented defaults and operator impact, especially because these media handlers default enabled once the endpoint is enabled.
  - confidence: 0.82
