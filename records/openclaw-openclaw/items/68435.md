## Summary

Keep open: current main still lacks the requested chat.completions audio/file handling, but this PR is not merge-ready because the audio path can accept failed or malformed input and places transcripts outside the active user turn.

## What This Changes

The PR adds audio, file, and video_url content-block handling to `/v1/chat/completions`, expands chatCompletions audio/files config/schema, and adds focused multimodal gateway/config tests.

## Best Possible Solution

Revise the PR so chat.completions validates media fail-closed, merges audio transcripts into the active user turn before prompt formatting, and aligns the new config surface with schema help, docs, and changelog before the changed gate.

## Reproduction Assessment

Yes. A source-level reproduction is clear: make `transcribeFirstAudio` return `undefined` for a request containing `input_audio`, or send a latest user turn with blank `input_audio.data` after prior history; the PR code proceeds without the supplied speech or falls back to stale prior content.

## Solution Assessment

No. Reusing the existing STT and file-extraction pipelines is the right direction, but the current patch should reject untranscribed or malformed audio and assemble transcripts into the active user message before prompt history/current-message formatting.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Reject audio when transcription produces no text:** `src/gateway/openai-http.ts:647-649`
  - body: `transcribeFirstAudio` returns `undefined` for provider/config/runtime failures and for missing transcripts, but this code only skips pushing a transcript and lets the request continue. For an audio-only turn, the gateway can return 200 with only the generic audio placeholder instead of the user's speech. Treat a supplied audio part with no transcript as `invalid_request_error`.
  - confidence: 0.91
- **[P2] Merge transcripts into the active user turn:** `src/gateway/openai-http.ts:1071`
  - body: `buildAgentPrompt` has already formatted history and marked the current message as the media-only placeholder before this prefix is added. In multi-turn requests, the actual speech ends up outside the current-message block, so the agent can answer the placeholder or stale context. Add the transcript before prompt assembly or replace the active turn content with it.
  - confidence: 0.86
- **[P2] Reject empty input_audio blocks:** `src/gateway/openai-http.ts:357-360`
  - body: When `input_audio.data` is missing or blank, this path returns `undefined` and the part is silently skipped. If that malformed block is the latest user turn after prior history, the prompt builder can ignore it and use older conversation content instead of returning a 400.
  - confidence: 0.88
- **[P3] Add help, docs, and changelog for the new config keys:** `src/config/types.gateway.ts:240-243`
  - body: This adds user-facing `gateway.http.endpoints.chatCompletions.audio` and `.files` config, but the PR head does not add schema help entries, docs, or an Unreleased changelog entry. Operators using `openclaw config` help or the public Gateway config docs will not see the new controls, and repo config-contract policy expects those surfaces to stay aligned.
  - confidence: 0.82
