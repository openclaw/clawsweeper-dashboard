## Summary

Keep open for normal PR review: current main still uses the hardcoded Codex terminal-idle default, and this PR is a focused fix with no blocking findings. Maintainers should explicitly accept that long-budget runs can now wait longer before stale-terminal cleanup.

## What This Changes

The PR passes the effective Codex run timeout into the app-server terminal-idle resolver, preserves explicit overrides, and adds focused resolver tests.

## Best Possible Solution

Land the bounded Codex app-server fix that ties terminal-idle to the effective run timeout while preserving explicit internal overrides and the 30-minute floor; handle any remaining diagnostic-copy polish as a separate follow-up if maintainers want it.

## Reproduction Assessment

Yes, at source level: current main passes only `options.turnTerminalIdleTimeoutMs` to the terminal-idle resolver, so a run with `params.timeoutMs` above 30 minutes still uses the 1,800,000ms default unless an internal override is supplied. I did not wait through a live 30-minute scheduled Codex turn in this read-only pass.

## Solution Assessment

Yes. Deriving the terminal-idle ceiling from the already-effective run timeout is the narrow maintainable fix and avoids adding a new public config option, while preserving explicit internal overrides.
