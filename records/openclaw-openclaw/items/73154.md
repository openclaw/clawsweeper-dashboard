## Summary

Keep open. The PR targets a real remaining refresh/reconnect data-loss gap, but the current patch is not merge-ready because it can drain restored follow-ups before the prior run is actually complete, silently replays attachment metadata without payload bytes, bypasses the shared session-switch helper added on main, and needs retention/changelog follow-through.

## What This Changes

The PR adds localStorage-backed WebChat chatQueue persistence, restores queues during lifecycle and session switching, persists queue mutations, and drains restored queues after gateway hello.

## Best Possible Solution

Implement durable WebChat queue persistence in the shared chat/session lifecycle, coordinate restored drains with confirmed active-run state, define attachment and retention behavior, add focused UI tests, and include the required user-facing changelog entry.

## Reproduction Assessment

Yes. A static reproduction is clear: queue a follow-up or attachment, refresh during an active run, then the PR restores before hello and drains after chatRunId is cleared while the old run may still be active; attachment replay restores metadata without payload bytes.

## Solution Assessment

No. localStorage may be part of the solution, but this patch is not the narrowest maintainable fix until it uses the shared session helper, waits for safe active-run ordering, and makes attachment/privacy behavior explicit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Wait for the active run before draining restored queues:** `ui/src/ui/app-gateway.ts:460-463`
  - body: On refresh during an active response, this unconditional hello-time flush can send restored follow-ups immediately after chatRunId is cleared even though the previous run may still be active server-side. The restored queue should wait for a confirmed idle/completed state before draining.
  - confidence: 0.84
- **[P2] Preserve or reject restored attachment payloads:** `ui/src/ui/chat-queue-persistence.ts:73`
  - body: Restored queue items can keep attachment metadata, but the payload bytes live in the in-memory attachment store and are gone after refresh. buildApiAttachments then drops those attachments, so replayed messages silently lose files unless the patch persists recoverable bytes or rejects those entries visibly.
  - confidence: 0.9
- **[P2] Restore queues through the shared session helper:** `ui/src/ui/app-render.ts:1575-1579`
  - body: Current main routes session switching through switchChatSession and app-render.helpers.ts owns queue save/restore. Adding a separate restore in this app-render callback duplicates the lifecycle and misses other switch paths; the durable restore should be integrated into the shared helper path.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `ui/src/ui/app-chat.ts:154`
  - body: This is a user-facing WebChat bug fix, but the PR patch does not update CHANGELOG.md. OpenClaw policy requires a single-line changelog entry before merge.
  - confidence: 0.87
