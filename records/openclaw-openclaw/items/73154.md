## Summary

Keep this PR open. It targets a real WebChat durability gap, but the latest patch is not merge-ready against current main: it drains restored messages before active-run ordering is established, does not handle restored attachment payloads under the current attachment side-store design, wires session restore into a stale inline path instead of the shared session helper, and lacks the required user-facing changelog entry.

## What This Changes

The PR adds localStorage-backed Control UI/WebChat chatQueue persistence, restores queued items during lifecycle and session changes, persists queue mutations, and drains restored items after gateway hello.

## Best Possible Solution

Implement durable queue restore on top of the current shared WebChat session lifecycle, with explicit active-run wait or ACK semantics, a defined attachment restore policy, focused regression tests, and a single-line changelog entry for the user-facing fix.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: refresh with queued messages during an active response, restore them from localStorage, then the PR's onHello path clears chatRunId and drains the queue before the still-active server-side run has finished; attachment loss follows from restoring metadata without the in-memory payload store.

## Solution Assessment

No. localStorage can be part of the fix, but this patch is not the best maintainable solution until it is rebased into the shared session helper flow and made explicit about active-run ordering, attachment recovery, and browser retention behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Wait for the active run before draining restored queues:** `ui/src/ui/app-gateway.ts:463-465`
  - body: The PR flushes restored queues immediately after hello, but this same handler has just cleared chatRunId for any run that was active before refresh. A queued follow-up restored from localStorage can be sent as a new chat.send before the server's still-running response for the same session finishes, violating the queue ordering users expect.
  - confidence: 0.83
- **[P2] Preserve attachment payloads or reject restored attachment queues:** `ui/src/ui/chat-queue-persistence.ts:76-78`
  - body: Current main keeps attachment bytes in attachment-payload-store while queue items carry metadata. After a browser refresh, restored parsed.items can list attachments but have no recoverable dataUrl in the side store, so buildApiAttachments filters them out and the replayed message loses its files. Persist a recoverable payload path or visibly skip these entries.
  - confidence: 0.9
- **[P2] Restore persisted queues through switchChatSession:** `ui/src/ui/app-render.ts:1578-1580`
  - body: Current main routes session changes through switchChatSession in app-render.helpers.ts, where queue save/restore now lives. Adding localStorage restore only to this older inline callback misses other session-switch paths and conflicts with the helper-owned lifecycle after rebasing.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `ui/src/ui/app-chat.ts:157`
  - body: This is a user-facing WebChat data-loss fix, but the PR does not update CHANGELOG.md. OpenClaw policy requires a single-line fix entry before merge.
  - confidence: 0.87
