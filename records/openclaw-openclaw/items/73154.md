## Summary

Keep open. Current main does not implement the durable localStorage queue behavior, so this PR is not obsolete, but the patch is not merge-ready because it misses current main's shared session-switch path, would restore attachment-bearing queue items without recoverable payload bytes, and introduces browser-side message retention that needs maintainer privacy review.

## What This Changes

The PR adds localStorage-backed WebChat chatQueue persistence and wires queue save, restore, and reconnect drain behavior through Control UI chat, lifecycle, gateway, and render paths.

## Best Possible Solution

Rebase the branch onto current main and implement durable queue restore through the shared chat queue/session-switch lifecycle, with an explicit attachment payload policy, explicit retention/clear behavior, focused coverage for refresh/reconnect/session-switch cases, and a user-facing changelog entry. Keep broader conversation-history and draft restore scoped to the linked issue unless this PR intentionally expands.

## Reproduction Assessment

Yes for the review findings. Current main stores queued attachment payload bytes outside chatQueue and sends only attachments with resolvable dataUrl values, and current main routes session changes through switchChatSession rather than the older inline app-render reset block.

## Solution Assessment

No as written. localStorage can be a narrow persistence layer for queued text messages, but the implementation should be rebased into the shared helper path and must make an explicit, tested choice for attachments and browser-local retention.
