## Summary

Keep open. The PR targets a real WebChat queue-loss problem, but the current patch still has blocking behavior issues around active-run ordering, attachment replay, and integration with the shared session-switch queue path, plus it lacks required real behavior proof.

## What This Changes

The branch adds localStorage-backed WebChat chatQueue persistence, restores queued items during lifecycle/session changes, and drains restored queues after gateway hello.

## Best Possible Solution

Implement durable WebChat queue persistence through the shared chat/session lifecycle, wait until the prior active run is safely complete before replay, define attachment and retention behavior, then add focused regression coverage plus real browser proof.

## Reproduction Assessment

Yes, source-level. The PR restores a queue before gateway hello, clears local active-run state, and can drain immediately while attachment payload bytes and prior run completion state are not recoverable from localStorage alone.

## Solution Assessment

No. localStorage may be part of the fix, but this patch is not the best solution until it uses the shared session helper, waits for safe active-run ordering, and defines attachment and retention semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Wait for active-run completion before draining restored queues:** `ui/src/ui/app-gateway.ts:463-465`
  - body: The PR clears local run state and then flushes restored localStorage queues immediately after gateway hello. On refresh while the prior run is still active server-side, `isChatBusy` sees no local run and can send follow-ups before the response they were queued behind finishes.
  - confidence: 0.84
- **[P2] Persist recoverable attachments or reject them visibly:** `ui/src/ui/chat-queue-persistence.ts:22-40`
  - body: Queued attachments are metadata-only while payload bytes live in the in-memory attachment store. After refresh, restored JSON entries can replay messages without their files unless the PR stores recoverable payloads safely or drops those queued items with visible feedback.
  - confidence: 0.9
- **[P2] Extend the shared session switch helper:** `ui/src/ui/app-render.ts:1577-1580`
  - body: Current main saves and restores queued messages through `switchChatSession` and `resetChatStateForSessionSwitch`. Adding localStorage restore in one render callback leaves other switch paths unhandled and can conflict with the existing per-session queue map.
  - confidence: 0.82
