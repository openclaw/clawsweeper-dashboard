## Summary

Keep open. The target bug is real, but this PR is not ready for merge against current main because the restore path is stale relative to the shared session-switch helper and attachment-bearing queued messages would replay without payloads after refresh; localStorage retention also needs explicit privacy review.

## What This Changes

The PR adds localStorage-backed WebChat chatQueue persistence and wires queue save, restore, and reconnect drain behavior through the Control UI chat, lifecycle, gateway, and render paths.

## Best Possible Solution

Rebase onto current main and implement queue durability through the central chat queue/session-switch helpers, with explicit attachment payload handling, retention/clear behavior, focused coverage for refresh/reconnect/session switch, and a user-facing changelog entry; keep broader conversation-history and draft restoration scoped to the linked issue unless this PR intentionally expands.

## Reproduction Assessment

Yes for the review findings: current main queues attachment metadata while keeping payload bytes in an in-memory store, and buildApiAttachments drops entries without a resolvable dataUrl. Current main also routes session changes through switchChatSession, while the PR restores localStorage data only in an older app-render callback path.

## Solution Assessment

No as written. localStorage can be a narrow text-queue persistence layer, but the implementation should live in the shared session-switch/queue lifecycle and must make an explicit, tested choice for attachments and local browser retention.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve attachment payloads when restoring queued messages:** `ui/src/ui/chat-queue-persistence.ts:22-23`
  - body: Queued attachments only carry metadata in chatQueue; their dataUrl payloads live in the in-memory payload store. After a browser refresh, restored items from localStorage still have attachments but no payload, and buildApiAttachments filters those out, so the queued message is replayed without its files. Persist a recoverable payload path or reject/skip attachment-bearing restored entries with visible feedback.

After rebasing, restoring persisted queues only at this caller would miss other session-switch paths and conflict with the helper-owned queue lifecycle.
  - confidence: 0.78
