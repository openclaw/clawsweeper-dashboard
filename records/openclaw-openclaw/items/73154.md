## Summary

Keep open. The PR targets a real WebChat queue-loss problem, but it is not merge-ready against current main: it is dirty, restores persistence in an older session-switch path that current main replaced with the shared `switchChatSession` helper, and replays attachment-bearing queued messages without recoverable attachment payloads. The localStorage retention model also needs explicit privacy/product review before merge.

## What This Changes

The PR adds localStorage-backed WebChat `chatQueue` persistence, restores/deduplicates queued items during lifecycle and session changes, and drains restored entries after gateway hello.

## Best Possible Solution

Rebase onto current main and implement queue durability through the central chat queue/session-switch helpers, with a clear attachment policy: either persist a recoverable payload path or refuse/drop attachment-bearing persisted entries with visible feedback. Add focused coverage for refresh restore, reconnect drain, session switch, TTL/quota behavior, and attachment-bearing queued messages; keep broader conversation-history, draft restoration, and server ACK semantics tracked by the linked issues unless this PR intentionally expands scope.

## Reproduction Assessment

Yes. Source inspection reproduces the review findings: queued attachments are metadata-only after refresh while `chat.send` requires a data URL, and the PR API reports the branch dirty against current main.

## Solution Assessment

No as written. localStorage may be acceptable for a narrow text queue, but this patch needs to be rebased into the shared session-switch helper and reconciled with the attachment payload model and privacy retention tradeoff before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve attachments before replaying persisted queue:** `ui/src/ui/chat-queue-persistence.ts:22-23`
  - body: Queued attachments are stored as metadata while their data URLs live only in the in-memory payload store. After a browser refresh, restored localStorage items still have an attachments array but no payload, and `buildApiAttachments` filters those out, so the queued message is sent without its files. Either do not persist/replay attachment-bearing items, or persist a recoverable payload path with user-visible fallback.
  - confidence: 0.89
- **[P2] Restore queues through the shared session switch helper:** `ui/src/ui/app-render.ts:1574-1578`
  - body: Current main routes session changes through `switchChatSession` and keeps per-session queues in `app-render.helpers.ts`. Adding restore logic only to this older inline app-render handler will miss other session-switch paths after rebase and conflicts with the helper-owned queue lifecycle. Move persistence save/restore into the shared helper path.
  - confidence: 0.78

## Security Review

Status: needs_attention

Concerns:
