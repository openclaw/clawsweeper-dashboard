## Summary

Keep open. The underlying WebChat refresh queue-loss problem is still real, but this branch should not merge as-is because it drains restored queues before proving the active run has finished, misses current main's shared session-switch queue lifecycle, drops restored attachment payloads, and adds browser-side retention that needs explicit privacy review.

## What This Changes

The PR adds localStorage-backed Control UI WebChat queue persistence, restores queued messages on lifecycle/session changes, persists queue mutations, and drains restored queues after gateway hello.

## Best Possible Solution

Rebase onto current main and implement durable queue restore in the shared chat queue/session lifecycle, with an explicit active-run wait/ACK strategy, tested attachment restore or visible rejection behavior, reviewed browser-retention semantics, focused UI tests, and a user-facing changelog entry.

## Reproduction Assessment

Yes. The review has a high-confidence static reproduction path: refresh with queued messages during an active run, restore from localStorage in handleConnected, then app-gateway drains after hello while client run state is reset; attachment loss follows from metadata-only queue entries plus buildApiAttachments filtering unresolved payloads.

## Solution Assessment

No. localStorage can be part of a narrow durability fix, but this implementation is not the best maintainable path until it is moved into the shared session-switch helper flow, coordinated with active-run state, and made explicit about attachment and privacy behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Wait for the active run before draining restored queues:** `ui/src/ui/app-gateway.ts:455-459`
  - body: On a browser refresh during an active response, handleConnected restores the queue and this hello path immediately flushes it after client chatRunId has been cleared. The gateway accepts a new chat.send with a new idempotency key while another run for the same session can still be active, so queued follow-ups can run before the response they were queued behind finishes.
  - confidence: 0.83
- **[P2] Preserve attachment payloads when restoring queued messages:** `ui/src/ui/chat-queue-persistence.ts:73`
  - body: Queued attachments only store metadata in chatQueue while their dataUrl bytes live in an in-memory side store. After refresh, restored items can still list attachments but buildApiAttachments cannot resolve payloads and drops them, so the replayed queued message loses its files. Persist a recoverable payload path or reject/skip attachment-bearing restored entries with visible feedback.
  - confidence: 0.9
- **[P2] Restore persisted queues through the shared session helper:** `ui/src/ui/app-render.ts:1574-1579`
  - body: Current main routes session changes through switchChatSession and app-render.helpers.ts owns queue save/restore. Adding persistence only to the older inline app-render callback misses other session-switch paths and conflicts with the helper-owned lifecycle added on main.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `ui/src/ui/app-chat.ts:154`
  - body: This is a user-facing WebChat bug fix, but the PR does not update CHANGELOG.md. OpenClaw policy requires a single-line user-facing fix entry credited to the contributor before merge.
