## Summary

This PR should stay open: the current branch is still merge-blocked by restored-queue drain ordering, attachment replay semantics, and integration with the shared session-switch helper on main. The earlier Greptile build/drain/quota comments were addressed, but the later Codex review blockers still match the current PR head.

## What This Changes

The PR adds localStorage persistence for WebChat chatQueue items, restores them during lifecycle/session changes, and flushes restored queues after gateway hello.

## Best Possible Solution

Implement durable WebChat queue persistence through the shared chat/session lifecycle with active-run completion coordination, explicit attachment restore or visible rejection behavior, reviewed retention semantics, and focused regression tests.

## Reproduction Assessment

Yes, source-level: refresh can restore a queue before gateway hello, the PR then flushes after chatRunId is cleared while the old run may still be active, and restored attachment metadata cannot recover the in-memory payload bytes. I did not run a browser E2E because this review was read-only.

## Solution Assessment

No. localStorage can be part of the fix, but this patch is not the best solution until it uses the shared session helper, coordinates restored drains with active-run completion, and defines attachment and retention behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Wait for the active run before draining restored queues:** `ui/src/ui/app-gateway.ts:455-463`
  - body: On a refresh during an active response, the PR restores queued messages before gateway hello and flushes them as soon as hello arrives. The old run may still be active server-side, so follow-ups can be sent before the response they were queued behind finishes.
  - confidence: 0.84
- **[P2] Preserve or reject attachment payloads before persisting:** `ui/src/ui/chat-queue-persistence.ts:22`
  - body: Queue entries only carry attachment metadata while payload bytes live in the in-memory attachment store. Serializing and restoring those entries can replay messages without their files unless persistence stores recoverable payloads or rejects those entries with visible feedback.
  - confidence: 0.9
- **[P2] Restore persisted queues through the shared session helper:** `ui/src/ui/app-render.ts:1574-1579`
  - body: Current main routes session changes through switchChatSession and app-render.helpers.ts, where queue save/restore now lives. Adding localStorage restore in one app-render callback bypasses that owner path and can miss or conflict with other session-switch flows.
  - confidence: 0.82
