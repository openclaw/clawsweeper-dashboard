## Summary

Keep open: current main still lacks the requested agent-scoped internal hook policy surface, and this PR still has a blocking policy-resolution bug plus missing real behavior proof.

## What This Changes

The PR adds agent/default/per-agent internal hook policy config, gates internal hook dispatch by effective policy, threads config through hook call sites, and updates docs/tests.

## Best Possible Solution

Land a rebased implementation that applies agent overlays only after confirmed agent scope, preserves non-agent lifecycle hooks under global policy, refreshes generated config baseline artifacts, and includes redacted real-run proof.

## Reproduction Assessment

Yes for the blocking PR finding: PR-head source shows gateway startup passes config without agentId, while policy resolution applies agent defaults and can derive the default agent from a non-agent session key. I did not live-run it because this is a read-only review, but the call path is direct.

## Solution Assessment

No. The feature direction may be useful, but this branch should not merge until agent overlays are gated on confirmed agent scope, generated config artifacts are refreshed, and real behavior proof is provided.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate agent defaults on real agent scope:** `src/hooks/internal-hooks.ts:307-308`
  - body: When triggerInternalHook receives config but no explicit agent, non-agent lifecycle keys like gateway:startup can fall back to the default agent and merge agents.defaults.hooks. A default allowlist such as ["command:new"] would unexpectedly suppress gateway startup handlers, so only apply agent defaults/per-agent overlays after explicit or parseable agent scope is confirmed.
  - confidence: 0.91
- **[P3] Regenerate the config baseline hash:** `src/config/zod-schema.hooks.ts:123`
  - body: This change extends the config schema but does not update docs/.generated/config-baseline.sha256. The config:docs:check lane compares the generated config baseline against the checked-in hash, so the schema/docs surface will drift until generated artifacts are refreshed.
  - confidence: 0.78
