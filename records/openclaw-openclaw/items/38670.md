## Summary

Keep open: current main still lacks the proposed agent-scoped hook policy surface, and the PR has a source-proven lifecycle-hook filtering bug plus missing real behavior proof, so it should not be closed as obsolete or merged as-is.

## What This Changes

The PR adds new hook policy config at `hooks.internal`, `agents.defaults.hooks`, and `agents.list[].hooks`, wires policy filtering into internal hook dispatch, and updates schema, docs, and tests.

## Best Possible Solution

Rebase and revise the PR so agent overlays apply only to confirmed agent-scoped events, generated config docs are aligned, event allowlist semantics are explicit, and redacted real-run proof is added.

## Reproduction Assessment

Yes for the PR-introduced blocker by source inspection: PR head passes `gateway:startup` with config but no agent id, then infers the configured default agent and applies `agents.defaults.hooks`. I did not run tests because this review is read-only.

## Solution Assessment

No. The feature direction is plausible, but the patch should first gate agent overlays on confirmed agent scope, align generated config artifacts, clarify event matching semantics, and include real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate agent policies on real agent scope:** `src/hooks/internal-hooks.ts:307-308`
  - body: `resolveEffectiveInternalHookPolicy` always reads `agents.defaults.hooks` before real agent scope is confirmed. Because `triggerInternalHook` can infer the default agent from non-agent keys such as `gateway:startup`, an allowlist like `agents.defaults.hooks.events: ["command:new"]` suppresses startup handlers; apply default/per-agent overlays only for explicit or parseable agent-scoped events.
  - confidence: 0.91
- **[P3] Regenerate the config baseline hash:** `src/config/zod-schema.hooks.ts:123`
  - body: This schema change adds new config surface under `hooks.internal`, `agents.defaults`, and `agents.list`, but the PR head does not include `docs/.generated/config-baseline.sha256`. Regenerate and commit the config baseline hash so `config:docs:check` stays aligned with the schema.
  - confidence: 0.78
