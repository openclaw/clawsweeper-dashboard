## Summary

Keep open: current main still lacks agent-scoped internal hook policy overrides, but this PR is not merge-ready because the PR-head resolver can apply agent defaults to non-agent lifecycle hooks, the config baseline is missing, the branch is conflicting, and the contributor has not provided real behavior proof.

## What This Changes

This PR adds global/default/per-agent internal hook policy config, applies it during internal hook dispatch, threads config through hook call sites, and updates hooks/config docs plus regression tests.

## Best Possible Solution

Land a rebased hook-policy resolver that applies agent overlays only for confirmed agent-scoped dispatch, keeps gateway/system lifecycle hooks governed by global policy, regenerates the config baseline, and documents the event-match contract.

## Reproduction Assessment

Yes, for the blocking PR bug by source inspection: configure agents.defaults.hooks.events to only allow command:new, dispatch gateway:startup with config but no agent id, and PR-head code resolves a default agent before applying the allowlist. This was not live-run in the read-only review.

## Solution Assessment

No. The feature direction is plausible, but the current implementation should gate agent overlays on explicit or parsed agent scope, keep non-agent lifecycle hooks on global policy, regenerate generated config metadata, and provide real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate agent defaults on real agent scope:** `src/hooks/internal-hooks.ts:307-308`
  - body: resolveEffectiveInternalHookPolicy merges cfg.agents.defaults.hooks before knowing the event is agent-scoped. With gateway startup passed only config, getAgentIdForEvent can derive the default agent from the non-agent session key, so agents.defaults.hooks.events like ["command:new"] suppresses gateway:startup handlers. Only apply default/per-agent overlays after an explicit agentId or parsed agent session key.
  - confidence: 0.91
