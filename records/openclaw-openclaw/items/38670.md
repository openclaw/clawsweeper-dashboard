## Summary

Keep open: current main still lacks agent-scoped internal hook policy, but this PR is conflicting and source inspection shows it can let agent defaults filter non-agent lifecycle hooks; it also lacks after-fix real behavior proof from a real setup.

## What This Changes

The PR adds global/default/per-agent internal hook policy config, threads config through internal hook dispatch call sites, and updates hooks/config docs and tests.

## Best Possible Solution

Rebase the feature onto current gateway startup code and land a scoped policy resolver that applies agent overlays only for confirmed agent dispatch, with config baseline/docs/tests and redacted real behavior proof.

## Reproduction Assessment

Yes for the blocking review finding: source inspection of the PR branch shows config-only lifecycle events can resolve to the default agent and merge `agents.defaults.hooks`, so `events: ["command:new"]` can filter `gateway:startup`. This was not live-run, but the call path is clear from the PR source.

## Solution Assessment

No. The feature direction may be useful, but this implementation should first gate agent overlays on explicit or parseable agent scope and keep gateway/system lifecycle hooks governed by global policy unless a maintainer intentionally changes that contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate agent defaults on real agent scope:** `src/hooks/internal-hooks.ts:307-308`
  - body: When `triggerInternalHook` is called with config but no explicit agent, `getAgentIdForEvent` derives an agent from any session key, and `resolveEffectiveInternalHookPolicy` merges `agents.defaults.hooks`.
