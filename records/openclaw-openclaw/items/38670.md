## Summary

Keep open: current main still lacks the requested agent-scoped hook policy surface, but this PR should not merge yet because the branch has a source-reproducible lifecycle-hook suppression bug, a security-sensitive policy concern, and no contributor real-run proof.

## What This Changes

The PR adds `agents.defaults.hooks` and `agents.list[].hooks` config overlays for internal hooks, threads config through hook dispatch call sites, updates schema/types/docs, and adds hook policy tests.

## Best Possible Solution

Rework the branch so agent hook overlays apply only to confirmed agent-scoped events, preserve non-agent lifecycle hooks under global policy, document exact event-key semantics, refresh generated config artifacts, and attach redacted real-run proof.

## Reproduction Assessment

Yes for the blocking PR finding. Source inspection of PR head shows `gateway:startup` passes config without `agentId`, agent resolution falls back from the non-agent session key to the default agent, and `resolveEffectiveInternalHookPolicy` applies `agents.defaults.hooks`.

## Solution Assessment

No. The feature direction may be useful, but this branch should not merge until agent overlays are gated on confirmed agent scope, generated config artifacts are refreshed, exact event semantics are documented, and real behavior proof is provided.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate agent defaults on real agent scope:** `src/hooks/internal-hooks.ts:307-308`
  - body: `resolveEffectiveInternalHookPolicy` merges `agents.defaults.hooks` before any real agent scope is confirmed. Because `triggerInternalHook` can derive the default agent from non-agent keys such as `gateway:startup`, an allowlist like `agents.defaults.hooks.events: ["command:new"]` suppresses gateway startup handlers. Apply defaults/per-agent overlays only for explicit or parseable agent-scoped events; otherwise use only global `hooks.internal`.
  - confidence: 0.91
