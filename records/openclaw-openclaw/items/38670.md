## Summary

Keep open: current main does not implement this feature, and the PR still has a blocking policy bug where agent defaults can affect non-agent lifecycle hooks. The existing ClawSweeper review remains applicable, with generated config metadata and changelog follow-through still needed before merge.

## What This Changes

The PR adds `hooks.internal`, `agents.defaults.hooks`, and `agents.list[].hooks` policy overlays for internal hook dispatch, plus docs and tests for the new config shape.

## Best Possible Solution

Revise and rebase the PR so agent overlays apply only to confirmed agent-scoped dispatch, gateway/system lifecycle hooks remain governed by global policy, and generated schema metadata, docs, changelog, and focused regressions land together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a non-agent event such as `gateway:startup` uses the `gateway:startup` session key, and the PR's config-only policy resolution can merge `agents.defaults.hooks.events: ["command:new"]` before a real agent scope is proven.

## Solution Assessment

No. The feature direction may be valid, but this implementation is not the narrowest maintainable solution until it gates agent overlays on real agent scope and preserves global lifecycle hook behavior.

## Security Review

Status: needs_attention

Summary: The patch changes hook execution policy and can let agent defaults suppress non-agent lifecycle hooks.

Concerns:

- **[medium] Agent defaults can suppress lifecycle hooks:** `src/hooks/internal-hooks.ts:305`
  - body: Because `agents.defaults.hooks` is merged even when dispatch is not tied to a real agent scope, a default event allowlist can prevent `gateway:startup` or similar lifecycle handlers from running. Deployments relying on those hooks for audit, DLP, or compliance could unintentionally bypass controls.
