## Summary

Keep open. Current main still lacks agent-scoped internal hook policy overrides, but the PR should not merge as-is because source inspection shows agent defaults can suppress non-agent lifecycle hooks and the contributor has not provided real runtime proof.

## What This Changes

The PR adds `hooks.internal.events`, `agents.defaults.hooks`, and `agents.list[].hooks` policy schema/types/docs/tests, tracks hook names during registration, and threads config/agent metadata into internal hook dispatch.

## Best Possible Solution

Land a scoped hook-policy resolver in the current dispatch/startup architecture that applies agent overlays only for confirmed agent dispatch, preserves global lifecycle hooks, refreshes generated config docs, and includes focused tests plus redacted runtime proof.

## Reproduction Assessment

Yes for the blocking PR finding: source inspection of the PR head shows config-only lifecycle events can resolve a default agent and merge `agents.defaults.hooks`, so an allowlist like `["command:new"]` can filter `gateway:startup`. I did not live-run the PR branch.

## Solution Assessment

No. The feature direction may be useful, but this implementation should first gate agent overlays on explicit or parseable agent scope, keep gateway/system lifecycle hooks governed by global policy, update generated config baselines, and provide redacted real behavior proof.

## Review Findings
