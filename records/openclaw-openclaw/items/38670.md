## Summary

Keep open: current main still lacks the proposed agent-scoped hook policy surface, but this PR should not merge as-is because the PR head can apply agent defaults to non-agent lifecycle hooks, lacks generated config baseline updates, has no real behavior proof, and is currently conflicting with main.

## What This Changes

This PR proposes agent-default and per-agent internal hook policy overlays, including config schema/types, runtime dispatch gating, tests, and hooks/config documentation.

## Best Possible Solution

Rebase onto current gateway startup code and land a scoped policy resolver that applies agent overlays only for explicit agent ids or canonical agent session keys, with updated config baseline/docs/tests and redacted real behavior proof.

## Reproduction Assessment

Yes for the blocking PR finding: source inspection of the PR head shows config-only lifecycle events can resolve to the default agent and merge agents.defaults.hooks, so an events allowlist can filter gateway:startup. I did not live-run it because the checkout is read-only, but the call path is clear from the PR source and current main startup code.

## Solution Assessment

No. The feature direction may be useful, but this implementation should first gate agent overlays on explicit or parseable agent scope, keep gateway/system lifecycle hooks under global policy unless intentionally changed, and refresh generated config baseline proof.
