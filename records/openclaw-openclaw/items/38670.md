## Summary

Keep open. Current main still lacks agent-scoped internal hook policy overrides, but this PR has a source-reproducible blocker where agent default policy can filter non-agent lifecycle hooks, and the contributor has not provided real behavior proof.

## What This Changes

The PR adds optional internal hook policy config under `hooks.internal`, `agents.defaults.hooks`, and `agents.list[].hooks`, threads config through hook dispatch call sites, and updates docs and tests.

## Best Possible Solution

Revise the PR so agent overlays apply only after explicit or parseable agent scope, regenerate the config docs baseline, clarify exact registration-key matching, and add redacted real-run proof before maintainer policy/security review.

## Reproduction Assessment

Yes. By source inspection, PR head dispatches `gateway:startup` with config and no `agentId`, then can infer a configured default agent and apply `agents.defaults.hooks` to a non-agent lifecycle event.

## Solution Assessment

No. The feature direction is plausible, but the current patch needs agent-scope gating, generated config baseline alignment, clearer event semantics, and real behavior proof before it is a maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate agent policy on real agent scope:** `src/hooks/internal-hooks.ts:307-308`
  - body: `resolveEffectiveInternalHookPolicy` reads `agents.defaults.hooks` before dispatch confirms that the event is agent-scoped. Because `triggerInternalHook` can infer the configured default agent from non-agent keys such as `gateway:startup`, an allowlist like `["command:new"]` suppresses startup handlers; apply agent overlays only for explicit or parseable agent-scoped events.

  - confidence: 0.78
