## Summary

Keep open: current main still has global-only internal hook dispatch, while this PR introduces a useful but not-yet-safe policy surface with a blocking agent-scope/security issue and missing release/schema follow-through.

## What This Changes

The PR adds `hooks.internal`, `agents.defaults.hooks`, and `agents.list[].hooks` policy overlays for internal hook dispatch, plus docs and tests for the new config shape.

## Best Possible Solution

Revise and rebase the PR so agent overlays apply only to confirmed agent-scoped dispatch, gateway/system lifecycle hooks remain governed by global policy, and generated schema metadata, docs, changelog, and focused regressions land together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a non-agent event such as `gateway:startup` falls back to the default agent, then `agents.defaults.hooks.events: ["command:new"]` can filter out the lifecycle registration.

## Solution Assessment

No. The feature direction may be valid, but this implementation is not the narrowest maintainable solution until it gates agent overlays on a real agent scope and preserves global lifecycle hook behavior.

## Security Review

Status: needs_attention

Summary: Needs attention: the patch changes hook execution policy and can let agent defaults suppress non-agent lifecycle hooks.

Concerns:

- **[medium] Agent defaults can suppress lifecycle hooks:** `src/hooks/internal-hooks.ts:305`
  - body: Because `agents.defaults.hooks` is merged even when dispatch is not tied to a real agent scope, a default event allowlist can prevent `gateway:startup` or similar lifecycle handlers from running. Deployments relying on those hooks for audit, DLP, or compliance could unintentionally bypass controls.
  - confidence: 0.86
