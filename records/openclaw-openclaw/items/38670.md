## Summary

Keep open. Current main does not implement agent-scoped internal hook policy overlays, and this PR still has a correctness/security blocker where agent defaults can filter gateway lifecycle hooks; it also needs generated schema metadata and a changelog entry before merge.

## What This Changes

The PR adds `agents.defaults.hooks` and `agents.list[].hooks` policy overlays for internal hook dispatch, threads config/agent metadata through hook trigger sites, updates schemas/docs, and adds regression tests.

## Best Possible Solution

Revise and rebase the PR so agent overlays apply only to confirmed agent-scoped events, gateway/system lifecycle hooks remain governed by global policy, and schema metadata, docs, changelog, and focused regressions land together.

## Reproduction Assessment

Yes. The PR source shows `gateway:startup` passes config without an agent id, the fallback can derive the default agent from a non-agent session key, and `agents.defaults.hooks.events` can then filter out the lifecycle registration.

## Solution Assessment

No. The requested direction may be useful, but this implementation is not the narrowest maintainable path until agent overlays are gated on explicit or parseable agent scope and non-agent lifecycle hooks remain on global policy.

## Review Findings

## Security Review

Status: needs_attention

Summary: The patch changes hook execution policy and can let agent defaults suppress non-agent lifecycle hooks.

Concerns:

- **[medium] Agent defaults can suppress lifecycle hooks:** `src/hooks/internal-hooks.ts:307`
  - body: Because `agents.defaults.hooks` is merged even when dispatch is not tied to a real agent scope, a default event allowlist can prevent `gateway:startup` or similar lifecycle handlers from running. Deployments relying on those hooks for audit, DLP, or compliance could unintentionally bypass controls.
  - confidence: 0.86
