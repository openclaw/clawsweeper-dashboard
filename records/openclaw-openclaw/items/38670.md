## Summary

Keep open: current main still lacks agent-scoped internal hook policy overrides, but the PR head has a concrete P2 policy/security defect and no contributor real behavior proof, so it needs revision and maintainer review rather than cleanup closure.

## What This Changes

The PR adds optional hook policy overrides under `hooks.internal`, `agents.defaults.hooks`, and `agents.list[].hooks`, threads config into internal hook dispatch, and updates docs and tests for the new config surface.

## Best Possible Solution

Revise the PR so agent overlays apply only after explicit or parseable agent scope, align generated config baseline files, clarify event-key matching semantics, and add redacted real-run proof before maintainer policy/security review.

## Reproduction Assessment

Yes for the PR defect by source inspection: `gateway:startup` is dispatched with config and no `agentId`, while the resolver applies agent defaults and can infer the configured default agent from a non-agent session key. I did not run tests because this was a read-only review.

## Solution Assessment

No. The feature direction is plausible, but this patch needs agent-scope gating, generated config baseline alignment, clearer event semantics, and real behavior proof before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate agent overlays on real agent scope:** `src/hooks/internal-hooks.ts:307-308`
  - body: `resolveEffectiveInternalHookPolicy` reads `agents.defaults.hooks` before dispatch proves the event is agent-scoped. Because `triggerInternalHook` can infer a default agent for non-agent keys such as `gateway:startup`, an allowlist like `["command:new"]` can suppress startup lifecycle handlers; apply agent overlays only for explicit or parseable agent-scoped events.
