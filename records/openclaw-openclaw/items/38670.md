## What This Changes

The branch adds agent-scoped internal hook policy config, resolver/filtering logic, config schemas/docs/tests, and config/agent metadata threading through hook dispatch call sites.

## Best Possible Solution

Rework policy resolution so agent overlays apply only with confirmed agent scope, refresh generated config baseline artifacts, clarify event-key semantics, and require redacted real-run proof before merge.

## Reproduction Assessment

Yes for the PR-introduced blocker: source inspection shows `gateway:startup` passes config without an agent id, the resolver infers the default agent from that non-agent key, and `agents.defaults.hooks` can then suppress lifecycle handlers. I did not run tests because this review is read-only.

## Solution Assessment

No. The feature direction is plausible, but this patch should first gate agent overlays on confirmed agent scope, align generated config artifacts, document the event-key matching contract, and include real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate agent defaults on real agent scope:** `src/hooks/internal-hooks.ts:307-308`
  - body: `resolveEffectiveInternalHookPolicy` always reads `agents.defaults.hooks` before a real agent scope is confirmed.   - confidence: 0.91
