## Summary

Keep open. The PR still adds a meaningful new hook-policy config surface that current main does not provide, but PR-head source still has a blocking scope bug and lacks contributor real-run proof for the new behavior.

## What This Changes

The PR adds agent-default and per-agent internal hook policy fields, applies them during internal hook dispatch, threads config context through hook call sites, and documents/tests the new config surface.

## Best Possible Solution

Rework the branch so agent hook overlays apply only after confirmed agent scope, preserve non-agent lifecycle hooks under global policy, document exact event-key semantics, refresh generated config artifacts, and include focused tests plus redacted real-run proof.

## Reproduction Assessment

Yes for the blocking PR finding. Source inspection of PR head shows gateway:startup passes config without agentId, policy resolution applies agents.defaults.hooks, and exact event filtering can suppress the gateway:startup registration.

## Solution Assessment

No. The feature direction may be useful, but this branch should not merge until agent overlays are gated on confirmed agent scope, generated config artifacts are refreshed, event-key semantics are documented, and real behavior proof is provided.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate agent defaults on real agent scope:** `src/hooks/internal-hooks.ts:307-308`
  - body: `resolveEffectiveInternalHookPolicy` merges `agents.defaults.hooks` before any real agent scope is confirmed. Because `triggerInternalHook` can derive the default agent from non-agent keys such as `gateway:startup`, an allowlist like `agents.defaults.hooks.events: ["command:new"]` suppresses gateway startup handlers. Apply defaults/per-agent overlays only for explicit or parseable agent-scoped events; otherwise use only global `hooks.internal`.
  - confidence: 0.91
- **[P3] Regenerate the config baseline hash:** `src/config/zod-schema.hooks.ts:123`
  - body: This schema change adds a new config surface, but the PR head does not contain `docs/.generated/config-baseline.sha256`. Refresh the generated config baseline artifacts with the schema/docs update so the config docs check does not drift.
  - confidence: 0.78
