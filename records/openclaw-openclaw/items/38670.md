## Summary

Keep open. Current main still has global-only internal hook dispatch and does not accept or document `agents.defaults.hooks` / `agents.list[].hooks`, so this PR is not implemented on main. The PR is a coherent implementation candidate, but it is not merge-ready because the proposed resolver can apply agent hook overlays to non-agent lifecycle events such as `gateway:startup`, which needs maintainer product/security review and a targeted revision.

## What This Changes

The PR adds agent-scoped internal hook policy config, threads config and agent metadata into internal hook dispatch call sites, gates dispatch by effective policy, records hook names in registrations, and updates hook/config docs and tests.

## Best Possible Solution

Keep the PR open as the implementation candidate, but require a revision that applies agent overlays only to confirmed agent-scoped events, keeps gateway/system lifecycle hooks governed by global hook policy, documents exact event-key matching or implements an explicit wildcard contract, and adds focused regression tests for precedence, named-hook gating, non-agent lifecycle events, and backward compatibility.

## Reproduction Assessment

Yes. On PR head, `gateway:startup` passes config without an agent id, `getAgentIdForEvent` can fall back from `gateway:startup` to the default agent, and `resolveEffectiveInternalHookPolicy` always merges `agents.defaults.hooks`; a defaults allowlist like `events: ["command:new"]` would skip a registered `gateway:startup` handler.

## Solution Assessment

No. The proposed direction may be acceptable, but this implementation is not the best fix as-is because it applies agent-scoped policy outside a real agent scope; the safer path is to resolve policy only when an explicit agent id or parseable agent session exists and leave non-agent lifecycle hooks on global policy.

Status: needs_attention
