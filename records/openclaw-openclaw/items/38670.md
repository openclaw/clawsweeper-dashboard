## Summary

Keep this PR open: current main still does not implement agent-scoped internal hook policy overrides, but the proposed patch has a real policy bug that can let agent defaults suppress non-agent lifecycle hooks before merge.

## What This Changes

The PR adds `hooks.internal`, `agents.defaults.hooks`, and `agents.list[].hooks` policy overlays for internal hook dispatch, plus config schema, docs, and tests.

## Best Possible Solution

Revise and rebase the PR so agent overlays apply only to confirmed agent-scoped events, non-agent lifecycle hooks stay governed by global policy, and schema metadata, docs, changelog, and focused regressions land together.

## Reproduction Assessment

Yes. The PR diff gives a source-level reproduction: `gateway:startup` has a non-agent session key, the new fallback resolves it to a default agent, and `agents.defaults.hooks.events` can then filter the lifecycle hook.

## Solution Assessment

No. The feature direction may be useful, but this implementation is not the best fix because agent defaults are merged before the event is proven to be agent-scoped; the safer path is explicit agent-scope resolution with gateway/system events left on global policy.

## Review Findings

## Security Review

Status: needs_attention

Summary: The diff changes hook execution policy and currently allows agent defaults to suppress non-agent lifecycle hooks.

Concerns:

- **[medium] Agent defaults can suppress lifecycle hooks:** `src/hooks/internal-hooks.ts:305`
  - body: Because `agents.defaults.hooks` is merged even when dispatch is not tied to a real agent scope, a default event allowlist can prevent `gateway:startup` or similar lifecycle handlers from running. Deployments relying on those hooks for audit, DLP, or compliance could unintentionally bypass controls.
