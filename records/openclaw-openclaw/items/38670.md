## Summary

Keep open: current main does not implement agent-scoped internal hook policy overlays, but this PR should not merge as-is because the branch can apply agent defaults to non-agent lifecycle hooks and is missing generated schema/changelog follow-through.

## What This Changes

The PR adds optional internal hook policy overlays under `hooks.internal`, `agents.defaults.hooks`, and `agents.list[].hooks`, threads config and agent context into hook dispatch, and updates docs/tests for the new config surface.

## Best Possible Solution

Revise and rebase the PR so agent overlays apply only to confirmed agent-scoped dispatches, lifecycle hooks remain governed by global policy, and schema metadata, docs, changelog, and focused regressions land together.

## Reproduction Assessment

Yes. Source inspection of the PR diff shows `gateway:startup` can pass config without an explicit agent id, while the fallback agent resolver maps non-agent session keys to `main`, letting `agents.defaults.hooks.events` suppress lifecycle registrations.

## Solution Assessment

No. The requested direction may be useful, but this implementation is not the narrowest maintainable solution until agent policy overlays are gated on explicit or parseable agent scope and generated schema/changelog outputs are updated.

## Security Review

Status: needs_attention

Summary: The PR changes hook execution policy and currently allows agent defaults to suppress non-agent lifecycle hooks.

Concerns:

- **[medium] Agent defaults can suppress lifecycle hooks:** `src/hooks/internal-hooks.ts:305`
  - body: Because `agents.defaults.hooks` is merged before a real agent scope is confirmed, a default event allowlist can prevent `gateway:startup` or similar lifecycle handlers from running. Deployments relying on those hooks for audit, DLP, or compliance could unintentionally bypass controls.
