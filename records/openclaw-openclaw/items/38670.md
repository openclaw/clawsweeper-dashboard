## Summary

Keep open: current main does not implement agent-scoped internal hook policy overrides, but this PR should not merge yet because PR-head policy resolution can apply agent defaults to non-agent lifecycle hooks, the generated config baseline is not updated, and the external contributor has not provided real behavior proof.

## What This Changes

The PR adds agent-scoped internal hook policy fields in config, schema, docs, tests, and runtime dispatch across internal hook call sites.

## Best Possible Solution

Land a rebased implementation that applies agent hook overlays only after confirmed agent scope, preserves non-agent lifecycle hooks under global policy, refreshes generated config docs artifacts, documents event-key semantics, and includes focused tests plus redacted real-run proof.

## Reproduction Assessment

Yes for the blocking PR finding: source inspection of PR head shows `gateway:startup` dispatch passes config without `agentId`, then policy resolution can apply `agents.defaults.hooks` and filter the `gateway:startup` registration. I did not live-run it because this is a read-only review.

## Solution Assessment

No. The feature direction may be useful, but this branch should not merge until agent overlays are gated on confirmed agent scope, generated config artifacts are refreshed, and real behavior proof is provided.

## Review Findings

- **[P2] Gate agent defaults on real agent scope:** `src/hooks/internal-hooks.ts:307-308`
