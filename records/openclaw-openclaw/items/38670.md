## Summary

Keep open. Current main does not implement agent-scoped internal hook policy overrides, so the PR is still the relevant implementation candidate, but the proposed patch is not merge-ready because agent defaults can suppress non-agent lifecycle hooks and the user-facing config change still needs release metadata follow-through.

## What This Changes

The PR adds agent-default and per-agent internal hook policy config, records hook names in internal hook registrations, threads config and agent metadata into hook dispatch call sites, gates dispatch by effective policy, and updates hook/config docs and tests.

## Best Possible Solution

Keep the PR open as the implementation path, but revise it so agent overlays apply only to confirmed agent-scoped events, gateway/system lifecycle hooks remain governed by global internal hook policy, current-main hook runtime refactors are preserved during rebase, and the config docs, generated schema metadata, changelog, and focused regressions all land together.

## Reproduction Assessment

Yes. On PR head, a config with agents.defaults.hooks.events set to only command:new and a registered gateway:startup hook follows the gateway startup call path with config but no agent id, resolves a default agent from the non-agent session key, and skips the lifecycle handler.

## Solution Assessment

No. The proposed direction can be valid, but this implementation applies agent-scoped policy outside a real agent scope; the narrower maintainable fix is to resolve agent overlays only from an explicit agent id or parseable agent session and leave non-agent lifecycle hooks on global policy.

## Security Review

Status: needs_attention

Summary: The patch changes an in-process hook execution policy path and can unintentionally suppress gateway lifecycle hooks that operators may use for audit or compliance.

Concerns:

- **[medium] Agent defaults can suppress lifecycle hooks:** `src/hooks/internal-hooks.ts:307`
  - body: Because agents.defaults.hooks is merged whenever config is present, non-agent lifecycle events such as gateway:startup can be filtered by an agent default allowlist. Deployments using internal hooks for startup audit, DLP, or compliance checks could bypass those controls unintentionally.
