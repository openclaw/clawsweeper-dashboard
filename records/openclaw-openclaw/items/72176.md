## Summary

Keep open: current main still lacks a generic duplicate-delivery guard for missing or regenerated inbound ids, and user transcript/SSE updates still lack stable user-turn idempotency that would prevent same-timestamp duplicate history entries.

## What This Changes

The issue reports intermittent duplicate user-message and assistant-message transcript delivery after upgrading to 2026.4.24 across WebChat and WeChat.

## Best Possible Solution

Add a generic Gateway/auto-reply idempotency path for missing or regenerated inbound ids, propagate stable user-turn metadata into transcript/session-history updates, and preserve legitimate repeated messages with focused regressions.

## Reproduction Assessment

Yes. The issue provides concrete `sessions_history` duplicate pairs, and current source still shows missing-id inbound dedupe bypass plus inline transcript updates that receive fresh sequence numbers without stable user-turn metadata.

## Solution Assessment

No current-main solution fully solves it. #72605 is useful for same-key WebChat races, but this broader report needs a generic fallback and transcript/session-history idempotency coverage.

## Repair Work Prompt

Fix the duplicate delivery regression described in https://github.com/openclaw/openclaw/issues/72176 on current main. Implement a generic Gateway/auto-reply idempotency path that covers inbound messages with missing or regenerated provider ids without suppressing legitimate repeated same-content messages. Start in `src/auto-reply/reply/inbound-dedupe.ts` and `src/auto-reply/reply/dispatch-from-config.ts`; use `MessageSid` when present, and add a conservative fallback scoped by provider/channel route, session/agent scope, peer/thread/account, normalized body, and timestamp or equivalent stable inbound metadata when `MessageSid` is missing. Also propagate stable user-turn metadata through `src/gateway/server-methods/chat.ts` so user transcript/session-history updates carry a deterministic `messageId` and/or `idempotencyKey`, and teach `src/gateway/session-history-state.ts` / `src/gateway/sessions-history-http.ts` to avoid appending duplicate inline events for the same user turn. Preserve existing same-key `chat.send` behavior and do not broaden #72605 unless needed for compatibility. Add focused regressions for missing-id inbound dedupe, duplicate regenerated `chat.send`/transcript updates, SSE duplicate suppression, and legitimate repeated messages. Add a `CHANGELOG.md` fix entry because this is user-facing. Do not edit unrelated block-streaming behavior from #70921.
