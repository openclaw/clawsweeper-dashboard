## Summary

Keep open: current main still has the missing-stable-message-id inbound dedupe gap, recent comments report the same duplicate-delivery symptom in v2026.5.2 and v2026.5.12, and the focused related fixes are still open rather than merged.

## What This Changes

The issue reports a 2026.4.24 regression where duplicate inbound deliveries create duplicate transcript entries and duplicate assistant replies across webchat, WeChat, and later Telegram reports.

## Best Possible Solution

Land or replace the focused shared inbound retry dedupe fix, keep the WebChat race work on its narrower gateway path, and close this issue only after equivalent code reaches main and release behavior is verified.

## Reproduction Assessment

Yes, source-level: a `MsgContext` with provider/peer/timestamp/body but no `MessageSid` produces no inbound dedupe key, so `claimInboundDedupe` is invalid and dispatch continues. Live comments also report the same symptom in v2026.5.2 and v2026.5.12.

## Solution Assessment

Yes, the best path is a narrow shared inbound dedupe fallback for idless retries, with the WebChat duplicate-send race handled separately in the gateway path. Closing before the related fixes land would be premature.
