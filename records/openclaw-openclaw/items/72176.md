## Summary

Keep open: current main still has the missing-stable-message-id inbound dedupe gap, and recent non-author comments report the same duplicate transcript delivery in v2026.5.2 and v2026.5.12. The focused fix remains an open PR, so closing this canonical regression report before a merge would be premature.

## What This Changes

The issue reports intermittent duplicate user and assistant message delivery after upgrading to v2026.4.24, with identical timestamps but different sequence numbers across multiple channels.

## Best Possible Solution

Land or replace the focused shared inbound retry dedupe fix, keep the WebChat race work on its narrower gateway path, and close this issue only after equivalent code reaches main or a release.

## Reproduction Assessment

Yes, by source inspection plus issue evidence. Current main still treats missing-MessageSid inbound messages as invalid dedupe claims that continue into dispatch, and commenters report the same duplicate-delivery symptom in v2026.5.2 and v2026.5.12.

## Solution Assessment

Yes, the best path is the narrow shared inbound dedupe fallback for idless retries, with WebChat duplicate-send races handled separately in the gateway path. Closing before the related fix lands would be premature.
