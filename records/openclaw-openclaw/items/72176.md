## Summary

Keep open. Current main still has the duplicate-delivery gaps described in the report, and the focused repair PR that linked this issue was closed unmerged, so there is no merged or shipped fix to close against.

## What This Changes

The issue reports an intermittent 2026.4.24 regression where identical inbound user messages reach session history twice across WebChat and WeChat, producing duplicate assistant replies.

## Best Possible Solution

End state should be a generic Gateway/auto-reply idempotency repair that catches retry duplicates when provider or client ids are missing or regenerated, propagates stable duplicate metadata into transcript/session-history updates, and preserves legitimate repeated messages through tight scoping, TTLs, and regression tests.

## Reproduction Assessment

Yes. The issue provides concrete sessions_history duplicate pairs, and current main has a source-level reproduction path: missing MessageSid produces an invalid inbound dedupe claim, regenerated chat.send idempotency keys start new runs, and inline history updates get fresh seq values without stable duplicate metadata.

## Solution Assessment

Yes, a focused replacement repair using the useful ideas from #72290 is the narrowest maintainable path. The fallback dedupe key must be scoped carefully so ordinary repeated user messages are not dropped.

## Repair Work Prompt

Fix the duplicate inbound message delivery regression tracked by https://github.com/openclaw/openclaw/issues/72176. Current main still skips inbound dedupe when MessageSid is missing, scopes chat.send dedupe only to the caller idempotencyKey, and emits inline user transcript/session-history updates without stable duplicate metadata. Use #72290 as useful source context but implement a conservative main-branch repair: add a tightly scoped fallback inbound dedupe key for retry duplicates without stable provider ids, propagate stable idempotency/message metadata into user transcript updates where appropriate, and ensure session-history inline updates do not append duplicates with fresh seq values for the same logical message. Do not broaden suppression enough to drop intentional repeated user messages; scope by channel/account/peer/session/thread and use a short retry window. Add focused regression tests and a user-facing CHANGELOG.md fix entry without inventing bot/maintainer thanks.
