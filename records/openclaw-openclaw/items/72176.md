## Summary

Keep open: current main still only dedupes stable inbound IDs and same-key WebChat retries, while this report and the latest comment describe regenerated or missing IDs across channels. The prior broad fix PR was closed unmerged and the open related PR is narrower.

## What This Changes

The issue reports intermittent duplicate user and assistant transcript delivery after 2026.4.24, with same timestamps but different sequence numbers across WebChat and WeChat.

## Best Possible Solution

Add generic Gateway/auto-reply idempotency for missing or regenerated inbound IDs and carry stable duplicate metadata into user transcript/session-history updates while preserving intentional repeated messages.

## Reproduction Assessment

Yes, by source inspection: missing `MessageSid` produces an invalid inbound claim that is not skipped, and inline transcript updates still allocate a fresh sequence number for each duplicate update. I did not run a live WebChat or WeChat gateway reproduction.

## Solution Assessment

No current-main solution fully solves the report. #72605 is useful for same-key WebChat attachment races, but the broader all-channel path still needs missing/regenerated-ID fallback plus user transcript idempotency.
