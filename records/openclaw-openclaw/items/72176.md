## Summary

Keep open: current main still has source-level duplicate-delivery gaps for missing/regenerated inbound ids and user transcript updates, while the related PR context only covers part of the WebChat same-key race.

## What This Changes

The issue reports intermittent duplicate user and assistant transcript entries after 2026.4.24 across WebChat and WeChat, with identical timestamps but new sequence numbers.

## Best Possible Solution

Implement one generic Gateway/auto-reply idempotency path that covers missing or regenerated inbound ids, propagates stable duplicate metadata into transcript/session-history updates, and preserves legitimate repeated messages with focused regressions.

## Reproduction Assessment

Yes. The report provides concrete sessions_history duplicate pairs, and current main has a source-level path where missing MessageSid yields an invalid dedupe claim, regenerated chat.send keys bypass key-only dedupe, and inline transcript updates get fresh sequence numbers.

## Solution Assessment

No current-main solution fully addresses the issue. #72605 is useful for same-id WebChat races, but this broader report needs a generic fallback for missing/regenerated ids plus user transcript idempotency coverage.
