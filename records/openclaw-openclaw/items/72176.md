## Summary

Keep open: current main still has a source-backed all-channel duplicate-delivery gap, and the later WebChat fixes only cover same-idempotency-key or UI coalescing cases rather than missing or regenerated inbound IDs plus user transcript idempotency.

## What This Changes

Prevent intermittent duplicate user messages and duplicate assistant replies after the 2026.4.24 regression across WebChat, WeChat, and other inbound channels.

## Best Possible Solution

Add a generic Gateway/auto-reply idempotency path for missing or regenerated inbound IDs, carry stable duplicate metadata into user transcript/session-history updates, and preserve legitimate repeated messages with narrow tests.

Yes, by source inspection: a missing `MessageSid` produces an `invalid` inbound dedupe claim that is not skipped, and duplicate user transcript updates without stable ids receive fresh sequence numbers. I did not run a live WebChat or WeChat reproduction in this read-only pass.

## Solution Assessment

No current-main solution fully solves this issue. The best path is a narrow generic dedupe/transcript-idempotency fix; the existing WebChat coalescing and same-key attachment-race coverage are useful but not sufficient for all-channel missing or regenerated inbound IDs.
