## Summary

Keep open: current main still lets missing-stable-message-id inbound retries bypass the inbound dedupe claim, and the active fix path is still an open PR rather than merged code. The issue has concrete session-history evidence plus a later report that the behavior still reproduced in v2026.5.2.

## What This Changes

The issue reports intermittent duplicate inbound user-message delivery and duplicate assistant replies after upgrading to 2026.4.24 across WebChat, WeChat, and other channels.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/72314 for idless inbound retries, coordinate any separate WebChat attachment race in https://github.com/openclaw/openclaw/pull/72605, then close after the fix reaches main or a release.

## Reproduction Assessment

Yes, by source inspection rather than a live channel run. Current main returns an invalid inbound dedupe claim when MessageSid is absent, and dispatch only suppresses duplicate or inflight claims.

## Solution Assessment

Yes, the active PR direction is the narrow maintainable fix for the central idless retry gap. The current issue should stay open until that PR or an equivalent fix lands, with the WebChat attachment race handled as a separate related slice.
