## Summary

Keep open. Current main still has the duplicate-delivery gap described in the report: inbound dedupe requires a stable `MessageSid`, `chat.send` request dedupe is scoped to the supplied idempotency key, and inline session-history user updates do not carry a stable message id. The related source PR #72290 is closed unmerged, so there is no merged fix to close against.

## What This Changes

The issue asks OpenClaw to stop intermittent duplicate user and assistant transcript delivery after 2026.4.24 when the same message is recorded with the same timestamp but different sequence numbers across WebChat and WeChat.

## Best Possible Solution

End state should be a generic Gateway/auto-reply idempotency repair that catches retry duplicates even when provider or client ids are regenerated, propagates stable duplicate metadata into session-history updates, and preserves legitimate repeated messages through tight scoping and regression tests.

## Reproduction Assessment

Yes. The issue provides concrete sessions_history duplicate pairs, and current main has a source-level path where missing `MessageSid` or regenerated `chat.send` idempotency keys bypass existing guards.

## Solution Assessment

Yes. A focused replacement for the useful parts of #72290 is the narrow maintainable path; closing under older duplicate-delivery fixes would leave this 2026.4.24 retry/transcript gap unaddressed.
