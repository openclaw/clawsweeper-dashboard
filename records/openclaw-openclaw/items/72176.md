## Summary

Keep open: current main still has unguarded duplicate-delivery paths for missing provider message ids and regenerated `chat.send` idempotency keys, and the newly related open PR appears to cover only the same-key WebChat race rather than the reported all-channel regression.

## What This Changes

Prevent intermittent duplicate inbound user messages and duplicate assistant replies after 2026.4.24 across WebChat, WeChat, and other channels.

## Best Possible Solution

End state should be one generic Gateway/auto-reply idempotency path that covers inbound retries with missing or regenerated ids, propagates stable duplicate metadata into transcript/session-history updates, and preserves legitimate repeated messages with tight scoping and regression tests.

## Reproduction Assessment

Yes. The issue includes concrete `sessions_history` duplicate pairs, and current main has a source-level reproduction path through missing `MessageSid`, regenerated `chat.send` keys, and inline session-history updates that assign fresh sequence numbers.

## Solution Assessment

No for a WebChat-only same-key repair as the complete solution. #72605 is useful overlapping work, but this issue needs a generic fallback for missing/regenerated ids plus transcript metadata coverage across channel paths.
