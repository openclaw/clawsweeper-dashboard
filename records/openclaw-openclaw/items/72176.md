## Summary

Keep open: current main has merged a narrow active WebChat text-send dedupe, but the broader cross-channel report still has source-level gaps for missing or regenerated inbound message IDs and user transcript/session-history idempotency. Open PRs cover important slices, but no merged current-main fix fully closes this issue.

## What This Changes

Prevent duplicate logical user messages in WebChat, WeChat, and other channels from producing duplicate transcript entries and assistant replies after the 2026.4.24 regression.

## Best Possible Solution

Coordinate the active related PRs, land an idless/regenerated-id inbound dedupe fix such as https://github.com/openclaw/openclaw/pull/72314, resolve the remaining WebChat attachment race if still relevant, and add user transcript/session-history idempotency coverage before closing this issue.

## Reproduction Assessment

Yes for the core failure class by source inspection, though not by a live channel run here. Missing MessageSid currently produces an invalid dedupe claim that dispatch does not skip, and user transcript updates can still receive fresh sequence numbers without a stable message id.

## Solution Assessment

No current-main solution fully solves the issue. The merged WebChat active-send dedupe is useful but intentionally narrow, while the broader idless inbound and transcript idempotency paths still need review and merge through the related fix PRs or an equivalent focused patch.
