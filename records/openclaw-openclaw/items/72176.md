## Summary

Keep open. Current main still only dedupes inbound delivery when a stable provider message id or exact `chat.send` idempotency key is reused; the reported retry shape with regenerated ids can still append duplicate user transcript updates and trigger duplicate assistant replies. The related fix PR #72290 is closed unmerged, while the newer Control UI coalescing only covers duplicate in-flight UI submits before a new run id is generated.

## What This Changes

The issue requests a fix for intermittent duplicate inbound user messages and duplicate assistant replies after upgrading from 2026.4.23 to 2026.4.24 across WebChat and WeChat.

## Best Possible Solution

End state should be a generic Gateway/auto-reply idempotency repair that catches retry duplicates when provider or client ids are missing/regenerated, propagates stable duplicate metadata into transcript updates, and adds regression tests that preserve legitimate repeated messages through narrow scoping and TTLs.

## Reproduction Assessment

Yes. The report gives concrete sessions_history duplicate pairs with identical timestamps and sequential seq numbers, and current main has a source-level reproduction path where missing `MessageSid` or regenerated `chat.send` idempotency keys bypass existing guards.

## Solution Assessment

Yes. The narrow maintainable direction is a generic idempotency fix in the Gateway/auto-reply layer, using #72290 as context but not limiting the repair to Control UI or WeChat-specific code.
