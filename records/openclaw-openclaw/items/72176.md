## Summary

Keep open. Current main still leaves the reported duplicate-delivery path unguarded when inbound retries lack a stable provider message id or when `chat.send` retries use a regenerated idempotency key, and the linked repair PR is closed unmerged.

## What This Changes

The issue requests fixing a 2026.4.24 regression where duplicate inbound user messages with identical timestamps but different seq numbers cause duplicate assistant replies across WebChat and WeChat.

## Best Possible Solution

End state should be a generic Gateway/auto-reply idempotency repair that catches retry duplicates when provider or client ids are missing or regenerated, propagates stable duplicate metadata into transcript/session-history updates, and preserves legitimate repeated messages through tight scoping, TTLs, and regression tests.

## Reproduction Assessment

Yes. The issue provides concrete `sessions_history` duplicate pairs, and current main has a source-level reproduction path: missing `MessageSid` produces an invalid inbound dedupe claim, regenerated `chat.send` keys create new runs, and inline history updates assign fresh sequence numbers.

## Solution Assessment

Yes. A focused generic idempotency repair, using #72290 as source context but tightening fallback scope, is the narrowest maintainable fix; a WebChat-only or WeChat-plugin-only change would not cover the reported all-channel layer.
