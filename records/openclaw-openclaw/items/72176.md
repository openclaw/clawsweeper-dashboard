## Summary

Keep open: current main still has source-level idempotency gaps for missing or regenerated inbound message IDs and for eager user transcript updates, and the latest issue comment says the problem still appears in v2026.5.2.

## What This Changes

The issue requests restoring exactly-once user-message delivery after 2026.4.24 so duplicate inbound turns do not create duplicate assistant replies across WebChat, WeChat, and other channels.

## Best Possible Solution

Land a generic Gateway/auto-reply idempotency fix that covers missing or regenerated inbound IDs and carries stable user-turn metadata through transcript and session-history updates.

## Reproduction Assessment

Yes, by source inspection rather than a live run: missing or regenerated `MessageSid` bypasses inbound dedupe, and duplicate inline user transcript updates receive fresh sequence numbers when no stable `messageId` is supplied.

## Solution Assessment

No current-main solution fully solves the issue. #72605 is useful for a narrower same-key WebChat race, but the broader report needs generic inbound fallback dedupe plus user transcript/session-history idempotency.
