## Summary

Keep open: current main still lacks the idless inbound dedupe fallback, live comments report the duplicate-delivery symptom through v2026.5.12, and the focused fix paths are still open PRs rather than merged code.

## What This Changes

The issue reports intermittent duplicate user and assistant message delivery after upgrading to OpenClaw 2026.4.24 across WebChat, WeChat, and later Telegram confirmations.

## Best Possible Solution

Ship the shared inbound retry dedupe fallback with regression coverage, preserve current gateway media handling while fixing the WebChat idempotency race, and close only after equivalent code reaches main or a release.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: missing MessageSid returns no inbound dedupe key, dispatch treats that as invalid rather than skipped, and the current WebChat attachment race test shows a same-id duplicate taking the started path.

## Solution Assessment

Yes, the narrow maintainable direction is a shared idless inbound dedupe fallback plus a separately ported WebChat preclaim around the current media-offload flow; closing before those land would be premature.
