## Summary

Keep open: current main still only dedupes inbound messages with stable provider message IDs or same `chat.send` idempotency keys, leaving the reported missing/regenerated-ID all-channel path and user transcript update path uncovered.

## What This Changes

The issue requests that OpenClaw stop intermittently delivering the same user message twice after 2026.4.24 across WebChat, WeChat, and other channels.

## Best Possible Solution

Add generic Gateway/auto-reply idempotency that covers missing or regenerated inbound IDs and carries stable duplicate metadata into user transcript/session-history updates without changing channel-specific policy.

## Reproduction Assessment

Yes, by source inspection rather than a live run: missing `MessageSid` produces an invalid inbound dedupe claim that is not skipped, and duplicate inline user transcript updates receive fresh sequence numbers when no stable id is passed.

## Solution Assessment

No current-main solution fully solves this issue. Existing same-key WebChat handling is useful but narrower than the all-channel missing/regenerated-ID and transcript-idempotency path reported here.
