## Repair Work Prompt

Fix the duplicate message delivery regression tracked by https://github.com/openclaw/openclaw/issues/72176. Current main still dedupes inbound dispatch only when MessageSid is present and dedupes chat.send only by the caller-provided idempotencyKey, so retries with missing/regenerated ids can append duplicate user transcript entries and trigger duplicate assistant replies. Use PR #72290 as source context, but implement the repair conservatively: add a tightly scoped fallback idempotency key for inbound retries when provider message ids are absent, propagate stable idempotency metadata into chat.send user transcript updates, and avoid suppressing legitimate repeated same-text messages by including route/session/account/thread/timestamp or a short TTL as appropriate. Keep the fix generic in Gateway/auto-reply code, not WebChat-only or WeChat-only. Add focused regression tests for missing MessageSid retry behavior, regenerated chat.send idempotency retry behavior, and session-history inline metadata. Add a CHANGELOG.md fix entry because this is user-facing. Do not change unrelated channel behavior, provider defaults, plugin APIs, or broad transcript repair policy.
