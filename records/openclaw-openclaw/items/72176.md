## Summary

Keep open. Current main still has the all-channel duplicate-delivery gap described in the report: inbound dedupe returns invalid without a stable provider message id, `chat.send` dedupe is scoped to the caller-provided idempotency key, and inline session-history user updates still do not carry stable duplicate metadata. The related fix PR #72290 is closed unmerged, and later WebChat-specific coalescing fixes do not cover WeChat or generic retry paths.

## What This Changes

The issue requests fixing a 2026.4.24 regression where identical user messages can be delivered twice through WebChat or WeChat, producing duplicate user and assistant transcript entries.

## Best Possible Solution

Implement a generic Gateway/auto-reply idempotency repair that catches retry duplicates when provider or client ids are missing or regenerated, propagates stable duplicate metadata into session-history updates, preserves legitimate repeated messages through narrow scoping and TTLs, and adds regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. The issue provides concrete `sessions_history` duplicate pairs, and current main has a source-level reproduction path: missing `MessageSid` yields an invalid inbound claim, regenerated `chat.send` keys bypass request dedupe, and inline history appends fresh seq values without stable ids.

## Solution Assessment

Yes. A focused replacement repair using #72290 as source context is the narrowest maintainable path; closing this under WebChat-only UI coalescing or older duplicate-delivery fixes would leave the WeChat/all-channel retry gap unaddressed.
