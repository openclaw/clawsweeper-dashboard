## Summary

Keep open: current main still has source-level gaps for duplicate inbound delivery when the inbound message id is missing/regenerated, and a fresh report says v2026.5.2 is still affected.

## What This Changes

The issue reports intermittent duplicate user and assistant transcript entries after upgrading from 2026.4.23 to 2026.4.24 across WebChat and WeChat.

## Best Possible Solution

Add a generic Gateway/auto-reply idempotency path that handles missing or regenerated inbound ids, carries stable id metadata into user transcript/session-history updates, and preserves legitimate repeated messages with targeted regressions.

## Reproduction Assessment

Yes, by source inspection: missing MessageSid produces an invalid inbound dedupe claim that is not skipped, and duplicate user transcript events without messageId receive fresh sequence numbers. I did not perform a live intermittent channel run in this read-only review.

## Solution Assessment

No current-main solution fully solves the issue. The open WebChat race PR is useful for same-key sends, but this report still needs the generic missing/regenerated-id fallback plus stable user transcript idempotency.
