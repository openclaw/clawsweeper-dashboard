## Repair Work Prompt

Fix openclaw/openclaw#72176 on current main. The bug is intermittent duplicate user-message delivery after 2026.4.24 across WebChat/WeChat/all-channel paths. Source evidence: `buildInboundDedupeKey` returns null without `MessageSid`; `dispatchReplyFromConfig` lets invalid claims proceed; `chat.send` user transcript updates lack stable user `messageId`/`idempotencyKey`; SSE inline updates assign fresh seq values. Implement a narrow generic fix in Gateway/auto-reply: add a stable fallback dedupe key for inbound contexts with missing/regenerated provider IDs using provider/account/peer/thread/session scope plus tightly bounded content/timestamp material, propagate stable user idempotency metadata into transcript/session-history updates, and preserve legitimate repeated messages. Coordinate with #72605 so same-key WebChat attachment race coverage remains intact; use #72290 only as source context, not as a blind cherry-pick. Do not add a config option, channel-specific hardcoding, or broad product-policy changes. Add focused regression tests and a single-line `CHANGELOG.md` fix entry for user-visible behavior.
