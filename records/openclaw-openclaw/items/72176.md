## Summary

Keep open: current main still has the idless inbound dedupe gap that matches this duplicate-delivery regression, and the active narrow fix is still an open PR rather than merged behavior.

## What This Changes

Reports a 2026.4.24 regression where duplicate inbound user messages reach the same session and trigger duplicate assistant replies across WebChat and WeChat.

## Best Possible Solution

Land the idless inbound retry dedupe in https://github.com/openclaw/openclaw/pull/72314, coordinate any remaining WebChat attachment guard separately, then verify session history before closing.

## Reproduction Assessment

Yes, source-reproducible: a current-main inbound context without MessageSid returns an invalid dedupe claim, and dispatch only suppresses duplicate or inflight claims. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No current-main solution fully solves the issue yet. The narrow maintainable path is the open idless inbound fallback PR, with the already-merged WebChat text dedupe treated as only a partial mitigation.
