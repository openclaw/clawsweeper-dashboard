## Summary

Keep open. Current main still leaves the all-channel duplicate-delivery path unguarded when retries lack a stable provider message id or when `chat.send` is retried with a fresh caller idempotency key; the related repair PR is closed unmerged.

## What This Changes

The issue reports a 2026.4.24 regression where duplicate inbound user messages receive fresh transcript sequence numbers and trigger duplicate assistant replies across WebChat and WeChat.

## Best Possible Solution

Implement a generic Gateway/auto-reply idempotency repair that catches retry duplicates without stable provider ids and propagates stable duplicate metadata into transcript/session-history updates.

## Reproduction Assessment

Yes. The report provides concrete `sessions_history` duplicate pairs, and current main has a source-level reproduction path: missing `MessageSid` produces an invalid inbound dedupe claim, regenerated `chat.send` keys create separate runs, and inline history assigns fresh sequence numbers.

## Solution Assessment

Yes. A focused generic idempotency repair is the narrowest maintainable path; WebChat-only submit coalescing or a WeChat-plugin-only patch would not cover the reported all-channel Gateway layer.

## Repair Work Prompt

Fix openclaw/openclaw#72176. Current main still only dedupes inbound auto-reply on provider `MessageSid` and `chat.send` on the caller idempotency key, so retries that lack a stable provider id or regenerate `chat.send` ids can create duplicate user turns with fresh sequence numbers. Use #72290 as source context, but do not blindly copy it: implement a generic Gateway/auto-reply idempotency repair with tight route/session/peer/body/timestamp scoping and bounded TTL so intentional repeated messages remain possible. Propagate stable idempotency/message identity into user transcript updates so session-history inline SSE can reuse identity instead of assigning a fresh duplicate seq. Keep the fix core/generic, not WebChat-only or WeChat-plugin-only. Add targeted regression tests and a single-line `CHANGELOG.md` fix entry with appropriate human credit if applicable. Do not change provider defaults, channel plugin defaults, or unrelated streaming/block-reply behavior.
