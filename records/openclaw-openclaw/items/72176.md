## Summary

Keep open. Current main still only dedupes this path when a stable provider `MessageSid` or reused `chat.send` idempotency key survives retries; a missing or regenerated identifier can still pass through inbound dispatch and session-history projection as a fresh message. The related source PR #72290 directly targets this issue but is closed unmerged, and I found no open PR that currently closes it.

## What This Changes

Fix intermittent duplicate user-message and assistant-reply delivery after 2026.4.24, especially when webchat or WeChat retries produce the same message with different sequence numbers.

## Best Possible Solution

Land a narrow Gateway/auto-reply repair that dedupes retry duplicates even when provider ids or `chat.send` keys are regenerated, carries stable metadata into session-history updates, and preserves legitimate repeated user messages through channel/account/session/thread/media/timestamp scoping and regression tests.

## Reproduction Assessment

Yes. The issue provides concrete `sessions_history` seq/timestamp evidence, and current main has a clear source path where missing or regenerated `MessageSid`/idempotency keys bypass existing guards and receive fresh session-history sequence numbers.

## Solution Assessment

Yes. A focused replacement for the useful parts of #72290 is the narrow maintainable direction; broad duplicate-delivery umbrellas or UI-only fixes do not cover this retry/transcript gap.
