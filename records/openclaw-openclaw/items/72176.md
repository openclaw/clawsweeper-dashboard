## Summary

Keep open. Current main still has the duplicate-delivery gaps described in the report: inbound dedupe is skipped when no stable provider message id is present, chat.send dedupe remains scoped to the caller-provided idempotency key, and inline session-history updates can still append fresh sequence numbers without stable duplicate metadata. The focused related PR is closed unmerged, so there is no shipped or main-branch fix to close against.

## What This Changes

The issue reports intermittent duplicate user-message delivery after upgrading to OpenClaw 2026.4.24, producing duplicate assistant replies across webchat and WeChat.

## Best Possible Solution

End state should be a generic Gateway/auto-reply idempotency repair that catches retry duplicates when provider or client ids are missing or regenerated, propagates stable duplicate metadata into transcript/session-history updates, and preserves legitimate repeated messages through tight scoping, TTLs, and regression tests.

## Reproduction Assessment

Yes. The issue provides concrete sessions_history duplicate pairs, and current main has a source-level reproduction path: missing MessageSid produces an invalid inbound dedupe claim, regenerated chat.send idempotency keys start new runs, and inline history updates get fresh seq values without a stable id.

## Solution Assessment

Yes, with guardrails. A focused replacement repair using the useful ideas from #72290 is the narrowest maintainable path, but the fallback dedupe key must be scoped carefully to avoid suppressing intentional repeated messages.
