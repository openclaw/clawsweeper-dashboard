## Summary

Keep open. Current main still has the reported duplicate-delivery gap: inbound dedupe drops out when there is no stable `MessageSid`, `chat.send` dedupe is scoped to the caller-supplied idempotency key, and inline session-history user updates still lack stable duplicate metadata. The related fix PR #72290 is closed unmerged, so there is no merged implementation to close against.

## What This Changes

The issue asks OpenClaw to prevent intermittent duplicate user-message delivery and duplicate assistant replies after 2026.4.24 across WebChat, WeChat, and other channel paths.

## Best Possible Solution

End state should be a generic Gateway/auto-reply idempotency repair that catches retry duplicates when provider or client ids are missing or regenerated, propagates stable duplicate metadata into session-history updates, and preserves legitimate repeated messages through narrow scoping, TTLs, and regression tests.

## Reproduction Assessment

Yes. The issue provides concrete `sessions_history` duplicate pairs, and current main has a source-level reproduction path: missing `MessageSid` produces an invalid inbound dedupe claim, regenerated `chat.send` idempotency keys start new runs, and inline history updates get fresh seq values without a stable id.

## Solution Assessment

Yes. A focused replacement repair using the useful ideas from #72290 is the narrowest maintainable path; closing this under older duplicate-delivery fixes would leave the 2026.4.24 retry/transcript gap unaddressed.
