## Summary

Keep open with a blocking review finding: this PR reverses the current ACPX probe-by-default readiness contract, so it is a product/default-policy change rather than a conservative bug fix, and the external proof is still controlled/mock-only.

## What This Changes

The PR changes ACPX startup so the embedded ACP backend registers without a startup probe by default, makes the startup probe opt-in via `OPENCLAW_ACPX_RUNTIME_STARTUP_PROBE=1`, updates ACP setup docs, and adds focused registration tests.

## Best Possible Solution

Keep the shipped probe-by-default readiness contract unless maintainers explicitly choose a new ACPX startup policy; handle the broader latency problem through dependency-aware sidecar decoupling with live Gateway proof.

## Reproduction Assessment

Yes, source-level: current main defaults to probing before `gateway ready`, while the PR diff flips that default back to opt-in. I did not run a live Gateway or ACP adapter in this read-only review.

## Solution Assessment

No, not as-is. The safer path is to preserve the current readiness contract and solve startup latency with dependency-aware sidecar decoupling, or make an explicit maintainer-approved policy change with live proof.
