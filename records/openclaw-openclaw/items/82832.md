## Summary

Keep open: the PR has supplied real Gateway terminal proof for the changed default, but the patch still reverses the shipped ACPX readiness contract that current source, docs, and tests define as probe-by-default. This needs maintainer policy/design review rather than conservative cleanup closure.

## What This Changes

The PR makes ACPX skip the embedded runtime startup probe by default, starts the inner ACPX service from the wrapper, updates ACP setup docs, and adds focused registration/service tests.

## Best Possible Solution

Keep the current probe-by-default readiness contract unless maintainers explicitly approve a new ACPX startup policy; address latency through dependency-aware sidecar decoupling or equivalent readiness-preserving design.

## Reproduction Assessment

Yes, source-level: current main source/docs/tests show the default startup probe is awaited, while the PR diff and proof show patched Gateway startup skipping that probe. I did not run a live Gateway locally because this is a read-only review.

## Solution Assessment

No, not as-is. The PR directly reverses a recently shipped readiness fix; the safer path is maintainer-approved sidecar decoupling or an explicit policy change with matching degraded-status semantics.
