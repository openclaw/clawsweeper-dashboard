## Summary

Keep open for maintainer review: the PR now supplies real Gateway startup proof, but the patch reverses the shipped ACPX probe-by-default readiness contract and should not be treated as a narrow bug fix without an explicit product/readiness decision.

## What This Changes

The PR changes the ACPX plugin to skip the embedded runtime startup probe by default, makes probing opt-in via `OPENCLAW_ACPX_RUNTIME_STARTUP_PROBE=1`, updates ACP setup docs, and adds focused service/register tests.

## Best Possible Solution

Keep the current probe-by-default readiness contract unless maintainers explicitly approve a new ACPX startup policy; solve the latency concern through dependency-aware sidecar decoupling or another design that preserves degraded-status semantics.

## Reproduction Assessment

Yes, source-level: current main docs, tests, and service code show the default startup probe is awaited, while the PR body shows patched Gateway startup skipping that probe. I did not run a live Gateway locally because this was a read-only review.

## Solution Assessment

No, not as-is. The PR is a direct default-policy reversal of a recently shipped readiness fix; the safer path is maintainer-approved sidecar decoupling or an explicit policy change with matching readiness semantics.
