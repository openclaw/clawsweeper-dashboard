## Summary

Keep open. Current main still has the response-prefix context gap on fast abort/routed reply paths, and the discussion already has reproduced root-cause analysis plus open closing PRs that should be reviewed or superseded before this issue closes.

## What This Changes

Populate `responsePrefix` template variables for early abort replies such as `stop` and `/stop` instead of sending `{model}` or `{thinkingLevel}` literally.

## Best Possible Solution

Land or supersede the linked fix PRs with one narrow implementation that derives safe prefix context for fast abort and routed replies, preserves static prefixes, and keeps the normal `onModelSelected` override path intact.

## Reproduction Assessment

Yes. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` or `/stop` during an active run, and current source shows the abort reply is emitted before model-selection context is populated while missing template values remain literal.

## Solution Assessment

Unclear which open PR is the best final patch without maintainer review. The safer direction is to use actual session/runtime model metadata where available and pass prefix context into routed delivery rather than relying only on config defaults.
