## Label Justifications

- `P2`: This is a normal-priority user-visible reply formatting bug with a focused reply-prefix and abort-delivery surface.
- `impact:session-state`: The bug is caused by missing model/thinking runtime context when an early abort reply is normalized.

## Summary

Keep open: current main and v2026.5.19 still have a source-level path where fast-abort replies can normalize a templated `responsePrefix` without model/thinking context, and https://github.com/openclaw/openclaw/pull/45315 remains the active open closing PR.

## What This Changes

Populate `responsePrefix` template variables such as `{model}` and `{thinkingLevel}` for early stop/abort replies instead of sending raw placeholders.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/45315 or a focused replacement that carries populated reply-prefix context through dispatcher fallback and routed fast-abort delivery, with regression coverage for abort replies and normal `onModelSelected` override behavior.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` or `/stop` before `onModelSelected` runs, and the routed fast-abort path can normalize the abort reply with no model/thinking context.

## Solution Assessment

Not applicable as a patch verdict for this issue. The maintainable fix is to pass populated prefix context into the early abort routing path rather than dropping prefixes or relying on raw config strings.
