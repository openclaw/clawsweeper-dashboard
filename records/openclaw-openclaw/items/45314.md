## Summary

Keep open. Current main and the latest release still have a source-reproducible fast-abort path where templated response prefixes can be normalized without model/thinking context, and an open paired PR from the reporter is still the active implementation candidate.

## What This Changes

The issue reports that stop and /stop abort replies can show literal responsePrefix template variables such as {model} and {thinkingLevel}.

## Best Possible Solution

Land a focused fix that builds a session/runtime-aware ResponsePrefixContext and passes it through both direct dispatcher delivery and routeReply, with direct and routed fast-abort regression tests.

## Reproduction Assessment

Yes, at source level. Configure messages.responsePrefix with {model} or {thinkingLevel}, start a run, then send stop or /stop before onModelSelected; current main can normalize the abort reply with a templated prefix and missing context.

## Solution Assessment

No complete current-main solution is present. The open PR's default pre-seeding is useful but the maintainable fix should also carry the populated prefix context into routed reply normalization so all abort delivery paths resolve templates.
