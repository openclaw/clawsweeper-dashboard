## What This Changes

The issue reports that early stop or `/stop` abort replies can leak unresolved `messages.responsePrefix` template variables such as `{model}` and `{thinkingLevel}`.

## Best Possible Solution

Land or replace the linked PR with a fix that derives abort-prefix context from active session/channel model state and carries that context through routed delivery, then close this issue from the merged fix.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure a templated `messages.responsePrefix`, trigger stop or `/stop` while a run is active, and the fast-abort reply can be normalized before model/thinking context is populated.

## Solution Assessment

Yes on the requested direction, but not by closing this issue yet. The best fix should use active session/channel model context and routed-delivery prefix context rather than only suppressing raw templates or seeding config defaults.
