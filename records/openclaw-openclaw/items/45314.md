## Label Justifications

- `P2`: This is a normal-priority user-visible reply formatting bug with a focused abort-delivery and reply-prefix surface.
- `impact:session-state`: The bug is caused by missing model/thinking runtime context when an early abort reply is normalized.

## Summary

Keep open: current main still has a source-level path where early abort replies can apply a templated responsePrefix before model/thinking context is populated, and the open linked PR remains the active implementation candidate.

## What This Changes

The issue reports that early `stop` and `/stop` abort replies leak raw `responsePrefix` template variables such as `{model}` and `{thinkingLevel}` instead of populated values.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/45315 or a focused replacement that supplies safe runtime/session prefix context to early abort and routed reply normalization with regression coverage.

## Reproduction Assessment

Yes, at source level. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` or `/stop` before `onModelSelected` runs, and the fast-abort path can normalize the abort reply with missing model/thinking context while the resolver preserves unresolved placeholders.

## Solution Assessment

Not applicable as a patch verdict for this issue. The maintainable fix is to carry populated prefix context from runtime/session state into early abort and routed reply normalization rather than dropping prefixes or relying on raw config strings.
