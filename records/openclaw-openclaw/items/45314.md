## Summary

Keep open. Current main still has the fast-abort response-prefix context gap, and open implementation PRs already target this bug with closing references, so this issue should remain open until maintainers land or supersede a canonical fix.

## What This Changes

The issue reports that early stop or abort replies prepend unresolved `messages.responsePrefix` template variables instead of resolved model and thinking values.

## Best Possible Solution

Land or supersede the linked fix PRs with one narrow implementation that provides safe runtime or session-derived prefix context to fast abort and routed command replies, preserves static prefixes and the normal `onModelSelected` override path, and adds regression coverage for both dispatcher and routed delivery.

## Reproduction Assessment

Yes. Configure `messages.responsePrefix` with `{model}` or `{thinkingLevel}`, trigger `stop` or `/stop` during an active run, and the current source path shows the abort reply is emitted before model-selection context is populated while the template resolver preserves missing values literally.

## Solution Assessment

Unclear which open PR is the best final patch without maintainer review. The safest direction is to derive prefix context from actual session/runtime metadata where available, avoid leaking placeholders, and preserve static identity prefixes rather than dropping the entire prefix.
