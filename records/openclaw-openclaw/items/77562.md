## Summary

Keep open: current main still routes Control UI typed `/new` through `sessions.create` without the documented `command:new` / reset hook path, and an open fix PR already targets this regression.

## What This Changes

Typing `/new` in the Control UI should create a fresh session while still firing `command:new`, `before_reset`, and session lifecycle hooks for the previous session.

## Best Possible Solution

Land or replace the existing focused fix so Control UI `/new` keeps fresh dashboard-session creation while emitting the documented hooks against the previous session, with regression coverage for both hook-emitting and ordinary programmatic `sessions.create` calls.

## Reproduction Assessment

Yes. Source inspection on current main shows typed `/new` reaches `sessions.create`, while the documented hook path is only present in reset/command handling, so the reported hook bypass is reproducible from code without a live Gateway run.

## Solution Assessment

Yes. The narrowest maintainable direction is an explicit internal hook-emission path for Control UI `/new` session creation, rather than making every public `sessions.create` call fire command hooks.
