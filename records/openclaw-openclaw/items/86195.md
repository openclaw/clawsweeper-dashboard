## Summary

Keep open: current main and the latest release document the opt-in Telegram native draft lane, but they still do not warn that Telegram Android can block typed follow-ups while a native draft is active.

## What This Changes

The issue asks OpenClaw to document the Telegram Android interactive-control tradeoff for `nativeToolProgress`/`sendMessageDraft` and recommend a non-freezing configuration for `/steer`, `/stop`, and mid-run follow-ups.

## Best Possible Solution

Add a prominent Telegram docs note that native `sendMessageDraft` tool-progress previews are passive-viewing UI and recommend `nativeToolProgress: false` for workflows that need `/steer`, `/stop`, corrections, or mid-run follow-ups.

## Reproduction Assessment

No, not with high-confidence live proof from this review alone. Source inspection confirms OpenClaw can keep an active native DM tool-progress draft, and the upstream Android PR points to the client-side send-button gate, but no redacted Android proof was attached here.

## Solution Assessment

Yes for the docs-first path. Documenting the passive-viewing tradeoff and the `nativeToolProgress: false` workaround is the narrow maintainable fix; presets, option renames, or doctor warnings should be separate maintainer choices if docs are not enough.
