## Summary

Keep open. Current main still has a source-visible path where WeCom gateway-mode media sends can lose the host media access context, and the inspected WeCom plugin falls back to posting the raw media path as text when upload fails, matching the report.

## What This Changes

Make `openclaw message send --channel wecom --media <local file>` deliver the file attachment instead of sending the local file path as text.

## Best Possible Solution

Ensure WeCom media sends load and upload the requested local file through OpenClaw's host media access policy and fail with an actionable error instead of silently sending a path; update the official catalog or upstream plugin if that is where the fix belongs.

## Reproduction Assessment

Yes, source-reproducible with medium confidence: the CLI media send can route to a gateway-mode WeCom plugin action without media access, and the inspected plugin posts the media path as fallback text when upload fails. I did not run a live WeCom send.

## Solution Assessment

Unclear for implementation ownership, but closing is not the best path. The narrow maintainable fix should preserve existing media policy while routing readable file bytes or approved local roots into the WeCom send path, with live WeCom proof after the change.
