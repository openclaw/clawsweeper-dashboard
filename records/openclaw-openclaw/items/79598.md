## Summary

Close because the reported behavior is isolated to the existing external WeCom channel plugin, while current OpenClaw core already carries `--media` through the generic outbound media contract and exposes local-media access hooks for plugins.

## What This Changes

The issue reports that sending a WeCom message with `openclaw message send --media` delivers the file path text instead of the media file.

## Best Possible Solution

Keep the OpenClaw core media/plugin contract as-is and fix the WeCom local-file upload fallback in the external WeCom plugin, then publish the plugin update through its normal npm/ClawHub path.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path at the plugin boundary: OpenClaw core forwards the media reference, and the external WeCom plugin has a text fallback that sends the media path when upload fails. I did not run a live WeCom delivery, but live proof is not needed to determine the repository ownership boundary.

## Solution Assessment

Yes, closing this core issue and routing the fix to the existing WeCom plugin is the narrowest maintainable solution. OpenClaw already exposes the needed media URL/local-read contract, so a core change would duplicate plugin-owned channel behavior.
