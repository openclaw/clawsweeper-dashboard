## Summary

Keep open: current main still lacks media-fetch-specific outbound sanitization, and this PR addresses a real privacy leak but needs targeted fixes before merge because it bypasses shared redacted error formatting for non-media failures and uses an unsafe recursive cause-chain helper.

## What This Changes

The PR adds a safe MediaFetchError message/helper, sanitizes media fetch failures in outbound attachment hydration, broadcast results, and message_sent delivery events, and adds focused tests.

## Best Possible Solution

Land a rebased media-specific sanitizer that maps MediaFetchError to one stable channel-safe message at outbound action and hook boundaries, preserves formatErrorMessage for all non-media failures, uses cycle-safe cause traversal, and records the user-facing privacy fix in the changelog.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: loadWebMedia can throw detailed MediaFetchError text, hydrateAttachmentPayload forwards it, and current broadcast/message_sent paths format that error without a media-safe substitution.

## Solution Assessment

No, not as proposed. The direction is right, but the patch should keep formatErrorMessage(err) as the fallback for every non-media failure and make MediaFetchError cause traversal cycle-safe before merge.
