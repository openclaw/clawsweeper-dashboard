## Summary

Keep open: current main still lacks the media-safe substitution in the targeted outbound paths, and the linked privacy report supports the bug, but this PR is not merge-ready because it lacks real behavior proof and introduces formatter and traversal regressions.

## What This Changes

The branch adds a media-fetch safe error message/helper, redacts MediaFetchError failures at attachment hydration, broadcast, and message_sent boundaries, and adds focused sanitizer tests.

## Best Possible Solution

Land a rebased media-specific sanitizer that maps MediaFetchError failures to one stable channel-safe message, keeps formatErrorMessage for every non-media fallback, uses cycle-safe cause traversal, and includes focused tests plus redacted real channel proof.

## Reproduction Assessment

Yes, from source inspection rather than a live run: current main builds detailed MediaFetchError strings and forwards loadWebMedia, broadcast, and message_sent failures without a media-specific safe substitution. I did not run a live channel scenario because this review is read-only.

## Solution Assessment

No, not as currently proposed. The media-specific sanitizer is the right direction, but the patch should preserve the shared formatter fallback, make cause traversal cycle-safe, rebase onto current main, and include real behavior proof.
