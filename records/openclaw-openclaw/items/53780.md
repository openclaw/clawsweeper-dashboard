## Summary

Keep open: the underlying privacy bug still appears valid on current main, but this PR is not merge-ready because it would regress existing non-media error formatting/redaction and the new cause-chain detector can recurse indefinitely.

## What This Changes

The PR adds a MediaFetchError safe-message helper, applies it during attachment hydration, broadcast result reporting, and delivery failure hooks, and adds focused sanitizer tests.

## Best Possible Solution

Land a rebased media-specific sanitizer that maps MediaFetchError to one stable channel-safe message at outbound action and hook boundaries, keeps formatErrorMessage as the fallback for every non-media failure, uses cycle-safe cause traversal, and includes the required changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: failed loadWebMedia produces detailed MediaFetchError text, attachment hydration forwards it, and current broadcast/message_sent paths format that error instead of substituting a stable media-safe message.

## Solution Assessment

No, not as proposed. The direction is right, but the patch should preserve the shared non-media error formatter, make MediaFetchError detection cycle-safe, and rebase onto current main before merge.
