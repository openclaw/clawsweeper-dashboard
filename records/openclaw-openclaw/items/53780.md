## Summary

Keep open. The PR targets a real privacy-sensitive media error leak that current main still has in outbound action, broadcast, and message_sent paths, but the patch needs correctness/security fixes and real behavior proof before merge.

## What This Changes

The branch adds a safe MediaFetchError message/helper, redacts media fetch failures at attachment hydration, broadcast-result, and message_sent failure boundaries, and adds focused sanitizer tests.

## Best Possible Solution

Land a rebased, reviewed media-specific sanitizer that maps MediaFetchError to one stable channel-safe message, preserves shared formatting/redaction for all other failures, uses cycle-safe cause traversal, and includes focused tests plus real Telegram/channel proof.

## Reproduction Assessment

Yes, from source inspection. Current main can create detailed MediaFetchError text, forwards loadWebMedia failures from attachment hydration, and formats those failures into broadcast results or message_sent hooks without a media-safe substitution.

## Solution Assessment

No, not as proposed. The direction is right, but the patch should preserve formatErrorMessage for every non-media failure and make MediaFetchError cause traversal cycle-safe before merge.
