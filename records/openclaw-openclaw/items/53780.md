## Summary

Keep open: this is a focused privacy fix for a real channel-visible media error leak, but the PR is not merge-ready because it is conflicting, lacks real behavior proof, and introduces outbound error-formatting and availability regressions.

## What This Changes

The PR adds a MediaFetchError safe message/helper, applies it in outbound attachment hydration, broadcast results, and message_sent failures, and adds focused media sanitizer tests.

## Best Possible Solution

Land a rebased media-specific sanitizer that maps MediaFetchError failures to one stable channel-safe message, keeps formatErrorMessage for all non-media fallbacks, uses an iterative seen-set cause walk, and includes focused tests plus redacted real channel proof.

## Reproduction Assessment

Yes, by source inspection: current main builds detailed MediaFetchError messages and forwards loadWebMedia, broadcast, and message_sent failures without a media-specific safe substitution. I did not run a live channel scenario because this review is read-only.

## Solution Assessment

No, not as currently proposed. The media-specific sanitizer is the right direction, but the patch should preserve the shared formatter fallback, make cause traversal cycle-safe, rebase onto current main, and include real behavior proof.
