## Summary

Keep this PR open. The core routing bug is real and the chosen stream-boundary fix is directionally right, but the latest patch still has a blocking cross-provider header mismatch and drops many current Unreleased changelog entries.

## What This Changes

The PR updates the Ollama stream path to derive the chat URL and SSRF policy from runtime `model.baseUrl`, wraps configured Ollama streams to stamp an effective base URL, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Keep the narrow Ollama-plugin fix, but make base URL, SSRF policy, and request headers come from the same runtime model/provider request, add a regression that proves cross-provider headers do not leak, and reapply the changelog entry without deleting current main's release notes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: register one Ollama-compatible provider first, then invoke the shared `api: "ollama"` stream with another runtime model carrying a different `baseUrl` and headers; current main ignores the later URL, while the PR routes the URL but still uses captured first-provider headers.

## Solution Assessment

No, not as currently patched. Resolving URL and SSRF policy per request is the right seam, but the fix must also resolve headers per request and preserve the current changelog entries before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route headers with the per-request URL:** `extensions/ollama/src/stream.ts:1029-1030`
  - body: When a later Ollama-compatible provider enters the first registered stream, this code switches `chatUrl` to the runtime `model.baseUrl` but the request still merges `defaultHeaders` captured from the first provider. That can send provider A's Authorization header to provider B's URL, or omit provider B's required headers. Resolve runtime `model.headers` from the same model/base URL source and cover the cross-provider case.
  - confidence: 0.9
- **[P2] Restore the removed changelog entries:** `CHANGELOG.md:19`
  - body: The PR replaces the top of the Unreleased Fixes section with the Ollama entry and drops many existing current-main fixes. Rebase or resolve the conflict by adding the Ollama bullet while preserving all existing Unreleased entries.
  - confidence: 0.96

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete cross-provider credential-routing risk by changing the request URL per runtime model while keeping first-provider headers captured in the stream closure.

Concerns:

- **[high] Potential cross-provider Authorization header leak:** `extensions/ollama/src/stream.ts:1029`
  - body: If the first registered Ollama stream has configured headers and a later provider supplies a different runtime `baseUrl`, the PR sends the request to the later URL while still merging the first stream's `defaultHeaders`; provider credentials can be sent to the wrong endpoint.
  - confidence: 0.88
