## Summary

Keep open. The underlying multi-provider Ollama routing bug is real from source inspection, but this PR is not correct yet because it can route a request to the runtime base URL while still using the first provider's captured headers, and it lacks real behavior proof.

## What This Changes

The PR changes Ollama native stream routing to derive the chat URL and SSRF policy from runtime model baseUrl, adds stream-runtime tests, and edits the changelog.

## Best Possible Solution

Resolve the URL, SSRF policy, and headers from the same per-request runtime provider data, with regression coverage for cross-provider URL/header pairing.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure two Ollama-compatible providers sharing `api: "ollama"`, register one first, then invoke the shared stream with a later runtime model carrying different `baseUrl` and headers.

## Solution Assessment

No. Per-request URL and SSRF recomputation is the right direction, but the patch must route headers from the same runtime model/provider data before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route headers with the per-request URL:** `extensions/ollama/src/stream.ts:1029-1030`
  - body: When a later Ollama-compatible provider enters the first registered stream, this code switches `chatUrl` to the runtime `model.baseUrl` while the request still merges `defaultHeaders` captured from the first provider. That can send provider A's Authorization or custom headers to provider B's URL, so resolve runtime headers from the same provider data and cover the cross-provider case.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The patch introduces a concrete cross-provider credential-routing risk by decoupling per-request URLs from per-request headers.
