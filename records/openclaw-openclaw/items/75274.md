The PR changes the Ollama stream path to resolve the chat URL and SSRF policy from runtime `model.baseUrl` per request, wraps configured streams to pass an effective base URL through, and adds an Unreleased changelog entry.

## Best Possible Solution

Rebase the PR onto current main, keep the changelog entry in the active Unreleased Fixes section, and add a focused `stream-runtime` regression test proving runtime `model.baseUrl` overrides both the fetch URL and matching SSRF policy for a stream created with a different default base URL.

Yes. Current main provides a source-level reproduction path: register one Ollama-compatible stream first, then invoke the shared `api: "ollama"` registry path with a runtime model whose `baseUrl` points elsewhere; the current stream closure still uses the first URL.

## Solution Assessment

Yes, with branch and test cleanup. Resolving routing at the Ollama stream boundary from the runtime model is narrower than redesigning the shared API registry, and it preserves existing default behavior when no runtime base URL is present.
