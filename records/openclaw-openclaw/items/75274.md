## What This Changes

The PR updates `extensions/ollama/src/stream.ts` so native Ollama streams resolve chat URLs and SSRF policy from the runtime model `baseUrl` per request, with a configured-stream wrapper that preserves runtime URLs before falling back to the creation-time URL.

## Best Possible Solution

Land the narrow Ollama-plugin routing fix after adding the required changelog entry and preferably a focused regression test for two configured providers sharing `api: "ollama"`; keep any provider-alias or manifest expansion as separate work unless the current API-owner hint path proves insufficient.

## Reproduction Assessment

Yes. The PR body gives a concrete cloud-plus-LAN setup, and current main confirms the failure path: `api: "ollama"` streams share a first-write registry slot while the existing stream closure bakes in one `chatUrl`. A focused unit test can reproduce it by invoking the first registered stream with a second runtime model carrying a different `baseUrl`.

## Solution Assessment

Yes, with a small process fix. Moving URL and SSRF-policy resolution into the Ollama stream path is the narrowest maintainable layer for the registry first-write behavior, and the updated wrapper now preserves the runtime `baseUrl`; the remaining merge readiness issue is release-note coverage and ideally regression proof.
