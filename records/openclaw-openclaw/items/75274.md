## Summary

Keep open; current main still has a real Ollama multi-provider routing surface, but this PR is not mergeable because it separates per-request URL routing from closure-captured headers and drops current changelog entries.

## What This Changes

The branch routes Ollama chat requests through a runtime `model.baseUrl` override, updates the configured stream wrapper/tests, and adds an Ollama changelog entry.

## Best Possible Solution

Resolve base URL, SSRF policy, and provider/model headers from the same per-request runtime provider data, preserve current changelog entries, and add a regression proving credentials do not cross providers.

## Reproduction Assessment

Yes. A source-level reproduction is clear: register one configured Ollama-compatible provider first, then invoke the shared `api: "ollama"` stream with another runtime model that has a different `baseUrl` and headers; current main uses the first closure, while this PR routes to the later URL with first-provider headers.

## Solution Assessment

No. Recomputing URL and SSRF policy per request is the right direction, but the current patch must also recompute headers from the same runtime model/provider request source and resolve the changelog conflict.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route headers with the per-request URL:** `extensions/ollama/src/stream.ts:1029-1030`
  - body: When a later Ollama-compatible provider enters the first registered stream, this code switches `chatUrl` to the runtime `model.baseUrl` but the request still merges `defaultHeaders` captured from the first provider. That can send provider A's Authorization header to provider B's URL or omit provider B's required headers, so resolve runtime headers from the same model/provider data and cover the cross-provider case.
  - confidence: 0.9
- **[P2] Restore the removed changelog entries:** `CHANGELOG.md:19`
  - body: The PR replaces the top of the Unreleased Fixes section with the Ollama entry and drops existing current-main fixes. Rebase or resolve the conflict by adding the Ollama bullet while preserving the current entries.
  - confidence: 0.96

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete cross-provider credential-routing risk by changing the request URL per runtime model while keeping first-provider headers captured in the stream closure.

Concerns:

- **[high] Potential cross-provider Authorization header leak:** `extensions/ollama/src/stream.ts:1029`
  - body: If the first registered Ollama stream has configured headers and a later provider supplies a different runtime `baseUrl`, the PR sends the request to the later URL while still merging the first stream's `defaultHeaders`; provider credentials can be sent to the wrong endpoint.
  - confidence: 0.88
