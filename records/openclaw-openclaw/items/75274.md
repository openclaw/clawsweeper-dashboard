## Summary

Keep open: the reported Ollama multi-provider routing bug is valid on current main, but this PR is not mergeable as-is because it changes the request URL per runtime model while still using closure-captured headers from the first registered provider.

## What This Changes

The PR changes Ollama stream routing to prefer a runtime model base URL for each request, adds regression tests, and edits the changelog.

## Best Possible Solution

Resolve the URL, SSRF policy, and headers from the same per-request runtime provider data, then prove the multi-provider case without crossing credentials between providers.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: register one configured Ollama-compatible provider first, then invoke the shared `api: "ollama"` stream with a later runtime model carrying a different `baseUrl` and headers. Current main keeps the first stream closure's URL and headers.

## Solution Assessment

No. Per-request URL and SSRF recomputation is the right direction, but the current patch must route headers from the same runtime model/provider data before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route headers with the per-request URL:** `extensions/ollama/src/stream.ts:1029-1030`
  - body: When a later Ollama-compatible provider enters the first registered stream, this code switches `chatUrl` to the runtime `model.baseUrl` but the request still merges `defaultHeaders` captured from the first provider. That can send provider A's Authorization header to provider B's URL or omit provider B's required headers, so resolve runtime headers from the same provider data and cover the cross-provider case.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The patch introduces a concrete cross-provider credential-routing risk by decoupling per-request URLs from per-request headers.

Concerns:

- **[high] Potential cross-provider Authorization header leak:** `extensions/ollama/src/stream.ts:1029`
  - body: The PR can send a request to the later provider's `model.baseUrl` while still merging the first registered provider's `defaultHeaders`, including Authorization or other custom credentials.
