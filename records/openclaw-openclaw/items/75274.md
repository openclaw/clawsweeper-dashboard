## Summary

Keep this PR open, but do not merge it as-is. The reported multi-provider Ollama routing bug is still present on current main, and the PR is aimed at the right plugin-owned path, but its wrapper overwrites the per-request runtime `baseUrl` with the first registered provider's captured URL, preserving the first-write routing failure it is meant to fix.

## What This Changes

The PR changes `extensions/ollama/src/stream.ts` so native Ollama streams can compute chat URL and SSRF policy from `model.baseUrl` per request and wraps configured streams to attach a resolved base URL.

## Best Possible Solution

Keep the fix in the Ollama plugin: make the registered stream function prefer a non-empty runtime model URL, and review runtime headers in the same path, with closure defaults only as fallback. Add a focused regression test that simulates two configured Ollama-compatible providers sharing `api: "ollama"` so the first registered stream is invoked with the second provider's runtime model.

## Reproduction Assessment

Yes. Current main confirms the reproduction path: the custom API registry drops later `ollama` stream registrations, the Ollama stream closure has a fixed URL, and runtime models carry the selected provider `baseUrl`. A unit test can reproduce this without live network by invoking the first registered configured stream with a second provider runtime model.

## Solution Assessment

No. Per-request URL resolution is the right layer, but this implementation is not the best fix because the wrapper overwrites the runtime `model.baseUrl` with the captured first provider URL. The safer repair is to preserve a non-empty runtime `model.baseUrl`, fall back to the configured default only when absent, and cover the two-provider path in tests.
