## Summary

Keep open: the branch targets a real Ollama first-write-wins routing bug, but it is not merge-ready because it routes the URL per runtime model while still using headers captured from the first provider, and its changelog hunk drops current entries.

## What This Changes

The PR changes Ollama chat streaming to derive the request URL from runtime `model.baseUrl`, wraps configured streams to stamp an effective base URL, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Resolve base URL, SSRF policy, and headers from the same per-request runtime provider/model data, preserve existing changelog entries, and add a regression that proves credentials do not cross providers.

## Reproduction Assessment

Yes. A source-level reproduction is clear: register one configured Ollama-compatible provider first, then invoke the shared `api: "ollama"` stream with a later runtime model carrying a different `baseUrl` and headers; the PR routes to the later URL while retaining first-provider default headers.

## Solution Assessment

No. Per-request URL recomputation is the right direction, but this patch must also route headers from the same runtime provider data and resolve the changelog conflict before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route headers with the per-request URL:** `extensions/ollama/src/stream.ts:1029-1030`
  - body: When a later Ollama-compatible provider enters the first registered stream, this code switches `chatUrl` to runtime `model.baseUrl` but the request still merges `defaultHeaders` captured from the first provider. That can send provider A's Authorization/custom headers to provider B's URL or omit provider B's required headers, so resolve runtime headers from the same provider data and cover the cross-provider case.
  - confidence: 0.9
- **[P2] Restore the removed changelog entries:** `CHANGELOG.md:19`
  - body: This hunk replaces the top of the Unreleased Fixes section with the Ollama entry and drops existing current-main fixes. Rebase or resolve the conflict by adding the Ollama bullet while preserving the current entries.
  - confidence: 0.96

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete cross-provider credential-routing risk by changing the request URL per runtime model while keeping first-provider headers captured in the stream closure.

Concerns:

- **[high] Potential cross-provider Authorization header leak:** `extensions/ollama/src/stream.ts:1029`
  - body: If the first registered Ollama stream has configured headers and a later provider supplies a different runtime `baseUrl`, the PR sends the request to the later URL while still merging the first stream's `defaultHeaders`; provider credentials can be sent to the wrong endpoint.
  - confidence: 0.88
