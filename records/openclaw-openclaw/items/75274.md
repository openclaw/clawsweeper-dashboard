## Summary

Keep this PR open: current main still has the Ollama first-registered stream routing problem, but this patch is not merge-ready because it routes the URL per runtime model while leaving provider headers captured from the first stream closure. That creates a concrete cross-provider credential-routing risk, and the branch also drops existing changelog entries from current main.

## What This Changes

The PR changes Ollama streaming to recompute the chat URL and SSRF policy from runtime `model.baseUrl`, wraps configured streams to stamp an effective base URL, adds runtime tests, and modifies `CHANGELOG.md` with an Ollama fix entry.

## Best Possible Solution

Keep the narrow Ollama-plugin direction, but resolve base URL, SSRF policy, and headers from the same per-request runtime model/provider data, then add a regression that proves cross-provider headers do not leak and rebase the changelog without deleting main's existing entries.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: register one configured Ollama-compatible provider first, then invoke the shared `api: "ollama"` stream with another runtime model carrying a different `baseUrl` and headers; main ignores the later URL, while this PR changes the URL but still uses the first provider's captured headers.

## Solution Assessment

No, not as currently patched. Recomputing URL and SSRF policy per request is the right seam, but the maintainable fix must route headers from the same runtime model/provider and preserve the current changelog entries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route headers with the per-request URL:** `extensions/ollama/src/stream.ts:1029-1030`
  - body: When a later Ollama-compatible provider enters the first registered stream, this code switches `chatUrl` to the runtime `model.baseUrl` but the request still merges `defaultHeaders` captured from the first provider. That can send provider A's Authorization header to provider B's URL, or omit provider B's required headers. Resolve runtime `model.headers` from the same model/base URL source and cover the cross-provider case.
  - confidence: 0.9
