## Summary

Keep open: this is a plausible fix for a real Ollama multi-provider routing bug, but the current patch still pairs a per-request URL with closure-captured headers and lacks contributor-provided real behavior proof.

## What This Changes

The PR recomputes Ollama native chat URL and SSRF policy from runtime `model.baseUrl`, wraps configured Ollama streams to stamp an effective base URL, adds stream-runtime URL tests, and edits `CHANGELOG.md`.

## Best Possible Solution

Resolve URL, SSRF policy, and default headers from the same per-request runtime provider data, then cover URL/header pairing for multiple Ollama-compatible providers.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure two Ollama-compatible providers sharing `api: "ollama"`, let the first register the shared stream, then invoke it with a later runtime model carrying a different `baseUrl` and headers.

## Solution Assessment

No. Per-request URL/SSRF recomputation is the right direction, but this patch must route headers from the same runtime model/provider data before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Route headers with the per-request URL:** `extensions/ollama/src/stream.ts:1029-1030`
  - body: When a later Ollama-compatible provider enters the first registered stream, this line switches `chatUrl` to the runtime `model.baseUrl` while the request still merges `defaultHeaders` captured from the first provider. That can send provider A's Authorization or custom headers to provider B's URL, so resolve the runtime headers from the same provider data and add coverage for the cross-provider URL/header pair.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The patch introduces a concrete cross-provider credential-routing risk by decoupling per-request URLs from per-request headers.

Concerns:

- **[high] Potential cross-provider Authorization header leak:** `extensions/ollama/src/stream.ts:1029`
  - body: The PR can send a request to a later provider's `model.baseUrl` while still merging the first registered provider's `defaultHeaders`, including Authorization or other custom credentials.
  - confidence: 0.88
