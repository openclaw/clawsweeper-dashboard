## Summary

Keep open: the PR targets a valid Ollama multi-provider routing bug, but the current head is not mergeable because request URLs can now diverge from the configured headers/credentials, and the branch is stale against current changelog history.

## What This Changes

The branch changes the Ollama plugin stream to derive the chat URL and SSRF policy from runtime `model.baseUrl`, wraps configured streams to stamp an effective base URL, adds stream runtime regression tests, and adds an Ollama changelog entry.

## Best Possible Solution

Keep the Ollama-plugin fix direction, but resolve base URL, SSRF policy, and request headers from the same runtime provider/model data, then rebase the branch so the changelog keeps current main history.

## Reproduction Assessment

Yes. Register two Ollama-compatible providers sharing `api: "ollama"`, give the first provider configured headers/base URL A, then invoke the shared stream with a runtime model for provider B carrying base URL and headers B; this PR routes the URL to B while still merging A's captured `defaultHeaders`.

## Solution Assessment

No, not as currently patched. Per-request URL and SSRF recomputation is the right seam, but the safe fix must also select headers from the same runtime provider and preserve current changelog entries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Route headers with the per-request URL:** `extensions/ollama/src/stream.ts:1029-1030`
  - body: When a later Ollama-compatible provider enters the first registered stream, this code switches `chatUrl` to the runtime `model.baseUrl` but the request still merges `defaultHeaders` captured from the first provider. That can send provider A's configured Authorization or custom header to provider B's URL, or omit provider B's required headers. Resolve runtime `model.headers` from the same model/base URL source and cover the cross-provider header case.
  - confidence: 0.9
- **[P2] Rebase without dropping changelog history:** `CHANGELOG.md:9-19`
  - body: The branch's `CHANGELOG.md` is stale against current main: compared with `11a268819e82`, it removes current Unreleased entries and the `2026.4.30` release section while adding the Ollama bullet. Rebase and add only the new Ollama entry without deleting existing history.
  - confidence: 0.95

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete cross-provider credential-routing risk by changing the request URL per runtime model while keeping first-provider headers captured in the stream closure.

Concerns:

- **[high] Potential cross-provider Authorization header leak:** `extensions/ollama/src/stream.ts:1029`
  - body: If the first registered Ollama stream has configured headers and a later provider supplies a different runtime `baseUrl`, the PR sends the request to the later URL while still merging the first stream's `defaultHeaders`; provider credentials or custom auth headers can be sent to the wrong endpoint.
  - confidence: 0.88
