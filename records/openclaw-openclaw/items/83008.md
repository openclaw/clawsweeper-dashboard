Keep open. Current main still contains the Telegram requested-stop worker and stale stop-timeout ingredients, and an open pull request already uses GitHub closing syntax for this issue, so the issue should remain open until that PR or a replacement fix lands.

## What This Changes

Fix Telegram channel hot reload so requested-stop ingress worker failures and stop timeouts do not leave stale task state that blocks restart.

## Best Possible Solution

Land a focused Telegram/gateway lifecycle fix, through https://github.com/openclaw/openclaw/pull/83024 or a replacement, that makes requested-stop worker errors and manual stop timeouts restartable during hot reload with regression coverage and real Telegram/gateway proof.

No live high-confidence reproduction was run in this read-only review. The issue includes concrete logs, and current source plus `src/gateway/server-channels.test.ts` make the stale-task timeout path source-reproducible during a `channels.telegram.*` hot reload.

## Solution Assessment
