## Summary

Keep open. Current main still has the reported shutdown/reload ingredients: isolated Telegram ingress worker failures propagate out of the polling session, gateway hot reload stops then immediately starts the channel, and the channel manager intentionally leaves stale task state in place when a manual stop times out.

## What This Changes

The issue reports that Telegram config hot reload can wedge isolated polling accounts when ingress shutdown exits nonzero or otherwise misses the stop grace window.

## Best Possible Solution

Add a focused regression around Telegram isolated-ingress reload shutdown, then fix the owner boundary so requested-stop worker failures and manual stop timeouts cannot leave stale channel task state blocking the reload restart.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection and existing tests show a narrow repro path with an injected isolated ingress worker or timed-out channel task during `channels.telegram.*` hot reload.

## Solution Assessment

Unclear that the proposed one-line worker exit-code change is the whole fix. A safer fix should first add a failing regression, then either treat requested-stop worker rejection as clean, catch abort-time worker failures in `#runIsolatedIngressCycle`, or make hot-reload stop timeout cleanup restartable without weakening the duplicate-start guard.
