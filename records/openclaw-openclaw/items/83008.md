## Summary

This should stay open: the report is credible against current main, and source inspection shows a Telegram hot-reload stop-timeout path can skip the replacement start after an isolated ingress worker stop. The one-line exit-code proposal may be useful, but the durable fix should cover the supervisor timeout and stale task state as well.

## What This Changes

The issue asks to prevent Telegram isolated polling ingress shutdown during `channels.telegram.*` hot reload from wedging the channel after the worker exits non-zero or stop times out.

## Best Possible Solution

Add a focused lifecycle fix that guarantees a `channels.telegram.*` hot reload eventually starts a fresh Telegram account after abort timeout or isolated worker stop failure, with regression coverage for both the supervisor timeout and isolated ingress abort path.

## Reproduction Assessment

Yes, source inspection plus the reporter's logs provide a high-confidence reproduction path: trigger a `channels.telegram.*` hot reload while isolated polling shutdown exceeds the supervisor timeout, then observe `startChannel` skip because the old task is still registered. I did not run a live Telegram setup in this read-only pass.

## Solution Assessment

No, the proposed exit-code-only patch is not clearly the best complete fix on current main. The safer repair should also address or test the hot-reload stop-timeout path where stale task state can skip the replacement start even if the worker later exits cleanly.
