## Review Findings

Overall confidence: 0.9

Full review comments:

- **[P1] Point the runtime entry at a valid setter:** `extensions/channel-broker/index.ts:17-18`
  - body: `defineBundledChannelEntry` will load `setChannelBrokerRuntime` from `./runtime-api.js` when the bundled channel runtime is registered, but that file does not export the setter. This leaves the new channel with a missing runtime export; either remove the runtime block if no PluginRuntime injection is needed, or expose a correctly typed PluginRuntime setter from the referenced module.
  - confidence: 0.94
- **[P2] Do not document inbound webhooks before wiring them:** `docs/channels/channel-broker.md:17-18`
  - body: The new docs say V1 supports signed inbound HTTP webhooks and describe `signingSecret`/`allowFrom`, but the plugin only declares receive capability metadata and no manifest/runtime route handles provider webhooks. Users would configure an inbound path that cannot deliver messages, so keep this documented as future work or add the actual signed webhook route.
  - confidence: 0.88
- **[P2] Validate broker receipts before trusting them:** `extensions/channel-broker/src/runtime.ts:46-50`
  - body: The HTTP transport accepts any object as a `BrokerReceiptV1`, and the send path then commits `status`, `messageIds`, provider metadata, and edit/delete tokens from that unchecked response. Validate version, request/provider/platform identity, status, and message id shape at this boundary before treating provider output as a sent receipt.
  - confidence: 0.87
