## What This Changes

This PR adds scheduler-owned `scheduleActivatedAtMs` cron state, uses it to bound restart/backoff catch-up, backfills legacy rows in doctor preflight, updates the cron protocol schema, and adds focused cron/doctor tests.

## Best Possible Solution

Rebase onto current main and land the scheduler-owned activation boundary plus doctor migration if maintainers accept the conservative migration-time activation semantics; otherwise replace it with an equivalent state model they explicitly choose.

## Solution Assessment

Yes, this is the best fix shape I found: scheduler-owned activation state plus doctor-owned legacy backfill matches the persisted-state ownership boundary. The remaining work is rebase/conflict refresh and maintainer acceptance of the conservative upgrade semantics, not a narrower code repair.
