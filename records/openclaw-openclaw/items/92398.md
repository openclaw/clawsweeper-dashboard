## Summary

## What This Changes

The PR refactors cron restart missed-slot replay to ignore previous cron slots at or before `updatedAtMs` and adds restart catch-up regressions for updated schedules, deferred agent turns, and error-backoff deferral.

## Best Possible Solution

Land this PR's `updatedAtMs`-based replay boundary if maintainers accept fail-safe suppression over late outbound sends, then close the companion PR and linked issue after merge.

## Reproduction Assessment

Yes. Current main reaches restart catch-up through `start()` and classifies cron missed slots with `previousRunAtMs > lastRunAtMs` without considering an updated future `nextRunAtMs`; the PR body also supplies before/after Gateway, CLI, SQLite, and log proof.

## Solution Assessment

Yes, with maintainer sign-off on the semantics. Reusing `updatedAtMs` is narrower than the companion PR's new persisted schedule timestamp, but maintainers should explicitly accept the metadata-only write tradeoff before merge.
