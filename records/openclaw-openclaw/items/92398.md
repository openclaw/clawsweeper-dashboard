## What This Changes

The PR adds scheduler-owned cron schedule activation state, uses it to bound restart/backoff missed-slot replay, adds a doctor backfill for legacy rows, updates the protocol state schema, and adds focused cron/doctor tests.

## Best Possible Solution

Land this PR, or an equivalent replacement, after maintainers explicitly accept the conservative migration-time activation boundary and keep exact-head proof/checks valid.

## Reproduction Assessment

Yes. Current main is source-reproducible: cron.update can move nextRunAtMs without recording schedule activation, and restart catch-up can still replay a previous cron slot solely because it is newer than lastRunAtMs.

## Solution Assessment

Yes, with maintainer acceptance required. A scheduler-owned activation timestamp plus doctor-owned legacy backfill is the cleanest owner-boundary fix; reusing updatedAtMs would be less precise for metadata-only edits, while changing legacy replay semantics remains a maintainer call.
