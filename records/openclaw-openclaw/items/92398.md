## Reproduction Assessment

Yes, at source level. Current main recomputes nextRunAtMs on cron.update without storing schedule activation, and restart catch-up can still replay a previous cron slot solely because it is newer than lastRunAtMs.

## Solution Assessment

Yes, this is the best fix shape I found: a scheduler-owned activation timestamp plus doctor-owned legacy backfill matches the state ownership boundary. The remaining question is maintainer acceptance of the conservative upgrade semantics, not a better code layer.
