## Label Justifications

- `merge-risk: 🚨 message-delivery`: The new cutoff intentionally changes whether a missed cron delivery is replayed after a subsequent job edit.

## Summary

The PR targets the correct cron startup-catch-up path, has convincing live before/after proof, and introduces no concrete correctness or security defect; it should remain open for merge review, with maintainers explicitly owning the documented safety-first choice that later job edits suppress best-effort replay of earlier slots.

## What This Changes

The PR prevents cron startup catch-up and pending error-backoff logic from replaying slots at or before the job’s latest write time, and adds three restart regression tests for API-updated schedules.

## Best Possible Solution

Keep one canonical missed-slot predicate shared by startup and backoff paths, and land this safety-first boundary if maintainers agree that any later job edit supersedes best-effort replay of earlier slots.

## Reproduction Assessment

Yes. The contributor reproduced the current-main failure through a real Gateway and CLI path, and the linked reporter timeline independently matches the same 120-second startup deferral.

## Solution Assessment

Yes, subject to an explicit safety-policy choice. The shared predicate is the narrowest maintainable fix for both startup replay and backoff re-entry without adding storage, config, or a parallel scheduling path.
