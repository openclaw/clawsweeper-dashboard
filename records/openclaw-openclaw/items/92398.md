## What This Changes

The PR adds scheduler-owned cron schedule activation state, bounds restart/backoff catch-up by that state, backfills legacy cron rows in doctor preflight, updates the cron protocol schema, and adds focused tests.

## Best Possible Solution

Land this scheduler-owned activation boundary and doctor migration once maintainers accept the conservative migration-time activation semantics, or replace it with an equivalent patch that preserves the same ownership model and proof.

## Reproduction Assessment

Yes, at source level. Current main recomputes nextRunAtMs on cron.update without recording schedule activation, and restart catch-up can replay a previous cron slot solely because it is newer than lastRunAtMs.

## Solution Assessment

Yes, this is the best fix shape I found: scheduler-owned activation state plus a doctor-owned legacy backfill matches the persisted-state ownership boundary. The remaining question is maintainer acceptance of the conservative upgrade semantics, not a narrower code repair.
