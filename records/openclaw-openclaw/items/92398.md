## Label Justifications

- `P1`: The PR fixes a user-facing cron regression where scheduled outbound jobs can execute at the wrong startup time after a schedule update.
- `merge-risk: 🚨 message-delivery`: Changing startup catch-up classification can intentionally suppress a missed outbound run in ambiguous timestamp windows, even though it prevents early delivery.

## Summary

No blocking correctness or security finding: current main still has the pre-update missed-slot replay path, and this PR cleanly bounds startup replay/backoff deferral with regression coverage and real runtime proof. The remaining maintainer call is accepting the documented safer behavior that ambiguous metadata-only updates skip best-effort catch-up rather than risk an early outbound run.

## What This Changes

The PR refactors cron startup missed-slot detection into a helper that rejects slots at or before `updatedAtMs`, uses it for startup replay and backoff deferral, and adds restart catch-up regressions.

## Best Possible Solution

Land this focused scheduler fix once maintainers accept the safer skip-on-ambiguous-`updatedAtMs` tradeoff and required checks remain green.

## Reproduction Assessment

Yes. Current main's startup catch-up path can infer a missed slot from the current cron expression and `lastRunAtMs` even when an API update already moved `nextRunAtMs` forward; the linked issue and PR proof give a concrete 120s deferral scenario, though I did not run local tests in this read-only review.

## Solution Assessment

Yes. Reusing `updatedAtMs` keeps the fix narrow and avoids a new persisted API surface while applying the same replay predicate to both startup catch-up and backoff deferral; a wider schedule-definition timestamp would only be better if maintainers reject the metadata-only update tradeoff.
