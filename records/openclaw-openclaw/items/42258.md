## Summary

Keep this PR open. Current main still has the stale `chatRunId` queue-blocking path and no watchdog, so the PR targets a real Control UI bug, but the branch is not ready to merge because it misses terminal `agent.wait` timeout snapshots, surfaces background probe errors to users, and lacks the required changelog entry for a user-facing fix.

## What This Changes

The PR adds a Control UI chat-run watchdog that tracks chat activity, probes `agent.wait` after idle time, clears stale run state, reloads history, flushes queued messages, and adds watchdog regression coverage.

## Best Possible Solution

Land a narrow Control UI watchdog that uses the existing `agent.wait` contract correctly: recover terminal snapshots including timeout-ended runs, keep probe-only failures non-visible while retrying, preserve existing queue semantics, and document the user-facing fix in the active changelog.

## Reproduction Assessment

Yes. The source path is high-confidence: set `chatRunId`, miss the matching terminal chat event, queue a normal follow-up, and `flushChatQueue` returns while `isChatBusy` stays true; `agent.wait` can then prove whether the gateway already has a terminal run snapshot.

## Solution Assessment

No, not yet. A client-side `agent.wait` watchdog is the narrowest maintainable direction, but this implementation must distinguish wait-deadline `timeout` from terminal timeout snapshots and avoid routing transient probe failures into visible chat errors.
