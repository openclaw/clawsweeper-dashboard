## What This Changes

Adds a Control UI client-side watchdog that probes `agent.wait` after chat inactivity to clear stale `chatRunId`, reload history, and flush queued chat messages.

## Best Possible Solution

Land a rebased Control UI recovery path that uses the current `agent.wait` contract, treats terminal timeout snapshots as finished, uses current run-lifecycle cleanup, suppresses transient probe errors, and includes redacted real browser/gateway proof.

## Reproduction Assessment

Yes, at source level. Leave `chatRunId` non-null after a missed terminal chat event, submit a normal message, and current main queues it while `flushChatQueue` returns because `isChatBusy` remains true.

## Solution Assessment

No, not as currently patched. A UI-side `agent.wait` recovery probe is the right narrow direction, but this implementation must handle terminal timeout snapshots, avoid visible background probe errors, and be rebased onto current run-lifecycle cleanup.

Overall correctness: patch is incorrect

- **[P2] Treat terminal timeout snapshots as finished:** `ui/src/ui/app-chat.ts:84`
  - body: `agent.wait` can return an already-ended run as `status: "timeout"` with terminal metadata such as `endedAt`, `stopReason`, `livenessState`, or `yielded`. This branch treats every timeout as still active, so a timed-out run can keep retrying and leave the queued message blocked.
  - confidence: 0.9

  - confidence: 0.82
