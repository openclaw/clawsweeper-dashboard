## Summary

Keep open: current main still has a source-reproducible stale `chatRunId` queue block, and this PR is not merge-ready because it misses terminal `agent.wait` timeout snapshots and lacks real browser/gateway proof.

## What This Changes

The PR adds a Control UI client-side chat-run watchdog that probes `agent.wait` after inactivity and clears stale busy state so queued chat messages can flush.

## Best Possible Solution

Land a rebased Control UI recovery path that uses authoritative `agent.wait` terminal metadata, stays aligned with the shared run-lifecycle helper, suppresses silent probe noise, preserves regression coverage, and includes redacted live browser/gateway proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: leave `chatRunId` non-null after a missed terminal chat event, submit a normal message, and current main queues it while `flushChatQueue` exits because `isChatBusy` remains true.

## Solution Assessment

Is this the best way to solve the issue? No, not as currently patched; a UI-side `agent.wait` watchdog is a narrow maintainable direction, but this branch must handle terminal timeout snapshots, avoid visible transient probe errors, and rebase onto current run-lifecycle code.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle terminal timeout snapshots as finished:** `ui/src/ui/app-chat.ts:84`
  - body: `agent.wait` can return an already-ended run as `status: "timeout"` with terminal metadata such as `endedAt`, `stopReason`, `livenessState`, or `yielded`. This watchdog treats every timeout as still active, so a timed-out or cancelled run can keep looping and leave the queued message blocked.
  - confidence: 0.88
- **[P3] Suppress transient watchdog probe errors:** `ui/src/ui/app-chat.ts:97-99`
  - body: The probe is a silent 50 ms background recovery check that retries automatically. Writing a thrown probe error into `host.lastError` can show a visible error banner even when the main gateway connection and run are otherwise healthy.
  - confidence: 0.82
