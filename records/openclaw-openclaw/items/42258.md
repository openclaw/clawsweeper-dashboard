## Summary

Keep open. Current main still has the source-reproducible stale Control UI busy latch path, so this PR is not obsolete, but the branch is not merge-ready because it mishandles terminal `agent.wait` timeout snapshots, can surface transient watchdog probe failures as visible UI errors, is conflicting, and lacks real browser/gateway proof.

## What This Changes

The PR adds a Control UI chat-run watchdog that probes `agent.wait` after inactivity, clears stale local run state, reloads chat history, flushes queued messages, and adds focused tests.

## Best Possible Solution

Land a rebased watchdog or equivalent UI recovery path that treats authoritative `agent.wait` terminal metadata as finished, keeps transient background probe failures out of visible UI errors, preserves focused regression coverage, and includes redacted real browser/gateway proof.

## Reproduction Assessment

Yes, at source level. Leave `chatRunId` non-null after a missed terminal chat event, submit another message, and current main keeps `isChatBusy` true so `flushChatQueue` returns before sending the queued item.

## Solution Assessment

Mostly, but not as currently patched. A UI-side `agent.wait` watchdog is a narrow maintainable fix, but this branch needs to treat terminal timeout/cancel snapshots as finished and avoid surfacing transient probe errors.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Treat terminal timeout snapshots as finished:** `ui/src/ui/app-chat.ts:84`
  - body: `agent.wait` can return an already-ended run as `status: "timeout"` with terminal metadata such as `endedAt`, `stopReason`, `livenessState`, or `yielded`. This watchdog treats every timeout as still active, so a timed-out or cancelled run can keep looping and leave the queued message blocked.
  - confidence: 0.87
- **[P3] Suppress transient watchdog probe errors:** `ui/src/ui/app-chat.ts:97-99`
  - body: The probe is a silent 50 ms background recovery check that retries automatically. Writing a thrown probe error into `host.lastError` can show a visible error banner even when the main gateway connection and run are otherwise healthy.
  - confidence: 0.82
