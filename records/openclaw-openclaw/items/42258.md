## Summary

Keep open: current main still lacks the Control UI inactivity recovery, and this PR remains a useful implementation candidate, but it should not merge as-is because it lacks real browser/gateway proof and mishandles terminal `agent.wait` timeout snapshots.

## What This Changes

The PR adds a Control UI chat watchdog that probes `agent.wait` after inactivity, clears stale run state, reloads history, flushes queued messages, and adds regression tests.

## Best Possible Solution

Land a rebased Control UI recovery path that treats authoritative `agent.wait` terminal metadata as finished, uses shared run-lifecycle cleanup, suppresses silent probe noise, preserves regression coverage, and includes redacted live browser/gateway proof.

## Reproduction Assessment

Yes, at source level. On current main, leave `chatRunId` non-null after a missed terminal chat event, submit a normal message, and the message queues while `flushChatQueue` returns because `isChatBusy` remains true.

## Solution Assessment

No, not as currently patched. A UI-side `agent.wait` recovery probe is the right narrow direction, but this implementation must handle terminal timeout metadata, avoid visible background probe errors, and be rebased onto current run-lifecycle cleanup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Treat terminal timeout snapshots as finished:** `ui/src/ui/app-chat.ts:84`
  - body: `agent.wait` can return an already-ended run as `status: "timeout"` with terminal metadata such as `endedAt`, `stopReason`, `livenessState`, or `yielded`. This branch treats every timeout as still active, so a timed-out or cancelled run can keep looping and leave the queued message blocked.
  - confidence: 0.9
- **[P3] Suppress transient watchdog probe errors:** `ui/src/ui/app-chat.ts:97-99`
  - body: The watchdog is a silent 50 ms background recovery probe that retries automatically. Writing a thrown probe error into `host.lastError` can show a visible error banner even when the main gateway connection and run are otherwise healthy.
  - confidence: 0.82
