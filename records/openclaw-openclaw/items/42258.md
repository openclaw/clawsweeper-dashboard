## Summary

Keep open: current main still lacks the stale-run recovery path, this PR is linked with closing intent to an open same-author bug report, and the watchdog approach remains useful but needs real behavior proof plus one cleanup before merge.

## What This Changes

The PR adds a Control UI chat-run watchdog that probes `agent.wait` after inactivity, clears stale run state, reloads chat history, flushes queued messages, and adds focused regression tests.

## Best Possible Solution

Rebase or replace the watchdog on current main, keep `agent.wait` as terminal-state confirmation, suppress transient probe errors from visible banners, add focused regressions, and land it with real browser/gateway proof.

## Reproduction Assessment

Yes, at source level. Leaving `chatRunId` non-null keeps `isChatBusy` true, normal sends enqueue, and `flushChatQueue` returns until a matching terminal event clears the run; no live browser/gateway reproduction was run in this read-only review.

## Solution Assessment

Mostly yes. A UI-side `agent.wait` watchdog is a narrow maintainable fix because it uses an existing backend contract without changing chat protocol or queue semantics, but transient watchdog probe errors should not be surfaced through `lastError`.
