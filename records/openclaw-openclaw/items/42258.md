Keep this PR open, but not mergeable as-is. Current main still has the stale `chatRunId` queue-blocking gap, and the watchdog direction is narrow, but the patch mishandles terminal `agent.wait` timeout snapshots, surfaces background probe failures as visible chat errors, and misses the required user-facing changelog entry.

## What This Changes

The PR adds a Control UI chat-run watchdog, activity tracking, lifecycle reset hooks, and regression tests across the chat app/controller/gateway UI paths.

## Best Possible Solution

Land a narrow Control UI recovery path that uses the existing `agent.wait` contract correctly: recover terminal snapshots including timeout-ended runs, keep wait-deadline timeouts active, suppress non-visible probe failures while retrying, preserve current queue semantics, and document the user-facing fix in the active changelog.

## Reproduction Assessment

Yes. The high-confidence source reproduction is to leave `chatRunId` set after a missed terminal chat event, queue a follow-up, and observe `flushChatQueue` returning while `isChatBusy` remains true; `agent.wait` can then distinguish an actually active run from a gateway-terminal snapshot.

## Solution Assessment

No, not yet. The client-side watchdog is the narrowest maintainable direction, but this implementation must distinguish wait-deadline `timeout` responses from terminal timeout snapshots and keep background probe failures out of visible chat error state.
