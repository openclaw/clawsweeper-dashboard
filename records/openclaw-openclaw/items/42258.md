## Summary

Keep open for repair: the PR addresses a valid Control UI stuck-chat bug, but as written it still leaves terminal timeout/abort snapshots stuck because it treats every `agent.wait` timeout response as an active run.

## What This Changes

The PR adds a Control UI chat-run watchdog that tracks inactivity, probes `agent.wait`, clears stale run state, reloads history, and flushes queued chat messages with new controller tests.

## Best Possible Solution

Land a client-side recovery path that distinguishes bare wait-deadline timeouts from terminal timeout/abort snapshots, clears stale UI state, flushes the queue, and records the user-facing fix in the changelog.

## Reproduction Assessment

Yes. The current-main source path is reproducible by inspection: queued messages do not flush while `chatRunId` remains set, and the Gateway already exposes `agent.wait` recovery for chat runs that finish without lifecycle events.

## Solution Assessment

No as-is. The watchdog direction is the narrowest maintainable fix, but it must handle terminal `timeout` snapshots separately from bare wait-deadline timeouts before it solves the linked timeout/abort cases.

## Review Findings

Full review comments:

- **[P2] Recover terminal timeout wait snapshots:** `ui/src/ui/app-chat.ts:84`
  - body: The watchdog treats every `agent.wait` timeout as still active, but Gateway also uses `status: "timeout"` for terminal aborted/timed-out snapshots when lifecycle metadata such as `endedAt` or `stopReason` is present. In that case this branch keeps `chatRunId` set and keeps the queue blocked, so the timeout/abort path from the linked reports remains stuck.
  - confidence: 0.91

A transient `agent.wait` or network failure can show a visible chat error even though the primary stream may still be healthy and the watchdog can retry; keep this diagnostic non-visible or only escalate after a real terminal failure.
  - confidence: 0.78
