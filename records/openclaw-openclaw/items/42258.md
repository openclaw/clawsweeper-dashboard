## Summary

Keep open: current main still has a source-reproducible stale busy latch, and this PR is a plausible fix path, but the branch is not merge-ready because it misses terminal timeout snapshots and lacks after-fix real browser/gateway proof.

## What This Changes

The PR adds a Control UI chat-run watchdog that probes `agent.wait` after inactivity, clears stale run state, reloads history, flushes queued messages, and adds focused tests.

## Best Possible Solution

Land a rebased watchdog or equivalent that treats authoritative `agent.wait` terminal snapshots as finished, keeps transient background probe failures out of visible errors, preserves regression coverage, and includes redacted real browser/gateway proof.

## Reproduction Assessment

Yes, at source level. On current main, a missed terminal chat event can leave `chatRunId` set; then `isChatBusy` stays true and `flushChatQueue` returns before dequeuing the next message.

## Solution Assessment

Mostly, but not as currently patched. A UI-side `agent.wait` watchdog is a narrow maintainable fix, but this branch must handle terminal timeout/cancel snapshots and avoid visible errors from transient background probe failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Handle terminal timeout wait snapshots:** `ui/src/ui/app-chat.ts:84`
  - body: `agent.wait` can return terminal timeout/cancel snapshots as `status: "timeout"` with metadata like `endedAt`, `stopReason`, `livenessState`, or `yielded`. This watchdog only recovers `"ok"`/`"error"`, so an already-ended run can keep looping and leave queued messages blocked.
  - confidence: 0.87
- **[P3] Suppress transient watchdog probe errors:** `ui/src/ui/app-chat.ts:97-99`
  - body: The watchdog probe runs silently in the background and retries, so assigning `host.lastError` when the short `agent.wait` request throws can show a visible error banner even though the primary connection may still be healthy.
  - confidence: 0.82
