## Summary

Keep this PR open. Current main still has the stale-busy Control UI shape, and this PR is the active implementation candidate for the linked bug, but the watchdog needs one correctness fix before merge because `agent.wait` can return a terminal `timeout` snapshot that the PR currently treats as still active.

## What This Changes

The PR branch adds a Control UI chat-run watchdog that records chat activity, probes `agent.wait` after idle time, clears stale run state, reloads history, flushes queued messages, and adds watchdog regression tests.

## Best Possible Solution

Keep the PR as the implementation candidate, but revise the watchdog so it recovers terminal timeout snapshots, keeps transient probe errors out of visible chat errors, acknowledges the added authenticated Gateway probe in the PR security notes, then rebase and validate the Control UI chat paths.

## Reproduction Assessment

Yes. The source path is high-confidence: set `chatRunId`, queue a normal message, miss the matching terminal chat event, and `flushChatQueue` returns while `isChatBusy` remains true; the existing backend `agent.wait` path can prove the run has already ended.

## Solution Assessment

No, not yet. A client-side `agent.wait` watchdog is the right narrow direction, but this implementation must distinguish wait-deadline `timeout` from terminal timeout snapshots and avoid surfacing transient probe failures as user-visible chat errors.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Recover terminal timeout snapshots:** `ui/src/ui/app-chat.ts:86`
  - body: `agent.wait` can return `status: "timeout"` for a terminal run snapshot, not only for an active wait that expired. Since the watchdog treats every timeout as still active and reschedules, the embedded timeout case can remain stuck forever even after the gateway has a terminal snapshot. Recover when the wait result is terminal, for example by checking terminal metadata such as `endedAt` rather than status alone.
  - confidence: 0.9
- **[P3] Do not surface transient watchdog probe errors:** `ui/src/ui/app-chat.ts:97-99`
  - body: This catch writes a silent background `agent.wait` probe failure into `host.lastError`, and the Control UI renders that field as a visible danger pill. A transient probe/network blip can therefore look like a chat failure while the run is still active; keep probe failures non-visible or track them separately while retrying.
  - confidence: 0.84
