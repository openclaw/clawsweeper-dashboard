## Summary

Keep open: current main still lacks the proposed block-reply hook, but this PR is not ready to merge because source inspection shows delivery-context and cancellation defects, real behavior proof is missing, and the gateway routing part has already landed elsewhere.

## What This Changes

The branch adds a public `before_block_reply` hook with trigger context through block-reply delivery, plus gateway hook session-key routing changes and tests.

## Best Possible Solution

Land a rebased hook-only implementation that preserves trigger and assistant-message context through coalescing, handles cancellation consistently on all block delivery paths, documents the public hook, and leaves already-merged gateway routing code out of the branch.

## Reproduction Assessment

Yes for the PR defects by source inspection: coalesced payloads drop context at `sendPayload(payload, undefined, ...)`, and the direct-send path marks content as sent before a hook cancellation can throw. I did not run tests because this was a read-only review.

## Solution Assessment

No. The hook direction is plausible, but the current patch is not the best implementation until it is rebased onto current main, preserves coalesced context, handles direct cancellation, documents the SDK surface, and removes the already-merged gateway routing overlap.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve trigger context through coalescer flushes:** `src/auto-reply/reply/block-reply-pipeline.ts:181`
  - body: `before_block_reply` depends on `context?.trigger`, but coalesced text is flushed with `sendPayload(payload, undefined, ...)`, so plugins cannot reliably distinguish `tool_start`, `text_end`, `message_end`, or `run_end` for the coalesced path. Store or merge the context with coalesced payloads and forward it on flush.
  - confidence: 0.91
- **[P2] Treat direct-send cancellation as non-delivery:** `src/auto-reply/reply/reply-delivery.ts:130`
  - body: The direct-send path records the block as sent before awaiting `onBlockReply`, but a `before_block_reply` cancellation now throws. Without a typed catch or delayed sent-key recording, final payload filtering can suppress content that the plugin cancelled before delivery.
  - confidence: 0.84
- **[P3] Document the public block-reply hook:** `src/plugins/types.ts:2125-2136`
  - body: This adds a public hook event/result contract, but the plugin docs do not describe `before_block_reply`, trigger values, or text/cancel behavior. Add the docs with the SDK surface so plugin authors can use the hook safely.
  - confidence: 0.82
