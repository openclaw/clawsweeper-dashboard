## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested `before_block_reply` hook, while the PR head has correctness gaps in trigger propagation and cancellation handling, and its gateway-session-key portion has already shipped separately.

## What This Changes

The PR adds a `before_block_reply` plugin hook with block-flush trigger metadata through the block-reply streaming path, plus gateway hook session-key rebinding changes and tests.

## Best Possible Solution

Land a rebased hook-only implementation on current main that preserves trigger and assistant-message context through coalescing, treats cancellation as non-delivery on every block path, documents the public API, and omits the already-shipped gateway routing change.

## Reproduction Assessment

Yes. A focused unit test can create `createBlockReplyPipeline` with coalescing enabled, enqueue text with a `tool_start` trigger, force flush, and assert the delivered context still has that trigger; a second direct-send test can use a canceling hook and assert delivery suppresses cleanly instead of throwing.

## Solution Assessment

No. The hook direction is plausible, but this patch is not the best implementation because it loses trigger metadata on the coalesced path, lets direct cancellation escape, and carries gateway work that current main already shipped separately.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve trigger context through coalescer flushes:** `src/auto-reply/reply/block-reply-pipeline.ts:179-181`
  - body: The hook reads `context?.trigger`, but coalesced text blocks are flushed without forwarding the context. With `blockStreamingCoalesce` enabled, `before_block_reply` receives no trigger for normal streamed blocks, so plugins cannot reliably distinguish tool-start narration from text or final flushes. Store or merge the context alongside coalesced payloads and forward it on flush.
  - confidence: 0.91
- **[P2] Handle cancellation on direct block sends:** `src/auto-reply/reply/reply-delivery.ts:130-136`
  - body: `BlockReplyCancelledError` is only recognized by the pipeline catch handler. Direct block delivery still awaits `params.onBlockReply(...)` here, so a plugin returning `{ cancel: true }` can throw through the media or no-pipeline path instead of suppressing delivery cleanly. Catch the typed cancellation or model direct sends with the same delivered/not-delivered result.
  - confidence: 0.84
- **[P3] Document the public hook before exposing it:** `src/plugins/types.ts:2125-2136`
  - body: This adds a new public plugin hook type, but the plugin hook docs do not describe `before_block_reply`, its trigger values, or cancel/text behavior, and the changelog entry only covers gateway session routing. Plugin authors need docs and a user-facing feature changelog entry to use the API correctly.
  - confidence: 0.82
