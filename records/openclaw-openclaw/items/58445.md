## Summary

Keep this PR open because current main still does not expose a `before_block_reply` plugin hook, so the proposed feature is not obsolete. The PR is not mergeable as-is: the PR head still drops trigger context on the coalesced block-reply path, which undermines the stated Slack/tool-start interception use case, and the branch also carries gateway session-key routing work that current main already has.

## What This Changes

The PR adds a `before_block_reply` plugin hook with block-reply trigger metadata through the streaming delivery path, plus gateway hook session-key rebinding changes, tests, and a changelog entry.

## Best Possible Solution

Keep the PR open and update it or replace it with a narrow branch that adds the hook on current main, preserves trigger and assistant-message context through coalescing, documents the public plugin API, and drops the already-landed gateway session-routing changes from the feature branch.

## Reproduction Assessment

Yes. A focused unit test can create a `createBlockReplyPipeline` with coalescing enabled, enqueue a text payload with `{ trigger: "tool_start" }`, force flush, and assert the delivered context still contains that trigger; on the PR head, `onFlush` calls `sendPayload(payload, undefined, ...)`, so that assertion would fail.

## Solution Assessment

No. The hook direction is plausible, but the current PR is not the best implementation because it loses trigger metadata on the coalesced path and mixes in gateway work that is already on main; the safer fix is a current-main rebase with context-preserving coalescer tests and docs/changelog alignment.
