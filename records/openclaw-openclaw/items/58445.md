## Summary

Keep open: current main still lacks the requested block-reply hook and trigger contract, but this PR is not ready to merge because it drops trigger context through coalescing, mishandles cancellation on direct sends, lacks public hook docs, and has no real Slack/runtime proof. The gateway session-key portion is already merged separately, so a rebase should keep the useful hook work and drop the overlapping gateway changes.

## What This Changes

Adds a public `before_block_reply` plugin hook with block-reply trigger context threaded through agent/reply delivery, plus overlapping gateway hook session-key rebinding code and tests.

## Best Possible Solution

Land a rebased hook-only implementation on current main that preserves trigger and assistant-message context through coalescing, handles cancellation consistently on every delivery path, documents the public API, and omits the already-merged gateway routing fix.

## Reproduction Assessment

Yes for the PR defects by source inspection: coalesced payloads drop context at `sendPayload(payload, undefined, ...)`, and direct delivery has no typed cancellation catch. I did not run tests because this was a read-only review.

## Solution Assessment

No. The hook direction is plausible, but the patch is not the best implementation until it is rebased onto current hook contracts, preserves context through coalescing, handles direct cancellation, and documents the new SDK surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve trigger context through coalescer flushes:** `src/auto-reply/reply/block-reply-pipeline.ts:181`
  - body: `before_block_reply` depends on `context?.trigger`, but coalescer `onFlush` calls `sendPayload(payload, undefined, ...)`, so coalesced text loses `tool_start`, `text_end`, `message_end`, or `run_end` before plugins can inspect it. Store or merge the context with coalesced payloads and forward it on flush.
  - confidence: 0.91
- **[P2] Treat direct-send cancellation as non-delivery:** `src/auto-reply/reply/reply-delivery.ts:130`
  - body: The direct-send path marks the content as directly sent and then awaits `params.onBlockReply(..., context)`, but only the pipeline catches `BlockReplyCancelledError`. A plugin cancellation can escape this path and leave final-reply suppression state behind; catch the typed cancellation or route direct sends through the same delivery result model.
  - confidence: 0.84
- **[P3] Document the public block-reply hook:** `src/plugins/types.ts:2125-2136`
  - body: This adds a new public hook event/result contract, but the plugin hook docs do not describe `before_block_reply`, its trigger values, or its text/cancel behavior. Add the public docs before exposing the SDK surface so plugin authors can use it correctly.
  - confidence: 0.82
