## Summary

Keep this PR open: current main still lacks `before_block_reply`, and the proposed hook addresses a real block-streaming extension gap, but the patch is not merge-ready because it drops trigger context through coalescing, lets direct-send cancellation escape as an error, lacks public docs/changelog coverage for the hook, and includes gateway session-routing work already present on main.

## What This Changes

The PR adds a `before_block_reply` plugin hook with trigger metadata through the block-reply streaming path and also changes gateway hook session-key rebinding.

## Best Possible Solution

Land a rebased hook-only implementation on current main that preserves trigger and assistant-message context through coalescing, treats cancellation as non-delivery on every block path, documents the public API, and leaves the already-shipped gateway routing fix out.

## Reproduction Assessment

Yes. A focused unit test can create `createBlockReplyPipeline` with coalescing enabled, enqueue text with `{ trigger: "tool_start" }`, force flush, and assert the delivered context still has that trigger; on the PR head, `onFlush` calls `sendPayload(payload, undefined, ...)`, so it fails. A second direct-send test can cancel from the hook with no pipeline and assert the handler does not throw.

## Solution Assessment

No. The hook direction is plausible, but this patch is not the best implementation because it loses trigger metadata on the coalesced path, lets direct cancellation escape, and mixes in already-landed gateway work; the safer path is a current-main rebase with context-preserving tests and docs/changelog alignment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve trigger context through coalescer flushes:** `src/auto-reply/reply/block-reply-pipeline.ts:179-181`
  - body: The hook reads `context?.trigger`, but coalesced text blocks are flushed with `sendPayload(payload, undefined, ...)`. With `blockStreamingCoalesce` enabled, `before_block_reply` receives no trigger for normal streamed blocks, so plugins cannot reliably distinguish tool-start narration from text or final flushes. Store or merge the context alongside coalesced payloads and forward it on flush.
  - confidence: 0.91
- **[P2] Handle cancellation on direct block sends:** `src/auto-reply/reply/reply-delivery.ts:130-136`
  - body: `BlockReplyCancelledError` is only recognized by the pipeline catch handler. Direct block delivery still awaits `params.onBlockReply(...)` here, so a plugin returning `{ cancel: true }` can throw through the media/no-pipeline path instead of suppressing delivery cleanly. Catch the typed cancellation or model direct sends with the same delivered/not-delivered result.
  - confidence: 0.84
- **[P3] Document the public hook before exposing it:** `src/plugins/types.ts:2125-2136`
  - body: This adds a new public plugin hook type, but the plugin hook docs do not describe `before_block_reply`, its trigger values, or cancel/text behavior, and the changelog entry only describes gateway session routing. Plugin authors need the public docs and feature changelog entry to use the new API correctly.
  - confidence: 0.82
