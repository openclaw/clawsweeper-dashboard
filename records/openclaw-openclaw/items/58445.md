## Summary

Keep this PR open. Current main still does not expose the requested block-reply hook, but the branch needs changes because the proposed hook loses trigger metadata on the coalesced path, cancellation is not handled cleanly on direct sends, the public hook is undocumented, and the PR has no after-fix real behavior proof.

## What This Changes

The PR adds a public `before_block_reply` plugin hook with block flush trigger metadata through agent/block-reply delivery and also includes gateway hook session-key rebinding changes/tests.

## Best Possible Solution

Land a rebased hook-only implementation on current main that preserves trigger and assistant-message context through coalescing, treats cancellation as non-delivery on every block path, documents the public API, and omits the already-merged gateway routing fix.

## Reproduction Assessment

Yes for the PR defects: a focused unit test can coalesce an enqueued block with `{ trigger: "tool_start" }` and observe the hook receives no trigger, and a direct-send cancellation test can show the typed cancellation escaping. I did not run those tests because this was a read-only review.

## Solution Assessment

No. The hook direction is plausible, but this patch is not the best implementation until it is rebased onto current main, preserves context through coalescing, handles cancellation consistently, and documents the new SDK surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve trigger context through coalescer flushes:** `src/auto-reply/reply/block-reply-pipeline.ts:179`
  - body: `before_block_reply` depends on `context?.trigger`, but coalesced text blocks are flushed with `sendPayload(payload, undefined, ...)`. With `blockStreamingCoalesce` enabled, plugins will not reliably know whether a coalesced block came from `tool_start`, `text_end`, `message_end`, or `run_end`; store/merge the context with coalesced payloads and forward it on flush.
  - confidence: 0.91
- **[P2] Treat direct-send cancellation as non-delivery:** `src/auto-reply/reply/reply-delivery.ts:130`
  - body: `BlockReplyCancelledError` is only handled by the pipeline catch path. Direct block delivery still awaits `params.onBlockReply(..., context)`, so a plugin returning `{ cancel: true }` can reject through the no-pipeline/media path instead of being suppressed cleanly; catch the typed cancellation or route direct sends through the same delivered/not-delivered result model.
  - confidence: 0.84
- **[P3] Document the public block-reply hook:** `src/plugins/types.ts:2122-2136`
  - body: This adds a new public plugin hook type and result contract, but the plugin hook docs do not describe `before_block_reply`, its trigger values, or its text/cancel behavior. Add the public docs before exposing the SDK surface so plugin authors can use it correctly.
  - confidence: 0.82
