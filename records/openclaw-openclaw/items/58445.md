## Summary

This PR should stay open because current main still does not expose the proposed `before_block_reply` hook, but the branch is not ready to merge: it introduces a new public plugin API, still has source-visible delivery defects, lacks real Slack/runtime proof, and overlaps gateway session-key work already merged on main.

## What This Changes

The PR adds a public `before_block_reply` plugin hook with block-reply trigger context through agent/reply delivery, plus gateway hook session-key rebinding changes and tests.

## Best Possible Solution

Land a rebased hook-focused implementation only if maintainers want this public plugin API, with trigger context preserved through coalescing, cancellation handled consistently on every delivery path, SDK docs updated, and the already-main gateway routing diff omitted.

## Reproduction Assessment

Yes for the PR defects by source inspection: the proposed coalescer flush drops trigger context, and direct-send cancellation can leave sent-key state behind. No high-confidence real reproduction was provided for the underlying Slack fragmentation fix after the PR changes.

## Solution Assessment

No. The hook direction is plausible, but this implementation is not the best merge path until it is rebased, narrowed to the hook API, preserves trigger context through coalescing, treats cancellation as non-delivery everywhere, and documents the public contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve trigger context through coalescer flushes:** `src/auto-reply/reply/block-reply-pipeline.ts:181`
  - body: `before_block_reply` depends on `context?.trigger`, but coalescer flushes forward `undefined`, so coalesced `tool_start`, `text_end`, `message_end`, and `run_end` blocks reach the hook without the trigger that plugins need. Store or merge the context with coalesced payloads and pass it through on flush.
  - confidence: 0.91
- **[P2] Treat direct-send cancellation as non-delivery:** `src/auto-reply/reply/reply-delivery.ts:128-129`
  - body: The direct-send path records the block content as sent before awaiting `onBlockReply`, and it does not catch `BlockReplyCancelledError`. A plugin cancellation can escape this path while leaving final-reply suppression state behind, so handle typed cancellation or route direct sends through the same delivery result model as the pipeline.
  - confidence: 0.84
- **[P3] Document the public block-reply hook:** `src/plugins/types.ts:2122-2134`
  - body: This adds a public hook event/result contract, but plugin docs do not describe `before_block_reply`, trigger values, text mutation, or cancellation behavior. Add the docs/API surface before exposing the hook so plugin authors have a stable contract.
  - confidence: 0.82
