## Summary

Keep open. Current main does not implement the new `before_block_reply` hook, but this PR still needs real Slack/runtime proof, public hook API review, and fixes for trigger preservation and cancellation semantics before it is safe to merge.

## What This Changes

Adds a public `before_block_reply` plugin hook with block-reply trigger context threaded through agent/reply delivery, plus gateway hook session-key routing changes and tests.

## Best Possible Solution

Land a rebased hook-only implementation that preserves trigger and assistant-message context through coalescing, handles cancellation consistently on every delivery path, documents the SDK contract, and omits the already-main gateway routing change.

## Reproduction Assessment

Yes for the PR defects by source inspection: coalesced payloads drop trigger context at `sendPayload(payload, undefined, ...)`, and direct sends have no typed cancellation handling. The underlying Slack fragmentation claim still lacks real after-fix runtime proof.

## Solution Assessment

No. The hook direction is plausible, but this patch is not the best implementation until it is rebased, preserves trigger context through coalescing, treats cancellation as non-delivery everywhere, and documents the public API.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve trigger context through coalescer flushes:** `src/auto-reply/reply/block-reply-pipeline.ts:181`
  - body: `before_block_reply` depends on `context?.trigger`, but coalescer flushes forward `undefined`, so coalesced `tool_start`, `text_end`, `message_end`, and `run_end` blocks reach the hook without the trigger that plugins need. Store or merge the context with coalesced payloads and pass it through on flush.
  - confidence: 0.91
- **[P2] Treat direct-send cancellation as non-delivery:** `src/auto-reply/reply/reply-delivery.ts:129-130`
  - body: The direct-send path records the block content as sent before awaiting `onBlockReply`, and it does not catch `BlockReplyCancelledError`. A plugin cancellation can escape this path while leaving final-reply suppression state behind, so handle typed cancellation or route direct sends through the same delivery result model as the pipeline.
  - confidence: 0.84
- **[P3] Document the public block-reply hook:** `src/plugins/types.ts:2125-2136`
  - body: This adds a public hook event/result contract, but plugin docs do not describe `before_block_reply`, trigger values, text mutation, or cancellation behavior. Add the docs/API surface before exposing the hook so plugin authors have a stable contract.
  - confidence: 0.82
