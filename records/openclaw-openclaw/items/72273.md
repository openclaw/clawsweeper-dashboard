## Summary

Keep open. Current main still hardcodes the Discord startup READY timeout at 15s and runtime READY timeout at 30s, and the only exposed Discord timeout setting is the separate `/gateway/bot` metadata lookup timeout. The report is fresh, source-backed, and narrow enough for a Discord plugin repair rather than closure.

## What This Changes

The issue requests configurable Discord gateway startup and runtime READY timeouts because the fixed 15s startup budget is too short for some multi-account Discord deployments.

## Best Possible Solution

Add Discord plugin-owned, account-inheritable startup READY and runtime READY timeout settings under `channels.discord` and `channels.discord.accounts.<accountId>`, preserve the current 15s/30s defaults, allow env fallbacks only when config is unset, and update schema/types/UI hints/generated metadata/docs/tests/changelog together.

## Reproduction Assessment

Yes. A high-confidence focused reproduction exists in the fake-timer lifecycle harness: keep `gateway.isConnected` false past 15s for startup, and emit `Gateway websocket opened` without READY for 30s at runtime; the multi-account stagger path is visible in `extensions/discord/src/channel.ts`.

## Solution Assessment

Yes. The narrow maintainable fix is to thread Discord account config into `runDiscordGatewayLifecycle`; editing bundled output or adding a core-owned `gateway.discord.*` key would cross the plugin ownership boundary and leave public config surfaces inconsistent.
