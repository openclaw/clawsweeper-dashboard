## Summary

Keep open. Current main still hardcodes the Discord gateway startup READY wait at 15s and runtime READY wait at 30s, while the public Discord config/schema/docs only expose the separate `/gateway/bot` metadata timeout. The issue is fresh, source-backed, narrowly scoped to the Discord plugin, and has a clear repair path.

## What This Changes

Request configurable Discord gateway startup and runtime READY timeouts because fixed 15s/30s budgets can expire in multi-account Discord setups with staggered provider startup.

## Best Possible Solution

Add Discord plugin-owned, account-inheritable READY timeout settings under `channels.discord` and `channels.discord.accounts.<id>`, with optional env fallbacks only when config is unset. Thread the resolved values from account config into `monitorDiscordProvider` and `runDiscordGatewayLifecycle`, and update schema/types/UI hints/docs/tests/changelog in the same repair.

## Reproduction Assessment

Yes. The actionable behavior is reproducible with the existing fake-timer lifecycle harness by keeping `gateway.isConnected` false past 15s for startup and by emitting `Gateway websocket opened` without READY for 30s at runtime; the multi-account stagger path is also deterministic from account index times the startup stagger.

## Solution Assessment

Yes, with a scope correction. Making these Discord plugin account settings configurable is the narrow maintainable fix; editing bundled output or adding a core `gateway.discord.*` key would cross the owner boundary and leave generated docs/schema out of sync.
