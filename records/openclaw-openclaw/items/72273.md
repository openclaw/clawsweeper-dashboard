## Summary

Keep open. Current main still hardcodes the Discord gateway startup READY wait at 15s and runtime READY wait at 30s, while multi-account Discord startup is intentionally staggered and only the unrelated `/gateway/bot` metadata timeout is configurable today.

## What This Changes

The issue requests configurable Discord gateway startup and runtime READY timeouts because fixed 15s/30s values can be too short for staggered multi-account Discord startup.

## Best Possible Solution

Add Discord plugin-owned, account-inheritable READY timeout settings under `channels.discord` and `channels.discord.accounts.<id>`, resolve optional env fallbacks only when config is unset, and thread the resolved values through `monitorDiscordProvider` into `runDiscordGatewayLifecycle` with schema/types/docs/tests kept in sync.

## Reproduction Assessment

Yes. Existing fake-timer lifecycle tests can reproduce the startup restart by keeping `gateway.isConnected` false past 15s and the runtime force-stop by emitting `Gateway websocket opened` without READY for 30s; the multi-account delay is covered by the 10s-per-account startup stagger path.

## Solution Assessment

Yes, with scope correction. The maintainable fix belongs in the Discord plugin account config/env surface and lifecycle parameters; editing bundled output or adding a core `gateway.discord.*` setting would cross the repository ownership boundary.
