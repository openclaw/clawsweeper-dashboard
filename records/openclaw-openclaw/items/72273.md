## Summary

Keep open. Current main still hardcodes the Discord gateway startup READY wait at 15s and runtime READY wait at 30s, while Discord multi-account startup is intentionally staggered and only the unrelated `/gateway/bot` metadata timeout is configurable.

## What This Changes

The issue requests configurable Discord gateway startup and runtime READY timeouts so multi-account startup staggering does not trigger fixed-timeout restart loops.

## Best Possible Solution

Add Discord plugin-owned READY timeout settings that are inheritable from `channels.discord` to `channels.discord.accounts.<id>`, with env fallbacks only when config is unset, then thread the resolved values into the lifecycle supervisor and keep schema, generated config docs, Discord docs, tests, and changelog aligned.

## Reproduction Assessment

Yes. The existing fake-timer lifecycle tests can reproduce the startup restart by keeping `gateway.isConnected` false past 15s and the runtime force-stop by emitting `Gateway websocket opened` without READY for 30s; the multi-account delay path is visible in the 10s-per-account startup stagger code.

## Solution Assessment

Yes, with scope correction. The maintainable fix belongs in the Discord plugin account/config/env surface and lifecycle parameters; editing bundled output or adding a core `gateway.discord.*` setting would cross the repository ownership boundary.
