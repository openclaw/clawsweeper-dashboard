## Summary

Keep open. Current main still has the reported Discord READY timeout behavior: startup readiness is fixed at 15 seconds, runtime READY after websocket-open is fixed at 30 seconds, and the exposed Discord config surface only covers `/gateway/bot` metadata lookup timeout. The report is narrow, reproducible from existing fake-timer lifecycle tests, and fits a Discord plugin-owned repair.

## What This Changes

Make Discord gateway startup READY and runtime READY timeouts configurable so multi-account startup staggering does not trigger premature gateway restarts.

## Best Possible Solution

Add Discord plugin-owned, account-inheritable READY timeout settings under `channels.discord` and `channels.discord.accounts.<accountId>`, with env fallbacks only when config is unset. Thread the resolved values into `monitorDiscordProvider` and `runDiscordGatewayLifecycle`, document the knobs, add schema/type/UI metadata, and update lifecycle tests for default and configured behavior.

## Reproduction Assessment

Yes. Existing fake-timer lifecycle tests can reproduce the startup restart by keeping `gateway.isConnected` false past 15 seconds, and the runtime force-stop by emitting `Gateway websocket opened` without READY for 30 seconds; the multi-account risk follows from the 10-second-per-account startup stagger.

## Solution Assessment

Yes, with a scope correction. The maintainable fix belongs in the Discord plugin/account config surface rather than a core `gateway.discord.*` namespace or bundled-output patch, because the behavior and stagger are Discord-owned.
