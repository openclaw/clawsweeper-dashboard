## Summary

Keep open. Current main still hardcodes the Discord startup READY wait at 15s and runtime READY wait at 30s, while the only adjacent configurable Discord timeout is for `/gateway/bot` metadata lookup; the reported multi-account stagger can plausibly outlive the startup budget.

## What This Changes

Request to make the Discord gateway startup and runtime READY timeout budgets configurable because the fixed 15s startup timeout can expire before staggered multi-account Discord providers finish starting.

## Best Possible Solution

Add Discord plugin-owned, account-inheritable READY timeout settings under `channels.discord` and `channels.discord.accounts.<id>`, with env fallbacks only when config is unset, then thread the resolved values through `monitorDiscordProvider` into `runDiscordGatewayLifecycle`; update schema/types/config metadata/docs/tests and consider a default that accounts for configured startup stagger rather than a fixed 15s.

## Reproduction Assessment

Yes. Existing fake-timer lifecycle tests can reproduce the startup restart by keeping `gateway.isConnected` false past 15s and the runtime force-stop by emitting `Gateway websocket opened` without READY for 30s; the multi-account part follows from the 10s-per-account startup stagger.

## Solution Assessment

Yes, with a scope correction. The narrow maintainable fix belongs in the Discord plugin account config/env surface and lifecycle parameters; editing bundled output or adding a core `gateway.discord.*` setting would cross the repo’s ownership boundary.
