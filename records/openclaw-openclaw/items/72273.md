Keep open. Current main still hardcodes the Discord startup/runtime READY waits and only exposes the separate `/gateway/bot` metadata timeout, so this review confirms the earlier ClawSweeper keep-open result against current main `e0cc374b07c160b430a1078efca7ecbc6b975f3a`. ## What This Changes

The issue asks for Discord gateway startup READY and runtime READY timeouts to become configurable because the fixed 15s startup wait can restart multi-account setups before staggered accounts finish connecting.

## Best Possible Solution

Add Discord plugin-owned, account-inheritable READY timeout settings under `channels.discord` and `channels.discord.accounts.<accountId>`, preserve the current 15s/30s defaults, allow env fallbacks only when config is unset, and update schema/types/UI hints/generated metadata/docs/tests/changelog together.

## Reproduction Assessment

Yes. The existing fake-timer lifecycle harness can reproduce the observable behavior by keeping `gateway.isConnected` false past 15s for startup and by emitting `Gateway websocket opened` without READY for 30s at runtime; the multi-account path follows from the deterministic 10s-per-account startup stagger.

## Solution Assessment

Yes. The best repair is a narrow Discord plugin config surface threaded into `runDiscordGatewayLifecycle`; editing bundled output or adding a core-owned `gateway.discord.*` key would cross the owner boundary and leave schema/docs surfaces inconsistent.
