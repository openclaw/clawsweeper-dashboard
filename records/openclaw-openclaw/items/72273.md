## Summary

Keep open. Current main still hardcodes the Discord startup READY wait at 15s and runtime READY wait at 30s, while multi-account Discord startup is explicitly staggered by 10 seconds per enabled account; the requested configurability remains a valid narrow Discord plugin repair.

## What This Changes

Request to make Discord startup and runtime gateway READY timeouts configurable instead of fixed 15s/30s budgets so staggered multi-account setups do not enter restart loops.

## Best Possible Solution

Add Discord plugin-owned, account-inheritable READY timeout settings under `channels.discord` and `channels.discord.accounts.<accountId>`, with env fallbacks only when config is unset, then pass the resolved values into the lifecycle and update schema/types/config metadata/docs/tests/changelog.

## Reproduction Assessment

Yes. Existing fake-timer lifecycle tests can reproduce both paths by keeping `gateway.isConnected` false past 15s for startup and emitting `Gateway websocket opened` without READY for 30s at runtime; the 10s-per-account startup stagger makes the reported 9-account setup plausibly exceed the fixed startup budget.

## Solution Assessment

Yes, with a scope correction. The narrow maintainable fix belongs in the Discord plugin config/env surface and lifecycle parameters; editing bundled release output or adding a core `gateway.discord.*` setting would cross the extension ownership boundary.
