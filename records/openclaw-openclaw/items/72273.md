## Summary

Keep open. Current main still hardcodes the Discord gateway startup READY wait at 15s and runtime READY wait at 30s, while multi-account Discord startup is intentionally staggered and only the unrelated `/gateway/bot` metadata timeout is configurable today. This remains a narrow, valid Discord plugin repair with a clear config/docs/test path and no matching open implementation PR found in the available context or read-only GitHub API search.

## What This Changes

Request to make Discord gateway startup/runtime READY timeouts configurable for multi-account setups where staggered provider startup can outlast the hardcoded waits.

## Best Possible Solution

Add Discord plugin-owned, account-inheritable READY timeout settings under `channels.discord` and `channels.discord.accounts.<id>`, resolve optional env fallbacks only when config is unset, and thread the resolved values through `monitorDiscordProvider` into `runDiscordGatewayLifecycle` with schema, types, generated config metadata, docs, tests, and changelog kept in sync.

## Reproduction Assessment

Yes. Existing fake-timer lifecycle tests can reproduce the startup restart by keeping `gateway.isConnected` false beyond 15s, and the runtime force-stop by emitting `Gateway websocket opened` without READY for 30s; the multi-account delay path is visible in the 10s-per-account startup stagger code.

## Solution Assessment

Yes, with scope correction. The maintainable fix belongs in the Discord plugin account config/env surface and lifecycle parameters, not in bundled output or a core `gateway.discord.*` setting that crosses the plugin ownership boundary.
