## Summary

Keep open. Current main still has fixed Discord gateway startup and runtime READY waits, while the public Discord config/schema/docs expose only the separate `/gateway/bot` metadata timeout. The report is fresh, source-backed, and narrow enough for a Discord plugin repair.

## What This Changes

Request to make the Discord gateway startup READY and runtime READY timeouts configurable because the current 15s and 30s fixed waits are too short for some multi-account setups.

## Best Possible Solution

Add Discord plugin-owned, account-inheritable READY timeout settings under `channels.discord` and `channels.discord.accounts.<id>`, with optional env fallbacks only when config is unset. Thread the resolved values into the lifecycle waits and keep core `gateway.discord.*` config out of the repair.

## Reproduction Assessment

Yes. The issue has a high-confidence code-level reproduction using the existing fake-timer lifecycle harness: keep `gateway.isConnected` false past 15s for startup, and emit `Gateway websocket opened` without READY for 30s at runtime.

## Solution Assessment

Yes. The narrow maintainable solution is to make the Discord plugin lifecycle waits configurable through the existing account config surface, plus docs/schema/tests; editing bundled output or adding a core-owned Discord key would be the wrong boundary.
