## Summary

The PR should stay open: current `main` and latest release still emit a blank line before the zsh `#compdef` directive, and this branch is the only open focused fix for that autoload requirement. The patch itself looks correct, but it remains non-merge-ready until the external contributor provides inspectable real zsh behavior proof and refreshes the branch/checks.

## What This Changes

The PR removes the leading newline from the generated zsh completion template so `#compdef` starts at byte/line 1 of `openclaw completion --shell zsh` output.

## Best Possible Solution

Land this focused template fix after the contributor adds redacted real zsh fpath/compinit proof and refreshes the branch so required checks run on the current base.

## Reproduction Assessment

Yes by source inspection: current `main` and `v2026.6.10` put a blank line before `#compdef`, while zsh `compinit` reads the first line of fpath completion files. I did not run an executed zsh fpath repro in this read-only review.

## Solution Assessment

Yes. Starting the generated zsh script directly with `#compdef` is the narrowest maintainable fix for fpath autoloading; changing profile sourcing or compinit deferral does not correct the generated file directive placement.
