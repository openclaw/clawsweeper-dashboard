## Summary

Keep open. Current main still emits zsh completion output with a leading blank line before `#compdef`, while this PR makes the directive the first character of the generated script. The change matches the zsh `compinit` autoload contract and has green check runs, but it still needs the required user-facing changelog entry before normal merge cleanup.

## What This Changes

The PR removes the leading newline before the generated zsh `#compdef` directive in `src/cli/completion-cli.ts` so fpath autoload sees it on line 1.

## Best Possible Solution

Land the one-line zsh completion template fix with a single-line Unreleased Fixes changelog entry crediting `@gigamonster256`; adding a focused first-line regression test would make the fix more durable without broadening the runtime change.

## Reproduction Assessment

Yes. Current main source inspection proves zsh output starts with a blank line, and the upstream zsh docs explain why fpath autoload ignores completion files whose first line does not start with `#compdef`; the PR author also reports `compinit -D` verification.

## Solution Assessment

Yes. Moving `#compdef` to the first character of the generated script is the narrowest maintainable fix for zsh fpath autoload compatibility while preserving sourced completion behavior; the safer completion path is to add release-note coverage before merge.
