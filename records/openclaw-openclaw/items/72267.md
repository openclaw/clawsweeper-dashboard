## Summary

Current `main` and the latest release still generate zsh completion output with a blank first line before `#compdef`, and zsh `compinit` only treats fpath completion files specially when the first line starts with a completion tag. The patch is the right narrow shape, but the external PR still needs inspectable real behavior proof before merge.

## What This Changes

The PR removes the leading newline before the generated zsh `#compdef` directive so the directive becomes the first line of `openclaw completion --shell zsh` output.

## Best Possible Solution

Land the narrow zsh generator fix after the contributor adds redacted terminal output, a terminal screenshot, or a short recording showing the patched completion file loaded from zsh `fpath` with `compinit`, then let required checks gate the exact head.

## Reproduction Assessment

Yes, source-level: current `main` and `v2026.6.6` emit a blank first line before `#compdef`, while zsh `compinit` uses the first line of fpath completion files to identify completion tags. I did not run a live zsh repro because `zsh` is not installed in this review environment.

## Solution Assessment

Yes, the proposed one-line template change is the narrowest maintainable fix in the zsh completion generator. Bash, fish, and PowerShell completion generators do not share the `#compdef` first-line invariant, so this does not need a broader cross-shell change.
