## Summary

Keep open. The PR is a valid narrow fix because current `main` still emits generated zsh completion text with a blank first line before `#compdef`, and zsh `compinit` only treats fpath-loaded files specially when the first line starts with a completion tag. The runtime diff is correct and security-neutral, but the branch still needs the required changelog entry for a user-visible CLI fix.

## What This Changes

The PR removes the leading newline before the generated zsh `#compdef` directive in `src/cli/completion-cli.ts` so the directive is the first line of the completion script.

## Best Possible Solution

Land the one-line zsh template fix with a single-line active-version `### Fixes` changelog entry crediting `@gigamonster256`; a focused `startsWith("#compdef openclaw")` regression test would make the behavior more durable without broadening the change.

## Reproduction Assessment

Yes. Current-main source inspection plus the upstream zsh `compinit` contract give a high-confidence reproduction path for fpath-loaded zsh completion files; local live zsh execution was not available because zsh is not installed in this checkout environment.

## Solution Assessment

Yes. Moving `#compdef` to the first character of the generated zsh script is the narrowest maintainable fix for fpath autoload compatibility, while preserving sourced completion behavior.
