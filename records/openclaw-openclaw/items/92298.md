## Summary

The reported cross-account pollution is plausible, but this patch is not merge-ready because it changes explicit and existing CODEX_HOME semantics, partitions broad persisted Codex state without an upgrade path, and has no real two-account after-fix proof.

## What This Changes

The PR derives a hashed per-auth-profile CODEX_HOME, updates bridge tests for the new paths, and adds token-usage and turn-payload debug logging.

## Best Possible Solution

Preserve an explicitly supplied CODEX_HOME exactly, then isolate only the upstream state proven to be account-owned or add a deliberate migration/seeding design for complete per-profile homes, with fresh-install, upgrade, and real two-account proof.

## Reproduction Assessment

Partially yes: current source reproduces the shared-home prerequisite and the linked issue gives a concrete two-account comparison on v2026.6.5, but this review did not reproduce or verify the claimed cached-token reduction after the patch.

## Solution Assessment

No. Per-account isolation is directionally plausible, but applying it to an explicit home and every CODEX_HOME-owned surface is broader than the proven defect and lacks upgrade handling.
