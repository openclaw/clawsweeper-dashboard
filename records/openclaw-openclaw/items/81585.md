## Summary

Keep open. This PR is plausible source work for https://github.com/openclaw/openclaw/issues/81514, but the current branch is not mergeable because it can clear a trailing `Exec failed` warning for the same assistant-success shape that current main intentionally surfaces as an error.

## What This Changes

The PR updates isolated cron payload outcome classification to treat trailing exec warnings with final assistant-visible text and earlier successful payloads as nonfatal, with regression tests for recovered exec and trailing-error boundaries.

## Best Possible Solution

Define a concrete recovery signal for isolated cron tool failures, then update `resolveCronPayloadOutcome` and tests so recovered fallbacks are order-agnostic without hiding claimed-success exec failures.

## Reproduction Assessment

Yes. Source inspection shows the PR predicate would treat `[assistant success text, Exec failed]` as nonfatal when `finalAssistantVisibleText` and a prior success payload exist, while current main intentionally emits that exec warning for assistant success claims.

## Solution Assessment

No. The PR is closer than the initial broad predicate, but final assistant text alone is still too weak; it should require an explicit failure acknowledgement, validated recovery sentinel, or maintainer-approved equivalent before suppressing the exec warning.
