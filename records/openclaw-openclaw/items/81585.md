## Summary

Keep open: this PR addresses a valid cron status bug, but the latest diff still clears fatal status for an assistant success claim followed by a trailing exec failure, which conflicts with the current payload contract that intentionally surfaces those exec failures.

## What This Changes

The PR changes isolated cron payload classification so a trailing exec warning can be treated as recovered when final assistant-visible text exists, and adds helper regression coverage for recovered and fatal trailing-error cases.

## Best Possible Solution

Define an explicit recovered-tool signal for isolated cron runs, such as a failure acknowledgement or validated sentinel, then update `resolveCronPayloadOutcome` and regression tests while keeping claimed-success exec failures fatal.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the PR predicate treats `[assistant success text, Exec failed]` as nonfatal when `finalAssistantVisibleText` exists, while current main intentionally emits that exec warning for assistant success claims.

## Solution Assessment

No. The PR narrows the first attempt, but final assistant text plus a prior non-error payload is not enough to prove recovery; the classifier should require an explicit recovery or acknowledgement signal before suppressing a trailing exec warning.
