## Summary

Keep open. The PR targets a real isolated-cron classifier problem, but the new recovered-exec branch would also mark claimed-success exec failures as nonfatal, conflicting with the current payload contract that intentionally surfaces those failures.

## What This Changes

The branch adds a recovered-exec warning bypass in `resolveCronPayloadOutcome` plus regression tests for recovered exec and fatal trailing-error boundaries.

## Best Possible Solution

Keep trailing exec warnings fatal unless the classifier has an explicit recovery or failure-acknowledgement signal, then cover both recovered fallback and claimed-success failure cases.

## Reproduction Assessment

Yes. Source inspection shows the PR predicate would treat `[assistant success text, Exec failed]` as nonfatal when `finalAssistantVisibleText` exists, while current main intentionally emits that exec warning for assistant success claims.

## Solution Assessment

No. Checking only for final assistant text is too weak; the classifier needs a concrete recovery or acknowledgement signal before suppressing a trailing exec warning.
