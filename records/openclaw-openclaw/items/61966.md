## Summary

This PR is a plausible fix area, but the current patch is not enough to merge: it reads plugin commands from the TUI client process, while normal gateway-backed plugin commands are registered in the gateway/runtime process. It also lacks external-contributor real behavior proof and is currently conflicting with main.

## What This Changes

The PR adds `listPluginCommands()` to TUI slash-command autocomplete so plugin-registered commands can be appended after built-in and gateway commands.

## Best Possible Solution

Expose executable plugin command metadata through the TUI backend/session command provider so autocomplete mirrors gateway and local runtime commands without loading plugin runtimes in the TUI client.

## Reproduction Assessment

Yes, at source level: current `getSlashCommands()` never reads plugin commands, and the PR's direct registry read can be shown not to cross the gateway/TUI process boundary. I did not run the TUI because this review is read-only and tests could create artifacts.

## Solution Assessment

No. The proposed local `listPluginCommands()` loop is too narrow because gateway-backed plugin commands live in another process; the maintainable fix should source command metadata through the TUI backend/protocol and add focused coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Fetch plugin commands from the TUI backend:** `src/tui/commands.ts:139`
  - body: `getSlashCommands()` runs in the TUI client process, but the normal gateway-backed plugin commands are registered in the gateway/runtime process. Reading `listPluginCommands()` here only sees this process's registry, so external plugin commands still will not appear in the common `openclaw tui` autocomplete path. Expose the command metadata through the TUI backend or gateway session data instead of reading the local registry directly.
