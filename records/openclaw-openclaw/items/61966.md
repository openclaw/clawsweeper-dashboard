## Summary

This PR should close as superseded: it targets a real TUI autocomplete gap, but the proposed local-registry loop does not cover the gateway/runtime plugin command source, and the newer open PR at https://github.com/openclaw/openclaw/pull/73984 is the canonical backend-based path for the same user-visible fix.

## What This Changes

The PR imports `listPluginCommands()` into `src/tui/commands.ts` and appends those plugin command names to `getSlashCommands()` autocomplete output.

## Best Possible Solution

Keep the remaining work on https://github.com/openclaw/openclaw/pull/73984: hydrate TUI autocomplete from the `commands.list`/TUI backend command inventory, preserve local built-in completions on collisions, and cover gateway plus embedded paths with focused tests.

## Reproduction Assessment

Yes. Source inspection shows current main builds TUI autocomplete from `getSlashCommands()` in the TUI process, while plugin commands are exposed through the gateway `commands.list` inventory; I did not run an interactive TUI in this read-only review.

## Solution Assessment

No. The local `listPluginCommands()` loop is not the best fix because gateway-loaded plugin commands belong to the runtime inventory; the backend-based path in https://github.com/openclaw/openclaw/pull/73984 is the narrower maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Fetch plugin commands from the TUI backend:** `src/tui/commands.ts:139`
  - body: `getSlashCommands()` runs in the TUI process, but gateway-loaded plugins register their commands in the gateway/runtime process and expose them through `commands.list`. Reading `listPluginCommands()` here can still leave external plugin commands missing in the common gateway TUI path, so autocomplete should hydrate from the backend command inventory instead.
