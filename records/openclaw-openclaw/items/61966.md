## Summary

Keep open: the PR targets a real TUI autocomplete gap, but the patch reads the TUI process-local plugin registry, so gateway/runtime plugin commands can still be absent; it also lacks contributor real behavior proof.

## What This Changes

The branch imports `listPluginCommands()` into `src/tui/commands.ts` and appends process-local plugin commands to `getSlashCommands()` autocomplete after built-in and gateway command aliases.

## Best Possible Solution

Add a `TuiBackend` command-inventory method backed by gateway `commands.list` and equivalent embedded inventory, merge that into autocomplete with TUI-local commands, and add focused regression tests.

## Reproduction Assessment

Yes, at source level. Current main populates TUI autocomplete locally from `getSlashCommands()`, while the backend `commands.list` inventory already includes plugin commands and is not fetched by the TUI.

## Solution Assessment

No. Reading `listPluginCommands()` inside `getSlashCommands()` only sees the local singleton and misses gateway/runtime plugin commands; the maintainable fix is to fetch backend command inventory through the TUI backend seam.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

This loop reads the TUI process-local plugin command singleton, so autocomplete can still omit external plugin commands even though `commands.list` exposes them from the backend; please merge the backend command inventory instead.
