## Summary

Keep open: the autocomplete gap is real, but this PR reads the wrong registry boundary for gateway-backed plugin commands. Current main already exposes plugin commands through the backend `commands.list` inventory, while this branch only reads the TUI process-local plugin singleton.

## What This Changes

The branch imports `listPluginCommands()` into `src/tui/commands.ts` and appends registered plugin commands to `getSlashCommands()` autocomplete after built-in and gateway aliases.

## Best Possible Solution

Wire TUI autocomplete to a `TuiBackend` command-inventory method backed by `commands.list` in gateway mode and equivalent local inventory in embedded mode, then dedupe with TUI-only commands and add focused regression tests.

## Reproduction Assessment

Yes, at source level. Current main builds TUI autocomplete from local `getSlashCommands()`, while plugin-aware runtime commands are exposed through backend `commands.list`; I did not run the interactive TUI in this read-only review.

## Solution Assessment

No. The PR reads the TUI process-local plugin command singleton, so gateway-registered plugin commands can still be missing; the maintainable fix is to fetch and merge backend runtime command inventory.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Fetch plugin commands from the backend inventory:** `src/tui/commands.ts:139`
  - body: When the TUI connects to a gateway, plugin commands are registered in the gateway/runtime process, not the TUI process. This loop reads the TUI process-local `pluginCommands` singleton, so autocomplete can still omit external plugin commands even though `commands.list` exposes them from the backend. Please fetch and merge the backend command inventory instead.
