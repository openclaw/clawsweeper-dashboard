## Summary

Keep open: the autocomplete gap is real, but this patch reads the TUI process's local plugin registry and does not solve the gateway-backed plugin command path; the external PR also still lacks after-fix real behavior proof.

## What This Changes

The PR imports `listPluginCommands()` into `src/tui/commands.ts` and appends plugin-registered command names to slash-command autocomplete.

## Best Possible Solution

Wire TUI autocomplete to the `commands.list` runtime inventory through `TuiBackend`, cover both gateway and local embedded backends, dedupe with TUI-only commands, and add focused regression tests.

## Reproduction Assessment

Yes, at source level: current main builds TUI autocomplete from `getSlashCommands()` in the TUI process, while plugin command inventory is available through `commands.list`. I did not run the interactive TUI because this review is read-only.

## Solution Assessment

No. The proposed local `listPluginCommands()` loop reads the wrong process boundary for gateway-backed plugin commands; the narrower maintainable fix is to fetch the backend command inventory and merge those entries into autocomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Reading `listPluginCommands()` here only sees the local process registry, so external plugin commands can still be absent from gateway-connected TUI autocomplete; source the inventory through the TUI backend instead.
