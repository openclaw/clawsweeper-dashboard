## Summary

Keep open: the reported autocomplete gap is real on current main, but this PR reads the process-local plugin registry from the TUI client and therefore does not fix gateway-backed plugin commands; it also lacks the required after-fix real behavior proof.

## What This Changes

The PR imports `listPluginCommands()` in `src/tui/commands.ts` and appends those plugin command names to `getSlashCommands()` autocomplete output.

## Best Possible Solution

Wire TUI autocomplete to the current `commands.list`/TUI backend command inventory, dedupe it with local TUI-only commands, and cover the gateway-backed plugin command path with focused tests.

## Reproduction Assessment

Yes, at source level: current main builds TUI autocomplete from `getSlashCommands()`, and that function never reads plugin commands or the existing `commands.list` runtime inventory. I did not run the interactive TUI because this review is read-only.

## Solution Assessment

No. The proposed direct `listPluginCommands()` loop is not the best fix because it reads only the TUI process registry; the maintainable path is to source command metadata through the TUI backend/gateway command inventory.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Fetch plugin commands from the TUI backend:** `src/tui/commands.ts:139`

Reading `listPluginCommands()` here only sees the local process registry, so external plugin commands still will not appear in the common gateway TUI autocomplete path. Source the inventory through the TUI backend/gateway RPC instead.
