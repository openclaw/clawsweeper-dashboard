## Summary

Keep open. The PR targets a real autocomplete gap, but the proposed fix reads the plugin-command registry from the TUI client process; gateway-backed TUI plugin commands are registered in the Gateway process and exposed through the existing runtime command catalog, so this patch would not make external plugin commands appear in the common Gateway TUI path.

## What This Changes

The PR adds `listPluginCommands()` to `src/tui/commands.ts` and appends normalized, de-duplicated plugin command names to TUI slash-command autocomplete.

## Best Possible Solution

Use the Gateway runtime command catalog for Gateway-backed TUI autocomplete, likely by adding a TUI backend command-list method that calls `commands.list`, then merge those entries with local TUI-only commands using the existing de-duplication rules. Add focused regression coverage and a user-facing changelog entry crediting the contributor.

## Reproduction Assessment

Yes. A static reproduction path is clear: enable/register a plugin command in the Gateway, start a separate `openclaw tui`, and observe that autocomplete is built in the TUI process without requesting `commands.list`; the PR’s local singleton lookup would still miss Gateway-registered plugin commands.

## Solution Assessment

No. Reading `listPluginCommands()` in `src/tui/commands.ts` is not the best fix because it uses the wrong process boundary for Gateway mode; the safer maintainable path is to consume the existing `commands.list` RPC and keep plugin loading out of the TUI client startup path.

## Review Findings
