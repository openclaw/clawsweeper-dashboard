## Summary

Keep open: the reported TUI autocomplete gap is source-reproducible, but this PR should not merge as written because it reads process-local plugin command state in the TUI client instead of the backend runtime command inventory exposed by `commands.list`. The PR also lacks after-fix real behavior proof and is currently conflicting with main.

## What This Changes

The branch imports `listPluginCommands()` into `src/tui/commands.ts` and adds those local plugin command entries to `getSlashCommands()` autocomplete with normalization and deduplication.

## Best Possible Solution

Wire TUI autocomplete to a `TuiBackend` command-inventory method backed by `commands.list` in gateway mode and equivalent local inventory in embedded mode, then dedupe with TUI-only commands and add focused regression tests.

## Reproduction Assessment

Yes, at source level: current main builds TUI autocomplete in the TUI process from `getSlashCommands()`, while the plugin-aware runtime command inventory is already exposed through `commands.list`. I did not run the interactive TUI because this review is read-only.

## Solution Assessment

No. The proposed local `listPluginCommands()` loop reads the wrong process boundary for gateway-backed plugin commands; the narrower maintainable fix is to fetch and merge the backend command inventory.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Fetch plugin commands from the backend inventory:** `src/tui/commands.ts:139`
  - body: When the TUI connects to a gateway, plugin commands are registered in the gateway/runtime process, not the TUI process. This loop reads the TUI process's local `pluginCommands` singleton, so autocomplete can still omit external plugin commands even though `commands.list` exposes them from the backend. Please fetch and merge the backend command inventory instead.
  - confidence: 0.89
