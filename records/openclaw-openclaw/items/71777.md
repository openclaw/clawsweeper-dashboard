## Summary

Keep open: current main and the latest release still expose `cron.run` as due/force queued manual execution, with no explicit scheduled-preview/as-scheduled contract for service environment or heartbeat/context parity. The protected `maintainer` label also prevents cleanup closure and points this to maintainer product/API review.

## What This Changes

Request that manual `openclaw cron run` either default to, or expose a flag for, a scheduled-equivalent execution environment covering workspace cwd, scheduler/service env, and heartbeat/context loading.

## Best Possible Solution

Define an additive scheduled-preview or as-scheduled cron contract through CLI, Gateway protocol, cron service, and runner seams, keeping workspace/env/context run-scoped and coordinated with the broader scheduler diagnostics work.

## Reproduction Assessment

No for the full macOS LaunchAgent and Windows service-environment divergence; this review did not run live platform schedulers. Source inspection does confirm the missing explicit `as-scheduled`/scheduled-preview surface on current main and v2026.6.10.

## Solution Assessment

Unclear for a concrete implementation until maintainers choose the public contract. A narrow cwd-only patch is not the best fix because current main already has run-scoped workspace seams and the issue also asks for service env plus heartbeat/context parity.
