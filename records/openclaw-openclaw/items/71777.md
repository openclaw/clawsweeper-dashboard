## Summary

Keep open: the issue has the protected `maintainer` label and the remaining request is a product/runner-contract question, not a safe cleanup close. Current main appears to have partially narrowed the old divergence by routing CLI manual runs through the Gateway cron service and passing a run-scoped workspace/context kind into the isolated cron runner, but there is still no `--as-scheduled` surface or live macOS/Windows proof that env and context parity are solved.

## What This Changes

The issue asks for manual `cron run` to faithfully reproduce scheduled cron execution, including cwd, service environment, and cron/heartbeat context loading, either by default or behind an explicit flag.

## Best Possible Solution

Define a maintainer-approved, run-scoped scheduled-environment contract for manual cron runs, then implement or document cwd, environment, and context parity through CLI/Gateway/service/runner seams with focused macOS and Windows proof.

## Reproduction Assessment

No. Current source inspection shows manual runs now go through the Gateway cron service and runner seams, but there is no high-confidence live reproduction of the reported LaunchAgent/Windows env and context mismatch on current main.

## Solution Assessment

Unclear. The right end state needs maintainer product judgment on whether parity is default behavior, an explicit flag, or documented current semantics, and any implementation should avoid process-global cwd mutation.
