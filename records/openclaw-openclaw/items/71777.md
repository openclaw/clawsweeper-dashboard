## Summary

Keep open: current main and v2026.6.10 still expose `cron.run` as queued `due`/`force` manual execution, with no explicit scheduled-preview or `as-scheduled` contract. The protected `maintainer` label and product-decision labels also make this a maintainer API decision rather than a cleanup close.

## What This Changes

The issue asks for manual `cron run` execution to reproduce the scheduled cron environment, including workspace cwd, service-like environment, and heartbeat/context loading parity.

## Best Possible Solution

Keep this open for a maintainer decision on the public CLI/Gateway contract, then implement one explicit scheduled-preview path that shares scheduled-run cwd/env/context semantics and has focused CLI, protocol, service, runner, and docs coverage.

## Reproduction Assessment

No for a high-confidence live platform reproduction: I did not run macOS LaunchAgent or Windows Scheduled Task comparisons. Source inspection does confirm the missing explicit `as-scheduled`/scheduled-preview surface on current main and v2026.6.10.

## Solution Assessment

Unclear until a maintainer chooses the API shape. An explicit `--as-scheduled` or RPC mode looks safer than changing the default manual-run behavior because `cron run` is already documented as a queued force-run.
