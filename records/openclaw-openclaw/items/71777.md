## Summary

Keep open: the issue has the protected `maintainer` label, and current `main`/latest release still do not expose an explicit scheduled-environment mode for `cron run`. Current code has partially narrowed the old gap by queuing manual runs through the Gateway cron lane and sharing the same execution core, but the remaining cwd/env/context contract needs maintainer judgment rather than cleanup closure.

## What This Changes

The issue requests that manual `cron run` reproduce the scheduled cron execution environment, including cwd, environment, and context loading.

## Best Possible Solution

Define the scheduled-preview contract explicitly, then implement any remaining parity through run-scoped workspace, environment, and context seams with docs and focused manual-versus-scheduled regression coverage.

## Reproduction Assessment

No high-confidence full current-main OS scheduler repro was established. Source inspection does show the missing scheduled-preview contract: the CLI and protocol still carry only job id plus `due`/`force` mode.

## Solution Assessment

No for the prior proposed implementation: mutating `process.cwd()` around an async Gateway cron run is not the best fix. The better direction is a run-scoped execution policy that reuses existing workspace/context seams and defines any environment parity deliberately.
