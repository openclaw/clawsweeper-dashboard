## Label Justifications

- `P2`: This is a normal-priority exec runtime bugfix with real but bounded impact on commands using leading-tilde workdirs.
- `merge-risk: 🚨 compatibility`: The PR changes how host exec workdirs are resolved and can choose a different directory for users with `OPENCLAW_HOME` set.

## Summary

Keep open: this is a focused fix for a real exec workdir bug, but the patch resolves host exec `~` paths through the OpenClaw-home helper and only supplies lint/unit-test proof, so it is not merge-ready.

## What This Changes

The branch imports `resolveUserPath` and resolves local exec workdirs before `statSync`, returning the resolved path when it is an existing directory.

## Best Possible Solution

Merge one narrow exec fix that expands leading `~` with OS-home semantics for host command paths, preserves fallback only for truly unavailable directories, and includes regression coverage plus real after-fix proof.

## Reproduction Assessment

Yes. Source inspection shows a local gateway exec call with `workdir: "~/..."` reaches `resolveWorkdir()`, which currently stats the literal path and falls back before command execution.

## Solution Assessment

No. Expanding before `statSync()` is the right layer, but `resolveUserPath()` is not the best helper for host exec because it resolves `~` through `OPENCLAW_HOME`; the safer fix is OS-home tilde expansion with focused regression coverage.
