## Summary

Keep open: the recovery approach is narrow and useful, but the PR still drops persisted Codex auth-profile metadata when recreating the missing app-server thread, which can reroute the retry under the wrong credentials and rewrite the binding without the original profile.

## What This Changes

This PR adds one-time `/codex bind` stale app-server thread recovery, regression coverage, and a changelog entry.

## Best Possible Solution

Preserve `authProfileId` through thread recreation and add an auth-profile recovery assertion before merging.

## Reproduction Assessment

Yes by source inspection: current main sends bound turns directly to `turn/start` and reports a thrown stale-thread error as a failed reply, while the PR test mocks that stale-thread path. I did not run it because this review is read-only and `pnpm` is unavailable in this checkout.

## Solution Assessment

No. Recreating and retrying once is the right shape, but the implementation should also preserve the saved `authProfileId` alongside the saved model, permission, sandbox, and service-tier metadata.
