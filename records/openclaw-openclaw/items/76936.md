## Summary

Keep this PR open, but it needs changes before merge: current main still lacks missing-thread recovery, while this PR's recovery path can drop a persisted Codex auth profile and retry under default credentials.

## What This Changes

The PR adds one-time stale `/codex bind` app-server thread recovery, regression coverage, and a changelog entry.

## Best Possible Solution

Preserve `authProfileId` through thread recreation and add an auth-profile recovery regression while keeping the PR's one-retry missing-thread behavior.

## Reproduction Assessment

Yes by source inspection. Current main sends bound turns directly to `turn/start` and reports stale-thread errors, while the PR test exercises that missing-thread path without covering persisted auth-profile metadata.

## Solution Assessment

No. Recreating and retrying once is the right narrow approach, but this PR should also carry the saved `authProfileId` into the recreated thread and rewritten binding.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve the binding auth profile during recovery:** `extensions/codex/src/conversation-binding.ts:372-381`
  - body: The recovery path recreates the app-server thread from the saved binding but does not forward `binding.authProfileId`. Existing bound turns already start the shared client with that field and binding files persist it, so after a restart this can retry and rewrite the binding under the wrong/default Codex credentials.
