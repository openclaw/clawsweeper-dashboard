## Summary

This PR should stay open because the recovery path still drops the bound Codex auth profile, which can make a restarted `/codex bind` conversation retry under the wrong/default credentials.

## What This Changes

The branch adds one-time missing-thread recovery for Codex conversation bindings, regression coverage, and a changelog entry.

## Best Possible Solution

Preserve `authProfileId` through thread recreation and add an auth-profile recovery regression before merging.

## Reproduction Assessment

Yes by source inspection: current main sends bound turns to `turn/start` and reports a thrown stale-thread error as a failed reply, while the PR test mocks that stale-thread path. I did not run the test because this review is read-only.

## Solution Assessment

No. Recreating and retrying is the right shape, but this PR should also preserve the saved auth profile alongside the saved model, permission, sandbox, and service-tier metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve the binding auth profile during recovery:** `extensions/codex/src/conversation-binding.ts:372-381`
  - body: The recovery path recreates the app-server thread from the old binding but only forwards cwd/model/provider and permission settings. Existing bound turns already start the shared client with `binding.authProfileId`, and binding files persist that field; after a restart, this path can recreate the thread without the bound Codex profile and rewrite the binding without it, so profile-specific or Codex OAuth chats may retry under the wrong/default credentials.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch has a concrete auth-isolation concern because recovery can drop a persisted Codex auth profile.

Concerns:

- **[medium] Recovery can switch auth profiles:** `extensions/codex/src/conversation-binding.ts:372`
  - body: `authProfileId` selects the app-server credential/profile used for a bound turn. Recreating the missing thread without carrying that field forward can route the retry through a different default profile and then persist the replacement binding without the original profile.
  - confidence: 0.84
