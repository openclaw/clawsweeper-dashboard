## Summary

Keep open for maintainer review. Current main has an unreleased `canvas.a2ui.reset` plus plugin-surface capability refresh path, but `canvas present` still preserves an explicit non-A2UI target by design, and the paired macOS A2UI rendering report could affect the real recovery path.

## What This Changes

The issue asks macOS Canvas to recover A2UI after the WebView navigates away or its capability URL expires, either by changing `canvas present` or by providing an explicit reset path.

## Best Possible Solution

Decide the intended Canvas UX: keep `canvas present` non-destructive and ship/document `canvas.a2ui.reset` as recovery, or explicitly redefine/add a reset-style command with live macOS proof.

## Reproduction Assessment

Yes for the narrow `canvas present` behavior: current source and smoke coverage show it does not force A2UI after an explicit user target. No live macOS proof was established for token-expiry recovery through `canvas.a2ui.reset`.

## Solution Assessment

Unclear. Forcing every `canvas present` back to A2UI is not clearly the best fix because it would change tested target-preservation semantics; a deliberate reset/recovery path with live proof is safer.
