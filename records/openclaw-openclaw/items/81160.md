## Summary

Keep open for maintainer review: current main has an unreleased `canvas.a2ui.reset` and capability-refresh path, but `canvas.present` still preserves explicit non-A2UI targets by design and the reporter has a paired open macOS A2UI delivery report that may affect the real recovery story.

## What This Changes

The issue asks macOS Canvas to recover A2UI after WebView navigation or cap-token expiry, either by making `canvas.present` load a fresh A2UI URL or by providing an agent-accessible reset path.

## Best Possible Solution

Decide the intended macOS Canvas UX: either keep `canvas.present` as a non-destructive show command and document/release `canvas.a2ui.reset` as the recovery path, or deliberately redefine a command that forces a fresh A2UI capability URL.

## Reproduction Assessment

Yes for the narrow `canvas.present` behavior: current source and smoke coverage show that after an explicit user target, present does not force A2UI. No live macOS proof was run for token-expiry recovery on current main.

## Solution Assessment

Unclear. The reset/capability-refresh path is a safer recovery shape than changing `present` blindly, but maintainers should decide whether target preservation or forced A2UI recovery is the product contract.
