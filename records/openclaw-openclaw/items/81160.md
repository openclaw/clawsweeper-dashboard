## Summary

Keep open for maintainer follow-up. Current main partially addresses recovery with `canvas.a2ui.reset` and capability refresh, but `canvas present` still preserves explicit non-A2UI targets and changing that behavior conflicts with existing macOS Canvas coverage.

## What This Changes

The issue requests a reliable macOS Canvas recovery path so `canvas present` restores A2UI with a fresh capability URL after navigation or token expiry.

## Best Possible Solution

Decide the intended Canvas UX, then either document and validate `canvas.a2ui.reset` as the recovery path or add a deliberately named reset/present command that refreshes A2UI without breaking targeted Canvas pages.

## Reproduction Assessment

Yes for the narrow `canvas present` behavior: current source and macOS smoke coverage show an explicit user target is not auto-overridden back to A2UI. No live current-main reproduction was established for cap-token expiry or reset failure.

## Solution Assessment

Unclear. Making every `canvas present` force A2UI is not clearly the best fix because it conflicts with tested target-preservation semantics; the safer direction is maintainer-approved reset semantics plus live macOS proof.
