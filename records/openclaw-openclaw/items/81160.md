## Summary

Keep open. Current main partially addresses recovery with `canvas.a2ui.reset` and a capability-refresh path, but `canvas present` still preserves explicit non-A2UI targets and changing that behavior needs a maintainer UX decision plus live macOS proof.

## What This Changes

The issue asks for macOS Canvas to recover to a fresh A2UI URL after navigation or cap-token expiry without requiring a gateway restart.

## Best Possible Solution

Decide whether `canvas present` is a non-destructive show command or an A2UI recovery command, then document and live-test the chosen recovery path with `canvas navigate`, cap-token refresh, `canvas.a2ui.reset`, and A2UI push on macOS.

## Reproduction Assessment

Yes for the narrow `canvas present` behavior: current source and smoke coverage show it does not force A2UI after an explicit user target. No live proof establishes that `canvas.a2ui.reset` still fails on current main.

## Solution Assessment

Unclear. Forcing every `canvas present` back to A2UI is not clearly the best fix because current tests preserve explicit Canvas targets; a documented explicit reset/recovery path may be safer.
