## Summary

Keep open for maintainer review. Current main has an unreleased `canvas.a2ui.reset` and capability-refresh recovery path, but `canvas present` still intentionally preserves an explicit non-A2UI target, and the same reporter has a paired open macOS A2UI rendering issue that may affect the real recovery story.

## What This Changes

The issue asks for macOS Canvas A2UI recovery after `canvas navigate` or cap-token expiry, either by making `canvas present` re-navigate to a fresh A2UI URL or by providing an equivalent reset/refresh path.

## Best Possible Solution

Decide the intended macOS Canvas UX: either keep `canvas present` as a non-destructive show command and document/release `canvas.a2ui.reset` as the recovery path, or add a clearly named A2UI reset/present command that forces a fresh capability URL without breaking targeted Canvas pages.

## Reproduction Assessment

Yes for the narrow `canvas present` behavior: current main source and smoke coverage show it does not force A2UI after an explicit user target. There is not yet high-confidence live proof that `canvas.a2ui.reset` still fails on current main.

## Solution Assessment

Unclear. Forcing every `canvas present` back to A2UI is not clearly the best fix because it would change tested target-preservation semantics; the safer maintainable direction is a deliberate reset/recovery path or an explicit maintainer decision to redefine `present`.
