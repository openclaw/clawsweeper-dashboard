## Summary

Keep open: this is a useful release backport, but the pending warning fallback is replayed after the normal inbound dispatch returns, outside the foreground freshness checks, so an older Discord turn can still post a stale warning after a newer answer. The PR also has only focused tests/Testbox proof, not real Discord delivery proof.

## What This Changes

Backports Discord recovered non-terminal tool-warning suppression to `release/2026.5.27`, adding a reply-payload metadata predicate/export plus regression coverage for recovered and warning-only cases.

## Best Possible Solution

Land the backport after pending fallback delivery is kept inside or guarded by the foreground freshness path and after redacted real Discord proof shows recovered warnings are suppressed while warning-only fallbacks still deliver.

## Reproduction Assessment

Yes at source level for the introduced race: the PR stores a pending warning, returns from `dispatchChannelInboundReply`, and only then replays it. I did not run live Discord, so the proof is source-path rather than end-to-end reproduction.

## Solution Assessment

No. Suppressing metadata-marked recovered warnings is the right direction, but replaying warning-only fallback delivery after dispatch is not the narrowest safe implementation because it bypasses the current freshness fence.

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Recheck freshness before replaying fallback warnings:** `extensions/discord/src/monitor/message-handler.process.ts:1070`
  - body: When the only pending Discord final is a metadata tool warning, this replay runs after `dispatchChannelInboundReply` has returned. The foreground reply fence performs its freshness cancellation inside `beforeDeliver` and is ended before this point, so a newer same-session/channel message that answers in the gap can no longer cancel the older fallback; the stale warning can be posted after the newer answer. Keep the fallback inside the dispatcher/fence path or repeat the same freshness check before sending it.
