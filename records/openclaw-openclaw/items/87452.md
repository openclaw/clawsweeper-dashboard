No, not as currently shaped: the suppression itself is the right behavior, but fallback warning delivery should stay inside the existing freshness fence or perform the same recheck before sending.

- **[P2] Recheck freshness before replaying fallback warnings:** `extensions/discord/src/monitor/message-handler.process.ts:873-875`
  - body: When an older Discord turn only has a metadata tool-warning final, this replay runs after `dispatchChannelInboundReply` returns. Keep this fallback inside the dispatcher/fence path or repeat the same freshness check before sending it.
