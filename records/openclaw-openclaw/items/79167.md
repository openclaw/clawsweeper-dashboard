## Summary

Keep open: current main has tool-name policy and workspace-only filesystem guards, but no supported per-path deny policy, and the requested cross-tool sensitive-path deny set needs maintainer security/product design.

## What This Changes

Add a configurable per-path filesystem deny list that applies uniformly to native file tools and MCP file-capable tools, with possible sensitive-file defaults.

## Best Possible Solution

Design a first-class resolved-path policy in the agent/tool dispatch layer, document precedence with sandbox/tool policy/hooks, and cover native, plugin, and MCP filesystem paths with regression tests.

## Reproduction Assessment

Not applicable as a feature request; source and docs confirm current main has no `denyPaths`/path-deny config and only supports tool-level policy plus `workspaceOnly`.

## Solution Assessment

Unclear: a dispatch-layer path policy is plausible, but maintainers should first define resolved-path matching, default deny patterns, MCP/plugin semantics, and interaction with `workspaceOnly`, sandboxing, and `before_tool_call`.
