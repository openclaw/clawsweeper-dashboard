Keep open. Current main still has a plausible doctor repair gap: the reported message is produced by plugin package-entry boundary validation, then surfaced as a top-level `plugins` config error that the existing stale/invalid plugin repair paths do not handle.

## What This Changes

The issue reports that after moving an OpenClaw configuration to another Windows drive and reinstalling OpenClaw 2026.3.7, plugin config validation fails with an extension entry escaping the package directory for `./index.ts`, and `openclaw doctor --fix` does not repair it.

## Best Possible Solution

Add a narrow doctor/registry repair path for configured installed plugins whose package-entry diagnostics block startup after config or install-root moves, while preserving strict package-directory containment checks.

## Reproduction Assessment

No, not a high-confidence live reproduction. Source inspection traces the reported current-main diagnostic and shows why the existing doctor repair paths do not handle that top-level registry error, but the full Windows moved-drive flow still needs a focused regression fixture.

## Solution Assessment

No close or stale cleanup is appropriate here. The maintainable fix is to make doctor repair or quarantine this plugin-install state without weakening package entry boundary validation.

## Repair Work Prompt

Fix the plugin doctor repair gap reported in https://github.com/openclaw/openclaw/issues/80592. Current main can surface `plugins: plugin: extension entry escapes package directory: ./index.ts` from `src/plugins/package-entry-resolution.ts`, but `openclaw doctor --fix` only repairs stale plugin refs when discovery has no errors and only quarantines `plugins.entries.<id>.config` schema issues. Add focused regression coverage for a configured installed plugin/package registry state that produces this package-entry diagnostic after a config/install-root move, then make doctor repair the state by refreshing/reinstalling the affected managed plugin record or quarantining/removing the affected configured plugin entry as appropriate. Do not relax package-directory boundary checks, symlink/hardlink checks, or plugin entry containment rules. Update the active changelog if the fix changes user-visible doctor behavior.
