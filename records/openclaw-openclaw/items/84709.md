## Summary

Keep open: the PR targets a real Codex cron regression, but it should not merge as-is because the fail-closed implementation has compatibility and security-boundary gaps plus no after-fix real cron run proof.

## What This Changes

The PR changes Codex cron tool-surface handling to preserve required dynamic exec, fail finite required-tool cron runs before model dispatch when tools are unavailable, and classify missing-tool summaries as fatal cron outcomes.

## Best Possible Solution

Keep the fail-closed cron contract, but make required-tool availability consistent: either hydrate/check explicit OpenClaw dynamic tools without widening native Code Mode, or deliberately map the whole Codex native shell/file surface with compatibility tests, scoped classifier logic, and live cron upgrade proof.

## Reproduction Assessment

Yes at source level: current main filters out Codex dynamic exec before applying toolsAllow and disables native Code Mode for finite allowlists, and https://github.com/openclaw/openclaw/issues/84141 supplies live Linux/macOS repro evidence. I did not run a live cron job in this read-only review.

## Solution Assessment

No. The fail-closed direction is right, but the current PR inconsistently maps Codex-native file tools and uses an overbroad summary classifier; the safer solution needs a clear native-surface contract plus real cron proof.

- **[P1] Handle the whole native file surface consistently:** `extensions/codex/src/app-server/run-attempt.ts:3633-3639`
  - body: With `toolsAllow: ["exec", "write"]`, this PR enables Codex native Code Mode because `exec` is present, but the required-tool check only treats `read` as satisfied by that native surface. The dynamic `write` tool is still filtered out as Codex-owned, so existing cron jobs using the original `exec,read,write` allowlist shape will fail before dispatch even though the native shell/file surface was just enabled. Either account for all native file tools consistently or fail this class explicitly without enabling Code Mode.
  - confidence: 0.86
- **[P2] Narrow the missing-tool summary tokens:** `src/cron/isolated-agent/helpers.ts:50-51`
  - body: The new fallback classifier runs on every cron summary with no structured error and matches broad substrings like `tool unavailable` and `unable to run required`. A successful monitoring cron that reports `no tool unavailable alerts` or summarizes an incident can now be marked fatal. Restrict this to structured/sentinel missing-tool output or pass the required-tool context into the outcome resolver.
  - confidence: 0.83
