## Summary

Keep open: this is a plausible fix for the linked isolated-cron exec regression, but it is not merge-ready because the after-fix proof is test-only and the diff changes security-sensitive restricted-allowlist behavior for Codex cron runs.

## What This Changes

The PR preserves explicitly allowed Codex dynamic exec/process tools for finite cron `toolsAllow` lists, enables native Codex app-server tooling for exec/process allowlists, fails missing required tool surfaces before dispatch, and classifies missing-tool summaries as fatal cron outcomes.

## Best Possible Solution

Land a fix only after maintainers confirm the Codex native tool-surface boundary and the contributor supplies a real patched cron/Codex run showing required exec exposure or the new fail-closed diagnostic before model dispatch.

## Reproduction Assessment

Yes at source level, with live user evidence in the linked bug: current main filters default Codex-owned dynamic exec before applying a finite allowlist and disables native Codex tooling for restricted allowlists, so `toolsAllow: ["exec"]` can still leave the cron run without exec.

## Solution Assessment

Partly: rehydrating explicitly allowed dynamic exec/process and failing closed before dispatch targets the right bug, but enabling native Codex code mode for finite exec/process allowlists and changing CLI cron upgrade behavior need maintainer security/compatibility approval plus real after-fix runtime proof.
