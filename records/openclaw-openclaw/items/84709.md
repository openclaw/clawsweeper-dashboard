## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Treat empty CLI allowlists as enforced policy:** `src/cron/isolated-agent/run-executor.ts:207-213`
  - body: `toolsAllow: []` is a defined runtime allowlist meaning no tools in embedded planning, and the CLI runner fails closed for any defined `toolsAllow` because it cannot enforce that policy. This new preflight only rejects non-empty finite lists, so a CLI-backed cron job that explicitly requests no tools still reaches `runCliAgent` without `toolsAllow` and can run with the CLI backend's native tools.
  - confidence: 0.9
- **[P1] Account for the full native file surface:** `extensions/codex/src/app-server/run-attempt.ts:3644-3651`
  - body: The PR enables native Code Mode whenever a finite allowlist requires `exec` or `process`, but the required-tool check only treats `read` as satisfied by that native surface. Cron jobs that request `exec` with `write`, `edit`, or `apply_patch` will fail with `TOOL_SURFACE_UNAVAILABLE` even though this code path enabled the native file surface that should satisfy those tools, so the accounting needs to cover the whole native surface or fail with a deliberate documented preflight.
  - confidence: 0.86
- **[P2] Narrow missing-tool summary matching:** `src/cron/isolated-agent/helpers.ts:45-53`
  - body: The new classifier marks a run fatal when any successful summary or output contains broad substrings like `tool unavailable` or `unable to run required`, without structured missing-tool metadata. A successful ops report that discusses an external unavailable tool or a historical unmet requirement can now be recorded as a cron failure, so this should key off structured failure signals or much more specific generated sentinel text.
  - confidence: 0.78
