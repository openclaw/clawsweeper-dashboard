## Summary

Keep this PR open: it targets a real P1 cron/Codex regression, but the current patch still has blocking tool-policy and security-boundary issues plus no after-fix real Gateway cron proof.

## What This Changes

The branch changes Codex app-server and isolated cron execution so finite cron `toolsAllow` can preserve required `exec`, fail before dispatch when required tools are missing, and classify missing-tool summaries as cron failures.

## Best Possible Solution

Keep the fail-closed direction, but make the required-tool contract exact for dynamic exec/process, native file tools, empty allowlists, and CLI forwarding before requiring redacted after-fix Gateway cron proof.

## Reproduction Assessment

Yes at source level: current main filters Codex-owned dynamic `exec` before finite `toolsAllow` filtering and disables native Code Mode for finite allowlists, while the linked issue provides live 2026.5.18 missing-exec evidence. I did not run a live cron job in this read-only review.

## Solution Assessment

No, not yet: the fix direction is plausible, but this patch does not consistently model native file tools or empty CLI allowlists and needs maintainer approval of the Codex native Code Mode boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Treat empty CLI allowlists as enforced policy:** `src/cron/isolated-agent/run-executor.ts:207`
  - body: `toolsAllow: []` is a defined runtime allowlist meaning no tools in the embedded planner, and the CLI runner already fails closed for any defined `toolsAllow` because it cannot enforce that policy. This new preflight only rejects non-empty finite lists, so a CLI-backed cron job that explicitly requests no tools still reaches `runCliAgent` without `toolsAllow` and can run with the CLI backend's native tools.
  - confidence: 0.9
- **[P1] Account for the full native file surface:** `extensions/codex/src/app-server/run-attempt.ts:3645-3651`
  - body: The PR enables native Code Mode whenever a finite allowlist requires `exec` or `process`, but the required-tool check only treats `read` as satisfied by that native surface. Cron jobs that request `exec` with `write`, `edit`, or `apply_patch` will fail with `TOOL_SURFACE_UNAVAILABLE` even though this code path enabled the native file surface that should satisfy those tools.
  - confidence: 0.86
- **[P2] Narrow missing-tool summary matching:** `src/cron/isolated-agent/helpers.ts:45-53`
  - body: The new classifier marks a run fatal when any successful summary/output contains broad substrings like `tool unavailable` or `unable to run required`, without structured missing-tool metadata. A successful ops report that discusses an external unavailable tool or a historical unmet requirement can now be recorded as a cron failure.
  - confidence: 0.78
