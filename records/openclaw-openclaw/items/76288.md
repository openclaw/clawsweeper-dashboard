## Summary

Keep open: the PR addresses the reported cron history correlation gap, but the current diff is not merge-ready because it changes canonical cron task run IDs and leaves the public hook and changelog surfaces incomplete.

## What This Changes

The PR threads the manual `cron.run` acknowledgement `runId` through queued execution, finished cron events, JSONL-backed run history, protocol Swift models, and a gateway regression assertion.

## Best Possible Solution

Preserve canonical `cron:<jobId>:<startedAt>` task ledger IDs while separately persisting the manual acknowledgement id in cron events/history, then align the hook API surface and changelog before merge.

## Reproduction Assessment

Yes. Current main returns a manual acknowledgement id from `cron.run` but does not carry it into the finished event or run-log read path; the PR's gateway assertion is a focused reproduction for that gap. The task-ledger regression is reproducible by comparing the PR's `manual:...` task id creation with maintenance recovery's `cron:` id parser.

## Solution Assessment

No, not as submitted. The history propagation is the right narrow direction, but replacing task ledger ids is unsafe; keep the ledger id canonical and expose the manual id only on the user-facing cron event/history surfaces.

  - body: This changes enqueued manual runs to create the task ledger record with the public `manual:...` id.
