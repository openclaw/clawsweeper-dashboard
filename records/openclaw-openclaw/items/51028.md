## Summary

Keep open: current main and the latest release still sort Sessions rows by stored `updatedAt`, and the requested meaningful-activity timestamp, reason label, and parent/child rollup are not implemented. The issue is valid but needs maintainer product/API direction because the discussion raises a real read-time versus persisted projection tradeoff.

## What This Changes

The issue requests that the Sessions panel sort by human-meaningful session activity rather than raw message or registry update timestamps.

## Best Possible Solution

Define a Gateway-owned meaningful-activity projection with explicit activity reason codes and parent/child rollup semantics, then expose it through `sessions.list` so the Control UI can sort and explain rows without overloading `updatedAt`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `sessions.list` and the Sessions view both order by `updatedAt`, and the row contract has no meaningful-activity timestamp or reason field.

## Solution Assessment

Unclear. The user problem is valid, but the issue discussion raises a real write-amplification concern for a stored `lastMeaningfulActivityAt`, so maintainers should decide the activity taxonomy and read-time versus persisted projection before implementation.
