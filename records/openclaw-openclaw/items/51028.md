## Summary

Keep open. Current main still orders and refreshes the Sessions panel from raw session-row `updatedAt`, while the gateway/UI contract does not expose a semantic meaningful-activity timestamp, visible-conversation timestamp, activity reason, or parent rollup sort key.

## What This Changes

The issue asks for the Sessions panel to sort by last meaningful user-visible activity, explain why a session moved, and roll child or sub-agent activity up to parent sessions while ignoring heartbeat/background churn.

## Best Possible Solution

Add an additive gateway-backed session activity contract: persist or derive raw event time, meaningful activity time, visible conversation time, and a concise activity reason; classify heartbeat/cron/internal writes so they do not bubble rows; roll child meaningful activity into parent list ordering; then update `sessions.list`, `sessions.changed`, Control UI sorting/rendering, docs, and focused tests while leaving the already-split #51030 label work separate.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: make one stored session receive only a heartbeat/background `updatedAt` touch while another has a newer real `lastInteractionAt`; current `listSessionsFromStore` and the Control UI still order by `updatedAt`, because no semantic activity field exists.

## Solution Assessment

Yes, the requested direction is the right maintainable solution if implemented additively in the gateway contract. A UI-only sort tweak would be insufficient because the gateway currently does not expose the semantic timestamp, reason, or parent rollup needed to distinguish meaningful activity from bookkeeping writes.
