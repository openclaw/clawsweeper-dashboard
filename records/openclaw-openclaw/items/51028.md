## Summary

Keep open: current main and the latest release still sort and limit session rows by stored `updatedAt`, while the requested meaningful-activity projection, reason label, and parent/child rollup contract are not implemented. The latest discussion also raises a real product/API decision about read-time projection versus a persisted sort key.

## What This Changes

Requests a session activity model and Sessions panel ordering/reason UI based on last meaningful activity rather than raw update/message timestamps.

## Best Possible Solution

Define a gateway-owned activity projection with explicit reason codes and parent/child rollup semantics, then expose it through `sessions.list` so the Control UI can sort and explain rows without overloading `updatedAt`.

## Reproduction Assessment

Yes, source-reproducible: current main and the latest release sort/filter session rows by `updatedAt`, and the session row contract lacks the requested meaningful-activity timestamp and reason field. I did not run a live UI scenario because this review is read-only.

## Solution Assessment

Unclear as specified: the user problem is valid, but the proposed persisted `lastMeaningfulActivityAt` may not be the best implementation after the write-amplification concern. The maintainable path is to decide the activity taxonomy and projection strategy before code changes.
