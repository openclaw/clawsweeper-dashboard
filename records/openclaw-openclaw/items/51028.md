## Label Justifications

- `impact:session-state`: The request changes how session activity state is classified, rolled up, and presented in session lists.

## Summary

Keep open: current main and the latest release still order session rows from `updatedAt`, while current docs and tests already distinguish user interaction freshness from noisy store-row mutations. The remaining work is a product/API decision for a gateway-owned meaningful-activity projection, reason label, and child-session rollup.

## What This Changes

The issue requests that the sessions panel order rows by a canonical last meaningful activity timestamp instead of raw last-message or row-update time, with child activity rollup and a human-readable reason label.

## Best Possible Solution

Define a gateway-owned meaningful-activity projection on session rows, using `lastInteractionAt` and other approved lifecycle signals as inputs, then sort/render the Control UI from that projection with tests for heartbeat noise and child-session rollup.

## Reproduction Assessment

Yes, from source inspection: current main defaults the sessions panel to `updated` descending, the table and live-update comparators use `updatedAt`, and the gateway list is also ordered by store-entry `updatedAt`. I did not run a browser scenario because this review is read-only.

## Solution Assessment

Unclear until maintainers choose the activity contract. The maintainable path is a gateway/protocol row projection with UI sorting and reason rendering, not ad hoc inference inside the sessions table alone.
