## Summary

Keep open. Current main still allows subagent completion announcements to be steered or direct-fallback delivered into an active Telegram requester session without a per-inbound turn guard, and the related #74286/#58443 reports remain open rather than fully superseding this v2026.5.2 Telegram reproduction.

## What This Changes

The issue asks OpenClaw to prevent stale or duplicate Telegram replies when normal user turns interleave with sub-agent completions, tool work, and heartbeat events.

## Best Possible Solution

Add a session-turn or inbound-message generation guard for background completion delivery and Telegram dispatch, preserving explicit background updates while suppressing stale final replies after newer user turns.

## Reproduction Assessment

Yes for source-level reproduction: current main can route a subagent completion into an active Telegram requester via steering or direct fallback without checking whether a newer inbound Telegram message superseded that completion. I did not live-run the full Telegram heartbeat/tool sequence.

## Solution Assessment

Yes, the best direction is a narrow turn-binding or generation check around background completion delivery plus regression tests for active requester interleaving. Direct fallback and announce-context changes alone are not sufficient unless they also prevent stale delivery after newer user input.
