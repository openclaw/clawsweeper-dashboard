## Summary

Keep open. Current main still has a plausible stale-turn path: sub-agent completions can be steered into an active requester run or sent as a direct fallback, while Telegram’s stale-final suppression is scoped to authorized abort commands rather than ordinary newer user turns; the related #74286 and #58443 do not fully cover this broader Telegram/interleaving report.

## What This Changes

The issue reports stale and duplicate Telegram direct-chat replies when user messages, sub-agent completion announcements, tool work, and heartbeat events overlap.

## Best Possible Solution

Add turn-bound delivery semantics that prevent background completion or heartbeat events from being emitted as replies to unrelated newer Telegram user messages, with regression coverage for the reported interleaving and a clear relationship to #74286.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows a concrete plausible path through active requester announce steering/direct fallback and abort-only Telegram supersession, but the exact Telegram transcript sequence still needs an executable regression.

## Solution Assessment

Unclear. The requested end state is right, but the best maintainable fix needs a maintainer decision on whether background completions should be suppressed, rebound to the original turn, or delivered as explicit proactive updates.
