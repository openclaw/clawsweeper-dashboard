## Summary

Keep open. Current main still has source-visible paths where background sub-agent completion output can be steered or directly delivered into an active Telegram requester without a per-inbound turn guard, while Telegram stale-final suppression currently covers authorized abort commands rather than ordinary newer user messages.

## What This Changes

Ensure Telegram direct-chat replies stay bound to the current inbound message when user messages, sub-agent completions, tool work, and heartbeat events overlap.

## Best Possible Solution

Add turn-bound delivery semantics for background completion and heartbeat events so late work is either suppressed, rebound to its original Telegram message, or delivered as an explicit background update with regression coverage for the reported interleaving.

## Reproduction Assessment

Yes, for source-level reproduction: current main can steer or direct-fallback sub-agent completion output into an active Telegram requester without a requester inbound-turn guard. I did not live-verify the full Telegram heartbeat/tool sequence.

## Solution Assessment

Unclear. A per-turn generation or delivery binding is the likely direction, but maintainers need to choose whether late background completions should be suppressed, rebound to the original Telegram message, or delivered as explicit background updates.
