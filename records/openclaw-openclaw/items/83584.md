## Label Justifications

- `P2`: The issue affects optional but real OpenAI-compatible HTTP integrations with bounded blast radius and meaningful media-delivery impact.
- `impact:security`: Raw local `MEDIA:/...` paths can be exposed to HTTP clients instead of being stripped or authorized.
- `impact:message-loss`: Generated images, audio, and documents can be dropped or rendered as unusable directive text on the HTTP surface.

## Summary

Keep open. Current `main` and latest `v2026.6.8` still serialize both OpenAI-compatible HTTP endpoints through text-only assistant output while the agent/media paths carry structured media metadata; the remaining work needs maintainer product/security direction for the public HTTP media shape and authorization model.

## What This Changes

The issue requests translating outbound OpenClaw `MEDIA:` attachments on `/v1/responses` and `/v1/chat/completions` into safe structured HTTP media output instead of exposing or dropping raw directive text.

## Best Possible Solution

Add a shared outbound-media response shaping layer for both OpenAI-compatible HTTP endpoints that strips raw `MEDIA:` from visible text, consumes `mediaUrl`/`mediaUrls`, validates references under the existing outbound media policy, and emits a documented authenticated media representation with streaming and non-streaming parity.

## Reproduction Assessment

Yes, at source level. Current `main` and latest `v2026.6.8` still drop `payloads[].mediaUrl` and `payloads[].mediaUrls` in both OpenAI-compatible HTTP serializers while the agent and media parsing paths carry those fields; no live Gateway request was run in this read-only review.

## Solution Assessment

Unclear as an implementation plan. Stripping raw `MEDIA:` text is necessary, but maintainers still need to choose the safe public HTTP media shape and authorization behavior before a repair PR changes the endpoint contract.
