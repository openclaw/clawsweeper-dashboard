## Label Justifications

- `P2`: This is a real Gateway HTTP media-delivery and filesystem-path exposure bug with bounded scope to authenticated OpenAI-compatible endpoint users.
- `impact:security`: The current response path can expose local filesystem paths from `MEDIA:` directives instead of converting or suppressing them.
- `impact:message-loss`: Generated images, audio, and documents are dropped or rendered as unusable directive text for OpenAI-compatible clients.

## Summary

Keep open. Current main still serializes the OpenAI-compatible HTTP surfaces as text-only assistant output while media already exists as delivery metadata elsewhere, and the remaining fix needs maintainer product/security direction for the public media shape and authorization model.

## What This Changes

The issue requests translating outbound `MEDIA:<path>` artifacts from Gateway OpenAI-compatible responses into safe image/file response content instead of passing raw directive text to clients.

## Best Possible Solution

Add a shared outbound-media shaping layer for the HTTP endpoints that strips `MEDIA:` from visible text, validates media references under existing outbound policy, and emits the maintainer-chosen authenticated URL, data URI, or compatibility fallback for each endpoint and streaming mode.

## Reproduction Assessment

Yes, at source level. Both HTTP handlers serialize only text from agent payloads while the agent result contract and shared media parser carry structured media metadata; I did not run a live gateway request in this read-only review.

## Solution Assessment

Unclear as a concrete implementation plan. Removing raw `MEDIA:` from visible HTTP text is necessary, but maintainers still need to choose the safe public media shape, authorization behavior, and streaming compatibility before implementation.
