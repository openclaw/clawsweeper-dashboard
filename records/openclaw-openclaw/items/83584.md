## Label Justifications

- `P2`: This is a real Gateway compatibility/media bug on authenticated OpenAI-compatible endpoints with limited blast radius.
- `impact:security`: The issue is about local filesystem paths being exposed to HTTP clients instead of being treated as non-text delivery metadata.
- `impact:message-loss`: Generated images, audio, and documents are not delivered as usable media through the affected HTTP client surface.

## Summary

Keep open: current main still serializes assistant payload text directly on both OpenAI-compatible HTTP endpoints, while `MEDIA:` directives are parsed and stripped only in shared/channel delivery paths. The root cause is clear, but the fix needs a maintainer security/product decision on the HTTP media exposure contract.

## What This Changes

The issue requests translating outbound `MEDIA:<path>` assistant directives on `/v1/responses` and `/v1/chat/completions` into structured OpenAI-compatible media instead of returning raw directive text.

## Best Possible Solution

Choose and document the HTTP outbound-media contract, then implement one shared response-shaping path that strips `MEDIA:` from visible text and emits authorized media references for both streaming and non-streaming Chat Completions and Responses.

## Reproduction Assessment

Yes from source inspection: both HTTP handlers serialize assistant payload text directly while the shared parser that strips and separates `MEDIA:` directives is not used on those paths. I did not run a live gateway request in this read-only review.

## Solution Assessment

Unclear as an implementation plan: translating outbound media is the right end state, but maintainers should first choose the safe HTTP response shape, authorization model, and streaming semantics.
