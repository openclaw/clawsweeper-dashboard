## Label Justifications

- `P2`: This is a real Gateway HTTP compatibility and media-delivery bug with bounded blast radius on optional authenticated endpoints.
- `impact:security`: The reported text-only path can expose host-local `MEDIA:/...` paths to HTTP clients instead of treating them as delivery metadata.
- `impact:message-loss`: Generated images, audio, and document artifacts are dropped or rendered as unusable directive text for OpenAI-compatible clients.

## Summary

Keep open. Current main still serializes both OpenAI-compatible HTTP surfaces as text-only assistant output while the shared outbound pipeline already separates `MEDIA:` directives into media metadata; the remaining fix needs maintainer product/security direction for the public media representation and authorization model.

## What This Changes

The issue requests translating outbound `MEDIA:` directives on `/v1/responses` and `/v1/chat/completions` into structured or authorized media output instead of exposing raw assistant text.

## Best Possible Solution

Add a shared outbound-media response shaping layer for the OpenAI-compatible HTTP endpoints that strips raw `MEDIA:` text, preserves media metadata, and emits a documented authorized URL/data representation or explicit compatibility fallback for streaming and non-streaming responses.

## Reproduction Assessment

Yes at source level. Current main joins only assistant text for both HTTP endpoints while the agent result and outbound delivery contracts carry `mediaUrl` / `mediaUrls`; I did not run a live Gateway request in this read-only review.

## Solution Assessment

Unclear as a concrete implementation plan. Stripping raw `MEDIA:` text is necessary, but maintainers still need to choose the safe HTTP media shape and authorization behavior before code changes the endpoint contract.
