## Label Justifications

- `P2`: This is a real HTTP integration bug with security and delivery impact, but it is bounded to OpenAI-compatible gateway clients and needs product/security design before implementation.
- `impact:security`: The reported failure can expose host-local filesystem paths to HTTP clients as raw assistant text.
- `impact:message-loss`: Generated media attachments are dropped from the HTTP response shape even though the agent result carries media metadata.

## Summary

Keep open. Current main and v2026.6.8 still serialize both OpenAI-compatible HTTP endpoints through text-only assistant output while the agent result contract carries media metadata, so the reported media loss/path-leak problem remains and needs maintainer product/security direction for the public HTTP media shape.

## What This Changes

Translate agent outbound media from `MEDIA:`/`mediaUrl(s)` on `/v1/responses` and `/v1/chat/completions` into safe HTTP-visible content instead of exposing raw local paths as assistant text.

## Best Possible Solution

Design one security-reviewed outbound-media contract for the OpenAI-compatible HTTP surfaces: strip raw directives, consume `mediaUrl(s)`, expose only scoped URLs or data content, update endpoint schemas/docs, and add focused tests for text, image, file, and denied-media cases.

## Reproduction Assessment

Yes, as a high-confidence source reproduction: both HTTP serializers read only payload text while the result type carries `mediaUrl(s)`, so media is not translated on current main. I did not run a live gateway/client reproduction in this read-only sweep.

## Solution Assessment

Unclear. The issue is valid, but the exact public shape for Chat Completions versus Responses and the authorization model for generated file access need maintainer product/security approval before this becomes a safe code-only fix.
