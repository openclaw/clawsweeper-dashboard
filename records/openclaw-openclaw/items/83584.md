## Label Justifications

- `P2`: This is a real Gateway HTTP compatibility and media-delivery bug on authenticated optional endpoints, with meaningful but bounded blast radius.
- `impact:security`: The current text-only serialization can expose host-local `MEDIA:/...` paths to HTTP clients instead of treating them as delivery metadata.
- `impact:message-loss`: Generated images, audio, and document artifacts are dropped or rendered as unusable directive text for OpenAI-compatible clients.

## Summary

Keep open. Current main and the latest release still serialize both OpenAI-compatible HTTP surfaces through text-only assistant output, while the shared agent result/media paths carry structured media metadata and already strip `MEDIA:` directives elsewhere; this still needs a maintainer product/security decision for the public HTTP media shape and authorization model.

## What This Changes

The issue asks OpenClaw’s OpenAI-compatible HTTP endpoints to translate agent outbound `MEDIA:` attachments into safe structured media output instead of leaking the raw directive as text.

## Best Possible Solution

Add one shared outbound-media response shaping layer for both OpenAI-compatible HTTP endpoints that strips `MEDIA:` from visible text, preserves structured media metadata, validates local and remote references under the existing outbound policy, and emits a documented authenticated URL, data URI, or explicitly chosen compatibility fallback for streaming and non-streaming responses.

## Reproduction Assessment

Yes, at source level. Both HTTP handlers still read only payload text or assistant deltas while the agent result contract and shared outbound pipeline carry media metadata; I did not run a live Gateway request in this read-only review.

## Solution Assessment

Unclear as a concrete implementation plan. Stripping raw `MEDIA:` text is necessary, but maintainers still need to choose the safe public HTTP media shape and authorization behavior before a repair PR changes the endpoint contract.
