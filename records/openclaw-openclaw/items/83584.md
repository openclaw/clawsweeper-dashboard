## Label Justifications

- `P2`: This is a real Gateway HTTP compatibility and media-delivery bug with a bounded endpoint surface, but the endpoint is authenticated and disabled by default.
- `impact:security`: The reported behavior can expose host filesystem paths from local `MEDIA:` directives to OpenAI-compatible HTTP clients.
- `impact:message-loss`: Generated images, audio, and document artifacts are not delivered as usable media through the OpenAI-compatible HTTP response surfaces.

## Summary

Keep open. Current main still drops structured outbound media on both OpenAI-compatible HTTP endpoints and can expose raw `MEDIA:` text, while the safe response shape needs maintainer product/security direction before a repair PR.

## What This Changes

The issue requests translating agent outbound `MEDIA:` directives and media payload fields into safe OpenAI-compatible HTTP media output instead of returning raw directive text.

## Best Possible Solution

Add a shared outbound-media shaping path for both HTTP endpoints that strips visible `MEDIA:` directives, validates local and remote media with the existing outbound policy, and emits a documented authenticated URL or data-URI response shape for streaming and non-streaming responses.

## Reproduction Assessment

Yes, from source inspection: both HTTP handlers serialize assistant payload text directly or stream text deltas, while the shared channel parser separates `MEDIA:` directives into media metadata. I did not run a live Gateway request in this read-only review.

## Solution Assessment

Unclear as a concrete implementation plan: stripping the raw directive is required, but the right outbound media shape differs between `/v1/chat/completions` and `/v1/responses` and should be chosen by maintainers before changing the public HTTP contract. ([platform.openai.com](https://platform.openai.com/docs/api-reference/chat/getMessages?lang=javascript&utm_source=openai))
