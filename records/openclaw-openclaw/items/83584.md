## Label Justifications

- `P2`: This is a real Gateway compatibility and media-delivery bug on optional authenticated HTTP endpoints, with meaningful but bounded blast radius.
- `impact:security`: The current text-only serialization can expose host-local `MEDIA:/...` paths to HTTP clients instead of treating them as non-text delivery metadata.
- `impact:message-loss`: Generated images, audio, and document artifacts can be dropped or rendered as unusable directive text for OpenAI-compatible clients.

## Summary

Keep open. Current main still serializes the OpenAI-compatible HTTP surfaces from assistant text only, while the existing outbound media pipeline already treats `MEDIA:` as delivery metadata; fixing this needs a product and security decision for the public HTTP media shape.

## What This Changes

The issue asks OpenClaw to translate outbound `MEDIA:<path-or-url>` directives on `/v1/responses` and `/v1/chat/completions` into safe structured media output instead of returning the directive as assistant text.

## Best Possible Solution

Add a shared outbound-media shaping layer for both HTTP endpoints that strips `MEDIA:` from visible text, preserves media metadata, validates local and remote references under the existing outbound policy, and emits a documented authenticated URL, data URI, or explicit compatibility fallback for streaming and non-streaming responses.

## Reproduction Assessment

Yes, at source level. Both HTTP handlers read only payload text or assistant deltas while the result contract and outbound pipeline carry media metadata; I did not run a live Gateway request in this read-only review.

## Solution Assessment

Unclear as a concrete implementation plan. Stripping raw `MEDIA:` text is necessary, but maintainers still need to choose the safe public HTTP media shape, authorization behavior, and compatibility default before a repair PR changes the endpoint contract.
