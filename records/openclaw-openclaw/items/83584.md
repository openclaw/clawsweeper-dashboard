## Label Justifications

- `P2`: This is a real Gateway compatibility/media bug with a limited surface and endpoints that are disabled/authenticated by default.
- `impact:security`: The reported behavior exposes host filesystem paths to HTTP clients instead of treating local `MEDIA:` directives as non-text delivery metadata.
- `impact:message-loss`: Generated images, audio, and documents are not delivered as usable media to OpenAI-compatible clients.

## Summary

Keep open. Current main still serializes OpenAI-compatible HTTP assistant output as plain text only, while the shared media parser and channel reply path already treat `MEDIA:` lines as delivery metadata; the remaining work is real but needs a maintainer decision on the safe HTTP output shape.

## What This Changes

The issue asks the Gateway OpenAI-compatible HTTP endpoints to translate outbound `MEDIA:<path-or-url>` directives into structured media output instead of returning the directive text to clients.

## Best Possible Solution

Add one shared outbound-media response shaping path for both HTTP endpoints that strips `MEDIA:` from visible text, validates local/remote media with the existing outbound policy, and emits a documented authenticated URL or data URI shape for streaming and non-streaming responses.

## Reproduction Assessment

Yes from source inspection: both HTTP handlers currently serialize agent payload text directly, while the shared parser would strip and separate `MEDIA:` directives on channel paths. I did not run a live gateway request in this read-only review.

## Solution Assessment

Unclear as an implementation plan: translating outbound media is the right end state, but maintainers should choose the safe HTTP media exposure shape before a repair PR changes the public response contract.
