## Label Justifications

- `P2`: This is a real Gateway compatibility and media-delivery bug on authenticated optional HTTP endpoints, with meaningful but bounded blast radius.
- `impact:security`: The current text-only serialization can expose host-local `MEDIA:/...` paths to HTTP clients instead of treating them as non-text delivery metadata.
- `impact:message-loss`: Generated images, audio, and document artifacts are dropped or rendered as unusable directive text for OpenAI-compatible clients.

## Summary

Keep open. Current main still serializes both OpenAI-compatible HTTP surfaces through text-only assistant output, while the shared outbound media path already treats `MEDIA:` lines and `mediaUrl`/`mediaUrls` as delivery metadata; the remaining fix needs a maintainer product/security decision for the public HTTP media shape and authorization model.

## What This Changes

The issue requests translating outbound `MEDIA:` directives and `mediaUrl`/`mediaUrls` from `/v1/responses` and `/v1/chat/completions` into safe OpenAI-compatible media output instead of leaking raw text.

## Best Possible Solution

Add one shared outbound-media response shaping layer for both HTTP endpoints that strips `MEDIA:` from visible text, preserves structured media metadata, validates local and remote references under the existing outbound policy, and emits a documented authenticated URL, data URI, or explicitly chosen compatibility fallback for streaming and non-streaming responses.

## Reproduction Assessment

Yes, at source level. Both HTTP handlers read only payload text or assistant deltas while the result contract and outbound pipeline carry media metadata; I did not run a live gateway request in this read-only review.

## Solution Assessment

Unclear as a concrete implementation plan. Stripping raw `MEDIA:` text is necessary, but maintainers still need to choose the safe public HTTP media shape and authorization behavior before a repair PR changes the endpoint contract.
