## Summary

Keep open. The revocation bug is real on current main and the log-then-terminate direction is narrow, but this PR is now stale against current main because it would re-mark secrets-reloader system events as trusted and it still lacks the required changelog entry.

## What This Changes

The PR updates gateway WebSocket revocation helpers to warn on `socket.close` failures, fall back to `terminate`, thread revocation loggers through reload paths, add unit tests, and mark one secrets-reloader system event as trusted.

## Best Possible Solution

Rebase the PR, keep the focused revocation log-and-terminate implementation, preserve current main's untrusted secrets-event handling, add an `Unreleased` changelog fix entry, then run the targeted gateway tests and changed gate before merge.

## Reproduction Assessment

Yes. Current main exposes the issue directly in the three bare `catch` blocks, and the PR's unit-test approach reproduces it by injecting throwing `socket.close` calls into the shared-auth and device-removal revocation helpers.

## Solution Assessment

No for the current PR head. The revocation fix is the narrow maintainable approach, but the stale `trusted: true` hunk should be dropped or changed to preserve current main's `trusted: false`, and the changelog entry still needs to be added.
