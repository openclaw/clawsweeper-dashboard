## Summary

Keep open. Current main has usage visibility, cost summaries, model fallback after provider-side rate-limit errors, context sizing knobs, and opt-in tool-loop detection, but it still does not provide built-in policy enforcement for per-request, per-session, daily token budgets, request caps, or rate limits before model dispatch.

## What This Changes

Request built-in token budget and request limit controls across request, session, daily, rate-limit, warning/interruption/fallback, and loop-detection scopes.

## Best Possible Solution

Define an additive core budget-policy contract at the gateway or agent-runtime model-dispatch boundary, with config/schema/docs/status visibility and tests for token, cost, and request-count decisions; keep loop detection configurable but separate unless maintainers intentionally fold it into the same policy surface.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request; the high-confidence verification path is source/docs inspection showing the requested config keys and pre-dispatch enforcement path are absent on current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the policy contract; a narrow implementation after that design decision would be safer than an autonomous broad fix that invents config shape, accounting authority, and fallback semantics.
