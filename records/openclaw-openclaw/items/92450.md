## Summary

Keep open: Telegram Bot API 10.1 introduced native rich-message table blocks on June 11, 2026, while current OpenClaw main still converts Markdown tables to text-oriented fallback formats and sends them through the older HTML message path.

## What This Changes

Add Telegram Bot API 10.1 rich-message table delivery for outbound Markdown tables while preserving existing send, edit, media, and streaming fallbacks.

## Best Possible Solution

Build Telegram InputRichMessage blocks from the existing Markdown IR, use the rich send/edit/draft methods consistently for supported table-bearing replies, and retain the current HTML path for unsupported content and recoverable API failures.

## Reproduction Assessment

Yes, at source level: current main transforms Markdown tables into text/code representations and contains no Telegram Bot API 10.1 rich-message method or native table-block serializer. The reporter's screenshot is consistent with that path.

## Solution Assessment

Yes in direction, but the best fix is a coherent Telegram rich-message delivery mode spanning final sends, edits, and streaming drafts—not a table detector added to one sendMessage call.
