## What This Changes

Add native Telegram rich-message rendering for outbound markdown tables while preserving readable fallback and existing send, edit, media, reply, and streaming behavior.

## Best Possible Solution

Add one Telegram-owned eligibility and transport layer that builds rich table payloads from original markdown for persistent send/edit paths, preserves reply and preview options, and normalizes rich-only constructs into the current HTML/plain fallback; keep answer drafts out of scope unless separately approved and proven live.

## Reproduction Assessment

Yes at source level: current main always routes tables through configured HTML code-or-bullet fallback, and two live reports show the resulting Telegram output. This review did not independently execute a current-main live Telegram send, so the status is source-reproducible rather than reproduced.

## Solution Assessment

The native rich-message direction is appropriate, but the broad linked implementation is not yet proven to be the best shape. The narrowest maintainable solution should preserve original markdown for rich tables, centralize normalized fallback, and leave persistent streaming semantics unchanged.
