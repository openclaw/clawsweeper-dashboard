## Summary

Keep this open: current main and the latest release still route Telegram table text through HTML/plain sendMessage/editMessageText fallbacks, while the remaining rich-message work is now being handled by open but not-yet-landable Telegram rich-message PRs that need compatibility and proof decisions.

## What This Changes

Request first-class Telegram native rich-message rendering for outbound markdown tables, with safe fallback to existing Telegram text delivery.

## Best Possible Solution

Land a Telegram-owned rich-message path for eligible final text send/edit table messages that preserves original markdown for rich payloads, keeps normalized HTML/plain fallback, and proves current-main upgrade behavior before expanding to streaming or draft surfaces.

## Reproduction Assessment

Yes at source level: current main renders markdown tables through Telegram HTML/plain fallback and has no native rich-message send path. I did not run a live Telegram bot repro in this read-only review, but the issue discussion adds screenshot and live-use evidence.

## Solution Assessment

Yes directionally, but not as an unconstrained transport replacement. The maintainable path is a narrow rich send/edit implementation with explicit fallback and live Telegram proof before broad streaming or draft adoption.
