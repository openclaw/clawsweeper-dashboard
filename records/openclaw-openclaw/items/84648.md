## Summary

Close as ClawHub/plugin work: SafeOps is an optional tool-preflight integration, current main already has a plugin hook and approval seam for this behavior, and the PR hard-codes a generated adapter into core exec with serious default-install and security-boundary risks.

## What This Changes

The PR adds a SafeOps adapter client and invokes it before `exec` dispatch in `src/agents/bash-tools.exec.ts`.

## Best Possible Solution

Publish SafeOps as a code plugin using `before_tool_call` and `requireApproval`; only open a narrow core PR if that plugin path lacks required generic exec metadata.

## Reproduction Assessment

Yes for the main PR regression by source inspection: the patched exec path always calls `beforeToolExecute`, and the adapter returns a deny result when `SAFEOPS_ADAPTER_TOKEN` is absent. I did not run the PR branch because this review was read-only.

## Solution Assessment

No. The maintainable boundary is the existing plugin `before_tool_call` and approval API, with any missing exec metadata handled as a narrow generic API change rather than a hard-coded SafeOps core adapter.
