## Label Justifications

- `P2`: This is a normal-priority optional integration PR with limited product blast radius, but its current core implementation would be unsafe to merge.
- `merge-risk: 🚨 compatibility`: The PR changes existing default exec behavior by denying all exec calls when SafeOps is not configured.
- `merge-risk: 🚨 security-boundary`: The PR inserts an external command-metadata preflight into the exec security boundary before existing approvals.
- `merge-risk: 🚨 availability`: The PR adds a network-dependent preflight to every exec call, so token, adapter, or timeout failures can make exec unavailable.

## Summary

Close as ClawHub/plugin work: SafeOps is an optional exec preflight integration, current main already has plugin hook and approval seams for this policy, and merging a hard-coded core adapter would create availability, compatibility, and security-boundary regressions.

## What This Changes

The PR adds a SafeOps adapter under `src/safeops` and calls it from `exec` before node, gateway, or direct command execution.

## Best Possible Solution

Publish SafeOps as a ClawHub code plugin using `before_tool_call` and `requireApproval`; only open a narrow core API proposal if that plugin path lacks generic exec metadata.

## Reproduction Assessment

Yes for the introduced PR regression by source inspection: `exec` always calls SafeOps, and the adapter denies when `SAFEOPS_ADAPTER_TOKEN` is absent. I did not run the PR branch because this review was read-only.

## Solution Assessment

No. The maintainable solution is the existing plugin `before_tool_call` and approval API, with any missing exec metadata handled as a narrow generic extension point rather than a hard-coded SafeOps core adapter.
