## Summary

Keep open: this recent maintainer-labeled Telegram reliability PR has a concrete correctness concern around duplicate-safe Telegram send retries, and the PR body does not provide after-fix live Telegram proof for the transport/reconnect path.

## What This Changes

Adds a Telegram polling-success hook that calls the delivery-queue reconnect drain for account-scoped pending Telegram entries, plus polling-session regression tests and a changelog entry.

## Best Possible Solution

Land a Telegram reconnect drain that reuses the shared delivery queue, but keep immediate backoff bypass limited to duplicate-safe send failures and require live Telegram transport proof before merge.

## Reproduction Assessment

Yes, source-level: current main's Telegram polling success path only updates status and does not call the shared reconnect delivery drain, while WhatsApp already uses that drain after reconnect. I did not run a live stale-socket injection in this read-only review.

## Solution Assessment

No, not yet. The hook belongs at the right ownership boundary, but the backoff-bypass predicate should preserve the existing duplicate-safe Telegram send contract and the transport behavior still needs real Telegram proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Keep reconnect drains duplicate-safe:** `extensions/telegram/src/polling-session.ts:56-59`
  - body: The new reconnect drain bypasses delivery backoff whenever the stored error string broadly matches a Telegram network-send failure. Existing Telegram send retry coverage intentionally refuses to treat a plain `Network request for 'sendMessage' failed!` envelope as safe because Telegram may have accepted the visible message before the client lost the response; bypassing backoff here can immediately replay the queued reply after reconnect and duplicate a user-visible Telegram message. Keep the drain, but only bypass backoff for the existing duplicate-safe send predicate or preserved low-level pre-connect evidence.
  - confidence: 0.86
