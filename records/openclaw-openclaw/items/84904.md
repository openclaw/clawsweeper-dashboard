## Summary

Keep open: the reported Responses replay bug is valid, but this PR should not merge as-is because it strips item IDs from every builder-produced Responses payload, including runtime paths that later force `store: true`, and it lacks after-fix real behavior proof.

## What This Changes

The PR changes the OpenAI Responses parameter builder and regression tests so prior reasoning, assistant text, and function-call output item IDs are omitted from replay payloads when building store-disabled Responses requests.

## Best Possible Solution

Keep the stateless `store: false` fix, but make replay IDs follow the final store policy and add coverage for both direct OpenAI `store: true` and proxy/custom `store: false` paths.

## Reproduction Assessment

Yes, by source inspection: current main builds Responses requests with `storeMode: "disable"` while still replaying prior item IDs for non-native-Codex Responses paths. I did not run a live provider reproduction in this read-only review.

## Solution Assessment

No, not as written: omitting IDs for `store: false` is the right direction, but the PR computes the gate before provider wrappers that can force final `store: true`. The maintainable fix is to preserve the existing direct OpenAI store-enabled path while disabling IDs for final store-disabled requests.
