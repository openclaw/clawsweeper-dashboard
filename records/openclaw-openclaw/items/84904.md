## Summary

Keep this PR open for normal merge review: the force-pushed version addresses the earlier ClawSweeper blocker by tying Responses item-id replay to the final store policy and now includes redacted real gateway proof for the store-disabled failure path. I found no discrete blocking correctness or security finding, but the change remains compatibility-sensitive because it alters session replay payloads across OpenAI Responses providers.

## What This Changes

The PR adds a replay-item-id option to the OpenAI Responses transport/wrapper path and updates regression tests so store-disabled requests omit prior server item IDs while direct OpenAI store-enabled requests preserve them.

## Best Possible Solution

Land the store-policy-aware replay gate with focused regression coverage, preserving item IDs only for final store-enabled requests and stripping them for store-disabled payloads.

## Reproduction Assessment

Yes. Source inspection on current main shows store-disabled Responses requests can still replay prior server-owned item IDs, and the PR body supplies redacted real gateway logs matching that failure; I did not rerun a live provider in this read-only review.

## Solution Assessment

Yes. Threading the final store policy into replay ID selection is the narrow maintainable fix: it avoids stale item IDs for `store:false` without changing direct OpenAI `store:true` replay semantics.
