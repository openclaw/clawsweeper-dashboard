## Summary

Keep open: the linked bug is valid, but this PR moves the hint into the actual config-write path and would print `openclaw doctor --fix` guidance after repairs were already applied in a non-`--fix` interactive run.

## What This Changes

The PR moves the doctor `--fix` trailer into the write-config branch and adds doctor health contribution tests for clean, pending, and `--fix` cases.

## Best Possible Solution

Keep the clean-run suppression, but base actionable `--fix` guidance on the existing pending-change/fix-hint paths rather than on `shouldWriteConfig` after a successful write.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a clean doctor run starts with matching `cfg` and `cfgForPersistence`, reaches the no-write branch with `shouldRepair: false`, and current main logs the trailer there.

## Solution Assessment

No. Moving the trailer into `shouldWriteConfig` fixes the clean no-write branch but misuses a flag that means the config is being written, so the narrower maintainable fix should preserve or emit hints only where pending repairs remain unapplied.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep the trailer out of the write branch:** `src/flows/doctor-health-contributions.ts:607-608`
  - body: `shouldWriteConfig` is set when doctor is about to persist changes, including a normal interactive run where the user accepts "Apply recommended config repairs now?" without `--fix`. The hint should stay on a pending/unapplied path, not after `replaceConfigFile()` succeeds.
