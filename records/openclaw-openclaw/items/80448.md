## Summary

Keep open: the PR targets a real doctor output bug, but the patch is not mergeable because it uses `shouldWriteConfig` as pending-change state, current CI shows new test type errors, and the supplied proof is only a mocked Vitest run.

## What This Changes

The PR moves the doctor `--fix` trailer into the config-write branch and adds regression tests for clean, pending, and `--fix` doctor write-config runs.

## Best Possible Solution

Carry an explicit pending/unapplied config-guidance state through the doctor config flow, suppress the trailer on clean runs, preserve actionable repair hints, fix the new test typing, and add live CLI proof.

## Reproduction Assessment

Yes. Source inspection and the linked report give a high-confidence path: after a clean `doctor --fix`, a clean non-repair doctor run reaches the no-write branch and current main logs the trailer solely because `shouldRepair` is false.

## Solution Assessment

No. Moving the trailer under `shouldWriteConfig` suppresses the clean run, but it uses the write decision as a proxy for unapplied pending repairs; the safer fix should model the pending/unapplied state directly.
