## Summary

Keep this PR open: it targets a real doctor output bug, but the current branch still emits the `--fix` hint after successful config writes and the after-fix proof is only a mocked/focused test run.

## What This Changes

The PR moves the doctor `--fix` trailer into the write-config branch and adds doctor health contribution tests for clean, pending-change, and `--fix` cases.

## Best Possible Solution

Suppress the generic trailer on clean no-write runs, but keep actionable `--fix` guidance tied to pending/unapplied repair hints rather than to the successful write path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path for the reported bug on current main, and the PR regression is also source-reproducible through accepted pending repairs returning `shouldWriteConfig: true` before `runWriteConfigHealth` writes the config.

## Solution Assessment

No. Moving the trailer into `shouldWriteConfig` fixes the clean no-write branch but makes successful writes print stale `--fix` guidance; the narrower fix should distinguish pending-unapplied hints from already-applied writes.
