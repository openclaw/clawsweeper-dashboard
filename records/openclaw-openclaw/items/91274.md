## Summary

Keep open: the cleanup is plausible, and the supplied live proof is useful, but the PR is not safe to merge as a standalone change because current QMD list output does not expose collection paths and the new migration can keep a stale unscoped collection when both old names exist.

## What This Changes

The PR changes QMD memory collection names from agent-suffixed values to unscoped base names and reverses the legacy migration from unscoped-to-scoped into scoped-to-unscoped.

## Best Possible Solution

Land the de-scoping only after the branch or base includes path-aware QMD collection reconciliation, a both-existing legacy-state regression test, and matching documentation updates.

## Reproduction Assessment

Yes. Source plus the QMD 2.5.3 CLI contract show that a both-existing scoped/unscoped collection state can be listed without paths, and this PR then skips migration/rebind for the unscoped name.

## Solution Assessment

No, not as a standalone merge. The cleanup becomes maintainable if it lands with the path-aware rebind behavior or after the related rebind PR is merged, plus docs and regression coverage for the upgrade case.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Handle both legacy names before switching targets:** `extensions/memory-core/src/memory/qmd-manager.ts:767-770`
  - body: `migrateLegacyScopedCollections()` currently returns as soon as the new unscoped name already exists. With QMD 2.5.x, `collection list` does not expose the path, and `shouldRebindCollection()` therefore treats an existing unscoped row as valid when the pattern matches. An upgrade from a state containing both `team-library` and `team-library-<agent>` can keep using the stale unscoped collection and leave the scoped one behind, which is the risky case this migration is meant to clean up. Please fold in a path-aware `collection show`/rebind path or add a migration that handles the both-existing case before changing the runtime names.
  - confidence: 0.87
- **[P3] Update the QMD collection-name documentation:** `packages/memory-host-sdk/src/host/backend-config.ts:348`
  - body: This change makes fresh QMD indexes use `memory-root` and `memory-dir`, but `docs/concepts/memory-qmd.md` still tells users to run `qmd search ... -c memory-root-main -c memory-dir-main`. After this line changes the canonical default names, that documented command no longer matches the collections OpenClaw creates.
  - confidence: 0.9
