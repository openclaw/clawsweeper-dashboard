## Summary

Keep open. The PR is directionally solid and has strong real-environment proof, but one upgrade edge remains: disambiguated custom QMD collections can miss the scoped-to-unscoped migration and then fail to add under the new unscoped name.

## What This Changes

The PR removes agent-id suffixing from managed QMD collection names, adds scoped-to-unscoped migration and rebind logic, and updates memory tests, config help, and docs.

## Best Possible Solution

Keep the unscoped naming direction only after the scoped-to-unscoped migration handles disambiguated custom legacy names with path-verified rebind coverage.

## Reproduction Assessment

Yes, at source level. A config with duplicate custom collection names can have shipped names like `notes-main` and `notes-main-2`, while this PR derives the second legacy probe as `notes-2-main` from the new disambiguated name.

## Solution Assessment

No, not yet. The unscoped naming approach is maintainable, but the migration needs a path-verified repair for disambiguated custom legacy conflicts before it is the best upgrade-safe solution.
