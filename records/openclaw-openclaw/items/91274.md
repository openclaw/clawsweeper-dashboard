## Summary

Keep open for maintainer review: the patch is a coherent implementation of the unscoped QMD collection-name proposal and I found no new blocking code finding, but it intentionally changes shipped derived-index identifiers and session collection names, so the remaining question is maintainer acceptance of the upgrade/reindex tradeoff.

## What This Changes

The PR renames managed QMD memory, custom, and session collections from agent-suffixed names to unscoped names, reverses the legacy collection migration, updates docs/help text, and adds upgrade/rebind regression coverage.

## Best Possible Solution

If maintainers accept the unscoped naming contract, land this shape after a current-main refresh with the scoped-to-unscoped migration, docs/help updates, and upgrade proof intact; otherwise keep the scoped identifiers and close the linked proposal.

## Reproduction Assessment

Not applicable as a refactor/compatibility PR rather than a bug report. Current main and v2026.6.6 source prove the existing scoped behavior, while the PR body supplies after-fix real QMD upgrade proof.

## Solution Assessment

Yes, if maintainers choose unscoped names: the change stays in the memory config resolver and QMD manager, updates docs/help/tests, and uses show-verified rebinds for upgrade safety. The safer alternative is to keep scoped names as the compatibility contract.
