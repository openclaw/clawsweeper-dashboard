## Summary

Keep open: the refactor is useful and the supplied real-environment proof is strong, but one upgrade-safety defect remains where custom legacy scoped collections can be removed without path or pattern ownership verification on name-only QMD list output.

## What This Changes

The PR removes agent-id suffixes from managed QMD collection names, reverses QMD migration to scoped-to-unscoped cleanup/rebind, and updates memory docs and tests.

## Best Possible Solution

Land the unscoped naming cleanup only after custom legacy collection removal requires verified ownership metadata or legacy-entry path enrichment, with the upgrade tests covering that case.

## Reproduction Assessment

Yes, at source level for the remaining review finding: a configured custom collection named `notes` with existing name-only `notes-<agentId>` list output reaches `canMigrateLegacyCollection` and returns true without verified path or pattern metadata. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet. Unscoped names are a reasonable cleanup because each agent has its own QMD index, but the custom legacy migration needs a stricter ownership check before this is the best safe upgrade path.
