## Summary

Keep open: the cleanup is useful and the supplied real-behavior proof is good, but one upgrade path still leaves a legacy session collection orphaned when a user-defined custom collection already owns the new `sessions` name.

## What This Changes

The PR removes agent-id suffixes from managed QMD collection names, reverses QMD collection migration from scoped to unscoped names, adds rebind/orphan cleanup tests, and updates related memory docs.

## Best Possible Solution

Keep the unscoped naming direction only after the scoped-to-unscoped migration handles the session-name collision upgrade path and proves it with a focused regression test.

## Reproduction Assessment

Yes. Source inspection shows that a custom collection named `sessions` makes the managed sessions collection resolve as `sessions-2`, while migration derives `sessions-2-<agentId>` and misses the shipped `sessions-<agentId>` collection.

## Solution Assessment

No. The overall unscoped-name approach is plausible, but the migration is not complete until engine-owned session cleanup is based on the original engine base name rather than the collision-resolved runtime name.
