This PR should stay open because it is a draft, carries the protected `maintainer` label, and proposes a new core CLI surface that needs maintainer product direction. The latest branch is also still functionally blocked: `openclaw lint` is documented as read-only, but its diagnostic paths call the observing config snapshot reader that can write config health state on normal reads.

## What This Changes

The PR adds a top-level `openclaw lint` command, diagnostic registry and runner, core lint checks, CLI startup policy wiring, focused tests, and docs for read-only structured diagnostics.

## Best Possible Solution

## Solution Assessment

No, not as-is. The standalone command is a coherent option, but the PR needs maintainer surface approval and non-observing config snapshot reads before it matches its read-only contract.
