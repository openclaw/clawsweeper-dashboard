## Summary

Keep this PR open: it is a protected, draft, maintainer-labeled feature PR with a coherent standalone lint proposal, but the current diff still violates its own read-only contract and needs maintainer product direction before merge.

## What This Changes

Adds a top-level `openclaw lint` command with JSON, severity, skip, and only options, plus a diagnostic registry/runner, built-in core checks, CLI startup wiring, docs, and focused tests.

## Best Possible Solution

Choose the diagnostics surface first, then land one implementation whose lint checks use a non-observing config snapshot path and reuse that snapshot for final validation.

## Reproduction Assessment

Not applicable for the central request because this PR adds a new CLI feature. The read-only contract bug is source-reproducible from the PR calling `readConfigFileSnapshot()` while current main can write config health state during observed reads.

## Solution Assessment

No, not as-is. The standalone command is a plausible solution, but it needs maintainer surface approval and non-observing config snapshot reads before it matches its stated contract.

- **[P2] Avoid writing config health state from lint:** `src/commands/lint.ts:38`
  - body: `openclaw lint` is documented as read-only, but this path calls `readConfigFileSnapshot()`.   - confidence: 0.88

Even if the top-level lint read is fixed, this check can still mutate config health state during a supposedly read-only lint run, so it should reuse the lint snapshot or use the same non-observing read path.
  - confidence: 0.84
