## Label Justifications

- `P2`: This is a normal-priority persistent session metadata bug with visible user impact but limited blast radius and no crash or security bypass.
- `impact:data-loss`: Replacement upserts can remove stored human-facing `label` and `displayName` values from persistent session records.
- `impact:session-state`: The affected fields are persistent session identity metadata used by session lists and channel-bound UI.

## Summary

Keep open: current main still persists replacement session entries without preserving stored human title fields, and the active release-targeted fix path is still open for maintainer/API-owner review.

## What This Changes

The issue reports that `upsertSessionEntry` can erase persistent session `label` and `displayName` metadata when replacement runtime metadata lacks those title fields.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/93072 with the narrow store-boundary title preservation, regression coverage, and documented exported-helper contract, then close this issue.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: store an entry with non-empty `label` and `displayName`, then call `upsertSessionEntry` for the same key with replacement metadata that omits or clears those fields and read the stored entry back; I did not run a mutating local repro because this review is read-only.

## Solution Assessment

Yes, with maintainer/API-owner acceptance. The store upsert boundary is the narrowest durable fix because every replacement write passes through it, while explicit rename and clear behavior can stay on the validated patch/replace paths.
