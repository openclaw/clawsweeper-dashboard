## Summary

Keep open: current main still has a source-level path where `upsertSessionEntry` persists the replacement entry without preserving existing non-empty human title fields, so the reported session-label loss remains a valid narrow bug.

## What This Changes

The issue requests that replacement session upserts preserve existing non-empty `label` and `displayName` unless the replacement supplies non-empty new values.

## Best Possible Solution

Patch `upsertSessionEntry` to preserve existing non-empty `label` and `displayName` only when the replacement lacks a non-empty value, and add focused regression coverage without changing ACP replacement semantics.

## Solution Assessment

Yes. The narrowest maintainable fix is in the session-store upsert path, preserving existing non-empty human title fields for empty replacement metadata while leaving explicit non-empty replacements and patch/update rename flows intact.

Fix https://github.com/openclaw/openclaw/issues/89003. Current `src/config/sessions/store.ts` `upsertSessionEntry` clones the replacement entry and persists it wholesale, so replacement metadata that omits, nulls, or empties `label`/`displayName` drops existing human session titles. Add a focused fix so existing non-empty `label` and `displayName` are preserved when the replacement lacks a non-empty value, while non-empty replacement values still replace the previous title.
