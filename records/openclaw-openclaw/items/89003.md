## Label Justifications

- `P2`: This is a normal-priority persistent session metadata bug with visible user impact and a bounded storage/session-list blast radius.
- `impact:data-loss`: Replacement upserts can remove stored human-facing `label` and `displayName` values from persistent session records.
- `impact:session-state`: The affected fields are persistent session identity metadata used by session lists and channel-bound UI.

## Summary

Keep open: current main and the latest release still replace stored session entries without preserving existing human title fields, and the open main-targeted fix candidate remains the canonical path rather than a reason to close the issue before merge.

## What This Changes

The issue reports that replacement `upsertSessionEntry` writes can erase persistent session `label` and `displayName` metadata, causing session lists to fall back to less stable origin titles.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/93072 with the narrow store-boundary title preservation and regression coverage, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: store an entry with non-empty `label` and `displayName`, then call `upsertSessionEntry` for the same key with replacement metadata that omits or empties those fields and read the stored entry back.

## Solution Assessment

Yes, subject to maintainer/API-owner compatibility acceptance. The store upsert boundary is the narrow durable fix because replacement writes pass through it, while explicit clears can remain on documented empty-string or patch/replace paths.
