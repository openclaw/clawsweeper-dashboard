## Label Justifications

- `P2`: This is a normal-priority persistent session metadata bug with visible user impact but limited storage-boundary blast radius.
- `impact:data-loss`: Replacement upserts can remove stored human-facing `label` and `displayName` values from persistent session records.
- `impact:session-state`: The affected fields are persistent session identity metadata used by session lists and channel-bound UI.

## Summary

Keep open: current main and the latest release still persist replacement session entries without preserving stored human title fields, and the open main-targeted PR is the active fix candidate awaiting maintainer/API-owner compatibility review.

## What This Changes

Persistent session replacement upserts should preserve stored human `label` and `displayName` fields when incoming runtime metadata omits those title fields.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/93072 with the narrow store-boundary title preservation, regression coverage, and documented helper contract, then close this issue.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: store an entry with non-empty `label` and `displayName`, then call `upsertSessionEntry` for the same key with replacement metadata that omits those fields and read the stored entry back; I did not run a mutating local repro in this read-only review.

## Solution Assessment

Yes, with maintainer/API-owner acceptance. The store upsert boundary is the narrowest durable fix because replacement writes pass through it, while explicit clear behavior can remain on documented empty-string or patch/replace paths.
