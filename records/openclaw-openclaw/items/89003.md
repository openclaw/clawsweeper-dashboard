## Label Justifications

- `P2`: This is a normal-priority persistent session metadata bug with visible user impact but limited blast radius and no crash or security bypass.
- `impact:data-loss`: Replacement upserts can overwrite or remove stored human-facing `label` and `displayName` values.
- `impact:session-state`: The affected fields are persistent session identity metadata used by session lists and channel-bound UI.

## Summary

Keep open: current main and the latest release still persist replacement session entries without preserving human title fields, and the open replacement PR is the current fix candidate.

## What This Changes

Persistent session upserts should preserve existing non-empty `label` and `displayName` values unless replacement metadata explicitly supplies title fields.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/93072 with the narrow store-boundary title preservation, focused regression coverage, and SDK contract wording, then close this issue.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: store a session entry with non-empty `label` and `displayName`, then call `upsertSessionEntry` for the same key with replacement metadata that omits those fields and read the stored entry back.

## Solution Assessment

Yes, with maintainer/API-owner acceptance. The store upsert boundary is the narrowest durable fix because every replacement write passes through it, while explicit rename and clear behavior can stay on the validated patch path.
