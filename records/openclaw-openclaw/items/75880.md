The PR adds a `--headed` browser-start CLI flag, browser-tool `headed` start parameter, `browserStart` headless override option, and docs/changelog coverage for request-local headed browser launches.

## Best Possible Solution

## Solution Assessment

Yes. Reusing the existing `/start?headless=false` contract and carrying it through CLI, client, and browser tool surfaces is the narrow maintainable direction; the remaining safer alternative is coverage/docs polish, not a new route or config mutation.
