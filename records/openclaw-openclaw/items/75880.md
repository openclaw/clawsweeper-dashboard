## Summary

Keep open. Current main still lacks the public headed-start surfaces, and this PR wires the right CLI, browser tool, and client-helper paths, but the main browser tool page still documents only the headless override.

## What This Changes

This PR exposes request-local headed browser starts through `openclaw browser start --headed`, browser tool `headed`/`headless` start parameters, `browserStart({ headless })`, docs, changelog, and focused tests.

## Best Possible Solution

Land one headed-start implementation after `docs/tools/browser.md` also documents `headless=false`, `--headed`, and the browser tool start override alongside the existing headless guidance.

## Reproduction Assessment

Yes. Current main has a working internal `POST /start?headless=false` path, while the CLI, browser tool, and client helper on main cannot send that false override.

## Solution Assessment

Yes for the implementation direction: reusing the existing request-local `/start?headless=` route is the narrow maintainable fix. The remaining issue is docs consistency and duplicate-PR coordination, not runtime design.
