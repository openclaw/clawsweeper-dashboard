## Summary

Keep open. Current main still lacks the public headed-start surface, and this PR wires the feature through the right runtime paths, but the main browser docs page still needs the same headed-start guidance and #75881 is an overlapping open implementation to coordinate.

## What This Changes

This PR adds request-local headed browser start support through `openclaw browser start --headed`, browser tool `headed`/`headless` parameters, the `browserStart` client helper, docs, changelog, and focused tests.

## Best Possible Solution

Land one headed-start implementation after `docs/tools/browser.md` also documents `headless=false`, `--headed`, and the browser tool start override alongside the existing headless guidance.

## Reproduction Assessment

Yes. Current main has a working internal `POST /start?headless=false` route path, while the CLI, browser tool, and `browserStart` client helper on main cannot send that false override.

## Solution Assessment

Yes for the implementation direction: reusing the existing request-local `/start?headless=` route is the narrow maintainable fix. The remaining issue is docs consistency plus choosing between overlapping PRs.
