## Summary

Keep open. The branch is a good narrow implementation path, but it still misses the linked tool API contract because explicit `headless: false` browser-tool starts are ignored rather than forwarded as a request-local headed launch.

## What This Changes

Adds request-local headed browser start wiring through `openclaw browser start --headed`, browser-tool start handling, the `browserStart` client helper, docs, changelog, and CLI regression tests.

## Best Possible Solution

Land one canonical implementation that keeps the request-local route/CLI/client design, supports both `headed: true` and explicit `headless: false` in the browser tool with conflict handling, preserves docs/changelog coverage, and then closes the linked feature issue plus duplicate PRs.

## Reproduction Assessment

Yes. Static inspection of current main reproduces the feature gap, and the latest PR diff reproduces the remaining tool gap because only `params.headed` is read when computing the start override.

## Solution Assessment

No, not as submitted. The route/CLI/client direction is the narrow maintainable solution, but the browser-tool start API should also accept explicit `headless` and reject ambiguous `headless`/`headed` combinations.
