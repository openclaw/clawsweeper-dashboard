## Summary

Keep open: this PR targets a real browser handoff gap, but the proposed browser tool proxy wiring would break node-proxied headed starts by putting `?headless=false` into the route path instead of the separate query object.

## What This Changes

The PR adds a headed runtime browser start surface across the CLI, browser tool schema and handler, `browserStart` client helper, docs, and changelog.

## Best Possible Solution

Revise the PR so headed starts preserve the existing path/query separation across direct, gateway, and node-proxy browser surfaces, with targeted coverage for the proxied browser tool path.

## Reproduction Assessment

Yes. For the PR-introduced blocker, call the browser tool with `action: "start"`, `headed: true`, and a node/proxy target; the proposed path becomes `/start?headless=false`, while the proxy dispatchers only honor `headless` from the separate query object.

## Solution Assessment

No. The CLI and client approach is narrow, but the browser tool proxy branch should keep `path: "/start"` and pass `{ headless: headlessOverride }` via `query` so node-proxied starts use the same route contract as other browser requests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Pass the proxy override through query params:** `extensions/browser/src/browser-tool.ts:570`
  - body: When `proxyRequest` is used, this builds `/start?headless=false` as the route path. The gateway and node proxy stacks dispatch `path` and `query` separately, and `normalizeBrowserRequestPath` does not split `?`, so node-proxied headed starts will miss the `/start` route instead of applying the override. Keep `path: "/start"` and pass the override in `query`.
  - confidence: 0.93
