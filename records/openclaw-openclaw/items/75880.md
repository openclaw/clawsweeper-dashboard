## Summary

Keep open: the PR targets a valid feature gap on current main, but the proposed browser-tool proxy wiring would break node-hosted headed starts by putting the query string into the route path.

## What This Changes

The PR adds a runtime headed browser-start surface across `browserStart`, `openclaw browser start --headed`, the browser tool schema/tool wiring, docs, and changelog.

## Best Possible Solution

Repair the PR by preserving path/query separation in the browser tool proxy path and adding a focused node-proxy test, while keeping the CLI/client/docs direction.

## Reproduction Assessment

Yes: after applying the PR, invoke the browser tool with `{ "action": "start", "target": "node", "headed": true }`; the proxy would receive `path: "/start?headless=false"`, while the dispatcher only matches `/start`. This was verified from the current node proxy and dispatcher contracts rather than a live run.

## Solution Assessment

No: exposing the existing `headless=false` route is the right direction, but the proxy implementation should pass `{ headless: false }` through the `query` object instead of embedding it in `path`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Pass headless through the proxy query object:** `extensions/browser/src/browser-tool.ts:570`
  - body: For node-hosted browsers, `browser.proxy` does not parse query strings out of `path`; it forwards only `params.query` to the dispatcher, which route-matches the path verbatim. With `headed: true`, this sends `/start?headless=false`, misses the `/start` route, and fails the node/browser handoff path; keep `path: "/start"` and send the override via `query`.
  - confidence: 0.92
