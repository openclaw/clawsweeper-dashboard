## Summary

Keep open: current main still has a source-reproducible Feishu group metadata bug where group session state can derive `groupId` from the sender open_id even though the route/session key and reply target use the group chat id.

## What This Changes

The issue reports that Feishu group message sessions store the sender open_id in `groupId` instead of the Feishu group chat ID, preventing group replies.

## Best Possible Solution

Fix the Feishu inbound context so group metadata is derived from the Feishu chat id while sender identity remains available through sender fields and prompt attribution.

## Reproduction Assessment

Yes, at source level: a group Feishu event with `chat_id=oc_*` and `sender.open_id=ou_*` currently reaches shared session metadata with `ChatType=group` and `From=feishu:ou_*`, which derives `groupId=ou_*`. I did not perform an end-to-end Feishu workspace run.

## Solution Assessment

Yes, the best direction is a narrow Feishu plugin fix that supplies group-derived context for session metadata while preserving sender-specific attribution; relaxing the embedded tool policy would be the wrong layer.
