## Summary

Keep open: current main still has the Feishu group metadata bug. The Feishu group route/session key can use the `oc_*` chat id, but the context used for session and embedded-run metadata still derives `groupId` from `From=feishu:ou_*`, matching the reporter's logs.

## What This Changes

Fix Feishu group-message handling so session/tool-policy group context uses the Feishu group `chat_id` (`oc_*`) instead of the sender `open_id` (`ou_*`).

## Best Possible Solution

Fix the Feishu inbound group context or supplied group-resolution seam so group metadata and embedded-run `groupId` come from the `oc_*` chat id while speaker attribution stays tied to the `ou_*` sender.

## Reproduction Assessment

Yes, at source level. A Feishu group event with `chat_id=oc_*` and `sender.open_id=ou_*` still builds `From=feishu:ou_*`, and current group metadata/run code derives `groupId` from `From`, matching the issue logs.

## Solution Assessment

Yes. The narrow fix is to correct Feishu's server-derived group context before it reaches session/tool metadata; relaxing the embedded tool-policy mismatch check would weaken the intended fail-closed guard.
