## Summary

This PR targets a real Telegram data-loss race and should stay open, but the patch is not merge-ready because it can still persist an offset that skips earlier pending updates and it leaves the existing tracker tests and required changelog out of sync.

## What This Changes

The PR moves Telegram update offset persistence from `beginUpdate` to `finishUpdate` for completed updates in `extensions/telegram/src/bot-update-tracker.ts`.

## Best Possible Solution

Keep the deferred-ack direction, but persist the highest contiguous completed update id, refresh the tracker regression tests, and add the required user-facing changelog entry.

## Reproduction Assessment

Yes. Current main’s source shows `acceptUpdateId` running during `beginUpdate`, and the tracker tests demonstrate restart replay is skipped once that accepted offset is restored.

## Solution Assessment

No, not as written. Deferring persistence until handler completion is the right direction, but persisting `update.updateId` directly still allows a later completed update to acknowledge an earlier pending one; the safer fix is to persist the contiguous safe completed watermark.
