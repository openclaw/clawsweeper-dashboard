## Summary

Keep open: the PR is aimed at the right Telegram data-loss window, but it still persists a raw completed update id, which can skip an earlier pending update after restart; the existing tracker tests also still assert the old begin-time persistence policy and the user-facing fix lacks a changelog entry.

## What This Changes

This PR moves Telegram update offset persistence from `beginUpdate` to `finishUpdate` after a completed handler.

## Best Possible Solution

Keep the deferred-ack direction, but persist only the highest contiguous completed update id, refresh the Telegram tracker/bot tests around pending lower ids and restart replay, and add the required changelog entry.

## Reproduction Assessment

Yes. Current-main source shows the offset is accepted in `beginUpdate`, and existing tracker tests show restart replay is skipped once that accepted offset is restored while `safeCompletedUpdateId` can remain behind a pending lower id.

## Solution Assessment

No, not as written. Deferring persistence until completion is the right narrow direction, but persisting `update.updateId` directly is not safe when a later update completes before an earlier one.
