## Summary

Keep open: this PR targets a real Telegram delivery-loss window, but the patch still can advance the persisted offset past lower pending updates and leaves the existing offset tests plus changelog stale.

## What This Changes

The PR branch changes `extensions/telegram/src/bot-update-tracker.ts` to stop acknowledging update ids in `beginUpdate` and instead acknowledge completed update ids in `finishUpdate`.

## Best Possible Solution

Land a corrected Telegram tracker change that persists only the highest contiguous completed update id after awaited handler delivery, keeps pending or failed lower ids replayable, updates tracker/bot regression coverage, and adds a single-line changelog fix entry.

## Reproduction Assessment

Yes from source inspection: current main persists the accepted update id in `beginUpdate`, and existing tests encode restart replay being skipped once that accepted offset is restored. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No, not as written. Deferring persistence until handler completion is the right direction, but the patch should persist the contiguous safe completed watermark rather than the raw completed id, and it needs matching tests and changelog coverage.
