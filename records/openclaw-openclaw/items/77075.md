## Summary

Keep open: the PR targets a valid Telegram offset data-loss bug, but the current patch can still persist past an earlier pending update and is missing the required test and changelog updates.

## What This Changes

The PR moves Telegram update offset persistence from `beginUpdate` to `finishUpdate` after completed handler execution in `extensions/telegram/src/bot-update-tracker.ts`.

## Best Possible Solution

Land a Telegram-owned fix that defers durable offset advancement until the highest contiguous completed update id advances, with regression coverage and a changelog entry.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main persists the Telegram offset during `beginUpdate`, and the durable monitor uses that stored id on restart. The existing tests also demonstrate the lower-pending-update gap that the PR must preserve.

## Solution Assessment

No, not as written. Deferring ack until completion is the right direction, but persisting `update.updateId` directly can still acknowledge earlier pending updates; the safer fix is to persist only the contiguous safe completed watermark.
