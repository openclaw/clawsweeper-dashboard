## Summary

Keep open: this PR addresses a real Telegram update-offset data-loss race, but the patch is not safe as written because it can still persist past earlier pending updates and it leaves required tests and changelog coverage stale.

## What This Changes

The PR moves Telegram update-offset persistence in `extensions/telegram/src/bot-update-tracker.ts` from `beginUpdate` to completed `finishUpdate` handling.

## Best Possible Solution

Keep the deferred-ack direction, but persist only the highest contiguous completed watermark bounded by pending/failed updates, refresh tracker regression tests, and add the changelog entry.

## Reproduction Assessment

Yes. Source inspection shows current main persists the update offset during `beginUpdate`, and the existing tracker tests show restart replay is skipped once that eager accepted offset is restored.

## Solution Assessment

No, not as written. Deferring persistence until handler completion is the right direction, but the PR should persist the safe contiguous completed watermark rather than the raw completed id, then update tests and changelog.
