## Summary

Keep open: the PR targets a valid Telegram data-loss race, but the patch still can persist a later completed update past an earlier pending one and it does not update the tracker tests or changelog.

## What This Changes

The PR moves Telegram update offset persistence from `beginUpdate` to completed `finishUpdate` handling in the Telegram update tracker.

## Best Possible Solution

Keep the deferred-ack direction, but persist only the highest contiguous completed update id, refresh the tracker regression tests, and add a user-facing changelog fix entry.

## Reproduction Assessment

Yes. Current main calls `acceptUpdateId` during `beginUpdate`, and the existing tracker tests encode the restart-from-persisted-offset skip behavior that makes the reported ack-before-delivery race source-reproducible.

## Solution Assessment

No, not as written. Deferring persistence until completion is the right direction, but the narrow maintainable fix should persist the contiguous safe completed watermark rather than the raw completed update id.
