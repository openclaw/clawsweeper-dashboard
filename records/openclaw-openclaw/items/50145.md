## Summary

Keep open. Current main still contains the reported stale replay path, and open PR #50147 is a closing-syntax implementation candidate, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that interrupt queue mode silently discard aborted runs with no new assistant output instead of replaying the previous assistant reply before the next reply.

## Best Possible Solution

Land PR #50147 or an equivalent narrow patch that suppresses payload construction and delivery for aborted interrupt runs with no new assistant output, while preserving the existing valid final-answer fallback for non-aborted runs.

## Reproduction Assessment

Yes. The issue provides concrete interrupt-mode steps, and current-main static inspection confirms the chain: interrupt abort, previous `lastAssistant` from session history, fallback payload construction, and delivery without an aborted-run gate.

## Solution Assessment

Yes. Guarding the aborted empty-attempt `lastAssistant` fallback and skipping aborted-run delivery is the narrow maintainable fix; the main safety requirement is focused regression coverage so valid fallback behavior remains intact.
