## Summary

Keep open. Current main still contains the reported stale replay path, and open PR #50147 remains the paired closing-syntax implementation candidate rather than a cleanup-close reason.

## What This Changes

The issue asks for `messages.queue.mode: "interrupt"` aborts to exit silently instead of delivering a previous assistant reply before the new response.

## Best Possible Solution

Land PR #50147 or an equivalent narrow patch that suppresses payload construction and delivery from aborted interrupt runs with no new assistant output, with focused payload-builder and reply-runner regression tests.

## Reproduction Assessment

Yes. The issue gives concrete interrupt-mode steps, a May 2 Telegram live repro confirms the symptom on v2026.4.29, and current-main static inspection confirms the stale `lastAssistant` fallback can still reach delivery.

## Solution Assessment

Yes, with regression coverage. Guarding the aborted empty-attempt `lastAssistant` fallback and skipping aborted-run delivery is the narrow maintainable fix, while the recent restart/tool-continuation variant should be checked as adjacent coverage.
