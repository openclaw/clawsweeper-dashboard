## What This Changes

The issue reports that `messages.queue.mode: "interrupt"` could abort an active run and then deliver the previous assistant reply before the replacement run answered.

## Best Possible Solution

Keep the shipped payload-boundary fix and treat the stale/conflicting fork PR as obsolete rather than reopening a second implementation path.

## Solution Assessment

Yes. The shipped `runAborted` payload-boundary fix is the safer solution because it suppresses stale aborted assistant output while preserving timeout/error payloads and reply-runner bookkeeping that a blanket aborted-run early return could skip.
