## Summary

Keep open. Current main still has the source path that can package a prior assistant message after an interrupt abort, and the paired fix PR at https://github.com/openclaw/openclaw/pull/50147 is still open and unmerged.

## What This Changes

The issue reports that interrupt queue mode can deliver a previous assistant reply from session history when an active run is aborted before producing new assistant text.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/50147 or an equivalent narrow patch that suppresses stale `lastAssistant` fallback for aborted empty interrupt attempts while preserving timeout, error, and compaction delivery behavior, with focused regression coverage.

## Reproduction Assessment

Yes. The issue gives concrete interrupt-mode steps, and current-main source inspection shows the abort path can retain an older `lastAssistant`, convert it into a payload, and deliver it without an aborted-run gate.

## Solution Assessment

Partly. Suppressing the stale `lastAssistant` fallback is the right fix direction, but a blanket early return for every `meta.aborted` result is too broad; the maintainable solution should target superseded empty-output interrupt runs.
