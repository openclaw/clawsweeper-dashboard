## Summary

Keep open. Current main still gates Anthropic adaptive-thinking payloads on Claude model-id substrings, and the public model compat config/schema cannot opt an opaque Anthropic-compatible proxy model into adaptive thinking.

## What This Changes

Request a config-backed way for custom Anthropic-compatible provider model entries to declare adaptive thinking support instead of relying on Claude model-id substrings.

## Best Possible Solution

Add a typed Anthropic adaptive-thinking compat signal and wire it through config schema, catalog normalization, thinking profile/session validation, Anthropic payload construction, docs, and focused regression tests while preserving existing Claude-id heuristics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a custom `anthropic-messages` provider with `reasoning: true` and an opaque Claude-backed id such as `zyb-high`; current main cannot expose or emit adaptive thinking for that id.

## Solution Assessment

Yes in direction, but the exact public API is not settled. A typed config-backed opt-in is more maintainable than hardcoding private proxy prefixes or enabling adaptive thinking for all opaque Anthropic-compatible models.
