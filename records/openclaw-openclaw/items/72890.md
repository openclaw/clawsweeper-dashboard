## Summary

Keep open. Current main still gates Anthropic adaptive thinking on built-in Claude model-id substrings, while configured custom model compat can expose OpenAI-style `xhigh` but cannot opt an opaque `anthropic-messages` proxy model into adaptive thinking.

## What This Changes

The issue requests config-backed Anthropic adaptive thinking support for custom provider model IDs instead of relying only on built-in model-id heuristics.

## Best Possible Solution

Add an explicit Anthropic adaptive-thinking model compat signal and wire it through config schema, catalog normalization, thinking profile/session validation, Anthropic transport payload construction, docs, and regression tests while preserving existing Claude-id heuristics and the non-opt-in fallback.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: configure a custom `models.providers.*` entry with `api: "anthropic-messages"`, `reasoning: true`, and an opaque Claude-backed id such as `zyb-high`; current main cannot expose or emit adaptive thinking for that id.

## Solution Assessment

Yes in direction, but the exact public API needs maintainer choice. A typed config-backed compat opt-in is narrower and more maintainable than hardcoding private corporate model prefixes or changing all opaque Anthropic-compatible proxies to adaptive by default.
