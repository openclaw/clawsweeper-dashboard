## Summary

Keep open. Current main and v2026.5.12 still gate Anthropic adaptive-thinking payloads on Claude model-id substrings, and configured custom provider models do not have a typed compat/config key to opt opaque Claude-backed aliases into adaptive thinking.

## What This Changes

Request a config-backed capability so custom Anthropic-compatible provider models with opaque IDs can opt into adaptive thinking.

## Best Possible Solution

Add a typed Anthropic adaptive-thinking capability and wire it through config schema, catalog normalization, thinking profile/session validation, Anthropic payload construction, docs, and focused regression tests while preserving existing Claude-id heuristics.

## Reproduction Assessment

Yes from source inspection. Configure a custom `anthropic-messages` provider with `reasoning: true` and an opaque Claude-backed id such as `zyb-high`; current main cannot expose or emit adaptive thinking for that id because the transport gate only checks model-id substrings.

## Solution Assessment

Yes in direction, but the exact public API is not settled. A typed explicit opt-in is more maintainable than hardcoding private proxy prefixes or enabling adaptive thinking for every opaque Anthropic-compatible model.
