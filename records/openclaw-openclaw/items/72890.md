## Summary

Keep open. Current main still gates Anthropic adaptive thinking on Claude Opus/Sonnet model-id substrings and only uses configured model compat to expose `xhigh`, so an opaque custom `anthropic-messages` proxy id cannot opt into `/think adaptive` or make the transport emit adaptive payloads.

## What This Changes

Allow custom configured `anthropic-messages` provider models to declare adaptive thinking support instead of relying only on Claude model-id heuristics.

## Best Possible Solution

Add a typed config-backed Anthropic adaptive-thinking opt-in under model compat, then have thinking-profile validation and Anthropic transport payload resolution honor it before falling back to model-id heuristics.

## Reproduction Assessment

Yes. Configure a custom `models.providers.my-proxy` entry with `api: "anthropic-messages"`, `reasoning: true`, and an opaque Claude-backed id such as `zyb-high`; static inspection shows current main neither validates `/think adaptive` from configured compat nor emits Anthropic adaptive payloads for that id.

## Solution Assessment

Yes. A config-backed compat opt-in is the narrow maintainable direction because it avoids hardcoding corporate prefixes and does not require a simple configured proxy to become a provider plugin.
