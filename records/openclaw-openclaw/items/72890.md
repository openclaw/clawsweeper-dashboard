## Summary

Keep open. Current main still makes Anthropic adaptive-thinking transport behavior depend on Claude model-id substrings, while custom configured `anthropic-messages` models can carry generic compat metadata but cannot declare adaptive-thinking support for opaque proxy model IDs.

## What This Changes

The issue requests a config-declared adaptive-thinking capability for custom Anthropic-compatible provider models with opaque model IDs.

## Best Possible Solution

Add a typed `compat`-backed adaptive-thinking capability for configured `anthropic-messages` models, feed it through thinking-profile validation and Anthropic transport payload resolution, keep the existing Claude model-id allowlist as fallback, and document a generic custom proxy example without customer-specific prefixes.

## Reproduction Assessment

Yes. A focused unit test can configure `models.providers.my-proxy` with `api: "anthropic-messages"`, an opaque model id, `reasoning: true`, and a non-off thinking level; current main will take the budget-token thinking path instead of sending `thinking.type: "adaptive"`.

## Solution Assessment

Yes. A typed config/compat capability is the narrowest maintainable solution because configured custom providers already carry model metadata, and hardcoding corporate prefixes or requiring a full provider plugin would not address the generic config surface reported here.
