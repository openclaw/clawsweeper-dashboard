## Summary

Keep open. Current main still has no configured-model metadata path for an opaque custom `anthropic-messages` Claude proxy to opt into Anthropic adaptive thinking; validation and transport are still driven by provider profiles or Claude model-id heuristics.

## What This Changes

The issue requests a config-declared adaptive-thinking capability for custom Anthropic-compatible provider models whose public model id does not match Claude naming heuristics.

## Best Possible Solution

Add a typed Anthropic adaptive-thinking opt-in under configured model `compat` metadata, then have both thinking-profile validation and Anthropic transport payload resolution honor it before falling back to model-id heuristics.

## Reproduction Assessment

Yes. Configure a custom `models.providers.my-proxy` entry with `api: "anthropic-messages"`, `reasoning: true`, and an opaque Claude-backed id such as `zyb-high`; current main has no config field that makes `/think adaptive` valid or makes the transport emit `thinking.type: "adaptive"` for that id.

## Solution Assessment

Yes. A config-backed compat flag is the narrowest maintainable fix because it avoids hardcoding corporate proxy prefixes and follows the existing configured-model metadata pattern used for reasoning efforts.
