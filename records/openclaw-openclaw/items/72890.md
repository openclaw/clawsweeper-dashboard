## Summary

Keep open. Current main still gates Anthropic adaptive thinking on hardcoded Claude model-id substrings, and configured custom `anthropic-messages` models can expose `xhigh` but not an adaptive-thinking capability.

## What This Changes

The issue requests a config-declared adaptive-thinking opt-in for custom Anthropic-compatible provider models.

## Best Possible Solution

Add a typed `compat` opt-in for Anthropic adaptive thinking on configured `anthropic-messages` models, then wire catalog normalization, thinking validation/exposure, transport payload/options, docs, tests, and changelog.

## Reproduction Assessment

Yes, from source inspection. Configure a custom `models.providers.*` model with `api: "anthropic-messages"`, `reasoning: true`, and an opaque Claude-backed id such as `zyb-high`; current main cannot expose adaptive support and the transport takes the non-adaptive budget-token branch.

## Solution Assessment

Yes. A typed config-backed compat opt-in is the narrow maintainable direction because it supports corporate proxies without hardcoding private prefixes or requiring a full provider plugin for a simple proxy.
