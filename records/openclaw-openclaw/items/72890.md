## Summary

Keep open. Current main still gates Anthropic adaptive thinking on Claude model-id substrings, while configured custom-model compat only exposes OpenAI-style `xhigh`, so opaque `anthropic-messages` proxy IDs still cannot opt into adaptive payloads.

## What This Changes

Request to let configured custom Anthropic-compatible providers declare adaptive-thinking support for opaque Claude-backed model IDs.

## Best Possible Solution

Add a documented per-model compat signal for Anthropic adaptive thinking, carry it through catalog normalization and thinking profiles, and have the Anthropic transport honor it while preserving existing Claude heuristics.

## Reproduction Assessment

Yes, from source inspection. A custom `models.providers.*` entry with `api: "anthropic-messages"` and an opaque id such as `zyb-high` routes through the Anthropic transport, but the adaptive gate returns false and emits the budget-token path.

## Solution Assessment

Yes. A typed config-backed compat opt-in is the narrow maintainable direction because it avoids private prefix allowlists while letting corporate Claude proxies declare the capability explicitly.
