## Summary

Keep open. Current main still hardcodes Anthropic adaptive-thinking support to recognized Claude model-id substrings, while custom configured model compat can only expose `xhigh`, so opaque `anthropic-messages` proxy ids still cannot opt into adaptive thinking through config.

## What This Changes

Request a config-backed model capability so custom Anthropic-compatible provider models with opaque ids can advertise adaptive thinking support.

## Best Possible Solution

Add an explicit model compat signal for Anthropic adaptive thinking and wire it through catalog normalization, thinking profile/session validation, Anthropic transport payload construction, docs, tests, and changelog while preserving existing Claude id heuristics.

## Reproduction Assessment

Yes, for the reported missing capability. Configure a custom `models.providers.*` entry with `api: "anthropic-messages"`, `reasoning: true`, and an opaque Claude-backed id such as `zyb-high`; current main reaches the Anthropic transport but cannot pass the adaptive gate, so it uses the budget-token thinking path.

## Solution Assessment

Yes, a typed config-backed compat opt-in is the narrow maintainable direction. It avoids hardcoding private proxy prefixes and keeps command menus, Gateway validation, transport payloads, docs, and generated schema aligned.
