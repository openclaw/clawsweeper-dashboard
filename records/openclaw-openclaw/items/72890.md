## Summary

Keep open. Current main still cannot let an opaque custom `anthropic-messages` proxy model advertise Anthropic adaptive thinking through configured model metadata; both validation and Anthropic payload shaping still fall back to hardcoded Claude model-id substrings.

## What This Changes

The issue requests a config-declared adaptive-thinking capability for custom Anthropic-compatible provider models so opaque proxy IDs can use `thinking.type: "adaptive"`.

## Best Possible Solution

Add a typed config-backed adaptive-thinking opt-in for configured `anthropic-messages` models, have thinking profile validation and Anthropic transport payload resolution honor it before the model-id fallback list, and document a generic proxy example with focused tests and a changelog entry.

## Reproduction Assessment

Yes. Configure `models.providers.my-proxy` with `api: "anthropic-messages"`, an opaque Claude-backed model id such as `zyb-high`, `reasoning: true`, and adaptive/non-off thinking; current main lacks a metadata path, so validation and transport resolve away from Anthropic adaptive payloads unless the id matches hardcoded Claude substrings.

## Solution Assessment

Yes. A typed model compat/profile opt-in is the narrowest maintainable fix because it supports opaque proxy IDs without hardcoding corporate prefixes or forcing a simple configured provider to become a full provider plugin.
