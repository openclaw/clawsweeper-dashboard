## Summary

Keep open. Current main still hardcodes Anthropic adaptive-thinking transport support to Claude model-id substrings, while custom model compat metadata only exposes OpenAI-style `xhigh`, so an opaque `anthropic-messages` proxy still cannot opt into adaptive thinking.

## What This Changes

Request to let custom Anthropic-compatible provider models declare adaptive-thinking support through model capability/config metadata instead of relying on hardcoded Claude model-id matching.

## Best Possible Solution

Add a typed, documented model compat opt-in for Anthropic adaptive thinking and wire it through catalog normalization, thinking policy/session validation, and Anthropic transport while preserving current model-id heuristics as fallback.

## Reproduction Assessment

Yes from source inspection. Configure a custom `models.providers.*` entry with `api: "anthropic-messages"`, `reasoning: true`, and an opaque Claude-backed id such as `zyb-high`; current main cannot expose `adaptive` from config and the transport takes the non-adaptive budget-token path.

## Solution Assessment

Yes. A typed config-backed compat opt-in is the narrow maintainable direction because it supports corporate proxies without hardcoding private model prefixes or moving provider-specific defaults into core.
