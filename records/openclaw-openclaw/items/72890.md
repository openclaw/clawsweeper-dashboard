## Summary

Keep open. Current main still cannot let a configured custom `anthropic-messages` proxy model with an opaque Claude-backed id opt into Anthropic adaptive thinking; adaptive support remains provider-profile or model-id driven, while configured compat metadata only extends `xhigh`.

## What This Changes

Allow custom Anthropic-compatible provider model definitions to declare adaptive thinking support instead of relying on hardcoded Claude model-id substrings.

## Best Possible Solution

Add a typed Anthropic adaptive-thinking opt-in under configured model compatibility metadata, then have thinking policy validation and Anthropic transport honor it before falling back to the hardcoded model-id heuristic.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure a custom `models.providers.*` entry with `api: "anthropic-messages"`, `reasoning: true`, and an opaque Claude-backed id such as `zyb-high`; current main has no configured metadata path that exposes `/think adaptive` or makes the transport send adaptive parameters for that id.

## Solution Assessment

Yes, with a schema-shape caveat. The narrowest maintainable fix is a typed config-backed compat opt-in, preferably on the existing `compat` metadata surface rather than hardcoding corporate prefixes or requiring a full provider plugin for a simple proxy.
