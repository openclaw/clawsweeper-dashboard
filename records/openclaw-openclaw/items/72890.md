## Summary

Keep open. Current main still cannot let a configured custom `anthropic-messages` proxy model with an opaque ID opt into Anthropic adaptive thinking; the runtime and validation paths remain tied to provider/plugin profiles or hardcoded Claude model-id substrings.

## What This Changes

Request adding a config-declared adaptive-thinking capability for custom Anthropic-compatible provider models so opaque proxy IDs can use provider-managed adaptive thinking.

## Best Possible Solution

Add a typed Anthropic adaptive-thinking opt-in under the existing model compatibility metadata, have both thinking-profile validation and Anthropic transport payload resolution honor it before the model-id fallback, then update docs, generated schema/help, tests, and the changelog.

## Reproduction Assessment

Yes. Configure a custom `models.providers.my-proxy` entry with `api: "anthropic-messages"`, `reasoning: true`, and an opaque Claude-backed model id such as `zyb-high`; current main has no configured metadata path that makes `/think adaptive` valid or makes the transport emit `thinking.type: "adaptive"` for that id.

## Solution Assessment

Yes. A typed config-backed compat opt-in is the narrowest maintainable repair because it supports opaque proxy IDs without hardcoding corporate prefixes or requiring a simple configured provider to become a full provider plugin.
