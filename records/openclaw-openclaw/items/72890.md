## Summary

Keep open. Current main still gates Anthropic adaptive payloads on Claude model-id substrings, while configured custom model compat only exposes `xhigh`, so an opaque `anthropic-messages` proxy id still cannot opt into `/think adaptive` or adaptive transport payloads.

## What This Changes

The issue requests a config-backed way for custom Anthropic-compatible provider models to declare adaptive-thinking support instead of relying on hardcoded Claude model-id matches.

## Best Possible Solution

Add a typed model-compat opt-in for Anthropic adaptive thinking and make both thinking-profile validation and Anthropic transport payload resolution honor it before falling back to model-id heuristics.

## Reproduction Assessment

Yes. Configure a custom `models.providers.my-proxy` entry with `api: "anthropic-messages"`, `reasoning: true`, and an opaque Claude-backed id such as `zyb-high`; static inspection shows `/think adaptive` is not exposed from compat and the transport would take the non-adaptive budget path.

## Solution Assessment

Yes. A config-backed compat opt-in is the narrow maintainable solution because it covers configured corporate proxies without hardcoding private prefixes or requiring a full provider plugin for a simple proxy.
