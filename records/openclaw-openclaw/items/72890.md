## Summary

Keep open. Current main still hardcodes Anthropic adaptive thinking support to Claude model-id substrings, while configured custom Anthropic Messages providers can only expose OpenAI-style `xhigh` through compat metadata and cannot opt opaque Claude-backed IDs into adaptive transport payloads.

## What This Changes

Allow custom Anthropic-compatible provider models to declare adaptive thinking support so opaque Claude-backed model IDs receive adaptive thinking payloads instead of budget-token thinking.

## Best Possible Solution

Add a typed model compat signal for Anthropic adaptive thinking and wire it through catalog normalization, thinking policy/session validation, Anthropic transport payload construction, docs, tests, and changelog while preserving existing Claude-id heuristics.

## Reproduction Assessment

Yes, from source inspection. A custom `models.providers.*` entry with `api: "anthropic-messages"`, `reasoning: true`, and an opaque Claude-backed id such as `zyb-high` reaches the Anthropic transport, but `supportsAdaptiveThinking()` returns false and the payload takes the non-adaptive budget-token path.

## Solution Assessment

Yes as a direction: a typed config-backed compat opt-in is narrower and more maintainable than hardcoding private prefixes or requiring every simple proxy to become a plugin. The exact public field name still needs maintainer decision.
