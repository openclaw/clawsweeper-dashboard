## Summary

Keep open. Current main still gates Anthropic adaptive thinking on Claude model-id substrings, while configured custom-model compat can expose OpenAI-style reasoning efforts but not Anthropic adaptive thinking; the reporter also reconfirmed the gap on v2026.5.2.

## What This Changes

Request that custom `anthropic-messages` provider model entries declare adaptive-thinking support so opaque Claude-backed IDs receive adaptive Anthropic payloads.

## Best Possible Solution

Add a typed model compat opt-in for Anthropic adaptive thinking and wire it through catalog normalization, thinking-profile validation, Anthropic transport payload construction, docs, tests, and changelog while preserving existing model-id heuristics.

## Reproduction Assessment

Yes, from source inspection. Configure a custom `models.providers.*` entry with `api: "anthropic-messages"`, `reasoning: true`, and an opaque Claude-backed id such as `zyb-high`; current main cannot expose `adaptive` from config and the transport will choose the non-adaptive budget-token path.

## Solution Assessment

Yes on direction, but the exact public config contract needs maintainer approval. A typed compat opt-in is narrower than hardcoding private prefixes and keeps opaque corporate proxies within the existing custom-provider model surface.
