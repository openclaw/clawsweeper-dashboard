## Summary

Keep open. Current main still gates Anthropic adaptive thinking on hardcoded Claude model-id substrings, so an opaque custom `anthropic-messages` proxy model cannot opt into adaptive payload shaping through configured model metadata.

## What This Changes

The issue requests a config-backed way for custom Anthropic-compatible provider models to declare adaptive thinking support for opaque proxy model IDs.

## Best Possible Solution

Add a typed Anthropic adaptive-thinking opt-in under the existing model compat metadata, make both thinking-profile validation and Anthropic transport payload resolution honor it before the model-id fallback, and document the custom proxy configuration with tests and a changelog entry.

## Reproduction Assessment

Yes. Configure a custom `models.providers.my-proxy` entry with `api: "anthropic-messages"`, `reasoning: true`, and an opaque Claude-backed model id such as `zyb-high`; current main has no metadata path that can make `/think adaptive` valid or make the transport emit `thinking.type: "adaptive"` for that id.

## Solution Assessment

Yes. A typed config-backed compat opt-in is the narrowest maintainable fix because it supports opaque proxy IDs without hardcoding corporate prefixes or forcing a simple configured provider to become a full provider plugin.
