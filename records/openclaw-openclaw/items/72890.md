## Summary

Keep open. Current main still hardcodes Anthropic adaptive transport support to Claude model-id substrings and only lets configured custom-model compat expose OpenAI-style `xhigh`, so the reported opaque `anthropic-messages` proxy cannot opt into adaptive thinking.

## What This Changes

The issue requests that configured custom Anthropic-compatible provider models be able to declare adaptive thinking support instead of depending on model-id heuristics.

## Best Possible Solution

Add an explicit model compat signal for Anthropic adaptive thinking and wire it through catalog normalization, thinking policy/session validation, Anthropic transport payload construction, docs, tests, and changelog while preserving existing heuristics.

## Reproduction Assessment

Yes from source inspection. Configure a custom `models.providers.*` entry with `api: "anthropic-messages"`, `reasoning: true`, and an opaque Claude-backed id such as `zyb-high`; current policy cannot expose `adaptive`, and the transport will take the non-adaptive budget-token path.

## Solution Assessment

Yes. A typed config-backed compat opt-in is the narrow maintainable solution because it supports configured corporate proxies without hardcoding private model prefixes or requiring a full provider plugin for a simple proxy.
