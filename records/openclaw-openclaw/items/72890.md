## Summary

Keep open. Current main still hard-codes Anthropic adaptive-thinking support to Claude model-id substrings and the custom provider model-compat schema has no adaptive-thinking capability flag, so the reported opaque `anthropic-messages` proxy id cannot opt into adaptive payloads through configuration.

## What This Changes

The issue requests a config-declared adaptive-thinking capability for custom Anthropic-compatible provider models before falling back to built-in model-id heuristics.

## Best Possible Solution

Add a typed model-compat opt-in for Anthropic adaptive thinking and make both thinking-profile selection and Anthropic transport payload construction honor it before falling back to model-id heuristics.

## Reproduction Assessment

Yes. Configure a custom `models.providers.*` entry with `api: "anthropic-messages"`, reasoning enabled, and an opaque Claude-backed id such as `zyb-high`; current static inspection shows `/think adaptive` is not exposed from config compat and the transport will not send adaptive payloads for that id.

## Solution Assessment

Yes. A config-backed compat opt-in is the narrow maintainable direction because it supports corporate proxies without hardcoding private model prefixes or forcing a full provider plugin for a simple Anthropic-compatible gateway.
