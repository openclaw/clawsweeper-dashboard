## Summary

Keep open. Current main still gates Anthropic adaptive-thinking payloads on Claude model-id substrings, and the public model config/compat path cannot opt an opaque Anthropic-compatible proxy model into adaptive thinking.

## What This Changes

Allow custom Anthropic-compatible provider models to declare adaptive thinking support via model/provider capabilities instead of relying only on hardcoded Claude model-id heuristics.

## Best Possible Solution

Add a typed Anthropic adaptive-thinking opt-in and wire it through config schema, catalog normalization, thinking profile/session validation, Anthropic payload construction, docs, and focused regression tests while preserving existing Claude-id heuristics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a custom anthropic-messages provider with reasoning enabled and an opaque Claude-backed id such as zyb-high; current main cannot expose or emit adaptive thinking for that id.

## Solution Assessment

Yes directionally, but the public API is not settled. A config-backed explicit opt-in is more maintainable than hardcoding private proxy prefixes or enabling adaptive thinking for every opaque Anthropic-compatible model.
