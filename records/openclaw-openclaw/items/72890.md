## Summary

Keep open. Current main still gates Anthropic adaptive thinking on hardcoded Claude model-id substrings, while configured model compat metadata only exposes xhigh to the thinking policy, so opaque custom Anthropic-compatible proxy IDs cannot opt into adaptive payloads.

## What This Changes

The issue requests a config-declared adaptive-thinking capability for custom Anthropic Messages provider models so opaque proxy IDs can receive Anthropic adaptive thinking parameters.

## Best Possible Solution

Add an additive, documented `models.providers.*.models[].compat` adaptive-thinking opt-in/opt-out for Anthropic Messages models, consume it consistently in thinking policy and Anthropic transport resolution, and keep the current Claude model-id detection only as the fallback.

## Reproduction Assessment

Yes. Configure a custom provider with `api: "anthropic-messages"`, an opaque model id such as `zyb-high`, `reasoning: true`, and an adaptive thinking request; current source routes that id through the non-adaptive budget-token branch because `supportsAdaptiveThinking("zyb-high")` is false and there is no config metadata path.

## Solution Assessment

Yes. A typed config-backed capability is the narrowest maintainable fix because it supports corporate or aggregator proxy IDs without hardcoding private prefixes or forcing a simple configured provider to become a full provider plugin.
