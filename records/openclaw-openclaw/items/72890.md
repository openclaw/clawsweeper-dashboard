## Summary

Keep open. Current main still blocks opaque Claude-backed custom `anthropic-messages` model IDs from `/think adaptive` and from adaptive Anthropic payloads because configured compat only affects `xhigh` while the transport adaptive gate is still model-id based.

## What This Changes

The request asks for configured custom Anthropic-compatible provider models to opt into adaptive thinking instead of relying only on Claude model-id substring checks.

## Best Possible Solution

Add a typed configured-model compat opt-in for Anthropic adaptive thinking, then make thinking-profile validation, gateway/session rows, agent command validation, and the Anthropic transport honor it before falling back to Claude model-id heuristics.

## Reproduction Assessment

Yes. Configure `models.providers.my-proxy` with `api: "anthropic-messages"`, `reasoning: true`, and an opaque Claude-backed model id such as `zyb-high`; current main has no config metadata path that makes `/think adaptive` valid or makes the transport emit adaptive thinking for that id.

## Solution Assessment

Yes. A config-backed compat opt-in is the narrowest maintainable fix because it avoids hardcoded corporate prefixes and does not require every simple Anthropic-compatible proxy to become a full provider plugin.
