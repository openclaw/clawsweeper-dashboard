## Summary

Keep open. Current main still has no config-backed way for an opaque custom `anthropic-messages` proxy model to opt into Anthropic adaptive thinking; the existing configured-model compat path only covers OpenAI-style reasoning efforts such as `xhigh`, while the Anthropic transport still gates adaptive payload shaping on hardcoded Claude model-id substrings.

## What This Changes

The issue requests that custom provider model metadata be able to declare Anthropic adaptive-thinking support instead of relying only on `supportsAdaptiveThinking()` hardcoded model-id matching.

## Best Possible Solution

Add a typed model compat opt-in for Anthropic adaptive thinking, have thinking-profile validation and Anthropic transport payload resolution honor it before model-id fallback, and document the custom proxy configuration with focused tests and a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure a custom `models.providers.my-proxy` entry with `api: "anthropic-messages"`, `reasoning: true`, and an opaque Claude-backed model id such as `zyb-high`; current main has no metadata path that can make `/think adaptive` valid or make the transport emit `thinking.type: "adaptive"` for that id.

## Solution Assessment

Yes. A typed config-backed compat opt-in is the narrowest maintainable fix because it supports opaque proxy IDs without hardcoding corporate prefixes or forcing a simple configured provider to become a full provider plugin.
