## Summary

Keep open. Current main still has no configured-model metadata path for an opaque `anthropic-messages` Claude proxy to opt into adaptive thinking; validation and the transport payload still rely on provider profiles or Claude model-id substrings.

## What This Changes

Request support for custom Anthropic-compatible provider models to declare adaptive-thinking capability instead of relying on hardcoded Claude model-id matching.

## Best Possible Solution

Add a typed adaptive-thinking opt-in under configured model `compat`, then have thinking-profile validation and Anthropic transport payload resolution honor it before falling back to Claude model-id heuristics.

## Reproduction Assessment

Yes. Configure a custom `models.providers.my-proxy` entry with `api: "anthropic-messages"`, `reasoning: true`, and an opaque Claude-backed model id such as `zyb-high`; current main has no metadata path that exposes `/think adaptive` or emits `thinking.type: "adaptive"` for that id.

## Solution Assessment

Yes. A config-backed `compat` opt-in is the narrowest maintainable solution because it avoids hardcoding corporate prefixes and matches the existing model metadata pattern used for configured `xhigh` reasoning.
