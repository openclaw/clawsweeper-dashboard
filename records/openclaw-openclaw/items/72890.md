## Summary

Keep open. Current main still hardcodes Anthropic adaptive transport support to Claude model-id substrings, and configured custom-model compat only exposes OpenAI-style reasoning effort metadata, so opaque `anthropic-messages` proxies cannot opt into adaptive thinking from config.

## What This Changes

The issue requests a config-declared way for custom Anthropic-compatible provider models with opaque IDs to opt into adaptive thinking support.

## Best Possible Solution

Add an explicit model compat signal for Anthropic adaptive thinking and wire it through catalog normalization, thinking policy/session validation, Anthropic transport payload construction, docs, tests, and changelog while preserving existing Claude id heuristics.

## Reproduction Assessment

Yes, from source inspection. A custom `models.providers.*` entry with `api: "anthropic-messages"`, `reasoning: true`, and an opaque Claude-backed id such as `zyb-high` reaches the Anthropic transport, but `supportsAdaptiveThinking(model.id)` returns false and the payload falls back to budget-token thinking.

## Solution Assessment

Yes. A typed config-backed compat opt-in is the narrow maintainable direction because it supports configured corporate proxies without hardcoding private model prefixes or requiring a provider plugin for a simple proxy.

## Repair Work Prompt

Implement config-backed adaptive thinking opt-in for custom `anthropic-messages` models. Add a typed model compat signal, likely under the existing `compat` metadata surface rather than the issue-body `capabilities` object, that lets configured model catalog entries advertise Anthropic adaptive thinking. Wire it through config types/schema, catalog normalization, thinking profile/session validation, and `src/agents/anthropic-transport-stream.ts` so an opaque reasoning-capable custom model can emit `thinking: { type: "adaptive" }` plus the appropriate `output_config.effort` when reasoning is enabled. Preserve existing Claude model-id heuristics as fallback, do not hardcode private prefixes such as `zyb-`, do not change OpenAI reasoning effort semantics, and update docs plus `CHANGELOG.md` for the new public config surface.
