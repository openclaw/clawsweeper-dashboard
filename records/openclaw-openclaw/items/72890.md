## Summary

Keep open. Current main still gates Anthropic adaptive thinking on Claude model-id substrings, and solving the opaque custom-provider case requires a public config/runtime capability decision rather than stale-item cleanup.

## What This Changes

The issue requests a config-backed way for custom Anthropic-compatible provider models to declare adaptive thinking support instead of relying only on built-in Claude model-id heuristics.

## Best Possible Solution

Add an explicit Anthropic adaptive-thinking compat/profile signal and wire it through config schema, catalog normalization, session thinking validation, Anthropic payload construction, docs, and regression tests while preserving existing Claude-id heuristics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a custom models.providers entry with api "anthropic-messages", reasoning enabled, and an opaque Claude-backed id such as zyb-high; current main routes it to the Anthropic transport but the adaptive gate returns false.

## Solution Assessment

Yes in direction, but the exact public API is not settled. A typed config-backed opt-in is narrower and more maintainable than hardcoding private prefixes or enabling adaptive thinking for all opaque Anthropic-compatible proxies.
