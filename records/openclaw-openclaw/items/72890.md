## Summary

Keep open. Current main still cannot let a configured custom `anthropic-messages` proxy model with an opaque Claude-backed id opt into Anthropic adaptive thinking; both `/think` validation and the Anthropic transport are still tied to provider profiles or hardcoded Claude model-id substrings.

## What This Changes

The issue requests a config-declared adaptive-thinking capability for custom Anthropic-compatible provider models whose IDs do not match OpenClaw's hardcoded Claude model substrings.

## Best Possible Solution

Add a typed Anthropic adaptive-thinking opt-in under configured model compatibility metadata, then have both thinking-profile validation and Anthropic transport payload resolution honor it before falling back to model-id heuristics. Keep provider plugins free to expose richer profiles, but do not require a simple configured corporate proxy to become a full plugin.

## Reproduction Assessment

Yes. Configure a custom `models.providers.my-proxy` entry with `api: "anthropic-messages"`, `reasoning: true`, and an opaque Claude-backed model id such as `zyb-high`; current main has no configured metadata path that makes `/think adaptive` valid or makes the transport emit `thinking.type: "adaptive"` for that id.

## Solution Assessment

Yes. A typed config-backed compat opt-in is the narrowest maintainable direction because it avoids hardcoding corporate prefixes and fits the existing model metadata pattern already used for configured `xhigh` reasoning.
