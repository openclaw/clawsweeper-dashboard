## Summary

Keep open. Current main still lacks a config-backed way for an opaque custom `anthropic-messages` model id to opt into Anthropic adaptive thinking, so the reported proxy setup still falls through the hardcoded model-id gate and uses budget-token extended thinking instead of `thinking: { type: "adaptive" }` plus `output_config.effort`.

## What This Changes

The issue requests a per-model custom-provider capability override so Anthropic-compatible proxy model ids can advertise adaptive thinking support without patching OpenClaw internals.

## Best Possible Solution

Add a typed per-model compat/profile opt-in for Anthropic adaptive thinking, honor it before model-id fallback in validation and Anthropic transport payload resolution, and document the custom proxy configuration with focused regression tests.

## Reproduction Assessment

Yes. Configure a custom `models.providers.my-proxy` entry with `api: "anthropic-messages"`, `reasoning: true`, an opaque Claude-backed model id such as `zyb-high`, and a non-off thinking level; current main routes that id through `supportsAdaptiveThinking(model.id) === false` and therefore avoids the adaptive payload path.

## Solution Assessment

Yes. A typed config-backed opt-in is the narrowest maintainable fix because it supports corporate proxy ids without hardcoding private prefixes or requiring a simple configured provider to become a full provider plugin.
