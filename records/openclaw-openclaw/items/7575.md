## Summary

Keep open. Current main still lacks Sysbox/sysbox-runc support and a Docker runtime selector, while the discussion explicitly keeps this as the canonical Sysbox/runtime-architecture tracker rather than completed by Docker-socket sandboxing or sandbox-common work.

## What This Changes

The issue requests Sysbox/sysbox-runc based Docker sandbox isolation, optionally including runtime selection, so OpenClaw can avoid Docker socket sharing for sandbox containers.

## Best Possible Solution

Keep this open until maintainers choose a concrete design: either add a conservative agents.defaults.sandbox.docker.runtime selector with schema/types, config hash behavior, Docker create --runtime wiring, docs, and tests; or implement Sysbox as a sandbox backend plugin using registerSandboxBackend, with host prerequisites and security tradeoffs documented.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence current-main verification path: source inspection shows docker.runtime is absent from the type/schema and Docker create args never emit --runtime.

## Solution Assessment

Unclear. The issue's direct host-level Sysbox plan should not be applied blindly; the maintainable path is a maintainer-approved design that uses either a narrow Docker runtime selector or the existing sandbox backend plugin seam.
