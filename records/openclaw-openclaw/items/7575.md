## Summary

Keep open. Current main still supports Docker-socket sandboxing and pluggable sandbox backends, but it does not implement Sysbox/sysbox-runc host runtime integration or a Docker `--runtime` selector for sandbox containers.

## What This Changes

Request to move sandbox isolation toward Sysbox-based container runtime isolation and optional runtime selection instead of relying only on Docker socket sharing.

## Best Possible Solution

Keep this as the canonical design tracker until maintainers choose either a conservative core `agents.defaults.sandbox.docker.runtime` selector with schema/create-args/docs/tests, or a Sysbox sandbox backend plugin with host prerequisites and security tradeoffs documented.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request; source and docs inspection on current main gives high-confidence proof that Sysbox/sysbox-runc support and `sandbox.docker.runtime` wiring are absent.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The direct host-level Sysbox plan should not be applied as routine cleanup; maintainers need to choose the core selector or plugin/backend path after security review.
