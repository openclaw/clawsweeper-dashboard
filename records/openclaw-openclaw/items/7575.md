## Summary

Keep open. Current main has Docker-socket sandboxing and a generic sandbox backend seam, but it still lacks Sysbox/sysbox-runc support or a Docker runtime selector, and the issue remains a security-sensitive architecture decision rather than a cleanup close.

## What This Changes

The issue requests Sysbox/sysbox-runc based sandbox isolation for Docker deployments, replacing the host Docker socket path with host-level runtime setup and matching OpenClaw config/docs support.

## Best Possible Solution

Keep this as the canonical sandbox runtime design tracker until maintainers approve either a strict core Docker runtime selector or a Sysbox sandbox backend plugin with host prerequisite docs, security guidance, and focused tests.

## Reproduction Assessment

Not applicable: this is a feature/security architecture request, not a bug with a failing reproduction path. Source inspection gives high-confidence proof that Sysbox/sysbox-runc and sandbox.docker.runtime remain absent from current main.

## Solution Assessment

Unclear: a core agents.defaults.sandbox.docker.runtime selector would solve part of the request, but the existing sandbox backend/plugin seam may be the more maintainable route. Maintainers should choose the design and security model before implementation.
