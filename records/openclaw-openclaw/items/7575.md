## Summary

Keep open. Current main still has Docker-socket sandboxing and a sandbox backend plugin seam, but it does not implement Sysbox/sysbox-runc, does not accept `agents.defaults.sandbox.docker.runtime`, and does not pass Docker `--runtime` when creating sandbox containers; the live discussion also keeps this scope separate from completed sandbox-common and Docker-socket work.

## What This Changes

Add Sysbox/sysbox-runc or Docker runtime selection support for stronger sandbox container isolation.

## Best Possible Solution

Keep this as the canonical Sysbox sandbox runtime design tracker until owners approve either a strict core Docker runtime selector or a plugin-backed Sysbox backend with host prerequisites, threat model, docs, and tests.

## Reproduction Assessment

Not applicable: this is a feature/security architecture request, not a broken existing behavior report. Source inspection gives high-confidence proof that the requested Sysbox/runtime selector path is absent on current main.

## Solution Assessment

Unclear: direct Docker `--runtime` selection is a narrow implementation, while a plugin-backed Sysbox backend may better contain host prerequisites and support boundaries. Maintainers should choose the API and threat-model path before implementation.
