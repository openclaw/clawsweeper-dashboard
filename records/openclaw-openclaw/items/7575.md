## Summary

Keep open. Current main still lacks Sysbox/sysbox-runc integration, a Docker sandbox runtime selector, and Docker create `--runtime` wiring; the live discussion also treats Docker-socket sandboxing and sandbox-common work as partial context rather than completion of this runtime-architecture request.

## What This Changes

Request host-level Sysbox runtime integration and optional Docker sandbox runtime selection so OpenClaw can run sandbox containers without mounting the host Docker socket.

## Best Possible Solution

Keep this as the canonical Sysbox sandbox runtime design tracker until maintainers approve either a strict core Docker runtime selector or a plugin-backed Sysbox backend with host prerequisites, rollback guidance, docs, and tests.

## Reproduction Assessment

Not applicable: this is a feature/security architecture request, not a broken existing behavior report. Current source and docs give a high-confidence source check that the requested Sysbox/runtime selector path is absent.

## Solution Assessment

Unclear: the requested direction is plausible, but adding host-level Sysbox directly to Docker deployment changes the trust boundary. Maintainers should choose between a narrow core Docker runtime selector and a Sysbox backend/plugin path before implementation.
