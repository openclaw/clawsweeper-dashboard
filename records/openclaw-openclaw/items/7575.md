## Summary

Keep open. Current main still lacks Sysbox/sysbox-runc support and a Docker `--runtime` selector; the newer sandbox backend seam may be the right implementation route, but this remains a security-sensitive runtime architecture decision for maintainers.

## What This Changes

Request to move Docker sandboxing toward Sysbox/sysbox-runc or runtime-selectable container isolation instead of Docker-socket-only sandbox orchestration.

## Best Possible Solution

Keep this as the canonical design tracker until maintainers choose either a narrow core Docker runtime selector with schema/create-args/docs/tests, or a Sysbox sandbox backend plugin with host prerequisites and security tradeoffs documented.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request; current-main verification is high-confidence because source and docs inspection shows no Sysbox implementation, no `sandbox.docker.runtime` key, and no Docker `--runtime` create wiring.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The direct host-level Sysbox plan should not be applied as a routine fix; maintainers need to choose between a core selector and a reviewed sandbox backend plugin.
