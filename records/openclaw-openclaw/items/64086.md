## Summary

Keep this issue open. Current main still lacks the requested plugin footer status contribution API and TUI consumption path, and open PR #72557 explicitly resolves this issue, so the maintainer path is to review, land, revise, or close that PR rather than close the issue early.

## What This Changes

Add a plugin API such as `registerStatusProvider` so plugins, especially context-engine plugins like ContextClaw, can contribute small status indicators to the TUI footer.

## Best Possible Solution

Keep this issue open until a reviewed implementation lands that exposes a stable plugin status-provider seam and either wires it into the TUI footer or clearly tracks the rendering follow-up with #53774.

## Reproduction Assessment

Yes. A read-only current-main check reproduces the gap: there are no status-provider symbols, the plugin API/builder/registry have no registration seam, and the TUI footer renders only fixed core fields.

## Solution Assessment

Unclear. A narrow `PluginStatusProvider` registration API matches existing plugin registration patterns, but the best final shape depends on maintainer decisions about return structure, footer UX, and whether storage and TUI consumption should ship together.
