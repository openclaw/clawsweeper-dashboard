## Summary

Keep this issue open. Current main still lacks the requested plugin footer status-provider API, registry storage, SDK export, and TUI footer read path, and open PR #72557 explicitly resolves this issue but is not merged.

## What This Changes

The issue requests an additive `registerStatusProvider` plugin API so context-engine and other plugins can contribute compact status indicators to the TUI footer.

## Best Possible Solution

Keep the issue open until a focused implementation lands that exposes a stable provider registration/storage contract and either wires it into the TUI footer or explicitly tracks the rendering step with #53774.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction of the gap: the plugin API, API builder, registry, SDK export surface, and TUI footer do not contain a status-provider seam.

## Solution Assessment

Unclear. A narrow additive plugin registration API matches existing plugin patterns, but the return shape and footer rendering contract should be settled through maintainer review of PR #72557 and coordination with #53774.
