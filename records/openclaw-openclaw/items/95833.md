## Summary

Keep open: current main and the latest release still have the retained session-lock cleanup path that can return or wait before releasing the physical `.jsonl.lock`, and the direct closing repair is still open rather than merged.

## What This Changes

The issue reports that subagent abort-settle/retained session-lock cleanup can leave a `.jsonl.lock` behind, causing later messages to the same session to fail until manual cleanup.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/96100 or an equivalent narrow active-scope lock-release fix, decide whether https://github.com/openclaw/openclaw/pull/95854 is unsafe or superseded, and handle abort-threshold default tuning separately.

## Reproduction Assessment

Yes, at source level: an active retained `withSessionWriteLock()` scope makes retained-lock cleanup return or wait before releasing the held physical lock. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Yes for keeping the issue open while the linked fix is unmerged; the narrowest maintainable repair appears to be the deferred active-scope release path in https://github.com/openclaw/openclaw/pull/96100. The default abort threshold is a separate maintainer product decision, not part of the core lock leak repair.
