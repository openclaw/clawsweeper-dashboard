## Label Justifications

- `P1`: The issue describes real sessions becoming unusable after embedded subagent abort cleanup, with subsequent messages failing until manual or watchdog lock recovery.
- `impact:session-state`: The failure is rooted in retained session transcript lock state and leaves `.jsonl.lock` ownership stale.
- `impact:message-loss`: Subsequent user messages to the locked session fail with a generic error instead of producing normal replies.

## Summary

Keep open: current main and v2026.6.10 still have retained session-lock abort/dispose paths that can return or wait before releasing the physical `.jsonl.lock`, and the active fixes are still open PRs rather than merged implementation.

## What This Changes

The issue reports that stuck embedded subagent abort cleanup can leave a session `.jsonl.lock` behind, causing later messages to fail until manual lock removal.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/96100 or an equivalent owner-approved retained-lock abort cleanup fix, then decide whether https://github.com/openclaw/openclaw/pull/95854 is still needed and handle abort-threshold tuning separately.

## Reproduction Assessment

Yes, at source level: current main can return or wait before releasing a retained held lock when cleanup runs during an active or never-settling retained write. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No final solution has landed yet. The narrowest maintainable path appears to be the deferred retained-lock abort cleanup in https://github.com/openclaw/openclaw/pull/96100, with the default timeout reduction treated as a separate maintainer decision.
