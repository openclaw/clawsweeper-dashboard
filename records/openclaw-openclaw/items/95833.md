## Label Justifications

- `P1`: The issue describes real user sessions becoming unusable after subagent abort cleanup, breaking subsequent agent/channel messages until manual intervention.
- `impact:session-state`: The failure is rooted in retained session transcript lock state and leaves the session’s `.jsonl.lock` ownership stale.
- `impact:message-loss`: Subsequent user messages to the locked session fail with a generic error and do not produce normal replies.

## Summary

Keep open: current main and the latest shipped release still have the retained-lock dispose path that can wait or return before releasing the physical `.jsonl.lock`, and the open linked repair has closing syntax for this issue but is not merged.

## What This Changes

The issue reports that embedded subagent abort/settle cleanup can leave a retained session `.jsonl.lock` behind, making later messages to that session fail until manual lock removal.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/95854 or an equivalent bounded dispose cleanup fix, then close this issue after merge; handle abort-threshold default tuning separately.

## Reproduction Assessment

Yes, at source level: start a retained `withSessionWriteLock()` operation that never resolves, then call `dispose()`; current main can wait or return before releasing the held physical lock. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Yes for the lock leak: a bounded retained-idle wait with force-release in final dispose is the narrow owner-boundary fix. Lowering `stuckSessionAbortMs` defaults is a separate product/default decision and should not be folded into the lock cleanup repair unless maintainers explicitly choose it.
