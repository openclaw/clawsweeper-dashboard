## Label Justifications

## Summary

Current main contains the merged deferred retained-lock release fix from https://github.com/openclaw/openclaw/pull/96100: abort release now marks the held lock for retry when the active write scope is still live, and the physical write-lock scope retries release after retained use unwinds. The fix is not in v2026.6.10, so this is a current-main-only close unless a later release ships it.

## What This Changes

The issue reports that embedded subagent abort cleanup can leak a session `.jsonl.lock`, causing later messages to fail until manual lock removal.

## Best Possible Solution

Keep the merged deferred-release implementation on main; route any remaining heap-growth, default-threshold tuning, or broader subagent delivery-lock behavior to narrower follow-up issues such as https://github.com/openclaw/openclaw/issues/86538 or https://github.com/openclaw/openclaw/issues/93383 when the symptoms differ.

## Reproduction Assessment

Yes. At source level, the pre-fix branch returned while a retained write scope was active; current main now has targeted tests and code for deferred release after the scope unwinds.

## Solution Assessment

Yes for the reported lock leak. Deferring release until after scope deactivation is the narrow owner-boundary fix and avoids force-releasing while the writer still owns the scope; abort-threshold default tuning is separate product policy.
