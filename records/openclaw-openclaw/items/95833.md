## Summary

Keep open: current main and latest release still have the retained session-lock cleanup shape that can return or wait before releasing the physical `.jsonl.lock`, and the active repair PRs are still open rather than merged.

## What This Changes

The issue reports that subagent abort cleanup can leave a session `.jsonl.lock` behind, causing later messages in that session to fail until manual lock removal.

## Best Possible Solution

Land a narrow maintainer-approved retained-lock cleanup fix, likely the fence-preserving path in https://github.com/openclaw/openclaw/pull/96100 plus any separately validated dispose gap, then handle abort-threshold default tuning as a separate product decision.

## Reproduction Assessment

Yes, at source level: a retained `withSessionWriteLock()` operation can keep `retainedLockUseCount` nonzero while abort or dispose cleanup returns before releasing the physical lock. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Unclear for the exact landing path: the narrow deferred-release PR appears safer for active retained writers, while the issue-closing bounded-dispose PR still has an unresolved force-release safety finding. The best fix should preserve transcript ownership and close the stale-lock leak without adding a late-writer race.
