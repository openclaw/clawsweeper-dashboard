## Label Justifications

- `P1`: A retained `.jsonl.lock` can make subsequent user messages to the affected session fail until manual lock removal.
- `impact:message-loss`: The reported symptom is that later messages return a generic failure instead of producing the expected agent reply.

## Summary

Keep open: current main and the latest release still have the retained-lock cleanup path that can wait indefinitely or return without releasing the physical session lock, and the linked fix PR is open rather than merged.

## What This Changes

The issue reports `.jsonl.lock` leaks after embedded subagent abort-settle timeouts, with a secondary request to reconsider the default `stuckSessionAbortMs` threshold.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/95854 for the lock leak, and split any default-threshold change into a separate maintainer-approved config/default decision.

## Reproduction Assessment

No live current-main reproduction was run. Source inspection gives a high-confidence path: a retained lock user can keep `waitForRetainedLockIdle()` pending, and dispose can return before `lock.release()` when the retained wait reports an active write scope.

## Solution Assessment

Yes for the central bug: the fix belongs in the embedded-attempt session-lock controller so final dispose/abort cleanup releases the physical lock. The default abort threshold is a separate product/config default choice, not the same repair.
