## Label Justifications

- `P1`: A shipped update path can silently drop managed gateway supervision, leaving Linux/WSL2 installs unavailable after reboot or process exit.
- `impact:crash-loop`: The issue is about process-level availability and loss of systemd lifecycle management after update.

## Summary

Keep open: this is a high-signal shipped update lifecycle bug, and current main still has a post-refresh health shortcut that can treat a surviving updated gateway process as sufficient without separately proving systemd supervision is still installed and active.

## What This Changes

The issue asks for `openclaw update` to preserve or recreate the managed systemd gateway unit instead of skipping reinstall/restart when a surviving gateway process already reports the updated version.

## Best Possible Solution

Make the package-update post-refresh shortcut prove the intended systemd unit is installed and supervisor-owned before declaring success; if the unit is missing, reinstall or restart under the chosen scope and add regression coverage for the surviving-process plus missing-unit case.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. The reporter provided concrete logs, and current source still contains the exact expected-version shortcut with no regression coverage for a missing unit plus surviving process.

## Solution Assessment

No, the current health-only shortcut is not the best final behavior. The maintainable fix should verify supervisor ownership and unit presence before success, coordinated with the existing systemd scope/provenance work.
