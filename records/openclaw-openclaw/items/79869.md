## Summary

Keep open. The narrowed current-main bug is still source-reproducible, but this PR still has a blocking deferred-wake correctness gap and lacks required after-fix real behavior proof.

## What This Changes

The branch lets targeted exec-event heartbeat wakes and focused regression tests run when `heartbeat.every` is `0m`.

## Best Possible Solution

Make disabled-periodic-heartbeat exec completion delivery reliable without losing deferred wake delivery, rebase over current heartbeat runner changes, and add real setup proof before merge.

## Reproduction Assessment

Yes for the narrowed bug by source inspection: configure `agents.defaults.heartbeat.every` to `0m`, let a background exec finish, and current main enqueues an `exec-event` wake that direct and scheduler paths treat as disabled. I did not run a live gateway repro in this read-only review.

## Solution Assessment

No. The narrow direction is plausible, but deferred fallback exec-event wakes need a retry or scheduling path; if heartbeat exceptions keep expanding, a dedicated system-event delivery path may be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Requeue deferred zero-interval exec events:** `src/infra/heartbeat-runner.ts:2245`
  - body: When `heartbeat.every` is `0m`, this fallback state lives outside `state.agents`. A second targeted `exec-event` within cooldown returns `skipped` with `not-due` or `min-spacing`, but the wake layer only requeues retryable busy skips and `scheduleNext()` ignores `state.execEventAgents`, so the queued completion can sit indefinitely unless another wake arrives. Requeue the wake for its due time or include the fallback state in scheduler re-arming.
  - confidence: 0.9
