## Summary

Keep open: this PR targets a real, source-reproducible disabled-heartbeat exec completion bug, but it still lacks after-fix real behavior proof and has a blocking deferred-wake correctness issue.

## What This Changes

The PR changes `src/infra/heartbeat-runner.ts` and focused heartbeat tests so targeted `exec-event` wakes can run when periodic heartbeat cadence is disabled with `every: "0m"`.

## Best Possible Solution

Deliver targeted exec completion events when periodic heartbeat cadence is disabled, preserve cooldown safety by rescheduling deferred fallback wakes, and merge only after focused tests plus real behavior proof show the completion reaches a real channel/session.

## Reproduction Assessment

Yes. By source inspection, configure `agents.defaults.heartbeat.every` to `0m`, let a background exec finish with notify-on-exit output, and current main requests an `exec-event` wake that both direct and scheduler paths treat as disabled; I did not run a live gateway repro in this read-only review.

## Solution Assessment

No. The PR direction is narrow and appropriate, but the current fallback returns cooldown-deferred zero-interval exec-event wakes without a retry or due-time schedule, so close-together completions can still remain queued until an unrelated wake arrives.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Requeue deferred zero-interval exec events:** `src/infra/heartbeat-runner.ts:2245`
  - body: When `heartbeat.every` is `0m`, this fallback state lives outside `state.agents`. A second targeted `exec-event` inside the cooldown returns `skipped` with `not-due` or `min-spacing` here, but the wake layer only requeues retryable busy skips and `scheduleNext()` only arms from `state.agents`, so the later queued completion can sit until some unrelated wake happens.
  - confidence: 0.9
