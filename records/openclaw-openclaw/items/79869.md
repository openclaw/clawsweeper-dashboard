## Summary

Keep open. The narrowed bug is valid and this PR targets the right subsystem, but the latest head still drops deferred zero-interval exec-event wakes and has no after-fix real behavior proof.

## What This Changes

The PR changes heartbeat runner dispatch and focused tests so targeted exec-event wakes can run when periodic heartbeats are disabled with `every: "0m"`.

## Best Possible Solution

Keep the narrow exec-event disabled-interval bypass, but make deferred fallback wakes requeue or schedule for their due time, cover close-together completions, and prove a real completion delivery with `every: "0m"`.

## Reproduction Assessment

Yes, by source inspection: configure `agents.defaults.heartbeat.every` to `0m`, let a background exec finish, and current main requests an `exec-event` wake that both direct and scheduler paths treat as disabled. I did not run a live gateway repro in this read-only review.

## Solution Assessment

No. The PR direction is narrow and appropriate, but the current fallback returns deferred zero-interval exec-event wakes without any retry or timer path, so two close completions can still leave a later completion queued indefinitely.
