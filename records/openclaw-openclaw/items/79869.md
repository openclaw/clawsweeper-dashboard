## Summary

Keep open. The PR targets a real source-reproducible gateway bug, but it still has a blocking scheduler/cooldown defect and lacks required after-fix real behavior proof from a real setup.

## What This Changes

The PR changes heartbeat runner dispatch and focused tests so targeted `exec-event` heartbeat wakes can run when `heartbeat.every` is `0m`.

## Best Possible Solution

Keep the narrow disabled-interval exec-event bypass, but make deferred fallback wakes requeue or schedule for their due time, update coverage for eventual delivery, and prove real background exec completion delivery.

## Reproduction Assessment

Yes, by source inspection: configure `agents.defaults.heartbeat.every` to `0m`, let a background exec finish with notify-on-exit output, and current main requests an `exec-event` wake that both direct and scheduler paths treat as disabled. I did not run a live gateway repro in this read-only review.

## Solution Assessment

No. The PR direction is narrow and appropriate, but the current fallback can drop deferred zero-interval exec-event wakes and the PR still lacks real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Requeue deferred zero-interval exec events:** `src/infra/heartbeat-runner.ts:2245`
  - body: When `heartbeat.every` is `0m`, this fallback state is outside `state.agents`; a deferred `exec-event` returns `skipped` with `not-due` or `min-spacing`, but `heartbeat-wake.ts` only requeues retryable busy skips and `scheduleNext()` only looks at `state.agents`. A second completion inside the cooldown can therefore stay queued indefinitely unless another wake arrives; the added scheduler test also fails in CI instead of proving eventual delivery.
  - confidence: 0.92
