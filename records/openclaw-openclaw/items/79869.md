## Summary

Keep open: the PR targets a valid source-reproducible bug, but the latest patch still drops deferred zero-interval exec-event wakes and the external PR lacks required after-fix real behavior proof.

## What This Changes

The PR updates heartbeat runner dispatch so targeted exec-event wakes can run with `heartbeat.every: "0m"` and adds focused scheduler/runHeartbeatOnce regression tests.

## Best Possible Solution

Keep the narrow exec-event bypass, but requeue or schedule deferred zero-interval exec-event wakes, cover two close-together completions, and add real behavior proof from a live setup.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: background exec completion enqueues an `exec-event` wake, while current main skips both direct and scheduler paths when `heartbeat.every` resolves to `0m`.

## Solution Assessment

No. The current PR handles first-wake delivery better than the initial version, but deferred fallback wakes are returned as ordinary `not-due`/`min-spacing` skips and are not retried or scheduled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Requeue deferred zero-interval exec events:** `src/infra/heartbeat-runner.ts:2243-2245`
  - body: When `heartbeat.every` is `0m`, the fallback agent lives only in `state.execEventAgents`. If a second targeted `exec-event` arrives inside the cooldown, this branch returns `skipped/not-due`, but `heartbeat-wake.ts` only retries busy skips and `scheduleNext()` exits because `state.agents` is empty. That leaves the queued completion stuck until an unrelated wake happens, so requeue or schedule the fallback wake for its next due time instead of dropping it.
  - confidence: 0.9
