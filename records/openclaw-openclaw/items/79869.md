## Summary

Keep open: current main still skips disabled-interval targeted `exec-event` wakes, but this PR is not merge-ready because it overrides explicit `target: "none"`, can drop deferred fallback wakes, and lacks real behavior proof.

## What This Changes

The PR changes the heartbeat runner and regression tests so targeted session-scoped `exec-event` wakes can run when `heartbeat.every` is `0m`.

## Best Possible Solution

A mergeable fix should let targeted exec-event wakes run with `heartbeat.every=0m` while preserving `target: "none"`, requeueing deferred wakes after cooldown, and adding redacted real gateway/agent proof.

## Reproduction Assessment

Yes, at source level: current main skips both `runHeartbeatOnce` and the scheduler path when `heartbeat.every=0m`, while background exec completion enqueues a session-scoped `exec-event` wake. I did not run a live gateway scenario in this read-only review.

## Solution Assessment

No. The PR is in a plausible owner boundary, but the current patch bypasses `target: "none"` and leaves deferred fallback wakes without retry ownership; the safer fix preserves delivery opt-out and owns delayed retry.
