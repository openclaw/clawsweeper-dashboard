## Summary

Keep open. The PR targets a real disabled-heartbeat exec-completion bug and moves in the right direction, but the latest patch still drops deferred zero-interval exec-event wakes, has a failing scheduler test, and lacks required after-fix real behavior proof.

## What This Changes

The PR changes heartbeat runner dispatch so targeted `exec-event` wakes can run when `heartbeat.every` is `0m`, and adds scheduler plus `runHeartbeatOnce` regression coverage.

## Best Possible Solution

Keep the narrow exec-event bypass, but requeue or schedule deferred zero-interval exec-event wakes so queued completions run after cooldown, update the regression test to cover that behavior, and add live proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: with `heartbeat.every: "0m"`, background exec completion requests an `exec-event` wake that both direct and scheduler heartbeat paths treat as disabled.

## Solution Assessment

No. The PR’s direction is narrow and appropriate, but the current patch returns deferred fallback wakes as ordinary skipped cooldown results that the wake layer does not retry, and the new scheduler test is failing in CI.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Requeue deferred zero-interval exec events:** `src/infra/heartbeat-runner.ts:2244-2245`
  - body: When a zero-interval targeted `exec-event` hits the fallback cooldown, this branch returns `skipped/not-due` or `skipped/min-spacing`. The wake layer only requeues retryable busy skips, and `scheduleNext()` only considers `state.agents`, so the completion can remain queued until an unrelated wake arrives. Requeue or schedule this fallback wake for its next due time instead of dropping it.
  - confidence: 0.92
- **[P2] Fix the zero-interval cooldown test expectation:** `src/infra/heartbeat-runner.scheduler.test.ts:632`
  - body: This new test advances fake time three times by 10s after the first run, which reaches the 30s fallback cooldown, so CI currently observes a second `runOnce` call. The test should stop before the cooldown if it only covers suppression, or preferably assert that a deferred wake stays pending and runs after the cooldown.
  - confidence: 0.88
