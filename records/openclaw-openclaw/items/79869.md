## Summary

Keep open. The PR fixes the scheduler/run gate for disabled heartbeat intervals, but it still lacks real behavior proof and appears to leave the default target-none delivery path consuming exec completion events without sending them to the user.

## What This Changes

The PR changes the heartbeat runner to let targeted `exec-event` wakes bypass disabled periodic intervals and adds scheduler plus `runHeartbeatOnce` regression coverage.

## Best Possible Solution

Keep the narrow exec-event bypass, but resolve exec completion delivery from the captured event/session route and add a regression that asserts the channel send when `heartbeat.every` is `0m` and heartbeat target is unset or `none`.

## Reproduction Assessment

Yes for source reproduction: current main drops the narrowed exec-event wake when the interval is disabled, and the PR branch still source-reproduces a target-none delivery gap. I did not run a live gateway/channel scenario in this read-only review.

## Solution Assessment

No. The implementation direction is right, but the current patch proves dispatch rather than user-visible completion delivery, so it is not yet the narrow complete fix for the linked regression.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Default exec-event delivery to the captured target:** `src/infra/heartbeat-runner.ts:1194`
  - body: With `heartbeat.every: "0m"` and the default or explicit `heartbeat.target: "none"`, this bypass now runs the heartbeat turn but leaves delivery resolution on `target-none`; the no-target branch then consumes the exec event and returns `ran` without sending the completion. Use the queued delivery context for exec events, for example an exec-event-only effective `target: "last"`, and assert the channel send rather than only that `getReplyFromConfig` ran.
