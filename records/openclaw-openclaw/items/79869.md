## Summary

Keep open. The PR targets a real disabled-heartbeat exec-event bug, but the patch can still consume completion events without delivering them under the default target-none route, skips cooldown bookkeeping for disabled-interval targeted wakes, and has no after-fix real behavior proof.

## What This Changes

The branch lets exec-event heartbeat wakes bypass a disabled periodic interval and adds scheduler/runHeartbeatOnce regression tests for heartbeat.every=0m.

## Best Possible Solution

Add a one-shot exec-event path that preserves the global heartbeat switch, uses captured delivery context for relayable completions, and keeps cooldown/flood bookkeeping, with focused tests plus real behavior proof.

## Reproduction Assessment

Yes. The narrowed current-main path is source-reproducible: background exec completion requests an exec-event wake, while heartbeat.every=0m leaves the scheduler without agent state and runHeartbeatOnce treats the interval as disabled.

## Solution Assessment

No. The PR is aimed at the right narrow area, but the current implementation does not preserve user-visible delivery under the default target-none route and bypasses the existing cooldown/flood guard for one-shot exec wakes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve captured delivery for zero-interval exec wakes:** `src/infra/heartbeat-runner.ts:1191`
  - body: With this bypass, exec-event wakes can run when heartbeat.every is 0m, but the normal delivery resolver still treats the default or explicit target-none route as no target and ignores the queued event's deliveryContext. The new test only asserts that the model ran, so a real completion can be consumed without sending the output back to the original chat; use the captured delivery context or an equivalent effective target for relayable exec completions before consuming the event.
  - confidence: 0.88
- **[P2] Keep cooldown bookkeeping for one-shot exec wakes:** `src/infra/heartbeat-runner.ts:2203-2206`
  - body: When heartbeat.every is 0m, state.agents has no HeartbeatAgentState, so this fallback calls runOnce directly and skips the evaluateWakeDeferral/recordRunBookkeeping path used by normal targeted exec wakes. Repeated background process exits for the same session can then trigger back-to-back model runs instead of respecting the min-spacing/flood guard documented for exec-event wakes.
  - confidence: 0.86
