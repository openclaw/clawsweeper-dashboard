## Summary

Keep open: the latest head addresses the earlier delivery-context and cooldown-bookkeeping concerns, but it still drops deferred zero-interval exec-event wakes and lacks after-fix real behavior proof.

## What This Changes

The PR lets targeted exec-event heartbeat wakes run when `heartbeat.every` is `0m`, relays captured delivery context, and adds scheduler/runHeartbeatOnce regression tests.

## Best Possible Solution

Keep the narrow exec-event bypass, but make deferred fallback exec wakes retry after cooldown while preserving captured delivery context and existing flood protection, then require real behavior proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main queues a scoped exec completion wake, then drops `runHeartbeatOnce` and scheduler dispatch when `heartbeat.every` resolves disabled.

## Solution Assessment

No. The direction is narrow and mostly correct, but the fallback cooldown path must requeue or arm a one-shot retry for deferred zero-interval exec events before this is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Requeue deferred zero-interval exec events:** `src/infra/heartbeat-runner.ts:2244-2245`
  - body: When `heartbeat.every` is `0m`, the fallback state lives only in `execEventAgents`, so a second exec completion within the 30s cooldown returns `skipped`/`not-due` or `min-spacing` here. `heartbeat-wake.ts` only requeues retryable busy reasons, and `scheduleNext()` returns early with no interval-backed agents, leaving that queued completion stuck until an unrelated wake.
