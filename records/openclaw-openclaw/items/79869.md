## Summary

Keep open: the PR addresses a source-reproducible disabled-heartbeat exec-event delivery bug linked to https://github.com/openclaw/openclaw/issues/62505, but latest head still has a fallback requeue gap, a failing scheduler test, and no real behavior proof.

## What This Changes

The PR changes `src/infra/heartbeat-runner.ts` to bypass disabled heartbeat intervals for targeted exec-event wakes and adds scheduler/runHeartbeatOnce regression tests.

## Best Possible Solution

Keep the narrow targeted exec-event bypass, schedule or requeue deferred zero-interval exec-event wakes, correct the cooldown test, and require live proof of background exec completion with `heartbeat.every: "0m"`.

## Reproduction Assessment

Yes for the narrowed bug by source inspection: background exec completion enqueues a system event and `exec-event` wake, while current main skips direct and scheduler dispatch when `heartbeat.every` resolves to `0m`. I did not run a live reproduction in this read-only review.

## Solution Assessment

No. The direction is narrow, but the latest patch returns deferred fallback wakes as non-retryable skips and its new cooldown test fails after 30 seconds; it needs fallback requeue/scheduling plus corrected coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Requeue deferred zero-interval exec events:** `src/infra/heartbeat-runner.ts:2243-2245`
  - body: When `heartbeat.every` is `0m`, fallback state lives only in `state.execEventAgents`. If a second targeted `exec-event` arrives during cooldown, this branch returns `skipped/not-due` or `min-spacing`; `heartbeat-wake.ts` only requeues retryable busy skips and `scheduleNext()` ignores `execEventAgents`, so the queued completion can sit until another unrelated wake. Requeue the wake or arm a timer for the fallback agent's next due time.
  - confidence: 0.91
- **[P2] Keep the cooldown test inside the cooldown:** `src/infra/heartbeat-runner.scheduler.test.ts:620-632`
  - body: This test advances 10s three times after the first run. With `DEFAULT_MIN_WAKE_SPACING_MS` at 30s and the extra 1ms timer advances, the third wake is no longer inside the cooldown, so CI sees a second run and the assertion fails. Shorten the elapsed time or assert the after-cooldown behavior separately.
  - confidence: 0.96
