## Summary

## What This Changes

The issue reports repeated bundled webhooks route registration during event-loop saturation from cron catchup, memory dreaming, and Control UI polling, and asks which plugin lifecycle path is re-entering `register()`.

## Best Possible Solution

Trace plugin activation and reload callers during the saturation scenarios, then add a focused regression proving webhooks routes register once per gateway lifecycle except explicit plugin config reload or gateway restart.

## Reproduction Assessment

Unclear: the issue includes credible timestamped logs from released Docker builds, and current source maps the log line to the webhooks registration path. I did not find a deterministic current-main command or test that reproduces re-entry under event-loop saturation.

## Solution Assessment

Unclear: a lifecycle-level fix is likely safer than only adding another webhooks idempotency guard, because route replacement already limits duplicate route-table entries but does not explain why registration re-enters.
