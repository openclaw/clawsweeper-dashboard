## Summary

Keep open: current main contains defensive same-plugin route replacement and several event-loop pressure mitigations, but I did not find code, tests, docs, changelog, or related PR evidence proving Webhooks registration is now single-shot under the reported saturation paths; merged #75986 explicitly left this cascade out of scope.

## What This Changes

The issue asks why the Webhooks plugin `register()` path is re-entered during event-loop saturation and whether a lifecycle guard should prevent repeated route registration.

## Best Possible Solution

Trace plugin activation and reload callers during saturation, then add a regression proving Webhooks routes register only on startup or explicit plugin config reload while preserving same-plugin route replacement.

## Reproduction Assessment

Unclear: the issue provides credible timestamped logs, and current source maps the repeated log line to the Webhooks `register()` path. I did not find a current-main command, test, or source path proving event-loop saturation itself triggers repeated plugin registration.

## Solution Assessment

Unclear: current `replaceExisting` behavior is useful containment but not the root-cause fix. The safer solution is to identify and constrain the lifecycle or reload caller before adding a Webhooks-only guard.
