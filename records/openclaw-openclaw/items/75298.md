## Summary

Keep this open. Current main still has evidence that the reported log line comes from the webhooks plugin entrypoint, and while adjacent safeguards exist for same-route replacement, pinned HTTP registries, deferred startup loading, and config-schema reads, I did not find source or test proof that event-loop saturation can no longer cause webhooks `register(api)` to re-enter during a single Gateway lifecycle.

## What This Changes

The issue asks OpenClaw to identify and prevent repeated webhooks plugin registration during one Gateway session when event-loop saturation follows cron catchup, memory-core dreaming, or Control UI polling.

## Best Possible Solution

Trace the plugin registry activation and deferred runtime attachment paths during the reported saturation windows, then add a focused regression proving webhooks routes register once per Gateway/plugin lifecycle unless an explicit config reload or Gateway restart occurs. Keep HTTP route replacement as a defensive registry invariant, not as the primary lifecycle fix.

## Reproduction Assessment

Unclear: the issue gives a credible log-based reproduction from released Docker builds and the current source maps the log line to the webhooks `register(api)` path. I did not find a deterministic current-main test or command that reproduces repeated registration under event-loop saturation.

## Solution Assessment

No current-main solution is proven yet. The adjacent route replacement and config-schema safeguards are useful, but the maintainable fix is to identify the re-entry caller and lock in a regression for Control UI polling, cron catchup, and memory-core dreaming pressure.
