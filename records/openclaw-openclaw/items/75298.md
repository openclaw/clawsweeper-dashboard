## Summary

Keep this open. Current main has adjacent protections for duplicate plugin HTTP-route entries, pinned route registries, and config-schema reads that previously reactivated plugin runtime, but I did not find source, test, docs, or release evidence proving that webhooks `register(api)` is no longer re-entered during event-loop saturation or that repeated webhooks registration is intended behavior.

## What This Changes

The issue reports that the bundled webhooks plugin repeatedly re-registers the same route during gateway event-loop saturation from cron catchup, memory dreaming, and Control UI polling.

## Best Possible Solution

Trace plugin registry activation and reload paths during the reported saturation windows, then add a focused regression that webhooks routes register once per Gateway/plugin lifecycle unless an explicit config reload or Gateway restart occurs. Keep route replacement as a defensive layer, not the primary fix.

## Reproduction Assessment

Unclear: the report provides credible timestamped operational evidence from released versions, and source inspection maps the log line to the webhooks registration path, but there is not yet a deterministic current-main command or test fixture that reproduces repeated registration under saturation.

## Solution Assessment

No: treating existing HTTP-route replacement or the prior config-schema guard as the solution would only mask one symptom. The safer fix is lifecycle tracing around plugin registry activation plus a regression that proves webhooks registration is not re-entered by Control UI polling, cron catchup, or memory pressure.
