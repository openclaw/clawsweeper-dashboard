## Summary

Keep open. Current main has adjacent safeguards for duplicate HTTP-route entries, route-registry pinning, plugin-load reentry, config-schema reads, and session-list event-loop yielding, but I did not find source or test evidence proving that the webhooks plugin `register(api)` is no longer re-entered during the reported event-loop saturation cases.

## What This Changes

The issue reports repeated webhooks route registration during high event-loop delay/utilization and asks whether plugin `register()` should be re-entered during a gateway lifecycle.

## Best Possible Solution

Trace the plugin registry activation/reload paths during saturated Control UI polling, cron catchup, and memory dreaming, then add a focused regression that webhooks routes register once per intended Gateway/plugin lifecycle unless an explicit config reload or Gateway restart occurs.

## Reproduction Assessment

Unclear: the report provides credible timestamped operational evidence from released versions, and current source maps the log line to the webhooks entrypoint, but there is not yet a deterministic current-main test or command that reproduces repeated registration under saturation.

## Solution Assessment

Not applicable to a proposed patch. Existing route replacement and registry pinning are useful defensive layers, but they are not enough to prove the underlying lifecycle re-entry is fixed.
