## Summary

Keep this open. Current main has strong adjacent safeguards for duplicate HTTP route table entries and for config-schema reads that previously reactivated plugin runtime, but I did not find source, test, docs, or release evidence proving that repeated webhooks `register()` entry under event-loop saturation is fixed or expected.

## What This Changes

The issue asks whether the webhooks plugin should re-register routes repeatedly during one Gateway lifecycle under event-loop saturation from cron catchup, Control UI polling, or memory dreaming, and requests the lifecycle guard or caller that should prevent it.

## Best Possible Solution

Trace plugin registry activation/reload during the reported saturation windows, then add a focused regression that webhooks routes register once per Gateway/plugin lifecycle unless an explicit config reload or Gateway restart occurs. Keep the existing route replacement behavior as a defensive layer, not as the only fix.

## Reproduction Assessment

Unclear: the issue has timestamped operational evidence from released versions, and source inspection maps the log line to the webhooks `register(api)` path, but there is not yet a deterministic current-main command or test fixture that reproduces repeated registration under saturation.

## Solution Assessment

No: closing as implemented would overfit adjacent route-table and config-schema fixes. The safer path is lifecycle tracing around plugin registry activation plus a targeted regression for webhooks registration under Control UI, cron, and memory pressure.
