## Summary

Keep this open. Current main has adjacent safeguards for HTTP-route replacement, pinned route registries, and config-schema reads that used to reactivate plugin runtime, but I did not find source, test, docs, or release evidence proving that the webhooks plugin's `register(api)` no longer re-enters during event-loop saturation or that repeated webhooks registration is intended behavior.

## What This Changes

The issue reports repeated webhooks route registration during event-loop saturation and asks whether the plugin lifecycle should prevent `webhooks` `register(api)` from re-entering during one gateway session.

## Best Possible Solution

Trace plugin registry activation and deferred reload paths during the reported saturation windows, then add a regression proving webhooks routes register once per Gateway/plugin lifecycle unless an explicit config reload or Gateway restart occurs. Keep same-plugin route replacement as a defensive route-table safeguard, not the primary lifecycle fix.

## Reproduction Assessment

Unclear: the report provides credible timestamped released-version logs, and current source maps the log line to the webhooks `register(api)` path. I did not find a deterministic current-main command or test fixture that reproduces repeated registration under event-loop saturation.

## Solution Assessment

No: treating existing HTTP-route replacement or the prior config-schema guard as the fix would only cover adjacent symptoms. The maintainable fix is to identify the lifecycle re-entry caller and prove with a focused regression that Control UI polling, cron catchup, and memory pressure do not re-register webhooks unexpectedly.
