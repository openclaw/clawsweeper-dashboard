## Summary

Keep this open. Current main has strong route-table idempotency and several related plugin metadata/reload fixes, but I did not find source, test, doc, or release evidence proving that repeated webhooks `register()` calls during event-loop saturation are fixed or expected.

## What This Changes

The issue reports that the webhooks plugin repeatedly re-registers the same configured route during event-loop saturation from cron catch-up, Control UI polling, and memory-core dreaming.

## Best Possible Solution

Trace plugin load/activation calls during the reported liveness-warning windows, then either remove the unintended re-entry or define an explicit idempotent webhooks lifecycle boundary. Add a focused regression that webhooks routes register once per Gateway/plugin lifecycle unless a real config change, plugin inventory change, or Gateway restart/reload occurs.

## Reproduction Assessment

Unclear: the report provides strong timestamped operational evidence, but there is no deterministic current-main command or test fixture that reproduces repeated webhooks `register()` calls under event-loop saturation.

## Solution Assessment

No: treating the existing route replacement guard as the fix would only cover duplicate route entries. The maintainable fix is to identify the lifecycle caller and test the relevant Control UI polling, cron catch-up, and memory-core dreaming pressure path.
