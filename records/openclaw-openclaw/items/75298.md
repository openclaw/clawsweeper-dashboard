## Summary

Keep open: current main has defensive route replacement and newer event-loop pressure mitigations for session/transcript reads, but I did not find code, tests, docs, changelog, or related-PR evidence proving webhooks registration is now single-shot under the reported saturation paths; merged #75986 explicitly left this cascade out of scope.

## What This Changes

The issue asks whether the bundled webhooks plugin should re-register routes repeatedly under event-loop saturation and what lifecycle path is causing that re-entry.

## Best Possible Solution

Trace plugin activation and reload callers during saturation, then add a regression proving webhooks routes register only on startup or explicit plugin config reload while preserving idempotent route replacement.

## Reproduction Assessment

Unclear: the issue has credible timestamped logs, and current source maps the repeated log line to the webhooks `register()` path. I did not find a current-main command, test, or source path proving event-loop saturation itself triggers repeated plugin registration.

## Solution Assessment

Unclear: the current route replacement behavior is a useful containment guard, but it is not the narrowest root-cause fix. The safer solution is to identify and constrain the lifecycle or reload caller before adding a webhooks-only guard.
