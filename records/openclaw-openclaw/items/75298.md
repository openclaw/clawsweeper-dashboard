## Summary

Keep open: current main contains idempotent route replacement and newer Control UI/event-loop mitigations, but there is still no source, test, docs, changelog, or related-PR proof that webhooks registration is single-shot under the reported saturation paths; merged #75986 explicitly treated this cascade as related but out of scope.

## What This Changes

Investigate and stop the bundled webhooks plugin from re-running route registration multiple times during event-loop saturation without an explicit gateway/plugin reload.

## Best Possible Solution

Trace plugin activation and reload callers during saturation, then constrain unexpected lifecycle reloads while preserving explicit plugin config reloads and idempotent route replacement.

## Reproduction Assessment

Unclear: the issue has credible timestamped logs, and current source maps the repeated log line to the webhooks `register()` path. I did not establish a current-main path that makes event-loop saturation, cron catchup, dreaming, or Control UI polling invoke plugin registration without config reload.

## Solution Assessment

Unclear: current route replacement is a useful containment guard, but it is not a root-cause fix. The safer path is to identify and constrain the lifecycle or reload caller before adding any webhooks-specific guard.
