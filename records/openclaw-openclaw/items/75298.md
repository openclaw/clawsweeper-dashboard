Keep open: current main still maps the repeated log line to the webhooks plugin registration path, and the later Control UI and heartbeat/event-loop mitigations do not prove that webhook registration is single-shot under the reported saturation paths.

## What This Changes

The issue reports repeated webhooks route registration during event-loop saturation from cron catchup, Control UI polling, and memory-core dreaming.

## Best Possible Solution

Trace plugin activation and reload callers under saturation, then ensure webhooks register only at startup or explicit plugin config reload while retaining idempotent route replacement.

Unclear: the report has credible timestamped release logs, and current source maps the repeated log line to the webhooks register path, but I did not establish a high-confidence current-main command or test that reproduces saturation-triggered re-registration.

## Solution Assessment
