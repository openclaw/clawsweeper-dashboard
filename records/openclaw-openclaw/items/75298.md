## Summary

Keep open: current main has adjacent route-registry and config-schema safeguards, but I did not find source, test, docs, or release evidence proving the webhooks plugin `register(api)` cannot re-enter under the reported event-loop saturation paths.

## What This Changes

The issue asks whether repeated webhooks route registration under event-loop saturation is expected and, if not, to identify or guard the plugin lifecycle re-entry.

## Best Possible Solution

Trace plugin registry activation and reload callers during the reported saturation triggers, then add a focused regression proving webhooks routes register once per Gateway lifecycle except explicit restart or plugin config reload.

## Reproduction Assessment

Unclear: the issue has credible timestamped logs from released builds, and current source maps the log line to the webhooks `register(api)` path. I did not find a deterministic current-main test or command that reproduces repeated registration under event-loop saturation.

## Solution Assessment

Unclear: the report correctly points at a lifecycle problem, but an idempotency-only webhooks guard is not the best fix unless the re-entry caller is understood. The safer path is to prove the plugin lifecycle contract and keep route replacement as a defensive fallback.
