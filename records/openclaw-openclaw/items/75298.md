## Summary

Keep this open. The current source still shows the reported log line is emitted only from the webhooks plugin registration path, while the existing route replacement, pinned route registries, config-schema guard, and sessions.list yielding are adjacent safeguards rather than proof that webhooks registration is single-shot under event-loop saturation.

## What This Changes

The issue requests investigation and a fix so the bundled webhooks plugin does not repeatedly re-register configured routes during gateway event-loop saturation from cron catchup, memory dreaming, or Control UI polling.

## Best Possible Solution

Trace the plugin/runtime lifecycle path that re-enters webhooks during saturated gateway windows, then add a focused regression proving webhooks routes register once per gateway/plugin lifecycle unless an explicit config reload or Gateway restart occurs. Keep same-plugin HTTP route replacement as defensive registry behavior, not as the primary fix.

## Reproduction Assessment

Unclear: the issue includes credible timestamped logs from released versions, and current source maps the repeated log line to the webhooks registration path. I did not find a deterministic current-main test or command that reproduces repeated webhooks registration under event-loop saturation.

## Solution Assessment

No: relying on route replacement, registry pinning, or the previous config-schema guard would treat symptoms or adjacent bugs. The maintainable solution is to isolate the re-entry caller and prove the expected lifecycle contract with a regression.
