## Summary

Keep open: current main has defensive HTTP route replacement, compatible active-registry reuse, and several event-loop pressure mitigations, but there is still no source, test, or release evidence proving webhooks registration is single-shot under the reported saturation paths. The related Control UI fix and broader CPU-spin cleanup do not cover this webhook re-registration cascade directly.

## What This Changes

The issue reports repeated webhooks route registration during event-loop saturation from cron catchup, Control UI polling, and memory-core dreaming.

## Best Possible Solution

Trace plugin activation and reload callers during saturation, then constrain the lifecycle so webhooks routes register only on startup or explicit plugin config reload while preserving idempotent route replacement.

## Reproduction Assessment

Unclear: the issue has credible timestamped release logs, and current source maps the repeated log line to the webhooks `register()` path. I did not find a high-confidence current-main command, test, or source path proving event-loop saturation itself still triggers repeated plugin registration.

## Solution Assessment

Unclear: there is no proposed patch, and the existing route replacement behavior is only a containment guard. The safer solution is to identify and constrain the reload caller before adding a webhooks-only guard or suppressing the log.
