## Summary

Keep open. Current main still lacks a Feishu delayed-ack implementation, but this external PR remains blocked by source-level correctness defects and missing after-fix real Feishu behavior proof.

## What This Changes

The PR adds an 8-second delayed Feishu p2p/private-message acknowledgment timer and Feishu bot tests for slow, fast, and group-chat behavior.

## Best Possible Solution

Land a Feishu-local revision that sends the delayed acknowledgment only after a real no-visible-output interval, cancels or suppresses it as soon as any visible Feishu reply starts, covers slow/fast/group/early-visible cases deterministically, and includes redacted real Feishu proof; broader configurable acknowledgment text remains tracked in https://github.com/openclaw/openclaw/issues/8285.

## Reproduction Assessment

Yes, at source level for the review findings. On the PR head, the delayed ack is canceled only after withReplyDispatcher settles while Feishu reply delivery can start earlier, and the fast test awaits a 9-second mock after advancing only 7,999 ms.

## Solution Assessment

No. The Feishu-local scope is plausible, but the current PR cancels too late for its no-visible-reply condition and needs a deterministic fast-path test plus early-visible-reply cancellation coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Cancel delayed ack when visible replies start:** `extensions/feishu/src/bot.ts:1835-1861`
  - body: The timer is only canceled after `withReplyDispatcher` settles, but Feishu reply dispatch can start streaming/card/plain-text/media output before settlement. A long run that already produced visible output can still send the delayed acknowledgment after 8 seconds, violating the PR's no-user-visible-reply condition and creating duplicate messages.
  - confidence: 0.9
- **[P2] Use a fast dispatcher in the fast-path test:** `extensions/feishu/src/bot.test.ts:536-537`
  - body: This test advances fake timers by only `7_999` ms and then awaits `pending`, but the shared dispatch mock resolves after `9_000` ms. Under fake timers, the promise cannot settle there, so the case can hang instead of validating that the ack is canceled before the threshold.
  - confidence: 0.88
