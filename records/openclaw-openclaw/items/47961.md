## Summary

Keep open: current main still lacks equivalent Feishu delayed-ack behavior, but this PR is not mergeable as-is because the timer can fire after a visible reply has already started, the fast-path test can hang under fake timers, and the external PR lacks real Feishu proof.

## What This Changes

The PR adds an 8-second delayed acknowledgment for slow Feishu p2p/private-message dispatches plus Feishu bot tests for slow, fast, and group-chat cases.

## Best Possible Solution

Land a Feishu-local revision that sends the delayed acknowledgment only after a real no-visible-output interval, cancels or suppresses it as soon as any visible Feishu reply starts, covers slow/fast/group/early-visible cases deterministically, and includes redacted real Feishu proof; broader cross-channel ack-text configuration remains better tracked in https://github.com/openclaw/openclaw/issues/8285.

## Reproduction Assessment

Yes, at source level. A Feishu p2p/private dispatch that emits visible output before 8 seconds but settles after 8 seconds leaves this PR's timer active, and the fast-path test awaits a 9-second fake-timer dispatch after advancing only 7,999 ms.

## Solution Assessment

No. The Feishu-local scope is plausible, but canceling only at outer dispatcher settlement is too late for the stated no-visible-reply condition, and the test needs a genuinely fast dispatcher or a clock advance that lets the pending dispatch settle.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Cancel delayed ack when visible replies start:** `extensions/feishu/src/bot.ts:1835-1861`
  - body: The timer is only canceled after `withReplyDispatcher` settles, but the Feishu reply dispatcher can start streaming/card/plain-text/media output before settlement. A long run that already produced visible output can still send the delayed acknowledgment after 8 seconds, which violates the PR body's no-user-visible-reply condition and creates duplicate/confusing messages.
  - confidence: 0.9
- **[P2] Use a fast dispatcher in the fast-path test:** `extensions/feishu/src/bot.test.ts:536-537`
  - body: The fast direct-message case advances fake timers by only 7,999 ms and then awaits `pending`, but the shared dispatch mock resolves after 9,000 ms. Under fake timers, that promise cannot settle there, so the test can hang instead of validating that the ack was canceled before the threshold.
  - confidence: 0.88
