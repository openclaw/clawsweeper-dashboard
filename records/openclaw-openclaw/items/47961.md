## Summary

Keep open: current main does not have the Feishu delayed-ack behavior, but this PR still has blocking correctness issues and lacks after-fix real Feishu proof.

## What This Changes

The PR adds an 8-second delayed acknowledgment for slow Feishu p2p/private-message dispatches plus Feishu bot tests for slow, fast, and group-chat cases.

## Best Possible Solution

Land a Feishu-local revision that cancels or suppresses the delayed ack on the first visible reply, keeps final replies intact, covers slow/fast/group/early-visible cases, and includes redacted real Feishu proof.

## Reproduction Assessment

Yes, at source level. A Feishu p2p/private dispatch that emits visible output before 8 seconds but settles after 8 seconds leaves this PR's timer active, and the fast-path fake-timer test awaits a 9-second mock after advancing only 7,999 ms.

## Solution Assessment

No. The Feishu-local direction is plausible, but canceling only at outer dispatcher settlement is too late for the stated no-user-visible-reply condition, and the fast-path test should use a fast dispatcher or advance the fake clock far enough to settle.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Cancel delayed ack when visible replies start:** `extensions/feishu/src/bot.ts:1833-1848`
  - body: The timer is only canceled after withReplyDispatcher settles, but the Feishu reply dispatcher can start streaming/card/plain-text/media output before settlement. A long run that already produced visible output can still send the delayed acknowledgment after 8 seconds, which violates the PR's no-user-visible-reply condition and creates duplicate/confusing messages.
  - confidence: 0.9
- **[P2] Use a fast dispatcher in the fast-path test:** `extensions/feishu/src/bot.test.ts:537`
  - body: The fast direct-message case advances fake timers by only 7,999 ms and then awaits pending, but the shared dispatch mock resolves after 9,000 ms. Under fake timers, that promise cannot settle there, so the test can hang instead of validating that the ack was canceled before the threshold.
  - confidence: 0.88
