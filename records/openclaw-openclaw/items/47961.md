## Summary

Keep open. Current main still does not implement this Feishu delayed-ack feature, but the PR has source-level correctness blockers and no contributor-supplied real Feishu behavior proof before merge.

## What This Changes

The PR adds an 8-second hardcoded delayed acknowledgment for slow Feishu p2p/private-message dispatches and tests slow, fast, and group-chat behavior.

## Best Possible Solution

Land a Feishu-local revision only after it suppresses the delayed acknowledgment as soon as visible Feishu output starts or is delivered, preserves final reply flow, and aligns text/timing with the broader acknowledgment UX decision.

## Reproduction Assessment

Yes, source-level for the PR defects. The diff cancels the delayed acknowledgment only after dispatcher settlement while Feishu replies can become visible before settlement, and the fast-path test awaits a 9-second mock after advancing only 7,999 ms.

## Solution Assessment

No. The Feishu-local direction is plausible, but this patch keys the delay to dispatcher settlement instead of visible reply start or delivery, and the user-visible text/timing still needs maintainer UX alignment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Cancel delayed ack when visible replies start:** `extensions/feishu/src/bot.ts:1833-1848`
  - body: The timer is only canceled after `withReplyDispatcher` settles, but Feishu can start or deliver streaming/card/plain-text/media output before settlement. A long DM that already produced visible output can still receive this extra acknowledgment after 8 seconds, violating the PR's no-visible-reply condition and creating duplicate messaging.
  - confidence: 0.9

  - body: The fast direct-message case advances fake timers by only 7,999 ms and then awaits `pending`, but the shared dispatch mock resolves after 9,000 ms. Under fake timers that promise cannot settle there, so the test can hang instead of validating fast-path cancellation.
  - confidence: 0.88
