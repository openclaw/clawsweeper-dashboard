## Summary

Keep open. This PR still proposes Feishu behavior that is not on current main, but the branch has source-level correctness defects, lacks real Feishu proof, and overlaps an open broader acknowledgment-text product/API discussion.

## What This Changes

The PR adds an 8-second delayed acknowledgment for Feishu p2p/private messages plus bot tests for slow, fast, and group-chat cases.

## Best Possible Solution

Land a Feishu-local or shared acknowledgment implementation that only fires after a real no-visible-output interval, cancels on first visible Feishu output, preserves final delivery, and aligns the text/timing contract with the broader acknowledgment-text direction.

## Reproduction Assessment

Yes, at source level for the PR defects. The PR schedules the ack before dispatch and cancels only after dispatcher settlement, while current Feishu reply dispatch can produce visible output before settlement; the fast-path test also awaits a 9-second fake-timer mock after advancing only 7,999 ms.

## Solution Assessment

No. The Feishu-local direction is plausible, but this implementation keys the delay to dispatcher settlement instead of first visible output and still needs maintainer UX alignment on text and timing.

- **[P2] Cancel delayed ack when visible replies start:** `extensions/feishu/src/bot.ts:1835-1861`

  - confidence: 0.88
