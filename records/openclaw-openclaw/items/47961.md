## Summary

Keep open. Current main still lacks this Feishu delayed-ack behavior, but the PR head has blocking correctness issues and no contributor-provided real Feishu proof for the changed runtime behavior.

## What This Changes

The branch adds a hard-coded 8-second delayed acknowledgment for Feishu p2p/private messages and bot tests for slow, fast, and group-chat cases.

## Best Possible Solution

Land a Feishu-local revision that sends the delayed acknowledgment only after a true no-visible-output interval, suppresses it as soon as any visible Feishu reply starts or is delivered, covers slow DM, fast DM, group chat, and early-visible-reply cancellation, and leaves the broader cross-channel ackText contract to https://github.com/openclaw/openclaw/issues/8285.

## Reproduction Assessment

Yes, at source level for the review findings. The PR cancels the delayed ack only after withReplyDispatcher settles while current Feishu reply delivery can start before settlement, and the fast-path test awaits a 9-second mock after advancing only 7,999 ms.

## Solution Assessment

No. The Feishu-local scope is plausible, but this implementation keys the delay to dispatcher settlement instead of visible reply start/delivery, and one added test is not deterministic under fake timers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Cancel delayed ack when visible replies start:** `extensions/feishu/src/bot.ts:1835-1861`
  - body: The timer is only canceled after withReplyDispatcher settles, but the Feishu reply dispatcher can start streaming, card, plain-text, or media output before settlement. A long run that already produced visible output can still send the delayed acknowledgment after 8 seconds, which violates the PR's no-user-visible-reply condition and creates duplicate/confusing messages.
  - confidence: 0.9
- **[P2] Use a fast dispatcher in the fast-path test:** `extensions/feishu/src/bot.test.ts:536-537`
  - body: The fast direct-message case advances fake timers by only 7,999 ms and then awaits pending, but the shared dispatch mock resolves after 9,000 ms. Under fake timers, that promise cannot settle there, so the test can hang instead of validating that the ack was canceled before the threshold.
  - confidence: 0.88
