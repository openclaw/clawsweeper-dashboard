## Summary

Keep open: the Feishu delayed-ack feature is not on current main, but this PR is not merge-ready because the branch can send the ack after visible output has already started, has a deterministic fake-timer test defect, and lacks real Feishu behavior proof.

## What This Changes

The PR adds an 8-second delayed acknowledgment for Feishu p2p/private-message dispatches plus Feishu bot tests for slow, fast, and group-chat behavior.

## Best Possible Solution

Land a Feishu-local revision that sends the delayed acknowledgment only after a real no-visible-output interval, suppresses it as soon as any visible Feishu reply starts or is delivered, preserves the final reply flow, and leaves the cross-channel config/API design to https://github.com/openclaw/openclaw/issues/8285.

## Reproduction Assessment

Yes, at source level for the PR defects. The branch cancels the delayed ack only after dispatcher settlement while current Feishu reply delivery can start before settlement, and the fast-path test awaits a 9-second mock after advancing only 7,999 ms.

## Solution Assessment

No. The Feishu-local scope is plausible, but this implementation keys the delay to dispatcher settlement instead of visible reply start/delivery, and one added test is not deterministic under fake timers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Cancel delayed ack when visible replies start:** `extensions/feishu/src/bot.ts:1835-1861`
  - body: The timer is canceled only after `withReplyDispatcher` settles, but Feishu can start or deliver streaming/card/plain-text/media replies before settlement. A long DM that already produced visible output can still receive the delayed ack at 8s, contradicting the PR's no-visible-reply condition and creating duplicate messaging.
  - confidence: 0.9

  - confidence: 0.88
