## What This Changes

The PR tightens Telegram `replyToMessageId` normalization to reject zero and negative numeric or string values and adds focused unit coverage while preserving existing thread-id parsing semantics.

## Best Possible Solution

Land the narrow Telegram plugin validation fix with a one-line `CHANGELOG.md` entry under `Unreleased` / `Fixes`, keeping the positive-ID rule in the shared reply-id normalizer and preserving negative Telegram group/thread ID handling in `parseTelegramThreadId`.

## Reproduction Assessment

Yes. Static inspection of current main is enough to reproduce the issue: the normalizer returns `0`, negative numbers, and parsed zero/negative strings instead of `undefined`, and the send path forwards that normalized value into Telegram reply parameters.

## Solution Assessment

Yes for the functional patch. Centralizing the `> 0` guard in `normalizeTelegramReplyToMessageId` is the narrowest maintainable fix because all relevant Telegram send surfaces already route through that helper; the safer remaining change is only the required changelog entry.
