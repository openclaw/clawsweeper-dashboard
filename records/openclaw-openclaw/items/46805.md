## Summary

Keep open: current main still has the Feishu mixed-ID lookup gap, and this branch should not merge as-is because it targets the old in-file resolver while main now uses `bot-sender-name.ts`; #73399 overlaps the active resolver and should be sequenced first.

## What This Changes

This PR adds Feishu `senderUserId` plumbing, changes sender-name lookup to try `user_id` before `open_id` and fallback IDs, and adds Feishu bot tests for priority, fallback, and cache behavior.

## Best Possible Solution

Port the ordered `user_id`/`open_id`/fallback lookup into `bot-sender-name.ts`, carry `senderUserId` through `FeishuMessageContext`, reconcile cache and fallback behavior with #73399, and add focused resolver/handler tests plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows mixed-ID events parse both IDs, but sender-name lookup still receives only `ctx.senderOpenId`, so `user_id` cannot be preferred when both IDs are present.

## Solution Assessment

No. The requested behavior is the right fix direction, but this branch needs to target the current `bot-sender-name.ts` resolver and be sequenced with #73399 instead of changing the old in-file resolver shape.
