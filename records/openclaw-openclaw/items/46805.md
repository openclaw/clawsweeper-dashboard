## Summary

Keep open: the requested Feishu sender-name behavior is still valid, but this branch is stale against current main. Current main resolves sender names through the split `bot-sender-name.ts` helper, while the PR implements ordered lookup in the old in-file resolver shape, so the intended `user_id` preference needs to be ported or coordinated with the adjacent Feishu display-name work in #73399.

## What This Changes

The PR adds Feishu `senderUserId` plumbing, changes sender display-name lookup to prefer `sender_id.user_id` before `open_id` and fallback IDs, and adds Feishu bot tests for priority and fallback behavior.

## Best Possible Solution

Port the ordered Feishu sender lookup into the current resolver boundary: keep an optional `senderUserId` on `FeishuMessageContext`, pass both user/open/fallback IDs from `handleFeishuMessage`, implement per-candidate cache/API lookup in `bot-sender-name.ts`, and preserve permission-error and cache behavior with focused regression coverage while coordinating with #73399.

## Reproduction Assessment

Yes. A static current-main reproduction is enough: send a Feishu event carrying both `sender_id.user_id` and `sender_id.open_id`; current main parses both but calls the sender-name resolver with only `ctx.senderOpenId`, so `user_id` cannot be preferred.

## Solution Assessment

No. The requested behavior is the right direction, but the branch is no longer in the best landing shape because current main moved sender-name resolution into `bot-sender-name.ts`; the safer fix is to adapt the change to that resolver boundary and reconcile it with #73399.
