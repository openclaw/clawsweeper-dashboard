## Summary

Keep open: current main still resolves Feishu sender names from `ctx.senderOpenId` only, so the central bug is real, but this external PR is not merge-ready because it conflicts with current Feishu sender-name refactors and lacks after-fix real Feishu behavior proof.

## What This Changes

The PR carries `sender_id.user_id` through Feishu inbound context, tries `user_id` before `open_id`/fallback IDs for sender-name lookup, and adds Feishu tests for priority, fallback, cache, and best-effort error handling.

## Best Possible Solution

Refresh the change onto current main by applying ordered `user_id` then `open_id` lookup inside `extensions/feishu/src/bot-sender-name.ts`, carry `senderUserId` through the Feishu message context, preserve current best-effort cache/error behavior, and add both focused tests and redacted real Feishu proof.

## Reproduction Assessment

We have a high-confidence source reproduction, not a live tenant reproduction: current main parses both IDs but sender-name lookup receives only `ctx.senderOpenId`, so a mixed-ID event cannot use `sender_id.user_id` for `contact.user.get`. The real placeholder-name symptom still needs redacted Feishu/Lark proof.

## Solution Assessment

Yes directionally: preferring `user_id` first with `open_id` fallback is the narrow Feishu plugin fix and matches the SDK contract. As submitted, it should be rebased into the current `bot-sender-name.ts` helper and coordinated with the open display-name fallback PR before merge.
