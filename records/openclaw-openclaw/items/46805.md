## Summary

Keep open: current main still resolves Feishu sender names from `ctx.senderOpenId` only, so the user_id-first fix remains useful. The PR needs a rebase onto the current `bot-sender-name.ts` helper and real Feishu behavior proof before merge; the related Feishu display-name PR does not implement this user_id preference.

## What This Changes

The PR changes Feishu inbound sender-name resolution to try `sender_id.user_id` before `sender_id.open_id` or fallback sender IDs, with tests for priority, cache, and fallback behavior.

## Best Possible Solution

Rebase the ordered user_id/open_id candidate logic into `extensions/feishu/src/bot-sender-name.ts`, pass both IDs from the Feishu event, keep the cache/fallback regression tests, and add redacted real Feishu proof.

## Reproduction Assessment

Yes at source level: current main accepts events with both IDs and extracts both, but the sender-name resolver receives only `ctx.senderOpenId` and performs a single contact lookup. I did not run a live Feishu tenant replay.

## Solution Assessment

Yes directionally: preferring `user_id` before `open_id` is a narrow Feishu-plugin fix for the reported attribution bug. The submitted branch should be rebased onto the current helper split and backed by real behavior proof before merge.
