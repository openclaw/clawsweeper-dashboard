## Summary

Keep open: the reported Feishu sender-name bug is valid, but this PR is stale against current main and now patches the old in-file resolver instead of the active `bot-sender-name.ts` resolver boundary.

## What This Changes

The branch adds Feishu `senderUserId` context plumbing, changes sender-name lookup to try `user_id` before `open_id` and fallback IDs, and adds Feishu bot tests for priority, cache, and fallback behavior.

## Best Possible Solution

Port the ordered `user_id`/`open_id`/fallback lookup into `bot-sender-name.ts`, carry `senderUserId` through `FeishuMessageContext`, reconcile it with #73399's cache and DM fallback changes, and add focused resolver/handler tests plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection is enough: current main parses both Feishu sender IDs but passes only `ctx.senderOpenId` into the single-candidate active resolver, so mixed-ID events cannot prefer `sender_id.user_id`.

## Solution Assessment

No. The requested behavior is the right fix direction, but this PR implements it in the old in-file resolver shape instead of the current `bot-sender-name.ts` resolver boundary.
