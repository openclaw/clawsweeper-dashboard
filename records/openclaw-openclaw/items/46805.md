## Summary

Keep open: the `user_id`-first lookup is still a useful Feishu attribution fix, but the branch is conflicting and targets the old inline resolver shape while current `main` uses `extensions/feishu/src/bot-sender-name.ts`. ## What This Changes

The PR changes Feishu sender-name resolution to prefer `sender_id.user_id`, fall back to `sender_id.open_id`, and adds mocked Feishu tests for ordering, caching, and fallback behavior.

## Best Possible Solution

Port the ordered `user_id` then `open_id` lookup into `extensions/feishu/src/bot-sender-name.ts`, preserve best-effort cache/error behavior, coordinate with adjacent Feishu display-name work, and require focused tests plus redacted live Feishu/Lark proof.

## Reproduction Assessment

Yes at source level, but not with a live tenant replay. Current main parses both Feishu sender IDs and then resolves only `ctx.senderOpenId`, so mixed-ID events cannot try `sender_id.user_id` first.

## Solution Assessment

No, not as submitted. The ordered lookup is the right narrow direction, but it must be rebased into `bot-sender-name.ts` and backed by real Feishu/Lark proof before merge.
