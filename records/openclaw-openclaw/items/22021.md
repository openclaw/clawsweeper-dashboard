## Summary

Keep open. Current main still lacks an actual-runtime-model response surface for the OpenAI-compatible Gateway endpoints: the handlers continue to use the request-side agent target model in responses, and the actual selected-model callback remains below the HTTP layer. The request is valid, but the streaming contract needs maintainer API judgment before an automated fix because normal headers are flushed before late fallback model selection can be known.

## What This Changes

The issue requests an HTTP response metadata surface, specifically `X-Actual-Model`, so clients can see which model actually handled each Gateway request.

## Best Possible Solution

Define and document a Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering non-streaming and streaming responses, then wire it from `onModelSelected` or equivalent run metadata with focused endpoint tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: exact searches find no actual-model response metadata, while the Gateway HTTP handlers emit the request-side `model` and selected-model data remains below the HTTP layer.

## Solution Assessment

Unclear. The requested header is plausible for non-streaming responses, but it is not sufficient for streaming fallback cases after headers flush; a maintainable fix needs a documented streaming metadata mechanism as well.
