## Summary

Keep open. Current main still lacks an actual-runtime-model transport surface on the OpenAI-compatible Gateway endpoints, and the remaining work is a public API-contract decision rather than conservative cleanup.

## What This Changes

Add response-level actual runtime model metadata, proposed as an `X-Actual-Model` header, for Gateway OpenAI-compatible HTTP responses.

## Best Possible Solution

Define and document a Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering non-streaming and streaming behavior, then wire it from existing selected-model plumbing with focused tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a bug reproduction because this is a feature request, but source inspection on current main confirms the observable gap: there is no actual-model response metadata and both HTTP handlers emit request-side `model` values.

## Solution Assessment

Is this the best way to solve the issue? No for a header-only implementation as a complete solution; a response header may work for non-streaming responses, but streaming fallback needs a documented SSE-compatible contract.
