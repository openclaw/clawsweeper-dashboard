## Summary

Keep open. Current main and the latest release still lack an actual-runtime-model response metadata surface for the Gateway OpenAI-compatible HTTP endpoints, and the remaining work needs a public API/streaming contract decision rather than conservative cleanup.

## What This Changes

The issue requests an HTTP response metadata surface, proposed as `X-Actual-Model`, so clients can see the actual model that handled each Gateway request.

## Best Possible Solution

Define one additive actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, including streaming behavior, then wire it from selected-model plumbing with focused tests and docs.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high-confidence proof that current main and v2026.5.12 still emit request-side model values and lack actual-runtime-model response metadata.

## Solution Assessment

No for the header-only proposal as a complete solution. A normal response header cannot cover late fallback selection in the current SSE path after headers flush, so the maintainable fix needs a documented streaming-compatible metadata contract.
