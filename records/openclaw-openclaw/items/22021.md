## Summary

Keep open: current main and the latest release still lack an actual-runtime-model response metadata contract for the OpenAI-compatible Gateway HTTP endpoints, and the remaining work is a public API/SSE product decision rather than already implemented or safely auto-fixable.

## What This Changes

The issue requests an `X-Actual-Model`-style HTTP response surface so clients can see the provider/model that actually served each Gateway request.

## Best Possible Solution

Define one documented actual-selected-model metadata contract across `/v1/chat/completions` and `/v1/responses`, including non-streaming and SSE behavior, then wire it from the existing selected-model signal with focused Gateway tests and live proxy proof.

## Reproduction Assessment

Yes. Source inspection on current main shows no actual-model Gateway response header or documented equivalent, and both OpenAI-compatible handlers return request-side model values while actual selected-model plumbing remains below the HTTP layer.

## Solution Assessment

No for the header-only proposal. The useful end state needs a unified body/header/SSE metadata contract so fallback attribution remains accurate after streaming headers have flushed.
