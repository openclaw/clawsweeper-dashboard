## Summary

Keep open. Current main still lacks an actual-runtime-model transport surface for the OpenAI-compatible Gateway endpoints, and maintainer discussion already identified the remaining blocker as API-contract shape rather than stale cleanup.

## What This Changes

The issue requests exposing the model that actually processed each Gateway HTTP request, proposed as an `X-Actual-Model` response header.

## Best Possible Solution

Define and document a Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering non-streaming and streaming behavior, then wire it from existing selected-model plumbing.

## Reproduction Assessment

Yes for the observable feature gap: source inspection on current main shows no actual-model response header or equivalent transport metadata, while both HTTP handlers emit request-side `model` values. This is not a bug reproduction in the strict sense because the issue requests a new public transport contract.

## Solution Assessment

Unclear. A response header may fit non-streaming responses, but streaming fallback needs maintainer API design because ordinary SSE response headers are flushed before late model selection is available.
