## Summary

Keep open. Current `main` and the latest release still lack an authoritative actual-runtime-model response surface for the OpenAI-compatible Gateway endpoints, and the remaining work needs an API contract decision because a plain response header cannot cover late fallback selection after SSE headers are flushed.

## What This Changes

The issue requests an HTTP response metadata surface that identifies the actual runtime model used for OpenAI-compatible Gateway requests.

## Best Possible Solution

Define and document one authoritative actual-model metadata contract across `/v1/chat/completions` and `/v1/responses`, using response fields or SSE events where streaming requires it and headers only where timing permits, then populate it from runtime model selection with primary and fallback coverage.

## Reproduction Assessment

Yes, at source level. Current-main and latest-release Gateway handlers return request-side model values and do not expose the runtime-selected model in headers, JSON response fields, or SSE metadata.

## Solution Assessment

No, the proposed header alone is not the best complete solution. A maintainable fix needs a shared HTTP/SSE metadata contract that can expose late fallback selection consistently on both Gateway endpoints.
