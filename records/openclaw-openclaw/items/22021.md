## Label Justifications

- `P2`: This is a concrete normal-priority interoperability and observability gap that affects client accuracy without causing an outage.
- `impact:session-state`: Clients currently cannot reliably distinguish requested or session-visible model identity from the model that executed a particular turn.
- `impact:auth-provider`: The missing metadata concerns the provider/model resolved after routing, authentication availability, and fallback.

## Summary

Current main and v2026.6.5 still do not expose authoritative per-request runtime-model metadata through the OpenAI-compatible Gateway endpoints. The request remains useful, but a response-header-only design cannot cover fallback selection after streaming headers are committed, so maintainers must first define the public buffered and streaming contract.

## What This Changes

Add authoritative actual-runtime-model metadata to Gateway OpenAI-compatible HTTP responses.

## Best Possible Solution

Define one documented actual-model contract across `/v1/chat/completions` and `/v1/responses`: populate authoritative selected provider/model metadata in buffered responses and expose equivalent in-band metadata for streams after fallback resolution.

## Reproduction Assessment

Yes. Current-main source provides a high-confidence path: request-side model values flow into HTTP response construction, while authoritative `onModelSelected` data is not wired into either Gateway compatibility handler.

## Solution Assessment

No. A universal response header is not the best complete solution because streaming headers are flushed before late fallback selection; the narrow maintainable solution needs explicit buffered and streaming representations.
