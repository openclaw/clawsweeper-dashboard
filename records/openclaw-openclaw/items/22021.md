## Summary

Keep open. Current main still does not expose the actual runtime model at the Gateway HTTP transport layer, and the requested header-only shape needs maintainer API-contract review because streaming responses flush headers before the model/fallback outcome is known.

## What This Changes

Add an `X-Actual-Model` response header or equivalent Gateway HTTP metadata so clients can see which model actually processed each request.

## Best Possible Solution

Define and document one Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering non-streaming and streaming behavior, then wire it from the existing selected-model callback with focused tests.

## Reproduction Assessment

Yes for the observable feature gap: source inspection on current main shows no actual-model response header or equivalent transport metadata, while both HTTP handlers emit request-side `model` values.

## Solution Assessment

No for a header-only proposal as the complete solution: a response header can cover non-streaming responses, but streaming fallback needs a documented late-bound/SSE-compatible metadata contract.
