## Summary

Keep open. Current main and v2026.5.12 still lack an actual-runtime-model response metadata surface for the Gateway OpenAI-compatible HTTP endpoints, and the remaining work needs a public API and streaming-contract decision rather than conservative cleanup.

## What This Changes

The issue requests a Gateway HTTP response metadata surface, specifically an `X-Actual-Model` header, exposing the model that actually processed each request.

## Best Possible Solution

Define one additive actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, including SSE behavior, then wire it from selected-model plumbing with focused tests and docs.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high-confidence proof that current main and v2026.5.12 still emit request-side model values and lack actual-runtime-model response metadata.

## Solution Assessment

No for the header-only proposal as a complete solution. A normal response header cannot cover late fallback selection in the current SSE path after headers flush, so the maintainable fix needs a documented streaming-compatible metadata contract.
