## Summary

Current main still lacks authoritative actual-runtime-model metadata on both OpenAI-compatible Gateway response surfaces. The requested observability remains useful, but it needs maintainer API judgment because a normal header cannot represent a fallback model selected after streaming headers are flushed; adjacent issues cover broader model observability but do not supersede this HTTP contract.

## What This Changes

Add authoritative actual-runtime-model metadata to Gateway OpenAI-compatible HTTP responses.

## Best Possible Solution

Define and document one authoritative actual-model metadata contract across `/v1/chat/completions` and `/v1/responses`, using response fields or events that work for streaming plus headers where timing permits, then populate it from runtime selection and cover primary and fallback cases.

## Reproduction Assessment

Yes, at the source level. Current-main handlers return request-side model values, contain no authoritative actual-model response surface, and flush SSE headers before the runtime-selected fallback is known.

## Solution Assessment

No, the proposed header alone is not the best complete solution. The maintainable fix needs a shared HTTP/SSE contract that can expose late fallback selection consistently on both Gateway endpoints.
