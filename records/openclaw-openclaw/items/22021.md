## Summary

Keep open: current main and v2026.6.8 still lack actual-runtime-model response metadata for the OpenAI-compatible Gateway endpoints, and the remaining work is a public API/SSE contract decision rather than a stale duplicate.

## What This Changes

Add an HTTP/transport response surface, proposed as `X-Actual-Model`, so clients can see which provider/model actually processed each Gateway request.

## Best Possible Solution

Define one documented actual-selected-model metadata contract across `/v1/chat/completions` and `/v1/responses`, including non-streaming and SSE behavior, then wire it from the existing selected-model signal with focused Gateway tests.

## Reproduction Assessment

Yes, at source level: current main's Gateway handlers return the request-side compatibility model and no actual-model response header exists. Streaming headers are flushed before late fallback attribution could be attached as a normal header.

## Solution Assessment

No, the proposed header is useful for non-streaming clients but is not the complete best solution. Maintainers should choose a unified body/header/SSE metadata contract so streaming fallback attribution remains accurate.
