- `impact:auth-provider`: The affected behavior sits in provider/model routing and fallback visibility for Gateway OpenAI-compatible requests.

## Summary

Keep open. Current main and v2026.5.19 still lack response-side actual-runtime-model metadata for the Gateway OpenAI-compatible HTTP endpoints; the narrower duplicate is already closed to this issue, while the open session/status issue and request-context header PR are adjacent rather than replacements.

## What This Changes

Request an HTTP response metadata surface, proposed as `X-Actual-Model`, exposing the provider/model that actually handled each Gateway OpenAI-compatible request.

## Best Possible Solution

Define one additive actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, including SSE behavior, then wire it from selected-model plumbing with focused Gateway tests and public docs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection on current main shows the Gateway handlers still emit request-side model values and lack actual-runtime-model response metadata.

## Solution Assessment

Is this the best way to solve the issue? No for a header-only change as the complete solution. Because streaming headers flush before late fallback selection may be known, the maintainable fix needs a documented streaming-compatible metadata contract.
