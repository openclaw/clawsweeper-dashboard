## Summary

Keep open. Current main and the latest release still lack an actual-runtime-model response surface for the OpenAI-compatible Gateway endpoints, and the proposed header-only shape still needs maintainer API-contract judgment because SSE headers are committed before late fallback attribution can be known.

## What This Changes

Request to expose the actual runtime provider/model used for each Gateway OpenAI-compatible HTTP response, proposed as an `X-Actual-Model` header.

## Best Possible Solution

Define one documented actual-selected-model metadata contract across `/v1/chat/completions` and `/v1/responses`, including JSON and SSE semantics, then wire it from the existing selected-model signal with focused Gateway tests.

## Reproduction Assessment

Yes, source-level. Current main still returns request-side compatibility model values and has no actual-model response header or equivalent metadata surface, while streaming headers are flushed before late fallback attribution could be attached as a normal header.

## Solution Assessment

No. The proposed header is useful for non-streaming clients but is not the complete best solution; maintainers should choose a unified body/header/SSE metadata contract so streaming fallback attribution remains accurate.
