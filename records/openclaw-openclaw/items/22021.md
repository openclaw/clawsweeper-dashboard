## Summary

Keep open. Current main and the latest release still lack an authoritative actual-runtime-model response surface for the OpenAI-compatible Gateway endpoints; the request remains valid, but a header-only design is not enough because streaming responses commit headers before late model fallback can be reported.

## What This Changes

The issue requests exposing the actual runtime model used for each OpenAI-compatible Gateway HTTP response, preferably via an `X-Actual-Model` response header.

## Best Possible Solution

Define one documented actual-selected-model metadata contract across `/v1/chat/completions` and `/v1/responses`, with non-streaming response metadata and an SSE-compatible event or field wired from the active model selection signal.

## Reproduction Assessment

Yes. Source inspection on current main shows both Gateway endpoints return the request-side compatibility model and no actual-model response header exists, while SSE headers are flushed before late fallback attribution could be attached as a header.

## Solution Assessment

No. The proposed header is useful for non-streaming clients but is not the complete best solution; maintainers should choose a unified body/header/SSE metadata contract so streaming fallback attribution remains accurate.
