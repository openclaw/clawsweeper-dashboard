## Summary

Keep open. Current main still lacks an actual-runtime-model response surface for the OpenAI-compatible Gateway endpoints, and the related open PR is adjacent tracing work rather than a shipped fix for this transport-level model visibility request.

## What This Changes

Add Gateway HTTP response metadata that exposes the model actually used for each request, especially when fallback or stale session state makes the requested model unreliable.

## Best Possible Solution

Define and document a Gateway actual-model metadata contract for /v1/chat/completions and /v1/responses, covering non-streaming and streaming behavior, then wire it from selected-model run metadata with focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: exact searches find no actual-model response metadata, while the compat HTTP handlers emit the request-side model and selected-model data remains below the HTTP layer.

## Solution Assessment

Unclear. A response header may be suitable for non-streaming responses, but streaming fallback cases need a documented SSE/body/trailer-style metadata mechanism because headers are flushed before the final selected model can be known.
