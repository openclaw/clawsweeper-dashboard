## Summary

Keep open. Current main still lacks an actual-runtime-model transport surface on the Gateway OpenAI-compatible HTTP endpoints, and the remaining work needs a maintainer API-contract decision rather than conservative cleanup.

## What This Changes

Add transport-level actual-runtime model metadata, proposed as an `X-Actual-Model` response header, to Gateway OpenAI-compatible responses.

## Best Possible Solution

Define and document one Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering non-streaming and streaming behavior, then thread selected provider/model data from the existing runtime callback/result path with focused tests.

## Reproduction Assessment

Yes for the observable feature gap: source inspection on current main shows no actual-model response header or equivalent transport metadata, while both HTTP handlers emit request-side `model` values.

## Solution Assessment

No for the header-only proposal as a complete solution. A response header may work for non-streaming responses, but streaming fallback needs a documented SSE-compatible, body, trailer, or otherwise late-bound metadata contract.
