## Summary

Keep open. Current main still lacks an actual-runtime-model response metadata surface for the Gateway OpenAI-compatible HTTP endpoints; the handlers continue to emit request-side model values, and the remaining work is a public API contract decision for non-streaming and streaming responses.

## What This Changes

Request to add an actual-runtime-model response metadata surface, proposed as `X-Actual-Model`, to Gateway OpenAI-compatible HTTP responses.

## Best Possible Solution

Define and document one additive Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, including non-streaming and streaming behavior, then wire it from selected-model plumbing with focused tests and docs.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high-confidence proof that current main still emits request-side model values and lacks actual-runtime-model response metadata.

## Solution Assessment

No for the header-only proposal as a complete solution. A response header may cover non-streaming responses, but streaming fallback requires a documented late-bound metadata contract such as an SSE event, response body field, trailer, or equivalent compatible mechanism.
