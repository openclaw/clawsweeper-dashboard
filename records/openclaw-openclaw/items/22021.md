## Summary

Keep open. Current main still lacks an actual-runtime-model transport surface for the OpenAI-compatible Gateway endpoints, and the remaining work is an API-contract decision rather than a stale session-metadata cleanup.

## What This Changes

The issue requests an `X-Actual-Model` HTTP response header exposing the provider/model that actually processed each Gateway OpenAI-compatible request.

## Best Possible Solution

Define and document a Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering non-streaming and streaming behavior, then wire selected provider/model data from the existing selection callback or run metadata.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: exact searches find no actual-model transport metadata, both compat HTTP handlers emit the request-side `model`, and selected-model data remains below the HTTP layer.

## Solution Assessment

Unclear. A response header may be the right non-streaming shape, but streaming fallback cases need a documented SSE/body/trailer-style metadata mechanism because ordinary headers are flushed too early.
