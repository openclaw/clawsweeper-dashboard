## Summary

Keep open: current main still lacks a response-side actual-runtime-model surface for the Gateway OpenAI-compatible HTTP endpoints, and the remaining work is a public API/SSE contract decision rather than a small header-only bug fix.

## What This Changes

Request an additive Gateway HTTP response metadata surface, proposed as `X-Actual-Model`, so clients can see the model that actually processed each OpenAI-compatible request.

## Best Possible Solution

Define one documented additive actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, including SSE behavior, then wire it from active model selection plumbing with focused Gateway tests and docs.

## Reproduction Assessment

Yes. Source inspection on current main shows the Gateway handlers still emit request-side model values and have no actual-runtime-model response metadata contract.

## Solution Assessment

No for the header-only proposal as the complete solution. The maintainable fix needs a documented streaming-compatible metadata contract because late fallback selection can happen after headers are flushed.
