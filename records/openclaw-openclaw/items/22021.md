## Summary

Keep open: current main and latest release still lack actual-runtime-model response metadata for the OpenAI-compatible Gateway endpoints, and the remaining work is a public API/SSE contract decision rather than a stale duplicate or already-implemented feature.

## What This Changes

The issue requests an HTTP response metadata surface, `X-Actual-Model`, so clients can see which runtime model actually handled each Gateway request.

## Best Possible Solution

Define one documented actual-selected-model metadata contract across `/v1/chat/completions` and `/v1/responses`, including non-streaming and SSE behavior, then wire it from the existing selected-model signal with focused Gateway tests.

## Reproduction Assessment

Yes, at source level: current main has no actual-model Gateway response header or documented equivalent, and both OpenAI-compatible handlers return request-side `model` values while the actual selected-model signal remains below the HTTP layer.

## Solution Assessment

No: the requested header is a useful non-streaming shape, but the best solution must define a unified body/header/SSE metadata contract so streaming fallback attribution remains accurate after headers are flushed.
