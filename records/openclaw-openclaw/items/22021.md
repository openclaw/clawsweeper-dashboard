## Summary

Keep open: current main and v2026.6.10 still lack an actual-runtime-model response metadata contract for the OpenAI-compatible Gateway HTTP endpoints, and this remains the canonical issue for the HTTP/SSE response surface.

## What This Changes

The issue requests adding an `X-Actual-Model` response metadata surface so HTTP clients can see which runtime model actually handled each Gateway request.

## Best Possible Solution

Keep this issue as the canonical Gateway HTTP actual-model metadata contract and define one documented design for `/v1/chat/completions` and `/v1/responses`, including non-streaming and streaming behavior.

## Reproduction Assessment

Yes. Source inspection shows the current Gateway response paths use the request-side `model` value and have no actual-model response header or selected-model callback wiring; I did not run a live HTTP request because the read-only source check proves the gap.

## Solution Assessment

No for the literal header-only proposal. A plain HTTP header cannot represent a fallback-selected model after SSE headers flush, so the best solution is a documented metadata contract that covers non-streaming bodies and streaming events or another compatible streaming surface.
