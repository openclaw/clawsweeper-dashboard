## Summary

Keep open. Current main still lacks an actual-runtime-model transport surface on the Gateway OpenAI-compatible HTTP endpoints, and the remaining work is a public API-contract decision rather than conservative cleanup.

## What This Changes

Request adding response-level actual-model metadata, proposed as `X-Actual-Model`, so HTTP clients can see which backend model handled each Gateway request.

## Best Possible Solution

Define and document a Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, including streaming behavior, then wire it from existing selected-model plumbing with focused tests.

## Reproduction Assessment

Yes for the observable feature gap: source inspection on current main shows no actual-model response header or equivalent transport metadata, while both HTTP handlers emit request-side `model` values. This is not a strict bug reproduction because the issue requests a new public transport contract.

## Solution Assessment

No for the header-only proposal as a complete solution. A response header may work for non-streaming responses, but streaming fallback needs maintainer API design because normal SSE response headers are flushed before late model selection can be known.
