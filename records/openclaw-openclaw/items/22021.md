## Summary

Keep open. Current main still lacks an actual-runtime-model transport surface for the OpenAI-compatible Gateway endpoints, and the remaining work needs maintainer API-contract judgment rather than stale cleanup.

## What This Changes

Add a Gateway HTTP response surface, proposed as `X-Actual-Model`, that exposes the runtime model that actually processed each OpenAI-compatible request.

## Best Possible Solution

Define and document a Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering non-streaming and streaming behavior, then wire selected provider/model data from existing selection callbacks or run metadata.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction of the gap: exact searches find no actual-model transport metadata, both compat handlers emit request-side `model`, and selected-model data remains below the HTTP layer.

## Solution Assessment

Unclear. A response header may be the right non-streaming shape, but streaming fallback cases need a documented SSE, body, trailer, or equivalent metadata mechanism because ordinary headers flush too early.
