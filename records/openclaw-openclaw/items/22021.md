## Summary

Keep open. Current main still lacks an actual-runtime-model transport surface on the Gateway OpenAI-compatible HTTP endpoints, and prior maintainer review plus the streaming code path make this an API-contract decision rather than conservative cleanup.

## What This Changes

The issue asks OpenClaw to expose the actual runtime model per HTTP response, preferably via an `X-Actual-Model` response header.

## Best Possible Solution

Define and document one Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering non-streaming and streaming behavior, then wire it from selected-model plumbing with focused tests.

## Reproduction Assessment

Yes for the observable feature gap: source inspection on current main shows no actual-model response header or equivalent transport metadata, while both Gateway HTTP handlers emit request-side `model` values. This is feature-gap proof rather than a failing existing bug contract.

## Solution Assessment

No for the header-only proposal as a complete solution; a response header can cover non-streaming responses, but streaming fallback needs a documented SSE-compatible or otherwise late-bound metadata contract.
