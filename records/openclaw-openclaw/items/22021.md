## Summary

Keep open. Current main still lacks an actual-runtime-model transport surface on the Gateway OpenAI-compatible HTTP endpoints, and the closed duplicate at https://github.com/openclaw/openclaw/issues/55450 points back to this issue as the canonical response-attribution contract.

## What This Changes

Request to expose the actual backend model used by each Gateway OpenAI-compatible HTTP request, proposed as an `X-Actual-Model` response header.

## Best Possible Solution

Define and document one Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, including non-streaming and streaming behavior, then wire it from selected-model plumbing with focused tests.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. High-confidence source inspection shows the requested transport metadata is absent and both HTTP handlers emit request-side model values.

## Solution Assessment

No for the header-only proposal as a complete solution. A response header may work for non-streaming responses, but streaming fallback needs a documented SSE-compatible or otherwise late-bound metadata contract.
