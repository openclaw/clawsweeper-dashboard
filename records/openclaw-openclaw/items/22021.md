## Summary

Keep open. Current main still lacks an actual-runtime-model response metadata surface for the Gateway OpenAI-compatible HTTP endpoints, and the remaining work is an API contract decision rather than conservative cleanup.

## What This Changes

The issue requests a Gateway HTTP response metadata surface that exposes the actual runtime model used for each OpenAI-compatible request.

## Best Possible Solution

Define and document one additive actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, including streaming behavior, then wire it from selected-model plumbing with focused tests and docs.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high-confidence proof that current main still emits request-side model values and lacks actual-runtime-model response metadata.

## Solution Assessment

No for the header-only proposal as a complete solution. A response header can cover some non-streaming responses, but streaming fallback requires a documented late-bound metadata contract such as an SSE event, response body field, trailer, or equivalent compatible mechanism.
