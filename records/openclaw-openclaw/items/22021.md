## Summary

Keep open. Current main still lacks an actual-runtime-model transport surface for the Gateway OpenAI-compatible HTTP endpoints; the prior session-metadata fixes and the related open context-header PR do not implement the requested response metadata.

## What This Changes

Request to add an HTTP response-level actual runtime model surface, proposed as `X-Actual-Model`, for Gateway OpenAI-compatible responses.

## Best Possible Solution

Define and document a Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering non-streaming and streaming behavior, then wire it from selected-model run metadata with focused tests and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: exact searches find no actual-model transport metadata, while both compat HTTP handlers emit the request-side `model` and selected-model data remains below the HTTP layer.

## Solution Assessment

Unclear. The requested header is plausible for non-streaming responses, but streaming fallback cases need a documented SSE event, response body field, trailer, or similar mechanism because headers are flushed before late model selection can be known.
