## Summary

Keep open. Current main still lacks the requested actual-runtime-model transport metadata on the OpenAI-compatible Gateway endpoints, and the existing discussion correctly identifies this as a remaining API-contract decision rather than stale cleanup.

## What This Changes

Request to add an `X-Actual-Model` HTTP response header, or equivalent transport metadata, showing which model actually processed each Gateway request.

## Best Possible Solution

Define and document a Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering both non-streaming and streaming behavior, then wire it from existing selected-model plumbing with focused endpoint tests.

## Reproduction Assessment

Yes for the observable feature gap: source inspection on current main shows no actual-model response header or equivalent transport metadata, while both HTTP handlers emit request-side `model` values. This is not a strict bug reproduction because the issue requests a new public transport contract.

## Solution Assessment

Unclear. A response header may work for non-streaming responses, but streaming fallback needs maintainer API design because normal SSE response headers are flushed before late model selection can be known.
