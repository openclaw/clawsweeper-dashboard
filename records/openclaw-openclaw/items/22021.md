## Summary

Keep open. Current main still lacks an actual-runtime-model transport surface on the OpenAI-compatible Gateway endpoints, and the remaining work is a public API-contract decision rather than conservative cleanup.

## What This Changes

The issue requests an `X-Actual-Model` response header, or equivalent HTTP transport metadata, that tells clients which backend model actually served each Gateway request.

## Best Possible Solution

Define and document a Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering non-streaming and streaming behavior, then wire it from existing selected-model plumbing with focused tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the observable feature gap: source inspection on current main shows no actual-model response header or equivalent transport metadata, while both HTTP handlers emit request-side `model` values.

## Solution Assessment

Is this the best way to solve the issue? No for the header-only proposal as a complete solution; a response header may work for non-streaming responses, but streaming fallback needs a documented SSE-compatible contract.
