## Summary

Keep open. Current main still lacks an actual-runtime-model transport surface for the OpenAI-compatible Gateway endpoints, and the existing maintainer discussion correctly identifies this as an API-contract gap rather than a stale session-metadata bug.

## What This Changes

Add an HTTP response surface that exposes the model that actually handled each Gateway request.

## Best Possible Solution

Define and document a Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering non-streaming and streaming behavior, then wire it from selected-model run metadata with focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: exact searches find no actual-model transport metadata, while both compat HTTP handlers emit the request-side `model` and selected-model data remains below the HTTP layer.

## Solution Assessment

Unclear. The requested header is plausible for non-streaming responses, but streaming fallback cases need a documented SSE/body/trailer-style metadata mechanism because headers are flushed before late model selection can be known.
