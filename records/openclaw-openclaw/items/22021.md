## Label Justifications

- `P2`: This is a normal-priority Gateway transparency enhancement with limited blast radius, not an outage or urgent regression.
- `impact:session-state`: The request is about per-request model state visibility when persisted or displayed session metadata can lag runtime routing.
- `impact:auth-provider`: The affected behavior sits in provider/model routing and fallback visibility for Gateway OpenAI-compatible requests.

## Summary

Keep open. Current main and the latest release still lack an actual-runtime-model response metadata surface for the Gateway OpenAI-compatible HTTP endpoints, and the remaining work needs a maintainer API/streaming contract decision rather than conservative cleanup.

## What This Changes

Add a Gateway HTTP response metadata surface, proposed as `X-Actual-Model`, so clients can see which provider/model actually handled each request.

## Best Possible Solution

Define one additive actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, including SSE behavior, then wire it from selected-model plumbing with focused tests and public docs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a strict bug reproduction because this is a feature request, but source inspection gives high-confidence proof that current main still emits request-side model values and lacks actual-runtime-model response metadata.

## Solution Assessment

Is this the best way to solve the issue? No for the header-only proposal as a complete solution; because streaming headers flush before late fallback selection, the maintainable fix needs a documented streaming-compatible metadata contract.
