## Summary

Keep open. Current main and v2026.5.19 still lack an actual-runtime-model response metadata surface for the Gateway OpenAI-compatible HTTP endpoints; https://github.com/openclaw/openclaw/issues/55450 was closed as a duplicate of this issue, while https://github.com/openclaw/openclaw/issues/51441, https://github.com/openclaw/openclaw/issues/7006, and https://github.com/openclaw/openclaw/pull/54985 are adjacent work rather than replacements.

## What This Changes

The issue requests an HTTP response metadata surface, specifically `X-Actual-Model`, so clients can see which runtime model actually handled each Gateway OpenAI-compatible request.

## Best Possible Solution

Define one additive actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, including SSE behavior, then wire it from selected-model plumbing with focused Gateway tests and public docs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives high-confidence proof for this feature gap: current main still emits request-side model values and lacks actual-runtime-model response metadata.

## Solution Assessment

Is this the best way to solve the issue? No for the header-only proposal as a complete solution; because streaming headers flush before late fallback selection, the maintainable fix needs a documented streaming-compatible metadata contract.
