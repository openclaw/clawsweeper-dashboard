## Summary

Keep open. Current main and the latest release still lack an actual-runtime-model response surface for the OpenAI-compatible Gateway endpoints, and the proposed header-only shape needs maintainer API-contract judgment because streaming responses flush headers before fallback attribution can be known.

## What This Changes

The issue requests an `X-Actual-Model` HTTP response header, or equivalent transport metadata, so clients can see which provider/model actually processed each Gateway request.

## Best Possible Solution

Define one documented actual-selected-model metadata contract across `/v1/chat/completions` and `/v1/responses`, including non-streaming and SSE semantics, then wire it from the existing selected-model signal with focused Gateway tests.

## Reproduction Assessment

Yes, at source level. Current main's Gateway handlers return the request-side compatibility model and no actual-model response header exists, while streaming headers are flushed before late fallback attribution could be attached as a header.

## Solution Assessment

No. The proposed header is useful for non-streaming clients but is not the complete best solution; maintainers should choose a unified body/header/SSE metadata contract so streaming fallback attribution remains accurate.
