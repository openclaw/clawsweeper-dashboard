## Summary

Keep open. Current main still lacks an actual-runtime-model response metadata surface for the Gateway OpenAI-compatible HTTP endpoints, and this remains the canonical transport contract issue rather than cleanup work that can be closed.

## What This Changes

The request is to expose the model that actually served each Gateway OpenAI-compatible HTTP request, proposed as an `X-Actual-Model` response header.

## Best Possible Solution

Define one additive actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, including SSE behavior, then wire it from selected-model plumbing with focused Gateway tests and public docs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a strict bug reproduction because this is a feature request, but source inspection gives high-confidence proof that current main still emits request-side model values and lacks actual-runtime-model response metadata.

## Solution Assessment

Is this the best way to solve the issue? No for the header-only proposal as a complete solution; because streaming headers flush before late fallback selection, the maintainable fix needs a documented streaming-compatible metadata contract.
