## Summary

Keep open: current main and v2026.6.5 still lack actual-runtime-model response metadata on the Gateway OpenAI-compatible HTTP surfaces, and the remaining work needs a public HTTP/SSE contract decision rather than cleanup.

## What This Changes

The issue requests an `X-Actual-Model` HTTP response header so clients can see which backend model actually served each Gateway request.

## Best Possible Solution

Define one documented additive actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, including streaming behavior, then wire it from active model-selection plumbing with focused Gateway tests and docs.

## Reproduction Assessment

Yes, source-level. Current main still carries request-side model values into Gateway JSON/SSE responses and lacks a documented actual-runtime-model response metadata surface.

## Solution Assessment

No for the header-only proposal as a complete fix. The maintainable solution needs a documented streaming-compatible metadata contract because fallback selection can happen after SSE headers are flushed.
