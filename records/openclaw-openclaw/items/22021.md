## Summary

Keep open: current main and v2026.6.5 still lack an actual-runtime-model response metadata surface for the OpenAI-compatible Gateway endpoints, and this remains a public HTTP/SSE contract decision rather than conservative cleanup.

## What This Changes

The issue requests a Gateway HTTP response header that exposes the actual runtime model used for each request.

## Best Possible Solution

Define one documented additive actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, including streaming semantics, then wire it from active model-selection plumbing with focused Gateway tests and docs.

## Reproduction Assessment

Yes, source-level. Current main still carries request-side model values into Gateway JSON/SSE responses and lacks a documented actual-runtime-model response metadata surface.

## Solution Assessment

No for the header-only proposal as a complete fix. The maintainable solution needs a documented streaming-compatible metadata contract because late fallback selection can happen after headers are flushed.
