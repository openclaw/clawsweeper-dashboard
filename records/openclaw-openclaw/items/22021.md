## Summary

Keep open. Current main still lacks an actual-runtime-model transport surface for the Gateway OpenAI-compatible HTTP endpoints, and the related closed duplicate points back to this issue as the canonical response-attribution contract.

## What This Changes

The issue requests an HTTP response header exposing the actual runtime model used for each Gateway OpenAI-compatible request.

## Best Possible Solution

Define and document one Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, including non-streaming and streaming behavior, then wire it from selected-model plumbing with focused tests.

## Reproduction Assessment

Not applicable as a strict bug reproduction because this is a feature request. Source inspection gives high-confidence proof that the requested transport metadata is absent and the handlers still emit request-side model values.

## Solution Assessment

No for the header-only proposal as a complete solution. A response header may cover non-streaming responses, but streaming fallback after `setSseHeaders` needs a documented SSE-compatible or otherwise late-bound metadata contract.
