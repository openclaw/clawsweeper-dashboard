## Summary

Keep open. Current main still lacks an actual-runtime-model transport surface on the Gateway OpenAI-compatible HTTP endpoints, and the remaining work is a public API contract decision rather than conservative cleanup.

## What This Changes

The issue requests an `X-Actual-Model` or equivalent Gateway HTTP response metadata surface so clients can tell which runtime model served each request.

## Best Possible Solution

Define and document one Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering non-streaming and streaming behavior, then wire it from existing selected-model plumbing with focused tests.

## Reproduction Assessment

Yes for the observable feature gap: source inspection on current main shows no actual-model response header or equivalent transport metadata, while both HTTP handlers emit request-side `model` values.

## Solution Assessment

No for the header-only proposal as a complete solution; a response header may work for non-streaming responses, but streaming fallback needs a documented SSE-compatible or otherwise late-bound metadata contract.
