## Summary

Keep open: current main still lacks a response-side actual-runtime-model surface for the Gateway OpenAI-compatible HTTP endpoints, and the related closed issue points back here while adjacent open work does not replace this API contract decision.

## What This Changes

The issue requests an HTTP response metadata surface that reveals the model that actually processed each Gateway OpenAI-compatible request.

## Best Possible Solution

Define and document one additive actual-model metadata contract for /v1/chat/completions and /v1/responses, including SSE behavior, then wire it from the active model selection plumbing with focused Gateway tests and docs.

## Solution Assessment

No for the header-only proposal as the complete solution. The maintainable fix needs a documented streaming-compatible metadata contract because late fallback selection can happen after headers are flushed.
