## What This Changes

The PR changes Android gateway readiness and notification forwarding so `notifications.changed` node events are queued until the gateway `connect` handshake completes and replayed after reconnect.

## Best Possible Solution

Land this Android lifecycle and notification-replay fix only after exact-head Android validation and redacted real Android app or emulator plus real gateway reconnect proof show notification events wait for connect and deliver after reconnect.

## Reproduction Assessment

Yes, at source level: current main publishes `currentConnection` before `conn.connect()` completes, notification forwarding sends through that readiness gate without retry, and the gateway rejects non-`connect` first requests. I did not run a live Android device or emulator reconnect scenario in this read-only review.

## Solution Assessment

Yes for the code direction: separating connecting and ready sockets, then queueing and reconciling `notifications.changed`, is the narrow owner-boundary fix for the linked bug. It is not merge-ready until exact-head validation and real Android/gateway reconnect proof are present.
