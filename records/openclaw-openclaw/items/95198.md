## Summary

Keep open: current main and v2026.6.8 still normalize one-segment OpenRouter model IDs to an `openrouter/<id>` runtime model and send `model.id` directly to the OpenAI-compatible transport, while the documented OpenRouter DeepSeek route uses `deepseek/deepseek-v4-flash`; no merged canonical fix owns this exact failure.

## What This Changes

The issue reports that the advertised OpenRouter short DeepSeek model ref resolves to an invalid OpenRouter route instead of the provider-local `deepseek/deepseek-v4-flash` model ID.

## Best Possible Solution

Align OpenRouter model listing, config validation, model selection, and transport so every advertised OpenRouter DeepSeek V4 ref maps to the documented provider-local `deepseek/...` request ID while preserving intentional router models such as `openrouter/openrouter/auto` and Fusion.

## Reproduction Assessment

Yes. The reported CLI flow is source-reproducible on current main: model selection normalizes the short OpenRouter ref to `openrouter/deepseek-v4-flash`, and the completions transport sends that value as the API `model` field; I did not run the live OpenRouter request.

## Solution Assessment

No current solution is in place. The best fix is a narrow consistency repair across model catalog/listing and OpenRouter request ID normalization, not a broad rewrite that strips all `openrouter/*` namespaces.
