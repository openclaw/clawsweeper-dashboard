## Summary

Keep open: current main and the latest release still preserve the short OpenRouter DeepSeek ref instead of expanding it to the upstream OpenRouter API slug, and the open linked PRs are candidate fixes rather than grounds to close this issue before merge.

## What This Changes

Fix OpenRouter API model normalization so catalog-listed short refs like `openrouter/deepseek-v4-flash` are sent to OpenRouter as `deepseek/deepseek-v4-flash` instead of a duplicated provider-prefixed ref.

## Best Possible Solution

Review and land one narrow OpenRouter-owned fix that expands the two catalog-listed short DeepSeek V4 refs while preserving native OpenRouter routes like `openrouter/auto`, then close this issue after merge.

## Reproduction Assessment

Yes, at source level: current main can normalize a bare OpenRouter DeepSeek model into `openrouter/deepseek-v4-flash`, and the OpenRouter API normalizer then preserves that short provider-prefixed ref. I did not run an authenticated live OpenRouter completion in this read-only review.

## Solution Assessment

Yes, the best fix is provider-owned API model normalization with focused regression tests for short DeepSeek V4 refs, native OpenRouter route preservation, and nested provider-qualified refs. Generic provider parsing should stay unchanged.
