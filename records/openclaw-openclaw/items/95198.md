## Summary

Keep open: current main and the latest release still preserve the reported short OpenRouter DeepSeek ref instead of mapping it to the upstream API slug, and several open closing PRs are still unmerged implementation candidates.

## What This Changes

The issue reports that OpenRouter short model refs such as `openrouter/deepseek-v4-flash` are accepted by catalog/config surfaces but are sent to OpenRouter with a duplicated provider prefix instead of the upstream `deepseek/...` slug.

## Best Possible Solution

Land one narrow OpenRouter-owned normalization fix, preferably the focused candidate at https://github.com/openclaw/openclaw/pull/95268 or an equivalent maintainer-chosen PR, then close this issue and supersede the duplicate candidate PRs.

## Reproduction Assessment

Yes at source level: current main can produce `openrouter/deepseek-v4-flash`, the OpenRouter API normalizer preserves it as API-facing, and the public OpenRouter catalog lists the namespaced `deepseek/...` slugs instead. I did not run an authenticated live completion in this read-only review.

## Solution Assessment

Yes: the best fix belongs in the OpenRouter provider normalizer, mapping only surfaced short aliases to upstream slugs while preserving native `openrouter/*` routes. Generic provider parsing or config migration would be broader than necessary for this bug.
