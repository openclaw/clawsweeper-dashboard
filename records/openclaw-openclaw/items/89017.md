## Summary

Keep this PR open, but the current patch is not merge-ready: it fixes the reported resume path by treating a matching Control UI session id as fresh, which also bypasses configured daily/idle rollover for ordinary Control UI sends.

## What This Changes

This PR threads `chat.send`'s client-provided `sessionId` into reply session initialization and reuses a matching current session entry, with gateway and session regression tests.

## Best Possible Solution

Honor the daily/idle reset contract by default, and add a distinct reconnect-resume signal or scoped policy if Control UI needs to suppress only disconnect-induced rollovers.

## Reproduction Assessment

Yes. Source inspection shows Control UI sends the current `sessionId` on follow-up `chat.send`, and the PR treats a matching requested id as fresh before stale reset handling.

## Solution Assessment

No. The current implementation is too broad because the same `sessionId` is used for ordinary sends, not only reconnect recovery; a reconnect-specific signal or policy that preserves configured rollovers is safer.
