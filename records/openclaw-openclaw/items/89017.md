## Label Justifications

## Summary

Keep open: the PR has a useful root-cause hook and credible live gateway proof, but the current implementation makes any matching Control UI session id count as fresh. Because Control UI already sends `currentSessionId` on ordinary sends, this can bypass configured idle/daily rollover for existing users.

## What This Changes

This PR passes the current webchat session id from `chat.send` into reply initialization and adds gateway/session tests for matching and stale requested ids.

## Best Possible Solution

Keep the gateway session-id handoff as an identity/resume signal, but preserve configured idle/daily rollover unless maintainers explicitly approve and document a Control UI resume override with focused tests.

## Reproduction Assessment

Yes for the merge blocker: source inspection shows `loadChatHistory` records `currentSessionId`, `requestChatSend` sends it on ordinary Control UI sends, and the PR treats a matching requested id as fresh even when configured freshness is stale. The original Mac sleep issue has reporter steps and PR live-gateway proof, but I did not rerun a physical sleep/reconnect scenario in this read-only review.

## Solution Assessment

No. Passing the id through is a plausible root-cause hook, but making it sufficient for `freshEntry` is too broad because normal Control UI sends include that id; the safer fix needs a narrower reconnect/resume rule or an explicit maintainer-approved policy change.
