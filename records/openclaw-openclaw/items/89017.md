## Summary

Keep this PR open: it targets a real Control UI session-continuity bug and includes useful live gateway proof, but the current patch makes a matching `sessionId` override configured idle/daily rollover for ordinary Control UI sends.

## What This Changes

The PR passes `chat.send`'s `sessionId` through reply options into `initSessionState` and adds gateway/session tests for matching and stale asserted session ids.

## Best Possible Solution

Keep the gateway session-id handoff, but make it a bounded resume/identity signal that preserves configured idle/daily rollover unless maintainers explicitly approve and document a Control UI resume override with tests.

## Reproduction Assessment

Yes for the merge blocker: source inspection shows Control UI stores `currentSessionId`, sends it on normal `chat.send`, and the PR treats a matching requested id as fresh even when configured freshness is stale. I did not rerun a physical Mac sleep/Docker/Chrome wake cycle.

## Solution Assessment

No. Passing the id through is a plausible root-cause hook, but making it sufficient for `freshEntry` is too broad unless maintainers explicitly choose that resume policy.
