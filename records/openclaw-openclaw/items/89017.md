## Summary

Keep open: the PR is a meaningful linked fix candidate for the Control UI reconnect session-loss bug, and current main still lacks the reconnect-only resume path. The latest diff appears to repair the earlier public protocol/SDK exposure issue, so the remaining blocker is maintainer acceptance of the scoped one-shot session freshness exception.

## What This Changes

The PR adds a private Control UI reconnect marker, Gateway stripping/validation, internal reply/session resume options, docs, and regression coverage so one matching reconnect send can preserve the visible session without adding a public `chat.send` resume field.

## Best Possible Solution

Land this PR only after maintainer acceptance of the scoped one-shot Control UI freshness exception; otherwise keep the public protocol and SDK closed and choose a different reconnect UX such as proactive stale-session notification.

## Reproduction Assessment

Yes, at source level: current main rotates stale daily/idle entries on the next send, and Control UI currently forwards only the visible `sessionId` without a reconnect-specific resume intent. I did not run the physical sleep/Docker/Chrome scenario in this read-only review.

## Solution Assessment

Yes, pending maintainer policy acceptance: the current diff is a bounded owner-boundary fix because it keeps the public Gateway protocol and Plugin SDK closed, scopes the marker to operator UI clients, requires the matching session id, and preserves ordinary stale-send rollover. The safer alternative, if maintainers reject the exception, is a proactive stale-session notice instead of implicit resume.
