## Summary

Keep open for maintainer/API review. The patch is coherent, has strong live Gateway proof, and I found no blocking code-review defect, but it adds a public `chat.send.resumeSession` parameter and a narrow daily/idle rollover exception that maintainers need to accept before merge.

## What This Changes

The PR adds an optional `chat.send.resumeSession` reconnect hint through Control UI, Gateway protocol and Swift generation, session initialization, docs, and regression tests so a visible webchat session can be preserved after reconnect.

## Best Possible Solution

Merge only after maintainers explicitly accept the additive protocol contract and narrow reconnect rollover exception; otherwise rework toward an internal Gateway-owned resume mechanism.

## Reproduction Assessment

Yes, from source and the linked report, but I did not live-reproduce the Mac sleep/Docker/Chrome path here. Current main sends the visible `sessionId` without a reconnect hint and `initSessionState` rotates stale entries when freshness fails.

## Solution Assessment

Yes, with a policy caveat: the implementation keeps the lifecycle decision in `initSessionState`, limits the exception to a matching current session, and preserves ordinary stale rollover. The remaining question is whether the public protocol hint is the contract maintainers want.
