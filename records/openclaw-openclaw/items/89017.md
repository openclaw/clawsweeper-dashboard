## Summary

Keep open for maintainer review: the patch appears technically correct with real Gateway proof and focused tests, but it intentionally adds a one-shot session-freshness exception for operator Control UI reconnects, so maintainers must accept that policy before merge.

## What This Changes

This PR adds a private Control UI reconnect marker, Gateway-side stripping/validation, internal reply/session resume options, docs, and regression coverage so one matching reconnect send can preserve the visible session while public `chat.send` and Plugin SDK surfaces stay closed.

## Best Possible Solution

Land this PR after maintainers accept the one-shot operator Control UI freshness exception and CI settles; otherwise choose a proactive stale-session notice or archive-discovery path that keeps rollover strict.

## Reproduction Assessment

Yes, at source level: current main rotates stale daily/idle entries on the next send, and Control UI currently forwards only the visible `sessionId` without a reconnect-specific resume intent. I did not run the physical sleep/Docker/Chrome scenario in this read-only review.

## Solution Assessment

Unclear until maintainer policy acceptance: the implementation is the narrowest code shape if preserving the visible reconnect session is desired, but a proactive stale-session notice would be the safer alternative if strict idle/daily rollover must always win.
