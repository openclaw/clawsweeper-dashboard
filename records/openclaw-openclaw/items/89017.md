## Summary

Keep open; the owner boundary is plausible, but the patch currently lets a matching client session id override configured idle/daily reset policy for normal Control UI sends, so it needs repair or an explicit maintainer decision before merge.

## What This Changes

The PR forwards the `chat.send` session id into reply initialization and reuses a stale session-store entry when that requested id matches, with gateway and session regression tests.

## Best Possible Solution

Preserve configured reset policy as the source of truth while fixing the reconnect data-loss path, or explicitly document and test any maintainer-approved decision that client-asserted WebChat sessions are exempt from idle/daily rollover.

## Reproduction Assessment

Yes from source inspection: current main has Control UI sending the current session id on `chat.send`, while reply initialization ignores it and stale entries roll over through `initSessionState`. I did not run the Mac sleep/Docker/Chrome reproduction live.

## Solution Assessment

No; forwarding the id into the owner lifecycle decision is the right area, but making that id override freshness is too broad because it breaks the documented idle/daily reset contract.
