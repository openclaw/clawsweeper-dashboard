## Label Justifications

- `P1`: This is a real Control UI session-continuity bug, and the current patch can break configured rollover behavior for existing users.
- `merge-risk: 🚨 compatibility`: The PR changes semantics for existing `session.reset` idle/daily configurations when Control UI sends include a session id.

## Summary

Keep open: the fix direction is plausible, but this patch is not merge-ready because the requested session id is not a reconnect-only signal. Current Control UI sends the loaded session id on ordinary visible-session sends, so the new reuse branch can bypass configured idle/daily rollover for existing users.

## What This Changes

The PR passes `chat.send`'s client-supplied session id into reply/session initialization and reuses a matching stale session entry, with gateway/session tests and live gateway proof.

## Best Possible Solution

Keep the fix in the gateway/session lifecycle path, but only preserve a stale session when there is a reconnect/resume signal or approved UI policy, and add tests proving normal stale sends still honor configured rollover.

## Reproduction Assessment

Yes, source-reproducible: current UI stores a session id from `chat.history` and sends it in ordinary `chat.send`, while current session initialization rotates stale entries when freshness fails. I did not run the physical Mac sleep/Docker/Chrome cycle.

## Solution Assessment

No, not as-is: plumbing the id through unconditionally is too broad because the id is not a reconnect-only signal. The narrower maintainable fix is a resume/admission policy that preserves configured rollovers for normal sends.
