## Best Possible Solution

Define one opt-in TUI timestamp display contract that uses persisted message timestamps consistently across live, pending, streaming, system, and replayed rows, then land or replace one reviewed implementation path.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives a high-confidence current-main behavior check that TUI rows do not render per-message timestamps today.

## Solution Assessment

Unclear until maintainers choose the product shape. The narrow maintainable direction is an opt-in TUI-owned display contract using stored message timestamps, not divergent ad hoc prefixes between live and replayed rows.
