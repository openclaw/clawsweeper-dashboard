## What This Changes

Request an optional TUI timestamp display, via config or toggle, for user and assistant chat messages.

## Best Possible Solution

Land a reviewed TUI timestamp implementation, likely by reworking https://github.com/openclaw/openclaw/pull/49069 or replacing it, that defines the accepted UX/config contract and carries timestamps as message data through live, pending, streaming, final, and history replay paths.

## Reproduction Assessment

Not applicable as a feature request. Source inspection on current main shows the TUI renders raw chat text and has no config, CLI flag, slash command, or replay path that exposes user-facing message timestamps.

## Solution Assessment

Unclear. Optional TUI timestamps are a reasonable direction, but maintainers still need to choose the accepted setting or toggle surface and require an implementation that uses stored message timestamps rather than transient rendered prefixes.
