## Summary

Keep open: current main and the latest v2026.6.10 release still do not render per-message timestamps in the TUI, and the remaining work is a product/config contract decision with open fork PR candidates rather than a stale or already-fixed report.

## What This Changes

The issue requests an optional user-visible timestamp display for user and assistant messages in the TUI chat log, via config or a slash-command toggle.

## Best Possible Solution

Define one opt-in TUI-owned timestamp display contract that uses persisted message timestamps consistently across live, pending, streaming, system, and replayed rows, then land or replace one reviewed implementation path.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives a high-confidence current-main behavior check that TUI rows do not render per-message timestamps today.

## Solution Assessment

Unclear until maintainers choose the product shape. The narrow maintainable direction is an opt-in TUI-owned display contract using stored message timestamps across all chat-log render paths, not divergent ad hoc prefixes.
