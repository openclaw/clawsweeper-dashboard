## Summary

Keep open: current main and v2026.6.8 still lack a user-facing TUI message timestamp display or validated `tui.showTimestamps` config, and two open fork PRs still claim this issue as the implementation path. This is a useful TUI ergonomics feature, but the config/display contract needs maintainer product review before another automated fix lane starts.

## What This Changes

Request an optional timestamp display for user and assistant messages in the terminal chat UI, with possible config or slash-command control.

## Best Possible Solution

Define one opt-in TUI-owned timestamp display contract that reuses persisted message timestamps across live and replayed rows, then review one linked PR or a narrowed replacement with schema, help, docs, and focused TUI tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives a high-confidence current-behavior check that TUI rows do not render per-message timestamps today.

## Solution Assessment

Unclear until maintainers choose the product shape. The narrow maintainable solution is an opt-in TUI display contract using stored message timestamps, not ad hoc prefixes that diverge between live and replayed rows.
