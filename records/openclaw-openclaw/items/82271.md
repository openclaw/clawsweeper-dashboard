## Summary

Close as implemented on current main: Discord final replies with explicit reply tags now bypass preview finalization and go through normal Discord reply delivery, with regression coverage for the `[[reply_to_current]]` path. The fix is current-main-only and is not included in `v2026.5.12`.

## What This Changes

Discord channel replies that include `[[reply_to_current]]` should still be delivered instead of being dropped before `sendMessage`.

## Best Possible Solution

Keep the current main implementation and include `bbf50a406eeeba3637e2dc8349657c12620aab07` in the next release so users on released builds receive the Discord reply-directive fix.

## Reproduction Assessment

No; I did not reproduce the dropped-message failure on current main. Current main has source and regression-test evidence showing final Discord payloads with `[[reply_to_current]]` fall back to normal delivery instead of preview editing.

## Solution Assessment

Yes; the current fix is narrow and maintainable because explicit reply directives disable only the preview-finalization shortcut and reuse the existing Discord delivery path that already carries reply metadata.
