## Summary

Keep open: current main still persists Telegram update offsets when an update is accepted while automatic final replies are delivered through a direct in-memory Telegram path, so the restart-during-delivery loss remains a plausible current-main bug. The discussion also adds useful narrowing evidence that cron and explicit message-tool sends work in the same topics, pointing at the automatic final-response handoff rather than Telegram transport or forum routing generally.

## What This Changes

Ensure Telegram long-poll updates are not durably acknowledged before the corresponding automatic forum-topic final reply has delivered or been durably handed off.

## Best Possible Solution

Land one focused fix that either defers persisted update watermarks until automatic reply delivery has completed or durably queues the final reply first, with regression coverage for forum-topic routing and restart recovery.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: persist an accepted update, generate or queue a final reply, interrupt before deliverReplies confirms, then restart with the persisted offset and the tracker skips the original update.

## Solution Assessment

Unclear whether the narrow offset-only PR is the best complete fix. The maintainable fix should prove either that handler-completion offset persistence covers the outbound send boundary or should reuse the durable delivery queue before advancing the watermark.
