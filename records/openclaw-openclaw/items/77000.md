## Summary

Keep open: current main still persists Telegram accepted offsets during `beginUpdate` and tests assert restart replays are skipped, while automatic final reply delivery goes through an in-memory `deliverReplies` path without a durable handoff. The issue also has an open closing PR, so it should stay open until a fix lands or that PR is closed.

## What This Changes

Telegram forum-topic inbound replies can be lost when the gateway restarts after the update watermark is persisted but before the automatic final `sendMessage` completes.

## Best Possible Solution

Land a focused fix that makes automatic Telegram final replies durable across gateway restarts while preserving forum thread routing, idempotency, and preview/final streaming behavior.

## Reproduction Assessment

Yes, as a source-level current-main path: `beginUpdate` persists the accepted update ID, a restarted tracker initialized from that offset skips the same update, and final reply delivery is not durably queued before the Telegram send. I did not run a live Telegram Bot API reproduction.

## Solution Assessment

Unclear whether the issue's Option A alone is the best complete fix: deferring offset persistence until handler completion addresses the reported window, but the existing write-ahead outbound queue may be the safer primitive for crash-after-transcript recovery. The open implementation PRs should be reviewed against duplicate-send and topic-routing behavior before merge.
