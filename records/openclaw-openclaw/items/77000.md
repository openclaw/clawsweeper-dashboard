## Summary

Close as implemented: current main and v2026.5.12 now spool Telegram updates locally before offset advancement and route final replies through durable outbound delivery, covering the restart window described in the report.

## What This Changes

Telegram forum-topic final replies could be lost if the gateway restarted after processing an update but before sendMessage confirmed.

## Best Possible Solution

Keep the shipped ingress-spool plus durable final-delivery implementation; remaining unrelated Telegram delivery symptoms should stay in narrower reports.

## Reproduction Assessment

No current-main failing path was established. The original production/source-level race is credible, but current main now changes both relevant handoffs: inbound updates survive locally after offset persistence and final replies enter durable delivery before platform send.

## Solution Assessment

Yes. Durable local ingress spooling plus durable final outbound queueing is a more complete fix than only delaying Telegram offsets, because it protects both the update replay source and the post-generation send handoff.
