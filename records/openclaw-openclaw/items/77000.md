## Summary

Keep open: current main still persists Telegram update watermarks at update acceptance while automatic final replies use a non-durable in-memory send path, and an open closing PR (#77075) should be reviewed before this issue is closed.

## What This Changes

The issue reports that Telegram long-poll update IDs are persisted before automatic final replies are confirmed, causing restart-during-delivery to lose forum-topic replies.

## Best Possible Solution

Keep this issue open until #77075 or a successor lands a restart-safe final-delivery fix with regression coverage for forum-topic routing and duplicate/idempotency behavior.

## Reproduction Assessment

Yes, as a source-level path: accept an update, persist its offset during beginUpdate, record or enqueue a final reply, then restart before deliverReplies confirms; the restarted tracker skips that update from the persisted watermark. I did not run a live Telegram Bot API restart test.

## Solution Assessment

Unclear for the issue's Option A alone: deferring offset persistence until handler completion narrows the race, but durable final delivery through the existing write-ahead queue may be the more maintainable fix if confirmed sends cannot be coupled exactly to update acknowledgement.
