## Summary

Keep open: current main still has a source-visible at-most-once Telegram window. The recent forum-topic delivery fallback helps when the process survives to turn cleanup, but it does not cover a process restart after the update offset is persisted and before Telegram final delivery is confirmed.

## What This Changes

Reports a Telegram long-poll/forum-topic data-loss race where update offsets are persisted before outbound final replies are confirmed, so a restart can permanently lose the reply.

## Best Possible Solution

Keep this open and make Telegram auto-replies either replayable until confirmed or durably recoverable before the accepted update offset becomes the only persisted state.

## Reproduction Assessment

Yes, at source level: current code persists accepted update ids before handler completion, while the Telegram final auto-reply path sends directly through `deliverReplies` without the shared durable queue. I did not run a live Telegram restart reproduction.

## Solution Assessment

No, current main is not the full solution for this issue: #76554-style fallback handles survived turn cleanup, but not process death between offset persistence and final send confirmation. The narrow maintainable fix belongs in the Telegram offset/final-delivery handoff rather than in docs or operator restart guidance.

## Repair Work Prompt

Fix the Telegram restart-during-final-delivery race reported in openclaw/openclaw#77000. Current source persists accepted `update_id`s before handler completion (`extensions/telegram/src/bot-update-tracker.ts`, `extensions/telegram/src/monitor.ts`), while forum-topic auto-replies go through `extensions/telegram/src/bot-message-dispatch.ts` -> `extensions/telegram/src/bot/delivery.replies.ts` instead of the shared write-ahead outbound queue. Implement a narrow reliability fix so an accepted Telegram update is not permanently lost if the gateway exits after assistant final text is produced but before the final Telegram send is confirmed. Prefer reusing existing durable delivery/recovery primitives where feasible; otherwise defer the persisted Telegram offset until the handler has reached a confirmed/suppressed final-delivery state. Do not add a config option, do not change unrelated channels, and preserve existing forum-topic thread routing, streaming preview behavior, and `NO_REPLY` suppression semantics. Add regression coverage for a blocked/failing final delivery where the update offset must not be the only durable state, plus coverage that successful final delivery still advances/dedupes normally. Add a user-facing changelog fix entry.
