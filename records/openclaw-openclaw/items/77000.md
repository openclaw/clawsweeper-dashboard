## Summary

Keep open: current main still persists Telegram update watermarks when an update is accepted, while automatic Telegram final replies are delivered through a direct in-memory path, so the restart-during-final-send loss remains a plausible current-main bug.

## What This Changes

Ensure Telegram forum-topic inbound turns cannot be acknowledged past restart before their automatic final reply is durably handed off or confirmed delivered.

## Best Possible Solution

Make Telegram automatic final delivery durable, preferably by reusing the existing write-ahead outbound delivery queue with forum-topic routing and idempotency, then adjust offset timing only if a focused regression proves it is still required.

## Reproduction Assessment

Yes, as a source-level reproduction: accept and persist a Telegram update, complete/write a final reply, interrupt before deliverReplies confirms, then restart with the persisted accepted offset and the tracker skips the original update. I did not run a live Telegram Bot API restart test.

## Solution Assessment

Unclear for the issue's Option A by itself, because offset deferral has duplicate and Bot API acknowledgement semantics to prove. The narrowest maintainable fix appears to be durable automatic final delivery through the existing outbound queue, with targeted tests for the restart window.

Fix the Telegram automatic final-reply restart-loss bug tracked by https://github.com/openclaw/openclaw/issues/77000. Current main persists accepted Telegram update offsets during beginUpdate, but automatic final replies are delivered through the direct in-memory deliverReplies path; a gateway restart after final transcript write but before sendMessage confirmation can permanently lose the visible forum-topic reply. Keep the fix scoped to Telegram automatic final delivery and shared outbound queue reuse: prefer durably handing final payloads to the existing write-ahead outbound delivery queue while preserving account, chat id, forum thread id, reply target, session context, hooks, media handling, and preview/final streaming semantics.
