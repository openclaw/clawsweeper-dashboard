## Summary

Keep open: current main still persists Telegram update watermarks at handler acceptance while automatic Telegram final replies are delivered through a direct in-memory path, so the restart-during-send loss described here remains a plausible current-main bug rather than a stale or duplicate report.

## What This Changes

Request to make Telegram long-poll/forum-topic replies durable across restarts so an inbound update is not permanently consumed before the outbound final reply is confirmed.

## Best Possible Solution

Make Telegram automatic final delivery durable, preferably by reusing the existing write-ahead outbound delivery queue for final payloads and preserving forum topic routing/idempotency on recovery.

## Reproduction Assessment

Yes, as a source-level reproduction: persist an accepted update, enqueue/record a final reply, interrupt before deliverReplies confirms, then restart with that persisted offset and the tracker skips the original update. I did not run a live Telegram Bot API reproduction.

## Solution Assessment

Unclear for the issue’s suggested Option A alone, because local offset deferral may not cover all grammY/getUpdates acknowledgement timing. The narrowest maintainable repair appears to be durable outbound final delivery with existing queue infrastructure, with offset semantics adjusted only if a focused regression proves it is also required.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77000. Telegram long-poll/forum-topic automatic final replies can be lost if the gateway stops after the inbound session/final transcript is recorded but before Telegram sendMessage confirms, because update offsets are persisted on acceptance and the automatic Telegram final delivery path is not durable. Implement a narrow durability fix without adding a config option: prefer reusing the existing write-ahead outbound delivery queue/recovery for Telegram automatic final replies, or otherwise gate offset persistence on confirmed durable delivery if a targeted regression proves that is safer. Preserve existing Telegram preview/final-edit behavior, quote/thread/topic routing, NO_REPLY handling, and message-tool delivery. Add regression coverage that simulates accepted update persistence, final reply dispatch, interruption before send confirmation, restart/recovery, and verifies the reply is retried in the same chat/thread without silently consuming the update. Do not change unrelated externalized plugin install flow or the #76388/#76735 polling-startup fix. Add a CHANGELOG.md entry because this is user-facing delivery reliability.
