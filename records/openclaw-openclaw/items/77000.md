## Summary

Keep open: current main still has the reported at-most-once window between persisted Telegram update acceptance and non-durable automatic final-reply delivery, so this is a plausible current bug rather than stale cleanup or a duplicate close.

## What This Changes

The issue asks OpenClaw to prevent Telegram forum-topic final replies from being lost when the gateway restarts after processing an inbound update but before outbound `sendMessage` confirms.

## Best Possible Solution

Make Telegram automatic final delivery durable, preferably by reusing the existing write-ahead outbound delivery queue for final payloads while preserving forum topic routing and idempotency.

## Reproduction Assessment

Yes, as a source-level reproduction: persist an accepted update, interrupt before `deliverReplies` confirms, then restart with that persisted offset and the tracker skips the original update. I did not run a live Telegram Bot API reproduction.

## Solution Assessment

Unclear whether deferring Telegram offsets alone is the best fix, because local offset semantics may not cover all grammY/getUpdates timing. The narrowest maintainable repair appears to be durable outbound final delivery using the existing queue, with offset changes only if regression coverage proves they are also required.

Fix https://github.com/openclaw/openclaw/issues/77000 as a narrow Telegram delivery durability bug.
