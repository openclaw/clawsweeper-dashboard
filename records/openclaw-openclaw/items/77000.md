## Summary

Keep open: current main still persists Telegram update offsets at update acceptance while automatic final replies go through a non-durable in-memory delivery path, and an open closing PR already targets this issue.

## What This Changes

The issue requests Telegram update acknowledgement/final-reply delivery ordering that prevents restart-time loss of forum-topic replies after the transcript has recorded an assistant response.

## Best Possible Solution

Land or supersede the linked fix with a regression that proves Telegram final replies are either durably queued or the update offset is not advanced until the automatic reply path has safely completed for forum topics.

## Reproduction Assessment

Yes, as a source-level reproduction: accept an update, persist its offset, interrupt before automatic final delivery confirms, then restart with that offset and the tracker skips the original update. I did not run a live Telegram Bot API reproduction.

## Solution Assessment

Unclear for the currently linked PR alone: moving offset persistence later is a plausible narrow fix, but it must be checked against the issue’s stronger sendMessage-confirmed or durable-queue requirement. The most maintainable solution should include focused regression coverage for the restart window and forum-topic routing.
