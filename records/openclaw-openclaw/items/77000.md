## Summary

Current main now covers the reported loss window: Telegram final replies use the channel-message durable send path, and the Telegram polling restart watermark is persisted only after successful handler dispatch using the safe completed update watermark. The fix is present on current main but I could not prove it is included in the latest shipped release.

## What This Changes

The issue reports that Telegram forum-topic updates can be acknowledged before the automatic final reply is durably delivered, causing replies to be lost across gateway restarts.

## Best Possible Solution

Keep the current durable final delivery and deferred safe-watermark implementation; the narrower open PR can be closed or rebased only if it adds unique regression coverage.

## Reproduction Assessment

Yes. The old failure path is source-reproducible from the issue evidence and earlier tracker behavior, while current main shows the watermark and final-delivery paths now run after successful dispatch through durable send handling.

## Solution Assessment

Yes. For the reported restart-during-final-send window, current main’s combination of safe completed watermark persistence and queued final delivery is the narrow maintainable solution; deeper platform-level polling source durability would be separate future hardening if new evidence requires it.
