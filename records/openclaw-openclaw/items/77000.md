## Summary

Keep open: current main still persists Telegram accepted-update watermarks before the automatic final reply has a durable delivery handoff, and the comments narrow this to the embedded-run final-delivery path rather than Telegram transport or forum-topic permissions.

## What This Changes

The issue asks OpenClaw to prevent Telegram forum-topic replies from being lost when a gateway restart happens after an inbound update is accepted but before the automatic final `sendMessage` succeeds.

## Best Possible Solution

Make Telegram automatic final-response delivery durable across gateway restarts by reusing the write-ahead outbound delivery queue or an equivalent per-turn durable handoff while preserving forum-topic routing metadata.

## Reproduction Assessment

Yes, as a source-level reproduction: accept and persist a Telegram update, let the run produce a final automatic reply, interrupt before `deliverReplies` confirms, then restart and observe the tracker skip that update because the accepted offset is already persisted. I did not run a live Telegram Bot API reproduction.

## Solution Assessment

Unclear for the issue's offset-deferral option alone, because duplicate-send behavior and grammY/getUpdates semantics need care. The narrowest maintainable fix appears to be durable outbound final delivery with idempotency, with offset timing adjusted only if a focused regression proves it is also required.

## Repair Work Prompt

Fix the Telegram automatic final-reply durability bug described in https://github.com/openclaw/openclaw/issues/77000. The observable failure is that a Telegram update can be accepted/persisted before the automatic final `sendMessage` has a durable handoff; if the gateway restarts after transcript finalization but before delivery confirmation, the update is skipped on restart and the user never receives the reply. Keep the fix in the Telegram plugin and shared outbound delivery surface as needed; do not add a broad new config option or change unrelated channel behavior. Prefer reusing the existing write-ahead outbound delivery queue for automatic final Telegram replies with per-turn/session idempotency and preserved forum-topic `message_thread_id` routing. Add regression coverage showing accepted offset persistence plus restart-before-send cannot permanently drop a completed automatic final reply. Update CHANGELOG.md because this is user-facing. Stop and ask for maintainer review if the implementation would require changing public plugin APIs or making a product-policy decision about duplicate delivery semantics.
