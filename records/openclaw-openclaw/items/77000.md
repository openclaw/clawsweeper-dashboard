## Summary

Keep open: current main still persists Telegram update offsets at handler acceptance while automatic Telegram final replies use a direct in-memory send path, and open PR #77075 is a closing implementation candidate that should be reviewed before this issue is resolved.

## What This Changes

Report asks OpenClaw to prevent Telegram forum-topic final replies from being lost when a gateway restart happens after update offset persistence but before outbound Telegram delivery confirms.

## Best Possible Solution

Land a regression-tested Telegram fix that makes completed forum-topic final replies survive gateway restarts, with delivery-aware offset persistence or durable queued final delivery plus idempotency.

## Reproduction Assessment

Yes, as a source-level reproduction: accept an update, persist the offset, interrupt before automatic deliverReplies confirms, then restart with that persisted offset and the tracker skips the original update. I did not run the live Telegram Bot API crash window.

## Solution Assessment

Unclear whether the open PR's handler-completion offset deferral is the complete fix, because the report is specifically about confirmation of outbound sendMessage. A delivery-aware offset change or reuse of the durable outbound queue is the safer end state if tests show handler completion can still precede visible delivery.
