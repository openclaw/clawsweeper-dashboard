## Summary

Keep open. Current main proves invalid-model 400s can be classified for fallback, but it does not prove the reported elapsed-time regression is fixed; the fallback loop still waits for the active candidate attempt to return before it can advance.

## What This Changes

The issue requests fail-fast model fallback for deterministic invalid-model or unsupported-parameter errors that reportedly stall about 30 seconds per bad candidate.

## Best Possible Solution

Add an elapsed-time regression for deterministic invalid-model and unsupported-parameter failures, then make that path advance immediately while preserving retry, cooldown, and idle-timeout behavior for retryable or genuinely slow providers.

## Reproduction Assessment

No. The report gives clear steps and a version comparison, and source inspection shows plausible wait points, but this read-only review did not establish a current-main failing run or an elapsed-time regression test.

## Solution Assessment

Unclear. The requested fail-fast behavior matches the documented fallback contract, but the implementation should first isolate whether the delay comes from SDK retry, transport response handling, stream error conversion, or the embedded idle-timeout wrapper.
