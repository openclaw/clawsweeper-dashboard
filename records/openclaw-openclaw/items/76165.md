## Summary

Keep open. Current main classifies immediate invalid-model 400s as fallback-worthy, but I found no source/test/release proof that the elapsed-time regression is fixed; the fallback runner still waits for each candidate attempt to return before it can continue.

## What This Changes

Make model fallback fail over immediately on deterministic invalid-model or unsupported-parameter errors instead of waiting for a timeout per fallback candidate.

## Best Possible Solution

Add an elapsed-time regression for deterministic invalid-model and unsupported-parameter failures, then fix the narrow transport or embedded-run layer that delays those errors while preserving normal retry, cooldown, and idle-timeout behavior for transient failures.

## Reproduction Assessment

No high-confidence current-main reproduction path was established. The reporter supplied concrete versioned steps and source shows the fallback runner waits sequentially, but existing tests only cover immediate invalid-model rejection, not the reported timeout-length delay.

## Solution Assessment

Unclear until the delay source is isolated. The requested fail-fast behavior is the right direction, but the repair should target the layer that turns deterministic 400s into delayed candidate attempts rather than broadening shared retry or cooldown policy.
