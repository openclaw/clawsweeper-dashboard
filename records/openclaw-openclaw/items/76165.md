## Summary

Keep open. Current main still only classifies invalid-model HTTP 400s after the provider error surfaces; I did not find source, docs, or elapsed-time test proof that deterministic invalid candidates bypass the reported per-candidate wait.

## What This Changes

The issue asks OpenClaw to fail over immediately from deterministic non-retryable model errors such as invalid-model or unsupported-parameter HTTP 400 responses instead of waiting about 30 seconds per fallback candidate.

## Best Possible Solution

Add narrow fail-fast handling for deterministic invalid model or unsupported-parameter failures while preserving normal timeouts for genuinely slow providers and covering elapsed-time behavior.

## Reproduction Assessment

No. The issue has plausible steps and current immediate-rejection tests, but there is no high-confidence elapsed-time reproduction against current main or live provider proof showing 30 seconds per invalid candidate.

## Solution Assessment

Unclear. The fail-fast outcome is correct, but the implementation should first identify whether the delay comes from SDK retry behavior, guarded fetch timeout, or embedded stream idle timeout before changing shared retry policy.
