## Summary

Keep open. Current main still has only classification and adjacent retry-after protections; I found no source, test, release, or related-issue evidence that the reported elapsed-time regression for deterministic invalid candidates is fixed.

## What This Changes

The issue reports that deterministic invalid fallback candidates such as HTTP 400 invalid-model or unsupported-parameter responses wait about 30 seconds each before trying the next configured model.

## Best Possible Solution

Add an elapsed-time regression for deterministic invalid-model and unsupported-parameter failures, then make that path advance to the next fallback immediately while preserving normal retry, cooldown, and idle-timeout behavior for transient or silent providers.

## Reproduction Assessment

No high-confidence live reproduction has been established. Source inspection shows the fallback loop still waits on each candidate attempt, while existing tests only prove invalid-model classification and not the reported elapsed-time fail-fast behavior.

## Solution Assessment

Unclear until the delay source is isolated. The requested fail-fast behavior is the right target, but the fix should land in the SDK/request/transport/embedded-run layer that holds deterministic 400s rather than broadening shared retry or idle-timeout policy.
