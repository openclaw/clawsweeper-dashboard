## Summary

Keep open. Current main recognizes invalid-model 400s as fallback-worthy, but I did not find a shipped/current fix or regression coverage proving deterministic 400-style failures bypass the reported per-candidate wait.

## What This Changes

The issue reports that deterministic invalid model or unsupported-parameter failures in a fallback chain wait about 30 seconds per candidate before trying the next model.

## Best Possible Solution

Add an elapsed-time regression for deterministic invalid-model and unsupported-parameter failures, then make only that path fail fast while preserving normal retry and timeout behavior for slow or retryable providers.

## Reproduction Assessment

No, not as a high-confidence live reproduction. Source inspection shows the fallback runner waits for each provider attempt before advancing, and current tests only cover immediate mocked invalid-model rejection rather than the reported elapsed-time delay.

## Solution Assessment

Unclear because there is no proposed patch and the delay source is not isolated. The requested fail-fast behavior is the right direction, but the safer fix is to target the SDK/request/embedded-run layer that holds deterministic 400s rather than weakening shared timeout policy.
