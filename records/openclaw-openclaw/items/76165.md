## Summary

Current main and shipped release notes already cover the reported invalid-candidate fallback delay: the fallback loop now accepts result classification from the embedded runner, classified terminal model failures advance to the configured fallback, and v2026.4.27 release notes explicitly shipped the fix for the original report.

## What This Changes

Fail over immediately through configured model fallbacks when invalid or non-retryable model candidates return deterministic 400-style failures instead of waiting about 30 seconds per candidate.

## Best Possible Solution

Keep the shipped fallback result-classification path; handle any future latency as a new current-main regression with fresh provider logs and model chain details.

## Reproduction Assessment

No current-main failing reproduction remains. The original affected-release path is concrete, and related live verification plus current source, tests, and v2026.4.27 release notes show the chain now advances instead of waiting through the old 30-second terminal failure.

## Solution Assessment

Yes. The current solution is maintainable because fallback classification is centralized at the model fallback boundary and preserves normal SDK retry behavior while treating unusable terminal model outputs as fallback failures.
