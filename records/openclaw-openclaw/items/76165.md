## Summary

Keep open. Current main handles immediate invalid-model 400s as fallback-worthy, but the reported elapsed-time regression is still not proved fixed or covered by a regression test; the runner still advances only after the active candidate attempt returns.

## What This Changes

Non-retryable model/provider errors such as invalid model IDs or unsupported parameters should move to the next configured fallback model without waiting for a retry or idle timeout.

## Best Possible Solution

Add an elapsed-time regression for deterministic invalid-model and unsupported-parameter failures, then make that path fail fast while preserving existing retry, cooldown, and idle-timeout behavior for retryable or slow providers.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows a plausible current-main path where the fallback runner cannot advance until the candidate attempt returns, while existing tests only cover immediate invalid-model rejection.

## Solution Assessment

Unclear until the delay source is isolated. The requested fail-fast behavior is the right direction, but the fix should target deterministic provider/SDK/embedded-run failures rather than broadening shared retry or timeout policy.
