## Summary

Keep open. Current main classifies invalid-model 400s as fallback-worthy once the error is returned, but the reported elapsed-time regression is not covered and the fallback runner still waits for each candidate attempt before it can advance.

## What This Changes

The issue asks OpenClaw to fail over immediately from deterministic invalid model or unsupported-parameter candidates instead of waiting about 30 seconds per failed candidate.

## Best Possible Solution

Add an elapsed-time regression for deterministic invalid-model and unsupported-parameter failures, then make only that path fail fast while preserving normal retry and timeout behavior for slow or retryable providers.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the fallback loop waits on each candidate and current tests only cover immediate invalid-model rejections, so the reported elapsed-time failure remains unprotected.

## Solution Assessment

Unclear until the delay source is isolated. The requested fail-fast behavior is the right direction, but the fix should target the SDK/request/embedded-run layer that holds deterministic 400s for 30 seconds rather than weakening shared timeout policy.
