## Summary

Close as implemented. Current main and v2026.4.27 include fallback classification for the same 30-second terminal invalid/incompatible model path, and the related #73115 maintainer verification confirmed the invalid OpenRouter cascade completes quickly on main.

## What This Changes

Report that v2026.4.24/.25 model fallback waits about 30 seconds per invalid HTTP 400 candidate instead of immediately advancing to the next configured model.

## Best Possible Solution

Keep the shipped v2026.4.27+ fallback classifier behavior and preserve the existing regression coverage for direct invalid-model errors and classified terminal results.

## Reproduction Assessment

Yes. The reproduction path is the invalid OpenRouter fallback chain from the related #73115 report; maintainer verification on main found it completed in about 3.4s instead of reproducing the 30-second-per-candidate delay.

## Solution Assessment

Yes. The shipped fix targets the fallback result-classification boundary, which is the narrow place where terminal invalid/incompatible model results can be rejected without changing retry policy for genuinely retryable provider failures.
