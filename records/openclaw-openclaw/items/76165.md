## Summary

Keep open. Current main classifies invalid-model 400s after the error surfaces, but the fallback loop still waits for each candidate attempt and the shared provider transport does not special-case deterministic 400 responses for fail-fast behavior.

## What This Changes

The issue asks OpenClaw to fail over immediately on deterministic invalid-model or unsupported-parameter 400 responses in a model fallback chain instead of waiting roughly 30 seconds per candidate.

## Best Possible Solution

Add a narrow fail-fast path and elapsed-time regression coverage for classified deterministic provider errors while preserving normal retry and timeout behavior for slow or transient providers.

## Reproduction Assessment

No high-confidence current-main reproduction yet. The issue gives clear versioned steps, and static code review shows the delay is plausible, but existing tests use immediate rejections and I found no live or elapsed-time proof that current main fails over within seconds.

## Solution Assessment

Unclear. The fail-fast direction is sound, but the best fix should first identify whether the wait comes from SDK retry handling, guarded fetch timeout, or embedded-run idle timeout before changing shared transport policy.

## Repair Work Prompt

Fix the model fallback latency reported in openclaw/openclaw#76165. Current main advances fallback after an invalid-model 400 only once the error surfaces, but deterministic invalid candidates can still wait roughly 30s before the next fallback. Investigate whether the delay comes from provider SDK retry behavior, `buildGuardedModelFetch`, embedded-run idle timeout handling, or model request timeout propagation. Add a narrow fail-fast path for deterministic non-retryable provider errors such as invalid model ID and unsupported parameter, without globally shortening slow-provider timeouts or disabling legitimate retries for 408/409/429/5xx. Add focused elapsed-time/unit coverage so an invalid first candidate reaches the next candidate promptly, and update user-facing changelog under the active version if behavior changes. Do not touch unrelated fallback policy, provider catalogs, or plugin loading.
