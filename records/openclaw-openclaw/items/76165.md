## Summary

Keep open. Current main classifies invalid-model 400s as fallback-worthy, but I found no proof that the reported elapsed-time regression is fixed; the fallback loop still awaits each candidate attempt, and embedded runs can still hold a silent model stream until the idle watchdog fires.

## What This Changes

The issue requests fail-fast model fallback for deterministic non-retryable invalid model or unsupported-parameter errors instead of waiting about 30 seconds per bad candidate.

## Best Possible Solution

Add an elapsed-time regression for delayed deterministic invalid-model and unsupported-parameter failures, then make that exact path fail fast without broadening shared retry or timeout policy.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the fallback runner waits on each candidate and current tests only cover immediate invalid-model rejection, so the elapsed-time failure remains plausible but unprotected.

## Solution Assessment

Unclear until the delay source is isolated. The requested fail-fast behavior is the right direction, but the fix should target the SDK/request/stream path that holds deterministic 400s rather than changing broad fallback or idle-timeout policy.
