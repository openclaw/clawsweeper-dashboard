## Summary

Keep open. Current main has the right fallback classification after an invalid-model 400 is returned, but the code and tests still do not prove that deterministic invalid candidates bypass the reported per-candidate wait before the next model is tried.

## What This Changes

The issue asks OpenClaw to fail over immediately from non-retryable invalid-model or unsupported-parameter candidates in a configured model fallback chain.

## Best Possible Solution

Keep normal timeouts for slow or hanging providers, but add a narrow fail-fast path and elapsed-time regression coverage for deterministic invalid-model and unsupported-parameter failures before fallback advances.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the fallback loop waits on each candidate and current tests only cover immediate invalid-model rejections, leaving the reported elapsed-time failure unprotected.

## Solution Assessment

Unclear until the delay source is isolated. The requested fail-fast behavior is the right direction, but the fix should target the SDK/request/embedded-run layer that holds deterministic 400s for 30 seconds rather than weakening shared timeout policy.
