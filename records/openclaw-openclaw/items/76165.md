## Summary

Keep open. Current main still lacks source, test, or release proof that the reported elapsed-time regression is fixed: immediate invalid-model 400s are classified for fallback, but the fallback loop still waits for each candidate attempt to return and the delayed idle-timeout path remains unprotected.

## What This Changes

Non-retryable invalid model or unsupported-parameter fallback candidates should advance to the next configured model immediately instead of waiting about 30 seconds per failed candidate.

## Best Possible Solution

Add elapsed-time regression coverage for deterministic 400 and unsupported-parameter failures, then fix the narrow provider transport or embedded-run path that converts those failures into idle waits while preserving retry, cooldown, and timeout behavior for retryable providers.

## Reproduction Assessment

No high-confidence current-main reproduction is established. Source inspection shows why a candidate can still wait until the stream/idle path returns, but the exact delayed invalid-400 path needs a focused regression or live/fake-provider proof.

## Solution Assessment

Unclear until the delay source is isolated. The requested fail-fast behavior is the right direction, but the repair should target deterministic provider errors in the transport/embedded-run path rather than broadly changing retry or idle-timeout policy.
