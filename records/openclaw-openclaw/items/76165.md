## Summary

Keep open. Current main still classifies invalid-model 400s for fallback only after the candidate attempt returns; the reported 30-second-per-candidate latency regression is not covered by tests or tied to a shipped fix.

## What This Changes

The issue asks model fallback to fail over immediately on deterministic invalid-model or unsupported-parameter 400s instead of waiting about 30 seconds per failed candidate.

## Best Possible Solution

Add an elapsed-time regression for deterministic provider 400/404/model-not-found and unsupported-parameter failures, then make only that path surface immediately while preserving retry and timeout behavior for slow or retryable providers.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows that fallback cannot advance until the current candidate attempt returns, but current tests only cover immediate mocked invalid-model failures, not the reported 30-second delay.

## Solution Assessment

Yes directionally. The narrow maintainable solution is to fix the SDK/request/embedded-run layer that holds deterministic provider errors, not to weaken the whole fallback loop or global timeout policy.
