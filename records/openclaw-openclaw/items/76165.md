## Summary

Keep open. Current main handles immediately rejected invalid-model 400s as fallback-worthy, but there is still no source, test, or release proof that the reported 30s-per-candidate latency regression is fixed.

## What This Changes

The issue reports that deterministic invalid fallback candidates wait about 30 seconds each before advancing to the next model.

## Best Possible Solution

Add an elapsed-time regression for deterministic invalid-model and unsupported-parameter failures, then make that path fail fast while preserving normal retry, cooldown, and idle-timeout behavior for retryable or slow providers.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows fallback cannot advance until the candidate attempt returns, and the default embedded idle watchdog provides a plausible 30s hold, but existing tests only cover immediate invalid-model rejection.

## Solution Assessment

Unclear until the delay source is isolated. The requested fail-fast behavior is the right direction, but the repair should target the SDK/request/stream layer that holds deterministic 400s rather than broadening shared fallback policy.
