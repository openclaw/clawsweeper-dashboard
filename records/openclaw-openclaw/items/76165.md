## Summary

Keep open. Current main still advances the fallback chain only after the current candidate attempt returns; invalid-model 400s are classified correctly, but there is no elapsed-time regression coverage or release/source proof that deterministic invalid candidates now bypass the reported stall.

## What This Changes

Fail model fallback over immediately for deterministic non-retryable invalid-model or unsupported-parameter candidates instead of waiting about 30 seconds per failed candidate.

## Best Possible Solution

Add a focused elapsed-time regression for deterministic invalid-model and unsupported-parameter failures, then fix the narrow transport/SDK/embedded-run path that holds those failures while preserving retry, cooldown, and idle-timeout behavior for genuinely retryable or silent providers.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the fallback runner waits on each candidate and current tests only cover immediate invalid-model rejection, so the elapsed-time regression remains unprotected.

## Solution Assessment

Unclear until the delay source is isolated. The requested fail-fast behavior is the right direction, but the repair should target the SDK/request/embedded-run layer that holds deterministic 400s rather than broadening shared retry or idle-timeout policy.
