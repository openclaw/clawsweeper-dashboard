## Summary

Keep open. Current main classifies invalid-model 400s correctly after the error reaches the fallback loop, but it still lacks a high-confidence elapsed-time reproduction proving deterministic invalid candidates bypass the reported 30s wait before fallback advances.

## What This Changes

The issue reports that deterministic invalid model or unsupported-parameter fallback candidates wait about 30 seconds each before trying the next configured model.

## Best Possible Solution

Keep normal timeouts for genuinely slow or hanging providers, but add focused elapsed-time regression coverage and a fail-fast path for deterministic invalid-model, unsupported-parameter, and not-found provider errors at the request/SDK boundary that delays them.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the fallback loop waits on each candidate and current tests only cover immediate invalid-model rejections, so the reported elapsed-time failure is plausible but not yet proven against current main.

## Solution Assessment

Unclear until the delay source is isolated. The requested fail-fast behavior is the right direction, but the fix should target the provider SDK/request/embedded-run layer that holds deterministic 400s rather than weakening shared timeouts for slow providers.
