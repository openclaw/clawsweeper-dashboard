## Summary

Keep open. Current main still proves only immediate invalid-model classification; there is no source, test, changelog, or release proof that the reported 30s-per-invalid-candidate fallback delay is fixed.

## What This Changes

The issue asks OpenClaw to fail over immediately on deterministic non-retryable model-candidate failures instead of waiting about 30 seconds per invalid fallback entry.

## Best Possible Solution

Add an elapsed-time regression for deterministic invalid-model and unsupported-parameter failures, then make that path fail fast while preserving normal retry, cooldown, and timeout behavior for retryable or slow providers.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the fallback runner waits on the current candidate and current tests only cover immediate invalid-model rejection, so the elapsed-time regression remains unprotected.

## Solution Assessment

Unclear until the delay source is isolated. The intended fail-fast behavior is right, but the repair should target the layer that turns deterministic 400s into a 30s wait rather than broadening shared retry or idle-timeout policy.
