## Summary

Keep open. Current main classifies immediate invalid-model 400s as fallback-worthy, but there is no source, test, or release proof that the reported elapsed-time regression is fixed; the fallback loop still cannot advance until the current candidate attempt returns.

## What This Changes

The issue asks for deterministic invalid-model or unsupported-parameter fallback candidates to fail over immediately instead of waiting about 30 seconds per candidate.

## Best Possible Solution

Add an elapsed-time regression for deterministic invalid-model and unsupported-parameter failures, then make that path fail fast while preserving normal retry, cooldown, and timeout behavior for retryable or slow providers.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the fallback runner waits on each candidate and current tests only cover immediate invalid-model rejection, so the elapsed-time regression remains unprotected.

## Solution Assessment

Unclear until the delay source is isolated. The requested fail-fast behavior is the right direction, but the repair should target the SDK/request/embedded-run layer that holds deterministic 400s rather than broadening shared retry or idle-timeout policy.
