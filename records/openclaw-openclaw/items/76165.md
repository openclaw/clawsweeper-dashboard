## Summary

Keep open. Current main classifies invalid-model 400s as fallback-worthy once the error is returned, but I did not find proof that deterministic invalid candidates bypass the reported per-candidate 30s wait; the shared transport retry cap explicitly does not special-case 400 responses, and existing tests use immediate rejections rather than elapsed-time behavior.

## What This Changes

The issue asks for deterministic non-retryable model errors such as invalid model IDs or unsupported parameters to advance to the next configured fallback immediately instead of waiting about 30 seconds per candidate.

## Best Possible Solution

Preserve normal timeouts for genuinely slow providers, but short-circuit deterministic invalid-model or unsupported-parameter failures into the existing fallback path without waiting for the provider request timeout.

## Reproduction Assessment

No. Static tests prove invalid-model 400 classification after an error is returned, but there is no elapsed-time regression test or live provider proof showing current main fails over within seconds.

## Solution Assessment

Unclear. The requested fail-fast direction is sound, but the implementation should first identify whether the delay comes from SDK retry behavior, guarded fetch timeout, or embedded-run timeout before changing shared timeout policy.
