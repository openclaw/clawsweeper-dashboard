## Summary

Keep open. Current main classifies invalid-model HTTP 400s as fallback-worthy once the error surfaces, but it still lacks proof that deterministic invalid candidates bypass the reported per-candidate wait before advancing.

## What This Changes

The issue requests fail-fast model fallback for deterministic invalid-model or unsupported-parameter errors so bad candidates do not consume about 30 seconds each before the next fallback model is tried.

## Best Possible Solution

Add elapsed-time regression coverage and a narrow fail-fast path for deterministic invalid-model and unsupported-parameter failures at the request/stream/embedded-run boundary, while preserving normal timeouts for slow or hanging providers.

## Reproduction Assessment

No high-confidence current-main reproduction was established. Source inspection confirms the fallback loop waits for each attempt and the tests only cover immediate invalid-model rejections, but the reported 30-second latency was not live-reproduced.

## Solution Assessment

Unclear until the delay source is isolated. The fail-fast direction is appropriate, but the fix should target the SDK/request/stream/embedded-run layer that holds deterministic 400s rather than weakening shared timeout policy.
