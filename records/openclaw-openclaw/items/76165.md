## Summary

Keep open. Current main recognizes invalid-model 400s as fallback-worthy after an error is returned, but it still lacks proof or coverage that deterministic failures advance without the reported per-candidate wait.

## What This Changes

The issue asks for deterministic non-retryable model errors in fallback chains to fail over immediately instead of waiting about 30 seconds per invalid candidate.

## Best Possible Solution

Add an elapsed-time regression for deterministic invalid-model and unsupported-parameter failures, then make only that path fail fast while preserving normal retries and slow-provider timeout behavior.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows a delayed candidate attempt would block the sequential fallback loop, but current tests only cover immediate invalid-model rejections.

## Solution Assessment

Yes, the requested fail-fast behavior for deterministic 400/404-style failures is the right direction. The fix should target the transport or runner path that holds those errors, not broadly reduce shared model request timeouts.
