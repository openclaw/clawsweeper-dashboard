## Summary

Keep open. The report is recent and actionable: current main classifies immediate invalid-model 400/404 failures for fallback, but I did not find an end-to-end timing guard proving deterministic provider rejections bypass the stream/SDK idle path before consuming a full per-candidate timeout.

## What This Changes

Report that invalid or non-retryable model candidates in a fallback chain can wait about 30 seconds each before moving to the next model on v2026.4.24-v2026.4.25.

## Best Possible Solution

Add a focused fast-fail path and regression coverage so deterministic HTTP 400/404 invalid-model or unsupported-parameter responses reach model fallback immediately, without being masked by SDK retry, stream iteration, or idle-timeout handling.

## Reproduction Assessment

Unclear. The issue gives concrete steps and affected versions, but this review did not live-run Groq/OpenRouter and current tests only reproduce an immediate mocked 400 rejection, not the observed 30-second per-candidate stall.

## Solution Assessment

Yes, a narrow fix in the provider transport/embedded stream/fallback boundary is the best direction; it should preserve retry behavior for real transient statuses while making deterministic 400/404 provider rejections terminal for that candidate immediately.
