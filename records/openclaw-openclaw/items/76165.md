## Summary

Keep open. Current main classifies immediate invalid-model 400s as fallback-worthy, but there is still no elapsed-time regression or source proof that deterministic provider failures bypass stream/request idle waits before the next fallback candidate runs.

## What This Changes

Fail fast through configured model fallbacks when deterministic invalid-model or unsupported-parameter HTTP 400 errors occur instead of spending about 30s on each failed candidate.

## Best Possible Solution

Add an elapsed-time regression for invalid-model and unsupported-parameter provider failures in the embedded fallback path, then make deterministic 400-class errors surface immediately while preserving retry, cooldown, and idle-timeout behavior for retryable or slow healthy providers.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows the fallback loop is serial and can be held by a candidate attempt, but current tests only cover immediate invalid-model rejection rather than the delayed provider/stream path reported here.

## Solution Assessment

Unclear until the delay source is isolated. The requested fail-fast behavior is the right direction, but the fix should target the layer that holds deterministic 400s instead of broadly changing shared retry or idle-timeout policy.
