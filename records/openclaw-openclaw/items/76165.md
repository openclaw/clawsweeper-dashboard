## Summary

Keep open. Current main classifies invalid-model HTTP 400 errors as fallback-worthy once the error surfaces, but I did not find evidence that deterministic invalid candidates bypass the reported per-candidate wait; the fallback loop is sequential, the transport retry bypass does not special-case 400 responses, and current coverage lacks an elapsed-time regression.

## What This Changes

The issue asks for model fallback to fail over immediately on deterministic non-retryable invalid-model or unsupported-parameter errors instead of waiting about 30 seconds per bad candidate.

## Best Possible Solution

Add a deterministic regression around slow deterministic invalid-model or unsupported-parameter candidates, then short-circuit those classified failures into the existing fallback path without weakening normal request timeouts for genuinely slow providers.

## Reproduction Assessment

No. The report has clear manual steps and version comparison, but current-main evidence only proves immediate-rejection classification, not a high-confidence elapsed-time reproduction.

## Solution Assessment

Unclear. The requested fail-fast behavior is the right product outcome, but the implementation should first isolate whether the delay happens in SDK retry, guarded fetch timeout, provider stream wrapping, or event-loop starvation.
