## Summary

Keep open. Current main classifies invalid-model HTTP 400s as fallback-worthy after the error is returned, but the reported problem is elapsed failover latency and I did not find source or test proof that deterministic invalid candidates bypass request or idle-timeout waits.

## What This Changes

Fail over immediately to the next configured model candidate when an invalid model or unsupported-parameter error is deterministic and non-retryable.

## Best Possible Solution

Add a focused fail-fast path and elapsed-time regression so deterministic invalid-model or unsupported-parameter failures enter the existing fallback path immediately while preserving normal request and idle timeouts for genuinely slow providers.

## Reproduction Assessment

No. The issue has plausible manual steps and version comparison, but current main coverage only proves immediate 400 classification and does not provide a deterministic elapsed-time or live-provider reproduction.

## Solution Assessment

Unclear. The fail-fast direction is the right end state, but the implementation should first pin down whether the wait comes from provider transport, pi-ai stream startup, or OpenClaw's embedded-run idle timeout before changing shared timeout policy.

## Repair Work Prompt

Fix the model fallback latency reported at https://github.com/openclaw/openclaw/issues/76165. Deterministic invalid model IDs and unsupported-parameter provider errors such as HTTP 400/404/422 should fall through to the next configured fallback candidate immediately once the provider response/error is available, without waiting for a per-candidate idle/request timeout. Start by adding a focused elapsed-time regression around `runWithModelFallback` plus the embedded-run/transport path that currently only tests immediate rejections. Preserve normal request and idle timeouts for genuinely slow providers and do not broadly disable SDK retries for retryable 408/409/429/5xx responses. Likely files are `src/agents/model-fallback.ts`, `src/agents/failover-error.ts`, `src/agents/provider-transport-fetch.ts`, `src/agents/pi-embedded-runner/run/llm-idle-timeout.ts`, `src/agents/pi-embedded-runner/run/attempt.ts`, and their colocated tests. Add a one-line `CHANGELOG.md` fix entry because this is user-facing. Validate with the targeted agent tests first, then `pnpm check:changed` in Testbox before handoff. Keep #75687/#75707 as context only; do not try to solve the broader event-loop saturation cluster in this PR.
