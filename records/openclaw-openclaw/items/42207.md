## Summary

Keep open. The hardening direction is legitimate, but the PR is not merge-ready because it leaves the host read path unguarded when broader filesystem access is enabled, uses lexical sensitive-path checks that symlinks can bypass, changes a public security default without the matching contract updates, and has no contributor real behavior proof.

## What This Changes

The PR flips the filesystem workspace-only default, adds a sensitive-path denylist helper and tests, and applies denylist checks to host write/edit operations.

## Best Possible Solution

Land an owner-reviewed filesystem access contract that covers defaults, canonical path checks, read/write/edit/apply_patch/media behavior, schema/help/docs, focused regression tests, and redacted real setup proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: with tools.fs.workspaceOnly=false, current main assembles the host read tool without a denylist, while this PR only adds sensitive-path checks to write/edit helpers.

## Solution Assessment

No. The hardening direction is reasonable, but this PR is not the best mergeable solution until host reads, symlink canonicalization, helper defaults, public docs/schema/help, and proof are handled together.
