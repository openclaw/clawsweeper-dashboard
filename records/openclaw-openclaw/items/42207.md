## Summary

Keep open: the security direction is important, but the branch is not merge-ready because it leaves the host read path outside the new denylist, uses bypassable lexical path checks, changes a compatibility-sensitive default without an upgrade contract, and still lacks real behavior proof.

## What This Changes

This PR flips unset filesystem tools to workspace-only by default, adds a hard-coded sensitive-path denylist, and wires that denylist into host write/edit operations with focused policy tests.

## Best Possible Solution

Land an owner-reviewed filesystem policy that either preserves the current default or deliberately flips it with migration/docs/proof, while enforcing canonicalized deny/allow behavior across read, write, edit, apply_patch, and media paths.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the review blocker: current main's workspaceOnly=false read branch returns the unguarded read wrapper, while this PR only wires the denylist into write/edit operations.

## Solution Assessment

No. The narrow denylist/default flip is not the best merge shape until it covers host reads, canonicalizes paths, updates the public contract, and resolves the default-policy decision with the broader filesystem policy work.
