## Summary

Keep open. The security hardening direction is plausible, but this PR is incomplete against its stated goal: it leaves host reads unguarded when broader filesystem access is enabled, uses lexical sensitive-path checks that symlinks can bypass, does not align the public config/docs contract, lacks real behavior proof, and needs maintainer/secops policy approval before merge.

## What This Changes

The branch flips `tools.fs.workspaceOnly` to default true, adds a sensitive-path denylist module and tests, and applies the denylist to host write/edit operations.

## Best Possible Solution

Define a maintainer-approved filesystem access contract for read/write/edit/apply_patch/media paths, including default semantics, sensitive-path handling, canonicalization, docs/schema updates, and regression tests, then implement that contract consistently.

## Reproduction Assessment

Yes. Source inspection is sufficient here: current main leaves unset/false `tools.fs.workspaceOnly` unrestricted for host reads, and the PR diff only adds sensitive-path checks to host write/edit while using lexical path matching; I did not run a live gateway in this read-only review.

## Solution Assessment

No. The hardening direction is reasonable, but this patch is not the best complete solution because it misses host reads, does not canonicalize symlink targets, leaves the public config contract stale, and changes a public default without a recorded security/product decision.
