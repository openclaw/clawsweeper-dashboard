## Summary

Keep open. This is a valid security-hardening direction, but the branch is not merge-ready because it misses the standalone read path, uses bypassable lexical path checks, changes an existing default without upgrade proof, and still lacks real behavior proof.

## What This Changes

The PR changes the filesystem tool policy default to workspace-only, adds a hardcoded sensitive path denylist, wires it into host write/edit operations, and adds unit tests for the default and denylist.

## Best Possible Solution

Land a maintainer-approved, realpath-aware filesystem policy layer covering read/write/edit/apply_patch/media/subagents, with upgrade docs and proof for any default change.

## Reproduction Assessment

Yes for the review blockers. Source inspection of the PR head and current main shows the standalone host read path remains unguarded when `workspaceOnly=false`, and the new denylist uses `path.resolve` rather than canonical realpath checks.

## Solution Assessment

No. A write/edit-only denylist plus an unconditional default flip is not the narrow maintainable fix; the safer path is one reviewed filesystem policy layer with explicit upgrade handling.
