## Summary

Keep open. Current main still uses the legacy filesystem policy, and this security-sensitive PR is not merge-ready because it leaves host reads unguarded, uses lexical sensitive-path matching, lacks public-contract updates, and has no after-fix real behavior proof.

## What This Changes

The PR flips `tools.fs.workspaceOnly` to default `true` and adds a sensitive-path denylist enforced for host write/edit operations with tests.

## Best Possible Solution

Land an owner-reviewed filesystem access contract that covers defaults, canonical path checks, read/write/edit/apply_patch/media behavior, schema/help/docs, focused regression tests, and redacted real setup proof.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence path: with `tools.fs.workspaceOnly=false`, current main assembles the host read tool without a denylist, while this PR only guards host write/edit helpers.

## Solution Assessment

No. The direction is reasonable, but the mergeable solution needs host reads, symlink-safe canonicalization, helper defaults, public docs/schema/help, and real behavior proof handled together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Enforce the denylist on host reads:** `src/agents/pi-tools.ts:690-700`
  - body: The sensitive-path checks only cover host write/edit operations. When broader host access is allowed, the standalone read tool is still assembled without this guard, so the credential-read exfiltration scenario described by the PR remains open.
  - confidence: 0.96
- **[P2] Canonicalize sensitive paths before matching prefixes:** `src/security/sensitive-paths.ts:64-65`
  - body: `isSensitivePath` checks `path.resolve()` and basename patterns, but filesystem access follows symlinks. A non-sensitive-looking symlink can point at a denied credential location and bypass the new protection.
  - confidence: 0.93
- **[P2] Update the public config contract for the new default:** `src/agents/tool-fs-policy.ts:12`
  - body: The runtime fallback changes unset config to workspace-only, but public types, schema help, and security docs still describe `tools.fs.workspaceOnly` as optional/default-false hardening. Operators would get conflicting upgrade guidance.
  - confidence: 0.88
- **[P2] Align host helper defaults with the new policy:** `src/agents/pi-tools.read.ts:724`
  - body: The host write/edit helper operations still default omitted options to `workspaceOnly=false`. Direct callers that omit options remain unrestricted even though the policy resolver would now treat unset as workspace-only.
  - confidence: 0.84
