## Summary

Keep open. The hardening goal is valid-looking and current main has not implemented it, but this PR is incomplete for its stated security boundary: it misses host reads, uses lexical sensitive-path matching, leaves helper defaults inconsistent, and lacks real after-fix proof.

## What This Changes

The PR changes the filesystem policy default to workspace-only, adds a sensitive-path denylist and tests, and wires that denylist into host write/edit operations.

## Best Possible Solution

Define one maintainer-approved filesystem access contract for read/write/edit/apply_patch, sensitive-path denial, symlink handling, config/docs, and tests, then implement that contract consistently across host and sandbox paths.

## Reproduction Assessment

Yes. Source inspection of current main and the PR diff shows the PR only adds denylist checks to host write/edit while host read remains unguarded when workspaceOnly=false, and the new denylist uses lexical path matching.

## Solution Assessment

No. The hardening direction is plausible, but this patch is not the best complete solution because it misses host reads, does not canonicalize sensitive targets, and changes a public default without aligning the config contract and docs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Enforce the denylist on host reads:** `src/agents/pi-tools.read.ts:724-736`
  - body: The new denylist is only called from host write/edit operations. In the workspaceOnly=false host path, read is still created from createReadTool(...) and pushed without a sensitive-path guard, so the credential-read exfiltration scenario described by the PR remains open.
  - confidence: 0.96

  - body: isSensitivePath uses path.resolve() and string-prefix matching, which does not dereference symlinks. A non-sensitive-looking symlink can point at ~/.ssh or another denied target and pass this check before filesystem access follows it.
  - confidence: 0.93
- **[P2] Align helper defaults with the new policy:** `src/agents/pi-tools.read.ts:724`
  - body: The host write/edit helpers still default omitted options to workspaceOnly=false. Direct callers that omit options would remain unrestricted even though this PR changes the policy layer to treat unset as workspace-only.
  - confidence: 0.86
- **[P2] Update the public config contract for the new default:** `src/agents/tool-fs-policy.ts:12`
  - body: Changing the effective default without updating type comments, generated help, security docs, and changelog leaves users and config surfaces with the old default:false/optional-hardening contract.
  - confidence: 0.88
