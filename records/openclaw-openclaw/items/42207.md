## Summary

Keep this PR open. Current main still defaults `tools.fs.workspaceOnly` to false and has no sensitive-path denylist for agent file tools, so the central hardening request is not implemented. The PR is a plausible security-hardening direction, but the submitted patch leaves the host `read` tool unguarded when `workspaceOnly=false`, uses lexical path checks that symlinks can bypass, and does not align config/docs/changelog with the default change.

## What This Changes

The PR changes `tools.fs.workspaceOnly` to default true, adds a new `src/security/sensitive-paths.ts` denylist with tests, and applies that denylist to host write/edit operations.

## Best Possible Solution

Keep this PR open as a source implementation candidate, but require a maintainer/secops pass before landing. The finished patch should make one explicit decision about the filesystem default, enforce any sensitive-path policy through one canonical resolver for read/write/edit before file access, cover symlink aliases, update config/schema/help/security docs/changelog, and coordinate with #70795 for read-path policy consistency.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: with `tools.fs.workspaceOnly=false`, current and proposed host read assembly returns an unguarded `createOpenClawReadTool`, while the PR only adds denylist checks to write/edit operations; a symlink bypass follows from the proposed `path.resolve()`-only implementation.

## Solution Assessment

No. The direction is reasonable, but this patch is not the best mergeable solution because it does not protect the read surface that motivates the PR and does not canonicalize paths before denylist checks; a safer fix would centralize policy enforcement and update docs/config/tests together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Enforce the denylist on host reads:** `src/agents/pi-tools.read.ts:728`
  - body: The denylist is only applied inside host write/edit operations. When `tools.fs.workspaceOnly` is explicitly false, `src/agents/pi-tools.ts` still builds the normal host `read` tool without this guard, so an injected agent can still read `~/.ssh/id_rsa` or similar files despite the new denylist.
  - confidence: 0.95
- **[P2] Canonicalize paths before checking sensitivity:** `src/security/sensitive-paths.ts:63`
  - body: `isSensitivePath` uses `path.resolve()`, which normalizes strings but does not follow symlinks. A workspace or temp symlink pointing at a blocked credential path can pass the denylist check and then be followed by the filesystem call, so the protection is bypassable.
  - confidence: 0.92
- **[P2] Align host factory defaults with the new policy:** `src/agents/pi-tools.read.ts:724`
  - body: After changing the policy default to workspace-only, the exported host write/edit factories still use `options?.workspaceOnly ?? false`. Any direct call without options remains unrestricted, contradicting the new default and leaving tests/helpers able to bypass the policy accidentally.
  - confidence: 0.84
