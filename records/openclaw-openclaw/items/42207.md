## Summary

Keep open: the hardening direction is valid, but this patch is not merge-ready because host reads remain unguarded, the denylist is lexical/symlink-bypassable, the public config contract stays stale, and contributor real behavior proof is absent.

## What This Changes

The PR flips `tools.fs.workspaceOnly` to default workspace-only, adds a hard-coded sensitive-path helper with tests, and applies that helper to host write/edit operations.

## Best Possible Solution

Land an owner-reviewed filesystem access contract that covers defaults, canonical path checks, read/write/edit/apply_patch/media behavior, schema/help/docs, focused regression tests, and redacted real setup proof.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence path: with `tools.fs.workspaceOnly=false`, current main assembles the host read tool without a denylist, while this PR only guards host write/edit helpers.

## Solution Assessment

No. The direction is reasonable, but the mergeable solution needs host reads, symlink-safe canonicalization, helper defaults, public docs/schema/help, and real behavior proof handled together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Enforce the denylist on host reads:** `src/agents/pi-tools.ts:688-699`
  - body: The new sensitive-path checks only cover host write/edit operations. When broader host access is allowed, the standalone read tool is still assembled without this guard, so the credential-read exfiltration scenario described by the PR remains open.
  - confidence: 0.96
- **[P2] Canonicalize paths before matching sensitive prefixes:** `src/security/sensitive-paths.ts:64-65`
  - body: `isSensitivePath` checks `path.resolve()` and basename patterns, but filesystem access follows symlinks. A non-sensitive-looking symlink can point at a denied credential location and bypass the new protection.
  - confidence: 0.93
- **[P2] Update the public config contract for the new default:** `src/agents/tool-fs-policy.ts:12`
  - body: The runtime fallback changes to workspace-only, but public types, schema help, security docs, and contract coverage still describe `tools.fs.workspaceOnly` as an optional/default-false guard, which would mislead operators.
  - confidence: 0.88
- **[P2] Align host helper defaults with the new policy:** `src/agents/pi-tools.read.ts:724`
  - body: The host write/edit helper operations still default omitted options to `workspaceOnly=false`. Direct callers that omit options remain unrestricted even though the policy resolver would now treat unset as workspace-only.
  - confidence: 0.84
