## Summary

Keep this PR open, but do not merge it as-is: the security direction is useful, yet the patch misses the main host-read exfiltration path, uses bypassable lexical denylist matching, and changes a public filesystem default without the needed contract updates.

## What This Changes

The PR flips the effective `tools.fs.workspaceOnly` default, adds a sensitive-path denylist helper with tests, and inserts denylist checks into host write/edit operations.

## Best Possible Solution

Finish this as one coordinated filesystem-policy hardening change: decide the public default, enforce canonical sensitive-path policy consistently across read/write/edit, update config/help/docs/changelog, and align the result with #70795.

## Reproduction Assessment

Yes for the blocking review findings. Source inspection shows the PR guards host write/edit paths but not the host `read` tool branch used when `tools.fs.workspaceOnly=false`, and the proposed denylist checks lexical paths rather than canonical targets.

## Solution Assessment

No. The hardening direction is reasonable, but this patch is not the best complete fix because it misses the motivating read-exfiltration surface and changes the public default without updating the surrounding contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Enforce the denylist on host reads:** `src/agents/pi-tools.read.ts:724-736`
  - body: The new denylist is only applied from host write/edit operations. With `tools.fs.workspaceOnly=false`, the host `read` tool is still assembled without this guard, so credential files remain readable through the exact exfiltration surface this PR is meant to close.
  - confidence: 0.96
- **[P2] Canonicalize targets before matching the denylist:** `src/security/sensitive-paths.ts:63-64`
  - body: `isSensitivePath` uses `path.resolve()` and string matching, which does not dereference symlinks. A symlink from an allowed-looking path to a denied credential path can bypass the check before the filesystem follows the link.
  - confidence: 0.93
- **[P2] Align helper defaults with the new policy:** `src/agents/pi-tools.read.ts:723-724`
  - body: After changing the policy default to workspace-only, the host write/edit factories still default omitted options to unrestricted mode. Direct callers that omit `workspaceOnly` would not get the new default advertised by this PR.
  - confidence: 0.86
- **[P2] Update the public config contract with the new default:** `src/agents/tool-fs-policy.ts:10-13`
  - body: This changes effective user-facing filesystem behavior, but the PR only updates one unit test. Config comments, generated help/docs, security guidance, and the changelog still need to describe the new default so users do not get contradictory policy guidance.
  - confidence: 0.88
