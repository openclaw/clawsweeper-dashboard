## Summary

Keep open and do not merge as-is. The security-hardening direction is useful, but the branch misses the motivating host read path, uses lexical denylist checks that can be bypassed by symlinks, leaves helper defaults and public config/docs inconsistent with the new default, and lacks external after-fix behavior proof.

## What This Changes

The PR flips `tools.fs.workspaceOnly` to default true, adds a sensitive-path denylist with tests, and applies that denylist to host write/edit operations.

## Best Possible Solution

Finish this as one maintainer-approved filesystem security policy: decide the default/profile behavior, enforce canonical sensitive-path checks across read/write/edit, update schema/help/docs/changelog/tests, and coordinate with https://github.com/openclaw/openclaw/issues/70795.

## Reproduction Assessment

Yes. Source inspection shows current main assembles an unguarded host `read` tool when `tools.fs.workspaceOnly=false`, while the PR only adds the denylist to host write/edit operations and its new helper uses lexical path matching.

## Solution Assessment

No. The hardening direction is plausible, but the patch is incomplete because it misses host reads, does not canonicalize targets, and changes a public default without aligning the config contract and docs.
