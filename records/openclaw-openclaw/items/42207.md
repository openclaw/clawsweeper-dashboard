## Summary

Keep open: the PR is valid security hardening work, but the current patch is not mergeable because the proposed denylist does not cover the host `read` path when `tools.fs.workspaceOnly=false`, the new path matcher is lexical rather than canonical, and the public config contract still documents the opposite default.

## What This Changes

The PR flips unset `tools.fs.workspaceOnly` handling to workspace-only by default, adds a sensitive-path denylist with tests, and applies that denylist to host write/edit operations.

## Best Possible Solution

Finish this as one filesystem-policy hardening change: decide the default, enforce sensitive-path checks through a shared read/write/edit resolver with canonical path validation, update schema/help/docs/changelog, and align the read-policy behavior with #70795.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with `tools.fs.workspaceOnly=false`, host `read` is returned without the proposed denylist, while the new helper described in the PR uses lexical path resolution rather than canonical realpath checks.

## Solution Assessment

No. The hardening direction is reasonable, but this patch is not the best complete fix because it leaves the motivating read surface uncovered and changes a public default without updating the contract or using the existing canonical filesystem safety patterns.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Enforce the denylist on host reads:** `src/agents/pi-tools.read.ts:724-736`
  - body: The new denylist is only wired into host write/edit operations. When `tools.fs.workspaceOnly=false`, the host `read` tool is still assembled without this guard, so credential files remain readable through the path the PR is meant to protect.
  - confidence: 0.95
- **[P2] Canonicalize paths before matching the denylist:** `src/security/sensitive-paths.ts:63-64`
  - body: `isSensitivePath` relies on lexical `path.resolve()` matching. A symlink from an allowed-looking path to a denied credential path can bypass that check before the filesystem follows the link, so the policy should validate canonical targets before access.
  - confidence: 0.92
- **[P2] Align host helper defaults with the new policy:** `src/agents/tool-fs-policy.ts:13`
  - body: After changing the policy default to workspace-only, direct host write/edit helper calls still default missing options to unrestricted mode. Callers that omit `workspaceOnly` would not get the new default the PR advertises.
  - confidence: 0.84
- **[P2] Update the public config contract with the new default:** `src/agents/tool-fs-policy.ts:33`
  - body: The PR changes effective behavior but leaves generated help, source config comments, security docs, and the changelog saying or implying that `tools.fs.workspaceOnly` defaults to false or is optional hardening. Users and config tooling would get contradictory guidance.
  - confidence: 0.88
