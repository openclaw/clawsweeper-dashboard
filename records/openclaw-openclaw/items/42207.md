## Summary

Keep open. Current main still documents and tests the unset `tools.fs.workspaceOnly` default as false, and this PR is not mergeable as-is because the proposed denylist does not cover the host read path that motivates the change and uses lexical path checks that symlink aliases can bypass.

## What This Changes

The PR changes the filesystem workspace-only default to true, adds a sensitive-path denylist, and applies that denylist to host write/edit operations with new tests.

## Best Possible Solution

Finish this as a cohesive filesystem-policy hardening change: decide the default explicitly, enforce sensitive-path policy through one canonical read/write/edit resolver, cover symlink aliases, and update config schema/help, security docs, changelog, and #70795 alignment together.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with `tools.fs.workspaceOnly=false`, host `read` is assembled without the proposed denylist, while the new helper uses `path.resolve()` rather than canonical realpath checks.

## Solution Assessment

No. The direction is reasonable hardening, but the current patch is not the narrowest maintainable fix because it leaves the motivating read surface uncovered and does not update the public config contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Enforce the denylist on host reads:** `src/agents/pi-tools.read.ts:724-736`
  - body: The new denylist only runs in host write/edit operations. When `tools.fs.workspaceOnly=false`, the host `read` tool remains the normal read wrapper, so an agent can still read credential paths that this PR intends to block.
  - confidence: 0.95
- **[P2] Canonicalize paths before matching the denylist:** `src/security/sensitive-paths.ts:63-64`
  - body: `isSensitivePath` uses lexical `path.resolve()` matching, which does not follow symlinks. A symlink outside a denied prefix can point at a denied credential path and bypass the new check.
  - confidence: 0.92
- **[P2] Keep host factory defaults aligned with the policy:** `src/agents/tool-fs-policy.ts:13`
  - body: After changing the policy default to workspace-only, the host write/edit factories still default omitted options to `false`, so direct helper callers stay unrestricted unless they pass the policy explicitly.
  - confidence: 0.84
- **[P2] Update the public config contract with the new default:** `src/agents/tool-fs-policy.ts:33`
  - body: Changing the effective default without updating config types, generated help/schema text, security docs, and changelog leaves users with published guidance that still says this setting defaults to false.
  - confidence: 0.88
