## Summary

This PR should stay open for maintainer review: the filesystem-security problem is real, but the branch still has concrete security-boundary gaps, missing real behavior proof, a conflicting head, and an upgrade-sensitive default change that current main has not adopted.

## What This Changes

The PR flips `tools.fs.workspaceOnly` to default to `true`, adds a sensitive path denylist, applies it to host write/edit paths, and adds policy/denylist tests.

## Best Possible Solution

Land one owner-reviewed filesystem policy layer that applies consistently to read, write, edit, apply_patch, media, sandbox, and subagent paths, with realpath-aware denial and explicit upgrade guidance for any default change.

## Reproduction Assessment

Yes for the review blockers: source inspection shows unset `tools.fs.workspaceOnly` still defaults to false on current main, and the PR's denylist does not cover host reads or canonical symlink resolution. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No, not as proposed: a denylist plus default flip is too narrow for this security boundary without realpath-aware canonicalization, full tool-surface enforcement, and an explicit compatibility decision. The safer path is a single filesystem policy layer with upgrade proof.
