## Summary

Keep open: the hardening direction is valid, but the PR is incomplete because it leaves the host read exfiltration path and symlink aliases uncovered while changing a public filesystem-policy default.

## What This Changes

The PR changes the filesystem policy default to workspace-only, adds a sensitive-path denylist with tests, and wires the denylist into host write/edit operations.

## Best Possible Solution

Finish this as one filesystem-policy hardening change: decide the default, enforce sensitive-path checks through a shared canonical read/write/edit resolver, update schema/help/docs/changelog, and coordinate the read-policy behavior with #70795.

## Reproduction Assessment

Yes. Source inspection shows current main assembles an unguarded host `read` tool when `tools.fs.workspaceOnly=false`, while the PR diff only adds the new denylist to host write/edit paths and uses lexical path matching.

## Solution Assessment

No. The hardening direction is reasonable, but this patch is not the best complete fix because it misses the motivating read surface and changes the public default without updating the contract or using canonical filesystem safety patterns.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Enforce the denylist on host reads:** `src/agents/pi-tools.read.ts:724-736`
  - body: The new denylist is only called from host write/edit operations. With `tools.fs.workspaceOnly=false`, the host read tool is still returned without this guard, so credential files remain readable through the surface the PR is meant to protect.
  - confidence: 0.95
- **[P2] Canonicalize targets before matching the denylist:** `src/security/sensitive-paths.ts:63-64`
  - body: `isSensitivePath` uses lexical `path.resolve()` matching. A symlink from an allowed-looking path to a denied credential path can bypass that check before the filesystem follows the link, so existing targets should be checked through canonical realpaths or the shared safe-open resolver.
  - confidence: 0.92
- **[P2] Align helper defaults with the new policy:** `src/agents/pi-tools.read.ts:723-724`
  - body: After flipping the policy default to workspace-only, the host write/edit factories still default omitted options to unrestricted mode. Direct callers that omit `workspaceOnly` would not get the new default advertised by the PR.
  - confidence: 0.84
- **[P2] Update the public config contract with the new default:** `src/agents/tool-fs-policy.ts:33`
  - body: The PR changes effective behavior but leaves generated help, source config comments, security docs, and the changelog saying or implying that `tools.fs.workspaceOnly` defaults to false or is optional hardening. Users and config tooling would get contradictory guidance.
  - confidence: 0.88
