## Summary

Keep open: this is useful security hardening, but it is not merge-ready because the denylist misses the standalone host read path, uses lexical path checks that symlinks can bypass, changes a compatibility-sensitive default without the full public contract, and lacks real behavior proof.

## What This Changes

The branch flips unset `tools.fs.workspaceOnly` to true, adds a sensitive-path denylist with tests, and wires that denylist into host write/edit operations.

## Best Possible Solution

Land an owner-reviewed filesystem policy that either narrows this PR to a safe interim denylist covering the full read/write/edit surface with canonical path checks and docs, or supersedes it through the broader PathGuard design.

## Reproduction Assessment

Yes. Source inspection shows current main and this PR's diff leave the host read path without a sensitive-path denylist when `tools.fs.workspaceOnly=false`; no live run is needed to prove that code path.

## Solution Assessment

No. The direction is reasonable, but the mergeable solution needs host reads, symlink-safe canonicalization, helper defaults, public docs/schema/help, and upgrade proof handled together.
