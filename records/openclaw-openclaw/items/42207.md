## Summary

Keep open. The security hardening direction is relevant, but the patch is incomplete: it leaves host reads unguarded when `workspaceOnly=false`, uses lexical sensitive-path checks that symlinks can bypass, leaves helper defaults/docs inconsistent with the proposed default flip, and has no after-fix real behavior proof.

## What This Changes

The PR changes filesystem tools to default to workspace-only mode, adds a new sensitive-path denylist module and tests, and wires that denylist into host write/edit operations.

## Best Possible Solution

Define a maintainer-approved filesystem access contract for read/write/edit/apply_patch, sensitive-path denial, symlink handling, config/docs, and tests, then implement it consistently across host and sandbox paths.

## Reproduction Assessment

Yes. Source inspection of current main and the PR diff shows the patch only adds denylist checks to host write/edit while host read remains unguarded when `workspaceOnly=false`, and the new denylist uses lexical path matching.

## Solution Assessment

No. The hardening direction is plausible, but this patch is not the best complete solution because it misses host reads, does not canonicalize sensitive targets, and changes a public default without aligning the config contract and docs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Enforce the denylist on host reads:** `src/agents/pi-tools.read.ts:778`
  - body: The PR adds `sensitivePathReason` only to host write/edit operations. In the `workspaceOnly=false` host path, the normal read tool is still created from `createReadTool(...)` and pushed without a sensitive-path guard, so the credential-read exfiltration scenario this PR is meant to block still works.
  - confidence: 0.96
- **[P2] Canonicalize paths before matching sensitive prefixes:** `src/security/sensitive-paths.ts:64`
  - body: `isSensitivePath` uses `path.resolve()` plus string-prefix and filename matching, but it does not dereference symlinks. An allowed-looking symlink path can point at a denied credential location and pass this check before filesystem access follows the target.
  - confidence: 0.93
- **[P2] Align helper defaults with the new policy:** `src/agents/pi-tools.read.ts:724`
  - body: The host write/edit helpers still default omitted options to `workspaceOnly=false`. Direct callers that omit options would remain unrestricted even though this PR changes the policy resolver to treat unset as workspace-only.
  - confidence: 0.86
- **[P2] Update the public config contract for the new default:** `src/agents/tool-fs-policy.ts:12`
  - body: Changing the effective default without updating generated config help, security docs, and related contract tests leaves users with the old `default: false` and optional-hardening contract while runtime behavior changes under them.
  - confidence: 0.88
