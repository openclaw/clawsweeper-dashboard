## Summary

Keep open. The security-hardening direction is valid, but the PR is not mergeable as-is: it misses the standalone host `read` path, uses lexical sensitive-path matching that symlinks can bypass, leaves the public config/docs contract inconsistent with the new default, has no real behavior proof, and overlaps broader active filesystem-policy work without fully superseding the default-policy decision.

## What This Changes

This PR changes `tools.fs.workspaceOnly` to default true, adds a sensitive-path denylist with tests, and applies that denylist to host write/edit operations.

## Best Possible Solution

Define a maintainer-approved filesystem access contract that covers default semantics, sensitive-path handling, canonicalization, read/write/edit/apply_patch/media coverage, config schema/help/docs, and regression tests before landing this kind of hardening.

## Reproduction Assessment

Yes. Source inspection is enough for the PR blockers: current main leaves unset/false `tools.fs.workspaceOnly` unrestricted for host reads, and the PR diff only adds sensitive-path checks to host write/edit while matching paths lexically.

## Solution Assessment

No. The hardening direction is reasonable, but this patch is incomplete; the safer solution is an owner-reviewed filesystem policy implementation that consistently covers all file-bearing surfaces and updates the public contract.

  - body: The PR applies `sensitivePathReason` only inside host write/edit operations. When `workspaceOnly=false`, the standalone host read tool is still created from `createReadTool(...)` and pushed without a sensitive-path guard, so the credential-read exfiltration scenario remains open.

  - confidence: 0.88

  - body: The host write/edit helper operations still default omitted options to `workspaceOnly=false`. Direct callers that omit options would remain unrestricted even though this PR changes the policy resolver to treat unset as workspace-only.
