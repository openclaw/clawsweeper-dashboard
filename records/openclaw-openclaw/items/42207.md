## Summary

Keep open but not merge-ready: the hardening direction is useful, but the patch leaves host reads unguarded, uses lexical sensitive-path checks that symlinks can bypass, changes a public default without contract/docs coverage, and lacks real behavior proof.

## What This Changes

The PR flips the filesystem workspace-only default to true, adds a sensitive-path denylist, wires it into host write/edit helpers, and updates/adds unit tests.

## Best Possible Solution

Settle and implement one filesystem access contract covering defaults, canonical path checks, read/write/edit/apply_patch/media behavior, schema/help/docs, focused regression tests, and real setup proof.

## Reproduction Assessment

Yes. Source inspection shows current main constructs the host read tool without a denylist when workspaceOnly is false, while the PR only guards write/edit and its new helper uses path.resolve/string matching.

## Solution Assessment

No. The hardening direction is reasonable, but this PR is not the safest merge path until read coverage, canonicalization, helper defaults, and public contract updates are handled together.
