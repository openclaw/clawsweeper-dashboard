## Summary

This PR should stay open for maintainer/security review: it changes a compatibility-sensitive filesystem policy, but the patch leaves host reads unprotected, uses lexical sensitive-path checks that symlink aliases can bypass, does not update the full public config/docs contract, and has no after-fix real behavior proof.

## What This Changes

The PR changes `tools.fs.workspaceOnly` to default true, adds a hardcoded sensitive-path denylist, applies that denylist to host write/edit helpers, and updates/adds unit tests.

## Best Possible Solution

Settle one filesystem access contract that covers defaults, sensitive-path handling, canonicalization, read/write/edit/apply_patch/media coverage, schema/help/docs, and tests before landing a security hardening change.

## Reproduction Assessment

Yes. Source inspection is enough for the PR blockers: current main has a separate unguarded host-read path when `workspaceOnly=false`, while the PR applies the new denylist only to write/edit helpers and matches sensitive paths lexically.

## Solution Assessment

No. The hardening direction is reasonable, but this patch is not the safest mergeable solution because it leaves the primary host-read exfiltration path and symlink bypass unresolved and changes the public default without a full contract update.

## Review Findings

Overall confidence: 0.94

Full review comments:

- **[P1] Enforce the denylist on host reads:** `src/agents/pi-tools.ts:654-659`
  - body: The PR only adds `sensitivePathReason` checks inside host write/edit operations. When broader host access is allowed, the standalone `read` tool is still created through the normal host read path and pushed without a sensitive-path guard, so the credential-read exfiltration scenario described in the PR remains open.
  - confidence: 0.96

  - body: The new sensitive-path helper uses `path.resolve()` and string/filename matching, but it does not dereference symlinks before filesystem access follows them. An allowed-looking symlink can therefore point at a denied credential location and bypass the protection.
  - confidence: 0.93
- **[P2] Update the public config contract for the new default:** `src/agents/tool-fs-policy.ts:12`
  - body: Changing the effective `tools.fs.workspaceOnly` fallback to true without updating the schema help, public types, security docs, and contract coverage leaves users with public guidance that still describes the default as false while runtime behavior changes.
  - confidence: 0.88
- **[P2] Align helper defaults with the new policy:** `src/agents/pi-tools.read.ts:724`
  - body: The host write/edit helper operations still default omitted options to `workspaceOnly=false`. Direct callers that omit options would remain unrestricted even though this PR changes the policy resolver to treat unset as workspace-only.
  - confidence: 0.84
