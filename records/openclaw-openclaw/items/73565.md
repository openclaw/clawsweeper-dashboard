## Summary

Keep open: current main still has direct session transcript append sinks that bypass the config-aware guard redaction path, and the linked closing PR is open and unmerged. Because that PR is the active implementation candidate, this issue should remain open until the fix lands or is superseded.

## What This Changes

The issue requests applying the existing transcript redaction guard to session transcript persistence entry points before message payloads are written to JSONL transcript files.

## Best Possible Solution

Review and land PR #73563 or an equivalent maintainer-approved fix that wraps each bare transcript append sink in guardSessionManager, threads config where redaction mode matters, preserves idempotency and transcript update semantics, and adds regression coverage for both redaction and redactSensitive="off" behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists by sending secret-like content through the config transcript append, CLI text-turn transcript, gateway inject helper, or Codex mirror while transcript redaction is enabled; current main shows direct appendMessage calls on those paths.

## Solution Assessment

Yes. Reusing guardSessionManager is the narrowest maintainable direction because it applies the existing before-write hook, transcript redaction, and tool-result guard infrastructure instead of adding a parallel scrubber.
