## Summary

Keep open: the linked bug is source-reproducible and current main still reuses fresh failed sessions by freshness/sessionId alone, but this PR has a blocking compatibility defect because it probes the raw stored transcript path and still lacks required real gateway/channel behavior proof.

## What This Changes

The PR adds a gateway agent reuse guard for failed sessions whose stored transcript path is missing and a regression test that expects session id rotation plus `sessionFile` clearing.

## Best Possible Solution

Keep the failed-lane recovery policy, but resolve stored transcript paths through the canonical session path helpers before the missing-file check, cover both missing and present resolved transcripts, and require redacted real gateway/channel recovery output before merge.

## Reproduction Assessment

Yes, source-reproducible: seed a fresh `status: "failed"` gateway session entry with an existing `sessionId` and missing `sessionFile`, then invoke the same session key and current main selects the old session id. I did not execute tests in this read-only review.

## Solution Assessment

No. The recovery direction is right, but the implementation should resolve `sessionFile` through the existing store-dir/agent-aware helper before checking existence and should prove the real gateway/channel recovery path.
