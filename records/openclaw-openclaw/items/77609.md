## What This Changes

The PR refactors embedded-runner runtime allowlist handling into a tool-construction plan, narrows local tool-family startup for `toolsAllow`, updates bundled MCP/LSP gating, and adds tests plus a changelog entry.

## Best Possible Solution

Keep the allowlist construction refactor, but make plugin-only tool resolution receive the same auth profile context as the full OpenClaw tool factory and cover auth-gated plugin/group allowlists.

## Reproduction Assessment

Yes for the review finding by source inspection. The plugin-only branch calls `resolveOpenClawPluginToolsForOptions` without `authProfileStore`, while that resolver needs the store to satisfy manifest auth signals for plugin tools.

## Solution Assessment

No. The PR direction is narrow and maintainable, but the plugin-only path must forward the same auth profile store as the normal `createOpenClawTools` path before this is the best fix.

## Review Findings

- **[P2] Forward auth profiles in plugin-only construction:** `src/agents/pi-tools.ts:661-688`
