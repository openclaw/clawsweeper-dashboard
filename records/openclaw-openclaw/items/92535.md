## Summary

Keep this open: current main and the latest release still contain the self-debug rewrite plus cron recovery classification that can treat final assistant-visible text as deliverable, and an open maintainer PR is explicitly paired as the fix candidate.

## What This Changes

The issue reports that isolated cron delivery can publish internal unavailable-tool self-debug text as a normal Telegram response.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/93207 with a structured exhausted-unavailable-tool signal that makes isolated cron fail closed without dispatching model self-debug text, plus focused runner/cron regressions and Telegram-visible proof.

## Reproduction Assessment

Yes. Current-main source still rewrites repeated unavailable-tool calls into assistant text, and cron can treat final assistant-visible text as recovery unless an explicit fatal signal or structured error is present; the reporter also supplied a real gateway and Telegram trace.

## Solution Assessment

Yes. A structured runner-to-cron failure signal consumed before delivery is the best primary fix; a delivery text filter would be brittle against model phrasing and legitimate user text.
