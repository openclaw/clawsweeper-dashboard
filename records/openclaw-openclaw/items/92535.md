## Label Justifications

- `P1`: A real isolated-cron Telegram workflow can publish internal runtime failure text as a successful chat reply and persist it in session state.
- `impact:session-state`: The bad assistant output is appended to the topic transcript and can later be replayed or ingested by memory/dreaming paths.

## Summary

Keep this open: current main still rewrites repeated unavailable-tool calls into plain assistant text and cron can treat final assistant-visible text as successful delivery. The linked fix path is the open draft PR at https://github.com/openclaw/openclaw/pull/92545, not a merged current-main implementation.

## What This Changes

Prevent isolated cron runs from publishing unavailable-tool self-debug text as successful Telegram output.

## Best Possible Solution

Land a structured runner-owned unavailable-tool failure signal that cron classifies as fatal before normal delivery, then verify the Telegram behavior and leave memory replay filtering to the existing follow-up at https://github.com/openclaw/openclaw/pull/80666.

## Reproduction Assessment

Yes. The reporter supplied a real gateway/Telegram trace, and current-main source still shows the unknown-tool rewrite plus cron final-text recovery path that can make that text deliverable.

## Solution Assessment

Yes for the structured-signal direction, not for a delivery text filter alone. The active draft PR follows the better owner boundary, but it still needs maintainer review and Telegram-facing proof before the issue can close.
