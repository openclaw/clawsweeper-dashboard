## Summary

Keep this issue open: current main still converts repeated unavailable-tool calls into the exact ordinary assistant prose seen in the production trace, and isolated-cron finalization can persist and deliver that prose as successful output. The related work covers adjacent tool-surface, scheduled-policy, A2A, memory-filtering, or terminal-fallback concerns, but no merged fix owns this exact terminal-outcome leak. Likely related people: @dutifulbob introduced the unknown-tool loop guard, and @osolmaz merged that work and is subscribed to this issue.

## What This Changes

Prevent isolated cron runs from publishing unavailable-tool self-debug text as successful Telegram output.

## Best Possible Solution

Emit a closed structured terminal failure when the unknown-tool loop guard trips, propagate it through the canonical agent-run terminal outcome, and have isolated cron retain operator diagnostics while skipping transcript success classification and user-chat delivery.

## Reproduction Assessment

Yes at the source level with high confidence. The supplied real-gateway trace contains the exact sentence hard-coded by current main’s unknown-tool loop rewrite, and current cron finalization can treat that ordinary assistant text as deliverable output; this review did not rerun a live Telegram scenario.

## Solution Assessment

No, a free-form delivery-text guard is not the best complete solution. Structured terminal failure propagation is channel-neutral, avoids false positives, and prevents the failure from being persisted as successful assistant output.
