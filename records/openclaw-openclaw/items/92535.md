## Summary

Keep this issue open: current main still has the source-level path that can turn repeated unavailable-tool calls into plain assistant text, and the issue is now paired with open closing fix PRs that need review rather than a cleanup close.

## What This Changes

The issue asks OpenClaw to prevent isolated cron runs from delivering internal unavailable-tool self-debug text to Telegram or other chat channels as successful assistant output.

## Best Possible Solution

Review and land the maintainer-owned fix in https://github.com/openclaw/openclaw/pull/93207, or deliberately select the competing structured-signal candidate, then close this issue after the chosen PR merges.

## Reproduction Assessment

Yes. The reporter supplied a real gateway and Telegram trace, and current-main source still shows the same chain: `exec` suggests `process`, the unknown-tool guard rewrites repeated missing-tool calls into assistant text, and cron can treat final assistant-visible text as recovery.

## Solution Assessment

Yes, the best fix is a structured exhausted-unavailable-tool signal owned by the runner and consumed by cron before delivery. A text filter is a weaker fallback because model phrasing can drift and legitimate user text could match.
