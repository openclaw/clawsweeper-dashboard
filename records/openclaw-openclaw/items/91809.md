## Summary

Current main partially reduces latency through a configured-only CLI path, a 750 ms browse fallback, and gateway/UI caching, but provider-wide wildcard selectors still deliberately require the full catalog and auth-backed visibility path. That matches the reporter’s configuration, so the central regression is not proven fixed; the more specific evidence here should remain canonical while the older UI-only report remains related.

## What This Changes

Restore fast `/models` and `openclaw models list` responses for multi-provider configurations using provider-wide wildcard model selectors.

## Best Possible Solution

Define one lifecycle-owned catalog snapshot and refresh contract that lets default and provider-wildcard views return bounded cached results immediately, refresh discovery off the request path, and preserve complete provider-scoped results once refreshed across CLI, slash-command, gateway, and UI callers.

## Reproduction Assessment

Yes at source level: provider-wide selectors still require full catalog browsing and auth-backed visibility on the `/models` path, matching the report. The exact current-main wall-clock timings were not rerun in the reporter’s WSL2 configuration.

## Solution Assessment

No complete fix is established yet. Current caching and configured-only fast paths are useful partial mitigations, but the best repair needs a maintainer-approved wildcard completeness and stale-refresh contract shared by all listing surfaces.
