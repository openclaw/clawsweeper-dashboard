## Summary

Keep open: the PR is not merge-ready because the new Claude CLI OAuth matcher depends on `claude-cli` appearing in the display message, but the single-candidate fallback path can rethrow a `FailoverError` whose message is only the raw Claude CLI text and whose provider is stored separately as metadata. The PR also has only test evidence, not real after-fix behavior proof.

## What This Changes

The PR extends OAuth refresh failure classification for Claude CLI 401 invalid-credentials messages, reorders external run failure handling ahead of generic provider-auth copy, and adds regression tests.

## Best Possible Solution

Classify Claude CLI invalid-credentials 401s using structured provider/status metadata or by promoting the CLI failure at the runner boundary, keep the shared OAuth reply precedence, add a raw-message regression, and then provide real behavior proof.

## Reproduction Assessment

Yes, at source level: current main routes a typed Claude CLI 401 auth failure through generic provider-auth handling before the OAuth reauth hint, and the PR still misses the raw single-candidate FailoverError shape. I did not run a live expired-token Claude CLI session.

## Solution Assessment

No: the shared classifier/reply-precedence location is reasonable, but this implementation is too narrow because it keys on the provider id being present in the message instead of using structured `FailoverError.provider` metadata.
