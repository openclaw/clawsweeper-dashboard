## Summary

Keep this PR open: the underlying log problem is real, but the proposed wording is still misleading for valid WhatsApp configurations where group messages can route without an explicit `groups` map.

## What This Changes

The PR adds a WhatsApp startup-log helper and tests so the inbound listener log reports DM-only, configured group count, or wildcard group routing based on `account.groups`.

## Best Possible Solution

Compute the startup log from the effective resolved group scope, including `groupPolicy`, `groupAllowFrom`, and wildcard/explicit `groups`, then back it with focused unit cases and redacted real startup-log proof.

## Reproduction Assessment

Yes, source-reproducible: current main always emits the static startup log, while the resolver can allow group routing without `account.groups`. I did not run a live WhatsApp Web bridge in this read-only review.

## Solution Assessment

No, not as proposed: counting only `account.groups` is too narrow for the current WhatsApp group-routing contract. The maintainable fix should log the effective group scope from the same resolved account/policy facts used by routing.
