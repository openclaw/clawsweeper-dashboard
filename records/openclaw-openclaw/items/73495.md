## Summary

Keep open. The PR is the right narrow shape for the Telegram doc/runtime mismatch, and the contributor already addressed earlier sanitizer and per-account merge feedback, but the latest scanner still has two small parity gaps against current Telegram runtime config resolution.

## What This Changes

This PR adds Telegram and streaming docs notes, a changelog entry, a new `openclaw doctor` warning scanner with tests for Telegram `replyToMode` versus tool-progress preview conflicts, and removes stale `await`s in active-memory tests.

## Best Possible Solution

Land the PR after tightening the scanner to share or faithfully mirror Telegram runtime streaming-mode and account-selection semantics, while keeping the runtime exclusion unchanged and using docs plus `openclaw doctor` as the user-facing guardrail for #73487.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a Telegram config with `replyToMode` not `off`, preview streaming active, and default tool-progress preview; current main creates no answer preview stream in that path, and the PR body includes matching live trace evidence.

## Solution Assessment

Yes, with small scanner fixes. Docs plus a doctor warning are the narrowest maintainable fix for the silent doc/runtime mismatch; a runtime fallback that emits separate progress messages while quote-reply is enabled is a larger product behavior change.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.87

Full review comments:

- **[P3] Mirror legacy streaming-off aliases before warning:** `src/commands/doctor/shared/telegram-replyto-streaming.ts:49`
  - body: The scanner only suppresses warnings for nested `streaming.mode: "off"`, but the runtime resolver also treats legacy `streamMode: "off"` and boolean `streaming: false` as preview-off. Those configs would get a doctor warning even though Telegram preview edits are already disabled at runtime, so the scanner should use or mirror `resolveChannelPreviewStreamMode`.
  - confidence: 0.89
- **[P3] Evaluate the implicit default account when accounts is empty:** `src/commands/doctor/shared/telegram-replyto-streaming.ts:70-84`
  - body: When `accounts` is `{}`, this branch returns without evaluating `channels.telegram`. Runtime account selection falls back to the implicit `default` account when no account IDs are configured, so a top-level `replyToMode: "first"` conflict would remain silent in exactly the default-account path.
  - confidence: 0.86
