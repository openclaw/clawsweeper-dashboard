## Summary

Keep open. The PR is a valid implementation candidate for the Telegram doc/runtime mismatch, but the latest scanner still diverges from runtime config resolution in two narrow edge cases and the PR now needs rebasing before merge.

## What This Changes

Adds Telegram and streaming docs notes, a changelog entry, a doctor warning with tests for Telegram replyToMode/tool-progress conflicts, and a small active-memory test cleanup.

## Best Possible Solution

Land the PR after the doctor scanner uses the same preview-mode and account-selection semantics as Telegram runtime, while keeping the runtime exclusion unchanged.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a Telegram config with replyToMode not off, preview streaming active, and tool-progress preview true; current main then creates no answer preview stream and drops progress updates.

## Solution Assessment

Yes, with the scanner parity fixes. Docs plus an openclaw doctor warning are the narrowest maintainable fix for the silent mismatch; a runtime fallback would be a larger delivery behavior change.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.87

Full review comments:

- **[P3] Mirror preview-mode aliases before warning:** `src/commands/doctor/shared/telegram-replyto-streaming.ts:49`
  - body: The scanner suppresses warnings only for nested `streaming.mode: "off"`, but runtime preview resolution also treats legacy `streamMode: "off"` and boolean `streaming: false` as preview-off. Those configs would get a doctor warning even though Telegram preview edits are already disabled, so this should call or faithfully mirror `resolveChannelPreviewStreamMode`.
  - confidence: 0.9
- **[P3] Evaluate the implicit default account for empty accounts:** `src/commands/doctor/shared/telegram-replyto-streaming.ts:74-87`
  - body: When `accounts` is `{}`, this branch returns without evaluating the top-level Telegram config. Runtime account selection falls back to the implicit `default` account when no account IDs exist, so a top-level `replyToMode: "first"` conflict would remain silent in that default-account path.
  - confidence: 0.87
