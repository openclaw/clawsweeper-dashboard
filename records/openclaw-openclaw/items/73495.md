## What This Changes

Adds Telegram and streaming docs notes, a changelog entry, and a doctor warning with tests for Telegram `replyToMode` versus tool-progress preview, plus two active-memory test cleanup edits.

## Best Possible Solution

Land a Telegram-owned doctor warning and docs note that use the existing channel doctor adapter plus Telegram account helpers, while leaving the runtime quote-reply behavior unchanged.

## Reproduction Assessment

Yes. Source inspection shows a Telegram turn with selected/native quote context disables the answer draft stream, and tool-progress preview is then gated off by `Boolean(answerLane.stream)`; the PR body also includes matching live trace evidence.

## Solution Assessment

No, not as currently implemented. Docs plus a doctor warning are the right narrow solution, but the scanner should live in the Telegram doctor adapter and use runtime account helpers so warnings match enabled-channel and account/default semantics.

- **[P2] Route this warning through the Telegram doctor adapter:** `src/commands/doctor/shared/preview-warnings.ts:146-160`
  - body: This direct import runs whenever any channel config exists, so a config with `channels.telegram.enabled: false` can still emit the new replyToMode/toolProgress warning even though Telegram is not active. `collectChannelDoctorPreviewWarnings` already applies channel doctor selection and filtering; put this scanner behind `telegramDoctor.collectPreviewWarnings` instead.
  - confidence: 0.86
- **[P3] Check the implicit default account when accounts is empty:** `src/commands/doctor/shared/telegram-replyto-streaming.ts:69-81`
  - body: When `accounts` is `{}`, `hasRecord(accounts)` is true, the loop has no entries, and the function returns before evaluating the top-level Telegram config. Runtime account selection falls back to the implicit `default` account in that case, so `replyToMode: "first"` can still suppress previews while doctor stays silent.
  - confidence: 0.9
