## Summary

Keep this PR open. Current main still has the Telegram quote-reply/tool-progress doc/runtime mismatch and no doctor warning, and this PR is a focused implementation candidate, but the latest head still has two small doctor-scanner parity bugs around empty `accounts` and legacy stream-mode aliases.

## What This Changes

The PR adds Telegram replyToMode/tool-progress conflict docs, a new `openclaw doctor` warning scanner with tests, a changelog entry, and removes two stale awaits in active-memory tests.

## Best Possible Solution

Land the PR after making the new doctor scanner mirror Telegram runtime resolution for account fallback and stream-mode aliases, keeping runtime delivery unchanged and documenting the existing trade-off. Any separate-message fallback for quote-reply mode should remain a separate product follow-up.

## Reproduction Assessment

Yes. The source-level reproduction is a Telegram config with `replyToMode` not `off`, active preview streaming, and default `streaming.preview.toolProgress`; current main shows the answer lane stream is not created when native quote reply is active, and the PR body includes live trace evidence for that path.

## Solution Assessment

Yes, the docs plus doctor warning approach is the narrowest maintainable fix for the silent doc/runtime mismatch. The implementation should be tightened to reuse or mirror Telegram's runtime account and stream-mode resolution so the warning neither misses empty-account fallback configs nor fires for legacy preview-off aliases.
