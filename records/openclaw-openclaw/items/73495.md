## Summary

Keep open. The paired bug is real, but this PR currently overstates the runtime conflict and adds a Telegram-specific core doctor scan that misses existing channel-doctor gating and compatibility semantics.

## What This Changes

The PR adds docs and changelog entries plus an `openclaw doctor` warning and tests for Telegram `replyToMode` versus `streaming.preview.toolProgress`, with a small active-memory test cleanup.

## Best Possible Solution

Keep the docs/doctor guardrail, but implement it through Telegram’s channel doctor adapter and scope the warning text/tests to current runtime behavior while mirroring preview-mode and account-selection semantics.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a Telegram turn with `replyToMode` not `off`, active preview streaming, default tool progress, and a `ReplyToIsQuote`/quote-text context; current main then creates no answer stream, so progress preview cannot render.

## Solution Assessment

No, not as written. Docs plus doctor are the right narrow shape, but the PR should not claim a blanket `replyToMode` conflict and should reuse the Telegram/channel doctor path instead of a core hardcoded scanner.

Full review comments:

- **[P2] Scope the warning to quote-reply turns:** `src/commands/doctor/shared/telegram-replyto-streaming.ts:44-52`
  - body: Current main only disables the answer preview stream when `replyToMode !== "off"` and `replyQuoteText != null`; plain Telegram turns with `replyToMode: "first"` still render tool-progress preview, and the existing test suite covers that path. This scanner treats every non-off `replyToMode` as a conflict, so `openclaw doctor` and the docs would tell users that progress cannot render for ordinary non-quote Telegram messages even though it can.
  - confidence: 0.91
- **[P2] Use the channel doctor adapter for Telegram warnings:** `src/commands/doctor/shared/preview-warnings.ts:149-153`
  - body: This new core-level import bypasses `collectChannelDoctorPreviewWarnings`, which already respects disabled channels/plugins and loads the Telegram doctor adapter. With `channels.telegram.enabled: false` or `plugins.entries.telegram.enabled: false`, the runtime will not run Telegram but this unconditional scanner can still emit a Telegram warning; move the check into `extensions/telegram/src/doctor.ts` or route it through the existing adapter path.
  - confidence: 0.88
- **[P3] Mirror legacy preview-off aliases before warning:** `src/commands/doctor/shared/telegram-replyto-streaming.ts:49`
  - body: The scanner suppresses conflicts only for nested `streaming.mode: "off"`, but Telegram runtime resolves preview mode through `resolveChannelPreviewStreamMode`, which also treats legacy `streamMode: "off"` and boolean `streaming: false` as preview-off. Those configs would get a warning even though preview edits are already disabled at runtime.
  - confidence: 0.89
- **[P3] Evaluate empty accounts as the default account:** `src/commands/doctor/shared/telegram-replyto-streaming.ts:73-87`
  - body: When `accounts` is `{}`, this branch returns after scanning zero entries. Runtime account selection falls back to the implicit `default` account when no account IDs exist, so top-level Telegram settings can still drive an active account and should be evaluated instead of skipped.
  - confidence: 0.87
