## Summary

Keep open: the PR is a useful attempt at the linked agent file-path bug, but the guard is wired into guarded/memory paths instead of the shared read/write/edit tool-parameter boundary, so default filesystem-tool execution can still pass hallucinated `.docodex` paths through unchanged.

## What This Changes

The branch adds a `validateToolFilePathExtension` helper in `agent-tools.read.ts`, calls it from workspace path guarding and memory path resolution, adds helper tests, and fixes one MCP capture test fixture.

## Best Possible Solution

Put exact known hallucinated document-extension handling at the shared agent path-param boundary for read/write/edit, keep free-form exec command text untouched, and cover both default `workspaceOnly=false` and guarded `workspaceOnly=true` paths.

## Reproduction Assessment

Yes for the PR defect: current source shows the new validation only runs through workspace guard or memory path resolution, while default read/write/edit tools bypass those wrappers. I did not run tests because this review is read-only.

## Solution Assessment

No. The PR is a plausible mitigation, but it is not the best fix because the guard belongs in shared path-param normalization for the structured file tools, with an exact known-pattern map instead of a broad substring denylist.
