## Summary

Keep open: current main and v2026.6.9 externalize Groq and provide doctor/update repair, but the reported raw npm upgrade plus Gateway/runtime path can still miss the external media plugin and fail with the generic provider-not-available error.

## What This Changes

The issue requests upgrade/runtime recovery or actionable diagnostics when a configured Groq audio media provider now requires the external `@openclaw/groq-provider` plugin.

## Best Possible Solution

Keep Groq externalized, but add a generic missing-official-media-provider diagnostic or repair path that uses the official catalog and prevents empty voice transcripts from being treated as valid input.

## Reproduction Assessment

Yes, source-reproducible: current source and v2026.6.9 show Groq excluded from the core package, media providers resolved only from loaded plugin capabilities, and configured Groq audio reaching the reported missing-provider throw when the plugin is absent.

## Solution Assessment

## Repair Work Prompt

Fix issue #95658 generically for externalized media-understanding providers. Preserve Groq as an external plugin. When `tools.media.audio.models` or shared media config references a provider whose official external plugin is missing or disabled, surface an actionable install/doctor hint such as `openclaw plugins install @openclaw/groq-provider` / `openclaw doctor --fix` instead of only `Media provider not available: groq`, and ensure channel audio paths do not treat a missing provider as a valid empty transcript. Prefer official catalog/contract helpers over hardcoded Groq production logic; Groq can be a test fixture. Keep existing doctor/update repair behavior and add focused regression tests for configured Groq audio with the plugin absent.
