## Summary

Keep open: the helper-level change is directionally useful, but the user-facing doctor preview path still gates this module on an explicit disabled-plugin config, so the reported `channels.discord`-only case will not emit the new warning. The PR also needs after-fix real CLI proof, since the body only lists focused tests and checks.

## What This Changes

The PR updates doctor channel-plugin blocker scanning and tests to add a missing-explicit-enablement warning for installed external channel plugins such as Discord.

## Best Possible Solution

Wire the user-facing doctor preview diagnostic to scan configured channel surfaces for untrusted external owners, preserve the explicit trust boundary, and add a preview-level regression plus real CLI proof.

## Reproduction Assessment

Yes, source-reproducible: with a `channels.discord` config and no `plugins.entries`, `collectDoctorPreviewWarnings` does not import the blocker scanner, while the plugin policy still marks an untrusted external owner ineffective.

## Solution Assessment

No: adding the helper hit is not sufficient while the preview caller keeps the old explicit-disable gate. The narrow fix is to relax that caller gate and prove the actual doctor warning output.
