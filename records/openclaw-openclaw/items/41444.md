## Label Justifications

- `P1`: The issue describes an Anthropic OAuth authentication regression that can turn valid provider requests into HTTP 401 failures.
- `impact:auth-provider`: The affected surface is Anthropic OAuth token handling and provider beta-header construction.

## Summary

Keep open. Current main partially moved 1M context away from the retired beta, but the reported direct Anthropic OAuth path can still lose `oauth-2025-04-20` when a model-level `anthropic-beta` header is merged after the seeded OAuth beta; the open closing PR at https://github.com/openclaw/openclaw/pull/41461 remains the implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue reports that Anthropic OAuth Bearer requests can fail with HTTP 401 when `context-1m-2025-08-07` is configured as a model-level `anthropic-beta` header.

## Best Possible Solution

Retarget or replace https://github.com/openclaw/openclaw/pull/41461 with a focused fix in Anthropic header composition that merges model-level beta headers without dropping OAuth-required betas, while preserving custom-endpoint suppression and adding regression coverage for the reported model-header configuration.

## Reproduction Assessment

Yes by source inspection, though not by a live call in this read-only review. A direct Anthropic model using an `sk-ant-oat` token and model-level `headers: { "anthropic-beta": "context-1m-2025-08-07" }` still follows a last-wins merge after the OAuth beta is seeded.

## Solution Assessment

No, not as the currently linked wrapper-only PR shape. The maintainable fix should target the active direct Anthropic transport header merge and keep the documented custom-endpoint beta suppression intact.
