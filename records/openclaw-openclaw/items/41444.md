## Label Justifications

- `P1`: The issue is a source-reproducible auth-provider regression that can make Anthropic OAuth/subscription-token model calls fail with HTTP 401 for affected configurations.
- `impact:auth-provider`: The reported failure is specifically about Anthropic OAuth token handling and provider request header construction.

## Summary

Keep open: current main and v2026.6.10 still have a source-reproducible Anthropic OAuth/header path where OAuth-required betas are added before configured model headers, so a model-level `anthropic-beta` value can replace `oauth-2025-04-20`; the only open closing PR is a conflicting fork PR that needs maintainer retargeting or replacement.

## What This Changes

This issue requests that Anthropic OAuth beta headers be preserved when 1M context is configured through model/provider headers rather than agent extra params.

## Best Possible Solution

Preserve OAuth-required Anthropic betas at the provider-owned header boundary for both extra-param and model-header paths, strip the retired 1M beta where appropriate, and add focused regression coverage for configured `models.providers.*.headers`.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence reproduction path: use an Anthropic OAuth token with configured `models.providers.*.headers["anthropic-beta"]` and no extra-param beta trigger, then the OAuth beta can be omitted or overwritten before the request is sent.

## Solution Assessment

No. Current main fixes the extra-param paths but not the reported model-header path; the best fix should merge provider/model `anthropic-beta` values with OAuth-required betas in one provider-owned header path rather than relying on wrapper activation from extra params.
