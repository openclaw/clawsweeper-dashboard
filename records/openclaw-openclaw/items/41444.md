## Label Justifications

- `P1`: The issue reports a high-confidence Anthropic OAuth regression that can turn valid direct provider requests into HTTP 401 failures.
- `impact:auth-provider`: The affected path controls Anthropic OAuth token routing and required beta headers for provider authentication.

## Summary

Keep open: current main still has a source-reproducible Anthropic OAuth path where model-level `anthropic-beta` headers can replace the seeded OAuth beta header, and the latest release tag shows the same header-order behavior. The existing cross-repo closing PR is open but conflicting and targets the older embedded-runner shape, so this issue should remain the canonical bug until a retargeted fix lands.

## What This Changes

The issue requests preserving Anthropic OAuth-required beta headers when `context-1m-2025-08-07` is configured through model-level provider headers instead of agent extra params.

## Best Possible Solution

Fix the active Anthropic plugin/direct-transport header composition so direct Anthropic OAuth requests preserve `oauth-2025-04-20` while custom endpoint beta suppression remains intact.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: use a direct Anthropic model with an `sk-ant-oat` token and model-level `headers: { "anthropic-beta": "context-1m-2025-08-07" }`; the wrapper does not run from model headers alone and transport header merging is last-wins.

## Solution Assessment

No, the older unconditional wrapper-gate fix is not the best current path. The fix should be retargeted to the active Anthropic plugin/direct transport header composition and preserve the documented custom-endpoint suppression behavior.
