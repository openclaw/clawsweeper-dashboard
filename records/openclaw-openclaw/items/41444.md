## Summary

Keep open. Current main still has the reported Anthropic OAuth/model-level beta-header gap in source, and the provided GitHub context shows an open same-author implementation PR with explicit closing intent, so conservative cleanup should wait for that PR to be reviewed, replaced, landed, or closed.

## What This Changes

Ensure Anthropic OAuth tokens still send the required `oauth-2025-04-20` beta when model-level `anthropic-beta` headers are configured, avoiding 401s from overwritten OAuth beta headers.

## Best Possible Solution

Keep the issue open and review, update, land, replace, or close PR #41461. The eventual code path should merge or protect Anthropic beta tokens at both the provider-wrapper and direct-transport boundaries, preserve `oauth-2025-04-20` for `sk-ant-oat-*` tokens, strip or ignore `context-1m-*` for OAuth per current docs, and add regression coverage for OAuth plus model-level `anthropic-beta`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a direct Anthropic model with `headers: { "anthropic-beta": "context-1m-2025-08-07" }` and use a `sk-ant-oat-*` token; current main builds OAuth beta defaults, then lets model headers overwrite them in the direct transport, while the plugin wrapper is not composed when no extra-param beta exists.

## Solution Assessment

No for the narrow always-apply-wrapper idea by itself. That can address the plugin wrapper path, but current main also has a direct transport merge path where model-level `anthropic-beta` overwrites `oauth-2025-04-20`, so the safer fix belongs at the actual beta-header merge boundary too.
