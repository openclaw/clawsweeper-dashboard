- `P1`: The issue describes an Anthropic OAuth authentication regression that can turn valid provider requests into HTTP 401 failures for affected users.
- `impact:auth-provider`: The affected behavior is Anthropic OAuth token handling and provider beta-header construction.

## Summary

Keep open. Current main still has a source-reproducible Anthropic OAuth beta-header overwrite path, and the open closing PR is still the implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue asks OpenClaw to preserve Anthropic's OAuth-required beta header when an OAuth token is used and `anthropic-beta` is supplied through model-level headers.

## Best Possible Solution

Review and either retarget or replace https://github.com/openclaw/openclaw/pull/41461 with a fix in the active Anthropic plugin/direct transport header composition that preserves direct Anthropic OAuth betas without adding implicit betas to custom endpoints.

## Reproduction Assessment

Yes. A direct Anthropic model using an `sk-ant-oat` token and model-level `headers: { "anthropic-beta": "context-1m-2025-08-07" }` follows current source paths where the OAuth beta wrapper is skipped or overwritten before dispatch.

## Solution Assessment

No, not as currently represented by the linked PR shape. The right fix should target the active Anthropic plugin/direct transport header composition and keep documented custom endpoint beta suppression intact.
