## Label Justifications

- `P1`: The report describes likely cross-user context injection in real multi-user memory deployments, but it is not a P0 security boundary bypass under the documented shared-gateway trust model.
- `impact:security`: The issue is about private remembered content being surfaced to the wrong channel participant, even though the formal security boundary is limited by policy.
- `impact:session-state`: The affected behavior is memory/session context selection and injection into a later model prompt.

## Summary

Keep open. Current `memory-lancedb` source shows long-term memory records and semantic recall are global to the plugin database, so the reported class is real enough to track, but OpenClaw's documented trust model does not support treating this as a P0 shared-gateway security boundary bypass.

## What This Changes

The issue requests sender-scoped semantic memory recall so memories captured for one Feishu sender are not injected into another sender's conversation.

## Best Possible Solution

Define generic memory scope metadata for `memory-lancedb` rows from runtime session/conversation identity, filter recall by requester scope, and document or automate how existing unscoped rows should be cleared, migrated, or treated.

## Reproduction Assessment

Yes for a focused source-level repro: populate `memory-lancedb` with one sender's memory, then recall from another session and the plugin searches the same table without any sender or session predicate. No live Feishu production reproduction was run in this read-only pass.

## Solution Assessment

No, a Feishu-only `sender_id` filter is too narrow as the permanent fix. The maintainable path is a channel-agnostic scope model using session/conversation/sender metadata plus a clear policy for legacy unscoped memories.
