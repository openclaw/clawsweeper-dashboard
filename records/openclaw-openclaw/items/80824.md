## Summary

Keep open: this is a useful active bug-fix PR and the OAuth-manager/Codex auth-bridge direction fits the current code boundary, but latest head still retains token-bearing OAuth credential JSON in the in-memory failure fingerprint.

## What This Changes

The PR adds an in-memory terminal OAuth refresh-failure cache keyed by credential state, threads forceOAuthRefresh through auth-profile resolution, removes Codex app-server expiry mutation on forced refresh, and adds focused regression tests.

## Best Possible Solution

Land the OAuth-manager cache and explicit forceRefresh path after replacing the raw credential fingerprint with a non-secret digest and letting normal CI/security gates pass.

## Reproduction Assessment

Yes, at source level: current main refreshes an unusable OAuth credential through one adapter.refreshCredential call on each resolution and has no terminal failure cache, while the Codex app-server forced-refresh path persists expires: 0 before resolving. The PR body also includes copied after-fix runtime output for the intended cache and forced-refresh behavior.

## Solution Assessment

Mostly yes: putting terminal failure caching in the OAuth manager and threading forceRefresh through the existing Codex auth bridge is the narrow maintainable fix. The safer version should store a deterministic non-secret digest instead of the raw serialized credential snapshot.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.84

Full review comments:

- **[P3] Avoid retaining OAuth secrets in failure fingerprints:** `src/agents/auth-profiles/oauth-manager.ts:269-272`
  - body: `refreshFailureCredentialFingerprint` serializes the full OAuth credential, and `rememberRefreshFailure` keeps that string in `cachedRefreshFailures`. Since OAuth credentials can include `access`, `refresh`, and `idToken` values, store a deterministic digest or other non-secret fingerprint so the cache does not retain token-bearing JSON after credentials change.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff has no dependency or workflow expansion, but the new in-memory credential fingerprint should not retain raw OAuth token material.

Concerns:

- **[low] Raw OAuth tokens retained in cache fingerprint:** `src/agents/auth-profiles/oauth-manager.ts:269`
  - body: The cache fingerprint is produced from `Object.entries(credential)` and then stored in `cachedRefreshFailures`, so token-bearing fields can remain in process memory longer than necessary. Hash or otherwise digest the canonical snapshot before caching it.
  - confidence: 0.78
