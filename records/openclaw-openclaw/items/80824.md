## Summary

Keep this PR open for normal review: the functional direction matches the current OAuth refresh path, but the external contribution still needs real behavior proof and one small secret-minimization fix before merge.

## What This Changes

The PR adds an OAuth manager terminal refresh-failure cache keyed by credential state, threads forced OAuth refresh through Codex app-server auth, and adds regression tests for cached failures and forced-refresh errors.

## Best Possible Solution

Land the OAuth-manager-level cache and explicit forced-refresh path after hashing the credential fingerprint and adding real setup proof that unchanged terminal failures stop retrying while changed credential metadata retries.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection of current main shows the expired/forced OAuth paths can call refresh again on each resolution without a terminal failure cache, and the PR adds targeted regression coverage for that path.

## Solution Assessment

Mostly yes: putting the cache in the OAuth manager and threading forceRefresh through the existing Codex auth bridge is the narrow owner-boundary fix. The safer version should hash the credential fingerprint and still provide real after-fix behavior proof before merge.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Avoid retaining OAuth secrets in failure fingerprints:** `src/agents/auth-profiles/oauth-manager.ts:269-272`
  - body: The fingerprint serializes the full OAuth credential object, including `access`, `refresh`, and possible `idToken` values, then stores that string in the long-lived failure cache. Please hash the canonical credential snapshot or store a non-secret digest so a terminal failure cache does not retain token-bearing strings after credentials change.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff has no dependency, workflow, or persistence expansion, but the in-memory failure fingerprint should not retain raw OAuth token material.

Concerns:

- **[low] Raw OAuth tokens retained in cache fingerprint:** `src/agents/auth-profiles/oauth-manager.ts:269`
  - body: `refreshFailureCredentialFingerprint` serializes the whole OAuth credential before storing it in `cachedRefreshFailures`, so access/refresh token strings can remain in process memory longer than necessary. Hash the canonicalized snapshot before caching it.
