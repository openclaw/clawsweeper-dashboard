## Summary

Keep open: current main still materializes configured providers into each generated models.json plan and merge mode preserves existing provider entries without an agent boundary, so the reported credential-boundary concern remains plausible and security-sensitive.

## What This Changes

The issue asks to scope generated per-agent models.json provider entries so an agent cannot read sibling agents' credential-bearing providers while retaining bridge cross-validation.

## Best Possible Solution

Define an owner-reviewed per-agent credential materialization rule that removes sibling credential-bearing providers from generated models.json while preserving safe shared catalog metadata, SecretRef marker behavior, and documented merge semantics.

## Reproduction Assessment

Yes, at source level: a config or existing models.json with multiple credential-bearing provider entries flows through planning and merge mode without an agent boundary. I did not run a live saqi bridge setup, so this is not a live reproduction.

## Solution Assessment

Unclear: per-agent credential isolation is the right security direction, but the safest fix should scope credential-bearing provider material rather than blindly dropping every non-own provider entry.
