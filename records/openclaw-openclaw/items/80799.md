- `impact:security`: The issue is about bearer/provider credentials becoming readable from another agent's generated config.

## Summary

Keep open: current `main` fixed the related plaintext SecretRef persistence bug, but the central per-agent provider scoping request is still not implemented. The current `models.json` planner still builds from the full provider map and merge mode can preserve already-present sibling provider entries, so this needs maintainer/security design before a fix.

## What This Changes

The issue asks for each agent's generated `models.json` to expose only that agent's provider entry instead of sibling providers and bearer keys.

## Best Possible Solution

Add a maintainer-approved per-agent provider ownership rule for generated `models.json`, then cover both new generation and cleanup of existing sibling entries without weakening the SecretRef marker contract.

## Reproduction Assessment

Yes from source, with medium confidence: current generation writes the root provider map and merge mode preserves self-contained existing provider entries. I did not run a live filesystem reproduction because this review is read-only and the reported path can write agent state.

## Solution Assessment

No, this is not solved yet; the related merged fix addresses plaintext SecretRef persistence, not scoping each agent's generated provider entries to its own provider. The best fix needs a maintainer decision on the canonical ownership signal before code changes.
