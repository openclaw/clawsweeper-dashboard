## Label Justifications

- `P1`: The issue reports a broken Browser `act` workflow affecting evaluate, click, wait, select, fill, and drag operations after gateway restart.
- `impact:session-state`: The failure is caused by browser tab target reference drift between route selection and action dispatch.

## Summary

Keep open. Current `main` and latest release `v2026.6.10` still contain the raw Browser `/act` targetId mismatch checks, while https://github.com/openclaw/openclaw/pull/96178 is the open linked candidate fix and should close this issue only after it merges.

## What This Changes

The issue reports Browser `act` calls failing with `ACT_TARGET_ID_MISMATCH` after gateway restart when callers reuse the returned tab target reference.

## Best Possible Solution

Land the narrow route-boundary canonicalization in https://github.com/openclaw/openclaw/pull/96178 after maintainer review, then close this issue when the fix reaches `main`.

## Reproduction Assessment

Yes. Source inspection shows current `main` accepts tab aliases during selection but then compares or dispatches raw `action.targetId`; the linked PR also includes before/after real Chrome/CDP proof, though I did not run a live browser repro in this read-only review.

## Solution Assessment

Yes. Canonicalizing accepted aliases at the `/act` route boundary is the narrow maintainable solution because the lower Playwright page lookup should continue receiving exact canonical target ids.
