## Summary

This is a valid current-main browser bug and should stay open because https://github.com/openclaw/openclaw/pull/96178 is the open closing fix candidate, not a merged implementation. Current main and the latest release still compare raw action target ids against the resolved canonical tab id, while the documented contract tells callers to reuse alias-style tab handles.

## What This Changes

The issue reports that Browser `act` requests fail with `ACT_TARGET_ID_MISMATCH` after gateway restart when callers pass the returned tab target reference.

## Best Possible Solution

Review and land or reject https://github.com/openclaw/openclaw/pull/96178; once a fix reaches `main`, close this issue with the merged commit and release provenance.

## Reproduction Assessment

Yes from source inspection, though not from a live Windows run in this review. Current main resolves the route tab through alias-aware selection, then rejects or dispatches the raw `action.targetId` instead of the canonical tab id, matching the reported 403 behavior.

## Solution Assessment

Yes, the linked PR's route-boundary canonicalization is the narrow maintainable fix because the route owns accepted tab aliases and the Playwright executor expects exact canonical target ids. The issue itself should remain open until that PR merges or maintainers choose a different fix.
