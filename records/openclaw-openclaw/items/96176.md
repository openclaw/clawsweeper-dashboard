## What This Changes

The issue reports that browser `act` operations fail with `action targetId must match request targetId` after `openclaw gateway restart` in v2026.6.9.

## Best Possible Solution

Repair the browser act route/tab identity boundary so request target and action target are compared after resolving supported tab handles to the same canonical raw target, while preserving the cross-tab mismatch guard.

## Reproduction Assessment

Yes from source for the guarded failure shape: `/tabs/open` can expose one raw target id, later tab listing can remap the stable alias to another live raw target, and `/act` then compares the original action id against the resolved tab id. I did not run the live Windows gateway restart path in this read-only sweep.

## Solution Assessment

Yes: the narrowest maintainable fix belongs in the browser route/tab identity boundary, not in agent-side retry policy. The fix should keep rejecting true cross-tab action overrides.

## Repair Work Prompt

Fix browser `act` targetId mismatch after gateway restart without weakening cross-tab safety. Cover the source case where `/tabs/open` returns a raw target id, later tab listing remaps the stable alias to a different live raw target, and `/act` rejects because the action payload still contains the old raw id. Compare the route-selected tab and action target after resolving supported handles (`suggestedTargetId`, `tabId`, label, raw targetId/prefix) to the same canonical raw id, and preserve `ACT_TARGET_ID_MISMATCH` for a truly different tab or nested batch override. Keep the change inside the browser plugin route/tab identity boundary; do not add config, broad fallback behavior, or agent-side retry policy. Related context: https://github.com/openclaw/openclaw/issues/78602 and adjacent non-fix PR https://github.com/openclaw/openclaw/pull/89086. No CHANGELOG.md edit.
