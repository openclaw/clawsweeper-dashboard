## Label Justifications

- `P1`: The PR addresses prompt-visible provider credential persistence, a high-priority security/auth workflow problem for real agent setups.
- `merge-risk: 🚨 compatibility`: Generated provider config changes from standalone plaintext `apiKey` values to an OpenClaw-managed marker for source-owned credentials.
- `merge-risk: 🚨 auth-provider`: If runtime auth storage does not supply the real credential, custom provider requests can send the marker as the provider API key or fail auth.
- `merge-risk: 🚨 security-boundary`: The diff changes where provider credentials live and how generated model catalogs distinguish secret values from non-secret auth markers.

## Summary

Keep open for maintainer review: current main still has the prompt-visible plaintext `models.json` serialization path, but this PR is not merge-ready because the new marker can become a literal provider credential on outbound requests and the branch is conflicting with current main's plugin-catalog sidecar work.

## What This Changes

The PR adds an `openclaw-managed` non-secret auth marker, sanitizes generated `models.json` provider `apiKey` values for source-managed credentials, and adds planner/registry regression tests.

## Best Possible Solution

Land one maintainer-selected Layer 1 implementation that removes prompt-visible plaintext from generated `models.json`, moves or reads source-owned plaintext credentials from a runtime-only auth path before writing any marker, and rebases cleanly over current plugin catalog sidecar behavior.

## Solution Assessment

No, not yet. Replacing plaintext with a marker is a plausible redaction shape, but the branch also needs runtime auth resolution or migration proof so the marker is never used as the actual provider credential.
