## Summary

Keep open: the latest head appears to address the earlier sanitizer gap and I found no blocking correctness finding, but this is still a security-sensitive auth contract change that needs maintainer acceptance before merge.

## What This Changes

The branch adds an `openclaw-managed` non-secret marker, sanitizes source-managed plaintext provider `apiKey` values from generated `models.json`, preserves models.json-only plaintext keys, and adds regression plus registry coverage.

## Best Possible Solution

Land one canonical Layer 1 fix that keeps pi registry compatibility, removes source-managed plaintext from generated `models.json`, and preserves models.json-only credentials unless maintainers explicitly choose a stricter migration.

## Reproduction Assessment

Yes. Source inspection shows current `main` serializes `finalProviders` directly into `models.json`, and the linked roadmap plus PR tests identify the plaintext provider-key persistence path; I did not execute tests because this review is read-only.

## Solution Assessment

Yes, with maintainer acceptance of the contract. The marker approach is narrower than deleting `apiKey`, preserves pi registry validation, and keeps models.json-only credentials, but the generated-config compatibility tradeoff should be explicit before merge.
