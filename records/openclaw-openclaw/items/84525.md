## Summary

Keep open: the PR addresses a real plaintext provider-key persistence bug and has useful proof, but the new sanitizer can still preserve an existing plaintext key for provider IDs that current code normalizes, leaving a supported upgrade path unredacted.

## What This Changes

The PR adds an `openclaw-managed` auth marker and sanitizes generated `models.json` provider `apiKey` values while preserving existing `models.json`-only plaintext keys and adding agent/model registry tests.

## Best Possible Solution

Normalize source-provider ownership before preserving existing plaintext `models.json` keys, add a regression for the whitespace-normalized upgrade case, then land the marker contract if maintainers accept it while keeping https://github.com/openclaw/openclaw/issues/11829 open for the remaining secrets roadmap.

## Reproduction Assessment

Yes. Source inspection shows current main writes `finalProviders` directly to generated `models.json`, and the merge path can carry plaintext provider `apiKey` values into that object.

## Solution Assessment

No, not as written. The marker-based approach is a reasonable Layer 1 fix, but the source ownership check must match the existing normalized provider-key contract before preserving old plaintext keys.
