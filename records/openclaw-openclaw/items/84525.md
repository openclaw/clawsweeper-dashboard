## Summary

Keep open: the security goal is valid, but the patch currently drops a documented credential source and violates the pi-coding-agent models.json contract for custom providers, so it needs a compatibility-safe revision before merge.

## What This Changes

The PR changes model configuration planning to omit plaintext provider apiKey values from generated models.json while keeping marker-based auth references and adds regression tests.

## Best Possible Solution

Keep plaintext out of model prompt/catalog serialization by writing a non-secret OpenClaw-managed marker or migrating credentials to the runtime auth store while preserving models.json validity and documented upgrade behavior.

## Reproduction Assessment

Yes, from source inspection: current main serializes finalProviders into models.json, and the PR removes plaintext apiKey values after merge. The merge blocker is also source-reproducible from pi-coding-agent 0.75.1 requiring apiKey for custom provider model entries.

## Solution Assessment

No. Stripping the field after merge is too blunt because models.json is both a documented credential source and an upstream registry contract; the safer fix keeps a non-secret auth marker or performs a credential migration with upgrade proof.
