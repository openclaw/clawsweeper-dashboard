## Summary

Keep open: this is useful security work, but the PR is not safe to merge as submitted because the new marker can become the literal request credential for config-only custom providers, and the branch is currently conflicting with current main's plugin catalog planner.

## Best Possible Solution

Land one canonical, rebased Layer 1 fix that writes a non-secret marker only when runtime request auth can resolve the source credential, preserves models.json-only keys, and keeps current plugin catalog sidecar behavior.

No, not as submitted. The marker approach is plausible, but it needs a runtime resolver or request-auth migration proof so source-config plaintext custom providers do not send the marker as the credential.
