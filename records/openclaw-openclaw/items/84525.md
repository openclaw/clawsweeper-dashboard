## Summary

Keep this PR open: it is a plausible Layer 1 fix for prompt-visible plaintext model provider keys, but the branch is currently conflicting against main and the auth-marker contract still needs maintainer selection among several competing implementations.

## What This Changes

The PR adds an `openclaw-managed` non-secret model auth marker, sanitizes generated `models.json` provider `apiKey` values before serialization, and adds planner/registry regression tests.

## Best Possible Solution

Land one maintainer-selected, rebased Layer 1 implementation that removes source-managed plaintext from generated `models.json`, preserves current plugin catalog behavior, and keeps explicit upgrade semantics for existing `models.json`-only credentials.

## Reproduction Assessment

Yes. Source inspection shows current main still serializes root provider configs into generated `models.json` without this sanitizer, and the PR proof exercises the planner plus registry path with a fake key.

## Solution Assessment

Mostly yes, pending maintainer acceptance and rebase. The marker is narrower than deleting `apiKey` because the current registry requires a non-empty key for custom model entries, but the branch is not currently a viable landing path.
