## Summary

Keep open: the generated-catalog secret leak is real and the contributor supplied useful live-output proof, but this patch is not merge-ready because its new marker is written into the same `apiKey` field that current registry code can later use as request auth, and the branch is conflicting with current main.

## What This Changes

The PR adds an `openclaw-managed` non-secret auth marker and sanitization coverage so generated `models.json` avoids source-configured plaintext provider API keys while preserving existing `models.json`-only keys.

## Best Possible Solution

Rework the fix on current main so generated catalog validation metadata stays separate from runtime request auth, sanitizes both root and plugin-owned catalog writes, and has explicit upgrade behavior for existing plaintext-only `models.json` credentials.

## Reproduction Assessment

Yes. Source inspection shows generated provider `apiKey` values flow through the catalog writer and current registry code consumes `apiKey` as request auth; the PR body also includes copied live output from the planner/discovery path with a fake key.

## Solution Assessment

No. The redaction goal is right, but writing a generic managed marker into the runtime `apiKey` field is not the best fix because current registry code treats that field as request authentication.
