## Label Justifications

- `P1`: The PR addresses prompt-visible plaintext model provider keys, a high-priority credential exposure problem affecting agent workflows.
- `merge-risk: 🚨 compatibility`: Generated provider config would change from plaintext apiKey values to a marker for source-owned credentials.
- `merge-risk: 🚨 auth-provider`: If runtime auth resolution does not pair correctly with the marker, custom provider requests could fail or send a marker as a credential.
- `merge-risk: 🚨 security-boundary`: The diff changes the boundary between persisted credentials and runtime-only credentials.

## Summary

Keep this PR open for maintainer review: it addresses a real credential-exposure gap, but the branch is currently conflicting and the new generated-model auth marker is an auth/security contract decision that should not be auto-cleaned up or merged without maintainer choice.

## What This Changes

The PR adds an `openclaw-managed` non-secret auth marker, sanitizes generated `models.json` provider `apiKey` values, and adds planner plus registry regression coverage.

## Best Possible Solution

Land one maintainer-selected, rebased Layer 1 implementation that redacts source-managed plaintext from both root and plugin-owned generated model catalogs while preserving explicit upgrade semantics for existing models.json-only credentials.

## Reproduction Assessment

Yes from source inspection: current main serializes root provider config into generated models.json without this sanitizer, and the docs describe generated models.json plaintext as agent-readable. I did not execute a current-main failing runtime request in this read-only review.

## Solution Assessment

Mostly yes, pending maintainer acceptance and rebase. The marker approach is narrower than deleting apiKey because the registry currently requires a non-empty key for custom model entries, but the final implementation must be reconciled with current plugin catalog writes and the chosen Layer 1 contract.
