## Summary

Keep this PR open for maintainer review: the security goal is valid, but the current patch still has a P1 auth/upgrade compatibility blocker and the new proof remains test-only rather than real runtime behavior.

## What This Changes

The PR adds a models.json provider sanitizer that drops plaintext apiKey values before serialization and adds regression tests for plaintext config, existing models.json merges, and marker preservation.

## Best Possible Solution

Keep plaintext out of prompt/catalog serialization by writing a non-secret auth marker or migrating credentials into the runtime auth store while preserving models.json validity for fresh installs and upgrades.

## Reproduction Assessment

Yes, source-level: current main serializes finalProviders into models.json, and the PR removes plaintext apiKey values after merge. The blocker is also source-reproducible from pi-coding-agent 0.75.3 requiring apiKey for non-built-in custom providers with model rows.

## Solution Assessment

No. Stripping the field after merge is too blunt because models.json is both a documented credential source and an upstream registry contract; a safer fix uses a non-secret marker or credential migration with upgrade proof.
