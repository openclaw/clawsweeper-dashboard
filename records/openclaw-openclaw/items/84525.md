## Label Justifications

- `P1`: The PR addresses plaintext model API keys being persisted into generated agent config, which is an urgent auth/security boundary problem.
- `merge-risk: 🚨 compatibility`: Changing generated source-managed `models.json` entries from plaintext keys to a marker can affect direct consumers that expected the file to be self-contained.

- `merge-risk: 🚨 security-boundary`: The sanitizer is meant to protect credentials but currently misses a supported normalized provider-key path.

## Summary

Keep open: this is a useful security fix for generated `models.json`, but the current patch still has one source-ownership gap that can preserve a plaintext key in a supported normalized-provider-key path.

## What This Changes

The PR adds an `openclaw-managed` non-secret auth marker, sanitizes source-managed plaintext provider `apiKey` values before writing generated `models.json`, preserves models.json-only plaintext keys, and adds focused model-config and pi registry tests.

## Best Possible Solution

Normalize source-provider ownership in the sanitizer, add the merge-mode normalized-key regression, then land the marker-based redaction if maintainers accept that generated source-managed `models.json` entries are no longer standalone secret-bearing configs while https://github.com/openclaw/openclaw/issues/11829 continues tracking the broader secrets roadmap.

## Reproduction Assessment

Yes, with source inspection rather than a local executed repro: current main serializes `finalProviders` directly into `models.json`, and the merge path can carry plaintext provider `apiKey` values into that object. The PR body also supplies after-fix terminal proof for the planner and pi registry boundary.

## Solution Assessment

No, not as currently patched. The marker-based direction fits the existing non-secret marker pattern, but the sanitizer should reuse normalized source-provider ownership before preserving existing plaintext keys.
