Keep open. ## What This Changes

The PR adds a Bedrock context-window literal-default provenance flag through discovery, model resolution, runtime guard call sites, and focused tests.

## Best Possible Solution

Repair and rebase the provenance approach so known Bedrock models keep their real lookup values, unknown literal fallbacks carry explicit provenance, runtime guard warnings fire on that provenance, and cache/type plumbing remains consistent.

## Reproduction Assessment

Yes. Source review is enough to reproduce the blockers: the provided diff reads an undeclared config variable, current ProviderRuntimeModel lacks the proposed field, and current-main guard tests show source=default does not warn by itself.

## Solution Assessment

No, not as submitted. The provenance-flag direction is maintainable and complementary to #65952, but it needs complete type propagation, profile fallback config threading, cache-key provenance, and current-main warning behavior before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Pass discovery config into profile fallback:** `extensions/amazon-bedrock/discovery.ts:293`
  - body: When an inference profile has no matching foundation model, this fallback reads `config?.defaultContextWindow`, but `config` is not defined inside `resolveInferenceProfiles`. TypeScript should reject it, and runtime discovery would throw in the fallback path; thread the discovery config into the helper instead.
  - confidence: 0.98
- **[P1] Add the provenance field to ProviderRuntimeModel:** `src/agents/pi-embedded-runner/run/setup.ts:115`
  - body: The new access on `params.runtimeModel` uses `ProviderRuntimeModel`, but that type does not declare `contextWindowIsLiteralDefault`. The same issue applies to the other runtime call sites, so the shared runtime model type needs the optional field or the changed typecheck lane will fail.
  - confidence: 0.94
- **[P2] Include default provenance in the discovery cache key:** `extensions/amazon-bedrock/discovery.ts:166-168`
  - body: The flag changes when `defaultContextWindow` is omitted versus explicitly set to the same numeric value, but the cache key only contains the resolved number. With refresh caching enabled, a cached implicit-32000 result can be reused for an explicit-32000 call, or the reverse, returning stale provenance until expiry.
  - confidence: 0.9
- **[P2] Add the source-default warning path:** `src/agents/context-window-guard.ts:46-53`
  - body: Against current main, changing the source to `default` does not make the guard warn for a 32000-token fallback; `evaluateContextWindowGuard` still only warns below the computed threshold, and current tests expect default source to pass. Add the source-default warning behavior here or make the dependency on #64901 explicit before this can deliver the stated warning.
  - confidence: 0.88
