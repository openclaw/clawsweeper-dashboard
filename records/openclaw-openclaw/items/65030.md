## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Pass discovery config into profile fallback:** `extensions/amazon-bedrock/discovery.ts:293`
  - body: When an inference profile has no matching foundation model, this new fallback reads `config?.defaultContextWindow`, but `config` is not defined inside `resolveInferenceProfiles`. TypeScript should reject it, and runtime discovery would throw in the fallback path; thread the discovery config into this helper instead.
  - confidence: 0.98
- **[P1] Add the provenance field to ProviderRuntimeModel:** `src/agents/pi-embedded-runner/run/setup.ts:115`
  - body: The new access on `params.runtimeModel` uses `ProviderRuntimeModel`, but that type does not declare `contextWindowIsLiteralDefault`. The same issue applies to the other runtime call sites, so the shared runtime model type needs the optional field or the typecheck lane will fail.
  - confidence: 0.94
- **[P2] Preserve discovered contextWindow when layering inline models:** `src/agents/pi-embedded-runner/model.ts:374`
  - body: The new layering path spreads `...inlineMatch` after `...discoveredModel`; partial inline entries can carry `contextWindow: undefined` and wipe out the discovered context window while preserving the new flag. Restore `contextWindow` from the discovered model when the inline model did not explicitly set it.
  - confidence: 0.9
- **[P2] Include provenance mode in the cache key:** `extensions/amazon-bedrock/discovery.ts:166-168`
  - body: The new flag differs when `defaultContextWindow` is omitted versus explicitly set to the same numeric value, but the cache key still only contains the resolved number. With refresh caching enabled, one provenance mode can reuse the other's cached discovery result until expiry.
  - confidence: 0.9
- **[P2] Warn when literal defaults become the source:** `src/agents/context-window-guard.ts:50-52`
  - body: Marking the fallback value as `source: "default"` does not currently make the guard warn, because `evaluateContextWindowGuard` only warns below the computed threshold and current tests expect default-source values to pass. Add the source-default warning path here or explicitly coordinate with the related warning PR before this can deliver the promised warning.
  - confidence: 0.88
