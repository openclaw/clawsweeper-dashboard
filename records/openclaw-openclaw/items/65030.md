Keep this PR open. Current main still lacks the explicit Bedrock fallback provenance needed for unknown/defaulted context windows, but the branch is not merge-ready because the submitted diff has concrete type, runtime, cache-key, and changelog blockers.

## What This Changes

The branch adds an optional `contextWindowIsLiteralDefault` flag through Bedrock discovery, model resolution, runtime context-window guard call sites, and focused tests.

## Best Possible Solution

Rebase onto current main, keep known Bedrock context windows as model metadata, mark only literal fallback values for unknown/defaulted Bedrock models as default-derived, thread that provenance through shared/runtime model types and guard call sites, and cover explicit defaults, unknown foundation models, unknown inference profiles, and warning behavior in focused tests.

Yes. Static source review reproduces the blockers: the PR diff reads an undeclared `config` variable in `resolveInferenceProfiles`, and current `ProviderRuntimeModel` plus pi-ai `Model` typing do not expose the new runtime field.

## Solution Assessment

No, not as submitted. The provenance-flag direction is plausible and complementary to the known-window lookup, but the branch needs rebase alignment, complete type propagation, cache-key provenance, and changelog coverage before it is the narrow maintainable fix.
