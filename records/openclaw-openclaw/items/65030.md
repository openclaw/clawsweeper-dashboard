## Summary

Keep open: current main fixed known Bedrock context windows through a lookup table, but it still does not carry provenance for unknown literal fallbacks, so this PR is not obsolete. The branch remains unmergeable as submitted because PR-head source still has blocking type/runtime issues, cache/provenance gaps, missing warning coordination, conflicts, and no real behavior proof.

## What This Changes

The PR adds a Bedrock context-window provenance flag and threads it through runtime model resolution so hardcoded discovery fallbacks can be treated as default-sourced values.

## Best Possible Solution

Land a rebased, coordinated provenance-plus-warning path for unknown Bedrock literal fallbacks that preserves known-model lookup behavior and carries provenance through the runtime model contract, cache key, overrides, and guard.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Pass discovery config into profile fallback:** `extensions/amazon-bedrock/discovery.ts:293`
  - body: When an inference profile has no matching foundation model, this fallback reads `config?.defaultContextWindow`, but `config` is not defined inside `resolveInferenceProfiles`. TypeScript should reject this, and the fallback path would throw at runtime; thread the discovery config into this helper instead.
  - confidence: 0.98
- **[P1] Add the provenance field to ProviderRuntimeModel:** `src/agents/pi-embedded-runner/run/setup.ts:115`
  - body: The new access on `params.runtimeModel` uses `ProviderRuntimeModel`, but that type does not declare `contextWindowIsLiteralDefault`. The same issue applies to the other runtime call sites, so the shared runtime model type needs the optional field or the typecheck lane will fail.
  - confidence: 0.95
- **[P2] Preserve discovered context windows when layering inline models:** `src/agents/pi-embedded-runner/model.ts:381`
  - body: The layering path spreads `...inlineMatch` after `...discoveredModel`; partial inline entries can carry `contextWindow: undefined` and wipe out the discovered context window while preserving the new flag. Restore the discovered `contextWindow` when the inline model did not explicitly set it.
  - confidence: 0.9
- **[P2] Include provenance mode in the Bedrock discovery cache key:** `extensions/amazon-bedrock/discovery.ts:168-170`
  - body: The new flag differs when `defaultContextWindow` is omitted versus explicitly set to the same numeric value, but the cache key still only contains the resolved number. With refresh caching enabled, one provenance mode can reuse the other's cached discovery result until expiry.
  - confidence: 0.89
- **[P2] Warn when literal defaults become the source:** `src/agents/context-window-guard.ts:51-53`
  - body: Marking the fallback value as `source: "default"` does not currently make the guard warn, because the guard still only warns when tokens are missing or below threshold. Add the default-source warning path or coordinate this with the related warning PR before this delivers the promised warning.
  - confidence: 0.88
