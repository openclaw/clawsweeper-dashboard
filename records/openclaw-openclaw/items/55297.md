## Summary

Keep open. Current main still has manual update paths for ClawHub skills and tracked plugins, while the existing startup auto-updater is scoped to OpenClaw package installs and only syncs plugins as part of an OpenClaw update run; the requested background updater remains unimplemented and needs maintainer product/security design.

## What This Changes

The issue requests opt-in configuration to automatically update installed ClawHub skills and plugins on gateway start or a configurable interval.

## Best Possible Solution

Ship a maintainer-approved opt-in update policy that reuses existing tracked skill/plugin update helpers but adds explicit operator consent, cadence controls, pinning or allowlist semantics, failure reporting, rollback guidance, audit metadata, and user-facing security documentation before applying third-party code in the background.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification is source/docs based: explicit `skills update` and `plugins update` paths exist, but no startup or interval updater is wired for installed skills/plugins.

## Solution Assessment

Unclear until maintainers decide the policy. A maintainable implementation should build on `updateSkillsFromClawHub`, `updateNpmInstalledPlugins`, and the existing update scheduler, but adding background third-party code updates without the threat-model controls would not be the best solution.
