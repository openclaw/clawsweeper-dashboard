## Summary

Keep open: the QMD timeout bug is real and the contributor supplied useful real behavior proof, but this patch applies the raised QMD timeout as the whole `memory_search` deadline even when the call does not query QMD. The broader overlapping PR at https://github.com/openclaw/openclaw/pull/95347 is open but currently conflicting, so it is not a safe superseding landing path.

## What This Changes

This PR changes memory-core to use QMD `limits.timeoutMs` as the outer `memory_search` deadline, adds a regression test, and updates QMD memory docs.

## Best Possible Solution

Keep the fix scoped to QMD-backed memory search work only, with the existing 15s guard preserved for supplement-only, builtin, and fallback branches.

## Reproduction Assessment

Yes. Current main and v2026.6.9 still use a fixed 15s `memory_search` wrapper while QMD docs/config expose `memory.qmd.limits.timeoutMs`; the contributor also provided redacted before/after QMD runtime logs showing a post-patch search completing after 23922ms.

## Solution Assessment

No. The useful direction is valid, but applying the QMD timeout as one whole-tool deadline is too broad because it also covers supplement-only and fallback work that should retain the existing 15s guard.
