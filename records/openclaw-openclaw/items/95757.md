## Summary

Keep open: current main still has the fixed 15s memory_search wrapper, and this PR is a useful candidate, but the latest head still lets same-call QMD-to-builtin fallback inherit the raised QMD budget and GitHub reports the branch as conflicting.

## What This Changes

The PR changes memory-core to apply memory.qmd.limits.timeoutMs to QMD-backed memory_search work, preserve the 15s guard for non-QMD paths, add regression tests, and update memory docs.

## Best Possible Solution

Keep the QMD timeout only for real QMD search work, keep setup, wiki-only, builtin, and fallback searches on the 15s guard, add same-call fallback coverage, then rebase before merge.

## Reproduction Assessment

Yes. Current main and v2026.6.10 still route memory_search through a fixed 15s wrapper while QMD config exposes a longer timeout; the PR-introduced fallback defect is source-reproducible through FallbackMemoryManager reusing opts.signal for builtin fallback.

## Solution Assessment

No. Scoping the QMD timeout to QMD work is the right direction, but choosing the deadline before manager.search misses same-call QMD-to-builtin fallback; the safer fix is a post-fallback/default 15s budget or a manager-owned timeout design.
