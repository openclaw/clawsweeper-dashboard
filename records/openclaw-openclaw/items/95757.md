## Summary

Keep this PR open: current main and v2026.6.10 still have the fixed 15s memory_search wrapper, and the PR is a useful candidate, but the latest branch is not merge-ready because same-call QMD-to-builtin fallback can still inherit the longer QMD deadline and the PR is currently conflicting.

## What This Changes

This PR makes memory-core use `memory.qmd.limits.timeoutMs` for QMD-backed `memory_search`, preserves the 15s guard for non-QMD phases, adds regression tests, and updates memory docs.

## Best Possible Solution

Keep the QMD timeout only for real QMD search work, keep manager setup, wiki-only, builtin, and every fallback search on the 15s guard, add coverage for same-call QMD-primary-to-builtin fallback, then rebase before merge.

## Reproduction Assessment

Yes. Current source and v2026.6.10 still show a fixed 15s memory_search wrapper while QMD config exposes a longer search timeout; the new fallback defect is source-reproducible because FallbackMemoryManager reuses the same AbortSignal for builtin fallback.

## Solution Assessment

No. Reusing the QMD timeout for real QMD search is the right direction, but this patch chooses the QMD deadline before `manager.search` and misses QMD-primary-to-builtin fallback inside the manager.
