## Summary

Keep open. Current main and the latest release still use the fixed 15s `memory_search` wrapper while QMD already has `memory.qmd.limits.timeoutMs`; this PR is a focused, technically sound candidate with sufficient before/after QMD runtime proof, but it needs maintainer acceptance because it changes how an existing config key affects interactive tool latency.

## What This Changes

The PR makes QMD-backed `memory_search` use `memory.qmd.limits.timeoutMs` only for live QMD search work, keeps setup/wiki/builtin/fallback paths on the 15s guard, bounds QMD CLI cleanup locally, and updates tests/docs.

## Best Possible Solution

Land this focused branch after required checks finish if maintainers accept using the existing QMD timeout as the live QMD search budget, then use the canonical issue to retire or redirect broader overlapping timeout proposals.

## Reproduction Assessment

Yes, source-reproducible. Current main and `v2026.6.11` still use the fixed 15s wrapper while the QMD config exposes `limits.timeoutMs`, and the PR thread includes before/after QMD runtime logs showing a 23922ms search succeeding under a 45000ms timeout.

## Solution Assessment

Yes as a technical fix, with maintainer policy acceptance still required. The latest head keeps non-QMD, setup, wiki-only, and same-call fallback work on the 15s guard and removed the earlier exported `close(timeoutMs)` API expansion.
