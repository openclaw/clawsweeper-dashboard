## Summary

Keep this PR open: the QMD timeout direction is useful and supported by real logs, but the current head is not merge-ready because a phase timeout can still wait on one-shot QMD cleanup before returning, and maintainers still need to accept the longer interactive wait policy.

## What This Changes

The PR makes QMD-backed `memory_search` use `memory.qmd.limits.timeoutMs` for live QMD search while keeping manager setup, wiki supplements, builtin search, and fallback paths on the 15s default guard.

## Best Possible Solution

Keep the longer budget only for live QMD search, keep setup/wiki/builtin/fallback and one-shot cleanup bounded, and document the availability tradeoff before merge.

## Solution Assessment

No. Scoping the longer timeout to live QMD search is the right direction and the current head fixed the earlier fallback defect, but one-shot CLI cleanup can still block a timeout response and needs a bounded cleanup repair before merge.
