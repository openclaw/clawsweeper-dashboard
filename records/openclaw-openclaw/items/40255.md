## Label Justifications

- `P1`: This is a documented heartbeat configuration regression that can break recurring agent workflows for affected users.
- `impact:session-state`: The issue changes the prompt/context state used by scheduled heartbeat turns instead of preserving the configured user prompt.
- `impact:message-loss`: Reporter evidence shows heartbeat-driven standing programs can silently return empty or `HEARTBEAT_OK`, preventing expected proactive messages.

## Summary

Keep open: current main and v2026.6.10 still route configured heartbeat prompts through the unconditional workspace-path hint path, while docs say the configured heartbeat prompt is sent verbatim; the only linked fix candidate is open, dirty, unmerged, and not a safe reason to close this canonical issue.

## What This Changes

The issue asks OpenClaw to stop mutating user-configured `heartbeat.prompt` text and ensure the configured heartbeat prompt is honored during heartbeat runs.

## Best Possible Solution

Keep this issue as the canonical tracker and land a narrow fix that preserves user-authored heartbeat prompts verbatim in the model-visible heartbeat path while keeping workspace-path guidance, HEARTBEAT.md directives, and `[OpenClaw heartbeat poll]` transcript markers separate and intentional.

## Reproduction Assessment

Yes. Source inspection shows a custom `heartbeat.prompt` containing `HEARTBEAT.md` is resolved as the base prompt and then still passed through the unconditional workspace-path hint helper before dispatch; I did not run a live packaged heartbeat in this read-only review.

## Solution Assessment

Yes, the right fix direction is narrow: separate user-authored heartbeat prompt text from generated/internal heartbeat guidance and prove the configured prompt reaches the model-visible body. Current main is not that solved state.
