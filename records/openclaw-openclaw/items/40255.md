## Label Justifications

- `P1`: The issue reports a documented heartbeat configuration regression that can break recurring agent workflows for affected users.
- `impact:session-state`: The bug changes the heartbeat prompt and transcript state used by scheduled agent turns instead of preserving configured user state.
- `impact:message-loss`: Reporter evidence shows heartbeat-driven standing programs can silently return empty or `HEARTBEAT_OK`, preventing expected proactive messages.

## Summary

Keep open: current main and the latest release still route configured heartbeat prompts through the same workspace-path hint path, while related duplicate evidence shows the transcript marker can still become the compiled prompt. This issue remains the canonical tracker; the linked fix candidate is open in a fork and currently conflicting, so it is not a completed fix.

## What This Changes

The issue asks OpenClaw to respect user-configured heartbeat prompts instead of appending or substituting internal heartbeat prompt text.

## Best Possible Solution

Keep this as the canonical tracker and land a narrow fix that preserves user-authored heartbeat prompts as the model-visible prompt, limits workspace-path guidance to default or internally generated prompts, and keeps `[OpenClaw heartbeat poll]` transcript-only with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows a configured heartbeat prompt containing `HEARTBEAT.md` reaches the runner, then current main still passes it through the unconditional workspace-path hint helper; related trajectory evidence shows the transcript marker can also become the compiled prompt.

## Solution Assessment

No, current main is not the solved state. The narrow maintainable fix is to separate user-authored heartbeat prompt text from generated heartbeat guidance and prove the resolved prompt, not the transcript marker, reaches the model-visible body.
