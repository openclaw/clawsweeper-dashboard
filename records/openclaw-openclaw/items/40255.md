## Label Justifications

- `P1`: This is a documented heartbeat configuration regression that can break recurring agent workflows for affected users.
- `impact:session-state`: The bug changes the prompt and transcript state used by scheduled heartbeat turns instead of preserving the configured user prompt.
- `impact:message-loss`: Reporter evidence shows heartbeat-driven standing programs can silently return empty or HEARTBEAT_OK, preventing expected proactive messages.

## Summary

Keep open: current main still does not satisfy the documented heartbeat prompt contract, the latest release carries the same prompt mutation/transcript-marker shape, and the issue is already the canonical tracker for the remaining heartbeat prompt-boundary bug.

## What This Changes

The issue reports that user-configured heartbeat prompts are no longer respected because OpenClaw mutates or replaces the configured prompt during heartbeat runs.

## Best Possible Solution

Keep this as the canonical tracker and land a narrow fix that preserves user-authored heartbeat prompts verbatim/model-visible, limits workspace-path guidance to default or internally generated prompts, and keeps `[OpenClaw heartbeat poll]` transcript-only with focused regression coverage.

## Reproduction Assessment

Yes, at source level. Configure a heartbeat prompt containing `HEARTBEAT.md`: current main resolves that configured text and then still passes it through the unconditional workspace-path hint helper, while related evidence shows the transcript marker can surface as the compiled prompt.

## Solution Assessment

No, current main is not the solved state. The narrow maintainable fix is to separate user-authored heartbeat prompt text from generated heartbeat guidance and prove the resolved prompt, not the transcript marker, reaches the model-visible body.
