- `impact:session-state`: The issue changes the model-visible heartbeat turn and transcript/context behavior away from the configured session prompt.
- `impact:message-loss`: Reporter evidence says heartbeat-driven standing programs return empty/HEARTBEAT_OK instead of producing the expected user-facing alerts.

## Summary

Keep open: current main and v2026.6.6 still violate the documented verbatim `heartbeat.prompt` contract through the unguarded workspace-path hint and the heartbeat transcript placeholder path. The newer report at https://github.com/openclaw/openclaw/issues/93405 was closed as duplicate back to this issue, and the linked fork PR is open, conflicting, and only covers part of the remaining behavior.

## What This Changes

The issue reports a regression where user-configured heartbeat prompts are no longer honored during heartbeat runs.

## Best Possible Solution

Keep this issue as the canonical tracker and land a narrow target-repo fix that sends the resolved configured heartbeat prompt to the model, keeps the poll placeholder transcript-only, and limits workspace-path hints to default or internal prompts.

## Reproduction Assessment

Yes from source inspection and reporter evidence: configure `agents.defaults.heartbeat.prompt` with text mentioning `HEARTBEAT.md`; current code resolves that prompt, appends extra path guidance, and the prompt/transcript split can make `[OpenClaw heartbeat poll]` model-visible. I did not run a live heartbeat in this read-only review.

## Solution Assessment

Yes, the maintainable fix is narrow: separate model-visible heartbeat prompt text from transcript placeholder text, and skip workspace hint injection for user-authored prompts while preserving it for default/internal prompts. The linked fork PR is directionally useful but incomplete for the newer prompt-placeholder evidence and not currently mergeable.
