## Summary

Keep this PR open, but it still needs changes before merge: the CJK dedupe direction is valid and proof is sufficient, while the current patch introduces a destructive empty-token merge for unsupported scripts in the dreaming dedupe path.

## What This Changes

The PR extracts memory-core tokenizer helpers into `extensions/memory-core/src/memory/tokenize.ts`, routes dreaming dedupe through CJK-aware text similarity, and adds focused regression tests.

## Best Possible Solution

Land the CJK dedupe extraction only after the dreaming dedupe path preserves exact-string fallback for empty-token snippets, or after tokenization is broadened with regression coverage for unsupported scripts.

## Reproduction Assessment

Yes. Current main's source and a read-only tokenizer probe show the reported CJK pair scores 0 with the ASCII-only dreaming tokenizer, and the PR diff/source path shows the new empty-token false-merge behavior.

## Solution Assessment

No. Extracting and sharing the CJK-aware tokenizer is the right shape, but routing destructive dreaming dedupe directly through a helper that treats two empty token sets as identical is not safe.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve fallback for empty-token snippets:** `extensions/memory-core/src/memory/tokenize.ts:59-60`
  - body: `tokenize` only emits ASCII and CJK-family tokens, so Cyrillic, Arabic, emoji-only, or punctuation-only snippets can produce empty sets on both sides. Because `dedupeEntries` now calls `textSimilarity`, this returns 1 and drops one same-path memory at normal thresholds; keep the old exact-string fallback for the dreaming dedupe path or broaden tokenization before using this helper there.
