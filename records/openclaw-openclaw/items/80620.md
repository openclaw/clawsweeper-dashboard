## Summary

Keep open: the CJK fix shape is good and the PR includes usable proof, but the patch introduces a real false-dedupe regression for non-tokenized non-CJK snippets by reusing a helper that returns similarity 1 for two empty token sets.

## What This Changes

The PR extracts the memory MMR CJK-aware tokenizer into a shared memory helper and routes dreaming dedupe through it with regression coverage.

## Best Possible Solution

Keep the shared CJK tokenizer extraction, but make the text-level similarity used by dreaming preserve exact-only fallback for empty token sets or add broader Unicode tokenization before merge.

## Reproduction Assessment

Yes, at source level. Current main’s dreaming dedupe drops CJK tokens and exact-matches empty token sets, and the read-only probe shows the PR fixes that CJK path while introducing an empty-token false merge for distinct Cyrillic/punctuation snippets.

## Solution Assessment

No. Reusing the CJK-aware tokenizer is the right direction, but routing dreaming dedupe through a helper that treats all two-empty-token strings as identical is not the safest complete fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve empty-token fallback for dreaming dedupe:** `extensions/memory-core/src/memory/tokenize.ts:59-60`
  - body: `textSimilarity` returns `1` when both token sets are empty. After `dreaming-phases.ts` uses it for same-path dedupe, two distinct snippets in scripts this tokenizer still does not cover, such as Cyrillic or Arabic, or punctuation-only notes, will be treated as duplicates and one can be dropped. The old dreaming logic only returned `1` for empty-token pairs when the normalized full strings matched, so keep that fallback or broaden tokenization before using this helper here.
  - confidence: 0.92
