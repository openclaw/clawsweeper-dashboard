## Summary

Keep open. The PR is aimed at a real memory-core bug and the shared CJK tokenizer extraction is the right direction, but the current diff still makes distinct unsupported-script or empty-token snippets from the same path compare as identical in dreaming dedupe.

## What This Changes

The PR extracts memory-core MMR tokenization into a shared helper, routes dreaming dedupe through it, re-exports the MMR helper API, and adds focused CJK dedupe regression tests.

## Best Possible Solution

Keep the shared CJK tokenization, add a dreaming-specific empty-token exact-match fallback or broader Unicode token coverage, and keep the managed-block leak tracked separately.

## Reproduction Assessment

Yes. Current main source and a focused read-only probe show the CJK dedupe miss, and the same probe shows the PR-head empty-token false merge for unsupported-script snippets.

## Solution Assessment

No, not yet. Extracting the CJK-aware tokenizer is maintainable, but using the shared raw textSimilarity result directly in dreaming dedupe loses current main's safer exact-match fallback for empty-token snippets.

## Review Findings

Add a dreaming-boundary fallback for empty token sets or broaden tokenization before routing this path through the shared helper.
