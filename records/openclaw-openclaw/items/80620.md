## Summary

Keep open: the CJK tokenizer extraction is directionally right and has sufficient terminal proof, but the exact automerge head still introduces a data-loss regression by treating distinct empty-token snippets as duplicates.

## What This Changes

This PR extracts memory-core’s CJK-aware MMR tokenizer into a shared helper, routes dreaming dedupe through it, re-exports the MMR helper API, and adds CJK dedupe regression tests.

## Best Possible Solution

Keep the shared CJK tokenizer extraction, but add a dreaming-specific empty-token exact-match fallback and leave the broader managed-block leak tracked until it is fixed separately.

## Reproduction Assessment

Yes. Current main source shows the CJK dedupe miss, and a read-only source probe shows this PR's empty-token false-merge regression for unsupported-script snippets.

## Solution Assessment

No. Sharing the tokenizer is the right direction, but using MMR's raw empty-set Jaccard result directly in dreaming dedupe is not the narrow safe fix because it drops distinct empty-token snippets.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve exact fallback for empty-token snippet pairs:** `extensions/memory-core/src/memory/tokenize.ts:59-60`
  - body: `dedupeEntries` now calls `textSimilarity`, whose shared Jaccard helper returns `1` when both token sets are empty. Because `tokenize` still emits only ASCII and CJK-family tokens, distinct Arabic, Cyrillic, emoji-only, or punctuation-only snippets from the same path are now treated as duplicates and one is dropped; current main only merged empty-token pairs when the normalized full strings matched exactly. Keep the shared tokenizer, but make dreaming dedupe preserve that exact-match fallback when both sides have no supported tokens.
  - confidence: 0.92
