## Summary

Keep open: the PR addresses a real memory-core bug, but the patch still introduces a blocking empty-token dedupe regression for non-CJK, non-ASCII snippets.

## What This Changes

The PR extracts memory-core's CJK-aware tokenizer into a shared helper, routes dreaming dedupe through it, and adds focused regression tests.

## Best Possible Solution

Keep the shared CJK-aware tokenizer, but use a dedupe-specific similarity wrapper that preserves exact-string fallback when both snippet token sets are empty.

## Reproduction Assessment

Yes. Source inspection shows the PR routes dedupe through a helper that returns 1 for two empty token sets, so two distinct Arabic, Cyrillic, emoji-only, or punctuation-only snippets from the same path will be merged incorrectly.

## Solution Assessment

No. Reusing the CJK-aware tokenizer is the right direction, but directly using the MMR textSimilarity helper changes the dedupe empty-token contract; a snippet-specific wrapper is the safer fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve empty-token fallback in dreaming dedupe:** `extensions/memory-core/src/dreaming-phases.ts:1357`
  - body: This call now uses shared textSimilarity, whose tokenizer only emits ASCII/CJK tokens and whose Jaccard helper returns 1 for two empty token sets. Two distinct Arabic, Cyrillic, emoji-only, or punctuation-only snippets from the same path will therefore score 1 and the later memory will be dropped under the default 0.9 dedupe threshold. Preserve the old exact-string fallback when both snippet token sets are empty.
