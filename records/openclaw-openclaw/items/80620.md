## Summary

Keep open. The shared-tokenizer direction matches the linked bug and the prior ClawSweeper review, but the patch changes dreaming dedupe so distinct same-path snippets in unsupported scripts can be treated as identical and dropped.

## What This Changes

The PR extracts memory-core's MMR tokenizer into a shared helper, routes dreaming dedupe through it, re-exports the MMR helper API, and adds CJK dedupe regression tests.

## Best Possible Solution

Keep the shared CJK tokenizer, but make dreaming dedupe preserve the old exact-text fallback when both snippets have no supported tokens, then leave or create separate tracking for the managed-block promotion leak.

## Reproduction Assessment

Yes. Current main source shows the CJK dedupe miss, and the PR head source plus a focused node probe show the introduced false-merge path for unsupported-script snippets.

## Solution Assessment

No. Sharing the tokenizer is the right direction, but using MMR's raw empty-set Jaccard result directly in dreaming dedupe is not the narrow safe fix because it drops distinct empty-token snippets.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve empty-token fallback for non-CJK snippets:** `extensions/memory-core/src/memory/tokenize.ts:59-60`
  - body: `dedupeEntries` now calls `textSimilarity`, and this branch returns `1` whenever both token sets are empty. Since `tokenize` still emits only ASCII and CJK-family tokens, distinct Arabic, Cyrillic, emoji-only, or punctuation-only snippets from the same path are now treated as duplicates and one is dropped; current main only merged empty-token pairs when the normalized full strings matched exactly. Keep the shared CJK tokenizer, but make the dreaming text comparison preserve that exact-match fallback for empty-token pairs or expand tokenization so these scripts do not collapse to empty.
  - confidence: 0.91
