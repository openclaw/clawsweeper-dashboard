## Summary

Keep open: the CJK tokenizer reuse is the right direction, but the PR currently introduces a blocking false-merge/data-loss case for same-path snippets whose text produces empty token sets.

## What This Changes

This PR extracts memory-core's CJK-aware MMR tokenizer into a shared helper, routes dreaming dedupe through it, and adds focused CJK/ASCII regression tests.

## Best Possible Solution

Land a narrow memory-core fix that keeps the CJK dedupe improvement while preserving exact-match-only behavior for unsupported empty-token snippets, with regression coverage for both cases.

## Reproduction Assessment

Yes. Current main's source shows the CJK pair falls through to exact-string-only similarity, and the PR diff shows distinct empty-token snippets would now score 1 in the destructive dedupe path.

## Solution Assessment

No. Extracting the CJK-aware tokenizer is maintainable, but routing dreaming dedupe directly through a generic helper that treats two empty token sets as identical is not safe; preserve the old empty-token full-string fallback or tokenize more Unicode scripts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve exact-match fallback for empty-token snippets:** `extensions/memory-core/src/memory/tokenize.ts:59-60`
  - body: `tokenize` only emits ASCII and CJK-family tokens, so distinct same-path snippets in scripts like Cyrillic or Arabic, or emoji/punctuation-only snippets, produce two empty sets. Because `dedupeEntries` now calls `textSimilarity` directly, `jaccardSimilarity` returns `1` and drops one entry at normal thresholds; current main only merged empty-token pairs when the normalized full strings matched. Keep the CJK fix, but preserve that fallback for the dreaming dedupe path.
  - confidence: 0.94
