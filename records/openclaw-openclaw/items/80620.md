## Summary

Keep open. The CJK tokenizer extraction is the right general fix path, but the patch changes dreaming dedupe so two distinct snippets with no ASCII or CJK tokens now compare as identical and one can be dropped.

## What This Changes

The branch extracts the memory MMR CJK-aware tokenization helpers into `extensions/memory-core/src/memory/tokenize.ts`, reuses them from dreaming dedupe, re-exports the old MMR API, and adds dreaming dedupe regression tests.

## Best Possible Solution

Keep the shared tokenizer extraction, preserve dreaming dedupe's old exact-match fallback when tokenization yields no comparable tokens, and coordinate the linked issue closure with the separate promotion-leak fix.

## Reproduction Assessment

Yes. Source inspection shows current main's CJK-only snippets fall into exact-match-only dedupe, while the PR's new `textSimilarity` path returns `1` for two empty token sets and can falsely merge distinct non-tokenized snippets.

## Solution Assessment

No, not as written. Sharing the CJK-aware tokenizer is the narrow maintainable direction, but the dreaming caller needs an empty-token exact-match guard or equivalent wrapper before this is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Preserve empty-token fallback in dreaming dedupe:** `extensions/memory-core/src/memory/tokenize.ts:59-60`
  - body: `dedupeEntries` now calls `textSimilarity`, and this helper returns `1` when both token sets are empty. Since `tokenize` only emits ASCII and CJK-family tokens, two distinct snippets in unsupported scripts or symbol-only content from the same path will now be treated as duplicates and one will be dropped whenever the threshold is `<= 1`; current main only merged empty-token pairs when the normalized full strings matched exactly. Keep the CJK-aware tokenizer, but preserve that fallback for the dreaming dedupe path.
  - confidence: 0.93
