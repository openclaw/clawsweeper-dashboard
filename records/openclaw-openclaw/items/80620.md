## Summary

Keep open: the CJK dedupe fix is pointed at the right memory-core bug, but the current patch introduces a blocking false-merge regression for same-path snippets whose text produces no ASCII/CJK tokens.

## What This Changes

The PR extracts the memory MMR CJK-aware tokenizer into a shared helper, routes dreaming dedupe through it, and adds dreaming dedupe regression tests.

## Best Possible Solution

Keep the PR open and repair the dreaming dedupe path so CJK snippets use shared tokenization without treating distinct empty-token snippets as identical; add a regression for that case.

## Reproduction Assessment

Yes. Current main source shows CJK-only snippets become empty token sets and fall back to exact string equality; the PR body also gives terminal proof for the intended CJK scenarios, though I did not run tests in this read-only review.

## Solution Assessment

No, not as currently written. Extracting the tokenizer is the right direction, but using textSimilarity directly in dreaming dedupe changes empty-token behavior and should preserve exact-match fallback or broaden tokenization before automerge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Preserve the empty-token fallback in dreaming dedupe:** `extensions/memory-core/src/dreaming-phases.ts:1360`
  - body: This now uses `textSimilarity`, whose tokenizer only emits ASCII/CJK tokens and whose Jaccard helper returns `1` for two empty sets. Distinct same-path snippets in Cyrillic, Arabic, emoji-only, or punctuation-only text will therefore meet any dedupe threshold <= 1 and one memory can be dropped; the old local logic only merged empty-token pairs when the normalized full strings matched exactly.
