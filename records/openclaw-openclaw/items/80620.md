## Summary

Keep open. The PR addresses a real source-reproducible memory-core CJK dedupe bug and includes usable terminal proof, but the current diff introduces a blocking false-merge path for snippets whose text produces empty token sets.

## What This Changes

This PR extracts memory-core's CJK-aware tokenizer into a shared helper, switches dreaming dedupe to it, and adds focused CJK dedupe regression tests.

## Best Possible Solution

Keep the shared CJK tokenizer direction, but preserve dreaming dedupe's old exact-normalized fallback for empty token sets and add non-tokenized-script regression coverage before landing.

## Reproduction Assessment

Yes. Current main's CJK dedupe failure is source-reproducible because tokenizeSnippet drops CJK and falls back to exact-match similarity; the PR-introduced empty-token false merge is also source-reproducible from the new call site and shared tokenizer behavior.

## Solution Assessment

No as written. Extracting the CJK tokenizer is the right maintainable shape, but calling generic textSimilarity directly loses the dedupe-specific exact-string fallback for tokenless snippets.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve empty-token fallback in dreaming dedupe:** `extensions/memory-core/src/dreaming-phases.ts:1360`
  - body: This call now uses textSimilarity, whose tokenizer only emits ASCII and CJK-family tokens. Two different Cyrillic, Arabic, emoji-only, or punctuation-only snippets from the same path therefore produce empty token sets, jaccardSimilarity returns 1, and dedupeEntries drops one at the default 0.9 threshold. The old helper only merged empty-token pairs when the normalized full strings matched, so please keep that dedupe-specific fallback or add token coverage before replacing this call.
  - confidence: 0.92
