## Summary

Keep open. The PR fixes the reported CJK dedupe direction and has adequate terminal proof, but it regresses dreaming dedupe for snippets whose text produces no ASCII/CJK tokens by treating distinct same-path snippets as duplicates.

## What This Changes

The PR extracts the memory MMR tokenizer into a shared helper, routes dreaming dedupe through it, re-exports the MMR similarity APIs, and adds CJK-focused regression tests.

## Best Possible Solution

Keep the shared CJK-aware tokenizer direction, but preserve dreaming dedupe's old exact-match behavior when tokenization yields empty sets and coordinate issue closure with the separate promotion-leak fix.

## Reproduction Assessment

Yes. Current main's source shows CJK-only snippets tokenize to empty sets and fall back to exact-match similarity, and a read-only probe confirms the PR fixes that while introducing an empty-token false-merge case.

## Solution Assessment

No as written. Extracting the CJK-aware tokenizer is the right maintainable direction, but the dreaming dedupe caller needs an explicit empty-token policy instead of inheriting MMR's empty-empty similarity of 1.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve the empty-token fallback in dreaming dedupe:** `extensions/memory-core/src/dreaming-phases.ts:1360`
  - body: `snippetSimilarity` returns 1 when both snippets produce empty token sets. In this dedupe path, that means two unrelated same-path snippets in scripts the tokenizer still does not cover, such as Cyrillic or Arabic, are treated as duplicates and one is dropped; current main only merged empty-token snippets when the normalized full strings matched. Keep that exact-match fallback for empty-token cases before applying the shared helper here.
  - confidence: 0.93
