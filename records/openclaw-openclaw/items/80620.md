## Summary

Keep open: the PR fixes a real CJK dreaming-dedupe bug, but the current patch still introduces a blocking empty-token regression in destructive memory dedupe for non-CJK scripts or punctuation-only snippets.

## What This Changes

This PR extracts memory-core's CJK-aware MMR tokenizer into a shared helper, routes dreaming dedupe through it, and adds focused regression tests.

## Best Possible Solution

Land the shared CJK tokenizer only after dreaming dedupe preserves exact-match behavior for empty-token snippets, or after broader Unicode tokenization covers those scripts with regression tests.

## Reproduction Assessment

Yes. Source inspection shows current main's CJK dedupe failure and the PR's empty-token false-merge path through dedupeEntries without needing a live workspace run.

## Solution Assessment

No as written. Extracting the CJK-aware tokenizer is the right direction, but destructive dreaming dedupe needs the old empty-token exact-match fallback or a broader Unicode tokenizer before this can safely merge.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Preserve empty-token fallback for non-CJK snippets:** `extensions/memory-core/src/memory/tokenize.ts:59-60`
  - body: `tokenize` only emits ASCII and CJK-family tokens, so distinct same-path snippets in scripts such as Cyrillic or Arabic, or emoji/punctuation-only snippets, can produce two empty token sets. Because `dedupeEntries` now calls this helper directly, `jaccardSimilarity` returns `1` and drops one entry at normal thresholds; current main only merged empty-token pairs when the normalized full strings matched. Keep the CJK fix, but preserve that fallback for the dreaming dedupe path.
  - confidence: 0.94
