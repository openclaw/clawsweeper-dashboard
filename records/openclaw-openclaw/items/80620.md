## Summary

Keep open and do not merge as-is. The CJK tokenizer direction is right, but the current patch changes dreaming dedupe semantics for snippets whose token sets are both empty, which can silently merge unrelated non-ASCII/non-CJK memories from the same path.

## What This Changes

The PR extracts memory-core CJK-aware tokenizer and similarity helpers from MMR into a shared module, uses them for dreaming-phase dedupe, and adds CJK dedupe regression tests.

## Best Possible Solution

Keep the shared CJK tokenizer extraction, but give dreaming dedupe a small wrapper that preserves exact normalized-string fallback when both snippets produce no supported tokens.

## Reproduction Assessment

Yes. The original CJK bug is source-reproducible from current main's ASCII-only tokenizer, and the PR-introduced regression is source-reproducible because `textSimilarity` returns 1 for two empty token sets while current dreaming dedupe only merges empty-token snippets on exact normalized equality.

## Solution Assessment

No. Extracting and sharing the CJK-aware tokenizer is the right direction, but the current PR needs a dedupe-specific empty-token fallback before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve empty-token exact-match fallback:** `extensions/memory-core/src/dreaming-phases.ts:1360`
  - body: `textSimilarity` returns `1` when both token sets are empty, so this call makes two distinct same-path snippets in unsupported scripts, emoji-only text, or punctuation-only text look identical. The old dedupe code only merged empty-token pairs when the normalized full strings matched exactly, so this can silently drop unrelated memories; keep the shared tokenizer but preserve the dedupe-specific empty-token fallback at this boundary.
  - confidence: 0.91
