## Summary

Keep open: the PR targets a real plugin dispatch bug and current main still lacks the guard, but the patch is not merge-ready because it turns `subscription.handle` into an unbound function call that can break method-style plugin handlers. The draft, maintainer-labeled state also means this should stay in author/maintainer follow-up rather than cleanup close.

## What This Changes

The branch guards plugin agent-event subscription metadata during dispatch and adds a contract test proving a malformed subscription row does not block a healthy subscriber.

## Best Possible Solution

Keep the core guard, but invoke the captured handler with the original subscription as receiver and add a regression for method-style handlers before moving the PR out of draft.

## Reproduction Assessment

Yes from source: current main dereferences `registration.subscription` before any guard, so a throwing subscription getter can abort dispatch before later subscribers run. The introduced receiver regression is also source-reproducible at PR head by registering a method-style `handle()` that reads `this`.

## Solution Assessment

No; guarding subscription metadata in `src/plugins/host-hook-runtime.ts` is the right layer, but the implementation should preserve the original subscription receiver while avoiding fragile re-reads.
