## Summary

Keep open: current main and v2026.6.8 already avoid EnvHttpProxyAgent when `NO_PROXY` matches, but the reporter's local-service workflow still falls into the documented strict SSRF path and open linked PRs are competing over that security-sensitive behavior.

## What This Changes

The issue requests that `web_fetch` with `tools.web.fetch.useTrustedEnvProxy` honor `NO_PROXY`/`no_proxy` for local and internal targets by bypassing the proxy and connecting directly.

## Best Possible Solution

Maintainers should choose the canonical policy: preserve the documented strict fallback, or land one security-reviewed PR that implements scoped NO_PROXY local access while preserving metadata/link-local protections, docs, and real web_fetch proof.

## Reproduction Assessment

Partially yes: source and tests show current main respects NO_PROXY by avoiding EnvHttpProxyAgent, and also show NO_PROXY-matched localhost/private targets stay on strict SSRF checks. I did not run a live proxy plus local service reproduction in this read-only review.

## Solution Assessment

Unclear. A narrow proxy-bypass-only fix is already covered by the shared guard, while direct local web_fetch access needs a maintainer-approved security policy rather than being implied by NO_PROXY alone.
