## Summary

Keep open: current main and v2026.6.9 already skip the env proxy when `NO_PROXY` matches, but NO_PROXY-matched local/private `web_fetch` targets still fall into the strict DNS-pinned SSRF path, so the reporter's local-service workflow remains unresolved behind open candidate PRs and a maintainer security/product decision.

## What This Changes

The issue requests that `web_fetch` honor `NO_PROXY`/`no_proxy` when `tools.web.fetch.useTrustedEnvProxy` is enabled so local and internal addresses bypass the configured environment proxy.

## Best Possible Solution

If maintainers approve NO_PROXY local access, implement it in `src/infra/net/fetch-guard.ts` as an exact-host trusted bypass that preserves metadata/link-local blocking, updates docs, adds focused SSRF tests, and includes real `web_fetch` proof; otherwise keep the documented strict fallback and close the competing PRs with that policy decision.

## Reproduction Assessment

Yes from source: current `web_fetch` skips EnvHttpProxyAgent when NO_PROXY matches, then uses the strict pinned SSRF path that blocks local/private targets. I did not run a live proxy plus local service in this read-only review.

## Solution Assessment

Unclear until maintainers decide the security policy. The maintainable fix belongs in the shared guarded-fetch layer, not a wrapper-only workaround, and direct local access must preserve metadata/link-local protections if approved.
