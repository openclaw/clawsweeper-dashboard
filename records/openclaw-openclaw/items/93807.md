## Summary

Keep open: current main and v2026.6.8 honor `NO_PROXY` by falling back to the strict DNS-pinned path, but that still does not satisfy the reporter's local/private-service workflow, and multiple open closing PRs already own the security-sensitive fix decision.

## What This Changes

The issue asks for `web_fetch` trusted-env-proxy mode to honor `NO_PROXY`/`no_proxy` so excluded local or private targets bypass the proxy instead of failing or being routed through it.

## Best Possible Solution

Choose the intended security semantics for `NO_PROXY` in trusted-env-proxy mode, then land one focused PR with direct/private-target tests, docs, and proof if that operator-controlled bypass is accepted.

## Reproduction Assessment

Yes for the remaining local/private-service failure: source inspection shows `NO_PROXY` makes the proxy gate false and the request then reaches the strict SSRF path, which blocks private/internal hosts. I did not establish a live repro that current main still routes public `NO_PROXY` matches through the proxy.

## Solution Assessment

Unclear until maintainers choose the security policy: the clean solution is likely in the shared guarded-fetch/proxy decision surface, but directly allowing private-network bypasses needs scoped tests and metadata/link-local protections.
