## Summary

Keep open: current main and v2026.6.8 already avoid EnvHttpProxyAgent when `NO_PROXY` matches, but the reporter's local-service workflow still falls into the documented strict SSRF path and the linked open PRs need a maintainer security/product decision.

## What This Changes

The issue reports that `web_fetch` with `tools.web.fetch.useTrustedEnvProxy` does not honor `NO_PROXY`/`no_proxy`, so local or internal URLs are still proxied.

## Best Possible Solution

Keep this issue open while maintainers review the linked PRs and choose the canonical security-preserving behavior for `NO_PROXY` matches.

## Reproduction Assessment

Partially yes: current main makes `NO_PROXY` matches ineligible for EnvHttpProxyAgent, then sends local/private targets through strict DNS-pinned SSRF checks that are documented to block private/internal hosts. I did not run a live proxy plus local-service repro, so this is not a high-confidence end-to-end reproduction.

## Solution Assessment

Unclear; forcing direct access for all `NO_PROXY` matches would solve the reporter workflow but may weaken the web_fetch SSRF boundary. The best solution is a maintainer-reviewed PR with explicit tests for both proxy bypass and local/private-address safety, likely through an explicit opt-in policy rather than an implicit `NO_PROXY` bypass.
