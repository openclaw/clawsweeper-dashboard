## Summary

Keep open: current main and v2026.6.9 already skip the EnvHttpProxyAgent when NO_PROXY matches, but NO_PROXY-matched localhost/private targets still fall into the strict DNS-pinned SSRF path, so the reported local-service workflow remains unresolved and is already paired with open candidate PRs.

## What This Changes

The issue requests that web_fetch trusted environment proxy mode honor NO_PROXY/no_proxy so matching local or private targets are reached directly instead of through the proxy.

## Best Possible Solution

Keep this issue open until maintainers either keep the documented strict fallback or land one security-reviewed shared-guard PR with scoped NO_PROXY local access, metadata/link-local protections, docs, tests, and real web_fetch proof.

## Solution Assessment

Unclear until maintainers decide the security policy. The maintainable fix belongs in the shared guarded-fetch layer, not a web_fetch wrapper-only workaround, and direct local access must preserve metadata/link-local protections if approved.
