## Summary

Keep open: current main and v2026.6.8 already avoid EnvHttpProxyAgent when `NO_PROXY` matches, but the reporter's local-service workflow still runs into the documented strict SSRF path, and multiple open PRs are already competing to decide that security-sensitive behavior.

## What This Changes

The issue requests that `web_fetch` with `tools.web.fetch.useTrustedEnvProxy` respect `NO_PROXY`/`no_proxy` so local or internal targets bypass the configured HTTP(S) proxy.

## Best Possible Solution

Keep this issue open while maintainers review the linked PRs and decide whether the correct end state is the documented strict NO_PROXY fallback or an explicit, tested, security-reviewed local-access policy for trusted web_fetch deployments.

## Reproduction Assessment

Partially yes: source and tests show NO_PROXY matches do not use EnvHttpProxyAgent, but they also show the current strict fallback blocks localhost/private/internal targets. That gives a high-confidence source path for the reported local-service failure, not for the exact claim that current main still routes those targets through the proxy.

## Solution Assessment

Unclear. A narrow proxy-bypass-only fix appears already covered by the shared guard, while direct local web_fetch access needs an explicit maintainer security/product decision instead of being implied by NO_PROXY alone.
