## Summary

Keep open. Current main still lacks a runtime path that refreshes model-facing tool registrations for already-active sessions after a plugin install; install/update paths refresh persisted registry state and still steer users toward a gateway restart, while agent tool arrays and Pi allowlists are built as setup-time snapshots.

## What This Changes

The issue requests that newly installed plugin tools become available to already-active agent sessions without requiring a gateway restart.

## Best Possible Solution

Add a generic runtime lifecycle path that re-resolves plugin-provided tools after a successful plugin install or registry refresh, applies the same effective tool-policy filters used during normal run setup, updates active sessions only at a safe turn boundary, and invalidates affected prompt/tool-inventory cache state without duplicating plugin runtime loads.

## Reproduction Assessment

Yes. The issue gives concrete install -> immediate use -> restart steps, and current-main source inspection confirms install refreshes registry metadata while the model-facing tool list and Pi allowlist are setup-time snapshots. I did not run a live plugin install because this was a read-only review.

## Solution Assessment

Partly. The requested direction is valid, but the safest maintainable fix is turn-boundary re-resolution through the existing policy/filter pipeline rather than arbitrary mid-stream mutation of active session state.

## Security Review

Summary: The issue is security-sensitive because dynamically exposing newly installed plugin tools to live sessions can widen a session's authority if policy is not reapplied.
