## Summary

Keep open. Current main still treats plugin-provided tools as a point-in-time run/session setup artifact: plugin install refreshes registry metadata and still emits a restart hint, while embedded agent and compaction sessions build fixed tool arrays and pass a fixed Pi tool allowlist during session creation. I found no active-session tool delta, registry-change subscription, or turn-boundary re-resolution path that would make newly installed plugin tools available to an already active session.

## What This Changes

Request to refresh active agent session toolsets after successful plugin installs so newly registered plugin tools can be used without restarting the Gateway or session.

## Best Possible Solution

Add a generic runtime lifecycle path that re-resolves plugin-provided tools after a successful plugin install or registry refresh, applies the same effective tool-policy filters used during normal run setup, updates active sessions only at a safe turn boundary, and invalidates affected prompt/tool-inventory cache state without duplicating plugin runtime loads.

## Reproduction Assessment

Yes. The issue gives concrete install -> immediate use -> gateway restart steps, and current-main source inspection confirms plugin install refreshes registry metadata while active agent sessions keep a setup-time tool allowlist snapshot.

## Solution Assessment

Unclear as a patch because this issue has no implementation. The requested direction is valid, but the safer maintainable shape is turn-boundary re-resolution through existing policy filters rather than arbitrary mid-stream mutation of active session state.
