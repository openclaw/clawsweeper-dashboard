## Summary

Keep open. Current main still has no active-session toolset refresh after plugin install: install/update paths refresh plugin registry state and still tell the user to restart, while model-facing tools are resolved into concrete arrays and Pi session allowlists during run/session setup. The newer `tools.effective` inventory can show current runtime availability, but I found no registry-change subscription, toolset-delta broadcast, or later `setActiveToolsByName` path that updates an already-active model session.

## What This Changes

Request to make tools registered by newly installed plugins available to already-active agent sessions without requiring a gateway or session restart.

## Best Possible Solution

Add a generic runtime lifecycle path that re-resolves plugin-provided tools after a successful plugin install or registry refresh, applies the same effective tool-policy filters used during normal run setup, updates active sessions only at a safe turn boundary, and invalidates affected prompt/tool-inventory cache state without duplicating plugin runtime loads.

## Reproduction Assessment

Yes. The issue gives concrete install -> immediate use -> restart steps, and current-main source inspection confirms plugin install refreshes registry metadata while the actual model-facing tool list and Pi allowlist are setup-time snapshots. I did not run a live plugin install because this was a read-only review, but the source path gives a high-confidence reproduction basis.

## Solution Assessment

Unclear as a patch because this issue has no implementation. The requested direction is valid, but the safest maintainable shape is turn-boundary re-resolution through the existing policy/filter pipeline rather than arbitrary mid-stream mutation of active session state.
