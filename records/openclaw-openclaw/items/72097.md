## Summary

Keep open. Current main still matches the reported stale-toolset gap: plugin install refreshes registry/install metadata and docs still tell users to restart the gateway, while the embedded agent run builds a concrete tool array and session allowlist only during run/session setup. The remaining fix is real but security-sensitive because it would expose newly installed plugin tools to already-active sessions.

## What This Changes

Request dynamic refresh of plugin-provided tools in already-active agent sessions after a successful plugin install.

## Best Possible Solution

Add a generic runtime lifecycle path that re-resolves plugin-provided tools after successful plugin install or registry refresh, applies the same effective tool-policy pipeline used during normal run setup, updates active sessions only at a safe turn boundary, and invalidates affected prompt/tool inventory caches without duplicating plugin runtime loads.

## Reproduction Assessment

Yes. The issue gives concrete install -> immediate use -> restart steps, and current-main source inspection confirms install refreshes registry state while active embedded sessions keep a session-created tool allowlist snapshot.

## Solution Assessment

Unclear as a patch because there is no proposed implementation in this issue. The requested direction is valid, but the safer maintainable shape is turn-boundary re-resolution through existing policy filters rather than arbitrary mid-stream mutation of active session state.
