## Summary

Keep open. Current main still refreshes persisted plugin registry state after install, but model-facing plugin tools are built into setup-time tool arrays and Pi session allowlists, and the public docs/command replies still direct users to restart the Gateway after plugin install/config changes.

## What This Changes

Request dynamic refresh of plugin-provided tools for already-active agent sessions after a successful plugin install.

## Best Possible Solution

Add a generic lifecycle path that re-resolves plugin tools after successful install or registry refresh, applies the same policy pipeline used during normal run setup, updates active sessions only at a safe turn boundary, and invalidates affected prompt/tool inventory cache state.

## Reproduction Assessment

Yes. The issue gives concrete install -> immediate use -> restart steps, and source inspection on current main confirms plugin install refreshes registry metadata while model-facing tools and Pi allowlists remain setup-time snapshots. I did not run a live plugin install because this was a read-only review.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is valid, but the safest maintainable solution is turn-boundary re-resolution through existing policy/filter code rather than arbitrary mid-stream mutation of active session state.

## Security Review

Status: needs_attention

Summary: The requested behavior would dynamically expose newly installed plugin tools to live sessions, so it must preserve existing tool authorization boundaries.

Concerns:

- **[medium] Reapply tool policy before live tool injection:** `src/agents/pi-tools.ts:765`
  - body: Normal tool construction applies owner-only, provider/model, group, sandbox, subagent, plugin allowlist, and schema-normalization filters before tools reach the model; any active-session refresh must reuse that pipeline before adding newly installed plugin tools.
  - confidence: 0.9
- **[medium] Refresh only at a safe turn boundary:** `src/agents/pi-embedded-runner/run/attempt.ts:1561`
  - body: The embedded runner builds the system prompt, customTools, and sessionToolAllowlist once for the run; mutating those mid-stream could desynchronize prompt text, SDK allowlists, and transcript tool results.
  - confidence: 0.86
