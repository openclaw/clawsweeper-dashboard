## Summary

Keep open. Current main still resolves plugin tools as a point-in-time run setup artifact: plugin install refreshes registry metadata and still tells the user to restart, while embedded agent sessions build `effectiveTools`, pass a fixed Pi tool allowlist into session creation, and call `setActiveToolsByName` once. I found no current active-session tool delta, turn-boundary re-resolution, or session allowlist update path after plugin install.

## What This Changes

Request to make newly installed plugin-provided tools available to already-active agent sessions without requiring a gateway/session restart.

## Best Possible Solution

Add a generic runtime lifecycle path that re-resolves plugin-provided tools after a successful plugin install or registry refresh, applies the same effective tool-policy filters used during normal run setup, updates active sessions only at a safe turn boundary, and invalidates affected prompt/tool-cache state without duplicating plugin runtime loads.

## Reproduction Assessment

Yes. The issue gives concrete install -> immediate use -> gateway restart steps, and current-main source inspection confirms install refreshes the registry while active agent sessions keep a session-created tool allowlist snapshot.

## Solution Assessment

Unclear as a patch because this issue has no implementation. The requested direction is valid, but the safer maintainable shape is turn-boundary re-resolution through existing policy filters rather than arbitrary mid-stream mutation of active session state.
