## Summary

Keep open: current main still selects plain provider-usage fetches without consulting the env-proxy helper, and this PR is the paired implementation for the still-open user report. The code change is narrow and plausible, but the external PR still lacks after-fix real behavior proof and has latest-head checks that are not green.

## What This Changes

The PR updates provider usage loading to prefer an explicit fetch override, otherwise an env-proxy-aware fetch, otherwise the normal global fetch, and adds a provider-usage regression test.

## Best Possible Solution

Keep this PR open and merge the narrow loader change only after real proxied Codex usage proof is added, the regression test observes fetch selection, and the relevant checks are green.

## Reproduction Assessment

Yes for source-level reproduction: current main passes plain resolveFetch(opts.fetch) into the provider usage hook, and Codex usage then calls the WHAM endpoint through that fetchFn. I did not run a live environment where direct chatgpt.com fetch is blocked but HTTP_PROXY/HTTPS_PROXY succeeds.

## Solution Assessment

Partly: the loader precedence proposed by the PR is the narrow maintainable fix and preserves explicit fetch overrides. The PR is not merge-ready until the test actually observes the selected fetch and the contributor supplies real after-fix proxy usage proof.
