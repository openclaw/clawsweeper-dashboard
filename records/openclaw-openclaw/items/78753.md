## Summary

Keep open: the linked bug is valid and this PR targets the right loader, but the current diff is not merge-ready because it changes fetch override precedence, lacks a changelog entry, and still has no submitted-branch real proxy proof.

## What This Changes

The PR changes `loadProviderUsageSummary` to prefer `resolveProxyFetchFromEnv(env)` for provider usage loads and adds two tests for proxy-env and no-proxy fetch behavior.

## Best Possible Solution

Keep the fix in the provider usage loader, but preserve explicit fetch overrides first, then fall back to env proxy fetch and normal fetch, with focused regression coverage, a changelog entry, and redacted real proxy proof.

## Reproduction Assessment

No live high-confidence reproduction against current `main` was established. Source inspection plus the linked issue logs show the loader still supplies plain `resolveFetch(opts.fetch)` to the Codex WHAM usage path while an existing env proxy helper is available.

## Solution Assessment

No, not as written. The narrow maintainable fix should keep `opts.fetch` precedence, then use `resolveProxyFetchFromEnv(env)`, then normal fetch; this PR reverses that contract and still needs changelog and real behavior proof.
