## Summary

Keep open: the PR targets a valid Codex provider-usage proxy gap, but the current patch is not mergeable because it bypasses explicit fetch overrides, lacks the required user-facing changelog entry, and still has no submitted-branch real behavior proof.

## What This Changes

The PR changes `loadProviderUsageSummary` to choose `resolveProxyFetchFromEnv(env)` before the normal fetch resolver and adds provider-usage tests for proxy-env and no-proxy behavior.

## Best Possible Solution

Keep the provider-usage loader fix, but preserve explicit `opts.fetch` precedence, use the env proxy only when no explicit fetch was supplied, add focused regression coverage and a changelog entry, then require redacted real proxy proof.

## Reproduction Assessment

No live high-confidence reproduction against current `main` was run. Source inspection plus the linked issue logs show the loader still supplies plain `resolveFetch(opts.fetch)` to the Codex WHAM usage path while an existing env-proxy helper is available.

## Solution Assessment

No, not as written. The narrow maintainable fix should keep an explicit fetch override first, then fall back to `resolveProxyFetchFromEnv(env)` and normal global fetch behavior.
