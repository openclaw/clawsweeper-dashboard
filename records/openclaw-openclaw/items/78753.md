## Summary

Keep open: the PR targets the right source-backed bug, but as written it bypasses explicit fetch injection when proxy env vars exist, lacks the required changelog entry, and still needs real behavior proof for the submitted branch.

## What This Changes

The PR imports `resolveProxyFetchFromEnv` and changes `loadProviderUsageSummary` to choose an env-proxy-aware fetch before falling back to the existing fetch resolver.

## Best Possible Solution

Update the PR to preserve explicit fetch overrides first, then fall back to env proxy fetch and normal fetch; add focused regression coverage, a changelog entry, and redacted real proxy proof before merge.

## Reproduction Assessment

No live high-confidence reproduction was established here. The source path is clear and the linked issue provides detailed reporter logs for `openclaw status --usage --json` with direct `chatgpt.com` egress blocked and HTTP(S) proxy env configured.

## Solution Assessment

No, not as written. The narrow fix should keep `opts.fetch` precedence, then use `resolveProxyFetchFromEnv(env)`, then normal fetch, with a regression test and changelog entry.

## Review Findings
