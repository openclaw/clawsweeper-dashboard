## Summary

This PR is still useful and not implemented on current main: current main and the latest release still reconnect when the actor is connected but the underlying task state is non-running. The runtime change looks like the right narrow fix after tracing the actor, caller, reconnect, receive-failure, tick, and send-failure paths, but the supplied real behavior proof is not enough for merge because the focused Swift test did not execute and no live macOS node-mode run was shown.

## What This Changes

The PR changes `GatewayChannelActor.connect()` to short-circuit on the actor's `connected` flag and adds a Swift regression test for a connected task whose `URLSessionTask.State` flips to `.suspended`.

## Best Possible Solution

Land the narrow actor guard change with an executed focused Swift regression test or redacted live macOS node-mode proof, then let the linked bug close on merge.

## Reproduction Assessment

No live reproduction was established in this review. The current-main source path is source-reproducible: repeated `connect()` calls can reconnect when `connected` remains true but `task.state` is non-running, which matches the linked report's suspected trigger.

## Solution Assessment

Yes, this appears to be the best narrow code fix for the suspected root cause because `GatewayChannelActor` already owns connection truth and its receive, tick, send, and shutdown paths clear that state before reconnecting. The safer alternative of suppressing macOS node logs would hide the symptom while leaving the duplicate WebSocket creation path intact.
