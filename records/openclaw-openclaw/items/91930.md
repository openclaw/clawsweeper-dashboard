## Label Justifications

- `P2`: The defect prevents discovery of supported models for one bundled provider but does not make the core runtime unusable.
- `impact:auth-provider`: The issue affects provider catalog routing, model availability, and model selection.

## Summary

The report remains valid on current main: OpenCode’s live Zen endpoint currently returns 48 models including the reported DeepSeek model, but OpenClaw’s OpenCode plugin does not own or refresh a Zen catalog, so CLI discovery depends on a narrower runtime registry. No open canonical PR or issue now owns the gap, and the remaining choice between mirroring live rows and maintaining a curated metadata-complete catalog requires maintainer product judgment.

## What This Changes

Make the OpenCode Zen model catalog exposed by OpenClaw include supported models beyond the single registry entry, including models advertised by the live Zen API.

## Best Possible Solution

Define one authoritative provider-owned Zen catalog policy, then implement either validated live discovery with metadata enrichment or a curated metadata-complete catalog, with CLI-list and runtime regression coverage for representative Claude, GPT, Gemini, and DeepSeek rows.

## Reproduction Assessment

Yes, with high confidence at the source and external-contract level: the live Zen endpoint contains the reported model, current docs promise a multi-model catalog, and the provider plugin has no catalog source that can expose those rows. A local authenticated CLI run was not needed to establish the missing ownership path.

## Solution Assessment

Unclear pending one product decision: maintainers must choose whether Zen discovery mirrors the live endpoint or remains curated. The implementation should stay provider-local rather than adding OpenCode-specific policy to generic model-list code.
