## Label Justifications

- `impact:auth-provider`: The issue concerns provider catalog visibility and model selection for the OpenCode runtime provider.

## Summary

The report remains necessary: current main documents OpenCode Zen as a multi-model catalog, the live Zen endpoint currently exposes 48 models including `deepseek-v4-flash-free`, and the OpenCode plugin does not provide a catalog registration that would surface that set through the provider-filtered CLI path.

## What This Changes

Make OpenCode Zen models exposed by the provider catalog available through `openclaw models list --all --provider opencode`.

## Best Possible Solution

Preserve one canonical OpenCode catalog owner, then expose the complete supported Zen set through the existing plugin catalog contract with verified metadata and provider-filtered CLI regression coverage rather than adding OpenCode-specific logic to core.

## Reproduction Assessment

Yes at source level: current docs promise multiple Zen model families, the live endpoint exposes 48 ids, and current OpenCode registration supplies no provider catalog capable of surfacing that set. A local CLI execution was not needed to establish the missing catalog path, though it should be part of fix validation.

## Solution Assessment

Unclear. The defect is real, but maintainers should first confirm whether Zen catalog authority remains the bundled pi registry or should move to a plugin-owned live adapter; either choice should use the existing catalog contract and retain complete model metadata.
