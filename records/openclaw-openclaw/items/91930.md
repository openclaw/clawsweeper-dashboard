## Label Justifications

- `impact:auth-provider`: The affected behavior is the provider's model catalog and model-routing selection surface.

## Summary

This is a valid current-main provider catalog bug, not a docs-only report: OpenCode advertises a multi-model Zen catalog and its live endpoint returns the reported model, while the bundled provider contributes no catalog and the generic registry path exposes only its narrower indexed set. The remaining maintainer decision is whether Zen should mirror the live endpoint or use an explicitly curated capability-enriched subset.

## What This Changes

Make OpenCode Zen model discovery include supported models available from the upstream catalog instead of listing only `opencode/claude-opus-4-6`.

## Best Possible Solution

Implement the Zen catalog inside `extensions/opencode` using the existing live provider-catalog seam, with capability-enriched known models and a deliberate policy for unknown live rows, then add provider-scoped CLI coverage and align the docs with that policy.

## Reproduction Assessment

Yes at source level: the upstream endpoint currently returns the reported model, the docs promise a multi-model Zen catalog, and current main has no OpenCode catalog hook before the CLI's registry fallback. The exact installed CLI command was not executed because this review is read-only and cannot generate runtime artifacts.

## Solution Assessment

No, treating this as documentation feedback alone is not the best solution. The narrow maintainable fix is a provider-owned Zen catalog using the same catalog contract as OpenCode Go, after maintainers choose dynamic versus curated exposure policy.
