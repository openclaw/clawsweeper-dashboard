## Summary

Keep open: current main and v2026.6.5 still present OpenCode Zen as a multi-model catalog, but the `opencode` plugin has no provider catalog hook or manifest `modelCatalog` rows, so provider-filtered model listing cannot surface the live Zen model set yet.

## What This Changes

The issue asks OpenClaw to make OpenCode Zen model listing/import match the live Zen catalog advertised by the provider docs.

## Best Possible Solution

Add a provider-owned OpenCode Zen catalog using the existing plugin catalog contract, with verified metadata and provider-filtered `models list` coverage, rather than adding OpenCode-specific logic to core.

## Reproduction Assessment

Yes at source level: current docs and the live Zen endpoint show a multi-model catalog, while the current Zen plugin and manifest lack any provider catalog hook or rows to expose it. I did not run the full CLI because this read-only checkout lacks `pnpm` and the source/API evidence is already decisive.

## Solution Assessment

Unclear for automation: the provider-owned catalog contract is the right layer, but maintainers still need to choose whether Zen should use static rows, live discovery, or a static-plus-live fallback like OpenCode Go.
