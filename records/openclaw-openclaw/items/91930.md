## Summary

This should stay open: current main still documents OpenCode Zen as a multi-model catalog, the live Zen endpoint exposes many models, and the bundled `opencode` provider still lacks the catalog/dynamic-model hooks that would make those rows appear through provider-filtered `models list`.

## What This Changes

Report that the OpenCode Zen docs/model-list flow only shows one local OpenClaw model even though the live Zen `/models` endpoint exposes many models including `deepseek-v4-flash-free`.

## Best Possible Solution

Make the OpenCode Zen docs and runtime agree: either add a provider-owned, upgrade-safe Zen catalog/dynamic-model path with focused model-list coverage and redacted live proof, or deliberately narrow the docs to the curated model surface maintainers want to support.

## Reproduction Assessment

Yes, at source level: current main's `opencode` plugin has no catalog/dynamic-model registration while the live Zen endpoint returns `deepseek-v4-flash-free` and many other models. I did not run a full authenticated `openclaw models list` workflow in this read-only review.

## Solution Assessment

Unclear until maintainers choose the catalog authority. The provider-local OpenCode plugin boundary is the right place, but the final shape should be upgrade-safe and either derive the live Zen catalog or intentionally document a curated subset.
