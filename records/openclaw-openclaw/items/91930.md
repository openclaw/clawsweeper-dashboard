## Summary

Keep open: current main still documents OpenCode Zen as a multi-model catalog and the live Zen endpoint returns many model ids, but the bundled Zen plugin exposes auth/runtime policy only and has no provider catalog hook or manifest catalog rows to make `models list --provider opencode` enumerate them.

## What This Changes

The issue reports that the OpenCode Zen docs and live `/zen/v1/models` catalog list many models, but `openclaw models list --all --provider opencode` only shows `opencode/claude-opus-4-6`.

## Best Possible Solution

Add a provider-owned OpenCode Zen catalog through the existing plugin catalog contract, backed by the chosen static, live, or hybrid metadata source and provider-filtered `models list` regression coverage.

## Reproduction Assessment

Yes, at source level: the live Zen endpoint returns the reported model ids, docs advertise a Zen catalog, and current Zen plugin/onboarding code has no catalog hook beyond the default Opus row. I did not run the authenticated CLI path in this read-only review.

## Solution Assessment

Unclear: the provider-owned catalog hook is the right layer, but maintainers still need to choose whether Zen's permanent catalog authority is static, live-discovered, or hybrid before implementation.
