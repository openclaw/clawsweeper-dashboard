- `impact:auth-provider`: The issue concerns provider catalog visibility and model selection for the OpenCode Zen runtime provider.

## Summary

Keep open: current main and v2026.6.5 still advertise OpenCode Zen as a multi-model catalog, but the Zen plugin has no model catalog hook or manifest catalog to surface the live Zen model set through provider-filtered model listing.

## What This Changes

The issue reports that OpenCode Zen live models such as `deepseek-v4-flash-free` are missing from `openclaw models list --all --provider opencode` despite the docs describing a multi-model Zen catalog.

## Best Possible Solution

Add a provider-owned OpenCode Zen catalog implementation using the existing plugin catalog contract, with verified metadata and provider-filtered `models list` regression coverage, rather than adding OpenCode-specific logic to core.

## Reproduction Assessment

Yes at source level: current docs and the live Zen endpoint show a multi-model catalog, while the current Zen plugin and manifest lack any catalog hook or rows to expose that catalog. I did not run the full CLI because this read-only review avoided repo-writing proof commands.

## Solution Assessment

Unclear. The provider-owned catalog contract is the right layer, but maintainers still need to choose whether Zen metadata should be static, live-discovered, or a static-plus-live fallback like OpenCode Go.
