## Label Justifications

- `P2`: The OpenCode provider remains partly usable, but incomplete catalog discovery blocks normal selection of supported Zen models for affected users.
- `impact:auth-provider`: The issue concerns provider-scoped model catalog visibility and model selection for OpenCode Zen.

## Summary

Keep open: current main and v2026.6.5 still advertise OpenCode Zen as a multi-model catalog, but the Zen plugin only seeds the default Opus model and does not expose a provider catalog hook or manifest catalog for the live Zen model set.

## What This Changes

The issue asks OpenCode Zen model discovery/docs to match the live Zen API catalog so provider-filtered model listing includes models such as `opencode/deepseek-v4-flash-free`.

## Best Possible Solution

Add a provider-owned OpenCode Zen catalog implementation through the existing plugin catalog contract, with maintainer-approved static/live/hybrid catalog authority, provider-filtered `models list` regression coverage, and aligned docs.

## Reproduction Assessment

Yes, at source level: current docs and the live Zen endpoint show a multi-model catalog, while the current Zen plugin and manifest lack any catalog hook or rows to expose that catalog. I did not run the full CLI because this review stayed read-only and avoided cache-writing commands.

## Solution Assessment

Unclear: the provider-owned catalog contract is the right layer, but maintainers still need to choose whether Zen metadata should be static, live-discovered, or static-plus-live fallback before implementation proceeds.
