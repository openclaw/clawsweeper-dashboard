## Summary

Keep open. Current main still exposes prompt-visible skills through a SKILL.md read/follow flow, while the implemented controls cover eligibility, visibility, install scanning, and prompt budgeting rather than the requested instruction isolation or role/memory/tool capability enforcement.

## What This Changes

Request instruction isolation for SKILL.md content, including role locking, filtering or monitoring, and per-skill capability enforcement.

## Best Possible Solution

Define a maintainer-approved trusted/untrusted skill precedence model, add typed capability metadata, enforce it through prompt assembly and runtime policy, and cover malicious SKILL.md cases with docs and focused regression tests.

## Reproduction Assessment

Yes for the missing-control behavior: current main source shows prompt-visible skills are listed with SKILL.md locations and the model is told to read and follow one. A deterministic malicious-skill exploit transcript is model-dependent, so I did not mark a runtime exploit as reproduced.

## Solution Assessment

Unclear as a final implementation plan. The requested isolation direction is valid, but delimiters or pattern filtering alone would be weaker than a typed capability contract with explicit prompt precedence, runtime enforcement, compatibility tests, and docs.
