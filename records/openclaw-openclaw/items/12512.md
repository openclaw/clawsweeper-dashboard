## Summary

Keep open. Current main still uses a model-facing available-skills catalog that tells the model to read and follow selected SKILL.md files, while the requested instruction isolation, role locking, per-skill memory/tool capabilities, and monitoring remain unresolved security design work.

## What This Changes

The issue requests a security boundary that isolates untrusted SKILL.md instructions from trusted agent context and enforces role, memory, and tool permissions for skills.

## Best Possible Solution

Define a maintainer-approved trusted/untrusted skill precedence model, add typed per-skill role/memory/tool capability metadata, enforce it through prompt assembly and runtime policy, and cover malicious SKILL.md cases with focused tests and docs.

## Reproduction Assessment

Yes for the missing-control behavior: current main source shows prompt-visible skills are listed with SKILL.md locations and the model is told to read and follow one. No deterministic exploit transcript was run, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear as a final implementation plan. The requested isolation direction is valid, but maintainers should choose a typed capability and prompt-precedence contract rather than landing only delimiter or pattern-filter prompt text.
