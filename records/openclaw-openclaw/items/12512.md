## Summary

Current main still treats installed skills as model-facing instructions: it catalogs eligible skills and explicitly directs the model to read and follow the selected SKILL.md. Existing controls cover prompt visibility, source/path containment, size limits, install scanning, and agent-level sandbox/tool policy, but they do not provide the requested per-skill role, memory, or tool authority boundary; the item therefore remains necessary and requires a maintainer security/product decision rather than automated implementation.

## What This Changes

Requests security isolation between SKILL.md instructions and the agent’s trusted context, including role, memory, tool, and monitoring controls.

## Best Possible Solution

Adopt a maintainer-approved skill trust model with typed provenance and capability metadata, enforce effective permissions at the runtime tool and memory boundaries rather than relying only on prompt filtering, and add adversarial tests proving untrusted skills cannot exceed those capabilities while trusted skills retain an explicit high-power path.

## Reproduction Assessment

Yes, at high confidence from source: current main instructs the model to read and follow a selected SKILL.md, while current skill metadata exposes visibility and invocation controls rather than per-skill role, memory, or tool authority. This reproduces the architectural gap, though no live exploit was executed.

## Solution Assessment

No, the proposed four phases are not yet the best implementation plan because prompt delimiters, signature filtering, and monitoring do not themselves enforce authority. The safer design is one typed trust/capability model enforced at prompt selection plus the actual tool and memory boundaries, with explicit trusted-skill compatibility behavior.
