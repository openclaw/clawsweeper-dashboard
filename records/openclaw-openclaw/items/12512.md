## Summary

Keep open: current main and v2026.6.10 still expose skills through a model-facing SKILL.md read-and-follow path, while instruction isolation, role locking, per-skill memory/tool authority, and runtime monitoring remain unresolved security/product design work.

## What This Changes

The issue requests a security boundary that treats SKILL.md content as untrusted instruction content instead of ordinary agent context.

## Best Possible Solution

Define one maintainer-approved skill trust and capability contract, then enforce it consistently through prompt selection, SKILL.md loading, tool execution, memory access, sandbox policy, regression tests, and docs.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 instruct the model to read and follow selected SKILL.md files, while skill policy only exposes invocation and prompt-visibility controls. I did not run a live malicious-skill exploit.

## Solution Assessment

Unclear as a final implementation path: the requested direction addresses a real security-hardening gap, but maintainers still need to define the trusted/untrusted skill contract, compatibility defaults, and enforcement scope.
