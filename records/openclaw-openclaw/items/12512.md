## Summary

Keep open: current main and v2026.6.8 still expose SKILL.md as a model-facing read-and-follow instruction surface, while existing controls are visibility, scanner, and tool-policy adjacent rather than the requested role, memory, tool, or behavioral-monitoring boundary.

## What This Changes

The issue requests instruction isolation for SKILL.md content so skill text cannot override agent role, memory, tool, or runtime policy.

## Best Possible Solution

Define one maintainer-approved skill trust and capability contract, then enforce it consistently through prompt assembly, SKILL.md loading, tool execution, memory access, sandbox policy, tests, and docs.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.8 still instruct the model to read and follow selected SKILL.md files, while skill policy only controls invocation and model visibility. I did not run a live malicious-skill exploit.

## Solution Assessment

Unclear as a final implementation path: the requested direction addresses a real security-hardening gap, but the best fix needs a typed trust and capability model enforced at runtime boundaries rather than prompt wording alone.
