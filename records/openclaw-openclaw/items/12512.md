## Summary

Keep open. Current main and the latest release still expose a prompt-facing skills catalog and instruct the model to read and follow selected SKILL.md files; the newer controls cover visibility, versioning, sandbox path materialization, and some static/proposal scanning, but not the requested hard instruction-isolation, role-locking, per-skill memory/tool capability enforcement, or runtime behavioral monitoring.

## What This Changes

Requests core security hardening so untrusted SKILL.md content cannot override agent role, memory, or tool policy when skills are loaded.

## Best Possible Solution

Define a maintainer-approved trusted/untrusted skill precedence model, add typed per-skill capability metadata for role, memory, and tool influence, enforce it through prompt assembly and runtime policy, and cover malicious SKILL.md cases with focused tests and docs.

## Reproduction Assessment

Yes, source-level. Current main still tells the model to read and follow matching SKILL.md files while the skill metadata and parser lack the requested role, memory, and tool capability boundaries.

## Solution Assessment

Unclear as an implementation path, because the request names the right security problem but spans prompt precedence, runtime enforcement, sandboxing, plugin compatibility, and security policy. The best fix needs maintainer design approval before code is queued.
