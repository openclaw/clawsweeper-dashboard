## Summary

Keep open. Current main and the latest release still expose prompt-visible SKILL.md files as model-read instruction sources, while the shipped controls are visibility, setup, and install-scanner adjacent rather than the requested instruction isolation, role locking, per-skill memory/tool boundary, or runtime monitoring model.

## What This Changes

The issue requests core security hardening that isolates SKILL.md instructions from trusted agent context and adds role, memory, tool, and monitoring boundaries for skills.

## Best Possible Solution

Define a maintainer-approved trusted/untrusted skill precedence model, add typed per-skill role, memory, tool, and monitoring metadata, enforce it through prompt assembly and runtime policy, and cover malicious SKILL.md cases with focused tests and docs.

## Reproduction Assessment

Yes for the source-level missing-control behavior: current main lists prompt-visible SKILL.md locations and instructs the model to read and follow them, while skill metadata lacks role, memory, tool, and monitoring capabilities. I did not run a live malicious-skill transcript in this read-only review.

## Solution Assessment

Unclear as a final implementation plan. The isolation direction is valid, but maintainers should choose a typed capability and prompt-precedence contract instead of landing only delimiter text, pattern filters, or scanner-only coverage.
