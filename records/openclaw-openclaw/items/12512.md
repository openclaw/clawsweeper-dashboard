## Summary

Keep open. Current main still makes SKILL.md a model-followed instruction source through the available-skills catalog, and the existing controls cover visibility, eligibility, containment, and some scanning rather than the requested instruction isolation, role locking, per-skill memory/tool boundaries, or runtime monitoring.

## What This Changes

The issue requests a new instruction-isolation boundary for SKILL.md content so skill instructions cannot override agent role, memory, tool, or policy behavior.

## Best Possible Solution

Define a maintainer-approved trusted/untrusted skill precedence model, add typed per-skill role, memory, and tool capability metadata, enforce it through prompt assembly and runtime policy, and cover malicious SKILL.md cases with focused tests and docs.

## Reproduction Assessment

Yes for the source-level missing-control behavior: current main lists prompt-visible SKILL.md locations and instructs the model to read and follow them, while skill metadata lacks role, memory, tool, and monitoring capabilities. I did not run a live malicious-skill transcript in this read-only review.

## Solution Assessment

Unclear as a final implementation plan. The isolation direction is valid, but maintainers should choose a typed capability and prompt-precedence contract instead of landing only delimiter text, pattern filters, or scanner-only coverage.
