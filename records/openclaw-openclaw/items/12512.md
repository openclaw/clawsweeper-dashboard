## Summary

Keep open: current main and the latest release still expose prompt-visible SKILL.md files as model-followed instruction sources, and the missing trusted/untrusted skill boundary remains a real security/product design item rather than cleanup that can be closed.

## What This Changes

The issue requests core security architecture that isolates untrusted SKILL.md instructions from trusted system prompt, role, memory, tool policy, and monitoring behavior.

## Best Possible Solution

Define a maintainer-approved trusted/untrusted skill precedence model, add typed per-skill role, memory, and tool capability metadata, enforce it through prompt assembly and runtime policy, and cover malicious SKILL.md cases with focused tests and docs.

## Reproduction Assessment

Yes for source-level reproduction: current main lists prompt-visible SKILL.md locations and instructs the model to read and follow them, while skill metadata lacks role, memory, tool, and monitoring capabilities. I did not run a live malicious-skill transcript in this read-only review.

## Solution Assessment

Unclear as a final implementation plan. The isolation direction is valid, but maintainers should choose a typed capability and prompt-precedence contract instead of landing only delimiter text, pattern filters, or scanner-only coverage.
