## Summary

Current main still presents matching SKILL.md files as model-facing specialized instructions, while skill policy only governs invocation visibility; the requested enforceable role, memory, and tool boundaries remain unimplemented and require a maintainer security/product decision.

## What This Changes

Add instruction isolation and per-skill capability enforcement so untrusted SKILL.md content cannot override agent role, memory, or tool policy.

## Best Possible Solution

Define a typed skill provenance and capability contract, decide which installed sources are trusted by default, enforce tool and memory access outside the model prompt, and add adversarial SKILL.md regression coverage while retaining current workflows for explicitly trusted skills.

## Reproduction Assessment

Yes at source level: current main instructs the model to read matching SKILL.md content, while the skill policy surface only controls visibility and invocation. There is no current per-skill role, memory, or tool capability enforcement to exercise as a failing runtime test.

## Solution Assessment

No, not exactly as proposed. Instruction isolation is the right objective, but delimiter precedence, role-lock wording, and regex signatures are not sufficient security boundaries; maintainers should first define typed trust and capabilities enforced by runtime owners.
