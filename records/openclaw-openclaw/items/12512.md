## Summary

Keep open. Current main still exposes prompt-visible skills through a model-facing catalog and tells the model to read and follow selected SKILL.md files, while the implemented controls cover visibility, eligibility, plugin hook prompt mutation, and generated-skill scanning rather than the requested per-skill instruction-isolation boundary.

## What This Changes

Request to isolate SKILL.md instructions from trusted agent context with role locking, per-skill role/memory/tool capabilities, and prompt-injection monitoring.

## Best Possible Solution

Define a maintainer-approved skill trust boundary, extend skill metadata/runtime policy with role, memory, and tool-influence controls, enforce it in prompt assembly and runtime execution, and document/test malicious SKILL.md cases.

## Reproduction Assessment

Yes for the missing-control behavior: current main can be inspected with any prompt-visible skill path, and tests/docs assert that the model is told to read and follow SKILL.md. A concrete exploit transcript is less deterministic because it depends on model compliance.

## Solution Assessment

Unclear as a direct implementation plan. The requested direction is valid, but delimiter or pattern filtering alone is weaker than a typed capability contract with explicit prompt precedence, runtime enforcement, compatibility tests, and docs.
