## Summary

Current main and the latest release still expose applicable skills to the model and instruct it to read and follow their SKILL.md files; existing visibility, scanning, sandbox, and tool-policy controls do not implement the requested per-skill instruction, memory, or capability boundary, and adjacent open proposals have not superseded this design work.

## What This Changes

Define and enforce isolation, role, memory, and tool-capability boundaries for instructions loaded from SKILL.md files.

## Best Possible Solution

Define trusted skill sources and typed per-skill capabilities, then enforce tool, sandbox, and memory effects through canonical runtime policy while treating prompt text only as guidance.

## Reproduction Assessment

Yes at the source level: current main explicitly directs the model to read and follow applicable SKILL.md instructions. The supplied report does not establish a live bypass of an existing hard policy, approval, sandbox, or authorization boundary.

## Solution Assessment

No, the proposed multi-phase package is not yet the narrowest maintainable solution; maintainers should first define the trust and compatibility contract, then implement typed hard-runtime capabilities rather than relying on prompt filtering or signature detection.
