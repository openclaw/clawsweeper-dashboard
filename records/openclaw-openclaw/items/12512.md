## Summary

Keep open. Current main still exposes prompt-visible `SKILL.md` paths and tells the model to read and follow selected skill files, while the requested trusted/untrusted boundary, role locking, per-skill memory/tool capability contract, and runtime monitoring are not implemented.

## What This Changes

The issue requests instruction isolation for `SKILL.md` content, including role locking, per-skill memory/tool capability metadata, and runtime monitoring.

## Best Possible Solution

Define a maintainer-approved trusted/untrusted skill precedence model, add typed per-skill role/memory/tool capability metadata, enforce it through prompt assembly and runtime policy, and cover malicious `SKILL.md` cases with focused tests and docs.

## Reproduction Assessment

Yes for the source-level missing-control behavior: current main lists prompt-visible `SKILL.md` locations and instructs the model to read and follow them. I did not run a live malicious-skill transcript, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear as a final implementation plan. The isolation direction is valid, but maintainers should choose a typed capability and prompt-precedence contract instead of landing only delimiter text or pattern filters.
