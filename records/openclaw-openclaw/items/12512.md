## Summary

Keep open: current main and the latest release still make selected SKILL.md files a model-read-and-follow instruction surface, while existing controls are visibility, eligibility, install/audit scanning, sandbox, and tool-policy hardening rather than the requested instruction-isolation, role, memory, and per-skill capability boundary.

## What This Changes

The issue requests instruction-isolation and permission boundaries for SKILL.md content so skill instructions cannot override agent/system behavior or influence tools and memory without policy enforcement.

## Best Possible Solution

Define one versioned skill trust and capability contract, then enforce it consistently through prompt selection, SKILL.md loading, tool execution, memory access, sandbox policy, tests, and docs.

## Solution Assessment

Unclear as a final implementation path. The requested security direction is valid, but the best fix needs a maintainer-approved typed trust and capability model enforced at runtime boundaries rather than prompt wording alone.
