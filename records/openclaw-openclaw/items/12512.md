## Summary

Keep open. Current main still uses a model-facing SKILL.md read/follow flow, and the existing controls cover visibility, eligibility, path/install safety, and prompt budgeting rather than the requested instruction isolation, role locking, per-skill memory/tool capability boundaries, or behavioral monitoring.

## What This Changes

Request to add a security boundary that isolates model-visible SKILL.md instructions from trusted agent context, including role locking, monitoring, and per-skill capability enforcement.

## Best Possible Solution

Define a maintainer-approved trusted/untrusted skill precedence model, add typed per-skill capability metadata, enforce it through prompt assembly and runtime policy, and cover malicious SKILL.md cases with docs and focused regression tests.

## Reproduction Assessment

Yes for the missing-control behavior: current main source shows prompt-visible skills are listed with SKILL.md locations and the model is told to read and follow one. A deterministic exploit transcript was not established, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear as a final implementation plan. The requested isolation direction is valid, but delimiter or pattern filtering alone would be weaker than a typed capability contract with explicit prompt precedence, runtime enforcement, compatibility tests, and docs.
