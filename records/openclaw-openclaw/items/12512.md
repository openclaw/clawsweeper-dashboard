## Summary

Keep open: current main and the latest release still make prompt-visible SKILL.md files model-followed instruction sources, while existing controls cover visibility, containment, install scanning, plugin prompt hooks, and broad tool policy rather than the requested per-skill instruction, role, memory, tool, and monitoring boundary.

## What This Changes

The issue requests instruction isolation and capability enforcement for SKILL.md content so skills cannot override agent role, memory, or tool behavior through prompt injection.

## Best Possible Solution

Define a maintainer-approved trusted/untrusted skill precedence model, add typed per-skill role, memory, and tool capability metadata, enforce it through prompt assembly and runtime policy, and cover malicious SKILL.md cases with focused tests and docs.

## Reproduction Assessment

Yes for source-level reproduction: current main lists prompt-visible SKILL.md locations and instructs the model to read and follow matching skills, while skill metadata lacks role, memory, tool, and monitoring capability fields. I did not run a live malicious-skill transcript in this read-only review.

## Solution Assessment

Unclear as a final implementation plan. The isolation direction is valid, but maintainers should choose a typed capability and prompt-precedence contract rather than landing only delimiter text, pattern filters, or scanner-only coverage.
