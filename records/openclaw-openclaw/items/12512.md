## Summary

Keep open. Current main still lists prompt-visible skills and tells the model to read and follow selected `SKILL.md` files; the requested instruction isolation, role/memory/tool capability contract, and monitoring remain unimplemented and require security design decisions.

## What This Changes

The issue requests instruction isolation for SKILL.md so skill content cannot override agent role, memory, or tool policy without an explicit capability contract.

## Best Possible Solution

Define a maintainer-approved trusted/untrusted skill precedence model, add typed per-skill role/memory/tool capability metadata, enforce it through prompt assembly and runtime policy, and cover malicious `SKILL.md` cases with focused tests and docs.

## Reproduction Assessment

Yes for the missing-control behavior: current main source shows prompt-visible skills are listed with `SKILL.md` locations and the model is told to read and follow one. I did not run a malicious skill transcript, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear as a final implementation plan. The isolation direction is valid, but maintainers should choose a typed capability and prompt-precedence contract rather than landing only delimiter or pattern-filter prompt text.
