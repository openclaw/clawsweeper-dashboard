## Summary

Keep open. Current main still exposes prompt-visible skills through a model-facing catalog and instructs the model to read and follow selected `SKILL.md` files, while the requested instruction isolation, role/memory/tool capability contract, and behavioral monitoring remain unimplemented and need maintainer security-design decisions.

## What This Changes

Request a core security architecture that isolates untrusted `SKILL.md` instructions from agent/system context and adds role, memory, tool, and monitoring boundaries for skills.

## Best Possible Solution

Define a maintainer-approved trusted/untrusted skill precedence model, add typed per-skill role/memory/tool capability metadata, enforce it through prompt assembly and runtime policy, and cover malicious `SKILL.md` cases with focused tests and docs.

## Reproduction Assessment

Yes for the missing-control behavior: current main source shows prompt-visible skills are listed with `SKILL.md` locations and the model is instructed to read and follow one. I did not run a malicious skill transcript, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Unclear as a final implementation plan. The isolation direction is valid, but maintainers should choose a typed capability and prompt-precedence contract rather than landing only delimiter text or ad hoc pattern filtering.
