## Summary

Keep open: current main and the latest release still expose skills as model-facing SKILL.md read-and-follow instructions, while the available controls cover visibility/invocation rather than instruction isolation, role locking, per-skill memory/tool authority, or behavioral monitoring.

## What This Changes

The issue requests a core security architecture that separates untrusted SKILL.md instructions from trusted agent/system context and adds role, memory, tool, and monitoring boundaries.

## Best Possible Solution

Define a maintainer-approved skill trust and capability contract, then enforce it consistently through prompt selection, SKILL.md loading, tool execution, memory access, sandbox policy, tests, and docs.

## Reproduction Assessment

Yes, at source level: current main instructs the model to read and follow selected SKILL.md files, while skill policy only exposes invocation and model-visibility controls. I did not run a live malicious-skill exploit.

## Solution Assessment

Unclear as a final implementation path: the requested direction addresses a real security-hardening gap, but the best fix needs a typed trust and capability model enforced at runtime boundaries rather than prompt wording alone.
