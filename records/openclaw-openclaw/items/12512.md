## Summary

Keep open: current main and v2026.6.8 still expose skills as model-facing read-and-follow SKILL.md instructions, while the implemented policy only controls visibility and invocation rather than role, memory, tool, or behavioral-monitoring boundaries.

## What This Changes

Request to add instruction isolation, role locking, capability policy, and monitoring around SKILL.md content so skill instructions cannot override agent role, memory, or tool authority.

## Best Possible Solution

Define one maintainer-approved skill trust and capability contract, then enforce it consistently through prompt selection, SKILL.md loading, tool execution, memory access, sandbox policy, tests, and docs.

## Reproduction Assessment

Yes, at source level: current main instructs the model to read and follow selected SKILL.md files, while skill policy only exposes invocation and model-visibility controls. I did not run a live malicious-skill exploit.

## Solution Assessment

Unclear as a final implementation path: the requested direction addresses a real security-hardening gap, but the best fix needs a typed trust and capability model enforced at runtime boundaries rather than prompt wording alone.
