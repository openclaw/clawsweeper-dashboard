## Summary

Current main and v2026.6.5 still expose a model-facing skills catalog and explicitly direct the agent to read and follow applicable SKILL.md instructions; existing controls cover discovery, install scanning, visibility, sandbox mounting, and agent-level tool policy, but no equivalent per-skill instruction, memory, or tool authority boundary was found. This remains necessary core security hardening, and the prior maintainer-side discussion also keeps it open pending a product and security design decision.

## What This Changes

Add enforceable isolation between trusted agent instructions and SKILL.md content, including per-skill limits on role, persistent-memory, and tool influence.

## Best Possible Solution

Define one typed skill trust policy with maintainer-approved defaults, enforce it at runtime-owned tool and persistent-memory boundaries across all skill invocation paths, and make prompt precedence accurately describe those guarantees rather than serving as the guarantee itself.

## Reproduction Assessment

Yes at the source-contract level: current main and its tests explicitly direct the model to read and follow applicable SKILL.md instructions, while no requested per-skill authority boundary is present. This read-only review did not execute a live malicious skill to establish every claimed exploit consequence.

## Solution Assessment

No, not exactly as proposed: delimiters, pattern detection, behavioral monitoring, and role-lock text are not sufficient authorization controls. The best solution is a typed policy enforced where tools and persistent memory are owned, with prompt text only communicating that enforced policy.
