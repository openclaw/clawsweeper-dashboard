## Summary

Keep open: current main and v2026.6.6 still expose model-visible skills by telling the agent to read and follow selected SKILL.md files, while the current policy surface only covers invocation and prompt visibility rather than a per-skill role, memory, or tool authority boundary.

## What This Changes

The issue requests instruction isolation and permission boundaries for SKILL.md content so skill instructions cannot override agent role, memory, or tool policy through prompt/context injection.

## Best Possible Solution

Adopt a maintainer-approved skill trust and capability contract, then enforce it consistently at prompt selection, SKILL.md loading, tool execution, memory access, sandbox policy, tests, and docs.

## Reproduction Assessment

Yes, at source level: current main instructs the model to read and follow selected SKILL.md files, while skill policy only exposes invocation and model-visibility controls. I did not run a live malicious-skill exploit.

## Solution Assessment

No final implementation path is approved yet. The requested direction is valid, but the best fix needs a typed trust/capability model enforced at runtime boundaries rather than prompt wording alone.
