## Summary

Current main and v2026.6.5 still expose a model-facing skills catalog and explicitly direct the agent to read and follow applicable SKILL.md instructions; existing controls cover discovery limits, visibility, installation, sandbox mounting, and agent-level tool policy, but no equivalent per-skill instruction, memory, or tool authority boundary is present. The request remains necessary core security hardening, while the proposed multi-phase architecture needs a maintainer product and security decision before implementation.

## What This Changes

Add instruction isolation and per-skill capability enforcement so untrusted SKILL.md content cannot override agent role, memory, or tool policy.

## Best Possible Solution

Define one maintainer-approved trusted-content and precedence model, add typed per-skill authority only where runtime enforcement is possible, enforce memory and tool access at canonical policy boundaries, and add malicious-skill regression tests plus upgrade documentation.

## Reproduction Assessment

Yes at the source level: current main tells the model to read and follow applicable SKILL.md content, while current skill metadata has no role, memory, or tool authority fields. This proves the missing boundary without claiming that a specific exploit was executed.

## Solution Assessment

Unclear as a concrete implementation: instruction isolation is the right problem area, but signature filtering, role locking, capability metadata, and monitoring should not become one implementation until maintainers define the trust, compatibility, and enforcement contract.
