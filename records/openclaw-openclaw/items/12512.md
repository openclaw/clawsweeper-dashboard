## Summary

Keep open. Current main still exposes prompt-visible skills through a SKILL.md read/follow flow, while existing controls cover visibility, eligibility, scanning, prompt budgeting, and tool/sandbox policy rather than the requested instruction isolation, role locking, memory/tool capability boundaries, or behavioral monitoring.

## What This Changes

The issue requests core security hardening so SKILL.md instructions are isolated from trusted agent/system instructions and governed by explicit role, memory, and tool capability boundaries.

## Best Possible Solution

Define a maintainer-approved trusted/untrusted skill precedence model, add typed per-skill capability metadata, enforce it through prompt assembly and runtime policy, and cover malicious SKILL.md cases with docs and focused regression tests.

## Reproduction Assessment

Yes for the missing-control behavior: current main source shows prompt-visible skills are listed with SKILL.md locations and the model is told to read and follow one. A deterministic malicious-skill exploit transcript is model-dependent, so I did not mark a runtime exploit as reproduced.

## Solution Assessment

Unclear as a final implementation plan. The requested isolation direction is valid, but delimiters or pattern filtering alone would be weaker than a typed capability contract with explicit prompt precedence, runtime enforcement, compatibility tests, and docs.
