## Summary

Keep open. Current main still exposes prompt-visible `SKILL.md` paths and tells the model to read and follow selected skill files; existing controls cover visibility, loading, plugin prompt hooks, and scanning, but not the requested instruction isolation, role locking, or per-skill memory/tool capability enforcement.

## What This Changes

Request an instruction-isolation and per-skill capability boundary so `SKILL.md` content cannot override agent role, memory, or tool policy.

## Best Possible Solution

Define a maintainer-approved trusted/untrusted skill precedence model, add typed per-skill role, memory, and tool capability metadata, enforce it through prompt assembly and runtime policy, and cover malicious `SKILL.md` cases with focused tests and docs.

## Reproduction Assessment

Yes for the source-level missing-control behavior: current main lists prompt-visible `SKILL.md` locations and instructs the model to read and follow them, while skill metadata lacks role, memory, tool, and monitoring capabilities. I did not run a live malicious-skill transcript.

## Solution Assessment

Unclear as a final implementation plan. The isolation direction is valid, but maintainers should choose a typed capability and prompt-precedence contract instead of landing only delimiter text or pattern filters.
