## Summary

Keep open. Current main still registers the bundled OpenRouter auto model as bare `auto`, while the catalog merge and Telegram `/models` picker paths preserve raw ids enough for `auto` and `openrouter/auto` to remain separate choices; the linked fix PR was closed unmerged.

## What This Changes

The issue asks to canonicalize the bundled OpenRouter auto model id so `auto` and `openrouter/auto` do not both appear in model pickers or persist as divergent session overrides.

## Best Possible Solution

Update the bundled OpenRouter provider catalog so the auto entry uses `openrouter/auto`, then add regression coverage that the catalog and picker surfaces expose only one OpenRouter auto choice while preserving unrelated OpenRouter model ids and media-understanding defaults.

## Reproduction Assessment

Yes. The issue provides concrete `/models` and session-state steps, and current main source plus the pinned dependency source show the same raw-id mismatch that would produce distinct `auto` and `openrouter/auto` entries.

## Solution Assessment

Yes. The narrowest maintainable fix is to canonicalize the bundled OpenRouter auto id and cover that behavior in tests; broad core/dependency normalization can remain a separate hardening follow-up.
