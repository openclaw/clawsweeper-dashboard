## Summary

Keep open. Current main still defines the bundled OpenRouter text catalog auto model as bare `auto`, while the pinned Pi catalog uses `openrouter/auto` and the Pi registry merge still compares raw provider/id strings; the closed fix PR did not land.

## What This Changes

The issue requests canonicalizing the bundled OpenRouter auto model id so the picker/catalog do not expose or persist a separate bare `auto` entry.

## Best Possible Solution

Canonicalize the bundled OpenRouter text catalog auto id to `openrouter/auto`, add regression coverage around catalog/picker dedupe, and preserve separate media-understanding defaults unless tests prove they share this text-model path.

## Reproduction Assessment

Yes. The issue gives concrete picker and session-state steps, and current main plus pinned upstream package source still show the raw-id mismatch needed to produce distinct `auto` and `openrouter/auto` catalog entries before higher-level normalization.

## Solution Assessment

Yes. The narrowest maintainable fix is to change the bundled OpenRouter text catalog id and add focused regression coverage; broader Pi registry normalization can remain a separate hardening follow-up.
