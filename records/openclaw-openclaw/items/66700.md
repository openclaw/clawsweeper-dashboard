## Summary

Keep this issue open. Current main still handles Nextcloud Talk webhook `object.content` as raw text and mention-gates group rooms by regexing that raw body, so rich-object placeholders such as `{mention-user1}` cannot reliably satisfy `requireMention: true`. The related #49869 PR is useful context for rich-object parsing, but it is open, not merged, and does not explicitly close this mention-gating issue.

## What This Changes

The issue requests that the bundled Nextcloud Talk plugin resolve `{mention-userN}` rich-object placeholders so bot mentions are detected in `requireMention: true` group rooms.

## Best Possible Solution

Add a plugin-local rich-object parser for Nextcloud Talk webhook content, carry enough resolved text and parameter-derived mention facts into `NextcloudTalkInboundMessage`, and have the group mention gate use the shared explicit mention/mention-decision helpers while preserving current raw-text behavior for ordinary messages. Add regression tests for `requireMention: true` accepting a bot `{mention-userN}` placeholder and rejecting a placeholder for another user.

## Reproduction Assessment

Yes. A source-level reproduction is clear: send a signed Nextcloud Talk `Create` webhook whose `object.content` JSON has `message: "hi {mention-user1}"` and a `parameters.mention-user1` entry for the bot; current main stores that JSON string as `text`, regex-checks the raw body, and drops the group message as unmentioned.

## Solution Assessment

Yes, fixing this inside `extensions/nextcloud-talk` is the right boundary. The safer implementation is not to disable `requireMention`, but to parse Nextcloud rich-object content and feed explicit mention evidence or resolved display text into the existing mention-gating contract.
