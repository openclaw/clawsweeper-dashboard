## Label Justifications

- `P1`: A supported nested subagent workflow can register but perform no work, stall the parent pipeline, and fail only after the 30-minute stale-context sweep.
- `impact:session-state`: The reported child session remains empty and its registered run terminates as lost execution context rather than producing a transcript.
- `impact:auth-provider`: The failure depends on mapping a canonical model provider and auth profile to the configured CLI execution runtime.

## Summary

Current main still reports and persists the canonical model provider at subagent spawn time, and the issue supplies a concrete v2026.6.5 cold depth-2 failure signature not covered by the merged followup repair. The issue should remain open, but the proposed persistence change needs validation because `modelProvider` may intentionally represent model identity while CLI execution runtime is resolved separately downstream.

## What This Changes

Resolve the configured CLI execution runtime early enough that cold depth-2 subagent spawns launch correctly, retain usable runtime state, and return accurate spawn metadata.

## Best Possible Solution

Prepare one canonical spawn execution plan containing separate model-provider, auth-provider, and CLI execution-runtime facts, then use the execution runtime for launch and active-run tracking while preserving canonical model identity in session state unless a tested contract proves otherwise.

## Reproduction Assessment

Yes at source level: current main deterministically persists and returns the raw canonical provider, and the sweeper contains the exact terminal error path described. A live cold depth-2 CLI launch reproduction is still needed to prove which pre-dispatch consumer causes the missing process.

## Solution Assessment

No, not yet as written: reusing `resolveCliRuntimeExecutionProvider` is the right primitive, but directly replacing persisted `modelProvider` with a runtime ID may mix two identities; the safer fix carries a separate resolved execution provider through registration and dispatch and proves upgrade-neutral behavior.
