## Summary

Keep open: current main and v2026.6.10 still leave the subagent/gateway provider metadata unresolved, while the downstream runner already resolves CLI execution, so this needs maintainer boundary choice plus live current-main depth-2 proof.

## What This Changes

The issue reports CLI-backed depth-2 subagent spawns being registered with a canonical provider instead of the CLI execution runtime, leading cold child runs to be swept as lost active context.

## Best Possible Solution

Keep session model/provider state canonical, choose a maintainer-approved execution-runtime resolution boundary for nested spawns, and prove cold depth-2 CLI-backed children launch and write transcripts.

## Reproduction Assessment

No high-confidence live current-main reproduction was established in this review. Source inspection and v2026.6.10 evidence confirm the reported raw-provider surfaces remain, but current attempt execution may already resolve CLI runtime at dispatch.

## Solution Assessment

Unclear. The original write-CLI-runtime-into-`modelProvider` shape is not the best fix; a dispatch-time execution-provider handoff that preserves canonical session state is safer and needs maintainer direction.
