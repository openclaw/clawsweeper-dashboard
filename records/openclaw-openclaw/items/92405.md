## Summary

Keep open: current main and the latest release still have the reported subagent/gateway resolver gap, and the related broad CLI-helper issue remains open rather than owning a merged spawn fix. The failure is source-plausible, but the proposed fix changes a provider/session-state boundary and should get maintainer-directed live reproduction and product review before implementation.

## What This Changes

The issue reports that nested `sessions_spawn` runs with CLI-backed canonical models persist/register the canonical provider instead of the executable CLI runtime, causing cold depth-2 child runs to lose execution context.

## Best Possible Solution

Keep the issue open and fix the runtime dispatch boundary so cold nested spawns resolve the CLI execution provider before registration/launch while preserving canonical model identity, with focused regression coverage and live CLI-backed proof.

## Reproduction Assessment

No high-confidence live repro was run here. Source inspection on current main clearly shows the raw provider persistence/registration path, but the exact 30-minute cold depth-2 CLI failure still needs a real CLI-backed reproduction.

## Solution Assessment

No, the proposed persistence-only fix is not proven to be the best fix yet. A safer owner-boundary solution should decide whether execution runtime belongs in session `modelProvider`, a separate registration/runtime field, or gateway dispatch metadata while retaining canonical model identity.
