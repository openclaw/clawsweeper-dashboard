## Summary

Current main still has a credible runtime-selection gap: child sessions persist canonical model identity, gateway admission records that canonical provider, and sibling CLI-backed paths explicitly resolve an execution runtime before dispatch. The issue should remain open, but its proposed persistence change is not yet safe because repository tests treat `modelProvider` as canonical identity and keep CLI runtime bindings separate; a maintainer should choose and prove one authoritative execution-resolution boundary with a live cold depth-2 reproduction.

## What This Changes

Fix cold depth-2 `sessions_spawn` runs that are accepted under a canonical provider but may never launch the configured CLI runtime.

## Best Possible Solution

Preserve canonical provider/model identity in session state, resolve the configured CLI execution runtime once at the earliest authoritative run-attempt boundary shared by nested spawns, and add focused plus live coverage for a binding-free depth-2 launch, transcript production, and terminal cleanup.

## Reproduction Assessment

No, not at high confidence yet. Current source confirms the raw canonical provider, the sibling resolver pattern, and the reported lost-context terminal path, but a live current-main cold depth-2 run or an end-to-end dispatch trace is still needed to prove that this routing gap causes the missing process launch.

## Solution Assessment

No, not exactly as proposed. Reusing `resolveCliRuntimeExecutionProvider` is directionally consistent with merged sibling work, but overwriting persisted `modelProvider` conflicts with existing canonical session-state expectations; the safer fix resolves execution runtime at a single authoritative run boundary.
