## What This Changes

The issue reports depth-2 CLI-backed subagent spawns losing active execution context when spawn/gateway registration uses the raw canonical provider instead of the configured execution runtime.

## Best Possible Solution

Keep this focused issue open until subagent spawn and gateway active-run registration resolve the execution runtime at the dispatch/runner boundary without corrupting canonical model/session state, with focused regression coverage and live cold depth-2 CLI proof.

## Reproduction Assessment

Yes at source level, but not as a live current-main run in this review. Current main still splits and persists the raw provider for subagent child model metadata, and gateway active-run registration reads that provider before dispatch.

## Solution Assessment

Unclear as a final implementation. The shared resolveCliRuntimeExecutionProvider pattern is likely the right direction, but maintainers need to choose the dispatch/session boundary so canonical modelProvider is not conflated with execution runtime identity.
