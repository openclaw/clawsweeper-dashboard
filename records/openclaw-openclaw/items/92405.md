## Best Possible Solution

Keep this open until a live cold depth-2 CLI-backed spawn proves where execution context is lost, then carry the CLI execution provider through the gateway/runner boundary without storing it as canonical `modelProvider`.

## Solution Assessment

No, the original persistence-side fix is not proven as the best solution because it would mix canonical model state with execution runtime identity. The safer direction is to preserve canonical `modelProvider` and carry a prepared execution provider through the registration or runner-selection boundary after live tracing confirms the gap.
