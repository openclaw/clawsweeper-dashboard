## Summary

Keep this open: current main still shows the raw canonical provider being persisted/reported in the spawn path, while the proposed fix shape crosses provider/auth/session-state boundaries and needs maintainer-directed live proof before implementation.

## What This Changes

The issue reports that nested `sessions_spawn` runs persist a canonical provider instead of the configured CLI execution runtime, causing cold depth-2 CLI-backed child sessions to lose execution context.

## Best Possible Solution

Preserve canonical `modelProvider`/`model` session identity, live-trace a cold depth-2 canonical-model spawn with a configured CLI runtime, and if the launch still loses context, carry a prepared execution-provider value through the registration/dispatch boundary with focused regression coverage.

## Reproduction Assessment

No high-confidence live reproduction in this pass. Source inspection shows raw canonical provider persistence and raw active-provider registration on current main, but the 30-minute cold depth-2 CLI-backed spawn was not run against the latest code.

## Solution Assessment

No, the issue's proposed persistence patch is not clearly the best fix because it may store execution runtime ids where docs and code expect canonical model providers. The safer solution is to keep canonical model state and add an explicit execution-runtime handoff at the runner or registration boundary after live proof.
