## Summary

Keep this open: current main still has an unresolved, compatibility-sensitive subagent/runtime routing question, and the exact cold depth-2 failure has not been disproved on current main. Source inspection shows the direct runner now resolves canonical Anthropic models to `claude-cli`, but the reported spawn surfaces still persist and report the canonical provider, so maintainers need live proof before choosing the execution-boundary fix.

## What This Changes

The issue reports that nested `sessions_spawn` runs with canonical Anthropic models mapped to `claude-cli` can stall and later fail with lost execution context because spawn paths retain the canonical provider instead of the CLI execution runtime.

## Best Possible Solution

Keep canonical session model identity intact, prove the cold depth-2 `claude-cli` spawn path live, then carry a prepared execution-provider fact through the gateway/agent boundary only where runner selection needs it.

## Reproduction Assessment

No, not at high confidence for current main. The source path and 2026.6.5 live evidence are credible, but the current attempt-execution resolver may already route the child run to `claude-cli`, so a fresh cold depth-2 run is needed.

## Solution Assessment

No, the proposed persistence fix is not the best solution as written. The safer direction is to preserve canonical `modelProvider` state and resolve or carry the CLI execution provider at the runner-selection boundary.
