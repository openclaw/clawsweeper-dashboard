## Summary

Current main still has a credible cold nested-spawn runtime-selection gap, and the issue provides strong source and released-runtime evidence, but its proposed rewrite of `modelProvider` and `resolvedProvider` conflicts with the repository’s explicit separation of canonical model identity from CLI execution runtime. Keep this open for a focused maintainer-owned reproduction and dispatch-boundary fix rather than applying the suggested persistence change verbatim.

## What This Changes

Fix cold depth-2 subagent runs that register but fail to launch when a canonical model provider is configured to execute through a CLI runtime.

## Best Possible Solution

Preserve canonical provider/model identity in session and spawn-response metadata, reproduce the cold depth-2 path, then resolve the configured CLI runtime at the earliest shared execution-dispatch boundary that currently loses it, with regression coverage for both cold CLI-backed and normal embedded/API-backed spawns.

## Solution Assessment

No, not as proposed: earlier runtime selection is a plausible fix direction, but rewriting canonical `modelProvider` and `resolvedProvider` conflicts with existing identity/runtime contracts. The safer fix is at a shared execution-dispatch boundary after reproducing the cold path.
