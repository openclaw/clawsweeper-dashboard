## Summary

Keep this open for maintainer follow-up: current main still stores and reports the raw canonical provider at the spawn/session surfaces, but the actual attempt path also has a downstream CLI-runtime resolver, so the reported cold depth-2 lost-context failure is plausible but not proven enough to close or auto-repair without a live repro and an owner decision on the right execution-runtime boundary.

## What This Changes

The issue requests fixing nested `sessions_spawn` so CLI-backed child runs use the configured CLI execution runtime instead of only persisting/reporting the raw canonical provider.

## Best Possible Solution

Resolve the CLI execution runtime at the dispatch/registration boundary while preserving canonical model identity in session state, then prove a cold depth-2 CLI-backed spawn launches and records a transcript.

## Reproduction Assessment

No high-confidence current-main live reproduction was established. Source inspection confirms raw provider persistence/reporting at the spawn and gateway surfaces, but current attempt-execution code also resolves canonical providers to CLI runtimes before launch.

## Solution Assessment

No, the suggested fix is not yet proven to be the best boundary. Persisting `claude-cli` as `modelProvider` could blur model identity and execution runtime, so maintainers should choose a narrower dispatch/registration fix and require live cold-spawn proof.
