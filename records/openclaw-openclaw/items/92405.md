## Summary

Keep open: current `main` and latest release still leave the reported spawn/gateway call sites without the CLI-runtime resolver, and no merged PR or open specific fix owns this exact nested-spawn gap. The issue needs maintainer-directed live reproduction and a product boundary decision before an automated fix because the proposed persistence fix touches session model/runtime identity.

## What This Changes

The issue reports that nested `sessions_spawn` persists a canonical provider instead of the configured CLI execution runtime, causing cold depth-2 CLI-backed child runs to lose execution context.

## Best Possible Solution

Resolve and carry the configured CLI execution provider at the execution-owner boundary for spawned child runs, keep model identity canonical or explicitly normalized where required, and add a cold depth-2 regression plus real CLI launch/transcript proof.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. Source inspection confirms the named resolver is still absent from the reported spawn/gateway call sites, but the full cold depth-2 CLI-backed failure needs a live trace.

## Solution Assessment

Unclear as proposed: resolving the CLI runtime for child execution is the right direction, but persisting or registering that value needs maintainer boundary judgment so canonical model state, auth mapping, thinking defaults, and runtime routing do not drift.
