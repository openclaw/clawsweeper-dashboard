## Summary

Keep open: current main does not contain a focused fix for the reported nested subagent CLI-runtime boundary, but the proposed fix changes documented/session model identity semantics and needs maintainer-directed live proof before implementation.

## What This Changes

The issue reports that nested `sessions_spawn` runs with canonical provider model refs can retain raw provider metadata instead of the configured CLI execution runtime, causing cold depth-2 child runs to stall and later fail as lost execution context.

## Best Possible Solution

Keep the issue open for a maintainer-owned fix that preserves canonical model identity, carries the resolved CLI execution provider at the runner/registration boundary, and proves a cold depth-2 spawn with a real CLI process and transcript.

## Reproduction Assessment

No, not at high confidence for the full failure. Source confirms the raw-provider metadata path, but current downstream attempt execution also resolves canonical providers to CLI runtimes, so a live cold depth-2 spawn trace is still needed.

## Solution Assessment

No, the proposed persistence/ack change is not clearly the best fix because it changes documented provider-prefix semantics. The safer direction is to carry a separate execution-runtime fact at the dispatch boundary while preserving canonical session model identity.
