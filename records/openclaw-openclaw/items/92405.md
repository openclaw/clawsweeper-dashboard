## Summary

Keep open: current main already resolves canonical providers to CLI execution providers at the runner boundary, so the proposed `modelProvider` persistence rewrite is not the best fix; the lost-context symptom still has a source-reproducible current-main path through pending subagent completion handoff handling and is now closely tracked by https://github.com/openclaw/openclaw/issues/93323 and related open PRs.

## What This Changes

The issue reports cold nested `sessions_spawn` runs for CLI-backed Anthropic models stalling with lost execution context and proposes resolving the execution runtime before child registration or dispatch.

## Best Possible Solution

Preserve canonical provider/model session state, keep CLI runtime selection at execution/dispatch boundaries, and land one maintainer-approved fix for pending subagent completion handoffs with live or gateway-level proof.

## Reproduction Assessment

Yes at source level, but not as a live run in this review. Current main still contains the pending direct-announce branch that can return `completion_handoff_pending`, and the registry still has the stale-context failure path that produces the reported terminal error.

## Solution Assessment

No for the proposed raw-provider persistence patch. The safer solution is to preserve canonical model identity, reuse `resolveCliRuntimeExecutionProvider` only at execution/dispatch boundaries, and resolve the pending completion-handoff behavior tracked by the newer related issue and PRs.
