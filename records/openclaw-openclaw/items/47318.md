## Summary

Keep this PR open, but it is not merge-ready: the main user problem is still real on current main, while the branch still warns from raw model text instead of the canonical model resolver, misses a supported per-agent primary path, prints unsanitized user-controlled text, and carries unrelated generated/runtime churn.

## What This Changes

The PR adds a non-blocking `openclaw config set` warning for selected agent model config paths when the model ref is missing from the model catalog.

## Best Possible Solution

Keep a narrow config-set warning, but implement it against the canonical post-mutation model resolver and alias index, cover supported default and per-agent primary paths, sanitize displayed refs, and remove unrelated generated/formatting changes.

## Reproduction Assessment

Yes. By source inspection, current main routes `config set` through `runConfigOperations`, normalizes and writes the config, and has no model-catalog warning hook for invalid model refs.

## Solution Assessment

No. A non-blocking warning is a reasonable direction, but this patch should resolve the normalized post-write model with the runtime alias/profile-aware resolver, cover per-agent `.model.primary`, sanitize terminal output, and drop unrelated churn.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Resolve model refs with the runtime resolver before warning:** `src/cli/config-cli.ts:770`
  - body: `warnIfModelNotInCatalog` parses the raw CLI value with `DEFAULT_PROVIDER` only. Valid configured aliases from `agents.defaults.models` are resolved by runtime through `buildModelAliasIndex`/`resolveModelRefFromString`, but this code reports them as missing catalog models, so the warning can mislead users for supported config values.
  - confidence: 0.9
- **[P2] Detect per-agent primary model paths:** `src/cli/config-cli.ts:752`
  - body: The detector matches `agents.list.<n>.model` but not `agents.list.<n>.model.primary`. Since agent model config can be an object with `primary` and `fallbacks`, setting a bad per-agent primary through `agents.list[0].model.primary` still follows the old silent-success path this PR is meant to fix.
  - confidence: 0.87
- **[P3] Sanitize model refs before writing warnings:** `src/cli/config-cli.ts:793`
  - body: The warning prints user-provided model text back to the terminal without the existing log sanitizer. A ref containing ANSI/control characters can spoof or obscure diagnostics, so route displayed refs through `sanitizeForLog` or the model warning sanitizer before interpolation.
  - confidence: 0.84
