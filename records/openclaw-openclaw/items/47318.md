## Summary

Keep this PR open: it targets a real config/model diagnostic gap, but the current branch is not merge-ready because the warning path still bypasses shared post-change model resolution, misses supported per-agent primary paths, prints unsanitized config-derived model text, is merge-conflicting, and carries unrelated generated/formatting churn.

## What This Changes

The branch adds a non-blocking model-catalog warning for selected `openclaw config set` agent model paths, plus focused tests and unrelated generated/formatting changes across plugin, script, agent, cron, gateway, and locale files.

## Best Possible Solution

Implement the diagnostic through the shared post-change model-resolution path, cover default and per-agent string/object primary shapes, sanitize all displayed model text, keep the branch scoped to config/model diagnostics, and require redacted real CLI proof for invalid and valid refs.

## Reproduction Assessment

Yes, by source inspection. Current main’s `runConfigOperations` accepts and writes config changes without a model-catalog warning hook, so the linked invalid-model scenario would still be accepted silently by `config set`.

## Solution Assessment

No, not as currently implemented. A warning-only diagnostic is a reasonable direction, but it should use the shared resolver against the post-change config and avoid unrelated branch churn.
