## Summary

Keep open. Current main still matches the reported Discord startup regression: application-ID lookup can now avoid `/oauth2/applications/@me` when configured or token-derived data is available, but native slash command deploy still retries Discord 429s up to three times during startup and bot identity lookup still hard-fails startup when `fetchUser("@me")` times out.

## What This Changes

Prevent Discord startup from timing out on OAuth/bot identity probes or flooding native slash command deploy PATCH retries/rate limits after v2026.4.29.

## Best Possible Solution

Discord startup should decouple core send/receive from optional REST-heavy setup: use configured or safely parsed IDs where available, make native command deploy 429s a single non-fatal skip or cooldown during startup, and let bot identity lookup failures degrade with clear status instead of repeatedly blocking monitor startup.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: mocking `client.deployCommands` to throw a Discord 429 follows the current three-attempt path, and forcing `fetchUser("@me")` to reject follows the hard startup failure path.

## Solution Assessment

No. Current main's application-ID mitigation is useful, but it is not the full fix because the deploy retry burst and hard bot identity startup dependency remain.
