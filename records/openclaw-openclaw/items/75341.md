## Summary

Keep open. Current main still matches the important parts of the Discord regression: the application-id lookup now has a useful configured/token-derived bypass, but native slash command deploy still retries a 429 three times during startup and bot identity resolution still awaits `fetchUser("@me")` as a startup-critical path.

## What This Changes

The issue asks OpenClaw to prevent Discord v2026.4.29 startup from timing out on OAuth/user `@me` probes and from amplifying native slash-command deploy PATCH 429s.

## Best Possible Solution

Discord startup should avoid turning upstream REST trouble into a startup failure or burst: keep the application-id bypass, make deploy 429s a single non-fatal cooldown/skip during startup, and let bot identity lookup degrade to cached or partial identity where message send/receive can still run safely.

## Reproduction Assessment

Yes. A focused mock reproduction is high-confidence: make `client.deployCommands` throw a Discord 429 to observe three deploy attempts, and make `fetchUser("@me")` reject to observe startup failing before lifecycle; only the reporter's exact high-RTT timing needs live conditions.

## Solution Assessment

No, current main is not the best final solution. It mitigates the `/oauth2/applications/@me` lookup path but leaves the startup deploy retry burst and hard bot identity dependency in place.
