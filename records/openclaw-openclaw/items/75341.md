## Summary

Keep open. Current main still preserves the reported Discord startup failure shape: one mitigation avoids some `/oauth2/applications/@me` calls via configured or token-parsed application IDs, but native slash-command deploy still retries 429s up to three startup attempts and bot identity lookup still hard-fails startup when `fetchUser("@me")` times out or rejects.

## What This Changes

The issue reports a v2026.4.29 Discord startup regression where OAuth/user identity probes time out and native slash-command deploy retries hit HTTP 429 during startup.

## Best Possible Solution

Discord startup should degrade instead of amplifying upstream REST trouble: avoid unnecessary application lookup, make native command deploy 429s a bounded non-fatal skip or cooldown during startup, and allow message send/receive plus gateway lifecycle to come up even when bot identity lookup is temporarily unavailable.

## Reproduction Assessment

Yes. The current tests already provide a high-confidence source-level reproduction path: mocking `client.deployCommands` to throw a Discord 429 produces three startup deploy attempts, and mocking `fetchUser("@me")` to reject makes startup throw before lifecycle begins.

## Solution Assessment

No. Current main's mitigation is useful but incomplete because it only reduces one application lookup path and log noise while leaving the deploy retry burst and hard bot identity startup dependency in place.
