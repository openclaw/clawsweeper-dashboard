## Summary

Keep open. Current main still preserves the reported Discord startup amplification: native slash-command deploy can retry ordinary 429s up to three times, and bot identity resolution still hard-fails startup on `fetchUser("@me")` errors even though the application-id lookup has a configured/parsed bypass.

## What This Changes

The issue reports a Discord v2026.4.29 startup regression where OAuth/app identity probes time out and native slash command deployment hits repeated HTTP 429 responses.

## Best Possible Solution

Discord startup should avoid amplifying REST trouble: use configured or parsed identity where safe, make ordinary deploy 429s a single non-fatal skip or cooldown that leaves existing commands active, and add focused tests for deploy 429s plus `fetchUser("@me")` timeout/fallback behavior.

## Reproduction Assessment

Partially yes. The native command 429 path has a high-confidence source/test reproduction because current main expects three deploy attempts; the live `@me` timeout path still depends on reporter network conditions, but current source makes that lookup startup-critical.

## Solution Assessment

No for current main. The existing mitigation improves logging and application-id lookup bypasses, but it does not prevent the deploy retry burst or make the bot identity timeout path non-amplifying.
