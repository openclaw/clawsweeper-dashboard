## Summary

Keep open. Current main has useful post-release Discord hardening, but it still preserves the two central failure paths from the report: startup native slash-command deploy retries rate-limited deploys up to three times, and bot identity resolution still hard-fails on `fetchUser("@me")`. The reporter provided concrete v2026.4.29 runtime logs, and the report remains distinct from the SecretRef regression in #74765.

## What This Changes

The issue asks Discord startup to avoid v2026.4.29 OAuth/user `@me` fetch timeouts and native slash-command deploy HTTP 429 bursts during gateway startup.

## Best Possible Solution

Discord startup should avoid amplifying REST trouble: prefer configured or token-derived application IDs, make bot identity timeouts non-amplifying where a usable fallback exists, and treat native deploy 429s as a single non-fatal skip or cooldown that leaves existing commands active. Targeted Discord startup/deploy tests should cover the 429 and `fetchUser("@me")` timeout paths.

## Reproduction Assessment

Partially yes. The deploy burst is source- and test-reproducible on current main because a repeated 429 leads to three deploy attempts; the exact `/users/@me` and `/oauth2/applications/@me` timeout timing depends on the reporter's network/Discord conditions, but the hard startup fetch paths are visible in source.

## Solution Assessment

No. Current main's partial hardening is not the best final solution because it mainly quiets or hardens rate-limit handling while preserving repeated deploy attempts and a hard bot-identity fetch failure path.
