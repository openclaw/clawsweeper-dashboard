## Summary

Keep open. Current main still preserves the reported Discord startup amplification: ordinary native slash-command deploy 429s can be retried up to three times, the monitor still hard-fails startup when `fetchUser("@me")` fails, and a startup pre-monitor probe can still call Discord user/application endpoints before monitoring starts.

## What This Changes

The issue reports a Discord v2026.4.29 startup regression where OAuth/app identity probes time out and native slash-command deployment hits repeated HTTP 429 responses.

## Best Possible Solution

Discord startup should avoid amplifying REST trouble: treat ordinary deploy 429s as a single non-fatal skip or cooldown that leaves existing commands active, avoid unnecessary application/user probes when configured or parsed identity is available, and keep startup fail-closed only when no safe bot identity can be derived.

## Reproduction Assessment

Yes for the deploy 429 path: current source and tests show three deploy attempts on ordinary rate limits. Unclear for the live `@me` timeout timing without the reporter's host/network, but current source clearly keeps `fetchUser("@me")` startup-critical.

## Solution Assessment

No for current main. The logging and application-id mitigations are useful but incomplete because they do not stop the retry burst, do not cover the startup pre-monitor probe, and do not make bot identity timeouts non-amplifying.
