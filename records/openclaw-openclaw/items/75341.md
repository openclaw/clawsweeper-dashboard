## Summary

Keep open. Current main still matches the core of the reported Discord startup regression: application ID lookup has a documented/configured-token mitigation, but native command deploy can still retry short 429s three times during startup and Discord bot identity lookup still fails the monitor before lifecycle starts.

## What This Changes

The issue reports a Discord v2026.4.29 startup regression where OAuth identity probes time out and native slash-command deploy PATCH calls hit HTTP 429 during gateway startup.

## Best Possible Solution

Discord startup should keep send/receive usable under upstream REST trouble: avoid a startup deploy retry burst on 429, log one actionable non-fatal cooldown/skip, preserve existing slash commands, and make bot identity lookup degrade or defer when a safe application/bot ID is already available.

## Reproduction Assessment

Yes. Source and existing tests provide deterministic paths: a mocked deploy 429 currently produces three deploy attempts, and a mocked `fetchUser("@me")` rejection currently fails monitor startup before lifecycle.

## Solution Assessment

No. Current main's application-ID mitigation is useful but incomplete because it leaves the deploy 429 amplification and the hard `/users/@me` startup dependency in place.
