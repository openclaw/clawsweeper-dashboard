## Summary

Keep open. Current main only partially mitigates the reported Discord v2026.4.29 regression: native slash-command deploy rate-limit logs are collapsed, but the source and tests still allow three deploy attempts on a 429, and bot identity lookup via `fetchUser("@me")` remains a hard startup dependency.

## What This Changes

The issue requests a Discord startup fix so OAuth/app identity fetch timeouts and native slash-command deploy 429s do not make v2026.4.29 startup unreliable or amplify REST traffic.

## Best Possible Solution

Discord startup should avoid amplifying REST trouble: use the existing configured/parsed application ID path where safe, avoid native command deployment while identity preflight is failing or rate-limited, and treat deploy 429s as a single non-fatal skip or cooldown that leaves existing commands active. Add focused Discord monitor/deploy tests for the 429 and `fetchUser("@me")` timeout paths.

## Reproduction Assessment

Yes for the native command 429 path: current source and tests show a 429 can produce three deploy attempts. Unclear for the live `@me` timeout path without the reporter's network conditions, but current main visibly makes `fetchUser("@me")` startup-critical.

## Solution Assessment

No. Current main's mitigation is useful but incomplete because it collapses repeated deploy 429 logging without preventing the retry burst or making the identity timeout path non-amplifying.
