## Summary

Keep open. Current main still only partially mitigates the reported Discord regression: application-ID resolution can avoid the `/oauth2/applications/@me` call and deploy rate-limit logging is quieter, but native command deploy still retries a 429 up to three times and bot identity lookup via `fetchUser("@me")` remains a hard startup dependency.

## What This Changes

The issue requests a Discord startup fix so v2026.4.29 no longer times out on Discord `@me` probes or bursts native slash-command deploy PATCH retries into 429s.

## Best Possible Solution

Discord startup should avoid amplifying upstream REST trouble: use configured or parsed application IDs where safe, make native command deploy 429s a single non-fatal skip or cooldown during startup, and ensure bot identity lookup failures degrade without repeatedly blocking startup while existing slash commands and message send/receive remain active.

## Reproduction Assessment

Yes, enough to act. Mocking `client.deployCommands` to throw a Discord 429 reproduces the three-attempt startup burst, and forcing `fetchUser("@me")` to reject reproduces the hard startup failure; only the reporter's exact high-RTT timeout timing needs live conditions.

## Solution Assessment

No. Current main's mitigation is useful but incomplete because it quiets logs and avoids one application lookup path while leaving the retry burst and hard bot identity startup dependency in place.
