## Summary

Keep open. Current main still relies on compaction prompt guidance, bounded identifier quality checks, and file-operation suffixes rather than a deterministic exact path-literal safeguard, and open PR #75093 remains the direct closing implementation candidate.

## What This Changes

The issue reports that context compaction can corrupt slash-separated file paths by inserting spaces before later tool calls.

## Best Possible Solution

Land PR #75093, or an equivalent narrow compaction safeguard change, so bounded path-like literals are preserved exactly across provider and built-in compaction without exposing hidden tool-result details.

## Reproduction Assessment

No. The reporter provided concrete failed tool-call paths, but a high-confidence reproduction would need the original long transcript or a deterministic MiniMax compaction replay; source review confirms the failure remains plausible on current main.

## Solution Assessment

Yes. Deterministic path-literal preservation in the compaction suffix is the narrow maintainable direction, and it is safer than relying on provider prompt adherence alone.
