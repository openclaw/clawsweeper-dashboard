## Summary

Keep open: current main still stores one browser-local assistant avatar override and applies it across agent identities, while the gateway/config path already models assistant avatar identity per agent.

## What This Changes

The issue reports that assigning an avatar from Quick Settings updates the avatar for every agent instead of storing a separate avatar per selected agent.

## Best Possible Solution

Store and load browser-local assistant avatar overrides by active agent id while preserving gateway-resolved per-agent identity fallback and a narrow legacy handling path for the old single override value.

## Reproduction Assessment

Yes, from source inspection: uploading an assistant avatar writes one browser-local assistant identity value, and the render/controller path applies that value regardless of the selected agent. I did not execute the browser flow in this read-only review.

## Solution Assessment

Yes: the narrow fix is to key the local Quick Settings assistant avatar override by agent id and leave the existing gateway/config per-agent identity contract intact. A config option or config.patch write is not needed.
