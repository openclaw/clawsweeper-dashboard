## Summary

Keep open: current main and the latest release still store the Control UI assistant avatar override in one global browser-local key, and four open linked PRs are already trying to close the issue. The next useful maintainer action is to choose and land one canonical scoped-storage fix, not close the issue or queue another repair branch.

## What This Changes

The issue asks for Control UI assistant avatar overrides to be scoped per selected agent instead of overwriting all agents through shared settings.

## Best Possible Solution

Land one canonical PR that scopes the assistant avatar override by current agent id across storage, render, assistant-identity, and bootstrap paths, with focused tests and browser proof for switch, reload, clear, and legacy fallback behavior.

## Reproduction Assessment

Yes. Source inspection shows every assistant-avatar override path reads or writes the same `openclaw.control.assistant.v1` localStorage key without using `assistantAgentId`, matching the reported cross-agent overwrite.

## Solution Assessment

Yes. Per-agent scoping in the Control UI storage helper plus threading the existing `assistantAgentId` through all callers is the narrowest maintainable fix; the remaining decision is which linked PR has the safest upgrade and proof story.
