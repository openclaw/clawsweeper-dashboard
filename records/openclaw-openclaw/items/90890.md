## Summary

This should stay open because the reported avatar bleed is still present on current main and in the latest release, while the linked PRs are open implementation candidates rather than merged fixes.

## What This Changes

The issue requests per-agent scoping for the Control UI assistant avatar override instead of one browser-local avatar being shared by every agent.

## Best Possible Solution

Land or adopt one focused linked PR that scopes local assistant avatar storage by agent id and deliberately preserves or migrates the existing global avatar value.

## Reproduction Assessment

Yes, at source level: current main reads and writes the assistant avatar override through one localStorage key and every caller uses that unscoped helper. I did not run a live browser session, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes: the narrow maintainable fix is to scope the existing local assistant avatar override by agent id and handle the legacy global key intentionally. The broader Agents-tab avatar upload request remains separate from this bug.
