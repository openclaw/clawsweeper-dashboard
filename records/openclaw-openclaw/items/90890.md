## Label Justifications

- `P2`: The bug affects per-agent Control UI avatar state but does not make the core runtime unusable.
- `impact:session-state`: The local assistant avatar preference is shared across agent sessions instead of staying scoped to the selected agent.

## Summary

Keep open. Current main and the latest release still use one unscoped browser-local assistant avatar key, and several open PRs already reference this issue with closing syntax, so the right cleanup path is to review and land one canonical fix rather than close or queue another repair.

## What This Changes

The issue reports that Control UI assistant avatar overrides are stored globally instead of per agent, so changing one agent's avatar changes every agent's avatar.

## Best Possible Solution

Land one canonical Control UI fix that scopes browser-local assistant avatar storage by agent ID, threads the active agent ID through all override read/write paths, and handles the legacy global key intentionally.

## Reproduction Assessment

Yes, from source: current main still calls loadLocalAssistantIdentity() and saveLocalAssistantIdentity() without an agent ID, and storage.ts uses the single key openclaw.control.assistant.v1. I did not run a live browser repro in this read-only pass.

## Solution Assessment

Yes, the narrow maintainable fix is to scope the local assistant avatar key by agent ID in the existing Control UI storage helper and pass assistantAgentId through the existing callers. The issue itself should remain open until one linked PR lands.
