## Summary

Keep this open: current main and the latest release still use one unscoped browser-local assistant avatar key, and four open PRs already reference this issue with closing syntax, so it should close only after one fix lands.

## What This Changes

The issue requests that Control UI assistant avatar overrides be stored per selected agent instead of shared across all agents in the same browser.

## Best Possible Solution

Review and land one focused closing PR that scopes the browser-local assistant avatar override by normalized agent id while preserving existing local avatar data, then close the duplicate PRs after this issue is fixed.

## Reproduction Assessment

Yes from source: current main reads and writes a single `openclaw.control.assistant.v1` localStorage entry regardless of `assistantAgentId`, and an existing bootstrap test shows that global local override applied to a worker agent. I did not run a live browser repro because this review is read-only.

## Solution Assessment

Yes: the narrow maintainable fix is to scope the browser-local assistant avatar storage and all load/save callers by the active agent id, with regression coverage for switching agents and legacy local avatar preservation.
