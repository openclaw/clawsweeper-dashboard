## Label Justifications

- `P2`: This is a normal-priority Control UI bug with limited blast radius: changing one agent's browser-local avatar affects other agents.
- `impact:session-state`: The affected state is per-agent assistant identity UI state being shared across agents in the same browser profile.

## Summary

Keep open: current main and the latest local release tag still store the browser-local assistant avatar override under one unscoped key, and all Control UI callers read or write it without an agent ID. Multiple open PRs already specifically target this issue, so the issue should wait for one reviewed fix to merge rather than being closed now.

## What This Changes

The issue requests that the Control UI assistant avatar override be bound to the selected agent instead of overwriting every agent in the browser.

## Best Possible Solution

Land one narrow Control UI fix that scopes the local assistant avatar override by normalized active agent ID across storage, bootstrap, render, and identity-controller paths, with legacy-key migration or an explicit compatibility decision plus focused regression proof.

## Reproduction Assessment

Yes, at source level: current main reads and writes one `openclaw.control.assistant.v1` localStorage entry from the settings render, bootstrap, and assistant-identity controller paths. I did not run the full browser flow, but the issue's agent-switching steps map directly to those unscoped calls.

## Solution Assessment

Yes: scoping the local assistant avatar override to the active assistant agent ID is the narrow maintainable fix. The remaining question is which open PR provides the best migration and proof, not whether the issue should close now.
