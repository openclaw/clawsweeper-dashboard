## Label Justifications

- `P2`: This is a real user-facing Control UI state bug with limited blast radius to browser-local assistant avatar preferences.
- `impact:session-state`: The issue mis-associates per-agent assistant avatar state by applying one local override across all selected agents.

## Summary

Keep open: current `main` and the latest release still use one browser-local assistant avatar override for all agents, so the reported cross-agent overwrite remains valid. Several open fix PRs already reference this issue, so the right cleanup path is to review and land one of those candidates rather than close the issue early.

## What This Changes

The issue asks for Control UI assistant avatar overrides to be scoped per agent instead of sharing one local avatar override across all agents.

## Best Possible Solution

Land one narrow Control UI fix that keys the local assistant avatar override by the selected agent id, preserves or intentionally migrates the existing unscoped browser value, and adds regression coverage for switching between two agents.

## Reproduction Assessment

Yes from source inspection: current `main` reads and writes a single `openclaw.control.assistant.v1` localStorage record, so two agents necessarily share the same local assistant avatar override. I did not run a browser session because this review is read-only, but the storage/controller path is direct.

## Solution Assessment

Yes, the narrow maintainable fix is to scope the existing browser-local assistant avatar storage by `assistantAgentId` and thread that id through the existing load/save call sites. A separate new settings surface or server-side avatar change is not needed for this bug.
