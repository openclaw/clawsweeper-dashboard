## Summary

Keep open: current main and the latest shipped release still store the Control UI assistant avatar override under one browser-wide key, and several open linked PRs are already competing to fix that exact bug. The issue should stay open until one focused fix lands.

## What This Changes

The issue asks for Control UI settings to bind browser-local assistant avatar overrides to the selected agent instead of one shared browser-wide assistant avatar.

## Best Possible Solution

Land one focused fix that scopes local assistant avatar storage by active agent id while preserving legacy global override behavior for upgrades, then close duplicate fix PRs after the chosen PR merges.

## Reproduction Assessment

Yes, source-reproducible: current main reads and writes one assistant avatar localStorage key and all relevant callers ignore `assistantAgentId`. I did not run a live browser session, but the source path directly matches the reported cross-agent overwrite.

## Solution Assessment

Yes, the narrow maintainable fix is to scope the browser-local assistant avatar override by normalized agent id and thread that scope through storage, bootstrap, identity refresh, and render callers. The open linked PRs already target that shape; the cleanest one should be reviewed and landed rather than creating another repair branch.
