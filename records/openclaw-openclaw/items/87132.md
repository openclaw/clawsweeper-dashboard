## Summary

Keep open: current main and v2026.5.22 still have the reported source-backed regression, where the Web UI usage request omits `agentId` while the gateway now defaults omitted `agentId` to the default agent.

## What This Changes

The issue asks to restore the Usage page agent filter for multi-agent setups by sending an explicit agent scope to the `sessions.usage` API.

## Best Possible Solution

Keep the backend default-agent scope and make the Usage UI choose and pass explicit agent scope for server-side usage loading, with regression coverage for non-default agents.

## Reproduction Assessment

Yes. A high-confidence source reproduction is available: load the Usage page in a multi-agent setup, observe the UI request lacks `agentId`, and the current backend default scopes the result to `main`.

## Solution Assessment

Yes. The maintainable fix is to pass explicit agent scope from the Usage UI to the existing `sessions.usage.agentId` protocol field, not to relax the backend security default.
