## Summary

Keep this PR open. Current main still does not implement a session working-context capsule, and this draft PR remains useful source work, but the proposed writer has a real temp-file collision bug and the durable local-state schema/privacy boundary still needs maintainer review before merge.

## What This Changes

The PR adds a new `src/agents/session-working-context.ts` utility and colocated Vitest tests for resolving, normalizing, reading, writing, and clearing a versioned JSON working-context capsule beside a session transcript.

## Best Possible Solution

Decide the durable schema and privacy model, fix the writer temp-file collision, then either land this primitive as the storage contract with focused tests or fold it into the first integration PR that updates and exposes working context across runtime, status, compaction, and new-session flows.

## Reproduction Assessment

Yes. The current-main gap is reproducible by targeted source/docs inspection, and the writer race is statically reproducible from the PR diff because concurrent same-process writes can share one temp path.

## Solution Assessment

Unclear as proposed. A small storage primitive is a reasonable first step, but this patch should not merge until the temp filename is unique per write and maintainers accept the durable schema/privacy boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use a unique temp path for each context write:** `src/agents/session-working-context.ts:176`
  - body: `writeWorkingContext` derives the temp filename from only `process.pid` and `Date.now()`. Two writes to the same session in the same process and millisecond can use the same temp file, causing one rename/unlink to race the other and either fail persistence or write the wrong final capsule. Use `randomUUID()` or the existing atomic JSON helper so each call gets a unique temp path.
  - confidence: 0.87
