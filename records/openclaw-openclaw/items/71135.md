## Summary

Keep open. The related working-context feature is still absent from current main, and this draft PR remains plausible source work, but it is not merge-ready because the proposed durable capsule write is collision-prone, uses default file permissions for sensitive local state, lacks real behavior proof, and still needs maintainer schema/privacy/storage guidance.

## What This Changes

The PR adds a new agents utility and tests for reading, writing, normalizing, and clearing a sibling `.context.json` session working-context capsule.

## Best Possible Solution

Keep the draft open for maintainer schema and storage guidance, then land a private, collision-safe persistence primitive coordinated with runtime, `/status`, compaction, and subagent integration for the canonical issue.

## Reproduction Assessment

Yes for the PR review findings: the PR diff shows the non-unique temp filename and default-permission write. For the underlying feature gap, current-main source search shows no structured working-context capsule implementation.

## Solution Assessment

No, not as-is. A storage primitive is a reasonable MVP direction, but this patch should use unique private atomic writes and wait for maintainer agreement on schema, storage location, retention, and integration lifecycle.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use a unique temp path for each write:** `src/agents/session-working-context.ts:171`
  - body: `writeWorkingContext` builds the temp filename from the target path, process id, and `Date.now()`. Two same-process writes to the same session in the same millisecond can pick the same temp file, so one write can rename or unlink the other writer's payload; add a per-call random suffix or reuse the existing atomic writer.
  - confidence: 0.87
- **[P2] Write context capsules with private permissions:** `src/agents/session-working-context.ts:174`
  - body: The capsule can persist cwd/repo paths, branch/remotes, sandbox state, and notes, but this `fs.writeFile` call uses default permissions. On permissive umasks the renamed capsule can be group/world-readable, so pass `mode: 0o600` or use `writeTextAtomic` before runtime code populates this file.
  - confidence: 0.84
