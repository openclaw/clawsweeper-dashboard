## Summary

Keep open: the PR has sufficient real-environment proof and addresses some prior comments, but the current head still lets OpenAI `role: "tool"` payload content pass into `inputMessages`, bypassing the independent `toolOutputs` privacy gate.

## What This Changes

The branch adds policy-gated model-call input/output content capture, routes `diagnostics.otel.captureContent` into the embedded runner, exports the shared resolver through the diagnostic SDK/API seam, and updates focused tests and SDK baselines.

## Best Possible Solution

Land the model-call capture fix after the extractor respects the separate content-class gates, with focused regression coverage for tool-role payloads and a separate follow-up for any remaining tool execution content capture.

## Reproduction Assessment

Yes. The blocking PR bug is source-reproducible with an OpenAI-style request payload containing `{ role: "tool", content: "tool result" }` while `inputMessages` is enabled and `toolOutputs` is disabled.

## Solution Assessment

No. Capturing model input at the `onPayload` seam and output from normalized chunks is the right shape, but the current extractor is not safe until it filters or separately gates tool-result payloads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep tool results out of inputMessages:** `src/agents/pi-embedded-runner/run/attempt.model-diagnostic-events.ts:183`
  - body: When `captureContent.inputMessages` is true but `toolOutputs` is false, `shouldCaptureRole` only rejects `system` and `developer`, so OpenAI-compatible `{ role: "tool", content: "tool result" }` messages are pushed into `inputMessages` and exported under `openclaw.content.input_messages`. Please filter tool/tool-result roles here or gate them on the matching tool-output policy, with a regression test for `inputMessages: true, toolOutputs: false`.
  - confidence: 0.9
