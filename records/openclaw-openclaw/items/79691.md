## Summary

Keep open. The latest head addresses the earlier model-content extraction findings and has sufficient live OTEL proof, but it now violates the bundled plugin boundary by re-exporting a core `src/**` helper from the diagnostics-otel plugin API, which current CI catches.

## What This Changes

The branch populates model-call diagnostic input/output message fields behind `diagnostics.otel.captureContent`, shares the content-capture policy resolver, and adds wrapper/OTEL service coverage.

## Best Possible Solution

Land the model-call capture fix through a boundary-compliant plugin SDK diagnostic surface, then separately implement or track real tool content emission before resolving the broader linked report.

## Reproduction Assessment

Yes. Current docs and the OTEL exporter expect opt-in content fields, while current main's model-call diagnostic event type and wrapper do not populate `inputMessages` or `outputMessages`; the PR body also supplies after-fix live OTEL output.

## Solution Assessment

No. The model-call seam and capture-policy direction are right, but the shared resolver must be exposed through a plugin SDK surface or kept plugin-local instead of importing core `src/**` from a bundled plugin.
