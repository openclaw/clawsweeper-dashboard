## Summary

Keep open: this targets a valid documented diagnostics bug, but the diff misses the real request/chunk seams, adds raw content to trusted diagnostic events before any capture-policy gate, and lacks after-fix real behavior proof.

## What This Changes

This PR adds diagnostic event fields and wrapper-side extraction of model input/output text for OTEL content-capture spans.

## Best Possible Solution

Wire content capture at the final request-payload and normalized output-delta seams, respect the existing captureContent policy before emitting sensitive fields, and land with focused regression tests plus redacted OTLP proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main documents `captureContent` and the OTEL plugin reads content fields, but the model-call event type and wrapper do not emit them. I did not run a live OTLP backend in this read-only review.

## Solution Assessment

No. The PR should use the final `onPayload` request and normalized stream/result chunks, and it needs a policy/redaction boundary before sensitive content is attached to diagnostic events.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Capture inputs from the emitted request payload:** `src/agents/pi-embedded-runner/run/attempt.model-diagnostic-events.ts:612`
  - body: The wrapper receives the provider model descriptor as `model`; the actual OpenAI request with `messages`/`input` is delivered through `options.onPayload` after provider wrappers can mutate it. Extracting from `model` leaves `inputMessages` empty for normal calls, so `captureContent.inputMessages` still won't export user prompts.
  - confidence: 0.93

  - body: Provider stream functions yield normalized chunks such as `{ type: "text_delta", delta }` and `{ type: "text", text }`. The new extractor only recognizes raw OpenAI wire chunks, so most real model streams still complete without `outputMessages`.
  - confidence: 0.89
