## Summary

Keep open: the PR fixes a real documented model-call content-capture gap, but the current head still lets tool-result messages flow into `inputMessages`, bypassing the separate `toolOutputs` privacy gate.

## What This Changes

The PR adds policy-gated `inputMessages` and `outputMessages` capture to model-call diagnostics, shares the OTEL content-capture resolver through the plugin SDK, and adds model diagnostic and OTEL service tests.

## Best Possible Solution

Land the model-call capture fix after the extractor respects the independent privacy gates, and keep any production tool-content capture work separately tracked if it remains out of scope.

## Reproduction Assessment

Yes. Current main source shows documented model-call content fields are not emitted, and the remaining PR regression is source-reproducible with an OpenAI-compatible payload containing a `role: "tool"` message while only `inputMessages` is enabled.

## Solution Assessment

No. Capturing at the final `onPayload` seam and normalized output stream is the right direction, but the current extractor must keep tool-result content out of `inputMessages` unless the separate tool-output policy explicitly allows it.
