## Summary

Keep this PR open: it targets a real documented diagnostics bug and includes credible live OTLP proof, but the current head still has two merge-blocking defects around the content-capture boundary.

## What This Changes

The branch adds model-call diagnostic content fields, captures request and response text in the embedded runner under an OTEL captureContent policy, wires that policy from attempt config, and adds focused wrapper tests.

## Best Possible Solution

Land opt-in model-call content capture only after raw collection is gated before buffering and the extractor covers the current OpenAI Responses payload shape with regression tests.

## Reproduction Assessment

Yes, source-backed. Docs expose captureContent, the OTEL exporter reads event.inputMessages/event.outputMessages, and current main's model-call event type/emitter do not populate those fields; this review did not run the live CLI path because the checkout was read-only.

## Solution Assessment

No. The PR uses the right general seam, but it must avoid buffering output when output capture is disabled and must parse Responses input_text arrays so the documented user prompt content is actually captured.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate output buffering behind captureContent:** `src/agents/pi-embedded-runner/run/attempt.model-diagnostic-events.ts:134-136`
  - body: `observeResponseChunk` now extracts every streamed text delta into `outputTextChunks` before the emit-time policy check. With `diagnostics.otel.captureContent.outputMessages` false, raw model output is still retained in memory until completion, so skip extraction and buffering unless output capture is enabled.
  - confidence: 0.9
- **[P2] Capture Responses input text parts:** `src/agents/pi-embedded-runner/run/attempt.model-diagnostic-events.ts:235-241`
  - body: OpenAI Responses requests built by current main put user prompts under `input[].content` as `{ type: "input_text", text }` parts, but this extractor only accepts string content for `input` array items. That means the main Responses path can record the system/developer string while dropping the actual user prompt from `inputMessages`.
  - confidence: 0.88
