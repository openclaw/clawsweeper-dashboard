## Summary

Keep this PR open: it fixes the right documented diagnostics gap and has useful live proof, but the current patch would leak system/developer prompt content through `inputMessages` and adds a plugin package boundary break.

## What This Changes

The PR adds model-call `inputMessages`/`outputMessages` capture for diagnostics OTEL, centralizes content-capture policy parsing, and extends wrapper and OTEL exporter tests.

## Best Possible Solution

Land the diagnostics content capture fix after the wrapper excludes system/developer prompt text from `inputMessages` unless the separate `systemPrompt` policy is intentionally implemented, and keep shared policy parsing on a public plugin SDK subpath.

## Reproduction Assessment

Yes, source-backed: current main lacks the documented model-call content fields while the OTEL exporter expects them. The PR body also supplies live OTEL output for the intended positive path.

## Solution Assessment

No. The proposed seam is appropriate, but the patch must preserve the independent `systemPrompt` privacy policy and the published plugin package boundary before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Exclude system prompts from inputMessages:** `src/agents/pi-embedded-runner/run/attempt.model-diagnostic-events.ts:183-185`
  - body: `extractInputMessages` pushes every chat/Responses message content into `inputMessages` without checking the role. Current Responses payload construction includes the system/developer prompt as a request message, so `captureContent.inputMessages=true` with `systemPrompt=false` would still export that prompt under `openclaw.content.input_messages`; skip `system`/`developer` roles here or gate them behind the separate `systemPrompt` policy.
  - confidence: 0.93
- **[P2] Keep the policy helper on the plugin SDK surface:** `extensions/diagnostics-otel/api.ts:1-4`
  - body: `@openclaw/diagnostics-otel` is published as an external plugin package, but its `api.ts` now re-exports from `../../src/infra/...`, which will not exist relative to the installed plugin package. Export the helper from `openclaw/plugin-sdk/diagnostic-runtime` and import it through that public subpath instead.
  - confidence: 0.88
