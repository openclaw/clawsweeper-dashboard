## Summary

Keep open: this PR targets a valid documented diagnostics bug and includes useful live OTEL proof, but the current head still has two merge-blocking issues around capture policy enforcement and Responses input extraction.

## What This Changes

The PR adds model-call `inputMessages`/`outputMessages` diagnostic fields, wires OTEL `captureContent` policy into embedded attempts, and extends wrapper tests for model-call content capture.

## Best Possible Solution

Land the model-call content capture fix after raw content collection is gated before buffering and the extractor covers current Responses text-part payloads with regression tests.

## Reproduction Assessment

Yes, source-backed. Current docs and the OTEL exporter expect opt-in content fields, while current main's model-call event type/emitter do not populate them; I did not run the live gateway path in this read-only review.

## Solution Assessment

No. The PR uses the right general seam, but it must avoid collecting raw output when output capture is disabled and parse Responses `input_text` content arrays before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate output buffering behind captureContent:** `src/agents/pi-embedded-runner/run/attempt.model-diagnostic-events.ts:134-136`
  - body: `observeResponseChunk` still extracts every streamed text delta into `outputTextChunks` before checking `contentCapture.outputMessages`. With output capture disabled, raw response text should not be collected in memory at all, so pass the policy into this path and skip extraction/buffering unless the subkey is enabled.
  - confidence: 0.91
- **[P2] Capture Responses input text parts:** `src/agents/pi-embedded-runner/run/attempt.model-diagnostic-events.ts:235-241`
  - body: Responses requests commonly carry user text as `input[].content` arrays with `{ type: "input_text", text }` parts, but this extractor only accepts string content for `input` array items. That leaves the main Responses shape without the user prompt in `inputMessages`.
  - confidence: 0.89
