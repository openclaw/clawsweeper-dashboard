## Summary

Keep open: the model-call capture direction is useful and the PR includes real OTEL proof, but the new input extractor can bypass the separate `toolOutputs` privacy gate by exporting tool-result messages under `inputMessages`.

## What This Changes

The PR adds policy-gated `inputMessages`/`outputMessages` capture to model-call diagnostics, shares the OTEL content-capture resolver through the plugin SDK, and adds model diagnostic and OTEL service tests.

## Best Possible Solution

Land the model-call capture fix after the extractor respects the independent content-class gates, and keep or split the linked tool-content work as a separate focused follow-up.

## Reproduction Assessment

Yes. Current main documents OTEL content capture but the model-call diagnostic emitter lacks input/output content fields; the PR finding is source-reproducible with an OpenAI chat payload containing a `role: "tool"` message while only `inputMessages` is enabled.

## Solution Assessment

Not yet. Capturing at the `onPayload` seam and normalizing streamed output is the right shape, but the extractor must avoid exporting tool-result content unless the corresponding tool-output gate is enabled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Keep tool results out of inputMessages:** `src/agents/pi-embedded-runner/run/attempt.model-diagnostic-events.ts:191-196`
  - body: When `captureContent.inputMessages` is true but `toolOutputs` is false, this loop still accepts OpenAI chat messages with `role: "tool"` and pushes their `content` into `inputMessages`. Existing provider payloads can contain `{ role: "tool", content: "tool result" }`, and the docs expose `toolOutputs` as a separate opt-in, so this bypasses that privacy gate and exports tool results under `openclaw.content.input_messages`. Please filter tool/tool-result content out of `inputMessages` or gate it on the matching tool-output policy.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: Needs attention: the PR can leak tool result content through `inputMessages` even when `captureContent.toolOutputs` is disabled.
