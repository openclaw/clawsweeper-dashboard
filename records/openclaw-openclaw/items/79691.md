## What This Changes

The PR adds model-call input/output content fields to diagnostics events, wires `diagnostics.otel.captureContent` into embedded attempts, and extends model diagnostic tests.

## Best Possible Solution

Land a narrow diagnostics fix that emits content only after explicit opt-in, supports existing Chat and Responses payload shapes, and accurately documents raw-versus-redacted boundaries.

## Reproduction Assessment

Yes, source-backed. Current docs and the OTEL exporter expect opt-in content fields, while current main's model-call event type/emitter do not populate them; the linked issue also provides a concrete gateway repro flow.

## Solution Assessment

No. The PR uses the right seam, but the latest head still needs to gate output extraction before buffering and parse Responses `input_text` content arrays before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Gate output buffering behind captureContent:** `src/agents/pi-embedded-runner/run/attempt.model-diagnostic-events.ts:134-136`
  - body: `observeResponseChunk` still extracts and stores every streamed text delta before checking `contentCapture.outputMessages`. With output capture disabled, diagnostics should not retain raw response text in memory, so pass the policy into this path and skip extraction/buffering unless the subkey is enabled.
  - confidence: 0.91
- **[P2] Capture Responses input text parts:** `src/agents/pi-embedded-runner/run/attempt.model-diagnostic-events.ts:238-241`
  - body: Responses requests commonly carry user text as `input[].content` arrays with `{ type: "input_text", text }` parts, but this extractor only accepts string `input[].content`. That leaves the main Responses shape without the user prompt in `inputMessages`.
  - confidence: 0.9
- **[P3] Fix the raw content field contract:** `src/infra/diagnostic-events.ts:462-465`
  - body: These new comments say the event fields are bounded and redacted, but the wrapper emits raw `state.inputMessages` and joined output chunks and leaves bounding/redaction to the OTEL consumer. Either enforce that contract before emitting or document these fields as raw opt-in content.
  - confidence: 0.83
