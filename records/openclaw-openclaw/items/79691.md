## Summary

Keep open: this PR targets a real documented diagnostics gap, but the current head still extracts inputs from the wrong object, misses normalized output chunks, lacks real behavior proof, and introduces a raw-content diagnostics bus exposure before policy/redaction.

## What This Changes

This PR adds model-call diagnostic event content fields and wrapper-side extraction of input/output text for OTEL content-capture spans.

## Best Possible Solution

Move content extraction to the final request-payload and normalized output seams, gate/bound/redact before trusted diagnostic emission, and land with focused tests plus redacted OTLP proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: docs and config expose `captureContent` input/output flags, the OTEL plugin reads `event.inputMessages`/`event.outputMessages`, and current main never emits those fields; I did not run a live OTLP backend.

## Solution Assessment

No. The PR targets the right documented bug, but it extracts at the wrong seams and bypasses the captureContent policy before internal emission; the safer path is to capture from final `onPayload` and normalized output events under an emission-time policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Capture inputs from the emitted payload:** `src/agents/pi-embedded-runner/run/attempt.model-diagnostic-events.ts:620`
  - body: At line 620 the wrapper calls `extractInputMessages(model)`, but the `model` argument is the provider model descriptor; the actual request with `messages` or `input` is observed through `options.onPayload` after provider wrappers and caller mutations. Normal calls can therefore keep `inputMessages` empty even with content capture enabled.
  - confidence: 0.93
- **[P2] Handle normalized stream chunks:** `src/agents/pi-embedded-runner/run/attempt.model-diagnostic-events.ts:100`
  - body: `observeResponseChunk` now sends every chunk to an extractor that only recognizes raw OpenAI wire formats, while providers in this repo emit normalized events such as `{ type: "text_delta", delta }` and `{ type: "text", text }`. Most real streams will still complete without `outputMessages`.
  - confidence: 0.9
