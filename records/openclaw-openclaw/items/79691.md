## Summary

Keep open: this PR addresses a documented diagnostics bug and the updated body now includes copied OTLP span proof, but the patch still buffers raw streamed output even when output content capture is disabled.

## What This Changes

The PR adds model-call diagnostic content fields, extracts input/output text in the embedded runner wrapper, passes the OTEL capture-content policy from attempt config, and adds focused wrapper tests.

## Best Possible Solution

Land opt-in model-call content capture that only buffers and emits content under the configured policy, preserves sanitized model-call hooks, and keeps OTEL redaction and size caps covered by tests plus live proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: docs and config expose captureContent, the OTEL plugin reads event.inputMessages/event.outputMessages, and current main's model-call event wrapper never populates those fields.

## Solution Assessment

No. The PR now targets the right seams, but output text collection must also be gated before buffering so default-disabled capture does not retain raw response content.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate output buffering behind captureContent:** `src/agents/pi-embedded-runner/run/attempt.model-diagnostic-events.ts:134-136`
  - body: `observeResponseChunk` now calls `extractTextFromChunk` and appends to `outputTextChunks` for every streamed model call. With the default `captureContent.outputMessages: false`, OpenClaw still retains raw response text in memory until completion, which breaks the opt-in boundary and duplicates large outputs; pass the policy into this path and skip extraction/buffering unless output capture is enabled.
  - confidence: 0.87
