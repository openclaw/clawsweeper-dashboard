## Summary

Keep open: this PR addresses a real documented OTEL model-content capture bug and includes live proof, but the current head still retains raw streamed output when output capture is disabled, misses the normal OpenResponses `input_text` shape, and would close a broader issue whose tool-content path remains unfixed.

## What This Changes

The PR adds model-call diagnostic `inputMessages`/`outputMessages` fields, captures model request/response content in the embedded runner under OTEL `captureContent` policy, and adds wrapper tests.

## Best Possible Solution

Land the model-content capture fix after tightening opt-out privacy and Responses parsing, then either keep https://github.com/openclaw/openclaw/issues/77391 open for tool content or extend the patch to cover tool input/output capture too.

## Reproduction Assessment

Yes, source-backed: current docs promise opt-in model content capture, the OTEL exporter reads those fields, and current main's model-call event/emitter never populate them. I did not rerun the live gateway path in this read-only review.

## Solution Assessment

No. The PR uses the right embedded-runner seam, but it must avoid retaining raw output when output capture is disabled, parse OpenResponses `input_text` parts, clarify the raw event contract, and avoid closing the broader tool-content issue unless that path is also fixed.
