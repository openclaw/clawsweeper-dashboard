## Summary

Keep open. Current main no longer shows the old Feishu-specific `tool call` string, but the attribution bug remains in the embedded tool lifecycle path: channel identity exists on the run params, is not passed into subscription params, and tool start/end logs still inherit the static `agent/embedded` logger.

## What This Changes

The issue requests that tool call lifecycle logs be tagged with the originating channel subsystem instead of a static Feishu or embedded-agent subsystem.

## Best Possible Solution

Add a typed channel-origin or logger seam from `runEmbeddedAttempt` into `subscribeEmbeddedPiSession`, derive a safe `gateway/channels/<channel>` logger for message-channel runs, keep `agent/embedded` as the fallback, and cover tool start/end attribution with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an embedded run with `messageChannel: "telegram"` followed by a tool start/end event: the runner has the channel, but the subscription layer drops it and handlers log through the static embedded logger.

## Solution Assessment

Yes. The narrow maintainable fix is to pass channel origin or an already-derived logger through the existing runner-to-subscriber seam; a channel-specific workaround or broad logging rewrite would be less targeted.
