## Summary

Keep open. Current main no longer has the old Feishu-specific `tool call` text, but the embedded tool lifecycle logs still inherit a static `agent/embedded` logger because the subscription layer is not given the originating channel identity.

## What This Changes

Request that tool lifecycle log entries use the originating channel subsystem instead of a static Feishu or embedded logger.

## Best Possible Solution

Thread a typed, normalized channel origin or logger into `subscribeEmbeddedPiSession`, use `gateway/channels/<channel>` for message-channel tool lifecycle logs, and keep `agent/embedded` as the fallback for channel-less runs.

## Reproduction Assessment

Yes, at source level: the runner has `messageChannel`/`messageProvider`, but the subscription params omit them and tool start/end logs call a static context logger. A live runtime repro would be any Telegram or Discord turn that executes a tool with debug logs enabled.

## Solution Assessment

Yes. The narrow maintainable fix is a typed channel-origin or logger seam at the runner-to-subscriber boundary, not channel-specific branches inside the tool handlers.
