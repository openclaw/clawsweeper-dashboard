## Summary

Keep open. Current main no longer has the old Feishu-specific `tool call` wording, but embedded tool start/end debug logs still inherit a static `agent/embedded` logger because the subscription path is not given the originating channel identity.

## What This Changes

The issue requests channel-specific subsystem attribution for embedded tool lifecycle logs instead of static Feishu or embedded logger attribution.

## Best Possible Solution

Thread a typed, normalized channel origin or logger from `runEmbeddedAttempt` into `subscribeEmbeddedPiSession`, use `gateway/channels/<channel>` for message-channel tool lifecycle logs, and keep `agent/embedded` as the fallback for channel-less runs.

## Reproduction Assessment

Yes, at source level: the runner has `messageChannel`/`messageProvider`, but the subscription params omit them and tool start/end logs call a static context logger. A live runtime repro would be any Telegram or Discord turn that executes a tool with debug logging enabled.

## Solution Assessment

Yes. The narrow maintainable fix is a typed channel-origin or logger seam at the runner-to-subscriber boundary, not channel-specific branches inside the tool handlers.
