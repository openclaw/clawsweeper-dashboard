## Summary

Keep open. Current main removed the old Feishu-specific tool-call wording, but the remaining attribution bug is still present: channel-originated embedded runs carry `messageChannel` into the runner and then drop it before the subscription tool handlers, which still log through a static `agent/embedded` logger.

## What This Changes

Ensure tool call lifecycle logs use the originating channel subsystem instead of a static Feishu or embedded logger.

## Best Possible Solution

Add a typed channel-origin or logger seam from `runEmbeddedAttempt` into `subscribeEmbeddedPiSession`, derive a safe channel subsystem for valid message-channel runs, keep `agent/embedded` as the fallback, and cover tool start/end attribution with focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a channel-originated embedded run, such as `messageChannel: "telegram"`, followed by a tool start/end event: the runner has the channel but the subscriber still injects the static embedded logger.

## Solution Assessment

Yes. The narrow maintainable repair is to pass channel origin or an already-derived logger through the existing runner-to-subscriber seam; a channel-specific workaround or broad logging rewrite would be less targeted.

## Repair Work Prompt

Fix tool lifecycle log subsystem attribution for channel-originated embedded runs. Current main carries `messageChannel` / `messageProvider` in `RunEmbeddedPiAgentParams`, but `buildEmbeddedSubscriptionParams` / `SubscribeEmbeddedPiSessionParams` do not pass channel origin or a logger into `subscribeEmbeddedPiSession`; tool start/end logs therefore use the static `agent/embedded` logger. Add a typed channel-origin or logger override seam from `runEmbeddedAttempt` to `subscribeEmbeddedPiSession`, derive `gateway/channels/<channel>` only for valid normalized message-channel values, and keep `agent/embedded` as the fallback for channel-less runs. Add focused regression coverage for Telegram/Discord-style channel attribution and fallback behavior. Preserve existing tool arg/result redaction and do not add config options, hardcoded channel-specific branches, or broad logging rewrites. Add a `CHANGELOG.md` fix entry because this repairs user-visible diagnostics.
