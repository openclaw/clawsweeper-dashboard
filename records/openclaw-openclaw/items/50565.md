## Summary

Keep open. The original Feishu-specific `tool call` wording is gone on current `main`, but the remaining logging attribution bug is still present: channel-originated embedded runs carry channel identity into the runner, then drop it before the subscription layer, so tool start/end debug logs inherit the static `agent/embedded` logger instead of a `gateway/channels/<channel>` subsystem.

## What This Changes

Tool lifecycle log entries should use the subsystem for the originating message channel instead of a static or wrong channel logger.

## Best Possible Solution

Add a typed channel-origin or logger seam from `runEmbeddedAttempt` into `subscribeEmbeddedPiSession`, derive a safe channel subsystem for valid message-channel runs, keep the embedded fallback for non-channel runs, and cover tool start/end attribution with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an embedded run with `messageChannel: "telegram"` followed by a tool start/end event: the runner has the channel, but subscription construction drops it and the tool handler logs through the static embedded logger.

## Solution Assessment

Yes. The narrow maintainable fix is to pass channel origin or an already-derived logger through the existing runner-to-subscriber seam; a channel-specific workaround or broad logging rewrite would be less targeted.
