## Summary

Keep open. Current main still appears vulnerable for the reported jjcc OpenAI-compatible Gemini route: cron's raw runtime schema contains nullable anyOf fields, Gemini cleanup is keyed from provider id only, and the OpenAI Completions transport forwards the already-normalized tool parameters without another schema pass.

## What This Changes

The issue reports that Gemini models behind a jjcc OpenAI-compatible provider reject the cron tool schema in v2026.6.1 because nullable cron fields are emitted as property-level anyOf schemas without a type.

## Best Possible Solution

Normalize Gemini-restricted tool schemas through the existing generic provider/model compatibility path when the active model identity or compat profile indicates Gemini, while preserving cron's nullable runtime validation and avoiding a jjcc-specific branch.

## Reproduction Assessment

Partially: there is no live jjcc request proof here, but source inspection shows current main can leave cron nullable anyOf fields in place for provider jjcc with a Gemini model id. The reporter's v2026.6.1 logs match the fields still emitted by the raw cron schema.

## Solution Assessment

Yes, a narrow generic schema-compatibility fix is the best direction. The merged DeepSeek fix is provider-specific, and the open Google/Gemini PRs found in search target native Google paths rather than this OpenAI-compatible jjcc route.
