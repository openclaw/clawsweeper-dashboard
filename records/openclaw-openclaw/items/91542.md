## Summary

Keep open. Current `main` and the latest `v2026.6.8` release still select Gemini schema cleanup from the provider id only, so `jjcc` with a Gemini model id can still miss the cleanup path; the open linked PR is the focused fix candidate and should land or be replaced before this issue closes.

## What This Changes

The issue reports Gemini model requests through a `jjcc` OpenAI-compatible provider failing because cron tool schemas expose nullable `anyOf` properties without property-level `type`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/91559, or an equivalent central tool-schema normalizer fix, then close this issue after the fix reaches `main`.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.8` leave `jjcc` with a Gemini model id outside the Gemini cleanup branch while cron schemas still contain nullable fields matching the report. A live jjcc/Gemini request was not run in this read-only review.

## Solution Assessment

Yes: the central tool-schema normalizer is the best boundary because it already receives provider id, model id, and model compatibility. A cron-specific or jjcc-specific rewrite would duplicate provider-schema policy in the wrong layer.
