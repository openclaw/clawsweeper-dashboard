## Summary

This maintainer-labeled draft PR should stay open: the latest head addresses the earlier ClawSweeper blockers, but source review found a remaining P2 regression in auto TTS provider selection for non-Gateway/local TTS paths.

## What This Changes

The PR threads prepared model, auth, provider, plugin, tool, media, and TTS runtime state through reply execution and Gateway startup, with scoped loader helpers, tests, docs, and a changelog entry.

## Best Possible Solution

Keep the prepared-runtime reuse, but preserve documented configured-provider auto-selection for local/non-Gateway TTS callers while using loaded-only scoping only when a prepared Gateway runtime registry is actually available.

## Reproduction Assessment

Yes. Source inspection of PR head shows a concrete path: local `tts.providers` or `tts.voices` with `messages.tts.provider` unset calls `getTtsProvider` before any speech providers are loaded, despite docs requiring first-configured-provider auto selection.

## Solution Assessment

No. Reusing prepared runtime surfaces is the right performance direction, but the latest branch is not mergeable until TTS auto-selection preserves the existing non-Gateway contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

- **[P2] Preserve auto TTS selection outside Gateway:** `extensions/speech-core/src/tts.ts:632`
  - body: When `messages.tts.provider` is unset, docs promise the first configured provider in registry order. Keep the scoped Gateway path, but fall back to configured provider discovery for non-pinned callers.

## Repair Work Prompt

On PR #76757, fix the remaining P2 regression where `getTtsProvider` now auto-selects only from loaded speech providers. Preserve the request-time/Gateway performance intent, but keep the documented behavior that when `messages.tts.provider` is unset OpenClaw picks the first configured provider in registry auto-select order for non-Gateway/local callers such as `openclaw capability tts providers` and `openclaw capability tts voices` without `--provider`. Prefer a small helper or caller option that uses loaded-only selection only when a prepared/pinned runtime registry exists. Add focused regression coverage for local TTS provider/voice resolution with provider unset and one configured provider. Do not broaden plugin loading on the hot Gateway reply path, do not change TTS config schema, and do not add an extra changelog entry unless the repair changes user-facing behavior beyond preserving the existing contract.
