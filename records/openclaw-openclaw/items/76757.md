## Summary

Keep open. The PR is still draft/protected and current head introduces correctness regressions in direct plugin tool invocation, configured PDF model-owner startup discovery, and custom OAuth provider token formatting.

## What This Changes

The PR threads prepared auth, model, plugin, tool, and startup runtime state through reply, embedded-runner, provider, and gateway startup paths to reduce repeated discovery.

## Best Possible Solution

Keep the prepared-runtime direction only after preserving existing direct tool invocation semantics, including all configured model-owner refs, and retaining documented provider auth hooks across gateway and non-gateway paths.

## Reproduction Assessment

Yes. Source inspection of PR head shows direct non-core tools.invoke requests no longer add the requested plugin tool as a standalone discovery allowlist entry, and optional plugin tool discovery still skips empty allowlists.

## Solution Assessment

No. The performance/refactor direction may be useful, but the current patch is not the best mergeable solution while it drops existing direct optional plugin tool behavior and misses configured PDF model-owner startup coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve requested plugin tools without profile allowlists:** `src/gateway/tool-resolution.ts:87-96`
  - body: `gatewayRequestedTools` now only reaches profile policies through `mergeAlsoAllowPolicy`; when there is no existing `allow` policy, the requested tool never reaches `pluginToolAllowlist`. Direct `tools.invoke` calls for optional plugin tools can therefore skip plugin discovery and return not found, so keep the standalone requested-tool allow entry or an equivalent path.
  - confidence: 0.9
- **[P2] Include pdfModel in configured model refs:** `src/config/model-refs.ts:32-37`
  - body: Gateway startup now starts model-owner plugins from `collectConfiguredModelRefs`, but this helper omits `agents.defaults.pdfModel`. A configured PDF provider plugin can be left out of the startup/prewarm runtime surfaces even though PDF models are part of the same config contract; collect `pdfModel` for defaults and per-agent entries too.
  - confidence: 0.86
- **[P2] Keep provider formatApiKey hooks on the OAuth path:** `src/agents/auth-profiles/oauth.ts:113-116`
  - body: For non-google providers, this now returns `credentials.access` unless the provider is already pinned in the gateway runtime registry. `formatApiKey` is a documented provider-plugin hook for custom runtime token shapes, and current main calls the plugin formatter from the OAuth profile path, so custom or non-gateway OAuth providers can silently lose required formatting.
  - confidence: 0.78
