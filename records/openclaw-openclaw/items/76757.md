## Summary

Keep open; this draft, maintainer-labeled PR still has a P2 correctness blocker where fallback providers can inherit an auth profile order prepared for a different provider.

## What This Changes

The PR threads prepared model, auth, provider, plugin, tool, media, TTS, and Gateway runtime state through reply execution to reduce repeated discovery, with tests, docs, and a changelog entry.

## Best Possible Solution

Keep the prepared-runtime reuse, but recompute or filter prepared auth order per fallback provider and add cross-provider fallback auth regression coverage before maintainer review.

## Reproduction Assessment

Yes. Source inspection of the PR head shows the reply path prepares auth order for the initial provider, then model fallback can invoke embedded execution for another provider while auth selection reuses that prepared order without recomputing.

## Solution Assessment

No. The performance direction is reasonable, but the current diff is not mergeable until provider-scoped auth ordering is recomputed or filtered for fallback candidates.

## Repair Work Prompt

On https://github.com/openclaw/openclaw/pull/76757, fix the prepared authProfileOrder reuse across model fallback providers. The bug is that get-reply-run prepares auth order for the initial provider, buildEmbeddedRunBaseParams carries it into every embedded fallback attempt, and prepareEmbeddedRunAuthSelection trusts it for the current provider. Update the implementation so a prepared order is reused only when it is valid for the current provider, or recompute/filter it with the prepared authStore and resolveAuthProfileOrder for the fallback provider. Preserve explicit user-locked profile validation, plugin-harness-owned transport behavior, and the prepared-runtime performance direction; do not broaden plugin/provider loading or change fallback candidate ordering. Add focused regression coverage for cross-provider fallback auth, preferably in src/agents/pi-embedded-runner/run/auth-selection.test.ts and/or the reply/model fallback tests. The existing changelog entry covers the performance feature unless the repair changes user-visible behavior.
