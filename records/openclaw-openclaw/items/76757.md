## Summary

Keep open: this draft MEMBER/maintainer XL PR has removed the earlier conflict markers, but the current head still introduces a source-reproducible gateway direct plugin-tool invocation regression before merge.

## What This Changes

The PR adds prepared reply/runtime plumbing so model, auth, provider, tool, session, embedded-runner, plugin, and gateway startup surfaces can be scoped once and reused across reply execution.

## Best Possible Solution

Restore the direct-invoke requested-tool allowlist behavior, add coverage for an optional plugin tool invoked without a pre-existing allow policy, then run the broad changed gate and provide perf evidence before maintainer review.

## Reproduction Assessment

Yes. Source inspection shows tools.invoke passes non-core tool names through gatewayRequestedTools, while PR head no longer adds that value as a standalone plugin discovery allowlist entry and optional plugin tool discovery skips empty allowlists.

## Solution Assessment

No. The prepared-runtime direction may be useful, but this PR is not the best mergeable solution while it can drop requested optional plugin tools from direct gateway invocation; the narrow repair is to preserve the previous gatewayRequestedTools allowlist path.
