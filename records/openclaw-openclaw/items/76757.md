## Summary

Keep open: source review found a fallback auth-order regression in the latest PR head, and the draft maintainer-labeled PR should stay in maintainer review rather than cleanup closure.

## What This Changes

The PR threads prepared model, auth, provider, plugin, tool, media, and TTS runtime state through reply execution and Gateway startup to reduce repeated runtime discovery.

## Best Possible Solution

Keep the prepared-runtime reuse, but recompute or filter prepared auth profile order when fallback changes provider, with a focused regression test for cross-provider fallback auth.

## Reproduction Assessment

Yes. Source inspection of PR head shows the reply path computes auth order for the initial provider, stores it on the follow-up run, and the embedded auth selector then reuses that order for fallback providers without recomputing.

## Solution Assessment

No. The performance direction and the latest TTS repair look sound, but the proposed path is not mergeable until fallback-provider auth selection avoids reusing an order scoped to another provider.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Recompute auth order when fallback changes provider:** `src/agents/pi-embedded-runner/run/auth-selection.ts:140-145`
  - body: The prepared `authProfileOrder` is computed for the initial reply provider, but model fallback can invoke this selector for a different provider. Because any prepared order bypasses `resolveAuthProfileOrder`, a fallback from one provider to another will iterate the primary provider's profile ids and can fail before considering valid profiles for the fallback provider. Only reuse the prepared order when it matches the current auth provider, or recompute/filter it for the fallback provider.
  - confidence: 0.87
