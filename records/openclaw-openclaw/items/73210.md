## Summary

Keep open. Current main still lacks the requested auto-TTS delivery policy and the auto-TTS prompt hint still describes TTS controls rather than speech-first writing guidance; the explicit TTS-tool duplicate-text fix and the broader closed spoken-rewrite request do not cover this narrower auto-TTS gap.

## What This Changes

Add a voice-only or caption-only text-delivery policy for successful auto-TTS replies and strengthen the auto-TTS system prompt so model replies are written for natural speech.

## Best Possible Solution

Add a backward-compatible messages.tts delivery policy with the existing double-delivery behavior as default, apply suppression only after auto-TTS succeeds and only for voice-style delivery, keep visible text for TTS or voice-send failures and non-voice file attachments, and add concise speech-first prompt guidance without taking on the broader spoken-rewrite pipeline.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: enable auto-TTS for a normal final reply on a voice-capable channel, and maybeApplyTtsToPayload returns generated audio while retaining the original visible reply text that channel delivery then captions or sends as follow-up text.

## Solution Assessment

Yes, with scope control. A generic TTS delivery policy plus a stronger auto-TTS prompt hint is the narrow maintainable path; channel-only special cases or a full spoken rewrite pass would be broader than this issue requires.
