## Summary

Keep open. Current main still lacks a configurable auto-TTS text-delivery policy, and the auto-TTS system prompt still describes TTS mechanics rather than instructing the model to write speech-first replies; the related explicit TTS-tool and silent-turn media fixes do not cover this auto-TTS request.

## What This Changes

The issue requests an auto-TTS voice-only/caption text-delivery mode plus stronger natural-speech prompt guidance when automatic TTS is enabled.

## Best Possible Solution

Add a backward-compatible generic TTS delivery policy with current double delivery as the default, suppress visible text only after successful auto-TTS voice-style audio delivery, preserve text fallback for TTS failures and non-voice attachments, and strengthen the prompt hint with concise speech-first writing guidance without taking on the broader spoken-rewrite pipeline.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable auto-TTS for a normal final reply and maybeApplyTtsToPayload returns generated audio while preserving visible reply text that the downstream channel delivery path captions or sends separately.

## Solution Assessment

Yes. The requested direction is the narrowest maintainable fix if scoped to a generic delivery policy plus prompt hint; channel-specific suppression alone would duplicate policy, while a full spoken-rewrite pass is broader than needed here.

Expected behavior: normal/default keeps visible text with generated audio; suppress hides visible text only after successful voice-style auto-TTS audio generation; TTS failure falls back to text; channels that cannot deliver native voice notes or only send file attachments should keep text context.
