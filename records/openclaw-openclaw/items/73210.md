## Summary

Keep open. Current main still lacks the requested auto-TTS text-delivery policy, and the auto-TTS system prompt hint still focuses on TTS mechanics rather than speech-first writing guidance. The explicit TTS-tool duplicate-text fix and the broader stale spoken-rewrite issue do not cover this narrower auto-TTS behavior.

## What This Changes

Request adds an auto-TTS voice-only/caption delivery mode and strengthens the auto-TTS prompt hint so generated replies are written for natural spoken output.

## Best Possible Solution

Add a backward-compatible messages.tts delivery policy with current double delivery as the default, apply text suppression only after auto-TTS succeeds on voice-style delivery, preserve text fallback for TTS/voice-send failures and non-voice file attachments, and add concise speech-first prompt guidance without implementing a full spoken-rewrite pipeline.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: enable auto-TTS for a normal final reply on a voice-capable channel and maybeApplyTtsToPayload returns generated audio while retaining visible reply text that downstream channel delivery captions or sends separately.

## Solution Assessment

Yes, with tight scope. A generic TTS delivery policy plus a stronger auto-TTS prompt hint is narrower and more maintainable than channel-specific suppression or a full spoken-rewrite pass.
