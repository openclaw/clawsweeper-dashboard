## Summary

Keep open. Current main still lacks both requested auto-TTS pieces: there is no text-delivery policy for successful auto-generated voice media, and the injected TTS prompt hint still gives operational instructions rather than speech-first writing guidance.

## What This Changes

Add an auto-TTS delivery policy for voice-only or caption-only text handling, and strengthen the auto-TTS system prompt so replies are written for natural spoken output.

## Best Possible Solution

Add a backward-compatible `messages.tts` delivery policy with the existing double-delivery behavior as default, apply suppress/caption behavior only after auto-TTS succeeds for voice-capable delivery, preserve text fallback when synthesis or voice delivery fails, and add concise speech-first prompt guidance without taking on the broader spoken-rewrite pipeline.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable auto-TTS on a normal final reply for a voice-capable channel, and the current auto-TTS path returns generated audio while retaining reply text that channel delivery still renders as a caption, follow-up, or card/text payload.

## Solution Assessment

Yes. A generic TTS delivery policy plus a stronger auto-TTS prompt hint is the narrow maintainable direction; channel-only special cases or a full spoken rewrite pass would be broader than this issue requires.
