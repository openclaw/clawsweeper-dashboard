## Summary

Keep open. Current main still lacks both requested pieces: auto-TTS successful audio delivery preserves visible reply text, and the system prompt hint is operational rather than speech-first.

## What This Changes

This issue requests an auto-TTS text-delivery policy for voice-only or caption-only replies plus stronger system prompt guidance for natural spoken output.

## Best Possible Solution

Add a backward-compatible generic auto-TTS delivery policy with `normal` as the default, suppress visible text only after successful voice-style audio delivery, keep text fallback on failure or non-voice attachments, and update speech-first prompt guidance, docs, config types, schema/help, and focused channel tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable `messages.tts.auto: "always"` for a normal final reply on Telegram, WhatsApp, Matrix, or Feishu, and the auto-TTS success path returns generated audio while preserving visible reply text for delivery.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix if implemented as a generic speech-core/config delivery policy plus prompt hint; a channel-only fix would duplicate policy, and a full spoken-rewrite pass is broader than this issue.
