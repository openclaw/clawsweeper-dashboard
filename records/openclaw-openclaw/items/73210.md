## Summary

Keep open. Current main still keeps visible reply text after successful auto-TTS voice generation, and the auto-TTS prompt hint still gives operational TTS instructions rather than speech-first writing guidance.

## What This Changes

Request a voice-only or caption-only delivery policy for successful auto-TTS plus a stronger speech-oriented system prompt hint.

## Best Possible Solution

Add a backward-compatible generic auto-TTS text-delivery policy with current double delivery as the default, suppress visible text only after successful voice-style audio delivery, preserve text fallback for failures and non-voice attachments, and strengthen the prompt hint without taking on the broader spoken-rewrite pipeline.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable auto-TTS for a final reply on a voice-capable channel, and maybeApplyTtsToPayload returns generated audio while preserving visible reply text for downstream caption or follow-up delivery.

## Solution Assessment

Yes. A generic speech-core/config delivery policy plus prompt hint is the narrow maintainable path; channel-only suppression would duplicate policy, while the older spoken-rewrite proposal is broader than this issue.
