## Summary

Keep open. Current main still lacks a configurable auto-TTS text-delivery policy, and the auto-TTS prompt hint still describes TTS mechanics rather than giving speech-first writing guidance. The related explicit TTS-tool fix and the older spoken-rewrite proposal do not cover this narrower auto-TTS final-reply behavior.

## What This Changes

Request adding an auto-TTS voice-only or caption-only delivery policy and improving the auto-TTS system prompt so generated replies are written for natural spoken output.

## Best Possible Solution

Add a backward-compatible `messages.tts` delivery policy with current double delivery as the default, apply text suppression only after successful auto-TTS voice-style audio delivery, preserve text fallback for TTS or send failures and non-voice attachments, and add concise speech-first prompt guidance without building a full spoken-rewrite pipeline.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: enable auto-TTS for a normal final reply on a voice-capable channel and `maybeApplyTtsToPayload` returns generated audio while retaining visible reply text that downstream delivery captions or sends separately.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix. A generic TTS delivery policy plus a stronger auto-TTS prompt hint is safer than channel-specific suppression and smaller than the stale full spoken-rewrite proposal.
