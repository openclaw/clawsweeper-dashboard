## Summary

Keep open. Current main still lacks an auto-TTS text-delivery policy and still gives the model operational TTS hints rather than speech-first writing guidance; the closed explicit-tool duplicate-text fix and the older stale spoken-rewrite issue do not resolve this narrower auto-TTS feature request.

## What This Changes

Request adds an auto-TTS voice-only/caption/normal text-delivery policy plus stronger system-prompt guidance for natural spoken replies.

## Best Possible Solution

Add a backward-compatible `messages.tts` delivery policy with current behavior as the default, suppress or caption visible text only after auto-TTS audio succeeds on voice-capable delivery, preserve text fallback when synthesis or voice delivery fails, and add concise speech-first guidance without implementing the full spoken-rewrite pipeline.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable auto-TTS on a final reply for Telegram or WhatsApp and current main keeps reply text on the synthesized voice payload, with no config, docs, or command path to suppress it.

## Solution Assessment

Yes, with scope control. A generic auto-TTS delivery policy plus a stronger prompt hint is the narrow maintainable solution; channel-only special cases or a full rewrite pass would be broader than this issue requires.
