## Summary

Keep open. Current main still lacks the requested auto-TTS text-delivery policy, and the auto-TTS prompt hint still tells the model about TTS mechanics rather than guiding speech-first writing. The related explicit TTS-tool duplicate-text fix and the older spoken-rewrite proposal do not cover this narrower auto-TTS behavior.

## What This Changes

The issue requests a configurable auto-TTS voice-only or caption-only delivery mode plus stronger speech-first system prompt guidance for spoken auto-TTS replies.

## Best Possible Solution

Add a backward-compatible generic TTS delivery policy with current double delivery as the default, suppress visible text only after successful auto-TTS voice-style audio delivery, keep text fallback for TTS failures and non-voice attachments, and strengthen the prompt hint with concise speech-first writing guidance without implementing the broader spoken-rewrite pipeline from #22028.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable auto-TTS for a normal final reply on a voice-capable channel and maybeApplyTtsToPayload returns generated audio while preserving visible reply text that the downstream channel delivery path captions or sends separately.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable solution if scoped to a generic delivery policy plus prompt hint. Channel-specific suppression alone would duplicate policy, while a full spoken-rewrite pass is broader than needed for this issue.

## Repair Work Prompt

Implement the auto-TTS delivery policy and prompt-hint improvement requested by https://github.com/openclaw/openclaw/issues/73210. Add a backward-compatible `messages.tts` text delivery policy, preserving current behavior by default, and apply it only after auto-TTS synthesis succeeds. Keep normal text fallback when TTS generation fails, when media/voice delivery is not available, or when the output is a non-voice file attachment. Support a voice-only/suppress mode, and only implement caption-only behavior if it can be represented without breaking overflow/fallback semantics on Telegram, WhatsApp, Matrix, Feishu, and generic plugin delivery. Strengthen `buildTtsSystemPromptHint()` with concise speech-first guidance: replies will be spoken aloud, use conversational sentences, avoid markdown/code/URLs/bullets when possible, and send technical details separately when needed. Do not implement the broader spoken-rewrite pipeline from #22028, do not change provider output formats, and do not move channel-owned behavior into core beyond a generic payload/config seam. Update config types/schema/help/labels/generated docs as required, `/tts` local preference command/docs if added, focused tests, public docs, and a user-facing CHANGELOG entry crediting `Thanks @altierac`.
