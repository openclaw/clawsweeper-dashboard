## Summary

Keep open. Current main still lacks the requested auto-TTS visible-text delivery policy and the auto-TTS system prompt still gives operational TTS instructions rather than speech-first writing guidance; the related explicit TTS-tool duplicate-text fix and the older spoken-rewrite issue do not cover this narrower auto-TTS gap.

## What This Changes

Request to add an auto-TTS voice-only or caption-only delivery mode while improving the auto-TTS prompt hint so generated replies are written for natural speech.

## Best Possible Solution

Add a backward-compatible generic auto-TTS text-delivery policy with current behavior as the default, wire it through config/types/schema, local /tts preferences and docs, apply suppress/caption handling only for successful voice-capable auto-TTS delivery, and strengthen the prompt hint with concise speech-first guidance without implementing a full spoken-rewrite pipeline.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: enable auto-TTS for a normal final reply on a voice-capable channel and maybeApplyTtsToPayload keeps the original text while adding audio media, which Telegram and similar channel paths still render as a caption or follow-up text.

## Solution Assessment

Yes. A generic TTS delivery policy plus a stronger prompt hint is the narrow maintainable fix; channel-only special cases or reviving the broader spoken-rewrite pass would solve more than this issue asks for.

## Repair Work Prompt

Implement the auto-TTS text-delivery feature requested in https://github.com/openclaw/openclaw/issues/73210. Add a backward-compatible policy under `messages.tts` such as `textDelivery: "normal" | "suppress" | "caption"` with `normal` preserving current behavior. Wire the field through config schema/types/help, effective TTS config resolution, local `/tts` preferences/commands if appropriate, and public TTS docs. Apply suppress/caption behavior only after auto-TTS synthesis succeeds for voice-capable delivery, preserve text on TTS or voice-delivery failure, and keep text on channels that cannot render native voice notes. Strengthen `buildTtsSystemPromptHint()` with concise guidance that auto-TTS replies will be spoken aloud and should use natural speech-friendly prose instead of bullets, code blocks, URLs, or dense markdown. Do not regress the explicit `/tts audio` or agent `tts` tool media-only behavior fixed for #70092, and do not change provider output format/transcoding policy except where needed for the delivery policy. Add a user-facing `CHANGELOG.md` entry credited with `Thanks @altierac`.
