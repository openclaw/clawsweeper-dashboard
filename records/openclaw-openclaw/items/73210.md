## Summary

Keep open. Current main still has both requested gaps: successful auto-TTS preserves visible reply text for voice-capable delivery, and the auto-TTS system prompt still gives operational TTS hints rather than speech-first writing guidance.

## What This Changes

Request a configurable auto-TTS text-delivery policy for voice-only or caption-only output, plus stronger system-prompt guidance for natural spoken replies.

## Best Possible Solution

Add a backward-compatible generic auto-TTS delivery policy with current double delivery as the default, suppress visible text only after successful native voice-style audio delivery, preserve text fallback for failures and non-voice attachments, and add concise speech-first prompt guidance without implementing the broader spoken-rewrite pipeline.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable `messages.tts.auto: "always"` for a normal final reply on Telegram, WhatsApp, Matrix, or Feishu, and the auto-TTS success path returns generated audio while preserving visible reply text for downstream delivery.

## Solution Assessment

Yes. The requested direction is the narrowest maintainable fix if implemented as generic speech-core/config delivery policy plus prompt hint; channel-only suppression would duplicate policy, while a full spoken-rewrite pass is broader than this issue.
