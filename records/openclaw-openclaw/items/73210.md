## Summary

Keep open. Current main still lacks the requested auto-TTS voice-only/text-delivery policy and the auto-TTS prompt hint still does not tell the model to write replies for spoken delivery.

## What This Changes

Add an auto-TTS delivery mode for voice-only or caption-only text handling and improve the auto-TTS prompt guidance for natural spoken replies.

## Best Possible Solution

Add a backward-compatible `messages.tts` text-delivery policy that defaults to current behavior, suppresses or captions text only after successful native voice-style audio delivery, and updates prompt/docs/commands with speech-first guidance.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: enable `messages.tts.auto: "always"` for a normal final reply on a voice-capable channel and the auto-TTS success payload retains visible text that channel delivery code sends as a caption or follow-up.

## Solution Assessment

Yes. A generic speech-core/config delivery policy plus a prompt-hint update is the narrow maintainable direction; pushing suppression into individual channels would duplicate policy, while the broader spoken-rewrite pass from #22028 is a separate larger feature.

## Repair Work Prompt

Implement the auto-TTS text-delivery feature requested by https://github.com/openclaw/openclaw/issues/73210. Add a backward-compatible generic policy under `messages.tts` such as `textDelivery: "normal" | "suppress" | "caption"`, defaulting to current behavior. Apply suppression only after TTS generation succeeds and only for native voice-style audio delivery; preserve visible text fallback when synthesis fails, when the reply already has non-TTS media, or when a channel can only deliver audio as a normal file attachment. Update `buildTtsSystemPromptHint` with concise speech-first guidance so auto-TTS replies are written for spoken delivery without making this a full spoken-rewrite pipeline. Add `/tts` help/status/preference handling only if the repo's TTS prefs pattern supports it cleanly; otherwise document the config surface. Update config types/schema/help/generated docs as required, public TTS/slash-command docs, and focused tests for speech-core, config, commands, outbound payload behavior, and Telegram/WhatsApp/Matrix/Feishu delivery. Do not change provider selection, output-format routing, or the explicit `tts` tool behavior fixed by #70092. Add a user-facing changelog entry unless maintainers decide the final patch is docs/internal only.
