## Summary

Keep open. Current main still preserves visible text after successful auto-TTS voice generation, and the auto-TTS prompt still gives operational TTS hints rather than speech-first writing guidance; the explicit `/tts audio` path and the older spoken-rewrite issue do not cover this request.

## What This Changes

Add an auto-TTS voice-only/caption delivery policy and strengthen the auto-TTS system prompt so generated replies are written for spoken output.

## Best Possible Solution

Add a backward-compatible generic auto-TTS delivery policy with current double delivery as the default, suppress visible text only after successful native voice-style audio delivery, preserve text fallback for failures and non-voice attachments, and add speech-first prompt guidance without implementing the broader spoken-rewrite pipeline.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable auto-TTS for a normal final reply on a voice-capable channel, and `maybeApplyTtsToPayload` returns generated audio while preserving visible reply text that downstream delivery captions or sends separately.

## Solution Assessment

Yes. The requested direction is the narrowest maintainable fix if scoped to a generic speech-core/config delivery policy plus prompt hint; channel-specific suppression alone would duplicate policy, while a full spoken-rewrite pass is broader than this issue.

## Repair Work Prompt

Fix the auto-TTS voice-only delivery feature request. Add a backward-compatible `messages.tts` delivery policy, likely `textDelivery: "normal" | "suppress" | "caption"` with `normal` preserving current behavior. Apply it only after TTS audio generation succeeds, preserve normal text fallback on TTS failure, and do not suppress text when the channel/output is only a regular audio file attachment rather than a voice-style delivery. Keep policy generic in speech-core/config/shared delivery seams; avoid hard-coding bundled plugin ids in core beyond existing channel capability seams. Add a `/tts` local preference only if it fits the existing prefs model, such as `/tts text off|caption|normal`, and update command help/docs. Strengthen `buildTtsSystemPromptHint` with concise speech-first guidance: replies will be spoken aloud, prefer natural sentences, avoid bullets/code blocks/URLs/markdown when possible, and use a separate text follow-up for technical detail. Include a user-facing CHANGELOG.md entry. Do not implement the broader spoken-rewrite LLM pass from the older stale issue.
