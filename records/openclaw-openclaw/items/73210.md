## Summary

Keep open. Current main still lacks both requested pieces: auto-TTS has no visible-text delivery policy, and the injected TTS prompt hint still gives operational TTS instructions rather than speech-first writing guidance. The related explicit TTS-tool duplicate-text fix and the closed spoken-rewrite request do not cover this narrower auto-TTS feature gap.

## What This Changes

The issue requests a voice-only or caption-only text delivery mode for auto-TTS plus stronger prompt guidance so auto-TTS replies are written naturally for spoken output.

## Best Possible Solution

Add a backward-compatible `messages.tts` delivery policy with the current double-delivery behavior as default, apply suppress/caption handling only after auto-TTS succeeds for voice-capable delivery, preserve text on TTS or voice delivery failure, update `/tts` local preferences and docs, and add concise speech-first prompt guidance without taking on the broader spoken-rewrite pipeline.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable auto-TTS for a normal final reply on a voice-capable channel and `maybeApplyTtsToPayload()` returns generated audio while retaining the original reply text, which Telegram/WhatsApp/Matrix/Feishu delivery paths still render as caption or follow-up text.

## Solution Assessment

Yes, with scope control. A generic TTS delivery policy plus stronger auto-TTS prompt hint is the narrow maintainable solution; channel-only special cases or a full spoken rewrite pass would be broader than this issue requires.
