## Summary

Keep open. Current main still lacks the requested auto-TTS voice-only delivery policy and speech-first prompt guidance; the existing auto-TTS success path preserves visible reply text and the config/command/docs surfaces do not expose a text-delivery mode.

## What This Changes

Request to add an auto-TTS voice-only or caption-only text-delivery mode and strengthen the auto-TTS system prompt for natural spoken output.

## Best Possible Solution

Add a backward-compatible generic auto-TTS text-delivery policy that defaults to current behavior, suppresses visible text only after successful native voice-style audio delivery, preserves text fallback for failures and non-voice attachments, and adds concise speech-first prompt guidance.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable `messages.tts.auto: "always"` for a normal final reply on a voice-capable channel and the auto-TTS success path returns audio while preserving visible reply text for delivery.

## Solution Assessment

Yes. The narrow maintainable direction is a generic speech-core/config delivery policy plus prompt-hint update; pushing suppression into individual channels would duplicate policy and a full spoken-rewrite pipeline would be broader than this issue.
