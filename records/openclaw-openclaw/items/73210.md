## Summary

Keep open. Current main still preserves visible text after successful auto-TTS voice generation, and the auto-TTS system prompt still gives operational TTS hints rather than speech-first writing guidance; the related explicit TTS-tool fix, stale spoken-rewrite issue, and closed unmerged silent-turn PR do not cover this request.

## What This Changes

The issue requests an auto-TTS voice-only or caption-only text delivery policy plus stronger natural-speech prompt guidance when automatic TTS is enabled.

## Best Possible Solution

Add a backward-compatible generic auto-TTS delivery policy with current double delivery as the default, suppress visible text only after successful voice-style audio delivery, preserve text fallback for failures and non-voice attachments, and strengthen the prompt hint without implementing the broader spoken-rewrite pipeline.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable auto-TTS for a normal final reply on a voice-capable channel and maybeApplyTtsToPayload returns generated audio while preserving visible reply text that downstream delivery captions or sends separately.

## Solution Assessment

Yes. The requested direction is the narrowest maintainable fix if scoped to a generic speech-core/config delivery policy plus prompt hint; channel-specific suppression alone would duplicate policy, while the full spoken-rewrite pass is broader than this issue.
