## Summary

Keep open: current main and the latest release still route Slack app_mention events without the requested per-inbound INFO receipt log, while Telegram has an explicit inbound log before dispatch. The request is narrow, in the Slack plugin owner boundary, aligned with channel stability/observability work, and no duplicate or merged fix was found.

## What This Changes

The issue requests INFO-level Slack inbound logging for app_mention events, aligned with Telegram inbound logging, so operators can diagnose silent routing failures from server logs.

## Best Possible Solution

Add a Slack-plugin-owned INFO receipt log for accepted app_mention events before downstream dispatch, with focused coverage proving one log line for channel mentions and no log for dropped DM duplicate app_mention events.

## Reproduction Assessment

Yes from source inspection: Slack routes accepted app_mention events into the message handler without a per-inbound INFO receipt log, while Telegram logs an inbound line before dispatch. I did not run a live Slack workspace reproduction in this read-only review.

## Solution Assessment

Yes: the narrow maintainable solution is a Slack-plugin-owned receipt log at the accepted app_mention/message-handler boundary, not a core channel-feedback change. Optional socket disconnect logging should remain a separate follow-up unless maintainers intentionally broaden scope.
