## Summary

Keep open. Current main and the latest release only have a narrow immediate post-compaction identical-tool-result guard, while this issue asks for a broader guarded-compaction feature with config, transcript-tail detection, scoring, instruction augmentation, post-compaction validation, reset recommendation, fixture coverage, and docs; open same-author stacked PRs still target those missing pieces.

## What This Changes

Request to add a loop-aware compaction saturation guard that detects degraded transcript tails, augments compaction, validates the result, and recommends reset only after guarded recovery fails.

## Best Possible Solution

Keep the shipped narrow post-compaction guard, then have maintainers review and either land, rebase, narrow, or replace the stacked PRs as one coherent guarded-compaction design.

## Reproduction Assessment

Not applicable as a strict bug reproduction: this is a broad feature/design request backed by production incident context. Current main has focused tests for the narrow post-compaction repeated-tool-result guard, but no sanitized fixture for the full detector, scorer, validator, and reset-recommendation pipeline.

## Solution Assessment

Unclear as a final implementation path. The current post-compaction guard is useful but too narrow to close this issue, while the proposed stacked path needs maintainer review because it adds new heuristics, config, validation, and escalation policy.
