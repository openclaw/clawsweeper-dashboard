## Summary

Keep open: current main has partial lineage support through stable session keys, lifecycle timestamps, and usage-family IDs, but it still rotates `sessionId` on `/new` and does not provide the requested per-agent session ledger with start time, kind, and thread identity for memory tools. The related continuity work is still an open PR and does not supersede this broader identity contract request.

## What This Changes

The issue requests a stable per-agent session identity or lightweight session ledger so memory and maintenance tools can identify related sessions across `/new` resets.

## Best Possible Solution

Define one gateway-owned session lineage contract, preferably in `sessions.json` and the Gateway/session APIs, that records stable session key, rotated session IDs, start/end/reason/kind metadata, and a documented consumer path for memory and cleanup tools.

## Reproduction Assessment

Not applicable: this is an enhancement request, not a report of broken documented behavior. Source inspection confirms the current behavior rotates `sessionId` and only partially preserves lineage through usage-family metadata.

## Solution Assessment

Unclear: the proposed `memory/<agent>-thread.md` table is a useful sketch, but the maintainable owner boundary likely belongs in gateway-owned session storage or APIs rather than an ad hoc memory file.
