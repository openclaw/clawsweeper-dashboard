## Summary

Keep open: current main has not clearly fixed the fresh v2026.6.10 report because recent content-bearing WhatsApp `append` upserts can still pass through access control before the stale-append skip, and unknown direct senders in pairing mode can still trigger an outbound pairing reply. This needs maintainer/security review plus live reconnect proof before changing pairing semantics.

## What This Changes

The issue reports that the WhatsApp plugin can send pairing-code messages to unintended contacts during reconnect after a Gateway restart.

## Best Possible Solution

Keep this issue open as the canonical v2026.6.10 WhatsApp security tracker, capture live reconnect/upsert evidence with redacted numbers, then tighten the WhatsApp inbound pairing gate so reconnect/history/contact-sync events cannot send pairing replies while the intended first-DM pairing contract is preserved or deliberately changed.

## Reproduction Assessment

No, not with high confidence from live behavior. Source inspection shows a plausible current-main path for recent content-bearing `append` DMs to reach the pairing send path during the reconnect grace window, but this review did not reproduce the reporter's random-address-book sends against a linked WhatsApp account.

## Solution Assessment

Unclear until the reconnect event is captured. The likely best fix is a WhatsApp-owned inbound gate that suppresses reconnect/history/contact-sync side effects before pairing access control, while preserving deliberate live first-DM pairing unless maintainers intentionally change that default.
