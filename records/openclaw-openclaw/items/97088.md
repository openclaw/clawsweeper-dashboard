- `impact:security`: The issue involves pairing-code disclosure and unsolicited contact outreach across the WhatsApp access-control boundary.

## Summary

Keep open: this is a fresh v2026.6.10 report of privacy-sensitive unsolicited WhatsApp pairing replies, and current main still has pairing-reply side effects reachable from the WhatsApp inbound pipeline. Existing guards reduce older known cases, but they do not prove reconnect/address-book sync events cannot still trigger outbound pairing messages.

## What This Changes

The issue reports that the WhatsApp plugin sends pairing-code messages to unintended contacts during reconnect without user or agent action.

## Best Possible Solution

Keep the issue open and fix the WhatsApp inbound boundary so reconnect, history, contact-sync, or stale events cannot send pairing replies unless a real current inbound DM from that sender triggered the challenge.

## Reproduction Assessment

No, not at high confidence. Source inspection shows the pairing-send path and plausible reconnect/upsert boundaries, but I did not run a live WhatsApp reconnect with a linked address book.

## Solution Assessment

Unclear until the exact Baileys event shape is captured, but the best likely fix is a WhatsApp-owned pre-access-control gate for reconnect/history/sync events rather than a new broad config option.
