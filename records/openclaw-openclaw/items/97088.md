## Summary

Keep open: current main and v2026.6.10 still contain a WhatsApp pairing-reply send path for unapproved direct-message events, and the fresh report needs maintainer/security review plus live reconnect evidence before changing pairing semantics.

Prevent WhatsApp reconnects from sending pairing-code messages to unintended contacts without an explicit live inbound sender or user-confirmed target.

## Best Possible Solution

Capture the live Baileys reconnect event shape, then add a WhatsApp-owned inbound gate that suppresses reconnect/history/contact-sync pairing replies while preserving intentional live first-DM pairing.

Unclear until the exact reconnect event is captured. The narrow fix should live in the WhatsApp inbound boundary, not in shared pairing formatting or a blanket removal of documented first-DM pairing.
