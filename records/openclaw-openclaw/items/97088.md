## Summary

Keep open: current main and v2026.6.10 still intentionally send WhatsApp pairing replies for content-bearing unknown DMs, while this fresh report says reconnect/address-book events produced those replies without user action; the exact live Baileys event shape remains unproven and security-sensitive.

## What This Changes

The issue asks the WhatsApp plugin to stop sending pairing-code messages to contacts during reconnect unless there is explicit user-approved or live inbound DM context.

## Best Possible Solution

Keep this as the canonical WhatsApp reconnect misdelivery tracker, capture redacted live reconnect evidence, then patch the WhatsApp-owned inbound gate so reconnect/history/contact-sync side effects never send pairing replies while deliberate first-DM pairing remains documented.

## Reproduction Assessment

No high-confidence live reproduction. Source inspection confirms the pairing-send side effect for content-bearing unknown WhatsApp DMs and existing reconnect guards, but not the reporter's random-contact reconnect event with a real linked account.

## Solution Assessment

Unclear. The likely best fix is a WhatsApp-owned reconnect/history/contact-sync gate before pairing replies, while broad disabling or confirmation for all pairing replies would need maintainer product/security direction.
