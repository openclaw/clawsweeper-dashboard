## Summary

Keep open: current main has several WhatsApp pairing guards, but the fresh v2026.6.10 report matches a remaining source-level gap where recent Baileys append/offline upserts can still reach pairing access control before append replay handling rules out side effects.

## What This Changes

The issue reports that WhatsApp reconnects can send unsolicited OpenClaw pairing messages from a linked personal account to unrelated contacts.

## Best Possible Solution

Add a WhatsApp-owned guard that prevents pairing replies for reconnect/offline replay upserts before any pairing side effect, with regression coverage for recent append DMs, stale append DMs, and normal live notify DMs.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review. Source inspection does show a plausible current-main path: a recent append/offline direct message with user content can reach pairing access control and send a pairing reply before append replay handling runs.

## Solution Assessment

Unclear until maintainers confirm the exact Baileys replay shape from logs, but the narrowest likely fix is not a new config flag. The better repair is to pass replay/upsert classification into the pairing decision, or move replay suppression before access-control side effects while preserving normal live pairing.
