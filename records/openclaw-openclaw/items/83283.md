## Summary

Keep open: current main still uses broad whole-command matching for WhatsApp doctor TUI discovery and can pass those PIDs into the kill path; the linked fix PR is open with closing syntax, so this issue should stay open until that PR lands or is closed.

## What This Changes

The issue reports that WhatsApp doctor repair should only signal confidently identified OpenClaw TUI clients, not unrelated processes whose argv contains matching terms.

## Best Possible Solution

Review and land or revise https://github.com/openclaw/openclaw/pull/83313 so process discovery is argv-structured, regression-tested, and either includes PID identity revalidation or records that as an explicit follow-up.

## Reproduction Assessment

Yes. A mocked `ps -axo pid=,command=` row such as a helper process whose argv contains `openclaw tui` would match the current full-command regex and flow into the termination helper on current main.

## Solution Assessment

Yes. Anchoring detection to the executable/subcommand shape, adding false-positive regression coverage, and revalidating process identity before signals is the narrow maintainable fix path.
