## Summary

Keep open: current main still matches local TUI candidates with a broad regex over the whole `ps` command string, then signals every returned PID without revalidating that it is still the expected OpenClaw TUI command.

## What This Changes

The issue asks `doctor --fix` WhatsApp responsiveness repair to signal only confidently identified OpenClaw TUI client processes.

## Best Possible Solution

Keep the WhatsApp responsiveness repair, but make process matching argv-structured and re-check stable process command metadata immediately before each signal.

## Reproduction Assessment

Yes. A focused mocked `ps -axo pid=,command=` line where a non-TUI command's arguments contain `openclaw-tui` would exercise the current matcher and then flow into the signaling helper.

## Solution Assessment

Yes. The suggested argv-aware, executable/subcommand-anchored detection plus pre-signal revalidation is the narrow maintainable fix and does not need a new config knob.
