## Summary

Keep open: current main still matches local TUI candidates with a broad regex over the whole `ps` command string, then signals every returned PID without revalidating that it is still an OpenClaw TUI command.

## What This Changes

The issue requests safer WhatsApp doctor repair so non-TUI processes whose argv contains TUI-looking words are not selected or terminated.

## Best Possible Solution

Keep the WhatsApp responsiveness repair, but make detection argv-structured and re-check same-command process metadata immediately before each signal, with focused regression coverage.

## Reproduction Assessment

Yes for a source-level path. A mocked `ps -axo pid=,command=` line where a non-TUI command's arguments contain `openclaw-tui` would pass the current matcher and flow into the signaling helper.

## Solution Assessment

Yes. The suggested argv-aware executable/subcommand anchoring plus pre-signal command revalidation is the narrow maintainable fix and does not need a config knob or product decision.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/83283. In `src/commands/doctor-whatsapp-responsiveness.ts`, make local TUI process detection argv-aware and anchored to the expected executable/subcommand shape so incidental arguments containing `openclaw-tui`, `openclaw tui`, `openclaw chat`, or `openclaw terminal` do not match. Revalidate stable process command metadata immediately before sending `SIGTERM` and again before any `SIGKILL`, so PID reuse or command drift cannot terminate unrelated processes. Add focused tests in `src/commands/doctor-whatsapp-responsiveness.test.ts` with mocked `ps` output for a non-TUI command whose argv contains TUI-looking words, and assert it is neither listed nor signaled. Preserve the current Windows no-op and the WhatsApp degraded-gateway repair behavior. Add a changelog entry because this changes user-visible `doctor --fix` safety.
