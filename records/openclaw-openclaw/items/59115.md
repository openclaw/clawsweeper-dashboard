## What This Changes

The PR adds Slack thread-history parsing and tests for forwarded/shared `is_share` attachments, appending their text or fallback to retained history messages.

## Best Possible Solution

Rebase the branch into `extensions/slack/src/monitor/thread.ts`, compute one capped renderable forwarded-attachment subset for both retention and rendering, update the active tests, and attach redacted Slack proof.

## Reproduction Assessment

Yes, at source level. A mocked `conversations.replies` page with a retained thread reply whose only readable content is an `is_share` attachment will be dropped by current main's `thread.ts` because it ignores `attachments`.

## Solution Assessment

No. The requested behavior is the right narrow fix, but this PR needs to target the active `thread.ts` helper and derive retention from the same capped renderable attachment subset used for output.
