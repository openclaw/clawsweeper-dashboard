## Summary

Keep open: the Slack thread-history bug is still present on current main, and this PR is useful in direction but not mergeable as-is because it targets the old `media.ts` implementation shape, still has a capped-retention edge case, and lacks after-fix real behavior proof.

## What This Changes

The PR adds forwarded/shared Slack attachment text or fallback content to thread history entries and adds unit coverage for forwarded, attachment-only, non-share, and image-only share cases.

## Best Possible Solution

Rebase onto current main, implement one capped forwarded-share extraction path in `extensions/slack/src/monitor/thread.ts` for both retention and formatting, keep the inbound format consistent, add focused regression tests, and attach redacted real Slack proof.

## Reproduction Assessment

Yes, at source level. Mock `conversations.replies` with a thread message whose `text` is empty and whose `attachments` contains an `is_share` item with `text` or `fallback`; current main filters it out because the helper only retains text or file messages.

## Solution Assessment

No as-is. The intended direction is narrow and maintainable, but the branch must be adapted to current `thread.ts` and use the same capped rendered share subset for both retention and output before merge.
