## Summary

Keep open. The bug is valid and current main still relays output-bearing structured exec completions, but this PR still suppresses every local structured SIGTERM completion without producer-side cleanup provenance, so it can hide actionable signaled command failures.

## What This Changes

This PR adds local-exec provenance to async exec system events, filters structured code 0 and SIGTERM completions out of heartbeat user relay, JSON-encodes relayable exec details, and adds focused heartbeat/exec tests.

## Best Possible Solution

Land a rebased fix that carries cleanup/restart provenance, or otherwise internalizes only proven-benign local exec completions while preserving relay for user-relevant signal/code failures.

## Reproduction Assessment

Yes, at source level. On current main, enqueue an output-bearing `Exec completed (..., code 0)` or `Exec failed (..., signal SIGTERM)` event for a heartbeat-capable session and the filter/runner path treats it as relayable; existing tests also assert output-bearing successful exec completion delivery.

## Solution Assessment

No, not yet. The local-exec provenance gate is a good narrow direction, but suppressing every local structured SIGTERM by text is broader than the reported restart-cleanup case unless cleanup provenance is added or maintainers explicitly accept that boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve relay for actionable SIGTERM failures:** `src/infra/heartbeat-events-filter.ts:12`
  - body: This makes every structured local `Exec failed (..., signal SIGTERM)` completion internal-only, but `maybeNotifyOnExit` emits the same shape for any background process killed by SIGTERM, not only restart cleanup. A user-started command terminated by SIGTERM would lose its completion output; carry cleanup provenance from the producer or keep signaled failures relayable unless the event is known cleanup.
  - confidence: 0.86
