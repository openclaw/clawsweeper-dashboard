## Summary

Keep open. Current main still has the noisy output-bearing exec-completion relay path, while this PR is a useful narrow fix but is currently unmergeable against main and still needs a changelog entry plus maintainer confirmation on the SIGTERM suppression boundary.

## What This Changes

The PR adds local-exec provenance to async exec completion events, filters trusted structured `code 0` and `signal SIGTERM` completions out of heartbeat user relay, suppresses visible heartbeat ack/reasoning paths for internal-only completions, JSON-encodes relayable exec prompt data, and adds focused regression tests.

## Best Possible Solution

Land a rebased narrow heartbeat fix that marks only trusted local benign exec completions internal, preserves relay for actionable failures, records the user-facing fix in `CHANGELOG.md`, and leaves the broader event-audience model to #69492 unless maintainers choose to fold this slice into that design.

## Reproduction Assessment

Yes. On current main, enqueue the structured `code 0`, `signal SIGTERM`, and `code 1` exec completion shapes from the PR body into a heartbeat session; source and existing tests show output-bearing `code 0` and `SIGTERM` completions remain relayable while `code 1` failures are also relayable.

## Solution Assessment

Yes, with rebase and semantics caveats. The PR's local-exec provenance plus per-event filtering is the narrowest tactical direction I found, but maintainers should confirm the SIGTERM boundary and adapt it to current main before merge.
