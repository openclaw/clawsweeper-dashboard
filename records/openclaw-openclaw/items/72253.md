## Summary

Keep open: the PR still over-suppresses output-bearing successful exec completions and all local SIGTERM completions, and it lacks the required real behavior proof for the live chat delivery path.

## What This Changes

The PR adds local-exec origin/context metadata, filters structured code 0 and SIGTERM exec completions out of heartbeat user relay, JSON-encodes remaining exec details, and adds heartbeat/exec tests.

## Best Possible Solution

Carry explicit cleanup or internal-audience provenance from the exec producer, suppress only proven cleanup or metadata-only completions, preserve output-bearing success and actionable signal failures, and verify the behavior in a real chat/runtime path.

## Reproduction Assessment

Yes, at source level. Current main can enqueue output-bearing `Exec completed (..., code 0)` or `Exec failed (..., signal SIGTERM)` events through `maybeNotifyOnExit`, and the heartbeat filter/runner path treats output-bearing structured completions as relayable.

## Solution Assessment

No. The local-exec source/context gate improves spoofing resistance, but exit-shape matching is still too broad; the safer fix is producer-side cleanup/internal provenance before suppressing user delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve relay for output-bearing success completions:** `src/infra/heartbeat-events-filter.ts:8-9`
  - body: The successful-completion regex still matches `Exec completed (..., code 0) :: <output>` and the runner treats matching local-exec events as internal-only. Current main has regression coverage that preserves output-bearing completion delivery to the original Telegram topic, so successful `notifyOnExit` commands with meaningful output would be silently dropped. Restrict suppression to metadata-only successes or events with explicit cleanup/internal provenance.
  - confidence: 0.88
- **[P2] Preserve relay for actionable SIGTERM failures:** `src/infra/heartbeat-events-filter.ts:10-11`
  - body: The SIGTERM regex treats every local `Exec failed (..., signal SIGTERM)` completion as cleanup, but `maybeNotifyOnExit` emits the same shape for any signaled background process and current events do not identify restart cleanup. A user-started command terminated by SIGTERM could lose its completion output; carry cleanup provenance from the producer before suppressing it.
  - confidence: 0.87
