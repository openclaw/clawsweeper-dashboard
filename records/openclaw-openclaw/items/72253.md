## Summary

Keep open: this is a real heartbeat/chat-noise problem, but the PR is not safe to merge because it suppresses output-bearing successful exec completions and all SIGTERM-shaped completions without producer-side cleanup provenance. Current main preserves output-bearing exec delivery, the branch conflicts with current main, and the contributor has not provided real runtime/chat proof.

## What This Changes

The PR adds local-exec origin/context metadata, changes heartbeat exec-event filtering to keep code 0 and SIGTERM structured completions internal, JSON-encodes relayable exec prompt details, and adds heartbeat/exec tests.

## Best Possible Solution

Carry explicit cleanup/internal provenance from the exec producer, suppress only proven cleanup or metadata-only completions, preserve output-bearing success and actionable signal failures, and verify the behavior in a real chat/runtime path.

## Reproduction Assessment

Yes, at source level. Current main can enqueue output-bearing `Exec completed (..., code 0)` and `Exec failed (..., signal SIGTERM)` events through `maybeNotifyOnExit`, and the heartbeat predicate treats output-bearing structured completions as relayable.

## Solution Assessment

No. The local-exec origin/context gate helps spoofing resistance, but exit-shape matching is still too broad; the maintainable fix is explicit cleanup/internal provenance before suppressing user delivery.

A user-started command terminated by SIGTERM could lose its completion output; carry cleanup provenance from the producer before suppressing it.
  - confidence: 0.87
