## Summary

Keep this PR open, but it is not merge-ready: the branch targets a real heartbeat exec-notification bug, yet its suppression rule is broader than current main's relay contract and would hide output-bearing success or SIGTERM completions without producer-side cleanup provenance.

## What This Changes

The PR adds local-exec metadata to queued exec events, filters structured code 0 and SIGTERM exec completions out of heartbeat user relay, suppresses visible heartbeat acknowledgments for those completions, and adds focused heartbeat/exec tests.

## Best Possible Solution

Carry explicit cleanup/internal provenance from the exec producer, suppress only proven cleanup or metadata-only completions, preserve output-bearing success and actionable signal failures, then verify the behavior in a real chat/runtime path.

## Reproduction Assessment

Yes, at source level: current main can enqueue output-bearing code 0 and SIGTERM exec summaries, and the heartbeat predicate treats output-bearing completions plus non-success structured completions as relayable. I did not run a live chat reproduction in this read-only review.

## Solution Assessment

No. The local-exec origin/context gate improves spoofing resistance, but exit-shape matching is still too broad; the maintainable fix is explicit cleanup/internal provenance before suppressing user delivery.
