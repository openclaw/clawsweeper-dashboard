## Summary

Keep this PR open, but it still needs changes before merge: the current diff can suppress user-relevant SIGTERM exec failures, and the PR body still has only test/build proof rather than after-fix real behavior proof.

## What This Changes

The branch adds local-exec origin/context metadata, filters structured code 0 and SIGTERM exec completions out of heartbeat user relay, JSON-encodes relayable exec details, and adds focused heartbeat/exec tests.

## Best Possible Solution

Rebase or replace this with a fix that carries cleanup/restart provenance, internalizes only proven-benign completions, and preserves relay for user-relevant signal/code failures.

## Reproduction Assessment

Yes, at source level. On current main, enqueue an output-bearing `Exec completed (..., code 0)` or `Exec failed (..., signal SIGTERM)` for a heartbeat-capable session and the filter/runner path treats it as relayable.

## Solution Assessment

No, not yet. The local-exec provenance gate is a useful direction, but text-only SIGTERM suppression is broader than the reported cleanup case unless the producer carries cleanup provenance or maintainers explicitly accept that policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve relay for actionable SIGTERM failures:** `src/infra/heartbeat-events-filter.ts:12`
  - body: This classifies every structured local `Exec failed (..., signal SIGTERM)` completion as internal-only, but `maybeNotifyOnExit` emits that same shape for any background process killed by SIGTERM. A user-started command terminated by SIGTERM would lose its completion output; carry cleanup provenance from the producer or keep signaled failures relayable unless the event is known cleanup.
  - confidence: 0.86
