## Summary

Keep open. The PR targets a real heartbeat exec-noise problem, but the current patch still suppresses every local structured SIGTERM completion without cleanup provenance, and the external PR still lacks after-fix real behavior proof for the chat delivery path.

## What This Changes

The PR adds local-exec origin/context metadata, classifies structured code 0 and SIGTERM exec completions as internal-only in heartbeat, JSON-encodes relayable exec details, and adds focused heartbeat/exec tests.

## Best Possible Solution

Carry explicit cleanup/restart provenance from the producer and internalize only proven-benign completions while keeping user-relevant signal/code failures relayable.

## Reproduction Assessment

Yes, at source level. On current main, enqueue an output-bearing `Exec completed (..., code 0)` or `Exec failed (..., signal SIGTERM)` for a heartbeat-capable session and the filter/runner path treats it as relayable.

## Solution Assessment

No, not yet. The local-exec origin/context gate is useful, but text-only SIGTERM suppression is broader than the reported cleanup case unless the producer carries cleanup provenance or maintainers explicitly accept that policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve relay for actionable SIGTERM failures:** `src/infra/heartbeat-events-filter.ts:12`
  - body: This classifies every structured local `Exec failed (..., signal SIGTERM)` completion as internal-only, but `maybeNotifyOnExit` emits that same shape for any background process killed by SIGTERM. A user-started command terminated by SIGTERM with useful output would be consumed silently; carry cleanup/restart provenance from the producer or keep signaled failures relayable unless the event is known cleanup.
