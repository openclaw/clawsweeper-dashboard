## Summary

Keep open. Current main still has the reported user-relay behavior for outputful structured exec completions, so this PR is not obsolete; the proposed direction is narrow and useful, but the branch is not merge-ready because GitHub reports it dirty against main and it still lacks the required user-facing changelog entry.

## What This Changes

The PR adds local-exec provenance to async exec completion events, filters trusted structured code-0 and SIGTERM completions out of heartbeat user relay, adjusts HEARTBEAT_OK/reasoning suppression, JSON-encodes relayable exec prompt data, and adds focused regression tests.

## Best Possible Solution

Land a rebased narrow heartbeat fix that marks only trusted local structured code-0 and cleanup SIGTERM exec completions internal, preserves user relay for actionable failures, records the user-facing behavior change in the changelog, and leaves the broader event-audience design to #69492.

## Reproduction Assessment

Yes. On current main, enqueue outputful `Exec completed (..., code 0) :: ...` or `Exec failed (..., signal SIGTERM) :: ...` for a heartbeat delivery target; the filter classifies parsed outputful completions as relayable and the runner keeps the user-relay path active.

## Solution Assessment

Yes, with integration caveats. The local-exec provenance gate plus per-event filtering is the narrow maintainable fix for this tactical bug, but the dirty branch, changelog policy, and SIGTERM semantics need maintainer/author follow-through before merge.
