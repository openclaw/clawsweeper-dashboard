## Summary

Keep open. The PR is directionally useful, but it is still incomplete against current main: it suppresses raw tool output only when an `ok` outcome reaches `readSubagentOutput`, while the frozen-result refresh path can still call completion capture without an outcome and persist the same raw tool/toolResult text. The related #74073 discussion and overlapping #73783/#74113 PRs point at the same subagent announce/capture race, so this needs a canonical maintainer choice rather than cleanup closure.

## What This Changes

The branch adds an `ok`-outcome guard before subagent output selection falls back to `latestRawText` and adds a regression test for successful tool-only histories.

## Best Possible Solution

Land one canonical subagent announce fix that prevents raw tool/toolResult text from becoming a successful completion result across direct announce, initial completion freeze, and frozen-result refresh paths, while preserving timeout/error partial-progress behavior and aligning tests, docs, and changelog with the resulting contract.

## Reproduction Assessment

Yes. A focused source-level reproduction is a successful subagent history whose latest visible message is a toolResult and no final assistant reply: current main returns `snapshot.latestRawText`, and after this PR any capture path that omits `outcome` still takes that raw fallback.

## Solution Assessment

No. The guard is a reasonable part of the fix, but gating only on `outcome?.status === "ok"` is not complete; the safer solution is to carry outcome through refresh captures or otherwise make successful completion captures consistently reject raw tool/toolResult fallback.
