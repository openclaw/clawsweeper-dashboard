## Summary

Keep open. Current main still drops the direct `/hooks/agent` body flag before dispatch, but current docs and prior security discussion point to this being an unresolved hook trust-boundary decision rather than a conservative cleanup close.

## What This Changes

The issue asks for direct `POST /hooks/agent` requests to honor `allowUnsafeExternalContent: true` instead of wrapping the prompt as external untrusted content.

## Best Possible Solution

Make the trust contract explicit: either document and test that direct `/hooks/agent` bodies cannot opt out, or add a deliberately gated direct-hook bypass with security docs and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main is a high-confidence reproduction: a raw direct hook body can include the field, `normalizeAgentPayload()` drops it, and downstream wrapping only skips when the synthetic `agentTurn` payload or Gmail config carries the opt-out.

## Solution Assessment

Unclear. Propagating the flag would satisfy the reporter's trusted dispatch flow, but it expands who can disable external-content markers, so maintainers should decide the hook-token trust boundary first.
