## Summary

Keep open. Current main still drops `allowUnsafeExternalContent` from direct `/hooks/agent` request bodies before dispatch, but the documented unsafe-content bypasses are config-owned mappings, Gmail config, and cron payloads, so the direct-body opt-out needs maintainer security/product judgment rather than an automatic one-line fix.

## What This Changes

Allow direct `POST /hooks/agent` callers to set `allowUnsafeExternalContent: true` so trusted hook dispatches arrive without external-content wrapping.

## Best Possible Solution

Make the trust contract explicit: either document and test that direct `/hooks/agent` bodies cannot opt out, or add a deliberately gated direct-hook bypass with security docs, replay-scope consideration, audit coverage, and focused regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: on current main, a raw `/hooks/agent` body can contain the field, `normalizeAgentPayload()` drops it, and downstream wrapping only skips when the synthetic `agentTurn` payload or Gmail config carries the bypass.

## Solution Assessment

Unclear. The suggested one-line propagation would fix the reporter's observed behavior, but it is not clearly the safest solution until maintainers decide whether direct hook callers may bypass external-content wrapping.
