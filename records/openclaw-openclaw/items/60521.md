## Summary

Keep open. Current main and the latest release still drop the direct `/hooks/agent` request-body flag, but the documented contract only advertises unsafe-content bypasses from config-owned mapping/Gmail surfaces and cron payloads, so accepting the requested direct HTTP-body bypass needs maintainer security/product judgment rather than cleanup closure or automatic repair.

## What This Changes

Allow direct `POST /hooks/agent` request bodies to set `allowUnsafeExternalContent` so trusted dispatch messages are not wrapped as untrusted external content.

## Best Possible Solution

Make the trust contract explicit: either document and test that direct `/hooks/agent` bodies cannot opt out, or add a deliberately gated direct-hook bypass with security docs and focused regression coverage.

## Reproduction Assessment

Yes, source-level reproduction is high confidence. A raw direct-hook body can contain the field, `normalizeAgentPayload()` drops it, and downstream wrapping only skips when the synthetic `agentTurn` payload or Gmail config carries the opt-out.

## Solution Assessment

Unclear. The proposed one-line propagation would fix the reporter's trusted dispatch use case, but it changes which hook-token callers can disable external-content wrapping, so maintainers should decide the trust boundary first.
