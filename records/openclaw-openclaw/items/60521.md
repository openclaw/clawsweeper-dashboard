## Summary

Keep open. Current main and v2026.5.18 still drop the direct request-body flag before dispatch, while docs only advertise server-side mapping/Gmail or cron payload bypasses, so maintainers need to decide whether direct hook-token callers may disable external-content wrapping.

## What This Changes

The issue asks OpenClaw to honor `allowUnsafeExternalContent: true` in direct `POST /hooks/agent` bodies so trusted hook dispatches are not wrapped as untrusted external content.

## Best Possible Solution

Make the direct hook trust contract explicit: either reject and document request-body bypasses with tests, or add a deliberately gated direct-hook bypass with security docs, audit coverage, and regression tests.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.5.18 shows a direct `/hooks/agent` body can include the field, `normalizeAgentPayload()` drops it, and downstream wrapping only skips when the internal agent-turn payload or Gmail config carries the opt-out.

## Solution Assessment

Unclear. Propagating the flag is the narrow mechanical change for the reporter's flow, but the maintainable fix needs an explicit maintainer decision on whether direct hook-token callers may disable external-content markers.
