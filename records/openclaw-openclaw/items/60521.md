## Summary

Keep open. Current main still drops the direct request-body flag before dispatch, while docs only advertise config-owned or cron-owned unsafe-content bypasses, so the remaining decision is whether direct hook callers should ever be allowed to disable external-content wrapping.

## What This Changes

Allow direct `POST /hooks/agent` request bodies with `allowUnsafeExternalContent: true` to skip external-content wrapping.

## Best Possible Solution

Make the trust contract explicit: either document and test that direct `/hooks/agent` bodies cannot opt out, or add a deliberately gated direct-hook bypass with security docs and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main is a high-confidence reproduction: a raw direct hook body can include the field, `normalizeAgentPayload()` drops it, and downstream wrapping only skips when the synthetic `agentTurn` payload or Gmail config carries the opt-out.

## Solution Assessment

Unclear. Propagating the flag is the narrow code change for the reporter’s flow, but it expands which hook-token callers can disable external-content markers, so the direct-hook trust boundary should be decided first.
