## Summary

Keep open. Current main still drops the direct `/hooks/agent` request-body flag before dispatch, but allowing that body field to bypass external-content wrapping is a security-sensitive trust-contract change that needs maintainer product/security direction.

## What This Changes

The issue requests that direct `/hooks/agent` requests honor `allowUnsafeExternalContent: true` instead of always wrapping the message as external untrusted content.

## Best Possible Solution

Make the hook trust contract explicit: either document and test that direct `/hooks/agent` bodies cannot opt out, or add a deliberately gated direct-hook bypass with security docs and focused regression coverage.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a raw direct hook body can contain the field, `normalizeAgentPayload()` drops it, and downstream wrapping only skips when the synthetic agent-turn payload or Gmail config carries the flag.

## Solution Assessment

Unclear. The proposed one-line propagation would satisfy the trusted automation use case, but it expands who can bypass external-content wrapping, so maintainers should decide the trust boundary first.
