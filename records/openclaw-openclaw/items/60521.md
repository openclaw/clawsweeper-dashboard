## Summary

Keep open. Current main still drops the raw direct-hook flag before dispatch, but docs and prior security discussion indicate direct HTTP-body opt-out may intentionally be outside the supported trust contract, so this needs maintainer security/product decision rather than cleanup close or automatic repair.

## What This Changes

The issue asks for direct `POST /hooks/agent` payloads with `allowUnsafeExternalContent: true` to reach agent turns without external-content warning wrappers.

## Best Possible Solution

Make the trust contract explicit: either document and test that direct `/hooks/agent` bodies cannot opt out, or add a deliberately gated direct-hook bypass with security docs and focused regression coverage.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a raw direct-hook body can contain the field, `normalizeAgentPayload()` drops it, and downstream wrapping only skips when the synthetic `agentTurn` payload contains the flag or Gmail config opts out.

## Solution Assessment

Unclear. The proposed one-line propagation would fix the reporter's trusted dispatch use case, but it changes which hook-token callers can disable external-content wrapping, so maintainers should decide the trust boundary first.
