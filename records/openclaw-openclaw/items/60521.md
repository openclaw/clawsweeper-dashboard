## Summary

Keep open. Current main still ignores the direct `/hooks/agent` request-body flag, but direct HTTP-body unsafe-content opt-out is not documented and related security context treats config-owned bypasses as the supported trust boundary, so this needs maintainer security/product decision rather than cleanup closure.

## What This Changes

Request that `POST /hooks/agent` honor `allowUnsafeExternalContent: true` from the request body and deliver the agent prompt unwrapped.

## Best Possible Solution

Make the trust contract explicit: either document and test that direct `/hooks/agent` bodies cannot opt out, or add a deliberately gated direct-hook bypass with security docs and focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a raw `/hooks/agent` body can contain the field, `normalizeAgentPayload()` drops it, and downstream wrapping only skips when the synthetic `agentTurn` payload or Gmail config carries the flag.

## Solution Assessment

Unclear. The proposed one-line propagation would satisfy the reporter's trusted dispatch use case, but it changes the hook-token trust boundary, so maintainers should decide the policy before implementation.
