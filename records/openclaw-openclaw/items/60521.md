## Summary

Keep open. Current main still drops the request flag in direct hook normalization before dispatch, while mapped hooks and the cron runner already preserve and honor it when present; because the flag disables external-content wrapping, the remaining decision is an explicit maintainer/security contract call for direct hook callers.

## What This Changes

The issue requests that direct `POST /hooks/agent` requests preserve `allowUnsafeExternalContent: true` so trusted hook-created agent turns are not wrapped as external untrusted content.

## Best Possible Solution

Make the direct hook trust contract explicit. If per-request bypass is intended, add the boolean to `HookAgentPayload` and `normalizeAgentPayload()`, include it in direct-hook replay scope, preserve it through dispatch, add regression coverage for direct `/hooks/agent`, and update hook/security docs with the same caution used for other unsafe bypass flags; otherwise reject or document the field as unsupported for direct hooks.

## Reproduction Assessment

Yes. Static inspection is enough: a direct `/hooks/agent` body can include the flag, but the normalizer drops it before the handler spreads `normalized.value` into dispatch, so the cron runner never receives the bypass value.

## Solution Assessment

Unclear until maintainers confirm the API contract. The narrow code fix is straightforward, but the safest maintainable solution depends on whether direct hook callers should be allowed to control an unsafe external-content bypass.
