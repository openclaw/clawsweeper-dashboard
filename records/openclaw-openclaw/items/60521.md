## Summary

Keep open. Current main still ignores `allowUnsafeExternalContent` on direct `/hooks/agent` request bodies, but current docs and prior security context indicate that direct HTTP-body opt-out may be intentionally unavailable, so this needs maintainer security/product judgment rather than a mechanical cleanup close.

## What This Changes

The issue requests honoring `allowUnsafeExternalContent: true` in direct `POST /hooks/agent` payloads so trusted hook-dispatched agent turns are not wrapped as external untrusted content.

## Best Possible Solution

Make the trust contract explicit: either document and test that direct `/hooks/agent` bodies cannot opt out, or add a deliberately gated direct-hook bypass with security docs, replay-scope consideration, and focused regression coverage.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: the raw request body can contain the field, `normalizeAgentPayload()` drops it, and downstream wrapping only skips when the synthetic `agentTurn` payload contains the flag.

## Solution Assessment

Unclear. The proposed one-line propagation would fix Atlas's observed behavior, but it is not clearly the safest solution until maintainers decide whether direct hook callers should be allowed to bypass external-content wrapping.
