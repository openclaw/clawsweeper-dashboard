## Summary

Keep open. Current main still drops the direct request-body flag before dispatch, while docs only advertise config-owned or cron-owned unsafe-content bypasses, so this remains an unresolved security/product boundary rather than a cleanup close.

## What This Changes

The issue asks for direct `POST /hooks/agent` JSON to propagate `allowUnsafeExternalContent: true` so trusted hook dispatches arrive without external-content wrapping.

## Best Possible Solution

Make the trust contract explicit: either document and test that direct `/hooks/agent` bodies cannot opt out, or add a deliberately gated direct-hook bypass with security docs and regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main is a high-confidence reproduction: a raw direct hook body can include the field, `normalizeAgentPayload()` drops it, and downstream wrapping only skips when the synthetic `agentTurn` payload or Gmail config carries the opt-out.

## Solution Assessment

Unclear. Propagating the flag would satisfy the reporter's trusted dispatch flow, but it expands which hook-token callers can disable external-content markers, so maintainers should decide the direct-hook trust boundary first.
