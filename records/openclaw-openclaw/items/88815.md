## Summary

This PR should remain open but is not merge-ready: the single-pin concept works in the supplied proof, while the current architecture has definite duplicate-delivery and multi-account routing defects plus an unresolved authorization-lifecycle boundary.

## Best Possible Solution

Define a channel-owned recipient capability keyed by channel and account, validate that capability against current destination policy at delivery time, suppress mirror-generated echo hooks for exactly-once fan-out, and only then expose the durable session, gateway, CLI, and Plugin SDK contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.98

Full review comments:

- **[P1] Prevent duplicate delivery before enabling multiple pins:** `src/infra/outbound/mirror-dispatch.ts:23-30`
  - body: With two pinned targets, each native mirror emits `message:sent`, and the post-hoc hook can deliver the same origin reply again to the sibling target. The command permits up to 16 pins, so this is a supported configuration rather than a hypothetical edge. Mark mirror-generated hook events and suppress their echo fan-out before landing.
  - confidence: 0.99
- **[P1] Route mirrors through the target Telegram account:** `src/infra/outbound/mirror-dispatch.ts:119-125`
  - body: The registry is keyed only by `channel` and first registration wins, but each Telegram bot core registers a closure over one account's bot, configuration, dependencies, and session runtime. A target whose `accountId` differs from the first registered account is therefore dispatched through the wrong account. Key dispatch by channel plus account or resolve an account-owned runtime per target.
  - confidence: 0.98
- **[P1] Revalidate destination authorization when delivering a pin:** `extensions/telegram/src/bot-message-context.ts:302-306`
  - body: `mirror: true` skips current account routing, group admission, sender allowlists, and topic requirements because authorization is assumed to have happened when `/pin` ran. Since the target is persisted, later policy or routing changes do not revoke delivery. Store a channel-owned authorization capability or revalidate the destination's current policy before dispatch.
  - confidence: 0.95
- **[P2] Remove the unrelated dependency graph change:** `package.json:1947`
  - body: The feature adds `bun-types` to root devDependencies and changes the lockfile, but none of the new mirror code imports or requires it. Repository automation blocks external dependency graph changes without explicit authorization, and this delta unnecessarily widens supply-chain scope. Restore `package.json` and `pnpm-lock.yaml` to the base versions.
  - confidence: 0.98
