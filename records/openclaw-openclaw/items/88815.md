## Label Justifications

- `merge-risk: 🚨 message-delivery`: Channel-only dispatcher lookup can send mirrored turns through the wrong Telegram account and routing context.

## Summary

The PR remains necessary as a new feature, but it is not merge-ready: the current design routes every Telegram mirror through the first account-scoped dispatcher registered for the channel, persists recipients while bypassing later destination admission changes, and includes an unrelated blocked dependency-graph change. The supplied Telegram screenshots are credible happy-path proof, but they do not exercise those blockers.

## What This Changes

The PR adds persisted session-bound cross-channel mirroring via `/pin`, gateway and CLI echo management, a plugin SDK mirror-dispatch API, and Telegram-native rendering of mirrored turns.

## Best Possible Solution

Keep native destination rendering, but make dispatch registration and lookup account-aware, validate the destination's current authorization and route before every mirrored delivery, define stale-pin cleanup and upgrade behavior, and remove the unrelated dependency changes.

## Reproduction Assessment

Yes at source level. Initializing two Telegram accounts retains only the first dispatcher, and changing destination policy after pinning cannot block the `mirror: true` path because the relevant checks are explicitly skipped.

## Solution Assessment

No. Reusing each channel's native rendering pipeline is the right direction, but the proposed channel-global dispatcher and historical-pin authorization model are not safe for account-scoped routing or later access revocation.
