Full review comments:

- **[P1] Share native echo options on chat.send assistant delivery:** `src/gateway/server-methods/chat.ts:3827-3839`
  - body: The message:sent hook passes `{ prefixed: false, filterTargets: ... }`, but chat.send assistant replies bypass that hook and call `fireEchoDeliveries` with no options here. That means web/control-ui/CLI turns still use the default `[echo]` assistant marker and, for streaming targets already handled by `launchStreamingEchoFanout`, can post a second final response after the native streamed Telegram message. Route this through the same helper/options as the message:sent path and cover the chat.send assistant case.
  - confidence: 0.93
- **[P1] Define the public channel-echo SDK contract before exporting it:** `scripts/lib/plugin-sdk-entrypoints.json:187`
  - body: Adding `channel-echo` to the public SDK entrypoint list makes `openclaw/plugin-sdk/channel-echo` a package contract, but the PR does not add the matching SDK docs/metadata/API-baseline classification required for new subpaths. Either keep this registry internal for this PR or publish it with the documented API lifecycle and contract tests so third-party plugins do not get an undocumented surface.
  - confidence: 0.86
- **[P1] Gate renderer registration to the owning channel:** `src/infra/outbound/echo-streaming.ts:53-55`
  - body: `registerEchoRendererFactory(channel, factory)` accepts an arbitrary channel string and the public SDK re-exports it, so any plugin that imports this subpath can replace the renderer for another channel such as `telegram`. This needs an ownership/uniqueness guard or a host-mediated registration path before the registry becomes public, otherwise echo content can be routed through the wrong channel renderer.
  - confidence: 0.78
