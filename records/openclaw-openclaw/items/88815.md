## Summary

Keep this PR open: the Telegram proof is strong and the channel-agnostic mirror direction is plausible, but the new public SDK registrar still trusts a plugin-supplied owner string for a delivery/revocation security boundary.

## What This Changes

The PR adds session pinning/channel echo via `/pin`, Gateway and CLI session echo controls, Telegram mirror/admission handling, a plugin SDK outbound registrar, protocol/Swift models, docs, and focused tests.

## Best Possible Solution

Keep the mirror concept, but bind mirror/admission ownership to host-provided plugin identity or keep the seam internal before exposing it as a public SDK contract.

## Reproduction Assessment

Yes. The blocking issue is source-reproducible: PR head exposes `createChannelOutboundRegistrar(owner: string)`, and the registry treats matching owner strings as the same owner.

## Solution Assessment

No. Mirroring through each target channel's own dispatch is a plausible direction, but the proposed public SDK registrar is not the best safe solution until ownership is bound by the plugin host rather than arbitrary plugin code.
