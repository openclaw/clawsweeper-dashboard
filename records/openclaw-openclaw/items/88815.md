## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Clear handled targets outside agent-run launches:** `src/infra/outbound/echo-streaming.ts:202`
  - body: A streamed target stays marked until the next `launchStreamingEchoFanout` call or abort dispose, but the post-hoc assistant echo filters are used by `message:sent` and chat.send for later replies too. Current dispatch can send command/abort replies via `routeReplyToOriginating` without starting a new agent run, so after one streamed echo the next command reply to the same pinned target is skipped as if it were a duplicate. Clear these marks when the post-hoc mirror has had its chance for the run, or key them by run instead of only by session.
  - confidence: 0.88
- **[P1] Enforce channel ownership for renderer registration:** `src/plugin-sdk/channel-echo.ts:20`
  - body: Exporting `registerEchoRendererFactory(channel, factory)` as a public SDK function makes the ownership contract documentary only: any plugin can register `telegram` first, and first-wins then prevents the real Telegram renderer from taking over. The registration needs to be bound to the loaded channel plugin's own id, or kept behind a core-owned/channel-owned seam that cannot be claimed by a foreign caller.
  - confidence: 0.9
- **[P2] Do not await unbounded renderer factories on the origin turn:** `src/infra/outbound/echo-streaming.ts:181`
  - body: `EchoRendererFactory` may return a Promise, and `runAgentTurnWithFallback` awaits `launchStreamingEchoFanout` before the model starts. That means a slow or hanging channel renderer factory for any pinned target stalls the origin conversation, despite the comment saying target rendering must not block the origin turn; make factory setup bounded/synchronous or move it off the critical path with a timeout/fallback.
  - confidence: 0.82
