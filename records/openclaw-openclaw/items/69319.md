## Best Possible Solution

Land a rebased, narrow Gateway uninstall RPC only after it is registered in the current descriptor registry, deletion is limited to ClawHub-originated or lockfile-tracked state, client input errors are classified correctly, current protocol/client artifacts are synced, and real behavior proof is added.

## Solution Assessment

No. The Gateway RPC direction is plausible, but this patch is not yet the best implementation because it needs a rebase, descriptor registration, scoped deletion, runtime-config use, input-error classification, synced client artifacts, and real behavior proof.

  - body: Current main builds core Gateway methods from `CORE_GATEWAY_METHOD_SPECS` and throws when a handler is not listed there.   - body: `uninstallSkillFromClawHub` deletes `targetDir` whenever it exists, even when there is no `.clawhub` origin metadata or lockfile entry. A slug collision with a manually managed or uploaded skill can therefore remove content outside the documented ClawHub uninstall scope.
  - confidence: 0.92
- **[P2] Use the request runtime config:** `src/gateway/server-methods/skills.ts:252-253`
  - body: The new handler calls `loadConfig()` directly while current adjacent skills handlers use `context.getRuntimeConfig()`. That can resolve a persisted/default workspace instead of the active Gateway runtime config for this request after rebase.
  - confidence: 0.86
- **[P2] Map invalid uninstall slugs to INVALID_REQUEST:** `src/gateway/server-methods/skills.ts:276`
  - body: The schema only enforces a non-empty string, so malformed values such as `../evil` pass validation and fail later in backend normalization. Mapping every backend failure to `UNAVAILABLE` misclassifies client input errors as server outages.
  - confidence: 0.88

  - body: When the directory is gone but the lockfile still contains the slug, the helper removes state while returning `removed: false`, causing the handler to say the skill was not installed even though cleanup happened.
