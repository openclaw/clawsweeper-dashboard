## Summary

Keep open. Current main still lacks `skills.uninstall`, so the PR is useful, but the proposed patch is not mergeable yet because the destructive ClawHub uninstall helper can delete untracked/manual workspace skill folders and the Gateway handler diverges from current request-context and error-classification patterns. The linked feature request is closed, while #63272 remains the separate CLI uninstall design track.

## What This Changes

This PR adds a Gateway `skills.uninstall` RPC with protocol schema/types, Swift generated models, handler registration and admin scope wiring, a ClawHub backend removal helper, docs, and focused backend plus Gateway handler tests.

## Best Possible Solution

Land a narrow Gateway `skills.uninstall` implementation that only removes ClawHub-proven or lockfile-tracked default-workspace skill installs, leaves manual/gateway-installer skill folders alone, maps slug/request validation failures to `INVALID_REQUEST`, uses `context.getRuntimeConfig()`, and keeps protocol schemas, generated clients, docs, scopes, and tests aligned. Leave CLI command design to #63272.

## Reproduction Assessment

Yes. From the PR diff, create `workspace/skills/<slug>` with no `.clawhub/origin.json` and no `.clawhub/lock.json` entry; `uninstallSkillFromClawHub` sees `dirExists` and removes it. A malformed slug such as `../evil` also passes the non-empty schema and is then mapped to `UNAVAILABLE` after backend normalization fails.

## Solution Assessment

No. The endpoint direction is reasonable, but this is not the best implementation until deletion is gated on ClawHub provenance, the handler uses the Gateway runtime config context, and client input errors are classified as request errors.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Refuse untracked skill folders:** `src/agents/skills-clawhub.ts:464-467`
  - body: `uninstallSkillFromClawHub` only returns the idempotent not-installed result when the directory is missing, so an existing manually managed or gateway-installer `skills/<slug>` folder with no ClawHub origin or lockfile entry is still deleted. Gate `fs.rm` on ClawHub provenance (`originBefore` or lock membership) and otherwise leave the folder in place.
  - confidence: 0.92
- **[P2] Use the Gateway runtime config context:** `src/gateway/server-methods/skills.ts:254-255`
  - body: The new handler calls a direct config load while adjacent skills handlers resolve workspaces through `context.getRuntimeConfig()`. That can target a stale or default workspace instead of the request's active runtime config; destructure `context` and mirror `skills.install`/`skills.update`.
  - confidence: 0.86
- **[P2] Classify bad uninstall slugs as invalid requests:** `src/gateway/server-methods/skills.ts:278`
  - body: Malformed non-empty slugs pass the schema and are rejected by `normalizeTrackedSlug`, but the handler maps every backend failure to `UNAVAILABLE`. Treat slug-normalization/path validation failures as `INVALID_REQUEST` so clients do not retry bad input as a server outage.
  - confidence: 0.88
