## Summary

Keep this PR open. Current main still lacks a `skills.uninstall` Gateway method, so the contribution is not obsolete, but the proposed patch is not merge-ready because the new destructive helper can delete untracked/manual workspace skill folders, client slug errors are surfaced as `UNAVAILABLE`, and the handler is stale against current Gateway runtime-config patterns.

## What This Changes

The PR adds a Gateway `skills.uninstall` RPC with ClawHub backend removal, protocol schema/types and Swift model updates, method registration/scope wiring, docs, and focused backend plus handler tests.

## Best Possible Solution

Land a narrow current-main-compatible Gateway `skills.uninstall` method that only removes ClawHub-proven or lockfile-tracked default-workspace skill installs, preserves manual skill folders, maps slug validation failures to `INVALID_REQUEST`, uses the handler runtime config context, and keeps schema exports, generated clients, method scopes, docs, and focused tests aligned. Leave the CLI uninstall design to #63272.

## Reproduction Assessment

Yes. Current main can be checked by searching and inspecting the method registry, which shows the RPC is absent; the PR defect can be reproduced from the diff by creating `workspace/skills/<slug>` without `.clawhub/origin.json` or lockfile membership and observing the helper take the `dirExists` removal path.

## Solution Assessment

No. The endpoint direction is reasonable, but this patch is not the best implementation until uninstall is gated to ClawHub provenance, request errors are classified correctly, and the handler follows current runtime-config patterns.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Refuse uninstalling untracked skill folders:** `src/agents/skills-clawhub.ts:464-467`
  - body: `uninstallSkillFromClawHub` only no-ops when the directory is missing, so an existing manually managed `skills/<slug>` folder with no `.clawhub` origin or lockfile entry is still deleted. Gate the `fs.rm` path on ClawHub provenance (`originBefore` or lock membership) and otherwise return the idempotent not-installed result.
  - confidence: 0.92
- **[P2] Use the Gateway runtime config context:** `src/gateway/server-methods/skills.ts:254-255`
  - body: Current main's skills handlers resolve workspaces through `context.getRuntimeConfig()`, but the new uninstall handler calls `loadConfig()` directly. This can use a stale/default config snapshot instead of the request's runtime config; mirror `skills.install`/`skills.update` and pass `context` into the handler.
  - confidence: 0.86
- **[P2] Return INVALID_REQUEST for bad uninstall slugs:** `src/gateway/server-methods/skills.ts:278`
  - body: Malformed non-empty slugs pass the schema, are rejected by `normalizeTrackedSlug`, and then every backend failure is mapped to `UNAVAILABLE`. That misclassifies client input such as `../evil` as a server outage; split slug-validation failures into `INVALID_REQUEST`.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch introduces a destructive admin RPC that can remove non-ClawHub/manual skill folders under the workspace because the deletion path is not gated on ClawHub provenance.

Concerns:

- **[high] Untracked skill deletion:** `src/agents/skills-clawhub.ts:466`
  - body: The helper reads ClawHub origin and lockfile state but deletes `targetDir` whenever the directory exists, so a slug collision with a manually managed workspace skill can be removed by `skills.uninstall` even though the RPC is documented as ClawHub-scope cleanup.
  - confidence: 0.92
