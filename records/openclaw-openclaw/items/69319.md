## Summary

Keep open: current main still lacks the Gateway uninstall RPC, but this PR still has blocking correctness and security issues around destructive deletion scope, runtime config selection, invalid-input classification, and missing real behavior proof.

## What This Changes

The branch adds a `skills.uninstall` Gateway RPC with protocol schema/types, a ClawHub uninstall helper, admin-scope registration, docs, Swift models, and focused tests.

## Best Possible Solution

Land a narrow Gateway uninstall implementation that removes only ClawHub-proven or lockfile-tracked default-workspace installs, uses the active Gateway runtime config, maps client slug errors to `INVALID_REQUEST`, and leaves CLI uninstall design to https://github.com/openclaw/openclaw/issues/63272.

## Reproduction Assessment

Yes for the PR-introduced issues by source inspection: an existing untracked `skills/<slug>` directory is deleted, and malformed non-empty slugs reach backend normalization before being reported as `UNAVAILABLE`. I did not execute the destructive flow because this review is read-only.

## Solution Assessment

No. The Gateway RPC direction is reasonable, but this patch is not ready until deletion is gated on ClawHub provenance, runtime config comes from the request context, client input errors are classified correctly, and real behavior proof is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Refuse untracked skill folders:** `src/agents/skills-clawhub.ts:464-466`
  - body: `uninstallSkillFromClawHub` deletes any existing `skills/<slug>` directory even when there is no `.clawhub` origin marker and no workspace lockfile entry. Gate removal on ClawHub provenance and otherwise return the idempotent not-installed result.
  - confidence: 0.92
- **[P2] Use the Gateway runtime config:** `src/gateway/server-methods/skills.ts:254-255`
  - body: The new handler calls `loadConfig()` directly while adjacent skills handlers use `context.getRuntimeConfig()`. That can target the persisted/default workspace instead of the request's active Gateway runtime config.
  - confidence: 0.86
- **[P2] Classify invalid slugs as INVALID_REQUEST:** `src/gateway/server-methods/skills.ts:278`
  - body: The schema only enforces a non-empty slug, so malformed values like `../evil` are rejected later by backend normalization, but the handler reports every backend failure as `UNAVAILABLE`. Treat slug/path validation failures as client request errors.
  - confidence: 0.88
- **[P3] Report lockfile-only cleanup accurately:** `src/agents/skills-clawhub.ts:472-474`
  - body: When only a stale lockfile entry exists, the helper removes that entry but returns `removed: false`, causing the handler to say the skill was not installed. Return a cleanup signal that reflects the lockfile mutation.
  - confidence: 0.74
