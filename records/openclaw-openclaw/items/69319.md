## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Refuse non-ClawHub skill directory removal:** `src/agents/skills-clawhub.ts:464-467`
  - body: `skills.uninstall` is documented as a ClawHub removal path, but this deletes `skills/<slug>` whenever the directory exists, even when there is no ClawHub origin metadata and no lockfile entry. A manually managed or differently installed skill with the same slug can be removed through this admin RPC, so gate recursive deletion on validated ClawHub provenance and treat untracked folders as not installed or a refusal.
  - confidence: 0.92
- **[P2] Map malformed slugs to INVALID_REQUEST:** `src/gateway/server-methods/skills.ts:278`
  - body: The schema only requires a non-empty slug, so values like `../evil` reach the backend slug normalizer and then come back through the catch-all `UNAVAILABLE` response. That turns client input errors into retryable server outages; distinguish slug validation failures and return `INVALID_REQUEST`.
  - confidence: 0.86
- **[P2] Preserve agent workspace routing parity:** `src/gateway/server-methods/skills.ts:254-259`
  - body: Current `skills.install` and `skills.update` accept `agentId` and resolve the selected agent workspace, but the new uninstall handler always targets the default workspace. Clients managing skills for a non-default agent can install or update there but cannot uninstall from the same workspace, so add `agentId` to the protocol and route through the same resolver before landing.
  - confidence: 0.83
