## Summary

This PR should stay open: current main still lacks `skills.uninstall`, but the proposed patch introduces a destructive ClawHub uninstall path that can delete untracked skill directories, uses persisted config instead of the Gateway runtime config, and still lacks after-fix real behavior proof.

## What This Changes

The PR adds a Gateway `skills.uninstall` RPC with protocol schemas/types, admin-scope registration, ClawHub backend deletion logic, Swift protocol models, docs, and focused tests.

## Best Possible Solution

Land a narrow Gateway uninstall RPC after it only removes ClawHub-originated or lockfile-tracked default-workspace skills, uses the active Gateway runtime config, maps client slug errors correctly, and includes real behavior proof.

## Reproduction Assessment

Yes for the PR-introduced defects by source inspection: the branch deletes an existing untracked skill directory and maps backend slug normalization failures to `UNAVAILABLE`. I did not execute the destructive path because this review is read-only.

## Solution Assessment

No. The Gateway RPC direction is reasonable, but this patch should gate deletion on ClawHub provenance, use request runtime config, classify invalid slugs as request errors, and add real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Refuse untracked skill folders:** `src/agents/skills-clawhub.ts:464-466`
  - body: `uninstallSkillFromClawHub` only treats the skill as not installed when the directory is absent, so any existing `skills/<slug>` folder is deleted even without a `.clawhub` origin marker or lockfile entry. A slug collision with a manually managed or gateway-installer skill would remove content outside the documented ClawHub uninstall scope; gate deletion on ClawHub provenance and otherwise return the idempotent not-installed result.
  - confidence: 0.92
- **[P2] Use the Gateway runtime config:** `src/gateway/server-methods/skills.ts:254-255`
  - body: The new handler calls `loadConfig()` directly while adjacent skills handlers use `context.getRuntimeConfig()`. That can target the persisted/default workspace instead of the active Gateway runtime config for this request, so resolve the default agent workspace from the request context.
  - confidence: 0.86
- **[P2] Classify invalid slugs as INVALID_REQUEST:** `src/gateway/server-methods/skills.ts:278`
  - body: The schema only requires a non-empty slug, so malformed values like `../evil` reach backend normalization and then every backend failure is returned as `UNAVAILABLE`. Slug/path validation failures are client request errors and should be reported as `INVALID_REQUEST` so clients do not retry them as server outages.
  - confidence: 0.88
- **[P3] Report lockfile-only cleanup accurately:** `src/agents/skills-clawhub.ts:472-474`
  - body: When the directory is already gone but the lockfile still contains the slug, the helper removes the lockfile entry while returning `removed: false`, causing the handler to say the skill was not installed. Return a cleanup signal that reflects the lockfile mutation so operators can tell that state changed.
  - confidence: 0.74
