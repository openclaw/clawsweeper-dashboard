## Summary

Keep open. Current main still lacks `skills.uninstall`, so the PR is not obsolete, but the proposed patch should not merge as-is because the destructive ClawHub uninstall helper can delete untracked workspace skill folders and the Gateway handler is stale against the current runtime-config/error-classification patterns.

## What This Changes

This PR adds a Gateway `skills.uninstall` RPC with protocol schema/types, generated Swift models, method registration/scope wiring, a ClawHub filesystem/lockfile uninstall helper, docs, and focused tests.

## Best Possible Solution

Land a narrow Gateway uninstall implementation that only removes ClawHub-proven or lockfile-tracked default-workspace skill installs, leaves manual/gateway-installer skill folders alone, uses the active Gateway runtime config context, maps slug/path validation failures to `INVALID_REQUEST`, keeps protocol/docs/generated clients/tests aligned, and adds the required user-facing changelog entry. Keep the separate CLI command design in #63272.

## Reproduction Assessment

Yes. On the PR head, create `<workspace>/skills/<slug>` without a `.clawhub/origin.json` marker or workspace `.clawhub/lock.json` entry; `fileExists(targetDir)` is enough to reach `fs.rm`. A malformed slug such as `../evil` also passes the non-empty schema and is later reported as `UNAVAILABLE`.

## Solution Assessment

No. The Gateway RPC direction is reasonable, but this implementation is not the best fix until deletion is gated on ClawHub provenance, runtime config comes from the request context, and client input errors are classified as invalid requests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Refuse untracked skill folders:** `src/agents/skills-clawhub.ts:464-466`
  - body: `uninstallSkillFromClawHub` only treats the skill as not installed when the directory is missing, so an existing manual or gateway-installer `skills/<slug>` folder with no ClawHub origin or lock entry is still deleted. Gate `fs.rm` on ClawHub provenance and return the idempotent result for untracked folders.
  - confidence: 0.92
- **[P2] Use the Gateway runtime config context:** `src/gateway/server-methods/skills.ts:254-255`
  - body: The new handler calls `loadConfig()` directly while current Gateway skills handlers resolve workspaces through `context.getRuntimeConfig()`. That can target a stale/default workspace instead of the request's active Gateway runtime config; destructure `context` and mirror the current `skills.install`/`skills.update` pattern.
  - confidence: 0.86
- **[P2] Classify bad uninstall slugs as invalid requests:** `src/gateway/server-methods/skills.ts:278`
  - body: Malformed non-empty slugs pass the schema and are rejected later by `normalizeTrackedSlug`, but the handler maps every backend failure to `UNAVAILABLE`. Treat slug-normalization/path validation failures as `INVALID_REQUEST` so clients do not retry bad input as a server outage.
  - confidence: 0.88
- **[P3] Report lockfile-only cleanup accurately:** `src/agents/skills-clawhub.ts:472-474`
  - body: When the directory is already gone but the lockfile still contains the slug, the helper drops the lockfile entry while returning `removed: false`, and the handler says the skill was not installed. Return a cleanup signal that reflects the lockfile mutation so operators are not told the call was a pure no-op.
  - confidence: 0.74
