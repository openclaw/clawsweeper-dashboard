## Summary

Keep open. Current main still has no `skills.uninstall` Gateway method, and this PR is useful, but the proposed implementation is not merge-ready because it adds a destructive admin RPC that can delete non-ClawHub skill folders and has request-routing/error-classification issues.

## What This Changes

This PR adds a Gateway `skills.uninstall` RPC with protocol schema/types, Swift models, handler registration and admin scope wiring, a ClawHub backend uninstall helper, docs, and focused tests.

## Best Possible Solution

Land a narrow Gateway uninstall implementation that only removes ClawHub-proven or lockfile-tracked default-workspace skill installs, leaves manual/gateway-installer skill folders alone, maps slug/request validation failures to `INVALID_REQUEST`, uses `context.getRuntimeConfig()`, and keeps protocol schemas, generated clients, docs, scopes, and tests aligned. Keep CLI command design in #63272.

## Reproduction Assessment

Yes. From the PR diff, create `workspace/skills/<slug>` without a `.clawhub/origin.json` marker or workspace `.clawhub/lock.json` entry; the helper still sees the directory and removes it. A malformed slug such as `../evil` also passes the non-empty schema and is then reported as `UNAVAILABLE` after backend normalization fails.

## Solution Assessment

No. The Gateway RPC direction is reasonable, but this implementation is not the best fix until destructive deletion is gated on ClawHub provenance, the active runtime config context is used, and client input errors are classified as invalid requests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Refuse untracked skill folders:** `src/agents/skills-clawhub.ts:464-467`
  - body: `uninstallSkillFromClawHub` only treats the skill as not installed when the directory is missing, so an existing manual or gateway-installer `skills/<slug>` folder with no ClawHub origin or lockfile entry is still deleted. Gate `fs.rm` on ClawHub provenance and otherwise return the idempotent not-installed result.
  - confidence: 0.92
- **[P2] Use the Gateway runtime config context:** `src/gateway/server-methods/skills.ts:254-255`
  - body: The new handler calls `loadConfig()` directly while adjacent skills handlers resolve workspaces through `context.getRuntimeConfig()`. That can target a stale/default workspace instead of the request's active Gateway runtime config; destructure `context` and mirror `skills.install`/`skills.update`.
  - confidence: 0.86
- **[P2] Classify bad uninstall slugs as invalid requests:** `src/gateway/server-methods/skills.ts:278`
  - body: Malformed non-empty slugs pass the schema and are rejected later by `normalizeTrackedSlug`, but the handler maps every backend failure to `UNAVAILABLE`. Treat slug-normalization/path validation failures as `INVALID_REQUEST` so clients do not retry bad input as a server outage.
  - confidence: 0.88
- **[P3] Report lockfile-only cleanup accurately:** `src/agents/skills-clawhub.ts:469-475`
  - body: When the directory is already gone but the lockfile still contains the slug, the helper drops the lockfile entry while returning `removed: false`, and the handler says the skill was not installed. Return a cleanup signal that reflects the lockfile mutation so operators are not told the call was a pure no-op.
  - confidence: 0.74
