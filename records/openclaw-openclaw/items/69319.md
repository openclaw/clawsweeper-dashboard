## Summary

Keep open. The PR implements a useful missing Gateway RPC, but the current patch still has blocking correctness and security issues around destructive deletion scope, runtime config selection, and request-error classification, and it lacks contributor real behavior proof.

## What This Changes

The PR adds a Gateway `skills.uninstall` RPC with protocol schema/types, Swift models, admin-scope method registration, a ClawHub uninstall helper, docs, and focused tests.

## Best Possible Solution

Land a narrow Gateway uninstall implementation that only removes ClawHub-proven or lockfile-tracked default-workspace skill installs, uses the active Gateway runtime config, classifies client input errors as `INVALID_REQUEST`, and keeps CLI uninstall design in the separate open issue.

## Reproduction Assessment

Yes. From the PR source diff, create `workspace/skills/<slug>` without a ClawHub origin marker or lockfile entry and the helper still deletes it; a malformed slug such as `../evil` also passes the non-empty schema and is reported as `UNAVAILABLE` after backend rejection.

## Solution Assessment

No. The Gateway RPC direction is reasonable, but this patch is not the best fix until destructive deletion is gated on ClawHub provenance, the active runtime config context is used, and malformed client input is classified as invalid request data.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Refuse untracked skill folders:** `src/agents/skills-clawhub.ts:464-466`
  - body: `uninstallSkillFromClawHub` deletes any existing `skills/<slug>` directory even when there is no ClawHub origin marker or workspace lockfile entry. Gate `fs.rm` on ClawHub provenance and otherwise return the idempotent not-installed result.
  - confidence: 0.92
- **[P2] Use the Gateway runtime config context:** `src/gateway/server-methods/skills.ts:254-255`
  - body: The new handler calls `loadConfig()` directly while adjacent skills handlers resolve workspaces through `context.getRuntimeConfig()`. That can target a stale/default workspace instead of the request's active Gateway runtime config.
  - confidence: 0.86
- **[P2] Classify bad uninstall slugs as invalid requests:** `src/gateway/server-methods/skills.ts:278`
  - body: Malformed non-empty slugs pass the schema and are rejected later by backend normalization, but the handler maps every backend failure to `UNAVAILABLE`. Treat slug/path validation failures as `INVALID_REQUEST` so clients do not retry bad input as an outage.
  - confidence: 0.88
- **[P3] Report lockfile-only cleanup accurately:** `src/agents/skills-clawhub.ts:472`
  - body: When the directory is already gone but the lockfile still contains the slug, the helper removes the lock entry while returning `removed: false`, and the handler says the skill was not installed. Return a cleanup signal that reflects the lockfile mutation.
  - confidence: 0.74
