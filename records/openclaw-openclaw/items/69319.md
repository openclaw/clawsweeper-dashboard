## Summary

Keep open. Current main still lacks the requested Gateway `skills.uninstall` RPC, but this PR should not merge as-is because the head diff still has blocking destructive-delete, runtime-config, error-classification, and proof gaps.

## What This Changes

Adds a Gateway `skills.uninstall` RPC with protocol schema/types, admin-scope registration, Swift models, a ClawHub backend uninstall helper, docs, and focused tests.

## Best Possible Solution

Land a narrow Gateway uninstall implementation that removes only ClawHub-proven or lockfile-tracked default-workspace skill installs, uses the active Gateway runtime config, maps client input errors to `INVALID_REQUEST`, and keeps CLI uninstall design in the separate open issue.

## Reproduction Assessment

Yes for the PR-introduced issues by source inspection: the helper deletes any existing target directory without ClawHub provenance, and malformed non-empty slugs reach backend normalization before being reported as `UNAVAILABLE`. I did not run tests because this review is read-only.

## Solution Assessment

No. The Gateway RPC direction is reasonable, but this patch is not the best fix until deletion is gated on ClawHub provenance, runtime config comes from the request context, client input errors are classified correctly, and real behavior proof is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Refuse untracked skill folders:** `src/agents/skills-clawhub.ts:464-466`
  - body: `uninstallSkillFromClawHub` deletes any existing `skills/<slug>` directory even when there is no ClawHub origin marker or workspace lockfile entry. Gate removal on ClawHub provenance and otherwise return the idempotent not-installed result.
  - confidence: 0.92
- **[P2] Use the Gateway runtime config context:** `src/gateway/server-methods/skills.ts:254-255`
  - body: The new handler calls `loadConfig()` directly while adjacent skills handlers resolve workspaces through `context.getRuntimeConfig()`. That can target a stale/default workspace instead of the request's active Gateway runtime config.
  - confidence: 0.86
- **[P2] Classify bad uninstall slugs as invalid requests:** `src/gateway/server-methods/skills.ts:278`
  - body: Malformed non-empty slugs pass the schema and are rejected later by backend normalization, but the handler maps every backend failure to `UNAVAILABLE`. Treat slug/path validation failures as `INVALID_REQUEST` so clients do not retry bad input as an outage.
  - confidence: 0.88
- **[P3] Report lockfile-only cleanup accurately:** `src/agents/skills-clawhub.ts:472`
  - body: When the directory is already gone but the lockfile still contains the slug, the helper removes the lock entry while returning `removed: false`, and the handler says the skill was not installed. Return a cleanup signal that reflects the lockfile mutation.
  - confidence: 0.74
