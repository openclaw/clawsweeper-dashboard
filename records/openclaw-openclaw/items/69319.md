## What This Changes

The PR adds a Gateway `skills.uninstall` RPC with protocol schema/types, admin-scope registration, ClawHub workspace deletion and lockfile cleanup, Swift client models, docs, and focused tests.

## Best Possible Solution

Land a narrow Gateway uninstall RPC only after it deletes ClawHub-originated or lockfile-tracked default-workspace skills, uses the active Gateway runtime config, classifies client slug errors correctly, and includes real behavior proof.

## Reproduction Assessment

Yes. The PR-introduced defects are source-reproducible: an existing untracked `skills/<slug>` directory reaches `fs.rm`, and malformed slugs rejected by backend normalization are returned as `UNAVAILABLE`; I did not execute the destructive path because this review is read-only.

## Solution Assessment

No. The RPC direction is reasonable, but this implementation should gate deletion on ClawHub provenance, use request runtime config, classify invalid slugs as request errors, and add real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Refuse untracked skill folders:** `src/agents/skills-clawhub.ts:464-466`
  - body: `uninstallSkillFromClawHub` only returns the idempotent not-installed response when the directory is absent. If `skills/<slug>` exists without `.clawhub` origin metadata or a lockfile entry, this still deletes it, so a slug collision can remove manually managed or uploaded skill content outside the documented ClawHub uninstall scope.
  - confidence: 0.92
- **[P2] Use the Gateway runtime config:** `src/gateway/server-methods/skills.ts:254-255`
  - body: The new handler calls `loadConfig()` directly while adjacent skills handlers use `context.getRuntimeConfig()`. That can target the persisted/default workspace instead of the active Gateway runtime config for this request, so resolve the default agent workspace from the request context.
  - confidence: 0.86
- **[P2] Classify invalid slugs as INVALID_REQUEST:** `src/gateway/server-methods/skills.ts:278`
  - body: The schema only enforces a non-empty slug, so malformed values such as `../evil` pass validation and fail later in backend normalization. Mapping every backend failure to `UNAVAILABLE` misclassifies client input errors as server outages and can lead clients to retry instead of correcting the request.
  - confidence: 0.88
- **[P3] Report lockfile-only cleanup accurately:** `src/agents/skills-clawhub.ts:472-474`
  - body: When the directory is already gone but the lockfile still contains the slug, the helper removes the lockfile entry while returning `removed: false`, causing the handler to say the skill was not installed even though state changed. Return a cleanup signal that distinguishes this case.
  - confidence: 0.74
