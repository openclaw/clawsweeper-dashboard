  - body: Current main builds core Gateway methods from `CORE_GATEWAY_METHOD_SPECS` and throws when a handler is not listed there.   - body: `uninstallSkillFromClawHub` deletes `targetDir` whenever it exists, even when there is no `.clawhub` origin metadata or lockfile entry. A slug collision with a manually managed or uploaded skill can therefore remove content outside the documented ClawHub uninstall scope.
  - confidence: 0.92
- **[P2] Use the request runtime config:** `src/gateway/server-methods/skills.ts:254-255`
  - body: The new handler calls `loadConfig()` directly while current adjacent skills handlers use `context.getRuntimeConfig()`. That can resolve a persisted/default workspace instead of the active Gateway runtime config for this request after rebase.
  - confidence: 0.86
- **[P2] Map invalid uninstall slugs to INVALID_REQUEST:** `src/gateway/server-methods/skills.ts:278`
  - body: The schema accepts any non-empty slug, so malformed values such as `../evil` pass validation and fail later in backend normalization. Mapping every backend failure to `UNAVAILABLE` misclassifies client input errors as server outages.
  - confidence: 0.88
- **[P3] Report lockfile-only cleanup accurately:** `src/agents/skills-clawhub.ts:472-474`
  - body: When the directory is gone but the lockfile still contains the slug, the helper removes state while returning `removed: false`, causing the handler to say the skill was not installed even though cleanup happened.
  - confidence: 0.74
