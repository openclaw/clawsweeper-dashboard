## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Refuse untracked skill folders:** `src/agents/skills-clawhub.ts:464-466`
  - body: `skills.uninstall` is scoped to ClawHub cleanup, but this branch deletes `skills/<slug>` whenever the directory exists, even when there is no ClawHub origin metadata or lockfile entry. A manually managed or uploaded skill with the same slug would be removed by a ClawHub uninstall request, so gate `fs.rm` on ClawHub provenance and report `removed: false` for untracked folders.
  - confidence: 0.94
- **[P1] Register the RPC in the core descriptor table:** `src/gateway/server-methods-list.ts:69`
  - body: Current main no longer uses this literal method list or the old scope group as the source of truth; core method listing, authorization, startup availability, and write policy come from `CORE_GATEWAY_METHOD_SPECS`. After rebase, adding only the old `server-methods-list` entry will leave `skills.uninstall` out of the canonical descriptor registry, so add it there instead.
  - confidence: 0.88
- **[P2] Use the request runtime config:** `src/gateway/server-methods/skills.ts:254-255`
  - body: The new handler loads config directly instead of using the Gateway request context, unlike current main's skills handlers. That can route uninstall to a different default workspace than the running Gateway request, so resolve the workspace from `context.getRuntimeConfig()` after rebasing.
  - confidence: 0.82
- **[P2] Map invalid uninstall slugs to INVALID_REQUEST:** `src/gateway/server-methods/skills.ts:278`
  - body: The schema only rejects empty strings; traversal slugs are rejected later by the backend and this handler maps every backend failure to `UNAVAILABLE`. Client input errors like `../evil` should be returned as `INVALID_REQUEST` so callers do not retry them as server outages.
  - confidence: 0.9
- **[P3] Report lockfile-only cleanup accurately:** `src/agents/skills-clawhub.ts:472-474`
  - body: When the directory is already gone but a ClawHub lockfile entry exists, this removes the lock entry but returns `removed: false`, leading the handler to say the skill was not installed. Return a distinct cleanup signal, or make `removed` reflect lockfile cleanup, so operators can tell a stale tracked install was repaired.
  - confidence: 0.78
