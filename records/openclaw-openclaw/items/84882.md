## What This Changes

The issue reports that memory-core Dreaming/recall normalization may silently delete date-only daily memory files while leaving dangling short-term recall references.

## Best Possible Solution

Identify the actual delete or truncate path, then add a regression that seeds date-only daily files, runs Dreaming with `separate` storage, and proves file existence/content plus recall-store references are preserved or explicitly audited.

## Reproduction Assessment

No. The report has detailed production evidence, but I did not find a high-confidence current-main reproduction path; the named normalizer rewrites recall JSON/lock state, while the separate-storage path reads daily files and writes phase reports under `memory/dreaming/`.

## Solution Assessment

Unclear. The requested guards are directionally sensible, but the maintainable fix should first locate the actual daily-file delete or truncate path and then add focused preservation and audit coverage around that path.
