## Summary

Keep open; the fallback direction is plausible, but the patch currently turns broad ClawHub artifact request failures into npm fallback, which can bypass ClawHub moderation or security blocks, and the real behavior proof is mock-only.

## What This Changes

This PR adds `ARTIFACT_UNAVAILABLE` handling so official ClawHub install and update failures can fall back to npm, updates related tests and changelog, and refreshes Codex prompt snapshots.

## Best Possible Solution

Keep the official npm fallback only for benign artifact-routing unavailability, fail closed on ClawHub moderation/security/auth/server errors, and require redacted live install or update proof before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the review finding: a non-404 `ClawHubRequestError` during artifact download becomes `ARTIFACT_UNAVAILABLE`, then the update/onboarding/doctor fallback predicates allow npm fallback. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. The maintainable fix is to distinguish benign artifact rollout/unavailable states from ClawHub moderation, auth, or security failures and keep those fail-closed.
