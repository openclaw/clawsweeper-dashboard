## Summary

Keep open: the protected `maintainer` label requires explicit maintainer handling, and current `main` still has a plausible source-level gap where migration provider options are keyed from raw `opts.provider` even though the apply path separately carries a resolved `providerId`.

## What This Changes

The issue requests that migrate provider options be computed from the resolved provider id so Codex-specific options survive default or normalized Codex apply paths.

## Best Possible Solution

Thread the resolved provider id into provider-option construction, use it for both plan and apply contexts, and add regression coverage for Codex `configPatchMode` when the raw provider option is missing or normalized separately.

## Reproduction Assessment

Yes, at source level: call `runMigrationApply` with `providerId: "codex"`, `opts.configPatchMode: "return"`, and no raw `opts.provider`; current code will build `undefined` provider options. I did not run that as a test because the checkout is read-only.

## Solution Assessment

Yes: using the resolved provider id in `buildMigrationProviderOptions` is the narrowest maintainable fix and keeps the migration provider context contract unchanged.
