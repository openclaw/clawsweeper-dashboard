## Summary

Keep open. Current main still has no timed exec approval decision, no `approvedUntil` or trust-window contract in the approvals schema, and no `/approve 30m` parsing; the exact same-author implementation PR was closed unmerged, while related trust-window PRs remain open and need maintainer security design choice.

## What This Changes

The issue requests time-bounded exec approvals so users can approve a command pattern for a finite duration such as 30 minutes or 1 hour, with chat, CLI, storage, and expiry handling.

## Best Possible Solution

Define one canonical timed exec approval model before merging code: either a per-command-pattern timed approval integrated into the existing approval resolver, or an explicit trust-window model with clear scope, authorization, audit, expiry, and restart semantics. The chosen path should update the protocol/schema, `/approve` handling, Control UI, approvals CLI, docs, and focused regression tests, then supersede competing open PRs once maintainers pick the design.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug, but yes for source-level verification: current main can be checked by reading the approval decision union, `/approve` parser, protocol schema, CLI renderer, and UI actions, all of which lack any duration/timed approval path.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The requested timed approval direction addresses the approval-fatigue gap, but storing `approvedUntil` in the approvals file may conflict with the non-persistence goal; maintainers should choose between command-pattern timed entries and the broader in-memory trust-window proposals before implementation.

Summary: This issue is security-sensitive because it would let commands bypass repeated approval prompts for a time window.
