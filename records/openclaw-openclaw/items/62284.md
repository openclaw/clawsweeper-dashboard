## Summary

Keep open. Current main still has no timed exec approval decision, no `approvedUntil` or trust-window contract in the approval schema, and no `/approve 30m` parsing, while the related implementation proposals remain open or closed unmerged and require maintainer security design choice.

## What This Changes

The issue requests time-bounded exec approvals so users can approve command execution for a finite duration such as 30 minutes or 1 hour.

## Best Possible Solution

Define one canonical timed exec approval model with clear scope, authorization, audit, expiry, and restart semantics, then update the resolver, protocol/schema, `/approve` handling, Control UI, approvals CLI, docs, and regression tests while superseding competing PRs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for source-level verification: current main can be checked by reading the approval decision union, `/approve` parser, protocol schema, CLI renderer, gateway method list, and UI actions, all of which lack a duration or timed approval path.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The time-bounded direction addresses approval fatigue, but the proposed `approvedUntil` file field conflicts with the non-persistence goal unless maintainers first choose file-backed command-pattern approvals versus an in-memory trust-window model.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it would let commands bypass repeated approval prompts for a time window.

Concerns:

  - body: A timed approval implementation needs a single enforcement point plus explicit approver authorization, per-agent/session/command binding, expiry behavior, audit visibility, and restart semantics; otherwise it could become an unintended long-lived exec bypass.
  - confidence: 0.9

  - body: The issue proposes adding `approvedUntil` to approval store entries while also saying timed approvals should not survive gateway restart. Maintainers should decide whether in-memory state, host-local file state, or both are authoritative before accepting a patch.
  - confidence: 0.82
