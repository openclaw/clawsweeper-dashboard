## Summary

Keep open: the gateway latency idea is useful, but this patch is not merge-ready because the new timing emissions do not preserve the existing `chat.send_timing` targeting/timing contract and the supplied proof is unit-test-only.

## What This Changes

Adds server-side gateway state for chat send receipt timing and emits new `first-output` and `completed` `chat.send_timing` phases from agent event handling, with gateway mocks/tests updated.

## Best Possible Solution

Carry the originating connection id and ack timestamp with the chat run, emit the new phases through the same targeted timing contract as the existing phases, update any intended consumers, and add live gateway/WebSocket proof.

## Reproduction Assessment

Yes for the review finding: source inspection shows the existing helper computes `ackToPhaseMs` from `ackedAtMs`, while the PR callback computes it from `receivedAtMs` and uses global broadcast. I did not run a live gateway because this review is read-only.

## Solution Assessment

No. The feature direction is plausible, but the maintainable fix is to preserve the existing per-request timing delivery/baseline semantics rather than adding a parallel runtime callback with less context.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the existing timing event contract:** `src/gateway/server-runtime-subscriptions.ts:78-82`
  - body: Existing `chat.send_timing` phases are sent to the requesting operator UI connection and compute `ackToPhaseMs` from the ack timestamp. This callback only has `receivedAtMs`, so it reports `ackToPhaseMs` as the same value as `receivedToPhaseMs`, then broadcasts the new phases to all read-scope clients. That makes the new metrics inaccurate and can pollute other Control UI clients' timing streams; carry the originating conn id plus ack timestamp, or omit fields you cannot compute correctly.
  - confidence: 0.9
