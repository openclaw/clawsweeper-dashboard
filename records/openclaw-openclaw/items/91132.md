Full review comments:

- **[P1] Target the new timing phases to the originating connection:** `src/gateway/server-runtime-subscriptions.ts:82`
  - body: This callback emits `chat.send_timing` through the broad gateway broadcast path, so any READ-scope client can receive first-output/completed timings for a run it did not start. Current main's timing helper stores the requesting `connId` and uses `broadcastToConnIds`; the new async phases need to preserve that scoping instead of widening delivery.
  - confidence: 0.9
- **[P2] Preserve the ACK baseline for async timing:** `src/gateway/server-runtime-subscriptions.ts:78-79`
  - body: `ackToPhaseMs` is computed from `receivedAtMs`, making it identical to `receivedToPhaseMs` and including pre-ACK handler work. Existing `chat.send_timing` phases distinguish `ackedAtMs` from `receivedAtMs`, so consumers would get misleading latency numbers from the new phases.
  - confidence: 0.88
- **[P2] Exercise the real async timing emitter in the test:** `src/gateway/server.chat.gateway-server-chat-b.test.ts:1693-1723`
  - body: The new test calls `broadcastToConnIds` directly to add `first-output` and `completed` events, so it would pass even if the production `emitServerTiming` wiring never emitted those phases. Drive the agent event handler/runtime subscription path instead so the regression test covers the code this PR adds.
  - confidence: 0.93
