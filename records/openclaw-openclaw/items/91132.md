## Summary

Keep open: the feature direction is useful, but this PR currently changes the `chat.send_timing` delivery path and emits incorrect timing fields for the new phases, so it should not merge as-is.

## What This Changes

Adds `first-output` and `completed` phases to the gateway `chat.send_timing` event by storing per-run send timestamps and emitting timing from agent delta/final handling.

## Best Possible Solution

Keep the additive phases, but carry the original timing context needed to reuse the current origin-connection delivery semantics and compute `ackToPhaseMs` from the actual ACK timestamp.

## Reproduction Assessment

Yes for the review findings: the PR diff directly shows broad `broadcast` delivery and identical received-time bases for both timing fields, while current source and tests show targeted origin-connection timing semantics. I did not run a live gateway because this was a read-only review.

## Solution Assessment

No; the PR is a plausible slice, but the best fix is to extend the existing timing contract rather than create a parallel agent-event timing path with less context. The new phases should carry origin connection and ACK timing data, then add focused gateway regression coverage.

  - confidence: 0.9
