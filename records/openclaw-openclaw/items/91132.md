Full review comments:

- **[P1] Route new timing phases to the originating connection:** `src/gateway/server-runtime-subscriptions.ts:82`
  - body: Current `chat.send_timing` milestones are sent with `broadcastToConnIds(..., new Set([connId]))` so only the originating operator UI receives per-run timing. This new callback uses `params.broadcast("chat.send_timing", ...)`, which sends the new async phases to every operator-read client; reuse the existing targeted timing path instead.
  - confidence: 0.9
- **[P2] Preserve the ACK baseline for async phase timing:** `src/gateway/server-runtime-subscriptions.ts:78-79`
  - body: The new payload sets both `ackToPhaseMs` and `receivedToPhaseMs` from `receivedAtMs`, so `ackToPhaseMs` includes pre-ACK work and no longer means what existing timing consumers expect. Carry the original `ackedAtMs` from `ChatRunTiming` and compute the two fields separately.
  - confidence: 0.86
- **[P2] Exercise the runtime emitter instead of injecting events:** `src/gateway/server.chat.gateway-server-chat-b.test.ts:1693-1722`
  - body: This test manually calls `broadcastToConnIds` with `first-output` and `completed` payloads after `chat.send` returns, so it can pass even if the production delta/final handlers never emit the new phases or emit them to the wrong recipients. Drive the agent event handler path that produces the real frames.
  - confidence: 0.82
