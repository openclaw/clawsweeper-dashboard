## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Target async timing to the originating UI connection:** `src/gateway/server-runtime-subscriptions.ts:82`
  - body: This callback uses `params.broadcast`, so each new per-run timing frame can go to every READ-scope operator client. Current `chat.send_timing` stores the originating connection and uses `broadcastToConnIds`; the async phases need the same targeting to avoid misdelivered timing data.
  - confidence: 0.94
- **[P2] Preserve the ACK timing baseline:** `src/gateway/server-runtime-subscriptions.ts:78-79`
  - body: Both `ackToPhaseMs` and `receivedToPhaseMs` are computed from `receivedAtMs`, making the two fields identical. Current timing events compute `ackToPhaseMs` from the ACK timestamp, so consumers lose the existing ACK-baseline meaning for these phases.
  - confidence: 0.9
- **[P1] Extend the existing first-assistant timing contract:** `src/gateway/server-chat.ts:641`
  - body: Current main already emits and parses `first-assistant-event` for first assistant output timing. Adding a separate `first-output` phase creates parallel semantics the current UI ignores; rebase onto the current timing path or use a maintainer-approved durable metadata contract instead.
  - confidence: 0.88
- **[P2] Drive the production emitter in the latency test:** `src/gateway/server.chat.gateway-server-chat-b.test.ts:1693-1694`
  - body: The new test manually calls `broadcastToConnIds` with fabricated `first-output` and `completed` payloads after `chat.send`. It would pass even if the production runtime callback never emitted those frames or sent them to the wrong recipients, so it should drive the agent event/subscription path instead.
  - confidence: 0.86
