## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Route late timing events to the originating connection:** `src/gateway/server-runtime-subscriptions.ts:82`
  - body: The new late timing callback sends `chat.send_timing` with `params.broadcast`, while existing chat-send timings are targeted to the requester with `broadcastToConnIds(new Set([connId]))`. In a gateway with multiple READ-scope clients, one run's `first-output`/`completed` timing can now be delivered to unrelated clients; reuse the existing per-run timing record and targeted sender.
  - confidence: 0.92
- **[P2] Compute ackToPhaseMs from the ACK timestamp:** `src/gateway/server-runtime-subscriptions.ts:78-79`
  - body: This uses the chat receive timestamp for `ackToPhaseMs`, making it equal to `receivedToPhaseMs` and including pre-ACK work such as session load and attachment preparation. Existing timing semantics compute `ackToPhaseMs` from the ACK timestamp, so late phases should carry or reuse `ackedAtMs`.
  - confidence: 0.9
- **[P2] Exercise the real emitter instead of injecting expected events:** `src/gateway/server.chat.gateway-server-chat-b.test.ts:1693-1722`
  - body: The added test manually calls `broadcastToConnIds` with the expected `first-output` and `completed` payloads after `chat.send`, so it would pass even if the production agent-event path never emitted the new phases. Drive the agent delta/final handler or subscription callback that the PR changes.
  - confidence: 0.95
