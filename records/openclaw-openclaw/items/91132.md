## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Route async timing to the originating connection:** `src/gateway/server-runtime-subscriptions.ts:82`
  - body: The existing `chat.send_timing` helper targets the originating Control UI connection with `broadcastToConnIds`, but this new async path calls `params.broadcast`, so every operator read client can receive timing for a send it did not originate. Thread the current `ChatRunTiming` owner/connId through these phases and target the original connection.
  - confidence: 0.91
- **[P2] Preserve the ack baseline for async phases:** `src/gateway/server-runtime-subscriptions.ts:78-79`
  - body: `ackToPhaseMs` is calculated from `receivedAtMs`, making it identical to `receivedToPhaseMs` for `first-output` and `completed`. Current timing consumers distinguish received-to-phase from ack-to-phase, so this path needs to store and use `ackedAtMs` as the existing `ChatRunTiming` path does.
  - confidence: 0.88
- **[P2] Exercise the runtime emitter instead of faking events:** `src/gateway/server.chat.gateway-server-chat-b.test.ts:1693-1723`
  - body: The added test manually calls `broadcastToConnIds` to create `first-output` and `completed` payloads after `chat.send`, so it would still pass if the production `emitServerTiming` hook never emitted either phase. Drive the agent delta/final handler or `createAgentEventHandler` path and assert the real emission calls.
  - confidence: 0.87
