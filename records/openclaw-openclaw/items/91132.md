## What This Changes

Adds `first-output` and `completed` phases to gateway `chat.send_timing` by tracking receive timestamps and first-output state, then emitting late timing from agent delta/final handling.

## Best Possible Solution

Carry the initiating connection id and ACK timestamp in run state, emit the new phases through the existing targeted timing contract, add focused coverage, and include redacted gateway/WebSocket proof.

## Reproduction Assessment

Yes. Source inspection of the PR head shows `params.broadcast` for late timing events and `ackToPhaseMs` derived from `receivedAtMs`, while current main uses targeted `broadcastToConnIds` and a distinct ACK timestamp.

## Solution Assessment

No. The maintainable fix is to extend the existing timing contract and state carried from `chat.send`, not add a parallel callback that loses recipient and ACK context.

Full review comments:

- **[P1] Target late timing events to the initiating UI:** `src/gateway/server-runtime-subscriptions.ts:82`
  - body: The new callback sends `first-output` and `completed` timing with `params.broadcast(...)`. Existing `chat.send_timing` milestones validate the Control UI client and call `broadcastToConnIds` for the initiating connection; because this event is READ-scoped, this can send one UI's per-send timing to every eligible operator WebSocket. Carry the origin conn id and use the targeted path.
  - confidence: 0.94
- **[P2] Preserve the ACK timing base for late phases:** `src/gateway/server-runtime-subscriptions.ts:78-79`
  - body: Both new fields are calculated from `receivedAtMs`, so `ackToPhaseMs` is identical to `receivedToPhaseMs`. Existing timing consumers treat `ackToPhaseMs` as time since the ACK; carry `chatSendAckedAtMs` or reuse the existing timing helper so late phases keep the same field contract.
  - confidence: 0.93
