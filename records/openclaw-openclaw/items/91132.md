## Best Possible Solution

Emit the new phases through the same operator-only, origin-connection timing path used by existing phases, preserve received/ACK/dispatch timing bases, update focused consumers/tests as needed, then add redacted real gateway/WebSocket proof.

## Reproduction Assessment

## Solution Assessment

No. The better fix is to extend the existing operator-targeted timing path or carry equivalent request-scoped state into the agent-event path, then prove it with focused gateway tests and real WebSocket output.

  - body: Existing `chat.send_timing` phases are emitted only to the Control UI connection that initiated `chat.send`; this callback uses the broad `broadcast` path, so a first-output/completed event for one send can be delivered to unrelated operator clients.   - confidence: 0.93
