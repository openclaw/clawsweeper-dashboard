## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Reuse the targeted chat timing path for async phases:** `src/gateway/server-runtime-subscriptions.ts:63`
  - body: This new callback sends `chat.send_timing` through `params.broadcast`, which reaches every READ-scope client. Existing chat-send timing records the originating `connId` in `ChatRunTiming` and uses `broadcastToConnIds`; the async phases should extend that path so unrelated clients do not receive another run's timing events.
  - confidence: 0.92
- **[P2] Compute ACK timing from the ACK timestamp:** `src/gateway/server-runtime-subscriptions.ts:75`
  - body: The payload sets `ackToPhaseMs` from `nowMs - receivedAtMs`, making it identical to `receivedToPhaseMs` and including pre-ACK work. Current timing code keeps `ackedAtMs` and `receivedAtMs` separate, so this field would be misleading for the new phases.
  - confidence: 0.9
- **[P2] Exercise the runtime emitter instead of injecting timing calls:** `src/gateway/server.chat.gateway-server-chat-b.test.ts:1668`
  - body: The new test manually calls `broadcastToConnIds` with the expected `first-output` and `completed` payloads, so it would pass even if `createAgentEventHandler` or `startGatewayEventSubscriptions` never emitted those events. Drive the actual handler path and assert the mock calls produced by production code.
  - confidence: 0.88
