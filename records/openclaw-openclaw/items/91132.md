- **[P1] Route async timing to the originating connection:** `src/gateway/server-runtime-subscriptions.ts:80`
  - body: This callback uses `params.broadcast`, so every READ-scope operator client can receive timing frames for a run they did not start. The existing timing path stores the originating `connId` and sends via `broadcastToConnIds`; route any new async phase through that same targeted path.
  - confidence: 0.93
- **[P2] Preserve the ACK baseline for async phases:** `src/gateway/server-runtime-subscriptions.ts:76-77`
  - body: `ackToPhaseMs` and `receivedToPhaseMs` are both computed from `receivedAtMs`, making the fields identical. Existing consumers expect `ackToPhaseMs` to subtract the ACK timestamp, so this would report misleading server timing data.
  - confidence: 0.9
- **[P1] Extend the shipped first-assistant timing path:** `src/gateway/server-chat.ts:625`
  - body: Current main already emits a targeted `first-assistant-event` timing event and the UI parses that phase. Adding a new `first-output` map and phase creates a parallel concept the current UI ignores, so the branch should rebase and extend the existing `ChatRunTiming` path instead.
  - confidence: 0.88
- **[P2] Exercise the runtime emitter instead of injecting events:** `src/gateway/server.chat.gateway-server-chat-b.test.ts:1665`
  - body: The new regression test manually calls `broadcastToConnIds` with fabricated `first-output` and `completed` payloads, so it can pass even if the production `emitServerTiming` wiring never emits those phases. Drive the agent event handler or gateway subscription path and assert the real emitted frames.
