## What This Changes

The PR adds `first-output` and `completed` phases to gateway `chat.send_timing` by storing per-run receive timestamps and emitting timing from agent delta/final handling.

## Best Possible Solution

Carry the initiating connection id and ACK timestamp forward with the run state, emit the new phases through the existing targeted timing contract, add focused multi-client/timing-base coverage, and then provide redacted real gateway/WebSocket proof.

## Reproduction Assessment

Yes. Source inspection of the PR head shows `params.broadcast` for late timing events and `ackToPhaseMs` derived from `receivedAtMs`, while current main uses targeted `broadcastToConnIds` and distinct ACK timing.

## Solution Assessment

No. The best fix is to extend the existing timing contract and state carried from `chat.send`, not add a parallel callback that loses recipient and ACK context.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Target late timing events to the initiating UI:** `src/gateway/server-runtime-subscriptions.ts:82`
  - body: This callback runs after the original `chat.send` handler, but it sends `chat.send_timing` with `params.broadcast(...)`. Existing timing milestones go through `emitOperatorChatSendServerTiming`, which verifies the Control UI client and calls `broadcastToConnIds` for only the initiating connection; since `chat.send_timing` is READ-scoped, this can leak one UI's per-send timing events to other operator-read WebSockets. Store the initiating connection with the run and use the targeted delivery path for these late phases too.
  - confidence: 0.94
- **[P2] Preserve the ACK timing base for late phases:** `src/gateway/server-runtime-subscriptions.ts:78-79`
  - body: The new payload sets `ackToPhaseMs` from `nowMs - receivedAtMs`, so `ackToPhaseMs` and `receivedToPhaseMs` are identical for `first-output` and `completed`. Existing `chat.send_timing` events define `ackToPhaseMs` relative to the post-ACK timestamp, and UI/diagnostic consumers use that field name as the ACK-relative duration. Carry `chatSendAckedAtMs` forward or reuse the existing timing helper so late phases keep the same contract.
  - confidence: 0.93
