## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Route async timing through ChatRunTiming:** `src/gateway/server-runtime-subscriptions.ts:80`
  - body: This callback broadcasts `chat.send_timing` with `params.broadcast`, which sends to every client with the event's READ/WRITE scope. Current main targets timing to the originating `connId` via `ChatRunTiming`/`broadcastToConnIds`, so these new phases can be delivered to unrelated operator clients and should reuse that path.
  - confidence: 0.93
- **[P2] Compute ack latency from the ACK timestamp:** `src/gateway/server-runtime-subscriptions.ts:76-77`
  - body: Here `ackToPhaseMs` is calculated from `receivedAtMs`, making it identical to `receivedToPhaseMs` and including pre-ACK work. Carry `ackedAtMs` from the existing timing entry so consumers get the same semantics as the existing phases.
  - confidence: 0.9
- **[P2] Drive the real timing emitter in the test:** `src/gateway/server.chat.gateway-server-chat-b.test.ts:1664`
  - body: The test fabricates `first-output` and `completed` by calling `broadcastToConnIds` itself after `chat.send`; it would pass even if the runtime emitter never fires or broadcasts to the wrong audience. Drive the agent-event/final path and assert the real emitted calls.
  - confidence: 0.86
