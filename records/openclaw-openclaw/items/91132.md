Full review comments:

- **[P1] Route new timing phases to the originating connection:** `src/gateway/server-runtime-subscriptions.ts:79`
  - body: Existing `chat.send_timing` emissions are targeted back to the sending Control UI connection with `broadcastToConnIds`; this callback uses broad `broadcast`, so any READ-scope client can receive timing for a chat send it did not start. Thread the existing run timing connection metadata through this path and assert targeted delivery.
  - confidence: 0.92
- **[P2] Preserve the ACK baseline for async phase timing:** `src/gateway/server-runtime-subscriptions.ts:75-76`
  - body: This makes `ackToPhaseMs` equal `receivedToPhaseMs`, but current timing code keeps ACK-relative timing based on `ackedAtMs`. Consumers would see pre-ACK session load and attachment work counted as post-ACK latency, so store/reuse the ACK timestamp or omit the field until it can be computed correctly.
  - confidence: 0.9
- **[P2] Exercise the runtime timing emitter in the test:** `src/gateway/server.chat.gateway-server-chat-b.test.ts:1667-1687`
  - body: The new test manually calls `broadcastToConnIds` with the expected `first-output` and `completed` payloads after `chat.send`, so it would pass even if the production agent delta/final handler never emitted those events. Drive the `createAgentEventHandler` delta/final path or a live-style gateway path instead.
  - confidence: 0.86
