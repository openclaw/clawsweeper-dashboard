## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Route timing phases to the originating connection:** `src/gateway/server-runtime-subscriptions.ts:82`
  - body: `chat.send_timing` is currently emitted to the requester with `broadcastToConnIds`, but this new async callback uses `params.broadcast`, so any read-scope client can receive timing for a run it did not send. Carry the request connection id or reuse the existing targeted timing helper.
  - confidence: 0.9
- **[P2] Preserve the ack baseline for phase timings:** `src/gateway/server-runtime-subscriptions.ts:76-77`
  - body: The new callback computes `ackToPhaseMs` from `receivedAtMs`, making it equal to `receivedToPhaseMs`. Existing timing consumers treat those fields as different baselines, so this should store/reuse the ack timestamp instead of deriving both from receipt time.
  - confidence: 0.86
- **[P2] Exercise the runtime emitter instead of injecting events:** `src/gateway/server.chat.gateway-server-chat-b.test.ts:1684`
  - body: The added test manually calls `broadcastToConnIds` with the expected `first-output` and `completed` payloads, so it would pass even if the new handler path never emitted those phases. Drive an agent delta/final event through the real handler and assert the actual broadcast.
  - confidence: 0.88
