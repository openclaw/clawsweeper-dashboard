## Summary

Keep open: this is a useful gateway telemetry feature, but the current patch is not merge-ready because the new timing callback loses the existing sender targeting and true ack baseline, and the added proof is still unit/mock output rather than a real gateway/WebSocket run.

## What This Changes

The PR adds gateway chat-run timing state and two new `chat.send_timing` phases, `first-output` and `completed`, with updated gateway context plumbing and tests.

## Best Possible Solution

Carry the originating connection id and ack timestamp with the chat run, emit the new phases through the same targeted timing contract as the existing phases, update the intended consumer or document server-only scope, and add redacted live gateway/WebSocket proof.

## Reproduction Assessment

Yes for the review findings: source inspection shows the PR callback globally broadcasts the new phases and computes `ackToPhaseMs` from `receivedAtMs`, while current main targets the requester and uses `ackedAtMs`. I did not run a live gateway because this review is read-only.

## Solution Assessment

No. The feature direction is plausible, but the best fix is to preserve the existing per-request timing delivery and ack-baseline semantics rather than adding a parallel callback with less context.
