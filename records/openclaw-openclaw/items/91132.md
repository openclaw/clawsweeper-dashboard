Overall correctness: patch is incorrect

Full review comments:

- **[P1] Target timing events to the originating operator connection:** `src/gateway/server-runtime-subscriptions.ts:82`
  - body: The new async phases use `params.broadcast`, so every READ-scope gateway client can receive timing for a run it did not start. The existing timing path records the sender connection and uses `broadcastToConnIds`; extend that path for these phases instead.
  - confidence: 0.94
- **[P2] Compute ackToPhaseMs from the ACK timestamp:** `src/gateway/server-runtime-subscriptions.ts:78`
  - body: `ackToPhaseMs` is currently computed from `receivedAtMs`, making it identical to `receivedToPhaseMs` and overstating ACK-relative latency. Store and use the ACK timestamp, as current main’s `ChatRunTiming` path does.
  - confidence: 0.9
- **[P2] Exercise the runtime emitter in the timing test:** `src/gateway/server.chat.gateway-server-chat-b.test.ts:1693-1712`
  - body: The added test manually calls `broadcastToConnIds` with the expected `first-output` and `completed` payloads after `chat.send`, so it does not prove the runtime handler emits those phases. Drive the agent event path or call the handler under test instead.
  - confidence: 0.91
