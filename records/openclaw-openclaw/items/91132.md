## Summary

Keep open: the latency phase idea is useful, but the patch currently emits the new async phases through a broader broadcast path, reports the ack metric from the wrong baseline, and still only supplies unit-test/source proof rather than a real gateway/WebSocket run.

## What This Changes

The PR adds `first-output` and `completed` phases to `chat.send_timing` by storing per-run receipt state in gateway chat state/context and emitting timing events from agent delta/final handling.

## Best Possible Solution

Preserve existing `chat.send_timing` delivery and metric semantics by carrying the sender connection and ack timestamp into async run state, emitting targeted phases, and adding redacted real WebSocket proof before merge.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the new async timing path uses untargeted `broadcast` and computes both timing fields from `receivedAtMs`. I did not run a live gateway/WebSocket scenario, and the PR body does not provide one.

## Solution Assessment

No; adding async phases is a plausible server-side step, but this implementation bypasses the established targeted timing helper and loses the ack baseline. The maintainable fix is to extend the existing timing contract instead of adding a parallel emission path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Route new timing phases to the originating connection:** `src/gateway/server-runtime-subscriptions.ts:82`
  - body: Existing `chat.send_timing` phases only go to the originating Control UI connection via `broadcastToConnIds`; this new async path calls `params.broadcast`, so every scoped operator client can receive another sender's `first-output` and `completed` timing events. Carry the requester connection/client into the run state and use the same targeted delivery semantics as the existing timing helper.
  - confidence: 0.93
- **[P2] Preserve the ack baseline for async timing phases:** `src/gateway/server-runtime-subscriptions.ts:78-79`
  - body: The new payload computes `ackToPhaseMs` from `receivedAtMs`, making it identical to `receivedToPhaseMs`. Existing timing events define `ackToPhaseMs` from the ack timestamp, so consumers will get misleading phase durations unless the ack timestamp is stored and propagated, or the ack-based field is omitted for phases that cannot compute it.
  - confidence: 0.9
- **[P2] Exercise the actual runtime emission path:** `src/gateway/server.chat.gateway-server-chat-b.test.ts:1693-1722`
  - body: This test manually calls `broadcastToConnIds` to create the `first-output` and `completed` events, so it would still pass if `createAgentEventHandler` never emitted them or emitted them through the wrong delivery path. Drive an agent delta/final through the new callback path and assert the actual gateway broadcast behavior.
  - confidence: 0.86
