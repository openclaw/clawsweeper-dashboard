## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Target late timing events to the initiating UI:** `src/gateway/server-runtime-subscriptions.ts:82`
  - body: The existing `chat.send_timing` path sends server milestones only to the Control UI connection that initiated `chat.send`. This new late callback uses `params.broadcast`, so `first-output` and `completed` frames fan out to every operator read/write WebSocket instead of the originating UI. Carry the original connection into this path and send with `broadcastToConnIds`.
  - confidence: 0.93
- **[P2] Preserve the ACK timing base for late phases:** `src/gateway/server-runtime-subscriptions.ts:78-79`
  - body: The existing event contract reports `ackToPhaseMs` from the ACK timestamp and `receivedToPhaseMs` from request receipt. Here both fields are computed from `receivedAtMs`, so clients get inflated ACK-to-phase values equal to total receive-to-phase latency. Store/pass the ACK timestamp or reuse the same timing helper.
  - confidence: 0.9
