## Summary

Keep this open. Current main has shipped useful mitigations for adjacent startup stalls and stale catalog reloads, but the core cold `models.list` path still awaits the full gateway model catalog before resolving the default/configured view, so slow provider catalog/auth work can still block the picker RPC.

## What This Changes

The issue requests that Gateway startup and `models.list` stay fast with many configured providers by avoiding slow startup-time provider validation/catalog work.

## Best Possible Solution

Keep the full catalog available for explicit discovery, but make normal picker-sized `models.list` responses come from configured rows, a warm/stale catalog, or a short bounded refresh; expensive provider catalog/auth work should run in the background or only for `view: "all"`.

## Reproduction Assessment

Yes for the blocking shape: a delayed or hanging `loadGatewayModelCatalog()` still blocks `models.list` before view resolution. The exact reported 30-85s timings require the reporter's provider/auth setup or a test hook that simulates slow provider catalog work.

## Solution Assessment

No, current main is only a partial solution. The safer direction is a bounded fast path for default/configured model views, while preserving the existing full-catalog behavior for explicit browse or diagnostics calls.
