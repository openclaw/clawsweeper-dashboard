## Summary

Keep open: current main has structured model-fallback diagnostics and trajectory capture, but no documented user-facing hook or channel notification for fallback events.

## What This Changes

Request a notification or hook event when OpenClaw falls back from a primary model to another configured model.

## Best Possible Solution

Define a deliberate public fallback event or hook contract first, then decide separately whether any channel-visible notification should be opt-in UX.

## Reproduction Assessment

Not applicable; this is a feature request, not a broken existing behavior report. Source and docs show fallback telemetry exists, but the requested user-facing notification or hook surface is not currently documented or wired.

## Solution Assessment

Unclear as proposed because the issue offers two viable product directions. A documented event/hook is the narrower maintainable direction than hardcoded cross-channel messages, unless maintainers choose an opt-in notification policy.
