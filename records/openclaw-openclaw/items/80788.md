## Summary

Keep open: current main and the latest release still use the old Discord REST response parser, while this PR provides a focused code-path fix with sufficient terminal proof and no blocking review findings.

## What This Changes

The PR changes the Discord REST client to decode raw gzip-looking response bodies before JSON coercion and adds read/search response-shape guards with focused tests.

## Best Possible Solution

Land the focused Discord REST read/search fix if maintainers accept the code-path proof, and keep broader probe/provider decompression policy tracked under https://github.com/openclaw/openclaw/issues/79752.

## Reproduction Assessment

Yes at source/proof level, but not as a live-credential repro: current main still uses response.text(), the linked report shows raw gzip bytes reaching read/search, and a local Response probe confirms the same raw-byte failure shape. I did not run a live Discord gateway smoke in this read-only review.

## Solution Assessment

Yes. Decoding at the Discord RequestClient boundary is the narrowest maintainable fix for one-shot Discord REST JSON responses, while read/search guards prevent binary strings or wrong shapes from leaking to callers.
