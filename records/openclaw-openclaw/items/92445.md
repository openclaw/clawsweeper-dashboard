## Summary

The reported Feishu audit rejection is credible and the supplied live run is useful, but this PR should remain open because its generic recursive card rewrite can corrupt action/control fields; green CI does not exercise that invariant, and the live proof covers only one plain-text service-number delivery.

## What This Changes

The PR adds Feishu-specific redaction of email addresses and Chinese phone-number formats across post messages, interactive cards, message edits, and streaming card updates, plus regression tests.

## Best Possible Solution

Keep redaction at the Feishu outbound owner boundary, but traverse only schema-known rendered text fields while preserving every control field byte-for-byte, then provide redacted diagnostic before/after evidence for the failing text path and one representative card path.

## Reproduction Assessment

Yes for the patch-introduced regression: a nested card control string matching an email or phone pattern is deterministically rewritten before serialization. The underlying Feishu 230028 rejection has credible after-fix live output, but no captured failing current-main request or payload.

## Solution Assessment

No. Redacting at the Feishu outbound boundary is appropriate, but recursively rewriting arbitrary card strings is broader than necessary and violates control-data integrity; schema-aware rendered-text traversal is safer.
