## Summary

This PR addresses a credible Feishu audit rejection at the correct channel boundary, but its recursive card traversal also rewrites non-display control fields and can break interactive actions; it additionally lacks required after-fix proof from a real Feishu workspace.

## What This Changes

The PR adds outbound Feishu redaction for email addresses and several Chinese phone-number formats across messages, cards, edits, and streaming updates.

## Best Possible Solution

Keep redaction inside the Feishu plugin, but traverse only rendered textual fields while preserving all card control data byte-for-byte, then provide redacted real-workspace evidence of the original audit rejection and successful delivery after the fix.

## Reproduction Assessment

Yes for the introduced regression: any nested card control string matching a redaction pattern is deterministically changed before serialization. The reported Feishu audit rejection itself has not been reproduced in a real workspace.

## Solution Assessment

No; redaction at the Feishu outbound boundary is the right ownership direction, but recursively rewriting arbitrary card values is broader than the visible-content failure and violates the distinction between display text and control data.
