## Summary

Keep open: current main still has a real descriptor-boundary inconsistency where the raw exported sub-CLI catalog contains the private `qa` command while the public getter APIs filter it unless the private QA env gate is enabled. The blast radius is narrower than the title suggests because the only production raw-export caller I found is `src/cli/argv.ts`, while root help and completion already use gated getters.

## What This Changes

The issue asks OpenClaw to stop exposing the private `qa` sub-CLI descriptor through `SUB_CLI_DESCRIPTORS` when `OPENCLAW_ENABLE_PRIVATE_QA_CLI` is not enabled.

## Best Possible Solution

Internalize or remove the unfiltered descriptor export, route argv/help classification through the gated descriptor APIs, and add regression coverage for private QA disabled and enabled states.

## Reproduction Assessment

Yes. Source inspection on current main shows the raw `SUB_CLI_DESCRIPTORS` export contains `qa` while the gated getter APIs filter `qa` when the private QA env flag is unset; I did not execute the CLI in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is to make the exported/callable descriptor surface share the same private QA gate, preferably by removing the raw export and updating `argv.ts` to use gated descriptor helpers.
