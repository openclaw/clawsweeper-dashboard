## Summary

Keep open: current main and v2026.6.6 still provide file-backed memory search plus thresholded promotion into `MEMORY.md`, not the requested typed write resolver with source-required classification, dedupe/merge, explicit conflicts, and revision history. The real-world implementation comment is useful design input, especially around confidence scoring, alias normalization, conflict tags, and append-only provenance.

## What This Changes

The issue requests a durable memory write pipeline that classifies candidate memories, attaches sources, deduplicates or merges before write, represents conflicts, and preserves revision history.

## Best Possible Solution

Settle one coordinated Memory MVP v1 contract, then implement the resolver inside the active memory plugin with SQLite-backed canonical records, provenance-preserving revisions, conservative dedupe/merge, and explicit conflict states.

## Reproduction Assessment

Not applicable: this is a new durable-memory write-path and product-contract request, not a bug report against existing documented behavior. Source and release inspection show the requested resolver is absent rather than broken.

## Solution Assessment

No, not as a standalone fix PR yet. The memory plugin is the right likely boundary, but the best solution requires aligning the open schema, provenance, review-flow, and ingestion-surface issues first.
