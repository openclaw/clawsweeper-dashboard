## Summary

Keep open: current main has adjacent short-term recall ranking, promotion safeguards, source line markers, and memory search indexing, but it still lacks the requested durable write resolver with typed classification, near-duplicate merge/update decisions, explicit conflict handling, revision history, and entity normalization. The related comments and open Memory MVP trackers are useful design context, not a terminal implementation or duplicate close path.

## What This Changes

The issue requests a conservative memory write pipeline that classifies candidate memories, attaches sources, deduplicates or merges before durable write, and routes conflicts for review.

## Best Possible Solution

Define the v1 durable memory record and provenance contract first, then implement a memory-core write resolver that gates low-confidence/source-less candidates, merges or updates with revision history, and surfaces explicit conflicts for operator review.

## Reproduction Assessment

Not applicable. This is an MVP feature/product request rather than a broken documented behavior, and source/docs inspection shows adjacent promotion/indexing pieces but no existing write-resolver contract to reproduce.

## Solution Assessment

Unclear as a standalone implementation. The requested resolver is a plausible end state, but the maintainable path is to sequence it with the open schema, provenance, review-flow, CLI-surface, and chunking/entity-extraction trackers rather than landing a narrow one-off patch.
