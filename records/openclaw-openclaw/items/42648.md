## Summary

Keep open: current main has partial memory promotion, retrieval indexing, and memory-wiki provenance surfaces, and the w1tv production comment adds useful design guidance, but the requested conservative resolver pipeline with classification, durable dedupe/merge, conflict handling, revision history, and mandatory source gating is still not implemented. The remaining work overlaps the open Memory MVP schema, provenance, review-flow, and CLI/operator-surface issues.

## What This Changes

The issue requests a v1 candidate-memory write pipeline that classifies candidates, attaches sources, detects duplicates or conflicts, resolves create/merge/review outcomes, and preserves provenance and revisions.

## Best Possible Solution

Define the v1 durable-memory write contract in the memory owner layer, then implement resolver-backed create/merge/conflict paths with provenance, revisions, review routing, docs, and tests in sequence with the open schema, provenance, and operator-flow trackers.

## Solution Assessment

Unclear as a final implementation, but keeping this as a product/design tracker is the right outcome. The maintainable path is to settle the durable record, provenance, and review semantics before exposing write/merge/conflict behavior as default OpenClaw memory infrastructure.
