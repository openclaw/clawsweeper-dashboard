## Summary

Keep open: current main has useful memory pieces, but it still does not provide the requested typed durable write resolver with source-required candidate classification, merge/update decisions, conflict state, and revision history. The real-world implementation comment is useful design input, and the related Memory MVP issues remain open rather than superseding this work.

## What This Changes

Request to add a conservative durable memory write pipeline that classifies candidates, requires source attribution, deduplicates or merges, preserves revisions, and routes conflicts to review.

## Best Possible Solution

Define the Memory MVP contract first, then implement one active memory write path through the memory plugin/SDK boundary with SQLite canonical records, source/provenance, resolver decisions, conflict review state, and revision history.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a Memory MVP feature/design request. Source inspection does show current main lacks the requested typed write resolver and canonical revision/conflict schema.

## Solution Assessment

No, the current scattered mechanisms are not the best final solution: memory-core, memory-lancedb, and memory-wiki each cover part of the space, but the maintainable path is a single owner-approved memory write contract instead of layering more ad hoc write behavior.
