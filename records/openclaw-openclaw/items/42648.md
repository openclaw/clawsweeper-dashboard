## Summary

Keep open: current main still has Markdown-backed memory search, staged short-term promotion, and recall/read tooling, but not the requested durable write resolver with classification, source-gated writes, near-duplicate/conflict resolution, revisions, and provenance-preserving merges. The w1tv production-experience comment is useful design input, and the open Memory MVP companion issues plus the separate Memory v2 PR show active design/implementation interest rather than an already-implemented core feature.

## What This Changes

The issue requests a durable memory write pipeline that classifies candidates, attaches sources, deduplicates or merges related records, preserves revision/provenance history, and routes conflicts or low-confidence writes for review.

## Best Possible Solution

Settle one Memory MVP contract for durable records, provenance, resolver decisions, conflicts, and review/edit/forget semantics, then implement it across memory-host-sdk and memory-core with docs and focused regression coverage.

## Reproduction Assessment

Not applicable as a bug reproduction. Source, docs, tests, current-main tool surface, and related-issue checks give a high-confidence state check that the requested durable write resolver does not exist yet.

## Solution Assessment

Unclear as a standalone implementation. The direction is reasonable, but the best solution needs maintainer agreement across durable schema, provenance, resolver policy, review/edit/forget flows, and how any open Memory v2 work should be sequenced.
