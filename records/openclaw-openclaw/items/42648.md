## Summary

Keep open: current main and the latest release still provide Markdown-backed memory search/read plus thresholded promotion into `MEMORY.md`, not the requested typed write resolver with classification, source-required dedupe/merge, conflict states, and revision history. The related Memory MVP issues and the real-world implementation comment are useful design input, but this remains an unsettled memory product/storage contract.

## What This Changes

Define a Memory MVP write pipeline that classifies candidate memories, attaches source/provenance, dedupes or merges near-duplicates, and routes conflicts before durable storage.

## Best Possible Solution

Settle the v1 memory record, provenance, conflict/review, and ingestion contracts across the open Memory MVP issues, then implement the resolver in the active memory plugin with SQLite-backed canonical records.

## Reproduction Assessment

Not applicable; this is a requested new memory write capability, not a broken existing behavior with a current-main repro. Source and docs show the requested resolver is absent from the current Markdown promotion model.

## Solution Assessment

Unclear as a standalone implementation; a conservative resolver is the right shape, but it depends on schema, provenance, conflict policy, and review-flow decisions that remain open.
