## Summary

Keep open: current main has thresholded short-term promotion, source markers, and marker-based duplicate avoidance, but it still does not implement the requested write resolver with typed classification, durable dedupe/merge, conflict-for-review handling, or revision history. The ClawHub smart-memory report and the later production-experience comment are useful design input, not evidence that the default OpenClaw memory pipeline is done.

## What This Changes

The issue requests a conservative Memory MVP write pipeline that classifies candidate memories, attaches source provenance, deduplicates or merges before durable writes, and routes conflicts for review.

## Best Possible Solution

Define the durable record, provenance, status, and revision model first, then implement one resolver-owned write path that gates all durable promotions and direct writes through classification, source checks, duplicate/merge resolution, and explicit conflict review.

## Reproduction Assessment

Not applicable. This is a Memory MVP feature/product request rather than a broken existing behavior, and current source/docs inspection gives a high-confidence state check that the requested resolver is absent.

## Solution Assessment

No, current main is not the best solution for the requested issue because append plus marker dedupe only handles one promotion path. The maintainable direction is to sequence this with the open schema, provenance, and review-flow work instead of adding another narrow append helper.
