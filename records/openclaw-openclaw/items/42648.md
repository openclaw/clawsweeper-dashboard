## Summary

Keep open: current main still exposes Markdown-backed memory search/retrieval plus thresholded short-term promotion into `MEMORY.md`, not a typed durable write resolver with source-required classification, duplicate/conflict detection, merge decisions, explicit conflict handling, and revision history. The related open Memory MVP issues and the real-world implementation comment are useful design input, but this remains a product/schema contract rather than a cleanup close.

## What This Changes

The issue requests a conservative Memory MVP write pipeline that classifies candidates, attaches provenance, deduplicates or merges near matches, surfaces conflicts, and preserves revision history before durable writes.

## Best Possible Solution

Settle the v1 memory record, provenance, merge/conflict, and review contracts across the linked Memory MVP issues, then implement the resolver in the active memory plugin with SQLite-backed canonical records and explicit review/conflict states.
