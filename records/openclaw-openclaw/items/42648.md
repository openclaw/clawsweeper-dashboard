## Summary

Keep open: current main still has Markdown-backed memory promotion/indexing and a Memory Wiki companion schema, but it does not implement the requested durable write resolver with classification, source-aware dedupe/merge, explicit conflicts, and revision-preserving write decisions; the related open Memory MVP issues show this is still active product and architecture work.

## What This Changes

The issue requests a Memory MVP write pipeline that classifies candidate memories, attaches source provenance, deduplicates or detects conflicts, resolves create/merge/review outcomes, and preserves revision history before durable writes.

## Best Possible Solution

Define one Memory MVP contract for durable records, resolver decisions, provenance, conflicts, revision history, and review/edit/forget semantics, then implement it in the memory plugin/host SDK with focused docs and regression coverage.

Not applicable as a bug reproduction; this is a feature and architecture request. The high-confidence check is source/docs inspection showing current main still has Markdown-backed promotion/indexing and no durable resolver write lifecycle.
