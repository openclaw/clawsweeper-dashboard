## Label Justifications

- `P2`: This is a normal-priority Telegram media ingestion bug with a clear source path and limited blast radius to failed inbound media resolution.
- `impact:message-loss`: A failed Telegram video download can leave the agent without the actual media attachment while the message continues processing.
- `impact:session-state`: The session transcript can record misleading media placeholder metadata that does not correspond to an accessible attachment.

## Summary

Keep open: current main still has the Telegram `getFile()` failure path that returns `null`, skips the user-visible failed-download warning, and can render unavailable media as placeholder metadata. A focused fix PR was closed unmerged, and a broader Telegram batch PR remains open, so this issue should stay open until an equivalent fix lands.

## What This Changes

The issue asks Telegram inbound media handling to surface `getFile()` or download failures and avoid sending a misleading placeholder-only video message when no attachment was resolved.

## Best Possible Solution

Land or split a focused Telegram plugin fix that carries media-resolution failure state to the handler, sends an accurate user warning, and avoids representing unavailable media as an attached video.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `ctx.getFile()` rejection returns `null`, `resolveMedia()` propagates `null`, the warning path only handles thrown errors, and body rendering can emit placeholder metadata with empty `allMedia`.

## Solution Assessment

Yes for keeping this open. The narrow maintainable fix belongs in the Telegram plugin media resolver, handler warning path, and inbound body construction; the linked PRs are candidates, not merged fixes.
