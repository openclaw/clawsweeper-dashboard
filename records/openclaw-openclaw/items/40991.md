- `impact:message-loss`: The reported Telegram video payload can reach the session without the actual media attachment.
- `impact:session-state`: The session transcript can imply media context exists when no attachment path is available, creating agent context drift.

## Summary

Keep open: current main and v2026.6.10 still have a source-proven Telegram path where `getFile()` failures return `null`, no failed-download warning is sent for that null result, and body rendering can still produce media placeholder context without an attachment. Related PRs are closed unmerged or only cover a partial local Bot API materialization cause.

## What This Changes

The issue asks for Telegram inbound video download failures to avoid entering sessions as placeholder-only media without an attachment or a clear failure state.

## Best Possible Solution

Consolidate Telegram failed-media handling so single-message and media-group getFile-null failures surface a rate-limited unavailable-media state, preserve useful file_id metadata, and avoid placeholder-only delivery.

## Reproduction Assessment

Yes, at source level: make Telegram `ctx.getFile()` fail for a video on current main; `resolveMedia()` returns `null`, the handler proceeds with `allMedia = []`, and body construction can emit placeholder/file_id context without an attachment. I did not run a live Telegram bot reproduction in this read-only review.

## Solution Assessment

No, current main is not the best solution because it still preserves ambiguous unavailable-media context without a clear failure state. The direct closed PR was also not the best final shape because it dropped fallback metadata and added unthrottled warning replies.
