## Label Justifications

- `P2`: This is a normal-priority Telegram media-ingestion bug with clear user impact but limited blast radius to Telegram inbound media.
- `impact:message-loss`: The Telegram message can reach the agent without the actual video attachment, leaving only placeholder text.
- `impact:session-state`: The session transcript can imply video media was present while the attachment state is empty, creating context drift.

## Summary

Keep open: current main and the latest release still have the source-level Telegram getFile-null path, and the linked fix attempts are closed/unmerged or external, so the reported video placeholder failure is not cleanup-closeable.

## What This Changes

The issue asks OpenClaw to stop silently converting failed Telegram inbound video downloads into a bare media placeholder with no attachment or visible failure.

## Best Possible Solution

Land one current extensions/telegram fix that carries explicit unavailable-media state through resolver/body handling, keeps useful fallback metadata, rate-limits user warnings, covers media groups, and proves both failed and normal media paths.

## Reproduction Assessment

Yes from source inspection, but not from a live Telegram run here. Force ctx.getFile() to throw a file-too-big error or exhaust retries for a video, and current main returns null, builds allMedia = [], and can still render a media placeholder without an attachment.

## Solution Assessment

No current-main solution exists yet. The best fix should make failed media explicit to the user and agent while preserving safe fallback metadata where useful.
