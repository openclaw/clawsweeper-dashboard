## Summary

Keep open: current main still has the Telegram null-media path where `getFile()` failure returns `null`, the single-message handler only warns on thrown media errors, and the body builder can still emit media placeholder text with no attachment. The focused target-repo PR is closed unmerged, and the remaining open batch PR is conflicting and still only an implementation candidate.

## What This Changes

The issue requests that Telegram inbound video failures from `getFile()` produce a real attachment or a clear unavailable-media warning instead of a misleading placeholder-only agent body.

## Best Possible Solution

Land or split a focused current-layout Telegram plugin fix that carries the `getFile()` failure state through media resolution, sends an accurate rate-limited warning, and makes any fallback placeholder clearly unavailable rather than attached.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a rejecting `ctx.getFile()` returns `null`, `resolveMedia()` propagates `null`, the handler does not warn on that path, and the body builder can still emit media placeholder text with empty `allMedia`. I did not run a live Telegram reproduction.

## Solution Assessment

Yes for keeping this open. The maintainable fix belongs in the Telegram media resolver, handler, and body construction path; the closed focused PR did not merge, and the open batch PR is only a conflicting candidate.
