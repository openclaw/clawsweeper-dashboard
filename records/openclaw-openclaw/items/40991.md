## Summary

Current main still has the reported Telegram null-media path, and open PR #41048 already references this issue with closing syntax, so this issue should stay open until that focused fix lands, is replaced, or is closed.

## What This Changes

The issue asks Telegram inbound media handling to avoid placeholder-only video messages when getFile() fails and no attachment is available.

## Best Possible Solution

Land PR #41048 or an equivalent narrow Telegram plugin patch that warns when media resolution returns null, preserves any caption text, and suppresses attachment-implying placeholders when no attachment exists.

## Reproduction Assessment

Yes. Current source and tests provide a high-confidence source-level reproduction: getFile failures return null, the handler does not warn on that null path, and the body test still expects a video placeholder when allMedia is empty.

## Solution Assessment

Yes. The narrowest maintainable solution is in the Telegram plugin resolver, single-message handler, and inbound-body builder with focused regression tests; no core API or product decision appears necessary.
