## Label Justifications

- `P2`: This is a normal-priority Telegram media ingestion bug with a clear source path and limited blast radius to failed inbound media resolution.
- `impact:message-loss`: A failed Telegram video download can leave the agent without the actual media attachment while the message continues processing.
- `impact:session-state`: The session transcript can record misleading media placeholder metadata that does not correspond to an accessible attachment.

## Summary

Keep open: current main still lets Telegram `getFile()` failures resolve to `null`, skips the user-visible failed-download warning on that path, and can build an inbound body with media placeholder metadata but no resolved attachment. The linked implementation PRs are open or unmerged, so this issue should stay open until an equivalent fix lands.

## What This Changes

The issue requests that Telegram inbound videos whose `getFile()` lookup fails do not enter the agent session as a bare `<media:video>` placeholder without an attachment or visible failure warning.

## Best Possible Solution

Land or split a focused Telegram plugin fix that carries `getFile()` failure state through media resolution, sends an accurate warning, and prevents unavailable media metadata from being represented as an attached video.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `ctx.getFile()` rejection returns `null`, `resolveMedia()` propagates `null`, the warning path only handles thrown errors, and body rendering can emit placeholder metadata with empty `allMedia`.

## Solution Assessment

Yes for keeping this open. The narrow maintainable fix belongs in the Telegram plugin media resolver, handler warning path, and message-body construction; the linked PRs are candidates, not merged fixes.
