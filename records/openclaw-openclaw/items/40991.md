- `impact:message-loss`: The inbound video's actual content is not delivered to the agent when Telegram file resolution fails.
- `impact:session-state`: The session can record a media marker that does not correspond to an available attachment.

## Summary

The bug remains on current main: Telegram `getFile()` failures return `null`, bypass the thrown-error warning, and can leave a media-only session message containing a bare placeholder without an attachment. The existing linked implementation should be revised because a later merged change intentionally preserves useful `file_id` recovery context.

## What This Changes

Surface an explicit Telegram media-download failure and avoid representing an unavailable inbound video as a successfully attached media item.

## Best Possible Solution

Return a closed media-resolution outcome that distinguishes unavailable downloads from intentionally unsupported media, then warn the user and render an explicit unavailable marker with `file_id` while preserving any caption.

## Reproduction Assessment

Yes, at source level with high confidence: make `ctx.getFile()` exhaust retries or return Telegram's file-too-big error for a video-only message, and current main proceeds with no attachment and a bare placeholder. A live Telegram reproduction was not run in this read-only review.

## Solution Assessment

No, the older linked patch is not the best current solution because suppressing all placeholders would discard the `file_id` recovery context added by https://github.com/openclaw/openclaw/pull/59948. The narrow maintainable fix is a typed failure outcome that preserves metadata and drives both the warning and body representation.
