## Label Justifications

- `P2`: This is a normal-priority Telegram media ingestion bug with limited blast radius but real user-visible attachment loss.
- `impact:message-loss`: The reported failure drops the actual inbound video attachment while still forwarding a placeholder to the agent.
- `impact:session-state`: The session transcript can claim media exists via <media:video> even when no usable attachment was attached.

## Summary

Keep open: current main and v2026.6.5 still allow Telegram getFile failures to return null, skip the thrown-error warning path, and build an empty-media placeholder body. The linked target-repo PR is closed unmerged, and the only open closing PR is still an unmerged fork PR, so the issue remains valid.

## What This Changes

The issue asks OpenClaw to avoid misleading Telegram inbound video placeholder-only messages when media resolution fails and to surface a clear download failure instead.

## Best Possible Solution

Land a focused current-layout Telegram fix that carries null media-resolution failures into an accurate, rate-limited user-visible warning and prevents placeholder-only agent bodies without dropping useful caption/file_id context.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source-level: getFile failures return null, the single-message handler only warns on thrown errors, and the body builder can emit a media placeholder with allMedia empty.

## Solution Assessment

Is this the best way to solve the issue? The issue's direction is right, but the stale linked branch is not the best final shape; the fix should be applied to extensions/telegram with warning throttling and metadata-preserving empty-media handling.
