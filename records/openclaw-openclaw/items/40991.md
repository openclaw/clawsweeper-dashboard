## Summary

Keep open: current main and the latest shipped tag still have the source-reproducible Telegram path where `getFile()` failure resolves to `null`, avoids the single-message failure reply path, and can render video metadata without an actual attachment. The direct target-repo fix PRs were closed unmerged, the open fork PR is not landed in this repo, and the related open document PR does not close this video failure mode.

## What This Changes

The issue asks OpenClaw to avoid silently turning failed Telegram inbound video downloads into a misleading `<media:video>` body with no attachment or visible failure.

## Best Possible Solution

Land a focused current-layout Telegram plugin fix that carries unavailable-media reason state through media resolution, warning, media-group, and inbound body rendering so failed video downloads are explicit without implying a real attachment exists.

## Reproduction Assessment

Yes, source-level: mock Telegram `ctx.getFile()` to reject with a file-too-big error or exhaust retries, and current main returns `null`, skips the single-message warning path, and can render video metadata with empty `allMedia`. ## Solution Assessment

Yes: the narrow maintainable solution belongs in the Telegram plugin media resolver/null-state plumbing, handler warning path, media-group handling, and body construction rather than broad core changes.
