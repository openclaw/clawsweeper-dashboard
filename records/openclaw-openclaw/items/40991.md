## Label Justifications

- `P2`: Telegram media ingestion can misrepresent or lose a usable video attachment, but the blast radius is bounded to one channel/media path.
- `impact:message-loss`: The reported Telegram video payload can reach the agent session without the actual media attachment.
- `impact:session-state`: The session transcript can imply media context is present even when no attachment path exists, creating prompt/context drift.

## Summary

Keep open: current main and v2026.6.10 still have the source-proven Telegram path where `getFile()` failures return `null`, no failed-download warning is sent, and the inbound body can still carry media placeholder text without an attachment.

## What This Changes

The issue asks for Telegram inbound video `getFile()` failures to attach real media or surface an explicit unavailable-media failure instead of a misleading placeholder-only body.

## Best Possible Solution

Consolidate Telegram failed-media handling so unavailable single-message media preserves useful metadata, adds an explicit reason or rate-limited user-visible warning, and covers captions, no-caption messages, media groups, and debounce paths.

## Reproduction Assessment

Yes, at source level: make Telegram `ctx.getFile()` fail for a video and current main returns `null`, proceeds with `allMedia = []`, and can render media placeholder text without an attachment. I did not run a live Telegram bot in this read-only review.

## Solution Assessment

No, not by simply dropping all placeholders. The maintainable fix should expose an unavailable-media state or warning while preserving useful fallback metadata such as `file_id` and avoiding unbounded warning replies.
