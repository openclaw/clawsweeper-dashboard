## Label Justifications

- `P2`: This is a normal-priority Telegram media-ingestion bug with clear user impact but limited blast radius to Telegram inbound media.
- `impact:message-loss`: The Telegram message can reach the agent without the actual video attachment, leaving only placeholder text.
- `impact:session-state`: The session transcript can imply video media was present while the attachment state is empty, creating context drift.

## Summary

Keep open: current main and the latest release still have the source-level Telegram getFile-null path, and the related fix attempts are unmerged or partial, so this still needs maintainer consolidation rather than cleanup closure.

## What This Changes

The issue asks for Telegram inbound video getFile failures to attach real media or surface an explicit failure instead of sending a misleading placeholder-only agent message.

## Best Possible Solution

Implement one current `extensions/telegram` fix that carries an explicit unavailable-media reason through resolver/body handling, rate-limits user warnings, preserves useful file_id metadata, covers media groups, and proves both failed and normal media paths.

## Reproduction Assessment

Yes from source, but not from a live Telegram run here. Force `ctx.getFile()` to throw a file-too-big error or exhaust retries for a video, and current main returns null, builds `allMedia = []`, and can still render a media placeholder without an attachment.

## Solution Assessment

No, current main is not the best solution because it preserves delivery while hiding the attachment failure. The fix should make failed media explicit to the user and agent while keeping safe fallback metadata where useful.
