## Label Justifications

- `impact:message-loss`: The reported Telegram video payload can reach the session without the actual media attachment.

## Summary

Keep open: current main and v2026.6.10 still have the source-proven Telegram path where `getFile()` failures return `null`, the single-message handler only warns for thrown media errors, and body rendering can preserve video placeholder/file_id context without an attachment.

## What This Changes

The issue reports Telegram inbound videos degrading to a bare `<media:video>` body when media resolution fails, without an attachment path or visible failure message.

## Best Possible Solution

Consolidate Telegram unavailable-media handling so single-message and media-group `getFile()` null failures surface a clear, rate-limited unavailable-media state while preserving useful file_id metadata and successful attachment behavior.

## Reproduction Assessment

Yes, at source level: make Telegram `ctx.getFile()` fail for a video on current main; `resolveMedia()` returns `null`, the handler proceeds with `allMedia = []`, and body construction can emit video placeholder/file_id context without an attachment path. I did not run a live Telegram bot reproduction in this read-only review.

## Solution Assessment

No, current main is not the best solution because it still lacks an explicit unavailable-media state for the null path. A safe fix should cover single messages, media groups, warning throttling, and metadata preservation rather than only suppressing placeholders.
