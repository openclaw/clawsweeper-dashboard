## Summary

Keep open: current main and the latest release still let Telegram `getFile()` failures return `null`, skip the user-visible media-download warning path, and build an inbound body that can preserve only media placeholder metadata without an attached file.

## What This Changes

The issue requests a Telegram media-ingestion fix so failed inbound video resolution surfaces an explicit failure instead of silently sending a placeholder-only message to the agent.

## Best Possible Solution

Land or split a focused Telegram plugin fix that carries the `getFile` failure reason through media resolution, sends an accurate rate-limited warning, and marks any fallback media metadata as unavailable rather than attached.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `ctx.getFile()` rejection returns `null`, `resolveMedia()` propagates `null`, the handler does not warn on that path, and body rendering can emit placeholder metadata with empty `allMedia`.

## Solution Assessment

Yes for keeping this open. The narrow maintainable fix belongs in the Telegram media resolver, handler, and body construction path; the linked PRs are implementation candidates, not merged fixes.
