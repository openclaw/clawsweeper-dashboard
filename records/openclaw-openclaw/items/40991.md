## Label Justifications

- `P2`: This is a normal-priority Telegram media ingestion bug with a clear source path and limited blast radius to failed inbound media resolution.
- `impact:message-loss`: A failed Telegram video download can leave the agent without the actual media attachment while the message continues processing.
- `impact:session-state`: The session transcript can record misleading media placeholder metadata that does not correspond to an accessible attachment.

## Summary

Keep open: current main still lets Telegram `getFile()` failures resolve to `null`, skips the user-visible failed-download warning on that path, and can enqueue a message body that represents unavailable media as placeholder metadata. The focused fix PR was closed unmerged and the broader Telegram batch PR is still open, so this should remain open until an equivalent fix lands.

## What This Changes

The issue asks Telegram inbound video handling to avoid misleading placeholder-only media messages when `getFile()` cannot resolve the attachment and to surface an explicit failure state.

## Best Possible Solution

Land or split a focused Telegram plugin fix that carries `getFile()` failure state through media resolution, sends an accurate user warning, and prevents unavailable media metadata from being represented as an attached video.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `ctx.getFile()` rejection returns `null`, `resolveMedia()` propagates `null`, the warning path only handles thrown errors, and body rendering can emit placeholder metadata with empty `allMedia`.

## Solution Assessment

Yes for keeping this open. The narrow maintainable fix belongs in the Telegram plugin media resolver, handler warning path, and message-body construction; the linked PRs are candidates, not merged fixes.
