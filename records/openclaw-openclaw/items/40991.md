## Summary

Keep open: current `main` still has the reported Telegram null-media path, so a `getFile()` failure can produce no attachment and no user-visible download warning. The issue is not fixed by the closed focused PR, and the remaining open batch PR is only an unmerged implementation candidate.

## What This Changes

Telegram inbound video should not enter the agent session as a misleading placeholder-only media message when `getFile()` fails without a downloaded attachment.

## Best Possible Solution

Land or split a narrow Telegram plugin fix so null `getFile()` failures surface a clear unavailable-media warning/state and no longer imply that a video attachment was delivered.

## Reproduction Assessment

Yes from source inspection: a rejecting `ctx.getFile()` makes `resolveMedia()` return `null`, the handler does not warn on that path, and the body builder can still emit the media placeholder with empty `allMedia`. I did not run a live Telegram reproduction.

## Solution Assessment

Yes for keeping this open. The maintainable fix belongs in the Telegram media resolver/handler/body path, preferably through a focused PR or a split from the existing broad Telegram batch PR.
