## Summary

Current `main` still has the reported Telegram null-media path: `getFile()` failures can return `null`, the single-message warning only runs for thrown media errors, and the inbound body builder can still emit media placeholder text when no attachment was resolved. The issue is also paired with open implementation PR #41048 and broader open PR #42059, so it should remain open until maintainers land, split, or close that fix path.

## What This Changes

Telegram inbound video/media failures from `getFile()` should produce a clear failed-download state instead of a misleading placeholder-only message with no attachment.

## Best Possible Solution

Resolve this through the Telegram plugin fix path: make failed media resolution distinct from a valid no-media message, send a visible warning for the `null` getFile failure path, avoid injecting misleading placeholder-only media text when no attachment exists, preserve caption text, and update Telegram resolver/body tests to lock the new behavior.
