## Label Justifications

- `P2`: Telegram media ingestion can misrepresent an unavailable attachment, but the issue is bounded to one channel/media path.
- `impact:message-loss`: The reported Telegram video payload can reach the session without the actual media attachment.
- `impact:session-state`: The session transcript can imply media was attached even when no media path exists, creating context drift.

## Summary

Keep open: current main and the latest release still have a source-proven Telegram path where `getFile()` failures resolve to `null`, no failed-download warning is sent, and body construction can still produce media placeholder text without an attachment.

## What This Changes

The issue reports Telegram inbound video download failures degrading into a placeholder-only `<media:video>` body without a real attachment path or user-visible failure.

## Best Possible Solution

Unify Telegram failed-media handling so failed single-message downloads expose an explicit unavailable-media state, preserve useful fallback context such as `file_id`, avoid unbounded warning replies, and cover the relevant caption, no-caption, media-group, and debounce paths.

## Reproduction Assessment

Yes, at source level: make Telegram `ctx.getFile()` fail for a video and current main returns `null`, proceeds with `allMedia = []`, and can render media placeholder text without an attachment. I did not run a live Telegram bot reproduction in this read-only review.

## Solution Assessment

No: suppressing all fallback placeholders is too broad because current behavior intentionally preserves useful `file_id` metadata for captioned failures. The better fix is an explicit unavailable-media reason that preserves metadata and avoids unbounded warning replies.
