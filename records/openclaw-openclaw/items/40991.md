- `impact:message-loss`: The reported video content can reach the agent as placeholder text while the actual media attachment is absent.
- `impact:session-state`: The session transcript can imply media was attached even when no media path exists, causing context drift.

## Summary

Keep open: current main and v2026.6.8 still allow Telegram getFile failures to resolve as null, continue without a media attachment, and render fallback media text, while direct fix attempts are closed, unmerged, or only partial.

## What This Changes

The issue asks Telegram inbound video getFile failures to surface an explicit unavailable/error state instead of entering the agent session as a bare media placeholder without an attachment.

## Best Possible Solution

Land one current extensions/telegram fix that carries an explicit unavailable-media reason through resolution/body handling, avoids unbounded warning spam, preserves useful file_id metadata, covers media groups, and proves failed and normal video paths.

## Reproduction Assessment

Yes from source: make Telegram ctx.getFile() fail with a file-too-big error or exhaust retries for a video, and current main returns null from media resolution while body construction can still render fallback media text with allMedia = []. I did not run a live Telegram bot probe in this read-only pass.

## Solution Assessment

No, current main is not the best solution because it preserves delivery while hiding the attachment failure. The fix should make failed media explicit to the user and agent without breaking successful media attachment or useful fallback metadata.
