- `P2`: This is a normal-priority Telegram media ingestion bug with clear source evidence and limited blast radius to failed inbound media resolution.
- `impact:message-loss`: A failed Telegram video download can leave the agent without the actual media attachment while processing continues.
- `impact:session-state`: The session transcript can record media placeholder metadata that does not correspond to an accessible attachment.

## Summary

Keep open: current main still returns null on Telegram getFile failures, single-message warnings only cover thrown download errors, and body construction can still send a media placeholder without an attachment. The target-repo fix PRs for this specific video case are closed unmerged, while the remaining open references are a fork PR on legacy paths and a document-only related PR.

## What This Changes

The issue asks Telegram inbound video download failures to surface an explicit unavailable-media state instead of sending a misleading placeholder-only agent message.

## Best Possible Solution

Land a focused current-layout Telegram plugin fix that carries unavailable-media reason state through resolver, warning, debounce, media-group, reply-context, and body-rendering paths without implying a real attachment exists.

## Reproduction Assessment

Yes, source-level: mock Telegram ctx.getFile to reject with a file-too-big error or exhaust retries, and current main returns null, skips the thrown-error warning path, and can render video metadata with no attachment. I did not run live Telegram proof because this review is read-only.

## Solution Assessment

Yes for keeping this issue open and queueing a focused repair: the narrow maintainable fix belongs in the Telegram plugin media resolver, handler warning path, debounce/media-group/reply plumbing, and body construction rather than core changes.
