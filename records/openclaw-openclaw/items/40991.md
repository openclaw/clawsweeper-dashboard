## Summary

Current main still has the reported Telegram null-media path, and this issue is already paired with open implementation PRs that use closing syntax. It should remain open until maintainers land, replace, or close that fix path.

## What This Changes

The issue requests that Telegram inbound video getFile failures surface a clear media-download failure and avoid misleading placeholder-only session bodies.

## Best Possible Solution

Land the focused Telegram plugin behavior change, preferably through #41048 or an equivalent extracted patch from #42059, so getFile null/failure is distinct from “no media,” users get a visible warning, captions are preserved, and placeholder-only bodies are suppressed when no attachment exists.

## Reproduction Assessment

Yes. Current source and tests provide a high-confidence code-level reproduction: mocked getFile failures return null, the handler does not warn on that null path, and the body test still expects a video placeholder when allMedia is empty.

## Solution Assessment

Yes. The narrowest maintainable fix is in the Telegram plugin’s resolver/handler/body-building path with focused regression tests; the broad batch PR is useful context but should not be required to solve this single issue.
