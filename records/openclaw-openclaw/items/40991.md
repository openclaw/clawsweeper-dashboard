## Label Justifications

- `P2`: This is a normal-priority Telegram media ingestion bug with a clear source path and limited blast radius to failed inbound media resolution.
- `impact:message-loss`: A failed Telegram video download can leave the agent without the actual media attachment while the message continues processing.
- `impact:session-state`: The session transcript can record misleading media placeholder metadata that does not correspond to an accessible attachment.

## Summary

Keep open: current main still has the Telegram media null-return path, the single-message warning only runs for thrown media errors, and body construction can still represent unavailable media as placeholder metadata. The focused fix PR is closed unmerged and the broader Telegram batch PR is still open, so this should stay open until an equivalent focused fix lands.

## What This Changes

Report requests Telegram inbound media handling to avoid placeholder-only video messages when `getFile()` resolution fails and to surface a clear failure warning.

## Best Possible Solution

Land or split a focused Telegram plugin fix that carries media-resolution failure state to the handler, sends an accurate rate-limited warning, and avoids representing unavailable media as an attached video while preserving useful file metadata.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `ctx.getFile()` rejection returns `null`, `resolveMedia()` propagates `null`, the user-visible warning path only handles thrown errors, and body rendering can emit placeholder metadata with empty `allMedia`.

## Solution Assessment

Yes for keeping this open. The maintainable fix belongs in the Telegram plugin media resolver, handler warning path, and inbound body construction, with care to preserve useful metadata and avoid unbounded warning spam.
