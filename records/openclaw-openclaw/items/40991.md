## Summary

Keep open: current main and the latest release still let Telegram getFile failures return null, continue without the media-download warning path, and build an agent body that can contain only media placeholder text with no saved attachment.

## What This Changes

Telegram inbound video getFile failures should surface an explicit unavailable/failure state instead of sending a placeholder-only agent body with no attachment.

## Best Possible Solution

Land or split a focused Telegram plugin fix that carries the getFile failure reason through media resolution, sends an accurate rate-limited warning, and marks any fallback placeholder as unavailable rather than attached.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: rejecting ctx.getFile() returns null, resolveMedia() propagates null, the handler does not warn on that path, and body rendering can still emit placeholder text with empty allMedia.

## Solution Assessment

Yes for keeping this open. The maintainable fix belongs in the Telegram media resolver, handler, and body construction path; the linked PRs are implementation candidates, not merged fixes.
