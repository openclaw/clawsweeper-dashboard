## Summary

Keep open: current main and v2026.5.22 still allow Telegram `getFile()` failures to return `null`, skip the single-message failure reply path, and render media metadata without an attached file. The earlier direct fix PRs were closed unmerged, while the open document-focused PR does not cover this video failure mode.

## What This Changes

The issue asks for failed Telegram inbound video resolution to attach the real media or surface an explicit unavailable-media state instead of a placeholder-only agent body.

## Best Possible Solution

Land a focused current-layout Telegram plugin fix that carries unavailable-media reason state through resolver, single-message, debounce, media-group, reply-context, and body rendering while preserving useful file_id metadata and rate-limiting user-visible warnings.

## Reproduction Assessment

Yes, source-level: mock Telegram `ctx.getFile()` to reject with a file-too-big error or exhaust retries, and current main returns `null`, skips the single-message warning path, and can render media metadata with empty `allMedia`. I did not run a live Telegram video because this review is read-only.

## Solution Assessment

Yes, the maintainable fix belongs in Telegram plugin media resolver/null-state plumbing, handler warning propagation, debounce/media-group handling, and body construction rather than broad core changes. The fix should preserve fallback metadata and avoid unthrottled auto-replies.
