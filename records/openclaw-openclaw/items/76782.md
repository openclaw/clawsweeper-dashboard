## Summary

Keep open: current main still routes skills through name/description/location only, so `triggers` support would be a new public SKILL.md metadata and routing contract. The ClawHub `eyes` example is useful demand signal, but OpenClaw core does not consume that field today.

## What This Changes

Request adds an optional `triggers` field to skill frontmatter and `<available_skills>` so exact keyword matching can augment description-based skill routing.

## Best Possible Solution

Define a backward-compatible trigger metadata contract and routing semantics, then parse and normalize trigger arrays, render them within prompt budget, update docs/tests, and coordinate with the related skill-matching issues.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior. Source inspection of current main gives a high-confidence absence check because skills are rendered and routed by name, description, and location only.

## Solution Assessment

Unclear: the prompt-field approach is plausible, but maintainers should first choose the public metadata shape and exact-match semantics. A safer implementation would keep the field optional, normalized, documented, and covered by prompt-budget tests.
