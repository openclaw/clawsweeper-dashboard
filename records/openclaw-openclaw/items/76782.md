## Summary

Keep open: current main still exposes skills to agents through name, description, and location only, so `triggers` would be a new public skill metadata and routing contract that needs maintainer API/product judgment.

## What This Changes

Add an optional `triggers` field to skill `SKILL.md` frontmatter and expose it in `<available_skills>` so agents can combine exact keyword routing with description matching.

## Best Possible Solution

Define the optional trigger metadata schema and routing semantics first, then implement normalized parsing/rendering with docs and regression coverage while preserving behavior for skills that omit it.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior. Source inspection gives a high-confidence absence check for trigger support in the prompt formatter, loader, skill entry contract, and docs.

## Solution Assessment

Unclear: the proposed prompt field is plausible, but maintainers should first choose the public metadata shape, matching semantics, normalization rules, and prompt-budget behavior.
