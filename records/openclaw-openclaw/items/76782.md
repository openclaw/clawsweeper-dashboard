## Summary

Keep open: current main still exposes skills to the agent as name/description/location only, so `triggers` support would be a new public skill metadata and routing contract. The ClawHub `eyes` example in the issue is useful demand signal, but OpenClaw core does not consume that field today.

## What This Changes

Request adding an optional SKILL.md `triggers` frontmatter field and including it in `<available_skills>` so skill routing can combine exact keyword matches with description-based selection.

## Best Possible Solution

Have maintainers decide the public metadata shape and routing semantics, then implement optional normalized trigger parsing/rendering with prompt-budget docs/tests and no behavior change for skills that omit it.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior. Source inspection of current main gives a high-confidence absence check for the requested `triggers` field.

## Solution Assessment

Unclear: the prompt-field approach is plausible, but maintainers should choose whether triggers live as top-level frontmatter or under `metadata.openclaw`, how exact matching interacts with model selection, and how to budget prompt tokens.
