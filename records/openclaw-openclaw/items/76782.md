Keep open: current main and v2026.6.10 still expose skills through `name`, `description`, `location`, and optional `version` only, while the requested trigger metadata and exact-match routing semantics remain an unresolved public skill-contract decision. ## What This Changes

The issue requests an optional `triggers` field in `SKILL.md` frontmatter so skill routing can use explicit keyword matching alongside semantic description matching.

## Best Possible Solution

Define the supported trigger metadata contract and routing semantics first, then update parsing, types, prompt rendering, compact-budget behavior, docs, Codex/native parity expectations, and focused tests together.

## Solution Assessment

Unclear until maintainers choose the public contract. A `triggers` field is plausible, but the best implementation must settle field placement, exact-match semantics, prompt-budget behavior, relationship to broader smart indexing, and Codex/native compatibility first.
