## Summary

Keep open. Current main has Telegram-specific DNS/family controls, but the transport still does not inherit the process `dns.getDefaultResultOrder()` value and still emits a warning when the sticky IPv4 fallback is promoted after a recoverable fetch failure.

## What This Changes

Request that Telegram Bot API transport honor process-level IPv4-first DNS policy or otherwise reduce noisy recovered IPv4 fallback warnings.

## Best Possible Solution

Teach Telegram transport to inherit process DNS defaults when no Telegram override is set and make recovered sticky IPv4 fallback logging debug-level or rate-limited, while preserving warnings for persistent or escalating network failures.

## Reproduction Assessment

Yes for the underlying current-main behavior: the existing mocked fetch path can trigger sticky IPv4 fallback with `autoSelectFamily` enabled, and source shows that promotion logs at warn level. The exact 24-hour frequency still depends on a flaky dual-stack host.

## Solution Assessment

Unclear as written: `ipv4first` should not be treated as identical to `family: 4`. A safer fix is to honor the process DNS default separately, keep the explicit Telegram `network.autoSelectFamily` control for IPv4-only behavior, and reduce warning noise only for successful recovery.
