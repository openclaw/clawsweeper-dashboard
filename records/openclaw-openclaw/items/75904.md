## Summary

Keep open. Current main still logs Telegram sticky IPv4 fallback promotion at warning level, and the linked open PR is an implementation candidate with closing syntax, so this issue should remain open until that PR merges or is closed.

## What This Changes

The issue requests quieter or DNS-aware Telegram IPv4 fallback behavior when polling recovers on dual-stack hosts already configured for IPv4-first DNS.

## Best Possible Solution

The shipped Telegram transport should log successful sticky IPv4 fallback promotion at debug level while preserving warning visibility for persistent or escalating network failures.

## Reproduction Assessment

Yes for the underlying behavior: the existing fetch test exercises recoverable timeout-to-sticky-IPv4 fallback, and current main shows that promotion logs at warn level. The exact operational count still depends on a flaky dual-stack host.

## Solution Assessment

Yes for the noise portion: downgrading recovered sticky fallback promotion is the narrowest maintainable fix. Treating IPv4-first DNS as forced IPv4 would be broader and should stay separate unless maintainers choose that policy.
