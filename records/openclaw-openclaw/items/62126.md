## Summary

Keep open. The proposed UX has sufficient screenshot proof and still targets a real onboarding gap, but the current head is conflicting with main and reintroduces managed provider discovery before the user can decline web_search, which violates the current configure contract.

## What This Changes

The PR adds a shared Codex-native web-search prompt for configure/onboarding, falling back to the managed provider picker when requested, with focused wizard tests.

## Best Possible Solution

If maintainers accept prompt-first onboarding, rebase the helper onto current main, preserve deferred managed-provider discovery, and validate both Codex-native and managed-provider paths; otherwise fold the idea into the broader configuration UX design.

## Reproduction Assessment

Yes for the PR regression: source inspection shows `resolveSearchProviderOptions` runs before the web_search opt-in, and current tests define that this must not happen when web search is disabled. The requested onboarding improvement itself is a feature, not a bug reproduction.

## Solution Assessment

Unclear until maintainers choose the UX. The shared helper is a reasonable prompt-first implementation shape, but it must preserve current deferred provider discovery and be rebased onto the latest plugin-backed setup flow.
