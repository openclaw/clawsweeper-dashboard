## Summary

Keep open: the throttle-control direction is plausible, but this PR should not merge as-is because it changes current-main Feishu streaming cadence, leaves public config surfaces out of sync, overlaps a competing streaming-config shape, and still lacks real Feishu/Lark behavior proof.

## What This Changes

The PR adds a Feishu `streamingThrottleMs` config field and passes it into `FeishuStreamingSession` as the streaming-card update throttle.

## Best Possible Solution

Define one Feishu-owned streaming throttle contract across schema, manifest, docs, runtime, and focused tests, preserving the current cadence unless maintainers intentionally choose a new API-rate tradeoff.

## Reproduction Assessment

Yes for the PR review defects: source inspection shows the branch would use a 100 ms fallback while current main has a tested 160 ms cadence and no public throttle surface. The original request itself is a feature/config change, not a current-main bug report.

## Solution Assessment

No. The throttle-control direction may be useful, but this branch is not the best merge path until it preserves current defaults or gets an explicit rate decision, publishes the config surface, and resolves the shape overlap with https://github.com/openclaw/openclaw/pull/42936.
