## Summary

Keep open: current main still lacks a user-facing AppTranslocation diagnostic/recovery path for Codex.app-backed app-server crashes or hangs. Main now prefers `/Applications/Codex.app` when resolving a managed macOS Codex binary, but that does not warn about already-running translocated Codex.app processes, and the generic `released=0` stuck-session portion remains only partially covered by separate recovery work.

## What This Changes

Requests macOS diagnostics and recovery for Codex.app AppTranslocation-related app-server crashes or hangs that leave embedded runs stuck.

## Best Possible Solution

Add a macOS-aware Codex diagnostic that warns when Codex.app or its app-server is running from AppTranslocation and gives clear `/Applications` plus full-restart guidance, while keeping generic lane-release recovery on its separate focused track.
