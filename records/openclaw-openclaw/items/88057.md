## Summary

Keep open: the direction matches ambient room-event session-state semantics, but the branch can persist an OpenClaw session id as a CLI backend session for direct room events when CLI hooks synthesize or block a result, and the PR lacks real behavior proof.

## What This Changes

The PR removes room-event CLI session suppression in the direct and queued auto-reply runners and updates tests to expect existing CLI bindings and returned CLI binding metadata to be reused.

## Best Possible Solution

Reuse actual CLI bindings for room events only when the CLI runner returns real CLI session metadata, preserve suppression for synthetic or blocked hook results, and land with targeted tests plus redacted real CLI room-event proof.

## Reproduction Assessment

Yes, source inspection shows current main suppresses CLI session reuse and returned session metadata for room-event direct runs, and the PR updates that behavior. I did not run a live CLI room-event scenario in this read-only review.

## Solution Assessment

No as submitted. Reusing real CLI bindings is the right narrow direction, but the direct runner still needs to distinguish real CLI session metadata from synthetic hook or blocked-run metadata before persisting it.
