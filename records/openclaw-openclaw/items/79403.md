## What This Changes

The PR adds `tools.web.search.maxResults` to the gateway config mutation allowlist and changes memory preflight compaction and memory flush to pass `AbortSignal.any([replyOperation.abortSignal])`, with regression tests for the new signal object.

## Best Possible Solution

Keep the memory fix focused on isolating auxiliary compaction/flush control flow from the parent turn, with a regression that simulates the actual child timeout/failure and live or log proof; split the `maxResults` allowlist change into its own PR or explicitly review it here.

## Reproduction Assessment

No. Current main clearly shares the parent signal at both call sites, but I did not establish a high-confidence path where the child can abort that read-only signal, and the PR proof only exercises mocked unit tests.

## Solution Assessment

No. `AbortSignal.any([parent])` creates a distinct signal object that follows the parent, but it does not isolate shared `replyOperation`/backend state or prove that auxiliary failures are downgraded instead of aborting the parent turn.
