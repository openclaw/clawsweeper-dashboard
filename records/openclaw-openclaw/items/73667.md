## Summary

Keep open. The PR addresses a real Active Memory/QMD reliability problem, but the current draft introduces a hook-timeout regression for valid custom recall budgets and overlaps with newer current-main reliability work that needs maintainer direction.

## What This Changes

The PR lowers Active Memory default timeout/setup-grace values, updates Active Memory docs/tests for shorter recall budgets, and adds randomized parallel QMD startup staggering in the Gateway memory startup path.

## Best Possible Solution

Finish the canonical #72015 reliability work with config-safe hook budgets, regression coverage for timeouts above the default, aligned docs/changelog, and QMD startup behavior that builds on current opt-in lazy initialization.

## Reproduction Assessment

Yes for the PR regression: configure Active Memory with timeoutMs above 3000 ms, such as the PR's own 5000 ms full-mode guidance. The branch registers a 4000 ms hook timeout, so the hook runner can cancel recall before the plugin watchdog for the configured budget returns.

## Solution Assessment

No, not as proposed. The lower-latency direction may be valid, but the hook envelope must be based on the resolved configured budget or documented cap, and QMD herd mitigation should be reconciled with current main's opt-in/lazy startup policy.
