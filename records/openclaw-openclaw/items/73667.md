## Summary

Keep open. The draft PR targets a real Active Memory/QMD reliability problem, but it is unsafe as-is because it can cap valid custom Active Memory recall budgets at a 4s hook timeout and it overlaps with newer current-main timeout and QMD startup mitigations that need maintainer direction.

## What This Changes

The PR lowers Active Memory default recall budgets and docs, removes default setup grace, and adds randomized QMD startup jitter with small test updates.

## Best Possible Solution

Land or replace this with a maintainer-approved reliability change that preserves config-derived hook budgets, adds regression coverage for custom timeouts above the default, includes docs/changelog, and reconciles QMD startup behavior with current main's opt-in lazy sync policy.

## Reproduction Assessment

Yes for the PR regression from source inspection: configure Active Memory with timeoutMs above 3000 ms, such as 5000 ms or 90000 ms, and the branch's fixed 4000 ms hook timeout can fire before the configured recall budget. I did not run tests because this was a read-only review.

## Solution Assessment

No, not as proposed. The latency direction is plausible, but the hook timeout must be sized from the resolved configured recall budget or cap, and QMD startup policy should be reconciled with newer current-main behavior before merge.
