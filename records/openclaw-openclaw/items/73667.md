## Summary

Keep this PR open. Current main has not adopted the lower Active Memory defaults, zero setup grace, docs guidance, or QMD startup jitter, but the branch has a blocking timeout-envelope regression that can cancel valid custom recall budgets before Active Memory's own watchdog finishes.

## What This Changes

The PR lowers Active Memory default timeout/setup-grace values, changes the registered `before_prompt_build` hook timeout, adds randomized QMD startup staggering, updates related tests, and revises Active Memory docs toward lower recall budgets.

## Best Possible Solution

Keep a focused implementation for #72015 that bounds default recall latency without cancelling valid custom timeout budgets, and choose one explicit QMD startup strategy with tests, docs, and release notes aligned to that policy.

## Reproduction Assessment

Yes for the PR regression: configure Active Memory with `timeoutMs` above 3000 ms, then the proposed fixed 4000 ms hook registration can time out before `maybeResolveActiveRecall` reaches its own `config.timeoutMs + setupGraceTimeoutMs` watchdog. The current hook runner proves this path because it enforces registration timeouts with `Promise.race`.

## Solution Assessment

No, not as proposed. The hook envelope should be derived from the resolved per-agent recall budget or documented cap, and the QMD startup/default-latency policy should be accepted deliberately rather than hidden behind a default-based hook timeout.
