## Summary

Keep open: this PR still targets real Active Memory/QMD reliability work and current main has not implemented the proposed lower 3s default or matching docs, but the branch is not merge-ready because its hook timeout can cancel supported custom recall budgets before Active Memory's own watchdog.

## What This Changes

The PR lowers Active Memory timeout defaults/docs, removes default setup grace, registers the before_prompt_build hook with a default-derived timeout, and adds randomized QMD startup jitter.

## Best Possible Solution

Land or replace this with a maintainer-approved reliability change that preserves config-derived hook budgets, covers custom timeout values above the default, includes changelog/docs, and aligns QMD startup behavior with current main's lazy boot-sync policy.

## Reproduction Assessment

Yes for the PR regression from source inspection: configure Active Memory with timeoutMs above 3000 ms, such as the PR docs' 5000 ms full-mode guidance, and the branch's 4000 ms hook timeout can fire first. I did not run tests in this read-only review.

## Solution Assessment

No, not as proposed. The latency direction is plausible, but the hook envelope must come from the resolved configured recall budget or supported cap, and the QMD startup change needs to be reconciled with current main's lazy sequential boot-sync behavior.
