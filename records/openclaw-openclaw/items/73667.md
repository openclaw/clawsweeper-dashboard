## Summary

Keep this draft PR open, but do not merge it as-is. The hook-timeout regression flagged in the existing review comments is real, and current main has since moved QMD startup toward opt-in/lazy sync plus separate Active Memory hard-timeout work rather than this branch's fixed default/jitter policy.

## What This Changes

The PR lowers Active Memory default/setup-grace timeout budgets, updates docs/tests for shorter recall, and adds randomized parallel QMD startup jitter.

## Best Possible Solution

Land a maintainer-approved reliability change that preserves configured recall budgets, tests custom timeout values, aligns QMD startup with the opt-in/lazy policy, and closes the remaining #72015 work through the canonical follow-up.

## Reproduction Assessment

Yes for the PR regression: configure Active Memory with timeoutMs above 3000 ms, such as 5000 ms or 90000 ms. The branch registers a 4000 ms hook timeout, and the hook runner enforces that before the plugin's configured recall budget can complete.

## Solution Assessment

No, not as proposed. The lower-latency goal is plausible, but the hook envelope must be derived from the resolved configured budget or documented cap, and QMD startup changes should fit current main's opt-in/lazy refresh policy.
