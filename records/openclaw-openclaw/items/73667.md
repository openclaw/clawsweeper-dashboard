## Summary

Keep open: this draft PR targets a real Active Memory reliability problem, but the branch introduces a definite hook-timeout regression and needs maintainer repair/review plus real behavior proof before merge.

## What This Changes

The PR lowers Active Memory default timeout/setup-grace values, updates Active Memory docs/tests, and adds jittered parallel QMD startup initialization.

## Best Possible Solution

Repair or replace the branch so hook budgets come from the resolved recall timeout, preserve current-main fast-fail/QMD startup mitigations, and settle the remaining Active Memory default-policy choices with maintainer review.

## Reproduction Assessment

Yes for the PR regression from source inspection: configure Active Memory with timeoutMs above 3000 ms on the branch, and the fixed 4000 ms hook timeout can fire before the configured recall budget. I did not run tests because this was a read-only review.

## Solution Assessment

No, not as proposed. The reliability direction is plausible, but the hook envelope must be sized from the resolved configured timeout or cap, and the QMD startup/default-latency policy should be reconciled with current main before merge.
