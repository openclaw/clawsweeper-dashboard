## Summary

Keep open. This draft external PR targets a real open Active Memory/QMD reliability issue, but the branch introduces a hook-timeout regression for configured recall budgets above the new default and still lacks real behavior proof.

## What This Changes

The PR lowers Active Memory default/setup-grace timeouts, changes before_prompt_build hook timeout sizing, adds jittered parallel QMD startup initialization, updates docs, and adjusts tests.

## Best Possible Solution

Repair or replace the branch so hook budgets preserve configured recall timeouts, keep current-main fast-fail/lazy QMD mitigations, and split broader nonblocking/default-policy decisions through the canonical reliability issue.

## Reproduction Assessment

Yes, for the PR regression by source inspection: configure Active Memory on the branch with timeoutMs above 3000 ms and the fixed 4000 ms hook timeout can fire before the configured recall budget. I did not run tests because this review was read-only.

## Solution Assessment

No, not as proposed. The reliability direction is plausible, but the hook envelope must be sized from the resolved configured timeout or cap, and the QMD/default-latency policy should be reconciled with current main before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Size the hook timeout from the configured recall budget:** `extensions/active-memory/index.ts:2434`
  - body: This branch registers the Active Memory before_prompt_build hook with a default-derived 4s timeout, while config.timeoutMs still accepts larger values and the docs recommend budgets above the default. The hook runner enforces the registration timeout before the plugin watchdog can return, so custom high-timeout recall can be silently dropped; derive the envelope from the resolved configured timeout or cap and cover a custom timeout above the default.
  - confidence: 0.93
