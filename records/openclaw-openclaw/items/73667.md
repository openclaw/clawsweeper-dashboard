## Summary

Keep this draft PR open for maintainer review, but do not merge it as-is. Current main has not implemented the PR's lower-default Active Memory policy or QMD startup jitter, and the branch introduces a concrete hook-timeout regression for valid custom `timeoutMs` configurations.

## What This Changes

The PR lowers Active Memory default recall/setup-grace budgets and docs, changes the registered `before_prompt_build` hook timeout to a fixed default-derived value, and adds randomized parallel QMD startup staggering.

## Best Possible Solution

Keep the config-derived hook envelope invariant, add regression coverage for custom timeouts above the default, and land a maintainer-approved lower-latency Active Memory/QMD startup policy with docs and changelog aligned to the linked reliability issue.

## Reproduction Assessment

Yes for the PR regression: configure Active Memory with `timeoutMs` above 3000 ms, such as the PR's own `full`-mode docs guidance of 5000 ms or higher. The branch's fixed 4000 ms hook registration can time out before the plugin's configured recall watchdog returns.

## Solution Assessment

No, not as proposed. Lowering defaults may be reasonable, but the hook timeout must be derived from the resolved configured recall budget or documented cap; QMD startup should use an explicit concurrency/lazy-start policy rather than a small random delay around broad `Promise.all`.
