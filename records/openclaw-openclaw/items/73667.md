## Summary

Keep this PR open. Current main has partial, different Active Memory/QMD reliability mitigations, but it does not implement this PR's lower default recall budget or jittered startup change, and the PR branch still introduces a blocking hook-timeout regression for custom Active Memory timeouts.

## What This Changes

The PR lowers Active Memory default timeout/setup-grace values, updates Active Memory docs/tests for shorter recall budgets, and adds randomized parallel QMD startup staggering in the Gateway memory startup path.

## Best Possible Solution

Land a maintainer-approved reliability change on top of current main that preserves the config-derived hook envelope, adds regression coverage for custom recall budgets above the default, aligns docs and changelog, and fits the existing QMD startup opt-in/lazy-start policy rather than adding broad randomized startup concurrency.

## Reproduction Assessment

Yes for the PR regression: configure Active Memory with `timeoutMs` above 3000 ms, such as the PR's own full-mode docs guidance of 5000 ms or higher. The branch's fixed 4000 ms hook timeout can fire before the plugin watchdog for the configured recall budget returns.

## Solution Assessment

No, not as proposed. The lower-latency direction may be valid, but the hook timeout must be sized from the resolved configured budget or documented cap, and QMD startup policy should be reconciled with current main's opt-in/lazy startup behavior.
