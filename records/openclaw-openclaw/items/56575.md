## Summary

Keep open: current main still does not provide this PR’s explicit time-based compact/reset maintenance API, but the PR is not mergeable as-is because source-level blockers remain and the external contributor has not provided real behavior proof.

## What This Changes

The PR adds per-model `sessionCacheTtl`/`timeBasedContextCompact` settings, session activity/cache timestamps, and gateway sweeps for idle compaction or reset modes.

## Best Possible Solution

Settle the session-maintenance API boundary first: reuse the existing session reset lifecycle for idle reset, keep cache compaction/pruning/keep-warm as documented opt-in prompt-cache behavior, then land a narrower implementation with focused tests and real behavior proof.

## Reproduction Assessment

Yes for the review blockers: PR-head source inspection shows the Moonshot mock/assertion contradiction, provider eligibility gating for reset TTLs, and stale cache-touch preference. The actual maintenance sweep behavior was not live-tested, and contributor real behavior proof is absent.

## Solution Assessment

No. The current PR is not the best solution as-is because it adds a separate reset policy beside the existing session reset contract and still has blocking implementation/test defects; the safer path is product/API review followed by a narrower implementation.

  - confidence: 0.84
