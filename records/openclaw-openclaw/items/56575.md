## Summary

Keep open: current main still does not implement the proposed `sessionCacheTtl`/`timeBasedContextCompact` compact/reset modes, and the PR has blocking source-level correctness issues plus missing real behavior proof.

## What This Changes

The PR adds per-model time-based session maintenance settings, session activity/cache timestamps, and a gateway sweep that can compact or reset idle sessions.

## Best Possible Solution

Settle the session-maintenance API boundary first, reuse the existing session reset lifecycle for reset behavior, and land a narrower opt-in prompt-cache maintenance design with docs, focused tests, and real behavior proof.

## Reproduction Assessment

Yes for the review blockers: PR-head source inspection shows the Moonshot mock/assertion contradiction, reset TTL provider gating, stale cache-touch baseline, and raw alias iteration. The full gateway compact/reset behavior was not live-tested.

## Solution Assessment

No as-is. The PR adds a parallel reset policy beside the existing session reset lifecycle and still has blocking correctness defects; a narrower design after maintainer API review is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Align the Moonshot mock with the assertions:** `src/agents/pi-embedded-runner/cache-ttl.test.ts:31-36`
  - body: The mock still returns `true` for normalized Moonshot while these changed assertions expect `false`. Since `isCacheTtlEligibleProvider` returns a defined plugin eligibility result directly, these assertions fail at runtime.
  - confidence: 0.96
- **[P2] Apply reset TTLs without cache eligibility gating:** `src/agents/pi-embedded-runner/cache-ttl.ts:170-172`
  - body: `resolveCacheTtlMs` returns `null` before reading the configured mode and TTL when the provider is not prompt-cache eligible. That makes configured `reset` mode silently do nothing for non-cache providers even though reset only needs idle timestamps.
  - confidence: 0.91
- **[P2] Use the newest activity timestamp for expiry:** `src/gateway/session-cache-maintenance.ts:53-57`
  - body: `resolveCacheTouchAt` prefers an older `lastCacheTouchAt` over later assistant activity. If maintenance is re-enabled after assistant turns continued, the sweep can reset or compact a recently active session immediately.
  - confidence: 0.84
- **[P2] Deduplicate maintenance by normalized session key:** `src/gateway/session-cache-maintenance.ts:311`
  - body: The sweep acts on raw store keys from a stale snapshot. When the store contains canonical and legacy aliases for one session, the same logical session can trigger duplicate reset or compaction work in a single pass.
  - confidence: 0.8
