Overall confidence: 0.9

Full review comments:

- **[P1] Apply reset mode without prompt-cache eligibility:** `src/agents/pi-embedded-runner/cache-ttl.ts:170-172`
  - body: `resolveCacheTtlMs` returns `null` before reading `timeBasedContextCompact` when the provider is not cache-TTL eligible, and the sweep skips maintenance when that value is null. That makes `timeBasedContextCompact: "reset"` a silent no-op for providers without prompt-cache support even though reset only needs idle timestamps.
  - confidence: 0.9
- **[P1] Use the newest activity when computing expiry:** `src/gateway/session-cache-maintenance.ts:53-57`
  - body: `resolveCacheTouchAt` always prefers `lastCacheTouchAt` over `lastAssistantMessageAt`. If maintenance is disabled and later re-enabled, an old cache-touch can survive while assistant turns continue, so the sweep can treat a recently active session as expired and reset or compact it immediately.
  - confidence: 0.86
- **[P2] Align the Moonshot mock with the new expectation:** `src/agents/pi-embedded-runner/cache-ttl.test.ts:31-36`
  - body: The mock at the top of this test file still returns `true` for `moonshot`, but the changed assertions now expect Moonshot eligibility to be `false`. These assertions will fail as written unless the mock or the expectation is corrected.
  - confidence: 0.94
- **[P2] Deduplicate aliases before maintaining sessions:** `src/gateway/session-cache-maintenance.ts:311-334`
  - body: The sweep iterates the raw store snapshot and acts on each key. Current session storage can contain legacy/canonical aliases for the same backing session, so one sweep can reset or compact the same session more than once unless it canonicalizes or dedupes before acting.
  - confidence: 0.79
