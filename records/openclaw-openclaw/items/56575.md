## Summary

Keep open: current main does not implement the new maintenance modes, and the PR still has blocking correctness issues plus a product/API overlap with the existing session reset lifecycle.

## What This Changes

The PR adds per-model `sessionCacheTtl` and `timeBasedContextCompact` settings, session activity/cache timestamps, and a gateway sweep that can compact or reset idle sessions.

## Best Possible Solution

Fold reset behavior into the existing `session.reset` lifecycle, keep any cache-specific addition focused on compaction or keep-warm behavior, and document/changelog the final supported config surface.

## Reproduction Assessment

Yes. The blocking defects are source-reproducible from the PR diff: the Moonshot mock contradicts changed assertions, reset TTL resolution is skipped for non-cache providers, and expiry can use an older cache-touch timestamp over newer activity.

## Solution Assessment

No. The proposed direction is not the best final shape yet because it adds a parallel reset mode beside current `session.reset`; the narrower maintainable path is to reuse the existing reset lifecycle and keep prompt-cache-specific logic separate.
