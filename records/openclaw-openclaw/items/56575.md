## Summary

Keep open: current main does not implement the proposed `sessionCacheTtl`/`timeBasedContextCompact` maintenance modes, and the PR still needs product/API direction, contributor real behavior proof, and fixes for source-level review blockers before merge.

## What This Changes

The PR adds per-model session cache TTL and explicit compact/reset maintenance modes, session activity/cache timestamps, and a gateway maintenance sweep.

## Best Possible Solution

Settle the session-maintenance API boundary first, reuse the existing idle reset lifecycle for reset behavior, keep prompt-cache maintenance explicitly opt-in and documented, then land a narrower implementation with focused tests and real behavior proof.

## Reproduction Assessment

Yes for the review blockers: PR-head source inspection shows the Moonshot mock/assertion contradiction, reset TTL provider gating, stale cache-touch preference, and raw alias iteration. The full gateway maintenance behavior was not live-tested.

## Solution Assessment

No. The PR is not the best solution as-is because it adds a parallel reset policy beside the existing `session.reset` lifecycle and still has blocking implementation/test defects; the safer path is product/API review followed by a narrower patch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Align the Moonshot mock with the new assertions:** `src/agents/pi-embedded-runner/cache-ttl.test.ts:31-36`
  - body: The mock still returns `true` for Moonshot while these changed assertions expect `false`. Since `isCacheTtlEligibleProvider` returns a defined plugin eligibility result directly, these assertions fail at runtime.
  - confidence: 0.96
- **[P2] Resolve reset TTLs before cache eligibility gating:** `src/agents/pi-embedded-runner/cache-ttl.ts:170-172`
  - body: `resolveCacheTtlMs` returns `null` before reading the configured mode and TTL when the provider is not prompt-cache eligible. That makes configured `reset` mode silently do nothing for non-cache providers even though reset only needs idle timestamps.
  - confidence: 0.91
- **[P2] Use the newest activity timestamp for expiry:** `src/gateway/session-cache-maintenance.ts:53-57`
  - body: `resolveCacheTouchAt` prefers `lastCacheTouchAt` over `lastAssistantMessageAt`. If maintenance is re-enabled after assistant turns continued, an old cache-touch timestamp can cause a recently active session to reset or compact immediately.
  - confidence: 0.84
- **[P2] Deduplicate maintenance by canonical session key:** `src/gateway/session-cache-maintenance.ts:311`
  - body: The sweep acts on raw `Object.entries(store)` keys. If the store contains both canonical and legacy aliases for one session, the same stale snapshot can trigger duplicate reset or compaction actions in a single pass.
  - confidence: 0.8
