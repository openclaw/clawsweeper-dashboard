## Summary

Keep open: this PR adds a real new session-maintenance config/runtime surface that is not on current main, but the current branch still has source-proven blockers and lacks after-fix real behavior proof.

## What This Changes

The branch adds per-model `sessionCacheTtl` and `timeBasedContextCompact` settings, session activity/cache timestamps, and a gateway sweep that can compact or reset idle sessions.

## Best Possible Solution

Reconcile prompt-cache maintenance with the existing `session.reset` lifecycle, keep any maintenance mode opt-in and documented, fix the source blockers, and require redacted runtime proof before merge.

## Reproduction Assessment

Yes for the review blockers: source inspection of the PR head shows the contradictory Moonshot mock/assertion, reset TTL provider gating, and stale cache-touch preference. The new maintenance behavior itself was not live-tested in this read-only review.

## Solution Assessment

No. The PR adds a parallel reset mode beside existing `session.reset` and still has blocking defects; the narrower maintainable path is to reuse the current reset lifecycle and isolate prompt-cache maintenance policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Align the Moonshot mock with the new assertions:** `src/agents/pi-embedded-runner/cache-ttl.test.ts:31-36`
  - body: The mock still returns `true` for Moonshot while these changed assertions expect `false`. Since `isCacheTtlEligibleProvider` returns a defined plugin eligibility result directly, these assertions fail at runtime.
  - confidence: 0.96
- **[P2] Resolve reset TTLs before cache eligibility gating:** `src/agents/pi-embedded-runner/cache-ttl.ts:170-172`
  - body: `resolveCacheTtlMs` returns `null` before reading the configured mode and TTL when the provider is not prompt-cache eligible. That makes configured `reset` mode a no-op for non-cache providers even though reset only needs idle timestamps.
  - confidence: 0.91
- **[P2] Use the newest activity timestamp for expiry:** `src/gateway/session-cache-maintenance.ts:53-57`
  - body: `resolveCacheTouchAt` prefers `lastCacheTouchAt` over `lastAssistantMessageAt`. If maintenance is re-enabled after assistant turns continued, an old cache-touch timestamp can cause an active session to reset or compact immediately.
  - confidence: 0.84
