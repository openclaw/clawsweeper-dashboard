## Summary

Keep open: current main does not implement the PR’s explicit `sessionCacheTtl` / `timeBasedContextCompact` maintenance surface, and the open PR still has blocking source-level defects plus no real behavior proof.

## What This Changes

The PR adds per-model time-based session maintenance settings, session activity/cache timestamps, and gateway sweeps for idle compaction or reset modes.

## Best Possible Solution

Settle the session-maintenance API boundary first, integrate reset behavior with the existing `session.reset` / `lastInteractionAt` lifecycle, and keep cache compaction as a documented opt-in prompt-cache feature with focused tests and real behavior proof.

## Reproduction Assessment

Yes for the review blockers: PR-head source inspection shows the Moonshot mock/assertion contradiction, provider eligibility gating for reset TTLs, and stale cache-touch preference. The actual maintenance sweep behavior was not live-tested, and the contributor has not provided real after-fix proof.

## Solution Assessment

No. The current PR is not the best solution as-is because it adds a separate reset policy beside the existing session reset contract and still has blocking implementation/test defects; the safer path is product/API review followed by a narrower implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Align the Moonshot mock with the new assertions:** `src/agents/pi-embedded-runner/cache-ttl.test.ts:31-36`
  - body: The mock still returns `true` for Moonshot while these changed assertions expect `false`. Since `isCacheTtlEligibleProvider` returns a defined plugin eligibility result directly, these assertions fail at runtime.
  - confidence: 0.96
- **[P2] Resolve reset TTLs before cache eligibility gating:** `src/agents/pi-embedded-runner/cache-ttl.ts:170-172`
  - body: `resolveCacheTtlMs` returns `null` before reading the configured mode and TTL when the provider is not prompt-cache eligible. That makes configured `reset` mode silently do nothing for non-cache providers even though reset only needs idle timestamps.
  - confidence: 0.91
- **[P2] Use the newest activity timestamp for expiry:** `src/gateway/session-cache-maintenance.ts:53-57`
  - body: `resolveCacheTouchAt` prefers `lastCacheTouchAt` over `lastAssistantMessageAt`. If maintenance is re-enabled after assistant turns continued, an old cache-touch timestamp can cause a recently active session to reset or compact immediately.
  - confidence: 0.84
