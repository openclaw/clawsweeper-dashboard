## Summary

Keep open: current main does not implement the proposed mode surface, and the PR still has blocking source-reproducible defects plus product/API overlap with existing session reset semantics.

## What This Changes

The PR adds per-model `sessionCacheTtl` and `timeBasedContextCompact` settings, session activity/cache timestamps, and a gateway sweep that can compact or reset idle sessions.

## Best Possible Solution

Reconcile the feature with the existing `session.reset` lifecycle, then land a documented and tested cache-specific maintenance path with changelog coverage.

## Reproduction Assessment

Yes. The blocking PR defects are source-reproducible from the diff: the Moonshot mock contradicts changed assertions, reset TTL resolution is skipped for non-cache providers, and expiry can use an older cache-touch timestamp over newer assistant activity.

## Solution Assessment

No. The proposed shape is not the best final solution yet because it adds a parallel reset mode beside the current `session.reset` lifecycle; the safer path is to reuse existing reset semantics and keep prompt-cache-specific maintenance separate.
