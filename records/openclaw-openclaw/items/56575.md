## Summary

Keep this PR open. Current main does not implement the new `sessionCacheTtl` / `timeBasedContextCompact` maintenance surface, and the branch still contains useful unique work, but it adds a new config/product policy and has blocking source-level review findings plus no real behavior proof.

## What This Changes

The branch adds per-model `sessionCacheTtl` and `timeBasedContextCompact` settings, session activity/cache timestamps, and gateway maintenance sweeps/tests for compact and reset modes.

## Best Possible Solution

Decide the session-maintenance product/API boundary first, then either integrate cache-aware maintenance with the existing reset lifecycle or split the prompt-cache compaction/timestamp pieces into a narrower, documented change.

## Reproduction Assessment

Yes for the review blockers: PR-head source inspection shows the contradictory Moonshot mock/assertions, provider eligibility gating for reset TTLs, and stale cache-touch preference. The new maintenance behavior itself was not live-tested in this read-only review.

## Solution Assessment

No. The current PR is not the best solution as-is because it adds a separate reset mode beside the existing `session.reset` contract and still has blocking defects; the safer path is to settle reset ownership, then implement cache maintenance narrowly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Align the Moonshot mock with the new assertions:** `src/agents/pi-embedded-runner/cache-ttl.test.ts:31-36`
  - body: The mock still returns `true` for Moonshot while these changed assertions expect `false`. Since `isCacheTtlEligibleProvider` returns a defined plugin eligibility result directly, these assertions fail at runtime.
  - confidence: 0.96
- **[P2] Resolve reset TTLs before cache eligibility gating:** `src/agents/pi-embedded-runner/cache-ttl.ts:170-172`
  - body: `resolveCacheTtlMs` returns `null` before reading the configured mode and TTL when the provider is not prompt-cache eligible. That makes configured `reset` mode a no-op for non-cache providers even though reset only needs idle timestamps.
  - confidence: 0.91
- **[P2] Use the newest activity timestamp for expiry:** `src/gateway/session-cache-maintenance.ts:53-57`
  - body: `resolveCacheTouchAt` prefers `lastCacheTouchAt` over `lastAssistantMessageAt`. If maintenance is re-enabled after assistant turns continued, an old cache-touch timestamp can cause an active session to reset or compact immediately.
  - confidence: 0.84
