## Summary

Keep this PR open, but it is not merge-ready. The branch still adds a real missing maintenance capability, while current main now has overlapping session reset semantics and the PR has concrete correctness defects that need reconciliation before merge.

## What This Changes

The PR adds per-model time-based `compact`/`reset` session maintenance settings, session activity/cache timestamps, a gateway maintenance sweep, Anthropic defaults, and related tests.

## Best Possible Solution

Rework the branch around current `session.reset` lifecycle semantics, keep only cache-aware idle compaction where it adds value, and document/changelog the final supported config surface before merge.

## Reproduction Assessment

Yes. The blocking defects are reproducible by source inspection of the PR diff: the Moonshot test mock contradicts the changed assertions, reset mode depends on cache-eligible TTL resolution, and expiry can use an older cache-touch timestamp over newer activity.

## Solution Assessment

No. The proposed implementation is not yet the best shape because it adds a parallel reset mode beside current `session.reset`; the safer direction is to reuse the existing lifecycle policy for resets and keep provider/cache-specific logic limited to compaction or keep-warm behavior.
