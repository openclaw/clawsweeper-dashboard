## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the PR's cache-aware idle compaction sweep and new per-model maintenance mode, while the PR head has blocking correctness issues and overlaps with newer main-branch session reset and cache-TTL behavior that it needs to reconcile before maintainer review can finish.

## What This Changes

The PR adds per-model `sessionCacheTtl` and `timeBasedContextCompact` settings, session activity/cache timestamps, a gateway maintenance sweep for idle compact/reset modes, Anthropic defaults, and related tests.

## Best Possible Solution

Rebase the branch onto current main, keep any cache-aware idle compaction behavior that still fits, align reset behavior with the existing `session.reset` lifecycle, preserve current provider cache eligibility contracts, and add docs/changelog coverage before merge.

## Reproduction Assessment

Yes. The PR defects are reproducible by source inspection: the Moonshot test mock contradicts the changed assertions, the reset mode cannot pass the sweep for non-cache providers because TTL resolution returns `null`, and the expiry baseline can prefer an older cache-touch timestamp over newer assistant activity.

## Solution Assessment

No. The current PR direction needs adjustment because it adds a parallel reset mode beside current `session.reset` and couples reset TTL resolution to prompt-cache eligibility; the narrower maintainable path is to reuse current session lifecycle semantics and make only the cache-compaction behavior provider/cache aware.
