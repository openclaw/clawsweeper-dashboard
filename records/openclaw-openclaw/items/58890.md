## Summary

Current main closes the reported availability failure: package updates launched from the managed gateway service now exit before package replacement, service refresh, or launchd activation can run, so the piped auto-update child can no longer reach the bootout-before-handoff crash path. The broader question of making package auto-update apply safely from a managed service is now tracked separately by the newer auto-update no-op report and its open PR.

## What This Changes

The issue reports that macOS auto-update can run the updater as a piped child of the gateway and then remove the LaunchAgent before a detached restart handoff exists, leaving the service unloaded.

## Best Possible Solution

Keep the shipped guard that prevents destructive package replacement from inside the managed gateway process, and handle any future safe auto-install design through the newer auto-update contract work rather than reopening this launchd crash path.

## Reproduction Assessment

Yes. The old source-level path is high confidence, and current main now short-circuits it: the auto-update child inherits gateway service markers, then updateCommand exits before package install, service refresh, or launchd bootout can execute.

## Solution Assessment

Yes for this issue's availability failure. Refusing service-process package replacement is the narrow shipped safety fix; a future helper-based auto-install flow should be designed under the newer auto-update no-op issue rather than by restoring the unsafe handoff ordering.
