## Summary

Keep this open. Current main still contains the core source pattern from the report: the gateway auto-updater launches a piped child, the package update restart flow refreshes the LaunchAgent before spawning the prepared restart script, and the install path still bootouts the LaunchAgent before bootstrap.

## What This Changes

The issue asks OpenClaw to make macOS built-in auto-update survive a gateway-owned LaunchAgent refresh where bootout closes the parent-held stdout/stderr pipes before restart handoff completes.

## Best Possible Solution

Make the auto-update restart handoff service-safe before any bootout-producing refresh runs, or move the auto-update child fully outside the live gateway-owned process/pipe tree, with focused unit coverage plus redacted macOS launchd proof.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection still shows the reported pipe capture and bootout-before-restart ordering on current main, but current EPIPE and launchd recovery hardening mean a macOS launchd run is needed to prove the exact crash path.

## Solution Assessment

No. The current partial launchd repairs are useful, but they do not resolve the auto-update subprocess pipe dependency or the refresh-before-restart ordering; maintainers should choose the service-safe handoff strategy before implementation.
