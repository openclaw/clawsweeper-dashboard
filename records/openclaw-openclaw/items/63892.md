## Summary

Keep open: current main still returns early on fresh persisted token totals before the token-threshold compaction check can run, and this issue is already paired with open implementation PRs for the same bug. The original `compactionCount` latch theory is corrected by source inspection; the remaining bug is the fresh-token fast path in `runPreflightCompactionIfNeeded`.

## What This Changes

Ensure proactive preflight compaction re-evaluates fresh persisted token totals on later turns so long-running sessions compact again before context overflow.

## Best Possible Solution

Ship a focused preflight fix that evaluates fresh persisted totals against `contextWindow - reserveTokensFloor - softThresholdTokens`, preserves the below-threshold fast path, keeps transcript-byte compaction independent, and adds a regression test for `totalTokensFresh: true` with cached tokens above the threshold.

## Reproduction Assessment

Yes. A deterministic source-level reproduction is a session entry with `sessionId`, `totalTokensFresh: true`, `totalTokens` above the preflight threshold, and no active transcript-byte trigger; current main returns at `agent-runner-memory.ts:454` before `shouldRunPreflightCompaction` runs.

## Solution Assessment

Yes. Gating or removing the fresh-token early return so fresh totals still reach the existing threshold helper is the narrowest maintainable fix; changing `compactionCount` semantics would address the original hypothesis, not the verified root cause.
