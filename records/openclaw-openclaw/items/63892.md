- `impact:session-state`: The report concerns persisted session token freshness, compaction checkpoints, and long-running context state.

## Summary

Keep open: current `main` and the latest tagged release still return early from the fresh persisted-token preflight path before evaluating the compaction threshold. The issue discussion corrected the original `compactionCount` theory to this fresh-token fast-path bug, and an open closing PR already tracks the focused fix.

## What This Changes

The issue reports that proactive preflight compaction should re-fire on later threshold crossings after a session has already compacted once.

## Best Possible Solution

Land one focused, rebased fix that lets fresh persisted totals at or above the computed threshold reach the existing preflight compaction gate while preserving stale transcript fallback and transcript-byte compaction behavior.

## Reproduction Assessment

Yes, at source level. A non-heartbeat, non-CLI run with `totalTokensFresh: true`, a positive persisted total above the threshold, and no active transcript-byte trigger returns before `shouldRunPreflightCompaction` runs.

## Solution Assessment

Yes for the corrected direction in the discussion and linked PRs: evaluate the existing threshold gate for fresh persisted totals before returning. Changing `compactionCount` semantics would address the original hypothesis, not the confirmed root cause.
