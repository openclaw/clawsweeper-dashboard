## Summary

Keep open: current main still returns from the fresh-token preflight path before token-threshold compaction is evaluated, and https://github.com/openclaw/openclaw/pull/64384 remains an open closing PR for this issue while https://github.com/openclaw/openclaw/pull/66716 overlaps the same fix area.

## What This Changes

The issue asks for proactive compaction to re-fire after a prior compaction when fresh persisted token totals grow back above the configured threshold.

## Best Possible Solution

Land or replace one focused preflight fix that lets fresh persisted totals at or above the computed token threshold reach the existing compaction gate, with regression coverage for the fresh above-threshold/no transcript-byte-trigger case.

## Reproduction Assessment

Yes. A deterministic source reproduction is a non-heartbeat, non-CLI session entry with `totalTokensFresh: true`, a positive `totalTokens` above `contextWindow - reserveTokensFloor - softThresholdTokens`, and no active transcript-byte trigger; current main returns before `shouldRunPreflightCompaction` runs.

## Solution Assessment

Yes for the corrected direction in the discussion and linked PRs: evaluate the existing threshold gate for fresh persisted totals before returning. Changing `compactionCount` semantics would address the original hypothesis, not the confirmed root cause.
