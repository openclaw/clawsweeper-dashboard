## Summary

Keep open: current main still short-circuits fresh persisted token totals before the token-threshold compaction check, and an open PR with closing syntax already tracks this issue while another open PR overlaps the same fix area.

## What This Changes

The issue asks for proactive compaction to re-fire on later turns after fresh persisted token totals grow back above the configured threshold.

## Best Possible Solution

Land or replace one focused preflight fix that lets fresh persisted totals at or above the computed threshold reach the existing compaction gate, with regression coverage for that path.

## Reproduction Assessment

Yes, at source level. A non-heartbeat, non-CLI session with `totalTokensFresh: true`, positive `totalTokens` above `contextWindow - reserveTokensFloor - softThresholdTokens`, and no active transcript-byte trigger returns before the threshold helper runs on current main.

## Solution Assessment

Yes for the corrected direction in the discussion and linked PRs: evaluate the existing threshold gate for fresh persisted totals before returning. Changing `compactionCount` semantics would address the original hypothesis, not the confirmed root cause.
