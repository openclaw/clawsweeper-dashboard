- `impact:session-state`: The report concerns persisted session token freshness, compaction checkpoints, and long-running context state.

## Summary

Keep open: current main and the latest tagged release still short-circuit the fresh persisted-token preflight path before evaluating the compaction threshold. The discussion corrected the original `compactionCount` theory to a fresh-token early-return bug, and https://github.com/openclaw/openclaw/pull/64384 already tracks the focused fix with closing syntax.

## What This Changes

The issue reports that proactive preflight compaction stops re-firing on long sessions after cached token totals become fresh, forcing later compactions through overflow retry instead.

## Best Possible Solution

Land one focused, rebased fix that lets fresh persisted totals at or above the computed threshold reach the existing preflight compaction gate while preserving transcript-byte compaction and stale transcript fallback behavior.

## Reproduction Assessment

Yes, at source level. A non-heartbeat, non-CLI run with `totalTokensFresh: true`, positive `totalTokens` above the computed threshold, and no active transcript-byte trigger returns before `shouldRunPreflightCompaction` runs.

## Solution Assessment

Yes for the corrected direction in the discussion and linked PRs: evaluate the existing threshold gate for fresh persisted totals before returning. Changing `compactionCount` semantics would address the original hypothesis, not the confirmed root cause.
