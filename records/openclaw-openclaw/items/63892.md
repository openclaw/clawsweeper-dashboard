## Summary

Keep open: current main still short-circuits the fresh persisted-token preflight path before evaluating the token threshold, and open PRs already track the same focused fix.

## What This Changes

The issue reports that proactive compaction stops re-firing after a successful checkpoint because fresh persisted token totals bypass the preflight threshold check.

## Best Possible Solution

Keep this canonical issue open until one focused, rebased fix lands so fresh persisted totals at or above the computed threshold reach the existing preflight compaction gate while preserving transcript-byte compaction and stale transcript fallback behavior.

## Reproduction Assessment

Yes, at source level. A non-heartbeat, non-CLI session with `totalTokensFresh: true`, positive `totalTokens` above `contextWindow - reserveTokensFloor - softThresholdTokens`, and no active transcript-byte trigger returns before the token threshold helper runs on current main.

## Solution Assessment

Yes for the corrected direction in the discussion and linked PRs: evaluate the existing threshold gate for fresh persisted totals before returning. Changing `compactionCount` semantics would address the original hypothesis, not the confirmed root cause.
