## Summary

Keep open: current main still short-circuits the fresh persisted-token preflight path before evaluating the token threshold, and an open PR with closing syntax already tracks the focused fix. The discussion corrected the original compactionCount hypothesis to the fresh-token early-return bug, which matches the source path.

## What This Changes

The issue asks for proactive preflight compaction to re-fire on later cycles when fresh persisted token totals cross the configured threshold after an earlier compaction.

## Best Possible Solution

Keep this issue open until one focused, rebased fix lands that lets fresh persisted totals at or above the computed threshold reach the existing preflight compaction gate while preserving transcript-byte compaction and stale transcript fallback behavior.

## Reproduction Assessment

Yes, at source level. A non-heartbeat, non-CLI run with `totalTokensFresh: true`, positive `totalTokens` above the computed threshold, and no active transcript-byte trigger returns before `shouldRunPreflightCompaction` runs on current main.

## Solution Assessment

Yes for the corrected direction in the discussion and linked PRs: evaluate the existing threshold gate for fresh persisted totals before returning. Changing `compactionCount` semantics would solve the original hypothesis, not the confirmed root cause.
