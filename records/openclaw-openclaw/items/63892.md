## Summary

Keep open: current main still returns from the fresh persisted-token preflight path before evaluating the token threshold, and open fix PRs already track the same root cause.

## What This Changes

The issue asks for proactive compaction to re-fire on later turns when fresh persisted token totals grow back over the compaction threshold after a checkpoint.

## Best Possible Solution

Keep this canonical issue open until one focused fix lands so fresh persisted totals at or above the computed threshold reach the existing preflight compaction gate with regression coverage.

## Reproduction Assessment

Yes, at source level. A non-heartbeat, non-CLI session with `totalTokensFresh: true`, positive `totalTokens` above `contextWindow - reserveTokensFloor - softThresholdTokens`, and no active transcript-byte trigger returns before the threshold helper runs on current main.

## Solution Assessment

Yes for the corrected direction in the discussion and linked PRs: evaluate the existing token threshold for fresh persisted totals before returning. Changing `compactionCount` semantics would address the original hypothesis, not the confirmed root cause.
