## Label Justifications

- `P1`: The issue affects normal Control UI chat input for CJK IME users, a core user-facing workflow for affected users.
- `impact:other`: The concrete impact is webchat input responsiveness and usability for CJK IME users, which does not fit the more specific owned impact labels.

## Summary

Keep open: current main and v2026.6.6 still resize the Control UI composer synchronously on every textarea input, so the reported CJK IME resize-lag path is not implemented. The active CJK composition PR is related, but it still calls the same synchronous resize path during composing input and is not a merged, safe canonical fix for this specific lag report.

## What This Changes

The issue requests coalescing Control UI webchat textarea auto-resize work with `requestAnimationFrame` to reduce Chinese/CJK IME typing lag.

## Best Possible Solution

Add one composer resize scheduler that coalesces `scrollHeight` reads and height writes with `requestAnimationFrame`, cover the behavior with focused tests, and validate real CJK IME typing in Chrome and Safari.

## Reproduction Assessment

Yes, at source level: current `handleInput` always calls `adjustTextareaHeight`, and that helper performs a synchronous `scrollHeight` read plus height writes on every input. I did not run a live macOS browser IME session, so this is not directly reproduced here.

## Solution Assessment

Unclear whether a resize-only patch solves every CJK symptom because composition draft synchronization is separately tracked by https://github.com/openclaw/openclaw/pull/91087. For this specific lag report, coalescing the resize path is the narrow maintainable fix.
