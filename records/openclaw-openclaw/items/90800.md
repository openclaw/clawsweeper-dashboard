## Summary

Keep open: current main and v2026.6.6 still resize the Control UI chat textarea synchronously on every input event, and the open related IME composition PR does not land or prove the requested resize coalescing fix.

## What This Changes

The issue asks OpenClaw to remove Chinese/CJK IME typing lag in the Control UI webchat composer by batching textarea auto-resize work instead of forcing layout on every input event.

## Best Possible Solution

Land a focused Control UI composer resize-coalescing fix with regression coverage and live CJK IME proof, either folded into a repaired narrow IME PR or shipped as a separate follow-up.

Yes, at source level: current main still performs a synchronous textarea resize on every input, and the reporter gives macOS Chrome/Safari Chinese IME steps. I did not run a live IME session, so this is not a directly reproduced runtime measurement.

## Solution Assessment

Yes, the maintainable direction is to coalesce composer resize work at the textarea boundary while preserving the local-draft and composition contracts. The current open composition PR is adjacent context, but it does not by itself implement the requested resize batching.
