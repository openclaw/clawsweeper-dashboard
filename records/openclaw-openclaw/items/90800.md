- `impact:other`: The concrete impact is webchat input responsiveness and IME usability, which does not fit the more specific owned impact labels.

## Summary

Keep open: current main and v2026.6.6 still call the synchronous textarea resize helper on every Control UI chat input, and the open CJK composition replacement PR does not coalesce that resize path.

## What This Changes

Batch Control UI WebChat textarea auto-resize work so Chinese/CJK IME composition typing no longer lags from synchronous `scrollHeight` layout on every input.

## Best Possible Solution

Coalesce the existing composer resize measurement/write with an owned rAF scheduler while preserving the local draft mirror and composition semantics, then prove smooth CJK IME input in Chrome and Safari.

## Reproduction Assessment

Yes, at source level: the current and released composer input handler synchronously resizes on every textarea input, and the reporter gives macOS Chrome/Safari Chinese IME steps. I did not run a live IME session, so this is not marked directly reproduced.

## Solution Assessment

Yes on direction: the narrow maintainable fix belongs in the composer resize scheduling path, not gateway or browser configuration. It should be reviewed against the open composition PR so batching resize does not duplicate or conflict with draft/composition event ownership.
