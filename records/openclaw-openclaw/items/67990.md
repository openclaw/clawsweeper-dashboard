## Summary

Keep open. Current main and v2026.5.18 include several backup hardening slices, but the central design is still incomplete: no user-configurable exclusion policy, native restore/import/full-mode surface, or service-aware gateway quiesce lifecycle exists, and the remaining work is split across open product/security-sensitive threads.

## What This Changes

The issue proposes a broader backup architecture with configurable exclusions, symlink-aware traversal, service-aware archiving, tar.gz output, and distinct config-only/full backup modes.

## Best Possible Solution

Keep this as the broad backup-design tracker while maintainers split the remaining work into reviewed slices for exclusion policy, restore/import safety, archive verification, live-file resilience, and gateway quiesce semantics.

## Risks / Open Questions

- Configurable exclusions reduce archive size but can also omit recovery-critical or sensitive state unless protected-path, manifest, verification, and docs policy are designed together.
- Native restore/import would write credentials-adjacent state, cron data, sessions, and workspaces, so path containment, permissions, rollback, and partial-failure behavior need explicit design.
- Service-aware gateway stop/start or quiesce behavior can disrupt managed installs and needs maintainer operations policy before implementation.
- Existing open items cover slices of the proposal, but no single canonical issue found in this pass covers the service-aware quiesce lifecycle.
