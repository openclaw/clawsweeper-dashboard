## Summary

Keep open. Current main and the latest shipped backup docs include several hardening slices, but the broader design is still incomplete: there is no user-configurable exclusion policy, native restore/import command, or service-aware gateway quiesce lifecycle, and those remaining pieces are product/security-sensitive.

## What This Changes

The issue proposes a broader backup architecture with service-aware archiving, configurable exclusions, symlink-safe traversal, tar.gz output, and config-only/full backup modes.

## Best Possible Solution

Keep this as the broad backup-design tracker while maintainers split exclusion policy, restore/import safety, archive verification, and gateway quiesce semantics into reviewed slices built on the current TypeScript backup primitives.

## Reproduction Assessment

Not applicable as a single bug reproduction; this is a design proposal. Source, docs, tests, release-tag inspection, and live related GitHub state give high-confidence proof that the remaining feature surfaces are absent or only partially covered.

## Solution Assessment

## Risks / Open Questions

- Configurable exclusions can make archives smaller while also making restores incomplete unless protected paths, manifest semantics, verification, and docs are designed together.
- Native restore/import would write credentials-adjacent state, cron data, sessions, and workspaces, so path containment, permissions, rollback, and partial-failure rules need explicit design.
- Service-aware stop/start or quiesce behavior can disrupt managed gateway installs and needs maintainer UX/operations policy before implementation.
- Open related work, especially https://github.com/openclaw/openclaw/pull/44288 and https://github.com/openclaw/openclaw/issues/13616, covers only slices of this proposal and should be coordinated rather than treated as complete.
