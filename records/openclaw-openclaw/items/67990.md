## Summary

Keep open. Current main has shipped useful backup primitives and the narrower installed-plugin node_modules pruning, but the central proposal is still not implemented: there is no configurable exclusion policy, no service-aware gateway quiesce lifecycle, no general live-file retry/skip behavior on main, and no restore/import/full-mode surface.

## What This Changes

The issue proposes a broader backup architecture with configurable exclusion rules, symlink-aware traversal, service-aware archiving, tar.gz output, and config-only/full modes.

## Best Possible Solution

Keep this as the broader backup design tracker and split it into maintainer-approved slices: a documented exclusion policy/config surface, general archive filtering and symlink/volatile-file semantics, an explicit quiesce/service lifecycle if desired, and restore/import disaster-recovery behavior coordinated with #13616, #54242, #67417, and #72251.

## Reproduction Assessment

Not applicable as a single reproduction because this is a design proposal. Source/docs/test inspection gives high-confidence proof that the requested configurable exclusions, service lifecycle, live-file resilience, and restore/full-mode surfaces are still absent, while the related race reports carry their own reproduction paths.

## Solution Assessment

No, not as one automated code change or as a direct port of the standalone implementation. The maintainable path is to preserve the current TypeScript backup primitives and have maintainers define the exclusion config, quiesce semantics, and restore/import contract before implementation.
