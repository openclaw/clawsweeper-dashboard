## What This Changes

The issue reports that the bundled Nostr channel repeatedly restarts after connecting because its DM subscription request is malformed and the provider does not stay running.

## Best Possible Solution

Land a narrow Nostr plugin fix that passes a plain `Filter` to `nostr-tools`, keeps `startNostrGatewayAccount` pending until abort, closes/destroys relay resources on shutdown, and adds regression coverage for REQ serialization and channel lifecycle.

## Reproduction Assessment

Yes at source level, but not with a live relay in this read-only review. Current main still passes an array into a `nostr-tools` API that expects one `Filter`, and the Nostr `startAccount` promise resolves into the generic auto-restart path.

## Solution Assessment

Yes. The best fix is a plugin-owned repair that aligns with the pinned dependency contract and the existing long-lived channel lifecycle pattern, without changing generic gateway policy.

## Repair Work Prompt

Fix the open Nostr restart-loop/DM subscription bug. In the Nostr plugin, make `startNostrBus` pass a plain Nostr `Filter` object to `SimplePool.subscribeMany` instead of an array and remove the unsafe cast; add a regression test that proves the subscription filter passed to `nostr-tools` is not nested and, if practical, that the serialized REQ has a plain filter object. In `startNostrGatewayAccount`, follow the existing long-lived channel pattern so startup stays pending until `ctx.abortSignal` aborts, then close the bus and clear active state; add lifecycle coverage similar to other channel tests. Also close/destroy underlying SimplePool relay resources on bus shutdown so restarts do not orphan connections. Keep the fix inside `extensions/nostr`; do not change generic gateway restart policy, plugin SDK contracts, dependency versions, or unrelated Nostr profile/security behavior. Add/update focused tests and a `CHANGELOG.md` fix entry for the user-facing channel repair.
