## Summary

Keep open: current main still has a narrow WhatsApp plugin bug where LID-to-phone identity resolution exists, but the command authorization and group owner-bypass paths do not receive the account auth directory needed to use it.

## What This Changes

Accept owner-only WhatsApp group control commands when Baileys reports the owner sender as a LID JID that maps back to the configured owner.

## Best Possible Solution

Thread the WhatsApp account authDir through command authorization and group owner-bypass identity resolution, then add regressions for LID-only group senders using owner control commands.

## Reproduction Assessment

Yes, at source level: construct a group WebInboundMessage whose sender is only a LID JID plus an authDir reverse mapping to an allowlisted owner, and current command auth has no authDir path to resolve it. I did not run a live WhatsApp transport.

## Solution Assessment

Yes. Passing the existing account authDir into the existing WhatsApp identity helpers is the narrowest maintainable fix and avoids new config, policy, or core behavior.
