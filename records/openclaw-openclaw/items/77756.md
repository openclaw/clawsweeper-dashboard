## Summary

Keep open: current main still has a narrow WhatsApp plugin authorization gap where LID-to-phone identity resolution is available but not threaded into the command authorization and group owner-bypass paths.

## What This Changes

Accept WhatsApp group control commands from the configured owner when Baileys reports the group sender as a resolvable LID JID.

## Best Possible Solution

Thread the WhatsApp account `authDir` through command authorization and group owner-bypass identity resolution, then add focused regressions for LID-only group senders using `/new` and `/stop`.

## Reproduction Assessment

Yes, at source level: current main can be driven with a group `WebInboundMessage` whose sender is only `456@lid` plus an authDir reverse mapping to an allowlisted owner, and the current command auth API has no way to use that authDir. I did not run the live WhatsApp transport.

## Solution Assessment

Yes, the narrow maintainable fix is to pass the existing account auth directory into the WhatsApp identity helpers at the command and owner-bypass call sites. This keeps the behavior plugin-owned and reuses the existing LID mapping contract instead of adding new policy or config.
