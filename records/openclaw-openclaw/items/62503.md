## Summary

Keep open: current main still has no devcontainer surface, but the latest PR head has blocking security/docs regressions and only stale real setup proof from before the latest head.

## What This Changes

The PR adds a VS Code devcontainer with a Dockerfile, post-create bootstrap, install docs/nav/labeler updates, zh-CN glossary changes, and file-content tests.

## Best Possible Solution

Keep the devcontainer direction, but run dependency installation without forwarded secrets, preserve the existing glossary while adding the new term, fix the Copilot wording, and require latest-head Docker plus Podman proof.

## Reproduction Assessment

Yes for the blocking review findings: the PR head source shows container-wide token forwarding, install-before-auth bootstrap ordering, glossary replacement, and stale Copilot wording. The devcontainer feature itself is not a bug reproduction path because current main lacks that surface.

## Solution Assessment

No. A devcontainer is a reasonable contributor-DX feature, but this implementation should not expose host secrets to install-time scripts or replace unrelated glossary entries.
