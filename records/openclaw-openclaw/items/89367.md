## Summary

Keep open: the wait-layer fix direction is still useful, but the patch is not merge-ready because the new fallback forwards only `timeoutPhase`-attributed hard timeouts while current terminal and SDK wait contracts also treat `status: timeout` plus `providerStarted: true` as a hard timeout. The external-PR proof is still test output only, so live parent/subagent behavior remains unproven.

## What This Changes

The branch exports the hard-timeout phase predicate, updates `waitForAgentJob`'s fallback timer to forward pending hard-timeout snapshots, and adds in-process regression coverage for hard and soft timeout races.

## Best Possible Solution

Update the fallback to use the full terminal hard-timeout contract, add providerStarted-only regression coverage, then land this as the wait-layer slice while keeping no-output delivery tracked under the canonical issue.

## Reproduction Assessment

Yes, source-reproducible: current main stores pending timeout snapshots but the outer `waitForAgentJob` timer only checks pending errors before returning `null`. I did not run a live parent/subagent timeout in this read-only review.

## Solution Assessment

No. The PR is in the right wait-layer area, but the phase-only check is narrower than the current hard-timeout contract; the maintainable fix should reuse the terminal outcome contract or explicitly cover providerStarted-only timeout snapshots.
