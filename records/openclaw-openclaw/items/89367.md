## Summary

Keep open: this is a focused wait-layer fix for the pending sub-agent timeout race, but the latest provider-started fallback gate re-derives the terminal outcome contract and can forward non-hard restart timeout snapshots before the retry grace. The PR also still has only in-process Vitest output, not redacted real parent/subagent runtime proof.

## What This Changes

The PR exports a timeout-phase helper, updates `waitForAgentJob` to forward selected pending timeout snapshots from the outer fallback timer, and adds gateway regression coverage.

## Best Possible Solution

Use the shared terminal outcome classification for the fallback gate, add a regression for provider-started restart or other non-hard timeout snapshots staying correctable, then attach redacted real parent/subagent runtime proof before maintainer landing.

## Reproduction Assessment

Yes, source-level: current main still has the pending-timeout wait fallback gap, and the PR's new provider-started gate can be compared directly against the shared terminal outcome contract and tests. I did not run a fresh live parent/subagent timeout in this read-only review.

## Solution Assessment

No, not yet: the fallback timer is the right layer for the wait-snapshot race, but this implementation should call the shared terminal outcome classification instead of duplicating part of the hard-timeout condition. It also still needs real runtime proof before merge.
