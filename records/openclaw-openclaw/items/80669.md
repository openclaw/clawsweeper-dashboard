## Summary

Keep open: the group/channel `message_tool` default is intentional and documented, but current main still has a source-level gap where pure `messages.*` config changes are classified as dynamic/no-op and skipped before runtime state is refreshed, so the documented `automatic` workaround may require a restart. Changing the default to `automatic` is a group privacy/product decision, not conservative cleanup.

## What This Changes

The issue asks to make WhatsApp group auto-replies visible by default and reports that the current `messages.groupChat.visibleReplies: "automatic"` workaround may not apply until a gateway restart.

## Best Possible Solution

Keep the safe group/channel default unless maintainers explicitly change it, fix the no-op config reload path so `messages.*` dynamic reads refresh runtime state without a full restart, and use https://github.com/openclaw/openclaw/issues/80647 for the broader cross-channel private-final diagnostic or guard.

## Reproduction Assessment

Yes, at source level. Current main classifies `messages.*` as a no-op reload plan and returns before `onHotReload`, while the same runtime still reads `cfg.messages.groupChat.visibleReplies` from the runtime config used for WhatsApp dispatch.

## Solution Assessment

No for flipping the default automatically; that changes the documented group/channel privacy contract. The safer maintainable first fix is to refresh runtime state for `messages.*` no-op reloads and leave default-policy changes to maintainer review.
