## Summary

Close as duplicate/superseded: this is the shared group/channel `message_tool_only` visible-reply behavior, not a WhatsApp transport failure. The global default reversal was already proposed and closed in https://github.com/openclaw/openclaw/pull/77808, while the remaining diagnostic/fallback policy is tracked by https://github.com/openclaw/openclaw/issues/77320.

## What This Changes

The issue asks OpenClaw to default WhatsApp group auto-replies to automatic visible delivery instead of requiring `messages.groupChat.visibleReplies: "automatic"`.

## Best Possible Solution

Keep the documented private-by-default group/channel contract, route diagnostic or fallback policy through https://github.com/openclaw/openclaw/issues/77320, and use `messages.groupChat.visibleReplies: "automatic"` for operators who want legacy auto-posting.

## Reproduction Assessment

Yes, at source level: an unset WhatsApp group turn follows the shared group/channel resolver to `message_tool_only`, which suppresses automatic final delivery while the agent can still process the message. That reproduces the reported symptom, but it matches the current documented policy rather than a transport bug.

## Solution Assessment

No, not as a standalone global default flip: changing unset group/channel replies to `automatic` would reverse the documented privacy contract already preserved by merged tests and closed review in https://github.com/openclaw/openclaw/pull/77808. The safer remaining path is the canonical diagnostic/fallback discussion in https://github.com/openclaw/openclaw/issues/77320.
