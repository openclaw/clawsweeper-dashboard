## Summary

Keep open: the PR is on the right path and includes real behavior proof, but the new auth helper bypasses canonical auth-profile ordering and the docs overstate Grok web_search auth-profile support that the patch does not implement.

## What This Changes

The PR passes auth-profile helper callbacks into plugin tool resolution, updates xAI x_search/code_execution to register and execute with xAI auth profiles, and refreshes docs, tests, and changelog.

## Best Possible Solution

Revise the PR so xAI tool auth uses canonical auth-profile ordering/eligibility and keep docs scoped to implemented auth sources unless Grok web_search auth-profile support is also added.

## Reproduction Assessment

Yes, source-level: current main's xAI tool factories gate on config/env while manifest availability can pass through auth profiles, and the linked reporter comment confirms the same mismatch from an auth-profile setup.

## Solution Assessment

No. The direction is narrow, but the current implementation should honor resolveAuthProfileOrder semantics and should not document Grok web_search auth-profile support unless the code implements that path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Honor configured auth-profile order when resolving tool keys:** `src/agents/openclaw-plugin-tools.ts:81`
  - body: The new helper walks `listProfilesForProvider()` directly, so a user with `auth.order.xai` can still have an omitted or cooled-down xAI profile tried first for `x_search`/`code_execution`. That diverges from the canonical auth-profile resolver, where profiles excluded by explicit order are not silently used, and can send the tool request with the wrong credential even though model auth would choose the configured profile. Please route this through `resolveAuthProfileOrder()` or an equivalent ordered/eligible helper before resolving profile material.
  - confidence: 0.86
- **[P3] Keep Grok web_search docs scoped to implemented auth:** `docs/providers/xai.md:123-124`
  - body: These lines say the bundled Grok `web_search` provider can use the xAI auth profile, but the PR only threads auth helpers into plugin tools and current `executeXaiWebSearchProviderTool()` still resolves credentials from scoped config or `XAI_API_KEY`. Users with only `openclaw onboard --auth-choice xai-api-key` would still hit the missing-key path for Grok web_search, so either implement that runtime path too or keep this doc change limited to `x_search` and `code_execution`.
  - confidence: 0.82
