## Summary

Keep open. Current main still has the dry-run text bug: send and poll results can carry `dryRun: true` while `handledBy` remains `core`, so the formatter's `handledBy === "dry-run"` branch is not reached before normal success text. An open linked fix PR with closing syntax exists, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue asks for `openclaw message send --dry-run` pretty-text output to render a dry-run indicator instead of a successful send line.

## Best Possible Solution

Land or replace the linked fix so send and poll dry-runs are recognized before success formatting, with focused text-mode regression coverage while preserving the existing no-delivery dry-run guard.

## Reproduction Assessment

Yes. The current-main source path is enough to reproduce with high confidence: a dry-run send can reach the formatter with `dryRun: true` and `handledBy: "core"`, so the dry-run text branch is skipped.

## Solution Assessment

Yes. The narrow maintainable solution is to make the result contract or formatter key off dry-run state before rendering send/poll success output, and the open linked PR is an implementation candidate for that approach.
