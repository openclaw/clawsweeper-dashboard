## Summary

Keep open. Current main still preserves the safe dry-run delivery behavior, but the text formatter can still render send/poll dry-runs as successful sends because those results carry `dryRun: true` while `handledBy` remains `core`.

## What This Changes

The issue asks for `openclaw message send --dry-run` pretty-text output to show a dry-run preview instead of the normal success message.

## Best Possible Solution

Update the message pretty-text formatter or result contract so send and poll dry-runs are keyed from `dryRun: true` before any success summary, while preserving the existing outbound dry-run short-circuit.

## Reproduction Assessment

Yes, source-reproducible with high confidence: a send or poll dry-run reaches the formatter with `dryRun: true` and `handledBy: "core"`, so the current `handledBy === "dry-run"` guard cannot catch it before the success summary path. I did not run the CLI command in this read-only review.

## Solution Assessment

Yes: the narrowest maintainable fix is to make the formatter/result contract recognize `dryRun: true` for send and poll results before rendering success text, with focused regression coverage for text mode.
