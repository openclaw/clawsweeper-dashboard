Rework the change on current main so the embedded compaction dispatcher passes the current `compaction_start` event into the handler, start events expose a safe `reason`, end events expose `completed`, `hasResult`, `willRetry`, and sanitized `errorMessage`, and current compaction count, token snapshot, retry, liveness, hook, reconciliation, fallback, and UI behavior remain unchanged.

## Solution Assessment

No. The requested direction is good, but this exact patch is not the best solution until it is rebased onto the current `compaction_start` / `compaction_end` path, includes `hasResult`, and keeps routine successful lifecycle logging below `warn`.
