## Summary

Keep open: current main and v2026.6.10 still resolve compaction providers through the plugin registry only, MCP tools remain a separate runtime/tool surface, and the issue has maintainer discussion signal rather than a safe cleanup close.

## What This Changes

Request to let `agents.defaults.compaction.provider` reference an MCP server/tool so MCP summarization tools can serve as compaction engines without a plugin wrapper.

## Best Possible Solution

Keep the issue open for maintainer design review; if accepted, define a narrow MCP compaction adapter contract across config schema, compaction runtime, MCP runtime, security policy, docs, and tests, otherwise document the plugin-wrapper path as the supported boundary.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.10 only resolve compaction providers through the plugin registry, while MCP tools are discovered and called through a separate session MCP runtime. This is a missing capability rather than a failing bug repro.

## Solution Assessment

Unclear; the MCP adapter is plausible, but the best solution needs maintainer direction because it adds config surface and makes MCP tool execution part of the compaction/session-state boundary.
