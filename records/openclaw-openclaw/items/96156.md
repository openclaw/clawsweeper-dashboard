## Summary

Keep open: current main and the latest release still route compaction providers only through the plugin-registered provider API, while MCP tools remain a separate runtime/tool surface. The reopened discussion explicitly says this needs maintainer design review rather than ClawHub cleanup.

## What This Changes

The issue requests allowing `agents.defaults.compaction.provider` to reference an MCP server tool as the compaction engine instead of requiring a plugin wrapper.

## Best Possible Solution

Define a narrow MCP compaction adapter contract with config shape, transcript and SecretRef handling, tool-policy semantics, fallback behavior, docs, and focused tests; otherwise document the plugin wrapper path as the supported boundary.

## Reproduction Assessment

Yes for the missing capability: source inspection shows compaction provider lookup only resolves plugin-registered providers, while MCP tool execution lives on a separate session MCP runtime. This is source-reproducible rather than a failing bug repro.

## Solution Assessment

Unclear; the MCP adapter is plausible, but the best shape requires maintainer direction because it crosses config, plugin API, MCP tool execution, and session-state security boundaries.

## Risks / Open Questions

- The requested adapter would send compacted session transcript content to a configured MCP tool, so maintainers need to define security, consent, tool-policy, and failure boundaries first.
- This would add or change compaction configuration under `agents.defaults.compaction`, which is compatibility-sensitive and needs a deliberate config contract.
- The existing MCP tool runtime is model-turn oriented, while compaction is a background session-state path, so reusing that runtime without a design decision could create surprising policy or lifecycle behavior.
