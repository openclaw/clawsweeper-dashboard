## Label Justifications

- `P2`: This is a normal-priority feature request affecting long-session compaction and advanced MCP users without breaking existing default workflows.
- `impact:session-state`: The request directly changes how transcript compaction summaries and successor session context would be produced.
- `impact:security`: The design would route potentially sensitive session transcript content through configured MCP tools, making the security boundary maintainer-visible.

## Summary

Keep open: current main still selects compaction providers only from the plugin-registered provider registry while MCP tools live on a separate runtime surface, and the issue was reopened for maintainer design discussion rather than cleanup closure.

## What This Changes

Request to let `agents.defaults.compaction.provider` reference an MCP server/tool as the compaction engine instead of requiring a plugin wrapper registered with `registerCompactionProvider()`.

## Best Possible Solution

Keep this issue open for maintainer design review; if accepted, define a narrow MCP compaction adapter contract with config schema, docs, security/tool-policy behavior, and tests, otherwise document the plugin-wrapper path as the supported boundary.

## Reproduction Assessment

Yes for the missing capability: current source shows `provider` resolves only to a plugin-registered compaction provider, while configured MCP tools are materialized through a separate bundle-MCP runtime. This is source-reproducible rather than a failing bug repro.

## Solution Assessment

Unclear; the MCP adapter is plausible, but the best solution needs maintainer direction because it adds config/API surface and makes MCP tool execution part of the compaction/session-state boundary.
