## Summary

Keep open: current main and v2026.6.10 still route compaction providers only through plugin-registered `registerCompactionProvider()` implementations, while MCP servers are a separate runtime/tool surface. The request would add new config and session-transcript execution behavior, and the issue was reopened with a direct discussion signal rather than being a safe cleanup close.

## What This Changes

Request allowing `agents.defaults.compaction.provider` to reference an MCP server/tool as the compaction engine instead of requiring a plugin wrapper around `registerCompactionProvider()`.

## Best Possible Solution

Keep this open for maintainer design review; if accepted, define a narrow MCP compaction adapter with explicit config schema, docs, tests, transcript/security behavior, and fallback semantics, otherwise document the plugin-provider wrapper as the supported boundary.

## Reproduction Assessment

Yes for the missing capability: current main's strict config schema has no MCP compaction fields, and the runtime resolves `provider` only through `getCompactionProvider()`. This is source-reproducible rather than a failing bug reproduction.

## Solution Assessment

Unclear; the MCP adapter is plausible, but it is not proven to be the best boundary because it adds config and executes MCP tools over session transcript data. Maintainers need to choose direct MCP compaction, a plugin-wrapper path, or a narrower plugin/API extension.
