## Summary

Close as ClawHub/plugin-scope: the MCP compaction adapter is useful, but current OpenClaw policy keeps optional MCP/provider integrations out of core when plugin and ClawHub paths can carry them.

## What This Changes

Request to allow `agents.defaults.compaction.provider` to target an MCP server/tool directly instead of only a plugin-registered compaction provider.

## Best Possible Solution

Keep core on the plugin-registered compaction/context-engine seams, and publish an MCP-to-compaction adapter through ClawHub; open a narrower core issue only if that adapter proves a missing plugin API.

## Reproduction Assessment

Not applicable: this is a feature request for a new compaction-provider mode, not a report of broken current behavior. Source inspection confirms current main supports plugin-registered providers but not the proposed direct MCP config shape.

## Solution Assessment

No: adding generic core `mcpServer`/`mcpTool` compaction config is not the best current path under the project’s lean-core/plugin boundary. A ClawHub plugin or context-engine plugin can own the adapter, with a narrower API-gap issue only if that implementation cannot be built against current seams.
