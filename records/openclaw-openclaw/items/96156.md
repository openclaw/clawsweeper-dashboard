## Summary

Keep open: current main and v2026.6.10 still route compaction provider selection through plugin-registered `CompactionProvider` ids, while MCP tool execution is a separate bundle-MCP runtime surface. The contributor discussion also includes a compaction owner saying this is not a ClawHub close and needs design discussion.

## What This Changes

Request to let `agents.defaults.compaction.provider` target an MCP server/tool so MCP summarizers can serve as compaction engines without a plugin wrapper.

## Best Possible Solution

Keep this open for maintainer API/product review; either define a narrow MCP compaction adapter contract with explicit security/config semantics, or document the plugin-wrapper boundary as the supported path.

## Reproduction Assessment

Yes for the missing capability: source inspection shows compaction `provider` resolves only to a plugin-registered provider, while MCP tools execute through a separate session MCP runtime. This is source-reproducible rather than a failing bug repro.

## Solution Assessment

Unclear. A direct MCP adapter is plausible, but it is not automatically the best solution because current plugins can wrap compaction providers and the new adapter would add config and transcript-execution policy surface.
