## Summary

Keep open. Current main still lets a top-level Feishu card-action callback id win over the real `context.open_message_id`, so the reported temporary-id streaming failure remains source-reproducible even after the adjacent fix for https://github.com/openclaw/openclaw/issues/71673.

## What This Changes

The issue reports that Feishu card-action callbacks can pass a temporary `card-action-c-*` id into streaming reply routing, causing Feishu to reject `open_message_id`.

## Best Possible Solution

Fix the Feishu plugin's card-action normalization so a real `context.open_message_id` is preferred over temporary `card-action-c-*` callback ids, temporary callback ids are never used as platform reply targets, and no-real-id callbacks plain-send immediately.

## Reproduction Assessment

Yes by source inspection, with a clear focused regression path: dispatch `card.action.trigger` with top-level `open_message_id: "card-action-c-..."`, `context.open_message_id: "om_original"`, and a reply-producing action, then assert the dispatcher receives `replyToMessageId: "om_original"`. I did not run tests because this review is read-only and tests can write caches/artifacts.

## Solution Assessment

Yes. The narrowest maintainable fix is at the Feishu card-action parser/synthetic-event boundary, not core reply dispatch, because the bug is Feishu-specific callback-id normalization while existing flat real-message-id behavior must remain intact.
