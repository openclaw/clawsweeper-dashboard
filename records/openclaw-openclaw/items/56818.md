## Summary

Keep open. The earlier Feishu card-action fixes cover missing callback fields, but current main still normalizes `open_message_id` with top-level precedence and then copies that chosen value into `context.open_message_id`, so the reported payload shape can still route the temporary `card-action-c-*` id into streaming reply delivery.

## What This Changes

The issue asks Feishu card-action callbacks to avoid using temporary `card-action-c-*` ids as streaming reply targets, using the original message id from callback context or falling back to a new message send instead.

## Best Possible Solution

Fix this inside the Feishu plugin boundary by normalizing card-action callback ids so `context.open_message_id` wins over a temporary top-level `card-action-c-*` id, and only fall back to plain-send semantics when no real `om_*` reply target is available. Add a focused lifecycle regression for that mixed-id payload shape.

## Reproduction Assessment

Yes. A `card.action.trigger` payload with top-level `open_message_id: "card-action-c-..."` and `context.open_message_id: "om_..."` statically reproduces the bad route: the parser chooses the temporary id, the synthetic event makes it the reply target, and streaming uses `im.message.reply` with that id.

## Solution Assessment

Yes, a narrow Feishu parser/normalization fix is the best path. This is plugin-owned behavior, not a core streaming change, and it should be covered by the existing card-action lifecycle test harness.

Fix Feishu card-action callback reply-target normalization for issue https://github.com/openclaw/openclaw/issues/56818. Current `parseFeishuCardActionEventPayload` in `extensions/feishu/src/monitor.account.ts` chooses top-level `open_message_id` before `context.open_message_id` and copies the result into context, so callbacks with top-level temporary `card-action-c-*` ids can still reach `FeishuStreamingSession.start` as invalid reply targets.
