## Summary

Keep open. Current main includes the adjacent #71673 fix for synthetic `card-action-*` message IDs, but the reported `open_message_id: card-action-c-*` path can still flow into Feishu streaming reply routing because the parser prefers the top-level callback ID over `context.open_message_id` and there is no guard for temporary card-action IDs.

## What This Changes

The issue reports that Feishu interactive card callbacks can pass a temporary `card-action-c-*` `open_message_id` into streaming replies, causing Feishu to reject the reply start request.

## Best Possible Solution

Keep the fix in the Feishu plugin: normalize card-action callback message IDs so temporary `card-action-c-*` values are never used as platform reply targets, prefer a valid `context.open_message_id` when present, and suppress reply metadata so streaming falls back to a create/send path when only a temporary callback ID exists.

## Reproduction Assessment

Yes. A source-level reproduction is to feed `parseFeishuCardActionEventPayload` a card callback whose top-level `open_message_id` is `card-action-c-*` and whose `context.open_message_id` is a real `om_*`; current main will choose the top-level value and pass it into `streaming.start`.

## Solution Assessment

No. The current #71673-era implementation is the right general direction, but it is incomplete for this issue because it trusts any provided `open_message_id` instead of recognizing Feishu temporary callback IDs and falling back to the nested real message ID or plain send.
