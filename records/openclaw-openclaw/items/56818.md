## Summary

Keep open. Current main still has a concrete Feishu card-action path where a top-level temporary `card-action-c-*` callback id can be normalized into `context.open_message_id`, then used as the streaming reply target, matching the reported `Invalid open_message_id` failure.

## What This Changes

The issue reports that Feishu interactive card callbacks can pass a temporary `card-action-c-*` `open_message_id` into streaming replies, causing Feishu to reject the reply target.

## Best Possible Solution

Fix the Feishu plugin's card-action normalization so `context.open_message_id` remains the preferred real reply target, top-level `card-action-c-*` callback ids are never used as platform reply targets, and callbacks without a real original message id fall back to plain sending. Add focused lifecycle coverage for the reported both-fields payload.

## Reproduction Assessment

Yes. A focused lifecycle test can dispatch `card.action.trigger` with top-level `open_message_id: "card-action-c-..."`, `context.open_message_id: "om_original"`, and a reply-producing action, then assert `createFeishuReplyDispatcher` receives `replyToMessageId: "om_original"` rather than the temporary id.

## Solution Assessment

Yes. The narrowest maintainable fix is in the Feishu card-action parser/synthetic-event boundary, not in core reply dispatch, because this is Feishu-specific callback-id normalization.

## Repair Work Prompt

Fix Feishu card action callback reply target selection for https://github.com/openclaw/openclaw/issues/56818. The reported payload has top-level `open_message_id: "card-action-c-*"` plus a real `context.open_message_id`; current `extensions/feishu/src/monitor.account.ts` picks the top-level value first and writes it back into context, then `extensions/feishu/src/card-action.ts` turns it into `reply_target_message_id`, causing streaming cards to call Feishu `message.reply` with the temporary id. Keep the change inside the Feishu plugin. Preserve SDK-style flat callbacks whose top-level `open_message_id` is a real message id. Prefer `context.open_message_id` when present; ignore or suppress top-level `card-action-c-*` callback ids when no real target exists so replies plain-send instead of using invalid reply metadata. Add focused regression coverage in the Feishu card-action lifecycle/unit tests for top-level temporary id plus real context id, and for the no-real-id fallback if needed. Add a user-facing `CHANGELOG.md` fix entry crediting the reporter. Do not change core reply dispatcher semantics or global channel contracts.
