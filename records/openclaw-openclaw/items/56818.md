## Summary

Keep open. Current main and v2026.5.22 still normalize Feishu card-action `open_message_id` by taking the top-level value before `context.open_message_id`, so a temporary `card-action-c-...` callback id can still become the streaming reply target; the related card-update PR is adjacent but does not close or solve this exact parser precedence bug.

## What This Changes

The issue reports that Feishu interactive-card callbacks can pass a temporary `card-action-c-...` ID into streaming reply startup instead of the original message ID.

## Best Possible Solution

Fix the Feishu card-action parser so a real `context.open_message_id` wins over temporary callback ids, while preserving SDK-flat real top-level ids and plain-send fallback when no real message id exists.

## Reproduction Assessment

Yes, as a source-level reproduction: parse a Feishu `card.action.trigger` payload with top-level `open_message_id: "card-action-c-..."`, `context.open_message_id: "om_original"`, and a reply-producing action, then assert the dispatcher uses `om_original`. I did not execute a live Feishu callback in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is at the Feishu card-action parser/synthetic-event boundary, before the streaming sender decides whether to call `message.reply`.
