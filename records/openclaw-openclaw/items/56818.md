## Summary

Keep open. Current main and the latest release still let a top-level Feishu card-action `open_message_id` win over `context.open_message_id`, so a temporary `card-action-c-...` callback id can still become the streaming reply target; the adjacent open card-update PR is related but does not resolve or close this report.

## What This Changes

The issue requests fixing Feishu card-action reply targeting so streaming replies do not use a temporary callback `open_message_id`.

## Best Possible Solution

Fix Feishu card-action normalization so a real `context.open_message_id` wins over temporary callback ids while preserving SDK-flat real top-level ids and the no-real-id plain-send fallback.

## Reproduction Assessment

Yes, as a source-level reproduction: dispatch a Feishu `card.action.trigger` payload with top-level `open_message_id: "card-action-c-..."`, `context.open_message_id: "om_original"`, and a reply-producing action, then assert the dispatcher uses `om_original`. I did not execute a live Feishu callback in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is at the Feishu card-action parser/synthetic-event boundary, not in the streaming sender, so the original platform message id is carried before any reply delivery path starts.
