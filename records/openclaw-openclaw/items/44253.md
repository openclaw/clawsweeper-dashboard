## Summary

Keep open. Current main still has no `selfDeny`, `delegateOnly`, `allowEscalate`, or equivalent parent-hidden/child-eligible tool scope, and the request adds a new security-sensitive public config contract rather than fixing a broken existing contract.

## What This Changes

Request a per-agent tool policy scope that hides tools from an orchestrator’s own model surface while leaving them eligible for spawned subagents.

## Best Possible Solution

Define a maintainer-approved delegate-only tool scope that hides selected tools from the parent model surface, keeps them eligible for subagent filtering, preserves hard deny precedence, and updates schema, docs, metadata, and focused policy tests together.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to remove a tool at the parent policy/profile stage and then try to re-allow it through `tools.subagents.tools`; current code and docs show subagent policy only narrows the already-resolved tool set.

## Solution Assessment

Unclear. `selfDeny` or `delegateOnly` is plausible, but the best solution needs maintainer agreement on the public config name, child-eligible pool semantics, and security precedence before implementation.
