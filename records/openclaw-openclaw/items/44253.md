## Summary

Keep open: current main and the latest release still lack a `selfDeny` or delegate-only tool policy, and the active tool inheritance path still propagates parent denies or restrictive allowlists into spawned child sessions. The request remains source-reproducible and canonical, but it needs maintainer product and security approval before implementation because it adds config surface for high-power tool delegation.

## What This Changes

The issue requests a per-agent `tools.selfDeny` or delegate-only policy so an orchestrator can pass tools to subagents while hiding those tools from its own model-visible surface.

## Best Possible Solution

Define a maintainer-approved delegate-only tool-policy model in core config, enforce it before model tool projection, preserve deny-wins semantics, and cover native subagent plus ACP behavior with focused tests and docs.

## Reproduction Assessment

Yes from source inspection: the current config has no delegate-only key, and parent denies or restrictive allowlists are propagated into spawned child sessions as inherited policy. I did not run a live scenario because the code path and existing tests already show the limitation.

## Solution Assessment

Unclear: `selfDeny` is plausible, but the best fix depends on maintainer choice of syntax, security semantics, deny precedence, and ACP/native subagent compatibility.
