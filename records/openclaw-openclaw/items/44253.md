## Summary

Keep open. Current main still has no `selfDeny`, `delegateOnly`, `allowEscalate`, or equivalent parent-hidden/child-eligible tool scope, and the latest delegated-tool work on main intentionally preserves inherited restrictions rather than allowing subagents to regain tools hidden from the parent.

## What This Changes

This issue requests a new per-agent tool policy option that hides selected tools from an orchestrator agent while keeping them eligible for its spawned subagents.

## Best Possible Solution

Define a maintainer-approved delegate-only tool scope that hides selected tools from the parent model surface, keeps them eligible for subagent filtering, preserves hard deny and inherited deny precedence, and updates schema, docs, metadata, and focused policy tests together.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to deny or profile-filter a tool at the parent level and then try to re-allow it through `tools.subagents.tools`; current code and docs show that subagent policy only narrows the already-resolved list.

## Solution Assessment

Unclear. `selfDeny` or `delegateOnly` is a plausible direction, but the best solution needs maintainer agreement on the public config name, child-eligible pool semantics, and security precedence before implementation.
