- `impact:security`: The requested behavior changes command and file-tool authorization boundaries between an orchestrator and its subagents.

## Summary

Keep this open: current main still has no delegate-only or self-deny tool policy, and the existing implementation deliberately carries the parent’s final tool restrictions into subagent sessions. The request is valid source-level feature/security work, but it needs maintainer product and security design before implementation because it adds a config surface and an exception to monotonic deny inheritance.

## What This Changes

Add a per-agent tool policy mode that hides selected tools from the orchestrator’s own model surface while still allowing those tools to be inherited by spawned subagents.

## Best Possible Solution

Design a maintainer-approved delegate-only tool policy that keeps `deny` absolute, documents the upgrade/security semantics, and proves both orchestrator self-surface and child inherited tool lists with focused tests.

## Reproduction Assessment

Yes, source-level. Current schema and runtime only support allow/deny-style filtering, and current subagent code forwards inherited parent restrictions into child sessions; I did not run a live gateway scenario.

## Solution Assessment

Unclear. `selfDeny` is a plausible shape, but the best solution depends on maintainer choice of config syntax, security semantics, and whether any narrower existing policy surface should own this.
