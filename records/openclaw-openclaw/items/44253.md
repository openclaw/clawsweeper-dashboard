## Summary

Keep this open: current main and latest release v2026.6.8 still lack a `selfDeny` or delegate-only tool policy, and the current pipeline remains intentionally non-escalating. The request is source-reproducible, but it needs maintainer product and security review before implementation because it adds config surface and changes command/file-tool authorization semantics.

## What This Changes

Add a per-agent delegate-only tool policy so an orchestrator can hide command/file tools from its own model surface while allowing selected subagents to inherit them.

## Best Possible Solution

Design a maintainer-approved delegate-only policy that removes selected tools only from the parent model surface while keeping normal `deny` absolute and auditable across native subagent and ACP paths.

## Reproduction Assessment

Yes. Source inspection shows strict agent tools schema only accepts the existing allow/alsoAllow/deny surfaces, and runtime captures parent deny/effective allow into inherited child policy with no self-only removal phase.

## Solution Assessment

Unclear. `selfDeny` is a plausible shape, but the best fix depends on maintainer-approved syntax, security semantics, upgrade behavior, and ACP/native inheritance behavior.
