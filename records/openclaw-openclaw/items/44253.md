## Summary

Keep open. Current main still lacks a `selfDeny`, `delegateOnly`, `allowEscalate`, or equivalent subagents-only tool scope; the schema, policy pipeline, docs, and tests continue to model subagent policy as a final narrowing step. This remains the canonical issue after the broader multi-agent umbrella was closed, but the fix is a new security-sensitive config contract that needs maintainer API and precedence approval.

## What This Changes

Request an optional per-agent delegate-only tool policy so orchestrator agents can hide tools such as `exec`, `write`, and `edit` from themselves while keeping them eligible for worker subagents.

## Best Possible Solution

Define a maintainer-approved subagents-only tool scope that hides selected tools from the parent model surface, keeps them eligible for subagent filtering, preserves absolute `deny` precedence, and updates schema, docs, generated config metadata, and focused policy tests together.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to remove a tool through profile or per-agent policy and then try to re-allow it through `tools.subagents.tools`; current source, docs, and tests show the subagent layer only narrows the already-resolved list.

## Solution Assessment

Unclear. `selfDeny` or `delegateOnly` is a plausible narrow direction, but the best solution needs maintainer agreement on the public config name, parent-hidden versus child-eligible pool semantics, and security precedence before implementation.
