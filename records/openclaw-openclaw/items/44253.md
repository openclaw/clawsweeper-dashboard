## Summary

Keep open. Current main still has no `selfDeny`, `delegateOnly`, `allowEscalate`, or equivalent parent-hidden/child-eligible tool scope, and this is a new security-sensitive public config contract rather than a broken existing behavior.

## What This Changes

Request to add a per-agent tool policy mode that hides selected tools from an orchestrator agent while keeping them eligible for its subagents.

## Best Possible Solution

Define a maintainer-approved delegate-only tool scope that hides selected tools from the parent model, keeps them eligible for subagent filtering, preserves hard-deny precedence, and updates schema, docs, metadata, and policy tests together.

## Reproduction Assessment

Yes, as a source-level current-state check: current main accepts only the existing strict per-agent tool policy fields, applies subagent policy after earlier filtering, and documents that subagent allowlists cannot add back tools removed earlier.

## Solution Assessment

Unclear. `selfDeny` or `delegateOnly` is plausible, but the best solution needs maintainer agreement on the public config name, child-eligible pool semantics, hard-deny precedence, and security review before implementation.

## Security Review

Status: needs_attention

Summary: The request is security-sensitive because it separates parent model tool visibility from child session access to powerful tools.

Concerns:

- **[medium] Preserve hard deny precedence:** `src/agents/tool-policy-pipeline.ts:158`
  - body: A delegate-only implementation must not let `tools.subagents.tools.allow` restore tools removed by global, agent, sender, inherited, sandbox, or explicit deny policy; otherwise parent deny could become a child grant for `exec`, `write`, or `edit`.
  - confidence: 0.9
- **[medium] Define the public model-surface split deliberately:** `src/config/zod-schema.agent-runtime.ts:703`
  - body: `AgentToolsSchema` is strict and currently has no parent-hidden child-eligible field, so adding one should be treated as a new public authorization contract rather than a local schema patch.
  - confidence: 0.86
- **[medium] Do not use exec approvals as the split boundary:** `src/infra/exec-approvals.ts:895`
  - body: Exec approval resolution is agent-keyed and does not model the difference between the orchestrator's provider-visible tool list and subagent child-session capability, so the tool-policy layer needs to carry this boundary explicitly.
  - confidence: 0.78
