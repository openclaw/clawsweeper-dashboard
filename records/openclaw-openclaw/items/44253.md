## Summary

Keep open: current main and the latest release still do not provide `selfDeny`, `delegateOnly`, or an equivalent parent-self-only deny surface. The request remains source-reproducible, but it changes command/file tool authorization semantics and needs maintainer product and security review before implementation.

## What This Changes

The issue requests a per-agent tool policy option that hides selected tools from an orchestrator agent itself while still allowing those tools to be delegated to subagents.

## Best Possible Solution

Design a maintainer-approved delegate-only policy that keeps ordinary `deny` absolute, removes selected tools only from the parent model surface, and makes delegated child access explicit, audited, documented, and covered by native/ACP/gateway tests.

## Reproduction Assessment

Yes, at source level. Current main lacks a self-only deny/delegate-only config key, and the existing schema, policy code, docs, and tests still make ordinary parent restrictions flow into child sessions.

## Solution Assessment

Unclear. `selfDeny` is a plausible shape, but the best fix needs maintainer-approved syntax, deny-wins semantics, audit behavior, and native/ACP/gateway inheritance coverage before implementation.
