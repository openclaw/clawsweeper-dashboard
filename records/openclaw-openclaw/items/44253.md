## Summary

Keep open. Current main still has no parent-hidden, child-eligible tool policy, and the request is a new security-sensitive public config contract that needs maintainer API and security design rather than conservative cleanup.

## What This Changes

The issue requests a per-agent delegate-only tool policy such as `tools.selfDeny` so an orchestrator can hide powerful tools from its own model surface while keeping them eligible for subagents.

## Best Possible Solution

Define a maintainer-approved delegate-only tool scope that hides selected tools from the parent model, keeps them eligible for subagent filtering, preserves hard-deny precedence, and updates schema, docs, metadata, and focused policy tests together.

## Reproduction Assessment

Yes, as a source-level current-state check. Current main accepts only the existing strict per-agent tool policy fields, applies subagent policy after earlier filtering, and documents that subagent allowlists cannot add back tools removed earlier.

## Solution Assessment

Unclear. `selfDeny` or `delegateOnly` is plausible, but the best solution needs maintainer agreement on the public config name, child-eligible pool semantics, hard-deny precedence, and security review before implementation.

## Security Review

Status: needs_attention

Summary: The requested config surface is security-sensitive because it separates parent model visibility from child access to powerful tools such as `exec`, `write`, and `edit`.

Concerns:

- **[medium] Preserve hard deny precedence:** `src/agents/pi-tools.policy.ts:107`
  - body: A delegate-only implementation must not let `tools.subagents.tools.allow` restore tools removed by global, agent, sender, inherited, or explicit deny policy; otherwise a parent deny could become a child grant for powerful tools.
  - confidence: 0.9
- **[medium] Define the public model-surface split deliberately:** `src/config/zod-schema.agent-runtime.ts:703`
  - body: `AgentToolsSchema` is strict and currently has no parent-hidden child-eligible field, so adding one should be treated as a new public authorization contract rather than a local schema patch.
  - confidence: 0.86
- **[medium] Do not use exec approvals as the split boundary:** `src/infra/exec-approvals.ts:895`
  - body: Exec approval resolution is keyed by agent policy and does not itself model the difference between an orchestrator's visible tool surface and child-session eligibility, so this boundary belongs in tool-policy semantics.
  - confidence: 0.78
