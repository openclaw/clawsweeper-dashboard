## Summary

Keep this open: current main and v2026.6.6 still do not provide a `selfDeny` or delegate-only tool policy, and the current pipeline still applies parent/agent policy before subagent filtering. The request is source-reproducible, but it needs maintainer product and security approval because it adds a config surface and creates an explicit exception to normal non-escalating tool inheritance.

## What This Changes

The issue requests a per-agent delegate-only tool policy so an orchestrator can hide tools from itself while still allowing subagents to receive those tools.

## Best Possible Solution

Design a maintainer-approved delegate-only policy that removes selected tools only from the parent model surface, keeps normal `deny` absolute, and proves native subagent plus ACP child-session behavior.

## Reproduction Assessment

Yes. Source inspection reproduces the limitation: parent/agent policy filters the tool set before subagent policy is applied, so a denied parent tool is not available for `tools.subagents.tools.allow` to restore.

## Solution Assessment

Unclear. `selfDeny` is a plausible shape, but the best solution depends on maintainer choice of syntax, security semantics, and interaction with inherited allow/deny behavior.
