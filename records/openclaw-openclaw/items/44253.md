## Summary

Keep open. Current main has per-agent tool policy and a final subagent tool-policy layer, but it does not implement `selfDeny`, `delegateOnly`, or another subagents-only tool scope; the current pipeline still filters parent/agent policy before applying subagent policy, so a parent-denied or profile-removed tool is unavailable to children. The related #50038 discussion explicitly names this orchestrator paradox but is closed as an umbrella item, and #59225 is adjacent schema slimming work rather than a replacement for delegate-only inheritance.

## What This Changes

Request to add a per-agent `tools.selfDeny` or delegate-only policy so an orchestrator can hide tools from its own model calls while still allowing eligible subagents to inherit them.

## Best Possible Solution

Define a maintainer-approved subagents-only tool scope that removes selected tools from the parent model surface while preserving them for eligible child sessions, with `deny` retaining absolute precedence and with schema, generated config metadata, docs, and focused policy tests updated together.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure an agent policy that removes `exec` and then attempt to allow `exec` through the subagent layer; the current pipeline applies the agent filter before the final subagent filter, and existing tests/docs cover the same cannot-restore behavior.

## Solution Assessment

Unclear. The requested `selfDeny` semantics are a plausible narrow fix, but the exact API name and precedence rules should be chosen by maintainers because this changes a security-sensitive tool inheritance contract.
