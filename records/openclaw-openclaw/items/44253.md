## Summary

Keep this open: current main and the latest release still lack a delegate-only or self-deny tool policy, while the current implementation deliberately carries parent tool restrictions into delegated child sessions. The request is source-reproducible, but it needs maintainer product and security design before implementation because it adds a config surface and a controlled exception to non-escalating tool inheritance.

## What This Changes

The issue requests a per-agent tool policy key that hides tools from an orchestrator's own model surface while allowing those tools to be delegated to subagents.

## Best Possible Solution

Design a maintainer-approved delegate-only tool policy that keeps normal deny absolute, documents the security semantics, and proves the orchestrator and child-session tool surfaces with focused tests.

## Reproduction Assessment

Yes, source-level. Current schema and runtime only support allow/deny-style filtering, and current subagent code forwards inherited parent restrictions into child sessions; no live gateway run was performed.

## Solution Assessment

Unclear. selfDeny is a plausible shape, but the best solution depends on maintainer choice of config syntax, security semantics, and whether a narrower existing policy surface should own this.
