## Summary

Keep open: current main and latest release still do not provide `selfDeny`, `delegateOnly`, or an equivalent parent-self-only deny surface. The gap is source-reproducible, but adding it changes command/file tool authorization semantics and needs maintainer product and security review.

## What This Changes

The issue requests a per-agent tool policy field that hides selected tools from an orchestrator agent while still allowing those tools to be inherited by delegated subagents.

## Best Possible Solution

Design a maintainer-approved delegate-only policy that keeps ordinary `deny` absolute while removing selected tools only from the parent model surface and preserving explicit auditability for delegated child access.

## Reproduction Assessment

Yes, at source level. Current main and `v2026.6.10` lack any self-only deny/delegate-only config key, and the current policy pipeline still makes child sessions inherit the caller's ordinary deny/effective tool boundary.

## Solution Assessment

Unclear. `selfDeny` is a plausible solution shape, but the best fix needs maintainer-approved syntax, deny-wins semantics, audit behavior, and native/ACP/gateway inheritance coverage.
