## Summary

Keep open. Current main still has no parent-self-only tool deny or equivalent delegate-only scope, and the requested behavior would add a security-sensitive public config contract that needs maintainer approval before implementation.

## What This Changes

Add a per-agent tool policy that hides selected tools from the parent agent while still allowing those tools to be inherited and narrowed by spawned subagents.

## Best Possible Solution

Define a maintainer-approved delegate-only or subagents-only tool scope, preserve hard `deny` precedence, then update schema/types/docs/generated config help and regression tests around parent filtering and subagent inheritance.

## Reproduction Assessment

Yes. A source-level reproduction is to deny a tool through profile or per-agent policy and then try to allow it through `tools.subagents.tools`; current code and docs show the subagent filter only narrows the already-resolved parent tool set.

## Solution Assessment

Unclear. `selfDeny` or `delegateOnly` is a plausible narrow design, but the best solution needs maintainer approval for the config name, exact processing point, and security precedence before code is queued.
