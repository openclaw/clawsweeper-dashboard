## Summary

Keep open: current main still exposes only allow/alsoAllow/deny tool policy, and the documented pipeline intentionally cannot re-grant a tool to subagents after an earlier parent policy removed it. The related multi-agent umbrella called out this same delegate-only gap, but did not ship or supersede a fix.

## What This Changes

The issue requests a per-agent delegate-only tool policy such as `tools.selfDeny` so an orchestrator can hide tools from itself while preserving them for worker subagents.

## Best Possible Solution

Design a core delegate-only tool policy contract, document deny precedence, and cover parent-self versus subagent inheritance across the shared policy pipeline and gateway/embedded runner call sites.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/config request. Source and docs do confirm the current limitation with high confidence: earlier policy filters remove tools before subagent policy runs, and no delegate-only config key exists.

## Solution Assessment

Unclear. A core policy seam is the right area, but maintainers should choose the public naming, precedence, and security semantics before implementation.
