## Summary

Keep open: current main and latest release still do not implement a `selfDeny`, `delegateOnly`, or equivalent parent-self-only tool denial surface. The gap is source-reproducible, but the fix changes command/file-tool authorization semantics and needs maintainer product and security review before implementation.

## What This Changes

The issue requests a per-agent delegate-only tool policy so an orchestrator can pass tools to subagents without seeing or directly using those tools itself.

## Best Possible Solution

Design a maintainer-approved delegate-only policy that keeps ordinary `deny` absolute while removing selected tools only from the parent model surface and preserving explicit auditability for delegated child access.

## Reproduction Assessment

Yes. Source inspection shows current main and latest release still lack any self-only deny/delegate-only config key and keep parent policy restrictions in inherited child policy.

## Solution Assessment

Unclear. `selfDeny` is a plausible solution shape, but the best fix needs maintainer-approved syntax, security semantics, and native/ACP/gateway inheritance coverage.
