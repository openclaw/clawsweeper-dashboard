## Summary

Keep open. Current main still has no `selfDeny`, `delegateOnly`, `allowEscalate`, or equivalent subagents-only tool scope, and the schema, policy pipeline, tests, and docs still describe monotonic filtering where subagent policy can only narrow the already-resolved tool set.

## What This Changes

Add a per-agent delegate-only tool policy so an orchestrator can pass tools such as `exec`, `write`, and `edit` to subagents while being config-blocked from using them directly.

## Best Possible Solution

Define a maintainer-approved subagents-only tool scope that hides selected tools from the parent model surface, keeps them eligible for subagent filtering, preserves absolute `deny` precedence, and updates schema, docs, generated config metadata, and focused policy tests together.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to remove a tool through profile or per-agent policy and then try to re-allow it through `tools.subagents.tools`; current source, tests, and docs show the subagent layer only narrows the already-resolved list.

## Solution Assessment

Unclear. `selfDeny` or `delegateOnly` is a plausible narrow direction, but the best solution needs maintainer agreement on the public config name, child-eligible pool semantics, and security precedence before implementation.
