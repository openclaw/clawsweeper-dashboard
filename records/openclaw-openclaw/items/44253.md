## Summary

Keep open. Current main still lacks `selfDeny`, `delegateOnly`, or another subagents-only tool scope; the strict per-agent tools schema and the monotonic policy pipeline still make this impossible as a config-level guarantee. The related #50038 umbrella names this same orchestrator paradox but is closed, while #59225 is adjacent schema-slimming work rather than a replacement.

## What This Changes

Add an optional per-agent delegate-only tool policy so orchestrator agents can hide tools from their own model surface while leaving them available to eligible subagents.

## Best Possible Solution

Define a maintainer-approved subagents-only tool scope that preserves hard `deny` precedence, hides selected tools from the parent model surface, and updates schema, generated config metadata, docs, and focused policy tests together.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to remove a tool through profile or agent policy and then attempt to allow it through `tools.subagents.tools`; current source and tests show the subagent layer only narrows the already-resolved tool set.

## Solution Assessment

Unclear. `selfDeny` or `delegateOnly` is a plausible narrow direction, but the exact config shape and security precedence rules should be chosen by maintainers before code changes.
