## Summary

Keep this open: current main and v2026.6.6 still do not provide a `selfDeny` or delegate-only tool policy, and the current pipeline still treats normal parent/agent deny as inherited by child sessions. The request is source-reproducible, but it needs maintainer product and security approval because it adds a config surface and creates an explicit exception to non-escalating tool inheritance.

## What This Changes

Requested support for a per-agent delegate-only tool deny list that hides tools from an orchestrator while still allowing those tools for spawned subagents.

## Best Possible Solution

Design a maintainer-approved delegate-only policy that removes selected tools only from the parent model surface, keeps normal `deny` absolute, and proves native subagent plus ACP child-session behavior.

## Reproduction Assessment

Yes. Source inspection reproduces the limitation: strict config accepts ordinary allow/deny only, and parent/agent policy filters the tool set before subagent and inherited policy can run.

## Solution Assessment

Unclear. `selfDeny` is a plausible shape, but the best solution depends on maintainer choice of syntax, security semantics, and interaction with inherited allow/deny behavior.
