## Summary

Keep this open: current main and v2026.6.6 still do not provide a `selfDeny` or delegate-only tool policy, and the existing pipeline still treats parent/agent deny and inherited child policy as normal non-escalating authorization. The request is source-reproducible, but it needs maintainer product and security approval because it adds a config surface and an explicit exception to tool inheritance semantics.

## What This Changes

The issue requests a per-agent `tools.selfDeny` or delegate-only policy so an orchestrator can pass selected tools to subagents while hiding those tools from its own model-visible surface.

## Best Possible Solution

Design a maintainer-approved delegate-only policy that removes selected tools only from the parent model surface, keeps normal `deny` absolute, and proves native subagent plus ACP child-session behavior.

## Reproduction Assessment

Yes. Source inspection reproduces the limitation: strict config accepts ordinary allow/deny only, and the parent policy filters the tool set before subagent and inherited policy are applied.

## Solution Assessment

Unclear. `selfDeny` is a plausible shape, but the best solution depends on maintainer choice of syntax, security semantics, upgrade behavior, and interaction with inherited allow/deny policy.
