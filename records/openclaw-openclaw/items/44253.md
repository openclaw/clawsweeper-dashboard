## Summary

Keep this open: current main and v2026.6.6 still lack a delegate-only or self-deny tool policy, and the current implementation intentionally carries parent tool allow/deny restrictions into child sessions. The request is source-reproducible, but it needs maintainer product and security design before implementation because it adds a config surface and a controlled exception to current non-escalating tool inheritance.

## What This Changes

The issue requests a per-agent `tools.selfDeny` or delegate-only policy that hides selected tools from an orchestrator while still allowing subagents to inherit them.

## Best Possible Solution

Design a maintainer-approved delegate-only tool policy that keeps normal deny absolute, documents the security semantics, and proves the orchestrator self-surface plus native subagent and ACP child-session tool surfaces with focused tests.

## Reproduction Assessment

Yes, source-level. Current main's strict schema, docs, runtime pipeline, and tests all show monotonic allow/deny inheritance with no separate parent-self removal stage; no live gateway run was needed for that source proof.

## Solution Assessment

Unclear. `selfDeny` is a plausible shape, but the best solution depends on maintainer choice of config syntax, security semantics, and interaction with ongoing inherited-tool fixes.
