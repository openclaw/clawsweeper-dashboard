## Summary

Keep open: this is a focused security bug-fix PR, but the latest head still has a source-reproducible disclosure path because suppressed BOOT.md echoes can fall through to generic tool failure logging with original raw params.

## What This Changes

The PR wraps BOOT.md content in internal-runtime-context delimiters, registers a session-scoped boot echo guard, sanitizes message-tool visible fields, and adds focused tests plus a changelog entry for fallback BOOT.md disclosure.

## Best Possible Solution

Keep the wrapper, message-tool sanitizer, and echo guard, but make deliberate echo suppression return a safe suppressed result or use a redacted diagnostic path before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: PR-head sanitization can produce an empty message, the current outbound runner throws on empty sends, and the generic adapter logs original raw tool params on that exception path.

## Solution Assessment

No. The PR touches the right layers, but suppressed boot echoes should not continue into the normal empty-send validation path where original BOOT.md-bearing params can be logged.
