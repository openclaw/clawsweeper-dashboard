## Summary

Keep open: the PR is aimed at a real prompt-disclosure bug and supplies useful real behavior proof, but latest head still leaves a concrete stringified-presentation bypass in the new message-tool sanitizer.

## What This Changes

The PR wraps BOOT.md content as internal runtime context, adds a session-scoped boot echo guard, sanitizes message-tool outbound visible fields, suppresses empty sanitized sends, and adds regression coverage plus a changelog entry.

## Best Possible Solution

Keep the layered BOOT.md/internal-runtime-context approach, but parse and sanitize JSON-string presentation payloads before delivery and cover that bypass with a focused regression test.

## Reproduction Assessment

Yes. Current main source gives a high-confidence reproduction path for the underlying bug: BOOT.md is ordinary prompt text, fallback retry preserves the original body, and the message-tool send path does not strip internal runtime context. The remaining PR blocker is also source-reproducible because string `presentation` skips the new sanitizer and is parsed later by `runMessageAction`.

## Solution Assessment

No. The proposed direction is narrow and maintainable, but it is incomplete until stringified presentation payloads are sanitized or rejected before `runMessageAction` parses and sends them.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Sanitize stringified presentation payloads:** `src/agents/tools/message-tool.ts:107-108`
  - body: The new sanitizer returns non-object `presentation` values unchanged, but `runMessageAction` still accepts JSON-string presentations and parses them before delivery. Please parse and sanitize string presentations before this early return, or move this sanitizer after the existing presentation parsing path.
