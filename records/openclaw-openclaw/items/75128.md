## Summary

Keep open. The PR is addressing a real BOOT.md prompt-disclosure path, but the added substantial-echo guard is still incomplete: it samples the boot prompt every 20 characters, so an unaligned exact 80-character copy can bypass the guard and still reach the message tool.

## What This Changes

The PR wraps BOOT.md content in internal-runtime-context delimiters, strips internal-runtime-context blocks from message-tool text fields, adds a per-session boot echo guard with tests, and adds a changelog entry for #53732.

## Best Possible Solution

Keep the delimiter wrapper and message-tool runtime-context strip, but revise the echo guard so it catches every outbound substring at or above the threshold, then add an unaligned exact 80-character slice regression test before merge.

## Reproduction Assessment

Yes. The PR algorithm can be reproduced with a source-level check: an outbound message equal to bootPrompt.slice(1, 81) is exactly 80 contiguous characters from the boot prompt, but the sampled scan returns false.

## Solution Assessment

No. The delimiter-based layers are the right direction, but the sampled echo detector is not dependable enough for a prompt-disclosure guard; complete window coverage or an equivalent rolling/fingerprint match is the safer narrow fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Scan every boot echo window:** `src/gateway/boot-echo-guard.ts:55-60`
  - body: `containsSubstantialBootEcho` is meant to suppress outbound text containing any 80+ contiguous characters from the boot prompt, but this loop samples only offsets spaced by `ECHO_SCAN_STEP` plus the tail.
