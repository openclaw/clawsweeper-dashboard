## Summary

Keep open: this PR targets a real prompt-disclosure bug and includes useful proof, but the new message-tool sanitizer still runs before the send path expands literal `\n`, leaving a delimiter-preserving bypass for outbound sends.

## What This Changes

The branch wraps BOOT.md in internal-runtime-context delimiters, adds a per-session boot echo guard, extends message-tool sanitization to runtime-context echoes, updates tests, and adds a changelog entry for https://github.com/openclaw/openclaw/issues/53732.

## Best Possible Solution

Keep the BOOT.md wrapper and echo guard, but move or repeat runtime-context stripping at the same boundary that normalizes outbound send text so all user-visible message-tool fields are sanitized after escaped-newline expansion.

## Reproduction Assessment

Yes, source-reproducible. Current main embeds BOOT.md directly, fallback retries preserve that body, and the message tool is a user-visible send path; the remaining PR blocker is also source-reproducible from sanitizer order versus later `\n` normalization.

## Solution Assessment

No. The PR changes the right layers, but the sanitizer order leaves a concrete bypass, so the narrow maintainable fix is to normalize escaped line breaks before `stripInternalRuntimeContext` or strip again after `handleSendAction` normalizes message text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Normalize escaped newlines before stripping context:** `src/agents/tools/message-tool.ts:793`
  - body: The new sanitizer runs before `handleSendAction` turns literal `\n` sequences in `message` into real newlines. A model can send `<<<BEGIN_OPENCLAW_INTERNAL_CONTEXT>>>\n...\n<<<END_OPENCLAW_INTERNAL_CONTEXT>>>`; this pass misses it, then the send handler normalizes it and can forward the internal block. Normalize escaped newlines before stripping, or run the same strip again after that conversion.
  - confidence: 0.86
