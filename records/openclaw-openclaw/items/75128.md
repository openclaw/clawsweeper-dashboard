## Summary

Keep open. The PR is aimed at the right boot and message-tool leak surfaces, but the new substantial-echo guard still samples the boot prompt in 20-character steps, so markerless unaligned 80-character BOOT.md echoes can pass through user-visible delivery.

## What This Changes

The PR wraps BOOT.md content in internal-runtime-context markers, strips runtime-context blocks from message-tool text fields, adds a per-session boot-echo guard with tests, and updates the changelog for #53732.

## Best Possible Solution

Keep the PR open and revise the boot echo guard to cover every outbound substring at or above the chosen threshold, either with a complete sliding-window scan or an equivalent fingerprint approach, plus regression coverage for unaligned 80-98 character slices.

## Reproduction Assessment

Yes. Against the PR head, use a boot prompt longer than 80 characters and an outbound message equal to an exact 80-character slice starting at offset 1; the function only checks offsets 0, 20, 40, and so on plus the tail, so that markerless echo can return false.

## Solution Assessment

No. The wrapper and message-tool runtime-context strip are good layers, but the sampled substring guard is not reliable enough for the security goal; complete window coverage or a fingerprint-based detector is the safer narrow fix.
