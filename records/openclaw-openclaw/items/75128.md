## Summary

Keep this PR open: the current main bug is real and the PR is pointed at the right boot/message-tool surfaces, but the latest patch still leaves a prompt-disclosure bypass and lacks required after-fix real behavior proof.

## What This Changes

The PR wraps BOOT.md content as internal runtime context, registers a per-session boot echo guard, extends message-tool sanitization including presentation fields, and adds tests plus a changelog entry.

## Best Possible Solution

Fix the sanitizer ordering so escaped-newline runtime-context blocks are stripped before any gateway/plugin send path, add focused regression coverage for that path, then add redacted real fallback-run proof before merge.

## Reproduction Assessment

Yes, source-reproducible. Current main embeds BOOT.md directly in the boot prompt, fallback retries preserve that body, and the message tool can deliver user-visible sends; this review did not perform a live Gemini fallback run.

## Solution Assessment

No. The PR targets the right layers, but the sanitizer runs before existing escaped-newline normalization and the PR still needs real behavior proof before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Normalize escaped newlines before stripping runtime context:** `src/agents/tools/message-tool.ts:785-788`
  - body: The new strip runs before `handleSendAction` turns literal `\\n` sequences in `message` into real newlines. A fallback model can emit `<<<BEGIN_OPENCLAW_INTERNAL_CONTEXT>>>\\n...\\n<<<END_OPENCLAW_INTERNAL_CONTEXT>>>`; this sanitizer misses it, then gateway-executed sends such as Telegram are forwarded to plugin dispatch with real marker lines before core delivery sanitization can run. Please normalize escaped newlines before stripping, or run the same runtime-context strip after that conversion for every send path.
  - confidence: 0.86
