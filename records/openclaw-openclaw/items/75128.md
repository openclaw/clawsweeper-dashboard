## Summary

Keep open. The PR addresses a real current-main prompt-disclosure path, but the latest head still has a blocking guard bug: the new BOOT.md echo detector samples the boot prompt every 20 characters, so an exact unaligned 80-character copy can bypass the message-tool suppression.

## What This Changes

The PR wraps BOOT.md prompt content in internal-runtime-context delimiters, sanitizes message-tool text fields, adds a per-session BOOT.md echo guard with tests, and adds a changelog entry for #53732.

## Best Possible Solution

Keep the delimiter wrapper and message-tool runtime-context strip, but replace the sampled echo detector with complete window coverage or an equivalent rolling/fingerprint check, then add an unaligned 80-character slice regression test before merge.

## Reproduction Assessment

Yes. Current main’s leak path is source-backed, and the PR-specific blocker reproduces with a deterministic check where bootPrompt.slice(1, 81) is an exact 80-character substring but containsSubstantialBootEcho returns false.

## Solution Assessment

No. The PR’s overall direction is maintainable, but the sampled guard is not dependable for prompt-disclosure hardening; exhaustive or rolling substring detection is the safer narrow fix.
