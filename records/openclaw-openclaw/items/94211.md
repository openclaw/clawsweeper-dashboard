## Summary

Keep open: the linked session-memory bug is real and neither current main nor v2026.6.8 fixes this reader path, but this PR's text-only deduplication is too broad and the contributor proof is only a Vitest run.

## What This Changes

The PR changes `src/hooks/bundled/session-memory/transcript.ts` to remember the last assistant text and skip consecutive assistant messages with identical extracted text.

## Best Possible Solution

Land one canonical consumer-layer fix that filters only actual duplicate transcript mirror rows while preserving legitimate repeated assistant text and standalone visible assistant rows, with regression coverage and redacted real memory-file proof.

## Reproduction Assessment

Yes, at source level: current main extracts every assistant text row from JSONL, so a raw thinking row plus a cleaned text row will duplicate in session-memory output. I did not run a live provider flow because this review is read-only.

## Solution Assessment

No. The PR is a plausible workaround, but text equality is broader than the reported raw/cleaned transcript artifact; the safer fix is a semantic transcript-mirror or parent-lineage filter with tests for legitimate repeated assistant rows.
