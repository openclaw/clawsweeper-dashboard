## Summary

Keep this draft PR open: the underlying Slack block-action routing bug is real on current main, but the current branch still broadens ordinary top-level DM routing in a way that conflicts with the existing stable-DM contract.

## What This Changes

The PR routes Slack block-action clicks through thread-aware Slack session handling, adds interactive message owner caching and wake behavior, adjusts threaded reply delivery/session repair behavior, and updates Slack/agent tests plus the changelog.

## Best Possible Solution

Keep the Slack-owned interaction fix direction, but preserve normal top-level DM session continuity and limit thread/owner routing to the Slack block-action click path before final Slack maintainer review.

## Reproduction Assessment

Yes at source level. Current main carries threadTs in the block-action payload but resolves the system-event session without threadTs, and PR head source shows a separate top-level DM routing regression against the stable-DM test contract.

## Solution Assessment

No. The Slack plugin is the right boundary, but the current draft is not the best final fix because it changes ordinary top-level DM session keys instead of keeping that repair scoped to interaction owner/thread routing.

## Review Findings

- **[P2] Preserve top-level DM session continuity:** `extensions/slack/src/monitor/message-handler/prepare-routing.ts:202-203`
  - body: This makes every direct message use `threadContext.messageThreadId` as the canonical thread id. Please keep normal top-level DM routing stable and use interaction owner/thread metadata only for the button-click path.
