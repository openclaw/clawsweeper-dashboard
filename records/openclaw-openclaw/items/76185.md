## Summary

Keep open. This PR addresses a real Slack interaction routing bug, but the current draft head introduces a top-level DM session-continuity regression and still needs narrowing before merge.

## What This Changes

The PR routes Slack block-action interactions through thread-aware Slack message handling, records interactive message ownership, wakes targeted sessions, adjusts threaded reply delivery/session repair behavior, and adds Slack/agent tests plus changelog entries.

## Best Possible Solution

Keep the fix in the Slack plugin, route button clicks with explicit interaction owner/thread metadata, preserve stable top-level DM session continuity, and rebase away already-implemented session repair changes before maintainer review.

## Reproduction Assessment

Yes. Source inspection shows current main carries threadTs in block-action payloads but cannot pass it into system-event session resolution, and PR-head source shows the introduced top-level DM session-key regression against an existing stable-DM contract.

## Solution Assessment

No. The Slack plugin is the right boundary, but this draft is not the best final fix because it broadens routing for ordinary top-level DMs instead of limiting the repair to block-action owner/thread routing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve top-level DM session continuity:** `extensions/slack/src/monitor/message-handler/prepare-routing.ts:203`
  - body: This makes every direct Slack message with `replyToMode="all"` use `threadContext.messageThreadId`, which is the top-level message timestamp. Ordinary DMs then move from the stable `agent:main:slack:direct:<user>` session into a fresh `:thread:<ts>` session per message, while the integration test in `prepare.test.ts` still expects the stable direct session. Keep normal top-level DMs on the direct session and carry block-action owner/thread metadata only for clicks.
  - confidence: 0.89
