## Summary

Keep open: current main still shows the Slack block-action thread-routing bug, but this draft PR broadens the fix by moving ordinary top-level Slack DMs with `replyToMode="all"` into per-message thread sessions, contradicting current tests and docs.

## What This Changes

The PR changes Slack block-action routing, interactive-message ownership caching, session wake behavior, threaded reply delivery, session repair tests, Slack prompt hints, and changelog entries for interactive Slack replies.

## Best Possible Solution

Keep the Slack-owned interaction routing fix, but scope it to explicit interaction owner/thread metadata while preserving the existing top-level DM session contract, then validate the focused Slack paths.

## Reproduction Assessment

Yes, from source inspection. Current main carries `threadTs` in the block-action payload but resolves the target session without it, and the PR-head routing regression is visible by comparing the changed direct-message routing with the existing stable-DM test/docs contract.

## Solution Assessment

No. The Slack plugin is the right boundary, but making all top-level DMs with `replyToMode="all"` thread-scoped is broader than the bug fix; the safer path is to route interaction clicks with owner/thread metadata without changing ordinary DM session keys.

## Review Findings

Please keep normal top-level DM routing stable and use explicit interaction owner/thread metadata for button clicks.
