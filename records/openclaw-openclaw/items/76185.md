## Summary

Keep open: the Slack-owned fix targets a real source-visible block-action routing gap, but this draft still changes ordinary top-level DM routing in a way that conflicts with current main's stable direct-DM contract and an unchanged integration test.

## What This Changes

The PR routes Slack block-action interactions through thread-aware session handling, adds interactive message owner caching/wake behavior, adjusts reply delivery/session repair behavior, and updates Slack tests, prompt hints, and changelog text.

## Best Possible Solution

Keep the fix inside the Slack plugin, route block actions using explicit interaction owner/thread metadata, and preserve ordinary top-level DM session continuity while leaving the already-main session repair behavior in place.

## Reproduction Assessment

Yes. Source inspection on current main shows Slack block-action payloads carry `threadTs` but session resolution cannot consume it, and the PR-head direct-message `canonicalThreadId` change is enough to route ordinary top-level `replyToMode="all"` DMs into per-message thread sessions.

## Solution Assessment

No. The Slack plugin is the right boundary, but the draft is not the best final fix because it broadens normal DM routing instead of limiting the repair to block-action ownership and thread wake/routing.
