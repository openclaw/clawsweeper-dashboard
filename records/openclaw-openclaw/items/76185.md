## Summary

Keep open: this PR targets a valid Slack thread-interaction gap on current main, but the draft head broadens ordinary top-level DM routing under `replyToMode="all"` and contradicts the existing direct-session contract; the session-repair portion is also already present on main.

## What This Changes

The PR changes Slack block-action routing, interactive message ownership, threaded reply delivery, session repair coverage, prompt hints, and changelog entries so Slack button clicks wake and reply through the owning thread session.

## Best Possible Solution

Land a minimal Slack-owned fix that threads block-action owner/thread metadata through system-event routing, wake, and reply delivery while preserving ordinary top-level DM session continuity and dropping already-main session-repair changes.

## Reproduction Assessment

Yes, by source inspection. Current main carries `threadTs` in Slack block-action payloads but resolves the queued system event without it, and the PR regression is source-reproducible because `threadContext.messageThreadId` becomes the top-level DM timestamp under `replyToMode="all"` while current tests expect the stable direct session.

## Solution Assessment

No. The Slack plugin is the right boundary, but the current draft is not the best final fix because it changes normal top-level DM routing instead of limiting the repair to explicit interaction owner/thread metadata.
