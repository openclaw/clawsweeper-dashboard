## Summary

Keep this draft PR open. Current main still drops Slack `threadTs` during block-action session-key resolution even though threaded inbound routing uses `:thread:<ts>` keys; the proposed diff is narrow, Slack-plugin-local, and I found no blocking correctness issue.

## What This Changes

The PR threads Slack block-action `threadTs` through system-event session resolution and adds focused monitor/interactions regression coverage.

## Best Possible Solution

Land this Slack-plugin-local fix after the draft is ready and the targeted Slack tests plus changed gate pass; keep broader wake-state concerns tracked separately under the existing Slack interaction follow-up.

## Reproduction Assessment

Yes. The static current-main path is enough to reproduce the routing mismatch: block actions parse and enqueue `threadTs`, but resolve the system-event session without it, so threaded callbacks target the base session key.

## Solution Assessment

Yes. Reusing `resolveThreadSessionKeys` inside the Slack system-event resolver is the narrowest maintainable fix because it aligns block-action events with the existing inbound Slack thread-routing contract without changing core APIs.
