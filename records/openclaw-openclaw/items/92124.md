- `P2`: This is a real, bounded memory-search/provider-routing regression with an open fix path but no evidence of broad runtime outage.
- `impact:auth-provider`: The failure is caused by provider/model routing normalization for an OpenAI-compatible embedding endpoint.
- `impact:session-state`: The same model identity is stored in memory index metadata, so the mismatch can pause persisted memory recall state until repaired.

## Summary

Keep open: current main and the latest release still strip the `openai/` embedding model prefix, and the linked fixing PR is still open with review/proof blockers. This issue should close only after that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to preserve `openai/...` embedding model names for non-native OpenAI-compatible memory endpoints instead of normalizing them to bare OpenAI model ids.

## Best Possible Solution

Land a narrow OpenAI embedding-provider fix that preserves `openai/` only for non-native base URLs, reuses parsed native OpenAI host detection, and proves both router request success and existing-index identity reuse.

## Reproduction Assessment

Yes, source-reproducible. Current main strips `openai/` before the model is used for both the embedding request and memory index identity; I did not run a live Requesty endpoint in this read-only review.

## Solution Assessment

Yes, the best fix belongs in the OpenAI embedding provider path. The implementation should conditionally preserve the qualified model for non-native OpenAI-compatible base URLs while keeping native OpenAI normalization intact.
