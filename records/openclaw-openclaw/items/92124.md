## Summary

Keep this issue open: current main and the latest release still strip the `openai/` embedding model prefix before request and memory-index identity use, and the linked fix path is an open pull request rather than a merged implementation.

## What This Changes

The issue reports that OpenClaw strips provider-qualified OpenAI embedding model IDs when `memorySearch.provider` is `openai` with a non-native OpenAI-compatible base URL, breaking Requesty embedding requests and existing memory index identity.

## Best Possible Solution

Land a corrected version of https://github.com/openclaw/openclaw/pull/92135, with native OpenAI URL detection aligned to the existing parsed-hostname rule and redacted router/index proof, then close this issue as implemented.

## Reproduction Assessment

Yes. Source inspection shows current main strips `openai/` before using `client.model` in both the embedding request body and memory index identity, matching the reported Requesty rejection and model mismatch path.

## Solution Assessment

Yes, conditional prefix preservation in the OpenAI embedding provider is the right owner boundary. The linked PR should first reuse the existing parsed native-OpenAI URL semantic and provide redacted router/index proof before it is the best mergeable fix.
