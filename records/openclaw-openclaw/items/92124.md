## Label Justifications

- `P1`: A shipped regression can stop memory indexing and search for existing custom-router configurations immediately after upgrade.
- `impact:session-state`: The runtime model identity can diverge from persisted vector-index metadata, pausing vector search and making the existing index unusable.
- `impact:auth-provider`: Provider/model routing is altered before the custom OpenAI-compatible endpoint receives the embedding request.

## Summary

Current main and v2026.6.5 deterministically strip the configured `openai/` prefix before sending the embedding request and before exposing the provider model identity, matching the router rejection and existing-index mismatch. No duplicate or open fix PR was found, so this valid regression should remain open for a focused repair.

## What This Changes

Preserve provider-qualified OpenAI embedding model names for non-native OpenAI-compatible memory endpoints while keeping native OpenAI behavior unchanged.

## Best Possible Solution

Resolve one endpoint-facing model identity during provider creation: preserve the configured qualified model for non-native custom endpoints, retain bare model IDs for native OpenAI, and reuse that resolved value for requests, cache metadata, status, and index identity.

## Reproduction Assessment

Yes. The supplied configuration deterministically enters current `normalizeModel`, removes `openai/`, and then uses that altered value for both the `/embeddings` payload and provider model identity; the review did not send a credentialed live Requesty request.

## Solution Assessment

Yes in direction, but the narrow maintainable fix is generic native-versus-custom endpoint model resolution with shared identity, not a Requesty-specific branch or a new configuration option.
