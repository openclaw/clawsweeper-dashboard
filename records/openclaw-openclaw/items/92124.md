## Label Justifications

- `P2`: This is a concrete memory search regression for a bounded OpenAI-compatible router setup, not a whole-runtime outage.
- `impact:auth-provider`: The failure is caused by provider/model routing and model-choice normalization for an OpenAI-compatible embedding endpoint.
- `impact:session-state`: The mismatch can pause vector search because the stored memory index identity no longer matches the runtime embedding model.

## Summary

Keep open: current main and the latest release still strip the `openai/` embedding prefix without considering the resolved base URL, and the focused fix is already tracked by an open linked PR that has not merged.

## What This Changes

The issue asks OpenClaw to preserve `openai/text-embedding-3-small` for memory search when `provider: "openai"` is routed through a non-native OpenAI-compatible embedding base URL such as Requesty.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/92135 with a narrow fix that preserves provider-qualified OpenAI embedding model strings only for non-native OpenAI-compatible base URLs, keeps native `api.openai.com` behavior unchanged, and proves existing memory-index identity remains usable.

## Reproduction Assessment

Yes from source inspection: current main strips `openai/` before using the embedding model in both the request body and runtime identity, matching the reported Requesty rejection and index mismatch. I did not run a live Requesty reproduction in this read-only review.

## Solution Assessment

Yes in direction: preserving the qualified model only for non-native OpenAI-compatible base URLs is the narrowest maintainable fix. The linked PR should still resolve its review blocker and provide redacted real router/index proof before merge.
