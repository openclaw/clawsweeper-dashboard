## Summary

Keep open. Current main still has the reported key-only follow-up queue cleanup, and PR #68839 remains the open closing implementation candidate, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue requests an identity guard around follow-up drain cleanup so a late drain cannot delete a newer queue entry for the same key after `/stop` or session reset.

## Best Possible Solution

Review PR #68839 as the canonical fix candidate and land a guarded cleanup plus regression test, or replace it with an equivalent narrow fix if review finds problems.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to park the first drain in an awaited follow-up run, clear the session queues via `/stop`, enqueue a replacement queue for the same key, then allow the first drain's `finally` to run and delete the current map entry by key. I did not run tests because this review was read-only.

## Solution Assessment

Yes for the proposed direction. Checking `FOLLOWUP_QUEUES.get(key) === queue` before deleting the map entry and clearing the callback is the narrowest maintainable fix; current main is not sufficient because it still mutates by key only.
