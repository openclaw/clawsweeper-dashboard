## Summary

Keep open: the latest branch narrowed the wildcard namespace to `ackReaction` and fixed two scanners, but it still makes `channels["*"].ackReaction` validate before the matching runtime behavior is on the merge result, and remaining read-only/status scanners still do not reserve the wildcard key.

## What This Changes

The PR adds `channels["*"]` as an accepted config key for `ackReaction` only, rejects other wildcard fields, and updates two channel enumeration helpers plus focused tests.

## Best Possible Solution

Define `channels["*"].ackReaction` as one supported contract across validation, runtime fallback, docs, scanner helpers, and tests, or keep `messages.ackReaction` as the supported cross-channel fallback.

## Reproduction Assessment

Yes, at source level: current main rejects `channels["*"]` during validation and `resolveAckReaction()` does not read that key. This is not an established documented bug contract; it is a new wildcard config behavior being proposed.

## Solution Assessment

No, not as a standalone PR. The latest branch is narrower than before, but the maintainable fix must align validation, runtime fallback, scanner treatment, docs or explicit scope, and tests before the new config namespace lands.
