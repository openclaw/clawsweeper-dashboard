## Summary

Keep this PR open: it usefully narrows `channels["*"]` to an ackReaction-only contract and updates several scanners, but the final runtime fallback is still object-level, so common configured channels without their own `ackReaction` ignore the wildcard. The posted proof shows validation passing, not the final user-visible ack reaction path.

## What This Changes

The PR reserves `channels["*"]` as an ackReaction-only channel config key, excludes it from configured-channel scanners, adds a wildcard ackReaction fallback, and adds focused tests.

## Best Possible Solution

Decide whether OpenClaw should support an ackReaction-only wildcard channel config; if yes, make the fallback field-level, keep explicit empty-string suppression, preserve scanner exclusions, add the missing regression test, and provide real channel proof.

## Reproduction Assessment

Yes for the PR defect: source inspection of the PR head shows the wildcard is only read when the concrete channel config object is absent. A configured channel with unrelated settings and no `ackReaction` would still fall through to `messages.ackReaction`.

## Solution Assessment

No. The best fix is a coherent, maintainer-accepted wildcard contract with field-level fallback and proof, not an object-level fallback that misses common configured-channel cases.
