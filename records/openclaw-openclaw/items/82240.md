## Summary

Keep open: this is a plausible additive Plugin SDK feature, but the PR currently accepts `longDescription` at the schema boundary and then drops it in the gateway request handler, so `plugin.approval.request` cannot deliver the long form to Telegram, Discord, or Matrix. The external proof is also only unit/API/format terminal output, not real channel behavior proof.

## What This Changes

Adds an optional `longDescription` plugin approval payload field, SDK exports, renderer preference logic for generic/Discord/Matrix approval surfaces, tests, API baseline hash updates, and a changelog entry.

## Best Possible Solution

Preserve `longDescription` from gateway validation through record creation, listing, forwarding, view-model rendering, channel renderers, SDK docs, and focused regression tests, then require real channel proof before merge.

## Reproduction Assessment

Yes for the PR defect: source inspection of the PR head shows the schema accepts `longDescription`, but the gateway handler does not copy it into the created approval request. I did not run a live channel flow because this review is read-only.

## Solution Assessment

No. The additive field/helper direction is plausible, but the implementation must preserve the field at gateway request creation and prove the real channel path before it solves the linked approval-context problem.
