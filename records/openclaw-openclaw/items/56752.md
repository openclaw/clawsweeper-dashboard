## Summary

Keep open. Current main still does not provide the requested desktop multi-window parallel-session behavior: native WebChat keeps a single normal window controller, native `/new` resets the current session, and the Control UI still blocks New Chat while a run or queue is active.

## What This Changes

The issue requests desktop support for creating another conversation window without pausing or interrupting existing active sessions.

## Best Possible Solution

Define a first-class per-window/per-session desktop flow backed by `sessions.create`, keep native window controllers keyed by session, and reserve reset behavior for explicit reset commands.

## Reproduction Assessment

Yes, at source level: current native WebChat uses one normal window controller and native `/new` still resets the selected session, while Control UI New Chat is blocked during active runs. No live desktop recording was performed.

## Solution Assessment

No current implementation fully solves the issue. The maintainable direction is a product-scoped per-window/per-session controller model using the existing session create/switch APIs instead of relying on reset semantics.
