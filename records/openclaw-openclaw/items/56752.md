## Summary

Keep open. Current main and v2026.5.20 still do not provide first-class parallel desktop/window sessions: native WebChat has one normal window controller, native `/new` still resets the selected session, and Control UI refuses New Chat while a run or queue is active.

## What This Changes

Request desktop/Control UI multi-window parallel sessions so creating another chat does not pause, reset, or block existing active sessions.

## Best Possible Solution

Define a first-class per-window/per-session desktop flow backed by `sessions.create`, keep native controllers keyed by session/window, preserve existing active runs by default, and reserve reset behavior for explicit reset commands.

## Reproduction Assessment

Yes, at source level: current main stores one normal native window controller, native `/new` calls `sessions.reset`, and Control UI returns an active-run error before `sessions.create` during active work. I did not run a live macOS desktop repro in this read-only review.

## Solution Assessment

No current implementation fully solves the issue. The maintainable direction is a product-scoped per-window/per-session controller model using existing session create/switch APIs instead of relying on reset semantics.
