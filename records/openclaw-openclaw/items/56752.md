## Summary

Keep open: current main and v2026.6.10 still do not provide the requested parallel macOS desktop chat windows. The source still has one normal WebChat window controller, macOS WebChat still does not implement `sessions.create`, and `/new` still falls back to resetting the current session when create-session support is missing.

## What This Changes

The issue requests multi-window desktop sessions so opening a new OpenClaw chat window does not pause, interrupt, or reset an existing active session.

## Best Possible Solution

Define the desktop session/window ownership model, then implement macOS WebChat windows keyed by session with `sessions.create` support and coverage proving one active run survives opening another chat window.

## Reproduction Assessment

Yes, at source level: WebChatManager owns one normal window and closes it for a different session, while macOS WebChat inherits the unsupported `createSession` path so `/new` resets instead of creating an independent window/session. I did not run a live macOS desktop smoke in this read-only review.

## Solution Assessment

No current implementation solves the request. The best path is a maintainer-approved desktop session/window model that reuses the existing gateway `sessions.create` RPC instead of adding a new generic session API.
