## Summary

Keep open. Current main still uses a single normal native WebChat window controller and reset-style new-session flows, while the backend `sessions.create` primitive is present but not wired into an independent multi-window desktop or Control UI product flow. The earlier ClawSweeper keep-open review still matches current source.

## What This Changes

The issue requests desktop multi-window conversation support so opening or starting another conversation does not pause or interrupt an existing active session.

## Best Possible Solution

Add a first-class multi-window/session UX that creates or selects independent session keys through `sessions.create`, keeps controller/view-model state per visible window, and reserves `sessions.reset` for explicit destructive reset or clear actions.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: open or select a different native chat session and `WebChatManager.show` closes the existing normal controller; use `/new` or the Control UI New session button and the path still routes through reset semantics that tests prove abort active work.

## Solution Assessment

No. The current reset-based behavior is not the best solution for parallel conversations; the maintainable direction is an explicit per-window/per-session model backed by `sessions.create` and non-blocking `chat.send`.
