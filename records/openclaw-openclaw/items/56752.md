## Summary

Keep open. Current main and the latest release still do not provide the requested parallel desktop/window session behavior: native chat is still keyed through one normal window controller, `/new` can still reset the selected session, and Control UI still blocks New Chat while a run or queue is active.

## What This Changes

The issue requests desktop multi-window parallel session support so creating or switching to a new chat window does not pause or interrupt an existing active session.

## Best Possible Solution

Define a first-class per-window/per-session desktop flow backed by `sessions.create`, keep native controllers keyed by session/window, preserve existing active runs by default, and reserve reset behavior for explicit reset commands.

## Reproduction Assessment

Yes, at source level: current native WebChat uses one normal window controller, native `/new` resets the selected session, and Control UI New Chat is blocked during active runs or queued messages. No live desktop recording was performed.

## Solution Assessment

No current implementation fully solves the issue. The maintainable direction is a product-scoped per-window/per-session controller model using existing session create/switch APIs instead of relying on reset semantics.
