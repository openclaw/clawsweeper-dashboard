## Summary

Keep open. Current main and v2026.5.22 still do not provide first-class parallel desktop/window sessions: native WebChat keeps one normal window controller, native `/new` still resets the selected session, and Control UI still blocks New Chat while an active run or queue exists. The linked merged work addressed adjacent WebChat create/switch and `/new` reset semantics, but not this central multi-window concurrency request.

## What This Changes

The issue requests multi-window parallel desktop sessions so creating a new conversation window does not pause, reset, or interrupt existing active sessions.

## Best Possible Solution

Define a first-class per-window/per-session desktop flow backed by `sessions.create`, keep native controllers keyed by session/window, preserve active runs by default, and reserve reset behavior for explicit reset commands.

## Reproduction Assessment

Yes, at source level: current main stores one normal native WebChat window controller, native `/new` calls the reset path, and Control UI returns the active-run message before `sessions.create` during active work. I did not run a live macOS desktop repro in this read-only review.

## Solution Assessment

No current implementation fully solves the issue. The maintainable direction is a product-scoped per-window/per-session controller model using existing session create/switch APIs instead of relying on reset semantics.
