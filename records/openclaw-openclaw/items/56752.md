## Summary

Keep open: current main and the latest release still keep macOS WebChat to one normal chat window at a time, and the native macOS chat transport still falls back from unsupported session creation to resetting the current session. No linked merged PR or canonical related item now owns the requested parallel desktop-window session behavior.

## What This Changes

Request multi-window desktop chat sessions so opening a new conversation window does not pause, reset, or interrupt an active session.

## Best Possible Solution

Define the desktop window/session ownership model, then implement macOS chat windows keyed by session with createSession support and regression coverage proving an active run in one window survives opening another.

## Reproduction Assessment

Yes, at source level: WebChatManager owns one normal window and the macOS chat transport lacks createSession, causing /new to fall back to reset. I did not run a live macOS desktop repro in this read-only pass.

## Solution Assessment

No current implementation solves the request. The best path is maintainer-approved desktop session/window ownership followed by focused macOS create-session, multi-window routing, and regression coverage.
