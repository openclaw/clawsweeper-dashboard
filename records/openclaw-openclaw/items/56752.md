## Summary

Keep open: current main and the latest release still do not provide independent macOS chat windows, and the macOS Swift chat path still falls back from unsupported session creation to resetting the current session.

## What This Changes

Request multi-window desktop chat sessions so opening a new conversation window does not pause, reset, or interrupt an existing active session.

## Best Possible Solution

Define the desktop multi-window/session ownership model, then make macOS chat windows keyed by session and implement sessions.create support with tests proving an active run in one window survives opening another.

## Reproduction Assessment

Yes, at source level: WebChatManager closes the existing normal chat window when switching session keys, and the Swift chat model falls back from unsupported createSession to resetSession on /new in the macOS transport. I did not run a live macOS desktop repro in this read-only review.

## Solution Assessment

No current implementation solves the issue. The best path is a maintainer-approved desktop session/window model, then a focused macOS implementation with create-session support instead of more reset behavior.
