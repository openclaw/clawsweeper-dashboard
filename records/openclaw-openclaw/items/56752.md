## Summary

Keep open: current main and v2026.6.10 still keep macOS WebChat to one normal chat window at a time, and macOS /new still falls back from unsupported session creation to resetting the current session. The requested parallel desktop-window session behavior remains missing and needs maintainer product direction before implementation.

## What This Changes

Request desktop multi-window chat sessions so opening a new OpenClaw conversation window does not pause, reset, or interrupt an existing active session.

## Best Possible Solution

Define the desktop session/window ownership model, then implement macOS WebChat windows keyed by session with createSession support and regression coverage proving an active run in one window survives opening another.

## Reproduction Assessment

Yes, at source level: WebChatManager owns one normal window and the macOS chat transport lacks createSession, causing /new to fall back to reset. I did not run a live macOS desktop reproduction in this read-only pass.

## Solution Assessment

No current implementation solves the request. The best path is maintainer-approved desktop session/window ownership followed by focused macOS create-session, multi-window routing, and regression coverage.
