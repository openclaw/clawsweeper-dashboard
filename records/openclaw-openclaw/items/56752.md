## Summary

Keep open. Current main still has the native desktop gap described here: normal WebChat uses one window controller, native `/new` still resets the selected session, and the Control UI session-creation path still refuses switching while a run or queue is active.

## What This Changes

Request multi-window parallel desktop sessions so opening or starting another conversation does not pause, reset, or interrupt an existing active session.

## Best Possible Solution

Define a first-class per-window/per-session desktop flow backed by `sessions.create`, keep native window controllers keyed by session, and reserve reset behavior for explicit resets of the selected session.

## Reproduction Assessment

No live high-confidence reproduction was performed. Source inspection is strong: native normal chat still replaces the single window controller for a different session, native `/new` still calls `sessions.reset`, and Control UI blocks new-session switching while a run or queue is active.

## Solution Assessment

No current implementation solves the desktop ask. The maintainable direction is a product-scoped per-window/per-session controller model using `sessions.create`, not relying on reset semantics for the active session.
