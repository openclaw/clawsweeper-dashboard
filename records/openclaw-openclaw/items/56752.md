## Summary

Keep open. Current main still has one normal native chat window controller, reset-backed `/new` paths, and only a backend `sessions.create` primitive that is not wired into an independent desktop multi-window flow; the provided discussion has no maintainer close decision or merged fixing PR.

## What This Changes

Request that OpenClaw desktop support multiple independent conversation windows/sessions so opening a new conversation does not pause or interrupt an existing active session.

## Best Possible Solution

Add an explicit desktop/control UI flow for independent conversation windows that creates or selects distinct session keys, keeps one chat controller per visible session/window, and uses existing non-blocking `chat.send` plus `sessions.create` rather than reset semantics for ordinary parallel work.

## Reproduction Assessment

Unclear for live reproduction: I did not run the macOS app, but the static reproduction path is strong. Opening or switching native chat goes through a single `WebChatManager` window controller, while `/new` and reset paths call `sessions.reset`, whose tests prove active work is aborted and runtime state is cleared.

## Solution Assessment

No, the current reset-backed new-session behavior is not the best way to solve parallel conversations. The maintainable direction is a per-window/per-session controller model backed by `sessions.create` and existing non-blocking chat sends.
