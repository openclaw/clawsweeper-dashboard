## Summary

Keep open: current main still has single normal macOS webchat window ownership and the shared Swift /new path still falls back to resetting the current session when session creation is unsupported, so the reported multi-window session behavior remains unimplemented.

## What This Changes

The issue requests desktop support for multiple active conversation windows so opening a new window does not pause, reset, or interrupt an existing session.

## Best Possible Solution

Define the desktop window/session ownership model, then implement macOS chat windows keyed by session with createSession support and regression coverage proving an active run in one window survives opening another.

## Reproduction Assessment

Yes, at source level: WebChatManager owns only one normal webchat window and the shared Swift /new path falls back to resetting when createSession is unsupported. I did not run a live macOS desktop repro in this read-only review.

## Solution Assessment

No current implementation solves the issue. The best path is a maintainer-approved desktop session/window model followed by focused macOS create-session and multi-window routing support.
