## Summary

Keep open. Current main has shipped the WebChat create/switch slice, but it still does not provide the requested desktop multi-window parallel-session behavior: native WebChat keeps one normal window controller per manager, native `/new` resets the selected session, and Control UI still blocks New session while a run or queue is active.

## What This Changes

The issue asks for OpenClaw desktop to support multiple concurrent conversation windows without pausing, interrupting, or resetting existing active sessions.

## Best Possible Solution

Define a first-class per-window/per-session desktop flow backed by `sessions.create` and session switching, keep active runs attached to their original session, and reserve reset behavior for explicit reset commands with documented concurrency limits.

## Reproduction Assessment

Yes, at source level: native WebChat still has a single normal window controller and `/new` reset semantics, while the Control UI helper blocks New session during active runs or queued messages. No live desktop recording was performed.

## Solution Assessment

No current implementation fully solves the issue. The maintainable direction is a product-scoped per-window/per-session controller model using the existing session create/switch APIs instead of relying on reset semantics.
