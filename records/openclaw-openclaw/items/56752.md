## Summary

Keep open: current main still has single normal macOS webchat window ownership and the native chat /new path still falls back to resetting the current session when createSession is unsupported, so the requested parallel desktop-window session behavior remains unimplemented.

## What This Changes

Request support for multiple OpenClaw desktop chat windows whose active sessions continue running independently when a new window or conversation is opened.

## Best Possible Solution

Define the desktop window/session ownership model, then implement macOS chat windows keyed by session with createSession support and regression coverage proving an active run in one window survives opening another.

## Reproduction Assessment

Yes, at source level: WebChatManager owns only one normal window and the native /new path can fall back to sessions.reset when createSession is unsupported. I did not run a live macOS desktop repro in this read-only pass.

## Solution Assessment

No current implementation solves the issue. The best path is a maintainer-approved desktop session/window model followed by focused macOS create-session and multi-window routing support.
