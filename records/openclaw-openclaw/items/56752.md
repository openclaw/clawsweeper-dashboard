## Summary

Keep open. Current main still lacks the requested desktop multi-window parallel-session behavior: native WebChat keeps one normal window controller, native `/new` still resets the selected session, and Control UI New Chat remains blocked while a run or queue is active.

## What This Changes

Request desktop multi-window parallel sessions so opening a new conversation window does not pause or interrupt existing active sessions.

## Best Possible Solution

Define a first-class per-window/per-session desktop flow backed by `sessions.create`, keep native window controllers keyed by session, and reserve reset behavior for explicit reset commands.

## Reproduction Assessment

Yes, at source level: current native WebChat uses one normal window controller, native `/new` resets the selected session, and Control UI New Chat is blocked during active runs or queued messages. No live desktop recording was performed.

## Solution Assessment

No current implementation fully solves the issue. The maintainable direction is a product-scoped per-window/per-session controller model using existing session create/switch APIs instead of relying on reset semantics.
