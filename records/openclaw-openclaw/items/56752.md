## Summary

Keep open. Current main and the latest shipped v2026.5.22 release still block or reset the paths this request cares about: Control UI New Chat is disabled during active work, native WebChat keeps one normal window controller per session, and typed native `/new` still routes through session reset behavior.

## What This Changes

Request first-class multi-window parallel desktop sessions so opening a new conversation window or session does not pause, reset, or interrupt an existing active session.

## Best Possible Solution

Define a first-class per-window/per-session desktop flow backed by `sessions.create`, keep native controllers keyed by session/window, preserve existing active runs by default, and reserve reset behavior for explicit reset commands.

## Reproduction Assessment

Yes, at source level: current main keeps singleton native window controllers, maps native `/new` to reset, and returns an active-run error before Control UI creates a new session. I did not run a live macOS desktop repro in this read-only review.

## Solution Assessment

No current implementation fully solves the issue. The maintainable direction is a product-scoped per-window/per-session controller model using existing session create/switch APIs instead of relying on reset semantics.
