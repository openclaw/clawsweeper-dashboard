## Summary

Keep open. Current main has a backend and Control UI session-creation path, but native desktop still keeps one normal WebChat window controller and native `/new` still resets the active session, so the requested parallel multi-window desktop flow is not implemented and needs product direction.

## What This Changes

Requests desktop multi-window conversations that create independent sessions without pausing or resetting active sessions.

## Best Possible Solution

Define a first-class multi-window session flow that creates or selects distinct session keys per conversation window, keeps native window controllers keyed by session, and preserves active runs unless the user explicitly resets that same session.

## Reproduction Assessment

No live high-confidence reproduction was performed. Source inspection is strong: native normal chat still replaces the single window controller for a different session, and native `/new` still calls `sessions.reset`.

## Solution Assessment

No current solution is in place for the native desktop ask. The maintainable direction is a product-scoped per-window/per-session controller model backed by `sessions.create`, not reset semantics on the active session.
