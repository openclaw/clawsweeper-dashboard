## Summary

Keep open: current main still enforces a single macOS webchat window and the macOS `/new` path falls back to resetting the current session instead of creating a parallel one, with no merged canonical PR or issue superseding this request.

## What This Changes

Request multi-window desktop chat support so a new OpenClaw conversation window does not pause or interrupt an existing active session.

## Best Possible Solution

Define the desktop product behavior first, then make macOS chat windows keyed by session and implement session creation without resetting or stealing active-run subscriptions from existing windows.

## Reproduction Assessment

Yes, at source level but not via a live desktop run: WebChatManager closes the previous chat window for a different session, and `/new` falls back to reset on macOS when createSession is unsupported. That explains the reported interruption path enough to keep the issue open.

## Solution Assessment

No current implementation solves the request. The best path is not a blind small fix; maintainers should decide the intended multi-window/session-routing model, then update macOS window ownership and create-session support with focused tests.
