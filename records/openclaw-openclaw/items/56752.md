## Summary

Keep open. Current main still exposes a single normal native WebChat window controller, and the visible new-session/reset paths still reset the active session rather than creating an independent parallel conversation window. The backend has a useful `sessions.create` primitive, but it is not wired into a desktop multi-window product flow, so this remains a valid feature request needing maintainer product direction.

## What This Changes

Request independent desktop conversation windows so opening or starting a second conversation does not pause, replace, or reset an existing active session.

## Best Possible Solution

Add an explicit multi-window/session flow that creates or selects distinct session keys for each conversation window, keeps one controller per visible native window or browser session view, and uses `sessions.create` plus non-blocking `chat.send` rather than reset semantics for ordinary parallel work.

## Reproduction Assessment

Unclear for live reproduction because I did not run the macOS app, but the static reproduction path is strong: opening a different native WebChat session replaces the single normal controller, while `/new` and reset paths still hit `sessions.reset`, whose tests prove it aborts active work for that session.

## Solution Assessment

No current solution is in place. Reset/new-session behavior is not the right fix for parallel conversations; the maintainable direction is a per-window/per-session controller model backed by the existing session creation and non-blocking chat primitives.
