## Summary

Keep open: current main still has a source-level mismatch between the documented trusted-proxy local password fallback and the auth pipeline, and the existing tests currently encode the rejecting behavior rather than the documented fallback.

## What This Changes

The issue reports that trusted-proxy gateway auth rejects same-host password fallback requests before the documented loopback fallback can run.

## Best Possible Solution

Adjust the trusted-proxy auth path so a no-forwarded-header loopback request with a valid configured password can authenticate locally, while proxy/forwarded evidence still runs through trusted-proxy requiredHeaders and allowUsers checks.

## Reproduction Assessment

Yes, by source inspection: the trusted-proxy branch returns the required-header or loopback failure before the password-auth branch can run, even for localDirect loopback requests with connectAuth.password. I did not run a live gateway in this read-only sweep.

## Solution Assessment

Yes, the narrow maintainable fix is to handle the documented localDirect password fallback before trusted-proxy requiredHeaders only when no forwarded-header evidence is present. Add focused coverage in src/gateway/auth.test.ts for the accepted fallback and the forwarded-header fail-closed cases.
