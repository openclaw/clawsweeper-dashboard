## Summary

Keep open: this is a real gateway auth contract mismatch. Current main and v2026.5.12 reject trusted-proxy local password fallback, but docs and credential planning still describe or select that fallback after a recent maintainer decision changed the runtime to fail closed.

## What This Changes

The issue reports that trusted-proxy gateway auth with `allowLoopback=true` rejects same-host callers with a valid `gateway.auth.password` even though the docs describe a local-direct password fallback.

## Best Possible Solution

Confirm the fail-closed trusted-proxy policy from https://github.com/openclaw/openclaw/issues/78684; if it stands, align docs and client credential planning with it, otherwise deliberately restore the fallback with focused fail-closed coverage.

## Reproduction Assessment

Yes. Source and v2026.5.12 release inspection show trusted-proxy failures return before password auth can run, while docs still describe a local-direct password fallback; I did not run a live gateway in this read-only review.

## Solution Assessment

No for the suggested runtime fallback as-is. The safer path is to first resolve the policy conflict with the recent fail-closed decision, then either align docs/credential planning to fail closed or restore fallback with explicit auth-boundary tests.
