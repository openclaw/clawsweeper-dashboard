## Summary

Keep open. Current main still has the ACP lifecycle gap this PR targets, and the gateway-disconnect recovery direction is narrow, but the patch is not merge-ready because the spawn health probe only handles thrown `getStatus` failures and still accepts resolved `dead` or `no-session` statuses; the live PR metadata also reports the branch as dirty.

## What This Changes

This PR adds ACP post-spawn status probing, allowlisted stale gateway-disconnect recovery for cancel/close/early-turn paths, and focused ACP manager plus `/acp` command regression tests.

## Best Possible Solution

Revise the spawn health guard to inspect the resolved `AcpRuntimeStatus` with the same `dead`/`no-session` classification used by `AcpSessionManager`, run the existing failed-spawn cleanup for unhealthy statuses, preserve the explicit gateway close-code allowlist, and add a `/acp spawn` regression for resolved unhealthy status results.

## Reproduction Assessment

Yes. The original bug and the remaining patch bug are reproducible with mocked ACP runtime paths: make spawn-time `runtime.getStatus` resolve `{ summary: "status=dead", details: { status: "dead" } }` and the current PR path will still bind/persist instead of cleaning up.

## Solution Assessment

No, not yet. The recovery approach is narrow and consistent with existing ACP manager behavior, but the spawn health probe must reject resolved unhealthy statuses and the branch needs rebase validation before this is the best maintainable fix.
