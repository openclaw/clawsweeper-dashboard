## Summary

Keep open. Current main still computes TaskFlow audit freshness from the flow record alone, and managed flows intentionally do not mirror child task activity into the flow timestamp, so the PR addresses a real false-positive path. The code change is narrow and aligned with the existing task-audit timestamp precedence; I found no functional or security blocker, but the PR still needs the required user-facing changelog entry and current-main validation before merge.

## What This Changes

The PR changes TaskFlow audit freshness to consider the latest timestamp across the flow record and linked task activity, with regression coverage for suppressing stale-running findings when a managed flow has an active linked child task.

## Best Possible Solution

Land a current-main version of this narrow audit-only fix with the regression test and a single-line changelog entry. Keep managed-flow lifecycle synchronization semantics unchanged unless maintainers choose a broader TaskFlow product change separately.

## Reproduction Assessment

Yes. On current main, a managed running flow with a stale `updatedAt` and a linked running task with recent `lastEventAt` still computes `stale_running` age from the flow timestamp only; the PR's new test fixture is a high-confidence reproduction path, though I did not execute it in this read-only review.

## Solution Assessment

Yes, with one process fix. Using linked task activity only for TaskFlow audit freshness is the narrowest maintainable solution because managed flows deliberately do not sync child task activity into the flow record; the PR should add the required changelog entry before merge.
