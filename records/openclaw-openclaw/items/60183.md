## What This Changes

The PR changes TaskFlow audit freshness to consider linked task activity timestamps and adds a regression test for suppressing false `stale_running` findings.

## Best Possible Solution

Land a refreshed PR that uses linked child activity to avoid false TaskFlow `stale_running` diagnostics while explicitly preserving the intended active-vs-terminal freshness rule and adding real audit proof.

## Reproduction Assessment

Yes, source-reproducible: current main computes the TaskFlow stale-running age from the flow timestamp before reading linked tasks, so an active child task with newer `lastEventAt` can still leave the parent flow flagged stale. I did not run tests because this was a read-only review.

## Solution Assessment

Mostly yes: using linked task activity in the audit path is the narrow owner-boundary fix because managed flows are intentionally not mirrored from child task progress. The remaining best-fix question is whether the freshness helper should include all linked tasks or only active linked tasks.
