## Summary

Keep open: current main still computes TaskFlow audit staleness from the parent flow timestamp, and the PR's code direction matches the existing task activity contract, but this external PR still lacks after-fix real behavior proof and fresh current-base validation before merge.

## What This Changes

The PR updates TaskFlow audit freshness to consider linked task activity and adds regression coverage for suppressing `stale_running` on active managed flows.

## Best Possible Solution

Land the narrow freshness fix after redacted after-fix CLI proof and current-base validation, with maintainers accepting or narrowing the all-linked-task freshness semantics.

## Reproduction Assessment

Yes, source-reproducible: current main computes TaskFlow staleness from the parent flow timestamp while managed flows do not mirror linked child progress. A linked child task can therefore have recent activity while the parent flow still produces `stale_running`.

## Solution Assessment

Yes for the code direction: using linked task activity and the existing task timestamp precedence is narrower than changing managed-flow sync semantics. If maintainers want stricter semantics, the only likely adjustment is limiting freshness to active linked tasks.
