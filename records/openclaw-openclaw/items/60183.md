## Summary

Keep open: current main still has the TaskFlow audit false-positive path and this PR is a focused fix, but it is not merge-ready without real behavior proof, a current-main refresh, and maintainer confirmation of linked-task freshness semantics.

## What This Changes

The PR changes TaskFlow audit stale-age calculation to include linked task activity timestamps and adds a regression test for an active child suppressing `stale_running`.

## Best Possible Solution

Refresh the branch, add redacted real `openclaw tasks audit` proof, and land a linked-task freshness rule that explicitly matches the chosen active-only or all-linked semantics.

## Reproduction Assessment

Yes, source-reproducible: current main computes TaskFlow stale age from the flow timestamp before loading linked tasks, while child task progress updates `lastEventAt` on the linked task. I did not run tests because this was a read-only review.

## Solution Assessment

Mostly yes: using linked task activity in TaskFlow audit is the right owner-boundary for managed flows that do not sync from child task progress. The unresolved best-fix question is whether freshness should include only active linked tasks or all linked tasks.
