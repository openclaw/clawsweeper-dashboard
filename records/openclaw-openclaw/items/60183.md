## Summary

Keep open: current main and the latest release still compute TaskFlow audit freshness from the parent flow timestamp only, while this PR is a focused source-reproducible fix. There are no blocking code findings, but it still needs contributor real-behavior proof, a current-base refresh, and maintainer acceptance of the all-linked-task freshness semantics.

## What This Changes

The PR changes TaskFlow audit freshness to use the most recent linked task activity timestamp and adds a regression test suppressing `stale_running` for active managed flows.

## Best Possible Solution

Merge a refreshed version after redacted real CLI proof and explicit acceptance or narrowing of the all-linked-task freshness semantics; keep terminal child reconciliation in the newer task-maintenance work.

## Reproduction Assessment

Yes, source-reproducible: current main calculates TaskFlow stale age from `flow.updatedAt ?? flow.createdAt` before linked-task activity is considered, so a running managed flow with a recent child `lastEventAt` can still cross the stale-running threshold.

## Solution Assessment

Yes, mostly: computing freshness in the audit owner path from linked task activity is the narrow fix. The open maintainer choice is whether terminal linked tasks should also count or whether the helper should use active linked tasks only.
