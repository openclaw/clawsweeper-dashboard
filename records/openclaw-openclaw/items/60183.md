## Summary

Keep open: current main and the latest release still compute TaskFlow audit freshness from the flow record only, and this PR is a focused bug-fix candidate. It is not merge-ready yet because the external PR has only test/CI proof, needs a current-main refresh for Dependency Guard, and still needs maintainer confirmation that freshness should consider all linked tasks rather than only active linked tasks.

## What This Changes

This PR changes TaskFlow audit staleness to compute the reference timestamp from the flow plus linked task activity and adds regression coverage for an active child task.

## Best Possible Solution

Refresh the branch on current main, keep the fix inside TaskFlow audit, add redacted before/after `openclaw tasks audit` output from a real affected ledger, and settle whether freshness should use active linked tasks or all linked task activity.

## Reproduction Assessment

Yes at source level: current main computes TaskFlow stale age from `flow.updatedAt`/`createdAt` before linked tasks are loaded, while child task progress is stored on `TaskRecord.lastEventAt`. I did not run tests because this review is read-only.

## Solution Assessment

Mostly yes: TaskFlow audit is the right owner boundary because managed child task progress is not mirrored onto the flow record. The remaining best-fix question is whether the reference timestamp should include all linked tasks or only active linked tasks.
