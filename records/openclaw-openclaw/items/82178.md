## Summary

Keep open: current main still implements and documents the process-denied path as synchronous, while the linked PR proposes a real tool-policy contract change that needs maintainer decision and real behavior proof before landing.

## What This Changes

Request design approval for finite-timeout `exec` auto-backgrounding when the `process` follow-up tool is hidden by policy.

## Best Possible Solution

Have maintainers decide the restricted-tool policy on the linked PR, require real process-denied runtime proof if the direction is accepted, and keep any true detached-job work separate.

## Reproduction Assessment

Yes for the current-main behavior: source, docs, and an existing test show that denying `process` makes `exec` run synchronously despite `yieldMs`. This is not a strict bug reproduction because current docs define that behavior today.

## Solution Assessment

Unclear until maintainer review: the linked PR is a focused implementation for finite-timeout hidden-process backgrounding, but it changes a documented tool-policy contract and should land only after policy approval and real behavior proof.
