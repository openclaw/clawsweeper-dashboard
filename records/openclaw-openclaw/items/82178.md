## Label Justifications

- `P2`: This is a normal-priority agent/tool execution design issue with real but bounded impact on restricted-tool long-running exec workflows.
- `impact:security`: The issue concerns whether hiding the process-management tool should still allow background host processes, which is a tool-policy security boundary.
- `impact:crash-loop`: The motivating failure is long-running exec work pinning a parent run until timeout and becoming vulnerable to process termination.

## Summary

Keep open: current main and the latest release still intentionally couple `exec` backgrounding to `process` visibility, while the only linked implementation PR is closed unmerged, so this remains a restricted-tool policy/security decision rather than cleanup.

## What This Changes

The issue asks whether finite-timeout `exec` backgrounding should remain available when policy hides the separate `process` management tool.

## Best Possible Solution

Settle the restricted-tool contract first, then land a current-main PR with the agent-tool policy change, timeoutless guard, aligned docs/tests, and real process-denied behavior proof if maintainers accept the new behavior.

## Reproduction Assessment

Yes for the current-main behavior: source, docs, and a focused test show `tools.deny: ["process"]` makes `exec` complete synchronously despite `yieldMs`. This is not a strict bug reproduction because current docs define that behavior today.

## Solution Assessment

Unclear until maintainer review: decoupling finite-timeout backgrounding from `process` is a plausible focused direction, but the best landing shape depends on an explicit restricted-tool security/product decision.
