## Label Justifications

- `P2`: This is a normal-priority agent/tool execution design issue with real but bounded impact on restricted-tool long-running exec workflows.
- `impact:security`: The issue concerns whether hiding the process-management tool should still allow background host processes, which is a tool-policy security boundary.
- `impact:crash-loop`: The motivating failure is long-running exec work pinning a parent run until heartbeat or run timeout and becoming vulnerable to termination.

## Summary

Keep open: current main and the latest release still couple `exec` backgrounding to `process` visibility, and the linked implementation PR is closed unmerged, so this remains a maintainer security/product decision rather than cleanup.

## What This Changes

Request to allow finite-timeout `exec` backgrounding when the `process` tool is hidden, while rejecting timeoutless hidden-process backgrounding and updating docs/tests if accepted.

## Best Possible Solution

Settle the restricted-tool contract first, then land a current-main PR with the policy change, timeoutless guard, completion-visibility behavior, aligned docs/tests, and real process-denied proof if maintainers accept it.

## Reproduction Assessment

Yes for the current-main behavior: source, docs, and focused tests show `tools.deny: ["process"]` makes `exec` complete synchronously despite `yieldMs`. This is not a strict bug reproduction because current docs define that behavior today.

## Solution Assessment

Unclear until maintainer review: decoupling finite-timeout backgrounding from `process` is a plausible focused direction, but the best landing shape depends on an explicit restricted-tool security/product decision.
