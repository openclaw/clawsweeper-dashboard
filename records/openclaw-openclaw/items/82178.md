## Label Justifications

- `P2`: This is a normal-priority agent/tool execution design issue with real but bounded impact on restricted-tool long-running exec workflows.
- `impact:security`: The issue explicitly concerns whether a denied process-management tool should still allow background host processes, which is a tool-policy security boundary.
- `impact:crash-loop`: The motivating failure is long-running exec work pinning a parent run until heartbeat/run timeout and becoming vulnerable to termination.

## Summary

Keep open: current main and the latest release still implement and document the process-denied path as synchronous, and the linked implementation PR was closed unmerged, so the remaining work is a real restricted-tool policy decision rather than cleanup.

## What This Changes

The issue requests a decision on allowing finite-timeout `exec` backgrounding when the separate `process` follow-up tool is hidden, while rejecting timeoutless hidden-process backgrounding.

## Best Possible Solution

Settle the restricted-tool contract first, then land a fresh current-main PR with the `agent-tools` policy change, timeoutless guard, completion-visibility decision, docs, and regression coverage if maintainers accept it.

## Reproduction Assessment

Yes for the current-main behavior: source, docs, and an existing test show `tools.deny: ["process"]` makes `exec` complete synchronously despite `yieldMs`. This is not a strict bug reproduction because current docs define that behavior today.

## Solution Assessment

Unclear until maintainer review: the closed linked PR was a focused candidate, but the best landing shape needs an explicit restricted-tool policy decision and current-main implementation over `agent-tools`.
