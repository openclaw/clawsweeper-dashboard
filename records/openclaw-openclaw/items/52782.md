## Summary

Keep open. The reported transcript cleanup mismatch is still real on current main, but the proposed one-line fix is stale against the extracted reset helper and remains incomplete for restored or thread-scoped sessions where the topic id is derived from the session key rather than the current message context.

## What This Changes

This PR changes reset transcript cleanup to pass the current message thread id into the fallback transcript path resolver.

## Best Possible Solution

Fix the current reset helper so fallback transcript cleanup uses the same topic id source as session creation: active message thread id first, then the thread-scoped session key, with persisted thread fields considered only when the session is known to be thread-scoped. Add focused reset-helper coverage for missing or stale `sessionFile`, session-key-derived topics, numeric thread ids, and non-thread sessions with stale thread metadata.

## Reproduction Assessment

Yes. A focused unit test can call `resetReplyRunSession` with `cleanupTranscripts: true`, a topic-scoped session key or message thread id, and a missing/stale `prevEntry.sessionFile`; current main will add the non-topic fallback path and leave the `*-topic-*` transcript behind.

## Solution Assessment

No. Passing only the current `MessageThreadId` was the right direction for one flow, but the maintainable fix belongs in the extracted reset helper and should mirror session creation’s thread-id resolution.
