## Summary

Keep open. The branch targets a real topic-transcript cleanup problem, but it is stale against current main and only patches the old inline call site while current resets go through `resetReplyRunSession`; it also still lacks real Telegram/topic behavior proof.

## What This Changes

The PR changes `runReplyAgent` to pass `sessionCtx.MessageThreadId` into `resolveSessionTranscriptPath` when deleting the previous transcript after a reset.

## Best Possible Solution

Retarget the fix to `resetReplyRunSession`: derive the cleanup topic id from the active message thread or parsed thread-scoped session key, preserve string and numeric topic ids, and add focused regression coverage.

## Reproduction Assessment

Yes, source-level. Call `resetReplyRunSession` with `cleanupTranscripts: true`, a topic-scoped `sessionKey`, no active `messageThreadId`, and no usable `sessionFile`; current main adds only the bare transcript fallback.

## Solution Assessment

No. Passing only `sessionCtx.MessageThreadId` at the stale inline call site is partial; the maintainable fix belongs in the current reset helper and should mirror session creation's active-or-session-key topic resolution.
