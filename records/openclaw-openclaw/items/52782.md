## Summary

Keep open: this is a plausible narrow bug fix, but the branch no longer touches the active reset path on current main. The current helper still has a topic cleanup gap, and the PR also lacks after-fix real behavior proof.

## What This Changes

The branch changes `runReplyAgent` to pass `sessionCtx.MessageThreadId` into `resolveSessionTranscriptPath` when deleting the previous transcript after a reset.

## Best Possible Solution

Fix the current reset helper to derive the cleanup topic id from the active message thread or the parsed thread-scoped session key, preserve string and numeric topic ids, and add focused regression coverage for missing or stale topic transcript metadata.

## Reproduction Assessment

Yes, source-level. Call `resetReplyRunSession` with `cleanupTranscripts: true`, a topic-scoped `sessionKey`, no active `messageThreadId`, and no usable `sessionFile`; current main adds only the bare transcript fallback.

## Solution Assessment

No. Passing only the active message thread id at the stale inline call site is partial; the maintainable fix belongs in the current reset helper and should mirror session creation's active-or-session-key topic resolution.
