## Summary

Keep open. The reported cleanup bug is still source-reproducible, but this PR is stale against current main because reset cleanup now runs through `resetReplyRunSession`, and the PR also lacks after-fix real Telegram/topic proof.

## What This Changes

The PR adds `sessionCtx.MessageThreadId` as the optional topic id when the previous transcript cleanup fallback resolves a transcript path in `runReplyAgent`.

## Best Possible Solution

Retarget topic-aware cleanup into `resetReplyRunSession`, derive the cleanup topic id from the active message thread or parsed topic-scoped session key while preserving string and numeric ids, add focused regression coverage, and include redacted Telegram/topic runtime proof.

## Reproduction Assessment

Yes, source-level. On current main, `resetReplyRunSession` with `cleanupTranscripts: true`, a topic-scoped `sessionKey`, no active `messageThreadId`, and a missing or stale `sessionFile` still adds only the bare transcript fallback.

## Solution Assessment

No. The one-line PR edits the old inline `runReplyAgent` cleanup site, while current main should solve this in `resetReplyRunSession` and mirror session creation's active-or-session-key topic resolution.
