## Summary

Keep this PR open: it points at a real transcript cleanup bug, but the patch is partial against current main because reset cleanup now lives in `resetReplyRunSession`, and the PR still lacks real Telegram/topic runtime proof.

## What This Changes

The PR adds `sessionCtx.MessageThreadId` to the transcript cleanup fallback in `runReplyAgent` when deleting a previous session transcript after reset.

## Best Possible Solution

Retarget the fix to `resetReplyRunSession`: derive the cleanup topic id from the active message thread or parsed thread-scoped session key, preserve string and numeric topic ids, add focused regression coverage, and provide redacted real Telegram/topic runtime proof.

## Reproduction Assessment

Yes, source-level: call `resetReplyRunSession` with `cleanupTranscripts: true`, a topic-scoped `sessionKey`, no active `messageThreadId`, and no usable `sessionFile`; current main adds only the bare transcript fallback.

## Solution Assessment

No. The one-line change targets the old inline fallback and only uses active `MessageThreadId`; the maintainable fix belongs in the current reset helper and should mirror session creation's active-or-session-key topic resolution.
