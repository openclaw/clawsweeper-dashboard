## Summary

Keep this PR open: the reported stale-transcript cleanup bug is real, and the Greptile/root-cause discussion correctly identifies the direct Telegram topic path mismatch, but the branch is incomplete against current main. Current main has moved reset cleanup into `agent-runner-session-reset.ts`, where the fallback still resolves the previous transcript without any topic id; the PR's older one-line fix also only covers active `MessageThreadId`, while restored or ACP-bound topic sessions can derive their topic from the thread-scoped session key.

## What This Changes

This PR adds `sessionCtx.MessageThreadId` to the reset transcript cleanup fallback in `src/auto-reply/reply/agent-runner.ts` so direct topic sessions resolve a topic-suffixed transcript path.

## Best Possible Solution

Port the cleanup fix into the current reset helper so previous-transcript fallback cleanup resolves topic ids the same way session creation and restored dispatch do: active message thread id first, then the thread-scoped session key, without resurrecting stale thread metadata for non-thread sessions.

## Reproduction Assessment

Yes. A focused unit test can call `resetReplyRunSession` with `cleanupTranscripts: true`, a topic-scoped session key, and a missing or stale `prevEntry.sessionFile`; current main will include the non-topic fallback path and leave the `*-topic-*` transcript behind.

## Solution Assessment

No. Passing only `sessionCtx.MessageThreadId` is a useful partial fix for direct topic turns, but the maintainable fix belongs in the extracted reset helper and must also handle session-key-derived topic ids and numeric thread ids.
