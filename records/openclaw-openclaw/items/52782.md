## Summary

Keep open: the underlying transcript-cleanup bug is valid, but this PR is stale/incomplete against current main. Reset cleanup now lives in `resetReplyRunSession`, where the fallback still omits the topic id and also needs the same session-key thread fallback used by session creation/restored dispatch flows.

## What This Changes

The PR adds `sessionCtx.MessageThreadId` to the reset transcript cleanup fallback path in `src/auto-reply/reply/agent-runner.ts`.

## Best Possible Solution

Update `resetReplyRunSession` to resolve cleanup topic ids from the active message thread id or the thread-scoped session key, and add focused regression coverage for missing or stale topic transcript cleanup.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `resetReplyRunSession` with `cleanupTranscripts: true`, a topic-scoped session key, and a missing or stale previous `sessionFile`; current main will add the non-topic fallback path and leave the topic-suffixed transcript behind.

## Solution Assessment

No. Passing only the active `MessageThreadId` at the old inline call site is not the narrowest current fix; the current helper should derive the topic id the same way session creation and dispatch do.
