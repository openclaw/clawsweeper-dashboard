## Summary

Keep open. The underlying bug is source-reproducible, but this PR is not ready because it edits the old inline cleanup path while current main routes resets through `resetReplyRunSession`, and the PR still lacks real Telegram/topic behavior proof.

## What This Changes

The PR adds `sessionCtx.MessageThreadId` as the optional topic id when `runReplyAgent` resolves a previous transcript cleanup fallback.

## Best Possible Solution

Move topic-aware cleanup into `resetReplyRunSession`: derive the cleanup topic id from the active message thread or parsed thread-scoped session key, preserve string and numeric ids, and add focused regression coverage plus redacted Telegram/topic proof.

## Reproduction Assessment

Yes, source-level. Call `resetReplyRunSession` with `cleanupTranscripts: true`, a topic-scoped `sessionKey`, no active `messageThreadId`, and a missing or stale `sessionFile`; current main adds only the bare transcript fallback.

## Solution Assessment

No. The one-line PR edits the old inline `runReplyAgent` cleanup site, while current main should solve this in `resetReplyRunSession` and mirror session creation's active-or-session-key topic resolution.
