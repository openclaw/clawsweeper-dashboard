## Summary

Keep open. Current main still builds strict AND FTS queries for default unicode61 memory search, and the provider-backed hybrid path does not broaden keyword search after a zero-row MATCH, so the reported CJK `textScore: 0` behavior remains plausible and source-reproduced.

## What This Changes

The issue requests fixing built-in `memory_search` BM25/FTS scoring for multi-term Chinese/CJK queries so keyword matches contribute textScore instead of remaining zero.

## Best Possible Solution

Add a tokenizer-aware CJK/multi-term fallback in the memory keyword path so hybrid search retries useful per-term or OR-style lexical candidates when the strict MATCH returns no rows, while preserving precise English/exact-term behavior where possible.

## Reproduction Assessment

Yes. Current source builds the reported strict AND MATCH expression, and an in-memory SQLite FTS5 repro with the same tokenizer pattern shows individual Chinese terms match while the combined AND expression returns zero rows.

## Solution Assessment

The simple OR suggestion is directionally useful but not necessarily the safest complete fix. A bounded zero-hit or CJK-aware fallback in `searchKeyword`/hybrid search would fix recall without broadly changing all English exact-search semantics.
