## Summary

Close as a duplicate: an older open issue from the same reporter has the same title, body, reproduction details, and ClawSweeper review, so the remaining memory-search bug should be tracked there.

## What This Changes

The issue reports that BM25/FTS5 keyword scoring in `memory_search` returns `textScore: 0` for multi-term Chinese queries because the FTS query builder joins tokens with `AND`.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/83384 as the canonical tracker and fix the hybrid memory keyword fallback there.

## Reproduction Assessment

Yes. Current source builds the strict AND query, and a focused in-memory SQLite FTS5 probe reproduced zero keyword rows for the multi-term Chinese AND query while the OR form returned rows.

## Solution Assessment

Yes for triage: closing this duplicate while keeping https://github.com/openclaw/openclaw/issues/83384 open is the cleanest path. The implementation should avoid a blanket AND-to-OR flip and add a targeted zero-hit or CJK-aware fallback in the memory keyword path.
