## Summary

Close as a duplicate. The same author opened the same title/body first at https://github.com/openclaw/openclaw/issues/83384, that canonical thread is still open, and current source checks show the underlying CJK BM25 issue remains valid there.

## What This Changes

The issue reports that multi-term Chinese memory searches get vector hits but no BM25/FTS5 keyword score because the memory FTS query builder joins all terms with AND.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/83384 as the canonical tracking thread and implement a targeted memory keyword fallback there, preserving strict matching first while broadening empty multi-term CJK/default-tokenizer results.

## Reproduction Assessment

Yes at source level. Current main builds the strict AND MATCH expression, the hybrid path leaves vector-only rows with textScore 0 when that keyword query returns no rows, and an in-memory unicode61 FTS5 probe reproduced the AND-zero/OR-hit pattern.

## Solution Assessment

No as a standalone tracking item because the earlier duplicate should own the remaining work. For the code fix, an unconditional OR replacement is not clearly safest; a bounded CJK or zero-keyword-hit fallback is the narrower maintainable path.
