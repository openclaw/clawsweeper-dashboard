## Summary

Keep open: current main has merged the compaction model fallback slice, but the requested circuit breaker still has unresolved lossy recovery, health/status, and transport-isolation behavior.

## What This Changes

The issue requests compaction fallback routing, forced truncation after repeated failures, Telegram transport decoupling, and gateway status visibility for stuck compaction.

## Best Possible Solution

Define a core per-session compaction failure policy that tracks failures, applies approved fallback routing, performs documented lossy recovery after a threshold, exposes health through status APIs, and keeps channel transports resilient without Telegram-specific core policy.

## Reproduction Assessment

Yes for the remaining current-main behavior: source and tests show capped compaction retries followed by context_overflow or compaction_failure rather than forced recovery. The full historical Telegram outage loop has concrete logs but was not live-reproduced in this review.

## Solution Assessment

No; the current merged fallback implementation is a useful partial fix, but it is not the complete circuit breaker requested here. The maintainable direction is one cohesive failure policy plus status and channel-lifecycle follow-through.
