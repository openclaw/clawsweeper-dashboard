## Summary

Keep open. Current main has useful partial mitigations for compaction fallback, bounded overflow retries, tool-result truncation, and lane timeouts, but it still does not provide the complete repeated-failure circuit breaker, force-truncate escalation, transport isolation, or compaction health/status policy requested here.

## What This Changes

The issue requests compaction recovery controls: fallback model retry, force-truncation after repeated failures, Telegram transport isolation, and gateway status visibility.

## Best Possible Solution

Define a maintainer-approved compaction recovery policy that combines existing model fallback and timeout behavior with a session-scoped failure counter, bounded lossy truncation or reset escalation, and operator-visible health without corrupting active transcripts or queued channel delivery.

## Reproduction Assessment

Yes, at source level: current main still has the compaction wait/unsubscribe path, bounded repeated overflow compaction attempts, and no complete repeated-failure force-truncate or health-status circuit breaker. I did not run a live Telegram reproduction of the original outage.

## Solution Assessment

Unclear as a single implementation: fallback retry and force-truncate are plausible, but the safe fix needs a maintainer decision on thresholds, whether configuration is added, and how background or lossy recovery interacts with session-lane ordering.
