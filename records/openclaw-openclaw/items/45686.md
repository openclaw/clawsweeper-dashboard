## Summary

Keep open: current main includes the compaction model fallback slice, but the requested circuit breaker is still incomplete because failed compaction can still end in a blocked overflow/compaction error, teardown still aborts compaction waits, and status surfaces only compaction counts rather than failure/stuck health.

## What This Changes

The issue requests a compaction circuit breaker that falls back to another model, force-truncates after repeated failures, keeps Telegram polling alive during compaction, and exposes compaction health in gateway status.

## Best Possible Solution

Define a core per-session compaction failure policy that tracks repeated failures, applies approved fallback routing, performs documented lossy recovery after a threshold, exposes health through status APIs, and decouples channel transport lifecycle from compaction waits without Telegram-specific core policy.

## Reproduction Assessment

Yes for the remaining current-main behavior: source and tests show capped compaction retries followed by context_overflow or compaction_failure rather than forced recovery, and unsubscribe still aborts compaction waits. The exact historical Telegram outage loop was not live-reproduced in this read-only review.

## Solution Assessment

No; the current merged fallback implementation is a useful partial fix but not the complete circuit breaker requested here. The safer maintainable direction is one cohesive failure policy plus status and channel-lifecycle follow-through.
