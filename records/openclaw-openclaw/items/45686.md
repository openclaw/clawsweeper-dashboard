## Summary

Keep open: current main has the compaction fallback slice, but the broader circuit-breaker request still has unresolved lossy recovery, health/status, and channel-lifecycle behavior that require maintainer policy.

## What This Changes

The issue requests a compaction circuit breaker with fallback model use, forced truncation after repeated failures, transport isolation, and gateway status visibility.

## Best Possible Solution

Define one maintainer-approved per-session compaction failure policy that tracks repeated failures, reuses fallback routing, escalates to documented lossy recovery, exposes health through status APIs, and prevents channel transports from being torn down by compaction internals.

## Reproduction Assessment

Yes for the remaining current-main behavior: source and tests show bounded compaction retries followed by blocked context_overflow or compaction_failure, and unsubscribe still aborts compaction waits. The full historical Telegram outage loop has concrete logs but was not live-reproduced in this read-only review.

## Solution Assessment

No, current main is only a partial solution. The model fallback slice is useful, but the best maintainable fix is a cohesive failure policy plus status and channel-lifecycle follow-through rather than another isolated retry branch.
