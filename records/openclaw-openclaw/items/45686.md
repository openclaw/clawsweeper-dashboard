## Summary

Keep open. Current main has partial mitigations for fallback compaction, bounded retry attempts, stuck-session recovery, and tool-result truncation, but it still lacks the full repeated-failure circuit breaker, forced lossy recovery policy, transport isolation, and compaction health/status requested here.

## What This Changes

Requests a compaction recovery circuit breaker with fallback model use, forced truncation after repeated failures, Telegram transport isolation, and gateway status visibility.

## Best Possible Solution

Define a maintainer-approved compaction recovery policy that combines existing model fallback and timeout handling with session-scoped failure accounting, bounded lossy recovery or rotation, and operator-visible health without corrupting active transcripts or queued channel delivery.

## Reproduction Assessment

Yes, at source level: current main still has the compaction wait/unsubscribe path, foreground lane serialization, bounded retry caps, and no full repeated-failure force-truncate or health-status breaker. I did not run a live Telegram reproduction of the original outage.

## Solution Assessment

Unclear as a single patch: fallback retry and forced truncation are plausible, but the safe fix needs maintainer decisions on thresholds, configuration, transcript loss, and whether compaction can run outside the foreground session lane.
