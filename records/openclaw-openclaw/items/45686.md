## Summary

Current main still only partially addresses the reported compaction death spiral. The code has retry caps, configurable/static compaction model selection, successful transcript rotation, and tool-result truncation, but it does not implement a repeated-failure fallback circuit breaker, forced lossy turn pruning after failed compactions, failing-compaction health status, or a clear transport-decoupling policy; the related fallback PR #74470 is still open and covers only one slice.

## What This Changes

The issue requests compaction reliability hardening with fallback model selection after repeated failures, force-truncation after a threshold, Telegram transport isolation, and gateway status visibility.

## Best Possible Solution

Define a core per-session compaction failure policy that tracks recent failures, uses an approved fallback route for implicit compaction, applies documented lossy recovery after a threshold, exposes live/failing compaction state through the session or gateway status API, and keeps channel transports resilient without embedding Telegram-specific policy in core.

## Reproduction Assessment

Yes for the current compaction failure behavior: existing runner tests and source inspection show capped retries followed by context_overflow or compaction_failure instead of fallback/forced pruning. The full Telegram outage loop has concrete historical logs but was not live-reproduced against current main.

## Solution Assessment

No; current main’s retry caps and static model override are useful hardening but not the best complete fix for this report. The safer direction is a cohesive core failure policy plus status/API and channel-lifecycle work, while treating #74470 as a partial fallback implementation candidate.
