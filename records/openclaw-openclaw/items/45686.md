## Summary

Keep open: current `main` partially hardens compaction failure handling, but it still does not implement the requested repeated-failure fallback circuit breaker, forced lossy transcript pruning, live stuck-compaction status, or full channel transport isolation. The related open PR only covers the fallback slice, so this still needs maintainer product and architecture follow-up rather than cleanup closure.

## What This Changes

Request a compaction circuit breaker that falls back to another model after repeated failures, force-truncates history after further failures, keeps Telegram polling resilient, and exposes compaction health in gateway status.

## Best Possible Solution

Define a core compaction failure policy that tracks per-session compaction failures, applies an approved fallback route/model for implicit compaction, performs documented lossy recovery after a threshold, exposes live/failing compaction state through session or gateway status, and keeps channel transports resilient through a channel-agnostic lane boundary.
