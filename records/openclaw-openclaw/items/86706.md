## Summary

Keep open. The useful behavior is focused, but it should not merge yet because it reuses existing DM history retention keys to load prior local iMessage rows, which can change upgrade behavior and expose sensitive context; the live proof is also textual only.

## What This Changes

The PR adds iMessage direct-DM Messages.app history seeding for fresh sessions by reading DM history limits, fetching `messages.history`, prepending formatted context, adding tests, and documenting the option.

## Best Possible Solution

Preserve the existing DM-turn retention contract and gate Messages.app seeding behind a distinct explicit iMessage opt-in, or record maintainer approval for reusing the existing keys with upgrade/security proof.

## Reproduction Assessment

Yes for the merge blocker: current main already treats `dmHistoryLimit` and `dms.*.historyLimit` as DM-turn retention, while the PR uses those same keys to fetch and inject local chat history. I did not run a live iMessage flow.

## Solution Assessment

No. The safer solution is a separate explicit iMessage seed/import option, or a maintainer-approved compatibility decision with upgrade tests and inspectable live proof.
