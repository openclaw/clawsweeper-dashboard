## Summary

Keep open: the PR is directionally aimed at the cron CLI watchdog gap, but it is not merge-ready because the new payload field is not fully wired into the public/agent-facing contract and the supplied proof is still mocked test output rather than a real resumed cron CLI run.

## What This Changes

The PR adds a bounded cron `payload.noOutputTimeoutMs` field and threads it through cron normalization, CLI runner params, watchdog resolution, regression tests, and the changelog.

## Best Possible Solution

Land a rebased cron payload API change that normalizes and preserves `noOutputTimeoutMs` across create/update/hooks, publishes the field in docs and cron tool guidance, and proves a real resumed CLI cron run uses the extended watchdog.

## Reproduction Assessment

Yes, at source level: current main has no payload timeout parameter in the cron CLI executor or watchdog resolver, so a cron payload value cannot affect the CLI no-output watchdog. I did not run a live cron job in this read-only review.

## Solution Assessment

No, not merge-ready yet. The core threading is the right general shape, but the patch needs the normalizer hint path, public docs/tool guidance, and real resumed cron CLI proof before it is the best maintainable solution.
