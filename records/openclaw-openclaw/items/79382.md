## Summary

Keep this PR open: the intended direction is plausible, but the patch does not actually thread the cron payload value into real cron CLI executions, so the user-visible behavior remains unchanged.

## What This Changes

The PR adds a cron payload `noOutputTimeoutMs` field through schemas/types/normalization and forwards a `RunCliAgentParams` override into the CLI watchdog resolver, with unit coverage and a changelog entry.

## Best Possible Solution

If maintainers want this per-job watchdog knob, wire `params.agentPayload?.noOutputTimeoutMs` into the cron CLI runner call, keep the resolver cap behavior bounded, and add cron-level regression coverage or live proof for a resumed CLI session.

## Reproduction Assessment

Yes, at source level: current main and the PR path show the cron executor never supplies a payload no-output timeout to `runCliAgent`, so the resolver cannot honor it in real cron runs. I did not run a live cron job because this review is read-only.

## Solution Assessment

No. The resolver helper change is part of the right shape, but the PR is not the best complete fix until the cron executor passes the payload value and the proof covers the resumed cron CLI path.
