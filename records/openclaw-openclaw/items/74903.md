## Summary

Keep open. Current main still matches the reported Feishu topic-session message-tool routing bug, and open closing PR #75364 plus overlapping PR #75159 should be reviewed or consolidated before the issue is closed.

## What This Changes

The issue asks Feishu `group_topic` and `group_topic_sender` message-tool sends to reply inside the triggering topic thread instead of posting as root group messages.

## Best Possible Solution

Review and land or replace the Feishu-owned runtime fix from #75364 so same-conversation `message(action=send)` in topic-scoped sessions inherits the inbound topic reply target, with focused channel-action regression tests and a changelog entry. A prompt hint like #75159 can be added only as defense in depth; delivery should not depend on model compliance.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: the shared prompt directs visible group output to `message(action=send)`, while current Feishu `send` leaves `replyToMessageId` undefined and sets `replyInThread: false`; existing tests assert those separate plain-send and explicit-thread behaviors but do not cover topic-scoped send inheritance.

## Solution Assessment

Yes. The best direction is the narrow Feishu plugin runtime fix represented by #75364 because the action handler has `sessionKey` and `toolContext.currentMessageId`, so it can preserve topic threads without relying on the model to choose `thread-reply`; the prompt-only PR is weaker as the primary fix.
