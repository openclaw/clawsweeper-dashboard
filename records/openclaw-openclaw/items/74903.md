## Summary

Keep open. Current main still matches the reported Feishu topic-session message-tool routing bug, and the issue is already paired with open closing PR #75364 plus overlapping PR #75159, so it should not be closed or queued for a third automated repair before those branches are reviewed or consolidated.

## What This Changes

The issue requests that Feishu `group_topic` and `group_topic_sender` sessions keep assistant `message(action=send)` replies inside the triggering topic thread instead of sending them as ordinary group messages.

## Best Possible Solution

Review and land or replace the Feishu-owned runtime fix from #75364 so same-conversation `message(action=send)` in topic-scoped sessions inherits the inbound topic reply target. A prompt hint like #75159 can be kept only as defense in depth; delivery should not depend on model compliance.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: the shared prompt directs visible group output to `message(action=send)`, while current Feishu `send` leaves `replyToMessageId` undefined and sets `replyInThread: false`; existing tests cover the separate plain-send and explicit-thread paths but not topic-scoped send inheritance.

## Solution Assessment

Yes. The narrow Feishu plugin runtime fix represented by #75364 is the best direction because the action handler has `sessionKey` and `toolContext.currentMessageId`, so it can preserve topic threads without relying on the model to choose `thread-reply`; the prompt-only PR is weaker as the primary fix.
