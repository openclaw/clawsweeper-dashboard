## Summary

Keep open. Current main still has the Feishu topic-session mismatch: shared group visible-reply guidance steers the assistant to `message(action=send)`, while the Feishu action handler only enables thread semantics for explicit `thread-reply`. The issue is also already paired with open closing PR #75364 and overlapping PR #75159, so it should remain open until maintainers land or consolidate one fix.

## What This Changes

The issue reports that Feishu `group_topic` and `group_topic_sender` sessions send message-tool visible replies to the group root when the assistant uses `action=send`, instead of keeping the reply in the original topic thread.

## Best Possible Solution

Land or replace the Feishu-owned runtime fix so same-conversation `message(action=send)` from `group_topic` and `group_topic_sender` sessions uses the inbound topic root/current message as the Feishu thread reply target, while preserving plain group `send` behavior and adding focused action-handler regression tests. A prompt hint can remain defense in depth, but the delivery invariant should not depend on model compliance.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: group message-tool mode tells the assistant to use `message(action=send)`, and the current Feishu action handler sends that action with no `replyToMessageId` and `replyInThread: false`; existing tests cover plain send and explicit thread-reply but not topic-mode send inheritance.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in the Feishu plugin runtime because the action handler has `sessionKey` and `toolContext.currentMessageId`; a prompt-only fix is weaker because it depends on the model choosing `thread-reply`.
