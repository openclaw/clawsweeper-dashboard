## Summary

Keep open. Current main still has the Feishu topic-session message-tool mismatch: the shared group prompt tells the assistant to use `message(action=send)`, while the Feishu action handler only sends thread replies for explicit `thread-reply`. The issue is also paired with open closing PR #75364 and alternate PR #75159, so repository policy says to leave this issue open until one fix lands or maintainers consolidate them.

## What This Changes

Feishu `group_topic` and `group_topic_sender` sessions should post assistant visible replies into the triggering topic thread instead of sending a plain group message when the assistant uses `message(action=send)`.

## Best Possible Solution

Land or replace the Feishu-owned runtime fix that makes same-conversation `message(action=send)` from `group_topic` and `group_topic_sender` sessions use the inbound topic root/current message as the Feishu thread reply target, while preserving plain group `send` behavior and adding focused action-handler regression tests. A prompt hint can be added as defense in depth, but the delivery invariant should not depend on model compliance.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: in a message-tool-only group turn, the prompt asks for `message(action=send)`, and current Feishu `send` resolves no `replyToMessageId` and sets `replyInThread: false`; existing tests assert that plain send behavior and lack topic-mode send inheritance coverage.

## Solution Assessment

Yes, the narrowest maintainable direction is the Feishu plugin runtime fix represented by #75364, because the bug is Feishu-specific delivery routing and the action handler has the session/tool context needed to preserve topic threads. A prompt-only change like #75159 is useful only as an additional nudge because it depends on the model choosing `thread-reply`.
