## Summary

Keep open. Current main does not implement this isolation surface, and the latest PR branch still has source-visible privacy gaps plus missing real two-user behavior proof for a default-on memory boundary.

## What This Changes

The PR adds user-scoped memory isolation across memory-core tool identity propagation, host SDK memory path guards, QMD manager scoping, memory config schema/types, SDK exports, and focused tests.

## Best Possible Solution

Land an owner-reviewed structural memory isolation design that consistently scopes identity, config defaults, read/search/write paths, QMD collections, shared memory semantics, migrations, docs, tests, and real two-user proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: default configs can stay unscoped, raw sender IDs can collide across channel/account scopes, QMD collections still index workspace-wide memory, and root flush files remain shared. I did not run tests or a live two-user setup because this review is read-only.

## Solution Assessment

No. The branch has useful pieces, but it is not the best mergeable solution until identity scope, default behavior, QMD/backend parity, write-path scoping, config semantics, and real behavior proof are coordinated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Enable isolation when the config block is omitted:** `extensions/memory-core/src/isolation-identity.ts:64-66`
  - body: `resolveMemorySearchConfig` defaults isolation to enabled, but this resolver treats a missing `memory.isolation` block as disabled and returns no `userId`. Existing configs would register memory tools and managers without the advertised default-on privacy boundary.
  - confidence: 0.94
- **[P1] Preserve channel and account scope in user IDs:** `extensions/memory-core/src/isolation-identity.ts:68-70`
  - body: This prefers raw `requesterSenderId` over the session key, so channel/account scope from keys like per-channel or per-account DMs is dropped before hashing the memory directory. Two different channels or accounts with the same sender id can collapse into one memory bucket.
  - confidence: 0.9
- **[P1] Scope QMD collections, not only QMD state:** `extensions/memory-core/src/memory/qmd-manager.ts:347-351`
  - body: The new per-user QMD state directory does not change `resolved.qmd.collections`, which still comes from the workspace `memory/**/*.md` collection. Each user's QMD index can therefore ingest other users' memory files despite having separate cache/state directories.
  - confidence: 0.9
- **[P1] Stop shared flush notes from entering every user index:** `packages/memory-host-sdk/src/host/internal.ts:262-263`
  - body: With `userId`, this still adds root-level `memory/*.md` files for every isolated user, while the existing flush plan writes conversation-derived durable notes to that root daily file. A flush from one user's session can become searchable by other users.
  - confidence: 0.88
- **[P2] Honor isolation scope and path encoding config:** `src/agents/memory-search.ts:424-426`
  - body: The schema accepts `scope`, `fallbackPolicy`, and `pathEncoding`, but the resolved runtime settings retain only `enabled`; identity resolution and path handling hard-code user/sender behavior and hash encoding. Valid config can parse while silently having no effect on the isolation boundary.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch introduces a privacy boundary but still leaves source-visible cross-user disclosure paths.

Concerns:

- **[high] Default configs can run unscoped:** `extensions/memory-core/src/isolation-identity.ts:64`
  - body: When `memory.isolation` is omitted, the PR's identity resolver treats isolation as disabled even though memory search resolves it as enabled by default, so existing multi-user deployments can still share memory.
  - confidence: 0.94
- **[high] Raw sender IDs can collide across scopes:** `extensions/memory-core/src/isolation-identity.ts:68`
  - body: Using `requesterSenderId` alone drops channel/account scope that current session keys can carry, so different accounts or channels with the same sender id can map to the same isolated memory directory.
  - confidence: 0.9
- **[high] QMD can index workspace-wide memory:** `packages/memory-host-sdk/src/host/backend-config.ts:349`
  - body: Per-user QMD state directories do not isolate content if default QMD collections still point at the shared workspace memory root and `memory/**/*.md`.
  - confidence: 0.9
- **[high] Memory flush writes remain shared:** `packages/memory-host-sdk/src/host/internal.ts:262`
  - body: The default flush target is still the root daily memory file, and the PR includes root-level memory files for every isolated user, leaving a cross-user disclosure path.
  - confidence: 0.88
