## Summary

Keep open: current main does not implement user-specific memory isolation, and this PR is still the active implementation track, but the latest head still has source-visible privacy-boundary gaps and lacks real behavior proof.

## What This Changes

The branch adds a default-on `memory.isolation` config surface, memory-core identity propagation, per-user file and QMD helpers, path encoding utilities, and scoped-memory tests.

## Best Possible Solution

Land an owner-reviewed memory privacy design that consistently scopes identity, read/search/write paths, QMD collections, flush targets, config semantics, migration behavior, docs, and real two-user validation.

## Reproduction Assessment

Yes. Source inspection of PR head shows default-config, QMD-collection, and shared-flush paths that bypass the intended isolation boundary; I did not run tests or a live two-user setup because this review is read-only.

## Solution Assessment

No. The direction is useful, but a default-on privacy boundary should not merge until backend parity, write-path scoping, config semantics, migration behavior, owner/security review, and real two-user proof are aligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Enable isolation when the config block is omitted:** `extensions/memory-core/src/isolation-identity.ts:64-66`
  - body: `resolveMemorySearchConfig` defaults isolation to enabled, but this resolver treats a missing `memory.isolation` block as disabled and returns no `userId`. Existing configs would register unscoped memory tools and managers, so the advertised default-on privacy boundary is not active.
  - confidence: 0.93
- **[P1] Scope QMD collections by user:** `extensions/memory-core/src/memory/qmd-manager.ts:347-351`
  - body: These lines only put QMD state under a user directory; the resolved default collections still target workspace-wide `memory/**/*.md` and are built without a `userId`. QMD users can therefore index and search other users' memory files despite isolated state paths.
  - confidence: 0.9
- **[P1] Route flush memory into user directories:** `packages/memory-host-sdk/src/host/internal.ts:262-263`
  - body: With a `userId`, `listMemoryFiles` still includes root-level `memory/*.md` files for every isolated user, while the flush plan writes durable notes to `memory/YYYY-MM-DD.md`. A flush from one session can become searchable by other users.
  - confidence: 0.88
- **[P2] Apply scope and path encoding settings:** `src/agents/memory-search.ts:424-426`
  - body: The schema accepts `scope`, `fallbackPolicy`, and `pathEncoding`, but the resolved runtime settings keep only `enabled`, and downstream calls use hard-coded user/hash behavior. Valid config can parse while silently not changing the isolation boundary.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch introduces a default-on privacy boundary but still leaves source-visible cross-user read, search, and write disclosure paths.

Concerns:

- **[high] Default configs can run unscoped:** `extensions/memory-core/src/isolation-identity.ts:64`
  - body: When `memory.isolation` is omitted, identity resolution treats isolation as disabled even though memory search resolves it as enabled by default, so existing multi-user deployments can still share memory.
