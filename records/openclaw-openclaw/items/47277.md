## Summary

Keep open. Current main does not implement user-specific memory isolation, and this external PR is still the active implementation track, but the branch is not merge-ready because source inspection still shows privacy-boundary gaps and the PR lacks real two-user behavior proof.

## What This Changes

The PR adds default-on memory isolation config, user identity propagation through memory-core tools, user-scoped builtin/QMD memory manager paths and caches, path-encoding helpers, SDK exports, and tests.

## Best Possible Solution

Land an owner-reviewed memory privacy design that consistently scopes identity, config semantics, read/search/write paths, QMD collections/state, flush targets, docs, migration behavior, and real two-user validation.

## Reproduction Assessment

Yes for the PR defects: PR-head source inspection shows default config, QMD collection, and shared flush/root-memory paths that can bypass the intended privacy boundary. I did not execute tests or a live two-user setup because this review is read-only.

## Solution Assessment

No. The branch contains useful building blocks, but a default-on privacy boundary should not merge until backend parity, write-path scoping, config semantics, and real two-user proof are coordinated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Enable isolation when the config block is omitted:** `extensions/memory-core/src/isolation-identity.ts:64-66`
  - body: `resolveMemorySearchConfig` advertises `isolation.enabled` as defaulting to true, but this resolver treats a missing `memory.isolation` block as disabled and returns no `userId`. Existing configs would register memory tools and QMD managers without the default-on privacy boundary.
  - confidence: 0.93
- **[P1] Scope QMD collections by user:** `extensions/memory-core/src/memory/qmd-manager.ts:347-351`
  - body: This only moves QMD state under a per-user directory; the resolved default collections still target the workspace root and `memory/**/*.md`. Each user's QMD index can therefore ingest other users' memory files despite isolated state paths.
  - confidence: 0.9
- **[P1] Route flush memory into user directories:** `packages/memory-host-sdk/src/host/internal.ts:262-263`
  - body: With a `userId`, `listMemoryFiles` still includes root-level `memory/*.md` files for every isolated user, while the existing flush plan writes durable notes to `memory/YYYY-MM-DD.md`. A flush from one session can become searchable by other users.
  - confidence: 0.88
- **[P2] Honor the accepted isolation options:** `src/agents/memory-search.ts:424-426`
  - body: The schema accepts `scope`, `fallbackPolicy`, and `pathEncoding`, but the resolved runtime settings keep only `enabled`, and the path helpers use the hard-coded default encoding. Valid config can parse while silently having no effect on the isolation boundary.
  - confidence: 0.82

## Security Review

Status: needs_attention
