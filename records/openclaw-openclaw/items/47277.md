- **[P1] Honor the default-on isolation contract:** `extensions/memory-core/src/isolation-identity.ts:25-27`
  - body: `resolveMemorySearchConfig` and the schema default isolation to enabled, but this resolver returns no policy when `memory.isolation` is omitted. Existing configs that do not add the new block will create tools with `userId` unset, so memory remains shared even though the PR advertises default-on isolation.
  - confidence: 0.93
- **[P1] Isolate the builtin index storage or query scope:** `extensions/memory-core/src/memory/manager.ts:170-171`
  - body: Adding `userId` to the manager cache key creates per-user manager objects, but they still open the same `settings.store.path` SQLite database and search without a user predicate. A second user's manager can see stale chunks from the first user before its own sync, and sync can delete the other user's indexed rows as stale.
  - confidence: 0.9
- **[P1] Scope QMD collections before indexing:** `extensions/memory-core/src/memory/qmd-manager.ts:348-352`
  - body: This scopes QMD's state directory by user, but the resolved QMD collections still point at the workspace-wide `MEMORY.md` and `memory/**/*.md` paths. Each per-user QMD manager would build its own index over all users' memory files, so QMD search still violates the isolation guarantee.
  - confidence: 0.89
- **[P2] Carry all isolation config fields into runtime decisions:** `src/agents/memory-search.ts:417-423`
  - body: The schema exposes `scope`, `fallbackPolicy`, and `pathEncoding`, but `resolveMemorySearchConfig` only preserves `enabled`. As a result, path guards and manager setup default to hash/user behavior even when config selects another documented mode, creating a misleading config surface for a security boundary.
  - confidence: 0.82
