## Summary

Keep open: this external PR still has source-provable P1 privacy defects and no real behavior proof, while current main does not already implement per-user memory isolation and no duplicate or superseding PR owns the work.

## What This Changes

The PR adds default-on per-user memory isolation across memory config, tool identity resolution, memory file access/listing, QMD state/cache scoping, SDK exports, and tests.

## Best Possible Solution

Land an owner-reviewed memory privacy design that consistently scopes identity, read/search/write paths, QMD collections/state, flush targets, config semantics, docs, and live two-user validation.

## Reproduction Assessment

Yes for the PR defects: source inspection of the latest head shows omitted memory.isolation disables identity resolution, QMD collections remain workspace-wide, and flush plans still write shared memory/YYYY-MM-DD.md. I did not run tests because this review is read-only.

## Solution Assessment

No. The current patch adds useful pieces, but a default-on privacy boundary is not safe until config semantics, backend parity, and write-path enforcement are coordinated and proven live.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Enable isolation when the config block is omitted:** `extensions/memory-core/src/isolation-identity.ts:64-66`
  - body: The schema and resolver make isolation default to enabled, but readIsolation returns null when memory.isolation is omitted. Existing configs therefore register memory tools with userId undefined and build unscoped managers, bypassing the advertised default-on privacy boundary.
  - confidence: 0.93
- **[P1] Scope QMD collections by user:** `extensions/memory-core/src/memory/qmd-manager.ts:347-351`
  - body: Passing userId into resolveQmdAgentDir only moves QMD's state/index directory. resolveMemoryBackendConfig still builds default collections from workspace MEMORY.md and memory/**/*.md with no user filter, so each per-user QMD index can ingest other users' memory files.
  - confidence: 0.9
- **[P1] Route flush writes into user memory directories:** `src/memory-host-sdk/host/internal.ts:213`
  - body: Isolated indexing still includes root memory/*.md files, while buildMemoryFlushPlan still targets memory/YYYY-MM-DD.md. A flush from one user's DM can be written to the shared root and then indexed for every other isolated user.
  - confidence: 0.87
- **[P2] Honor the accepted isolation options:** `src/agents/memory-search.ts:424-426`
  - body: The config schema accepts scope and pathEncoding, but the resolved settings keep only enabled; downstream path checks and QMD path creation always use hard-coded user/hash behavior. Users can set valid options that parse yet do not affect the runtime boundary.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch introduces a privacy boundary but leaves source-provable cross-user read, search, and write disclosure paths.

Concerns:

- **[high] Default configs can run unscoped:** `extensions/memory-core/src/isolation-identity.ts:64`
  - body: When memory.isolation is omitted, identity resolution treats isolation as disabled even though memory search resolves isolation as enabled by default, so existing multi-user deployments can still share memory.
  - confidence: 0.93
- **[high] QMD can index all user memory:** `extensions/memory-core/src/memory/qmd-manager.ts:347`
  - body: Per-user QMD state directories do not isolate content if QMD collections still target workspace-wide memory roots; each user index can contain other users' files.
  - confidence: 0.9
- **[high] Compaction writes to the shared memory root:** `src/memory-host-sdk/host/internal.ts:213`
  - body: The patch permits root memory files for every isolated user while the existing flush plan writes durable notes to the root daily file path, creating a cross-user disclosure path.
  - confidence: 0.87
