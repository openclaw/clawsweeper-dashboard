## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope memory identities by route context:** `extensions/memory-core/src/isolation-identity.ts:69-70`
  - body: `senderId` is only the raw requester id, and the fallback parser also keeps only the token after `direct`; current session routing supports per-channel and per-account DM scopes. Two channels/accounts that emit the same sender id will share the same memory bucket, so derive the isolation key from trusted route/session metadata instead of raw sender text.
  - confidence: 0.91
- **[P1] Route flush writes into isolated memory buckets:** `packages/memory-host-sdk/src/host/internal.ts:263`
  - body: Isolated scans still include root-level `memory/*.md` for every user, but the memory flush plan writes private compaction notes to `memory/YYYY-MM-DD.md`. A DM compaction for one user can therefore be indexed and returned for another user, so flush/write paths need the same user scope before root files are shared.
  - confidence: 0.89
- **[P1] Apply default isolation consistently:** `extensions/memory-core/src/isolation-identity.ts:27-29`
  - body: The schema and resolved search config default isolation to enabled, but this resolver treats a missing `memory.isolation` block as no isolation and returns `undefined`. Existing configs without the new block will keep using shared memory despite the advertised default-on behavior; either make the feature explicitly opt-in or make all runtime gates honor the same default.
  - confidence: 0.86
- **[P2] Honor the full isolation config surface:** `src/agents/memory-search.ts:424-426`
  - body: `memory.isolation` accepts `scope`, `fallbackPolicy`, and `pathEncoding`, but resolved search settings only carry `enabled`, and downstream list/read/QMD calls keep default hash/user behavior. Users can set accepted config values that the runtime ignores, so either wire these fields end-to-end or remove the unsupported knobs.
  - confidence: 0.84
