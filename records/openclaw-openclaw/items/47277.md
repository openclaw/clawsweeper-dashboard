Full review comments:

- **[P1] Apply default isolation when the config block is absent:** `extensions/memory-core/src/isolation-identity.ts:27-29`
  - body: `resolveMemorySearchConfig` on this branch defaults isolation to enabled when `memory.isolation` is omitted, but this helper returns `undefined` before resolving any identity whenever the block is absent. That means fresh/default configs run `memory_search`/`memory_get` unscoped despite the advertised default-on isolation; either consume the same defaults here or change the default/upgrade semantics consistently.
  - confidence: 0.91
- **[P1] Include channel and account scope in sender identities:** `extensions/memory-core/src/isolation-identity.ts:68-70`
  - body: This returns the raw `senderId` before considering the trusted channel/account/session context. Many transports only provide local sender IDs, so two channels or accounts with the same sender string can hash to the same memory bucket; derive the key from a trusted channel/account/sender tuple or require a globally unique requester identity.
  - confidence: 0.86
- **[P1] Keep memory-flush notes out of shared root scans:** `src/memory-host-sdk/host/internal.ts:213`
  - body: In isolation mode this still indexes root-level `memory/*.md` for every user, while the memory-core flush plan writes private compaction notes to `memory/YYYY-MM-DD.md`. A DM compaction flush can therefore be searched by other isolated users; route flushes to the scoped user path or stop treating root daily notes as shared when isolation is enabled.
  - confidence: 0.9
- **[P2] Bound per-user memory manager lifetimes:** `extensions/memory-core/src/memory/manager.ts:169-177`
  - body: Adding `userId` to the default manager cache key creates one long-lived manager per sender, and those managers own SQLite, watcher, and timer state with no idle eviction here. Busy shared-channel deployments can accumulate resources indefinitely; use per-request scoping or bounded teardown before caching by user.
  - confidence: 0.78
