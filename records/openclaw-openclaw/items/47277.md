## Summary

Keep open: current main does not implement this feature, and the latest PR still has source-provable privacy gaps plus missing real behavior proof, so it is not safe to merge or close.

## What This Changes

The PR adds default-on user-scoped memory isolation across memory config, memory-core tool identity resolution, host memory path/list/read handling, QMD cache/state scoping, SDK exports, and tests.

## Best Possible Solution

Land an owner-reviewed memory privacy design that consistently scopes identity, read/search/write paths, QMD collections/state, flush targets, config semantics, docs, and live two-user validation.

## Reproduction Assessment

Yes for the PR defects: source inspection of the latest head shows omitted memory.isolation disables identity resolution, QMD collections remain workspace-wide, and flush plans still write shared memory/YYYY-MM-DD.md. I did not run tests because this review is read-only.

## Solution Assessment

No. The current patch adds useful pieces, but a default-on privacy boundary is not safe until config semantics, backend parity, write-path enforcement, and live two-user proof are coordinated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Enable isolation when the config block is omitted:** `extensions/memory-core/src/isolation-identity.ts:64-66`
  - body: The schema and resolver make isolation default to enabled, but readIsolation returns null when memory.isolation is omitted. Existing configs therefore register memory tools with userId undefined and build unscoped managers, bypassing the advertised default-on privacy boundary.
  - confidence: 0.93
- **[P1] Scope QMD collections by user:** `extensions/memory-core/src/memory/qmd-manager.ts:347-351`
  - body: This scopes QMD state/cache directories by user, but resolveMemoryBackendConfig still builds default collections from the workspace MEMORY.md and memory/**/*.md roots. Each per-user QMD index can still ingest other users' memory files.
  - confidence: 0.9
- **[P1] Route flush writes into user memory directories:** `src/memory-host-sdk/host/internal.ts:213`
  - body: In isolation mode listMemoryFiles still includes root memory/*.md files for every user, while buildMemoryFlushPlan still writes compaction notes to memory/YYYY-MM-DD.md. A flush from one user's session can land in the shared root and then be indexed for other users.
  - confidence: 0.87
- **[P2] Honor the accepted isolation options:** `src/agents/memory-search.ts:424-426`
  - body: The config schema accepts scope and pathEncoding, but the resolved runtime settings keep only enabled; downstream path checks and QMD path creation always use hard-coded user/hash behavior. Users can set valid options that parse yet do not affect the runtime boundary.
  - confidence: 0.82
