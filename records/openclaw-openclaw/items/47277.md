## Summary

Keep open. The proposed privacy feature is useful, and the production-oriented discussion about storage/query/write isolation is relevant, but the latest head still has source-reproducible cross-user read/search/write gaps and no real multi-user behavior proof.

## What This Changes

The PR adds default-on memory isolation config, userId resolution/plumbing through memory-core tools, builtin memory path checks, QMD cache/state scoping, and related tests.

## Best Possible Solution

Land an owner-reviewed privacy design that enforces identity consistently across tool registration, read/search/write paths, builtin and QMD backends, flush writes, config/docs, and live two-user validation.

## Reproduction Assessment

Yes for the PR defects: source inspection of the latest head shows absent isolation config remains unscoped, QMD collections still target workspace-wide memory, and memory flush writes stay under the shared root path. I did not run tests because this review is read-only.

## Solution Assessment

No. The current patch is not the best way to solve this because it adds a default-on privacy boundary over only part of the memory system; the safer path is a coordinated design with backend parity and write-path enforcement before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Enable isolation when the config block is omitted:** `extensions/memory-core/src/isolation-identity.ts:64-66`
  - body: The resolver returns `undefined` whenever `memory.isolation` is absent, while `resolveMemorySearchConfig` treats isolation as enabled by default. Existing memory-enabled configs without the new block will create unscoped managers and bypass the advertised privacy boundary.
  - confidence: 0.92
- **[P1] Scope QMD collections by user:** `extensions/memory-core/src/memory/qmd-manager.ts:347-351`
  - body: This scopes only QMD's per-user state directory. The resolved default collections still point at workspace `MEMORY.md` and `memory/**/*.md`, so each user-specific QMD index can ingest every user's memory files.
  - confidence: 0.9
- **[P1] Route flush writes into user memory directories:** `src/memory-host-sdk/host/internal.ts:213`
  - body: In isolated mode this still indexes root `memory/*.md` files for every user, while memory-core's flush plan writes durable notes to `memory/YYYY-MM-DD.md`. A DM flush can therefore become shared search context for other isolated users.
  - confidence: 0.86
- **[P2] Honor the accepted isolation options:** `src/agents/memory-search.ts:424-426`
  - body: The schema accepts `scope` and `pathEncoding`, but the resolved settings keep only `enabled` and downstream calls use hard-coded user/hash behavior. Users can set valid options that parse successfully but do not affect isolation.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: This PR introduces a privacy boundary while leaving concrete cross-user read/search/write disclosure paths.

Concerns:

- **[high] Default configs can run unscoped:** `extensions/memory-core/src/isolation-identity.ts:64`
  - body: When `memory.isolation` is omitted, the identity resolver treats isolation as disabled even though the search config resolves it as enabled by default, so existing multi-user deployments can still share memory.
  - confidence: 0.92
- **[high] QMD can index all user memory:** `extensions/memory-core/src/memory/qmd-manager.ts:347`
  - body: Per-user QMD state directories do not isolate content if QMD collections still point at workspace-wide memory roots; each user index can contain other users' files.
  - confidence: 0.9
- **[high] Compaction writes to shared memory root:** `src/memory-host-sdk/host/internal.ts:213`
  - body: The patch permits root memory files for every isolated user while the existing flush plan writes durable notes to the root daily file path, creating a cross-user disclosure path.
  - confidence: 0.86
