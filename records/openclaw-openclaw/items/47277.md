## Summary

Keep open: current main still uses shared memory paths, and the PR is a security-sensitive feature that remains incomplete as a privacy boundary. The discussion also contains useful production context from KinthAI calling for storage, query, and write isolation, which supports the feature direction but not this partial implementation.

## What This Changes

The PR threads optional sender/user identity through memory-core tools and memory-host file access, adds memory isolation config types, and attempts per-user memory scoping for builtin and QMD memory backends.

## Best Possible Solution

Land a maintainer-reviewed isolation design that fails closed, canonicalizes user identity into safe storage keys, scopes storage/search/read/write paths consistently across builtin and QMD, and documents the shared-memory policy with regression coverage.

## Reproduction Assessment

Yes. Source inspection of current main plus the provided PR diff gives a high-confidence static reproduction path: current main is shared, while the PR leaves unscoped or unsafe paths in QMD, sender fallback, memory-host path checks, and schema validation.

## Solution Assessment

No. The current PR is a partial default-on privacy boundary; the safer solution is an owner-reviewed design with fail-closed identity, backend parity, write-path enforcement, and validated configuration.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope QMD collections by user:** `extensions/memory-core/src/memory/qmd-manager.ts:256`
  - body: The PR accepts `userId` here and forwards it from the caller, but the QMD manager still uses agent-scoped state, collection roots, cache identity, and query behavior. Deployments using `memory.backend: "qmd"` can remain agent-wide under the new isolation guarantee.
  - confidence: 0.9
- **[P1] Fall back to session identity when senderId is absent:** `extensions/memory-core/src/tools.shared.ts:59`
  - body: The tool context scopes memory only from optional `senderId`. If plugin context omits `requesterSenderId`, valid session-key-only executions still create unscoped managers despite the added session-key helper, so isolation can silently disappear.
  - confidence: 0.86
- **[P1] Encode user IDs before joining memory paths:** `packages/memory-host-sdk/src/host/internal.ts:181`
  - body: The isolated filesystem path is derived from raw sender/session identifiers. Values containing path separators, dot segments, or platform-invalid characters can escape, collapse, or fail the intended per-user directory boundary.
  - confidence: 0.78
- **[P2] Add the isolation option to the schema:** `src/config/types.memory.ts:25`
  - body: The PR adds and reads `memory.isolation`, but the strict `MemorySchema` is not updated, so users cannot configure the advertised `memory.isolation.enabled: false` legacy switch.
  - confidence: 0.89
