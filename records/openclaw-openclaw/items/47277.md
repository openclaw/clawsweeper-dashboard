## Summary

Keep open: this is a meaningful privacy feature, but the latest head still has source-visible cross-user read/search/write gaps and lacks real two-user behavior proof, so it is not ready to merge or close as cleanup.

## What This Changes

The PR adds default-on per-user memory isolation config, sender/session identity plumbing, user-encoded memory paths, QMD state/cache scoping, SDK exports, and tests.

## Best Possible Solution

Land an owner-reviewed privacy design that enforces identity consistently across memory tool registration, builtin and QMD read/search/write paths, flush writes, config options, docs, and live two-user validation.

## Reproduction Assessment

Yes for the PR defects: source inspection of the latest head shows the default config path is unscoped, QMD collections still point at workspace memory roots, and flush writes remain under the shared root. I did not run tests because this review is read-only.

## Solution Assessment

No. The current patch is not the best way to solve this because it layers a default-on privacy boundary over only part of the memory system; the safer path is a coordinated design with backend parity, write-path enforcement, honored config semantics, and live proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Enable isolation when the config block is omitted:** `extensions/memory-core/src/isolation-identity.ts:64-66`
  - body: `resolveMemorySearchConfig` treats isolation as enabled by default, but this resolver returns `undefined` when `memory.isolation` is absent. Existing memory configs without the new block will create unscoped tools/managers and bypass the advertised default privacy boundary.
  - confidence: 0.93
- **[P1] Scope QMD collections by user:** `extensions/memory-core/src/memory/qmd-manager.ts:347-351`
  - body: This only moves QMD state under a per-user directory. The resolved default collections still point at workspace `MEMORY.md` and `memory/**/*.md`, so each user's QMD index can ingest every user's memory files.
  - confidence: 0.9
- **[P1] Route flush writes into user memory directories:** `src/memory-host-sdk/host/internal.ts:213`
  - body: Isolated scans still include root `memory/*.md` files, while the memory-core flush plan writes durable notes to `memory/YYYY-MM-DD.md`. A DM flush can therefore become shared search context for other isolated users.
  - confidence: 0.87
- **[P2] Honor the accepted isolation options:** `src/agents/memory-search.ts:424-426`
  - body: The schema accepts `scope` and `pathEncoding`, but the resolved settings keep only `enabled` and downstream calls use hard-coded user/hash behavior. Users can set valid options that parse successfully but do not affect isolation behavior.
  - confidence: 0.82

## Security Review
