## Summary

Keep open. Current main still uses shared memory search/read behavior, and this PR is not safe to merge as-is because it adds a default-on privacy boundary while leaving QMD scoping, identity fallback, path-safe user IDs, and config schema coverage incomplete.

## What This Changes

This PR adds default-on per-user memory isolation across memory-core tool context, memory search manager plumbing, memory-host file discovery/read checks, memory config types, and session-key helpers.

## Best Possible Solution

Replace or substantially rework the branch with a maintainer-reviewed memory isolation design that canonicalizes user identity safely, enforces storage/query/read/write scope for both builtin and QMD backends, updates config schema/help/docs, and adds regression coverage for direct, channel, group, disabled-isolation, and cross-platform identity cases.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main has no user-scoped memory manager/read path, and the PR diff still leaves concrete unscoped QMD and sender-missing paths. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The proposed PR is not the best way to solve the issue because it adds a partial default-on privacy boundary without complete backend parity, safe identity mapping, schema validation, and write-path enforcement; the safer path is an owner-reviewed design before merge.
