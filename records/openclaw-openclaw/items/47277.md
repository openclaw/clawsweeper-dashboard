## Summary

Keep open. Current main does not already implement memory isolation, and this PR is a plausible feature branch, but the latest head still has source-visible privacy gaps plus no real two-user behavior proof.

## What This Changes

The PR adds memory-core user identity propagation, memory.isolation config/schema, per-user filesystem/QMD state helpers, SDK path guards, and tests for user-scoped memory search/get.

## Best Possible Solution

Land an owner-reviewed memory privacy design that consistently scopes identity, config semantics, read/search/write paths, QMD collections/state, flush targets, docs, migration behavior, and real two-user validation.

## Reproduction Assessment

Yes for the PR defects: source inspection of PR head shows config-default, QMD-collection, and flush-root paths that bypass the intended isolation boundary. I did not run tests or a live two-user setup because this review is read-only.

## Solution Assessment

No. The branch contains useful pieces, but a default-on privacy boundary should not merge until backend parity, write-path scoping, config semantics, owner/security review, and real two-user proof are aligned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Enable isolation when the config block is omitted:** `extensions/memory-core/src/isolation-identity.ts:64-66`
  - body: `resolveMemorySearchConfig` defaults memory isolation to enabled, but this resolver treats a missing `memory.isolation` block as disabled and returns no `userId`. Existing configs would register unscoped memory tools and managers, so the advertised default-on privacy boundary is not active.
  - confidence: 0.93
- **[P1] Scope QMD collections by user:** `extensions/memory-core/src/memory/qmd-manager.ts:347-351`
  - body: These lines only put QMD state under a user directory; the resolved default collections still target workspace-wide `memory/**/*.md` and are built without a `userId`. QMD users can therefore index and search other users' memory files despite isolated state paths.
  - confidence: 0.9
- **[P1] Route flush memory into user directories:** `packages/memory-host-sdk/src/host/internal.ts:262-263`
  - body: With a `userId`, `listMemoryFiles` still includes root-level `memory/*.md` files for every isolated user, while the flush plan writes durable notes to `memory/YYYY-MM-DD.md`. A flush from one session can become searchable by other users.
  - confidence: 0.88
- **[P2] Apply scope and path encoding settings:** `src/agents/memory-search.ts:424-426`
  - body: The schema accepts `scope`, `fallbackPolicy`, and `pathEncoding`, but the resolved runtime settings keep only `enabled`, and downstream calls use hard-coded user/hash behavior. Valid config can parse while silently not changing the isolation boundary.
  - confidence: 0.82

## Security Review
