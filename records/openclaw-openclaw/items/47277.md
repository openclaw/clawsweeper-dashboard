## Label Justifications

## Summary

The PR is still necessary as a product capability, but the current head is not mergeable: its advertised default is inconsistent, raw sender identities can collide across accounts/channels, normal memory flushes remain globally readable, documented config modes are ignored, and per-user managers are cached without bounds. The discussion usefully established the need for storage/query/write isolation, but no real two-user proof demonstrates that contract end to end.

## What This Changes

Adds configurable user-scoped memory isolation across memory tool identity resolution, builtin and QMD indexing/state, SDK path validation, and memory configuration.

## Best Possible Solution

Design one canonical tenant identity owned by the inbound/session layer, pass it as a prepared fact to every memory writer, reader, indexer, and backend, store tenant state in a bounded lifecycle-owned SQLite design, and ship the behavior only with explicit shared-memory semantics plus tested fresh-install and doctor migration paths.

## Reproduction Assessment

Yes. On the PR head, run two users with omitted `memory.isolation` to observe shared managers, or enable isolation and trigger the standard compaction flush to show both users can access the same root `memory/YYYY-MM-DD.md`; equal raw sender IDs across channels also deterministically select the same bucket.

## Solution Assessment

No. The proposed per-tool/per-manager user ID threading is not the narrowest maintainable solution because tenant identity, writes, shared-memory policy, persistence, lifecycle, and migration must be defined at one earlier owner boundary.

- **[P1] Make the advertised default actually isolate memory:** `extensions/memory-core/src/isolation-identity.ts:27-29`
  - body: When `memory.isolation` is omitted, `resolveMemorySearchConfig` sets isolation enabled, but `readIsolation()` returns `null`, so the tool context supplies no user ID and both builtin and QMD paths remain shared. This creates a false default-on security guarantee; resolve one canonical default consistently or keep isolation explicitly opt-in.
  - confidence: 0.99
- **[P1] Namespace sender identity by channel and account:** `extensions/memory-core/src/isolation-identity.ts:68-70`
  - body: Returning `requesterSenderId` verbatim lets identical raw IDs from different channels or configured accounts hash to the same memory directory and QMD state. Derive a canonical channel/account/peer identity before this layer rather than treating a transport-local sender ID as globally unique.
  - confidence: 0.96
- **[P1] Route normal memory flushes into the isolated write scope:** `src/memory-host-sdk/host/internal.ts:109-115`
  - body: The new guard allows every root `memory/*.md` file, while the existing compaction flush still writes user-derived durable notes to `memory/YYYY-MM-DD.md`. Once either user flushes, all isolated users can search/read those notes, so the claimed boundary fails on the normal write path.
  - confidence: 0.99
- **[P2] Implement or remove the unused isolation modes:** `src/config/zod-schema.ts:116-124`
  - body: The schema promises `scope` and `pathEncoding`, but runtime identity selection ignores `scope` and path construction always uses the helper's default hash encoding. Configurations such as `scope: "session"` or `pathEncoding: "whitelist"` are accepted yet have no effect, so either wire them through every backend or remove them before exposing the contract.
  - confidence: 0.98
- **[P1] Bound the per-user manager cache:** `extensions/memory-core/src/memory/manager.ts:169-186`
  - body: Appending `userId` creates a process-global cached `MemoryIndexManager` for every distinct sender, and those managers own database/watcher/sync resources with no eviction or idle teardown. A busy multi-user deployment will grow resources with user cardinality; use bounded lifecycle ownership or a shared manager with query-level tenant scoping.
  - confidence: 0.94
