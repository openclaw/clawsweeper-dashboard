## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor default isolation without explicit config:** `extensions/memory-core/src/isolation-identity.ts:64-66`
  - body: `resolveMemorySearchConfig` defaults `settings.isolation.enabled` to true when `memory.isolation` is absent, but this helper treats that same missing block as disabled and returns `undefined`. Existing configs with no new block therefore create unscoped memory managers despite the new default-on contract, so the isolation boundary is silently bypassed.
  - confidence: 0.92
- **[P1] Scope memory flush output before sharing root files:** `packages/memory-host-sdk/src/host/internal.ts:262-263`
  - body: In isolation mode this still exposes root-level `memory/*.md` to every user, but the current memory flush plan writes per-session durable notes to `memory/YYYY-MM-DD.md`. A compaction flush from one user will land in that shared root and then be returned to other isolated users, bypassing the privacy guarantee.
  - confidence: 0.9
- **[P2] Bound per-user manager cache entries:** `extensions/memory-core/src/memory/manager.ts:169-171`
  - body: Adding `userId` to the long-lived manager cache key creates a separate default `MemoryIndexManager` per sender, and those managers start watchers, session listeners, interval sync, and SQLite state until explicitly closed. A busy shared channel can accumulate unbounded managers and resources; use bounded eviction/idle teardown or a shared scoped manager design before enabling this broadly.
  - confidence: 0.83
