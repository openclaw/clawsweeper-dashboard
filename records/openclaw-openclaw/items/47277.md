Full review comments:

- **[P1] Honor default-on isolation when config is absent:** `extensions/memory-core/src/isolation-identity.ts:65-66`
  - body: `memory.isolation` is advertised and resolved as default-on, but this resolver returns `undefined` whenever the block is absent. Existing configs therefore create memory tools with no `userId`, so `memory_search`/`memory_get` use unscoped managers despite the new default.
  - confidence: 0.92
- **[P1] Scope QMD collections, not only manager state:** `extensions/memory-core/src/memory/search-manager.ts:104-105`
  - body: Passing `userId` into `QmdMemoryManager.create()` only splits manager state; `resolved` was already built from workspace-wide `MEMORY.md` and `memory/**/*.md` collections. QMD users can still index/search the whole workspace memory corpus unless collection resolution is user-scoped too.
  - confidence: 0.88
- **[P1] Keep private flush output out of shared root:** `src/memory-host-sdk/host/internal.ts:213`
  - body: For isolated users, `listMemoryFiles()` still includes root-level `memory/*.md` for everyone. Current memory flush writes private compaction notes to `memory/YYYY-MM-DD.md`, so a default flush can create shared content that every isolated user then indexes.
  - confidence: 0.9
- **[P2] Implement or remove accepted isolation options:** `src/config/zod-schema.ts:122-126`
  - body: The schema accepts `scope`, `fallbackPolicy`, and `pathEncoding`, but the runtime only partially consumes them; for example `resolveMemorySearchConfig()` preserves only `enabled`, and read/list paths use the default encoding. Shipping accepted-but-ignored isolation knobs would create a misleading config contract.
  - confidence: 0.82
