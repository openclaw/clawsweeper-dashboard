  - body: The PR advertises default-on isolation and resolveMemorySearchConfig defaults enabled to true, but readIsolation returns null when memory.isolation is omitted, causing resolveIsolationIdentity to return undefined and the tools to use unscoped memory for ordinary configs.
  - confidence: 0.93
- **[P1] Namespace sender identities before using them as user IDs:** `extensions/memory-core/src/isolation-identity.ts:66-67`
  - body: Returning the raw senderId means identical IDs from different channels/accounts can share one memory bucket, and the direct-session fallback only keeps the token after direct. The isolation key needs a canonical channel/account/user identity, not just the raw sender fragment.
  - confidence: 0.88
- **[P1] Stop shared flush files from bypassing per-user memory:** `packages/memory-host-sdk/src/host/internal.ts:256`
  - body: In isolation mode this branch still includes root memory/*.md for every user, while current flush writes durable notes to memory/YYYY-MM-DD.md. Once compaction flushes, one user's notes can become visible to every isolated user unless write scoping and shared-memory semantics are handled together.
  - confidence: 0.86
- **[P1] Bound the per-user manager lifecycle:** `extensions/memory-core/src/memory/manager.ts:168`
  - body: Adding userId to the default manager cache key creates one long-lived MemoryIndexManager per sender, including SQLite state, watchers, and timers, with no eviction or shared-manager strategy. Busy multi-user deployments can accumulate resources until the process becomes unstable.
  - confidence: 0.82
