## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor default-on isolation when config is absent:** `extensions/memory-core/src/isolation-identity.ts:65-66`
  - body: `resolveMemorySearchConfig` defaults isolation to enabled when `cfg.memory?.isolation` is missing, but this resolver treats the absent block as `null` and returns no identity. Fresh/default configs therefore pass no `userId` into memory managers and silently use shared memory instead of the advertised default-on isolation.
  - confidence: 0.93
- **[P1] Scope flush-written daily memory before sharing root files:** `src/memory-host-sdk/host/internal.ts:213`
  - body: With a `userId`, `listMemoryFiles` scans the user directory and then still adds root `memory/*.md` files. Current flush behavior writes durable session notes to `memory/YYYY-MM-DD.md`, so one user's flush note can be indexed and returned to every isolated user.
  - confidence: 0.91
- **[P2] Wire validated isolation options into runtime behavior:** `src/agents/memory-search.ts:425-426`
  - body: The schema accepts `scope`, `fallbackPolicy`, and `pathEncoding`, but the resolved runtime shape only preserves `enabled`, and manager/list/read calls use hard-coded defaults. Users can set validated isolation options and get different behavior than configured, which is risky for a privacy boundary.
  - confidence: 0.86
- **[P1] Bound per-user memory manager cache entries:** `extensions/memory-core/src/memory/manager.ts:170-171`
  - body: Adding `userId` to the default manager cache key creates one long-lived `MemoryIndexManager` per sender; default managers start watchers/listeners/interval sync and the shared cache has no idle eviction. A busy multi-user channel can accumulate SQLite handles and watchers indefinitely.
  - confidence: 0.82
