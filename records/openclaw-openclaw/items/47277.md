Full review comments:

- **[P1] Honor default-on isolation when config is absent:** `extensions/memory-core/src/isolation-identity.ts:27-29`
  - body: `resolveMemorySearchConfig()` defaults isolation to enabled when `memory.isolation` is absent, but this resolver returns `null` for the same absent block and then returns `undefined` as the user id. Existing configs that do not contain the new block can therefore still create unscoped memory managers even though the PR advertises default-on isolation.
  - confidence: 0.9
- **[P2] Remove or enforce unsupported isolation options:** `src/config/types.memory.ts:22-33`
  - body: The schema and exported type accept `scope` and `pathEncoding`, but the runtime identity/path code hard-codes user identity and hash encoding instead of reading those fields. Exposing security-relevant knobs that do not actually change behavior will mislead operators and make upgrade/debug behavior drift from config.
  - confidence: 0.82
- **[P2] Bound per-user manager cache growth:** `extensions/memory-core/src/memory/manager.ts:169-171`
  - body: Adding `userId` to this long-lived cache key can create one default `MemoryIndexManager` per sender, and each non-status manager opens SQLite plus file/session/interval watchers. Multi-user deployments need bounded eviction, idle teardown, or a shared-manager design before this can be safe to run under load.
  - confidence: 0.78
