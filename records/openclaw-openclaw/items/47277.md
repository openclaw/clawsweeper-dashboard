## Summary

Keep open. Current main still has shared memory behavior, and this PR is not merge-ready because it adds a default-on privacy boundary with static cross-user gaps in QMD, optional sender identity handling, raw user path handling, and config validation.

## What This Changes

The branch adds default-on per-user memory isolation plumbing through memory-core tool context, memory-host file checks, memory search managers, config types, and session-key helpers.

## Best Possible Solution

Replace or substantially rework the branch with an owner-reviewed design that canonicalizes user identity safely, scopes storage/search/read/write consistently for builtin and QMD, defines shared-memory behavior explicitly, and updates schema, docs, and regression coverage together.

## Reproduction Assessment

Yes. Source inspection of current main plus the PR diff gives a high-confidence static reproduction path: current main is shared, and the PR leaves unscoped or unsafe paths in QMD, sender fallback, memory-host path checks, and schema validation.

## Solution Assessment

No. The proposed patch is a partial default-on privacy boundary; the safer solution is a maintainer-owned isolation design with fail-closed identity, backend parity, write-path enforcement, and validated configuration.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope QMD collections by user:** `extensions/memory-core/src/memory/qmd-manager.ts:256`
  - body: `userId` is accepted here and forwarded from the caller, but the QMD manager still uses per-agent state, collection roots, cache identity, and query behavior. Deployments using `memory.backend: "qmd"` can remain agent-wide under the new isolation guarantee.
  - confidence: 0.9
- **[P1] Fall back to session identity when senderId is absent:** `extensions/memory-core/src/tools.shared.ts:59`
  - body: The tool context scopes memory only from optional `senderId`. If plugin context omits `requesterSenderId`, valid session-key-only executions still create unscoped managers despite the added session-key helper, so isolation can silently disappear.
  - confidence: 0.86
- **[P1] Encode user IDs before joining memory paths:** `packages/memory-host-sdk/src/host/internal.ts:181`
  - body: The isolated filesystem path is derived from raw sender/session identifiers. Values containing path separators, dot segments, or platform-invalid characters can escape, collapse, or fail the intended per-user directory boundary.
  - confidence: 0.78
- **[P2] Add the isolation option to the schema:** `src/config/types.memory.ts:25`
  - body: The PR adds and reads `memory.isolation`, but the strict `MemorySchema` is not updated, so users cannot configure the advertised `memory.isolation.enabled: false` legacy switch.
  - confidence: 0.89
