Full review comments:

- **[P1] Honor default-on isolation when config is absent:** `src/agents/memory-search.ts:425`
  - body: `resolveMemorySearchConfig` now treats missing `memory.isolation` as enabled, but the tool identity path treats an absent isolation block as no isolation and returns no `userId`. On the existing-user upgrade path, `memory_search`/`memory_get` can still create unscoped managers despite the default-on privacy promise, so the absent-config case must either fail closed or the feature must be explicitly opt-in.
  - confidence: 0.91
- **[P2] Remove or enforce unsupported isolation options:** `src/config/zod-schema.ts:122-126`
  - body: The schema accepts `scope` and `pathEncoding`, but the runtime paths reviewed only propagate `enabled` and otherwise default to user/hash behavior at the call sites. Users can set accepted config that has no effect, so either remove those fields for now or thread and test them end to end.
  - confidence: 0.84
- **[P2] Bound per-user manager cache growth:** `extensions/memory-core/src/memory/search-manager.ts:63-64`
  - body: Including `userId` in QMD manager cache keys creates one cached manager per sender, while the manager layer owns SQLite handles, watchers, timers, and subscriptions. In a busy multi-user channel this can grow without an idle close or eviction policy, so add a bounded lifecycle or use a shared manager with per-request scoping.
  - confidence: 0.78
