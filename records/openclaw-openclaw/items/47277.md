## Summary

Keep this PR open. Current main still does not implement per-user memory isolation, and the PR head remains an open, unmerged, non-mergeable implementation with blocking privacy-boundary gaps in QMD, identity derivation, raw user-id path handling, and config validation.

## What This Changes

This PR adds default-on user-specific memory isolation across the memory-core plugin tools, memory manager/file discovery and reads, memory config types, and session-key utilities.

## Best Possible Solution

Keep the work open, but require a maintainer-reviewed memory isolation design that scopes identity consistently across storage, query, reads, writes/flushes, builtin and QMD backends, config schema/help/docs, and regression tests before merge.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: current main has no user scope in memory tool context, file discovery, or reads, while the PR head still leaves QMD and some tool executions unscoped.

## Solution Assessment

No. The current PR is not the best way to solve the issue because it adds a partial default-on privacy boundary without complete backend, identity, schema, and write-path enforcement; a narrower, complete memory-core and memory-host-sdk design is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope QMD collections by user:** `extensions/memory-core/src/memory/qmd-manager.ts:256`
  - body: `userId` is accepted here but never stored or used when creating QMD collections, so `memory.backend: "qmd"` still indexes and searches the agent-wide memory set. That bypasses the isolation guarantee for QMD deployments.
  - confidence: 0.9
- **[P1] Fall back to the session key for user scope:** `extensions/memory-core/src/tools.shared.ts:59`
  - body: This returns a scoped `userId` only when `senderId` is present, but `requesterSenderId` is optional and the new session-key extractor is never used here. Session-key-only direct runs therefore create unscoped managers and can read shared memory despite default-on isolation.
  - confidence: 0.86
- **[P2] Add the isolation option to the schema:** `src/config/types.memory.ts:25`
  - body: The PR adds and reads `memory.isolation`, but the strict `MemorySchema` still rejects that key. Users cannot set `memory.isolation.enabled: false`, so the advertised legacy compatibility switch is not actually configurable.
  - confidence: 0.89
- **[P1] Encode user IDs before joining paths:** `src/memory-host-sdk/host/internal.ts:181`
  - body: `userId` is used directly as a filesystem segment for the isolated memory directory. Sender/session identifiers that contain path separators or dot segments can escape or collapse buckets, so isolation should map identities to a guaranteed-safe encoded segment before path checks and joins.
  - confidence: 0.78
