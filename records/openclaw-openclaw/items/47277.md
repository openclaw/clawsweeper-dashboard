Full review comments:

- **[P1] Honor default-on isolation when config is absent:** `extensions/memory-core/src/isolation-identity.ts:27-29`
  - body: `readIsolation()` returns `null` when `memory.isolation` is missing, so `resolveIsolationIdentity()` returns no `userId` even though the memory config resolver treats absent isolation as enabled. Existing configs without the new block can therefore keep running unscoped while the resolved settings claim isolation is on.
  - confidence: 0.93
- **[P1] Stop sharing flush-written root memory across users:** `src/memory-host-sdk/host/internal.ts:213`
  - body: The isolated listing still includes root-level `memory/*.md` for every user, but the current flush path writes durable notes to `memory/YYYY-MM-DD.md`. A note flushed during one user's session can therefore be indexed and returned to other isolated users.
  - confidence: 0.91
- **[P1] Scope QMD collections by user, not only state:** `packages/memory-host-sdk/src/host/backend-config.ts:366-367`
  - body: The PR gives each user a separate QMD state/cache path, but the default QMD collections still index workspace `MEMORY.md` and `memory/**/*.md`. Each per-user manager will still ingest the shared memory tree unless the collection set or query layer is user-scoped.
  - confidence: 0.9
- **[P2] Apply the new isolation options at runtime:** `src/agents/memory-search.ts:420-426`
  - body: The schema accepts `scope`, `fallbackPolicy`, and `pathEncoding`, but the runtime paths shown here only resolve `enabled`, while other code keeps hard-coded defaults. Operators can set options that appear valid but do not consistently control memory isolation behavior.
  - confidence: 0.84
