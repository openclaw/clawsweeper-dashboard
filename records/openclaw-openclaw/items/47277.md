## Summary

Keep open: current main does not have the requested memory isolation surface, and this PR is still the active implementation candidate, but it is not merge-ready because source inspection of the latest PR head still shows privacy-boundary gaps and no real two-user behavior proof. The KinthAI production comment usefully reinforces that storage, query, and write isolation all need to be proven together.

## What This Changes

The PR adds user-scoped memory isolation across config, memory-core tool identity propagation, builtin/QMD memory managers, host SDK path guards, SDK exports, and tests.

## Best Possible Solution

Land an owner-reviewed memory privacy design that consistently scopes identity, config semantics, read/search/write paths, QMD collections/state, flush targets, docs, migration behavior, and real two-user validation.

## Reproduction Assessment

Yes for the PR defects: PR-head source inspection shows default config, QMD collection, and shared flush/root-memory paths that can bypass the intended privacy boundary. I did not execute tests or a live two-user setup because this review is read-only.

## Solution Assessment

No. The branch contains useful building blocks, but a default-on privacy boundary should not merge until backend parity, write-path scoping, config semantics, and real two-user proof are coordinated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Enable isolation when the config block is omitted:** `extensions/memory-core/src/isolation-identity.ts:64-66`
  - body: `resolveMemorySearchConfig` defaults isolation to enabled, but this resolver treats a missing `memory.isolation` block as disabled and returns no `userId`. Existing configs would register memory tools and managers without the advertised default-on privacy boundary.
  - confidence: 0.93
- **[P1] Scope QMD collections by user:** `extensions/memory-core/src/memory/qmd-manager.ts:347-351`
  - body: This creates a per-user QMD state directory, but the resolved default collections still target the workspace memory root and `memory/**/*.md`. Each user's QMD index can therefore ingest other users' memory files despite isolated state paths.
  - confidence: 0.9
- **[P1] Route flush memory into user directories:** `packages/memory-host-sdk/src/host/internal.ts:262-263`
  - body: With a `userId`, `listMemoryFiles` still includes root-level `memory/*.md` files for every isolated user, while the existing flush plan writes durable notes to `memory/YYYY-MM-DD.md`. A flush from one session can become searchable by other users.
  - confidence: 0.88
- **[P2] Honor accepted isolation scope and path encoding:** `src/agents/memory-search.ts:424-426`
  - body: The schema accepts `scope` and `pathEncoding`, but the resolved runtime settings keep only `enabled`, and callers use hard-coded user scope and hash encoding. Valid config can parse while silently having no effect on the isolation boundary.
  - confidence: 0.82
