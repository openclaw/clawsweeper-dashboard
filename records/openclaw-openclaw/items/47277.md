## Summary

Keep open. This PR adds a security-sensitive privacy boundary, but the branch still leaves user scope incomplete across QMD, identity derivation, filesystem path handling, and config validation, so it should not merge as-is.

## What This Changes

The branch adds default-on per-user memory isolation plumbing for memory-core tools, memory search managers, memory-host file checks, config types, and session-key helpers.

## Best Possible Solution

Replace or substantially rework the branch with an owner-reviewed design that canonicalizes user identity safely, enforces storage/search/read/write scope for builtin and QMD backends, and updates schema, docs, and regression coverage together.

## Reproduction Assessment

Yes. Source inspection of current main plus the PR diff gives a high-confidence static reproduction path: current main is shared, and the branch still leaves concrete unscoped or unsafe paths in QMD, sender fallback, and memory-host path checks. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The proposed patch is not the best way to solve this because it adds a partial default-on privacy boundary without complete backend parity, safe identity mapping, schema validation, and write-path policy; the safer path is a maintainer-owned isolation design.

  - body: The PR adds and reads `memory.isolation`, but the strict `MemorySchema` is not updated, so users cannot actually configure the advertised `memory.isolation.enabled: false` legacy switch.
