## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor default-on isolation without a config block:** `extensions/memory-core/src/isolation-identity.ts:64-66`
  - body: `resolveMemorySearchConfig` resolves a missing `memory.isolation` block as enabled, but this resolver returns `undefined` when the block is absent. Default and upgraded configs therefore create memory tools with no `userId`, so search/read paths fall back to unscoped memory despite the new default claiming isolation.
  - confidence: 0.93
- **[P2] Do not accept isolation options until runtime honors them:** `src/config/zod-schema.ts:122-126`
  - body: The schema accepts `scope` and `pathEncoding`, but the runtime paths reviewed only carry `enabled` and partly read `fallbackPolicy`; path helpers still default to hash mode and identity always prefers sender/direct user. Operators can set accepted isolation options that silently do not change behavior, which is unsafe for a security boundary.
  - confidence: 0.87
- **[P2] Bound per-user memory manager cache growth:** `extensions/memory-core/src/memory/manager.ts:169-171`
  - body: Adding `userId` to the default manager cache key creates a long-lived manager per sender. Those managers start watchers, session listeners, interval sync, and SQLite handles, while the shared cache only shrinks on explicit close; a busy channel can accumulate resources until availability degrades.
  - confidence: 0.8
