- **[P1] Honor default isolation when config is absent:** `extensions/memory-core/src/isolation-identity.ts:27-28`
  - body: `readIsolation()` returns `null` when `cfg.memory.isolation` is missing, so `resolveIsolationIdentity()` returns `undefined` even though the PR's schema/resolved settings advertise isolation as default-on.   - confidence: 0.9
- **[P1] Scope flush output before sharing root memory:** `src/memory-host-sdk/host/internal.ts:213`
  - body: In isolated mode this still indexes root `memory/*.md` for every user, while memory-core's default flush target remains `memory/YYYY-MM-DD.md`. A DM compaction can write private durable notes to root memory and then every isolated user can read/search them.
  - confidence: 0.9
- **[P2] Honor the new isolation config fields:** `src/agents/memory-search.ts:420-426`
  - body: The schema accepts `scope`, `fallbackPolicy`, and `pathEncoding`, but the resolved runtime isolation config keeps only `enabled`, and the path/QMD callers default to hash/user behavior. Users can set accepted config that has no consistent effect across the new isolation paths.
  - confidence: 0.84
- **[P2] Bound per-user memory manager cache entries:** `extensions/memory-core/src/memory/manager.ts:169-171`
  - body: Adding `userId` to the manager cache key creates one long-lived builtin/QMD manager per sender; default managers start watchers/listeners/timers and hold SQLite or QMD state until explicit close. A busy multi-user deployment needs bounded eviction, idle teardown, or a shared scoped-manager design.
  - confidence: 0.83
