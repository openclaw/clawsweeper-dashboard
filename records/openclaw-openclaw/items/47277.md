## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Apply the default isolation mode when config is omitted:** `extensions/memory-core/src/isolation-identity.ts:27-29`
  - body: `resolveMemorySearchConfig` defaults isolation to enabled when `memory.isolation` is absent, but this resolver returns `undefined` for the same config shape. That means the advertised default-on mode can create unscoped managers/tools and silently use shared memory unless the user adds an explicit isolation block.
  - confidence: 0.94
- **[P1] Scope QMD collections by the resolved user:** `extensions/memory-core/src/memory/qmd-manager.ts:347-351`
  - body: Passing `userId` into the QMD state directory only gives each user a private index file; the resolved QMD collections still point at the whole workspace `memory/**/*.md` tree. Each user's QMD manager can therefore index other users' files into its private database, so QMD isolation is still not structural.
  - confidence: 0.91
- **[P1] Keep memory-flush notes out of the shared root:** `src/memory-host-sdk/host/internal.ts:213`
  - body: In isolated mode this still indexes root `memory/*.md` for every user, while the automatic memory flush writes session content to `memory/YYYY-MM-DD.md`. A private conversation can be flushed into the shared root and then searched by another isolated user.
  - confidence: 0.9
- **[P2] Wire every advertised isolation option into runtime:** `src/config/zod-schema.ts:122-126`
  - body: The schema accepts `scope`, `fallbackPolicy`, and `pathEncoding`, but the runtime mostly carries only `enabled`; `scope` is ignored and path encoding stays hard-coded to hash. Exposing config that does not control behavior will give operators a false sense that they selected a stricter partition mode.
  - confidence: 0.86
