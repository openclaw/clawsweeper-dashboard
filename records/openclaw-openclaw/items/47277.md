## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Scope identities by route context:** `extensions/memory-core/src/isolation-identity.ts:69-70`
  - body: `resolveIsolationIdentity` prefers the raw `senderId` before looking at the session key, but the documented secure DM mode distinguishes channel/account+sender identities. In `dmScope: "main"` or multi-account/channel setups, two different users with the same provider-local sender ID can share one memory bucket, which breaks the isolation guarantee.
  - confidence: 0.9
- **[P1] Apply default isolation consistently:** `src/agents/memory-search.ts:282-285`
  - body: This adds `isolation.enabled` as default-on in the resolved memory search settings, but `readIsolation()` returns `null` when `cfg.memory.isolation` is absent. Default configs therefore create tools with no `userId` while the manager layer still believes isolation is enabled, leaving searches/reads unscoped instead of fail-closed or isolated.
  - confidence: 0.88
- **[P1] Route flush writes into isolated memory buckets:** `packages/memory-host-sdk/src/host/internal.ts:262-263`
  - body: When `userId` is present, `listMemoryFiles` still indexes root-level `memory/*.md` for every user. The current flush plan writes compaction memories to `memory/YYYY-MM-DD.md`, so private notes captured during one user's session remain in a shared path and can be returned to other isolated users.
  - confidence: 0.92
- **[P2] Honor the full isolation config surface:** `src/config/zod-schema.ts:122-126`
  - body: The schema accepts `scope`, `fallbackPolicy`, and `pathEncoding`, but the runtime only partly consumes `enabled`; QMD and file-read paths still default to hash encoding and the configured scope is not enforced. Shipping accepted but ineffective isolation settings creates a misleading security/config contract.
  - confidence: 0.86
