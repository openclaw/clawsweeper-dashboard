## Summary

Keep open: current main does not implement user-specific memory isolation, and this PR is still not mergeable because source inspection of the latest head shows remaining cross-user privacy gaps plus missing real two-user behavior proof.

## What This Changes

The PR adds default-on user-scoped memory isolation across memory config/schema, memory-core tool identity propagation, builtin and QMD memory manager scoping, path-encoding helpers, SDK exports, and tests.

## Best Possible Solution

Land an owner-reviewed memory privacy design that consistently scopes identity, read/search/write paths, QMD collections/state, flush targets, config semantics, docs, and live two-user validation.

## Reproduction Assessment

Yes for the PR defects: PR-head source inspection shows default config, QMD collection, and flush/root-memory mismatches that can bypass the intended privacy boundary. I did not run tests because this review is read-only.

## Solution Assessment

No. The branch contains useful building blocks, but a default-on privacy boundary is not the best merge shape until backend parity, write-path scoping, config semantics, and real two-user proof are coordinated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Enable isolation when the config block is omitted:** `extensions/memory-core/src/isolation-identity.ts:64-66`
  - body: The search config resolves `isolation.enabled` to `true` by default, but `resolveIsolationIdentity` returns `undefined` when `memory.isolation` is absent. Existing configs then register memory tools with no `userId` and build unscoped managers, bypassing the advertised default-on boundary.
  - confidence: 0.93
- **[P1] Scope QMD collections by user:** `extensions/memory-core/src/memory/qmd-manager.ts:347-351`
  - body: This only puts QMD state under `users/<encoded>`, while the resolved default collections still point at workspace `MEMORY.md` and `memory/**/*.md`. Each user's QMD manager can still ingest other users' memory files, so search isolation is incomplete.
  - confidence: 0.9
- **[P1] Route memory flush files into user directories:** `packages/memory-host-sdk/src/host/internal.ts:262-263`
  - body: With a `userId`, `listMemoryFiles` still includes root `memory/*.md` files for every user, and the existing flush plan writes durable compaction notes to `memory/YYYY-MM-DD.md`. A flush from one session can become searchable by other isolated users.
  - confidence: 0.88
- **[P2] Honor accepted isolation options:** `src/agents/memory-search.ts:424-426`
  - body: The schema accepts `scope` and `pathEncoding`, but the runtime-resolved settings keep only `enabled` and the path helpers keep using hard-coded hash encoding. Valid config can parse while silently having no effect on the isolation boundary.
  - confidence: 0.82
