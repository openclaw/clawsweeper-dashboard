## Summary

Keep open: the reported auto-reply precedence bug is source-reproducible and the PR targets the right code path, but this external draft PR lacks real behavior proof and the diff regresses the lazy catalog-loading fast path for global-default-only configs.

## What This Changes

The PR changes auto-reply thinking-default resolution so per-model `params.thinking` can override global `thinkingDefault`, with a focused regression test.

## Best Possible Solution

Update auto-reply to preserve per-agent overrides, honor per-model `params.thinking` before the global default, and keep runtime catalog hydration only for cases where catalog metadata is actually needed.

## Reproduction Assessment

Yes, from source inspection. Current main returns the global auto-reply `thinkingDefault` before reaching the shared resolver that already implements per-model `params.thinking` precedence.

## Solution Assessment

No, the current PR diff is not the best complete fix because it removes the global-default no-hydration path. The safer fix should preserve lazy catalog behavior while allowing per-model config to win.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve the global-default fast path:** `src/auto-reply/reply/model-selection.ts:280-281`
  - body: When only `agents.defaults.thinkingDefault` is configured, this deletion makes auto-reply call `resolveThinkingCatalog()` before the shared resolver can return the global default. If the selected model is absent from the configured catalog or lacks `reasoning` metadata, that unnecessarily loads the runtime model catalog on ordinary replies; keep the per-model override fix while returning the global default without hydration when no per-model override applies.
