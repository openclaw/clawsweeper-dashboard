## Summary

Keep open: current main still has a WhatsApp-specific final delivery gap. WhatsApp normalizes assistant text and plain-text scaffolding, but the compact emoji tool-trace line stripper is still Discord-local, so the reported leak remains source-reproducible.

## What This Changes

The issue reports WhatsApp outbound delivery exposing internal OpenClaw tool-trace lines that Discord strips and asks for channel-agnostic final-message sanitization.

## Best Possible Solution

Centralize the compact internal trace-line stripper in the shared assistant-visible/plain-text sanitizer path used by final channel delivery, apply it to WhatsApp, and add focused regression tests for the reported 🛠️/🩹 lines plus preservation of intentional tool-progress payloads.

## Reproduction Assessment

Yes for source-level reproduction: current main routes WhatsApp final text through normalizeWhatsAppPayloadText and sanitizeForPlainText, neither of which strips the reported compact 🛠️/🩹 trace lines. I did not run a live WhatsApp send because this was a read-only review.

## Solution Assessment

Yes, the best fix is to move the Discord-local trace-line stripping into a shared final-message sanitizer and consume it from WhatsApp rather than copying Discord code directly. The repair should preserve explicit tool/progress delivery boundaries where those messages are intentionally visible.
