## Summary

Keep open. Current main still fully reads cron run-log JSONL files before pagination in both per-job and all-jobs paths, and the issue is also paired with open unmerged PR #57088, so it should not be closed before that PR is reviewed, landed, or replaced.

## What This Changes

The issue asks for a hard cap or bounded read path for cron run-log JSONL pagination so oversized logs cannot be fully loaded into memory.

## Best Possible Solution

Keep the issue open until #57088 or a replacement lands. The desired end state is a race-safe bounded read for both per-job and all-jobs cron run-log pagination, preferably tail-reading a configured byte window without mutating files; if pruning is used, it should be serialized with appends and avoid another unbounded full-file read.

## Reproduction Assessment

Yes. A high-confidence source-level repro is to create an oversized `runs/<jobId>.jsonl` and call `readCronRunLogEntriesPage` or `cron.runs`; current main reads the whole file before filtering and slicing, and the all-jobs path does the same for every JSONL file.

## Solution Assessment

No for current main. The current implementation still exposes the reported memory behavior; the safest fix is a bounded read implementation, with #57088 treated as an implementation candidate that still needs review against the race and full-read-prune concerns.
