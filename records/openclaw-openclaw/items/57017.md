## Summary

Keep open. Current main still fully reads cron run-log JSONL files before pagination in the per-job and all-jobs paths, and the issue is paired with an open unmerged implementation PR, so closing would orphan the remaining fix.

## What This Changes

Request a race-safe size-capped cron run-log read path so oversized JSONL logs cannot be fully loaded into memory during pagination.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/57088 with a bounded, race-safe implementation for both per-job and all-jobs cron run-log pagination, honoring cron.runLog settings and adding oversized-read regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create an oversized runs/<jobId>.jsonl and call readCronRunLogEntriesPage or cron.runs; current main reads the complete file text before pagination.

## Solution Assessment

No for current main. The implementation still performs full-file reads, and the maintainable fix should bound read memory, preferably by tail-reading without mutating logs or by a serialized best-effort prune if maintainers accept that approach.
