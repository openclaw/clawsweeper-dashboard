## Summary

Keep open. Current main and the latest release still read cron run-log JSONL files into memory before pagination, and an open closing PR remains the active implementation path rather than a reason to close this issue early.

## What This Changes

The issue reports that cron run-log pagination can load oversized JSONL files into memory without a hard read cap.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/57088 with a current-main fix that bounds per-job and all-jobs cron run-log read memory, respects cron.runLog settings, preserves fs-safe semantics, and adds oversized-read regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create an oversized cron/runs/<jobId>.jsonl and call readCronRunLogEntriesPage or cron.runs; current main reads full JSONL files before pagination.

## Solution Assessment

No, current main is not fixed yet. The maintainable fix should use bounded reads or serialized best-effort pruning for both per-job and all-jobs paths while preserving existing filters, config, and fs-safe contracts.
