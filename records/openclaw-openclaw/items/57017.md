Keep open. Current main and the latest release still read cron run-log JSONL files into memory before pagination, and the open closing PR remains the active implementation path rather than a reason to close this issue early.

## What This Changes

The issue reports that cron run-log pagination can load an oversized JSONL file fully into memory before slicing results.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/57088 with a current-main fix that bounds per-job and all-jobs cron run-log reads while preserving fs-safe access, runtime prune settings, filters, and oversized-read coverage.

## Reproduction Assessment

Yes from source inspection. Create an oversized `cron/runs/<jobId>.jsonl` file and call `readCronRunLogEntriesPage` or the `cron.runs` all-jobs path; current main reads full JSONL contents before pagination.

## Solution Assessment

No, current main is not fixed yet. The maintainable fix should use bounded reads or serialized best-effort pruning for both per-job and all-jobs paths while preserving existing filters, config, and fs-safe contracts.
