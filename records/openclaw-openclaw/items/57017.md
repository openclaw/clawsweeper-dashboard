- `impact:crash-loop`: The reported failure mode is an OOM crash or severe GC pressure when reading oversized cron run-log files.

## Summary

Keep open. Current main and v2026.5.18 still read entire cron run-log JSONL files before pagination on both per-job and all-jobs paths, and the issue is already paired with open closing PR https://github.com/openclaw/openclaw/pull/57088.

## What This Changes

Requests bounding cron run-log reads so oversized JSONL files cannot be loaded fully into memory before pagination.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/57088 with a current-main fix that bounds per-job and all-jobs cron run-log read memory, respects cron.runLog settings, preserves fs-safe write semantics, and adds oversized-read regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create an oversized cron/runs/<jobId>.jsonl and call readCronRunLogEntriesPage or cron.runs; current main reads the full file before pagination.

## Solution Assessment

No for current main. The maintainable fix should use bounded reads or a serialized best-effort prune for both per-job and all-jobs paths while preserving existing config, fs-safety, and query contracts.
