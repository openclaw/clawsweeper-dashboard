## Summary

Keep open. Current main and the latest release still read cron run-log JSONL files fully before pagination, and the issue is paired with an open implementation PR that should be repaired, replaced, or closed before the issue is resolved.

## What This Changes

Add a bounded read or race-safe defensive pruning path so oversized cron run-log JSONL files cannot be fully loaded before pagination.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/57088 with a current-main fix that bounds per-job and all-jobs cron run-log read memory, honors cron.runLog settings, preserves fs-safe writes and search fields, and adds oversized-read regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create an oversized cron/runs/<jobId>.jsonl and call readCronRunLogEntriesPage or cron.runs; current main reads the full file before pagination.

## Solution Assessment

No for current main. The maintainable fix should use a bounded read or a serialized best-effort prune for both per-job and all-jobs paths while preserving existing config, fs-safety, and query contracts.
