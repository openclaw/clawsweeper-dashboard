## Summary

Keep open: the PR is aimed at the right narrow Control UI gap, but the current diff fetches `derivedTitle` without routing it through the actual picker option-label path, so rows that only have a derived title can still show the parsed/raw session key in the dropdown.

## What This Changes

The PR adds `includeDerivedTitles: true` to Control UI chat session picker `sessions.list` requests, adds `derivedTitle` to the UI session row type, and tests `resolveSessionDisplayName` fallback behavior.

## Best Possible Solution

Route `derivedTitle` through the actual picker option-label path, update the strict request-shape tests, and provide live Control UI proof of the dropdown rendering readable titles.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: picker rows render labels from `resolveSessionOptionGroups`, and that path still ignores rows where `derivedTitle` is the only friendly name. I did not run tests because this review is read-only.

## Solution Assessment

Is this the best way to solve the issue? No: the direction is right, but updating only `resolveSessionDisplayName` misses the dropdown option-label helper that actually renders the picker list.
