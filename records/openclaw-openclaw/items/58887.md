## Summary

This remains open because current main still sends the normal Telegram typing cue only after media resolution and audio transcription have already completed for voice/audio messages, matching the prior review and leaving a real Telegram UX gap.

## What This Changes

Request an immediate Telegram typing indicator when a voice message is received, before audio download/STT transcription delays the agent loop.

## Best Possible Solution

Add a Telegram-owned early typing path for eligible voice/audio messages after cheap authorization/thread checks but before media download and STT, with regression coverage for ordering and ignored-message cases.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the Telegram handler awaits resolveMedia, then context construction awaits resolveTelegramInboundBody/transcribeFirstAudio, before the existing context.sendTyping call is reached.

## Solution Assessment

Yes. The narrow maintainable fix is in the Telegram plugin ingress/context layer using existing sendChatAction behavior, not a broad core typing policy change.
