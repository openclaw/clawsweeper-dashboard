## Summary

Keep this issue open. Current main still delays Telegram typing for voice/audio messages until after media download and preflight transcription finish, and the adjacent Telegram performance PR does not cover the requested pre-STT typing signal.

## What This Changes

Request that Telegram voice/audio messages emit a typing indicator immediately after receipt, before media download and STT transcription delay the normal reply-loop typing cue.

## Best Possible Solution

Add a narrow, authorized early-typing path for Telegram voice/audio messages before media download and `transcribeFirstAudio`, with focused tests proving the cue fires before the slow voice path for allowed messages and does not fire for unauthorized or dropped messages.

## Reproduction Assessment

Yes. The static call path is enough to reproduce the ordering: `resolveMedia` and `transcribeFirstAudio` are awaited before `context.sendTyping()` can run, and a focused Telegram unit test can assert that ordering with mocked media/STT delays.

## Solution Assessment

Yes. A Telegram-first handler fix is the narrowest maintainable solution for this issue, reusing the existing `sendChatActionHandler` and threading helpers while preserving the current access gates.
