## Summary

Keep open: current main has several useful partial fixes for configured and forward-compatible models, but the RFC’s broader invariant is not fully implemented or accepted as a product contract across list/status/runtime/capability/docs. The discussion already points to a phased design approach, and the remaining work is architectural rather than conservative cleanup.

## What This Changes

The issue asks OpenClaw to treat `models.providers`-injected models as first-class forward-compatible models across status, list, runtime, capability gates, validation, and docs.

## Best Possible Solution

Define a small canonical resolved-model contract with identity, transport, capabilities, provenance, and availability, then migrate status/list/runtime/picker/capability consumers to it in reviewed phases and document the supported limits.

## Reproduction Assessment

Not applicable as a direct bug reproduction: this is an RFC for a product/design invariant. Source inspection confirms partial, surface-specific implementations rather than one single failing current-main path.

## Solution Assessment

No single one-off fix is the best solution. The maintainable path is a maintainer-approved canonical resolution descriptor and phased consumer migration, preserving provider/plugin ownership for provider-specific policy.
