## Summary

Current main treats configured provider models as first-class across the relevant surfaces: catalog/list visibility, runtime fallback resolution, availability fallback, model picker views, and thinking capability gates now consume configured model metadata or provider-owned compatibility hooks. The same implementation is present in the shipped `v2026.5.3` tag, so this RFC can be closed as implemented while narrower provider-specific regressions can stay in their own issues.

## What This Changes

The issue proposes making `models.providers.<provider>.models` injected models a first-class forward-compatibility path across status, list, runtime, availability, capability gates, and docs.

## Best Possible Solution

Keep the current generic configured-catalog, runtime-resolution, and provider-compat hooks, and track future provider-specific drift as narrow follow-up issues rather than keeping this broad RFC open.

## Reproduction Assessment

Not applicable. This is an RFC/design-alignment request, and source plus test/docs inspection shows the requested invariant is now represented in the current implementation.

## Solution Assessment

Yes. The current approach is the maintainable direction: configured provider metadata feeds generic catalog/list/runtime paths, while provider-owned hooks and `compat` metadata handle transport and capability differences without adding more core allowlists.
