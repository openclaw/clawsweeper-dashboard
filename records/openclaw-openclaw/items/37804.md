## Summary

Keep open: current main has useful partial support for config-injected provider models, but it still does not implement the RFC's central invariant of one canonical resolved-model descriptor consumed by runtime, CLI status/list, Gateway/UI pickers, availability/auth logic, capability gates, validation, and docs.

## What This Changes

The issue requests treating `models.providers.<provider>.models` entries as first-class forward-compat models across status, list, runtime, availability/auth heuristics, capability gates, validation, docs, and picker-style consumers.

## Best Possible Solution

Define a shared resolved-model result under the model selection/catalog layer that carries identity, provider/model transport/API, capability metadata, configured/discovered/synthetic provenance, availability/auth/probe state, and unsupported reason, then migrate runtime resolution, `models status`, `models list`, Gateway/UI model pickers, auto-reply/session thinking gates, validation, and docs to consume it in planned phases.

## Reproduction Assessment

Not applicable as a single bug reproduction: this is an RFC/design invariant, not one failing command. There is high-confidence source evidence that the remaining behavior is still split across separate runtime, catalog, CLI row, Gateway, and thinking-policy paths.

## Solution Assessment

No, the current partial fixes are not the best final solution for the full issue. The maintainable path is the phased shared-descriptor design described in the discussion, with one source of truth and focused migrations rather than more one-off allowlist/list/status patches.
