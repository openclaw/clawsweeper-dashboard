## Summary

Keep open: current main has useful partial support for configured provider models, but the RFC's shared resolved-model descriptor is still not implemented across runtime, CLI status/list, Gateway/UI pickers, availability/auth, capability policy, validation, and docs.

## What This Changes

The issue requests making `models.providers.<provider>.models` entries a first-class forward-compatible model source across status, listing, runtime, capability gates, validation, and documentation.

## Best Possible Solution

Define a shared resolved-model result in the model selection/catalog layer that carries identity, transport/API, capabilities, provenance, availability/auth/probe state, and unsupported reason, then migrate runtime, CLI, Gateway/UI, auto-reply/session thinking policy, validation, and docs to consume it in phases.

## Reproduction Assessment

Not applicable as a single bug reproduction: this is an RFC/design invariant, not one failing command. Source inspection gives high confidence that current main still computes model answers through separate catalog, runtime, status/list, Gateway, and thinking-policy paths.

## Solution Assessment

No: the current partial support is useful, but it is not the best final solution for the RFC. The maintainable path is a phased shared-descriptor design, with focused migrations and regression coverage for discovered, configured-only, and synthetic forward-compatible models.
