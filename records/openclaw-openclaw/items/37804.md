## Summary

Keep open: current main has substantial partial support for configured provider models, but the RFC's broader invariant is not terminally resolved because model resolution is still split across CLI list/status, Gateway catalog, runtime, and thinking-policy paths, with related capability drift still tracked separately.

## What This Changes

Recognize `models.providers`-injected provider models as first-class forward-compat models across status, list, runtime, availability, capability gates, validation, and docs.

## Best Possible Solution

Have maintainers define the canonical model-resolution contract, then either close this RFC once current catalog/runtime/thinking surfaces are judged sufficient or split remaining gaps into narrower follow-up issues.

## Reproduction Assessment

Not applicable as a feature/RFC rather than a concrete bug report. Source review shows partial current-main support across the named surfaces, but no single failing reproduction path is defined for this umbrella item.

## Solution Assessment
