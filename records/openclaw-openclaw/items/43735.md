## Summary

Keep open. Current main discovers workspace skills, but the stale persisted snapshot path still appears valid: process-local skill versions start at 0, so a persisted `skillsSnapshot.version: 0` can be treated as current after restart even when the on-disk skill inventory changed.

## What This Changes

The issue asks for valid workspace skills under `<workspace>/skills` to appear reliably in the agent `<available_skills>` context instead of only a stale subset.

## Best Possible Solution

Land #71497 or an equivalent shared refresh-state fix that invalidates old persisted skill snapshots after restart while preserving source precedence, skill gates, containment checks, and prompt-size limits.

## Reproduction Assessment

Yes. A high-confidence source reproduction is: persist a session with `skillsSnapshot.version: 0`, stop OpenClaw, change the workspace skills inventory, restart, then the gateway and CLI paths compare cached version 0 against process-local version 0 and reuse the old snapshot.

## Solution Assessment

No, current main is not the solved state. The narrow maintainable solution is restart-safe invalidation in the shared refresh-state layer, not unbounded skill prompt injection or a separate reload command.
