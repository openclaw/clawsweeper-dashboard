## Summary

Keep open. Current main supports workspace skill discovery, including one-level grouped skill directories, but the persisted skills snapshot refresh path still relies on process-local numeric versions that restart at `0`; a stale version-0 snapshot can still be reused after restart. The related open draft maintainer PR #71497 appears to target the same root cause, so this should stay open for that PR or an equivalent fix instead of being closed or queued as a competing repair.

## What This Changes

The issue asks for valid workspace skills to be reliably discovered and surfaced in the agent `<available_skills>` context instead of disappearing or requiring manual directory inspection.

## Best Possible Solution

Land #71497 or an equivalent restart-safe skills snapshot invalidation fix that rebuilds persisted snapshots after process restarts when the effective skill inventory changed, while preserving source precedence, skill gates, containment checks, and prompt-size limits.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is a persisted session `skillsSnapshot.version` of `0`, a skill inventory change while OpenClaw is stopped, then a restart where `getSkillsSnapshotVersion()` is still `0` and both gateway and CLI refresh checks treat the old snapshot as current.

## Solution Assessment

No, current main is not the solved state. The maintainable direction is restart-safe invalidation in the shared refresh path, not forcing every discovered skill into the prompt without caps.
