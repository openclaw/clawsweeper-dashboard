## Summary

Keep open. Current main supports workspace skill discovery, but the stale persisted snapshot path still appears valid: after restart, a cached version-0 `skillsSnapshot` can compare as current when the process-local skill version is also 0, so the reported missing `<available_skills>` behavior is not fully fixed.

## What This Changes

The issue requests that valid workspace skills under `~/.openclaw/workspace/skills/` reliably appear in the agent’s initial `<available_skills>` context.

## Best Possible Solution

Land #71497 or an equivalent restart-safe snapshot invalidation fix that rebuilds stale persisted skill snapshots after process restarts while preserving source precedence, skill gates, containment checks, and prompt-size limits.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is a persisted session with `skillsSnapshot.version: 0`, a skill inventory change while OpenClaw is stopped, and then a restart where `getSkillsSnapshotVersion()` is still 0 so both gateway and CLI refresh checks reuse the old snapshot.

## Solution Assessment

No, current main is not the solved state. The maintainable fix is restart-safe invalidation in the shared snapshot refresh path, not removing the bounded prompt and discovery limits.
