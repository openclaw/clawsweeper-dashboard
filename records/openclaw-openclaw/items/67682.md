## Summary

Keep open. Current main still exposes MCP credentials only at global `mcp.servers.*` scope, and the corroborating comments describe the same multi-agent attribution failure for Plane/Feishu-style deployments; related open items cover broader env scoping or remote caller-context headers, but none has merged the requested per-agent MCP credential isolation.

## What This Changes

Add per-agent MCP environment or credential overrides so multiple agents can use the same MCP server with agent-specific identity attribution.

## Best Possible Solution

Design a schema-backed per-agent MCP identity overlay for env and headers, with SecretRef/redaction support, isolated effective runtime config and cache keys, docs, and regression tests; coordinate it with the broader per-agent env and MCP scoping issues rather than closing this as duplicate now.

## Reproduction Assessment

Yes. The limitation is source-reproducible on current main: agent entries do not accept an MCP/env override, MCP credentials remain under top-level `mcp.servers.*`, and the runtime merge/fingerprint path only consumes global server config.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is valid, but injecting env into an already-started shared MCP process would be risky; a per-agent effective MCP config with isolated clients/processes and cache keys is the safer design.

## Security Review

Status: needs_attention

Summary: The issue is credential-sensitive even without a patch, because per-agent MCP credentials must be isolated and redacted by design.
