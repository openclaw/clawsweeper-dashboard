## Summary

Keep open. Current main still has only global MCP server credentials and no agent-qualified env/header overlay, so the reported multi-agent identity attribution use case remains unsolved and needs maintainer design because it is credential-sensitive core config/runtime work.

## What This Changes

The issue requests per-agent MCP credential overrides so multiple agents can use the same MCP server with distinct identities for attributed writes.

## Best Possible Solution

Design and implement a schema-backed per-agent MCP identity overlay for env and headers, with SecretRef support, redaction hints, effective-config merge/fingerprinting, isolated clients or processes, docs, and regression tests proving two agents can use one server id with different credentials.

## Reproduction Assessment

Yes. The limitation is source-reproducible on current main: the proposed agent-level MCP shape is not part of the strict agent schema/type contract, and runtime paths consume only top-level `cfg.mcp.servers`.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is valid, but mutating env per call on an already-started shared MCP server is not the best design; the safer path is a per-agent effective MCP config with isolated runtime/cache boundaries.
