## Summary

Keep this issue open. Current main still models MCP credentials as global `mcp.servers` entries, while the agent schema, typed agent config, MCP merge path, and session runtime cache have no agent-qualified env/header overlay for same-server identity attribution.

## What This Changes

Request per-agent MCP server credential overrides so different agents can write to the same MCP-backed service under their own identities.

## Best Possible Solution

Add a schema-backed per-agent MCP identity overlay for env and headers, support redacted secret references, merge it into the effective MCP config before embedded Pi and CLI backend MCP creation, include it in runtime/cache fingerprints, and add regression tests proving two agents can use one server id with different credentials.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: the proposed `agents.list[].mcp.overrides` shape is not accepted by the strict agent schema, and the runtime merge/cache paths only consume top-level `cfg.mcp.servers`.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is valid, but mutating env per call on an already-started shared MCP server would be unsafe; the maintainable path is an effective per-agent config overlay with isolated clients/processes or cache keys.
