## Summary

The datetime naming goal remains useful, but this branch is not merge-ready: its first post-upgrade cleanup intentionally deletes the four numbered rollback files created by released versions, and timestamp collisions can silently drop another backup. Current main and v2026.6.5 still use the five-deep numbered ring, so this is neither implemented nor safely superseded; the linked issue should remain open with this PR until the transition is lossless and real behavior proof is added.

## What This Changes

The PR replaces numbered configuration backup rotation with UTC datetime-stamped filenames and updates cleanup, permission hardening, and regression tests for the new scheme.

## Best Possible Solution

Keep the five-backup policy, allocate collision-safe datetime names, and add one explicit upgrade path that converts or temporarily retains numbered backups until the datetime history reaches full capacity.

## Reproduction Assessment

Yes. Seed `.bak` plus `.bak.1`–`.bak.4` as a released installation would, then run the proposed maintenance once; source and the added test expectations show all numbered history files are deleted after only one datetime entry is created.

## Solution Assessment

No. Datetime filenames are maintainable, but the branch does not implement the linked issue's migration and collision requirements; a lossless single-owner transition is the safer solution.
