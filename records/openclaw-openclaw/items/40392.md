## Summary

Keep this PR open. Current main still uses the numbered backup ring, but this PR is not mergeable as-is because it would delete existing numbered recovery backups during migration and can overwrite a timestamped backup on same-millisecond rotations; the linked same-author issue and parallel PR make this a maintainer consolidation/author-follow-up case rather than a cleanup close.

## What This Changes

The PR changes config backup rotation and its colocated tests to use UTC millisecond timestamped `.bak.<timestamp>` files, directory-scanned chmod/cleanup, and bounded datetime retention instead of the numbered `.bak.1`-`.bak.4` ring.

## Best Possible Solution

Land one consolidated datetime-backup implementation that preserves or migrates existing numbered backups until equivalent timestamped retention exists, uses collision-safe destination names, keeps bounded retention and owner-only permissions, updates focused tests, and adds the required user-facing changelog entry.

## Reproduction Assessment

Yes. On the PR head, create existing `openclaw.json.bak.1` through `.bak.4` files and run `cleanOrphanBackups` or `maintainConfigBackups`; the non-datetime branch unlinks them. Calling `rotateConfigBackups` twice with the same `Date` also targets the same timestamped destination.

## Solution Assessment

No. The datetime naming direction matches the linked feature request, but this patch is not the best mergeable form until it preserves or migrates legacy numbered backups and chooses unique collision-safe destinations instead of documenting silent overwrite behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve legacy backups during timestamp migration:** `src/config/backup-rotation.ts:147`
  - body: On upgrade, existing `openclaw.json.bak.1` through `.bak.4` entries take this non-datetime branch and are unlinked on the first maintenance cycle. Since only the current `.bak` is renamed to one timestamped file before the new `.bak` is copied, users can immediately lose up to four rollback snapshots; keep or migrate the numeric ring until equivalent datetime retention exists.
  - confidence: 0.9
- **[P2] Avoid clobbering backups on timestamp collisions:** `src/config/backup-rotation.ts:53`
  - body: `rotateConfigBackups` uses the timestamp as the whole destination name. Two writes with the same millisecond target the same file, so POSIX rename overwrites the previous backup and Windows can fail into the best-effort catch; choose a unique fallback suffix before renaming.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/config/backup-rotation.ts:46-49`
  - body: This changes user-visible config backup filenames and recovery behavior, but the PR only updates source and tests. OpenClaw policy requires a `CHANGELOG.md` entry for user-facing config changes before merge.
  - confidence: 0.86
