## Summary

Keep this PR open for human review: current main still uses the numbered backup ring, the linked same-author feature request remains open, and there is a competing open implementation for the same datetime-backup behavior. The patch still has merge-blocking data-retention and collision issues, and the contributor has not provided real behavior proof.

## What This Changes

The PR changes config backup rotation and tests from numbered `.bak.1`-`.bak.4` slots to UTC millisecond timestamped `.bak.<YYYYMMDD-HHmmssSSS>` files with directory-scanned permission hardening and cleanup.

## Best Possible Solution

Land one consolidated datetime-backup implementation that preserves or migrates existing numbered backups, uses collision-safe destination names, keeps bounded retention and owner-only permissions, and updates the focused tests together.

## Reproduction Assessment

Yes for the PR defects: source inspection gives a high-confidence path by creating existing `openclaw.json.bak.1` through `.bak.4` files and running the PR cleanup path, or by rotating twice with the same `Date` so both rotations target the same timestamped destination. The underlying request is a feature change, not a broken current-main bug.

## Solution Assessment

No. The datetime naming direction matches the linked feature request, but this patch is not the best mergeable solution until it preserves or migrates legacy numbered backups, uses collision-safe unique destinations, and maintainers choose the canonical scope among the related open backup items.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve legacy backups during timestamp migration:** `src/config/backup-rotation.ts:146-149`
  - body: On upgrade, existing `openclaw.json.bak.1` through `.bak.4` entries take the non-datetime cleanup path and are unlinked even though only one timestamped backup can exist at that point. Keep or migrate the numeric ring until equivalent datetime retention exists so users do not lose rollback snapshots.
  - confidence: 0.9
- **[P2] Avoid clobbering backups on timestamp collisions:** `src/config/backup-rotation.ts:53`
  - body: `rotateConfigBackups` uses the timestamp as the whole destination name. Two writes in the same millisecond target the same file, so POSIX rename overwrites the previous backup and Windows can fall into the best-effort catch; choose a unique fallback suffix before renaming.
  - confidence: 0.84
