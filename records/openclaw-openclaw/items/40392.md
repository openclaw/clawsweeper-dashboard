## Summary

Keep this PR open, but do not merge it as-is. Current main and the latest release still use the numbered config backup ring, so the requested datetime behavior is not already implemented; however, this PR introduces migration-time backup loss and same-timestamp clobbering, and it is paired with the same-author issue plus a parallel open implementation PR that maintainers should consolidate.

## What This Changes

The PR branch changes `src/config/backup-rotation.ts` and its colocated tests to replace numbered `.bak.N` config backup rotation with UTC millisecond timestamped backup names, directory-scanned chmod/cleanup, and bounded datetime retention.

## Best Possible Solution

Keep one datetime-backup implementation path open, preserve or migrate existing numbered recovery slots until equivalent timestamped retention exists, choose collision-safe timestamp destinations, keep bounded retention and owner-only permissions, and validate the final behavior in `src/config/config.backup-rotation.test.ts` before landing.

## Reproduction Assessment

Yes. On the PR head, a config directory containing existing `openclaw.json.bak.1` through `.bak.4` will hit the non-datetime branch in `cleanOrphanBackups` and unlink those files on the first maintenance cycle; calling `rotateConfigBackups` twice with the same `now` also targets the same timestamped destination.

## Solution Assessment

No. The datetime naming direction matches the linked feature request, but this patch is not the best mergeable form until it preserves or migrates legacy numbered backups and uses collision-safe destinations instead of documenting silent overwrite behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve legacy backups during the timestamp migration:** `src/config/backup-rotation.ts:147`
  - body: On upgrade, existing `openclaw.json.bak.1` through `.bak.4` entries take this non-datetime branch and are unlinked on the first write. Since only the current `.bak` is renamed to one timestamped file before the new `.bak` is copied, users can immediately lose up to four rollback snapshots; keep or migrate the numeric ring until equivalent datetime retention exists.
  - confidence: 0.9
- **[P2] Avoid clobbering backups on timestamp collisions:** `src/config/backup-rotation.ts:53`
  - body: `rotateConfigBackups` still uses the timestamp as the whole destination name. Two writes with the same millisecond target the same file, so POSIX rename overwrites the previous backup and Windows can fail into the best-effort catch; choose a unique fallback suffix before renaming.
  - confidence: 0.82
