## Summary

Keep open: current main still uses numbered config backup rotation, and this PR is a useful but not merge-ready implementation because it deletes legacy recovery backups, can clobber same-millisecond timestamp backups, lacks the required changelog entry, and currently conflicts with main.

## What This Changes

The PR changes config backup rotation and tests from numbered `.bak.1`-`.bak.4` slots to UTC millisecond timestamped `.bak.<YYYYMMDD-HHmmssSSS>` files with directory-scanned hardening and cleanup.

## Best Possible Solution

Land one consolidated datetime-backup implementation that preserves or migrates existing numbered backups until equivalent timestamped retention exists, chooses collision-safe destination names, keeps bounded retention and owner-only permissions, updates focused tests, and adds the required user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection of the PR diff gives a high-confidence path: create existing `openclaw.json.bak.1` through `.bak.4` files and run cleanup or maintenance to hit the unlink branch, and call rotation twice with the same `Date` to target the same timestamped destination.

## Solution Assessment

No. The datetime naming direction matches the linked feature request, but this patch is not the best mergeable form until it preserves or migrates legacy numbered backups, uses unique collision-safe destinations, rebases against current main, and records the user-facing behavior change in the changelog.

  - body: On upgrade, existing `openclaw.json.bak.1` through `.bak.4` entries take the non-datetime cleanup path and are unlinked even though only one timestamped backup can exist at that point.
