## Summary

Keep open: this is a plausible implementation for the timestamp-backup feature, but the current branch still has blocking backup-retention and collision defects, lacks required real behavior proof, conflicts with main, and overlaps another open implementation path.

## What This Changes

The PR changes config backup rotation from numbered `.bak.N` files to UTC millisecond timestamped `.bak.<YYYYMMDD-HHmmssSSS>` files and updates cleanup, permission hardening, and focused backup tests.

## Best Possible Solution

Land one canonical timestamp-backup implementation that preserves or migrates numbered backups, uses collision-safe names, keeps bounded owner-only backups, and includes focused tests plus real config-write proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the migration cleanup unlinks legacy numbered backups, and two rotations with the same timestamp target the same destination. The underlying request is a feature change, not a broken current-main bug.

## Solution Assessment

No. The datetime naming direction matches the linked feature request, but this patch is not the best mergeable solution until it preserves or migrates legacy numbered backups, uses collision-safe unique destinations, resolves conflicts, and clears the real behavior proof gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve legacy backup snapshots during migration:** `src/config/backup-rotation.ts:146-147`
  - body: On upgrade, existing `openclaw.json.bak.1` through `.bak.4` entries take the non-datetime cleanup path and are unlinked even though only one timestamped backup can exist at that point. Keep or migrate the numeric ring until equivalent datetime retention exists so users do not lose rollback snapshots.
  - confidence: 0.9
- **[P2] Avoid clobbering backups on timestamp collisions:** `src/config/backup-rotation.ts:53`
  - body: `rotateConfigBackups` uses the timestamp as the whole destination name. Two writes in the same millisecond target the same file, so POSIX rename overwrites the previous backup and Windows can fall into the best-effort catch; choose a unique fallback suffix before renaming.
  - confidence: 0.84
