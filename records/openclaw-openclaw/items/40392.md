## Summary

Keep open: the datetime backup idea is still useful, but this branch deletes shipped numeric backup slots during upgrade, still has a timestamp-collision backup-loss path, and has no after-fix real behavior proof.

## What This Changes

The branch replaces the five-slot numeric config backup ring with UTC millisecond timestamped backup siblings and updates cleanup, permission hardening, and focused tests.

## Best Possible Solution

Land one backup-format migration that preserves or converts legacy numeric backups, bounds datetime retention, handles collisions deterministically, keeps owner-only hardening, and proves fresh-install plus upgrade behavior; otherwise fold the work into https://github.com/openclaw/openclaw/issues/20460.

Full review comments:

- **[P1] Preserve legacy backup slots during migration:** `src/config/backup-rotation.ts:143-147`
  - body: On upgrade, existing `.bak.1` through `.bak.4` files are shipped recovery points, but this cleanup now treats every non-datetime suffix as an orphan and deletes it. After the first post-upgrade config write users would have only the new primary `.bak` plus one timestamped backup, so migrate or retain legacy slots until the datetime set safely reaches capacity.
  - confidence: 0.95
- **[P2] Make timestamp rotation collision-safe:** `src/config/backup-rotation.ts:53-56`
  - body: The new rotation destination is only the millisecond timestamp, so two config writes with the same `now` value target the same file. `rename` can replace the previous backup on POSIX, and the caught failure path can drop the rotation on Windows, so add deterministic collision handling instead of documenting the loss.
  - confidence: 0.87
