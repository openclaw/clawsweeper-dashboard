## Summary

Keep this PR open. Current main and v2026.4.27 still use the numbered config backup ring, so the request is not implemented elsewhere, but the PR branch still has a blocking migration-time data-retention bug and an unresolved collision-safe destination question that should be settled with #39923 and #39982 before merge.

## What This Changes

The PR branch changes `src/config/backup-rotation.ts` and its colocated tests to replace numbered `.bak.N` config backup rotation with UTC millisecond timestamped backup names, directory-scanned chmod/cleanup, and bounded datetime retention.

## Best Possible Solution

Keep one datetime-backup implementation path open and make it preserve or migrate existing numbered recovery slots until equivalent timestamped retention exists, choose collision-safe timestamp destinations, keep bounded retention and owner-only permissions, and validate the behavior in `src/config/config.backup-rotation.test.ts` before landing.

## Reproduction Assessment

Yes. On the PR head, a config directory containing existing `openclaw.json.bak.1` through `.bak.4` will hit the non-datetime branch in `cleanOrphanBackups` on the first maintenance cycle and unlink those files, while only the primary `.bak` is converted to a timestamped file.

## Solution Assessment

No. The datetime naming direction matches #39923, but this implementation is not the best mergeable form until it preserves or migrates legacy numbered backups and uses collision-safe destinations instead of documenting silent overwrite behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve legacy backups during the timestamp migration:** `src/config/backup-rotation.ts:147`
  - body: On upgrade, existing `openclaw.json.bak.1` through `.bak.4` entries take this non-datetime branch and are unlinked on the first write. Since only the current `.bak` is renamed to one timestamped file before the new `.bak` is copied, users can immediately lose up to four rollback snapshots; keep or migrate the numeric ring until equivalent datetime retention exists.
  - confidence: 0.9
- **[P2] Avoid clobbering backups on timestamp collisions:** `src/config/backup-rotation.ts:53`
  - body: `rotateConfigBackups` still uses the timestamp as the whole destination name. Two writes with the same millisecond target the same file, so POSIX rename overwrites the previous backup and Windows can fail into the best-effort catch; choose a unique fallback suffix before renaming.
  - confidence: 0.82
