## Summary

Keep open: current main and `v2026.5.7` still use the numbered backup ring, while this PR is an active but currently incorrect implementation of the timestamp-backup feature and overlaps with https://github.com/openclaw/openclaw/pull/39982.

## What This Changes

This PR replaces numbered config backup rotation with UTC millisecond timestamped backup files and updates cleanup, permission hardening, and focused backup rotation tests.

## Best Possible Solution

Land one canonical timestamp-backup implementation that preserves or converts numbered backups, allocates collision-safe names, keeps bounded owner-only retention, and includes focused tests plus real config-write proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the PR head unlinks legacy numbered backups and two rotations with the same timestamp target the same destination. The underlying request is a feature change, not a broken current-main bug.

## Solution Assessment

No. The backup-rotation module is the right boundary, but this branch is not mergeable until it preserves or migrates legacy backups, uses collision-safe destinations, clears the proof gate, and is chosen over the parallel implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve legacy backup snapshots during migration:** `src/config/backup-rotation.ts:146-147`
  - body: On upgrade, existing `openclaw.json.bak.1` through `.bak.4` entries take the non-datetime cleanup path and are unlinked even though only one timestamped backup can exist at that point. Keep or migrate the numeric ring until equivalent datetime retention exists so users do not lose rollback snapshots.
  - confidence: 0.9
- **[P2] Avoid clobbering backups on timestamp collisions:** `src/config/backup-rotation.ts:53`
  - body: `rotateConfigBackups` uses the timestamp as the whole destination name. Two writes in the same millisecond target the same file, so POSIX rename overwrites the previous backup and Windows can fall into the best-effort catch; choose a unique fallback suffix before renaming.
  - confidence: 0.84
