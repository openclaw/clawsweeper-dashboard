## Summary

Keep open: current main still uses numbered config backup rotation, and this PR is a plausible feature implementation, but it is not mergeable as-is because it deletes legacy numbered backups, can overwrite same-millisecond backups, and lacks real behavior proof.

## What This Changes

The branch changes config backup rotation and tests from numbered `.bak.1`-`.bak.4` slots to UTC millisecond timestamped `.bak.<YYYYMMDD-HHmmssSSS>` files with directory-scanned permission hardening and pruning.

## Best Possible Solution

Land one consolidated datetime-backup implementation that preserves or migrates legacy numbered backups, uses collision-safe unique names, keeps bounded retention and owner-only permissions, and includes focused tests plus real config-write proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: existing `.bak.1` through `.bak.4` files hit the new non-datetime cleanup path, and rotating twice with the same `Date` targets the same timestamped destination. The underlying request is a feature change, not a broken current-main bug.

## Solution Assessment

No. The file boundary is appropriate, but this patch should preserve or migrate legacy rollback history, use collision-safe destinations, and provide real behavior proof; maintainers also need to choose one canonical implementation alongside https://github.com/openclaw/openclaw/pull/39982.

  - body: On upgrade, existing `openclaw.json.bak.1` through `.bak.4` entries take the non-datetime cleanup path and are unlinked even though only one timestamped backup can exist at that point.   - confidence: 0.84
