## Summary

Keep open: current main and the latest release still use numbered config backup rotation, while this PR still has source-backed migration and collision defects plus missing real behavior proof before merge.

## What This Changes

This PR changes config backup rotation from numbered `.bak.N` files to UTC millisecond timestamped `.bak.<YYYYMMDD-HHmmssSSS>` files and updates cleanup, permission hardening, and focused tests.

## Best Possible Solution

Land one canonical config backup timestamp implementation that preserves or converts existing numbered backups, guarantees unique bounded owner-only backups, and covers active config write paths with tests and real proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the cleanup path unlinks legacy numbered backups, and the rotation path uses only the timestamp as the destination name. The underlying timestamp-backup request is a feature change, not a broken current-main bug.

## Solution Assessment

No. `src/config/backup-rotation.ts` is the right boundary, but this patch is not mergeable until it preserves or migrates legacy numbered backups, uses collision-safe destinations, and clears the real behavior proof gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve legacy backup snapshots during migration:** `src/config/backup-rotation.ts:145-147`
  - body: On upgrade, existing `openclaw.json.bak.1` through `.bak.4` entries take the non-datetime cleanup path and are unlinked even though only one timestamped backup can exist at that point. Keep or migrate the numeric ring until equivalent datetime retention exists so users do not lose rollback snapshots.
  - confidence: 0.9
- **[P2] Avoid clobbering backups on timestamp collisions:** `src/config/backup-rotation.ts:53`
  - body: `rotateConfigBackups` uses the timestamp as the whole destination name. Two writes in the same millisecond target the same file, so POSIX rename overwrites the previous backup and Windows can fall into the best-effort catch; choose a unique fallback suffix before renaming.
  - confidence: 0.84
