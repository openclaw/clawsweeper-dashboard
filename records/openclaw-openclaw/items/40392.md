## Summary

Keep open. The PR addresses a useful recovery-ergonomics request, but it is not merge-ready because the upgrade path deletes existing numbered config backups, same-millisecond rotations can still lose a backup, and the external PR has no real after-fix behavior proof.

## What This Changes

This PR changes config backup rotation from adjacent numeric `.bak.N` slots to timestamped `.bak.YYYYMMDD-HHmmssSSS` siblings and updates the focused backup-rotation tests.

## Best Possible Solution

Keep the timestamped naming only with an upgrade-safe plan that preserves or migrates legacy numbered backups, adds collision-safe destination selection, and includes real after-fix proof of the backup files produced during an upgrade write.

## Reproduction Assessment

Yes for the review findings from source: current main preserves numbered backups, while the PR head deletes non-datetime suffixes and can overwrite same-millisecond timestamp targets. I did not run tests because this review is read-only and tests would create artifacts.

## Solution Assessment

No. Timestamped names are a plausible direction, but the best fix needs upgrade-safe legacy retention or migration plus collision handling before this can be considered the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve legacy backups during timestamp migration:** `src/config/backup-rotation.ts:146-147`
  - body: Current releases treat `.bak.1` through `.bak.4` as managed recovery points, but this branch classifies every non-datetime suffix as an orphan and unlinks it. On the first config write after upgrade, users with a full numeric ring would collapse from five historical backups to only `.bak` plus one timestamped file, losing rollback points the linked request explicitly said to preserve or migrate.
  - confidence: 0.94
- **[P2] Add a real collision fallback for timestamped slots:** `src/config/backup-rotation.ts:53-55`
  - body: The destination name is only the millisecond timestamp. If two rotations receive the same millisecond, POSIX `rename` replaces the previous timestamped backup and Windows errors into the best-effort catch, so one backup is silently lost. Add a deterministic collision suffix or retry path instead of only documenting the overwrite case.
  - confidence: 0.84
