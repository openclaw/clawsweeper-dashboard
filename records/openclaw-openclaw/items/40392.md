## Summary

Keep open: the branch addresses a real config-backup feature request, but current main and the latest release still use numbered backups, and this PR has blocking data-integrity issues plus missing real behavior proof before merge.

## What This Changes

The PR changes config backup rotation from numbered `.bak.N` slots to UTC millisecond timestamped `.bak.<YYYYMMDD-HHmmssSSS>` files and updates cleanup, permission hardening, and focused tests.

## Best Possible Solution

Land one canonical timestamp-backup implementation that preserves or migrates existing numbered backups, uses collision-safe destination names, keeps bounded retention and owner-only permissions, updates focused tests, and includes real config-write proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the migration cleanup unlinks legacy numbered backups, and two rotations with the same `Date` target the same timestamped destination. The underlying request is a feature change, not a broken current-main bug.

## Solution Assessment

No. The datetime naming direction matches the linked feature request, but this patch is not the best mergeable solution until it preserves or migrates legacy numbered backups, uses collision-safe unique destinations, and maintainers choose the canonical scope among the related open backup items.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve legacy backups during timestamp migration:** `src/config/backup-rotation.ts:146-147`
  - body: On upgrade, existing `openclaw.json.bak.1` through `.bak.4` entries take the non-datetime cleanup path and are unlinked even though only one timestamped backup can exist at that point. Keep or migrate the numeric ring until equivalent datetime retention exists so users do not lose rollback snapshots.
  - confidence: 0.9
- **[P2] Avoid clobbering backups on timestamp collisions:** `src/config/backup-rotation.ts:53`
  - body: `rotateConfigBackups` uses the timestamp as the whole destination name. Two writes in the same millisecond target the same file, so POSIX rename overwrites the previous backup and Windows can fall into the best-effort catch; choose a unique fallback suffix before renaming.
  - confidence: 0.84
