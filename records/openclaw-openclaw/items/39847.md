## Summary

Keep open: current main still lacks source-level outbound metadata stripping, and the issue is paired with an open closing PR, so it should remain the canonical report until that fix lands or is replaced.

## What This Changes

Report that echoed OpenClaw inbound metadata can pass through outbound reply normalization and be delivered visibly to Discord or other channel users.

## Best Possible Solution

Land one canonical fix, likely https://github.com/openclaw/openclaw/pull/50520 or a replacement, that strips leading inbound metadata at the shared outbound payload plan boundary with regression coverage for metadata plus directives/media.

## Reproduction Assessment

Yes. Source inspection shows `buildInboundUserContextPrefix` emits Conversation/Sender JSON blocks, while `createOutboundPayloadPlanEntry` parses `payload.text` directly and neither the parser nor channel delivery calls the existing metadata stripper before visible sends.

## Solution Assessment

Yes. Sanitizing once at the shared outbound payload plan boundary before directive parsing is the narrow maintainable fix; per-channel or compiled-bundle patches would miss other delivery paths.
