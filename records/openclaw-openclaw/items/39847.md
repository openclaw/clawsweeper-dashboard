## Summary

Keep open: current main and the latest release still parse outbound reply text before inbound metadata stripping, and the open closing PR has not landed yet.

## What This Changes

Strip echoed OpenClaw inbound metadata from outbound delivery before reply directive parsing so it cannot appear in Discord or other user-visible channel replies.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/50520 or a clean replacement that strips leading inbound metadata at the shared outbound payload planner before directive parsing, with focused regression tests and redacted real channel proof.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to pass `createOutboundPayloadPlan` or `normalizeReplyPayloadsForDelivery` a payload whose text starts with generated inbound metadata; current main parses that raw text before inbound metadata stripping.

## Solution Assessment

Yes. The shared outbound payload planner before directive parsing is the narrow maintainable fix point; compiled-bundle or channel-specific patches would leave other delivery paths uncovered.
