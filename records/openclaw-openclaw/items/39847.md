## Label Justifications

- `P1`: This is an active channel-delivery security bug that can expose internal metadata to real users on shared chat surfaces.
- `impact:security`: The issue concerns AI-facing internal conversation and sender metadata becoming visible in outbound messages.

## Summary

Keep open: current `main` and the latest release still parse outbound reply text before any inbound-metadata stripping, and https://github.com/openclaw/openclaw/pull/50520 remains the open closing PR for this fix.

## What This Changes

The issue asks OpenClaw to strip echoed inbound metadata from outbound channel delivery before Discord or other chat surfaces can display it.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/50520 or a clean replacement that strips leading inbound metadata at the shared outbound payload planner, preserves directive and media parsing, and includes outbound regression coverage plus real channel proof.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to pass `createOutboundPayloadPlan` or `normalizeReplyPayloadsForDelivery` a text payload beginning with generated inbound metadata; current `main` parses and plans that raw text before any inbound-metadata stripping.

## Solution Assessment

Yes. The shared outbound payload planner before directive parsing is the narrow maintainable fix point; compiled-bundle or channel-specific patches would leave other delivery paths uncovered.
