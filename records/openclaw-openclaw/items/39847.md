Keep open: current main and v2026.6.6 still parse raw outbound payload text before inbound-metadata stripping, and the open linked repair PR at https://github.com/openclaw/openclaw/pull/50520 has not merged.

## What This Changes

The issue reports that echoed OpenClaw inbound metadata blocks can pass through shared outbound payload normalization and become visible channel messages.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/50520 or an equivalent core outbound-plan fix that strips leading inbound metadata before directive parsing, then close with merged commit and release provenance.

## Reproduction Assessment

Yes, at source level: current main still feeds raw `payload.text` into `parseReplyDirectives`, and downstream outbound sanitization does not remove inbound metadata blocks. I did not run a live Discord repro because this review is read-only.

## Solution Assessment

Yes: stripping leading inbound metadata at the shared outbound payload-plan boundary before directive parsing is the narrow maintainable fix because delivery projections flow through that plan. A Discord-only compiled-bundle patch would miss sibling paths and be overwritten by updates.
