Keep open: current main and the latest shipped release still parse outbound reply text before any inbound-metadata stripping, and the open closing PR remains unmerged. The issue has a clear source-level reproduction and should stay open until https://github.com/openclaw/openclaw/pull/50520 or an equivalent replacement lands.

## What This Changes

The issue requests stripping echoed OpenClaw inbound metadata from outbound reply payloads before channel delivery so internal context is not visible in Discord or other chat surfaces.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/50520 or a clean replacement that strips leading inbound metadata at the shared outbound payload planner, preserves directive/media parsing, and covers generated metadata labels with tests and real channel proof.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to pass normalizeReplyPayloadsForDelivery a payload beginning with generated inbound metadata; current main sends that text into parseReplyDirectives before any inbound metadata stripping.

## Solution Assessment

Yes. The shared outbound payload planner before directive parsing is the narrow maintainable fix point; channel-specific compiled-bundle patches would leave other delivery callers uncovered.
