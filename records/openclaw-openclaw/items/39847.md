## Summary

Keep open: current `main` and release `v2026.5.20` still parse outbound reply text before inbound metadata stripping, while https://github.com/openclaw/openclaw/pull/50520 remains the open but conflicting closing PR for this fix.

## What This Changes

This issue requests stripping echoed inbound metadata from outbound channel replies before delivery.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/50520 or a clean replacement that strips leading inbound metadata at the shared outbound payload planner before directive parsing, with focused regression tests and redacted real channel proof.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to pass `createOutboundPayloadPlan` or `normalizeReplyPayloadsForDelivery` a text payload beginning with generated inbound metadata; current `main` parses that raw text before inbound metadata stripping.

## Solution Assessment

Yes. The shared outbound payload planner before directive parsing is the narrow maintainable fix point; compiled-bundle or channel-specific patches would leave other delivery paths uncovered.
