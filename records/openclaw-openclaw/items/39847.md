## Summary

Keep open: current main and the latest release still parse outbound reply text without calling the inbound metadata stripper, while the canonical closing PR remains open and conflicting.

## What This Changes

The issue reports that outbound channel delivery can expose model-echoed inbound metadata because shared reply payload normalization does not strip inbound metadata before parsing and delivery.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/50520 or a clean replacement that strips inbound metadata at the shared outbound payload planner before directive parsing, with focused regression coverage and real channel proof.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to pass `createOutboundPayloadPlan` or `normalizeReplyPayloadsForDelivery` a text payload beginning with generated inbound metadata; current main parses that text without calling the inbound metadata stripper.

## Solution Assessment

Yes. The shared outbound payload planner before directive parsing is the narrow maintainable fix point because channel-specific or compiled-bundle patches leave other delivery paths uncovered.
