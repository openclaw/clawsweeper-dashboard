## Summary

Keep open: current main still sends outbound reply text into directive parsing before any inbound metadata stripper runs, and the linked closing PR is still open rather than merged.

## What This Changes

Strip echoed OpenClaw inbound metadata from outbound reply payloads before channel delivery, especially Discord thread messages.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/50520 or a replacement that strips leading inbound metadata at the shared outbound payload planner, preserves directive/media parsing, and covers generated metadata labels.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to pass `normalizeReplyPayloadsForDelivery` a payload beginning with generated inbound metadata; current main sends that text into `parseReplyDirectives` before any inbound-metadata stripping.

## Solution Assessment

Yes. The shared outbound payload planner is the narrow maintainable fix point; per-channel compiled-bundle patches or later channel sanitizers would leave other planner callers uncovered.
