## Best Possible Solution

Land or replace the linked fix with a current-main patch that strips leading generated inbound metadata before the first outbound directive parse, preserves current planner behavior, and includes focused outbound/sanitizer/channel proof.

Yes, source-level. A reply payload containing leading Conversation/Sender metadata currently flows through createOutboundPayloadPlanEntry and parseReplyDirectives without inbound metadata stripping before channel delivery projection.

## Solution Assessment

Yes in boundary, with one refinement. The shared outbound payload planner before the first directive parse is the narrow maintainable fix, but it should use the current leading-only sanitizer shape rather than a broad downstream Discord-only patch.
