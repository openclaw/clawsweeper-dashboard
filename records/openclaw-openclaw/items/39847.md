## Summary

Keep open. Current main at e47a7448e90ff75f2d21082a1c33a18695b49162 still parses raw outbound payload text before directive handling and does not apply `stripInboundMetadata()` in the shared delivery planner, while related implementation PRs #39894, #50520, and #51653 are still unmerged closing candidates.

## What This Changes

Request to strip OpenClaw-injected inbound metadata from outbound reply payloads before directive parsing so echoed Conversation/Sender blocks are not delivered to Discord or other channels.

## Best Possible Solution

Keep this issue open as the canonical bug, review and land one focused source-level fix in `src/infra/outbound/payloads.ts` that strips inbound metadata before directive parsing in the shared outbound plan, add regression coverage in `src/infra/outbound/payloads.test.ts`, then close or supersede the duplicate implementation PRs with a maintainer rationale.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to pass a reply payload whose `text` begins with `Conversation info (untrusted metadata):` / `Sender (untrusted metadata):` fenced JSON blocks through `normalizeReplyPayloadsForDelivery()` or `createOutboundPayloadPlan()`; current main parses and projects that raw text without calling `stripInboundMetadata()`.

## Solution Assessment

Yes. The issue’s suggested shared-planner fix is the narrowest maintainable direction because it covers Discord and other channel delivery callers before directive parsing, instead of patching individual compiled bundles or adapters.
