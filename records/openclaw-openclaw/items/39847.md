## What This Changes

Request to strip echoed inbound metadata from outbound reply payloads before directive parsing and channel delivery so Discord users do not see internal context blocks.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/50520 or an equivalent core outbound-plan fix that strips leading inbound metadata before `parseReplyDirectives`, then close with merged commit and release provenance.

## Reproduction Assessment

Yes, at source level: current main still feeds raw `payload.text` into `parseReplyDirectives`, and the downstream outbound sanitizer does not remove inbound metadata blocks. I did not run a live Discord repro because this review is read-only.

## Solution Assessment

Yes: stripping leading inbound metadata at the shared outbound payload-plan boundary before directive parsing is the narrow maintainable fix because delivery projections flow through that plan. A Discord-only compiled-bundle patch would miss sibling delivery paths and be overwritten by updates.
