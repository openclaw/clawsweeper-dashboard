## Summary

Keep open: current main and the latest release still parse outbound payload text without stripping echoed inbound metadata, and the open linked pull request is the active fix candidate that should be reviewed or refreshed before this issue closes.

## What This Changes

The issue reports that echoed inbound metadata blocks can pass through outbound delivery to Discord because the shared delivery normalization path does not strip them before parsing reply directives.

## Best Possible Solution

Refresh and land https://github.com/openclaw/openclaw/pull/50520, or replace it with an equivalent narrow fix that strips only leading inbound metadata at the shared outbound payload-plan boundary before directive parsing.

## Reproduction Assessment

Yes for a source-level reproduction: current main passes payload text directly to parseReplyDirectives in the shared outbound payload plan, so a leading metadata block remains part of the normalized delivery text. I did not run a test because this review is read-only.

## Solution Assessment

Yes: a leading-only sanitizer call at the shared outbound payload-plan boundary is the narrowest maintainable fix because it removes echoed prompt-prefix metadata before directive parsing while preserving metadata-looking examples later in normal replies.
