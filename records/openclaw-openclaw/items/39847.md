## Summary

Keep open: current main still does not strip inbound metadata in the shared outbound payload normalization path, and open PRs explicitly reference this issue with closing syntax, so the issue should remain open until one canonical fix lands or is replaced.

## What This Changes

The issue asks OpenClaw to prevent echoed inbound metadata blocks from leaking to Discord or other channel delivery by stripping them during outbound reply payload normalization.

## Best Possible Solution

Land one canonical fix that sanitizes text with `stripInboundMetadata` in the shared outbound payload planning path before directive parsing, with regression coverage proving metadata removal while preserving media and reply directives.

## Reproduction Assessment

Yes, source-reproducible. A payload containing `Conversation info (untrusted metadata):` is parsed by `parseReplyDirectives()` in `createOutboundPayloadPlanEntry` and projected into delivery without the existing metadata stripper being called.

## Solution Assessment

Yes. The shared outbound normalizer is the narrowest maintainable fix point because it covers channel delivery projections before adapter-specific sanitizers and avoids per-channel workaround patches.
