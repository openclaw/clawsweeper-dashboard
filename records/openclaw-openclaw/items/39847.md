## Summary

Keep open: current main and the latest release still route outbound payload text through reply-directive parsing without calling the inbound-metadata sanitizer, and an open closing PR already owns the proposed source-level fix.

## What This Changes

The issue reports that echoed inbound metadata can leak into user-visible outbound channel messages because the outbound delivery normalizer does not strip internal metadata before delivery.

## Best Possible Solution

Refresh and review https://github.com/openclaw/openclaw/pull/50520, or land a narrow replacement if that branch cannot be updated, so the shared outbound payload planner strips leading inbound metadata before directive parsing with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main and `v2026.6.8` still parse raw outbound payload text before any inbound metadata stripping, while the exact sanitizer helpers live elsewhere and are not wired into the outbound planner.

## Solution Assessment

Yes, the shared outbound payload plan is the right layer because it feeds direct delivery and transport projections before channel adapters. The linked PR’s `stripLeadingInboundMetadata` direction matches that boundary, but it needs refresh and review before landing.
