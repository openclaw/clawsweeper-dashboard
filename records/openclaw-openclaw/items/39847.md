## Summary

Keep open: current main still plans outbound replies from raw `payload.text` without calling the existing inbound-metadata stripper, and an open closing PR is already the canonical fix path.

## What This Changes

Request source-level stripping of echoed inbound metadata before outbound channel delivery so users do not see Conversation/Sender JSON blocks.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/50520 or a replacement that strips leading inbound metadata at the shared outbound payload plan boundary with regression coverage for directives/media and generated metadata labels.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to pass a reply payload beginning with generated Conversation/Sender metadata blocks through `normalizeReplyPayloadsForDelivery`; current main sends the raw text into directive parsing unstripped.

## Solution Assessment

Yes. Sanitizing once at `createOutboundPayloadPlanEntry` before directive parsing is the narrow shared fix; per-channel or compiled-bundle patches would miss other delivery paths.
