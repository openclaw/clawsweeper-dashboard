## Summary

Keep open: current main still lacks the MCP scope selector and this PR is the canonical replacement path for the linked feature, but merge should wait on the effective-scope contract for cached device tokens and the current unmergeable head.

## What This Changes

The branch adds a repeatable `openclaw mcp serve --scope` option limited to `operator.read`, `operator.write`, and `operator.approvals`, threads it through the MCP server/bridge, and updates docs, tests, and changelog.

## Best Possible Solution

Land this or an equivalent narrow selector after resolving cached-token semantics and mergeability, then use it as the canonical implementation for #73864 while leaving #74030 superseded.

## Reproduction Assessment

Yes. Source inspection shows current main lacks the CLI/server/bridge scope path, and existing GatewayClient/auth tests show why cached device-token reuse can preserve broader scopes than a later explicit request.

## Solution Assessment

No, not as-is. The allowlist and plumbing are the right narrow direction, but the least-privilege contract should either make explicit scopes effective for reused identities or document and test the required fresh-profile/token-rotation path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Clarify cached-token scope reuse:** `docs/cli/mcp.md:261`
  - body: This presents `--scope` as the way to create least-privilege bridge identities, but existing MCP identities can reconnect with stored device-token scopes because `GatewayClient` substitutes cached token scopes when reusing a stored token. Add code to force explicit scopes to narrow or reissue the token, or document and test that users need fresh profiles or token rotation before relying on this for least privilege.
