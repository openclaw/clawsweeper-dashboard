## Summary

Keep open. The PR addresses a real missing MCP scope-selection surface, but it is not merge-ready because explicit least-privilege scopes can still be bypassed by cached or persisted device-token scope precedence.

## What This Changes

The PR adds a repeatable `openclaw mcp serve --scope` allowlist for read/write/approvals, threads selected scopes through the MCP server and bridge to `GatewayClient`, and updates docs, changelog, and focused tests.

## Best Possible Solution

Update this PR so explicit MCP `--scope` selections govern the effective bridge authorization even with cached/persisted device tokens, preserve the existing default when omitted, and add regression coverage before using it to resolve #73864 and leave #74030 superseded.

## Reproduction Assessment

Yes. Seed or reuse a device token with broader scopes, then construct an MCP/Gateway client with narrower requested scopes; current tests at `src/gateway/client.test.ts:803` and `src/gateway/server.auth.default-token.suite.ts:267` show stored or persisted scopes win.

## Solution Assessment

No. The allowlist and plumbing are the right direction, but the patch is incomplete until the cached/persisted token path cannot make `--scope operator.read` connect with broader effective scopes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor explicit MCP scopes for cached tokens:** `src/mcp/channel-bridge.ts:126`
