## Summary

Keep open: current main still lacks MCP serve scope selection, and this PR is the active implementation candidate, but the patch needs maintainer review because explicit least-privilege scopes can still be bypassed by reused device-token scopes.

## What This Changes

This PR adds repeatable `openclaw mcp serve --scope` support limited to read/write/approvals, threads selected scopes through the MCP server and bridge, and updates docs, changelog, and focused tests.

## Best Possible Solution

Update this PR, or an equivalent narrow implementation, so the documented least-privilege MCP bridge profiles have a clear effective-auth contract, then use it to resolve #73864 and leave #74030 superseded.

## Reproduction Assessment

Yes. Seed or reuse a stored device token with broader scopes, then start or construct the MCP bridge with a narrower explicit scope and no shared token/password; current GatewayClient tests show stored scopes are sent instead of the requested scopes.

## Solution Assessment

No. The allowlist and plumbing are the right shape, but the patch is incomplete until cached-token behavior cannot silently defeat the documented least-privilege scope selection or the contract is explicitly constrained to fresh separate identities.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor explicit MCP scopes with cached tokens:** `src/mcp/channel-bridge.ts:126`
  - body: This forwards `--scope` to `GatewayClient`, but MCP clients use the default device identity and `GatewayClient` reuses stored device-token scopes before requested scopes when no shared token/password is supplied. A previously paired broader identity can run `openclaw mcp serve --scope operator.read` and still connect with broader effective scopes, which undercuts the least-privilege bridge profile this flag documents.
  - confidence: 0.86
