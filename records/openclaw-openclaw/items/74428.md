## Summary

Keep this PR open. Current main still lacks MCP serve scope selection and the PR is the active implementation candidate, but it needs maintainer review for cached device-token effective-scope semantics and is currently dirty against main.

## What This Changes

The branch adds repeatable `openclaw mcp serve --scope` selection limited to `operator.read`, `operator.write`, and `operator.approvals`, threads it through the MCP server/bridge, and updates docs, changelog, and focused tests.

## Best Possible Solution

Resolve the cached-token contract, rebase this or an equivalent narrow implementation, then land it as the canonical fix for #73864 while leaving #74030 superseded.

## Reproduction Assessment

Yes. Source inspection shows current main has no `mcp serve --scope` path and the bridge still passes a fixed scope array; the PR diff adds the missing selector and plumbing.

## Solution Assessment

No, not as-is. The allowlist and plumbing are the narrow direction, but the least-privilege contract must account for cached device-token scope reuse before users can rely on explicit scopes for existing profiles.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Document cached-token scope reuse:** `docs/cli/mcp.md:261`
  - body: This presents `--scope` as sufficient for least-privilege bridge identities, but `GatewayClient` loads the default device identity and reuses stored device-token scopes on reconnect. An existing broad profile can still report broader scopes after `--scope operator.read`; either force explicit scopes to narrow/reissue the token, or document and test the fresh-profile/token-rotation requirement.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff adds a least-privilege scope control, but existing cached device-token reuse can preserve broader effective permissions than the explicit MCP scope request.

Concerns:

- **[medium] Cached tokens can preserve broader scopes:** `src/mcp/channel-bridge.ts:123`
  - body: `OpenClawChannelBridge` forwards explicit scopes to `GatewayClient`, but `GatewayClient` can replace that request with stored device-token scopes when reusing a cached token. That makes the new least-privilege guidance incomplete for existing broad MCP profiles.
