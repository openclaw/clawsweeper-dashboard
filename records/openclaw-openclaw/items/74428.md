## Summary

Keep open: current main still lacks the MCP serve scope selector, and this PR is the focused implementation candidate, but it should not merge until explicit scope behavior is resolved for cached or persisted device tokens.

## What This Changes

The PR adds a repeatable `openclaw mcp serve --scope` allowlist for read/write/approvals, threads it through the MCP server and bridge to `GatewayClient`, and updates docs, changelog, and focused tests.

## Best Possible Solution

Update this PR so explicit MCP scope selections either govern the effective bridge authorization or clearly require a fresh/profile-specific pairing flow, add regression coverage, then use it to resolve #73864 and keep #74030 superseded.

## Reproduction Assessment

Yes. Seed or reuse a stored device token with broader scopes, then connect with narrower requested scopes; current `GatewayClient` and gateway auth tests show persisted token scopes win over the requested list.

## Solution Assessment

No. The allowlist and plumbing are the right shape, but passing scopes only as `GatewayClient` options is not enough while cached device-token scopes can replace the explicit MCP request.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor explicit MCP scopes when reusing device tokens:** `src/mcp/channel-bridge.ts:126`
  - body: This only forwards explicit MCP scopes into `GatewayClient`, but current client auth replaces `opts.scopes` with stored device-token scopes whenever it reuses a cached token. A profile with an existing broader token can run `openclaw mcp serve --scope operator.read` and still connect with broader effective scopes, which defeats the least-privilege bridge profile this flag documents.
