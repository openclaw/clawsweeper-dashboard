## Summary

Keep open. The PR is a useful narrow implementation for the open MCP scope-selection request, but it is not merge-ready because explicit `--scope` values can still be bypassed by the existing cached device-token scope precedence in `GatewayClient`.

## What This Changes

The PR adds a repeatable `openclaw mcp serve --scope` option limited to `operator.read`, `operator.write`, and `operator.approvals`, threads it through the MCP server and bridge, and updates docs, changelog, and focused tests.

## Best Possible Solution

Update this PR so explicit MCP `--scope` selections control the effective Gateway connect scopes even on cached device-token reconnects, preserve the current default read/write/approvals behavior when `--scope` is omitted, add a regression test for the cached-token path, then use the merged PR to resolve #73864 and leave #74030 superseded.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to seed a stored device token with broader scopes and construct `GatewayClient` with narrower `opts.scopes`; the existing Gateway client test at `src/gateway/client.test.ts:803` already proves stored scopes are sent instead.

## Solution Assessment

No. Passing `operatorScopes` into `GatewayClient` is the right direction, but it is not sufficient while cached device-token scopes take precedence over requested scopes; the safer fix must address that cached-token path or explicitly isolate bridge identities.

## Review Findings

- **[P2] Honor explicit MCP scopes for cached tokens:** `src/mcp/channel-bridge.ts:126`
