## Summary

Keep open. This PR is a useful implementation candidate for MCP bridge scope selection, but current main still lacks the feature and the branch needs maintainer review for cached device-token scope semantics plus mergeability before it can safely land.

## What This Changes

The PR adds a repeatable `openclaw mcp serve --scope` allowlist for `operator.read`, `operator.write`, and `operator.approvals`, threads selected scopes through the MCP server/bridge, updates MCP docs/changelog, and adds focused tests.

## Best Possible Solution

Land this or an equivalent narrow implementation after maintainer review, preserving the default read/write/approvals behavior when `--scope` is omitted, making the cached-token contract explicit, and using it as the canonical path for the linked configurable-scope request.

## Reproduction Assessment

Yes. Source inspection of current main shows no `--scope` CLI/server/bridge path, and the existing GatewayClient auth code plus tests show cached device-token scopes can remain broader than a later explicit request.

## Solution Assessment

No, not as-is. The allowlist and plumbing are the right narrow direction, but the least-privilege contract should either make explicit scopes effective for reused identities or document and test the required fresh-profile/token-rotation path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Clarify cached-token scope reuse:** `docs/cli/mcp.md:261`
  - body: This presents `--scope` as the way to create least-privilege bridge identities, but an existing MCP identity can reconnect with stored device-token scopes because `GatewayClient` substitutes cached token scopes when reusing a stored token. Add code to force explicit scopes to narrow or reissue the token, or document and test that users need fresh profiles or token rotation before relying on this for least privilege.
  - confidence: 0.88
