## Summary

Keep open. Current main still lacks configurable MCP serve scopes, and this PR is the active narrow implementation candidate, but it should not merge as-is because the least-privilege guidance does not account for reused device-token effective scopes and the external PR has no after-fix real behavior proof.

## What This Changes

The PR adds a repeatable `openclaw mcp serve --scope` option limited to `operator.read`, `operator.write`, and `operator.approvals`, threads it through the MCP server/bridge, and updates docs, tests, and changelog.

## Best Possible Solution

Land this or an equivalent narrow implementation after it documents or handles reused-token scope semantics, includes real MCP/gateway proof, and then use it as the canonical fix for the linked feature request.

## Reproduction Assessment

Yes. Source inspection on current main shows no `--scope` option, no `operatorScopes` serve option, and a fixed MCP bridge scope list; a live gateway is not needed to verify the missing capability.

## Solution Assessment

No, not as-is. The allowlist and plumbing are the right narrow direction, but the least-privilege contract needs to handle or document reused device-token scopes, and the contributor still needs real behavior proof.

## Review Findings

Overall correctness: patch is incorrect
