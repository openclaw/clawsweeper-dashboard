## Summary

Keep open: current main still lacks the requested MCP serve scope selector, and this PR is the canonical replacement path for #74030 / #73864, but it needs an auth-scope follow-up before merge because an already paired broader bridge identity can still defeat the least-privilege behavior being documented.

## What This Changes

This PR adds repeatable `openclaw mcp serve --scope` parsing limited to read/write/approvals, threads selected scopes through the MCP server and bridge to `GatewayClient`, and updates docs, changelog, and focused tests.

## Best Possible Solution

Land this or an equivalent narrow selector only after the effective-scope contract is explicit, either by making explicit `--scope` narrow reused MCP bridge tokens or by documenting and testing the required fresh-profile/token-rotation path.

## Reproduction Assessment

Yes. Current main can be reproduced by source inspection of the missing CLI/server/bridge scope path, and the PR-specific concern is covered by existing auth tests that show a reconnect with narrower requested scopes keeps prior persisted scopes.

## Solution Assessment

No. The allowlist and plumbing are the right direction, but the patch is incomplete until maintainers decide and cover how explicit MCP scopes interact with existing paired device tokens.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Honor explicit MCP scopes for reused device tokens:** `src/mcp/channel-bridge.ts:126`
  - body: The new flag is documented for least-privilege bridge profiles, but this line only forwards scopes into `GatewayClient`; the existing auth path can return a previously issued broader device token when the requested scopes are a subset of that token. A bridge that was already paired with read/write/approvals can later run `--scope operator.read` and still receive the broader token, so the least-privilege contract needs code or docs/tests for fresh identity/rotation.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The patch introduces a least-privilege auth-scope control, but existing device-token reuse can preserve broader effective permissions than the explicit MCP scope request.

Concerns:

- **[medium] Explicit MCP scopes can be overridden:** `src/mcp/channel-bridge.ts:126`
  - body: Existing paired device tokens are reused when the requested scopes are within the token's stored broader scopes, so an explicit read-only MCP profile can still connect with write/approvals if it reuses a previously paired bridge identity.
  - confidence: 0.87
