## Summary

Keep open: this is still the canonical implementation candidate for MCP serve scope selection, but it needs real MCP/gateway behavior proof and cached-token scope semantics clarified before merge.

## What This Changes

The PR adds a repeatable `openclaw mcp serve --scope` flag limited to `operator.read`, `operator.write`, and `operator.approvals`, threads it through the MCP server/bridge, and updates docs, tests, and changelog.

## Best Possible Solution

Land this or an equivalent narrow implementation after it documents or handles cached device-token scope reuse, includes real MCP/gateway proof, and remains the canonical fix for https://github.com/openclaw/openclaw/issues/73864.

## Reproduction Assessment

Yes. Source inspection on current main shows no `--scope` option, no `operatorScopes` serve option, and a fixed MCP bridge scope list; I did not run a live gateway because the missing capability is visible in source.

## Solution Assessment

No, not as-is. The allowlisted CLI flag is the narrow direction, but the least-privilege docs/contract need to cover existing cached device-token scopes and the contributor still needs real behavior proof.
