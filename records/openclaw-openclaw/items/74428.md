## Summary

Keep open. The requested MCP scope-selection feature is still missing from current main, and this PR is the useful implementation candidate, but the proposed plumbing is not merge-ready because explicit least-privilege scopes can be bypassed by cached broader device-token scopes.

## What This Changes

The PR adds a repeatable `openclaw mcp serve --scope` option limited to `operator.read`, `operator.write`, and `operator.approvals`, threads it through the MCP server and bridge, updates docs and changelog, and adds focused CLI/server tests.

## Best Possible Solution

Update this PR so explicit MCP `--scope` selections control the effective Gateway connect scope even when cached device-token auth is reused, preserve the current default read/write/approvals behavior when omitted, add a regression for the cached-token path, then land it as the canonical fix for #73864 and keep #74030 superseded.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to run the bridge with an explicit narrow scope while `GatewayClient` has a stored broader device token; current `resolveConnectScopes()` and the existing cached-token test show the stored scopes win before caller-provided scopes.

## Solution Assessment

No, not as-is. The allowlisted CLI/API shape is a good narrow direction, but the implementation must also make explicit scopes determine the effective cached-device-token connection or isolate bridge identities before the docs can promise least privilege.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor explicit MCP scopes with cached tokens:** `src/mcp/channel-bridge.ts:123`
  - body: Passing `operatorScopes` into `GatewayClient` is not enough because `GatewayClient` reuses stored device-token scopes before `opts.scopes`. A device with a cached broader token can run `openclaw mcp serve --scope operator.read` and still connect with the broader stored scopes, defeating the least-privilege profile this flag documents. Please make explicit MCP scopes control the effective cached-token connect scopes, or isolate the bridge identity, and add a regression for that path.
  - confidence: 0.87
