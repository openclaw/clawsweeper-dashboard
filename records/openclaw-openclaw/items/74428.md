## Summary

Keep this PR open: it implements the requested MCP scope plumbing, but it still needs maintainer review because explicit `--scope` is framed as least-privilege while cached device-token reuse can preserve broader effective scopes, and the PR is currently reported unmergeable against main.

## What This Changes

The PR adds repeatable `openclaw mcp serve --scope` parsing for the read/write/approvals allowlist, threads selected scopes through the MCP server and bridge, and updates docs, tests, and changelog.

## Best Possible Solution

Land this or an equivalent narrow implementation after deciding cached-token semantics, with docs/tests that make any fresh-profile or token-rotation requirement explicit if reused tokens are not narrowed.

## Reproduction Assessment

Yes. Source inspection of current main shows no `--scope` CLI/server/bridge path and fixed MCP bridge scopes; the cached-token concern is also source/test-backed by `resolveConnectScopes()` and the gateway auth scope-reuse test.

## Solution Assessment

No, not as-is. The allowlist and plumbing are the right narrow direction, but the least-privilege contract should either make explicit scopes effective for reused identities or document and test the required fresh-profile/token-rotation path.

## Security Review

Status: needs_attention

Summary: The diff adds a least-privilege control, but cached device-token reuse can leave effective scopes broader than the explicit MCP scope request.

Concerns:

- **[medium] Explicit MCP scopes can be overridden by cached tokens:** `src/mcp/channel-bridge.ts:123`
  - body: The PR forwards selected scopes into `GatewayClient`, but current client behavior replaces caller-requested scopes with stored device-token scopes when reusing a cached token. An existing broad MCP identity can therefore reconnect with `--scope operator.read` while keeping broader effective permissions unless the code or docs require a fresh identity or token rotation.
