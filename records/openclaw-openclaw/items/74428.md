## Summary

Keep open: current main still lacks MCP serve scope selection, and this PR remains the active implementation candidate for the linked feature request. The branch looks narrow, but the auth/token reuse contract should be confirmed by maintainers before merge.

## What This Changes

This PR adds a repeatable `openclaw mcp serve --scope` selector limited to read/write/approvals, threads it through the MCP server and bridge, and updates docs, changelog, and focused tests.

## Best Possible Solution

Land this or an equivalent narrow selector after maintainer review, with explicit docs or tests for fresh bridge identity/token cache semantics, then resolve #73864 and leave #74030 superseded.

## Reproduction Assessment

Yes. Current main can be reproduced by source inspection: the CLI has no `--scope` option and the bridge always requests read/write/approvals; the cached-token caveat is also covered by an existing gateway auth test.

## Solution Assessment

Unclear. The allowlist and plumbing are the narrow maintainable direction, but maintainers should decide whether explicit MCP scopes are only the requested pairing scope or must also narrow reused device tokens.
