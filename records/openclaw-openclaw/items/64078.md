## Summary

Keep open. Current main still preserves the reported Slack routing split for ordinary top-level room/channel messages under `replyToMode="all"`: reply metadata points at the message timestamp, but the inbound session key can remain the shared channel session. The issue also has an open paired PR with closing syntax, so it should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that Slack channel/group top-level messages with `replyToMode="all"` use a thread-scoped inbound session key aligned with reply delivery to avoid duplicate processing and cross-sender context bleed.

## Best Possible Solution

Review and land #64080, or an equivalent Slack-owned patch, so room/channel routing derives one conversation thread id for both inbound `SessionKey` and reply `MessageThreadId`: parent `thread_ts` for real thread replies, message `ts` for ordinary top-level `replyToMode="all"` turns, and no thread id for modes that intentionally keep channel continuity.

## Reproduction Assessment

Yes. A source-level reproduction is clear from `resolveSlackRoutingContext` with a Slack channel message, `isRoomish: true`, no `thread_ts`, no seeded room thread, and `replyToMode="all"`: the session key stays on the base channel while `MessageThreadId` resolves to the message timestamp; the issue also includes two live reproduction reports.

## Solution Assessment

Yes for the requested direction. Unifying the routed thread id with the reply-delivery thread id is the narrowest maintainable fix, provided it keeps `replyToMode=off`, `first`, `batched`, runtime bindings, seeded app mentions, true thread replies, and existing DM behavior intact.
