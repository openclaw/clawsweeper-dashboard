## Summary

Keep open: the PR targets a valid current-main Ollama stderr bug, but the proposed quiet condition is too broad and would regress an existing alternate loopback provider expectation; it also still needs real behavior proof before merge.

## What This Changes

The PR changes Ollama discovery to silence unreachable-host warnings whenever there is no explicit `models.providers.ollama` config and adds a mocked provider-discovery regression test.

## Best Possible Solution

Narrow the quiet logic so stale ambient keys are silent only when there is no explicit Ollama configuration anywhere, while configured alternate Ollama providers remain loud; then add the changelog entry and real CLI proof.

## Reproduction Assessment

Yes, at source level. Current main makes a real `OLLAMA_API_KEY` with no explicit `models.providers.ollama` set `quiet: false`, and `buildOllamaProvider` warns on an unreachable localhost probe.

## Solution Assessment

No. The fix direction is right for stale ambient keys, but `quiet: !hasMeaningfulExplicitConfig` is broader than the existing discovery contract and would silence explicitly configured alternate loopback Ollama providers.
