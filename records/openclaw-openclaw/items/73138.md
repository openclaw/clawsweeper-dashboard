## Summary

Keep open. The PR is useful but not merge-ready: source review found one runtime behavior blocker around stale cached guardrail config and one config-contract blocker around the new public schema surface. The dedicated security pass did not find a concrete supply-chain or secret-handling regression.

## What This Changes

The PR adds an observe-only MCP runtime guardrails module and tests, wires it around bundled MCP tool calls, exposes guardrail snapshots on session MCP runtimes, and adds `mcp.runtimeGuardrails` to MCP config types, Zod schema, and generated base schema.

## Best Possible Solution

Merge this direction only after `mcp.runtimeGuardrails` participates in the same runtime invalidation/hot-apply path as other relevant MCP runtime config, and after the public config contract is complete with help, labels, docs, generated baseline hash, changelog, and focused tests. Keep production behavior observe-only unless a separate reviewed change intentionally exposes enforcement.

## Reproduction Assessment

Yes for the runtime blocker: create a cached session MCP runtime, call `getOrCreate` again with unchanged `mcp.servers` but changed `mcp.runtimeGuardrails`, and the manager will return the existing runtime because the fingerprint only reflects MCP servers. The docs/metadata blocker is a source-contract check rather than a runtime reproduction.

## Solution Assessment

No, not as-is. The wrapper is a plausible narrow implementation, but the supported config surface needs hot-apply semantics and documentation/baseline parity before it is a maintainable solution; a safer alternative is keeping the guardrail config internal until those contracts are complete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Recreate runtimes when guardrails config changes:** `src/agents/pi-bundle-mcp-runtime.ts:197-199`
  - body: `guardrails` is created once from `params.cfg?.mcp?.runtimeGuardrails`, but cached session runtimes are only recreated when `loadSessionMcpConfig()` returns a different fingerprint, and current main derives that fingerprint from `loaded.mcpServers` only. Editing thresholds or tool annotations under the new config will keep returning the existing runtime and continue using stale guardrails despite the documented `mcp.*` hot-apply behavior. Include this subtree in the runtime fingerprint or otherwise dispose/recreate when it changes.
  - confidence: 0.9
- **[P2] Align the config docs contract before shipping:** `src/config/zod-schema.ts:269`
  - body: Adding `mcp.runtimeGuardrails` to the strict config schema makes it a supported public key, but this PR does not update the matching help, labels, configuration reference, generated baseline hash, or changelog. The generated schema applies `FIELD_HELP`/`FIELD_LABELS`, and `pnpm config:docs:check` tracks the baseline, so this would ship an undocumented and drifted config surface.
  - confidence: 0.89
