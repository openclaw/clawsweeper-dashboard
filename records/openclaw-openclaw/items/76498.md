## Summary

Keep open. Current main already separates OpenAI Platform API-key billing from Codex OAuth in the general OpenAI docs and implementation, but the Talk-specific docs and error path still do not explicitly tell users that OpenAI Realtime Talk needs funded OpenAI Platform API-key billing rather than ChatGPT/Codex subscription quota.

## What This Changes

Clarify OpenAI Realtime Talk setup and errors so users know it uses funded OpenAI Platform `OPENAI_API_KEY` billing, not ChatGPT/Codex subscription quota.

## Best Possible Solution

Add a targeted OpenAI Realtime Talk docs note and, if feasible, map missing-key or insufficient-quota startup failures to a diagnostic that points to funded OpenAI Platform API-key billing while leaving provider routing unchanged.

## Reproduction Assessment

Not applicable: this is a docs/UX clarification rather than a broken defined behavior. Source inspection confirms the relevant path uses `talk.provider: "openai"` with `OPENAI_API_KEY`/provider API-key billing and currently passes raw provider errors through.

## Solution Assessment

Yes: focused docs and diagnostic copy is the narrow maintainable solution. Routing realtime audio through `openai-codex` would be the wrong direction because that id is not registered as a realtime voice provider.
