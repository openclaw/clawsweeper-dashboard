## Summary

Keep open. Current main separates OpenAI Platform API-key billing from Codex OAuth in the provider docs and the implementation really uses an OpenAI API key for Realtime Talk, but the Talk-specific docs and runtime error path still do not explicitly explain that ChatGPT/Codex subscription quota does not fund OpenAI Realtime Talk audio.

## What This Changes

Request clearer Talk docs or error copy explaining that OpenAI Realtime Talk needs funded OpenAI Platform API-key billing, not ChatGPT/Codex subscription or `openai-codex` quota.

## Best Possible Solution

Add a targeted OpenAI Realtime Talk note in the provider and Control UI Talk docs, and optionally map OpenAI Realtime quota/missing-key startup failures to a diagnostic that points at funded OpenAI Platform `OPENAI_API_KEY` billing while leaving provider routing unchanged.

## Reproduction Assessment

Not applicable: this is a docs/UX clarification rather than a broken defined contract. Source inspection confirms the relevant behavior: OpenAI Realtime Talk uses the OpenAI API-key path and currently passes raw provider errors through.

## Solution Assessment

Yes: a focused docs and diagnostic-copy change is the maintainable fix. Routing realtime audio through `openai-codex` would be the wrong solution because `openai-codex` is not a registered realtime voice provider.
