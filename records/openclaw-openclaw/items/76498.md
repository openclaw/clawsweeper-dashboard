## Summary

Keep open. Current `main` no longer supports the issue's original absolute wording because OpenAI Realtime Talk can use an `openai-codex` OAuth fallback when no direct key is configured, but the direct API-key `insufficient_quota` confusion still deserves a targeted docs/UX clarification and an open linked PR already tracks the work.

## What This Changes

The issue asks OpenClaw to clarify OpenAI Realtime Talk billing/auth so users do not confuse direct OpenAI Platform API-key quota with ChatGPT/Codex subscription quota or the `openai-codex` provider id.

## Best Possible Solution

Update the linked docs PR or land equivalent wording that explains direct OpenAI Platform API-key billing/quota failures while preserving the current `openai` realtime provider id, `talk.realtime.*` keys, and supported `openai-codex` OAuth fallback.

## Reproduction Assessment

Not applicable: this is a docs/UX clarification, not a failing defined runtime contract. Source inspection gives high confidence about the current auth paths and the remaining documentation gap, but no live billing failure was exercised.

## Solution Assessment

Yes, with updated scope: a narrow docs/diagnostic clarification is the right fix. The wording should not say Realtime requires only `OPENAI_API_KEY`, because current source and tests support `openai-codex` OAuth fallback when no direct key is configured.
