## Summary

Keep open. Current main still has a prompt-visible Slack skill that describes a callable `slack` tool while the Claude Code harness is constrained to OpenClaw MCP-prefixed bundle tools, and the requested Slack workspace/app admin actions are still absent; user-token read plumbing has landed, but it does not close the central gaps.

## What This Changes

Requests making Slack actions callable from Claude Code/ToolSearch and adding gated Slack workspace/app administration operations.

## Best Possible Solution

Choose the Claude/MCP exposure contract first, then implement a disabled-by-default Slack admin action group with SecretRef-compatible tokens, scope documentation, and focused Slack dispatcher/config/tool-surface tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the Slack skill advertises `slack`, Claude Code only allows `mcp__openclaw__*`, bundle MCP loading skips channel entries, and the Slack dispatcher lacks the requested admin actions.

## Solution Assessment

Unclear. The proposed MCP bridge plus admin action group is plausible, but maintainers should first settle the callable-tool boundary and high-privilege Slack token policy; skill gating is a safer interim if no callable Slack bridge is approved.
