## Summary

Keep open. Current main still matches the report: Slack actions are implemented as a channel message-action surface, the Claude/ACPX MCP paths do not expose a Slack-specific callable tool, and the requested workspace/app administration actions are absent. The reporter's follow-up usefully narrows this to a maintainer architecture and security decision rather than a conservative cleanup close.

## What This Changes

The issue requests exposing the existing Slack action surface to Claude Code/tool search and adding opt-in Slack admin actions for channel, member, and app-manifest management.

## Best Possible Solution

Define the Claude/ACPX tool-exposure contract first, then split implementation into a Slack-owned bridge or skill-loading correction plus a separate opt-in admin action group. The admin path should be disabled by default, SecretRef-compatible, scope-documented, and covered by Slack action runtime, config schema, MCP/tool discovery, and Claude/ACPX visibility tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `skills/slack/SKILL.md` advertises a `slack` tool, while current Slack actions are channel-message adapters and the MCP bridge code does not register or expose a Slack-specific callable tool.

## Solution Assessment

Unclear. The reporter's proposed MCP bridge is plausible, but the narrowest maintainable fix depends on maintainer choice between bridging, suppressing unreachable skill guidance, or both; the high-privilege admin operations should be treated as a separate opt-in feature.
