## Summary

Keep open. Current main still matches the report: Slack's implemented actions remain a Slack channel message-action surface, the Claude/OpenClaw MCP bridge does not expose a callable Slack tool, and the requested Slack workspace/app administration actions are still absent. The reporter's follow-up usefully narrowed this to a maintainer/security decision about tool exposure and privileged Slack token policy, not a conservative cleanup close.

## What This Changes

The issue requests exposing the existing Slack action surface to Claude Code/ToolSearch and adding opt-in Slack admin actions with user/app-configuration token plumbing.

## Best Possible Solution

Define the Claude/ACPX Slack tool-exposure contract first, then split implementation into a Slack-owned bridge or skill-loading fix plus a separate opt-in admin action group. The admin path should be disabled by default, use SecretRef-compatible user/app-configuration token resolution, document required Slack scopes, and add coverage for Slack action runtime, config schema, MCP/tool discovery, ACPX/Claude visibility, and public docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main loads a Slack skill that tells Claude about a `slack` tool, while the Claude/OpenClaw MCP path exposes no Slack callable tool and the Slack runtime/docs contain no requested admin actions.

## Solution Assessment

Unclear. The proposed direction is plausible, but the narrowest maintainable fix depends on maintainer choice between adding a Slack MCP bridge, suppressing unreachable skill guidance, or both; privileged admin actions should be reviewed as a separate opt-in feature.
