## Summary

Keep open. Current main still matches the central report: Slack is implemented as a channel message-action surface, the prompt-visible Slack skill describes a callable `slack` tool, Claude Code is restricted to OpenClaw MCP-prefixed tools, and neither the core tools bridge nor the channel MCP bridge exposes Slack message/admin actions. The admin-token part is narrower than originally filed because `userToken` and guarded user-token writes already exist, but app-configuration tokens and the requested workspace/app administration actions are still missing and need maintainer security direction.

## What This Changes

The issue requests making Slack channel actions callable from Claude Code/ToolSearch and adding opt-in Slack workspace/app administration actions such as channel creation, invites, member lookup, member listing, Slack app manifest create/update, and channel archive.

## Best Possible Solution

Define the Claude/ACPX Slack tool exposure contract first, then split the work into a Slack-owned callable bridge or skill-gating fix plus a separate Slack admin action group. The admin group should be disabled by default, SecretRef-compatible, scope-documented, account-aware, and covered by Slack action runtime, config schema, MCP/tool discovery, and Claude/ACPX visibility tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main has a Slack skill and Slack channel actions, but Claude Code is allowed only `mcp__openclaw__*` tools and the current MCP bridges do not expose Slack actions or admin operations.

## Solution Assessment

Unclear. The reporter’s bridge-versus-skill-gating split is the right decision point, but maintainers should choose that contract before implementation; the high-privilege admin operations should be reviewed as a separate opt-in feature.
