## Summary

Keep open. Current main has partial Slack user-token plumbing, but Slack is still surfaced through shared message/channel paths rather than a Slack ToolSearch/MCP tool, and the requested workspace/app administration actions are still missing; this needs maintainer direction on both Claude/MCP exposure and high-privilege Slack authority.

## What This Changes

Request to expose Slack actions to Claude Code/ToolSearch and add gated Slack workspace/app administration actions with admin-token configuration.

## Best Possible Solution

Align Slack skill guidance with the actual Claude tool surface first, then add a disabled-by-default Slack admin action group with SecretRef-compatible tokens, documented scopes, and tests for token selection, gating, and MCP/message-tool exposure.

## Reproduction Assessment

Yes. A source-level reproduction is enough: the Slack skill advertises `slack`, Claude allows only `mcp__openclaw__*`, bundle discovery only includes bundle-format plugins, Slack is a channel entry, and the Slack dispatcher/action discovery lacks the requested admin cases.

## Solution Assessment

Unclear. The proposed direction is plausible, but maintainers should choose the bridge versus skill-loading behavior and the admin-token policy before a repair PR starts.
