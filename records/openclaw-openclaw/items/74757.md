## Summary

Keep open. Current main still matches the central report: Slack actions exist as a Slack channel message-action surface and prompt-visible skill guidance, but Claude Code's OpenClaw MCP exposure has no Slack callable tool, and the requested Slack workspace/app administration actions remain absent. The issue discussion also frames the next step as a maintainer architecture and security decision, so this should not be cleanup-closed.

## What This Changes

The issue asks OpenClaw to make Slack actions callable from Claude Code/ToolSearch and add opt-in Slack workspace/app administration actions with admin-token plumbing.

## Best Possible Solution

Define the Claude/ACPX tool-exposure contract first, then split implementation into a Slack-owned callable bridge or harness skill-gating fix plus a separate opt-in admin action group. The admin path should be disabled by default, use SecretRef-compatible user/app-configuration token resolution, document required Slack scopes, and add tests for Slack action runtime, config schema, MCP/tool discovery, and Claude/ACPX visibility.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: Slack skills can be visible to Claude Code, but Slack actions are only channel message actions, Claude Code is allowlisted to `mcp__openclaw__*`, and the OpenClaw tools MCP bridge currently has no Slack callable tool.

## Solution Assessment

Unclear. The proposed bridge plus admin group is plausible, but the narrowest maintainable fix depends on maintainer choice between adding a Slack MCP bridge, suppressing unreachable skill guidance, or both; high-privilege admin actions should be reviewed as a separate opt-in feature.
