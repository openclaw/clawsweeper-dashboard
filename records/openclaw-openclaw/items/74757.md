## Summary

Keep open. Current main still matches the report: Slack has channel-owned message actions and a prompt-visible Slack skill, but Claude Code is restricted to OpenClaw MCP-prefixed tools and the current MCP bridge paths do not expose Slack channel actions or the requested Slack workspace/app administration operations. The issue is active and security-sensitive enough that maintainers should choose the exposure contract and token policy before implementation.

## What This Changes

Expose Slack actions to Claude Code/ToolSearch and add gated Slack workspace/app administration actions such as channel create/invite, user lookup/listing, app manifest create/update, and channel archive.

## Best Possible Solution

Define the Claude/ACPX Slack exposure contract first, then split the implementation into a Slack-owned callable bridge or skill-gating fix plus a separate opt-in Slack admin action group. The admin group should be disabled by default, SecretRef-compatible, scope-documented, account-aware, and covered by Slack action runtime, config schema, MCP/tool discovery, and Claude/ACPX visibility tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main has a prompt-visible Slack skill and Slack channel actions, while Claude Code is restricted to `mcp__openclaw__*` tools and the current MCP bridge paths do not expose Slack action/admin operations.

## Solution Assessment

Unclear. The reporter's bridge-versus-skill-gating framing is the right decision point, but maintainers should choose that contract first; the high-privilege admin operations are safer as a separate opt-in feature rather than bundled into an automatic visibility fix.
