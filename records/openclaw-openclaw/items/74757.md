## Summary

Keep open. Current main still leaves the central product decision unresolved: Slack has channel-owned actions and the generic `message` tool can route some channel actions, but the bundled Slack skill still tells Claude to use a `slack` tool, Slack is not a bundle/MCP plugin exposing a dedicated `slack` tool, and the requested workspace/app administration actions are not implemented. The discussion is recent and the reporter is explicitly waiting for maintainer direction on the MCP bridge versus skill-gating shape before sending a PR.

## What This Changes

Request to make Slack actions discoverable to Claude Code/ToolSearch and add opt-in Slack admin operations with user-token and app-configuration-token plumbing.

## Best Possible Solution

Define the Claude/ACPX Slack tool exposure contract first, then split implementation into a Slack-owned callable bridge or skill-gating fix plus a separate disabled-by-default Slack admin action group. The admin group should be SecretRef-compatible, scope-documented, account-aware, and covered by Slack action runtime, config schema, MCP/tool discovery, and Claude visibility tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main has a Slack skill and channel actions, but no dedicated Slack MCP/bundle tool and no requested admin/app-manifest actions; the prompt-visible skill therefore still does not match the callable `slack` surface described by the report.

## Solution Assessment

Unclear. The reporter's proposed split is a good starting point, but maintainers should decide the bridge-versus-skill-gating contract and approve the high-privilege admin-token policy before implementation.
