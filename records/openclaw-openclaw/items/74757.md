## Summary

Keep open. Current main still matches the central report: Slack actions remain a Slack channel message-action surface, Claude Code is limited to OpenClaw MCP-prefixed tools, no Slack MCP/tool bridge is present, and the requested Slack workspace/app administration actions are still missing. Current main does include partial user-token plumbing for reads and guarded writes, so the remaining work is narrower than the original token-plumbing request but still needs maintainer/security direction.

## What This Changes

Request to expose the existing Slack action surface to Claude Code/ToolSearch and add opt-in Slack administration actions for channel, member, and app-manifest management.

## Best Possible Solution

Define the Claude/ACPX tool-exposure contract first, then split implementation into a Slack-owned callable bridge or skill-gating fix plus a separate opt-in Slack admin action group. The admin path should be disabled by default, SecretRef-compatible, scope-documented, and covered by Slack action runtime, config schema, MCP/tool discovery, and Claude/ACPX visibility tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main has a prompt-visible Slack skill and Slack channel actions, but neither the OpenClaw-tools MCP bridge nor the plugin-tools MCP bridge exposes a callable Slack tool to Claude Code.

## Solution Assessment

Unclear. The proposed MCP bridge is plausible, but maintainers should first choose between bridge-only, skill-gating-only, or both; the admin operations should be reviewed as a separate high-privilege opt-in feature.
