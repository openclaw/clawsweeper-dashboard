## Summary

Keep open. Current main still matches the report: Slack actions exist, but they are Slack channel message actions rather than a Claude/MCP-discoverable `slack` tool, and the requested workspace/app administration actions plus app-configuration-token plumbing are still absent. The reporter’s follow-up correctly identifies this as a product/security boundary decision rather than a missed one-line registration.

## What This Changes

Expose Slack channel actions to Claude Code/ToolSearch and add opt-in Slack admin operations for channel, member, and app-manifest management.

## Best Possible Solution

Define the Claude/ACPX Slack exposure contract first, then split the work into a Slack-owned callable bridge or skill-gating fix and a separate opt-in Slack admin action group. The admin group should be default-off, SecretRef-compatible, scope-documented, account-aware, and covered by Slack action runtime, config schema, MCP/tool discovery, and Claude/ACPX visibility tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main has a prompt-visible Slack skill and Slack channel-owned actions, while Claude Code is constrained to `mcp__openclaw__*` and the MCP bridges do not register Slack action or admin tools.

## Solution Assessment

Unclear. The reporter’s bridge-versus-skill-gating options are the right decision point, but maintainers should choose that contract first; the high-privilege admin operations are better reviewed as a separate opt-in feature rather than bundled into an automatic visibility fix.
