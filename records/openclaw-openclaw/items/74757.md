## Summary

Keep open. Current main still matches the core report: Slack actions are implemented as a Slack channel message-action surface, the bundled Slack skill still describes a callable `slack` tool, Claude Code is restricted to OpenClaw MCP-prefixed tools, and the current MCP bridges do not expose Slack channel actions or the requested workspace/app administration operations. The existing `userToken` support narrows the token-plumbing gap, but app-configuration-token support and admin actions still need maintainer security/product direction.

## What This Changes

Request exposing Slack actions to Claude Code/ToolSearch and adding Slack workspace/app administration actions such as channel creation, invites, user lookup/listing, app manifest creation/update, and channel archival.

## Best Possible Solution

Define the Claude/ACPX Slack exposure contract first, then split the implementation into a Slack-owned callable bridge or skill-gating fix plus a separate opt-in Slack admin action group. The admin group should be disabled by default, SecretRef-compatible, scope-documented, account-aware, and covered by Slack action runtime, config schema, MCP/tool discovery, and Claude/ACPX visibility tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main has a prompt-visible Slack skill and channel-owned Slack actions, while Claude Code is restricted to `mcp__openclaw__*` tools and the available MCP bridges do not expose Slack action/admin operations.

## Solution Assessment

Unclear. The reporter’s bridge-versus-skill-gating framing is the right decision point, but maintainers should choose that contract first; the high-privilege admin operations are better reviewed as a separate opt-in feature rather than bundled into an automatic visibility fix.
