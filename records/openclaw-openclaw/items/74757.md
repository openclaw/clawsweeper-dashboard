## Summary

Keep open. Current main still matches the report: Slack has channel-owned message actions and a prompt-visible Slack skill, but Claude/ACPX tool exposure is limited to MCP bridges that do not surface those Slack actions, and the requested workspace/app admin operations are not implemented. The reporter's follow-up correctly identified this as an architecture/security decision rather than a missed single registration.

## What This Changes

Request to make Slack actions discoverable/callable from Claude Code ToolSearch and add gated Slack admin actions with user/app-configuration token plumbing.

## Best Possible Solution

Define the Slack tool exposure contract first, then split implementation into a Slack-owned callable bridge or skill-gating fix plus a separate opt-in Slack admin action group. The admin group should be disabled by default, SecretRef-compatible, scope-documented, account-aware, and covered by Slack action runtime, config schema, MCP/Claude visibility, docs, and changelog updates.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main loads a Slack skill and Slack channel actions, but the Claude CLI/backend only allows OpenClaw MCP-prefixed tools and the current MCP bridges do not expose Slack channel actions or admin operations.

## Solution Assessment

Unclear. The reporter's proposed bridge plus admin action group is plausible, but maintainers should first choose the bridge versus skill-gating contract and approve the high-privilege Slack token policy; a safer implementation is likely two separate PRs.
