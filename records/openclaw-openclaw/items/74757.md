## Summary

Keep open. Current main has partial Slack user-token/read support, but the prompt-visible Slack skill still names a `slack` tool while Claude is limited to OpenClaw MCP tools, and the requested Slack workspace/app administration actions are still absent.

## What This Changes

Request Slack tool discoverability for Claude sessions plus Slack workspace/app administration actions with admin-token plumbing and action gating.

## Best Possible Solution

Align Slack skill loading with the actual callable Claude surface or add a thin Slack MCP bridge, then add a disabled-by-default admin action group with SecretRef-compatible admin tokens, documented scopes, and focused dispatcher/config tests.

## Reproduction Assessment

Yes. A source-level reproduction exists: the Slack skill advertises `slack`, Claude accepts only `mcp__openclaw__*`, bundle MCP discovery skips non-bundle channel plugins, and the Slack dispatcher/action list lacks the requested admin actions.

## Solution Assessment

Unclear. The reporter's MCP bridge plus admin action group is plausible, but maintainers should first choose the Claude exposure shape and admin-token policy because both affect security and plugin architecture.
