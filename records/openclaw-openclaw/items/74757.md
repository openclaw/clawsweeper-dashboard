## Summary

Keep open. Current main still does not expose Slack channel actions as a Claude-discoverable `slack`/`mcp__openclaw__slack_*` tool and still lacks the requested Slack workspace/app administration actions; the remaining work needs maintainer product and security direction before implementation.

## What This Changes

Add Claude-visible Slack action tooling and opt-in Slack admin actions for channel/app creation, invites, member lookup, member listing, app manifests, and archiving.

## Best Possible Solution

Define the Claude/ACPX Slack exposure contract first, then split implementation into a skill-loading or Slack-action bridge fix plus a separate disabled-by-default Slack admin action group with documented scopes and SecretRef-compatible tokens.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the Slack skill is loaded as if a `slack` tool exists, while Slack actions remain channel-owned and the available Claude/MCP bridge does not expose Slack action or admin methods.

## Solution Assessment

Unclear. The reporter’s bridge-versus-skill-gating options frame the right decision, but the maintainable path should be chosen by maintainers before adding high-privilege admin actions.
