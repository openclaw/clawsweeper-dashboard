## Summary

Keep open. Current main still leaves the central problem unresolved: Slack actions remain a Slack channel message-action surface, Claude Code is restricted to OpenClaw MCP-prefixed tools, the OpenClaw-tools MCP bridge does not expose Slack actions, and the requested Slack workspace/app administration actions are not present as callable Slack tooling.

## What This Changes

The issue requests making Slack actions discoverable to Claude Code/ToolSearch and adding opt-in Slack admin actions for channel creation, invites, user/member lookup, app manifest create/update, and archiving.

## Best Possible Solution

Define the Claude/ACPX tool-exposure contract first, then split the work into a Slack-owned bridge or skill-loading fix plus a separate opt-in admin action group. The admin path should be disabled by default, SecretRef-compatible, scope-documented, and covered by tests for Slack action runtime, config schema, MCP/tool discovery, and Claude visibility.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure Slack so the bundled skill is eligible, run through Claude Code, and the prompt-visible `slack` instructions have no matching OpenClaw MCP Slack tool because current main only wires Slack actions as channel message actions.

## Solution Assessment

Unclear. The proposed direction is plausible, but the best maintainable fix depends on maintainer choice between a Slack MCP bridge, conditional skill suppression, or both; the high-privilege admin operations should be reviewed as a separate opt-in feature.
