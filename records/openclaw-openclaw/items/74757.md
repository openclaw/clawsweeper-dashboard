## Summary

Keep open. Current main still leaves the reported user problem unresolved: Slack actions remain a channel-message action surface, the Claude/ACPX MCP bridge does not expose a callable Slack tool, and the requested Slack workspace/app administration actions are absent. Current main has `userToken` support, but that only covers read/write token selection for existing actions, not `appConfigToken` or admin actions.

## What This Changes

The issue requests exposing Slack actions to Claude Code/ToolSearch and adding opt-in Slack workspace/app administration actions with user/app-configuration token plumbing.

## Best Possible Solution

Define the Claude/ACPX tool-exposure contract first, then split the work into a Slack-owned MCP/tool bridge or skill-loading fix plus a separate opt-in Slack admin action group. The admin path should stay disabled by default, support SecretRef resolution, document required Slack scopes, and cover channel/app/user operations with targeted tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the Slack skill is prompt-visible, Slack actions are implemented behind the channel message-action adapter, the OpenClaw-tools MCP bridge has no Slack tool, and targeted searches show the requested admin endpoints are not implemented.

## Solution Assessment

Unclear. The reporter’s proposed direction is plausible, but the best fix depends on maintainer choice between adding a Slack MCP bridge, suppressing unreachable Slack skill guidance for Claude Code, or doing both; the admin operations should be reviewed as a separate high-privilege feature.
