## Summary

Keep open: current main and the latest release still replay old outbound delivery-queue entries without a queued-age expiry policy, and the open implementation candidate has not merged and still needs TTL/default/config policy alignment.

## What This Changes

The issue requests configurable TTL/expiry handling for stale delivery-queue messages so old queued channel messages are not replayed on gateway restart.

## Best Possible Solution

Keep this as the canonical TTL policy issue, then land or replace the open candidate with a backward-compatible expiry policy shared by startup recovery and reconnect drains after maintainers decide config shape, default, expire action, and inbound scope.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with retryCount 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main treats it as eligible, and gateway startup calls that path without a TTL option.

## Solution Assessment

Unclear as proposed. The outbound recovery boundary is right, but the TTL default, config shape, expire action, reconnect-drain behavior, and inbound scope need maintainer policy alignment before this can close.
