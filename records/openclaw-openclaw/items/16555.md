- `impact:message-loss`: The reported behavior causes stale or duplicate outbound messages to be replayed into user channels after gateway recovery.

## Summary

Keep open: current main and the latest release still replay pending outbound delivery entries by retry and backoff policy without any queued-age TTL, and the duplicate report plus closed implementation attempts route back to this issue as the canonical policy item.

## What This Changes

The issue requests an opt-in TTL/expiry policy for persisted outbound delivery queue messages so stale queued messages are not replayed on gateway restart.

## Best Possible Solution

Keep this as the canonical outbound stale-replay policy item, then land a narrow current-main fix against the SQLite-backed outbound delivery queue that expires stale pending entries after claim/re-read and aligns config, generated docs/schema, and tests with the maintainer-chosen default and action.

## Reproduction Assessment

Yes, at source level. Seed an old pending outbound entry with retryCount 0 and no lastAttemptAt; current recovery treats it as eligible and no current config/schema field can set an age cutoff.

## Solution Assessment

Unclear as the final implementation shape. Core outbound recovery is the right layer, but maintainers still need to choose the default, config surface, and expiry action before a safe implementation PR.
