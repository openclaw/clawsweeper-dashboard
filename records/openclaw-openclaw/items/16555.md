## Summary

Keep open. Current main still has no age or expiry gate for outbound delivery-queue recovery, the source path still replays old first-attempt entries on gateway startup, and the focused open PR remains an unresolved implementation candidate rather than a merged fix.

## What This Changes

The issue requests an optional expiry policy for outbound delivery queue entries so stale queued messages are not replayed after gateway restart.

## Best Possible Solution

Land or replace a focused outbound recovery expiry policy that skips or quarantines stale queued replies while preserving recent crash recovery, reconnect drains, budget deferrals, and uncertain-send reconciliation.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with `retryCount: 0` and no `lastAttemptAt`, then run `recoverPendingDeliveries`; current main treats it as eligible, and gateway startup calls that recovery path.

## Solution Assessment

Unclear as proposed. The outbound recovery boundary is right, but the operator-facing config, default TTL, expiry action, reconnect behavior, and uncertain-send handling need maintainer alignment before this can close.
