## Label Justifications

- `impact:message-loss`: The issue concerns stale, duplicate, and wrongly timed channel-message delivery during queue recovery.

## Summary

Current main still replays pending outbound entries without a general queued-age expiry policy. Keep this issue open while the active implementation PR is reviewed, because it only expires old first-attempt entries during startup recovery and still needs maintainer agreement on compatibility, scope, and default behavior.

## What This Changes

Add bounded expiry handling for stale outbound delivery-queue entries so gateway recovery does not replay obsolete messages.

## Best Possible Solution

Define one closed expiry policy at the durable delivery-queue owner boundary, apply it consistently to startup and reconnect drains, preserve or explicitly approve compatibility behavior, and prove both stale quarantine and legitimate crash recovery in a real gateway setup.

## Reproduction Assessment

Yes at source level: current main persists enqueuedAt but its startup and reconnect recovery paths have no general age-expiry decision. This read-only review did not execute a live crash/restart reproduction.

## Solution Assessment

Unclear. Expiry belongs in the delivery-queue owner boundary, but the active PR's unconditional ten-minute first-attempt cutoff is narrower than the request and needs maintainer agreement on default behavior, retry-entry treatment, and reconnect symmetry.
