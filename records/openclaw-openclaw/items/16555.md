## Summary

Keep open. Current main and the latest release still lack an outbound delivery-queue TTL or expiry action, and the focused implementation candidate is now closed, so this issue remains the canonical place for the policy decision.

## What This Changes

Request to add a configurable TTL/expiry policy for outbound delivery queue recovery so stale pending messages are not replayed after gateway restart.

## Best Possible Solution

Define a closed outbound expiry policy at the delivery-queue owner boundary, apply it consistently to startup recovery and reconnect drains, and update config schema, generated docs, and focused tests after maintainers choose the compatibility default.

## Reproduction Assessment

Yes at source level. Seed an old pending outbound delivery entry and run startup recovery; current main uses `enqueuedAt` for ordering but has no age-expiry decision before replay.

## Solution Assessment

Unclear until maintainers choose the policy. Expiry belongs in the delivery-queue owner boundary, but the TTL default, opt-in behavior, and expired-entry action are product and compatibility decisions.
