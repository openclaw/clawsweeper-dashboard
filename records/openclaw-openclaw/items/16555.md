## Label Justifications

- `P2`: Stale recovery can repeatedly deliver old or duplicate channel messages, but it does not make the core runtime unusable.
- `impact:message-loss`: The requested queue policy controls whether outbound messages are duplicated, wrongly replayed, or suppressed during recovery.

## Summary

Keep open. Current main and the latest release still lack a queued-age expiry policy for outbound recovery, and the open focused PR is not merged and is narrower than the full configurable/shared-path behavior requested.

## What This Changes

Request adds a TTL/expiry policy for outbound delivery queue messages so stale queued messages are not replayed after gateway restart.

## Best Possible Solution

Define one auditable stale-entry policy at the queue policy or storage boundary, apply it consistently to startup recovery and reconnect drains, and keep operator configuration minimal with focused recovery/storage/config tests.

## Reproduction Assessment

Yes, at source level. Seed an old pending outbound delivery entry with retryCount 0 and no lastAttemptAt, then run startup recovery; current main treats it as eligible and gateway startup invokes that path without an age policy.

## Solution Assessment

No, the broad three-action config proposal and the current startup-only PR are not yet the best complete solution. The maintainable direction is a shared expiry invariant with explicit retry semantics, terminal-state handling, and an agreed config/storage boundary.
