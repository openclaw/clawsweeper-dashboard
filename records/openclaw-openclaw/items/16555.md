## Summary

Keep open: current main and the latest release still lack a configurable outbound delivery-queue TTL/expiry gate, and the focused implementation candidate is still open and unmerged.

## What This Changes

The issue requests an optional TTL/expiry policy for persisted outbound delivery queue entries so stale messages are not replayed after gateway restart.

## Best Possible Solution

Land or replace a focused outbound delivery expiry policy that prevents stale queue replay while preserving recent crash recovery, reconnect drains, backoff, and unknown-send reconciliation semantics.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with retryCount 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main treats it as eligible and gateway startup calls that path.

## Solution Assessment

Unclear as proposed. The delivery-queue recovery boundary is right, but TTL duration, default behavior, expire action, config location, and inbound scope need maintainer alignment before the open candidate PR or a replacement can close this.
