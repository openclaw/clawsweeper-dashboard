## Summary

Keep open: current main still lacks an outbound delivery-queue age cutoff or `messages.delivery` config surface, and the focused implementation candidate is open, conflicting, and has review/product-policy feedback to resolve.

## What This Changes

Add configurable expiry for outbound delivery queue messages so stale queued messages are not replayed after gateway restart.

## Best Possible Solution

Land or replace the focused outbound recovery expiry policy so stale pending deliveries are skipped or quarantined while recent legitimate crash recovery still works, with config/schema/docs updates only after maintainers settle the operator-facing policy.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery with `retryCount: 0` and no `lastAttemptAt`, then run `recoverPendingDeliveries`; current main treats it as eligible and gateway startup calls that recovery path.

## Solution Assessment

Unclear as proposed. The shared outbound recovery boundary is the right place, but the config surface, default TTL/action, and the open PR's attempted-send predicate need maintainer alignment and repair before merge.
