## Summary

Keep open. Current main still has no delivery-queue age cutoff or `messages.delivery` config surface, while the production repro and closed unmerged outbox PRs show this remains real work rather than cleanup-close material.

## What This Changes

The issue requests an optional TTL/expiry policy for outbound delivery queue entries so stale queued messages are not replayed after gateway restarts.

## Best Possible Solution

Add a narrow optional expiry policy to the existing file-backed outbound delivery queue, enforced in both startup recovery and reconnect drains, with config/schema/docs/tests aligned and default behavior unchanged.

## Reproduction Assessment

Yes. Seed a `delivery-queue/*.json` entry with an old `enqueuedAt` and otherwise eligible retry/backoff state, then invoke startup recovery or a matching reconnect drain; source inspection shows current main will still consider it for delivery because no age check exists.

## Solution Assessment

Yes. The narrow maintainable direction is an opt-in TTL/expiry policy on the current queue; the broader SQLite/outbox rewrites referenced in the discussion are closed and unmerged, so they should remain prior art rather than the active fix path.
