## Summary

Keep open. Current main still has no age or expiry gate for pending outbound delivery queue recovery, and the focused implementation path remains open with proof, conflict, and policy blockers rather than merged.

## What This Changes

Request adding configurable age-based expiry for outbound delivery queue entries so stale pending messages are not replayed after gateway restarts.

## Best Possible Solution

Land or replace a focused outbound recovery expiry policy that skips or quarantines stale queued replies while preserving recent crash recovery, active claims, reconnect drains, and uncertain-send reconciliation; treat inbound freshness as a separate follow-up if needed.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with `retryCount: 0` and no `lastAttemptAt`, then run `recoverPendingDeliveries`; current main treats it as eligible and gateway startup calls that path.

## Solution Assessment

Unclear as proposed. The outbound recovery boundary is right, but the operator-facing config, default TTL, expire action, reconnect behavior, and uncertain-send handling need maintainer alignment before this can close.
