## Summary

Keep open. Current main still lacks an outbound delivery-queue age or expiry gate, and the focused implementation PR remains open rather than merged; the remaining work also needs maintainer alignment on TTL/default/action policy.

## What This Changes

Request adds configurable TTL/expiry for persisted outbound delivery queue entries so stale messages are skipped or quarantined instead of replayed after gateway restart.

## Best Possible Solution

Land or replace a focused outbound expiry policy that quarantines stale queued replies while preserving recent crash recovery, reconnect drains, budget deferrals, and uncertain-send reconciliation.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with `retryCount: 0` and no `lastAttemptAt`, then run `recoverPendingDeliveries`; current main treats it as eligible and gateway startup calls that recovery path.

## Solution Assessment

Unclear as proposed. The outbound recovery boundary is right, but the config surface, default TTL, expire action, reconnect behavior, and uncertain-send handling need maintainer alignment before this can close.
