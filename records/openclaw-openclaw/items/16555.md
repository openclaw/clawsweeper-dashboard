## Summary

Keep open: current main and v2026.6.6 still recover outbound delivery queue entries by max-retry and backoff only, with no queued-age expiry or accepted `messages.delivery.*` config surface; the related TTL PRs are closed and unmerged, so this remains the canonical policy issue.

## What This Changes

Request to add configurable expiry/TTL for outbound delivery queue entries so stale queued messages are skipped, deleted, or moved to failed instead of replayed after restart.

## Best Possible Solution

Define a narrow outbound delivery expiry policy at the core queue boundary, preserve default compatibility or explicitly choose a new default, add strict config/schema/docs and tests for startup recovery and reconnect drain, and track the inbound mirror separately unless maintainers deliberately expand scope.

## Reproduction Assessment

Yes at source level: create an old pending outbound delivery with `retryCount: 0` and no `lastAttemptAt`; `recoverPendingDeliveries` loads it and `isEntryEligibleForRecoveryRetry` returns eligible without an age check. No live gateway restart was run because this review is read-only.

## Solution Assessment

Unclear as an implementation path: adding expiry at the delivery queue boundary is likely the right layer, but the threshold, default, and expired-entry action are product/config decisions. A hardcoded one-off cutoff without maintainer alignment would not be the best fix.
