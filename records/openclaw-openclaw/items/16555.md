## Summary

Keep open: current main and the latest release still lack a queued-age expiry policy for outbound delivery recovery, and the remaining work is already paired with an open but conflicting implementation candidate plus unresolved TTL/config/default/inbound-scope decisions.

## What This Changes

The issue requests configurable TTL/expiry behavior for stale outbound delivery queue messages so old entries are not replayed after gateway restart.

## Best Possible Solution

Keep this as the canonical TTL policy issue, then land or replace the open candidate with a backward-compatible expiry policy that covers startup recovery and reconnect drains after maintainers decide the config shape, default, expire action, and inbound scope.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with retryCount 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main treats it as eligible and gateway startup calls that path without a TTL option.

## Solution Assessment

Unclear as proposed. The outbound recovery boundary is right, but the TTL default, config shape, expire action, reconnect-drain behavior, and inbound scope need maintainer policy alignment before this can close.
