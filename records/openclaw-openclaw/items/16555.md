## Summary

Keep open. Current main still has the file-backed outbound delivery queue replay otherwise eligible pending entries by retry/backoff/max-retry rules only, with no queued-age cutoff and no `messages.delivery` config surface; the related SQLite/outbox attempts in the discussion are closed and unmerged prior art rather than a canonical fix.

## What This Changes

The issue requests configurable TTL/expiry handling for outbound delivery-queue messages so stale queued entries are not replayed on gateway restart.

## Best Possible Solution

Add a narrow optional expiry policy to the existing file-backed outbound delivery queue, enforced in both startup recovery and reconnect drains, with config/schema/docs/tests aligned and default behavior unchanged.

## Reproduction Assessment

Yes. Seed a `delivery-queue/*.json` entry with an old `enqueuedAt` and otherwise eligible retry/backoff state, then invoke startup recovery or a matching reconnect drain; source inspection shows current main will still consider it for delivery because no age check exists.

## Solution Assessment

Yes. The narrow maintainable direction is an opt-in TTL/expiry policy on the current queue; the broader SQLite/outbox rewrites referenced in the discussion are closed and unmerged, so they should remain prior art rather than the active fix path.

## Repair Work Prompt

Fix #16555 by adding optional TTL/expiry handling for the existing file-backed outbound delivery queue. Keep the default behavior backward-compatible: no expiry unless `messages.delivery.maxAgeMs` is configured. Add typed and strict config support for `messages.delivery.maxAgeMs` and an expiry action that safely moves expired entries to failed/quarantine rather than deleting by default. Enforce the policy in both `recoverPendingDeliveries` and `drainPendingDeliveries`, using `enqueuedAt` as the age source, and log a concise summary when entries expire. Do not revive the broad SQLite/message-lifecycle rewrites from the closed PRs; use them only as prior art. Add regression coverage for startup recovery, reconnect drain, storage/policy helpers, and config acceptance/rejection. Add a user-facing changelog entry and update generated config docs/schema if the repo requires it. Stop and ask if the implementation would change default no-expiry behavior or require a broader storage migration.
