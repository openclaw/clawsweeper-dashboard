## Summary

Keep open. Current main still has no outbound delivery-queue age cutoff, and the existing open implementation candidate remains unmerged with review/proof/policy blockers.

## What This Changes

The issue requests TTL or expiry handling for stale outbound delivery queue messages so gateway restart recovery does not replay old queued sends.

## Best Possible Solution

Keep this as the canonical outbound delivery-queue freshness tracker and land or replace a focused recovery expiry policy that preserves legitimate recent crash recovery while quarantining stale queued replies.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with retryCount: 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main treats it as eligible and gateway startup calls that path.

## Solution Assessment

Unclear as proposed. The outbound recovery boundary is right, but the default TTL, action vocabulary, config surface, and reconnect/budget-deferral semantics need maintainer alignment.
