## Summary

Keep open: current main and v2026.6.5 still lack an outbound delivery-queue TTL/expiry policy, and the only focused open candidate has not merged and still needs maintainer policy alignment.

## What This Changes

Add an operator-controlled TTL/expiry policy for outbound delivery-queue recovery so stale queued messages are not replayed after gateway restarts.

## Best Possible Solution

Keep this as the canonical TTL policy issue, then land or replace the focused candidate with a backward-compatible expiry policy shared by startup recovery and reconnect drains after maintainers decide the config shape, default, expire action, and inbound scope.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery with retryCount 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main treats it as eligible and gateway startup calls that path without a TTL option.

## Solution Assessment

Unclear as proposed. The outbound recovery boundary is right, but the TTL default, config shape, expire action, reconnect-drain behavior, and inbound scope need maintainer policy alignment before this can close.
