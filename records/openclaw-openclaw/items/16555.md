## Summary

Keep open: current main and the latest tagged release still lack an outbound delivery-queue age/TTL policy, while related implementation PRs are closed/unmerged and the live issue is explicitly marked for maintainer/product review.

## What This Changes

The issue requests configurable TTL/expiry handling for delivery queue messages so stale outbound entries are not replayed after gateway restart.

## Best Possible Solution

Add a narrow, current-schema outbound delivery freshness policy that preserves existing default behavior unless maintainers choose a new default, with coverage for startup recovery, reconnect drain, storage state, config acceptance, and logging.

## Reproduction Assessment

Yes from source inspection: pending outbound delivery entries older than any desired TTL are still loaded and replay-eligible because current recovery/drain code has no age cutoff. I did not run a live gateway reproduction because this review is read-only and tests would write artifacts.

## Solution Assessment

Unclear as a product decision: the narrow owner boundary is `src/infra/outbound` plus `messages` config, but maintainers still need to choose whether TTL is opt-in only, which actions are supported, and whether inbound freshness belongs in a separate follow-up.
