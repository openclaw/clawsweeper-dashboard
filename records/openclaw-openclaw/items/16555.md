## Summary

Keep open. Current main still lacks an outbound delivery-queue TTL/expiry policy: startup recovery and reconnect drains use retry/backoff/max-retry state, not queued age, and strict messages config still has no `messages.delivery` surface. The production Telegram/systemd reproduction remains applicable, while related SQLite/outbox TTL PRs are closed and unmerged rather than canonical fixes.

## What This Changes

The issue requests configurable TTL/expiry for persisted outbound delivery queue messages so stale entries are skipped, deleted, or quarantined instead of replayed after gateway restarts.

## Best Possible Solution

Implement a narrow core expiry policy for the existing file-backed outbound delivery queue: optional strict `messages.delivery` config, stale-entry expiry in both startup recovery and reconnect drains, no behavior change when unset, clear logs, generated config docs, and focused storage/policy/recovery/config tests.

## Reproduction Assessment

Yes. Seed a delivery-queue JSON entry with an `enqueuedAt` far in the past and otherwise eligible retry/backoff state, then invoke startup recovery or a matching reconnect drain; current main will consider it for delivery because no age check exists.

## Solution Assessment

Yes. A direct TTL/expiry policy on the current file-backed queue is the narrowest maintainable fix for the reported stale replay behavior; the broader SQLite/outbox rewrites are useful prior art but are closed and unmerged.
