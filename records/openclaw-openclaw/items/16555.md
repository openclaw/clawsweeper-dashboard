## Summary

Keep open. Current main still has no age or expiry gate for outbound delivery-queue recovery, and the focused implementation path remains open with conflict, proof, and TTL policy blockers rather than merged.

## What This Changes

The issue requests configurable expiry for stale outbound delivery queue entries so gateway restart recovery does not replay old messages.

## Best Possible Solution

Land or replace a focused outbound recovery expiry policy that skips or quarantines stale queued replies while preserving recent crash recovery, reconnect drains, budget deferrals, and uncertain-send reconciliation.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with `retryCount: 0` and no `lastAttemptAt`, then run `recoverPendingDeliveries`; current main treats it as eligible and gateway startup calls that path.

## Solution Assessment

Unclear as proposed. The outbound recovery boundary is right, but the operator-facing config, default TTL, expiry action, reconnect behavior, and uncertain-send handling need maintainer alignment before this can close.
