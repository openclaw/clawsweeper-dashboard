## Label Justifications

- `P2`: Stale recovery causes recurring duplicate or out-of-context channel delivery, but it does not make the core runtime unusable.
- `impact:message-loss`: The queue policy determines whether outbound messages are duplicated, wrongly replayed, or suppressed during recovery.

## Summary

Current main and the latest release still lack a queued-age expiry policy, while the focused implementation PR remains open. That PR is narrower than the request—it handles only never-attempted entries during startup—so maintainers still need to settle shared-path semantics, configuration, and the canonical storage owner.

## What This Changes

Add a configurable age limit and terminal action for stale persisted outbound messages so gateway recovery does not replay obsolete deliveries.

## Best Possible Solution

Define one auditable age and terminal-state policy at the canonical queue policy or storage boundary, apply it consistently to startup and reconnect consumers, and keep operator configuration minimal while moving durable delivery state toward SQLite.

## Reproduction Assessment

Yes at source level: production logs show old pending entries being recovered, and current source routes pending entries through startup and reconnect consumers without a queued-age terminal policy. This read-only review did not run a live crash/restart reproduction.

## Solution Assessment

No, neither the issue's broad three-action configuration nor the linked PR's startup-only first-attempt cutoff is yet the best complete solution. The maintainable fix is a shared expiry invariant with explicit retry semantics, auditable terminal state, and an agreed storage owner.
