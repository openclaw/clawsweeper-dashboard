## Summary

Keep open: current main and latest release still replay pending outbound delivery entries by retry/max-retry policy without any queued-age TTL or accepted `messages.delivery.*` expiry config, and the duplicate report plus focused PR attempts now route back to this issue.

## What This Changes

Request to add configurable TTL/expiry handling for outbound delivery queue messages so stale entries are not dumped into channels after gateway restart.

## Best Possible Solution

Keep this issue as the canonical outbound stale-replay policy item, then land a narrow current-main fix that expires stale pending outbound entries after re-reading/claiming them and aligns config/schema/docs/tests with the chosen default.

## Reproduction Assessment

Yes, at source level. Seed an old pending outbound entry with `retryCount: 0` and no `lastAttemptAt`; current recovery treats it as eligible and no current config/schema field can set an age cutoff.

## Solution Assessment

Unclear as a final code shape. A core outbound recovery expiry policy is the right layer, but maintainers still need to choose the default, config surface, and expiry action before a safe implementation PR.
