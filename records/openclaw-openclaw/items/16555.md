- `impact:message-loss`: The report is about stale or duplicate channel messages being replayed from the outbound delivery queue after gateway restart.

## Summary

Keep open: current main still replays pending outbound delivery queue entries without queued-age expiry, the strict messages config has no TTL/expire-action surface, and the focused implementation candidate remains open rather than merged.

## What This Changes

Request a configurable TTL/expiry policy for outbound delivery queue entries so stale pending messages are not replayed after gateway restart.

## Best Possible Solution

Keep this as the canonical policy issue, then land or replace the open candidate with a backward-compatible expiry policy shared by startup recovery and reconnect drains after maintainers decide the default, config shape, expire action, and inbound scope.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with retryCount 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main treats it as eligible and gateway startup calls that path.

## Solution Assessment

Unclear as proposed. The outbound recovery boundary is correct, but the requested TTL default, config shape, expire action, and inbound scope are product-policy decisions before the open candidate or a replacement can close this.
