## Summary

Keep open. Current main still replays pending outbound delivery entries on startup without an age cutoff, the requested config surface is absent, and the focused implementation PR remains open with merge/proof/policy blockers.

## What This Changes

The issue requests configurable TTL/expiry handling for persisted delivery-queue messages so stale outbound messages are not replayed after gateway restart.

## Best Possible Solution

Land or replace the focused outbound recovery expiry policy so stale never-attempted queued replies are skipped or quarantined while legitimate recent crash recovery remains intact, with config/docs added only if maintainers choose an operator-facing TTL surface.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with `retryCount: 0` and no `lastAttemptAt`, then run `recoverPendingDeliveries`; current main treats it as eligible and gateway startup calls that recovery path.

## Solution Assessment

Unclear as proposed. The shared outbound recovery boundary is the right place, but the TTL default, action vocabulary, config surface, and open PR repair/proof need maintainer alignment before this can close.
