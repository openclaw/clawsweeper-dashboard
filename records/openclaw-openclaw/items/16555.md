## Summary

Keep open: current main and latest tagged release still replay pending outbound delivery entries without an age/TTL policy, and the related duplicate issue plus stale PR point back to this issue as the canonical remaining config/default decision.

## What This Changes

Issue requests a configurable outbound delivery-queue TTL or expiry action to prevent stale queued messages from replaying after gateway restart.

## Best Possible Solution

Define a closed outbound expiry policy at the current SQLite-backed delivery queue recovery/reconnect boundary, wire it through strict config and docs if approved, and cover startup plus reconnect drain with focused tests.

## Reproduction Assessment

Yes, at source level: seed an old pending outbound delivery with retryCount 0 and no lastAttemptAt, then recoverPendingDeliveries treats it as eligible without comparing age against enqueuedAt.

## Solution Assessment

Partly: the delivery queue recovery/reconnect boundary is the right owner, but the proposed config keys, default TTL, and expireAction vocabulary need maintainer approval.
