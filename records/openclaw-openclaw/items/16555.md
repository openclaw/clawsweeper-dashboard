## Summary

Keep open: current main and latest release still lack an outbound delivery-queue TTL/expiry policy, and the focused implementation candidate at https://github.com/openclaw/openclaw/pull/54900 is still open rather than merged.

## What This Changes

Requests configurable TTL/expiry handling for outbound delivery queue messages so stale queued replies are not replayed after gateway restarts.

## Best Possible Solution

Land or replace the focused outbound expiry fix with maintainer-approved defaults/config semantics, tests for startup recovery and reconnect drains, and clear separation from any broader inbound freshness follow-up.

## Reproduction Assessment

Yes at source level. Seed an old pending outbound delivery with retryCount 0 and no lastAttemptAt, then startup recovery treats it as eligible because current main has no age/expiry check before draining it.

## Solution Assessment

Unclear until maintainers decide the policy shape. A core outbound expiry policy is the right layer, but the default, config surface, expire action, and whether inbound freshness belongs in the same change still need alignment.
