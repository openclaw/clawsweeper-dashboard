## Summary

Keep open. Current main still has no outbound delivery-queue age cutoff or `messages.delivery` config surface, and the focused implementation candidate remains open and unmerged.

## What This Changes

Request a configurable TTL/expiry policy for outbound delivery-queue entries so stale queued messages are not replayed after gateway restart.

## Best Possible Solution

Land or replace the focused shared outbound recovery fix so old stale pending deliveries are skipped or quarantined while recent crash recovery remains intact, then document any operator-facing config maintainers choose to expose.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with `retryCount: 0` and no `lastAttemptAt`, then run `recoverPendingDeliveries`; current main treats it as eligible and calls the delivery path.

## Solution Assessment

Unclear as proposed. The shared outbound recovery boundary is the right place to solve this, but the user-facing TTL default and expiry action need maintainer alignment, and the open implementation candidate still needs repair before merge.
