## Summary

Keep open. Current main still has no outbound delivery-queue age or expiry gate, so old pending entries can be replayed on gateway startup; the focused implementation at https://github.com/openclaw/openclaw/pull/54900 is still open and has review/proof/policy blockers rather than being a merged fix.

## What This Changes

Request to add configurable TTL or expiry handling for outbound delivery queue messages so stale pending entries are skipped, deleted, or quarantined during gateway recovery.

## Best Possible Solution

Land or replace the focused outbound recovery expiry policy so stale queued replies are skipped or quarantined without regressing legitimate recent crash recovery, and split any broader inbound freshness policy into a deliberate follow-up if maintainers want that scope.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with retryCount: 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main treats it as eligible and gateway startup calls that recovery path.

## Solution Assessment

Unclear as proposed. The outbound recovery boundary is the right place, but the operator-facing TTL surface, default, expiry action, reconnect behavior, and uncertain-send reconciliation need maintainer alignment before this can close.
