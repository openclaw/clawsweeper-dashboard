## What This Changes

Request to add configurable TTL/expiry for outbound delivery queue messages so stale queued replies are not replayed after gateway restart.

## Best Possible Solution

Define a closed outbound expiry policy at the current SQLite-backed delivery queue boundary, wire it through strict config and docs if maintainers approve the new surface, and cover startup recovery plus reconnect drain with focused tests.

## Reproduction Assessment

Yes, at source level. Seed an old pending outbound delivery entry with retryCount 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main considers first replay after crash eligible and does not compare Date.now() with enqueuedAt.

## Solution Assessment

Unclear until maintainers choose the policy. A narrow outbound expiry policy in delivery queue recovery is the right owner boundary, but the default, threshold, and expireAction vocabulary need approval before implementation.
