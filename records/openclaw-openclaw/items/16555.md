## Label Justifications

- `P2`: The issue is a normal-priority delivery reliability improvement with clear user impact but a bounded outbound queue surface and an unresolved product-policy choice.
- `impact:message-loss`: The reported behavior can replay stale channel messages after restart, causing duplicate or stale outbound delivery in messaging integrations.

## Summary

Keep open: current main and latest release still recover pending outbound delivery entries without any queued-age or TTL policy, and the related implementation attempts are closed unmerged rather than canonical fixes.

## What This Changes

Requested configurable TTL/expiry handling for persisted delivery queue messages so stale outbound entries are not replayed after gateway restart.

## Best Possible Solution

Define a closed outbound expiry policy at the current SQLite-backed delivery queue recovery and reconnect-drain boundary, then wire approved config, docs, and focused tests around that policy.

## Reproduction Assessment

Yes, at source level: an old pending outbound entry with `retryCount: 0` and no `lastAttemptAt` remains immediately eligible because recovery checks retry/backoff state, not age against `enqueuedAt`. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

Partly: the recovery and reconnect-drain boundary is the right owner for outbound expiry, but the proposed config key, default TTL, and expire action vocabulary need maintainer approval before implementation.
