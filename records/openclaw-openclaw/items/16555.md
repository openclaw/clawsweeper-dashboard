## Summary

Keep open. Current main still replays pending outbound delivery entries on startup without an age cutoff, the requested config surface is absent, and the focused implementation PR is still open with review/proof/policy blockers.

## What This Changes

Add a TTL/expiry policy for delivery-queue messages so stale queued outbound entries are skipped or quarantined instead of replayed after gateway restart.

## Best Possible Solution

Land or replace a focused outbound recovery expiry policy that prevents stale queued replies from replaying while preserving legitimate recent crash recovery, with config/docs added only if maintainers choose an operator-facing TTL surface.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with retryCount: 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main treats it as eligible and gateway startup calls that recovery path.

## Solution Assessment

Unclear as proposed. The shared outbound recovery boundary is right, but the default TTL, action vocabulary, config surface, and relationship to the inbound freshness request need maintainer alignment before this can close.
