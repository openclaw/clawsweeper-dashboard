## Summary

Keep open. Current main still replays first-attempt pending delivery-queue entries without an age cutoff, the requested config surface is absent, and the focused implementation candidate is still open and unmerged.

## What This Changes

Request a configurable TTL/expiry policy for outbound delivery-queue messages so stale pending entries are skipped, deleted, or moved to failed during recovery.

## Best Possible Solution

Land or replace the focused recovery-policy fix so old stale pending deliveries are quarantined or skipped while recent crash recovery remains intact, then document any operator-facing config that maintainers choose to expose.

## Reproduction Assessment

Yes. A high-confidence source path is to seed an old pending queue entry with `retryCount: 0` and no `lastAttemptAt`, then run `recoverPendingDeliveries`; current main treats it as eligible and calls the delivery path.

## Solution Assessment

Unclear as proposed. The shared outbound recovery boundary is the right place to solve this, but the TTL/default/action surface needs maintainer choice and the open candidate still needs repair before merge.
