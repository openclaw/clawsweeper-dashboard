## Summary

Keep open. Current main still lacks outbound delivery-queue TTL/expiry: startup recovery and reconnect drains replay pending JSON entries using retry/backoff/max-retry rules, not queued age, and strict messages config still has no `messages.delivery` surface. The production Telegram/systemd reproduction and closed SQLite/outbox PRs remain useful context, but no merged canonical fix is present.

## What This Changes

Add optional TTL/expiry controls for file-backed outbound delivery queue messages so stale pending entries are not replayed after gateway restart.

## Best Possible Solution

Implement an optional, documented expiry policy in the existing file-backed outbound delivery queue, enforcing it in both startup recovery and reconnect drains with focused regression coverage and no SQLite/outbox rewrite.

## Reproduction Assessment

Yes. Seed a `delivery-queue/*.json` entry with an old `enqueuedAt` and otherwise eligible retry/backoff state, then invoke startup recovery or a matching reconnect drain; source inspection shows current main will consider it for delivery because no age check exists.

## Solution Assessment

Yes. A narrow optional TTL/expiry policy on the current file-backed queue is the maintainable fix for stale replay; the broader SQLite/outbox PRs are prior art but are closed and unmerged.
