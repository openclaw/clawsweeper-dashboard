## Summary

Keep open. Current main still has no outbound delivery queue TTL/expiry policy, startup recovery can replay old first-attempt entries, and the focused implementation remains open with review/proof/policy blockers.

## What This Changes

The issue requests an optional TTL/expiry policy for outbound delivery queue entries so stale queued messages are not replayed after gateway restart.

## Best Possible Solution

Land or replace a focused outbound recovery expiry policy that prevents stale queued replies from replaying while preserving legitimate recent crash recovery, with config/docs added only if maintainers choose an operator-facing TTL surface.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with retryCount: 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main treats it as eligible and gateway startup calls that recovery path.

## Solution Assessment

Unclear as proposed. The shared outbound recovery boundary is the right place, but the default TTL, expiry action, config surface, and interaction with reconnect/budget deferral need maintainer alignment before this can close.
