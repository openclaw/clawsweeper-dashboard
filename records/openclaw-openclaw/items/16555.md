## Summary

Keep open. Current main still has no delivery-queue age cutoff or `messages.delivery` config surface, and the focused open implementation PR remains unmerged with review feedback to address.

## What This Changes

Add a configurable TTL/expiry policy for outbound delivery queue entries so stale pending messages are skipped or quarantined instead of replayed after gateway restart.

## Best Possible Solution

Land or replace the focused outbound recovery fix so stale pending deliveries are skipped or quarantined while legitimate recent crash recovery remains intact, with docs/schema updates if maintainers choose an operator-facing config surface.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with `retryCount: 0` and no `lastAttemptAt`, then run `recoverPendingDeliveries`; current main treats it as eligible and the gateway startup path calls recovery with the real outbound delivery function.

## Solution Assessment

Unclear as proposed. The shared outbound recovery boundary is the right place to solve this, but the TTL default/action and the open PR's attempted-send predicate need maintainer alignment and repair before merge.
