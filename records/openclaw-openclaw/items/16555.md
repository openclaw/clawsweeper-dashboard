## Summary

Keep open. Current main still replays old pending outbound delivery entries without an age or expiry gate, the messages config has no delivery TTL surface, and the focused implementation at https://github.com/openclaw/openclaw/pull/54900 remains open rather than merged.

## What This Changes

Add a configurable TTL or expiry policy for outbound delivery-queue messages so stale queued replies are not replayed after gateway restart.

## Best Possible Solution

Land or replace the focused outbound recovery expiry policy so stale queued replies are skipped or quarantined without regressing legitimate recent crash recovery, and split inbound freshness into a deliberate follow-up if maintainers want that broader scope.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with `retryCount: 0` and no `lastAttemptAt`, then run `recoverPendingDeliveries`; current main treats it as eligible and gateway startup calls that recovery path.

## Solution Assessment

Unclear as proposed. The outbound recovery boundary is the right place, but the operator-facing TTL surface, default, expiry action, reconnect behavior, and uncertain-send reconciliation need maintainer alignment before this can close.
