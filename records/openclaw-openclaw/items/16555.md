## Summary

Keep open: current main still has no queued-age expiry for outbound delivery recovery, and the remaining TTL/default/config policy is still being handled through an open implementation candidate rather than a merged fix.

## What This Changes

Request configurable TTL/expiry for delivery queue messages so stale pending outbound messages are not replayed after gateway restarts.

## Best Possible Solution

Keep this as the canonical TTL policy issue, then land or replace the open candidate with a backward-compatible outbound expiry policy shared by startup recovery and reconnect drains after maintainers decide the config shape, default, expire action, and inbound scope.

## Reproduction Assessment

Yes, at source level. Seed an old pending delivery entry with retryCount 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main treats it as eligible and gateway startup calls that path.

## Solution Assessment

Unclear as proposed. The outbound recovery boundary is correct, but the TTL default, config shape, expire action, reconnect behavior, and inbound scope need maintainer policy alignment before this can close.
