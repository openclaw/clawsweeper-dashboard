## Summary

Keep open. Current main still has the file-backed outbound delivery queue replay path without queued-age expiry, and the strict messages config still has no `messages.delivery` TTL/expire-action surface; the related SQLite/outbox attempts are closed and unmerged prior art, not a canonical fix.

## What This Changes

Add an optional TTL/expiry policy for outbound delivery-queue messages so stale pending entries are not replayed after gateway restart or reconnect recovery.

## Best Possible Solution

Implement an optional, documented expiry policy on the existing file-backed outbound delivery queue, enforced in both startup recovery and reconnect drains while preserving the default no-TTL behavior.

## Reproduction Assessment

Yes. Seed a `delivery-queue/*.json` entry with an old `enqueuedAt` and otherwise eligible retry/backoff state, then invoke startup recovery or a matching reconnect drain; source inspection shows current main will consider it for delivery because no age check exists.

## Solution Assessment

Yes. A narrow optional TTL/expiry policy on the current file-backed queue is the maintainable path; the broader SQLite/outbox rewrites are closed and unmerged, so they should be treated as prior art rather than the active fix.

## Repair Work Prompt

Implement optional TTL/expiry for the existing file-backed outbound delivery queue. Add a backwards-compatible `messages.delivery` config surface, with no TTL by default unless maintainers have already chosen otherwise, and a closed expire-action enum that can safely move expired entries to failed/quarantine or skip/delete only if explicitly configured. Enforce the policy in both `recoverPendingDeliveries` and `drainPendingDeliveries`, using `enqueuedAt` for age, and log concise expiry summaries without logging message bodies or recipients. Do not replace the file queue with SQLite/outbox and do not change unrelated delivery semantics. Add focused tests for startup recovery, reconnect drain, storage/policy handling, strict config acceptance/rejection, and generated config docs if the schema output changes. Add a user-facing changelog entry for the new config behavior.
