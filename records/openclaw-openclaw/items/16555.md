## Summary

Keep open: current main and v2026.6.6 still have no outbound delivery-queue age gate or accepted config surface for the requested TTL/expiry policy, while the related implementation PRs are closed unmerged and the remaining default/action choice needs maintainer direction.

## What This Changes

The issue requests configurable TTL/expiry handling for outbound delivery queue messages so stale queued replies are not replayed after gateway restart.

## Best Possible Solution

Define one closed outbound expiry policy at the delivery-queue owner boundary, apply it consistently to startup recovery and reconnect drains, then update strict config, generated config docs, and focused queue tests after maintainers choose the default and action vocabulary.

## Reproduction Assessment

Yes, at source level. Seed an old pending outbound delivery with retryCount 0 and no lastAttemptAt, then run recoverPendingDeliveries; current main treats it as eligible and has no age predicate, with production comments corroborating the restart replay symptom.

## Solution Assessment

Unclear as a final product shape. A narrow outbound expiry policy is the right owner boundary, but the config key, default behavior, expire action, and whether inbound freshness belongs in this issue need maintainer decisions first.
