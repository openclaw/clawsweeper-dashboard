## Summary

Keep open. Current main still has no outbound delivery-queue age cutoff or `messages.delivery` config surface, and the focused stale-recovery PR is still open and unmerged, so the reported restart replay problem remains active work.

## What This Changes

Add configurable TTL/expiry handling for outbound delivery queue recovery so stale pending messages are not replayed after gateway restart.

## Best Possible Solution

Land a narrow shared outbound recovery freshness policy that quarantines or skips old stale queued entries while preserving recent crash recovery, with config/schema/docs only if maintainers choose an operator-facing knob.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed an old pending delivery entry with `retryCount: 0` and no `lastAttemptAt`, then run startup recovery; current main treats it as eligible and calls the delivery path.

## Solution Assessment

Unclear as written. The shared outbound recovery boundary is the right place to solve this, but the exact TTL/default/action surface needs maintainer choice and the current open implementation candidate still needs repair or replacement.
