## What This Changes

This PR adds optional AbortSignal handling to QMD memory search and the shared QMD CLI runner, with regression tests for direct, grouped, and mcporter subprocess cancellation.

## Best Possible Solution

Land the QMD subprocess abort coverage after maintainers accept optional AbortSignal semantics on the exported QMD helper and normal gates are resolved; if that helper should stay narrower, keep the same route coverage behind a memory-core-owned wrapper.

## Reproduction Assessment

Yes at source level. Current main passes a deadline signal into memory managers, but QMD ignores it before spawning qmd/mcporter subprocesses; the PR discussion also includes terminal proof that a real child process dies promptly on abort.

## Solution Assessment

Yes, with maintainer acceptance of the exported helper contract. Adding cancellation at the spawn boundary and threading the existing signal through direct, grouped, and mcporter routes is narrower than changing timeout policy, and the separate configured-timeout work remains adjacent rather than a replacement.
