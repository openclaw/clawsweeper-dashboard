## Summary

Keep open. Current main still has the reported Slack custom-invoke media-context drop, and this PR is a narrow, source-matched fix with regression coverage and a changelog entry; the remaining work is maintainer review plus rebase/exact-head validation, not closure or a separate repair job.

## What This Changes

This PR augments Slack action invocation context with `mediaLocalRoots` and `mediaReadFile`, forwards that context through both custom invoke and fallback runtime paths, adds a Slack `upload-file` regression test, and records the user-facing fix in `CHANGELOG.md`.

## Best Possible Solution

Land this PR, or an equivalent Slack-wrapper change, after rebase and exact-head validation so Slack custom invokes and fallback runtime actions receive the same core-computed media access while keeping the existing local-media guard intact.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to route Slack `upload-file` through `createSlackActions(..., { invoke })` with `mediaLocalRoots` and `mediaReadFile` on the channel action context; current main forwards only raw `toolContext` to that custom invoke.

## Solution Assessment

Yes. Building the augmented context once in `createSlackActions` is the narrowest maintainable fix because it preserves core media-root computation and Slack upload validation instead of widening local path policy.
