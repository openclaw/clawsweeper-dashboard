## Summary

Keep open. Current main still drops Slack action media access on the custom invoke path, and the PR is a narrow, reviewable fix with a regression test and changelog entry; I did not find a blocking correctness or security issue in the proposed patch.

## What This Changes

The PR merges Slack action `toolContext` with `mediaLocalRoots`/`mediaReadFile` for both custom and fallback invokes, adds a custom-invoke `upload-file` regression test, and records the Slack fix in the changelog.

## Best Possible Solution

Land this PR, or an equivalent Slack-wrapper fix, after exact-head validation. The end state should keep core-owned media-root resolution and Slack's existing local-media guard intact while giving custom invoke and fallback runtime paths the same trusted media context.

## Reproduction Assessment

Yes. A source-level reproduction is to call `createSlackActions` with `options.invoke` for `upload-file` while the channel action context contains `mediaLocalRoots` and `mediaReadFile`; current main forwards only raw `toolContext`, so the Slack upload runtime sees no trusted roots.

## Solution Assessment

Yes. Fixing the Slack wrapper boundary is the narrowest maintainable solution because it forwards core-computed media access without widening path validation, upload authorization, or token handling.
