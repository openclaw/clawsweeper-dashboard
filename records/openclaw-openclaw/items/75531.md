## Summary

Keep open. Current main still drops Slack action media access on the custom invoke path, while this PR applies a narrow Slack-wrapper fix with a focused regression test and changelog entry; the remaining path is normal maintainer review and exact-head CI.

## What This Changes

This PR augments Slack action context with `mediaLocalRoots` and `mediaReadFile`, forwards it through custom invoke and fallback runtime paths, adds a custom-invoke `upload-file` regression test, and records the fix in the changelog.

## Best Possible Solution

Land this PR, or an equivalent Slack-wrapper fix, once exact-head validation is green so Slack custom invoke and fallback runtime paths receive the same core-computed media access while keeping the existing local-media guard intact.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to route Slack `upload-file` through `createSlackActions(..., { invoke })` with `mediaLocalRoots` and `mediaReadFile` on the channel action context; current main forwards only raw `toolContext` to that custom invoke.

## Solution Assessment

Yes. Building the augmented context at `createSlackActions` is the narrowest maintainable fix because it preserves core media-root computation and Slack upload validation instead of widening local path policy.
