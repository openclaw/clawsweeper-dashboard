## Summary

Keep open. Current main still routes Slack `user:<id>` proactive sends through `conversations.open` before `chat.postMessage`, while the report shows direct `chat.postMessage(channel=<USER_ID>)` succeeds and the failure is associated with `missing_scope`; this is a real Slack plugin delivery gap, not an implemented or duplicate cleanup item.

## What This Changes

The issue requests that proactive Slack DM delivery work when the same bot can already send a direct Slack `chat.postMessage` to the target user ID.

## Best Possible Solution

Fix the Slack plugin sender so text/block-only proactive DM sends to `user:<id>` can deliver via direct `chat.postMessage(channel=U...)` when `conversations.open` is unavailable or unnecessary, while retaining D-channel resolution for media uploads and any path that requires a conversation id.

## Reproduction Assessment

Yes. A deterministic unit repro is to mock `client.conversations.open` to reject with Slack `missing_scope`, mock `client.chat.postMessage` to succeed, and call `sendMessageSlack('user:U078T4P5FHP', 'hello', ...)`; current main rejects before posting.

## Solution Assessment

Yes, a narrow Slack plugin change is the best path. This should stay in `extensions/slack` because core already delegates outbound delivery to the channel plugin, and the Slack-specific distinction is text/block post behavior versus media upload channel-id requirements.
