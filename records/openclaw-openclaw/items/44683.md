## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Trigger rollouts when rendered config or secrets change:** `charts/openclaw/templates/deployment.tpl:15-18`
  - body: The pod template has no checksum annotation for the rendered ConfigMap or Secret, so `helm upgrade` can change config or secret values without creating a new ReplicaSet. Because config is copied only at startup and provider keys are env vars, upgrades can report success while the running pod keeps stale config and credentials.
  - confidence: 0.9
- **[P2] Keep service.port synchronized with gateway.port:** `charts/openclaw/values.yaml:84`
  - body: The chart wires container ports and probes to `.Values.service.port`, but the default OpenClaw config hardcodes `"port": 18789`. If an operator overrides `service.port`, Kubernetes probes and routes to the new port while the gateway still listens on 18789.
  - confidence: 0.9
- **[P2] Avoid duplicating the chart name in Helm resource names:** `charts/openclaw/templates/_helpers.tpl:9`
  - body: The documented install command uses release name `openclaw`, but `openclaw.fullname` renders resources as `openclaw-openclaw` and `openclaw-openclaw-home-pvc`. That diverges from the current manifest names and can create a second deployment and empty PVC during adoption.
  - confidence: 0.86
- **[P2] Fail when an external Secret name is omitted:** `charts/openclaw/templates/secret.tpl:1`
  - body: When `secret.create=false` and `secret.existingSecret` is empty, the helper still returns the generated secret name but no Secret is created. The Deployment then references a missing `OPENCLAW_GATEWAY_TOKEN` Secret and pods fail at startup instead of Helm failing clearly.
  - confidence: 0.88
- **[P2] Guard multi-replica installs with a ReadWriteOnce PVC:** `charts/openclaw/values.yaml:4`
  - body: The chart exposes `replicaCount`, but the default persistent volume is `ReadWriteOnce`. Setting replicas above 1 can leave pods stuck pending on multi-node clusters, and `strategy: Recreate` does not prevent multiple steady-state replicas.
  - confidence: 0.78
- **[P2] Do not advertise Service exposure with loopback bind:** `charts/openclaw/templates/NOTES.txt:2-8`
  - body: `NOTES.txt` prints NodePort and LoadBalancer access instructions whenever `service.type` changes, but the default config still binds the gateway to loopback. Those generated URLs will refuse connections unless the chart validates or clearly requires a non-loopback bind for exposed Service modes.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The chart adds no new third-party code execution, but it currently teaches users to pass gateway and provider secrets through Helm values.

Concerns:

- **[medium] Prefer existing Secrets over CLI-passed keys:** `charts/openclaw/README.md:10`
  - body: The README install example passes the gateway token and provider key with `helm --set`, and the chart renders those values into a Helm-managed Secret. That can expose credentials through shell history, process arguments, rendered output, and Helm release state; prefer `secret.existingSecret` or add explicit safe-secret guidance before merge.
  - confidence: 0.82
