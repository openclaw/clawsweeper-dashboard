## Summary

Keep open: current main still only has the Kustomize/raw manifest Kubernetes path, so the Helm chart is not obsolete, but this branch should not merge yet because source review shows blocking chart upgrade, naming, port, secret, proof, and security issues.

## What This Changes

The branch adds a first-party Helm chart under `charts/openclaw` with templated Deployment, Service, PVC, Secret, ConfigMap, notes, README, values, and `.helmignore`.

## Best Possible Solution

Land a maintainer-approved Helm chart only after it matches the current Kubernetes manifests and docs, handles upgrades, ports, naming, replicas, and secrets safely, and includes real install/upgrade proof from a cluster.

## Reproduction Assessment

Yes for the review blockers from source inspection: overriding `service.port` leaves `gateway.port` at `18789`, the default `openclaw` release renders duplicate names, and ConfigMap/Secret value changes do not alter the pod template. I did not run a live cluster, so the PR still needs real install and upgrade proof.

## Solution Assessment

No. A Helm chart can be a reasonable deployment path, but this branch needs safer chart conventions, secret handling, source guards, and maintainer product approval before it is the best core solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Trigger rollouts when config or secrets change:** `charts/openclaw/templates/deployment.tpl:16-17`
  - body: The pod template has only static labels, so a Helm upgrade that changes `config.openclawJson`, `config.agentsMd`, or chart-managed secrets can report success without creating a new ReplicaSet. Those inputs are only consumed at container startup, so users keep running stale config or tokens until they manually restart the Deployment.
  - confidence: 0.9
- **[P2] Preserve existing names for the default release:** `charts/openclaw/templates/_helpers.tpl:8`
  - body: The documented install uses release name `openclaw`, but this helper renders resources as `openclaw-openclaw`. That is not equivalent to the existing manifests and would create a second Deployment/Service/PVC instead of adopting the current `openclaw` objects and data.
  - confidence: 0.88
- **[P2] Keep the Service port in sync with the gateway config:** `charts/openclaw/values.yaml:23`
  - body: Changing `service.port` updates the container port and probes, but the embedded `openclaw.json` still tells the gateway to listen on `18789`. Operators who override the chart port get failing probes and an unreachable Service unless they also hand-edit JSON.
  - confidence: 0.87
- **[P2] Fail when an existing Secret name is required:** `charts/openclaw/templates/secret.tpl:1`
  - body: With `secret.create=false` and no `secret.existingSecret`, the chart renders a Deployment that references a generated Secret name that the chart never creates. Helm should fail during rendering instead of letting pods enter `CreateContainerConfigError`.
  - confidence: 0.86
- **[P3] Guard multi-replica installs with ReadWriteOnce persistence:** `charts/openclaw/values.yaml:4`
  - body: The chart exposes `replicaCount`, but the default persistent volume access mode is `ReadWriteOnce`. Setting replicas above one can leave pods unschedulable or sharing a single-node volume unexpectedly, so the chart should validate or clearly constrain that combination.
  - confidence: 0.7

## Security Review

Status: needs_attention

Summary: Needs attention: the documented Helm `--set` secret flow would put gateway and provider secrets into shell and Helm release surfaces instead of the current temp-file deploy flow.

Concerns:

- **[medium] Avoid passing provider secrets through Helm values:** `charts/openclaw/README.md:8`
  - body: The README recommends inline `--set secret.gatewayToken` and provider API keys, and the chart renders those values into a Helm-managed Secret. Helm exposes release values/manifests through normal `helm get` commands and warns that generated Secrets can appear in output, so prefer an existing Secret or a safer secret-input pattern before merging this as a first-party deployment path.
  - confidence: 0.78
