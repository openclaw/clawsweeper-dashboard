## Summary

Keep open. Current main does not have the requested Helm chart, and the proposed chart still has blocking correctness and security-model issues before it can become an owned OpenClaw deployment surface.

## What This Changes

The PR adds an additive `charts/openclaw/` Helm chart with chart metadata, values, templates, install notes, and README for deploying OpenClaw on Kubernetes alongside the existing raw manifests.

## Best Possible Solution

Revise the chart into an owner-approved, tested deployment surface that preserves the raw-manifest resource and data contract, uses an external-Secret-first credential flow, restarts pods on config and secret changes, validates unsupported value combinations, updates docs and changelog, and clearly documents whether Helm is supported in this repository.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the PR defects: render or inspect the chart with changed `service.port`, `secret.create=false` without `secret.existingSecret`, NodePort/LoadBalancer service types, changed config/secret values, and `replicaCount > 1` with the default ReadWriteOnce PVC.

## Solution Assessment

Is this the best way to solve the issue? No, not as submitted. A Helm chart may be reasonable, but the current patch is not the narrowest maintainable solution until the chart contract, secret handling, rollout behavior, and ownership decision are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Trigger rollouts for chart config and secret changes:** `charts/openclaw/templates/deployment.tpl:15-17`
  - body: The pod template has only static labels and no checksum annotations for the rendered ConfigMap or Secret. Since the init container copies config into the PVC and secrets are injected as env vars, `helm upgrade` can succeed while the running pod keeps old config or tokens until a manual restart.
  - confidence: 0.9
- **[P1] Avoid storing credentials in Helm release values:** `charts/openclaw/README.md:8-11`
  - body: The install docs pass `secret.gatewayToken` and provider keys through `--set`, and the chart models those credentials under values. Helm stores release values, so this can retain long-lived gateway tokens and provider keys outside the intended Secret-only workflow.
  - confidence: 0.86
- **[P2] Default the chart to a current OpenClaw image:** `charts/openclaw/values.yaml:6-9`
  - body: The chart hardcodes `2026.3.12`/`2026.3.12-slim` while current main is already on `2026.4.30` with latest release `v2026.4.29`. Merging this now would make fresh Helm installs deploy an old gateway by default.
  - confidence: 0.86
- **[P2] Preserve existing Kubernetes resource names:** `charts/openclaw/templates/_helpers.tpl:8-9`
  - body: The fullname helper always renders `<release>-<chart>`, so the documented release name `openclaw` creates resources like `openclaw-openclaw` and a new PVC. That breaks the stated equivalence with the existing `openclaw` objects and prevents an in-place migration from the raw manifests.
  - confidence: 0.91
- **[P2] Keep the gateway config port tied to service.port:** `charts/openclaw/values.yaml:83-84`
  - body: `config.openclawJson` hardcodes gateway port `18789` while the chart exposes `service.port` for the container, probes, and Service. Overriding `service.port` renders Kubernetes resources for one port while the gateway listens on another.
  - confidence: 0.88
- **[P2] Require existingSecret when secret creation is disabled:** `charts/openclaw/templates/secret.tpl:1`
  - body: When `secret.create=false` and `secret.existingSecret` is empty, the helper still generates a Secret name but this template creates no Secret. The Deployment then references a required token Secret that does not exist, leaving pods in `CreateContainerConfigError` instead of failing at render time.
  - confidence: 0.88
- **[P2] Do not advertise unreachable Service URLs:** `charts/openclaw/templates/NOTES.txt:2-8`
  - body: The notes print NodePort and LoadBalancer access instructions, but the default config keeps the gateway bound to loopback. Those Service paths route to the pod IP, so the rendered instructions lead operators to a dead endpoint unless the chart also changes or warns about the bind setting.
  - confidence: 0.82
- **[P3] Guard replicaCount with the default RWO PVC:** `charts/openclaw/values.yaml:4`
  - body: The chart exposes `replicaCount` while the default PVC access mode is `ReadWriteOnce`. Setting more than one replica can leave extra pods pending on multi-node clusters, and `strategy: Recreate` does not prevent multiple steady-state replicas.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `charts/openclaw/Chart.yaml:1`
  - body: This adds a user-facing deployment feature, but the PR does not add a `CHANGELOG.md` entry. OpenClaw policy requires user-facing `feat` changes to be recorded before merge.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: Security review needs attention because the chart’s default install path routes long-lived operator credentials through Helm values.

Concerns:

- **[medium] Credentials flow through Helm values:** `charts/openclaw/README.md:10`
  - body: The README tells users to pass `secret.gatewayToken` and provider API keys with `--set`, and `values.yaml` defines those fields as chart values. This can expose tokens through shell history and Helm release state; prefer an external Secret-only path or another flow that avoids storing plaintext values in the release.
  - confidence: 0.86
