## Summary

Keep open. Current main does not already implement a Helm chart, and the existing Kubernetes docs deliberately present Kustomize/raw manifests as the supported path. The PR is useful source work, but as submitted it needs maintainer product/security review plus blocking chart fixes before it should merge.

## What This Changes

The PR adds an additive `charts/openclaw/` Helm chart with chart metadata, values, deployment/service/configmap/PVC/secret templates, install notes, README, and `.helmignore` for Kubernetes deployment.

## Best Possible Solution

Either leave the supported in-repo Kubernetes path as the current Kustomize/raw-manifest flow and point Helm users to layer their own chart externally, or accept Helm as an owner-approved deployment surface with a redesigned secret model, migration-compatible resource names, rollout triggers, validation guards, docs, and changelog coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a current-main bug because main has no chart, but yes for the PR defects: render the chart with changed `service.port`, `secret.create=false` without `secret.existingSecret`, NodePort/LoadBalancer service types, `replicaCount > 1`, or changed config/secret values and inspect the templates.

## Solution Assessment

Is this the best way to solve the issue? No, not as submitted. A Helm chart may be a reasonable direction if maintainers want to own it, but this patch is not yet the narrowest maintainable implementation because it conflicts with the documented Kustomize-first surface and needs security and template fixes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Trigger rollouts for config and secret changes:** `charts/openclaw/templates/deployment.tpl:15-17`
  - body: The pod template has only static labels and no checksum annotations for the rendered ConfigMap or Secret. Because the init container copies config into the PVC and credentials are injected as env vars, `helm upgrade` can succeed while the running pod keeps old config or tokens until someone manually restarts the Deployment.
  - confidence: 0.9
- **[P1] Avoid storing credentials as Helm values:** `charts/openclaw/README.md:8-11`
  - body: The install docs pass `secret.gatewayToken` and provider keys through `--set`, and the chart models those credentials under values. Helm release values are inspectable release data, so this can retain long-lived gateway tokens and provider keys outside the intended Secret-only workflow.
  - confidence: 0.86
- **[P2] Preserve existing Kubernetes resource names:** `charts/openclaw/templates/_helpers.tpl:8-9`
  - body: The fullname helper always renders `<release>-<chart>`, so the documented release name `openclaw` creates resources like `openclaw-openclaw` and a new PVC. That breaks the stated equivalence with the existing manifests and prevents an in-place migration from the current `openclaw` objects.
  - confidence: 0.91
- **[P2] Keep the gateway port tied to service.port:** `charts/openclaw/values.yaml:84`
  - body: `config.openclawJson` hardcodes gateway port `18789` while the chart exposes `service.port` for the container, probes, and Service. Overriding `service.port` renders Kubernetes resources for one port while the gateway listens on another, causing failed probes or unreachable Service traffic.
  - confidence: 0.88
- **[P2] Require an existing secret when creation is disabled:** `charts/openclaw/templates/secret.tpl:1`
  - body: When `secret.create=false` and `secret.existingSecret` is empty, this template creates no Secret while the Deployment still references the helper-generated Secret name. Pods then fail with a missing required token Secret instead of Helm failing at render time.
  - confidence: 0.88
- **[P2] Do not advertise unreachable external Service URLs:** `charts/openclaw/templates/NOTES.txt:2-8`
  - body: The notes print NodePort and LoadBalancer access instructions, but the default config keeps the gateway bound to loopback. Those Service paths route to the pod IP, so the rendered instructions lead operators to a dead endpoint unless the chart also changes or warns about the bind setting.
  - confidence: 0.82
- **[P3] Guard replicas with a ReadWriteOnce PVC:** `charts/openclaw/values.yaml:4`
  - body: The chart exposes `replicaCount` while the default PVC access mode is `ReadWriteOnce`. Setting more than one replica can leave extra pods pending on multi-node clusters, and `strategy: Recreate` does not prevent multiple steady-state replicas.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `charts/openclaw/Chart.yaml:1`
  - body: This adds a user-facing deployment feature, but the PR does not add a `CHANGELOG.md` entry. OpenClaw policy requires user-facing `feat` changes to be recorded before merge.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff does not add CI or dependency execution, but the chart's value model can retain operator credentials outside the intended Kubernetes Secret workflow.

Concerns:

- **[medium] Credentials flow through Helm values:** `charts/openclaw/README.md:10`
  - body: The README tells users to pass `secret.gatewayToken` and provider API keys with `--set`, and `values.yaml` defines those fields as chart values. Helm release values can be inspected later, so this can expose long-lived OpenClaw gateway tokens and model-provider keys through shell history and release state; prefer an external Secret-only path or another flow that avoids storing plaintext credentials in release values.
  - confidence: 0.86
