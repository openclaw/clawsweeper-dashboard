## Summary

Keep open: current main does not contain a Helm chart, and this PR still has blocking deploy correctness and credential-handling issues plus missing real behavior proof. The current Kubernetes docs prefer Kustomize and say Helm can be layered on top, so merging a core chart also needs maintainer product and ownership judgment.

## What This Changes

The PR adds a new `charts/openclaw` Helm chart with values, templates, README, helpers, ConfigMap, Deployment, Service, PVC, Secret, and install notes for deploying OpenClaw Gateway on Kubernetes.

## Best Possible Solution

Decide whether OpenClaw core should own a Helm chart; if yes, land a chart that preserves current Kubernetes deploy semantics, uses safer secret guidance, validates unsupported values, and includes real install/upgrade proof.

## Reproduction Assessment

Yes for the blocking review findings by source inspection: the chart lacks a pod-template rollout trigger for startup-only config/secret inputs, and `service.port` can diverge from the gateway's configured listen port. No live cluster reproduction was run in this read-only review.

## Solution Assessment

No. A Helm chart may be useful, but this implementation is not the narrowest maintainable merge path until it matches the existing deploy behavior, handles secrets safely, and maintainers confirm core chart ownership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Trigger pod rollouts when chart inputs change:** `charts/openclaw/templates/deployment.tpl:15-17`
  - body: The chart renders ConfigMap and Secret changes, but the Deployment pod template has no checksum or other changing annotation. Because `openclaw.json` and `AGENTS.md` are copied by the init container into the PVC and API keys are consumed as env vars, `helm upgrade` can report success while the running pod keeps stale config and credentials until a manual restart.
  - confidence: 0.93
- **[P2] Keep service.port aligned with gateway.port:** `charts/openclaw/values.yaml:82`
  - body: Overriding `service.port` changes the container port and probes, but the generated `openclaw.json` still tells the gateway to listen on 18789. Any non-default port makes readiness/liveness and Service routing target a port the gateway is not using unless operators also rewrite the embedded JSON.
  - confidence: 0.91
- **[P2] Fail when external Secret name is missing:** `charts/openclaw/templates/secret.tpl:1`
  - body: With `secret.create=false` and an empty `secret.existingSecret`, the helper still returns a generated Secret name while no Secret is rendered. The Deployment then references a required `OPENCLAW_GATEWAY_TOKEN` key from a Secret that does not exist, producing a pod startup failure instead of a clear Helm render error.
  - confidence: 0.89
- **[P2] Avoid putting provider secrets in the install command:** `charts/openclaw/README.md:7-10`
  - body: The quick start asks users to pass the gateway token and provider API key through `--set`, which can expose real credentials through shell history or process listings. Make the primary path create or reference an existing Kubernetes Secret, and reserve inline values for clearly marked local-only examples.
  - confidence: 0.86
- **[P2] Use a fullname helper compatible with existing objects:** `charts/openclaw/templates/_helpers.tpl:6-9`
  - body: The documented `helm upgrade --install openclaw` renders names like `openclaw-openclaw`, while the existing manifests use `openclaw`, `openclaw-config`, and `openclaw-home-pvc`. That breaks the stated equivalent migration path by creating a second Deployment and PVC instead of targeting the current objects.
  - confidence: 0.82
