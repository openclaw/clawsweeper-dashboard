## Summary

Keep open. Current main still has only the Kustomize/raw-manifest Kubernetes path, and this PR is the active implementation candidate for Helm support. The proposed chart is not safe to merge as submitted because it adds a new deployment surface with unresolved rollout, naming, port/config, secret-handling, PVC, changelog, and product-ownership concerns.

## What This Changes

The PR adds an additive `charts/openclaw/` Helm chart with chart metadata, values, templates, install notes, and README for deploying OpenClaw on Kubernetes.

## Best Possible Solution

Revise the chart into an owner-approved, tested deployment surface that preserves the raw-manifest object/data contract, avoids Helm-stored credentials by default, restarts on config and secret changes, validates unsupported values combinations, updates docs and changelog, and has a clear maintainer decision that Helm belongs in this repository.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a current-main bug because main has no chart, but the PR defects have clear static reproduction paths: render with changed `service.port`, `secret.create=false` without `existingSecret`, NodePort/LoadBalancer service types, `replicaCount > 1`, or changed config/secret values and inspect the templates.

## Solution Assessment

Is this the best way to solve the issue? No, not as submitted. A Helm chart may be a reasonable direction, but this patch needs maintainer product approval plus chart fixes before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Trigger rollouts for config and secret changes:** `charts/openclaw/templates/deployment.tpl:15-17`
  - body: The pod template has only static labels and no checksum annotations for the rendered ConfigMap or Secret. Because the init container copies config into the PVC and secrets are injected as env vars, `helm upgrade` can succeed while the running pod keeps old config or tokens until someone restarts the Deployment manually.
  - confidence: 0.9
- **[P1] Avoid storing credentials as Helm values:** `charts/openclaw/README.md:7-10`
  - body: The install docs pass `secret.gatewayToken` and provider keys through `--set`, and the chart models those credentials under values. Helm stores release values and exposes them via release inspection commands, so this can retain gateway tokens and provider keys outside the intended Secret-only workflow.
  - confidence: 0.84
- **[P2] Preserve existing Kubernetes resource names:** `charts/openclaw/templates/_helpers.tpl:8-9`
  - body: The fullname helper always renders `<release>-<chart>`, so the documented release name `openclaw` creates resources like `openclaw-openclaw` and a new PVC. That breaks the stated equivalence with the existing manifests and prevents an in-place migration from the current `openclaw` objects.
  - confidence: 0.91
- **[P2] Keep the gateway port tied to service.port:** `charts/openclaw/values.yaml:83-84`
  - body: `config.openclawJson` hardcodes gateway port `18789` while the chart exposes `service.port` for the container and Service. Overriding `service.port` renders Kubernetes resources for one port while the gateway listens on another, causing failed probes or unreachable Service traffic.
  - confidence: 0.88
- **[P2] Require an existing secret when creation is disabled:** `charts/openclaw/templates/secret.tpl:1`
  - body: When `secret.create=false` and `secret.existingSecret` is empty, the helper still generates a Secret name but this template creates no Secret. The Deployment then references a required token Secret that does not exist, leaving pods in `CreateContainerConfigError` instead of failing at render time.
  - confidence: 0.88
- **[P2] Do not advertise unreachable NodePort or LoadBalancer URLs:** `charts/openclaw/templates/NOTES.txt:3-8`
  - body: The notes print NodePort and LoadBalancer access instructions, but the default config keeps the gateway bound to loopback. Those Service paths route to the pod IP, so the rendered instructions lead operators to a dead endpoint unless the chart also changes or warns about the bind setting.
  - confidence: 0.82
- **[P3] Guard replicas with a ReadWriteOnce PVC:** `charts/openclaw/values.yaml:4`
  - body: The chart exposes `replicaCount` while the default PVC access mode is `ReadWriteOnce`. Setting more than one replica can leave extra pods pending on multi-node clusters, and `strategy: Recreate` does not prevent multiple steady-state replicas.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `charts/openclaw/Chart.yaml:1`
  - body: This adds a user-facing deployment feature, but the PR does not add a `CHANGELOG.md` entry. OpenClaw policy requires user-facing `feat` changes to be recorded before merge.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff does not add new CI or dependency execution, but its Helm value model can retain operator credentials outside the intended Kubernetes Secret workflow.

Concerns:

- **[medium] Credentials flow through Helm values:** `charts/openclaw/README.md:9`
  - body: The README tells users to pass `secret.gatewayToken` and provider API keys with `--set`, and `values.yaml` defines those fields as chart values. That can expose long-lived OpenClaw gateway tokens and model-provider keys through shell history and Helm release state; prefer an external Secret-only path or file-based secret creation that avoids storing plaintext values in the release.
  - confidence: 0.84
