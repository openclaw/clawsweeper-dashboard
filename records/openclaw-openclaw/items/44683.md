## Summary

Keep open: this PR adds a real new deployment surface, but current main does not already provide a Helm chart and the branch still has correctness, credential-handling, and real-proof blockers before maintainers can consider it.

## What This Changes

The branch adds an additive Helm chart under `charts/openclaw` with chart metadata, values, templates, NOTES, README, and `.helmignore`.

## Best Possible Solution

If OpenClaw owns a Helm chart, it should match the raw-manifest resource/data contract, avoid credential values by default, restart pods on config or Secret changes, validate unsupported values, update Kubernetes docs, and include real install/upgrade proof.

## Reproduction Assessment

Yes for the review defects: source inspection of the PR chart shows the missing rollout triggers, credential-in-values path, hardcoded image/port, Secret validation gap, and PVC replica footgun. I did not run a real cluster install.

## Solution Assessment

No. A Helm deployment path may be useful, but this branch is not the best merge path until maintainers approve owning that surface and the chart fixes the correctness and security gaps with real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Trigger rollouts for config and secret changes:** `charts/openclaw/templates/deployment.tpl:15-17`
  - body: The pod template only has static labels and no checksum annotations for the rendered ConfigMap or Secret. Because this chart copies config during init and injects credentials as env vars, `helm upgrade` can succeed while the running pod keeps stale config or tokens until a manual restart.
  - confidence: 0.9
- **[P1] Avoid putting long-lived credentials in Helm values:** `charts/openclaw/README.md:10-11`
  - body: The README tells operators to pass the gateway token and provider API keys through `--set`, and the chart models those secrets as values. Helm release values are retrievable later, so this stores long-lived credentials outside the intended Secret-only workflow.
  - confidence: 0.88
- **[P2] Default to a current OpenClaw image:** `charts/openclaw/values.yaml:8`
  - body: The chart hardcodes `2026.3.12-slim`, while the review context identifies the latest stable release as `v2026.5.7` and current main is already on a later beta. Fresh Helm installs would deploy an old gateway by default.
  - confidence: 0.86
- **[P2] Preserve raw-manifest resource names:** `charts/openclaw/templates/_helpers.tpl:7-9`
  - body: The fullname helper always renders `<release>-<chart>`, so the documented release name `openclaw` creates resources like `openclaw-openclaw` and a new PVC. That diverges from the existing `openclaw` resources and breaks the stated equivalent migration path.
  - confidence: 0.9
- **[P2] Tie the gateway config port to service.port:** `charts/openclaw/values.yaml:83-84`
  - body: `config.openclawJson` hardcodes gateway port `18789` while the Deployment, probes, and Service use `.Values.service.port`. Overriding `service.port` renders Kubernetes resources for one port while the gateway still listens on another.
  - confidence: 0.88
- **[P2] Require existingSecret when secret creation is disabled:** `charts/openclaw/templates/secret.tpl:1`
  - body: When `secret.create=false` and `secret.existingSecret` is empty, the Secret template renders nothing but the Deployment still references the generated Secret name. Pods then fail at startup instead of Helm failing with a clear render-time error.
  - confidence: 0.88
- **[P2] Do not print unreachable Service endpoints:** `charts/openclaw/templates/NOTES.txt:2-8`
  - body: The NOTES output advertises NodePort and LoadBalancer access paths, but the default config keeps the gateway bound to loopback. Those Service paths route to the pod IP, so the rendered instructions lead to a dead endpoint unless the bind setting is also changed or clearly warned about.
  - confidence: 0.82
- **[P3] Guard multiple replicas with the RWO PVC default:** `charts/openclaw/values.yaml:4`
  - body: The chart exposes `replicaCount` while the default PVC access mode is `ReadWriteOnce`. Setting more than one replica can leave extra pods pending on multi-node clusters, and `strategy: Recreate` does not prevent multiple steady-state replicas.
  - confidence: 0.78
