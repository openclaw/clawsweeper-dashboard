## Summary

## What This Changes

## Best Possible Solution

Decide whether OpenClaw should own a first-party Helm chart; if yes, make it share the Kubernetes contract with the current manifests, use safe secret defaults, validate unsupported values, force config/secret rollouts, and prove fresh install plus upgrade in a real cluster.

## Solution Assessment

No, not as currently proposed. A first-party Helm chart needs an explicit support decision and a hardened migration, auth, rollout, and validation contract rather than a render-only copy of the existing manifests.

Overall confidence: 0.88

Full review comments:

- **[P1] Do not duplicate the Kubernetes deployment source:** `charts/openclaw/Chart.yaml:1`
  - body: Current docs ship Kustomize manifests and explicitly say Helm can be layered on top. This chart reimplements the same Deployment, Service, PVC, and ConfigMap as a second first-party source without proving the existing path is insufficient, which creates support drift unless maintainers first approve a single supported deployment contract.
  - confidence: 0.82
- **[P1] Preserve existing Kubernetes object names:** `charts/openclaw/templates/_helpers.tpl:9`
  - body: With the documented release name `openclaw`, this helper renders resources like `openclaw-openclaw` and `openclaw-openclaw-home-pvc`, while the current manifests use `openclaw` and `openclaw-home-pvc`. Adopting the chart would create a new Deployment/PVC instead of migrating the existing install and data.
  - confidence: 0.91
- **[P1] Trigger rollouts for config and secret changes:** `charts/openclaw/templates/deployment.tpl:15-17`
  - body: Helm upgrades that change `config.openclawJson`, `config.agentsMd`, or `secret.*` update only the ConfigMap/Secret. The pod template is unchanged, while the init container and env vars consume those values only at pod start, so upgrades can succeed while pods keep stale config or credentials.
  - confidence: 0.9
- **[P1] Avoid passing credentials through Helm values:** `charts/openclaw/README.md:10-11`
  - body: The primary install command puts the gateway token and provider key in `--set`, which can expose secrets through shell history, process listings, CI logs, and Helm release state. A first-party chart should prefer an existing Kubernetes Secret or another documented secret-management path.
  - confidence: 0.88
- **[P2] Keep the gateway port single-sourced:** `charts/openclaw/values.yaml:84`
  - body: Changing `service.port` updates the container port, probes, and Service, but the embedded `openclaw.json` still makes the gateway bind to 18789. A normal Helm port override can therefore fail readiness and make the Service unreachable unless the user also rewrites JSON by hand.
  - confidence: 0.9
- **[P2] Validate external secret configuration at render time:** `charts/openclaw/templates/secret.tpl:1`
  - body: When `secret.create=false` and `secret.existingSecret` is empty, the helper still points the Deployment at a generated secret name that this chart never creates. Helm should fail with a clear message instead of producing pods stuck in `CreateContainerConfigError`.
  - confidence: 0.86
- **[P2] Guard multi-replica persistent deployments:** `charts/openclaw/values.yaml:4`
  - body: The chart exposes `replicaCount` while defaulting persistence to a single `ReadWriteOnce` PVC. Setting replicas above one can leave extra pods pending or competing for the same state volume, so the chart needs validation or an explicit unsupported-mode guard.
  - confidence: 0.78
