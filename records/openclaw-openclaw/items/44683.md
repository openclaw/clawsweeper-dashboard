## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Trigger rollouts when config or secrets change:** `charts/openclaw/templates/deployment.tpl:15-17`
  - body: The pod template has no checksum annotation for the rendered ConfigMap or Secret, so `helm upgrade` can change config or secret values without creating a new ReplicaSet. Since config is copied only at startup and credentials are env vars, the running pod can keep stale config or keys until a manual restart.
  - confidence: 0.9
- **[P2] Keep the Service port tied to gateway.port:** `charts/openclaw/values.yaml:24`
  - body: The chart wires container ports and probes to `.Values.service.port`, but the default OpenClaw config hardcodes `gateway.port` to `18789`. If an operator changes only `service.port`, Kubernetes probes and Service traffic target a port the gateway is not listening on.
  - confidence: 0.9
- **[P2] Use names that can adopt current Kubernetes resources:** `charts/openclaw/templates/_helpers.tpl:9`
  - body: The helper always renders `<release>-<chart>`, so the documented release name `openclaw` creates resources like `openclaw-openclaw` and `openclaw-openclaw-home-pvc`. That diverges from the existing manifest names and can create a second deployment/PVC instead of adopting current objects and data.
  - confidence: 0.86
- **[P2] Fail when an external Secret name is omitted:** `charts/openclaw/templates/secret.tpl:1`
  - body: When `secret.create=false` and `secret.existingSecret` is empty, no Secret is rendered but the Deployment still references the generated Secret name for a required gateway token. Helm should fail early with a clear validation error instead of producing pods stuck on a missing Secret.
  - confidence: 0.88
- **[P2] Guard multi-replica installs with ReadWriteOnce PVCs:** `charts/openclaw/values.yaml:4`
  - body: The chart exposes `replicaCount`, but the default persistent volume uses `ReadWriteOnce`. Setting replicas above one can leave pods pending on multi-node clusters, and `strategy: Recreate` only affects updates, not steady-state replica count.
  - confidence: 0.78
- **[P2] Do not emit exposure URLs for loopback-bound installs:** `charts/openclaw/templates/NOTES.txt:2-8`
  - body: `NOTES.txt` prints NodePort and LoadBalancer access instructions when `service.type` changes, but the default config still binds the gateway to loopback. Those generated URLs can refuse connections unless the chart validates or clearly requires a non-loopback bind for Service/Ingress exposure.
  - confidence: 0.82
- **[P2] Default the chart to the current release image:** `charts/openclaw/values.yaml:8`
  - body: The chart defaults to `2026.3.12-slim` and `appVersion: 2026.3.12`, while the current release is `v2026.5.18`. Merging this now would make the new install path deploy an older OpenClaw build by default.
  - confidence: 0.82
- **[P2] Avoid teaching command-line provider secret input:** `charts/openclaw/README.md:8-11`
  - body: The README install example teaches users to pass the gateway token and provider key via `helm --set`. Official deployment docs should steer users toward a pre-created Secret or stronger secret-handling guidance before this becomes a supported path.
  - confidence: 0.82
