## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Trigger rollouts when rendered config or secrets change:** `charts/openclaw/templates/deployment.tpl:15-18`
  - body: The pod template has no checksum annotation for the rendered ConfigMap or Secret, so `helm upgrade` can change `config.openclawJson`, `config.agentsMd`, or `secret.*` without creating a new ReplicaSet. Because the init container copies config only at startup and provider tokens are injected as env vars, upgrades can report success while the running pod keeps stale config and credentials until manually restarted.
  - confidence: 0.9
- **[P2] Keep service.port synchronized with gateway.port:** `charts/openclaw/values.yaml:82-84`
  - body: The chart wires container ports and probes to `.Values.service.port`, but the default OpenClaw config still hardcodes `"port": 18789` inside `config.openclawJson`. If an operator overrides `service.port`, Kubernetes will probe and route to that new port while the gateway still listens on 18789, making the pod unhealthy or unreachable unless the user also edits raw JSON by hand.
  - confidence: 0.9
- **[P2] Avoid duplicating the chart name in Helm resource names:** `charts/openclaw/templates/_helpers.tpl:7-9`
  - body: The documented install command uses release name `openclaw`, but `openclaw.fullname` renders resources as `openclaw-openclaw` and `openclaw-openclaw-home-pvc`. That does not match the existing raw manifest names like `openclaw` and `openclaw-home-pvc`, so this chart is not equivalent for users migrating from the current manifests and can create a second deployment and empty PVC.
  - confidence: 0.86
- **[P2] Fail when an external Secret name is omitted:** `charts/openclaw/templates/secret.tpl:1`
  - body: When `secret.create=false` and `secret.existingSecret` is empty, the helper still returns the generated secret name but `secret.tpl` does not create it. The Deployment then references `OPENCLAW_GATEWAY_TOKEN` from a non-existent Secret, producing a pod startup failure instead of a clear Helm render error.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff does not add new code execution, but the install docs currently teach passing live tokens and provider keys through Helm `--set`, which is a concrete credential-handling footgun.

Concerns:
