## Summary

Keep open. The Helm chart is not already on current main and the PR is still a plausible feature, but it should not merge as-is because it adds a new core deployment surface with source-visible chart correctness issues, a secrets-handling concern, and no real install/upgrade proof.

## What This Changes

Adds a new `charts/openclaw` Helm chart with templated ConfigMap, Deployment, Service, PVC, Secret, values, install notes, and README for Kubernetes deployments.

## Best Possible Solution

If OpenClaw wants a core-owned Helm path, land a chart that mirrors the current Kustomize behavior, validates unsafe values, restarts pods on config/secret changes, aligns docs and release defaults, and gives safer external-Secret guidance; otherwise keep Helm as an operator-layered deployment outside core.

## Reproduction Assessment

Yes for the blocking review findings: the name mismatch, port mismatch, missing rollout trigger, invalid external-secret mode, and loopback Service notes are visible directly in the PR templates and values. I did not establish a live Kubernetes install in this read-only review.

## Solution Assessment

No. A Helm chart may be a reasonable deployment option, but this implementation is not the narrowest maintainable merge path until it mirrors the existing manifests more faithfully, validates unsafe values, handles rollout semantics, and proves real install/upgrade behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restart pods when chart config or secrets change:** `charts/openclaw/templates/deployment.tpl:15-17`
  - body: The pod template has no checksum or rollout annotation, so `helm upgrade` changes to `config.openclawJson`, `config.agentsMd`, or `secret.*` update only the ConfigMap/Secret. Because config is copied by an init container and credentials are env vars, running pods keep stale config or tokens until someone manually restarts the Deployment.
  - confidence: 0.91
- **[P2] Keep rendered names equivalent to existing objects:** `charts/openclaw/templates/_helpers.tpl:9`
  - body: The README installs the chart with release name `openclaw`, but this helper renders names as `openclaw-openclaw`. That creates a second Deployment/Service/PVC instead of matching the existing `openclaw` objects and `openclaw-home-pvc`, breaking the promised equivalent migration path.
  - confidence: 0.9
- **[P2] Tie the gateway listener port to service.port:** `charts/openclaw/values.yaml:84`
  - body: `service.port` drives the container port and probes, but `config.openclawJson` still hard-codes `gateway.port` to 18789. Overriding `service.port` makes the probes and Service target a port the gateway is not listening on unless users also edit raw JSON by hand.
  - confidence: 0.89
- **[P2] Fail when an external Secret name is required:** `charts/openclaw/templates/_helpers.tpl:29-30`
  - body: With `secret.create=false` and no `secret.existingSecret`, the helper still returns a generated name while `secret.tpl` emits no Secret. The Deployment then references a missing non-optional `OPENCLAW_GATEWAY_TOKEN`, so pods fail at startup instead of Helm rejecting the invalid values.
  - confidence: 0.88
- **[P2] Warn before advertising non-port-forward Service URLs:** `charts/openclaw/templates/NOTES.txt:2-8`
  - body: The install notes print NodePort and LoadBalancer access instructions when `service.type` changes, but the default gateway config binds to loopback. Those URLs will refuse connections unless users also change the bind setting, so the chart should fail or clearly warn in the generated notes for non-port-forward exposure.
  - confidence: 0.8
- **[P3] Guard multi-replica installs that share the default PVC:** `charts/openclaw/values.yaml:4`
  - body: `replicaCount` can be raised while `persistence.enabled` and `ReadWriteOnce` remain the defaults, which asks multiple pods to share one RWO PVC. Add chart validation or a prominent values note so extra pods do not sit pending on multi-node clusters.
  - confidence: 0.72

## Security Review

Status: needs_attention

Summary: Needs attention: the chart introduces a values-based secrets path that can persist long-lived tokens in Helm release metadata, although it does not change CI, dependency resolution, or release scripts.

Concerns:

- **[medium] Keep long-lived credentials out of Helm values:** `charts/openclaw/README.md:9`
  - body: The README tells users to pass the gateway token and provider keys with `--set`, and the chart stores them under `secret.*` values. Helm can later expose supplied release values, so the safer chart path should prefer or document pre-created Kubernetes Secrets for real credentials.
