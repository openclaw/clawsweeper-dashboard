## Summary

## What This Changes

The PR adds an official Helm chart under `charts/openclaw/` with Chart.yaml, values, templates, README, and NOTES as a values-driven Kubernetes deployment alternative to `scripts/k8s/manifests/`.

## Best Possible Solution

If OpenClaw wants to maintain an official Helm chart, land it only after deployment-owner approval, hardened migration and secret semantics, and redacted real install plus upgrade proof; otherwise keep Helm as an external layer over the Kustomize manifests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Trigger rollouts when config or secrets change:** `charts/openclaw/templates/deployment.tpl:15-17`
  - body: The pod template has only static labels, so changes to `config.openclawJson`, `config.agentsMd`, or `secret.*` will not create a new ReplicaSet. Those inputs are copied or injected only at startup, so `helm upgrade` can succeed while the gateway keeps stale config and credentials until a manual restart.
  - confidence: 0.91
- **[P1] Preserve existing Kubernetes object names:** `charts/openclaw/templates/_helpers.tpl:7-9`
  - body: The fullname helper always renders `<release>-openclaw`; with the documented release name `openclaw`, this creates `openclaw-openclaw` and a new PVC instead of adopting the existing `openclaw` resources. That breaks the stated equivalence and can split existing deployment state during migration.
  - confidence: 0.9
- **[P1] Do not document provider secrets through Helm values:** `charts/openclaw/README.md:8-9`
  - body: The install example tells operators to pass the gateway token and provider API key through `helm --set`. Helm stores release values in release metadata by default, so an official path should prefer a pre-created Secret or another safer secret handoff and make inline values an explicitly warned testing-only path.
  - confidence: 0.86
- **[P2] Keep service.port and gateway.port in sync:** `charts/openclaw/values.yaml:84`
  - body: `service.port`, `containerPort`, and the probes are templated, but `config.openclawJson` still hardcodes gateway port `18789`. If an operator changes only `service.port`, Kubernetes targets a port the gateway is not listening on and the pod fails readiness or becomes unreachable.
  - confidence: 0.88
- **[P2] Require an existing Secret when secret creation is disabled:** `charts/openclaw/templates/secret.tpl:1`
  - body: When `secret.create=false` and `secret.existingSecret` is empty, the chart renders no Secret but still points the Deployment at the generated secret name. Helm should fail at template time instead of installing pods that enter `CreateContainerConfigError`.
  - confidence: 0.89
- **[P2] Do not default new installs to an old image tag:** `charts/openclaw/values.yaml:8`
  - body: The chart pins `2026.3.12-slim`, while the current raw manifest uses the moving `slim` tag and the target repo release context is newer. A first-party chart should either track the documented manifest default or be updated by the release process so new installs do not start on an old gateway build.
  - confidence: 0.82
- **[P2] Guard unsupported replica and PVC combinations:** `charts/openclaw/values.yaml:4`
  - body: The chart exposes `replicaCount` while mounting one OpenClaw home PVC into every pod. With the default ReadWriteOnce PVC this can either block pods on multi-node clusters or allow multiple pods on one node to share mutable gateway state, so the chart needs a template guard or an explicit supported HA design.
  - confidence: 0.8
