## Summary

Keep this PR open, but it is not ready to merge. Current main still only has the raw Kubernetes manifests, so the Helm chart is unique work, but the proposed chart has blocking upgrade/migration issues, security-sensitive secret-handling concerns, and still needs redacted real behavior proof from an actual Kind/Kubernetes install or upgrade path; after adding proof, updating the PR body should trigger a fresh ClawSweeper review, or a maintainer can comment `@clawsweeper re-review`.

## What This Changes

The PR adds a new `charts/openclaw` Helm chart with values, templated Kubernetes resources, secrets, persistence, service notes, and a README as a Helm alternative to `scripts/k8s/manifests`.

## Best Possible Solution

Keep the Helm chart path only if it preserves the raw-manifest migration semantics, has clear secret/config ownership, rolls pods on value-driven startup changes, and includes real Kind or Kubernetes proof for install and upgrade behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Trigger rollouts for config and secret changes:** `charts/openclaw/templates/deployment.tpl:15-17`
  - body: The pod template has only static labels and no checksum annotation for the rendered ConfigMap or Secret, so `helm upgrade` can change `config.openclawJson`, `config.agentsMd`, or `secret.*` values without creating a new ReplicaSet. Those inputs are read only at startup through the init container or env vars, so upgrades can report success while the running gateway keeps stale config or credentials until a manual restart.
  - confidence: 0.91
- **[P2] Preserve raw-manifest object names for migration:** `charts/openclaw/templates/_helpers.tpl:9`
  - body: The documented release name `openclaw` renders resources like `openclaw-openclaw` and `openclaw-openclaw-home-pvc`. The existing manifests use `openclaw` and `openclaw-home-pvc`, so adopting the chart creates a parallel Deployment and fresh PVC instead of being equivalent to the current kustomize path.
  - confidence: 0.9
- **[P2] Fail when an external secret name is missing:** `charts/openclaw/templates/secret.tpl:1`
  - body: When `secret.create=false` is set without `secret.existingSecret`, the Secret template renders nothing but the Deployment still references the generated secret name for `OPENCLAW_GATEWAY_TOKEN`. That makes Helm render successfully and leaves pods to fail later with a missing required Secret, so add a template-time `fail` guard.
  - confidence: 0.9
- **[P2] Keep the gateway port and Service port in sync:** `charts/openclaw/values.yaml:84`
  - body: Overriding `service.port` updates the Service, container port, and probes, but the embedded `config.openclawJson.gateway.port` stays at `18789`. The gateway will bind the old port while Kubernetes probes and routes to the new one, making the release unhealthy unless operators manually edit the JSON too.
  - confidence: 0.88
- **[P2] Deploy the current image by default:** `charts/openclaw/values.yaml:8`
  - body: The chart defaults to `2026.3.12-slim` and appVersion `2026.3.12`, while the current release context is `v2026.5.7` and the raw manifest uses the moving `slim` image tag. Merging this now would make the new install path deploy an older gateway by default.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The chart introduces security-sensitive deployment defaults around credentials and config/secret rollout behavior that should be fixed before merge.

Concerns:

- **[medium] Credentials are passed through Helm values:** `charts/openclaw/README.md:10`
  - body: The README tells users to pass the gateway token and provider API key through `--set`, and Helm release records store override Config values. That can duplicate long-lived credentials in Helm release history in addition to the application Secret; prefer an existing Secret flow, `--set-file` guidance where appropriate, and explicit warnings about redaction and release-value storage.
  - confidence: 0.82
- **[medium] Security config updates may not reach running pods:** `charts/openclaw/templates/deployment.tpl:15`
  - body: The Deployment lacks checksum annotations or another rollout trigger for ConfigMap and Secret value changes. Gateway auth, bind, Control UI, and token/provider-key changes can remain stale after a successful Helm upgrade until an operator manually restarts pods.
  - confidence: 0.86
