## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Trigger rollouts when rendered config or secrets change:** `charts/openclaw/templates/deployment.tpl:15-17`
  - body: The pod template has no checksum annotation for the rendered ConfigMap or Secret, so `helm upgrade` can change config or secret values without creating a new ReplicaSet. Because config is copied only at startup and provider keys are env vars, upgrades can report success while the running pod keeps stale config and credentials. Kubernetes only rolls a Deployment when `.spec.template` changes. ([kubernetes.io](https://kubernetes.io/docs/concepts/workloads/controllers/deployment/?source=post_page-----8db6c301b18--------------------------------&utm_source=openai))
  - confidence: 0.9
- **[P2] Keep service.port synchronized with gateway.port:** `charts/openclaw/values.yaml:24`
  - body: The chart wires container ports and probes to `.Values.service.port`, but the default OpenClaw config hardcodes `"port": 18789`. If an operator overrides `service.port`, Kubernetes probes and routes to the new port while the gateway still listens on 18789.
  - confidence: 0.9
- **[P2] Avoid duplicating the chart name in resource names:** `charts/openclaw/templates/_helpers.tpl:9`
  - body: The documented release name `openclaw` renders resources like `openclaw-openclaw` and `openclaw-openclaw-home-pvc`, diverging from the current manifest names and creating a second deployment/PVC during adoption instead of reusing the current objects/data.
  - confidence: 0.86
- **[P2] Fail when an external Secret name is omitted:** `charts/openclaw/templates/secret.tpl:1`
  - body: When `secret.create=false` and `secret.existingSecret` is empty, the helper still returns a generated Secret name but no Secret is created. The Deployment then references a missing required gateway token Secret and pods fail at startup instead of Helm failing clearly.
  - confidence: 0.88
- **[P2] Guard multi-replica installs with a ReadWriteOnce PVC:** `charts/openclaw/values.yaml:4`
  - body: The chart exposes `replicaCount`, but the default persistent volume is `ReadWriteOnce`, which Kubernetes defines as read-write mountable by a single node. Setting replicas above 1 can leave pods pending on multi-node clusters, and `strategy: Recreate` does not prevent multiple steady-state replicas. ([kubernetes.io](https://kubernetes.io/docs/concepts/storage/persistent-volumes/?source=post_page---------------------------&utm_source=openai))
  - confidence: 0.78
- **[P2] Do not advertise Service exposure with loopback bind:** `charts/openclaw/templates/NOTES.txt:2-8`
  - body: `NOTES.txt` prints NodePort and LoadBalancer access instructions when `service.type` changes, but the default config still binds the gateway to loopback. Those generated URLs can refuse connections unless the chart validates or clearly requires a non-loopback bind for Service/Ingress exposure.
  - confidence: 0.82
- **[P2] Default the chart to the current OpenClaw image:** `charts/openclaw/values.yaml:8`
  - body: The chart defaults to `2026.3.12-slim` and `appVersion: 2026.3.12`, while current main and the review context are after the `v2026.5.12` release. Merging this as-is would make the new install path deploy an older OpenClaw build by default.
  - confidence: 0.82
- **[P2] Avoid passing secrets through --set examples:** `charts/openclaw/README.md:8-10`
  - body: The README install example teaches users to pass the gateway token and provider key via `helm --set`, which Helm documents as command-line value input; the chart then stores those values in Kubernetes Secrets, which Kubernetes cautions are stored unencrypted in etcd by default unless extra protections are enabled. Prefer an existing Secret or safer secret guidance before merge. ([docs.helm.sh](https://docs.helm.sh/docs/helm/helm_install/?utm_source=openai))
  - confidence: 0.82
