## Summary

Keep open: current main has a Kustomize-based Kubernetes path and explicitly frames Helm as something operators can layer on top, while this PR adds a new user-facing deployment surface with unresolved chart correctness, credential-handling, and ownership questions.

## What This Changes

The PR adds an additive `charts/openclaw/` Helm chart with chart metadata, values, templates, README, install notes, Secret/ConfigMap/Deployment/Service/PVC resources, and Helm ignore rules for Kubernetes deployment.

## Best Possible Solution

If OpenClaw wants Helm in core, land an owner-approved chart that preserves the raw-manifest contract, avoids plaintext credential values by default, restarts pods on config and secret changes, validates unsupported value combinations, tracks current release images, and updates Kubernetes docs plus the changelog.

## Reproduction Assessment

Yes. The PR defects are source-reproducible from the added chart templates and values: config/secret upgrades do not change the pod template, credentials are passed through values, resource names diverge from current manifests, and several value combinations render broken deployments.

## Solution Assessment

No. A Helm chart may be a valid deployment option, but this branch is not the best merge path until maintainers approve core ownership and the chart fixes credential handling, rollout behavior, resource equivalence, stale image defaults, validation, docs, and changelog coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Add rollout triggers for config and secret changes:** `charts/openclaw/templates/deployment.tpl:15-17`
  - body: The pod template has only static labels and no checksum annotations for the rendered ConfigMap or Secret. Because config is copied by the init container and secrets are injected as env vars, `helm upgrade` can succeed while the running pod keeps stale config or tokens until a manual restart.
  - confidence: 0.9
- **[P1] Avoid storing credentials in Helm release values:** `charts/openclaw/README.md:8-11`
  - body: The README tells users to pass the gateway token and provider API keys through `--set`, and the chart models those credentials under values. Helm release values are retrievable later, so this retains long-lived credentials outside the intended Secret-only workflow.
  - confidence: 0.88
- **[P2] Default the chart to a current OpenClaw image:** `charts/openclaw/values.yaml:8`
  - body: The chart hardcodes `2026.3.12-slim` while the provided review context says the latest release is `v2026.5.2`. Fresh Helm installs would deploy an old gateway by default.
  - confidence: 0.86
- **[P2] Preserve existing Kubernetes resource names:** `charts/openclaw/templates/_helpers.tpl:9`
  - body: The fullname helper always renders `<release>-<chart>`, so the documented release name `openclaw` creates resources like `openclaw-openclaw` and a fresh PVC. That breaks the stated equivalence with the existing `openclaw` objects and prevents an in-place migration from the raw manifests.
  - confidence: 0.91
- **[P2] Tie gateway config port to service.port:** `charts/openclaw/values.yaml:84`
  - body: `config.openclawJson` hardcodes gateway port `18789` while the chart exposes `service.port` for the container, probes, and Service. Overriding `service.port` renders Kubernetes resources for one port while the gateway listens on another.
  - confidence: 0.88
- **[P2] Require existingSecret when secret creation is disabled:** `charts/openclaw/templates/secret.tpl:1`
  - body: When `secret.create=false` and `secret.existingSecret` is empty, the helper still generates a Secret name but this template creates no Secret. The Deployment then references a required token Secret that does not exist, leaving pods stuck at startup instead of failing at render time.
  - confidence: 0.88
- **[P2] Do not advertise unreachable Service URLs:** `charts/openclaw/templates/NOTES.txt:2-8`
  - body: The notes print NodePort and LoadBalancer access instructions, but the default config keeps the gateway bound to loopback. Those Service paths route to the pod IP, so the rendered instructions lead operators to a dead endpoint unless the chart also changes or warns about the bind setting.
  - confidence: 0.82
- **[P3] Guard replicaCount with the default RWO PVC:** `charts/openclaw/values.yaml:4`
  - body: The chart exposes `replicaCount` while the default PVC access mode is `ReadWriteOnce`. Setting more than one replica can leave extra pods pending on multi-node clusters, and `strategy: Recreate` does not prevent multiple steady-state replicas.
  - confidence: 0.78
- **[P3] Add the required changelog entry:** `charts/openclaw/Chart.yaml:1`
  - body: This adds a user-facing deployment feature, but the PR does not add a `CHANGELOG.md` entry. OpenClaw policy requires user-facing `feat` changes to be recorded before merge.
  - confidence: 0.86
