## Summary

Keep open: current main still has no Helm chart, the Kubernetes docs currently present Kustomize as the supported in-repo path, and this external PR still has blocking chart correctness issues plus missing real install/upgrade proof.

## What This Changes

Adds a new `charts/openclaw` Helm chart with values, templates, README, and NOTES for installing OpenClaw on Kubernetes.

## Best Possible Solution

If OpenClaw wants a core-owned Helm path, land a chart that mirrors current manifests, validates unsafe values, restarts pods on config or secret changes, aligns the docs with the deployment policy, and includes real install/upgrade proof; otherwise keep Helm as an operator-layered chart outside core.

## Reproduction Assessment

Yes for the blocking review findings: the template names, port mismatch, missing rollout trigger, invalid external-Secret mode, and loopback exposure notes are visible from the PR source against current main. I did not run a live Kubernetes install in this read-only review.

## Solution Assessment

No. A Helm chart could be useful, but this implementation is not the narrowest maintainable merge path until maintainers confirm the core-owned chart direction and the chart fixes the source-visible correctness and secret-handling issues.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Restart pods when chart inputs change:** `charts/openclaw/templates/deployment.tpl:15-17`
  - body: The pod template has no checksum annotations, so `helm upgrade` changes to ConfigMap or Secret values can leave the running gateway on stale copied config and env vars until a manual restart. Helm documents checksum annotations for this exact rollout pattern. ([helm.sh](https://helm.sh/docs/howto/charts_tips_and_tricks/))
  - confidence: 0.91
- **[P2] Keep chart names equivalent to existing objects:** `charts/openclaw/templates/_helpers.tpl:9`
  - body: The documented release name `openclaw` renders names like `openclaw-openclaw`, creating new objects and a fresh PVC instead of matching the existing `openclaw` resources promised by the migration-equivalent chart.
  - confidence: 0.9
- **[P2] Tie the gateway listener port to service.port:** `charts/openclaw/values.yaml:84`
  - body: `service.port` drives the container port and probes, but the gateway JSON still binds to `18789`; overriding the value makes Kubernetes target a port the gateway is not listening on. ([kubernetes.io](https://kubernetes.io/docs/concepts/services-networking/service/))
  - confidence: 0.89
- **[P2] Fail when an external Secret name is required:** `charts/openclaw/templates/_helpers.tpl:29-30`
  - body: With `secret.create=false` and no `secret.existingSecret`, the chart emits no Secret but the Deployment references a generated non-optional Secret name, so pods fail at startup instead of Helm rejecting invalid values. ([kubernetes.io](https://kubernetes.io/docs/concepts/configuration/secret/))
  - confidence: 0.88
- **[P2] Warn before printing non-port-forward Service access:** `charts/openclaw/templates/NOTES.txt:2-8`
  - body: The NOTES template prints NodePort and LoadBalancer access instructions, but the default gateway config binds to loopback; current docs state that loopback works for port-forward but not Service or Ingress paths to the pod IP.
  - confidence: 0.8
- **[P3] Guard multi-replica installs with the default PVC:** `charts/openclaw/values.yaml:4`
  - body: The chart lets users raise `replicaCount` while the default PVC remains `ReadWriteOnce`; on multi-node clusters, replicas scheduled to different nodes cannot all mount that volume. Kubernetes defines ReadWriteOnce as read-write by a single node. ([kubernetes.io](https://kubernetes.io/docs/concepts/storage/persistent-volumes/?utm_source=openai))
  - confidence: 0.72
