## Summary

Keep open. Current main still ships only the Kustomize/raw-manifest Kubernetes path and has no `charts/openclaw` chart, so this PR is not implemented or obsolete; however, the proposed chart has blocking correctness and security issues that need revision and maintainer product review before merge.

## What This Changes

The PR adds a new `charts/openclaw/` Helm chart with values, templates, Secret/ConfigMap/Deployment/Service/PVC resources, NOTES output, and chart README intended to mirror the existing `scripts/k8s` manifests.

## Best Possible Solution

An acceptable chart should be owner-approved, preserve the raw-manifest object/data contract for migration, keep credentials in an existing/external Secret flow, couple Service and gateway config safely, validate invalid value combinations, trigger rollouts on config/secret changes, and update docs/changelog plus chart validation before landing.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a current-main bug because main has no chart, but the PR defects have clear static reproduction paths: render with changed `service.port`, `secret.create=false` without `existingSecret`, changed config/secret values, or release name `openclaw` and inspect the generated manifests.

## Solution Assessment

Is this the best way to solve the issue? No, not as submitted. A Helm chart may be a reasonable deployment direction if maintainers want it in core, but this patch needs correctness, migration, secret-handling, validation, docs, and ownership fixes first.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Trigger rollouts for config and secret changes:** `charts/openclaw/templates/deployment.tpl:15-17`
  - body: The pod template has no checksum annotations for the rendered ConfigMap or Secret. Because config is copied by the init container and secrets are injected as env vars, `helm upgrade` can succeed while the running pod keeps old config or tokens until someone restarts the Deployment manually.
  - confidence: 0.9
- **[P1] Avoid storing credentials as Helm values:** `charts/openclaw/README.md:8-10`
  - body: The install docs pass `secret.gatewayToken` and provider keys through `--set`, and the chart models those credentials under values. Helm release information includes chart contents and values files, so this can retain long-lived tokens in shell history and release state instead of a Secret-reference-only workflow.
  - confidence: 0.86
- **[P2] Preserve existing Kubernetes resource names:** `charts/openclaw/templates/_helpers.tpl:8-9`
  - body: The fullname helper always renders `<release>-<chart>`, so the documented release name `openclaw` creates resources like `openclaw-openclaw` and a new PVC. That breaks the stated equivalence with the existing manifests and prevents an in-place migration from the current `openclaw` objects.
  - confidence: 0.91
- **[P2] Keep the gateway port tied to service.port:** `charts/openclaw/values.yaml:83-84`
  - body: `config.openclawJson` hardcodes gateway port `18789` while the chart exposes `service.port` for the container, probes, and Service. Overriding `service.port` renders Kubernetes resources for one port while the gateway still listens on another, causing failed probes or unreachable Service traffic.
  - confidence: 0.88
- **[P2] Require an existing secret when creation is disabled:** `charts/openclaw/templates/secret.tpl:1`
  - body: When `secret.create=false` and `secret.existingSecret` is empty, the helper still generates a Secret name but this template creates no Secret. The Deployment then references a required token Secret that does not exist, leaving pods in `CreateContainerConfigError` instead of failing at render time.
  - confidence: 0.87
