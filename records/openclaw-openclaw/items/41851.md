## Summary

Keep open: the branch targets a stale sanitizer helper in `errors.ts`, while current `main` routes user-facing replies through the extracted `sanitize-user-facing-text.ts` helper where retained duplicate blocks still use `block.trim()`. It also lacks the required after-fix real behavior proof for an external PR.

## What This Changes

The branch adds an `openclaw-*.log` ignore pattern and changes `src/agents/pi-embedded-helpers/errors.ts` so duplicate block rebuilding appends the original block instead of `block.trim()`.

## Best Possible Solution

Move retained-block whitespace preservation into `src/agents/pi-embedded-helpers/sanitize-user-facing-text.ts`, add a duplicated indented-block regression test, and leave broader duplicate heuristics to https://github.com/openclaw/openclaw/issues/14629.

## Reproduction Assessment

Yes, from source inspection. `sanitizeUserFacingText("Intro\n\n    x\n\n    x")` reaches the active duplicate-collapse helper, removes the duplicate block, and appends the retained block with `block.trim()`, stripping its indentation.

## Solution Assessment

No, not as-is. Preserving the retained block is the narrow maintainable fix, but this branch applies it to the stale `errors.ts` location instead of the active extracted sanitizer and lacks regression coverage.
