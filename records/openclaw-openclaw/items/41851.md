## Summary

Keep open: the reported bug is plausible and narrow, but this PR is not merge-ready because it changes the old `errors.ts` helper while current `main` runs the duplicate-block sanitizer from `sanitize-user-facing-text.ts`, where the trimming behavior still exists.

## What This Changes

The PR adds an `openclaw-*.log` ignore pattern and changes duplicate-block rebuilding in `src/agents/pi-embedded-helpers/errors.ts` to append the original block instead of `block.trim()`.

## Best Possible Solution

Apply the retained-block whitespace preservation in the active sanitizer, add an indented duplicate-block regression test, and leave broader duplicate-detection heuristics to https://github.com/openclaw/openclaw/issues/14629.

## Reproduction Assessment

Yes, from source inspection: outbound replies call `sanitizeUserFacingText()`, whose active duplicate-block helper still pushes `block.trim()` when a duplicate is collapsed. I did not run a runtime reproduction because this review was read-only.

## Solution Assessment

No, not as this branch is written. Preserving the original retained block is the right narrow fix, but it needs to move from the stale `errors.ts` helper to `sanitize-user-facing-text.ts` with regression coverage.
