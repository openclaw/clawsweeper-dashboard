## Summary

This PR is still useful and should stay open, but it remains blocked by a compatibility regression: the new startup hard timer hard-codes a 600 second fallback before config-backed timeout resolution, so configured finite local-agent timeouts can be ignored or shortened. The terminal proof is strong for explicit `--timeout` behavior, and the remaining repair is narrow.

## What This Changes

The PR adds CLI-level hard timeout handling for local and embedded-fallback agent runs, updates the CLI docs, and adds focused timeout, signal, and exit tests.

## Best Possible Solution

Land the CLI hard timeout after the startup boundary preserves the same config-backed local-agent timeout contract as the command path, with regression coverage for explicit, omitted, configured, and disabled timeouts.

## Reproduction Assessment

Yes for the blocking PR defect: source inspection shows the startup hard timer uses a 600s fallback before config-backed timeout resolution. The original no-completion CLI hang also has credible terminal proof in the PR body for explicit-timeout paths.

## Solution Assessment

No, not as submitted: the hard-timeout approach is the right repair shape, but the startup boundary must use the same configured timeout contract as the local agent command path before merge. The safer repair is to resolve config consistently or only arm the early timer for explicit `--timeout`.
