## Summary

Keep this PR open: the hard-timeout direction is useful, but the current diff can break users who rely on a configured local-agent timeout when `--timeout` is omitted, and the PR still has only mocked proof for the real wedged CLI behavior.

## What This Changes

Adds CLI-level hard timeout handling for `openclaw agent --local` and embedded fallback runs, plus focused tests and `docs/cli/agent.md` timeout documentation.

## Best Possible Solution

Keep the hard-timeout helper, but preserve the existing config-aware timeout semantics for omitted `--timeout`, add regression coverage for configured defaults, and require redacted live terminal/log proof before merge.

Yes at source level: current main awaits local and fallback `agentCommand` promises directly, so a non-settling embedded run can keep the CLI alive. There is not yet high-confidence live exact-head proof from a real wedged local agent command.

## Solution Assessment

No, not as currently written: the embedded-run boundary is a maintainable direction, but the startup timer must preserve the existing config-aware omitted-timeout contract before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P1] Preserve configured timeout for omitted --timeout:** `src/cli/run-main.ts:115`
  - body: This startup guard runs before config is read and falls back to `600` whenever `--timeout` is omitted. Current agent timeout semantics use `cfg.agents?.defaults?.timeoutSeconds ?? 600`, so an operator with a longer configured default can have `openclaw agent --local` killed at 630s during startup/registration instead of using their configured budget. Please defer this timer until the config-backed value is known, or only arm the pre-registration guard for explicit `--timeout` values.
  - confidence: 0.88
