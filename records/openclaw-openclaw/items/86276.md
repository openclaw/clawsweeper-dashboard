## Label Justifications

- `P2`: This is a normal-priority CLI automation fix with limited blast radius, but it affects timeout behavior for local agent runs.
- `merge-risk: 🚨 compatibility`: The PR can change existing omitted-timeout `--local` runs from the embedded 48-hour default to a 600s plus 30s hard exit when no config timeout is set.
- `merge-risk: 🚨 availability`: The new hard timer intentionally exits the CLI process with 124 for stuck local/fallback runs, which can terminate long-running local automation if the timeout policy is wrong.

## Summary

Keep open: the hard-timeout direction addresses a real source-reproducible local/fallback hang class, but the diff changes omitted `--local` timeout behavior to a 600s hard boundary and still lacks exact-head real behavior proof.

## What This Changes

Adds a bounded embedded-agent helper around explicit `--local` and embedded fallback runs, exits 124 after the resolved timeout plus 30s, updates focused tests, and documents the CLI timeout behavior.

## Best Possible Solution

Land a CLI hard timeout only after preserving or explicitly approving the omitted-timeout upgrade semantics, adding focused coverage for that choice, and attaching exact-head terminal/log proof for a wedged local run.

## Reproduction Assessment

Yes, source-level: current main awaits the local or fallback `agentCommand` promise directly, so a never-settling embedded run keeps `agentCliCommand` pending. I did not establish a live wedged CLI reproduction in this read-only review.

## Solution Assessment

No, not yet: the hard-timeout wrapper is a reasonable direction, but the current diff silently changes omitted local timeout behavior unless maintainers explicitly accept that migration. The safer path is to preserve unset-config semantics or document and prove the new CLI default as an intentional compatibility break.
