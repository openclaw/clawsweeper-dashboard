## Summary

Keep open. Current main still has a clear stale Control UI run-state path: Stop rendering uses either `chatRunId` or `sessionsResult.hasActiveRun`, terminal chat events clear `chatRunId`, but the session event path can leave a previously true `hasActiveRun` row in place.

## What This Changes

The issue asks for WebUI run-state reconciliation so the Stop button returns to Send after a completed assistant reply without refreshing the page.

## Best Possible Solution

Add a regression-tested Control UI reconciliation path that clears the current session row's stale `hasActiveRun` on matching terminal chat/session events or refreshes sessions after terminal run cleanup, without changing abort semantics.

## Reproduction Assessment

Yes, by source inspection: start from a UI state where the current session row has `hasActiveRun: true`, then deliver the matching final chat event. `chatRunId` clears, but the partial `sessions.changed` path can preserve the stale row value that keeps `canAbort` true.

## Solution Assessment

Yes, the narrowest fix is to reconcile the existing Control UI session/run state on terminal events or after the final sessions lifecycle update. A broader protocol change is not required unless maintainers prefer emitting `hasActiveRun` in chat lifecycle snapshots.
