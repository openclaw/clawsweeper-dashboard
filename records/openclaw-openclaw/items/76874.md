## Summary

Keep open. Current main still has a source-reproducible stale Control UI run-state path, and the provided timeline shows open PR #76894 uses closing syntax for this issue, so this should remain open until that PR is reviewed and merged or closed.

## What This Changes

The issue asks for the WebUI Stop button to return to Send after a completed assistant reply, with stale running state reconciled before later user messages are accepted.

## Best Possible Solution

Review and land or close PR #76894; the durable fix should reconcile cached Control UI session active-run state on terminal chat lifecycle events and keep regression coverage for that path.

## Reproduction Assessment

Yes, by source inspection: seed the current session row with `hasActiveRun: true`, process a matching final chat event, and `chatRunId` clears while the cached session row can stay active. That leaves `hasAbortableSessionRun` true and the Stop button rendered.

## Solution Assessment

Yes. The narrow maintainable fix is to reconcile the cached session `hasActiveRun` flag on matching terminal chat events, or force an equivalent session-row refresh, while preserving the existing run-id filtering for unrelated terminal events.
