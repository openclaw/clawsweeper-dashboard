## Summary

Keep open. Current main still has a source-reproducible stale Control UI run-state path, and open PR #76894 references this issue with closing syntax, so the issue should remain open until that PR is reviewed and merged or closed.

## What This Changes

The issue asks for the Control UI to clear stale active-run state so the Stop button returns to Send after a completed assistant reply.

## Best Possible Solution

Review and land or close PR #76894; the merged fix should reconcile the cached `hasActiveRun` flag on matching terminal chat events and keep regression coverage around the Stop/Send state.

## Reproduction Assessment

Yes, by source inspection: start with the current session row cached as `hasActiveRun: true`, then deliver a matching terminal chat event. `chatRunId` clears, but the cached session row can keep `canAbort` true because the lifecycle payload omits `hasActiveRun` and the UI skips a full sessions reload for chat-turn updates.

## Solution Assessment

Yes. The narrow maintainable fix is to reconcile the cached session active-run flag on matching terminal chat lifecycle completion, or otherwise refresh the session row after terminal cleanup; the linked PR appears scoped to that boundary but is not merged into current main.
