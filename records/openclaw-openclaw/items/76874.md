## Summary

Keep open; current main still has a credible stale run-state path where the frontend clears `chatRunId` on the terminal chat event but keeps a previously applied `hasActiveRun: true` session row, so the Stop control can remain visible.

## What This Changes

The issue requests that the Control UI automatically return from Stop to Send after a completed assistant reply is visible.

## Best Possible Solution

Make terminal run cleanup and Control UI session state agree so `hasActiveRun` is cleared or refreshed after the active chat run actually completes, with regression coverage for final chat events plus stale session rows.

## Reproduction Assessment

Yes, from source inspection: seed the current session row with `hasActiveRun: true`, process a matching terminal chat event that clears `chatRunId`, and the Stop predicate remains true unless the session row is also reconciled. I did not run a live browser reproduction.

## Solution Assessment

Yes, a narrow lifecycle/session-state reconciliation is the best direction; a broad refresh loop would be heavier and should only be used if the gateway cannot emit an authoritative post-cleanup session update.
