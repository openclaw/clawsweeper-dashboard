## Label Justifications

- `P2`: The report describes a real Dreaming ingestion regression with limited scope to Memory Core session-corpus state rather than a whole-runtime outage.
- `impact:session-state`: The failure prevented new session transcripts from entering Dreaming state and diary generation.
- `impact:data-loss`: The issue is about user session transcript content being skipped by the ingestion corpus, even though the underlying session files still existed.

## Summary

Current main and shipped releases now route Memory Core Dreaming session-ingestion state through SQLite-backed plugin state, so the reported WSL2 9p chmod/stat failure no longer blocks the Dreaming ingestion path.

## What This Changes

The issue reports Memory Core Dreaming session-corpus ingestion freezing on WSL2 because the workspace-root private file-store permission check rejected `/mnt/*` 9p mounts as mode 777.

## Best Possible Solution

Keep the SQLite-backed Memory Core Dreaming state implementation shipped in `v2026.6.5` and later; any remaining Dreaming cron symptoms should stay on narrower active reports with fresh logs.

## Reproduction Assessment

Yes at source level: the reporter's WSL2 9p logs match the locked fs-safe chmod/stat behavior, and current main shows the affected Memory Core Dreaming state path has moved off that private file-store persistence. I did not run a live WSL2 mount reproduction in this read-only review.

## Solution Assessment

Yes: moving Dreaming runtime state to SQLite plugin state is the narrow maintainable fix for this ingestion-state failure because it avoids non-portable workspace-root permission semantics without weakening private secret-file enforcement globally.
