## Summary

Keep this PR open, but not merge-ready. Current main still lacks a transient cooldown for Telegram sendChatAction failures, and the PR addresses the right handler with useful regression coverage; however, the latest diff also narrows the shared Telegram rate-limit helper used by message delivery and webhook retry paths, which is broader than the stated fix.

## What This Changes

The PR adds Telegram sendChatAction transient network/rate-limit/5xx cooldown state and regression tests, injects backoff runtime helpers for tests, narrows Telegram rate-limit detection to structured error_code values, and adds a changelog entry.

## Best Possible Solution

Land a version that keeps the dedicated sendChatAction transient cooldown and regression coverage while preserving the exported Telegram rate-limit helper's existing fallback behavior. If chat-action cooldown needs stricter classification, add a private chat-action-only predicate instead of narrowing shared delivery/webhook retry semantics.

## Reproduction Assessment

Yes. The original bug is reproducible at unit level by making sendChatAction throw transient 500/network/429 errors and observing current main retry each call; the PR regression is reproducible against the current wrapped retry_after-without-error_code rate-limit fixture.

## Solution Assessment

No, not as currently diffed. The handler-local cooldown is the right fix boundary, but changing the shared rate-limit helper is not the narrowest maintainable solution because it affects send and webhook retry callers outside this PR's stated scope.
