Keep this PR open. Current main still has only the Telegram 401 sendChatAction guard, while this PR adds the right narrow transient cooldown behavior; however, the branch also narrows the shared Telegram rate-limit helper used by message delivery and webhook retry paths, so it needs one focused repair before merge.

## What This Changes

The PR adds a Telegram sendChatAction transient cooldown with regression tests, test runtime injection for the handler, structured rate-limit detection changes, and a changelog entry.

## Best Possible Solution

Land a version that keeps the dedicated Telegram sendChatAction transient cooldown and regression coverage while preserving the existing shared Telegram rate-limit helper contract for send and webhook retry paths. Yes. The current-main unit test for non-401 sendChatAction failures shows repeated 500 calls are retried and rejected without cooldown, and the PR regression is reproducible by checking isTelegramRateLimitError against the existing wrapped retry_after fixture.

## Solution Assessment

No, not as currently diffed. The sendChatAction cooldown is the right narrow fix, but narrowing the exported rate-limit helper is broader than the PR's stated scope; restoring the helper fallback or adding a chat-action-only strict helper is safer.
