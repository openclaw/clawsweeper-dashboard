## Summary

Keep open. The sendChatAction cooldown is useful and current main still lacks it, but the PR branch narrows a shared Telegram rate-limit helper in a way that would drop existing wrapped retry_after retry behavior for send and webhook callers.

## What This Changes

This PR adds a Telegram sendChatAction transient cooldown with regression tests, updates shared Telegram rate-limit detection tests, and adds an Unreleased changelog entry.

## Best Possible Solution

Land the sendChatAction transient cooldown and regression coverage, but preserve the existing shared Telegram rate-limit helper contract. If the chat-action path needs stricter classification, add a separate chat-action-only helper instead of narrowing the exported helper used by send and webhook retry paths.

## Reproduction Assessment

Yes. Current main's unit coverage shows non-401 sendChatAction failures are retried on every call, and the PR regression is visible by comparing the PR head helper against current main's wrapped retry_after rate-limit tests.

## Solution Assessment

No, not as currently diffed. The cooldown is the right narrow fix, but changing the shared rate-limit helper is broader than the stated sendChatAction scope; restoring the fallback or using a local strict helper is safer.
