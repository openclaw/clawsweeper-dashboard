## Summary

Keep this PR open: the Telegram sendChatAction cooldown is still useful and current main does not implement it, but the latest head regresses the shared Telegram rate-limit helper used outside typing indicators.

## What This Changes

The branch adds transient cooldown state to the Telegram sendChatAction handler, expands unit coverage, changes shared Telegram rate-limit helper tests/behavior, and adds a changelog entry.

## Best Possible Solution

Land the sendChatAction transient cooldown and regression coverage after preserving the shared rate-limit fallback contract, or make any stricter 429 handling local to chat actions.

## Reproduction Assessment

Yes. Current main source and tests show non-401 sendChatAction failures are retried on every call; the PR-specific regression is reproducible from the diff against the shared rate-limit helper and existing wrapped retry_after send test.

## Solution Assessment

No, not as currently diffed. The cooldown is the right narrow fix, but changing the shared isTelegramRateLimitError contract is broader than the stated sendChatAction scope and can break message/webhook retries.
