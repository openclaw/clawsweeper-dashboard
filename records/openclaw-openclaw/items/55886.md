## What This Changes

## Best Possible Solution

Layer transient cooldown into the current Telegram sendChatAction handler while preserving minInterval coalescing and broad shared rate-limit classification, then land with focused tests and redacted live Telegram transport proof.

## Reproduction Assessment

Yes, at source level. Current main routes typing through this handler, the handler only special-cases 401 failures before rethrowing, and the dedicated tests still expect repeated non-401 failures to reject without transient cooldown.

## Solution Assessment

No, not as currently diffed. Handler-local cooldown is the right boundary, but this head must preserve current chat-action coalescing and shared rate-limit fallback behavior before it is a maintainable fix.
