## Summary

Keep open. Current main has real runtime-dependency and startup mitigations, including newer hot-path repair reductions, but the reported Windows npm-global startup/status/recovery regression is not fully proven fixed: `gateway status` still relies on WebSocket probing, the local status fallback does not directly cover pre-connect `ECONNREFUSED`, and Windows service recovery still has waits that can explain a 30-60s degraded path.

## What This Changes

The issue requests faster Windows Gateway startup, status probing, and service recovery for npm-global installs, especially stale process handling, `ECONNREFUSED` probe transitions, missing `gateway.cmd` repair, and bundled runtime dependency installs.

## Best Possible Solution

Keep the issue open and split the remaining work by root cause after a Windows trace: preserve the runtime-deps mitigations already on main, link the narrower runtime-deps tracker where applicable, then implement a focused `gateway status` readiness/fallback improvement or Windows service recovery change only where current-main evidence confirms the delay.
