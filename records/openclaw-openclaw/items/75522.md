## Summary

Keep open. Current main contains partial startup/runtime-dependency mitigations, but the reported Windows Gateway startup/status/recovery problem is not proven fixed: the lock still defaults to a 5000ms timeout, local status fallback still uses Gateway RPC rather than HTTP readiness, WebSocket probe failures can still surface as connect failures, and Windows service recovery still has multi-second process/port waits.

## What This Changes

Reports slow Windows Gateway startup, status probing, and service recovery after npm-global install, especially stale process cleanup, `connect ECONNREFUSED` probes, missing `gateway.cmd` repair, bundled runtime-deps staging, and Bonjour/provider stalls.

## Best Possible Solution

Keep this open and split the work by measured root cause: preserve the related runtime-deps repair from #75500, collect a fresh Windows `OPENCLAW_GATEWAY_STARTUP_TRACE=1` run for the missing-`gateway.cmd` scenario, then make focused changes only where the trace proves remaining lock, status/readiness, service recovery, or plugin/mDNS startup delay.

## Reproduction Assessment

Unclear. The report includes concrete Windows steps and detailed logs, but this read-only review did not live-run the npm-global `gateway.cmd` deletion scenario against current main; source inspection shows the reported paths remain plausible.

## Solution Assessment

Unclear. The requested direction is too broad for one maintainable patch; the safer path is a targeted Windows trace followed by narrow fixes for the proven bottleneck rather than closing as implemented or bundling unrelated startup changes.
