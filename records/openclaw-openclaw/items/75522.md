## Summary

Keep open. Current main has useful runtime-dependency and readiness mitigations, but it does not prove the broader Windows Gateway startup/status/recovery regression is fixed: the lock timeout remains 5000ms, local status fallback still uses Gateway RPC rather than direct HTTP readiness, and Windows service cleanup still has multi-second kill and port-release waits.

## What This Changes

The issue requests faster, more reliable Windows Gateway startup/status/service recovery for npm-global installs, including stale-process cleanup, readiness probing, plugin runtime dependency staging, and clearer diagnostics.

## Best Possible Solution

Keep this open, preserve the already-landed runtime-deps mitigations, collect a fresh Windows `OPENCLAW_GATEWAY_STARTUP_TRACE=1` trace for the gateway.cmd deletion scenario, then split fixes by measured root cause across status/readiness probing, lock/service recovery timing, runtime dependency staging, and mDNS/provider stalls.

## Reproduction Assessment

Unclear. The report has concrete Windows steps and logs, and current source still contains the relevant paths, but this read-only review did not live-run the Windows npm-global `gateway.cmd` deletion scenario on current main.

## Solution Assessment

No for a one-shot broad change. The maintainable path is to use the Windows trace to choose narrow fixes, rather than combining lock timeout, HTTP readiness fallback, dependency packaging, service repair, and mDNS/provider startup changes in one PR.
