## Summary

Keep open. Current main has several real mitigations for gateway startup and bundled runtime dependencies, but it does not prove the reported Windows startup/status/recovery regression is fully fixed: gateway startup still has a pre-start bundled dependency repair path, the local status fallback does not directly cover pre-connect ECONNREFUSED cases, and the 5s gateway lock timeout the report calls out is still present.

## What This Changes

The issue asks OpenClaw to make Windows gateway startup, status probing, and service recovery faster and clearer after stale processes, missing gateway.cmd, and bundled plugin runtime dependency staging delays.

## Best Possible Solution

Keep this open and split the work by root cause after a Windows repro: route bundled runtime dependency staging to the existing runtime-deps trackers, verify gateway.cmd deletion and Scheduled Task recovery on current main, and implement a narrow local status/readiness classification improvement only if the WebSocket probe still misclassifies a listening but not ready gateway.

Unclear. The issue includes credible Windows logs and concrete steps, and source review confirms several relevant code paths, but this read-only review did not live-run Windows npm-global startup/recovery on current main.

## Solution Assessment

Unclear. Current main contains useful partial fixes, but the best maintainable solution is likely a set of targeted fixes after Windows trace evidence rather than closing this as already solved or making one broad probe/dependency change.
