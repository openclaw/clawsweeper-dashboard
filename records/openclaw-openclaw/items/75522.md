## Summary

Keep open. Current main contains meaningful runtime-dependency and startup mitigations, but the Windows Gateway startup/status/recovery regression described here is not proven fixed: status fallback still depends on Gateway RPC/WebSocket, the gateway lock still defaults to a 5000 ms acquisition window, and Windows Scheduled Task stop/restart still has multi-second process and port-release waits.

## What This Changes

This issue requests faster and clearer Windows Gateway startup, status probing, and service recovery, including stale-process cleanup, readiness fallback behavior, bundled plugin dependency staging, and diagnostics.

## Best Possible Solution

Keep the issue open and split the remaining work by root cause: preserve the shipped/runtime-deps improvements, collect a fresh Windows `OPENCLAW_GATEWAY_STARTUP_TRACE=1` trace for the `gateway.cmd` deletion scenario, then make focused changes only where the trace proves remaining lock, status/readiness, service recovery, mDNS, or provider-startup delays.
