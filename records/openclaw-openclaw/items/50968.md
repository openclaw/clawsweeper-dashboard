Keep open: the underlying bug is still present on current main, but this branch is not merge-ready because its LiteLLM changes target deleted command-owned files, the PR is currently conflicting, and one token-metadata path can still downgrade stored limits.

## What This Changes

The PR adds LiteLLM-compatible `/model/info` probes during Copilot Proxy and LiteLLM onboarding so configured models can use proxy-reported context and output-token limits instead of static defaults.

## Best Possible Solution

Port model-info probing into the active plugin-owned LiteLLM and Copilot Proxy onboarding paths, require complete token metadata before patching persisted limits, preserve stored/manual limits on fallback, and add focused tests plus redacted real behavior proof.

## Reproduction Assessment

Yes, source-level. Current main still assigns static 128000/8192 limits in Copilot Proxy and LiteLLM onboarding, so a proxy reporting larger `/model/info` limits would still be configured with the smaller values; I did not run a live proxy.

## Solution Assessment

No. Querying model-info is the right direction, but this branch targets deleted LiteLLM command files and treats partial token metadata as discovered; the safer solution is to port the logic to the plugin-owned onboarding paths with stricter metadata handling.
