## Summary

Keep open: current main still hardcodes LiteLLM and copilot-proxy model limits, but this PR is stale against the pluginized LiteLLM onboarding path, has one remaining correctness issue, and still needs real behavior proof before merge.

## What This Changes

The PR makes copilot-proxy and LiteLLM onboarding query a LiteLLM-compatible `/v1/model/info` endpoint and use returned token limits when writing model definitions.

## Best Possible Solution

Land a refreshed change that implements model-info discovery in current plugin-owned LiteLLM and copilot-proxy onboarding, preserves existing limits when fields are absent or fetch fails, and adds regression coverage for large-context models.

## Reproduction Assessment

Yes. Source inspection on current main shows both onboarding paths still write 128k/8192 defaults; the concrete runtime repro is to run onboarding against a LiteLLM-compatible proxy whose `/model/info` advertises a larger context window and inspect the written model config.

## Solution Assessment

No, not as-is. The endpoint-discovery direction is the right fix, but the PR must be rebased onto the pluginized LiteLLM path and must avoid treating partial token metadata as authoritative.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Avoid treating partial LiteLLM info as discovered:** `src/commands/onboard-auth.config-litellm.ts:95`
  - body: If `/v1/model/info` returns a matching `model_info` with only one token-limit field, this still sets `discovered: true` while substituting 128k/8192 for the missing field. The later patch then treats those defaults as authoritative and can overwrite a previously correct or manually tuned existing model entry; patch only fields actually returned, or keep `discovered` false unless both limits are numeric.
  - confidence: 0.86
