## Summary

Keep open. The underlying provider-onboarding bug is still present on current main, but this PR is not merge-ready because its LiteLLM half targets deleted legacy command files and the remaining probe logic can treat incomplete token metadata as authoritative.

## What This Changes

The PR adds LiteLLM-compatible `/model/info` probing during onboarding for Copilot Proxy and legacy LiteLLM command setup so discovered `max_input_tokens` and `max_output_tokens` can replace static 128k/8192 model limits.

## Best Possible Solution

Implement the fix in the current plugin-owned onboarding paths: add a small LiteLLM-compatible model-info probe for `extensions/litellm` and `extensions/copilot-proxy`, map complete `max_input_tokens`/`max_output_tokens` metadata into model definitions, preserve existing tuned limits when probing fails or returns partial data, and cover first-run plus re-auth behavior with focused plugin tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure LiteLLM or Copilot Proxy with a `/v1/model/info` response reporting a larger context window, run onboarding, and observe that current main still writes static 128000/8192 limits from the plugin model builders.

## Solution Assessment

No. The intended direction is right, but this branch is not the best current solution because it attaches the LiteLLM implementation to deleted legacy command files and can patch fallback defaults as if incomplete metadata were discovered.
