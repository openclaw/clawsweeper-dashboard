## Summary

Keep open. Current main still has the reported gap: transcript-backed usage recovery only treats provider/model identity, nonzero usage, derived totals, or cost as a usable snapshot, so a contentful local/OpenAI-compatible transcript with no usage telemetry still returns no fallback estimate and can leave status/session context usage empty or effectively zero.

## What This Changes

The issue requests a conservative transcript-derived context estimate for local/OpenAI-compatible sessions when provider usage telemetry is missing or zero.

## Best Possible Solution

Add a generic bounded fallback in the gateway transcript usage path: provider telemetry remains authoritative, but contentful transcripts with no usable usage snapshot produce a conservative prompt/context estimate that status and session-list displays treat as usable. Cover the content-only no-telemetry case with regression tests and keep the stale-total guard intact.

## Reproduction Assessment

Yes. The current `usage-empty` fixture creates a contentful transcript with no provider usage telemetry and asserts the usage reader returns null, which matches the reported empty or zero context display path without needing a live local model.

## Solution Assessment

Yes. A narrow extractor/status fallback is the best repair path because it preserves provider-reported usage and existing stale-token protections while giving local no-telemetry sessions a conservative context meter.
