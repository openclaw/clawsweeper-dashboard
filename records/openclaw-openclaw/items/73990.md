## Summary

Keep open. Current main still has the reported no-telemetry path: contentful local/OpenAI-compatible transcripts without meaningful assistant usage produce no usable transcript usage snapshot, and the status/session-list consumers then keep context usage empty or zero. The related open report #75357 is about requesting upstream streaming usage; this issue still covers the separate fallback case when usage telemetry is absent.

## What This Changes

The issue requests a conservative transcript-derived context estimate for local/OpenAI-compatible sessions when provider usage telemetry is missing or zero.

## Best Possible Solution

Add a generic bounded transcript-content fallback in the gateway/status usage path: provider usage stays authoritative, but contentful transcripts with no usable usage snapshot produce a conservative prompt/context estimate that status and session-list displays treat as usable.

## Reproduction Assessment

Yes. The current `usage-empty` fixture creates a contentful transcript with no usage telemetry and expects the transcript reader to return null, which matches the reported empty or zero context display path.

## Solution Assessment

Yes. A narrow extractor/status fallback is the best maintainable direction because it preserves real provider telemetry, handles no-telemetry local sessions generically, and leaves #75357's transport-specific usage capture as separate work.
