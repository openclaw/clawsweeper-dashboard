## Summary

Keep open. Current main still drops contentful local/OpenAI-compatible transcripts when provider usage telemetry is absent: the transcript extractor returns null without a usage/model snapshot, and both status and session-list fallbacks require a usable positive usage snapshot before they can show context progress. The open #75357 transport-telemetry report is related but does not supersede this no-telemetry fallback request.

## What This Changes

Request a conservative transcript-derived context usage fallback for local/OpenAI-compatible sessions when provider usage telemetry is missing or zero.

## Best Possible Solution

Add a generic, bounded transcript-content fallback in the Gateway/status usage path: preserve valid provider usage first, then estimate a conservative prompt/context token count from visible transcript text only when no positive usage-derived context total exists, with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes. The current `usage-empty` fixture creates a contentful user/assistant transcript without usage telemetry and asserts that transcript usage is null; the status/session-list paths then drop that missing snapshot before formatting context usage.

## Solution Assessment

Yes. The requested narrow extractor/status fallback is the best maintainable direction because provider transports and usage normalization remain authoritative whenever they supply usable telemetry.
