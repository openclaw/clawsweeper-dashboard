## Summary

Keep open. Current main still lacks a transcript-content fallback when provider usage telemetry is absent: contentful local/OpenAI-compatible transcripts can produce no usable usage snapshot, and both status and session-list consumers drop null or non-positive totals before displaying context usage. The related #75357 transport report is adjacent, but it does not replace this no-telemetry fallback request.

## What This Changes

Add a conservative transcript-derived context estimate for local/OpenAI-compatible sessions when provider usage telemetry is missing or zero.

## Best Possible Solution

Implement a generic, bounded transcript-content fallback in the gateway/status usage path: keep provider telemetry authoritative when present, otherwise estimate prompt/context tokens from renderable transcript text and mark that fallback as usable for status and session-list displays.

## Reproduction Assessment

Yes. The current `usage-empty` fixture creates a contentful transcript with no usage telemetry and expects the reader to return null, which matches the reported permanent zero/empty context display path.

## Solution Assessment

Yes. A narrow extractor/status fallback is the maintainable fix because provider transports remain authoritative when they provide usable telemetry, while local no-telemetry sessions still get a conservative context meter.
