## Summary

Keep this open. Current main still drops contentful no-telemetry transcripts in the usage extractor, so status and session-list context displays can remain empty or effectively zero despite the docs saying transcript fallback should prevent custom-provider sessions from collapsing to zero.

## What This Changes

Local/OpenAI-compatible sessions without provider usage telemetry should fall back to a conservative transcript-derived context estimate instead of showing zero or empty context usage.

## Best Possible Solution

Add a generic bounded fallback in the gateway/status transcript usage path: provider telemetry stays authoritative, but contentful transcripts with no usable usage snapshot produce a conservative prompt/context estimate that status and session-list displays treat as usable. Reuse the repo's existing token-estimation helpers where possible, and add regression coverage across the extractor, session list, status rendering, and `session_status`.

## Reproduction Assessment

Yes. The current `usage-empty` fixture creates a contentful transcript with no assistant usage telemetry and expects the reader to return null, matching the reported empty or zero context display path.

## Solution Assessment

Yes. A narrow extractor/status fallback is the maintainable fix because it preserves provider telemetry while giving local no-telemetry sessions a conservative context meter; a provider-specific workaround would leave other OpenAI-compatible no-telemetry backends with the same gap.
