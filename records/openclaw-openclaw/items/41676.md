## Summary

Keep open: current main still directs macOS launchd `gateway.err.log` to an uncapped service stderr file, while the existing 100 MB rotation applies to the structured gateway logger path. The recent live macOS disk-pressure comments and related JSONL/backup reports are useful context, but they do not supersede this daemon service-log retention bug.

## What This Changes

The issue requests bounded rotation or rate limiting so repeated gateway WebSocket heartbeat output cannot grow `gateway.err.log` without a size or retention cap.

## Best Possible Solution

Add bounded retention for daemon-managed gateway stdout/stderr logs, preferably reusing the existing `logging.maxFileBytes` default/archive policy while keeping JSONL diagnostics and backup pruning in their separate linked issues.
