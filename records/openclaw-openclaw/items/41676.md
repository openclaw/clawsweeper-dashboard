- `P1`: Real macOS installs reported gateway logs reaching multi-GB sizes and leaving very low free disk space, which can degrade desktop gateway availability.

## Summary

Keep open. Current main and the latest release suppress macOS LaunchAgent stderr, so the original `gateway.err.log` spam path is partially mitigated, but daemon-owned gateway log retention is still not implemented and launchd stdout remains a fixed append target without a max-size or rotation policy.

## What This Changes

The issue asks for rate limiting or deduplication of repeated gateway heartbeat timeout logs plus bounded rotation or max-size controls for runaway gateway log files.

## Best Possible Solution

Land or replace the focused daemon log-retention work with current-main semantics: keep stderr suppression, bound the active launchd stdout sink, decide any legacy stderr cleanup policy explicitly, and leave backup amplification and diagnostic JSONL rotation to their separate linked items.

## Reproduction Assessment

Yes, at source level for the remaining retention gap: current main still writes macOS LaunchAgent stdout to a fixed file and has no daemon max-size or rotation implementation. I did not live-reproduce the original heartbeat spam loop on current main.

## Solution Assessment

No, current main is only a partial solution because stderr is suppressed but the active daemon stdout log remains unbounded. The maintainable fix is a maintainer-approved daemon retention policy with focused tests for install, restart, and legacy cleanup behavior.
