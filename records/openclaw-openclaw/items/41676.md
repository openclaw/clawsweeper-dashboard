## Summary

Keep open. Current main still bounds the internal JSONL logger, but the managed macOS Gateway stdout/stderr files are fixed `gateway.log` and `gateway.err.log` paths, and console capture still mirrors every `console.error` to stderr, so repeated heartbeat output can still grow the service error log without a cap. Related #76407/#75380 cover diagnostic JSONL files, not this LaunchAgent stderr surface.

## What This Changes

The issue asks for bounded or rate-limited handling of repeated Gateway WebSocket heartbeat timeout output so `gateway.err.log` cannot grow without limit.

## Best Possible Solution

Bound managed Gateway stdout/stderr with a default retention policy tied to the existing logging limits where possible, and add generic storm handling for repeated low-value errors without special-casing only the heartbeat string.

## Reproduction Assessment

No high-confidence live reproduction for the exact heartbeat storm was established in this read-only review. Source inspection plus two user reports show a credible current-main path: managed launchd stderr is fixed and unbounded while console errors continue to be mirrored there.

## Solution Assessment

Yes, the maintainable direction is to cap the managed service stdout/stderr surface, preferably reusing the existing `logging.maxFileBytes` model, and avoid a string-specific heartbeat workaround. The related diagnostic JSONL PR is not enough because it does not touch `gateway.err.log`.
