## Summary

Keep open. Current main mitigates the original macOS stderr spam by sending launchd stderr to `/dev/null`, but the launchd-owned stdout log remains a fixed append-only file with no daemon retention hook, and the focused retention PR is still open and conflicting.

## What This Changes

The issue requests bounded, rate-limited, or configurable gateway daemon log retention so repeated heartbeat failures cannot grow gateway logs to multi-GB files.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/79892 with current-main daemon retention for launchd-owned logs, while keeping diagnostic JSONL rotation and backup amplification scoped to their separate canonical issues.

## Reproduction Assessment

Yes, for the remaining daemon retention gap by source inspection: current main still writes launchd stdout to a fixed `gateway.log` path without a max-size or rotation hook. The original active stderr heartbeat-spam path is partly mitigated because macOS launchd stderr is now suppressed to `/dev/null`.

## Solution Assessment

No, current main is only a partial solution. The narrow maintainable fix is daemon-owned retention at launchd install/restart boundaries, with default/config semantics decided by maintainers.
