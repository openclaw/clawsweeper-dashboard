## Label Justifications

- `P1`: Independent users observed multi-gigabyte gateway logs and near-disk-exhaustion, creating an urgent reliability problem for long-running desktop gateways.
- `impact:crash-loop`: Unbounded output from an always-running gateway can exhaust disk and make the gateway or host unavailable.

## Summary

Current main still writes launchd-managed gateway stdout and stderr to persistent files without a retention bound, and two independent macOS reports show multi-gigabyte growth. The exact historical heartbeat message is absent from current source, but that does not resolve the general disk-exhaustion path; related open work covers different diagnostic JSONL files and backup amplification, so this issue remains the canonical gateway service-log item.

## What This Changes

Add bounded retention for gateway stdout/stderr logs and prevent repeated WebSocket heartbeat timeout output from filling disk.

## Best Possible Solution

Adopt a lifecycle-owned bounded default for launchd gateway stdout/stderr, prove the active-process truncate or reopen behavior on a real macOS install, and address heartbeat-specific flooding only where a current emitter is identified; keep diagnostic JSONL rotation and backup exclusions in their separate canonical items.

## Reproduction Assessment

Yes for the central issue: current source demonstrably directs launchd gateway output to persistent files without a tested retention bound, and two installations provide concrete multi-gigabyte results. The exact historical heartbeat-timeout emitter is no longer identifiable in current source.

## Solution Assessment

No, the suggested multi-option configuration surface is not yet the best first solution. A bounded daemon-owned default with real launchd proof is narrower and more maintainable, with operator knobs considered only if maintainers decide the default is insufficient.
