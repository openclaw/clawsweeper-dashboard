## Summary

Keep open: current main still leaves macOS launchd gateway stdout/stderr service logs uncapped, while the existing 100 MB rotation only covers the separate structured logger file. Recent duplicate-user evidence and source inspection make the remaining daemon service-log retention bug actionable.

## What This Changes

The issue asks OpenClaw to prevent `gateway.log` and `gateway.err.log` from growing without bound under repeated gateway/WebSocket error output.

## Best Possible Solution

Add bounded retention for daemon-owned gateway stdout/stderr logs, preferably reusing the existing `logging.maxFileBytes` size and archive model while keeping diagnostic JSONL rotation and backup pruning in their separate issues.

## Reproduction Assessment

Yes, for the remaining bug by source inspection: current main maps macOS launchd stdout/stderr to `gateway.log` and `gateway.err.log` and emits those paths directly in the plist with no retention layer. I did not live-reproduce the old heartbeat-spam loop because that exact string is no longer present in current source.

## Solution Assessment

Yes: the narrow maintainable solution is service-log retention around the launchd stdout/stderr files, likely sharing the existing logger rotation defaults. Broader WebSocket heartbeat changes, diagnostic JSONL rotation, and backup exclusions should stay separate unless fresh evidence ties them to this same root cause.

## Repair Work Prompt

Fix the open gateway service-log retention bug reported at https://github.com/openclaw/openclaw/issues/41676. Current main rotates the structured `logging.file` JSONL log, but macOS launchd still writes daemon stdout/stderr directly to `$OPENCLAW_STATE_DIR/logs/gateway.log` and `gateway.err.log` with no cap. Add bounded retention for those daemon-owned service logs, preferably by reusing or mirroring the existing `logging.maxFileBytes` default/archive behavior without adding a new product policy knob unless the existing config cannot apply cleanly.
