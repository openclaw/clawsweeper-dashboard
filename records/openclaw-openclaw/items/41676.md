## Summary

Current main still writes gateway stdout/stderr through platform service log files without a repository-level max-size or rotation policy for `gateway.err.log`, and the repeated heartbeat line is still emitted by the WebSocket heartbeat path. The report is valid and narrow enough for a focused fix, while related items only cover other logs or backup amplification.

## What This Changes

The issue requests bounded gateway error logging and de-duplication or rate limiting for repeated WebSocket heartbeat timeout lines that can make `gateway.err.log` grow to multi-GB sizes.

## Best Possible Solution

Add a bounded gateway log lifecycle for active `gateway.log` and `gateway.err.log`, plus targeted rate limiting or aggregation for repeated heartbeat timeout stderr lines, with tests covering rotation/retention and duplicate suppression.

## Reproduction Assessment

No. Current source shows the exact repeated stderr line and the unbounded active `gateway.err.log` path, but I did not reproduce the long-lived WebSocket failure loop against current main in a live gateway.

## Solution Assessment

Unclear. The requested bounded log policy is directionally right, but maintainers should choose whether the durable fix belongs in launchd service setup, gateway-managed rotation, a shared logging helper, heartbeat rate limiting, or a combination.
