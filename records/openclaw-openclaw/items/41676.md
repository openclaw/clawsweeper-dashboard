## Summary

Keep open: current main still leaves the macOS daemon-managed `gateway.err.log` path as an uncapped service stderr file, while the existing 100 MB rotation applies to the structured gateway logger rather than launchd stdout/stderr. Recent discussion also adds live macOS disk-pressure evidence, and the related JSONL diagnostics and backup issues cover adjacent but separate surfaces.

## What This Changes

Bound daemon-managed gateway stdout/stderr logging so repeated WebSocket heartbeat or close noise cannot grow `gateway.err.log` without limit.

## Best Possible Solution

Add bounded retention for daemon-managed gateway stdout/stderr logs, preferably reusing the existing `logging.maxFileBytes` default/archive policy, while leaving JSONL diagnostic rotation and upgrade backup pruning to their separate linked items.

## Reproduction Assessment

Yes for the unbounded log-growth path: source inspection shows current main directs launchd stderr/stdout to plain service files with no cap. I did not live-reproduce the exact heartbeat-spam loop in this worker.

## Solution Assessment

Yes: a narrow service-log retention fix is the best maintainable path for this issue. It should avoid expanding into #75380/#76407 JSONL diagnostics or #76502 backup policy except for cross-references.
