## Summary

Keep open: current main still routes launchd-managed gateway stdout/stderr to plain `gateway.log` and `gateway.err.log` files with no active size cap, while the existing 100 MB rotation applies to the structured JSONL gateway logger. The later related reports cover JSONL diagnostics and backup amplification, not this daemon stderr/stdout retention path.

## What This Changes

The issue requests bounded rotation, rate limiting, or max-size control for repeated gateway heartbeat timeout output that can make `gateway.err.log` grow to multi-GB sizes.

## Best Possible Solution

Add bounded retention for daemon-managed gateway stdout/stderr logs, preferably reusing the existing `logging.maxFileBytes` default/archive policy while handling launchd’s open-file behavior correctly.

## Reproduction Assessment

Yes for the unbounded growth path by source inspection: launchd writes to plain `gateway.log` and `gateway.err.log` files with no cap. I did not live-reproduce the exact heartbeat timeout spam loop on macOS current main.

## Solution Assessment

Yes: a narrow daemon service-log retention fix is the best maintainable direction. It should avoid expanding into #75380/#76407 JSONL diagnostics or #76502 backup policy except for cross-references.
