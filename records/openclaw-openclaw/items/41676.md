## Summary

Keep open. Current main still routes macOS launchd stderr directly to `gateway.err.log` without a retention hook, while `logging.maxFileBytes` only covers the structured Gateway file logger; the newer live comment showing a 13 GB active `gateway.err.log` confirms this remains actionable.

## What This Changes

Request bounded retention or rate limiting for `gateway.err.log` when repeated WebSocket heartbeat timeout messages flood macOS Gateway supervisor logs.

## Best Possible Solution

Add bounded retention for macOS launchd supervisor logs, keeping it separate from structured JSONL logging and documenting the default policy; treat heartbeat-line dedupe and backup-directory cleanup as related follow-ups unless they naturally fall out of the same narrow fix.

## Reproduction Assessment

Yes for the unbounded supervisor-log class: current source resolves launchd stderr to `gateway.err.log` and writes it through `StandardErrorPath` without a rotation or truncation hook. I did not live-reproduce the exact heartbeat flood, and that exact text is not present in first-party source.

## Solution Assessment

Yes, bounded supervisor-log retention is the narrowest maintainable first fix because it protects users from any noisy stderr producer. Producer-specific heartbeat dedupe is safer as a follow-up once the source of that exact line is identified.
