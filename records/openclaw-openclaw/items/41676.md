## Summary

Keep open. Current main and v2026.5.19 partially mitigate the original macOS stderr path by sending launchd stderr to `/dev/null`, but the launchd-owned stdout sink is still a fixed file with no daemon retention hook, and the focused retention PR remains open and conflicting.

## What This Changes

The issue requests bounded gateway logging for repeated heartbeat timeout spam, with max-size or rotation control for gateway logs.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/79892 with current-main daemon retention for launchd-owned stdout, make any legacy stderr cleanup explicit, and keep diagnostic JSONL rotation and backup amplification scoped to their separate open issues.

## Reproduction Assessment

Yes, at source level for the remaining daemon retention gap: current main still writes macOS launchd stdout to a fixed path and has no daemon retention implementation. I did not live-reproduce the original heartbeat spam loop on current main.

## Solution Assessment

No, current main is only a partial solution. The narrow maintainable path is daemon-owned retention at launchd install/restart boundaries, with maintainer-approved default and configuration semantics.
