## Summary

Keep open. Current main suppresses macOS launchd stderr so the original active `gateway.err.log` sink is no longer the same, but it still has no daemon-owned retention for the actual launchd stdout log and the related retention PR is still open and conflicting.

## What This Changes

The issue asks for bounded rotation or retention for macOS gateway service logs so repeated heartbeat timeout output cannot grow `gateway.err.log` without limit.

## Best Possible Solution

Rebase or replace https://github.com/openclaw/openclaw/pull/79892 so it bounds the launchd-owned log files that current main actually writes, documents the suppressed stderr behavior, and leaves heartbeat dedupe, CPU-loop, and backup amplification follow-ups in their narrower issue tracks.

## Reproduction Assessment

Yes for the remaining daemon stdout retention gap by source inspection: current main still writes launchd stdout to a fixed `gateway.log` path without a retention hook. No live current-main reproduction was established for the original heartbeat spam into active `gateway.err.log`, and current Darwin stderr is now `/dev/null`.

## Solution Assessment

No, not fully solved yet. Suppressing stderr addresses the exact active `gateway.err.log` sink, but the maintainable fix still needs a current-main retention policy for actual launchd-owned logs and a rebased decision on any legacy stderr cleanup or heartbeat dedupe.
