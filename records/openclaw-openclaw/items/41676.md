## Summary

Keep open: current main still routes macOS launchd gateway stdout/stderr directly to `gateway.log` and `gateway.err.log` without a service-log retention cap, while the existing 100 MB rotation applies to the separate structured logger path. The exact legacy heartbeat timeout string is no longer present in current source, but recent user evidence and the uncapped launchd sink show the disk-growth bug remains actionable.

## What This Changes

Requests bounded retention, rotation, or rate limiting for gateway service logs so repeated WebSocket heartbeat timeout output cannot grow `gateway.err.log` indefinitely.

## Best Possible Solution

Add bounded retention for daemon-owned gateway stdout/stderr logs, preferably reusing the existing `logging.maxFileBytes` default/archive policy, while leaving JSONL diagnostics and backup pruning to their separate linked items.

## Reproduction Assessment

Yes, for the remaining bug: source inspection shows current main directs macOS launchd stdout/stderr to plain `gateway.log` and `gateway.err.log` files with no max-size enforcement. The old exact heartbeat-spam string was not found in current source, so that specific loop was not live-reproduced.

## Solution Assessment

Yes: the narrow maintainable fix is service-log retention around the launchd stdout/stderr files, not broad WebSocket heartbeat changes or the separate JSONL diagnostics and backup surfaces.
