## Summary

Keep open. Current main has bounded structured file logs, but the daemon service still maps launchd stdout/stderr directly to `gateway.log` and `gateway.err.log` without a max-size or retention policy; related diagnostics and backup items do not cover this active service-log sink.

## What This Changes

The issue asks for bounded or rotated `gateway.err.log` handling and suppression of repeated WebSocket heartbeat timeout spam.

## Best Possible Solution

Add a documented bounded lifecycle for daemon-owned `gateway.log` and `gateway.err.log`, with focused daemon tests and targeted suppression or aggregation for repeated low-value stderr loops.

## Reproduction Assessment

No high-confidence live reproduction for the original heartbeat loop. Source inspection does confirm the active macOS service stderr file is still an uncapped plain `gateway.err.log`, and later user evidence reports the same disk-growth symptom on live installs.

## Solution Assessment

Unclear. Bounded service-log retention is the right direction, but maintainers should choose the policy and integration point before an automated fix changes launchd-managed log behavior.
