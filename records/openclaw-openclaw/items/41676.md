- `P1`: Multiple macOS reports show gateway logs reaching multi-GB sizes and driving low-disk availability risk on desktop installs.
- `impact:crash-loop`: The issue is about process-level availability and disk-pressure risk from runaway gateway service logging.

## Summary

Keep open. Current main and the latest release suppress active macOS launchd stderr, but they still have no daemon-owned retention or max-size policy for launchd stdout, and the focused retention PR remains open, conflicting, and policy-sensitive.

## What This Changes

The issue asks for bounded or rate-limited gateway service logging so repeated heartbeat or gateway errors cannot grow `gateway.err.log`/`gateway.log` indefinitely.

## Best Possible Solution

Land or replace the focused daemon log-retention work for launchd-owned stdout with maintainer-approved defaults and controls, keep current stderr suppression, and leave backup amplification plus diagnostic JSONL rotation to their separate canonical items.

## Reproduction Assessment

Yes, at source level for the remaining gap: current main still writes macOS launchd stdout to a fixed append target and has no daemon retention implementation. I did not live-reproduce the original heartbeat spam loop against current main.

## Solution Assessment

No, current main is only a partial mitigation because stderr is suppressed while launchd-owned stdout remains unbounded. The maintainable solution needs daemon-owned retention at install/restart boundaries plus an approved default/config policy.
