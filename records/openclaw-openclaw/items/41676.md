## Summary

Keep open. Current main partially mitigates the original macOS stderr spam by suppressing launchd stderr, but launchd stdout is still a fixed append-only file with no daemon retention hook, and the focused retention PR remains open and dirty.

## What This Changes

The issue requests bounded rotation, max-size control, or deduplication for macOS gateway logs that can grow to multi-GB files under repeated heartbeat or socket failures.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/79892 with current-main daemon retention for launchd-owned logs, while keeping diagnostic JSONL rotation and backup amplification scoped to their separate canonical issues.

## Reproduction Assessment

Yes, by source inspection for the remaining daemon retention gap: current main still writes launchd stdout to a fixed path without a max-size or rotation hook. The original active stderr spam path is partly mitigated because macOS launchd stderr is now suppressed.

## Solution Assessment

No, current main is only a partial solution. The narrow maintainable fix is daemon-owned retention at launchd install/restart boundaries, with default and configuration semantics decided by maintainers.
