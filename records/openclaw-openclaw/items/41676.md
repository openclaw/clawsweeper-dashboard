## Label Justifications

- `P1`: Runaway gateway logs have reached multi-GB sizes on real macOS installs and can create disk-pressure and desktop reliability failures.
- `impact:crash-loop`: The issue is about process-level availability pressure from runaway gateway logging and possible disk exhaustion.

## Summary

Keep open. Current main mitigates the original macOS stderr sink by sending launchd stderr to `/dev/null`, but macOS launchd stdout is still a fixed append-only file with no daemon retention hook, and the focused retention PR remains open and dirty against main.

## What This Changes

The issue asks for bounded gateway log growth through heartbeat-noise suppression and log rotation or max-size controls for gateway stdout/stderr logs.

## Best Possible Solution

Rebase, revise, or replace https://github.com/openclaw/openclaw/pull/79892 with current-main daemon retention for launchd-owned stdout, make legacy stderr cleanup an explicit decision, and keep backup amplification and diagnostic JSONL rotation in their separate open threads.

## Reproduction Assessment

Yes, at source level for the remaining daemon retention gap: current main still writes macOS launchd stdout to a fixed path and has no daemon retention implementation. I did not live-reproduce the original heartbeat spam loop on current main.

## Solution Assessment

No, current main is only a partial mitigation. The maintainable path is daemon-owned retention at launchd install/restart boundaries, with maintainer-approved defaults and configuration semantics.
