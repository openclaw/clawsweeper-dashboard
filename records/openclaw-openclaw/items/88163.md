## Summary

Keep open. The central pairing-token stability fix is useful, but the PR should not merge as-is because the new reuse guard can preserve malformed stored token values that current main would remint during approval.

## What This Changes

The PR changes device pairing approval to reuse a valid existing same-key device token during metadata refreshes while continuing to rotate for scope expansion, key repair, revoked or issuer-tagged tokens, and malformed role metadata.

## Best Possible Solution

Keep the bounded same-key token reuse behavior, but rotate instead of reusing when the stored token value is missing or blank, with a regression test for that persisted-state repair path.

## Reproduction Assessment

Yes from source inspection: current main remints requested role tokens on approval, while the PR can reuse an unrevoked entry without validating the stored token value. I did not run tests because this review was read-only.

## Solution Assessment

No as submitted. The direction is the narrow maintainable fix, but the reuse predicate should also prove the stored token value is usable before preserving it.

Full review comments:

- **[P2] Reject malformed stored tokens before reuse:** `src/infra/device-pairing.ts:543`
  - body: `canReuseExistingApprovedToken` treats any unrevoked token entry as reusable without checking that `token` is a non-empty string. Please rotate unless the stored token value is structurally usable and add a regression for that state.
