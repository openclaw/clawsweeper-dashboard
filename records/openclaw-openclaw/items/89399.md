## Summary

Keep open: current main has partial SecretRef infrastructure for gateway-snapshot command resolution, local fallback, and read-only degradation, but it still intentionally treats exec SecretRef execution as opt-in for doctor/config/secrets flows. That does not fully implement the requested consistent exec-resolution layer across CLI, gateway, and runtime contexts.

## What This Changes

Requested consistent exec-based SecretRef resolution across CLI, gateway, and runtime contexts so diagnostics and non-interactive environments do not report false missing secrets.

## Best Possible Solution

Define one security-aware SecretRef resolution policy for CLI diagnostics, gateway snapshots, and service runtimes, keeping exec execution explicit and documenting when gateway snapshot resolution versus local execution is used.

## Reproduction Assessment

No high-confidence failing current-main reproduction is available from the issue alone. Source inspection does show the relevant behavior class: some CLI paths use gateway snapshot/local fallback, while doctor and secrets/config checks keep exec providers opt-in.

## Solution Assessment

Unclear; standardization is likely useful, but the best implementation needs a maintainer decision on the security policy for when CLI tools may execute or delegate exec SecretRefs.
