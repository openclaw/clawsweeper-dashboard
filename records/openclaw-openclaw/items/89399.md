## Summary

Keep open: current main has targeted gateway-backed SecretRef resolution for command paths and explicit `--allow-exec` diagnostics, but it does not implement the requested consistent exec-based resolution layer across CLI, gateway, doctor, and runtime contexts. The linked external resolver is useful reference context, but the core change remains a security- and compatibility-sensitive product decision.

## What This Changes

Request standardized exec-based SecretRef resolution so CLI, gateway, diagnostics, and runtime contexts behave consistently without changing SecretRef semantics.

## Best Possible Solution

Design a secops-reviewed resolver/delegation contract that preserves safe defaults, uses gateway snapshots for supported command paths, and keeps exec provider execution explicit for diagnostics and headless environments.

## Reproduction Assessment

No high-confidence live reproduction is available. Source inspection does show the relevant divergence: command paths can use gateway snapshot/local fallback, while doctor and diagnostics intentionally skip exec SecretRefs unless `--allow-exec` is set.

## Solution Assessment

Unclear as a direct implementation because a universal resolver would change security and fallback policy. The safer solution is an explicit maintainer-approved design for which contexts may delegate to gateway resolution, run exec locally, or degrade.
