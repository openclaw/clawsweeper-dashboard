## Summary

Keep open: current main and v2026.6.9 support exec SecretRefs, runtime snapshots, explicit command-path opt-ins, and pluginIntegration providers, but they still do not define the requested shared OS/session/headless exec-resolution contract across CLI, gateway, and runtime contexts.

## What This Changes

The issue requests a standardized, backward-compatible exec SecretRef resolution layer so CLI, gateway, diagnostics, scheduled tasks, and agent runtime contexts resolve the same provider consistently.

## Best Possible Solution

Define a maintainer-approved generic exec SecretRef context contract or plugin API extension that preserves explicit exec opt-in, documents headless limitations, and proves at least one real service-context keychain flow before implementation.

## Reproduction Assessment

No high-confidence live reproduction was established for a concrete keychain or scheduler setup. Source inspection does show current main lacks a shared cross-context exec resolver contract and still resolves exec providers from the caller/provider env model.

## Solution Assessment

Unclear: a shared resolver contract may be the right direction, but the best boundary depends on maintainer choices about core versus plugin ownership, explicit exec consent, and credential-session security.
