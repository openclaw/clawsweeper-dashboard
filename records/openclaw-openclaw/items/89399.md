## Summary

Keep open: current main and the latest release have explicit exec opt-ins, runtime snapshots, command-path fallback behavior, and pluginIntegration provider seams, but they still do not define the requested cross-context OS/session/headless exec SecretRef contract.

## What This Changes

The issue requests a standardized exec SecretRef resolution layer so CLI, gateway, diagnostics, and agent runtime contexts behave consistently for keychain-style providers.

## Best Possible Solution

Define a maintainer-approved generic exec SecretRef context contract or plugin API extension that preserves explicit exec opt-in, documents headless limitations, and proves at least one real service-context keychain flow before implementation.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does show current main lacks a shared cross-context exec resolver contract and still resolves exec providers from the caller/provider env model.

## Solution Assessment

Unclear: a shared resolver contract may be the right direction, but the best boundary depends on maintainer choices about core versus plugin ownership, explicit exec consent, and credential-session security.
