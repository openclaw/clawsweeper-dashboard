## Summary

Keep open: current main still skips tool-id normalization for same-model OpenAI Responses replay and then splits stored ids directly into `function_call` and `function_call_output` payload fields; PR #74850 is already the open linked fix candidate.

## What This Changes

The issue requests OpenAI Responses replay id normalization so stored `{call_id}|{item_id}` tool identifiers cannot produce invalid `function_call` or `function_call_output` payload fields.

## Best Possible Solution

Review and land PR #74850, or an equivalent focused patch, that normalizes malformed or overlong OpenAI Responses replay ids at the replay boundary while preserving canonical reasoning replay pairs and adding regression coverage.

## Reproduction Assessment

Yes. The issue includes a no-credential payload-capture repro, and current main still skips normalization for same-model stored ids before splitting them into Responses payload fields.

## Solution Assessment

Yes. A Responses-specific replay-boundary normalizer is the narrowest maintainable fix because enabling generic OpenAI tool-id sanitization would risk regressing native reasoning replay that intentionally preserves canonical `call_*|fc_*` pairs.
