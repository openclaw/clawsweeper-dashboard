## Summary

Keep open. Current main still has a same-model OpenAI Responses replay path that can pass malformed or overlong `call_id|item_id` pairs through to the next provider payload, and open PR #74850 is already the linked implementation candidate with closing syntax.

## What This Changes

The issue reports that OpenAI Responses replay can re-emit stored `call_id|item_id` tool identifiers without normalizing malformed or overlong call and item ids, causing follow-up provider request rejection after tool use.

## Best Possible Solution

Review and land PR #74850, or an equivalent focused patch, that normalizes invalid OpenAI Responses `call_id` and `fc_*` item ids at the Responses replay boundary while preserving canonical reasoning replay pairs and adding regression coverage for malformed and overlong ids.

## Reproduction Assessment

Yes. The issue includes a no-credential payload-capture reproduction, and current source inspection shows the same-model replay path skips the transport normalizer before raw id splitting into Responses payload fields.

## Solution Assessment

Yes. A Responses-specific replay-boundary normalizer is the narrowest maintainable direction because current OpenAI replay policy deliberately avoids generic tool-call id sanitization to preserve native reasoning replay.
