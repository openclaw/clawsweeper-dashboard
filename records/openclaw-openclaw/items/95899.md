## Summary

Keep open: the PR targets a real restart-recovery UX gap, but it changes a fail-closed assistant-tail branch into automatic resume without the confirmation or idempotency boundary required by the linked issue. It also only proves the behavior with unit tests, and the requested no-retype behavior still has uncovered cases.

## What This Changes

The PR changes main-session restart recovery to extract the last user transcript message for assistant-tail interruptions and resume the session with a recovery prompt, plus focused unit-test updates.

## Best Possible Solution

Preserve the no-retype direction, but land it with a confirm-to-continue or idempotency-proven assistant-tail boundary, a transcript lookup that can find the original request beyond the recent tail window, and an explicit follow-up or design for stale approval-pending recovery.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.9` send the unresumable notice and fail a marked running main session when the recent transcript tail is an assistant message. I did not run a live gateway-restart reproduction in this read-only review.

## Solution Assessment

No. Re-presenting the user request is the right direction, but this implementation is not the best fix until assistant-tail recovery is confirm/idempotency-gated and the recovered request lookup is not limited to the recent transcript window.
