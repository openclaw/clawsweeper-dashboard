## Label Justifications

- `P1`: The PR changes a security-sensitive command-execution path for untrusted channel auto-replies and still has a source-proven fail-open blocker.
- `merge-risk: 🚨 compatibility`: The intended deny default changes behavior for existing non-owner channel workflows that relied on implicit exec access or stored exec policy.
- `merge-risk: 🚨 security-boundary`: Merging this PR as-is can leave unauthenticated channel senders with a path to permissive host exec when partial agent defaults are configured.

## Summary

Current main still lacks sender-aware auto-reply exec override resolution, so the security direction is useful, but the PR should stay open because the head patch can still fail open for partial operator exec defaults and it lacks real channel/runtime proof.

## What This Changes

The PR adds `senderIsOwner` to auto-reply exec override resolution, forwards the command owner flag from reply directive resolution, and adds resolver tests for owner and non-owner exec override behavior.

## Best Possible Solution

Revise the non-owner path so user-controlled inline/session exec fields are ignored and incomplete operator defaults fail closed or are completed with `security=deny` and `ask=always`, then rebase and coordinate the final command-execution boundary with https://github.com/openclaw/openclaw/pull/64719.

## Reproduction Assessment

Yes, at source level. Evaluate the PR-head resolver with `senderIsOwner=false` and partial `agentExecDefaults` such as `{ host: "gateway" }`, then follow current exec fallback where missing `security` and `ask` resolve to `full` and `off`.

## Solution Assessment

No. The direction is right, but returning partial operator defaults unchanged is not a safe command-execution boundary; the safer path is to require complete explicit safe policy or fill missing policy fields with deny/always for non-owner senders.
