## Label Justifications

- `P1`: The PR changes a live command-execution security boundary for untrusted channel auto-replies and still has a source-proven fail-open blocker.
- `merge-risk: 🚨 compatibility`: The deny default changes behavior for existing non-owner channel workflows that relied on implicit exec access or stored exec policy.
- `merge-risk: 🚨 security-boundary`: Returning partial agent exec defaults unchanged can preserve permissive host exec for non-owner channel senders.

## Summary

Keep open: the security direction is useful, but the PR still has a source-proven fail-open case for partial agent exec defaults, conflicts with current main, overlaps with broader channel tool hardening, and lacks real behavior proof.

## What This Changes

This PR passes `senderIsOwner` into auto-reply exec override resolution, denies exec by default for non-owner senders without overrides, ignores non-owner inline/session exec overrides, and adds resolver tests.

## Best Possible Solution

Align the non-owner exec boundary with current tool-policy direction, ignore user-controlled inline/session exec state for non-owner senders, fail closed or complete incomplete operator defaults with `security=deny` and `ask=always`, and coordinate the final boundary with https://github.com/openclaw/openclaw/pull/64719 before merge.

## Reproduction Assessment

Yes, at source level. Evaluate the PR-head resolver with `senderIsOwner=false` and partial `agentExecDefaults` such as `{ host: "gateway" }`; the returned overrides omit `security` and `ask`, and current exec runtime fills those with `full` and `off`.

## Solution Assessment

No. The owner/non-owner split is the right area, but returning partial agent defaults unchanged is not a safe fail-closed solution; maintainers should choose and document how explicit non-owner exec policy works.
