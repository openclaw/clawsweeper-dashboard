## Summary

Keep open: this is a real security hardening path, but the PR branch still has a fail-open case for partial non-owner exec defaults and only provides unit-test proof for a runtime command-execution boundary.

## What This Changes

The PR adds sender ownership to auto-reply exec override resolution, forwards `command.senderIsOwner`, and adds unit coverage for non-owner deny defaults plus ignored inline/session exec overrides.

## Best Possible Solution

Revise the non-owner auto-reply exec boundary so user-controlled inline/session exec fields are ignored and missing non-owner policy fields fail closed, then coordinate the final tool-access model with https://github.com/openclaw/openclaw/pull/64719 before merge.

## Reproduction Assessment

Yes. A focused source-level reproduction is to call the PR version of `resolveReplyExecOverrides` with `senderIsOwner: false` and `agentExecDefaults` containing only `host` or `node`, then follow downstream exec defaults to `security=full` and `ask=off`.

## Solution Assessment

No. Ignoring inline/session overrides for non-owner senders is the right direction, but returning partial operator defaults unchanged is not the narrowest safe fix for a command-execution policy boundary.
