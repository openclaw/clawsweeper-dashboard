## Summary

Keep this PR open. Current main still lacks sender-aware auto-reply exec override resolution, but the proposed patch has a blocking security fail-open for partial operator exec defaults and overlaps with the broader dangerous-tool hardening in #64719.

## What This Changes

The PR adds `senderIsOwner` to auto-reply exec override resolution, forwards the command owner flag from the directive call site, and adds unit coverage for non-owner deny defaults plus ignored inline/session exec overrides.

## Best Possible Solution

Revise the non-owner auto-reply exec boundary so user-controlled inline/session exec fields are ignored and missing operator policy fields fail closed with `security=deny` and `ask=always`, then coordinate the final tool-access model with #64719 before merge.

## Reproduction Assessment

Yes. A focused unit reproduction is to call the PR version of `resolveReplyExecOverrides` with `senderIsOwner: false` and `agentExecDefaults` containing only `host` or `node`, then verify downstream exec construction can still fall through to host defaults of `full` and `off`.

## Solution Assessment

No. The PR is the right direction, but returning partial operator defaults as-is is not the safest fix; merge in deny/always for missing non-owner policy fields or require explicit safe `security` and `ask` before allowing non-owner exec.
