## Summary

Keep open: the PR is aimed at a real command-execution security gap, but the current branch still returns partial non-owner operator exec defaults unchanged, so missing `security` and `ask` can fall through to permissive host defaults; it also lacks real channel/runtime proof and is currently conflicting with main.

## What This Changes

The PR adds `senderIsOwner`-aware auto-reply exec override resolution, forwards the owner flag at the directive call site, and adds resolver tests for non-owner deny defaults plus ignored inline/session exec overrides.

## Best Possible Solution

Revise the non-owner path so user-controlled inline/session exec fields are ignored and incomplete operator defaults either fail closed or are completed with safe `security=deny` and `ask=always`, then rebase and coordinate the final command-execution boundary with https://github.com/openclaw/openclaw/pull/64719.

## Reproduction Assessment

## Solution Assessment

No. The direction is right, but returning partial operator defaults unchanged is not a safe command-execution boundary; require complete safe policy or fill missing fields with deny/always for non-owner senders.
