## Summary

Keep this PR open, but do not merge it as-is. Current main already avoids treating empty cron output as an interim acknowledgement, while the remaining raw-stdout output recovery is still missing and this PR's transcript-scan approach introduces correctness and output-integrity problems.

## What This Changes

The PR adds keyword-based raw-stdout prompt detection, scans the session transcript for the latest successful toolResult text, uses that text as cron output when payload text is empty, and adds a regression test.

## Best Possible Solution

Expose a verified latest successful current-attempt tool result from the embedded runner, feed it into the cron outcome path only when payload text and final assistant-visible text are empty, and preserve the interim retry unless recovery proves terminal success.

## Reproduction Assessment

Yes. A focused mock of an isolated cron run with empty payloads and no final assistant-visible text shows the raw-stdout output gap on current main, and the PR diff directly shows the introduced retry suppression when a prompt matches the raw-stdout keyword gate.

## Solution Assessment

No. The requested behavior is reasonable, but prompt keyword detection plus transcript-file recovery is not the narrowest maintainable fix; a typed current-attempt runner result would be safer and more reliable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate retry suppression on recovered tool output:** `src/cron/isolated-agent/run.ts:730`
  - body: `rawStdoutCronRequest` is derived from prompt text, then used to disable the interim-ack follow-up before `rawStdoutToolResultText` is proven. A matching raw-stdout job that first replies with an acknowledgement and no successful tool result will now exit incomplete instead of retrying to finish the cron task.
  - confidence: 0.92
- **[P2] Reject tool results with unverified timestamps:** `src/cron/isolated-agent/run.ts:226-228`
  - body: `readLatestSuccessfulToolResultText` only skips entries when the timestamp parses and is older than `runStartedAt`, so missing or unparsable timestamps are accepted. That can promote a stale transcript toolResult as this run's output; require verified current-run state or a valid timestamp at or after the run start.
  - confidence: 0.83
- **[P3] Use an isolated current-run transcript fixture:** `src/cron/isolated-agent/run.interim-retry.test.ts:102-109`
  - body: The test writes a fixed `/tmp/transcript.jsonl` before the runner captures `runStartedAt`, so the timestamp filter can exclude the fixture under clock advancement and the path can collide across tests. Use a per-test path and a timestamp guaranteed to fall inside the run window.
  - confidence: 0.86

## Security Review
