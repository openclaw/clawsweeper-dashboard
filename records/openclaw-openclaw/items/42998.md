## Summary

Keep open for maintainer handling: current main already covers the empty/NO_REPLY duplicate-rerun portion via https://github.com/openclaw/openclaw/pull/41401, but this PR still attempts meaningful raw tool-result output recovery and the proposed implementation has correctness/security blockers plus missing real behavior proof.

## What This Changes

The PR adds raw-stdout prompt detection, reads the latest successful toolResult from the session transcript, suppresses interim-ack retries for those cron requests, and adds a regression test.

## Best Possible Solution

Expose verified current-attempt tool-result text from the embedded runner to cron under an explicit completion contract, while preserving current main's empty/NO_REPLY no-retry behavior.

## Reproduction Assessment

Yes, by source inspection rather than a live cron run: current main no longer retries empty payloads, but arbitrary non-NO_REPLY toolResult text is still not an input to resolveCronPayloadOutcome. The PR-introduced blockers are also source-reproducible from the diff.

## Solution Assessment

No. Prompt keyword detection plus transcript-file scanning is not the best fix because it can suppress valid retries and promote stale output; typed current-attempt runner state is the safer contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate retry suppression on recovered tool output:** `src/cron/isolated-agent/run.ts:730`
  - body: The raw-stdout guard disables the interim-ack follow-up based on prompt text before a successful tool result has been recovered. A raw-stdout job that first replies with an acknowledgement and no captured tool result can now exit incomplete instead of running the continuation that completes the task.
  - confidence: 0.92
- **[P2] Reject tool results with unverified timestamps:** `src/cron/isolated-agent/run.ts:226-228`
  - body: The transcript reader only skips entries when a timestamp parses and is older than runStartedAt, so missing or unparsable timestamps are accepted. That can promote stale transcript output as this run's cron result; require verified current-run state or a valid timestamp inside the run window.
  - confidence: 0.83
- **[P3] Use an isolated current-run transcript fixture:** `src/cron/isolated-agent/run.interim-retry.test.ts:102-109`
  - body: The test writes a fixed /tmp/transcript.jsonl before the runner captures runStartedAt, so clock advancement can exclude the fixture and the path can collide across tests. Use a per-test path and a timestamp guaranteed to be in the run window.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff adds no dependency or CI supply-chain changes, but transcript-based cron output promotion creates a concrete output integrity and privacy concern.

Concerns:

- **[medium] Avoid transcript-based cron output promotion:** `src/cron/isolated-agent/run.ts:226`
  - body: The patch reads the session transcript and promotes the latest successful toolResult text to cron output based on prompt keywords and permissive timestamp handling. If a stale or malformed transcript entry is selected, OpenClaw could deliver unrelated tool output to the cron delivery target; prefer verified current-attempt runner state.
