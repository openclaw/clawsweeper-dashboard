## Summary

Keep this PR open, but do not merge it as-is: current main already prevents empty cron output from being treated as an interim acknowledgement, while the PR’s raw-output recovery path still relies on prompt keywords and transcript scraping that can suppress valid continuations or promote unrelated tool output.

## What This Changes

The PR adds raw-stdout prompt detection, scans the session transcript for the latest successful toolResult text, uses that text as cron output when payload text is empty, and adds an interim-retry regression test.

## Best Possible Solution

Expose verified current-attempt tool-result text from the embedded runner to cron under an explicit completion contract, and keep the generic interim-ack retry path intact.

## Reproduction Assessment

Yes for the remaining raw-output gap by source inspection: an isolated cron run with empty payloads and a successful non-NO_REPLY toolResult still leaves outputText undefined on current main. The original empty-payload retry is not reproduced on current main because empty text is explicitly not classified as interim.

## Solution Assessment

No. Prompt keyword detection plus transcript-file recovery is not the best fix because it can suppress valid continuation retries and promote stale or unrelated transcript output; typed current-attempt runner state would be safer.
