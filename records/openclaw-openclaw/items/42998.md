## Summary

Keep open: current main has already shipped the empty/NO_REPLY no-rerun behavior, but this PR also tries to promote arbitrary raw tool stdout and its transcript-based patch has blocking correctness and output-integrity issues.

## What This Changes

The PR adds raw-stdout prompt detection, scans the session transcript for the latest successful toolResult text, uses that as cron output when payload text is empty, and adds a regression test.

## Best Possible Solution

Expose verified current-attempt tool result text from the embedded runner and let cron consume it only under a clear completion contract, while preserving the generic interim acknowledgement retry path.

## Reproduction Assessment

Yes. Source inspection shows a cron isolated run with empty payloads, no final assistant-visible text, and a successful non-NO_REPLY toolResult still leaves resolveCronPayloadOutcome without raw stdout text.

## Solution Assessment

No. Prompt keyword detection plus transcript-file recovery is not the best fix because it can suppress valid continuation retries and can promote stale or unrelated transcript output; a typed current-attempt runner result would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate retry suppression on recovered tool output:** `src/cron/isolated-agent/run.ts:730`
  - body: The raw-stdout guard disables the interim-ack follow-up based on prompt text before a successful tool result has been recovered. A raw-stdout job that first replies with an acknowledgement and no captured tool result can now exit incomplete instead of running the continuation that completes the task.
  - confidence: 0.92
- **[P2] Reject tool results with unverified timestamps:** `src/cron/isolated-agent/run.ts:226-228`
  - body: The transcript reader only skips entries when a timestamp parses and is older than runStartedAt, so missing or unparsable timestamps are accepted. That can promote stale transcript output as this run's cron result; require verified current-run state or a valid timestamp inside the run window.
  - confidence: 0.83
- **[P3] Use an isolated current-run transcript fixture:** `src/cron/isolated-agent/run.interim-retry.test.ts:102-109`
  - body: The test writes a fixed /tmp/transcript.jsonl before the runner captures runStartedAt, so clock advancement can exclude the fixture and the path can collide across tests. Use a per-test path and a timestamp guaranteed to be in the run window.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff adds no dependency or CI supply-chain changes, but transcript-based output promotion creates a concrete cron output integrity/privacy concern.

Concerns:

- **[medium] Avoid transcript-based cron output promotion:** `src/cron/isolated-agent/run.ts:203`
  - body: The patch reads the session transcript and promotes the latest successful toolResult text to cron output based on prompt keywords and permissive timestamp handling. If a stale or malformed transcript entry is selected, OpenClaw could deliver unrelated tool output to the cron delivery target; prefer verified current-attempt runner state.
