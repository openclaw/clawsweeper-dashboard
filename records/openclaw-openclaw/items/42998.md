## Summary

Keep open. Current main has removed the narrow empty-output retry symptom, but it still does not implement the PR's intended raw-stdout completion behavior, and the proposed patch introduces blocking retry and transcript-integrity problems.

## What This Changes

The PR adds raw-stdout prompt detection, scans the session transcript for the latest successful toolResult text, uses that text as the cron output when payload text is empty, and adds a regression test.

## Best Possible Solution

Expose a verified latest successful current-attempt tool result from the embedded runner, feed it into the cron outcome path only when payload text and final assistant-visible text are empty, preserve the interim retry unless recovery proves terminal success, and cover raw stdout success plus no-tool, stale, errored, malformed, and exact `NO_REPLY` cases.

## Reproduction Assessment

Yes. A focused mock of an isolated cron run with empty payloads and empty final assistant-visible text shows current main returns no raw stdout output, and the PR diff shows a matching raw-stdout prompt would skip retry even if no successful tool result was recovered.

## Solution Assessment

No. The requested product behavior is reasonable, but the PR's keyword gate plus transcript-file scan is not the safest solution; a typed current-attempt result from the embedded runner would be narrower and more reliable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate the retry skip on recovered tool output:** `src/cron/isolated-agent/run.ts:730`
  - body: `rawStdoutCronRequest` is derived from prompt text, then used to disable the interim-ack follow-up before `rawStdoutToolResultText` is proven. A matching raw-stdout job that first replies with an acknowledgement and no successful tool result will now exit incomplete instead of retrying to finish the cron task.
  - confidence: 0.92
- **[P2] Reject transcript entries without valid run timestamps:** `src/cron/isolated-agent/run.ts:226-228`
  - body: `readLatestSuccessfulToolResultText` only skips entries when the timestamp parses and is older than `runStartedAt`, so missing or unparsable timestamps are accepted. That can promote a stale transcript toolResult as this run's output; require verified current-run state or a valid timestamp at or after the run start.
  - confidence: 0.81
- **[P3] Use an isolated transcript fixture timestamp:** `src/cron/isolated-agent/run.interim-retry.test.ts:102-109`
  - body: The test writes `/tmp/transcript.jsonl` before `runCronIsolatedAgentTurn` captures `runStartedAt`, so the new timestamp filter can exclude the fixture under normal clock advancement, and the fixed path can collide across runs. Use a per-test path and write a timestamp guaranteed to fall inside the run window.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR adds no dependency or CI supply-chain changes, but its transcript-based output promotion creates a concrete cron output integrity/privacy concern.

Concerns:
