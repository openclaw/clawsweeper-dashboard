## Summary

Keep this PR open, but do not merge it as-is. Current main already avoids treating empty cron output as an interim acknowledgement, while the remaining raw-stdout tool-result recovery is still missing and this patch has correctness plus output-integrity concerns.

## What This Changes

The PR adds raw-stdout prompt detection, scans the session transcript for the latest successful toolResult text, uses that text as cron output when payload text is empty, and adds a regression test.

## Best Possible Solution

Expose a verified latest successful current-attempt tool result from the embedded runner and let cron consume it only when payload text and final assistant-visible text are empty, while preserving interim retry otherwise.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction for the remaining output gap: a cron isolated run with empty payloads, no final assistant-visible text, and a successful tool result leaves `resolveCronPayloadOutcome` without raw stdout text.

## Solution Assessment

No. The requested behavior is reasonable, but prompt keyword detection plus transcript-file recovery is not the narrowest maintainable fix; a typed current-attempt runner result would be safer and more reliable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Gate retry suppression on recovered tool output:** `src/cron/isolated-agent/run.ts:730`
  - body: `rawStdoutCronRequest` disables the interim-ack follow-up before a successful `rawStdoutToolResultText` is proven. A matching raw-stdout job that first replies with an acknowledgement and no captured tool result can now exit incomplete instead of running the continuation that completes the cron task.
  - confidence: 0.92
- **[P2] Reject tool results with unverified timestamps:** `src/cron/isolated-agent/run.ts:226-228`
  - body: `readLatestSuccessfulToolResultText` only skips entries when a timestamp parses and is older than `runStartedAt`, so missing or unparsable timestamps are accepted. That can promote stale transcript output as this run's cron result; require verified current-run state or a valid timestamp inside the run window.
  - confidence: 0.83
- **[P3] Use an isolated current-run transcript fixture:** `src/cron/isolated-agent/run.interim-retry.test.ts:102-109`
  - body: The test writes a fixed `/tmp/transcript.jsonl` before the runner captures `runStartedAt`, so the timestamp filter can exclude the fixture under clock advancement and the path can collide across tests. Use a per-test path and a timestamp guaranteed to be in the run window.
  - confidence: 0.86
