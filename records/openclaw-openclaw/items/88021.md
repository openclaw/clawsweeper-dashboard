## Label Justifications

- `P2`: This is a bounded restart-recovery delivery improvement with user-visible channel reply loss, but it is not a security, data-loss, crash-loop, or urgent runtime-wide outage.
- `impact:message-loss`: The reported recovered final reply is written to session output but not delivered to the original channel.
- `impact:session-state`: The remaining behavior depends on restart-aborted main-session state and stored delivery context metadata.

## Summary

Keep open: current main and the latest release still hard-code restart recovery resumes with `deliver: false`, so the channel reply loss described here remains possible. The fix needs a maintainer decision on the freshness and target-validation boundary before recovered finals are delivered back into channel queues.

## What This Changes

Request optional channel delivery of final replies produced by main-session restart recovery when a valid delivery context still exists.

## Best Possible Solution

Add a generic restart-recovery delivery policy that uses only a validated, fresh delivery context and preserves session-only fallback for no-target or unsafe cases.

## Reproduction Assessment

No high-confidence live reproduction was established. The source path is reproducible from current main because `resumeMainSession` always sends `deliver: false` and the matching test asserts that behavior.

## Solution Assessment

Unclear. Flipping every resumed call to `deliver: true` is not the best fix because existing freshness coverage shows stale channel finals are dangerous; the maintainable path is a gated recovery-delivery policy.
