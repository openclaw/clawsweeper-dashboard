## Summary

Keep this PR open, but it is not ready to merge: the branch adds a useful pending-question context seam, yet the exact head does not implement the claimed resolved-commitment dismissal path and has a retry-cache bug that can re-drain pending questions inconsistently.

## What This Changes

The branch adds session heartbeat pending-question state, queues delivered commitment heartbeat text into that state, drains it into user-triggered prompt builds, reserves the session slot, and adds focused tests.

## Best Possible Solution

Land a corrected narrow fix that persists delivered heartbeat question context once, dismisses resolved pending commitments before delivery, preserves retry drain semantics, and reconciles the overlapping mirror work from https://github.com/openclaw/openclaw/pull/79504.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the PR blockers by source inspection: the exact head still has no resolved extraction/dismissal path, and the retry cache can re-drain absent pending-question context; I did not run tests because this review is read-only.

## Solution Assessment

Is this the best way to solve the issue? Not yet: the pending-question state direction is reasonable, but the branch must implement the resolved-commitment contract it claims and cache empty drain results before it is the narrow maintainable fix.

Store an explicit drained value or sentinel.
