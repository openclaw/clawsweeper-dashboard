## Summary

Keep open: the durable pending-question approach addresses a real source-level gap, but the diff currently broadens transcript mirroring to every visible heartbeat payload instead of only commitment questions, and the run cache does not remember empty pending-question drains.

## What This Changes

The branch adds heartbeat pending-question session state, queues delivered commitment heartbeat questions, drains them into the next user-triggered prompt build with run-scoped caching, and reserves the new session key.

## Best Possible Solution

Keep the pending-question queue as a core session/prompt fix, but scope transcript mirroring to the commitment-question path and make run-scoped pending-question drainage stable for both present and absent context.

## Reproduction Assessment

Yes, at source level: current main can deliver heartbeat commitment questions from an isolated heartbeat run, while prompt construction has no durable pending-question drain and replay filters delivery-mirror assistant entries. I did not run a live Telegram scenario in this read-only review.

## Solution Assessment

No as written: the pending-question queue is the right narrow direction, but the implementation should not start mirroring unrelated visible heartbeat payloads into target transcripts and should cache empty drains per run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Scope heartbeat mirroring to commitment questions:** `src/infra/heartbeat-runner.ts:1965-1975`
  - body: `heartbeatMirror` is built for any visible heartbeat payload, including periodic task, exec, and cron alerts where `dueCommitmentIds` is empty. Passing that mirror to `sendDurableMessageBatch` makes those non-commitment heartbeat runs append `delivery-mirror` assistant records to the target session; current main did not mirror this heartbeat path, and the new pending-question enqueue is already gated to delivered commitments. Scope the mirror the same way, or remove it if the pending queue is enough.
  - confidence: 0.87
- **[P3] Cache empty pending-question drains:** `src/agents/pi-embedded-runner/run/attempt.prompt-helpers.ts:115-127`
  - body: When the first prompt build for a `runId` finds no pending heartbeat question, the cache entry omits `heartbeatPendingQuestionContext`. A retry with the same `runId` falls through and calls the destructive drain again, so retry stability only holds for non-empty context. Store a sentinel or explicit boolean so every first-attempt drain result, including `undefined`, is reused.
  - confidence: 0.78
