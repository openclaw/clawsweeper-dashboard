## What This Changes

The PR adds a shared control-diagnostic token matcher and uses it to suppress leading `NO_SUBPROCESS_STARTED` only for Matrix isolated cron delivery, with regression tests.

## Best Possible Solution

Land the narrow suppression only if maintainers accept that `NO_SUBPROCESS_STARTED` is a Matrix cron control diagnostic rather than a deliverable Matrix message.

## Reproduction Assessment

Yes from source inspection: current `main` only normalizes `NO_REPLY`, so a synthesized leading `NO_SUBPROCESS_STARTED: ...` text would pass through `dispatchCronDelivery` to outbound delivery when a Matrix cron target is resolved. I did not run the test suite because this review is read-only and test commands can write caches or artifacts.

## Solution Assessment

Yes, conditionally: the implementation is the narrowest maintainable fix if maintainers accept Matrix-only suppression. If they do not, the better solution is a channel-neutral or explicitly documented diagnostic delivery policy.
