## Summary

Keep open. Current main still has the existing-config installer fallthrough: after detecting a config and logging that onboarding is skipped, the same branch continues to `Starting setup` and can execute `openclaw onboard`. The large model-list and OpenRouter selection paths appear mitigated, so the remaining actionable part is a narrow installer bug.

## What This Changes

The issue asks to fix Umbrel setup onboarding so users can navigate past model selection, choose OpenRouter, and avoid being forced through the wizard when a config already exists.

## Best Possible Solution

Make the existing-config finalization path terminate or bypass onboarding immediately after doctor/dashboard handling, then add a focused install.sh regression test proving that an existing `OPENCLAW_CONFIG_PATH` or legacy config cannot fall through to `Starting setup` or `openclaw onboard`. Keep model-picker/OpenRouter changes out of scope unless new current-version Umbrel evidence shows they still fail independently.
