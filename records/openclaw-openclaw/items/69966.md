## Summary

Keep open. Current main still models these bootstrap profile knobs as global defaults, the per-agent schema does not accept the requested fields, runtime bootstrap/context resolution reads defaults only, and the Control UI preset path still patches `agents.defaults`. The related #69988 work is useful precedent for per-agent config merging, but it is scoped to compaction rather than these bootstrap/context fields.

## What This Changes

Request per-agent overrides for `contextInjection`, `bootstrapMaxChars`, and `bootstrapTotalMaxChars` so heterogeneous agents do not have to share one global bootstrap profile.

## Best Possible Solution

Add direct optional per-agent overrides under `agents.list[]` with fallback to `agents.defaults`, thread effective values by session agent id through bootstrap/context assembly and reports, and update schema/generated docs/Control UI/docs. Leave a reusable named profile registry for a later product decision unless maintainers explicitly want that larger layer.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a config using the requested `agents.list[]` fields would be rejected by the strict per-agent schema, and the current runtime resolvers ignore agent id and read only `agents.defaults`.

## Solution Assessment

Yes for the direct-override variant. Adding `agents.list[]` overrides with defaults inheritance is the narrowest maintainable fix and matches existing per-agent `contextLimits` precedent; named reusable profiles are broader than needed for the first pass.
