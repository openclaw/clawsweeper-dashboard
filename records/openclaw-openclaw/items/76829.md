## Summary

Keep open: current main still allows prompt `format` failures to reach shared auth-profile cooldown state, and two open PRs are already paired with this issue by closing syntax. See https://github.com/openclaw/openclaw/issues/76829.

## What This Changes

The issue requests that malformed transcript/request-format failures be repaired or isolated so one corrupted session cannot put the shared provider auth profile into cooldown.

## Best Possible Solution

Land one canonical narrow fix that excludes session-local `format` failures from shared auth-profile health while preserving true auth, billing, rate-limit, and overload cooldown behavior.

## Reproduction Assessment

Yes for source-level reproduction, though no live Telegram/Anthropic run was exercised. A prompt 400 classified as `format` flows through `resolveAuthProfileFailureReason` into `markAuthProfileFailure`, which writes shared cooldown state.

## Solution Assessment

Yes, the narrow maintainable solution is to add the `format` carve-out at the auth-profile failure-policy boundary and cover it with focused tests, then review or close the duplicate open PRs accordingly.
