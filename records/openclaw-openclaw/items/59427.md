## Summary

Keep open. Current main has useful opt-in mitigations, but the default normal short-reply path still uses the full prompt/bootstrap/tool assembly model rather than automatic lazy or lightweight context loading.

## What This Changes

The issue requests reducing the default per-message context load for short replies through lazy context loading or a lightweight/full mode policy.

## Best Possible Solution

Define a maintainer-approved default context-cost policy that keeps full context where required, reduces repeated bootstrap/tool-heavy prompt cost for safe short continuations, preserves explicit full mode, and coordinates with the open tool-schema and bootstrap-selection trackers.

## Reproduction Assessment

Yes for the structural issue: current source shows the default path still builds the normal prompt, bootstrap context, and tool surface unless opt-in controls are set. The exact reported token counts need a representative `/context detail` or provider usage check in the reporter’s configured workspace.

## Solution Assessment

Unclear as a concrete implementation. The requested lazy/default lightweight behavior is plausible, but the existing opt-in modes are partial mitigations and a default change needs maintainer policy across bootstrap, tools, memory, safety, and compatibility.
