## Summary

Keep open. Current main has several useful opt-in reductions, but the ordinary default short-reply path still assembles the full OpenClaw prompt, bootstrap context, and tool surface unless users explicitly opt into narrower modes; changing that default remains a cross-cutting product and compatibility decision.

## What This Changes

The issue asks OpenClaw to avoid loading a roughly 40K-token baseline context on every message by lazily or lightly loading prompt, bootstrap, tools, and memory context when possible.

## Best Possible Solution

Define a maintainer-approved default context-cost policy that preserves explicit full mode, coordinates with https://github.com/openclaw/openclaw/issues/14785, https://github.com/openclaw/openclaw/issues/14619, and https://github.com/openclaw/openclaw/issues/61001, and reduces repeated prompt/bootstrap/tool cost only where compatibility is proven.

## Reproduction Assessment

Yes for the source-level structural problem: current main still defaults to full prompt, bootstrap, and tool assembly unless opt-in controls are set. The exact 38K/43K counts still need a representative `/context detail` or provider usage check in the reporter's configured workspace.

## Solution Assessment

Unclear as a concrete implementation. Lazy/default lightweight loading is plausible, but the maintainable fix needs a maintainer policy choice across bootstrap files, tools, memory, safety, and prompt-cache compatibility rather than a narrow code-only patch.
