## Summary

Keep open. Current main has useful opt-in context reductions, but the default short-reply path still builds the normal OpenClaw system prompt, bootstrap context, skills/memory sections, and direct tool surface unless users configure narrower modes; changing that default is a product/API decision, not a conservative cleanup close.

## What This Changes

The issue requests reducing fixed per-message context cost by lazy-loading context for short replies and adding a lightweight/full context mode.

## Best Possible Solution

Define a maintainer-approved default context-cost policy that keeps an explicit full mode, coordinates with the open tool-schema/tooling/bootstrap trackers, and makes any default lazy-loading change only where compatibility and prompt-cache behavior are proven.

## Reproduction Assessment

Yes for the source-level default path: current main still defaults to full prompt/bootstrap/tool assembly unless opt-in settings are configured. The reporter's exact 38K/43K counts still need a representative `/context detail` or provider usage measurement in their workspace.

## Solution Assessment

Unclear as a concrete implementation. Lazy loading and lightweight/full modes are plausible, but the maintainable fix needs a product/API decision across bootstrap files, tools, memory, safety, and prompt-cache compatibility rather than a narrow code-only patch.
