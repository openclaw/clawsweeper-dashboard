## Summary

Keep open: current main still forwards an unchecked parsed `ltm search --limit` value into LanceDB, and this PR is a narrow useful fix with tests, but the changelog wording still needs a small correction and the PR head has failing checks.

## What This Changes

The PR validates memory-lancedb `ltm search --limit`, warns and falls back to 5 for invalid values, adds focused CLI regression tests, and adds a changelog entry.

## Best Possible Solution

Land the scoped CLI validation, warning, and regression tests after correcting the changelog/docs wording and getting the PR head checks green.

## Reproduction Assessment

Yes. On current main, a non-numeric `opts.limit` makes `Number.parseInt` return `NaN`, and the handler passes that value through `MemoryDB.search` into LanceDB `.limit(limit)`.

## Solution Assessment

Yes, mostly. Validating at the memory-lancedb CLI boundary is the narrow maintainable fix; the safer small adjustment is to remove the inaccurate changelog wording or document `ltm search --limit` explicitly.
