## Summary

Keep open. Current main still has a plausible gap for doctor-offered git/source updates on Windows fallback installs: that path runs update doctor and returns without the full `openclaw update` service restart handoff, while update-mode doctor stages non-systemd service metadata instead of installing or migrating a Scheduled Task.

## What This Changes

The issue asks OpenClaw to repair native Windows doctor/update behavior so Startup-folder fallback gateway installs are migrated or clearly reported, launchers are refreshed, and stale gateway processes are replaced after `dist` changes.

## Best Possible Solution

Teach the Windows doctor/update path to refresh the fallback launcher, attempt Scheduled Task migration when allowed, and restart or replace the running gateway, or emit a clear skipped-migration warning when it cannot.

## Reproduction Assessment

No live high-confidence reproduction was run in this Linux checkout. Source inspection supports the reported path: doctor-offered git update runs update doctor without the full restart handoff, and Windows update-mode doctor stages rather than installs/migrates the service.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: reuse the existing Windows install/restart primitives to migrate or clearly report fallback state and replace stale gateway processes after update.
