## Summary

Keep open: current main still lacks the requested lifecycle phases, and this PR is the right area of work but remains incomplete because `agent.wait` can still miss or overwrite `startup-failed` with the trailing `end` event.

## What This Changes

The PR adds embedded subagent first-progress/startup-failed lifecycle events, changes subagent announce cleanup ordering, maps startup failures through registry/session state, updates docs/changelog, and adds lifecycle tests.

## Best Possible Solution

Land the lifecycle signal work after `agent.wait` preserves `startup-failed` as the terminal startup error across the startup-failed-to-end sequence, with focused gateway regression coverage.

## Reproduction Assessment

Yes. Source inspection of PR head shows aborted no-progress runs emit `startup-failed` followed by `end`, while `waitForAgentJob` still ignores `startup-failed` in active waits and can cache the trailing `end` as ok.

## Solution Assessment

No. Additive first-progress/startup-failed phases are a maintainable direction, but the PR needs the gateway wait/cache consumer to treat startup-failed as terminal and resist the follow-up end overwrite.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve startup failures through trailing end events:** `src/gateway/server-methods/agent-job.ts:189`
  - body: The new no-progress abort path emits `startup-failed` and then the normal `end`. This file only adds `startup-failed` to the global listener; active `waitForAgentJob` listeners still drop that phase, and the following `end` can record/cache `ok`, so `agent.wait` can miss the startup failure. Handle `startup-failed` in active waits and keep it terminal across the follow-up end.
