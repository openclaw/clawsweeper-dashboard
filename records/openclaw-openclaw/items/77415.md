## Summary

Keep open: the PR targets a real subagent startup-silence bug that current main does not already solve, but the proposed gateway wait handling still lets the new startup-failure signal be missed or overwritten by the follow-up terminal end event.

## What This Changes

The PR adds embedded subagent first-progress/startup-failed lifecycle signaling, adjusts subagent announce cleanup ordering, maps startup failures through registry/gateway session state, and updates tests, docs, and changelog.

## Best Possible Solution

Land the lifecycle signal work after every gateway wait/session consumer preserves startup-failed as the terminal startup error through the immediately following end event, with regression coverage for that exact event sequence.

## Reproduction Assessment

Yes, source-reproducible: the PR diff makes handleAgentEnd emit startup-failed and then end for aborted no-progress runs, while waitForAgentJob's active listener still only handles end/error. I did not run tests in this read-only review.

## Solution Assessment

No: additive first-progress/startup-failed phases are the right direction, but the current patch is incomplete until agent.wait cannot miss startup-failed or overwrite it with the follow-up end snapshot.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve startup failures for active waits:** `src/gateway/server-methods/agent-job.ts:151`
  - body: The new embedded path emits `startup-failed` and then still emits the normal lifecycle `end` for the same aborted run. `waitForAgentJob`'s active subscription below still ignores every phase except `end`/`error`, so active `agent.wait` calls can miss the startup failure and then resolve/cache the follow-up `end` as `ok`. Please make the wait/cache path preserve `startup-failed` through that terminal sequence and add a regression that emits `startup-failed` followed by `end`.
  - confidence: 0.9
