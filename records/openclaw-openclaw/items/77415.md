## Summary

Keep open: this PR targets the real subagent startup-silence bug, but the current patch still lets active `agent.wait` callers miss `startup-failed` and resolve the following `end` as `ok`.

## What This Changes

The PR adds embedded subagent `first-progress` and `startup-failed` lifecycle signals, adjusts subagent announce cleanup ordering, maps startup failures into registry/session state, and updates tests, docs, and changelog.

## Best Possible Solution

Land after `startup-failed` is preserved as the terminal startup error across both cached and active `agent.wait` listeners, with docs updated for the new lifecycle phases.

## Reproduction Assessment

Yes. Source inspection shows the PR emits `startup-failed` and then `end` for aborted no-progress runs, while the active `waitForAgentJob` listener still ignores `startup-failed` and can resolve the follow-up `end` as `ok`.

## Solution Assessment

No. Additive lifecycle phases are the right direction for #58776, but the patch is incomplete until every gateway wait/cache consumer preserves `startup-failed` through the terminal event sequence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve startup failures for active waits:** `src/gateway/server-methods/agent-job.ts:182-185`
  - body: The new shared listener recognizes `startup-failed`, but the active `waitForAgentJob` subscription below still ignores every phase except `end` and `error`. Since the embedded handler emits `startup-failed` and then `end` for the same no-progress aborted run, a live `agent.wait` can miss the startup failure and resolve/cache the later `end` as `ok`; please treat `startup-failed` as terminal in the active wait path and cover `startup-failed` followed by `end`.
  - confidence: 0.9
- **[P3] Update the lifecycle phase docs:** `docs/tools/subagents.md:409-413`
  - body: This page now documents `first-progress` and `startup-failed`, but `docs/concepts/agent-loop.md` still says lifecycle events are only `start | end | error` and that `agent.wait` waits only for `end/error`. Please update that docs page too so the public lifecycle contract is consistent after this PR merges.
  - confidence: 0.78
