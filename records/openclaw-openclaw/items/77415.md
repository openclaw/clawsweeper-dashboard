## Summary

Keep open: current main still lacks the requested subagent startup lifecycle phases, and this PR moves in the right direction, but the gateway wait/cache path can still turn the new startup-failed event into an ok result when the trailing end event arrives.

## What This Changes

The PR adds embedded subagent first-progress/startup-failed lifecycle events, adjusts subagent announce cleanup ordering, maps startup failures through registry/session state, and updates docs, tests, and changelog.

## Best Possible Solution

Land the lifecycle signal work after agent-job preserves startup-failed as the terminal startup error across the startup-failed -> end sequence, with focused regression coverage for agent.wait.

## Reproduction Assessment

Yes, source-reproducible: PR head emits startup-failed and then end for aborted no-progress runs, while waitForAgentJob still ignores startup-failed in active waits and can consume the trailing end as ok. I did not run tests in this read-only review.

## Solution Assessment

No: additive first-progress/startup-failed phases are the right direction, but the current patch is incomplete until gateway wait/cache consumers preserve startup-failed through the follow-up end event.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve startup failures through the trailing end event:** `src/gateway/server-methods/agent-job.ts:189`
  - body: The embedded handler now emits startup-failed and then the normal end for an aborted no-progress run.
