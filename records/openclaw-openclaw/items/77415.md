## Summary

Keep open: the PR is useful and targets a real lifecycle gap, but it is not merge-ready because `startup-failed` can still be lost or overwritten by gateway consumers. https://github.com/openclaw/openclaw/pull/77415

## What This Changes

The PR adds embedded subagent `first-progress` and `startup-failed` lifecycle signals, adjusts subagent announce cleanup behavior, and updates docs, changelog, and lifecycle tests.

## Best Possible Solution

Land the lifecycle signal work after all gateway wait, cache, session-store, and session-subscriber consumers preserve `startup-failed` as the terminal startup error across the subsequent `end` event.

## Reproduction Assessment

Yes, source-reproducible: PR head emits `startup-failed` followed by `end` for aborted no-progress runs, while active `agent.wait` ignores `startup-failed` and can consume the trailing `end`. I did not execute tests in this read-only review.

## Solution Assessment

No: additive lifecycle phases are the right direction, but the current patch is incomplete until gateway wait/session consumers preserve the startup failure instead of allowing the follow-up `end` to win.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve startup failures in active agent.wait:** `src/gateway/server-methods/agent-job.ts:189`
  - body: The global cache listener now recognizes `startup-failed`, but the per-call `waitForAgentJob` listener still ignores every phase except `end` and `error`. Since the embedded handler emits `startup-failed` and then the normal `end` for the same aborted no-progress run, an active `agent.wait` can miss the failure and resolve/cache ok from the trailing end; handle `startup-failed` in the active waiter and keep that snapshot terminal through the following end.
  - confidence: 0.92
- **[P2] Thread startup-failed through session updates:** `src/gateway/session-lifecycle-state.ts:43`
  - body: The new `session-lifecycle-state` mapping is not reached by the gateway session event pipeline because `server-chat` only finalizes and persists `start`, `end`, and `error`. With this PR's `startup-failed` then `end` sequence, session subscribers and persisted session status can still end on the later `end` status instead of `failed`; include `startup-failed` in the session finalization path or preserve the failed snapshot across the trailing end.
  - confidence: 0.84
