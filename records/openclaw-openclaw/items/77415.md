## Summary

Keep open. Current main still lacks the startup lifecycle behavior, and this PR is the active successor for that bug, but the branch is not merge-ready because it still preclaims cleanup too early and its proof is regression-test output rather than an observed real aborted startup.

## What This Changes

The PR adds first-progress/startup-failed subagent lifecycle handling, gateway wait/session-state updates, cleanup sequencing changes, docs, changelog, and focused tests.

## Best Possible Solution

Land a consolidated lifecycle fix that keeps the additive startup phases, moves the cleanup lease to the point where cleanup actually starts, and includes redacted real-runtime proof from an observed aborted subagent startup.

## Reproduction Assessment

Yes, at source level: current main accepts only start/end/error in the affected consumers and maps aborted lifecycle end events to timeout. I did not run the reporter's live corpus or force a real Gateway startup abort in this read-only review.

## Solution Assessment

No, not as currently written. The first-progress/startup-failed direction is the right narrow fix, but the cleanup preclaim needs to move after the awaited pre-cleanup paths and the external PR still needs real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Move cleanup preclaim after the hook path:** `src/agents/subagent-registry-lifecycle.ts:789-796`
  - body: This preclaims `cleanupHandled` before later awaited lifecycle work, including the non-deferred ended hook. If that hook rejects for a no-completion run, `completeSubagentRun` exits before `startSubagentAnnounceCleanupFlow`, but the persisted record now says cleanup is already handled, blocking later announce/finalize cleanup. Claim the lease immediately before starting cleanup, after those awaited paths.
