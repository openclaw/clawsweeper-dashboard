## Summary

Keep open; the PR addresses the right lifecycle gap, but it leaves `agent.wait` inconsistent so active waiters can still miss `startup-failed` and report the following legacy `end` as success.

## What This Changes

The PR adds subagent `first-progress` and `startup-failed` lifecycle events, adjusts completion announce cleanup, and updates gateway/session mapping, docs, changelog, and focused tests.

## Best Possible Solution

Preserve the additive lifecycle design, but make startup failures propagate consistently through embedded events, subagent registry, session state, and `agent.wait` without being downgraded by the following compatibility `end` event.

## Reproduction Assessment

Yes at source level: the PR emits `startup-failed` before a normal `end`, while `waitForAgentJob` active waiters still ignore that phase and can finish from the following `end` as `ok`. I did not run tests in this read-only review.

## Solution Assessment

No; the overall direction is right, but the implementation is incomplete until `agent.wait` handles `startup-failed` on both cached and active waiter paths and prevents the subsequent legacy `end` from replacing the failure.
