## Summary

Keep open: current main still has the source-level duplication path, and the existing narrow fix PR for this bug remains open rather than merged.

## What This Changes

The issue asks OpenClaw to avoid adding the scheduled-reminder wrapper when a main-session cron `systemEvent` with `delivery.mode: "none"` is already injected into the session.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/44973 or an equivalent narrow marker-and-skip fix with regression coverage and real cron/heartbeat proof, while preserving ordinary cron reminder wrappers and `delivery.mode` semantics.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: main-session cron queues a `cron:` system event, heartbeat peeks that event into the cron wrapper, and reply setup drains the same queue into `System` lines.

## Solution Assessment

Yes. The narrow maintainable fix is to identify main-session cron system events already injected into history and skip only those from cron reminder wrapping, rather than changing general delivery or heartbeat behavior.
