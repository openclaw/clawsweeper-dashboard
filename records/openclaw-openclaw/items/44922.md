## Summary

Keep open: current main and the latest release still contain the duplicate main-session cron prompt path, and the related remediation is still in open or closed-unmerged PRs rather than landed code.

## What This Changes

Cron `sessionTarget: "main"` jobs with `payload.kind: "systemEvent"` and `delivery.mode: "none"` should avoid adding an extra scheduled-reminder wrapper when the event is already present in the main session.

## Best Possible Solution

Land a focused remediation, or a corrected canonical PR, that skips only main-session cron system events already injected as System context while preserving ordinary cron reminders, no-relay behavior, and regression coverage.

## Reproduction Assessment

Yes, at source level. Current main queues a main-session `cron:` system event, heartbeat wraps cron-tagged pending events as scheduled reminders, and reply setup can drain generic queued system events into `System:` prompt context.

## Solution Assessment

Yes. The narrow maintainable fix is to distinguish main-session cron system events that are already injected into history and skip only those from cron reminder wrapping, rather than changing the whole cron delivery contract.
