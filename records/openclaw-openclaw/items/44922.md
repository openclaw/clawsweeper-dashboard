## Summary

Keep open: current main still has the reported main-session cron duplication path, and the targeted fix remains an open PR rather than merged behavior.

## What This Changes

Cron jobs using `sessionTarget: "main"`, `payload.kind: "systemEvent"`, and `delivery.mode: "none"` should inject only the system event without adding a scheduled-reminder wrapper.

## Best Possible Solution

Land, rebase, or replace the targeted fix so main-session cron system events are tagged and excluded only from redundant cron wrapper prompt construction, with focused regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: the main-session cron job enqueues a `cron:` system event, then heartbeat selects that pending event and builds the scheduled-reminder prompt; I did not run a live cron execution in this read-only pass.

## Solution Assessment

Yes. The tag-and-skip approach described in the discussion and open PR is the narrowest maintainable fix because it preserves delivery semantics while suppressing only already-injected main-session cron content.
