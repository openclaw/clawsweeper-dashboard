## Summary

Keep open: current main and the latest shipped tag still contain the cron-to-heartbeat duplicate prompt path, and the related fix PRs are open rather than merged.

## What This Changes

Cron main-session systemEvent jobs should not add a scheduled-reminder wrapper when delivery.mode is none.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/44973 or an equivalent focused remediation that skips only main-session cron systemEvents already injected as System context, with regression coverage and proof that ordinary cron reminders still work.

## Reproduction Assessment

Yes, at source level. Current main queues a main-session `cron:` system event, heartbeat wraps cron-tagged pending events as scheduled reminders, and reply setup can drain the same queued text into the prompt as `System:` context.

## Solution Assessment

Yes, the narrow maintainable fix is to distinguish main-session cron system events that are already injected into history and skip only those from cron reminder wrapping. Broader main-session cron hardening should be a maintainer choice, not required to resolve this duplicate-wrapper bug.
