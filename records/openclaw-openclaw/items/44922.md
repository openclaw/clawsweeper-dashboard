## Summary

Current main still has the reported duplicate model-input path, and the related focused fix has not landed; this issue should stay open.

## What This Changes

A main-session cron job with a systemEvent payload and `delivery.mode: "none"` should not surface the same payload both as a raw `System:` event and as a scheduled-reminder wrapper.

## Best Possible Solution

Land the linked focused fix or an equivalent narrow change that distinguishes main-session cron systemEvent injections from cron reminders, with regression coverage and redacted real cron/heartbeat proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a main-session cron systemEvent is queued with a `cron:` context key, heartbeat peeks it into a reminder wrapper, and reply setup drains the same event into `System:` lines. I did not run a live cron job because this review is read-only.

## Solution Assessment

Yes, the linked fix direction is the narrow maintainable path: mark only main-session injected cron events and skip only those from the wrapper while preserving ordinary cron reminder handling.
