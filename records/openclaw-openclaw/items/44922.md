## Summary

Keep this open: current main still enqueues main-session cron systemEvents with a plain `cron:<jobId>` context key, and heartbeat prompt construction still wraps that same queued event as a scheduled reminder. The linked fix PR is the right follow-up path, but it has not merged.

## What This Changes

A main-session cron `systemEvent` with `delivery.mode: "none"` should avoid adding a second scheduled-reminder heartbeat wrapper for the same event text.

## Best Possible Solution

Review and land a narrow fix that marks main-session injected cron systemEvents and excludes only those already-injected events from the heartbeat reminder wrapper, with focused regression coverage.

## Reproduction Assessment

Yes, by source inspection: a main cron `systemEvent` is queued with `contextKey=cron:<jobId>`, then heartbeat treats the same queued text as a cron event and wraps it in the reminder prompt. I did not run a live cron schedule in this read-only review.

## Solution Assessment

Yes, the tag-and-skip direction from the linked PR is the narrowest maintainable fix because it preserves normal cron reminder prompts while avoiding duplication only for main-session events already injected into history.
