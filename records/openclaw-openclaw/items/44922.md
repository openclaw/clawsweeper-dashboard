## Summary

Keep open. Current main still has the duplicate construction path: main-session cron enqueues a cron-tagged system event, and heartbeat prompt construction still wraps that same cron-tagged event into the scheduled-reminder prompt; open PR #44973 remains the targeted fix candidate.

## What This Changes

Cron jobs using `sessionTarget: "main"` and `payload.kind: "systemEvent"` should not surface the same text both as a queued system event and as a scheduled-reminder heartbeat prompt when delivery is disabled.

## Best Possible Solution

Land PR #44973 or an equivalent narrow fix that marks main-session cron systemEvent injections and excludes only those already-injected events from cron reminder prompt construction while preserving isolated cron reminders and outbound delivery semantics.

## Reproduction Assessment

Yes. The issue provides a concrete cron job configuration, and current-main source inspection shows the same `cron:<job.id>` event can be queued into the main session, drained as a System line, and selected again for the heartbeat cron prompt.

## Solution Assessment

Yes. The tag-and-skip direction discussed on the issue is the narrowest maintainable fix because `delivery.mode: "none"` controls outbound relay, while the bug is duplicate prompt construction for already-injected main-session events.
