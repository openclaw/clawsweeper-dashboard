## Summary

Keep open: current main still has the cron-to-heartbeat duplication path, and the exact fix PR is still open rather than merged. The issue has enough source-level proof and related discussion to remain actionable.

## What This Changes

The issue requests that main-session cron jobs with a systemEvent payload and delivery.mode "none" avoid adding a second scheduled-reminder wrapper to the session context.

## Best Possible Solution

Rebase and review https://github.com/openclaw/openclaw/pull/44973, or land an equivalent narrow fix that distinguishes already-injected main-session cron systemEvents from cron reminders before building the heartbeat wrapper.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a main-session cron systemEvent is queued with a `cron:` context key, heartbeat peeks it into the cron wrapper, and reply setup drains the same event into `System:` lines.

## Solution Assessment

Yes. The narrow maintainable fix is to mark or otherwise identify main-session injected cron systemEvents and skip only those from the wrapper while preserving normal cron reminder handling.
