## Summary

Keep open: current main and the latest shipped release still contain the source-level duplicate cron prompt path, the focused fix PR was closed unmerged, and the broader remediation PR is still open with review blockers rather than landed proof.

## What This Changes

The issue reports that main-session cron `systemEvent` jobs with `delivery.mode: "none"` surface the same text twice in session/model context: once as a system event and once as a scheduled-reminder wrapper.

## Best Possible Solution

Land a focused remediation, or a corrected canonical PR, that skips only main-session cron system events already injected as System context while preserving ordinary cron reminders, no-relay behavior, and regression coverage.

## Reproduction Assessment

Yes, at source level. Current main queues a main-session `cron:` system event, heartbeat wraps cron-tagged pending events as scheduled reminders, and reply setup can drain the same queued text into `System:` context.

## Solution Assessment

Yes, the narrow maintainable solution is to distinguish main-session cron system events that are already injected into history and skip only those from cron reminder wrapping. The broader open PR needs maintainer review before it can be the canonical implementation path.
