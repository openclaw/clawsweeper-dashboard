## What This Changes

The issue reports that a main-session cron systemEvent is both injected as a system message and repeated as a scheduled-reminder heartbeat prompt despite `delivery.mode: "none"`.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/44973 or an equivalent marker-and-skip fix with regression coverage and redacted cron/heartbeat proof, while preserving normal cron wrapper and delivery semantics.

## Reproduction Assessment

Yes. Current main queues a main-session `cron:` system event, heartbeat peeks and wraps the same text as a scheduled reminder, and reply setup can drain that queue into `System:` lines.

## Solution Assessment

Yes. The narrow maintainable solution is to identify main-session cron system events already injected into history and skip only those from cron reminder wrapping, without changing broader delivery or heartbeat behavior.
