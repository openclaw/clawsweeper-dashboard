## Summary

Keep open: current main still has a source-level doctor migration bug where the primary-model rewrite can create a canonical `openai/*` entry with generated Codex runtime policy before the legacy `openai-codex/*` model-map entry is merged, letting the generated entry override an explicit `agentRuntime.id: "pi"`. The related prompt-cache issue is useful context for why users need the PI opt-out, but it does not supersede this narrower config-preservation bug.

## What This Changes

The issue requests that `openclaw doctor --fix` preserve intentional PI/OAuth runtime metadata when migrating legacy `openai-codex/*` model config to canonical `openai/*` refs.

## Best Possible Solution

Keep the documented migration to canonical `openai/*` refs, but preserve explicit legacy model-entry metadata, especially concrete `agentRuntime.id: "pi"` and aliases, when merging the old key into the canonical model-map entry.

## Reproduction Assessment

Yes, with high-confidence source reproduction: the current repair order can generate a canonical Codex runtime entry from the primary ref before the legacy PI model-map entry is merged, and the merge order lets the generated entry win. I did not run doctor or tests because this review is read-only.

## Solution Assessment

Yes for the narrow bug: preserve explicit model-scoped runtime and alias metadata while still migrating legacy refs to canonical `openai/*`. Making Codex opt-in by default is a broader product-policy question and should not be part of the repair.
