## Summary

Keep open: current source supports a real `doctor --fix` regression where a generated canonical Codex runtime policy can override an explicit legacy model-scoped `agentRuntime.id: "pi"` during `openai-codex/*` route repair. The documented migration to `openai/*` should remain, but explicit PI routing metadata must survive it.

## What This Changes

The issue asks doctor route repair to preserve intentional PI/OAuth routing when migrating `openai-codex/*` model refs to canonical `openai/*` refs.

## Best Possible Solution

Preserve explicit legacy model-entry metadata, especially `agentRuntime.id: "pi"`, while still migrating legacy `openai-codex/*` refs to canonical `openai/*` according to the current docs.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the current repair order can create `openai/gpt-5.4` with `agentRuntime.id: "codex"` before the legacy `openai-codex/gpt-5.4` model entry with `agentRuntime.id: "pi"` is merged, and the merge order lets the generated canonical entry win. I did not run doctor or tests because this review was read-only.

## Solution Assessment

Yes for the narrow bug: keep the documented canonical route migration but make the merge preserve explicit concrete runtime policy and aliases from the legacy entry. The broader request to make Codex runtime opt-in by default is a separate product-policy discussion, not required for this fix.
