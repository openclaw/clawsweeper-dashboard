## Summary

Keep open: current main still has a narrow `doctor --fix` migration bug where a generated canonical Codex runtime policy can override an explicit legacy model-scoped `agentRuntime.id: "pi"` after `openai-codex/*` refs are rewritten. The related prompt-cache report is useful context, but this issue has its own source-level config-preservation failure.

## What This Changes

The issue asks for `doctor --fix` to preserve intentional PI/OAuth runtime metadata when migrating legacy `openai-codex/*` model refs to canonical `openai/*` refs.

## Best Possible Solution

Preserve explicit concrete legacy model-entry metadata, especially `agentRuntime.id: "pi"` and aliases, while still migrating legacy `openai-codex/*` refs to canonical `openai/*` according to the documented contract.

## Reproduction Assessment

Yes, at source level: the current repair order can create a canonical `openai/*` Codex entry before merging the legacy `openai-codex/*` entry with `agentRuntime.id: "pi"`, and the merge order lets the generated canonical entry win. I did not run doctor or tests because this was a read-only review.

## Solution Assessment

Yes for the narrow bug: keep the documented canonical route migration but change the merge/preservation logic so explicit concrete legacy runtime policy and aliases survive. Making Codex runtime opt-in by default is a separate product-policy decision, not required for this fix.
