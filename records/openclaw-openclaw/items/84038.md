## Label Justifications

- `P1`: The report describes a doctor/update repair path that can break an existing agent runtime/auth workflow and switch token-cost behavior for real users.
- `impact:data-loss`: The affected doctor repair can drop or overwrite user-authored model-map metadata such as aliases and `agentRuntime` config.
- `impact:auth-provider`: The failure mode changes model namespace, selected runtime, OAuth/API-key preference, and Codex/OpenAI auth routing.

## Summary

Keep open: current main still has a source-level path where doctor repair can rewrite an intentional `openai-codex/*` primary to `openai/*`, create a `codex` runtime policy for that canonical ref, and then let that newly-created canonical entry override the old per-model `agentRuntime.id: "pi"` during the legacy model-map merge.

## What This Changes

The issue asks doctor repair to preserve intentional PI-runtime and OpenAI Codex OAuth routing when migrating legacy `openai-codex/*` model refs.

## Best Possible Solution

Fix the doctor repair ordering or merge precedence so explicit PI/auth intent on a legacy model-map entry survives primary-route migration, then add a regression test for the exact config shape from this issue.

## Reproduction Assessment

Yes, at source level. With the reported config shape, the primary rewrite creates a canonical `openai/*` Codex runtime policy before the legacy model-map entry is merged, and the later merge lets that canonical entry override the explicit PI policy.

## Solution Assessment

Yes for the narrow bug: preserving explicit runtime/auth intent during the existing doctor repair is the maintainable fix. The broader request to make Codex experimental or opt-in should be treated separately if maintainers want to revisit defaults.
