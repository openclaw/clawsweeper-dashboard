## Summary

Keep this PR open for normal maintainer review: it is the canonical open landing path for a real web_fetch HTML entity decoding bug, current main and v2026.6.10 still have the broken decoder, and I found no blocking patch defect after checking the runtime caller, shared decoder, sibling caller, tests, related PRs, and CI/proof context.

## What This Changes

The PR routes web_fetch entity decoding through the shared HTML decoder, tightens that decoder's named and numeric entity parsing, and adds focused regression coverage for astral, escaped-ampersand, uppercase, malformed, and BMP entities.

## Best Possible Solution

Land this PR or an equivalent shared-decoder fix after ordinary maintainer review, because it removes duplicated web_fetch decoder behavior and preserves the existing entity contracts with focused regression coverage.

## Reproduction Assessment

Yes. Current main and v2026.6.10 still show the fromCharCode numeric decoder and ampersand-first replacement chain, and the PR body provides before/after output through exported htmlToMarkdown; I did not run tests in this read-only review.

## Solution Assessment

Yes. Routing web_fetch through decodeHtmlEntityAt with a single left-to-right pass is the narrow maintainable fix; a local fromCodePoint helper or replacement-chain reorder would leave duplicate decoder behavior in place.
