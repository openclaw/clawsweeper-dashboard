## Summary

Keep this PR open: the re-review fixed the earlier shared-decoder concern, but the new shared-decoder path now drops current web_fetch support for uppercase or mixed-case named entities, which is a small but real compatibility regression in extracted page text.

## What This Changes

The branch changes web_fetch entity decoding to call `decodeHtmlEntityAt` in a single left-to-right pass and adds regression tests for astral numeric entities, escaped `&amp;#39;`, and lowercase/BMP entities.

## Best Possible Solution

Keep the shared single-pass decoder approach, but preserve current case-insensitive support for the existing named entity subset and cover it with regression tests before merge.

## Reproduction Assessment

Yes. Source inspection gives a focused repro: current main's `/gi` replacements decode `&AMP;` or `&LT;`, while PR head calls a lowercase-only `decodeHtmlEntityAt`, so `htmlToMarkdown('<p>A &LT; B</p>')` would stop matching current behavior.

## Solution Assessment

No, not quite. The shared single-pass decoder is the right shape after the earlier review, but the best fix must also preserve the existing named-entity compatibility before landing.
