## Summary

Keep open; the PR correctly moves web_fetch toward shared single-pass entity decoding and has usable proof, but it is not merge-ready because the shared helper is not equivalent to the decoder it replaces. As written it can stop decoding uppercase supported named entities and can consume malformed numeric entities via lenient `parseInt`, so fetched page text can still be rewritten incorrectly.

## What This Changes

This PR routes web_fetch HTML entity decoding through the shared `decodeHtmlEntityAt` helper and adds regression tests for astral numeric entities, escaped `&amp;`, and BMP entities.

## Best Possible Solution

Keep the shared single-pass path, but make the shared decoder or web_fetch wrapper preserve current valid entity behavior and strict malformed-entity preservation, with focused regression tests.

## Reproduction Assessment

Yes. Source inspection shows current main decodes `&amp;` before numeric entities and uses `String.fromCharCode` for numeric code points; the PR body also includes before/after live output for the exported `htmlToMarkdown` path.

## Solution Assessment

No. Reusing the shared helper is the right boundary, but as-is it changes web_fetch's existing entity contract; the best fix is to extend or tighten the shared helper or wrapper with regression coverage.
