## Summary

Current main and v2026.5.22 now include the OpenAI-compatible stream parser fix that keeps parallel tool-call argument buffers separate, which addresses the reported MiniMax/OpenAI-compatible file-tool argument corruption path well enough to close this cleanup item.

## What This Changes

The issue reports that Windows file-tool paths containing `system` are being sent as `ystem`, causing ENOENT failures.

## Best Possible Solution

Keep the shipped per-index OpenAI-compatible stream parser fix, and ask for a new current-version report only if v2026.5.22+ still shows raw MiniMax tool-call chunks mutating a path literal.

## Reproduction Assessment

Yes, enough for cleanup at source level: the report had concrete Windows ENOENT logs, and the merged parser fix reproduced the OpenAI-compatible class where parallel tool-call deltas corrupt `exec`/`read` arguments. I did not live-replay the exact MiniMax Windows workspace path.

## Solution Assessment

Yes. Fixing the OpenAI-compatible stream parser to key argument accumulation by `toolCall.index`/id is the maintainable boundary fix; path-normalization changes would not address streamed argument corruption.
