## Summary

Keep open: the PR is a plausible fix path for the linked delivery-ownership feature, but it is not merge-ready because a persisted/raw `completionOwner` can still misroute retries, the proof is synthetic-only, and the new routing contract needs maintainer signoff.

## What This Changes

The PR adds a native subagent `completionOwner` enum, persists it through subagent run and retry payloads, exposes it on `sessions_spawn`, rejects it for ACP, adjusts completion routing, and updates focused tests and prompt snapshots.

## Best Possible Solution

Land a narrowed PR that normalizes owner values at raw and persisted boundaries, preserves legacy behavior for unknown state, splits or justifies the queue debounce change, and includes live Gateway/channel proof plus maintainer signoff for the native-only owner contract.

## Reproduction Assessment

Yes for the PR-head defect: source inspection shows raw restored run records can feed an unnormalized `completionOwner` into retry routing. The linked feature request itself is not a current-main bug reproduction and still lacks a minimal upstream runtime case.

## Solution Assessment

No, not as currently proposed. The direction is plausible, but it needs durable enum normalization, live transport proof, and maintainer signoff on native-only owner semantics before merge.
