## Summary

Keep open: the latest PR head adds the native completion-owner contract, but persisted owner values can still bypass normalization before routing, and the supplied proof is synthetic rather than a real Gateway/channel run.

## What This Changes

The PR adds a native `completionOwner` field across `sessions_spawn`, subagent spawn/registry persistence, final-delivery routing, hook types, prompt snapshots, and focused regression tests.

## Best Possible Solution

Normalize and validate `completionOwner` at persistence and routing boundaries, preserve legacy child-binding-first behavior for missing or unknown values, then land the native-only contract with maintainer approval and live Gateway/channel proof.

## Reproduction Assessment

No high-confidence current-main bug reproduction exists for the linked feature request. For the PR defect, the source path is high confidence: persisted JSON is restored without enum validation and the router branches on `undefined` versus non-`work-thread-final`.

## Solution Assessment

No, not as-is. The implementation direction is reasonable, but persisted/routing owner values need normalization before the child/work-thread fallback is skipped, and the PR still needs real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Normalize completionOwner before routing:** `src/agents/subagent-announce-delivery.ts:400`
  - body: The new route treats any restored owner value other than `work-thread-final` or `undefined` as requester-owned. Persisted run records are restored from JSON by spreading stored fields into `SubagentRunRecord`, so a stale or unrecognized `completionOwner` string can skip the child/work-thread fallback and misroute the complete final. Normalize the value at restore or before this branch.
