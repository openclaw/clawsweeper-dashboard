## Summary

Keep open: this is a recent substantive feature PR paired with an open feature issue, but it still needs maintainer semantics review, real behavior proof, and a fix for a legacy-value routing edge.

## What This Changes

The PR adds native subagent `completionOwner` values across `sessions_spawn`, subagent run registration, final-delivery routing, retry persistence, hook event types, tests, and prompt snapshots.

## Best Possible Solution

Normalize completion-owner values at persisted/routing boundaries, preserve child-binding-first behavior for missing or unknown values, then land the native-only contract after maintainer approval and live Gateway/channel proof.

## Reproduction Assessment

Not applicable as a current-main bug reproduction because the linked work is a feature/design request. For the PR finding, source inspection gives a high-confidence path: persisted records are restored without owner normalization and the new resolver branches on the raw owner value.

## Solution Assessment

No, not as-is. The implementation direction is scoped, but the routing boundary should normalize owner values before changing delivery ownership, and the PR still needs live behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Normalize completionOwner before routing:** `src/agents/subagent-announce-delivery.ts:368`
  - body: The new router uses `params.completionOwner` directly to decide whether to skip the child binding. Subagent records are restored from JSON by spreading stored fields into `SubagentRunRecord`, so a stale or unrecognized persisted `completionOwner` is truthy and not `work-thread-final`; that sends the full final through the requester path instead of preserving the child-binding-first legacy route promised by this PR. Normalize the value here, or at restore, and only skip the child binding for the supported requester-owned values.
  - confidence: 0.87
