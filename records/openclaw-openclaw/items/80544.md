## Summary

Keep open: the native completion-owner direction matches the linked feature request, but the latest head still has a runtime normalization bug for persisted owner values and the supplied proof is synthetic-only, so this should not merge yet.

## What This Changes

This PR adds a native `completionOwner` field across `sessions_spawn`, subagent run persistence, final-delivery routing, hook payloads, prompt snapshots, and regression tests.

## Best Possible Solution

Normalize owner values at persisted and routing boundaries, preserve the legacy requester-then-child fallback for missing or unknown values, then land the native-only contract after maintainer architecture review and live Gateway/channel proof.

## Reproduction Assessment

Not applicable as a current-main bug reproduction because this is a feature PR. The PR finding is source-reproducible: persisted JSON can restore raw owner values, and the new router branches on the raw value before deciding whether to use the child fallback.

## Solution Assessment

No, not as-is. The implementation direction is plausible, but owner values should be normalized before routing and replay, and the PR still needs live behavior proof for the channel/Gateway path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Normalize completionOwner before routing:** `src/agents/subagent-announce-delivery.ts:400`
  - body: `completionOwner` can come back from persisted JSON as a raw string because registry restore spreads stored fields directly into the run record. When that value is stale or unrecognized, this check treats it as explicitly present and skips the child-bound fallback, so a completion can route to the requester/origin path instead of preserving missing-owner behavior. Normalize at restore or at this routing boundary before deciding to suppress the child fallback.
