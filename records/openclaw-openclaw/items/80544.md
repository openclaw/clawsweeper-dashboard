## Summary

Keep this PR open: the native completion-owner direction is useful for the linked longwork ownership request, but the latest head still has a source-reproducible routing bug for unnormalized persisted owner values and the supplied proof remains synthetic rather than a real Gateway/channel run.

## What This Changes

The PR adds a native `completionOwner` contract across `sessions_spawn`, subagent spawn/registry persistence, final-delivery routing, hook event types, prompt snapshots, and regression tests.

## Best Possible Solution

Normalize owner values at persisted and routing boundaries, keep child-binding-first behavior for missing or unknown values, then land the native-only contract after maintainer architecture approval and live Gateway/channel proof.

## Reproduction Assessment

Not applicable as a current-main bug reproduction because the linked work is a feature/design request. For the PR finding, source inspection gives a high-confidence path: persisted records can restore raw owner values, and the router branches on the raw value.

## Solution Assessment

No, not as-is. The implementation direction is reasonable, but persisted/routing owner values should be normalized before changing delivery ownership, and the PR still needs real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Normalize completionOwner before routing:** `src/agents/subagent-announce-delivery.ts:368`
  - body: The new router uses `params.completionOwner` directly to decide whether to skip the child binding. Persisted run records are restored from JSON by spreading stored fields into `SubagentRunRecord`, so a stale or unrecognized stored owner is truthy and not `work-thread-final`; that routes the complete final through the requester path instead of preserving the promised child-binding-first legacy route. Normalize at restore or at this routing boundary before skipping the child route.
