## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Import Type from the declared package:** `src/agents/tools/task-tool.ts:1`
  - body: Both new tool files import `@sinclair/typebox`, but `package.json` only declares `typebox` and existing tools import from `typebox`. Under the repo's package layout this can fail module resolution before the task/team tools load, so switch the new files to the declared package import.
  - confidence: 0.96
- **[P2] Preserve missing-read errors before wrapping denials:** `src/agents/sandbox/fs-bridge-path-safety.ts:74-84`
  - body: `openReadableFile` now wraps every failed boundary open as `permission_denied`, including ordinary missing-file opens. Current sandbox read tests preserve ENOENT/no-such-file semantics, so callers that distinguish not-found from permission failures lose the expected diagnostic.
  - confidence: 0.91
- **[P2] Register task and team in the core catalog:** `src/agents/openclaw-tools.ts:238-242`
  - body: The PR constructs `task` and `team` as core tools but does not add them to `CORE_TOOL_DEFINITIONS`. Profile allowlists, `group:openclaw`, tool sections, and known-core-tool checks are derived from that catalog, so these tools will not behave consistently as first-class core tools.
  - confidence: 0.89
- **[P2] Use flat enum schemas for task:** `src/agents/tools/task-tool.ts:31-39`
  - body: `TaskToolSchema` uses `Type.Union(...Type.Literal...)` for `action`, `runtime`, and `notifyPolicy`, which emits `anyOf`. Existing tool schemas use `stringEnum` and `optionalStringEnum` because some providers reject `anyOf`, so this can make the new task tool unusable with those providers.
  - confidence: 0.94
- **[P2] Use a flat enum schema for team:** `src/agents/tools/team-tool.ts:22`
  - body: `TeamToolSchema.action` uses `Type.Union(...Type.Literal...)`, producing the same `anyOf` shape that the repo's schema helper avoids for provider compatibility. Use the shared flat enum helper here as well.
  - confidence: 0.93
- **[P2] Map allowed-mount escapes to workspace_boundary:** `src/agents/fs-permission-denied.ts:22-25`
  - body: The classifier does not recognize the current bridge message `Sandbox path escapes allowed mounts`, while this PR wraps that exact message from local and remote sandbox path guards. Those denials become `reason=unknown` instead of `workspace_boundary`.
  - confidence: 0.94
- **[P2] Reject unknown task actions before output lookup:** `src/agents/tools/task-tool.ts:232-241`
  - body: After the explicit task actions, the handler falls through to the output path. Direct invoke paths call `execute` without relying on TypeBox validation, so an unknown action becomes a misleading `task not found` response instead of a clear unsupported-action error.
  - confidence: 0.88
- **[P2] Translate missing team updates to ToolInputError:** `src/agents/tools/team-tool.ts:87-93`
  - body: `updateTeam()` throws a plain `Error` when the team id is unknown, and this branch does not translate it. Direct tool invocation maps input errors differently from execution failures, so a user-correctable missing team update can surface as a 500-style tool failure.
  - confidence: 0.87
- **[P2] Consolidate the sandbox readiness note:** `src/commands/doctor-sandbox.ts:240-253`
  - body: `runSandboxHealth` calls the new readiness note before `maybeRepairSandboxImages`, but the non-Docker no-local-support branch also emits capability-probe details. That makes doctor print overlapping sandbox readiness output for the same condition.
  - confidence: 0.84
