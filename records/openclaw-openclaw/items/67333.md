## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Import Type from the declared package:** `src/agents/tools/task-tool.ts:1`
  - body: Both new tool files import `@sinclair/typebox`, but the repo declares and uses `typebox`. Under the current package layout, these imports can fail before the new tools load.
  - confidence: 0.96
- **[P2] Preserve missing-read errors before wrapping denials:** `src/agents/sandbox/fs-bridge-path-safety.ts:80-84`
  - body: `openReadableFile` now wraps every failed boundary open as `permission_denied`, including ordinary missing-file opens. Callers that distinguish not-found from permission failures lose the expected ENOENT-style diagnostic.
  - confidence: 0.91
- **[P2] Register task and team in the core catalog:** `src/agents/openclaw-tools.ts:238-242`
  - body: The PR constructs `task` and `team` as core tools but does not add them to `CORE_TOOL_DEFINITIONS`. Profile allowlists, `group:openclaw`, tool sections, and known-core-tool checks are catalog-derived, so these tools will not behave as first-class core tools.
  - confidence: 0.89
- **[P2] Use flat enum schemas for task:** `src/agents/tools/task-tool.ts:31-39`
  - body: `TaskToolSchema` uses `Type.Union(...Type.Literal...)` for `action`, `runtime`, and `notifyPolicy`, which emits `anyOf`. Existing tool schemas use flat enum helpers because some providers reject `anyOf`.
  - confidence: 0.94
- **[P2] Use a flat enum schema for team:** `src/agents/tools/team-tool.ts:22`
  - body: `TeamToolSchema.action` uses `Type.Union(...Type.Literal...)`, producing the same provider-incompatible `anyOf` shape. Use the shared flat enum helper here as well.
  - confidence: 0.93
- **[P2] Map allowed-mount escapes to workspace_boundary:** `src/agents/fs-permission-denied.ts:22-25`
  - body: The classifier does not recognize the bridge message `Sandbox path escapes allowed mounts`, while this PR wraps that exact message from sandbox path guards. Those denials become `reason=unknown` instead of `workspace_boundary`.
  - confidence: 0.94
- **[P2] Reject unknown task actions before output lookup:** `src/agents/tools/task-tool.ts:232-241`
  - body: After the explicit task actions, the handler falls through to the output path. Direct invoke paths can call `execute` without relying on TypeBox validation, so an unknown action becomes a misleading task lookup error.
  - confidence: 0.88
- **[P2] Translate missing team updates to ToolInputError:** `src/agents/tools/team-tool.ts:87-93`
  - body: `updateTeam()` throws a plain `Error` when the team id is unknown, and this branch does not translate it. A user-correctable missing team update can surface as a tool execution failure instead of an input error.
  - confidence: 0.87
- **[P2] Consolidate the sandbox readiness note:** `src/commands/doctor-sandbox.ts:241-253`
  - body: `runSandboxHealth` emits the new readiness note before `maybeRepairSandboxImages`, but the non-Docker no-local-support branch also emits capability-probe details, producing overlapping doctor output for the same condition.
  - confidence: 0.84
