- **[P1] Use flat enums for the task tool schema:** `src/agents/tools/task-tool.ts:31-39`
  - body: `Type.Union([...Type.Literal])` produces an `anyOf` schema for `action`, `runtime`, and `notifyPolicy`. Current provider-facing tools use `stringEnum`/`optionalStringEnum` because some validators reject `anyOf`, so registering this new core tool can break tool setup.
  - confidence: 0.95
- **[P1] Use a provider-safe enum for the team action:** `src/agents/tools/team-tool.ts:22`
  - body: This new core tool exposes `action` as a TypeBox union, which compiles to `anyOf`. Use the shared flat enum helper instead so the tool schema follows the same provider contract as the existing action-dispatched tools.
  - confidence: 0.93
- **[P1] Preserve missing-file errors for sandbox reads:** `src/agents/sandbox/fs-bridge-path-safety.ts:80-84`
  - body: Wrapping every failed `openBoundaryWithinRequiredMount` result as `permission_denied` changes missing-file reads from ENOENT-style behavior into permission failures. Current memory-flush append logic treats missing sandbox files as an empty first write, so this can make first-time sandbox writes fail instead of creating the file.
  - confidence: 0.9
- **[P2] Avoid duplicating the sandbox capability note:** `src/commands/doctor-sandbox.ts:241-253`
  - body: For non-docker backends without local sandbox support, `runSandboxHealth` first emits `noteSandboxCapabilityReadiness`, then `maybeRepairSandboxImages` emits another capability-probe note. Doctor output should consolidate this into one warning path.
  - confidence: 0.82
