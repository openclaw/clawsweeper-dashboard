## Summary

Keep open: this PR targets a real QMD/mcporter regression, but the current patch strips the same QMD cache env that local `qmd mcp` servers need when mcporter launches them, and the supplied proof is test/code-only rather than a live mcporter run.

## What This Changes

This PR adds a separate `mcporterEnv` for `QmdMemoryManager` that removes `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` from mcporter spawns and updates the adjacent test expectation.

## Best Possible Solution

Land a per-target env fix that prevents mcporter's own config discovery from seeing QMD-scoped XDG while preserving or injecting the per-agent QMD env for any `qmd mcp` server mcporter launches, then back it with live mcporter 0.10+ proof.

## Reproduction Assessment

Yes. The linked issue has concrete logs, current source still passes QMD-scoped XDG to mcporter, and mcporter v0.10.2 source confirms `XDG_CONFIG_HOME` controls config discovery; I did not run the live daemon scenario in this read-only review.

## Solution Assessment

No. The proposed direction is close, but deleting `XDG_CACHE_HOME` from the mcporter process also removes the per-agent cache env from local stdio `qmd mcp` servers that mcporter launches.
