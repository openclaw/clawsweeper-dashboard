## Summary

Keep open: the PR targets a real qmd/mcporter regression, but the patch strips the same XDG cache env that a local `qmd mcp` stdio server inherits from mcporter, so it can move searches off the agent-specific index.

## What This Changes

The PR adds a separate mcporter spawn environment without XDG config/cache overrides and updates the adjacent qmd-manager test expectation.

## Best Possible Solution

Land a per-target env fix that prevents mcporter config discovery from seeing QMD-scoped XDG while preserving or injecting the per-agent QMD env for any `qmd mcp` server mcporter launches.

## Reproduction Assessment

Yes, at source level: current main sends QMD-scoped XDG vars to mcporter, mcporter 0.10.2 uses `XDG_CONFIG_HOME` for config discovery, and the linked issue provides concrete failure logs. I did not run the live daemon scenario in this read-only review.

## Solution Assessment

No. The proposed env split fixes mcporter config discovery but is too broad for supported local `qmd mcp` server setups because mcporter passes its process env to stdio servers.

Full review comments:

- **[P1] Preserve the qmd server env when stripping mcporter discovery env:** `extensions/memory-core/src/memory/qmd-manager.ts:458-459`
  - body: `mcporter` v0.10.2 launches stdio servers with its own `process.env` plus server overrides, and OpenClaw's supported mcporter mode is a configured server running `qmd mcp`. Removing `XDG_CACHE_HOME` here means that local qmd server no longer sees the per-agent cache where direct `qmd update` writes `qmd/index.sqlite`, so searches can hit the default cache instead of the agent index. Please separate mcporter config discovery from the qmd server env or inject the per-agent QMD env into the server config.
