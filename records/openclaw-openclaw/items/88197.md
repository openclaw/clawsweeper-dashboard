## Summary

Keep open: current main and the latest release still delete one-shot `deleteAfterRun` cron jobs only after `ok` runs, while terminal failures are deliberately disabled and retained for inspection. The direct fix PR was closed unmerged, so the remaining decision is cron retention policy rather than conservative cleanup.

## What This Changes

The issue requests removal of failed or timed-out `deleteAfterRun` cron jobs so failed one-shot isolated cron jobs do not accumulate as disabled orphans.

## Best Possible Solution

Make the `deleteAfterRun` terminal-failure contract explicit first; if maintainers approve failure deletion, align `applyJobResult`, docs, and regression tests, otherwise keep failed-job retention and add bounded cleanup or audit tooling for failed one-shot orphans.

## Reproduction Assessment

Yes. Source inspection shows a `deleteAfterRun: true` one-shot job whose run status is `error` cannot satisfy the `result.status === "ok"` deletion gate, so it is retried or disabled instead of removed.

## Solution Assessment

Unclear. Deleting after terminal failure is a small code change, but it conflicts with current docs, tests, and the source comment preserving failed-job state for inspection, so the best fix needs maintainer policy first.
