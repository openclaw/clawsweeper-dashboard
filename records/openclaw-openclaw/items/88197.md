## Summary

Keep open: current main and the latest release still implement success-only `deleteAfterRun` deletion, and the only PR that directly tried terminal-failure deletion is closed unmerged after identifying the needed policy decision.

## What This Changes

The issue asks for `deleteAfterRun: true` cron jobs to be deleted after failed or timed-out runs so disabled failed one-shot jobs do not accumulate.

## Best Possible Solution

Make the terminal-failure contract explicit first; if maintainers approve failure deletion, align runtime, docs, and regression tests, otherwise keep failed-job retention and add bounded cleanup or audit tooling.

## Reproduction Assessment

Yes. Source inspection shows a `deleteAfterRun: true` `at` job whose run status is `error` cannot satisfy the `result.status === "ok"` deletion gate, so it is retried or disabled instead of removed.

## Solution Assessment

Unclear. Deleting after terminal failure would stop orphan buildup, but current docs and source intentionally preserve failed job state for inspection, so the best fix depends on maintainer policy.
