## Summary

Keep open: current main and v2026.6.9 still implement success-only `deleteAfterRun` job deletion, while terminal failures are retried or disabled and retained for inspection. The direct fix PR was closed unmerged after the contributor called out the unresolved retention-policy decision, so this is not conservative cleanup yet.

## What This Changes

The issue asks OpenClaw to delete or otherwise clean up one-shot `deleteAfterRun` cron jobs after failed isolated runs so disabled error-state orphans do not accumulate.

## Best Possible Solution

Make the terminal-failure contract explicit first; if maintainers approve failure deletion, align `applyJobResult`, CLI/docs, and regression tests, otherwise keep failed-job retention and add bounded cleanup or audit tooling for orphaned failed one-shots.

## Reproduction Assessment

Yes, at source level. A `deleteAfterRun: true` one-shot job whose run returns `error` cannot satisfy the `result.status === "ok"` deletion gate, so it is retried or disabled and retained instead of removed.

## Solution Assessment

Unclear. Deleting after terminal failure is a small runtime change, but it changes the documented success-only deletion and failed-state inspection contract, so the best fix needs a maintainer policy call first.
