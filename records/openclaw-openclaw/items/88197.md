## Label Justifications

- `P2`: The report describes accumulating disabled cron jobs after failures, but the current behavior is bounded to cron cleanup semantics and needs product judgment rather than emergency handling.
- `impact:session-state`: The affected state is persisted cron/job session state that can drift or accumulate after isolated runs fail.

## Summary

Current main supports the report’s source hypothesis, but it also documents and implements success-only deletion for one-shot cron jobs with failed jobs intentionally retained for inspection. This should stay open for a maintainer policy decision rather than be closed or queued as an automatic bug fix.

## What This Changes

The issue asks for `deleteAfterRun: true` cron jobs to be removed after failed or timed-out runs so disabled isolated cron jobs do not accumulate.

## Best Possible Solution

Maintainers should choose the one-shot failure policy first: keep success-only deletion and add bounded cleanup tooling for disabled failed jobs, or intentionally redefine `deleteAfterRun` with matching docs and regression tests.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a `deleteAfterRun: true` `at` job whose run status is `error` cannot satisfy the `result.status === "ok"` deletion gate and is disabled or retried instead of removed. I did not run a live gateway repro because this review is read-only.

## Solution Assessment

Unclear as a fix direction: deleting after every failed run would solve orphan accumulation, but it conflicts with current docs and the code's stated goal of preserving error state. A safer maintainer decision may be cleanup tooling or an explicit policy change with docs/tests.
