## Summary

Keep open: current main and v2026.6.8 still delete one-shot `deleteAfterRun` jobs only after `ok` runs, while terminal failures are deliberately disabled and retained for inspection. The direct fix attempt is closed unmerged, so the remaining work is a maintainer policy decision rather than conservative cleanup.

## What This Changes

Requests that one-shot cron jobs with `deleteAfterRun` be removed after failed or timed-out runs so disabled listener-spawn isolated jobs do not accumulate.

## Best Possible Solution

Make the terminal-failure contract explicit first; if maintainers approve failure deletion, align `applyJobResult`, docs, and regression tests, otherwise keep failed-job retention and add a bounded cleanup or audit path.

## Reproduction Assessment

Yes. Source inspection shows a `deleteAfterRun: true` one-shot job whose run status is `error` cannot satisfy the `result.status === "ok"` deletion gate, so it is retried or disabled instead of removed.

## Solution Assessment

Unclear. Deleting after terminal failure is a small code change, but it conflicts with current docs/tests and the source comment preserving failed-job state for inspection, so the best fix needs maintainer policy first.
