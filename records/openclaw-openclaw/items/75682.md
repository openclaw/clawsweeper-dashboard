## Summary

Keep open: the PR's stated safety goal is plausible but undocumented, and the current patch regresses existing session status model/provider identity handling by bypassing the shared resolver that preserves stored providers and configured-provider inference.

## What This Changes

The PR changes `session_status` model override and status-card label selection to avoid using `providerForCard` when the derived provider is missing.

## Best Possible Solution

Keep `resolveSessionModelIdentityRef` as the status-card identity source and add only the narrow undefined-provider guard needed for the actual crash path, with focused regression coverage and a changelog entry if user-visible status output changes.

## Reproduction Assessment

Yes for the PR regression: source inspection against the diff and existing status tests show the branch would drop provider identity for stored `modelProvider` and inferred custom-provider runtime models. The original undefined-provider crash is not reproducible from the blank PR body alone.

## Solution Assessment

No. Replacing the shared resolver with local field checks is not the best fix because it bypasses existing provider inference and slash-model parsing; a smaller guard around the existing resolver-derived values is safer.
