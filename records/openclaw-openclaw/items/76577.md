## Summary

Keep open for maintainer review: the reported install failure is plausible, but the proposed fix appears to use pnpm's build-script allowlist for an error thrown by pnpm's exotic-subdependency resolver path.

## What This Changes

This PR adds `@whiskeysockets/libsignal-node` to root and workspace `onlyBuiltDependencies` and adds a WhatsApp/pnpm changelog entry.

## Best Possible Solution

Confirm the current 2026.5.3 package/install path first, then fix the actual exotic-subdependency policy or the official WhatsApp package install flow with targeted regression coverage instead of relying on an ineffective build-script allowlist.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source-level, yes: with `blockExoticSubdeps` enabled, pnpm's resolver throws on exotic subdeps independently of `onlyBuiltDependencies`, matching the reported error path; I did not run a fresh install in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No: `onlyBuiltDependencies` gates lifecycle/build scripts, while the reported `ERR_PNPM_EXOTIC_SUBDEP` is thrown earlier by the resolver, so the fix should target the current package/install policy path after a package smoke.
