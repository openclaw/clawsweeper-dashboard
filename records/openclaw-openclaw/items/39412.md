## Summary

Keep open: current main already supersedes the daemon-token persistence/audit part with a newer config-backed flow, but the provider `*_FILE` env-var support is still unique and this PR introduces credential-handling regressions that block merge.

## What This Changes

The PR adds file-backed provider auth env-var lookup and rewrites gateway daemon install/audit behavior to prefer config-backed gateway tokens, with related tests.

## Best Possible Solution

Rewrite this as a focused provider-secret loading change aligned with SecretRef policy, preserving direct-env precedence, list separators, `_FILE` prefix-scan exclusion, diagnostics for unreadable configured files, and current main's daemon-token flow.

## Reproduction Assessment

Yes. Source inspection shows current main still reads provider auth from direct env vars only, while the PR-head rotation, key-list parsing, and unreadable-file behaviors are visible in the diff.

## Solution Assessment

No. The file-backed env direction may be useful, but this PR mixes it with stale daemon-token internals and introduces concrete credential-handling regressions; a narrower provider-secret change is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor explicit install tokens during rotation:** `src/cli/daemon-cli/install.ts:47-48`
  - body: When `openclaw gateway install --token <new-token>` runs with an existing `gateway.auth.token`, this early return keeps the old config token while reporting persistence success. The later install path can then omit the service env token, so the operator's explicit rotation is ignored.
  - confidence: 0.88
- **[P2] Preserve separators for file-backed key lists:** `src/agents/live-auth-keys.ts:111-114`
  - body: `<PROVIDER>_API_KEYS` now flows through `resolveSecretEnvValue()`, which normalizes file contents before `parseKeyList()` can split them. A file containing `key1\nkey2` becomes one concatenated invalid key, breaking multi-key rotation.
  - confidence: 0.84
- **[P3] Report unreadable configured secret files:** `src/utils/secret-env.ts:25-32`
  - body: If a `_FILE` env var points at a typo or unreadable path, all read errors are swallowed and the resolver returns `null`, which is indistinguishable from no configuration and leaves operators with a later auth failure or fallback credential.
  - confidence: 0.74

## Security Review
