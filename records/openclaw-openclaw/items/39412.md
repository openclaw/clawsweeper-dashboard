## Summary

Keep this PR open, but not merge-ready: the provider `*_FILE` idea remains a meaningful unsuperseded feature, while the daemon-token half is stale against current main and the patch still has concrete credential-handling defects plus missing real behavior proof.

## What This Changes

The PR adds file-backed provider auth env-var resolution and rewrites gateway daemon install/audit behavior to prefer config-backed gateway tokens, with related tests.

## Best Possible Solution

Rewrite this as a focused provider-secret loading change aligned with the SecretRef policy, preserving list separators and diagnostics, while leaving current main's daemon-token flow intact.

## Reproduction Assessment

Yes for review purposes. Source inspection shows current main still lacks provider `*_FILE` lookup, and the PR diff makes the token-rotation, key-list parsing, and silent unreadable-file behaviors directly traceable.

## Solution Assessment

No. The provider file-backed env direction may be useful, but this PR mixes it with stale daemon internals and introduces credential-handling regressions; a narrower provider-secret patch is the safer path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor explicit install tokens during rotation:** `src/cli/daemon-cli/install.ts:47-48`
  - body: When `openclaw gateway install --token <new-token>` runs with an existing `gateway.auth.token`, this early return keeps the old config token while reporting persistence success. The later install path can then omit the service env token, so the operator's explicit rotation is ignored.
  - confidence: 0.88
- **[P2] Preserve separators for file-backed key lists:** `src/agents/live-auth-keys.ts:111-114`
  - body: `<PROVIDER>_API_KEYS` now flows through `resolveSecretEnvValue()`, which normalizes file contents before `parseKeyList()` can split them. A file containing `key1\nkey2` becomes one concatenated invalid key, breaking multi-key rotation.
  - confidence: 0.84
- **[P3] Report unreadable configured secret files:** `src/utils/secret-env.ts:25-32`
  - body: If a `_FILE` env var points at a typo or unreadable path, all read errors are swallowed and the resolver returns `null`, which is indistinguishable from no configuration and leaves operators with a later auth failure or fallback credential.
  - confidence: 0.74
