## Summary

Keep this PR open, but it should not merge as-is: the contributor proof gate is still missing, and source review shows blocking regressions in explicit gateway token rotation and file-backed key-list parsing.

## What This Changes

The PR adds file-backed provider auth env-var resolution, changes daemon gateway install/audit handling toward config-backed tokens, and adds auth/daemon tests.

## Best Possible Solution

Rework the PR around the current SecretRef/auth resolver and resolveGatewayInstallToken contracts, preserve key-list parsing and explicit token rotation, and add redacted real behavior proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR diff and current normalization/token-resolution code shows the token-rotation early return, newline separator loss, and silent file-read failure paths. I did not run tests because this was a read-only review.

## Solution Assessment

No. The daemon part should stay on the current shared resolveGatewayInstallToken path, and provider file-backed secrets should be reconciled with the existing SecretRef file-provider policy instead of adding a silent scalar env-file resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor explicit install tokens during rotation:** `src/cli/daemon-cli/install.ts:47-48`
  - body: When `openclaw gateway install --token <new-token>` runs with an existing `gateway.auth.token`, this early return keeps the old config token and reports persistence success. The later install plan can omit the service env token, so the user-supplied rotation is ignored.
  - confidence: 0.88
- **[P2] Preserve separators for file-backed key lists:** `src/agents/live-auth-keys.ts:111-114`
  - body: `<PROVIDER>_API_KEYS` now goes through scalar secret normalization before `parseKeyList()`. A file containing newline-separated keys is normalized into one concatenated invalid key, breaking multi-key rotation.
  - confidence: 0.84
- **[P3] Report unreadable configured secret files:** `src/utils/secret-env.ts:31-32`
  - body: If a `_FILE` env var points at a typo or unreadable path, the resolver swallows the read error and returns `null`, leaving operators with a later auth failure or fallback credential and no clue that the configured secret file failed to load.
  - confidence: 0.74
