## Summary

Keep open: this is a plausible credential-hardening PR, but the current branch still has blocking token-rotation, key-list parsing, and secret-file diagnostic issues, lacks required real behavior proof, and overlaps current main's newer SecretRef and gateway install-token contracts.

## What This Changes

Adds `_FILE` provider API-key env loading, changes daemon gateway install/audit logic to avoid embedding gateway tokens in service env, and adds tests for those paths.

## Best Possible Solution

Rebase onto current main, drop the superseded local daemon persistence helper, and either rely on documented SecretRef file providers or propose a SecretRef-compatible `*_FILE` design with diagnostics and list parsing coverage.

## Reproduction Assessment

Yes for the PR review findings: source inspection at PR head shows the explicit-token early return, scalar normalization before list parsing, and silent `*_FILE` read-failure path. I did not run live daemon/provider commands because this review is read-only and contributor real behavior proof is still missing.

## Solution Assessment

No. The daemon portion should be reconciled with current main's shared `resolveGatewayInstallToken` path, and provider file-backed secrets need a deliberate SecretRef-compatible design instead of a separate silent file-read alias.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor explicit install tokens during rotation:** `src/cli/daemon-cli/install.ts:47-48`
  - body: When `openclaw gateway install --token <new-token>` runs with an existing `gateway.auth.token`, this early return keeps the old config token and reports persistence success. The later install plan can omit the service env token, so the user-supplied rotation is ignored.
  - confidence: 0.88
- **[P2] Preserve separators for file-backed key lists:** `src/agents/live-auth-keys.ts:111-114`
  - body: `<PROVIDER>_API_KEYS` now goes through scalar secret normalization before `parseKeyList()`. A file containing newline-separated keys is normalized into one concatenated invalid key, breaking multi-key rotation.
  - confidence: 0.84
- **[P3] Report unreadable configured secret files:** `src/utils/secret-env.ts:31-32`
  - body: If a `_FILE` env var points at a typo or unreadable path, the resolver swallows the read error and returns `null`, leaving operators with a later auth failure or fallback credential and no clue that the configured secret file failed to load.
  - confidence: 0.74

## Security Review

Status: needs_attention
