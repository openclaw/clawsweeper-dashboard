## Summary

Keep open: current main still lacks provider `*_FILE` auth support, while the daemon-token portion is stale against newer main behavior and the PR has blocking credential handling defects.

## What This Changes

The PR adds provider auth `*_FILE` env-var support and changes daemon gateway token install/audit behavior to prefer config-backed auth.

## Best Possible Solution

Rewrite this as a focused provider-secret loading change that preserves direct-env precedence, keeps list separators intact, excludes `_FILE` names from prefix scans, reports unreadable configured files, and leaves current daemon token handling in place.

## Reproduction Assessment

Yes. Source inspection shows current main still only reads direct provider env vars, and the PR-head credential rotation, key-list parsing, and unreadable-file behaviors are visible in the diff.

## Solution Assessment

No. The file-backed env direction may be useful, but this PR is not the best current solution because it mixes a stale daemon-token rewrite with provider auth changes and introduces concrete credential handling regressions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor explicit install tokens during rotation:** `src/cli/daemon-cli/install.ts:47-48`
  - body: When `openclaw gateway install --token <new-token>` runs with an existing `gateway.auth.token`, this early return keeps the old config token while reporting persistence success. The later install path can then omit the service env token, so the operator's explicit rotation is ignored.
  - confidence: 0.88
- **[P2] Preserve separators for file-backed key lists:** `src/agents/live-auth-keys.ts:111-114`
  - body: `<PROVIDER>_API_KEYS` now flows through `resolveSecretEnvValue()`, which normalizes file contents before `parseKeyList()` can split them. A file containing `key1\nkey2` becomes one concatenated invalid key, breaking multi-key rotation.
  - confidence: 0.84
- **[P3] Report unreadable configured secret files:** `src/utils/secret-env.ts:25-32`
  - body: If a `_FILE` env var points at a typo or unreadable path, all read errors are swallowed and the resolver returns `null`, which is indistinguishable from no configuration. Operators then get a later auth failure or fallback credential without a useful diagnostic.
  - confidence: 0.74

## Security Review

Status: needs_attention
