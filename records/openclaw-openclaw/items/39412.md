## Summary

Keep this PR open, but it should not merge as-is: the daemon-token half is now largely superseded by current main, while the remaining provider `_FILE` alias work has real security/design concerns and still lacks after-fix real behavior proof.

## What This Changes

The branch adds `_FILE` provider auth env support through a new secret-env resolver and changes daemon gateway installs/audits toward config-backed tokens.

## Best Possible Solution

Rebase onto current main, drop the superseded daemon token persistence path, and either rely on documented SecretRef file providers or propose a SecretRef-compatible file-backed env design with diagnostics and list parsing coverage.

## Reproduction Assessment

Yes for the PR review findings: source inspection at PR head shows the explicit-token early return, scalar normalization before list parsing, and silent *_FILE read-failure path. I did not run live daemon/provider commands because this review is read-only and contributor real behavior proof is still missing.

## Solution Assessment

No. The daemon portion should be reconciled with current main's shared resolveGatewayInstallToken path, and provider file-backed secrets need a deliberate SecretRef-compatible design instead of a separate silent file-read alias.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor explicit install tokens during rotation:** `src/cli/daemon-cli/install.ts:47-48`
  - body: When `openclaw gateway install --token <new-token>` runs with an existing `gateway.auth.token`, this early return keeps the old config token and reports persistence success. The later install plan can omit the service env token, so the user-supplied rotation is ignored.
  - confidence: 0.88
- **[P2] Preserve separators for file-backed key lists:** `src/agents/live-auth-keys.ts:111-114`
  - body: `<PROVIDER>_API_KEYS` now goes through scalar secret normalization before `parseKeyList()`. A file containing newline-separated keys is normalized into one concatenated invalid key, breaking multi-key rotation.
  - confidence: 0.84
- **[P3] Report unreadable configured secret files:** `src/utils/secret-env.ts:31-32`
  - body: If a `_FILE` env var points at a typo or unreadable path, the resolver swallows the read error and returns `null`, leaving operators with a later auth failure or fallback credential and no clue that the configured secret file failed to load.
  - confidence: 0.74

## Security Review

Status: needs_attention

Summary: The diff changes credential loading and gateway token persistence and has concrete token-rotation, SecretRef policy, and diagnostic risks.

Concerns:

- **[medium] Explicit gateway token rotation can be ignored:** `src/cli/daemon-cli/install.ts:47`
  - body: The daemon install path can keep an existing config token while treating persistence as successful, leaving the old gateway credential active after an explicit `--token` rotation attempt.
  - confidence: 0.88
- **[medium] File env aliases bypass SecretRef file-provider policy:** `src/utils/secret-env.ts:25`
  - body: The resolver reads arbitrary paths directly while the current SecretRef file provider has structured resolution, path/permission checks, byte and timeout limits, and explicit diagnostics; adding another credential file path needs maintainer security review.
  - confidence: 0.7
- **[low] Unreadable *_FILE secrets fail silently:** `src/utils/secret-env.ts:31`
  - body: Swallowing configured secret-file read errors hides typos or permission failures and can allow an unintended fallback credential path without an operator-visible diagnostic.
  - confidence: 0.72
