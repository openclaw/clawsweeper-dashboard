## Summary

Keep open: the provider `_FILE` idea remains unique, but this branch is not mergeable as-is because it lacks real behavior proof and still carries credential-handling regressions; the daemon-token half is already superseded on current `main`.

## What This Changes

The PR adds file-backed provider auth env-var lookup and changes daemon gateway install/audit behavior and tests toward config-backed gateway tokens.

## Best Possible Solution

Rewrite this as a narrow provider secret-loading change aligned with SecretRef policy, preserving direct-env precedence, list separators, visible diagnostics for configured file failures, and current main's daemon-token resolver.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main is direct-env only, while the PR diff shows the explicit-token early return, list normalization before splitting, and swallowed file-read errors.

## Solution Assessment

No. The file-backed env direction may be useful, but this branch mixes it with stale daemon-token internals and should be narrowed/rebased to the current SecretRef-aware token flow.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor explicit install tokens during rotation:** `src/cli/daemon-cli/install.ts:47-48`
  - body: When `openclaw gateway install --token <new-token>` runs with an existing `gateway.auth.token`, this early return keeps the old config token and reports persistence success. The later install path can then omit the service env token, so the requested rotation is ignored.
  - confidence: 0.88
- **[P2] Preserve separators for file-backed key lists:** `src/agents/live-auth-keys.ts:111-114`
  - body: `<PROVIDER>_API_KEYS` now flows through `resolveSecretEnvValue()`, which normalizes file contents before `parseKeyList()` can split them. A file containing newline-separated keys becomes one concatenated invalid key, breaking multi-key rotation.
  - confidence: 0.84
- **[P3] Report unreadable configured secret files:** `src/utils/secret-env.ts:25-32`
  - body: If a `_FILE` env var points at a typo or unreadable path, all read errors are swallowed and the resolver returns `null`, which is indistinguishable from no configuration and leaves operators with a later auth failure or fallback credential.
  - confidence: 0.74

## Security Review

Status: needs_attention

Summary: The credential-loading and daemon auth changes introduce concrete token-rotation and silent secret-file diagnostics concerns.

Concerns:

- **[medium] Explicit gateway token rotation can be ignored:** `src/cli/daemon-cli/install.ts:47`
  - body: The proposed daemon install path can keep an existing config token while treating persistence as successful, causing an explicit `--token` rotation to be dropped and leaving the old gateway credential active.
  - confidence: 0.88
- **[low] Unreadable `_FILE` secrets fail silently:** `src/utils/secret-env.ts:31`
  - body: Silently swallowing file-read errors can hide misconfigured secret files and may allow unintended fallback credentials to be used without an operator-visible warning.
  - confidence: 0.72
