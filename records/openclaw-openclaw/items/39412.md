## Summary

Keep open: the provider file-backed secret-loading change is still unique, but the branch is stale against current main's daemon-token implementation, has concrete credential-handling regressions, and lacks real behavior proof.

## What This Changes

The PR adds `*_FILE` provider auth env-var lookup, changes daemon gateway token persistence/audit behavior, and adds related tests.

## Best Possible Solution

Rebase or rewrite this as a focused provider-secret loading change aligned with SecretRef policy, preserving current main's daemon-token flow and adding diagnostics for configured secret-file failures.

## Reproduction Assessment

Yes. Source inspection shows current main still reads provider auth from direct env vars only, and the PR diff exposes the token-rotation, list-separator, and unreadable-file behaviors without needing a full runtime setup.

## Solution Assessment

No. The file-backed env direction may be useful, but this branch mixes it with obsolete daemon-token internals and introduces credential-handling regressions; a narrower provider-secret change is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor explicit install tokens during rotation:** `src/cli/daemon-cli/install.ts:47-48`
  - body: When `openclaw gateway install --token <new-token>` runs with an existing `gateway.auth.token`, this early return keeps the old config token while treating persistence as successful. The later install path can then omit the service env token, so the explicit rotation is ignored.
  - confidence: 0.88
- **[P2] Preserve separators for file-backed key lists:** `src/agents/live-auth-keys.ts:111-114`
  - body: `<PROVIDER>_API_KEYS` now flows through `resolveSecretEnvValue()`, which normalizes file contents before `parseKeyList()` can split them. A file containing `key1\nkey2` becomes one concatenated invalid key, breaking multi-key rotation.
  - confidence: 0.84
- **[P3] Report unreadable configured secret files:** `src/utils/secret-env.ts:25-32`
  - body: If a `_FILE` env var points at a typo or unreadable path, all read errors are swallowed and the resolver returns `null`, which is indistinguishable from no configuration and leaves operators with a later auth failure or fallback credential.
  - confidence: 0.74
