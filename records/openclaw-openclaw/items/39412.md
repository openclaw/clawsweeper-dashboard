## Summary

Keep open: current main already covers much of the daemon token direction through a newer shared gateway install-token path, but this PR still contains unique provider `*_FILE` work and has concrete correctness, proof, and credential-policy blockers before it can merge.

## What This Changes

The PR adds file-backed provider auth environment variable lookup, changes gateway daemon install/audit handling toward config-backed tokens, and adds auth/daemon coverage.

## Best Possible Solution

Rework the PR around the current SecretRef/auth resolver and `resolveGatewayInstallToken` contracts, preserve key-list parsing and explicit token rotation, and add redacted real behavior proof before merge.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR diff shows the existing-token early return, list separator loss, and swallowed file-read errors. I did not run tests because this was a read-only review.

## Solution Assessment

No. The daemon part should stay on the current shared `resolveGatewayInstallToken` path, and provider file-backed secrets should be reconciled with the existing SecretRef file-provider policy instead of adding a silent scalar env-file resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor explicit install tokens during rotation:** `src/cli/daemon-cli/install.ts:43-45`
  - body: When `openclaw gateway install --token <new-token>` runs with an existing `gateway.auth.token`, this early return keeps the old config token and reports persistence success. The later install plan can omit the service env token, so the user-supplied rotation is ignored.
  - confidence: 0.88
- **[P2] Preserve separators for file-backed key lists:** `src/agents/live-auth-keys.ts:111-114`
  - body: `<PROVIDER>_API_KEYS` now goes through scalar secret normalization before `parseKeyList()`. A file containing newline-separated keys is normalized into one concatenated invalid key, breaking multi-key rotation.
  - confidence: 0.84
- **[P3] Report unreadable configured secret files:** `src/utils/secret-env.ts:31-32`
  - body: If a `_FILE` env var points at a typo or unreadable path, the resolver swallows the read error and returns `null`, leaving operators with a later auth failure or fallback credential and no clue that the configured secret file failed to load.
  - confidence: 0.74

## Security Review

Status: needs_attention

Summary: The diff changes credential loading and daemon token persistence and still has concrete token-rotation and secret-file diagnostic risks.

Concerns:

- **[medium] Explicit gateway token rotation can be ignored:** `src/cli/daemon-cli/install.ts:43`
  - body: The proposed daemon install path can keep an existing config token while treating persistence as successful, leaving the old gateway credential active after an explicit `--token` rotation attempt.
  - confidence: 0.88
- **[low] Unreadable `_FILE` secrets fail silently:** `src/utils/secret-env.ts:31`
  - body: Swallowing configured secret-file read errors hides misconfigured secret files and can allow an unintended fallback credential path without an operator-visible diagnostic.
  - confidence: 0.72
