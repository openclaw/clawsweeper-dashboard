## Summary

Keep open for maintainer review rather than cleanup-close: current main covers the daemon-token direction, but the provider credential alias work is still unique and this branch has blocking correctness/security findings plus no real behavior proof.

## What This Changes

The branch adds `_FILE` provider auth env resolution, moves daemon gateway token persistence toward config-backed installs, and updates auth/daemon audit tests.

## Best Possible Solution

Rebase or replace this with a narrower maintainer-approved credential-loading change that uses current SecretRef/auth resolver seams, preserves list parsing and token rotation semantics, and surfaces configured file-read failures.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR diff shows the token-rotation early return, newline separator loss, and silent file-read failure paths. No after-fix real setup proof was supplied by the contributor.

## Solution Assessment

No. Current main has a better daemon-token seam, and the provider `_FILE` support should be reconciled with the SecretRef credential contract instead of adding a silent scalar env-file resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor explicit install tokens during rotation:** `src/cli/daemon-cli/install.ts:47-48`
  - body: When `openclaw gateway install --token <new-token>` runs with an existing `gateway.auth.token`, this early return keeps the old config token and reports persistence success. The later install plan can omit the service env token, so the user-supplied rotation is ignored.
  - confidence: 0.88
- **[P2] Preserve separators for file-backed key lists:** `src/agents/live-auth-keys.ts:111-114`
  - body: `<PROVIDER>_API_KEYS` now goes through `resolveSecretEnvValue()` before `parseKeyList()`. A file containing newline-separated keys is normalized into one concatenated invalid key, breaking multi-key rotation.
  - confidence: 0.84
- **[P3] Report unreadable configured secret files:** `src/utils/secret-env.ts:31-32`
  - body: If a `_FILE` env var points at a typo or unreadable path, the resolver swallows the read error and returns `null`, leaving operators with a later auth failure or fallback credential and no clue that the configured secret file failed to load.
  - confidence: 0.74

## Security Review

Status: needs_attention

Summary: The diff changes credential loading and daemon token persistence and introduces concrete token-rotation and secret-file diagnostic risks.

Concerns:

- **[medium] Explicit gateway token rotation can be ignored:** `src/cli/daemon-cli/install.ts:47`
  - body: The proposed daemon install path can keep an existing config token while treating persistence as successful, leaving the old gateway credential active after an explicit `--token` rotation attempt.
  - confidence: 0.88
- **[low] Unreadable `_FILE` secrets fail silently:** `src/utils/secret-env.ts:31`
  - body: Swallowing all configured secret-file read errors hides misconfigured secret files and can allow unintended fallback credentials to be used without an operator-visible warning.
  - confidence: 0.72
