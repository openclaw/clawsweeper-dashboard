## Summary

Keep open: current main has superseded much of the daemon-token direction, but provider `*_FILE` env support is still not implemented and this branch has correctness and credential-handling blockers before it can merge.

## What This Changes

The PR adds `*_FILE` provider secret env-var lookup, changes daemon gateway installs toward config-backed tokens, and updates related auth/audit tests.

## Best Possible Solution

Land a narrower credential-loading change only if maintainers want provider `*_FILE` env compatibility, preserving list separators, surfacing configured file-read failures, and leaving current main's shared daemon-token resolver intact.

## Reproduction Assessment

Yes. Source inspection of the PR branch reproduces the introduced defects, and current-main source inspection confirms the provider `*_FILE` feature is not already implemented.

## Solution Assessment

No. The useful provider-secret idea needs a SecretRef-aligned implementation, while the daemon-token changes are stale and conflict with the current shared install-token direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor explicit install tokens during rotation:** `src/cli/daemon-cli/install.ts:47-48`
  - body: When `openclaw gateway install --token <new-token>` runs with an existing `gateway.auth.token`, this early return keeps the old config token and reports persistence success. The later install plan can omit the service env token, so the user-supplied rotation is ignored.
  - confidence: 0.88
- **[P2] Preserve separators for file-backed key lists:** `src/agents/live-auth-keys.ts:111-114`
  - body: `<PROVIDER>_API_KEYS` now goes through `resolveSecretEnvValue()` before `parseKeyList()`. A file containing newline-separated keys is normalized into one concatenated invalid key, breaking multi-key rotation.
  - confidence: 0.84
- **[P3] Report unreadable configured secret files:** `src/utils/secret-env.ts:31-32`
  - body: If a `_FILE` env var points at a typo or unreadable path, the resolver swallows the read error and returns `null`. Operators then see a later auth failure or fallback credential with no indication that the configured secret file failed to load.
  - confidence: 0.74

## Security Review

Status: needs_attention

Summary: The diff changes credential loading and daemon token persistence and introduces concrete token-rotation and secret-file diagnostic risks.

Concerns:

- **[medium] Explicit gateway token rotation can be ignored:** `src/cli/daemon-cli/install.ts:47`
  - body: The proposed daemon install path can keep an existing config token while treating persistence as successful, leaving the old gateway credential active after an explicit `--token` rotation attempt.
  - confidence: 0.88
- **[low] Unreadable `_FILE` secrets fail silently:** `src/utils/secret-env.ts:31`
  - body: Swallowing all configured secret-file read errors hides misconfigured secret files and can allow unintended fallback credentials to be used without an operator-visible warning.
  - confidence: 0.72
