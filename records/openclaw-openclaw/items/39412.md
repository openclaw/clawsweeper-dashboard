## Summary

Current main already covers the useful daemon gateway-token behavior through the shared install-token path and covers file-backed provider credentials through the shipped SecretRef file-provider contract. The remaining `*_FILE` env-alias part is obsolete/risky against that contract, and the stale branch still has blocking auth/security review findings plus no real behavior proof.

## What This Changes

The branch adds `*_FILE` provider API-key resolution, moves daemon gateway token handling toward config-backed installs, and updates auth/install/audit tests.

## Best Possible Solution

Keep the shipped SecretRef file-provider and shared `resolveGatewayInstallToken` daemon implementation; discuss any future `*_FILE` alias as a new SecretRef-compatible security/product design.

## Reproduction Assessment

Yes for the PR review findings: source inspection at PR head shows the token-rotation early return, list-separator loss, and silent file-read failure paths. This is not a current-main bug reproduction because current main uses the shared daemon-token and SecretRef file-provider paths instead.

## Solution Assessment

No. Current main's shared daemon-token helper plus the documented SecretRef file-provider contract is the narrower maintainable solution; the PR's separate `*_FILE` resolver bypasses that security policy and still has merge-blocking defects.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor explicit install tokens during rotation:** `src/cli/daemon-cli/install.ts:47-48`
  - body: When `openclaw gateway install --token <new-token>` runs with an existing `gateway.auth.token`, this early return keeps the old config token and reports persistence success. The later install plan can omit the service env token, so the user-supplied rotation is ignored.
  - confidence: 0.88
- **[P2] Preserve separators for file-backed key lists:** `src/agents/live-auth-keys.ts:111-114`
  - body: `<PROVIDER>_API_KEYS` now goes through scalar secret normalization before `parseKeyList()`. A file containing newline-separated keys is normalized into one concatenated invalid key, breaking multi-key rotation.
  - confidence: 0.84
- **[P3] Report unreadable configured secret files:** `src/utils/secret-env.ts:31-32`
  - body: If a `_FILE` env var points at a typo or unreadable path, the resolver swallows the read error and returns `null`, leaving operators with a later auth failure or fallback credential and no clue that the configured secret file failed to load.
  - confidence: 0.74

## Security Review

Status: needs_attention

Summary: The patch changes credential loading and gateway token persistence and still has concrete token-rotation, SecretRef policy, and diagnostic risks.

Concerns:

- **[medium] Explicit gateway token rotation can be ignored:** `src/cli/daemon-cli/install.ts:47`
  - body: The daemon install path can keep an existing config token while treating persistence as successful, leaving the old gateway credential active after an explicit `--token` rotation attempt.
  - confidence: 0.88
- **[medium] File env aliases bypass SecretRef file-provider policy:** `src/utils/secret-env.ts:25`
  - body: The new resolver reads arbitrary paths directly instead of using the current SecretRef file provider with structured resolution, path/permission checks, byte and timeout limits, and explicit diagnostics.
  - confidence: 0.7
- **[low] Unreadable `*_FILE` secrets fail silently:** `src/utils/secret-env.ts:31`
  - body: Swallowing configured secret-file read errors hides typos or permission failures and can allow an unintended fallback credential path without operator-visible diagnostics.
  - confidence: 0.72
