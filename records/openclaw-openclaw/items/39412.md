## Summary

Keep open: this PR still contains security-sensitive credential-loading changes with concrete correctness findings, no real behavior proof, and a credential-policy question around adding `*_FILE` aliases outside the existing SecretRef contract.

## What This Changes

The PR adds file-backed provider auth env-var aliases, routes provider auth readers through a new secret-env resolver, and changes daemon gateway installs and drift checks toward config-backed gateway token auth.

## Best Possible Solution

Rebase onto current main, keep the shared gateway install token path, and implement any provider file-backed env support through a SecretRef-compatible design with explicit diagnostics and multi-key parsing coverage.

## Reproduction Assessment

Yes for the PR review findings: source inspection shows the explicit-token early return, scalar normalization before list parsing, and silent `_FILE` read-failure path. I did not run tests or live daemon/provider commands because this was a read-only review and the missing gate is real behavior proof.

## Solution Assessment

No. The daemon portion should be reconciled with current main's shared `resolveGatewayInstallToken` path, and provider file-backed secrets need a deliberate SecretRef-compatible design rather than a separate silent file-read alias.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor explicit install tokens during rotation:** `src/cli/daemon-cli/install.ts:47-48`
  - body: When `openclaw gateway install --token <new-token>` runs with an existing `gateway.auth.token`, this early return keeps the old config token and reports persistence success. The later install plan can omit the service env token, so the user-supplied rotation is ignored.
  - confidence: 0.88
- **[P2] Preserve separators for file-backed key lists:** `src/agents/live-auth-keys.ts:111-114`
  - body: `<PROVIDER>_API_KEYS` now goes through scalar secret normalization before `parseKeyList()`. A file containing newline-separated keys is normalized into one concatenated invalid key, breaking multi-key rotation.
  - confidence: 0.84
- **[P3] Report unreadable configured secret files:** `src/utils/secret-env.ts:31-32`
  - body: If a `_FILE` env var points at a typo or unreadable path, the resolver swallows the read error and returns `null`, leaving operators with a later auth failure or fallback credential and no clue that the configured secret file failed to load.
  - confidence: 0.74

## Security Review

Status: needs_attention

Summary: The diff touches credential loading and daemon token persistence and has concrete token-rotation, diagnostic, and credential-policy concerns.

Concerns:

- **[medium] Explicit gateway token rotation can be ignored:** `src/cli/daemon-cli/install.ts:47`
  - body: The proposed daemon install path can keep an existing config token while treating persistence as successful, leaving the old gateway credential active after an explicit `--token` rotation attempt.
  - confidence: 0.88
- **[low] Unreadable `_FILE` secrets fail silently:** `src/utils/secret-env.ts:31`
  - body: Swallowing configured secret-file read errors hides misconfigured secret files and can allow an unintended fallback credential path without an operator-visible diagnostic.
  - confidence: 0.72
- **[medium] File env aliases bypass SecretRef file-provider policy:** `src/utils/secret-env.ts:25`
  - body: The resolver reads arbitrary file paths directly while current SecretRef file-provider paths use structured resolution, permission checks, and diagnostics; this policy choice needs maintainer review before adding another credential file path.
  - confidence: 0.7
