## Summary

Keep open: this security-sensitive PR still has actionable correctness findings, missing real behavior proof, and a credential-policy question around adding `*_FILE` aliases beside the existing SecretRef file-provider surface.

## What This Changes

The PR adds file-backed provider secret env var loading via `*_FILE`, persists daemon gateway tokens into config-backed installs, and updates auth/daemon drift tests.

## Best Possible Solution

Rebase onto current main, keep the shared daemon token resolver, and design provider file-backed secret support through or alongside SecretRef with explicit diagnostics and multi-key parsing coverage.

## Reproduction Assessment

Yes for the PR review findings: source inspection of the PR diff shows the token-rotation early return, scalar normalization before list parsing, and silent `_FILE` read failure path. I did not run tests because this is a read-only review.

## Solution Assessment

No. The daemon portion should stay on current main's shared `resolveGatewayInstallToken` implementation, and provider file-backed secrets should be reconciled with the existing SecretRef file-provider policy before adding a new env alias convention.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor explicit install tokens during rotation:** `src/cli/daemon-cli/install.ts:47-48`
  - body: When `openclaw gateway install --token <new-token>` runs with an existing `gateway.auth.token`, this early return keeps the old config token and reports persistence success. The later install plan can omit the service env token, so the user-supplied rotation is ignored.
  - confidence: 0.88
- **[P2] Preserve separators for file-backed key lists:** `src/agents/live-auth-keys.ts:111-114`
  - body: `<PROVIDER>_API_KEYS` now goes through scalar secret normalization before `parseKeyList()`. A file containing newline-separated keys is normalized into one concatenated invalid key, breaking multi-key rotation.
  - confidence: 0.84
- **[P3] Report unreadable configured secret files:** `src/utils/secret-env.ts:31-32`
  - body: If a `_FILE` env var points at a typo or unreadable path, the resolver swallows the read error and returns `null`, leaving operators with a later auth failure or fallback credential and no clue that the configured secret file failed to load.
  - confidence: 0.74

## Security Review

Status: needs_attention

Summary: The diff changes credential loading and daemon token persistence and has concrete token-rotation, diagnostic, and credential-policy concerns.

Concerns:

- **[medium] Explicit gateway token rotation can be ignored:** `src/cli/daemon-cli/install.ts:47`
  - body: The proposed daemon install path can keep an existing config token while treating persistence as successful, leaving the old gateway credential active after an explicit `--token` rotation attempt.
  - confidence: 0.88
- **[low] Unreadable `_FILE` secrets fail silently:** `src/utils/secret-env.ts:31`
  - body: Swallowing configured secret-file read errors hides misconfigured secret files and can allow an unintended fallback credential path without an operator-visible diagnostic.
  - confidence: 0.72
- **[medium] File env aliases bypass SecretRef file-provider policy:** `src/utils/secret-env.ts:25`
  - body: The resolver reads arbitrary file paths directly while current SecretRef file-provider paths use structured resolution and diagnostics; this policy choice needs maintainer review before adding another credential file path.
  - confidence: 0.7
