## Summary

Keep this PR open, but it is not merge-ready: the provider `*_FILE` direction is useful security work, while the daemon-token portion is largely superseded by current main and the branch still has credential-handling regressions plus no real behavior proof.

## What This Changes

The PR adds file-backed provider auth env-var lookup via `*_FILE`, changes daemon gateway install/audit behavior to rely on config-backed tokens, and adds related tests.

## Best Possible Solution

Rewrite this as a focused, SecretRef-aligned provider secret-loading change, preserving direct-env precedence, list separators, clear diagnostics for configured files, and current main's daemon-token flow.

## Reproduction Assessment

Yes. Source inspection of the PR diff shows the explicit-token rotation, file-backed list parsing, and unreadable-file diagnostic problems, and current-main source confirms the daemon token path has already moved on.

## Solution Assessment

No. The provider file-backed env idea may be worth pursuing, but this PR mixes it with stale daemon-token internals and introduces credential-handling regressions; a narrower provider-secret patch is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor explicit install tokens during rotation:** `src/cli/daemon-cli/install.ts:47-48`
  - body: When `openclaw gateway install --token <new-token>` runs with an existing `gateway.auth.token`, this early return treats the old config token as persisted and later suppresses the service env token. The user-supplied rotation is ignored and the old credential remains active.
  - confidence: 0.88
- **[P2] Preserve separators for file-backed key lists:** `src/agents/live-auth-keys.ts:111-114`
  - body: `<PROVIDER>_API_KEYS` now goes through `resolveSecretEnvValue()` before `parseKeyList()`. A file containing newline-separated keys can be normalized into one concatenated invalid key, breaking multi-key rotation.
  - confidence: 0.84
- **[P3] Report unreadable configured secret files:** `src/utils/secret-env.ts:25-32`
  - body: If a `_FILE` env var points at a typo or unreadable path, the resolver swallows all read errors and returns `null`. Operators then see a later auth failure or fallback credential with no indication that the configured secret file failed to load.
  - confidence: 0.74
