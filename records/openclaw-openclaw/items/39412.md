## Summary

Keep open: current main has already moved the daemon-token handling toward a shared config-backed resolver, but this PR still proposes provider `*_FILE` auth aliases and remains blocked by correctness/security findings plus missing real behavior proof.

## What This Changes

The PR adds `*_FILE` provider auth env-var lookup, changes daemon gateway token persistence and audit behavior, and adds auth/daemon tests.

## Best Possible Solution

Rebase around current `resolveGatewayInstallToken` and SecretRef file-provider contracts, fix token rotation/list parsing/diagnostics, then add redacted runtime proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR diff and current normalization/token-resolution code shows the token-rotation early return, newline separator loss, and silent file-read failure paths. I did not run tests because this was a read-only review.

## Solution Assessment

No. The daemon portion should stay on current main's shared `resolveGatewayInstallToken` path, and provider file-backed secrets should be reconciled with the existing SecretRef file-provider policy instead of adding a silent scalar env-file resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor explicit install tokens during rotation:** `src/cli/daemon-cli/install.ts:47-48`
  - body: When `openclaw gateway install --token <new-token>` runs with an existing `gateway.auth.token`, this early return keeps the old config token and reports persistence success. The later install plan can omit the service env token, so the user-supplied rotation is ignored.
  - confidence: 0.88
- **[P2] Preserve separators for file-backed key lists:** `src/agents/live-auth-keys.ts:111-114`
  - body: `<PROVIDER>_API_KEYS` now goes through scalar secret normalization before `parseKeyList()`. A file containing newline-separated keys is normalized into one concatenated invalid key, breaking multi-key rotation.
  - confidence: 0.84
- **[P3] Report unreadable configured secret files:** `src/utils/secret-env.ts:31-32`
  - body: If a `_FILE` env var points at a typo or unreadable path, the resolver swallows the read error and returns `null`, leaving operators with a later auth failure or fallback credential and no clue that the configured secret file failed to load.
  - confidence: 0.74

## Security Review

Status: needs_attention

Summary: The diff changes credential loading and daemon token persistence and still has concrete token-rotation and secret-file diagnostic risks.

Concerns:

- **[medium] Explicit gateway token rotation can be ignored:** `src/cli/daemon-cli/install.ts:47`
  - body: The proposed daemon install path can keep an existing config token while treating persistence as successful, leaving the old gateway credential active after an explicit `--token` rotation attempt.
  - confidence: 0.88
- **[low] Unreadable `_FILE` secrets fail silently:** `src/utils/secret-env.ts:31`
  - body: Swallowing configured secret-file read errors hides misconfigured secret files and can allow an unintended fallback credential path without an operator-visible diagnostic.
  - confidence: 0.72
- **[medium] File-backed env aliases bypass the SecretRef file-provider contract:** `src/utils/secret-env.ts:25`
  - body: The new resolver reads arbitrary file paths directly, while the current SecretRef file provider uses bounded secure file reads and structured resolution errors; that policy choice needs maintainer review before adding another credential file path.
  - confidence: 0.7
