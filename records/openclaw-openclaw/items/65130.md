## Summary

Current main already fixes the reported container/global npm update failure through a shipped staged-prefix update flow. Instead of adding broad npm `--force`, OpenClaw now installs npm global updates into a temporary prefix, verifies that staged package, then swaps the package tree and bin shim into the real global prefix, so an existing `/usr/local/bin/openclaw` no longer blocks the npm install step.

## What This Changes

The issue asks OpenClaw package-manager updates in Docker/container installs to replace an existing global OpenClaw binary instead of failing with npm `EEXIST` at `/usr/local/bin/openclaw`.

## Best Possible Solution

Keep the staged-prefix package update implementation and close this report as shipped. The older `--force` PR direction is superseded by a safer implementation that avoids npm overwriting the real global bin path during install and then performs OpenClaw-controlled verification and swap.

## Reproduction Assessment

Yes. The original failure path maps to the global npm update flow, and current source plus focused tests show that npm now installs into a staged prefix before any live `/usr/local/bin/openclaw` shim is replaced, so the reported `EEXIST` condition should no longer reproduce on the updated path.

## Solution Assessment

Yes. The shipped staged-prefix and swap approach is narrower and safer than adding npm `--force` to every global update because it avoids npm touching the existing live bin path during install while preserving explicit verification and rollback-oriented shim handling.
