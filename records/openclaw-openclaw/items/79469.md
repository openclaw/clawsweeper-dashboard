## Summary

Keep open: current main and the latest release still treat WebChat reasoning payloads as delivery-only data and exclude them from completed transcript replay, while the Control UI can only render thinking that is already present in the stored message object. The active related PRs address pieces of live reasoning streaming, not the end-to-end completed-reply visibility and retention contract requested here.

## What This Changes

The issue asks for completed WebChat/Control UI replies to keep reasoning visible or manually expandable after streaming finishes.

## Best Possible Solution

Define a WebChat reasoning disclosure and retention contract that reuses existing reasoning and Show Thinking gates, distinguishes transient delivery payloads from durable transcript content, and updates gateway, Control UI, tests, and docs together.

## Reproduction Assessment

Yes, at source level: current docs, gateway code, and tests exclude `isReasoning` WebChat payloads from completed replay, while the UI only renders thinking already present in stored message content. I did not run a live browser reproduction.

## Solution Assessment

Unclear until maintainers define the UX/privacy contract. A blanket always-show replay is riskier than a scoped design that reuses existing reasoning/session gates and explicitly decides what reasoning, if any, is retained after completion.
