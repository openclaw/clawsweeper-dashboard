## Summary

Keep open: current main still treats WebChat reasoning payloads as transient delivery data and excludes them from completed transcript replay, while the Control UI can only render reasoning that is already present as persisted thinking content. The prior ClawSweeper review reached the same conclusion, and the open related PRs have not merged a complete end-to-end fix.

## What This Changes

Requests a Control UI/WebChat way to keep reasoning visible or expandable after a response completes.

## Best Possible Solution

Define a WebChat reasoning disclosure and retention contract that preserves existing `/reasoning` and Show Thinking gates, distinguishes transient delivery payloads from durable transcript content, and updates gateway plus Control UI coverage together.

## Reproduction Assessment

Yes, at source level: current docs, gateway code, and tests deliberately exclude `isReasoning` WebChat payloads from completed replay, while the UI only renders reasoning already present in stored message content. I did not run a live browser reproduction.

## Solution Assessment

Unclear until maintainers define the UX/privacy contract. A blanket always-show replay would be riskier than a scoped design that reuses existing reasoning/session gates and explicitly decides what reasoning, if any, is retained after completion.
