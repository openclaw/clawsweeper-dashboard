## Summary

Keep open: current main still intentionally excludes reasoning-only WebChat payloads from completed transcript replay, while the Control UI can only show reasoning when a persisted thinking block exists and the existing display/session gates are enabled. This is a valid WebChat UX/product request, not conservative cleanup.

## What This Changes

Request a WebChat/Control UI affordance to keep completed reasoning visible or manually expandable after a response finishes.

## Best Possible Solution

Define a Control UI/WebChat reasoning disclosure contract that reuses existing reasoning/session gates, clearly separates transient delivery payloads from durable transcript content, and adds focused WebChat render and gateway replay coverage.

## Reproduction Assessment

Yes, at source level: current docs, gateway code, and tests deliberately exclude `isReasoning` WebChat payloads from completed replay, while the UI only renders reasoning that remains in persisted message content. I did not run a live browser reproduction.

## Solution Assessment

Unclear until maintainer UX/privacy direction is set. The likely maintainable path is a scoped WebChat reasoning-disclosure design over the existing `reasoningLevel` and Show Thinking gates, not an unconditional replay of all transient reasoning payloads.
