## Summary

Keep open. Current main still has the source-level stale-terminal-assistant path: incomplete-turn classification and payload construction can key off `lastAssistant=toolUse` even when `currentAttemptAssistant=stop` contains the final visible answer; the text-only deliverability part is already fixed.

## What This Changes

The issue requests a bug fix for embedded runs that misclassify a completed post-`update_plan` assistant stop turn as incomplete and drop the final text before channel delivery.

## Best Possible Solution

Land a narrow terminal-assistant freshness fix in the embedded runner, preserving the existing true-terminal-`toolUse` guard and tracking broader observability/recovery ideas separately.

## Reproduction Assessment

Yes for a source-level reproduction: synthesize an attempt with `lastAssistant.stopReason="toolUse"`, `currentAttemptAssistant.stopReason="stop"`, and visible `assistantTexts`; current main still enters the incomplete-turn branch. A live WhatsApp/OpenRouter reproduction was not run in this read-only review.

## Solution Assessment

Yes, the narrow best fix is to select the freshest terminal assistant state, `currentAttemptAssistant ?? lastAssistant`, at the classifier, payload, final-metadata, and logging boundary. That addresses this bug without weakening the established guard for truly unfinished tool-use turns.
