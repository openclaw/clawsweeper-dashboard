## Summary

Keep open: current main and v2026.6.6 mitigate the original zero-send path by making text-only payloads deliverable, but the stale `lastAssistant` incomplete-turn false positive still appears present and can still replace the real final answer with an error payload. A closed unmerged PR attempted the exact fix, so the remaining work is narrow rather than obsolete.

## What This Changes

The issue reports a silent channel send miss when a final `stop` assistant turn after `update_plan` is misclassified from a stale previous `toolUse` assistant snapshot and the collected final text is dropped.

## Best Possible Solution

Land a narrow regression fix equivalent to the useful part of https://github.com/openclaw/openclaw/pull/80931 on the current `embedded-agent-runner` paths, and leave recovery commands or broader observability work for separate items.

## Reproduction Assessment

Yes at source level: current main still lets stale `lastAssistant.stopReason === "toolUse"` drive incomplete-turn classification even when a fresher `currentAttemptAssistant` exists. I did not run a live WhatsApp/OpenRouter reproduction in this read-only pass.

## Solution Assessment

Yes, with a narrower scope than the original P0-P2 list: use the actual current terminal assistant for classification and preserve real assistant text when warning fallback runs. Do not rework text-only deliverability, because current main and v2026.6.6 already cover that leg.
