## Summary

Keep open. Current main has the text-only deliverability mitigation, but the embedded-runner incomplete-turn path still reads stale `lastAssistant` for the terminal guard and still replaces collected final assistant text with the generic incomplete-turn error when that branch fires; the focused linked PR was closed unmerged.

## What This Changes

Fix a silent embedded-runner send miss where stale incomplete-turn state after `update_plan` can discard a completed `stopReason="stop"` final answer.

## Best Possible Solution

Use the freshest terminal assistant state for incomplete-turn classification, payload construction, and final assistant metadata, add a stale-`lastAssistant` regression, and track broader recovery or observability ideas separately.

## Reproduction Assessment

Yes for source-level reproduction: an attempt with `lastAssistant.stopReason="toolUse"`, `currentAttemptAssistant.stopReason="stop"`, and non-empty final `assistantTexts` still follows stale `lastAssistant` checks on current main. I did not establish the full live WhatsApp path in this read-only review.

## Solution Assessment

No, current main is not the best solution because it only mitigates text-only deliverability and still replaces the real final answer with a generic incomplete-turn error. The narrow maintainable fix is to select `currentAttemptAssistant ?? lastAssistant` at the classifier and payload boundaries while preserving the true terminal-toolUse guard.
