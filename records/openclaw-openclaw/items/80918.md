## Summary

Keep open. Current main has the text-only deliverability mitigation, but the embedded-runner incomplete-turn path can still use stale `lastAssistant` state and then replace the real final-answer payloads with the generic incomplete-turn error.

## What This Changes

The issue reports that embedded-runner incomplete-turn handling can drop a final plain-text answer after an `update_plan` tool call and mark the run successful without sending a channel message.

## Best Possible Solution

Use the freshest terminal assistant state for incomplete-turn classification, payload construction, and final assistant metadata, add a stale-`lastAssistant` regression, and track broader recovery or observability ideas separately.

## Reproduction Assessment

Yes for source-level reproduction: an attempt with stale `lastAssistant.stopReason="toolUse"`, fresh `currentAttemptAssistant.stopReason="stop"`, and non-empty final text still follows the incomplete-turn branch. I did not establish the full live WhatsApp channel path in this read-only review.

## Solution Assessment

No, current main is not the best solution because it only mitigates text-only deliverability and still replaces the real final answer with a generic incomplete-turn error. The narrow maintainable fix is to select `currentAttemptAssistant ?? lastAssistant` at the classifier and payload boundaries while preserving the true terminal-toolUse guard.
