## Summary

Keep open. Current main still has the stale `lastAssistant` path in the incomplete-turn classifier and still replaces collected final assistant text with the generic incomplete-turn error when that branch fires; the linked focused fix was closed unmerged, while text-only deliverability itself has already been mitigated.

## What This Changes

The issue asks OpenClaw to preserve and deliver a final `stop` assistant answer after `update_plan` even when `lastAssistant` still points at the previous `toolUse` snapshot.

## Best Possible Solution

Use the freshest terminal assistant state for incomplete-turn classification, payload construction, and final assistant metadata, add a stale-`lastAssistant` regression, and keep broader recovery or observability work separate.

## Reproduction Assessment

Yes for source-level reproduction: an attempt with `lastAssistant.stopReason = "toolUse"`, `currentAttemptAssistant.stopReason = "stop"`, non-empty `assistantTexts`, and `payloadCount > 0` still follows the stale `lastAssistant` classifier and incomplete-branch return on current main. I did not establish the full live WhatsApp path in this read-only review.

## Solution Assessment

Yes, the narrow maintainable solution is to select `currentAttemptAssistant ?? lastAssistant` at the classifier and payload/final-metadata boundaries while preserving the real terminal-`toolUse` guard. The closed unmerged PR demonstrates that shape but is not on main.
