## Summary

Keep open. Current main still uses stale `attempt.lastAssistant` for the terminal tool-use incomplete-turn branch, so the reported final-answer discard path remains source-reproducible; the linked fix PR was closed unmerged and no merged canonical fix supersedes this issue.

## What This Changes

The issue reports that a post-`update_plan` final `stopReason=stop` text turn can be discarded when incomplete-turn detection reads a stale `toolUse` assistant snapshot, causing a silent WhatsApp send miss.

## Best Possible Solution

Land a narrow embedded-runner fix that uses the freshest terminal assistant state for incomplete-turn classification and final payload metadata, with regression coverage for the `update_plan` stale-`lastAssistant` shape while preserving the existing true terminal `toolUse` guard.

## Reproduction Assessment

Yes for a source-level reproduction: construct an attempt with `lastAssistant.stopReason = "toolUse"`, a fresh `currentAttemptAssistant.stopReason = "stop"`, and final visible assistant text, and current main still lets the stale `lastAssistant` drive incomplete-turn handling. I did not reproduce the full live WhatsApp/OpenRouter delivery path.

## Solution Assessment

Yes, the narrow fix direction from the closed PR is the best current path: select `currentAttemptAssistant ?? lastAssistant` at the classifier and payload boundary, then keep broader alarms, failed-queue records, and recovery CLI work out of the first repair.
