## Summary

Keep open. Current main has fixed the text-only deliverability filter, but the embedded-runner path still classifies from stale `lastAssistant` and still drops collected final assistant text when the incomplete-turn branch fires; the linked focused PR was closed unmerged.

## What This Changes

The issue asks OpenClaw to preserve and deliver the final `stopReason=stop` assistant text after `update_plan` instead of treating the run as incomplete from a stale `lastAssistant=toolUse` snapshot.

## Best Possible Solution

Use the freshest terminal assistant state for incomplete-turn classification, payload construction, and final assistant metadata, with a focused regression for stale `lastAssistant=toolUse` after `update_plan`.

## Reproduction Assessment

Yes for source-level reproduction: an attempt with `lastAssistant.stopReason="toolUse"`, `currentAttemptAssistant.stopReason="stop"`, and non-empty final `assistantTexts` still follows the incomplete-turn branch because current main checks `lastAssistant`. I did not establish the full live WhatsApp path in this read-only review.

## Solution Assessment

No, current main is not the best solution because it only fixes text-only deliverability and still replaces the real final answer with a generic incomplete-turn error. The narrow maintainable fix is the closed PR's direction: select `currentAttemptAssistant ?? lastAssistant` at the classifier and payload boundaries while preserving the true terminal-toolUse guard.
