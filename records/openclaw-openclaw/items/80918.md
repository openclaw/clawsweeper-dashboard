## Summary

Keep open. Current main still has the stale `lastAssistant` path that can classify a completed post-`update_plan` final answer as incomplete and then replace collected assistant text with a generic error, although the separate text-only deliverability bug is already fixed; https://github.com/openclaw/openclaw/pull/80931 is a useful but closed, unmerged candidate fix.

## What This Changes

The issue reports that embedded agent runs can discard a final text response after an `update_plan` tool turn when stale incomplete-turn state classifies the terminal assistant as `toolUse`.

## Best Possible Solution

Apply the narrow terminal-assistant selection fix on current main, preserving the true terminal-toolUse guard for https://github.com/openclaw/openclaw/issues/76477 and leaving broader observability/recovery ideas for separate follow-up work.

## Reproduction Assessment

Yes for a source-level reproduction: use an attempt with stale `lastAssistant.stopReason = "toolUse"`, fresh `currentAttemptAssistant.stopReason = "stop"` with visible text, and `payloadCount = 1`; current main still keys the incomplete-turn guard on `lastAssistant`. I did not establish a live WhatsApp/OpenRouter reproduction in this read-only review.

## Solution Assessment

No, current main is not the best solution because it only fixes text-only deliverability and still drops real final text after a false incomplete classification. The narrow maintainable fix is to use `currentAttemptAssistant ?? lastAssistant` at the classifier and payload/final metadata boundaries, with a focused regression test.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/80918 on current main. The embedded runner already computes `currentAttemptAssistant`, but `resolveIncompleteTurnPayloadText`, final assistant metadata, and `buildEmbeddedRunPayloads` still rely on stale `lastAssistant`, so a run shaped as `assistant(toolUse update_plan) -> toolResult(empty) -> assistant(stop, visible final text)` can be falsely classified incomplete and lose the real final answer. Use https://github.com/openclaw/openclaw/pull/80931 as reference, but implement against current main. Select `terminalAssistant = attempt.currentAttemptAssistant ?? attempt.lastAssistant` at the incomplete-turn classifier and payload/final metadata boundaries. Preserve the existing protection for truly terminal tool-use turns with only pre-tool narration from https://github.com/openclaw/openclaw/issues/76477. Add focused helper and runner regression tests for stale `lastAssistant=toolUse` plus fresh `currentAttemptAssistant=stop` after `update_plan`. Do not add broader recovery commands, delivery-queue changes, or product/config options in this PR. Add a changelog entry if maintainers treat the user-visible data-loss fix as release-facing.
