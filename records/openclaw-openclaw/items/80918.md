## Summary

Keep open. The reported zero-send leg is not fully supported by current source or the inspected v2026.5.7 npm package because text-only fallback payloads are deliverable, but current `main` still has the source-provable false incomplete-turn/data-loss path: it keys the terminal check on `lastAssistant.stopReason` and the incomplete branch returns only the generic error payload instead of preserving collected assistant text.

## What This Changes

The issue asks OpenClaw to stop dropping a real final assistant answer when an embedded run follows a final tool call such as `update_plan` with a plain-text `stopReason="stop"` turn.

## Best Possible Solution

Keep the issue open and land a focused runner fix that selects the freshest terminal assistant message or synchronizes `lastAssistant`, preserves non-empty `assistantTexts` through any fallback, and leaves recovery CLI or broader observability work for a separate decision.

## Reproduction Assessment

Yes for the remaining false incomplete-turn/data-loss path: a source-level regression can build an attempt with visible final `assistantTexts` and stale `lastAssistant.stopReason="toolUse"`, which current `resolveIncompleteTurnPayloadText` treats as incomplete. The full zero-send symptom is not source-proven because current delivery filtering accepts text-only fallback payloads.

## Solution Assessment

Yes, if scoped to the P0 runner behavior rather than the whole wishlist. The narrowest maintainable fix is to base terminal classification on the actual latest assistant turn and to keep collected assistant text deliverable when fallback classification still happens.
