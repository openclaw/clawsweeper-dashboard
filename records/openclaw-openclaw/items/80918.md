## Summary

Keep open: current main and the latest release still pass stale `lastAssistant` into the final incomplete-turn classifier, and https://github.com/openclaw/openclaw/pull/94637 is the open linked candidate fix. Closing this issue before that PR lands would lose the active implementation path for a source-reproducible message-loss bug.

## What This Changes

The issue reports that a stale `lastAssistant.stopReason` can misclassify a clean final assistant turn after `update_plan`, replacing the real final text with an incomplete-turn warning and causing a channel send miss.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/94637, or replace it with an equivalent narrow classifier fix plus regression coverage that preserves the incomplete-turn safety contract.

## Reproduction Assessment

Yes, at source level: current main still computes a fresh `currentAttemptAssistant` but passes stale `lastAssistant` into the final classifier call. I did not run a live WhatsApp/OpenRouter reproduction in this read-only review.

## Solution Assessment

Yes: the best narrow fix is to classify the final incomplete-turn check with the resolved current-attempt assistant while keeping the early tool-use guard that protects the existing incomplete-turn safety contract.
