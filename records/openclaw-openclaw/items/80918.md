## Summary

Keep open. Current main still has the stale `lastAssistant` source path that can classify a completed post-`update_plan` final answer as incomplete, and the focused fix PR was closed unmerged; only the text-only deliverability part is already fixed.

## What This Changes

The issue reports that embedded runs can discard a final `stopReason="stop"` assistant answer after `update_plan` when stale `lastAssistant.stopReason="toolUse"` drives incomplete-turn handling.

## Best Possible Solution

Use the freshest terminal assistant state at the incomplete-turn and payload/final-metadata boundaries, and add a regression for stale `lastAssistant=toolUse` plus fresh `currentAttemptAssistant=stop` after `update_plan`.

## Reproduction Assessment

Yes at source level, though not as a live WhatsApp/OpenRouter run in this review. A current-main attempt with `assistantTexts`, stale `lastAssistant.stopReason="toolUse"`, fresh `currentAttemptAssistant.stopReason="stop"`, and `payloadCount=1` still reaches the incomplete-turn error path.

## Solution Assessment

No current implementation fully solves it. The narrow maintainable fix is to select `currentAttemptAssistant ?? lastAssistant` for incomplete-turn detection, payload construction, and final assistant metadata while keeping true terminal tool-use turns protected.
