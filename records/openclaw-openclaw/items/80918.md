## Summary

Keep open. Current `main` and the latest shipped release still use `attempt.lastAssistant` for the incomplete-turn classifier, so the stale `toolUse` snapshot can still override a later completed `currentAttemptAssistant` final answer.

## What This Changes

The issue reports that embedded runs can silently drop a final text reply when a stale `lastAssistant.stopReason = "toolUse"` snapshot survives past a later `stop` assistant turn after `update_plan`.

## Best Possible Solution

Land a narrow embedded-runner fix that selects `currentAttemptAssistant ?? lastAssistant` for incomplete-turn classification, payload construction, and final assistant metadata, with a regression preserving the older terminal `toolUse` guard.

## Reproduction Assessment

Yes for source-level reproduction: construct an attempt with `lastAssistant.stopReason = "toolUse"`, `currentAttemptAssistant.stopReason = "stop"`, and non-empty `assistantTexts`; current `main` still classifies from stale `lastAssistant`. I did not run the full live WhatsApp path.

## Solution Assessment

No, current `main` is not the best solution because it only fixes text-only deliverability while leaving the stale classifier and incomplete-branch replacement path intact. The maintainable fix is to use the freshest terminal assistant state while keeping the real terminal `toolUse` incomplete-turn guard.
