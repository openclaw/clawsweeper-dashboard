## Summary

Keep open. Current main has fixed the text-only deliverability part, but the embedded-runner incomplete-turn path still keys the terminal tool-use guard off stale `lastAssistant` and still returns only the generic error payload when that guard fires, so the reported post-`update_plan` final-answer loss remains actionable.

## What This Changes

The issue reports that embedded runs can silently drop a post-`update_plan` final text turn when stale incomplete-turn state classifies the run as terminal `toolUse`.

## Best Possible Solution

Land a narrow embedded-runner fix that uses the freshest terminal assistant state for incomplete-turn classification and payload/final metadata, with a regression for stale `lastAssistant=toolUse` plus fresh `currentAttemptAssistant=stop` after `update_plan`, while preserving the true terminal `toolUse` guard from https://github.com/openclaw/openclaw/issues/76477.

## Reproduction Assessment

Yes for source-level reproduction: construct an attempt with nonempty `assistantTexts`, stale `lastAssistant.stopReason = "toolUse"`, and fresh `currentAttemptAssistant.stopReason = "stop"`; current main still classifies through `lastAssistant` and can enter the incomplete branch. I did not run the live WhatsApp/OpenRouter path.

## Solution Assessment

Yes, the best repair is the narrow direction already demonstrated by the closed unmerged PR: select `currentAttemptAssistant ?? lastAssistant` at the classifier and payload boundary, and add focused regression coverage without weakening the existing interrupted-tool-use protection.
