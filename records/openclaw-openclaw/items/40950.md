## Summary

Keep open. Current main still lacks a configurable inbound media store/sandbox cap, so the contribution is not obsolete, but the submitted branch is stale and incorrect as a merge candidate.

## What This Changes

The PR adds an optional top-level `media.maxBytes` config field plus media-store getter/setter plumbing and updates storage, sandbox staging, and config metadata surfaces toward a configurable inbound media byte limit.

## Best Possible Solution

Ship a refreshed current-main patch that defines one bounded effective inbound media store/sandbox limit, initializes it from config, uses it consistently in storage/staging/chat preflight, preserves lower caller caps, and updates generated metadata, docs, changelog, and focused tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: configure a sandboxed agent and send a non-image attachment larger than 5 MiB but below the chat attachment parse cap; current main rejects it against `MEDIA_MAX_BYTES` before any configurable limit can apply.

## Solution Assessment

No. The requested direction is plausible, but this PR is not the best mergeable solution because it leaves missing imports, runtime wiring, current chat preflight behavior, generated metadata, and cap precedence incomplete.
