## Summary

Keep open: current main still lacks the requested configurable media staging cap, but this PR head is conflicting, misses runtime imports/startup wiring, can weaken caller-specific caps, and lacks after-fix real gateway proof.

## What This Changes

The PR adds a `media.maxBytes` config field, mutable media-store limit helpers, and config/help metadata so inbound media store and sandbox staging limits can be raised above the 5 MiB default.

## Best Possible Solution

Land a rebased replacement for https://github.com/openclaw/openclaw/issues/40880 that defines a bounded `media.maxBytes` contract and threads it through media store, sandbox staging, chat preflight, help/docs, and regression tests while preserving stricter caller-specific caps.

## Reproduction Assessment

Yes, by source inspection. Current main still has fixed 5 MiB media-store, sandbox staging, and chat preflight caps, and gateway coverage asserts a 6 MiB sandboxed PDF is rejected.

## Solution Assessment

No. A bounded `media.maxBytes` setting is plausible, but this branch is stale/conflicting, misses imports/startup wiring, and does not preserve caller-specific cap precedence.
