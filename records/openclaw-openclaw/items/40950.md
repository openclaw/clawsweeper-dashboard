## Summary

Keep open. Current main still lacks a configurable `media.maxBytes` limit, but this PR is stale against current main and still has blocking compile/runtime, startup wiring, and media-cap contract issues before it can merge.

## What This Changes

The PR adds a `media.maxBytes` config field plus media-store getter/setter plumbing, then switches media storage, staging, and config metadata surfaces toward a configurable inbound media byte limit.

## Best Possible Solution

Ship a refreshed current-main patch that defines one bounded effective sandbox/store media limit, initializes it from config during gateway startup or reload, uses it consistently in media storage, sandbox staging, and chat preflight, preserves lower caller/channel caps, and updates generated config metadata plus focused tests/docs.

## Reproduction Assessment

Yes. Current main gives a high-confidence static reproduction path: configure a sandboxed agent and send a non-image attachment larger than 5 MiB but below the chat attachment cap; `chat.send` rejects it against `MEDIA_MAX_BYTES` before staging.

## Solution Assessment

No. The requested direction is plausible, but this submitted PR is not the best mergeable solution because it is stale, misses current startup/preflight paths, and breaks import/cap semantics; a refreshed narrower patch is safer.
