## Summary

Keep open: current main still hardcodes the 5 MiB inbound media store, sandbox staging, and chat preflight limits, so the underlying request is still real, but this PR head is conflicting and not merge-ready.

## What This Changes

The PR adds a top-level `media.maxBytes` config field, mutable media-store limit helpers, and updates media store, media route, sandbox staging, and config metadata paths.

## Best Possible Solution

Land a rebased replacement for https://github.com/openclaw/openclaw/issues/40880 that defines a bounded `media.maxBytes` contract and threads it through media store, sandbox staging, chat preflight, help/docs, and regression tests while preserving stricter caller-specific caps.

## Reproduction Assessment

Yes, by source inspection. Current main still has fixed 5 MiB store, sandbox staging, and chat preflight caps, and current gateway coverage asserts a 6 MiB sandboxed PDF is rejected.

## Solution Assessment

No. A bounded `media.maxBytes` setting is plausible, but this PR head misses imports/startup wiring, weakens caller-specific caps, conflicts with main, and lacks real gateway proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Import the media limit getter before staging media:** `src/auto-reply/reply/stage-sandbox-media.ts:81`
  - body: `stageSandboxMedia` now calls `getMediaMaxBytes()`, but this file still imports only `getMediaDir` and `MEDIA_MAX_BYTES`. Each staging attempt will hit an undeclared identifier instead of copying valid inbound media into the sandbox.
  - confidence: 0.95
- **[P1] Import the media limit getter before serving media:** `src/media/server.ts:54`
  - body: The media route passes `getMediaMaxBytes()` to `readFileWithinRoot`, but that symbol is not imported or re-exported here. Valid media reads will throw before the file can be served.
  - confidence: 0.93
- **[P2] Initialize the configured media limit on gateway boot:** `src/config/zod-schema.ts:517-522`
  - body: Adding `media.maxBytes` to the schema does not apply it by itself. The PR head has no startup call to `setMediaMaxBytes`, so the effective limit stays at the default and the new config option has no effect.
  - confidence: 0.92
- **[P2] Preserve stricter caller media caps:** `src/media/store.ts:253-254`
  - body: The download path checks the module-global limit directly, so raising `media.maxBytes` can bypass smaller per-channel or per-call limits that current callers pass into media-store operations.
  - confidence: 0.84
