## Summary

Keep open. Current main still hardcodes the 5 MiB media store, sandbox staging, and chat preflight limits, but this PR is not mergeable as-is because it has unresolved correctness blockers, widens caller media caps, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a top-level `media.maxBytes` config field and attempts to replace hardcoded media store, media route, and sandbox staging limits with a mutable effective media byte limit.

## Best Possible Solution

Ship a current-main repair for https://github.com/openclaw/openclaw/issues/40880 that adds a bounded `media.maxBytes` contract, wires startup and chat preflight, preserves stricter caller caps, updates config metadata/docs, adds regression coverage, and includes real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main shows a sandboxed non-image attachment above 5 MiB still follows the fixed store, staging, and chat preflight caps; I did not run a live gateway.

## Solution Assessment

No. The requested direction is plausible, but this PR is not the best mergeable fix until it imports the new getter, applies config at startup, aligns chat preflight, preserves caller caps, and provides real behavior proof.

  - confidence: 0.9

  - body: Current `chat.send` rejects sandboxed non-image offloads against `MEDIA_MAX_BYTES` before staging runs. Without updating that preflight, files above 5 MiB still fail even if staging later reads the new config.
  - confidence: 0.88

  - body: The remote download guard compares streamed bytes against the module-global media limit directly. If an operator raises `media.maxBytes`, callers that pass a smaller channel or per-call cap can persist payloads that should remain rejected.
  - confidence: 0.84
