## Summary

Keep open: current main still enforces the fixed 5 MiB media-store and sandbox-staging limit, but this external PR cannot merge as-is because the head is conflicting, lacks real behavior proof, and still has concrete runtime/config wiring defects.

## What This Changes

The PR adds a `media.maxBytes` config field plus media-store limit helpers, schema/help/type metadata, and swaps several media-store, staging, and media-route checks from the fixed 5 MiB constant to a configurable limit.

## Best Possible Solution

Land a rebased, maintainer-reviewed implementation for https://github.com/openclaw/openclaw/issues/40880 that defines a bounded `media.maxBytes` contract and threads the resolved limit through media store, sandbox staging, gateway preflight, help/docs, and regression tests while preserving stricter caller-specific caps.

## Reproduction Assessment

Yes, by source inspection. Current main still has fixed 5 MiB store, sandbox staging, and chat preflight caps, and current gateway coverage asserts a 6 MiB sandboxed PDF is rejected.

## Solution Assessment

No. A bounded `media.maxBytes` setting is plausible, but this branch is conflicting, lacks startup wiring, calls an unimported getter in changed runtime paths, and needs a clear resource-limit precedence decision.
