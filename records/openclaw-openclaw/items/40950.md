## Summary

Keep open: current main still has the fixed 5 MiB media-store, sandbox-staging, and chat preflight cap, but this PR head is conflicting, still has correctness blockers, and lacks after-fix real gateway proof.

## What This Changes

The PR adds `media.maxBytes` config metadata and a mutable media-store limit, then changes media serving and sandbox staging paths to call `getMediaMaxBytes()`.

## Best Possible Solution

Land a rebased replacement that defines a bounded `media.maxBytes` contract and threads it through media store, sandbox staging, chat preflight, docs/help, and regression tests while preserving stricter caller-specific caps.

## Reproduction Assessment

Yes, by source inspection. Current main still has fixed 5 MiB store, sandbox staging, and chat preflight caps, and current gateway coverage asserts a 6 MiB sandboxed PDF is rejected.

## Solution Assessment

No. A bounded `media.maxBytes` setting is a plausible direction, but this PR head misses imports, never initializes the setting at startup, weakens caller-specific caps, is conflicting, and lacks real gateway proof.

  - body: Adding `media.maxBytes` to the schema does not apply it by itself. The PR head has no startup call to `setMediaMaxBytes`, so the effective limit stays at the default and the new config option has no effect.
  - confidence: 0.92

  - body: The download path checks the module-global limit directly, so raising `media.maxBytes` can bypass smaller per-channel or per-call limits that current main passes into media-store operations.
  - confidence: 0.84
