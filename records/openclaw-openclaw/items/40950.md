## Summary

This PR still targets a real current-main configurability gap, but it is not mergeable as-is: the head is conflicting, still has compile/runtime breakages around unimported `getMediaMaxBytes`, does not initialize the configured limit on gateway boot, and weakens caller-specific media caps.

## What This Changes

The PR adds a `media.maxBytes` config field and tries to replace the fixed 5 MiB media store and sandbox staging cap with a mutable limit read from gateway config.

## Best Possible Solution

Ship a current-main repair for https://github.com/openclaw/openclaw/issues/40880 that defines a bounded `media.maxBytes` contract and threads it through store, sandbox staging, chat preflight, docs, and tests while preserving stricter caller or channel caps.

## Reproduction Assessment

Yes. Source inspection on current main shows the fixed 5 MiB store, sandbox staging, and chat preflight caps, and a current gateway test constructs a 6 MiB sandboxed PDF expecting rejection.

## Solution Assessment

No. The direction is plausible, but this PR head is conflicting, calls an unimported getter in two paths, never initializes the configured limit at startup, can weaken caller-specific caps, and lacks real gateway proof.
