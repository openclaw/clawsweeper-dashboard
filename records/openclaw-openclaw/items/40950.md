## Summary

Keep open. Current main still has the fixed 5 MiB media-store, sandbox-staging, and chat preflight cap, but this PR head is conflicting, still has concrete correctness defects, and lacks required after-fix real behavior proof.

## What This Changes

The PR proposes a top-level `media.maxBytes` config field, mutable media-store limit helpers, gateway startup wiring, config metadata, and getter-based media store/staging/serving limits.

## Best Possible Solution

Land a rebased replacement for https://github.com/openclaw/openclaw/issues/40880 that defines a bounded `media.maxBytes` contract and threads it through media store, sandbox staging, chat preflight, help/docs, and regression tests while preserving stricter caller-specific caps.

## Reproduction Assessment

Yes, by source inspection. Current main still has fixed 5 MiB store, sandbox staging, and chat preflight caps, and current gateway coverage asserts a 6 MiB sandboxed PDF is rejected.

## Solution Assessment

No. A bounded `media.maxBytes` setting is plausible, but this PR head misses imports/startup wiring, weakens or bypasses caller-specific caps, is conflicting against current main, and lacks real gateway proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Import the media limit getter before staging media:** `src/auto-reply/reply/stage-sandbox-media.ts:81`
  - body: `stageSandboxMedia` now calls `getMediaMaxBytes()`, but the module still imports only `getMediaDir` and `MEDIA_MAX_BYTES`. Each staging attempt will hit an undeclared identifier inside the `try`, then log and skip valid inbound media instead of copying it into the sandbox.
  - confidence: 0.95
- **[P1] Import the media limit getter before serving media:** `src/media/server.ts:54`
  - body: The `/media/:id` handler passes `getMediaMaxBytes()` to `readFileWithinRoot`, but that symbol is not imported. Valid media IDs will throw before the file can be served and fall into the generic error path.
  - confidence: 0.93
- **[P2] Initialize the configured media limit on gateway boot:** `src/config/zod-schema.ts:517-522`
  - body: Adding `media.maxBytes` to the schema does not apply it by itself. The PR head has no startup call to `setMediaMaxBytes`, so the effective limit stays at the default and the new config option has no effect.
  - confidence: 0.92
- **[P2] Preserve caller-specific media caps:** `src/media/store.ts:253-254`
  - body: The remote download path checks the module-global limit directly and does not carry a caller `maxBytes` through redirects, so raising the global setting can bypass smaller per-channel or per-call limits for persisted remote media.
  - confidence: 0.84
