## Summary

Keep open. The PR targets a real current-main configurability gap, but the submitted head is not mergeable as-is because it leaves blocking runtime/type issues, does not apply the new config at gateway startup, still leaves chat preflight hardcoded, and lacks real behavior proof.

## What This Changes

The PR adds a top-level `media.maxBytes` config field and tries to replace the hardcoded media store, media route, and sandbox staging limit with a mutable media byte limit.

## Best Possible Solution

Ship a current-main repair for https://github.com/openclaw/openclaw/issues/40880 that adds a bounded `media.maxBytes` contract, applies it at startup, threads it through store/staging/chat, preserves stricter caller caps, updates docs/tests, and includes real gateway proof.

## Reproduction Assessment

Yes. Source inspection on current main shows the media store, sandbox staging, and chat preflight still use the fixed `MEDIA_MAX_BYTES` path, and a current test asserts rejection of a 6 MiB sandbox PDF.

## Solution Assessment

No. The requested direction is plausible, but this PR is not the best mergeable fix until the getter imports, startup wiring, chat preflight, caller-cap precedence, and real behavior proof are handled against current main.

  - body: `stageSandboxMedia` now calls `getMediaMaxBytes()`, but this module does not import that symbol. In runtime, valid inbound files would hit a `ReferenceError` inside the staging loop and be skipped instead of copied into the sandbox.

  - confidence: 0.9

  - body: The patch adds a schema field and setter, but the PR head never calls `setMediaMaxBytes` during `startGatewayServer`. As submitted, accepting `media.maxBytes` in config does not change the effective limit.
  - confidence: 0.92
- **[P2] Align chat preflight with the configured staging cap:** `src/config/zod-schema.ts:517-522`
  - body: `chat.send` still rejects sandboxed non-image offloads against the fixed `MEDIA_MAX_BYTES` preflight before staging runs. Files above 5 MiB still fail even if staging later reads the new configured cap.
  - confidence: 0.88

  - body: `saveMediaBuffer` now defaults to the module-global limit, but callers that pass their own larger channel or per-call cap still bypass a lower `media.maxBytes`. If the new field is the documented store maximum, use the stricter effective cap.
  - confidence: 0.82
