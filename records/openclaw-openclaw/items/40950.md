## Summary

Keep open: current main still has the fixed 5 MiB media store, sandbox staging, and chat preflight limits, but this PR head is not mergeable as-is because the new config field is not wired at gateway startup, calls an unimported getter in two paths, and still needs real gateway proof plus resource-limit review.

## What This Changes

The PR adds a top-level `media.maxBytes` config field and attempts to thread a mutable media-store limit through config metadata, gateway startup, media storage, media serving, and sandbox staging.

## Best Possible Solution

Ship a current-main repair for https://github.com/openclaw/openclaw/issues/40880 that defines a bounded `media.maxBytes` contract and threads it through store, sandbox staging, chat preflight, docs, and tests while preserving stricter caller or channel caps.

## Reproduction Assessment

Yes. Source inspection on current main shows the fixed 5 MiB store, sandbox staging, and chat preflight caps, and current test coverage constructs a 6 MiB sandboxed PDF expecting rejection; I did not run tests in this read-only review.

## Solution Assessment

No. The direction is plausible, but this PR head does not wire the config at startup, calls an unimported getter in two paths, can weaken stricter caller caps, and needs current-main rebase plus real gateway proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Import the getter before staging media:** `src/auto-reply/reply/stage-sandbox-media.ts:81`
  - body: `stageSandboxMedia` now calls `getMediaMaxBytes()` but this file still imports only `getMediaDir` and `MEDIA_MAX_BYTES` from the media store. The branch will fail to build; if emitted, staging would throw before copying inbound media.
  - confidence: 0.94
- **[P1] Import the getter before serving media:** `src/media/server.ts:54`
  - body: The media route now passes `getMediaMaxBytes()` to `readFileWithinRoot`, but that symbol is not imported from `server.runtime.js`. This breaks the route at compile time and would make valid media reads fail at runtime.
  - confidence: 0.9
- **[P2] Initialize the configured limit on gateway boot:** `src/config/zod-schema.ts:517-522`
  - body: Adding `media.maxBytes` to the schema is not enough: the PR head never calls `setMediaMaxBytes` from `startGatewayServer`, so the new config knob remains at the default and does not fix files above 5 MiB.
  - confidence: 0.91
- **[P2] Preserve stricter caller media caps:** `src/media/store.ts:253-254`
  - body: The remote download guard now checks the module-global `mediaMaxBytes` instead of a caller-provided cap. Once the global cap is raised, channel or per-call paths that intentionally enforce a smaller limit can persist larger media than they allowed before.
  - confidence: 0.84
