## Summary

Keep open. The underlying media-size configurability request is still valid on current main, but this PR is not mergeable as-is: it has unresolved import/startup/preflight defects, is conflicting with current main, and lacks required real behavior proof.

## What This Changes

The PR adds an optional `media.maxBytes` config field and a mutable media-store limit, then switches some store, staging, and media-route limit checks from the fixed 5 MiB constant to the new getter.

## Best Possible Solution

Ship a current-main repair for https://github.com/openclaw/openclaw/issues/40880 that adds a bounded `media.maxBytes` contract, wires startup/store/staging/chat preflight, preserves stricter explicit caps, updates config docs and tests, and includes real gateway proof.

## Reproduction Assessment

Yes. Source inspection on current main shows the store, sandbox staging, and chat preflight still use the fixed 5 MiB cap, and current tests assert rejection for a 6 MiB sandboxed non-image attachment; I did not run a live gateway in this read-only review.

## Solution Assessment

No. The requested direction is plausible, but this PR is not the best mergeable fix until it is rebased onto current main, wires the configured value through current startup and chat paths, preserves caller caps, and provides real behavior proof.

Full review comments:

- **[P1] Import the media limit getter in sandbox staging:** `src/auto-reply/reply/stage-sandbox-media.ts:81`
  - body: `stageSandboxMedia` now calls `getMediaMaxBytes()`, but this module still imports only `getMediaDir` and `MEDIA_MAX_BYTES`. The changed staging path will fail typecheck or throw at runtime before valid inbound files can be copied.
  - confidence: 0.96
- **[P1] Import the media limit getter in the media route:** `src/media/server.ts:54`
  - body: The `/media/:id` handler calls `getMediaMaxBytes()` without importing it from the runtime module. When this handler runs, valid media reads can fail instead of being served.
  - confidence: 0.9
- **[P2] Apply media.maxBytes during gateway startup:** `src/config/zod-schema.ts:517-522`
  - body: The patch adds the schema field and setter, but the gateway startup path never calls `setMediaMaxBytes`, so configuring `media.maxBytes` leaves the effective store and staging limit at the 5 MiB default.
  - confidence: 0.91
- **[P2] Align chat preflight with the configured staging cap:** `src/gateway/server-methods/chat.ts:927-934`
  - body: Current main rejects sandboxed non-image offloads against `MEDIA_MAX_BYTES` before staging runs. Without updating that preflight to the resolved configured cap, files above 5 MiB still fail even if staging later uses `media.maxBytes`.
  - confidence: 0.88
- **[P2] Enforce media.maxBytes as the global ceiling:** `src/media/store.ts:427-430`
  - body: `saveMediaBuffer` only uses `mediaMaxBytes` as the default when callers omit `maxBytes`. Existing channel or tool paths that pass a higher explicit cap can still persist payloads above the operator's configured global maximum.
  - confidence: 0.82
