Current main still lacks a configurable inbound media store/sandbox cap, so the contribution is not obsolete, but the submitted branch is stale and incorrect as a merge candidate: it misses imports, does not apply the new config at startup, does not update the current chat preflight path, and can bypass stricter caller/channel caps.

## What This Changes

The PR adds an optional top-level `media.maxBytes` config field, media-store getter/setter plumbing, and updates media storage, sandbox staging, and config metadata surfaces toward a configurable inbound media byte limit.

## Best Possible Solution

Ship a refreshed current-main patch that defines one bounded effective inbound media store/sandbox limit, initializes it from config during gateway startup or reload, uses it consistently in media storage, sandbox staging, and chat preflight, preserves lower caller/channel caps, and updates generated config metadata, docs, changelog, and focused tests.

Yes. Current main gives a high-confidence static reproduction path: configure a sandboxed agent and send a non-image attachment larger than 5 MiB but below the chat attachment parse cap; the code rejects it against the fixed `MEDIA_MAX_BYTES` path before any configurable limit can apply.
