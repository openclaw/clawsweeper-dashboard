## Summary

Keep open: current main still has the hardcoded 5 MiB media store/sandbox cap, but this PR is stale and incorrect because the final head calls new getters without imports and never wires the new config into gateway startup.

## What This Changes

The PR adds a `media.maxBytes` config field and replaces some hardcoded inbound media store/staging limits with mutable getter/setter accessors.

## Best Possible Solution

Ship a current-main replacement that adds one bounded inbound media store/sandbox limit, applies it during startup/reload, aligns chat preflight, preserves stricter caller caps, updates config metadata/docs/changelog, and adds regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main still rejects sandboxed media above `MEDIA_MAX_BYTES` in the store, staging, and chat preflight paths; the PR head defects are also reproducible from the raw diff without a live gateway run.

## Solution Assessment

No. The requested direction is reasonable, but this PR is not the best mergeable fix because it misses imports/startup wiring, is stale against current main, and does not preserve all cap contracts.

  - confidence: 0.92

  - body: The changed remote download path checks only the module-global `mediaMaxBytes`. Any caller or channel that passed a smaller cap loses that stricter guard, so raising the global setting can allow larger persisted payloads than that caller intended.

  - confidence: 0.81
