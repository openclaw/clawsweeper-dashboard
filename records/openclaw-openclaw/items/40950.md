## Summary

Keep open. The user problem still exists on current main, and this PR is the active closing implementation path, but the current PR head has blocking runtime/type errors, does not initialize the new setting, can weaken narrower media caps, and lacks real behavior proof.

## What This Changes

The PR adds a top-level `media.maxBytes` config field and rewires media store, media serving, and sandbox staging paths to use a mutable media size limit.

## Best Possible Solution

Ship a rebased current-main repair for https://github.com/openclaw/openclaw/issues/40880 that defines a bounded `media.maxBytes` contract and threads it through media store, sandbox staging, chat preflight, docs/help, and regression tests while preserving stricter caller or channel caps.

## Reproduction Assessment

Yes, by source inspection. Current main still has fixed 5 MiB store, sandbox staging, and chat preflight caps, and the current gateway test constructs a 6 MiB PDF expecting sandbox-limit rejection.

## Solution Assessment

No. A bounded `media.maxBytes` setting is a plausible direction, but this PR head misses required imports, never initializes the setting at startup, weakens caller-specific caps, and lacks real gateway proof.
