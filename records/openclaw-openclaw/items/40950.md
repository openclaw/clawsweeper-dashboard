## Summary

Keep open. Current main still has the fixed 5 MiB media store/sandbox cap and the linked media-limit issue remains open, but this PR is not mergeable as submitted because it is stale against current main and leaves import, startup wiring, chat preflight, caller-cap precedence, generated metadata, and changelog gaps.

## What This Changes

The PR adds an optional top-level `media.maxBytes` setting and media-store getter/setter plumbing, then updates media storage, sandbox staging, config help, labels, and types toward a configurable inbound media limit.

## Best Possible Solution

Land a refreshed current-main implementation that defines one bounded effective inbound media store/sandbox limit, initializes it from config, uses it in storage, staging, and chat preflight, and never overrides stricter caller or channel caps.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a sandboxed agent and send a non-image attachment above 5 MiB but below the chat attachment parse cap; current main rejects it against `MEDIA_MAX_BYTES` before any configurable limit exists.

## Solution Assessment

No. The requested direction is plausible, but the submitted diff is incomplete and stale; the safer solution is a current-main patch that wires the config through all effective limit checks and preserves stricter caps.
