## Summary

Keep open. The underlying configurability request is still valid on current main, but this PR is stale/conflicting, lacks real end-to-end behavior proof, and still leaves correctness/security gaps around chat preflight and caller-specific media caps.

## What This Changes

The PR adds a `media.maxBytes` config field and mutable media-store limit, then updates config metadata plus store, staging, and media-route code to use that limit.

## Best Possible Solution

Ship a current-main repair for https://github.com/openclaw/openclaw/issues/40880 that adds a bounded `media.maxBytes` contract, threads it through store, sandbox staging, and chat preflight, preserves stricter caller caps, updates docs/tests, and includes real behavior proof.

## Reproduction Assessment

Yes. Source inspection on current main shows sandboxed non-image attachments above 5 MiB hit the fixed `MEDIA_MAX_BYTES` chat preflight and staging cap, and the current gateway test asserts that rejection path with a 6 MiB PDF; I did not run a live gateway.

## Solution Assessment

No. The requested direction is plausible, but this PR is not the best mergeable fix until it rebases onto current main, updates chat preflight, preserves caller-specific caps, and adds real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Wire chat preflight to the configured sandbox cap:** `src/auto-reply/reply/stage-sandbox-media.ts:82`
  - body: Changing `stageSandboxMedia` to read `getMediaMaxBytes()` is not enough because current `chat.send` rejects sandboxed non-image offloads above `MEDIA_MAX_BYTES` before staging runs. With this PR, a 10 MiB PDF can still fail before the new configured staging limit is used.
  - confidence: 0.89
- **[P2] Preserve caller-specific media size limits:** `src/media/store.ts:250`
  - body: The store download path now checks the module-global `mediaMaxBytes` directly. If an operator raises `media.maxBytes`, callers that pass a stricter per-channel or per-call cap can persist payloads above their intended limit, increasing disk exposure and breaking existing limit contracts.
  - confidence: 0.86
