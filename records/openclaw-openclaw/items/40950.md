## Summary

Keep this PR open, but it is not merge-ready: current main and the latest release still have the fixed 5 MiB sandbox/media-store cap, while the PR head is stale, lacks real behavior proof, and still has blocking runtime/config wiring defects.

## What This Changes

The PR proposes a top-level `media.maxBytes` config field and media-store helpers so media store, sandbox staging, and media serving limits can be raised above the 5 MiB default.

## Best Possible Solution

Land a rebased, maintainer-reviewed implementation for https://github.com/openclaw/openclaw/issues/40880 that defines a bounded `media.maxBytes` contract, threads the resolved effective limit through current store, sandbox staging, gateway preflight, help/docs, and regression tests, and preserves stricter existing caller-specific caps.

## Reproduction Assessment

Yes. Source inspection on current main shows fixed 5 MiB store, sandbox staging, and chat preflight caps, and current gateway coverage asserts a 6 MiB sandboxed PDF is rejected.

## Solution Assessment

No. A bounded config-backed media staging limit is plausible, but this PR head is stale, lacks startup wiring, calls unimported getters, and needs a clear resource-limit precedence decision before it is the maintainable fix.
