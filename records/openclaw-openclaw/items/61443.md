## Summary

Keep open. Current main does not implement provider-level retry config, and this PR is a plausible active implementation, but it still needs maintainer scope review, a rebase, and contributor-supplied real behavior proof before merge.

## What This Changes

The PR adds opt-in `models.providers.<name>.retry` config, provider retry runner plumbing, abort-aware retry support, OpenAI-compatible transport wrapping, generated config metadata, tests, and changelog text.

## Best Possible Solution

Choose one canonical LLM retry contract, scope the config to transports that honor it or wire all supported provider families, preserve timeout/abort/failover semantics, update public docs/schema/tests, and require redacted real behavior proof before merge.

## Reproduction Assessment

Yes for the review blocker: source inspection of the PR head shows a generic provider retry config while only OpenAI-compatible stream factories consume the retry runner. Real provider retry behavior is not reproduced because no after-fix runtime proof is present.

## Solution Assessment

No. Provider retry is a reasonable feature direction, but this implementation should not merge until the config scope is enforced or wired across supported transports, the branch is rebased, and real behavior proof is supplied.
