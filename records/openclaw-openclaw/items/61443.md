## Summary

Keep this PR open: current main still has no provider-level retry config, so the contribution is not obsolete, but the branch is not merge-ready. The current PR head is dirty against main, overlaps the still-open #38640 retry implementation, and still exposes generic provider retry config while only wiring OpenAI-compatible transports; it also needs a rebase that preserves current main's timeout-aware OpenAI SDK request options and current config schema fields.

## What This Changes

The branch adds opt-in `models.providers.<name>.retry` config, provider retry policy/registry code, abort-aware retry support, OpenAI-compatible transport wrapping, generated config metadata, and focused retry/test fixture updates.

## Best Possible Solution

Land one canonical provider-retry design after maintainers choose between this PR and #38640. The final version should be opt-in, either genuinely cover every provider API that accepts the config or validate/document a narrower OpenAI-compatible scope, preserve current timeout/request-option and config-schema behavior from main, regenerate public config metadata, and include focused tests for config reload, transport wiring, abort handling, Retry-After handling, and unsupported-provider behavior.

## Reproduction Assessment

Not applicable as a normal user bug reproduction; this is a feature PR review. The blocking behavior is statically reproducible by comparing PR head `86bed8e9230eead8859d6ddfee204d6368a2d4f3` with current main: generic retry config is accepted while only OpenAI-compatible transports consult the retry registry, and the PR helper is signal-only while main's helper is timeout-aware.

## Solution Assessment

No, not yet. The approach is plausible, but the current branch is not the best merge shape until it rebases cleanly, preserves current main's timeout/config contracts, and resolves whether provider retry is generic across LLM transports or explicitly scoped to OpenAI-compatible providers.
