## Summary

Keep this PR open, but the current branch is not merge-ready. It adds a useful provider-retry direction, yet the public config is generic while only OpenAI-compatible transports use the runner, and the branch is stale against current main's provider schema and transport timeout work.

## What This Changes

The PR adds opt-in `models.providers.<name>.retry` config, provider retry policy/registry code, abort-aware retry support, OpenAI-compatible transport wrapping, generated config metadata, and focused tests.

## Best Possible Solution

Land one canonical provider-retry design that is either wired through all supported LLM transports or explicitly scoped and validated as OpenAI-compatible-only, then rebase on current main, preserve timeout/schema behavior, add changelog/docs, and cover reload plus unsupported-provider behavior in tests.

## Reproduction Assessment

Yes for the blocking review issue: static inspection is enough to reproduce it by configuring `models.providers.anthropic.retry` and tracing transport creation, where Anthropic uses `createAnthropicMessagesTransportStreamFn()` and never consults the PR's provider retry registry.

## Solution Assessment

No. The approach is plausible, but accepting generic provider retry config while only applying it to OpenAI-compatible stream creation is not the narrowest maintainable solution for the Anthropic/provider retry reports this PR references.
