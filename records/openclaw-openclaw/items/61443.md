## Summary

Keep open: the feature is still useful and current main does not provide this configurable LLM-provider retry surface, but this PR needs human review because it exposes a general provider config while only OpenAI-compatible transports consume it, carries compatibility/availability risk, is stale against current transport work, and lacks real behavior proof.

## What This Changes

The PR adds an opt-in `models.providers.<name>.retry` config, a provider retry runner registry, OpenAI-compatible stream retry wrapping, retry policy tests, generated config metadata, and test fixture updates.

## Best Possible Solution

Carry retry as a resolved runtime model/provider request policy and either implement/test the supported provider families or narrow the schema/docs to OpenAI-compatible providers, then rebase on current main and add real behavior proof.

## Reproduction Assessment

Yes for the review finding: source inspection shows the schema accepts `retry` for every model provider while only the OpenAI-compatible transport creates and consumes provider retry runners. No live failure reproduction was run in this read-only review.

## Solution Assessment

No. The PR is a plausible implementation for OpenAI-compatible providers, but the best fix needs a maintainer-approved provider retry contract that matches the public config surface and current runtime owner boundaries.
