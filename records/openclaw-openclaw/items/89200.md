## Summary

Keep this issue open: current main still removes the session store entry and archives the primary transcript on `sessions.delete`, but it does not call the existing trajectory sidecar cleanup used by prune/cap maintenance.

## What This Changes

Deleting a session should also remove that session's `.trajectory.jsonl` runtime file and `.trajectory-path.json` pointer file.

## Best Possible Solution

Reuse the existing trajectory cleanup helper from the delete path, preserving referenced surviving sessions and existing plugin lifecycle behavior.

## Reproduction Assessment

Yes by source inspection: `sessions.delete` archives the transcript only, while the trajectory cleanup helper is called from prune/cap maintenance but not from the delete RPC. I did not run a filesystem repro because this review is read-only.

## Solution Assessment

Yes: the narrowest maintainable fix is to reuse `removeRemovedSessionTrajectoryArtifacts` or an equivalent existing helper after deletion, with tests proving sidecars are removed only when the session id is no longer referenced.
