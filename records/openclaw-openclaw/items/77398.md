## Summary

Keep open. Current main still leaves normal Discord REST calls on the process/global fetch path and opens the Discord gateway WebSocket without a Discord-owned IPv4-preferred lookup or fallback, so the reported `NODE_OPTIONS=--dns-result-order=ipv4first` workaround remains a plausible current-main regression path.

## What This Changes

The issue requests a fix for Discord inbound startup failures on IPv4-only networks after the 2026.5.x networking changes.

## Best Possible Solution

Add a Discord plugin-owned transport policy that gives Discord REST and gateway WebSocket startup an IPv4-preferred or IPv4-fallback path without setting process-wide launcher `NODE_OPTIONS`.

## Reproduction Assessment

Yes. Source inspection plus the issue's two-machine workaround gives a high-confidence reproduction path, though I did not establish it live on an affected IPv4-only Windows network.

## Solution Assessment

No, the process-wide `NODE_OPTIONS` workaround is useful proof but not the best repository fix. The maintainable fix should live in the Discord plugin transport layer and preserve the existing proxy and SSRF boundaries.
