## Summary

Keep open: current main has related Discord startup hardening, but the Discord REST and gateway WebSocket paths still do not apply a Discord-owned IPv4-first/DNS-family policy, so the reported IPv4-only network regression remains plausible and actionable.

## What This Changes

Restore Discord inbound connectivity on IPv4-only/no-AAAA networks in 2026.5.x without requiring users to set `NODE_OPTIONS=--dns-result-order=ipv4first`.

## Best Possible Solution

Discord should own an IPv4-safe transport policy across REST metadata, RequestClient calls, and the `ws` gateway socket so inbound works on IPv4-only networks without process-wide `NODE_OPTIONS`.

## Reproduction Assessment

No live current-main reproduction was established, because this needs the affected Windows/network DNS environment. Source inspection plus the reporter's concrete two-machine workaround makes the failure path clear enough to keep open and queue a focused fix.

## Solution Assessment

No. The launcher-level `NODE_OPTIONS` workaround is useful for users, but a Discord-owned transport fix is the narrower maintainable solution because it can cover REST, metadata, and `ws` while preserving unrelated process traffic.
