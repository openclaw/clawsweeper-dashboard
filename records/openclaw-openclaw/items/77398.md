## Summary

Keep open. Current main still lacks a Discord-owned IPv4/DNS fallback on the default REST and gateway WebSocket paths, and an open unmerged PR already targets the issue with closing syntax, so the issue should stay open until that fix lands or is replaced.

## What This Changes

The issue reports a 2026.5.x Discord inbound regression where IPv4-only networks never reach gateway READY unless the gateway is launched with IPv4-first DNS ordering.

## Best Possible Solution

Land or replace PR #77449 with a Discord plugin-owned REST and gateway DNS policy that prefers or falls back to IPv4 for Discord hosts, plus targeted regression tests and a changelog entry.

## Reproduction Assessment

Yes. The issue gives a concrete two-machine Windows IPv4-only-network path and current main still leaves default Discord REST and gateway WebSocket creation without a lookup/family fallback; I did not perform the live network reproduction.

## Solution Assessment

Yes. A Discord plugin-owned transport fix is narrower and safer than setting process-wide `NODE_OPTIONS`, and the linked open PR is aimed at that boundary but still needs review/merge or replacement.
