## Summary

Keep open. Current main still lacks a Discord-owned DNS/family fallback on the default REST and gateway WebSocket paths, while the docs describe Discord as ready for inbound gateway use and the issue includes a two-machine workaround that points directly at DNS result ordering.

## What This Changes

The issue asks OpenClaw to restore Discord inbound on IPv4-only networks by handling Discord REST and WebSocket DNS family selection without requiring a process-wide env workaround.

## Best Possible Solution

Add a Discord plugin-owned transport policy for REST and gateway WebSocket startup that prefers or falls back to IPv4 for Discord hosts, with targeted regression coverage and a changelog entry, without setting process-wide NODE_OPTIONS from the launcher.

## Reproduction Assessment

Yes. Source inspection on current main plus the issue’s two-machine workaround gives a high-confidence reproduction path, although I did not establish it live on an affected Windows IPv4-only network.

## Solution Assessment

No. The NODE_OPTIONS workaround is useful proof, but the maintainable repository fix should be local to the Discord plugin transport paths rather than a process-wide launcher default.
