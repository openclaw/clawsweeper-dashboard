## Summary

Keep open. Current main still leaves key Discord REST paths on the process/global fetch path and opens the gateway socket without a Discord-specific DNS order or IPv4 fallback, so the reported `NODE_OPTIONS=--dns-result-order=ipv4first` workaround matches a plausible current source-level regression.

## What This Changes

Restore Discord inbound connectivity on IPv4-only networks after 2026.5.x without requiring users to set a process-wide DNS result order workaround.

## Best Possible Solution

Add a Discord plugin-owned transport policy for REST and gateway WebSocket startup that prefers or falls back to IPv4 for Discord hosts, with targeted regression tests and a changelog entry, without setting process-wide `NODE_OPTIONS` from the launcher.

## Reproduction Assessment

Yes, source inspection plus the issue’s two-machine workaround gives a high-confidence reproduction path, but I did not establish it live on an affected IPv4-only Windows network. The current default REST and WebSocket paths lack a Discord-owned DNS/family fallback, so this is source-reproducible on current main.

## Solution Assessment

No, the process-wide `NODE_OPTIONS` workaround is useful proof but not the best repository fix. The maintainable fix should live in the Discord plugin transport layer, following the existing Telegram-style owner-local networking pattern where appropriate.
