## Summary

Keep open. Current main still routes normal agent-scoped `workspaceAccess: "rw"` sandboxes around the skill-sync path, and this PR is not safe to merge because it both imports a non-existent export and leaves the intended sync block unreachable for the reported configuration.

## What This Changes

The PR removes the inner `workspaceAccess !== "rw"` guard around sandbox skill sync, passes managed and bundled skill directories to the sync call, and changes the Overview language selector to prefer persisted settings.

## Best Possible Solution

Revise or replace this with a focused sandbox fix that prepares sandbox-readable copies of eligible non-workspace skills for rw sessions, rewrites prompt-visible skill locations to those readable copies, preserves current filtering and filesystem confinement, and adds a targeted rw-mode regression test. Leave the Overview locale change out of the sandbox PR.

## Reproduction Assessment

Yes. The issue has a high-confidence static reproduction path: configure an agent-scoped sandbox with `workspaceAccess: "rw"`, then trace `resolveSandboxContext()` through `runEmbeddedAttempt()` to see skill loading use the host workspace while non-workspace skill paths remain outside the sandbox filesystem mounts.

## Solution Assessment

No. This PR is not the best way to solve the issue because the import breaks module loading and the sync remains inside the branch skipped by normal rw sandboxes; the safer direction is an explicit rw skill-mirror path with prompt-location rewriting and regression coverage.
