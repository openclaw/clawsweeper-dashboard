## Summary

Keep open: this PR is not mergeable as-is because it turns a service-specific Docker bootstrap into the default image path and introduces build, runtime, and supply-chain regressions against the current generic Docker contract.

## What This Changes

The PR modifies the Docker image to install a Python skill venv, copy a baked team workspace and config templates, and run a custom entrypoint that seeds OpenClaw config from environment variables.

## Best Possible Solution

Keep the official Dockerfile generic, and move team-specific workspace/config seeding into an opt-in derived image, Compose overlay, or external deployment script with pinned dependencies and auditable source content.

## Reproduction Assessment

Yes for the PR defects from source: the default Docker build reaches an unconditional venv/pip step while current build args are empty, and every container command would execute the entrypoint's required env checks before the gateway command. I did not run Docker because this was a read-only review.

## Solution Assessment

No: hard-coding Telegram, Google, and a team workspace into the official image is not the narrow maintainable solution. The safer path is an opt-in deployment wrapper or derivative Dockerfile that preserves the generic image contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Keep the default Docker build self-contained:** `Dockerfile:260-262`
  - body: The new venv layer runs unconditionally, but the base runtime only installs `python3` and leaves `OPENCLAW_DOCKER_APT_PACKAGES` empty by default. The patch comment says python3-venv/pip/gettext-base must be passed in, so the documented default Docker build path now fails before the image is produced.
  - confidence: 0.91
- **[P1] Keep the entrypoint compatible with existing Docker setup:** `scripts/openclaw-entrypoint.sh:15-18`
  - body: This entrypoint exits unless TELEGRAM_BOT_TOKEN, LLM_API_KEY, USER_TIMEZONE, and USER_NAME are set, and Dockerfile installs it as the global entrypoint. Existing Compose setup and docs make channel/provider setup optional, so gateway startup, pre-start onboarding, and CLI helper containers can fail before OpenClaw runs.
  - confidence: 0.94
- **[P2] Remove the unresolved workspace gitlink:** `openclaw-team-workspace:1`
  - body: The PR adds `openclaw-team-workspace` as a gitlink but does not add a `.gitmodules` URL, while the Dockerfile copies that path into the image. Standard clones cannot populate or audit that tree, so the build depends on opaque workspace content and may copy an empty or missing directory.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff introduces concrete supply-chain and privacy-sensitive risks in the release Docker image path.

Concerns:

- **[medium] Unpinned PyPI install in Docker build:** `Dockerfile:261`
  - body: `pip install websockets` resolves code from PyPI at image build time outside the repo lockfile and without a pinned version or hash, making official image builds less reproducible and more exposed to upstream compromise.
  - confidence: 0.9
- **[medium] Opaque workspace gitlink copied into image:** `openclaw-team-workspace:1`
  - body: The proposed image copies a gitlink-backed workspace with no `.gitmodules` URL in the patch; if populated externally, arbitrary agent instructions or skills would be baked into the official image without normal source review.
  - confidence: 0.86
- **[low] Entrypoint prints generated USER.md content:** `scripts/openclaw-entrypoint.sh:74`
  - body: The new startup script prints the generated USER.md to container logs, which can expose user identity/profile fields in deployment logs and support bundles.
  - confidence: 0.78
