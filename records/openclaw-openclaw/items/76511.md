## Summary

Keep open for maintainer handling: the PR is not merge-ready because it turns the official Docker image into a team-specific bootstrap, breaks existing default Docker assumptions from source inspection, and adds supply-chain/privacy risks that need product/security direction.

## What This Changes

The PR adds a custom Docker bootstrap with a Python skill venv, baked team workspace, config templates, and an entrypoint that seeds OpenClaw state from environment variables.

## Best Possible Solution

Keep the official Docker image generic, and put this team bootstrap in a derived image, Compose overlay, or plugin/bundle with pinned dependencies and auditable workspace content.

## Reproduction Assessment

Yes for the PR defects from source: the default build reaches an unconditional venv/pip step while current build args are empty, and gateway startup would run required custom env checks before OpenClaw starts. I did not run Docker because this review is read-only.

## Solution Assessment

No: hard-coding Telegram, Google, a team workspace, and first-run config seeding into the official image is not the narrow maintainable path. A derived deployment wrapper or bundle-style plugin keeps core generic and security reviewable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Keep the default Docker build self-contained:** `Dockerfile:260-262`
  - body: The new venv layer runs unconditionally, but the runtime image only installs `python3` by default and `OPENCLAW_DOCKER_APT_PACKAGES` remains empty. The documented default Docker build can therefore fail before producing an image unless callers know to pass extra packages that are only mentioned in the new comment.
  - confidence: 0.91
- **[P1] Keep the entrypoint compatible with existing Docker setup:** `scripts/openclaw-entrypoint.sh:15-18`
  - body: The global entrypoint exits unless TELEGRAM_BOT_TOKEN, LLM_API_KEY, USER_TIMEZONE, and USER_NAME are set. Existing Compose/docs make channel and provider setup optional, so normal gateway startup can fail before OpenClaw runs.
  - confidence: 0.94
- **[P2] Use the persisted Docker workspace mount:** `scripts/openclaw-entrypoint.sh:5`
  - body: The entrypoint seeds `/home/node/workspace`, but current Compose persists `/home/node/.openclaw/workspace`. Agent workspace changes would bypass the configured host mount and be lost on container replacement.
  - confidence: 0.86
- **[P2] Remove the unresolved workspace gitlink:** `openclaw-team-workspace:1`
  - body: The PR adds `openclaw-team-workspace` as a gitlink without a `.gitmodules` URL, while the Dockerfile copies it into the image. Standard clones cannot populate or audit that tree, so builds depend on opaque content.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff introduces concrete supply-chain and privacy-sensitive concerns in the official Docker image path.

Concerns:

- **[medium] Unpinned PyPI install in Docker build:** `Dockerfile:261`
  - body: `pip install websockets` resolves executable code from PyPI at image build time outside the repo lockfile and without a pinned version or hash, weakening reproducibility for official image builds.
  - confidence: 0.9
- **[medium] Opaque workspace gitlink copied into image:** `openclaw-team-workspace:1`
  - body: The proposed image copies a gitlink-backed workspace with no `.gitmodules` URL in the patch; if populated externally, arbitrary agent instructions or skills could be baked into the official image without normal source review.
  - confidence: 0.86
- **[low] Generated user profile is printed to logs:** `scripts/openclaw-entrypoint.sh:74`
  - body: The entrypoint prints the generated USER.md content to container logs, which can expose user identity/profile fields in deployment logs and support bundles.
  - confidence: 0.78
