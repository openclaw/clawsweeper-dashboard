## Summary

Keep open: current main and v2026.6.6 still expose only `ok`, `error`, and `skipped` cron run statuses, and unresolved delivery targets still become `error`; the linked implementation PR is open but conflicting, so the requested status split remains unmerged.

## What This Changes

The issue requests a cron run status split so successful execution with failed delivery surfaces as a warning instead of an error.

## Best Possible Solution

Review, rebase, and land or replace https://github.com/openclaw/openclaw/pull/80139 for the additive warning-status contract, while keeping the narrower delivery-channel propagation bug in https://github.com/openclaw/openclaw/issues/92460 separate.

## Reproduction Assessment

Yes. Source inspection on current main shows no `warning` cron run status, delivery-target failures still return `error`, and `error` still drives scheduler failure counters; I did not run a live cron job because this review is read-only.

## Solution Assessment

Yes as a product direction, but it is not implemented on main yet. The narrow maintainable path is an additive warning-state contract across core storage/API/UI that preserves true execution failures as errors and keeps delivery diagnostics separate.
