## Summary

Keep open: current main and v2026.6.10 still expose cron run status as `ok | error | skipped`, while the linked broad warning-status PR remains open and conflicting. This issue is still the canonical product/compatibility decision thread for whether cron should add a first-class partial-success warning state or choose a narrower delivery-status model.

## What This Changes

Request a cron execution-versus-delivery status split so successful runs with failed result delivery can surface as a warning instead of an execution error.

## Best Possible Solution

Keep this issue as the canonical decision thread until maintainers either land a retargeted first-class `warning` status across cron/API/UI surfaces or explicitly choose a narrower delivery-status-only model.

## Reproduction Assessment

Yes. Current main is source-reproducible: cron status is only `ok | error | skipped`, unresolved delivery failures can still return `status: "error"`, and scheduler persistence consumes that status as the primary run outcome.

## Solution Assessment

Unclear as final product direction. An execution/delivery split is the right shape for the reported problem, but a first-class `warning` status changes a public cron/Gateway/UI/plugin contract and should land only through the linked reviewed PR or an explicit maintainer-approved alternative.
