- `impact:message-loss`: The affected workflows involve cron completion delivery failure or unclear `not-delivered` state after successful execution.

## Summary

Keep open: current main and v2026.6.10 still expose cron run status as `ok | error | skipped`, while an open linked PR is the broad implementation candidate for the requested `warning` status and remains unmerged.

## What This Changes

The issue asks cron to separate task execution success from post-execution delivery failure so Control UI and run history can show a warning instead of an execution error.

## Best Possible Solution

Keep this issue as the canonical decision thread until maintainers either land a retargeted `warning` status implementation or explicitly choose a narrower delivery-status-only model.

## Reproduction Assessment

Yes, source-reproducible: current main has no `warning` cron status, unresolved delivery failures can still produce `status: "error"`, and the scheduler persists that status as the primary run outcome.

## Solution Assessment

Unclear as final product direction. An execution/delivery split is the right shape, but a first-class `warning` status changes public cron, Gateway, UI, and plugin-facing contracts and should land through the linked reviewed PR or an explicit maintainer-approved alternative.
