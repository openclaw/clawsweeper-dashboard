## Label Justifications

- `P2`: This is a medium-impact cron/status correctness improvement with limited blast radius but real operator confusion and alert fatigue.
- `impact:message-loss`: The issue centers on final announce delivery failures and `not-delivered` state being collapsed into the primary cron error status.

## Summary

Keep open: current main and the latest release still expose cron run status as `ok | error | skipped`, and delivery-target/direct delivery failures can still flow through `status: "error"`. The open linked implementation PR is the active candidate, but it is unmerged and the new `warning` status is a compatibility-sensitive public cron contract.

## What This Changes

Request a cron status model that separates successful execution from failed final delivery and lets Control UI/API surfaces show delivery-only failures as warnings.

## Best Possible Solution

Land a reviewed, compatibility-safe execution/delivery split for cron status across scheduler state, gateway schema, run-log filters, CLI/UI labels, docs, and regression tests, or explicitly choose a different public partial-success model.

## Reproduction Assessment

Yes, source-reproducible: current main has only `ok`, `error`, and `skipped`, and unresolved/direct cron delivery failures still return `status: "error"` before scheduler persistence. I did not run a live cron job because this review is read-only.

## Solution Assessment

Unclear as final product direction: the execution/delivery split is the right shape for the reported problem, but a new first-class `warning` status is a public compatibility-sensitive contract and should land through the linked reviewed PR or an explicit maintainer-approved alternative.
