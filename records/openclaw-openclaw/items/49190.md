## Summary

Keep open. Current main still conflates non-best-effort cron delivery-target/direct-delivery failures with execution failures by persisting `error`, and the cron protocol plus Control UI still have no `warning` run status; related #64413 is adjacent delivery-reason work but does not replace the requested status/UI split.

## What This Changes

Request to classify cron runs where execution succeeds but announce delivery fails as `warning` instead of `error`, with persisted/API/Control UI status support.

## Best Possible Solution

Add an additive cron-owned warning outcome, or an explicit execution/delivery split that serializes to warning for successful execution plus failed post-execution delivery. Keep existing delivery diagnostics visible, avoid execution-error backoff/failure alerts for warning runs, update gateway schemas and Control UI filters/rendering, and coordinate with #64413 without waiting on its broader normalized reason-code work.

## Reproduction Assessment

Yes. A source-level reproduction is clear: an isolated `agentTurn` cron job with announce delivery and no resolvable delivery target reaches `failDeliveryTarget`, which current main records as `error` while preserving summary/output fields.

## Solution Assessment

Yes. The requested additive `warning` status or equivalent execution/delivery split is the narrow maintainable fix; relying only on `lastDeliveryStatus` leaves the primary persisted/API/UI status misleading.
