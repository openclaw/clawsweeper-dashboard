## Summary

Keep open: current main still cannot represent the requested warning state, delivery-target/direct-delivery failures still return `error`, and the issue has an open linked implementation PR that should be reviewed rather than closing the issue early.

## What This Changes

The issue requests cron run status semantics that distinguish successful execution with failed result delivery from true execution failure, surfacing the former as a warning in API/UI state.

## Best Possible Solution

Review, repair if needed, and land or replace https://github.com/openclaw/openclaw/pull/80139 for the warning-status contract, while coordinating broader reason-code work through https://github.com/openclaw/openclaw/issues/64413.

## Reproduction Assessment

Yes. Source inspection on current main shows no `warning` cron run status, delivery-target/direct-delivery failures still return `error`, and `error` still drives scheduler error counters; I did not run a mutating cron job in this read-only review.

## Solution Assessment

Yes as a product direction, but it is not implemented on main yet. The narrow maintainable solution is an additive warning-state contract that preserves true execution failures as errors while keeping delivery diagnostics separate.
