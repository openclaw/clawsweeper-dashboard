## Summary

Keep open. Current main still records this partial-success cron path as `error`, and the issue is already owned by an open closing PR, so it should remain open until that PR lands or is explicitly closed.

## What This Changes

Request cron runs to represent successful execution with failed final delivery as `warning` instead of `error` across persisted status, API, and Control UI.

## Best Possible Solution

Review and land or intentionally close https://github.com/openclaw/openclaw/pull/80139, keeping the broader normalized delivery-reason taxonomy in https://github.com/openclaw/openclaw/issues/64413 separate.

## Reproduction Assessment

Yes, from source inspection rather than a live run: an isolated agentTurn with requested announce delivery and unresolved target reaches `failDeliveryTarget`, returns `status: "error"`, and then increments cron error state in `applyJobResult`. A full runtime repro would use the issue's isolated announce/no-channel setup.

## Solution Assessment

Yes, the proposed first-class `warning` status is the right narrow direction for this partial-success case, provided it is added across persistence, Gateway schema, UI, CLI/docs, and plugin hook surfaces. The current best path is the existing open closing PR, not a separate cleanup close or duplicate repair job.
