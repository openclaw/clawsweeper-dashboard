## Summary

Keep open: current main and v2026.6.6 still expose only `ok`, `error`, and `skipped` cron run statuses, delivery-target failures still record `error`, and the linked implementation PR remains open and conflicting.

## What This Changes

Request an additive cron partial-success status that separates task execution success from final delivery failure and renders that state as a warning in Control UI.

## Best Possible Solution

Review, rebase, and land or replace https://github.com/openclaw/openclaw/pull/80139 with an additive warning-status contract across cron storage, Gateway/API, CLI, and Control UI while keeping https://github.com/openclaw/openclaw/issues/92460 as a separate delivery-channel propagation bug.

## Reproduction Assessment

Yes. Source inspection on current main shows no `warning` cron run status, delivery-target failures still return `error`, and `error` still drives scheduler failure counters; I did not run a live cron job in this read-only review.

## Solution Assessment

Mostly yes. The maintainable direction is an additive execution-vs-delivery status split, but maintainers still need to approve the public status contract and land or replace the open implementation PR.
