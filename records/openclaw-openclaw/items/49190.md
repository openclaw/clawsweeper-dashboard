## Summary

Keep open. Current main still models cron runs as `ok`/`error`/`skipped` and the unresolved announce-delivery path can still return `error`, while an open implementation PR explicitly closes this issue and should be reviewed or landed before cleanup.

## What This Changes

Request adding a cron warning status or execution/delivery split so successful isolated cron execution with failed announce delivery no longer appears as an execution error.

## Best Possible Solution

Land or revise the open additive warning-status implementation so cron persistence, Gateway protocol, CLI/UI, plugin hooks, docs, and regression tests all preserve the execution-versus-delivery distinction.

## Reproduction Assessment

Yes. Source inspection shows the unresolved non-best-effort delivery path returns `status: "error"`, and the cron/API/UI status contracts currently have no `warning` value.

## Solution Assessment

Yes, the maintainable direction is an additive warning status or equivalent execution/delivery split; the concrete code path should be resolved through the linked open PR rather than closing this issue early.
