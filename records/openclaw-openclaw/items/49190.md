- `impact:message-loss`: The issue centers on announce delivery failures being conflated with execution failures and hiding the true notification-delivery outcome.

## Summary

Keep open: current main and the latest stable release still have only `ok`, `error`, and `skipped` cron run statuses, delivery-target failures still return `error`, and the linked implementation at https://github.com/openclaw/openclaw/pull/80139 is open and unmerged.

## What This Changes

Request to split cron execution status from delivery status so a successful isolated run with failed announce delivery can surface as `warning` instead of `error`.

## Best Possible Solution

Review, refresh, and land or replace https://github.com/openclaw/openclaw/pull/80139 after the additive warning-status contract is accepted and the PR preserves current cron delivery/state behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main lacks `warning`, delivery-target failures still return `status: "error"`, and `error` still drives cron error counters; I did not run a mutating cron repro in this read-only review.

## Solution Assessment

Yes as a product direction, but it is not implemented on main yet. The narrow maintainable path is an additive warning-state contract that keeps fatal execution failures as errors while preserving delivery diagnostics separately.
