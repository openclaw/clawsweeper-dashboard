## Summary

Keep open: current main and the latest release still expose only `ok`, `error`, and `skipped` cron run statuses, and delivery-target/direct-delivery failures still return `error`; the linked fixing PR is open and conflicting rather than merged.

## What This Changes

The issue requests an additive cron status model that records successful execution with failed delivery as a warning so Control UI does not show those runs as execution errors.

## Best Possible Solution

Review, rebase, and land or replace https://github.com/openclaw/openclaw/pull/80139 for the additive warning-status contract, while leaving broader delivery reason-code normalization to https://github.com/openclaw/openclaw/issues/64413.

## Reproduction Assessment

Yes. Source inspection on current main shows no `warning` cron run status, delivery-target/direct-delivery failures still return `error`, and `error` still drives scheduler error counters.

## Solution Assessment

Yes as a product direction, but it is not implemented on main yet. The narrow maintainable path is an additive warning-state contract that preserves true execution failures as errors while keeping delivery diagnostics separate.
