## Summary

Keep open. Current main still has no cron `warning` run status, and delivery-target plus direct-delivery failures can still persist the whole run as `error` even when execution produced output; related #64413 is adjacent delivery-reason work, not a replacement for the requested status/UI split.

## What This Changes

Add a cron warning or execution-vs-delivery status split so Control UI distinguishes post-execution delivery failures from true task failures.

## Best Possible Solution

Add an additive cron-owned warning outcome, or an equivalent execution/delivery split serialized as warning for successful execution plus failed post-execution delivery, while keeping delivery diagnostics visible and excluding warning runs from execution-error backoff.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: an isolated `agentTurn` cron with requested announce delivery, no resolvable delivery target, and non-best-effort delivery reaches `failDeliveryTarget`, which current main records as `status: "error"`.

## Solution Assessment

Yes. A narrow additive warning status or explicit execution/delivery split is the maintainable fix because the current `ok | error | skipped` model plus delivery metadata still leaves primary persisted/UI status ambiguous.
