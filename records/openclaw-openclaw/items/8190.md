## Summary

Keep open. Current main still has only single-session/current-session reset plus adjacent per-session CLI auth invalidation; it lacks an admin-scoped, all-session, context-preserving auth refresh, and the related auth-migration/auth-epoch issues cover narrower problems.

## What This Changes

Request adds a global session auth refresh/reset that invalidates cached auth for all active channel sessions after auth changes while preserving conversation context.

## Best Possible Solution

Add an additive admin-scoped session auth refresh capability for all or selected active sessions that invalidates auth-backed runtime and CLI bindings without hard transcript rollover, preserves session ids/context where feasible, and documents failure semantics.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken behavior against an existing contract. Source inspection gives high confidence that current main only exposes single-session/current-session reset paths and no global context-preserving auth-refresh action.

## Solution Assessment

Unclear as proposed. Looping `sessions.reset` is not the best fix because it hard-rolls session ids/transcripts; the maintainable direction is a purpose-built admin-scoped auth refresh lifecycle.
