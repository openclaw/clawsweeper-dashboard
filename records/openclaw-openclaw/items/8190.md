## Summary

Keep open. Current main still only exposes single-session/current-session reset paths, while the requested global, context-preserving auth refresh is a new Gateway/session lifecycle capability and is not superseded by the related auth-migration issue.

## What This Changes

The issue requests a global, context-preserving session auth refresh/reset when switching auth backends or credentials.

## Best Possible Solution

Add an additive admin-scoped session auth refresh capability for all or selected active sessions that invalidates auth-backed runtime/CLI bindings without hard transcript rollover, preserves session ids/context where feasible, documents failure semantics, and coordinates with auth-store migration work.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken behavior against an existing contract. Source inspection gives high confidence that current main only exposes single-session/current-session reset paths and no global auth-refresh action.

## Solution Assessment

Unclear as proposed. Reusing or looping sessions.reset would hard-roll session ids/transcripts, so the maintainable direction is a purpose-built admin-scoped auth refresh lifecycle rather than broad use of the existing reset path.
