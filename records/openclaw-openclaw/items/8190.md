## Summary

Keep open. Current main still exposes per-session hard reset, current-session soft reset, CLI binding invalidation, and SecretRef reload support, but it does not provide the requested admin-scoped global active-session auth refresh that preserves session context across channels.

## What This Changes

Request an additive global session auth reset or refresh capability so active sessions adopt changed provider credentials without manual per-channel reset or context loss.

## Best Possible Solution

Add an owner-reviewed, admin-scoped sessions.refreshAuth-style capability that enumerates active sessions, invalidates auth-backed runtime and CLI bindings, preserves session ids/transcripts/context where safe, reports affected sessions, and documents coordination with host-wide auth migration.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main check that the existing surfaces are per-session reset, current-session chat reset, and provider/auth reload helpers rather than global active-session auth refresh.

## Solution Assessment

Yes for the direction, but it should be an additive admin-scoped auth refresh rather than a broad loop over the current hard reset service. The current reset path rolls session identity and transcripts, which conflicts with the requested context-preserving behavior.
