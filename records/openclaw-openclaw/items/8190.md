## Summary

Keep open. Current main still exposes only single-session hard reset and current-session soft reset; the requested global, context-preserving auth refresh remains a Gateway/session lifecycle feature that needs maintainer design.

## What This Changes

The issue requests a global, context-preserving session auth reset or refresh when switching auth backends or API keys.

## Best Possible Solution

Add an additive admin-scoped session/auth refresh capability that can target all or selected active sessions, invalidate auth-backed runtime and CLI bindings without hard transcript rollover, preserve session ids/context where feasible, and document failure and active-run semantics.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection on current main is high-confidence: auth/session reset surfaces are single-session or current-session only, and there is no global auth-refresh method.

## Solution Assessment

Unclear as proposed. Looping the current sessions.reset path would create new session ids and transcripts, so the maintainable direction is a purpose-built auth-refresh lifecycle with admin scoping and context-preservation semantics.
