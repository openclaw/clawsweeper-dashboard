## Label Justifications

- `P2`: This is a normal-priority feature with clear multi-channel user value, but it needs design review rather than urgent regression handling.
- `impact:session-state`: The requested behavior must preserve active session ids, transcripts, memory, and context while refreshing auth state.
- `impact:security`: A global auth refresh could redirect credentials for active conversations and needs explicit authorization and audit semantics.
- `impact:auth-provider`: The feature is specifically about switching provider accounts, API keys, auth profiles, and cached auth bindings.

## Summary

Keep open. Current main still has per-session hard reset, current-session soft reset, per-run CLI auth invalidation, and SecretRef reload, but no admin-scoped all-session auth refresh that preserves active session context; the related auth-migration issue is adjacent rather than a substitute.

## What This Changes

Request a global Gateway/session action that refreshes or resets all active sessions after an auth/provider switch while preserving conversation context.

## Best Possible Solution

Add an owner-reviewed, admin-scoped sessions.refreshAuth-style capability that enumerates active session stores, invalidates auth-backed runtime/CLI bindings, preserves session context, and documents coordination with host-wide auth migration.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main check: Gateway exposes single-session hard reset, current-session soft reset, per-run CLI auth invalidation, and SecretRef reload, but no global active-session auth refresh surface.

## Solution Assessment

Yes, keeping this open is the right disposition. The maintainable solution is an additive admin-scoped soft auth refresh path with explicit security and session-state semantics, not looping the existing hard reset over every session.
