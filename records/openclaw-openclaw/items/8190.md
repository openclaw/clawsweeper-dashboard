## Label Justifications

- `P2`: This is a normal-priority feature with clear user value, but it needs design review rather than urgent regression handling.
- `impact:session-state`: The requested behavior must preserve active session ids, transcripts, memory, and context while refreshing auth state.
- `impact:security`: A global auth refresh can redirect credentials across active conversations and needs explicit authorization and audit semantics.
- `impact:auth-provider`: The feature is specifically about switching provider accounts, auth profiles, API keys, and cached auth bindings.

## Summary

Keep open. Current main and the latest release still expose per-session reset and per-run CLI auth invalidation, but not an admin-scoped all-session auth refresh that preserves active session context, so this remains a feature/security design item.

## What This Changes

Request a global session auth refresh/reset command for auth backend or API-key switches that preserves active session context.

## Best Possible Solution

Add an owner-reviewed, admin-scoped sessions.refreshAuth-style capability that enumerates active session stores, invalidates auth-backed runtime/CLI bindings, preserves session context, and documents coordination with host-wide auth migration.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main check: Gateway exposes single-session hard reset, current-session soft reset, and per-run CLI auth invalidation, but no global active-session auth refresh surface.

## Solution Assessment

Yes, keeping this open is the right disposition. The maintainable solution is an additive admin-scoped soft auth refresh path with explicit security and session-state semantics, not looping the existing hard reset over every session.
