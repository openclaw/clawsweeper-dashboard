- `impact:security`: A global auth refresh can redirect credentials across active conversations and needs explicit authorization/audit semantics.
- `impact:auth-provider`: The feature is specifically about switching provider accounts, auth profiles, API keys, and cached auth bindings.

## Summary

Keep open. Current main still has only single-session hard reset and current-session soft reset paths; the related auth-switching issues cover account/order migration rather than this active-session auth refresh lifecycle.

## What This Changes

Request an admin/global session auth refresh or reset action that updates active sessions after auth/provider changes while preserving conversation context.

## Best Possible Solution

Add an owner-reviewed, admin-scoped sessions.refreshAuth-style capability that enumerates active session stores, invalidates auth-backed runtime and CLI bindings, preserves session ids/transcripts/context where possible, and documents coordination with host-wide auth migration.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main check: Gateway exposes single-session hard reset and current-session soft reset, but no global active-session auth refresh surface.

## Solution Assessment

Yes, keeping this open is the right disposition. The maintainable solution is an additive soft auth refresh path with explicit security and session-state semantics, not looping the existing hard reset over every session.
