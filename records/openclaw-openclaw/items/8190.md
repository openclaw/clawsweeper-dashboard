## Summary

Keep open: current main and v2026.6.10 still provide single-session reset plus per-session CLI auth invalidation, not a global, context-preserving auth refresh across active sessions. The related issues cover narrower provider-switch, account-switch, or auth-epoch slices and do not supersede this broader Gateway/session lifecycle request.

## What This Changes

Request a global Gateway/session action that refreshes auth across all active sessions after auth or provider changes while preserving session context.

## Best Possible Solution

Design an additive, maintainer-approved Gateway/session auth-refresh contract that enumerates active sessions, invalidates stale auth-backed runtime state, preserves intended context and user selections, and documents the admin/security semantics.

## Reproduction Assessment

Yes, at source level: current main has a single-key sessions.reset schema/handler and per-session CLI auth invalidation, while no global auth-refresh/reset method exists. I did not run a live multi-channel auth-switch scenario in this read-only review.

## Solution Assessment

Unclear as a final implementation: a global Gateway action is plausible, but maintainers need to define admin scope, preservation rules, and stale-auth versus intentional pin semantics before a safe fix lands.
