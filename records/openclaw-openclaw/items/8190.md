## Summary

Keep open: current main and v2026.6.10 still expose single-session reset plus per-session CLI auth invalidation, not a global, context-preserving auth refresh across active sessions. The related issues cover narrower account switching or provider-switch stale-state slices and do not supersede this broader lifecycle/API design request.

## What This Changes

The issue requests a global, context-preserving session auth refresh/reset when auth backends or API keys change.

## Best Possible Solution

Keep this as the canonical tracker for a maintainer-designed global auth refresh/reset contract that enumerates active sessions, invalidates auth-backed runtime state, preserves intended context and user selections, and documents the admin/security semantics.

## Reproduction Assessment

Yes at source level: current main has a single-key sessions.reset schema/handler and per-session CLI auth invalidation, while docs still tell users to reset the affected chat after auth changes. I did not run a live multi-channel auth-switch scenario in this read-only review.

## Solution Assessment

Unclear as a final implementation: a global Gateway action is plausible, but maintainers need to define admin scope, preservation rules, and stale-auth versus intentional pin semantics before a safe fix lands.
