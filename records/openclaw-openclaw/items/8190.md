## Summary

Keep open. Current main still exposes single-session reset paths only, and the related auth-switching issues cover account/order migration rather than the active-session auth refresh lifecycle requested here.

## What This Changes

The issue requests a global auth/session reset or auth refresh capability that updates all active sessions after auth backend changes while preserving conversation context.

## Best Possible Solution

Add an owner-reviewed, admin-scoped sessions.refreshAuth-style capability that enumerates active session stores, invalidates auth-backed runtime and CLI bindings, preserves session ids/transcripts/context where possible, and documents how it coordinates with host-wide auth profile migration.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main check: the Gateway exposes single-session hard reset and current-session soft reset, but no global auth refresh surface.

## Solution Assessment

Yes, keeping this open is the right disposition. The best fix is not to call the existing hard reset for every session, but to design an additive soft auth refresh path with explicit security and session-state semantics.
