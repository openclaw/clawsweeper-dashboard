## Label Justifications

- `P2`: This is a clear multi-channel auth improvement with security implications, but it is not an active outage or urgent regression.
- `impact:session-state`: The requested behavior must preserve active session ids, transcripts, memory, and context while refreshing auth state.
- `impact:security`: A global auth refresh can change which credentials active conversations use and therefore needs explicit authorization and audit semantics.
- `impact:auth-provider`: The request is specifically about switching provider accounts, API keys, auth profiles, and cached auth bindings.

## Summary

Keep open. Current main still exposes only single-session hard reset, current-session soft reset, SecretRef reload, and auth-identity-aware CLI resume behavior; it does not add an admin-scoped all-session auth refresh that preserves active session context.

## What This Changes

Request an OpenClaw Gateway capability to refresh or reset auth across all active channel sessions after an auth/provider switch while preserving conversation context.

## Best Possible Solution

Add an owner-reviewed, admin-scoped sessions.refreshAuth-style capability that enumerates active session stores, invalidates auth-backed runtime and CLI bindings, preserves session ids/transcripts/context where possible, and documents coordination with host-wide auth migration.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main check: Gateway exposes single-session hard reset, current-session soft reset, SecretRef reload, and CLI auth identity guards, but no global active-session auth refresh surface.

## Solution Assessment

Yes for the direction, but it should be solved as an additive admin-scoped auth refresh rather than by looping the existing hard reset service over every session. The current hard reset service rolls session ids and transcripts, which does not match the requested context-preserving behavior.

## Risks / Open Questions

- A global auth refresh could redirect active conversations to different provider credentials unless admin authorization, target selection, audit/reporting, and stale credential invalidation semantics are explicit.
- The implementation must preserve session ids, transcripts, queued work, and context while avoiding reuse of stale CLI/runtime auth bindings.
- The design should coordinate with https://github.com/openclaw/openclaw/issues/61709 so stored auth migration and active-session auth refresh do not diverge.
