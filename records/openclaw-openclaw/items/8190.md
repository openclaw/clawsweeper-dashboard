## Summary

Keep open. Current main still exposes single-session reset and current-session soft reset only; the requested global, context-preserving auth refresh remains a cross-Gateway/session lifecycle feature that needs maintainer design.

## What This Changes

Request a Gateway-level action or command that refreshes auth across all active sessions after auth backend or API-key changes while preserving conversation context.

## Best Possible Solution

Add an additive admin-scoped session/auth refresh capability that can target all or selected active sessions, invalidate auth-backed runtime and CLI bindings without hard transcript rollover, preserve session ids/context where feasible, and document failure and active-run semantics.

## Reproduction Assessment

Not applicable as a feature request. Source and docs verification on current main is high-confidence: auth remains pinned per session, and existing reset surfaces are single-session or current-session only.

## Solution Assessment

Unclear as proposed. Looping the current sessions.reset path would create new session ids/transcripts, so the maintainable direction is a purpose-built auth-refresh lifecycle capability with admin scoping and context-preservation semantics.
