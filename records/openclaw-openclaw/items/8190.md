## Summary

Keep open: current main and v2026.6.10 still expose per-session reset and per-session CLI auth invalidation, not the requested global, context-preserving auth refresh. Related issues cover narrower provider-switch, cross-runtime continuity, or session-state resolver work, but none supersedes this broader lifecycle/API design request.

## What This Changes

The issue requests a global, context-preserving reset or auth-refresh capability so all active sessions pick up changed provider credentials without manually resetting each channel.

## Best Possible Solution

Define an additive Gateway/session auth-refresh contract that can target all active sessions, invalidate stale runtime credential bindings, preserve session ids and transcripts where safe, and specify admin authorization plus in-flight behavior.

## Reproduction Assessment

Yes, at source level: current main exposes only single-session hard reset plus per-session CLI-binding/auth-epoch invalidation, with no all-session context-preserving auth-refresh contract. I did not run a live multi-channel provider/auth switch.

## Solution Assessment

No for a hard global reset: current reset rotates session ids and transcript files. The maintainable direction is an additive auth/runtime invalidation contract that preserves session records and explicit user choices.
