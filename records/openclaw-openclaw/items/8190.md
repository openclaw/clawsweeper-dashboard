- `impact:auth-provider`: The issue is specifically about switching provider accounts, API keys, auth profiles, and cached auth bindings.

## Summary

Keep open. Current main still has per-session hard reset, current-session soft reset, auth-status cache invalidation, and provider auth removal/abort behavior, but no admin-scoped global active-session auth refresh that preserves session context; the related auth migration and model-default issues are adjacent rather than replacements.

## What This Changes

Request a global, context-preserving active-session auth refresh/reset when provider credentials or auth backends change.

## Best Possible Solution

Add an owner-reviewed, admin-scoped sessions.refreshAuth-style capability that enumerates active session stores, invalidates auth-backed runtime and CLI bindings, preserves session ids/transcripts/context where safe, reports affected sessions, and documents coordination with host-wide auth migration.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main check: Gateway exposes per-session hard reset, current-session soft reset, CLI auth invalidation, and SecretRef/auth-status reload paths, but no global active-session auth refresh surface.

## Solution Assessment

Yes for the direction, but it should be solved as an additive admin-scoped auth refresh rather than by looping the existing hard reset service over every session. The current hard reset rolls session ids and transcripts, which does not match the requested context-preserving behavior.
