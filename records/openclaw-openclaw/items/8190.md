## Summary

Current main and v2026.6.5 still have only single-session destructive reset, while normal session reconstruction preserves provider/auth selections; the related open issues cover cross-agent profile migration and a Windows-specific stale-route reproduction, not this global context-preserving lifecycle capability.

## What This Changes

Requests a global, context-preserving session auth refresh when provider credentials or auth backends change.

## Best Possible Solution

Define one admin-authorized refresh transaction that updates the intended auth profile state, enumerates canonical agent session stores, invalidates only auto/runtime credential and provider bindings, preserves session IDs and transcripts, and reports active sessions that could not be safely refreshed.

## Reproduction Assessment

Yes at source level: current main preserves session auth/provider selections during ordinary reconstruction, and its only reset API addresses one session and creates a new session identity. A live multi-provider switch was not run, but the missing global capability is unambiguous from protocol and runtime source.

## Solution Assessment

No, a bulk invocation of the existing reset is not the best solution because it rolls session identity and transcript state. The best direction is a narrow, authorized auth/runtime refresh transaction that preserves explicit user choices and canonical conversation state.

- The overlap with https://github.com/openclaw/openclaw/issues/61709 must be resolved so profile-store migration and active-session invalidation do not become competing partial transactions.
