## Summary

Keep open: current main and v2026.6.6 still lack a global, context-preserving auth/session refresh contract; existing reset paths are single-session and either rotate session state or handle narrower CLI auth reuse.

## What This Changes

Request to add a global auth/session refresh or reset path that updates all active sessions after auth/provider changes while preserving conversation context.

## Best Possible Solution

Add a core auth/session invalidation contract that refreshes stale runtime credential bindings across active sessions, preserves session IDs and transcripts, honors explicit user selections, and defines admin authorization plus in-flight-turn behavior.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 expose only single-session reset and runtime-specific auth epoch handling, with no all-session auth-refresh contract. I did not run a live multi-channel auth-switch repro.

## Solution Assessment

No, a hard global reset is not the best implementation because current reset rotates session IDs and transcript state. The safer direction is auth-generation invalidation at persistent runtime boundaries while preserving session records and explicit user choices.
