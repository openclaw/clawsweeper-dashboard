## Summary

Keep open: current main and v2026.6.6 still lack a global, context-preserving auth/session refresh contract; existing reset paths are single-session and either rotate session state or clear only the addressed session's runtime bindings.

## What This Changes

The issue requests a global session/auth refresh action that updates active sessions after auth or provider changes while preserving conversation context.

## Best Possible Solution

Add a core auth/session invalidation contract that refreshes stale runtime credential bindings across active sessions, preserves session ids and transcripts, honors explicit user selections, and defines admin authorization plus in-flight-turn behavior.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 expose only single-session hard reset plus current-session soft reset, with no all-session auth-refresh contract. I did not run a live multi-channel provider/auth switch.

## Solution Assessment

No, a hard global reset is not the best implementation because current reset rotates session ids and transcript files. The safer direction is auth/runtime invalidation at persistent runtime boundaries while preserving session records and explicit user choices.
