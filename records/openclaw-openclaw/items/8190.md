## Summary

Keep open. Current main and v2026.5.22 still provide per-session reset, current-session soft reset, and SecretRef reload, but not a global active-session auth refresh/reset that preserves conversation context across auth switches.

## What This Changes

The issue requests an admin or gateway-level way to refresh/reset auth for all active sessions after provider/account changes while preserving session memory and context.

## Best Possible Solution

Add an owner-reviewed, admin-scoped sessions.refreshAuth-style capability that enumerates active sessions, invalidates auth-backed runtime and CLI bindings, preserves session ids/transcripts/context where safe, reports affected sessions, and documents coordination with secrets.reload and auth migration.

## Reproduction Assessment

Not applicable as a feature request. Source, docs, tests, and latest-release inspection give high confidence that OpenClaw exposes per-session reset and current-session soft reset rather than global active-session auth refresh.

## Solution Assessment

Yes for the direction, but not as a broad loop over the current hard reset service. The maintainable solution should add explicit admin-scoped auth-refresh semantics that preserve session identity and transcripts while invalidating auth-backed bindings.
