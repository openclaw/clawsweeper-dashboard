## Summary

Keep open. Current main has per-request CLI auth-epoch invalidation and per-agent auth order controls, but it still lacks a global, context-preserving all-session auth refresh/reset action; `sessions.reset` remains a single-key hard rollover.

## What This Changes

Request a Gateway/session capability to refresh or reset auth for all active sessions after auth switches while preserving conversation context.

## Best Possible Solution

Add an additive admin-scoped session/auth refresh capability that can target all or selected active sessions, invalidate auth-backed runtime and CLI bindings without hard transcript rollover, preserve session ids/context where feasible, and document active-run and failure semantics.

## Reproduction Assessment

Not applicable; this is a feature request. Source inspection is high-confidence that current reset surfaces are single-session/current-session scoped and there is no global auth-refresh method.

## Solution Assessment

Unclear as proposed. Looping the existing `sessions.reset` path would create new session ids and archived transcripts, so the maintainable direction is a purpose-built admin-scoped auth-refresh lifecycle that reuses current auth-epoch/runtime-routing checks where possible.
