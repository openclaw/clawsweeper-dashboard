## Summary

Keep open: current main and the latest release still lack a global, context-preserving auth/session refresh contract; existing reset paths are single-session and current auth-epoch logic only invalidates one CLI binding on a later turn.

## What This Changes

Request a global auth-switch session refresh/reset action that invalidates stale auth across active sessions while preserving conversation context.

## Best Possible Solution

Define a core Gateway/session auth-refresh contract that can target all active sessions, invalidate stale runtime credential bindings, preserve session ids and transcripts where safe, honor explicit user selections, and specify admin authorization plus in-flight behavior.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 expose only single-session hard reset plus per-session auth-epoch invalidation, with no all-session auth-refresh contract. I did not run a live multi-channel provider/auth switch.

## Solution Assessment

No, a hard global reset is not the best shape because current reset rotates session ids and transcript files. The safer direction is an auth/runtime invalidation contract that preserves session records and explicit user choices.
