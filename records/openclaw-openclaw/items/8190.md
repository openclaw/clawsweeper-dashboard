## Summary

Keep open. Current main and the latest release still lack a global, context-preserving all-session auth refresh/reset; narrower CLI auth-epoch and runtime refresh work helps adjacent stale-auth cases but does not satisfy this cross-session Gateway/session lifecycle request.

## What This Changes

The issue requests a global, context-preserving session auth refresh/reset when auth backends or provider credentials change.

## Best Possible Solution

Keep this as the canonical maintainer design tracker and define an admin-authorized auth-generation/runtime invalidation contract that refreshes stale credentials across active sessions while preserving session rows, transcripts, and explicit user choices.

## Reproduction Assessment

Yes at source level: the Gateway protocol, method registry, and handler still expose only a single-session `sessions.reset`, and the implementation rotates session identity/transcripts instead of providing a global auth-only refresh. I did not run a live multi-channel credential-switch scenario.

## Solution Assessment

No, a global hard reset is not the best implementation because the existing reset path is intentionally destructive for session identity. The maintainable direction is an admin-authorized auth-generation/runtime invalidation contract that preserves session rows and explicit per-session choices.
