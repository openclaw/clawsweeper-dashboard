## Summary

Keep open. Current main still exposes only single-session reset/auth invalidation paths, and the existing hard reset creates a new session id/transcript rather than the requested all-session, auth-only refresh with preserved context. The related auth-order and auth-migration work is useful context but does not supersede this cross-session lifecycle feature.

## What This Changes

Request to add a gateway-level global auth refresh/reset action that updates all active sessions after auth backend or API-key changes while preserving conversation context.

## Best Possible Solution

Keep this as the canonical tracker for a core Gateway/session lifecycle capability: add an additive all-sessions auth refresh/reset API, enumerate active session stores safely, invalidate auth-backed runtime and CLI bindings/auth epochs, preserve session ids/transcripts/context where feasible, define admin-scope semantics, and cover the protocol, Gateway handler, session store, channel behavior, and docs. Treat #61709 as related auth-store migration work, not a replacement.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence source-inspection path for the missing behavior: current protocol, method registry, handler, hard reset service, and soft reset command all remain single-session scoped.

## Solution Assessment

Unclear until maintainers settle the API contract. Reusing the existing hard reset globally would not be the best solution because it rolls sessions and archives transcripts; the safer direction is an auth-only refresh capability with explicit all-sessions scope and context preservation.
