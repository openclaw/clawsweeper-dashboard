## Summary

Keep open. Current main still has only single-session/current-session reset paths, while the requested global, context-preserving auth refresh is a new Gateway/session lifecycle feature with security-sensitive semantics.

## What This Changes

The issue requests a Gateway-level action or command to refresh/reset auth across all active channel sessions while preserving conversation context after auth/backend switches.

## Best Possible Solution

Add an additive admin-scoped session/auth refresh capability that can target all or selected active sessions, invalidate auth-backed runtime and CLI bindings without hard transcript rollover, preserve session ids/context where feasible, and document failure and active-run behavior.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken behavior against an existing contract. Source inspection gives high confidence that current main only exposes single-session/current-session reset paths and no global auth-refresh action.

## Solution Assessment

Unclear as proposed. Reusing or looping sessions.reset would hard-roll session ids/transcripts, so the maintainable direction is a purpose-built admin-scoped auth refresh lifecycle rather than broad use of the existing reset path.
