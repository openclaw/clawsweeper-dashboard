## Summary

Current main still lacks the requested global, context-preserving auth refresh/reset: Gateway exposes a single-key hard reset, while auth-epoch logic only invalidates reusable CLI bindings for individual sessions on the next run.

## What This Changes

The issue requests a global, context-preserving session auth refresh/reset when users switch provider credentials or auth backends.

## Best Possible Solution

Design an additive admin-scoped Gateway/session capability such as `sessions.refreshAuth` with all-session targeting, explicit credential/session safety semantics, context preservation, and focused Gateway/session-store/channel tests.

## Reproduction Assessment

Yes at source level: current Gateway protocol and method handling only expose a single-key reset, while CLI auth epochs only invalidate reusable CLI sessions during run preparation. I did not establish a live multi-channel auth-switch reproduction in this read-only review.

## Solution Assessment

No current implementation fully solves it: the existing hard reset is too destructive for context-preserving auth refresh, and CLI auth epochs only cover part of the cache problem. The best fix needs an explicit global refresh contract rather than repurposing `sessions.reset` blindly.
