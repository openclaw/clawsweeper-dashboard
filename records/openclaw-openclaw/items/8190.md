## Summary

Keep open. Current main still exposes only single-session/current-session reset paths, while the requested behavior is a new global, context-preserving auth refresh across active sessions after an auth switch; the related auth-order and auth-migration items do not cover active session runtime/cache invalidation.

## What This Changes

Request a global, context-preserving session auth refresh/reset so active channel sessions pick up changed auth backends or API keys without per-session manual resets.

## Best Possible Solution

Add an additive admin-scoped session auth refresh capability for all or selected active sessions that invalidates auth-backed runtime and CLI bindings without hard transcript rollover, preserves session ids/context where feasible, and documents failure semantics.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken behavior against an existing contract. Source inspection gives high confidence that current main only exposes single-session/current-session reset paths and no global auth-refresh action.

## Solution Assessment

Unclear as proposed. Reusing or looping sessions.reset would hard-roll session ids/transcripts, so the maintainable direction is a purpose-built admin-scoped auth refresh lifecycle rather than broad use of the existing reset path.
