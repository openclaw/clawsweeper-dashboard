## Summary

Keep open. Current main still has only keyed per-session reset and per-session soft CLI binding cleanup, while the requested all-session auth refresh with context preservation remains a new Gateway/session lifecycle feature that needs maintainer API design.

## What This Changes

Add a global auth-refresh/reset capability that updates all active sessions after auth/provider switches while preserving conversation context.

## Best Possible Solution

Define an additive Gateway/session API for all-session auth refresh that invalidates auth-backed runtime and CLI bindings while preserving session ids, transcripts, and conversation context wherever the session type supports it.

## Reproduction Assessment

Yes for source-inspection verification, not as a runtime bug reproduction. Current main documents per-session auth stickiness and exposes only keyed `sessions.reset` plus per-session soft reset cleanup, with no all-sessions auth-refresh method.

## Solution Assessment

Unclear until maintainers settle the API contract. Reusing the existing hard reset globally is not the best solution because it creates new session ids and archives transcripts; the safer direction is an auth-only refresh with explicit all-session scope.
