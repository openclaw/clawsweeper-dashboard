## Label Justifications

- `P2`: Stale session routing after account switches is a meaningful multi-channel reliability and usability problem without emergency-level impact.
- `impact:session-state`: Persisted per-session routing state can diverge from the current global authentication configuration.
- `impact:auth-provider`: Existing sessions can remain pinned to an old provider account or auth profile after an operator switches credentials.
- `impact:security`: A global credential-selection operation requires explicit operator authorization and safe handling of intentionally locked profiles.

## Summary

Keep open: current main refreshes runtime credential snapshots, but persisted session-level provider, model, and auth-profile selections can still retain the prior route, and no global context-preserving refresh capability exists.

## What This Changes

Requests a global, context-preserving refresh of session authentication after provider or credential changes.

## Best Possible Solution

Add an operator-authorized, additive session-routing refresh that clears only eligible auto-derived provider/auth pins across selected agent stores while preserving session IDs, transcripts, and context; retain intentional user-locked choices unless explicitly requested.

## Reproduction Assessment

Yes at the source level: current main persists session routing and auth-profile selections, while its public reset API handles one session through a hard rollover. This read-only review did not exercise a live multi-channel environment.

## Solution Assessment

No, a global hard reset is not the best solution because it replaces session identity and interrupts active work. A narrower refresh of eligible session routing and auth selections is safer and preserves conversation state.
