## Summary

Keep open. Current main fixes stale credentials for reusable CLI sessions, but it still has no global, context-preserving auth refresh contract, and related open reports show persistent provider/runtime state can remain stale after switching.

## What This Changes

Request a global, context-preserving refresh of authentication used by active sessions after provider credentials or auth backends change.

## Best Possible Solution

Add one core auth-generation contract consumed by every persistent runtime: invalidate only stale runtime credential bindings after canonical auth changes, preserve session IDs and transcripts, retain explicit user-selected session profiles unless deliberately included, and define admin authorization plus in-flight-turn behavior.

## Reproduction Assessment

Yes at source level: current main still lacks an all-session auth-refresh contract, and persistent-runtime reports provide concrete adjacent failures. There is not yet one high-confidence live reproduction demonstrating the same stale-auth behavior across every channel and runtime.

## Solution Assessment

No, a global hard session reset is not the best implementation because the existing reset rotates session state. The safer direction is generation-based invalidation at each persistent runtime boundary while preserving canonical session records and explicit user choices.

## Risks / Open Questions

- A global operation could override intentional per-session auth-profile choices unless default-derived and explicit selections have distinct semantics.

- The implementation must define behavior for in-flight turns and avoid rotating session IDs, archiving transcripts, or losing context merely because credentials changed.
- Credential invalidation requires an admin-authorized protocol boundary and must not expose credential fingerprints or secret material.
