## Summary

Keep open. Current main still exposes only single-session Gateway reset and current-session `/reset soft`; there is no global, context-preserving auth refresh/reset action, and the related auth migration/switching issues do not supersede active session cache invalidation.

## What This Changes

Request an admin/global session reset or auth-refresh action that invalidates cached auth across active channel sessions while preserving conversation context.

## Best Possible Solution

Add an additive admin-scoped session/auth refresh capability that targets all or selected active sessions, invalidates auth-backed runtime and CLI bindings without hard transcript rollover, preserves session ids/context where feasible, and documents active-run and failure semantics.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken behavior against an existing contract. Source inspection gives high confidence that current main only exposes single-session/current-session reset paths and no global auth-refresh action.

## Solution Assessment

Unclear as proposed. Reusing or looping `sessions.reset` would hard-roll transcripts and session ids, so the maintainable direction is a purpose-built admin-scoped auth refresh lifecycle rather than broad use of the existing reset path.
