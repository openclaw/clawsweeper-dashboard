## Summary

Keep open. Current main still has single-session hard reset, current-session soft reset, per-run CLI auth invalidation, and SecretRef reload, but no admin-scoped all-session auth refresh that preserves active session context; the related auth migration and no-model account switch issues do not cover this lifecycle gap.

## What This Changes

Request a global Gateway/session action that refreshes auth for all active sessions after auth or provider changes while preserving conversation context.

## Best Possible Solution

Add an owner-reviewed, admin-scoped sessions.refreshAuth-style capability that enumerates active session stores, invalidates auth-backed runtime and CLI bindings, preserves session context, and documents coordination with host-wide auth migration.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence current-main check: Gateway exposes single-session hard reset, current-session soft reset, per-run CLI auth invalidation, and SecretRef reload, but no global active-session auth refresh surface.

## Solution Assessment

Yes, keeping this open is the right disposition. The maintainable solution is an additive admin-scoped soft auth refresh path with explicit security and session-state semantics, not looping the existing hard reset over every session.
