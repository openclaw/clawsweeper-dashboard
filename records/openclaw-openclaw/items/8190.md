## Summary

Keep open. Current main still exposes only single-session/current-session reset paths, and the requested global, context-preserving auth refresh is a new security-sensitive Gateway/session capability rather than implemented behavior.

## What This Changes

Add a gateway/global action to refresh or reset auth caches across all active sessions after auth/provider changes while preserving conversation context.

## Best Possible Solution

Add an additive admin-scoped session/auth refresh capability that can target all or selected active sessions, invalidate auth-backed runtime and CLI bindings without hard transcript rollover, preserve session ids/context where feasible, and document failure and active-run semantics.

## Reproduction Assessment

Not applicable; this is a feature request rather than broken behavior against an existing contract. Source inspection gives high confidence that current main only exposes single-session/current-session reset paths and no global auth-refresh action.

## Solution Assessment

Unclear as proposed. Reusing or looping sessions.reset would hard-roll transcripts and session ids, so the maintainable direction is a purpose-built admin-scoped auth refresh lifecycle rather than broad use of the existing reset path.
