## Summary

Keep open: current main and v2026.6.6 still lack a global, context-preserving auth/session refresh; the implemented paths are single-session hard reset or per-session CLI auth-epoch invalidation, and related issues cover narrower overlaps rather than superseding this design.

## What This Changes

Request a global auth/session refresh when providers or credentials change, while preserving active conversation context across channels.

## Best Possible Solution

Define a core Gateway/session auth-refresh contract that can target all active sessions, invalidate stale runtime credential bindings, preserve session ids and transcripts where safe, honor explicit user selections, and specify admin authorization plus in-flight behavior.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.6 expose only single-session hard reset plus per-session auth-epoch invalidation, with no all-session context-preserving auth-refresh contract. I did not run a live multi-channel provider/auth switch.

## Solution Assessment

No, a hard global reset is not the best shape because current reset rotates session ids and transcript files. The safer direction is an additive auth/runtime invalidation contract that preserves session records and explicit user choices.
