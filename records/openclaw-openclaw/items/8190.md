## Summary

Keep open. Current main still only supports single-session reset/soft-reset, the documented auth behavior remains session-pinned, and the related auth-switching issues do not cover the requested global, context-preserving active-session auth refresh.

## What This Changes

The issue requests a global, context-preserving session auth refresh/reset when switching providers, accounts, or API keys.

## Best Possible Solution

Define an additive admin-scoped Gateway/session capability that refreshes auth-backed runtime bindings across selected active sessions while preserving session ids, transcripts, and context where possible.

## Reproduction Assessment

Not applicable as a feature request. Current-main source and docs give a high-confidence check: auth profiles are pinned per session, and only single-session reset/soft-reset surfaces exist today.

## Solution Assessment

Unclear as proposed. A Gateway action such as sessions.refreshAuth may be the right direction, but the best solution needs owner-approved protocol, security, and context-preservation semantics before implementation.
