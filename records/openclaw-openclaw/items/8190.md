## Summary

Keep open: current main and the latest v2026.6.8 release still provide single-session reset and per-session CLI auth invalidation, not the requested global, context-preserving auth refresh. The related Windows provider-switch issue at https://github.com/openclaw/openclaw/issues/88372 partially overlaps but tracks a narrower concrete repro rather than superseding this design work.

## What This Changes

The issue requests a global auth-switch session reset or refresh that updates active sessions to current credentials while preserving conversation context.

## Best Possible Solution

Define a core Gateway/session auth-refresh contract that can target all active sessions, invalidate stale runtime credential bindings, preserve session ids and transcripts where safe, honor explicit user selections, and specify admin authorization plus in-flight behavior.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.8 expose only single-session hard reset plus per-session auth-epoch or CLI-binding invalidation, with no all-session context-preserving auth-refresh contract. I did not run a live multi-channel provider/auth switch.

## Solution Assessment

No, a hard global reset is not the best shape because current reset rotates session ids and transcript files. The safer direction is an additive auth/runtime invalidation contract that preserves session records and explicit user choices.
