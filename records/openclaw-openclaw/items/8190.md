## Summary

Keep open. Current main still has only single-session/current-session reset surfaces plus per-request CLI auth-epoch checks; it does not provide a global, context-preserving auth refresh/reset across active sessions.

## What This Changes

Request a global Gateway/session action to refresh or reset all active sessions after auth/provider changes while preserving conversation context.

## Best Possible Solution

Add an additive admin-scoped session/auth refresh capability that can target all or selected active sessions, invalidate auth-backed runtime and CLI bindings without hard transcript rollover, preserve session ids/context where feasible, and document active-run and failure semantics.

## Reproduction Assessment

Not applicable; this is a feature request. Source inspection gives high confidence that current reset surfaces are single-session/current-session scoped and no global auth-refresh method exists.

## Solution Assessment

Unclear as proposed. Looping the existing sessions.reset path would mint new session ids and archive transcripts, so the maintainable direction is a purpose-built admin-scoped auth-refresh lifecycle rather than a broad hard reset.
