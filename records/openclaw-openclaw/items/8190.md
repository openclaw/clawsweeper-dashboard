## Summary

Keep open. Current main and v2026.6.5 still lack an explicit global, context-preserving auth refresh, while existing reset behavior is single-session and destructive; related issues cover profile migration and a Windows-specific stale-route repro rather than replacing this cross-session lifecycle request.

## What This Changes

Request a global, context-preserving refresh of active sessions when provider credentials or auth selection changes.

## Best Possible Solution

Add an operator-admin, auth-specific refresh operation that advances canonical auth/runtime generations across targeted active sessions while preserving session ids, transcripts, and context, with explicit runtime-specific invalidation and audit events rather than invoking hard reset.

## Reproduction Assessment

Yes for the capability gap: current main exposes only key-scoped session reset and no global auth-refresh operation. The broader claim that every runtime permanently caches credentials is not universal because current CLI and embedded runtimes already contain auth epoch and refresh mechanisms.

## Solution Assessment

No, globally hard-resetting sessions is not the best solution because the current reset service changes session identity and archives transcripts. The better design is an additive auth-generation refresh that preserves canonical session state and delegates invalidation to each runtime owner.
