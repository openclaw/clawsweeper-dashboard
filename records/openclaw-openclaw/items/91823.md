## Summary

Keep open: current main and v2026.6.5 still allow a source-reproducible path where Linux auto-update falls back to spawning `openclaw update` as a Gateway descendant when only generic OpenClaw service markers are present, and the CLI then emits the exact ancestry guard reported here.

## What This Changes

The issue reports that Linux systemd user-service auto-update invokes the updater inside the Gateway process tree, causing `openclaw update` to refuse the managed-service restart.

## Best Possible Solution

Add a narrow Linux managed-service auto-update repair: avoid direct child `openclaw update` when the Gateway has OpenClaw service markers but no systemd-specific supervisor hint, and either start a verified external systemd handoff or return the existing manual-update guidance before spawning the updater.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a Linux Gateway process with only OpenClaw service markers is not classified as systemd-supervised, so auto-update can spawn `openclaw update` directly with inherited Gateway ancestry. I did not run a live systemd package update in this read-only review.

## Solution Assessment

No, current main is not sufficient because v2026.6.5 already contains the current systemd-run handoff source and the reporter still hit the ancestry guard. The maintainable fix is to close the unsupervised fallback gap or fail before spawning the updater.

Preserve the existing custom-supervisor caution outside the auto-update path: do not broadly classify all Linux service markers as systemd unless the code can prove an OpenClaw-managed systemd Gateway service.
