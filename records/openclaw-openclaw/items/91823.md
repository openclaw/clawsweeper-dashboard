## Summary

Keep open: current main still has a source-reproducible Linux managed-service gap where startup auto-update can miss the systemd handoff path and fall back to spawning `openclaw update` as a Gateway descendant, which then hits the reported ancestry guard.

## What This Changes

The issue reports Linux systemd user-service auto-update failing after the rollout window because the updater still runs inside the Gateway process tree instead of a detached managed-service handoff.

## Best Possible Solution

Make startup auto-update fail safe for Linux managed Gateway services: use the verified `systemd-run --user --scope` handoff when service ownership is identifiable, or refuse before spawning a Gateway child and surface the manual update command.

## Reproduction Assessment

Yes, source-level. A Linux Gateway process with OpenClaw gateway service markers but without the systemd-specific hints returns no supervisor, so auto-update can spawn `openclaw update` directly and then hit the current package-update ancestry guard.

## Solution Assessment

No, current main is not sufficient. The best fix is a narrow managed-service handoff/fail-safe repair, not weakening the CLI ancestry guard that protects manual in-service package updates.
