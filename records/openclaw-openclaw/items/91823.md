## What This Changes

The issue reports that Linux systemd user-service auto-update can invoke `openclaw update` as a Gateway descendant instead of handing off outside the Gateway process tree.

## Best Possible Solution

Merge https://github.com/openclaw/openclaw/pull/91833 or an equivalent narrow fix that routes Linux Gateway service-marker auto-updates through the existing managed systemd handoff, then let the issue close after the fix lands.

## Reproduction Assessment

Yes from source inspection: a Linux Gateway process with only generic OpenClaw gateway service markers is not classified as systemd-supervised, so startup auto-update can spawn the updater directly under the Gateway process tree. I did not run a live systemd package update in this read-only review.

## Solution Assessment

Yes, the best shape is to reuse the existing managed-service handoff rather than add another updater path. The open linked PR appears to own that repair, so this issue should remain open until that PR merges or maintainers choose a replacement path.
