## Summary

Keep open: this active draft/protected PR still has blocking correctness issues. The hidden VBS launcher detaches the long-running node process from scheduler lifecycle control, the changelog diff corrupts existing release-note text, and the real Windows node-login proof is still insufficient.

## What This Changes

The PR opts Windows node service installs into a generated hidden `.vbs` launcher, updates scheduled-task/startup fallback tests, and adds a changelog entry.

## Best Possible Solution

Keep the node-only hidden-launcher direction, but make the Windows task or OpenClaw lifecycle code continue to observe and stop the real node host, then land with focused lifecycle tests and redacted real Windows node-login proof.

## Reproduction Assessment

Yes for source reproduction: the linked report gives Windows 11 sign-in steps and evidence, docs define the node host as a background service, and current main schedules `node.cmd` directly. I did not perform a live Windows sign-in flow in this read-only review.

## Solution Assessment

No as written: a hidden launcher is the right narrow direction, but `Run(..., False)` detaches the node process from the task lifecycle and the branch corrupts the changelog. A safer fix keeps the task/process tracked or adds node-specific runtime and termination logic before opting node installs into hidden startup.
