## Summary

Keep open: the PR targets a valid Windows node-service bug, but the current hidden launcher detaches the node process from Task Scheduler, which appears to break status/stop/restart semantics, and the PR is still draft/protected with insufficient real node login proof.

## What This Changes

The PR opts Windows node installs into a generated hidden `.vbs` launcher for scheduled-task and Startup-folder startup, adds focused daemon tests, and records a changelog entry.

## Best Possible Solution

Keep the node-only hidden-launcher direction, but make Task Scheduler or OpenClaw lifecycle code continue to track and stop the long-running node process, then add Windows lifecycle coverage and real redacted node login proof before landing.

## Reproduction Assessment

Yes for source reproduction: the linked report gives Windows 11 sign-in steps, docs define the node host as a background service, and current main schedules `node.cmd` directly. I did not perform a live Windows sign-in flow in this read-only review.

## Solution Assessment

No as written: the hidden-launcher approach is the right narrow direction, but `Run(..., False)` detaches the child process from the scheduled task and undermines service lifecycle control. The safer fix is to keep the launcher/task tracked or add node-process-aware runtime and termination logic with regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep the hidden task attached to the node process:** `src/daemon/schtasks.ts:334`
  - body: The generated VBS uses `WScript.Shell.Run(..., 0, False)`, so the scheduled-task action exits immediately after launching `node.cmd`. For the node service this means Task Scheduler can report Ready/Last Result=0 while the node host keeps running, and the existing status/stop/restart code only talks to the completed task plus a gateway-listener fallback, so `openclaw node status` can report stopped and `openclaw node stop` may not terminate the node process. Keep the launcher waiting/tracked or add node-specific process tracking before opting node installs into it.
  - confidence: 0.9
