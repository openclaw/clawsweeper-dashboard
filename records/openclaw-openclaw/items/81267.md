## Summary

Keep open: this is a plausible Windows node service bug fix, but the patch still has a startup fallback compatibility regression and the PR body does not prove the successful Scheduled Task login path after the change.

## What This Changes

The PR makes Windows node installs opt into a hidden `.vbs` launcher for scheduled-task and Startup-folder launch paths, with daemon tests and a changelog entry.

## Best Possible Solution

Keep the hidden launcher approach, but preserve detection/removal of legacy `.cmd` Startup-folder entries and require redacted real Windows Scheduled Task proof before merge.

## Reproduction Assessment

Yes. The linked bug gives concrete Windows 11 steps and evidence, and current main source confirms the node Scheduled Task targets the generated `.cmd` script directly; I did not run a Windows sign-in path in this read-only checkout.

## Solution Assessment

No, not yet. A hidden launcher is the right direction, but this patch should keep compatibility with existing `.cmd` Startup-folder fallback installs and should prove the real scheduled-task login behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep removing legacy Startup .cmd entries:** `src/daemon/schtasks.ts:84-88`
  - body: Changing `resolveStartupEntryPath` to return only `.vbs` when the hidden flag is set breaks upgraded node installs that already fell back to the Startup folder on current main. Those installs created `OpenClaw Node.cmd`; after this PR every node manage/status path sets `OPENCLAW_WINDOWS_TASK_HIDDEN_LAUNCHER`, so `uninstall`, `stop`, `restart`, `status`, and runtime fallback only probe the new `.vbs` path and leave or miss the old login item. Please keep probing/removing the legacy `.cmd` entry while writing `.vbs` for new hidden installs.
  - confidence: 0.89
- **[P2] Quote the command passed to WScript.Run:** `src/daemon/schtasks.ts:323`
  - body: The generated VBS passes the raw script path as the full `Run` command string. If the state dir or profile path contains spaces, the hidden launcher command becomes ambiguous and can resolve the wrong executable before `node.cmd`; the existing `/TR` path handles this with explicit quoting, so the VBS launcher should embed quoted command text rather than only a VBScript string literal.
  - confidence: 0.74
