## Summary

Keep open: the PR targets a real Windows node service bug, but it currently has blocking correctness issues. The hidden launcher detaches the Scheduled Task from the node process, which breaks the existing status/stop/restart model, and the changelog edit accidentally corrupts existing UTF-8 text.

## What This Changes

The PR adds a node-only hidden Windows launcher that stages `node.cmd` plus `node.vbs` and points Scheduled Task/Startup fallback entries at the hidden launcher.

## Best Possible Solution

Preserve a hidden node launch path without detaching it from service lifecycle semantics, restore the changelog to clean UTF-8, and add proof that Windows node status/stop/restart still work after login.

## Reproduction Assessment

Yes. The reported root path is source-reproducible because current main stages the Windows node task as `node.cmd`, and the PR proof confirms no hidden launcher exists before the patch; this review did not run a live Windows sign-in cycle.

## Solution Assessment

No, not as written. A hidden launcher is a reasonable direction, but launching it with `bWaitOnReturn=False` detaches Task Scheduler from the node process and breaks the service management contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep the hidden launcher attached to the task:** `src/daemon/schtasks.ts:334`
  - body: The generated VBS runs `node.cmd` with `bWaitOnReturn=False`, so `wscript` exits immediately and Task Scheduler will report the task as completed while the node host continues detached. `openclaw node status` then goes through the existing scheduled-task runtime path, which only has a gateway-listener fallback for non-running tasks, and `stop`/`restart` also operate on the task rather than the detached node process. Please keep the hidden launcher tied to the node process or add node-specific lifecycle detection/control before switching `/TR` to the VBS file.
  - confidence: 0.91
- **[P2] Restore the changelog encoding:** `CHANGELOG.md:302`
  - body: The changelog diff re-encodes existing UTF-8 text into mojibake and adds a BOM at the start of the file. That corrupts published release notes unrelated to this Windows node fix, so the branch should restore the file and leave only the intended clean release-note entry, or let maintainers handle the changelog during landing.
  - confidence: 0.98
