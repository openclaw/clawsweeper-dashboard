## Label Justifications

- `P2`: Feishu interactive-card payload loss affects real plugin workflows but is scoped to one channel plugin.
- `impact:data-loss`: Structured form and selector values from Feishu callbacks are discarded before plugins or agents can use them.
- `impact:message-loss`: The synthetic channel message created from the card callback omits part of the inbound user interaction payload.

## Summary

Keep open: current main and v2026.5.20 still strip Feishu card-action sibling fields before agent dispatch, and open implementation PRs with closing links already target this issue while maintainers choose the canonical payload contract.

## What This Changes

The issue requests preserving Feishu card-action form_value, input_value, name, option, and options fields in the synthetic message delivered to agents while keeping simple button clicks compatible.

## Best Possible Solution

Land one canonical Feishu plugin patch that preserves selected sibling action fields from parser through decoder/fallback/handler dispatch, keeps legacy command handling compatible, adds regression coverage, includes redacted live Feishu proof, and then closes this issue after merge.

## Reproduction Assessment

Yes, source-level. A raw card.action.trigger payload with form/input/select sibling fields is normalized to only action.value and action.tag, then fallback dispatch serializes only action.value.

## Solution Assessment

No final implementation is on main yet. A handler-only change would be incomplete; the durable fix should start at parseFeishuCardActionEventPayload and align the event type, decoder/fallback behavior, handler dispatch, tests, and selected payload shape.
