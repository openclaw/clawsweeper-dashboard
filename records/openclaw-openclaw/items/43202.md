## Summary

Keep open: current main still only preserves `form_value`, `option`, and `options` in the Feishu card-action pipeline, while this report also asks for `input_value`, `name`, and other raw action sibling fields, and there are open implementation PRs with closing syntax that should be reviewed instead of closing the issue early.

## What This Changes

The issue requests that Feishu card-action callbacks pass structured action sibling fields such as `form_value`, `input_value`, `name`, `option`, and `options` through to the synthetic agent message while preserving simple button behavior.

## Best Possible Solution

Land or close one canonical Feishu plugin PR that preserves the agreed action fields at the parser boundary, carries them through decoder/fallback/handler dispatch, keeps legacy command handling compatible, adds regression coverage for the selected contract, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes, source-level. A raw Feishu `card.action.trigger` payload containing `input_value` or `name` is reconstructed without those fields in `parseFeishuCardActionEventPayload`, and the current tests only lock in `form_value`/`option`/`options`.

## Solution Assessment

Unclear as a final contract. The maintainable path is not another handler-only patch; it should define the canonical Feishu action payload shape at the parser/decoder boundary and review the linked PRs against that contract.
