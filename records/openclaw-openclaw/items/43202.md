## Label Justifications

- `impact:data-loss`: Submitted Feishu form, input, and selection values are dropped before the agent can consume them.
- `impact:message-loss`: The synthetic inbound message omits meaningful parts of the user's Feishu card interaction.

## Summary

Keep this open: current main and the latest release still drop Feishu card-action fields outside `action.value`, and the issue is paired with open conflict-blocked implementation PRs that need one maintainer-selected payload contract before landing.

## What This Changes

Pass Feishu card-action `form_value`, `input_value`, `name`, `option`, `options`, and similar action fields through to the synthetic agent message while preserving simple button behavior.

## Best Possible Solution

Land one Feishu plugin contract that preserves card-action interaction fields from the raw parser through decoder and handler, keeps existing button command behavior stable, and covers the live callback path with focused tests and redacted Feishu proof.

## Reproduction Assessment

Yes, at source level: the current live callback parser keeps only `action.value` and `action.tag`, and the downstream fallback derives agent text only from `action.value`. I did not run a live Feishu callback in this read-only review.

## Solution Assessment

No implementation is on main yet. The best fix is a parser-to-decoder-to-handler contract that preserves current command routing and structured-action validation, not closing the issue while linked PRs remain open and conflicting.
