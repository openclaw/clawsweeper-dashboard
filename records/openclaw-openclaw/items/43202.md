## Summary

Keep open: current main and the latest release still normalize Feishu card-action callbacks down to `action.value` and `action.tag`, so the requested form/input/select sibling fields are still dropped. The issue also has open paired implementation work with closing syntax, so it should remain open until that PR is landed, replaced, or closed.

## What This Changes

The issue requests Feishu card-action callbacks to pass `form_value`, `input_value`, `name`, `option`, `options`, and similar sibling action fields through to the agent instead of only `action.value`.

## Best Possible Solution

Review, refresh, and land or replace one canonical Feishu plugin patch that preserves the missing fields at the parser boundary, carries them through fallback/handler dispatch, adds regression coverage, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes, source-level. A raw `card.action.trigger` payload with sibling form/input/select fields is normalized to only `action.value` and `action.tag`, and fallback dispatch then serializes only `action.value`.

## Solution Assessment

No for the handler-only proposed fix. The durable solution should start at `parseFeishuCardActionEventPayload`, then align parser, event type, fallback serialization, handler dispatch, tests, and the chosen payload contract across the overlapping Feishu PRs.
