## Summary

Keep open: current main and v2026.6.6 still drop Feishu card-action fields outside `action.value`, and the linked implementation PRs are open/conflict-blocked rather than merged fixes.

## What This Changes

The issue requests passing Feishu card action fields such as `form_value`, `input_value`, `name`, `option`, and `options` through to the synthetic agent message instead of only using `action.value`.

## Best Possible Solution

Land one canonical Feishu plugin change that preserves raw interactive-card fields through the parser, decoder, and handler while retaining existing command routing and structured-action safeguards.

## Reproduction Assessment

Yes by source inspection: current main strips adjacent Feishu action fields in `parseFeishuCardActionEventPayload` and the fallback message path reads only `action.value`. I did not run a live Feishu callback because this review is read-only and no credentials are available.

## Solution Assessment

No implementation is on main; the right fix should be Feishu plugin-local but must define one command-compatible agent-facing payload contract before landing. A handler-only patch would be too narrow because current main also normalizes the raw callback in `monitor.account.ts`.
