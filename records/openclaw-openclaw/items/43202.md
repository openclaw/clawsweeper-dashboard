## Summary

Keep open. Current main still discards the requested Feishu action sibling fields in the parser/type/dispatch path, and the provided GitHub context shows open implementation candidates including same-author PR #43953 with closing syntax, so cleanup closure would be premature.

## What This Changes

Request Feishu card action callbacks to pass through `action.form_value`, `action.input_value`, `action.name`, `action.option`, and `action.options` to the synthetic agent message while preserving legacy button-only behavior.

## Best Possible Solution

Preserve the Feishu interactive fields from `parseFeishuCardActionEventPayload` through the typed event/decoder into the synthetic message content, emit the richer payload only when those fields exist, keep value-only button behavior unchanged, and add parser-to-handler regression coverage.

## Reproduction Assessment

Yes. A source-level reproduction is available: feed a `card.action.trigger` payload containing `action.value` plus `action.form_value` or `action.input_value` into the current parser; the returned event keeps only `action.value` and `action.tag`, so the handler cannot forward the extra fields.

## Solution Assessment

Yes, the narrow maintainable fix belongs in the Feishu plugin parser, event type, decoder/fallback, and handler path. A handler-only patch would be incomplete because live callbacks already lose the fields in `parseFeishuCardActionEventPayload`.
