## Summary

Keep this open: current main and the latest release still narrow Feishu card callbacks to `action.value`/`tag`, while the reporter's closing PR and an overlapping Feishu PR remain open and conflict-blocked. The remaining work is to choose and land one canonical typed payload contract, not to close the issue before a fix merges.

## What This Changes

The issue requests that Feishu card action callbacks pass `form_value`, `input_value`, `name`, `option`, `options`, and similar component fields through to the synthetic agent message while preserving simple button behavior.

## Best Possible Solution

Land one Feishu plugin fix that preserves the requested fields at the parser boundary, defines a bounded typed payload contract, keeps existing button-command behavior and validation safeguards, and includes focused tests plus redacted live Feishu proof.

## Reproduction Assessment

Yes. At source level, current main normalizes a card-action callback to only `action.value` and `action.tag`, then builds the synthetic message from `action.value`; this read-only review did not run a live Feishu callback.

## Solution Assessment

No final implementation is on main. The best fix is not an unbounded pass-through in the handler alone, but a parser-to-decoder-to-handler contract that preserves existing command routing and validation.
