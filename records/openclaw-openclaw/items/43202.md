## Label Justifications

- `P2`: The issue blocks structured Feishu card form and selector workflows but is limited to one bundled channel plugin.
- `impact:data-loss`: Submitted form, input, and selection values are discarded before agents or plugins can consume them.
- `impact:message-loss`: The synthetic inbound message omits meaningful parts of the user's Feishu card interaction payload.

## Summary

Keep this open: current main and v2026.6.6 still narrow Feishu card-action callbacks to `action.value`/`tag`, while the reporter's closing PR and an overlapping Feishu PR remain open and conflict-blocked. The remaining work is to pick one command-compatible payload contract and land it, not close this issue before a fix merges.

## What This Changes

Pass Feishu card-action `form_value`, `input_value`, `name`, `option`, and `options` fields through to the synthetic agent message instead of dropping them.

## Best Possible Solution

Land one Feishu plugin patch that preserves raw callback fields at the parser boundary, carries them through the decoder/handler without bypassing structured-action safeguards, preserves existing button-command behavior, and adds focused tests plus redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level. The production `card.action.trigger` path normalizes callbacks to `action.value`/`tag` only, then builds the synthetic message from `action.value`; this read-only review did not run a live Feishu callback.

## Solution Assessment

No implementation is on main yet. The best fix is not a handler-only pass-through, but a parser-to-decoder-to-handler contract that preserves current command routing and structured-action validation.
