## Label Justifications

- `impact:data-loss`: Submitted Feishu form, input, and selection values are dropped before the agent can consume them.
- `impact:message-loss`: The synthetic inbound message omits meaningful parts of the user's Feishu card interaction.

## Summary

Keep this open: current main and the latest release still drop Feishu card-action fields outside `action.value`, and open closing PRs already cover the same work but need maintainer selection of the agent-facing payload contract before one branch is repaired or landed.

## What This Changes

Pass Feishu card-action fields such as `form_value`, `input_value`, `name`, `option`, and `options` through to the synthetic agent message while preserving existing simple button behavior.

## Best Possible Solution

Land one canonical Feishu plugin change that preserves card-action interaction fields from the raw parser through decoder and handler, keeps existing button command behavior stable, and includes focused tests plus redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level: the current live callback parser keeps only `action.value` and `action.tag`, and the fallback message builder reads only `action.value`. I did not run a live Feishu callback in this read-only review.

## Solution Assessment

No implementation is on main yet. The best fix is a parser-to-decoder-to-handler contract that preserves current command routing and structured-action validation, not closing the issue while linked PRs remain open and conflicting.
