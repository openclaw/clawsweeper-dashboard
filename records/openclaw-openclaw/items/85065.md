## Label Justifications

- `P1`: The report describes an active regression that can block resumed Claude CLI agent/channel replies for real users on the latest release.
- `impact:message-loss`: The failure surfaces in Telegram/TUI replies and can suppress a normal agent response after idle resume.
- `impact:auth-provider`: The bug is in provider/model runtime routing between Anthropic, `claude-cli`, and embedded harness selection.

## Summary

Keep open: current main and the latest release still have a source-level path where a configured CLI runtime alias can reach embedded harness selection and throw `MissingAgentHarnessError`, while the docs say `claude-cli` is a CLI backend alias, not an embedded harness id.

## What This Changes

The issue requests that `claude-cli` and other CLI runtime aliases avoid hard missing-harness failures when registration or resume timing leaves no matching embedded harness registered.

## Best Possible Solution

Teach the harness selection/resume boundary that CLI runtime aliases are PI-owned CLI backends, not embedded plugin harness ids, and add regression coverage for `claude-cli` plus at least one other CLI alias without weakening fail-closed plugin-runtime behavior.

## Reproduction Assessment

No live current-main reproduction was run. Source inspection gives a high-signal path: an unregistered `claude-cli` runtime can still reach `selectAgentHarnessDecision()` and throw, despite the docs saying CLI aliases must not be treated as embedded harness ids.

## Solution Assessment

The requested direction is valid, but the safest fix is narrower than a blanket plugin fallback: route CLI aliases through the existing CLI/PI boundary and keep strict missing-harness errors for real embedded plugin runtimes.
