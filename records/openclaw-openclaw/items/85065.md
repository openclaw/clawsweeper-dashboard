## Label Justifications

- `P1`: The report describes an active runtime routing regression that can block resumed CLI-backed agent/channel replies for real users.
- `impact:message-loss`: The failure surfaces as a channel/TUI warning instead of a normal agent response after a resumed turn.
- `impact:session-state`: The reported trigger is idle-session resume, where stored CLI session state and runtime selection can drift.
- `impact:auth-provider`: The bug is in provider/model runtime routing between Anthropic, `claude-cli`, and embedded harness selection.

## Summary

Keep open: current `main` and `v2026.5.20` still have the source-level `claude-cli` missing-harness path, but an open focused fix PR appears to cover the same selector fallback and should be reviewed or landed before this issue is closed.

## What This Changes

The issue reports that configured CLI runtime aliases such as `claude-cli` can reach embedded harness selection during idle-session resume and throw `MissingAgentHarnessError` when the plugin harness registry snapshot is not populated.

## Best Possible Solution

Review and land the narrow selector fallback in https://github.com/openclaw/openclaw/pull/84966 if CI and maintainer review pass, then close this issue with the merge commit and release proof if it also covers the idle-resume path.

## Reproduction Assessment

No live current-main reproduction was run for the exact idle-resume race. Source inspection shows a high-signal path where a valid CLI alias can still reach `selectAgentHarness` and throw, and related live proof in https://github.com/openclaw/openclaw/pull/84966 confirms the same selector failure class for `claude-cli`.

## Solution Assessment

Yes, the best fix direction is the narrow one: distinguish known CLI runtime aliases from embedded plugin harness ids at selector/helper boundaries, while preserving strict missing-harness errors for real plugin runtimes.
