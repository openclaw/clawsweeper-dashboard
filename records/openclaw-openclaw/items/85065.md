## Summary

Keep open: current main still has a source-level path where an explicit CLI runtime alias such as `claude-cli` can be treated as a missing embedded harness, and the exact idle-resume trigger has not been fixed or superseded by a merged change. Related open reports cover compaction and upgrade variants, but they do not fully replace this direct resume/selection-boundary report.

## What This Changes

The issue reports that `claude-cli` can be selected as an embedded agent harness during idle session resume and throw `MissingAgentHarnessError` instead of staying on the CLI runtime path.

## Best Possible Solution

Make all direct harness-selection preflight or resume call sites treat CLI runtime aliases as CLI-owned runtime policy, while preserving strict missing-harness errors for real embedded plugin runtimes.

## Reproduction Assessment

Not for the exact idle-resume race in a live setup. Source inspection does show a clear current-main path where `claude-cli` remains a valid CLI alias but direct harness selection still throws when it is treated as an embedded harness id.

## Solution Assessment

Yes, with a narrower boundary than a blanket fallback. The maintainable fix is to route CLI aliases through the existing CLI/PI-aware seams and keep missing-harness failures for true plugin runtimes.
