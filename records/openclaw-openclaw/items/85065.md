## Label Justifications

- `P1`: The original report described an urgent runtime regression that could break resumed agent/channel workflows for real `claude-cli` users.
- `impact:session-state`: The failure was tied to resumed or compacting live CLI sessions and session harness state.
- `impact:message-loss`: The error could replace or block channel replies instead of delivering model output.
- `impact:auth-provider`: The affected path is provider/runtime harness routing for the `claude-cli` Anthropic CLI backend.

## Summary

Current main and shipped releases already cover the reported `claude-cli` CLI-runtime alias failure paths: harness selection routes provider-owned CLI aliases to the OpenClaw harness instead of throwing, and compaction/model-fallback preflight bypasses CLI runtime aliases before the missing-harness check.

## What This Changes

The issue reports `claude-cli` harness selection throwing when registration is unavailable during session resume or compaction, and asks for a CLI-runtime fallback instead of a hard `MissingAgentHarnessError`.

## Best Possible Solution

Close this issue as fixed in `v2026.6.1` and leave the broader registry reload race to https://github.com/openclaw/openclaw/issues/86342.

## Reproduction Assessment

No current-main failure path remains: source inspection shows CLI alias selection, compaction, and model-fallback preflight all bypass the missing-harness throw. The old 2026.5.x failure is source-backed by the merged PR diffs and reporter logs.

## Solution Assessment

Yes: the landed implementation keeps real plugin harnesses fail-closed while treating provider-owned CLI runtime aliases as OpenClaw/CLI dispatch placeholders, which is narrower than adding broad missing-harness fallback behavior.
