## Summary

Keep open: current main has related iOS Talk pieces and Gateway Talk relay APIs, but the requested iOS mode that records on-device audio and uses the Gateway-configured STT plus TTS pipeline is not fully implemented as described.

## What This Changes

The issue requests iOS Talk Mode support that records audio on iPhone, routes STT through the Gateway, sends the transcript to the agent, and streams Gateway-generated TTS audio back for playback.

## Best Possible Solution

Define one first-party iOS Gateway voice path that reuses `talk.catalog`, `talk.session.*`, and `talk.speak` where possible, then document how it maps to configured STT/TTS providers.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request; source inspection shows the current native iOS Talk path uses local speech recognition, while Gateway transcription relay exists separately and is not wired into that native path.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the product path; the safest direction is to compose the existing Gateway Talk relay/session APIs rather than add a parallel iOS-only audio protocol.
