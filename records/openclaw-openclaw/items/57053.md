## Summary

Keep open. Current main still exposes `session_status` as rendered status text plus minimal structured metadata, not a stable remaining time/token budget contract; the related open token-usage, sub-agent-capacity, and session-cap issues are adjacent but do not fully supersede this request.

## What This Changes

Request to expose remaining session time/token budget as structured data from `session_status` or a new lightweight `session_budget` tool.

## Best Possible Solution

Define an additive documented budget contract backed by authoritative time and token accounting, add focused session-status/tool tests, and update the session-tool and token-use docs so skill authors can rely on it.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection give a high-confidence current-behavior check: `session_status` returns rendered `statusText` and does not expose the requested structured budget fields.

## Solution Assessment

Unclear until maintainers choose the public schema and semantics. Structured budget data is a maintainable direction, but the exact time/token sources and visibility rules should be settled before implementation.

## Risks / Open Questions

- Maintainers still need to choose the public surface: `session_status.details.budget`, a dedicated `session_budget` tool, or coordination with adjacent usage/capacity APIs.
- `remaining_tokens` needs a precise source definition so callers do not confuse context-window headroom, transcript usage totals, provider quota windows, and configured hard caps.
- `remaining_seconds` needs an authoritative current-run deadline or timeout source; sub-agent run timeouts and caller session lifetime are not currently the same contract.
- Any structured budget payload should preserve existing session visibility and privacy boundaries before exposing usage, cost, or provider quota metadata.
