## Summary

Keep open. Current main still exposes `session_status` as rendered status text plus minimal metadata, while the requested structured remaining time/token budget contract is absent and the related open usage/capacity issues only cover adjacent parts of the problem.

## What This Changes

Expose remaining session budget values, such as remaining seconds/tokens and elapsed/timeout data, as structured output from `session_status` or a new `session_budget` tool.

## Best Possible Solution

Define an additive documented budget contract backed by authoritative time and token accounting, then cover it with focused session-status/tool tests and docs so skill authors can rely on it.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs inspection give a high-confidence current-behavior check: `session_status` returns rendered `statusText` and does not expose the requested structured budget fields.

## Solution Assessment

Unclear until maintainers choose the public API shape and semantics. Structured budget data is a maintainable direction, but the exact fields, sources, and visibility rules should be settled before implementation.

## Risks / Open Questions

- The public schema still needs an authoritative definition for `remaining_tokens`, since context-window headroom, transcript totals, provider quota windows, and configured caps are different concepts.
- `remaining_seconds` needs a clear source of truth for the current run deadline or timeout; caller session lifetime and sub-agent run timeout are not currently one shared contract.
- Any structured budget payload must preserve existing session visibility, sandbox, and privacy boundaries before exposing usage, cost, or provider quota metadata.
