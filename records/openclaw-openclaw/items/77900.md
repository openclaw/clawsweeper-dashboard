## Summary

Keep open: current main still has a Telegram Bot API transport fallback cascade and polling recovery, but no host-level breaker/cooldown that would stop repeated failures against an unhealthy Telegram endpoint.

## What This Changes

Add backoff or circuit-breaker behavior for Telegram inbound `getUpdates` and transport fallback failures during unreachable-network windows.

## Best Possible Solution

Add a Telegram-owned host health cooldown around Bot API transport fallback and polling/media fetch paths, with regression tests; keep any cross-channel framework extraction aligned with #41899 after the Telegram failure mode is bounded.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show the relevant current-main path: Telegram fetch can escalate to a sticky pinned fallback IP and polling has no host-level breaker to suppress repeated unhealthy endpoint attempts.

## Solution Assessment

Unclear that the full configurable shared breaker proposed in the report is the best first patch. A narrower Telegram-local breaker/cooldown with fixed safe defaults is the most maintainable initial fix, with shared framework work left to #41899 if multiple channels need the same API.
