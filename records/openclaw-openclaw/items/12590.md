## Summary

Keep open. Current main still matches the reported counter-synchronization path, and open PR #51421 already targets this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that memoryFlush runs on alternating auto-compaction cycles because the flush marker is persisted with the post-increment compaction count.

## Best Possible Solution

Land #51421 or an equivalent focused patch that preserves the pre-increment flush marker after flush-triggered compaction and updates regression coverage for consecutive eligible auto-compaction cycles.

## Reproduction Assessment

Yes, at source level. On current main, a flush-triggered compaction can persist both counters to the same post-increment value, and the next `shouldRunMemoryFlush` call skips because the counters match.

## Solution Assessment

Yes. The narrowest maintainable fix is to stop recording the post-increment count as the flush marker, then update the runner/gate expectations and consecutive-cycle coverage.
