## Summary

Keep open. Current main and latest v2026.5.2 still persist the post-increment compaction count as `memoryFlushCompactionCount`, while the flush gate skips equal counters; open PR #51421 already targets this bug with closing syntax.

## What This Changes

The issue asks for `memoryFlush` to run before every eligible auto-compaction instead of being skipped after flush-triggered compaction synchronizes the counters.

## Best Possible Solution

Land #51421 or an equivalent focused patch that preserves the pre-increment flush marker after flush-triggered compaction, updates the synchronized-counter test expectation, and adds coverage for consecutive eligible auto-compaction cycles.

## Reproduction Assessment

Yes. Source inspection on current main shows a flush-triggered compaction can persist equal `compactionCount` and `memoryFlushCompactionCount`, and the next eligible cycle then hits the equal-counter gate and skips the flush.

## Solution Assessment

Yes. The narrow maintainable fix is to stop recording the post-increment `nextCount` as the flush marker, with regression coverage for consecutive eligible auto-compaction cycles and a coordination check against #47143.
