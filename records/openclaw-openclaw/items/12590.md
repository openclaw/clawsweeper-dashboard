## Summary

Keep open. Current main and the latest release still contain the reported counter-synchronization path, and open PR #51421 targets the issue with closing syntax.

## What This Changes

The issue reports that `memoryFlush` only fires on alternating auto-compaction cycles because the flush marker is persisted with the post-increment compaction count.

## Best Possible Solution

Land #51421 or an equivalent focused patch that preserves the pre-increment flush marker after flush-triggered compaction and adds consecutive eligible auto-compaction coverage.

## Reproduction Assessment

Yes, at source level. On current main, a flush-triggered compaction can persist both counters to the same post-increment value, and the next `shouldRunMemoryFlush` call skips because the counters match.

## Solution Assessment

Yes. The narrowest maintainable fix is to stop recording the post-increment count as the flush marker, then update the runner/gate expectations and regression coverage.
