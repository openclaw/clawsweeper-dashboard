## Summary

Keep open. Current main still contains the counter synchronization path that can make `memoryFlush` skip the next eligible auto-compaction, and open PR #51421 already targets this bug with closing syntax, so the issue should remain open until that PR or an equivalent patch lands.

## What This Changes

The issue reports that `memoryFlush` records the post-increment compaction count after a flush-triggered compaction, causing later eligible auto-compactions to skip the flush instead of running once per cycle.

## Best Possible Solution

Land #51421 or an equivalent focused patch that preserves the pre-increment flush marker after flush-triggered compaction, updates the synchronized-counter test expectation, and adds regression coverage for consecutive eligible auto-compaction cycles.

## Reproduction Assessment

Yes. The source path is high-confidence: a flush-triggered compaction can persist equal `compactionCount` and `memoryFlushCompactionCount`, and the next eligible cycle then hits the equal-counter gate and skips the flush.

## Solution Assessment

Yes. The narrow maintainable fix is to stop recording the post-increment `nextCount` as the flush marker, with coverage for consecutive eligible auto-compaction cycles and a coordination check against #47143.
