## Summary

Keep open. Current main and the latest release still contain the reported counter-synchronization path, and open PR #51421 targets this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that configured memoryFlush run before every eligible auto-compaction cycle instead of being skipped when compaction and flush counters become synchronized.

## Best Possible Solution

Land #51421 or an equivalent focused patch that preserves the pre-increment flush marker after flush-triggered compaction and adds regression coverage for consecutive eligible auto-compaction cycles.

## Reproduction Assessment

Yes, at source level. Current main can persist both counters to the same post-increment value after a flush-triggered compaction, and the next shouldRunMemoryFlush call skips because the counters match.

## Solution Assessment

Yes. The narrowest maintainable fix is to stop recording the post-increment compaction count as the flush marker, then update the runner/gate expectations and consecutive-cycle coverage.
