## Summary

Keep open. Current main still has the reported counter synchronization path, and open PR #51421 already targets this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for `memoryFlush` to run before every eligible auto-compaction cycle instead of being skipped after the flush marker is synchronized with the post-increment compaction count.

## Best Possible Solution

Land #51421 or an equivalent focused patch that preserves the pre-increment flush marker, updates the synchronized-counter test expectation, and adds coverage for consecutive eligible auto-compaction cycles.

## Reproduction Assessment

Yes. The source path is high-confidence: a flush-triggered compaction can persist equal `compactionCount` and `memoryFlushCompactionCount`, and the next eligible cycle then hits the equal-counter gate and skips the flush.

## Solution Assessment

Yes. The narrow maintainable solution is to stop recording the post-increment `nextCount` as the flush marker, with regression coverage for consecutive eligible compactions.
