## Summary

Keep open: the reported no-config premise is contradicted by current source/docs, but the logs still describe a real session-state/message-loss failure that overlaps active long-run recovery and session-lock work rather than a safely closeable duplicate.

## What This Changes

The issue asks OpenClaw to avoid failing legitimate long local vLLM model calls that cross the diagnostic stall threshold, preferably with configurable thresholds, graceful user-visible status, and recovery after slow calls complete.

## Best Possible Solution

Treat this as a session-recovery policy follow-up: document/use the existing global diagnostics thresholds, review the open session-takeover and WhatsApp delivery PRs, then decide whether local-provider model calls need a provider-aware progress/timeout contract.

## Reproduction Assessment

Yes at source level for the watchdog path: current tests and code show stale active model calls can be classified as stalled and become abort-eligible at diagnostics.stuckSessionAbortMs. No live vLLM run reproduced the exact EmbeddedAttemptSessionTakeoverError timing from the report.

## Solution Assessment

Unclear. The existing global diagnostics knobs already solve part of the request, while the remaining per-provider timeout/status/recovery behavior needs a maintainer decision and coordination with the open session-takeover and WhatsApp delivery work.
