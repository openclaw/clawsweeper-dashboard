## Summary

Keep open: current main and v2026.6.9 have the diagnostic knobs and stream-progress partial fixes, but they still deliberately classify silent vLLM-shaped model calls as stalled and request active abort after `diagnostics.stuckSessionAbortMs`. Related active-run recovery work overlaps, especially https://github.com/openclaw/openclaw/issues/88870 and https://github.com/openclaw/openclaw/pull/88925, but it does not clearly replace the remaining local silent-model policy decision.

## What This Changes

The issue reports that diagnostic stall detection can abort legitimate long local vLLM model calls, causing failed sessions and lost channel replies.

## Best Possible Solution

Keep the existing threshold and stream-progress repairs, then have maintainers decide whether silent local model calls should be waited out, require explicit operator tuning, or use a different recovery path coordinated with the active-run abort work.

## Reproduction Assessment

Yes, at source level: current tests and classifier code reproduce the remaining silent-model-call abort boundary after `diagnostics.stuckSessionAbortMs`. No live vLLM run was attempted in this read-only review.

## Solution Assessment

Unclear. The current configuration and stream-progress fixes are good partial repairs, but the remaining behavior is a maintainer tradeoff between preserving slow silent model calls and unwedging genuinely stale sessions.
