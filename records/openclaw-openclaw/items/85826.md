## Summary

Keep open: current main and the reported release already expose the requested diagnostic timeout knobs, but source and tests still show long silent model calls can be classified as stalled and become active-abort recovery candidates, so the remaining message-loss/session-failure path needs maintainer recovery-policy review rather than cleanup closure.

## What This Changes

The issue reports that long local vLLM model calls can be classified as stalled and eventually fail the session, losing user-visible replies.

## Best Possible Solution

Keep the existing diagnostics and provider timeout knobs, then decide a provider-aware session recovery policy for long silent local model calls before changing active-abort or user-visible still-working behavior.

## Reproduction Assessment

Yes at source level, but not as a live vLLM run. Current code and tests show active model calls can be classified as stalled and become active-abort candidates after the abort threshold; the exact takeover timing from the report was not reproduced here.

## Solution Assessment

Unclear. Adding another timeout knob is not the best first fix because diagnostics and provider timeout controls already exist; the maintainable path is a deliberate recovery policy for long silent local model calls.
