## Summary

Keep open: current main and the latest release already have diagnostics and provider timeout knobs, but source and tests still show long silent active model calls can be classified as stalled and become active-abort recovery candidates. The remaining work is session-recovery policy, not conservative cleanup closure.

## What This Changes

The issue asks for configurable or more graceful handling of long local-vLLM model calls that can be classified as stalled and fail the session before a channel reply is delivered.

## Best Possible Solution

Keep the existing diagnostics and provider timeout knobs, then decide a coherent recovery policy for long silent local model calls that preserves slow-but-valid runs, prevents session-file races, and surfaces a channel-visible outcome when recovery aborts work.

## Reproduction Assessment

Yes at source level, but not as a live vLLM run. Current code and tests show active model calls can be classified as stalled after the diagnostics threshold and can become active-abort candidates after the abort threshold; the exact takeover timing from the report was not reproduced here.

## Solution Assessment

Unclear. A new hard-coded or duplicate timeout knob is not the best first fix because global diagnostics and provider timeout controls already exist; the maintainable path is a deliberate recovery policy for long silent local model calls.
