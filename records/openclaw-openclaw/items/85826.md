## Summary

Keep open: current main and the reported release already have configurable diagnostics/provider timeout knobs, but source still shows long silent active model calls can become stalled and active-abort eligible, so the remaining session-failure/message-loss behavior needs maintainer recovery-policy review.

## What This Changes

The issue reports that long local vLLM model calls can be classified as stalled and later abort an active session, causing failed sessions and lost channel replies.

## Best Possible Solution

Keep the existing diagnostics and provider-timeout knobs, then decide a focused recovery policy for long silent local model calls that avoids losing channel replies while still clearing genuinely orphaned runs.

## Reproduction Assessment

Yes at source level, but not as a live vLLM run. Current code and tests show active model calls can be classified as stalled after the diagnostics threshold and can become active-abort candidates after the abort threshold; the exact takeover timing from the report was not reproduced here.

## Solution Assessment

Unclear. A new duplicate timeout knob is not the best first fix because global diagnostics and provider timeout controls already exist; the maintainable path is a deliberate recovery policy for long silent local model calls.
