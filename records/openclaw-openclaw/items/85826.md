## Summary

Keep open: current main and the reported release already have global diagnostic and provider timeout knobs, so the no-config premise is partly contradicted, but source and tests still show long silent model calls can be classified as stalled and become active-abort recovery candidates. The reported takeover and message-loss path needs maintainer session-recovery policy review rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to avoid killing legitimate long local vLLM model calls and to support configurable thresholds, visible progress, and recovery instead of failed sessions.

## Best Possible Solution

Keep this as a session-recovery policy follow-up: preserve the existing diagnostics and provider timeout controls, then decide how long silent local model calls should report progress, avoid unsafe active aborts, and surface user-visible still-working or timeout state.

## Reproduction Assessment

Yes at source level, but not as a live vLLM run. Current main classifies active model calls with stale progress as stalled_agent_run and can request allowActiveAbort after diagnostics.stuckSessionAbortMs; the specific vLLM timing and takeover sequence was not reproduced.

## Solution Assessment

Unclear. A new hard-coded or duplicate timeout knob is not the best first fix because global diagnostics and provider request timeout controls already exist; the maintainable path is a deliberate recovery policy for long silent local model calls.
