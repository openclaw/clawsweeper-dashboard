## Summary

Keep open: the hard-coded/no-config premise is partly wrong because current main and v2026.5.20 expose diagnostics and provider timeout knobs, but source and tests still show silent model calls can be classified as stalled and abort-drained, so the remaining message/session-loss behavior needs maintainer recovery-policy review.

## What This Changes

The issue asks OpenClaw to stop long local vLLM model calls from being misclassified or aborted by stall detection and to provide tunable thresholds, visible still-working behavior, and recovery without losing channel replies.

## Best Possible Solution

Keep the existing global knobs documented, then decide a provider-aware recovery policy that distinguishes slow-but-healthy local model calls from orphaned work, preserves channel delivery, and is proven with focused source tests plus a live vLLM-compatible reproduction.

## Reproduction Assessment

Yes at source level, but not as a live vLLM run. Current source and tests show model_call work with no progress becomes stalled and can request active-abort after diagnostics.stuckSessionAbortMs; the exact takeover timing and channel loss were not reproduced here.

## Solution Assessment

Unclear. Existing diagnostics and provider timeout knobs cover part of the request, so the maintainable path is a deliberate recovery policy for long silent local model calls rather than another unconditional timeout knob.
