## Summary

Keep open: current main has partially addressed the report with configurable diagnostics, stream-progress tracking, and a main-only silent-model classification improvement, but silent local model calls are still intentionally active-abort eligible after `diagnostics.stuckSessionAbortMs`, and the latest release still has the older warning-threshold stalled classification.

## What This Changes

The issue reports long local vLLM model calls being classified as stalled and eventually failing sessions because diagnostic stall thresholds and recovery behavior are not sufficient for slow or silent model work.

## Best Possible Solution

Keep this issue open as the canonical local-model stall/recovery policy item, preserve the current stream-progress and configuration improvements, and have maintainers decide the remaining silent-model abort behavior alongside the session/transcript fence work.

## Reproduction Assessment

Yes at source level for the remaining boundary, but not as a live current-main vLLM run. Current tests show streamed calls stay fresh while silent local model calls still become `session.stalled` and request active abort after `diagnostics.stuckSessionAbortMs`.

## Solution Assessment

Unclear. The current configuration and stream-progress fixes are the right partial path, but the best remaining fix requires maintainer judgment on whether silent/non-streaming model calls should be recovered by abort, waited out longer, or handled through a different session-recovery policy.
