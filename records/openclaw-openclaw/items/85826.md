## Summary

Keep open: current main and the latest release already contradict the “no configuration knob” part, but source and tests still show long silent active model calls can be classified as stalled and become active-abort recovery candidates, so the remaining message-loss/session-failure behavior needs maintainer recovery-policy review rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to stop treating legitimate long local vLLM model calls as stalled sessions that can fail the run and drop channel replies.

## Best Possible Solution

Keep the existing global timeout knobs as the documented mitigation, then make a maintainer decision on provider-aware progress, active-abort limits, and user-visible still-working/recovery semantics for long silent local model calls.

## Reproduction Assessment

Yes at source level, but not as a live vLLM run. Current source and tests show active model calls can be classified as stalled after the diagnostic threshold and can become active-abort recovery candidates after `diagnostics.stuckSessionAbortMs`; the exact takeover timing from the reported hardware/model was not replayed.

## Solution Assessment

Unclear. A new duplicate stall knob is not the best first fix because global diagnostics thresholds and provider timeout controls already exist; the maintainable path is a deliberate session-recovery policy for long silent local model calls.
