## Summary

Keep open: current source and the reported release already provide diagnostics timeout knobs, but long silent model calls can still become stalled and active-abort eligible, so the remaining takeover/message-loss behavior needs maintainer session-recovery policy review.

## What This Changes

The issue asks OpenClaw to avoid failing long local vLLM model calls as stalled sessions, preserve/deliver user-visible replies, and recover gracefully after slow model calls.

## Best Possible Solution

Treat this as a session-recovery policy follow-up: document/use the existing diagnostics and provider timeout knobs now, then decide whether long local model calls need provider-aware progress, different active-abort semantics, or channel-visible still-working behavior.

## Reproduction Assessment

Yes at source level, but not as a live vLLM run. Current code and tests show active model calls can be classified as stalled after the diagnostics threshold and can become active-abort candidates after the abort threshold; the exact takeover timing from the report was not reproduced here.

## Solution Assessment

Unclear. A duplicate timeout field is not the best first fix because global diagnostics, provider timeout, and agent run timeout controls already exist; the maintainable fix needs an explicit decision on long silent local-model progress and active-abort behavior.
