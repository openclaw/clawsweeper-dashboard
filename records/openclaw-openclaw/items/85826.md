## Summary

Keep open: current main and v2026.6.9 have partially addressed the report with documented diagnostic thresholds and stream-progress tracking, but silent local vLLM-shaped model calls are still intentionally classified as stalled and actively aborted after `diagnostics.stuckSessionAbortMs`.

## What This Changes

The issue reports that diagnostic stall detection can classify legitimate long local vLLM model calls as stalled and eventually fail the session with lost channel output.

## Best Possible Solution

Keep this issue open as the canonical local-model stall/recovery policy item, preserve the existing config and stream-progress fixes, and have maintainers decide the remaining silent-model abort behavior alongside the session/transcript fence work.

## Reproduction Assessment

Yes at source level for the remaining boundary: current tests and classifier code show silent vLLM-shaped model calls still become stalled and request active abort at `diagnostics.stuckSessionAbortMs`. No live current-main vLLM run was attempted in this read-only review.

## Solution Assessment

Unclear. The current configuration and stream-progress fixes are good partial repairs, but the best remaining fix requires maintainer judgment on whether silent/non-streaming model calls should be recovered by abort, waited out longer, or handled through a different recovery policy.
