## Label Justifications

- `P2`: This is a normal-priority Feishu delivery reliability bug with a clear plugin-owned surface and limited channel-specific blast radius.
- `impact:message-loss`: The reported rate-limit failures can drop outbound Feishu messages after the agent has generated a reply.

## Summary

Keep open. Current main still treats Feishu message delivery rate-limit failures as terminal for text/card, media, and initial streaming-card sends, the latest shipped tag has the same behavior, and the related pacing, media-idempotency, and delivery-notification issues track adjacent but distinct work.

## What This Changes

Add bounded retry/backoff handling for Feishu outbound API rate-limit failures such as 2200, 11232, and 429 so failed sends are not silently lost.

## Best Possible Solution

Add a Feishu-owned bounded retry/backoff helper for explicit retryable delivery failures, apply it consistently to create/reply sends, and keep proactive pacing, media idempotency, and user notification contracts on their existing related trackers unless maintainers merge the scopes.

## Reproduction Assessment

Yes, at source level: mock Feishu client.im.message.create or reply to throw a retryable SDK error or return a retryable nonzero code, and current main makes one delivery attempt before throwing. I did not run a live Feishu rate-limit scenario in this read-only review.

## Solution Assessment

Yes. A narrow Feishu-plugin retry wrapper around outbound delivery sends is the maintainable fix; it should reuse or factor retryable classification and avoid a core retry layer or new public config option.
