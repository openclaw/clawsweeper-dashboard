## Summary

Keep open. Current main and the latest shipped release still treat Feishu outbound delivery rate-limit failures as terminal for text/card, media, and initial streaming-card sends; existing backoff classification is limited to typing indicators, and the related pacing/idempotency reports track separate work.

## What This Changes

The issue asks the Feishu plugin to retry retryable outbound message send failures caused by Feishu API rate-limit or flow-control responses.

## Best Possible Solution

Add a Feishu-owned bounded retry/backoff helper for explicit retryable delivery failures, apply it consistently to create/reply sends, and keep proactive pacing plus media idempotency on their separate trackers.

## Reproduction Assessment

Yes, at source level: mock Feishu client.im.message.create or reply to throw, or return a retryable nonzero code, and current main makes one delivery attempt before throwing. I did not run a live Feishu rate-limit scenario in this read-only review.

## Solution Assessment

Yes. A narrow Feishu-plugin retry wrapper around outbound delivery sends is the maintainable fix; it should reuse or factor retryable classification and avoid a core retry layer or new public config option.
