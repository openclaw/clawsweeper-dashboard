## Summary

Keep open. Current main still handles Feishu delivery sends as one-shot message create/reply calls for text/cards, media, reply fallback, and initial streaming cards; existing backoff classification is scoped to typing reactions and the related pacing and duplicate-media issues remain distinct rather than superseding this retry bug.

## What This Changes

The issue requests bounded retry/backoff handling for Feishu outbound message sends when Feishu returns rate-limit or flow-control errors.

## Best Possible Solution

Add a Feishu-plugin-owned bounded retry/backoff helper for outbound delivery sends, share retryable error classification with typing where practical, and apply it consistently without adding a core retry layer or public config option.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock Feishu message create/reply to throw or return a retryable rate-limit response; current main performs one delivery call and throws without a second attempt.

## Solution Assessment

Yes. A narrow Feishu-owned retry wrapper is the best direction because Feishu retryability, response shapes, and duplicate-delivery risk are plugin-specific; a generic core retry layer would be broader and riskier.
