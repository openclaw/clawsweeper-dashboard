## Summary

Keep open. Current main still treats Feishu delivery rate-limit or flow-control failures as terminal across text/card, media, and initial streaming-card sends; the existing retryable/backoff classification is only wired to typing indicators, and the related Feishu pacing and duplicate-media reports track different remaining work.

## What This Changes

Add bounded retry/backoff for Feishu outbound message sends when Feishu returns retryable rate-limit or flow-control errors.

## Best Possible Solution

Add a Feishu-plugin-owned bounded retry/backoff helper for explicit retryable delivery failures, apply it consistently to create/reply sends, and leave proactive pacing or media idempotency to their separate trackers.

## Reproduction Assessment

Yes, at source level: mock Feishu client.im.message.create/reply to throw or return a retryable rate-limit response and current main makes one delivery attempt before throwing. I did not run tests because this review is read-only and test commands can create cache artifacts.

## Solution Assessment

Yes. A narrow Feishu-owned retry wrapper around outbound delivery sends is the maintainable fix; it should reuse or factor retryable classification and avoid a core retry layer or public config option.
