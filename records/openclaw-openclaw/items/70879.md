## Label Justifications

- `P2`: This is a normal-priority Feishu delivery reliability bug with a clear plugin-owned surface and limited channel-specific blast radius.
- `impact:message-loss`: The reported rate-limit failures can cause outbound Feishu messages to be dropped after a terminal send failure.

## Summary

Keep open. Current main and the latest shipped release still treat Feishu delivery-side rate-limit failures as terminal for text/card, media, and initial streaming-card sends; the only existing Feishu backoff classifier is wired to typing reactions, not outbound delivery.

## What This Changes

The issue asks the Feishu plugin to retry retryable API rate-limit send failures with bounded exponential backoff instead of dropping the outbound message after one failed attempt.

## Best Possible Solution

Add a Feishu-owned bounded retry/backoff helper for explicit retryable delivery failures, apply it consistently to message create/reply sends, and keep proactive pacing, media idempotency, and user notification behavior on their separate tracks unless the same narrow helper needs shared tests.

## Reproduction Assessment

Yes, at source level: mock Feishu client.im.message.create or reply to throw a retryable SDK error, or return a retryable nonzero code, and current main makes one delivery attempt before throwing. I did not run a live Feishu rate-limit scenario in this read-only review.

## Solution Assessment

Yes. A narrow Feishu-plugin retry wrapper around outbound delivery sends is the maintainable repair; it should reuse or factor retryable classification and avoid a core retry layer or new public config option.
