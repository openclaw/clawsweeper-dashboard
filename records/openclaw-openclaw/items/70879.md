## Summary

Keep open. Current main and v2026.5.20 still make Feishu text/card, media, and initial streaming-card delivery sends as one-shot calls; the existing rate-limit/backoff classifier is scoped to typing indicators, and there is no closing PR for this delivery retry gap.

## What This Changes

The issue asks for Feishu outbound message sends to retry retryable API rate-limit failures such as 2200, 11232, and 429 instead of treating them as terminal delivery failures.

## Best Possible Solution

Add a Feishu-owned bounded retry/backoff helper for explicit retryable delivery failures, apply it consistently to create/reply sends, and keep proactive pacing, duplicate-media idempotency, and broad failure notification on their separate trackers.

## Reproduction Assessment

Yes at source level: mock Feishu client.im.message.create or reply to throw, or return a retryable nonzero code, and current main makes one delivery attempt before throwing. I did not run a live Feishu rate-limit scenario in this read-only review.

## Solution Assessment

Yes. A narrow Feishu-plugin retry wrapper around outbound delivery sends is the maintainable fix; it should reuse or factor retryable classification and avoid a core retry layer or new public config option.
