## Summary

Keep open. Current main still sends Feishu delivery create/reply calls as one-shot operations for text, cards, media, and initial streaming cards; the existing backoff handling is scoped to typing indicators, and the related pacing and duplicate-media reports track different remaining work.

## What This Changes

The issue requests bounded retry/backoff for Feishu message delivery when API rate-limit or flow-control responses are returned.

## Best Possible Solution

Add a Feishu-plugin-owned bounded retry/backoff helper for outbound message delivery, reuse or factor retryable classification where practical, and apply it consistently without adding a core retry layer or public config option.

## Reproduction Assessment

Yes, at source level: mock Feishu client.im.message.create/reply to throw or return a retryable rate-limit response and current main makes one delivery attempt before throwing. I did not run tests because this review is read-only and tests may create cache artifacts.

## Solution Assessment

Yes. A narrow Feishu-owned retry wrapper around delivery sends is the best direction because retryability, response shapes, and duplicate-delivery risk are plugin-specific; proactive pacing remains separately tracked.
