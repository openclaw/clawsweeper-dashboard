## Summary

Keep open. Current main and the latest checked release still send Feishu text/post messages, cards, media messages, and initial streaming cards with a single outbound API call, while existing Feishu backoff classification is limited to typing reactions and does not protect message delivery. The related pacing request #14206 is broader and #64448 is a useful duplicate-delivery caution, but neither supersedes this narrower retry reliability bug.

## What This Changes

Feishu outbound message delivery should add bounded retry/backoff handling for retryable rate-limit or flow-control API errors.

## Best Possible Solution

Add a Feishu-plugin-owned bounded retry helper around outbound delivery, covering text/post, interactive cards, media message create/reply, and initial streaming-card sends. It should classify retryable Feishu response bodies and thrown SDK/Axios errors, preserve withdrawn-reply fallback and thread safety, avoid broad core retry for non-idempotent sends, and document/configure retry behavior only if a user-facing knob is added.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists by mocking a retryable Feishu SDK response or thrown error in the current send helpers; current main performs one create/reply call and rejects without a second attempt. Existing send.reply-fallback coverage also asserts a rate-limited reply error is rethrown rather than retried.

## Solution Assessment

Yes. A narrow Feishu-owned retry wrapper is the best direction because Feishu message sends are platform-specific and can be non-idempotent; a generic core/channel retry layer would be more likely to duplicate deliveries or affect unrelated plugins.
