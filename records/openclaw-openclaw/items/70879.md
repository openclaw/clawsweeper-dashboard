## Label Justifications

- `P2`: This is a normal-priority Feishu delivery reliability bug with a clear plugin-owned surface and limited channel-specific blast radius.
- `impact:message-loss`: The reported rate-limit failures can cause outbound Feishu messages to be dropped after a terminal send failure.

## Summary

Keep open. Current main and the latest shipped tag still make one terminal Feishu delivery attempt for text/card replies, media sends, and initial streaming-card sends, while existing backoff handling is limited to typing indicators and related open issues track proactive pacing, idempotency, or generic delivery-failure notification separately.

## What This Changes

The issue requests bounded retry/backoff for Feishu outbound message sends that fail with retryable rate-limit or internal-error responses.

## Best Possible Solution

Add a Feishu-owned bounded retry/backoff helper for explicit retryable delivery failures, apply it consistently to create/reply sends, and keep proactive pacing, media idempotency, and generic failure notifications on their separate trackers unless maintainers merge the scopes.

## Reproduction Assessment

Yes, at source level: mock Feishu client.im.message.create or reply to throw a retryable SDK error, or return a retryable nonzero code, and current main makes one delivery attempt before throwing. I did not run a live Feishu rate-limit scenario in this read-only review.

## Solution Assessment

Yes. A narrow Feishu-plugin retry wrapper around outbound delivery sends is the maintainable fix; it should reuse or factor retryable classification and avoid a core retry layer, new public config option, or generic notification work in this repair.
