## Summary

Keep open. Current main still performs Feishu delivery sends as one-shot API calls for text/post messages, reply fallback, media delivery, and initial streaming cards; existing retry/backoff classification is local to typing reactions and does not cover the reported rate-limit delivery failures. The prior ClawSweeper comment identified the same gap, and the open related pacing and duplicate-delivery reports do not supersede this narrower reliability bug.

## What This Changes

Add bounded retry/backoff handling for Feishu message delivery when retryable rate-limit or transient Feishu API errors occur.

## Best Possible Solution

Implement a Feishu-plugin-owned bounded retry helper for outbound delivery sends, sharing retryable error classification across text/post, reply fallback, media delivery, and initial streaming-card sends while preserving withdrawn-reply fallback behavior and avoiding a core-wide retry layer.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists by mocking a retryable Feishu response body or thrown SDK/Axios error in the current Feishu send helpers; current main performs one delivery call and fails without a second attempt.

## Solution Assessment

Yes. A narrow Feishu-owned retry wrapper is the best direction because Feishu retryability, response codes, and idempotency risks are plugin-specific; a generic core/channel retry would be broader and risk duplicate sends in unrelated channels.
