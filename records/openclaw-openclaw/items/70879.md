## Summary

Keep open. Current main still treats Feishu message delivery rate-limit responses as terminal send failures across text/card, media, and initial streaming-card sends; the existing backoff handling is limited to typing indicators, and the related pacing and duplicate-media issues track different work.

## What This Changes

Add bounded retry/backoff handling for Feishu outbound message sends when Feishu returns retryable rate-limit or flow-control errors.

## Best Possible Solution

Add a Feishu-plugin-owned bounded retry/backoff helper for outbound message delivery, apply it consistently to create/reply sends, and keep proactive pacing or idempotency work in their separate trackers.

## Reproduction Assessment

Yes, at source level: mock Feishu client.im.message.create/reply to throw or return a retryable rate-limit response and current main makes one delivery attempt before throwing. I did not run tests because this review is read-only and tests may create cache artifacts.

## Solution Assessment

Yes. A narrow Feishu-owned retry wrapper around delivery sends is the maintainable fix; it should reuse/factor retryable classification where practical and avoid adding a core retry layer or public config option.
