## Label Justifications

- `impact:message-loss`: The reported rate-limit failures can cause outbound Feishu messages to be dropped after a terminal send failure.

## Summary

Keep open. Current main and v2026.5.20 still treat Feishu outbound rate-limit send failures as terminal in text/card, media, and initial streaming-card delivery paths; the existing backoff classifier is only wired to typing indicators, while proactive pacing, duplicate media idempotency, and delivery-failure notification are separate open items.

## What This Changes

The issue requests bounded retry and backoff handling for Feishu outbound send failures caused by rate limiting.

## Best Possible Solution

Add a Feishu-owned bounded retry/backoff helper for explicit retryable outbound delivery failures across create/reply text, cards, media, and initial streaming-card sends, while leaving proactive pacing, idempotency, and delivery-failure notification on their separate trackers.

## Reproduction Assessment

Yes, at source level: mock Feishu client.im.message.create or reply to throw a retryable SDK error, or return a retryable nonzero response code, and current main makes one delivery attempt before throwing. I did not run a live Feishu rate-limit scenario in this read-only review.

## Solution Assessment

Yes. A narrow Feishu-plugin retry wrapper around outbound delivery sends is the maintainable fix; delivery-failure notification belongs to https://github.com/openclaw/openclaw/issues/46830 and should not be bundled into the retry repair.
