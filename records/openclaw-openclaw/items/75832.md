## Summary

Keep open. Current main still has the provider-identity fix from #74666, but the approval follow-up `agent` continuation does not carry elevated availability/default state into the embedded exec tool path, so the reported second-command failure remains plausible and narrow.

## What This Changes

The issue reports that elevated exec availability is lost after the first `allow-once` approved command completes in a Telegram follow-up run on v2026.4.29.

## Best Possible Solution

Add a narrow internal approval-follow-up path that reconstructs or propagates only elevated availability and default level for the trusted continuation, with regression coverage proving each `allow-once` approval remains single-command only.

## Reproduction Assessment

Yes. The issue provides concrete runtime steps, and current-main source gives a high-confidence static reproduction path through a follow-up `agent` RPC that preserves provider identity but drops elevated defaults before the exec gate.

## Solution Assessment

No. #74666 was a useful partial fix for provider identity, but the maintainable fix also needs scoped elevated-default propagation or reconstruction across approval follow-up continuations.
