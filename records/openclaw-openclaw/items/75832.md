## Summary

Keep open. Current main still preserves Telegram provider identity from PR #74666, but the approval follow-up `agent` continuation does not carry elevated availability/default state, so a later explicit elevated exec can fail before registering a fresh approval.

## What This Changes

Investigate a Telegram elevated exec approval follow-up where a second elevated command in the same assistant turn fails instead of prompting for a fresh approval.

## Best Possible Solution

Add a narrow, tested internal follow-up path that preserves only the elevated availability/default level needed to request fresh approvals, while keeping prior `allow-once` decisions non-reusable.

## Reproduction Assessment

Yes. The issue gives concrete runtime steps, and current-main source provides a high-confidence static path through the follow-up `agent` RPC without elevated defaults into the exec gate.

## Solution Assessment

No. PR #74666 was a useful provider-identity fix, but the remaining maintainable solution must also propagate scoped elevated defaults through approval follow-up continuations without broadening host-exec trust.
