## Summary

Keep open. The report matches current main: approved async exec followups are resumed through the gateway `agent` path, but that path forwards channel/provider context without carrying `bashElevated` into the embedded runner. PR #74666 covered the provider/channel half of the problem, but not the elevated-defaults half that produces the reported enabled-gate error.

## What This Changes

The issue reports that Telegram direct-chat elevated exec approval works once, then subsequent elevated exec calls in the same assistant turn fail as unavailable on v2026.4.29.

## Best Possible Solution

Preserve or recompute the already-authorized elevated defaults for exec-approval followup agent runs through a typed internal-only context, then add regression coverage proving a second elevated exec in that followup prompts again instead of failing closed as unavailable.

## Reproduction Assessment

Yes. The issue gives deterministic Telegram steps, and current main exposes a focused code-level reproduction: an approved async exec followup enters the gateway `agent` path, which currently reaches `runEmbeddedPiAgent` without `bashElevated`.

## Solution Assessment

No current solution is in place. The narrow maintainable fix is not another channel fallback; it is carrying or recomputing elevated permission state for trusted approval followup runs without widening user-controlled RPC authority.
