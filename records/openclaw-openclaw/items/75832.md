## Summary

Keep open. Current main still contains the provider-identity fix from #74666, but the approval-follow-up `agent` continuation has no path to carry elevated availability/default state into the embedded exec tool, so the reported same-turn second-command failure remains valid and security-sensitive.

## What This Changes

The issue asks OpenClaw to keep elevated exec available for subsequent elevated commands in the same Telegram approval-follow-up turn after an `allow-once` command completes.

## Best Possible Solution

Add a narrow internal approval-follow-up elevated-state path plus regression coverage proving the second elevated command gets a fresh approval and prior `allow-once` authority is not reused.

## Reproduction Assessment

Yes, source-reproducible. The issue gives deterministic Telegram/sandboxed steps, and current main shows follow-up agent runs preserve provider identity but do not carry elevated defaults into the embedded exec path.

## Solution Assessment

No. #74666 was a useful partial fix for provider identity, but the maintainable fix also needs scoped elevated-default propagation or reconstruction across approval-follow-up continuations.
