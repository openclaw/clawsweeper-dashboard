## Summary

Keep open. Current main still preserves provider identity from #74666, but the approval-follow-up path still has no elevated availability/default state, so the reported failure remains valid and security-sensitive.

## What This Changes

The issue reports that after one approved elevated exec in Telegram, a second elevated exec in the same assistant turn fails instead of prompting for a fresh approval.

## Best Possible Solution

Add a narrow trusted approval-follow-up path that reconstructs or propagates only elevated availability and default level, with regression coverage proving every `allow-once` approval remains single-command only.

## Reproduction Assessment

Yes. The issue gives deterministic steps, and current-main source shows the follow-up `agent` RPC preserves provider identity but drops elevated defaults before the embedded exec gate; I did not run a live Telegram sandbox reproduction.

## Solution Assessment

No. #74666 was a useful partial fix for provider identity, but the maintainable fix also needs scoped elevated-default propagation or reconstruction across approval follow-up continuations.
