## Summary

Keep open. Current main still preserves Telegram channel identity for the approval follow-up, but the Gateway `agent` continuation path does not carry elevated availability/default state, so a later explicit elevated exec can fail at the elevated gate before registering a fresh approval.

## What This Changes

Reports that a second elevated exec in the same Telegram assistant turn fails after an `allow-once` approval instead of prompting again.

## Best Possible Solution

Add a narrow, tested internal follow-up path that preserves elevated availability/default level so later elevated exec calls can request fresh approvals while keeping `allow-once` single-command semantics intact.

## Reproduction Assessment

Yes. The issue provides concrete runtime steps, and current-main source gives a high-confidence static reproduction path through the follow-up `agent` RPC without elevated defaults into the exec gate.

## Solution Assessment

No. PR #74666 was a useful provider-identity fix, but the maintainable solution also needs scoped elevated-default propagation through approval follow-up continuations without broadening host-exec trust.
