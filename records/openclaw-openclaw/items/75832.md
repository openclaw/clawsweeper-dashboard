## Summary

Keep open. Current main still has the PR #74666 provider-preservation fix, but the approval follow-up continuation cannot carry elevated availability/defaults through the Gateway `agent` RPC, so a later same-turn elevated exec can still fail before registering a fresh approval.

## What This Changes

Report asks to make repeated Telegram elevated exec commands in one assistant turn prompt again after `allow-once` instead of failing the elevated gate.

## Best Possible Solution

Add a narrow, tested internal approval-follow-up continuation path that re-derives or carries only elevated availability/default level for the originating trusted session while preserving single-command `allow-once` semantics.

## Reproduction Assessment

Yes. The issue gives concrete runtime steps, and current-main source shows the follow-up `agent` path lacks elevated defaults before the exec gate that emits the reported error.

## Solution Assessment

No. PR #74666 fixed provider identity, but the maintainable fix also needs scoped elevated-default propagation through the internal follow-up path without broadening host-exec authority.
