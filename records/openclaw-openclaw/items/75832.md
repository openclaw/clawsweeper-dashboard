## Summary

Keep open. Current main still matches the reported failure mode: the approval followup path preserves Telegram channel identity from the prior fix, but it still does not carry or recompute elevated exec defaults for the resumed agent turn, so a later explicit elevated exec can fail before reaching the approval prompt.

## What This Changes

The issue reports that Telegram direct-chat elevated exec approval works once on v2026.4.29, then subsequent elevated exec calls in the same assistant turn fail as unavailable.

## Best Possible Solution

Add a narrow, tested internal propagation or recomputation path for elevated availability/default state in exec-approval followup agent runs, while preserving one-shot approval decisions and keeping elevated flags out of general user-controlled gateway agent requests.

## Reproduction Assessment

Yes. The issue provides concrete Telegram steps, and current main gives a focused static reproduction path: an approved async exec followup enters the gateway agent path, which reaches the embedded runner without bashElevated.

## Solution Assessment

No. PR #74666 fixed the provider/channel half of the failure, but the complete fix also needs elevated runtime defaults to survive the trusted approval followup path without bypassing per-command approval prompts.
