## Summary

Keep open. The report is still valid on current main: the prior merged fix preserved the Telegram provider identity for approval follow-up runs, but the Gateway `agent` follow-up path still does not propagate elevated availability/default state into the embedded run, so a second elevated exec can fail closed before a new approval prompt is registered.

## What This Changes

The issue requests a fix for Telegram elevated exec follow-up runs where a second elevated command in the same assistant turn fails after an allow-once approval, despite the earlier PR #74666 provider-context fix.

## Best Possible Solution

Add a narrow, tested propagation path for elevated availability and default level into exec-approval follow-up Gateway `agent` runs, while keeping approval decisions one-shot and preserving the existing provider/channel checks.

## Reproduction Assessment

Yes. The issue provides a deterministic Telegram/docker repro, and current source gives a high-confidence static reproduction: follow-up runs preserve `messageProvider` but enter the embedded run without `bashElevated`, causing explicit elevated exec to fail at the enabled gate before approval registration.

## Solution Assessment

No. PR #74666 was a useful partial fix for provider identity, but the complete fix must also preserve the original elevated runtime defaults through the follow-up run without widening approval authority.
