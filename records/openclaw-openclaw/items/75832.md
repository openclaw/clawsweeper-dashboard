## Summary

Keep open. Current main contains the PR #74666 provider-context fallback, but the Gateway `agent` continuation still cannot carry elevated defaults into the embedded exec tools, so a second elevated exec in the approval follow-up turn can fail at the elevated availability gate before a fresh approval prompt is registered.

## What This Changes

Report that Telegram direct-chat elevated exec approval follow-up runs lose elevated availability after the first approved command in the same assistant turn.

## Best Possible Solution

Preserve scoped elevated availability/default mode across internal exec-approval follow-up continuations while keeping `allow-once` single-command semantics and not exposing elevated authority to arbitrary external `agent` RPC callers.

## Reproduction Assessment

Yes. The issue gives deterministic v2026.4.29 Telegram direct-chat steps, and current-main source shows the follow-up `agent` RPC lacks a path to carry elevated defaults into the embedded exec tool, producing the reported gate error before approval registration.

## Solution Assessment

No. PR #74666 was a valid provider-identity fix, but the remaining maintainable solution needs a narrow internal elevated-default propagation path through the approval follow-up continuation with explicit security tests.
