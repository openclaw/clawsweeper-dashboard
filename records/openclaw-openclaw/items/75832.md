## Summary

Keep open. Current main still has the provider-identity fix from #74666, but the approval follow-up `agent` continuation does not carry elevated availability/default state into the embedded exec path, so the reported second-command failure remains valid and security-sensitive.

## What This Changes

The issue reports that after one approved elevated exec in Telegram, a second elevated exec in the same assistant turn fails instead of prompting for a fresh approval.

## Best Possible Solution

Add a narrow internal approval-follow-up path that reconstructs or propagates only elevated availability and default level for the trusted continuation, with regression coverage proving each `allow-once` approval remains single-command only.

## Reproduction Assessment

Yes. The issue gives deterministic runtime steps, and current-main source shows the follow-up `agent` RPC preserves provider identity but drops elevated defaults before the embedded exec gate.

## Solution Assessment

No. #74666 was a useful partial fix for provider identity, but the maintainable fix also needs scoped elevated-default propagation or reconstruction across approval follow-up continuations.
