## Summary

Keep open. Current main still has the provider-identity fix from PR #74666, but the Gateway approval-follow-up `agent` continuation has no contract or dispatch path for scoped elevated availability/defaults, so a second explicit elevated exec can still fail before a fresh approval request is registered.

## What This Changes

The issue reports that Telegram elevated exec approval state drops after one approved command, causing later elevated exec calls in the same assistant turn to fail instead of prompting again.

## Best Possible Solution

Add a narrow internal approval-follow-up path that preserves only elevated availability/default level for requesting fresh approvals, while keeping consumed approvals non-reusable and arbitrary Gateway `agent` calls unprivileged.

## Reproduction Assessment

Yes. The issue gives concrete steps, and current-main source provides a high-confidence static path where the follow-up run preserves provider identity but loses elevated defaults before the exec approval gate.

## Solution Assessment

No. PR #74666 was the right narrow provider-identity fix, but the remaining issue needs a separately scoped elevated-continuation contract and tests around fresh approval registration.
