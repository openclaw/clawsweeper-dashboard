## Summary

Keep open. Current main still matches the report: PR #74666 preserves the Telegram message-provider context, but the Gateway agent follow-up path still has no way to carry elevated availability/default state into the resumed embedded run.

## What This Changes

The issue reports that after one allow-once elevated exec approval, a second elevated exec in the same Telegram turn fails instead of prompting again.

## Best Possible Solution

Add a narrow, tested follow-up-only propagation path for elevated availability/default level so later elevated exec calls can register fresh approvals without carrying over prior allow-once decisions.

## Reproduction Assessment

Yes. The issue includes concrete runtime steps, and current source gives a static reproduction: the follow-up agent run preserves provider identity but has no elevated defaults, so the next explicit elevated exec can fail before approval registration.

## Solution Assessment

No. PR #74666 was a useful partial fix for provider identity, but the complete maintainable fix must also preserve only the elevated runtime defaults needed to re-enter normal approval flow.
