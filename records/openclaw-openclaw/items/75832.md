## Summary

Keep open. The report matches current main: the shipped PR #74666 preserves Telegram channel identity for approval follow-up runs, but the Gateway `agent` follow-up path still does not propagate the elevated availability/default state needed for a second explicit elevated exec to reach approval registration.

## What This Changes

The issue requests that elevated exec approval follow-up turns keep prompting for subsequent elevated commands in the same assistant turn after an `allow-once` approval.

## Best Possible Solution

Add a narrow, tested propagation path for elevated availability/default level into exec-approval follow-up agent runs so later elevated exec calls can register a fresh approval request while preserving one-shot approval semantics.

## Reproduction Assessment

Yes. The issue includes concrete runtime steps, and current source gives a high-confidence static reproduction: the follow-up run preserves `messageProvider` but enters the embedded agent path without `bashElevated`, so the next explicit elevated exec can fail at the enabled gate before prompting.

## Solution Assessment

No. PR #74666 was a useful partial fix for provider identity, but the complete maintainable fix also needs to preserve the original elevated runtime defaults through the follow-up run without bypassing per-command exec approvals.
