## Summary

Keep open. Current main still carries the #74666 provider-identity fix, but the approval-follow-up `agent` continuation still has no path to carry or reconstruct elevated availability/default state for embedded exec tools, so the reported same-turn second-command failure remains a plausible current bug.

## What This Changes

Request to make elevated exec ask-mode register a fresh approval prompt for a second elevated command in the same assistant turn after the first `allow-once` command completes.

## Best Possible Solution

Add a narrow trusted approval-follow-up path that reconstructs or propagates only elevated availability and default level for the continuation, with regression coverage proving each `allow-once` approval remains single-command only.

## Reproduction Assessment

Yes. The issue gives a concrete same-turn Telegram repro, and current-main source provides a high-confidence static path where the approval follow-up preserves provider identity but drops elevated defaults before embedded exec gating.

## Solution Assessment

No. #74666 was a useful partial fix for provider identity, but the maintainable fix also needs scoped elevated-default propagation or reconstruction across approval-follow-up continuations.
