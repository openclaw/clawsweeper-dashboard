## Summary

Keep open. Current main still only preserves the Telegram provider identity for exec-approval follow-up runs; it does not carry elevated availability/default state into the Gateway agent continuation, so a second explicit elevated exec can still fail the enabled gate before registering a fresh approval.

## What This Changes

The issue asks OpenClaw to keep per-command elevated exec approvals available for multiple elevated commands in one Telegram assistant turn after v2026.4.29.

## Best Possible Solution

Add a narrow, tested propagation path for elevated availability/default level into exec-approval follow-up agent runs, while preserving one-shot approval decisions and avoiding a broad public protocol escape hatch.

## Reproduction Assessment

Yes. The issue provides concrete runtime steps, and current source gives a high-confidence static reproduction: the follow-up run preserves messageProvider but not bashElevated, so the next explicit elevated exec can fail before approval registration.

## Solution Assessment

No. PR #74666 was a useful partial fix for provider identity, but the maintainable fix also needs elevated runtime defaults to survive the follow-up run without bypassing per-command approval semantics.
