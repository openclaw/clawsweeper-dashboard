## Summary

Keep open. Current main still includes the provider-identity fix from the related PR, but the approval follow-up `agent` continuation still has no path to carry elevated availability/default state into the embedded exec tool, so the reported second-command failure remains valid and security-sensitive.

## What This Changes

The issue reports that a Telegram sandboxed approval follow-up can run one elevated exec command, then loses elevated availability for a second elevated exec in the same assistant turn.

## Best Possible Solution

Add a narrow trusted approval-follow-up path that reconstructs or propagates only elevated availability and default level for the continuation, with regression coverage proving every `allow-once` approval remains single-command only.

## Reproduction Assessment

Yes. The issue gives deterministic runtime steps, and current-main source shows the approval follow-up `agent` path preserves provider identity but drops elevated defaults before the embedded exec gate.

## Solution Assessment

No. The related PR was a useful partial fix for provider identity, but the maintainable fix also needs scoped elevated-default propagation or reconstruction across approval follow-up continuations.
