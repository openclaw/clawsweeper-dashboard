## Summary

Keep open: the reported duplicate-delivery bug looks source-reproducible, but this PR introduces two blocking delivery regressions and still lacks real Telegram behavior proof for the visible reply change.

## What This Changes

The PR adds a dispatch-time guard that suppresses exact silent final replies after `hasRepliedRef` indicates a message-tool delivery, with focused dispatch tests for silent and visible final replies.

## Best Possible Solution

Keep the dispatch-level suppression only after it preserves all non-text sendable payload fields, treats a suppressed silent final as successful for durable pending-final cleanup, and includes real Telegram proof.

## Reproduction Assessment

Yes, source-reproducible: on current main, a `dispatchReplyFromConfig` call with `replyOptions.hasRepliedRef.value === true` and a resolver returning `{ text: "NO_REPLY" }` still reaches final delivery. I did not run tests because this review is read-only.

## Solution Assessment

No. Suppressing at dispatch is plausible, but this implementation must also clear durable pending-final state and preserve presentation, interactive, channel-data, or audio-only payload content.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Clear pending delivery when suppressing the silent final:** `src/auto-reply/reply/dispatch-from-config.ts:1584`
  - body: Agent runs store `pendingFinalDeliveryText` before dispatch, and this file only clears it after an attempted final delivery succeeds. This new `continue` skips `sendFinalPayload` and never calls `clearPendingFinalDeliveryAfterSuccess`, so a message-tool turn ending in `NO_REPLY` can leave stale pending final state in the session.
  - confidence: 0.86
- **[P2] Preserve non-text sendable payload fields:** `src/auto-reply/reply/dispatch-from-config.ts:1033`
  - body: This guard decides there is no other content by clearing `text` and checking `resolveSendableOutboundReplyParts`, but that helper only knows text and media. Final payloads can still be sendable through `presentation`, `interactive`, `channelData`, or `audioAsVoice`, so those payloads would now be dropped whenever the text is exact `NO_REPLY` after message-tool delivery.
  - confidence: 0.78
