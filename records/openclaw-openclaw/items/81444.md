## Summary

Keep open: the reported duplicate-final behavior is source-reproducible, but this PR still has two correctness blockers and the author confirmed live Telegram proof is still pending.

## What This Changes

Adds a dispatch guard and regression tests to skip an exact `NO_REPLY` final when `hasRepliedRef` indicates a message-tool reply already happened.

## Best Possible Solution

Keep the dispatch-level fix direction, but preserve full reply payload content, clear durable pending-final state when suppression succeeds, and add redacted live Telegram proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main sends a final reply after a tool result in the dispatch path, while the docs define exact `NO_REPLY` as non-user-visible. I did not run tests because this review is read-only.

## Solution Assessment

No. Dispatch-level suppression is plausible, but this patch is not the narrow safe fix until it uses the full reply-payload content contract and handles pending-final cleanup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Clear pending delivery when suppressing the silent final:** `src/auto-reply/reply/dispatch-from-config.ts:1580-1582`
  - body: Agent runs persist `pendingFinalDeliveryText` before dispatch, and cleanup only runs after `attemptedFinalDelivery` is set. This early `continue` suppresses the final before setting that flag, so restart recovery can retain stale pending-final state and replay a message that was intentionally handled.
  - confidence: 0.86
- **[P2] Check structured payload fields before suppressing:** `src/auto-reply/reply/dispatch-from-config.ts:1035`
  - body: The helper treats a `NO_REPLY` payload with no text/media as empty by calling `resolveSendableOutboundReplyParts` after deleting text. That ignores `presentation`, `interactive`, and `channelData` content covered by `hasReplyPayloadContent`, so a message-tool turn could silently drop a deliverable structured final.
  - confidence: 0.82
