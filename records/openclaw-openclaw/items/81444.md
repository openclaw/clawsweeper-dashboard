## Summary

Keep open, but do not merge as-is: the reported duplicate-final behavior is valid, while this patch still skips durable pending-final cleanup and can drop structured sendable payloads. The supplied proof is source-level Vitest output, not real Telegram behavior proof.

## What This Changes

Adds a dispatch guard and focused tests to suppress exact `NO_REPLY` final payloads when `hasRepliedRef` shows a message-tool delivery already happened.

## Best Possible Solution

Keep a dispatch-level fix, but make suppressed silent finals clear durable pending-final state, use the full reply-payload content contract, and capture real Telegram proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main sends `{ text: "NO_REPLY" }` as a final after a tool result in the dispatch path, and existing dispatch tests assert that behavior. I did not execute tests because this review is read-only.

## Solution Assessment

No. Suppression at dispatch is plausible, but this implementation is not the narrow safe fix until it preserves structured payload content and handles pending-final cleanup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Clear pending delivery when suppressing the silent final:** `src/auto-reply/reply/dispatch-from-config.ts:1584`
  - body: Agent runs persist `pendingFinalDeliveryText` before dispatch, and cleanup only happens after `attemptedFinalDelivery` is set. This early `continue` treats the silent final as handled but skips `clearPendingFinalDeliveryAfterSuccess`, so a message-tool turn ending in `NO_REPLY` can leave stale pending-final state for restart recovery.
  - confidence: 0.86
- **[P2] Check structured payload fields before suppressing:** `src/auto-reply/reply/dispatch-from-config.ts:1035`
  - body: The helper decides a `NO_REPLY` payload has no other content with `resolveSendableOutboundReplyParts({ ...reply, text: undefined })`, but that helper only covers text and media. Final payloads with `presentation`, `interactive`, `channelData`, or `audioAsVoice` would be dropped after any message-tool delivery even though those fields are deliverable.
  - confidence: 0.82
