## Summary

Keep open: this focused bug-fix PR is useful, but the current head still has two blocking correctness issues and only source-level test proof for a Telegram-visible path.

## What This Changes

Adds a dispatch guard to suppress exact `NO_REPLY` finals after `hasRepliedRef` shows message-tool delivery, plus dispatch regression tests for silent and non-silent finals.

## Best Possible Solution

Keep the dispatch-level fix direction, but make suppression use the full reply-payload content contract, clear durable pending-final state when consuming the silent final, and require redacted live Telegram proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main's dispatch path sends final `{ text: "NO_REPLY" }` payloads after tool-result handling, and existing tests assert that behavior. I did not execute tests because this review is read-only.

## Solution Assessment

No. Suppression at dispatch is plausible, but this implementation should preserve structured payload content and clear pending-final state before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Clear pending delivery when suppressing the silent final:** `src/auto-reply/reply/dispatch-from-config.ts:1581-1582`
  - body: This early `continue` runs before `attemptedFinalDelivery` is set, but agent runs persist `pendingFinalDeliveryText` before dispatch and cleanup only runs after an attempted final succeeds. A message-tool turn ending in exact `NO_REPLY` can therefore leave stale pending-final state for restart recovery.
  - confidence: 0.87
- **[P2] Use full reply-payload content before suppressing:** `src/auto-reply/reply/dispatch-from-config.ts:1035`
  - body: The helper clears `text` and asks `resolveSendableOutboundReplyParts` whether content remains, but that helper only considers text and media. `ReplyPayload` and `hasReplyPayloadContent` also treat `presentation`, `interactive`, and `channelData` as deliverable, so a `NO_REPLY` payload carrying those fields would be dropped after message-tool delivery.
  - confidence: 0.83
