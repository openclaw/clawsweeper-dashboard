## Summary

Keep open: the underlying Telegram duplicate-final behavior is source-reproducible, but this branch still has correctness blockers and only source-level test proof; the author has acknowledged live Telegram proof is still pending.

## What This Changes

The PR adds a dispatch guard and regression tests to skip exact `NO_REPLY` final replies when `hasRepliedRef` indicates a message tool already replied.

## Best Possible Solution

Converge on one fix that suppresses only text-only silent finals after actual same-conversation message-tool delivery, clears durable pending-final state, preserves rich/media payloads, and includes live Telegram proof.

## Reproduction Assessment

Yes, source-reproducible: current main sends a final `{ text: "NO_REPLY" }` through the dispatch path after a tool result, and the final loop has no `hasRepliedRef` suppression. I did not run tests because this review is read-only.

## Solution Assessment

No. Suppression after message-tool delivery is the right direction, but this implementation should clear pending-final state and use the full reply-payload content contract before dropping a final payload.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Clear pending delivery when suppressing the silent final:** `src/auto-reply/reply/dispatch-from-config.ts:1584`
  - body: This early `continue` treats the final `NO_REPLY` as handled but skips the later `attemptedFinalDelivery` path that clears `pendingFinalDelivery*`. Since agent runs persist pending final text before dispatch, a message-tool turn ending in `NO_REPLY` can leave stale retry state for restart recovery.
  - confidence: 0.86
- **[P2] Check full payload content before suppressing:** `src/auto-reply/reply/dispatch-from-config.ts:1035`
  - body: The helper checks only text and media by calling `resolveSendableOutboundReplyParts`, then suppresses the whole payload. Final payloads with `presentation`, `interactive`, or non-empty `channelData` would be dropped after message-tool delivery even though the broader reply content contract treats those fields as content.
  - confidence: 0.82
