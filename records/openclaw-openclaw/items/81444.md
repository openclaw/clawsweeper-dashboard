## Summary

Keep open: this is a plausible bug fix for a source-reproducible Telegram duplicate-reply path, but the current patch still drops some sendable payloads, skips pending-final cleanup, and lacks live Telegram proof. It also overlaps the still-open https://github.com/openclaw/openclaw/pull/79379, so maintainers should converge on one repaired fix path.

## What This Changes

The PR suppresses exact `NO_REPLY` final replies after message-tool delivery evidence has marked the source conversation as already replied, and adds dispatch regression tests.

## Best Possible Solution

Keep a dispatch-level fix only if it clears durable pending-final state, uses the full reply-payload content contract before suppressing, preserves normal rich/media delivery, and converges with https://github.com/openclaw/openclaw/pull/79379 before live Telegram proof is attached.

## Reproduction Assessment

Yes, source-reproducible: current main has dispatch coverage where a tool result is followed by final `{ text: "NO_REPLY" }` and still expects a final reply to be queued. I did not execute tests in this read-only pass.

## Solution Assessment

No. Suppression at dispatch is plausible, but this implementation should first preserve rich payload content and clear pending-final retry state when it intentionally handles the silent final.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Clear pending final state when suppressing silent finals:** `src/auto-reply/reply/dispatch-from-config.ts:1584`
  - body: Agent runs persist `pendingFinalDeliveryText` before dispatch, and cleanup only happens after `attemptedFinalDelivery` is set. This early `continue` suppresses the final but never reaches `clearPendingFinalDeliveryAfterSuccess`, so restart recovery can keep stale pending-final state after the message tool already delivered the visible reply.
  - confidence: 0.86
- **[P2] Use the full payload content contract before suppressing:** `src/auto-reply/reply/dispatch-from-config.ts:1035`
  - body: `resolveSendableOutboundReplyParts({ ...reply, text: undefined })` only checks text and media. A final payload with exact `NO_REPLY` plus `presentation`, `interactive`, or `channelData` would be dropped after message-tool delivery even though the normal reply-content contract treats those fields as sendable.
  - confidence: 0.82
