## Summary

This should stay open: current main still has a source-level path where a successful block stream causes an unsent text-only final reply to be removed before dispatch.

## What This Changes

The issue reports that Feishu group chat text replies are dropped when `blockStreaming` is enabled and earlier block-stream output makes the final text-only payload look suppressible.

## Best Possible Solution

Change block-stream final suppression to preserve text-only final payloads unless `blockReplyPipeline.hasSentPayload` proves that same text was already delivered, while keeping media de-duplication intact.

## Reproduction Assessment

Yes, as a source-level reproduction: call `buildReplyPayloads` with `blockStreamingEnabled: true`, a pipeline reporting `didStream() === true` and `isAborted() === false`, `hasSentPayload() === false`, and a text-only final payload; current code returns no payloads. I did not run a live Feishu account reproduction in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is in the core final-payload suppression logic, with regression coverage for unsent text-only finals after unrelated block-stream output. Feishu defaults and new config options should not be changed.
