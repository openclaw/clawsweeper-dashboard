## Summary

Keep open: current main and the latest release still contain the core text-only final-payload suppression after successful block streaming, and Feishu still has a channel-side block path that drops block payloads unless they can be consumed by the streaming-card fallback. The related repair PRs are open but not clean canonical fixes yet, so there is still meaningful maintainer work here.

## What This Changes

The issue reports that Feishu replies generated by the model can be silently dropped when `blockStreaming` is enabled, especially text-only final replies after block output.

## Best Possible Solution

Repair the shared final-payload filter so unsent text-only final replies survive block streaming without duplicating already-streamed chunks, and keep Feishu's real completed-block delivery work coordinated with https://github.com/openclaw/openclaw/issues/55027.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.6` still drop text-only final payloads after a successful non-aborted block stream, and the focused unit test still expects that drop. I did not run a live Feishu reproduction in this read-only review.

## Solution Assessment

No current open PR is the best solution as-is. The narrow shared repair needs to preserve coalesced-stream dedupe, while Feishu channel-side completed-block delivery should be coordinated with the existing canonical Feishu issue rather than solved by an unapproved competing config surface.
