## Summary

Keep open: current main and the latest release still contain the shared block-streaming final-text suppression path, and Feishu still drops block payloads unless they can be consumed by the streaming-card path. The related open PRs are not clean canonical fixes, so this remains valid maintainer work.

## What This Changes

The issue reports Feishu text replies being silently dropped when blockStreaming is enabled, including a later report that the behavior also appears in 1:1 Feishu DMs.

## Best Possible Solution

Land one coordinated fix that preserves already-streamed dedupe while delivering unsent final text, and resolve Feishu blockStreaming behavior when streaming-card delivery is disabled.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence core reproduction path: call buildReplyPayloads with blockStreamingEnabled true, didStream true, isAborted false, hasSentPayload false, and a text-only final payload; the current test asserts zero outbound payloads.

## Solution Assessment

Unclear as a single automated fix. Preserving unsent final text is a likely core repair, but Feishu blockStreaming behavior overlaps with broader Feishu streaming-mode/design work and needs maintainer coordination.
