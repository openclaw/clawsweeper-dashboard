## Summary

Keep open: current main still has a source-level path where text-only final replies are removed after successful block streaming, and the related open fix PR has not landed and still needs rework/proof.

## What This Changes

The issue reports Feishu group-chat text replies being discarded when `blockStreaming` is enabled and the block pipeline streamed earlier content.

## Best Possible Solution

Land or replace the existing core reply-payload fix so successful block streaming suppresses only payloads proven delivered while preserving coalesced text, media, error, and direct-send dedupe behavior.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: call `buildReplyPayloads` with block streaming enabled, `didStream() === true`, `isAborted() === false`, `hasSentPayload() === false`, and a text-only final payload; current main returns no reply payloads. I did not run a live Feishu account reproduction.

## Solution Assessment

Yes, the right direction is a narrow core payload-suppression fix, not a Feishu config workaround. The existing open PR direction needs coalesced-stream-safe dedupe before it is the best implementation.
