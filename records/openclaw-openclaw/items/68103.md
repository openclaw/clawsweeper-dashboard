Overall confidence: 0.9

Full review comments:

- **[P1] Pass auth options instead of a built auth method:** `extensions/nvidia/index.ts:24`
  - body: `defineSingleProviderPluginEntry` already wraps each auth option with `createProviderApiKeyAuthMethod`. Passing an already-created auth method here gives the helper the wrong object shape, so setup/auth metadata can be registered with missing option fields such as `methodId`, `optionKey`, or `envVar`.
  - confidence: 0.92
- **[P1] Write NVIDIA defaults under models.providers:** `extensions/nvidia/index.ts:35-45`
  - body: This `applyConfig` writes a top-level `providers` object, but OpenClaw model provider config lives under `models.providers`. Onboarding through this auth method would leave NVIDIA defaults in the wrong config location.
  - confidence: 0.9
- **[P2] Keep static catalog rows visible without credentials:** `extensions/nvidia/index.ts:69-70`
  - body: When no API key is available, this live catalog returns `null` and the branch does not register a static fallback catalog, so bundled NVIDIA rows disappear from unauthenticated listing and picker paths even though current main exposes safe static rows before auth.
  - confidence: 0.86
- **[P2] Declare the NVIDIA discovery config before reading it:** `extensions/nvidia/index.ts:63-64`
  - body: The branch reads `ctx.config?.models?.nvidiaDiscovery?.enabled`, but that key is not part of the current config contract or plugin schema. Add a typed plugin config/schema surface or remove the undeclared compatibility path before using it.
  - confidence: 0.83
- **[P2] Normalize NVIDIA ids before catalog lookup:** `extensions/nvidia/index.ts:89`
  - body: Dynamic model resolution can receive provider-stripped ids, while the new catalog stores ids such as `nvidia/nemotron-...`. This exact comparison misses known rows, drops catalog metadata, and can return a bare model id for NVIDIA runtime calls.
  - confidence: 0.88
