## Summary

Keep open. Current main has a shipped static NVIDIA provider, but it does not implement this PR's NIM `/models` discovery or expanded dynamic catalog work; the PR remains potentially useful, but its current head conflicts with the current single-provider auth/config contracts and with the merged NVIDIA onboarding/literal-prefix behavior.

## What This Changes

The PR expands the bundled NVIDIA provider docs/catalog, adds a NVIDIA models helper with optional NIM `/models` discovery, and wires dynamic catalog/model resolution plus API-key auth into the NVIDIA plugin.

## Best Possible Solution

Keep the PR open and request changes: rebase on current main, preserve NVIDIA's current onboarding and literal-prefix semantics, keep discovery provider-owned, align static manifest/catalog metadata with runtime discovery, use the single-provider auth option shape, write config through `models.providers`, normalize prefixed and provider-stripped NVIDIA ids, and add focused tests for discovery fallback, metadata retention, config patches, and secret-safe fetch behavior.

## Reproduction Assessment

Not applicable as a user bug report; this is a feature PR. The merge blockers are reproducible by static inspection of the PR head against current `defineSingleProviderPluginEntry`, `OpenClawConfig`, and NVIDIA manifest/provider code.

## Solution Assessment

No. The requested NVIDIA NIM discovery direction is reasonable, but this PR is not the best current implementation until it is rebased onto the merged NVIDIA provider design and brought back inside the manifest-first provider/auth contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Use single-provider auth option objects:** `extensions/nvidia/index.ts:24`
  - body: `defineSingleProviderPluginEntry` maps each `provider.auth` entry into `createProviderApiKeyAuthMethod()` itself. Passing the already-created auth method here leaves option fields such as `methodId`, `optionKey`, and `envVar` missing when the wrapper registers NVIDIA auth, so setup can fail or typecheck can reject the provider entry.
  - confidence: 0.9
- **[P1] Write NVIDIA config under models.providers:** `extensions/nvidia/index.ts:37-44`
  - body: This auth `applyConfig` writes a top-level `providers` object, but OpenClaw provider config lives under `models.providers`. The onboarding patch would not configure the actual NVIDIA provider path and does not match the current `OpenClawConfig` shape.
  - confidence: 0.88
- **[P2] Normalize NVIDIA ids before catalog lookup:** `extensions/nvidia/index.ts:89`
  - body: `resolveDynamicModel` compares `ctx.modelId` directly with catalog ids like `nvidia/nemotron-...`. When model refs are parsed into provider-stripped ids, known models miss this lookup, drop catalog metadata, and fall back to a bare text-only model id.
  - confidence: 0.84
