## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Pass auth options instead of a built auth method:** `extensions/nvidia/index.ts:24`
  - body: `defineSingleProviderPluginEntry` maps `provider.auth` option objects and calls `createProviderApiKeyAuthMethod` itself. Passing an already-built auth method here gives the wrapper the wrong shape, so setup/auth registration can read missing fields such as `methodId`, `optionKey`, or `envVar`.
  - confidence: 0.92
- **[P1] Write NVIDIA defaults under models.providers:** `extensions/nvidia/index.ts:35-45`
  - body: This `applyConfig` creates a top-level `providers` object, but OpenClaw provider config is stored under `models.providers`. Onboarding through this auth method would leave NVIDIA defaults in an ignored location.
  - confidence: 0.9
- **[P2] Keep static catalog rows visible without credentials:** `extensions/nvidia/index.ts:69-70`
  - body: When no API key is available, the live catalog returns `null` and no `staticRun` fallback is registered. That removes the unauthenticated static NVIDIA rows that current main exposes for listing and picker paths.
  - confidence: 0.86
- **[P2] Normalize NVIDIA IDs before catalog lookup:** `extensions/nvidia/index.ts:89`
  - body: Dynamic resolution can receive a provider-stripped model ID, while `NVIDIA_CATALOGED_MODELS` stores entries such as `nvidia/nemotron-...`. The exact comparison misses known rows, drops metadata, and can return a bare runtime model ID.
  - confidence: 0.88
- **[P2] Declare the NVIDIA discovery config before reading it:** `extensions/nvidia/index.ts:63-64`
  - body: The branch reads `ctx.config?.models?.nvidiaDiscovery?.enabled`, but that key is not part of the current config contract or plugin schema. Either declare a typed plugin config/schema surface or remove the undeclared compatibility path.
  - confidence: 0.83
