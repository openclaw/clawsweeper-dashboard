## Summary

Keep open. Current main has a first-class NVIDIA provider, but it is still static; this PR contains useful unshipped NIM discovery/catalog work, and the current branch has concrete merge-blocking auth/config/model-id contract bugs that should be fixed rather than swept closed.

## What This Changes

The PR expands NVIDIA provider docs/catalog data, adds a helper for NVIDIA NIM `/models` discovery, and wires dynamic catalog/model resolution into the bundled NVIDIA provider.

## Best Possible Solution

Rework the PR on current main as a focused NVIDIA provider update that keeps the existing manifest/onboarding/auth contracts, adds bounded NIM discovery with static fallback metadata, preserves NVIDIA model-prefix semantics, and covers auth, config, discovery fallback, and metadata with targeted tests.

## Reproduction Assessment

Not applicable as a user bug report; this is a feature PR. The merge blockers are high-confidence from static inspection of the PR head against current `defineSingleProviderPluginEntry`, `OpenClawConfig`, and NVIDIA catalog contracts.

## Solution Assessment

No. Dynamic NVIDIA NIM discovery is a reasonable provider-local direction, but this branch is not the best implementation because it bypasses the current auth helper shape, writes provider config to the wrong path, and does not normalize NVIDIA model IDs before catalog lookup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Pass auth option objects to the single-provider helper:** `extensions/nvidia/index.ts:24`
  - body: `defineSingleProviderPluginEntry` already wraps each `provider.auth` entry with `createProviderApiKeyAuthMethod()`. Passing the created auth method here leaves the wrapper without option fields such as `methodId`, `optionKey`, and `envVar`, so NVIDIA setup/auth registration can fail or fail type checking.
  - confidence: 0.92
- **[P1] Write NVIDIA defaults under models.providers:** `extensions/nvidia/index.ts:35-45`
  - body: This `applyConfig` patch writes a top-level `providers` object, but OpenClaw model providers live under `models.providers`. The returned config patch will not configure the NVIDIA model provider path and conflicts with the `OpenClawConfig` shape.
  - confidence: 0.9
- **[P2] Keep discovery config on a declared plugin surface:** `extensions/nvidia/index.ts:63-64`
  - body: `ctx.config?.models?.nvidiaDiscovery` adds an undeclared `models` key; current `ModelsConfig` only carries legacy discovery toggles for other providers. Put this under the plugin config/schema or add a documented typed config surface before reading it.
  - confidence: 0.83
- **[P2] Normalize IDs before NVIDIA catalog lookup:** `extensions/nvidia/index.ts:89`
  - body: Dynamic resolution can receive provider-stripped model IDs, while the new catalog stores NVIDIA-owned IDs such as `nvidia/nemotron-...`. This exact comparison misses known rows, drops metadata, and can return a bare upstream model ID.
  - confidence: 0.87
