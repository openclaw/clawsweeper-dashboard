Full review comments:

- **[P1] Use auth option objects with the single-provider helper:** `extensions/nvidia/index.ts:24`
  - body: `defineSingleProviderPluginEntry` expects auth option objects and wraps them with `createProviderApiKeyAuthMethod` itself. Passing a built auth method here gives the wrapper the wrong shape, so setup/auth registration can read missing option fields such as `methodId`, `optionKey`, or `envVar`.
  - confidence: 0.92
- **[P1] Write NVIDIA defaults under models.providers:** `extensions/nvidia/index.ts:37-44`
  - body: This `applyConfig` creates a top-level `providers` object, but OpenClaw provider config is stored under `models.providers`. Onboarding through this auth method would leave NVIDIA defaults in an ignored location.
  - confidence: 0.9
- **[P1] Preserve NVIDIA literal model IDs during lookup:** `extensions/nvidia/index.ts:88-105`
  - body: Dynamic resolution receives provider-stripped model IDs, while the catalog stores entries such as `nvidia/nemotron-...`; the exact lookup misses known rows and can fall back to a bare model ID that conflicts with the current literal-prefix NVIDIA behavior.
  - confidence: 0.88
- **[P2] Keep static catalog rows visible without credentials:** `extensions/nvidia/index.ts:65-70`
  - body: When no API key is available, the live catalog returns `null` and no `staticRun` fallback is registered. That removes the unauthenticated static NVIDIA rows current main exposes for listing and picker paths.
  - confidence: 0.86
- **[P2] Declare discovery config before reading it:** `extensions/nvidia/index.ts:63-64`
  - body: The branch reads `ctx.config?.models?.nvidiaDiscovery?.enabled`, but that key is not part of the current config contract or plugin schema. Either declare a typed plugin config/schema surface or remove the undeclared compatibility path.
  - confidence: 0.83
