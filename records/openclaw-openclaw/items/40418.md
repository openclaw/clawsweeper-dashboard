## Summary

Keep open: current main and v2026.6.10 still provide only mechanical reset-memory capture, artifact sanitization, and bounded startup loading; they do not implement the requested structured synthesis, dedupe, continuation package, transcript anchors, or start-of-session summary generation. The remaining work is product- and security-sensitive memory lifecycle design, not conservative cleanup.

## What This Changes

The issue requests automatic preservation, synthesis, and rehydration of session memory around `/new` session resets.

## Best Possible Solution

Define a maintainer-approved memory lifecycle contract first: an opt-in, sanitized continuation package with curated summaries, transcript receipts or anchors, retention and redaction controls, untrusted historical-context framing, and tests across `/new`, `/reset`, compaction, daily/idle rollover, and startup hydration.

## Reproduction Assessment

Not applicable: this is a feature/design request, not a broken existing behavior report. Source inspection confirms current main has partial capture, sanitization, and startup loading, but not the requested synthesis workflow.

## Solution Assessment

No for the raw full-dump shape in the issue body: automatic broad transcript export is not the safest maintainable default. A bounded, sanitized, opt-in continuation package with explicit retention, provenance, and trust boundaries is the better direction.
