## Summary

Current main partially addresses session continuity with sanitized command-reset memory capture, bounded startup loading of daily/slugged memory notes, and compaction-triggered agentic flushes, but it does not implement the requested automatic synthesis, dedupe, anchor, or full-transcript continuation package around resets. This remains useful but needs maintainer product and security design before code work.

## What This Changes

The issue requests automatic preservation and synthesis of session memory around `/new` so resets carry a compact durable continuation package and links to full transcripts.

## Best Possible Solution

Design a maintainer-approved reset/memory lifecycle path that extends the existing memory plugin or flush contracts, or documents plugin alternatives, with explicit defaults, retention/redaction rules, and bounded startup/reset behavior.

## Reproduction Assessment

Yes at source level for the missing behavior: current main captures sanitized recent messages on `/new`/`/reset`, loads bounded existing notes at startup, and runs agentic memory flush only on the reply/compaction path; I did not run a live reset because this review is read-only and the request is a feature/design gap.

## Solution Assessment

Unclear: reset-time agentic flush is plausible and tracked more concretely by https://github.com/openclaw/openclaw/issues/45608, but defaults, blocking/background behavior, provider use, transcript retention, dedupe, and security boundaries need maintainer decision.
