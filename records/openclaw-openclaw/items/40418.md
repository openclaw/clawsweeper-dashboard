## Summary

Keep open: current main and v2026.6.6 cover part of the continuity problem with raw session-memory artifacts, startup-context loading, and pre-compaction memory flush, but they do not implement structured /new synthesis, anchored transcript links, or a settled privacy/security lifecycle. The linked synthesis PR is closed unmerged and conflicting, and the raw-memory poisoning follow-up is still open.

## What This Changes

Requests automatic /new session memory preservation: save reset context, export or link full transcripts, synthesize compact daily memory, and hydrate the next session from that summary.

## Best Possible Solution

Define the continuation package in the memory plugin/session-reset boundary: preserve full transcripts through the existing session store, write a bounded sanitized summary with receipt links, and make retention, quoting, indexing, and reset hydration explicit and reviewable.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Source inspection confirms the current behavior is raw reset-memory capture plus separate startup-context and compaction-flush paths, not the requested synthesis workflow.

## Solution Assessment

No: the proposed raw `.session_memory_preserve.json` and broad transcript dump is not the best default by itself. The safer solution is a memory-plugin-owned, bounded, sanitized continuation package with explicit retention and trust boundaries.
