## Summary

Current main preserves a recent conversation excerpt during `/new` and `/reset`, but it does not provide the requested synthesis, durable-fact deduplication, transcript linking, or compact next-session hydration. The request remains meaningful, while its canonical owner, retention policy, and safe reinjection behavior require maintainer product and security decisions.

## What This Changes

Add automatic preservation and synthesis of session context across `/new`, with durable transcript access and a compact continuation package for the next session.

## Best Possible Solution

Define one canonical memory lifecycle that keeps complete transcripts in the existing SQLite session store and lets the selected memory owner generate a small, sanitized, deduplicated continuation artifact containing durable facts, decisions, open actions, and stable transcript references.

## Reproduction Assessment

Not applicable; this is a feature request rather than a broken existing contract. Source inspection does provide a high-confidence baseline: current main captures only a recent excerpt into a memory file during reset.

## Solution Assessment

No; the proposed JSON and Markdown sidecar workflow is not the best repository-aligned solution. A database-backed design under one canonical memory owner should address synthesis, deduplication, retention, sanitization, and successor-session hydration together.

- Implementing the proposed JSON and Markdown sidecars would create another file-backed state path beside the canonical SQLite session store and conflict with current storage architecture.
