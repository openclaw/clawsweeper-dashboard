## Label Justifications

- `P2`: This is a meaningful session-memory continuity improvement with limited blast radius, but it needs product/security decisions rather than emergency handling.
- `impact:session-state`: The request is about preserving and rehydrating session memory across reset and session-boundary workflows.
- `impact:data-loss`: The reported gap can leave useful decisions, corrections, and follow-up context unavailable after a reset unless it is explicitly preserved.
- `impact:security`: The proposed automatic memory persistence and synthesis can retain or transmit sensitive transcript content.

## Summary

Keep open: current main and v2026.6.8 cover explicit `/new` and `/reset` session-memory capture plus startup loading of recent memory files, but the requested synthesis, transcript-linking, and privacy-safe memory finalization behavior is still a product/security design item with only partial overlap in related trackers.

## What This Changes

Request automatic session-memory preservation and synthesis around `/new` resets, including durable summaries and links back to full transcripts.

## Best Possible Solution

Define one opt-in, bounded memory-finalization contract that builds on the existing session-memory and startup-context paths, with explicit privacy, dedupe, retention, summary, and transcript-reference semantics.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/design request. Source review does show current main has command-only capture and startup loading, while synthesis, transcript linking, and broader finalization semantics are absent.

## Solution Assessment

Unclear as requested: a raw `.session_memory_preserve.json` or full transcript dump would not be the best default. The safer direction is a bounded opt-in lifecycle contract that reuses the existing memory/startup paths and resolves privacy, dedupe, retention, and poisoning concerns first.
