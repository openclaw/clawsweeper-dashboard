## Summary

Keep open: current main and v2026.5.7 still use one full-query FTS lookup for provider-backed hybrid memory search, so the central fix remains useful. PR head still needs changes because it routes no-provider FTS-only search through the same capped fallback helper, reducing current fallback recall.

## What This Changes

This PR refactors memory-core hybrid keyword search through a capped per-keyword FTS fallback, re-enables hybrid keyword tests, adds a fan-out bound test, and updates the changelog.

## Best Possible Solution

Keep the bounded provider-backed hybrid FTS fallback, preserve current no-provider FTS-only recall, add regression coverage for that boundary, then refresh and validate the PR.

## Reproduction Assessment

Yes, by source inspection. Current main searches every extracted keyword in no-provider FTS-only mode, while PR head routes that path through a helper that slices fallback terms to six, so a no-provider query whose only indexed hit is later than the cap can be lost.

## Solution Assessment

No, not as-is. The helper and cap are a good shape for provider-backed hybrid fan-out, but the no-provider FTS-only path should keep its uncapped fallback unless maintainers intentionally choose a recall reduction and add coverage for it.
