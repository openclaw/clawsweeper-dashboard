## Summary

Keep open: current main still throws before `cron.status` or `cron.list` can run in self-remove-scoped cron tool contexts, matching the v2026.5.4 logs in the report. The remaining fix is real, but it changes a cron tool authorization boundary and should get maintainer review before implementation.

## What This Changes

The issue asks for cron-triggered heartbeat/status workflows to safely query their own cron metadata with `status` or `list` without reopening broad cron mutation permissions.

## Best Possible Solution

Add a scoped read-only introspection path for cron-triggered non-owner jobs that permits `status` and tightly filtered/self `list` while preserving the existing mutation guard for add/update/run/remove.

## Reproduction Assessment

Yes, from source inspection and existing tests: a self-remove-scoped cron tool rejects `list` before dispatch, and the same guard would reject `status`. I did not run a live gateway in this read-only review.

## Solution Assessment

Unclear pending maintainer policy review. The narrow maintainable direction is read-only self-introspection only, with `list` forced to the current job or current agent scope and all mutation paths still denied.
