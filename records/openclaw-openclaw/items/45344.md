## Summary

Keep this PR open: current main still lacks configured-agent validation for explicit OpenAI-compatible gateway agent selectors, so the reported bug remains real. The submitted branch is the right general direction, but it is not merge-ready because it is stale against current default-agent/model-selector behavior and would introduce compatibility regressions around omitted allowlists and default aliases.

## What This Changes

The PR adds an `InvalidGatewayAgentIdError`, validates explicit OpenAI-compatible gateway agent selectors against configured agent IDs in chat and Responses HTTP handlers, and adds regression tests for unknown header/model selectors.

## Best Possible Solution

Land a rebased gateway patch that validates only explicit agent selectors against the current runtime config, returns `400 invalid_request_error` for unknown selectors on `/v1/chat/completions` and `/v1/responses`, preserves implicit default routing plus `openclaw/default` and configured-default aliases, and adds focused tests for known, unknown, and default selectors across the helper and both HTTP endpoints.

## Reproduction Assessment

Yes. The PR body gives concrete HTTP requests, and current-main code inspection shows explicit header/model selectors are normalized and used without configured-agent validation before dispatch; live verification was not run because this was a read-only review.

## Solution Assessment

No, not as submitted. The validation direction is the narrow maintainable fix, but the branch should be rebased onto the current runtime config/default-selector contract and should avoid breaking omitted-validation callers or `openclaw/default` routing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Skip validation when the allowlist is absent:** `src/gateway/http-utils.ts:84-85`
  - body: `knownAgentIds` is optional, but `!knownAgentIds?.includes(...)` throws for every explicit header/model selector when callers omit it. Current compat surfaces such as embeddings still call `resolveAgentIdForRequest` without a list, so a rebase would turn valid explicit selectors into unexpected `InvalidGatewayAgentIdError`s; either guard only when the list is supplied or make all callers pass one.
  - confidence: 0.88
- **[P2] Preserve default-agent selector aliases:** `src/gateway/http-utils.ts:100-105`
  - body: The new resolver treats `openclaw/default` as the literal agent id `default` and falls back to hard-coded `main`. Current main routes `openclaw` and `openclaw/default` through `resolveDefaultAgentId`, so configs whose default agent is not a literal `main` entry would reject or route to the wrong agent.
  - confidence: 0.86
