## Summary

Keep open. Current main still lacks configured-agent rejection for explicit OpenAI-compatible HTTP agent selections, so the central fix remains useful, but this PR is stale against current default-agent and alias routing and lacks real after-fix proof.

## What This Changes

Adds gateway validation for explicit agent IDs in Chat Completions and Responses HTTP handling, maps unknown IDs to 400 errors, and adds focused gateway tests.

## Best Possible Solution

Validate explicit gateway agent selections against configured agents plus the effective default alias in the shared resolver, preserve current agent-first/default alias behavior, return 400 for unknown explicit IDs in both HTTP endpoints, and land with focused tests plus real gateway proof.

## Reproduction Assessment

Yes. From source inspection, current main accepts unknown explicit header/model agent IDs because the shared resolver returns them before any configured-agent validation; I did not run a live gateway in this read-only review.

## Solution Assessment

No. The direction is right, but the branch should be rebased to preserve configured default-agent routing and `openclaw/default`, and it needs after-fix real behavior proof before merge.

- **[P2] Preserve configured default-agent routing:** `src/gateway/http-utils.ts:105`
  - body: Current main routes unselected `openclaw` requests through the configured default agent, but this branch falls back to literal `main`.   - confidence: 0.88
- **[P2] Preserve the default model alias before validation:** `src/gateway/http-utils.ts:84`
  - body: Current main treats `model: "openclaw/default"` as a stable alias for the configured default agent. This branch parses it as an explicit agent id named `default` and rejects it unless such an agent exists, so existing clients using the documented alias would get a 400.
  - confidence: 0.86
- **[P3] Make omitted allowlist behavior explicit:** `src/gateway/http-utils.ts:84`
  - body: `knownAgentIds` is optional on the resolver parameters, but this check throws for every explicit header/model selection when the option is omitted. Either require the list on validating paths or guard `undefined` as no validation so future direct callers do not get surprising rejections.
  - confidence: 0.82
