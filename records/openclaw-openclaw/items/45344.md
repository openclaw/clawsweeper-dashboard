## Summary

Keep open: the PR addresses a real gateway routing bug, but the branch is stale against current main, would regress configured default-agent routing and the `openclaw/default` alias, and still lacks after-fix real gateway proof.

## What This Changes

The PR adds configured-agent allowlist validation for explicit OpenAI-compatible gateway agent selectors and returns 400 errors for unknown IDs in Chat Completions and Responses.

## Best Possible Solution

Rebase the validation onto current gateway code so only explicit unknown agent selectors are rejected while preserving configured default routing, `openclaw/default`, header precedence, and `x-openclaw-model`, then add focused regressions and redacted live gateway proof.

## Reproduction Assessment

Yes, source-level. Current main accepts explicit header/model agent IDs without a configured-agent allowlist and both OpenAI-compatible handlers use those resolved values before dispatch; I did not run a live gateway in this read-only review.

## Solution Assessment

No. The validation direction is right, but this branch is not the best merge shape until it is rebased onto current gateway contracts and includes after-fix real behavior proof.

  - confidence: 0.89

  - body: `knownAgentIds` is optional on the resolver params, but this condition rejects every explicit selector when it is omitted. Make the allowlist required on validating paths or treat `undefined` as no validation so direct callers have a coherent contract.
  - confidence: 0.84
