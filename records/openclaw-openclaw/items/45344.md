## Summary

Keep open: the gateway bug is still source-reproducible on current main, and the PR is a plausible focused fix, but the submitted patch has blocking regressions around optional allowlists and configured default-agent aliases. The external PR also lacks after-fix real behavior proof from a real gateway run.

## What This Changes

The PR adds allowlist validation for explicit OpenAI-compatible chat and Responses agent selectors and maps unknown IDs to 400 invalid_request_error, with helper and endpoint tests.

## Best Possible Solution

Land a rebased gateway patch that validates explicit selectors against configured agents while preserving configured default aliases, optional helper semantics, model override behavior, and focused endpoint tests.

## Reproduction Assessment

Yes, source-level. Current main resolves explicit header and model agent selectors without configured-agent validation before OpenAI-compatible chat and Responses dispatch; live HTTP was not run in this read-only review.

## Solution Assessment

No, not as submitted. The HTTP-boundary validation direction is right, but the patch needs rebase work and fixes for optional allowlists and configured default-agent aliases before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Skip validation when no allowlist is supplied:** `src/gateway/http-utils.ts:84-85`
  - body: `knownAgentIds` is optional, but this condition throws for every explicit header/model selector when callers omit it. Existing and future helper callers that do not pass a list would get unexpected `InvalidGatewayAgentIdError`s, so guard only when the list is supplied or make the allowlist required everywhere.
  - confidence: 0.88
- **[P2] Preserve configured default-agent aliases:** `src/gateway/http-utils.ts:100-105`
  - body: This resolver treats `openclaw/default` as the literal agent id `default` and falls back to hard-coded `main`. Current docs and main route `openclaw` and `openclaw/default` through the configured default agent, so configs whose default agent is not literal `main` would reject or route incorrectly.
  - confidence: 0.86
