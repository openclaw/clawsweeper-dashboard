## Summary

Keep open: current main still has the strict validator path that rejects nested `toolResult`/`tool_result`-style content blocks and replaces the middleware result with the generic post-processing error. The linked routing and announce-timing issues are useful cluster context, but neither supersedes this middleware data-loss path and I found no open PR carrying the fix.

## What This Changes

The issue requests preserving nested tool-result-like content through tool-result middleware instead of replacing the result with `Tool output unavailable due to post-processing error`.

## Best Possible Solution

Add a bounded coercion path in `src/agents/harness/tool-result-middleware.ts` that preserves valid text/image blocks, flattens recognized nested tool-result-like blocks into safe text within existing limits, preserves validated details, and still fails closed for unsalvageable middleware output.

## Reproduction Assessment

Yes, at source level: a middleware candidate containing `content: [{ type: "toolResult", ... }]` cannot pass the current content-block validator and will take the fallback error path. I did not execute the live Codex/channel flow during this read-only review.

## Solution Assessment

Yes, a narrow middleware coercion/sanitization fix is the maintainable direction because the Codex bridge already expects post-middleware output to be converted to text/image app-server items. Expanding the validator alone would not safely bridge arbitrary nested tool-result payloads into the existing output contract.
