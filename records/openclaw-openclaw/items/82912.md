## Summary

Keep open: current main still has a source-provable path where tool-result middleware rejects non-text/image content blocks and replaces the tool result with the generic post-processing error. The linked companion routing issue is related but does not supersede this middleware data-loss path.

## What This Changes

The issue requests that tool-result middleware preserve or coerce nested toolResult/tool_result/tool/function content blocks instead of discarding the whole tool result.

## Best Possible Solution

Add a bounded coercion path in the middleware runner that preserves valid text/image blocks, flattens recognized nested tool-result-like blocks into safe text within existing limits, and keeps strict failure behavior for unsalvageable outputs.

## Reproduction Assessment

Yes, from source inspection: a middleware result containing `content: [{ type: "toolResult", ... }]` cannot pass the current content-block validator and will take the fallback error path. I did not execute the live Codex/channel flow in this read-only pass.

## Solution Assessment

Yes, a narrow coercion/sanitization fix in `src/agents/harness/tool-result-middleware.ts` is the maintainable direction because the Codex bridge already expects middleware output to be converted back to text/image before returning to app-server.
