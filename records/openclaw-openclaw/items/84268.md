## Summary

Keep this PR open, but it should not merge as-is: the bug is real and the supplied terminal proof is useful, yet the branch does not account for current main’s durable source-reply mirror and can duplicate visible replies after reload while also reconstructing history from raw tool arguments.

## What This Changes

The PR adds gateway chat-history and live session-history projection logic that mirrors successful current-session `message.send` tool replies into assistant display history when the final assistant message is `NO_REPLY`.

## Best Possible Solution

Keep the legacy-history backfill idea, but make it defer to the current `sourceReplyTranscriptMirror`/delivery-mirror path and mirror only sanitized delivered content when no durable mirror already exists.

## Reproduction Assessment

Yes, by source inspection: current main already appends durable source-reply delivery mirrors, and this PR flushes a second synthetic mirror at `NO_REPLY` without checking for that persisted reply. I did not run tests because this review is read-only.

## Solution Assessment

No, not as written. A projection fallback is reasonable for legacy transcripts, but the maintainable fix should integrate with the current delivery-mirror contract and sanitizer instead of independently reconstructing visible replies from raw tool-call arguments.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Skip synthetic mirrors when a persisted mirror already exists:** `src/gateway/chat-display-projection.ts:826`
  - body: Current `main` already mirrors successful internal source replies into the transcript through `sourceReplyTranscriptMirror` and `appendAssistantMessageToSessionTranscript`. For a normal new transcript shaped like message tool call, successful result, `NO_REPLY`, then the persisted delivery-mirror assistant message, flushing here inserts an extra assistant copy before the dropped `NO_REPLY`, so reload history can show the same reply twice. Guard this fallback behind the absence of an existing delivery mirror or use the canonical mirror path directly.
  - confidence: 0.88
- **[P2] Mirror sanitized delivered text instead of raw tool arguments:** `src/gateway/chat-display-projection.ts:557`
  - body: The message tool strips formatted reasoning tags and normalizes reply payloads before delivery, but this projection mirror reads the recorded `message`/`caption` argument and only strips inline reply/audio directives. A raw tool argument containing stripped reasoning or pre-normalized media text can reappear in reload history even though it was not delivered that way; build the mirror from the sanitized result/source-reply payload when available or run the same sanitizer.
  - confidence: 0.82
