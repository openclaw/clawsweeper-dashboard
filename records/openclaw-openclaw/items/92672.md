## Label Justifications

- `P1`: The issue describes a broken real user-facing agent/channel workflow where provider failures can make OpenClaw appear frozen or silent.
- `impact:message-loss`: The central failure mode is that the expected channel-visible failure notice is not delivered to the originating user or thread.
- `impact:session-state`: Rate-limit failures can persist quota suspension state while the user receives no visible explanation of the session state change.
- `impact:auth-provider`: The trigger class is provider rate-limit, billing, overload, timeout, and model fallback behavior.

## Summary

Keep open: current main still has source-backed failover paths where rate-limit/billing/overload failures are recorded or thrown but not consistently delivered as visible channel notices, and the issue is paired with an open unmerged closing PR that does not yet land the delivery hook.

## What This Changes

The issue requests a core message-lifecycle change so model failover failures such as rate limits, billing exhaustion, overloads, and timeouts produce an immediate user-visible channel notification instead of silent dead air.

## Best Possible Solution

Implement one core-owned gateway-failure notice path that reuses failover metadata and message-lifecycle origin tags, adds focused regression coverage for primary and followup failures, and lands through the linked PR or a replacement that proves real channel behavior.

## Reproduction Assessment

Yes, source-level: force the embedded/followup model-fallback path to throw a rate-limit or billing `FailoverError`, and current main records or logs the failure without a guaranteed channel notice on the inspected paths. I did not run a live provider-quota reproduction in this read-only review.

## Solution Assessment

Partly: the issue's core-level gateway-failure delivery direction is the right owner boundary, but the metadata-only linked PR is not enough and user-facing copy/localization needs maintainer direction before merge.
