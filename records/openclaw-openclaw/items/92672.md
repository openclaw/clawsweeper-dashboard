## Summary

Keep open: current main and v2026.6.6 still have failover/followup paths that can log or throw a structured provider failure without delivering a user-visible channel notice, and the open closing PR has not landed. ## What This Changes

Requests a core/channel failure-notice path for rate-limit, billing, and overloaded model failover so users see an immediate message instead of a silent stalled turn.

## Best Possible Solution

Review, repair, and land a narrow core failure-notice path that reuses shared failover classification, marks internal runtime/provider notices for delivery despite normal source suppression when appropriate, and includes focused tests plus redacted real channel/runtime proof.

## Reproduction Assessment

Yes, source-level: forcing a provider billing/rate-limit failure through the current failover or followup path reaches code that throws or logs and returns without a channel-visible notice. I did not run a live provider-quota reproduction in a real channel during this read-only review.

## Solution Assessment

No current implementation solves it. The requested direction is plausible, but the best fix should unify with the existing internal failure-notice bypass and shared failover classifiers rather than merge metadata-only or one-channel plumbing.
