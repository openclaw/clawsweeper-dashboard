## Summary

Keep open. Current main still only denies unlisted Telegram groups through the existing allowlist path; it does not expose the requested configurable ignore, warn, cooldown, or auto-leave behavior, and the same-author PR #50682 is still the paired implementation path.

## What This Changes

Request to add Telegram unknown-group handling so unapproved groups can be ignored, warned, or warned and left before assistant execution.

## Best Possible Solution

Land PR #50682 or a narrow successor that implements the unknown-group action in the bundled Telegram plugin preflight, updates public config types/schema/generated metadata/docs, shares behavior across normal messages and native commands, and validates cooldown plus forum-topic delivery before closing this issue.

## Reproduction Assessment

Yes. Configure Telegram with an explicit groups allowlist, send from an unlisted group, and current main will hit group-chat-not-allowed, drop/log the normal message or send the static native-command denial, with no configurable warning, cooldown, or leave action.

## Solution Assessment

Yes. Telegram inbound preflight is the right boundary because it runs before assistant execution and inference cost; the maintainable implementation should trigger only from the unknown-group chat-allowlist denial and reuse one action helper across message and native-command paths.
