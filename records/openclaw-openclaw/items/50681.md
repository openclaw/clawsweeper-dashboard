## Summary

Keep open. Current main still only denies unlisted Telegram groups by dropping/logging normal messages or returning a static native-command denial, and it does not expose the requested configurable ignore, warn, cooldown, or auto-leave behavior; open PR #50682 remains the paired implementation path.

## What This Changes

The issue requests Telegram config and runtime handling to ignore, warn, or leave unknown groups before assistant execution.

## Best Possible Solution

Land PR #50682 or a narrow successor that implements the action in the bundled Telegram plugin preflight, shares behavior across normal messages and native commands, and updates the public config/docs/test surfaces before closing this issue.

## Reproduction Assessment

Yes, source-level. Configure Telegram with a groups allowlist and send from an unlisted group; current main reaches group-chat-not-allowed, drops/logs normal messages or sends the static native-command denial, and has no configurable warn/cooldown/leave path.

## Solution Assessment

Yes. Telegram inbound preflight is the right boundary because it runs before assistant execution; the narrow maintainable solution is to handle only the unknown group-chat-not-allowed branch and reuse one action helper across message and native-command paths.
