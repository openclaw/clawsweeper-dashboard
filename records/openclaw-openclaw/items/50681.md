## Summary

Keep open. Current main still fail-closes unknown Telegram groups by denying/dropping them, but it does not provide the requested configurable ignore, warning, cooldown, or auto-leave action; the same-author PR #50682 remains open as the paired implementation path.

## What This Changes

Request to add Telegram unknown-group handling that can ignore, warn, or leave unapproved groups before assistant execution.

## Best Possible Solution

Land PR #50682 or a narrow successor that implements the behavior in Telegram inbound preflight, updates public config types/schema/generated metadata/docs, shares the action path across normal messages and native commands, preserves forum-topic warning delivery, and validates cooldown plus approved-group behavior before this issue is closed.

## Reproduction Assessment

Yes. Configure Telegram with an explicit groups allowlist, send from an unlisted group, and current main reaches group-chat-not-allowed/not-allowed without any configurable warning or leaveChat path; the same behavior is visible in source and existing tests.

## Solution Assessment

Yes. Telegram inbound preflight is the right boundary because it runs before assistant execution; the maintainable implementation should distinguish unknown chat denial from sender authorization failure and reuse one helper for normal messages and native commands.
