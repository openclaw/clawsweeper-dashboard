## Summary

Keep this PR open because it addresses a real Claude approval-boundary bug, but the submitted implementation is not merge-safe: it makes a non-contract `session_key` mandatory, lacks after-fix real behavior proof, and is currently conflicting with main.

## What This Changes

The PR adds a required Claude permission `session_key`, stores it with pending permission requests, checks `yes/no <request_id>` replies against the inbound session, and adds focused regression tests.

## Best Possible Solution

Land a rebased security fix that preserves Claude Code's four-field permission-request contract, derives and stores the originating OpenClaw session server-side, rejects cross-session replies, validates binding data, and includes focused tests plus real MCP/Claude proof.

## Reproduction Assessment

Yes, by source inspection. Current main records pending Claude permissions by request id only and resolves any matching `yes/no <id>` message regardless of inbound session, so a focused bridge regression can seed one trusted pending request and send an attacker-session reply.

## Solution Assessment

No. Binding replies to the originating session is the right direction, but making undocumented `session_key` mandatory is not the safest implementation; OpenClaw should preserve the upstream schema and derive the binding from server-side context.
