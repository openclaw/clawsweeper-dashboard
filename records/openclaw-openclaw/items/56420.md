## Summary

Keep open: current main still accepts Claude permission replies by request id only, but this PR needs narrow changes before merge because it breaks an existing MCP channel sender, does not normalize/reject blank session keys, and lacks the required changelog entry.

## What This Changes

The PR adds a `session_key` field to Claude permission-request notifications, stores it with pending MCP permission requests, and adds tests for rejecting replies from other sessions.

## Best Possible Solution

Land a rebased session-bound permission reply fix that keeps all in-repo senders compatible, stores only normalized non-empty session keys, includes regression coverage and a changelog entry, and composes cleanly with #71648 if it lands first.

## Reproduction Assessment

Yes. On current main, a pending Claude permission keyed as `abcde` will be accepted from any non-empty session message containing `yes abcde` or `no abcde` because the handler checks only the request id; this review verified the path by source inspection rather than by running tests.

## Solution Assessment

No, not as submitted. Binding replies to the originating session is the right fix boundary, but the patch should update the remaining sender contract and normalize/reject blank session keys before merge.
