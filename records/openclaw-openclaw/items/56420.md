## Summary

Keep open. Current main still accepts Claude permission replies by request id only, and this PR targets the right security boundary, but the branch is not merge-ready because it makes `session_key` required without updating every covered sender, stores unnormalized/blank session keys, lacks the required changelog entry, and is currently dirty against main.

## What This Changes

The PR adds `session_key` to Claude MCP permission-request notifications, stores it with pending Claude permissions, and only accepts `yes/no <request_id>` replies from the matching session.

## Best Possible Solution

Land a rebased session-bound permission reply fix with a normalized non-empty `session_key` contract, all in-repo permission-request senders updated, regression coverage, a changelog entry, and clean composition with #71648 if that PR lands first.

## Reproduction Assessment

Yes. On current main, seed `pendingClaudePermissions` for `abcde` and send `handleSessionMessageEvent` from another non-empty `sessionKey` with `yes abcde`; the source path checks only the request id and emits the permission notification.

## Solution Assessment

No, not as submitted. Binding replies to the originating session is the narrow maintainable fix, but the patch must keep covered senders compatible and reject or normalize blank/padded session keys before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep permission-request senders in sync:** `src/mcp/channel-shared.ts:122`
  - body: By making `session_key` required in `ClaudePermissionRequestSchema`, this patch requires every `notifications/claude/channel/permission_request` sender to include it. The Docker MCP channel e2e client still sends that notification without `session_key`, so that covered flow will fail schema validation once this lands; update the caller or keep the contract backward-compatible while deriving the session another way.

A blank or padded request key can enter `pendingClaudePermissions` but never match a reply; normalize and require a non-empty session key before storing it.
  - confidence: 0.88
