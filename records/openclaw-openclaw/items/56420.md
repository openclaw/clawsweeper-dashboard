## Summary

Keep this PR open: current main still accepts Claude permission replies by request id only, so the security bug is real, but the submitted patch has narrow contract and changelog blockers before merge.

## What This Changes

The PR adds a required `session_key` to Claude permission-request notifications, stores that key with pending Claude permissions, accepts `yes/no <request_id>` only from the matching session, and adds regression coverage.

## Best Possible Solution

Land a rebased session-bound permission reply fix with a normalized non-empty `session_key` contract, all in-repo senders updated, a changelog entry, and clean composition with #71648 if that pending-map lifecycle PR lands first.

## Reproduction Assessment

Yes. On current main, seed `pendingClaudePermissions` for `abcde` and send a `session.message` from any other non-empty `sessionKey` with `yes abcde`; the handler checks only the request id and emits the permission notification.

## Solution Assessment

No, not as submitted. Binding replies to the originating session is the right fix boundary, but the patch should keep senders compatible and normalize/reject blank session keys before merge.
