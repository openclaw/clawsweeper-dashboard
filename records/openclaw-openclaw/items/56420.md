## Summary

Keep this PR open, but not merge-ready: the underlying approval-spoofing bug is real, while this branch makes a required `session_key` protocol change that conflicts with Claude Code's documented permission-request contract and still lacks after-fix real behavior proof.

## What This Changes

The PR adds `session_key` to Claude MCP permission-request handling, stores it with pending permission entries, checks it before accepting `yes/no` replies, and adds focused tests.

## Best Possible Solution

Land a rebased design that preserves Claude Code's four-field permission-request schema, derives the originating session from OpenClaw-side context, rejects cross-session replies, normalizes non-empty session data, and includes focused tests plus real MCP/Claude proof.

## Reproduction Assessment

Yes, by source inspection: current main records pending Claude permissions by request id only and resolves any matching `yes/no <id>` message regardless of the inbound session. A focused bridge regression can seed one trusted pending permission, send an attacker-session reply, and observe the unbound approval path.

## Solution Assessment

No. Binding replies to the originating session is the right direction, but requiring an undocumented `session_key` from Claude Code is not the safest implementation; OpenClaw should preserve the documented request schema and derive the binding from server-side context.

- **[P1] Keep Claude permission requests on the documented schema:** `src/mcp/channel-shared.ts:122`
  - body: Claude Code documents `permission_request` params as `request_id`, `tool_name`, `description`, and `input_preview`; it does not send `session_key`. Making this field required can reject real permission requests before the bridge records them, so legitimate remote approvals can stop working.
  - confidence: 0.94
- **[P3] Reject blank session keys before storing permissions:** `src/mcp/channel-shared.ts:122`
  - body: The new schema accepts empty strings and the server stores the raw value, while reply handling later uses `toText()`, which trims blank strings to `undefined`. A blank or whitespace-only value creates a pending permission that can never match a reply.
  - confidence: 0.82
- **[P3] Normalize stored session keys before comparison:** `src/mcp/channel-server.ts:43`
  - body: `params.session_key` is passed through raw, but inbound message session keys are normalized by `toText()` before comparison. Surrounding whitespace in the notification would make the legitimate session unable to resolve its own pending permission.
  - confidence: 0.79
