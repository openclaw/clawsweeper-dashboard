## Summary

Keep open: current main still accepts Claude permission replies by request id only, and this PR fixes the right security boundary but has merge-blocking sender compatibility and session-key normalization gaps plus a missing changelog entry.

## What This Changes

The PR adds a required `session_key` to Claude permission-request notifications, stores it on pending Claude permissions, rejects replies from other sessions, and adds MCP channel regression tests.

## Best Possible Solution

Land a rebased session-bound permission reply fix that updates all in-repo permission-request senders, normalizes and requires non-empty session keys, adds the required changelog entry, and composes cleanly with #71648 if that pending-map lifecycle change lands first.

## Reproduction Assessment

Yes. On current main, seed `pendingClaudePermissions` for `abcde` and send `yes abcde` from any other non-empty `sessionKey`; the handler checks only the request id before emitting the permission notification.

## Solution Assessment

No, not as submitted. Binding replies to the originating session is the right fix boundary, but the patch must keep in-repo senders compatible and normalize/reject blank session keys before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep permission-request senders in sync:** `src/mcp/channel-shared.ts:122`
  - body: By making `session_key` required in `ClaudePermissionRequestSchema`, this patch requires every sender of `notifications/claude/channel/permission_request` to include it. The Docker MCP channel e2e client still sends that notification without `session_key`, so that covered flow will fail schema validation once this lands; update the caller or keep the contract backward-compatible while deriving the session another way.
  - confidence: 0.87

A blank or padded request key can enter `pendingClaudePermissions` but never match a reply; normalize and require a non-empty session key before storing it.
  - confidence: 0.88

  - body: This is a user-facing MCP channel security hardening fix, but the PR does not add an Unreleased changelog entry. Add a single-line `### Fixes` entry with public wording and contributor credit.
