## Summary

Keep open. Current main still accepts Claude permission replies by request id alone, and this PR moves in the right direction, but the patch is not merge-ready because the new required `session_key` contract is incomplete, blank keys can become permanently unresolved pending permissions, and the required changelog entry is missing.

## What This Changes

The PR adds `session_key` to Claude permission request notifications, stores it with pending Claude permissions, accepts yes/no replies only from the matching session, and adds regression coverage for cross-session rejection.

## Best Possible Solution

Land the session-bound permission reply fix with a trimmed, non-empty `session_key` contract, update all in-repo senders and MCP docs/changelog, and make sure the pending-map lifecycle work in #71648 composes with the final stored request shape.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction on current main is to create a pending Claude permission for `abcde`, then send a `session.message` from a different non-empty `sessionKey` containing `yes abcde`; the handler checks only the request id and emits the permission notification.

## Solution Assessment

No, not as submitted. Binding replies to the originating session is the right fix boundary, but the patch should normalize and reject blank session keys and update existing permission-request senders before merge.
