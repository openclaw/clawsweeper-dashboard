## Summary

Keep open: current main still has the request-id-only Claude permission approval path, and this PR is directionally the right security fix, but it is not merge-ready because it makes `session_key` required without updating an existing packaged MCP channel smoke producer and it lacks after-fix real behavior proof.

## What This Changes

The PR adds `session_key` to Claude permission request notifications, stores it with pending MCP Claude permissions, and only accepts `yes/no` replies from the same session with regression tests.

## Best Possible Solution

Keep the session-bound permission design, update all `permission_request` producers and schema normalization, then land after acceptable live MCP proof and current-head validation.

## Reproduction Assessment

Yes by source inspection. Current main stores pending Claude permissions by request id only and resolves any matching `yes/no <id>` message without checking the message session; I did not run tests because this review must keep the checkout read-only.

## Solution Assessment

No, not yet. Binding pending permissions to the originating session is the narrow maintainable fix, but this implementation is incomplete until repository producers are updated, `session_key` is normalized/non-empty, and real behavior proof is supplied.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Update the MCP channel smoke client for session_key:** `src/mcp/channel-shared.ts:129`
  - body: By making `session_key` required here, the existing packaged Docker MCP channel client still sends `notifications/claude/channel/permission_request` without that field. The schema will reject that notification before `handleClaudePermissionRequest` runs, so the smoke flow will time out waiting for the permission notification after `yes abcde`. Please update that producer to pass the session key it already read from the channel notification.
  - confidence: 0.92
- **[P3] Reject or normalize blank session keys:** `src/mcp/channel-shared.ts:129`
  - body: `z.string()` accepts `""` and whitespace, while the reply path reads inbound message sessions through `toText()`, which rejects blank strings. A blank stored `sessionKey` would create a pending permission that can never be resolved; use a trimmed non-empty schema boundary before storing it.
  - confidence: 0.78
