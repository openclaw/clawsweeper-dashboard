## Summary

Keep open: current main still accepts Claude permission replies by request id only, and the PR targets the right security boundary, but it should not merge as-is because it breaks an in-repo permission-request sender, can strand blank or padded session keys, and lacks external real behavior proof.

## What This Changes

The PR adds `session_key` to Claude MCP permission-request notifications, stores it with pending Claude permissions, and accepts yes/no replies only from the matching session.

## Best Possible Solution

Land a rebased session-bound permission reply fix that uses a normalized non-empty session key, keeps all in-repo senders compatible, and composes cleanly with https://github.com/openclaw/openclaw/pull/71648 if that pending-map lifecycle PR lands first.

## Reproduction Assessment

Yes, by source inspection. On current main, seed `pendingClaudePermissions` for `abcde` and send a `session.message` from a different non-empty `sessionKey` with `yes abcde`; the handler checks only the request id and emits the permission notification.

## Solution Assessment

No, not as submitted. Binding replies to the originating session is the right fix boundary, but the patch should keep senders in sync and normalize/reject blank session keys before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep permission-request senders in sync:** `src/mcp/channel-shared.ts:119`
  - body: By making `session_key` required in `ClaudePermissionRequestSchema`, this patch requires every sender of `notifications/claude/channel/permission_request` to include it. The Docker MCP channel e2e client still sends that notification without `session_key`, so that covered flow will fail schema validation once this lands; update that sender or keep the contract backward-compatible while deriving the session another way.
  - confidence: 0.88
- **[P2] Normalize and reject blank permission session keys:** `src/mcp/channel-server.ts:43`
  - body: `params.session_key` is stored raw while replies compare against `toText(payload.sessionKey)`, which trims and treats blank strings as absent. A blank or padded request key can enter `pendingClaudePermissions` but never match a reply; normalize and require a non-empty session key before storing it.
  - confidence: 0.87
