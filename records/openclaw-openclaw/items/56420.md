## Summary

This PR addresses a real security bug that is still present on current main, but it should stay open because the branch is conflicting, lacks required real behavior proof, and introduces a required `session_key` without updating every current permission-request producer.

## What This Changes

The PR adds a Claude MCP permission `session_key`, stores it with pending permissions, rejects `yes/no` replies from other sessions, and adds regression coverage.

## Best Possible Solution

Land a rebased session-binding fix that uses trimmed non-empty session keys, updates every permission-request producer, and includes real MCP/Claude channel proof.

## Reproduction Assessment

Yes, by source inspection. Current main stores Claude permission requests without a session identifier and resolves any matching `yes/no <id>` message by request id only, so a focused regression can call the bridge with a trusted request and an attacker-session reply.

## Solution Assessment

No, not yet. Binding pending permissions to the originating session is the right direction, but this patch needs producer updates, trimmed non-empty session-key handling, a rebase, and real behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Update every permission request producer for session_key:** `src/mcp/channel-shared.ts:122`
  - body: By making `session_key` required, the Docker MCP channel smoke client still sends `notifications/claude/channel/permission_request` without it. That notification will fail schema parsing before the bridge stores the pending permission, so the smoke/client flow can time out instead of producing the expected permission reply.
  - confidence: 0.92
- **[P3] Reject blank session keys before storing permissions:** `src/mcp/channel-shared.ts:122`
  - body: The new schema accepts empty and whitespace-only strings, while reply handling reads inbound sessions through `toText()`, which trims and turns blanks into undefined. A blank stored `sessionKey` can never match a reply, so require a trimmed non-empty value before storing it.
  - confidence: 0.78
