## Summary

Keep open. Current main still has the cross-session Claude permission reply bug, but this PR needs source-branch fixes and contributor-supplied real behavior proof before it is merge-ready.

## What This Changes

The PR adds a `session_key` field to Claude MCP permission-request notifications, stores it with pending Claude permissions, and only accepts `yes/no <id>` replies from the matching session.

## Best Possible Solution

Land a rebased session-bound permission reply fix that preserves or updates all permission-request senders, stores a normalized non-empty session key, and composes with the pending-map lifecycle work if it lands first.

## Reproduction Assessment

Yes, by source inspection. On current main, a different session can send `yes <request_id>` for an existing pending Claude permission because the reply handler checks only the request id before sending the permission notification.

## Solution Assessment

No, not as submitted. Binding replies to the originating session is the right fix boundary, but the patch should keep senders compatible and normalize/reject blank session keys before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep permission-request senders in sync:** `src/mcp/channel-shared.ts:119`
  - body: By making `session_key` required in `ClaudePermissionRequestSchema`, this patch requires every sender of `notifications/claude/channel/permission_request` to include it. The Docker MCP channel e2e client still sends that notification without `session_key`, so that covered flow will fail schema validation once this lands; update that sender or keep the contract backward-compatible while deriving the session another way.
  - confidence: 0.88
- **[P2] Normalize and reject blank permission session keys:** `src/mcp/channel-server.ts:43`
  - body: `params.session_key` is stored raw while replies compare against `toText(payload.sessionKey)`, which trims and treats blank strings as absent. A blank or padded request key can enter `pendingClaudePermissions` but never match a reply; normalize and require a non-empty session key before storing it.
  - confidence: 0.87
