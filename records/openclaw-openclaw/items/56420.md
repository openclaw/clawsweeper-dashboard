## Summary

Keep open: current main still accepts Claude permission replies by short request id alone, and this PR is the right direction but remains incomplete because it misses an existing producer, accepts blank session keys, lacks after-fix real behavior proof, and is currently conflicting with main.

## What This Changes

The PR adds a required Claude MCP permission `session_key`, stores it with pending permission state, and only accepts `yes/no` replies from the matching session.

## Best Possible Solution

Land a rebased session-binding fix that uses a trimmed non-empty session key, updates every permission request producer, and proves the real MCP/Claude permission flow.

## Reproduction Assessment

Yes, by source inspection. Current main stores Claude permission requests without a session identifier and resolves any matching `yes/no <id>` message by request id only; a focused regression can call the bridge with a trusted request and an attacker-session reply.

## Solution Assessment

No, not yet. Binding pending permissions to the originating session is the narrow maintainable fix, but this PR must update all producers, reject or normalize blank session keys, and provide live behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Update all permission_request producers for session_key:** `src/mcp/channel-shared.ts:129`
  - body: By making `session_key` required here, the existing Docker MCP channel smoke client still sends `notifications/claude/channel/permission_request` without that field. The schema will reject the notification before `handleClaudePermissionRequest` runs, so the smoke flow can time out waiting for the permission notification after `yes abcde`; pass the session key from that producer too.
  - confidence: 0.92
- **[P3] Reject or normalize blank session keys:** `src/mcp/channel-shared.ts:129`
  - body: `z.string()` accepts empty and whitespace-only values, while the reply path reads inbound message sessions through `toText()`, which trims and rejects blanks. A blank stored `sessionKey` would create a pending permission that can never be resolved; use a trimmed non-empty schema boundary before storing it.
  - confidence: 0.78
