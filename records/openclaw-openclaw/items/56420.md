## Summary

Keep open. The PR targets a real MCP/Claude approval-boundary bug, but the submitted fix makes `session_key` a required Claude permission-request field even though Claude Code documents the notification as a four-field contract, so it can break legitimate permission relay before the security fix takes effect. ([code.claude.com](https://code.claude.com/docs/en/channels-reference))

## What This Changes

The PR adds `session_key` to Claude MCP permission requests, stores it with pending permission entries, checks it before accepting `yes/no <id>` replies, and adds focused tests.

## Best Possible Solution

Land a rebased design that preserves Claude Code's four-field `permission_request` schema, derives the originating OpenClaw session server-side, rejects cross-session replies, normalizes non-empty session data, and includes real MCP/Claude proof.

## Reproduction Assessment

Yes, by source inspection: current main stores pending Claude permissions by request id only and resolves any matching `yes/no <id>` message regardless of the inbound session. A focused bridge regression can seed one trusted pending permission, send an attacker-session reply, and observe the unbound approval path.

## Solution Assessment

No. Binding replies to the originating session is the right direction, but requiring an undocumented `session_key` from Claude Code is not the safest implementation; OpenClaw should preserve the documented request schema and derive the binding from server-side context.
