## Summary

Keep open: current main still has the request-id-only Claude permission reply path, and this PR is useful security hardening, but the patch is incomplete because it breaks an existing repository producer, accepts blank session keys, lacks real behavior proof, and is currently conflicting.

## What This Changes

The PR adds session keys to Claude MCP permission requests, stores them with pending permission state, and accepts `yes/no` replies only from the originating session.

## Best Possible Solution

Keep the session-bound permission design, update every repository `permission_request` producer and normalize a non-empty session key at the schema boundary, then land after rebase, maintainer review, and real behavior proof.

## Reproduction Assessment

Yes, by source inspection. Current main stores Claude permission requests without a session identifier and resolves any matching `yes/no <id>` message by request id only; I did not run tests because this review is read-only.

## Solution Assessment

No, not yet. Binding pending permissions to the originating session is the narrow maintainable fix, but this PR must update all producers, reject or normalize blank session keys, and provide live behavior proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Update the MCP channel smoke client for session_key:** `src/mcp/channel-shared.ts:129`
  - body: By making `session_key` required here, the existing packaged Docker MCP channel client still sends `notifications/claude/channel/permission_request` without that field. The schema will reject that notification before `handleClaudePermissionRequest` runs, so the smoke flow can time out waiting for the permission notification after `yes abcde`; update the producer to pass the session key it already uses for the channel notification.
  - confidence: 0.92
- **[P3] Reject or normalize blank session keys:** `src/mcp/channel-shared.ts:129`
  - body: `z.string()` accepts empty and whitespace-only values, while the reply path reads inbound message sessions through `toText()`, which rejects blanks. A blank stored `sessionKey` would create a pending permission that can never be resolved; use a trimmed non-empty schema boundary before storing it.
  - confidence: 0.78
