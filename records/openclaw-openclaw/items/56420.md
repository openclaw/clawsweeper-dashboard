## Summary

Keep open: this targets a real security-sensitive permission-routing gap, but the current implementation is not merge-safe because it changes Claude Code's documented permission request schema and still lacks real behavior proof.

## What This Changes

The PR adds `session_key` to Claude MCP permission requests, stores it with pending permissions, gates `yes/no <id>` replies by session, and adds focused regression tests.

## Best Possible Solution

Land a rebased design that preserves Claude Code's documented permission request schema, binds request IDs to OpenClaw-side originating session context, rejects cross-session replies, validates normalized non-empty session data, and proves the real MCP/Claude flow.

## Reproduction Assessment

Yes, by source inspection. Current main records pending Claude permissions by request id only and resolves any matching `yes/no <id>` message regardless of the inbound session.

## Solution Assessment

No. Binding replies to the originating session is the right direction, but requiring an undocumented `session_key` from Claude Code is not the safest implementation; OpenClaw should preserve the documented request schema and derive the binding from server-side context.

- **[P1] Keep Claude permission requests on the documented schema:** `src/mcp/channel-shared.ts:122`
  - body: Claude Code documents `permission_request` params as `request_id`, `tool_name`, `description`, and `input_preview`; it does not send `session_key`. Making that field required rejects real permission requests and the current Docker smoke notification before the bridge records the request, so legitimate remote approvals can stop working. ([code.claude.com](https://code.claude.com/docs/en/channels-reference?utm_source=openai))
  - confidence: 0.94
- **[P3] Reject blank session keys before storing permissions:** `src/mcp/channel-shared.ts:122`
  - body: The new schema accepts empty strings, and the server stores the raw value while reply handling later uses `toText()`, which trims blanks to `undefined`. A blank or whitespace-only value creates a pending permission that can never match a reply, so require a normalized non-empty session key wherever the final design obtains it.
  - confidence: 0.82
