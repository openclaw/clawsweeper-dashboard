## Summary

Keep open: current main still accepts Claude permission replies by request id alone, and this PR is a useful security-direction fix, but the submitted patch has two merge-blocking compatibility/normalization defects and lacks required after-fix real behavior proof.

## What This Changes

The PR adds a `session_key` field to Claude MCP permission-request notifications, stores it with pending Claude permissions, checks matching sessions before accepting `yes/no` replies, and adds regression coverage.

## Best Possible Solution

Land a rebased session-bound permission reply fix that uses a normalized non-empty session key, keeps every in-repo permission-request sender compatible, adds focused regression coverage, and composes cleanly with the pending map lifecycle work if that lands first.

## Reproduction Assessment

Yes, by source inspection. On current main, seed `pendingClaudePermissions` for `abcde` and send a `session.message` from a different non-empty `sessionKey` with `yes abcde`; the handler checks only the request id and emits the permission notification.

## Solution Assessment

No, not as submitted. Binding permission replies to the originating session is the right fix boundary, but this patch should preserve/update all senders and normalize/reject blank session keys before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep permission-request senders in sync:** `src/mcp/channel-shared.ts:119`
  - body: By making `session_key` required in `ClaudePermissionRequestSchema`, this patch requires every sender of `notifications/claude/channel/permission_request` to include it. The Docker MCP channel e2e client still sends that notification without `session_key`, so that covered flow will fail schema validation once this lands; update the sender or keep the contract backward-compatible while deriving the session another way.
  - confidence: 0.88
- **[P2] Normalize and reject blank permission session keys:** `src/mcp/channel-server.ts:43`
  - body: `params.session_key` is stored raw while replies compare against `toText(payload.sessionKey)`, which trims and treats blank strings as absent. A blank or padded request key can enter `pendingClaudePermissions` but never match a reply; normalize and require a non-empty session key before storing it.
  - confidence: 0.87
