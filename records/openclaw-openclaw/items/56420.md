## Summary

Keep open. Current main still accepts Claude permission replies by request id alone, so the security hardening is valid, but the proposed patch is not merge-ready: it changes the permission-request schema without updating all in-repo senders and it permits blank/raw session keys that can leave permission requests unresolved.

## What This Changes

The branch adds a required `session_key` to Claude permission request notifications, stores that key with pending Claude permissions, gates yes/no replies on a matching session, and adds MCP channel regression coverage.

## Best Possible Solution

Land the session-bound permission reply fix after normalizing and requiring a non-empty `session_key`, updating all in-repo senders plus MCP docs and changelog, and composing the stored request shape with the pending-map lifecycle work in #71648.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction on current main is to create a pending Claude permission for `abcde`, then send a `session.message` from a different non-empty `sessionKey` containing `yes abcde`; the handler checks only the request id and emits the permission notification.

## Solution Assessment

No, not as submitted. Binding permission replies to the originating session is the right boundary, but the patch should keep senders compatible and normalize/reject blank session keys before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep permission-request senders in sync:** `src/mcp/channel-shared.ts:122`
  - body: By making `session_key` required in the notification schema, this patch requires every `notifications/claude/channel/permission_request` sender to include it. The Docker MCP channel e2e client still sends that notification without `session_key`, so that flow will fail schema validation before the handler runs; update that caller or keep the boundary backward-compatible while deriving the session another way.
  - confidence: 0.88
- **[P2] Reject blank permission session keys:** `src/mcp/channel-shared.ts:122`
  - body: The new schema accepts any string and the server stores `params.session_key` raw, while replies normalize `payload.sessionKey` through `toText`. Empty or whitespace-only request keys can be stored but never match a reply, leaving the permission unresolved; trim and require a non-empty session key at the schema/storage boundary.
  - confidence: 0.89
