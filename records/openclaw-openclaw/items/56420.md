## Summary

Keep this PR open: current main still has the unbound Claude permission-reply behavior, but this branch is not merge-ready because it makes an upstream permission-request field mandatory, lacks real behavior proof, and is currently conflicting with main.

## What This Changes

The PR requires `session_key` in Claude MCP permission requests, stores it on pending permission entries, gates `yes/no <request_id>` replies on matching inbound `sessionKey`, and adds focused tests.

## Best Possible Solution

Land a rebased security fix that preserves Claude Code's four-field permission-request contract, derives and stores the originating OpenClaw session server-side, rejects cross-session replies, validates non-empty binding data, and includes focused tests plus real MCP/Claude proof.

## Reproduction Assessment

Yes, by source inspection. Current main records pending Claude permissions by request id only and resolves any matching `yes/no <id>` message regardless of inbound session, so a focused bridge regression can seed one trusted pending request and send an attacker-session reply.

## Solution Assessment

No. Binding replies to the originating session is the right direction, but making undocumented `session_key` mandatory is not the safest implementation; OpenClaw should preserve the upstream schema and derive the binding from server-side context.
