## Summary

Keep open. Current main still accepts Claude permission replies by short request id only, but this PR is not merge-ready because the new required `session_key` contract is not propagated to all in-repo senders, does not reject/normalize blank keys, and needs the required user-facing changelog entry.

## What This Changes

The PR adds `session_key` to Claude permission request notifications, stores it with pending Claude permissions, checks it before accepting `yes/no <id>` replies, and adds regression coverage for cross-session rejection.

## Best Possible Solution

Land a rebased session-bound permission reply fix with a normalized non-empty `session_key` contract, updated in-repo senders/docs/changelog, and composition with #71648 if that pending-map lifecycle PR lands first.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction on current main is to seed `pendingClaudePermissions` for `abcde`, then send a `session.message` from a different non-empty `sessionKey` containing `yes abcde`; the handler checks only the request id and emits the permission notification.

## Solution Assessment

No, not as submitted. Binding replies to the originating session is the right fix boundary, but the patch should update every sender of the required notification field and reject/normalize blank session keys before merge.

## Review Findings

Full review comments:

- **[P2] Keep permission-request senders in sync:** `src/mcp/channel-shared.ts:122`
  - body: By making `session_key` required in `ClaudePermissionRequestSchema`, this patch requires every sender of `notifications/claude/channel/permission_request` to include it. The Docker MCP channel e2e client still sends that notification without `session_key`, so that covered flow will fail schema validation once this lands; update the caller or keep the contract backward-compatible while deriving the session another way.
  - confidence: 0.87
- **[P2] Reject blank permission session keys:** `src/mcp/channel-shared.ts:122`
  - body: `session_key: z.string()` accepts empty or whitespace-only strings, and the server stores `params.session_key` while replies compare against `toText(payload.sessionKey)`, which trims and treats blanks as absent. A blank or padded request key can enter `pendingClaudePermissions` but never match a reply; normalize and require a non-empty session key before storing it.
  - confidence: 0.88
- **[P3] Add a changelog entry:** `CHANGELOG.md:14`
  - body: This is a user-facing MCP channel security hardening fix, but the PR does not add an Unreleased changelog entry. Add a single-line `### Fixes` entry with public wording and contributor credit.
