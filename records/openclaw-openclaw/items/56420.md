## Summary

Keep this PR open because the underlying cross-session approval risk is still present on current main, but this patch should not merge as written: it makes `session_key` a required Claude Code permission request field even though the current Claude Code channel contract sends only `request_id`, `tool_name`, `description`, and `input_preview`. The PR also has only test/check claims, not after-fix real MCP/Claude Code behavior proof.

## What This Changes

This PR adds a `session_key` field to Claude permission request handling, stores it with pending permission requests, rejects yes/no replies from other sessions, and adds regression tests for trusted versus attacker sessions.

## Best Possible Solution

Preserve the security goal, but redesign it around the supported Claude Code channel contract or an explicitly reviewed upstream/API correlation field, then add regression coverage and real relay proof.

## Reproduction Assessment

Yes, from source inspection: current main stores pending Claude permissions by short request id and accepts a matching `yes/no <id>` from any inbound session. I did not execute the regression test because this was a read-only review.

## Solution Assessment

No. The session-binding idea is the right security direction, but making `session_key` required on `notifications/claude/channel/permission_request` is not the best fix because it violates the current Claude Code channel contract.
