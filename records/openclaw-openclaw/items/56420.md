## Summary

Keep open. Current main still accepts Claude permission replies by short request id alone, so the PR targets a real security-sensitive MCP channel bug, but the submitted patch is not merge-ready because the new required `session_key` contract is not propagated to the Docker MCP e2e client and the schema/storage path accepts blank or unnormalized session keys.

## What This Changes

The PR adds `session_key` to Claude permission request notifications, stores it with pending Claude permissions, requires yes/no replies to come from the matching session, and adds regression coverage for cross-session rejection.

## Best Possible Solution

Land the session-bound permission reply fix with a trimmed, non-empty `session_key` contract, update all in-repo senders plus MCP docs and changelog, and make sure the pending-map lifecycle work in #71648 composes with the new stored request shape.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction on current main is to create a pending Claude permission for `abcde`, then send a `session.message` from a different non-empty `sessionKey` containing `yes abcde`; the handler checks only the request id and emits the permission notification.

## Solution Assessment

No, not as submitted. Binding replies to the originating session is the right boundary, but the patch should normalize/reject blank session keys and update the existing permission-request sender contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Keep permission-request senders in sync:** `src/mcp/channel-shared.ts:122`
  - body: By making `session_key` required in the notification schema, this patch requires every `notifications/claude/channel/permission_request` sender to include it. The current Docker MCP channel e2e client still sends that notification without `session_key`, so that flow will fail schema validation once this lands; update that caller or keep the contract backward-compatible while deriving the session another way.
  - confidence: 0.86
- **[P2] Reject blank permission session keys:** `src/mcp/channel-shared.ts:122`
  - body: This accepts `session_key` as a plain `z.string()` and stores `params.session_key` raw, while replies read `payload.sessionKey` through `toText`, which trims and converts blanks to `undefined`. Empty or whitespace-only request keys can enter `pendingClaudePermissions` but never match a reply, leaving the permission unresolved; use a trimmed non-empty schema and store the normalized value.
  - confidence: 0.88

  - body: This is a user-facing MCP channel fix/security hardening, but the PR does not add an `Unreleased` changelog entry. Add a single-line `### Fixes` entry with appropriate public wording and contributor credit.
