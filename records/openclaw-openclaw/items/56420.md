## Summary

Keep open. Current main still accepts a Claude permission reply from any session that knows the short request id, and this PR targets the right security boundary, but the submitted patch is not merge-ready because it changes the permission-request contract without updating every in-repo sender and accepts unresolved blank session keys.

## What This Changes

The PR adds `session_key` to Claude permission request notifications, stores it with pending Claude permissions, requires `yes/no <id>` replies to come from the matching session, and adds regression coverage for cross-session rejection.

## Best Possible Solution

Land the session-bound permission reply fix with a trimmed, non-empty `session_key` contract, update every in-repo permission-request sender, document the MCP notification contract, add the required changelog entry, and compose the stored request shape with the pending-map lifecycle work in #71648.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction on current main is to seed `pendingClaudePermissions` for `abcde`, then send a `session.message` from a different non-empty `sessionKey` containing `yes abcde`; the handler checks only the request id and emits the permission notification.

## Solution Assessment

No, not as submitted. Binding replies to the originating session is the right fix boundary, but the patch should normalize and reject blank session keys and update the existing sender/docs/changelog contract before merge.
