## Summary

Keep open: the PR targets a real approval-boundary bug, but the current patch adds a required Claude Code permission-request field outside the upstream contract and the branch is conflicting, so it needs redesign and maintainer/security review before merge.

## What This Changes

The PR adds a required Claude permission `session_key`, stores it with pending permission requests, checks `yes/no <request_id>` replies against the inbound session, and adds focused regression tests.

## Best Possible Solution

Preserve Claude Code's permission-request schema, derive or persist the originating session from OpenClaw-owned channel context, rebase around the pending-map lifecycle PR, and add both focused tests and redacted real MCP proof.

## Reproduction Assessment

Yes, source-reproducible with high confidence. Current main stores pending Claude permissions by request id only and resolves any matching `yes/no <id>` message regardless of which inbound session sent it; I did not run tests because this review is read-only.

## Solution Assessment

No, not as proposed. Binding replies to session is the right security direction, but adding a required non-contract `session_key` to the Claude Code notification schema is not the safest implementation path.
