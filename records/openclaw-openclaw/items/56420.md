## Summary

Keep open. Current main still accepts Claude MCP permission replies by short request id from any non-empty session, so the security fix is still useful, but this PR needs source fixes and real behavior proof before merge.

## What This Changes

The PR adds `session_key` to Claude MCP permission-request notifications, stores it on pending permissions, gates yes/no replies by session key, and adds session-binding tests.

## Best Possible Solution

Land a rebased session-bound permission reply fix that carries a normalized non-empty session key from every permission-request sender into pending entries and composes with https://github.com/openclaw/openclaw/pull/71648 if that lifecycle work lands first.

## Reproduction Assessment

Yes, by source inspection. Current main stores Claude permission requests by short request id only and accepts a later `yes/no <id>` from any non-empty session, while the docs describe the linked conversation as the reply source.

## Solution Assessment

No, not as submitted. Binding replies to the originating session is the right fix boundary, but the patch must keep all permission-request senders in sync and normalize or reject blank session keys before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep permission-request senders in sync:** `src/mcp/channel-shared.ts:119`
  - body: By making `session_key` required in `ClaudePermissionRequestSchema`, this patch requires every sender of `notifications/claude/channel/permission_request` to include it. The Docker MCP channel e2e client still sends that notification without `session_key`, so that covered flow will fail schema validation once this lands.
  - confidence: 0.9
- **[P2] Normalize and reject blank permission session keys:** `src/mcp/channel-server.ts:43`
  - body: `params.session_key` is stored raw while replies compare against `toText(payload.sessionKey)`, which trims and treats blank strings as absent. A blank or padded request key can enter `pendingClaudePermissions` but never match a reply; normalize and require a non-empty session key before storing it.
  - confidence: 0.88
