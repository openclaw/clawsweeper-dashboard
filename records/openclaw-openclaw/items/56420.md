## Summary

Keep open: current main still accepts any matching Claude permission reply by short request id, and this PR targets the right security boundary, but the patch is not merge-ready because it adds a required `session_key` that is not in Claude Code's documented four-field `permission_request` contract, is conflicting with current main, and lacks real behavior proof. ([code.claude.com](https://code.claude.com/docs/en/channels-reference?utm_source=openai))

## What This Changes

The PR adds a required `session_key` to Claude MCP permission requests, stores it with pending permissions, rejects replies from other sessions, and adds regression coverage.

## Best Possible Solution

Land a rebased design that keeps Claude Code's documented permission request schema, binds request IDs to OpenClaw-side originating session context, rejects cross-session replies, validates non-empty normalized session data, and proves the real MCP/Claude flow.

## Reproduction Assessment

Yes, by source inspection: current main records pending Claude permissions by request id only and resolves any matching `yes/no <id>` message regardless of the inbound session. A focused bridge regression can seed one trusted pending permission, send an attacker-session reply, and observe the existing unbound approval path.

## Solution Assessment

No, not yet. Binding replies to the originating session is the right direction, but requiring an undocumented `session_key` from Claude Code is not the safest implementation; OpenClaw should preserve the documented request schema and derive the binding from server-side context.

  - body: Claude Code documents `permission_request` params as `request_id`, `tool_name`, `description`, and `input_preview`; it does not send `session_key`. ([code.claude.com](https://code.claude.com/docs/en/channels-reference?utm_source=openai))
  - confidence: 0.94
- **[P3] Reject blank session keys before storing permissions:** `src/mcp/channel-shared.ts:122`
  - body: The new schema accepts empty strings, and the server stores the raw value while reply handling later uses `toText()`, which trims blanks to `undefined`. A blank or whitespace-only value creates a pending permission that can never match a reply, so require a normalized non-empty session key wherever the final design obtains it.
  - confidence: 0.82
