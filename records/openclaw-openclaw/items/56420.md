## Summary

Keep open. Current main still has the request-id-only Claude permission reply path, but this PR is not merge-ready because its required `session_key` contract is not applied to an existing Docker MCP channel sender, blank or padded session keys can leave permissions unresolved, and the required changelog entry is missing.

## What This Changes

The PR adds a required `session_key` to Claude permission request notifications, stores it with pending permissions, accepts only matching-session `yes/no <id>` replies, and adds MCP channel regression tests.

## Best Possible Solution

Land a rebased session-bound permission reply fix with a normalized non-empty `session_key` contract, synced senders/docs/changelog, and composition with #71648 if that pending-map lifecycle PR lands first.

## Reproduction Assessment

Yes. A high-confidence source reproduction on current main is to seed `pendingClaudePermissions` for `abcde`, then send a `session.message` from a different non-empty `sessionKey` containing `yes abcde`; the handler checks only the request id and emits the permission notification.

## Solution Assessment

No, not as submitted. Binding replies to the originating session is the right fix boundary, but the patch should update every sender of the required notification field, reject or normalize blank session keys, add the changelog entry, and rebase cleanly.
