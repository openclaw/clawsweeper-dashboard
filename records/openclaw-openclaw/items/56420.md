## Summary

Keep open: the underlying permission-spoofing issue is still present on current main, but this PR is not merge-ready because it breaks a covered permission-request sender, stores unnormalized session keys, and lacks after-fix real behavior proof.

## What This Changes

The PR adds a Claude MCP permission `session_key`, stores it with pending permission requests, and only accepts `yes/no <id>` replies from the matching session.

## Best Possible Solution

Land a rebased session-bound permission reply fix that carries a normalized non-empty session key from every permission-request sender into the pending entry, updates focused coverage, and composes with the pending-map lifecycle work if it lands first.

## Reproduction Assessment

Yes, by source inspection. Current main stores Claude permission requests by short request id only and accepts any later non-empty session that sends `yes/no <id>`, so the spoofing path is clear without a live daemon run.

## Solution Assessment

No, not as submitted. Binding replies to the originating session is the right fix boundary, but the patch must keep senders in sync and normalize or reject blank session keys before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep permission-request senders in sync:** `src/mcp/channel-shared.ts:119`
  - body: By making `session_key` required in `ClaudePermissionRequestSchema`, this patch requires every sender of `notifications/claude/channel/permission_request` to include it. The Docker MCP channel e2e client still sends that notification without `session_key`, so that covered flow will fail schema validation once this lands.
  - confidence: 0.9
- **[P2] Normalize and reject blank permission session keys:** `src/mcp/channel-server.ts:43`
  - body: `params.session_key` is stored raw while replies compare against `toText(payload.sessionKey)`, which trims and treats blank strings as absent. A blank or padded request key can enter `pendingClaudePermissions` but never match a reply; normalize and require a non-empty session key before storing it.
  - confidence: 0.88
