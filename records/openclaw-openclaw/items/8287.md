## Summary

Current main still lacks node-advertised tool schemas or node-owned tool inventory, and no viable canonical item or merged change supersedes the request. The issue should remain open because the capability is real but needs a maintainer product and security decision on the abstraction.

## What This Changes

Allow connected nodes to advertise agent tool schemas that the Gateway exposes and routes without a wrapper plugin.

## Best Possible Solution

Add one additive Gateway-owned node command-schema capability with strict validation, provenance, authorization, deterministic lifecycle, and catalog support, then expose it through the smallest maintainer-approved model surface—preferably the generic nodes tool unless namespaced first-class tools demonstrate clear UX value without weakening policy.

## Reproduction Assessment

Yes at source level: current main accepts node commands but not their agent-tool definitions or schemas, and the runtime exposes only the generic nodes tool. This is a missing capability rather than a regression requiring a failing runtime reproduction.

## Solution Assessment

No, the proposed first-class dynamic-tool design is not yet proven best. A Gateway-owned command-schema contract projected through the existing nodes abstraction is a narrower alternative that should be compared before expanding the model tool namespace.

## Risks / Open Questions

- Node-defined model tools expand a remote execution boundary: schemas, names, descriptions, and command mappings must not bypass pairing, node command policy, approvals, or tools.allow/tools.deny.
- Dynamic connection lifecycle can make model tool inventories nondeterministic or stale unless ordering, reconnect idempotency, conflict handling, and in-flight calls have one Gateway owner.
- Synthesizing many first-class tools may increase prompt and schema churn; enriching the existing nodes tool with validated command schemas may be the safer product shape.
