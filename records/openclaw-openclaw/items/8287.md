## Summary

Keep open: current main and latest release still expose node caps/commands and plugin tools, but not node-advertised LLM tool definitions, node-owned tool inventory, or a Gateway lifecycle/policy contract for synthesizing node-backed tools. This remains a valid feature/design item rather than cleanup-close material.

## What This Changes

Request a Gateway node protocol/runtime path for connected nodes to advertise LLM agent tools that route directly through node.invoke instead of requiring wrapper plugins and /tools/invoke loopback.

## Best Possible Solution

Keep this open for a maintainer-approved generic node tool/schema advertisement contract with strict Gateway validation, node provenance, command-policy enforcement, deterministic lifecycle, catalog/effective inventory support, docs, and tests; service-specific integrations should remain plugin/ClawHub-owned.

## Reproduction Assessment

Yes at source level: current main and v2026.6.6 accept node command names but not node-owned tool definitions or schemas, and runtime exposes only the generic nodes tool. This is a missing capability rather than a broken existing behavior with a failing runtime repro.

## Solution Assessment

Unclear; first-class dynamic node tools may be the right UX, but maintainers should compare that against a narrower Gateway-owned command-schema contract projected through the existing nodes abstraction before expanding the model tool namespace.
