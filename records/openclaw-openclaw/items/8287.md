## Summary

Current main still exposes one static `nodes` agent tool and stores only node capability and command names; it has no protocol, registry lifecycle, or tool-inventory path for node-advertised schemas. The narrower command-schema report was closed in favor of this issue, so this remains the canonical feature and security-design thread.

## What This Changes

Allow connected nodes to advertise schema-defined agent tools that the gateway dynamically exposes and routes through node invocation.

## Best Possible Solution

Keep this issue as the canonical design thread and define one additive, validated node command-schema contract with explicit opt-in model-tool exposure, authenticated node provenance, deterministic run snapshots, collision/version rules, disconnect cleanup, and enforcement through the existing `node.invoke` authorization path.

## Reproduction Assessment

Not applicable: this requests a new protocol and runtime capability rather than reporting broken established behavior. Current-main source inspection does provide high-confidence confirmation that node tool/schema advertisement is absent.

## Solution Assessment

Not yet. The desired outcome is useful, but accepting an arbitrary `tools` array directly is incomplete; the maintainable solution should unify node command schemas and controlled model-tool exposure under one generic, security-reviewed protocol contract.

## Risks / Open Questions

- A paired or compromised node could advertise deceptive names or overbroad schemas unless every generated tool is bound to its authenticated node session and permitted command set.

- Naming collisions, reconnect idempotency, schema-version compatibility, provenance in catalog/effective inventory, and `tools.allow`/`tools.deny` semantics remain product decisions.
