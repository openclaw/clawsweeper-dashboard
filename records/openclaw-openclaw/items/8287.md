## Summary

Keep open; current main and the latest release still accept node caps/commands only and expose agent tools from core/plugin/channel/MCP paths, so node-advertised first-class LLM tools remain a real Gateway protocol/runtime design gap.

## What This Changes

The issue requests a Gateway/node protocol feature that lets connected nodes advertise agent tool definitions and route those tool calls directly through node invocation without a wrapper plugin.

## Best Possible Solution

Keep this as the canonical design thread for an additive, maintainer-approved node tool-advertisement contract with validated schemas, node provenance in inventories, policy-aware execution through node.invoke, reconnect/unregister lifecycle handling, and docs for naming and security behavior.

## Reproduction Assessment

Yes for the source-level feature gap, not as a runtime bug reproduction. Current main shows node connections carry caps/commands/permissions, NodeRegistry stores command strings, and tool inventories have no node-backed tool source.

## Solution Assessment

Unclear until maintainers choose the product/security boundary. The requested direction is plausible, but the maintainable solution is a designed additive protocol/runtime extension rather than an autonomous quick fix or a Tesla-specific wrapper.
