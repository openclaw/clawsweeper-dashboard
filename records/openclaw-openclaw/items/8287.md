- `impact:security`: The requested capability would let remote nodes define model-callable execution surface, so it affects trust, authorization, and policy boundaries.

## Summary

Keep open: current main and v2026.6.6 still model nodes as command hosts behind the static `nodes` tool, not as dynamic first-class model tool providers; the closed command-schema issue was already routed here as the canonical design thread.

## What This Changes

Request to let connected nodes advertise model-callable agent tools directly through the Gateway instead of requiring wrapper plugins.

## Best Possible Solution

Keep this as the canonical maintainer design thread for additive node tool/schema advertisement, dynamic node-backed tool lifecycle, node provenance in catalog/effective inventory, and policy routing through existing node.invoke authorization.

## Reproduction Assessment

Yes at source level: current main has node command names, `node.invoke`, plugin-registered tools, and MCP tool projection, but no node-advertised first-class LLM tool schema or dynamic node tool registration path. This is a feature gap, not a broken existing behavior with a runtime repro.

## Solution Assessment

Yes, keeping this open for maintainer-approved protocol/runtime design is the best current path. A wrapper plugin is the supported workaround, but it does not solve the requested node-owned advertisement and lifecycle model.
