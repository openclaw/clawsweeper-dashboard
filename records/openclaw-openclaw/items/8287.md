## Label Justifications

- `P2`: This is a useful bounded product capability with real integration demand, but existing plugin and generic nodes-tool paths remain functional.
- `impact:security`: The proposal would let connected nodes supply model-visible schemas and executable commands, expanding the node-to-agent trust and authorization boundary.

## Summary

Current main removes the original token-reading/HTTP-loopback pain through the plugin node runtime, but it still does not let a connected node advertise model-visible tool definitions; the remaining protocol and authorization design is substantive and has been explicitly retained as the canonical discussion.

## What This Changes

Request connected nodes to advertise first-class agent tool schemas that the Gateway dynamically exposes and routes without a wrapper plugin.

## Best Possible Solution

Keep the plugin-owned wrapper as the supported near-term path, and design any core addition as an additive, validated node advertisement contract whose model-visible tools retain node provenance and execute through the existing `node.invoke` authorization path.

## Reproduction Assessment

Not applicable: this is a feature request, not broken defined behavior. Current-main source inspection gives a high-confidence capability check because the strict connect schema, NodeRegistry state, and agent tool assembly have no node-advertised tool-definition path.

## Solution Assessment

Partly: direct node advertisement is a coherent way to achieve zero-plugin dynamic tools, but the issue's original HTTP workaround should now use `api.runtime.nodes`; maintainers still need to decide whether node-authored model schemas are an acceptable core trust boundary.
