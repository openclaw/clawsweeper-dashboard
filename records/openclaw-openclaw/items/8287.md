## Summary

Keep this canonical feature request open. Current main still exposes node command names rather than node-advertised tool schemas, and the narrower command-schema request was intentionally consolidated here; however, automatically turning remote node declarations into model-callable tools needs a maintainer-approved product and security design.

## What This Changes

Add an additive Gateway/node protocol that lets connected nodes advertise first-class LLM tool schemas and dynamically provide node-backed agent tools.

## Best Possible Solution

Keep integration-specific tool definitions plugin-owned and add a narrow, authorized in-process node command/schema runtime seam that removes HTTP and token loopback; pursue automatic node-to-LLM tool synthesis only if maintainers explicitly approve a stricter trusted advertisement contract with provenance, lifecycle, collision, and policy rules.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection provides high-confidence confirmation that current main carries node command names but has no node-advertised tool schemas or dynamic node tool source.

## Solution Assessment

No, the proposed automatic registration is not yet established as the best solution. A plugin-owned tool definition backed by a narrow in-process node invocation/schema API better matches current ownership and security boundaries unless maintainers determine that nodes should become a separately trusted tool source.

## Risks / Open Questions

- A connected node would be able to influence model-visible executable affordances, so the design needs a clear trust boundary rather than treating successful pairing as permission to publish arbitrary LLM tools.
- Schema size and complexity limits, reserved names, collisions across nodes/plugins/core, provenance, reconnect idempotency, and disconnect behavior remain product and protocol decisions.
- Authorization must define how advertised tools compose with declared command checks, plugin node-invoke policy, approvals, and tools.allow/tools.deny without creating a second or weaker policy path.
- Automatically changing the available tool set as nodes connect or disconnect may affect prompt determinism, active sessions, caching, and user expectations.
