- `impact:security`: The request would let remote nodes define model-callable execution surface, affecting authorization, provenance, and tool policy boundaries.

## Summary

Keep open: current main and v2026.6.8 still expose nodes as declared command hosts behind the static `nodes` tool, not as first-class node-advertised model tools; this remains a core protocol/runtime design item with security implications.

## What This Changes

Request to let connected nodes register first-class LLM agent tools through the Gateway protocol instead of requiring wrapper plugins.

## Best Possible Solution

Keep this as the canonical maintainer design thread for additive node tool/schema advertisement, dynamic node-backed tool lifecycle, node provenance in catalog/effective inventory, and policy-preserving routing through existing node.invoke authorization.

## Reproduction Assessment

Yes at source level: current main has node command names, `node.invoke`, plugin-registered tools, and MCP/plugin tool inventory, but no node-advertised first-class LLM tool schema or dynamic node tool registration path. This is a feature gap, not a broken existing behavior with a runtime repro.

## Solution Assessment

Yes, keeping this open for maintainer-approved protocol/runtime design is the best current path. A wrapper plugin remains the supported workaround, but it does not solve node-owned advertisement, lifecycle, or tool provenance.
