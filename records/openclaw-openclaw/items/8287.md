## Summary

Keep open. Current main still has node-declared commands plus one static `nodes` tool, while first-class custom LLM tools are still registered through plugins; the issue discussion and openclaw-tescmd workaround describe a real core protocol/runtime gap rather than something already implemented or safely closeable.

## What This Changes

Allow connected nodes to advertise tool definitions that the Gateway exposes as first-class LLM tools and routes through node invocation.

## Best Possible Solution

Add an additive Gateway protocol and runtime contract for validated node-advertised tools, with NodeRegistry lifecycle state, synthesized node-backed agent tools, inventory provenance, node.invoke policy enforcement, and docs for naming, reconnect, pairing, and allow/deny behavior.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection on current main confirms the feature gap: connect accepts node caps/commands but not tool definitions, and agent tool assembly exposes static core tools plus plugin tools only.

## Solution Assessment

Unclear until maintainer design. The requested direction addresses a real core protocol/runtime gap, but the maintainable solution should be an additive, validated node tool contract with policy and provenance semantics rather than a one-off wrapper around existing `node.invoke`.
