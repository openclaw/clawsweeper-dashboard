## Summary

Keep open for maintainer design review: current main still exposes node capabilities through the generic `nodes` tool and plugin tools, but connected nodes cannot advertise validated first-class LLM tool definitions. The narrower command-schema report was closed in favor of this broader canonical request, and the existing `openclaw-tescmd` wrapper remains useful evidence of the integration friction rather than a replacement for the missing protocol capability.

## What This Changes

Allow connected nodes to advertise schemas that become first-class LLM tools without requiring a wrapper plugin.

## Best Possible Solution

Define an additive, validated node tool-advertisement contract with explicit node provenance, deterministic connection lifecycle, collision rules, and execution routed exclusively through existing `node.invoke` authorization and tool-policy enforcement.

## Reproduction Assessment

Not applicable: this is a new capability request rather than broken established behavior, and source inspection directly confirms the missing protocol and runtime surfaces.

## Solution Assessment

Unclear: direct node registration is coherent and removes real indirection, but maintainers must choose between first-class node tools and a narrower plugin SDK invocation seam while defining the permanent trust and lifecycle contract.

## Risks / Open Questions

- A paired node defining arbitrary model-visible tools could widen command execution beyond the operator's intended policy unless schema validation, node provenance, pairing trust, name collisions, reconnect lifecycle, and `tools.allow`/`tools.deny` semantics are designed together.
- Dynamically changing the tool inventory during a session could create stale prompt or tool-schema state unless availability changes have one explicit lifecycle owner and deterministic session behavior.
- A parallel node-tool execution path could drift from existing `node.invoke`, command allowlisting, and system-run approval enforcement.
