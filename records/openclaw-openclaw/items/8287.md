## Summary

Keep open. Current main still supports node-declared capabilities/commands and plugin-registered tools that can call nodes, but it does not let connected nodes advertise first-class LLM tool definitions or add/remove node-backed agent tools dynamically; the recent maintainer/Codex discussion also explicitly kept this feature open.

## What This Changes

Allow connected nodes to advertise typed agent tools during the Gateway connection lifecycle so the Gateway can expose them directly to the LLM and route calls through node invocation without a wrapper plugin.

## Best Possible Solution

Add an additive Gateway protocol extension for node-advertised tool definitions, validate and store those definitions per connected NodeRegistry session, synthesize node-backed agent tools and catalog/effective inventory entries with node provenance, route execution through existing node.invoke policy gates, and document naming, reconnect, pairing, and allow/deny behavior.

## Reproduction Assessment

Not applicable as a bug reproduction, but yes for the feature gap: source inspection on current main shows node connect accepts command names but not tool definitions, and the agent runtime only exposes the static nodes tool plus plugin tools.

## Solution Assessment

Unclear until maintainer design. The requested direction addresses a real core protocol/runtime gap, but the maintainable solution needs an additive contract and policy model rather than a narrow one-off wrapper around existing node.invoke.
