## Summary

Keep open. Current main still drops `messageId` on the Slack `read` path, so exact Slack message or thread-reply lookup remains unresolved; the item has no protected labels and no paired implementation was found.

## What This Changes

The issue requests Slack `message` tool `read` support for fetching a specific channel message or thread reply by `messageId`, optionally combined with `threadId`.

## Best Possible Solution

Add exact-message read support inside the Slack plugin path: expose and forward `messageId` for `read`, make `readSlackMessages` use Slack timestamp bounds plus filtering for both channel and thread reads, preserve existing thread history behavior, and update focused tests, docs, and changelog.

## Reproduction Assessment

Yes. The source path is deterministic: a Slack `message` read call with `messageId` reaches the dispatcher/runtime/helper without that parameter being read or forwarded, so current main cannot return only the requested timestamp.

## Solution Assessment

Yes, with one boundary note. The narrow maintainable solution is plugin-local and additive; Slack URL parsing should be added only at the existing Slack target/URL parsing surface or left as a small follow-up if no such surface exists.
