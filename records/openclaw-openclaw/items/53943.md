## Summary

Keep this open. Current main still lacks the Slack exact-message read path: the Slack read dispatcher, runtime case, and read helper only carry history/thread parameters, so a `messageId` for a specific thread reply is still ignored; the related reports cover adjacent thread behavior but not this exact lookup.

## What This Changes

The issue requests Slack `message(action: "read")` support for `messageId`, including filtering to a specific thread reply when both `messageId` and `threadId` are provided.

## Best Possible Solution

Add exact-message read support in the Slack plugin path: expose and forward `messageId` for `read`, make `readSlackMessages` return only the requested Slack timestamp for threaded and channel reads, preserve existing `threadId` behavior, and update focused tests plus public message docs.

Yes. A high-confidence source trace reproduces it: pass `messageId` to Slack `read`, and the current dispatcher/runtime/helper path never forwards or uses that value, so no live Slack workspace is needed to confirm the bug shape.

## Solution Assessment
