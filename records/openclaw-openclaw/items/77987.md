## What This Changes

The branch adds `RateLimitConfig.maxEntries` with a 10,000 default, enforces oldest non-locked entry eviction on new auth-rate-limit entries, and adds flood plus lockout-preservation tests.

## Best Possible Solution

Land the insert-time cap after maintainers explicitly choose the cap semantics and API boundary: SDK-only default, or public `gateway.auth.rateLimit.maxEntries` wiring with schema, docs, and focused validation.

## Reproduction Assessment

Yes. Source inspection shows current main inserts one entry per unique non-loopback scope/IP in `recordFailure` and has no insert-time cap before timer/manual prune runs.

## Solution Assessment

Mostly yes. A gateway-owned insert-time cap is the narrow fix for the reported unique-IP failed-auth flood, but the default/API boundary and saturated-lockout semantics need maintainer acceptance.

## Risks / Open Questions

- The new 10,000-entry default changes exported limiter behavior under high-cardinality failed-auth traffic by evicting oldest non-locked entries instead of retaining every failed-attempt key until prune.
- `maxEntries` is added to the exported `RateLimitConfig` and plugin SDK surface, but current public `gateway.auth.rateLimit` schema and docs do not accept or describe it, so operator tuning remains an explicit API-boundary choice.
- The lockout-preservation rule means a cap saturated by active lockouts will drop new non-locked failure entries until lockouts expire; maintainers should accept that security tradeoff before merge.
