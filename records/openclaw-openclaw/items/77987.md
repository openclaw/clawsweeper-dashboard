## Summary

Keep this PR open for maintainer review: the source change is a focused and credible fix for the unbounded auth-rate-limit map, with sufficient runtime proof, but it adds an exported limiter config/default surface that needs an explicit decision on API-only versus public operator configuration before merge.

## What This Changes

The PR adds `RateLimitConfig.maxEntries` with a 10,000 default, evicts oldest non-locked auth-rate-limit entries on insert, and adds unique-IP flood regression tests.

## Best Possible Solution

Land the bounded-entry mitigation after maintainers decide whether `maxEntries` stays API-only or is wired through the public gateway config type, Zod schema, and docs.

## Reproduction Assessment

Yes. Current `main` shows `recordFailure` creates one map entry per unique non-exempt failed auth key and only prune removes entries later, so the unbounded-between-prunes behavior is source-reproducible.

## Solution Assessment

Mostly yes. An insert-time cap in the gateway-owned limiter is the narrow mitigation, but the public config/API boundary should be decided before merge.
