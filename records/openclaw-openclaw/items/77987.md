## Summary

Keep open: the limiter cap is a plausible narrow hardening fix and the supplied real behavior proof is sufficient, but the patch still exposes `maxEntries` on `RateLimitConfig` without aligning the public `gateway.auth.rateLimit` type, strict schema, docs, or schema coverage, so operator config for the new knob would be rejected.

## What This Changes

The PR adds a default 10,000-entry cap to the gateway auth rate limiter, evicts oldest non-locked entries on new inserts, adds two limiter tests, and updates the changelog.

## Best Possible Solution

Land the bounded auth limiter after either making `maxEntries` internal/test-only or fully adding `gateway.auth.rateLimit.maxEntries` to config types, strict schema, docs, and schema coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows `recordFailure` inserts one Map entry per new non-exempt scope/IP key while only `prune` removes stale unlocked entries, so a unique-IP failure loop before prune grows with unique key count.

## Solution Assessment

No, not yet. The insertion-time cap with locked-entry preservation is the right narrow repair, but the branch should either wire `maxEntries` through the public Gateway config contract or keep the cap internal.
