## Summary

Keep this PR open: it addresses a valid gateway auth rate-limiter memory-bound gap and includes after-fix runtime proof, but the advertised `maxEntries` knob is not wired into the public gateway config contract.

## What This Changes

The PR adds a `maxEntries` cap with FIFO eviction of non-locked entries to the gateway auth rate limiter, adds two regression tests, and adds a changelog entry.

## Best Possible Solution

Land a repaired version that keeps the bounded limiter behavior and aligns `maxEntries` across gateway config types, strict schema, generated metadata, docs, and targeted coverage.

## Reproduction Assessment

Yes. Current main is source-reproducible: `recordFailure` inserts one `Map` entry per new non-exempt `{scope, ip}` and only periodic/window pruning removes entries, so a unique-IP flood grows until pruning runs.

## Solution Assessment

No, not quite. The in-limiter cap is the right narrow repair, but the new `maxEntries` config field must be accepted and documented by the gateway config contract before the PR is complete.
