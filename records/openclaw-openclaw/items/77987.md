## Summary

Keep open: current main and v2026.6.6 still have the unbounded auth-rate-limit entries map, and this PR is a plausible narrow mitigation with sufficient runtime proof. The remaining pre-merge question is whether the new `maxEntries` option should stay an SDK/internal limiter option or be wired into public gateway config and docs.

## What This Changes

The PR adds a default 10,000-entry cap to the gateway auth rate limiter, evicts oldest non-locked entries on new inserts, and adds unique-IP flood regression tests.

## Best Possible Solution

Land the insert-time cap after the API/config boundary is explicit: either keep `maxEntries` as an SDK/internal-only default or wire it through gateway config schema/docs with matching validation.

## Reproduction Assessment

Yes. Current main shows `recordFailure` inserts one entry per unique non-exempt failed auth key and only prune later removes entries, so a unique-IP flood before prune is source-reproducible.

## Solution Assessment

Mostly yes. The insert-time cap in the gateway-owned limiter is the narrow fix, but maintainers still need to decide whether the new cap remains SDK/internal-only or becomes public gateway config.
