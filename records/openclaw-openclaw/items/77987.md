## Summary

Keep open: the limiter cap is a focused fix with sufficient runtime proof, but the new `maxEntries` config field is not wired into the strict gateway config contract, so the advertised knob would be rejected by OpenClaw config validation.

## What This Changes

The PR adds a default 10,000-entry cap to the gateway auth rate limiter, evicts oldest non-locked entries on insert, adds two regression tests, and updates the changelog.

## Best Possible Solution

Land the bounded limiter behavior after aligning `maxEntries` across gateway config types, strict schema, generated metadata, docs, and focused regression coverage.

## Reproduction Assessment

Yes. Current main is source-reproducible: `recordFailure` inserts one Map entry per new non-exempt `{scope, ip}` and only window/prune cleanup removes entries, so a unique-IP flood grows until pruning runs.

## Solution Assessment

No, not quite. The in-limiter cap with lockout-preserving eviction is the right narrow repair, but the PR should either keep the cap non-configurable or wire `maxEntries` through the public gateway config contract it now implies.
