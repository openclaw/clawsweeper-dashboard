## Summary

Keep open: current main and v2026.6.8 still lack the insertion-time auth limiter cap, and this PR is the viable candidate fix with sufficient live-output proof. The remaining blocker is maintainer acceptance of the new default cap and whether `maxEntries` stays SDK/internal-only or becomes public `gateway.auth.rateLimit` configuration.

## What This Changes

The branch adds optional `RateLimitConfig.maxEntries` with a 10,000 default, evicts oldest non-locked limiter entries on new inserts, and adds unique-IP flood plus lockout-preservation regression tests.

## Best Possible Solution

Land the insert-time cap after maintainers explicitly choose the API boundary: keep `maxEntries` SDK/internal-only with the default tradeoff, or wire it through public gateway config schema, docs, and validation.

## Reproduction Assessment

Yes. Source inspection shows current main inserts one entry per unique non-loopback scope/IP in `recordFailure` and has no insert-time cap before timer/manual prune runs; the PR body also supplies live-output proof for the fixed path.

## Solution Assessment

Mostly yes. A gateway-owned insert-time cap that skips active lockouts is the narrow fix for the reported map growth, but the PR should merge only after the SDK-only versus public gateway config boundary is explicit.
