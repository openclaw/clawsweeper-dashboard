## Summary

Keep open: the reported memory-bound issue is valid and the limiter-side fix shape is narrow, but this head still leaves `maxEntries` inconsistent with the gateway config contract if it is meant to be configurable before merge.

## What This Changes

The PR adds a default 10,000-entry cap to the gateway auth rate limiter, evicts oldest non-locked entries on insert, adds regression tests, and updates the changelog.

## Best Possible Solution

Land the bounded limiter behavior after aligning `maxEntries` with the public config type, strict schema, help/docs, and schema coverage, or make the cap an internal constant instead of a configurable field.

## Reproduction Assessment

Yes. Current main is source-reproducible: `recordFailure` inserts one Map entry per new non-exempt `{scope, ip}`, and no hard size cap runs before the existing prune path.

## Solution Assessment

No, not quite. The in-limiter FIFO cap with locked-entry preservation is the right narrow repair, but the PR should either wire `maxEntries` through the gateway config contract or stop presenting it as configurable.
