## Summary

Keep open: current main and v2026.6.8 still lack the auth-rate-limit entry cap, and this PR is the open candidate fix for the linked gateway auth hardening issue. The patch looks functionally sound, but merging should wait for maintainer confirmation on the new default cap and SDK/public config boundary.

## What This Changes

The PR adds optional `RateLimitConfig.maxEntries` with a 10,000 default, enforces oldest non-locked entry eviction on new auth-rate-limit inserts, and adds flood plus lockout-preservation tests.

## Best Possible Solution

Land the insert-time cap after maintainers explicitly choose the API boundary and cap semantics: SDK-only default, or public `gateway.auth.rateLimit.maxEntries` wiring with schema, docs, and upgrade proof.

## Reproduction Assessment

Yes. Source inspection shows current main inserts one entry per unique non-loopback scope/IP in `recordFailure` and has no insert-time cap before timer/manual prune runs.

## Solution Assessment

Mostly yes. A gateway-owned insert-time cap that skips active lockouts is the narrow fix for the reported one-failure unique-IP flood, but the default/API boundary and non-absolute cap semantics need maintainer acceptance.

## Risks / Open Questions

- The new 10,000-entry default changes exported limiter behavior under high-cardinality failed-auth traffic by evicting oldest non-locked entries instead of retaining every failed-attempt key until prune.

- Because active lockouts are intentionally preserved, the cap is not an absolute memory ceiling when locked entries alone exceed `maxEntries`; maintainers should accept or clarify that security tradeoff before merge.
