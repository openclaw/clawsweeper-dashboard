## Summary

Keep open: the PR addresses a real gateway auth-rate-limit hardening gap and the supplied live proof is sufficient, but the diff still adds `RateLimitConfig.maxEntries` without aligning the public `gateway.auth.rateLimit` contract, so it should not merge as-is.

## What This Changes

The PR adds a 10,000-entry FIFO cap to the gateway auth rate limiter, preserves locked-out entries during eviction, adds regression tests, and updates the changelog.

## Best Possible Solution

Land the bounded limiter after either making the cap an internal constant or fully adding `gateway.auth.rateLimit.maxEntries` to the type, strict schema, docs, and config coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows `recordFailure` inserts one Map entry per new non-exempt scope/IP key, while only prune removes stale entries, so a unique-IP failure loop before prune grows with unique key count.

## Solution Assessment

No, not yet. The insertion-time cap with locked-entry preservation is the right narrow repair, but the PR should either wire `maxEntries` through the public config contract or keep the cap internal.
