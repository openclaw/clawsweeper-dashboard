## Best Possible Solution

## Solution Assessment

No, not as-is. The insert-time cap is the right layer for the reported flood, but the new cap value needs normalization and maintainers should choose the SDK-only versus public-config boundary.

## Risks / Open Questions

- `maxEntries` is now an exported limiter option but raw `0`, negative, non-finite, or infinite values can weaken or disable failure retention unless normalized.
- The default cap intentionally changes high-cardinality failed-auth behavior by evicting older non-locked entries before prune would have removed them, so maintainers should accept that compatibility tradeoff explicitly.

- The PR does not wire `maxEntries` through public `gateway.auth.rateLimit` config/schema/docs, so maintainers need to decide whether SDK-only tuning is intentional.
