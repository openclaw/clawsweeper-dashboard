## Summary

Keep this PR open for maintainer follow-up. Current main still does not expose agent reasoning context to `message_sending`, so the requested API gap remains real, but the PR is not merge-ready against current main because the hook type surface moved and the implementation only populates the new field from the CLI agent delivery path.

## What This Changes

The PR adds an optional `agentContext` field to `message_sending` hook events, threads selected embedded-agent metadata through outbound delivery, adds helper tests, and documents the hook contract in the changelog/JSDoc.

## Best Possible Solution

Add a backward-compatible typed reasoning context on the current `src/plugins/hook-message.types.ts` message hook contract, build it from current `EmbeddedPiRunMeta` fields, and thread it through shared delivery paths used by both CLI-delivered agent turns and normal channel replies. Cover the contract with focused hook/delivery tests and document retry, non-agent delivery, model identity, and stop-reason semantics in the public plugin hooks docs.

## Reproduction Assessment

Yes. A static reproduction is enough: inspect current main’s `PluginHookMessageSendingEvent` and `applyMessageSendingHook()` and confirm a `message_sending` hook cannot receive tool-call, token-usage, or agent identity context; the normal `routeReply()` path also reaches delivery without such a parameter.

## Solution Assessment

No, not as-is. The optional field is a plausible narrow direction, but the PR should be updated to the current `hook-message.types.ts` surface and populated from the shared agent reply delivery path, not only from `deliverAgentCommandResult()`.
