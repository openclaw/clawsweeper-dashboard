## Summary

Keep open: the requested hook provenance is still useful and not implemented on current main, but this PR cannot merge as-is because it targets stale hook and delivery paths, lacks contributor real-behavior proof, and still leaves partial policy-context coverage.

## What This Changes

The PR adds an optional `agentContext` payload for `message_sending` plugin hooks and threads tool names, token usage, context fill, session identity, channel, and response length from embedded agent runs into outbound delivery.

## Best Possible Solution

Define one maintainer-approved typed outbound provenance context on the active `message_sending` event and durable send paths, coordinated with stopReason and model-provenance work, with docs, focused tests, and redacted real-send proof.

## Reproduction Assessment

Yes, source-level: current main and v2026.5.12 show `message_sending` receives delivery metadata only, while agent run metadata exists upstream. I did not run a live channel send in this read-only review.

## Solution Assessment

No. The branch is a useful prototype, but the maintainable fix needs an approved public API shape on `src/plugins/hook-message.types.ts` and the active durable delivery path, not the stale type and command shim edited here.
