## Summary

Keep this PR open: the feature is still not implemented on current main, but the patch is stale and partial for the advertised public hook contract. https://github.com/openclaw/openclaw/pull/39761

## What This Changes

The PR adds optional `agentContext` metadata to `message_sending` hook events, threads embedded-agent tool/token/session data into outbound delivery, and adds tests plus a changelog entry.

## Best Possible Solution

Rebase onto current main and define the public hook contract on the current hook-message and outbound delivery surfaces, with tests that either populate `agentContext` for all agent-run deliveries or explicitly lock a narrower best-effort scope.

## Reproduction Assessment

Yes, source-level. The PR builds `agentContext` only in the old CLI delivery path, while current main's common reply delivery path still calls `deliverOutboundPayloads` without any agent-run metadata and the current hook event type has no such field.

## Solution Assessment

No. The direction is useful, but the patch is stale and partial; the safer solution is a current-main rebase plus a maintainer decision on delivery-surface coverage before merging a public plugin hook contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Update the current hook event type surface:** `src/plugins/types.ts:611-616`
  - body: On current main, `PluginHookMessageSendingEvent` lives in `src/plugins/hook-message.types.ts` and is re-exported through `hook-types`. Adding `agentContext` only in `src/plugins/types.ts` leaves the actual hook runner/event contract unchanged after rebase, so the new `deliver.ts` event object will either fail typecheck or expose a different SDK type than the runner uses.
  - confidence: 0.82
- **[P2] Populate agent context for every agent delivery:** `src/commands/agent/delivery.ts:223-226`
  - body: The public JSDoc says `agentContext` is present when an outbound message originates from an agent run, but this patch only builds it in the CLI agent command delivery path. Current main's inbound reply/follow-up and cron agent delivery paths still call `deliverOutboundPayloads` without this parameter, so policy hooks see `undefined` for common agent replies and can silently skip enforcement.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: No dependency or CI supply-chain issue was found, but the new policy-hook metadata is inconsistently populated across agent delivery paths.

Concerns:

- **[medium] Avoid policy guard gaps:** `src/commands/agent/delivery.ts:223`
  - body: Because `agentContext` is intended to support pre-send validation guards, populating it only on one delivery path can allow common agent replies to bypass hooks that intended to block or rewrite messages based on tool/session metadata.
