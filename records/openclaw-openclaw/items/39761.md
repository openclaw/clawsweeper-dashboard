## Summary

Keep this PR open, but it is not merge-ready: the requested plugin API is still missing on current main, and the branch is stale against the current hook contract and normal reply delivery paths.

## What This Changes

The PR adds optional `agentContext` metadata to `message_sending` hook events, threads selected embedded-agent metadata through CLI agent delivery and outbound delivery, and adds tests plus a changelog entry.

## Best Possible Solution

Add a backward-compatible reasoning context on the current `src/plugins/hook-message.types.ts` contract, derive it from current embedded run metadata, and thread it through all `message_sending` paths that deliver agent-originated replies, with docs and tests defining retry and non-agent behavior.

## Reproduction Assessment

Yes. Static source inspection on current main shows the hook event lacks `agentContext`, and normal inbound replies call `runMessageSending()` directly without any delivery-layer context.

## Solution Assessment

No, not as-is. The optional field is a reasonable direction, but the patch targets stale hook type and command-delivery surfaces and leaves common reply paths without the new metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Populate agent context on normal replies:** `src/commands/agent/delivery.ts:232`
  - body: The new field is only built in the CLI agent command delivery path. Normal inbound replies use `buildMessageSendingBeforeDeliver()` and call `runMessageSending()` directly with only `{ content, to }`, so hooks enforcing tool/session policy would still see missing context on a common agent reply path.
  - confidence: 0.9
- **[P2] Move the contract change to hook-message types:** `src/plugins/types.ts:611-616`
  - body: Current main defines and re-exports message hook event types from `src/plugins/hook-message.types.ts` through `hook-types.ts`. Adding `agentContext` only to the older `src/plugins/types.ts` location will not update the current public `PluginHookMessageSendingEvent` contract after rebase.
  - confidence: 0.88
- **[P2] Accumulate tool metadata across attempts:** `src/agents/pi-embedded-runner/run.ts:935`
  - body: `runEmbeddedPiAgent` can retry or fail over before returning, but the patch exposes only the latest attempt's `toolMetas`. If an earlier attempt used tools and a later attempt succeeds without them, policy hooks can misclassify the turn as having no tool use.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: No supply-chain changes were found, but incomplete hook metadata coverage creates a concrete policy-enforcement bypass risk.

Concerns:

- **[medium] Partial agentContext coverage can bypass policy hooks:** `src/commands/agent/delivery.ts:232`
  - body: A plugin may block sends when `agentContext.toolCallCount` is zero or missing, but the PR only supplies the field from one CLI delivery path; normal chat agent replies would still receive missing context and may be allowed by hooks that treat missing context as non-agent delivery.
  - confidence: 0.86
- **[medium] Retry metadata loss can weaken tool-use guards:** `src/agents/pi-embedded-runner/run.ts:935`
  - body: Because the exposed tool metadata comes from the latest attempt rather than the whole run, failover or retry paths can hide earlier tool usage and lead policy hooks to make the wrong allow/block decision.
  - confidence: 0.74
