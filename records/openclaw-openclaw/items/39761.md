## Summary

Keep open for maintainer review: the feature is useful and not implemented on current main, but this branch is conflicting, targets stale command-delivery paths, lacks real behavior proof, and needs a coordinated public hook provenance/security contract before it is mergeable.

## What This Changes

The PR adds optional agent tool, token, session, channel, and response-length context to the `message_sending` plugin hook and threads it from the embedded runner into outbound delivery.

## Best Possible Solution

Define one backward-compatible typed outbound provenance context for `message_sending`, rebase it onto `src/agents/command/delivery.ts` and durable send/retry paths, then document and prove the real hook behavior before merge.

## Reproduction Assessment

Yes, source-level: current main's hook event and `applyMessageSendingHook()` still expose delivery-only metadata, while active agent command delivery sends through durable delivery without an agent provenance carrier. I did not run a live channel send because this is a read-only review.

## Solution Assessment

No, not as-is. The direction is useful, but the branch must be rebased onto the active durable delivery contract and the public provenance/security semantics need maintainer approval before this is the maintainable fix.
