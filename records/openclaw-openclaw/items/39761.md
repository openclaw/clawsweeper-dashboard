## Summary

Keep open for maintainer review: current main does not implement the requested hook metadata, but the PR is stale against the current hook/durable delivery surfaces, has correctness blockers, and lacks required real behavior proof.

## What This Changes

The PR adds a `MessageSendingAgentContext` type, embedded-runner tool/token metadata plumbing, outbound hook threading, unit tests, and a changelog entry for `message_sending` hooks.

## Best Possible Solution

Define the hook contract on current main's active `hook-message` and durable send surfaces, with tests and docs that either cover all agent-originated deliveries or explicitly define a narrower best-effort contract.

## Reproduction Assessment

Yes, source-level. Current main lacks `agentContext` on the active `PluginHookMessageSendingEvent`, and the PR head wires the field through pre-refactor CLI/direct-delivery surfaces while common durable reply paths still send no agent-run metadata.

## Solution Assessment

No. The direction is useful, but the patch should be rebased onto current main and should settle the public hook coverage contract before exposing a policy-relevant field.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Move agentContext onto the active hook event type:** `src/plugins/types.ts:607-620`
  - body: On current main, `PluginHookMessageSendingEvent` lives in `src/plugins/hook-message.types.ts` and is re-exported through `hook-types`. Adding `agentContext` only here leaves the active hook runner/SDK event type unchanged after rebase, so hook authors would not get the promised field on the real public contract.
  - confidence: 0.86
- **[P2] Thread agent context through durable agent sends:** `src/commands/agent/delivery.ts:223-235`
  - body: This builds `agentContext` only in the old CLI delivery path and passes it to direct outbound delivery. Current main's CLI agent delivery, inbound replies, follow-up replies, and turn delivery use `sendDurableMessageBatch`, so common agent-originated messages would still reach `message_sending` hooks with no agent metadata.
  - confidence: 0.88
- **[P2] Accumulate tool metadata across runner attempts:** `src/agents/pi-embedded-runner/run.ts:940`
  - body: This assignment preserves only the latest attempt's tool metadata, and success returns still expose `attempt.toolMetas`. If an earlier retry/failover attempt used tools and the final attempt did not, hooks can see `toolCallCount: 0` for a run that did use tools, which breaks the policy-guard use case described by the PR.
  - confidence: 0.8
