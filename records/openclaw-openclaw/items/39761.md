## Summary

Keep open. The feature is still useful, but the branch is stale and conflicting against current main, misses the active public hook and durable-delivery surfaces, and still lacks contributor real-behavior proof for a security-sensitive hook provenance change.

## What This Changes

The PR adds an optional agent reasoning context to outbound `message_sending` hooks, including tool names, token usage, context fill, agent/session identity, channel, response length, tests, and a changelog entry.

## Best Possible Solution

Define one maintainer-approved typed outbound provenance context on the active `message_sending` event and durable send paths, coordinated with the open stopReason and model-provenance work, with docs, focused tests, and redacted real-send proof.

## Reproduction Assessment

Yes, source-level: current main and `v2026.5.12` expose only delivery-level `message_sending` data while adjacent agent metadata exists upstream. I did not run a live channel send in this read-only review.

## Solution Assessment

No. The branch is a useful prototype, but the maintainable fix needs an approved public API shape on `src/plugins/hook-message.types.ts` and the active durable delivery/recovery paths, not the stale command shim edited here.

  - body: This adds `agentContext` to `src/plugins/types.ts`, but current main imports and re-exports the public `message_sending` event from `src/plugins/hook-message.types.ts`. After a rebase, hook authors and the runner would still be on different contracts.

  - confidence: 0.94
- **[P2] Persist policy context for queued retries:** `src/infra/outbound/deliver.ts:238-242`
  - body: The PR documents that `agentContext` is not persisted in the write-ahead queue, but delivery is queued before the hook runs. If recovery replays the queued payload after a crash, policy hooks can see `agentContext: undefined` and make a different allow/block decision than the original send.
  - confidence: 0.86
- **[P2] Accumulate tool calls across attempts:** `src/agents/pi-embedded-runner/run.ts:1475`
  - body: The success return still publishes only `attempt.toolMetas` from the final attempt, while the latest retry-limit fix only tracks the latest attempt for error exits. If an earlier retry or failover attempt used tools and the final attempt does not, policy hooks can see `toolCallCount: 0` for a turn that did use tools.
  - confidence: 0.82
