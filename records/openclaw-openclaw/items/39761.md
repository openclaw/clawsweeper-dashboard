## Summary

Keep open. Current main still lacks agent reasoning context on `message_sending`, so the API gap is real, but the branch is not merge-ready because it targets an older hook type location and only covers one delivery path for metadata that plugin policy hooks may treat as security-sensitive.

## What This Changes

The PR adds optional `agentContext` metadata to outbound `message_sending` hook events, threads selected embedded-agent run metadata through CLI agent delivery, and adds tests plus a changelog entry.

## Best Possible Solution

Add a backward-compatible typed reasoning context on the current `src/plugins/hook-message.types.ts` contract, derive it from current embedded run metadata, and thread it through the shared reply and outbound delivery paths that actually invoke `message_sending`. The public docs and tests should spell out retry behavior, non-agent sends, model/provider identity, and stop/tool-use semantics.

## Reproduction Assessment

Yes. Static source inspection on current main reproduces the gap: the event type and hook construction omit agent reasoning context, and normal inbound replies call `runMessageSending()` without such metadata.

## Solution Assessment

No, not as-is. The optional field is a reasonable direction, but the implementation should be rebased onto the current hook-message type surface and cover normal reply dispatch, not only CLI agent command delivery.
