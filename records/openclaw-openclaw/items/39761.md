## Summary

Keep open. The feature is still useful and not implemented on current main, but this PR is stale against the active hook/durable delivery surfaces, only threads the new context through one old agent command path, and still lacks contributor real-behavior proof.

## What This Changes

This PR adds optional `agentContext` metadata to `message_sending` hook events, threads embedded-runner tool/token/session data into outbound delivery, and adds tests plus a changelog entry.

## Best Possible Solution

Rebase onto current main and define one backward-compatible agent completion/provenance context on the active hook-message and durable send surfaces, with tests and docs for the chosen coverage.

## Reproduction Assessment

Yes, source-level for the review findings. Current main lacks `agentContext` on the active hook event and call sites, while the PR only threads the proposed field through an old CLI command delivery path.

## Solution Assessment

No. The direction is useful, but this patch is stale and partial; the safer solution is a current-main rebase plus maintainer API-scope approval for which agent-originated delivery paths must populate the public field.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Update the active hook event type:** `src/plugins/types.ts:611-616`
  - body: On current main, `PluginHookMessageSendingEvent` lives in `src/plugins/hook-message.types.ts` and is re-exported through `hook-types`. Adding `agentContext` only to this stale `src/plugins/types.ts` surface leaves the actual hook runner contract unchanged after rebase, so plugin authors will not get the advertised field from the active SDK type.

Current main's durable reply and turn delivery paths also call `sendDurableMessageBatch`, so hooks can silently miss policy context for common agent replies.
  - confidence: 0.88
