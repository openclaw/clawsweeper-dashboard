## Summary

Keep open. The requested hook/provenance capability is still missing on current main, but this PR is not merge-ready because it is conflicting, targets stale hook/delivery paths, lacks real behavior proof, and leaves policy-sensitive retry semantics unresolved.

## What This Changes

The PR adds an optional `agentContext` payload to `message_sending` hooks and threads tool names, token usage, context fill, session identity, channel, and response length from embedded agent runs toward outbound delivery.

## Best Possible Solution

Define one maintainer-approved typed outbound provenance/agent-completion context on the active `message_sending` event and durable delivery paths, including explicit retry semantics, then rebase or replace this PR with focused tests, docs, and redacted real behavior proof.

## Reproduction Assessment

Yes, source-level: current main's active `message_sending` event and delivery call expose only delivery metadata, while agent run metadata carries upstream completion/provenance signals. I did not run a live channel send in this read-only review.

## Solution Assessment

No, not in its current form. The feature direction is useful, but the PR must move to the active hook/durable delivery surfaces and settle a consistent provenance contract, especially for retries and non-command agent deliveries.
