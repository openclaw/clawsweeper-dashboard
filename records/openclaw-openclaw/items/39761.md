## Summary

Keep open. The requested hook provenance is still useful, but the branch targets stale hook and delivery surfaces on current main, still publishes only final-attempt tool metadata on the success path, and has no real behavior proof.

## What This Changes

The PR adds an optional `agentContext` payload to `message_sending` hooks by threading embedded-runner tool metadata, token usage, context fill, session/channel identity, tests, and changelog through outbound delivery.

## Best Possible Solution

Define one maintainer-approved typed outbound provenance context on the active message hook and durable send paths, covering related stopReason/model provenance needs, with docs, focused tests, and redacted real-send proof.

## Reproduction Assessment

Yes, source-level: current main’s `PluginHookMessageSendingEvent` and `applyMessageSendingHook()` carry only delivery-level metadata and no `agentContext`. I did not run a live send in this read-only review.

## Solution Assessment

No. The direction is useful, but this branch targets stale files and incomplete run metadata; maintainers should approve the public API shape and rebase it onto the active hook contract and durable delivery path.
