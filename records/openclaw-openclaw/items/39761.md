## Summary

Keep open. The feature direction is useful, but the PR is not correct on current main: it edits stale hook/delivery surfaces, leaves active durable delivery paths without the new context, overlaps broader hook provenance requests, and lacks required real behavior proof.

## What This Changes

The PR adds an optional `agentContext` payload to `message_sending` hooks by threading embedded-runner tool metadata, token usage, context fill, session/channel identity, tests, and a changelog entry through outbound delivery.

## Best Possible Solution

Define one maintainer-approved typed outbound provenance context on the active `message_sending` event and durable send paths, coordinated with stopReason/model-provenance requests, with docs, focused tests, and redacted real-send proof.

## Reproduction Assessment

Yes, source-level: current main’s `PluginHookMessageSendingEvent` and `applyMessageSendingHook()` carry only delivery metadata, while active command delivery uses `sendDurableMessageBatch()` without an agent context carrier. I did not run a live send in this read-only review.

## Solution Assessment

No. The branch is a useful prototype, but the maintainable solution needs a maintainer-approved public API shape on the active hook contract and durable delivery paths rather than the stale files this PR edits.
