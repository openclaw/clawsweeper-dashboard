## Summary

Keep open. Current main still lacks `agentContext`, and this PR remains a useful but stale and partial public hook API change: it targets old hook/delivery files, leaves common durable reply paths without metadata, and lacks real behavior proof.

## What This Changes

The PR adds an optional `agentContext` object to `message_sending` hook events, threads embedded-runner tool/token/session metadata into outbound delivery, and adds tests plus a changelog entry.

## Best Possible Solution

Define a current-main, backward-compatible hook context on `src/plugins/hook-message.types.ts` and thread it through all intended agent-originated delivery surfaces, or document a deliberately narrower best-effort contract.

## Reproduction Assessment

Yes, source-level. Current main's hook event type and `applyMessageSendingHook()` omit `agentContext`, while this PR only updates stale pre-refactor surfaces and leaves current durable reply paths without the metadata.

## Solution Assessment

No. The direction is useful, but this patch is stale and partial; the maintainable solution is a current-main rebase plus a maintainer decision on which delivery surfaces the public hook field must cover.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Update the active hook event contract:** `src/plugins/types.ts:611-616`
  - body: On current main, `PluginHookMessageSendingEvent` is defined in `src/plugins/hook-message.types.ts` and re-exported through `hook-types`. Adding `agentContext` only to `src/plugins/types.ts` leaves the active hook runner contract unchanged after rebase, so plugin authors would not get the promised typed field on the current SDK surface.
  - confidence: 0.88

Current main also sends common agent replies through durable delivery without this metadata, so hooks can silently skip policy checks for those replies.
  - confidence: 0.86

## Security Review

Status: needs_attention
