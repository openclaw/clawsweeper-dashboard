## Summary

Keep open: current main still lacks the requested agent reasoning context, and this PR is not ready because it targets stale hook contract surfaces and only threads metadata through a narrow delivery path while common agent reply paths would still emit `message_sending` events without the new context.

## What This Changes

The PR adds optional `agentContext` metadata to `message_sending` hook events, propagates selected embedded-agent metadata into outbound delivery, and adds tests plus a changelog entry.

## Best Possible Solution

Define a backward-compatible `message_sending` agent metadata contract in `hook-message.types.ts`, populate it from every agent-originated delivery path that can supply run metadata, and document/test the explicit behavior for retries, non-agent sends, and unavailable metadata.

## Reproduction Assessment

Yes for the PR review findings. Source inspection on current main shows the hook event lacks `agentContext`, outbound hook construction only carries delivery metadata, and inbound reply hooks still call `runMessageSending()` with only `{ content, to }`.

## Solution Assessment

No, not as currently proposed. The optional field is a plausible direction, but the patch needs to target the current public hook contract and provide coherent coverage for normal agent reply paths before it is the narrow maintainable fix.

  - body: Current main imports and re-exports message hook event contracts from `src/plugins/hook-message.types.ts` through `hook-types.ts`.   - body: The new field is built only in the CLI agent command delivery path. Current inbound replies still call `runMessageSending()` directly with only `{ content, to }`, and routed replies enter outbound delivery without run metadata, so hooks enforcing tool/session policy would miss `agentContext` on common agent-originated replies.
  - confidence: 0.9
- **[P2] Accumulate tool metadata across run attempts:** `src/agents/pi-embedded-runner/run.ts:935-937`
  - body: The patch publishes only the latest attempt's `toolMetas`. `runEmbeddedPiAgent` can retry or fail over before returning, so an earlier tool-using attempt followed by a final no-tool attempt can expose `toolCallCount: 0` and cause verification hooks to make the wrong allow/block decision.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: No supply-chain changes were found, but the diff creates concrete policy-hook bypass risks by exposing incomplete agent metadata coverage.

Concerns:

- **[medium] Partial agentContext coverage can bypass policy hooks:** `src/commands/agent/delivery.ts:220`
  - body: A plugin may allow sends when `agentContext` is missing, as shown in the PR's example guard. Because the PR supplies the field only from one command delivery path, normal chat replies can still reach `message_sending` without the new policy context.
  - confidence: 0.86
- **[medium] Latest-attempt tool metadata can mislead verification guards:** `src/agents/pi-embedded-runner/run.ts:935`
  - body: Exposing only the latest attempt's tool metadata can hide tool activity from earlier retry/failover attempts, which weakens hooks that use `toolCallCount` to verify whether a completion was actually checked.
  - confidence: 0.74
