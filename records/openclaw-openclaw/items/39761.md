## Summary

Keep open for maintainer review: current main does not implement `agentContext`, and the PR is useful but stale against current hook/delivery structure and still needs a public API scope decision before merge.

## What This Changes

The PR adds an optional `agentContext` payload to `message_sending` hook events with tool names, token usage, context fill, agent/session identity, channel, and response length.

## Best Possible Solution

Rebase onto current main and define the public hook contract in the current hook-message and outbound delivery surfaces, either populating `agentContext` for all agent-run deliveries or documenting and testing a narrower scope explicitly.

## Reproduction Assessment

Yes, source-level. The PR diff only threads `agentContext` through `deliverAgentCommandResult`, while current main has auto-reply and follow-up agent deliveries that call `deliverOutboundPayloads` without any such parameter.

## Solution Assessment

No. The direction is useful, but the current patch is stale and partial for the advertised hook contract; a rebase plus a maintainer decision on delivery-surface coverage is the safer path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Populate agent context for every agent delivery:** `src/commands/agent/delivery.ts:220-224`
  - body: The new public field is advertised as present when an outbound message originates from an agent run, but this patch only builds it in the CLI agent command delivery path. Normal inbound auto-replies and follow-up runs still deliver through `routeReply`/`deliverOutboundPayloads` without this parameter, so hooks using the example guard will see `agentContext` as undefined and skip enforcement for common agent replies.
