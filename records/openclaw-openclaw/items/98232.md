## Summary

Keep open: current main and v2026.6.11 still have credible paths where a managed update/restart failure can stamp `managed-service-handoff-failed` without reconciling that a LaunchAgent replacement gateway is already healthy. Related merged and open work covers adjacent recovery or busy-port restart pieces, but no current merged item fully owns the false failure banner and sentinel success contract.

## What This Changes

The issue reports a false `managed-service-handoff-failed` during macOS LaunchAgent update when the replacement gateway is already healthy.

## Best Possible Solution

Teach the managed update/restart flow to convert a failed child update or busy expected LaunchAgent listener into success only after proving the replacement gateway is on the expected version, RPC-ready, and admin-capable, while keeping wrong-version or unreachable replacements as real failures.

## Reproduction Assessment

Yes at source level: current main can mark `managed-service-handoff-failed` after a package-update restart failure without rechecking that the replacement gateway is healthy and on the expected version. I did not run the full macOS launchd timing race live in this read-only review.

## Solution Assessment

Unclear for an automated fix: the likely best direction is to reuse the existing post-update health/version/RPC verification before writing a final sentinel, but maintainers should approve the exact success contract before converting a failed child update into an `ok` handoff.
