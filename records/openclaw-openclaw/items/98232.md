## Summary

Keep open: current main still has a credible managed update path where a child update/restart failure records `managed-service-handoff-failed` without reconciling that the expected LaunchAgent replacement gateway may already be healthy. Related work covers adjacent stop/restart races, but no merged or open item fully owns this narrower false failure banner after a healthy replacement.

## What This Changes

The issue reports a macOS LaunchAgent managed update/restart false failure when the replacement gateway is already healthy, running the new version, and admin-capable.

## Best Possible Solution

The managed update/restart flow should treat an expected busy LaunchAgent listener as success only after probing the replacement gateway for the expected version, RPC readiness, and admin capability, then write a successful sentinel instead of `managed-service-handoff-failed`.

## Reproduction Assessment

Yes at source level: current main can mark `managed-service-handoff-failed` after a package-update restart failure without rechecking whether the replacement Gateway is already healthy and on the expected version. I did not reproduce the full race live on macOS launchd.

## Solution Assessment

Unclear for an automated fix: the right direction is to reuse the existing post-update health/version probe, but maintainers should confirm the exact success contract before converting a failed child update into a successful handoff.
