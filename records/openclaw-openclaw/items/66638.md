## Summary

Keep open. Current main still has completion-focused announce plumbing and generic opt-in task state updates, but it does not provide framework-owned immediate requester-channel spawn acknowledgements, native per-run subagent progress lifecycle events, or durable announce persistence.

## What This Changes

Request decoupled subagent lifecycle notifications so spawn, progress, completion, failure, cancellation, and retryable delivery are emitted per run without waiting on the parent session lane.

## Best Possible Solution

Keep this as an umbrella architecture request for a versioned runtime-owned lifecycle delivery contract with immediate requester-channel spawn acknowledgement, per-run progress and terminal events, durable idempotent retry storage, and safe direct delivery limited to framework status events.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the implementation gap: current source and docs show accepted tool results, internal spawn/SSE hooks, completion-specific announce delivery, generic opt-in task notifications, and in-memory announce queueing rather than the requested lifecycle channel.

## Solution Assessment

Is this the best way to solve the issue? Unclear at the detailed design level, but the requested direction is the right class of fix; completion-only tweaks, prompt guidance, or per-channel sidebands would not cover spawn acknowledgement, progress, and restart-resilient delivery consistently.
