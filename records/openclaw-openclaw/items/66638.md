## Summary

Keep open. Current main has completion-focused announce delivery, in-process lifecycle/SSE hooks, and pending final-delivery retry state, but it still does not provide first-class spawn/progress/terminal notifications with durable requester-channel delivery independent of the parent session lane.

## What This Changes

Requested decoupling of subagent spawn, progress, completion, failure, and cancellation notifications from parent session-lane batching with durable per-run delivery.

## Best Possible Solution

Define a versioned runtime lifecycle delivery contract with per-event receipts, durable idempotent retry, and clear channel-direct policy, then coordinate it with the narrower completion-spool and lane-fairness work.

## Reproduction Assessment

Yes for the current-main implementation gap: source and docs show completion-only announce delivery, internal lifecycle hooks, sessions.changed broadcasts, and pending final-delivery retry, but no durable requester-channel spawn/progress/terminal lifecycle stream. I did not run a live WhatsApp starvation reproduction.

## Solution Assessment

Unclear at detailed design level, but a runtime-owned lifecycle delivery contract is the right class of solution. Prompt guidance, completion-only delivery, or per-channel sidebands would not consistently cover the requested ordering, durability, and lane-decoupling semantics.
