## Summary

Keep open. Current main has more completion and task-delivery plumbing than the earlier review, but it still does not provide runtime-owned immediate requester-channel spawn acknowledgements, native per-run progress lifecycle events, or a durable first-class lifecycle announce queue decoupled from the parent/session lane.

## What This Changes

Request to decouple subagent spawn, progress, completion, failure, and cancellation notifications from parent session-lane batching with per-run lifecycle events and durable retry delivery.

## Best Possible Solution

Keep this as an umbrella architecture issue for a versioned runtime-owned lifecycle delivery contract with immediate requester-channel spawn ACK, per-run progress and terminal events, safe direct delivery, and durable idempotent retry storage.

## Reproduction Assessment

Yes, for the implementation gap: current source shows accepted tool results, internal hooks/SSE session events, terminal task_completion announces, and a process-local announce queue rather than the requested requester-channel lifecycle stream. I did not run a live WhatsApp lane-starvation scenario in this read-only review.

## Solution Assessment

Unclear at the detailed design level, but the current direction is the right class of fix. Current main is partial; prompt guidance, completion-only delivery, or per-channel sidebands would not cover spawn acknowledgement, progress, and restart-resilient lifecycle delivery consistently.
