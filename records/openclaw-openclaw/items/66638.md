## Summary

Keep open. Current main has useful completion delivery plumbing, an internal spawn hook/SSE lifecycle event, an in-memory announce queue, and opt-in task state-change notifications, but it still does not implement the central request: framework-owned requester-channel spawn acknowledgements, native per-run progress lifecycle delivery, or durable announce persistence across gateway restarts/channel outages.

## What This Changes

Request framework-level subagent lifecycle notifications that deliver spawn, progress, completion, failure, and cancellation events independently of the parent session lane, with durable retry semantics.

## Best Possible Solution

Keep this as the umbrella runtime architecture request. The end state should be a versioned, runtime-owned subagent lifecycle delivery contract backed by the subagent registry and task ledger: immediate requester-channel spawn acknowledgement, per-run progress and terminal events, durable idempotent announce retry, and safe direct delivery only for framework status events, coordinated with #22358 and #54488.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the implementation gap: current-main source and docs show the spawn path only returns an accepted tool result plus internal events, completion uses a `task_completion` announce path, and queued announce state is in memory rather than durable.

## Solution Assessment

Is this the best way to solve the issue? Unclear at the detailed design level, but the requested direction is the right class of fix; narrow completion-only tweaks or per-channel sidebands would not cover spawn acknowledgements, progress events, and restart-resilient delivery consistently.
