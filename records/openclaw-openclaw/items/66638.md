## Summary

Keep open. Current main has completion and task-delivery plumbing, but it still lacks the requested runtime-owned immediate requester-channel spawn acknowledgement, native per-run progress lifecycle stream, and durable first-class lifecycle announce queue independent of the parent session lane.

## What This Changes

Request a runtime architecture that delivers subagent spawn, progress, completion, failure, and cancellation notifications promptly and durably outside parent session-lane batching.

## Best Possible Solution

Keep this as an umbrella architecture issue for a versioned runtime-owned lifecycle delivery contract with immediate requester-channel spawn ACK, per-run progress and terminal events, safe direct delivery policy, and durable idempotent retry storage.

## Reproduction Assessment

Yes, for the implementation gap: current source shows accepted spawn results, internal hooks/SSE session events, terminal task_completion announces, and a process-local announce queue rather than the requested requester-channel lifecycle stream. I did not run a live WhatsApp lane-starvation scenario.

## Solution Assessment

Unclear at detailed design level, but a runtime-owned lifecycle event contract is the right class of solution. Completion-only delivery, prompt guidance, or per-channel sidebands would not cover spawn acknowledgement, progress, and restart-resilient delivery consistently.
