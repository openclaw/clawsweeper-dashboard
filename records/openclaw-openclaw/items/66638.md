## Summary

Keep open. Current main has partial completion delivery, internal lifecycle broadcasts, and recent fixes for stale/collect-mode completion delivery, but it still lacks the requested first-class per-run spawn/progress/terminal notification stream with durable requester-channel delivery decoupled from the parent session lane.

## What This Changes

The issue requests runtime-owned per-run subagent lifecycle notifications with immediate spawn/progress/completion delivery, durable retry, and safe channel-direct routing independent of parent session-lane batching.

## Best Possible Solution

Define a versioned runtime-owned lifecycle delivery contract covering spawn, progress, terminal states, durable idempotent retry, per-run ordering, and safe channel-direct policy without using parent session turns as the implicit transport.

## Reproduction Assessment

Yes for the implementation gap: current docs/source expose non-blocking spawn, internal sessions.changed broadcast, and final task_completion handoff, but no requester-channel spawn/progress/terminal stream. I did not establish a live WhatsApp starvation replay.

## Solution Assessment

Unclear at detailed design level, but the requested runtime-owned lifecycle delivery contract is the right class of solution. Prompt guidance, completion-only delivery, or per-channel sidebands would not cover ordering, durability, and lane-decoupling semantics consistently.
