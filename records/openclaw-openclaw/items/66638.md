## Summary

Keep open: current main has completion announce plumbing and retry/queue improvements, but it still does not implement the requested runtime-owned, per-run spawn/progress/terminal notification stream with durable requester-channel delivery independent of the parent session lane.

## What This Changes

Request first-class subagent lifecycle notifications that deliver spawn acknowledgement, progress, completion/failure, and retryable channel delivery outside parent session-lane batching.

## Best Possible Solution

Define a maintainer-approved lifecycle delivery contract with per-event receipts, durable retry/idempotency, per-run ordering, and explicit channel-direct policy, then land it with source tests plus real channel proof.

## Reproduction Assessment

Yes for the implementation gap: current source and docs show completion announce, hooks, sessions.changed, and queue/retry behavior, but not the requested first-class spawn/progress/terminal lifecycle stream. I did not run a live WhatsApp starvation reproduction.

## Solution Assessment

Unclear at detailed design level, but the requested runtime-owned lifecycle delivery contract is the right class of solution. Completion-only fixes, prompt guidance, or per-channel sidebands would leave inconsistent ordering, durability, and lane-decoupling behavior.
