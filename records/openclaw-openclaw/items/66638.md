## Summary

Keep open. Current main still has useful task records, completion-focused subagent announce delivery, direct/queued terminal delivery, and opt-in generic task state updates, but it does not provide framework-owned immediate requester-channel spawn acknowledgements, native per-run subagent progress lifecycle events, or durable persisted announce retry storage.

## What This Changes

Introduce per-run subagent lifecycle notifications with immediate spawn/progress/completion delivery independent of parent session-lane batching.

## Best Possible Solution

Keep this as an architecture item for a versioned runtime-owned lifecycle delivery contract with immediate requester-channel spawn ACK, per-run progress and terminal events, durable idempotent retry storage, and scoped direct delivery for framework status events.

## Reproduction Assessment

Yes for the implementation gap: current source and docs show accepted tool results, internal hooks/SSE lifecycle emission, completion-specific announce delivery, generic opt-in task notifications, and in-memory announce queueing rather than the requested lifecycle channel. A live WhatsApp latency reproduction was not needed to verify that the requested contract is still missing from current main.

## Solution Assessment

Unclear at the detailed API level, but the requested direction is the right class of fix. Completion-only tweaks, prompt guidance, or per-channel sidebands would not consistently cover spawn acknowledgement, progress, terminal status, and restart-resilient delivery.
