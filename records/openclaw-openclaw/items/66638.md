## Summary

Keep open. Current main has useful completion delivery, task notification policy, persisted pending-final-delivery state, spawn hooks, and sessions.changed broadcasts, but it still does not provide the requested first-class requester-channel spawn/progress/terminal lifecycle stream with per-event delivery receipts and durable lane-independent retry.

## What This Changes

The issue requests first-class per-run subagent spawn/progress/completion notifications that bypass parent session-lane batching and retry durably.

## Best Possible Solution

Define a versioned runtime lifecycle delivery contract with immediate spawn acknowledgement, progress and terminal events, durable idempotent receipt storage, and a safe channel-direct policy, then prove it with overlapping subagents and a delayed parent turn.

## Reproduction Assessment

Yes for the current-main implementation gap: source and docs show completion-centric delivery, internal spawn hooks, sessions.changed broadcasts, and pending final-delivery retry, but no first-class requester-channel spawn/progress/terminal lifecycle stream. I did not run a live WhatsApp starvation reproduction.

## Solution Assessment

Unclear at detailed design level, but a runtime-owned lifecycle delivery contract is the right class of solution. Prompt guidance, completion-only delivery, or per-channel sidebands would not cover the requested ordering, durability, and lane-decoupling semantics consistently.
