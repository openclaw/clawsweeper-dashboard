## Summary

Keep open. Current main has partial completion and final-delivery retry plumbing, but it still lacks the requested first-class runtime lifecycle delivery contract, so closing as implemented or duplicate would lose remaining architecture work.

## What This Changes

Request per-run subagent lifecycle notifications with immediate spawn acknowledgement, progress, terminal events, durable retry, and delivery independent of the parent session lane.

## Best Possible Solution

Define a versioned runtime-owned lifecycle delivery contract that covers spawn, progress, terminal states, durable idempotent retry, and safe channel-direct policy without making parent session turns the implicit transport.

## Reproduction Assessment

Yes for the current-main implementation gap: source and docs show accepted spawn, hook/SSE events, final task_completion delivery, and durable final-delivery retry, but no requester-channel spawn/progress lifecycle stream. I did not run a live WhatsApp starvation scenario.

## Solution Assessment

Unclear at detailed design level, but a runtime-owned lifecycle event contract is the right class of solution. Completion-only delivery, prompt guidance, or per-channel sidebands would not consistently cover spawn acknowledgement, progress, and restart-resilient delivery.
