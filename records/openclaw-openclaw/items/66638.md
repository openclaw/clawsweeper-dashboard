## Summary

Keep open. Current main has direct/queued terminal completion plumbing and generic opt-in task state-change notifications, but it still lacks the framework-owned immediate requester-channel spawn acknowledgement, native per-run subagent progress lifecycle events, and durable announce persistence requested here.

## What This Changes

Request decoupled subagent lifecycle notifications so spawn, progress, completion, failure, cancellation, and retryable delivery are emitted per run without waiting on the parent session lane.

## Best Possible Solution

Keep this as an umbrella architecture request for a versioned runtime-owned lifecycle delivery contract: immediate requester-channel spawn acknowledgement, per-run progress and terminal events, durable idempotent retry storage, and safe direct delivery limited to framework status events, coordinated with #22358 and #54488.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the implementation gap: current source and docs show only accepted tool results, internal spawn/SSE hooks, completion-specific announce delivery, and in-memory announce queueing.

## Solution Assessment

Is this the best way to solve the issue? Unclear at the detailed design level, but the requested direction is the right class of fix; completion-only tweaks or per-channel sidebands would not cover spawn acknowledgement, progress, and restart-resilient delivery consistently.
