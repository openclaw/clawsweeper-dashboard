## Summary

Keep open for maintainer review: current main still treats OpenAI-compatible Chat Completions remote `image_url` fetching as opt-in and private-network-blocked, while the reported Open WebUI/WSL2 placeholder URL flow is not documented, covered by image-upload smoke tests, or implemented by a merged fix.

## What This Changes

The issue requests that Open WebUI image uploads remain usable through OpenClaw to lmstudio when the generated media URL uses `placeholder.openclaw.cloud` in a WSL2/custom-gateway-port setup.

## Best Possible Solution

Define a documented Open WebUI image-upload contract: prefer data URI or genuinely reachable image URLs, or add a narrow authenticated media ingest/proxy path that works in WSL2 without broadly allowing private-network fetches.

## Reproduction Assessment

No high-confidence live reproduction is available. Source inspection shows the relevant remote `image_url` path would reject or SSRF-guard the placeholder URL on current main, but the exact WSL2/Open WebUI/lmstudio chain was not run and raw logs are missing.

## Solution Assessment

Unclear. Hosts-file workarounds or broadly allowing private-network fetches are not the right fix; maintainers should choose a narrow media-ingest or documented client payload contract that keeps SSRF hardening intact.
