## Summary

Keep open. Current main matches the reported failure pattern for a URL-only image handoff: the OpenAI-compatible image path does not define `placeholder.openclaw.cloud`, rejects URL image fetches by default, and uses SSRF guardrails that block loopback/private resolutions when URL fetching is enabled.

## What This Changes

Request support for Open WebUI image uploads whose image URL points at `placeholder.openclaw.cloud` while the Gateway runs on a custom WSL2 port.

## Best Possible Solution

Preserve the SSRF guardrails while defining a supported Open WebUI image path, preferably data URI/base64 ingestion or an explicit trusted media bridge/rewrite with docs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source inspection shows a URL-only `image_url` for `placeholder.openclaw.cloud` will not be treated as local uploaded media, and a loopback hosts entry is blocked by the URL-fetch SSRF policy; a live WSL2/Open WebUI/lmstudio run and raw request logs are still missing.

## Solution Assessment

Is this the best way to solve the issue? Unclear: a blind hostname or loopback bypass would weaken the media URL fetch boundary, so the maintainable path is a deliberate Open WebUI media contract or trusted rewrite/bridge rather than an ad hoc exception.
