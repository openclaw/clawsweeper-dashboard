## Summary

Keep open for maintainer review: current main and v2026.5.7 still reject OpenAI-compatible remote image_url fetches by default, and enabled URL fetches keep private-network SSRF blocking. This is a real Open WebUI/WSL2 compatibility gap, but the right fix needs a product/security decision rather than conservative cleanup.

## What This Changes

The issue requests that Open WebUI image uploads work through OpenClaw to lmstudio on WSL2 when the uploaded image is represented as a placeholder.openclaw.cloud media URL and the gateway runs on a custom port.

## Best Possible Solution

Define a documented and tested Open WebUI image-upload contract: require data URIs or reachable image URLs, or add a narrow authenticated gateway media/proxy path that works on WSL2 without broadly allowing private-network URL fetches.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection does show that a remote image_url in the latest user message is rejected by default and, if enabled, still goes through a private-network SSRF guard, which matches the reported failure class.

## Solution Assessment

Unclear. A hosts-file workaround or broad private-network allow would not be the best fix; maintainers should choose a narrow Open WebUI media-ingest contract that keeps URL-fetch hardening intact.
