## Summary

Keep open. Current main and v2026.6.5 still only support sticky `/model` session overrides plus per-job cron model selection; the requested turn-counted auto-revert is a new model/session policy and config surface that needs maintainer product/API direction rather than cleanup closure.

## What This Changes

The issue requests `/model <name> --turns N` plus optional per-agent model limit config that warns and reverts the session to its default model after the turn budget.

## Best Possible Solution

Define one canonical model override lifetime policy, then implement turn budgets and notifications at the session/Gateway boundary with allowlist/auth checks, state persistence, tests, and public docs.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request. Source inspection gives high-confidence absence of the requested surface because `/model` writes a persistent `sessions.patch` model, `sessions.patch` has no turns/scope field, and `chat.send` has no model field.

## Solution Assessment

Unclear until maintainers choose the product/API shape. The maintainable direction is one shared model-lifetime contract that fits related one-shot override and fallback-persistence work, rather than adding competing ad hoc controls.
