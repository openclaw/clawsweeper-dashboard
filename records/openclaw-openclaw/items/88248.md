## Summary

Keep open: the PR targets a real Control UI/Gateway chat error visibility bug, but the patch still drops visible error bubbles for session-matching runs that the UI adopts on the error event itself. A small controller/test follow-up is needed before merge.

## What This Changes

The PR adds assistant-style chat error payloads from Gateway error broadcasts and teaches the Control UI chat controller to render active-run error events as visible assistant messages.

## Best Possible Solution

Keep the gateway error payload shape, but adjust Control UI error-message dedupe so server-provided or synthesized errors render once for active and adopted matching runs while true locally handled/orphan failures stay suppressed.

## Reproduction Assessment

Yes. Current main source shows chat `error` events only set `lastError`, and the PR body gives a Crabbox/WebVNC missing-auth repro; the review finding is also source-reproducible by sending a session-matching error event when `chatRunId` is initially null.

## Solution Assessment

No. The gateway-side payload addition is reasonable, but the UI-side guard is too broad because it discards adopted matching-run error messages; a narrower dedupe plus regression test is the safer fix.
