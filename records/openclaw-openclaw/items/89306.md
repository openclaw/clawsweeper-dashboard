## Label Justifications

- `P1`: The report describes a broken Android gateway authorization/setup workflow that can block users from completing setup on custom-port deployments.
- `impact:auth-provider`: The affected path is Android gateway authorization and device/node pairing before setup can complete.
- `impact:other`: The issue also affects setup availability for deployments where default TLS ports cannot be exposed.

## Summary

Current main and the v2026.6.6 release cover the central report: Android preserves explicit non-443 TLS/WebSocket ports, surfaces pairing/approval guidance, and pairing state now refreshes or reuses unchanged pending requests instead of rotating the request ID for the same approval surface.

## What This Changes

The issue reports Android authorization failing silently and retrying indefinitely when the gateway is reachable only on a non-443 TLS/WebSocket port.

## Best Possible Solution

Keep the released Android endpoint and pairing-request behavior; ask for a fresh report only if a current Android build still shows a specific custom-port failure with redacted diagnostics.

## Reproduction Assessment

No high-confidence current-main failure path remains from source inspection: explicit non-443 TLS ports are preserved and unchanged pairing requests are refreshed/reused. This review did not live-run the exact Android custom-domain deployment, so confidence is source-based rather than live-device based.

## Solution Assessment

Yes: the best current solution is the released combination of explicit endpoint-port preservation, visible pairing guidance, and bounded/reused pairing state in the shared gateway/session paths rather than a new Android-only config option.
