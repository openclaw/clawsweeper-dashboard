## Summary

Keep open: current main still lacks the installer no-op optimization, and this PR is the canonical implementation thread, but the proposed skip path has a reproduced trust-boundary blocker, exits before installer finalization, and lacks real behavior proof.

## What This Changes

The PR adds a `scripts/install.sh` npm-install skip when the detected installed version equals the target version, plus `--force` and `OPENCLAW_FORCE_INSTALL` overrides.

## Best Possible Solution

Implement the optimization by checking the npm-managed package or a verified npm-global binary, keep installer finalization and `--verify` running on no-op installs, and add focused installer tests/docs for the new flag and skip behavior.

## Reproduction Assessment

Yes for the PR-head blocker: a MEMBER review gives a concrete fake-PATH reproduction, and the PR-head source shows the generic resolver drives the skip. This is not a current-main bug reproduction because current main does not implement the skip.

## Solution Assessment

No. The current PR compares the target version with a generic CLI resolver and exits before finalization; the maintainable solution should prove the npm-managed install is current and keep post-install checks running.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Verify the npm-managed package before skipping install:** `scripts/install.sh:1989-1990`
  - body: The skip path compares the target version with `resolve_openclaw_version()`, which can return any `openclaw` found earlier on PATH before checking npm's global package tree. A wrapper or different install with the same semver can make the installer return "already up to date" even when the npm-managed package is absent or stale, so base this skip on the npm global package or a verified npm-global binary.
  - confidence: 0.93
- **[P2] Keep finalization on the skip path:** `scripts/install.sh:2340-2341`
  - body: Returning from `main` when `INSTALL_SKIPPED=1` bypasses the existing finalization path, including loaded gateway refresh and explicit `--verify` execution. The install step can no-op, but the installer should still run the post-install diagnostics and verification the user requested.
  - confidence: 0.87
