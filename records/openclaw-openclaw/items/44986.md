## Summary

Keep open: current main still lacks the requested installer no-op path, but this PR is not merge-ready because the skip trusts a PATH-resolved `openclaw` and can bypass required finalization; the external PR also lacks real behavior proof.

## What This Changes

This PR adds an npm-install skip in `scripts/install.sh` when the detected local version matches the resolved target version, plus `--force` and `OPENCLAW_FORCE_INSTALL` overrides.

## Best Possible Solution

Keep the canonical installer optimization only after it verifies the npm-managed package or npm-global binary before skipping, still runs finalization and requested verification, and includes redacted real installer proof.

## Reproduction Assessment

Yes for the PR-head blocker: the diff shows the skip uses the generic version resolver, and a MEMBER review reproduced a fake-PATH `openclaw` causing a false skip. This is not a current-main bug because the skip is not on current main.

## Solution Assessment

No. The current PR is not the best fix because it compares against a PATH-resolved CLI and can return before finalization; the narrower maintainable path is to verify the npm-managed install itself and keep post-install diagnostics running.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Verify the npm-managed package before skipping install:** `scripts/install.sh:1988-1989`
  - body: The skip path compares the target version with `resolve_openclaw_version()`, which can return any `openclaw` found earlier on PATH before checking npm's global package tree. A wrapper or different install with the same semver can make the installer return "already up to date" even when the npm-managed package is absent or stale, so base this skip on the npm global package or a verified npm-global binary.
  - confidence: 0.93
- **[P2] Run finalization when skipping the npm install:** `scripts/install.sh:2340-2341`
  - body: Returning from `main` when `INSTALL_SKIPPED=1` skips the existing finalization path, including loaded gateway refresh, onboarding follow-through, and explicit `--verify` execution. The install step can no-op, but the installer should still run the post-install diagnostics and verification the user requested.
  - confidence: 0.87
