## Summary

Keep open: the PR is useful source work, but the current skip condition can falsely skip npm installation by trusting a PATH-resolved `openclaw` before verifying the npm-managed package, and current main has not implemented the optimization.

## What This Changes

The PR adds an npm install skip to `scripts/install.sh` when the detected local version matches the target version, plus `--force` and `OPENCLAW_FORCE_INSTALL` overrides.

## Best Possible Solution

Keep the optimization, but base the skip on the npm-managed package or npm-global bin identity for the active npm environment, preserve installer finalization, and add focused tests plus docs/changelog coverage.

## Reproduction Assessment

Yes. The PR-introduced defect has a high-confidence reproduction from maintainer review using a fake PATH `openclaw`, and current source confirms the resolver checks PATH before the npm global package tree.

## Solution Assessment

No. Comparing the target against a generic resolved CLI version is not the narrowest maintainable solution; the skip should verify the npm-owned package or that the resolved binary belongs to the npm global install being updated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Verify the npm-managed package before skipping install:** `scripts/install.sh:1990`
  - body: The skip path compares the target version with `resolve_openclaw_version()`, which can read any `openclaw` earlier on PATH before checking npm's global package tree. A wrapper or different install with the same semver can make the npm installer return "already up to date" even when the npm-managed package is absent or stale, so base the skip on the npm global package/bin being updated.
  - confidence: 0.93

Repo policy expects user-facing installer behavior to be documented in the public installer docs and recorded in `CHANGELOG.md`, so users can discover `--force` and the no-op behavior outside script help.
  - confidence: 0.82
