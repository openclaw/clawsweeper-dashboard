## Summary

Keep open: current main does not implement the requested installer no-op, and this PR is still the active implementation path for the closed related feature request, but the patch has a correctness blocker in the skip condition.

## What This Changes

The PR adds `--force`/`OPENCLAW_FORCE_INSTALL` and skips the npm install in `scripts/install.sh` when the detected local OpenClaw version matches the target npm version.

## Best Possible Solution

Keep the optimization, but make the no-op decision from the npm-managed package or npm-global bin identity for the environment being updated, then add focused tests plus public installer docs and changelog coverage.

## Reproduction Assessment

Yes for the PR-introduced defect: the maintainer review describes a concrete fake-PATH reproduction, and current source confirms the resolver checks PATH before npm root. The requested installer optimization itself is not a current-main bug reproduction.

## Solution Assessment

No. Comparing the target against a generic resolved CLI version is not the narrowest maintainable solution; the skip should verify the npm-managed package or the npm global binary that this installer would update.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Verify the npm-managed package before skipping install:** `scripts/install.sh:1990`
  - body: The skip path compares the target version with `resolve_openclaw_version()`, which can return any `openclaw` earlier on PATH before checking npm's global package tree. A wrapper or different install with the same semver can make the npm installer return up-to-date when the npm-managed package is absent or stale, so base the skip on the npm global package/bin being updated.
  - confidence: 0.93

  - confidence: 0.82
