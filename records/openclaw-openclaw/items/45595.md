## Summary

Keep open. Current main still compares path-shaped exec allowlist entries against the PATH-resolved symlink identity while approval-backed execution and planned shell execution prefer the realpath, and the open closing PR should be reviewed rather than closing this issue early.

## What This Changes

The issue asks OpenClaw to use one consistent executable identity for exec approval matching, safe-bin/skill trust, allow-always persistence, and audit metadata when commands resolve through symlinks.

## Best Possible Solution

Land #73543 or a focused replacement that makes allowlist matching, allow-always persistence, skill/safe-bin trust, and audit metadata agree on the executable path identity, with regression coverage for symlink and realpath allowlist entries.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a command resolution with resolvedPath=/opt/homebrew/bin/rg and resolvedRealPath=/opt/homebrew/Cellar/ripgrep/14.1.1/bin/rg: matchAllowlist() checks resolvedPath, while approval-backed execution pins resolvedRealPath.

## Solution Assessment

Unclear. Consulting resolvedRealPath in the matcher is necessary, but the maintainable fix should also settle the reported persistence, safe-bin/skill trust, and metadata surfaces so trust and execution do not drift.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because current main can approve one executable path identity while executing another.

Concerns:

- **[medium] Approval identity can drift across symlink target changes:** `src/infra/exec-command-resolution.ts:376`
  - body: Path-shaped allowlist matching checks resolution.resolvedPath, but approval-backed execution uses resolution.execution.resolvedRealPath when available. A symlink-path approval can therefore keep matching after the symlink target changes while a different real executable runs.
