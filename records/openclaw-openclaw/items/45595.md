## Summary

Keep open. Current main still compares path-shaped exec allowlist patterns against the PATH-resolved symlink path while approval-backed execution pins the realpath, and the linked closing fix PR is still open and unmerged.

## What This Changes

The issue asks OpenClaw to use one canonical executable identity for exec approval matching, persistence, trust checks, audit metadata, and execution when PATH binaries resolve through symlinks.

## Best Possible Solution

Review and land #73543 or an equivalent focused fix, then explicitly scope any remaining realpath/symlink trust behavior across safe-bin, skill auto-allow, persistence, and audit metadata.

## Reproduction Assessment

Yes. A code-level reproduction is a resolution with resolvedPath=/opt/homebrew/bin/rg and resolvedRealPath=/opt/homebrew/Cellar/ripgrep/14.1.1/bin/rg: current matchAllowlist() checks only the former, while approved execution later runs the latter.

## Solution Assessment

Unclear. Consulting resolvedRealPath in the matcher is the necessary primary fix, but the maintainable end state should also define how durable approvals and adjacent trust surfaces treat symlink paths versus real executable paths.
