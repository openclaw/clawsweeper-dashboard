## Summary

Keep open. Current main still checks path-shaped exec allowlist patterns against the PATH-resolved symlink path while approved execution pins the realpath, and #73543 is open, unmerged, and explicitly linked as a closing fix candidate.

## What This Changes

The issue requests a consistent executable identity for exec approval matching, safe-bin trust, skill auto-allow, durable approval persistence, and audit metadata when PATH resolution encounters symlinked binaries.

## Best Possible Solution

Land #73543 or an equivalent focused fix after review, with regression coverage for realpath allowlist matching through symlinks and target-drift behavior. If the accepted matcher change intentionally preserves symlink-path entries, record that as the product decision and split explicit follow-ups for persistence, safe-bin trust, skill auto-allow, and audit metadata as needed.

## Reproduction Assessment

Yes. A code-level reproduction is a resolution with resolvedPath=/opt/homebrew/bin/rg and resolvedRealPath=/opt/homebrew/Cellar/ripgrep/14.1.1/bin/rg: current matchAllowlist() checks only the former, while approval-backed execution later uses the latter.

## Solution Assessment

Unclear. Consulting the realpath in the matcher is the narrow primary fix, but the maintainable solution should explicitly define whether durable approvals and symlink-pattern allowlist entries are trusted as symlinks or as the real executable that actually runs.
