## Summary

Keep open: current main and v2026.5.12 still have the reported approval/execution path identity mismatch, and https://github.com/openclaw/openclaw/pull/73543 already targets this issue with closing syntax but is not merged.

## What This Changes

The issue asks OpenClaw to use one canonical executable identity for exec approval matching, trust checks, persistence, metadata, and approved execution when PATH entries are symlinks.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/73543 with a focused, maintainer-approved executable identity fix that aligns matching, trust checks, persistence, audit metadata, and execution, with symlink/realpath regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction: an `ExecutableResolution` with `resolvedPath=/opt/homebrew/bin/rg` and `resolvedRealPath=/opt/homebrew/Cellar/ripgrep/14.1.1/bin/rg` will be matched against the symlink path while approved execution pins the realpath.

## Solution Assessment

Unclear as a complete implementation. Canonicalizing executable identity is the right direction, but the final fix should cover the broader approval/trust/persistence surfaces and make an explicit security-policy choice about symlink-path approvals.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because approval can be granted for one executable path identity while execution runs another.

Concerns:

- **[medium] Approval identity can drift from execution identity:** `src/infra/exec-command-resolution.ts:368`
  - body: Path-shaped allowlist matching checks `resolution.resolvedPath`, but approval-backed execution uses `resolution.execution.resolvedRealPath` when available, so a symlink-path approval can keep matching after the symlink target changes while a different real executable runs.
  - confidence: 0.94
