## Summary

Keep open. Current main still matches path-shaped exec allowlist entries against the PATH-resolved symlink identity while approval-backed execution pins the real executable path, and the linked closing PR remains open for maintainer review.

## What This Changes

The issue requests canonicalizing exec approval trust, persistence, and execution around the same executable identity when PATH resolution crosses symlinks.

## Best Possible Solution

Land #73543 or a focused replacement that makes allowlist matching, allow-always persistence, skill/safe-bin trust, and audit metadata use a consistent executable identity with regression coverage for symlink and realpath allowlist entries.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a resolution with resolvedPath=/opt/homebrew/bin/rg and resolvedRealPath=/opt/homebrew/Cellar/ripgrep/14.1.1/bin/rg: matchAllowlist() checks the former while approved execution pins the latter.

## Solution Assessment

Unclear. Consulting resolvedRealPath in the matcher is necessary, but the maintainable fix should also settle whether symlink-pattern approvals are canonicalized so a changed symlink target invalidates old trust.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because current main can approve one executable path identity while executing another.

Concerns:

- **[medium] Approval identity can drift across symlink target changes:** `src/infra/exec-command-resolution.ts:376`
  - body: Path-shaped allowlist matching checks resolution.resolvedPath, but approval-backed execution uses resolution.execution.resolvedRealPath when available. A symlink-path approval can therefore keep matching after the symlink target changes while a different real executable runs.
