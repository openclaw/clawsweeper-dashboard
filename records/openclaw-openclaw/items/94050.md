## Summary

Keep open: the linked bug is real and this PR is a plausible implementation candidate, but the current branch is still too broad because it treats changing completed exec output as no progress and only supplies test/typecheck proof.

## What This Changes

The PR removes exec output text from completed and failed exec result hashes and updates loop-detection tests to expect critical/global escalation when exec output varies.

## Best Possible Solution

Split exec hashing by status: keep completed output as a progress signal, strip volatile output from failed exec outcomes, update focused tests, and add real behavior proof.

## Reproduction Assessment

Yes, by source inspection: current main hashes failed exec output into `resultHash`, while the no-progress streak breaks when hashes differ. I did not run tests because this is a read-only review.

## Solution Assessment

No. Stripping volatile output is a valid direction for failed exec loops, but this branch also strips completed output even though current tests and docs treat changing completed output as progress.
