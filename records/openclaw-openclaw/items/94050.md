## Summary

Keep open for now: the linked failed-exec bug is valid, but this PR is not merge-ready because it broadens the hash change to completed exec output and drops stable failed-exec discriminators.

## What This Changes

This PR changes `hashExecToolOutcome` so completed and failed exec outcomes hash only status, exit code, and timeout, and updates two loop-detection tests to expect critical/global escalation with varying exec output.

## Best Possible Solution

Split completed and failed exec hashing: keep completed output in the hash, and for failed exec exclude volatile output while preserving status, exitCode, timedOut, failureKind, and exitSignal with focused regression tests.

## Reproduction Assessment

Yes, by source inspection: current main hashes failed exec output into `resultHash`, while the critical and global blockers require identical hashes. I did not execute a local reproducer in this read-only review.

## Solution Assessment

No. The output-stripping direction is plausible for failed exec loops, but this PR changes completed-output progress semantics and omits stable failed-exec metadata.
