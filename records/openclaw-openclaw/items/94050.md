## Summary

Keep open: this PR targets a valid linked exec loop-detection bug, but the patch is still broader than the safest fix because it treats changing completed exec output as no progress. It also still has only typecheck/Vitest proof, not direct after-fix runtime proof from a real stuck exec scenario.

## What This Changes

The PR removes output text from completed and failed exec result hashes and updates two loop-detection tests to expect critical/global escalation when exec output varies.

## Best Possible Solution

Split exec outcome hashing by status: keep completed output as a progress signal, strip volatile failed output, preserve focused regression tests, and add direct redacted runtime proof.

## Reproduction Assessment

Yes at source level: current main hashes failed exec output into `resultHash`, while `getNoProgressStreak` breaks when hashes differ. I did not run a live current-main stuck-exec scenario because this review is read-only.

## Solution Assessment

No. Stripping failed exec output is the right narrow direction, but stripping completed output contradicts the current warning-only fixture and docs because changing successful output is progress.
