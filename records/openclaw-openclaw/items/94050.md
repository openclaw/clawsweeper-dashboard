## Summary

Keep open: the linked failed-exec no-progress bug is still present on current main and the latest release, and this PR is the active candidate fix, but the current patch still has an availability-sensitive fingerprint gap and only test-output proof.

## What This Changes

This PR splits exec outcome hashing so completed exec keeps output while failed exec hashes only status, exitCode, and timedOut, and updates the failed-output loop test to expect global circuit-breaker escalation.

## Best Possible Solution

Keep the failed-output volatility fix, but include stable failed-exec fields such as `failureKind` and `exitSignal` in the hash, add distinct-metadata regression coverage, and require direct after-fix runtime proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main hashes failed exec output into resultHash and `getNoProgressStreak` stops when that hash changes. I did not run a live failed-exec loop because this cleanup review is read-only.

## Solution Assessment

No. Stripping volatile failed output is the right narrow direction, but the best implementation should preserve stable failed-exec metadata instead of hashing only status, exitCode, and timedOut.
