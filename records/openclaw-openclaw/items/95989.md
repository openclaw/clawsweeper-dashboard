## Summary

Keep open: the PR targets a real Matrix message-loss area, but the submitted fix is too broad and reintroduces the exact two-member DM misclassification class that current Matrix routing guards were built to prevent.

## What This Changes

The PR changes the Matrix direct-room tracker to force-refresh `m.direct` for strict two-member rooms after cache seed and then classify them as DMs even when no explicit or promotion path matched, with direct tracker test expectations updated accordingly.

## Best Possible Solution

Keep any useful forced-refresh behavior behind the existing Matrix DM evidence model: classify after refresh only when `m.direct` now contains the room, otherwise preserve recent-invite/per-room metadata gates and add live Matrix proof for the intended post-startup DM case.

## Reproduction Assessment

Yes, at source level for the introduced regression: the diff adds an unconditional post-seed `return true` after every promotion gate and changes existing non-DM/veto expectations to true. I did not run a live Matrix homeserver in this read-only review.

## Solution Assessment

No. The forced refresh is a plausible piece, but the unconditional exact-two-member fallback is not the best fix because current main and prior Matrix fixes require explicit or gated promotion evidence after `m.direct` has seeded.
