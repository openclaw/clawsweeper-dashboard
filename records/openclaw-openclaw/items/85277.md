## Summary

Keep open for maintainer review: the patch is a focused bug fix and I did not find a blocking correctness issue, but it intentionally changes OpenAI Responses item-id replay semantics and should be coordinated with the adjacent replay-recovery PRs before merge.

## What This Changes

The PR adds an OpenAI Responses replay-id helper, uses it when converting assistant text items, and adds coverage for signed message ids when paired reasoning is absent.

## Best Possible Solution

Land one coherent OpenAI Responses replay policy: store-disabled requests send no provider-owned ids, store-enabled requests only replay internally paired ids, and retry/session-reset recovery follows that same contract.

## Reproduction Assessment

Yes at source level: current main replays textSignature ids whenever Responses item-id replay is enabled, so a pruned-reasoning transcript can send a signed message id without its paired reasoning item. I did not run a live provider failure in this read-only review.

## Solution Assessment

Yes, with maintainer coordination: the helper is narrow and preserves unsigned fallback ids while requiring a replayed reasoning predecessor for signed ids. The remaining question is merge order and final replay policy across the adjacent Responses recovery PRs, not a concrete defect in this patch.
