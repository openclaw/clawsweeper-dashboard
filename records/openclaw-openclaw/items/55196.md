## Summary

Keep this open. Current main still lacks a `brave.http` diagnostics flag, while the bundled Brave plugin does make outbound Brave API calls and uses the shared search cache without targeted request/status/timing/cache logging. The request is narrow, belongs in the existing Brave plugin, and can use current Plugin SDK diagnostics/logging surfaces without a new core API.

## What This Changes

Add an opt-in `brave.http` diagnostics flag for Brave `web_search` requests, responses, timings, and cache hit/miss visibility.

## Best Possible Solution

Add the flag inside the bundled Brave plugin: thread the full config or a precomputed `brave.http` enablement into the runtime, emit redacted opt-in diagnostics for request URL/query/params, status, duration, and cache hit/miss/write events, and document the flag in the Brave Search and diagnostics docs.

## Reproduction Assessment

Yes. This is a feature gap rather than a runtime crash: source inspection on current main shows Brave search executes network and cache paths, while `rg` confirms no `brave.http` flag or Brave diagnostics logger exists.

## Solution Assessment

Yes. The narrowest maintainable solution is provider-owned diagnostics in `extensions/brave`, reusing the existing Plugin SDK diagnostic flag and subsystem logger APIs; a new core diagnostics API is not needed.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/55196 by adding an opt-in `brave.http` diagnostics flag to the bundled Brave web-search plugin. Keep the change local to `extensions/brave` unless a tiny generic SDK use is already exported. Thread the full OpenClaw config or a precomputed flag from `createBraveWebSearchProvider().createTool(ctx)` into the Brave runtime, and use `isDiagnosticFlagEnabled("brave.http", cfg)` plus a subsystem logger following the Telegram `telegram.http` pattern. Log sanitized request URL/query/params, HTTP status, duration, and cache hit/miss/write information for both `web` and `llm-context` modes. Add focused tests around disabled/enabled diagnostics, cache hit/miss, and both Brave modes where practical. Update `docs/diagnostics/flags.md`, `docs/tools/brave-search.md`, the legacy `docs/brave-search.md` if it remains mirrored, and add a user-facing `CHANGELOG.md` entry with valid attribution. Do not change Brave API request semantics, cache keys, cache TTL behavior, or global logging level behavior.
