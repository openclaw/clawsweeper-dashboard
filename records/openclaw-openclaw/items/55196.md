## Summary

Keep open. Current main still lacks a `brave.http` diagnostics flag in the Brave plugin or docs, while the Brave runtime performs outbound Brave API requests and cache reads/writes without targeted diagnostic logging. The existing plugin SDK context and diagnostics helpers make this a narrow bundled-plugin feature rather than a new core API decision.

## What This Changes

The issue requests an opt-in `brave.http` diagnostics flag for logging Brave `web_search` requests, response status/timing, and cache hit/miss behavior.

## Best Possible Solution

Add Brave-owned diagnostic logging inside the bundled Brave web search provider: pass the full config or a precomputed `brave.http` flag into the runtime, emit token-redacted request/status/duration/cache hit/miss logs for both `web` and `llm-context` modes, and update diagnostics, Brave Search docs, tests, and changelog together.

## Repair Work Prompt

Implement the requested `brave.http` diagnostics flag for the bundled Brave web search provider. Keep the behavior Brave-owned unless an existing public SDK helper is clearly insufficient; do not add provider-specific core policy. Thread the full OpenClaw config or a precomputed enabled flag from `createBraveWebSearchProvider().createTool(ctx)` into the Brave runtime, use `isDiagnosticFlagEnabled("brave.http", config)` plus a plugin-SDK subsystem logger, and emit opt-in, credential-redacted diagnostics for both Brave `web` and `llm-context` modes: outbound endpoint/params/query, response status and duration, cache hit/miss, and cache write. Never log `X-Subscription-Token`, resolved credential values, or raw response bodies. Treat search query logging as sensitive but acceptable only when the user explicitly enables the flag. Add focused tests around disabled/enabled diagnostics, token redaction, both modes where practical, and cache hit/miss logging. Update `docs/diagnostics/flags.md`, `docs/tools/brave-search.md`, the legacy `docs/brave-search.md` if it mirrors the public Brave page, and add a single-line user-facing `CHANGELOG.md` entry with `Thanks @mecampbellsoup`. Do not change unrelated web search providers or global logging semantics.
