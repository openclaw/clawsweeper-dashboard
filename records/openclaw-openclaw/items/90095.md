## Summary

Keep this PR open for normal review and merge consideration. Current `main` and `v2026.6.1` still allow the empty-response fallback on the skipped/failed automatic delivery lane, while the PR applies the existing message-tool-only suppression to that path and adds a focused regression; I found no blocking correctness issue.

## What This Changes

Adds a Telegram fallback guard and regression test so message-tool-only turns with skipped non-silent final text do not send the generic empty-response fallback.

## Best Possible Solution

Land the narrow Telegram predicate/test if required checks pass, preserving real dispatch-error fallback while suppressing only the empty-response fallback for message-tool-only skipped/failed automatic final delivery.

## Reproduction Assessment

Yes, source-level reproduction is clear: current main sends the empty-response fallback when the automatic lane records a non-silent skip/failure and no delivery, even when the dispatch result is `message_tool_only`. The PR regression models that exact skipped final path.

## Solution Assessment

Yes, this is the best narrow fix I found: the Telegram fallback owner already tracks `suppressSilentReplyFallback`, and applying it to the skipped/failed empty-response branch avoids a broader source-delivery refactor while preserving dispatch-error fallback behavior.
