## Summary

Keep open: the PR targets a real current-main session-delivery recovery gap, but it should not merge as-is because the new catch path clears the unknown-send marker for every thrown delivery failure, including production partial-sent failures that already have platform results and can still be replayed.

## What This Changes

The PR adds persisted session-delivery recovery markers and a regression test so restart recovery refuses to blindly replay unacked entries whose delivery may already have started.

## Best Possible Solution

Keep the marker for sent-before-error or partial-sent failures, clear it only for known pre-send retryable failures, and add focused coverage before landing the session-queue reconciliation change.

## Reproduction Assessment

Yes. Current main has a clear source-level reproduction: session recovery delivers before acking and QueuedSessionDelivery has no recovery marker, while the PR test models the crash-before-ack state and second recovery pass.

## Solution Assessment

No, not yet. The marker direction matches the outbound queue, but clearing it for every thrown delivery failure misses the production partial-sent path that already reports platform results.
