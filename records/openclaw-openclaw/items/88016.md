## Summary

Keep open: the PR addresses a real session-delivery replay bug, but one partial-send storage-error path can still clear the recovery marker and make an already-sent reply replayable. The broader fail-safe move to `failed/` is also a maintainer acceptance decision because it changes crash recovery from blind at-least-once replay to duplicate-avoidance.

## What This Changes

Adds durable recovery markers to session-delivery queue entries, refuses marked unacked restart-continuation deliveries during recovery, marks partial sends as unknown, and adds focused recovery/sentinel tests.

## Best Possible Solution

Fix the partial-send marker-write failure path, then land only after maintainers explicitly accept the fail-safe `failed/` semantics or choose a reconciliation-backed design.

## Solution Assessment
