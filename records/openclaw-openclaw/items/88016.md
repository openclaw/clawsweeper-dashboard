## Summary

Keep this PR open: it targets a real current-main session-delivery replay bug and the overall direction is useful, but the patch still has a concrete recovery edge that can reintroduce duplicate replay after a successful delivery if persisting the post-send marker fails.

## What This Changes

The PR adds durable recovery markers to the session-delivery queue, refuses blind replay of marked unacked entries, marks partial sends as unknown, and adds focused recovery tests plus a sentinel test mock update.

## Best Possible Solution

Land this only after the post-delivery marker failure path remains non-replayable, with a regression test for that edge and an explicit maintainer decision on the fail-safe `failed/` semantics until session-side reconciliation exists.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main delivers a session entry before acking it and stores no recovery marker, so an unacked post-send entry can be recovered as fresh. I did not execute the repro because this review is read-only.

## Solution Assessment

No, not yet. The proposed direction is the right ownership boundary, but it must preserve a non-replayable marker after a returned delivery even when the `unknown_after_send` write fails.
