## Summary

Keep open: current main still has a source-backed gap where successful embedded-run `assistantTexts` are captured for delivery and trajectory, but no inspected success path reconciles a missing visible assistant turn back into the active Pi session transcript branch. The newer comments strengthen the report across WebChat, Discord, Telegram, WhatsApp, and multiple providers; related #77033 fixed overflow replay only, not this broader missing-assistant invariant.

## What This Changes

The issue requests that every successful embedded PI run with visible assistant text persist an equivalent visible assistant turn in the active transcript branch before later user turns are appended.

## Best Possible Solution

Add a focused embedded PI runner/session-manager invariant repair after successful attempts with non-empty visible assistant text, appending exactly one equivalent visible assistant message under the session lock when the active branch lacks it.

## Reproduction Assessment

No live high-confidence reproduction was established. The issue/comments provide concrete trajectory/transcript evidence, and current source shows successful `assistantTexts` can be delivered and recorded without a post-success active-branch reconciliation step.

## Solution Assessment

Yes: the narrowest maintainable solution is a Pi embedded-runner/session-manager invariant repair, not channel-specific WebChat or messaging delivery mirroring. The fix should no-op when Pi already persisted an equivalent assistant turn and should not add a new config knob.
