## Summary

Keep open: current main still captures and delivers successful embedded-run `assistantTexts` and records them in trajectory, but I did not find a post-success invariant that verifies an equivalent visible assistant turn exists on the active session branch. The follow-up comments materially strengthen the report across channels/providers, while related #77033 covers overflow replay and #76969/#76968 cover the neighboring orphaned-user repair rather than this missing-successful-assistant case.

## What This Changes

Ensure every successful embedded PI run with visible assistant text persists an equivalent assistant turn in the active transcript branch so later turns do not re-answer older prompts.

## Best Possible Solution

Add a focused embedded PI runner/session-manager invariant that no-ops when the assistant turn already exists and appends exactly one equivalent visible assistant turn after successful attempts that produced visible `assistantTexts`.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review. The issue and comments provide concrete trajectory/transcript evidence, and current source shows successful `assistantTexts` can be delivered and recorded in trajectory without a post-success active-branch reconciliation step.

## Solution Assessment

Yes: the narrow maintainable solution is an embedded PI runner/session-manager invariant, not a gateway/WebChat/channel mirror. The existing overflow and orphaned-user fixes are adjacent but do not cover successful visible assistant text missing from the active branch.

## Repair Work Prompt

Fix this issue in the embedded PI runner/session layer. After a successful attempt with non-empty visible `assistantTexts`, ensure the active persisted transcript branch contains exactly one equivalent visible assistant message after the triggering prompt before any later user turn can be appended. Prefer a small helper near `src/agents/pi-embedded-runner/run/attempt.ts`; use the session manager/lock path, no-op when an equivalent assistant message is already reachable, ignore whitespace-equivalent duplicates, and no-op cleanly when the manager/session path is unavailable. Do not add gateway/WebChat/channel-side mirroring of normal agent final text, do not change plugin hook semantics, and do not make cleanup timeout configurability part of this PR unless a regression proves it is required. Add regression coverage for missing assistant append, already-persisted no-op, whitespace-equivalent duplicate no-op, and absent manager/session no-op. Add a user-facing changelog fix entry.
