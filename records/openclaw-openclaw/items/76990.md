## Summary

Keep open: current main still has a source-backed gap where successful embedded-run `assistantTexts` are captured for output and trajectory, but no inspected PI success path enforces an equivalent visible assistant turn on the active transcript branch. The later comments add confirming evidence across Control UI, Discord, Telegram, WhatsApp, and multiple providers; related #77033 covers overflow replay, while #76969/#76968 cover the neighboring orphaned-user repair.

## What This Changes

A successful embedded PI run with visible assistant text should persist an equivalent visible assistant turn on the active transcript branch before later prompts are assembled.

## Best Possible Solution

Add a focused PI embedded-runner transcript invariant repair after successful attempts with non-empty visible assistant text, using the session manager/lock path and no-oping when an equivalent assistant turn already exists.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review. The issue and comments provide concrete trajectory/transcript evidence, and current source shows captured successful `assistantTexts` without a fallback active-branch reconciliation step.

## Solution Assessment

Yes: the narrowest maintainable fix is an embedded PI runner/session-manager invariant, not a channel-specific delivery mirror or a WebChat-side transcript write. Related hook and cleanup hypotheses should be covered by regression tests but do not replace the missing-successful-assistant persistence check.

In the embedded PI runner, after a successful attempt with non-empty visible `assistantTexts`, verify under the session write/manager path that an equivalent visible assistant turn exists on the active branch after the triggering user prompt; append exactly one normal visible assistant message through `SessionManager.appendMessage(...)` or an equivalent branch-safe helper when it is missing.
