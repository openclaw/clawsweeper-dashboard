## Summary

Keep open: current main still captures successful embedded-run `assistantTexts` and records them in trajectory/output data, but I did not find a success-path invariant that verifies or repairs an equivalent visible assistant turn on the active session branch. The related discussion keeps this as the canonical remaining transcript-invariant issue; https://github.com/openclaw/openclaw/issues/81286 was closed as a duplicate pointing back here.

## What This Changes

The issue requests that every successful embedded PI runner response with visible assistant text persist an equivalent visible assistant turn on the active transcript branch before later turns assemble context.

## Best Possible Solution

Add a focused PI embedded-runner/session-manager invariant that repairs only missing successful visible assistant turns on the active branch, while preserving Pi ownership of normal transcript persistence and WebChat's no-mirror boundary.

## Reproduction Assessment

No live current-main reproduction was established. Source inspection and the issue evidence show the runner captures successful `assistantTexts` and uses active session messages for later prompts without a visible-assistant branch reconciliation step.

## Solution Assessment

Yes: the narrow maintainable direction is a PI embedded-runner/session-manager invariant repair, not a channel-specific delivery mirror or a broad prompt-history rewrite. It should coordinate with the adjacent orphan-user persistence work but remain scoped to missing successful assistant turns.
