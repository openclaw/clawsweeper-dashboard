## Summary

Keep open: current main still has a source-backed gap where successful embedded-run `assistantTexts` are captured for delivery/trajectory, but the inspected PI success path does not enforce an equivalent visible assistant turn on the active transcript branch. The related #77033 and #77046 fixes cover neighboring overflow and Codex app-server mirror paths, not this embedded PI invariant.

## What This Changes

The issue requests that successful embedded PI runner replies with visible assistant text always persist an equivalent visible assistant turn on the active transcript branch.

## Best Possible Solution

Add a focused embedded PI runner/session-manager invariant repair after successful attempts with non-empty visible `assistantTexts`, no-oping when an equivalent visible assistant turn already exists and covering duplicate/whitespace cases.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review. The issue and comments provide concrete trajectory/transcript evidence, and current source shows successful `assistantTexts` can be captured and returned without an inspected active-branch reconciliation step.

## Solution Assessment

Yes: the narrow maintainable solution is an embedded PI runner/session-manager invariant, not a channel-specific delivery mirror or WebChat-side transcript write. The fix should preserve Pi ownership and add regression coverage around already-persisted replies.

Fix the embedded PI runner transcript invariant for this issue: after a successful attempt with non-empty visible `assistantTexts`, ensure the active persisted transcript branch has exactly one equivalent visible assistant turn after the triggering prompt branch. Keep the fix inside the embedded PI runner/session-manager boundary; do not mirror normal WebChat/channel delivery payloads into JSONL, do not add a config option, and avoid duplicating messages when Pi already persisted an equivalent assistant reply, including whitespace-equivalent text. Add focused regression coverage for no-op when already persisted, append/repair when missing, whitespace-equivalent duplicate avoidance, and graceful no-op when the manager/session path is unavailable. Also make sure the existing overflow replay behavior from #77033 and the orphaned-user repair cluster (#76888/#76968/#76969) still pass. Add a user-facing changelog fix entry if behavior changes.
