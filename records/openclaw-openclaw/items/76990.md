## Summary

Keep open: current main still lacks a source-visible invariant that reconciles successful embedded-run `assistantTexts` back into the active persisted transcript branch. The added comments strengthen the report across Control UI, Discord, Telegram, WhatsApp, and multiple providers; merged #77033 covers overflow replay, while #76969/#76968 cover the neighboring orphaned-user repair rather than this missing successful-assistant persistence path.

## What This Changes

Ensure successful embedded PI runs that produce visible assistant text leave an equivalent visible assistant turn on the active persisted transcript branch.

## Best Possible Solution

Add a PI embedded-runner/session-manager invariant that, after a successful run with non-empty visible assistant text, no-ops when an equivalent assistant turn already exists and otherwise persists exactly one visible assistant turn on the active branch before later user turns can parent from it.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review. The issue and comments provide concrete trajectory/transcript evidence, and current source shows successful `assistantTexts` are captured and recorded without a current-main fallback that reconciles a missing active-branch assistant turn.

## Solution Assessment

Yes: a runner/session invariant is the narrowest maintainable direction because the documented model says Pi owns durable normal assistant turns. A WebChat or channel-side mirror would fight the existing boundary and risk duplicate transcript entries.
