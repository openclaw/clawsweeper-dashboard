## Summary

Keep open: current main still has a source-backed gap where successful embedded-run `assistantTexts` are captured for delivery/trajectory, but the inspected PI success path does not enforce an equivalent visible assistant turn on the active transcript branch. The follow-up comments add cross-channel/provider evidence, while related #77033 and #76969 cover neighboring overflow/orphan repair paths rather than this exact invariant.

## What This Changes

The issue requests that successful embedded PI runs with visible `assistantTexts` reliably persist an equivalent visible assistant turn on the active session transcript branch before later user turns.

## Best Possible Solution

Add a focused PI embedded-runner/session-manager invariant repair after successful attempts with non-empty visible assistant text, under the session lock and no-oping when an equivalent assistant turn is already reachable on the active branch.

## Reproduction Assessment

Yes, source-reproducible: current main captures and records `assistantTexts` for successful attempts without a post-success active-branch reconciliation step. I did not execute a live channel session, but the issue and comments provide the transcript/trajectory comparison shape to turn into a focused regression test.

## Solution Assessment

Yes: the narrowest maintainable fix is in the embedded PI runner/session-manager boundary, not in channel delivery mirroring or WebChat transcript writes. The repair should preserve Pi ownership and only append when the visible assistant reply is missing from the active branch.
