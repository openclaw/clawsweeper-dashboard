## Label Justifications

- `P2`: This is a normal-priority plugin SDK/runtime ergonomics bug with a clear source-level explanation and limited blast radius, but it can make operator hook policies appear installed while remaining inert.

## Summary

Keep open: current main shows typed plugin hooks are wired through `api.on(...)`, but the reported probe uses `api.registerHook(...)`, which registers internal hook events and can still be shown as ready by the hooks CLI even when the event name is a typed plugin hook name that no internal dispatcher fires.

## What This Changes

The issue reports that user-installed plugin hook handlers registered with `api.registerHook` show ready in `openclaw hooks list` but do not fire for `gateway_start` or `before_agent_run` at runtime.

## Best Possible Solution

Make the SDK and CLI contract unambiguous: typed plugin hooks should use `api.on`, while `api.registerHook` should either clearly reject/warn on typed hook names or gain an explicit compatibility path with focused tests.

## Reproduction Assessment

Yes, source-reproducible: a plugin using `api.registerHook("gateway_start")` or `api.registerHook("before_agent_run")` is recorded as an internal hook entry, while the runtime dispatch path checks `registry.typedHooks` populated by `api.on(...)`. I did not run the live macOS/Telegram flow in this read-only review.

## Solution Assessment

Unclear: dispatching typed hook names from `api.registerHook` would match the reporter's expectation, but it changes a public SDK compatibility surface. The safer maintainable path needs maintainer choice between explicit diagnostics/docs or a tested compatibility bridge.
