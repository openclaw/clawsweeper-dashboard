## Label Justifications

- `P2`: The issue describes a real plugin hook dispatch gap with limited scope to plugin authors and operators relying on hook-based guardrails.
- `impact:security`: Ready-looking lifecycle hooks that do not dispatch can silently disable plugin guardrails such as `before_agent_run` policy checks.

## Summary

Close as a duplicate: current main still shows the same `api.registerHook(...)` typed-hook compatibility gap, and the remaining work is already tracked by the open canonical report at https://github.com/openclaw/openclaw/issues/77285.

## What This Changes

The issue reports that user-installed plugin hooks registered with `api.registerHook("gateway_start" | "before_agent_run", ...)` appear ready but are not invoked during gateway startup or agent turns.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/77285 as the canonical item for deciding whether typed hook names passed to `api.registerHook(...)` should alias to `api.on(...)`, fail fast, or receive clearer diagnostics and docs.

## Reproduction Assessment

Yes, source-based: `api.registerHook(...)` records internal hook entries while `gateway_start` and `before_agent_run` dispatch through the typed hook runner, so those handler bodies are not reached through this registration path.

## Solution Assessment

Yes for cleanup: closing this duplicate keeps the extra evidence linked while leaving the API compatibility decision on the existing canonical report.
