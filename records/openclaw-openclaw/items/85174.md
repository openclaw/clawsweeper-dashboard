## Summary

Close as duplicate: the remaining bug is the same `api.registerHook(...)` typed-hook compatibility gap already tracked by the open canonical issue at https://github.com/openclaw/openclaw/issues/77285, and this issue's extra `gateway_start`, `before_agent_run`, and hooks-list evidence should be preserved there.

## What This Changes

The issue reports that user-installed plugins can register typed lifecycle hook names with `api.registerHook(...)` and see them as ready, but their handlers do not run during gateway startup or agent turns.

## Best Possible Solution

Keep the remaining SDK compatibility decision on https://github.com/openclaw/openclaw/issues/77285, then either reject/warn on typed hook names passed to `registerHook(...)` or add a tested compatibility bridge to the typed `api.on(...)` path.

## Reproduction Assessment

Yes. Source inspection shows a plugin using `api.registerHook("gateway_start")` or `api.registerHook("before_agent_run")` is stored as an internal hook entry, while the runtime dispatch paths check `registry.typedHooks` populated by `api.on(...)`; I did not run the live macOS/Telegram flow.

## Solution Assessment
