## Summary

Keep open: current main still has a source-reproducible mismatch where `api.registerHook(...)` records plugin-managed internal hook entries that can appear ready in `openclaw hooks list`, while runtime lifecycle dispatch for `gateway_start` and `before_agent_run` only reads typed hooks registered through `api.on(...)`.

## What This Changes

The issue reports that user-installed plugin hooks registered with `api.registerHook("gateway_start" | "before_agent_run", ...)` show as ready but do not fire during gateway startup or agent turns.

## Best Possible Solution

Make the plugin SDK and hooks CLI contract unambiguous so typed lifecycle hooks use `api.on(...)`, while `api.registerHook(...)` either rejects or warns on typed lifecycle names or deliberately bridges them with focused compatibility tests.

## Reproduction Assessment

Yes, source-reproducible: a plugin using `api.registerHook("gateway_start")` or `api.registerHook("before_agent_run")` is recorded under `registry.hooks`, while the runtime dispatcher checks `registry.typedHooks` populated by `api.on(...)`. I did not run the live macOS/Telegram reproduction.

## Solution Assessment

Unclear: dispatching typed hook names from `api.registerHook` would match the reporter's expectation, but it changes a public SDK compatibility surface. Maintainers should first choose between explicit diagnostics/docs and a tested compatibility bridge.
