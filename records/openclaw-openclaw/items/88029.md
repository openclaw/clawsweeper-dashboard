## Summary

Keep open for maintainer review: the patch appears to be a focused and correct fix for credential-store durability, with strong real behavior proof, but it changes the plaintext credential persistence path used by every provider and should get secops/owner acceptance before merge.

## What This Changes

The branch changes both `FileAuthStorageBackend` sync and async `auth.json` persistence paths from direct writes to durable `replaceFileAtomicSync` and adds regression tests for interrupted writes and parent directory mode.

## Best Possible Solution

Land the focused atomic-write fix after secops/agent owner review confirms `replaceFileAtomicSync` is the desired credential persistence primitive and the focused auth-storage proof remains green.

## Reproduction Assessment

Yes. Current main source shows direct `writeFileSync` persistence and the `loadError` guard that prevents later overwrite after parse failure; the PR body also supplies a real-process 20/20 before versus 0/20 after proof for the interrupted-write path.

## Solution Assessment

Yes, this is the best narrow fix shape: both implicated write sites live inside `FileAuthStorageBackend`, and the patch reuses the existing fs-safe atomic replacement helper rather than adding a new storage abstraction. The remaining question is owner acceptance of credential-store compatibility semantics, not a different code layer.
