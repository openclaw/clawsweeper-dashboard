## Summary

Keep open: the PR fixes the raw `O_TRUNC` credential write for write-failure cases, but the new atomic replace calls leave fs-safe durability flags disabled while power loss is part of the stated failure model.

## What This Changes

The PR changes `AuthStorage` credential persistence to write `auth.json` through `replaceFileAtomicSync` and adds a regression test for interrupted writes preserving existing credentials.

## Best Possible Solution

Use the existing fs-safe atomic replace helper with durable temp-file and parent-directory flushes on both credential write paths while keeping the `AuthStorage` API and `auth.json` format unchanged.

## Reproduction Assessment

Yes. The PR supplies a real child-process before/after reproduction for interrupted writes, and source inspection confirms current main still uses the raw `auth.json` write paths that can leave `loadError` blocking later persistence.

## Solution Assessment

No, not quite. Temp-file plus rename is the right shape, but the current calls omit fs-safe durability flags, so the best fix is the same helper with `syncTempFile` and `syncParentDir` enabled or an explicitly narrowed failure scope.
