## Label Justifications

- `P1`: The PR fixes a broken terminal TUI run workflow where switching away during an in-flight run can leave the returned session blank/idle and lose the final result from the user's view.

## Summary

No blocking findings. The PR remains necessary because current main still rebuilds switch-back history only from persisted messages while live run events are scoped away from the switched client; the proposed gateway snapshot and TUI adoption path directly address that gap with focused tests and real TUI proof.

## What This Changes

Adds an optional gateway-sourced `inFlightRun` snapshot to `chat.history` and teaches the TUI session switch path to adopt that run and streaming status on switch-back, with gateway and TUI regression tests.

## Best Possible Solution

Land the additive gateway-sourced snapshot and TUI adoption path after normal CI and maintainer review, keeping recovery at the gateway state boundary rather than adding a client-only live-run store.

## Reproduction Assessment

Yes, by source inspection: current main clears the TUI active run on switch and `chat.history` lacks any in-flight snapshot while live events are delivered to scoped session keys. I did not run a failing current-main live repro, but the code path and after-fix recording make the behavior high-confidence.

## Solution Assessment

Yes. The gateway is the right owner because it still has the authoritative abort controller and run buffer after the client switches away; a client-only store would not see events delivered to another session key.
