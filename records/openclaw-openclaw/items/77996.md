Full review comments:

- **[P1] Rebase archived RPCs onto current Gateway registration:** `src/gateway/server-methods-list.ts:119-120`
  - body: Current main no longer uses this old method-list/protocol shape as the authoritative core RPC contract: core methods are classified in `src/gateway/methods/core-descriptors.ts`, lazy-loaded from `src/gateway/server-methods.ts`, and schemas live under `packages/gateway-protocol`. As written, a rebase that only keeps these added names here will leave `sessions.archived.*` unadvertised, unscoped, or undispatched on the current Gateway path.
  - confidence: 0.91
- **[P2] Keep archive discovery out of sessions.list row builds:** `src/gateway/session-utils.ts:1712`
  - body: `sessions.list` is a hot UI/startup path and current main already documents synchronous transcript I/O as the dominant blocker there. Calling `enumerateArchivedTranscriptsInDir()` from every main-row build adds a synchronous `readdir`/`stat` scan to that path, so agents with many reset/deleted archives can stall Gateway polling and WebSocket responsiveness; load archives only when the row is expanded or through a bounded cached projection.
  - confidence: 0.86
- **[P2] Reuse the existing chat renderer for archived messages:** `ui/src/ui/views/sessions.ts:1128-1154`
  - body: The archived viewer reimplements only a narrow subset of chat content blocks, while the live chat renderer already handles `image_url`, `input_image`, transcript `MediaPaths`, managed/local image tickets, attachments, and tool/reasoning rendering. Archived transcripts using those existing shapes will still hide images or show broken placeholders, so the viewer should feed archived messages through the shared chat rendering path instead of maintaining a second renderer.
  - confidence: 0.78
