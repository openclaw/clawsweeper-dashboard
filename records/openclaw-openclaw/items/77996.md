## Summary

Keep open: this is still the active implementation candidate for the linked WebChat history problem, but the PR has blocking correctness and security issues before merge.

## What This Changes

The PR adds Gateway archived-transcript list/read RPCs, attaches archive previews to session rows, renders archived reset/deleted transcripts with image blocks in the Control UI Sessions panel, and adds related tests plus a changelog entry.

## Best Possible Solution

Keep the archived-history browser direction, but route archived reads through the same display-safe, byte-bounded history projection used by chat.history and sync generated Control UI locale outputs before normal maintainer review proceeds.

## Reproduction Assessment

Yes. Source inspection shows reset/delete transcripts are archived on disk while current main has no sessions.archived RPC or archived-transcript Sessions UI surface, and the linked issue includes a concrete reset archive filename from OpenClaw 2026.5.3-1.

## Solution Assessment

No, not yet. The API/UI direction is plausible, but sessions.archived.read should reuse existing chat-history projection and byte-budget safeguards, and the generated Control UI locale outputs must be committed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Bound archived transcript reads before returning messages:** `src/gateway/session-utils.fs.ts:754-765`
  - body: `sessions.archived.read` returns raw flattened transcript messages with only a count limit. Unlike `chat.history`, this skips display projection, oversized-message replacement, and final byte budgeting, so a reset archive with large base64/tool payloads can exceed the WebSocket/UI budget and expose raw transcript internals to read-scoped clients.
  - confidence: 0.89
- **[P2] Regenerate Control UI locale outputs:** `ui/src/i18n/locales/en.ts:217-226`

  - confidence: 0.92
