## Summary

Keep open: current main still has the source-level diagnostic bug, and two open closing PRs already target the same CLI/tool-name confusion, so this should not be closed before one implementation merges.

## What This Changes

The issue asks the CLI to emit an agent-tool diagnostic instead of suggesting `plugins.allow` when an unknown root command is a plugin-owned agent tool name.

## Best Possible Solution

Land a single narrow CLI diagnostic fix that resolves manifest `contracts.tools` ownership before the generic `plugins.allow` fallback, then let the merged PR close this issue.

## Reproduction Assessment

Yes. Source inspection shows a restrictive `plugins.allow` plus an input token absent from built-ins, reserved roots, plugin ids, and `commandAliases` still falls through to the misleading allowlist message, with no `contracts.tools` lookup.

## Solution Assessment

Yes. The narrow maintainable fix is a manifest-backed tool-owner diagnostic in the existing missing-command policy; the issue should remain open until one of the linked PRs implementing that direction merges.
