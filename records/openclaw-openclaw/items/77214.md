## Summary

Keep open: current main still has the misleading `plugins.allow` diagnostic path for unknown root commands, and the linked fix PRs are still open rather than merged.

## What This Changes

The issue asks the CLI to identify agent tool names as model-use tools instead of suggesting that users add tool names to `plugins.allow`.

## Best Possible Solution

Land one narrow manifest-backed fix that checks `contracts.tools` before the misleading allowlist fallback, then let GitHub close this issue from the merged PR.

## Reproduction Assessment

Yes. Source inspection shows a restrictive `plugins.allow` plus an input token that is not a built-in, plugin id, or `commandAliases` entry reaches the `plugins.allow excludes` fallback with no `contracts.tools` lookup.

## Solution Assessment

Yes. The narrow maintainable solution is to use existing manifest/control-plane tool ownership metadata for diagnostics while preserving plugin-id and command-alias behavior.
