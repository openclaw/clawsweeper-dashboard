## Summary

Keep open: current `main` still has a source-level path where an unknown root command under a restrictive `plugins.allow` can receive plugin-allowlist guidance even though agent tools are not CLI subcommands and `contracts.tools` already declares tool ownership.

## What This Changes

The issue asks for the CLI unknown-subcommand diagnostic to recognize enabled plugin-owned agent tool names and avoid suggesting `plugins.allow` edits for tool ids.

## Best Possible Solution

Add a generic manifest-backed agent-tool diagnostic that says the token is an agent/model tool owned by the relevant plugin, while preserving the existing plugin-id, plugin CLI root, and runtime slash-command messages.

## Reproduction Assessment

Yes, at source level: with a restrictive `plugins.allow` and an input token that is not a built-in, reserved root, plugin id, or `commandAliases` entry, the current policy emits the misleading allowlist message. The missing branch is a lookup against manifest-declared `contracts.tools`.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: use existing manifest/control-plane metadata to improve diagnostics without making agent tools executable as CLI subcommands or broadening plugin runtime loading.

Preserve the existing behavior for real plugin ids, plugin CLI roots, disabled plugin entries, and `commandAliases` runtime slash commands.
