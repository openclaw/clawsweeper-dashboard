## Summary

Keep open: current main still has no `contracts.tools` ownership lookup in CLI unknown-command diagnostics, and the linked closing PRs are still open implementation candidates.

## What This Changes

Request a CLI diagnostic that recognizes plugin-owned agent tool names and stops directing users to add tool names to `plugins.allow`.

## Best Possible Solution

Land one narrow manifest-backed tool-owner diagnostic that preserves existing plugin-id and CLI-alias behavior, then let the merged closing PR resolve this issue.

## Reproduction Assessment

Yes, source-level: the missing-command policy still falls through to the `plugins.allow` token suggestion without consulting `contracts.tools`, and the earlier unowned-root guard also checks only plugin CLI metadata. I did not execute the CLI in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to add a manifest-backed tool-owner resolver beside the existing command-alias resolver, not to make agent tools CLI subcommands or loosen config protection.
