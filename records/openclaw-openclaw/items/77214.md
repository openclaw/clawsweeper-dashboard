## Summary

Keep open: current main still has the misleading `plugins.allow` fallback for agent-tool-shaped unknown subcommands, and two open PRs (#77220 and #77221) already reference this issue with closing syntax, so this should close only after one fix lands.

## What This Changes

The issue requests a CLI diagnostic that recognizes manifest-declared agent tool names and tells users to use them from agent/model tool-use instead of suggesting `plugins.allow`.

## Best Possible Solution

Land or dedupe one narrow fix PR that adds manifest-backed `contracts.tools` ownership lookup to the missing-command policy while preserving the existing plugin-id, plugin CLI, and runtime slash-command diagnostics.

## Reproduction Assessment

Yes, at source level: an input such as `lcm_recent` with a non-empty `plugins.allow` list reaches the fallback at `src/cli/run-main-policy.ts:171` because current main only checks command aliases, not `contracts.tools`. I did not execute the CLI because this review was constrained to read-only inspection.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: consult existing manifest/tool ownership metadata before emitting the allowlist suggestion, without making agent tools executable as CLI subcommands.
