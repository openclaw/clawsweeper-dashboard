## Summary

Keep open. Current main still publishes runtime config snapshots verbatim, while Slack Socket Mode reads bot/app tokens from the active runtime config path, so the reported sentinel leak remains source-reproducible.

## What This Changes

The issue reports Slack Socket Mode restart loops when `__OPENCLAW_REDACTED__` reaches live Slack token configuration despite valid real tokens.

## Best Possible Solution

Fix shared runtime snapshot publication so redacted placeholders are restored from the paired source snapshot or rejected before activation, with focused config/secrets and Slack regression coverage.

## Reproduction Assessment

Yes, at source level. A runtime snapshot containing Slack botToken/appToken sentinels with real paired source values is still published unchanged, and Slack startup reads those active config fields.

## Solution Assessment

Yes. The shared runtime snapshot layer is the maintainable fix boundary; a Slack-only token guard would leave the same poisoned runtime config available to other credential consumers, but secops should approve the restore-versus-reject behavior.
