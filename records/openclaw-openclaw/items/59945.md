## Summary

Keep open. Current main and the latest release still publish runtime config snapshots without restoring or rejecting redaction sentinels, while Slack Socket Mode resolves bot/app tokens from that active config path.

## What This Changes

Slack Socket Mode should keep using real bot/app tokens when runtime config snapshots contain redacted placeholders instead of failing startup and restarting.

## Best Possible Solution

Fix shared runtime snapshot publication so redacted placeholders are restored from the paired source snapshot or rejected before activation, with focused config/secrets and Slack regression coverage.

## Reproduction Assessment

Yes, at source level. A runtime snapshot containing Slack botToken/appToken sentinels with real paired source values is still published unchanged, and Slack startup reads those active config fields.

## Solution Assessment

Yes, the shared runtime snapshot layer is the maintainable fix boundary. A Slack-only token guard would leave the same poisoned runtime config available to other credential consumers, but the restore-versus-reject policy needs maintainer/security approval.
