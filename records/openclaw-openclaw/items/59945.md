## Summary

Keep open. Current main still publishes runtime config snapshots verbatim while Slack startup reads bot/app tokens from that active runtime config; the restart loop is capped now, but the redacted credential failure remains actionable and no closing PR is linked.

## What This Changes

Fix Slack Socket Mode startup so redacted config sentinels never become live Slack bot/app token values that trigger channel restarts.

## Best Possible Solution

Fix shared runtime config publication to restore paired source values or reject redaction sentinels before they become active runtime config, with focused Slack/config/secrets regression coverage and secops-aware review.

## Reproduction Assessment

Yes, at source level. Publish a runtime snapshot whose Slack botToken/appToken are __OPENCLAW_REDACTED__ while sourceConfig has real values; getRuntimeConfig returns the active runtime snapshot and Slack startup reads those credential fields.

## Solution Assessment

Yes. The shared runtime snapshot layer is the narrow maintainable fix boundary; a Slack-only guard would leave the same sentinel publication bug available to other credential consumers.
