## Summary

Keep open. Current main still leaves the reported root cause in place: runtime config snapshots can be published without restoring or rejecting redaction sentinels, and Slack Socket Mode startup reads bot/app tokens from that active runtime config path. The restart-loop wording is partly softened by the current restart cap, but the credential/sentinel failure remains actionable.

## What This Changes

Slack Socket Mode should connect with valid bot/app tokens instead of consuming the reserved redaction sentinel from an active runtime snapshot.

## Best Possible Solution

Fix the shared runtime config publication path so a paired source snapshot is used to restore redacted placeholders, or the snapshot is rejected before it becomes live config. Add focused config/secrets tests using Slack botToken and appToken sentinel leakage, and keep Slack-specific code as a credential consumer rather than a workaround site.

## Reproduction Assessment

Yes, at source level: publish a runtime snapshot with Slack botToken/appToken set to the redaction sentinel and a paired source snapshot with real values, then read through getRuntimeConfig and Slack account startup resolution. A live Slack Socket Mode run was not performed in this read-only review.

## Solution Assessment

Yes, the shared runtime snapshot layer is the right fix boundary. A Slack-only guard would leave the same redaction-sentinel publication bug available to other channel, provider, and plugin credential paths.
