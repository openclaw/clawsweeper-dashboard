## Summary

Keep open. Current main still publishes runtime config snapshots without restoring or rejecting redaction sentinels, and Slack Socket Mode still reads bot/app credentials from that active runtime config path.

## What This Changes

The issue requests a fix for Slack Socket Mode startup failing when redacted token sentinels leak into the active runtime config snapshot.

## Best Possible Solution

Fix shared runtime snapshot publication so redacted placeholders are restored from the paired source snapshot or rejected before activation, with focused config/secrets and Slack regression coverage.

## Reproduction Assessment

Yes, at source level. A runtime snapshot containing Slack botToken/appToken sentinels with real paired source values is still published unchanged, and Slack startup reads those active config fields.

## Solution Assessment

Yes, the shared runtime snapshot layer is the maintainable fix boundary. A Slack-only token guard would leave the same poisoned runtime config available to other credential consumers, but restore-versus-reject policy needs maintainer/security approval.
