## Summary

Keep open. Current main still publishes runtime config snapshots verbatim while Slack startup reads bot/app tokens from that active snapshot; the restart loop is now capped, but the redacted-token failure path remains source-reproducible.

## What This Changes

The issue reports Slack Socket Mode failing when redacted token sentinels reach live runtime config despite valid Slack tokens.

## Best Possible Solution

Fix the shared runtime snapshot publication path so redacted placeholders are restored from the paired source snapshot or rejected before they become active, then add focused config/secrets and Slack regression coverage.

## Reproduction Assessment

Yes, at source level. Publishing a runtime snapshot whose Slack botToken/appToken are `__OPENCLAW_REDACTED__` while the paired source snapshot has real values leaves the sentinel in `getRuntimeConfig()`, and Slack startup reads those token fields from that active config.

## Solution Assessment

Yes. The shared runtime snapshot layer is the right fix boundary because a Slack-only guard would leave the same poisoned runtime config available to other credential consumers.
