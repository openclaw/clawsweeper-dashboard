## Summary

Keep open. Current main still publishes runtime config snapshots verbatim while Slack startup reads bot/app tokens from that active snapshot; the restart loop is now capped, but the redacted-token failure remains valid and touches shared secret/redaction behavior.

## What This Changes

The issue reports Slack Socket Mode failing when a redacted runtime config snapshot leaks `__OPENCLAW_REDACTED__` into live bot/app token fields.

## Best Possible Solution

Fix the shared runtime config publication path to restore paired source values or reject redacted placeholders before they become active, with focused config/secrets/Slack regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level repro exists by publishing a runtime snapshot whose Slack botToken/appToken are `__OPENCLAW_REDACTED__` while the paired source snapshot has real values, then reading through `getRuntimeConfig()` and Slack account startup resolution.

## Solution Assessment

Yes. The shared runtime snapshot layer is the right fix boundary; a Slack-only guard would leave the same sentinel publication bug available to other credential consumers.
