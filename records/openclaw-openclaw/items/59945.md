## Summary

Keep open. Current main still publishes runtime config snapshots verbatim while Slack startup reads bot/app tokens from that active runtime path; the restart behavior is now capped, but the redacted credential failure remains actionable.

## What This Changes

Slack Socket Mode should connect with valid configured tokens instead of consuming `__OPENCLAW_REDACTED__` from a runtime snapshot and failing channel startup.

## Best Possible Solution

Fix the shared runtime config publication path to restore paired source values or reject redacted placeholders before they become live config, with focused config/secrets/Slack regression coverage.

## Reproduction Assessment

Yes. At source level, publish a runtime snapshot whose Slack botToken/appToken are `__OPENCLAW_REDACTED__` while the paired source snapshot has real values, then read through `getRuntimeConfig()` and Slack account startup resolution.

## Solution Assessment

Yes. The shared runtime snapshot layer is the right fix boundary; a Slack-only guard would leave the same sentinel publication bug available to other credential consumers.
