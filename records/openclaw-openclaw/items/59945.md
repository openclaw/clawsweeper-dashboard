## Label Justifications

- `P1`: The report describes a configured Slack channel startup regression that can repeatedly restart provider tasks for affected users.
- `impact:crash-loop`: The observed failure mode is repeated Slack provider startup failure followed by channel auto-restart attempts.
- `impact:auth-provider`: The bug is about redaction state poisoning Slack credential resolution for bot/app tokens.

## Summary

Keep open. Current main and the latest release still publish active runtime config snapshots verbatim before Slack reads bot/app token fields, so the poisoned-snapshot failure remains source-reproducible even though the exact current log wording may have shifted.

## What This Changes

The issue asks OpenClaw to stop Slack Socket Mode from consuming the reserved redaction sentinel as live bot/app token data and entering repeated provider restarts.

## Best Possible Solution

Publish only de-redacted or rejected runtime snapshots in the shared config/secrets runtime path, then cover Slack Socket Mode bot/app token sentinels and generic redaction restoration with focused regression tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: setRuntimeConfigSnapshot(...) still publishes a runtime config containing Slack botToken/appToken sentinels unchanged, and Slack startup reads those fields from getRuntimeConfig(). I did not run a live Slack workspace repro.

## Solution Assessment

Is this the best way to solve the issue? Yes for the fix boundary: the shared runtime snapshot layer should restore or reject sentinels before activation. The exact restore-versus-reject policy needs maintainer/security direction because a Slack-only guard would leave the same poisoned config available to other credential consumers.
