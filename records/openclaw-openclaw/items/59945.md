## Summary

Keep open. Current main still permits a runtime config snapshot containing redaction sentinels to become active, while Slack startup reads bot/app tokens from that active config path; the exact live Slack log wording was not verified, but the broken source path remains clear.

## What This Changes

Fix Slack Socket Mode startup so valid Slack bot/app tokens are used instead of redacted runtime snapshot sentinels, avoiding restart loops.

## Best Possible Solution

Fix shared runtime snapshot publication so redacted placeholders are restored from the paired source snapshot or rejected before activation, then add focused config/secrets and Slack regression coverage.

## Reproduction Assessment

Yes, at source level. Publishing a runtime snapshot whose Slack botToken/appToken fields contain __OPENCLAW_REDACTED__ while sourceConfig has real values leaves that snapshot active, and Slack startup reads those active config fields.

## Solution Assessment

Yes. The shared runtime snapshot layer is the maintainable fix boundary because a Slack-only guard would leave the same poisoned runtime config available to other credential consumers.
