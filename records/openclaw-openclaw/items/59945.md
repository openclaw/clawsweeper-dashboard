## Summary

Keep open. Current main and v2026.5.22 still publish active runtime config snapshots verbatim before Slack resolves bot/app tokens, so redacted-sentinel poisoning remains source-reproducible even though current restart/log details may differ from the original report.

## What This Changes

The issue asks OpenClaw to stop Slack Socket Mode from consuming redacted credential sentinels as live bot/app tokens and repeatedly restarting.

## Best Possible Solution

Fix the shared runtime snapshot activation path to restore or reject redaction sentinels before publishing active config, then cover Slack bot/app token poisoning and generic redaction restore/reject behavior with focused regression tests.

## Reproduction Assessment

Yes, at source level: setRuntimeConfigSnapshot still publishes supplied runtime config unchanged, and Slack startup reads botToken/appToken from that active runtime config. I did not run a live Slack workspace repro.

## Solution Assessment

Yes for the boundary: a Slack-only guard would leave the same poisoned active config available to other credential consumers, so the fix belongs in shared runtime snapshot activation. The exact restore-versus-reject policy needs maintainer/security direction.
