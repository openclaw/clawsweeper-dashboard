## Summary

Keep open. Current main and the latest release still run memoryFlush only before the main reply turn, while completed in-turn compactions are counted after execution without re-entering any memory-flush path.

## What This Changes

The issue asks OpenClaw to stop runtime-triggered in-turn auto-compactions from bypassing enabled pre-compaction memoryFlush writes.

## Best Possible Solution

Add shared post-compaction memory-flush handling for completed compaction cycles in both main replies and queued follow-ups, with regression coverage proving one durable flush per new compaction cycle.

## Reproduction Assessment

Yes, at source level. An embedded Pi session that completes runtime auto-compaction during the main or follow-up turn increments autoCompactionCount after execution without re-entering the memoryFlush path.

## Solution Assessment

Yes. The maintainable fix is to tie memoryFlush handling to actual completed compaction cycles in the existing reply/follow-up accounting paths; documenting the gap would preserve behavior that contradicts the current memoryFlush contract.
