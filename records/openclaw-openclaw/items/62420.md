## Summary

Keep open: current main and v2026.5.19 still run memoryFlush only in the pre-turn main reply path, while completed in-turn main and follow-up compactions are counted afterward without re-entering the flush path. The related open fix at https://github.com/openclaw/openclaw/pull/84792 only reorders pre-run maintenance for https://github.com/openclaw/openclaw/issues/84695 and does not cover this post-turn compaction gap.

## What This Changes

Report that enabled compaction.memoryFlush can miss durable memory writes when auto-compaction completes during a reply turn rather than during pre-turn maintenance.

## Best Possible Solution

Add shared post-compaction memoryFlush handling for completed runtime compaction cycles in both main replies and queued follow-ups, preserving the existing pre-turn flush and proving one durable flush per new compaction cycle.

## Reproduction Assessment

Yes, at source level. Current main and v2026.5.19 show completed in-turn compactions being counted after execution without re-entering the memoryFlush path; no live transport run was performed in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to attach memoryFlush handling to completed compaction-cycle accounting in the existing main and follow-up paths; documentation-only would preserve behavior that contradicts the current once-per-cycle memoryFlush contract.
