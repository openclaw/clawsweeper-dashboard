## Summary

Keep open: current main and the latest release still run memoryFlush before the reply turn, while completed in-turn main and follow-up compactions are only counted afterward without re-entering the flush path.

## What This Changes

The issue asks OpenClaw to ensure enabled compaction memoryFlush cannot be bypassed by runtime-triggered in-turn auto-compaction.

## Best Possible Solution

Add shared post-compaction memory-flush handling for completed auto-compaction cycles in both main replies and queued follow-ups, with regression coverage proving one durable flush per new compaction cycle.

## Reproduction Assessment

Yes, at source level. An embedded Pi session that completes runtime auto-compaction during the main or follow-up turn increments autoCompactionCount after execution without re-entering the memoryFlush path.

## Solution Assessment

Yes. The maintainable fix is to tie memoryFlush handling to actual completed compaction cycles in the existing reply and follow-up accounting paths; documenting the gap would preserve behavior that contradicts the current memoryFlush contract.
