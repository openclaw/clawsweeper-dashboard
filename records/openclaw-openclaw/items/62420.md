## Summary

Keep open. Current main still has the reported bypass: `runMemoryFlushIfNeeded()` runs before the reply turn, while in-turn embedded compaction is counted after the run and the post-run accounting path does not re-enter memory flush or update `memoryFlushCompactionCount`.

## What This Changes

Ensure memory flush cannot be bypassed when embedded Pi auto-compaction happens during a turn rather than before the turn starts.

## Best Possible Solution

Tie memory flush state to actual embedded Pi compaction completion/count, covering both main replies and queued followups, with regression tests proving one flush per new compaction cycle.

## Reproduction Assessment

Yes, at source level. An embedded Pi session with `agents.defaults.compaction.memoryFlush.enabled=true` that crosses the runtime auto-compaction threshold during the main turn will record `autoCompactionCount` after execution without re-entering the memory-flush path.

## Solution Assessment

Yes. The narrow maintainable fix is runtime accounting that runs or audits memory flush for actual compaction cycles; a docs-only clarification would preserve the mismatch with existing docs and config help.
