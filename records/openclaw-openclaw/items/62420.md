## Summary

The issue remains valid on current main: memory flush is only invoked before the main agent turn, while embedded Pi runtime compaction can complete during the turn and is only counted after the run returns. The docs still promise a pre-compaction memory flush, and I did not find a second production path that flushes when main-turn auto-compaction is observed.

## What This Changes

The issue requests that enabled compaction memory flushes cover runtime-triggered in-turn auto-compaction, or that the limitation be documented clearly.

## Best Possible Solution

Make the embedded Pi runtime path ensure enabled memory flush cannot fall behind an actual auto-compaction cycle. Prefer a pre-compaction flush at a reliable runtime boundary; if that is not available, add a best-effort post-compaction audit/flush after the observed compaction count is persisted, and document any remaining limitation precisely.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path is to run an embedded Pi session with `agents.defaults.compaction.memoryFlush.enabled=true` and force auto-compaction during the main turn; current main has only a pre-turn flush call, while the later compaction event/count path has no flush retry.

## Solution Assessment

Unclear until implementation. The requested direction is correct, but the best narrow fix is likely either a true pre-compaction hook if the runtime can safely pause before compaction, or a targeted post-compaction audit when `compactionCount` advances beyond `memoryFlushCompactionCount`.
