## Summary

Keep open. Current main and the latest release still have the reported ordering gap: memory flush runs only before the reply turn, while in-turn compaction is counted after the embedded run without re-entering memory flush. The merged adjacent fix at https://github.com/openclaw/openclaw/pull/51421 fixed the every-other-cycle counter bug, but it did not add coverage for compactions that complete during the main or follow-up turn.

## What This Changes

Ensure enabled memoryFlush covers auto-compactions that happen during an active turn, not only pre-turn checks.

## Best Possible Solution

Add shared post-compaction memory-flush handling for actual completed compaction cycles in both main replies and queued follow-ups, with regression tests proving one flush per new compaction cycle.

## Reproduction Assessment

Yes, at source level. An embedded Pi session with agents.defaults.compaction.memoryFlush.enabled=true that crosses the runtime auto-compaction threshold during a main or follow-up turn will count autoCompactionCount after execution without re-entering the memory-flush path.

## Solution Assessment

Yes. The narrow maintainable fix is to tie memory-flush state to actual compaction completion/count in the existing reply and follow-up accounting paths; a docs-only clarification would preserve the mismatch with the documented memoryFlush behavior.
