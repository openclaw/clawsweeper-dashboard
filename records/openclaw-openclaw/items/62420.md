## Summary

Current `main` still matches the reported bypass: memory flush is invoked before the main reply turn, while runtime compaction observed during the turn is only counted afterward and does not re-enter the flush path.

## What This Changes

Ensure enabled compaction memory flush cannot be skipped when auto-compaction happens during an agent turn rather than during the pre-turn check.

## Best Possible Solution

Tie memory flush state to actual compaction completion/count for embedded Pi runs, preferably before runtime compaction and otherwise with a targeted post-compaction audit that runs once per new compaction cycle.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an embedded Pi session with `agents.defaults.compaction.memoryFlush.enabled=true` that crosses the runtime auto-compaction threshold during the main turn; current `main` records that compaction after the turn without a flush retry.

## Solution Assessment

Yes, the requested runtime fix direction is the narrow maintainable path. A docs-only clarification would preserve the mismatch between enabled memory flush and counted auto-compaction cycles.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/62420. Observable bug: with `agents.defaults.compaction.memoryFlush.enabled=true`, embedded Pi auto-compaction can occur during the main turn and later increment `compactionCount` without writing memory files or advancing `memoryFlushCompactionCount`, because `runMemoryFlushIfNeeded()` only runs before `runAgentTurnWithFallback()`. Implement a narrow runtime/accounting fix so enabled memory flush cannot fall behind a real auto-compaction cycle. Prefer a reliable pre-compaction flush boundary if one exists; otherwise add a best-effort post-compaction audit after `autoCompactionCount` is persisted that runs at most once per newly observed compaction cycle and updates memory-flush metadata consistently. Check whether the queued follow-up runner needs the same treatment. Do not change unrelated compaction policy, provider defaults, plugin APIs, or dependency versions. Add focused regression coverage proving that an in-turn embedded compaction causes the memory-flush path or audit to run and that repeated accounting does not double flush. Update user-facing docs/changelog if behavior or documented semantics change; because this is user-facing config behavior, include a `CHANGELOG.md` fix entry unless maintainers mark it internal-only.
