## Summary

Keep open. Current main still only collects image-input metadata for custom/OpenAI-compatible onboarding and still writes fixed context/max-token defaults; the related runtime image-metadata bug is closed, but this setup-time metadata collection request is not implemented.

## What This Changes

Request optional onboarding prompts for custom/OpenAI-compatible model metadata: context window, max output tokens, and supported modalities.

## Best Possible Solution

Extend the shared custom-provider onboarding/config path to collect optional model metadata, preserve current defaults when blank, map modality choices to the existing `input` field, and coordinate with the open metadata-prompt PR instead of duplicating it blindly.

## Reproduction Assessment

Yes. A source-backed reproduction is to follow the current custom-provider onboarding path after model selection: it asks for endpoint ID, alias, and image support, then writes fixed context/max-token defaults without metadata prompts.

## Solution Assessment

Yes, setup-time optional metadata collection is the narrow maintainable direction, but it should be implemented in the shared custom-provider setup path and aligned with the existing open PR rather than only adding a one-off configure hook.
