## Summary

Keep open. Current main still does not collect context-window, max-output-token, or general modality metadata during custom/OpenAI-compatible onboarding; it only collects image-input support and writes fixed defaults, while the linked implementation PR remains open and not merged.

## What This Changes

Request to add optional context-window, max-output-token, and modality prompts to the OpenAI-compatible/custom provider onboarding CLI flow.

## Best Possible Solution

Extend the shared custom-provider onboarding/config path to collect optional model metadata, preserve defaults when left blank, map modality choices to `ModelDefinitionConfig.input`, and coordinate with https://github.com/openclaw/openclaw/pull/43493 rather than starting a duplicate branch.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to follow `custom-api-key` onboarding after model verification: current main asks for endpoint ID, alias, and image support, then writes fixed `contextWindow` and `maxTokens` defaults without the requested metadata prompts.

## Solution Assessment

Yes, the requested setup-time metadata collection is a reasonable direction. The maintainable version should live in the shared custom-provider setup/config path and reuse the existing model config contract, with maintainer judgment on audio/video scope and configure-vs-onboard parity.
