## Summary

Keep open. Current main partially covers the modality side by inferring or asking about image input for custom providers, and related runtime handling for configured image metadata has shipped, but the onboarding flow still does not collect context window or max output token metadata and does not expose a general model-input modality prompt.

## What This Changes

Request interactive OpenAI-compatible custom-provider onboarding prompts for context window, max output tokens, and supported model modalities.

## Best Possible Solution

Add optional model-metadata collection to the custom/OpenAI-compatible onboarding path and shared config builder: blank numeric inputs should preserve current defaults, modality selection should map to the existing model `input` field, and docs/tests should cover both interactive and non-interactive setup without changing unrelated runtime capability behavior.

## Reproduction Assessment

Yes. A high-confidence reproduction path is source-backed: follow `openclaw onboard` into `custom-api-key`/OpenAI-compatible setup and the code only reaches model ID, endpoint ID, alias, and image-input confirmation before writing fixed context/max-token defaults.

## Solution Assessment

Yes, the narrowest maintainable solution is to extend the existing custom-provider onboarding/config helpers rather than ask users to edit config manually or change Gateway runtime capability checks. Related #64213 already handled the runtime image-metadata lookup side and should be left intact.
