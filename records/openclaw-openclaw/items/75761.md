## Summary

Keep this PR open. The fast-fail direction is useful, but the latest patch now treats any formatted tool output with unavailable-shaped JSON as terminal, so a `memory_get` read failure can abort recall even though the intended terminal condition is `memory_search`; it also still lacks the required changelog entry.

## What This Changes

The PR updates Active Memory to parse embedded recall tool output, fast-fail clean zero-hit and unavailable memory search results by aborting the hidden run, and adds focused Vitest coverage.

## Best Possible Solution

Keep the fast-fail design, but make the terminal detector apply only to actual `memory_search` output, add a regression for non-search unavailable output, and add the required single-line changelog entry before normal CI and readiness review.

## Reproduction Assessment

Yes. Current main has no `onToolResult` fast-fail path, and the PR tests reproduce the intended zero-hit/unavailable `Memory Search` paths; the newly identified false-positive can be reproduced with a formatted `Memory Get` payload containing `disabled: true` and `error`.

## Solution Assessment

No as written. The runtime approach is narrow for zero-hit `memory_search`, but the implementation should preserve tool identity before classifying terminal payloads and should complete OpenClaw changelog policy.
