## Summary

Keep open: the PR addresses real Active Memory stall paths with a plugin-local use of existing embedded-runner callbacks, and I found no definite diff-introduced functional or security bug. Maintainer review is still needed because the branch intentionally changes setup-grace timeout semantics and exact-head CI is currently red on visible unrelated xai lint annotations.

## What This Changes

The branch updates Active Memory runtime/tests and CHANGELOG to fast-fail empty or unavailable Memory Search tool output, classify embedded timeout boilerplate as timeout, and enforce `config.timeoutMs` as the recall watchdog.

## Best Possible Solution

Land the plugin-local fix after maintainers choose the hard user-visible timeout policy and required checks are green; keep QMD retrieval quality work in #75860.

## Reproduction Assessment

Yes. On current main, Active Memory does not observe formatted `memory_search` tool output during recall and uses a setup-grace-extended watchdog, so empty/unavailable Memory Search output cannot end recall early.

## Solution Assessment

Yes, with a policy caveat. The implementation stays in the Active Memory plugin and uses an existing embedded-runner callback seam, but maintainers should confirm that `config.timeoutMs` is the intended hard recall budget.
