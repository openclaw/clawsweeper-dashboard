## Summary

Keep this PR open. Current main still lacks an Active Memory path that observes formatted `memory_search` tool output and stops recall early on clean zero-hit or unavailable search results, while the PR adds that handling with focused regression coverage and no blocking review findings found in this read-only pass.

## What This Changes

The PR updates the Active Memory plugin, its tests, and the changelog to fast-fail terminal Memory Search output, avoid treating Memory Get misses as terminal, classify embedded timeout boilerplate as timeout, and enforce the configured recall timeout as the visible deadline.

## Best Possible Solution

Land the narrow owner-plugin fix after normal maintainer review and CI, keeping the behavior inside `extensions/active-memory` and using the existing embedded `onToolResult` callback rather than adding a broader core API.

## Reproduction Assessment

Yes. A focused mock of current main where formatted Memory Search emits `results: []` and `debug.hits: 0` reproduces the stalled path because no tool-output callback can stop recall early; the PR adds that regression and related timeout cases.

## Solution Assessment

Yes. The proposed approach is the narrow maintainable fix for this PR: observe formatted Memory Search output inside the Active Memory plugin, preserve unrelated Memory Get misses, and keep the embedded-runner contract unchanged.
