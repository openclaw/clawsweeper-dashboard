## Summary

Keep this PR open. Current main still has the targeted Active Memory recall behavior, and the PR uses an existing embedded-runner callback seam, but it is not cleanup-closeable because the branch is dirty/unmergeable, has failing aggregate checks, and needs maintainer confirmation on the setup-grace versus hard-timeout lifecycle policy.

## What This Changes

This PR changes Active Memory to parse embedded Memory Search tool output, fast-fail terminal empty or unavailable searches, treat timeout boilerplate as no recall, enforce `timeoutMs` as the recall watchdog, and add focused tests plus a changelog entry.

## Best Possible Solution

Land a rebased, green Active Memory plugin-local fix after maintainers confirm whether `setupGraceTimeoutMs` should remain extra setup budget or whether `timeoutMs` alone is the user-visible hard recall budget.

## Reproduction Assessment

Yes. On current main, a mocked zero-hit `memory_search` result cannot end recall early because Active Memory does not observe formatted tool output, and the watchdog path currently uses `timeoutMs + setupGraceTimeoutMs`.

## Solution Assessment

Unclear as a merge decision. The fast-fail and timeout-boilerplate pieces use the narrow existing callback seam, but the hard-timeout change alters documented setup-grace behavior and should be confirmed before merge.
