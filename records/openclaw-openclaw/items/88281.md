## Summary

Keep open: the PR targets a real P1 gateway restart blocker, but the current patch introduces a process-boundary regression where standalone task maintenance can mark a live gateway-owned ACP turn as lost.

## What This Changes

The PR adds an `AcpSessionManager.hasActiveTurn` probe, routes ACP task maintenance through it, and adds task-registry regression tests for stale and live ACP tasks.

## Best Possible Solution

Make ACP task reclamation depend on an authoritative liveness source for the process doing the mutation: gateway-owned maintenance can use the in-memory turn map, while standalone CLI should query the gateway or retain persisted ACP entries conservatively.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: run a long ACP turn in the gateway, wait past the reconcile grace, then run standalone `openclaw tasks maintenance --apply`. The CLI process has an empty ACP active-turn map, so this patch can treat the live task as unbacked.

## Solution Assessment

No, not as currently implemented. The gateway crash cleanup idea is the right direction, but the fix must be process-aware so standalone maintenance does not use a non-authoritative in-process map.
