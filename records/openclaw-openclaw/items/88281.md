## Summary

Keep open. The PR targets the right source-visible ACP zombie-task bug, but the new process-local liveness probe is installed in the shared default maintenance runtime, so standalone task maintenance can misclassify live gateway ACP turns as lost.

## What This Changes

The PR adds `AcpSessionManager.hasActiveTurn`, switches ACP task backing checks from persisted session-entry existence to the active-turn map, and adds focused task-registry tests.

## Best Possible Solution

Keep the live-turn probe for gateway-owned reconciliation, but make non-gateway or offline task maintenance conservative or gateway-authoritative before it can mark ACP tasks lost.

## Reproduction Assessment

Yes. Source inspection shows the introduced regression path: standalone task maintenance calls the same default runtime, but its local ACP manager cannot see active turns owned by the gateway process.

## Solution Assessment

No as currently wired. The core idea is a good gateway-owned liveness check, but the implementation needs an authoritative-process boundary before merge.

## Repair Work Prompt

Repair this PR so ACP active-turn liveness is authoritative only where the gateway process owns the in-memory ACP session manager. The current patch routes all task maintenance through `getAcpSessionManager().hasActiveTurn`, which makes standalone CLI maintenance misclassify live gateway ACP turns as missing because the CLI process has an empty active-turn map. Preserve the intended zombie cleanup for gateway-owned maintenance, preserve conservative persisted-entry retention or use a real gateway-owned liveness query for offline/CLI maintenance, and do not add user config, plugin API, or SDK surface. Add focused regression coverage for both paths: an old running ACP task with a persisted session entry and no active turn in gateway-authoritative mode is reclaimed, while the same persisted entry is retained when maintenance is not ACP-runtime-authoritative. Include command-level or maintenance-runtime tests that cover `openclaw tasks maintenance --apply`/standalone behavior.
