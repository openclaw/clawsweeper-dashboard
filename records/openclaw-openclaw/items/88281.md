## Summary

Keep open. The latest branch fixes the earlier initialization-window problem by marking ACP liveness before runtime initialization, but it still clears the marker per attempt before awaited failover/reconcile cleanup, leaving a source-visible false-lost window for long ACP turns that fail over after the reconcile grace.

## What This Changes

The PR adds an internal ACP active-turn liveness registry, routes ACP task maintenance through a gateway-authoritative ACP live-turn probe, renames the maintenance authority flag/diagnostic reason, wires gateway startup, and adds focused regression tests.

## Best Possible Solution

Keep the active-turn design, but make ACP liveness cover the full prompt task lifecycle across initialization, retries, failover cleanup, and terminal marking, with focused regression coverage and an accepted/documented diagnostic reason change.

## Reproduction Assessment

Yes. Source inspection shows current main keeps a crashed ACP task backed by a surviving session entry, and the PR body models the running task plus surviving entry plus missing live-turn state that blocks restart/update.

## Solution Assessment

No, not yet. The live-turn gate is the right narrow direction, but the marker lifetime should be whole-turn scoped so backend failover/cleanup cannot create another false-lost window.
