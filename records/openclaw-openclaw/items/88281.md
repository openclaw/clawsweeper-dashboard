## Summary

Keep open for maintainer review. Current main still has the ACP persisted-entry backing behavior and the PR's live-turn approach is well targeted, but the patch exposes the new probe through the plugin SDK and carries compatibility/session-state risk that should be resolved or accepted before merge.

## What This Changes

The PR adds an ACP live-turn liveness probe, gates ACP task reconciliation on gateway-authoritative runtime state, wires gateway startup to that authority flag, and adds task-registry/ACP regression tests.

## Best Possible Solution

Land a narrow gateway-authoritative ACP liveness fix after keeping the live-turn probe core-internal or explicitly approving it as SDK-visible, with maintainers deciding whether targeted component proof is enough for this restart/update path.

## Reproduction Assessment

Yes, source-reproducible. Current main treats a surviving ACP session-store entry as a backing session, and the linked report plus PR tests model the running task plus surviving entry plus no live turn state that wedges restart/update.

## Solution Assessment

Mostly yes: the live-turn gate is narrower than adding TTL or force-restart knobs and follows the existing process-authority pattern. The remaining issue is that the liveness probe should be internal or explicitly accepted as SDK-visible before merge.
