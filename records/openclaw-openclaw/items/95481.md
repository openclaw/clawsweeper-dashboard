## Label Justifications

- `P1`: The PR targets P1 completion-delivery loss and still has a route identity defect that can misroute detached-task completions across task scopes.
- `merge-risk: 🚨 message-delivery`: A default-scoped route lease can be overwritten or read by another task sharing the same raw runId, wrongly targeting completion messages.
- `merge-risk: 🚨 session-state`: The durable lease stores requester/session routing state separately from the scoped task registry but the lifecycle hooks do not pass the task scope.
- `merge-risk: 🚨 security-boundary`: Cross-scope reuse of requester_origin_json can expose or reuse another requester route for completion delivery.

## Summary

Keep open: this is a plausible storage foundation, but the production lifecycle hooks still use the default lease scope, so same-runId tasks in different scopes can still overwrite or leave active requester-route state.

## What This Changes

This PR adds a SQLite-backed task-route lease table/API, executor acquire/settle hooks, cleanup, docs, tests, and a lifecycle repro for detached-task delivery origin recovery.

## Best Possible Solution

Land the storage foundation only after lifecycle hooks use the TaskRecord scope tuple, docs/comments match that invariant, and maintainers accept the durable route-state contract before the companion cron wiring lands.

## Reproduction Assessment

Yes, source-level: current main scopes same-runId task operations by runtime/session facts, while this PR head's production acquire and settle hooks call the scoped lease API without passing any scope. A focused executor test with two same-runId pending tasks in different scopes would reproduce the default-scope overwrite/read behavior.

## Solution Assessment

No; the scoped schema/API is the right direction, but the current integration is not the best fix because the main lifecycle path still uses the default scope. The maintainable repair is to derive lease scope from TaskRecord at acquire time and from the updated TaskRecord rows at settle time.
