## Summary

Keep open for contributor and maintainer follow-up: the feature direction is plausible, but the patch has compile-time defects, changes the loop-detection policy boundary, and lacks required real behavior proof.

## What This Changes

The PR adds a transformContext-based consecutive tool-error guard to embedded agent runs, wires it into attempt cleanup, and extends diagnostic loop events with a new detector value.

## Best Possible Solution

Fold any needed consecutive-error protection into the existing loop-detection and unknown-tool guard contracts, preserve transform cleanup order, add focused helper coverage, and require real behavior proof before merge.

No. The PR body shows repeated `Tool bash not found` snippets, but it does not provide a real OpenClaw environment, exact steps, expected and actual behavior, or after-fix runtime evidence.

## Solution Assessment

No. The direction may be useful, but this patch is not the best implementation because it does not compile as written, bypasses the existing loop-detection configuration boundary, and leaves cleanup order unsafe.

Full review comments:

- **[P1] Use the repository pi-agent-core package:** `src/agents/pi-embedded-runner/run/consecutive-error-guard.ts:16`
  - body: `package.json` only declares `@earendil-works/pi-agent-core`, and current source imports that package throughout the repo. This new type import points at `@mariozechner/pi-agent-core`, so TypeScript/module resolution fails before the guard can build.
  - confidence: 0.99
- **[P1] Declare or remove the sessionId option:** `src/agents/pi-embedded-runner/run/attempt.ts:2066`
  - body: `installConsecutiveErrorGuard` is called with `sessionId`, but `ConsecutiveErrorGuardParams` does not define that property. Because this is an object literal, TypeScript rejects the call; either add and use the field or stop passing it.
  - confidence: 0.98
- **[P2] Respect the existing loop-detection configuration:** `src/agents/pi-embedded-runner/run/attempt.ts:2059-2067`
  - body: `tools.loopDetection.enabled` defaults to false and current detectors return no block when disabled, but this installs a hard-aborting guard on every embedded attempt with hard-coded thresholds. That changes default runtime behavior and bypasses users who disabled loop detection.
  - confidence: 0.93
- **[P3] Unwind transformContext guards in reverse install order:** `src/agents/pi-embedded-runner/run/attempt.subscription-cleanup.ts:76-80`
  - body: This guard is installed after the tool-result/history-image transforms, but cleanup removes the older transform chain first and then restores the newly captured wrapper. That can leave `agent.transformContext` pointing at a stale disposed wrapper during cleanup; dispose the newest guard first or compose it into the existing remover.
  - confidence: 0.82
