## Label Justifications

- `P2`: This is a channel-specific reliability bug fix with real user impact but limited blast radius and no emergency signal.
- `merge-risk: 🚨 compatibility`: The PR changes existing unbounded LINE sends into hard 8s/10s timeouts, which can fail slow existing setups that previously waited.
- `merge-risk: 🚨 message-delivery`: The timeout does not cancel the underlying SDK request, so reply-to-push fallback can duplicate a message if LINE accepts the original reply late.

## Summary

Keep open: the patch addresses a real source-proven LINE hang, but it is not merge-ready because proof is mocked only, the branch is currently conflicting, and the non-canceling timeout can turn a late-success reply into a fallback duplicate send.

## What This Changes

The PR wraps LINE push and reply sends in 10s and 8s wall-clock timeouts, adds fake-timer send-helper coverage, and adds a changelog entry.

## Best Possible Solution

Use a bounded LINE send path that does not create fallback duplicate delivery, or explicitly approve the at-least-once delivery tradeoff with redacted real LINE proof before merge.

## Reproduction Assessment

Yes from source inspection: current main awaits LINE push/reply SDK promises without a timeout, and the v11.0.0 SDK fetch wrapper has no signal or timeout parameter. I did not run a live LINE hang because this review is read-only and no live token proof was provided.

## Solution Assessment

No as-is: the wall-clock wrapper is the narrowest way to unblock hangs, but it is not the best merge shape for reply fallback unless the late-settling request is canceled/idempotent or maintainers explicitly accept the duplicate-delivery risk.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Avoid fallback while the LINE reply is still in flight:** `extensions/line/src/send.ts:267-273`
  - body: `withTimeout` only races the SDK promise; it does not abort `client.replyMessage`. The existing callers catch any `replyMessageLine` rejection and send the same content via push, so a slow LINE reply that is accepted after the 8s local timeout can be followed by the fallback push and duplicate the user-visible message. Please make the LINE request cancellable/idempotent, or keep this timeout out of the reply-to-push fallback path.
  - confidence: 0.86
