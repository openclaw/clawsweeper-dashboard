- `P2`: This is a focused LINE channel reliability fix with limited blast radius, but it needs safer delivery semantics before merge.
- `merge-risk: 🚨 compatibility`: The PR changes existing slow-but-successful LINE sends into timeout errors after fixed 8s/10s budgets.

## Summary

Keep open: the PR targets a real LINE delivery gap, but the patch treats non-aborted LINE requests as failed after a wall-clock timeout, so existing fallback paths can send duplicates if the original request later succeeds; it also has only mocked proof for an external behavior change.

## What This Changes

The PR wraps LINE `pushMessage` and `replyMessage` calls in fixed `withTimeout` budgets, adds fake-timer send tests, and adds a changelog fix entry.

## Best Possible Solution

Use a cancelable LINE transport or an idempotent, duplicate-safe fallback design, then prove the normal, timeout, and fallback paths with redacted live LINE evidence before merge.

## Reproduction Assessment

No high-confidence live reproduction is present. Source and dependency inspection show unbounded awaits and no SDK abort plumbing, and the PR's mocked tests can simulate a hung promise, but no real LINE stalled-send or downstream fallback proof was supplied.

## Solution Assessment

No. Bounding the awaited call is the right area, but using a non-canceling timeout as a send failure is not the safest fix because fallback callers may duplicate delivery; a cancelable or idempotent path is safer.

- **[P1] Make timed-out sends duplicate-safe before fallback:** `extensions/line/src/send.ts:221-227`

Please make this path cancelable or otherwise duplicate-safe before treating timeout as send failure.
