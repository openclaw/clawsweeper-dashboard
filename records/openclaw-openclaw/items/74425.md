## Summary

Keep this PR open, but do not merge it as-is: current main still lacks this Windows CLI shutdown fix, while the proposed launcher-wide hard exit introduces observable CLI regressions.

## What This Changes

The PR adds a final `process.exit(process.exitCode ?? 0)` call after launcher help handling or built entry import completes.

## Best Possible Solution

Implement a narrower lifecycle fix that closes or drains the short-lived CLI resource keeping Windows commands alive while preserving async version/help output, JSON/piped output, and foreground command lifetimes.

## Reproduction Assessment

Yes for the PR regression: source inspection shows the root `--version` fast path can return from entry module evaluation before its async version output runs, then hit the new hard launcher exit. The underlying Windows hang has concrete reporter steps in #74378 but still needs Windows confirmation after revision.

## Solution Assessment

No. A launcher-wide `process.exit()` is broader than the bug, conflicts with the async root version fast path, and should be replaced by a drained or owner-specific shutdown boundary.
