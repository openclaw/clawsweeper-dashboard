## Summary

Keep open. The linked Windows regression still has a credible reproduction path, but this patch is not merge-ready because the blanket launcher hard-exit can truncate pending stdout/stderr and the user-facing fix lacks the required changelog entry.

## What This Changes

The PR appends an unconditional `process.exit(process.exitCode ?? 0)` with explanatory comments to the root `openclaw.mjs` launcher after the help/import path completes.

## Best Possible Solution

Fix the Windows short-lived CLI hang at the narrow cleanup or launcher-completion boundary while preserving `process.exitCode`, allowing stdout/stderr to drain, keeping foreground commands alive until stopped, and documenting the user-facing fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes. Related #74378 gives a high-confidence Windows npm reproduction for `status` and `--version`, including process observations after 3 and 13 seconds and a separate healthy gateway process; this review did not run a live Windows smoke.

## Solution Assessment

No. A final unconditional `process.exit()` may solve the lingering-process symptom, but Node’s process contract makes it unsafe for pending stdout/stderr; the safer direction is targeted resource cleanup or an explicit drain before any hard exit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Drain output before forcing launcher exit:** `openclaw.mjs:317`
  - body: Calling `process.exit(...)` at the launcher boundary applies to every completed CLI command. Node documents that this can terminate before pending stdout/stderr writes finish, so help, JSON, or piped output can be truncated; drain the streams first or fix the leaking handle instead.
  - confidence: 0.87
