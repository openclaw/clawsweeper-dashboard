## Summary

Keep open: the linked Windows CLI hang is a valid bug target, but this PR’s launcher-wide hard exit is broader than the completion path it is trying to fix and can break existing version/help/output behavior.

## What This Changes

The PR adds a final `process.exit(process.exitCode ?? 0)` at the end of `openclaw.mjs` after launcher help or entry import handling completes.

## Best Possible Solution

Use a narrower lifecycle cleanup for completed short-lived CLI commands that closes the leaking handle or waits for settled output, while preserving root version/help output and foreground command lifetimes.

## Reproduction Assessment

Yes for the PR regression: source inspection plus local Node checks show the new final hard exit can run before async version output and stream drains complete. The original Windows hang has reporter reproduction in the linked issue, but this review did not run a Windows npm smoke.

## Solution Assessment

No. A launcher-wide `process.exit()` is broader than the bug; the maintainable fix should target the leaking completion path or explicitly await or drain safe shutdown boundaries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve root version output:** `openclaw.mjs:317`
  - body: The new final `process.exit(...)` runs after `await import('./dist/entry.js')`, but the root `--version` path starts `resolveVersion().then(...)` and returns from module evaluation before that promise prints. This can make `openclaw --version` exit 0 with no output; await that path or avoid a launcher-wide hard exit.
  - confidence: 0.91
- **[P2] Drain output before forcing launcher exit:** `openclaw.mjs:317`
  - body: This hard exit applies after help and routed command output. `process.exit()` can terminate before pending stdout/stderr writes drain, so piped help/status/JSON output can be truncated; fix the leaking handle or explicitly wait for stream completion before exiting.
  - confidence: 0.86
