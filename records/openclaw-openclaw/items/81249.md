## Label Justifications

- `P2`: This is a real but bounded managed-proxy/Ollama embedding failure affecting self-hosted users, not a full core runtime outage.
- `impact:security`: The issue concerns the managed-proxy SSRF boundary and whether loopback traffic can bypass the filtering proxy.
- `impact:auth-provider`: The observable failure is in local Ollama provider routing for memory embeddings under managed proxy mode.

## Summary

Close as implemented on current main. The merged Ollama path now gives configured host-local loopback `/api/embed` requests a narrow guarded direct route under the default loopback mode while preserving SSRF checks, proxy mode, block mode, and the absence of a broad `NO_PROXY` bypass; the fix appears current-main-only rather than shipped in `v2026.5.22`.

## What This Changes

The issue asks for local Ollama memory embedding requests to keep working with `proxy.enabled=true` without reintroducing a broad local-network proxy bypass.

## Best Possible Solution

Keep the narrow exact-origin Ollama embedding bypass on main, avoid adding a general `proxy.bypass` knob, and have affected users retest from current main or the next release that contains the fix.

## Reproduction Assessment

Yes. The pre-fix release path is source-reproducible because the shipped Ollama embedding provider used the ordinary strict guarded fetch for `/api/embed` while managed proxy mode cleared `NO_PROXY` and routed strict fetches through the proxy; current main no longer uses that path for exact configured loopback embedding origins.

## Solution Assessment

Yes. The current main solution is narrower than the requested broad bypass: it is exact-origin, host-local loopback only, DNS-pinned, redirect-aware, private to bundled/internal use, and controlled by the existing `proxy.loopbackMode` policy.
