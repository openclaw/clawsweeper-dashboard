## Summary

Keep open. The linked merged provider SSRF work helps configured custom/local provider origins, but current main still routes normal loopback HTTP, including native Ollama embedding fetches, through the managed proxy while clearing NO_PROXY, so the requested local Ollama proxy exception remains an unresolved security/product decision.

## What This Changes

Request a secure config path for local Ollama embedding traffic to bypass operator-managed proxy routing without reopening general SSRF bypasses.

## Best Possible Solution

Define an owner-approved narrow local model-server proxy exception, or explicitly document that operator proxy allowlisting is the supported path; any new config must include SSRF and managed-proxy regression coverage.

## Reproduction Assessment

No live high-confidence reproduction was run. Source inspection shows proxy.enabled clears NO_PROXY and routes normal loopback HTTP through the managed proxy while Ollama embeddings use guarded fetch to /api/embed, so the routing premise is source-reproducible but the exact 503 depends on the reporter's Gluetun/container setup.

## Solution Assessment

Unclear. The linked exact-origin provider SSRF fix is useful adjacent work, but the best solution for this issue requires a narrower managed-proxy/local-model policy decision rather than treating general NO_PROXY or a broad bypass list as safe by default.
