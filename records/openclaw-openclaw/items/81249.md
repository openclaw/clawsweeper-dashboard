## Label Justifications

- `P2`: The issue affected a real but bounded self-hosted Ollama embedding workflow under managed proxy mode, not the whole runtime.
- `impact:security`: The requested behavior touches the managed-proxy SSRF boundary and local loopback bypass policy.
- `impact:auth-provider`: The observable failure is provider routing for local Ollama memory embeddings when proxy mode is active.

## Summary

Current main now implements the narrow Ollama memory-embedding fix: bundled Ollama `/api/embed` requests can bypass the managed proxy only for the exact configured host-local loopback origin after SSRF/DNS validation, while broad `NO_PROXY` or general local-network bypasses remain unsupported.

## What This Changes

The issue asks for local Ollama memory embeddings to work under managed proxy mode without re-enabling broad NO_PROXY-style bypasses.

## Best Possible Solution

Keep the merged exact configured-loopback Ollama embedding implementation on main, leave broad local-network bypasses out of the proxy contract, and track the separate browser/CDP path in its own PR.

## Reproduction Assessment

Yes. Before the fix, the source path was high-confidence: configure Ollama embeddings to `http://127.0.0.1:11434` with a managed proxy that cannot reach host loopback, and the embedding request went through that proxy; current main now has focused regression coverage for the corrected path.

## Solution Assessment

Yes. The merged fix is narrower than the proposed `proxy.bypass`/`allowLocal` knob because it applies only to the exact configured Ollama loopback embedding origin after SSRF/DNS validation and keeps broad local-network traffic on the proxy path.
