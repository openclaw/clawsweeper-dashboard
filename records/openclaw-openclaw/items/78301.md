## Summary

Keep open: current main still lacks plugin-aware diagnostics for the reported fetch/header failure path, while the runtime-injection part needs maintainer interpretation because current code and docs still expose `api.runtime`. The request is useful, but the suggested header-stripping behavior touches security hardening and should be designed deliberately.

## What This Changes

Request host-side diagnostics for invalid or legacy plugin contracts, including manual `Content-Length` failures, runtime migration warnings, and plugin-author migration docs.

## Best Possible Solution

Add targeted host diagnostics that preserve existing SSRF/undici hardening, name the offending plugin and root cause for invalid request headers, and clarify the current runtime-entrypoint migration path in SDK docs/tests.

## Reproduction Assessment

Yes for the fetch diagnostic path: source inspection plus a Node 22 snippet reproduce the generic `TypeError: fetch failed` with the real undici cause hidden under `err.cause`. No full OpenClaw plus Tencent plugin restart-loop reproduction was run in this checkout.

## Solution Assessment

Unclear as proposed: better diagnostics are the right direction, but stripping and warning on manual `Content-Length` would soften a security-relevant hard failure. A safer first fix is to preserve rejection while surfacing plugin id, request context, and undici cause, then separately document runtime migration expectations.
