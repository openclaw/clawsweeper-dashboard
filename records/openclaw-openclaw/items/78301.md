## Summary

Keep open: current main still does not provide plugin-aware diagnostics for the hidden undici `Content-Length` failure, and the newer duplicate-module/runtime-store concern remains a real diagnostics gap rather than a resolved duplicate. The request is useful, but the exact fix needs maintainer judgment because stripping invalid `Content-Length` would change security-sensitive hard-fail behavior.

## What This Changes

The issue asks for clearer plugin-loader diagnostics for invalid plugin contracts, including hidden undici `Content-Length` failures, corrected runtime-entry expectations, duplicate module-instance symptoms, and migration guidance for plugin authors.

## Best Possible Solution

Keep the issue open and design a targeted diagnostics contract that preserves undici/SSRF hardening, names the offending plugin and root cause, and documents or warns about duplicate module-instance runtime storage with `createPluginRuntimeStore`.

## Reproduction Assessment

Yes for the fetch diagnostic path: current source plus a Node 22 fetch probe reproduce the generic `TypeError: fetch failed` with the actionable undici cause hidden under `err.cause`. The full OpenClaw plus Tencent plugin restart loop was not rerun here, but two issue commenters provided real environment verification.

## Solution Assessment

Unclear as proposed: better diagnostics are the right direction, but stripping manual `Content-Length` would soften a security-relevant hard failure. A safer first design is to preserve rejection while surfacing plugin id, request context, and undici cause, then add duplicate-module/runtime-store guidance separately.
