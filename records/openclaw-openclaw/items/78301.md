## Summary

Keep open: current main still lacks a plugin-aware diagnostic for the ordinary `Content-Length`/undici failure path, and the reporter's corrected module-duplication case remains a real SDK/loader diagnostic question. The original runtime-declaration example was corrected in discussion, so the remaining work should focus on error context, duplicate module/runtime-store guidance, and migration docs rather than treating `api.runtime` as removed.

## What This Changes

Request clearer plugin loader/runtime diagnostics for invalid plugin-authored `Content-Length` headers, duplicate module/runtime-store pitfalls, and 2026.4.x to 2026.5.x plugin migration guidance.

## Best Possible Solution

Keep the hardening by default, add targeted plugin-aware diagnostics that surface the undici cause and likely offending header, and document the runtime-store/runtimeExtensions migration path without deprecating `api.runtime`.

## Reproduction Assessment

Yes for the central fetch diagnostic path: source shows ordinary plugin `Content-Length` headers reach undici, and a focused Node fetch reproduces the generic `TypeError` with the actionable cause under `err.cause`. The duplicate-module case is supported by the reporter's corrected investigation and current runtime-store docs, but I did not run the Tencent plugin end to end.

## Solution Assessment

Unclear as proposed: better diagnostics are the right direction, but automatically stripping `Content-Length` changes security semantics. The safer maintainable path is to preserve the fail-closed behavior unless maintainers choose otherwise, then surface plugin id, request context, and `err.cause` plus docs guidance.

- Duplicate-module detection needs a precise trigger based on actual distinct resolved module paths/URLs to avoid noisy false positives for valid `extensions` plus `runtimeExtensions` packages.
