## Summary

Keep open: current main still does not add plugin-aware diagnostics for the reported invalid `Content-Length` fetch failure or for duplicate runtime module instances, even though the external Weixin plugin has since fixed its own bugs. The useful core request remains a diagnostics/product decision, not an auto-close candidate.

## What This Changes

The issue asks OpenClaw to add clearer plugin-loader/runtime diagnostics for invalid plugin contracts, including hidden undici `Content-Length` failures, duplicate module-instance pitfalls, and migration guidance.

## Best Possible Solution

Preserve the hardened fetch behavior, add targeted diagnostics that name the plugin and root undici cause, and document or warn about runtime-store-safe patterns for split source/runtime module loading.

## Reproduction Assessment

Yes for the fetch diagnostic path: current source still delegates the mismatched `Content-Length` failure without translating `err.cause`, and a local Node snippet reproduces the generic `TypeError: fetch failed` wrapper. The full Tencent restart loop was not rerun, and that external plugin has since fixed the exact 2.4.1 bugs in 2.4.2.

## Solution Assessment

Unclear as proposed: better diagnostics are the right direction, but silently stripping `Content-Length` changes a security boundary. The safer maintainer-owned solution is to keep rejection hard, enrich the error/log context, and handle duplicate-module/runtime-store guidance as diagnostics or docs.
