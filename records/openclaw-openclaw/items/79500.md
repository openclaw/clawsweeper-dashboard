## Summary

Keep open. Current main still returns without appending `DREAMS.md` when the dreaming narrative run finishes but `getSessionMessages` sees no assistant text, and multiple current-release comments report the same orphan-transcript/no-diary behavior; only the legacy context-engine initialization half is fixed.

## What This Changes

The issue reports that memory-core dreaming writes phase reports but fails to append generated narrative subagent replies to `DREAMS.md`, with a related legacy context-engine initialization warning.

## Best Possible Solution

Make the detached Dream Diary narrative result durable before cleanup, or read the actual generated assistant text from a reliable run/transcript source, while keeping deterministic fallback as a last-resort no-text path.

## Reproduction Assessment

Yes, source-level: current main has a deterministic path where `waitForRun` succeeds, `getSessionMessages` returns no assistant text, and the function returns before writing `DREAMS.md`. I did not run the full overnight cron scenario.

## Solution Assessment

Unclear as a final fix. https://github.com/openclaw/openclaw/pull/78698 is a useful fallback workaround, but the best full fix should preserve the generated narrative by making session registration/transcript visibility reliable before cleanup.
