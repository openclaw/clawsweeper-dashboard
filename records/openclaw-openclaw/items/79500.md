## Summary

Keep open. Current main still returns without appending `DREAMS.md` when a completed Dream Diary run yields no visible assistant message, v2026.5.12 still ships that path, and fresh reporter comments reproduce the symptom across `claude-cli`, MiniMax, and OpenAI-backed installs; only the legacy context-engine initialization half is already fixed.

## What This Changes

The issue reports that generated memory-core Dream Diary narrative text is left in detached/orphaned subagent transcripts instead of being appended to `DREAMS.md`, with an additional CLI legacy context-engine initialization warning.

## Best Possible Solution

Make Dream Diary subagent completion durable before cleanup, preferably by reading the actual run result or a flushed/registered transcript, and reserve deterministic fallback prose for genuine terminal/no-text failures.

## Reproduction Assessment

Yes, at source and reporter-evidence level. Current main has a clear path where `waitForRun` succeeds, `sessions.get` returns empty messages for an unregistered or not-yet-flushed narrative session, and `generateAndAppendDreamNarrative` returns without writing `DREAMS.md`; I did not execute the overnight cron scenario.

## Solution Assessment

Unclear as a final fix. Current main fixes the context-engine initialization warning, and https://github.com/openclaw/openclaw/pull/78698 improves silent no-text outcomes with fallback prose, but preserving the generated narrative likely needs a transcript/session registration or run-result extraction fix.
