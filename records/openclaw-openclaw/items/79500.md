## Summary

Keep open. Current main still has the source-backed wait-then-read Dream Diary path that can return before appending `DREAMS.md` when no assistant text is visible, while the legacy context-engine initialization half is already fixed and the remaining no-write behavior is being handled by the open fallback PR.

## What This Changes

The issue reports that CLI-backed memory-core Dream Diary subagent output is generated but not appended to `DREAMS.md` because `waitForRun` is not a transcript-flush barrier and CLI compaction previously missed context-engine initialization.

## Best Possible Solution

Preserve the CLI compaction initialization guard, then fix the remaining Dream Diary no-text path either by landing an accepted fallback write or by extracting/waiting for the actual generated narrative before cleanup.

## Reproduction Assessment

Yes, source-reproducible with high confidence: a CLI-backed run can emit lifecycle success before transcript persistence, and the Dream Diary code immediately reads session messages and returns without appending if no assistant text is visible. I did not execute the full overnight cron scenario.

## Solution Assessment

Unclear as a final fix. The current open PR addresses the silent no-write class with fallback prose, but the report specifically says real generated text exists later, so a transcript-flush barrier or run-result extraction path may be the more faithful fix.
