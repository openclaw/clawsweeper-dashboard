## Label Justifications

- `P2`: The bug affects a bounded memory-core Dream Diary output path with real user reports, but it does not make the core runtime unusable.
- `impact:data-loss`: The report is about generated diary text being present in transcripts but not persisted into DREAMS.md, then potentially archived by cleanup.
- `impact:session-state`: The failure centers on detached subagent session registration, transcript visibility, and getSessionMessages returning no assistant content.

## Summary

Keep open. Current main and the latest release still have the Dream Diary wait-then-read/no-text return path, while the legacy context-engine initialization half is already fixed; the active fallback PR is related but does not fully settle whether generated narrative text should be preserved.

## What This Changes

The issue reports that memory-core dreaming generates narrative subagent text but fails to append it to DREAMS.md because the diary path reads an unregistered or not-yet-flushed subagent session, with a separate legacy context-engine initialization warning.

## Best Possible Solution

Make Dream Diary narration read from a durable run result or transcript/session-store signal before cleanup, while keeping the CLI context-engine initialization guard and using fallback prose only for true terminal/no-text outcomes if maintainers accept that behavior.

## Reproduction Assessment

Yes, at source level. Current main can complete waitForRun with status ok, immediately read getSessionMessages, and return without appending DREAMS.md when no assistant text is visible; multiple reporters supplied matching v2026.5.12 logs, but I did not run a live overnight cron sweep.

## Solution Assessment

Unclear as a final fix. Current main fixes the context-engine initialization half, and https://github.com/openclaw/openclaw/pull/78698 covers fallback/no-text writes, but the best durable fix may need to wait for or extract the actual generated narrative instead of replacing it with fallback text.
