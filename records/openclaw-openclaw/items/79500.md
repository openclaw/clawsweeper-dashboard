## Summary

Keep open. Current main and the latest release still read Dream Diary messages immediately after `waitForRun({ status: ok })` and return without appending `DREAMS.md` when no assistant text is visible; the separate legacy context-engine initialization gap is already fixed, and https://github.com/openclaw/openclaw/pull/78698 is the active fallback-oriented candidate for the remaining no-text behavior.

## What This Changes

The issue reports that CLI-backed memory-core dreaming can generate narrative text but fail to append it to `DREAMS.md` because transcript persistence races `waitForRun` and CLI compaction may run before legacy context engines are initialized.

## Best Possible Solution

Keep the context-engine initialization guard, then either land the focused fallback fix if maintainers accept fallback diary prose for no-text runs or replace it with a transcript-flush/run-result extraction fix that preserves the generated narrative.

## Reproduction Assessment

Yes, source and report evidence give a high-confidence path: a CLI-backed run can report `agent.wait` success before transcript persistence, and the Dream Diary code immediately reads session messages and returns without appending if no assistant text is visible. I did not execute the full overnight cron scenario.

## Solution Assessment

Unclear as a final fix. Current main only fixes the context-engine initialization half; https://github.com/openclaw/openclaw/pull/78698 covers timeout/no-text fallback, but maintainers should decide whether fallback is acceptable for this report's generated-text race or whether the code should wait for/pull the actual flushed narrative.
