## Summary

Keep open: current main still matches the reported Dream Diary race and the CLI compaction context-engine initialization gap, while overlapping open PRs only cover parts of the problem.

## What This Changes

The issue asks for CLI-backed dreaming narrative output to reliably reach `DREAMS.md` instead of being lost when `agent.wait` returns before transcript persistence and CLI compaction has not initialized context engines.

## Best Possible Solution

Land or adapt the existing CLI compaction initialization fix, then make Dream Diary extraction depend on a flushed assistant transcript or the completed run payload and cover both paths with focused regressions.

## Reproduction Assessment

Yes, source inspection plus the reporter's logs give a high-confidence path: a CLI-backed dreaming narrative run can resolve `agent.wait` on lifecycle `end` before transcript persistence makes the assistant message visible to `getSessionMessages`. I did not run the live overnight Darwin sweep in this read-only review.

## Solution Assessment

Yes, the maintainable fix is narrow runtime repair: initialize context engines before CLI compaction resolution and make narrative extraction read from a transcript-flushed signal or run result, rather than adding new config or accepting fallback-only output as the full fix.
