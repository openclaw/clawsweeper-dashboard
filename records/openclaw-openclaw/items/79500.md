## Summary

Keep open. Current main and v2026.5.12 still have the wait-for-run then immediate session-message read path that can return before appending `DREAMS.md`; the legacy context-engine initialization half is fixed, but the generated-narrative loss is still active and has a fresh v2026.5.12 confirmation.

## What This Changes

The issue reports that Dream Diary narrative text generated by dreaming subagents is not appended to `DREAMS.md` because the diary path reads session messages before transcript persistence is complete, with a separate CLI context-engine initialization warning.

## Best Possible Solution

Preserve generated diary text by making run completion wait for transcript persistence or by extracting the assistant narrative from a reliable run result payload, while keeping the CLI context-engine initialization guard and using fallback prose only as a deliberate mitigation.

## Reproduction Assessment

Yes, with high confidence from source and reporter logs. A CLI or similar subagent run can report `ok` before session transcript persistence is visible, and current main immediately reads session messages and returns before appending `DREAMS.md` when no assistant text is found.

## Solution Assessment

No current-main solution exists for the generated-text race. The narrow maintainable fix is a transcript-flush/run-result extraction path; https://github.com/openclaw/openclaw/pull/78698 is useful fallback work but needs maintainer review because it may write fallback text instead of the actual generated narrative.
