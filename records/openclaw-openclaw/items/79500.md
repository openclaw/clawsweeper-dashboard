## Summary

Keep open: current main fixed the legacy context-engine initialization part, but the Dream Diary path still treats `agent.wait` success as enough and reads the session transcript once before the CLI assistant message is guaranteed to be visible.

## What This Changes

The issue reports that CLI-backed `memory-core` dreaming generates narrative text but fails to append it to `DREAMS.md` because transcript persistence can lag behind `waitForRun`, with a related legacy context-engine initialization warning.

## Best Possible Solution

Keep the context-engine initialization fix and make Dream Diary extraction wait for a flushed assistant transcript or use an existing completed-run payload before warning, falling back only after that bounded read path fails.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main can resolve `waitForRun` from lifecycle `end` before `persistCliTurnTranscript` appends the CLI assistant message, and `memory-core` then reads `getSessionMessages` once. I did not run the live 3am cron sweep.

## Solution Assessment

Yes, the best fix is a narrow runtime ordering repair. The fallback PR helps avoid an empty diary, but it should not replace a generated narrative that becomes visible milliseconds after `agent.wait`.

Implement a narrow barrier/retry or existing-result-payload extraction path so generated narrative text is appended to `DREAMS.md` when it appears shortly after wait completion.
