## Summary

Keep open. Current main still returns before appending `DREAMS.md` when the Dream Diary subagent reports success but `sessions.get` has no assistant text; the CLI context-engine initialization half is fixed, and https://github.com/openclaw/openclaw/pull/78698 is an active fallback-oriented fix candidate rather than a merged resolution.

## What This Changes

The issue reports that memory-core dreaming writes phase artifacts but loses generated Dream Diary text before it reaches `DREAMS.md`, with an additional CLI compaction context-engine initialization gap reported on older builds.

## Best Possible Solution

Make the Dream Diary writer durable by reading the actual completed narrative from a flushed transcript or run result, keep the CLI context-engine initialization guard, and use the fallback PR only as a safety net for terminal/no-text runs.

## Reproduction Assessment

Yes, at source level: a dreaming narrative run that reaches `status: ok` before `sessions.get` can see an assistant message follows the current no-text warning and return path without appending `DREAMS.md`. I did not execute the full cron scenario.

## Solution Assessment

Unclear as a final fix. The open fallback PR is useful for silent no-text outcomes, but the best durable fix should preserve the generated narrative by waiting for or directly reading the completed assistant output.
