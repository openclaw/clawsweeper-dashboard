## Summary

Keep open: current main has fixed the legacy context-engine initialization gap, but the Dream Diary race remains source-visible because `agent.wait` can still resolve before CLI transcript persistence makes the assistant message available to `getSessionMessages`.

## What This Changes

The issue reports that CLI-backed Dream Diary subagent output is generated but never appended to `DREAMS.md` because narrative extraction reads session messages too early and CLI compaction previously missed context-engine initialization.

## Best Possible Solution

Make Dream Diary extraction depend on a flushed assistant transcript or an already-available run result payload, while preserving the current context-engine initialization fix and existing Dream Diary defaults.

## Reproduction Assessment

Yes, source and reporter log evidence give a high-confidence path: a CLI-backed dreaming run can return `agent.wait` status `ok` before `persistCliTurnTranscript` appends the assistant message, causing `getSessionMessages` to return no narrative text.

## Solution Assessment

Yes, a narrow runtime fix is the best path: do not add a new user config or change Dream Diary policy; instead make the existing narrative read wait for durable assistant text or consume a reliable completed-run payload.
