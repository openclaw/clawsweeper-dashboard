## Summary

Keep open: current main has fixed the CLI context-engine initialization gap, but the Dream Diary path still treats `agent.wait` success as a transcript-flush barrier and immediately reads session messages, so the reported dropped-narrative failure remains source-reproducible.

## What This Changes

The issue reports that CLI-backed `memory-core` dreaming generates Dream Diary prose, but the prose is not appended to `DREAMS.md` because session transcript persistence can lag behind `waitForRun`.

## Best Possible Solution

Make Dream Diary extraction depend on flushed assistant output or a run-result payload, keep the existing CLI context-engine init fix, and add focused regression coverage for delayed CLI transcript visibility.

## Reproduction Assessment

Yes, source inspection and the reporter's logs provide a high-confidence source-level reproduction path: a CLI-backed Dream Diary run can report `agent.wait` success before the assistant transcript is visible to `getSessionMessages`. I did not execute the live scheduled Darwin sweep.

## Solution Assessment

Yes, this should be a narrow runtime fix rather than a new feature: use a flushed transcript signal or completed-run assistant payload before appending `DREAMS.md`, and leave the separate machine-only opt-out request for its own product decision.
