## Summary

Keep open: current main fixed the legacy context-engine initialization half, but the Dream Diary extraction still waits for run completion and immediately reads session history even though `agent.wait` is not shown to be a transcript-flush barrier.

## What This Changes

The issue reports that memory-core dreaming narrative subagent output can be generated but never appended to `DREAMS.md` because run completion can race transcript persistence and CLI compaction previously missed legacy context-engine initialization.

## Best Possible Solution

Make Dream Diary extraction depend on flushed assistant output or the completed run payload, keep the current CLI context-engine initialization fix, and preserve generated narrative text rather than only masking the race with fallback prose.

## Reproduction Assessment

Yes at source level: a CLI-backed narrative run can report `agent.wait` success from lifecycle end before transcript persistence completes, and the issue includes logs showing the no-text warning before the assistant JSONL write. I did not run the overnight Darwin sweep locally in this read-only review.

## Solution Assessment

Yes, the narrow fix is to read from a flushed transcript or run result before appending `DREAMS.md`; the context-engine initialization part is already fixed by https://github.com/openclaw/openclaw/pull/79484. The fallback approach in https://github.com/openclaw/openclaw/pull/78698 is useful for terminal no-text cases but should not replace preserving the generated narrative when it exists.
