## Summary

Keep open: current main has fixed the legacy context-engine initialization part, but the central Dream Diary race remains because the narrative path still treats `waitForRun` success as sufficient and immediately reads session messages before CLI transcript persistence is guaranteed visible. Related open PRs address timeout or fallback behavior, not preserving the late generated assistant narrative reported here.

## What This Changes

Fix CLI-backed memory-core Dream Diary runs so generated narrative text reliably reaches `DREAMS.md` instead of being left only in orphaned session transcripts.

## Best Possible Solution

Make the Dream Diary path append the real completed assistant narrative only after it is available, using a bounded post-ok transcript settle/read or an existing typed run-result text seam, while keeping the current CLI context-engine initialization fix.

## Reproduction Assessment

Yes for source-level reproduction: current main still calls `waitForRun`, then immediately reads `getSessionMessages`, while `agent.wait` can resolve from lifecycle/dedupe before CLI transcript persistence is complete. I did not run the full live Darwin `claude-cli` cron sweep.

## Solution Assessment

Yes: the narrowest maintainable fix is a runtime ordering repair for the Dream Diary extraction path, not a new config option or timeout policy. Existing related PRs for fallback and timeout policy should be coordinated with, but they do not replace preserving the real generated text.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79500 on current main. The remaining bug is that `extensions/memory-core/src/dreaming-narrative.ts` treats `subagent.waitForRun({ status: "ok" })` as enough, immediately calls `getSessionMessages`, and drops the real CLI-generated narrative if transcript persistence becomes visible a few milliseconds later. Keep the existing CLI compaction `ensureContextEnginesInitialized()` behavior; do not add a new timeout/config policy and do not duplicate the fallback-only work from https://github.com/openclaw/openclaw/pull/78698 or the timeout-policy work from https://github.com/openclaw/openclaw/pull/76342. Prefer a bounded post-ok retry/settle read in memory-core, or use an existing typed run-result assistant-text seam if one already exists without expanding public API. Add focused regression coverage where `waitForRun` returns ok, the first `getSessionMessages` has no assistant text, a later bounded read sees the assistant narrative, and `DREAMS.md` receives that real text. Update changelog if the fix is user-visible, and stop if the implementation requires a new public plugin/runtime API decision.
