## Summary

Keep open: the legacy context-engine registration half is now fixed on current main and in v2026.5.12, but the core Dream Diary race remains. Current code still treats `agent.wait` success as enough to read session messages, while the agent lifecycle can report success before CLI transcript persistence appends the assistant text.

## What This Changes

The issue reports that CLI-backed memory-core dreaming can generate diary prose but fail to append it to `DREAMS.md` because narrative extraction races transcript persistence and previously missed context-engine initialization.

## Best Possible Solution

Keep the CLI context-engine initializer fix, then make Dream Diary narrative extraction wait for durable assistant text or read it from an already-completed run result before returning no text.

## Reproduction Assessment

Yes, source-level: a CLI-backed narrative run can return `agent.wait` status `ok` from lifecycle before transcript persistence finishes, and `generateAndAppendDreamNarrative` immediately reads session messages and exits if the assistant text is not visible. I did not run the live nightly Darwin sweep.

## Solution Assessment

Yes, the best fix is narrow runtime behavior: preserve the existing context-engine initializer guard and add a transcript-flush/read barrier or durable result-based extraction for the Dream Diary path, without adding a timeout or configuration feature.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/79500 on current main. The remaining bug is that memory-core Dream Diary narrative generation awaits `subagent.waitForRun({ status: "ok" })` and immediately calls `getSessionMessages`; `agent.wait` can resolve from the lifecycle end before CLI transcript persistence appends the assistant message, so `DREAMS.md` is not updated. Implement a narrow fix that makes narrative extraction depend on durable assistant text, preferably by adding a bounded retry/read barrier around `getSessionMessages` in `extensions/memory-core/src/dreaming-narrative.ts` or by using an existing durable run-result payload if one is already available. Preserve the CLI context-engine initializer guard in `src/agents/command/cli-compaction.ts`; do not add a new timeout/config option or change unrelated Dream Diary fallback policy. Add regression coverage where `getSessionMessages` first returns no assistant message after a successful wait and then returns the assistant narrative, and verify `DREAMS.md` is appended. Add a changelog fix entry if this becomes a user-facing repair PR.
