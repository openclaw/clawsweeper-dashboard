## Label Justifications

- `P2`: This was a real user-visible memory-core Dream Diary output loss, but the affected surface is limited to dreaming/session persistence rather than core runtime availability.
- `impact:data-loss`: The reported failure drops or orphans generated Dream Diary text instead of appending it to `DREAMS.md`.
- `impact:session-state`: The root cause depends on subagent run/session/transcript state becoming visible before `getSessionMessages` reads it.

## Summary

Close as implemented on current main. The Dream Diary code still reads the narrative through `waitForRun` then `getSessionMessages`, but current main now defers the terminal lifecycle event that powers `agent.wait` until after transcript persistence and CLI compaction, and the legacy context-engine initialization gap is also fixed; the fix is not in v2026.5.20 yet.

## What This Changes

The issue reports that memory-core dreaming generates narrative subagent output but fails to append it to `DREAMS.md` because the diary writer reads no assistant text after `waitForRun`, with an earlier CLI context-engine initialization warning as a compounding defect.

## Best Possible Solution

Keep the current main post-turn lifecycle barrier as the generic fix for plugin-runtime `waitForRun`, and leave deterministic fallback diary behavior for genuinely missing assistant text to the separate fallback PR.

## Reproduction Assessment

Yes. The report has multiple real-install confirmations, and source history shows v2026.5.20 could emit terminal lifecycle before transcript persistence while memory-core immediately read session messages after `waitForRun`; I did not rerun an overnight cron sweep.

## Solution Assessment

Yes. Deferring terminal lifecycle `end` until after transcript persistence and compaction fixes the generic `waitForRun` contract, which is cleaner than adding memory-core polling around one diary path; fallback diary prose remains a separate no-output policy path.
