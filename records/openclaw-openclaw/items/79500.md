## Summary

Keep open. Current main and v2026.5.12 still have the remaining Dream Diary no-text race: after a successful subagent wait, memory-core immediately reads session messages and returns without appending `DREAMS.md` when no assistant text is visible; the legacy context-engine initialization half is fixed, but recent issue comments confirm the diary path still fails on v2026.5.12 and non-claude-cli runtimes.

## What This Changes

The issue asks for Dream Diary narrative text generated by memory-core dreaming subagents to reliably reach `DREAMS.md` instead of being stranded in late-written session transcripts.

## Best Possible Solution

Keep the CLI compaction initialization guard, then make the Dream Diary path read from a transcript-flushed signal or run-result payload so completed generated narratives are appended reliably; fallback-only behavior can remain a separate mitigation if maintainers accept it.

## Reproduction Assessment

Yes, source-level and report-backed. A completed subagent run can produce lifecycle success before `persistCliTurnTranscript` flushes, while `generateAndAppendDreamNarrative` immediately reads session messages and returns without appending when no assistant text is visible; I did not execute the full overnight cron path.

## Solution Assessment

Unclear for the currently linked fallback approach. A fallback write prevents an empty diary, but the best bug fix should preserve the generated narrative by waiting for transcript persistence or extracting the final assistant text from the run result.
