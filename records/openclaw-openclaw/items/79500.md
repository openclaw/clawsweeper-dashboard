## Summary

Keep open. Current main still has the Dream Diary no-text path: after a successful subagent wait it reads session messages and returns without appending `DREAMS.md` when none are visible, while recent comments report the same failure on v2026.5.12; the legacy context-engine initialization half is fixed, but the generated-narrative handoff is not.

## What This Changes

Fix memory-core dreaming so generated narrative subagent text is appended to `DREAMS.md` instead of being lost after successful runs, and keep the CLI compaction context engine initialized.

## Best Possible Solution

Keep the CLI compaction initialization guard, then fix the Dream Diary handoff so memory-core appends the actual generated narrative after transcript/session registration is durable, with fallback diary text only as a deliberate terminal-path backup.

## Reproduction Assessment

Yes. Source inspection plus multiple v2026.5.12 reports give a high-confidence path: Dream Diary waits for the run, reads session messages, and returns without appending when the session lookup is empty or not yet flushed; I did not run the overnight cron scenario.

## Solution Assessment

Unclear as a final fix. The open fallback PR is a useful workaround for terminal/no-text outcomes, but the best root fix should preserve and append the generated narrative by fixing the session registration/flush boundary or reading the run result directly.
