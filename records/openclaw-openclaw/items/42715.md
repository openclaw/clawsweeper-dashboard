## Summary

Keep this open: current main has fallback decision warnings and missing remote-provider auth now fails before sending an unauthenticated request, but auth-classified failures still intentionally fall through the model fallback chain and the requested gateway-level credential sharing is not present.

## What This Changes

The issue asks OpenClaw to avoid silent fallback when a second Cloudflare AI Gateway provider lacks `apiKey`, ideally by sharing gateway credentials, failing loudly on auth errors, and documenting multi-provider setup.

## Best Possible Solution

Maintainers should decide the policy boundary first, then implement either a targeted fail-closed/notice path for selected-model auth failures, a documented gateway credential inheritance model, or a narrower docs-only clarification.

## Reproduction Assessment

Yes for the source-level fallback behavior: current tests and code show 401/auth-classified failures can fall back to another configured/default model. The exact live Cloudflare AI Gateway 401/2009 path was not exercised.

## Solution Assessment

No clear best fix yet. A blanket auth-error fail-closed change would fight existing fallback behavior, while gateway-level `apiKey` inheritance would add new config semantics, so maintainer direction is needed before implementation.
