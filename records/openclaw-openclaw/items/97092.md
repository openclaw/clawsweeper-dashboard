## Summary

Keep open: the patch looks like the right core-level safety fix for the linked bug, but the protected maintainer label and mock-only after-fix proof mean this should stay on the maintainer review path before merge.

## What This Changes

The PR gates agent-core tool execution on `stopReason: "toolUse"`, strips tool-call blocks from non-executable final assistant messages, and adds a regression test for length-truncated tool-call streams.

## Best Possible Solution

Land the core guard after maintainers accept the safety-first stopReason contract and get adequate after-fix runtime proof, while keeping provider adapters responsible for marking complete tool-call turns as `toolUse`.

## Reproduction Assessment

Yes. Current main source shows `runLoop` executes every returned `toolCall` regardless of `stopReason`, and the linked report provides live before-fix logs; I did not run tests because this review is read-only.

## Solution Assessment

Yes, this appears to be the best fix layer: agent-core owns final tool dispatch and replay context, while provider adapters already normalize complete tool-use turns to `toolUse`. The remaining question is proof and compatibility, not an alternate code location.
