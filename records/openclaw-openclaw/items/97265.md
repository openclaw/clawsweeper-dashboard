## What This Changes

The PR changes `tools.effective` to load global session stores with the requested agent id, limits requested-agent ownership override to canonical `global` keys, and adds unit, handler-integration, and Gateway coverage.

## Best Possible Solution

Review and land this focused gateway fix after required CI completes, keeping `tools.effective` global-agent store selection aligned with sibling session and chat RPC ownership rules.

## Reproduction Assessment

Yes at source level: current main validates the requested agent but then loads and resolves `global` without that agent, so a non-default requested agent can mismatch the default session agent. I did not run tests locally because this review is read-only.

## Solution Assessment

Yes. Passing the requested agent into the existing store-target resolver, then allowing the session-agent override only for canonical `global`, is the narrow maintainable fix and avoids duplicating sibling validation logic.
