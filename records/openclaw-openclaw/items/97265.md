## Summary

## What This Changes

## Best Possible Solution

Keep the selected-global store scoping fix, but narrow the session-agent override to canonical global keys, add a regression for known-agent/non-global mismatch rejection, and require redacted real Gateway or Control UI proof before merge.

## Reproduction Assessment

Yes. Current main and v2026.6.10 show the reported global-session bug from source, and the patch regression is also source-reproducible because `resolveSessionAgentId` prefers explicit `agentId` over the session key.

## Solution Assessment

No. Passing the agent id into `loadSessionEntry` is the right narrow direction for global sessions, but passing it unconditionally into `resolveSessionAgentId` is too broad and bypasses the non-global mismatch guard.
