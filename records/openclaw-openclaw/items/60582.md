## Label Justifications

- `P2`: This is a bounded Discord improvement with meaningful workflow impact but no demonstrated emergency or core-runtime outage.
- `impact:security`: Category-level entries change the authorization boundary by granting access to every current and future child channel.
- `impact:message-loss`: Messages from unlisted child channels are denied even when operators intend the containing category to be allowed.

## Summary

Current main still lacks category-level matching for ordinary Discord child channels: the shared resolver supports parent fallback, but inbound preflight supplies parent metadata only for threads. The narrower stable-category-ID request remains useful alongside the broader glob proposal and requires maintainer product/security approval because it authorizes future child channels automatically.

## What This Changes

Add category-level Discord guild allowlist entries whose settings apply to child channels without per-channel config changes or gateway restarts.

## Best Possible Solution

Reuse the canonical direct/parent/wildcard matcher, carry stable category metadata into ordinary channel resolution, define precedence as direct channel over category over wildcard, and extend thread resolution to reach the grandparent category with focused allow, deny, and override tests plus Discord documentation.

## Reproduction Assessment

Yes, at high confidence from current source: ordinary child-channel preflight does not pass the fetched category parent ID to the existing fallback matcher, whereas thread messages pass their immediate parent channel. A live Discord reproduction was not run during this read-only review.

## Solution Assessment

Yes in general, but the maintainable path is not a new `category:` syntax or parallel config block; stable category IDs should use the existing `channels` map and canonical parent-fallback precedence, with explicit thread-grandparent handling.
