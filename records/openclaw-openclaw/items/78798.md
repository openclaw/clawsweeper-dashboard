## Summary

Keep open: current main has adjacent plugin command registration, argument parsing, command listing, and `continueAgent`, but it still lacks the requested Plugin SDK contract for a command handler to pass a replacement same-turn user message body to the LLM.

## What This Changes

The issue requests a Plugin SDK API for plugin-registered slash commands that run custom logic, then either pass the remaining command text through to the LLM or block continuation.

## Best Possible Solution

Add a backward-compatible plugin command continuation result that can explicitly replace the same-turn user body, document it in the Plugin SDK and slash-command docs, and cover text and native command paths with focused regression tests.

## Reproduction Assessment

Not applicable: this is a feature request for a new public Plugin SDK behavior, not a bug with an existing expected result. Source inspection confirms the adjacent command registration behavior exists, while the requested pass-through result shape is absent on current main.

## Solution Assessment

No: the current workaround of message hooks plus manual slash-command parsing is not the best long-term solution for this API gap. The maintainable path is an additive typed command-result/body-rewrite contract that fits existing command authorization, continuation, and help/list integration.
