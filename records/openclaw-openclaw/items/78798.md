## Summary

Keep open: current main has plugin slash-command registration, argument parsing, help/list integration, and `continueAgent`, but it does not expose the requested typed pass-through result that rewrites the same-turn user message to the remaining command text.

## What This Changes

The issue requests a plugin API for registering slash commands that can run plugin logic, parse command arguments, optionally block handling, and pass the remaining message through to the LLM.

## Best Possible Solution

Add a backward-compatible plugin command continuation contract that can explicitly replace the same-turn user body, document it in the Plugin SDK and slash-command docs, and cover both text and native command paths with regression tests.

## Reproduction Assessment

Not applicable: this is a feature request for a new public plugin SDK behavior, not a bug with an existing expected result. Source inspection confirms the adjacent command registration behavior exists but the requested pass-through result shape does not.

## Solution Assessment

No: the current workaround of `message_received` or `before_prompt_build` plus manual parsing is not the best long-term solution for this API gap. The maintainable path is an additive command-result/body-rewrite contract that matches existing command authorization and help integration.
