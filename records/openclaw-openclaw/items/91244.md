## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Bind trusted-upstream to a source boundary:** `extensions/slack/src/monitor/provider.ts:166-171`
  - body: `createSlackTrustedUpstreamHttpHandler` treats the configured header as the only trust assertion before dispatching to Bolt. Because the schema defaults that value to `true` and the Slack webhook route is plugin-auth rather than gateway trusted-proxy auth, a directly reachable gateway or a proxy that preserves client headers can accept forged Slack envelopes; add loopback/trusted-proxy source validation or require an unguessable upstream assertion before dispatch.
  - confidence: 0.9
- **[P2] Add operator guardrails for the trust mode:** `src/config/zod-schema.providers-core.ts:951-963`
  - body: Adding `trusted-upstream` to the Slack schema exposes a security-sensitive transport, but this branch does not add Slack docs, doctor guidance, or security-audit coverage for risky trusted-upstream deployments. Operators would have no first-party warning comparable to gateway trusted-proxy before enabling this mode.
  - confidence: 0.86
