Full review comments:

- **[P1] Bind trusted-upstream to a source boundary before dispatch:** `extensions/slack/src/monitor/provider.ts:166-171`
  - body: `createSlackTrustedUpstreamHttpHandler` treats the configured header as the only trust assertion before dispatching to Bolt. Because the schema defaults that value to `true` and the Slack webhook route is plugin-auth rather than gateway-auth, a directly reachable gateway or a proxy that fails to strip client headers can accept forged Slack envelopes. Add loopback/trusted-proxy source validation or require an unguessable upstream assertion before dispatch.
  - confidence: 0.9
- **[P2] Add operator guardrails for the new trust mode:** `src/config/zod-schema.providers-core.ts:951-963`
  - body: This new mode changes Slack verification from OpenClaw-owned Slack signatures to an upstream assertion, but the branch does not update Slack docs, Slack security audit, or doctor guidance. Operators would have no first-party warning comparable to gateway trusted-proxy before enabling a risky deployment shape.
  - confidence: 0.84
