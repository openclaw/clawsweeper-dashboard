## Summary

Keep open: the parser change is the right direction for fresh OpenRouter catalog loads, but the PR misses cache invalidation, so upgraded users with an existing stale capability cache may keep the bad context budget.

## What This Changes

The PR updates OpenRouter dynamic model capability parsing to prefer endpoint-specific context length from `top_provider` and adds regression coverage for that catalog shape.

## Best Possible Solution

Merge the endpoint-specific context parsing with a cache invalidation path, such as bumping the OpenRouter disk cache version and covering the stale-cache upgrade case in a focused test.

## Reproduction Assessment

Yes. Source inspection plus the live OpenRouter catalog and linked logs reproduce the root mismatch: current cached parsing can combine a `1000000`-token top-level context window with a `262144`-token endpoint output cap for the reported model.

## Solution Assessment

No, not yet. The parser change is the right narrow fix for fresh catalog loads, but it needs cache invalidation so the affected upgrade path actually receives the new parsed `contextWindow`.
