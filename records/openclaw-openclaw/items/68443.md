## Summary

Keep open. The Feishu chat-history feature is useful, but the PR still has a chat-versus-thread routing bug, inconsistent pagination normalization, missing inspectable live Feishu proof, and an unresolved bulk-read/action-contract boundary.

## What This Changes

The PR adds a Feishu message-tool `list` action, a `listFeishuChatMessages` helper using `im.message.list`, shared action metadata, and Feishu action-surface test updates.

## Best Possible Solution

Land a Feishu-owned history path that is explicitly approved for bulk reads, resolves only chat IDs for chat history, keeps topic history on the thread helper, clamps pagination consistently, returns cursor metadata, documents scopes, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes for the review findings: PR-head source inspection and Feishu SDK/API contract evidence show a thread ID can flow into a chat container call, and raw limits are inconsistently normalized. I did not exercise a live Feishu tenant in this read-only review.

## Solution Assessment

No. The feature direction is plausible, but this branch should reject or correctly route thread IDs, normalize pagination once, settle `read` versus `list` and bulk-read exposure semantics, and provide real Feishu proof before merge.

- **[P1] Resolve only chat IDs for chat history:** `extensions/feishu/src/channel.ts:758-760`
