## Summary

Keep open: current main still lacks the proposed Feishu chat-history `list` action, but this PR should not merge as-is because source inspection shows a thread ID can be sent to the chat-history endpoint, pagination is not normalized before the Feishu call, and contributor-provided live Feishu proof is still missing.

## What This Changes

The PR adds a Feishu shared message `list` action, runtime helper, shared action-name and target-mode entries, and action-surface test updates for reading chat history by chat/container ID.

## Best Possible Solution

Land a Feishu-owned message-history action that uses the existing chat-target resolver, rejects or separately routes thread IDs, normalizes pagination before the API call, documents required scopes, adds focused tests, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR diff shows `ctx.params.threadId` can be sent to `container_id_type: "chat"`, and invalid numeric limits can flow into `page_size`. No live Feishu reproduction was run in this read-only review.

## Solution Assessment

No. The feature direction is plausible, but this implementation should first reuse the existing Feishu chat-target resolver, remove or route the thread-ID fallback, normalize pagination, and provide real Feishu proof.
