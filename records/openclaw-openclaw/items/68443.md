## Summary

Keep open. Current main still lacks the Feishu chat-history message-list capability, and this PR is a plausible implementation path, but the branch still has chat/thread ID routing and pagination correctness blockers plus missing inspectable live Feishu proof.

## What This Changes

This PR adds a Feishu message `list` action, runtime helper, shared action metadata, and action-surface tests to read chat history by chat/container ID.

## Best Possible Solution

Land a Feishu-owned chat-history action that normalizes chat targets, rejects or separately routes thread IDs, clamps pagination once, covers the handler/helper with focused tests, documents scopes and cursor behavior, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head shows raw chat targets or a thread ID can flow into a chat-history endpoint, and unnormalized limits flow into `page_size`. I did not exercise a live Feishu tenant in this read-only review.

## Solution Assessment

No. The feature direction is plausible, but this branch should use the existing chat ID resolver, reject or route thread IDs correctly, normalize pagination in one place, and provide real Feishu proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Resolve only chat IDs for chat history:** `extensions/feishu/src/channel.ts:758-760`
  - body: The new `list` branch reads `channelId`/`chatId`/`target` as raw strings and can fall back to `threadId`, but the helper always calls Feishu with `container_id_type: "chat"`. That sends prefixed chat targets unnormalized and can send topic thread IDs to the chat-history endpoint, so this should use the existing chat ID resolver and reject or route thread IDs to the thread path.
  - confidence: 0.94
- **[P2] Normalize page size before calling Feishu:** `extensions/feishu/src/send.ts:457-478`
  - body: `limit` is passed through as any number, `Math.min(limit, 50)` is sent as `page_size`, and the original value is reused in `.slice(0, limit)`. Clamp to a supported integer range once and use that value consistently so zero, negative, fractional, and over-50 limits do not produce inconsistent API calls or results.
  - confidence: 0.86
