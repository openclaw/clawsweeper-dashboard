## Summary

Keep open. Current main still lacks Feishu chat-history pagination, and this PR remains a plausible implementation path, but the live diff still has source-level routing and pagination blockers plus no inspectable real Feishu proof.

## What This Changes

The PR adds a Feishu channel message `list` action for reading chat history by chat/container ID, wires it through runtime/action metadata, and updates Feishu action-surface tests.

## Best Possible Solution

Land a Feishu-owned message-history action that uses the existing chat-target resolver, rejects or separately routes thread IDs, normalizes pagination once, adds focused tests/docs for scopes and cursor behavior, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the live PR diff shows a thread ID can be passed to a chat-history endpoint and unnormalized numeric limits can flow into `page_size`. No live Feishu tenant was exercised in this read-only review.

## Solution Assessment

No. The feature direction is plausible, but this branch bypasses existing Feishu chat-target normalization, mixes thread IDs with chat IDs, and needs pagination normalization plus real Feishu proof before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Resolve only chat IDs for chat history:** `extensions/feishu/src/channel.ts:756-757`
  - body: The new `list` action reads `channelId`/`chatId`/`target` directly and can fall back to `ctx.params.threadId`, but the helper calls Feishu with `container_id_type: "chat"`. Prefixed chat targets should go through the existing resolver, and Feishu thread IDs belong on the existing thread-list path instead of being sent to the chat-history endpoint.
  - confidence: 0.94
- **[P2] Normalize page size before the API call:** `extensions/feishu/src/send.ts:457-478`
  - body: `limit` is accepted as any number, then `Math.min(limit, 50)` is passed as `page_size` while `.slice(0, limit)` still uses the original value. Zero, negative, fractional, and over-50 values are not handled consistently; clamp to a supported integer range once and use that value everywhere.
  - confidence: 0.86
