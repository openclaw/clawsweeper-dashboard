## Summary

Keep open. Current main still lacks Feishu chat-history listing, but this PR is not ready to merge because source inspection confirms the thread-id/chat-id routing and pagination defects, and no inspectable live Feishu proof has been posted.

## What This Changes

The PR adds a Feishu channel message `list` action for reading chat history by chat/container ID, wires it through the Feishu runtime and shared action metadata, and updates Feishu action-surface tests.

## Best Possible Solution

Land a Feishu-owned chat-history read/list path that uses the existing chat-target resolver, rejects or separately routes thread IDs, normalizes pagination before the API call, adds focused handler tests and docs for required scopes, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR diff shows a thread ID can be passed into a chat-history endpoint and invalid numeric limits can flow into `page_size`. No live Feishu reproduction was run in this read-only review.

## Solution Assessment

No. The feature direction is useful, but this branch bypasses existing Feishu target normalization, mixes thread IDs with chat IDs, accepts invalid limit values, and lacks real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Resolve only real Feishu chat IDs:** `extensions/feishu/src/channel.ts:758-760`
  - body: The new `list` branch falls back to `ctx.params.threadId` and passes that value to `listFeishuChatMessages`, whose request uses `container_id_type: "chat"`. Thread IDs belong on the existing thread path, and prefixed group/channel targets should go through the existing chat-ID resolver before calling the chat-history endpoint.
  - confidence: 0.94
- **[P2] Clamp page size before calling Feishu:** `extensions/feishu/src/send.ts:457-478`
  - body: `limit` is accepted as any number and only upper-bounded with `Math.min(limit, 50)`, so `0`, negative values, and non-integers can be sent as `page_size` while `.slice(0, limit)` still uses the original value. Normalize to a supported integer range once and use that value consistently.
  - confidence: 0.86
