## Summary

Keep open. The Feishu chat-history feature is useful, but the branch is not merge-ready because it lacks inspectable live Feishu proof, can route a topic thread ID into the chat-history endpoint, handles pagination limits inconsistently, and expands bulk history access without a settled public/security contract.

## What This Changes

The branch adds a Feishu message-tool `list` action, a chat-history helper using `im.message.list`, shared message-action metadata, and Feishu action-surface test updates.

## Best Possible Solution

Land a Feishu-owned history path that is explicitly approved for bulk reads, resolves only chat IDs for chat history, keeps topic history on the thread helper, clamps pagination, returns cursor metadata, documents scopes, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes for the review findings: PR-head source inspection and Feishu API docs show a thread ID can flow into a chat container call, and raw limits flow into a documented 1-50 page-size API. I did not exercise a live Feishu tenant in this read-only review.

## Solution Assessment

No. The feature direction is plausible, but this branch should reject or correctly route thread IDs, normalize pagination in one place, settle `read` versus `list` and bulk-read exposure semantics, and provide real Feishu proof before merge.

Full review comments:

- **[P1] Resolve only chat IDs for chat history:** `extensions/feishu/src/channel.ts:758-760`
  - body: The new `list` branch falls back to `ctx.params.threadId`, but `listFeishuChatMessages` always calls Feishu with `container_id_type: "chat"`.   - confidence: 0.95

  - confidence: 0.9
