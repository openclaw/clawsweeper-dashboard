## Summary

Keep this PR open. Current main still lacks Feishu chat-history listing, but the branch has blocking, narrow correctness issues in ID routing and pagination before it should merge.

## What This Changes

The PR adds a Feishu shared message `list` action, a `listFeishuChatMessages` helper using `im.message.list`, action discovery tests, and shared message-action target metadata.

## Best Possible Solution

Land a Feishu-owned chat-history action that only sends chat IDs to the chat-history endpoint, rejects or routes thread IDs through the existing thread helper, clamps pagination inputs, and includes focused tests, docs/scope guidance, and a changelog entry.

## Reproduction Assessment

Yes. A static reproduction is enough: call the proposed `list` action with only `threadId`; the branch assigns that value to `chatId` and sends it to `im.message.list` with `container_id_type: "chat"`.

## Solution Assessment

No. The feature direction is reasonable, but this patch should not merge until ID-type routing and limit normalization are fixed and the user-facing release/docs/test coverage is completed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Stop treating thread IDs as chat IDs:** `extensions/feishu/src/channel.ts:756-757`
  - body: The new `list` branch falls back to `ctx.params.threadId` when no chat ID is supplied, but `listFeishuChatMessages` calls `im.message.list` with `container_id_type: "chat"`. Calls with only a thread ID will hit the wrong Feishu container type; remove this fallback or route thread IDs to the existing thread-list path.
  - confidence: 0.94
- **[P2] Clamp list page size before the API call:** `extensions/feishu/src/send.ts:454`
  - body: `limit` flows directly into `page_size: Math.min(limit, 50)`, so zero, negative, or fractional values can be sent to Feishu. Normalize a single positive integer page size before the request and use that same value for any local trimming.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/channels/plugins/message-action-names.ts:9`
  - body: This adds a user-facing Feishu message action, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires user-facing `feat` changes to include a changelog entry before merge.
  - confidence: 0.86
