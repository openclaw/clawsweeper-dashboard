## Summary

Keep open. Current main still lacks Feishu chat-history listing, but this PR is not merge-ready because source review shows chat/thread ID routing and limit-normalization bugs, missing inspectable real Feishu proof, and an unresolved product choice around `read` versus a new generic `list` action.

## What This Changes

The PR adds a Feishu message `list` action, chat-history helper, shared message-action metadata, and action-surface tests for reading messages by chat/container ID.

## Best Possible Solution

Land a Feishu-owned chat-history path that normalizes chat targets, rejects or separately routes thread IDs, clamps pagination once, returns cursor metadata, documents scopes and semantics, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head shows a thread ID can flow into a chat-history endpoint and raw limits can flow into `page_size`. I did not exercise a live Feishu tenant in this read-only review.

## Solution Assessment

No. The feature direction is plausible, but this branch should resolve only real chat IDs, reject or correctly route thread IDs, normalize pagination in one place, and provide real Feishu proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Resolve only chat IDs for chat history:** `extensions/feishu/src/channel.ts:758-760`
  - body: The new `list` branch accepts raw `channelId`/`chatId`/`target` and then falls back to `threadId`, but `listFeishuChatMessages` always calls Feishu with `container_id_type: "chat"`. That can send prefixed targets unnormalized and topic thread IDs to the chat-history endpoint; use the existing chat ID resolver and reject or route thread IDs to the thread path.
  - confidence: 0.94
- **[P2] Normalize page size before calling Feishu:** `extensions/feishu/src/send.ts:457-478`
  - body: `limit` can be zero, negative, fractional, or greater than 50; `Math.min(limit, 50)` goes straight to `page_size` while the original value is reused for `.slice(0, limit)`. Clamp once to a supported integer range and use that value consistently for the API call and result handling.
  - confidence: 0.86
