## Summary

Keep open: current main still contains the reported blanket successful-stream final-payload suppression path, and the focused test suite still asserts that an unsent final text payload is dropped after `didStream()` with no abort.

## What This Changes

Fix WhatsApp/block-streaming reply delivery so a partial streamed block does not suppress a distinct complete final assistant reply.

## Best Possible Solution

Replace blanket successful-stream final dropping with proof-based per-payload suppression, and add regression coverage for partial streamed text followed by a longer unsent completed final reply.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: `buildReplyPayloads` with block streaming enabled, `didStream() = true`, `isAborted() = false`, `hasSentPayload() = false`, and a final text payload returns no reply payloads. I did not run a live WhatsApp repro in this read-only review.

## Solution Assessment

Yes, the narrow maintainable fix is to suppress final payloads only when `hasSentPayload` or `directlySentBlockKeys` proves that same payload was already delivered, while preserving the existing media/error behavior.
