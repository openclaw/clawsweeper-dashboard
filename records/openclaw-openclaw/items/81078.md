## Summary

Keep open: current main still suppresses text-only final payloads after any successful block stream without proving that the final text was already sent, so the reported WhatsApp-visible truncation remains actionable.

## What This Changes

The issue reports that WhatsApp block streaming can drop the completed final assistant reply after delivering only a partial streamed block.

## Best Possible Solution

Replace the blanket successful-stream final drop with proof-based per-payload suppression using `hasSentPayload` and `directlySentBlockKeys`, while keeping the existing media and error safeguards.

## Reproduction Assessment

Yes at source level: call `buildReplyPayloads` with block streaming enabled, `didStream()=true`, `isAborted()=false`, `hasSentPayload()=false`, and a text final payload, and current main returns no reply payloads. I did not run a live WhatsApp reproduction in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is to use the existing sent-payload proof API for successful-stream finals instead of adding a new config or channel-specific policy.
