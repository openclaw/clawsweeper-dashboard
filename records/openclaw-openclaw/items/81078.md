- `impact:message-loss`: The central failure mode is silent suppression of the final complete outbound message after partial block delivery.

## Summary

Keep open: current main and the latest release still contain the broad successful-stream final suppression path, and the focused test still expects text-only final payloads to be dropped even when the block pipeline says that exact payload was not sent. A prior focused fix PR is useful evidence for the repair shape, but it is closed unmerged and no open PR currently owns the fix.

## What This Changes

WhatsApp block streaming can omit the completed final reply after partial block delivery because final payload suppression is too broad.

## Best Possible Solution

Change buildReplyPayloads() so successful block streaming suppresses only payloads proven already delivered by blockReplyPipeline.hasSentPayload() or direct block-send keys, while preserving the existing media-only follow-up behavior.

## Reproduction Assessment

Yes from source inspection: current main still drops text-only final payloads whenever block streaming did stream and did not abort, and the adjacent test codifies that behavior. I did not execute the test shard because the review contract is read-only.

## Solution Assessment

No, the current blanket final-payload drop is not the best solution. The repair should dedupe against proven sent payloads rather than treating any successful partial stream as equivalent to the completed final text.
