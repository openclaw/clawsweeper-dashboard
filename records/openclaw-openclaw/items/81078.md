## Summary

Keep open: current main and the latest release still drop text-only final payloads after a successful block stream, and the focused remaining work is already represented by https://github.com/openclaw/openclaw/pull/93079.

## What This Changes

The issue requests that WhatsApp block streaming deliver the complete final reply after partial streamed content unless the exact final payload was already sent.

## Best Possible Solution

Land a focused generic reply-payload fix that only drops final text when the block pipeline or direct block-send evidence proves that exact final content was delivered, with regression coverage for text-only finals and media follow-ups.

## Reproduction Assessment

Yes. Source inspection shows a high-confidence path: a successful block stream with a text-only final payload enters shouldDropFinalPayloads and returns null without checking whether that final payload was sent.

## Solution Assessment

Yes. The narrow shared payload-builder repair is the best shape because WhatsApp reaches this generic block reply path, and the fix can preserve existing dedupe/media behavior without adding a WhatsApp-specific workaround.
