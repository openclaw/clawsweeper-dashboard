## Label Justifications

- `P1`: The issue reports silent truncation of user-visible WhatsApp replies in an active messaging workflow.
- `impact:message-loss`: The reported failure suppresses the complete final assistant reply after only partial streamed content reaches the channel.

## Summary

Keep open: current main and the latest release still suppress text-only final replies after a successful block stream, and the active fix path is the open linked PR at https://github.com/openclaw/openclaw/pull/93079.

## What This Changes

The issue requests that WhatsApp/block-streamed final replies be delivered unless the block reply pipeline proves the same final payload was already sent.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/93079, or an equivalent narrow guard, then close this issue once current main contains the fix.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a successful, non-aborted block stream with a text-only final reaches a branch that returns no final payload before consulting hasSentPayload, and the current test suite asserts that behavior.

## Solution Assessment

Yes. The best fix is a narrow shared buildReplyPayloads guard that drops text-only finals only when the block pipeline proves the final was already sent, rather than adding WhatsApp-specific policy or disabling block streaming.
