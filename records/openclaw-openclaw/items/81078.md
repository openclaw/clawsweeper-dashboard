## Summary

Keep open: current main still has the successful-stream final-payload suppression path, and the focused test codifies that an unsent text-only final payload is dropped. The open fix PR at https://github.com/openclaw/openclaw/pull/62157 targets the same root problem, but it is unmerged and prior review flagged a coalesced-stream duplication risk plus missing inspectable real behavior proof.

## What This Changes

The issue reports that WhatsApp block streaming can drop the completed final reply after a partial streamed block was delivered.

## Best Possible Solution

Fix `buildReplyPayloads` so genuinely unsent final text is delivered after partial block streaming while preserving proof-based suppression for already-streamed/coalesced text, direct-send keys, media-only follow-ups, and error payloads.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: call `buildReplyPayloads` with block streaming enabled, `didStream() === true`, `isAborted() === false`, `hasSentPayload() === false`, and a text-only final payload; current main returns no reply payloads. The checked-in focused test asserts that exact behavior.

## Solution Assessment

Yes in direction, with one important constraint: final suppression should become coverage-aware rather than only exact per-payload matching. The open PR at https://github.com/openclaw/openclaw/pull/62157 needs rework to preserve coalesced-stream dedupe before it is the best fix.
