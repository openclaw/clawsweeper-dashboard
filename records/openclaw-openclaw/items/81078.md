## Summary

Keep open: current main and the latest release still suppress text-only final payloads after successful block streaming, and the open fix PR at https://github.com/openclaw/openclaw/pull/93079 has not merged yet.

## What This Changes

Ensure WhatsApp/block-streamed replies deliver the complete final text payload unless pipeline or direct-send evidence proves that exact payload was already sent.

## Best Possible Solution

Review and land the focused generic buildReplyPayloads fix so text-only finals are suppressed only when the block pipeline or direct-send tracking proves the exact final payload was already delivered.

## Reproduction Assessment

Yes from source: a successful non-aborted block stream with hasSentPayload returning false reaches the text-only return null branch, and the focused test still expects the final payload to be dropped. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

Yes, the current requested direction is the best code shape: use the existing sent-payload/direct-send evidence instead of a WhatsApp-specific workaround or disabling block streaming. The open PR still needs merge-ready runtime proof before this issue can close.
