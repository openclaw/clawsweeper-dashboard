## Summary

Keep open: current main still has the successful-block-stream branch that drops text-only final payloads without proving the final text was sent. An existing open PR targets the same root cause, but that PR already has correctness and real-behavior-proof blockers, so this issue remains a useful canonical WhatsApp loss-of-final report.

## What This Changes

The issue reports incomplete WhatsApp final replies when block streaming has already delivered a partial text block and the completed final text is then suppressed.

## Best Possible Solution

Update `buildReplyPayloads` so successful block streams suppress finals only with delivery evidence, while preserving coalesced-stream duplicate suppression, media/error safeguards, and WhatsApp blockStreaming behavior.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: call `buildReplyPayloads` with block streaming enabled, `didStream()=true`, `isAborted()=false`, `hasSentPayload()=false`, and a text final payload, and the current branch returns no reply payloads. The focused test in `src/auto-reply/reply/agent-runner-payloads.test.ts` asserts that behavior.

## Solution Assessment

Yes on the issue direction, but not by landing the already-open PR as submitted. The narrow maintainable fix should use proof-based final suppression while explicitly preserving coalesced-stream, media, error, and direct-send dedupe behavior.
