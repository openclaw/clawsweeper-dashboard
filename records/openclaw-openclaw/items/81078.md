## Summary

Keep open: current main still suppresses unsent text-only final payloads after any successful block stream, and the existing focused test codifies that behavior. A related open PR targets the same bug, but it has a known correctness blocker and is not merged, so the issue should remain trackable.

## What This Changes

The issue requests that WhatsApp/block-streaming final replies be delivered unless the reply pipeline can prove the final payload was already sent.

## Best Possible Solution

Update buildReplyPayloads so successful block streams suppress finals only with proof that the same text/media was already delivered, including coverage-aware handling for coalesced streams and focused regression tests.

## Reproduction Assessment

Yes, at source level. A buildReplyPayloads call with blockStreamingEnabled=true, didStream()=true, isAborted()=false, hasSentPayload()=false, and a final text payload currently returns no reply payloads, as the checked-in test asserts.

## Solution Assessment

Yes for the requested direction, but not by blindly landing the existing open PR. The maintainable fix should be current-main-aware and preserve coalesced-stream, media, direct-send, and error safeguards while changing text final suppression.
