## Summary

Keep open: current main still suppresses text-only final payloads after any successful block stream even when the pipeline cannot prove the final text was already sent; a related open PR exists, but it is not merged and has a correctness/proof blocker.

## What This Changes

The issue reports that WhatsApp block streaming can drop a completed final reply after partial streamed text because final-payload suppression does not require proof that the same payload was already delivered.

## Best Possible Solution

Land a corrected proof-based final-payload filter in `buildReplyPayloads` that delivers genuinely unsent completed text while preserving coalesced-stream and media dedupe safeguards, with focused regression coverage.

## Reproduction Assessment

Yes, at source level: `buildReplyPayloads` with block streaming enabled, `didStream()=true`, `isAborted()=false`, `hasSentPayload()=false`, and a final text payload returns no reply payloads on current main. The checked-in focused test asserts that behavior.

## Solution Assessment

Yes for the reported direction: suppress finals only when `hasSentPayload` or direct block-send evidence proves delivery. The related open PR needs rework because a naive per-payload filter can reintroduce duplicates for coalesced streamed text.
