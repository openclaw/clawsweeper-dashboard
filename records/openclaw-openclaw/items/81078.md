## Summary

Keep open: current main still has the successful block-stream path that suppresses text-only final payloads without proving that the same final text was already delivered, and the focused tests still codify that loss-of-final behavior.

## What This Changes

The issue requests a fix so WhatsApp block streaming does not drop the complete final assistant reply after only a partial streamed block was delivered.

## Best Possible Solution

Change `buildReplyPayloads` to suppress successful block-stream final payloads per payload only when `hasSentPayload(payload)` or `directlySentBlockKeys` proves that content was already delivered, while keeping existing error, fallback-notice, media, and dedupe safeguards.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: call `buildReplyPayloads` with block streaming enabled, `didStream()=true`, `isAborted()=false`, `hasSentPayload()=false`, and a text final payload; current main returns no reply payloads.

## Solution Assessment

Yes. The narrowest maintainable fix is to replace the blanket successful-stream text drop with proof-based per-payload suppression using the existing block reply pipeline API and direct-send keys.
