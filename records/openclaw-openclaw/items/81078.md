## Summary

Keep open: current main and v2026.6.6 still suppress text-only final payloads after successful block streaming, and the active fix path is the open linked PR rather than cleanup-close.

## What This Changes

The issue asks OpenClaw to preserve complete WhatsApp final replies after partial block streaming unless the block reply pipeline proves that exact final payload was already delivered.

## Best Possible Solution

Review and land the focused linked PR once it satisfies the real-runtime proof gate for partial block streaming followed by complete final delivery and exact duplicate suppression.

## Reproduction Assessment

Yes from source: a successful stream with a text-only final payload reaches the `!reply.hasMedia` branch and returns no final payload without checking whether that final text was sent. I did not run a live WhatsApp repro in this read-only review.

## Solution Assessment

Yes, the best fix direction is narrow: use existing sent-payload evidence before dropping a text-only final, while preserving media-only and exact-duplicate safeguards. The open linked PR appears to own that path.
