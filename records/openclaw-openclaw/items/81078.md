## Summary

Keep open: current main still has the blanket successful-stream final-payload suppression path, and existing focused tests codify that an unsent final text payload is dropped when `didStream()` is true and the pipeline is not aborted.

## What This Changes

Fix block-streaming final-payload suppression so WhatsApp delivers a complete final reply after partial streamed text unless the pipeline can prove the final payload was already sent.

## Best Possible Solution

Change `buildReplyPayloads` to suppress final payloads only when `hasSentPayload` or `directlySentBlockKeys` proves the same payload was already delivered, with regression coverage for partial streamed text followed by a longer completed final reply.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: `buildReplyPayloads` with `blockStreamingEnabled=true`, `didStream()=true`, `isAborted()=false`, `hasSentPayload()=false`, and a final text payload returns no reply payloads on current main. The existing focused test asserts exactly that behavior.

## Solution Assessment

Yes, the reporter's direction is the narrowest maintainable fix: replace the blanket successful-stream drop with proof-based per-payload suppression while preserving the existing media/TTS safeguards.
