## Summary

Keep open: current main still has the successful-block-stream path that suppresses text-only final payloads without checking whether that final text was actually sent, and the focused test suite currently codifies that loss-of-final behavior.

## What This Changes

The issue asks OpenClaw to deliver the completed WhatsApp final reply after partial block streaming unless the same final payload was proven already sent.

## Best Possible Solution

Change `buildReplyPayloads` to apply proof-based per-payload final suppression for successful block streams, while keeping the existing media and error safeguards.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: call `buildReplyPayloads` with block streaming enabled, `didStream()=true`, `isAborted()=false`, `hasSentPayload()=false`, and a text final payload, and current main returns no reply payloads. The focused test in `src/auto-reply/reply/agent-runner-payloads.test.ts` asserts exactly that behavior.

## Solution Assessment

Yes, the narrowest maintainable fix is to replace the blanket successful-stream text drop with per-payload suppression based on `hasSentPayload` or `directlySentBlockKeys`. That matches the existing pipeline API and the streaming docs without adding a new config or product policy.
