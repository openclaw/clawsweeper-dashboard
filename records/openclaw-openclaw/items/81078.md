## What This Changes

The issue reports that WhatsApp block streaming can suppress the complete final assistant reply after a partial streamed block was delivered.

## Best Possible Solution

Change the shared final-payload filter so successful block streaming suppresses only payloads proven delivered by the block pipeline or direct block-send evidence, while preserving existing media-only follow-up behavior.

## Repair Work Prompt

Fix the WhatsApp-visible block-streaming final suppression bug in the shared auto-reply payload builder. Current `buildReplyPayloads()` drops text-only final payloads whenever `blockStreamingEnabled` is true, `blockReplyPipeline.didStream()` is true, and the pipeline is not aborted; change it so a final payload is skipped only when `blockReplyPipeline.hasSentPayload(payload)` or direct block-send evidence proves that same final content was already delivered. Preserve existing media-only follow-up behavior when streamed text was sent but final media was not, preserve error/fallback payload behavior, and do not add WhatsApp-specific policy or new config. Use the closed unmerged source PR https://github.com/openclaw/openclaw/pull/90569 only as reference; produce a fresh narrow implementation and regression coverage.
