## Summary

Keep open: current main still suppresses text-only final payloads after successful block streaming without proving the final text was sent. An open PR targets the same surface, but it is not merged and prior review found it incomplete, so this issue remains a valid tracker.

## What This Changes

The issue asks OpenClaw to deliver complete WhatsApp final replies after partial block streaming unless the reply pipeline can prove the same final content was already sent.

## Best Possible Solution

Replace the blanket successful-stream text drop with coalescing-safe, proof-based final suppression that only skips content already delivered by the block pipeline or direct block keys.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: call `buildReplyPayloads` with block streaming enabled, `didStream()=true`, `isAborted()=false`, `hasSentPayload()=false`, and a text-only final payload; current main returns no reply payloads and the focused test asserts that behavior.

## Solution Assessment

Yes for the issue direction: the narrow fix is proof-based final suppression rather than a new config or policy. The open PR needs a coalescing-safe repair before it is the best implementation.
