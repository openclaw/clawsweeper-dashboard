## Summary

Keep open. Current main still has the shared provider-resolution gap: a non-empty active media-understanding registry can return before configured/compat Deepgram is loaded, so the reported Discord voice STT failure remains actionable.

## What This Changes

The issue requests that Discord voice STT keep resolving configured Deepgram media providers across follow-up turns when the active media provider registry is only partially populated.

## Best Possible Solution

Extend the shared capability provider resolver to reconcile requested tools.media.audio.models providers for mediaUnderstandingProviders against a partial active registry by provider id, then cover active OpenAI/MiniMax plus configured Deepgram with a focused regression test. Keep the fix generic in the plugin runtime boundary, with no Discord- or Deepgram-specific special case.

## Reproduction Assessment

Yes. The issue gives concrete Discord voice plus Deepgram config and logs, and current main still has the static resolver path that can reproduce the missing-provider condition when active media providers exist but Deepgram is only available through compat/config-shaped loading.

## Solution Assessment

Yes, with a narrow implementation. The best fix is shared requested-provider reconciliation for media audio providers; an unconditional merge of all active and compat providers would be broader than needed and should preserve the active-registry fast path where safe.
