## Summary

Keep open. The PR targets a real source-reproducible read-image media bug that is still present on current main, but it is not merge-ready because it lacks real Feishu/Telegram proof and the proposed start-arg fallback can promote local paths before binding the behavior to the raw registered `read` tool.

## What This Changes

The PR adds start-argument fallback media extraction for embedded image tool results, threads tool start args into non-verbose tool-result handling, and adds extractor/handler regression tests.

## Best Possible Solution

Update this PR or a replacement against current main's media artifact and pending-media flow, scope arg-derived local paths to the raw registered `read` tool under existing trust gates, and attach redacted Feishu or Telegram proof.

## Reproduction Assessment

Yes. Source inspection on current main shows pathless image content still returns no media, while the handler has original start args available but does not pass them to media extraction; I did not run a live Feishu or Telegram repro in this read-only review.

## Solution Assessment

No, not as submitted. The fallback direction is right, but the current diff needs to be rebased onto the current artifact/pending-media flow and limited to the `read` tool contract before it is a safe fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope start-arg media fallback to read:** `src/agents/pi-embedded-subscribe.tools.ts:205-211`
  - body: The helper promotes any path-like start arg to outbound media whenever image content exists, before it knows which tool contract produced that result. Pass the raw tool name into this fallback and only accept arg paths for the trusted `read` tool; otherwise another trusted image-producing tool with an unrelated input path can queue the wrong local file.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff broadens local file media delivery from tool start args without binding that fallback to the `read` tool contract.

Concerns:

- **[medium] Tool-agnostic start args can become local media:** `src/agents/pi-embedded-subscribe.tools.ts:205`
  - body: `extractToolResultMediaPathsWithStartArgs` reads path-like args without knowing which tool produced the image result, so a trusted image-result tool with an unrelated input path could cause that local file to be queued as media. Limit the fallback to the raw registered `read` tool and preserve the existing trusted media gates.
  - confidence: 0.76
