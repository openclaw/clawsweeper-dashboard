## Summary

Keep open. The bug still exists on current main, but this PR is not merge-ready because the fallback is tool-agnostic, the branch targets an older direct media-emission shape, and after-fix real channel proof is still missing.

## What This Changes

The PR adds a start-argument media path fallback for embedded image tool results, threads tool start args into media extraction, and adds extractor/handler regression tests.

## Best Possible Solution

Adapt the fallback to current main by passing rawToolName and startArgs into media artifact extraction, accepting arg paths only for the trusted raw read tool, then queueing through the existing pending-media path with live redacted Feishu or Telegram proof.

## Reproduction Assessment

Yes, source-level. Current main drops a Pi-style read image result with image content but no MEDIA: token or details.path even though handleToolExecutionEnd still has the original read args.

## Solution Assessment

No, not as submitted. The fix direction is right, but the fallback should be scoped to the raw read tool and rebased into the current media artifact and pending-media queue path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope start-arg media fallback to read:** `src/agents/pi-embedded-subscribe.tools.ts:205-211`
