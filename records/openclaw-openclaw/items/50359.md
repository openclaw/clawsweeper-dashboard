## Summary

Keep open. The underlying read-image media bug is still source-reproducible on current main and this PR is paired with the canonical open issue at https://github.com/openclaw/openclaw/issues/41744, but the PR should not merge as submitted because it lacks real behavior proof and its start-args fallback is too broad for local media delivery.

## What This Changes

The PR adds start-argument fallback media extraction for embedded read-image tool results, threads start args into tool-result output, and adds extractor and handler regression tests.

## Best Possible Solution

Update the PR against current main's `ToolResultMediaArtifact` and pending-media flow, scope arg-derived local paths to the registered `read` tool with existing trust gates, and add redacted live Feishu or Telegram proof.

## Reproduction Assessment

Yes, source-level. Current main still drops the pathless Pi-style read image result shape because extraction has no start-args input, while the test suite preserves that behavior as an empty result.

## Solution Assessment

No, not as submitted. The fallback direction is right, but it needs raw-tool scoping to `read` and adaptation to the current media artifact and pending-media queue path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope start-arg media fallback to read:** `src/agents/pi-embedded-subscribe.tools.ts:205-211`
  - body: The helper promotes any path-like start arg to outbound media whenever image content exists, before it knows which tool contract produced that result. Pass the raw tool name into this fallback and only accept arg paths for the trusted `read` tool; otherwise another trusted image-producing tool with an unrelated input path can queue the wrong local file.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff broadens local file media delivery from tool start args without binding that fallback to the `read` tool contract.

Concerns:

- **[medium] Tool-agnostic start args can become local media:** `src/agents/pi-embedded-subscribe.tools.ts:205`
  - body: `extractToolResultMediaPathsWithStartArgs` reads path-like args without knowing which tool produced the image result, so a trusted image-result tool with an unrelated input path could cause that local file to be queued as media. Limit the fallback to the raw registered `read` tool and preserve the existing trusted media gates.
  - confidence: 0.76
