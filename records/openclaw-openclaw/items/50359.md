## Summary

## What This Changes

The PR adds fallback media extraction from original tool start args when embedded image results lack `details.path`, threads those args through tool-result handling, and adds extractor plus handler regression tests.

## Best Possible Solution

Port the fallback onto current `extractToolResultMediaArtifact` and the pending-media queue, scope start-arg fallback to exact trusted `read` results, keep local-media trust gates intact, and prove delivery with a redacted Feishu or Telegram run.

## Reproduction Assessment

Yes, source-level. Current main still returns no media for pathless read image content while retaining original tool args elsewhere in the handler, and the checked-in test asserts the empty-media behavior.

## Solution Assessment

No, not as currently shaped. Falling back to read start args is the narrow fix direction, but this PR needs a current-main port and a read-specific trust gate before it is the best mergeable solution.
