## Summary

Keep open. The PR targets a real remaining read-image media bug, but the branch is not merge-ready because the start-args fallback is tool-agnostic, stale against the current pending-media artifact path, and still lacks after-fix live behavior proof.

## What This Changes

The PR adds a start-arg path/file_path/filePath fallback for embedded tool-result media extraction, threads tool_execution_start args into the handler, and adds extractor/handler regression tests.

## Best Possible Solution

Rebase the fallback into current `extractToolResultMediaArtifact` and pending-media queueing, accept start-arg paths only for the trusted raw `read` contract, preserve existing local-media gates, and land with redacted live Feishu or Telegram proof.

## Reproduction Assessment

Yes, source-level. Current main returns no media artifact for image content without `MEDIA:` text or `details.path`, while `handleToolExecutionEnd` has the original args and the upstream read tool emits pathless image content.

## Solution Assessment

No, not as submitted. The fix direction is right, but it needs read-tool scoping and a rebase onto the current media artifact/pending queue path before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Scope start-arg media fallback to read:** `src/agents/pi-embedded-subscribe.tools.ts:205-211`
  - body: The new helper promotes any `path`, `file_path`, or `filePath` start arg to outbound media whenever image content exists, before the caller filters by trusted tool name. Thread the raw tool name into this fallback and only accept arg paths for the trusted `read` contract; otherwise another trusted image-producing tool with an unrelated input path could queue the wrong local file.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff broadens local file media delivery from tool start args without binding that fallback to the `read` tool contract.

Concerns:

- **[medium] Tool-agnostic start args can become local media:** `src/agents/pi-embedded-subscribe.tools.ts:205`
  - body: `extractToolResultMediaPathsWithStartArgs` reads path-like args without knowing which tool produced the result, so a trusted image-result tool with an unrelated input path could cause that local file to be queued as media. Limit the fallback to the raw `read` tool and preserve the existing trusted media gates.
