## Summary

Keep open. Current main still has the read-image media promotion gap this PR targets, but the branch is not mergeable as submitted because the start-args fallback is tool-agnostic and the contributor has not provided real after-fix Feishu or Telegram behavior proof.

## What This Changes

The PR adds start-argument fallback media extraction for embedded read-image tool results and regression tests for extractor and handler delivery.

## Best Possible Solution

Rebase onto current main, pass raw tool name plus start args into media artifact extraction, accept arg-derived local paths only for the registered `read` tool, and add redacted live Feishu or Telegram delivery proof.

## Reproduction Assessment

Yes, source-level. Current main still drops the Pi-style read image result shape with image content but no `MEDIA:` text or `details.path`, while the handler does not pass the available start args into extraction.

## Solution Assessment

No, not as submitted. The fallback direction is right, but it should be scoped to the raw registered `read` tool and adapted to the current media artifact and pending-media queue path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope start-arg media fallback to read:** `src/agents/pi-embedded-subscribe.tools.ts:205-211`
  - body: The helper promotes any path-like start arg to outbound media whenever image content exists, before it knows which tool contract produced that result. Pass the raw tool name into this fallback and only accept arg paths for the trusted `read` tool; otherwise another trusted image-producing tool with an unrelated input path can queue the wrong local file.
  - confidence: 0.86
