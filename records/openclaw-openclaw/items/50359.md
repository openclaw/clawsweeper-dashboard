## Summary

Keep open. The underlying read-image media bug is still present on current main, but this PR is not ready because it adds a tool-agnostic start-args path fallback and still lacks after-fix real channel proof.

## What This Changes

The PR threads original tool start args into embedded tool-result media extraction so read image results without `details.path` can populate outbound media, with extractor and handler regression tests.

## Best Possible Solution

Rebase the fallback into current `extractToolResultMediaArtifact` and pending-media queueing, accept start-arg paths only for the trusted raw `read` tool contract, and land after redacted Feishu or Telegram proof.

## Reproduction Assessment

Yes, source-level. Current main still returns no media for image content without `MEDIA:` text or `details.path`, while `handleToolExecutionEnd` has the original args but does not pass them into extraction.

## Solution Assessment

No, not as submitted. The fix direction is right, but it needs read-tool scoping and rebasing onto the current pending-media artifact path before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Scope start-arg media fallback to read:** `src/agents/pi-embedded-subscribe.tools.ts:205-211`
  - body: The new helper is called for every tool result, but it promotes any `path`, `file_path`, or `filePath` start arg to outbound media whenever image content exists. Thread the raw tool name into this fallback and only accept the arg path for the trusted `read` contract; otherwise a different trusted image-producing tool with an input path could send the wrong local file.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff broadens local file media delivery from model-provided start args without tying that fallback to the read-tool contract.

Concerns:

- **[medium] Tool-agnostic start args can become local media:** `src/agents/pi-embedded-subscribe.tools.ts:205`
  - body: `extractToolResultMediaPathsWithStartArgs` reads path-like args without knowing which tool produced the result, so a trusted image-result tool with an unrelated input path could cause that local file to be queued as media. Limit the fallback to the raw `read` tool and preserve the existing trusted media gates.
  - confidence: 0.72
