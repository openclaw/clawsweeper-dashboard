## Summary

Keep open. Current main still has the shared pathless read-image media promotion gap, and this PR targets the right bug, but the branch is dirty against current main’s refactored media artifact path and still lacks PR-head real behavior proof.

## What This Changes

The PR adds a start-args fallback for embedded image tool results without `details.path`, threads the original tool args into media emission, and adds extractor/handler regression tests.

## Best Possible Solution

Port the fallback onto current main by extending the shared artifact extraction/handler path to accept original read start args, preserve the existing trust filters and metadata, add regression tests for `path`, `file_path`, and `filePath`, then require redacted real Feishu or Telegram behavior proof before merge.

## Reproduction Assessment

Yes, source-level. Current main still has a path where a Pi read image result with image content but no `MEDIA:` text or `details.path` produces no queued media even though the original read args are available in the handler.

## Solution Assessment

Mostly yes, but not as the branch is currently shaped. The start-args fallback is the narrow shared fix, but it needs to be ported from the older `extractToolResultMediaPaths` flow into current main’s artifact-based pending-media flow.
