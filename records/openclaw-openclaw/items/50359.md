## Summary

Keep open: the reported bug is still visible from current-main source, but the submitted patch still has a concrete security/correctness blocker and lacks required real behavior proof.

## What This Changes

The PR adds start-argument fallback media extraction for embedded read-image tool results and tests extractor plus handler media delivery.

## Best Possible Solution

Rebase the fix onto current main, pass raw tool name plus start args into media artifact extraction, allow arg-derived local paths only for the registered `read` tool, and queue through the existing pending-media path with redacted live channel proof.

## Reproduction Assessment

Yes, source-level. Current main still drops a Pi-style read image result with image content but no `MEDIA:` token or `details.path`, even though the handler has the original start args available.

## Solution Assessment

No, not as submitted. The fallback direction is right, but it should be scoped to the raw registered `read` tool and adapted to current main's media artifact and pending-media queue path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope start-arg media fallback to read:** `src/agents/pi-embedded-subscribe.tools.ts:205-211`
  - body: The helper promotes any path-like start arg to outbound media whenever image content exists, before it knows which tool contract produced that result. Pass the raw tool name into this fallback and only accept arg paths for the trusted `read` tool; otherwise another trusted image-producing tool with an unrelated input path can queue the wrong local file.
  - confidence: 0.86

## Security Review

Status: needs_attention
