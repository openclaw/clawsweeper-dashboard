## Summary

Keep open. The linked bug remains source-reproducible on current main and the PR direction is useful, but the branch needs adaptation to the current pending-media implementation, tool-scoped fallback logic, and contributor-supplied real behavior proof before merge.

## What This Changes

The PR adds a start-args fallback to embedded tool-result media extraction for read image outputs and tests `path`, `file_path`, and `filePath` delivery paths.

## Best Possible Solution

Adapt the fallback into current `extractToolResultMediaArtifact` and pending-media queueing, scoped to trusted read tool args, with regression tests plus redacted live Feishu or Telegram proof.

## Reproduction Assessment

Yes, source-level. The pinned read tool can return image content without `details.path`, and current main still drops that shape because the extractor never receives `startData.args`.

## Solution Assessment

No, not as submitted. The fix direction is right, but it should be rebased onto the current media artifact/pending queue path and limited to the read-tool contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Scope start-arg media fallback to read:** `src/agents/pi-embedded-subscribe.tools.ts:205-211`
  - body: This helper is called for every tool result, but it treats any `path`, `file_path`, or `filePath` start arg as an output attachment whenever image content exists. Thread the tool name into the extractor and only use this fallback for the read tool, otherwise another trusted image-producing tool with an input path can send the wrong local file.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff broadens local file media delivery from model-provided start args without tying that fallback to the read tool contract.

Concerns:

- **[medium] Tool-agnostic start args can become local media:** `src/agents/pi-embedded-subscribe.tools.ts:205`
  - body: `extractToolResultMediaPathsWithStartArgs` reads `path`-like args without knowing which tool produced the result, so any trusted image-result tool with a stray or unrelated path arg could cause that local path to be sent as media. Limit this fallback to the read tool and keep current raw-tool-name trust gates.
  - confidence: 0.72
