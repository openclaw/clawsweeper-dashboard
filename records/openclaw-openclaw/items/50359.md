## Summary

Keep open. The underlying read-image media loss is still present on current main, and this PR targets the right surface, but the branch needs a scoped rebase, real Feishu or Telegram behavior proof, and a safer tool-scoped fallback before merge.

## What This Changes

The PR threads embedded tool start args into media extraction so read image results without `details.path` can populate outbound media, with extractor and handler regression tests.

## Best Possible Solution

Rebase onto the current media artifact and pending-media queue path, pass raw tool name plus start args into extraction, accept arg-derived local paths only for the registered `read` tool, and add redacted live channel proof.

## Reproduction Assessment

Yes, source-level. Current main still drops a Pi-style read image result with image content but no `MEDIA:` token or `details.path`, even though the handler has the original start args available.

## Solution Assessment

No, not as submitted. The fix direction is right, but the start-arg fallback should be scoped to the raw `read` tool and adapted to current main's media artifact and pending-media queue path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope start-arg media fallback to read:** `src/agents/pi-embedded-subscribe.tools.ts:205-211`
