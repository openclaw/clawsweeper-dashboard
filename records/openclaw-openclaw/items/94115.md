## Summary

Keep open: this PR targets a real nodes CLI UX gap, but it should not merge as-is because the new hint points users to `openclaw nodes status` even though current code and docs use `openclaw nodes pending` or `openclaw nodes list` to enumerate pending pairing requests. It also still needs after-fix real behavior proof from a patched CLI run.

## What This Changes

The PR adds a `runNodesCommand` error hint when a nodes CLI gateway error message contains `unknown requestId`.

## Best Possible Solution

Handle unknown node pairing request IDs in the nodes CLI by pointing to `openclaw nodes pending` or `openclaw nodes list`, or by displaying current pending IDs directly, without changing gateway behavior.

## Reproduction Assessment

Yes at source level: current main returns `unknown requestId` from `node.pair.approve` and the nodes CLI wrapper prints it without a specific hint. I did not run a live companion-app pairing flow in this read-only review.

## Solution Assessment

No, not as submitted: the wrapper is a reasonable small place for this mitigation, but the hint should use the canonical pending/list command or show pending IDs directly. The current wording points to a status view that is not the pending-request list.
