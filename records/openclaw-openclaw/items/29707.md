## Summary

Keep open: current main still forces direct Amazon Nova Bedrock model IDs onto the no-cache path, while AWS now documents Nova prompt caching and Nova-compatible `cachePoint` examples. The older Nova `cachePoint` crash fix is useful history, but it does not implement native Nova prompt-cache support. ([docs.aws.amazon.com](https://docs.aws.amazon.com/bedrock/latest/userguide/prompt-caching.html))

## What This Changes

Request native Amazon Nova prompt-cache support in the Amazon Bedrock provider by allowing `cacheRetention` for eligible Nova models and avoiding Claude-only cache metadata.

## Best Possible Solution

Add provider-owned Nova prompt-cache support in the Amazon Bedrock plugin, with Nova-specific eligibility, AWS-compatible `cachePoint` shaping, focused tests, public docs, and changelog coverage while keeping unsupported non-Anthropic Bedrock models on the no-cache path.

## Reproduction Assessment

Yes for the current unsupported behavior: source inspection shows direct Nova IDs lose explicit `cacheRetention` and then hit the provider no-cache wrapper. No live AWS request was run during this read-only review.

## Solution Assessment

Yes, the maintainable direction is a narrow Amazon Bedrock provider enhancement, not removing the entire non-Anthropic guard. The implementation should follow AWS Nova cache semantics and avoid Claude-only TTL behavior without model-specific proof.
