## Summary

Keep open: current main and the latest release still force direct Amazon Nova Bedrock model IDs through the no-cache path, while the request remains a narrow provider capability repair with clear source-level proof.

## What This Changes

Request native Amazon Nova prompt caching in the Bedrock provider by allowing cache retention for documented Nova models without leaking unsupported Anthropic-only cache metadata to other Bedrock families.

## Best Possible Solution

Add provider-owned Nova prompt-cache handling in the bundled Amazon Bedrock plugin, limited to documented cache-capable Nova IDs, while keeping unsupported Bedrock families on the no-cache path.

## Reproduction Assessment

Yes, at source level: direct Nova IDs go through the Bedrock no-cache wrapper, current cache-retention gates exclude them, and current tests assert cacheRetention: "none" for amazon.nova-micro-v1:0. I did not run a live Nova request.

## Solution Assessment

Yes, but only with a narrow provider-owned shape: recognize documented Nova cache-capable models and emit Bedrock/Nova-compatible cachePoint handling instead of globally removing the non-Anthropic no-cache guard.
