## Summary

Keep open: current main still treats direct Amazon Nova Bedrock model IDs as non-cache-eligible and routes them through the provider no-cache wrapper, while the current AWS contract supports Nova prompt caching. This is a focused Amazon Bedrock provider enhancement with clear source proof and no canonical merged fix yet.

## What This Changes

Request native Amazon Nova prompt-caching support in the Amazon Bedrock provider by removing the hard no-cache path and avoiding Claude-only cache metadata where unsupported.

## Best Possible Solution

Add provider-owned Nova prompt-cache support in the bundled Amazon Bedrock plugin, while keeping unsupported non-Anthropic Bedrock models on the no-cache path and documenting the exact Nova TTL behavior.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the unsupported behavior: direct Nova IDs are not cache-retention eligible and are wrapped with `cacheRetention: "none"`; no live AWS call was performed.

## Solution Assessment

Yes. The narrow maintainable direction is an Amazon Bedrock provider enhancement that recognizes Nova cache eligibility, preserves explicit retention only where valid, and injects Nova-compatible cache points without applying Claude-only TTL semantics.
