## Summary

Keep open: current main still drops explicit cache retention for direct Amazon Nova Bedrock model IDs and forces non-Anthropic Bedrock streams to `cacheRetention: "none"`. The earlier Nova rejection history in https://github.com/openclaw/openclaw/issues/20815 and https://github.com/openclaw/openclaw/pull/20866 explains why the guard exists, but AWS now documents Nova-compatible prompt caching, so focused provider work remains.

## What This Changes

Request native Amazon Nova prompt caching on Bedrock by allowing `cacheRetention` for Nova model IDs and using AWS-compatible cache-point handling instead of hard-forcing no-cache.

## Best Possible Solution

Add provider-owned Nova prompt-cache support in the Amazon Bedrock plugin, with explicit Nova eligibility, AWS-compatible cachePoint shaping, focused tests, public docs, and changelog coverage while leaving unsupported non-Anthropic Bedrock models on the no-cache path.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the unsupported behavior: direct Nova IDs are not cache-retention eligible and are wrapped with `cacheRetention: "none"`; no live AWS call was performed.

## Solution Assessment

Yes. The narrow maintainable direction is an Amazon Bedrock provider enhancement that follows the AWS Nova cache contract and avoids reusing Claude-only TTL semantics without model-specific proof.
