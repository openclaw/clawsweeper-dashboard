## Summary

Keep open. Current main still treats direct Amazon Nova Bedrock IDs as non-cache-eligible, forces their stream options to `cacheRetention: "none"`, and has tests/docs that preserve that behavior, while AWS now documents Nova prompt caching support for Bedrock Converse/ConverseStream. 

## What This Changes

Request native Amazon Nova prompt caching on Bedrock by allowing `cacheRetention` and using Nova-compatible Bedrock cache points instead of the current non-Anthropic no-cache path.

## Best Possible Solution

Ship a bundled Amazon Bedrock provider update that recognizes supported Nova IDs, preserves explicit cache settings where valid, injects only Bedrock/Nova-compatible cache checkpoints, and keeps unsupported Bedrock families on the no-cache path.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current main resolves explicit Nova `cacheRetention` to `undefined`, then the Amazon Bedrock plugin wraps direct Nova streams with `cacheRetention: "none"`.

## Solution Assessment

Yes. A focused Amazon Bedrock plugin plus shared retention-helper change is the narrowest maintainable path; the safer alternative to broad removal of the no-cache wrapper is a Nova-family allowlist with provider-specific cache-point shaping.
