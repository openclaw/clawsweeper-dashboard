## Summary

Keep open: current main still blocks direct Amazon Nova Bedrock prompt caching, and the remaining work is a narrow bundled-provider enhancement rather than a stale or already-implemented request.

## What This Changes

The issue requests native prompt caching support for Amazon Nova models on Amazon Bedrock by allowing cache retention and using Nova-compatible cache checkpoints.

## Best Possible Solution

Add provider-owned Nova prompt-cache support in the bundled Amazon Bedrock plugin, with Nova-safe cachePoint handling, explicit cacheRetention pass-through, focused tests, docs, and a changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows explicit cacheRetention for direct amazon.nova-* Bedrock IDs is dropped by resolveCacheRetention and the provider wrapper then forces non-Anthropic direct Bedrock models to cacheRetention: "none".

## Solution Assessment

Yes. The narrow maintainable direction is an Amazon Bedrock provider change that recognizes AWS-supported Nova IDs and injects only Nova-compatible cache checkpoints while keeping unsupported non-Anthropic Bedrock models on the no-cache path.

Fix native prompt caching support for Amazon Nova on the bundled Amazon Bedrock provider. Current main drops explicit cacheRetention for direct amazon.nova-* IDs and then forces non-Anthropic direct Bedrock models to cacheRetention: "none". Teach the provider/cache-retention path to recognize AWS-supported Nova model IDs, preserve explicit cacheRetention where valid, and add Nova-compatible cachePoint handling without reusing Claude-only ttl: "1h" semantics unless AWS documents it for that Nova model. Keep unsupported non-Anthropic Bedrock models on the no-cache path. Update focused tests, public prompt-caching/Bedrock docs, and CHANGELOG.md.
