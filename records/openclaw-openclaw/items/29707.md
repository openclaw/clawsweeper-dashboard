## Summary

Keep open: current main still excludes direct Amazon Nova Bedrock models from cache-retention eligibility and then forces them through the Bedrock no-cache wrapper, while AWS now documents explicit prompt caching for Nova. This remains a narrow provider enhancement with clear source-level proof and no merged canonical fix.

## What This Changes

The issue requests native Amazon Nova prompt caching support in the Amazon Bedrock provider instead of forcing Nova requests to `cacheRetention: "none"`.

## Best Possible Solution

Add provider-owned Nova prompt-cache support in the bundled Amazon Bedrock plugin, with explicit Nova model gating, Nova-safe `cachePoint` payload shaping, 5-minute TTL handling, tests, docs, and changelog while keeping unsupported non-Anthropic Bedrock models on the no-cache path.

## Reproduction Assessment

Yes, source-level. Current main drops direct Nova cache settings in the resolver and then routes direct Nova model IDs through the Bedrock wrapper that forces `cacheRetention: "none"`; no live AWS call was run.

## Solution Assessment

Yes. The best fix is a narrow Amazon Bedrock provider implementation for supported Nova model IDs and TTL semantics, not a broad cacheRetention passthrough for every non-Anthropic Bedrock model.
