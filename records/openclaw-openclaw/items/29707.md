## Summary

Keep open: current main and the latest release still force direct non-Anthropic Bedrock models, including Amazon Nova, to `cacheRetention: "none"`, while current AWS docs show Nova prompt caching support. No merged canonical fix owns the remaining Nova-specific capability work.

## What This Changes

Request native prompt caching for Amazon Nova on Bedrock by allowing cache retention and using Nova-compatible cache-point handling instead of the non-Anthropic no-cache path.

## Best Possible Solution

Add provider-owned Nova cache capability handling in the Amazon Bedrock plugin, with Nova-specific checkpoint placement and TTL rules while keeping unsupported Bedrock families on the no-cache path.

## Reproduction Assessment

Yes at source level: direct Nova IDs fail the current Bedrock cache-eligibility checks and are wrapped with `cacheRetention: "none"`. I did not run a live Nova request in this read-only review.

## Solution Assessment

No to globally removing the restriction or only passing cacheRetention through. The maintainable path is a model-family capability policy that emits Nova-compatible checkpoints and preserves the existing no-cache behavior for unsupported Bedrock models.
