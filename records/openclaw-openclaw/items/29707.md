## Summary

Keep open: current main still excludes direct Amazon Nova Bedrock model IDs from cache-retention eligibility, forces them through the Bedrock no-cache wrapper, and documents that non-Anthropic Bedrock models are runtime-disabled while AWS now documents Nova prompt caching support. ([docs.aws.amazon.com](https://docs.aws.amazon.com/bedrock/latest/userguide/prompt-caching.html))

## What This Changes

Request native prompt-cache support for Amazon Nova models on Bedrock instead of forcing all non-Anthropic Bedrock models to `cacheRetention: "none"`.

## Best Possible Solution

Add Nova-aware Bedrock prompt-cache handling in the Amazon Bedrock plugin and shared cache-retention helpers, limited to AWS-documented Nova IDs and compatible default-TTL cache points while keeping unsupported non-Anthropic Bedrock models on the no-cache path.

## Reproduction Assessment

Yes. Source inspection and existing tests show direct Nova IDs currently drop explicit `cacheRetention` and then get forced through the Bedrock no-cache wrapper; no live AWS call is needed to prove current-main behavior.

## Solution Assessment

Yes, with one refinement. A provider-owned Nova implementation is the narrowest maintainable path, but it should use Bedrock/Nova-compatible `cachePoint` handling and avoid Claude-only 1-hour TTL behavior unless AWS documents that support for Nova.
