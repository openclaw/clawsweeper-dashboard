## Summary

Keep open: current main still drops explicit `cacheRetention` for direct `amazon.nova-*` Bedrock IDs and then forces those models through `cacheRetention: "none"`. The prior discussion usefully explains why the old guard existed via https://github.com/openclaw/openclaw/issues/20815 and https://github.com/openclaw/openclaw/pull/20866, but current AWS documentation now describes Nova prompt caching and recommends explicit prompt caching for cost savings.

## What This Changes

Request native Amazon Nova prompt caching on the Amazon Bedrock provider by replacing the blanket non-Anthropic no-cache path with Nova-compatible cache handling.

## Best Possible Solution

Add provider-owned Nova prompt-cache support in the Amazon Bedrock plugin, with explicit Nova model eligibility, AWS-compatible checkpoint shaping, focused tests, public docs, and changelog coverage while keeping unsupported non-Anthropic Bedrock models on the no-cache path.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the unsupported behavior: direct Nova IDs are not cache-retention eligible and are wrapped with `cacheRetention: "none"`; no live AWS call was performed.

## Solution Assessment

Yes, the maintainable direction is a narrow Amazon Bedrock provider enhancement rather than a core policy change. The fix should follow the AWS Nova contract and avoid reusing Claude-only TTL semantics without model-specific proof.
