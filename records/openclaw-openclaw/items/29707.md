## Summary

Keep open: current main and the latest release still exclude direct Amazon Nova Bedrock model IDs from cache-retention eligibility and force them through the no-cache Bedrock wrapper, while AWS documents Nova prompt caching as supported. This remains a focused Amazon Bedrock provider enhancement, not cleanup to close.

## What This Changes

The issue requests native Amazon Nova prompt-caching support on Bedrock by allowing supported Nova model IDs to use cache retention without sending incompatible cache-point payloads.

## Best Possible Solution

Add provider-owned Nova prompt-cache support in the Amazon Bedrock plugin with AWS-documented model and TTL gating, Bedrock/Nova-compatible `cachePoint` handling, focused tests, docs, and changelog, while keeping unsupported non-Anthropic Bedrock models on the no-cache path.

## Reproduction Assessment

Yes, source-level: current main drops explicit `cacheRetention` for direct Nova in `resolveCacheRetention` and then forces direct Nova requests through `cacheRetention: "none"`. I did not run a live AWS Bedrock Nova request during this read-only review.

## Solution Assessment

Yes, the requested direction is the maintainable shape if implemented narrowly. The fix should add Nova-specific cache eligibility and Bedrock-compatible payload handling, not pass cache retention through for every non-Anthropic Bedrock model.
