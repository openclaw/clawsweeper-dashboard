## Summary

Keep open: current main and the latest release still keep direct Amazon Nova Bedrock model IDs off the cache-retention path and force them through the no-cache wrapper, while AWS documents Nova prompt caching support. This remains a focused Amazon Bedrock provider enhancement with a clear implementation and validation path.

## What This Changes

Request Amazon Bedrock Nova models to be recognized as prompt-cache eligible, pass `cacheRetention`, and use Nova-compatible `cachePoint` payloads instead of Anthropic-only handling.

## Best Possible Solution

Implement provider-owned Nova prompt caching in the Amazon Bedrock plugin with AWS-documented Nova eligibility and TTL gating, Bedrock/Nova-compatible `cachePoint` placement, unsupported-model no-cache preservation, focused tests, public docs, and live Bedrock proof when credentials are available.

## Reproduction Assessment

Yes, source-level: direct Nova model IDs are excluded by the cache-retention resolver, forced through the no-cache Bedrock wrapper, and covered by tests that assert this exclusion. I did not run a live AWS Nova request during this read-only review.

## Solution Assessment

Yes, the requested direction is the maintainable solution if implemented narrowly. The fix should add Nova-specific cache eligibility and Bedrock-compatible payload handling, not pass cache retention through for every non-Anthropic Bedrock model.
