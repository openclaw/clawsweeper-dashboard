## Summary

Keep open: current main still treats direct Amazon Nova Bedrock model IDs as non-cacheable, while current AWS Bedrock docs document prompt caching support for Nova and recommend explicit prompt caching for cost savings and more consistent performance. This is valid provider work, not conservative cleanup to close.

## What This Changes

The issue requests native prompt-caching support for Amazon Nova models through the Amazon Bedrock provider, including allowing cache retention and avoiding Claude-only cache metadata assumptions.

## Best Possible Solution

Add provider-owned Nova prompt-cache support in the Amazon Bedrock plugin and shared cache-retention helpers, limited to AWS-supported Nova IDs, with Nova-safe checkpoint injection plus tests, public docs, and changelog coverage.

## Reproduction Assessment

Yes. Source inspection shows direct Nova IDs drop explicit cacheRetention and then go through the Bedrock no-cache wrapper; current tests and docs encode that same disabled behavior.

## Solution Assessment

Yes. A narrow Amazon Bedrock provider enhancement is the right direction, but the fix should add Nova-specific eligibility and payload handling instead of globally removing the non-Anthropic guard or relying only on pi-ai pass-through.
