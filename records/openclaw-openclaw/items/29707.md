## Summary

Keep open. Current main still deliberately treats direct Amazon Nova Bedrock model IDs as non-Anthropic: explicit `cacheRetention` is dropped and the Amazon Bedrock plugin forces the stream through `cacheRetention: "none"`, while official Bedrock docs now document Nova prompt caching through `cachePoint` on `system` and `messages`. ([docs.aws.amazon.com](https://docs.aws.amazon.com/bedrock/latest/userguide/prompt-caching.html?utm_source=openai))

## What This Changes

Request native prompt-cache support for Amazon Nova models through the bundled Amazon Bedrock provider.

## Best Possible Solution

Add provider-owned Nova prompt-cache support in the bundled Amazon Bedrock plugin using the existing `cacheRetention` knob and Bedrock-compatible `cachePoint` payload shaping, while keeping unsupported Bedrock families on the no-cache path.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high confidence that current main still intentionally blocks the requested Nova cache path through the retention gate, provider wrapper, tests, and docs.

## Solution Assessment

Yes. A focused Amazon Bedrock plugin/runtime-cache update is the narrowest maintainable direction because the public config knob already exists and no new core extension API is needed.
