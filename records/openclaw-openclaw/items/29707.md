## Summary

Keep open: current main and the latest shipped release still exclude direct Amazon Nova Bedrock model IDs from cache-retention eligibility and force them through `cacheRetention: "none"`. The earlier linked Nova `cachePoint` failure in https://github.com/openclaw/openclaw/issues/20815 and workaround PR https://github.com/openclaw/openclaw/pull/20866 explain why the guard exists, but this feature request now needs a provider-owned Nova prompt-caching implementation rather than cleanup closure.

## What This Changes

Request native Amazon Nova prompt-caching support in the Amazon Bedrock provider by preserving `cacheRetention` and using Nova-compatible Bedrock cache-point shaping.

## Best Possible Solution

Implement Nova prompt-cache support inside the Amazon Bedrock plugin and shared cache-retention eligibility helpers, with model-specific AWS-compatible cache-point shaping, tests, docs, and changelog coverage while keeping unsupported non-Nova non-Anthropic models on the no-cache path.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: direct `amazon.nova-*` Bedrock IDs are not cache-retention eligible and are wrapped with `cacheRetention: "none"`; no live AWS call was performed.

## Solution Assessment

Yes. The narrow maintainable fix is provider-owned Nova prompt-cache support in the Amazon Bedrock plugin and cache-retention helpers, not a broad core cache-policy change or a blind reuse of Claude-only TTL behavior.
