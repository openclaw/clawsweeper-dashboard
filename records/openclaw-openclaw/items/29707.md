## Summary

Keep this issue open. Current main still blocks direct Amazon Nova Bedrock models from receiving prompt-cache retention and then forces those direct non-Anthropic Bedrock calls through the no-cache wrapper, while current AWS Bedrock docs describe Nova prompt caching and recommend opting into explicit prompt caching for cost and performance benefits.

## What This Changes

Request support for native Amazon Nova prompt caching on Bedrock by allowing Nova cache retention and avoiding incompatible cache metadata behavior.

## Best Possible Solution

Add provider-owned Nova prompt-cache support in the bundled Amazon Bedrock plugin and shared cache-retention helpers: recognize supported direct, region-prefixed, global, and resolvable application-profile Nova model IDs; preserve explicit cacheRetention: "none"; inject only Bedrock/Nova-compatible cachePoint checkpoints in fields AWS documents as supported; avoid Claude-only 1-hour TTL for Nova unless AWS documents it; and update tests, docs, and changelog together.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: resolveCacheRetention({ cacheRetention: "long" }, "amazon-bedrock", "openai-completions", "amazon.nova-micro-v1:0") returns undefined, and the Amazon Bedrock wrapper then forces direct Nova streams to cacheRetention: "none".

## Solution Assessment

Yes, with one implementation constraint. A focused Amazon Bedrock provider repair is the narrowest maintainable solution, but it should treat Nova cachePoint fields and TTL semantics as Bedrock/Nova-specific rather than reusing Claude-only long-TTL behavior wholesale.
