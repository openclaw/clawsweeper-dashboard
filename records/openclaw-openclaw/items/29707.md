## Summary

Keep open: current main and v2026.5.19 still exclude direct Amazon Nova Bedrock model IDs from cache-retention eligibility and force them through the Bedrock no-cache wrapper, while current AWS Bedrock docs describe Nova prompt caching and Nova-compatible cachePoint usage. ([docs.aws.amazon.com](https://docs.aws.amazon.com/bedrock/latest/userguide/prompt-caching.html))

## What This Changes

Request native Amazon Nova prompt-caching support on Bedrock by allowing supported Nova model IDs to use cache retention and Nova-compatible cache points instead of the current no-cache path.

## Best Possible Solution

Add provider-owned Nova prompt-cache support in the bundled Amazon Bedrock plugin: recognize supported direct, region-prefixed, and global Nova IDs; preserve explicit retention only where AWS supports it; inject only Bedrock/Nova-compatible cachePoint checkpoints; keep unsupported non-Anthropic Bedrock models on the no-cache path; and update tests, public docs, and changelog together.

## Reproduction Assessment

Yes, source-level. Current main drops explicit cacheRetention for direct Nova Bedrock IDs and then wraps those requests with cacheRetention: "none"; no live AWS reproduction was run.

## Solution Assessment

Yes. The maintainable path is a Bedrock-provider-owned Nova cache implementation, not a broad resolver passthrough or reuse of Claude-only TTL semantics.
