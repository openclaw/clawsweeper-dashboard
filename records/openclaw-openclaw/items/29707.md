## Summary

Keep open. Current main still blocks direct Amazon Nova Bedrock prompt caching despite AWS Bedrock documentation supporting Nova prompt caching, and the remaining repair is narrow enough for a provider-owned fix.

## What This Changes

Request Amazon Bedrock Nova models to be recognized as prompt-cache eligible, pass `cacheRetention` through, and use Nova-compatible cache-point handling instead of disabling caching.

## Best Possible Solution

Add Nova-specific prompt-cache eligibility and Bedrock `cachePoint` injection/pass-through in the bundled Amazon Bedrock plugin and shared retention helpers, with tests, public docs, and changelog updated together.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `resolveCacheRetention({ cacheRetention: "long" }, "amazon-bedrock", "openai-completions", "amazon.nova-micro-v1:0")` resolves to `undefined`, and direct Nova streams then receive `cacheRetention: "none"` from the wrapper.

## Solution Assessment

Yes. A focused Amazon Bedrock provider repair is the narrowest maintainable solution; it should add Nova-compatible `cachePoint` handling rather than blindly reusing Claude-only cache semantics or only removing the no-cache wrapper.

## Repair Work Prompt

Fix OpenClaw issue https://github.com/openclaw/openclaw/issues/29707. Current main drops `cacheRetention` for direct Amazon Nova Bedrock model IDs and then forces direct Nova streams through `createBedrockNoCacheWrapper`, even though AWS Bedrock supports Nova prompt caching. Implement provider-owned Nova prompt-cache support in the bundled `amazon-bedrock` plugin and shared cache-retention helpers. Recognize supported Nova model IDs including `amazon.nova-2-lite-v1:0`, `amazon.nova-pro-v1:0`, `amazon.nova-lite-v1:0`, `amazon.nova-micro-v1:0`, and region/global-prefixed variants. Preserve explicit valid `cacheRetention` for those Nova IDs, inject only Bedrock/Nova-compatible `cachePoint: { type: "default" }` checkpoints in supported `system` and `messages` fields, and do not inject Anthropic `cache_control`. Do not map Nova `cacheRetention: "long"` to `ttl: "1h"` unless current AWS docs or a live Bedrock proof confirms 1-hour TTL for that Nova model; default/no-ttl 5-minute behavior is safer. Keep unsupported non-Anthropic Bedrock families such as Mistral on the no-cache path. Update colocated tests, `docs/reference/prompt-caching.md`, `docs/providers/bedrock.md` if needed, and `CHANGELOG.md` with a user-facing fix entry. Do not introduce bundled provider IDs into core beyond generic helper seams already used for provider cache semantics.
