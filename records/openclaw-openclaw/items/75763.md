## Summary

Keep open. Current main still preserves the stricter #47487 tool-profile behavior, but the reported silent cron UX remains source-backed: `config validate` stays schema-only, doctor preview warnings do not flag ignored `tools.exec`/`tools.fs` sections, and cron only promotes a narrow denial-token set to fatal status.

## What This Changes

The issue asks OpenClaw to make `messaging` profile tool-section mismatches loud for cron users instead of recording tool-unavailable output as a successful delivered run.

## Best Possible Solution

Preserve strict profile authorization, then add explicit config/doctor diagnostics for ignored tool sections and cron failure classification for tool-unavailable final output.

## Reproduction Assessment

Yes. The source-level path is high confidence: `tools.profile: "messaging"` plus configured `tools.exec` leaves `profileAlsoAllow` undefined, and reported final assistant text outside the current denial tokens can flow to cron as non-fatal delivered output.

## Solution Assessment

Yes. The narrow maintainable fix is diagnostics and run-status classification, not reverting the #47487 removal of implicit tool grants.

## Repair Work Prompt

Fix the silent cron/tool-profile regression reported in https://github.com/openclaw/openclaw/issues/75763. Preserve the #47487 security model: do not restore implicit `exec`, `process`, `read`, `write`, or `edit` grants from configured `tools.exec`/`tools.fs` sections. Add a loud diagnostic path for configs where a restrictive profile such as `messaging` has configured tool sections that are not granted by the active profile or explicit `alsoAllow`; cover `openclaw doctor` and, if appropriate, `openclaw config validate` output/JSON without breaking schema-only invalid-config semantics unexpectedly. Also make isolated cron runs classify tool-unavailable final output caused by missing `exec`/`process` access as a non-ok run status, preferably using structured failure metadata where available and tight text fallback patterns where not. Avoid broad natural-language false positives. Update focused tests and add a user-facing changelog entry because cron/config UX changes are user-visible.
