## Summary

Keep open. Current main intentionally preserves the #47487 security change, but the reported silent cron UX gap still appears valid: `messaging` does not grant `exec`/`process`, configured `tools.exec` only emits a runtime warning, doctor/config validation do not flag the mismatch, and cron only marks narrated failures as errors when they match existing denial tokens.

## What This Changes

The issue asks OpenClaw to surface restrictive tool-profile mismatches for isolated cron jobs instead of reporting a successful run after the agent delivers an exec/process-unavailable message.

## Best Possible Solution

Preserve the stricter #47487 tool-policy behavior, but add a loud diagnostic path for restrictive profiles with configured tool sections and missing `alsoAllow`, plus cron classification for unavailable-tool final output so affected runs become visible errors instead of successful deliveries.

## Reproduction Assessment

Yes. Source inspection reproduces the core condition: `tools.profile: "messaging"` excludes `exec`/`process`, `tools.exec` only produces a runtime warning, and cron marks outcomes `ok` unless the final payload matches the current fatal-error classifier.

## Solution Assessment

No, the current shipped behavior is not the best end state for this issue. The security fix should stay, but the silent UX should be fixed with a targeted doctor/config diagnostic and/or cron fatal classification rather than by restoring implicit tool grants.
