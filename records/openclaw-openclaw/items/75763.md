## Summary

Keep open. Current main still preserves the stricter #47487 tool-profile behavior, but the reported silent cron UX gap remains: the `messaging` profile does not include `exec`/`process`, configured `tools.exec` only produces a runtime warning unless `alsoAllow` is explicit, and the isolated-cron fatal classifier does not cover the reporter's quoted tool-unavailable output strings.

## What This Changes

The issue requests non-silent handling for `tools.profile: "messaging"` plus configured exec/fs sections after the 2026.4.29 tool-policy tightening, because isolated cron jobs can deliver tool-denial text as successful output.

## Best Possible Solution

Preserve the stricter tool authorization model, add a visible diagnostic for configured exec/fs sections hidden by restrictive profiles, and extend isolated-cron denial classification so tool-unavailable final output is recorded as an error or warning instead of a green delivery.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `tools.profile: "messaging"` plus `tools.exec` leaves `profileAlsoAllow` unset, and final assistant text that lacks the current denial tokens can flow through isolated cron as `status: "ok"`.

## Solution Assessment

Yes for the diagnostic/classification direction, and no for restoring implicit profile widening. The narrow maintainable fix is to surface the mismatch and classify the affected cron output while keeping #47487's stricter authorization intact.
