## Summary

Keep open. Current main still preserves the stricter #47487 authorization change but does not make the reported cron/config failure mode loud enough: validation remains schema-only, doctor preview has no ignored exec/fs section warning, and cron status still depends on a narrow denial-text classifier that misses the reported unavailable-tool wording.

## What This Changes

The issue asks OpenClaw to surface restrictive `messaging` tool-profile configs where configured `tools.exec` or `tools.fs` sections are ignored without matching `alsoAllow`, especially in isolated cron runs.

## Best Possible Solution

Preserve strict profile authorization, then add shared config/doctor diagnostics and narrow cron fatal classification for specific unavailable-tool outputs.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `tools.profile: "messaging"` plus `tools.exec` leaves `profileAlsoAllow` undefined, `config validate` still reports schema validity, and cron only fails the run when structured errors or known denial tokens are present.

## Solution Assessment

Yes. The maintainable fix is louder diagnostics and targeted cron status classification, not restoring implicit `exec`/`process` grants removed for #47487.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75763 by making the `tools.profile: "messaging"` plus configured `tools.exec`/`tools.fs` mismatch non-silent. Preserve the #47487 security model: do not restore implicit grants and do not add `exec`, `process`, `read`, `write`, or `edit` to restrictive profiles automatically. Add a shared diagnostic for configured exec/fs sections that are not granted by the active profile plus explicit `alsoAllow`, wire it into `openclaw doctor` preview and `openclaw config validate` output/JSON as appropriate, and add narrowly scoped isolated-cron fatal classification for specific unavailable-tool text such as model-visible `exec`/`process` not available messages. Avoid broad `not runnable` matching unless paired with clear tool-policy context. Add focused tests and a user-facing changelog entry.
