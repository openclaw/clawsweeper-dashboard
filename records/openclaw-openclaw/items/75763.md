## Summary

Keep open. Current main still implements and documents the stricter 2026.4.29 tool-profile behavior from #47487, but the reported silent cron UX gap remains: validation and doctor do not surface this mismatch, and cron can still record ok delivery for the reported tool-unavailable text.

## What This Changes

Make the 2026.4.29 `messaging` tool-profile migration non-silent when configured `tools.exec`/`tools.fs` sections are ignored without explicit `alsoAllow`.

## Best Possible Solution

Preserve strict profile authorization, then add loud config/doctor diagnostics for ignored tool sections and cron failure classification for tool-unavailable final output.

## Reproduction Assessment

Yes. The source-level path is high confidence: `tools.profile: "messaging"` plus `tools.exec` leaves `profileAlsoAllow` undefined, and reported final assistant text outside the current denial tokens can flow to cron as non-fatal output.

## Solution Assessment

Yes. The maintainable fix is diagnostics and run-status classification, not restoring implicit grants removed for #47487.
