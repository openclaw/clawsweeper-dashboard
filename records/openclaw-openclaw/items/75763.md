## Summary

Keep open. Current main still preserves the stricter #47487 tool-policy behavior, but the reported silent UX gap remains: `config validate` does not surface the mismatch, doctor preview does not diagnose configured `tools.exec`/`tools.fs` under restrictive profiles, and cron only marks a narrow set of denial strings fatal.

## What This Changes

The issue reports that restrictive `messaging` tool profiles silently omit configured `exec`/`process` tools for isolated cron runs while cron still reports successful delivery.

## Best Possible Solution

Preserve strict profile authorization, then add shared diagnostics for configured-but-ungranted tool sections and targeted cron fatal classification for unavailable-tool outputs.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `tools.profile: "messaging"` plus `tools.exec` still leaves `profileAlsoAllow` undefined, validation stays schema-only, and the cron classifier does not match the reported unavailable-tool wording unless it arrives through an existing structured error path.

## Solution Assessment

Yes. The maintainable fix is louder diagnostics and narrow cron status classification, not restoring the implicit `exec`/`process` grants removed for #47487.

## Repair Work Prompt

Fix openclaw/openclaw#75763 without restoring implicit tool grants. Preserve the #47487 security behavior: `tools.exec` and `tools.fs` under `messaging` or `minimal` must not add `exec`, `process`, `read`, `write`, or `edit` unless explicit `alsoAllow` grants them. Add a shared diagnostic for restrictive profiles with configured tool sections whose implied tools are not allowed, surface it in `openclaw doctor` preview and `openclaw config validate` output/JSON, and add focused tests. Also extend isolated cron fatal classification only for specific unavailable-tool/profile-policy outputs matching the reported cases, so cron runs that deliver "this session doesn't have exec available" or equivalent unavailable-tool text record an error instead of `status: ok`. Keep classification narrow to avoid false positives in normal reports. Add a user-facing changelog fix entry.
