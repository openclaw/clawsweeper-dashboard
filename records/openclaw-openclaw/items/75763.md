## Summary

Keep open. Current main still preserves the stricter #47487 tool-policy behavior, but the silent cron UX gap remains: configured `tools.exec` under `tools.profile: "messaging"` only emits a runtime warning, config/doctor do not flag the mismatch, and cron reports `ok` unless final output matches the current fatal-denial classifier.

## What This Changes

The issue asks OpenClaw to surface restrictive tool-profile configs that drop expected `exec`/`process` access, especially for isolated cron runs, instead of reporting delivered tool-unavailable text as a successful run.

## Best Possible Solution

Preserve the stricter tool-profile authorization model, but add a loud diagnostic path for restrictive profiles with configured tool sections and missing `alsoAllow`, plus cron classification for tool-unavailable final output so affected runs show as visible errors instead of green deliveries.

## Reproduction Assessment

Yes. The source-level path is high confidence: `tools.profile: "messaging"` plus `tools.exec` leaves `profileAlsoAllow` unset, and a final assistant message that lacks the current denial tokens flows through cron as `status: "ok"`.

## Solution Assessment

Yes. The best fix is targeted diagnostics and cron error classification while preserving #47487; restoring implicit `exec`/`process` access would solve the symptom by undoing the intended security change.

## Repair Work Prompt

Preserve the #47487 security behavior: do not restore implicit `exec`, `process`, or fs grants from configured tool sections. Add a high-signal diagnostic for restrictive profiles (`messaging`/`minimal`) where `tools.exec` or `tools.fs` is configured but the active profile lacks matching explicit `alsoAllow`; cover global and per-agent tools.
