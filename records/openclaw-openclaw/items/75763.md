## Summary

Keep open. Current main still intentionally blocks implicit `exec`/`process` grants under `messaging`, but the reported silent cron UX gap remains because validation/doctor do not flag the mismatch and cron only fails when final output matches the current denial classifier.

## What This Changes

Make restrictive `messaging` profile tool-section mismatches and cron tool-denial outputs visible instead of reporting affected delivered cron runs as `ok`.

## Best Possible Solution

Preserve strict #47487 authorization while adding loud config/doctor diagnostics and cron failure classification for unavailable required tools so affected runs are not green deliveries.

## Reproduction Assessment

Yes. The source-level path is high confidence: `tools.profile: "messaging"` plus configured `tools.exec` leaves `profileAlsoAllow` unset, and final assistant text outside the current denial token list can flow through cron as non-fatal output.

## Solution Assessment

Yes. The narrow maintainable fix is diagnostics and run-status classification, not restoring implicit grants that #47487 intentionally removed.

## Repair Work Prompt

Fix the silent cron/tool-profile mismatch reported in https://github.com/openclaw/openclaw/issues/75763. Preserve #47487 strict authorization: do not restore implicit `exec`, `process`, `read`, `write`, or `edit` grants from configured `tools.exec`/`tools.fs` sections. Add focused diagnostics so `tools.profile: "messaging"` or another restrictive profile plus configured `tools.exec`/`tools.fs` without explicit `tools.alsoAllow` is surfaced by doctor and, if appropriate, config validation output. Also tighten isolated cron status classification so model-visible tool-unavailable final output such as `doesn't have exec available` or `not found or not runnable` is not recorded as a green delivered `ok` run. Prefer a shared helper for detecting filtered configured tool sections and targeted cron classifier coverage over broad text heuristics. Add a user-facing `CHANGELOG.md` fix entry. Do not change provider defaults, tool-profile membership, or the #47487 security semantics.
