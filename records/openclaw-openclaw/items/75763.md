## Summary

Keep open. Current main still preserves the stricter #47487 tool-profile behavior, but the reported silent cron UX gap remains valid: `messaging` does not include `exec`/`process`, `tools.exec` only drives a startup warning unless `alsoAllow` is explicit, doctor/config validation do not surface the mismatch, and cron only turns selected structured/error-token payloads into error status.

## What This Changes

The issue asks OpenClaw to stop silently reporting isolated cron runs as successful when a restrictive `messaging` tool profile plus configured `tools.exec` leaves `exec`/`process` unavailable.

## Best Possible Solution

Preserve the strict #47487 tool-profile enforcement, but add a visible diagnostic for restrictive profiles with configured `tools.exec`/`tools.fs` and missing explicit grants, plus cron error classification for unavailable-tool final output so affected runs are visible failures instead of green deliveries.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `tools.profile: "messaging"` plus `tools.exec: {}` resolves without `exec`/`process`, emits only a startup warning, and cron records `ok` unless the final output or metadata trips the current fatal classifier.

## Solution Assessment

No, the current behavior is not the best end state. The maintainable fix is not to restore implicit `exec` grants, but to surface the mismatch through doctor/config validation and/or structured cron failure classification.
