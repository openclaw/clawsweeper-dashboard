## Summary

Keep open. Current main intentionally preserves the stricter #47487 tool-profile behavior, but the reported silent cron UX gap remains: validation still accepts the config, doctor preview does not flag the mismatch, and the cron denial classifier misses the reported unavailable-tool wording.

## What This Changes

Report requests a non-silent failure path when `tools.profile: "messaging"` is combined with configured `tools.exec`/`tools.fs` but no explicit `alsoAllow` grants the needed tools.

## Best Possible Solution

Preserve strict profile authorization, then add shared config/doctor diagnostics and targeted cron fatal classification for unavailable-tool outputs from isolated runs.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `tools.profile: "messaging"` plus `tools.exec` leaves `profileAlsoAllow` undefined, `config validate` still reports validity, and cron only treats a narrow set of denial strings or structured failures as fatal.

## Solution Assessment

Yes. The maintainable fix is louder diagnostics and targeted cron status classification, not restoring implicit `exec`/`process` grants removed for #47487.
