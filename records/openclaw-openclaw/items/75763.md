## Summary

Keep open. Current main intentionally preserves the stricter #47487 tool-profile behavior, but the reported silent cron UX gap remains: validation still reports the config as valid, doctor has no warning for configured exec/fs sections that the active profile will not grant, and cron only turns final text into an error for a narrow denial phrase set.

## What This Changes

The issue asks OpenClaw to surface restrictive tool-profile/exec mismatches loudly so isolated cron runs do not look successful when exec/process are unavailable.

## Best Possible Solution

Preserve strict profile authorization while adding loud config/doctor diagnostics for restrictive profiles with ignored configured tool sections, plus narrowly scoped cron classification so unavailable exec/process runs are not green deliveries.

## Reproduction Assessment

Yes. The source-level path is high confidence: `tools.profile: "messaging"` plus configured `tools.exec` leaves `profileAlsoAllow` unset, and reported final assistant text outside the current denial token list can flow through cron as `status: "ok"`.

## Solution Assessment

Yes. The narrow maintainable direction is diagnostics and status classification, not restoring the implicit grants that #47487 intentionally removed.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75763. Preserve the #47487 behavior: configured `tools.exec`/`tools.fs` must not implicitly grant `exec`, `process`, `read`, `write`, or `edit` under restrictive profiles. Add loud diagnostics for configs where global or agent `tools.profile` is `messaging`/`minimal` (or otherwise restrictive) and configured `tools.exec`/`tools.fs` expects tools not granted by explicit `alsoAllow`; cover `openclaw doctor` and, if maintainable without breaking compatibility, `openclaw config validate` warning/error output. Extend isolated cron classification so unavailable exec/process/bash outcomes like `doesn't have exec available`, `tool "exec" ... isn't available`, or `not found or not runnable` do not record green `ok` deliveries; prefer structured failure metadata where possible and keep final-text fallback narrow to avoid false positives. Add focused tests, update user-facing docs/changelog if behavior changes, and do not change unrelated model allowlist behavior from #75753.
