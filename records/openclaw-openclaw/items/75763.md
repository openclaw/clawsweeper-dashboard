## Summary

Keep open. Current main preserves the stricter #47487 security behavior, but the reported silent cron UX gap remains: the config is only warned at runtime, `config validate` still accepts it as valid, doctor preview does not flag this mismatch, and cron only classifies a narrow set of denial strings as fatal.

## What This Changes

The issue asks OpenClaw to loudly surface restrictive `messaging` tool-profile configs where `tools.exec` or `tools.fs` sections are ignored without matching `alsoAllow`, especially for isolated cron runs.

## Best Possible Solution

Preserve strict profile authorization, then add shared config/doctor diagnostics and narrow cron fatal classification for specific unavailable-tool outputs.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `tools.profile: "messaging"` plus `tools.exec` leaves `profileAlsoAllow` undefined, `config validate` still reports schema validity, and cron only fails the run when structured errors or known denial tokens are present.

## Solution Assessment

Yes. The maintainable fix is louder diagnostics and targeted cron status classification, not restoring implicit `exec`/`process` grants removed for #47487.

Preserve the #47487 security model: do not restore implicit grants and do not add `exec`, `process`, `read`, `write`, or `edit` to restrictive profiles automatically. Add a shared diagnostic for configured exec/fs sections that are not granted by the active profile plus explicit `alsoAllow`, wire it into `openclaw doctor` preview and `openclaw config validate` output/JSON as appropriate, and add narrowly scoped isolated-cron fatal classification for specific unavailable-tool text such as model-visible `exec`/`process` not available messages.
