## Summary

Keep open. Current main still preserves the stricter #47487 tool-profile behavior, but the reported silent cron UX gap remains source-backed: the config only warns at runtime, `config validate` remains schema-only, doctor preview lacks this mismatch, and cron does not classify the reported unavailable-tool wording as fatal.

## What This Changes

The issue asks OpenClaw to stop silently treating isolated cron turns as successful when a `messaging` tool profile plus configured `tools.exec` leaves `exec`/`process` unavailable.

## Best Possible Solution

Preserve strict tool-profile authorization, then add shared config/doctor diagnostics and targeted cron fatal classification for unavailable-tool outputs.

## Reproduction Assessment

Yes, source-level. `tools.profile: "messaging"` plus `tools.exec` leaves `profileAlsoAllow` undefined, `config validate` still reports schema validity only, and the cron denial classifier misses the unavailable-tool wording shown by current main.

## Solution Assessment

Yes. The narrow maintainable fix is louder diagnostics plus targeted cron status classification, not restoring implicit `exec`/`process` grants removed for #47487.

## Repair Work Prompt

Fix the silent isolated-cron tool-policy failure from https://github.com/openclaw/openclaw/issues/75763. Preserve the #47487 security behavior: do not restore implicit `exec`/`process` or fs grants from configured tool sections. Add a shared diagnostic for configs where restrictive profiles such as `messaging` have configured `tools.exec` or `tools.fs` sections whose tools are not granted by the active profile/explicit `alsoAllow`; surface it through doctor preview and `openclaw config validate` in a user-visible way consistent with existing CLI JSON/text behavior. Extend cron isolated-agent fatal classification only for specific unavailable-tool outputs such as the current unknown-tool guard text (`I can't use the tool "exec" here because it isn't available`) and the reported `doesn't have exec available` wording, so blocked commands do not record `status: "ok"`. Add focused regression tests and update docs/changelog if user-facing CLI/runtime behavior changes. Do not broaden generic error matching enough to classify ordinary narrative failures as policy errors.
