## Summary

Keep open. Current main preserves the stricter profile behavior from #47487, but the reported silent cron UX gap still appears unfixed: validation/doctor do not flag this mismatch, and cron can still classify unrecognized tool-unavailable final text as a successful run.

## What This Changes

Request to make the 2026.4.29 tool-profile regression non-silent when isolated cron jobs use the `messaging` profile with configured exec/fs sections but no explicit `alsoAllow`.

## Best Possible Solution

Preserve strict profile authorization, then add loud config/doctor diagnostics for ignored tool sections and cron failure classification for tool-unavailable isolated-run output.

## Reproduction Assessment

Yes. The source-level reproduction path is high confidence: `tools.profile: "messaging"` plus `tools.exec` leaves `profileAlsoAllow` undefined, `config validate` only reports schema validity, and unrecognized final assistant text can flow through cron without setting `hasFatalErrorPayload`.

## Solution Assessment

Yes for the diagnostics/status direction, but not for restoring back-compat implicit grants. The narrow maintainable fix is to surface the mismatch in validation/doctor and classify the affected cron runs as warnings/errors while keeping explicit `alsoAllow` as the authorization mechanism.

## Repair Work Prompt

Fix openclaw/openclaw#75763 without reverting or weakening the #47487 security model. Preserve the rule that configured `tools.exec`/`tools.fs` do not grant tools under restrictive profiles; do not add implicit `exec`, `process`, `read`, `write`, or `edit` grants. Add user-visible diagnostics so `tools.profile: "messaging"` or `"minimal"` plus configured `tools.exec`/`tools.fs` without matching `tools.alsoAllow` is surfaced by `openclaw doctor` and, if appropriate for existing config validate semantics, by `openclaw config validate`/JSON output. Also make isolated cron runs fail or warn loudly when the final assistant-visible text clearly reports tool unavailability from the missing exec/process path, including the reported `doesn't have exec available` and `not found or not runnable` cases, preferably via structured failure metadata where possible rather than an overly broad text grep. Add focused regression tests for tool policy, doctor/config diagnostics, and cron outcome classification. Add a user-facing changelog fix entry; do not touch unrelated tool-profile behavior or plugin/channel surfaces.
