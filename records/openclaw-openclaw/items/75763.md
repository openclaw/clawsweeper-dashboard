## Summary

Keep open. Current main still preserves the stricter #47487 tool-profile behavior, but the reported silent cron UX gap remains source-backed: config validation is schema-only, doctor preview lacks this mismatch, and cron denial classification does not catch the unavailable-tool wording reported here.

## What This Changes

The issue asks OpenClaw to stop silently treating isolated cron turns as successful when a restrictive `messaging` tool profile drops configured `exec`/`process` access.

## Best Possible Solution

Preserve strict tool-profile authorization, add shared diagnostics for configured tool sections that restrictive profiles do not grant, and classify the relevant unavailable-tool cron output as a failed run.

## Reproduction Assessment

Yes, source-level. `tools.profile: "messaging"` plus `tools.exec` still leaves `exec`/`process` outside `profileAlsoAllow`, `config validate` stays schema-only, doctor lacks the diagnostic, and the cron denial tokens miss the reported unavailable-tool wording.

## Solution Assessment

Yes. The narrow maintainable repair is louder validation/doctor diagnostics plus targeted cron fatal classification, not restoring implicit `exec`/`process` grants removed for #47487.

## Repair Work Prompt

Fix issue https://github.com/openclaw/openclaw/issues/75763 on current main. Preserve the #47487 strict tool-profile behavior: do not implicitly grant `exec`, `process`, `read`, `write`, or `edit` from configured `tools.exec`/`tools.fs`. Add a shared diagnostic for configs where a restrictive profile (`messaging`/`minimal`) has configured `tools.exec` or `tools.fs` but lacks matching explicit `alsoAllow`, and surface it through `openclaw doctor` preview and `openclaw config validate` in a user-visible way. Add targeted isolated-cron classification for unavailable-tool outputs matching the reported `exec`/`process` missing-tool wording so runs do not remain `status: "ok"` when the delivered output is the tool-policy failure. Update focused tests and add a single-line `CHANGELOG.md` fix entry. Do not add a config option and do not loosen authorization.
