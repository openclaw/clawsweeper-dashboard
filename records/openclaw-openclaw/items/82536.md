## Summary

Keep open: the underlying reload overhead is credible, but this patch is not merge-ready because it reaches into private Pi loader internals, its rewritten test mock is likely broken by Vitest hoisting, and the PR still lacks real after-fix benchmark proof.

## What This Changes

The PR changes embedded Pi runs to await an async resource-loader helper that skips `reload()` and directly loads inline extension factories, plus updates the attempt, compact, and resource-loader test paths.

## Best Possible Solution

Use a public-contract embedded resource-loader fast path, or an upstream Pi API, that avoids package discovery while preserving settings reload semantics, inline extensions, source info, conflict handling, and real Windows/large-tree timing proof.

## Reproduction Assessment

No high-confidence real reproduction was run in this read-only review. Source inspection confirms current main still awaits the dependency `reload()` path and the dependency calls `packageManager.resolve()`, but the PR provides no after-fix benchmark proof.

## Solution Assessment

No. Directly invoking private `DefaultResourceLoader` internals with `any` is not the narrowest maintainable solution; a public fast path or upstream contract with real timing proof is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Avoid reaching into Pi private loader internals:** `src/agents/pi-embedded-runner/resource-loader.ts:50-52`
  - body: `loadExtensionFactories` and `extensionsResult` are private in the published Pi declarations, but this fast path calls and mutates them via `any`. That makes embedded runs depend on an implementation detail that can break on a dependency update; please use a public API/upstream fast path or keep the reload contract intact.
  - confidence: 0.88
- **[P2] Hoist the shared mock before the module mock:** `src/agents/pi-embedded-runner/resource-loader.test.ts:24`
  - body: Vitest hoists `vi.mock()` factories, so the factory can run before `mockLoadExtensionFactories` is initialized. This test file should put the shared mock in `vi.hoisted()` or keep the mock state inside a hoisted object before referencing it from the module factory.
  - confidence: 0.86
