## Summary

Keep open: current main still has no disabled-mode diagnostic signal, and this PR is a narrow implementation with green head checks, but it needs a small docs wording cleanup before maintainer merge.

## What This Changes

The PR refactors tool-loop detection to compute would-fire verdicts even when disabled, emits deduplicated `[loop-detection-diag]` INFO lines, and updates tests, docs, and changelog.

## Best Possible Solution

Land the diagnostic change after aligning the disabled-mode docs and cache terminology, leaving default-enforcement and post-compaction policy to their separate items.

## Reproduction Assessment

Yes. Source inspection of current main shows `detectToolCallLoop` exits before detector analysis when disabled, and a repo search finds no `[loop-detection-diag]` tag.

## Solution Assessment

Mostly yes. Computing the existing verdict and logging only an INFO diagnostic while still returning `{ stuck: false }` is narrow and preserves enforcement behavior, but the docs should match the new disabled-mode semantics.
