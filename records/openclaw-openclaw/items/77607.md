## Summary

Keep open: the diagnostic idea is understandable, but this PR currently regresses a merged loop-detection fix on current main, conflicts with the stated default-off/no-extra-log-noise policy, and still lacks real behavior proof.

## What This Changes

The PR refactors tool-loop detection to compute verdicts even when disabled, emit deduplicated `[loop-detection-diag]` INFO logs, and update related tests, docs, and changelog.

## Best Possible Solution

Rebase onto current main, preserve the generic no-progress critical block, and only land disabled-mode diagnostics if maintainers approve the policy or the diagnostics become an explicit opt-in path with real runtime proof.

## Reproduction Assessment

Yes. Source inspection shows current main has no disabled-mode diagnostic path, while the PR head emits one when a disabled detector would fire; the same source pass also shows the PR head drops current main's generic no-progress critical branch.

## Solution Assessment

No. The current patch is not the best merge shape because it regresses current main behavior and changes disabled-mode logging policy without explicit approval; the safer path is a rebase plus maintainer-approved or opt-in diagnostics.
