## Summary

Keep open: current `main` still treats top-level MCP server `env` values as scalar-only and the secrets audit target registry does not include `mcp.servers.*.env.*`. The request is concrete and aligns with existing SecretInput support for the acpx plugin's MCP env surface, but it is a new core SecretRef surface rather than an already-implemented cleanup item.

## What This Changes

Request support for SecretRef objects in `mcp.servers.<name>.env` and include MCP server environment values in `openclaw secrets audit`.

## Best Possible Solution

Extend the existing SecretInput target machinery to top-level `mcp.servers.*.env.*`, resolve refs before MCP server launch without logging values, and add audit/schema/runtime coverage that mirrors the acpx MCP env contract.

## Reproduction Assessment

## Solution Assessment

Yes, the maintainable path is to reuse the existing SecretInput registry and runtime assignment flow rather than adding ad hoc MCP spawn-time resolution. The fix should include schema/type updates, audit target registration, runtime snapshot resolution, and focused tests.
