## Summary

Keep open: current main still limits top-level `mcp.servers.*.env` to scalar values, while SecretRef and audit coverage exists for plugin-declared `plugins.entries.acpx.config.mcpServers.*.env.*` rather than the top-level MCP registry.

## What This Changes

Request top-level `mcp.servers.*.env` values to accept SecretRef objects and have `openclaw secrets audit` report plaintext MCP env credentials.

## Best Possible Solution

Add first-class SecretInput support for top-level `mcp.servers.*.env.*` across schema, types, audit/configure target registration, runtime materialization, and docs while preserving existing scalar env values.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a SecretRef object under `mcp.servers.<name>.env.*` is outside the current scalar schema and audit target registry. I did not run a live CLI repro in this read-only sweep.

## Solution Assessment

Yes in direction, but it needs owner review: extending the existing SecretInput contract to the top-level MCP registry is cleaner than routing users through the acpx plugin config, provided resolution happens before MCP runtimes receive env values.
