## Summary

Keep open. Current main still has the Feishu-specific gate split described in the discussion: the command writes session activation state, but Feishu's pre-dispatch mention gate only reads config, so unmentioned group messages can still reach the agent after `/activation mention`.

## What This Changes

Feishu group chats should honor `/activation mention` by requiring @mentions after the command instead of continuing to answer unmentioned messages.

## Best Possible Solution

Implement the Feishu-owned bridge so persisted per-session group activation overrides the config-derived `requireMention` gate after access checks and route/session resolution, with focused tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with Feishu configured to allow unmentioned group messages, `/activation mention` persists `groupActivation`, but the next unmentioned group message is admitted by a config-only Feishu gate before that session state can hard-block dispatch.

## Solution Assessment

Yes. The narrow maintainable fix is to bridge Feishu's hard admission gate to the persisted activation state for the selected session; prompt-only or docs-only changes would leave the runtime gate disconnected from the successful command.
