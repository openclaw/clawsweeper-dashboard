## Summary

Keep open. Current main still matches the reported Feishu activation gap: `/activation mention` persists generic group activation state, but Feishu's hard group-message admission gate still derives `requireMention` from config before routed session activation can affect dispatch. Hollychou924's root-cause comment and the current source agree on the same split, and no open PR directly references this issue as an implementation candidate.

## What This Changes

Feishu group `/activation mention` should make the bot ignore ordinary unmentioned group messages after the command succeeds.

## Best Possible Solution

Implement the behavior in the Feishu plugin after group and sender allowlist checks: resolve the effective routed group session/session key, treat persisted `groupActivation: "mention"` as requiring a bot mention and `"always"` as allowing unmentioned dispatch, and fall back to config-derived `requireMention` when no activation is stored. Update focused Feishu tests plus Feishu/group docs and changelog so the command confirmation matches runtime behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: with Feishu config allowing unmentioned group messages, `/activation mention` persists `groupActivation`, but the next unmentioned group message is admitted by a config-only Feishu gate before that session state is read.

## Solution Assessment

Yes. The narrow maintainable fix is Feishu-owned admission logic that bridges persisted activation into the hard mention gate after access checks; a prompt-only change or core startup migration would leave Feishu dispatch disconnected from the command state.
