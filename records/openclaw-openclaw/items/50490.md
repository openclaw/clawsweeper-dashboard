## Summary

Keep open. Current main and v2026.5.22 still match the reported Feishu gap: `/activation mention` persists generic group activation state, but Feishu's pre-dispatch group admission still derives `requireMention` from Feishu config and group policy before the later session activation state can suppress non-mention messages.

## What This Changes

The issue requests that Feishu group chats honor `/activation mention` by responding only when the bot is mentioned.

## Best Possible Solution

Implement the behavior inside the Feishu plugin by applying persisted group activation to the existing mention admission decision for the resolved target session, while preserving current group/sender allowlists and config-derived `requireMention` defaults.

## Reproduction Assessment

Yes, by source inspection: `/activation mention` persists `sessionEntry.groupActivation`, while Feishu still derives the hard mention gate from config-only `resolveFeishuReplyPolicy`. I did not run a live Feishu workspace, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes. The narrow maintainable fix is to feed the resolved session activation state into Feishu's existing admission decision, not to add another config knob or rely on the model to self-filter after dispatch.
