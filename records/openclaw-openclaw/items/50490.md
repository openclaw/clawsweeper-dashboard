## Label Justifications

- `P2`: This is a normal-priority channel regression with a focused Feishu group-admission surface and an open fix path.
- `impact:session-state`: The persisted `groupActivation` session state is accepted by the command but not enforced by Feishu ingress.
- `impact:message-loss`: The channel admits messages that the selected activation mode should suppress, so group message-delivery semantics are wrong.

## Summary

Keep open. The Feishu current-main gate still computes `requireMention` from static config/policy and does not read the session `groupActivation` written by `/activation`; the open linked PR is the active fix path, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Feishu group chats should honor `/activation mention` by stopping non-mention messages from entering the agent run path after the session mode changes.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/87709, or an equivalent Feishu plugin fix that derives group admission from persisted `groupActivation` before dispatch while preserving static config fallback.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `/activation mention` persists `groupActivation`, while Feishu group admission only uses config-derived `requireMention` and never reads that session value.

## Solution Assessment

Yes. The narrow best fix is at the Feishu ingress boundary, keeping the shared command persistence intact and making Feishu mirror sibling channels that enforce session activation before dispatch.
