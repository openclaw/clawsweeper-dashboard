## Summary

Keep open. Current main still has the Feishu-specific gap: `/activation mention` persists group activation state, but Feishu admits group messages using config-derived `requireMention` before that session state can affect the hard mention gate.

## What This Changes

Feishu group `/activation mention` should make the bot require a direct mention instead of continuing to answer ordinary unmentioned group messages.

## Best Possible Solution

Apply persisted group activation to Feishu's existing mention admission decision for the resolved target session, while preserving current group/sender allowlists and config-derived defaults.

## Reproduction Assessment

Yes, by source inspection: `/activation mention` persists `groupActivation`, while Feishu's hard gate still derives `requireMention` from config and group policy only. I did not run a live Feishu workspace, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

Yes. The narrow maintainable fix is to feed the resolved session activation state into Feishu's existing admission decision, not to add another config knob or rely on the model to self-filter after dispatch.
