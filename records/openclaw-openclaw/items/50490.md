- `impact:message-loss`: The issue is about Feishu delivering unwanted group replies after the user switched to mention-only activation.

## Summary

Keep open. Current main still has the Feishu-specific activation gap: `/activation mention` stores session activation, but Feishu's pre-dispatch mention gate still decides admission from config-derived `requireMention` before the session activation state can hard-block unmentioned group messages.

## What This Changes

Feishu group `/activation mention` should switch the group to mention-only replies instead of continuing to respond to unmentioned messages.

## Best Possible Solution

Implement the Feishu-owned bridge so the effective target session's persisted group activation is resolved before the hard mention gate, with config-derived `requireMention` as the default and focused coverage for normal, broadcast, and topic-scoped group sessions.

## Reproduction Assessment

Yes, with source-level confidence. Configure Feishu so unmentioned group messages are admitted, send `/activation mention`, and the persisted `groupActivation` remains downstream of the Feishu config-only mention gate for the next unmentioned message.

## Solution Assessment

Yes. The narrow maintainable fix is to resolve the effective Feishu route/session early enough to fold persisted activation into the hard mention gate; prompt-only or docs-only changes would leave the successful command disconnected from runtime behavior.
