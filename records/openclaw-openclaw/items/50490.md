## Summary

Keep open. Current main still matches the Feishu-specific gap described by Hollychou924: `/activation mention` persists session activation state, but Feishu admits or drops group messages through a config-only `requireMention` gate before the persisted activation can affect runtime behavior.

## What This Changes

The issue requests that Feishu group chats honor `/activation mention` by responding only when the bot is mentioned after the command succeeds.

## Best Possible Solution

Implement the bridge in the Feishu plugin: keep config-derived `requireMention` as the default, but let persisted per-session `groupActivation` override Feishu's hard mention admission after group and sender access checks, with focused tests and docs/changelog updates.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: with Feishu config allowing unmentioned group messages, `/activation mention` persists `groupActivation`, but the next unmentioned group message is admitted by a Feishu gate that only reads config and group policy.

## Solution Assessment

Yes. The narrowest maintainable fix is Feishu-owned admission logic aligned with the Telegram pattern; prompt-only changes or core startup migrations would leave the hard Feishu gate disconnected from persisted command state.
