## Summary

Keep open. Current main still has the Feishu-specific split described in the issue discussion: `/activation mention` persists session activation state, but Feishu's pre-dispatch activation gate still derives `requireMention` from config only, so unmentioned group messages can be admitted when config allows them.

## What This Changes

The issue reports that Feishu group `/activation mention` appears to succeed but does not stop replies to later unmentioned group messages.

## Best Possible Solution

Implement the Feishu-owned bridge so persisted per-session group activation overrides the config-derived `requireMention` default at the Feishu activation gate, with focused regression coverage for normal, broadcast, and topic-scoped group sessions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with Feishu configured to allow unmentioned group messages, `/activation mention` persists `groupActivation`, but the next unmentioned group message is admitted by a config-only Feishu gate before that session state can hard-block dispatch.

## Solution Assessment

Yes. The narrow maintainable fix is to resolve the effective Feishu route/session early enough to fold persisted activation into the hard mention gate; prompt-only or docs-only changes would leave runtime behavior disconnected from the command.
