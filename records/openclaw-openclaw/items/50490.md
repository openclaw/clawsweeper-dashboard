## Label Justifications

- `P2`: This is a real Feishu group workflow regression with a narrow channel/session blast radius and a config workaround.
- `impact:message-loss`: The bug is in message admission/suppression behavior: Feishu admits group messages that activation state says should require a mention.

## Summary

Keep open. Current main and v2026.5.22 still have the Feishu-specific gap: `/activation mention` persists group activation state, but Feishu group admission still derives `requireMention` from config/group policy before the session activation state can affect routing; the earlier comment’s workaround of setting `channels.feishu.requireMention: true` remains accurate but does not fix the command path.

## What This Changes

The issue requests that Feishu group `/activation mention` actually switch the group from always-on replies to mention-only replies.

## Best Possible Solution

Wire Feishu group admission to the resolved session’s activation state before the mention filter, preserving group/sender allowlists and config defaults, then add a regression showing `/activation mention` suppresses later non-mention group messages.

## Reproduction Assessment

Yes, source-level. A Feishu group with config-derived `requireMention: false` can persist `groupActivation: mention`, but the next non-mention message still passes the Feishu gate because that gate does not read the session entry.

## Solution Assessment

Yes. The narrow maintainable fix is in the Feishu plugin admission path, likely after route/session resolution but before mention-based drop, rather than adding a new config knob or relying on model self-filtering.
