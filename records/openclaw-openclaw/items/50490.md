- `P2`: This is a clear Feishu group-chat regression with a bounded channel/session-state surface, but it is not a core-runtime outage.
- `impact:message-loss`: Feishu routes messages for visible replies even after the session has been switched to mention-only activation, so channel delivery behavior is wrong.

## Summary

Keep open. Current `main` and the latest release still have the Feishu-specific gap: `/activation mention` persists session activation, but Feishu decides whether to admit unmentioned group messages from config-derived `requireMention` before that session state can affect dispatch.

## What This Changes

The issue requests that Feishu group chats honor `/activation mention` by suppressing replies to unmentioned messages after the command succeeds.

## Best Possible Solution

Have the Feishu plugin resolve effective persisted group activation for the target group session before the hard mention gate, while preserving config-derived `requireMention` as the default and existing group/sender admission checks.

## Reproduction Assessment

Yes, with source-level confidence. Configure Feishu so unmentioned group messages are admitted, send `/activation mention`, then send an unmentioned group message; current source still gates Feishu admission from config-only `requireMention` before persisted activation can apply.

## Solution Assessment

Yes. The narrow maintainable solution is to resolve effective Feishu route/session activation before the hard mention gate; prompt-only or docs-only changes would leave a successful command disconnected from runtime routing.
