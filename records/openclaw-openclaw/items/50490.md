## Summary

Keep open. Current main still has the Feishu-specific split described in the discussion: `/activation mention` persists group activation state, but Feishu's pre-dispatch gate derives `requireMention` from config only, so unmentioned group messages can still reach the agent when config allows them.

## What This Changes

Report asks Feishu group `/activation mention` to stop replies to unmentioned group messages after switching the session to mention-only activation.

## Best Possible Solution

Implement the Feishu-owned bridge so persisted per-session group activation overrides the config-derived `requireMention` default at the Feishu activation gate, with focused coverage for normal, broadcast, and topic-scoped group sessions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with Feishu configured to allow unmentioned group messages, `/activation mention` persists `groupActivation`, but the next unmentioned message is admitted by a config-only Feishu gate before that session state can hard-block dispatch.

## Solution Assessment

Yes. The narrow maintainable fix is to resolve the effective Feishu route/session early enough to fold persisted activation into the hard mention gate; prompt-only or docs-only changes would leave runtime behavior disconnected from the successful command.
