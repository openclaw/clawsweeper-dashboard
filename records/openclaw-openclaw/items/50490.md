## Summary

Keep open. Current main still has the Feishu-specific activation gap described in the discussion: `/activation mention` persists session activation, but Feishu's pre-dispatch mention gate still uses config-derived `requireMention` before that session state can hard-block unmentioned group messages.

## What This Changes

The issue asks Feishu group chats to honor `/activation mention` by stopping responses to ordinary unmentioned group messages after the command succeeds.

## Best Possible Solution

Implement the Feishu-owned bridge so persisted per-session group activation overrides the config-derived `requireMention` default at the Feishu activation gate, with focused coverage for normal, broadcast, and topic-scoped group sessions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: with Feishu configured to allow unmentioned group messages, `/activation mention` persists `groupActivation`, but the next unmentioned message is still admitted by Feishu's config-only gate before that session state can hard-block dispatch.

## Solution Assessment

Yes. The narrow maintainable fix is to resolve the effective Feishu route/session early enough to fold persisted activation into the hard mention gate; prompt-only or docs-only changes would leave runtime behavior disconnected from the successful command.
