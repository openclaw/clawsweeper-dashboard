## Summary

Keep open. Current `main` and `v2026.5.22` still have the Feishu-specific gap: `/activation mention` persists group activation state, but Feishu admission still uses config-derived `requireMention` before that session state can affect routing.

## What This Changes

Feishu group chats should honor `/activation mention` by suppressing unmentioned group messages after the mode is toggled.

## Best Possible Solution

Have the Feishu plugin resolve effective persisted group activation for the target group session before the hard mention gate, while keeping config-derived `requireMention` as the default and preserving existing admission checks.

Yes. The narrow maintainable solution is to resolve Feishu route/session activation before the hard mention gate; prompt-only or docs-only changes would leave a successful command disconnected from runtime routing.

## Repair Work Prompt

Fix the Feishu group activation bug from https://github.com/openclaw/openclaw/issues/50490. In Feishu group chats, `/activation mention` currently persists `sessionEntry.groupActivation` but the Feishu pre-dispatch mention gate still uses config-only `requireMention`, so unmentioned messages keep reaching the agent when `groupPolicy: open` or `requireMention: false` admits them. Implement a bounded Feishu-plugin fix that resolves effective persisted activation for the same target group session before `resolveFeishuGroupSenderActivationIngressAccess`, preserves config-derived `requireMention` as the default, and does not bypass group admission, sender allowlists, broadcast dispatch, topic-scoped sessions, or command authorization. Add focused regression tests for `/activation mention` suppressing unmentioned Feishu group messages while @mentions still dispatch, and update docs/changelog only if user-facing behavior/docs are changed. Do not add a new config option or move owner-specific Feishu policy into core.
