## Summary

Keep open. Current main still routes the macOS node service through `openclaw node run`, that command is not an invalid-config bypass, and the node runner still reads strict runtime config before opening the Gateway connection. Gateway startup has last-known-good recovery for invalid effective config, but the node startup path does not share that policy, so this remains a real behavior/policy issue rather than cleanup-closeable.

## What This Changes

The issue asks for the macOS `ai.openclaw.node` service to handle a config that the Gateway can use instead of failing with `Config invalid` and refusing to connect.

## Best Possible Solution

Define the intended `openclaw node run` contract for invalid effective config. Either give node a narrow best-effort/last-known-good startup path for the TLS/auth/plugin-registry data it needs, or keep startup strict but emit node-specific diagnostics and repair guidance, with regression coverage and `docs/cli/node.md` updated if behavior changes.
