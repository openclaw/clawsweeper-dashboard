## Summary

Keep open. Current main still separates Gateway recovery from node-host startup: the macOS node service runs `openclaw node run`, that path is not invalid-config allowlisted, and the node runner still reads strict runtime config before connecting, so the reported failure shape remains a real behavior and policy gap.

## What This Changes

The issue reports that the macOS `ai.openclaw.node` launchd service fails with `Config invalid` even though the Gateway can continue using the same `openclaw.json`.

## Best Possible Solution

Define the intended `openclaw node run` startup contract, then implement that contract narrowly: either add node-specific best-effort or last-known-good config recovery for the fields the node needs, or keep node startup strict but emit precise node-service diagnostics and repair guidance. Add focused regression coverage and update `docs/cli/node.md` if the behavior changes.

## Reproduction Assessment

Yes for the code-path failure shape: with an invalid effective config, `openclaw node run` is not allowlisted by the config guard and the node runner then depends on strict `getRuntimeConfig()` before connecting. The reporter's exact config is missing, so we cannot live-reproduce their specific macOS launchd instance.

## Solution Assessment

Unclear. Copying Gateway recovery into node startup may solve the service symptom, but it changes fail-closed config semantics for node TLS/auth/plugin data; a maintainer should choose between narrow recovery and stricter diagnostics before implementation.
