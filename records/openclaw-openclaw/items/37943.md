## Summary

Keep open. Current main still has a clear invalid-config guard path for the reported log, but the issue only proves JSON syntax with `jq`; it does not include the effective config, `openclaw config validate --json`, or launchd service environment needed to confirm a node-only parser/config-path bug.

## What This Changes

The issue reports that the macOS `ai.openclaw.node` launchd service logs invalid config while the gateway accepts the same `openclaw.json`.

## Best Possible Solution

Collect structured config validation output plus the node service plist/environment; if the config is schema-valid and both processes use the same path, fix the node startup/config-path path while preserving fail-closed behavior for schema-invalid config.

## Reproduction Assessment

Yes for the source-level invalid-config log path: current `node run` still goes through the config guard and can emit the reported note when the effective config is invalid. No for the stronger claim that a schema-valid config fails only in the node service, because the report lacks OpenClaw validation output and the effective config.

## Solution Assessment

Unclear. There is no proposed patch, and treating `jq` success as OpenClaw config validity would be the wrong boundary; the narrow maintainable path is first to prove whether this is schema invalidity, path/profile divergence, or a true node-only parser mismatch.
