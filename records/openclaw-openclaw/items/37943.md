## Summary

Keep open. Current main explains the reported log as the invalid-config guard path rather than a JSON parser failure, and Gateway now also fails closed on schema-invalid config; the report still lacks the effective config or `openclaw config validate` output needed to prove a node-only parser mismatch.

## What This Changes

The issue asks OpenClaw to resolve a macOS `ai.openclaw.node` launchd startup failure that reports `Config invalid` even though `openclaw.json` is JSON-valid and the Gateway had been using it.

## Best Possible Solution

Keep node and Gateway startup consistent on schema-invalid config, and require `openclaw config validate --json` plus the affected config issue before changing node recovery behavior.

## Reproduction Assessment

Yes for the source-level invalid-config path: `openclaw node run` still runs doctor preflight/config guard and can emit the reported note when the effective config is invalid. No for the stronger claim that a schema-valid config fails, because the issue includes only `jq` validation and no OpenClaw validation output.

## Solution Assessment

Unclear. Treating `jq` success as valid OpenClaw config is not correct, and allowing node startup to continue on invalid schema would change the fail-closed startup policy.
