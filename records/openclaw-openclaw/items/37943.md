## Summary

Keep open. Current main still matches the April 26 ClawSweeper analysis: the macOS node service launches the normal `openclaw node run` path, that command is not invalid-config allowlisted or guard-bypassed, and the node runner still reads strict runtime config before connecting to the Gateway.

## What This Changes

The issue asks for the macOS `ai.openclaw.node` launchd service to handle the same usable OpenClaw config as the Gateway instead of logging invalid config and failing to connect.

## Best Possible Solution

Define the intended `openclaw node run` startup contract, then implement a narrow regression-tested node path and update `docs/cli/node.md` if behavior changes.

## Reproduction Assessment

Yes for the source-level failure shape: with an invalid effective config that Gateway can recover from, current main still sends the node service through strict `openclaw node run` startup. No live reproduction of the reporter's exact macOS launchd instance is possible from the provided raw data because the effective config and validation issue are missing.

## Solution Assessment

Unclear. Copying Gateway recovery into node startup may solve the service symptom, but the node path consumes auth, TLS, SecretRef, and plugin data, so a maintainer should choose between narrow recovery and strict fail-closed diagnostics before implementation.
