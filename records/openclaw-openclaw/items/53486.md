## Summary

Keep open. Current main and the latest release still route Feishu card-shaped JSON supplied as the plain `message`/`text` value through the text send path, while the focused fix PR was closed unmerged for lack of live Feishu proof.

## What This Changes

Fix Feishu proactive sends so valid interactive-card JSON passed via `message(action=send)` renders as a native interactive card instead of raw text.

## Best Possible Solution

Add Feishu-local compatibility detection for valid native interactive-card JSON strings in `send` and `thread-reply`, sanitize through the existing native-card sanitizer, route valid cards to `sendCardFeishu`, and leave malformed, non-card, or oversized JSON on the text path.

## Reproduction Assessment

Yes, by source inspection rather than live Feishu. The reported payload supplies only a plain `message` string, and current main only creates Feishu cards from `presentation` or structured `channelData` before falling through to `sendMessageFeishu`.

## Solution Assessment

Yes for the fix direction. Keeping compatibility parsing inside the Feishu plugin runtime and reusing the existing sanitizer is narrower than adding Feishu-specific parsing to shared core.
