## Summary

Keep this issue open. Current main still routes valid Feishu interactive-card JSON supplied through the plain `message` parameter as text/post content; the existing native-card support covers structured `presentation`, `interactive`, and `channelData.feishu.card` payloads, but not this compatibility regression.

## What This Changes

The issue requests that Feishu `message(action=send)` detect valid interactive-card JSON strings in the plain `message` field and send them as native Feishu interactive cards instead of plain text.

## Best Possible Solution

Add Feishu-local compatibility detection for valid native interactive-card JSON strings in `send` and `thread-reply`, sanitize through the existing native-card safety path, route valid cards to `sendCardFeishu`, and leave malformed card JSON or ordinary JSON on the text path. Keep the shared message tool provider-agnostic unless more channels need the same seam.

## Reproduction Assessment

Yes. The issue includes concrete steps and a sample card payload, a later comment confirms the regression on OpenClaw 2026.4.25, and current main’s source path shows a plain `message` string cannot become a Feishu native card.

## Solution Assessment

Yes. The narrowest maintainable fix is in Feishu-owned send handling using the existing sanitizer and `sendCardFeishu`; moving Feishu-specific JSON parsing into shared core would be broader than needed unless a generic plugin API gap appears.
