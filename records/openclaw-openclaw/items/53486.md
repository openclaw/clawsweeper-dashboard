## Summary

Keep open: current main still sends valid Feishu card JSON supplied as plain `message`/`text` through the text path, while the direct fix PR closed unmerged for lack of live Feishu proof and the merged native-card work only covers structured `channelData`/`presentation` payloads.

## What This Changes

Reported regression where Feishu proactive `message(action=send)` renders card JSON in the plain `message` parameter as raw text instead of an interactive card.

## Best Possible Solution

Add Feishu-local compatibility detection for valid native interactive-card JSON in plain `message`/`text` for `send` and `thread-reply`, sanitize through the existing native-card sanitizer, route valid cards to `sendCardFeishu`, and leave malformed or non-card JSON on the text path.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: the issue provides a concrete card JSON payload, and current main only builds Feishu cards from `presentation` or structured channel data before falling plain text through `sendMessageFeishu`. I did not run a live Feishu workspace.

## Solution Assessment

Yes, the best fix shape is Feishu-local parsing and sanitization for valid card-shaped plain text before the text fallback. Adding Feishu-native fields or parsing to the shared message tool would be the wrong owner boundary.
