## Summary

Keep open. Current main still routes Feishu card-shaped JSON in the plain `message`/`text` parameter through the plain text sender, and the only focused fix candidate, https://github.com/openclaw/openclaw/pull/77268, was closed unmerged because live Feishu proof was unavailable.

## What This Changes

Restore Feishu proactive send/thread-reply so valid interactive-card JSON in `message` or `text` renders as a Feishu card rather than raw text.

## Best Possible Solution

Add a Feishu-local compatibility parser for valid native interactive-card JSON in `send` and `thread-reply`, reuse the existing sanitizer, and leave malformed, non-card, or oversized JSON on the plain-text path.

## Reproduction Assessment

Yes, by source inspection rather than live Feishu. The supplied JSON has no `presentation` or `channelData`, and current main reads it as text before falling through to `sendMessageFeishu`.

## Solution Assessment

Yes for the code direction. Keeping Feishu card compatibility parsing inside the Feishu plugin runtime and reusing `sanitizeNativeFeishuCard` is narrower than adding Feishu-specific parsing to shared core.
