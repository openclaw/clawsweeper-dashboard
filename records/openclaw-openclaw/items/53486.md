## Summary

Keep open: current main and v2026.6.10 still send Feishu proactive `message` card JSON as plain text unless a structured presentation or `channelData.feishu.card` is already present, and the same-fix PRs are closed unmerged after blocking on live Feishu proof.

## What This Changes

The issue reports a Feishu regression where proactive `message(action=send)` renders card-shaped JSON from the `message` parameter as raw text instead of an interactive card.

## Best Possible Solution

Keep this issue open until a Feishu-local, sanitizer-backed fix lands with focused routing coverage and redacted live Feishu proof, or a maintainer explicitly accepts a proof override for the credential-gated path.

## Reproduction Assessment

Yes, at source level. Current main reads plain `message`/`text` but only builds a Feishu card from `presentation` or structured channel data, so card-shaped JSON falls through to `sendMessageFeishu`; I did not run a live Feishu tenant send.

## Solution Assessment

Yes for the likely code direction, but not complete yet. The maintainable fix belongs in the Feishu plugin boundary, reusing the existing sanitizer/native-card path rather than adding provider-specific parsing to shared core, with live proof or a maintainer override before closure.
