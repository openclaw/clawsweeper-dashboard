## Summary

Keep open. Current main and the latest release still route Feishu card-shaped JSON supplied as plain `message` text through the text-send path, and the focused fix PR was closed unmerged for lack of live Feishu proof.

## What This Changes

The issue asks to restore Feishu proactive sends so valid interactive-card JSON in the plain `message`/`text` parameter renders as an interactive card instead of raw text.

## Best Possible Solution

Add Feishu-local compatibility detection for valid native interactive-card JSON strings in `send` and `thread-reply`, reuse the existing sanitizer and legacy-command guard, and leave malformed, ordinary, or oversized JSON on the text path.

## Reproduction Assessment

Yes, by source inspection rather than live Feishu. The reported payload supplies only a plain `message` string, and current main only creates Feishu cards from `presentation` or structured `channelData` before falling through to `sendMessageFeishu`.

## Solution Assessment

Yes for the fix direction. Keeping compatibility parsing inside the Feishu plugin runtime and reusing the existing sanitizer is narrower than adding Feishu-specific parsing to the shared message runner.
