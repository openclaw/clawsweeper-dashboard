## Summary

Keep this issue open. Current main still routes Feishu card JSON supplied as the plain `message` string through the text/post path; native-card support only covers structured `presentation`, `interactive`, or `channelData.feishu.card` payloads, and the merged adjacent native-card PR does not cover this compatibility regression.

## What This Changes

The issue requests that Feishu `message(action=send)` detect valid interactive-card JSON strings in the plain `message` field and send them as native Feishu interactive cards instead of plain text.

## Best Possible Solution

Add Feishu-local compatibility detection for valid native interactive-card JSON strings in `send` and `thread-reply`, sanitize through the existing native-card safety path, route valid cards to `sendCardFeishu`, and keep malformed card JSON or ordinary JSON on the text path. Keep shared message-tool core provider-agnostic unless more channels need a generic seam.

## Reproduction Assessment

Yes. The issue provides concrete steps and a sample payload, a later comment confirms the same regression on OpenClaw 2026.4.25, and current main’s source path shows a plain `message` string cannot become a Feishu native card today.

## Solution Assessment

Yes. The narrowest maintainable fix is in Feishu-owned send handling using the existing sanitizer and `sendCardFeishu`; moving Feishu-specific JSON parsing into shared core would be broader than needed unless a generic plugin API gap appears.
