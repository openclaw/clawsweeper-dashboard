## Summary

Keep open: the active-run control direction is useful and has real Discord VC proof, but the PR still has two source-backed correctness issues in status reporting and gateway-relay active-run state handling.

## What This Changes

This PR adds realtime voice active-run status, steer, cancel, and follow-up control across Gateway Talk methods, WebUI realtime transports, Discord voice, SDK exports, protocol schemas, docs, and focused tests.

## Best Possible Solution

Keep the centralized active-run control seam, but require status to report the underlying active work and require relay transcript interception to be gated by verified live run state or reliable cleanup before merge.

## Reproduction Assessment

Yes for the PR findings: source inspection shows control progress is emitted as a tool event and then selected by status formatting, and gateway relay transcript interception depends on activeAgentRuns state that is cleared only after server-side tool-result handling. A current-main bug repro is not applicable because these defects are introduced by this PR branch.

## Solution Assessment

No, not yet. The centralization is the right shape, but the PR should fix status filtering and stale relay active-run cleanup before it is the best maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Skip control-tool progress when formatting status:** `src/talk/agent-run-control-shared.ts:303`
  - body: `formatRealtimeVoiceAgentStatus` selects the newest `tool.*` event, but this PR also records active-run control as `tool.progress` with `name: "openclaw_agent_control"`. After one status/steer/cancel event, a later status request can say OpenClaw is working in `openclaw_agent_control` instead of reporting the underlying active tool or diagnostic activity; filter control-tool events before choosing the status source.
  - confidence: 0.9
- **[P2] Verify live relay state before intercepting transcripts:** `src/gateway/talk-realtime-relay.ts:300-301`
  - body: The relay now treats any final transcript as active-run control when `activeAgentRuns.size > 0`, but those entries are cleared only when `talk.session.submitToolResult` is processed. Because the browser sends that result fire-and-forget, a missed or rejected submit can leave stale entries and repeatedly swallow phrases like "status" or "cancel" instead of forwarding them to the provider; gate this on verifiable live run state or add reliable cleanup/retry.
  - confidence: 0.86
