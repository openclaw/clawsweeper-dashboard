## Summary

## What This Changes

The branch buffers block-kind replies in `createReplyDispatcher` until a final payload arrives, drops them on NO_REPLY, adds an internal opt-out, and adds dispatcher regression tests.

## Best Possible Solution

Make the turn-level decision use the canonical silent-payload contract before any buffer flush, preserve or explicitly redesign the block delivery/idle contract with tests, then prove the leak shape in a redacted live Telegram run.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main dispatches a block before a later final NO_REPLY, and the PR still flushes buffered blocks when the final is the supported JSON silent envelope because it checks only exact text NO_REPLY.

## Solution Assessment

No, not yet. Buffer-until-final is a plausible shape, but the implementation must share the canonical silent-payload detector and settle the caller delivery contract before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Suppress buffered blocks for every silent final form:** `src/auto-reply/reply/reply-dispatcher.ts:275`
  - body: `normalizeReplyPayload` treats both bare `NO_REPLY` and `{"action":"NO_REPLY"}` as silent, but this new branch only checks `isSilentReplyText` before deciding to drop the buffer. With a buffered block followed by a JSON silent final, execution skips this branch, flushes the block as a real reply, and only then the final normalizes to null, leaving the substrate leak open for an existing supported silent payload shape.
  - confidence: 0.94
- **[P2] Do not report buffered blocks as idle delivery:** `src/auto-reply/reply/reply-dispatcher.ts:270-271`
  - body: This path returns `true` after only pushing a block into `bufferedBlocks`; it does not add the payload to `sendChain` or `pending` until a later final arrives. Callers such as the same-channel `onBlockReply` path wait for `waitForReplyDispatcherIdle()` when `sendBlockReply` returns true, and the existing test expects delivery to have started before that callback resolves, so this changes the dispatcher contract without updating the caller or coverage.
  - confidence: 0.86
