## Summary

Keep open. Current main still exposes `tools.fs.workspaceOnly` as one coarse boolean, while the requested per-operation read/search versus write/edit/apply_patch policy remains a core security/config design change rather than a stale or already-implemented cleanup item.

## What This Changes

The issue asks for `tools.fs.workspaceOnly` to support per-operation filesystem policy so reads/search can be broader while write/edit/apply_patch remain workspace-confined.

## Best Possible Solution

Add a backward-compatible core filesystem policy extension that keeps the legacy boolean, introduces a reviewed operation-scoped or allowed-roots contract, threads it through all filesystem enforcement layers, and updates config schema/help/docs/tests together.

## Reproduction Assessment

Yes, as a capability gap by source inspection. Current main only accepts a boolean `tools.fs.workspaceOnly`, so the proposed per-operation object policy is rejected by the strict schema and cannot be represented in runtime policy.

## Solution Assessment

Unclear. The requested direction is plausible, but maintainers should choose the safest operation map or allowlisted-root contract and coordinate it with #70795 before implementation.
