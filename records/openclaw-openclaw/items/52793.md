## Summary

Keep open. Current main still models `tools.fs.workspaceOnly` as one boolean and threads that single policy through read/write/edit/apply_patch, so granular per-operation filesystem policy remains unimplemented and needs security-minded core config design.

## What This Changes

The issue requests per-tool or per-operation `tools.fs.workspaceOnly` policy so read/search access can be widened while write/edit/apply_patch remain workspace-bound.

## Best Possible Solution

Add a backward-compatible filesystem policy contract that keeps the legacy boolean, defines reviewed operation or allowed-root semantics, threads it through core host/sandbox/media/plugin paths, and updates schema, help, docs, and regression tests proving widened reads do not weaken writes.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: current main only accepts and resolves a boolean `tools.fs.workspaceOnly`, so a per-operation object policy cannot be represented by the strict schema or runtime policy.

## Solution Assessment

Unclear. The requested direction is reasonable, but maintainers should choose the safest contract, such as operation-specific rules versus allowed read roots, and coordinate it with #70795 before implementation.
