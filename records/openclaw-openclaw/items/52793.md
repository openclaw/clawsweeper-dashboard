## Summary

Keep open. Current main still exposes `tools.fs.workspaceOnly` as one boolean and wires that same policy through filesystem tools, so the requested read/search versus write/edit/apply_patch split remains unimplemented and needs security/API design.

## What This Changes

The issue requests per-operation filesystem workspace policy so read/search can be broader than write/edit/apply_patch.

## Best Possible Solution

Add a backward-compatible core filesystem policy extension that preserves the legacy boolean, defines a reviewed per-operation or allowed-roots contract, threads it through core/plugin/media/sandbox filesystem paths, and updates schema/help/docs plus regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a config object under `tools.fs.workspaceOnly` would be rejected by the strict schema, and the runtime has only one boolean policy to apply.

## Solution Assessment

Unclear. The requested direction is reasonable and belongs in core, but maintainers should choose the safest operation or allowed-root contract before implementation.
