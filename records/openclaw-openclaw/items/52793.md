## Summary

Keep open. Current main still exposes `tools.fs.workspaceOnly` as one coarse boolean, and the runtime threads that single flag through read, write, edit, and the `apply_patch` umbrella behavior rather than accepting per-operation filesystem policy.

## What This Changes

The issue requests a granular filesystem workspace policy so read/search can be allowed outside the workspace while write/edit/apply_patch remain workspace-contained.

## Best Possible Solution

Add a backward-compatible core filesystem policy extension: keep the legacy boolean, add either an operation-specific object form or explicit read/search allowlisted roots, update config types/schema/generated help/docs, thread the richer policy through tool creation and plugin/media contexts, and add tests proving read/search can reach approved external roots while write/edit/apply_patch remain confined.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: the proposed object form is rejected by the strict schema, and current tests/runtime show one boolean controls read/write/edit behavior rather than separate operation policies.

## Solution Assessment

Unclear on the exact JSON shape, but the fix belongs in core rather than a plugin workaround because the typed config, schema, and runtime policy currently expose only one coarse filesystem flag. Maintainers should choose the safest object/allowlist contract before implementation.
