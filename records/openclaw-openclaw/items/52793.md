## Summary

Keep open. Current main still has one coarse `tools.fs.workspaceOnly` boolean, so the requested per-operation read/search versus write/edit/apply_patch policy remains unimplemented and needs maintainer security/API design.

## What This Changes

The issue requests per-operation filesystem workspace policy so read/search can be broader while write/edit/apply_patch remain confined.

## Best Possible Solution

Add a backward-compatible core filesystem policy extension that keeps the legacy boolean, introduces a reviewed operation or allowed-roots contract, threads it through all core/plugin/media filesystem paths, updates config schema/help/docs, and adds regression coverage proving writes remain confined when reads/search are widened.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main only accepts a boolean `tools.fs.workspaceOnly`, so the proposed object/per-operation policy is rejected by the strict schema and cannot be represented in the runtime policy.

## Solution Assessment

Unclear. The requested direction is reasonable and belongs in core, but maintainers should choose the safest operation/allowlist contract and coordinate it with #70795 before implementation.
