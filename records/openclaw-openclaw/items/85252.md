## Summary

Keep open. Current main still propagates non-missing read errors from the memory-core session/read paths and memory-wiki compile path, while the related page-summary issue only covers one overlapping call site rather than the session-start/session-delta retry surface.

## What This Changes

Retry transient raw errno -11 read failures with bounded backoff in memory-core sync and memory-wiki compile paths for iCloud/FileProvider-backed memory files.

## Best Possible Solution

Add a shared bounded transient-read helper in the memory-host layer, use it from memory-core session/file reads and memory-wiki page reads, and preserve current missing-file and permanent-error behavior.

## Reproduction Assessment

Yes for a source-level reproduction: inject a transient `NodeJS.ErrnoException` from `FileHandle.read`, `readRegularFile`, or `fs.readFile`, and current main will propagate it through the memory sync or compile paths. I did not establish a live iCloud/FileProvider reproduction in this read-only Linux review.

## Solution Assessment

Yes: the narrowest maintainable fix is a shared bounded retry/backoff wrapper for transient read errors, with focused regression tests. A new config knob or dist-only patch is not the right first step.
