## Label Justifications

- `P2`: The bug affects memory sync and wiki compile reliability for iCloud/FileProvider-backed storage, but the blast radius is scoped to that filesystem class.
- `impact:session-state`: The report says session-start and session-delta memory sync can fail, which can leave a session without expected prior memory state.

## Summary

Keep open: current main still rethrows transient non-missing read errors in the memory read paths, and the open linked fix PR has not merged yet, so this issue should remain the canonical tracking item until that PR lands or is declined.

## What This Changes

Add bounded retry/backoff for transient FileProvider-style read failures in memory-core, memory-host, and memory-wiki sync/compile paths.

## Best Possible Solution

Review and either land or decline https://github.com/openclaw/openclaw/pull/85351; after it merges, close this issue as implemented while keeping the separate concurrency work tracked independently.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction: inject a transient NodeJS ErrnoException from readRegularFile, fs.readFile, fs.open, or FileHandle.read and current main will rethrow it through the memory sync/compile paths. I did not establish a live iCloud/FileProvider race on this runner.

## Solution Assessment

Yes: a narrow shared bounded retry helper for transient memory read failures is the maintainable fix shape, with missing-file and permanent-error behavior preserved. The issue should stay open because the linked implementation PR is still open.
