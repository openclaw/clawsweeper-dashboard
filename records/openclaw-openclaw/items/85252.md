## Label Justifications

- `P2`: The issue affects memory sync and wiki compile reliability for iCloud/FileProvider-backed stores, but the blast radius is scoped to that filesystem class rather than the whole runtime.
- `impact:session-state`: The report says session-start and session-delta memory sync can fail, which can leave a session without the expected memory snapshot or recent memory state.

## Summary

Keep open. Current main and the latest release still propagate non-missing read errors from the memory-core/session and memory-wiki compile paths, and the open Memory Wiki PR only covers https://github.com/openclaw/openclaw/issues/68738 rather than this issue's memory-core session read retry surface.

## What This Changes

The issue asks OpenClaw to retry transient EAGAIN/EWOULDBLOCK read failures in memory-core and memory-wiki file reads on iCloud/FileProvider-backed paths.

## Best Possible Solution

Add a shared, bounded memory-owned read retry helper for transient FileProvider-style read failures, use it from memory-core/session and memory-wiki read paths as applicable, and preserve missing-file and permanent-error behavior.

## Reproduction Assessment

Yes for a source-level reproduction: inject a transient `NodeJS.ErrnoException` from `FileHandle.read`, `readRegularFile`, or `fs.readFile` and current main will propagate it through the memory sync/compile paths. I did not establish a live iCloud/FileProvider repro on this runner.

## Solution Assessment

Yes. The narrow maintainable fix is a shared bounded read retry/backoff helper with targeted tests, not a dist-only patch or a new config knob first; coordinate the Memory Wiki concurrency portion with https://github.com/openclaw/openclaw/pull/84458.
