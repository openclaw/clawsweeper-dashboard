## Summary

Keep open. Current main and the latest shipped release still propagate non-missing read errors from the memory-core/session and memory-wiki compile paths, and the linked page-summary issue does not cover the session-start/session-delta read retry surface.

## What This Changes

The issue asks for bounded retry/backoff around memory-core and memory-wiki reads that surface raw errno -11 failures from iCloud/FileProvider-backed files.

## Best Possible Solution

Add a shared, bounded memory-host read retry helper for transient FileProvider-style read failures, use it from memory-core and memory-wiki read paths, and keep missing-file and permanent-error behavior unchanged.

## Reproduction Assessment

Yes for a focused source-level repro: inject a transient `NodeJS.ErrnoException` from `FileHandle.read`, `readRegularFile`, or `fs.readFile` and current main will propagate it through the memory sync/compile paths. I did not establish a live iCloud/FileProvider repro on this runner.

## Solution Assessment

Yes, the best fix is a narrow shared read wrapper with bounded retry/backoff and targeted tests, not a dist-only patch or a new config knob first. The memory-wiki page-summary concurrency issue should be coordinated with https://github.com/openclaw/openclaw/issues/68738 rather than duplicated blindly.
