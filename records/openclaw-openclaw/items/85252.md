## Summary

Keep open. Current main still propagates non-missing read errors from the memory-host, session transcript, and memory-wiki compile paths, and the issue is already paired with an open closing PR that has not merged.

## What This Changes

The issue requests bounded retry/backoff for transient iCloud/FileProvider-backed read failures in memory-core sync and memory-wiki compile paths.

## Best Possible Solution

Review and land or replace the existing bounded memory read retry PR, keeping missing-file and permanent-error behavior unchanged and leaving the broader wiki read-concurrency issue tracked separately.

## Reproduction Assessment

Yes for a source-level reproduction: inject a transient `NodeJS.ErrnoException` from `readRegularFile`, `fs.readFile`, `fs.open`, or `FileHandle.read` and the current paths rethrow it after only handling missing files. I did not establish a live iCloud/FileProvider repro in this review environment.

## Solution Assessment

Yes, a shared bounded retry helper applied to the affected memory read seams with focused regression tests is the narrow maintainable solution. Because https://github.com/openclaw/openclaw/pull/85351 is already open with closing syntax, the issue should stay open until that PR lands or maintainers choose a replacement.
