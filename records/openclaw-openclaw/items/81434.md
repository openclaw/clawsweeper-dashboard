## Summary

Keep this PR open because it addresses a real heartbeat scaffolding leak, but it should not merge yet: the new sanitizer ordering can expose wrapped tool-call JSON that current main's XML tool-call stripper hides, and the latest head also needs fresh real behavior proof.

## What This Changes

The branch extends assistant-visible/user-facing sanitization for Codex scaffolding, applies it to heartbeat delivery paths, requires the heartbeat response tool in tool mode, and adds regression tests.

## Best Possible Solution

Land one focused heartbeat sanitizer fix after preserving full tool-call block stripping, robustly handling file_contents blocks, and refreshing latest-head real behavior proof.

## Reproduction Assessment

Yes, at source level. Current main sends heartbeat reply text through normalization paths that do not apply the canonical assistant-visible sanitizer, and https://github.com/openclaw/openclaw/issues/81369 supplies concrete leaked scaffolding examples.

## Solution Assessment

No, not as written. Reusing the canonical sanitizer is the right direction, but the new marker-only pass must not run before the full XML tool-call block stripper, and latest-head real behavior proof needs to cover the added heartbeat-tool mode change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep wrapped tool-call payloads hidden:** `src/shared/text/assistant-visible-text.ts:760`
  - body: The new scaffolding pass runs before `stripToolCallXmlTags`, so a normal block like `<tool_calls>\n[{...}]\n</tool_calls>` has only its wrapper lines removed and the JSON payload is left as visible text. That regresses the existing sanitizer contract and can leak the tool arguments this PR is trying to suppress; run the full XML block stripper before marker-only cleanup or make this pass remove the entire wrapped block.
  - confidence: 0.93
- **[P2] Close file_contents blocks inside copied fences:** `src/shared/text/assistant-visible-text.ts:644-645`
  - body: `findCodeRegions` can mark a copied Markdown fence inside a `<file_contents>` excerpt as code through the end of the string. When that happens, this `continue` skips the real `</file_contents>` close tag, leaves `inFileContentsBlock` true, and drops any visible text after the block. Once a file_contents block is open, its close tag should be honored even if the copied file contents contain fences.
  - confidence: 0.84
