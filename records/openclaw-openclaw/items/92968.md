## Summary

Keep open: the PR targets a real status/context-window bug, but the proposed warmup is still fire-and-forget, so the first cold status response can remain stale, and this external PR has no real behavior proof on its head.

## What This Changes

The PR changes context-window lookup, Gateway startup, status summary/text call sites, and tests so status paths trigger background model context-window cache warming.

## Best Possible Solution

Make first cold status deterministic by preparing or awaiting bounded read-only context metadata before formatting, keep catalog loading behind lazy boundaries, and prove the fresh-start first-status output.

## Reproduction Assessment

Yes, source-reproducible for the PR blocker: PR-head code starts warmup with void and the added test expects the first status-style lookup to remain 200_000 until async warmup is flushed.

## Solution Assessment

No. Delegating status resolution to the shared resolver is useful, but fire-and-forget warmup plus static catalog imports is only a plausible mitigation and does not satisfy deterministic first-call status correctness.
