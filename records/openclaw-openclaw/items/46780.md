## Summary

Keep open. Current main still has tool, agent, message, session, gateway, dispatch, and install hook surfaces, but no skill execution lifecycle boundary; the enterprise permission-check comment makes this a real plugin API and security-design gap rather than stale cleanup or ClawHub-only work.

## What This Changes

The issue requests before/after or success/failure lifecycle hooks for skill invocation so plugins or hooks can log, analyze, monitor, and enforce policy around skills.

## Best Possible Solution

Add an additive, documented typed plugin hook surface around explicit skill invocation: a pre-invocation decision hook with allow/block/approval semantics and a post-invocation observation hook with redacted context, dispatch mode, outcome, duration, and failure summary.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug; this is a feature/API gap. Static inspection of the hook catalogs, hook runner, internal hook event types, skill metadata, docs, and slash-skill dispatch path gives high confidence that the requested lifecycle boundary is absent on current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear for the proposed JSON script-hook shape. A typed plugin API with explicit blocking, approval, timeout, ordering, and redaction semantics is the safer maintainable direction, especially for the enterprise permission-check use case raised in discussion.
