## Summary

Keep open. Current main has typed plugin hooks for tools, agent turns, messages, sessions, gateway lifecycle, install policy, and dispatch, plus internal script hooks for command/session/agent/gateway/message events, but it still lacks a skill execution lifecycle boundary for before/after invocation or success/failure observation. The discussion adds a concrete enterprise permission-check use case, so this is a real plugin API/security design gap rather than stale cleanup or ClawHub-only work.

## What This Changes

The issue requests before/after or success/failure lifecycle hooks for skill invocation so hooks or plugins can log, analyze, monitor, and enforce policy around skills.

## Best Possible Solution

Add an additive, documented typed plugin hook surface around explicit skill invocation: a pre-invocation decision hook that can allow, block, or require approval with bounded reason metadata, plus a post-invocation observation hook carrying redacted skill name, invocation source, agent/session/run context, dispatch mode, outcome, duration, and failure summary. If maintainers want model-selected skill tracking too, define that separately because prompt-visible skills do not always have a deterministic execution boundary.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug; this is a feature/API gap. Static inspection of the hook catalogs, hook runner, internal hook event types, skill metadata, docs, and slash-skill dispatch path gives high confidence that the requested lifecycle boundary is absent on current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear for the proposed JSON script-hook shape. A typed plugin API with explicit blocking, approval, timeout, ordering, and redaction semantics is the safer maintainable direction, especially for the enterprise permission-check use case raised in discussion.
