## Summary

Keep open. Current main still lacks a skill execution lifecycle hook boundary, and the enterprise permission-check comment makes this a real plugin API and security-design question rather than stale cleanup or ClawHub-only work.

## What This Changes

The issue requests before/after or success/failure hooks around skill invocation for logging, analytics, monitoring, debugging, and policy enforcement.

## Best Possible Solution

Add an additive, documented typed plugin hook surface around explicit skill invocation, with a pre-invocation decision hook and a post-invocation observation hook carrying redacted context, dispatch mode, outcome, duration, and failure summary.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug; static inspection of the current hook catalogs, hook runner, internal hook event types, skill metadata, docs, and slash-skill dispatch path gives high confidence that the requested lifecycle boundary is absent.

## Solution Assessment

Is this the best way to solve the issue? Unclear for the proposed JSON script-hook shape; a typed plugin API with explicit blocking, approval, timeout, ordering, and redaction semantics is the safer maintainable direction.
