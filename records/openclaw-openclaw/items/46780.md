## Summary

Keep open. Current main still exposes tool-call and install-policy hooks but not per-skill before/after execution or success/failure events with skill attribution; the related diagnostics PR is open and observability-only, while the setup-hook issue covers install/update rather than invocation-time gating.

## What This Changes

The issue requests skill lifecycle hooks such as beforeExecute/afterExecute or skill invoke/success/fail events for logging, analytics, debugging, monitoring, and permission checks.

## Best Possible Solution

Add a versioned plugin/API lifecycle contract for skill invocation and outcomes, with explicit skill identity, blocking semantics, success/failure phases, privacy boundaries, and a documented relationship to existing tool hooks and diagnostics events.

## Reproduction Assessment

Not applicable. This is a feature/API request; source and docs inspection on current main give a high-confidence check that the requested skill lifecycle hook names and payload contract are absent.

## Solution Assessment

Unclear. A first-class plugin lifecycle hook may be the right direction, but maintainers need to decide event timing, skill attribution, failure semantics, permission behavior, and overlap with diagnostics telemetry before implementation.

## Risks / Open Questions

- Skill invocation is not one uniform runtime path today: skills may be prompt-only, user-invocable, native-harness-backed, or tool-dispatched, so the event boundary needs careful definition.
- A before-execute hook used for permissions becomes a security boundary and needs explicit failure, timeout, data exposure, and redaction semantics.
- The open diagnostics PR may help analytics but does not satisfy the permission-gating use case unless maintainers deliberately expand it into a blocking lifecycle API.
