## Summary

Keep open. Current main still has tool-, run-, install-, session-, and gateway-scoped hooks, but no per-skill before/after execution hook or success/failure event with reliable skill attribution; the related OTel and setup-hook threads cover different lifecycle moments and do not close the requested authorization boundary.

## What This Changes

The issue requests before/after skill execution lifecycle hooks and success/failure attribution for logging, analytics, debugging, monitoring, and permission checks.

## Best Possible Solution

Define a versioned skill execution lifecycle contract with skill identity, invocation source, blocking semantics, success/failure phases, privacy boundaries, and a documented relationship to tool hooks, diagnostics, and permission manifests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs inspection gives a high-confidence current-main check that the requested skill execution hook names and payload contract are absent.

## Solution Assessment

Unclear. A first-class plugin lifecycle hook may be the right direction, but maintainers need to decide execution boundaries, skill attribution, blocking behavior, failure semantics, and overlap with diagnostics before implementation.

## Risks / Open Questions

- A before-execute hook used for permission checks becomes a security boundary and needs explicit fail-closed/fail-open, timeout, redaction, and payload semantics.
- Skill use is not a single runtime path today: skills can be prompt-discovered instructions, slash commands, direct tool-dispatched commands, plugin-shipped skills, or harness-native inputs.
- The open diagnostics PR may help analytics and tracing, but it would not by itself provide blocking authorization or a public skill lifecycle hook contract.
