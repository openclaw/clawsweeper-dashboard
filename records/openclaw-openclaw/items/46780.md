## Summary

Keep open. Current main now has bounded `skill.used` diagnostics, but it still does not provide a skill-attributed before/after execution hook, success/failure lifecycle event, or blocking skill authorization contract; the related setup-hook and telemetry work do not supersede this execution-lifecycle API decision.

## What This Changes

The issue requests before/after skill execution hooks or skill invoke/success/fail events for logging, analytics, debugging, monitoring, and possible authorization checks.

## Best Possible Solution

Define a first-class, versioned skill execution lifecycle contract with reliable skill identity, invocation source, blocking semantics, success/failure phases, timeout/error behavior, privacy boundaries, and documented overlap with tool hooks, diagnostics, setup hooks, runtime events, and permission manifests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs inspection on current main verify the requested hook names, skill identity payload, and before/after outcome contract are absent.

## Solution Assessment

Unclear. A first-class plugin lifecycle event may be the right direction, but maintainers need to decide attribution, blocking behavior, failure semantics, privacy, and overlap with diagnostics/runtime events before implementation.
