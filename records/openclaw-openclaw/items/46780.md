## Summary

Keep open. Current main has tool-, install-, run-, session-, gateway-, and generic runtime event surfaces, but it still lacks a reliable per-skill before/after execution hook, success/failure lifecycle event, or blocking skill authorization boundary with skill attribution.

## What This Changes

The issue requests skill before/after execution lifecycle hooks, or equivalent skill invoke/success/failure events, for analytics, debugging, monitoring, self-improvement, and permission enforcement.

## Best Possible Solution

Define a first-class, versioned skill execution lifecycle contract with reliable skill identity, invocation source, blocking semantics, success/failure phases, timeout/error behavior, privacy boundaries, and a documented relationship to diagnostics, tool hooks, setup hooks, runtime events, and permission manifests.

## Reproduction Assessment

Not applicable as a bug reproduction. Source and docs inspection give a high-confidence current-main absence check for the requested skill execution hook names, skill identity payload, and before/after outcome contract.

## Solution Assessment

Unclear. A first-class plugin lifecycle event may be the right direction, but maintainers need to decide attribution, blocking behavior, failure semantics, privacy, and overlap with diagnostics/runtime events before implementation.

## Risks / Open Questions

- A before-execute skill hook used for permissions would become a security boundary, so fail-closed behavior, timeout handling, redaction, and exposed prompt/env context need explicit design.
- Skill use is not one uniform runtime path today: skills can be prompt instructions, slash commands, command-dispatched tools, plugin-shipped skills, or harness-native inputs, so a shallow before_tool_call wrapper could miss or misattribute execution.
- The related diagnostics PR may help observability if it lands, but telemetry alone would not satisfy blocking authorization or reliable lifecycle semantics.
