## Summary

Keep open: current main has adjacent tool, run, install, session, and gateway hooks, but no per-skill before/after execution hook or success/failure event with reliable skill attribution. The related OTel PR and setup-hook issue cover different lifecycle moments, so this still needs maintainer product and security review.

## What This Changes

The issue requests before/after skill execution lifecycle hooks or plugin events so operators can log, monitor, analyze, or gate skill use.

## Best Possible Solution

Define a narrow public plugin-hook contract for skill selection/invocation only if maintainers decide skill execution should become a first-class runtime event; keep existing tool/install hooks as adjacent surfaces and require fail-closed security semantics before adding script-style config hooks.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source and docs inspection gives a high-confidence current-main check that the requested skill execution hook surface is absent.

## Solution Assessment

Unclear until maintainers choose the product/security contract. Existing tool and install hooks solve adjacent problems, but they are not a complete substitute for skill-scoped before/after execution events.
