## Summary

Keep open. The report is new against v2026.5.2 and current main still constructs the PDF tool during `openclaw-tools` prep when PDF availability is detected, but source inspection suggests `pdfjs-dist` itself is only imported inside extraction execution, so the remaining fix should profile and defer the confirmed PDF factory hot path rather than assume the import site.

## What This Changes

Reduce per-agent-run `pdf` tool staging overhead by deferring expensive PDF setup until a PDF is actually used.

## Best Possible Solution

Make PDF tool registration a lightweight descriptor or lazy wrapper, keep `pdfjs-dist` and document-extract loading inside fallback execution, and add focused regression coverage for staging-time imports/timing.

## Reproduction Assessment

No, not at high confidence from this review alone. The issue has concrete v2026.5.2 timing data and current source still runs `createPdfTool` during staging, but I did not live-reproduce the multi-second delay and current source does not prove `pdfjs-dist` imports before first PDF use.

## Solution Assessment

Unclear. Truly lazy-loading `pdfjs-dist` appears to be the intended current structure already, so the best repair is to profile `createPdfTool`/model resolution first and defer only the confirmed staging hot path; adding a new PDF enabled flag is a broader fallback, not the first fix.
