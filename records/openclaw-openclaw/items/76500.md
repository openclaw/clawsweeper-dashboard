## Summary

Keep open. The v2026.5.2 traces point to a fresh, narrow `pdf` tool prep hotspot, but current source does not prove the specific `pdfjs-dist` eager-import diagnosis; the next maintainer step should split timing inside PDF factory/model-resolution/document-extractor loading before changing tool availability or config policy.

## What This Changes

The issue asks OpenClaw to remove roughly 3.5s of per-agent-run PDF tool initialization overhead by making PDF extraction setup truly lazy or disableable.

## Best Possible Solution

Add focused PDF factory substage timing, then make only the proven expensive part lazy or cached while preserving the documented PDF tool availability contract.

## Reproduction Assessment

Source-reproducible for the eager PDF factory path, not high-confidence for the `pdfjs-dist` subclaim. The issue has repeated v2026.5.2 traces and current main still constructs the PDF tool during `core-plugin-tool` prep, but source inspection does not prove `pdfjs-dist` is imported during registration.

## Solution Assessment

Unclear. Truly lazy `pdfjs-dist` loading is the right shape if that import is actually on the prep path, but current source points first to measuring PDF factory/model-resolution/document-extractor substages and then removing the confirmed cost.
