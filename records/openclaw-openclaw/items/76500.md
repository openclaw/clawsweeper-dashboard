## Summary

Keep open. Current main still constructs the PDF tool during `openclaw-tools` prep and resolves PDF model config before any PDF is used, while the cited `pdfjs-dist` import remains execution-only in source, so the remaining work is a narrow performance fix after profiling the actual staging hot path.

## What This Changes

The issue requests reducing unused PDF tool startup overhead in the `core-plugin-tool stages` path, especially around `pdf-tool` staging and PDF.js loading.

## Best Possible Solution

Make PDF tool registration lightweight, defer PDF model resolution and document extraction work until execution, and keep `pdfjs-dist` inside the fallback path for actual PDF processing.

## Reproduction Assessment

No high-confidence live reproduction was established. There is a source-level current-main path: `createOpenClawTools` still constructs `createPdfTool` during tool prep and `createPdfTool` resolves model config before execution, matching the reported `pdf-tool` staging bucket.

## Solution Assessment

Unclear as stated. Lazy-loading `pdfjs-dist` appears to be the intended current structure already, so the best repair is to profile `createPdfTool` and defer the confirmed registration-time work, likely following the image tool's deferred model-resolution pattern.
