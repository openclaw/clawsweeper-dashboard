## Summary

Keep open. Current main still constructs the PDF tool during `openclaw-tools:pdf-tool` prep and resolves PDF model config before any PDF is used; related Windows reports corroborate the same staging surface, but current source suggests the actual `pdfjs-dist` import is still inside execution-time extraction fallback, so the repair should profile and defer the confirmed hot path.

## What This Changes

The issue requests reducing unused PDF tool startup overhead in the `core-plugin-tool stages` path, especially around `pdf-tool` staging and `pdfjs-dist` loading.

## Best Possible Solution

Make PDF tool registration a lightweight descriptor or lazy wrapper, defer model resolution and document extraction work until execution, and add regression coverage proving registration does not load extraction fallback code.

## Reproduction Assessment

No high-confidence live reproduction was established. There is a source-level reproduction path for the staging surface because current main still calls `createPdfTool` during `openclaw-tools:pdf-tool` prep and resolves model config before execution, matching the issue's timing bucket.

## Solution Assessment

Unclear whether the suggested `pdfjs-dist` root cause is exact. The narrow maintainable fix is to profile the PDF factory path and defer the confirmed expensive registration work; adding a new PDF enabled flag should be a fallback product choice, not the first repair.
