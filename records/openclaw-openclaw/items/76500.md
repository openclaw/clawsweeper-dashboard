## Summary

Keep open. Current main still constructs the `pdf` tool during OpenClaw tool preparation when PDF availability is detected, and the canonical duplicate comments on #76607/#76637 point back here; however, source inspection still does not prove the reported `pdfjs-dist` dynamic import itself happens before first PDF execution.

## What This Changes

The issue asks OpenClaw to remove per-agent-run `pdf` tool staging overhead by keeping PDF/document extraction setup lazy until actual PDF use.

## Best Possible Solution

Keep this as the canonical PDF staging performance bug, make PDF registration a lightweight lazy wrapper, and defer model/document-extract/`pdfjs-dist` work until actual PDF execution with focused regression coverage.

## Reproduction Assessment

No high-confidence live reproduction was established here. The issue and duplicate reports provide concrete v2026.5.2 timing data, and current source still shows the shared PDF factory path during tool prep, but the specific eager `pdfjs-dist` import remains unproven by source inspection.

## Solution Assessment

Unclear for the exact root-cause claim. The narrow maintainable fix is to profile the `openclaw-tools:pdf-tool` stage and make the confirmed factory/model/document-extract work lazy; adding a new PDF enabled flag or pre-warm policy is broader than the first repair.
