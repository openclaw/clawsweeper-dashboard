## Summary

Keep open. Current main still constructs the PDF tool during `openclaw-tools` prep and resolves PDF model config before any PDF is used, but source inspection does not support the stated `pdfjs-dist` eager-import root cause, so the fix should profile and defer the confirmed PDF factory hot path.

## What This Changes

The issue requests reducing unused PDF tool startup overhead in the `core-plugin-tool stages` path, especially around `pdf-tool` staging and `pdfjs-dist` loading.

## Best Possible Solution

Make PDF tool registration a lightweight descriptor or lazy wrapper, keep `pdfjs-dist` and document-extract loading inside execution-only fallback paths, and add focused regression coverage for staging-time imports or timing.

## Reproduction Assessment

No high-confidence live reproduction was established. There is a source-level recognition path: current main still calls `createPdfTool` during tool prep and resolves model config before execution, while the reported timing data identifies the same `pdf-tool` stage.

## Solution Assessment

Unclear. Lazy-loading `pdfjs-dist` appears to be the intended current structure already, so the best repair is to profile `createPdfTool` and related factory dependencies first, then defer only the confirmed staging hot path; a new PDF enabled flag is broader than the first fix should be.
