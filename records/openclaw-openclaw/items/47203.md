## Summary

Keep open. Current main still drops navigate timeout/load-state controls before the browser route and Playwright call, so the reported 20s browser navigate timeout race remains a valid browser plugin bug; related #72653 is broader and open, but this issue is the more specific canonical report.

## What This Changes

The issue reports that browser navigate times out on Linux VNC because navigate does not propagate load-state or timeout controls through the browser tool path.

## Best Possible Solution

Add a focused browser plugin fix that validates and propagates navigate timeout/load-state controls through the tool, CLI/client, proxy route, and Playwright navigation guard while preserving SSRF checks and current default semantics unless an explicit option is provided.

## Reproduction Assessment

Yes. The issue provides concrete Linux VNC steps, and current-main source inspection gives a high-confidence unit-level reproduction: pass `timeoutMs` or `loadState` to browser navigate and observe both are dropped before the route or `page.goto`.

## Solution Assessment

Yes for a narrow pass-through and timeout-budget repair. A raw-CDP fallback or default wait-state change may be useful later, but it is broader than needed to fix the demonstrated option-drop and timeout-race path.
