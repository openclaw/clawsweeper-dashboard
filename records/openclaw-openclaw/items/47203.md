## Summary

Keep open. Current main and the latest release still leave the browser navigate path with a fixed 20s outer/client budget and no forwarding of navigate timeout or wait-state controls, so the Linux VNC timeout report remains actionable.

## What This Changes

The issue asks for `browser navigate` to stop timing out on Linux VNC when the browser control service itself is reachable and direct navigation works.

## Best Possible Solution

Fix the browser plugin navigate stack so caller timeout and supported wait controls reach the client, proxy, route, and Playwright/Chrome MCP helpers with an outer timeout that cannot race the inner navigation deadline.

## Reproduction Assessment

Yes, at source level. The issue gives concrete Linux VNC steps and current main independently shows the navigate path dropping timing/wait controls before the fixed 20s client timeout and default Playwright load wait; I did not run the live VNC environment.

## Solution Assessment

Yes for a narrow fix. Threading normalized navigate timeout data through the existing browser plugin seams and adding a client-side slack budget is more maintainable than raising global timeouts or switching to raw CDP by default.
