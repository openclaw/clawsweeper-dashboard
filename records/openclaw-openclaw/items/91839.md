## Summary

The reported blast radius is not resolved by the related open issues: current main still treats unavailable-model wording as neither a permanent announce-delivery error nor a narrow terminal completion outcome, and completion delivery can combine internal retries with registry retries and history reconciliation. The gateway-wide starvation itself still needs a focused reproduction or RPC-rate trace, so this should remain open as the cross-surface availability report rather than be closed into one narrower sibling.

## What This Changes

Prevent a terminal unavailable-model failure in a subagent run from amplifying through completion announce and history activity into gateway event-loop starvation.

## Best Possible Solution

Represent an unavailable-model child result as a structured terminal outcome, deliver exactly one bounded failure completion to the parent, and add a regression test that caps announce attempts and `chat.history` calls while proving unrelated health RPCs remain serviceable.

## Reproduction Assessment

Partly yes. Current main clearly contains the reported terminal-error classification gap, nested announce retry surfaces, and server-side history reads, but the full event-loop starvation has not been reproduced with a controlled provider or RPC-rate trace.

## Solution Assessment

No, not as one combined change. The narrow runtime fix should bound terminal completion propagation and history work; model-reference validation and service watchdog recovery should remain separately owned defenses in https://github.com/openclaw/openclaw/issues/84212 and the systemd recovery track.
