## Summary

Keep open. Current main and the latest release still use stable WhatsApp group JIDs for group config matching, and the open linked PR is the active candidate for the requested default-safe name-matching behavior.

## What This Changes

The issue asks for `channels.whatsapp.groups` entries to match either the stable WhatsApp group JID or the exact current WhatsApp group subject/name, with JID keys taking precedence.

## Best Possible Solution

Keep this issue open while maintainers review https://github.com/openclaw/openclaw/pull/91180; land a default-off, JID-first, audited opt-in implementation or explicitly decide to keep WhatsApp group config JID-only.

## Reproduction Assessment

Yes, at source level. Current main captures `groupSubject` from WhatsApp metadata but group admission, require-mention defaults, tool policy, and group system prompts resolve from stable group IDs only.

## Solution Assessment

Unclear until maintainer review. A WhatsApp-local, default-off, JID-first opt-in with dangerous-flag auditing looks like the safest shape; unconditional mutable-name authorization would not be the best solution.
