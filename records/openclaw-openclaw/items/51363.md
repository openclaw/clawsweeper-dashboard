## Summary

Keep open. Current main and latest release still derive Docker and sandbox-browser container names from the sandbox scope key before adding any instance/workspace identity, while the linked fix PR was closed unmerged and its commits are not contained in main or any tag checked.

## What This Changes

The issue reports Docker sandbox container name collisions across co-hosted OpenClaw instances that share the default agent key, allowing exec operations to use another instance's mounted workspace.

## Best Possible Solution

Land a focused sandbox-runtime fix that includes stable workspace or instance identity in Docker and browser container slugs for shared/agent scopes, preserves session-scope stability, and adds regression coverage for two workspaces using the same agent scope.

## Reproduction Assessment

Yes, source-reproducible. Current main computes Docker and browser container names from `shared` or the agent/session scope key, while different `workspaceDir` values only affect mounts/config hashes after the name is chosen; I did not run a live Docker repro in this read-only review.

## Solution Assessment

Yes for the likely implementation shape, but no for closing this issue. The narrowest maintainable fix is to add workspace identity to shared/agent Docker and browser container slugs with explicit maintainer approval for upgrade cleanup behavior.
