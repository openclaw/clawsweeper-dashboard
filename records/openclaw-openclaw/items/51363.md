## Label Justifications

- `P1`: The report describes sandboxed exec operations reading or writing another co-hosted instance's workspace, which can affect real multi-instance users now.
- `impact:data-loss`: A reused sandbox container can write into the wrong instance workspace and contaminate files.
- `impact:security`: The bug weakens the Docker sandbox boundary by reusing a container mounted for a different OpenClaw instance.
- `impact:session-state`: Using the wrong mounted workspace can make an agent operate on another instance's session and workspace context.

## Summary

Keep open. Current main and the latest release still derive Docker and sandbox-browser container names from the sandbox scope key before instance/workspace identity participates, while the direct fix PR closed unmerged and the broader multi-instance issue does not own this narrow isolation defect.

## What This Changes

The issue asks for Docker sandbox container names to include stable instance identity so multiple HOME-isolated OpenClaw instances on one host do not share a container.

## Best Possible Solution

Patch Docker and browser sandbox naming so shared/agent scopes include stable instance or workspace identity before Docker lookup, preserve intentional session-scope stability, and add regression plus upgrade proof.

## Reproduction Assessment

Yes, source-reproducible. Current main derives Docker and browser container names from `shared` or `agent:main` before workspace identity is considered, so two HOME-isolated default-agent instances still converge on the same name.

## Solution Assessment

Unclear that the exact HOME-hash proposal is the final best fix. The best fix should use stable identity already available to the sandbox layer, cover both Docker and browser containers, and make session-scope plus upgrade behavior explicit.
