## Label Justifications

- `P1`: The report describes sandboxed exec operations reading or writing another co-hosted instance's workspace, which can affect real multi-instance users now.
- `impact:data-loss`: A reused sandbox container can write into the wrong instance workspace and contaminate files.

## Summary

Keep open. Current main and the latest release still derive Docker and sandbox-browser container names from the sandbox scope key before instance/workspace identity participates, while the direct fix PR closed unmerged and the broader multi-instance issue is only adjacent.

## What This Changes

The issue reports that co-hosted HOME-isolated OpenClaw instances using the default agent sandbox scope can derive the same Docker sandbox container name and share a mounted workspace.

## Best Possible Solution

Patch Docker and sandbox-browser naming so shared and agent scopes include stable instance/workspace identity, keep session-scope naming intentional, and add regression plus real two-HOME Docker proof.

## Reproduction Assessment

Yes, source-reproducible. Current main derives Docker and browser container names from `shared` or `agent:main` before workspace identity participates, so two HOME-isolated default-agent instances still converge on the same name.

## Solution Assessment

Unclear that the exact HOME-hash proposal is the final best fix. The best fix should use stable identity already available to the sandbox layer, cover both Docker and browser containers, and make session-scope plus upgrade behavior explicit.
