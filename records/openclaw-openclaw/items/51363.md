- `impact:security`: The issue is about Docker sandbox isolation failing across co-hosted OpenClaw instances.
- `impact:data-loss`: The reported collision can route exec reads and writes into the wrong instance workspace.
- `impact:session-state`: Cross-instance workspace reuse can contaminate the session context and state the agent is operating on.

## Summary

Keep open. Current main and the latest release still derive Docker and sandbox-browser container names from the sandbox scope key only, and the related fix PR was closed unmerged, so co-hosted HOME-isolated instances using the default agent can still collide.

## What This Changes

The issue requests unique Docker sandbox container names per co-hosted OpenClaw instance so HOME-isolated instances using the default agent do not share one container.

## Best Possible Solution

Add a stable per-instance discriminator to Docker and sandbox-browser container-name inputs for agent and shared scopes, preserve session-scope naming, and cover the migration behavior in tests and release notes.

## Reproduction Assessment

Yes from source inspection, though not from a live Docker run. Default agent scope resolves to agent:main, slugifies to agent-main-f331f052, and Docker/browser container names ignore workspaceDir.

## Solution Assessment

Yes on direction, but the exact discriminator and upgrade behavior need maintainer approval. The narrow maintainable fix is to include stable per-instance identity for agent/shared scopes while preserving session-scope names.
