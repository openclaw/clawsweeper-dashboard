## Label Justifications

- `P1`: The report describes sandboxed exec operations reading or writing another co-hosted instance's workspace, which can affect real multi-instance users now.
- `impact:data-loss`: A reused sandbox container can write into the wrong instance workspace and contaminate files.

- `impact:session-state`: Using the wrong mounted workspace can make an agent operate on another instance's session and workspace context.

## Summary

Keep open: current main and v2026.6.8 still derive Docker and sandbox-browser container names from the sandbox scope key before instance or workspace identity participates, while the direct fix PR closed unmerged and the broader multi-instance feature only partially overlaps.

## What This Changes

The issue asks for Docker sandbox container names to include instance identity so co-hosted OpenClaw instances with the default agent do not share a container.

## Best Possible Solution

Patch Docker and browser sandbox naming so shared/agent scopes include stable instance or workspace identity before Docker lookup, preserve session-scope stability, and add regression plus upgrade proof.

## Reproduction Assessment

Yes, source-reproducible. Current main derives Docker and browser container names from `shared` or `agent:main` before workspace identity is considered, so two HOME-isolated default-agent instances still converge on the same name.

## Solution Assessment

Unclear until maintainer review. Including stable workspace or instance identity in shared/agent sandbox names is the narrow likely fix, but maintainers should confirm the identity source and upgrade/orphan-container behavior before repair.
