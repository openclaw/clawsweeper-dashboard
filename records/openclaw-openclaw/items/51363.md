## Label Justifications

- `P1`: The source-proven bug can make co-hosted OpenClaw instances share an exec sandbox container and operate on the wrong workspace.
- `impact:data-loss`: A shared sandbox container can route writes from one instance into another instance's workspace.
- `impact:security`: The issue weakens the Docker sandbox isolation boundary between separately configured instances on the same host.
- `impact:session-state`: Wrong-container reuse can expose or mutate another instance's workspace, agent context, and session files.

## Summary

Keep open. Current main and v2026.6.10 still derive Docker and sandbox-browser runtime names from shared/agent scope keys before workspace identity participates, while https://github.com/openclaw/openclaw/pull/96866 is an open candidate fix rather than a merged implementation.

## What This Changes

The issue reports Docker sandbox container name collisions across co-hosted OpenClaw instances that reuse the default agent key, causing cross-instance sandbox workspace reuse.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/96866 or an equivalent security-reviewed fix that scopes non-shared sandbox runtime identity by resolved workspace across Docker, browser, SSH, recreate commands, docs, and regression tests.

## Reproduction Assessment

Yes, by source inspection: current main derives Docker and browser container names from shared/agent scope keys without workspace identity, and a running same-name container with mismatched hash can be kept as hot. I did not run a live Docker two-HOME repro in this read-only review.

## Solution Assessment

Yes on direction, but it needs maintainer/security review: workspace-scoping the sandbox lifetime key is a better owner-boundary fix than a Docker-container-name-only patch because it covers sibling runtime identity, browser, SSH, and recreate behavior.
