## Label Justifications

- `P1`: The source-backed bug affects a real sandbox isolation boundary for co-hosted OpenClaw instances.
- `impact:security`: The issue is about Docker sandbox isolation failing across instances that share one Docker daemon.
- `impact:data-loss`: The collision can route exec reads and writes into the wrong instance workspace.
- `impact:session-state`: Cross-instance workspace reuse can contaminate the session context and state an agent operates on.

## Summary

Keep open. Current main and the latest shipped tag still build Docker and sandbox-browser runtime names from the sandbox scope key, while the related fix PR was closed unmerged, so co-hosted HOME-isolated instances can still collide.

## What This Changes

The issue asks OpenClaw to make Docker sandbox container names unique per co-hosted instance so HOME-isolated instances using the default agent do not share one runtime.

## Best Possible Solution

Add a stable per-instance discriminator to Docker and sandbox-browser names for agent/shared scopes, preserve session-scope naming, and cover migration behavior in tests and release notes.

## Reproduction Assessment

Yes from source inspection, though not from a live Docker run. Default agent scope resolves to agent:main, slugifies to agent-main-f331f052, and Docker/browser container names ignore workspaceDir.

## Solution Assessment

Yes on direction, but the exact discriminator and upgrade behavior need maintainer approval. The narrow maintainable fix is to include stable per-instance identity for agent/shared scopes while preserving session-scope names.
