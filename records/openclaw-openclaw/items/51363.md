## Label Justifications

- `P1`: A source-backed sandbox runtime collision can break real co-hosted agent workflows and expose one instance workspace to another.
- `impact:data-loss`: The reported collision can route exec reads and writes into the wrong mounted workspace.
- `impact:security`: The issue concerns sandbox isolation between configured OpenClaw instances on the same Docker host.
- `impact:session-state`: Sharing one sandbox runtime across instances can mis-associate agent workspace and session context.

## Summary

Keep open. Current main and the latest release still derive Docker and sandbox-browser runtime names from the sandbox scope key rather than an instance identity; workspace identity only changes the config hash, and a running hash-mismatched container can still be treated as hot and reused.

## What This Changes

The issue asks OpenClaw to make Docker sandbox container names unique per co-hosted instance so separate homes using the default agent do not share one runtime.

## Best Possible Solution

Add stable instance identity to Docker and sandbox-browser runtime names for shared and agent scopes, or fail closed on foreign-instance hash mismatches, with upgrade cleanup expectations and regression proof for two distinct homes using the same agent key.

## Reproduction Assessment

Yes, source-reproducible. Two HOME-isolated instances using the default agent scope still resolve to the same scope-derived Docker and browser container names, and a running mismatched hash can be treated as hot instead of rejected.

## Solution Assessment

No, current main is not the best solution yet. It detects workspace differences through the config hash but leaves the collision-prone runtime name unchanged, so the fix should move instance identity into naming or explicitly reject foreign containers.
