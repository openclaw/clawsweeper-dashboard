## Label Justifications

- `P1`: A source-backed sandbox collision can break real co-hosted agent workflows and expose one instance's workspace to another.
- `impact:data-loss`: The reported collision can route exec reads and writes into the wrong mounted workspace.
- `impact:security`: The issue concerns sandbox isolation between configured OpenClaw instances on the same Docker host.
- `impact:session-state`: Sharing one sandbox runtime across instances can mis-associate agent workspace and session context.

## Summary

Keep open. Current main and the latest release still derive Docker and sandbox-browser container names from the sandbox scope key, while workspace identity only affects the config hash; a running hash-mismatched container is treated as hot and reused, so the reported cross-instance sandbox collision remains source-reproducible.

## What This Changes

Docker sandbox containers should be named with an instance discriminator so co-hosted OpenClaw instances using the same default agent key do not share one runtime.

## Best Possible Solution

Add a stable instance identity to both Docker exec and sandbox-browser runtime names, or fail closed on foreign-instance hash mismatches, with regression coverage for two distinct workspaces using the same agent-scoped key.

## Reproduction Assessment

Yes. Source inspection shows two instances with the same agent-scoped key still compute the same Docker and browser container names, and a running mismatched container logs a recreate hint rather than refusing reuse.

## Solution Assessment

No, current main is not the best solution yet. It detects workspace differences through the config hash but leaves the collision-prone name unchanged, so the fix should move instance identity into the runtime name or explicitly fail closed on foreign containers.
