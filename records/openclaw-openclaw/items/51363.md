## Summary

Keep open. Current main and v2026.6.8 still build Docker and sandbox-browser container names from the sandbox scope key before instance or workspace identity participates, while the direct fix PR closed unmerged and the broader multi-instance issue only partially overlaps this narrower isolation bug.

## What This Changes

The issue reports Docker sandbox container-name collisions across multiple HOME-isolated OpenClaw instances using the default agent key, causing sandboxed exec to reuse another instance's mounted workspace.

## Best Possible Solution

Patch Docker and browser sandbox naming so agent/shared scopes include stable instance or workspace identity before Docker lookup, preserve intentional session-scope stability, and add regression plus upgrade proof.

## Reproduction Assessment

Yes, source-reproducible. Current main derives Docker and browser container names from `shared` or `agent:main` before workspace identity is considered, so two HOME-isolated default-agent instances still converge on the same name.

## Solution Assessment

Unclear for the exact HOME-hash proposal. The best fix should use stable identity already available to the sandbox layer, cover both Docker and browser containers, preserve session-scope stability, and make upgrade/orphan behavior explicit.
