## Summary

Keep open. Current main still derives Docker sandbox and sandbox-browser container names from the sandbox scope key only, so co-hosted instances using the default main agent can still collide; the linked fix PR is closed unmerged.

## What This Changes

The issue requests unique Docker sandbox container names per OpenClaw instance so multiple HOME-isolated instances on the same host do not share one sandbox container.

## Best Possible Solution

Implement the fix in the core Docker sandbox owner module by mixing a stable instance identity, such as resolved OpenClaw state/workspace root or HOME-derived identity, into both Docker and sandbox-browser container-name slug inputs while preserving per-scope reuse within one instance. Add regression tests for two different instance roots with the same default main agent and document the one-time container recreate/orphan behavior.

## Reproduction Assessment

Yes. With sandbox mode all and default agent scope, two HOME-isolated instances using the default main agent both resolve to agent:main and current code slugifies only that value into the same container name.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix, but it should cover both src/agents/sandbox/docker.ts and src/agents/sandbox/browser.ts and avoid dynamic discriminators like gateway port.
