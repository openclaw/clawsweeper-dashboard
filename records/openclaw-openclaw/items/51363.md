## Label Justifications

- `P1`: The report describes a sandbox isolation regression where exec operations can read or write another co-hosted instance's workspace, affecting real multi-instance users.
- `impact:data-loss`: A shared container can write into the wrong instance workspace, creating cross-instance file contamination.
- `impact:security`: The bug weakens the Docker sandbox boundary by reusing a container mounted for a different OpenClaw instance.
- `impact:session-state`: Using the wrong mounted workspace can make an agent operate on another instance's session and workspace context.

## Summary

Keep open. Current main and v2026.6.8 still choose Docker and browser sandbox container names from the sandbox scope slug before workspace identity is considered; the direct fix PR was closed unmerged, and the broader multi-instance feature does not cover this narrow sandbox isolation bug.

## What This Changes

Ensure Docker sandbox containers are uniquely named across co-hosted OpenClaw instances with different HOME/workspace roots so exec and browser sandbox state cannot be shared accidentally.

## Best Possible Solution

Patch Docker and browser sandbox runtime naming so non-session shared/agent scopes include a stable instance identity before Docker lookup, preserve intentional session-scope stability, and add regression coverage for two workspace roots using the default main agent.

## Reproduction Assessment

Yes, source-reproducible. Current main derives Docker and browser container names from `shared` or `agent:main` before workspace identity is considered, so two HOME-isolated default-agent instances still converge on the same container name.

## Solution Assessment

Unclear that the exact HOME-hash proposal is the final best fix. The best fix should use the stable instance/workspace identity already available to the sandbox layer, cover both Docker and browser containers, and make the session-scope and upgrade behavior explicit.
