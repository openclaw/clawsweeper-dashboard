## Label Justifications

- `P1`: The source-proven bug can make co-hosted OpenClaw instances share an exec sandbox container and operate on the wrong workspace.
- `impact:data-loss`: A shared container mount can route writes from one instance into another instance's workspace.
- `impact:security`: The bug weakens the Docker sandbox isolation boundary between separately configured instances on the same host.
- `impact:session-state`: The wrong workspace/container can expose or mutate another instance's agent context and session files.

## Summary

Keep open. Current main and v2026.6.10 still derive Docker and sandbox-browser container names from shared/agent scope keys before instance identity participates, and the direct fix PR is closed unmerged while the broader multi-instance issue is only adjacent.

## What This Changes

The issue requests per-instance Docker sandbox and browser container names so co-hosted OpenClaw instances with the same default agent key do not share one container.

## Best Possible Solution

Add instance identity to shared/agent Docker and browser sandbox container slugs, keep session-scope names stable unless intentionally changed, and cover same-agent/different-workspace collisions with focused tests.

## Reproduction Assessment

Yes by source inspection: shared/agent scope names are independent of workspace identity, and a running foreign container with no local registry timestamp is kept as hot rather than recreated. I did not run Docker because this review is read-only, but current and release source match the reported two-HOME reproduction.

## Solution Assessment

Unclear until maintainer/security review decides the upgrade behavior. The best narrow shape appears to be the closed PR's corrected direction: include workspace identity for shared/agent scope, avoid unnecessary session-scope renames, and add browser coverage too.
