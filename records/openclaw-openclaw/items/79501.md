## Summary

Keep open: this is a recent feature PR that is not implemented on current main and still has a source-reproducible runner integration bug. It also needs maintainer product judgment because model-driven proactive outreach changes core heartbeat behavior.

## What This Changes

The PR adds an ambient heartbeat mode, a maxMessages config key, an ambient prompt helper, runner wiring, schema/docs updates, and focused prompt/schema tests.

## Best Possible Solution

If maintainers want ambient mode in core, make heartbeat.mode="ambient" select the structured response-tool/message-tool path before legacy prompt selection, update generated config docs, add runner-level coverage, and require real scheduled-run proof.

## Reproduction Assessment

Not applicable as a current-main bug; this is a feature PR. The main blocker is source-reproducible from prompt selection: ambient mode is added inside the response-tool helper, while default/automatic non-Codex heartbeats still choose the legacy HEARTBEAT_OK prompt before that helper runs.

## Solution Assessment

No. Reusing the existing scheduler is plausible, but this implementation needs ambient mode to drive the response-tool/message-tool path, generated config baseline alignment, runner coverage, and stronger scheduled-run proof before it is maintainable.
