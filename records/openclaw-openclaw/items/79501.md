## Summary

Keep open: the PR is not obsolete on current main, but the proposed ambient mode has a blocking runner-path bug and missing generated config baseline coverage. The supplied terminal proof covers schema/helper checks, not an actual scheduled ambient heartbeat run, and proactive ambient outreach also needs maintainer product approval before merge.

## What This Changes

The PR adds `heartbeat.mode = "ambient"` and `heartbeat.maxMessages` with prompt helper, runner wiring, config schema/type updates, heartbeat docs, and focused prompt/schema tests.

## Best Possible Solution

If maintainers want this in core, ambient mode should drive the structured response-tool/message-tool heartbeat path consistently, update generated config artifacts, and include runner-level proof before merge.

## Reproduction Assessment

Not applicable as a current-main bug report; this is a feature PR. The main blocker is source-reproducible from the runner prompt-selection path: ambient handling is added behind a gate that remains false for default/non-Codex automatic heartbeat runs.

## Solution Assessment

No: reusing the existing scheduler is a reasonable direction, but this implementation needs ambient mode to control prompt/tool selection and generated config artifacts before it is the narrow maintainable solution.
