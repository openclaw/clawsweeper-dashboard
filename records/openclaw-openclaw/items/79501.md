## Summary

Keep open: current main does not include the ambient heartbeat config, but this PR only wires ambient behavior into the response-tool prompt path, so supported automatic/non-response-tool heartbeat runs can still use legacy HEARTBEAT_OK semantics with maxMessages ignored. The branch is also currently conflicting with main and the supplied terminal proof does not exercise an actual scheduled heartbeat run.

## What This Changes

Adds heartbeat.mode = "ambient" and heartbeat.maxMessages to heartbeat config/schema/docs, plus an ambient prompt helper used by heartbeat response-tool prompt resolution.

## Best Possible Solution

If maintainers want this feature, route ambient mode at the runner decision point before the response-tool/automatic split, cover both prompt paths with runner tests, and require scheduled-run proof after the branch is refreshed.

## Reproduction Assessment

Yes, for a source-level PR defect: with heartbeat.mode = "ambient" and an automatic/non-response-tool heartbeat path, the runner bypasses the new ambient helper and uses the legacy prompt. I did not execute the PR branch because this is a read-only review.

## Solution Assessment

No. The helper is a plausible building block, but the maintainable fix is runner-level mode resolution before the response-tool/automatic split, or an explicitly validated response-tool-only ambient contract.
