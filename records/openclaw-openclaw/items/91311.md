## Summary

This PR remains necessary because current main only supports loading through trusted skill symlinks, not Skill Workshop apply writes. The patch uses a separate default-false permission, applies the same trusted-root policy to the skill file and support-file writes/removals, and now has convincing current-head real CLI proof; no discrete correctness bug was found, but maintainers should explicitly accept the intentional filesystem security-boundary expansion before merge.

## What This Changes

The PR adds `skills.workshop.allowSymlinkTargetWrites`, threads active config through all Skill Workshop apply entry points, and validates skill and support-file operations against trusted symlink target roots.

## Best Possible Solution

Keep load trust and generated-write authority separate, default writes off, and land the single canonical resolver only after the relevant owners explicitly accept the narrowly configured external-write boundary.

## Reproduction Assessment

Not applicable as a broken-current-behavior bug. The PR has a high-confidence current-head demonstration of the proposed allowed and denied modes using real symlinks and the real CLI.

## Solution Assessment

Yes, subject to owner acceptance: a separate default-false write permission reusing the existing narrow trust list is safer and more maintainable than making read trust implicitly writable or adding entry-point-specific exceptions.
