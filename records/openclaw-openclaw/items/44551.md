## Summary

Keep this open: current main already has substantial Control UI Chinese localization and a browser/setting based locale path, but the exact audit/tool action examples in the report still resolve through English shared tool-display metadata rather than locale strings.

## What This Changes

Request Chinese display support for the Gateway Control UI, especially Audit/Event Log or tool action names such as exec, read, cron run, subagents list, browser, plus other visible UI text.

## Best Possible Solution

Add a focused Control UI i18n layer for user-visible tool/action/event labels, covering the reporter's exec/read/cron/subagents/browser examples and obvious nearby labels, while keeping the generated locale pipeline and shared tool-display contract intact.

## Reproduction Assessment

Yes. Set the Control UI locale to zh-CN or use a zh browser locale, then inspect chat tool cards or event-log entries for exec/read/browser/cron/subagents; the surrounding UI localizes, but those identifiers still come from English shared metadata or raw event names.

## Solution Assessment

Yes for a narrow follow-up, not for closing this as-is. The existing locale infrastructure is the right foundation, but the remaining best fix is to route tool/action display labels through i18n keys instead of returning spec.title/spec.label directly from the shared JSON.

Fix the remaining Chinese localization gap for the Gateway Control UI reported in https://github.com/openclaw/openclaw/issues/44551. Current main already supports zh-CN/zh-TW locale selection, but chat tool cards and event/action labels still show English metadata from apps/shared/OpenClawKit/Sources/OpenClawKit/Resources/tool-display.json or raw event ids.
