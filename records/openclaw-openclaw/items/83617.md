## Summary

Keep open: the report now has live v2026.5.20 confirmation, current main still routes cron `payload.model` through the same allowlist resolver, and the documented canonical `openai/gpt-*` plus `agentRuntime.id: "codex"` shape should not require the legacy `openai-codex/*` workaround.

## What This Changes

The issue requests that isolated cron `payload.model` values using canonical OpenAI refs be accepted when the same refs are configured in `agents.defaults.models` with Codex runtime policy.

## Best Possible Solution

Keep canonical `openai/gpt-*` cron payloads valid when their matching `agents.defaults.models` entry selects the Codex runtime, and add regression coverage for both direct config and doctor-migrated legacy `openai-codex/*` configs.

## Reproduction Assessment

Yes for released behavior, but not by a current-main live run in this review. The issue has v2026.5.20 live confirmation and current main has not changed the cron allowlist resolver since that release, so a focused current-main fixture should be the first repair step.

## Solution Assessment

Yes, the best path is to repair the existing cron allowlist/model-resolution contract and cover the doctor-migrated canonical OpenAI/Codex shape, not to require users to keep using legacy `openai-codex/*` cron refs.
