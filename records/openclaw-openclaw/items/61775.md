## Summary

Keep open: current main still has only the minimal root Makefile, and the latest PR head still leaves ordered compound workflows as ordinary prerequisites that can run concurrently under parallel Make.

## What This Changes

The PR expands the root Makefile into contributor convenience targets for dependency install, build/UI build, cleanup, quality checks, tests, development, version output, and landing-gate workflows.

## Best Possible Solution

Keep the Makefile convenience layer, but make ordered workflows explicit sequential recipes so parallel Make cannot reorder clean/build or gate steps.

## Reproduction Assessment

Yes. Static inspection of latest PR head shows `rebuild` and `landing-gate` are prerequisite-only compound targets; GNU Make can run ordinary prerequisites concurrently with `-j` or inherited `MAKEFLAGS`.

## Solution Assessment

No, not as written. The thin wrapper approach is maintainable, but ordered compound targets should invoke `$(MAKE)` one step at a time or otherwise prevent parallel reordering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Sequence rebuild after cleaning:** `Makefile:35`
  - body: `rebuild` is declared with `clean-dist build` prerequisites, so `make -j rebuild` can run clean and build concurrently. That can remove `dist` while the build is producing it; make this target a sequential recipe that invokes `clean-dist` before `build`.
  - confidence: 0.9
- **[P2] Sequence landing gate steps:** `Makefile:87`
  - body: `landing-gate` uses `check test build` prerequisites, which GNU Make may run concurrently under `-j` or inherited `MAKEFLAGS`. That defeats the promised check-then-test-then-build gate and can overlap broad commands in one worktree; call the targets sequentially from the recipe.
  - confidence: 0.92
