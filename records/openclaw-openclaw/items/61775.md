## Summary

Keep open. Current main still has only the minimal `build` Makefile target, so the PR is not obsolete, but the latest head is not ready as written: the ordered `rebuild` and `landing-gate` workflows are encoded as prerequisites that GNU Make can run concurrently under `-j` or inherited `MAKEFLAGS`. The prior `.PHONY` and `format` review issues appear fixed.

## What This Changes

The PR expands the root `Makefile` from a single `build` target into contributor convenience targets for help, dependency install, build/UI build, cleanup, quality checks, tests, development, version output, and a landing gate, all delegating to existing `pnpm` scripts.

## Best Possible Solution

Keep the Makefile as a thin contributor convenience layer, but express ordered compound workflows as explicit sequential recipe steps that call the existing targets one at a time while preserving the existing `pnpm` script contract.

## Reproduction Assessment

Yes. The issue is reproducible by static inspection plus GNU Make's documented `-j` behavior: `make -j rebuild` or `make -j landing-gate` can run prerequisite recipes concurrently instead of enforcing the intended order.

## Solution Assessment

No, not as written. Delegating to existing `pnpm` scripts is the right direction, but ordered workflows should be recipes or another non-parallel sequencing mechanism rather than ordinary prerequisites.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Sequence rebuild after cleaning:** `Makefile:35`
  - body: `rebuild` is modeled as prerequisites, so `make -j rebuild` can run `clean-dist` and `build` at the same time. That lets the clean step remove `dist` while the build is producing it; make this target an explicit sequential recipe instead.
  - confidence: 0.9
- **[P2] Sequence the landing gate commands explicitly:** `Makefile:87`
  - body: `landing-gate` uses `check test build` prerequisites, which GNU Make may execute concurrently under `-j` or inherited `MAKEFLAGS`. That breaks the documented check-then-test-then-build gate and can overlap broad repo commands; call the targets sequentially from the recipe instead.
  - confidence: 0.92
