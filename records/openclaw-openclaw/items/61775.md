## Summary

Keep open. The Makefile convenience layer is useful and still delegates to existing `pnpm` scripts, but latest head leaves ordered workflows as normal prerequisites, so `make -j` or inherited `MAKEFLAGS` can overlap clean/build and check/test/build steps.

## What This Changes

The PR expands the root `Makefile` into make-based contributor convenience targets for dependency install, build/UI build, cleanup, quality checks, tests, development, version output, and landing-gate workflows.

## Best Possible Solution

Keep the Makefile convenience layer after converting ordered compound targets into sequential recipes that preserve the existing `pnpm` script contract.

## Reproduction Assessment

Yes. Static inspection of PR head shows `rebuild` and `landing-gate` are prerequisite-only compound targets; GNU Make 4.3 supports `-j/--jobs`, so `make -j rebuild` or `make -j landing-gate` can violate the intended order.

## Solution Assessment

No, not as written. The thin wrapper approach is maintainable, but ordered workflows should call `$(MAKE)` targets one at a time from recipe lines instead of relying on ordinary prerequisites.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Sequence rebuild after cleaning:** `Makefile:35`
  - body: `rebuild` is declared as `clean-dist build` prerequisites, so `make -j rebuild` can run the clean and build recipes concurrently. That can remove `dist` while `build` is producing it; make this target a sequential recipe that invokes `clean-dist` before `build`.
  - confidence: 0.9
- **[P2] Sequence landing gate steps:** `Makefile:87`
  - body: `landing-gate` uses `check test build` prerequisites, which GNU Make may run concurrently under `-j` or inherited `MAKEFLAGS`. That defeats the promised check-then-test-then-build gate and can overlap broad test/build commands in one worktree; call the targets sequentially from the recipe.
  - confidence: 0.92
