## Summary

Keep open: this PR is a plausible developer-workflow change, but it is not ready to merge because the ordered Make workflows are still unordered under parallel Make, real behavior proof is missing, and current main removed the root Makefile after the PR was opened.

## What This Changes

The PR expands the root Makefile into a developer convenience wrapper for dependency installation, builds, tests, checks, formatting, and a landing-gate target.

## Best Possible Solution

If maintainers still want a root Makefile, rebase over the removal and keep it as a thin pnpm wrapper with sequential recipes for ordered targets; otherwise close this PR in favor of the documented pnpm scripts.

## Reproduction Assessment

Yes. Static inspection of the PR head is enough: GNU Make may run ordinary prerequisites in parallel, so the `rebuild` and `landing-gate` prerequisite lists can violate their intended order.

## Solution Assessment

No. The wrapper approach can be maintainable only if maintainers want this surface, but ordered workflows should be recipes that call `$(MAKE)` sequentially and the branch must account for current main's Makefile removal.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Sequence rebuild after cleaning:** `Makefile:35`
  - body: `rebuild` is declared as `clean-dist build` prerequisites, so `make -j rebuild` can run clean and build concurrently and remove `dist` while the build is producing it. Make this a sequential recipe that invokes `clean-dist` before `build`.
  - confidence: 0.9
- **[P2] Sequence landing gate steps:** `Makefile:87`
  - body: `landing-gate` uses `check test build` prerequisites, which GNU Make can run concurrently under `-j` or inherited `MAKEFLAGS`. That defeats the promised check-then-test-then-build gate and can overlap broad commands in one worktree; call the targets sequentially from the recipe.
  - confidence: 0.92
