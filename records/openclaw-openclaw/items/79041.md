## Summary

Keep open: current main and the latest release still forward Webhooks `run_task.childSessionKey` into TaskFlow without a route-session-tree guard, and this PR contains the focused fix plus real Gateway proof. It is not merge-ready as submitted because GitHub reports the head as conflicting and the new guard should be rebased onto the current session accessor seam.

## What This Changes

The PR adds a Webhooks `run_task.childSessionKey` route-tree guard, regression coverage, and docs for the accepted child-session contract.

## Best Possible Solution

Rebase this PR, keep the Webhooks-owned fail-closed session-tree guard, use the current row accessor for the session lookup or document a specific escape-hatch reason, and rerun focused proof before merge.

## Reproduction Assessment

Yes, source-reproducible: current main still passes Webhooks `run_task.childSessionKey` directly into the bound TaskFlow runtime, TaskFlow stores it, and cancellation later acts on that key. I did not run live tests in this read-only review; the PR body supplies after-fix terminal proof from a real loopback Gateway route.

## Solution Assessment

Mostly yes: enforcing the guard at the Webhooks route boundary is the narrow owner-boundary fix because Webhooks owns the untrusted HTTP input while generic TaskFlow still serves trusted plugin callers. The submitted branch still needs rebase and accessor alignment before it is the best mergeable version.
