## Summary

Keep open: the PR addresses a real Webhooks session-boundary bug that is still present on current main and the latest release, but the branch is currently conflicting and the stricter 403 behavior is compatibility-sensitive.

## What This Changes

Adds a Webhooks `run_task.childSessionKey` route-session-tree guard, focused handler/plugin-route tests, and docs for the child session key contract.

## Best Possible Solution

Land a rebased Webhooks-owned guard that uses current single-row session access, documents the new `childSessionKey` contract, and treats the stricter 403 behavior as an intentional security compatibility change.

## Reproduction Assessment

Yes, source-reproducible: current main forwards Webhooks `run_task.childSessionKey` into TaskFlow unchecked, TaskFlow stores it, and cancellation later targets that session key. I did not run a live repro in this read-only pass.

## Solution Assessment

Mostly yes: enforcing the guard at the Webhooks route boundary is the right owner boundary because Webhooks owns the external input. The current branch still needs rebase and session-accessor alignment before it is the best mergeable fix.
