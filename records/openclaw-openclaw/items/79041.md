## Summary

## What This Changes

The PR adds a Webhooks `run_task.childSessionKey` session-tree authorization guard, regression coverage for raw and registered routes, and documentation for the stricter child-session contract.

## Best Possible Solution

Land a rebased Webhooks-owned guard that rejects non-session sentinels, uses the current session-entry accessor seam, documents the stricter contract, and explicitly accepts the compatibility break.

## Reproduction Assessment

Yes. Current source and v2026.6.8 both show Webhooks forwarding `childSessionKey` unchecked into TaskFlow, and TaskFlow later persists and cancels by that stored key; I did not run a live failing repro in this read-only review.

## Solution Assessment

No, not yet. Enforcing the guard at the Webhooks route boundary is the right owner layer, but this branch needs the `current` sentinel repaired and the dirty branch rebased before it is the best mergeable solution.
