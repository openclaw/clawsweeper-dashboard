## Summary

The PR is a useful partial fix, but it is not merge-ready because captured plugin approval IDs can still be routed to the exec resolver after queue pruning, and no inspectable after-fix Control UI proof is attached.

## What This Changes

The branch updates the Control UI approval prompt so rendered buttons pass an approval ID into `handleExecApprovalDecision`, and updates the handler/type signature to accept that optional ID.

## Best Possible Solution

Carry render-time approval identity as both ID and kind or resolver method, cover stale-render exec and plugin approval clicks, and merge only after redacted real Control UI proof shows the decision RPC is sent and blocked work proceeds.

## Reproduction Assessment

Yes, by source inspection: render an approval, let the expiry timeout prune `execApprovalQueue`, then click the still-rendered button. Current main reads live queue state and PR head fixes ID capture for exec approvals, but the same stale window remains source-reproducible for plugin approvals because kind is lost.

## Solution Assessment

No. Passing only the ID is a good partial fix for exec approvals, but the shared approval UI needs to preserve the rendered kind or resolver method so plugin approvals do not fall back to the exec resolver after pruning.
