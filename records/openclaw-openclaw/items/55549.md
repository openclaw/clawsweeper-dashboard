## Summary

Keep open: this PR addresses a real current-main Control UI approval race, but the patch still re-derives plugin approval kind from mutable queue state and lacks inspectable after-fix real behavior proof.

## What This Changes

The PR changes Control UI approval buttons to pass a rendered approval ID into `handleExecApprovalDecision` and updates the handler/view-state signature for that optional ID.

## Best Possible Solution

Carry render-time approval identity as ID plus kind or resolver method, add focused stale-render coverage for exec and plugin approvals, and merge only after redacted runtime proof shows the Control UI click submits the approval and the blocked command proceeds.

## Reproduction Assessment

Yes by source inspection: render an approval, let the expiry timeout prune `execApprovalQueue`, then click the still-rendered button. Current main reads live queue state and returns before sending an approval RPC.

## Solution Assessment

No, not as currently patched. Passing render-time identity is the right direction, but the handler must preserve approval kind or resolver method instead of re-deriving it from queue state that may already be gone.
