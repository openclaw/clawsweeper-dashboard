## Summary

Keep this PR open: it targets a real Control UI approval race still present on current main, but the patch is incomplete because captured IDs can lose the plugin approval resolver kind after queue pruning, and the external contributor still has not attached after-fix real behavior proof.

## What This Changes

The PR updates Control UI approval buttons to pass a render-time approval ID into `handleExecApprovalDecision` and updates the handler/type signature to use that ID.

## Best Possible Solution

Capture render-time approval identity as ID plus kind or resolver method, add targeted stale-render coverage for exec and plugin approvals, and require redacted after-fix runtime proof before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: render an approval, let the expiry timer remove it from `execApprovalQueue`, then click the still-rendered button; current main reads `execApprovalQueue[0]` and returns before sending an approval RPC.

## Solution Assessment

No, not as-is. Capturing render-time identity is the right direction, but the patch should capture kind or resolver method with the ID so plugin approvals continue using `plugin.approval.resolve` after pruning.
