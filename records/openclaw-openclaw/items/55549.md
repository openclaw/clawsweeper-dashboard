## Summary

Keep open: current main still has the original live-queue race, and this PR is not merge-ready because the captured ID is not enough to preserve plugin approval resolver routing after the queue is pruned.

## What This Changes

The PR widens the Control UI approval decision handler to accept an optional approval ID and passes the rendered approval ID from the modal action buttons.

## Best Possible Solution

Capture stable resolver metadata with the rendered approval, cover stale-render exec and plugin clicks in focused UI tests, and add the required Unreleased/Fixes changelog line.

## Reproduction Assessment

Yes. Current main can be reproduced by rendering an approval, pruning `execApprovalQueue`, and clicking a still-rendered action button; the PR-branch plugin gap follows the same setup with a rendered plugin approval and an empty live queue.

## Solution Assessment

No, not as-is. Capturing only the ID fixes the exec case but loses resolver kind for stale plugin approvals; the narrow maintainable fix is to capture kind or method with the ID, or otherwise route `plugin:` IDs safely.
