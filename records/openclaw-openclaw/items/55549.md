## Summary

Keep open: the reported stale-render approval race is still source-reproducible on current main, but this branch is not merge-ready because captured plugin approval IDs can still be sent to the exec resolver after queue pruning, and no inspectable after-fix runtime proof is attached.

## What This Changes

The PR changes Control UI approval buttons to pass a rendered approval ID into `handleExecApprovalDecision` and updates the handler/view-state signature to accept that optional ID.

## Best Possible Solution

Carry render-time approval identity as ID plus kind or resolver method, add focused stale-render coverage for exec and plugin approvals, and merge only after redacted runtime proof shows the Control UI click submits the approval and the blocked command proceeds.

## Reproduction Assessment

Yes. By source inspection, render an approval, let the expiry timeout prune `execApprovalQueue`, then click the still-rendered button; current main reads the live queue and returns before sending an approval RPC.

## Solution Assessment

No, not as-is. Passing render-time identity is the right direction, but the handler must preserve kind or resolver method instead of re-deriving it from queue state that may already be gone.
