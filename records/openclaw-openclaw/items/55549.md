## Summary

Keep open: the PR targets a real Control UI stale-render approval race, but it should not merge as-is because captured plugin approval IDs can still be resolved through the exec resolver, and the external contributor has not attached real after-fix behavior proof.

## What This Changes

The branch changes the Control UI approval buttons to pass a rendered approval id into `handleExecApprovalDecision` and updates the handler/view-state signature to accept that optional id.

## Best Possible Solution

Capture render-time approval identity as id plus kind or resolver method, add focused stale-render tests for exec and plugin approvals, and merge only after redacted runtime proof is attached.

## Reproduction Assessment

Yes, by source inspection: render an approval, let the expiry timeout prune `execApprovalQueue`, then click the still-rendered button. Current main reads the live queue and returns before sending an approval RPC, and the PR's plugin path is source-reproducible by pruning a rendered plugin approval before click.

## Solution Assessment

No, not as-is. Passing render-time identity is the right direction, but the handler must preserve kind or resolver method instead of re-deriving it from a queue entry that may already be gone.
