## Summary

Keep this PR open, but it is not merge-ready: the exec approval direction is useful, yet the proposed handler can still misroute stale-render plugin approvals and the PR has no inspectable after-fix Control UI proof.

## What This Changes

The PR changes the Control UI approval view and handler so approval buttons pass a captured approval ID into `handleExecApprovalDecision`.

## Best Possible Solution

Carry rendered approval identity as ID plus kind or resolver method, cover stale-render exec and plugin approval clicks, then merge after redacted Control UI proof shows the approval request and command continuation.

## Reproduction Assessment

Yes, source-reproducible: render an approval, let the expiry timer prune `execApprovalQueue`, then click the still-rendered button. Current main returns before sending a request, and the PR fixes only the ID half of that stale-render state.

## Solution Assessment

No. Capturing the ID is the right direction for exec approvals, but the durable identity also needs the approval kind or resolver method so plugin approvals do not fall back to the wrong backend method.
