## Summary

Keep open: the branch fixes the exec approval ID race directionally, but it still resolves stale-render plugin approval clicks through the exec resolver after queue pruning and lacks inspectable real behavior proof.

## What This Changes

The PR changes Control UI approval buttons to pass a rendered approval ID into `handleExecApprovalDecision`, and updates the handler/view-state signature to accept that optional ID.

## Best Possible Solution

Carry render-time approval identity as ID plus kind or resolver method, cover stale-render exec and plugin approval clicks, then merge after redacted Control UI proof shows the decision is submitted and blocked work proceeds.

## Reproduction Assessment

Yes, by source inspection: render an approval, let the expiry timeout prune `execApprovalQueue`, then click the still-rendered button. Current main returns before sending an approval RPC; PR head fixes ID capture for exec approvals but loses plugin kind in the stale window.

## Solution Assessment

No, not as currently patched. Passing render-time identity is the right direction, but the handler must preserve approval kind or resolver method instead of re-deriving it from queue state that may already be gone.
