## Summary

Keep this PR open, but do not merge it as-is. The Control UI bug is real on current main, but the branch captures only the approval id; after plugin approvals were added, a stale rendered plugin approval can no longer recover its resolver kind and falls back to the exec resolver.

## What This Changes

The branch changes Control UI approval buttons to pass a render-time approval id into `handleExecApprovalDecision` and widens the handler/view-state signature to accept it.

## Best Possible Solution

Capture stable resolver metadata with the rendered approval, keep the legacy fallback for older callers, add stale-render exec and plugin UI coverage, and add one Unreleased/Fixes changelog line.

## Reproduction Assessment

Yes. On current main, render an approval, let the expiry cleanup remove it from `execApprovalQueue`, then click the still-rendered button; the handler reads the now-empty queue and returns without an RPC. On the PR branch, the same setup with a plugin approval re-routes to the exec resolver.

## Solution Assessment

No, not as-is. Capturing only the id is too little state for the shared exec/plugin overlay; the narrow maintainable fix is to capture the approval kind or resolver method alongside the id.
