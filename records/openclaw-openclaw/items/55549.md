## Summary

Keep open: the stale-render approval bug is source-reproducible and the PR is directionally useful, but the current patch loses approval kind when the queued entry has already expired and still lacks inspectable after-fix real behavior proof.

## What This Changes

The PR passes the rendered Control UI approval ID into the approval decision handler and updates the view-state signature so approval button clicks no longer depend only on the live queue head.

## Best Possible Solution

Carry render-time approval identity as ID plus kind or resolver method, cover stale-render exec and plugin approval clicks, then merge after redacted real Control UI proof shows the approval decision is submitted and the blocked command proceeds.

## Reproduction Assessment

Yes by source inspection: render an approval, let the expiry timeout prune `execApprovalQueue`, then click the still-rendered button. Current main reads live queue state and returns before sending an approval RPC.

## Solution Assessment

No, not as currently patched. Passing render-time identity is the right direction, but the handler must preserve approval kind or resolver method instead of re-deriving it from queue state that may already be gone.
