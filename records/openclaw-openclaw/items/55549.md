## Summary

Keep open: the PR targets a real Control UI approval race, but PR head preserves only the approval ID and can still route stale-render plugin approvals to the exec resolver; it also lacks inspectable real behavior proof.

## What This Changes

The PR changes the Control UI approval prompt and handler so rendered approval buttons pass an approval ID into `handleExecApprovalDecision`.

## Best Possible Solution

Carry render-time approval identity as ID plus kind or resolver method, cover stale-render exec and plugin approval clicks, then merge after redacted real Control UI proof shows the decision is submitted and blocked work proceeds.

## Reproduction Assessment

Yes, by source inspection: render an approval, let the expiry timeout prune `execApprovalQueue`, then click the still-rendered button. Current main reads live queue state and returns before sending an approval RPC; PR head fixes ID capture but reproduces the same stale-window failure for plugin approvals by losing kind.

## Solution Assessment

No, not as currently patched. Passing render-time identity is the right direction, but the handler must preserve approval kind or resolver method instead of re-deriving it from queue state that may already be gone.
