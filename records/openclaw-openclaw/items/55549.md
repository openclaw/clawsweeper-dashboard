## Summary

Keep open: the PR addresses a real Control UI approval stale-render race on current main, but the patch is not merge-ready because it preserves only the approval ID and can route pruned plugin approvals through the exec resolver. It also lacks attached after-fix real browser/WebSocket proof despite claiming DevTools verification.

## What This Changes

The PR updates Control UI approval buttons to pass a render-time approval ID into `handleExecApprovalDecision` and updates the handler/type signature to accept that optional ID.

## Best Possible Solution

Capture render-time approval identity as ID plus kind or resolver method, add focused stale-render coverage for exec and plugin approvals, and require redacted after-fix runtime proof before merge.

## Reproduction Assessment

Yes, by source inspection: render an approval, let `enqueueApprovalRequest` prune `execApprovalQueue`, then click the still-rendered button. Current main reads `execApprovalQueue[0]` and returns before sending an approval RPC when that entry is gone.

## Solution Assessment

No, not as-is. Passing render-time identity is the right direction, but the patch must carry kind or resolver method with the ID so plugin approvals still use `plugin.approval.resolve` after pruning.
