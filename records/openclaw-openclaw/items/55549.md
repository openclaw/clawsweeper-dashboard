## Summary

Keep open. Current main still has the stale Control UI approval queue bug, and this PR is useful but incomplete because the captured id is not enough to preserve exec-vs-plugin resolver routing in the same stale-render window.

## What This Changes

The PR changes Control UI approval buttons to pass a render-time approval id into `handleExecApprovalDecision` and widens the handler/view-state signature for that optional id.

## Best Possible Solution

Keep the render-time capture approach, but pass stable approval kind or resolver metadata with the id, add stale-render tests for exec and plugin approvals, and add the required Unreleased/Fixes changelog entry.

## Reproduction Assessment

Yes. Current main can be reproduced by rendering an approval, letting `execApprovalQueue` be pruned, then clicking a still-rendered approval button; the PR-branch plugin gap follows the same setup with a rendered plugin approval and an empty live queue.

## Solution Assessment

No, not as-is. Passing only the id is too little state for the shared exec/plugin overlay; the narrow maintainable fix is to capture the resolver kind or method alongside the id while keeping the existing fallback for legacy callers.
