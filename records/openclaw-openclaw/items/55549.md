## Summary

Keep this PR open. Current main still has the Control UI stale-render approval race, and the PR's render-time capture approach is the right direction for exec approvals, but the branch is not mergeable as-is because it captures only the approval id and then re-derives the approval kind from the mutable queue; after cleanup, stale plugin approval clicks route to the exec resolver instead of the plugin resolver.

## What This Changes

The PR updates Control UI approval buttons to pass a render-time approval id into `handleExecApprovalDecision` and widens the shared handler signature to accept that optional id.

## Best Possible Solution

Keep the render-time capture design, but capture stable routing metadata too, either the approval kind or the exact resolver method, and cover stale-render button clicks for both exec and plugin approvals before merging.

## Reproduction Assessment

Yes. The current-main bug can be reproduced by rendering an approval, pruning `execApprovalQueue`, then clicking the still-rendered button; the PR-branch plugin regression follows the same path, with the queue lookup missing and defaulting to `exec.approval.resolve`.

## Solution Assessment

No, not as-is. Passing only the id is too little state for a shared exec/plugin overlay; the narrow maintainable fix is to pass a stable approval reference or resolver along with the captured id and keep the live-queue fallback only for legacy callers.
