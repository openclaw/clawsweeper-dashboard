## Summary

Keep open: the PR targets a real Control UI approval race, but the patch still loses plugin approval kind when a rendered entry has already been pruned, and there is no inspectable real behavior proof attached.

## What This Changes

The PR updates Control UI approval buttons to pass a rendered approval ID into `handleExecApprovalDecision` instead of having the handler read the live queue head at click time.

## Best Possible Solution

Carry render-time approval identity as ID plus kind or resolver method, cover stale-render exec and plugin approval clicks, then merge after redacted Control UI proof shows the decision is submitted and blocked work proceeds.

## Reproduction Assessment

Yes, by source inspection: render an approval, let the expiry timeout prune `execApprovalQueue`, then click the still-rendered button. Current main returns before sending an approval RPC; PR head fixes ID capture for exec approvals but loses plugin kind in that same stale window.

## Solution Assessment

No, not as currently patched. Passing render-time identity is the right direction, but the handler must preserve approval kind or resolver method instead of re-deriving it from queue state that may already be gone.
