## Summary

Keep open: current main still has the stale-render approval race this PR targets, but the branch is not correct as-is because it captures only the approval id and can route pruned plugin approvals to the exec resolver. The PR also still lacks attached after-fix real behavior proof.

## What This Changes

The PR changes the Control UI approval buttons to pass the rendered approval id into `handleExecApprovalDecision`, which then resolves that id instead of always reading the current first queue entry.

## Best Possible Solution

Capture render-time approval identity as id plus kind or resolver method, add focused stale-render coverage for exec and plugin approvals, and require redacted after-fix runtime proof before merge.

## Reproduction Assessment

Yes, by source inspection: render an approval, let `enqueueApprovalRequest` prune `execApprovalQueue`, then click the still-rendered button. Current main reads the live queue entry at click time and returns before sending the approval RPC when that entry is gone.

## Solution Assessment

No, not as-is. Passing render-time identity is the right direction, but the PR must also carry kind or resolver method so pruned plugin approvals still use `plugin.approval.resolve`.
