## Summary

Keep open. Current main and the latest release still send Feishu streamed final replies as interactive CardKit messages without a searchable fallback, while the focused fallback PR closed unmerged because live tenant search proof and owner tradeoff acceptance were missing.

## What This Changes

The issue asks OpenClaw to make Feishu streaming-card reply content searchable in Feishu conversation search, or to document/replace the delivery mode that prevents searchability.

## Best Possible Solution

Feishu owners should decide the product path with live tenant proof: document the limitation, restore/plain-default behavior through the renderMode discussion, or revive a default-off searchable fallback with explicit acceptance of duplicate delivery and quota tradeoffs.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available from this review. Source inspection shows current main and v2026.6.6 still emit final streamed Feishu replies as interactive cards, but the native Feishu search miss needs live tenant proof.

## Solution Assessment

Unclear. A default-off post fallback is a plausible code path, but it was closed unmerged because the live search outcome and duplicate-message/quota tradeoffs need Feishu owner approval; the related renderMode issue may solve only the default plain-reply part, not all streaming-card searchability.
