## Summary

Keep open. Current main still matches the reported Feishu behavior: streaming card replies are delivered through the CardKit streaming path without a plain-text searchable fallback, the Feishu docs only document disabling streaming, and the related closing PR #74808 is still open rather than merged.

## What This Changes

The issue asks for Feishu streaming-card replies to be searchable in Feishu conversation search, or for the limitation to be documented if Feishu cannot index updated interactive-card content.

## Best Possible Solution

Land or deliberately replace PR #74808 with a Feishu-owned opt-in searchable fallback and docs that explain the visible duplicate-message tradeoff; keep the behavior in the Feishu plugin unless another channel needs the same contract.

## Reproduction Assessment

Yes. The issue gives a concrete Feishu UI path, and current main matches the code-level precondition: streaming card replies use CardKit interactive updates while tests assert no plain-text fallback is sent.

## Solution Assessment

Yes. A Feishu-specific opt-in fallback plus docs is the narrowest maintainable direction because the limitation is tied to Feishu interactive-card indexing, not a shared core reply contract.
