## Summary

Keep open. Current main and release v2026.6.10 still route Feishu streamed final replies through interactive CardKit messages with no searchable post/text fallback, the direct fallback PR https://github.com/openclaw/openclaw/pull/74808 was closed unmerged, and the related default-behavior issue https://github.com/openclaw/openclaw/issues/91384 covers only a narrower subset of the remaining product decision.

## What This Changes

Feishu streamed replies delivered as interactive CardKit cards are reported as missing from Feishu conversation search, and the reporter asks for a searchable fallback, alternate delivery shape, or documentation.

## Best Possible Solution

Keep this open until Feishu owners choose the product path: restore stricter renderMode:auto text-vs-card behavior, add a live-proven default-off searchable fallback, or document a confirmed Feishu search limitation.

## Reproduction Assessment

Unclear end to end, but source-reproducible for the OpenClaw side: current main emits streamed final replies as Feishu interactive cards with no post/text fallback. This review did not live-reproduce Feishu conversation search indexing.

## Solution Assessment

Unclear. The closed default-off post fallback is a plausible implementation, but it is only the best solution if live tenant proof confirms the search result and owners accept the duplicate-delivery tradeoff; the related renderMode:auto fix covers only default plain replies.
