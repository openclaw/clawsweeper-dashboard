## Summary

The report matches the documented group/channel visible-reply policy, not a Feishu transport failure: current main processes the turn but keeps normal final text private unless the model uses the `message` tool or legacy automatic replies are enabled.

## What This Changes

The issue reports that Feishu group messages in release/2026.5.3 process without errors but do not post a visible reply.

## Best Possible Solution

Keep the shared group/channel reply policy as-is, route this symptom to the group visible-reply docs, and reopen only with redacted session/tool/send-failure logs if a `message` tool call fails to post.

## Reproduction Assessment

Yes for the observed no-visible-reply outcome from source: group/channel turns default to `message_tool_only`, and tests cover final text producing `queuedFinal=false` with no `sendFinalReply`. No high-confidence Feishu API send failure is reproduced because no message-tool/send-error session log was provided.

## Solution Assessment

Yes. The narrow maintainable path is config/model guidance, not a Feishu plugin change; set `messages.groupChat.visibleReplies: "automatic"` for legacy automatic room replies or use a model that reliably calls `message(action=send)`.
