## Summary

Keep open: current `main` and the latest release still omit `maxLinesPerMessage` from the direct `openclaw message send` formatting path, and the linked fixing PR is open rather than merged.

## What This Changes

The issue reports that direct Discord CLI sends ignore configured `maxLinesPerMessage`, split at the Discord default, and return an incomplete send receipt.

## Best Possible Solution

Land or replace the linked fixing PR so direct sends resolve the configured Discord line limit before calling the existing outbound adapter, then verify whether receipt reporting needs a narrower follow-up.

## Reproduction Assessment

Yes from source: current `sendMessage()` does not pass configured `maxLinesPerMessage`, while the Discord adapter reads that field and falls back to 17 lines when it is absent. I did not run a live Discord send in this read-only review.

## Solution Assessment

Yes: the best path is the open closing PR or an equivalent narrow change in `src/infra/outbound/message.ts`, not a new config option or docs-only clarification. Receipt completeness may be evaluated after the formatting plumbing is fixed.
