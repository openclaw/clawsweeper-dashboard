## What This Changes

The issue reports that Discord CLI message sends ignore the existing `channels.discord.maxLinesPerMessage` setting, split at the default 17-line limit, and return a receipt that does not expose every Discord message id created by chunking.

## Best Possible Solution

Review and land or reject the linked PR; after the max-lines plumbing lands, track the multi-message receipt-id concern separately if it falls outside that narrow fix.

## Reproduction Assessment

Yes from source, though I did not run a live Discord send in this read-only review. Current main still forwards only `parseMode` from the direct-send path while the Discord adapter falls back to 17 lines when `maxLinesPerMessage` is absent.

## Solution Assessment

Yes for the central bug: plumbing the existing channel/account `maxLinesPerMessage` setting into direct-send formatting is the narrow maintainable fix. The receipt-id concern may need a separate follow-up if it expands the outbound result contract.
