## Label Justifications

- `P2`: This is a reproducible Discord CLI delivery-formatting bug with limited blast radius rather than a channel-wide outage.
- `impact:message-loss`: One logical CLI send can be split into multiple Discord messages while the send receipt reports only one visible part/message id.

## Summary

Keep open: current main and the latest release still omit `maxLinesPerMessage` from the direct `openclaw message send` path, and the linked fixing PR is open rather than merged.

## What This Changes

The issue requests that Discord CLI sends honor the configured `channels.discord.maxLinesPerMessage` line cap, or make the default chunking behavior explicit.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/91885 with a channel-owned formatting fix, then verify a real Discord CLI send/readback honors the configured line cap.

## Reproduction Assessment

Yes. The current source path shows direct `sendMessage` forwards only `parseMode`, while the Discord adapter chunks using `formatting.maxLinesPerMessage` or its 17-line default; I did not run a live Discord send in this read-only review.

## Solution Assessment

Yes for the end state: honoring the existing Discord config in CLI sends is the narrowest product fix. The implementation should avoid duplicating Discord account/default resolution in generic core unless maintainers explicitly bless that shared contract.
