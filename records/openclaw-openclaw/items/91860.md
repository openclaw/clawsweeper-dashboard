## Summary

This should stay open: current main still has a source-backed gap where the generic CLI outbound path only passes explicit formatting, while Discord config-derived `maxLinesPerMessage` is not guaranteed to reach the adapter chunker. Related live-reply reports were fixed by a merged PR, but no open canonical item or current-main fix covers this CLI send case.

## What This Changes

The issue reports that `openclaw message send --channel discord` ignores the existing `channels.discord.maxLinesPerMessage` config, splits around the 17-line default, and may return incomplete multi-message receipt metadata.

## Best Possible Solution

Honor the existing Discord max-lines config in the CLI/direct outbound path without adding a new config surface, then verify and fix multi-chunk receipt metadata only if that path still drops created Discord message ids.

## Reproduction Assessment

No live high-confidence reproduction was executed in this read-only review. The source path is clear enough to reproduce with a focused test: a Discord CLI/direct outbound send with no explicit formatting reaches the adapter without config-derived `maxLinesPerMessage`, so chunking can fall back to 17 lines.

## Solution Assessment

Yes, the best fix is to thread or derive the existing Discord config value at the Discord-owned outbound boundary. Adding a new CLI flag or documenting the mismatch would leave the documented config and CLI behavior inconsistent.
