## Label Justifications

- `P2`: This is a high-confidence, limited-blast-radius Discord CLI delivery bug with an active focused implementation PR.
- `impact:message-loss`: One logical CLI send can unexpectedly become multiple Discord messages while additional delivered message identifiers are easy to miss in the result.

## Summary

Current `main` still ignores the configured Discord line cap on direct CLI sends, and the issue has an open closing PR that now needs owner-boundary review and live Discord proof before landing. The receipt already supports multiple platform IDs, so the central remaining defect is configuration propagation rather than a new CLI option.

## What This Changes

Make Discord CLI sends honor the configured `maxLinesPerMessage` value and clearly expose the identifiers for messages produced by chunking.

## Best Possible Solution

Expose or reuse a channel-owned effective outbound-formatting resolver, forward its result through durable CLI delivery, and prove with a real Discord send that the configured line cap and complete `platformMessageIds` receipt are observable.

## Reproduction Assessment

Yes. Current `main` deterministically omits the line-cap formatting value, and the reporter supplied exact released-version commands plus Discord readback showing the 17-line fallback; this read-only review did not repeat the live transport run.

## Solution Assessment

The proposed configuration plumbing addresses the correct seam, but the PR's core-owned Discord account/config resolver is not the best ownership shape. A channel-owned effective-formatting contract is safer and avoids a second source of truth.
