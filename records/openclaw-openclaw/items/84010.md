Full review comments:

- **[P2] Document the exact provider alias pattern:** `docs/gateway/secrets-plan-contract.md:71`
  - body: This says provider aliases are any alphanumeric/underscore/hyphen string, but current main accepts only `^[a-z][a-z0-9_-]{0,63}$`. Users could copy uppercase, digit-leading, or overlong aliases that `isSecretsApplyPlan` rejects.
  - confidence: 0.94
- **[P2] Align providerDeletes ordering with the apply code:** `docs/gateway/secrets-plan-contract.md:83`
  - body: This says provider deletes run after targets, but `projectPlanState` runs provider mutations before target mutations and `applyProviderPlanMutations` deletes before upserting. A plan containing the same alias in both fields keeps the upserted provider today, so this publishes the wrong contract.
  - confidence: 0.95
- **[P2] Do not say apply prints the configure count summary:** `docs/gateway/secrets-plan-contract.md:87-90`
  - body: This section says `openclaw secrets apply --from ...` logs `Plan: targets=...`, but the apply command only logs dry-run/applied/no-changes results; the count summary is emitted by `secrets configure` before optional apply.
  - confidence: 0.92
- **[P2] Avoid the unsafe exec-provider copy-paste example:** `docs/gateway/secrets-plan-contract.md:65`
  - body: The example sets `allowInsecurePath: true` without explaining the trusted-path restriction used by existing secrets docs. Please remove it from the normal example or add the same safety caveat so users do not bypass exec path checks casually.
  - confidence: 0.78
