## Summary

Keep open. The feature is valid and has user interest, and the current head addressed some early bot-review feedback, but the patch is stale against current main and exposes new access-control config before enforcing it consistently on transcript/status surfaces or generated metadata.

## What This Changes

The PR adds per-agent `agents.list[].tools.sessions.visibility` and `agentToAgent` schema/types/help plus partial session list/send policy wiring and tests.

## Best Possible Solution

Choose the canonical per-agent session/A2A policy shape, port it through `src/plugin-sdk/session-visibility.ts`, enforce it consistently across list/history/send/status and SDK consumers, then regenerate schema metadata and add changelog/docs coverage.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure global `tools.sessions.visibility: "all"` with a per-agent requester override such as `sessions.visibility: "self"`, then call `sessions_history` or `session_status` for another agent; the PR does not thread requester tools into those surfaces.

## Solution Assessment

No. Per-agent overrides are a plausible solution, but this patch is not the best merge path until it is rebased onto the current plugin-sdk policy helper and applies the policy to every session access surface.
