## Summary

Keep open. This is a useful attempt at a real per-agent session/A2A access-control feature, but it is not mergeable as-is because the PR accepts new policy config while only partially enforcing it, lacks contributor real behavior proof, and overlaps active related proposals that need maintainer policy alignment.

## What This Changes

The PR adds optional per-agent `agents.list[].tools.sessions` and `agents.list[].tools.agentToAgent` config, updates list/send policy wiring and config metadata, and adds sessions-access tests plus a session path regression test.

## Best Possible Solution

Consolidate the feature in the current `src/plugin-sdk/session-visibility.ts` policy helper, wire all four session tools consistently, refresh generated config metadata/docs/tests, and align the outcome with the related open session-access proposals.

## Reproduction Assessment

Yes for the PR review findings: source and diff inspection show the branch accepts new per-agent config while only list/send get requester-agent policy wiring, and the generated config baseline is not updated. The requested behavior itself is a new config capability, not a current-main bug reproduction.

## Solution Assessment

No. Per-agent scoping is a reasonable direction, but this patch is not the best merge path until enforcement is centralized in the current helper, history/status are covered, generated metadata is refreshed, and maintainers choose the final access-control semantics.
