## Summary

Keep open: the PR is aimed at a real Telegram stream-preview failure mode, but the proposed backoff is not merge-ready because ordinary preview updates can still flush before the `retry_after` deadline. It also would close broader delivery-queue issue #51628 without touching that queue/replay path, and it lacks the required changelog entry.

## What This Changes

The PR changes Telegram stream previews to retry rate-limited sends/edits after `retry_after`, retry transient edits, retry pre-connect initial sends, and export a retry-after extractor.

## Best Possible Solution

Make the Telegram draft-stream loop honor retry deadlines across all flush paths, add focused regression coverage and a changelog entry, and keep broader outbound queue/replay work on its existing issues.

## Reproduction Assessment

Yes, by source inspection. Current main stops the preview stream on any send/edit exception, and the PR head has a clear source-level backoff bypass when `update()` arrives before the separate retry timer fires.

## Solution Assessment

No. The PR is in the right owner module, but the backoff must be integrated with the stream loop or update scheduling so every retry path respects the deadline; it also should not close the broader delivery-queue replay issue.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Honor backoff before flushing updates:** `extensions/telegram/src/draft-stream.ts:255-258`
  - body: After a 429, this schedules a side timer and returns `false`, so the shared loop keeps the text pending. If another preview update arrives before the timer fires, `loop.update()` sees no loop timer or in-flight send and can call `flush()` immediately, retrying before Telegram's `retry_after` window. Gate normal update-triggered flushes while backoff is active, or move the retry deadline into the loop.
  - confidence: 0.87
