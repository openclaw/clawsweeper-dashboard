## Summary

Keep open: current main still has a source-visible discoverability gap for rejected cron `payload.model` values. The run now records diagnostics, but the rejection returns before cron delivery runs and there is no warning at the model-rejection point; related open issues cover specific model allowlist failures, not this generic visibility problem.

## What This Changes

The issue requests visible logging and announcement for cron or `/hooks/agent` runs rejected because `payload.model` is outside the configured allowlist.

## Best Possible Solution

Keep strict model validation, but log the rejected model and allowlist hint at cron preflight and route that specific failure through the visible cron or hook failure announcement path with focused regression coverage.

## Reproduction Assessment

Yes, source-level: a nonempty cron `payload.model` is rejected before execution and delivery, while docs and tests show this is the explicit validation path. I did not run an end-to-end gateway cron because this review is read-only and `pnpm` is unavailable in the checkout environment.

## Solution Assessment

Yes for the minimum fix: improve warning and announcement visibility while preserving fail-closed allowlist behavior. Restoring silent fallback-to-default should stay out of the repair unless maintainers make a separate product decision.
