## Summary

Keep open. Current main still lacks Telegram session lifecycle control-lane routing, and this PR is close but not mergeable as-is because the broad `/reset` predicate also catches `/reset soft`, which current code treats as an in-place mutation that should remain serialized with the active session. The earlier Greptile duplication note appears addressed by the shared helper extraction, but the soft-reset regression and required changelog entry still need repair.

## What This Changes

The PR extracts a shared Telegram command resolver, routes registered `/new` and `/reset` text commands to the Telegram `:control` sequentializer lane, and adds sequential-key tests for bare `/new`, bare `/reset`, and `/new@bot` routing.

## Best Possible Solution

Land a repaired version that keeps the control-lane bypass for `/new` and hard `/reset` lifecycle commands, explicitly keeps `/reset soft` forms on the normal chat/topic lane, adds focused sequential-key and middleware regressions, and records the user-facing Telegram fix in the changelog.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: on the PR head, `/reset soft` is accepted under the `/reset` alias and the new predicate returns the control lane; a focused regression should assert `/reset soft` remains on the normal Telegram chat/topic key while bare `/reset` bypasses.

## Solution Assessment

No. The PR direction is useful, but the current predicate is broader than the maintainable fix because it conflates hard reset/new-session control commands with soft reset's same-session mutation path.
