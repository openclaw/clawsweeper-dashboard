## Summary

Keep open: the PR addresses a real Telegram queueing problem, but the current patch routes all `/reset` forms onto the control lane, including `/reset soft`, and no real Telegram after-fix proof has been posted.

## What This Changes

The PR adds Telegram sequential-key logic that routes `/new` and `/reset` messages to a per-chat control lane so they bypass the normal chat/topic queue.

## Best Possible Solution

Narrow control-lane routing to hard session lifecycle resets, keep `/reset soft` serialized, add focused sequential-key coverage, and attach redacted live Telegram proof for the busy-turn path.

## Reproduction Assessment

No live Telegram reproduction is attached, but source inspection is high-confidence: current main serializes `/reset`, while the PR's alias-based helper would match accepted-argument forms like `/reset soft` and route them to `:control`.

## Solution Assessment

No: the control-lane direction is plausible for hard `/new` and bare `/reset`, but the current patch is too broad because it does not preserve the existing soft reset semantics or add proof.
