## Summary

This PR should stay open because the current head still has the previously identified soft-reset routing bug and still lacks contributor-supplied real behavior proof. The core direction is useful, but `/reset soft` must remain serialized with the active session instead of bypassing the queue.

## What This Changes

The PR adds Telegram sequential-key routing that sends `/new` and `/reset` to a per-chat control lane, plus focused sequential-key tests.

## Best Possible Solution

Land a repaired version that bypasses `/new` and hard `/reset`, keeps `/reset soft` on the normal chat/topic lane, adds a focused regression, and includes redacted real Telegram or equivalent terminal/log proof.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: the PR predicate routes any command resolved to key `reset` to `:control`, while current normalization resolves `/reset soft` as `/reset` and current session code treats soft reset as same-session reuse.

## Solution Assessment

No. The approach is right for `/new` and hard `/reset`, but it must inspect the reset form or parsed reset mode so `/reset soft` stays serialized on the normal lane.
