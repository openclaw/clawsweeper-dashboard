## Summary

Keep open: the PR is a useful Telegram bug fix direction, but it still lacks contributor-supplied real behavior proof and the current predicate is too broad for `/reset soft`, which mutates the existing session and should stay serialized.

## What This Changes

The PR routes Telegram `/new` and `/reset` aliases through a new session-control command predicate so they use the `telegram:<chatId>:control` lane, and adds sequential-key tests for `/new`, `/reset`, and bot-suffixed `/new`.

## Best Possible Solution

Land a repaired version that bypasses `/new` and hard `/reset`, keeps `/reset soft` on the normal chat/topic lane, adds a focused regression for the soft-reset case, and includes redacted real Telegram or terminal proof before merge.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence. On the PR head, `/reset soft` resolves through the argument-accepting `/reset` alias and reaches the new key-only control-lane predicate while current main handles soft reset as an in-place session mutation.

## Solution Assessment

No. The direction is right for `/new` and hard `/reset`, but the implementation should inspect the raw reset form or parsed reset mode so `/reset soft` remains serialized on the normal lane.
