## Summary

Keep open. The PR targets a real Telegram queueing gap that current main still has, but the proposed predicate routes every `/reset` form to the control lane, including `/reset soft`, which current main treats as an in-place session mutation that should remain serialized; it also lacks the required changelog entry.

## What This Changes

The PR extracts a shared Telegram command resolver and routes registered `/new` and `/reset` commands to the Telegram `:control` sequentializer lane, with unit coverage for `/new`, `/reset`, and bot-username-suffixed `/new`.

## Best Possible Solution

Keep the control-lane bypass for bare `/new` and hard `/reset`, but explicitly keep `/reset soft` variants on the normal Telegram chat/topic lane, then add focused sequential-key and middleware regressions plus a single-line `CHANGELOG.md` entry crediting the contributor.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: on the PR head, `/reset soft` resolves through the accepted-args `/reset` alias and `getTelegramSequentialKey` returns `telegram:<chatId>:control`; a focused regression should assert it remains on the normal chat/topic key.

## Solution Assessment

No. The current PR direction is close, but the broad command-key predicate is not the narrowest maintainable fix because it conflates hard reset with soft reset; excluding soft-reset forms before returning the control lane is safer.
