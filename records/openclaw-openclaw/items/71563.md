## Summary

Keep this PR open: it addresses a real Telegram queueing gap, but the current head still routes same-session soft resets onto the bypass lane and has no contributor-supplied real behavior proof.

## What This Changes

The PR adds Telegram sequential-key logic and tests so `/new` and `/reset` resolve to a per-chat `:control` lane instead of the normal per-topic lane.

## Best Possible Solution

Land a repaired version that bypasses `/new` and hard `/reset`, keeps soft-reset forms serialized on the normal chat/topic lane, adds focused regression coverage, and includes redacted real Telegram or equivalent runtime proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: PR head routes any resolved `reset` command to `:control`, while current command normalization and reset tests show `/reset soft` is a valid same-session reset form.

## Solution Assessment

No. The control-lane direction is appropriate for `/new` and hard `/reset`, but the implementation should inspect the reset form or parsed reset mode so soft resets stay on the serialized lane.
