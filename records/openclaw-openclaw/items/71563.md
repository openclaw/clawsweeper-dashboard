## Summary

Keep open: current main and v2026.6.8 still route `/new` and `/reset` through the normal Telegram lane, and this PR is a plausible focused fix, but it is not merge-ready without real Telegram proof and upgrade-safety confidence for session-mutating commands bypassing the queue.

## What This Changes

This PR adds Telegram sequential-key detection for `/new` and `/reset` so those session lifecycle commands return the chat control lane instead of the per-chat or per-topic lane.

## Best Possible Solution

Merge only after focused Telegram proof and regression coverage show `/new` and `/reset` bypass the queue without stale replies or session-state drift.

## Reproduction Assessment

Yes, at source level: current main maps `/new` and `/reset` to the ordinary topic/default lane, and grammY sequentialize waits on same-key updates. I did not run a live Telegram reproduction in this read-only review, so the reproduction is source-backed rather than observed.

## Solution Assessment

Unclear as merge-ready: routing these commands to the control lane is an acceptable mitigation, but it needs proof that session reset and reply-fence behavior remain correct while an old turn is active. A shared command-resolution helper plus focused tests would be the cleaner implementation shape if maintainers want this path.
