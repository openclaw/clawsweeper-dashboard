## Summary

Keep this PR open. The reported Telegram queueing problem is valid and the control-lane direction is appropriate, but the current PR head still routes `/reset soft` to the control lane because it matches the registered `reset` command key without checking the reset mode; that would let an in-place soft reset interleave with an active turn on the same session.

## What This Changes

The PR extracts a shared Telegram control-command resolver and routes registered `/new` and `/reset` commands to the Telegram control sequentializer lane, with sequential-key unit coverage for `/new`, `/reset`, and `/new@bot`.

## Best Possible Solution

Keep the control-lane bypass for `/new` and hard reset commands, but explicitly keep soft-reset forms serialized on the normal Telegram chat/topic lane, then add focused unit and middleware regression coverage plus the required changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is clear: on the PR head, `getTelegramSequentialKey` would classify `/reset soft` as `telegram:<chatId>:control` because `/reset` accepts args and alias resolution returns `/reset`; a focused regression should assert that `/reset soft` remains on `telegram:<chatId>` or the topic key.

## Solution Assessment

No. The broad command-key predicate is not the best fix because it treats soft reset as a hard session lifecycle command; the safer implementation checks the normalized/raw reset form and excludes `/reset soft` variants before returning the control lane.
