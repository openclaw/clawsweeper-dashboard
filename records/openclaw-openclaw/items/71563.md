The PR modifies Telegram sequential-key routing and tests so `/new` and `/reset` resolve to a per-chat control lane.

## Best Possible Solution

Land a repaired version that bypasses `/new` and hard `/reset`, keeps soft reset forms serialized on the normal lane, adds a focused regression, and includes redacted real Telegram or equivalent terminal/log proof.

No. The direction is right for `/new` and hard `/reset`, but the implementation must exclude soft reset forms or parse reset mode before choosing the control lane.
