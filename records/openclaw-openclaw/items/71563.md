## Summary

Keep this PR open. The direction is useful and current main still lacks the `/new`/hard `/reset` Telegram control-lane bypass, but the latest PR head routes every registered `/reset` form to the control lane, including `/reset soft`, which current main treats as an in-place same-session mutation that should stay serialized with the active turn.

## What This Changes

The PR extracts shared Telegram command resolution, routes registered `/new` and `/reset` messages to the Telegram control sequentializer lane, and adds sequential-key tests for bare `/new`, bare `/reset`, and `/new@bot`.

## Best Possible Solution

Land a repaired version that keeps bare `/new` and hard `/reset` forms on the Telegram control lane, explicitly leaves `/reset soft` and `/reset: soft` forms on the normal chat/topic lane, adds focused regressions for both paths, and records the user-facing Telegram fix in the changelog.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: on the PR head, `/reset soft` normalizes to the `/reset` command alias and the new key-only predicate routes it to `telegram:<chatId>:control`; a focused sequential-key test should assert that soft reset stays on `telegram:<chatId>` or the topic key.

## Solution Assessment

No. The PR is aimed at the right Telegram queueing problem, but the current predicate is broader than the maintainable fix because it conflates hard session lifecycle commands with `/reset soft` same-session mutation behavior.
