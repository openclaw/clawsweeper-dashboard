## Summary

Keep open: current main and the latest release still do not define how Telegram `channel_post` commands should handle sender-scoped per-chat `allowFrom`, and the linked open PR explicitly leaves that authorization policy unsettled.

## What This Changes

The issue asks OpenClaw to define and document the Telegram `channel_post` authorization rule when `groups[chatId].allowFrom` is configured but Telegram does not expose a normal human sender.

## Best Possible Solution

Choose and document one Telegram `channel_post` authorization policy, then update the Telegram native command path, regression tests, and public docs with live allowed/denied Telegram proof.

## Reproduction Assessment

Partly: source inspection shows the current command auth path depends on `msg.from` and rejects sender-required allowlists when no sender is available. I did not run a live Telegram channel_post reproduction in this read-only review.

## Solution Assessment

No implementation path is best yet because maintainers must choose the security semantics first. The maintainable path is a narrow policy decision followed by code, tests, docs, and live Telegram proof for that chosen behavior.
