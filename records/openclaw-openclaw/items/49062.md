## Summary

Keep open: current main and v2026.6.8 still do not define the Telegram channel_post plus sender-scoped per-chat allowFrom rule, and the paired open PR explicitly leaves that authorization policy unsettled.

## What This Changes

The issue asks OpenClaw to define and document how Telegram channel_post commands should handle sender-scoped per-chat allowFrom when Telegram does not expose a normal human sender.

## Best Possible Solution

Choose one Telegram channel_post authorization policy, then update the Telegram extension code, focused auth tests, public docs, and live allowed/denied Telegram proof to match it.

## Reproduction Assessment

Partly: source inspection shows the current command auth path depends on msg.from and the sender-scoped allowFrom helper fails when senderId is absent. I did not run a live Telegram channel_post reproduction in this read-only review.

## Solution Assessment

No implementation path is best yet because maintainers must choose the security semantics first. The maintainable path is a narrow policy decision followed by code, tests, docs, and live Telegram proof for the chosen behavior.
