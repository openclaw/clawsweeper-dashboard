## Summary

Keep open: current main and the latest shipped release still do not define Telegram `channel_post` semantics for sender-scoped `groups[chatId].allowFrom`, and the paired open PR at https://github.com/openclaw/openclaw/pull/49063 explicitly leaves that policy out of scope.

## What This Changes

The issue asks maintainers to define and document how Telegram `channel_post` native commands should handle sender-scoped `groups[chatId].allowFrom`.

## Best Possible Solution

Choose one Telegram `channel_post` authorization rule, implement it in the Telegram plugin, document it at https://docs.openclaw.ai/channels/telegram, add focused auth coverage, and require live allowed/denied Telegram proof before landing.

## Reproduction Assessment

Yes, source-level: current main still has grammY command matching that can include `channelPost`, while OpenClaw native handlers read `ctx.message` and the sender-scoped override helper requires a sender id. I did not run a live Telegram channel session in this read-only review.

## Solution Assessment

No, not yet. The paired PR solves only the explicit-chat channel command path; the maintainable fix for this issue is a maintainer-approved auth rule plus docs, regression tests, and live Telegram proof.
