## Summary

Keep open: current main still lacks a defined Telegram `channel_post` policy for sender-scoped `groups[chatId].allowFrom`, and the paired open PR at https://github.com/openclaw/openclaw/pull/49063 explicitly leaves that semantics decision out of scope.

## What This Changes

The issue asks OpenClaw to define and document Telegram `channel_post` authorization when a per-chat sender-scoped `groups[chatId].allowFrom` override is present.

## Best Possible Solution

Choose one Telegram `channel_post` authorization policy, implement it in the Telegram plugin, document it on the public Telegram docs page, add focused auth coverage, and require live Telegram proof for allowed and denied channel command behavior.

## Reproduction Assessment

Yes, source-level. Current main still routes native commands through `ctx.message`, sender-scoped overrides require sender identity, and the pinned grammY contracts expose `channel_post` separately with optional `Message.from`.

## Solution Assessment

No automated fix should choose the policy here. The maintainable path is a maintainer-approved authorization rule plus docs, regression tests, and live Telegram proof; the paired PR only handles explicitly allowed channel chats.
