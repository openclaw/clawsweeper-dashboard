## Summary

Keep this issue open. Current main still has no explicit Telegram `channel_post` policy for sender-scoped per-chat `allowFrom`, and the related same-author PR remains open while explicitly leaving that policy out of scope.

## What This Changes

Define and document Telegram `channel_post` authorization behavior when `groups[chatId].allowFrom` is configured.

## Best Possible Solution

Add an explicit Telegram `channel_post` auth rule for sender-scoped `allowFrom`, cover it with targeted native-command and group-access tests, and document the chosen behavior on the public Telegram channel docs page.

## Reproduction Assessment

Yes. The current-main static path is high-confidence: Telegram native command handlers return when `ctx.message` is absent, and the auth helpers require sender identity for sender-scoped `allowFrom`; the related PR is the implementation candidate for channel command delivery but leaves this policy unsettled.

## Solution Assessment

Unclear until a maintainer chooses the product/security policy. The related PR may be useful for explicit-chat `channel_post` command execution, but it is not a complete solution for this issue because it deliberately leaves sender-scoped `allowFrom` semantics undefined.
