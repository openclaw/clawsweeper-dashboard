## Summary

Keep open: current main and v2026.6.6 still leave Telegram channel_post native-command authorization undefined for sender-scoped per-chat allowFrom, and the linked implementation PR is open, conflicting, and explicitly does not settle that policy.

## What This Changes

The issue asks maintainers to define and document how Telegram channel_post native commands should behave when sender-scoped groups[chatId].allowFrom is configured.

## Best Possible Solution

Choose and document the channel_post auth policy first, then implement native-command handling so channel identity, command allowlists, group/channel policy, callbacks, tests, and live Telegram proof all follow that policy.

## Reproduction Assessment

Yes at source level, but not through live Telegram proof here: current main's native command handlers read ctx.message, while sender-scoped allowFrom paths require a sender ID that channel_post does not provide as a normal human sender.

## Solution Assessment

No current solution is in place. The related PR is only a partial path because it is open, conflicting, proof-blocked, and explicitly does not define sender-scoped groups[chatId].allowFrom semantics for channel_post.
