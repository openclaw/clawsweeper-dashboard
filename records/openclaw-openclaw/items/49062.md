## Summary

Keep open. Current main still does not define a Telegram `channel_post` native-command auth policy for sender-scoped per-chat `allowFrom`, and the related open PR explicitly leaves that semantics question out of scope.

## What This Changes

Define and document how Telegram native commands authorize `channel_post` updates when `groups[chatId].allowFrom` is configured.

## Best Possible Solution

Choose one explicit `channel_post` auth rule, enforce it in Telegram native-command auth, add targeted regression coverage, and document the rule on the Telegram channel docs page.

## Reproduction Assessment

Yes, by source inspection rather than live Telegram: current native command handlers only read `ctx.message`, while the auth helpers require sender identity when sender allowlists are enforced. The missing piece is an explicit current-main policy for sender-less `channel_post` commands with per-chat `allowFrom`.

## Solution Assessment

Unclear until a maintainer chooses the product/security policy. The related PR may address explicit-chat command delivery, but it is not the best complete fix for this issue because it deliberately leaves sender-scoped `allowFrom` semantics unresolved.
