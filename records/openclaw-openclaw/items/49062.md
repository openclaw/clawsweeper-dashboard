## Summary

Keep open: current main still lacks an explicit Telegram `channel_post` policy for sender-scoped per-chat `allowFrom`, the public Telegram docs do not document a channel-post exception, and the paired same-author PR remains open while explicitly leaving this policy unresolved.

## What This Changes

The issue asks OpenClaw to define how Telegram `channel_post` native commands should behave when `groups[chatId].allowFrom` is configured but no originating human sender is available.

## Best Possible Solution

Maintainers should choose one Telegram `channel_post` authorization rule, implement it in the Telegram plugin, document it at https://docs.openclaw.ai/channels/telegram, and verify allowed and denied behavior with focused tests plus live Telegram proof.

## Reproduction Assessment

Yes, source-level. Current main has no explicit native-command `channel_post` auth branch, and the shared sender-scoped `allowFrom` helper fails closed when sender identity is required but unavailable.

## Solution Assessment

No current implementation settles the issue. The maintainable fix is a maintainer-approved `channel_post` authorization rule with docs, tests, and live Telegram proof rather than closing around the open partial PR.
