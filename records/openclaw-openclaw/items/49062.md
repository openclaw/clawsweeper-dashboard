## Summary

This should stay open for maintainer follow-up. Current main does not define the requested `channel_post` sender-scoped policy, the related open PR explicitly leaves `groups[chatId].allowFrom` semantics out of scope, and Telegram's Bot API makes `Message.from` optional for channel messages while grammY documents `ctx.msg` as the cross-message shortcut. ([core.telegram.org](https://core.telegram.org/bots/api%20))

## What This Changes

The issue asks OpenClaw to define and document Telegram `channel_post` native-command authorization when sender-scoped `groups[chatId].allowFrom` is configured.

## Best Possible Solution

Define the Telegram `channel_post` auth policy in the Telegram plugin, document it on the public Telegram channel page, and cover explicit chat allow plus sender-scoped override cases with regression tests and live Telegram proof.

## Reproduction Assessment

No high-confidence current-main runtime reproduction was established. The source path is clear: current handlers do not process `channel_post` commands, and the sender-scoped auth helpers would fail closed when a per-chat `allowFrom` requires a missing `msg.from` sender.

## Solution Assessment

Unclear until maintainers choose the policy. The narrow maintainable solution is either an explicit chat-only `channel_post` auth rule or an explicit unsupported/reject path, with docs and tests rather than an implicit fallback.
