## Summary

Keep open: current main still lacks a defined Telegram channel_post sender-scoped authorization policy, and the paired open PR explicitly leaves groups[chatId].allowFrom semantics out of scope. This is an authorization/product semantics decision, not a strict already-defined bug fix.

## What This Changes

The issue asks maintainers to define and document how Telegram channel_post native commands should behave when per-chat groups[chatId].allowFrom sender-scoped authorization is configured.

## Best Possible Solution

Define one channel_post authorization rule in the Telegram plugin, document it in the Telegram channel docs, cover it with focused auth tests, and require real Telegram proof before landing because this changes command authorization behavior.

## Reproduction Assessment

Yes, source-level. Current main’s native command handlers return when ctx.message is absent, and the Telegram/grammY contracts show channel_post data lives outside ctx.message while Message.from may be unavailable for channel messages.

## Solution Assessment

No, the paired PR is not enough for this issue because it explicitly leaves groups[chatId].allowFrom semantics undefined. The maintainable fix is a maintainer-approved policy plus docs, tests, and live Telegram proof.
