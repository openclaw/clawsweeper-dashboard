## Summary

Keep open: current main and the latest release still lack a defined Telegram `channel_post` policy for sender-scoped `groups[chatId].allowFrom`, and the paired open PR explicitly leaves that authorization semantics decision out of scope.

## What This Changes

Define and document how Telegram `channel_post` native commands should handle per-chat sender-scoped `groups[chatId].allowFrom` when Telegram does not provide a human sender.

## Best Possible Solution

Choose one Telegram `channel_post` authorization policy, implement it in the Telegram plugin, document it on the Telegram channel docs page, add focused auth coverage, and require live Telegram proof for allowed and denied channel command behavior.

## Reproduction Assessment

Yes, source-level. Current main’s native command callbacks still read `ctx.message`, while Telegram and grammY expose channel posts through a separate `channel_post`/`ctx.msg` surface, and Telegram `Message.from` may be empty for channel messages. ([core.telegram.org](https://core.telegram.org/bots/api%20%20?utm_source=openai))

## Solution Assessment

No automated fix should choose the policy here. The maintainable path is a maintainer-approved authorization rule plus docs, tests, and live Telegram proof; the paired PR is explicitly not enough for the `groups[chatId].allowFrom` semantics.
