## What This Changes

The issue asks OpenClaw to explicitly define and document how Telegram channel_post native commands behave when sender-scoped groups[chatId].allowFrom is configured.

## Best Possible Solution

Choose one Telegram channel_post authorization policy, then update the Telegram plugin code, focused auth tests, public docs, and live allowed/denied Telegram proof to match it.

Do we have a high-confidence way to reproduce the issue? Source-reproducible only: current code reads msg.from for command sender identity and the sender-scoped override path fails when senderId is absent, but I did not run a live Telegram channel_post reproduction.
