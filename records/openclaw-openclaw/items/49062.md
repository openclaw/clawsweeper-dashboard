## Summary

Keep open: current main still does not define Telegram `channel_post` behavior for sender-scoped per-chat `allowFrom`, and the paired open PR explicitly leaves that policy out of scope. This is an authorization semantics decision rather than a narrow already-defined bug fix.

## What This Changes

Request to define and document how Telegram `channel_post` native commands should behave when `groups[chatId].allowFrom` is configured but Telegram does not expose the originating human sender.

## Best Possible Solution

Define one Telegram `channel_post` authorization rule in the plugin, document it in the Telegram channel docs, cover it with focused auth tests, and require live Telegram proof before landing.

## Reproduction Assessment

Yes, source-level. Current main’s native command path does not consume `channel_post`, and the versioned grammY/types contracts show `channel_post` is separate from `message` while `Message.from` may be absent for channel messages.

## Solution Assessment

No, not as an automated bug fix. The maintainable path is a maintainer-approved authorization policy plus docs, regression tests, and live Telegram proof; the paired open PR intentionally does not settle the `groups[chatId].allowFrom` rule.
