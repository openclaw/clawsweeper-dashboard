## Summary

Keep open: current main covers useful pieces of Telegram 409 recovery, Telegram context filtering, and stale webchat delivery-context precedence, but the bundled report is not fully implemented because CLI reseed history still renders prior transcript as authoritative context without timestamp/freshness semantics.

## What This Changes

The issue asks OpenClaw to improve Telegram polling conflict recovery, prevent Telegram sessions from routing outbound replies through stale webchat metadata, and stop stale prior messages from being reinjected as current CLI session history.

## Best Possible Solution

Keep this issue open as the umbrella until maintainers split or canonicalize the remaining mechanisms: route Telegram 409 lifecycle work to https://github.com/openclaw/openclaw/issues/75852, keep the delivery-context precedence coverage, and decide the CLI reseed-history freshness contract.
