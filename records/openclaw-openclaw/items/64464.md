## Summary

Keep open. Current main still does not synthesize `DISPLAY=:0` for visible WSL2 managed Chrome launches; it only falls back to headless for implicit Linux no-display starts and errors for explicit headed starts, while the related upstream PRs for this exact behavior are closed unmerged.

## What This Changes

The issue requests a browser-plugin fix so WSL2 headed Chrome launches receive `DISPLAY=:0` when the parent environment does not provide a display.

## Best Possible Solution

Add a browser-plugin-scoped WSL2 headed launch path: for local managed Chrome starts only, when headed mode is explicitly selected and no display variable is inherited, detect WSL2 via the plugin SDK runtime helper and pass `DISPLAY=:0` to the Chrome child process while leaving headless, remote CDP, existing-session, and user-provided display values unchanged.

## Reproduction Assessment

Yes. The high-confidence reproduction is a current-main code path: local managed Linux profile, WSL2, `headless=false`, and no `DISPLAY`/`WAYLAND_DISPLAY` returns the explicit no-display error before spawn, and the spawn env lacks any `DISPLAY=:0` fallback.

## Solution Assessment

Yes, with tighter scope than the original line-level suggestion. A WSL2-only, no-inherited-display, local-managed-headed fallback is the narrow maintainable solution; a generic all-Linux `DISPLAY=:0` injection would be riskier.
