## Summary

Keep open. Current main still does not synthesize a WSL2 `DISPLAY=:0` fallback for visible managed Chrome launches; explicit headed Linux starts with no inherited display still fail before spawn, and all target-repo PRs for this exact behavior are closed unmerged.

## What This Changes

The issue requests a browser-plugin fix so headed WSL2 Chrome launches get `DISPLAY=:0` when no display variable is inherited.

## Best Possible Solution

Add a browser-plugin-scoped WSL2 headed launch fallback only for local managed Chrome starts when headed mode is explicit and no display variable is inherited, preserving existing display values, headless launches, remote CDP, and existing-session profiles.

## Reproduction Assessment

Yes, at source level. On current main, local managed Linux with explicit headed mode and no `DISPLAY` or `WAYLAND_DISPLAY` returns the no-display error before spawn, and the spawn env has no WSL2 `DISPLAY=:0` fallback.

## Solution Assessment

Yes, if scoped tighter than the original line-level suggestion. The maintainable fix is WSL2-only, only when no display is inherited, and wired into the current `prepareOomScoreAdjustedSpawn(..., { env })` path with regression coverage.
