## Summary

Keep open. Current main and the latest release still do not synthesize a WSL2 `DISPLAY=:0` fallback for visible managed Chrome launches, but prior maintainer discussion cautions against broad auto-injection, so this needs an explicit support-boundary decision before a repair lane.

## What This Changes

The issue asks browser control to set `DISPLAY=:0` for WSL2 visible Chrome launches when `headless: false` and Chrome would otherwise fail to initialize.

## Best Possible Solution

Make an explicit support-boundary call: either land a guarded browser-plugin WSL2 fallback with production-path tests, or document remote CDP/Xvfb/desktop-session as the supported path and close with that rationale.

## Reproduction Assessment

Yes, at source level. On current main, a local managed Linux profile with explicit headed mode and no inherited `DISPLAY` or `WAYLAND_DISPLAY` returns the no-display error before launch, and the spawn env has no WSL2 fallback.

## Solution Assessment

Unclear. The literal `DISPLAY=:0` fix is too broad for all WSL2 setups; the maintainable path is either a tightly guarded WSL2 fallback or a documented decision not to support synthetic display guessing.
