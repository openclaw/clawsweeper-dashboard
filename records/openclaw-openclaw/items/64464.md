## Summary

Keep open. Current main still handles only the adjacent Linux no-display case by falling back to headless for implicit starts and by returning an explicit no-display error for headed starts; it does not make visible WSL2 Chrome launches work by adding `DISPLAY=:0`. The related upstream PRs for this exact change are closed and unmerged, so this remains a narrow browser plugin bug rather than cleanup-ready.

## What This Changes

The issue asks OpenClaw's browser plugin to set `DISPLAY=:0` for WSL2 headed Chrome launches when Chrome is spawned without a display environment.

## Best Possible Solution

Add a browser-plugin-scoped WSL2 headed launch path: for local managed Chrome starts only, when headed mode is explicitly selected and no display variable is inherited, detect WSL2 via the plugin SDK runtime helper and provide `DISPLAY=:0` to the Chrome child process while leaving headless, remote CDP, existing-session, and user-provided display values unchanged. Update the Linux/WSL docs and add focused spawn-env/config tests.

## Reproduction Assessment

Yes. The high-confidence reproduction is a code-level current-main path: local managed Linux profile, WSL2, `headless=false`, and no `DISPLAY`/`WAYLAND_DISPLAY` returns the explicit no-display error before spawn; the spawn env also lacks any WSL2 `DISPLAY=:0` fallback.

## Solution Assessment

Yes, with tighter scope than the original line-level suggestion. A WSL2-only, no-inherited-display, local-managed-headed fallback is the narrow maintainable solution; a generic all-Linux `DISPLAY=:0` injection would be riskier.
