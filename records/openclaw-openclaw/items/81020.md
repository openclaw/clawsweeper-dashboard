## Summary

Keep this open: the reported Windows daemon supervision failure is plausible and not implemented or superseded on current main, but the exact `P1D` creation path is not present in current source or the reported release source, so a maintainer should choose the Windows service contract before a fix is attempted.

## What This Changes

Request durable Windows Scheduled Task watchdog behavior so `openclaw onboard --install-daemon` does not lose automatic gateway restart after the task repetition duration expires.

## Best Possible Solution

Choose the native Windows service contract, then make install/update/doctor repair a durable gateway watchdog task with regression coverage and update the Windows service docs to describe the supported restart behavior.

## Reproduction Assessment

Unclear for the exact `P1D` origin: current main and the reported release source do not create that value. Source inspection does show a current repair gap because existing tasks are updated with `/Change /TR` only, so stale finite trigger settings would survive reinstall/update.

## Solution Assessment

Unclear. A one-line `P1D` to `P9999D` change does not map to current main, so the maintainable fix likely needs a choice between recreating/auditing task triggers and introducing a dedicated watchdog task.
