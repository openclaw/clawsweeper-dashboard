## Summary

Keep this open: the report describes a plausible Windows daemon-supervision gap, and current main does not provide enough evidence for any safe close reason. The exact `P1D` creation path is not present in source, but existing Scheduled Tasks are updated with `/Change /TR` only, so stale trigger/repetition settings would not be repaired by reinstall/update.

## What This Changes

The issue reports that Windows daemon installation leaves a Scheduled Task watchdog repetition duration that expires after one day, causing gateway auto-restart to stop after day one.

## Best Possible Solution

Decide the Windows service contract, then make install/update either recreate or repair durable gateway supervision settings, or add a dedicated watchdog task with regression coverage, docs, and live Windows proof.

## Reproduction Assessment

Unclear for the exact `P1D` origin because current main does not create that value. Source inspection does show a reproducible repair gap: existing Scheduled Tasks are updated with `/Change /TR` only, so stale finite repetition settings would survive reinstall/update.

## Solution Assessment

Unclear. Changing `P1D` to `P9999D` would only be the best fix if that creation path exists; current main uses `schtasks /SC ONLOGON`, so the maintainable solution likely needs a maintainer decision between trigger recreation/repair and a dedicated watchdog task.
