## Summary

Keep open: this points at a real Windows service-supervision gap, but current main and the reported release do not create the reported `P1D` repetition setting, and the existing-task update path intentionally preserves Task Scheduler settings. The remaining decision is a Windows service-contract/product choice, not a conservative cleanup close.

## What This Changes

Reports that Windows daemon install creates or retains a scheduled-task repetition duration that expires after one day, leaving crashed Gateways without automatic restart.

## Best Possible Solution

Choose the intended native Windows service contract, then make install/update either recreate or explicitly repair a durable Gateway watchdog trigger, or document that Windows Scheduled Tasks are logon-start only and point users to a supported watchdog path.

## Reproduction Assessment

Unclear. The exact `P1D` creation path is not present in current main or v2026.5.7 source, but source inspection does show that an existing task with stale finite repetition settings would be updated with `/Change /TR` only and not repaired.

## Solution Assessment

Unclear. Changing `P1D` to `P9999D` would be narrow only if OpenClaw currently created that trigger; current main uses `ONLOGON` plus preservation updates, so the safer fix needs maintainer choice between trigger repair/recreation and a dedicated watchdog task.
