## Summary

Keep open. The report is specific, current, and plausible against current main: cron announce runs intentionally expose the message tool, then fall back to runner-owned delivery when the tool did not verify a send to the target, so a WhatsApp target-resolution or source-delivery mismatch can be masked as a successful cron run. I did not find a merged fix or canonical issue that fully covers the wrong-destination WhatsApp group case.

## What This Changes

Issue reports that scheduled WhatsApp group announce jobs with tools enabled can fail a model-initiated message send, fall back to the owner DM, and still report delivered.

## Best Possible Solution

Make cron announce delivery enforce the configured explicit target: failed or off-target message-tool attempts should fall back only to that resolved target, and any target mismatch or fallback failure should mark the run not delivered with a clear diagnostic.

## Reproduction Assessment

Yes at source level for the tool-enabled announce path: current main keeps the message tool available, treats only verified target sends as satisfying delivery, and then falls back to runner-owned delivery. The exact WhatsApp group-to-DM reroute still needs a focused fixture or live reproduction.

## Solution Assessment

Unclear. The narrow direction is to enforce explicit cron announce targets and fail visibly on target mismatch; simply disabling all tools for announce jobs would be a broader behavior change than the report requires.
