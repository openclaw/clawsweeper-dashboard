## Summary

Keep open. The report is recent, concrete, and conflicts with the documented cron announce contract: explicit announce delivery should either reach the configured target or report not-delivered, not silently resolve to an owner DM with delivered=true.

## What This Changes

The issue reports cron announce jobs with enabled tools delivering WhatsApp group output to the owner DM while reporting successful delivery.

## Best Possible Solution

Cron announce should make the configured `delivery.channel` and `delivery.to` authoritative for fallback delivery; failed or unverified message-tool sends should never mark the run delivered or change the resolved destination.

## Reproduction Assessment

No, not from this read-only pass. The issue has concrete CLI steps and the current source has the relevant message-tool/direct-fallback path, but I did not establish a live WhatsApp current-main repro.

## Solution Assessment

Unclear until the exact failure mode is reproduced. The likely best fix is a bounded cron delivery/target-resolution correction, not a broad change that disables tools for announce jobs.
