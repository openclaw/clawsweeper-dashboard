## Label Justifications

- `P1`: The original issue described scheduled group messages going to the wrong or unverified destination while cron recorded successful delivery.
- `impact:message-loss`: The affected behavior was false-success or misrouted channel delivery for scheduled cron messages.

## Summary

Current main and the latest release now require explicit message-tool target evidence before cron announce suppresses direct fallback, so the reported false-success/misdelivery path is fixed by the merged target-evidence PR.

## What This Changes

Fix cron announce jobs so tool-enabled runs deliver to the configured target instead of silently treating targetless message-tool activity as successful group delivery.

## Best Possible Solution

Keep the merged explicit-target-evidence implementation and close this issue as fixed; new reports should include a fresh cron run log if a different channel still misdelivers.

## Reproduction Assessment

Yes. The pre-fix source path could treat `didSendViaMessagingTool=true` with empty target metadata as verified delivery, and the merged regression test now covers that exact shape on current main.

## Solution Assessment

Yes. Requiring explicit target evidence only for cron announce delivery is the narrow owner-boundary fix because it preserves legacy message-tool-owned flows while keeping cron direct fallback for unverified sends.
