## Label Justifications

- `P1`: The original report described scheduled messages going to the wrong recipient while the job reported successful delivery.
- `impact:message-loss`: The issue is about cron chat messages being misdelivered or silently not delivered to the intended group.

## Summary

Current main now makes cron announce delivery target-authoritative: failed or unverified message-tool sends do not satisfy delivery, recovered `Message failed` warnings stay diagnostic when final text exists, and fallback sends through the resolved configured target instead of silently becoming an owner DM.

## What This Changes

Cron announce jobs should deliver final text to the configured WhatsApp group with tools enabled, without silently falling back to an owner DM while reporting success.

## Best Possible Solution

Keep the current main behavior: cron announce may leave tools enabled, but final fallback delivery remains bound to the configured resolved target and delivery status reflects actual send success.

## Reproduction Assessment

No high-confidence failing current-main reproduction remains from this read-only pass. The report has concrete old-release steps, but current source now handles the reported message-warning and target-fallback paths.

## Solution Assessment

Yes. The current implementation fixes the delivery contract without disabling tools globally: it treats recovered message-tool failures as diagnostic and sends fallback output through the resolved cron target.
