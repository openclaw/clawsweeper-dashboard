## Summary

Keep open. Current main still lacks stable cron job IDs and duplicate-ID validation, but the proposed patch has three concrete merge blockers around Swift source compatibility, Gateway error mapping, and the required changelog entry.

## What This Changes

The PR adds optional cron job IDs across cron service creation, Gateway schema, CLI `cron add --id`, agent cron-tool recovery, docs, generated Swift protocol models, and tests, plus duplicate-ID rejection in `CronService.add`.

## Best Possible Solution

Land the additive stable cron-ID feature after preserving Swift protocol source compatibility, mapping duplicate custom IDs to a first-class Gateway `INVALID_REQUEST`, adding the Unreleased changelog entry with contributor credit, and validating the generated clients and cron surfaces together.

## Reproduction Assessment

Yes. Current-main source inspection reproduces the missing behavior: cron creation always generates a UUID, `add` has no duplicate guard, and the Gateway schema rejects top-level `id`; the PR blockers are directly reproducible from the diff and current handler/generator behavior.

## Solution Assessment

No. The service, CLI, Gateway schema, docs, and test direction is narrow and maintainable, but this is not the best merge shape until the Swift initializer, duplicate-error mapping, and changelog blockers are fixed.

  - body: Adding `id: String?` as a required initializer argument changes the public `CronAddParams(...)` call signature even though the wire field is optional.   - confidence: 0.9

  - confidence: 0.86

  - body: This exposes user-facing `openclaw cron add --id` and Gateway `cron.add` custom-ID behavior, but the PR files do not include `CHANGELOG.md`. Add one single-line Unreleased `### Changes` entry with contributor credit before merge.
