## Summary

Keep open. Current main still lacks custom cron job IDs, and this PR is a plausible implementation, but it has blocking Swift compatibility and Gateway error-mapping issues plus missing after-fix real behavior proof.

## What This Changes

Adds optional stable cron job IDs across the CLI, Gateway protocol, cron service, docs, generated Swift models, and tests, with case-insensitive duplicate-id rejection.

## Best Possible Solution

Converge on one implementation that adds validated create-time cron IDs through CLI, Gateway, service, generated clients, docs, and focused tests while preserving UUID defaults and first-class validation errors.

## Reproduction Assessment

Yes. Source inspection on current main shows UUID-only service creation, no Gateway `cron.add` `id` field, and no CLI `--id` flag; the PR-specific blockers are visible from the diff plus current Gateway and Swift generator code.

## Solution Assessment

No. The feature direction is narrow and maintainable, but this PR is not the best merge shape until Swift initializer compatibility and duplicate-id Gateway error mapping are fixed and real behavior proof is added.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve Swift initializer compatibility:** `apps/shared/OpenClawKit/Sources/OpenClawProtocol/GatewayModels.swift:4142`
  - body: Adding `id: String?` as a required initializer argument changes the public `CronAddParams(...)` call signature even though the wire field is optional. Default generated optional initializer parameters to `nil` or add a compatibility overload, then regenerate both Swift model copies so existing callers keep compiling.
  - confidence: 0.9
- **[P2] Map duplicate IDs to validation errors:** `src/cron/service/ops.ts:326`
  - body: The duplicate guard throws a plain `Error`, but `cron.add` only maps `TypeError` and `RangeError` from `context.cron.add` to `INVALID_REQUEST`. A duplicate ID through Gateway or CLI will bypass the promised validation response unless this error is mapped or thrown as a validation error.
  - confidence: 0.86
