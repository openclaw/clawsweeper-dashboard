## Summary

Keep open, but this PR is not merge-ready: it raises the Skill Workshop `apply` approval window to 300s without aligning the current Codex `skill_workshop` dynamic-tool watchdog, so the agent-tool path can still time out before the new approval window is useful. The PR also still lacks real large-proposal approval-flow proof.

## What This Changes

This PR adds an `apply`-only `timeoutMs: 300_000` to Skill Workshop lifecycle approval requests while leaving `reject` and `quarantine` on the default approval timeout.

## Best Possible Solution

Choose one timeout policy before merge: either keep the approval window within the selected tool watchdog, or raise every affected `skill_workshop` watchdog above the approval ceiling with focused tests and real approval-flow proof.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current main uses a 120s approval default for Skill Workshop lifecycle approval, returns `Approval timed out` on expiry, and Codex `skill_workshop` dynamic tools still have a 90s watchdog. I did not run a live large-proposal approval flow in this read-only review.

## Solution Assessment

No. Raising `apply` approval to 300s is a plausible response to the 120s approval-window complaint, but it is not the best fix as submitted because it does not align the current Codex watchdogs or provide real approval-flow proof.
