## Summary

Keep open. Current main still does not implement the central agent-prefixed Slack thread parser fix, and the paired same-author issue remains open. The PR is a relevant implementation candidate, but the proposed parser change still has a blocking edge-case bug for agent-prefixed main-session keys whose third segment is also a channel surface.

## What This Changes

The PR changes session group-key parsing to skip agent prefixes, adjusts restart-sentinel delivery-context fallback ordering, and adds regression tests for agent-prefixed Slack thread routing plus incomplete delivery contexts.

## Best Possible Solution

Keep the PR open as the implementation candidate, but revise the parser to reuse or mirror parseRawSessionConversationRef() so agent-prefixed keys are only treated as provider-encoded conversation keys when they include an explicit group/channel kind. Rebase the restart-sentinel fallback change onto current main and keep parsed thread targets ahead of stale base-session routes while preserving useful partial hints such as accountId.

## Reproduction Assessment

Yes. The static reproduction path is high confidence: resolveGroupSessionKey() on current main with From=agent:builder:slack:channel:C123:thread:..., Provider=webchat, and ChatType=channel reads head=agent and can return a webchat-normalized group key. The PR finding reproduces similarly with From=agent:main:webchat, where the proposed startIndex makes webchat the head but leaves the id empty.

## Solution Assessment

No, not as-is. The PR targets the right surfaces, but the parser change should be guarded by an explicit group/channel segment or by parseRawSessionConversationRef(); the restart-sentinel part should then be reviewed after rebase against the current routing code.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Guard agent-prefixed parsing with a group kind:** `src/config/sessions/group.ts:77`
  - body: The new `startIndex = 2` branch skips `agent:<id>:` for every three-segment agent key before checking whether the remaining body is really a group/channel key. For `From = "agent:main:webchat"` with channel chat type, `headIsSurface` becomes true, `id` is empty, and the function returns `null` instead of falling back to the provider context, so channel-scoped main sessions can lose group routing. Require an explicit `group`/`channel` segment after the provider, or reuse the existing raw session conversation parser.
  - confidence: 0.9
