## Summary

Keep open: current main still has source-visible agent-prefixed session parser drift, and this PR is a plausible paired implementation for the open report, but the patch still has a parser bug and lacks real Slack/A2A proof.

## What This Changes

The PR changes group-session key parsing and restart-sentinel delivery-context merging, with regression tests, to preserve Slack thread routing for agent-prefixed A2A sessions.

## Best Possible Solution

Land a rebased narrow fix that uses guarded shared session-key parsing, keeps current route facts ahead of base fallback, preserves thread-target policy unless intentionally changed, and includes focused tests plus redacted Slack runtime proof.

## Reproduction Assessment

Yes, at source level. Current main can read agent:builder:slack:channel:C123:thread:T as head=agent and fall back to Provider=webchat, while the PR diff introduces a separate agent:main:webchat parser edge case.

## Solution Assessment

No, not as-is. The parser should skip an agent prefix only when the remaining key has an explicit group/channel shape, or reuse parseRawSessionConversationRef(), and the branch still needs rebase plus real Slack proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Guard agent-prefixed parsing with an explicit group kind:** `src/config/sessions/group.ts:77-78`
  - body: The new startIndex branch treats any agent:<id>:<surface> key as provider-encoded. With From="agent:main:webchat" and ChatType="channel", head becomes webchat, the id slice is empty, and the resolver returns null instead of falling back to provider-context channel routing. Only skip the agent prefix when the remaining key has provider plus group/channel plus id, or reuse parseRawSessionConversationRef().
