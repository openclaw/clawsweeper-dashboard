## Summary

Keep open: current main still shows the source-visible agent-prefixed Slack parser drift, but this PR is not ready because the branch introduces a parser edge case and still lacks real Slack/A2A after-fix proof.

## What This Changes

This PR changes group-session key parsing, restart-sentinel delivery-context merging, and related Vitest coverage to preserve Slack thread routing for agent-prefixed A2A sessions.

## Best Possible Solution

Land a rebased narrow fix that uses guarded shared session-key parsing, keeps explicit/current routing facts ahead of base fallback, preserves current `sessions_send` thread-target policy unless maintainers intentionally change it, and includes focused regression tests plus redacted Slack runtime proof.

## Reproduction Assessment

Yes, at source level. Current main can read `agent:builder:slack:channel:C123:thread:T` with `head=agent`, miss the encoded Slack provider, fall back to `Provider=webchat`, and persist that channel through group session metadata.

## Solution Assessment

No, not as-is. The parser should only skip an `agent:<id>` prefix when the remaining key has provider plus explicit `group` or `channel` shape, preferably by reusing or mirroring `parseRawSessionConversationRef()`, and the restart fallback should be reviewed against current thread-target policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Guard agent-prefixed parsing with explicit group shape:** `src/config/sessions/group.ts:77-78`
  - body: The new `startIndex` branch skips any `agent:<id>:...` key before checking for a group/channel segment. For `From="agent:main:webchat"` with `ChatType="channel"`, `startIndex` points at `webchat`, the id slice is empty, and the resolver returns `null` instead of falling back to provider-context channel routing. Only skip the prefix when the remaining body has provider + group/channel + id, or reuse `parseRawSessionConversationRef()`.
  - confidence: 0.9
