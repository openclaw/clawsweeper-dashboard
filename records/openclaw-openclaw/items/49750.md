## Summary

Keep open: the branch targets a real current-main session-routing bug, but the proposed parser change still has a blocking edge case and the external PR lacks required real Slack/A2A behavior proof.

## What This Changes

The PR updates group-session key parsing, restart-sentinel delivery-context merging, and related Vitest coverage to preserve Slack thread routing for agent-prefixed A2A sessions.

## Best Possible Solution

Land a rebased narrow fix that uses guarded shared session-key parsing, preserves current sessions_send thread-target policy unless maintainers intentionally change it, and includes focused regression tests plus redacted Slack/A2A runtime proof.

## Reproduction Assessment

Yes, at source level. Current main can read agent:builder:slack:channel:C123:thread:T with head=agent, miss the encoded Slack provider, fall back to Provider=webchat, and persist that channel through group session metadata.

## Solution Assessment

No, not as-is. The parser should only skip an agent:<id> prefix when the remaining body has provider plus explicit group/channel shape, preferably by reusing or mirroring parseRawSessionConversationRef(), and the restart fallback should be reviewed against current thread-target policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Guard agent-prefixed parsing with an explicit group shape:** `src/config/sessions/group.ts:77-78`

Only skip the prefix when the remaining body has provider + group/channel + id, or reuse the existing session-key parser.
