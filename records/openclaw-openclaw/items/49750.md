## Summary

Keep open: current main still has the source-visible Slack A2A routing bug, but this PR is not ready to merge because it still has a parser regression, lacks real Slack/A2A proof, and is now conflicting with main.

## What This Changes

The PR changes group-session key parsing, restart-sentinel delivery-context precedence, and related Vitest coverage to preserve Slack thread routing for agent-prefixed A2A sessions.

## Best Possible Solution

Land or replace this with a rebased narrow fix that uses guarded shared session-key parsing, preserves the intended restart/thread-target policy, and includes focused regression tests plus redacted Slack/A2A runtime proof.

## Reproduction Assessment

Yes, at source level. Current main can read From=agent:builder:slack:channel:C123:thread:T with Provider=webchat, miss the encoded Slack provider, and persist webchat through group session metadata.

## Solution Assessment

No, not as-is. The parser should only skip agent:<id> when the remaining body has provider plus explicit group/channel shape, preferably by reusing or mirroring parseRawSessionConversationRef(), and restart fallback precedence needs maintainer review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Guard agent-prefixed parsing with an explicit group shape:** `src/config/sessions/group.ts:77-78`
  - body: The new startIndex branch treats any agent:<id>:<surface> key as a provider-bearing group key before verifying that the remaining body contains group/channel plus an id. With From="agent:main:webchat" and ChatType="channel", headIsSurface is true, id is empty, and resolveGroupSessionKey returns null instead of falling back to provider-context channel routing. Only skip the prefix when the remaining body has provider + group/channel + id, or reuse parseRawSessionConversationRef().
