## Summary

This PR targets a real Discord status-probe performance problem, but it should stay open because the current branch is not mergeable as-is: it drops current voice/stage audit requirements, adds a test that timed out in CI, and still lacks real Discord status-probe proof.

## What This Changes

The PR adds bounded parallel Discord channel permission audits, a 30-second in-memory permission-context cache, and regression tests for ordered concurrency and cache reuse.

## Best Possible Solution

Refresh the Discord plugin patch on current main so status audits use bounded ordered concurrency and short per-token/guild permission-context reuse while preserving channel-type requirements, then prove it with deterministic tests and redacted real status-probe output.

## Reproduction Assessment

Yes, at source level: current main routes `channels status --probe` into Discord `auditAccount`, audits configured channels serially, and refetches bot/guild/member context per channel. The claimed real timeout is not live-reproduced because no redacted status-probe run is provided.

## Solution Assessment

No, not as-is. Bounded concurrency plus a short context cache is the right repair direction, but the branch must preserve voice/stage permission requirements, replace the hanging timer-based test, refresh against main, and add real behavior proof before merge.
