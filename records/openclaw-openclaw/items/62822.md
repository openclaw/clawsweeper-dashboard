## Summary

Keep open: this is a valid Discord status-probe performance fix direction, but the current PR head is conflicting, regresses current voice/stage audit requirements, has a CI timeout in the new audit test, and lacks required real behavior proof.

## What This Changes

The PR adds bounded parallel Discord permission audits, a 30-second in-memory bot/guild permission-context cache, and regression tests for ordered concurrency and cache reuse.

## Best Possible Solution

Refresh the Discord plugin patch on current main so status audits use bounded ordered concurrency and short per-token/guild permission-context reuse while preserving channel-type requirements, then prove it with deterministic tests and redacted real status-probe output.

## Reproduction Assessment

Yes, at source level: current main routes `channels status --probe` into Discord `auditAccount`, serially awaits each configured channel, and refetches bot/guild/member permission context per channel. The claimed real timeout is not live-reproduced in the PR because no redacted status-probe run is provided.

## Solution Assessment

No, not as-is. Bounded concurrency plus a short context cache is the right repair direction, but the branch must preserve voice/stage permission requirements, replace the hanging timer-based test, refresh against main, and add real behavior proof before merge.
