## Summary

Keep open: the Discord status-probe performance problem is still present on current main, but this PR is not ready to merge because it is stale against current audit semantics, its new concurrency test times out in CI, and it lacks after-fix real behavior proof.

## What This Changes

The PR adds bounded ordered concurrency for Discord channel permission audits, a 30-second in-memory Discord bot/guild permission-context cache, and regression tests for ordered audits plus cache reuse.

## Best Possible Solution

Refresh the Discord plugin change on current main so audits run with bounded ordered concurrency, permission context is reused briefly per token/guild, current channel-type requirements are preserved, and deterministic tests plus redacted real status-probe proof demonstrate the improvement.

## Reproduction Assessment

Yes. Source inspection on current main shows `channels status --probe` reaches Discord `auditAccount`, then serially awaits each configured numeric channel and refetches bot/guild/member context for every permission check.

## Solution Assessment

No, not as-is. Bounded concurrency plus a short permission-context cache is the right direction, but the branch must preserve current voice/stage audit semantics, replace the timer-dependent hanging test with deterministic synchronization, and add real behavior proof before merge.
