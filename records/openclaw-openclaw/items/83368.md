## Summary

Keep open: the PR fixes raw/canonical alias write and delete drift, but preserved aliases can still appear as duplicate visible session rows because the list path enumerates every store key without collapsing raw aliases.

## What This Changes

The PR preserves raw external gateway session aliases during canonicalization, synchronizes them on replacement writes, deletes them with the canonical key, and adds gateway regression tests.

## Best Possible Solution

Keep raw aliases as internal lookup aliases, synchronize and delete them across mutations, and collapse them from list/search/count output before landing.

## Reproduction Assessment

Yes, at source level. Current main canonicalizes a raw structured key into an agent-scoped key and prunes the raw candidate, while the PR preserves both keys and the remaining list path still enumerates every store entry.

## Solution Assessment

No, not yet. The alias-sync boundary is the right direction, but aliases need to stay internal to lookup/mutation paths so one logical session is not exposed twice.
