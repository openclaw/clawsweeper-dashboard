## Summary

Keep open: the PR fixes many raw/canonical alias write paths, but current PR head still has two source-reproducible P1 session-state freshness blockers. A canonical visible key can still ignore a newer hidden raw alias before mutation, and session listing hides raw aliases before choosing the freshest logical row.

## What This Changes

The PR adds shared gateway session-store alias helpers and routes session, compaction, resolver, node-event, and plugin host-hook writes through raw/canonical alias synchronization.

## Best Possible Solution

Treat each canonical key and its preserved raw aliases as one logical freshness group before every mutation and before session list search/filter/count/order/limit behavior, with raw-newer regression coverage.

## Reproduction Assessment

Yes. Seed an older canonical row plus a newer hidden raw alias for the same external key, then mutate or list by the canonical visible key; source inspection shows the PR head still ignores or drops the raw alias before freshness selection.

## Solution Assessment

No. The shared alias helper direction is maintainable, but the implementation must make alias-group freshness the unit of selection before mutation and list filtering to avoid stale canonical rows winning.
