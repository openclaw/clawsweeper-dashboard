## Summary

Keep this PR open, but do not merge it yet. The shared alias-helper direction is sound, but a visible canonical-key mutation can still overwrite a newer hidden raw alias because preserved aliases are synchronized after, not included in, freshness selection.

## What This Changes

The PR adds shared gateway session-store alias helpers and routes gateway mutations, compaction/checkpoint writes, resolver/node-event writeback, reset, and plugin host-hook session mutations through raw/canonical alias synchronization with regression coverage.

## Best Possible Solution

Keep the centralized alias helper approach, but select the freshest entry across canonical candidates and preserved raw aliases before any alias sync/write/delete, then lock that behavior with a raw-newer canonical-key regression.

## Reproduction Assessment

Yes, at source level. Current main has no raw-alias preservation, and the remaining PR regression follows directly when a canonical visible key is passed while the hidden raw alias has the newer updatedAt.

## Solution Assessment

No, not yet. Centralizing alias handling is the maintainable direction, but preserved aliases must participate in freshness selection before synchronization.

## Review Findings

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve raw aliases in freshness selection:** `src/gateway/session-utils.ts:945-949`
  - body: When callers use the visible canonical key, `target.storeKeys` contains only canonical variants, while the hidden raw alias is discovered separately in `preservedAliasKeys`. The new sync then copies the canonical entry over that alias, so if external inbound traffic updated the raw alias after the canonical row, a later `sessions.patch`/reset/compact/plugin mutation from the listed key can lose the newer session or plugin state. Include preserved aliases in the freshest-match selection before syncing and add a raw-newer canonical-key regression.
  - confidence: 0.91
