## Summary

Keep open. The PR addresses the right gateway session-state bug and has strong proof, but it still leaves a merge-blocking alias lifecycle gap when clients mutate the canonical row that is now the only visible list row.

## What This Changes

The PR preserves raw external gateway session aliases during canonicalization, syncs aliases across several write/delete/reset paths, hides preserved aliases from session lists, and adds gateway regression tests.

## Best Possible Solution

Make raw/canonical alias discovery bidirectional so both raw-key and canonical-key mutations sync or delete the same alias set, with regression coverage for canonical-key patch, reset, delete, and list flows.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: seed raw+canonical aliases, list sessions to get only the canonical row, then mutate that canonical key; the PR only returns preserved aliases for raw-key migration, so the raw row can remain stale.

## Solution Assessment

No, not yet. The shared helper direction is the right fix, but the solution needs canonical-key alias discovery before the list-collapse behavior is safe to merge.
