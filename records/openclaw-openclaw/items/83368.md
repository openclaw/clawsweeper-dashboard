## Summary

Keep open. The PR is active and useful, and the terminal proof is sufficient for this backend path, but two P1 session-state blockers remain: hidden raw aliases are still excluded from freshness selection and are filtered out of list/search/active handling before the alias group is collapsed.

## What This Changes

The PR adds shared gateway session-store alias helpers and routes gateway session mutations, compaction/checkpoint writes, reset, resolver sync, node events, and plugin host-hook writes through raw/canonical alias synchronization with regression coverage.

## Best Possible Solution

Keep the shared alias contract, but treat raw and canonical aliases as one freshness group before mutation and before list/search/active/count behavior, with raw-newer regressions.

## Reproduction Assessment

Yes, source-level on the PR head. A store with a visible canonical key and a newer hidden raw alias still runs migration freshness over `target.storeKeys` only, and list filtering drops the raw alias before search/active/count can use its fresher metadata.

## Solution Assessment

No, not as-is. The shared helper direction is maintainable, but the fix needs alias-group freshness selection for writes and list collapse before filters to avoid replacing one session split with another stale-state path.
