## Best Possible Solution

Keep the shared alias-helper direction, but treat canonical and preserved raw aliases as one logical alias group for freshness selection before every mutation and before list filtering/search/count/order/limit.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create canonical and raw alias entries for the same logical session with the raw alias newer, then mutate or list by the visible canonical key.

## Solution Assessment

No, not yet. The helper boundary is the right direction, but syncing aliases after selecting a stale canonical entry is not enough; the alias group needs one freshness decision before reads and writes.
