## Summary

The PR is useful and the backend proof is sufficient, but the latest head still has session-state blockers: canonical-key mutation can sync stale canonical data over a fresher hidden raw alias, and session listing hides raw aliases before choosing the freshest logical row.

## What This Changes

The branch adds shared gateway session-store alias helpers, routes gateway/session/plugin mutations through them, and expands regression coverage for raw/canonical external session aliases.

## Best Possible Solution

Keep the shared alias helper direction, but make canonical/raw alias groups choose one freshest entry before any sync, mutation, search, active filtering, ordering, count, or limit behavior.

## Reproduction Assessment

Yes, source-level for the review findings: a store with a canonical key and a fresher hidden raw alias is enough to see that migration selects only `target.storeKeys` and list filtering drops the raw alias before freshness-sensitive filters. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No for the current patch shape: the shared alias contract is the right direction, but it must choose the freshest alias-group entry before sync and list filtering. The safer repair is a focused freshness/collapse fix with raw-newer regression coverage.
