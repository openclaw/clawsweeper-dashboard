## Summary

Keep open: the shared alias direction is useful and the backend proof is strong, but canonical-key flows still can ignore a fresher hidden raw alias before syncing or listing, leaving session-state blockers before merge.

## What This Changes

The PR adds shared gateway raw-session alias helpers and routes session mutation, compaction/checkpoint, resolver, node-event, and plugin host-hook paths through raw/canonical alias synchronization.

## Best Possible Solution

Treat each canonical key and its preserved raw aliases as one logical freshness group before every mutation and before list/search/filter/count/order/limit behavior, with raw-newer regressions.

## Reproduction Assessment

Yes, source-reproducible. Seed a store with an older canonical entry and a newer hidden raw alias for the same external key, then mutate by the canonical visible key or list/search/filter the store; the PR head still does not choose the raw alias as the freshest logical row first.

## Solution Assessment

No, not yet. The shared alias-helper approach is the right direction, but the fix must make alias groups freshness-aware before sync and listing rather than only copying or hiding alias rows afterward.
