## Summary

Keep open: the PR is useful and the shared alias-helper direction is right, but the latest diff still has two blocking alias freshness bugs. Preserved raw aliases are rediscovered, yet canonical-key writes and session listing still do not treat the raw/canonical pair as one freshness group before mutation or visibility filtering.

## What This Changes

The PR adds shared gateway raw-session-alias helpers and routes gateway session mutations, compaction/checkpoint paths, resolver sync, node events, and plugin host-hook state through raw/canonical alias synchronization.

## Best Possible Solution

Keep the shared alias-helper approach, but treat each canonical key and its preserved raw aliases as one logical freshness group before every mutation and before list/search/count/order/limit behavior.

## Reproduction Assessment

Yes, source-reproducible. Seed a store with an older canonical entry and a newer hidden raw alias for the same external key, then mutate by the canonical visible key or list/search/count the store; the diff shows the raw alias is not part of freshness selection before sync/filtering.

## Solution Assessment

No. The helper boundary is maintainable, but the implementation must select the freshest alias-group entry before syncing writes and before visible list filtering to avoid stale or destructive behavior.
