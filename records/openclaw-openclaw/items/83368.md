## Summary

Keep open: the shared alias direction is useful and the backend proof is sufficient, but the current head still has a P1 session-state bug because canonical-key migration selects freshness only from canonical target keys before syncing rediscovered raw aliases.

## What This Changes

The branch adds shared gateway session-store alias helpers and routes gateway/session, compaction/checkpoint, resolver/node-event, reset, and plugin host-hook mutations through them with regression tests.

## Best Possible Solution

Keep the shared alias contract, but choose the freshest raw/canonical alias entry before hiding or syncing aliases so old split stores upgrade without losing state.

## Reproduction Assessment

Yes, at source level. On PR head, a canonical visible-key mutation computes preserved raw aliases but selects the freshest entry only from target.storeKeys, so a newer hidden raw alias is not considered before alias sync.

## Solution Assessment

No, not yet. Centralizing alias handling is the right shape, but preserved raw aliases must participate in freshness selection before synchronization and before list-style canonical follow-ups mutate state.
