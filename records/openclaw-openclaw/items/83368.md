## Summary

Keep open: the PR is useful and well-scoped, but it is not merge-ready because canonical-key mutation and list filtering can still ignore a newer hidden raw alias before selecting the logical session entry.

## What This Changes

The PR adds shared gateway session-store alias helpers and routes gateway, compaction, resolver, node-event, and plugin host-hook writes through raw/canonical alias synchronization.

## Best Possible Solution

Treat each canonical key and its preserved raw aliases as one logical freshness group before every mutation and before list/search/filter/count/order/limit behavior, with raw-newer regressions.

## Reproduction Assessment

Yes. Seed an older canonical entry plus a newer hidden raw alias, then mutate or list by the canonical visible key; source inspection shows the PR head still ignores or drops the raw alias before freshness selection.

## Solution Assessment

No. The shared alias-sync helper is the right direction, but the patch must choose the freshest alias-group entry before writes and list filtering to be safe.
