## Summary

This PR should stay open: the shared raw/canonical alias direction is useful and the supplied backend proof is strong, but canonical visible-key writes can still overwrite newer hidden raw alias state because freshness selection ignores preserved aliases before syncing.

## What This Changes

The PR adds shared gateway session-store alias helpers and routes session mutations, compaction, reset, resolver, node-event, and plugin host-hook writes through them with regression coverage.

## Best Possible Solution

Keep the shared alias contract, but choose the freshest entry across canonical target keys and preserved raw aliases before any alias sync, replacement write, prune, reset, patch, or compaction follow-up.

## Reproduction Assessment

Yes, source-level. Seed a store with an older canonical entry and a newer hidden raw alias, then call migrateAndPruneGatewaySessionStoreKey with the canonical key returned by listing; the PR head discovers the raw alias but selects freshness only from target.storeKeys.

## Solution Assessment

No. The shared helper direction is maintainable, but it must make freshness selection alias-aware before syncing or it can replace the newer raw state with stale canonical state.
