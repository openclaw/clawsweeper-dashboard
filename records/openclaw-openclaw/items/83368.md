## Summary

Keep this PR open: the shared alias contract and regression coverage are useful, and the supplied terminal proof is strong, but PR head still has two P1 session-state blockers where fresher hidden raw-alias state can be overwritten or hidden when callers use the visible canonical key.

## What This Changes

The PR adds gateway session-store alias helpers and routes gateway mutations, compaction/checkpoint writes, resolver sync, node-event touches, and plugin host-hook writes through raw/canonical session-alias synchronization with tests.

## Best Possible Solution

Keep the shared alias contract, but treat raw and canonical aliases as one freshness group before any mutation and before list/search/active/count collapse.

## Reproduction Assessment

Yes for the PR findings, source-level: a store with the same logical canonical/raw alias pair and a newer hidden raw entry is enough to see that canonical-key mutation chooses only `target.storeKeys`, then syncs stale canonical state over the alias. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No, not yet: the shared helper is the right direction, but the PR must collapse alias groups by freshness before writes and before visible list filtering. That is narrower than a product redesign and should be covered by raw-newer regressions.
