## What This Changes

## Best Possible Solution

Keep the shared alias contract, but make freshness selection include preserved raw alias keys before any write, sync, prune, delete, or list-style canonical follow-up mutates state.

## Reproduction Assessment

Yes, source-level: create a store with an older canonical entry and a newer hidden raw alias, then call `migrateAndPruneGatewaySessionStoreKey` with the canonical key returned by listing. The current PR head discovers the raw alias but selects freshness only from `target.storeKeys`, so the newer raw state is not retained before alias sync.

## Solution Assessment

No, not yet. Centralizing alias handling is the right shape, but preserved raw aliases must participate in freshness selection before synchronization and before canonical-key follow-up mutations.
