## Summary

Keep open: current main and the latest release still have the source-reproducible active-run metadata gap, and two open PRs now own candidate fixes rather than a merged implementation.

## What This Changes

The issue reports that WhatsApp with `messages.tts.auto: "inbound"` can answer steered inbound audio as plain text when the audio arrives during an already-active reply run.

## Best Possible Solution

Keep this issue as the canonical user report until the operation-local fix in https://github.com/openclaw/openclaw/pull/95596, or an equivalent maintainer-chosen fix, lands and proves the active-steer audio-to-voice path.

## Reproduction Assessment

Yes. Source inspection shows a steered audio inbound records `currentInboundAudio`, the active steer queue call does not carry it, and final inbound-auto TTS still gates on the original text message context.

## Solution Assessment

Yes. The operation-local marker in the linked maintainer PR is the narrowest visible fix because it carries the transient audio fact to final dispatch without adding durable session state; a wrapper-only TTS gate would not fix the missing metadata handoff.
