## Summary

Keep open: current main still appears to compute Auto-TTS `inboundAudio` from the original dispatch context while active-run steering forwards only prompt text, so the documented voice-in/voice-out behavior can still be missed in the reported text-then-audio timing.

## What This Changes

The issue reports that WhatsApp `messages.tts.auto: "inbound"` can deliver text instead of voice when an audio inbound is steered into an already active session run.

## Best Possible Solution

Preserve steered inbound media/reply-origin metadata for the active run so any audio consumed by `steer` can authorize `tts.auto: "inbound"` at final delivery without adding a WhatsApp-specific special case.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection gives a medium-confidence path: a text-started active run can accept a later audio steer while final TTS gating still sees the original context's `inboundAudio=false`.

## Solution Assessment

Yes. The narrow maintainable fix is to carry steered inbound media state into the active run; adding another default-dispatch pre-gate would not solve stale `false` metadata because the speech runtime already performs the inbound gate.
