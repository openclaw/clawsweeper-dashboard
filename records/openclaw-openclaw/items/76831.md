## Summary

Keep open: current main still appears to let a text-originated active run consume later WhatsApp audio as steering text while final Auto-TTS delivery keeps the original non-audio dispatch context.

## What This Changes

A WhatsApp bug report asks that `messages.tts.auto: "inbound"` preserve voice replies when an audio DM is steered into an already active session run.

## Best Possible Solution

Preserve a steered inbound audio/reply-origin signal through the active run and use it at dispatch, preferably as sticky consumed-inbound metadata, without adding a config knob or changing provider defaults.

## Reproduction Assessment

No high-confidence live reproduction was established here. Source inspection gives a medium-confidence path: WhatsApp audio metadata reaches dispatch normally, but default `steer` injects only prompt text into the active run while final TTS uses the original text message's `inboundAudio=false` context.

## Solution Assessment

Yes, the best repair is to carry steered inbound media/reply-origin metadata into active-run delivery. Adding only another default-dispatch pre-gate would not solve the stale-context case because the speech runtime already gates `inbound` mode on `inboundAudio`.
