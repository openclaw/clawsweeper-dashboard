## Summary

Keep open: current main still has a source-visible active-steer metadata gap where the second audio turn can be injected into the active run as text while final reply TTS stays keyed to the original non-audio inbound context.

## What This Changes

The issue asks OpenClaw to preserve inbound-audio/reply-origin metadata for WhatsApp `messages.tts.auto: "inbound"` when an audio message is steered into an already-active session run.

## Best Possible Solution

Propagate the steered inbound's audio/reply-origin facts into the active run's dispatch/TTS decision, preferably as a sticky current-turn audio fact plus focused tests for text-then-audio active session steering.

## Reproduction Assessment

Yes for a source-level reproduction: active streaming `steer` forwards only prompt text/options, while `auto: "inbound"` TTS requires `inboundAudio: true`. I did not run a live WhatsApp/TTS round trip in this read-only pass.

## Solution Assessment

Yes, the maintainable fix is to thread existing current-inbound audio/reply-origin metadata through active steering rather than adding a WhatsApp-specific fallback or new config option.
