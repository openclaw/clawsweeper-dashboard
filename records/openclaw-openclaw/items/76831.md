## Summary

Keep open: current main still appears to derive `inboundAudio` from the original dispatch context, so a text-started active run can consume a later audio steer while final delivery still sees a text origin for Auto-TTS gating.

## What This Changes

Ensure WhatsApp `messages.tts.auto: "inbound"` still emits a voice reply when an audio inbound is steered into an already-active run after an earlier text inbound.

## Best Possible Solution

Carry the steered inbound's audio/reply-origin metadata into the active run, either as a sticky `inboundAudio` signal over consumed inbounds or as an updated active reply-origin record, with regression coverage for text-then-audio steer timing.

## Reproduction Assessment

No live WhatsApp reproduction was established in this read-only review. Source inspection gives a medium-confidence path: the active steer injects only prompt text while final TTS gating still depends on the original dispatch context's `inboundAudio`.

## Solution Assessment

Yes, keeping this open is the right outcome. The maintainable fix should preserve steered inbound media metadata for dispatch rather than adding a redundant session-only TTS pre-gate in default dispatch.
