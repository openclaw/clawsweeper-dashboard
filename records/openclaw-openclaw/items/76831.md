## Summary

Keep open: current main still appears to let a text-started active run consume later WhatsApp audio through steering while final TTS dispatch evaluates the original text context, so the documented voice-in/voice-out behavior can still be missed.

## What This Changes

The issue requests that WhatsApp `messages.tts.auto: "inbound"` preserve inbound-audio reply metadata when an audio message is steered into an already-active session run.

## Best Possible Solution

Carry steered inbound audio or reply-origin metadata into the active run, preferably as a sticky consumed-inbound audio signal for final dispatch, with focused regression coverage for text-then-audio steering.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection gives a medium-confidence path: default `steer` injects the audio transcript into the active run while final TTS dispatch still evaluates the original text context's `inboundAudio=false`.

## Solution Assessment

Yes, the maintainable direction is to preserve steered inbound media metadata for final delivery. Adding only another default-dispatch pre-gate would not fix the stale `inboundAudio` value that suppresses synthesis.
