## Summary

Keep open: current main still has a source-level path where a steered WhatsApp audio message can be consumed by the active run while final TTS delivery still evaluates the original text message context, leaving `inboundAudio` false for `auto: "inbound"`.

## What This Changes

The issue reports that WhatsApp audio steered into an already-running session can be answered as plain text despite `messages.tts.auto: "inbound"`.

## Best Possible Solution

Carry a steered inbound audio signal into the active run's delivery context, preferably as sticky consumed-inbound metadata, and cover text-then-audio steering with a deterministic regression test.

## Reproduction Assessment

No live WhatsApp reproduction was established here. Source inspection gives a medium-confidence reproduction path: a text-originated active steer run can consume a later audio prompt while final delivery still uses the original context's `inboundAudio=false`.

## Solution Assessment

Yes, the narrow maintainable fix is to preserve steered inbound media metadata for dispatch; adding only another default-dispatch pre-gate would not address the stale `inboundAudio` value because the speech runtime already performs that gate.
