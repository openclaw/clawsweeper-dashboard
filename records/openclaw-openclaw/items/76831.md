## Summary

Keep open: current main still has the source path where a WhatsApp audio message steered into an active run can affect the prompt while final Auto-TTS still gates on the original text-originated dispatch context.

## What This Changes

The issue reports that WhatsApp `messages.tts.auto: "inbound"` can deliver a text reply instead of voice when an audio message is steered into an already-active session run.

## Best Possible Solution

Carry steered inbound audio or reply-origin metadata into the active run, preferably as a sticky `inboundAudio` signal or explicit active reply-origin update, and cover text-then-audio steering with a deterministic regression.

## Reproduction Assessment

No, there is not a high-confidence live reproduction from this review. Source inspection gives a medium-confidence path: `steer` injects only the audio transcript prompt into the active run while final TTS still gates on the original dispatch context's `inboundAudio`.

## Solution Assessment

Yes, metadata propagation is the best fix direction. Adding another dispatch-local `inboundAudio` pre-gate would not solve stale `inboundAudio=false` because the speech runtime already performs that gate.
