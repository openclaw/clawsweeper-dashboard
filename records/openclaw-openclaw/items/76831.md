## Summary

Keep open: current main still appears to let a text-originated active run consume a later WhatsApp audio steer while final Auto-TTS delivery is gated from the original non-audio dispatch context.

## What This Changes

The issue requests that WhatsApp replies under `messages.tts.auto: "inbound"` remain voice replies when an audio inbound is steered into an already-active text-originated run.

## Best Possible Solution

Carry steered inbound media/reply-origin state into the active run, preferably as a sticky `inboundAudio` signal or updated active reply-origin metadata, with deterministic regression coverage for text-then-audio steering.

## Reproduction Assessment

No live WhatsApp reproduction was established in this read-only review. Source inspection gives a medium-confidence reproduction path: a default `steer` run can inject the later audio transcript while final dispatch still sees the original text context's `inboundAudio=false`.

## Solution Assessment

Yes, the best fix is to preserve steered inbound audio metadata through the active-run dispatch boundary rather than adding a redundant pre-gate in default dispatch. The runtime already enforces the `inbound` gate correctly when it receives the right signal.

## Repair Work Prompt

Fix the WhatsApp/default-dispatch Auto-TTS `inbound` race where a text-originated active run consumes a later audio inbound via default `messages.queue.mode=steer`, but final delivery still uses the original non-audio dispatch context and skips TTS. Keep the fix in the auto-reply/queue/dispatch ownership boundary: carry steered inbound audio/reply-origin metadata into the active run, either as sticky `inboundAudio` over consumed inbounds or updated active reply-origin metadata. Do not add a new config option, do not special-case WhatsApp in core if a generic queue/dispatch seam is sufficient, and do not weaken the speech runtime's `autoMode === "inbound" && !inboundAudio` gate. Add focused regression coverage for text-then-audio steering, plus any WhatsApp preflight/dispatch tests needed to prove audio metadata still reaches TTS. Add a changelog fix entry because this is user-facing.
