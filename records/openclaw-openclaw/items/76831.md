## Summary

Keep open: current main still appears to gate Auto-TTS from the original dispatch context, while active steering only queues prompt text into the existing run. That leaves a plausible current-main path where a text-started run consumes a later WhatsApp audio steer but final delivery still sees `inboundAudio=false`.

## What This Changes

Ensure WhatsApp/default steer-mode replies honor `messages.tts.auto: "inbound"` when a later steered inbound message is audio.

## Best Possible Solution

Carry steered inbound audio/reply-origin metadata into the active run so final dispatch sees the audio input that shaped the reply, with regression coverage for text-then-audio steering.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection gives a medium-confidence current-main path: default steer injects only prompt text while final TTS dispatch still gates on the original context's `inboundAudio`.

## Solution Assessment

Yes, the best fix appears to be preserving steered inbound media/reply-origin metadata for dispatch. Adding only another default-dispatch pre-gate would not turn stale `inboundAudio=false` into a voice reply because the speech runtime already enforces that gate.
