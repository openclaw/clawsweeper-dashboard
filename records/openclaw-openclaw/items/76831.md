## Summary

Keep this open: current main still has a source-provable gap where streaming `steer` mode queues only the steered prompt into the active embedded run, while final TTS dispatch still gates `inbound` mode from the original run context's `inboundAudio` value.

## What This Changes

The issue reports that WhatsApp can deliver a text reply instead of voice when an audio inbound is steered into an already-active session with `messages.tts.auto: "inbound"`.

## Best Possible Solution

Preserve the steered inbound's audio/reply-origin metadata through the active streaming run so final dispatch can synthesize voice when any consumed inbound that shaped the reply was audio.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction path: current main drops the steered inbound's audio metadata before final TTS dispatch, and the runtime gate requires `inboundAudio === true`. No live WhatsApp repro was run during this read-only review.

## Solution Assessment

No for the wrapper-only fix suggested as option one; the better fix is to carry active steered inbound audio/reply-origin metadata into the final dispatch decision without adding a channel-specific special case.
