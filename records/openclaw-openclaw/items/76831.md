## Summary

Keep open: current main still snapshots `inboundAudio` from the run-originating dispatch context while active-run steering queues only text, so the reported WhatsApp text-then-audio race remains a plausible documented-behavior bug.

## What This Changes

The issue asks for WhatsApp auto-TTS `inbound` replies to remain voice when an audio message is steered into an already-active run that began from text.

## Best Possible Solution

Carry steered inbound media/reply-origin state into the active run so dispatch sees the audio message that was consumed by the model, with a regression for text-started steering followed by audio.

## Reproduction Assessment

No, there is not a high-confidence live reproduction from this review. Source inspection gives a medium-confidence path: default `steer` injects the audio transcript into the active run while final TTS delivery still uses the original text context's `inboundAudio=false`.

## Solution Assessment

Yes, the narrow maintainable fix is to preserve steered inbound media/reply-origin metadata for dispatch. Adding only another default-dispatch pre-gate would not solve the stale metadata path because the speech runtime already gates on `inboundAudio`.
