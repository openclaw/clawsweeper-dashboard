## Summary

Keep open: current main still appears to preserve the reported source path where a steered WhatsApp audio message can reach the active Codex run as prompt text while final Auto-TTS still gates on the original text-originated dispatch context.

## What This Changes

The issue reports that WhatsApp `messages.tts.auto: "inbound"` can reply with text instead of voice when an audio DM is steered into an active text-originated session run.

## Best Possible Solution

Carry steered inbound audio or reply-origin metadata into the active run, preferably as a sticky `inboundAudio` signal or explicit active reply-origin update, and lock it with a deterministic text-then-audio steering regression.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection gives a medium-confidence path: current `steer` injects prompt text into the active run while final TTS delivery still gates from the original dispatch context's `inboundAudio`.

## Solution Assessment

Yes, metadata propagation is the best fix direction. Adding another dispatch-local TTS pre-gate would not fix a stale `inboundAudio=false` value because the speech runtime already performs the inbound gate.
