## What This Changes

The issue asks OpenClaw to preserve audio-origin TTS metadata when a WhatsApp voice note is steered into an already-active text-originated run.

## Best Possible Solution

Carry or accumulate the active steered inbound's audio/reply-origin metadata into final default dispatch, then cover the text-origin plus audio-steer race with focused regression tests.

## Reproduction Assessment

Unclear for a live WhatsApp timing repro in this review; source inspection gives a deterministic test shape: active text-origin run, later audio inbound accepted through steer mode, and final dispatch still gating Auto-TTS from the original context.

## Solution Assessment

Yes, the best fix is to propagate or sticky-OR audio-origin metadata across the steer boundary. Mirroring the ACP wrapper gate alone is not sufficient because the shared TTS runtime already gates on `inboundAudio`.
