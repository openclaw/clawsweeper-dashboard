## Summary

Keep open: current main still gates auto-inbound TTS from the original dispatch context, while active steering only feeds prompt text from the later audio inbound into the running turn, so the reported text-then-audio race remains a plausible existing-behavior bug.

## What This Changes

The issue requests that WhatsApp replies stay voice-delivered under `messages.tts.auto: "inbound"` when an audio inbound is steered into an already-active session run.

## Best Possible Solution

Track consumed-inbound media state across active steering and use it for final TTS/reply-origin decisions, with a focused text-then-audio steer regression on the default dispatch path.

## Reproduction Assessment

No; there is not a high-confidence live reproduction from this review. Source inspection gives a medium-confidence path: text-origin dispatch computes `inboundAudio=false`, active steer queues only later audio prompt text, and speech runtime blocks inbound TTS unless `inboundAudio` is true.

## Solution Assessment

Yes; keeping this open is the right outcome. The narrow maintainable fix is to preserve consumed-inbound audio/reply-origin metadata through active steering rather than adding a redundant default-dispatch pre-gate.
