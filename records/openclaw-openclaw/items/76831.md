## Summary

Keep open: current main still appears to gate Auto-TTS on the original finalized dispatch context, while active steering only injects prompt text into the running session, so a text-originated WhatsApp run can consume later audio content but still deliver the final reply as text.

## What This Changes

The issue asks OpenClaw to preserve inbound audio metadata for WhatsApp `messages.tts.auto: "inbound"` when an audio message is steered into an already-active run.

## Best Possible Solution

Carry the steered inbound's media/TTS-origin metadata into the active run, either as a sticky `inboundAudio` signal or updated active reply-origin record, with regression coverage for text-then-audio steering.

## Reproduction Assessment

No high-confidence live reproduction was established here. Source inspection gives a medium-confidence reproduction path: start a text-originated active run, steer a WhatsApp audio inbound into it, and final dispatch still evaluates `inboundAudio` from the original text context.

## Solution Assessment

Yes, keeping this open is the right outcome. The narrow maintainable fix is to propagate consumed steered inbound audio metadata into dispatch, not to add another default-dispatch pre-gate that still reads the wrong origin.
