## Summary

Keep open: current main still has a source-reproducible active-run metadata gap where a steered audio inbound can shape the assistant reply while final TTS dispatch still evaluates `auto: "inbound"` against the original text-run context.

## What This Changes

The issue requests that WhatsApp `messages.tts.auto: "inbound"` replies remain voice replies when an audio message is steered into an already-active session run.

## Best Possible Solution

Preserve steered inbound audio and reply-origin metadata through the active run so final dispatch can synthesize voice when any consumed inbound that shaped the reply was audio.

## Reproduction Assessment

Yes for a high-confidence source-level path: current main drops the steered inbound's audio metadata before final TTS dispatch, and the runtime gate requires `inboundAudio === true`. No live WhatsApp repro was run in this read-only review.

## Solution Assessment

No for a wrapper-only inboundAudio gate: current main already gates inside the speech runtime. The narrower maintainable fix is to carry active-run reply-origin/audio metadata from the steered inbound into final dispatch.
