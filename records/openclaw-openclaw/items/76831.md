## Summary

Keep open: current main still has the source-reproducible active-run metadata gap, and the linked fix PR is open but not merged, so this issue remains the canonical tracking item.

## What This Changes

Issue reports that WhatsApp `tts.auto: "inbound"` can deliver a text reply instead of voice when an audio inbound is steered into an already-active session run.

## Best Possible Solution

Land a reviewed fix that carries steered inbound audio metadata through the active session to final dispatch and clears the one-shot marker after consumption; the linked PR is the active candidate.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: a steered audio inbound sets `currentInboundAudio`, but the streaming steer branch does not carry that fact, and final TTS dispatch still relies on the original `ctx` inbound-audio value. No live WhatsApp repro was run in this read-only review.

## Solution Assessment

Yes, carrying the steered inbound audio fact through the session entry into dispatch is the narrow maintainable fix. A wrapper-only TTS gate is not enough because the speech runtime already gates on `inboundAudio`; the missing fact is the active-run metadata handoff.
