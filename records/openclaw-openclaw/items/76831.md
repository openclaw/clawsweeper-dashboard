## Summary

Keep open: current main still appears to gate Auto-TTS from the original dispatch context during active `steer`, so a text-started run can consume a later WhatsApp audio message while final delivery still sees non-audio metadata.

## What This Changes

The issue reports that WhatsApp replies are sent as plain text instead of voice when `messages.tts.auto: "inbound"` and a later audio inbound is steered into an already-active run.

## Best Possible Solution

Carry steered inbound audio/reply-origin metadata into the active run, preferably as sticky consumed-inbound audio state or updated active reply-origin data, with regression coverage for text-then-audio steering.

## Reproduction Assessment

No high-confidence live reproduction was established in this read-only review. Source inspection gives a medium-confidence path: active `steer` injects prompt text into the running model while dispatch still gates TTS from the original context's `inboundAudio`.

## Solution Assessment

Yes: preserving active-run media/reply-origin metadata is the narrowest maintainable direction. Adding only another default-dispatch pre-gate would not fix stale `inboundAudio=false` when the later consumed message is the audio one.
