## Summary

Keep open: current main still appears to gate Auto-TTS from the original dispatch context, so a text-started active run can consume a later WhatsApp audio steer while final delivery still sees `inboundAudio=false`. The existing issue discussion and prior ClawSweeper comment are consistent with the current source path.

## What This Changes

The issue asks for WhatsApp `tts.auto: "inbound"` replies to remain voice replies when an audio message is steered into an already-active text-started session run.

## Best Possible Solution

Carry the steered inbound audio/reply-origin signal into the active run, either as sticky `inboundAudio` over consumed inbounds or as updated active reply-origin metadata, with focused regression coverage for text-then-audio steering.

## Reproduction Assessment

No live WhatsApp reproduction was established in this read-only review. Source inspection gives a medium-confidence path: default `steer` injects prompt text into an active run while final TTS gating still uses the original context's `inboundAudio`.

## Solution Assessment

Yes, the likely best fix is to preserve steered inbound media metadata for dispatch rather than adding another redundant TTS pre-gate. The speech runtime already has the `inbound` gate; the missing piece is the active-run metadata signal.
