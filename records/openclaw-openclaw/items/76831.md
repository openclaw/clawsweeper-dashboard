## Summary

Keep open: current main still has the source path where a later WhatsApp audio prompt can be steered into the active model turn while final Auto-TTS still uses the original dispatch context for `inboundAudio` and session TTS state.

## What This Changes

The issue asks OpenClaw to deliver a voice reply for WhatsApp `messages.tts.auto: "inbound"` when an audio message arrives during an already-active steered session run.

## Best Possible Solution

Carry steered inbound audio or explicit reply-origin media metadata into the active run, preferably as a sticky `inboundAudio` signal for the next reply, and cover text-then-audio steering with deterministic regression coverage.

## Reproduction Assessment

No, there is not a high-confidence live reproduction from this review. Source inspection gives a medium-confidence path: the steered audio transcript can enter the active run while final TTS still gates on the original text dispatch context's `inboundAudio`.

## Solution Assessment

Yes, metadata propagation is the best fix direction. Adding only another dispatch-local `inboundAudio` pre-gate would not address stale `inboundAudio=false`, because the speech runtime already performs that gate.
