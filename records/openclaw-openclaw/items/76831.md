Keep open: current main and v2026.6.10 still drop the steered inbound-audio fact before final TTS dispatch, and the open candidate fixes have not merged yet.

## What This Changes

The issue reports that WhatsApp active steer-mode runs can answer a later inbound audio message but still deliver the final reply as text when `messages.tts.auto` is `inbound`.

## Best Possible Solution

Keep this issue as the canonical user report until https://github.com/openclaw/openclaw/pull/95596 or an equivalent maintainer-chosen fix lands and proves the active-steer audio-to-voice path.

## Reproduction Assessment

Yes. Source inspection shows the steered audio fact is recorded before active steering, not carried across the queue boundary, and final inbound-auto TTS gates from the original text context.

## Solution Assessment

Yes. The best path is an operation-local metadata handoff like https://github.com/openclaw/openclaw/pull/95596; adding only another dispatch wrapper gate would not fix the missing active-run audio fact.
