## Summary

Keep open: current `main` still supports the reported failure path. WhatsApp preserves inbound audio metadata, but active-run steering queues only prompt text while final default dispatch computes `inboundAudio` from the original dispatch context, so `tts.auto: "inbound"` can be silently skipped for a later steered audio message.

## What This Changes

The issue asks OpenClaw to keep WhatsApp `messages.tts.auto: "inbound"` voice replies when an audio inbound is steered into an already-active text-originated session run.

## Best Possible Solution

Preserve the steered inbound audio/reply-origin signal across the active run so final dispatch sees the consumed audio message when deciding `tts.auto: "inbound"`.

## Reproduction Assessment

No high-confidence live reproduction was run here. Source inspection gives a medium-confidence reproduction path: start a text-originated active run, steer a later WhatsApp audio prompt into it, and final dispatch still evaluates `inboundAudio` from the original text context.

## Solution Assessment

Yes, the best fix is to carry steered inbound media/reply-origin metadata forward, likely as sticky inbound-audio state or an updated active reply origin. Adding only a redundant default-dispatch pre-gate would not fix stale false metadata because the speech runtime already performs the inbound gate.
