## Summary

Keep open: current main still appears to preserve the reported failure path, where a WhatsApp audio message steered into an already-running text-originated session can reach the model prompt while final Auto-TTS delivery still gates on the original dispatch context. No merged PR or canonical follow-up was found for the remaining WhatsApp active-steer metadata gap.

## What This Changes

The issue asks OpenClaw to make WhatsApp `messages.tts.auto: "inbound"` synthesize voice replies when an audio DM is steered into an already active session run.

## Best Possible Solution

Carry steered inbound audio or reply-origin metadata into the active run, preferably as a sticky `inboundAudio` signal or explicit active reply-origin update, with a deterministic text-then-audio steering regression.

## Reproduction Assessment

No high-confidence live reproduction was established in this review. Source inspection gives a medium-confidence path: default `steer` injects only prompt text into the active embedded run while final dispatch still gates TTS from the original context's `inboundAudio`.

## Solution Assessment

Yes, metadata propagation is the best direction; adding another dispatch-local TTS pre-gate would not fix a stale `inboundAudio=false` value because the speech runtime already performs the inbound gate.
