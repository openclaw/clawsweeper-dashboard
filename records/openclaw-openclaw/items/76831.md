## Summary

Keep open: current main still appears to lose the inbound-audio signal when a WhatsApp audio message is steered into an active text-started run, so documented `tts.auto: "inbound"` behavior can silently fall back to text. The existing issue discussion and the prior ClawSweeper comment remain consistent with the current source path.

## What This Changes

The issue asks OpenClaw to preserve voice-in/voice-out Auto-TTS behavior when a WhatsApp audio message is steered into an already active session run.

## Best Possible Solution

Carry the steered inbound audio/reply-origin signal into the active run, preferably as sticky consumed-inbound metadata or updated active reply-origin state, with regression coverage for text-then-audio steering.

## Reproduction Assessment

No live WhatsApp reproduction was established in this read-only review. Source inspection gives a medium-confidence path: `steer` injects the audio transcript into the active run while final TTS gating still sees the original text context's `inboundAudio=false`.

## Solution Assessment

Yes: preserving steered inbound media metadata is the narrowest maintainable fix. Mirroring another pre-gate in default dispatch would not solve stale `inboundAudio` when the reply content came from a later steered audio message.
