## Summary

Keep open: current main still appears to gate Auto-TTS from the original dispatch context, so an active text-started steer can ingest a later audio message while final delivery still sees a text-origin `inboundAudio=false`.

## What This Changes

Fix WhatsApp `messages.tts.auto: "inbound"` so replies to audio messages consumed during an active steered run are delivered as voice instead of plain text.

## Best Possible Solution

Carry the steered inbound's audio/reply-origin metadata into the active run as a sticky `inboundAudio` signal or updated active reply-origin record, with regression coverage for text-then-audio steer timing.

## Reproduction Assessment

No, there is not a high-confidence live reproduction from this review. Source inspection gives a medium-confidence path: active steer injects prompt text while final TTS delivery still depends on the original dispatch context's `inboundAudio`.

## Solution Assessment

Yes, keeping this open is the best current outcome. The narrow maintainable fix should preserve steered inbound media metadata for dispatch; adding only a redundant default-dispatch pre-gate would not fix a stale false `inboundAudio` snapshot.
