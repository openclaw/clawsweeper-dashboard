## Summary

Keep open: current main still appears to compute Auto-TTS eligibility from the original dispatch context while steer-mode injects only prompt text into the active run, so a text-started turn can consume later audio content but still dispatch with `inboundAudio=false`.

## What This Changes

The issue asks OpenClaw to preserve WhatsApp voice-in/voice-out behavior when an audio DM is steered into an already-active run started by a text DM.

## Best Possible Solution

Carry steered inbound audio/reply-origin state into the active run for dispatch, preferably as sticky consumed-inbound audio metadata or updated active reply-origin metadata with regression coverage for text-then-audio steering.

## Reproduction Assessment

No live WhatsApp reproduction was established here, but source inspection gives a medium-confidence reproduction path: a text-origin active steer run injects the later audio transcript as prompt text while final dispatch still evaluates TTS from the original text context.

## Solution Assessment

Yes: the narrow maintainable fix is to preserve steered inbound media/reply-origin metadata through the active-run boundary; adding another default-dispatch pre-gate would not fix stale `inboundAudio=false`.
