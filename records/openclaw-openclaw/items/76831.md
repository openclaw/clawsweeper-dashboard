## Summary

Keep open: current main still appears to derive Auto-TTS `inboundAudio` from the original dispatch context, so a text-started active run can consume a later audio steer while final delivery still sees a text origin for `auto: "inbound"`.

## What This Changes

The issue reports that WhatsApp replies can be sent as text instead of voice when `messages.tts.auto: "inbound"` and a later audio inbound is steered into an already-active run.

## Best Possible Solution

Carry the steered inbound's audio/reply-origin metadata into the active run, either as a sticky `inboundAudio` signal over consumed inbounds or an updated active reply-origin record, with regression coverage for text-then-audio steer timing.

## Reproduction Assessment

No live high-confidence WhatsApp reproduction was established here. Source inspection gives a medium-confidence path: active steer injects only prompt text, while final Auto-TTS still gates on the original dispatch context's `inboundAudio`.

## Solution Assessment

Yes, keeping this open is the right outcome. The maintainable fix should preserve steered inbound media metadata for dispatch; adding only a redundant default-dispatch pre-gate would not help when `inboundAudio` remains false.
