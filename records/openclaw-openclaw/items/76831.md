## Summary

Keep open: current main still appears to use the original dispatch context for final Auto-TTS gating, while steer mode injects only the later prompt text into the active run. The issue matches documented voice-in/voice-out behavior and the prior ClawSweeper source review remains consistent with the current code.

## What This Changes

Preserve inbound-audio metadata when a WhatsApp audio message is steered into an active run so `messages.tts.auto: "inbound"` can synthesize the reply as voice.

## Best Possible Solution

Carry a sticky inbound-audio or active reply-origin signal from steered inbounds into the active run dispatch metadata, with regression coverage for text-then-audio steering.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection gives a medium-confidence path: default `steer` injects the later audio transcript into the active run while final TTS gating still uses the original context’s `inboundAudio`.

## Solution Assessment

Yes, preserving steered inbound media metadata is the narrow maintainable fix. Adding another default-dispatch pre-gate would not fix stale `inboundAudio=false` reaching the speech runtime.

Keep the fix in the shared reply/steer metadata boundary unless source inspection proves the WhatsApp plugin owns it. Do not add a new config option or change provider/channel defaults.
