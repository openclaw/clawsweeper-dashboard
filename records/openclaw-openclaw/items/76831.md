## Summary

Keep open: current main still appears vulnerable in the active-run steer path, where a later WhatsApp audio message can be injected into the running turn without updating the final dispatch metadata used by `tts.auto: "inbound"`.

## What This Changes

The issue requests that WhatsApp replies remain voice replies when an audio inbound is steered into an already-active session run under `messages.tts.auto: "inbound"`.

## Best Possible Solution

Carry the steered inbound's audio/reply-origin signal into active-run final dispatch, preferably as a scoped sticky `inboundAudio`/reply-origin update for consumed queued messages, with regression coverage for WhatsApp audio under Codex steer mode.

## Reproduction Assessment

Yes, from source: a text-origin active run computes `inboundAudio=false`, while a later audio message steered into that run is passed as prompt text without media metadata before final TTS dispatch. I did not live-run the WhatsApp race, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No, closing or only mirroring the ACP-side gate would not be the best fix; current speech runtime already gates inbound mode, so the narrower durable fix is to preserve steered audio metadata through the active-run dispatch boundary.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76831. The observable bug is that with WhatsApp and `messages.tts.auto: "inbound"`, an audio inbound steered into an active text-origin session can influence the model response but final delivery still uses the original text inbound's `inboundAudio=false`, so no TTS audio is synthesized. Do not add a config option and do not move WhatsApp-specific behavior into core. Prefer a generic active-run steering metadata fix: preserve or sticky-OR the queued inbound audio/reply-origin signal through the active run so final dispatch can apply TTS when any consumed steered inbound was audio, without making unrelated non-audio turns voice. Verify the existing speech runtime gate remains the source of truth; do not fix this only by duplicating the ACP gate. Add focused regression coverage for active-run steer plus inbound audio metadata, and update `CHANGELOG.md` because this is user-facing. Related context: #72034 was the earlier Feishu TTS inbound report, but current main already has the runtime inbound gate.
