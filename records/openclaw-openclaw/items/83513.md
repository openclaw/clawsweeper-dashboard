## Label Justifications

- `P2`: This is a focused Control UI webchat bug with a clear source path and limited blast radius, but it can make the chat appear stuck for affected users.
- `impact:session-state`: The failure is caused by stale run/session state (`hasActiveRun`/`canAbort`) being converted into visible chat stream state after the local stream was cleared.

## Summary

Keep open: current `main` still has a render-time fallback that turns `stream: null` into an empty stream when the current session is abortable, and the chat item builder renders that empty stream as the reading indicator. The related earlier issue was closed stale and the linked PR was not merged, so there is no implemented fix to close against.

## What This Changes

The issue reports a Control UI webchat bug where the reading indicator can persist after an assistant response because abortability state can manufacture an empty stream after `chatStream` was cleared.

## Best Possible Solution

Gate the reading indicator on explicit pending stream/loading state, not abortability alone: `stream: null` after a final response should stay null, while `stream: ""` or a clearly loading active run can still show the dots.

## Reproduction Assessment

Yes from source inspection: render `renderChat` with `canAbort: true`, an `onAbort` handler, `stream: null`, and `loading: false`; it produces `displayStream === ""`, and `buildChatItems` renders the reading indicator. I did not run a live browser/gateway repro in this read-only sweep.

## Solution Assessment

Yes, with one constraint: removing the `canAbort` fallback or narrowing it to an actual loading/pending state is the maintainable fix, but the existing active-loading indicator behavior should be preserved or deliberately updated with tests.
