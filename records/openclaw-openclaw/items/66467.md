## Summary

Keep open. Current main still routes ACPX session updates through the published ACPX/ACP SDK path where `usage_update.used` is schema-required as a number, so the reported `used: null` notification is still a valid source-level bug rather than cleanup-closeable stale noise.

## What This Changes

Accept ACP `session/update` `usage_update` notifications from ACPX when usage fields such as `used` are null or omitted instead of rejecting the notification.

## Best Possible Solution

Publish or consume an ACPX/ACP SDK fix that normalizes or drops nullable `usage_update` usage fields before SDK validation, then add focused regression coverage for `used: null`.

## Reproduction Assessment

Yes, at source level: pass a `session/update` `usage_update` with `used: null` through the current ACPX client path and the pinned ACP SDK validates `zSessionNotification` before dispatch, where `used` is required to be numeric.

## Solution Assessment

Unclear. Accepting or dropping nullable usage fields is the right behavior, but the maintainable location should be selected at the ACPX/ACP SDK boundary rather than in unrelated Discord delivery code.
