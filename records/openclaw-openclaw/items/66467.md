## Summary

Keep open. Current main still has the ACPX/ACP SDK validation mismatch: `@openclaw/acpx` is pinned to `acpx` 0.6.1, whose ACP client path validates `session/update` before ACPX's lenient usage projection can omit a nullable `used` value.

## What This Changes

The issue requests that ACP/acpx `usage_update` notifications with `used: null` be accepted or ignored instead of failing Discord-bound sessions with `Invalid params`.

## Best Possible Solution

Make the ACPX runtime boundary contract-consistent by normalizing or dropping nullable usage fields before SDK validation, or by consuming a published ACPX/ACP SDK release with that behavior, then add focused regression coverage for `used: null`.

## Reproduction Assessment

Yes. A high-confidence source-level path is to pass a `session/update` `usage_update` with `used: null` through the pinned ACPX client path; the SDK parses `zSessionNotification` first and rejects the required numeric `used` field before ACPX projection runs.

## Solution Assessment

Unclear. Accepting or dropping nullable usage fields is the right behavior, but the maintainable fix should be chosen at the ACPX/ACP SDK boundary rather than in unrelated Discord delivery code.
