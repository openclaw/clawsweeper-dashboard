## Summary

Keep open. Current main still routes the affected ACPX path through published `acpx` 0.7.0 and `@agentclientprotocol/sdk` 0.21.1, whose `session/update` schema rejects `usage_update.used: null` before callback dispatch; no current OpenClaw shim or duplicate fix covers this validation path.

## What This Changes

The issue asks OpenClaw to accept or safely ignore ACP `session/update` `usage_update` notifications when `used` is `null` or omitted.

## Best Possible Solution

Normalize or drop nullable `usage_update` fields before ACP SDK notification validation, preferably in `acpx` or the ACP SDK contract, then update OpenClaw's pinned dependency and add focused regression coverage for `used: null`.

## Reproduction Assessment

Yes, at source level: current main resolves ACPX through SDK 0.21.1, whose schema requires numeric `used` and validates `session/update` before dispatch. I did not run a live Discord ACP session in this read-only review.

## Solution Assessment

Unclear. The expected behavior is clear, but the maintainable repair should be chosen at the `acpx` / ACP SDK boundary; a local OpenClaw compatibility shim would need owner approval.
