## Summary

Keep open. Current main still pins the ACPX path to `acpx` 0.7.0 and `@agentclientprotocol/sdk` 0.21.0, whose `session/update` schema requires `usage_update.used` to be numeric before dispatch, so the reported `used: null` notification remains a valid source-level bug rather than cleanup-closeable noise.

## What This Changes

The issue asks OpenClaw to accept or safely ignore ACP `usage_update` notifications where `used` is `null` in Discord-bound ACP sessions.

## Best Possible Solution

Normalize or drop nullable `usage_update` usage fields before ACP SDK notification validation, then add focused regression coverage proving `used: null` no longer rejects the whole notification.

## Reproduction Assessment

Yes, at source level: pass a `session/update` `usage_update` with `used: null` through the current ACPX client path and the pinned ACP SDK validates `zSessionNotification` before dispatch, where `used` is required to be numeric.

## Solution Assessment

Unclear. Accepting or dropping nullable usage fields is the right behavior, but the maintainable location should be selected at the ACPX/ACP SDK boundary rather than in unrelated Discord delivery code.
