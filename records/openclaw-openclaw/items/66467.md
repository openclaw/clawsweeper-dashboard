## Summary

Keep open. Current main still has the ACPX/ACP SDK validation mismatch described by the reporter: the bundled ACPX plugin remains pinned to `acpx` 0.6.1, that dependency still brings in an ACP SDK schema requiring numeric `usage_update.used`, and the published SDK validates `session/update` before ACPX's later projection can omit nullable usage fields.

## What This Changes

The issue requests that ACP `session/update` `usage_update` notifications with `used: null` be accepted or safely ignored instead of failing JSON-RPC param validation in Discord-bound ACP sessions.

## Best Possible Solution

Make the ACPX runtime boundary contract-consistent: normalize or omit nullable `usage_update` fields before ACP SDK validation, or consume an upstream ACPX/ACP SDK release that accepts the intended shape, then add focused regression coverage for `used: null` on the ACPX-backed `session/update` path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path is to send a `session/update` `usage_update` with `used: null` through the `ClientSideConnection` used by `acpx@0.6.1`; the SDK validates against `zSessionNotification` and rejects the required numeric `used` field before dispatch.

## Solution Assessment

Unclear. Accepting or dropping nullable usage fields is the right product behavior, but the maintainable fix should be chosen at the ACPX/ACP SDK boundary rather than patched in unrelated Discord delivery code.
