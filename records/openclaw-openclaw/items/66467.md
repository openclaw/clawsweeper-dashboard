## Summary

Keep open. Current main still pins the ACPX path to `acpx` 0.7.0 and `@agentclientprotocol/sdk` 0.21.0; that SDK validates `session/update` before dispatch and requires `usage_update.used` to be numeric, while the OpenClaw ACPX wrapper only delegates the runtime stream, so the reported `used: null` failure remains a valid source-level bug.

## What This Changes

The issue reports ACP `session/update` `usage_update` notifications failing validation when the ACP runtime sends `used: null`.

## Best Possible Solution

Normalize or drop nullable `usage_update` fields before ACP SDK notification validation, preferably in `acpx` or the ACP SDK contract, then update OpenClaw's pinned dependency and add regression coverage for `used: null`.

## Reproduction Assessment

Yes, at source level: `acpx` 0.7.0 depends on `@agentclientprotocol/sdk` 0.21.0, whose `ClientSideConnection` parses `session/update` params with `zSessionNotification` before callback dispatch, and `zUsageUpdate` requires `used: number`. A live Discord session was not run in this read-only review.

## Solution Assessment

Unclear. Accepting or dropping nullable usage fields is the right behavior, but the maintainable fix should be chosen at the ACPX/ACP SDK boundary rather than in unrelated Discord delivery code.
