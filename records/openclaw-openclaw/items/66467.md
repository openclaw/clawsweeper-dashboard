## Summary

Keep open. Current main still has the same ACPX/ACP SDK validation mismatch: the bundled ACPX path delegates to `acpx` 0.6.1, whose ACP client connection validates `session/update` before its lenient runtime projection can drop nullable usage fields, and the current published ACP SDK schema still requires numeric `usage_update.used`.

## What This Changes

Report that Discord-bound ACPX sessions reject `session/update` `usage_update` notifications when `used` is `null`.

## Best Possible Solution

Make the ACPX runtime boundary contract-consistent: normalize or omit nullable `usage_update` fields before `@agentclientprotocol/sdk` validation, or consume an upstream ACPX/ACP SDK release that accepts the intended shape, then add regression coverage for `used: null` on the ACPX-backed session/update path.

## Reproduction Assessment

Yes. A source-level reproduction path is clear: send a `session/update` `usage_update` with `used: null` through the `ClientSideConnection` used by `acpx@0.6.1`; the SDK parses against `zUsageUpdate.used: z.number()` before dispatch and returns `-32602 Invalid params`.

## Solution Assessment

Unclear. Accepting or ignoring nullable usage fields matches the observable user problem, but the best implementation should be decided at the ACPX/ACP SDK boundary rather than patched ad hoc in unrelated Discord delivery code.
