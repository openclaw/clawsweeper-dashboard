## Summary

This PR should stay open for maintainer review: current main still undercounts non-multiple-of-three video byte limits, and the proposed one-line helper change matches the existing base64 sizing primitive with focused regression coverage. I found no blocking correctness or security findings.

## What This Changes

The PR changes video max-base64 limit resolution to reuse the existing base64 size estimator and adds boundary tests for non-multiple-of-three raw byte limits.

## Best Possible Solution

Merge this narrow helper fix with its regression test once maintainers are satisfied with the normal CI gate.

## Reproduction Assessment

Yes. Source inspection shows current main rounds the configured raw byte cap down while the caller rounds actual payload size up to base64 quanta, so a video at a non-multiple-of-three raw cap can be rejected before provider submission.

## Solution Assessment

Yes. Reusing estimateBase64Size in the resolver is the narrowest maintainable fix because it makes both sides of the comparison use the same base64 sizing primitive without changing config or the global cap.
