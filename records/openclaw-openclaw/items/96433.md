## Summary

Keep open. This is a focused, source-backed bug fix with no blocking patch findings: current main floors the configured raw video byte limit while the caller compares it to a ceil-rounded base64 payload size, and the PR aligns both sides while preserving the existing global cap.

## What This Changes

The PR changes video base64 limit resolution to reuse the shared base64-size estimator and adds a focused unit test for boundary rounding and the global cap.

## Best Possible Solution

Merge the focused helper fix with its regression coverage after required CI is green, keeping the existing global base64 payload cap unchanged.

## Reproduction Assessment

Yes, at source level: current main uses floor-rounded expansion for the configured cap while the caller uses ceil-rounded base64 sizing for the actual video payload. A raw size where `maxBytes % 3 !== 0` can be within the raw cap but exceed the incorrectly rounded base64 cap.

## Solution Assessment

Yes. Reusing `estimateBase64Size` in the resolver is the narrowest maintainable fix because it makes the configured limit calculation symmetric with the caller's actual payload estimate and does not add config or compatibility surface.
