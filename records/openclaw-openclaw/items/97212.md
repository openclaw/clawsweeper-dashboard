## Summary

Keep open for maintainer review: the patch targets the source-proven null-to-string coercion path and has sufficient after-fix proof, but it deliberately changes shared validation semantics and needs maintainer acceptance of that compatibility risk.

## What This Changes

The PR updates shared LLM tool-argument validation to preserve null for nullable anyOf unions and adds a regression test for the MCP optional-null case.

## Best Possible Solution

Land the focused validation-layer fix after maintainers accept the nullable-union compatibility change and exact-head checks remain clean.

## Reproduction Assessment

Yes from source: current main validates plain JSON-schema union branches after coercing each branch, so anyOf with string before null can turn null into an accepted string branch before the null branch is tried. The linked issue and PR proof describe the matching MCP failure path.

## Solution Assessment

Yes. The best layer is coerceWithUnionSchema because the unwanted conversion happens inside the shared plain JSON-schema validation pass before tool execution; stripping nulls later at the MCP boundary would be broader and can erase meaningful nullable arguments.
