## Summary

Keep open: current main has partial Telegram forum topic-name support for normal inbound messages, but native slash commands, successful create/edit topic actions, recovered dispatch contexts, and restart/system continuation paths can still carry only the topic id into trusted context.

## What This Changes

The issue asks OpenClaw to reliably include Telegram forum topic names in trusted agent context whenever the channel has observed, created, edited, or reconstructed that topic context.

## Best Possible Solution

Keep the fix inside the Telegram channel boundary by making one canonical topic-name resolver/cache update path feed normal inbound, native commands, successful create/edit actions, recovered dispatch, and restart/system delivery contexts.

## Reproduction Assessment

Yes, at source level: current main builds native Telegram command and restart/recovered delivery contexts with MessageThreadId but without TopicName, while normal inbound has a cache-backed TopicName flow. I did not run live Telegram because this review is read-only.

## Solution Assessment

Yes: extending the existing Telegram topic-name cache/resolver across the missing Telegram-owned paths is narrower than adding a core workaround or external plugin hook.
