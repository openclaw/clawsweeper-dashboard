## Label Justifications

- `P1`: This is a source-supported Telegram channel workflow bug that can block routed agent commands in active forum-topic use.
- `impact:session-state`: The reported failure is stale or incomplete trusted turn/session metadata: the session has a topic id but loses the human topic name before agent context.

## Summary

Keep open. Current main includes the earlier topic-name support from https://github.com/openclaw/openclaw/pull/65973 for normal inbound Telegram forum messages, but the reliability gap remains in native slash-command context, recovered topic dispatch context, and create/edit topic cache writes, so the latest main and release do not solve the central user problem.

## What This Changes

Telegram forum-topic agent turns should include the human-readable topic name in trusted metadata wherever OpenClaw already knows the title, including native commands and reconstructed turns.

## Best Possible Solution

Extend the existing Telegram topic-name cache/context plumbing so known topic names are persisted on topic create/edit and carried into normal inbound, native command, and recovered/system turn metadata before prompt construction.

## Reproduction Assessment

Yes, source-level. Current main's native slash-command context sets MessageThreadId and IsForum but not TopicName, while prompt metadata only emits topic_name when ctx.TopicName is present; no live Telegram run was performed in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing Telegram topic-name cache and context fields across the missing turn paths, not add a new resolver/catalog or config option.
