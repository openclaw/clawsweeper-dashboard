## Summary

Keep open: the reported regression is valid and the PR is aimed at the right transport layer, but the current patch still needs changes before merge because it diverges from the existing global family-selection policy, overwrites explicit connect hints, and lacks the required changelog entry.

## What This Changes

The PR adds `autoSelectFamily` connect options to per-request undici HTTP/1 dispatchers used by guarded provider fetches so IPv4-only hosts can fall back when DNS returns AAAA records.

## Best Possible Solution

Share the existing global `autoSelectFamily` decision with per-request guarded-fetch dispatchers, preserve explicit caller connect overrides, add focused regression coverage, and include a single-line user-facing changelog fix entry.

## Reproduction Assessment

Yes, source-level: current main creates pinned guarded dispatchers with only the pinned lookup and the existing test asserts `autoSelectFamily` is absent. The linked reports provide concrete IPv4-only/AAAA host steps, but I did not run that live network setup.

## Solution Assessment

No: adding Happy Eyeballs to per-request dispatchers is the right direction, but this implementation should reuse the global WSL2-aware policy and avoid clobbering explicit transport options before it lands.
