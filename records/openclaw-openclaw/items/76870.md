## Summary

Keep open: the regression is valid and source-reproducible, but this PR still needs changes because it diverges from the existing global family-selection policy, can overwrite explicit connect options, and is missing the required changelog entry.

## What This Changes

The PR adds Happy Eyeballs `autoSelectFamily` options to HTTP/1 undici dispatchers used by guarded provider fetches and updates the SSRF dispatcher test expectation.

## Best Possible Solution

Share the existing global `autoSelectFamily` decision with per-request dispatchers, preserve explicit caller overrides, add focused regression coverage for WSL2 and explicit false settings, and document the user-facing fix in the changelog.

## Reproduction Assessment

Yes, at source level: current main routes provider requests through `fetchWithSsrFGuard` into `createPinnedDispatcher`, and the existing test asserts the pinned dispatcher does not receive `autoSelectFamily`. I did not run the live IPv4-only network scenario.

## Solution Assessment

No: adding Happy Eyeballs to per-request dispatchers is the right direction, but this implementation should reuse the WSL2-aware global policy and avoid clobbering explicit transport options before merge.
