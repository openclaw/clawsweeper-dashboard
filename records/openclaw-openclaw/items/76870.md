## Summary

Keep open: the regression and proposed direction are valid, but the PR still diverges from the existing global family-selection policy, overwrites explicit connect overrides, and is missing the required changelog entry.

## What This Changes

The PR adds autoSelectFamily connect options to per-request HTTP/1 undici dispatchers and updates pinned SSRF dispatcher coverage for IPv4 fallback on broken IPv6 egress.

## Best Possible Solution

Share the global autoSelectFamily decision with per-request dispatchers, preserve explicit caller connect overrides, add focused regression coverage, and include the required Unreleased/Fixes changelog entry.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: createPinnedDispatcher installs only the pinned lookup, and the existing test asserts autoSelectFamily is undefined on that dispatcher; the linked issues add concrete IPv4-only/AAAA network logs.

## Solution Assessment

No. Adding Happy Eyeballs to per-request dispatchers is the right fix direction, but this implementation should reuse the existing global policy and avoid overwriting explicit transport hints before merge.
