## Summary

Keep open: the linked Homebrew auto-update bug is still valid on current main, but this PR has a narrow blocking defect because the new skip branch de-duplicates its hint using notification fields it does not persist for all supported auto-update configurations.

## What This Changes

The PR adds Homebrew package-root detection, skips gateway auto-update for detected Homebrew-managed installs with a brew-upgrade log hint, adds infra tests, and adds a changelog entry.

## Best Possible Solution

The merged solution should keep Homebrew-managed installs out of the npm-based background updater while making the brew-upgrade hint one-shot per version/tag for every auto-update configuration, with explicit `openclaw update` CLI parity left out of scope unless maintainers request it.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available with a Homebrew package root, a newer resolved package version, `update.auto.enabled: true`, and `update.checkOnStart: false`; the PR branch reads unset notification fields and writes state without setting them, so the hint can repeat on later eligible checks.

## Solution Assessment

No as proposed. The Homebrew skip-and-hint direction is narrow and maintainable for the linked bug, but the implementation must persist the notification state before it satisfies its own one-shot hint contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Persist the Homebrew skip notification:** `src/infra/update-startup.ts:438`
  - body: When `update.checkOnStart` is false, current main still runs auto-update but does not populate `lastNotifiedVersion` or `lastNotifiedTag`. This new Homebrew branch reads those fields to suppress repeat logs, then writes `nextState` without setting them, so the same Homebrew hint can repeat on later eligible checks instead of once per version/tag. Set the fields before `writeState` and cover the disabled-startup-hint case.
  - confidence: 0.88
