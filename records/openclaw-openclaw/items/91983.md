## Summary

Current main repairs Anthropic invalid-signature replay failures by retrying without thinking blocks, but the issue reports a newer and independently meaningful defect: signatures produced on successful streams are being persisted invalid, including ordinary-sized values. The available current code prevents a stuck session after rejection but does not validate signatures or explain how successful generation writes corrupt metadata, so the central write-time bug remains open.

## What This Changes

Prevent invalid Anthropic thinking signatures from being persisted and poisoning later session replay, including corruption that is not detectable by signature length.

## Best Possible Solution

Isolate signature assembly per Anthropic generation attempt, preserve only the successful attempt’s provider metadata, and keep the existing rejection-driven retry that removes invalid thinking blocks; add a regression fixture that simulates recovery/retry streams and proves no signature bytes leak across attempts.

## Reproduction Assessment

Yes for the user-visible failure: replaying an invalid persisted Anthropic thinking signature deterministically reaches the current invalid-signature recovery path, and the reporter provides repeated fleet evidence. The successful-stream corruption producer is not yet reproduced locally, so a fix should first capture the retry/attempt sequence in a focused fixture.

## Solution Assessment

No, the requested size cap is not the best complete fix because normal-sized invalid signatures are also reported. The narrow maintainable solution is per-attempt accumulator isolation plus successful-attempt-only persistence, retaining rejection-driven transcript recovery as the fail-safe.
