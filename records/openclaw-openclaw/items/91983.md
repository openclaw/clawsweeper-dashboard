## Summary

This is a credible, current Anthropic session-corruption bug: current main persists and replays signed thinking metadata, and its existing rejection-driven recovery does not explain or prevent the reported fresh corrupt writes across many sessions.

## What This Changes

Prevent invalid Anthropic thinking signatures from being persisted and repeatedly replayed into unusable sessions.

## Best Possible Solution

Make signature state strictly request- and content-block-scoped, reject or reset on impossible retry/event transitions, and retain a narrow Anthropic-rejection recovery path for already-persisted invalid signatures without using a length heuristic.

## Reproduction Assessment

Yes for the failure path: current main forwards persisted same-model signatures to Anthropic, and the provider rejection matches the existing recovery error class. The write-time corruption trigger itself still needs a deterministic stream or retry fixture.

## Solution Assessment

No, a signature length cap is not the best solution because the reporter found normal-sized invalid values; the maintainable fix is lifecycle-correct assembly plus explicit rejection-driven recovery.
