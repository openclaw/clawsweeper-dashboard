## Summary

Keep open: the runtime change is a plausible narrow fix for the Windows root-alias fast-path gap, but the external PR still lacks real Windows after-fix behavior proof and its test change deletes the only Windows case instead of locking the new expected behavior.

## What This Changes

This PR removes the Windows override in `src/plugin-sdk/root-alias.cjs` so the plugin-sdk root-alias loader respects `tryNative`, and deletes the old Windows contract case.

## Best Possible Solution

Land the narrow root-alias pass-through with an explicit Windows contract case expecting `tryNative: true`, plus redacted Windows terminal/log proof that the patched root alias uses the native path without alias or fallback regressions.

## Reproduction Assessment

Yes, at source level: current main forces `getModuleLoader(true)` to `false` on `win32`, and the contract harness currently asserts that Windows production/dist behavior. I did not execute a Windows runtime in this read-only review.

## Solution Assessment

Yes for the runtime direction: removing the root-alias-only Windows override is the narrowest maintainable fix and matches the related native-loader policy. It is not merge-ready until the Windows case is restored as a positive regression assertion and real after-fix Windows proof is added.
