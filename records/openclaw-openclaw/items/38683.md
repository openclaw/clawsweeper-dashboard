## Summary

Close as superseded. The useful recovery idea is already covered by the explicit doctor repair path from #18544, while the later merged Telegram/open-DM policy in #74112 intentionally keeps startup validation fail-closed and makes this schema-time wildcard insertion the wrong direction.

## What This Changes

The PR adds a Telegram config schema normalizer that injects `"*"` into top-level or account `allowFrom` when `dmPolicy` is `open`, and changes one config validation test to expect that auto-heal.

## Best Possible Solution

Keep the shipped explicit doctor repair plus fail-closed schema/runtime policy; any future startup recovery should be an audited maintainer-approved design that never silently converts restrictive Telegram allowlists into public access.

## Reproduction Assessment

Yes. The PR diff itself is enough to reproduce the introduced regression: it appends `"*"` to non-wildcard Telegram `allowFrom` before validation, while current docs and runtime treat `"*"` as public Telegram DM access.

## Solution Assessment

No. Schema-time wildcard insertion is not the best solution because it silently broadens access; current main's explicit doctor repair and fail-closed validation are the safer maintained path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not widen restrictive Telegram allowlists:** `src/config/zod-schema.providers-core.ts:153`
  - body: Current main treats `allowFrom` without `"*"` as a safety boundary and only makes `dmPolicy="open"` public when an effective wildcard is present. This helper appends `"*"` to restrictive Telegram allowlists before validation, so a sender-limited config can become public DM access.
  - confidence: 0.9
