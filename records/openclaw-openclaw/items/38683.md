## Summary

Keep open for maintainer review, but do not merge as-is. The PR's schema-time Telegram auto-heal conflicts with current main's explicit public-DM contract: current code and docs require an effective wildcard before `dmPolicy="open"` behaves as public access, while this patch silently appends `"*"` to restrictive allowlists and can reintroduce the public exposure class addressed by #73756/#74112.

## What This Changes

The PR adds a Telegram config schema normalizer that inserts `"*"` into top-level or account `allowFrom` when `dmPolicy` is `open`, and updates one config validation test to expect Telegram auto-heal instead of rejection.

## Best Possible Solution

Keep the current fail-closed contract unless maintainers explicitly choose a different product/security policy. If startup recovery is still wanted, implement it as an explicit, audited repair path that does not silently turn a restrictive Telegram allowlist into public access, and cover top-level, account, inherited, missing, empty, and non-wildcard cases in Telegram and shared doctor tests.

## Reproduction Assessment

Yes. The review reproduction is static and high-confidence: the patch appends `"*"` to restrictive Telegram `allowFrom` before validation, while current main docs/runtime treat an effective wildcard as the explicit public-access signal.

## Solution Assessment

No. Schema-time mutation is not the best fix under current main because it silently broadens access and conflicts with the shipped doctor-side repair path; a maintainer-approved repair should stay explicit and preserve restrictive allowlists unless the operator intentionally configures public access.
