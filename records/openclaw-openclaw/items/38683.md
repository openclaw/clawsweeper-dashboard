## Summary

Close as superseded: the PR’s schema-time Telegram wildcard insertion conflicts with the current fail-closed access-control policy, while the safer explicit repair direction is already represented by doctor repair work and later Telegram/open-DM policy alignment.

## What This Changes

The PR adds a Telegram config schema normalizer that injects `"*"` into top-level or account `allowFrom` when `dmPolicy` is `open`, and updates a config validation test to expect auto-heal.

## Best Possible Solution

Keep schema validation fail-closed and use an explicit, logged doctor/security repair or audit path for intentional public Telegram DM access rather than mutating access policy during validation.

## Reproduction Assessment

Yes. The high-confidence reproduction is static: the PR diff appends `"*"` to Telegram `allowFrom` for `dmPolicy="open"`, while current docs and runtime treat wildcard as the explicit public-access signal.

## Solution Assessment

No. Schema-time auto-heal is not the best solution because it silently broadens access; current fail-closed validation plus explicit doctor/audit repair is the narrower maintainable path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Do not widen restrictive Telegram allowlists:** `src/config/zod-schema.providers-core.ts:153`
  - body: Current main treats a non-wildcard `allowFrom` as a sender boundary and only makes `dmPolicy="open"` public when an effective wildcard is present. This helper appends `"*"` during schema validation, so a config that was rejected or limited to specific sender IDs becomes public DM access.
  - confidence: 0.9
