## Summary

Keep open and do not merge as-is. The PR targets a real Telegram cold-start footgun, but its schema-time auto-heal would silently convert restrictive Telegram `allowFrom` lists into wildcard public DM access, conflicting with current main's documented fail-closed policy and runtime guardrails.

## What This Changes

The PR adds a Telegram config schema normalizer that inserts `"*"` into top-level or account `allowFrom` when `dmPolicy` is `open`, and updates one config validation test to expect Telegram auto-heal instead of rejection.

## Best Possible Solution

Keep the fail-closed schema/runtime contract unless maintainers explicitly approve a public-access recovery flow. If startup recovery is still desired, it should be an explicit audited repair path that does not silently turn restrictive Telegram allowlists into wildcard public DM access, with coverage for top-level, account, inherited, missing, empty, and non-wildcard cases.

## Reproduction Assessment

Yes. The high-confidence reproduction is static: the PR diff appends `"*"` to restrictive Telegram `allowFrom` before validation, while current docs and runtime treat wildcard as the explicit public-access signal.

## Solution Assessment

No. Schema-time mutation is not the best solution because it silently broadens access; the safer direction is explicit doctor/security repair or a maintainer-approved recovery design that preserves restrictive allowlists unless the operator intentionally configures public access.
