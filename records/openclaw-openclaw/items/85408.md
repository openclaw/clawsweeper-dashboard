## Summary

Keep open: current main and v2026.6.8 still keep Claude CLI runs on a strict generated MCP config with `--allowedTools mcp__openclaw__*`, so user/project Claude MCP access is not implemented and the remaining work is a security-sensitive product decision.

## What This Changes

The issue requests an opt-in way for `openclaw agent --agent claude` to expose user/project MCP servers instead of sealing Claude CLI sessions to OpenClaw's loopback MCP server.

## Best Possible Solution

Define an explicit opt-in Claude CLI MCP contract that keeps default loopback-only isolation, prefers OpenClaw-managed `mcp.servers`, derives matching Claude allowlist patterns for approved servers, and documents/tests fresh and resume behavior with security proof.

## Reproduction Assessment

Yes, source-reproducible: current Claude CLI backend args still allow only `mcp__openclaw__*`, and the runner injects a strict generated MCP config that blocks ambient user/project MCP discovery. I did not run a live Claude CLI session in this read-only review.

## Solution Assessment

Unclear until maintainers choose the trust boundary. The safest direction is an explicit opt-in contract around OpenClaw-managed `mcp.servers` and derived Claude allowlists, not silent ambient import from Claude user/project files.
