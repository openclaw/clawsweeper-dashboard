## Summary

Keep open: current main still routes the direct Claude CLI backend through a strict generated MCP config and a default `mcp__openclaw__*` tool allowlist, so the requested user/project MCP access is not implemented. Relaxing that boundary is config-, compatibility-, and security-sensitive, so this needs maintainer design review rather than an automatic close or repair PR.

## What This Changes

Allow direct `openclaw agent` Claude CLI runs to access configured user/project MCP servers beyond the built-in OpenClaw loopback MCP server.

## Best Possible Solution

Define an explicit opt-in contract that keeps default loopback-only isolation, uses OpenClaw `mcp.servers` as the canonical supported registry where possible, and derives Claude allowlist patterns only for maintainer-approved configured servers or per-agent allowlist entries.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: resolve the bundled `claude-cli` backend and prepare bundle MCP, and the resulting argv keeps `--allowedTools mcp__openclaw__*` while strict MCP config is regenerated. I did not run a live Claude CLI session in this read-only review.

## Solution Assessment

Unclear: the proposed knobs are plausible, but the best path should be chosen by maintainers because OpenClaw already has `mcp.servers` and intentionally strict Claude loopback isolation. A safer direction is explicit opt-in allowlisting with tests for both default isolation and enabled non-OpenClaw MCP access.
