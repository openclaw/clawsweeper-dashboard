## Summary

Keep open: current main and v2026.6.6 still keep the bundled Claude CLI backend on a strict generated MCP config with the native Claude allowlist set to `mcp__openclaw__*`, so the requested user/project MCP access is not safely implemented. The remaining work is a security- and config-sensitive product decision, not a conservative cleanup close.

## What This Changes

The issue requests an opt-in way for `openclaw agent` Claude CLI runs to use additional user/project MCP servers instead of only the OpenClaw loopback MCP surface.

## Best Possible Solution

Define an explicit opt-in Claude CLI MCP contract that keeps default loopback-only isolation, prefers OpenClaw `mcp.servers` as the supported registry, derives matching Claude allowlist patterns only for approved servers, and covers fresh/resume behavior with docs and security tests.

## Reproduction Assessment

Yes. Source inspection shows the bundled `claude-cli` path still combines `--allowedTools mcp__openclaw__*` with a regenerated strict MCP config; I did not run a live Claude CLI session in this read-only review.

## Solution Assessment

Unclear until maintainers choose the trust boundary. The safest direction is an explicit opt-in contract around OpenClaw-managed `mcp.servers` and derived Claude allowlists, rather than silently importing ambient Claude user/project MCP config.
