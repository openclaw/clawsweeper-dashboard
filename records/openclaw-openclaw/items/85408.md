## Summary

Keep open: current main still seals Claude CLI runs with a generated strict MCP config, and while OpenClaw-managed `mcp.servers` is already supported, the requested user-scope/project Claude MCP passthrough remains a new security-sensitive product/config decision.

## What This Changes

The issue requests a way for `openclaw agent --agent claude` to include user/project Claude MCP servers instead of only the generated OpenClaw MCP overlay.

## Best Possible Solution

Keep the strict default, then add an explicit documented opt-in only if maintainers want Claude user/project MCP passthrough, preferably by projecting approved servers through OpenClaw-owned `mcp.servers` or a per-agent allowlist with security review.

## Reproduction Assessment

Yes, at source level for the remaining user-scope/project Claude MCP request: current main always injects `--strict-mcp-config --mcp-config` for Claude bundle MCP runs, and Claude's contract says that ignores other MCP configurations. I did not run a live Claude CLI session, and the already-supported OpenClaw `mcp.servers` path is a different reproduction path.

## Solution Assessment

Unclear. Reusing ambient Claude MCP config may be convenient, but the safer maintainable direction is likely an explicit OpenClaw-owned opt-in that preserves the loopback bridge and canonical tool/security policy.
