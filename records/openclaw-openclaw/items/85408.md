## Summary

Keep open: current main and v2026.6.8 still launch the bundled Claude CLI path with a generated strict MCP config and the native Claude allowlist set to `mcp__openclaw__*`. A merged related fix now includes OpenClaw `mcp.servers` in the generated config, but the requested user/project Claude MCP access and matching allowlist contract remain a security-sensitive product decision.

## What This Changes

The issue asks for an opt-in way for `openclaw agent --agent claude` to expose user/project MCP servers instead of only the OpenClaw loopback MCP server.

## Best Possible Solution

Define a default-off Claude CLI MCP contract that keeps loopback-only isolation by default and exposes only explicitly approved OpenClaw-managed or user-selected MCP servers with matching Claude allowlist patterns, docs, tests, and real backend proof.

## Reproduction Assessment

Yes. Source inspection shows the bundled `claude-cli` path still combines `--allowedTools mcp__openclaw__*` with a regenerated strict MCP config; I did not run a live Claude CLI session in this read-only review.

## Solution Assessment

Unclear until maintainers choose the trust boundary. The safest direction is an explicit opt-in contract around OpenClaw-managed or explicitly selected MCP servers and derived Claude allowlists, rather than silently importing ambient Claude user/project MCP config.
