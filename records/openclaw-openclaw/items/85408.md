## Summary

Keep open: current main and the latest release still launch Claude CLI with a loopback-only native allowlist and a strict generated MCP config. OpenClaw-managed `mcp.servers` are now included in that generated file, but ambient Claude user/project MCP discovery and matching non-OpenClaw allowlist policy remain a security-sensitive product decision.

## What This Changes

Requests a safe opt-in path for `openclaw agent` Claude CLI sessions to use user/project Claude MCP servers and non-OpenClaw `mcp__*` tools instead of only the generated OpenClaw loopback.

## Best Possible Solution

Define a default-off Claude CLI MCP contract that keeps loopback isolation by default and exposes only explicitly approved OpenClaw-managed or user-selected MCP servers with derived allowlist patterns, docs, regression tests, and real Claude backend proof.

## Reproduction Assessment

Yes, source-level. Current main still combines `--allowedTools mcp__openclaw__*` with a generated strict MCP config; I did not run a live Claude CLI turn in this read-only review.

## Solution Assessment

Unclear until maintainers choose the trust boundary. The safest shape is explicit default-off selection of approved MCP sources and allowlist patterns, not silent ambient import from Claude user/project config.
