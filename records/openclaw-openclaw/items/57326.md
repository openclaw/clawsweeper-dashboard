## Label Justifications

- `P1`: Multiple CLI-backed helper paths can still route or persist runtime state incorrectly, breaking real agent/channel workflows for CLI-runtime users.
- `impact:security`: Prompts and auth context intended for a local CLI backend can still enter embedded/API helper execution when helper paths bypass CLI dispatch.
- `impact:session-state`: The subagent persistence path can store raw provider state instead of the resolved CLI runtime for later turns.
- `impact:auth-provider`: The report centers on provider, runtime, model, and auth-profile selection for CLI-backed models.

## Summary

Keep open. Current main has fixed the queued-followup and cron portions of the report, but model auth probes, LLM slug generation, and initial child-session runtime persistence still have source-level paths that bypass or fail to persist the configured CLI runtime; an open closing PR should be reviewed rather than closing the issue first.

## What This Changes

The issue asks for CLI-backed helper paths to use the same runtime resolution and CLI dispatch path as normal agent turns.

## Best Possible Solution

Review and land a rebased narrow fix or replacement for the remaining probe, slug, and child-session persistence paths, while keeping distinct compaction and stale-pin hardening in their own linked PRs.

## Reproduction Assessment

Yes, at source level. Configure a canonical provider model with a CLI runtime such as claude-cli, then trigger models status --probe, LLM slug generation, or child-session model persistence; current main still uses embedded execution or raw provider persistence in those paths.

## Solution Assessment

Yes for the direction, but current main is incomplete. The maintainable fix is to reuse resolveCliRuntimeExecutionProvider and the existing CLI dispatch contract at each remaining helper boundary without broadening runtime namespaces.
