## Summary

Keep open. Current main still has source-level helper paths that call the embedded runner directly for queued followups, model auth probes, and LLM slug generation, and the issue is paired with an open closing PR that should be reviewed, rebased, replaced, or rejected before this issue is closed.

## What This Changes

The issue asks for CLI-backed helper runs to use the same CLI dispatch decision as the main agent runner instead of bypassing into embedded/API execution.

## Best Possible Solution

Land a rebased or replacement helper-path fix that reuses the main runner's CLI runtime resolution and dispatch pattern, keep the merged cron fix, and split the subagent persistence hardening if it should not ride with the helper repair.

## Reproduction Assessment

Yes, at source level. Configure a canonical provider with a CLI runtime and trigger queued followup execution, models status --probe, or LLM slug generation; those helpers still call runEmbeddedPiAgent without resolving the CLI execution provider first.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix for the helper paths. The open PR should be rebased or replaced so each helper preserves its existing auth, session, routing, timeout, cancellation, and tool-policy semantics while sharing the main CLI dispatch decision.
