## Summary

Keep open. Current main still has helper paths that call the embedded runner directly for queued followups, model auth probes, and LLM slug generation, and the issue is paired with open closing PR https://github.com/openclaw/openclaw/pull/72499.

## What This Changes

The issue asks for CLI-backed helper execution paths to share the main runner’s CLI runtime resolution and dispatch behavior.

## Best Possible Solution

Review, rebase, or replace https://github.com/openclaw/openclaw/pull/72499 so queued followups, probes, and slug generation share the main CLI dispatch pattern, while keeping the current-main cron fix and splitting subagent persistence hardening if needed.

## Reproduction Assessment

Yes, at source level. Configure a canonical provider with a CLI runtime and trigger queued followup execution, `openclaw models status --probe`, or LLM slug generation; current main still calls `runEmbeddedPiAgent` directly in those helpers without resolving the CLI execution provider first.

## Solution Assessment

Yes. Reusing the main runner’s `resolveCliRuntimeExecutionProvider` plus `isCliProvider` plus `runCliAgent` pattern is the narrow maintainable direction, provided each helper preserves its existing auth, session, timeout, cancellation, and tool-policy semantics.
