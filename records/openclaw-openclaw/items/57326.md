## Label Justifications

- `P1`: Multiple helper paths can still misroute CLI-backed real agent/channel workflows through the wrong runtime.
- `impact:security`: Prompts intended for a local CLI backend can still flow through embedded/API helper execution.
- `impact:session-state`: The remaining subagent and helper paths can persist or reuse provider/runtime/session state inconsistently.
- `impact:auth-provider`: The bug is centered on provider, runtime, and auth-profile selection for CLI-backed models.

## Summary

Keep open. Current main has picked up the shared CLI-routing pattern for queued followups and cron, but model auth probes, LLM slug generation, and child-session runtime-model persistence still have source-level paths that call or persist the canonical provider without resolving the configured CLI runtime first; the issue is also paired with the open closing PR https://github.com/openclaw/openclaw/pull/72499.

## What This Changes

The issue asks for all CLI-backed helper paths to use the same CLI dispatch/runtime-resolution path as normal agent turns.

## Best Possible Solution

Land a rebased or replacement fix for the remaining probe, slug, and child-session persistence paths using `resolveCliRuntimeExecutionProvider` and CLI dispatch, while preserving the already-merged followup and cron behavior.

## Reproduction Assessment

Yes, at source level. Configure a canonical provider model with a CLI runtime such as `agentRuntime.id: "claude-cli"`, then trigger `models status --probe`, LLM slug generation, or child-session model persistence; current main still uses embedded execution or raw provider persistence in those paths.

## Solution Assessment

Yes for the direction, but not complete on current main. The maintainable fix is to reuse the existing CLI runtime-resolution and dispatch contract while preserving each helper's auth, session, timeout, cancellation, and tool-policy semantics.
