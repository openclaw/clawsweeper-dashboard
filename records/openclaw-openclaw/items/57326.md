## Label Justifications

- `P1`: Multiple CLI-backed helper paths can still route or persist runtime state incorrectly, breaking real agent/channel workflows for CLI-runtime users.
- `impact:security`: Prompts and auth context intended for a local CLI backend can still enter embedded/API helper execution when helpers bypass CLI dispatch.
- `impact:session-state`: The subagent persistence path can store raw provider state instead of the resolved CLI runtime for later turns.
- `impact:auth-provider`: The report centers on provider, runtime, model, and auth-profile selection for CLI-backed models.

## Summary

Keep open. Current main fixes the queued-followup and cron portions, but `models status --probe`, opt-in LLM slug generation, and initial subagent runtime persistence still have source-level CLI runtime gaps; the broad helper PR was closed unmerged, and the open compaction PR only covers one adjacent slice.

## What This Changes

The issue asks for every CLI-backed helper path to use the same resolved CLI dispatch decision as the main agent runner instead of falling through to embedded/API execution or raw provider persistence.

## Best Possible Solution

Land a narrow maintainer-reviewed fix that resolves the CLI runtime before model probes, slug generation, and child-session runtime persistence, while letting the compaction PR and broader PI billing-policy issue remain separate.

## Reproduction Assessment

Yes, at source level. Configure a canonical provider model with a CLI runtime such as `claude-cli`, then trigger `models status --probe`, opt-in LLM slug generation, or initial subagent session persistence; current main still uses embedded execution or raw provider persistence in those paths.

## Solution Assessment

Yes for the reported direction, but current main is incomplete. The maintainable fix is to reuse `resolveCliRuntimeExecutionProvider` and the existing CLI dispatch contract at each remaining helper boundary without treating CLI backend ids as provider hook aliases or embedded harness ids.
