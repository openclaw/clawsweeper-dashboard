## Summary

Keep open. Current main still leaves the reported helper surfaces on direct embedded/API paths or checks CLI-ness before resolving the configured CLI runtime, and the issue is paired with open closing PR #72499, so it should stay open until that PR or an equivalent fix lands or is rejected.

## What This Changes

The issue requests routing queued followups, model auth probes, LLM slug generation, and cron runs through CLI runtime resolution and runCliAgent when the selected runtime is CLI-backed.

## Best Possible Solution

Keep this issue open and review, rebase, land, or reject PR #72499, or land an equivalent scoped patch. The fix should reuse the main runner’s CLI runtime resolution before isCliProvider, dispatch CLI-backed helper runs through runCliAgent, preserve each helper’s auth/session/routing/cancellation context, and keep the later persisted-pin hardening separate unless maintainers deliberately fold it in.

## Reproduction Assessment

Yes. Current source inspection gives a high-confidence reproduction path: configure a canonical provider with a CLI runtime and trigger queued followup, probe, slug, or cron execution; those helper paths still use embedded dispatch or pre-resolution CLI checks on main.

## Solution Assessment

Yes for the requested direction. The narrow maintainable solution is to share the main runner’s resolveCliRuntimeExecutionProvider -> isCliProvider -> runCliAgent pattern in the helper paths, with focused regression tests for each surface.
