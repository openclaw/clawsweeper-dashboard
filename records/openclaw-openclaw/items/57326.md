## Label Justifications

- `P1`: Multiple CLI-backed helper paths can still route or persist runtime state incorrectly, breaking real agent/channel workflows for CLI-runtime users.
- `impact:security`: Prompts and auth context intended for a local CLI backend can still enter embedded/API helper execution when helpers bypass CLI dispatch.
- `impact:session-state`: The subagent persistence path can store raw provider state instead of the resolved CLI runtime for later turns.
- `impact:auth-provider`: The report centers on provider, runtime, model, and auth-profile selection for CLI-backed models.

## Summary

Keep open. Current main fixes the queued-followup and cron portions, but model auth probes, LLM slug generation, child-session model persistence, and the related in-process compaction path still have source-level CLI runtime gaps; the broad helper PR that claimed to close this was closed unmerged, while https://github.com/openclaw/openclaw/pull/84966 only covers the compaction slice.

## What This Changes

The issue requests that CLI-backed helper paths use the same resolved CLI runtime dispatch as normal agent turns.

## Best Possible Solution

Land a narrow maintainer-reviewed fix that resolves the CLI runtime before each remaining helper boundary, while keeping broader PI/cliBackends billing policy in the canonical follow-up issue.

## Reproduction Assessment

Yes, at source level. Configure a canonical provider model with a CLI runtime such as claude-cli, then trigger models status --probe, LLM slug generation, or initial child-session model persistence; current main still uses embedded execution or raw provider persistence in those paths.

## Solution Assessment

Yes for the direction, but current main is incomplete. The maintainable fix is to reuse resolveCliRuntimeExecutionProvider and the existing CLI dispatch contract at each remaining helper boundary without broadening runtime namespaces.
