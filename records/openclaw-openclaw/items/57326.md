## Summary

Keep open: current main and the latest release still contain the reported diagnostic/slug embedded helper paths and raw subagent/gateway provider boundaries. Queued followups and cron have been fixed on main, but the remaining source gaps plus recent 2026.6.5 repro comments mean this is not implemented or safely superseded yet.

## What This Changes

The issue asks OpenClaw to route CLI-backed helper and dispatch paths through the same CLI runtime resolution used by the main agent runner.

## Best Possible Solution

Preserve canonical provider/model session state and carry a resolved CLI execution provider at helper and runner boundaries, using the queued-followup and cron fixes as the sibling pattern while splitting or linking subagent completion-handoff work where appropriate.

## Reproduction Assessment

Yes, at source level: current main still routes `models status --probe` and the LLM slug generator through embedded execution, and still lacks a prepared CLI execution-provider fact at the inspected subagent/gateway boundary. I did not run a live CLI-backed reproduction in this read-only review.

## Solution Assessment

No, not as a persistence-only patch that stores CLI runtime ids in canonical `modelProvider` fields. The safer fix is to keep canonical model identity intact and resolve CLI execution providers at the helper or dispatch boundary, with maintainers deciding how to split the subagent overlap.
