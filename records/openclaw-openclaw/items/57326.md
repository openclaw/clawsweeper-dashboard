## Label Justifications

- `P1`: Multiple production reports show CLI-backed workflows misrouting or failing in helper paths that affect real agent/channel execution.
- `impact:security`: Prompts expected to stay on a local CLI backend can still be sent through embedded/API helper paths.
- `impact:auth-provider`: The bug is about provider/runtime/auth-profile routing for CLI-backed models such as claude-cli.
- `impact:session-state`: Several reported paths involve preserving active runtime config, session bindings, or persisted child-session modelProvider state.

## Summary

Keep open. Current main has picked up the shared CLI-routing pattern for queued followups and cron, but model auth probes, LLM slug generation, and subagent runtime-model persistence still have source-level paths that use or persist canonical provider/model data without resolving the configured CLI runtime first; the issue is also paired with the open closing PR https://github.com/openclaw/openclaw/pull/72499.

## What This Changes

The issue asks CLI-backed helper paths using canonical provider refs to dispatch through the configured CLI backend instead of embedded/API helpers.

## Best Possible Solution

Land a rebased or replacement fix for the remaining probe, slug, and child-session runtime paths using the shared CLI dispatch/runtime-resolution contract, then split broader PI-harness billing asymmetry into its own canonical item if it remains.

## Reproduction Assessment

Yes, at source level. Configure a canonical provider model with a CLI runtime such as agentRuntime.id=claude-cli, then trigger models status --probe, LLM slug generation, or child-session model persistence; current main still uses embedded execution or raw provider persistence in those paths.

## Solution Assessment

Yes for the direction, but not complete on current main. The maintainable fix is to reuse resolveCliRuntimeExecutionProvider, isCliProvider, and runCliAgent while preserving each helper's auth, session, timeout, cancellation, and tool-policy semantics.
