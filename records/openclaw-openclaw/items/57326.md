## Summary

Keep open. Current main has partially fixed the cron runtime-selection branch, but queued followups, model auth probes, LLM slug generation, and the newer subagent runtime persistence report still have source-level paths that do not resolve CLI runtime aliases before embedded/API dispatch; the issue is also paired with an open closing PR that should be reviewed, rebased, landed, replaced, or rejected before closing.

## What This Changes

The issue asks for CLI-backed helper paths to use the same CLI runtime resolution and `runCliAgent()` dispatch as the main agent runner.

## Best Possible Solution

Keep the issue open, rebase or replace the paired helper-path PR for followup/probe/slug dispatch, keep the already-merged cron fix, and split the newer subagent runtime-persistence hardening if it should not ride with the helper fix.

## Reproduction Assessment

Yes, at source level. Configure a canonical provider with a CLI runtime and trigger queued followup execution, `models status --probe`, or LLM slug generation; those current-main helpers still enter `runEmbeddedPiAgent` directly instead of resolving to `runCliAgent`.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix for the helper paths: reuse the main runner's `resolveCliRuntimeExecutionProvider` to `isCliProvider` to `runCliAgent` pattern while preserving each helper's auth, session, routing, timeout, tool, and cancellation semantics.
