## Label Justifications

- `P1`: Multiple CLI-backed helper paths can still break real agent/channel workflows for users relying on local CLI runtimes.
- `impact:security`: Prompts and auth context intended for a local CLI backend can still enter embedded/API helper execution when helpers bypass CLI dispatch.
- `impact:session-state`: The subagent persistence path can store raw provider state instead of the resolved CLI runtime for later turns.
- `impact:auth-provider`: The report centers on provider, runtime, model, and auth-profile selection for CLI-backed models.

## Summary

Keep open. Current main has fixed the cron, followup, and compaction slices, but `models status --probe`, LLM slug generation, and initial subagent model persistence still bypass or fail to persist the resolved CLI runtime, so the central helper-routing problem remains real.

## What This Changes

The issue asks for CLI-backed helper paths to resolve CLI runtimes consistently and dispatch or persist through the CLI runner path instead of embedded/API helper paths.

## Best Possible Solution

Land a narrow maintainer-reviewed fix that reuses `resolveCliRuntimeExecutionProvider` and the existing CLI dispatch/session contracts at each remaining helper boundary, with focused tests and redacted real CLI-backed smoke proof.

## Reproduction Assessment

Yes, at source level. Configure a canonical provider model with a CLI runtime such as `anthropic/...` plus `agentRuntime.id: "claude-cli"`, then trigger `models status --probe`, LLM slug generation, or initial child-session model persistence; current main still uses embedded execution or raw provider persistence in those paths.

## Solution Assessment

Yes for the direction, but current main is incomplete. The maintainable fix is to reuse the existing CLI runtime resolver and CLI dispatch contract at the remaining helper boundaries without broadening runtime namespaces or adding a new config mode.
