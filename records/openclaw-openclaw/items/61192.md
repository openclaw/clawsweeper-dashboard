## Summary

Keep open. Current main still lacks a root-cause-proven fix for the reported Telegram forum-topic 400: raw Telegram display names still enter context, current OpenAI/Copilot paths do not show that value becoming a Chat Completions `messages[].name`, and the target-repo fix attempts were closed unmerged after reviewers asked for a redacted failing outbound payload.

## What This Changes

The issue reports Telegram forum-topic messages failing with GitHub Copilot/OpenAI models when participant display names contain characters that are invalid for an OpenAI message `name` field.

## Best Possible Solution

Capture a redacted failing provider payload, identify the exact rejected field, then patch only that serializer or provider boundary while preserving raw sender names for policy and display.

## Reproduction Assessment

Unclear. The report has concrete Telegram forum-topic steps and an error log, but I did not have a live Telegram plus GitHub Copilot run and current source does not show Telegram `SenderName` becoming an OpenAI Chat Completions `messages[].name` field.

## Solution Assessment

Unclear. Sanitizing a guessed sender-name field is not the best proven fix; the safer path is to use the rejected outbound payload to target the actual provider-bound field and keep raw display names intact upstream.
