## Summary

Keep open. Current main still does not have a root-cause-proven fix for the reported Telegram forum-topic failure: raw Telegram sender names are preserved as prompt/context data, the inspected OpenAI serializers do not show that value becoming a Chat Completions `messages[i].name`, and both target-repo fix attempts were closed unmerged after reviewers asked for a redacted failing outbound payload.

## What This Changes

Telegram forum-topic messages from users with non-ASCII display names are reported to make GitHub Copilot OpenAI models return `400 invalid_request_body` and put the provider profile into cooldown.

## Best Possible Solution

Capture a redacted failing provider payload, identify the exact rejected field, then patch only that serializer or provider boundary while preserving raw sender names for policy and display.

## Reproduction Assessment

Unclear. The report has concrete Telegram forum-topic steps and an error log, but I did not have a live Telegram plus GitHub Copilot run and current source does not show Telegram `SenderName` becoming an OpenAI Chat Completions `messages[i].name` field.

## Solution Assessment

Unclear. Sanitizing a guessed sender-name field is not the best proven fix; the safer path is to use the rejected outbound payload to target the actual provider-bound field and keep raw display names intact upstream.
