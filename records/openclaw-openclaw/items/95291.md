## What This Changes

The issue reports Feishu/Lark `message` tool file and image sends failing with CDN/connection upload errors while standalone Lark SDK uploads succeed with the same credentials and files.

## Best Possible Solution

Review and land a narrow Feishu client wrapper fix, or close the linked candidate if rejected; then close this issue only after merged proof shows message-tool file/image sends succeed through the SDK upload path.

## Reproduction Assessment

No independent high-confidence live current-main reproduction was run in this review. The reporter gives deterministic Feishu 2026.6.6 failures, and current source still traces `filePath`/`media` sends into SDK multipart Buffer uploads through OpenClaw's timeout wrapper.

## Solution Assessment

Yes for the linked candidate direction: the wrapper boundary is where OpenClaw diverges from the standalone SDK upload path, so fixing request shaping there is narrower than adding downstream fallback behavior. The implementation still needs to land before this issue can close.
