## Summary

Keep open: the underlying NO_REPLY leak in outbound message-tool sends still exists on current main and the latest release, but this PR is not merge-ready because it can silently drop substantive group/channel sends that merely contain the token.

## What This Changes

The branch adds NO_REPLY parsing and suppressed-success handling to core outbound message send payload construction, plus Vitest coverage for direct trailing-token sends and token-only dry-run sends.

## Best Possible Solution

Use the existing silent-token helpers to suppress exact token-only sends and strip leading/trailing control tokens while preserving substantive visible text across direct and group/channel contexts, then add focused tests plus redacted real send proof.

## Reproduction Assessment

Yes from source inspection: current main and v2026.6.5 pass outbound send text through buildSendPayloadParts without silent-token handling, while docs define exact NO_REPLY as non-deliverable. I did not run tests because this review is read-only.

## Solution Assessment

No. The direction is useful, but the implementation is too broad for group/channel sends and should preserve substantive visible text instead of erasing it whenever the literal token appears.
