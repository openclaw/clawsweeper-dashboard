## Summary

Keep open: the PR targets a real Google Gemma routing gap, but the current patch is not mergeable because it can create inconsistent `api`/`baseUrl` pairs for dynamic CLI-sourced Gemma models and explicit provider configs.

## What This Changes

The PR forces dynamic and explicit Google Gemma models toward `google-generative-ai` and adds a provider-policy regression test.

## Best Possible Solution

Keep the fix in the Google plugin, but switch Gemma transport atomically: pair native `api` with a compatible native base URL, and normalize explicit configs only when the runtime API/baseUrl combination remains valid for all affected models.

## Reproduction Assessment

Yes, at source level. Current main still leaves explicit Gemma configs on `openai-completions`, and the PR diff creates reproducible inconsistent `api`/`baseUrl` cases for CLI template clones and omitted model-level APIs.

## Solution Assessment

No. The PR targets the right Google plugin surfaces, but `api` and `baseUrl` need to change together and provider-level normalization must preserve OpenAI-compatible consumers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Patch Gemma baseUrl when forcing the native API:** `extensions/google/provider-models.ts:175-176`
  - body: Setting `api: "google-generative-ai"` unconditionally also affects `google-gemini-cli` clones, where template selection can keep the CLI `https://cloudcode-pa.googleapis.com` base URL. The native transport then appends `/models/...:streamGenerateContent` to that incompatible host/path, so CLI Gemma requests can fail; patch or select a native Generative Language base URL together with the API switch.
  - confidence: 0.91
- **[P1] Gate baseUrl normalization on native Gemma consumers:** `extensions/google/provider-policy.ts:108-115`
  - body: The new explicit-Gemma check strips `/openai` from the provider base URL whenever any Gemma id exists, but `normalizeProviderModels` only changes models with explicit `api: "openai-completions"`. A Gemma model that inherits the provider API can still send OpenAI-compatible requests to `/v1beta`, and sibling OpenAI-compatible models under the same provider lose their valid endpoint.
  - confidence: 0.89
