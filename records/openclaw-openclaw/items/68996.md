## Summary

Keep open: the PR addresses a real Google Gemma routing bug, but the current patch can leave `api` and `baseUrl` inconsistent for CLI-sourced Gemma clones and can strip `/openai` from provider config even when some models still use OpenAI-compatible transport.

## What This Changes

The branch forces Gemma dynamic models and explicit Gemma provider config entries onto `google-generative-ai`, with Google provider-policy regression coverage.

## Best Possible Solution

Switch Gemma to native Google transport atomically in the Google plugin, updating both `api` and a compatible `baseUrl` only for affected Gemma entries while preserving OpenAI-compatible sibling models.

## Reproduction Assessment

Yes, at source level. The PR diff and current runtime contracts show the two failing paths: CLI-sourced Gemma clones can retain a CLI baseUrl after an API switch, and provider config normalization can strip `/openai` while some models still use `openai-completions`.

## Solution Assessment

No. The direction is right, but the implementation needs to keep `api` and `baseUrl` compatible and scope provider-level baseUrl normalization to models whose API is actually switched.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Patch Gemma baseUrl when forcing native API:** `extensions/google/provider-models.ts:177`
  - body: Setting the Gemma patch to `api: "google-generative-ai"` can still clone from a `google-gemini-cli` template first, preserving `https://cloudcode-pa.googleapis.com` as `baseUrl`. The native Google transport then builds `:streamGenerateContent` on that incompatible host, so either patch a compatible native baseUrl with the API switch or avoid switching CLI-sourced clones.
  - confidence: 0.91
- **[P1] Gate Gemma baseUrl normalization on an API switch:** `extensions/google/provider-policy.ts:111-118`
  - body: The new Gemma predicate strips `/openai` from the provider baseUrl whenever any Gemma id exists, but `normalizeProviderModels` only switches models whose `model.api` is explicitly `openai-completions`. A Gemma model that omits `api`, or an OpenAI-compatible sibling, can still use `providerConfig.api` while requests are sent to the native base path; gate normalization on the actual model API switch.
  - confidence: 0.89
