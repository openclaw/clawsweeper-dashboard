## Summary

Keep open: this PR targets a real Google Gemma routing bug, but the current branch still introduces two source-proven `api`/`baseUrl` mismatches that should block merge until repaired.

## What This Changes

The branch forces Gemma dynamic resolution and explicit Google provider model configs from OpenAI-compatible transport to `google-generative-ai`, with a provider-policy regression test.

## Best Possible Solution

Switch Gemma to native Google transport atomically in the Google plugin, keeping `api` and `baseUrl` compatible while preserving explicit OpenAI-compatible sibling entries.

## Reproduction Assessment

Yes, at source level. Current main and the PR diff expose both failing paths, and the PR discussion includes before/after endpoint checks plus copied after-fix E2E output; I did not run a credentialed Google call.

## Solution Assessment

No. The PR targets the right provider-owned surfaces, but the implementation must keep `api` and `baseUrl` compatible and avoid provider-level normalization that breaks OpenAI-compatible siblings.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Patch Gemma baseUrl with native API:** `extensions/google/provider-models.ts:175-176`
  - body: The new Gemma patch forces `api: "google-generative-ai"` after cloning, but template selection tries the requested provider first and Gemma does not set `preferExternalFirstForCli`; for `google-gemini-cli` clones this can preserve `https://cloudcode-pa.googleapis.com`. The native stream builder then appends `/models/...:streamGenerateContent`, so CLI Gemma requests route to an incompatible host/path. Patch a compatible baseUrl with the API switch or avoid switching CLI-sourced clones.
  - confidence: 0.91
- **[P1] Gate Gemma baseUrl normalization on API switch:** `extensions/google/provider-policy.ts:108-115`
  - body: This returns true for any Gemma model id, even if a model omits `api` and continues to inherit `provider.api = "openai-completions"`. That strips `/openai` from the provider baseUrl while OpenAI-compatible sibling models still use the provider transport, sending them to the native path. Only normalize the provider baseUrl when the selected Gemma entry is actually switched to native, or keep native baseUrl model-scoped.
  - confidence: 0.89
