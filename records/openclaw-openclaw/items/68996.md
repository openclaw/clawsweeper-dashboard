## Summary

Keep open: the PR is aimed at a real Google Gemma routing bug, but the current branch still introduces inconsistent `api`/`baseUrl` combinations for CLI-template clones and some explicit configs. The remaining work is narrow and provider-local, so this is a good repair-lane candidate rather than a cleanup close.

## What This Changes

The branch forces dynamic and explicit Google Gemma models toward `google-generative-ai` and adds a provider policy regression test.

## Best Possible Solution

Keep the fix in the Google plugin, but switch Gemma transport atomically: choose or patch a compatible native base URL with `api`, and only normalize provider base URLs when the affected model actually moves to native transport.

## Reproduction Assessment

Yes, at source level. Current main still leaves explicit Gemma configs pinned to `openai-completions`, and the PR diff exposes reproducible inconsistent `api`/`baseUrl` cases for CLI templates and omitted model-level APIs.

## Solution Assessment

No. The PR targets the right Google plugin surfaces, but `api` and `baseUrl` need to change together and explicit OpenAI-compatible provider settings must be preserved for unaffected models.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Patch Gemma baseUrl when forcing the native API:** `extensions/google/provider-models.ts:175-176`
  - body: The Gemma branch now forces `api: "google-generative-ai"` for every clone, including `google-gemini-cli` resolutions that check CLI templates first. Those templates carry the Cloud Code base URL, and the native transport builds `:streamGenerateContent` from the retained `baseUrl`, so CLI Gemma calls can be sent to an incompatible host/path. Patch the base URL together with the API or select only compatible native templates.
  - confidence: 0.91
- **[P1] Gate baseUrl normalization on actual Gemma API switches:** `extensions/google/provider-policy.ts:108-115`
  - body: The new explicit-Gemma check triggers provider-level native base URL normalization even when a Gemma model omits `model.api`, but `normalizeProviderModels` only switches models whose API is explicitly `openai-completions`. That can leave runtime falling back to provider `openai-completions` while `/openai` has been stripped from the base URL, and can also break sibling OpenAI-compatible models under the same provider config.
  - confidence: 0.89
