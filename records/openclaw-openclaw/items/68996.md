## Summary

Keep open; do not merge as-is. The PR targets a real Google Gemma transport problem, but the diff introduces blocking routing regressions for Gemini CLI template clones and mixed or implicit OpenAI-compatible Google configs. Google’s Gemma docs support the native Gemini API direction for Gemma 4. ([ai.google.dev](https://ai.google.dev/gemma/docs/core/gemma_on_gemini_api))

## What This Changes

The PR forces Gemma model resolution and explicit Google provider config normalization onto the native `google-generative-ai` transport and adds Google provider regression tests.

## Best Possible Solution

Keep the Gemma fix in the Google plugin, but make the transport switch atomic: dynamic Gemma clones should only use `google-generative-ai` with a compatible Generative Language base URL or template source, and explicit-config normalization should not rewrite a provider-level OpenAI-compatible base URL unless the affected model/provider API is actually native or the model gets its own native base URL. Add regression coverage for `google-gemini-cli` Gemma resolution and mixed Google configs containing both Gemma and OpenAI-compatible models.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to resolve a `google-gemini-cli` Gemma model when only a CLI `gemini-3-flash-preview` template is available, and to normalize a `google` provider config whose baseUrl ends in `/openai` with a Gemma model that omits model-level `api`.

## Solution Assessment

No. The proposed direction is close, but changing `api` without a compatible baseUrl/template and triggering provider-level native baseUrl normalization from a Gemma id alone is not the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep CLI Gemma clones on a compatible base URL:** `extensions/google/provider-models.ts:175-176`
  - body: The Gemma branch now overrides `api` for every clone, but `google-gemini-cli` resolution still tries CLI templates first. Those templates carry `baseUrl: https://cloudcode-pa.googleapis.com`, so the native Google transport will build `https://cloudcode-pa.googleapis.com/models/<gemma>:streamGenerateContent` instead of using the Generative Language endpoint. Scope the override to compatible Google templates or patch the baseUrl/source together.
  - confidence: 0.91
- **[P1] Avoid rewriting OpenAI configs before switching the model:** `extensions/google/provider-policy.ts:108-115`
  - body: The new Gemma-id check makes provider normalization strip `/openai` from the provider baseUrl even when a Gemma entry still inherits `provider.api: "openai-completions"` or when sibling models remain OpenAI-compatible. That sends chat-completions traffic to `/v1beta` instead of `/v1beta/openai`. Gate native baseUrl normalization on an actual API switch, or carry the native baseUrl on the forced Gemma model only.
  - confidence: 0.88
