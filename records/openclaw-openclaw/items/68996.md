## Summary

Keep open: this is a real Google Gemma routing fix direction, but the current diff can leave `api` and `baseUrl` mismatched for CLI-sourced Gemma clones and can over-normalize provider base URLs for configs that still use OpenAI-compatible transport.

## What This Changes

The PR forces dynamic and explicit Google Gemma model configs onto `google-generative-ai`, normalizes affected provider base URLs away from `/openai`, and adds Google provider regression coverage.

## Best Possible Solution

Switch Gemma to native Google transport atomically in the Google plugin, updating `api` and compatible `baseUrl` only for affected Gemma entries while preserving explicit OpenAI-compatible sibling models.

## Reproduction Assessment

Yes, at source level. The PR diff and current runtime contracts show that dynamic Gemma clones can retain a CLI `baseUrl` after forcing native API, and provider config normalization can strip `/openai` while a model still inherits `openai-completions`.

## Solution Assessment

No. The direction is right, but this implementation must keep `api` and `baseUrl` compatible and scope provider-level baseUrl normalization to an actual Gemma API switch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Patch Gemma baseUrl when forcing native API:** `extensions/google/provider-models.ts:175-176`
  - body: For Gemma dynamic resolution, `patch = { api: "google-generative-ai" }` can still preserve a `google-gemini-cli` template `baseUrl` such as `https://cloudcode-pa.googleapis.com`. The native transport then builds `:streamGenerateContent` from that retained base URL, so patch a compatible native base URL with the API switch or avoid switching CLI-only clones.
  - confidence: 0.91
- **[P1] Gate baseUrl normalization on an API switch:** `extensions/google/provider-policy.ts:108-115`
  - body: The new Gemma trigger normalizes provider `baseUrl` whenever any Gemma id exists, but `normalizeProviderModels` only switches models whose own `api` is exactly `openai-completions`. A Gemma model that omits `api` can still inherit provider `api: "openai-completions"` while the base URL has been stripped to `/v1beta`, breaking that model and OpenAI-compatible siblings.
  - confidence: 0.89
