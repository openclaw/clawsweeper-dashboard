## Summary

Keep open: this is a real Google Gemma routing fix direction with sufficient copied after-fix live output, but the current patch still introduces incompatible API/baseUrl combinations that can break CLI-sourced Gemma clones and explicit OpenAI-compatible provider configs.

## What This Changes

The PR forces Google Gemma dynamic and explicit model entries onto `google-generative-ai` routing and adds a provider-policy regression test.

## Best Possible Solution

Switch Gemma to native Google transport atomically in the Google plugin: update API and compatible native baseUrl only for affected Gemma entries, preserve OpenAI-compatible sibling models and CLI-only templates, and add focused regression coverage.

## Reproduction Assessment

Yes, at source level. The PR diff and current runtime contracts show a CLI-sourced Gemma clone can retain a CLI baseUrl after forcing native API, and provider config normalization can strip `/openai` while a model still inherits `openai-completions`.

## Solution Assessment

No. The direction is right, but the implementation must keep API and baseUrl compatible and scope provider-level baseUrl normalization to the exact Gemma models whose API is switched.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Patch Gemma baseUrl when forcing native API:** `extensions/google/provider-models.ts:175-176`
  - body: Setting `api: "google-generative-ai"` for every Gemma clone can preserve a `google-gemini-cli` template `baseUrl` such as `https://cloudcode-pa.googleapis.com`. The native transport then builds `:streamGenerateContent` requests from that retained baseUrl, so CLI-sourced Gemma models can route to an incompatible host/path; patch a compatible native baseUrl with the API switch or avoid switching CLI-only clones.
  - confidence: 0.91
- **[P1] Gate baseUrl normalization on an API switch:** `extensions/google/provider-policy.ts:108-115`
  - body: This Gemma check strips `/openai` from the provider baseUrl whenever any Gemma id exists, even when a model omits `api` and still inherits `providerConfig.api = "openai-completions"`. That sends OpenAI-compatible requests to the native path and can also break sibling OpenAI-compatible models under the same provider config; only normalize the provider baseUrl when the affected model API is actually switched.
  - confidence: 0.89
