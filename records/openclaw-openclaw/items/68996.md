## Summary

Keep open: this PR targets a real Google Gemma routing bug and includes usable contributor live-output proof, but the current diff can create incompatible API/baseUrl pairs for CLI-sourced clones and explicit OpenAI-compatible provider configs.

## What This Changes

The PR forces Gemma forward-compat and explicit Google provider model configs from OpenAI-compatible routing to `google-generative-ai` and adds a provider-policy regression test.

## Best Possible Solution

Switch Gemma to native Google transport atomically in the Google plugin, keeping API and baseUrl compatible for affected Gemma models while preserving OpenAI-compatible sibling models and CLI-only templates.

## Reproduction Assessment

Yes, at source level. The PR diff and current runtime contracts show a CLI-sourced Gemma clone can retain a CLI baseUrl after forcing native API, and provider config normalization can strip `/openai` while a model still inherits `openai-completions`.

## Solution Assessment

No. The direction is right, but the implementation must keep API and baseUrl compatible and scope provider-level baseUrl normalization to the exact Gemma models whose API is switched.

- **[P1] Patch Gemma baseUrl when forcing native API:** `extensions/google/provider-models.ts:175`
  - body: Setting `api: "google-generative-ai"` for every Gemma clone can preserve a `google-gemini-cli` template `baseUrl` such as `https://cloudcode-pa.googleapis.com`. That sends OpenAI-compatible requests to the native path and can also break sibling OpenAI-compatible models under the same provider config; only normalize the provider baseUrl when the affected model API is actually switched.
  - confidence: 0.89
