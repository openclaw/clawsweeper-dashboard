## Summary

Keep open: the branch addresses a real Google Gemma routing bug, and current main does not fully cover explicit Gemma configs pinned to OpenAI-compatible transport. The PR is not mergeable as-is because two source-proven API/baseUrl compatibility blockers remain in the Google plugin path.

## What This Changes

The PR forces Gemma dynamic models and explicit Gemma provider config entries to use `google-generative-ai`, with Google provider-policy regression coverage.

## Best Possible Solution

Switch Gemma to native Google transport atomically in the Google plugin, updating `api` and compatible `baseUrl` only for affected Gemma entries while preserving explicit OpenAI-compatible sibling models.

## Reproduction Assessment

Yes, at source level. The PR diff and current code show that dynamic Gemma clones can retain a CLI baseUrl after forcing native API, and provider config normalization can strip `/openai` while a model still inherits `openai-completions`.

## Solution Assessment

No. The direction is right, but the implementation must keep `api` and `baseUrl` compatible and must scope provider-level baseUrl normalization to an actual Gemma API switch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Patch Gemma baseUrl when forcing native API:** `extensions/google/provider-models.ts:175-176`
  - body: Setting `api: "google-generative-ai"` for every Gemma clone can preserve a `google-gemini-cli` template baseUrl such as `https://cloudcode-pa.googleapis.com`. The native transport then builds `:streamGenerateContent` requests from that retained baseUrl, so CLI-sourced Gemma models can route to an incompatible host/path; patch a compatible native baseUrl with the API switch or avoid switching CLI-only clones.
  - confidence: 0.91
- **[P1] Gate baseUrl normalization on an API switch:** `extensions/google/provider-policy.ts:108-115`
  - body: This Gemma check strips `/openai` from the provider baseUrl whenever any Gemma id exists, even when a model omits `api` and still inherits `providerConfig.api = "openai-completions"`. That sends OpenAI-compatible requests to the native path and can also break sibling OpenAI-compatible models under the same provider config; only normalize the provider baseUrl when the affected model API is actually switched.
  - confidence: 0.89
