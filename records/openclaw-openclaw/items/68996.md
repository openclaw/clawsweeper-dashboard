## Summary

Keep this PR open, but it should not merge as-is. Current main still has the Gemma/native-API routing gap, and the PR is pointed at the right Google plugin boundary, but the patch can create inconsistent `api`/`baseUrl` pairs for `google-gemini-cli` clones and explicit OpenAI-compatible Google configs.

## What This Changes

The PR forces Google Gemma model resolution and explicit Google provider config normalization toward the native `google-generative-ai` transport and adds Google provider policy/model regression tests.

## Best Possible Solution

Keep the fix in the Google plugin, but make the Gemma transport switch atomic: dynamic Gemma resolution should only use `google-generative-ai` with a compatible Generative Language base URL or template source, explicit-config normalization should switch model API and base URL together without breaking mixed OpenAI-compatible configs, and the PR should add focused regression tests plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to resolve a `google-gemini-cli` Gemma model when only a CLI `gemini-3-flash-preview` template is available, and to normalize a Google provider config whose base URL ends in `/v1beta/openai` with a Gemma entry that omits model-level `api`. The current code and PR diff show the mismatched `api`/`baseUrl` paths without needing live credentials.

## Solution Assessment

No. The PR targets the right owner module, but changing `api` without a compatible base URL/template and triggering provider-level native base URL normalization from a Gemma id alone is not the narrowest maintainable fix. The safer approach is to patch or select the native model transport as one coherent model-level change and preserve explicit OpenAI-compatible provider settings for unaffected models.
