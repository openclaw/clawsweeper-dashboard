## Summary

Keep open: the PR targets a real Google Gemma routing problem and has usable copied live output, but the patch is not mergeable because it can pair the native API with incompatible CLI base URLs and can strip `/openai` from provider configs without actually switching the affected model to native transport.

## What This Changes

The PR changes Google Gemma dynamic model resolution and explicit provider config normalization so Gemma models use the native Google Generative AI transport instead of the OpenAI-compatible endpoint.

## Best Possible Solution

Keep the fix in the Google plugin, but make Gemma transport switching atomic: change `api` and `baseUrl` together, and only normalize provider-level base URLs when the affected model is actually moved to native transport.

## Reproduction Assessment

Yes, at source level. Current main still has the explicit-config gap, and the PR diff exposes deterministic inconsistent `api`/`baseUrl` cases for CLI templates and model entries that omit a model-level API.

## Solution Assessment

No. The PR targets the right Google plugin surfaces, but the implementation needs to switch API and base URL as one transport decision and preserve OpenAI-compatible provider settings for unaffected models.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Patch Gemma baseUrl when forcing the native API:** `extensions/google/provider-models.ts:175-176`
  - body: The Gemma branch forces `api: "google-generative-ai"` for every clone, including `google-gemini-cli` resolutions that can source a CLI template first. Those templates carry the Cloud Code base URL, so the native transport can build `:streamGenerateContent` requests against an incompatible host/path; patch the base URL together with the API or only force native transport for compatible templates.
  - confidence: 0.91
- **[P1] Gate baseUrl normalization on actual Gemma API switches:** `extensions/google/provider-policy.ts:108-115`
  - body: The new explicit-Gemma check triggers provider-level native base URL normalization even when a Gemma model omits `model.api`, while `normalizeProviderModels` only switches models whose API is explicitly `openai-completions`. That can leave the model falling back to provider `openai-completions` after `/openai` has been stripped, and can also break sibling OpenAI-compatible models in the same provider config.
  - confidence: 0.89
