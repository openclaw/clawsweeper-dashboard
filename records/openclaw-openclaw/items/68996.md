## Summary

Keep open: the PR addresses a real Google Gemma routing bug and includes usable live-output proof, but the current diff still creates inconsistent `api`/`baseUrl` combinations for CLI-sourced clones and explicit configs with omitted model APIs, so it should not merge as-is.

## What This Changes

The PR changes Google Gemma model resolution and explicit provider config normalization so Gemma models use the native Google Generative AI API instead of the OpenAI-compatible endpoint.

## Best Possible Solution

Keep the fix in the Google plugin, but switch Gemma transport atomically so `api` and `baseUrl` remain compatible and explicit config normalization preserves sibling OpenAI-compatible models.

## Reproduction Assessment

Yes, at source level. Current main and the PR diff show the relevant model/config paths, and the PR discussion provides before/after endpoint and E2E output; no credentialed Google call was run during this read-only review.

## Solution Assessment

No. The PR targets the correct Google plugin surfaces, but the API switch must carry a compatible baseUrl and explicit config normalization must only rewrite baseUrl when the Gemma model actually switches to native transport.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Patch Gemma baseUrl when forcing native API:** `extensions/google/provider-models.ts:175-176`
  - body: Setting `api: "google-generative-ai"` for every Gemma clone keeps the template `baseUrl`. For `google-gemini-cli`, template selection can clone from the Cloud Code host first, and the native transport will then append `/models/...:streamGenerateContent` to an incompatible endpoint.
  - confidence: 0.91
- **[P1] Gate Gemma baseUrl normalization on an API switch:** `extensions/google/provider-policy.ts:108-115`
  - body: The new explicit-Gemma check strips `/openai` from the provider base URL whenever any Gemma id exists, but models with omitted `api` can still inherit `provider.api = "openai-completions"`. That sends OpenAI-compatible requests to the native path and can also break sibling OpenAI-compatible models under the same provider config.
  - confidence: 0.89
