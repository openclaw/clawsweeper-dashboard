## Summary

Keep open. Current main still lacks a core assistant-output repetition circuit breaker, and the open related PR is an implementation candidate rather than shipped behavior; the requested halt-and-confirm semantics still need maintainer product/API judgment.

## What This Changes

Add a core streaming guard that detects repeated assistant output, halts the stream, alerts the user, and requires confirmation before resume or cancel.

## Best Possible Solution

Review PR #39961 or a replacement as the canonical implementation path, then land a conservative core guard at the assistant stream subscription or runner boundary with documented config, provider-safe abort behavior, final-flush suppression, and clear stop/cancel user messaging.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature request. The high-confidence absence check is source-backed: repeated assistant text_delta chunks currently flow through append/emit paths without any repetition threshold or halt-and-confirm state.

## Solution Assessment

Unclear. A core stream guard is the right area, but the exact halt-and-confirm design needs maintainer judgment because aborting a provider stream is supported while resuming the same stream after approval is not established by the inspected code.
