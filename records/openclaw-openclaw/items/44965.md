## Summary

Current main still lacks a semantic text-stream repetition breaker, and no viable open or merged PR owns the remaining work. The reliability problem remains valid, but the requested manual-resume workflow needs a product decision; likely related people include steipete and vincentkoc for current runtime ownership, with Liu Yuan and Sk Akram connected to adjacent watchdog and loop-detection history.

## What This Changes

Requests a core streaming repetition circuit breaker that halts repeated output, alerts the user, and waits for manual approval or cancellation.

## Best Possible Solution

Add a channel-independent repetition detector at the canonical assistant stream boundary that aborts with a distinct terminal outcome, quarantines buffered repetitive output, and removes or marks the aborted turn before a fresh user-triggered retry; do not resume the same provider stream.

## Reproduction Assessment

Yes at source level: continuous chunks refresh the activity-based idle watchdog, while the current stream path has no semantic repetition detector. The historical reports are concrete, but no deterministic current-main model reproduction was run.

## Solution Assessment

No, not exactly as proposed. Automatic abort and output quarantine are a maintainable core reliability safeguard, but `/approve` should start a clean retry rather than attempt to resume the same provider stream, and transcript handling needs an explicit terminal-state design.
