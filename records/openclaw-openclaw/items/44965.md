## Summary

Keep open: current main and v2026.6.8 still lack a semantic streamed-text repetition guard, while existing safeguards cover tool-call loops, duplicate delivery events, or idle silence rather than active repeated assistant text. The closed implementation PR is no longer a viable landing path, so this issue remains the canonical tracker and needs maintainer product direction before implementation.

## What This Changes

The issue requests a core stream-level safeguard that detects repeated assistant text, halts the stream, alerts the user, and requires manual confirmation before continuing.

## Best Possible Solution

Keep this as the canonical tracker and design a bounded core stream/session guard with replay-safe abort, no buffered spam leak, clear user alerting, transcript cleanup policy, and focused current-main regression coverage before implementation lands.

## Reproduction Assessment

Yes at source level: current main streams assistant text without semantic repetition detection, while the existing loop detector is tool-call scoped. I did not run a live model-loop reproduction in this read-only review.

## Solution Assessment

Unclear. A stream-level guard is plausible and current main lacks it, but the halt-and-confirm flow needs maintainer direction on detection scope, abort behavior, false positives, and history cleanup.
