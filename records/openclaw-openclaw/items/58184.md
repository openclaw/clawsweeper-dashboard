## Summary

Keep this PR open. The latest diff is still not merge-ready: it introduces a test-file syntax error, keeps a blanket `wss:` source that leaves the advertised secure WebSocket exfiltration path open, and drops `ws:` despite current code and docs still supporting remote `gatewayUrl=ws://...` Control UI workflows.

## What This Changes

The PR changes the Control UI CSP helper and tests to remove `ws:`, keep a blanket `wss:` source, add an `extraConnectSrc` option, and add regression tests around those connect-src values.

## Best Possible Solution

Define the Control UI connection policy explicitly: allow same-origin by default and, if remote gateways remain supported, add exact configured gateway origins without blanket `ws:` or `wss:` scheme sources. Tests and docs should cover the exfiltration block plus the retained or intentionally retired remote `gatewayUrl` workflow.

## Reproduction Assessment

Yes. The current failure path is reproducible by static inspection: the PR diff contains an extra `});`, still emits `connect-src 'self' wss:`, and removes `ws:` while current code and docs still support `gatewayUrl=ws://...` remote WebSocket targets.

## Solution Assessment

No. This patch is not the best way to solve the issue because it neither compiles cleanly nor closes the `wss://` exfiltration class, and it regresses a documented `ws://` remote gateway path. A narrow exact-origin allowlist or same-origin proxy design is safer.
