## Summary

Keep open: the security-hardening goal is plausible, but the latest branch still permits arbitrary `wss:` egress, regresses current OpenAI Realtime/tweakcn `connect-src` origins and documented `ws://` gatewayUrl workflows, and lacks real-browser proof.

## What This Changes

This PR changes the Control UI CSP helper to emit `connect-src 'self' wss:` with an optional extra source list and adds tests for that policy.

## Best Possible Solution

Define an explicit Control UI `connect-src` policy that blocks arbitrary WebSocket exfiltration while preserving supported Realtime, tweakcn, and gatewayUrl flows through exact origins, same-origin relay/proxying, or an intentional documented deprecation.

## Reproduction Assessment

Yes at source level: current main includes blanket `ws:`/`wss:` scheme sources, and the PR head can be compared directly against the browser paths that use `fetch()` and `new WebSocket(...)`. I did not run a real browser CSP repro, so runtime proof remains a contributor gate.

## Solution Assessment

No. The current PR is not the best fix because it leaves arbitrary `wss:` egress, drops current required HTTP connect origins, and blocks documented plaintext remote gateway URLs without a maintainer decision.
