## Summary

Keep this PR open, but do not merge the current branch: it is a plausible CSP hardening direction, yet the latest diff still permits blanket `wss:` egress, drops current required HTTP connect origins, and changes documented remote gateway behavior without a maintainer security/product decision.

## What This Changes

The PR changes the Control UI CSP builder and tests to remove blanket `ws:` access, keep `wss:`, and add optional extra `connect-src` origins.

## Best Possible Solution

Define an explicit Control UI connection policy that defaults to same-origin, preserves current provider/theme origins, and handles remote gateways through validated exact origins or a documented deprecation path.

## Reproduction Assessment

Yes. Static inspection is enough: the PR head emits `connect-src 'self' wss:`, while current main still opens configured `gatewayUrl` values via `new WebSocket(...)` and fetches OpenAI Realtime plus tweakcn over HTTPS.

## Solution Assessment

No. The current patch keeps a broad secure WebSocket exfiltration channel, drops required current HTTP origins, and changes documented remote `ws://` behavior without a clear product/security policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Remove the blanket wss: source:** `src/gateway/control-ui-csp.ts:58`
  - body: The patched directive still emits `connect-src 'self' wss:`. A scheme source permits WebSocket connections to arbitrary secure hosts, so injected Control UI script could still exfiltrate over `wss://...`; this does not deliver the stated CSP hardening.
  - confidence: 0.94
- **[P2] Keep required HTTP connect origins:** `src/gateway/control-ui-csp.ts:58`
  - body: Current main allows `https://api.openai.com` for browser Realtime offers and `https://tweakcn.com` for theme imports. Replacing the directive with only `'self' wss:` drops both origins and would break those Control UI features after rebase.
  - confidence: 0.9
- **[P2] Preserve documented ws gateway URLs:** `src/gateway/control-ui-csp.ts:58`
  - body: Hardcoding `wss:` while removing `ws:` blocks cross-origin plaintext gateway targets, but the UI still accepts `gatewayUrl`, opens it with `new WebSocket(...)`, and the docs still show `gatewayUrl=ws://<gateway-host>:18789`. Preserve this with exact origins or retire it deliberately with docs and tests.
  - confidence: 0.88
