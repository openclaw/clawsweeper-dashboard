## Summary

Keep this PR open, but do not merge it as-is. The CSP concern is real, but the latest diff still permits arbitrary `wss:` connections and also breaks currently documented `ws://` remote gateway workflows.

## What This Changes

The PR changes the Control UI CSP helper and tests to remove `ws:` from `connect-src`, retain a blanket `wss:` scheme source, and add an `extraConnectSrc` option/test path.

## Best Possible Solution

Revise the CSP design so legitimate Control UI gateway connections are either same-origin/proxied or allowed by exact configured origins, while avoiding blanket `ws:` or `wss:` scheme sources. The tests and docs should cover both the exfiltration block and the supported remote-gateway workflow.

## Reproduction Assessment

Yes. The diff can be checked statically: `connect-src 'self' wss:` still allows arbitrary secure WebSocket targets, while documented `gatewayUrl=ws://...` remote dev URLs no longer match the patched policy.

## Solution Assessment

No. The proposed patch is not the best fix because it leaves the secure-WebSocket exfiltration channel open and removes support for a documented plaintext remote-gateway path; an exact-origin allowlist or same-origin proxy is the safer maintainable direction.
