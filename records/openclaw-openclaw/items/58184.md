## Summary

Keep open: the security-hardening goal is plausible, but this PR is not correct as-is because the latest head still allows arbitrary `wss:` egress, drops current required HTTP origins, and would block supported `ws://` gateway URL workflows without a maintainer product decision.

## What This Changes

The PR changes the Control UI CSP helper and tests to remove the blanket `ws:` scheme while still allowing `wss:` and optional extra `connect-src` origins.

## Best Possible Solution

Define an explicit Control UI `connect-src` policy that blocks arbitrary WebSocket egress while preserving OpenAI Realtime, tweakcn imports, and supported `gatewayUrl` workflows through exact origins, same-origin proxying, or documented deprecation.

## Reproduction Assessment

Yes. Source inspection of the latest PR head and current main shows the PR emits `connect-src 'self' wss:` while current main and docs still require HTTP origins plus `ws://`/`wss://` gateway URL support; I did not run a live browser CSP repro.

## Solution Assessment

No. Keeping blanket `wss:` while dropping required HTTP origins and `ws://` support is not the narrow maintainable fix; the safer direction needs an explicit allowlist/proxy/deprecation decision.
