## Summary

Keep open: the PR targets a real CSP hardening concern, but the current branch is not mergeable as-is because it still allows arbitrary secure WebSocket egress, drops required current-main HTTP origins, blocks documented ws:// remote gateway URLs, and lacks after-fix real browser proof.

## What This Changes

The PR changes the Control UI CSP helper to emit `connect-src 'self' wss:` with optional extra origins and adds tests for that policy.

## Best Possible Solution

Define an explicit Control UI `connect-src` strategy that blocks arbitrary WebSocket egress while preserving OpenAI Realtime, tweakcn imports, and documented remote `gatewayUrl` workflows through exact origins, proxying, or a deliberate docs-backed deprecation.

## Reproduction Assessment

Yes for the review findings: static inspection of the PR head and current main shows the changed directive, required current origins, and documented cross-origin gateway paths. I did not run a live browser CSP repro in this read-only review.

## Solution Assessment

No. The current patch is not the best fix because it keeps blanket `wss:`, removes current HTTP connect origins, and breaks documented `ws://` gatewayUrl use; the safer path is an explicit policy decision with matching tests and browser proof.
