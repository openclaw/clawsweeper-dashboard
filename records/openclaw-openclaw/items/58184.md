## Summary

Keep this PR open, but do not merge the current head. The hardening goal is valid, yet the latest diff still leaves arbitrary secure WebSocket connections allowed via `wss:` and removes documented `ws://` remote-gateway support without a replacement exact-origin policy.

## What This Changes

The PR changes the Control UI CSP helper and tests to drop the blanket `ws:` scheme source, keep a blanket `wss:` scheme source, and add an `extraConnectSrc` option/test path.

## Best Possible Solution

Revise the CSP design so legitimate Control UI gateway connections are either same-origin/proxied or allowed by exact configured `ws://`/`wss://` origins, while removing blanket WebSocket scheme sources. The tests and docs should cover both the exfiltration block and the supported remote-gateway workflow.

## Reproduction Assessment

Yes. The regression is reproducible by inspecting the emitted policy and runtime path: the PR still emits `connect-src 'self' wss:` for arbitrary secure WebSocket targets, and the documented `gatewayUrl=ws://...` path still flows into `new WebSocket(...)` but would no longer match the patched policy.

## Solution Assessment

No. The current patch is not the best fix because it leaves the `wss://` exfiltration channel open and removes `ws://` remote-gateway support; an exact-origin allowlist or same-origin proxy policy is the narrower maintainable direction.
