## Summary

Keep open: the PR is a plausible security-hardening direction, but the current head still permits arbitrary `wss:` egress, drops current browser HTTP origins, and blocks documented `ws://` gatewayUrl workflows without maintainer product/security approval or real browser proof.

## What This Changes

This PR changes the Control UI CSP helper and tests to emit `connect-src 'self' wss:` with optional extra sources instead of the current `ws: wss:` connect policy.

## Best Possible Solution

Define an explicit Control UI `connect-src` policy that blocks arbitrary WebSocket egress while preserving OpenAI Realtime, tweakcn imports, and supported gatewayUrl workflows through exact origins, same-origin proxying, or documented deprecation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: compare current main's CSP header with the PR head and exercise a browser console/network path for external WebSocket egress; contributor live browser proof is still missing.

## Solution Assessment

Is this the best way to solve the issue? No. The current diff leaves arbitrary `wss:` egress, drops required HTTP origins, and blocks documented `ws://` gateway URLs; the safer fix needs an explicit origin/proxy/deprecation decision.
