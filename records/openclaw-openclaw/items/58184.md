## Summary

Keep this PR open, but do not merge the current patch: the syntax-error regression from the earlier review has been fixed, while the latest diff still leaves the main secure-WebSocket exfiltration class open and regresses documented plaintext remote `gatewayUrl` workflows.

## What This Changes

The PR changes the Control UI CSP helper and tests to remove the blanket `ws:` source, keep a blanket `wss:` source, add an `extraConnectSrc` option, and assert those connect-src values.

## Best Possible Solution

Define the Control UI connection policy explicitly: keep same-origin connections by default, remove blanket scheme sources, and either add narrowly validated exact gateway origins for supported remote workflows or intentionally retire those workflows with docs and tests updated together.

## Reproduction Assessment

Yes. Static inspection is enough here: the latest PR head emits `connect-src 'self' wss:`, while current main still accepts remote `gatewayUrl` values and passes them directly to `new WebSocket(...)`.

## Solution Assessment

No. The current patch is not the best way to solve the issue because it keeps the `wss://` exfiltration channel and breaks documented `ws://` remote gateway targets; an exact-origin allowlist, same-origin proxy, or explicit deprecation path would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Remove the blanket wss: source:** `src/gateway/control-ui-csp.ts:58`
  - body: The patched directive still emits `connect-src 'self' wss:`. A scheme source allows WebSocket connections to any secure host, so injected Control UI script can still exfiltrate over `wss://...`; this does not deliver the stated CSP hardening.
  - confidence: 0.94
- **[P2] Preserve documented ws:// gatewayUrl support:** `src/gateway/control-ui-csp.ts:58`
  - body: Dropping `ws:` blocks cross-origin plaintext gateway targets when the Control UI is served with this CSP, but the UI still accepts `gatewayUrl` and opens it via `new WebSocket(...)`, and the docs still show `gatewayUrl=ws://<gateway-host>:18789`. Either preserve this with exact origins or retire it deliberately with docs/tests.
  - confidence: 0.9
