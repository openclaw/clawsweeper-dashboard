## Summary

Keep open: current main still lacks a high-confidence fix for the reported saturation. Shared JSON writes clean their temp path, but cron still has a separate atomic-write path without final cleanup, Control UI still polls `node.list`, and the current `node.list` handler does not show the claimed 2-second probe path.

## What This Changes

The issue asks OpenClaw to stop persistent gateway event-loop saturation attributed to accumulating atomic-write temp files and repeated slow Control UI `node.list` polling.

## Best Possible Solution

Profile a live macOS gateway with Control UI connected and no paired nodes, then land the confirmed narrow fixes: cron temp cleanup/startup orphan cleanup and any measured node polling or caching change.

## Reproduction Assessment

Unclear. The report has concrete logs, file counts, version, and environment, and cron has a source-visible orphan-temp failure path, but I did not run a live gateway and current `node.list` does not expose the claimed 2-second probe.

## Solution Assessment

Unclear. Cron atomic cleanup is a narrow maintainable fix, but returning early from `node.list` or adding a timeout knob is not grounded in the current handler; profiling should decide whether UI polling, pairing-file reads, or another gateway hot path is responsible.
