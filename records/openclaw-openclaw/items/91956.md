## Label Justifications

- `impact:message-loss`: The issue is about duplicated and noisy Feishu streaming-card delivery obscuring the intended assistant reply.

## Summary

Keep open: current main and v2026.6.6 still have the Feishu streaming path that combines reasoning, answer text, and transient tool status into one mutable card, with snapshot merging that can duplicate diverging full-card updates. The related open Feishu streaming PRs are useful context, but none has merged or become a clean canonical replacement.

## What This Changes

The issue asks Feishu streaming replies to stop repeatedly showing reasoning text and interleaved tool execution status in the visible user reply.

## Best Possible Solution

Coordinate one Feishu-owned dispatcher/streaming-card repair that treats each card update as the latest authoritative snapshot, keeps transient tool status out of final close, suppresses duplicate reasoning/status accumulation, and lands with focused regression coverage plus redacted Feishu proof.

## Reproduction Assessment

Yes, at source level: the reporter supplied a Feishu screenshot, and current main still sends combined reasoning/status/answer snapshots through mergeStreamingText where diverging full snapshots can concatenate. I did not run a credentialed live Feishu repro.

## Solution Assessment

Yes on the owner boundary: this should be fixed in the Feishu plugin dispatcher/streaming-card state handling, not by disabling streaming globally or adding another config option. The exact branch should be chosen from the overlapping open Feishu fixes or replaced by one narrow canonical PR.
