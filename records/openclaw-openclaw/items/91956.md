## Summary

This issue remains necessary because current main still combines optional reasoning previews, answer snapshots, and transient tool progress in one Feishu card, while the linked issues cover adjacent card-loss, overwrite, and delivery-mode problems rather than this distinct repeated-reasoning and tool-detail leakage symptom. Likely related people: @vincentkoc (high confidence, reasoning-preview and dispatcher history) and @steipete (medium confidence, streaming-card and dispatcher history).

## What This Changes

Prevent Feishu streaming replies from repeating reasoning and exposing internal tool-call details while preserving the intended final answer.

## Best Possible Solution

Capture the affected turn’s normalized callback sequence, then fix the earliest canonical boundary so reasoning snapshots are permission-gated and deduplicated while Feishu displays only safe transient progress and a clean final answer.

## Reproduction Assessment

No, not yet at high confidence. The report gives a repeatable user trigger and current source contains every implicated display path, but the exact normalized callback sequence from xiaomi-tp/mimo-v2.5 has not been captured or reproduced on current main.

## Solution Assessment

Unclear at the exact code layer. The best fix is not to disable Feishu streaming or every reasoning preview; it is to correct callback classification or snapshot deduplication at its canonical owner and retain only explicitly enabled, safe progress output.
