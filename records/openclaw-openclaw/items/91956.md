## Label Justifications

- `P1`: The latest released Feishu path can make common tool-using replies unreadable by repeatedly mixing transient content into the answer card.
- `impact:message-loss`: Repeated reasoning and tool status obscure the intended final response enough that usable message content is effectively suppressed.

## Summary

Current main still intentionally merges Feishu reasoning previews, answer snapshots, and tool execution status into one mutable streaming card, and no later fix or canonical issue clearly covers the reported repeated-reasoning plus status-interleaving combination. The report is credible, but the desired rule for transient tool status needs owner judgment because current tests and implementation treat that status as deliberate progress UI rather than accidental raw tool leakage.

## What This Changes

Prevent Feishu streaming cards from repeatedly showing reasoning snapshots and interleaving tool execution status so the user receives a readable final answer.

## Best Possible Solution

Unify Feishu with the canonical channel progress composition rules: replace cumulative reasoning snapshots instead of appending them, keep any approved tool status transient and visually separate, and atomically replace progress with the clean final answer.

## Reproduction Assessment

Partly. Current source provides a high-confidence path for reasoning, answer snapshots, and tool status to share one Feishu card, but no attached event trace or live run establishes exactly which callback repeats the affected provider's reasoning.

## Solution Assessment

Not fully yet. Snapshot replacement and clean final takeover are the narrow maintainable fix, but maintainers must first confirm the intended Feishu policy for deliberately rendered tool-progress lines rather than treating all such lines as accidental leakage.
