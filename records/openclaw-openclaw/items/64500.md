## Summary

Keep open. Current main still allows the reported changing-output ping-pong loop class: the ping-pong detector only escalates to critical when both sides have stable no-progress hashes, and the before-tool hook has no persisted pair-level block for the alternate tool. The related session-global breaker PR is adjacent useful work, but it does not close this pair/enforcement gap.

## What This Changes

The issue asks for tool-loop/circuit-breaker enforcement to stop sustained A/B ping-pong tool loops by blocking the offending pair instead of only the current tool/signature.

## Best Possible Solution

Add run/session-scoped pair-aware enforcement to the existing loop detector and before-tool hook so a sustained offending A/B signature can block both sides while preserving conservative progress safeguards for legitimate alternating work.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: seed alternating A/B calls with changing per-side result hashes and current main reaches the ping-pong warning branch, not the critical branch; the hook also only vetoes the current attempted call when a critical detector fires.

## Solution Assessment

Yes. Pair-aware enforcement using the existing run/session history and thresholds is the narrowest maintainable direction; the active session-global PR should be coordinated with, not treated as a complete fix for changing-output ping-pong loops.
