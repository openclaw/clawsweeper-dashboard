## Summary

Keep open. Current main still has the reported enforcement gap: sustained ping-pong detection can stay warning-only when per-side outcomes change, and the before-tool hook only vetoes the current attempted tool without persisting a pair-level block for the alternate tool.

## What This Changes

The issue requests pair-aware circuit-breaker enforcement so sustained A/B tool-call loops stop both sides instead of only the tool that trips the threshold.

## Best Possible Solution

Add run/session-scoped pair enforcement to the existing loop detector and before-tool hook so a sustained offending A/B signature can block both sides without adding a new config knob, while preserving conservative progress safeguards for non-looping workflows.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: seed alternating A/B calls with changing per-side result hashes and the current detector reaches warning but cannot reach the critical ping-pong branch. Separately, when the global breaker fires for one side, the hook returns only a current-call veto and has no pair-block state for the alternate tool.

## Solution Assessment

Yes. Pair-aware enforcement using existing run-scoped history and thresholds is the narrow maintainable fix; adding another config knob or dropping all progress checks globally would be a broader product change.
