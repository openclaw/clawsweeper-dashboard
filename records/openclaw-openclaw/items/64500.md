## Summary

Keep open. Current main still allows the reported loop class: changing-output ping-pong stays warning-only, and the before-tool hook has no persisted pair-level block for the alternate tool; the related session-global PR is useful but does not close this specific pair/enforcement gap.

## What This Changes

The issue asks for loop/circuit-breaker enforcement to stop sustained alternating tool-pair loops instead of only blocking the currently repeated tool signature.

## Best Possible Solution

Add run/session-scoped pair-aware enforcement to the existing loop detector and before-tool hook so a sustained offending A/B signature can block both sides, while preserving conservative progress safeguards for legitimate alternating work.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: seed alternating A/B calls with changing per-side result hashes and current main reaches the ping-pong warning branch but cannot reach the critical branch. Separately, the hook only vetoes the current attempted call and has no pair-block state for the alternate tool.

## Solution Assessment

Yes, pair-aware enforcement using existing run/session history and thresholds is the narrowest maintainable direction. The active session-global PR should be treated as adjacent work, not a complete fix for changing-output ping-pong loops.
