## Summary

Keep open. Current main still has the reported pair-aware enforcement gap: ping-pong detection only becomes critical when both sides have stable result hashes, and the before-tool-call hook blocks only the current invocation instead of persisting a block for both tools in the alternating pair.

## What This Changes

The issue requests pair-aware circuit breaker enforcement for sustained alternating tool calls so ping-pong loops block both tools rather than only the last triggering tool.

## Best Possible Solution

Implement pair-aware loop enforcement in the agent tool-loop detector and before-tool-call hook: track alternating signatures as one scoped pair, trip a conservative pair breaker for sustained A/B loops, and block both sides within the active run/session while preserving current defaults and single-tool no-progress behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to record alternating A/B tool calls with changing per-side result hashes; current tests show that this reaches ping-pong warning but not critical blocking, and the hook has no pair-level block state.

## Solution Assessment

Yes. The narrowest maintainable solution is a run/session-scoped pair-aware breaker that reuses the existing loopDetection thresholds and diagnostics, rather than blunt per-agent tool denial or unrelated streaming-loop changes.
