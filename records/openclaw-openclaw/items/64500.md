## Summary

Keep open. Current main still has the reported pair-aware enforcement gap: sustained ping-pong alternation is only blocked when both sides produce stable result hashes, and the global circuit breaker still blocks the current tool/signature rather than the alternating pair.

## What This Changes

The issue asks for tool-loop circuit breaking to be pair-aware so alternating tool pairs such as memory_search and memory_get cannot continue by switching to the other tool after one side is blocked.

## Best Possible Solution

Implement pair-aware loop enforcement in the agent tool-loop detector and before-tool-call hook: track the alternating pair/signatures, trip a conservative pair-scoped breaker for sustained A/B loops, and block both sides of the offending pair within the active run/session while preserving current defaults and existing unknown-tool behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to record alternating A/B tool calls with changing per-side result hashes: current tests already show that this reaches ping-pong warning but not critical blocking, and the hook has no state to block the alternate paired tool.

## Solution Assessment

Yes, the requested pair-aware guard is the narrow maintainable direction. The safest implementation should reuse existing loop-detection thresholds and run scoping, add pair metadata/state only where needed, and avoid unrelated changes such as enabling loop detection by default or changing generic_repeat semantics.
