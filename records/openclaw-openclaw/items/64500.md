## Summary

Keep open. Current main still has the reported enforcement gap: ping-pong detection only becomes critical when both sides have stable result hashes, and the before-tool hook only blocks the current attempted call rather than remembering and blocking both signatures in the alternating pair.

## What This Changes

The issue requests pair-aware enforcement for tool ping-pong loops so the circuit breaker blocks both sides of an alternating tool pair instead of only the current tool.

## Best Possible Solution

Add a conservative pair-scoped enforcement path for sustained A/B tool alternation: track canonical pair signatures within the active run/session, trip only after the configured thresholds, and block both sides of the offending pair without changing unrelated generic-repeat or streaming-loop behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an alternating A/B tool history with changing per-side result hashes: current main reaches ping-pong warning-only behavior, while the hook has no pair-block state to stop the alternate tool after one side trips.

## Solution Assessment

Yes. Pair-aware loop enforcement is the narrow maintainable fix; the safer implementation is to reuse existing run scoping and thresholds rather than enabling loop detection by default or making all changing-output ping-pong critical immediately.
