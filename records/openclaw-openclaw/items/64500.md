## Summary

Keep open. Current main still has the reported enforcement gap: sustained ping-pong alternation with changing per-side result hashes stays warning-only, and critical hook handling vetoes only the current tool call without remembering the paired signature.

## What This Changes

The issue asks for `tools.loopDetection` to enforce ping-pong circuit breaking at the tool-pair/signature-pair level so sustained A/B loops cannot continue by alternating tools.

## Best Possible Solution

Add pair-aware enforcement to the existing loop detector/hook path so a sustained offending A/B signature pair trips within the active run and blocks both sides without adding a new config knob.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: record alternating A/B calls with changing per-side result hashes and current tests/source show the result remains warning-only at the critical threshold. The hook also has no pair-block state, so a current-call veto cannot automatically block the alternate tool.

## Solution Assessment

Yes. Pair-aware enforcement using the existing thresholds and run-scoped history is the narrowest maintainable direction; dropping all progress checks globally would be riskier than constraining the offending pair.
