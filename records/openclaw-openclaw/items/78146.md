## Summary

Keep this PR open: it adds useful redacted lineage instrumentation, but the proposed full-context logging can mislabel an available prior response id as a requested chain id, and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds redacted OpenAI WebSocket request and completion lineage debug logging plus focused planner tests.

## Best Possible Solution

Keep the instrumentation narrow, make the lineage fields reflect only what was actually sent on each request, and require redacted runtime proof from the affected WebSocket path before merge.

## Reproduction Assessment

Yes for the PR-introduced review finding: source inspection shows the full-context branch strips `previous_response_id` from the payload while the PR still records the same id as debug lineage. No live reproduction of the original stale-replay bug is included in the PR evidence.

## Solution Assessment

No; the diagnostic direction is reasonable, but the field semantics should distinguish an available prior id from a `previous_response_id` actually sent on the request. The safer fix is to omit `previousResponseId` for full-context sends or rename it as a non-requested candidate field.
