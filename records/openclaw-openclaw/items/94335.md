## Summary

Keep open: the linked bug is real and not fixed on current main, but this PR is not merge-ready because it converts any non-empty captured display output into a successful subagent completion instead of proving an explicit current-run terminal success.

## What This Changes

The PR adds a lost-context sweeper branch that completes a stale subagent run as `ok` when `captureSubagentCompletionReply()` returns non-empty text, plus three focused registry tests.

## Best Possible Solution

Constrain lost-context recovery to an explicit current-run terminal-success boundary, preserve the lost-context error for non-terminal/no-output/read-failure cases, and require redacted real runtime proof before merge.

## Reproduction Assessment

Yes, source-level: current main can fall from running session-store reconciliation to the lost-context fallback while the announce path can still capture child output. I did not run a live Agent Teams repro in this read-only review.

## Solution Assessment

No. The bugfix direction is valid, but this patch uses display-oriented captured output as the success predicate; the safer repair is an explicit current-run terminal success signal such as private-transcript or persisted session status.
