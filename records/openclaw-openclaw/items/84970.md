## Label Justifications

- `P1`: A beta-blocking session pollution bug can repeatedly grow hidden context and disrupt long-lived agent runs.
- `impact:data-loss`: The report is about synthetic operational prompts corrupting persisted user/session history.
- `impact:session-state`: The affected surface is canonical session transcript and context-engine/LCM state drift.

## Summary

Keep open: this is a protected beta-blocker and the current source still routes the boot-md synthetic prompt through the canonical agent session key before context-engine/LCM assembly, so the reported session pollution remains credible.

## What This Changes

The issue requests that bundled boot-md startup checks stop persisting synthetic BOOT.md prompts into the canonical main session transcript and LCM context.

## Best Possible Solution

Run boot-md in a truly isolated or non-persistent boot context, or mark boot-generated messages internal so transcript assembly and context-engine/LCM ingestion exclude them, with a restart regression and a doctor cleanup warning for polluted sessions.

## Reproduction Assessment

Yes. The provided v2026.5.19 logs give a concrete installed reproduction, and current source shows boot-md still sends the synthetic prompt under the canonical session key used by context-engine lifecycle calls.

## Solution Assessment

No. The current implementation generates a boot session id and restores the main mapping, but the context-engine path still receives the canonical session key, so the safer fix is to isolate or mark boot turns internal before transcript/LCM persistence.
