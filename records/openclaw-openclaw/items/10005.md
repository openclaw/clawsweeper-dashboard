## Summary

Keep open: current main and the latest release still route both chat `/status` and the `session_status` tool through the shared rendered status-card text path, with no documented suppression or JSON-only visibility setting. The request is useful and source-reproducible, but it adds a new config/output contract with privacy implications, so it needs maintainer product/security direction before implementation.

## What This Changes

The issue requests a config option to hide the rendered session status card for `/status` and `session_status`, while preserving current behavior by default.

## Best Possible Solution

Add one documented shared status visibility contract for both chat `/status` and `session_status`, preserving today’s rendered card by default and covering privacy-sensitive fields with schema, docs, and focused tests.

## Reproduction Assessment

Yes for source-level verification, not as a bug reproduction: current main routes both `/status` and `session_status` through rendered status-card text with no suppression config. I did not run the app because the read-only source, docs, tests, release snapshot, and related issue context establish the path directly.

## Solution Assessment

Unclear until maintainers choose the output contract. The maintainable direction is a central shared visibility option covering both status paths, not per-channel special cases, with the current rendered card kept as the default.
