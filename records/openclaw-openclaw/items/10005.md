## Summary

Keep open: current main still renders both chat `/status` and `session_status` through the shared status-card text path with no documented suppression, JSON-only, or visibility setting. The remaining work is a product/security decision about the public output contract, not a narrow bug fix.

## What This Changes

The issue requests a config option to hide the rendered `/status` and `session_status` status card while preserving current behavior by default.

## Best Possible Solution

Add one documented shared status visibility contract for `/status` and `session_status`, preserving today’s card by default and covering usage/privacy behavior with schema, docs, and focused tests.

## Reproduction Assessment

Yes for source-level verification, not as a bug reproduction: current main routes both `/status` and `session_status` through rendered status-card text with no suppression config. I did not run the app because the read-only source, docs, tests, and related issue context establish the path directly.

## Solution Assessment

Unclear until maintainers choose the output contract. The narrow maintainable direction is a central shared status visibility option rather than per-channel special cases, with the current rendered card kept as the default.
