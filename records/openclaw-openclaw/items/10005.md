## Summary

Keep this issue open. Current main still renders both chat `/status` and `session_status` through the shared status-card path with no `session.statusCard`, `agents.defaults.statusCard`, hide flag, or JSON-only chat/tool alternative, and the prior issue discussion identifies this as the remaining canonical tracker after the narrower privacy request was closed as superseded.

## What This Changes

The issue requests a documented configuration option to suppress the rendered session status card for chat `/status` and the `session_status` tool while preserving current behavior by default.

## Best Possible Solution

Add an explicit, documented status-card visibility control for chat `/status` and `session_status` in the shared status builder path, preserve current rendered output by default, update config types/schema/help/docs, and add focused tests for both command and tool behavior while keeping CLI `openclaw status --json` separate.

## Reproduction Assessment

Not applicable as a runtime bug report, but there is a high-confidence verification path: read the current `buildStatusReply`, `buildStatusText`, `buildStatusMessage`, and `session_status` code and confirm they always return rendered status text without a config-controlled suppression branch.

## Solution Assessment

Unclear until a maintainer chooses the public config key and hidden-output semantics. The narrow maintainable direction is to thread one explicit setting through the shared status text/card path and cover both chat-command and agent-tool callers, rather than treating the existing CLI JSON status surface as the fix.
