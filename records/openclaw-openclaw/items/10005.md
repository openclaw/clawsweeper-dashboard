## Summary

Keep open: current main still renders `/status` and `session_status` through the shared status-card text path with no documented visibility, JSON-only, or suppression setting, and the related privacy request was closed into this item as the canonical tracker. The remaining decision is product/security policy for the output contract, not a narrow bug fix.

## What This Changes

The issue requests a configuration option to hide or suppress the rendered session status card for chat `/status` and the `session_status` tool.

## Best Possible Solution

Add one documented shared status visibility contract for `/status` and `session_status`, preserving today’s card by default and covering the provider usage/privacy case with schema, help/docs, and focused regression tests.

## Reproduction Assessment

Yes for source-level verification, not as a bug reproduction: current main routes both `/status` and `session_status` through rendered status-card text with no suppression config. I did not run the app because the read-only source, docs, tests, and related issue context establish the path directly.

## Solution Assessment

Unclear until maintainers choose the output contract. The narrow maintainable direction is a central shared status visibility option rather than per-channel special cases, with the current rendered card kept as the default.
