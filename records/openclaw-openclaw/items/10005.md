## Summary

Keep this open. Current main still unconditionally renders status-card text for chat `/status` and `session_status`, and the requested config/output behavior needs a maintainer product choice before implementation.

## What This Changes

Add a config-controlled way to hide or suppress the rendered `/status` and `session_status` status card while preserving current behavior by default.

## Best Possible Solution

Add one documented status-card visibility setting in the shared status builder path, keep rendered cards enabled by default, and cover both chat-command and agent-tool callers once maintainers choose the public semantics.

## Reproduction Assessment

Not applicable as a runtime bug report; this is a feature request. Source inspection gives a high-confidence verification path: current `buildStatusReply`, `buildStatusText`, `buildStatusMessage`, and `session_status` always produce rendered status text without a config-controlled suppression branch.

## Solution Assessment

Unclear until a maintainer chooses the public key and hidden-output semantics. The narrow maintainable direction is to thread one explicit setting through the shared status-card builder rather than treating the existing CLI `status --json` mode as the fix.
