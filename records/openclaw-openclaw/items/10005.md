## Label Justifications

- `P3`: This is a useful but low-urgency optional output and privacy enhancement rather than a broken core workflow.
- `impact:security`: The status card can expose provider usage or billing information to users of secondary agents in multi-user deployments.

## Summary

Current main still renders the shared status card for both authorized chat `/status` requests and `session_status`, with no supported hide, JSON-only, or section-redaction setting. This remains the canonical tracker, but the whole-card, per-agent privacy, and tool-return semantics require a maintainer product decision before implementation.

## What This Changes

Add a configuration control that hides or replaces the rendered session status card for chat `/status` and the `session_status` tool.

## Best Possible Solution

Define one structured status model with an explicit presentation and field-redaction policy, preserve the current rendered card by default, and apply the chosen scope consistently to chat and tool consumers with schema, help, docs, and focused tests.

## Reproduction Assessment

Yes at source level: current main always renders the shared status text for the chat command and tool, and the searched configuration surface contains no supported suppression or redaction control. This is a feature-gap reproduction rather than a failing existing contract.

## Solution Assessment

No, the proposed boolean alone is not yet the best solution because it leaves tool output semantics, per-agent scope, and sensitive-field redaction unresolved. A canonical structured status representation with explicit presentation and privacy policy is safer and more extensible.
