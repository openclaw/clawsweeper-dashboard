## Summary

Current main still unconditionally renders the shared status card for both chat `/status` and `session_status`; the request remains necessary, but its global versus per-agent scope and hidden-output behavior require a maintainer product and privacy decision.

## What This Changes

Requests a configurable way to suppress or change the rendered session status card for chat `/status` and the `session_status` tool.

## Best Possible Solution

Define one default-preserving visibility contract shared by `/status` and `session_status`, explicitly deciding global versus per-agent scope and whether hidden mode returns reduced text, structured details, or no visible reply; then align schema, help, docs, and focused tests across both entry points.

## Reproduction Assessment

Yes at source level: both chat `/status` and `session_status` unconditionally produce rendered text through the shared status builder, and existing tests assert the card output. No runtime execution was necessary to establish that the requested control is absent.

## Solution Assessment

Unclear. A boolean whole-card switch is plausible, but the consolidated privacy use case may require per-agent or field-level policy, so maintainers should choose the public contract before implementation.
