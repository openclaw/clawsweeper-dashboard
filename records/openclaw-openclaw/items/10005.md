The issue requests a config-controlled way to hide or replace the rendered `/status` and `session_status` card while preserving current behavior by default.

## Best Possible Solution

Add one shared, documented status visibility contract for chat `/status` and `session_status`, preserving the rendered card by default and covering config schema, help/docs, and focused tests after maintainers choose the privacy/output shape.

## Reproduction Assessment

Yes, as a source-level feature-gap check rather than a broken existing behavior reproduction. Current main and `v2026.6.5` both route `/status` and `session_status` through rendered status-card text with no suppression config or documented equivalent.

## Solution Assessment

Unclear until maintainers choose the output contract. The maintainable direction is a central shared visibility option covering both status paths, not per-channel special cases, with the current rendered card kept as the default.
