## Summary

Keep open: current main and the latest release still render chat `/status` and `session_status` through the shared status-card text path with no supported hide, section-filter, or JSON-only visibility setting. The request is valid, but it needs maintainer product/security direction before adding another compatibility-sensitive config surface.

## What This Changes

Issue requests a configuration option to hide or suppress the rendered session status card for chat `/status` and the `session_status` tool.

## Best Possible Solution

Define one shared, privacy-aware status presentation contract for chat `/status` and `session_status`, keeping the full card as the default while supporting the maintainer-approved reduced or section-filtered mode.

## Reproduction Assessment

Yes, at source level: both current-main entry points still use the shared rendered-text status path, and the config schemas expose no status-card visibility or section-suppression setting. I did not run a live chat/tool invocation because the source and existing tests show the current contract clearly.

## Solution Assessment

No, the proposed boolean is a plausible starting point but not yet the best complete solution. Maintainers still need to choose global versus per-agent scope and define whether hidden output means reduced text, structured details only, section filtering, or no visible content.
