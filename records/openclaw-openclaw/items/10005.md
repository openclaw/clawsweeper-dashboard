## Summary

Keep open: current main and latest release still render chat `/status` and `session_status` through the shared status-card path with no documented hide, JSON-only, or section-visibility control, and related reports point back to this issue as the canonical tracker.

## What This Changes

Request to add a configuration option that suppresses the rendered chat/session status card while preserving current behavior by default.

## Best Possible Solution

Add one shared, documented status visibility contract for both chat `/status` and `session_status`, preserving the rendered card by default and covering schema, help/docs, and regression tests for the chosen privacy/output behavior.

## Reproduction Assessment

Yes for source-level verification of the current absence, not as a bug reproduction. Current main and v2026.5.20 both route `/status` and `session_status` through rendered status-card text with no suppression config or documented equivalent.

## Solution Assessment

Unclear until maintainers choose the output contract. The maintainable direction is a central shared visibility option covering both status paths, not per-channel special cases, with the current rendered card kept as the default.
