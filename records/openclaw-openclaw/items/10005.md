## Summary

Keep open: current main and the latest release still render `/status` and `session_status` through the shared status-card text path with no documented suppression or JSON-only visibility setting. This is a useful feature request with a privacy angle, but it needs maintainer product/security direction before implementation.

## What This Changes

Add a configuration option to hide the rendered chat `/status` and `session_status` status card while preserving current rendering by default.

## Best Possible Solution

Add one documented shared status visibility contract for both chat `/status` and `session_status`, preserving today’s card by default and covering privacy-sensitive usage fields with schema, docs, and focused tests.

## Reproduction Assessment

Yes for source-level verification, not as a bug reproduction: current main routes both `/status` and `session_status` through rendered status-card text with no suppression config. I did not run the app because the read-only source, docs, release snapshot, and related issue context establish the path directly.

## Solution Assessment

Unclear until maintainers choose the output contract. The maintainable direction is a central shared visibility option covering both status paths, not per-channel special cases, with the current rendered card kept as the default.
