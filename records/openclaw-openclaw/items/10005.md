## Summary

Keep open: current main still renders `/status` and `session_status` through the shared text status-card path with no supported visibility, section-filtering, JSON-only, or hide-card setting. The narrower per-agent usage/billing privacy request is closed as superseded by this issue, so this remains the canonical tracker and needs maintainer product/security judgment.

## What This Changes

The issue requests a configuration option to hide or reduce the rendered `/status` and `session_status` status card.

## Best Possible Solution

Define one canonical, privacy-aware status presentation contract shared by chat and tool callers, preserving the full card by default while supporting an explicitly chosen reduced, structured, or section-filtered mode with schema, help, docs, and tests.

## Reproduction Assessment

Yes at source level: both current-main entry points use the shared rendered-text path, and the config schemas do not expose a status-card visibility or section-suppression setting. I did not run a live chat/tool invocation because source and tests already show the contract clearly.

## Solution Assessment

No, the proposed boolean is not yet the best complete solution because maintainers still need to choose global versus per-agent scope and define whether hidden output is empty, reduced text, section-filtered, or structured data. The maintainable path is a single shared status presentation contract used by both `/status` and `session_status`.
