## Summary

Keep open: current main and the latest release still route chat `/status` and the `session_status` tool through the rendered status-card path with no supported hide, section-filter, or JSON-only visibility setting. The related per-agent usage/billing suppression request was closed as superseded by this issue, so this remains the canonical tracker pending maintainer product/security direction.

## What This Changes

Request adds a config-controlled way to hide or reduce the rendered status card returned by chat `/status` and the `session_status` tool.

## Best Possible Solution

Define one shared, privacy-aware status presentation contract for both chat and tool callers, keeping the full card as the default while supporting the maintainer-approved reduced or section-filtered mode.

## Reproduction Assessment

Yes, at source level: both current-main entry points use the shared rendered-text status path, and the config schemas expose no status-card visibility or section-suppression setting. I did not run a live chat/tool invocation because the source and existing tests already show the current contract clearly.

## Solution Assessment

No, the proposed boolean is a plausible starting point but not yet the best complete solution. Maintainers still need to choose global versus per-agent scope and define whether hidden output means reduced text, structured details only, section filtering, or no visible content.
