## Summary

Current main and v2026.6.5 still expose the full shared status card with no supported hide, JSON-only, or section-visibility control, and no open replacement PR owns the work. This remains the canonical tracker, but implementation requires a maintainer product/security decision because whole-card suppression, sensitive-field suppression, and tool output semantics are different contracts.

## What This Changes

Add a configurable way to suppress rendered status-card output from chat `/status` and the `session_status` tool.

## Best Possible Solution

Define one explicit status-disclosure policy at the shared status boundary, preserving current output by default while separately specifying chat reply behavior, tool result structure, and whether privacy requires whole-card or sensitive-section suppression.

## Reproduction Assessment

Yes at source level: current main and v2026.6.5 unconditionally build and return the rendered card, while existing session visibility settings only control which session may be queried. This is a demonstrated feature gap, not a regression against an existing hide contract.

## Solution Assessment

No, the proposed boolean plus JSON-only behavior is not yet the best-specified solution. Maintainers should first decide whole-card versus sensitive-section suppression and define a useful, privacy-safe `session_status` result when rendered text is disabled.
