Keep open: current main still has no documented status-card visibility setting for chat `/status` or `session_status`, and the related reports point back to this issue as the canonical tracker.

## What This Changes

The issue requests a config option to suppress the rendered session status card for chat `/status` and `session_status` while preserving current rendering by default.

## Best Possible Solution

Pick a documented status-card visibility contract, then implement it once through the shared status builder/tool path with schema, help/docs, and focused chat-command plus `session_status` coverage.

## Reproduction Assessment

This is not a bug reproduction, but there is a high-confidence source verification path: current main routes both `/status` and `session_status` through rendered status-card text with no suppression config. I did not run the app because this review was constrained to read-only inspection.

## Solution Assessment

Unclear until maintainers choose the product contract. The maintainable path is a central shared status visibility option rather than per-channel special casing, with the default preserving today’s card output.
