## Summary

Keep open: current main and the latest shipped release still render chat `/status` and `session_status` through the shared status-card text path with no documented hide, JSON-only, or section-visibility setting. This remains the canonical tracker after the related privacy request at https://github.com/openclaw/openclaw/issues/69445 was closed as superseded, but the fix needs maintainer product/security direction because it creates a public config and output contract.

## What This Changes

Request adding a configuration option to suppress the rendered `/status` and `session_status` status card, leaving structured status data available.

## Best Possible Solution

Add one shared, documented status visibility contract for both chat `/status` and `session_status`, preserving today’s rendered card by default and covering privacy-sensitive fields with schema, docs, and focused tests.

## Reproduction Assessment

Yes for source-level verification, not as a bug reproduction: current `main` routes both `/status` and `session_status` through rendered status-card text with no suppression config. I did not run the app because the read-only source, docs, tests, latest-release snapshot, and related issue context establish the path directly.

## Solution Assessment

Unclear until maintainers choose the output contract. The maintainable direction is a central shared visibility option covering both status paths, not per-channel special cases, with the current rendered card kept as the default.
