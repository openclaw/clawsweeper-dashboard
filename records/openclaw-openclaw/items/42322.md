## Summary

Keep open: current main still has no Feishu `replyToMode`, and this PR is a plausible parity feature, but the branch is not merge-ready because first-mode streaming can still reply twice and the new config is not exposed through public discovery/docs surfaces.

## What This Changes

The PR adds a Feishu `replyToMode` config option and threads it from schema/config resolution into Feishu reply dispatch for `all`, `first`, and `off` reply behavior.

## Best Possible Solution

Land a rebased Feishu-owned parity implementation that applies one reply-target policy across streaming cards, text/card chunks, media, and fallbacks, then exposes the setting through manifest metadata, docs, focused tests, and maintainer-owned changelog handling.

## Reproduction Assessment

Yes. Source inspection of PR head shows `replyToMode: "first"` consumes the reply target for later sends, but `streaming.start()` still receives the original target, so a streamed card followed by media can attach twice.

## Solution Assessment

No, not as-is. The feature direction is narrow and consistent with other channels, but the implementation should centralize reply-target consumption and publish the new setting through supported config metadata/docs before merge.
