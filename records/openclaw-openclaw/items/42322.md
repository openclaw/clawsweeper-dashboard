## Summary

Keep this PR open. Current main still lacks the requested Feishu `replyToMode` surface, so this is not an implemented-on-main close, but the branch is not merge-ready because `first` mode can still reply twice in the streaming-card path and the public metadata/docs surface is not aligned.

## What This Changes

The PR adds a Feishu `replyToMode` config option and passes it from Feishu message handling into the reply dispatcher for `off`, `first`, and `all` reply-target behavior.

## Best Possible Solution

Land a Feishu-owned current-main implementation only if maintainers want this parity feature: resolve `replyToMode` with group over merged account/top-level precedence, apply a single reply-target consumption policy across streaming-card creation, text/card chunks, media sends, and typing behavior, then expose the setting in manifest/docs/changelog with focused regression tests for `off`, `first`, `all`, precedence, and streaming-plus-media delivery.

## Reproduction Assessment

Yes. A focused dispatcher test can reproduce the remaining PR bug by using `replyToMode: "first"`, starting a streaming card, then sending media; the PR head passes the reply target to both `streaming.start()` and the first consumed media send.

## Solution Assessment

No, not as-is. The requested direction is plausible, but the implementation should centralize reply-target consumption and update the manifest/docs surface instead of adding a schema-only option with a partial dispatcher policy.
