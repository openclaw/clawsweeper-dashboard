## Summary

Keep open: current main still lacks Feishu `replyToMode`, but this PR head still has a source-reproducible `first`-mode streaming bug and incomplete public config surfaces.

## What This Changes

The PR adds Feishu `replyToMode` config, resolves it during inbound handling, and passes it into the Feishu reply dispatcher for `all`, `first`, and `off` reply behavior.

## Best Possible Solution

Land a rebased Feishu-owned `replyToMode` implementation that applies one reply-target policy across streaming cards, text/card chunks, media, and fallbacks, with manifest/docs/changelog and focused regression tests.

## Reproduction Assessment

Yes. Source inspection of PR head shows `streaming.start()` receives the original reply target while later sends use `consumeReplyToMessageId()`, so `replyToMode: "first"` can still attach a streamed card and subsequent media as native replies.

## Solution Assessment

No, not as-is. The direction is narrow and consistent with other channels, but the implementation should consume the reply target through one dispatcher helper and publish the new setting through supported metadata/docs/changelog surfaces.
