## Summary

Keep open. Current main still lacks the Feishu `replyToMode` config, and the PR is useful source work, but the patch is not merge-ready because `first` mode can still attach more than one native reply and the public config surface is not fully exposed.

## What This Changes

The PR adds Feishu `replyToMode` schema/config resolution and passes the selected mode into Feishu reply dispatch for `off`, `first`, and `all` native reply-target behavior.

## Best Possible Solution

Land a rebased Feishu-owned parity implementation that applies one reply-target policy across streaming cards, text/card chunks, media and fallbacks, then updates manifest metadata, docs, changelog, and focused regression tests.

## Reproduction Assessment

Yes. Source inspection of PR head shows `replyToMode: "first"` consumes the reply target for later sends, but `streaming.start()` still receives the original target first, so a streamed card followed by media can attach twice.

## Solution Assessment

No, not as-is. The feature direction is plausible Feishu parity, but the implementation should centralize reply-target consumption and publish the new config through the supported metadata/docs/changelog surfaces before merge.
