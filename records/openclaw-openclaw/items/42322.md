## Summary

Keep this PR open: current main still lacks an explicit Feishu `replyToMode`, but the branch is not safe to merge because `first` mode misses the streaming-card path and the contributor has not provided real Feishu/Lark behavior proof.

## What This Changes

The PR adds a Feishu `replyToMode` config and threads it into Feishu bot dispatch so replies can attach to all, first, or no outbound messages.

## Best Possible Solution

Decide whether Feishu should expose `replyToMode` in addition to `replyInThread`; if yes, rebase and wire one reply-target policy through schema, manifest/docs, tests, streaming, media, fallback, card, and text paths.

## Reproduction Assessment

Yes, for the review findings: source inspection of PR head shows `streaming.start()` still uses the raw reply target while later sends consume it, and current main has no Feishu `replyToMode` schema or dispatcher support. I did not live-run a Feishu/Lark workspace.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the implementation should first reconcile the maintainer direction from related Feishu reply-threading reviews and then apply one policy across all outbound send paths and public config surfaces.
