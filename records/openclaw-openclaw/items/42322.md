## Summary

Keep open. The feature is still absent from current main, and the PR is not merge-ready because `replyToMode="first"` misses the streaming-card path, the new config key is not exposed through public Feishu metadata/docs, and the contributor has not provided real Feishu/Lark behavior proof.

## What This Changes

The branch adds a Feishu `replyToMode` config option and threads it through the Feishu bot and reply dispatcher for `all`, `first`, and `off` reply behavior.

## Best Possible Solution

Implement one Feishu reply-target policy across schema, manifest/docs, tests, streaming, media, card, and text paths, or keep `replyInThread` as the only supported Feishu reply-control setting.

## Reproduction Assessment

Yes, for the review findings: PR-head source shows `streaming.start` uses the raw reply target while later sends consume it, so `first` mode can attach more than one native reply when streaming and media both send. I did not live-run a Feishu/Lark workspace.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the implementation needs one consistent reply-target policy across all outbound send paths and public config surfaces before merge.
