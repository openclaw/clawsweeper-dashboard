## Summary

Keep open: current main still lacks Feishu `replyToMode`, so the PR is not obsolete, but the patch is not merge-ready because `first` mode can still use the reply target twice with streaming plus media and the public config surface is not documented or exposed in plugin metadata.

## What This Changes

The PR adds Feishu `replyToMode` (`all`, `first`, `off`) to the config schema and threads it through bot dispatch and the reply dispatcher to control native reply targets.

## Best Possible Solution

Rebase and implement one Feishu reply-target policy across schema, manifest/docs, tests, streaming, media, card, and text paths, then provide redacted real Feishu/Lark runtime proof.

## Reproduction Assessment

Yes, at source level for the PR findings: PR head passes raw `replyToMessageId` to `streaming.start()` while the new consume helper only clears later send paths. I did not live-run a Feishu/Lark workspace.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the patch needs a single reply-target policy across all outbound paths plus public config surfaces before merge.
