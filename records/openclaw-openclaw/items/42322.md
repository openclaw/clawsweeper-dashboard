## What This Changes

The PR adds optional Feishu `replyToMode` config values and threads that value through Feishu bot dispatch and reply-target selection.

## Best Possible Solution

Rebase onto current main, centralize one reply-target consumer across streaming, card, text, media, and fallback sends, expose the config in Feishu manifest/docs, and attach redacted Feishu/Lark proof for the supported modes.

## Reproduction Assessment

Yes for the PR defect at source level: the PR head passes the raw reply target into `startStreaming()` while first-mode state is only updated by later `consumeReplyToMessageId()` calls. I did not live-run a Feishu/Lark workspace.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the implementation needs centralized reply-target consumption and public discovery/docs before merge.
