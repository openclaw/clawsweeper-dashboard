## Summary

Keep open. Current main still does not expose Feishu `replyToMode`, so this PR is not obsolete, but it is not merge-ready: `first` mode misses the streaming-card startup path, public config surfaces are incomplete, the branch is dirty against main, and contributor real behavior proof is still missing.

## What This Changes

The PR adds a Feishu `replyToMode` config key and threads it into the Feishu bot and reply dispatcher for `all`, `first`, and `off` reply-to behavior.

## Best Possible Solution

Rebase onto current main and implement one Feishu reply-target policy across schema, manifest/docs metadata, tests, streaming, media, card, and text paths, with redacted live Feishu/Lark proof.

## Reproduction Assessment

Yes for the PR review findings: PR-head source shows `streaming.start` uses raw `replyToMessageId` while later sends consume it, and current main lacks Feishu `replyToMode`. I did not live-run a Feishu/Lark workspace.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the implementation should apply one reply-target policy across all outbound send paths and public config surfaces before merge.

## Review Findings

  - confidence: 0.78
