## Summary

Keep open: current main still has no Feishu `replyToMode`, and this PR is not merge-ready because first-mode streaming bypasses the new consume helper, public config surfaces are incomplete, and no real Feishu/Lark behavior proof is supplied.

## What This Changes

Adds Feishu `replyToMode` config schema entries and dispatcher plumbing to control native reply targeting with `all`, `first`, and `off` modes.

## Best Possible Solution

Decide the Feishu product contract first; if `replyToMode` should exist, rebase and implement one policy across schema, manifest/docs, tests, streaming, media, card, and text paths with redacted runtime proof.

## Reproduction Assessment

Yes, at source level for the PR findings: PR head starts streaming with raw `replyToMessageId` before the new consume helper can mark the first reply used, and current main has no Feishu `replyToMode` schema or dispatcher support. I did not live-run a Feishu/Lark workspace.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the implementation needs a rebase, one reply-target policy across all outbound paths, public config surface updates, and maintainer agreement on adding this config next to `replyInThread`.
