## Summary

Keep this PR open: current main still does not expose Feishu `replyToMode`, and the branch is useful, but it is not mergeable as-is because first-mode streaming can still create more than one native reply target, the public config surfaces are incomplete, and external Feishu/Lark runtime proof is missing.

## What This Changes

The PR adds a Feishu `replyToMode` config option and threads all/first/off reply-target policy through Feishu bot dispatch and reply delivery.

## Best Possible Solution

Land a rebased Feishu reply-target policy that covers streaming, text, card, and media sends consistently, updates schema/manifest/docs/tests, and includes redacted Feishu/Lark runtime proof.

## Reproduction Assessment

Yes for the PR findings at source level: at PR head the first-mode helper is used by later sends, while `startStreaming()` still receives the raw reply target. I did not live-run a Feishu/Lark workspace.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the implementation should centralize reply-target consumption across streaming and non-streaming sends and update public Feishu config surfaces before merge.
