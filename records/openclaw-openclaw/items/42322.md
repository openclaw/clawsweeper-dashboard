## Summary

## What This Changes

The PR adds a Feishu `replyToMode` config key and threads all/first/off reply-target selection through the Feishu bot and reply dispatcher.

## Best Possible Solution

Land a rebased Feishu reply-target policy that uses one consumption path across streaming, text, card, and media sends, updates schema/manifest/docs/tests, and includes redacted Feishu/Lark proof.

## Reproduction Assessment

Yes for the PR defects at source level: the PR head adds first-mode state only in `consumeReplyToMessageId()`, while `startStreaming()` still receives the raw reply target. I did not live-run a Feishu/Lark workspace.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the implementation should centralize reply-target consumption across streaming and non-streaming sends and update public Feishu config surfaces before merge.
