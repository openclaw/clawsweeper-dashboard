## Summary

Keep open: current main still lacks Feishu `replyToMode`, and the PR has a source-level `first`-mode gap in streaming-plus-media dispatch, omits public config surfaces, lacks real Feishu/Lark runtime proof, and is currently conflicting.

## What This Changes

The PR adds a Feishu `replyToMode` config option and threads it through the Feishu bot and reply dispatcher for `all`, `first`, and `off` reply behavior.

## Best Possible Solution

Rebase and implement one Feishu reply-target policy across schema, manifest/docs/generated metadata, tests, streaming, media, card, and text paths, or explicitly keep `replyInThread` as the sole Feishu control.

## Reproduction Assessment

Yes, for the review findings: PR-head source shows `streaming.start()` still uses raw `replyToMessageId` while later sends consume it, and current main lacks Feishu `replyToMode` schema or dispatcher support. I did not live-run a Feishu/Lark workspace.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the implementation should first reconcile Feishu `replyInThread` behavior and apply one policy across all outbound send paths and public config surfaces.

## Review Findings

Overall correctness: patch is incorrect
