## Summary

Keep this PR open. Current main still does not expose Feishu `replyToMode`, and the PR is a relevant implementation candidate, but it is not merge-ready because `replyToMode: "first"` is not applied consistently to streaming-card creation and the public config metadata/docs/test coverage are still incomplete.

## What This Changes

This PR adds a Feishu `replyToMode` option (`off`/`first`/`all`) to the Feishu config schema and passes it into the reply dispatcher so reply-to metadata can be omitted or consumed after the first outbound send.

## Best Possible Solution

Land a rebased Feishu-owned implementation that resolves `replyToMode` with group over account/top-level precedence, applies one reply-target consumption policy across streaming-card creation, text/card chunks, media sends, and typing behavior, then updates manifest/docs/generated metadata plus focused regression tests for `off`, `first`, `all`, precedence, and streaming-plus-media delivery.

## Reproduction Assessment

Yes. A focused dispatcher test can reproduce the remaining PR bug by setting `replyToMode: "first"`, delivering a markdown/card streaming response with media, and asserting only one outbound operation receives `replyToMessageId`; the current PR diff would let the streaming card and first media send both use it.

## Solution Assessment

No, not as-is. The requested direction is reasonable, but the implementation should centralize reply-target consumption so every Feishu outbound path, including streaming-card startup, observes the same `first`/`off`/`all` policy and the public config surfaces stay aligned.
