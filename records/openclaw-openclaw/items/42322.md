## Summary

Keep open. The Feishu replyToMode feature is not on current main, but the PR still has a source-level bug in first-mode streaming behavior, lacks public config surface updates, and has no after-fix Feishu/Lark runtime proof.

## What This Changes

The PR adds a Feishu replyToMode config option and threads it through the Feishu bot and reply dispatcher for all, first, and off reply behavior.

## Best Possible Solution

Implement one Feishu reply-target policy across schema, manifest/docs/generated metadata, tests, streaming, media, card, and text paths, or keep replyInThread as the supported Feishu control.

## Reproduction Assessment

Yes, for the review findings: PR-head source shows streaming.start uses raw replyToMessageId while later sends consume it, and current main lacks Feishu replyToMode schema or dispatcher support. I did not live-run a Feishu/Lark workspace.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the implementation should apply one reply-target policy across all outbound send paths and public config surfaces before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Count streaming sends toward first-reply mode:** `extensions/feishu/src/reply-dispatcher.ts:314`
  - body: With replyToMode set to "first", streaming.start still receives the raw replyToMessageId before consumeReplyToMessageId is called. A streamed answer that also sends media can thread both the streaming card and the first media send, so first mode can produce multiple native replies; make streaming startup consume the same single-use target.
  - confidence: 0.9
- **[P3] Expose replyToMode in public config surfaces:** `extensions/feishu/src/config-schema.ts:184`
  - body: The schema accepts replyToMode, but the PR does not update the Feishu manifest/docs/generated config metadata that setup and discovery use. Users would get a supported runtime key that public config surfaces do not advertise; add those surfaces with the schema change.
  - confidence: 0.78
