## Summary

Keep this PR open, but it is not merge-ready. Current main still lacks the requested Feishu `replyToMode` surface, and the branch is useful parity work, but the PR head leaves streaming-card creation outside first-mode consumption and does not update the manifest/docs/changelog/test contract.

## What This Changes

The PR adds a Feishu `replyToMode` config option and passes it into Feishu reply dispatch for `off`, `first`, and `all` reply-target behavior.

## Best Possible Solution

Land a Feishu-owned parity implementation after rebasing to current main, centralizing reply-target consumption across streaming, text/card chunks, media, and typing behavior, then expose the setting through schema, manifest, docs, changelog, and focused regression tests.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: on the PR head, configure `replyToMode: "first"`, start a streaming card, then send media; `streaming.start()` still gets the original reply target and the first media send consumes it again.

## Solution Assessment

No, not as-is. The direction is plausible Feishu parity, but the implementation should use one mode-aware reply-target policy for every outbound path and update the supported metadata/docs/test surfaces.
