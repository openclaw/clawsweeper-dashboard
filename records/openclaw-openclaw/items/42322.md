## Summary

Keep this PR open for maintainer handling: current main still does not expose Feishu `replyToMode`, but the branch is conflicting, lacks real Feishu/Lark proof, and still has source-level gaps around `first` mode and public config discovery.

## What This Changes

The PR adds Feishu `replyToMode` config and dispatcher plumbing so Feishu replies can attach to all, first, or no outbound messages.

## Best Possible Solution

If Feishu should expose explicit `replyToMode`, rebase and wire one reply-target policy through schema, manifest/docs, tests, streaming, media, fallback, card, and text paths.

## Reproduction Assessment

Yes, for the review findings: source inspection of PR head shows `streaming.start()` still uses the raw reply target while later sends consume it, and current main has no Feishu `replyToMode` schema or dispatcher support. I did not live-run a Feishu/Lark workspace.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the implementation should first reconcile the related `replyInThread` maintainer direction and then apply one policy across all outbound send paths and public config surfaces.
