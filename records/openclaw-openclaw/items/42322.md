## Summary

Keep open: current main still lacks Feishu `replyToMode`, and the related attempts are closed or unrelated, but this PR is not merge-ready because `first` mode misses the streaming-card path and external Feishu/Lark behavior proof is still absent.

## What This Changes

The PR adds a Feishu `replyToMode` config key and threads `all`/`first`/`off` reply-target behavior from config resolution into Feishu reply dispatch.

## Best Possible Solution

Land a rebased Feishu parity implementation that applies one reply-target policy across streaming cards, chunked text/cards, media, and fallbacks, then exposes the setting through manifest metadata, docs, focused tests, and maintainer-owned changelog handling.

## Reproduction Assessment

Yes, for the review findings: source inspection of PR head shows `streaming.start()` still uses the original reply target while later sends consume it, and current main has no Feishu `replyToMode` schema or dispatcher support. I did not live-run a Feishu/Lark workspace.

## Solution Assessment

No, not as-is. The feature direction is narrow and consistent with other channels, but the implementation should centralize reply-target consumption and publish the new setting through supported config metadata/docs before merge.
