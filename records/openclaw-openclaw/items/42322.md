## Summary

Keep open. Current main still lacks Feishu replyToMode support, and this PR is a plausible feature, but PR head has a source-visible first-mode streaming bug, missing public config surfaces, and no contributor-supplied real Feishu/Lark behavior proof.

## What This Changes

Adds a Feishu `replyToMode` config option to the schema, bot, and reply dispatcher to control native reply targeting with `all`, `first`, and `off` modes.

## Best Possible Solution

Rebase and implement one Feishu reply-target policy across schema, manifest/docs, tests, streaming, media, card, and text paths, then provide redacted Feishu/Lark runtime proof.

## Reproduction Assessment

Yes, at source level for the PR findings: PR head starts streaming with raw `replyToMessageId` before the new consume helper can mark the first reply used, and current main has no Feishu `replyToMode` schema or dispatcher support. I did not live-run a Feishu/Lark workspace.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the implementation needs one reply-target policy across all outbound paths and public config surfaces before merge.
