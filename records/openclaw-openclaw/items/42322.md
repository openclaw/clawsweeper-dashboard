## Summary

Keep open: current main and the latest release still do not expose Feishu `replyToMode`, but this PR is not mergeable as-is because `first` mode can still reply more than once on streaming sends and the public config surfaces/proof are incomplete.

## What This Changes

The PR adds a Feishu `replyToMode` config option and threads it through bot dispatch and the reply dispatcher to control native reply targeting with `all`, `first`, and `off` modes.

## Best Possible Solution

Land a rebased Feishu reply-target policy that consistently covers schema, manifest/docs, tests, streaming, media, card, and text sends, with redacted Feishu/Lark runtime proof.

## Reproduction Assessment

Yes, at source level for the PR findings: PR head starts streaming with raw `replyToMessageId` before the new consume helper can mark the first reply used. I did not live-run a Feishu/Lark workspace.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the patch needs one reply-target policy across all outbound paths plus public config surfaces and real runtime proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9
