Keep open: current main still does not expose Feishu `replyToMode`, and this is a focused feature branch rather than cleanup-only churn. ## What This Changes

The PR adds a Feishu `replyToMode` config setting and dispatcher state intended to control whether replies attach to all, first, or no triggering messages.

## Best Possible Solution

Refresh the branch on current main, settle the Feishu reply policy, route every outbound path through one reply-target consumer, align docs/manifest discovery, and prove `all`/`first`/`off` in a real Feishu/Lark chat before merge.

## Reproduction Assessment

Yes for the PR defect at source level: the PR head starts streaming with raw `replyToMessageId` while first-mode state changes only inside `consumeReplyToMessageId()`. I did not run a live Feishu/Lark workspace.

## Solution Assessment

No, not as-is. A maintainable fix should centralize reply-target consumption across streaming, text, and media, then align schema, docs, and plugin discovery for the new config surface.
