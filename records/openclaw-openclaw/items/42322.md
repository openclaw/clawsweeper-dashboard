## Summary

Keep open: current main still lacks Feishu `replyToMode` and the related `replyToMode` attempts are closed unmerged, but this PR still needs changes and contributor proof before it is mergeable.

## What This Changes

The PR adds a Feishu `replyToMode` config option to schema/runtime paths and threads it into the reply dispatcher for `all`, `first`, and `off` reply behavior.

## Best Possible Solution

Rebase and implement one Feishu reply-target policy across schema, manifest/docs, tests, streaming, media, card, and text paths, or explicitly keep `replyInThread` as the sole Feishu control if that is the product decision.

## Reproduction Assessment

Yes, for the review findings: source inspection of PR head shows streaming startup still uses raw `replyToMessageId` while later sends consume it, and current main has no Feishu `replyToMode` schema or dispatcher support. I did not live-run a Feishu/Lark workspace.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the implementation should first reconcile the existing Feishu `replyInThread` direction and apply one policy across all outbound send paths and public config surfaces.
