## Summary

Keep open: current main still lacks Feishu `replyToMode`, but this PR is not ready because `first` mode still lets streaming-card startup bypass the reply-target consumer and the new public setting is not exposed through the supported config surfaces.

## What This Changes

This PR adds Feishu `replyToMode` schema/config resolution and threads the selected mode into Feishu reply dispatch so replies can be `all`, `first`, or `off`.

## Best Possible Solution

Land a rebased Feishu-owned parity implementation that applies one reply-target policy across streaming cards, text/card chunks, media, and fallbacks, then updates manifest metadata, docs, changelog, and focused regression tests.

## Reproduction Assessment

Yes. Source inspection of PR head shows `replyToMode: "first"` consumes the reply target for later sends, but `streaming.start()` still receives the original target, so a streamed card followed by media can attach twice.

## Solution Assessment

No, not as-is. The feature direction is narrow and consistent with other channels, but the implementation should centralize reply-target consumption and publish the new setting through supported metadata/docs/changelog surfaces before merge.
