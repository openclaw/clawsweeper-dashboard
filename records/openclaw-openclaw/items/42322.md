## Summary

Keep this PR open, but it is not ready to merge: the feature is still absent from current main and the PR head has a source-level correctness bug in `replyToMode: "first"`, plus missing public config surfaces and no real Feishu/Lark behavior proof.

## What This Changes

The PR adds a Feishu `replyToMode` config option and threads it through the bot and reply dispatcher to control native reply targeting with `all`, `first`, and `off` modes.

## Best Possible Solution

Rebase and implement one Feishu reply-target policy across schema, manifest/docs, tests, streaming, media, card, and text paths, then provide redacted real Feishu/Lark runtime proof.

## Reproduction Assessment

Yes, at source level for the PR findings: PR head starts streaming with raw `replyToMessageId` before the new consume helper can mark the first reply used, while current main and the latest release have no Feishu `replyToMode` support. I did not live-run a Feishu/Lark workspace.

## Solution Assessment

No, not as-is. The feature direction is plausible, but the patch needs one reply-target policy across all outbound paths plus public config surfaces before merge.
