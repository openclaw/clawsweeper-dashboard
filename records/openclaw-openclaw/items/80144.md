## Summary

Keep open. The PR addresses a valid schema-backed memory-search reliability feature request, but it should not merge yet because the new optional query-result cache can retain expired entries indefinitely for many unique queries.

## What This Changes

Adds schema-backed `memorySearch.query` timeout, cache TTL, and retry controls, wires them into builtin memory search runtime, and updates docs, config metadata, tests, generated hashes, and changelog.

## Best Possible Solution

Land the PR after adding bounded query-cache cleanup and regression coverage, while preserving the additive config shape and the core-versus-memory-core ownership boundary.

## Reproduction Assessment

Not applicable as a feature/config PR rather than a bug reproduction. Current-main source inspection confirms the requested keys are absent, and the PR diff plus tests exercise the proposed config and runtime paths.

## Solution Assessment

No, not yet. The schema/resolver work in core and runtime wiring in `memory-core` are the right boundary, but the query-result cache needs expiry sweeping or a size cap before this is the best landing shape.

- **[P2] Expire old query cache entries before storing more:** `extensions/memory-core/src/memory/manager.ts:375-378`
