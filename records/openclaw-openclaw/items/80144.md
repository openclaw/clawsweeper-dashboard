## Summary

Keep open. Current main still lacks the requested memory search query reliability controls, and this PR is the active implementation path, but the patch should not merge until the new query cache avoids caching degraded provider-failure fallbacks and has an expiry sweep or size bound.

## What This Changes

The PR adds schema/resolver/docs support for `memorySearch.query` timeout, cache TTL, and retry fields, plus builtin memory-core runtime cache/retry/fallback behavior and tests.

## Best Possible Solution

Land an owner-reviewed version that only caches fully successful query results and bounds or sweeps the query cache, with focused regression coverage preserving the additive config contract.

## Reproduction Assessment

Not applicable as a feature/config PR rather than a broken existing contract. Source inspection confirms current main lacks the requested fields, and the PR plus focused tests exercise the proposed implementation path.

## Solution Assessment

No, not yet. The schema/resolver and memory-core runtime boundary are the right direction, but the query cache needs cacheability and lifecycle fixes before this is the best landing shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Avoid caching degraded provider-failure results:** `extensions/memory-core/src/memory/manager.ts:494-496`
  - body: `searchWithProvider` can return keyword fallback or `[]` after a retryable provider failure, but the caller still stores that result when `cacheTtlMs` is enabled. A temporary embedding outage can therefore make later identical queries skip the provider and reuse degraded results for the full TTL, so only cache results when the provider path actually succeeded or carry a cacheable/degraded flag back from the provider search.
  - confidence: 0.9
- **[P2] Expire old query cache entries before storing more:** `extensions/memory-core/src/memory/manager.ts:375-378`
  - body: Expired entries are removed only when the same key is read or when the whole cache is cleared. A long-running gateway with many unique queries can retain stale result arrays indefinitely, so sweep expired entries or enforce a size bound before adding new cache entries.
  - confidence: 0.88
