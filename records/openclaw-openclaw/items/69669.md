## Summary

Keep open: current main and the latest release still define bound ACP follow-ups as direct ACP prompt text, and switching to parent-rewritten follow-ups is a product/session contract decision not covered by the related delivery PRs.

## What This Changes

Requests changing thread-bound ACP follow-ups so the parent assistant rewrites shorthand user messages into explicit child tasks by default, with verbatim relay only when explicitly requested.

## Best Possible Solution

ACP/session owners should choose the follow-up contract, then update runtime dispatch, accepted-note wording, public docs, and regression tests together while keeping current direct bound ACP behavior documented until that decision lands.

## Reproduction Assessment

Yes, at source level: current main retargets a bound conversation to the ACP session, sends BodyForAgent as ACP prompt text, and the regression test asserts raw "continue" reaches runTurn while the parent replyResolver is skipped. This proves the current behavior, but it conflicts with the requested new contract rather than a currently documented bug.

## Solution Assessment

Unclear/no until maintainers choose the product contract. The requested parent-rewrite default may be useful, but the narrow maintainable solution is a coordinated runtime/docs/test contract change after owner decision rather than a one-off pass-through patch.
