## Summary

Keep open: current main and v2026.6.10 still define bound ACP follow-ups as direct prompt text to the bound ACP session, while the requested parent-rewrite default remains a product/session contract decision. The adjacent open delivery PR preserves direct bound-session semantics, and the closed hub-delegated PR moved a different non-thread delegation slice to an external plugin rather than implementing this thread-bound behavior.

## What This Changes

The issue asks for thread-bound ACP follow-ups to remain parent-orchestrated by default, with raw verbatim forwarding only on explicit request.

## Best Possible Solution

ACP/session owners should decide the follow-up contract first, then update runtime dispatch, accepted-note text, public docs, and regression tests consistently while keeping the current direct contract intact until then.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.10 route a bound ACP conversation directly to the ACP session and test that raw follow-up text reaches the ACP runtime while parent reply resolution is skipped.

## Solution Assessment

No, not as an autonomous fix yet. The requested default may be useful, but the maintainable path is an owner-approved runtime/docs/test contract change rather than a one-off pass-through patch.
