## Summary

Keep open. Current `main` still encodes direct bound-ACP follow-up routing: bound conversations are retargeted to the ACP session before normal reply generation, ACP dispatch sends inbound prompt text to `runTurn`, and regression coverage asserts raw shorthand text reaches the ACP runtime while the parent reply resolver is skipped. The same-author PR #69675 is still open and docs-only, so this needs maintainer ACP contract review before a runtime change.

## What This Changes

Request that thread-bound ACP follow-up messages stay parent-orchestrated by default, with verbatim relay only when the user explicitly asks for exact forwarding.

## Best Possible Solution

Keep this issue open while maintainers decide the ACP contract and review #69675. If parent orchestration is accepted, implement a focused runtime distinction between parent-owned shorthand continuation and explicit verbatim ACP relay, then update the bound dispatch path, accepted-note wording, docs, and regression tests together.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current `main`: the bound ACP dispatch regression sends `BodyForAgent: "continue"`, asserts `runtime.runTurn` receives that exact text, and asserts the parent `replyResolver` is not called.

## Solution Assessment

Unclear. The requested behavior may be the right UX for parent-orchestrated coding workflows, but current docs and tests intentionally encode direct ACP routing, so the best fix depends on a maintainer semantics decision rather than an automatic patch.
