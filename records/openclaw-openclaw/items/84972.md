## Summary

Keep open: this is a plausible Anthropic long-context misclassification fix, but the current PR routes the error into generic overflow recovery without proving the retry leaves the long-context/request-eligibility path.

## What This Changes

This PR adds an Anthropic long-context usage-error classifier, routes that string through context-overflow handling before billing exclusion, exports the helper, adds focused tests, and updates troubleshooting docs.

## Best Possible Solution

Make the recovery provider-aware: either retry under the standard Anthropic window without the 1M request mode and prove that path, or keep this response in eligibility/billing handling with operator guidance.

## Reproduction Assessment

Yes at source level: current main matches the exact body as billing and excludes billing before overflow handling. I did not see a live Anthropic long-context 429 reproduced against the patched runtime.

## Solution Assessment

No: the classifier reorder alone is not the best fix because current overflow recovery still uses the resolved 1M token budget and context1m request behavior. The safer solution needs provider-aware standard-tier retry semantics or an explicit decision to keep this in eligibility handling.
