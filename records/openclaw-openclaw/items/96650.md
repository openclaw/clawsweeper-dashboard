## Summary

Keep open: the PR targets the right Slack outbound mirror-session bug, but the patch currently changes Slack's behavior when the new explicitness flag is omitted, which can affect existing explicit `replyToId` callers, and it still lacks real Slack proof.

## What This Changes

Adds an optional `replyToIsExplicit` route parameter, forwards it through message-action outbound mirror routing, and makes Slack prefer the current thread session for inherited replies while adding focused route tests.

## Best Possible Solution

Land one narrow Slack route fix that treats explicitness as tri-state: explicit `true` stays reply-first, inherited `false` uses current-session-first, and omitted/unknown preserves the old reply-first default, with gateway/API coverage and redacted live Slack proof.

## Reproduction Assessment

Yes. Source inspection shows current main lets Slack inherited child reply IDs win over a recovered thread session, and the linked issue includes deployment evidence of mirror-only sibling sessions; I did not run a live Slack workspace replay.

## Solution Assessment

No. The proposed fix is the right layer, but the current implementation treats unknown explicitness as inherited; the safer fix preserves the old reply-first default for omitted fields and only changes the known inherited path.
