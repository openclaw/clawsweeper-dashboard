## Summary

Keep open; this PR is not safe to merge because the added `feishu_doc` implementation calls an unsupported `client.docx.documentComment` surface, while current main already routes Feishu document comments through Drive-backed `feishu_drive` APIs and the remaining public edit/delete contract needs maintainer direction.

## What This Changes

Adds Feishu `feishu_doc` comment actions, schema/docs entries, and mocked tests for listing, getting, creating, updating, and deleting Docx comments.

## Best Possible Solution

Keep one coherent Feishu comment interface: either keep `feishu_drive` canonical, or add maintainer-approved `feishu_doc` wrappers over verified Drive APIs with pagination, supported reply/root semantics, docs, tests, and a changelog entry.

## Reproduction Assessment

Yes. The failure is source-reproducible by comparing the PR diff with the pinned SDK types: the PR calls `(client.docx as any).documentComment`, while comments are exposed under Drive APIs and no `documentComment` member exists.

## Solution Assessment

No. The direct `docx.documentComment` implementation is not the best fix; a typed Drive-backed wrapper, or keeping `feishu_drive` canonical, is safer and matches current main behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use the real Drive comment APIs:** `extensions/feishu/src/docx.ts:953`
  - body: The pinned Lark SDK exposes comments as `client.drive.fileComment` and `client.drive.fileCommentReply`, while `client.docx` has document/block resources and no `documentComment` member. Because this casts `docx` to `any`, a real client will dereference an absent API and every new comment action will fail at runtime.
  - confidence: 0.94
- **[P2] Expose only supported edit and delete semantics:** `extensions/feishu/src/doc-schema.ts:208-218`
  - body: `update_comment` and `delete_comment` only accept `comment_id`, but the verified Drive API updates/deletes replies via `fileCommentReply` and requires a `reply_id`; root comments expose solve-state patching, not arbitrary content update/delete through this shape. Please align the schema, docs, and dispatcher with operations Feishu can actually call.
  - confidence: 0.86
- **[P2] Page through comment listings:** `extensions/feishu/src/docx.ts:963-972`
  - body: `list_comments` performs one list request and reports `total` from that page's item count. The Feishu comment list API is paginated with `has_more` and `page_token`, so active documents can omit later comments and return a misleading count unless the tool returns pagination metadata or iterates all pages.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `extensions/feishu/src/doc-schema.ts:181`
  - body: This adds user-facing Feishu tool actions but the PR does not touch `CHANGELOG.md`. Repo policy requires a changelog entry for user-facing `feat` changes before merge.
  - confidence: 0.82
