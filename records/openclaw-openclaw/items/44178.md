## Summary

Close as superseded: merged #58497 shipped the Feishu document-comment collaboration surface through `feishu_drive`, and this PR now duplicates that area while using an API shape that does not match the SDK or current implementation.

## What This Changes

The PR adds Feishu document comment actions to `feishu_doc`, updates its schema and skill docs, and adds mocked tests for list/get/create/update/delete comment operations.

## Best Possible Solution

Keep the shipped `feishu_drive` comment implementation as the canonical surface; any future edit/delete/resolve additions should be a new Drive-API-based follow-up with explicit schema and tests.

## Reproduction Assessment

Yes. Source inspection is enough: the PR calls `(client.docx as any).documentComment`, while the locked SDK and current Feishu code expose document comments through Drive APIs, so the new actions have a high-confidence failure path.

## Solution Assessment

No. The already-merged Drive-based implementation is the maintainable solution; adding a parallel `feishu_doc` surface around an untyped, mismatched SDK path would duplicate and break the existing Feishu comment flow.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Use the Drive comment API surface:** `extensions/feishu/src/docx.ts:956-958`
  - body: The SDK does not expose `client.docx.documentComment`; cloud-document comments are under Drive comment APIs with `file_token` and `file_type`. These new tool actions will fail at runtime before performing any comment operation.
  - confidence: 0.93
- **[P2] Preserve comment pagination metadata:** `extensions/feishu/src/docx.ts:965-967`
  - body: `list_comments` makes one request and reports `total` from that page's item count, so paginated documents would silently omit later comments and return a misleading count. The current Drive tool surfaces `has_more` and `page_token` instead.
  - confidence: 0.88
