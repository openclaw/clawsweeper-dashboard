## Summary

Keep open: current main covers Feishu comments through `feishu_drive`, but this PR is not mergeable because the new `feishu_doc` implementation calls a nonexistent SDK surface, duplicates the existing Drive-backed comment tooling, and exposes unsupported edit/delete semantics.

## What This Changes

The PR adds Feishu `feishu_doc` comment actions, schema entries, skill documentation, and mocked tests for listing, getting, creating, updating, and deleting document comments.

## Best Possible Solution

Keep one coherent Feishu comment interface: either keep `feishu_drive` canonical, or add maintainer-approved `feishu_doc` wrappers over verified Drive APIs with pagination, supported reply/root semantics, docs, tests, and a changelog entry.

## Reproduction Assessment

Yes. The failure is source-reproducible by comparing the PR diff with the pinned SDK types: the PR calls `client.docx.documentComment`, while the SDK exposes comments under `client.drive.fileComment` and `client.drive.fileCommentReply`.

## Solution Assessment

No. The PR's direct `docx.documentComment` implementation is not the best fix; a typed Drive-backed wrapper, or keeping `feishu_drive` canonical, is safer and matches current main behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the real Drive comment APIs:** `extensions/feishu/src/docx.ts:953`
  - body: The pinned Lark SDK exposes comments as `client.drive.fileComment` / `fileCommentReply`, while `client.docx` has document/block resources and no `documentComment` member. Because this casts `docx` to `any`, a real client will dereference an absent API and every new comment action will fail at runtime.
  - confidence: 0.93
- **[P2] Expose only supported edit and delete semantics:** `extensions/feishu/src/doc-schema.ts:208-218`
  - body: `update_comment` and `delete_comment` only accept `comment_id`, but the verified Drive API updates/deletes replies via `fileCommentReply` and requires a `reply_id`; root comments expose solve-state patching, not arbitrary content update/delete through this shape. Please align the schema, docs, and dispatcher with operations Feishu can actually call.
  - confidence: 0.84
- **[P2] Page through comment listings:** `extensions/feishu/src/docx.ts:963-965`
  - body: `list_comments` performs one list request and reports `total` from that page's item count. The Feishu comment list API is paginated with `has_more` / `page_token`, so active documents can omit later comments and return a misleading count unless the tool returns pagination metadata or iterates all pages.
  - confidence: 0.89
