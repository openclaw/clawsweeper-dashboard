## Summary

Keep open: this PR is not mergeable as-is. The current Feishu plugin already has a Drive-backed comment surface, while the PR adds a parallel `feishu_doc` surface that calls a `docx.documentComment` SDK member that is absent from the pinned Lark SDK and also drops comment-list pagination.

## What This Changes

The PR adds `list_comments`, `get_comment`, `create_comment`, `update_comment`, and `delete_comment` actions to `feishu_doc`, updates the Feishu doc skill text, and adds docx tests for those comment operations.

## Best Possible Solution

Expose one coherent Feishu comment interface: either keep `feishu_drive` as canonical, or add maintainer-approved `feishu_doc` wrappers over the verified Drive comment APIs with pagination, reply-vs-root comment semantics, docs, tests, and a changelog entry.

## Reproduction Assessment

Yes. The failure is reproducible by inspecting the PR diff against the pinned SDK types: the real client exposes Drive comment resources, while the PR calls `client.docx.documentComment`; current main also shows the existing `feishu_drive` comment path.

## Solution Assessment

No. The PR's direct `docx.documentComment` implementation is not the best fix because it bypasses the dependency contract and duplicates the existing Drive comment surface; a typed Drive-backed wrapper or keeping `feishu_drive` canonical is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Use the real Drive comment API:** `extensions/feishu/src/docx.ts:953`
  - body: The pinned Lark SDK exposes comments under `client.drive.fileComment` / `fileCommentReply`, while `client.docx` has no `documentComment` member. Because this casts `docx` to `any`, a real client will dereference an absent API and every new comment action will fail at runtime.
  - confidence: 0.92
- **[P2] Expose only supported edit and delete semantics:** `extensions/feishu/src/doc-schema.ts:208`
  - body: `update_comment` and `delete_comment` are documented as editing or removing a comment with only `comment_id`, but the verified Drive API separates root comments from replies: root comments have list/get/create/solve-state operations, while reply update/delete requires a `reply_id`. Please align the schema/docs with operations that can be called correctly.
  - confidence: 0.82
- **[P2] Page through comment listings:** `extensions/feishu/src/docx.ts:963-965`
  - body: `list_comments` performs one list request and reports `total` from that page's item count. Feishu comment listing is paginated (`has_more` / `page_token`), so active documents can omit later comments and return a misleading count. Return pagination metadata or iterate pages before presenting this as a full list.
  - confidence: 0.88
