## Summary

Keep open: the Feishu no-allowFrom group-command bug is real, but this PR is not safe as-is because it changes shared command authorization semantics, can regress DM command gating, and leaves the root SDK mirror inconsistent.

## What This Changes

The PR changes the shared command-gating helper so all-unconfigured authorizers allow commands when access groups are enabled, and updates the helper unit tests.

## Best Possible Solution

Scope the no-allowFrom allow behavior to Feishu group command authorization, or add an explicit mirrored helper mode, while preserving DM fail-closed contracts and adding regression coverage plus a changelog entry.

## Reproduction Assessment

Yes, from source inspection. Current main builds an unconfigured Feishu group command authorizer when no `allowFrom` exists and the shared helper returns false under `useAccessGroups=true`; the PR's shared fallback also source-reproduces the introduced open-DM authorization regression.

## Solution Assessment

No. The current PR fixes the Feishu symptom by changing shared helper semantics for every `useAccessGroups=true` caller; a scoped Feishu group-command fix, or an explicit shared mode mirrored into the root SDK, is the narrower maintainable path.
