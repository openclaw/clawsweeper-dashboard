## Summary

Keep open: this PR targets a real, still-open Feishu command authorization bug, but the patch is not merge-ready because it widens shared command authorization, leaves the root SDK shim inconsistent, lacks after-fix runtime proof, and has failing checks on the current head.

## What This Changes

This PR changes the shared command-gating helper to authorize access-groups commands when no authorizer is configured, and updates command-gating tests for that behavior.

## Best Possible Solution

Land a narrow Feishu command-authorization fix that authorizes admitted senders when no explicit command allowlist is configured, while preserving explicit allowlist and owner-command enforcement; only change shared helper semantics with an explicit mode and SDK shim parity.

## Reproduction Assessment

Yes, source-level: current main can carry an admitted Feishu slash command into `CommandAuthorized=false` when no command allowlist is configured, and `runPreparedReply` then drops unauthorized whole-message control commands. I did not run a live Feishu workspace.

## Solution Assessment

No. The bug-fix direction is valid, but this patch changes a shared helper too broadly and leaves the root SDK shim divergent; a Feishu-scoped fix or explicit shared mode with parity is safer.
