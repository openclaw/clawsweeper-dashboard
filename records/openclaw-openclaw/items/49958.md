## Summary

Keep open: current main still source-reproduces the linked Feishu slash-command authorization bug, but this PR is not merge-ready because it broadens a shared command-authorization helper beyond the Feishu case, leaves the root SDK shim on different semantics, and lacks contributor-supplied real behavior proof.

## What This Changes

This PR changes `resolveCommandAuthorizedFromAuthorizers` so all-unconfigured `useAccessGroups` authorizers allow commands, and updates command-gating tests for that behavior.

## Best Possible Solution

Land or replace this with a Feishu-scoped command-authorization fix that authorizes admitted senders when no explicit command allowlist is configured, preserves explicit allowlists and owner-command enforcement, and keeps SDK entrypoints consistent.

## Reproduction Assessment

Yes, source-level. Current main can carry a Feishu group slash command with no command allowlist into `CommandAuthorized=false`, and the reply path then drops unauthorized whole-message control commands; I did not run a live Feishu workspace.

## Solution Assessment

No. The bug-fix direction is valid, but this branch changes a shared authorization helper globally and leaves the root SDK shim divergent; a Feishu-scoped fix or explicit shared mode with parity is safer.
