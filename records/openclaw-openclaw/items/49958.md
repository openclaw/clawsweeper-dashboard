## Summary

Keep open: the Feishu slash-command bug is still source-reproducible on current main, but this PR is not merge-ready because it broadens shared command authorization beyond the reported Feishu path and lacks after-fix real behavior proof.

## What This Changes

This PR changes shared command gating so all-unconfigured `useAccessGroups` authorizers allow commands, and updates command-gating unit tests for that behavior.

## Best Possible Solution

Land a narrow Feishu command-authorization fix that authorizes admitted senders when no explicit command allowlist is configured, preserves explicit allowlist denial and owner-command enforcement, and keeps SDK entrypoints consistent.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, at source level: current main can carry an admitted Feishu slash command with no command allowlist into `CommandAuthorized=false`, and the reply path then drops unauthorized whole-message control commands; I did not run a live Feishu workspace.

## Solution Assessment

Is this the best way to solve the issue? No. The bug-fix direction is valid, but this branch changes a shared helper too broadly and leaves the root SDK shim divergent; a Feishu-scoped fix or explicit shared mode with parity is safer.
