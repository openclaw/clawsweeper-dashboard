## Summary

Keep open: the Feishu slash-command bug is real, but this PR changes shared command authorization too broadly, still lacks after-fix real Feishu/OpenClaw behavior proof, and already has unresolved review findings on the same head.

## What This Changes

The PR adds a no-configured-authorizer allow branch to the shared command gate and updates command-gating unit tests for that behavior.

## Best Possible Solution

Land a Feishu-scoped command-authorization fix, or a clearly scoped helper mode, that authorizes admitted Feishu senders when no explicit command allowlist is configured while preserving shared fail-closed behavior and SDK parity.

## Reproduction Assessment

Yes, at source level: current main can pass an admitted Feishu slash command with an empty command allowlist into the shared access-groups command gate, which denies all-unconfigured authorizers. I did not run a live Feishu workspace in this read-only review.

## Solution Assessment

No. The intended Feishu bug fix is valid, but changing the shared helper is not the narrowest maintainable solution because DM, generic ingress, and SDK command-gate paths rely on the existing fail-closed semantics.
