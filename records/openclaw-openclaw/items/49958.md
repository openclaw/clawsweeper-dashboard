## Summary

Keep this PR open, but do not merge the current patch. It targets a real Feishu group-command bug tracked by https://github.com/openclaw/openclaw/issues/49915, yet the branch changes a shared authorization helper in a way that can authorize control commands outside the intended Feishu group flow and leaves the root SDK mirror inconsistent.

## What This Changes

The PR adds an all-unconfigured-authorizers allow fallback to `resolveCommandAuthorizedFromAuthorizers` when `useAccessGroups` is enabled, and updates the command-gating unit tests to expect that behavior.

## Best Possible Solution

Fix the Feishu no-`allowFrom` group-command case at the narrow group-command boundary, or add an explicit helper mode that distinguishes admitted group chats from DM command auth. Keep root SDK mirror behavior aligned and cover Feishu group success, configured-deny, and DM no-allowlist negative cases.

## Reproduction Assessment

Yes. Current main reproduces from source and tests: an admitted Feishu group with no sender `allowFrom` reaches the command helper as `{ configured: false, allowed: false }`, the helper returns false, and the Feishu regression test still records `CommandAuthorized: false`.

## Solution Assessment

No. The proposed patch fixes the Feishu symptom by changing a shared helper contract for every `useAccessGroups=true` caller, and it misses the root SDK CJS mirror; a scoped Feishu/group-command fix or explicit helper mode is the safer maintainable path.
