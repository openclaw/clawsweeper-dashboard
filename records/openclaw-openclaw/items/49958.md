## Summary

Keep open: the underlying Feishu slash-command bug is real, but this branch broadens shared command authorization, leaves SDK semantics inconsistent, and lacks after-fix real behavior proof.

## What This Changes

The PR changes the shared command-gating helper so access-groups commands are authorized when no authorizer is configured, and updates command-gating unit tests to expect that.

## Best Possible Solution

Land a Feishu-scoped command-authorization fix, or an explicit shared mode with SDK shim parity, that authorizes admitted Feishu senders when no command allowlist is configured while preserving explicit allowlist denial.

## Reproduction Assessment

Yes, at source level: current main can carry an admitted Feishu group slash command with no command allowlist into `CommandAuthorized=false`, and the reply path then drops unauthorized whole-message control commands without a reply. I did not run a live Feishu workspace in this read-only review.

## Solution Assessment

No. The bug-fix direction is valid, but this patch changes a shared helper too broadly; a Feishu-scoped fix or explicit helper mode with SDK shim parity is safer.
