## Summary

Keep this PR open. Current main still has the Feishu group slash-command denial that the PR targets, but the proposed shared-helper fallback is too broad for the current command-auth contracts and also leaves the root SDK CommonJS mirror with different semantics.

## What This Changes

The branch changes the shared command-gating helper to allow commands when access groups are enabled and no authorizer is configured, and updates the colocated command-gating tests for that behavior.

## Best Possible Solution

Revise the fix so an already admitted Feishu group with no sender `allowFrom` is treated as unrestricted for that group command flow, while preserving DM no-allowlist command restrictions. If the shared helper contract changes, keep the root SDK CJS mirror aligned and add direct Feishu group regression coverage plus DM negative coverage.

## Reproduction Assessment

Yes. Current main reproduces from source and tests: an explicitly configured Feishu group with no sender `allowFrom` reaches command authorization as `{ configured: false, allowed: false }`, the shared helper returns false, and whole-message control commands are then silently dropped.

## Solution Assessment

No. The proposed direction fixes the Feishu symptom but applies the no-config allow rule at the shared helper level, which is broader than the Feishu group-only bug and leaves an SDK mirror inconsistent; a scoped group-command path or an explicit helper mode would be safer.
