## Summary

Keep open: the Feishu slash-command bug is still source-reproducible, but this PR changes shared command authorization too broadly and lacks required real Feishu/OpenClaw behavior proof.

## What This Changes

The PR changes the shared command-gating helper so access-groups mode allows all-unconfigured authorizers, and updates command-gating unit tests.

## Best Possible Solution

Land a narrow Feishu command-authorization fix, or an explicit mirrored helper mode, that authorizes admitted senders when no command allowlist is configured while preserving explicit allowlist and DM fail-closed contracts.

## Reproduction Assessment

Yes, source-reproducible. Current main can carry a Feishu group slash command with empty command allowlists into `CommandAuthorized=false`, and the reply path then drops unauthorized whole-message control commands.

## Solution Assessment

No. The broad shared-helper fallback is not the narrowest maintainable fix; Feishu should handle the admitted-sender no-command-allowlist case directly, or the shared contract must be explicit and mirrored in the root SDK.

## Review Findings

- **[P1] Scope the no-config fallback to Feishu group commands:** `src/channels/command-gating.ts:31-32`
  - body: The new all-unconfigured allow branch is shared by every `useAccessGroups=true` caller, not only Feishu groups. DM command paths build all-unconfigured authorizers when `allowFrom` and `groupAllowFrom` are empty, so this can flip open-DM no-allowlist command handling from unauthorized to authorized despite existing tests that drop those commands.
  - confidence: 0.91

  - body: This changes the TypeScript helper’s access-groups semantics but leaves the CommonJS root SDK mirror on the old behavior. Plugins using the root SDK entrypoint can therefore get different authorization results from scoped/runtime imports for the same no-authorizer configuration.
  - confidence: 0.86
