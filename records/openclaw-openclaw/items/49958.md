## Summary

Keep open for revision: the underlying Feishu slash-command authorization bug is still source-reproducible on current main, but this PR is not merge-ready because it broadens a shared authorization helper, leaves the root SDK shim inconsistent, and still lacks contributor real behavior proof.

## What This Changes

The PR changes shared command authorization so all-unconfigured `useAccessGroups` authorizers allow commands, and updates command-gating tests to expect that behavior.

## Best Possible Solution

Land a narrow Feishu command-authorization fix that authorizes admitted senders when no explicit command allowlist is configured, while preserving explicit allowlist and owner-command enforcement and keeping SDK entrypoints consistent.

## Reproduction Assessment

Yes, source-level: current main can carry an admitted Feishu slash command with no command allowlist into `CommandAuthorized=false`, and the reply path then drops unauthorized whole-message control commands. I did not run a live Feishu workspace.

## Solution Assessment

No. The bug-fix direction is valid, but this patch changes a shared helper too broadly and leaves the root SDK shim divergent; a Feishu-scoped fix or explicit shared mode with parity is safer.
