## Summary

Keep open, but this PR is not merge-ready: current main still source-reproduces the Feishu no-allowlist slash-command problem, while the proposed shared-helper fallback is too broad and can widen unrelated DM command authorization.

## What This Changes

This PR changes shared command-gating behavior in `src/channels/command-gating.ts` and expands the helper tests in `src/channels/command-gating.test.ts`.

## Best Possible Solution

Use a scoped Feishu/channel-ingress command-authorization fix or an explicit helper mode that authorizes already-admitted senders only when no command allowlist exists, while preserving DM/open-policy and explicit-allowlist fail-closed behavior.

## Reproduction Assessment

Yes, source-level. Current main denies all-unconfigured `useAccessGroups=true` command authorizers, Feishu can build that empty command-allowlist path for slash commands, and unauthorized whole-message commands are then dropped without a reply.

## Solution Assessment

No. The current PR fixes the symptom through a global helper semantic change, but the safer solution is a scoped Feishu or explicit-mode fix with tests for DM, group, explicit allowlist, and SDK alias behavior.
