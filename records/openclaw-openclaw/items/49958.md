## Label Justifications

- `P1`: The underlying Feishu slash-command workflow is user-facing and currently broken, while the proposed fix touches shared authorization behavior.
- `merge-risk: 🚨 compatibility`: The PR changes existing command-gate semantics for all `useAccessGroups=true` callers with empty command allowlists, which can alter existing deployments.
- `merge-risk: 🚨 security-boundary`: The diff broadens who can be command-authorized across shared channel and SDK paths, including control-command surfaces outside Feishu.

## Summary

Keep open: the Feishu slash-command bug is still source-reproducible on current main, but this PR fixes it by broadening a shared command-authorization helper, leaves the root SDK shim with different semantics, and still lacks contributor real behavior proof.

## What This Changes

The PR changes shared command gating so all-unconfigured `useAccessGroups` authorizers allow commands and updates command-gating unit tests for that behavior.

## Best Possible Solution

Land a narrow Feishu command-authorization fix that authorizes admitted senders when no explicit command allowlist is configured, while preserving explicit allowlist denial and SDK entrypoint consistency.

## Reproduction Assessment

Yes, source-level: current main can carry an admitted Feishu slash command with no command allowlist into `CommandAuthorized=false`, and the reply path then returns no response for unauthorized whole-message control commands. I did not run a live Feishu workspace.

## Solution Assessment

No. The bug-fix direction is valid, but this implementation changes a shared helper too broadly and leaves the root SDK shim divergent; a Feishu-scoped fix or explicit shared mode with parity is safer.
