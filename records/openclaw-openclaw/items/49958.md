## Summary

Keep open: the Feishu command-authorization bug remains source-reproducible on current main, but this PR is not merge-ready because it broadens shared command-gating semantics beyond Feishu and lacks required after-fix real behavior proof.

## What This Changes

The PR changes shared command-gating to authorize access-groups command senders when no authorizer is configured and updates the command-gating unit tests to expect that behavior.

## Best Possible Solution

Land a Feishu-scoped command-authorization fix that authorizes admitted Feishu senders when no explicit command allowlist is configured, while preserving shared fail-closed semantics and SDK parity.

## Reproduction Assessment

Yes, at source level: current main can carry an admitted Feishu slash command with no command allowlist into `CommandAuthorized=false`, and whole-message unauthorized commands are then dropped without a reply. I did not run a live Feishu workspace in this read-only review.

## Solution Assessment

No. The bug fix direction is valid, but this branch changes a shared helper too broadly; the safer solution is a Feishu-scoped authorization fix or an explicit helper mode with SDK shim parity and regression coverage.
