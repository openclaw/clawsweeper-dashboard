## Summary

Keep open: the PR fixes the schema rejection narrowly, but it accepts Qwen thinking formats without carrying them through OpenClaw's boundary-aware request builder or catalog normalization, and the contributor has not provided real after-fix behavior proof.

## What This Changes

This PR expands the model compat `thinkingFormat` type/schema to accept `qwen` and `qwen-chat-template` and adds schema parse tests.

## Best Possible Solution

Implement the full config-to-payload contract if generic `thinkingFormat` Qwen values are accepted, or keep Qwen thinking control in the documented provider-owned params path and leave the core compat enum closed.

## Reproduction Assessment

Yes, for the schema rejection path: current main's schema/type excludes the two Qwen values and the PR changes that direct parse gate. I did not run a live gateway in this read-only review, and the PR does not include after-fix real behavior proof.

## Solution Assessment

No. Adding enum values is necessary for the proposed config surface, but it is not sufficient because OpenClaw's request builder and catalog normalization still do not preserve or apply those Qwen formats.
