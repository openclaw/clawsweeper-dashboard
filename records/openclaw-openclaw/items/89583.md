## What This Changes

The PR wraps unreadable `packages/llm-core` tool parameter schemas in controlled errors and adds regression coverage for a plain JSON-schema `properties` proxy.

## Best Possible Solution

Keep the schema hardening, but snapshot/read the tool schema once and wrap only schema-read, conversion, compile, and schema-coercion failures while keeping value-side argument failures on the validation-failure path.

## Reproduction Assessment

Source-level yes: current main passes `tool.parameters` directly into TypeBox conversion/compilation and plain JSON-schema coercion, and TypeBox source shows those paths traverse schema fields that can throw. I did not execute tests because this review is read-only.

## Solution Assessment

No. Wrapping unreadable schema traversal is the right direction, but this PR is not the best final shape because it also wraps `validator.Check`/`Errors` value traversal as invalid schema failures.
