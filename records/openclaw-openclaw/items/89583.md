## Summary

The PR remains useful but should stay open for maintainer handling: current main and the latest release still have the unwrapped LLM-core validation path, while this maintainer draft overlaps two other open drafts for the same unreadable-schema boundary and changes a plugin-visible error contract.

## What This Changes

This PR wraps `validateToolArguments()` schema conversion, compilation, plain JSON-schema coercion, checking, and error enumeration so unreadable tool parameter schemas become controlled `Invalid parameter schema` errors, with a regression test for a throwing JSON-schema `properties` proxy.

## Best Possible Solution

Land one canonical LLM-core unreadable-schema guard with stable plugin-facing error semantics and focused regression coverage, then close or supersede the overlapping draft branches.

## Reproduction Assessment

Yes from source inspection: current main and `v2026.6.9` pass `tool.parameters` into TypeBox and local JSON-schema traversal without a schema-error boundary, and the pinned TypeBox package enumerates schema properties in these paths. I did not run tests because this cleanup review is read-only.

## Solution Assessment

Unclear as the final best fix. `validateToolArguments()` is the right boundary, but maintainers still need to choose whether this narrow wrapper or a broader schema snapshot/preflight branch is the canonical implementation.
