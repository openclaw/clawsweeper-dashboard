## Summary

Keep open. Current main has shipped the narrow transport retry classifier fix, but the reported large reindex failure is not fully resolved because exhausted transport failures still abort the batch, sync warnings still flatten causes with `String(err)`, and failed safe reindex still removes the temp database instead of preserving resumable progress.

## What This Changes

The issue asks OpenClaw memory search to survive large remote-embedding reindex failures without losing roughly 40K chunks of progress and to log enough fetch-cause detail to diagnose the provider failure.

## Best Possible Solution

Keep the shipped retry-classifier fix, then land or replace the batch-splitting retry-exhaustion work and resolve checkpoint/resume plus redacted cause-aware logging through the linked memory reliability items.

## Reproduction Assessment

Yes, at source level. A mocked embedding provider can still exhaust retries with `TypeError("fetch failed")`, causing the batch to throw through safe reindex and the temp database to be removed; the full 40K-chunk provider run was not repeated.

## Solution Assessment

No, this is not ready to close as solved. Current main handles the narrow transient retry classifier subset, but the remaining batch-split, diagnostics, and resumable-progress behavior need the linked follow-up work or an explicit maintainer split.
