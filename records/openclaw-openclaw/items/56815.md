## Summary

Keep open. Current main still leaves the reported large remote-embedding reindex failure mode unresolved: transport-shaped `TypeError("fetch failed")` is not retryable, async memory sync warnings still flatten errors, and failed safe reindex still deletes the temp database. Related #44167 is an open implementation candidate for the retry/split subset, while #26772 tracks the broader checkpoint/resume problem, so this issue should stay open until those pieces land or maintainers split the scope deliberately.

## What This Changes

The issue requests more resilient memory_search reindexing for large remote embedding runs that fail with `TypeError: fetch failed`, lose temp-index progress, and lack stack/cause diagnostics.

## Best Possible Solution

Keep this issue open while landing #44167 or an equivalent narrow memory-core fix for transient embedding transport retry and ordered batch splitting, then add redacted stack/cause-aware sync logging and decide the failed-reindex recovery design with #26772 so atomic swap safety is preserved while large runs can resume or retain useful checkpoint state.

## Reproduction Assessment

Yes. A focused source-level repro can mock the memory embedding provider to throw `TypeError("fetch failed")` and observe that the current retry classifier does not retry it; the full 40K-chunk provider run was not repeated in this read-only review.

## Solution Assessment

No, not as a close or single-subset fix. #44167 appears aimed at the narrow transport retry portion, but this issue also needs diagnostics and a checkpoint/resume or temp-index preservation decision before the reported user problem is fully addressed.
