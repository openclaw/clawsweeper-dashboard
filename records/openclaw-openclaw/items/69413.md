## Summary

Current main still misses native Signal mention metadata in the mention gate, and the PR's core fix remains useful; keep it open because this external PR still lacks real Signal after-fix proof and the operator docs still contradict the new behavior.

## What This Changes

The PR adds Signal native mention metadata detection for the bot account, wires it into group mention gating, and adds Signal mention-gating tests for uuid, phone, other-participant, and malformed-span cases.

## Best Possible Solution

Land the focused Signal plugin fix after contributor-provided real Signal proof and an operator-facing docs update, while leaving metadata-only activation policy as a separate maintainer follow-up if desired.

## Reproduction Assessment

Yes, by source inspection: a Signal group with requireMention true, a display-name mentionPattern such as @caio, and native mention metadata for the bot hydrates to @uuid/@phone, so current main's display-name regex does not set wasMentioned and the message is skipped. I did not run a live Signal account reproduction in this read-only review.

## Solution Assessment

Yes for the targeted bug: matching structured Signal mention metadata against deps.account/deps.accountUuid inside the Signal plugin is the narrowest owner-local fix. The merge-ready path still needs docs and real behavior proof rather than a broader core mention-policy change.
