## Summary

Keep this PR open. Current main and the latest shipped release still gate Signal group mentions through text regex matching after hydrating native mention placeholders, so the native display-name-vs-uuid/phone mismatch remains unfixed; the PR is a focused Signal plugin fix and looks functionally sound after the span-validation follow-up, but it still needs maintainer review plus docs/changelog follow-through.

## What This Changes

The PR changes Signal inbound handling to OR metadata-backed native bot-mention detection into `wasMentioned`, adds `doesSignalMentionTargetBot`, and covers uuid, phone, other-participant, and forged-span cases in the Signal mention-gating tests.

## Best Possible Solution

Land or rework the focused Signal plugin change after preserving the raw-message span validation, adding the docs/changelog updates, and deciding whether metadata-only native mention detection should also make `canDetectMention` true or be tracked as a separate follow-up.

## Reproduction Assessment

Yes. The code-level reproduction is high-confidence: a Signal group with `requireMention: true`, a display-name mention regex such as `@caio`, and a native mention whose metadata contains the bot uuid/phone will hydrate to `@<uuid>` or `@<phone>` and fail current main's text regex gate.

## Solution Assessment

Yes for the core code path: matching structured Signal mention metadata against the bot's own uuid/phone inside the Signal plugin is the narrowest maintainable fix, and the second commit adds the important span check. The merge-ready solution should also update docs/changelog and make an explicit scope call on the pre-existing metadata-only `canDetectMention` gap.
