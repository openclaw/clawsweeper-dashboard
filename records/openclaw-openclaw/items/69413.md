## Summary

Keep open. Current main still has the Signal mention-gating gap, and the PR's code path appears to address the bug after the span-validation follow-up, but the branch still lacks the required user-facing docs/changelog update for a Signal behavior change.

## What This Changes

The PR adds Signal native mention metadata matching against the bot account in inbound group mention gating and adds coverage for UUID, phone, non-bot, and malformed-span cases.

## Best Possible Solution

Ship the plugin-local native metadata matcher with raw-message span validation, update the Signal operator docs and changelog, and track any broader metadata-only `canDetectMention` policy as an explicit follow-up if maintainers want that behavior.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a Signal group with `requireMention: true`, a display-name mention pattern such as `@caio`, and a native mention whose metadata contains the bot UUID or phone; current main hydrates the placeholder to `@<uuid>` or `@<phone>`, the display-name regex does not match, and the message is skipped.

## Solution Assessment

Yes for the targeted bug: matching structured Signal mention metadata against the configured bot account inside the Signal plugin is the narrowest maintainable fix. The merge-ready version should add docs/changelog and leave the metadata-only gating policy as a deliberate follow-up decision.
