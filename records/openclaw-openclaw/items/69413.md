## Summary

Keep open. Current main still has the Signal bug the PR targets, and the proposed implementation is a focused Signal plugin fix with the earlier forged-metadata concern addressed by span validation. The remaining merge blocker is narrow docs/changelog follow-through for a user-visible behavior change, plus a maintainer scope call on whether native metadata alone should make mention detection available.

## What This Changes

The PR adds Signal native mention metadata detection against the bot account, wires it into group mention gating, and adds mention-gating coverage for uuid, phone, other-participant, and malformed-span cases.

## Best Possible Solution

Land the focused Signal plugin fix after adding the missing operator-facing docs and changelog entry, preserving the raw-message span validation and keeping any broader metadata-only mention-detection policy as an explicit follow-up if maintainers want that behavior.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a Signal group with `requireMention: true`, a display-name mention pattern such as `@caio`, and a native mention whose metadata contains the bot uuid or phone; current main hydrates the placeholder to `@<uuid>` or `@<phone>`, the display-name regex does not match, and the message is skipped.

## Solution Assessment

Yes for the targeted bug: matching structured Signal mention metadata against the configured bot account inside the Signal plugin is the narrowest maintainable fix. The merge-ready version should add docs/changelog and keep the metadata-only `canDetectMention` behavior as an explicit scope decision.
