## Summary

Keep open: this targets a real Codex native hook relay pressure problem, but the default systemd wrapper can fail before the OpenClaw relay runs and upstream Codex treats that as a failed hook rather than a policy block, so the PR needs repair plus real behavior proof before merge.

## What This Changes

The branch wraps Linux native hook relay commands in a user `systemd-run --scope` with CPU, memory, and task limits when a user runtime and `systemd-run` path are present, and adds a focused command-builder test.

## Best Possible Solution

Make the systemd wrapping fail-safe and viability-checked before default use, or keep the original relay command as the default while maintainers choose the broader concurrency or lightweight-relay design for the canonical fan-out issue.

## Reproduction Assessment

Yes for the PR-introduced defect: source inspection shows the wrapper is selected by path presence, while upstream Codex treats nonzero PreToolUse hook command exits as failed hooks without blocking. I did not run a live systemd user-scope failure reproduction.

## Solution Assessment

No as submitted: a systemd scope may be a plausible mitigation, but the default path needs viability checks, fail-safe fallback, and real proof before it is the best fix. The broader P1 fan-out issue may still need aggregate concurrency or a lightweight relay path.
