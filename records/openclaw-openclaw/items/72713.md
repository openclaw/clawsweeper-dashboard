## Summary

Keep open: the linked bug is still valid on current main, but this draft external PR is not ready because it removes macOS approvals-store serialization, lacks real behavior proof, and overlaps the narrower open implementation at https://github.com/openclaw/openclaw/pull/72650.

## What This Changes

The branch broadens Node and macOS exec-approvals path validation to allow trusted first-level `.openclaw`/state-dir symlinks, adds hardening tests, and documents the supported CLI approvals shape.

## Best Possible Solution

Land one security-reviewed implementation that allows only a validated first-level `.openclaw` symlink while preserving macOS file locking and final-file/deeper-symlink protections, then supersede the duplicate branch cleanly.

## Reproduction Assessment

Yes. Source inspection shows `openclaw exec-policy preset yolo` calls `saveExecApprovals()`, and current main's `ensureDir()` rejects symlinked `.openclaw`; the current store test also asserts that rejection.

## Solution Assessment

No. The trusted symlink direction is plausible, but this branch drops macOS file locking around security-sensitive approvals I/O, lacks contributor real behavior proof, and overlaps https://github.com/openclaw/openclaw/pull/72650.
