## Summary

Keep open: the PR addresses a real current-main bug, but it is draft, security-sensitive exec-approvals path work, still removes macOS approvals-store serialization, lacks the required changelog entry, and overlaps the narrower open #72650 implementation.

## What This Changes

The branch broadens Node and macOS exec-approvals path validation to allow trusted first-level `.openclaw` or state-directory symlinks, adds path-hardening tests, and documents the supported symlink shape.

## Best Possible Solution

Land one rebased, reviewed implementation that allows only the trusted first-level `.openclaw` symlink shape, preserves deeper-symlink and symlinked-file rejection, keeps macOS approvals file locking around validation plus I/O, updates docs and changelog, and then closes #72572 after merge.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction path: `openclaw exec-policy preset yolo` writes through `saveExecApprovals()`, reaches `ensureDir()`, and rejects `.openclaw` when that path component is a symlink; the current test suite also locks in that rejection.

## Solution Assessment

No, not as-is. The implementation direction is plausible, but this draft should preserve the current macOS file lock, add the required changelog entry, and be reconciled with #72650 before it is the safest maintainable fix.
