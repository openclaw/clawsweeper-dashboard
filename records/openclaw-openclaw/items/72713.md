## Summary

Keep open: the PR addresses a real, source-reproducible symlinked `.openclaw` exec-policy failure, but it is draft/conflicting, lacks contributor real behavior proof, overlaps another open implementation candidate, and introduces a blocking macOS approvals lock regression.

## What This Changes

The branch broadens Node and macOS exec-approvals path validation to allow trusted first-level `.openclaw` or state-dir symlinks, adds hardening tests, and documents the supported approvals layout.

## Best Possible Solution

Land one security-reviewed implementation that permits only a validated first-level `.openclaw` symlink while preserving macOS file locking and final-file, chained-symlink, and deeper-symlink protections.

## Reproduction Assessment

Yes, from source inspection. With `.openclaw` as a symlink under the resolved home directory, `openclaw exec-policy preset yolo` reaches `saveExecApprovals()`, and current main rejects the symlinked approvals directory before writing.

## Solution Assessment

No, not as-is. The trusted first-level symlink direction is plausible, but the final patch should preserve existing macOS `withFileLock` serialization and consolidate with the other open implementation candidate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86
