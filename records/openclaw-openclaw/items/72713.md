## Summary

Keep open: current main still reproduces the `~/.openclaw` symlink rejection, but this draft PR is not ready because it overlaps #72650 and regresses current macOS approvals-store serialization.

## What This Changes

The PR broadens Node and macOS exec-approvals path validation to allow trusted first-level `.openclaw` or state-dir symlinks, adds hardening tests, and documents the supported symlink shape.

## Best Possible Solution

Land one reviewed canonical fix for #72572 that allows only the trusted first-level `.openclaw` symlink shape, preserves current macOS locking and deeper-symlink rejection, and includes docs plus changelog coverage.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: `openclaw exec-policy preset yolo` writes through `saveExecApprovals()`, reaches `ensureDir()`, and rejects a symlinked `.openclaw` component below the trusted home root.

## Solution Assessment

No, not as-is. The trusted symlink direction is plausible, but this branch should preserve macOS approvals locking, add the required changelog entry, and be reconciled with #72650 before it is the safest fix.
