## Summary

Keep open: the underlying bug is still present on current main, but this draft PR is not ready to merge because it drops current macOS approvals-store serialization while changing security-sensitive symlink handling, and it also lacks the required changelog entry for a user-facing fix.

## What This Changes

The PR broadens Node and macOS exec-approvals path validation to allow trusted first-level `~/.openclaw` or state-directory symlinks, adds path-hardening tests, and documents the supported symlink shape.

## Best Possible Solution

Land one rebased, reviewed implementation that allows only the trusted first-level `.openclaw` symlink shape, preserves deeper-symlink and symlinked-file rejection, keeps macOS approvals file locking around validation plus I/O, updates docs and changelog, and then closes #72572 after merge.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: `openclaw exec-policy preset yolo` writes through `saveExecApprovals()`, reaches `ensureDir()`, and rejects `~/.openclaw` when that path component is a symlink; the current test suite still asserts that rejection.

## Solution Assessment

No, not as-is. The implementation direction is plausible, but this draft should preserve the current macOS file lock, add the required changelog entry, and be reconciled with #72650 before it is the safest maintainable fix.
