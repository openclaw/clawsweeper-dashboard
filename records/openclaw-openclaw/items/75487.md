## Summary

Keep open. Current main still computes interval heartbeat phase slots without active-hours or timezone input, checks `activeHours` only when the timer fires, and PR #75597 is the open unmerged implementation candidate.

## What This Changes

The issue requests heartbeat interval scheduling to honor the configured `activeHours.timezone` and window when choosing the next run time, rather than only skipping after a UTC-phase tick fires.

## Best Possible Solution

Land PR #75597 or an equivalent narrow scheduler change that seeks forward to the next phase-aligned slot inside `activeHours`, while preserving the runtime guard for targeted and manual wakes.

## Reproduction Assessment

Yes. A focused fake-timer scheduler test can configure `every: "4h"` with `activeHours` in `Asia/Shanghai`, start during quiet local hours, and observe current main arming or consuming quiet UTC-phase slots instead of the first in-window slot.

## Solution Assessment

Yes. Scheduler-level active-hours-aware phase seeking is the narrowest maintainable fix for the reported behavior; fixed local-time triggers and startup immediate-run semantics should remain separate feature requests.
