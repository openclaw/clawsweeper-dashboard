## Summary

Keep open. Current main still matches the reported scheduler bug: heartbeat phase selection is computed without active-hours or timezone input, then quiet-hour slots are skipped only after the timer fires. The issue is also paired with open PR #75597 using closing syntax, so cleanup should wait for that PR to merge or be closed.

## What This Changes

Request that heartbeat interval scheduling account for `heartbeat.activeHours.timezone` and window when choosing the next phase-aligned run, instead of firing quiet-hour slots and skipping afterward.

## Best Possible Solution

Land PR #75597 or an equivalent narrow scheduler change that preserves deterministic per-agent phase selection while seeking forward past quiet-hour slots. Keep the runtime `isWithinActiveHours` guard for manual and targeted wakes, and add focused coverage for explicit IANA timezones, overnight windows, invalid windows, and startup/config-reload scheduling.

## Reproduction Assessment

Yes. A fake-timer scheduler test can configure `every: "4h"` with `activeHours` in `Asia/Shanghai`, start during quiet local hours, and observe current main arming or consuming UTC-aligned quiet slots instead of scheduling directly to the first in-window slot.

## Solution Assessment

Yes. The narrowest maintainable fix is the report's option A at scheduler level; options B and C add new public scheduling or startup semantics and should remain separate enhancements.
