## Summary

Keep open. Current main still computes heartbeat phase slots without `activeHours` or timezone input, then applies `activeHours` only as a runtime skip guard; open PR #75597 already references this issue with closing syntax and is the implementation candidate.

## What This Changes

Make heartbeat interval scheduling choose phase-aligned ticks within the configured `activeHours` timezone instead of arming raw epoch-phase ticks that may be skipped later.

## Best Possible Solution

Review and land PR #75597, or an equivalent narrow scheduler change that seeks forward to the first phase-aligned slot inside `activeHours` while preserving the runtime guard for targeted/manual wakes.

## Reproduction Assessment

Yes. A focused helper or fake-timer scheduler test can configure `every: "4h"` with `activeHours.timezone: "Asia/Shanghai"`, choose a phase whose next slots are quiet locally, and show current main arming those quiet slots before `runHeartbeatOnce` skips them as `quiet-hours`.

## Solution Assessment

Yes. Active-hours-aware phase seeking in the scheduler is the narrowest maintainable fix for the reported behavior; fixed local-time schedules and immediate startup runs should remain separate feature requests.
