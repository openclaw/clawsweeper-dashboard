## Summary

Keep open. Current main still matches the reported behavior: heartbeat interval phase calculation is UTC-epoch based and active-hours timezone checks are applied as a runtime skip, not as part of selecting the next scheduled phase. Because open PR #75597 uses closing syntax for this issue and remains unmerged, the issue should stay open for that PR to be reviewed or replaced rather than closed by cleanup.

## What This Changes

The issue requests that heartbeat interval scheduling respect the configured `activeHours.timezone` and active window when choosing the next due time, instead of firing UTC-aligned phases and skipping outside the window at execution time.

## Best Possible Solution

Review and land PR #75597 or an equivalent narrow scheduler fix that preserves deterministic per-agent phase spreading while seeking forward to the first phase slot inside the configured active-hours window. Keep the runtime active-hours guard for manual or targeted wakes, and cover explicit IANA timezones, overnight windows, invalid windows, startup/config reload scheduling, docs, and changelog as appropriate.

## Reproduction Assessment

Yes. A focused fake-timer scheduler test can configure `every: "4h"` with `activeHours.timezone: "Asia/Shanghai"`, start during quiet local hours, and verify that current main schedules or consumes quiet UTC-aligned phase slots instead of arming the first in-window slot.

## Solution Assessment

Yes, the best fix direction is the report's option A at the scheduler level. Options B and C would add new public scheduling or startup semantics and should be treated as separate enhancements unless maintainers intentionally broaden the scope.
