## Summary

Keep open. Current `main` still schedules heartbeat intervals by deterministic UTC-epoch phase and only applies `activeHours` as a run-time skip, and the issue is already paired with open PR #75597 using closing syntax, so cleanup should wait for that PR to land or be closed.

## What This Changes

The issue asks heartbeat interval scheduling to honor `activeHours.timezone` and schedule the next phase inside the configured local active-hours window instead of firing UTC-aligned phase slots that are later skipped as quiet hours.

## Best Possible Solution

Land PR #75597 or an equivalent narrow scheduler change that preserves deterministic per-agent phase selection while seeking forward past quiet-hour phase slots. Keep `isWithinActiveHours` as the run-time guard for manual/targeted wakes and cover explicit IANA timezones, overnight/zero-width windows, startup/config reload scheduling, and quiet-hour interval skips with focused tests plus docs/changelog updates.

## Reproduction Assessment

Yes. A fake-timer scheduler test can configure `every: "4h"` with `activeHours.timezone: "Asia/Shanghai"`, start during local quiet hours, and observe current `main` scheduling or consuming UTC-aligned quiet slots instead of arming the first in-window slot.

## Solution Assessment

Yes. The narrowest maintainable fix is the report's option A at the scheduler level; options B and C add new public scheduling or startup-trigger semantics and should remain separate enhancements.
