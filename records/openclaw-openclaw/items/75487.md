## Summary

Keep open. Current main still uses `activeHours.timezone` only as an execution-time guard, while interval scheduling and phase advancement remain based on the deterministic UTC-millisecond phase, so quiet-hour slots can be armed and consumed before the next in-window heartbeat. The existing ClawSweeper comment reached the same narrow conclusion, and the broader phase/jitter request in #35799 does not supersede this active-hours bug.

## What This Changes

The issue requests that heartbeat `activeHours.timezone` and its local-time window determine the next scheduled heartbeat time instead of only filtering a UTC-phase tick after it fires.

## Best Possible Solution

Add a scheduler-level active-hours due-time helper that reuses the existing timezone/window semantics, preserves deterministic per-agent phase, and skips quiet-hour phase slots when computing `nextDueMs`; keep the run-time guard as a safety net and document the resulting behavior.

## Reproduction Assessment

Yes. A focused fake-timer scheduler test can start from a quiet local time with `every: "4h"` and `activeHours.timezone: "Asia/Shanghai"`, then verify current main arms a quiet UTC phase instead of the first phase inside the configured local window.

## Solution Assessment

Yes. The best narrow fix is option A from the report: make interval scheduling choose the next active-hours-compliant phase, while leaving fixed local-time phase configuration and startup-immediate behavior as separate product enhancements.

## Repair Work Prompt

Fix the heartbeat active-hours scheduler bug reported in this issue narrowly. Current heartbeat scheduling computes deterministic interval phases without considering `heartbeat.activeHours`, then `runHeartbeatOnce` skips quiet-hour ticks and the runner advances to the next phase. Preserve deterministic per-agent phase, but when computing interval `nextDueMs` for configured active hours, skip phase slots outside the configured local-time window and arm the timer for the first phase inside that window. Reuse or share the existing `isWithinActiveHours` semantics so normal, overnight, invalid timezone fallback, zero-width, and full-day windows behave consistently. Keep the existing `runHeartbeatOnce` active-hours guard for targeted/manual wakes and safety. Do not add local fixed-time phase configuration, startup-immediate heartbeat behavior, or broader #35799 public jitter/prime controls in this PR. Add focused tests for Asia/Shanghai normal windows, overnight windows, invalid/full-day behavior, and schedule advancement after a quiet-hour skip. Update `docs/gateway/heartbeat.md` wording if needed and add a single-line user-facing `CHANGELOG.md` fix entry crediting `Thanks @raydoomed`. Validate with the targeted heartbeat tests and `pnpm check:changed` via the normal maintainer/Testbox path.
