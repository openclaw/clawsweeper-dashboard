## Summary

Keep open. Current main still matches the reported bug: active-hours timezone logic is only an execution guard, while interval phase selection remains timezone/window-agnostic. The issue is also paired with open PR #75597 using closing syntax, so the right cleanup path is to review and land or close that PR, not close this issue early.

## What This Changes

The issue asks for heartbeat `activeHours.timezone` to affect the next scheduled heartbeat phase instead of only skipping executions after UTC-aligned phase ticks fire.

## Best Possible Solution

Land PR #75597 or an equivalent narrow scheduler change that preserves deterministic per-agent phase selection while seeking forward past quiet-hour phase slots. Keep the runtime `isWithinActiveHours` guard for manual and targeted wakes, and cover explicit IANA timezones, overnight windows, invalid windows, and startup/config-reload scheduling with focused tests plus docs/changelog updates.

Yes. A fake-timer scheduler test can configure `every: "4h"` with `activeHours` in `Asia/Shanghai`, start during quiet local hours, and observe that current main arms or consumes quiet UTC-aligned phase slots instead of scheduling directly to the first in-window slot.

Yes. The narrowest maintainable fix is option A from the report, implemented at scheduler level; options B and C add new public scheduling/startup semantics and should remain separate enhancements.
