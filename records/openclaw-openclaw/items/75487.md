## Summary

Keep open. Current `main` still computes heartbeat interval due times without `activeHours` or timezone input, then applies `activeHours` only as a runtime skip; the issue is also paired with open PR #75597 using closing syntax, so cleanup should wait for that PR to land or be closed.

## What This Changes

The issue asks heartbeat interval scheduling to account for `activeHours.timezone` and the configured local window when choosing the next phase-aligned run, instead of firing quiet-hour slots and skipping them afterward.

## Best Possible Solution

Land PR #75597 or an equivalent narrow scheduler change that preserves deterministic per-agent phase selection while seeking forward past phase-aligned slots outside `activeHours`; keep the runtime guard for manual and targeted wakes, and add focused coverage for IANA timezones, overnight windows, invalid windows, startup, and config reload scheduling.

## Reproduction Assessment

Yes. A focused fake-timer scheduler test can configure `every: "4h"` with `activeHours` in `Asia/Shanghai`, start during quiet local hours, and observe current main arming or consuming quiet UTC-aligned phase slots instead of scheduling directly to the first in-window slot.

## Solution Assessment

Yes. The scheduler-level active-hours-aware phase seek described by the paired PR is the narrowest maintainable direction; local-time phase configuration and immediate startup trigger behavior would be separate public scheduling features.
