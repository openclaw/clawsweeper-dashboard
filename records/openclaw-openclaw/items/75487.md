## Summary

Keep open. Current main still computes interval heartbeat phase slots without active-hours or timezone input, applies `activeHours` only as a runtime quiet-hours guard, and the issue is paired with open implementation PR #75597 using closing syntax.

## What This Changes

Heartbeat active-hours configuration should affect the next interval wake time in the configured timezone instead of only skipping runs after a UTC-phase timer fires.

## Best Possible Solution

Land PR #75597 or an equivalent narrow scheduler change that seeks forward to the next phase-aligned slot inside `activeHours` while preserving the runtime guard for targeted and manual wakes.

## Reproduction Assessment

Yes. A focused fake-timer scheduler test can configure `every: "4h"` with `activeHours` in `Asia/Shanghai`, start during quiet local hours, and observe current main arming or consuming quiet UTC-phase slots instead of scheduling directly to the first in-window slot.

## Solution Assessment

Yes. Scheduler-level active-hours-aware phase seeking is the narrowest maintainable fix for the reported behavior; fixed local-time triggers and immediate startup runs should remain separate feature requests.
