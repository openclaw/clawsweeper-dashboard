## Summary

Keep open. Current main honors `activeHours.timezone` only as a run-time guard, while the interval scheduler still computes and advances deterministic phase ticks without considering the active-hours window; #35799 is related broader phase/jitter work but does not supersede this narrower scheduling bug.

## What This Changes

The issue requests that heartbeat interval scheduling honor `heartbeat.activeHours.timezone` when choosing the next scheduled tick, instead of only skipping runs after UTC-aligned phase ticks fire.

## Best Possible Solution

Add a scheduler-level path that preserves the existing deterministic per-agent phase but skips phase slots that fall outside `activeHours` in the resolved timezone. Keep `runHeartbeatOnce`'s active-hours guard for manual and targeted wakes, and leave local fixed-time phases or startup-immediate behavior as separate enhancements.

## Reproduction Assessment

Yes. The current code path is enough to reproduce with a focused fake-timer scheduler test using `every: "4h"` plus `activeHours` in `Asia/Shanghai`: the timer arms the next stable phase without active-hours filtering, then `runHeartbeatOnce` skips it as `quiet-hours`.

## Solution Assessment

Yes, option A is the best narrow fix. Computing the next active-hours-compliant phase in the scheduler matches the docs and avoids adding new public phase-alignment or startup-immediate semantics for this bug.
