## Summary

Keep open. Current main still computes heartbeat interval phase slots without `activeHours` or timezone input, applies active-hours only as a runtime quiet-hours guard, and PR #75597 is the open closing implementation candidate.

## What This Changes

The issue asks for heartbeat `activeHours.timezone` to influence the next scheduled phase slot instead of only skipping runs after UTC-aligned timers fire.

## Best Possible Solution

Review and land PR #75597 or an equivalent narrow scheduler fix that seeks to the next phase-aligned slot inside `activeHours` while preserving the runtime guard for targeted and manual wakes.

## Reproduction Assessment

Yes. A focused fake-timer scheduler test can configure `every: "4h"` with `activeHours.timezone: "Asia/Shanghai"`, start during quiet local hours, and observe current main arming or consuming UTC-phase quiet slots.

## Solution Assessment

Yes. Scheduler-level active-hours-aware phase seeking is the narrowest maintainable fix for the reported behavior; fixed local-time triggers and immediate startup runs should remain separate feature requests.
