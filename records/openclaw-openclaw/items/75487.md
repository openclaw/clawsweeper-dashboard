## Summary

Keep open. Current main validates `activeHours.timezone` when deciding whether a heartbeat run may execute, but the scheduler still computes and advances stable interval phases without considering the active-hours window, so quiet-hour phase ticks can be consumed before the first in-window run. This is related to broader heartbeat phase controls in #35799, but it is a narrower active-hours scheduling bug rather than a duplicate.

## What This Changes

The issue requests that heartbeat `activeHours.timezone` and its local-time window determine the next scheduled heartbeat phase instead of only filtering execution after a UTC-based phase tick fires.

## Best Possible Solution

Add a scheduler-level helper that preserves the existing deterministic phase but skips quiet-hour phase slots when computing `nextDueMs`, while keeping `runHeartbeatOnce`'s active-hours guard for manual/targeted wakes. Cover normal, overnight, invalid, and explicit-IANA timezone windows with focused scheduler tests and update heartbeat docs/changelog to describe the behavior precisely.

## Reproduction Assessment

Yes. A focused fake-timer scheduler test can configure `every: "4h"` with `activeHours` in `Asia/Shanghai`, start from a quiet local time, and show that current main arms/calls the quiet UTC phase before advancing instead of scheduling directly to the first in-window phase.

## Solution Assessment

Yes for a narrow fix: make the interval scheduler compute the next active-hours-compliant phase and leave local fixed-time phase or startup-immediate options as separate enhancements. That is smaller and less risky than adding new public scheduling fields for this report.
