## Summary

Keep open. Current `main` still computes heartbeat phase slots without active-hours input and only applies `activeHours` as a runtime skip guard; GitHub currently shows PR #75597 open as the closing implementation candidate. ŅłĆciteŅłéturn0view0Ņłéturn0view1Ņłü

## What This Changes

The issue asks heartbeat interval scheduling to respect `activeHours.timezone` when choosing the next phase-aligned tick instead of waking during quiet local hours and skipping afterward.

## Best Possible Solution

Review and land PR #75597, or an equivalent narrow scheduler change that seeks forward to the next phase-aligned slot inside `activeHours` while preserving the runtime guard for targeted wakes.

## Reproduction Assessment

Yes. A focused fake-timer scheduler test can configure `every: "4h"` with `activeHours.timezone: "Asia/Shanghai"`, start during quiet local hours, and observe current `main` consuming quiet UTC-phase slots instead of arming the first in-window slot.

## Solution Assessment

Yes. Scheduler-level active-hours-aware phase seeking is the narrowest maintainable fix for the reported behavior; fixed local-time triggers and immediate startup runs should remain separate feature decisions.
