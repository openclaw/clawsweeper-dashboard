## Summary

Keep open. Current main still computes interval heartbeat phase slots without `activeHours` or timezone input, applies active-hours only as a runtime skip guard, and the issue is paired with open PR #75597 using closing syntax.

## What This Changes

Request heartbeat interval scheduling to choose the next phase-aligned tick inside the configured `activeHours` timezone instead of waking on UTC-phase slots that are later skipped.

## Best Possible Solution

Review and land PR #75597 or an equivalent narrow scheduler change that seeks to the first phase-aligned in-window slot while preserving targeted and manual wake cooldown behavior.

## Reproduction Assessment

Yes. A focused fake-timer scheduler test can configure `every: "4h"` with `activeHours` in `Asia/Shanghai`, start during quiet local hours, and observe current main arming or consuming quiet UTC-phase slots before the first in-window slot.

## Solution Assessment

Yes. Scheduler-level active-hours-aware phase seeking is the narrowest maintainable fix for the reported behavior; fixed local-time triggers and immediate startup runs should remain separate feature requests.
