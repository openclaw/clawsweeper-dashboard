## Summary

Keep open. Current main still computes heartbeat interval phase slots without active-hours or timezone input, applies `activeHours` only as a runtime guard, and open PR #75597 is already the closing implementation candidate.

## What This Changes

Heartbeat interval scheduling should account for the configured `activeHours` timezone/window when choosing the next phase-aligned run.

## Best Possible Solution

Review and land PR #75597, or an equivalent narrow scheduler change that seeks forward to the next phase-aligned slot inside `activeHours` while preserving the runtime guard for targeted/manual wakes.

## Reproduction Assessment

Yes. A focused fake-timer scheduler test can configure `every: "4h"` with `activeHours` in `Asia/Shanghai`, start during quiet local hours, and observe current main scheduling or consuming quiet UTC-phase slots instead of arming the first in-window slot.

## Solution Assessment

Yes. Scheduler-level active-hours-aware phase seeking is the narrowest maintainable fix for the reported behavior; fixed local-time triggers and immediate startup runs should stay separate feature requests.
