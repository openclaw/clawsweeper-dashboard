## Summary

Keep open. Current main still computes heartbeat interval phases without active-hours or timezone input, then applies `activeHours` only as an execution-time quiet-hours guard, so the reported UTC-phase consumption behavior remains a source-backed scheduler bug rather than an implemented or duplicate cleanup item.

## What This Changes

The issue requests that heartbeat `activeHours.timezone` affect the next scheduled heartbeat time instead of only skipping runs after UTC-aligned interval ticks fire.

## Best Possible Solution

Add scheduler-level active-hours filtering that preserves the existing deterministic per-agent phase but computes `nextDueMs` as the next phase inside the configured local window, while keeping the execution-time quiet-hours guard as a backstop. Document the resulting behavior and cover explicit timezone, overnight window, invalid config, and unchanged no-active-hours cases.

## Reproduction Assessment

Yes. A fake-timer scheduler test can start from a quiet local time with `every: "4h"` and `activeHours.timezone: "Asia/Shanghai"`, then show current main arms or consumes quiet UTC-aligned phase ticks instead of scheduling directly to the first in-window phase.

## Solution Assessment

Yes for the issue's option A. Adjusting next-due calculation is narrower than adding new public fixed-time phase fields or startup-immediate semantics, and it matches the existing docs promise for active-hours behavior.
