## Summary

Keep open: current main has generic Gateway startup tracing and coarse ACPX registration/probe logs, but no ACPX-specific subphase timings; this is a new observability/logging decision rather than a closeable implemented or duplicate issue.

## What This Changes

Add structured ACPX startup subphase timing logs or spans so a cold-start delay can be localized.

## Best Possible Solution

Extend the existing startup trace/diagnostics path, or add explicit ACPX phase logs if maintainers want them always visible, covering config/auth prep, state directory setup, stale lease reap, runtime construction, registration, and optional probe/doctor timing with focused regression tests.

## Reproduction Assessment

No high-confidence live reproduction was established for the reported 58-second Windows cold-start gap. Source inspection does give medium-confidence proof that current main lacks ACPX-specific subphase timing between ACPX service start and backend registration.

## Solution Assessment

Unclear. Adding ACPX startup instrumentation is plausible and likely useful, but maintainers should decide whether it belongs in default logs, the existing startup trace/diagnostics timeline, or both before an automated cleanup PR changes logging behavior.
