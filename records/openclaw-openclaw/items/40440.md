## Summary

Keep open. Current main still records skipped Telegram group media history as text-only entries and later builds media context only from the current message plus reply media, while the two related fix attempts at https://github.com/openclaw/openclaw/pull/40470 and https://github.com/openclaw/openclaw/pull/40490 are closed unmerged.

## What This Changes

The issue asks Telegram group history to preserve media paths and types for no-mention media messages so a later bot mention can access those attachments.

## Best Possible Solution

Add an additive media-bearing pending-history contract, preserve skipped Telegram media for a bounded safe lifetime, filter missing refs, and merge surviving history media into later mention context.

## Reproduction Assessment

Yes, source-level. In a Telegram requireMention group, downloaded allMedia reaches resolveTelegramInboundBody, the no-mention skip branch records only rawBody history, and later context construction builds MediaPaths only from current or replied-to media.

## Solution Assessment

Yes for a narrow repair. The maintainable path is optional mediaPaths/mediaTypes on pending history plus bounded retention and surviving-path filtering, implemented on the current extensions/telegram paths rather than reviving the closed legacy PR branches as-is.

## Repair Work Prompt

Fix Telegram group media history for requireMention/no-mention messages. When Telegram downloads media for a group message that is skipped because the bot was not mentioned, preserve media path/type refs in the pending history entry and replay surviving refs into the later mention context so MediaPaths/MediaTypes include the recent history media. Keep the shared contract additive, avoid a new config flag, and do not move owner-specific behavior out of the Telegram plugin. Handle retention safely: no unbounded cache growth, no unbounded fs.utimes burst, no stale/missing path injection, and preserve mediaTypes index alignment where it matters. Add focused regression coverage around the skipped group message path, later replay context, inbound history shape, and media retention/filtering. Add a user-facing changelog entry if the fix changes shipped behavior.
