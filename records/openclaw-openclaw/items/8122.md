## Summary

Current main and v2026.6.5 still cannot express or resolve a Telegram group/topic-specific reset policy. The closed implementation PR did not land, and the open agent/binding request overlaps the policy design but does not fully cover this transport conversation scope, so maintainers should keep this issue open for a canonical product decision.

## What This Changes

Allow selected Telegram groups or topics to override global, chat-type, and channel session reset policy.

## Best Possible Solution

Define one prepared, account-aware conversation reset override and documented precedence model, then let approved Telegram group/topic and agent/binding owners supply that override before the existing reset resolver.

## Reproduction Assessment

Yes, at source level with high confidence: current main's strict Telegram schemas, configuration types, reset-policy caller, resolver, tests, and release code all lack a group/topic-specific policy path. This is a missing capability rather than a runtime failure that needs execution to demonstrate.

## Solution Assessment

Unclear as proposed: the nested Telegram setting is plausible, but the best durable solution must reconcile it with https://github.com/openclaw/openclaw/issues/60659 and establish one account-aware precedence contract rather than parallel policy systems.
