## Summary

The requested group-specific lifecycle is still missing on current main, but the open generic policy request at https://github.com/openclaw/openclaw/issues/60659 now owns the same remaining work and can represent an exact Telegram group through the existing channel, account, and peer binding matcher.

## What This Changes

Allow selected Telegram groups to use a different session reset policy from other groups and channels.

## Best Possible Solution

Continue the work in https://github.com/openclaw/openclaw/issues/60659 by adding one canonical binding- or prepared-route reset policy with exact account/peer precedence, then document this Telegram group example there.

## Reproduction Assessment

Yes at the source-contract level: current main has only global, chat-type, and channel reset overrides, with no peer- or binding-specific input. This demonstrates a missing capability rather than a regression in documented behavior.

## Solution Assessment

No, the proposed Telegram-only nested configuration is not the best long-term source of truth. Extending the existing account- and peer-aware binding policy avoids a competing channel-specific lifecycle path and also covers the canonical multi-agent use cases.
