## Label Justifications

- `impact:session-state`: The missing scope can retain unrelated conversation context and cause session-state contamination in groups intended for independent tasks.

## Summary

Keep open. Current main still has no per-Telegram-group or per-topic reset setting, and the broader per-agent/per-binding request overlaps but does not yet establish the product decision or exact configuration contract needed for this group-scoped use case.

## What This Changes

Add per-group, and potentially per-topic, session reset policy so selected Telegram conversations can use shorter or one-shot session lifecycles without changing other groups.

## Best Possible Solution

Define a single account- and route-aware conversation reset-policy contract with explicit precedence, then expose Telegram group/topic settings through it or make binding policy the documented way to cover this use case.

## Reproduction Assessment

Yes at the source level. Current strict Telegram schemas reject the proposed group/topic reset field, and the runtime reset path has no conversation-specific configuration input.

## Solution Assessment

No, a Telegram group-ID-only helper is not the best fix. The maintainable solution is an account-aware conversation or binding policy seam with a documented precedence order and focused multi-account tests.
