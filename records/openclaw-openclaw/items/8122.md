## Summary

Current main still lacks any group-, topic-, binding-, agent-, or conversation-specific reset policy, while the linked implementation remains closed unmerged. The request is still necessary, but the durable owner boundary requires a maintainer product decision: the original Telegram-only shape does not safely cover multi-account routing and overlaps the broader per-agent or per-binding request.

## What This Changes

Add a per-Telegram-group session reset policy so selected one-shot groups expire context independently of other conversations.

## Best Possible Solution

Define one account-aware prepared conversation or binding reset override at the shared session initializer, then map Telegram group/topic or generic binding configuration into it with explicit precedence over channel, type, and global policy.

## Reproduction Assessment

Yes, at high confidence from source inspection: Telegram group/topic configuration has no reset field, and session initialization consumes only global, type, and channel policy.

## Solution Assessment

No, the originally proposed Telegram-only lookup is not the best final design because it is not account-aware and may duplicate the broader agent or binding policy surface; a prepared shared override with owner-specific configuration mapping is safer.
