## Summary

Keep open, but not mergeable as-is: the proposed acknowledgment runs before Feishu ingress checks, bypasses established typing lifecycle/config semantics, and has no real Feishu proof; the native typing direction is also tracked separately.

## What This Changes

The PR adds an immediate Feishu `Typing` message reaction in `handleFeishuMessage()` after receipt logging and before routing or policy checks.

## Best Possible Solution

If immediate acknowledgement remains desired, implement it as a managed post-eligibility Feishu signal that respects `typingIndicator` and cleanup, or align the work with the native typing/status path.

## Reproduction Assessment

Yes for source-level review: the PR diff places the reaction before current-main group, DM, mention, sender, and empty-message return paths. I did not run live Feishu/Lark verification.

## Solution Assessment

No. A direct receipt-time reaction is not the narrowest maintainable solution because it bypasses ingress policy, the existing `typingIndicator` setting, and managed cleanup; a post-eligibility lifecycle hook or native typing API is safer.
