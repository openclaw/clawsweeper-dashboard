## Summary

Keep open. The PR targets a real Feishu UX gap, but the patch emits a visible `Typing` reaction before ingress checks, bypasses the existing typing lifecycle/config, and lacks real Feishu/Lark behavior proof.

## What This Changes

This PR adds an immediate Feishu `Typing` message reaction to inbound messages in `handleFeishuMessage()` before routing and policy checks.

## Best Possible Solution

If immediate acknowledgement remains desired, implement it as a managed post-eligibility Feishu signal that respects `typingIndicator` and cleanup, or defer the work to the native typing/status path tracked separately.

## Reproduction Assessment

Yes. The PR diff places the reaction call immediately after receipt logging, while current main has later returns for disabled/disallowed groups, sender allowlists, unmentioned group messages, disallowed DMs, and empty-message skips.

## Solution Assessment

No. A direct receipt-time reaction is not the narrowest maintainable fix because it bypasses ingress policy, the existing `typingIndicator` setting, and cleanup state; a post-eligibility managed signal or confirmed native typing API is safer.
