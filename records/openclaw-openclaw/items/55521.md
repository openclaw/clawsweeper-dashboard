## Summary

Keep open. The PR still has blocking correctness issues: the proposed acknowledgment reaction would run before Feishu group, sender, and mention eligibility gates, and it bypasses the existing typing setting and cleanup lifecycle. The earlier Greptile and ClawSweeper comments are consistent with the current-main code review, and the linked #69572 leaves a real product/API question about adding more reaction-based typing behavior.

## What This Changes

The PR adds an unconditional Feishu `im.messageReaction.create` call in `handleFeishuMessage()` to add a `Typing` reaction immediately after inbound message logging.

## Best Possible Solution

If immediate Feishu feedback is still desired, put it behind the same reply-eligibility decision as dispatch, reuse or extend the managed typing lifecycle so reactions are removed, respect `typingIndicator`, and add focused tests for disabled groups, group/sender allowlists, mention-required groups, and disabled typing indicators while resolving the native-typing question in #69572.

## Reproduction Assessment

Yes. The issue is reproducible by static path inspection: the PR inserts the reaction before current-main early returns for group policy, allowlists, and mention gating, and existing tests show those paths intentionally avoid dispatch/reply.

## Solution Assessment

No. A direct unconditional reaction in `handleFeishuMessage()` is not the narrowest maintainable solution because Feishu typing behavior already has config, age, keepalive, and cleanup semantics in the reply lifecycle; the safer path is to integrate there after eligibility is known or wait for the native typing API decision.
