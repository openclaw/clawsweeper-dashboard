## Summary

Keep open. The branch is a focused Feishu UX feature, but the diff emits a visible `Typing` reaction before Feishu ingress checks, bypasses the existing typing toggle and cleanup lifecycle, and still lacks real Feishu/Lark proof.

## What This Changes

The branch adds a direct Feishu `Typing` message-reaction API call in `handleFeishuMessage()` immediately after receipt logging.

## Best Possible Solution

Implement any immediate Feishu acknowledgement as a managed post-eligibility signal that respects `typingIndicator`, cleans up after the reply, and is aligned with the native typing/status decision.

## Reproduction Assessment

Yes for source-level review: the PR diff places the reaction before current-main group, DM, mention, sender, and empty-message return paths. I did not run live Feishu/Lark verification.

## Solution Assessment

No. A direct receipt-time reaction is not the narrowest maintainable solution because it bypasses ingress policy, the existing `typingIndicator` setting, and managed cleanup; a post-eligibility lifecycle hook or native typing API is safer.
