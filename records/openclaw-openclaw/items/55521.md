## Summary

Keep open; this PR is not merge-ready because it sends a visible Feishu acknowledgement before admission gates, ignores the existing typing toggle, bypasses the managed typing lifecycle, and still lacks real Feishu/Lark proof.

## What This Changes

The PR adds an unconditional Feishu `messageReaction.create` Typing emoji acknowledgement in `handleFeishuMessage` immediately after message receipt logging.

## Best Possible Solution

Implement any Feishu acknowledgement only after admission gates, behind `typingIndicator`, through the managed typing lifecycle, after maintainer direction on reaction-based versus native typing, with focused tests and live Feishu/Lark proof.

## Reproduction Assessment

Yes, at source level. The diff inserts the reaction before existing disabled/disallowed/unmentioned group returns and before DM admission, though no live Feishu tenant run was performed in this read-only review.

## Solution Assessment

No. A raw early reaction is not the safest solution; the maintainable path needs admission gating, `typingIndicator` semantics, managed cleanup/backoff behavior, and maintainer direction on reaction versus native typing.
