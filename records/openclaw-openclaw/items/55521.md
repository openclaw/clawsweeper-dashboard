## Summary

Keep open for contributor/maintainer follow-up: the PR still emits a visible Feishu `Typing` reaction before current-main ingress gates and bypasses the managed typing lifecycle, and it has no real Feishu proof.

## What This Changes

This PR adds an immediate Feishu `Typing` message reaction in `handleFeishuMessage()` after receipt logging and before routing or mention checks.

## Best Possible Solution

If immediate acknowledgement remains desired, implement it as a managed post-eligibility Feishu signal that respects `typingIndicator` and cleanup, or defer it to the native typing/status work tracked at https://github.com/openclaw/openclaw/issues/69572.

## Reproduction Assessment

Yes. The PR diff places the reaction call immediately after receipt logging, while current main has later returns for disabled/disallowed groups, sender allowlists, unmentioned group messages, disallowed DMs, and empty-message skips.

## Solution Assessment

No. A direct receipt-time reaction is not the narrowest maintainable fix because it bypasses ingress policy, the existing `typingIndicator` setting, and cleanup state; a post-eligibility managed signal or confirmed native typing API is safer.
