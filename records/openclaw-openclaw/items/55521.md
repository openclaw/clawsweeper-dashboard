## Summary

Keep open: the PR is not safe to merge as written because the new Feishu reaction runs before existing no-reply gates, bypasses the documented typing toggle, and is not tied into the managed cleanup lifecycle. The open native-typing issue #69572 also means maintainers should decide whether to continue reaction-based typing before repairing this PR.

## What This Changes

The PR adds a direct Feishu `Typing` `messageReaction.create` call immediately after the received-message log in `handleFeishuMessage()`.

## Best Possible Solution

If immediate Feishu feedback remains desired, implement it through a Feishu-owned managed typing path only after reply eligibility is known, with cleanup and `typingIndicator` support, or replace the reaction approach with the confirmed native typing/status API tracked by #69572.

## Reproduction Assessment

Yes. Applying the PR diff makes the reaction call execute before current-main early returns for disabled groups, allowlist failures, unmentioned mention-required groups, and unauthorized DMs; current tests cover several of those no-dispatch paths.

## Solution Assessment

No. An unconditional reaction in `handleFeishuMessage()` is not the narrowest maintainable solution because Feishu typing already has a managed lifecycle with config, stale-message suppression, state, and cleanup semantics, and #69572 may require a native typing API instead.
