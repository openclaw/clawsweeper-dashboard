## Summary

Keep open. The PR is a focused Feishu UX feature, but the current patch sends a visible Typing reaction before admission gates, ignores the existing typingIndicator contract, bypasses the managed reaction lifecycle, and still lacks real Feishu/Lark proof.

## What This Changes

The PR adds a direct Feishu `im.messageReaction.create` call that places a `Typing` reaction on inbound messages immediately after receipt logging.

## Best Possible Solution

Implement any Feishu acknowledgement only after dispatch eligibility is known, behind `typingIndicator`, through the managed lifecycle or a maintainer-approved native typing path, with focused tests and real Feishu/Lark proof.

## Reproduction Assessment

Yes, at source level. The PR diff inserts the reaction before current-main disabled/disallowed/unmentioned group returns and before DM admission, though no live Feishu tenant run was performed.

## Solution Assessment

No. A raw early reaction is not the safest solution; the maintainable path needs admission gating, existing `typingIndicator` semantics, managed cleanup/backoff behavior, and maintainer direction on reaction versus native typing.

  - confidence: 0.86
