## Summary

Keep this open: current main and v2026.6.8 still drive due inferred-commitment delivery from stored metadata without pre-send source-context revalidation, while the visible implementation candidate remains open and conflicting.

## What This Changes

The issue requests revalidating inferred open-loop commitments against current conversation/session/artifact context before sending visible follow-ups.

## Best Possible Solution

Land or replace the candidate with a maintainer-approved revalidation path that suppresses completed or superseded inferred commitments before visible delivery and preserves enough context for any follow-up that is sent.

## Reproduction Assessment

Yes. I did not run a fresh live Slack or Telegram repro, but current source still selects due commitments from stored state, builds the visible prompt from metadata including suggestedText, and disables tools/source replay for the delivery turn.

## Solution Assessment

Yes as a direction, not as current implementation. Pre-send revalidation and dismissal in the commitment/heartbeat/session-state path is the maintainable fix; current main lacks it, and https://github.com/openclaw/openclaw/pull/81382 is visible but still conflicting.
