## Label Justifications

- `P2`: A shipped opt-in core follow-up feature can send stale user-facing messages, but the blast radius is limited to inferred commitments.
- `impact:session-state`: The report is about stale commitment/session state being treated as current context after later conversation or artifact state changed.
- `impact:message-loss`: The stale standalone follow-up can create confusing visible messages and replies without enough current conversation context.

## Summary

Keep this open: current main and the latest release still deliver due inferred commitments from stored metadata without replaying source context or tools, and the direct implementation candidate is still open rather than merged.

## What This Changes

The issue requests a revalidation-first commitment delivery flow so inferred open-loop follow-ups are suppressed or dismissed when newer conversation, artifact, memory, or session state shows the loop is completed or superseded.

## Best Possible Solution

Land or replace a maintainer-approved commitment revalidation path that suppresses completed or superseded inferred follow-ups before visible delivery and preserves enough reply context for any follow-up that is sent.

## Reproduction Assessment

Yes. Source inspection shows due commitments are selected from stored records, rendered into a metadata-only prompt, and run with tools disabled; the issue comments add live Slack and Telegram recurrence evidence.

## Solution Assessment

Yes as a direction, but it is not implemented on main. Pre-send revalidation and suppression is the right boundary; https://github.com/openclaw/openclaw/pull/81382 is the visible candidate but still needs refresh and maintainer product review.
