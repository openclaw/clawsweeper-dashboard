## Label Justifications

- `P1`: The report describes a fresh regression duplicating scheduled automation side effects and visible Telegram deliveries for real cron users.
- `impact:message-loss`: The affected behavior duplicates outbound Telegram messages and other channel-visible work from one scheduled task.
- `impact:session-state`: One scheduled run identity can produce multiple attempts and conflicting cleanup/success terminal states.
- `impact:auth-provider`: The effective provider/model can flip from the configured non-Codex payload model to the Codex primary/fallback route.

## Summary

Keep open: current main still has a source-backed path where scheduled cron payload-model runs use configured fallback chains after an attempt-level failure, and the reported cleanup-after-side-effects case is not covered by existing cron fallback tests. The exact MiniMax scheduled-only cleanup failure still needs live or focused timer-path proof before maintainers choose the safest policy change.

## What This Changes

The issue reports scheduled isolated cron jobs with a non-Codex `payload.model` being replayed on the Codex primary/fallback model after a cleanup terminal state, duplicating Telegram messages and other side effects.

## Best Possible Solution

Make scheduled cron fallback side-effect-aware: keep configured fallback chains for pre-reply/provider failures, but treat a normal assistant stop or committed delivery/file/tool side effects as terminal for that scheduled run, with timer-path regression coverage for a non-Codex payload model and Codex fallback.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does show the relevant current-main path: scheduled isolated cron jobs honor `payload.model`, pass configured fallbacks to `runWithModelFallback`, and a post-output cleanup rejection can advance to a later Codex candidate.

## Solution Assessment

Unclear; disabling all payload-model fallback would contradict current docs and tests. The safer solution is side-effect-aware terminal handling that preserves fallback for clean provider failures before any user-visible or stateful work is committed.
