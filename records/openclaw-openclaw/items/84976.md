## Summary

Keep open: the report is fresh, specific, and still plausible on current main. Source inspection confirms scheduled isolated cron jobs honor `payload.model` first while keeping configured fallbacks, so a Codex primary/fallback can be reached after a non-terminal first attempt; the exact `minimax -> cleanup -> codex -> success` scheduled-only path still needs live or focused regression proof before changing fallback semantics.

## What This Changes

The issue reports that scheduled isolated cron `agentTurn` jobs using a non-Codex `payload.model` can execute once on the requested model, end as `cleanup`, then replay on the Codex primary model and duplicate side effects.

## Best Possible Solution

Define a side-effect-aware cron execution contract that treats a normal assistant stop or committed delivery/file/tool side effect as terminal for that scheduled run, while preserving fallback for clean pre-side-effect provider failures.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection confirms the relevant current-main ingredients: scheduled jobs enter the timer path, payload-model cron runs keep configured fallbacks, and fallback execution can advance to a later Codex route after a non-terminal first attempt.

## Solution Assessment

Unclear; the best fix depends on the intended contract for cron payload-model overrides after side effects. The safer direction is to suppress fallback replay only after work has been committed or the assistant stopped normally, while retaining fallback for clean provider failures.
