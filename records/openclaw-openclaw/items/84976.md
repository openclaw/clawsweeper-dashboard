## Summary

Keep open: the report is fresh, specific, and still plausible on current main because cron payload-model runs can still use the configured fallback chain, including a Codex primary/fallback, after the first provider attempt fails or is treated as non-terminal. The exact MiniMax scheduled-only cleanup status still needs live or focused regression proof before choosing the safest fix.

## What This Changes

The issue reports scheduled cron agent turns with a non-Codex `payload.model` being re-executed on the agent's Codex primary model, causing duplicate Telegram sends and repeated file/state side effects.

## Best Possible Solution

Define a cron execution contract that treats an attempt with committed side effects or a normal assistant stop as terminal for that scheduled run, or otherwise blocks fallback replay after side effects; cover the scheduled timer path with a non-Codex payload model and Codex primary/fallback regression test.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection does confirm the relevant current-main path: scheduled isolated cron jobs honor `payload.model` first, then can still enter `runWithModelFallback` with configured fallbacks that may include the Codex model described in the report.

## Solution Assessment

Unclear; the narrowest fix depends on whether maintainers want `payload.model` cron jobs to keep configured fallbacks after side effects. A safer direction is to prevent fallback replay once an attempt has committed delivery or file/tool side effects, while preserving intentional fallback behavior for clean pre-side-effect failures.
