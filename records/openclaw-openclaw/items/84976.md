## Summary

Keep open: the issue describes a credible current regression in scheduled cron execution where a non-Codex payload-model run can complete useful work, end as cleanup, and then replay through the configured Codex fallback, duplicating side effects. Source inspection on current main still shows cron payload-model jobs are run through the model fallback path, and the reporter’s scheduled-run evidence narrows the failure to timer-fired runs rather than the debug force-run path.

## What This Changes

Scheduled isolated cron jobs with a non-Codex `payload.model` should execute once on that configured model and avoid replaying completed side-effecting work on the agent’s Codex primary or fallback model.

## Best Possible Solution

Add a scheduled-timer regression fix that treats a clean completed attempt or committed side effects as terminal for a cron fire, while preserving documented configured fallbacks before any side-effecting work is committed.

## Reproduction Assessment

Yes, source-level and reporter evidence give a high-confidence scheduled-timer reproduction path: timer-fired non-Codex `payload.model` cron jobs can enter the configured fallback chain after a cleanup terminal state. I did not run the live current-main MiniMax/Telegram schedule because it requires real provider/channel credentials and scheduled firing.

## Solution Assessment

Unclear until maintainers choose the fallback boundary. Disabling all fallbacks for `payload.model` would contradict current tests, so the better direction is a side-effect-aware terminal gate or equivalent completion policy for scheduled cron attempts.
