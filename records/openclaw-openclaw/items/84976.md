## Summary

Keep open: the report is fresh and specific, and current main still has a plausible path where a scheduled payload-model cron run can fall through the configured fallback chain and re-run on the Codex route after a non-terminal first attempt. The exact scheduled-only cleanup-to-retry trigger still needs live or focused regression proof before changing the fallback contract.

## What This Changes

The issue reports scheduled isolated cron agentTurn jobs with a non-Codex payload model being retried on the Codex primary or fallback model, duplicating Telegram and file side effects.

## Best Possible Solution

Define a side-effect-aware cron fallback contract: once an attempt reaches a normal assistant stop or commits message/file/tool side effects, the scheduled run should be terminal or fail without replaying on another provider, with regression coverage for the scheduled timer path.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection confirms the relevant current-main path where a scheduled payload-model cron run can still enter `runWithModelFallback` with configured Codex fallbacks, but the exact cleanup-to-retry trigger needs live or focused test proof.

## Solution Assessment

Unclear; the narrow fix depends on whether maintainers want configured fallbacks to remain eligible after side effects. A safer end state is side-effect-aware fallback suppression while preserving fallback behavior for clean pre-side-effect provider failures.
