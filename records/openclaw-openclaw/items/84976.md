## Summary

Current main and the v2026.6.6 release already cover the reported duplicate side-effect loop: committed message-tool delivery and other observable delivery/progress now make the embedded attempt terminally successful and are excluded from fallback classification, so the scheduler should not re-drive that completed run on the Codex fallback model.

## What This Changes

The issue reports scheduled cron agent turns re-running after a non-Codex payload model completes, causing duplicate Telegram sends and file/state writes on the primary Codex model.

## Best Possible Solution

Keep the shipped terminal-delivery and fallback-classification behavior; reopen or file a new report only with v2026.6.6-or-newer scheduled-run evidence showing the same runId/sessionKey executes twice after committed delivery.

## Reproduction Assessment

Yes for the original regression: the reporter provided live scheduled-fire evidence on 2026.5.18, and the source path now clearly shows why committed delivery should be terminal. I did not establish a failing current-main repro because current main contains the fix proof.

## Solution Assessment

Yes. Treating committed outbound delivery and durable progress as terminal success, and excluding it from fallback classification, is the shared fix boundary without hard-coding MiniMax or Codex-specific behavior.
