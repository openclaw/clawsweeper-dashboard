## Summary

Keep open: the added live scheduled-run evidence makes the regression credible, and current main still documents and tests that cron `payload.model` runs can walk configured fallbacks. The remaining maintainer decision is how to stop fallback replay after a clean stop or committed side effects without breaking intentional pre-side-effect cron failover.

## What This Changes

The issue reports that scheduled isolated cron jobs with a non-Codex `payload.model` can run once on that model, commit Telegram/file side effects, then replay on the Codex primary/fallback model and duplicate the work.

## Best Possible Solution

Add a scheduled-timer regression fix that treats committed delivery/file/tool side effects or a clean assistant stop as terminal for that cron fire, while preserving documented configured fallbacks for clean pre-side-effect failures.

## Reproduction Assessment

Yes for the affected scheduled-timer path: the reporter supplied detailed live 2026.5.18 evidence from real scheduled fires, and current main still contains the source path that honors `payload.model` first then permits configured fallback candidates. I did not run a current-main live cron/MiniMax/Telegram reproduction in this read-only review.

## Solution Assessment

Unclear as a code change until maintainers choose the fallback boundary. Disabling all fallbacks for `payload.model` would contradict current docs/tests, so the safer fix is a side-effect-aware terminal gate that preserves fallback before any committed work.
