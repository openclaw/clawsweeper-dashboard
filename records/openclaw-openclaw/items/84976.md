## Label Justifications

- `P1`: The report describes a fresh scheduled automation regression that duplicates Telegram deliveries and local side effects for real cron users.
- `impact:message-loss`: One scheduled fire can produce duplicate outbound Telegram messages or repeated visible channel work.
- `impact:session-state`: The same run identity is reported to produce two session starts with conflicting cleanup and success terminal states.
- `impact:auth-provider`: The effective execution provider changes from the configured non-Codex payload model to the Codex primary/fallback model during replay.

## Summary

Keep open: the reporter's scheduled-fire evidence plus current source show payload-model cron runs can still walk configured fallback candidates, and the remaining decision is how to make clean or side-effecting attempts terminal without removing documented pre-side-effect fallback.

## What This Changes

Scheduled isolated cron jobs with a non-Codex `payload.model` should complete once on that model instead of replaying on the agent's Codex primary and duplicating side effects.

## Best Possible Solution

Add a scheduled-cron terminal gate that treats a clean assistant stop or committed message/file/tool side effects as terminal for that fire, while preserving documented configured fallbacks for pre-side-effect model failures and `payload.fallbacks: []` strict mode.

## Reproduction Assessment

Yes, at source level and with supplied live scheduled-fire evidence: current main still honors `payload.model` first and permits configured fallback candidates, and the reporter showed the scheduled MiniMax to Codex replay with duplicated side effects. I did not run a live current-main MiniMax/Telegram cron in this read-only review.

## Solution Assessment

Unclear until maintainers choose the fallback boundary. Disabling all fallbacks for `payload.model` would conflict with current docs and tests, so the maintainable fix is a side-effect-aware terminal policy rather than a blanket fallback removal.
