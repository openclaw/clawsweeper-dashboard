## Summary

Keep open: current main still has a source-backed path where a WebChat .docx upload is accepted, routed through managed media storage, and can fail when the temp-file `FileHandle.sync()` raises `EPERM` with no Windows-specific fallback.

## What This Changes

Request that Windows WebChat .docx uploads be saved as managed media instead of failing with `EPERM` from `fsync`.

## Best Possible Solution

Fix media buffer persistence so Windows `fsync`/flush failures do not reject accepted .docx uploads, while preserving the temp-file write-and-rename flow and adding regression coverage for the EPERM path.

## Reproduction Assessment

No live high-confidence reproduction was established in this read-only review. Source inspection gives a clear current-main path: WebChat sends .docx as a file, `chat.send` offloads it through `saveMediaBuffer`, and an `EPERM` from `FileHandle.sync()` propagates as `MediaOffloadError`.

## Solution Assessment

Yes, a narrow code fix is the maintainable direction: keep .docx/non-video upload support and make `src/media/store.ts` tolerate or avoid the Windows sync failure without adding a config option or changing upload policy.
