## Summary

Keep open: current main still routes WebChat `.docx` uploads through the attachment offload path, and the media store still calls `FileHandle.sync()` before rename with no Windows `EPERM` handling.

## What This Changes

The issue reports that WebChat `.docx` uploads fail on Windows with an `EPERM` from `fsync` while saving intercepted media.

## Best Possible Solution

Make the media-store sync step Windows-safe, keep the temp-file write and rename design, and add a focused regression for `EPERM` during temp-file sync.

## Reproduction Assessment

No, not as a live current-main reproduction. The issue body gives concrete Windows steps and source inspection maps them to the current `saveMediaBuffer` sync path, but native Windows or simulated `EPERM` proof is still needed.

## Solution Assessment

Yes, a narrow media-store fix is the best direction. The MIME and WebChat acceptance paths already handle `.docx`, so the fix should target sync/open behavior or intentional `EPERM` handling without changing attachment policy.
