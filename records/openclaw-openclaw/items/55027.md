- `impact:message-loss`: The affected behavior drops or withholds block chunks in the Feishu channel instead of delivering them progressively.

## Summary

Keep open: current main and the latest release still route enabled Feishu block chunks through the streaming-card fallback or drop them, while current docs describe `blockStreaming: true` as completed-block streaming before the final reply.

## What This Changes

The issue asks for Feishu to deliver each completed block progressively instead of relying on CardKit typewriter playback or dropping block chunks.

## Best Possible Solution

Update the existing Feishu plugin dispatcher and regression tests so `blockStreaming: true` delivers completed blocks progressively through a real Feishu send path, while preserving the default-off safeguard, final-card dedupe, and no-visible-reply fallback behavior.

## Reproduction Assessment

Yes. Source inspection shows the Feishu dispatcher returns block payloads when they cannot be consumed by a streaming card, and otherwise updates/closes the streaming card instead of sending an immediate completed-block message.

## Solution Assessment

No. Current main is a fallback/card-preview implementation, while the documented `blockStreaming` contract and this issue call for real completed-block delivery when Feishu block streaming is explicitly enabled.
