## Summary

Keep open. Current main exposes Feishu `blockStreaming`, but block payload delivery is still tied to the Feishu streaming-card path: with preview streaming disabled, block chunks return before any normal Feishu send helper runs, and with preview streaming enabled they are consumed by Card Kit playback rather than true completed-block messages.

## What This Changes

The issue requests Feishu `blockStreaming` to send each completed block immediately as a real Feishu message/card instead of dropping it or replaying it through Card Kit preview streaming.

## Best Possible Solution

Keep the existing config/schema support, but change the Feishu plugin so explicit `blockStreaming: true` delivers `{ kind: "block" }` through normal Feishu send helpers independent of Card Kit preview streaming, with duplicate-final and media protections preserved.

## Reproduction Assessment

Yes. A high-confidence source reproduction is `channels.feishu.blockStreaming: true` with `channels.feishu.streaming: false`: core block callbacks are enabled, but the dispatcher returns from the block path before sending anything to Feishu.

## Solution Assessment

Yes. A plugin-local repair is the narrowest maintainable fix because docs already define block streaming as normal channel messages and Feishu already has text/card send helpers; the broader character-streaming PR direction is not needed for this issue.
