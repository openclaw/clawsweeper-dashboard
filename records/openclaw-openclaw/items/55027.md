## Summary

Keep open. Current main still hard-disables Feishu's generic block-streaming path and only accepts block payload text when it can be folded into the Feishu streaming-card preview path, so the requested true progressive block delivery remains missing. The related closed Feishu reports address silent drops and duplicate streaming-card behavior, but they do not supersede this narrower delivery gap.

## What This Changes

The issue requests true progressive Feishu block delivery when runtime block chunks are emitted, without relying on Card Kit playback animation or dropping blocks when streaming cards are disabled.

## Best Possible Solution

Implement the fix inside the Feishu plugin boundary: resolve explicit top-level and per-account Feishu block-streaming config, allow the core block reply pipeline only when that setting is enabled, deliver `info.kind === "block"` through normal Feishu text/card sends, and keep Card Kit preview streaming as a separate mode with regression coverage for silent-drop and duplicate-final protections.
