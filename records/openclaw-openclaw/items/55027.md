## Summary

Keep open. Current main and the latest shipped release still route Feishu block payloads through the streaming-card path: with preview streaming disabled they return before any Feishu send helper runs, and with preview streaming enabled they are consumed by Card Kit playback rather than delivered as normal completed-block messages.

## What This Changes

The issue requests true Feishu block streaming so completed runtime blocks are delivered progressively instead of being dropped or replayed through card-side animation.

## Best Possible Solution

Keep the existing config/schema surface, but make the Feishu plugin deliver explicit `blockStreaming: true` block payloads through normal Feishu send helpers independently of Card Kit preview playback, with duplicate-final and media protections preserved.

## Reproduction Assessment

Yes, at source level. Configure Feishu with `blockStreaming: true` and `streaming: false`, then deliver a `{ kind: "block" }` payload: core block callbacks are enabled, but the Feishu dispatcher returns before any normal send helper runs.

## Solution Assessment

Yes. A Feishu plugin-local repair is the narrowest maintainable path because docs already define block streaming as normal channel messages and the plugin already has the card/text send helpers needed for completed-block delivery.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/55027 in the Feishu plugin. Current main enables core block callbacks when `account.config.blockStreaming === true`, but `deliver(..., { kind: "block" })` returns unless the Feishu streaming-card path is active; with `streaming: false` blocks are dropped, and with `streaming: true` blocks are consumed by Card Kit playback. Change `extensions/feishu/src/reply-dispatcher.ts` so explicit Feishu `blockStreaming: true` delivers completed block payloads through the normal Feishu send helpers independent of `FeishuStreamingSession`, preserving default `blockStreaming` disabled behavior, `renderMode: "raw"` behavior, media delivery, duplicate-final suppression, and existing streaming-card final/partial preview behavior where it should still apply. Add focused regression coverage in `extensions/feishu/src/reply-dispatcher.test.ts` for `blockStreaming: true` plus `streaming: false` and for `blockStreaming: true` plus `streaming: true` to prove block payloads reach normal Feishu sends rather than being dropped or only closed through a streaming card. Add a changelog entry for the user-visible Feishu fix; do not change unrelated channel/core streaming policy or the separate core final-payload suppression issue tracked by https://github.com/openclaw/openclaw/issues/78942 / https://github.com/openclaw/openclaw/pull/62157 unless it is strictly necessary and separately justified.
