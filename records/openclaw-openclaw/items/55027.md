## Summary

Keep open: current main and v2026.6.8 still drop Feishu block payloads unless the streaming-card path can consume them, and an open PR now explicitly owns the fix path. The issue should remain open until https://github.com/openclaw/openclaw/pull/94250 merges or is replaced.

## What This Changes

The issue asks Feishu to deliver emitted block chunks progressively as messages instead of dropping them when the streaming-card path is disabled or replaying them through CardKit animation.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/94250 or an equivalent narrow Feishu dispatcher repair, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, at source level: with Feishu `blockStreaming: true`, block payloads still return without delivery when `useStreamingCard` is false. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Yes for the narrow bug boundary: repairing the existing Feishu `blockStreaming` contract in the dispatcher is the maintainable path. The segment-mode proposal is broader and should stay separate unless maintainers choose that product direction.
