## Summary

Keep open: current main still has the Feishu dispatcher path that returns from completed block payloads unless a streaming card can consume them, and the latest release contains the same behavior. The related open PRs cover nearby Feishu streaming or core final-payload paths but no landed change closes this specific dispatcher bug.

## What This Changes

The issue requests true progressive Feishu block delivery so completed block chunks are sent immediately through normal Feishu messages instead of being dropped or replayed only through a streaming card.

## Best Possible Solution

Implement a Feishu plugin-local dispatcher fallback that sends explicit blockStreaming completed blocks through normal Feishu send helpers when no streaming card is active, with regression coverage for streaming false/card true and duplicate-final behavior.

## Reproduction Assessment

Yes, at source level. Configure Feishu with blockStreaming true and streaming false, then deliver a payload with info.kind === "block"; current main returns before normal Feishu send helpers run.

## Solution Assessment

Yes. A narrow Feishu dispatcher repair is the best fit because docs and config already define completed-block delivery separately from streaming-card preview playback.
