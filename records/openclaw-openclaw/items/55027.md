## Label Justifications

- `P2`: This is a bounded Feishu channel delivery bug with a workaround, but it still drops user-visible assistant block output.
- `impact:message-loss`: Generated assistant block text can be dropped instead of delivered to the Feishu chat unless a streaming card consumes it.

## Summary

Keep open: current main and v2026.6.10 still drop Feishu block payloads when the streaming-card path is unavailable, while the docs describe `blockStreaming` as completed-block delivery. The issue already has an open closing PR that owns the narrow dispatcher fix, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

Feishu should deliver enabled block-streamed assistant blocks immediately instead of dropping them or only replaying them through CardKit streaming animation.

## Best Possible Solution

Land a narrow Feishu dispatcher fix that sends enabled block payloads without CardKit animation and dedupes matching finals, then close this issue after it reaches main.

## Reproduction Assessment

Yes, at source level: with Feishu `blockStreaming: true`, a block payload returns without delivery when `useStreamingCard` is false. I did not run a live Feishu tenant reproduction in this read-only review.

## Solution Assessment

Yes for the narrow bug boundary: repairing the existing Feishu dispatcher `blockStreaming` contract is the maintainable fix. The segment-mode proposal is broader product/config work and should stay separate unless maintainers choose that direction.
