## Summary

Keep open. Current main and v2026.5.18 still route Feishu block payloads through the streaming-card path: with preview streaming disabled they return before normal Feishu send helpers run, and with preview streaming enabled they are consumed by Card Kit playback instead of delivered as completed-block messages.

## What This Changes

The issue requests Feishu to deliver each completed runtime block immediately as a normal Feishu message when block streaming is enabled, without relying on Card Kit playback or dropping blocks when preview streaming is disabled.

## Best Possible Solution

Keep the existing Feishu config surface, but make explicit `blockStreaming: true` block payloads deliver through normal Feishu send helpers independently of Card Kit preview streaming while preserving duplicate-final, media, and streaming-card safeguards.

## Reproduction Assessment

Yes, at source level. Configure Feishu with `blockStreaming: true` and `streaming: false`, then deliver a `{ kind: "block" }` payload: the current dispatcher returns before any normal Feishu send helper runs.

## Solution Assessment

Yes. A Feishu plugin-local repair is the narrowest maintainable path because docs already define block streaming as completed channel messages and the plugin already has card/text send helpers for completed-block delivery.
