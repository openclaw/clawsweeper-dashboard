## Summary

Keep open: current main still drops Feishu block payloads unless they can be consumed by the streaming-card path, and the latest shipped release contains the same behavior. The related open core PR targets final-payload suppression after block streaming, but it does not close this Feishu dispatcher bug.

## What This Changes

The issue requests Feishu to deliver completed block-streaming chunks as normal progressive messages when `blockStreaming` is enabled instead of dropping them or only feeding them into streaming-card playback.

## Best Possible Solution

Implement a Feishu plugin-local dispatcher fix so explicit `blockStreaming: true` block payloads deliver through normal Feishu send helpers independently of Card Kit preview streaming, while preserving existing duplicate-final, media, and streaming-card safeguards.

## Reproduction Assessment

Yes, at source level. Configure Feishu with `blockStreaming: true` and `streaming: false`, then deliver a `{ kind: "block" }` payload: current main returns before any normal Feishu send helper runs.

## Solution Assessment

Yes. A Feishu plugin-local repair is the narrowest maintainable path because docs already define block streaming as completed channel messages and the Feishu config surface already exists.
