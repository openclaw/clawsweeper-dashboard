## Summary

Keep open. Current main still disables Feishu’s generic block-streaming path and only consumes block chunks when they can be folded into the Card Kit streaming-card preview, so the requested true progressive block delivery remains missing. The related closed Feishu reports address silent-drop prevention and duplicate-card suppression, but they do not supersede this narrower behavior gap.

## What This Changes

Request Feishu to deliver runtime block chunks progressively as normal Feishu replies when block streaming is enabled, instead of dropping them or replaying them through Card Kit preview animation.

## Best Possible Solution

Implement the fix inside the Feishu plugin boundary: resolve explicit top-level and per-account block-streaming config using the shared channel streaming helpers, let the core block pipeline run only when that setting is enabled, deliver block chunks through the existing Feishu normal text/card send helpers, and keep Card Kit preview streaming as a separate mode with duplicate-final protections intact.

## Reproduction Assessment

Yes. A high-confidence unit reproduction already exists: create the Feishu dispatcher, invoke the captured `deliver` callback with `{ kind: "block" }`, and current tests/source show no Feishu send occurs unless the block can be consumed by the streaming-card path.

## Solution Assessment

Yes. A plugin-local repair is the narrowest maintainable solution because core already supports `disableBlockStreaming: false`, Feishu already has normal send helpers, and the missing behavior is the Feishu dispatcher/config decision that hard-disables and then drops block chunks.
