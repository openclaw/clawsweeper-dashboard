## Summary

Keep open: current main still drops Feishu completed-block payloads unless the streaming-card path can consume them, while the docs define block streaming as normal completed-block channel delivery. The latest release contains the same dispatcher behavior, and the related open PRs cover adjacent Feishu/core streaming paths rather than a landed fix for this issue.

## What This Changes

The issue requests true Feishu completed-block delivery so runtime-emitted block chunks are sent immediately instead of being dropped or replayed only through streaming-card animation.

## Best Possible Solution

Send explicit Feishu `blockStreaming` completed blocks through normal Feishu send helpers instead of returning or relying solely on streaming cards, while preserving duplicate-final suppression and preview-card behavior for partial/final previews.

## Reproduction Assessment

Yes, at source level. Configure Feishu with `blockStreaming: true` and `streaming: false`, then deliver a payload with `info.kind === "block"`; current main returns before normal Feishu send helpers run.

## Solution Assessment

Yes. A narrow Feishu dispatcher repair is the maintainable path because docs and config already define completed-block delivery separately from streaming-card preview playback.
