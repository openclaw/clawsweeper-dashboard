## Summary

Keep open: current main and latest release still route Feishu block payloads only through the streaming-card path, or return before normal send helpers when that path is disabled. The docs/config already define `blockStreaming` as completed-block channel delivery, and related open PRs overlap but no merged narrow fix owns this issue yet.

## What This Changes

Feishu should deliver block-streaming blocks immediately as normal Feishu messages/cards instead of dropping them or only replaying them through an interactive streaming card.

## Best Possible Solution

Fix Feishu's plugin-local dispatcher so explicit `blockStreaming: true` block payloads are delivered through normal Feishu card/text send helpers without depending on Card Kit playback, while preserving preview streaming and duplicate-final safeguards.

## Reproduction Assessment

Yes, at source level. Configure Feishu with `blockStreaming: true`, `streaming: false`, and `renderMode: "card"`, then deliver a `{ kind: "block" }` payload: current main returns before any normal Feishu send helper runs.

## Solution Assessment

Yes. The narrow maintainable fix is in the Feishu dispatcher because the existing config and docs already define completed-block delivery separately from interactive-card preview streaming.
