## Summary

Keep open: current main and the latest release still drop Feishu block payloads when the streaming-card path cannot consume them, and https://github.com/openclaw/openclaw/pull/94250 is an open closing PR for the narrow dispatcher fix. This issue should close only after that PR, or an equivalent reviewed replacement, lands on main.

## What This Changes

Feishu should deliver emitted block replies progressively as real Feishu messages instead of relying on CardKit playback or dropping blocks outside the streaming-card path.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/94250, or an equivalent narrow Feishu dispatcher repair, then close this issue once the fix reaches main.

## Reproduction Assessment

Yes. Source inspection shows Feishu opts into core block callbacks when `blockStreaming: true`, but the dispatcher returns from `info.kind === "block"` when no streaming card is usable; I did not run a live Feishu tenant test.

## Solution Assessment

Yes for the narrow bug boundary. The maintainable fix is to repair the existing Feishu `blockStreaming` dispatcher contract, and the open closing PR is already the candidate path rather than a reason to queue duplicate work.
