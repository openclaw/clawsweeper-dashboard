## Summary

Keep open. Current main now accepts Feishu `blockStreaming` config, but the central behavior is still missing: Feishu block chunks are only consumed by the Card Kit streaming-card path and are still dropped when preview streaming is disabled, so the related closed schema/duplicate-card fixes do not supersede this issue.

## What This Changes

Feishu should deliver runtime block chunks progressively as real channel sends when `blockStreaming` is enabled, instead of dropping them or routing them only through streaming-card playback.

## Best Possible Solution

Keep the existing config/schema support, but change the Feishu plugin so explicit block streaming delivers `{ kind: "block" }` through normal Feishu send helpers independent of Card Kit preview streaming, with duplicate-final protections preserved.

## Reproduction Assessment

Yes. A high-confidence source reproduction is `channels.feishu.blockStreaming: true` with `channels.feishu.streaming: false`: core block callbacks are enabled, but the dispatcher returns from the block path before sending because `streamingEnabled` is false.

## Solution Assessment

Yes. A plugin-local repair is the narrowest maintainable fix because core already supports block replies and Feishu already has normal text/card send helpers; the missing behavior is the Feishu dispatcher’s block-delivery decision.
