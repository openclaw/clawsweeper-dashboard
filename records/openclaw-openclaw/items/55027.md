## Summary

Keep open: current main and the latest release still route Feishu block payloads through the streaming-card path or drop them, so the reported lack of true completed-block delivery remains. The related open segment-mode PR is useful context but is not a clean canonical fix yet because it adds a parallel config mode and is currently blocked/conflicting.

## What This Changes

The issue requests Feishu `blockStreaming` to deliver each completed runtime block immediately instead of relying on CardKit typewriter playback or dropping block chunks when streaming cards are disabled.

## Best Possible Solution

Repair the existing Feishu `blockStreaming` contract so completed blocks are sent immediately through normal Feishu text/card delivery, while `streaming` remains the CardKit live-card control.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.8` still return from the Feishu block delivery path when no streaming card can consume the block, and the streaming-card path still uses CardKit playback. I did not run a live Feishu reproduction in this read-only review.

## Solution Assessment

Yes for an existing-contract repair: the narrowest maintainable solution is to make Feishu `blockStreaming` own immediate completed-block sends without adding another config knob. The open segment-mode PR is a broader alternative and needs maintainer cleanup before it can be canonical.
