## Summary

Keep open: the feature direction is useful, but the current PR is not merge-ready because it still makes warn-mode cleanup destructive, removes orphan checkpoints without the existing age guard, can under-report files already removed by maintenance, has inaccurate docs for public retention keys/defaults, and still lacks real CLI behavior proof.

## What This Changes

The PR adds archived transcript and orphan checkpoint cleanup planning/reporting to `openclaw sessions cleanup`, updates `SessionCleanupSummary` output, and adds docs, changelog, and focused tests.

## Best Possible Solution

Integrate archive cleanup reporting with the existing maintenance helpers so dry-run and JSON can preview archived deletions, apply-mode counts are collected before mutation, warn mode remains non-destructive, checkpoint cleanup keeps the age guard, and docs use the public retention keys/defaults.

## Reproduction Assessment

Yes. Source inspection of PR head gives deterministic fixture paths for the defects: non-dry-run warn mode reaches the new unlink planner, and a fresh unindexed checkpoint reaches deletion without an mtime cutoff; I did not run a write-producing local repro because this review is read-only.

## Solution Assessment

No. The requested cleanup/reporting direction is useful, but the current implementation duplicates the archive sweep after existing maintenance and weakens established warn-mode, retention, and reporting contracts; reusing or extending the existing cleanup contracts is safer.

- **[P2] Gate archive unlinking on enforce mode:** `src/config/sessions/cleanup-service.ts:680-685`
  - body: `mode` can still be `warn` when cleanup runs without `--enforce`, but this new call passes `dryRun: false` for every non-dry-run invocation. That makes warn-mode cleanup delete archived files, so keep this planner in preview mode unless the resolved mode is enforcing.
  - confidence: 0.94
- **[P2] Keep checkpoint cleanup behind the age guard:** `src/config/sessions/cleanup-service.ts:347-350`
  - body: This branch removes any checkpoint whose session id is absent from the index, regardless of mtime. Current unreferenced-artifact cleanup keeps the `file.mtimeMs <= Date.now() - olderThanMs` cutoff, so fresh or in-flight checkpoints can be removed here; reuse that sweep or apply the same cutoff.
  - confidence: 0.9
- **[P2] Report archived removals before maintenance deletes them:** `src/config/sessions/cleanup-service.ts:680-685`
  - body: In enforce mode, `updateSessionStore` already runs deleted/reset archive cleanup before this planner executes. The new `archivedFiles` summary can therefore report zero removals even though the same command removed files; collect counts from the existing helper or plan before applying.
  - confidence: 0.86
- **[P3] Use public retention keys and defaults:** `docs/cli/sessions.md:125-126`
  - body: The docs expose the internal `pruneAfterMs` name and say unset reset retention leaves reset archives in place. The public key is `session.maintenance.pruneAfter`, and unset `resetArchiveRetention` defaults to that window; only `false` disables reset archive cleanup.
  - confidence: 0.9
