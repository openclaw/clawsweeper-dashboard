## Summary

Keep this PR open: the direction is useful, but the current patch still weakens cleanup safety by deleting archive files in warn mode, removes orphan checkpoints without the existing age guard, can underreport files already removed by current maintenance, and lacks real behavior proof.

## What This Changes

The PR adds archived transcript cleanup planning/reporting to `openclaw sessions cleanup`, adds an `archivedFiles` summary field, updates dry-run output, docs, changelog, and focused tests.

## Best Possible Solution

Integrate archive cleanup reporting with the existing maintenance helpers so warn mode remains non-destructive, retention and mtime guards are preserved, and CLI/JSON counts reflect files before deletion.

## Reproduction Assessment

Yes. Source inspection of the PR diff and current main shows the unconditional non-dry-run planner call, missing checkpoint mtime guard, and post-maintenance reporting path without needing a write-producing local run.

## Solution Assessment

No. The requested direction is useful, but the current implementation duplicates existing maintenance cleanup and weakens established warn-mode, retention, and reporting contracts; it should reuse or extend those contracts instead.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Gate archive unlinking on enforce mode:** `src/config/sessions/cleanup-service.ts:680-685`
  - body: `mode` can still be `warn` when cleanup runs without `--enforce`, but this new call passes `dryRun: false` for every non-dry-run invocation. That lets warn-mode cleanup delete archive files, so only execute archive deletion when the resolved mode is enforcing or keep the planner in preview mode for warn.
  - confidence: 0.94
- **[P2] Keep checkpoint cleanup behind the age guard:** `src/config/sessions/cleanup-service.ts:336-350`
  - body: The checkpoint branch removes any checkpoint whose session id is absent from the index, regardless of mtime. Current unreferenced-artifact cleanup keeps the `file.mtimeMs <= Date.now() - olderThanMs` cutoff, so fresh or in-flight checkpoints can be removed here; reuse that sweep or apply the same cutoff.
  - confidence: 0.9
- **[P2] Report archived removals before maintenance deletes them:** `src/config/sessions/cleanup-service.ts:679-685`
  - body: In enforce mode, `updateSessionStore` already runs deleted/reset archive cleanup before this planner executes. The new `archivedFiles` summary can therefore report zero removals even though the same command removed files; collect counts from the existing helper or plan before applying.
  - confidence: 0.86
- **[P3] Use public retention keys and defaults:** `docs/cli/sessions.md:125-126`
  - body: The docs expose the internal `pruneAfterMs` name and say unset reset retention leaves reset archives in place. The public key is `session.maintenance.pruneAfter`, and unset `resetArchiveRetention` defaults to that window; only `false` disables reset archive cleanup.
  - confidence: 0.9
