## Summary

Keep open. The PR targets a valid sessions-cleanup gap, but the current head still has merge-blocking cleanup bugs: non-enforce warn-mode runs can unlink archived files, and checkpoint cleanup can delete live checkpoints whose filename basename differs from the session ID.

## What This Changes

The PR wires archived transcript and orphan checkpoint cleanup into `openclaw sessions cleanup`, adds an `archivedFiles` summary/output field, and updates docs, changelog, and regression tests.

## Best Possible Solution

Land the feature after archived cleanup shares the command’s warn/enforce semantics, checkpoint preservation uses exact metadata-referenced paths, public docs match config keys/defaults, and the branch is rebased onto current main.

## Reproduction Assessment

Yes, source-level. On the PR head, a non-dry-run cleanup in warn mode reaches archive deletion with `dryRun: false`, and a live checkpoint can be treated as orphaned when its file basename is not the live session ID.

## Solution Assessment

No, not yet. Co-locating archived cleanup with `runSessionsCleanup` is the right boundary, but deletion must be gated by effective enforce mode and checkpoint liveness should use exact referenced paths instead of filename-prefix inference.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Respect warn mode before deleting archives:** `src/config/sessions/cleanup-service.ts:482-487`
  - body: This runs archived-file cleanup with `dryRun: false` for every non-dry-run command after `updateSessionStore`, even when the resolved cleanup mode is `warn`. Current cleanup treats warn mode as non-mutating unless `--enforce` is passed, so a default cleanup can now unlink archived files while the index cleanup stayed warning-only.
  - confidence: 0.94
- **[P2] Preserve referenced checkpoint paths:** `src/config/sessions/cleanup-service.ts:262-276`
  - body: The planner considers a checkpoint live only when the filename-derived basename is in the live session ID set. Checkpoint files are named from the transcript basename and the authoritative live reference is `compactionCheckpoints[].preCompaction.sessionFile`, so topic/custom/legacy transcript basenames can lose still-referenced checkpoints.
  - confidence: 0.92
- **[P3] Correct archived-retention docs:** `docs/cli/sessions.md:105-106`
  - body: These docs use the internal `session.maintenance.pruneAfterMs` name and say reset archives are left unless configured. The public key is `session.maintenance.pruneAfter`, and an unset `resetArchiveRetention` defaults to the prune window unless explicitly set to `false`.
  - confidence: 0.95
