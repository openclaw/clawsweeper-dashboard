## Summary

Keep this PR open. It addresses a real sessions-cleanup gap, but the current patch can delete a still-referenced compaction checkpoint when the checkpoint filename prefix differs from the live `sessionId`, and the new docs still describe the retention config incorrectly.

## What This Changes

The PR adds archived transcript and orphan checkpoint cleanup to `openclaw sessions cleanup`, exposes an `archivedFiles` summary, and updates tests, docs, and the changelog.

## Best Possible Solution

Land the feature after checkpoint cleanup uses exact referenced checkpoint paths from store metadata, keeps dry-run/enforce parity, fixes the public config wording, and adds a regression fixture for a custom or legacy `sessionFile` checkpoint.

## Reproduction Assessment

Yes. A source-level reproduction is a `sessions.json` entry with `sessionId: "live"` and `compactionCheckpoints[].preCompaction.sessionFile` pointing at `legacy.checkpoint.<uuid>.jsonl`; the PR planner treats `legacy` as orphaned even though current metadata still references the path.

## Solution Assessment

No, not yet. Co-locating archived cleanup with `runSessionsCleanup` is the right boundary, but checkpoint preservation should match current disk-budget path-reference semantics before this is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve referenced checkpoint paths:** `src/config/sessions/cleanup-service.ts:269-276`
  - body: This check only keeps a checkpoint when the filename prefix matches a live `sessionId`. Existing checkpoint metadata stores the authoritative `compactionCheckpoints[].preCompaction.sessionFile` path, and checkpoint names are derived from the current `sessionFile`; a live entry with a legacy or custom transcript basename would have its still-listed checkpoint deleted by `cleanup --enforce`. Build the reference set from checkpoint paths, matching disk-budget cleanup, and add that fixture.
  - confidence: 0.9
- **[P3] Correct archived-retention docs:** `docs/cli/sessions.md:105-106`
  - body: The new archived-files docs name the internal `pruneAfterMs` field and say reset archives are left in place unless configured. The public key is `session.maintenance.pruneAfter`, and `resetArchiveRetention` defaults to that prune window unless explicitly set to `false`, so this would send operators to the wrong config behavior.
  - confidence: 0.95
