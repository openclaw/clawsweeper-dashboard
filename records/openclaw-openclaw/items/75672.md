## Summary

Keep this PR open. Current main still lacks the requested archived-file cleanup in `openclaw sessions cleanup`, and the PR is a useful implementation candidate for open #75658, but the proposed orphan-checkpoint logic is too broad and can delete valid checkpoint snapshots for sessions whose transcript filename does not equal `entry.sessionId`. The PR also needs the normal changelog follow-through before merge.

## What This Changes

The PR adds archived transcript cleanup planning and execution to `openclaw sessions cleanup`, exposes an `archivedFiles` summary in text/JSON output, and adds tests for deleted/reset archive and orphan checkpoint cleanup.

## Best Possible Solution

Keep the PR open and revise the archived-file plan to preserve exact referenced checkpoint paths from `compactionCheckpoints[].preCompaction.sessionFile`, preferably by sharing or mirroring the existing disk-budget reference calculation. Then add targeted regression coverage for explicit/topic `sessionFile` checkpoint retention and add an Unreleased changelog entry for the cleanup behavior.

## Reproduction Assessment

Yes. A high-confidence source-level repro is a live store entry with `sessionId: "sess"`, `sessionFile: "sess-topic-1.jsonl"`, and `compactionCheckpoints[].preCompaction.sessionFile` pointing at `sess-topic-1.checkpoint.<uuid>.jsonl`; the PR's filename-prefix check does not see `sess-topic-1` in the live session-id set and will plan/remove that referenced checkpoint.

## Solution Assessment

No. The direction is right, but the checkpoint orphan test is not the safest maintainable solution because current code already models checkpoint liveness by exact referenced paths rather than by session-id prefixes.
