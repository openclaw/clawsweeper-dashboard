## Summary

Keep open: the cleanup/reporting direction is useful for the linked sessions cleanup problem, but the current PR head still has a source/build blocker, unsafe destructive cleanup semantics, inaccurate apply reporting, docs/changelog issues, and no real behavior proof.

## What This Changes

The PR adds archived transcript cleanup planning/reporting to `openclaw sessions cleanup`, dry-run text output, docs/changelog text, and focused tests for deleted/reset/checkpoint files.

## Best Possible Solution

Fold archive cleanup reporting into the existing maintenance path so it uses `params.maintenance`, preserves warn/enforce and mtime-retention guards, reports deletions accurately, documents public config keys/defaults, and is backed by redacted real CLI output.

## Reproduction Assessment

Yes. Source inspection of the current PR head shows the undefined `maintenance` reference, warn-mode deletion path, checkpoint removal without the current-main mtime cutoff, and docs/default mismatch; no runtime test was run because this was a read-only review.

## Solution Assessment

No. The intended archive cleanup/reporting is useful, but this implementation duplicates a post-maintenance sweep and breaks established cleanup safety/reporting contracts; extending the existing maintenance helper/reporting path is safer.
