## Summary

Keep open: the feature is useful, but the current diff still has two blocking cleanup-semantics bugs around warn mode and checkpoint preservation, plus a docs mismatch on public config names/defaults.

## What This Changes

The PR adds archived transcript/checkpoint cleanup to `openclaw sessions cleanup`, exposes an `archivedFiles` summary, and updates CLI output, docs, changelog, and regression tests.

## Best Possible Solution

Land the feature after archived-file cleanup obeys warn/enforce semantics, preserves exact metadata-referenced checkpoint files, and documents the public retention keys/defaults correctly.

## Reproduction Assessment

Yes, source-level. With `session.maintenance.mode: "warn"` and no `--enforce`, the PR still reaches archive deletion with `dryRun: false`, and a live checkpoint whose transcript basename differs from `sessionId` is treated as orphaned by the filename-prefix check.

## Solution Assessment

No, not yet. Co-locating archive cleanup with `runSessionsCleanup` is the right boundary, but the implementation should gate deletion on enforce semantics and preserve checkpoint files by exact referenced paths from session metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Respect warn mode before deleting archives:** `src/config/sessions/cleanup-service.ts:481-487`
  - body: This always runs archived-file cleanup with `dryRun: false` after `updateSessionStore`, even when the resolved cleanup mode is `warn` and `--enforce` was not passed. Current cleanup treats warn mode as non-mutating, so a default `openclaw sessions cleanup` can now unlink `.deleted`, `.reset`, and checkpoint files despite warn mode.
  - confidence: 0.93
- **[P2] Preserve referenced checkpoint paths:** `src/config/sessions/cleanup-service.ts:262-276`
  - body: This keeps checkpoints only when the filename prefix matches a live `sessionId`. Checkpoint files are named from the transcript basename, while the authoritative live reference is `compactionCheckpoints[].preCompaction.sessionFile`, so a live session with a custom or legacy transcript basename can lose its still-referenced checkpoint during cleanup.
  - confidence: 0.91
- **[P3] Correct archived-retention docs:** `docs/cli/sessions.md:105-106`
  - body: The new docs use the internal `session.maintenance.pruneAfterMs` name and say reset archives are left in place unless configured. The public key is `session.maintenance.pruneAfter`, and `resetArchiveRetention` defaults to that prune window unless explicitly set to `false`.
  - confidence: 0.95
