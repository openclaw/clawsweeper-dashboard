## Summary

Keep open: this still targets the real cleanup gap tracked by the linked issue, but the current PR head is not mergeable as-is because it can delete archived files in warn mode, removes orphan checkpoints without the existing age safety guard, misstates retention defaults in docs, and still lacks real behavior proof.

## What This Changes

The PR adds archived session transcript cleanup planning/reporting for `.deleted`, `.reset`, and orphan checkpoint files to `openclaw sessions cleanup`, with CLI output, docs, changelog, and tests.

## Best Possible Solution

Land a scoped cleanup-command artifact plan only after it respects warn/enforce mode, uses the existing age/reference safety guards, documents public config keys and defaults, and includes redacted real CLI proof.

## Reproduction Assessment

Yes. Source inspection reproduces the main blockers: with `session.maintenance.mode` set to `warn`, non-dry-run cleanup still reaches archive deletion, and orphan-looking checkpoints are queued for removal without an mtime cutoff.

## Solution Assessment

No. The direction is useful, but the current patch should reuse or match the established cleanup safety semantics instead of adding a more aggressive parallel deletion path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate archive cleanup on enforce mode:** `src/config/sessions/cleanup-service.ts:674-680`
  - body: `mode` can be `warn` when cleanup runs without `--enforce`, and current cleanup deliberately avoids destructive artifact pruning in that mode. This new call still unlinks archived files with `dryRun: false`, so warn-mode cleanup would mutate disk unexpectedly; skip archive deletion unless the resolved mode is enforcing.
  - confidence: 0.93
- **[P2] Keep checkpoint cleanup behind the age guard:** `src/config/sessions/cleanup-service.ts:335-349`
  - body: This branch removes any checkpoint whose session id is absent from the index immediately. Existing unreferenced-artifact cleanup only removes compaction checkpoints after the `pruneAfter` mtime cutoff, so fresh or in-flight checkpoint files can be deleted here; preserve the existing age guard or delegate checkpoint cleanup to that path.
  - confidence: 0.86
- **[P3] Correct archived retention docs:** `docs/cli/sessions.md:125-126`
  - body: The new docs use the internal `pruneAfterMs` name and say reset archives are left alone when retention is unset. The public config key is `session.maintenance.pruneAfter`, and unset `resetArchiveRetention` defaults to that prune window; only `false` disables reset archive cleanup.
  - confidence: 0.9
