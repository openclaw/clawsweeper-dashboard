## Summary

Keep open. The feature is useful for #75658, but the current PR head can still delete archived files while cleanup is in warn mode and can remove still-referenced compaction checkpoints because orphan detection uses filename prefixes instead of stored checkpoint paths; the docs also describe the wrong public retention key/default.

## What This Changes

This PR adds archived transcript and checkpoint cleanup to `openclaw sessions cleanup`, exposes an `archivedFiles` summary, and updates docs, changelog, and regression tests.

## Best Possible Solution

Land the feature after archive cleanup respects warn/enforce semantics, checkpoint pruning preserves exact metadata-referenced checkpoint files, and public docs match the config schema/defaults.

## Reproduction Assessment

Yes, source-level. The warn-mode path still calls the archive planner with `dryRun: false`, and a live entry with `sessionId: "live"` but a checkpoint file named from a legacy/custom transcript basename would be treated as orphaned by the PR's filename-prefix check.

## Solution Assessment

No, not yet. Co-locating archive cleanup with `runSessionsCleanup` is the right boundary, but the implementation should reuse the existing warn/enforce contract and preserve checkpoint files via exact referenced paths from post-cleanup metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Respect warn mode before deleting archives:** `src/config/sessions/cleanup-service.ts:482-487`
  - body: This always runs the archive planner with `dryRun: false` after `updateSessionStore`, even when the resolved cleanup mode is `warn` and `--enforce` was not passed. Existing cleanup treats warn mode as non-mutating, so a default `openclaw sessions cleanup` can now unlink `.deleted`, `.reset`, and checkpoint files without the configured enforcement mode.
  - confidence: 0.92
- **[P2] Preserve referenced checkpoint paths:** `src/config/sessions/cleanup-service.ts:269-271`
  - body: This keeps checkpoints only when the filename prefix matches a live `sessionId`. Checkpoint files are named from the transcript basename, while the authoritative reference is `compactionCheckpoints[].preCompaction.sessionFile`, so a live session with a custom or legacy transcript basename can lose its still-referenced checkpoint during `cleanup --enforce`.
  - confidence: 0.9
- **[P3] Correct archived-retention docs:** `docs/cli/sessions.md:105-106`
  - body: The new docs use the internal `session.maintenance.pruneAfterMs` name and say reset archives are left in place unless configured. The public key is `session.maintenance.pruneAfter`, and `resetArchiveRetention` defaults to that prune window unless explicitly set to `false`.
  - confidence: 0.95
