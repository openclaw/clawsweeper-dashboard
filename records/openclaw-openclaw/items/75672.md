## Summary

Keep this PR open: the feature targets a real cleanup gap, but the current patch would still delete checkpoint files that are referenced by live session metadata and the new retention docs point operators at the wrong config behavior.

## What This Changes

The PR adds archived session transcript and orphan checkpoint cleanup to sessions cleanup, surfaces an `archivedFiles` summary, and updates tests, docs, and the changelog.

## Best Possible Solution

Land this feature after archived checkpoint cleanup preserves exact checkpoint paths referenced by store metadata, dry-run/enforce behavior stays aligned, and the public docs describe the existing retention keys and defaults correctly.

## Reproduction Assessment

Yes. A source-level reproduction is a live store entry with `sessionId: "live"` and `compactionCheckpoints[].preCompaction.sessionFile` pointing at `legacy.checkpoint.<uuid>.jsonl`; the patch classifies `legacy` as orphaned even though the current metadata still references that path.

## Solution Assessment

No. Co-locating archived-file cleanup with `runSessionsCleanup` is the right boundary, but the orphan-checkpoint detector must use current checkpoint path-reference semantics before this is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve referenced checkpoint paths:** `src/config/sessions/cleanup-service.ts:269-276`
  - body: This treats a checkpoint as live only when the filename prefix matches a referenced session id. Existing metadata stores the authoritative `compactionCheckpoints[].preCompaction.sessionFile` path, and snapshots are named from the current `sessionFile` basename, so a live entry with a legacy or custom basename can have its still-referenced checkpoint deleted by `cleanup --enforce`. Build the keep set from referenced checkpoint paths and add that fixture.
  - confidence: 0.9
- **[P3] Correct archived-retention docs:** `docs/cli/sessions.md:105-106`
  - body: The new docs name internal `session.maintenance.pruneAfterMs` and say reset archives are left in place unless configured. The public key is `session.maintenance.pruneAfter`, and reset archive cleanup defaults to that prune window unless `resetArchiveRetention` is `false`, so this would mislead operators tuning cleanup.
  - confidence: 0.95
