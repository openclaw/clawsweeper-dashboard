## Summary

Keep open: the PR targets a real cleanup/reporting gap, but the current patch can delete still-referenced compaction checkpoints when the checkpoint filename basename differs from the live `sessionId`, and the new docs misstate the public retention config.

## What This Changes

The PR adds archived transcript and orphan checkpoint cleanup to `openclaw sessions cleanup`, surfaces an `archivedFiles` summary in text/JSON output, and updates docs, changelog, and regression tests.

## Best Possible Solution

Land the feature after checkpoint cleanup uses exact post-cleanup referenced checkpoint paths from store metadata and the public docs match the config schema/defaults.

## Reproduction Assessment

Yes, source-level. A live store entry with `sessionId: "live"` and `compactionCheckpoints[].preCompaction.sessionFile` pointing at `legacy.checkpoint.<uuid>.jsonl` is still referenced by current metadata semantics, but the PR planner treats `legacy` as orphaned because it only compares the filename prefix to live session ids.

## Solution Assessment

No, not yet. Co-locating archive cleanup with `runSessionsCleanup` is the right boundary, but checkpoint preservation should follow the existing exact-path reference contract and the docs should use the public config keys/defaults.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve referenced checkpoint paths:** `src/config/sessions/cleanup-service.ts:269-271`
  - body: This keeps checkpoints only when the filename prefix matches a live `sessionId`. Existing metadata stores the authoritative `compactionCheckpoints[].preCompaction.sessionFile` path, and checkpoint names are derived from the current transcript basename, so a live session with a legacy/custom transcript basename can have its still-referenced checkpoint removed by `cleanup --enforce`. Build the reference set from checkpoint paths and add that fixture.
  - confidence: 0.9
- **[P3] Correct archived-retention docs:** `docs/cli/sessions.md:105-106`
  - body: The added docs name the internal `pruneAfterMs` field and say reset archives are left in place unless configured. The public key is `session.maintenance.pruneAfter`, and `resetArchiveRetention` defaults to that prune window unless explicitly set to `false`, so this sends operators to the wrong config behavior.
  - confidence: 0.95
