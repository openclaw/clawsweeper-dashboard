## Summary

Keep open: the PR targets a valid sessions cleanup gap, but the current head can delete archived files in warn mode and can remove still-referenced compaction checkpoints; the new docs also misstate the public retention keys/defaults.

## What This Changes

The branch adds archived transcript/checkpoint cleanup to `openclaw sessions cleanup`, exposes an `archivedFiles` summary, and updates CLI rendering, docs, changelog, and tests.

## Best Possible Solution

Land the feature after archived-file cleanup obeys warn/enforce semantics, preserves checkpoint files by exact metadata-referenced paths, and documents the public retention keys/defaults correctly.

## Reproduction Assessment

Yes, source-level. Warn mode reaches archive deletion with `dryRun: false`, and checkpoint preservation is filename-prefix based while current checkpoint cleanup uses exact `preCompaction.sessionFile` references.

## Solution Assessment

No, not yet. Co-locating cleanup in `runSessionsCleanup` is the right boundary, but deletion must be gated by enforce semantics and checkpoint liveness must use the existing exact referenced-path contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Respect warn mode before deleting archives:** `src/config/sessions/cleanup-service.ts:481-487`
  - body: This always runs archived-file cleanup with `dryRun: false` after the store update, even when the resolved mode is `warn`. Current cleanup treats warn mode as reporting-only, so a default cleanup command can unlink archived files without `--enforce`.
  - confidence: 0.93
- **[P2] Preserve referenced checkpoint paths:** `src/config/sessions/cleanup-service.ts:262-270`
  - body: This treats a checkpoint as live only when its filename prefix matches a live `sessionId`. Checkpoint snapshots are referenced by exact `compactionCheckpoints[].preCompaction.sessionFile` paths, so a custom or legacy transcript basename can lose a still-referenced checkpoint.
  - confidence: 0.91
- **[P3] Correct archived-retention docs:** `docs/cli/sessions.md:105-106`
  - body: The docs use the internal `session.maintenance.pruneAfterMs` name and say reset archives are left in place unless configured. The public key is `session.maintenance.pruneAfter`, and `resetArchiveRetention` defaults to that prune window unless set to `false`.
  - confidence: 0.95
