## Summary

Keep open. Current main still uses only process-local Teams sent-message markers, and this PR is a plausible narrow migration to the existing SDK keyed store, but the latest patch has a P2 correctness bug that renews recovered markers for another full TTL and the PR is protected for maintainer handling.

## What This Changes

The PR adds Microsoft Teams sent-message marker persistence using the SDK keyed store behind the existing in-memory cache, switches reply-to-bot checks to async recovery, exposes optional runtime access, and updates tests plus the changelog.

## Best Possible Solution

Preserve the original `sentAt` during persistent recovery, keep this Teams-only split scoped to the Teams plugin, and then let maintainers decide whether it should land separately from #75053.

## Reproduction Assessment

Yes for the review finding: store a marker, clear memory, have persistent lookup hit near the original 24-hour expiry, then observe the PR backfills memory with `Date.now()` and extends the marker. I did not run that test because this pass is read-only, but the timestamp flow is clear in the PR source.

## Solution Assessment

No as written. The SDK keyed-store approach is the right narrow mechanism, but persistent lookup should return the stored record or `sentAt` and pass that timestamp into the memory backfill instead of renewing it at lookup time.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88
