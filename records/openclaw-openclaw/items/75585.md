## Summary

Keep open. Current `main` still has only a process-local Microsoft Teams sent-message cache, and this PR is a plausible Teams-scoped persistence change, but the latest patch still renews recovered markers for another full TTL and the protected `maintainer` label leaves the product decision with maintainers.

## What This Changes

The PR adds best-effort Microsoft Teams sent-message marker persistence through the SDK keyed store, switches reply-to-bot lookup to an async persistent recovery path, exposes optional Teams runtime access, and updates tests plus the changelog.

## Best Possible Solution

Keep the Teams-only migration scoped to the Teams plugin, preserve the existing 24-hour marker semantics when recovering from SQLite, and then let maintainers decide whether this split should land separately from the broader persistence rollout.

## Reproduction Assessment

Yes for the review finding: register a marker, clear memory, have persistent lookup succeed near the original 24-hour expiry, then advance past the original expiry and observe the PR keeps the in-memory marker alive because it backfilled with `Date.now()`. Current main is also statically reproducible as process-local because no Teams persistent store path exists.

## Solution Assessment

No as written. The SDK keyed-store approach is the right narrow mechanism, but persistent lookup should return the stored `sentAt` and backfill memory with that timestamp instead of renewing the marker at lookup time.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve the original marker TTL on persistent backfill:** `extensions/msteams/src/sent-message-cache.ts:164`
