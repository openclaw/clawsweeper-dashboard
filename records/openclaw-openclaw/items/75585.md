## Summary

Keep open. This draft PR has the protected `maintainer` label, current `main` still uses process-local Microsoft Teams sent-message markers, and the PR is a real implementation rather than obsolete cleanup. The patch is not merge-ready as written because a persistent hit backfills memory with the lookup time, extending the existing 24-hour marker lifetime after restart.

## What This Changes

The PR adds best-effort SDK keyed-store persistence for Microsoft Teams sent-message markers, wires reply-to-bot checks through an async persistent lookup, and updates tests, Teams docs, and the changelog.

## Best Possible Solution

Keep the Teams-only migration scoped to the Teams plugin, preserve the existing 24-hour marker semantics when recovering from SQLite, and then let a maintainer decide whether this default-on split should land separately from the broader #75053 rollout.

## Reproduction Assessment

Yes for the review finding: with fake timers, register a marker, clear memory, make a persistent lookup just before the original 24-hour expiry, then advance past the original expiry and observe the PR keeps the in-memory marker alive. For the feature itself, current main is statically reproducible as process-local because the Teams marker cache has no persistent store path.

## Solution Assessment

No as written. The SDK keyed-store approach is the right narrow mechanism for a bundled plugin, but the persistent lookup should return the stored `sentAt` and backfill memory with that timestamp instead of `Date.now()`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve the original marker TTL on persistent backfill:** `extensions/msteams/src/sent-message-cache.ts:164`
  - body: When a marker is recovered from SQLite near the 24-hour expiry, this writes it back to memory with `Date.now()`, so the in-memory fast path can keep classifying replies as `reply_to_bot` for another full TTL after the original persistent record expires. Return the stored `sentAt` from `lookupPersistentSentMessage` and pass that into `rememberSentMessageInMemory` so restart recovery does not extend marker lifetime beyond the existing cache semantics.
  - confidence: 0.87
