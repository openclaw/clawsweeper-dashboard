## Summary

Keep open. Current main still uses a fixed above-trigger delete confirmation with no viewport measurement, flip, or clamp logic, and the latest release has the same relevant implementation. The related placement-fix PRs were closed unmerged, while the remaining open related PR addresses listener cleanup rather than overflow placement.

## What This Changes

The issue reports that the Control UI chat delete confirmation popover can render outside the visible chat boundary, making the delete action unreachable.

## Best Possible Solution

Add viewport-aware placement for the Control UI chat delete confirmation: measure the trigger and popover after insertion, choose above or below based on available space, clamp horizontally within the viewport, preserve skip-confirm/cancel/delete/outside-click behavior, and cover top-edge, bottom-edge, and narrow-width cases in focused tests. Keep the broader localization suggestion separate from this bug fix.

## Reproduction Assessment

Yes. The source path is deterministic: clicking a chat message delete button near the top or horizontal edge opens an absolutely positioned popover above the trigger inside a clipped chat thread, with no current-main measurement or flip/clamp path.

## Solution Assessment

Yes. A narrow viewport-aware placement fix in the grouped chat delete popover is the maintainable solution; the reporter’s workaround of removing `position: absolute` may avoid this overflow but would put the confirmation back into document flow and risk shifting chat layout.
