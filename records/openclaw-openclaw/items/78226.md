## Summary

Keep open: the linked bug remains source-reproducible on current main, but this PR is not mergeable as submitted because it has a missing import, leaves a stale initialization writeback path, has stale CLI test mocks, and still lacks after-fix real behavior proof.

## What This Changes

The PR adds locked, hash-guarded exec-approval writeback across node, gateway, and CLI paths so stale execution metadata should not restore revoked policy.

## Best Possible Solution

Keep the CAS/lock direction, but correct the branch by adding or replacing the missing process-scoped queue helper, guarding every exec-approvals write including initialization/defaulting, updating CLI tests, and adding redacted real node/gateway proof.

## Reproduction Assessment

Yes. Source inspection on current main shows resolveExecApprovals loads approvals before an async skill-bin refresh and recordAllowlistUse later saves that stale object without a freshness check, while admin writes do check baseHash.

## Solution Assessment

No as submitted. The locked CAS metadata-write approach is the right narrow shape, but this branch is non-buildable, leaves ensureExecApprovals() outside the guard, and needs test/proof follow-through.
