## Summary

Keep open: current main still has the user-boundary-only `turnSucceeded` annotation introduced by the merged UI fix, existing tests miss adjacent agent-initiated turns, and no open duplicate PR currently owns this follow-up.

## What This Changes

The issue reports that back-to-back agent-initiated turns can cause a later assistant reply to mark an earlier failed tool turn as succeeded in chat rendering.

## Best Possible Solution

Add a focused chat-item regression test for adjacent agent-initiated turns and update the annotation to scope success to the actual turn boundary without regressing the earlier within-turn suppression.

## Reproduction Assessment

Yes from source inspection: the current backward pass resets only at user groups, so adjacent agent-initiated turns can share the later reply flag. I did not run the repro test because this review is read-only and tests may create artifacts.

## Solution Assessment

Yes in direction: the narrow fix belongs in the shared `buildChatItems` annotation path with regression coverage, not in individual Control UI/WebChat renderers. The implementation should use durable turn/run metadata rather than a naive assistant-boundary reset because the earlier fix intentionally supports multi-tool single turns.
