## Summary

Keep this PR open. Current main still uses `reserveTokensFloor` for both compaction threshold gates, while this PR applies a shared effective-reserve helper with focused regression coverage and no concrete PR-introduced correctness finding.

## What This Changes

The PR adds shared effective Pi compaction reserve resolution, applies it to preflight compaction and memory-flush gates, renames the gate helper parameter, adds regression tests, and updates the changelog.

## Best Possible Solution

Land this PR or an equivalent narrow core fix after exact-head validation, keeping effective reserve resolution shared so Pi settings, preflight compaction, and memory flush agree on reserve semantics.

## Reproduction Assessment

Yes. On current main, configuring `agents.defaults.compaction.reserveTokens` above `reserveTokensFloor` is source-reproducible: both threshold gates still subtract only the floor while the config contract defines `reserveTokens` as the effective headroom.

## Solution Assessment

Yes. Centralizing effective reserve calculation in `pi-settings` and feeding that value into both gates is the narrow maintainable fix, and it avoids changing the public `MemoryFlushPlan` plugin surface.
