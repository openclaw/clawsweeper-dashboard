Keep open: current main still has the floor-only compaction gates this PR targets, and the PR is a focused fix with no blocking review finding found; it needs maintainer review, rebase/mergeability confirmation, and exact-head validation rather than cleanup closure.

## What This Changes

The PR adds a shared effective Pi compaction reserve resolver, updates preflight compaction and memory-flush thresholds to use it, renames the gate parameter, adds regression tests, and records the fix in the changelog.

## Best Possible Solution

Land this PR or an equivalent narrow core fix after rebase and exact-head validation, keeping effective reserve resolution shared so Pi settings, preflight compaction, and memory flush agree on reserve semantics.

## Reproduction Assessment

Yes. On current main, configure `agents.defaults.compaction.reserveTokens` above `reserveTokensFloor` and inspect the source paths: both threshold gates still compute from the floor-only value, while the config contract says `reserveTokens` is the effective headroom.

## Solution Assessment

Yes. Centralizing effective reserve calculation in `pi-settings` and feeding that value into both gates is the narrow maintainable fix, and it avoids changing the public `MemoryFlushPlan` plugin surface.
