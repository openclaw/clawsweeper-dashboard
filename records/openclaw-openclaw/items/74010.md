## Summary

Keep open: current main still uses `reserveTokensFloor` in both compaction gates while the config contract makes `reserveTokens` the user-facing headroom, and the PR is a narrow fix with no patch-introduced correctness or security finding. It still needs a rebase/validation refresh because the branch is dirty against current main and exact-head checks are stale/failing outside the touched compaction surface.

## What This Changes

The PR adds shared effective Pi compaction reserve resolution, applies it to preflight and memory-flush gates, renames the gate parameter, adds focused regression tests, and updates the changelog.

## Best Possible Solution

Rebase this PR or an equivalent narrow replacement onto current main, rerun touched-surface and changed-gate validation, then land it so Pi settings, preflight compaction, and memory flush share the same effective reserve semantics.

## Reproduction Assessment

Yes. Source inspection on current main shows both gates subtract only `reserveTokensFloor`; for a 100k context with `reserveTokens: 30000`, floor 20k, and soft threshold 4k, current main uses a 76k threshold where the config contract implies 66k.

## Solution Assessment

Yes. Centralizing effective reserve resolution in `pi-settings` and passing that value into both gates is the narrow maintainable fix, and it avoids changing the public `MemoryFlushPlan` plugin contract.
