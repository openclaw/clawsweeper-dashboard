## Summary

Keep this PR open. Current main still uses `reserveTokensFloor` for both compaction gates, while the PR's shared effective-reserve helper and focused tests address the reported `reserveTokens` asymmetry without changing the plugin `MemoryFlushPlan` contract.

## What This Changes

The PR adds shared effective Pi compaction reserve resolution, applies it to preflight compaction and memory-flush gates, renames the gate parameter, adds regression tests, and updates the changelog.

## Best Possible Solution

Land this PR or an equivalent narrow core fix after exact-head validation, keeping effective reserve resolution shared so Pi settings, preflight compaction, and memory flush agree on reserve semantics.

## Reproduction Assessment

Yes. On current main, configuring `agents.defaults.compaction.reserveTokens` above `reserveTokensFloor` can be verified by source inspection: both threshold gates still read only the floor while the config contract defines `reserveTokens` as the effective headroom.

## Solution Assessment

Yes. Centralizing the effective reserve calculation in `pi-settings` and passing that value into both gates is the narrow maintainable fix, and it preserves the existing `MemoryFlushPlan` plugin surface.
