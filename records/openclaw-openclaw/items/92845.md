## What This Changes

## Best Possible Solution

Land the docs clarification as the narrow fix; any change to auto-start the standalone HTTP listener should be handled separately as a product and security decision.

## Reproduction Assessment

Yes. By source inspection, current main docs omit the eager env requirement while code and tests keep the standalone listener lazy unless `OPENCLAW_EAGER_BROWSER_CONTROL_SERVER` is truthy; I did not run a live gateway in this read-only review.

## Solution Assessment

Yes. Documenting the existing opt-in requirement in the API reference is the narrowest maintainable fix; auto-starting the listener would change exposure and startup behavior and should be separate.
