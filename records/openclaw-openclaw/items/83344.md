## Summary

Keep open: current main still has a source-proven WebChat history-tail gap where `chat.history` bounds the raw transcript tail before display projection, and the session preview/title path can still select silent/internal tail records such as `NO_REPLY`. The queue/draft portion overlaps https://github.com/openclaw/openclaw/issues/51549 and https://github.com/openclaw/openclaw/pull/73154, but this issue has distinct history-window and preview evidence worth fixing or tracking directly.

## What This Changes

The issue reports that WebChat refresh can hide recent visible conversation history and stale session preview text when the transcript tail is dominated by internal tool or silent records.

## Best Possible Solution

Add a shared visible-history tail-read path so `chat.history` and session previews gather enough displayable user/assistant records before applying limits, while leaving queue/draft persistence to the existing refresh-state issue and PR.

## Reproduction Assessment

Yes, at source level. Current main caps the raw transcript records before display projection in `chat.history`, and the preview path scans bounded tail windows without applying the same silent/internal filtering; the reporter also supplied a real transcript shape that matches this path.

## Solution Assessment

Yes for a narrow repair. The maintainable fix is to reuse or share the visible-history over-read/projection logic for RPC history and previews, not to solve this by adding local queue/draft storage or by forcing the broader pagination PR into this issue.

Fix https://github.com/openclaw/openclaw/issues/83344 on current main. Scope: WebChat refresh via RPC `chat.history` can return too few visible messages when the newest raw transcript records are internal/tool/silent because `chat.history` passes the requested limit as the raw `maxMessages` and `parseRecentTranscriptTailMessages` slices raw messages before display projection.
