## Summary

Keep open: current main has partial WebChat history restoration, but source inspection still supports the reported failure class when bounded transcript tails are dominated by internal/silent rows, and the linked broader WebChat refresh issue remains open for queue/draft persistence.

## What This Changes

The issue asks WebChat refresh to restore recent visible conversation history and accurate session previews even when the transcript tail is mostly internal tool traffic, with related queue and draft preservation concerns.

## Best Possible Solution

Fix the WebChat/gateway history and preview paths to collect enough recent display-visible transcript rows before applying the UI window, while handling queue and draft persistence through the broader refresh-state decision tracked by the related open issue.

## Reproduction Assessment

Yes, source inspection gives a concrete repro path: build a WebChat transcript whose recent bounded tail is mostly internal/tool/silent rows, then refresh and observe too few visible rows plus stale preview metadata. This review did not run a live browser F5 repro, so the structured status is source-reproducible rather than reproduced.

## Solution Assessment

Unclear for the whole issue because the visible-history/preview fix is a narrow maintainable bug fix, but queue and draft persistence require a product decision about storage lifetime and privacy. A safer path is to split the narrow history-tail fix from the broader refresh-state persistence work in the related open issue.
