## Summary

Keep open: current main has gateway-backed history reload, but the source still supports the internal-heavy tail failure for visible WebChat history and preview metadata, while the queue/draft pieces remain tied to the broader refresh-state discussion.

## What This Changes

The issue asks for WebChat refresh to restore recent user-visible conversation and previews even when the transcript tail is mostly internal tool, silent, or heartbeat traffic.

## Best Possible Solution

Fix history and preview retrieval to gather enough recent display-visible transcript rows before applying UI limits, and handle queue/draft refresh persistence through the broader state-lifecycle decision tracked by the related open issue and PR.

## Reproduction Assessment

Yes, at source level: create a WebChat transcript whose latest 100 raw records are mostly internal/tool/silent entries, then refresh so chat.history and preview extraction select too few display-visible rows. I did not run a live browser F5 repro, so this is source-reproducible rather than fully reproduced.

## Solution Assessment

Unclear for the whole issue. The narrow history/preview fix should live in the gateway/history projection path, while queue and draft persistence need maintainer-approved retention, attachment, and replay semantics before the open broader PR or a replacement is the best fix.
