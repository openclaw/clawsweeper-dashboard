## Label Justifications

- `P2`: This is a normal-priority WebChat behavior bug with visible state/message-context loss but no evidence of transcript deletion or core runtime outage.
- `impact:message-loss`: The UI can make recent user-visible conversation messages appear missing after refresh even though the transcript remains on disk.
- `impact:session-state`: The report concerns WebChat session refresh state, transcript projection, queue state, draft state, and stale session previews.

## Summary

Keep open: current main still loads a bounded raw WebChat transcript tail before display filtering, and session previews still use bounded tail scans, so the reported internal/tool-heavy tail failure remains source-reproducible. The broader queue/draft refresh work is also still open through related review paths rather than shipped on main.

## What This Changes

The issue asks WebChat refresh/history and session previews to restore recent human-visible messages when the transcript tail is dominated by internal/tool/silent records.

## Best Possible Solution

Fix the WebChat/gateway history and preview paths to collect enough recent display-visible transcript rows before applying UI limits, and handle queue/draft persistence through the broader refresh-state decision with explicit retention and replay semantics.

## Reproduction Assessment

Yes, at source level: build a WebChat transcript whose recent bounded tail is mostly internal/tool/silent rows, then refresh and observe that chat.history and preview scans can return too few visible rows or stale preview text. I did not perform a live browser F5 run in this read-only review.

## Solution Assessment

Unclear for the whole issue: the history-tail and preview repair is a narrow maintainable bug fix, but queue and draft persistence need a maintainer decision about storage lifetime, privacy, attachments, and replay timing.
