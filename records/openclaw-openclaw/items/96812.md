## Label Justifications

- `P1`: The PR targets a provider-stream bug that can replace live opencode-go answers with stalled errors for real users.
- `merge-risk: 🚨 compatibility`: The unconditional `toolcall_start` liveness change can alter timeout behavior for existing slow first-event opencode-go streams.
- `merge-risk: 🚨 availability`: The diff changes when opencode-go streams are kept alive versus aborted, and the current shape can prematurely abort a valid stream.

## Summary

Keep open: the linked opencode-go watchdog bug is still valid on current main, but this PR is not merge-ready because it treats `toolcall_start` as unconditional progress and can shorten the protected first-event window before any real provider delta arrives.

## What This Changes

The PR adds `text_end`, `thinking_end`, `toolcall_start`, and `toolcall_end` to the opencode-go stalled-stream liveness predicate.

## Best Possible Solution

Gate boundary liveness after the first real provider delta, add regressions for both post-delta boundary liveness and pre-delta `toolcall_start`, then provide after-fix runtime proof.

## Reproduction Assessment

Yes. Current main forwards boundary events but only re-arms idle liveness for deltas, and the linked issue includes a focused failing timing test; I did not run tests in this read-only review.

## Solution Assessment

No, not as submitted. The correct layer is the provider-owned opencode-go wrapper, but the implementation should preserve the first-event contract by accepting boundary liveness only after real provider progress begins.
