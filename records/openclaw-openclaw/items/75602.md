## Summary

Keep open. The PR is draft and has the protected `maintainer` label, and the patch does not fix the linked iMessage stdin failure path from #75438; it only touches MCP stdio transport and generic process exec. I also found a blocking correctness issue in the MCP change because the returned promise can resolve before the new write callback receives an async write error.

## What This Changes

The PR adds a write callback around `OpenClawStdioClientTransport.send()` writes and installs a child stdin `error` listener before `runCommandWithTimeout()` writes input.

## Best Possible Solution

Keep this PR open for maintainer review, but either retarget it as a smaller generic exec/MCP hardening PR with tests and changelog coverage, or replace it with the focused iMessage fix from #75473 so #75438 is resolved in the owner plugin path.

## Reproduction Assessment

Yes. Current main statically shows the unguarded iMessage write, and focused Node probes reproduce both the child stdin `EPIPE` uncaught shape and the write-callback ordering problem in the MCP-style promise pattern.

## Solution Assessment

No. The current patch is not the best fix for the linked issue because it misses `extensions/imessage/src/client.ts` and the MCP promise still resolves before async write callback errors can reject it; the narrower maintainable fix is local stdin stream/write-callback handling in the iMessage RPC client, with generic exec hardening split or clearly scoped.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Fix the linked iMessage stdin path:** `src/agents/mcp-stdio-transport.ts:142`
  - body: This PR cites #75438, but that report's crashing write is still `IMessageRpcClient.request()` in `extensions/imessage/src/client.ts`, which this patch does not touch. Merging this as-is would leave the reported `imsg rpc` EPIPE path open, so the fix needs to cover the iMessage client or stop claiming that issue as its target.
  - confidence: 0.93
- **[P2] Wait for the write callback before resolving:** `src/agents/mcp-stdio-transport.ts:150`
  - body: On the usual `stdin.write()` true path, this promise resolves immediately. If Node later calls the write callback with `EPIPE`, the promise is already fulfilled, so `send()` still reports success for the async write failure. Resolve from the write callback on success, or otherwise keep the promise unsettled until the write outcome is known.
  - confidence: 0.88
- **[P3] Add the required changelog entry:** `src/process/exec.ts:360`
  - body: This is a user-facing fix for gateway child-process crash handling, but the PR only changes source files. Repo policy requires a `CHANGELOG.md` entry for user-visible fixes, so the release notes would miss this behavior change.
  - confidence: 0.9
