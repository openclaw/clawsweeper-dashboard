## Summary

Keep open. The cleanup problem is still visible in current main’s refactored code, but the branch is stale/dirty against the split registry modules and needs porting before it can land; the security pass found no supply-chain concern.

## What This Changes

The PR changes subagent cleanup bookkeeping so termination and orphan-prune cleanup finalize consistently, delete-mode termination becomes idempotent, and deferred cleanup retries continue after pruning.

## Best Possible Solution

Port the cleanup invariant into the current split modules: termination cleanup should finalize/delete once per updated runId, orphan pruning should wake any deferred cleanup retries through the lifecycle seam, and focused regression tests should cover multi-run childSessionKey delete cleanup plus deferred retry wakeup.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: add two delete-mode run records with the same childSessionKey, call markSubagentRunTerminated with that childSessionKey, and inspect listSubagentRunsForRequester or the registry map; current code updates both rows but only delete-finalizes one child-session entry.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is: the PR’s direction is right, but current main moved the relevant code into subagent-registry-run-manager.ts, subagent-registry-lifecycle.ts, and subagent-registry-helpers.ts, so the maintainable fix is a narrow port with tests rather than landing the stale branch unchanged.
