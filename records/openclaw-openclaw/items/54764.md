## Summary

Keep open. The cleanup bug the PR targets is still visible on current main, but the branch edits the old monolithic registry shape and needs a narrow port to the current split registry modules before it can be merged.

## What This Changes

The PR refactors subagent cleanup bookkeeping so termination and orphan-prune cleanup share final cleanup state handling, delete-mode termination becomes idempotent, and deferred cleanup retries continue after orphan pruning.

## Best Possible Solution

Port the cleanup invariant into the current split modules: terminate/delete cleanup should finalize every updated run id, orphan pruning should wake deferred cleanup retry handling through the lifecycle seam, and focused regression tests should cover shared child-session delete cleanup plus orphan-prune retry wakeup.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: register two delete-mode subagent run records with the same `childSessionKey`, call `markSubagentRunTerminated` by that child session, and inspect the registry; current main updates both records but only delete-finalizes one grouped entry.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is. The proposed behavior is right, but the maintainable fix is a narrow port to `subagent-registry-run-manager.ts`, `subagent-registry-lifecycle.ts`, and `subagent-registry-helpers.ts` with current-layout tests.
