## Summary

Keep open: the PR addresses a real `models set` feedback gap, but the current head is not mergeable as-is because it uses side-effectful catalog loading for a warning, bypasses the established config mutation path, drops current Codex repair warnings, and still lacks real CLI proof.

## What This Changes

The PR changes `models set` to load the model catalog, warn when the selected provider/model is missing, and then write the selected default model directly.

## Best Possible Solution

Implement warning-only catalog-miss feedback inside the current `updateConfig` path, use read-only catalog access, preserve Codex repair warnings, and update the PR body/proof to match the intended behavior.

## Reproduction Assessment

Yes, by source inspection: current main's `modelsSetCommand` writes the resolved model through `applyDefaultModelPrimaryUpdate` without any catalog lookup, so typo feedback is absent. I did not run the CLI because this read-only review must not mutate user config.

## Solution Assessment

No. Warning-only catalog feedback may be the right direction, but this branch must preserve the config mutation contract, avoid catalog refresh side effects, and keep the Codex repair warning path before it is the best implementation.
