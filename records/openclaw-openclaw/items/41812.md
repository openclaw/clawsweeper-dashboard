## Summary

Keep open: current main still lacks catalog feedback for `models set`, and the PR resolved the earlier hard-reject/duplicate-validation concerns, but the latest head still bypasses the config mutation contract and lacks the required changelog entry.

## What This Changes

The PR changes `models set` to load the model catalog, warn when the resolved model is absent, then save the chosen default model directly with `writeConfigFile`.

## Best Possible Solution

Keep warning-only catalog-miss feedback consistent with the runtime allowlist, preserve conflict-safe/include-aware config mutation semantics, and document the user-facing CLI fix in `CHANGELOG.md`.

## Reproduction Assessment

Yes, by source inspection. Current main's `modelsSetCommand` saves a syntactically valid model ref through `applyDefaultModelPrimaryUpdate` without loading the catalog; I did not run the CLI because this review is read-only and the command mutates user config.

## Solution Assessment

No. The warning-only catalog feedback is aligned with the runtime allowlist contract, but the final save should use the existing `updateConfig`/`replaceConfigFile` mutation path instead of writing an object derived from a pre-catalog snapshot.
