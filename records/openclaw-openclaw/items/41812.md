## Summary

Keep open. Current main still has no catalog feedback in `models set`, so the PR is not obsolete, but the latest patch bypasses the established conflict-safe config persistence path and should not merge as-is.

## What This Changes

The PR changes `models set` to load the model catalog, warn when the resolved provider/model is absent, and then persist the selected default model through a direct config write.

## Best Possible Solution

Keep the catalog-miss feedback, but route the final mutation through `updateConfig` or `replaceConfigFile` with a valid snapshot/hash so concurrent edits and include-owned config sections are preserved. Add focused tests for the warning behavior and for preserving the config write contract, plus the required user-facing changelog entry.

## Reproduction Assessment

Yes. Current main reproduces the original gap by source inspection because `modelsSetCommand` never loads the catalog; the PR regression is reproducible from the diff whenever config changes during the awaited catalog load or the relevant config section is include-owned.

## Solution Assessment

No. Warning on catalog misses is a reasonable narrow direction, but this implementation is not the best fix because it replaces the conflict-safe config helper with a direct write; the safer alternative is to keep the catalog check and perform the final write through the existing mutation contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the conflict-safe config write path:** `src/commands/models/set.ts:30`
  - body: The new flow loads config, awaits catalog loading, then writes an `updated` object derived from that earlier snapshot with `writeConfigFile(updated)`.
