The PR makes `models set` load the model catalog, warn for unresolved provider/model refs, then persist the selected default through a direct config write.

## Best Possible Solution

Keep the catalog-miss warning, preserve existing config mutation semantics, and add regression coverage plus changelog before merge.

No. Warning on catalog miss is a maintainable direction, but the final mutation should stay on `updateConfig` or `replaceConfigFile` with base-hash/include semantics and should include tests plus changelog text.
