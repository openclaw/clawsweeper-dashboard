## Summary

Keep open. The requested typo feedback is plausible and the current-main gap is source-reproducible, but this PR bypasses current config-write and Codex repair contracts, uses a side-effectful catalog path, and still lacks required real behavior proof.

## What This Changes

The PR adds model-catalog lookup to `models set`, warning when the requested provider/model is not found before persisting the default model.

## Best Possible Solution

Add warning-only catalog feedback with `loadModelCatalog({ readOnly: true })` while preserving `updateConfig` and post-write Codex repair warnings, then land it with redacted CLI proof.

## Reproduction Assessment

Yes, by source inspection: current main's `modelsSetCommand` updates the default model through `updateConfig` without any catalog lookup, so typo feedback is absent. I did not run the CLI because this read-only review must not mutate user config.

## Solution Assessment

No, not as implemented. Warning-only catalog feedback is a better fit than hard rejection for the current allowlist model, but it should use the read-only catalog path and stay inside the existing config-update and Codex repair flow.
