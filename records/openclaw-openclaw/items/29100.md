## Summary

Keep open. Current main still lacks a supported pricing overlay for auto-discovered Amazon Bedrock models: the plugin discovery schema and typed config omit cost metadata, and the discovery path still assigns zero-cost defaults to discovered foundation and inference-profile models.

## What This Changes

The issue requests a backward-compatible way to attach per-model cost metadata to Amazon Bedrock auto-discovery without duplicating explicit provider model entries.

## Best Possible Solution

Land a plugin-owned Bedrock discovery cost overlay on the current `plugins.entries.amazon-bedrock.config.discovery` surface, with legacy migration compatibility, cache-key handling, docs, tests, and coordinated handling of the aws-sdk display path tracked by #53286.

## Reproduction Assessment

Yes. Source inspection on current main is enough: the strict Bedrock plugin discovery schema rejects a costs field, the typed discovery config has no such field, and discoverBedrockModels assigns zero-cost metadata to discovered models.

## Solution Assessment

Yes. A Bedrock plugin-owned overlay is the narrow maintainable direction; duplicating explicit provider blocks is the current workaround, while a global models.costs overlay would be broader than this provider-owned request.

## Repair Work Prompt

Implement cost metadata support for auto-discovered Amazon Bedrock models. Add a backward-compatible costs map under `plugins.entries.amazon-bedrock.config.discovery` keyed by model ID, migrate legacy `models.bedrockDiscovery` content into that plugin config when present, apply full or partial cost overlays to discovered foundation models and inference profiles, include normalized costs in discovery cache identity, and keep discovery/request behavior unchanged. Use closed PR #31160 only as source context; do not move Bedrock-owned policy into core or hard-code an AWS pricing catalog. Update Bedrock docs, relevant config/help/schema artifacts if required by the repo workflow, focused tests, and CHANGELOG.md because this is user-facing. Coordinate or explicitly test the aws-sdk cost-display overlap tracked by #53286 so the metadata is usable in status/usage surfaces or the remaining display work is clearly scoped.
