## Summary

Keep open. Current main and the latest tagged release still lack a supported cost overlay for auto-discovered Amazon Bedrock models, the direct implementation PR was closed unmerged, and the remaining display behavior for `aws-sdk` cost output is separately tracked.

## What This Changes

Add a supported cost map for Amazon Bedrock auto-discovered models so usage and status surfaces can estimate USD without duplicating explicit provider model entries.

## Best Possible Solution

Add a plugin-owned `plugins.entries.amazon-bedrock.config.discovery.costs` overlay, apply it during Bedrock discovery, include it in the discovery cache key, and keep `aws-sdk` visible-cost display policy coordinated with the separate open issue.

## Reproduction Assessment

Yes. Source inspection is enough: the strict plugin schema and shared type omit a pricing map, and discovered Bedrock models are assigned zero-cost metadata today.

## Solution Assessment

Yes. A Bedrock plugin-owned overlay is the narrowest maintainable direction; a top-level `models.costs` map would be broader product surface and should wait for explicit maintainer direction.

Implement https://github.com/openclaw/openclaw/issues/29100 as a narrow Amazon Bedrock plugin feature. Add a plugin-owned `plugins.entries.amazon-bedrock.config.discovery.costs` map keyed by Bedrock model ID with optional nonnegative `input`, `output`, `cacheRead`, and `cacheWrite` values in USD per 1M tokens. Do not add a top-level `models.costs` overlay without maintainer approval. Apply the overlay when converting discovered foundation models and inherited inference profiles, keep unspecified cost fields at zero, and include a deterministic normalized cost fingerprint in the discovery cache key. Preserve legacy `models.bedrockDiscovery` migration behavior by moving or merging any legacy `costs` object into the plugin-owned discovery config if present. Update Bedrock docs and focused tests.
