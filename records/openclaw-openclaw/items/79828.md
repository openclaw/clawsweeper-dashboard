## Summary

Keep open. Source inspection supports a real package/runtime dependency bug: the image tool can still require `sharp` when an image needs optimization, but the published root package does not install `sharp`; the existing fallback only covers in-limit non-HEIC media.

## What This Changes

The issue asks OpenClaw to make the documented image tool work on default Debian/global npm installs when it receives JPEG attachments without a separately installed `sharp` package.

## Best Possible Solution

Ensure the packaged npm install resolves the media-understanding image optimizer dependency, and add package-acceptance or postpublish coverage so bundled plugin runtime imports cannot pass without installable dependencies.

## Reproduction Assessment

Yes from source inspection, but not from a live package install in this read-only review. A JPEG that requires optimization goes through `loadWebMedia` to `resizeToJpeg`, which reaches the bundled `sharp` import while the root package install does not provide `sharp`.

## Solution Assessment

Yes, the narrow maintainable direction is packaging/runtime dependency coverage, not Telegram-specific handling. Preserve the existing in-limit fallback, but make default packaged installs able to optimize images that exceed the cap.
