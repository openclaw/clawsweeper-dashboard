## Summary

Keep open. Current main still matches the report: Anthropic onboarding stores a local `anthropic:claude-cli` OAuth profile, but the runtime external-CLI resolver overlays fresh Claude CLI credentials as `runtime-only` and does not update the persisted profile, so an expired local profile can keep re-triggering the bootstrap path and DEBUG log. The fix choice touches the existing runtime-only external-auth policy, so it should get auth-profile owner review rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to keep Claude CLI-derived local OAuth profiles fresh from the external CLI credentials, or otherwise stop repeated bootstrap DEBUG logging after the local profile expires.

## Best Possible Solution

Adopt an owner-approved policy: safely persist fresher matching credentials for stored, non-bootstrapOnly external CLI profiles under the auth-store lock, or intentionally keep runtime-only overlays and make the DEBUG line state-change/rate limited with docs coverage.

## Reproduction Assessment

Yes, source-reproducible: an expired stored `anthropic:claude-cli` profile plus fresher Claude CLI credentials will pass through `resolveExternalCliAuthProfiles`, return a runtime-only overlay, and emit the reported DEBUG message. I did not establish a live Windows/Claude CLI reproduction cycle.

## Solution Assessment

Unclear. Option A looks like the right user-visible fix if the stored Claude CLI profile is meant to stay canonical, but it changes credential persistence semantics; if the runtime-only policy is intentional, the safer fix is to de-dupe or rate-limit the DEBUG log and document the behavior.
