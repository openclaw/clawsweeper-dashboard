## Summary

Keep open. Current main still overlays fresh Claude CLI credentials as runtime-only when the stored `anthropic:claude-cli` profile is expired, so the stale stored profile and repeated bootstrap DEBUG path remain; the linked hclsys commits look like a proposed Option A fix, but they are out-of-main/diverged and not shipped proof.

## What This Changes

The issue requests that stored external-CLI OAuth profiles such as `anthropic:claude-cli` resync from fresher Claude CLI credentials or stop producing per-resolution bootstrap debug logs.

## Best Possible Solution

Make an explicit auth-profile policy decision, then either persist fresher matching credentials for stored non-bootstrap-only external CLI profiles under the auth-store lock or intentionally keep runtime-only overlays while de-duplicating the bootstrap DEBUG log and documenting the behavior.

## Reproduction Assessment

Yes, source-reproducible: an expired stored `anthropic:claude-cli` OAuth profile plus fresher Claude CLI credentials will pass through `resolveExternalCliAuthProfiles`, return a runtime-only overlay, and emit the reported debug message. I did not establish a live Windows Claude CLI refresh cycle.

## Solution Assessment

Unclear. The linked hclsys commits implement the reporter’s preferred persistence direction, but writing credentials from the runtime overlay path changes auth-store semantics, so owner review should choose the final persistence or log-rate-limit design before merge.
