## Summary

Keep open. Current main still matches the report: an expired stored `anthropic:claude-cli` OAuth profile can be overlaid with fresh Claude CLI credentials at runtime while the stored profile remains stale, and the exact DEBUG bootstrap line is still emitted on that path; the right fix needs auth-profile policy review because it may change documented runtime-only external-CLI credential semantics.

## What This Changes

The issue asks OpenClaw to stop stale local Claude CLI OAuth profiles from repeatedly triggering external-CLI bootstrap DEBUG logs after the external Claude CLI credentials refresh.

## Best Possible Solution

Have the auth-profile owner choose one policy and cover it with tests: either safely persist fresher same-identity Claude CLI credentials under the auth-store lock, or keep runtime-only overlays and make the DEBUG line state-change or rate limited with docs coverage.

## Reproduction Assessment

Yes, source-reproducible: an expired stored `anthropic:claude-cli` OAuth profile plus fresher Claude CLI credentials enters `resolveExternalCliAuthProfiles`, returns a runtime-only overlay, and emits the reported DEBUG message. I did not establish a live Windows/Claude CLI 1-2 day refresh cycle.

## Solution Assessment

Unclear. Persisting the fresher external credential would address the stale-profile root cause, but it changes the runtime-only external-CLI credential policy; rate-limiting or state-change logging is safer if the policy is intentional.
