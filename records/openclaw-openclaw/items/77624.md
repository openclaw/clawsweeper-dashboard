## Summary

Keep open: although the reporter later said a fix was tried, current main still resolves inline static auth-profile secrets with trim-only handling and can pass a U+2026 masked key into runtime auth, where Node rejects it as an Authorization header ByteString.

## What This Changes

The issue asks OpenClaw to prevent masked Unicode API keys persisted in per-agent auth profiles from crashing subagent startup.

## Best Possible Solution

Normalize or reject persisted static credential material at the auth-profile resolve and upsert boundaries so malformed masked values become auth failures or fallbacks, not request setup crashes.

## Reproduction Assessment

Yes, as a source-reproducible path: current main preserves U+2026 through auth-profile resolution, and a direct Node `Headers` check throws the reported ByteString TypeError for that value.

## Solution Assessment

Yes: the narrow maintainable fix is to reuse the existing secret normalizer in generic auth-profile credential resolution/upsert paths, not add OpenRouter- or subagent-specific handling.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77624 on current main. Persisted per-agent auth profiles can contain masked Unicode static credentials such as `sk-or-…50ec`; `resolveApiKeyForProfile` currently trims inline `api_key.key` and `token.token` values without the existing ByteString-safe normalizer, and the embedded runner can pass that value into provider Authorization header setup. Reuse `normalizeSecretInput`/`normalizeOptionalSecretInput` at the generic auth-profile resolve boundary for inline static key/token values, normalize any lock-based static credential upsert path that can persist key/token material, and avoid provider-specific OpenRouter handling. Add focused regression coverage proving U+2026 in stored `api_key` and token profiles cannot reach resolved runtime auth/header construction and instead sanitizes or fails as normal auth. Add a single-line `CHANGELOG.md` Fixes entry unless maintainers classify the change as purely internal. Do not change OAuth SecretRef policy or broad auth precedence behavior.
