## Summary

Keep open: current main still has a source-reproducible path where a persisted agent auth profile containing U+2026 is accepted as credential material and can reach provider Authorization header construction, while an existing normalizer was specifically added to prevent ByteString header crashes.

## What This Changes

The issue asks OpenClaw to stop subagent crashes when agent-scoped `auth-profiles.json` contains a masked API key string with U+2026 and to avoid writing masked credentials as real secrets.

## Best Possible Solution

Normalize or reject persisted profile credential material at the auth profile resolution boundary, add regression coverage for masked Unicode credentials, and then separately trace any UI/config path that writes display-truncated secrets back to disk.

## Reproduction Assessment

Yes. Source inspection shows persisted `api_key` values are trimmed but not normalized before runtime auth injection, and a direct Node `Headers` check reproduces the reported ByteString throw for `Bearer sk-or-…50ec`.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing secret normalizer or reject malformed persisted credentials at the auth-profile read/resolve boundary, rather than adding provider-specific OpenRouter handling.

## Repair Work Prompt

Fix the subagent crash reported in https://github.com/openclaw/openclaw/issues/77624. On current main, persisted agent auth profiles can contain masked API key strings such as `sk-or-…50ec`; `coercePersistedAuthProfileStore`/`resolveApiKeyForProfile` preserve that Unicode credential and `auth-controller.ts` can pass it into provider runtime auth, where Node `Headers` throws a ByteString error. Reuse the existing `src/utils/normalize-secret-input.ts` behavior, or reject malformed persisted credentials at the auth-profile read/resolve boundary, so non-Latin-1 credential material cannot reach Authorization header construction and instead fails as a normal auth/config error. Add focused regression coverage for persisted `api_key` profiles containing U+2026. Do not add provider-specific OpenRouter handling and do not introduce a new config option. Investigate the masked-write path only if it is in the same auth-profile save/update code; otherwise leave that as a follow-up in the PR notes. Add a changelog fix entry because this is user-facing.
