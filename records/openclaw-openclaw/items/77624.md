## Summary

Keep open: current main still has a source-reproducible path where a persisted agent auth profile containing U+2026 is treated as a usable credential and can reach provider auth setup, while the existing secret normalizer was written specifically to avoid ByteString header crashes.

## What This Changes

The issue reports that subagent runs can crash when an agent-scoped `auth-profiles.json` stores a display-masked API key such as `sk-or-…50ec`.

## Best Possible Solution

Normalize or reject persisted profile credential material at the auth profile resolution boundary, and add regression coverage proving masked Unicode credentials cannot reach provider Authorization header construction.

## Reproduction Assessment

Yes: source inspection plus a direct Node `Headers` check gives a high-confidence path. A stored `api_key` value with U+2026 is trimmed but not normalized before it can be passed to provider auth, and Node throws the reported ByteString error for that header value.

## Solution Assessment

Yes: the narrow fix should reuse the existing secret normalizer in auth profile read/resolve and any locked upsert path, rather than adding provider-specific OpenRouter handling.
