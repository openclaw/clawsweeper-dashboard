## Label Justifications

## Summary

Keep open: current `main` still has a source-reproducible path where a main-session one-shot can enqueue its system event, receive `runHeartbeatOnce` as `skipped: disabled`, then persist as a skipped disabled `at` job instead of deleting after success. The report's broader claim that every session target and payload kind is affected still needs focused regression coverage.

## What This Changes

The issue reports one-shot `schedule.kind: "at"` cron jobs being skipped as `disabled` and left behind instead of executing and honoring `deleteAfterRun`.

## Best Possible Solution

Change main-session wake-now handling so an already-enqueued system event is not recorded as a skipped disabled one-shot solely because immediate heartbeat is disabled, and add a regression test for the `disabled` heartbeat result plus existing busy-heartbeat cases.

## Reproduction Assessment

Yes from source inspection, but not from an executed local run in this read-only review. A main `systemEvent` one-shot with default `wakeMode: "now"` can receive `runHeartbeatOnce` as `skipped: disabled`, after which `applyJobResult` disables the `at` job instead of deleting it.

## Solution Assessment

Yes, the best fix appears to be in cron's main-session wake-now result handling and one-shot state transition tests. The current code treats heartbeat-disabled wake delivery as the whole cron run outcome even after the system event has already been enqueued.
