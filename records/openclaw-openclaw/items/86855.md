## Summary

Keep open: the linked bug is still real, but this PR is not merge-ready because it suppresses warnings based on `fileTarget`, which current source records from tool arguments before any write/edit success is known.

## What This Changes

The PR adds a `timedOut && fileTarget` warning-suppression branch in `resolveToolErrorWarningPolicy()` plus regression tests for write/edit timeout warning behavior.

## Best Possible Solution

Keep the narrow warning-policy fix, but gate suppression on a source-backed signal that the file mutation completed and only the acknowledgement/response path timed out, with regression coverage for real timeouts that still warn.

## Reproduction Assessment

Yes, at source level: current main routes write/edit `lastToolError` values through the mutating-tool warning branch, so a timed-out mutating write with no user-facing failure acknowledgement produces the extra warning payload. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

No. The requested bug fix is valid, but this implementation uses argument-derived `fileTarget` as success proof; the safer repair needs a post-mutation completion or acknowledgement-timeout signal before suppressing the warning.
