## Summary

Keep open. The feature is not on current main, but this PR still has blocking behavior regressions in the proposed Feishu typing-reaction path and lacks after-fix real Feishu proof.

## What This Changes

Adds Feishu account/channel `ackReaction` config and docs, extends `resolveAckReaction` with a Feishu channel default, and passes the resolved value to Feishu typing reaction creation.

## Best Possible Solution

Land a corrected Feishu reaction-customization feature that preserves empty-disable semantics, validates against Feishu emoji type names before API calls, updates plugin metadata, and leaves native typing replacement to https://github.com/openclaw/openclaw/issues/69572.

## Reproduction Assessment

Yes. The blocking PR regressions are source-reproducible: the shared resolver preserves `""` as disable while the PR falls back to `Typing`, and aliases like `party_parrot` pass the PR regex but are absent from the checked Feishu emoji constants.

## Solution Assessment

No. The direction is maintainable, but the patch should preserve explicit disable, use a Feishu-specific validation or fallback, and keep plugin metadata aligned with the new config field.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Skip typing for empty ackReaction:** `extensions/feishu/src/typing.ts:125-131`
  - body: `resolveAckReaction` treats an explicit empty string as the shared disable signal, but this branch computes `isFeishuCompatible` with `!!resolvedEmoji` and then falls back to `Typing`. Configs such as `messages.ackReaction: ""` or `channels.feishu.accounts.<id>.ackReaction: ""` would still call `messageReaction.create`, so please skip dispatch when the resolved value is exactly empty.
  - confidence: 0.93
- **[P2] Validate Feishu emoji names before sending aliases:** `extensions/feishu/src/typing.ts:125-131`
  - body: The `/^\w+$/` guard filters Unicode emoji but still accepts cross-channel aliases such as `eyes` or `party_parrot` from global `messages.ackReaction`. Those are not Feishu emoji type names in the checked constants, so Feishu can reject the reaction and the non-backoff catch path will silently drop the typing indicator; validate against known Feishu names or fall back to `Typing` before the API call.
  - confidence: 0.86
- **[P3] Update the Feishu plugin manifest schema:** `extensions/feishu/src/config-schema.ts:187`
  - body: Adding `ackReaction` only to the runtime Zod schema leaves `extensions/feishu/openclaw.plugin.json` stale, so discovery/setup surfaces that rely on plugin metadata will not see the new top-level or per-account config field. Please add the field to the manifest schema alongside `typingIndicator`.
  - confidence: 0.78
