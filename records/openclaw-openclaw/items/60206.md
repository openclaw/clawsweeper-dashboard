## Summary

Keep open. The PR addresses a real Feishu parity gap, but the current patch still breaks the shared empty-string disable contract, can forward non-Feishu word aliases into the Feishu API, lacks the required changelog entry, and currently reports a dirty merge state against main.

## What This Changes

The PR adds Feishu account/channel `ackReaction` config and docs, extends shared ack-reaction resolution with a Feishu channel default, and passes the resolved value into Feishu typing-reaction creation with tests.

## Best Possible Solution

Land a corrected Feishu `ackReaction` parity change that preserves explicit disable semantics, rejects or safely falls back for non-Feishu global aliases, documents valid Feishu emoji type names, adds release-note coverage, and leaves the native Typing API question to #69572.

## Reproduction Assessment

Yes. A focused static reproduction is `messages.ackReaction: ""` or `channels.feishu.accounts.<id>.ackReaction: ""`: the shared resolver returns `""`, while the PR typing path falls back to `Typing` and still calls the Feishu reaction API. A second check is `messages.ackReaction: "party_parrot"`, which passes the PR regex but is not a Feishu emoji type in the checked constants.

## Solution Assessment

No. The direction is maintainable, but the current implementation should distinguish explicit disable from invalid-value fallback and should validate Feishu emoji type names more tightly before sending them to the API.

  - body: The shared resolver intentionally returns `""` for explicit disable values, and current tests cover that contract. This branch passes that empty string into `addTypingIndicator`, but `!!resolvedEmoji` makes it fall back to `Typing` and still call `messageReaction.create`, so `messages.ackReaction: ""` or a Feishu account/channel empty override no longer suppresses typing reactions. Validate against known Feishu emoji types or fall back for non-Feishu values before calling the API.
  - confidence: 0.83
