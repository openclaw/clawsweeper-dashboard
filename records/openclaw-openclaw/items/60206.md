## Summary

Keep open. The PR is useful Feishu parity work, but the current diff still breaks the shared empty-string disable contract and accepts non-Feishu word aliases before sending the reaction API call.

## What This Changes

The PR adds Feishu account/channel `ackReaction` config and docs, threads shared ack-reaction resolution into Feishu reply dispatch, and lets Feishu typing reactions use a resolved emoji type.

## Best Possible Solution

Land a corrected Feishu `ackReaction` parity change that preserves explicit disable semantics, validates or falls back for non-Feishu emoji type values, documents the valid Feishu type-name requirement, and leaves the native Typing API direction to #69572.

## Reproduction Assessment

Yes. A focused static reproduction is `messages.ackReaction: ""` or `channels.feishu.accounts.<id>.ackReaction: ""`, where the shared resolver returns `""` but the PR typing path falls back to `Typing` and still calls Feishu reactions; another is `messages.ackReaction: "party_parrot"`, which passes `/^\w+$/` but is not a Feishu emoji type.

## Solution Assessment

No. The overall direction is narrow and maintainable, but the current implementation should distinguish explicit disable from invalid-value fallback and validate Feishu emoji type names more tightly before calling the Feishu API.
