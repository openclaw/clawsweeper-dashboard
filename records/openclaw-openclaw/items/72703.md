## Summary

Keep open. Current main still routes queued runtime system events into user prompt text, and the linked closing PR remains open rather than merged.

## What This Changes

Background task, deferred maintenance, and heartbeat-related system notices are reported as being injected into model input as user-role messages rather than runtime system context.

## Best Possible Solution

Land a narrow fix that routes queued runtime system events through runtime system context while keeping user-authored prompt and transcript fields clean, and split any heartbeat role-policy change into a separate maintainer decision.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: mock or enqueue a formatted system event, run prepared reply assembly, and inspect commandBody/followupRun.prompt versus extraSystemPrompt; I did not run tests because this read-only review should not create local artifacts.

## Solution Assessment

Yes for the queued system-event portion: the prepared-reply prompt assembly seam is the narrow maintainable fix point. The heartbeat-prompt part is not the same fix because current docs intentionally describe heartbeats as user messages.
