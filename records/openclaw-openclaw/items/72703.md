## Summary

Keep open. Current main still sends drained runtime system events through user prompt fields, and the open linked PR remains the tracked implementation path rather than a reason to close this issue before merge.

## What This Changes

The issue asks for background task updates, deferred maintenance notices, and similar runtime-generated notifications to stop entering the user-message prompt stream and instead use system/runtime context.

## Best Possible Solution

Land the linked PR or an equivalent narrow change that routes drained runtime system events into runtime system context, keeps them out of user-authored prompt and transcript fields, and preserves trusted/untrusted ownership semantics.

## Reproduction Assessment

Yes. Current main has a high-confidence source/test reproduction by mocking `drainFormattedSystemEvents` in `runPreparedReply`; the focused test asserts the system line appears in `commandBody` and is absent from runtime system prompt context.

## Solution Assessment

Yes. Moving queued runtime system events at the reply prompt assembly seam is the narrow maintainable fix; task delivery policy and intentional heartbeat user-message behavior should stay separate unless maintainers choose a broader product change.
