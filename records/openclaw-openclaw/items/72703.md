## Summary

Keep open. Current main still routes drained queued system events into user prompt text, and the unmerged closing PR #72761 is the correct implementation path to review rather than closing this issue early.

## What This Changes

The issue requests that background task updates, deferred maintenance notices, and heartbeat/system-event prompts be delivered through system context or another non-user channel instead of user-role prompt text.

## Best Possible Solution

Land #72761 or an equivalent narrow change that routes drained runtime system events into extraSystemPrompt or another non-user runtime context, keeps them out of commandBody, followupRun.prompt, and transcript prompt text, and preserves the existing trusted/untrusted event ownership behavior.

## Reproduction Assessment

Yes. Current main provides a high-confidence source and test reproduction path: mocked drainFormattedSystemEvents output is asserted to appear in commandBody/user prompt text and not in extraSystemPrompt.

## Solution Assessment

Yes. Moving drained runtime system events to runtime system context is the narrow maintainable fix for the reported queued-event bug; changing heartbeat prompt semantics should remain a separate maintainer-scoped decision if desired.
