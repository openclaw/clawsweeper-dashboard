## Summary

Keep open. Current main still routes drained runtime system events into prompt-body fields that are used as user-visible model input, and the open closing PR is the right implementation path to review rather than closing this issue before merge.

## What This Changes

The issue asks for background task updates, deferred maintenance notifications, and related runtime system events to be delivered as system context instead of user-role prompt text.

## Best Possible Solution

Land #72761 or an equivalent narrow change that moves drained runtime system events into `extraSystemPrompt` or another runtime system-context channel, keeps them out of `commandBody`, `followupRun.prompt`, and transcript prompt text, and preserves trusted/untrusted event ownership semantics.

## Reproduction Assessment

Yes. The current source and focused tests provide a high-confidence reproduction path: mock `drainFormattedSystemEvents`, run `runPreparedReply`, and inspect that the event appears in `commandBody` and `followupRun.prompt` instead of runtime system context.

## Solution Assessment

Yes for the queued runtime system-event bug. Routing drained events to runtime system context is the narrow maintainable fix; changing heartbeat prompt semantics should be handled separately if maintainers want that broader behavior change.
