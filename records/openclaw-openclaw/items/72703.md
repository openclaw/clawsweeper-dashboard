## Summary

Keep open. Current main still routes drained runtime system events into prompt-body fields used as user-visible model input, and the open closing PR is the appropriate implementation path to review instead of closing this issue early.

## What This Changes

The issue requests that background task updates, deferred maintenance notifications, and heartbeat prompts avoid being injected as user-role/user prompt messages.

## Best Possible Solution

Land #72761 or an equivalent narrow change that moves drained runtime system events into extraSystemPrompt or another runtime system-context channel, keeps them out of commandBody, followupRun.prompt, and transcript prompt text, and preserves trusted/untrusted event ownership semantics.

## Reproduction Assessment

Yes. Current main has a high-confidence source/test reproduction: a mocked drainFormattedSystemEvents result is asserted to appear in commandBody and followupRun.prompt, and background task progress reaches that same queue via queueTaskSystemEvent.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable fix for queued runtime system events; heartbeat prompt semantics should remain a separate maintainer decision if product behavior needs to change there.
