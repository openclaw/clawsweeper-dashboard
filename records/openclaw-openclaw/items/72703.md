## Summary

Keep open. Current main still routes drained runtime system events into user prompt fields, and the open closing PR remains the tracked implementation path rather than grounds to close this issue before merge.

## What This Changes

The issue requests that background task updates, deferred maintenance notifications, and similar runtime-generated notices be delivered as system/runtime context instead of user-role prompt text.

## Best Possible Solution

Land PR #72761 or an equivalent narrow change that routes drained runtime system events into runtime system context, keeps them out of user-authored prompt and transcript fields, and preserves trusted/untrusted ownership semantics.

## Reproduction Assessment

Yes. Current main has a high-confidence source/test reproduction by mocking `drainFormattedSystemEvents` in `runPreparedReply`; the focused test asserts the system line appears in `commandBody` and `followupRun.prompt`.

## Solution Assessment

Yes. Moving queued runtime system events at the reply prompt assembly seam is the narrow maintainable fix; task delivery policy and intentional heartbeat user-message behavior should stay separate unless maintainers choose a broader product change.
