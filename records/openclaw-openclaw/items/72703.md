## Summary

Keep open. Current main still routes drained runtime system events into prompt-body fields that become user-visible model input, and the open closing PR is the right implementation path to review before this issue is closed.

## What This Changes

Move background task updates, deferred maintenance notices, and other queued runtime system events out of user-role prompt text and into system-role or runtime system context.

## Best Possible Solution

Land #72761 or an equivalent narrow change that routes drained runtime system events into extraSystemPrompt or another runtime system-context channel, keeps them out of commandBody, followupRun.prompt, and transcript prompt text, and preserves trusted/untrusted event ownership semantics.

## Reproduction Assessment

Yes. Current main has a high-confidence source/test reproduction: mocked drainFormattedSystemEvents output is asserted to appear in commandBody and followupRun.prompt, and background task progress reaches the same queue through enqueueSystemEvent.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable fix for queued runtime system events; broader heartbeat prompt semantics should be handled separately if maintainers want that behavior changed.
