## Summary

Keep open. Current main still routes drained runtime system events into user prompt fields, and the open unmerged closing PR is the active implementation path rather than a reason to close this issue before merge.

## What This Changes

Route background task and deferred runtime system-event updates into system context instead of user prompt/message fields.

## Best Possible Solution

Land PR #72761 or an equivalent narrow change that moves drained runtime system events into runtime system context while keeping user-authored prompt and transcript fields clean.

## Reproduction Assessment

Yes. Current main has a source-backed reproduction by mocking `drainFormattedSystemEvents` in `runPreparedReply`; focused tests already assert the event text appears in `commandBody` and `followupRun.prompt`.

## Solution Assessment

Yes for the system-event portion. Moving drained runtime system events at the reply prompt assembly seam is the narrow maintainable fix; heartbeat prompt role policy should stay a separate explicit maintainer decision.
