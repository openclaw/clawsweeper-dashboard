## Summary

Keep open. Current main still drains queued runtime system events into prompt-body fields used for user-visible model input, and the open closing PR is still the right implementation path to review rather than closing this issue early.

## What This Changes

The issue asks for background task updates, deferred maintenance notices, and related runtime notifications to be delivered as system context instead of user-role prompt text.

## Best Possible Solution

Land #72761 or an equivalent narrow change that routes drained runtime system events into `extraSystemPrompt` or another runtime system-context channel, keeps them out of `commandBody`, `followupRun.prompt`, and transcript prompt text, and preserves the existing trusted/untrusted event ownership behavior.

## Reproduction Assessment

Yes. The current source and focused tests provide a high-confidence reproduction path: mocked `drainFormattedSystemEvents` output is asserted to appear in `commandBody` and `followupRun.prompt` rather than runtime system context.

## Solution Assessment

Yes for the queued runtime system-event bug. Moving drained events to runtime system context is the narrow maintainable fix; changing heartbeat prompt semantics should be a separate maintainer-scoped decision if desired.
