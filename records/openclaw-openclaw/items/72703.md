## Summary

Keep open. Current main still routes drained runtime system events into prompt-body fields used as user-visible model input, and the open closing PR is the right implementation path to review rather than closing this issue early.

## What This Changes

The issue asks to keep runtime-generated background task and maintenance updates out of user-role prompt text by routing them through system/runtime context or a dedicated non-user channel.

## Best Possible Solution

Land #72761 or an equivalent narrow change that moves drained runtime system events into extraSystemPrompt or another runtime system-context channel, keeps them out of commandBody, followupRun.prompt, and transcript prompt text, and preserves existing trusted/untrusted event ownership semantics.

## Reproduction Assessment

Yes. Current main has a high-confidence source/test reproduction: mocked drainFormattedSystemEvents output is asserted to appear in commandBody and followupRun.prompt, while background task progress reaches the same queue via enqueueSystemEvent.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable fix for queued runtime system events; broader heartbeat prompt semantics should be handled separately if maintainers want that policy changed.
