## What This Changes

The PR adds a TUI `historyLimit === 0` branch that clears local chat state and skips `client.loadHistory`, plus a regression test for that branch.

## Best Possible Solution

Either intentionally support `--history-limit 0` end to end by updating CLI validation, focused docs/tests, and the current session-clearing/remembering path, or keep the positive-integer CLI contract and close this PR as not planned.

## Reproduction Assessment

Yes, partially by source inspection: a direct `TuiOptions.historyLimit = 0` still reaches `client.loadHistory({ limit: 0 })`, which violates the Gateway schema. The public CLI path on current main now rejects zero before the TUI path, so the advertised disable-history flow is not currently reproduced end to end.

## Solution Assessment

No. The lower-level guard is plausible, but it is not the best current-base solution unless the CLI contract is updated and the current session-state side effects are preserved.
