## Summary

Keep open: current main still forwards `--history-limit 0` to the gateway, so the bug is source-reproducible, but this external PR needs real behavior proof and two branch fixes before merge.

## What This Changes

The PR adds a TUI zero-history branch that skips `chat.history` when `historyLimit === 0` and adds a regression test.

## Best Possible Solution

Update the branch to keep the TUI-side zero-history guard while preserving current session-load side effects and package imports, then merge after redacted terminal/live proof is added.

## Reproduction Assessment

Yes, by source inspection: the CLI preserves `0`, TUI forwards it as `limit: 0`, and the gateway `chat.history` schema rejects limits below `1`. I did not run the TUI flow because this review must keep the checkout read-only.

## Solution Assessment

No, not as written. Handling zero before calling `chat.history` is the right boundary, but the patch needs to preserve session remembering and update the stale test import before it is the narrow maintainable fix.
