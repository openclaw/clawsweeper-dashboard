## Summary

Keep open: current main still can forward a programmatic TUI `historyLimit: 0` to `chat.history`, but this PR is not merge-ready because it only adds a lower-level guard while the public CLI still rejects zero and current main has newer selected-session side effects the branch does not preserve.

## What This Changes

The PR adds a TUI `loadHistory` early return for `historyLimit === 0` and a focused unit test that expects no gateway history call.

## Best Possible Solution

Either support zero-history mode end to end at the CLI and TUI session boundary while preserving current session side effects, or keep the positive-integer public contract and avoid merging the lower-level-only guard.

## Reproduction Assessment

Yes, by source inspection for the lower-level TUI options path: `createSessionActions` forwards `historyLimit: 0` to `chat.history`, whose protocol schema requires `limit >= 1`. The public command in the PR body is not reproduced end to end because current CLI parsing rejects zero before `runTui`.

## Solution Assessment

No, not yet. The lower-level guard is plausible, but it is not the best current-base fix unless maintainers accept zero as a public no-history mode, update CLI validation/docs/tests, and preserve current selected-session side effects.
