## Summary

Keep this PR open, but it is not merge-ready: the zero-history idea is useful, current main still cannot support the advertised CLI flow, and the branch needs a rebase plus real behavior proof before maintainers can judge it cleanly.

## What This Changes

The PR adds a TUI zero-history branch that skips gateway history loading when `historyLimit` is `0` and adds regression coverage for that path.

## Best Possible Solution

Make zero-history an end-to-end TUI contract only if maintainers want that mode: allow zero at CLI parsing, document/test it, skip `chat.history`, and preserve current session-clearing plus session-remembering behavior.

## Reproduction Assessment

Yes, by source inspection: current main rejects `--history-limit 0` at `src/cli/tui-cli.ts:47`, and a direct `TuiOptions.historyLimit = 0` would reach `client.loadHistory({ limit: 0 })` in `src/tui/tui-session-actions.ts:411`. I did not run the CLI or tests because this review is read-only.

## Solution Assessment

No. The internal guard is a plausible partial fix, but the best current-main solution must decide the public zero-limit contract, update CLI/docs/tests, and preserve session remembering.
