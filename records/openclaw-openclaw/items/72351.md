## Summary

Keep open: the zero-history bug is source-reproducible and the TUI-side guard is the right boundary, but the current PR head still has two P2 correctness blockers and lacks required real behavior proof.

## What This Changes

The PR adds a `historyLimit === 0` branch in TUI history loading plus a regression test so zero history skips `chat.history`.

## Best Possible Solution

Keep the TUI-side zero-history guard, update it to preserve session remembering and current package imports, then merge after redacted terminal or live-output proof is posted.

## Reproduction Assessment

Yes, by source inspection: the CLI preserves `0`, TUI forwards it as `limit: 0`, and the gateway `chat.history` schema rejects limits below `1`. I did not run the TUI flow because this review is read-only.

## Solution Assessment

No, not as written. Handling zero before calling `chat.history` is the narrow owner-boundary fix, but this branch must preserve session remembering and use the current pi-tui package name.
