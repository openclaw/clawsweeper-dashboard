## Summary

Keep open: the underlying TUI bug is source-reproducible on current main, but this PR should not merge until the zero-history path preserves remembered sessions, the test is rebased to the current pi-tui package, and real behavior proof is posted.

## What This Changes

The PR adds a TUI `historyLimit === 0` branch that skips gateway history loading and adds a regression test for `openclaw tui --history-limit 0`.

## Best Possible Solution

Implement the TUI-side zero-history guard while preserving normal session-load side effects, use current package imports in the regression test, document `0` if it is intended public behavior, and merge only after redacted terminal/live proof is added.

## Reproduction Assessment

Yes, by source inspection: the CLI preserves `0`, TUI forwards it as `limit: 0`, and the gateway `chat.history` schema rejects limits below `1`. I did not run the TUI command because this review was required to keep the checkout read-only.

## Solution Assessment

No, not as written. Handling zero before calling `chat.history` is the right boundary, but the patch needs to preserve session remembering and update the stale test import before it is the narrow maintainable fix.
