## Summary

Keep open: current main still has the zero-limit TUI bug and the PR fixes the right boundary, but the patch needs two correctness updates plus redacted real behavior proof before merge.

## What This Changes

The PR adds an early `historyLimit === 0` branch in TUI history loading that skips backend history requests and adds a regression test.

## Best Possible Solution

Keep the TUI-side zero guard, preserve selected-session persistence, update the test to the current pi-tui package, cover the persistence side effect, and merge after redacted real behavior proof is posted.

## Reproduction Assessment

Yes, by source inspection: the CLI preserves `0`, the TUI forwards it as `limit: 0`, and the gateway schema rejects limits below `1`. I did not run the interactive TUI command because this review is read-only.

## Solution Assessment

No, not as written. Handling zero in TUI before calling `chat.history` is the right boundary, but the early return must keep `rememberSessionKey` behavior and the test must use the current package import.
