## Best Possible Solution

Either keep the shipped positive-integer CLI contract, or implement zero-history mode end to end through CLI parsing, docs, CLI smoke coverage, the current session-state helper path, and real TUI proof.

## Reproduction Assessment

Yes for the lower-level options path: source inspection shows `createSessionActions` forwards `opts.historyLimit` to `chat.history`, whose schema rejects `0`. The exact public command does not reproduce on current main or `v2026.6.8` because CLI validation rejects zero before `runTui`.

## Solution Assessment

No, not yet. A lower-level guard is only plausible after maintainers accept zero-history semantics and the branch is rebased to update CLI parsing, docs/tests, current package imports, and remembered-session side effects.
