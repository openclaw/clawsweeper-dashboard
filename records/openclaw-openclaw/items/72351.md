## What This Changes

The PR adds a TUI zero-history branch that skips `chat.history` when `historyLimit` is `0` and adds a regression test for that branch.

## Best Possible Solution

Either intentionally support `--history-limit 0` end to end by updating CLI validation, docs/tests, and the current session-clearing/remembering path, or keep the positive-integer CLI contract and close this PR as not planned.

## Reproduction Assessment

Yes, by source inspection: a direct `TuiOptions.historyLimit = 0` still reaches `client.loadHistory({ limit: 0 })`, while the public CLI on current main now rejects zero before that path. I did not run the CLI or tests because this review is read-only.

## Solution Assessment

No. The lower-level guard is plausible, but it is not the best current-base solution unless it also updates the CLI contract and preserves the current session-state side effects.
