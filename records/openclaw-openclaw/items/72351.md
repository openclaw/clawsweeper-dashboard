## Summary

Keep this PR open: current main has not implemented zero-history TUI behavior, but the branch is not merge-ready because it changes public CLI semantics below the parser, is stale against current TUI dependencies, and still lacks real behavior proof.

## What This Changes

The PR adds a TUI history-loading branch for `historyLimit === 0` plus a regression test for avoiding a gateway `chat.history` call.

## Best Possible Solution

Either keep the shipped positive-integer CLI contract, or implement zero-history mode end to end through CLI parsing, docs, CLI smoke coverage, and the current `clearDisplayedSession`/`rememberSessionKey` path.

## Reproduction Assessment

Yes for the lower-level options path: source inspection shows `createSessionActions` would forward `opts.historyLimit: 0` to `chat.history`, whose schema rejects zero. The exact public command does not reproduce on current main or v2026.6.9 because CLI validation rejects zero first.

## Solution Assessment

No, not yet. A lower-level guard is only a plausible piece after maintainers accept zero-history semantics and the branch is rebased to update CLI parsing, docs/tests, current package imports, and remembered-session side effects.
