## Summary

Keep this PR open. Current main still forwards `--history-limit 0` into `chat.history`, whose schema rejects zero, and this PR fixes that path narrowly with regression coverage; the remaining blocker is the required docs and changelog update for a user-facing TUI behavior change.

## What This Changes

The PR adds a TUI-side `historyLimit === 0` no-history branch that clears local display state, skips the gateway history request, refreshes session info, and adds a regression test.

## Best Possible Solution

Merge the narrow TUI-side guard after documenting that `--history-limit 0` disables initial history loading and adding a user-facing changelog entry; keep the gateway `chat.history` RPC contract restricted to positive limits.

## Reproduction Assessment

Yes. Current main parses `--history-limit 0`, forwards `limit: 0` to `client.loadHistory`, and the gateway `chat.history` schema rejects limits below 1; the PR test captures the expected no-gateway-call path.

## Solution Assessment

Yes for the runtime fix. Handling zero in the TUI before calling `chat.history` is narrower than relaxing the gateway protocol for every client, but the public docs and changelog should be updated before merge.
