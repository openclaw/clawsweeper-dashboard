## Summary

Keep this PR open: current main still has the zero-limit failure and the TUI-side guard is the right direction, but the patch is not ready against current main because the added test uses a stale pi-tui package name, the zero-history branch skips selected-session persistence, and the external PR lacks after-fix real behavior proof.

## What This Changes

The PR adds a `historyLimit === 0` TUI path that clears local chat state, renders the session marker, skips backend history loading, and adds a regression test.

## Best Possible Solution

Keep the TUI-side zero guard, but update it to preserve session remembering, rebase the test to current dependencies, document the zero-history semantics, and merge only after redacted real behavior proof is added.

## Reproduction Assessment

Yes, by source inspection against current main: the CLI preserves `0`, the TUI forwards it as `limit: 0`, and the gateway schema rejects limits below `1`. I did not run the local command because this review must keep the checkout read-only.

## Solution Assessment

No, not as written. Handling zero in the TUI before calling `chat.history` is the right owner boundary, but this patch must preserve the existing session-remembering side effect and update stale test imports before merge.
