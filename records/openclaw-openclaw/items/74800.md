## Summary

Keep open: current main still returns plugin command handler results without normalization and then dereferences the result in the generic and Telegram native command paths. The issue is already paired with open PR #74989, which uses closing syntax and targets the same bug, so the right cleanup path is to review and land or close that PR rather than close this issue now.

## What This Changes

The issue asks OpenClaw to tolerate plugin command handlers that return undefined so Telegram/custom command dispatch does not crash with the generic fallback.

## Best Possible Solution

Review and land #74989, or an equivalent narrow fix, that normalizes nullish/non-object plugin command returns at executePluginCommand, preserves continueAgent behavior, defensively protects generic and Telegram native consumers, and adds focused regression coverage.

## Reproduction Assessment

Yes. A registered plugin command whose handler completes without returning a payload follows the current source path from executePluginCommand to an unguarded result.continueAgent read; the Telegram native path has the same unsafe raw-result consumption.

## Solution Assessment

Yes. Normalizing at the shared plugin command executor boundary is the narrowest maintainable fix, with caller-side guards in the generic and Telegram native paths as defense-in-depth.
