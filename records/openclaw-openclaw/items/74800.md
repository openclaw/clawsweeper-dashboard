## Summary

Keep open: current main still returns plugin command handler results unchanged and dereferences the result in the shared plugin-command path, while the Telegram native path also reads the raw result. PR #74989 is the open closing implementation candidate, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Handle Telegram/plugin command handlers that return no payload without surfacing the generic Telegram dispatch error.

## Best Possible Solution

Normalize nullish and non-object plugin command returns at the shared executor boundary, keep defensive caller guards, and add regression coverage for shared and Telegram-native command paths.

## Reproduction Assessment

Yes. A plugin command handler that completes without returning a payload can flow through `executePluginCommand` unchanged and then hit `result.continueAgent` in `handlePluginCommand`; the Telegram native path has the same raw-result consumption risk.

## Solution Assessment

Yes. The narrow maintainable direction is the one described by PR #74989: normalize at the executor boundary and preserve `continueAgent` semantics with focused regression tests.
