## Summary

Keep open: current main still returns the raw plugin command handler result and dereferences it in the generic command path, while the Telegram native path also consumes the raw result. Open PR #74989 is the focused implementation candidate, but it is still open and unmerged in the provided GitHub context.

## What This Changes

Guard Telegram/plugin command handling so undefined command handler results do not crash dispatch and instead become a no-op reply or clear handler error.

## Best Possible Solution

Ship a narrow fix that normalizes nullish and non-object plugin command returns at the shared executor boundary, keeps defensive caller guards, and locks the behavior with regression tests and a changelog entry.

## Reproduction Assessment

Yes. A plugin command handler that completes without returning a payload can flow through `executePluginCommand` unchanged and then hit `result.continueAgent` in `handlePluginCommand`; the Telegram native path has the same raw-result consumption.

## Solution Assessment

Yes. Normalizing at the shared executor boundary with defensive caller guards is the narrow maintainable direction, and PR #74989 is the current candidate to review rather than closing this issue before merge.
