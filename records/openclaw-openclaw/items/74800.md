## Summary

Keep open: current main still returns raw plugin command handler results and then dereferences `result.continueAgent`, so an undefined handler return can still trigger the reported Telegram failure. PR #74989 is the focused closing candidate, but it remains open and unmerged in the provided GitHub context.

## What This Changes

A Telegram/plugin command handler that returns no payload should not crash command dispatch or surface the generic Telegram error.

## Best Possible Solution

Land PR #74989 or an equivalent narrow fix that normalizes nullish and non-object plugin command returns at the executor boundary, keeps caller-side guards, and adds regression tests plus a changelog entry.

## Reproduction Assessment

Yes. A plugin command handler that completes without returning a payload can flow through `executePluginCommand` unchanged and then hit `result.continueAgent` in `handlePluginCommand`; the Telegram native command path has the same raw-result consumption.

## Solution Assessment

Yes. The proposed fix direction is the narrow maintainable solution because it protects all plugin command callers at the shared executor boundary while preserving `continueAgent` behavior and adding defensive caller coverage.
