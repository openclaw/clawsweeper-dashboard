## Summary

Keep open: current main still allows a plugin command handler to return `undefined` and then dereferences that result in the shared command path; PR #74989 is the open, unmerged closing candidate for the same failure.

## What This Changes

Telegram plugin command handlers that return no payload should be normalized so command dispatch does not throw and show the generic Telegram error.

## Best Possible Solution

Land or close PR #74989 after review; the desired end state is executor-level normalization with defensive caller guards, regression tests, and the required changelog entry.

## Reproduction Assessment

Yes. A plugin command handler that completes without returning a payload can flow through `executePluginCommand` unchanged and then hit `result.continueAgent` in `handlePluginCommand`; the Telegram native path has the same raw-result exposure.

## Solution Assessment

Yes. Normalizing nullish and non-object results at the shared executor boundary, with defensive guards at the two command consumers, is the narrow maintainable fix because it protects current and future plugin command callers without changing command ownership.
