## Summary

Keep open. Current main still returns raw plugin command handler results and dereferences them in the generic and Telegram native command paths, so the reported undefined-result crash remains valid. The issue is also paired with open closing PR #74989, so the right cleanup path is PR review/merge rather than closing this issue first.

## What This Changes

The issue requests defensive handling for Telegram/plugin command handlers that return undefined so users do not see the generic Telegram failure message.

## Best Possible Solution

Review and land PR #74989, or an equivalent narrow fix, that normalizes nullish and non-object plugin command returns at executePluginCommand, keeps defensive guards in the generic and Telegram native consumers, preserves continueAgent behavior, adds regression tests, and includes the required user-facing changelog entry.

## Reproduction Assessment

Yes. A plugin command handler that completes without returning a payload follows the current source path from executePluginCommand to result.continueAgent in handlePluginCommand; the Telegram native path has the same unsafe raw-result consumption.

## Solution Assessment

Yes. Normalizing at the shared executor boundary is the narrowest maintainable fix, with caller-side guards as defense-in-depth for mocks or wrappers that bypass the shared executor.
