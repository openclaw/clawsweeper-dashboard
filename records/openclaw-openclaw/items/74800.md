## Summary

Keep open: current `main` still returns the raw plugin command handler value and dereferences it in the generic command path, while the Telegram native path also consumes the raw result. PR #74989 is the focused closing implementation candidate, but it is still open and unmerged.

## What This Changes

The issue requests nullish-result hardening for Telegram/plugin command handlers that complete without returning a reply payload.

## Best Possible Solution

Ship a narrow fix that normalizes nullish and non-object plugin command returns at the shared executor boundary, keeps defensive caller guards, and locks the behavior with regression tests and a changelog entry.

## Reproduction Assessment

Yes. A plugin command handler that completes without returning a payload can flow through `executePluginCommand` unchanged and then hit `result.continueAgent` in `handlePluginCommand`; the Telegram native path has the same raw-result consumption.

## Solution Assessment

Yes. The direction in PR #74989 is the narrow maintainable fix because it protects all plugin command callers at the executor boundary while preserving `continueAgent` behavior and adding caller-side defense.
