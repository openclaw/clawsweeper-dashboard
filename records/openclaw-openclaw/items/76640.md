## Summary

Keep open. Current main and v2026.5.2 still reopen saved media temp files read-only and await `handle.sync()`; Telegram inbound media uses that shared path before transcription, and open #76593 targets the same bug but is not merged.

## What This Changes

The issue asks OpenClaw to fix Telegram voice/media downloads on Windows when media persistence fails during temp-file `fsync` before Groq transcription.

## Best Possible Solution

Review and land #76593, or an equivalent shared media-store fix, with the required changelog entry and focused regression coverage while preserving atomic rename and existing media security checks.

## Reproduction Assessment

No live high-confidence Windows reproduction was run. The source-level reproduction is clear: make `FileHandle.sync()` fail with Windows-style `EPERM` inside `saveMediaBuffer()` and Telegram will hit the current media failure reply.

## Solution Assessment

Yes. The best fix belongs in the shared media-store temp-file persistence helper; #76593's `r+` handle change is the narrow code direction, but it still needs changelog and merge review.
