## Summary

Keep open. Current checked-out main still lets `FileHandle.sync()` errors propagate from the shared media-store write path into Telegram's inbound media failure reply, and open PR #76752 already references this issue with closing syntax.

## What This Changes

Telegram media downloads on Windows should keep working when a successful file write is followed by `EPERM: operation not permitted, fsync`.

## Best Possible Solution

Land a narrow shared media-store fix, likely through PR #76752 or an equivalent patch, that treats only successful-write Windows `EPERM` sync failures as best-effort while preserving other errors, rename behavior, cleanup, tests, and changelog coverage.

## Reproduction Assessment

Yes. A source-level reproduction can mock `FileHandle.sync()` inside `saveMediaBuffer()` to throw Windows `EPERM` after `fs.writeFile()` succeeds, and the issue discussion now also includes a live Windows 2026.5.2 log showing the same Telegram failure.

## Solution Assessment

Yes. The narrowest maintainable fix is in the shared media-store write path with focused regression coverage; because PR #76752 is already open with closing syntax, this issue should stay open until that PR or an equivalent fix lands.
