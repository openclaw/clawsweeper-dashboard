## Summary

Keep open. Current main still routes Telegram media downloads through `saveMediaBuffer()`, whose temp-file `FileHandle.sync()` error would propagate into the Telegram media failure reply; the related Telegram media reports cover network fallback, not this Windows `fsync` failure mode.

## What This Changes

The issue reports that Telegram voice/media downloads on Windows fail with `EPERM: operation not permitted, fsync` before Groq audio transcription can consume the file.

## Best Possible Solution

Keep the issue open and fix the shared media-store write path to tolerate the Windows temp-file `fsync` failure narrowly, with focused regression coverage and a user-facing changelog entry.

## Reproduction Assessment

No live Windows reproduction was established in this read-only review. Source inspection gives a clear reproduction path: mock `FileHandle.sync()` to throw Windows `EPERM` during `saveMediaBuffer()` and verify Telegram falls into the current media failure reply.

## Solution Assessment

Yes, a narrow media-store fix is the best direction. The safer repair is to tolerate only known unsupported/permission sync errors after a successful temp write, while leaving actual writes, size checks, renames, and cleanup semantics intact.

Fix https://github.com/openclaw/openclaw/issues/76640. Current Telegram inbound media downloads eventually call `saveMediaBuffer()` in `src/media/store.ts`; on Windows, a successful temp-file write can still fail at `FileHandle.sync()` with `EPERM`, causing Telegram to send `Failed to download media` and preventing voice transcription.
