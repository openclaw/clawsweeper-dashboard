## Summary

Keep open. This is a valid source-backed Windows regression, and the open closing PR #76752 is the right implementation candidate, so this issue should stay open until that PR lands or is replaced.

## What This Changes

Report that Telegram voice/image media downloads on Windows npm 2026.5.2 fail with `EPERM` during media-store `fsync` and return the generic failed-to-download message.

## Best Possible Solution

Review and land #76752 or an equivalent narrow media-store fix with focused mocked `fsync` `EPERM` regression coverage and a changelog entry, then let the closing reference resolve this issue.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: make `FileHandle.sync()` throw `EPERM` during `saveMediaBuffer()` and Telegram will send the reported failure response. The discussion also includes a live Windows 2026.5.2 reproduction, but I did not run a Windows current-main test locally.

## Solution Assessment

Yes, fixing the shared `writeSavedMediaBuffer()` sync handling is the narrow maintainable direction. The issue should not be closed early because #76752 is still open and specifically references this issue with closing syntax.
