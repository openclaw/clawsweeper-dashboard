## Summary

Keep open. Current main still propagates `FileHandle.sync()` failures from the shared media save path into Telegram’s media-download error reply, and #76752 is an open, unmerged closing PR for this exact failure mode.

## What This Changes

Telegram media downloads on Windows should tolerate a post-write `fsync` `EPERM` so images and voice notes can be saved and processed instead of failing with the generic media-download reply.

## Best Possible Solution

Land or replace #76752 with a narrow media-store fix that treats only known post-write sync permission/unsupported errors as best-effort, preserves write/rename/cleanup failures, adds focused regression coverage, and includes a user-facing changelog entry.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: make `handle.sync()` throw a Windows-style `EPERM` after `saveMediaBuffer()` writes the temp file, then Telegram reaches the existing media failure reply. A live Windows comment also reports the same `fsync` error on 2026.5.2 and recovery after downgrading to 2026.4.22.

## Solution Assessment

Yes, the narrow shared media-store direction is the best fix boundary because Telegram uses that path for inbound media and the failure occurs after a successful temp write. The issue should stay open while the linked closing PR is reviewed, merged, or replaced.
