## Summary

Keep open. Current main still lets `FileHandle.sync()` failures propagate from the shared media-store write path into Telegram's media failure reply, and open PR #76752 already references this issue with closing syntax, so this should remain open until that fix or an equivalent one lands.

## What This Changes

Telegram media downloads on Windows should tolerate `EPERM` from the shared media-store `fsync` path so voice notes and attachments can be saved and transcribed.

## Best Possible Solution

Land #76752 or an equivalent narrow shared media-store fix that treats only successful-write sync `EPERM`/related Windows flush failures as best-effort while preserving other errors, rename behavior, cleanup, tests, and changelog coverage.

## Reproduction Assessment

Yes, at source level: mock the temp-file `FileHandle.sync()` inside `saveMediaBuffer()` to throw Windows `EPERM` after `fs.writeFile()` succeeds, then verify Telegram reaches the current media failure reply. A live Windows comment also reports 2026.5.2 failing and 2026.4.22 working.

## Solution Assessment

Yes. The narrowest maintainable direction is a shared media-store fix plus focused regression coverage; because #76752 is an open closing PR, this issue should stay open while maintainers review or replace that PR.
