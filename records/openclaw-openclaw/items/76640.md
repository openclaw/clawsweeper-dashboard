## Summary

Keep open. Current main still awaits temp-file `fsync` in `saveMediaBuffer()` without an `EPERM` guard, Telegram media downloads route through that helper, and #76752 is an open unmerged closing PR rather than a reason to close this issue.

## What This Changes

The issue reports a Windows regression where Telegram voice/image media downloads fail with `EPERM: operation not permitted, fsync` during media persistence.

## Best Possible Solution

Review and land #76752 or a narrow replacement that handles Windows `EPERM` from temp-file fsync in the shared media store, with focused regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes. The issue discussion now has a Windows 11/npm reproduction on 2026.5.2 plus a downgrade-to-2026.4.22 comparison, and source inspection shows current main still routes Telegram media through an unguarded `handle.sync()` in `saveMediaBuffer()`.

## Solution Assessment

Yes, the shared media-store path is the right repair boundary. A Telegram-only workaround would leave the same `saveMediaBuffer()` failure available to other inbound media callers; the active PR should be reviewed for narrow errno handling and regression coverage before merge.
