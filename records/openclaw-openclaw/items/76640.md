## Summary

Keep open. Current main still propagates `FileHandle.sync()` failures from the shared media-store write path into Telegram's media failure reply, the Windows follow-up confirms the regression on v2026.5.2, and open PR #76752 already references this issue with closing syntax.

## What This Changes

Fix Telegram inbound media downloads on Windows so successful media writes are not failed by `EPERM` from `fsync` before transcription or media handling.

## Best Possible Solution

Land or replace #76752 with a narrow shared media-store fix that treats only successful-write Windows `EPERM` sync failures as best-effort while preserving other write, rename, cleanup, test, and changelog behavior.

## Reproduction Assessment

Yes, at source level: make `handle.sync()` throw `EPERM` after `fs.writeFile()` succeeds in `saveMediaBuffer()`, then Telegram reaches the existing media failure reply. The live Windows comment gives additional v2026.5.2 confirmation, but I did not run a live current-main Windows case.

## Solution Assessment

Yes. The narrowest maintainable fix is in the shared media-store write path, not Telegram-specific handling, because Telegram reaches the failure through `saveMediaBuffer()` and other errors should still propagate.
