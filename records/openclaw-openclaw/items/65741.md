## Summary

Keep open: the `supergroup` normalization is a plausible narrow fix, but the PR also changes Telegram group authorization by treating route bindings as permission, targets a removed `src/telegram` handler path for the group gate, and still lacks after-fix real Telegram proof.

## What This Changes

The branch maps `supergroup` to `group`, adds a Telegram-only hot-reload restart delay, and changes legacy Telegram handler code so explicit peer bindings bypass group allowlist drops.

## Best Possible Solution

Land a focused current-main replacement: add `supergroup` normalization with regression coverage, keep Telegram group authorization in `channels.telegram.groups` plus sender allowlists unless maintainers approve and document a new binding authorization model, and only add reload 409 mitigation with targeted proof.

## Reproduction Assessment

Source-level yes for the normalization and group-gate claims: current main returns `undefined` for `normalizeChatType("supergroup")`, and the active handler returns before inbound processing when the group allowlist/policy rejects a message. No high-confidence live reproduction was established for the hot-reload 409 claim.

## Solution Assessment

No. The normalization change is narrow, but the binding bypass crosses the documented Telegram authorization boundary and the handler patch targets a removed path; the safer path is a current plugin-runtime patch plus explicit maintainer approval for any authorization change.
