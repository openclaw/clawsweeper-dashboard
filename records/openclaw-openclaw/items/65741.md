## Summary

Keep open: the `supergroup` normalization looks like useful bug-fix work, but this PR is not mergeable as-is because it patches a removed Telegram handler path, changes Telegram group authorization semantics, and has no real Telegram behavior proof.

## What This Changes

The branch maps `supergroup` to `group`, adds a Telegram-only hot-reload restart pause, and lets explicit Telegram peer bindings bypass group allowlist drops in the legacy handler.

## Best Possible Solution

Land a focused current-main replacement: add `supergroup` normalization with regression coverage, keep Telegram group authorization in `channels.telegram.groups` plus sender allowlists unless maintainers approve and document a new binding authorization model, and add any reload 409 mitigation only with proof and a targeted test.

## Reproduction Assessment

Source-level yes for the normalization and group-gate paths: current main still returns `undefined` for `normalizeChatType("supergroup")`, and the active handler returns before inbound processing when group allowlist/policy denies access. No high-confidence live reproduction was established for the hot-reload 409 claim.

## Solution Assessment

No. The normalization change is narrow, but the PR's binding bypass crosses the documented Telegram authorization boundary and the handler patch targets a removed path; the safer fix is a current plugin-runtime patch plus explicit maintainer approval for any authorization change.
