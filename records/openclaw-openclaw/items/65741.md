## Summary

Keep open: the PR still contains a useful Telegram routing bug fix, but it is not mergeable as-is because it widens Telegram group authorization, patches a removed legacy handler path, and lacks after-fix real Telegram proof.

## What This Changes

The branch maps Telegram `supergroup` chat type to `group`, adds a Telegram-only hot-reload restart pause, and lets explicit Telegram peer bindings bypass group allowlist drops in the old handler.

## Best Possible Solution

Land a focused current-main replacement: add `supergroup` normalization with regression coverage, keep Telegram group authorization in `channels.telegram.groups` plus sender allowlists unless maintainers approve and document a new binding authorization model, and add any reload 409 mitigation only with live proof and targeted tests.

## Reproduction Assessment

Yes at source level for the normalization and group-gate paths: current main still returns no `ChatType` for `supergroup`, and the active Telegram handler returns before inbound processing when group allowlists block access. No high-confidence live reproduction was established for the hot-reload 409 claim.

## Solution Assessment

No. The `supergroup` normalization is narrow, but treating bindings as group authorization crosses the documented Telegram security boundary and the group-gate patch targets a removed legacy path; the safer route is a current plugin-runtime patch plus maintainer approval for any authorization change.
