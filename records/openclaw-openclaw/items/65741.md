## Summary

Keep open: the useful `supergroup` normalization is still missing on current main, but the branch also changes Telegram authorization in a way current docs and maintainer notes reserve for explicit group/sender allowlists, and the group-gate patch targets a removed handler path. The external PR also still lacks real Telegram after-fix proof.

## What This Changes

The branch maps Telegram `supergroup` chat type to `group`, adds a Telegram-only hot-reload restart pause, and lets explicit peer bindings bypass Telegram group allowlist drops in the legacy Telegram handler.

## Best Possible Solution

Land a focused current-main replacement: add `supergroup` normalization with regression coverage, keep Telegram group authorization in `channels.telegram.groups` plus sender allowlists unless maintainers approve and document a new binding authorization model, and require live proof for any reload 409 mitigation.

## Reproduction Assessment

Yes at source level: current main still returns no `ChatType` for `supergroup`, and the active Telegram handler returns before inbound processing when group allowlists reject access. I did not establish a live Telegram reproduction for the hot-reload 409 claim.

## Solution Assessment

No. The `supergroup` normalization is narrow, but using route bindings as group authorization crosses the documented Telegram security boundary and the group-gate patch is aimed at a removed legacy handler path.
