## Summary

Keep open: the `supergroup` normalization is useful, but the PR also changes Telegram group authorization, patches a removed legacy handler path, and lacks real Telegram behavior proof.

## What This Changes

The PR normalizes Telegram `supergroup` chat types as groups, lets explicit Telegram peer bindings bypass group allowlist drops in the legacy handler, and adds a Telegram-only restart pause during hot reload.

## Best Possible Solution

Land a current-main replacement that adds `supergroup` normalization with regression coverage, keeps Telegram group authorization in explicit group/sender allowlists unless maintainers approve a documented binding model, and adds any 409 restart mitigation only with proof.

## Reproduction Assessment

Source-level yes for the normalization and group-gate paths: current main still returns `undefined` for `supergroup`, and the active handler returns before inbound processing when Telegram group policy rejects a message. I did not establish a live Telegram reproduction for the hot-reload 409 claim.

## Solution Assessment

No. The normalization change is narrow, but the binding bypass crosses the documented Telegram authorization boundary and the handler patch targets a legacy path instead of the active plugin runtime.
