## Summary

Keep this PR open for maintainer review. Current main still has a plausible `supergroup` normalization gap, but the Telegram handler part is stale against the current plugin-owned path and the binding-as-authorization change conflicts with current group access-control docs/tests; the hot-reload delay also needs to be reconciled with current Telegram polling 409 recovery.

## What This Changes

The PR maps Telegram `supergroup` chat types to `group`, lets explicit Telegram peer bindings bypass group allowlist drops in the old Telegram handler, and adds a Telegram-only 1.5s pause between channel stop/start during hot reload.

## Best Possible Solution

Land a focused current-main update: add `supergroup` normalization with route/normalization tests, keep group authorization tied to `channels.telegram.groups` and sender allowlists unless maintainers explicitly approve a documented binding authorization model, and only add a Telegram-scoped reload pause if a current gateway or Telegram regression test proves it is still needed on top of existing polling recovery.

## Reproduction Assessment

Yes for the code-level routing gaps: `normalizeChatType("supergroup")` currently returns `undefined`, and current Telegram group handling drops non-allowlisted groups before inbound routing. The hot-reload 409 claim does not have a high-confidence reproduction path in this review; current main has a related polling 409 recovery test instead.

## Solution Assessment

No. The `supergroup` mapping is narrow and likely useful, but the handler patch targets a removed path and changes the Telegram access-control boundary; a safer solution is a current plugin-path patch with explicit maintainer approval, docs, and tests for any binding-as-authorization behavior.
