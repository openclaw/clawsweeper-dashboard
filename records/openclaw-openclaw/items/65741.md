## Summary

Keep open: the PR contains a useful narrow `supergroup` normalization fix, but it is not mergeable as-is because the binding bypass broadens Telegram group authorization, the group-gate edit targets a stale pre-extension handler path, and the external PR still lacks real Telegram proof.

## What This Changes

The PR adds Telegram `supergroup` chat-type normalization, a Telegram-only restart delay, and legacy handler logic that lets peer bindings bypass group allowlist drops.

## Best Possible Solution

Land a focused replacement that adds `supergroup` normalization with regression coverage, keeps Telegram group authorization in explicit group and sender allowlists unless a new policy is approved and documented, and only adds reload 409 mitigation with live proof.

## Reproduction Assessment

Yes, at source level: current main and the latest release do not normalize `supergroup`, and current main blocks unlisted Telegram groups through the active `extensions/telegram` runtime. I did not establish a live Telegram reproduction for the hot-reload 409 claim.

## Solution Assessment

No. The `supergroup` alias is a narrow maintainable fix, but binding-based group authorization conflicts with the documented security boundary and the group-gate edit is pointed at a removed legacy handler path.
