## Summary

Keep open, but this PR should not merge as-is: the `supergroup` normalization is a valid narrow fix, while the branch is stale against the active Telegram plugin path, broadens Telegram group authorization, and still lacks real Telegram behavior proof.

## What This Changes

The branch adds Telegram `supergroup` chat-type normalization, a Telegram-only hot-reload restart delay, and legacy Telegram handler logic that lets explicit peer bindings bypass group allowlist drops.

## Best Possible Solution

Land a focused replacement: add `supergroup` normalization with regression coverage, keep Telegram group authorization in explicit group/sender allowlists unless maintainers approve and document a new binding authorization model, and require live proof for any reload 409 mitigation.

## Reproduction Assessment

Yes at source level: current main still does not normalize `supergroup`, and the active Telegram runtime checks group allowlists before inbound processing. I did not establish a live Telegram reproduction for the hot-reload 409 claim.

## Solution Assessment

No. The `supergroup` normalization is narrow and maintainable, but using route bindings as group authorization crosses the documented Telegram security boundary and the group-gate patch is aimed at a removed legacy handler path.
