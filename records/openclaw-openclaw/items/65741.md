## Summary

Keep open. Current main still lacks the useful `supergroup` routing alias, but this PR is not merge-ready because it changes the Telegram group authorization boundary, edits a removed legacy runtime path, is currently conflicting, and still lacks after-fix real Telegram proof.

## What This Changes

The PR adds `supergroup` chat-type normalization, a Telegram-only restart delay during gateway reloads, and legacy Telegram handler logic that lets explicit peer bindings bypass group allowlist drops.

## Best Possible Solution

Land a focused replacement that adds `supergroup` normalization with regression coverage, preserves explicit Telegram group and sender allowlists unless a policy change is approved and documented, and evaluates any reload 409 mitigation against the active polling/reload paths with live proof.

## Reproduction Assessment

Yes, at source level: current main and the latest release do not normalize `supergroup`, and current main blocks unlisted Telegram groups through the active `extensions/telegram` runtime. I did not establish a live Telegram reproduction for the hot-reload 409 claim.

## Solution Assessment

No. The shared `supergroup` alias is a narrow maintainable fix, but this PR bundles it with a Telegram authorization policy change, a stale legacy handler edit, and an unproven reload delay.
