## Summary

Keep this PR open for maintainer review. The branch contains one likely useful compatibility fix, but the group-routing change targets a removed Telegram handler path and also changes the documented Telegram group access boundary by letting bindings authorize otherwise non-allowlisted groups.

## What This Changes

The PR maps `supergroup` to `group` in chat type normalization, adds a Telegram-scoped hot-reload restart pause, and lets explicit Telegram peer bindings bypass group allowlist drops in the legacy Telegram handler.

## Best Possible Solution

Land a focused current-main replacement: add `supergroup` normalization with routing tests, keep Telegram group authorization tied to `channels.telegram.groups` and sender allowlists unless maintainers explicitly approve a new binding authorization model, and add any reload pause only with a targeted current-main 409 regression test.

## Reproduction Assessment

Yes for the code-level gaps: current main still returns `undefined` for `normalizeChatType("supergroup")`, and non-allowlisted Telegram groups are blocked before inbound processing. The hot-reload 409 claim is unclear without a live or test reproduction; current main has related polling 409 recovery tests instead.

## Solution Assessment

No. The `supergroup` normalization is narrow and useful, but the handler patch is stale and the binding bypass crosses an access-control boundary; a safer solution is a current plugin-path patch with tests and explicit maintainer approval for any authorization semantics change.
