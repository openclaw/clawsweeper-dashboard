## Summary

Keep this open. Current main still lacks the Copilot-owned `compat.supportsEagerToolInputStreaming: false` metadata on the GitHub Copilot Claude model paths, and open PR #75393 is a closing implementation candidate rather than merged proof.

## What This Changes

The issue reports that GitHub Copilot Claude tool requests still send `tools[].eager_input_streaming` on v2026.4.29, causing Copilot's Anthropic-compatible proxy to reject the request schema.

## Best Possible Solution

Land a provider-local GitHub Copilot fix that ensures every Copilot model resolved with `anthropic-messages` carries `compat.supportsEagerToolInputStreaming: false`, with focused coverage for helper-built defaults, forward-compatible synthetic Claude IDs, and any manifest/static catalog rows that feed runtime model resolution.

## Reproduction Assessment

Yes. The issue gives a concrete v2026.4.29 CLI command and gateway logs, and current source plus the pi-ai contract confirm the same model metadata gap still routes Copilot Claude tool requests into the eager-streaming default.

## Solution Assessment

Yes. A narrow Copilot provider metadata fix is the best direction because the rejection is specific to Copilot's Anthropic-compatible proxy; changing core Anthropic behavior or the pi-ai default would be broader than necessary.
