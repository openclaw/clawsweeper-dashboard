## Summary

Keep this open. Current main still routes Copilot Claude model IDs through the Anthropic Messages path without setting `compat.supportsEagerToolInputStreaming: false`, and the pi-ai contract defaults that flag to true, matching the reporter's `tools.0.custom.eager_input_streaming` rejection. Open PR #75393 is an active closing implementation candidate, but it is not merged proof and should be reviewed against all Copilot model-definition surfaces before this issue closes.

## What This Changes

The issue requests a GitHub Copilot provider fix so Anthropic-compatible Copilot models omit pi-ai's per-tool `eager_input_streaming` field that Copilot rejects.

## Best Possible Solution

Land one provider-owned Copilot fix that guarantees every Copilot model using `anthropic-messages` receives `compat.supportsEagerToolInputStreaming: false`, with focused tests for default/static Claude rows and synthetic forward-compatible Claude rows; avoid changing core Anthropic behavior for other providers.

## Reproduction Assessment

Yes. The issue includes a concrete v2026.4.29 CLI command and gateway logs, and current source plus pi-ai docs/source explain the exact rejected field path without needing a live paid/provider request in this read-only review.

## Solution Assessment

Yes, the provider-local compat fix is the narrowest maintainable direction. A runtime normalization hook or complete metadata coverage across manifest/default/dynamic Copilot Claude model definitions is safer than changing the generic Anthropic transport.
