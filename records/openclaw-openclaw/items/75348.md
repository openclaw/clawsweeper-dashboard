## Summary

Keep this open. Current main still lacks the Copilot-owned `compat.supportsEagerToolInputStreaming: false` metadata on the inspected Copilot Claude model paths, while the exact bundled pi-ai dependency still defaults Anthropic-compatible tool payloads to include `eager_input_streaming`; open PR #75393 is an active closing implementation candidate, not merged proof.

## What This Changes

The issue reports that GitHub Copilot Anthropic-compatible model requests still send `eager_input_streaming` through pi-ai on v2026.4.29 and asks for Copilot model compat metadata to disable it.

## Best Possible Solution

Land a provider-local GitHub Copilot fix that ensures every Copilot model resolved with `anthropic-messages` carries `compat.supportsEagerToolInputStreaming: false`, with focused coverage for default/helper-built Claude IDs, forward-compatible synthetic Claude IDs, and any catalog-derived static rows that feed runtime model resolution.

## Reproduction Assessment

Yes. The issue includes a concrete v2026.4.29 CLI command and gateway logs, and source plus exact pi-ai tarball inspection confirms current main can still route Copilot Claude models into pi-ai's default eager tool streaming path without a disabling compat flag.

## Solution Assessment

Yes. A narrow Copilot provider metadata fix is the best direction because the rejection is specific to Copilot's Anthropic-compatible proxy; changing core Anthropic behavior or the pi-ai default would be broader than necessary.
