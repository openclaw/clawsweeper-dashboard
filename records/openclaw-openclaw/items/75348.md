## Summary

Keep open: current main still lacks the Copilot-owned `supportsEagerToolInputStreaming: false` metadata needed to stop pi-ai from adding `tools[].eager_input_streaming`, and #75393 is still an open, unmerged closing PR rather than implementation evidence.

## What This Changes

Disable pi-ai eager tool input streaming for Copilot Anthropic-compatible models so Copilot Claude tool calls are not rejected by the Copilot proxy.

## Best Possible Solution

Apply a Copilot-owned compat override for every Anthropic-compatible Copilot model source and preserve `supportsEagerToolInputStreaming` through catalog/config normalization, with focused regression coverage and a changelog entry.

## Reproduction Assessment

Yes. The issue gives a concrete v2026.4.29 command and gateway log, and current main source plus the pinned pi-ai 0.71.1 contract show the same metadata gap remains; I did not run a live Copilot request in this read-only review.

## Solution Assessment

Yes, if applied comprehensively. A narrow Copilot provider metadata fix is the maintainable solution; changing core transports or the dependency default would affect other Anthropic-compatible providers that may legitimately support eager tool input streaming.
