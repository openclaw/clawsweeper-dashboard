## Summary

Keep this open: current `main` still leaves GitHub Copilot Claude model definitions without `compat.supportsEagerToolInputStreaming: false`, so pi-ai can still emit the rejected `tools[].eager_input_streaming` field. The linked #75393 remains open and unmerged, so it is an implementation candidate rather than closure evidence.

## What This Changes

The issue requests a Copilot provider-local compat flag so Anthropic-compatible GitHub Copilot models do not send pi-ai eager tool input streaming metadata.

## Best Possible Solution

Land #75393 or a replacement that applies the Copilot-owned Anthropic compat flag across dynamic defaults, dynamic fallback resolution, and any static catalog rows that feed runtime selection, with focused tests for each path.

## Reproduction Assessment

Yes. The issue provides a concrete v2026.4.29 command and gateway log, and current `main` plus the pinned pi-ai 0.71.1 source show the same metadata path still defaults to sending `eager_input_streaming`.

## Solution Assessment

Yes. A narrow GitHub Copilot provider metadata fix is the maintainable solution; changing core transport behavior or pi-ai defaults would affect providers that may legitimately support eager tool input streaming.
