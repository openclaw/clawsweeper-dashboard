## Summary

Keep this open. Current main still lacks the Copilot-owned `compat.supportsEagerToolInputStreaming: false` metadata on the Copilot Anthropic model paths, and the open closing PR #75393 is an implementation candidate rather than proof that this issue is fixed.

## What This Changes

The issue requests disabling pi-ai eager tool input streaming for GitHub Copilot Anthropic model definitions so Copilot Claude requests stop sending the rejected `tools[].eager_input_streaming` field.

## Best Possible Solution

Land a Copilot-provider fix that normalizes every Copilot model using `anthropic-messages` to `compat.supportsEagerToolInputStreaming: false`, with tests covering the manifest/static catalog, default builder, and forward-compatible synthetic model path.

## Reproduction Assessment

Yes. The report gives a concrete v2026.4.29 CLI command and gateway logs, and source inspection confirms the relevant current-main Copilot model metadata still lacks the pi-ai compat flag that suppresses the rejected field.

## Solution Assessment

Yes for the requested direction, but the implementation should cover all Copilot Anthropic model sources. A provider-local compat override is narrower and safer than changing pi-ai defaults or OpenClaw's generic Anthropic transport.
