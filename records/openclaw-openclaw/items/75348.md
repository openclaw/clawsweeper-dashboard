## Summary

Keep this issue open. Current main still lacks the Copilot-owned `compat.supportsEagerToolInputStreaming: false` metadata on the inspected Copilot Claude model paths, the published pi-ai dependency defaults that flag to true for Anthropic-compatible requests, and open PR #75393 is an active closing implementation candidate rather than merged proof.

## What This Changes

The issue requests a GitHub Copilot provider fix so Copilot Claude models do not send pi-ai's `tools[].eager_input_streaming` field to a proxy that rejects it.

## Best Possible Solution

Land a provider-local GitHub Copilot fix that ensures every Copilot model using `anthropic-messages` carries `compat.supportsEagerToolInputStreaming: false`, with focused coverage for registered/default Claude IDs and synthetic forward-compatible Claude IDs, without changing core Anthropic behavior for other providers.
