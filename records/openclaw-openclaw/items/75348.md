## Summary

Keep this open. Current main still lacks the Copilot-owned `compat.supportsEagerToolInputStreaming: false` metadata on the Copilot Claude model paths, while the pinned pi-ai dependency still defaults Anthropic-compatible tool payloads to `eager_input_streaming: true`; the linked PR is open and unmerged, so it is not implementation proof yet.

## What This Changes

The issue reports that GitHub Copilot Claude models still reject `tools[].eager_input_streaming` on v2026.4.29 and requests a Copilot model metadata fix.

## Best Possible Solution

Land a provider-local GitHub Copilot fix so every Copilot model resolved with `anthropic-messages` carries `compat.supportsEagerToolInputStreaming: false`, with focused coverage for default helper-built models, forward-compatible synthetic Claude IDs, and any manifest-backed catalog rows that reach runtime model resolution.

## Reproduction Assessment

Yes. The issue gives a concrete v2026.4.29 CLI command and gateway logs, and current source plus the pi-ai 0.71.1 contract confirm the same metadata gap still routes Copilot Claude tools into the eager-streaming default.

## Solution Assessment

Yes. A narrow Copilot provider metadata fix is the best direction because the rejection is specific to Copilot's Anthropic-compatible proxy; changing core Anthropic transport behavior or pi-ai defaults would be broader than necessary.
