## Summary

Keep open: current main still lacks the Copilot-owned compat metadata needed to stop pi-ai from adding `tools[].eager_input_streaming`, and #75393 is still an open unmerged closing candidate rather than implementation evidence.

## What This Changes

The issue reports that GitHub Copilot Claude models still fail because pi-ai adds `tools[].eager_input_streaming` unless Copilot model definitions set `compat.supportsEagerToolInputStreaming: false`.

## Best Possible Solution

Land #75393 or a replacement that applies the Copilot provider-local compat flag across every Anthropic-compatible Copilot model source, preserves it through catalog/config normalization where needed, and adds focused regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. The issue gives a concrete v2026.4.29 command and gateway log, and current main source plus the pinned pi-ai 0.71.1 contract show the same metadata gap remains; I did not run a live Copilot request in this read-only review.

## Solution Assessment

Yes. A narrow Copilot provider metadata fix is the maintainable solution; changing core transports or the dependency default would affect other Anthropic-compatible providers that may legitimately support eager tool input streaming.
