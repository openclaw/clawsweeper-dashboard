## Summary

Keep this open. Current main still lacks the Copilot-owned `compat.supportsEagerToolInputStreaming: false` metadata on the Copilot Claude model paths, while the pinned `@mariozechner/pi-ai` dependency defaults Anthropic-compatible tool requests to include `eager_input_streaming`; PR #75393 is an open closing implementation candidate, not merged proof.

## What This Changes

Report that GitHub Copilot Claude models still reject `tools[].eager_input_streaming` on v2026.4.29 and need Copilot Anthropic model definitions to disable eager tool input streaming.

## Best Possible Solution

Land a provider-local GitHub Copilot fix that ensures every Copilot model resolved with `anthropic-messages` carries `compat.supportsEagerToolInputStreaming: false`, with focused coverage for helper-built defaults, forward-compatible synthetic Claude IDs, and any manifest/static catalog rows that feed runtime model resolution.

## Reproduction Assessment

Yes. The issue gives a concrete v2026.4.29 CLI command and gateway logs, and current source plus the exact pi-ai 0.71.1 contract confirm the same model metadata gap still routes Copilot Claude tools into the eager-streaming default.

## Solution Assessment

Yes. A narrow Copilot provider metadata fix is the best direction because the rejection is specific to Copilot's Anthropic-compatible proxy; changing core Anthropic behavior or the pi-ai default would be broader than necessary.
