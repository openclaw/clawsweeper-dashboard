## Summary

Keep this open. Current main still lacks the Copilot-owned `compat.supportsEagerToolInputStreaming: false` metadata on the Anthropic-compatible Copilot model paths, and PR #75393 is an open closing implementation candidate rather than merged evidence.

## What This Changes

The issue reports that GitHub Copilot Claude requests still include `eager_input_streaming` through the pi-ai Anthropic-compatible path and are rejected on v2026.4.29.

## Best Possible Solution

Land a provider-local Copilot fix that sets `compat.supportsEagerToolInputStreaming: false` for every Copilot model resolved with `anthropic-messages`, including helper-built defaults, synthetic Claude IDs, and static catalog rows if they participate in runtime resolution.

## Reproduction Assessment

Yes. The issue gives a concrete v2026.4.29 CLI command and gateway logs, and current source plus the pi-ai 0.71.1 contract confirm the same metadata gap still exists on current main.

## Solution Assessment

Yes. A narrow Copilot provider metadata fix is the maintainable path because the rejection is specific to Copilot's Anthropic-compatible proxy; changing core Anthropic transport behavior or pi-ai defaults would be broader than necessary.
