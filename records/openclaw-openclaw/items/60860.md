## Summary

Keep this PR open for revision: the dedicated `google-vertex` direction is useful and addresses real related reports, but the current branch still has blocking ADC token-routing and endpoint-normalization defects. Current `main` is not equivalent either, so this is normal PR review rather than an obsolete cleanup close.

## What This Changes

The PR adds a dedicated Google Vertex AI provider in the Google plugin with ADC token minting, Vertex endpoint routing, region/project helpers, a Gemini catalog, registration changes, and tests.

## Best Possible Solution

Land one Google-plugin-owned Vertex path that preserves guarded fetch, scopes ADC token caching to credential identity, normalizes global/regional/template endpoints, supports authorized_user and service_account ADC, and includes focused tests plus changelog coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a guarded/injected fetch, rotate ADC credentials while the cache is warm, or use a `{location}` Vertex base URL and the PR bypasses the intended fetch path, reuses the old token, or emits a literal templated host.

## Solution Assessment

No, not as-is. A dedicated Google Vertex provider is a maintainable direction, but this branch needs the guarded-fetch, credential-scoped cache, templated-host normalization, and changelog fixes before merge.

- **[P2] Pass guarded fetch into ADC token minting:** `extensions/google/transport-stream.ts:281`
  - body: `resolveGoogleRequestApiKey()` mints ADC tokens before building the guarded model fetch, and `resolveGoogleVertexAdcToken()` uses global `fetch` internally.   - confidence: 0.82
