## Summary

Keep open. The PR still carries useful Google Vertex work that is not fully on current main, but latest PR head has blocking credential-exchange, token-cache, and templated-endpoint defects, plus missing real behavior proof before merge.

## What This Changes

The PR adds a dedicated Google Vertex AI provider path in the Google plugin with ADC auth, Vertex endpoint routing, provider catalog/registration changes, and focused tests.

## Best Possible Solution

Land or supersede one Google-plugin-owned Vertex implementation that preserves guarded token exchange, scopes ADC token caching to credential identity, normalizes regional/global/template endpoints, covers both authorized_user and service_account ADC, and includes focused tests plus redacted live-run proof.

## Reproduction Assessment

Yes, source inspection gives high-confidence reproduction paths: use an injected or guarded fetch, rotate ADC credentials while the token cache is warm, or configure a templated Vertex base URL and PR head bypasses the guard, reuses the prior token, or emits a literal templated host.

## Solution Assessment

No, not as-is. A dedicated Google Vertex provider is a maintainable direction, but this branch needs guarded-fetch token minting, credential-scoped caching, endpoint-template normalization, and real behavior proof before merge; maintainers may also choose to consolidate with the open google-genai direction.
