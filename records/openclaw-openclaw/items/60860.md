## Summary

Keep open. The PR is still useful because current main only partially supports Google Vertex ADC, but the latest head still has merge-blocking source-level issues in ADC refresh routing, credential cache isolation, templated Vertex endpoint handling, and changelog coverage.

## What This Changes

The PR adds a dedicated Google Vertex provider path in the Google plugin, including ADC-backed provider/catalog resolution, Vertex request URL construction, service-account and authorized-user token exchange, manifest auth metadata, and focused Google transport tests.

## Best Possible Solution

Land or replace this with a maintainer-approved Google plugin implementation that gives `google-vertex` exact provider ownership, keeps selection-time and runtime ADC semantics aligned, supports authorized_user and service_account credentials, preserves injected fetch/proxy behavior, keys token caching by credential identity, handles global/regional/template Vertex endpoints, and includes focused auth/routing tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path: call the PR transport with an injected `options.fetch`, switch ADC credentials while an unexpired cached token exists, or use a `{location}-aiplatform.googleapis.com` base URL and observe the PR code route through the wrong path.

## Solution Assessment

No, not as-is. A dedicated Google Vertex provider is a maintainable direction for the linked bugs, but this patch should first fix ADC fetch injection, credential-keyed caching, templated endpoint substitution, CI failures, and changelog coverage.

## Security Review

Status: needs_attention

Summary: Needs attention because the PR adds Google ADC token minting with concrete credential-isolation and network-path concerns.

Concerns:

- **[medium] Unkeyed ADC token cache can reuse prior credentials:** `extensions/google/vertex-adc.ts:151`
  - body: The new cache returns an unexpired token before checking the active ADC file path, refresh token, service-account identity, or env. Credential rotation can keep using the previous account's bearer token until expiry.
