## Summary

Keep this PR open. Current main only has partial Google Vertex support, while the PR head still introduces blocking auth, compatibility, and credential-isolation regressions despite the overall direction being useful.

## What This Changes

The PR adds a dedicated Google Vertex provider in the Google plugin with ADC-backed auth, Vertex endpoint routing, a Gemini 3.x catalog, provider registration, and focused transport/catalog/ADC tests.

## Best Possible Solution

Land or replace this with a maintainer-approved Google plugin implementation that gives google-vertex exact provider ownership without breaking existing api: "google-vertex" configs, aligns selection-time and runtime ADC semantics, supports authorized_user and service_account credentials, preserves injected fetch/proxy behavior, keys token caching by credential identity, handles global/regional/template endpoints, and includes focused auth/routing tests plus a changelog entry.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path: use an existing google-vertex config with api: "google-vertex", call the PR transport with an injected options.fetch, switch ADC credentials while an unexpired token is cached, or use a {location}-templated Vertex base URL.

## Solution Assessment

No, not as-is. A dedicated Google Vertex provider is the right maintainable direction for the linked reports, but this patch needs the exact-provider compatibility, ADC fetch injection, credential-keyed caching, templated endpoint handling, CI, and changelog issues fixed first.

## Security Review

Status: needs_attention

Summary: Needs attention because the PR adds Google ADC token minting with concrete credential-isolation and network-path concerns.

Concerns:

- **[medium] Unkeyed ADC token cache can reuse prior credentials:** `extensions/google/vertex-adc.ts:151`
  - body: The new cache returns an unexpired token before checking the active ADC file path, refresh token, service-account identity, or env. Credential rotation can keep using the previous account's bearer token until expiry.
  - confidence: 0.86
- **[medium] ADC token refresh bypasses the injected fetch path:** `extensions/google/transport-stream.ts:281`
  - body: The transport does not pass the caller's injected fetch into ADC token refresh, so credential minting can bypass configured proxy, test, or restricted network paths even when the model request is guarded.
  - confidence: 0.84
