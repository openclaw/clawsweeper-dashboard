## Summary

Keep open. Current main has partial Google Vertex support, but it is not equivalent to this PR’s dedicated provider/catalog/service-account direction, and the latest PR head still has concrete auth-sensitive correctness and security blockers before merge.

## What This Changes

The PR adds a dedicated `google-vertex` provider in the Google plugin with ADC-backed auth/catalog setup, Vertex endpoint routing, authorized_user and service-account token exchange, transport tests, and provider manifest metadata.

## Best Possible Solution

Land or replace this with a maintainer-approved Google plugin implementation that gives `google-vertex` exact provider ownership, keeps selection-time and runtime ADC semantics aligned, supports authorized_user and service_account credentials, preserves injected fetch/proxy behavior, keys token caching by credential identity, handles global/regional/template Vertex endpoints, and includes focused auth/routing tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. The review has a high-confidence source-level reproduction path from the PR discussion and diff: the PR head calls ADC token resolution without the stream fetch option, returns cached ADC tokens before checking the active credential source, and preserves literal `{location}` hosts through URL composition.

## Solution Assessment

No, not as-is. A dedicated Google Vertex provider is a maintainable direction for #58775 and #60736, but this patch should first fix ADC fetch injection, credential-keyed caching, templated endpoint substitution, and changelog coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass the injected fetch into Vertex ADC refresh:** `extensions/google/transport-stream.ts:281`
  - body: `resolveGoogleRequestApiKey()` calls `resolveGoogleVertexAdcToken()` before building the guarded fetch and without passing `options.fetch`. Proxy-aware or test callers can route the model request correctly while ADC refresh bypasses that configured network path.
  - confidence: 0.88
- **[P2] Key the ADC token cache by credential source:** `extensions/google/vertex-adc.ts:151-152`
  - body: The resolver returns any unexpired `cachedToken` before reading the active ADC file path, env, refresh token, or service-account identity. A long-running gateway that switches credentials can continue sending Vertex requests with the previous account token until expiry.
  - confidence: 0.86
- **[P2] Replace templated Vertex hosts before composing URLs:** `extensions/google/transport-stream.ts:223-224`
  - body: `new URL()` accepts `https://{location}-aiplatform.googleapis.com`, so template-style Vertex base URLs are preserved as a literal `{location}` host. Substitute the resolved region before returning the origin, matching current main behavior for templated endpoints.
  - confidence: 0.8
- **[P3] Add the required changelog entry:** `extensions/google/index.ts:281`
  - body: This registers a user-facing `google-vertex` provider feature, but the PR does not touch `CHANGELOG.md`. Repo policy requires a user-facing `feat` change to add an active-version changelog entry with appropriate contributor credit.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: Needs attention because the PR adds Google ADC token minting with concrete credential-isolation and network-path concerns.

Concerns:

- **[medium] Unkeyed ADC token cache can reuse prior credentials:** `extensions/google/vertex-adc.ts:151`
  - body: The new cache returns an unexpired token before checking the active ADC file path, refresh token, service-account identity, or env. Credential rotation can keep using the previous account's bearer token until expiry.
  - confidence: 0.86
- **[medium] ADC token refresh bypasses the injected fetch path:** `extensions/google/transport-stream.ts:281`
  - body: The transport does not pass the caller's injected fetch into ADC token refresh, so credential minting can bypass configured proxy, test, or restricted network paths even when the model request is guarded.
  - confidence: 0.84
