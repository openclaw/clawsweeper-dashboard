## Summary

Keep open. Current main has partial Google Vertex support for authorized_user ADC and Vertex URL routing, but it is not equivalent to this PR's dedicated google-vertex provider and service-account ADC work; the latest PR head still has merge-blocking auth/transport defects around injected fetch handling and credential-scoped token caching.

## What This Changes

The PR adds a dedicated google-vertex provider in the bundled Google plugin, Vertex region/catalog helpers, authorized_user and service_account ADC token exchange, Vertex request routing, manifest auth metadata, and focused Google transport tests.

## Best Possible Solution

Land or replace this with a maintainer-approved Google plugin implementation that gives google-vertex exact provider ownership, keeps selection-time and runtime ADC semantics aligned, supports authorized_user and service_account credentials, preserves injected fetch/proxy behavior, keys token caching by credential identity, and includes focused routing/auth tests plus user-facing setup docs.

## Reproduction Assessment

Yes for the PR findings. The latest branch source shows resolveGoogleVertexAdcToken() is called without the stream's injected fetch option, and the token resolver returns cachedToken before reading the current ADC/env state, so focused unit tests can reproduce both paths without live Google access.

## Solution Assessment

No, not as-is. The dedicated-provider direction is likely the maintainable boundary, but this patch should preserve fetch injection and credential-keyed token caching before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass the transport fetch into ADC token refresh:** `extensions/google/transport-stream.ts:281`
  - body: resolveGoogleVertexAdcToken() now falls back to global fetch because the stream never passes options.fetch. Callers that supply a proxy-aware or test fetch will send the model request through the guarded path, but token refresh will bypass that network path.
  - confidence: 0.88
- **[P2] Key the ADC token cache by credential source:** `extensions/google/vertex-adc.ts:151-152`
  - body: The cache returns any unexpired token before reading the current ADC path, credential contents, or env. In a long-running gateway, switching GOOGLE_APPLICATION_CREDENTIALS or service accounts can continue sending Vertex requests with the previous account's token until expiry.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Security review needs attention because the PR adds ADC token minting and currently has concrete credential isolation and network-path issues.

Concerns:

- **[medium] Unkeyed ADC token cache can reuse prior credentials:** `extensions/google/vertex-adc.ts:151`
  - body: The new cache returns an unexpired token without checking the current credential path, refresh token, service-account identity, or project env. A gateway that rotates or switches Google credentials can send Vertex requests using a previous account's token.
  - confidence: 0.84
- **[medium] ADC token refresh bypasses the injected fetch path:** `extensions/google/transport-stream.ts:281`
  - body: The stream does not pass options.fetch into the new ADC token resolver, and the resolver uses global fetch for token exchange. Proxy-aware callers or restricted network transports can have model requests guarded while credential minting bypasses that configured path.
  - confidence: 0.82
