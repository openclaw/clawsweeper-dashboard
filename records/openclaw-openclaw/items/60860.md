## Summary

Keep this PR open. Current main has partial Google Vertex ADC support, but it does not implement the PR's dedicated provider and service-account ADC path; the PR is still useful but not mergeable because the latest branch has concrete auth/network correctness issues in the new ADC flow.

## What This Changes

The PR adds a dedicated `google-vertex` provider in the bundled Google plugin with ADC-backed catalog/config resolution, Vertex request URL routing, authorized_user and service-account token exchange, manifest auth metadata, and focused Google transport tests.

## Best Possible Solution

Land or replace this with a maintainer-approved Google plugin implementation that gives `google-vertex` exact provider ownership, keeps selection-time and runtime ADC semantics aligned, supports both authorized_user and service_account credentials, preserves injected fetch/proxy behavior, keys token caching by credential identity, handles global/regional/template Vertex endpoints, and includes focused auth/routing tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. The PR findings are reproducible from source-level checks: the token resolver is called without the stream fetch option, the cache returns before reading the current ADC source, and a templated Vertex base URL would be composed with the literal `{location}` host.

## Solution Assessment

No, not as-is. A dedicated Google Vertex provider is a maintainable direction, but this patch should first fix the ADC fetch injection, credential-keyed caching, templated endpoint handling, and required changelog coverage.

## Security Review

Status: needs_attention

Summary: Needs attention because the PR adds Google ADC token minting and currently has concrete credential isolation and network-path issues.

Concerns:

- **[medium] Unkeyed ADC token cache can reuse prior credentials:** `extensions/google/vertex-adc.ts:151`
  - body: The new cache returns an unexpired token before checking the active ADC file path, refresh token, service-account email, or env. Credential rotation can keep using the previous account's bearer token until expiry.
  - confidence: 0.84
- **[medium] ADC token refresh bypasses the injected fetch path:** `extensions/google/transport-stream.ts:281`
  - body: The transport does not pass the caller's injected fetch into ADC token refresh, so credential minting can bypass configured proxy/test/restricted network paths even when the model request is guarded.
  - confidence: 0.82
