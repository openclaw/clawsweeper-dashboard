## Summary

Keep open: current main still has a source-level path where Discord channel/thread turns resolve to message-tool-only delivery before the Discord progress draft stream is created, so the documented progress draft can still be absent. The related open PR at https://github.com/openclaw/openclaw/pull/80042 addresses adjacent verbose/tool-progress delivery, but its diff does not remove the draft-stream creation gate highlighted here.

## What This Changes

Restore Discord `streaming.mode: "progress"` so channel/thread turns create and update the visible status draft during tool-heavy work instead of staying silent until the final response.

## Best Possible Solution

Keep final-reply privacy for `message_tool_only` turns, but let Discord progress mode create a channel-owned progress draft and receive useful work callbacks for those turns, with focused regression coverage.

## Reproduction Assessment

Yes, from source inspection: Discord channel turns can resolve to `message_tool_only`, that flag is passed into the draft-preview controller, and the controller refuses to create a draft stream. I did not run a live Discord session.

## Solution Assessment

Yes, the requested direction is the narrow maintainable boundary: restore the progress draft and progress callbacks without making normal final replies public. The active related PR is useful but appears adjacent rather than sufficient for the draft-creation gate.
