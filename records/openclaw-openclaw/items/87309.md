## Summary

No blocking findings. The PR targets the real source-level gap behind https://github.com/openclaw/openclaw/issues/62809: current main rotates isolated cron run session identity and currently uses that rotating session id as OpenAI prompt-cache affinity, while the proposed patch adds a separate stable hashed affinity key and preserves transcript/session isolation.

## What This Changes

The PR adds an internal `promptCacheKey` for embedded isolated cron runs, threads it through embedded runner stream resolution, and makes OpenAI Responses and opted-in Completions payload builders prefer it over `sessionId`.

## Best Possible Solution

Land the focused internal prompt-cache-affinity fix once required CI passes, keeping isolated cron transcript sessions fresh while using a stable hashed cache key for OpenClaw-managed OpenAI transports.

## Reproduction Assessment

Yes, from source inspection: current main forces fresh isolated cron sessions and the OpenAI transport derives `prompt_cache_key` from that rotating `sessionId`. I did not run a live OpenAI cache telemetry reproduction.

## Solution Assessment

Yes, the proposed internal key is the narrowest maintainable fix because it separates provider cache affinity from transcript/session identity without adding user config or changing session-bound and CLI cron behavior.
