## Summary

Keep open: current main still appears to route the Claude stream-json final `result` usage into CLI `lastCallUsage`, so the reporter's summed `cache_read_input_tokens` can still become a fresh session context snapshot and drive the CLI compaction gate.

## What This Changes

Fix Claude CLI stream-json usage accounting so summed `result` cache-read values from tool sub-calls do not poison session totals or failed compaction.

## Best Possible Solution

Parse Claude stream-json so OpenClaw's context snapshot uses the latest real model-call usage, then keep aggregate result usage out of compaction-triggering session totals.

## Reproduction Assessment

Yes at source level: a Claude stream-json fixture with multiple assistant usage records and a final result whose cacheRead is their sum would flow through `parseCliJsonl` into `lastCallUsage` and session `totalTokens`. I did not run a live Claude CLI account in this read-only review.

## Solution Assessment

Yes, a narrow parser/accounting repair is the right direction: use a trusted latest model-call usage snapshot for context, and keep aggregate provider telemetry from driving preemptive compaction.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/85573. Current main parses the final Claude stream-json `result.usage.cache_read_input_tokens` as `output.usage`, then copies it to `agentMeta.lastCallUsage` and stores it as fresh CLI `SessionEntry.totalTokens`; reporter evidence shows that `result` cache-read is an aggregate across tool sub-calls, not the final API call. Implement a narrow repair in the CLI JSONL parser/session-accounting path so Claude stream-json context snapshots use the latest real model-call usage, not aggregate result usage. Prefer tracking the latest deduped assistant/message usage in `src/agents/cli-output.ts` for `claude-stream-json` and returning that as the context snapshot while preserving aggregate result usage only for cost/telemetry if needed. Add regression coverage with multiple assistant usage records and a final result whose cacheRead is their sum; assert persisted CLI `totalTokens` uses the last assistant usage, not the result sum. Also consider a defensive stale/impossible-snapshot guard only if it stays narrow. Do not disable all CLI compaction, change provider defaults, or weaken the existing guard against treating cumulative CLI usage as context. Add a changelog entry because this fixes a user-visible stall.
