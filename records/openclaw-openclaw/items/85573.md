## Summary

Keep open: current main still routes the claude-cli stream-json final `result.usage` into `lastCallUsage`, while the parser does not read the assistant-event usage snapshots that the report shows are needed to avoid inflated cache-read context totals.

## What This Changes

The issue requests a fix for claude-cli stream-json cache-read accounting so tool-heavy turns do not inflate session token snapshots and trigger ineffective compaction stalls.

## Best Possible Solution

Land a focused claude-cli stream-json accounting fix that preserves the latest deduped assistant-call usage for context snapshots while keeping aggregate result usage available only for aggregate run or cost accounting.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: a fixture with assistant `message.usage` cache snapshots followed by a final `result.usage` sum will currently expose only the final result usage as `lastCallUsage`. I did not run live claude-cli in this read-only pass.

## Solution Assessment

Yes for the accounting layer: the maintainable fix is to carry the latest deduped assistant-call usage into `lastCallUsage` for context snapshots instead of treating the final result usage as the last call. Broader claude-cli compaction ownership should be handled only if real context pressure remains after that fix.
