## Label Justifications

- `P1`: The report describes a deterministic provider-backed agent stall across active channels on tool-heavy claude-cli sessions, with manual reset as the recovery path.
- `impact:message-loss`: The user-visible failure mode is that Telegram, dashboard, and other active channels stop receiving agent replies.
- `impact:session-state`: The root path is session token accounting and compaction state for claude-cli-backed conversations.

## Summary

Keep open. Current main still lets claude-cli stream-json final `result.usage` become both aggregate run usage and `agentMeta.lastCallUsage`, so the fresh CLI session context snapshot can still be based on cumulative cache-read data; three open closing PRs are active fix candidates but none has merged.

## What This Changes

The issue requests fixing claude-cli stream-json usage accounting so cumulative `result.cache_read_input_tokens` does not inflate session token state and trigger failed compaction.

## Best Possible Solution

Land a focused fix that separates aggregate run usage from the latest assistant-call prompt/cache snapshot for CLI session context accounting, with regression coverage and redacted real claude-cli proof; handle claude-cli compaction ownership separately only if needed after accounting is correct.

## Reproduction Assessment

Yes, at source level: current main copies the claude-cli final result usage into `lastCallUsage`, then session-store persists that value as a fresh CLI context snapshot and the compaction gate consumes it. I did not run the live claude-cli binary in this read-only review.

## Solution Assessment

Yes for the accounting layer: the narrow maintainable fix is to preserve aggregate result usage for run/cost accounting while carrying the latest assistant-call usage into `lastCallUsage`. The approach in https://github.com/openclaw/openclaw/pull/85600 is closest conceptually, but the linked PRs still need review/proof before this issue can close.
