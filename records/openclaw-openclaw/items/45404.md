## Summary

Keep open. Current main still canonicalizes managed browser state under the `openclaw` profile and has safe trash helpers for explicit profile reset/delete, but there is still no doctor diagnostic or repair path that scans for stale unconfigured `browser/clawd/user-data` residue from the rename period.

## What This Changes

Detect legacy managed Brave/Chromium profile residue under `browser/clawd/user-data` and offer safe doctor-style cleanup while keeping `browser/openclaw/user-data` canonical.

## Best Possible Solution

Add a browser-owned doctor diagnostic that warns on confirmed unused legacy `clawd` profile residue and only moves it to a safe trash/archive location during explicit repair.

## Reproduction Assessment

Yes, from source inspection rather than a live migrated home. A home containing `browser/clawd/user-data` beside `browser/openclaw/user-data` is not reported because current browser doctor code reads configured profiles and never scans legacy sibling directories.

## Solution Assessment

Yes. A doctor/browser diagnostic with opt-in trash/archive repair is the narrow maintainable direction; startup migration or unconditional deletion would be riskier.
