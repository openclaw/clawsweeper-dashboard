## Summary

Keep open. Current main still treats `openclaw` as the canonical managed browser profile and preserves `clawd` only as a config compatibility alias, but there is still no doctor/browser cleanup path that detects or safely archives an unconfigured legacy `browser/clawd/user-data` directory.

## What This Changes

The issue asks OpenClaw to detect leftover rename-era `~/.openclaw/browser/clawd/user-data` browser profile residue, warn clearly, and offer safe archival while documenting the canonical `openclaw` managed profile path.

## Best Possible Solution

Implement a browser-owned doctor/repair check that scans for stale unconfigured `CONFIG_DIR/browser/clawd/user-data` only when the canonical managed profile is `openclaw`, emits a clear warning, and offers explicit safe archival or Trash movement. Keep the legacy `clawd` driver alias intact and document the canonical managed profile path in the browser and doctor docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to place an unconfigured `CONFIG_DIR/browser/clawd/user-data` beside the canonical `openclaw` profile; current doctor and browser doctor code only inspect configured/default profiles and will not report or archive that directory.

## Solution Assessment

Yes. A doctor/repair warning plus explicit archival flow is the narrowest maintainable fix; startup migration or silent deletion would create unnecessary data-loss risk for browser profile state.
