## Summary

Keep open. Current main uses `openclaw` as the canonical managed browser profile and still has no doctor/repair path that detects a stale `browser/clawd/user-data` sibling, so the prior source-pass comment remains accurate.

## What This Changes

The issue requests a doctor or repair cleanup that detects legacy managed Brave profile residue under `browser/clawd/user-data` after the rename and safely archives it.

## Best Possible Solution

Add a browser-owned doctor diagnostic that warns on legacy `clawd` profile residue and, in repair mode, moves only safe unconfigured residue to Trash/archive after confirming `openclaw` is the active managed profile.

## Reproduction Assessment

Yes. From source inspection, a migrated home containing `~/.openclaw/browser/clawd/user-data` would not be reported by current doctor because the browser health check never scans that sibling path.

## Solution Assessment

Yes. A doctor/browser diagnostic with opt-in archival is the narrowest maintainable solution; startup migration or unconditional deletion would be riskier.
