## Summary

Keep open. Current main uses `openclaw` as the canonical managed browser profile and still has no browser-specific doctor or migration path for leftover `clawd` profile directories, so the reported cleanup gap remains real and narrowly scoped.

## What This Changes

Request a low-priority doctor or repair cleanup that detects legacy `~/.openclaw/browser/clawd/user-data` residue, confirms the active managed profile is `openclaw`, and offers safe archival instead of deletion.

## Best Possible Solution

Add an idempotent browser doctor/repair check that reports stale `~/.openclaw/browser/clawd` profile data, never deletes automatically, and offers an explicit archive/trash path with docs for the canonical managed profile.

## Reproduction Assessment

Yes, at source level. If `~/.openclaw/browser/clawd/user-data` exists, current main resolves active managed browser state through the `openclaw` profile and has no scanner or doctor repair for that legacy directory.

## Solution Assessment

Yes. A doctor/repair warning with opt-in archival matches existing migration policy better than startup deletion, and it can later be surfaced through the broader cache inventory work if maintainers prefer that UI.
