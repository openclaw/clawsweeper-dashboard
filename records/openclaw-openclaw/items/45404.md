## Summary

Keep open. Current main uses `openclaw` as the canonical managed browser profile and keeps `clawd` only as a config compatibility alias, but it still has no doctor or repair path for detecting and safely archiving leftover `CONFIG_DIR/browser/clawd/user-data` residue.

## What This Changes

Request a low-priority doctor/repair cleanup that detects legacy `clawd` managed Brave/browser profile directories after the rename and offers safe archival instead of silent deletion.

## Best Possible Solution

Implement a browser-owned doctor warning and explicit `--fix`/repair archival path for unconfigured legacy `CONFIG_DIR/browser/clawd/user-data` when the active managed profile is `openclaw`, while preserving the legacy `clawd` driver alias and documenting the canonical profile path.

## Reproduction Assessment

Yes. A source-level reproduction is to place an unconfigured `CONFIG_DIR/browser/clawd/user-data` beside the canonical `openclaw` profile; current doctor/browser doctor code only inspects configured/default profiles and will not report or archive that directory.

## Solution Assessment

Yes. The requested doctor/repair warning plus explicit archival is the narrowest maintainable solution; startup migration or silent deletion would create unnecessary data-loss risk for browser profile state.
