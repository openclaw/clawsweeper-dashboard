## Summary

Keep open. Current main still has additive prompt-context and system-prompt hook support, but no public hook result field, merge rule, shared resolver output, or runner consumption path that lets a plugin replace the submitted user prompt.

## What This Changes

Request adding an opt-in prompt replacement result for `before_agent_start`/`before_prompt_build` so plugins can truncate or redact an oversized current user prompt before model submission.

## Best Possible Solution

Add and document an explicit user-prompt replacement result field, keep legacy compatibility, pass it through the shared hook merger/resolver, and apply it consistently before additive context in embedded, CLI, and Codex runner paths.

## Reproduction Assessment

Yes, at source level. A plugin cannot return a typed user-prompt replacement because current main lacks the hook result field, allowlist entry, merge output, shared resolver output, and runner application path.

## Solution Assessment

Likely yes, but it needs a maintainer API decision. An explicit prompt replacement field governed by existing prompt-mutation policy and applied before prepend/append context is the narrowest maintainable direction; existing `systemPrompt` and context additions do not solve oversized current prompts.
