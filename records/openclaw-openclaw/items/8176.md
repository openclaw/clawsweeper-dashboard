## Label Justifications

- `P2`: Prompt hook replacement is a bounded but maintainer-visible plugin API improvement with clear context-limit and redaction value.
- `impact:security`: The requested hook capability controls whether trusted plugins can redact or rewrite sensitive user prompt text before model-provider submission.

## Summary

Keep open: current main and v2026.6.6 still expose only additive prompt/context and system-prompt hook fields; no plugin-owned user prompt replacement flows through embedded, CLI, or Codex paths. The remaining work is a security-sensitive plugin API decision, not stale cleanup.

## What This Changes

The issue requests a hook result field that lets plugins replace the effective user prompt before model submission.

## Best Possible Solution

Keep this open until maintainers approve or reject a shared, documented prompt replacement contract for before_prompt_build plus legacy before_agent_start, with consistent filtering, precedence, runners, docs, and tests.

## Reproduction Assessment

Yes. Source inspection on current main and v2026.6.6 shows hook result types, merge logic, embedded, CLI, and Codex paths only support additive context and system-prompt fields, not replacing the submitted user prompt.

## Solution Assessment

No, not as a legacy-only before_agent_start patch. The maintainable solution is a shared prompt-build replacement contract with legacy before_agent_start compatibility plus explicit precedence and security filtering.
