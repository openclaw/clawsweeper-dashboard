## Summary

Keep open. Current main and the latest release still support additive prompt context and system-prompt mutation, but they do not expose or apply a plugin-owned replacement for the submitted user prompt; the remaining work is a product and security-sensitive plugin API decision rather than conservative cleanup.

## What This Changes

The issue requests a hook result field that lets plugins replace the submitted user prompt before the model call, with prepend/append context wrapping the effective prompt.

## Best Possible Solution

Keep this open until maintainers either approve and implement a documented prompt replacement contract across before_prompt_build and legacy before_agent_start, or explicitly reject that plugin API direction.

## Reproduction Assessment

Yes at source level: current main has no public replacement field, no merge rule for one, and no embedded/CLI/Codex runner path that applies a hook-owned replacement prompt. I did not run a live oversized-prompt scenario in this read-only review.

## Solution Assessment

Unclear as a product decision: adding replacement support is a plausible and useful plugin API extension, but the best fix needs maintainer-approved security semantics and cross-runner behavior. The narrow implementation should prefer modern before_prompt_build with legacy before_agent_start compatibility rather than a legacy-only field.

- A complete implementation needs sibling runner parity or explicit unsupported-surface documentation for embedded, CLI, and Codex harness paths.
