## Label Justifications

- `P2`: This is a bounded, strategically aligned plugin API enhancement with limited immediate blast radius but cross-runner contract work.
- `impact:security`: The requested hook would let in-process plugins replace user instructions before model invocation, affecting trust, audit, and prompt-integrity boundaries.

## Summary

Current main still lacks a public user-prompt replacement result and all inspected prompt paths remain additive, so the requested capability is still necessary; the remaining blocker is an explicit maintainer decision on the security-sensitive plugin contract and precedence.

## What This Changes

Request plugin prompt-build hooks to replace the outgoing user prompt before model invocation.

## Best Possible Solution

Add one documented replacement field to the modern before_prompt_build contract, inherit it through legacy before_agent_start, gate it with existing prompt-injection policy, define deterministic priority semantics, and apply it consistently before additive context in embedded, CLI, and Codex runners with focused contract and runtime tests.

## Reproduction Assessment

Yes, at source level: the exhaustive hook result type, merge function, shared resolver, CLI assembly, Codex bridge, and docs all lack a user-prompt replacement path on current main.

## Solution Assessment

No, the original minimal embedded-runner-only proposal is not the best complete fix; the maintainable solution is a modern public hook contract with security gating, deterministic precedence, and parity across every supported runner.
