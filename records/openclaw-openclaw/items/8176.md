## Summary

Keep open: current main and the latest release still expose additive prompt/context and system-prompt hook fields only, so plugins still cannot replace the submitted user prompt through `before_prompt_build` or legacy `before_agent_start`. The remaining work is a security-sensitive plugin API decision; the direct implementation PR is closed unmerged and the broader `llm_input` interception PR is open, conflicting, and a different boundary.

## What This Changes

Request an opt-in hook result that lets plugins replace the submitted user prompt before model submission so trusted plugins can truncate or redact oversized input.

## Best Possible Solution

Keep this as the canonical tracker until maintainers either approve and document an explicit user-prompt replacement API across shared hook surfaces or reject the capability; if approved, update types, mutation filtering, merge precedence, embedded/CLI/Codex runners, docs, and regression tests together.

## Reproduction Assessment

Yes. Source inspection shows current hook result types, merge logic, shared prompt helpers, embedded runner, CLI runner, and Codex helper all lack a user-prompt replacement field and only wrap the original prompt with additive context.

## Solution Assessment

Unclear as an implementation direction. A replacement field is plausible, but maintainers need to choose the public API name, ordering, audit/security semantics, and whether the broader `llm_input` interception boundary should own this instead.
