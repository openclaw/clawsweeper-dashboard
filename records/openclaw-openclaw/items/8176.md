## Label Justifications

- `P2`: The missing extension point blocks bounded plugin redaction and context-limit workflows but does not break OpenClaw’s default agent execution.
- `impact:security`: The requested capability authorizes in-process plugins to redact or replace user-submitted content immediately before model execution.

## Summary

Keep open: current main and v2026.6.5 still provide additive user-context and system-prompt mutations, but no public hook result or runner path can replace the submitted user prompt. The request fits OpenClaw’s plugin direction, while the public API precedence and security semantics require explicit maintainer approval.

## What This Changes

Add a plugin hook result field that replaces the submitted user prompt before model execution, with additive context applied around the replacement.

## Best Possible Solution

Add one typed replacement field to before_prompt_build, inherit it through legacy before_agent_start, and implement replacement-then-prepend/append ordering consistently across every runner with security-policy handling, documentation, and focused contract tests.

## Reproduction Assessment

Yes at source level: a plugin can return additive prompt mutations, but current main has no typed replacement field, merge rule, or runner branch capable of consuming a replacement. A live model call is unnecessary to establish the missing API path.

## Solution Assessment

Yes in overall direction, but not as a legacy-hook-only addition. The narrow maintainable design is a modern before_prompt_build contract inherited by before_agent_start and implemented consistently across embedded, CLI, and Codex runners.

## Risks / Open Questions

- A replacement hook can silently remove or rewrite user intent, so maintainers must define precedence when multiple plugins return replacements and how the mutation is exposed to audit or diagnostic surfaces.
- The existing prompt-injection policy and before_agent_run fail-closed boundary need an explicit rule for whether prompt replacement is permitted, filtered, or disabled together with other prompt mutations.
- A one-runner implementation would create behavior drift across embedded, CLI, and Codex harnesses.
