- `P2`: The issue is a normal-priority plugin API enhancement with clear source evidence but a bounded, non-emergency blast radius.

## Summary

Keep open: current main and the latest release still expose additive prompt/context and system-prompt hook fields only, so plugins still cannot replace the submitted user prompt through `before_prompt_build` or legacy `before_agent_start`. The remaining work is a security-sensitive plugin API decision, not a narrow bug cleanup.

## What This Changes

Request to add a prompt replacement result field for plugin prompt hooks so plugins can shrink or redact the current user prompt before the model call.

## Best Possible Solution

Keep this issue as the canonical tracker until maintainers either approve and document an explicit prompt replacement API across the shared hook surfaces or reject the capability.

## Reproduction Assessment

Yes, from source inspection: current hook result types, merge logic, and embedded/CLI runners all lack a replacement prompt field and only wrap the original prompt with additive context. I did not run a live model scenario because the gap is directly visible in the current source.

## Solution Assessment

Unclear as an implementation direction: a prompt replacement field is plausible, but maintainers need to decide the API name, precedence, audit/security semantics, and whether the broader `llm_input` interception boundary should own this instead.
