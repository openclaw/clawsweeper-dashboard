## Summary

Keep open. Current main and v2026.6.8 still support additive prompt context and system-prompt mutations, but they do not expose or apply a hook-owned replacement for the current user prompt; the direct implementation attempts are closed unmerged and the broader open LLM-hook PR is not a clean canonical fix for this hook API decision.

## What This Changes

The issue requests an opt-in plugin hook result that can replace the current user prompt before model submission, with context fields applied around the effective prompt.

## Best Possible Solution

Define a maintainer-approved user-prompt replacement contract for `before_prompt_build` and legacy `before_agent_start`, or explicitly choose the broader LLM-input hook as the supported boundary, then implement it consistently across embedded, CLI, Codex harness, docs, policy filtering, and regression tests.

## Reproduction Assessment

Yes at source level, though this is a feature gap rather than a bug reproduction. Current main and v2026.6.8 expose only additive prompt-context fields and system-prompt mutation, so a plugin cannot replace the current user prompt through these hooks.

## Solution Assessment

Unclear. A prompt-build replacement field is plausible and fits the plugin API direction, but maintainers still need to choose the API boundary, precedence, policy filtering, and runner coverage, especially alongside https://github.com/openclaw/openclaw/pull/61485.
