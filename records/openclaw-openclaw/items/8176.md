## Summary

Keep open: current main and v2026.6.10 still expose additive prompt/context and system-prompt hook fields only, so plugins cannot replace the submitted user prompt through `before_prompt_build` or legacy `before_agent_start`. The remaining work is a plugin API and security-boundary decision, not conservative cleanup.

## What This Changes

The issue requests an opt-in hook result field that lets plugins replace the effective user prompt before model submission.

## Best Possible Solution

Keep this as the canonical tracker until maintainers either approve and document a prompt replacement API across shared hook surfaces or explicitly reject the capability.

## Reproduction Assessment

Yes, source-level: current hook result types, merge logic, embedded runner, CLI runner, and docs all show additive context/system-prompt support but no user-prompt replacement path. I did not run a live provider scenario because the feature gap is directly visible in source.

## Solution Assessment

Unclear as an implementation direction until maintainers decide the API and trust boundary. A prompt replacement field is plausible, but the best implementation must settle naming, precedence, `allowPromptInjection` behavior, observability, and whether the broader LLM-input hook boundary should own this instead.
