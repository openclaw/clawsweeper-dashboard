## Summary

Keep open. Current main still supports additive prompt context and system-prompt mutations, but there is no hook result field, allowlist entry, merge rule, shared resolver output, or runner path that lets a plugin replace the submitted user prompt.

## What This Changes

The issue requests a prompt replacement hook result so plugins can truncate or redact oversized current user prompts before additive prompt context is applied.

## Best Possible Solution

Add an explicit prompt replacement field to the modern and legacy hook contracts, document precedence and ordering, and apply it consistently before prepend/append context across embedded, CLI, and Codex runner paths after maintainer API approval.

## Reproduction Assessment

Yes, at source level. Current main lacks the typed hook result field, prompt-mutation allowlist entry, merge output, shared resolver output, and runner application path needed for prompt replacement.

## Solution Assessment

Likely yes, but it needs maintainer API approval. An explicit prompt replacement field applied before additive context is the narrowest maintainable direction; existing `systemPrompt` and context additions do not shrink an oversized current user prompt.
