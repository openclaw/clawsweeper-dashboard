## Label Justifications

- `P2`: The missing capability blocks some plugin-based context mitigation but does not break OpenClaw's default agent workflow.
- `impact:security`: The requested hook can redact or replace user input before model submission, directly affecting prompt integrity and policy enforcement.

## Summary

Current main still lacks user-prompt replacement: prompt-build hooks only add context or alter system instructions, while the newer final gate can inspect or block but cannot transform and continue. The capability fits OpenClaw's plugin direction, but its public precedence, auditability, and security semantics require explicit maintainer approval.

## What This Changes

Requests a plugin hook result that can replace the submitted user prompt before model execution.

## Best Possible Solution

Add one canonical replacement field to `before_prompt_build`, define deterministic multi-plugin precedence and audit/security semantics, wrap the replacement with prepend/append context, and apply it consistently in embedded, CLI, and Codex runtimes before `before_agent_run` performs final inspection.

## Reproduction Assessment

Yes at source level: prompt-build hooks only add context or alter system-prompt text, while `before_agent_run` can inspect and block but cannot return transformed prompt text. A live Anthropic context-limit failure was not reproduced during this read-only review.

## Solution Assessment

No, the original legacy-`before_agent_start`-only proposal is not the best long-term fix. The maintainable shape is a canonical `before_prompt_build` contract with legacy inheritance, explicit precedence and security rules, and parity across all agent harnesses.

## Risks / Open Questions

- Allowing an in-process plugin to replace user input can silently change user intent or remove policy-relevant text unless the API defines trust, audit, and final-inspection behavior.
- Multiple prompt-mutating plugins need deterministic precedence; an ambiguous first-wins or last-wins rule could produce order-dependent redaction and context-limit behavior.
- A partial implementation would create harness divergence unless embedded, CLI, and Codex paths all submit the same effective prompt and preserve the final fail-closed gate.
