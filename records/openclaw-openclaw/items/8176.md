## Summary

Keep open: current main and the latest release still lack a public hook result field or runner path that replaces the submitted user prompt; the open implementation candidate needs maintainer API and security review before this can close.

## What This Changes

Requests a Plugin SDK prompt-build or legacy before_agent_start result field that lets trusted plugins replace the current user prompt before provider submission.

## Best Possible Solution

Define or explicitly reject a first-class before_prompt_build prompt replacement API with legacy before_agent_start compatibility, deterministic precedence, policy filtering, embedded/CLI/Codex parity, docs, regression tests, and redacted real plugin-run proof.

## Reproduction Assessment

Yes, at source level: current hook types and mutation allowlists contain only additive/system fields, and embedded, CLI, and Codex paths all build from the original prompt plus context additions. I did not run a live provider call because the missing public contract is clear from source inspection.

## Solution Assessment

No for a legacy-only promptOverride field. The maintainable solution is a first-class before_prompt_build replacement contract with legacy compatibility, runner parity, policy filtering, and explicit security semantics.
