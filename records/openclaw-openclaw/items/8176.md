## Summary

Keep open: current main and the latest release still support additive prompt context and system-prompt mutations, but they do not expose or apply a user-prompt replacement hook. The open implementation candidate at https://github.com/openclaw/openclaw/pull/57630 covers the same capability, but it remains open/conflicting and needs product/security follow-up rather than cleanup closure.

## What This Changes

The issue requests a Plugin SDK hook result field that can replace the user prompt before model submission.

## Best Possible Solution

Review or replace https://github.com/openclaw/openclaw/pull/57630 with a focused public prompt replacement contract that applies consistently across embedded, CLI, and Codex prompt assembly, with docs, tests, and security-reviewed precedence semantics.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a Plugin SDK capability request. Source inspection gives high confidence that current main and the latest release lack a replacement-prompt field while supporting additive prompt context.

## Solution Assessment

No. A partial legacy-only `promptOverride` would be too narrow; the maintainable solution is an explicit prompt-build API with legacy compatibility and consistent embedded, CLI, and Codex behavior.

## Security Review

Status: needs_attention

Summary: The requested API is security-sensitive because plugins may rely on it to redact or shrink user input before provider calls.

Concerns:

- **[medium] Avoid partial redaction semantics:**
  - body: If OpenClaw publishes a prompt replacement field, every prompt assembly path and prompt-injection filtering rule must honor it; otherwise a plugin could believe sensitive text was redacted while another runner still sends the original prompt.
