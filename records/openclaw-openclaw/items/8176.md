## Summary

Keep open: current main and v2026.5.20 still expose additive prompt/context fields and system-prompt mutation, but no hook result field or runner path that replaces the submitted user prompt. This remains a public Plugin SDK and security-sensitive prompt redaction contract, with an open implementation candidate under review.

## What This Changes

The issue requests a hook result field that lets `before_prompt_build` and legacy `before_agent_start` replace the submitted user prompt before provider submission.

## Best Possible Solution

Define or explicitly reject a first-class `before_prompt_build` prompt replacement API with legacy `before_agent_start` compatibility, deterministic precedence, policy filtering, embedded/CLI/Codex parity, docs, regression tests, and redacted real plugin-run proof.

## Reproduction Assessment

Yes, at source level: current hook types and mutation allowlists contain only additive/system fields, and the embedded, CLI, and Codex paths all build from the original prompt plus context additions. I did not run a live provider call because the missing public contract is clear from source inspection.

## Solution Assessment

No for a legacy-only `promptOverride` field. The maintainable solution is a first-class `before_prompt_build` replacement contract with legacy compatibility, runner parity, policy filtering, and explicit security semantics.
