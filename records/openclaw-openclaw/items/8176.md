## Summary

Keep open: current main and the latest release still lack a public hook result field and runner path that can replace the submitted user prompt, while the open implementation candidate still needs maintainer API/security review before it can resolve this issue.

## What This Changes

Request to let plugin prompt hooks replace the submitted user prompt before model invocation so plugins can truncate or redact oversized current prompts.

## Best Possible Solution

Define or explicitly reject a first-class before_prompt_build prompt replacement API with legacy before_agent_start compatibility, deterministic precedence, policy filtering, embedded/CLI/Codex parity, docs, regression tests, and redacted real plugin-run proof.

## Reproduction Assessment

Yes, at source level: current hook types and mutation allowlists contain only additive/system fields, and embedded, CLI, and Codex prompt paths all build from the original prompt plus context additions. I did not run a live provider call because the missing public contract is clear from source inspection.

## Solution Assessment

No for a legacy-only promptOverride field. The maintainable solution is a first-class before_prompt_build replacement contract with legacy compatibility, runner parity, policy filtering, and explicit security semantics.
