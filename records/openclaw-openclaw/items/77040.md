## Summary

Keep open: current main still has a source-reproducible hang path because `channels capabilities` advertises a timeout but awaits channel capability hooks directly, and the Discord diagnostic hook has an unbounded permission-fetch path.

## What This Changes

The issue asks for `openclaw channels capabilities --channel discord` to return or fail within a bounded timeout instead of hanging and leaving CPU-bound `openclaw-channels` child processes.

## Best Possible Solution

Enforce the existing capabilities timeout around all plugin probe/diagnostic work and keep dependency staging or plugin setup discovery bounded, while preserving existing Discord semantics.

## Reproduction Assessment

Yes, from source: a configured channel plugin whose `probeAccount` or `buildCapabilitiesDiagnostics` promise never settles will keep `channels capabilities` waiting because the command awaits those hooks directly. The exact Discord CPU-bound child process was not live-reproduced.

## Solution Assessment

Yes: a narrow repair should enforce a command-level timeout/fallback around plugin hook and loader work. Changing the existing optional-channel install-on-demand behavior should be treated as a separate product choice unless the fix proves it is directly required to stop the hang.
