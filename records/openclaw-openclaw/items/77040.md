## Summary

Keep open: current main still has an unbounded capability-hook await path, and the explicit Discord diagnostic path can still enter heavier plugin resolution work before returning.

## What This Changes

The issue reports that `openclaw channels capabilities --channel discord` hangs, stages plugin runtime dependencies, and leaves a CPU-bound `openclaw-channels` process instead of returning bounded diagnostics.

## Best Possible Solution

Bound the diagnostic path end to end: enforce the existing timeout around capability hooks and keep already-configured Discord inspection on a lightweight read-only path that does not do unnecessary plugin mutation work.

## Reproduction Assessment

Yes, from source: a configured channel plugin whose probe or capability diagnostics hook never settles will keep `channels capabilities` waiting because the command awaits those hooks directly. The exact Discord CPU-bound child process was not live-reproduced in this read-only pass.

## Solution Assessment

Yes, a narrow repair is appropriate: enforce the existing `--timeout` budget around all capability hook work and avoid unnecessary install/registry refresh work for read-only Discord diagnostics. That preserves the current CLI contract without adding configuration.
