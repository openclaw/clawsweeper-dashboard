## Summary

Keep open: current `main` still leaves the documented `channels capabilities` timeout as data passed into plugin hooks rather than an end-to-end command bound, and the explicit-channel path can still enter install/setup plugin discovery before reporting capabilities.

## What This Changes

The issue reports that `openclaw channels capabilities --channel discord` hangs and leaves an `openclaw-channels` child process pegging CPU instead of returning quickly or timing out.

## Best Possible Solution

Keep the diagnostic read-only and bounded end to end: enforce the existing timeout around capability probes/diagnostics and avoid dependency-staging or config mutation during capability inspection.

## Reproduction Assessment

Yes, partially from source. The live CPU-bound child process was not reproduced here, but current `main` still has unbounded awaits around capability plugin hooks and a setup/install discovery path on the reported command.

## Solution Assessment

Yes, a narrow repair is the best path. The command already has a timeout option, so the maintainable fix is to apply that budget around the whole capability hook path and keep install/repair work out of this read-only diagnostic.
