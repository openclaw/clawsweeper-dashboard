## Summary

Keep open: current main still has a plausible bounded-diagnostic bug in the `channels capabilities` path. The Discord HTTP probe itself is bounded, but the command still awaits plugin hooks directly and can enter setup/plugin loading and config-refresh work for an explicit channel.

## What This Changes

The issue reports that `openclaw channels capabilities --channel discord` hangs and leaves a CPU-bound `openclaw-channels` child process instead of returning bounded Discord capability diagnostics.

## Best Possible Solution

Keep the diagnostic bounded end to end: enforce the requested timeout around capability plugin hooks and keep read-only capability inspection away from install/runtime-dependency repair work unless that mutation is an explicit setup operation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: source inspection shows current-main unbounded awaits around capability hooks and a loader/install path for this command, but I did not live-reproduce the reported CPU-bound child process.

## Solution Assessment

Is this the best way to solve the issue? Yes, a narrow repair should bound the existing diagnostic command rather than add a new mode or config option; changing broader plugin install semantics should be avoided unless needed to keep the command read-only.
