## Summary

Keep open: current main still has a source-reproducible unbounded `channels capabilities` path, and the explicit-channel path can still enter plugin setup/install resolution before returning diagnostics.

## What This Changes

`openclaw channels capabilities --channel discord` is reported to hang during Discord diagnostics, leave a CPU-bound `openclaw-channels` child process, and ignore the expected bounded timeout behavior.

## Best Possible Solution

Keep the diagnostic command bounded under the existing timeout, prefer already-registered/read-only channel data for configured channels, and prevent dependency staging or plugin loader work from blocking capability output indefinitely.

## Reproduction Assessment

Yes, from source: a configured channel plugin whose `probeAccount` or `buildCapabilitiesDiagnostics` promise never settles will keep `channels capabilities` waiting despite `--timeout`. I did not live-reproduce the Discord CPU-bound child process itself.

## Solution Assessment

Yes: the narrow maintainable fix is to enforce the existing timeout across all capability discovery work and avoid unnecessary setup/install resolution for already configured channels. That fixes the broken diagnostic behavior without adding a new mode or config knob.
