## Summary

The command remains a documented Discord diagnostic, but it can still load configured channel plugins and the targeted Discord permission audit does not carry the CLI timeout into the REST permission fetch path.

## What This Changes

The issue reports that `openclaw channels capabilities --channel discord` hangs, leaves a CPU-bound `openclaw-channels` child process, and should instead return capability information or fail with a bounded error.

## Best Possible Solution

Make the diagnostic command a bounded read-only path that avoids full channel runtime activation where metadata is enough and enforces the CLI timeout through Discord probes and targeted permission checks.

## Reproduction Assessment

No live production reproduction was run, so we do not have a high-confidence runtime repro. Source inspection on current main still shows the reported diagnostic path can await unbounded Discord permission work after loading configured channel plugins.

## Solution Assessment

Yes, the narrow repair is still the best direction. Keep `channels capabilities` read-only and bounded, and carry the existing CLI timeout through Discord diagnostics instead of adding a new mode or config option.
