## Summary

Keep open: current main still preserves the risky diagnostic shape. The Discord targeted capabilities path receives a CLI timeout but does not pass it into permission lookup, and the latest release shows the same behavior, so this issue is still necessary.

## What This Changes

The issue reports that `openclaw channels capabilities --channel discord` hangs, leaves a CPU-bound `openclaw-channels` child process, and should instead finish quickly or fail with a bounded diagnostic error.

## Best Possible Solution

Make `channels capabilities` a bounded read-only diagnostic path: carry the CLI timeout through Discord target permission diagnostics and avoid full configured plugin activation when metadata/setup artifacts are enough.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection does show the reported command’s Discord target path still awaits permission diagnostics without enforcing the CLI timeout, matching the expected hang class.

## Solution Assessment

Yes, a narrow repair is the best path. The current adapter contract already supplies `timeoutMs`, so the fix should propagate that through Discord permission/audit fetches and add regression coverage rather than adding a new config option or broader product mode.
