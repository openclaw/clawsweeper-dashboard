## Summary

Keep open: current main still has direct, unbounded awaits around channel capability hooks, and the explicit-channel path can still enter install/config/registry refresh work that matches the reported staging symptom.

## What This Changes

The issue requests that `openclaw channels capabilities --channel discord` return promptly or fail with a bounded timeout instead of hanging and leaving CPU-bound child processes.

## Best Possible Solution

Keep the diagnostic path bounded end to end by wrapping capability status/diagnostic hooks in `--timeout` handling and keeping read-only capability inspection off install/registry mutation paths unless setup is explicit.

## Reproduction Assessment

Yes, from source for the core failure mode: a configured channel plugin whose capability probe or diagnostics hook never settles will keep the command waiting. The exact Discord CPU-bound child process was not live-reproduced.

## Solution Assessment

Yes, the narrow maintainable fix is to enforce the existing CLI timeout around all capability hook work and avoid unnecessary install/registry refresh work in this diagnostic path. This should not require new Discord semantics or a config knob.
