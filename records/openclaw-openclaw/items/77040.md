## Summary

Keep open: current main still leaves the documented Discord capability diagnostic vulnerable to an unbounded capability hook/discovery path, and the prior ClawSweeper comment plus the linked plugin-loader overhead report support treating this as valid follow-up work.

## What This Changes

The issue reports that `openclaw channels capabilities --channel discord` can hang, leave an `openclaw-channels` child at high CPU, and should return capability data or fail within a bounded timeout.

## Best Possible Solution

Bound the capabilities command end to end with timeout-aware plugin hook handling, keep read-only capability inspection away from unnecessary install/registry mutation work, and add focused regression coverage for hanging hooks and configured Discord capability probes.

## Reproduction Assessment

Partially: source inspection gives a high-confidence path for an unbounded hang around capability hooks and plugin discovery, but the exact production `openclaw-channels` 99% CPU process was not live-reproduced in this read-only review.

## Solution Assessment

Yes, a narrow repair is the best path: enforce the existing `--timeout` budget around capability probes/diagnostics and avoid read-only capability inspection triggering dependency staging unless setup is explicitly required.
