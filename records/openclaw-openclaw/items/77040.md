## Summary

Keep open: the report describes a valid existing diagnostic command hanging under a configured Discord channel, and current main still lacks a command-level timeout around capability plugin hooks even though the Discord HTTP probes themselves are bounded.

## What This Changes

`openclaw channels capabilities --channel discord` hangs during Discord capability diagnostics and leaves a CPU-bound child process instead of returning quickly or failing with a bounded error.

## Best Possible Solution

Keep the diagnostic command bounded end to end: enforce the CLI timeout around plugin capability hooks and avoid dependency-staging/config mutation on the read-only capability path unless the user explicitly runs setup/install.

## Reproduction Assessment

Yes, partially from source. The exact CPU-bound child process was not live-reproduced, but current main has unbounded awaits around capability plugin hooks and an explicit install/registry-refresh path that can enter plugin loading during the reported command.

## Solution Assessment

Yes, a narrow fix should be possible. The maintainable direction is to wrap capability probes/diagnostics in command-level timeout handling and keep read-only capability inspection off install/runtime-dependency repair paths unless setup is explicitly requested.
