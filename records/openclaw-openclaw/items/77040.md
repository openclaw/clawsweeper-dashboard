## Summary

Keep open: current main still has an unbounded capability-hook await path, and the explicit-channel command can still enter plugin install/registry refresh work that matches the reported dependency-staging symptom.

## What This Changes

The issue asks for `openclaw channels capabilities --channel discord` to return promptly or fail with a bounded timeout instead of hanging and leaving CPU-bound channel child processes.

## Best Possible Solution

Keep the diagnostic path bounded end to end: wrap capability hooks in command-level timeout handling and keep read-only capability inspection off install/registry mutation paths unless setup is explicitly requested.

## Reproduction Assessment

Yes, from source: a configured channel plugin whose probe or diagnostics hook never settles will keep `channels capabilities` waiting because the command awaits those hooks directly. The exact Discord CPU-bound child process was not live-reproduced.

## Solution Assessment

Yes, a narrow repair is appropriate: enforce the existing CLI timeout around all capability hook work and avoid read-only capability commands triggering install/registry refresh work. This should not require changing Discord product semantics or adding new configuration.
