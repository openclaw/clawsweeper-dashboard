## Summary

Keep open: current main still has unbounded awaits in `channels capabilities`, and the CLI path still preloads configured channel plugins, so the reported hang/CPU symptom remains actionable under the existing `--timeout` contract.

## What This Changes

The issue reports that `openclaw channels capabilities --channel discord` hangs, emits plugin dependency staging output, and leaves an `openclaw-channels` child process at about 99% CPU instead of returning bounded Discord capability diagnostics.

## Best Possible Solution

Bound the capabilities diagnostic path end to end with the existing CLI timeout, including plugin hook work and any capability-triggered plugin loading, while preserving Discord semantics and avoiding new configuration.

## Reproduction Assessment

Yes, from source: a configured enabled channel plugin whose `probeAccount` or `buildCapabilitiesDiagnostics` promise never settles will keep `channelsCapabilitiesCommand` waiting despite `--timeout`. The exact CPU-bound Discord child process was not live-reproduced.

## Solution Assessment

Yes, the narrowest maintainable fix is to enforce the existing `--timeout` across capability probe/diagnostic and plugin-load work, with an actionable timeout error. A broader change to install-on-demand policy should be avoided unless maintainers explicitly want that behavior changed.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77040. On current main, `openclaw channels capabilities --channel discord` can hang because the command preloads configured channel plugins and directly awaits plugin `probeAccount` / `buildCapabilitiesDiagnostics` hooks while Discord target diagnostics ignore the passed timeout. Implement bounded timeout handling for the capabilities command path so probe/diagnostics and any capability-triggered plugin loading fail with an actionable timeout instead of hanging or leaving channel helper processes running. Add regression coverage using a never-settling channel hook and, if touching Discord diagnostics, ensure the target permission audit path respects the timeout. Preserve existing public behavior where possible, especially the current explicit optional-channel install behavior unless the change deliberately updates the test/docs contract. Do not add a new config option or change Discord capability semantics. Add a user-facing `CHANGELOG.md` fix entry.
