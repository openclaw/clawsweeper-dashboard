## Summary

Keep open: current main still has an unbounded `channels capabilities` command path around plugin capability hooks, and the explicit-channel path can still enter plugin install/registry refresh work. The Discord HTTP probe itself is bounded, so the remaining problem is the broader CLI/plugin capability flow reported here.

## What This Changes

The issue reports that `openclaw channels capabilities --channel discord` hangs and leaves an `openclaw-channels` child process consuming CPU instead of returning capability information or timing out.

## Best Possible Solution

Keep `channels capabilities` bounded end to end: enforce the CLI timeout around capability hooks and make the read-only capability path avoid dependency staging or heavy setup loading unless setup/install is explicitly requested.

## Reproduction Assessment

Yes, from source for the unbounded command path: current main awaits capability plugin hooks directly after parsing `--timeout`. The exact CPU-bound `openclaw-channels` process was not live-reproduced in this read-only review.

## Solution Assessment

Yes for a narrow repair: wrap capability probes/diagnostics in command-level timeout handling and add regression coverage. Any broader change to optional channel install/setup behavior should be limited to preventing read-only diagnostics from doing setup work unexpectedly.
