## Summary

Keep open: current `main` still has an unbounded capability-hook await path, and the Discord targeted diagnostics path still does live permission work without carrying the CLI timeout through that call.

## What This Changes

The issue reports that `openclaw channels capabilities --channel discord` hangs and leaves an `openclaw-channels` child pegging CPU instead of returning bounded diagnostics.

## Best Possible Solution

Bound the `channels capabilities` diagnostic path end to end and keep read-only capability inspection from triggering install or registry mutation unless setup is explicitly requested.

## Reproduction Assessment

Yes, from source: a configured channel plugin whose probe or diagnostics hook never settles will keep `channels capabilities` waiting because the command awaits those hooks directly. The exact Discord CPU-bound child process was not live-reproduced.

## Solution Assessment

Yes, a narrow repair is the best path: enforce the existing CLI timeout around capability hook work and avoid read-only capability commands triggering install/registry refresh work unnecessarily.
