The issue reports that `openclaw channels capabilities --channel discord` hangs and leaves an `openclaw-channels` child process pegging CPU instead of returning capability data or timing out.

## Best Possible Solution

Keep this issue open and make `channels capabilities` a bounded read-only diagnostic path that honors the CLI timeout through Discord permission checks and avoids unnecessary heavy plugin activation where the lightweight channel surface is sufficient.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection does reproduce the unbounded class: the CLI timeout reaches the adapter contract, but Discord targeted permission diagnostics do not carry it into the REST permission lookup.

## Solution Assessment

Yes, a narrow repair is the best path. The adapter contract already has `timeoutMs`, so the fix should propagate that deadline through Discord permission diagnostics and add regression coverage before considering broader plugin-loader changes.
