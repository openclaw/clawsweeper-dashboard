## Summary

Keep open: current main still awaits channel capability hooks without an outer command timeout, and the explicit-channel path can still enter install/registry refresh work that matches the reported dependency-staging symptom.

## What This Changes

The issue asks for `openclaw channels capabilities --channel discord` to return bounded Discord capability diagnostics instead of hanging and leaving a CPU-bound channel child process.

## Best Possible Solution

Bound the capability diagnostic path end to end and keep read-only capability inspection out of plugin install or registry mutation work unless setup is explicitly requested.

## Reproduction Assessment

Yes, from source: a configured channel plugin whose `probeAccount` or `buildCapabilitiesDiagnostics` promise never settles will keep `channels capabilities` waiting despite the CLI timeout option. The exact Discord CPU-bound child process was not live-reproduced.

## Solution Assessment

Yes: the narrow maintainable fix is to enforce the existing CLI timeout around all capability hook work and make the explicit-channel diagnostic path read-only unless installation/setup is intentionally requested.
