## Summary

Keep open: current main still has unbounded awaits in the channel capabilities command and an explicit plugin install/registry-refresh path for a read-style diagnostic, so the reported Discord hang remains actionable.

## What This Changes

Make `openclaw channels capabilities --channel discord` return promptly or fail with a bounded timeout instead of hanging and leaving CPU-bound channel processes.

## Best Possible Solution

Bound the capability diagnostic end to end by enforcing timeouts around plugin capability hooks and Discord target diagnostics while preserving existing install semantics unless separately reviewed.

## Reproduction Assessment

Yes, from source inspection rather than a live production run. The docs expose a timeout, but current main forwards that value into plugin hooks without wrapping the awaited hook or setup/plugin-loading path that the report says became CPU-bound.

## Solution Assessment

Yes, a narrow repair is available. Add command-level timeout handling and regression tests for slow capability hooks, and propagate/observe the timeout in Discord capability diagnostics without changing broader channel setup policy.
