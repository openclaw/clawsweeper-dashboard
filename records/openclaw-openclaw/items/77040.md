## Summary

Keep open: current main still has unbounded awaits and mutation-capable plugin resolution in the documented diagnostic command; the Discord HTTP probe itself is bounded, so the remaining risk sits in the broader command/plugin path.

## What This Changes

The issue asks for `openclaw channels capabilities --channel discord` to return capability data quickly or fail with a bounded timeout instead of hanging and leaving CPU-bound channel child processes.

## Best Possible Solution

Make the diagnostic bounded end to end with actionable timeout errors; if explicit-channel install/staging remains part of capabilities, it must also be cancellable and must not leave child processes behind.

## Reproduction Assessment

Partially. Source inspection of current main shows a credible unbounded command path through plugin hooks and install/registry refresh, but I did not live-reproduce the CPU-bound child process.

## Solution Assessment

Yes. A narrow repair should enforce the existing command timeout across plugin selection, probes, diagnostics, and any invoked install/registry work while preserving current channel capability semantics unless maintainers explicitly choose a read-only policy change.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77040. `openclaw channels capabilities --channel discord` can hang during a diagnostic and leave an `openclaw-channels` child pegging CPU. Keep the existing user-facing command semantics, but make the capabilities command bounded end to end: apply the normalized command timeout around explicit channel plugin resolution, `probeAccount`, `buildCapabilitiesDiagnostics`, and any install/registry-refresh work they can invoke; convert timeouts into actionable probe/diagnostic errors instead of hanging. Ensure timed-out child/staging work is cancelled or cannot be left CPU-bound. Preserve current optional-channel install behavior unless the implementation clearly requires a policy change; if it does, stop and ask for maintainer review rather than silently making `channels capabilities` strictly read-only. Add regression coverage for a never-resolving capability hook and for the explicit channel path so config writes/registry refreshes are either bounded or avoided by design. Add a user-facing `CHANGELOG.md` fix entry. Do not change Discord channel product behavior, plugin IDs, provider defaults, or public plugin API shape unless the timeout repair strictly requires it.
