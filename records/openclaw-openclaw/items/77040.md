## Summary

Keep open: current main and the latest release still await channel capability hooks directly, and the Discord targeted diagnostics path does live permission work without propagating the CLI timeout through that call. The exact CPU-bound child process was not live-reproduced, but the bounded-timeout contract is still not enforced end to end.

## What This Changes

The issue asks for `openclaw channels capabilities --channel discord` to complete quickly or fail with a bounded timeout instead of hanging and leaving CPU-bound `openclaw-channels` processes.

## Best Possible Solution

Enforce a command-level timeout/fallback around capability probes and diagnostics, add regression coverage with never-settling hooks, and keep any install-on-demand policy change separate unless needed for the fix.

## Reproduction Assessment

Yes, from source for the unbounded wait: a configured channel plugin whose probe or diagnostics hook never settles will keep `channels capabilities` awaiting that hook. I did not live-reproduce the Discord-specific 99% CPU child process.

## Solution Assessment

Yes: the narrow maintainable fix is to enforce the existing CLI timeout around capability hook work and add focused regression tests. Broader plugin loader or install-staging changes should wait for profiling or maintainer policy direction.
