## Summary

Keep open: current main still has an unbounded capability hook await path, and the explicit Discord channel path can still enter install/registry refresh work consistent with the reported dependency-staging symptom.

## What This Changes

The issue reports that `openclaw channels capabilities --channel discord` hangs, leaves a CPU-bound `openclaw-channels` child process, and should return or time out with actionable output.

## Best Possible Solution

Make `channels capabilities` bounded end to end and keep capability inspection read-only unless the user explicitly runs setup or install flows.

## Reproduction Assessment

Yes, from source for the hang class: a configured channel plugin whose probe or diagnostics hook never settles will keep the command waiting because current main awaits those hooks directly. The exact Discord CPU-bound child process was not live-reproduced.

## Solution Assessment

Yes, a narrow repair is the best fit: enforce the existing CLI timeout around all capability hook work and avoid read-only capability commands triggering install or registry refresh work. This should not require new Discord semantics or configuration.

## Repair Work Prompt

Fix the `openclaw channels capabilities --channel discord` hang reported in https://github.com/openclaw/openclaw/issues/77040. Keep the change scoped to the diagnostic command and channel plugin resolution path: wrap `probeAccount` and `buildCapabilitiesDiagnostics` with command-level timeout handling, make timeout errors render as actionable probe/diagnostic failures instead of hanging, and stop `channels capabilities` from performing install/config/registry mutation work unless an explicit setup/install command is used. Add focused regression tests for a never-settling probe or diagnostics hook and for explicit channel capability inspection remaining read-only. Do not add a new config option or change Discord capability semantics. Add a one-line `CHANGELOG.md` Fixes entry because this affects user-facing CLI behavior.
