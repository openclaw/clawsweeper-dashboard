## Summary

Keep open: current main and the latest release still preserve the reported risky diagnostic path. The CLI accepts a timeout, but Discord targeted capability diagnostics await permission fetching without consuming or enforcing that timeout, and the command remains a supported Discord voice diagnostic.

## What This Changes

The issue reports that `openclaw channels capabilities --channel discord` hangs and leaves CPU-bound `openclaw-channels` child processes instead of returning or timing out.

## Best Possible Solution

Make `channels capabilities` a bounded, lightweight diagnostic path by enforcing `--timeout` through Discord target permission checks and avoiding full configured-channel runtime activation when metadata/setup artifacts are enough.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection still shows the likely failure path: the CLI timeout reaches the diagnostic contract, but Discord targeted permission diagnostics ignore it before awaiting REST permission fetches.

## Solution Assessment

Yes, the best fix is narrow. Keep the supported diagnostic read-only and bounded rather than adding a new mode or broad compatibility layer.
