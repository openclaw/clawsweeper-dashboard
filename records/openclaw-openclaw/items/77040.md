## Summary

Keep open: current main still exposes a bounded `channels capabilities` timeout contract but directly awaits capability hooks and can enter plugin install/registry refresh work on the explicit channel path, so the reported hang is not obsolete. The linked #77029 plugin-loader overhead context is relevant, but it does not supersede this command-specific timeout bug.

## What This Changes

The issue reports that `openclaw channels capabilities --channel discord` hangs and leaves a CPU-bound `openclaw-channels` child process instead of returning bounded capability diagnostics.

## Best Possible Solution

Bound the capability command end to end with the existing `--timeout` budget, including plugin probe/diagnostic hooks and Discord permission diagnostics, while preserving current CLI semantics unless a minimal bounded fallback is required.

## Reproduction Assessment

Yes, source-reproducible: current main directly awaits the relevant plugin hooks and Discord diagnostics while the documented command offers `--timeout`. The exact CPU-bound child was not live-reproduced, but the unbounded source path remains present.

## Solution Assessment

Yes, the narrow maintainable fix is to enforce the existing timeout around the command's hook and diagnostic boundaries, then add regression coverage with never-settling hooks. Broader plugin-loader caching belongs outside this focused repair unless it is needed to keep this command bounded.
