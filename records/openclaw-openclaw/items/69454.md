## Summary

Keep open. Current main still has the unbounded Control UI session refreshes, immediate `sessions.changed` reload, global node/device work, and `index.html` `no-cache` behavior this PR targets, so the PR is not obsolete or implemented on main. The branch still needs a functional fix for delayed `sessions.changed` reloads after tab switches, and #45930 is overlapping but narrower polling-interval work rather than a superseding fix.

## What This Changes

The PR bounds Control UI chat/session session-list refreshes, scopes node polling and node/device preloads to the Nodes tab, debounces `sessions.changed` reloads away from Chat, and serves Control UI `index.html` with `Cache-Control: no-store`, with targeted tests.

## Best Possible Solution

Land a refreshed, narrow version that keeps the Control UI churn reductions and `index.html` no-store hardening, preserves current session-event payload application, and guards any delayed session reload against tab changes. Keep interval tuning coordinated with #45930 so the two fixes do not drift into conflicting polling policy.

## Reproduction Assessment

Yes. On the PR head, a high-confidence unit path is to fire `sessions.changed` while on Overview, switch `host.tab` to Chat before advancing the 750 ms timer, and observe that the scheduled callback still calls `loadSessions`. Current-main inspection also confirms the original churn paths are still present.

## Solution Assessment

No, not as-is. The direction is narrow and maintainable, but the delayed callback should re-check `shouldReloadSessionsForEvent(host)` or be cleared on tab changes before `loadSessions` runs; the branch also needs a current-main refresh and targeted test proof.
