## Summary

Keep open: current main and latest v2026.6.9 still let an oversized assistant placeholder count as renderable final text, so Control UI can skip the automatic history reload that would hydrate the persisted audio attachment. The report is source-plausible, but it still needs live browser/Gateway proof and a maintainer choice because several open WebChat media/reload PRs overlap without owning this exact symptom.

## What This Changes

The issue asks Control UI to render large audio attachments inline as soon as the message arrives instead of showing `message too large` until manual refresh.

## Best Possible Solution

Keep this issue open until maintainers land a focused first-render hydration fix for oversized/truncated media finals, or verify that an existing WebChat media/reload PR fixes the exact audio placeholder case.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live repro was established, but source inspection shows a plausible current-main path where the oversized placeholder is treated as renderable final text and skips the automatic history reload that manual refresh performs.

## Solution Assessment

Is this the best way to solve the issue? The requested behavior is the right end state, but the implementation should be narrow: reload or hydrate only oversized/truncated media finals while preserving the no-flicker optimization for normal renderable finals.
