## Summary

Keep open: current main can render audio attachments inline, but the reported refresh-only behavior still points to a live Control UI hydration/reload gap where an oversized placeholder can remain visible until manual history refresh.

## What This Changes

The issue reports that Control UI shows a large generated audio attachment as "message too large" until manual refresh, instead of rendering the inline audio player immediately.

## Best Possible Solution

Keep `chat.history` bounded, but make the live Control UI final/session-message path hydrate the persisted attachment projection when the first visible row is the oversized placeholder so the audio player appears without manual refresh.

## Reproduction Assessment

No, not as a live high-confidence repro: the report has concrete steps and current source shows the likely placeholder/reload path, but I did not run the Windows/Edge `music_generate` scenario.

## Solution Assessment

Yes as a direction: the narrow fix should preserve the bounded history placeholder and no-flicker optimization for normal finals while reloading or hydrating the placeholder case.
