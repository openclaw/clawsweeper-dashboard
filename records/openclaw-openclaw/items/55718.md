## Summary

Keep open. Current main still has channel account display names and stored session label/displayName support, but there is no account-config-to-session-metadata bridge for newly created inbound sessions; the remaining decision is the public config spelling, since `name` already exists while the issue proposes `displayLabel`.

## What This Changes

Request to configure friendly channel-account labels and have new Control UI session rows use them instead of technical session keys or origin labels.

## Best Possible Solution

Add a generic session metadata path that resolves the selected channel account display name, seeds only new or unlabeled sessions, and documents/tests the chosen config spelling.

## Reproduction Assessment

Yes, at source level. Configure a channel account with `name` or the proposed `displayLabel`, create a new inbound session, and current `recordSessionMetaFromInbound` cannot use that config because only message context reaches metadata derivation.

## Solution Assessment

Unclear. The requested behavior is useful, but reusing existing account `name` appears safer than adding a second label field unless maintainers intentionally want `displayLabel` as an alias.
