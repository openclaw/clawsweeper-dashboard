## Summary

Keep open. Current `main` still supports manual stored session labels and account `name` values for account/status surfaces, but it does not seed new inbound channel sessions from channel account config. The related closed custom-session-label items cover manual or UI display behavior; they do not cover account-config-driven defaults for newly created channel sessions.

## What This Changes

Add a channel-account display label configuration path that automatically gives newly created Control UI channel sessions a friendly label instead of a raw session key or technical origin label.

## Best Possible Solution

Implement a narrow shared channel/session metadata enhancement that resolves the configured account display name for the inbound channel/account pair, seeds stored session display data for new or unlabeled sessions, and documents the supported config spelling with regression coverage across session storage, gateway rows, and Control UI rendering.

## Reproduction Assessment

Yes. A code-level reproduction is clear: configure a channel account with a friendly `name` or proposed `displayLabel`, create an inbound session with that `AccountId`, and the current `recordSessionMetaFromInbound` path has no config input or account-label resolver to use it.

## Solution Assessment

Unclear for the exact proposed field name. Reusing the existing account `name` field as the default session label source is the narrowest maintainable path; adding `displayLabel` should be an intentional documented alias only if maintainers want that public spelling.
