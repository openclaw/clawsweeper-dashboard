## Summary

Keep open. Current main still does not seed Control UI session labels from channel account configuration; the existing account `name` field and stored session `label`/`displayName` paths are present, but there is no config-to-session metadata bridge for newly created inbound sessions.

## What This Changes

The issue requests configuration-driven friendly display labels for channel accounts so newly created Control UI sessions show human-readable names without editing `sessions.json`.

## Best Possible Solution

Add a narrow shared channel/session metadata path that resolves the configured display name for the inbound channel account, seeds only new or unlabeled sessions, and documents the chosen config spelling with regression coverage.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure a channel account with a friendly `name` or proposed `displayLabel`, create an inbound session for that account, and current `recordSessionMetaFromInbound` cannot use the account config because it only passes message context into metadata derivation.

## Solution Assessment

Unclear. The requested `displayLabel` field is not clearly the best standalone API because account `name` already exists; the safer direction is to reuse `name` as the default session label source unless maintainers intentionally want `displayLabel` as an alias.
