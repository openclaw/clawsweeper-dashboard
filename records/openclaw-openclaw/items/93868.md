## What This Changes

The PR changes `tts.status` and speech-core provider ordering so status diagnostics reuse one speech-provider inventory/configured-state pass and provider-order resolution can consume that prebuilt inventory.

## Best Possible Solution

Move the first `tts.status` yield before synchronous TTS config/prefs work, keep the single provider inventory/configured-state pass, and add redacted real Gateway or Control UI fanout proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main's `tts.status` has no await and runs config, prefs, fallback-provider filtering, and provider-state diagnostics synchronously before responding; I did not run the full dashboard cold-start timing scenario.

## Solution Assessment

No. The provider-inventory reuse is the right narrow layer, but the first yield needs to move before the synchronous config/prefs segment to fully address the reported fanout front-block.
