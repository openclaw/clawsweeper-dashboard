## Label Justifications

- `P2`: This is a normal-priority Control UI correctness fix for multi-base-path deployments, with limited but real wrong-Gateway impact.
- `merge-risk: 🚨 compatibility`: The diff changes shipped localStorage fallback and migration behavior that existing browser profiles may rely on for saved custom Gateway settings.
- `merge-risk: 🚨 message-delivery`: A preserved stale Gateway URL can route WebSocket traffic, and therefore user messages, to a sibling backend.
- `merge-risk: 🚨 session-state`: The affected persisted settings include session selection, so accepting or resetting the wrong record can associate the page with the wrong session surface.

## Summary

Keep open: the branch fixes the main absolute sibling-base-path legacy fallback and now includes useful terminal proof, but the new custom-endpoint exception is too broad and can still preserve a legacy same-origin sibling path such as `/gateway-a/ws` when the current page is `/gateway-b`.

## What This Changes

The PR changes Control UI settings load/save to gate legacy global localStorage fallback by page-derived Gateway path, migrate compatible legacy/custom endpoints, remove the legacy key on save, and add storage regression tests.

## Best Possible Solution

Keep the storage-boundary fix, but narrow the legacy custom-endpoint exception so root `/ws` remains migratable while sibling base-path `/.../ws` records are rejected or explicitly migrated to the current page-derived Gateway.

## Reproduction Assessment

Yes by source inspection: seed the legacy global settings key with `/gateway-a/ws` or `wss://example.com/gateway-a/ws`, load `/gateway-b/chat` with no current scoped record, and the PR head's exception accepts the stale URL. `connectGateway()` then passes that URL to `GatewayBrowserClient`.

## Solution Assessment

No. The storage boundary is the right place to fix the issue, but the current exception is too broad; the safer fix should distinguish root custom `/ws` endpoints from sibling base-path `/.../ws` records.
