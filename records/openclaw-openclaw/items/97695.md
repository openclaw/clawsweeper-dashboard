## Label Justifications

- `P2`: This is a normal-priority Control UI correctness fix for multi-base-path deployments, with limited but real wrong-Gateway impact.
- `merge-risk: 🚨 compatibility`: The PR changes shipped browser localStorage fallback and migration behavior that existing browser profiles may rely on for saved custom Gateway settings.
- `merge-risk: 🚨 message-delivery`: A preserved stale Gateway URL can route WebSocket traffic, and therefore user messages, to a sibling backend.
- `merge-risk: 🚨 session-state`: The affected persisted settings include session selection, so accepting or resetting the wrong record can associate the page with the wrong session surface.

## Summary

Keep open: this PR is a useful candidate for the canonical Control UI wrong-Gateway bug, but the current helper still treats any same-origin path ending in `ws` as a migratable custom endpoint, so a legacy sibling `/gateway-a/ws` record can survive when loading `/gateway-b`.

## What This Changes

The PR changes Control UI settings load/save to gate legacy global localStorage fallback by page-derived Gateway scope, write page-scoped settings, remove the shared legacy key, and add storage regression tests.

## Best Possible Solution

Keep the storage-boundary fix, but narrow the legacy custom-endpoint exception so root `/ws` remains migratable while sibling base-path `/.../ws` records are rejected or migrated to the current page-derived Gateway.

## Reproduction Assessment

Yes by source inspection: seed the legacy global settings key with `/gateway-a/ws` or `wss://example.com/gateway-a/ws`, load `/gateway-b/chat` with no scoped record, and PR head accepts the stale URL before `connectGateway()` passes it to the WebSocket client.

## Solution Assessment

No. The storage boundary is the right layer, but the current exception is too broad because it distinguishes root `/ws` from neither sibling base-path `/.../ws` nor absolute same-origin equivalents.
