## Summary

Keep open. The PR addresses a real duplicate-final-payload path, but the patch is broader than the product contract: it would suppress every normal assistant answer after any successful message tool text send, including proactive or cross-target sends where the original requester still needs a confirmation. The PR is also draft and has the protected `maintainer` label, so it needs maintainer handling rather than cleanup closure.

## What This Changes

The PR changes `buildEmbeddedRunPayloads` to suppress assistant answer payloads when `didSendViaMessagingTool` is true and adds regression tests for streamed assistant text and final-answer fallback suppression after message-tool delivery.

## Best Possible Solution

Keep the PR open and narrow the suppression so it only drops final payloads that are known duplicates of the message-tool delivery, or sends that targeted the current/original chat as the final visible reply. Preserve normal final confirmations after proactive or cross-target sends, and keep the deterministic approval suppression behavior unchanged.

## Reproduction Assessment

Yes. A unit-level reproduction is `buildEmbeddedRunPayloads({ assistantTexts: ["Done, I told Jane."], didSendViaMessagingTool: true, ...defaults })`: the PR’s new condition would return no normal assistant payload even though the prompt contract allows proactive message sends followed by a requester-facing confirmation.

## Solution Assessment

No. Suppressing final assistant answers solely from `didSendViaMessagingTool` is not the narrowest maintainable fix; the safer path is to compare sent text/target against the final payload or pass richer send metadata into the payload builder.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve replies after proactive message sends:** `src/agents/pi-embedded-runner/run/payloads.ts:356`
  - body: `didSendViaMessagingTool` only means some message text was committed successfully. The message tool is also advertised for proactive sends, and the silent-reply guidance is scoped to using it to deliver the user-visible reply; with this blanket condition, sending a message to another target and then answering the original requester will produce no outbound payload. Please suppress only duplicate or same-target final-delivery answers, or pass enough sent-text/target data to distinguish that case.
  - confidence: 0.82
