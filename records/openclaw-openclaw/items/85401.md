## Summary

Keep open. This is a fresh production report on v2026.5.20 with attachment loss and local-path exposure; current main has generic MEDIA parsing and WhatsApp document-send support, but the WhatsApp auto-reply path still has enough unparsed-directive handoff risk that it needs a focused regression test and fix rather than closure.

## What This Changes

Investigate a WhatsApp group auto-reply case where a final assistant `MEDIA:` directive for a PDF/document was shown as chat text instead of delivered as an attachment.

## Best Possible Solution

Normalize reply directives before every WhatsApp auto-reply delivery branch, preserve existing media path policy, and add regression coverage for group final PDF/document `MEDIA:` directives plus failed-media fallback that never exposes local paths.

## Reproduction Assessment

Yes, at source level for the vulnerable branch: if raw `MEDIA:` text reaches WhatsApp delivery without parsed `mediaUrl`/`mediaUrls`, `deliverWebReply` treats it as text-only. The live current-main WhatsApp path was not rerun in this read-only review.

## Solution Assessment

Yes, the best fix is a narrow reuse of existing reply directive/media-path normalization at the WhatsApp dispatch boundary. Adding a separate WhatsApp parser or changing CLI media send would be the wrong boundary.
