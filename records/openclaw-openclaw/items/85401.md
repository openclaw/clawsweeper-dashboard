## Summary

Keep open: current main and the latest release cover the successful final-reply media path better than the original report, including PDF/document sends, but the blocked absolute-host-path path still collapses to a generic WhatsApp-visible media failure rather than a sanitized actionable policy error.

## What This Changes

The issue reports WhatsApp auto-reply final `MEDIA:` PDF/document directives rendering as raw text or degrading to a generic failure instead of being delivered as attachments or rejected with a sanitized explanation.

## Best Possible Solution

Keep the issue open for a maintainer/security decision, then add a sanitized error taxonomy for blocked local media that preserves current successful workspace/relative media delivery and never exposes raw host paths.

## Reproduction Assessment

Yes, at source level. Current main has tests and code paths showing blocked absolute host-local reply media is dropped and converted to a generic warning; I did not run a live WhatsApp group reproduction in this read-only review.

## Solution Assessment

No, the current behavior is only a partial mitigation: it strips media instead of leaking the directive, but it does not provide the sanitized actionable local-media-root rejection the report asks for. The safer fix is to map `LocalMediaAccessError` reasons into approved user-visible text while keeping raw paths out of chat.
