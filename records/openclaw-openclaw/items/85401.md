## Summary

Keep open: current main and v2026.6.9 no longer look like the original raw-path leak in many `MEDIA:` cases, but blocked WhatsApp final-reply media still degrades to the generic `⚠️ Media failed.` text instead of the requested sanitized, actionable local-media-policy error.

## What This Changes

The issue asks OpenClaw to parse WhatsApp final-reply `MEDIA:` PDF/document directives, deliver allowed attachments, and avoid exposing raw local paths when delivery is blocked.

## Best Possible Solution

Define a maintainer-approved sanitized `path-not-allowed` fallback in the shared reply-media policy, then add focused WhatsApp PDF/document coverage that preserves allowed-root denial and strips raw directives.

## Reproduction Assessment

Yes, at source level: current main still drops blocked reply media and appends/sends the generic `⚠️ Media failed.` warning, with tests preserving that behavior. I did not run a live WhatsApp group reproduction in this read-only review.

## Solution Assessment

No for closing: current parser and normalizer address much of the raw `MEDIA:` leakage, but they do not provide the requested sanitized, actionable allowed-roots rejection. The best fix needs a maintainer-approved shared failure contract before implementation.
