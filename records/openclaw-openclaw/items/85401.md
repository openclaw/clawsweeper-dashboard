## Summary

Keep open: current main and v2026.6.8 handle the successful final-reply media path better than the original report, but the remaining blocked absolute-host-path case still collapses to generic WhatsApp-visible failure text instead of a sanitized actionable local-media policy error.

## What This Changes

The issue requests that WhatsApp final assistant `MEDIA:` PDF/document directives be parsed and stripped from visible text, then either delivered as attachments or replaced with a sanitized actionable error.

## Best Possible Solution

Keep the issue open for maintainer/security review, then add a sanitized local-media rejection taxonomy that preserves successful workspace/relative media delivery and never exposes raw host paths in chat.

## Reproduction Assessment

Yes, at source level. Current main has tests and code paths showing blocked absolute host-local reply media is dropped and converted to a generic warning; I did not run a live WhatsApp group reproduction in this read-only review.

## Solution Assessment

No, current main is only a partial mitigation. The maintainable fix is to map `LocalMediaAccessError` reasons into approved sanitized user-visible text while keeping raw paths out of WhatsApp-visible output.
