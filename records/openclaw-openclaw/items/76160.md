## Summary

Keep open. Current main still has no duplicate-name guard in the shared cron service, and an open closing PR (#76180) is already paired with the issue, so cleanup should wait for that PR to land or be closed.

## What This Changes

The issue asks for cron add/edit to prevent duplicate cron job names, provide an explicit replace path, and expose the behavior consistently through CLI, Gateway, and agent-tool surfaces.

## Best Possible Solution

Enforce normalized per-agent cron name uniqueness in the shared cron service path, include disabled jobs in the collision check, and land the behavior through the existing Gateway/CLI/agent-tool contract with tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: two `cron.add` calls with the same name take the unconditional append path, and `cron.update` can patch a name before replacing only by ID.

## Solution Assessment

Yes for the service-layer uniqueness direction; unclear whether the optional `--replace` flow belongs in the first fix because the already-open PR mainly covers rejection.
