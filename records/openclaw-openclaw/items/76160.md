## Summary

Keep open. Current main still creates and updates cron jobs without any name-collision check, and the default list path still filters disabled rows, so the reported duplicate-name risk remains valid and narrow enough for a focused fix.

## What This Changes

The issue requests cron add/edit to treat job names as unique, reject collisions by default, support an explicit replace path, and make legacy duplicate-name cases discoverable.

## Best Possible Solution

Enforce normalized per-agent cron name uniqueness in the cron service add/update path, add an explicit replace contract for create, and cover CLI, Gateway RPC, and agent-tool behavior through that shared seam.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: two `cron.add` calls with the same name take the unconditional `state.store.jobs.push(job)` path, and `cron.update` can patch `name` without checking sibling jobs.

## Solution Assessment

Yes. A Gateway/service-layer fix is the narrowest maintainable solution because CLI, agent tool, and future callers already converge on `cron.add` and `cron.update`; CLI-only validation would leave other surfaces broken.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76160 on current main. Implement duplicate cron job name protection at the shared cron service/Gateway boundary, not only in CLI parsing: `cron.add` should reject a normalized name collision within the same effective agent job set by default, including disabled rows; `cron.update` should reject renaming a job onto another job's normalized name while excluding the job being edited; the agent-facing cron tool should receive the same structured invalid-request error through the Gateway path. Add an explicit `--replace` / `replace` create path that overwrites the existing row in place while preserving its `id` and replacing the other create fields atomically. Reuse existing normalization patterns such as `normalizeLowercaseStringOrEmpty` where user-facing name lookup is already case-insensitive. Do not broaden cron remove semantics, do not change custom-ID behavior from adjacent PR #75065 unless it has already landed, and avoid startup/load-time migrations for legacy duplicates. Add focused tests for add collision refusal, add replace preserving ID, update-name collision refusal, disabled rows participating in uniqueness, and agent-tool/Gateway error shape. Add a single-line `CHANGELOG.md` fix entry for the user-facing cron behavior.
