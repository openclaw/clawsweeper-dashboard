## Summary

Keep open. Current main still allows duplicate cron job names in the shared add/update service path, and an open PR explicitly references this issue with closing syntax, so the issue should remain open until that PR lands or is closed.

## What This Changes

The issue asks for cron add/edit to reject duplicate job-name collisions by default, with optional replacement behavior and clearer duplicate visibility.

## Best Possible Solution

Review and land or close #76445; the end state should enforce normalized cron name uniqueness in the shared cron service add/update path with regression coverage for disabled existing jobs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: two add calls with the same normalized name both reach the unconditional store append, and update can patch name before replacing by id without a sibling check.

## Solution Assessment

Yes for a shared service-layer uniqueness check. CLI-only validation would miss Gateway and agent-tool callers; --replace and duplicate-list warnings should be explicit API/CLI/tool follow-up choices if included.
