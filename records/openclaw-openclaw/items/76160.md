## Summary

Keep open. Current main still allows duplicate cron names through the shared service add/update paths, and the linked closing PR #76180 is open rather than merged, so this issue should stay open until that fix is reviewed and landed or replaced.

## What This Changes

The issue requests cron job name uniqueness for add/edit, with an optional replace path and duplicate visibility so duplicate-named jobs cannot silently persist.

## Best Possible Solution

Land a service-layer uniqueness check shared by CLI, Gateway RPC, and the agent cron tool, then decide whether `--replace` and duplicate-list warnings belong in the same PR or a follow-up.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: two `cron.add` calls with the same normalized name take the unconditional push path, and `cron.update` can patch a job name without checking other jobs.

## Solution Assessment

Yes for the core direction: the shared cron service/Gateway path is the narrowest maintainable place to enforce uniqueness. The exact `--replace` UX still needs maintainer confirmation because the open PR context does not show that part implemented.
