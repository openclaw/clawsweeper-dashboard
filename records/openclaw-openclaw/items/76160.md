## Summary

Keep open. Current main still accepts duplicate cron job names through the shared service add/update paths, default listing still filters disabled rows, and the linked fix PR was closed unmerged.

## What This Changes

Request to make cron job names unique on add/edit, with collision errors instead of silently creating ambiguous duplicate rows.

## Best Possible Solution

Enforce normalized cron name uniqueness in the shared cron service add/update path, propagate a clear Gateway error to CLI and agent-tool callers, and add regression coverage including disabled existing jobs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: two `cron.add` calls with the same normalized name both reach `jobs.push`, and `cron.update` can patch `name` before replacing by id with no sibling check.

## Solution Assessment

Yes for a shared service-layer validation. CLI-only validation would miss Gateway and agent-tool callers; replacement/list-warning UX should be explicit follow-up if maintainers want it.

Add a helper in the cron service layer to find an existing job by normalized name, make `add` reject any collision including disabled jobs, and make `update` reject renaming onto a different existing job while allowing unchanged same-job names.
