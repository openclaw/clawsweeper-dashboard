## Summary

Keep open. Current main still permits duplicate cron names in the shared add/update service paths, and open PR #76445 uses closing syntax for this issue, so the issue should remain open until that PR or another fix lands.

## What This Changes

The issue requests duplicate cron job names be rejected on add/edit, with optional replacement and legacy duplicate visibility UX.

## Best Possible Solution

Land a reviewed service-layer uniqueness fix that covers Gateway, CLI, and agent tool callers, then decide separately whether `--replace` and duplicate-list warnings belong in the same public contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: repeated adds with the same normalized name both reach `jobs.push`, and update can patch `name` before replacing by id without checking sibling jobs.

## Solution Assessment

Yes for a shared cron service validation fix. The broader replacement/list-warning UX is useful but should be an explicit maintainer product decision rather than an implied side effect of the bug fix.
