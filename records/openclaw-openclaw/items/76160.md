## Summary

Keep open. Current main still allows duplicate cron names through the shared add/update service paths, and open PR #76180 is already paired as the implementation path, so this issue should remain open until that PR lands or is closed.

## What This Changes

The issue requests per-agent cron job name uniqueness for add/edit, an optional replace path, and consistent CLI, Gateway, and agent-tool errors.

## Best Possible Solution

Review and land #76180 or an equivalent service-layer fix that enforces normalized name uniqueness across enabled and disabled jobs, then decide whether `--replace` belongs in the same closeout.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: two `cron.add` calls with the same normalized name take the unconditional append path, and `cron.update` can patch a name before replacing by ID only.

## Solution Assessment

Yes. A shared cron service add/update check is the narrowest maintainable fix because CLI, Gateway RPC, and the agent cron tool already converge on that service boundary.
