## Summary

Keep open. Current main still accepts duplicate cron job names through the shared add/update service paths, and the only linked fix PR was closed unmerged, so the reported collision risk remains live.

## What This Changes

Request validation that prevents or explicitly replaces duplicate cron job names across cron add/edit surfaces.

## Best Possible Solution

Enforce normalized cron name uniqueness in the shared cron service add/update path, expose any accepted replacement behavior consistently through Gateway, CLI, and the agent tool, and add regression coverage for disabled existing jobs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: two add calls with the same normalized name both reach the unconditional `jobs.push`, and update can patch `name` before replacing by id with no sibling check.

## Solution Assessment

Yes for a shared service-layer fix. CLI-only validation would miss Gateway and agent-tool callers; `--replace` should be wired as an explicit API/CLI/tool contract if included.
