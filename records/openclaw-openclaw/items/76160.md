## Summary

Keep this issue open because current main still accepts duplicate cron names and an open linked implementation PR is already the right path to review before closing.

## What This Changes

The issue asks cron add/edit to enforce unique job names, reject duplicate-name collisions by default, and optionally support replacing an existing job in place.

## Best Possible Solution

Review and land or revise the linked service-layer uniqueness fix, with a maintainer decision on whether --replace belongs in the same change or a follow-up.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main appends new cron jobs without checking existing names, allows name patches during update, and hides disabled rows from default listing unless --all/includeDisabled is used.

## Solution Assessment

Yes for the service-layer validation boundary, because CLI and agent tool calls both route through Gateway cron.add/cron.update into CronService. The --replace mode is a separate UX/product decision that should be resolved in the linked PR or a follow-up.
