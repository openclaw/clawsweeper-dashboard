## Summary

Keep this PR open: the Feishu send retry direction is useful and source-backed, but the latest branch is not merge-ready because the retry loop still misses fulfilled Feishu body-code rate-limit responses and the expanded 11232/HTTP 429 behavior lacks matching tests and refreshed real-environment proof.

## What This Changes

The PR adds bounded retry/backoff for selected Feishu message-send rate-limit responses and routes direct, reply, and streaming-card sends through that retry helper with new send retry/concurrency tests.

## Best Possible Solution

Keep the PR open until the helper retries both thrown and fulfilled retryable Feishu send responses, tests cover the expanded 230020/11232/HTTP 429 signals, and the author or a maintainer supplies or explicitly accepts real proof for the expanded live behavior.

## Reproduction Assessment

Yes, from source inspection: current main sends direct messages through a no-retry wrapper and replies through a raw SDK call, so a Feishu rate-limit rejection fails immediately. The contributor also supplied live 230020 stress output, but I did not run a live Feishu reproduction in this read-only review.

## Solution Assessment

No, not yet: bounded retry in the Feishu send helper is the right direction, but the current implementation only retries rejected Axios shapes and does not inspect fulfilled body-code failures before returning. The safer fix is to classify retryable response bodies before success assertion, then test and prove the expanded signals.
