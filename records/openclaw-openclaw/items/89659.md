## Summary

Keep open: current main still sends Feishu messages without retrying 230020 rate-limit responses, while this PR adds useful bounded retry behavior with live direct-send and reply-send proof. The remaining gate is maintainer or Feishu-owner acceptance of the outbound delivery semantics, especially whether retrying non-idempotent create/reply calls is acceptable without Feishu `uuid` dedupe.

## What This Changes

The PR adds 230020-only bounded retry to Feishu API send calls, routes text/card/streaming direct and reply sends through it, and adds retry/concurrency tests plus live proof.

## Best Possible Solution

Land a Feishu-owned retry fix that keeps the 230020-only bounded retry, either with explicit owner signoff on latency/idempotency or with `uuid` dedupe reused across each retried message create/reply operation.

## Reproduction Assessment

Yes. Current main clearly has one-shot Feishu send paths, and the PR provides live Feishu stress output showing 230020 burst failures recovered after retry; I did not rerun the live Feishu scenario in this read-only review.

## Solution Assessment

Acceptable but not conclusively best without owner signoff. The narrow retry helper fixes the observed 230020 loss mode, but Feishu's documented `uuid` dedupe is the safer permanent shape for retried POST sends.
