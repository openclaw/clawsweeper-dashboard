## Summary

This PR has useful direct-send retry proof, but it should not merge as the closing fix yet because Feishu text/card replies and streaming-card sends still bypass the new retry wrapper, and 230006 is retried by default despite official docs treating it as a bot-ability configuration error.

## What This Changes

The PR adds bounded retry/backoff for Feishu `requestFeishuApi` calls on 230020/230006 and adds direct-send retry/concurrency tests.

## Best Possible Solution

Complete one Feishu message-send retry boundary for direct sends, replies, media, and streaming-card sends, and keep 230006 out of the default retry set unless maintainers accept it with live proof and contract tests.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: current main wraps direct sends without retry and Feishu docs define 230020 as a send/reply frequency-limit error; the contributor's live test shows direct `message.create` initially hit 230020 and then succeeded after retries. I did not rerun live Feishu credentials from this read-only review.

## Solution Assessment

No, this is a useful mitigation but not the best complete fix yet. The maintainable fix is to share one retry boundary across all Feishu message create/reply send paths and keep retryable codes aligned with Feishu's documented contracts or live proof.
