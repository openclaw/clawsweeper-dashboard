## Summary

Keep open: the Feishu retry direction is useful and still needed on current main, but the PR retries visible send operations without stable `uuid` idempotency and lacks after-fix proof for the newly changed streaming-card path.

## What This Changes

The PR adds bounded Feishu `230020` send retries in the shared API helper, routes reply and streaming-card sends through it, and adds direct-send retry tests plus live direct/reply proof.

## Best Possible Solution

Land Feishu send retries only after every retried visible send reuses a stable per-send UUID across attempts, with focused tests and redacted live proof for the full intended scope or a narrower retry scope.

## Reproduction Assessment

Yes. Source inspection shows the PR retries the same visible Feishu send closures while the `message.create` and `message.reply` payloads omit Feishu's SDK-supported `uuid` idempotency field.

## Solution Assessment

No. Retrying `230020` is the right repair direction, but the best fix must make retried visible sends idempotent or narrow the retry to paths with proven pre-send failure semantics.
