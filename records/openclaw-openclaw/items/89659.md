## Summary

Keep open: this PR is a live implementation candidate for Feishu 230020 send-rate-limit recovery and is not superseded, with strong after-fix proof. The remaining blocker is maintainer acceptance of the Feishu outbound retry semantics, especially repeated create/reply calls without Feishu `uuid` idempotency.

## What This Changes

The PR adds bounded 230020 retry handling to Feishu `requestFeishuApi`, routes text/card/streaming direct and reply sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Land a Feishu-owned retry fix that keeps 230020-only bounded retries, with either explicit owner acceptance of the latency/idempotency tradeoff or `uuid` dedupe reused across each retried create/reply operation.

## Reproduction Assessment

Yes, by source and supplied live proof. Current main and v2026.6.1 have one-shot Feishu send paths, and the PR body supplies after-fix live Feishu stress output showing 230020 burst failures recovered by retry; I did not rerun the live Feishu scenario in this read-only review.

## Solution Assessment

Acceptable but not conclusively best without Feishu-owner signoff. The narrow helper fixes the observed 230020 loss mode, but Feishu `uuid` idempotency is the safer permanent shape for retried POST sends if maintainers want to avoid duplicate-delivery ambiguity.
