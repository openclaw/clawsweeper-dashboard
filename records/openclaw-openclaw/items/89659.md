## Summary

Keep open for maintainer review: the patch looks directionally correct and has live Feishu proof, but helper-level retry changes several Feishu send surfaces and remains a bounded delivery mitigation, so a Feishu owner should accept that behavior before merge.

## What This Changes

This PR adds bounded retry/backoff for Feishu 230020 message send rate-limit errors in requestFeishuApi, routes reply and streaming-card sends through that helper, and adds retry/concurrency tests.

## Best Possible Solution

Land the bounded 230020 retry after Feishu owner acceptance of the shared-helper scope, then track durable queueing or backpressure separately if maintainers need stronger burst guarantees.

## Reproduction Assessment

Yes. Current main has no retry in requestFeishuApi, and the PR body supplies live Feishu direct/reply stress output showing 230020 failures recover after the patch; I did not run live Feishu credentials in this read-only review.

## Solution Assessment

Acceptable mitigation, but not a complete delivery guarantee. The helper-level retry is a maintainable place for SDK-thrown Feishu send rate limits, but maintainer acceptance is needed because the scope is broader than the exact proof and the retry remains bounded.
