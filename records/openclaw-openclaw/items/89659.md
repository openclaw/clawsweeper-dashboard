## Summary

Keep open: the PR has good live proof for direct Feishu `message.create` retries, but it still leaves visible reply and streaming-card sends outside the retry path and retries `230006`, which the current Feishu/Lark docs describe as a bot-ability setup error rather than a transient frequency limit.

The PR adds bounded retry/backoff handling for selected Feishu send error codes in `requestFeishuApi` and adds retry/concurrency tests for direct message sends.

## Best Possible Solution

Retry only documented transient Feishu message frequency-limit failures, especially `230020`, across direct, reply, media, and streaming-card message create/reply paths while surfacing setup/configuration errors immediately.
