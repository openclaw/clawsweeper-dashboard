## Summary

Keep open: the direct-send retry is useful and has live proof for Feishu `230020`, but the branch still leaves text/card replies and streaming-card sends outside the retry path and retries `230006` even though Feishu documents it as a bot-ability setup error.

## What This Changes

The PR adds bounded retry/backoff in Feishu `requestFeishuApi` for `230020` and `230006` and adds direct-send retry/concurrency tests.

## Best Possible Solution

Land a Feishu transport retry that covers every visible `message.create` and `message.reply` send path for documented transient rate limits, preserves reply fallback semantics, and does not retry `230006` without official contract proof.

## Reproduction Assessment

Yes at source level: current main has single-attempt Feishu send/reply paths, and a mocked `230020` response follows a no-retry failure path for direct callers on main and for reply/streaming paths after this PR. I did not run tests because this review is read-only.

## Solution Assessment

No. The shared helper is the right repair layer for Feishu API sends, but this PR needs to route all visible create/reply send paths through it and remove or formally justify `230006` before it is the best fix.
