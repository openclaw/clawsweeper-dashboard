## Summary

## What This Changes

The branch adds a Feishu `requestFeishuApi` retry loop for thrown 230020 send rate-limit errors, routes direct/reply/streaming-card sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Merge the bounded 230020 retry if Feishu owners accept the delivery tradeoff; use a separate follow-up for per-chat scheduling or `uuid` idempotency hardening if they want stricter duplicate protection.

## Reproduction Assessment

Yes. Current main shows Feishu send paths fail after a single thrown provider error, Feishu docs define 230020 as a retryable frequency-limit condition, and the PR body includes live direct/reply stress commands that exercised the changed path after the patch.

## Solution Assessment

Yes for the reported failure mode. A per-chat scheduler would be a broader throughput feature, but a shared bounded retry in the Feishu message API wrapper is the narrow maintainable fix for transient 230020 send failures.
