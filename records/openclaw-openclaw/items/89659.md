## Summary

Keep this PR open, but not merge-ready: the retry direction is useful for the linked Feishu message-loss issue, yet the patch replays visible Feishu sends without stable `uuid` de-duplication keys, so it can trade lost messages for duplicate messages.

## What This Changes

## Best Possible Solution

Keep the shared retry boundary, but generate one stable Feishu uuid per logical visible send, reuse it for all retry attempts across text, card, streaming-card, and media create/reply paths, then validate with focused tests and redacted live proof.

The duplicate-send risk is source-reproducible from replaying send closures without uuid.

## Solution Assessment

No, not as submitted. The shared retry helper is the right owner boundary, but retrying visible Feishu sends should be paired with per-logical-send uuid idempotency and refreshed redacted live proof before merge.
