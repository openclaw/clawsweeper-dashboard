## Summary

Keep open. Current main still sends Feishu messages without retrying 230020, while this PR adds a narrow bounded retry with sufficient live Feishu proof; the remaining blocker is maintainer acceptance of the outbound message-delivery tradeoff, not an automated code repair.

## What This Changes

The PR adds Feishu 230020 send retry/backoff in the shared API wrapper, routes reply and streaming-card sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Land a bounded Feishu-owner-approved 230020 retry after exact-head CI, keeping the retry scoped and documented as a delivery-reliability tradeoff rather than a general queue.

## Reproduction Assessment

Yes, by source and external contract: current main has no retry in requestFeishuApi/reply sends, and Feishu docs identify 230020 as a frequency-limit error that should be retried after reducing frequency. I did not run live Feishu credentials myself, but the PR supplies live after-fix output.

## Solution Assessment

Yes, this is an acceptable narrow fix shape: the shared Feishu API wrapper is already used by direct/media sends and can own the SDK-thrown 230020 policy without duplicating retry logic at every caller. The maintainer decision is whether this bounded retry is the desired delivery behavior.
