## Summary

Keep open for maintainer/Feishu-owner review: the patch appears to address the Feishu 230020 send-rate-limit path with strong live proof, but it intentionally changes outbound delivery timing and should be accepted as a channel policy before merge.

## What This Changes

The PR adds bounded retry/backoff for Feishu 230020 send rate-limit errors in the shared API wrapper, routes reply and streaming-card sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Land the bounded 230020 retry only after Feishu-owner acceptance and exact-head CI, keeping it documented as a delivery-reliability mitigation rather than a durable queue or idempotency layer.

## Reproduction Assessment

Yes, by source and external contract: current main does not retry thrown 230020 send failures, and Feishu documents 230020 as a frequency-limit error that should be retried after lowering request frequency. I did not run live Feishu credentials in this read-only review, but the PR body supplies live after-fix direct and reply stress output.

## Solution Assessment

Yes, this is an acceptable narrow fix shape if maintainers accept the delivery tradeoff: the shared Feishu API wrapper already owns direct/media request failures, and the PR brings reply and streaming-card sends onto the same path. If maintainers need guaranteed delivery or idempotency, that should be a separate queue-level design rather than stretching this helper patch.
