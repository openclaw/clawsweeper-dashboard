## Summary

Keep open: the retry is a useful fix for the Feishu rate-limit message-loss bug, and the supplied live proof is strong, but the patch installs send-specific retry behavior as the default for the shared Feishu API wrapper that existing upload/media paths also use.

## What This Changes

The PR adds bounded 230020 retry/backoff to Feishu API requests, routes text reply and streaming-card sends through the wrapper, and adds retry/concurrency tests.

## Best Possible Solution

Keep the bounded 230020 retry, but make it an explicit Feishu message-send retry mode/helper and cover direct, reply, streaming-card, media send, and upload no-retry behavior.

## Reproduction Assessment

Yes at source/proof level: current main calls Feishu sends once through requestFeishuApi or directly, and the PR supplies live Feishu logs showing 230020 under concurrent direct/reply send load. I did not rerun live Feishu credentials in this read-only review.

## Solution Assessment

No as currently shaped: retrying 230020 for Feishu message sends is the right direction, but making it the default in the shared wrapper is broader than the observed send failure. The narrower maintainable fix is an opt-in send retry helper or option.
