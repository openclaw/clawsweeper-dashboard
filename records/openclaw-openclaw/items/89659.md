## Summary

## Best Possible Solution

Keep the shared retry approach only after each visible Feishu send operation generates one stable `uuid` and reuses it for every retry across text, media, reply, and streaming-card paths, with focused tests plus refreshed redacted live direct/reply proof.

## Solution Assessment

No: bounded retry in the Feishu plugin is the right layer, but the current implementation is not the best fix until visible writes reuse a stable `uuid` across retries to avoid duplicate delivery.
