## Summary

This PR should stay open, but it is not merge-ready because the new Feishu retry loop can reissue message create/reply requests without a stable Feishu `uuid`, which leaves duplicate delivery possible after an ambiguous send outcome.

## What This Changes

The branch adds bounded retry/backoff for Feishu `230020` send rate-limit errors, routes text/reply/streaming-card sends through the shared helper, and adds retry/concurrency tests.

## Best Possible Solution

Keep the Feishu-plugin retry boundary, but add stable per-logical-send Feishu `uuid` dedupe across retried create/reply paths and make the bounded retry budget an explicit maintainer-owned tradeoff.
