## Summary

Keep this PR open: it is a useful Feishu delivery fix with strong live proof, but it is not merge-ready because the new retry loop replays Feishu message POSTs without the platform's native `uuid` dedupe key, which can duplicate delivered messages after ambiguous failures.

## What This Changes

The PR changes Feishu `requestFeishuApi` to retry code 230020 with two linear-backoff attempts, routes reply and streaming-card sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Keep the 230020 retry direction, but make every retried Feishu create/reply message send use one stable native `uuid` per logical send and get maintainer acceptance for the retry budget.

## Reproduction Assessment

Yes for the current-main failure by source and contributor live proof: current main wraps Feishu send errors but does not retry 230020, and the PR proof shows live 230020 bursts recover after retries. I did not rerun live Feishu tests in this read-only review.

## Solution Assessment

No as submitted. Helper-level retry is the right general direction, but Feishu send/reply retries need stable `uuid` dedupe on every retried message POST before this is the best safe fix.
