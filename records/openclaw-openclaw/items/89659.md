## Summary

Keep open: the reliability fix is useful, but this PR retries visible Feishu sends without stable `uuid` idempotency and expands retry behavior to streaming-card sends without matching proof/tests.

## What This Changes

The PR adds bounded retry handling for Feishu send rate-limit error `230020` and routes direct, reply, and streaming-card message sends through the shared Feishu API wrapper.

## Best Possible Solution

Land Feishu retry only after every retried visible `im.message.create`/`reply` send reuses a stable UUID, with focused tests and redacted live proof for each changed visible path or a narrower retry scope.

## Reproduction Assessment

Yes. By source inspection, the PR retries the same Feishu send closures up to two times while the affected `im.message.create` and `im.message.reply` payloads omit the SDK-supported `uuid` field.

## Solution Assessment

No. Retrying observed `230020` rate-limit failures is the right direction, but the best fix must make retried visible sends idempotent or narrow the retry scope to paths with proven pre-send failure semantics.
