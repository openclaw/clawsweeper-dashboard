## Best Possible Solution

Keep the central bounded retry helper, but make it retry both thrown HTTP/Axios rate limits and resolved Feishu nonzero rate-limit responses across direct, reply, and streaming-card sends, with focused tests and refreshed live proof.

## Reproduction Assessment

## Solution Assessment

No. The central Feishu helper is the right layer, but the best fix must classify retryable result objects before returning and cover the documented send rate-limit codes or explain exclusions with tests.

- **[P1] Retry resolved Feishu rate-limit responses:** `extensions/feishu/src/comment-shared.ts:140`
  - body: `requestFeishuApi` only retries errors that are thrown. The Feishu SDK returns successful HTTP bodies as `{ code, msg, data }`, and the message create/reply methods are typed that way, so a retryable body like `{ code: 11232 }` or `{ code: 230020 }` can return here and then fail in `assertFeishuMessageApiSuccess`/`sendRes.code !== 0` without retry. Move the retryable-code check to cover resolved responses too.
  - confidence: 0.92
- **[P1] Cover the documented send rate-limit codes:** `extensions/feishu/src/comment-shared.ts:95`

([open.feishu.cn](https://open.feishu.cn/document/server-docs/api-call-guide/generic-error-code?lang=zh-CN&utm_source=openai)) Since this PR closes the broader Feishu message-loss report, either handle those codes or explicitly exclude them with endpoint-contract rationale and focused tests.
  - confidence: 0.83
