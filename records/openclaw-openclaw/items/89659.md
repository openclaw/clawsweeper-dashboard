## Summary

Keep open: the retry direction is useful for Feishu rate-limit delivery, but the patch currently replays non-idempotent Feishu `message.create` and `message.reply` calls. Feishu provides a native `uuid` dedupe field for these APIs, and the PR needs to reuse a stable value per logical send before merge.

## What This Changes

The PR adds bounded 500ms/1000ms retry handling for Feishu `230020` send rate-limit errors and routes direct, reply, and streaming-card send paths through the shared retry helper.

## Best Possible Solution

Add stable Feishu `uuid` reuse per logical outgoing create/reply send, cover every retried Feishu send surface with focused tests, then refresh redacted live proof showing retry success without duplicate delivery.

## Reproduction Assessment

Yes from source inspection and the PR's live Feishu proof: current `main` fails immediately on retryable Feishu send rate-limit errors, while the PR replays the request closure. The duplicate-delivery blocker is also source-reproducible because retried create/reply payloads omit Feishu's documented `uuid` dedupe field.

## Solution Assessment

No. The shared retry helper is the right general owner boundary, but the best fix must make each retried Feishu send idempotent with native `uuid` reuse and then prove the live channel does not duplicate messages.
