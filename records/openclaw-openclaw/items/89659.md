## Summary

This PR should stay open because the Feishu retry direction is useful and backed by live proof, but the current patch is not merge-ready: helper-level retry reaches multiple non-idempotent message create/reply paths, and the streaming-card payload change is not aligned with existing sibling tests.

## What This Changes

The PR adds bounded Feishu 230020 retry/backoff in requestFeishuApi, routes reply and streaming-card sends through the helper, and adds retry/concurrency tests for text sends.

## Best Possible Solution

Keep the bounded retry idea, but make retried Feishu sends idempotent across text/card/media/streaming-card payloads and either remove or fully validate the unrelated streaming-card append-content change.

## Reproduction Assessment

Yes for the central bug from source and supplied proof: current main wraps Feishu API errors without retry, and the PR includes live Feishu stress output showing retries recover 230020 sends. I did not run a live Feishu reproduction during this read-only review.

## Solution Assessment

No: bounded retry is the right direction, but the current patch is not the best merge shape until retries are idempotent and the streaming-card payload change is either removed or tested against the intended CardKit contract.
