## Summary

Keep this PR open: the patch addresses a real Feishu message-delivery bug with the right shared send wrapper and strong supplied live proof, and I found no blocking code-review finding. ## What This Changes

The PR adds bounded retry handling around Feishu message send APIs for transient `230020` rate-limit responses, covers direct/reply/streaming-card sends, and adds retry/concurrency tests.

## Best Possible Solution

Land the shared-wrapper retry if maintainers accept the bounded retry policy, and track broader Feishu pacing or other error-code families separately instead of expanding this PR without proof.

## Reproduction Assessment

Yes for source-level reproduction: current main has message send paths that either lack retry in `requestFeishuApi` or bypass it, and Feishu docs identify `230020` as a retry-after-lowering-frequency error for create/reply. I did not rerun live Feishu credentials in this read-only review.

## Solution Assessment

Yes, this is the best fix shape if maintainers accept the bounded retry budget: putting retry in `requestFeishuApi` covers direct, reply, streaming-card, and existing media message sends without duplicating channel-specific send logic. The safer alternative is only to tune the budget/jitter before merge, not to move the retry to a narrower call site.
