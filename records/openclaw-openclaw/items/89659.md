## Summary

This PR addresses a real Feishu rate-limit message-loss path, and current main/latest release still send once, but the patch is not merge-ready because it retries visible Feishu create/reply writes without a stable dedupe `uuid`, which can turn an ambiguous successful write into duplicate delivered messages.

## What This Changes

The PR adds bounded retry/backoff for Feishu `230020` send rate-limit errors in shared Feishu API handling, routes direct/reply/streaming-card sends through that retry path, and adds retry/concurrency tests.

## Best Possible Solution

Keep the bounded retry approach, but generate and reuse one stable Feishu `uuid` per logical visible create/reply across text, media, and streaming-card starts, cover it in tests, then rerun redacted live Feishu proof.

## Reproduction Assessment

Yes, source-reproducible: current main and `v2026.6.1` call Feishu sends once, while Feishu docs define `230020` as a frequency-limit condition to retry. This read-only review did not run live Feishu.

## Solution Assessment

No: shared bounded retry is the right layer, but the proposed fix is incomplete until retried visible sends are idempotent with stable `uuid` values.
