## Summary

Keep open: the retry target is a real Feishu delivery problem, but this branch is not merge-ready because the retry loop lacks Feishu `uuid` idempotency and the branch also changes streaming-card update semantics from cumulative content to append-only content.

## What This Changes

The branch adds bounded retry for Feishu `230020` send/reply errors in the shared API helper, wraps direct/reply/streaming-card sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Keep the retry idea, but make each logical visible Feishu send/reply idempotent with a stable `uuid` reused across attempts, remove or separately prove the streaming-card append change, then rerun redacted live direct and reply proof.

## Reproduction Assessment

The blocking review findings are also source-reproducible from the PR diff and current streaming-card tests.

## Solution Assessment

No: helper-level retry is a plausible mitigation, but the best fix must make visible sends idempotent with Feishu `uuid` and keep unrelated streaming-card behavior out unless it has separate proof.
