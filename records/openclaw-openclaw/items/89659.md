## Summary

Keep open for maintainer review: current main still sends Feishu 230020 failures through a no-retry helper, while this PR adds a bounded retry path with credible live Feishu proof. The remaining question is whether Feishu owners accept the outbound latency/idempotency tradeoff before merge.

## What This Changes

The branch adds 230020-only bounded retry to Feishu `requestFeishuApi`, routes reply and streaming-card sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Merge after a Feishu owner accepts the bounded retry/idempotency tradeoff, or require stable per-send `uuid` idempotency on retried create/reply calls before merge.

## Reproduction Assessment

Yes, we have a high-confidence source/proof path: current main has no retry in `requestFeishuApi`, official Feishu/Lark docs identify 230020 as a send/reply frequency-limit error, and the PR body includes live stress output that hit 230020 and recovered after retry. I did not run a live Feishu test in this read-only review.

## Solution Assessment

Acceptable mitigation, not fully owner-settled as the best final policy: centralizing 230020 retry in the Feishu API helper covers the right send paths, but Feishu's documented `uuid` idempotency support means maintainers should decide whether retry-only is enough before merge.
