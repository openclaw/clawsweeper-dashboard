## Summary

Keep this PR open: the retry direction is useful for the linked Feishu message-loss bug, but the current patch retries visible Feishu message POSTs without stable native `uuid` values, so an ambiguous first attempt can be replayed as a duplicate message.

## What This Changes

The PR adds bounded retry/backoff for Feishu `230020` send rate-limit responses across direct, reply, and streaming-card send paths, with new retry/concurrency tests.

## Best Possible Solution

Keep the bounded `230020` retry behavior, but make every retryable Feishu message create/reply idempotent with one stable native `uuid` per logical send before merge.

## Reproduction Assessment

Yes, at source/proof level: the linked issue and PR body show live Feishu rate-limit failures, and current main has no retry loop in the shared send wrapper. I did not rerun a live Feishu environment in this read-only review.

## Solution Assessment

No. Bounded retry in the Feishu plugin is the right layer, but the PR is not the best safe fix until retries are paired with stable Feishu `uuid` values reused across attempts.
