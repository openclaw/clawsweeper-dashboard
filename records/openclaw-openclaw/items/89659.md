## Summary

Keep open. The retry direction addresses a real Feishu rate-limit message-loss path, but the patch reissues message POSTs without Feishu `uuid` dedupe, so it can trade dropped messages for duplicate messages under ambiguous retry outcomes.

## What This Changes

The branch adds retry/backoff for Feishu `230020` send rate-limit errors in shared Feishu API calls, routes reply and streaming-card sends through that helper, and adds retry/concurrency tests.

## Best Possible Solution

Keep the Feishu-extension retry approach, but add per-logical-send stable `uuid` dedupe for every retried create/reply path and leave the bounded retry budget as an explicit maintainer-owned tradeoff.

## Reproduction Assessment

## Solution Assessment

No, not as-is: retrying in the Feishu plugin is the right boundary, but the best fix must make retried create/reply POSTs idempotent with stable `uuid` values before merge.
