## Summary

Keep open for maintainer review. The patch appears technically sound and has strong live Feishu proof, but it intentionally changes Feishu delivery semantics by adding bounded shared-helper retry, so a Feishu owner should accept the retry budget and residual burst behavior before merge.

## What This Changes

The PR adds bounded retry/backoff for Feishu 230020 send rate-limit errors, routes reply and streaming-card sends through the retry helper, and adds retry/concurrency tests.

## Best Possible Solution

Land the bounded 230020 retry only after a Feishu owner accepts the shared-helper scope and residual burst behavior; track durable per-chat throttling or queueing separately if maintainers require stronger delivery guarantees.

## Reproduction Assessment

Yes, at source level: current main wraps the first thrown Feishu send error without retry, while reply and streaming-card sends bypass that wrapper. Live current-main reproduction would require Feishu credentials, but the PR supplies live after-fix Feishu proof.

## Solution Assessment

Yes as an acceptable bounded mitigation, not as a complete durable queue. The cleaner final state is a Feishu-owner-approved retry budget now, with separate throttling or queueing work only if maintainers want stronger burst guarantees.
