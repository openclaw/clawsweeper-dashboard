## Summary

Keep open for maintainer review: the patch is a meaningful Feishu delivery reliability fix with sufficient live proof, but the retry policy now sits in the shared Feishu API helper and should be explicitly accepted or narrowed before merge.

## What This Changes

The PR adds bounded retry/backoff for Feishu send rate-limit code 230020 in `requestFeishuApi`, routes reply and streaming-card sends through that helper, and adds retry/concurrency tests.

## Best Possible Solution

Land a Feishu-owned bounded retry after maintainers accept the exact retryable code set and helper scope; otherwise narrow the retry wrapper to explicit message delivery sends before merge.

## Reproduction Assessment

Yes, at source level: current main performs one Feishu message send attempt and a thrown AxiosError with body code 230020 would be wrapped as a terminal send failure. I did not run a live Feishu rate-limit reproduction in this read-only review.

## Solution Assessment

Yes, with maintainer acceptance of scope: a bounded Feishu-owned retry around message delivery is the right layer and avoids a core retry/config surface. The only remaining solution-fit question is whether the shared helper should own that retry policy or a narrower send-only wrapper should.
