## Summary

This PR is still necessary and should stay open for maintainer review: current main lacks retry on Feishu reply and streaming-card sends, while the branch adds bounded `230020` retry coverage with supplied live Feishu proof. The remaining blocker is not a cleanup close reason; it is Feishu owner acceptance of the shared retry scope, fixed retry budget, and no-idempotency-key tradeoff before merge.

## What This Changes

The PR adds bounded retry/backoff for Feishu `230020` send rate-limit responses in the shared Feishu request helper, routes reply and streaming-card sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Land this only after a Feishu owner accepts the shared retry scope and duplicate-send tradeoff, or asks for a narrower/idempotent retry design with refreshed direct and reply proof.

## Reproduction Assessment

Yes. Current main has clear no-retry source paths for Feishu reply and streaming-card sends, and the PR supplies live Feishu direct/reply stress output showing `230020` recovery after the patch; I did not run a live Feishu account in this read-only review.

## Solution Assessment

Unclear as the best final policy until a Feishu owner signs off. Bounded retry is a plausible repair for transient send rate limits, but the broader helper scope and missing send idempotency keys make owner acceptance or a narrower/idempotent variant the safer merge path.
