## Summary

Keep this PR open: the Feishu rate-limit retry fix is addressing a real message-loss path, but the current retry loop can resend non-idempotent message POSTs without Feishu `uuid` de-duplication, so it is not merge-ready.

## What This Changes

The PR adds bounded retry/backoff for Feishu `230020` send rate-limit errors in the shared Feishu API helper and routes direct, reply, and streaming-card send paths through it with new retry/concurrency tests.

## Best Possible Solution

Land bounded Feishu send retries only after each logical `message.create` and `message.reply` retry reuses a stable Feishu `uuid`, with focused tests and redacted live proof for direct, reply, and streaming-card sends.

## Reproduction Assessment

Yes from source, though I did not run a live current-main Feishu stress test in this read-only review. Current main sends direct/reply messages once through `requestFeishuApi`, so an Axios-shaped `230020` response throws without retry.

## Solution Assessment

No: bounded retry is the right direction, but this implementation should be made idempotent before merge. The safer solution is to reuse a stable Feishu `uuid` for every retryable logical send and then prove no duplicate delivery.
