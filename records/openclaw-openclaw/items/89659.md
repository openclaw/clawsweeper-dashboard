## Summary

Keep this PR open: the Feishu rate-limit retry bug is real and the chosen shared helper boundary is reasonable, but the current patch is not merge-ready because it retries visible send calls without making the retried payloads idempotent and the supplied proof only covers direct/reply text stress sends.

## What This Changes

This PR adds bounded 500/1000 ms retry for Feishu send rate-limit code 230020 in requestFeishuApi, routes reply and streaming-card sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Keep the shared retry direction, but add one stable Feishu uuid per logical visible send across text/card/media/streaming create and reply paths, cover those payloads in focused tests, and require redacted live proof before merge.

## Reproduction Assessment

Yes for the PR-introduced risk from source: the new tests and helper show the same send closure can be invoked three times on 230020, while inspected create/reply payloads omit a stable uuid. I did not run tests in this read-only review.

## Solution Assessment

No. Retrying Feishu sends through the shared helper is an acceptable boundary, but the best fix must make every retried visible send idempotent and prove the broader send surface before merge.
