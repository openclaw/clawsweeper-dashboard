## Summary

Keep this PR open: it addresses a real Feishu send-rate-limit failure with useful live proof, but the retry loop can now re-send visible Feishu create/reply requests without a stable Feishu `uuid`, which leaves a concrete duplicate-message risk after ambiguous server delivery.

## What This Changes

Adds bounded `230020` retry/backoff to Feishu API requests, routes reply and streaming-card send paths through that helper, and adds Feishu retry/concurrency tests.

## Best Possible Solution

Keep the Feishu retry direction, but make retried create/reply sends idempotent with stable `uuid` reuse and have maintainers explicitly accept the bounded retry budget before merge.

## Reproduction Assessment

Yes at source level, though I did not run live Feishu: current main has no retry in `requestFeishuApi`, and the PR’s live terminal output shows `230020` send bursts recovering only after the patch.

## Solution Assessment

No as submitted. Retrying in the Feishu plugin is the right layer, but the maintainable fix must preserve idempotency with stable Feishu `uuid` values and settle the retry-budget policy.
