## Summary

Keep open: the retry belongs in the Feishu plugin, but this PR is not merge-ready because it replays side-effecting Feishu sends without stable `uuid` idempotency and the supplied proof does not show the repaired no-duplicate behavior.

## What This Changes

The PR changes Feishu's shared API helper to retry `230020` rate-limit failures, routes reply and streaming-card sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Keep the Feishu-plugin retry, but land it only after stable Feishu `uuid` values and focused tests/proof cover every retried `message.create` and `message.reply` path.

## Reproduction Assessment

Yes from source, though I did not run a live current-main Feishu stress test in this read-only review. Current main sends Feishu messages once through the SDK/helper and wraps a `230020` failure without retry, while the linked live proof shows this failure class under concurrency.

## Solution Assessment

No. Retrying in the Feishu plugin helper is the right layer, but the best fix must make each replayed send idempotent with Feishu `uuid` values and cover the direct, reply, media, and streaming-card paths.
