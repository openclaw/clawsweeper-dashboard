## Summary

Keep open: current main and the latest release still lack Feishu send-rate-limit retry, and this PR contains a plausible bounded fix with live direct/reply proof, but maintainer review should decide the broadened helper-level retry scope before merge.

## What This Changes

The PR adds bounded 230020 retry/backoff to the shared Feishu API helper, routes text/card replies and streaming-card initial sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Land a bounded 230020 retry only after Feishu-owner review accepts the intended send-surface scope, and keep or open narrower follow-up tracking for other Feishu rate-limit codes if they remain reproducible.

## Reproduction Assessment

Yes. Current main and v2026.6.1 show no retry in requestFeishuApi and direct reply sends bypass the helper, while the PR body provides live Feishu stress output for direct and reply sends hitting 230020 and succeeding after retry.

## Solution Assessment

Mostly yes. Centralizing a bounded retry around the Feishu SDK's thrown AxiosError shape is maintainable, but the final merge should either accept or narrow the extra media and streaming-card retry surface.
