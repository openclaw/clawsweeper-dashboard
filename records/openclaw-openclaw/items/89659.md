## Summary

Keep this PR open for changes. The direct-send retry proof is useful, but the patch is still too narrow for the Feishu message-loss bug because reply and streaming-card sends can hit the same documented rate-limit path while bypassing the changed helper, and `230006` is documented as a bot-ability setup error rather than a retryable frequency-limit error.

## What This Changes

The PR adds bounded retries to Feishu `requestFeishuApi` for selected send error codes and adds retry/concurrency tests for direct message sends.

## Best Possible Solution

Use one Feishu message-send retry path for documented transient `230020` failures across direct, reply, media, and streaming-card sends, and leave bot-ability setup errors out of the default retry set unless maintainers explicitly accept that compatibility tradeoff.

## Reproduction Assessment

Yes. The direct-send failure is supported by the PR's live Feishu proof, and the remaining sibling-path gap is source-reproducible because Feishu documents `230020` on reply sends while current main reply and streaming-card message sends bypass the changed helper.

## Solution Assessment

No. A helper-level retry is a good owner-boundary shape, but this PR is not the best complete fix until all Feishu message-send surfaces share it and the retryable-code set excludes documented configuration errors or has maintainer-approved proof.
