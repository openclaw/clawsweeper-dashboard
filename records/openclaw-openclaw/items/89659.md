## Summary

Keep this PR open but not merge-ready: the retry direction matches the Feishu delivery bug, but the current patch puts send-specific retry behavior in the generic Feishu API helper and the supplied live proof is unredacted and narrower than the delivery surfaces changed.

## What This Changes

Adds bounded retry/backoff for Feishu 230020 send rate-limit errors in `requestFeishuApi`, routes reply and streaming-card sends through that helper, and adds retry/concurrency tests.

## Best Possible Solution

Scope 230020 retry to explicit Feishu message create/reply send behavior or an opt-in helper, add focused tests and redacted proof for any approved media/card scope, and keep broader rate-limit codes tracked separately.

## Reproduction Assessment

The PR's copied live output also shows direct/reply bursts hitting 230020 and succeeding after retry, but I did not run a live Feishu tenant test in this read-only review.

## Solution Assessment

No, not yet. Retrying 230020 for Feishu message create/reply is the right direction, but placing the policy in `requestFeishuApi` is broader than the proved failure and needs a narrower helper or explicit Feishu owner acceptance with broader proof.
