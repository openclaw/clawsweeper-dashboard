## Summary

Keep open: the Feishu retry direction addresses a real message-loss bug, but this PR is not merge-ready because it replays visible sends without stable Feishu `uuid` dedupe and the supplied proof only covers direct/reply text stress sends.

## What This Changes

Adds bounded retry/backoff for Feishu send rate-limit code 230020 through `requestFeishuApi`, routes reply and streaming-card sends through that helper, and adds retry/concurrency tests.

## Best Possible Solution

Add one stable Feishu `uuid` per logical visible send and reuse it across every retry, then cover text, reply, media, and streaming-card retry behavior with focused tests plus redacted live proof.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.1 make only one attempt for the relevant Feishu send paths, while the PR body shows live 230020 failures during direct/reply stress sends. I did not run a live Feishu repro in this read-only review.

## Solution Assessment

No: bounded retry is the right direction, but replaying visible sends without stable Feishu request dedupe is not the best merge shape. The safer fix is idempotent retry across every retried create/reply payload, or a narrower retry surface until all affected paths are covered.
