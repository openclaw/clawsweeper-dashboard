## Summary

Keep this PR open: the Feishu rate-limit retry direction is useful and tied to a real message-loss issue, but the current patch retries visible send operations without Feishu request-deduplication and does not cover the non-throwing API-error response shape already handled elsewhere in the plugin.

## What This Changes

The PR adds bounded retry/backoff for Feishu message send rate-limit code 230020, routes text reply and streaming-card send paths through the shared Feishu API wrapper, and adds retry/concurrency tests.

## Best Possible Solution

Land a Feishu-local retry fix after each logical text/card/media/streaming send generates one stable uuid reused across retries, both thrown and response-body 230020 shapes are tested, and redacted live direct/reply proof is refreshed.

## Reproduction Assessment

Yes at source level. Current main sends Feishu messages without retry on reply/card paths, and the PR body plus official Feishu docs show 230020 is a rate-limit condition, but I did not run live Feishu proof in this read-only review.

## Solution Assessment

No. Retrying rate-limited sends is the right owner boundary, but the best fix must make retried Feishu sends idempotent with a stable uuid and cover thrown plus response-body 230020 shapes before this is merge-ready.

Full review comments:

- **[P1] Reuse Feishu UUIDs before retrying sends:** `extensions/feishu/src/comment-shared.ts:133`
  - body: This retry loop replays the same message send callback on 230020, but the retried create/reply/card payloads do not set Feishu's `uuid` de-duplication field. Feishu documents that the same `uuid` allows at most one successful send/reply within an hour; without generating one uuid per logical send and reusing it across retries, an ambiguous accepted send can be replayed into a duplicate visible message.
  - confidence: 0.91
- **[P2] Retry Feishu rate-limit response bodies:** `extensions/feishu/src/comment-shared.ts:133`
  - body: The helper only retries thrown AxiosError-shaped 230020 failures, then returns any resolved response directly. Existing Feishu code already documents that the SDK can surface API errors as normal response bodies, so a `{ code: 230020 }` response will still fall through to `assertFeishuMessageApiSuccess` without retrying.
  - confidence: 0.82
