## Summary

Keep this PR open: the Feishu retry direction is valid, but the current patch retries visible `message.create` and `message.reply` writes without Feishu's stable `uuid` dedupe key, so it can trade message loss for duplicate delivery under ambiguous write failures.

## What This Changes

The PR adds bounded retry handling for Feishu `230020` send rate-limit errors across direct, reply, and streaming-card send paths, with retry and concurrency tests.

## Best Possible Solution

Keep the Feishu plugin retry, but merge only after every retried visible send reuses one stable Feishu `uuid` per logical message and fresh redacted live proof shows no lost or duplicated direct/reply messages.

## Reproduction Assessment

Yes. Source inspection shows current main sends through a single-attempt Feishu API wrapper, and the PR reruns create/reply closures for `230020` while those payloads omit Feishu's documented dedupe `uuid`.

## Solution Assessment

No. The Feishu plugin is the right owner boundary for bounded retry, but the best fix must make retried visible writes idempotent across text, card, media, and streaming-card send variants.
