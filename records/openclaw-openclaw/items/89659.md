## Summary

Keep open: the PR is aimed at the right Feishu delivery-loss bug, but the retry helper only retries thrown SDK/Axios errors and can still skip retry when the SDK resolves with a non-zero retryable response code; the final Feishu retryable-code scope also still needs a 11233 decision and proof.

## What This Changes

The PR adds bounded retry/backoff handling for Feishu message send rate-limit responses across direct, reply, and streaming-card send paths with new retry and concurrency tests.

## Best Possible Solution

Keep the bounded Feishu-owned helper, but make it retry both thrown and resolved retryable send responses, then cover or explicitly exclude each documented send-rate-limit code with focused tests and refreshed proof.

Yes. From source inspection, a resolved Feishu send response with `code: 230020` or `code: 11232` returns from `requestFeishuApi` without retry and then fails at the existing send success assertion; I did not run live Feishu because this review is read-only and has no credentials.

## Solution Assessment

No. The shared helper is the right layer, but the proposed implementation is incomplete until it covers resolved API-level rate-limit responses and settles the documented 11233 send-limit code.
