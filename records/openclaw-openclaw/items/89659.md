## Summary

Keep open: the retry direction and live Feishu proof are useful, but the patch is not merge-ready because retries are not idempotent and still miss retryable 230020 responses that arrive as normal SDK response bodies.

## What This Changes

The PR adds bounded 230020 retry/backoff to the shared Feishu API helper, routes text/reply/streaming-card sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Before merge, make every logical Feishu send retry idempotently with a stable uuid, retry both thrown and returned-body 230020 failures, and have maintainers explicitly accept whether bounded retry is enough for high-burst reply scenarios.

## Reproduction Assessment

Yes. Current main source shows Feishu send paths have no bounded retry, and the linked issue plus PR live output show 230020 rate-limit sends failing before this patch; I did not rerun a live Feishu stress test in this read-only review.

## Solution Assessment

No. Retrying in the shared Feishu send helper is a plausible layer, but the best fix must use Feishu's documented uuid dedupe and handle the SDK's normal response-body error shape before returning.

- **[P1] Make retried Feishu sends idempotent:** `extensions/feishu/src/comment-shared.ts:133`
  - body: `requestFeishuApi` can now execute the same `im.message.create` or `im.message.reply` operation up to three times, but the send payloads still omit Feishu's `uuid` request-dedupe field. Feishu documents `uuid` for create/reply as allowing only one successful send/reply per hour for the same UUID, so a retry after Feishu accepted the first request but the client saw a retryable error can create duplicate visible messages. Generate a stable UUID per logical send and reuse it for every retry across text, media, and streaming-card create/reply paths.
  - confidence: 0.92
- **[P1] Retry 230020 response bodies before returning:** `extensions/feishu/src/comment-shared.ts:133`
  - body: The helper retries only caught exceptions. The pinned SDK types return `{ code, msg, data }`, and the sibling typing path already documents that Feishu SDK calls can resolve with API-level error codes; if `message.create` or `message.reply` resolves with `{ code: 230020 }`, this line returns it and the later success check throws without retry. Detect retryable response codes inside `requestFeishuApi` before returning and add a returned-body 230020 regression test.
  - confidence: 0.89
