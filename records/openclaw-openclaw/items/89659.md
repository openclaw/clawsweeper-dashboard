## Summary

This PR should stay open for Feishu-owner maintainer review: the patch is a focused and source-backed delivery reliability fix with sufficient live proof, but merging it changes Feishu send semantics from immediate failure to bounded delayed retry.

## What This Changes

The PR adds bounded retry/backoff for Feishu 230020 send rate-limit errors in the shared API wrapper, routes direct/reply/streaming-card sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Land the bounded 230020 retry only after Feishu-owner acceptance and exact-head validation, treating it as a delivery-reliability mitigation rather than a durable queue or idempotency layer.

## Reproduction Assessment

## Solution Assessment

Yes, if maintainers accept the bounded retry tradeoff. The shared Feishu API wrapper is the narrow owner for thrown SDK send failures, while guaranteed delivery or idempotency would be a separate queue-level design.
