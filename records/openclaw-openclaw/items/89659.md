## Summary

Keep open: the retry direction matches the Feishu rate-limit problem, but the patch retries visible create/reply writes without Feishu `uuid` idempotency, so merging can duplicate delivered messages and the supplied live proof needs redaction and rerun after that repair.

## What This Changes

This PR adds bounded `230020` retry handling to Feishu API sends/replies, wraps streaming-card starts, and adds retry/concurrency tests.

## Best Possible Solution

Keep the shared retry approach, but add one stable Feishu `uuid` per logical visible send/reply and reuse it across every retry for text, media, and streaming-card starts, then accept redacted live proof.

## Reproduction Assessment

Yes, source-reproducible: current main and `v2026.6.1` call Feishu sends once without retry, and Feishu docs define `230020` as a retryable frequency limit; I did not rerun a live Feishu scenario in this read-only review.

## Solution Assessment

No: shared bounded retry is the right layer, but the best fix must make each retried visible create/reply idempotent with Feishu `uuid` across text, media, and streaming-card starts before merge.
