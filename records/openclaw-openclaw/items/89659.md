## Summary

Keep open: the PR addresses a real Feishu message-loss path, but the current patch is not merge-ready because retrying Feishu sends without a stable idempotency uuid can duplicate delivered messages, and the retry scope now exceeds the direct/reply live proof supplied.

## What This Changes

The PR adds bounded retry/backoff for Feishu 230020 send errors in `requestFeishuApi`, routes reply and streaming-card initial sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Land a Feishu send retry that reuses one stable Feishu `uuid` per logical message send and only includes streaming-card/media surfaces after focused tests, live proof, or Feishu-owner acceptance.

## Reproduction Assessment

Yes, from source and supplied proof: current main has Feishu send paths without retry, while the PR body shows live 230020 rate-limit responses under concurrent direct/reply sends. I did not run a live Feishu account in this read-only review.

## Solution Assessment

No as written: helper-level retry is a plausible mitigation, but it is too broad and not idempotent for message sends. The safer fix is stable uuid reuse plus scoped proof for every send path that inherits retry.
