## Summary

Keep open. The retry direction is useful, but this PR is not merge-ready because it retries `230006` even though Feishu documents that as bot ability not activated, and the main reply-send path still bypasses the new retry helper. ([open.feishu.cn](https://open.feishu.cn/document/server-docs/im-v1/message/create?lang=zh-CN&utm_source=openai))

## What This Changes

The branch adds bounded retries in Feishu `requestFeishuApi` for `230020`/`230006` and adds retry/concurrency tests for direct message sends.

## Best Possible Solution

Land this after the Feishu plugin retries only documented transient send failures and routes direct, reply, media, and streaming-card send paths through one bounded retry shape with focused coverage.

## Reproduction Assessment

Yes, at source level. Current main has one-shot Feishu message create/reply paths and terminal nonzero-response assertions; the PR tests only the direct create wrapper path, not reply send retry behavior.

## Solution Assessment

No. A Feishu-owned bounded retry helper is the right direction, but this implementation is too narrow and includes an API code Feishu documents as a configuration error rather than a transient rate limit.
