## Summary

Keep this PR open: the Feishu rate-limit retry direction is useful for the linked message-loss bug, but the patch is not merge-ready because retried Feishu create/reply sends do not reuse Feishu's documented `uuid` idempotency key.

## What This Changes

The branch adds bounded retry handling for Feishu message send rate-limit errors and adds retry/concurrency tests for direct and reply send paths.

## Best Possible Solution

Keep the retry behavior, but make every retried Feishu message create/reply path generate one stable `uuid` per logical send and reuse it across attempts, with tests proving retry reuse and distinct UUIDs for distinct sends.

## Reproduction Assessment

Yes at source level: current main has Feishu send paths that can surface `230020` without retry, and the PR body includes live Feishu stress output showing recovery after retry. I did not run a live Feishu reproduction in this read-only review.

## Solution Assessment

No. Retrying `230020` is the right general direction, but the best fix must include Feishu `uuid` idempotency for every retried message create/reply path and then refresh real proof after that repair.
