## Summary

Keep open: this is useful Feishu message-delivery work, but the current patch is not merge-ready because the retry helper only retries thrown SDK errors and still misses retryable Feishu response bodies that the SDK can resolve normally.

## What This Changes

The PR adds bounded retry/backoff handling for Feishu message send rate-limit responses across direct, reply, and streaming-card sends, plus retry and concurrency tests.

## Best Possible Solution

Keep the shared Feishu send wrapper as the owner, but make it retry both thrown and resolved explicit rate-limit signals, add focused tests for 230020/11232/429 and non-retryables, then refresh redacted live proof before merge.

## Reproduction Assessment

Yes. From the SDK 1.66.0 source, a mocked request that resolves `{ code: 11232 }` or `{ code: 230020 }` will return from `requestFeishuApi` without retry and then fail once in the send success assertion; this was source-verified rather than executed because the review is read-only.

## Solution Assessment

No. The shared wrapper is the right layer, but the current implementation is too narrow because it only handles thrown AxiosError shapes instead of the resolved response body shape exposed by the SDK.
