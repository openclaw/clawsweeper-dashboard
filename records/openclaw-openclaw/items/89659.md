## What This Changes

Adds bounded retry/backoff for Feishu 230020 send rate-limit errors in the shared Feishu API helper, routes reply and streaming-card initial sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Land the bounded Feishu 230020 retry if the Feishu owner accepts rejected-send semantics and helper scope; otherwise narrow retry to explicit message create/reply calls or add uuid-based idempotency before merge.

## Reproduction Assessment

Yes, source-reproducible: current main sends direct Feishu messages through a one-shot helper and reply/streaming-card sends without retry, so a thrown Feishu 230020 fails rather than retrying. I did not run a live current-main Feishu stress test in this read-only review.

## Solution Assessment

Yes, this is an acceptable plugin-local fix shape because the affected send paths already live in the Feishu plugin. The remaining best-fix question is whether maintainers want helper-wide retry without uuid idempotency or a narrower/idempotent retry path.
