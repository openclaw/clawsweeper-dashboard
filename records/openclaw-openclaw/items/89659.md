## Summary

Keep this PR open. The direct-send retry proof is useful, but the patch is still too narrow for the linked message-loss bug and it retries a documented Feishu configuration error without live proof that it is transient.

## What This Changes

The PR adds bounded retry/backoff for Feishu API calls that go through `requestFeishuApi`, adds a 230020/230006 retry-code detector, and adds direct-send retry/concurrency tests.

## Best Possible Solution

Use one Feishu message-send retry path for direct, reply, media, and streaming-card sends on documented transient 230020, with separate maintainer-approved proof before retrying 230006 by default.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: make Feishu `message.create` or `message.reply` throw an Axios-shaped error with `response.data.code = 230020`; current main wraps or throws it once, while the PR only retries callers routed through `requestFeishuApi`. I did not run a live current-main Feishu failure because the PR already provides live after-fix proof for direct sends.

## Solution Assessment

No. The helper approach is maintainable, but this PR is not the best complete fix until all visible Feishu send paths share it and 230006 is removed or separately justified with live API proof.
