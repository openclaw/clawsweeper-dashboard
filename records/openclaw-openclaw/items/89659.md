## Summary

Keep open: the live Feishu proof supports the retry direction, but the current retry loop can duplicate visible messages because retried create/reply sends do not use a stable Feishu `uuid`, and it still misses retryable nonzero response bodies. The documented reply-burst ceiling also needs maintainer acceptance before the linked message-loss issue is treated as solved.

## What This Changes

The PR adds bounded linear retry for Feishu `230020` send rate-limit errors, routes direct/reply/streaming-card sends through the shared helper, and adds retry/concurrency tests.

## Best Possible Solution

Land Feishu send retry after each logical message send has stable Feishu `uuid` idempotency, retry handles both thrown and resolved `230020` shapes, and maintainers decide whether bounded retry is enough or reply bursts need pacing.

## Solution Assessment

No. The retry direction is useful, but the current implementation is not the best fix until it makes retries idempotent and covers retryable response bodies; reply-burst pacing remains an owner decision.

- **[P1] Make retried Feishu sends idempotent:** `extensions/feishu/src/comment-shared.ts:133`
  - body: This retry loop replays the same `im.message.create`/`im.message.reply` POST when `230020` is thrown, but none of the send paths pass the Feishu `data.uuid` that the pinned SDK exposes for create/reply. If the first attempt reaches Feishu and the client loses the response, the retry can create a second visible message; generate one stable logical-send UUID and reuse it for every attempt across text/card/media/streaming sends.
  - confidence: 0.88
- **[P2] Retry rate-limit response bodies before returning:** `extensions/feishu/src/comment-shared.ts:133`
  - body: The helper only retries caught Axios errors, but the Feishu SDK returns normal response bodies for successful HTTP responses and existing callers throw after `requestFeishuApi` returns. A resolved `{ code: 230020 }` therefore still bypasses retry and fails as message loss, so detect retryable response codes before returning or move send-result validation into the retry boundary.
  - confidence: 0.8
