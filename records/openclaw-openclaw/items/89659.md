## Summary

Keep this PR open: the retry direction addresses the Feishu rate-limit message-loss bug, but the current patch retries visible sends without a stable Feishu `uuid`, so it can trade dropped messages for duplicate messages before it is safe to merge.

## What This Changes

The branch adds bounded retry/backoff for Feishu `230020` send errors, routes direct/reply/streaming-card sends through the retry helper, and adds retry/concurrency tests.

## Best Possible Solution

Keep the retry approach, but make each logical Feishu create/reply/card send generate one stable `uuid`, reuse it across all retry attempts, cover reuse and per-message uniqueness in tests, and refresh redacted live Feishu proof after the repair.

## Reproduction Assessment

Yes, source-level: current main has no retry in `requestFeishuApi`, and the PR/live proof show the target Feishu send rate-limit path. I did not run a live Feishu account in this read-only review.

## Solution Assessment

No; bounded retry is the right direction, but retrying visible sends without stable Feishu `uuid` reuse is not the best merge shape. The maintainable fix is idempotent retry at the send-call boundary plus focused tests.
