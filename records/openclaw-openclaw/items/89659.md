## Summary

Keep open: the Feishu send retry is useful and backed by live proof, but this patch also changes streaming-card updates from cumulative text to suffix-only payloads, which conflicts with current tests and Feishu's Card Kit contract.

## What This Changes

The branch adds bounded retry/backoff for Feishu 230020 send rate-limit errors, routes Feishu direct/reply/streaming-card sends through the shared wrapper, adds retry/concurrency tests, and changes streaming-card update payloads to send suffix content.

## Best Possible Solution

Keep cumulative Feishu streaming-card content updates, keep the bounded 230020 retry for message create/reply/send-card paths, and merge only after the focused Feishu retry and streaming-card tests pass.

## Reproduction Assessment

Yes. The original rate-limit failure is source-reproducible from current main's one-shot requestFeishuApi wrapper and live PR proof, and the review blocker is source-reproducible because the PR sends suffix content where current tests and Feishu docs require cumulative content.

## Solution Assessment

No. The retry wrapper is the right narrow owner for 230020 send failures, but the unrelated streaming-card suffix change is not part of the best fix and should be removed or separately proven.
