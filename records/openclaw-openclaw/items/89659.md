## Summary

Keep open: the PR is a useful Feishu message-loss fix direction, but the latest 11232/HTTP 429 expansion is still incomplete because the retry loop only classifies thrown Axios errors and can miss SDK-resolved body-level rate-limit responses.

## What This Changes

The PR adds bounded Feishu send retry/backoff in the shared request helper, routes reply and streaming-card send paths through it, and adds retry/concurrency tests.

## Best Possible Solution

Keep the bounded retry approach, but make the shared helper classify retryable Feishu body codes from resolved responses as well as thrown Axios errors, with focused tests and updated real-environment proof for the final supported signals.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: make requestFeishuApi receive a resolved response body like { code: 11232, msg: "rate limit" } and it returns without retry before downstream send success checks throw. I did not run tests because this review is required to keep the checkout read-only.

## Solution Assessment

No: the retry helper is the right owner boundary, but the current implementation only covers thrown errors and not the SDK-resolved body shape already acknowledged elsewhere in the Feishu plugin. The best fix is one canonical classifier used inside the retry loop for both thrown and resolved responses.
