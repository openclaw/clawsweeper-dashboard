## Summary

Keep this PR open: the retry direction is useful and backed by live Feishu output, but the current patch retries non-idempotent message POSTs and can duplicate visible Feishu messages unless each logical send reuses a stable Feishu `uuid`.

## What This Changes

The PR adds bounded retry/backoff for Feishu `230020` send rate-limit errors in the shared Feishu API helper, routes text replies and streaming-card sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Land the retry behavior only after it is idempotent with stable per-logical-send `uuid` reuse across every retried create/reply path, then have maintainers accept the fixed retry budget or switch to pacing.

## Reproduction Assessment

Yes at source level: current main's shared helper performs one attempt, while reply and streaming-card send paths bypass any retry handling, so a thrown `230020` send error exits immediately. I did not run a live current-main Feishu rate-limit reproduction in this read-only review.

## Solution Assessment

No, not yet. Bounded retry is a plausible fix, but the best maintainable version must make retried message POSTs idempotent with Feishu `uuid` and settle whether the fixed two-retry budget is acceptable.
