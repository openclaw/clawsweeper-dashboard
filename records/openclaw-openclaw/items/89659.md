## Summary

Keep open: the live Feishu proof supports the retry direction, but the patch retries visible message-send POSTs without using the SDK-supported `uuid` idempotency field, so an ambiguous first-attempt success can become duplicate chat delivery on retry.

## What This Changes

The PR adds bounded `230020` retry/backoff to Feishu API requests, routes direct, reply, media, and streaming-card send paths through that helper, and adds retry/concurrency tests.

## Best Possible Solution

Reuse one stable Feishu `uuid` per logical send across all retry attempts, keep retryable codes narrow, and have maintainers accept or adjust the retry budget before merge.

## Reproduction Assessment

Yes at source/proof level: the PR body includes live Feishu direct and reply stress output for `230020`, and the changed source clearly retries those send callbacks. I did not run live Feishu locally during this read-only review.

## Solution Assessment

No, not yet: centralizing bounded retry in `requestFeishuApi` is a plausible owner-boundary fix, but it is not the best mergeable form until retried visible sends reuse stable Feishu `uuid` values and maintainers accept the retry budget.
