## Summary

Keep open: the PR is a useful Feishu message-loss fix, and maintainer discussion already pushed it toward the right retryable-code boundary, but the current helper still only retries thrown Axios errors. Feishu SDK success-status responses resolve to `{code,msg,data}`, so a resolved 11232/230020 body can still bypass the retry loop and fail after `requestFeishuApi` returns.

## What This Changes

The PR adds bounded retry/backoff handling for Feishu message create/reply/streaming-card sends on 230020, 11232, and HTTP 429, plus retry and concurrency tests.

## Best Possible Solution

Centralize Feishu send retry classification so the helper handles both thrown Axios 429/errors and resolved Feishu response-body codes before success assertions, then add focused tests and refreshed redacted proof or an explicit maintainer-accepted note for 11232/429.
