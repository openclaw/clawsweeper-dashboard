## Summary

Keep this PR open because it is still a useful Feishu message-loss fix, but the latest patch is not merge-ready: the retry loop only handles thrown rate-limit errors and can still miss SDK-resolved Feishu error bodies such as `code: 11232`.

## What This Changes

The PR adds bounded retry/backoff around Feishu message send calls and adds Feishu send retry/concurrency tests.

## Best Possible Solution

Classify retryable returned Feishu response bodies inside `requestFeishuApi` before returning, keep non-retryable codes immediate, preserve withdrawn-reply fallback behavior, and add redacted proof or a maintainer-accepted explanation for hard-to-force live signals.

Yes from source: the SDK returns resolved message responses with `{ code?, msg?, data? }`, and the PR returns those bodies before checking retryable codes. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet: wrapping the shared send helper is the right layer, but the helper must classify both thrown AxiosError shapes and resolved Feishu response bodies to be the best fix.
