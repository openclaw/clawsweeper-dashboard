## Summary

Keep this PR open: the Feishu retry direction is useful and still needed, but the current patch is not merge-ready because the shared retry loop can repeat visible message writes without reusing Feishu's idempotency `uuid`, and the live proof needs a redacted refresh after that repair.

## What This Changes

The branch adds bounded retry for Feishu `230020` send rate-limit errors in `requestFeishuApi`, routes direct/reply and streaming-card sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Keep the Feishu plugin retry, but merge only after each logical visible send reuses one stable Feishu `uuid` across retries and redacted live proof confirms delivery without loss or duplicates.

## Reproduction Assessment

Yes, source-reproducible: current main has a single-attempt Feishu API wrapper and direct/reply sends wrap a `230020` SDK error as a failed send. I did not run live Feishu in this read-only review.

## Solution Assessment

No, not as written. The Feishu plugin is the right owner boundary for bounded retry, but the best fix must add stable `uuid` reuse across retried visible message writes and refresh live proof after that repair.
