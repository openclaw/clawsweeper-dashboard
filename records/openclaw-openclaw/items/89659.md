## Summary

Keep this PR open: it is a useful Feishu message-loss fix direction with relevant live proof, but it is not merge-ready because the shared retry loop can resubmit visible Feishu sends without a stable `uuid`, and the supplied proof needs to be redacted and rerun after that repair.

## What This Changes

The PR adds bounded retry/backoff for Feishu `230020` send-rate-limit errors in the shared Feishu API wrapper, routes reply and streaming-card initial sends through that wrapper, and adds direct-send retry/concurrency tests.

## Best Possible Solution

Make each retried logical visible send reuse a stable Feishu `uuid`, keep the retry scope limited to `230020`, cover direct/reply/card/media paths with focused tests, and refresh redacted live Feishu proof before merge.
