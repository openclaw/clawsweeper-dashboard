## Summary

Keep open: the Feishu retry direction is useful, but this PR is not merge-ready because it replays message sends without a stable native idempotency key, includes an unrelated streaming-card content change that conflicts with current tests, and the live proof is unredacted and narrower than the affected send surfaces.

## What This Changes

The PR adds bounded retry/backoff for Feishu 230020 send rate-limit errors across direct, reply, and streaming-card sends, plus new retry/concurrency tests.

## Best Possible Solution

Make retries opt-in or send-scoped, reuse a stable Feishu uuid for every retried message operation, remove the unrelated streaming-card append-payload change unless it is separately proven and tested, and either broaden the fix for the linked issue or remove the closing reference.

  - body: This change switches streaming-card updates from the cumulative text to only the appended suffix. Current tests for this module assert the Feishu CardKit update payload remains cumulative, including throttled and failed-update retry cases; sending only the suffix would truncate or replace visible card content if the endpoint keeps the existing contract. Remove this unrelated change or update the streaming-card contract with focused tests and real proof.
  - confidence: 0.92
- **[P1] Make Feishu send retries idempotent:** `extensions/feishu/src/comment-shared.ts:133-136`
  - body: The retry loop re-executes create/reply sends, but the message payloads still omit Feishu's native `uuid` field exposed by the SDK for both create and reply. If a retried request is accepted or partially processed before the caller sees the rate-limit error, the retry can create duplicate visible messages; generate one stable uuid per logical send and reuse it across all retry attempts.
  - confidence: 0.78
