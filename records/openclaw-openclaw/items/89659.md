## Summary

Keep this PR open: the Feishu rate-limit retry direction is useful and backed by live proof, but the current retry loop can replay non-idempotent message sends without a stable Feishu `uuid`.

## What This Changes

The PR adds bounded retry/backoff for Feishu `230020` send rate-limit errors in the shared Feishu API helper, routes direct/reply/streaming-card sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Land the Feishu send retry only after all retried `message.create` and `message.reply` operations reuse stable per-send Feishu `uuid` values and the bounded retry budget is an explicit maintainer choice.

## Solution Assessment

No, not yet. Retrying Feishu `230020` at the send helper is a reasonable owner-boundary fix, but the retry needs stable Feishu `uuid` idempotency and an accepted retry-budget policy before it is the best solution.
