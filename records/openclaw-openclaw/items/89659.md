## Summary

Keep this PR open, but it is not merge-ready: bounded Feishu send retry is the right direction, yet the current patch replays visible create/reply sends without Feishu's documented `uuid` request dedupe and the supplied proof does not cover the full retry surface.

## What This Changes

Adds bounded retry/backoff for Feishu message API rate-limit code 230020 through `requestFeishuApi`, routes reply and streaming-card sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Add one stable Feishu `uuid` per logical visible send and reuse it across every retry, then prove direct, reply, media, and streaming-card retry behavior with focused tests and redacted live evidence.

## Reproduction Assessment

Yes, at source level: current main makes one Feishu delivery attempt for the reviewed paths, and the PR body includes live Feishu output showing `230020` during concurrent direct/reply sends. I did not run a live Feishu repro in this read-only review.

## Solution Assessment

No: bounded retry is the right direction, but replaying visible sends without stable Feishu request dedupe is not the best merge shape. The safer fix is idempotent retry across every retried create/reply payload, or a narrower retry surface.
