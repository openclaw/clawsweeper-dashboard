## Summary

Keep open: the PR addresses a real Feishu message-loss path, but the patch is not merge-ready because the new retry loop can duplicate non-idempotent sends and still misses SDK responses that resolve with a rate-limit body.

## What This Changes

The PR changes Feishu's shared API wrapper to retry send rate-limit code 230020 twice with linear backoff, routes reply and streaming-card sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Land Feishu retry only after each logical create/reply/card/media send reuses a stable Feishu uuid across attempts, body-level 230020 responses are retried inside the helper, and maintainers decide whether bounded retry is enough or pacing is needed.

## Reproduction Assessment

Yes. Current main has a single-attempt Feishu API wrapper, and the PR's live Feishu proof demonstrates 230020 rate-limit failures under concurrent direct and reply sends; I did not run a live Feishu reproduction in this read-only review.

## Solution Assessment

No. Retrying in the shared wrapper is a plausible layer, but the best fix also needs idempotent uuid reuse, body-response retry handling, and an explicit maintainer decision on the remaining bounded-retry ceiling.
