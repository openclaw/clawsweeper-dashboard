## Summary

Keep open: the retry direction is still needed because current main and the latest release lack Feishu send retry, but this patch retries visible message writes without a stable Feishu `uuid`, so a retry after an ambiguous delivery can duplicate messages.

## What This Changes

The PR adds bounded retry handling for Feishu `230020` send/reply rate-limit errors and adds retry/concurrency tests for the Feishu channel plugin.

## Best Possible Solution

Keep the shared retry approach, but add one stable Feishu `uuid` per logical visible send/reply and reuse it across all retries for text, media, and streaming-card starts before accepting redacted live proof.

## Reproduction Assessment

Yes, source-reproducible: current main and `v2026.6.1` call Feishu sends once without retry, while the linked issue and PR proof show live `230020` send-rate-limit failures under concurrency. I did not rerun a live Feishu scenario in this read-only review.

## Solution Assessment

No: helper-level bounded retry is the right layer, but the best fix must make retried visible writes idempotent with Feishu `uuid` before merge. The repair should cover direct, reply, media, and streaming-card start sends rather than only the text path.
