## Summary

Keep open. The Feishu retry direction is valid, but this PR replays message create/reply POSTs without a stable Feishu `uuid`, so an ambiguous first send can become a duplicate visible chat message; the supplied proof is also unredacted and narrower than the affected send surface.

## What This Changes

The PR adds bounded retry/backoff for Feishu `230020` send failures in the shared API helper, routes reply and streaming-card sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Keep this PR open and land a Feishu-owned bounded retry that passes one stable `uuid` per logical create/reply send across text, card, media, and streaming-card delivery, with focused idempotency tests and redacted live proof.

## Reproduction Assessment

Yes, at source level. Current main makes one Feishu delivery attempt before terminal error handling, and the PR body provides live stress output for direct/reply text rate-limit retries, though this read-only review did not run a live Feishu scenario.

## Solution Assessment

No. Retrying inside the Feishu plugin is the right layer, but the best fix must make each logical send idempotent with a stable Feishu `uuid` and prove the broader helper-level call-site impact before merge.
