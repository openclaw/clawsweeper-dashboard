## What This Changes

Adds bounded retry handling for Feishu message send rate-limit responses across direct, reply, and streaming-card send paths, plus retry and concurrency tests.

## Best Possible Solution

Add one stable Feishu `uuid` per logical visible send and reuse it across every retry, then prove direct, reply, media, and streaming-card send retries with focused tests and redacted live evidence.

## Reproduction Assessment

Source-reproducible: current main lacks reply-path retry, and the PR body includes live Feishu terminal output showing `230020` under concurrent direct/reply sends. I did not run a live Feishu repro from this checkout.

## Solution Assessment

No: bounded retry is the right direction, but replaying visible sends without stable Feishu request dedupe is not the best merge shape yet. The safer fix is idempotent retry across every retried create/reply payload, or a narrower retry surface.
