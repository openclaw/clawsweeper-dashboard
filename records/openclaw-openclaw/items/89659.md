## Summary

Keep this PR open: the 230020 retry/backoff is a useful direct-send mitigation with credible live proof, but it is not ready as the closing fix because Feishu reply and streaming-card send paths still bypass the retry helper, and 230006 is documented by Feishu as a bot-ability configuration error rather than a rate limit.

## What This Changes

The PR adds bounded retry/backoff to Feishu `requestFeishuApi` for body codes 230020 and 230006 and adds direct-send retry/concurrency tests.

## Best Possible Solution

Keep the bounded 230020 retry shape, apply it through the shared Feishu message-send helper for create/reply/streaming send surfaces, and leave 230006 non-retryable unless maintainers accept concrete vendor or live proof.

## Reproduction Assessment

Yes, source-reproducible with high confidence: current main has one-shot Feishu message send paths, and Feishu docs list 230020 on create/reply message APIs with the same rate limit. I did not run a live current-main Feishu failure repro in this read-only review.

## Solution Assessment

No: the bounded 230020 retry helper is the right direction, but this PR applies it too narrowly and adds 230006 without contract proof. The safer fix is a shared message-send retry path for create/reply/streaming sends, limited to documented transient codes.
