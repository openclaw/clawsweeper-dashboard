## Summary

Keep open: the Feishu retry direction is useful, but this PR is not merge-ready because visible retried sends should reuse Feishu's `uuid` idempotency field and the supplied live proof should be refreshed with redacted identifiers after that repair.

## What This Changes

The PR adds bounded retry/backoff for Feishu `230020` send rate-limit errors across direct sends, replies, and streaming-card sends, with new retry and concurrency tests.

## Best Possible Solution

Keep the bounded Feishu retry, but generate one stable UUID per logical visible send and reuse it for every retry across direct, reply, and streaming-card initial sends, with focused tests and redacted live Feishu proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main wraps Feishu direct sends with a helper that does not retry and replies are one-shot, so a Feishu `230020` response becomes a send failure. I did not run a live Feishu current-main stress test in this read-only review.

## Solution Assessment

No. Bounded retry in the Feishu plugin is the right layer, but the best fix needs stable Feishu `uuid` reuse for retried visible sends and refreshed redacted live proof after that repair.
