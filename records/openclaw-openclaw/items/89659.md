## Summary

The PR is still useful and should stay open: current main still sends Feishu direct, reply, and streaming-card messages without retrying the observed `230020` send-rate-limit response, while this branch adds a bounded shared retry path with tests and live Feishu proof. ## What This Changes

The PR adds bounded retry/backoff for Feishu `230020` send rate-limit errors in the shared API wrapper, routes direct, reply, and streaming-card send paths through it, and adds retry/concurrency tests.

## Best Possible Solution

Land the plugin-local bounded retry if Feishu owners accept the delayed-send semantics, while keeping durable queueing or stronger delivery guarantees as separate product work.

## Reproduction Assessment

Yes from source and PR proof, though I did not rerun the live Feishu scenario in this read-only review. Current main wraps Feishu send failures without retry, and the PR body includes live direct/reply stress output showing `230020` hits recover after the patch.

## Solution Assessment

Mostly yes, pending owner acceptance of the delivery tradeoff. A plugin-local bounded retry in the shared Feishu API wrapper is narrower than adding a core queue, but maintainers should explicitly accept that this is burst mitigation rather than durable delivery.
