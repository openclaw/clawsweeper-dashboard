## Summary

Keep open. The PR targets a real Feishu delivery bug and includes convincing live proof, but the retry loop still needs message-delivery fixes before merge because it replays non-idempotent sends and only retries thrown Axios-style errors.

## What This Changes

Adds bounded retry/backoff for Feishu `230020` send rate-limit errors in the shared Feishu API helper, routes text and streaming-card create/reply sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Keep the Feishu-owned retry helper, but make every retried message create/reply path use a stable per-logical-send `uuid`, handle both thrown and resolved `230020` response shapes, and leave the retry budget as an explicit maintainer-approved limit.

## Solution Assessment

No, not yet. A Feishu-owned retry helper is the right layer, but the current implementation needs stable request ids and non-throwing response handling before it is the best fix.
