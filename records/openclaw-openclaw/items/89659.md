## Summary

Keep this PR open for maintainer review: the patch is a focused Feishu delivery reliability fix with strong live proof, but it intentionally changes outbound delivery semantics by retrying message create/reply calls without wiring Feishu's documented `uuid` de-duplication field into those retried sends.

## What This Changes

The PR adds bounded retry handling for Feishu message send rate-limit code 230020 across direct, reply, and streaming-card send paths with new retry/concurrency tests.

## Best Possible Solution

Land a Feishu-owned bounded retry only after the Feishu owner accepts the no-`uuid` tradeoff or the PR adds per-logical-send de-duplication for retried create/reply calls.

## Reproduction Assessment

Yes, at source level: current main performs one Feishu create/reply attempt and then throws or fails on a rate-limit error. The PR also supplies live Feishu terminal proof showing 230020 failures recovered after retry on direct and reply sends.

## Solution Assessment

The owner boundary is right: a Feishu-local retry wrapper is better than a core retry layer or new config option. The remaining question is whether bounded retries without Feishu `uuid` de-duplication are the best final delivery semantics.
