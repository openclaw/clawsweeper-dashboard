## Summary

Keep this PR open, but it is not merge-ready. The live Feishu proof is useful and the direction is right, but the retry loop can reissue visible sends without Feishu `uuid` idempotency, and the helper still lets a retryable Feishu body response fail after returning instead of backing off.

## What This Changes

The PR adds bounded retry/backoff for Feishu `230020` send rate-limit errors, routes direct/reply/streaming-card sends through the shared helper, and adds retry/concurrency tests.

## Best Possible Solution

Land a retry implementation that reuses a stable Feishu `uuid` per logical send, retries both thrown and returned `230020` responses, and has an explicit maintainer decision on the residual reply-burst ceiling.

## Reproduction Assessment

Yes at source level. The PR head retries thrown `230020` errors, while create/reply payloads omit Feishu `uuid` and the helper returns resolved response bodies before checking for retryable `code: 230020`.

## Solution Assessment

No, not yet. Retrying Feishu sends is the right layer, but the best fix must make retries idempotent and handle both thrown and returned rate-limit responses before merge.

Full review comments:

- **[P1] Make retried Feishu sends idempotent:** `extensions/feishu/src/comment-shared.ts:133`
  - body: The new retry loop calls the same send closure again after a thrown `230020`, but the Feishu create/reply payloads in `send.ts` and `streaming-card.ts` still omit `data.uuid`. Feishu's create/reply API uses `uuid` to dedupe repeated sends for one hour, so a request that reached Feishu but failed before the client received the response can be sent again and produce duplicate visible messages. Generate a stable uuid per logical send and reuse it for every retry, with tests that assert retried create and reply calls carry the same uuid.
  - confidence: 0.88
- **[P2] Retry rate-limit response bodies before returning:** `extensions/feishu/src/comment-shared.ts:133`
  - body: `requestFeishuApi` returns any resolved SDK body immediately, so a resolved `{ code: 230020 }` flows to `assertFeishuMessageApiSuccess` and fails without backoff. The Feishu send code already treats nonzero response bodies as normal SDK results, so fold `code === 230020` results into the same retry path before returning.
  - confidence: 0.77
