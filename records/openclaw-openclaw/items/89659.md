## Summary

Keep open: the retry direction is valid and the contributor supplied useful live Feishu proof for direct/reply sends, but the patch still has blocking message-delivery risk because retried message sends are not idempotent and the streaming-card retry expansion is broader than the proof.

## What This Changes

Adds bounded retry/backoff for Feishu 230020 send errors in `requestFeishuApi`, routes reply and streaming-card initial sends through that helper, and adds retry/concurrency tests.

## Best Possible Solution

Land a narrow Feishu send retry that reuses one stable Feishu `uuid` per logical retried `message.create`/`message.reply` payload and only includes streaming-card paths after focused tests, live proof, or explicit Feishu-owner acceptance.

## Reproduction Assessment

Yes, from source and supplied proof: current main has Feishu send paths without retry, while the PR body shows live 230020 rate-limit responses under concurrent direct/reply sends. I did not run a live Feishu account in this read-only review.

## Solution Assessment

No as written: helper-level retry is a plausible mitigation, but it is too broad and not idempotent for message sends. The safer fix is stable uuid reuse plus scoped proof for every send path that inherits retry.

- **[P1] Reuse a Feishu uuid across retried sends:** `extensions/feishu/src/comment-shared.ts:139-141`
  - body: `requestFeishuApi` now retries any caller that throws Feishu code 230020, but the affected `message.create`/`message.reply` payloads do not pass the SDK's `uuid` field. Without one stable uuid per logical OpenClaw send, a retry can create duplicate delivered messages if Feishu accepts the first send but the client still receives a retryable response. Generate the uuid outside the retried closure and reuse it across retry attempts, or get explicit Feishu-owner acceptance of the 230020 semantics.
  - confidence: 0.82

Add focused tests/live proof for these modes or keep this PR scoped to the proven send paths.
