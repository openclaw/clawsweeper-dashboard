## Summary

Keep open: the retry direction is useful, but the current patch retries visible Feishu sends without stable Feishu `uuid` idempotency, so an ambiguous retry can duplicate chat messages; redacted post-repair live proof is also still needed.

## What This Changes

The PR adds bounded retry/backoff for Feishu `230020` send rate-limit errors, routes reply and streaming-card initial sends through the shared Feishu API wrapper, and adds retry/concurrency tests.

## Best Possible Solution

Keep bounded retry, but generate one stable Feishu `uuid` per logical visible send, reuse it across retry attempts, cover uuid reuse and distinct-send uniqueness in tests, and refresh redacted live Feishu proof after repair.

## Solution Assessment

No; bounded retry is the right direction, but the best merge shape must make visible sends idempotent with Feishu uuid reuse and refresh redacted live proof after that change.
