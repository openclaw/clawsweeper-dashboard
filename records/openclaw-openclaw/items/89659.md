## Summary

The PR should stay open because the retry direction is useful, but the current patch can trade Feishu rate-limit message loss for duplicate delivery: it replays send calls without a stable Feishu `uuid` idempotency key.

## What This Changes

The PR adds bounded retries for Feishu `230020` send rate-limit errors in the shared API helper, routes reply and streaming-card sends through that helper, and adds retry/concurrency tests.

## Best Possible Solution

Keep the bounded `230020` retry idea, but generate and reuse a stable Feishu `uuid` for every retried message create/reply payload and prove both rate-limit recovery and no-duplicate delivery before merge.

Yes for the original source-level problem: current main has Feishu send paths with no retry for thrown `230020` errors on reply/streaming-card paths, and the linked issue reports message loss under rate limits. I did not run a live Feishu reproduction in this read-only review.

## Solution Assessment

No. Retrying `230020` is the right direction, but the best fix needs stable Feishu `uuid` reuse on every retried create/reply send path so retrying does not trade message loss for duplicate delivery.
