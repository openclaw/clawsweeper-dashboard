## Summary

Keep this PR open: it addresses a real Feishu message-loss path, but the current retry implementation is not merge-ready because retried create/reply/card sends do not reuse Feishu's idempotency `uuid`, and the live proof should be refreshed with redacted identifiers after that repair.

## What This Changes

The PR adds bounded retry/backoff for Feishu `230020` send rate-limit errors across direct text sends, reply sends, and initial streaming-card sends, plus retry/concurrency tests.

## Best Possible Solution

Land the retry behavior only after each logical Feishu send generates and reuses one stable `uuid` across retry attempts, with focused tests and refreshed redacted live Feishu proof for direct and reply sends.

## Reproduction Assessment

Yes at source level: current main has no retry in `requestFeishuApi`, and the linked issue plus PR proof show live Feishu `230020` send failures. I did not run a live Feishu account in this read-only review.

## Solution Assessment

No. Retrying `230020` in the Feishu plugin is the right area, but the best fix must make retries idempotent with stable `uuid` values before it is safe to merge.
