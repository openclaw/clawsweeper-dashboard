## Summary

## What This Changes

The branch adds bounded retry/backoff for Feishu send rate-limit code 230020, routes reply and streaming-card sends through the shared helper, and adds retry/concurrency tests.

## Best Possible Solution

Land the bounded 230020 retry only after Feishu owners accept the delivery-semantics tradeoff and normal CI is green, keeping retries constrained to explicit Feishu frequency-limit responses.

## Reproduction Assessment

Yes, by source inspection current main throws Feishu send errors without retry, and the PR body supplies live direct/reply stress output that triggers 230020 and succeeds after the patch. I did not rerun the live Feishu scenario in this read-only review.

## Solution Assessment

Yes, this is an acceptable mitigation: it targets the documented 230020 frequency-limit response with bounded retries and keeps non-rate-limit codes such as 230006 fail-fast. The safer alternative, if maintainers need stronger duplicate protection, is adding Feishu uuid/idempotency or per-chat queueing before merge.
