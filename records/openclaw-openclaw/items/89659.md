## What This Changes

This PR adds bounded linear-backoff retry handling for Feishu 230020 send rate-limit Axios errors, wraps direct/reply/card send paths through the helper, and adds retry/concurrency tests.

## Best Possible Solution

Land this after Feishu-owner review accepts bounded shared-helper retry as the delivery policy; track durable queueing or stricter backpressure separately if maintainers want stronger guarantees.

## Reproduction Assessment

Yes for source-level reproduction: current main sends Feishu message/reply calls without retrying 230020, and official Feishu docs classify 230020 as a frequency-limit error. I did not run a live Feishu stress test in this read-only review, but the PR body supplies live after-fix terminal proof.

## Solution Assessment

Mostly yes: a small bounded retry in the Feishu helper is an acceptable maintainable mitigation for transient 230020 errors across SDK send paths. Durable queuing or stronger backpressure would be a larger product decision, not a narrow repair to this PR.
