## Summary

## What This Changes

The PR adds bounded retry/backoff for Feishu `230020` rate-limit errors in `requestFeishuApi`, routes reply and streaming-card sends through it, and adds retry/concurrency tests.

## Best Possible Solution

Land the bounded retry only after Feishu owners accept the shared text/media/card send scope and the non-durable two-retry budget; use a narrower send-only helper if that scope is not intended.

## Reproduction Assessment

Yes, at source level: current main has no retry in `requestFeishuApi`, leaves reply/card sends as one-shot SDK calls, and the linked report describes lost sends on Feishu rate limits. I did not run a live Feishu reproduction in this read-only review.

## Solution Assessment

Unclear as a final product answer, but acceptable as a bounded mitigation: central retry is maintainable, yet broader than plain text sends and not a durable queue. Feishu owner approval should decide whether this scope and budget are the best fix.
