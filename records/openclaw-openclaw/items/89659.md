## Summary

## What This Changes

The PR adds bounded retry/backoff for Feishu 230020 send rate-limit errors by changing `requestFeishuApi`, routing text replies and streaming-card sends through it, and adding retry/concurrency tests.

## Best Possible Solution

Land a scoped Feishu send retry that is opt-in for proven message create/reply paths, or explicitly prove the broader media/card scope, with redacted real proof and linked issue wording aligned to any remaining rate-limit work.

## Reproduction Assessment

Yes for the reported direct/reply Feishu 230020 send failures via the PR's live stress-output, but I did not independently run live Feishu. Current main and v2026.6.1 lack retry in `requestFeishuApi`.

## Solution Assessment

No as submitted. Retrying 230020 is the right direction, but putting it in the shared helper is broader than the proved behavior; the best fix is opt-in send retry or explicit proof for every affected path.
