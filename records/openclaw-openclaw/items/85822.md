## What This Changes

The issue reports Discord replies being held for roughly 48 seconds after `embedded run done` because silent post-run lane work still blocks delivery.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85829 or an equivalent fix that makes post-run auth-success bookkeeping non-reply-blocking with diagnostics, then close this issue after the fix reaches `main`.

## Reproduction Assessment

No; this review did not run a live Discord reproduction. The reporter supplied concrete 2026.5.20 logs, and current `main` provides a credible source-level reproduction path by delaying or contending `markAuthProfileSuccess` after `embedded run done`.

## Solution Assessment

Yes, the narrow maintainable direction is to instrument and de-risk post-run finalization so reply delivery is not held by best-effort auth-success bookkeeping. Stuck-session recovery is not the best fix for this gap because the lane eventually completes and the blocking work is still on the normal success path.
