## Summary

Keep open. ## What This Changes

The issue reports Discord/Codex turns where the session lane remains held for roughly 48 seconds after embedded-run completion because post-run auth/OAuth bookkeeping can block reply delivery without trace diagnostics.

## Best Possible Solution

Review and land, revise, or close the linked nonblocking auth-success PR; close this issue only after main contains an accepted repair for the reply-blocking post-run path.

## Reproduction Assessment

Yes at source level: current main and `v2026.6.8` both await auth-profile success bookkeeping after `embedded run done`, and the issue comments provide production Discord timing for the same gap. I did not run a live Discord repro in this read-only cleanup review.

## Solution Assessment

Yes, the linked PR is the best narrow repair if maintainers accept the tradeoff: it moves post-run auth-success bookkeeping off the reply path and adds slow/failure diagnostics. The stale-lock fix alone is not enough because the runner still awaits the bookkeeping on current main.
