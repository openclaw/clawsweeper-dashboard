## Label Justifications

- `P1`: The report describes every Discord DM turn in the current release taking about 50-65 seconds despite the model call finishing much earlier, which is an urgent channel workflow regression.
- `impact:message-loss`: Replies are not delivered to Discord while the lane remains active after the embedded run completes.
- `impact:session-state`: The observed delay is tied to session/global lane retention and diagnostic session state after run completion.

## Summary

Keep open: current main still has a credible source-level path for the reported silent post-run delay. The embedded runner logs `embedded run done` and then awaits auth-profile success bookkeeping while the session/global lane task is still active; that bookkeeping waits on the auth-store file lock with a retry budget that can plausibly match the observed gap, and the existing stuck-session recovery intentionally does not release active reply/lane work.

## What This Changes

The issue reports Discord DM replies on v2026.5.20 being held for roughly 48 seconds after `embedded run done` and before `lane task done`, with no trace log activity during the gap.

## Best Possible Solution

Add explicit post-run finalization timing and make non-critical auth-profile success bookkeeping bounded or non-reply-blocking, while tracking the bubblewrap stderr as a separate Codex plugin follow-up if it still reproduces.

## Reproduction Assessment

No live high-confidence reproduction was run in this review. The reporter gives a concrete v2026.5.20 Discord DM reproduction, and current source inspection provides a credible reproducible path by delaying or contending `markAuthProfileSuccess` after `embedded run done`.

## Solution Assessment

No, the current stuck-session recovery is not the best solution for this specific gap because it intentionally preserves active reply/lane work. The narrower fix is to instrument and de-risk post-run finalization so reply delivery is not held by best-effort auth bookkeeping.
