## Summary

Keep open: current main still contains the source-level unsafe path where raw task completion results can be used as direct external fallback content, and the write-ahead queue still persists payloads before final channel normalization. The related open reports cover broader channel/text leaks, but this issue adds distinct Telegram plus durable retry evidence that is not superseded.

## What This Changes

The issue requests hardening so Telegram subagent completion fallback and delivery recovery cannot store or send raw child/internal context after mediated announce failure.

## Best Possible Solution

Fail closed across the fallback and queue layers: never direct-deliver raw `event.result`, strip or reject internal fences before persistence/retry, and cover mediated-failure paths with regression tests.

## Reproduction Assessment

Yes at source level: inject a `task_completion` event whose `result` contains internal delimiters, force the requester announce call to fail or return no visible output, and current main routes the extracted result to direct fallback send. I did not run a live Telegram send in this read-only review.

## Solution Assessment

Yes, the report's defense-in-depth direction is the right shape; a Telegram-only sanitizer would be too narrow because raw fallback selection and pre-normalization queue persistence both need fail-closed handling.
