## Summary

Keep this issue open. Current main still has source-backed evidence that Discord native commands and Slack slash commands pass provider-startup config into route resolution while `session.dmScope` is the documented session-key contract; Telegram already uses a fresh runtime config path, and open PR https://github.com/openclaw/openclaw/pull/39617 still references this issue with closing syntax.

## What This Changes

The issue asks slash/native command routing in Discord, Telegram, and Slack DMs to respect fresh `session.dmScope`-aware config instead of a provider-startup snapshot.

## Best Possible Solution

Land, retarget, or replace https://github.com/openclaw/openclaw/pull/39617 with a narrow plugin-owned fix that refreshes route-sensitive config for Discord and Slack native command routing, preserves Telegram’s fresh-runtime path, and adds focused DM session-key regression coverage.

## Reproduction Assessment

Yes, by source inspection. With a non-main `session.dmScope`, `resolveAgentRoute` will compute a peer-scoped DM key only if the slash/native command path passes a fresh config object, and current Discord and Slack paths still pass captured startup config.

## Solution Assessment

Yes for the issue direction, but not for closing it now. The maintainable solution is a small channel-owned route-config refresh for Discord and Slack plus regression coverage, while the existing linked PR needs retargeting or replacement before it can land.
