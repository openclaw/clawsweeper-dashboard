## Summary

Keep open: the underlying routing bug still exists on current main, but this PR edits an obsolete iMessage helper body instead of the shared parser now used by the live path.

## What This Changes

The PR branch adds decimal-only guards to the old iMessage `chat_id` parsing helper plus iMessage regressions for hex `chat_id` rejection and allowlist false-match prevention.

## Best Possible Solution

Port the decimal-only `chat_id` validation to `src/channels/plugins/chat-target-prefixes.ts`, keep invalid allowlist entries as non-matches, use neutral send-target error text, and add focused shared/iMessage regressions.

## Reproduction Assessment

Yes. `Number.parseInt` reproduces the truncation to `2`, and current main still uses that parse-and-finite-check pattern in both shared send and allowlist parser paths.

## Solution Assessment

No. Strict decimal validation is the right direction, but this branch applies it to the pre-refactor helper; the maintainable fix point is the shared channel-target parser used by the current iMessage path.
