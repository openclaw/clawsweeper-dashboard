## Summary

Keep open: the underlying iMessage routing bug is valid and current main still has the parse-and-finite-check behavior in the shared parser, but this PR now targets an obsolete helper body and lacks real iMessage after-fix proof before merge.

## What This Changes

The PR branch adds decimal-only `chat_id:` guards in `extensions/imessage/src/target-parsing-helpers.ts` and iMessage regression tests for hex rejection plus allowlist false-match prevention.

## Best Possible Solution

Port the decimal-only `chat_id:` validation to `src/channels/plugins/chat-target-prefixes.ts`, keep invalid allowlist `chat_id:` entries as non-matches, use neutral invalid-target guidance, and add focused shared/iMessage regression coverage.

## Reproduction Assessment

Yes, by source inspection: current main still feeds `chat_id:` values into `Number.parseInt(value, 10)` in both shared parser paths, and the reported identifier parses to `2`. I did not run a full iMessage setup in this read-only review.

## Solution Assessment

No. Strict decimal validation is the right fix direction, but this branch applies it to the old iMessage helper body; the maintainable fix point is the current shared channel-target parser.
