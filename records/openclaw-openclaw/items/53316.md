## Summary

Keep open. Current main still has the `parseInt` truncation bug and this PR is useful source work, but the branch is not mergeable as-is because it edits the old iMessage-local helper body rather than the shared parser now used by iMessage and BlueBubbles.

## What This Changes

The PR adds decimal-only guards to iMessage `chat_id` target and allowlist parsing, plus regression tests for hex `chat_id` rejection and allowlist false-match prevention.

## Best Possible Solution

The end state should be shared-parser validation that treats `chat_id` as decimal-only across iMessage and BlueBubbles while preserving `chat_guid` and `chat_identifier` behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: the sample hex identifier parses to `2` with `Number.parseInt`, and current main feeds both send and allowlist `chat_id` values through that parse-and-finite-check path.

## Solution Assessment

No. The PR's intended validation is right, but the implementation is no longer the best fix because current main requires the guard in `src/channels/plugins/chat-target-prefixes.ts`, not the old iMessage-local helper body.
