## Summary

Keep open. The reported misrouting bug is still present on current main and in the latest release, but this PR targets the old iMessage-local helper while current iMessage and BlueBubbles both route through the shared chat-target parser, so the branch should not merge as-is.

## What This Changes

The PR adds decimal-only validation to iMessage-local `chat_id` target and allowlist parsing, plus iMessage regression tests for rejecting hex `chat_id` values and preventing allowlist false matches.

## Best Possible Solution

Land a narrow shared-parser fix in `src/channels/plugins/chat-target-prefixes.ts` that requires decimal-only `chat_id` values before parsing, throws a neutral invalid-`chat_id` error for send targets, silently rejects invalid allowlist `chat_id` entries, preserves `chat_guid` and `chat_identifier` behavior, and adds shared/iMessage/BlueBubbles regression coverage plus a user-facing changelog entry with reporter and contributor credit.

## Reproduction Assessment

Yes. There is a high-confidence static reproduction: Node truncates the sample hex identifier to `2`, and current main still passes the result through both shared send and allowlist parser paths without validating the full string.

## Solution Assessment

No. The PR's direction is correct, but it is not the best current implementation because the live iMessage and BlueBubbles paths now use the shared parser rather than the old iMessage-local helper body.
