## Summary

Keep open: this is a valid iMessage routing/privacy bug, but the PR branch no longer fixes current main because it adds the guard to the old iMessage helper body while current iMessage parsing flows through the shared channel-target parser that still uses `Number.parseInt` without decimal-only validation.

## What This Changes

The PR branch adds decimal-only `chat_id:` validation to an iMessage target parsing helper and adds regression tests for hex identifiers and allowlist false matches.

## Best Possible Solution

Land a narrow shared-parser fix in `src/channels/plugins/chat-target-prefixes.ts` that requires decimal-only `chat_id:` values before parsing, silently rejects invalid allowlist entries, uses neutral invalid-target guidance, and covers the shared/iMessage regression path.

## Reproduction Assessment

Yes. `Number.parseInt("2ecba0e20f3a4299b0efb228a5990731", 10)` returns `2`, and current main still uses that parse-and-finite-check pattern in the shared send and allowlist parser paths.

## Solution Assessment

No. Strict decimal validation is the right direction, but this branch applies it to the obsolete iMessage helper body; the maintainable fix point is the current shared channel-target parser.
