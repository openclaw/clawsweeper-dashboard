## Summary

Keep open. The underlying iMessage `chat_id:` truncation bug still exists on current `main` and the latest stable tag, but this external PR now targets an obsolete helper body and lacks real behavior proof from an iMessage setup.

## What This Changes

The PR adds decimal-only guards for iMessage `chat_id:` parsing and regression tests for hex rejection plus allowlist false-match prevention.

## Best Possible Solution

Land a narrow shared-parser fix in `src/channels/plugins/chat-target-prefixes.ts` that requires decimal-only `chat_id:` values, silently rejects invalid allowlist `chat_id:` entries, adds focused regression coverage, and leaves changelog credit for maintainer landing.

## Reproduction Assessment

Yes, by source inspection: current `main` still feeds `chat_id:` values into `Number.parseInt(value, 10)` in both shared parser paths, and the reported identifier parses to `2`. I did not run a full iMessage setup in this read-only review.

## Solution Assessment

No. Strict decimal validation is the right fix direction, but this branch applies it to the old iMessage helper body; the maintainable fix point is the current shared channel-target parser with neutral invalid-target guidance.
