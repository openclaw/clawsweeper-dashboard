## Summary

Keep open: the reported routing bug is still present on current main by source inspection, but this PR patches the old iMessage helper body instead of the shared parser now used by iMessage. The fix direction is valid, but the branch needs real behavior proof and a port/rebase or replacement shared-parser patch before merge.

## What This Changes

The PR branch adds decimal-only guards for iMessage chat_id parsing and regression tests for hex chat_id rejection plus allowlist false-match prevention.

## Best Possible Solution

Land a narrow shared-parser fix in `src/channels/plugins/chat-target-prefixes.ts` that requires decimal-only `chat_id:` values before parsing, keeps invalid allowlist entries as non-matches, adds focused shared/iMessage regression coverage, and lets maintainers handle the user-facing changelog entry.

## Reproduction Assessment

Yes, by source inspection and a focused parseInt check: current main passes `chat_id:` values to `Number.parseInt(value, 10)` in both send and allowlist paths, and Node parses the reported leading-digit hex identifier as `2`. I did not run a real iMessage send in this read-only review.

## Solution Assessment

No. Strict decimal validation is the right fix direction, but this branch applies it to the old iMessage helper body; the maintainable fix point is the current shared channel-target parser with neutral invalid-target guidance.
