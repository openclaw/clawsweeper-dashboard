## Summary

Keep this PR open. The sessionKey normalization direction matches the reported bug, but the patch also turns the literal agent id `"null"` into a clear/unset operation even though current agent-id normalization accepts `null` as a valid id, and the PR still lacks real behavior proof beyond unit tests.

## What This Changes

The PR changes cron input normalization so literal string `"null"` becomes `null` for `agentId` and `sessionKey`, with normalization tests for create and patch inputs.

## Best Possible Solution

Keep the string `"null"` clear workaround focused on the reported sessionKey path, or make reserving `null` as an invalid agent id an explicit documented and tested compatibility decision before merging.

## Reproduction Assessment

Yes by source inspection: current main exposes string-only cron tool fields, preserves `"null"` as a non-empty string in normalization, and stores non-empty string session keys in the cron service. I did not run a live gateway round-trip in this read-only review.

## Solution Assessment

No as written: normalizing the model-produced `sessionKey: "null"` sentinel is the narrow fix, but applying the same sentinel to `agentId` conflicts with the current valid agent-id contract. The safer implementation either leaves `agentId` strings alone or explicitly reserves that id with maintainer approval and coverage.
