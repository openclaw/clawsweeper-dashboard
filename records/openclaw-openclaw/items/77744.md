## Summary

Keep open: current main still has a source-level path matching the reported Docker/llama.cpp timeout, because explicit provider timeouts are clamped to the 120s idle watchdog unless the base URL is classified as local or the run is cron-triggered; the reporter's bare hostname is not classified local.

## What This Changes

The issue asks OpenClaw to honor the supported `models.providers.<id>.timeoutSeconds` setting for long local llama.cpp prefills after the legacy `agents.defaults.llm.idleTimeoutSeconds` key was removed.

## Best Possible Solution

Keep the provider-scoped timeout contract and make explicit provider request timeouts reach the model idle watchdog for self-hosted/local OpenAI-compatible providers without restoring the retired `agents.defaults.llm` key.

## Reproduction Assessment

Yes for the code-level mismatch: with `modelRequestTimeoutMs=14400000` and a base URL like `http://cerebro-mac:8080/v1`, the current resolver follows the non-local branch and caps the idle watchdog at 120s. I did not run the full Docker plus 150k-token llama.cpp prefill.

## Solution Assessment

Yes. The narrow maintainable fix is to align the existing `models.providers.<id>.timeoutSeconds` contract with the idle watchdog for this self-hosted provider path, not to reintroduce `agents.defaults.llm` or add a new timeout hierarchy here.
