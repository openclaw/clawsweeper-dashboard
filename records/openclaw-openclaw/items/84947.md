## Summary

Keep open: the plugin contract direction fits the plugin boundary and the PR supplies real happy-path proof, but the patch introduces a process-global registry rollback gap for the new embedding providers.

## What This Changes

The branch adds a general embedding provider plugin contract, registry, SDK subpath/export, loader/status metadata plumbing, tests, docs, and changelog entry.

## Best Possible Solution

Keep the contract direction, but make generic embedding providers follow the same loader snapshot and rollback discipline as memory embedding providers before merging the public SDK subpath.

## Reproduction Assessment

Yes, source-reproducible: the new registration path writes process-global embedding provider state, while the loader rollback paths only restore memory embedding provider state. A snapshot or failed plugin load can therefore leave the new registry populated after the registry itself is not activated or is rolled back.

## Solution Assessment

No, not yet: the contract shape is plausible, but the implementation needs the same global-state restore discipline as memory embedding providers. The safer path is to fix that rollback gap and add regression coverage before reviewing the public API shape for merge.
