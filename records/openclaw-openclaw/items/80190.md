## Summary

Keep open, but do not merge as written. Current main already accepts `hooks.allowConversationAccess` in host config and treats it as the raw-conversation hook trust gate; this PR would make that gate mutable through the agent-facing Gateway tool allowlist.

## What This Changes

The PR adds `plugins.entries.*.hooks.allowConversationAccess` to the agent Gateway config mutation allowlist and adds guard tests for setting and toggling it through `config.patch`.

## Best Possible Solution

Preserve or restore local plugin hook policy through upgrade/doctor/config-management paths, or make an explicit maintainer-approved human-operated path, without letting the agent Gateway allowlist grant raw-conversation access by default.

## Reproduction Assessment

Yes, by source inspection: current main lacks this path in the Gateway tool allowlist, so changing `plugins.entries.<id>.hooks.allowConversationAccess` through the agent `config.patch` path reaches the protected-path rejection before the write is forwarded.

## Solution Assessment

No. The PR fixes the reconfiguration symptom by widening the model-to-operator trust boundary; a safer fix should target upgrade/config preservation or another explicit host-controlled approval path.
