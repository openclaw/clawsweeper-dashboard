## Summary

Keep open: the reported guard-path failure is source-reproducible, but this PR fixes it by letting agent-driven `config.patch` toggle a raw-conversation hook permission that current docs and code treat as a model-to-operator trust boundary.

## What This Changes

The branch adds `plugins.entries.*.hooks.allowConversationAccess` to the agent gateway tool config-patch allowlist and adds guard coverage for allowing that field.

## Best Possible Solution

Keep `allowConversationAccess` as explicit host/operator approval, and solve the upgrade/config-patch pain through an operator-confirmed config path or upgrade preservation rather than an unrestricted agent allowlist entry.

## Reproduction Assessment

Yes, by source inspection. Current main omits the plugin hook path from the agent gateway tool allowlist, and the guard rejects any changed config path outside that list.

## Solution Assessment

No. The patch fixes the symptom by allowlisting a raw-conversation access trust bit for agent-driven `config.patch`; a safer solution would preserve or set host config through an explicit operator approval path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep conversation hook access out of the agent allowlist:** `src/agents/tools/gateway-tool.ts:65`
  - body: `allowConversationAccess` is the host approval bit that lets non-bundled plugins receive raw conversation content through hooks such as `agent_end`, `llm_input`, and `llm_output`. The gateway tool guard is explicitly the model-to-operator boundary for agent-driven config writes, so adding this path lets a prompt-influenced agent grant a plugin transcript access through a surface designed to fail closed. Leave this protected or add a separate human-confirmed approval flow.
  - confidence: 0.88
