## Summary

Keep open and do not merge as written: the PR addresses a real `gateway config.patch` denial, but it does so by letting the agent-facing Gateway tool grant raw-conversation hook access to non-bundled plugins, which is a security boundary change needing maintainer judgment.

## What This Changes

The PR adds `plugins.entries.*.hooks.allowConversationAccess` to the agent Gateway tool config mutation allowlist and adds guard tests for setting and toggling it through `config.patch`.

## Best Possible Solution

Preserve or restore local plugin hook policy through upgrade/doctor/config management, or add an explicitly maintainer-approved human-operated approval path, without making raw-conversation access agent-mutable by default.

## Reproduction Assessment

Yes. Source inspection shows current main and the latest release lack this path in `ALLOWED_GATEWAY_CONFIG_PATHS`, so a `config.patch` changing `plugins.entries.<id>.hooks.allowConversationAccess` reaches the protected-path rejection before the RPC write is forwarded.

## Solution Assessment

No. The PR fixes the denial symptom, but it changes the model-to-operator trust boundary for a raw-conversation access gate; the safer solution is upgrade/config preservation or an explicit human-operated config path.
