## Summary

Keep open: current main still applies the implicit floor to reserveTokens-only configs, and this PR is a focused attempt to align the affected runtime, memory, Codex, docs, and tests. I found no blocking correctness or security defect, but maintainers should consciously accept the compatibility and availability tradeoff for existing reserveTokens-only setups before merging.

## What This Changes

The PR changes compaction budget resolution so explicit agents.defaults.compaction.reserveTokens is honored when reserveTokensFloor is omitted, and updates agent, memory-core, Codex, tests, and docs accordingly.

## Best Possible Solution

Align all compaction budget resolvers around one documented contract: reserveTokens is authoritative when set, reserveTokensFloor is only an explicit lower bound, and overlapping compaction-budget PRs should converge on that same semantics before merge.

## Reproduction Assessment

Yes. Current main is source-reproducible: applyAgentCompactionSettingsFromConfig, Codex projection, and Codex startup binding all raise reserveTokens-only configs through the implicit 20000 floor, and the PR changes focused tests to cover the new expectation.

## Solution Assessment

Yes, if maintainers accept the compatibility shift. The PR touches the relevant owner surfaces and tests; the safer alternative is to preserve current default semantics and add an explicit opt-in or migration if the hidden floor remains a shipped contract.
