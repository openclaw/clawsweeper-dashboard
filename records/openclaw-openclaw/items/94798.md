## Summary

Keep this PR open for maintainer review: the patch is focused and covers the sibling runtime paths, but it changes shipped compaction config semantics and the supplied terminal proof still does not execute a resolver or agent/compaction path that prints the effective reserve.

## What This Changes

This PR changes compaction reserve resolution so explicit `agents.defaults.compaction.reserveTokens` bypasses the implicit 20,000-token floor when `reserveTokensFloor` is omitted, across agent settings, memory flush, Codex projection/startup, tests, and docs.

## Best Possible Solution

Preserve the shipped implicit-floor behavior by default unless maintainers explicitly approve the omitted-floor semantic change; if approved, require runtime proof that prints the effective reserve and PR-body release-note context for the upgrade-visible behavior.

## Reproduction Assessment

Yes, at source level: current main and `v2026.6.10` raise explicit `reserveTokens` by the implicit floor when `reserveTokensFloor` is omitted. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet: the implementation is plausible and sibling-aware, but the current documented solution is `reserveTokensFloor: 0`, so changing omitted-floor semantics needs maintainer approval and stronger effective-value runtime proof.
