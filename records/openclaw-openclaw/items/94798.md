## Summary

Keep open: the PR is a focused attempt to change compaction reserve semantics, but it currently bypasses the existing small-context reserve-floor cap when an explicit floor is configured and has no real behavior proof.

## What This Changes

The PR changes compaction default handling so an explicit `agents.defaults.compaction.reserveTokens` is not raised by the implicit `reserveTokensFloor` default, and updates related docs and tests.

## Best Possible Solution

Preserve the current small-context cap for any configured floor, add coverage for omitted versus explicit floors, and require real upgrade/runtime proof before accepting the changed default semantics.

## Reproduction Assessment

Yes, source-level: with a small `contextTokenBudget`, an explicit `reserveTokens`, and an explicit `reserveTokensFloor` above the safe max, the PR's new branch uses the uncapped explicit floor and can return a reserve larger than the current code's capped limit.

## Solution Assessment

No. Respecting an omitted floor may be plausible, but the implementation should still use the context-capped floor when `reserveTokensFloor` is explicitly configured and should prove the config compatibility change in a real setup.
