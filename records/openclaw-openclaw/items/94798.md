## Summary

Keep this PR open, but it is not merge-ready: the patch has a concrete P1 cap regression, changes upgrade-sensitive compaction defaults, and still has no real behavior proof from the external contributor.

## What This Changes

The branch changes compaction reserve handling so explicit `agents.defaults.compaction.reserveTokens` is honored when `reserveTokensFloor` is omitted across core settings, Codex projection, memory flush, docs, and tests.

## Best Possible Solution

Keep the omitted-floor behavior only if explicit floors still pass through the existing context-window cap, add a regression test for a too-high explicit floor, and require redacted runtime proof plus maintainer compatibility approval before merge.

## Reproduction Assessment

Yes, source-level: with `contextTokenBudget: 16_384`, `reserveTokens: 2_048`, and `reserveTokensFloor: 20_000`, the PR branch returns the uncapped floor instead of the existing safe cap of 8,384. I did not run a live OpenClaw repro because this review is read-only and the source path is direct.

## Solution Assessment

No. Respecting an omitted floor may be a reasonable compatibility direction, but the implementation should still use the context-capped floor when `reserveTokensFloor` is explicitly configured and should prove the upgrade behavior in a real setup.
