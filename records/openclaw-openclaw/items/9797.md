## Summary

Keep open. Current main and the latest tagged release still lack the requested agent-facing `queue_status` tool or equivalent stable queue snapshot surface; the existing code has internal lane snapshots plus logs/metrics, but no registered tool, no `diagnostics.queue` Gateway RPC, and no CLI diagnostics queue command. The related #43618 work remains relevant but, per the provided PR context, explicitly does not add the in-agent tool requested here.

## What This Changes

The issue requests an agent-facing `queue_status` tool that exposes queue depth, active run counts, and per-agent workload so dispatcher agents can route work to idle agents or lanes.

## Best Possible Solution

Keep this issue as the canonical in-agent dispatch surface request. The maintainable end state is a shared read-only queue snapshot provider in core, reused by operator diagnostics where applicable, with a `queue_status` agent tool registered through the catalog/profile/policy system and documented/test-covered visibility rules.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. The current-main verification path is high-confidence: source search plus agent-tool, catalog, Gateway method, docs, and release-tag inspection show the requested surface is absent.

## Solution Assessment

Yes, the requested direction is plausible, but the best implementation should not be a bespoke agent-only dump of raw lane internals. It should reuse or define a shared queue snapshot contract, then expose only the safe subset through an agent tool with explicit permissions.
