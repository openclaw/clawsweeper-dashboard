## Summary

Keep this open: current main still lacks an agent-facing `queue_status` tool or equivalent stable queue snapshot surface for dispatch decisions, and the related operator diagnostics PR is open and explicitly out of scope for the in-agent tool.

## What This Changes

The issue requests a `queue_status` tool exposing lane depth, active concurrency, and per-agent workload so dispatcher agents can route work intelligently.

## Best Possible Solution

Build a shared read-only queue snapshot provider, then expose it through an agent-facing `queue_status` tool with catalog/profile/policy wiring, docs, and focused tests; reuse https://github.com/openclaw/openclaw/pull/43618 if it lands first.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high-confidence proof that current main does not yet expose the requested agent-facing queue snapshot surface.

## Solution Assessment

Unclear until maintainers approve the stable API and privacy contract. The narrowest maintainable direction is a shared snapshot backend with an agent-facing tool rather than asking agents to infer load from `agents_list`, `sessions_list`, or logs.
