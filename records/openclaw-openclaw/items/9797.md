## Summary

Keep open: current main and v2026.6.10 still lack an agent-facing `queue_status` tool or equivalent stable workload/capacity surface. Internal lane snapshot helpers now exist, but they are not exposed through the agent tool catalog, Gateway queue diagnostics, CLI, or docs, and the open item remains the canonical queue-visibility tracker.

## What This Changes

Request an agent-facing `queue_status` tool that exposes lane queue depth, active versus maximum concurrency, and per-agent workload so agents can route tasks intelligently with `sessions_spawn`.

## Best Possible Solution

Keep this issue as the canonical queue-visibility tracker and implement a maintainer-approved shared read-only queue snapshot provider, then expose it through a policy-gated `queue_status` agent tool with catalog/profile/docs/tests and carefully bounded metadata.

## Reproduction Assessment

Yes, as a source-level feature gap: current main has internal lane snapshots but no registered `queue_status`, `sessions_capacity`, or `diagnostics.queue` surface. I did not run a live workload because the requested public API does not exist.

## Solution Assessment

Unclear until maintainers choose the public metadata contract. The likely best shape is a shared snapshot backend reused by an agent-facing tool and any future operator surface, with explicit read-only permissions and redaction rules.
