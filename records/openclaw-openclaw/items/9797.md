## Summary

Keep open: current main and the latest release still lack an agent-facing `queue_status` or equivalent capacity/workload tool, and the related queue diagnostics PR was closed unmerged after explicitly scoping out the in-agent surface.

## What This Changes

Request to add an agent-facing `queue_status` tool exposing lane capacity, queued/active work, and per-agent workload so dispatcher agents can route work intelligently.

## Best Possible Solution

Define one shared read-only queue snapshot provider, then expose the smallest agent-safe workload/capacity facts through the maintainer-chosen tool, catalog, policy, docs, and tests.

## Reproduction Assessment

Yes, at source level: current main has internal lane snapshot helpers, but the supported agent tool assembly, tool catalog, Gateway diagnostics surface, and latest release have no queue or capacity status tool. This is a missing feature/API surface, not a broken existing contract.

## Solution Assessment

Unclear: a dedicated `queue_status` tool is plausible, but maintainers should first choose whether the minimal capacity facts belong in a new tool, `session_status`, or structured `sessions_spawn` responses and define visibility rules.
