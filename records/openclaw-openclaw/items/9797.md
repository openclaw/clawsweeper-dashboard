## Summary

Current main and the latest release contain the shared all-lane queue snapshot primitive, but neither exposes queue or concurrency capacity to agents; the closed, unmerged operator diagnostics PR explicitly excluded the requested in-agent surface, so meaningful product work remains.

## What This Changes

Expose queue depth, active concurrency, and caller-visible workload capacity to agents so they can preflight intelligent task dispatch.

## Best Possible Solution

Expose a minimal permission-filtered capacity view over `getCommandLaneSnapshots()` and canonical subagent runtime facts, preferably through the existing session/subagent management surface unless a separate tool provides a clearly cleaner contract.

## Reproduction Assessment

Yes at source level: current main exposes all-lane queue snapshots internally, while the agent tool catalog and policy have no queue or capacity preflight surface under the requested or synonymous names.

## Solution Assessment

The shared snapshot backend is the best foundation, but the exact placement, per-agent semantics, and caller visibility rules require maintainer product judgment before implementation.
