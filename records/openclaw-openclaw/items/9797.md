## Summary

Keep open: current main still lacks the requested agent-facing `queue_status` tool or equivalent stable queue snapshot API for dispatch decisions. Internal lane snapshot helpers now exist, but they are used by runtime diagnostics/recovery paths only and are not wired into the agent tool catalog, Gateway diagnostics API, CLI queue diagnostics, or docs as a stable user-facing surface.

## What This Changes

Add an agent-facing `queue_status` tool that exposes lane and per-agent workload state so agents can dispatch subagent work intelligently.

## Best Possible Solution

Define one shared read-only queue snapshot backend, then expose a policy-gated `queue_status` agent tool with catalog/profile registration, docs, and focused tests after maintainers settle schema and visibility rules.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/API request. Source inspection gives high confidence the requested public surface is absent: search found no `queue_status` or `diagnostics.queue`, while only internal lane snapshot helpers exist.

## Solution Assessment

Unclear until maintainers approve the agent-visible API shape and metadata visibility rules. The maintainable direction is a shared snapshot provider plus a policy-gated tool, not inference from `agents_list`, `sessions_list`, logs, or raw queue internals.
