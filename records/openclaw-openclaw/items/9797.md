## Summary

Keep open. Current main still lacks the requested agent-facing `queue_status` tool or equivalent stable queue snapshot surface; existing queue snapshots, diagnostics logs, and telemetry remain internal/operator-adjacent rather than a dispatch tool for agents.

## What This Changes

The issue requests a new agent-facing `queue_status` tool that exposes queue/lane and per-agent workload status for smarter task dispatch.

## Best Possible Solution

Keep this as the canonical in-agent dispatch request: define or reuse a shared read-only queue snapshot contract, then expose a safe subset through `queue_status` with catalog/profile/policy wiring, docs, and tests.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. The source verification path is high-confidence: current main lacks the requested tool, catalog entry, Gateway queue RPC, CLI queue diagnostics command, and user-facing docs for a stable queue snapshot.

## Solution Assessment

Yes, the requested direction is plausible, but the best implementation should not dump raw lane internals directly to agents. It should coordinate with #43618, settle the snapshot contract and visibility model, then expose the agent tool through the existing tool policy system.
