## Summary

Keep this open. Current main still lacks the requested agent-facing `queue_status` tool or equivalent stable dispatcher queue snapshot surface; the related queue diagnostics work is still an open operator/RPC/CLI PR and explicitly does not add the in-agent tool.

## What This Changes

The issue requests an agent-facing `queue_status` tool that exposes lane and per-agent workload data for dispatcher-style routing.

## Best Possible Solution

Define one shared read-only queue snapshot contract, then expose `queue_status` through the agent tool catalog/profile/policy surface with docs and focused tests, reusing the operator diagnostics backend from https://github.com/openclaw/openclaw/pull/43618 if it lands first.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection on current main gives high-confidence evidence that the requested stable agent-facing queue-status surface is still missing.

## Solution Assessment

Yes, the requested direction is best solved by a shared read-only queue snapshot backend plus a narrow agent-facing tool. The implementation should not dump raw internal lane state directly or bloat `agents_list` without a maintainer-approved schema and visibility model.
