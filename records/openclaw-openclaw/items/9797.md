## Summary

Keep open: current main and the latest release still have only internal queue/lane state plus log/stability diagnostics, not an agent-facing `queue_status` or equivalent capacity/workload surface for dispatch decisions.

## What This Changes

The issue requests a new agent-facing `queue_status` tool that exposes lane queue depth, active versus maximum concurrency, and per-agent workload so agents can dispatch tasks intelligently.

## Best Possible Solution

Define one shared read-only queue snapshot provider, then expose the smallest agent-safe capacity/workload facts through the chosen tool surface and existing catalog/policy system without adding core dispatch policy.

## Reproduction Assessment

Yes, at source level: current main has internal lane snapshots, while the supported agent tool catalog and runtime tool assembly have no queue or capacity status surface. This is a missing feature/API contract rather than a failing existing behavior.

## Solution Assessment

Unclear: a dedicated `queue_status` tool is plausible, but maintainers should first choose whether the minimal capacity facts belong in a new tool, `session_status`, or structured `sessions_spawn` responses and define the visibility boundary.
