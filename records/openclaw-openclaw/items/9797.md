## Summary

Keep open: current main and the latest release still lack an agent-facing `queue_status` or equivalent workload/capacity surface, and the related operator-diagnostics PR was closed unmerged while explicitly excluding an in-agent tool.

## What This Changes

Request for a model-facing `queue_status` tool that exposes lane queue depth, active versus maximum concurrency, and per-agent workload so agents can dispatch tasks intelligently.

## Best Possible Solution

Define one shared read-only queue snapshot provider, then expose the smallest agent-safe capacity/workload facts through the chosen tool, catalog, and policy surface.

## Reproduction Assessment

Yes, at source level: current main has internal lane snapshots, while the supported agent tool catalog and runtime tool assembly have no queue or capacity status surface. This is a missing feature/API contract rather than a failing existing behavior.

## Solution Assessment

Unclear: a dedicated `queue_status` tool is plausible, but maintainers should first choose whether the minimal capacity facts belong in a new tool, `session_status`, or structured `sessions_spawn` responses and define visibility rules.
