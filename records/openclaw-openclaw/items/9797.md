## Summary

Keep open: current main and the latest release still lack an agent-facing `queue_status` tool or equivalent capacity/workload surface. Current code has internal lane snapshot helpers, but they are not wired into the agent tool catalog, existing session/agent tools, Gateway queue diagnostics, or a shipped operator command.

## What This Changes

The issue requests a model-facing `queue_status` tool that lets agents inspect queue depth, lane concurrency, and per-agent workload before dispatching work with `sessions_spawn`.

## Best Possible Solution

Implement one shared read-only queue snapshot backend in core, then expose a filtered `queue_status` agent tool with catalog/profile/policy wiring, docs, and focused tests after maintainers settle the metadata and visibility contract.

## Reproduction Assessment

Yes, as a source-level feature gap: current main has no `queue_status`/capacity tool in the agent catalog or registration, and existing `agents_list`, `sessions_list`, and `session_status` outputs do not expose queue depth or active lane counts. I did not run a live agent session because the missing model-facing surface is clear from source and docs inspection.

## Solution Assessment

Unclear pending maintainer product direction: the proposed `queue_status` direction is plausible, but the best implementation should reuse a shared queue snapshot backend rather than invent a separate model-only view. The remaining decision is the stable shape and visibility policy for workload metadata.
