## Summary

Keep open. Current main has internal queue snapshot helpers and queue diagnostics logs, but it still lacks the requested agent-facing `queue_status` tool or an equivalent dispatcher-oriented queue snapshot contract; the related queue diagnostics PR remains open and explicitly excludes the in-agent tool.

## What This Changes

Add an agent-facing `queue_status` tool that exposes lane queue depth, active runs, and per-agent workload so dispatcher agents can route work intelligently.

## Best Possible Solution

Implement one shared read-only queue snapshot backend, then expose `queue_status` through the agent tool factory, catalog, profile, and policy surfaces with permissions, docs, and focused tests, reusing https://github.com/openclaw/openclaw/pull/43618 if it lands first.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection against current main gives high-confidence evidence that the requested stable agent-facing queue-status surface is not present.

## Solution Assessment

Yes at the design level: a shared read-only snapshot provider plus an agent-facing tool is the narrow maintainable path. The existing `agents_list` and `sessions_list` workaround does not expose lane depth or active-run state, and the open operator diagnostics PR explicitly does not add the in-agent tool.
