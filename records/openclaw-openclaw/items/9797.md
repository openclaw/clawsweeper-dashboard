## Summary

Keep open: current main and the latest release still lack an agent-facing `queue_status` or equivalent workload/capacity tool. Internal lane snapshot helpers exist, but they are not exposed through the agent tool catalog, tool registration, Gateway queue diagnostics, or documented session/subagent tools.

## What This Changes

Request an agent-facing `queue_status` tool that reports queue depth, active runs, concurrency limits, and per-agent workload so dispatchers can route `sessions_spawn` work intelligently.

## Best Possible Solution

Implement a shared read-only queue snapshot provider in core, then expose it through a maintainer-approved `queue_status` agent tool with catalog/profile/policy wiring, visibility rules, docs, and focused tests.

## Reproduction Assessment

Yes, as a source-level feature gap: current main has no `queue_status` or equivalent queue-capacity tool in the agent tool catalog, registration path, or Gateway diagnostics RPC. I did not run a live dispatcher workload because the requested tool surface does not exist.

## Solution Assessment

Unclear until maintainers choose the product/API contract. The strongest direction is a shared read-only queue snapshot backend with a narrow agent-facing tool, rather than separate bespoke snapshots for agents, CLI, and UI.
