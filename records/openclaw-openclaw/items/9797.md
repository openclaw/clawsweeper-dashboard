## Summary

Keep open. Current main has internal lane snapshot helpers, but it still lacks the requested agent-facing `queue_status` tool or an equivalent dispatcher-oriented queue snapshot contract; the related diagnostics work remains an open PR and explicitly does not add the in-agent tool.

## What This Changes

Requests a new agent-facing `queue_status` tool that reports lane queue depth, active concurrency, and per-agent workload for dispatcher agents.

## Best Possible Solution

Define or reuse one shared read-only queue snapshot backend, then expose `queue_status` through the agent tool catalog/profile/policy surface with permissions, docs, and focused tests.

## Reproduction Assessment

Not applicable as a feature request. Source, docs, and related-PR inspection on current main give high-confidence evidence that the requested stable agent-facing queue-status surface is absent.

## Solution Assessment

Yes at the design level: a shared read-only snapshot provider plus an agent-facing tool is the narrow maintainable direction. The open diagnostics PR can supply part of the backend if it lands, but it intentionally does not close the agent-tool request.
