Keep this issue open: current main still lacks the requested agent-facing `queue_status` tool or equivalent stable in-agent queue snapshot surface, and the related operator diagnostics work is explicitly scoped as separate backend/operator work.

## What This Changes

Request an agent-facing `queue_status` tool that exposes lane and per-agent workload state so agents can dispatch tasks to less-busy agents.

## Best Possible Solution

Build a shared read-only queue snapshot backend, then expose it through an agent-facing `queue_status` tool with catalog/profile/policy wiring, docs, and focused tests; reuse the operator diagnostics backend if https://github.com/openclaw/openclaw/pull/43618 lands first.

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high-confidence proof that current main lacks the requested agent-facing queue snapshot surface.

## Solution Assessment

Yes, the best direction is a shared snapshot provider plus an agent-facing tool rather than asking agents to infer workload from `agents_list`, `sessions_list`, or logs. The exact fields, visibility, and permissions still need maintainer product/API judgment.
