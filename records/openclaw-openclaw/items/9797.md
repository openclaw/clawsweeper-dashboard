## Summary

Keep open: current main and the latest release still lack an agent-facing `queue_status` or equivalent capacity/workload tool, while the related operator diagnostics PR is closed unmerged and explicitly did not add the in-agent surface.

## What This Changes

Request an agent-facing `queue_status` workload/capacity tool so dispatcher agents can route `sessions_spawn` calls based on live queue and agent load.

## Best Possible Solution

Define one shared read-only queue snapshot provider, then expose the smallest agent-safe workload/capacity facts through the maintainer-chosen tool, catalog, policy, docs, and tests.

## Reproduction Assessment

Yes, at source level: current main has internal lane snapshot helpers, but the supported agent tool assembly, tool catalog, and Gateway diagnostics surface have no queue or capacity status tool. This is a missing feature/API surface, not a broken existing contract.

## Solution Assessment

Unclear: a dedicated `queue_status` tool is plausible, but maintainers should first choose whether the minimal capacity facts belong in a new tool, `session_status`, or structured `sessions_spawn` responses and define visibility rules.
