## Summary

Keep this open: current main still lacks the requested agent-facing queue snapshot tool, and the related operator diagnostics PR is open and explicitly out of scope for in-agent dispatch.

## What This Changes

The issue requests an agent-facing `queue_status` tool that reports lane depth, active concurrency, and per-agent workload so agents can dispatch work intelligently.

## Best Possible Solution

Build or reuse a shared read-only queue snapshot backend, then expose it through `queue_status` with catalog/profile/policy wiring, docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection on current main gives high-confidence proof that no agent-facing `queue_status` or equivalent stable queue snapshot surface exists yet.

## Solution Assessment

Unclear until maintainers approve the API and metadata-visibility contract. The narrow maintainable direction is a shared snapshot provider plus an agent-facing tool, rather than asking agents to infer load from `agents_list`, `sessions_list`, or logs.
