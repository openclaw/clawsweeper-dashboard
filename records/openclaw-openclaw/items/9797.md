## Summary

Keep open: current main and the latest release still lack an agent-facing `queue_status` or equivalent workload/capacity tool, while the available queue snapshot state remains internal and the related operator-diagnostics PR was closed unmerged.

## What This Changes

The issue requests a model-facing queue-status surface so agents can inspect lane depth, active concurrency, and per-agent workload before dispatching tasks.

## Best Possible Solution

Define one shared read-only queue snapshot provider, then expose the smallest agent-safe workload and capacity facts through the chosen model-facing tool, catalog, and policy surface.

## Reproduction Assessment

Yes, at source level: current main has internal lane snapshot helpers, while the supported agent tool assembly and catalog have no queue or capacity status surface. This is a missing feature/API contract rather than a failing existing behavior.

## Solution Assessment

Unclear: a dedicated queue_status tool is plausible, but maintainers should first choose whether the minimal capacity facts belong in a new tool, session_status, or structured sessions_spawn responses and define visibility rules.
