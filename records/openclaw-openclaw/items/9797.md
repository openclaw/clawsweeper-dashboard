## Summary

Keep this open: current main still lacks an agent-facing `queue_status` tool or equivalent stable queue snapshot surface, and the related diagnostics PR explicitly excludes in-agent dispatch.

## What This Changes

Request an agent-facing `queue_status` tool that exposes lane and per-agent workload so dispatchers can choose idle targets for `sessions_spawn`.

## Best Possible Solution

Define a shared read-only queue snapshot backend, then expose `queue_status` through the agent tool catalog/profile/policy surface with visibility filtering, docs, and focused tests.

## Solution Assessment

Unclear until maintainers approve the agent-visible API shape and metadata-visibility rules. The maintainable direction is a shared snapshot provider plus an agent-facing tool, not inference from `agents_list`, `sessions_list`, or verbose logs.
