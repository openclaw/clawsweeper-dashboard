## Summary

Keep this issue open: current main still lacks an agent-facing `queue_status` tool or equivalent stable queue snapshot surface for dispatch decisions, and the related operator diagnostics PR is still open and explicitly excludes the in-agent tool.

## What This Changes

Request an agent-facing `queue_status` tool that exposes lane and per-agent workload so agents can route `sessions_spawn` work intelligently.

## Best Possible Solution

Build a shared read-only queue snapshot backend, then expose it through an agent-facing `queue_status` tool with catalog/profile/policy wiring, docs, and focused tests; reuse the operator diagnostics backend if https://github.com/openclaw/openclaw/pull/43618 lands first.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high-confidence proof that current main lacks the requested agent-facing queue snapshot surface.

## Solution Assessment

Yes, the best direction is a shared snapshot provider plus an agent-facing tool rather than asking agents to infer workload from `agents_list`, `sessions_list`, or logs. The open operator diagnostics PR can be a backend candidate, but it does not complete this issue by itself.
