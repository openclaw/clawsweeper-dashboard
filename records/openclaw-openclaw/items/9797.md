## Summary

Keep open: current main and the latest release still lack an agent-facing `queue_status` or equivalent workload/capacity tool. Internal lane snapshot helpers exist, but they are not exposed through the agent tool catalog, tool registration, Gateway queue diagnostics, or documented session/subagent tools.

## What This Changes

The issue requests a read-only agent-facing queue/workload status surface so agents can route `sessions_spawn` work to idle capacity instead of guessing.

## Best Possible Solution

Implement one shared read-only queue snapshot backend in core, then expose a filtered `queue_status` agent tool with catalog/profile/policy wiring, docs, and focused tests after maintainers settle the metadata and visibility contract.

## Reproduction Assessment

Yes, as a source-level feature gap: current main has no `queue_status` or `sessions_capacity` tool in the catalog or tool registration, and the existing session/agent tools do not expose lane queue depth or active-vs-max counts. I did not run a live agent session because the missing model-facing surface is clear from source and docs inspection.

## Solution Assessment

Unclear pending maintainer product direction: the proposed `queue_status` tool is plausible, but the best fix should reuse a shared queue snapshot backend instead of creating a separate model-only view. The remaining decision is the stable metadata shape plus visibility and permission policy.
