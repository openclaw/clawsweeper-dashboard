## Summary

Keep open: this is a focused bug-fix PR with adequate real-environment proof, but one blocking correctness issue remains in the global npm update path because the freshness probe does not use the same npm command that the install will run.

## What This Changes

The PR bypasses npm freshness quarantine for OpenClaw-managed npm update/install, plugin metadata/pack/install flows, hosted installer scripts, docs, tests, and changelog.

## Best Possible Solution

Keep the freshness-bypass direction, but make the bypass helper derive npm config from the resolved npm command and child env used by the install, with a regression test for prefix-local npm or a configured global `before` file not visible to plain `npm`.

## Reproduction Assessment

Yes. The linked issue gives concrete npm `ETARGET` logs with a moving `before` cutoff, and current main's update/install sources do not neutralize `min-release-age` or `before` before running npm child processes.

## Solution Assessment

No, not yet. The proposed bypass is the right owner-boundary fix, but global update args must probe freshness with the selected npm command/env rather than plain `npm` to cover Homebrew or prefix-local npm installs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Probe config with the selected npm binary:** `src/infra/update-global.ts:732`
  - body: `globalInstallArgs` installs with `resolved.command`, but the new freshness helper probes config by running plain `npm`. When OpenClaw selects a prefix-local npm such as a Homebrew npm that is not the `npm` on `PATH`, the probe can miss that npm's `before` policy and append `--min-release-age=0`; npm treats `before` and `min-release-age` as mutually exclusive, so the update can still fail in the reported setup. Pass the resolved npm command and child env into the probe or compute the override beside the selected command.
  - confidence: 0.86
