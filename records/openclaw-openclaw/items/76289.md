## Summary

Keep open. Current main honors the configured cron wall-clock timeout, but resumed CLI-backed cron runs can still be terminated by the 180-second no-output watchdog before a 600-second cron deadline, so the reported user-visible timeout cap is still plausible and narrowly actionable.

## What This Changes

The issue reports that cron job `timeoutSeconds` values above 180 seconds are not honored and long cron runs are terminated around 180 seconds without a warning.

## Best Possible Solution

Make explicit cron `timeoutSeconds` consistently govern lower runtime watchdog ceilings, or emit a clear diagnostic when a smaller CLI no-output or model-idle guard is the actual terminating deadline.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a continuation/resumed CLI-backed cron run with `timeoutSeconds: 600`: cron passes `timeoutMs=600000`, but `useResume=true` selects the resume watchdog profile capped at `180000`.

## Solution Assessment

No. The current outer cron timeout policy is correct, but lower watchdog behavior can still preempt the configured cron timeout without making the distinct cap clear to the user.
