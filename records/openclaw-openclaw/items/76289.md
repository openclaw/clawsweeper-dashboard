## Summary

Keep open. Current main fixes the older queue/startup timeout budget issue and honors `timeoutSeconds` for the outer cron deadline, but a resumed CLI-backed cron run can still hit the default 180-second no-output watchdog before a configured 600-second cron timeout.

## What This Changes

The issue asks for cron `timeoutSeconds: 600` to be honored instead of silently terminating long cron jobs around 180 seconds.

## Best Possible Solution

Make explicit cron agent-turn timeouts consistently govern lower runtime watchdog ceilings, or surface a clear user-visible diagnostic when a smaller no-output or idle guard ends the run first.

## Reproduction Assessment

Yes, at source level. A persisted CLI cron continuation with `timeoutSeconds: 600` sends `timeoutMs=600000` into `runCliAgent`, but `resolveCliNoOutputTimeoutMs` with `useResume: true` still returns the 180000 ms default cap.

## Solution Assessment

No, current behavior is not the best way to solve the issue. The outer cron timeout policy is now correct, but the lower CLI and embedded watchdogs can still end the run earlier without tying that result clearly to the configured cron timeout.
