## Summary

Keep open. Current main honors `timeoutSeconds` in the outer cron timer, but a CLI-backed/resumed cron run can still hit the 180-second no-output watchdog before a configured 600-second cron timeout, so the reported user-visible problem is not fully resolved.

## What This Changes

Cron `timeoutSeconds` should allow long-running agent-turn cron jobs to run for the configured duration instead of being silently terminated around 180 seconds.

## Best Possible Solution

Make explicit cron `timeoutSeconds` consistently govern lower runtime watchdog ceilings, or surface a clear diagnostic/config path when a smaller no-output or idle guard terminates the run first.

## Reproduction Assessment

Yes. A code-level reproduction is a CLI-backed cron continuation run with `timeoutSeconds: 600`: `runCliAgent` receives `timeoutMs=600000`, but `resolveCliNoOutputTimeoutMs(... useResume: true)` clamps the no-output watchdog to 180000 ms.

## Solution Assessment

No. The outer cron timeout policy is correct, but the current lower watchdog behavior can still preempt the configured cron timeout without making that distinct cap clear to the user.

Fix the cron timeoutSeconds preemption bug reported in https://github.com/openclaw/openclaw/issues/76289. Current outer cron timeout handling honors payload.timeoutSeconds, but CLI-backed resumed cron runs can still be killed by the 180s CLI no-output watchdog, and embedded runs have a separate idle watchdog clamp.
