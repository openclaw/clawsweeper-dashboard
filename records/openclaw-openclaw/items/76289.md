## Summary

Keep open. The version-specific queue-time budget bug was fixed by #41783, but current `main` still has a source-level path where a resumed CLI cron run with `timeoutSeconds: 600` can be terminated by the 180-second no-output watchdog before the configured cron deadline.

## What This Changes

The issue asks for cron `timeoutSeconds` values above 180 seconds to be honored, or for any lower hard cap to be made explicit to users.

## Best Possible Solution

Align lower runtime watchdog behavior with explicit cron `timeoutSeconds`, or surface distinct documented diagnostics when a lower no-output or idle guard terminates the run first.

## Reproduction Assessment

Yes, at source level. A resumed CLI-backed cron run with `timeoutSeconds: 600` passes `timeoutMs=600000` to `runCliAgent`, but `resolveCliNoOutputTimeoutMs` uses the resume watchdog profile and clamps the no-output watchdog to 180000 ms.

## Solution Assessment

No, current `main` is not a sufficient solution. It fixes the #41783 queue-time budget issue, but the lower CLI and embedded watchdogs can still preempt the configured cron deadline without making that policy clear to the user.
