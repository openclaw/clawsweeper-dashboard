## Summary

Keep open. Current main fixes adjacent 2026.4.10 timeout causes, but CLI continuation cron runs can still pass `useResume=true` and clamp the no-output watchdog to 180000 ms while the outer cron timeout is 600000 ms; embedded runs also have a lower implicit idle watchdog path.

## What This Changes

The issue asks cron `timeoutSeconds` and `openclaw cron update --timeout` to honor values above 180 seconds or clearly warn when a lower cap is applied.

## Best Possible Solution

Make explicit cron agent-turn timeouts either set the effective lower watchdog ceilings for that run or emit a clear diagnostic when a smaller idle/no-output guard is what terminates first.

## Reproduction Assessment

Yes, at source level. A CLI-backed cron continuation run with `timeoutSeconds: 600` can pass a stored CLI session ID, set `useResume=true`, and resolve the no-output watchdog to the 180000 ms resume-profile maximum.

## Solution Assessment

No, current main is only a partial solution. The shipped outer-timer and fresh-session fixes address adjacent symptoms, but they do not prevent or explain the remaining lower watchdog caps.
