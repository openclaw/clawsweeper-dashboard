## Summary

Keep open: current main and v2026.6.8 still route Microsoft Teams SharePoint/OneDrive attachment sends through raw proactive delivery without the existing thread root, and open linked PRs such as https://github.com/openclaw/openclaw/pull/93837 and https://github.com/openclaw/openclaw/pull/94348 are active fix candidates rather than merged provenance.

## What This Changes

The issue reports Microsoft Teams channel replies with attachments posting as top-level messages when `channels.msteams.replyStyle` is `thread`.

## Best Possible Solution

Land one narrow Microsoft Teams plugin fix that passes the existing thread root through SharePoint/OneDrive attachment send paths while preserving explicit top-level behavior.

## Reproduction Assessment

Yes, at source level: a threaded Teams channel file send reaches SharePoint or OneDrive attachment branches that bypass the existing thread-aware sendMSTeamsMessages path. I did not run a live Teams tenant reproduction in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is to reuse the existing threadActivityId contract in the Teams attachment send path, not add config or change reply-style semantics.
