- `P2`: The issue affects Microsoft Teams attachment reply placement in a specific channel mode without crash, security, or broad runtime outage.
- `impact:message-loss`: Attachment replies are delivered to the wrong Teams location, losing the expected thread relationship.

## Summary

Keep open: current main and the latest release still send Microsoft Teams SharePoint/OneDrive attachment replies through raw proactive delivery without the existing thread root, and https://github.com/openclaw/openclaw/pull/94348 is an open closing PR rather than merged provenance.

## What This Changes

The issue reports that Microsoft Teams replies with file attachments should stay in the configured channel thread instead of posting as new top-level messages.

## Best Possible Solution

Land one narrow Microsoft Teams plugin fix that passes the existing thread root through SharePoint and OneDrive attachment send paths while preserving explicit top-level behavior.

## Reproduction Assessment

Yes, at source level: a threaded Teams channel file send reaches the SharePoint or OneDrive attachment branches that bypass the existing thread-aware sendMSTeamsMessages path. I did not run a live Teams tenant reproduction in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is to reuse the existing threadActivityId contract in the Teams attachment send path, not add config or change reply-style semantics.
