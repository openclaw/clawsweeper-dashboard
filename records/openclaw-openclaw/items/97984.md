## Summary

Keep open: current main and the latest release still provide terminal/local reauth guidance and existing device-code primitives, but they do not implement a channel-safe reauth session that sends pairing material, dedupes active flows, persists the refreshed profile, or reports completion back to the originating conversation.

## What This Changes

The issue requests a channel-originated model reauth flow for hard OAuth/profile failures, initially using OpenAI device-code pairing instead of terminal-only recovery commands.

## Best Possible Solution

Define a gateway/provider auth-recovery contract that starts and dedupes OpenAI device-code reauth only for authorized channel contexts, persists through the auth-profile store, and reports safe completion, expiry, or operator-needed states back to the originating conversation.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection confirms current main and `v2026.6.10` still only provide terminal/local reauth guidance while device-code auth is not wired into channel recovery.

## Solution Assessment

Yes directionally, but not as an automation-ready patch. The maintainable path is a small explicit auth-recovery contract and security-reviewed channel authorization model, not shelling out to the interactive CLI from a channel failure path.
