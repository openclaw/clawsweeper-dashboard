## Summary

Keep this PR open: it targets a real source-reproducible CLI prompt/tool-surface mismatch, but the branch can still advertise loopback tools when no loopback MCP runtime/config was actually injected.

## What This Changes

The PR reuses the MCP loopback scoped-tool resolver for CLI prompt/report construction and adds prompt-tool-name hashing to CLI session reuse.

## Best Possible Solution

Gate prompt-visible loopback tools on the same active loopback runtime/config that is injected into the CLI backend, keep the deterministic session invalidation, and land with focused fallback coverage.

Yes at source level: current main passes tools: [] into the CLI prompt/report path, and Memory Recall is gated on memory_search/memory_get in availableTools. I did not run a live CLI turn in this read-only review.

## Solution Assessment

No, not as-is: reusing the loopback resolver is the right direction, but prompt tools should be tied to an active injected loopback runtime before the CLI prompt advertises them.
