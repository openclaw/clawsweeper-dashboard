## Summary

Keep open: the PR targets a real source-reproducible CLI prompt/tool mismatch, but the patch can advertise loopback tools when no loopback runtime was injected, and the attached proof is unit/mock-only rather than a real CLI run.

## What This Changes

The PR reuses the MCP loopback scoped-tool resolver for bundled CLI prompt construction, reports those prompt-visible tools, and adds prompt-tool hash session invalidation.

## Best Possible Solution

Gate prompt-visible tools on an active loopback runtime so the prompt mirrors the injected MCP config/tools/list, keep deterministic session invalidation for prompt-tool changes, and merge after redacted live CLI proof.

## Reproduction Assessment

Yes at source level: current main passes tools: [] into buildCliAgentSystemPrompt while Memory Recall returns no lines when memory_search/memory_get are absent from availableTools. I did not run a live CLI repro in this read-only review.

## Solution Assessment

No, not as-is: sharing the loopback resolver is the right direction, but the prompt tool list should be tied to an active injected loopback runtime before advertising tools. A real CLI fresh/existing-session proof should verify the final path.
