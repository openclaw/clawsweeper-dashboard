## Summary

Keep open: the PR addresses the right CLI prompt/tool mismatch, but the branch currently fails `check-test-types`, lacks required real CLI behavior proof, and has a loopback-availability edge case that should be fixed before merge.

## What This Changes

The PR adds a shared loopback scoped-tool resolver, feeds its tools into CLI prompt and report generation, and stores a prompt tool-name hash in CLI session fingerprints.

## Best Possible Solution

Land the loopback resolver approach only after it is gated on the actual injected loopback runtime, the test type failure is fixed, and a redacted real CLI bundled-MCP proof shows Memory Recall guidance with callable loopback tools.

## Reproduction Assessment

Yes from source inspection: current main passes `tools: []` into CLI prompt construction while the loopback separately resolves policy-filtered tools, and memory-core suppresses Memory Recall when the available tool set is empty. I did not run a live `claude-cli` turn in this read-only review.

## Solution Assessment

Partly: reusing the loopback scoped-tool resolver is the right boundary, and hashing the prompt-visible tool names is a reasonable session reset mechanism. The current branch still needs the type-check fix, a loopback-availability guard, and real CLI proof before it is the best mergeable solution.
