## Summary

This should stay open: current main still treats a single OpenClaw Gateway listener on wildcard/LAN bind as a busy-port warning, even though non-loopback Gateway binds are a documented supported deployment mode when auth is configured.

## What This Changes

The issue asks status and doctor diagnostics to treat a healthy OpenClaw Gateway listener bound to LAN or wildcard addresses as expected instead of a port conflict.

## Best Possible Solution

Add a shared expected-Gateway listener predicate for single Gateway loopback or wildcard listeners plus the existing same-PID dual-stack loopback case, then use it consistently in status and doctor diagnostics without changing real port-conflict handling.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a `PortUsage` with one OpenClaw Gateway listener at `*:18789`, `0.0.0.0:18789`, or `[::]:18789` is neither empty nor dual-stack loopback, so current status emits a warning and shared diagnostics include “already in use.” I did not mark it `reproduced` because this review did not run a live Gateway or tests.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable fix: centralize expected Gateway listener detection in `src/infra/ports-format.ts` and reuse it from status and doctor while preserving warnings for foreign or ambiguous owners.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/77939. Current `openclaw status --all` and the doctor Gateway port diagnostic path treat a healthy managed Gateway listener on wildcard/LAN bind as a port conflict. Add a shared predicate in `src/infra/ports-format.ts` that treats expected OpenClaw Gateway listeners as OK when they are either the existing same-PID dual-stack loopback case or a single Gateway listener on the configured port with a loopback/wildcard address such as `127.0.0.1`, `localhost`, `::1`, `0.0.0.0`, `::`, or `*`. Export it through `src/infra/ports.ts`, use it in `src/commands/status-all/diagnosis.ts` and `src/commands/doctor-gateway-daemon-flow.ts`, and keep warnings for unknown processes, SSH tunnels, mixed Gateway plus non-Gateway listeners, multiple unrelated listeners, and multiple Gateway PIDs. Add focused regression coverage in the existing port-format, status diagnosis, and doctor Gateway daemon tests. Add a single-line user-facing changelog fix entry. Do not change Gateway startup port availability, auth policy, bind-mode semantics, or `handlePortError` behavior unless a test proves it is necessary.
