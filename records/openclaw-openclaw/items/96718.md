## Summary

Keep open: this PR is not merge-ready because it filters nulls after tool-call validation, while the source path can already coerce nullable string fields to empty strings before MCP execution; it also changes explicit JSON null semantics and lacks real MCP proof.

## What This Changes

The PR strips top-level `null` properties from materialized MCP tool inputs before `runtime.callTool(...)` and makes a small unrelated test variable change.

## Best Possible Solution

Preserve explicit JSON null through tool validation and MCP dispatch, fixing the nullable-string coercion path and only omitting fields when schema/default semantics explicitly require absence.

## Reproduction Assessment

Yes, source-reproducible with medium confidence: for the reported `anyOf [string, null]` shape, current validation can coerce null to an empty string before `execute(...)`, then MCP runtime forwards that record. I did not run the awslabs EKS MCP server or capture the raw wire payload.

## Solution Assessment

No. Since MCP execution receives already validated arguments, the maintainable fix should preserve nullable values in validation/coercion or apply schema-aware omission before validation, not unconditionally strip nulls after validation.
