## Summary

Keep open. Current main appears to have fixed the hook session rebinding and plugin webhook route-registry pieces, but the BlueBubbles phone-format routing mismatch remains: DM peer routing and BlueBubbles conversation binding still compare the normalized sender/conversation strings exactly, and the current normalizer does not equate `+1...` with `1...`.

## What This Changes

The issue reports BlueBubbles routed messages using the wrong agent context, intermittent `/bluebubbles-webhook` 404s, and peer-specific routing misses when phone handles alternate between `+1...` and `1...` formats.

## Best Possible Solution

Implement BlueBubbles-owned phone-handle canonicalization for DM peer routing and configured/runtime conversation binding so `+1XXXXXXXXXX` and `1XXXXXXXXXX` select the same peer before wildcard/catch-all routing, while preserving the current hook session rebinding and plugin HTTP route registry behavior.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a BlueBubbles config with a peer-specific binding for `+1XXXXXXXXXX`, a wildcard/catch-all binding, and an inbound DM whose sender normalizes to `1XXXXXXXXXX`; current source shows the peer-specific binding will not match before wildcard fallback.

## Solution Assessment

Yes, the narrowest maintainable fix is channel-owned canonical matching in the BlueBubbles route/conversation helpers. Changing generic `resolveAgentRoute` peer equality directly would be riskier because other channels rely on exact peer ids.
