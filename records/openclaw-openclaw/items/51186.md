## Summary

Keep open. Current main appears to have fixed the `/hooks/agent` agent-session rebinding path and the plugin webhook route-registry 404 class, but the BlueBubbles plus/no-plus phone-handle mismatch is still present before peer-specific routing can win over wildcard or catch-all routing.

## What This Changes

The issue asks OpenClaw to prevent BlueBubbles routed messages from using the wrong agent context, keep `/bluebubbles-webhook` reachable when registered, and normalize phone handles such as `+1...` and `1...` for routing.

## Best Possible Solution

Implement BlueBubbles-owned phone-handle canonicalization so configured `+1...`, inbound `1...`, route peer ids, allowFrom checks, and ACP conversation ids converge before wildcard or catch-all routing; coordinate with #62101 if that PR lands first.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a BlueBubbles DM with a peer binding for `+1XXXXXXXXXX`, a catch-all binding, and an inbound sender of `1XXXXXXXXXX`; current main preserves those as different ids before exact peer matching.

## Solution Assessment

Yes for the remaining work. The narrow maintainable fix belongs in the BlueBubbles handle/conversation normalization path rather than changing generic `resolveAgentRoute` equality for every channel.
