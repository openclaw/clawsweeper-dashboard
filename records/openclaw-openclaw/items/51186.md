## Summary

Keep open: current main appears to have fixed the explicit `/hooks/agent` target-agent isolation and stale plugin-route webhook 404 paths, but the BlueBubbles phone-format binding miss still appears possible from source inspection.

## What This Changes

The issue reports BlueBubbles inbound routing using the wrong agent context, intermittent `/bluebubbles-webhook` 404s, and peer-specific bindings missing when handles differ by `+1...` versus `1...` format.

## Best Possible Solution

Keep the current hook and webhook registry fixes, and add a BlueBubbles-owned handle matching or canonicalization path that safely treats equivalent phone variants as the same peer while preserving email and explicit chat-target behavior.

## Reproduction Assessment

No, not a full live reproduction. Source inspection gives a clear current-main path for the phone-format miss: BlueBubbles preserves the leading `+`, and routing compares peer IDs exactly, so `1...` and `+1...` can diverge.

## Solution Assessment

No. Current main looks like the right direction for the hook and webhook registry parts, but the issue is not fully solved until BlueBubbles peer matching handles safe phone-format variants in the plugin-owned routing surface.
