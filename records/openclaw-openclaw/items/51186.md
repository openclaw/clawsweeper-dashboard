## Summary

Keep open. Current main appears to have addressed the `/hooks/agent` session rebinding and plugin webhook route-registry failures, but the BlueBubbles phone-format routing mismatch is still present in source: `+1...` and `1...` can still diverge before peer-specific routing falls through to a wildcard or catch-all agent.

## What This Changes

The issue reports BlueBubbles wrong-agent routing through `/hooks/agent`, intermittent `/bluebubbles-webhook` 404s, and peer matching misses between `+1...` and `1...` phone-handle formats.

## Best Possible Solution

Implement BlueBubbles-owned phone-handle canonicalization for both DM peer route matching and configured/runtime conversation binding so `+1XXXXXXXXXX` and `1XXXXXXXXXX` select the same peer before wildcard or catch-all routing, coordinating with the narrower allowFrom normalization in #62101 if that PR lands first.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a BlueBubbles DM with a peer-specific binding for `+1XXXXXXXXXX`, a catch-all binding, and an inbound sender string of `1XXXXXXXXXX`; current main passes the raw sender to exact peer matching and the normalizer still does not canonicalize the plus/no-plus variants.

## Solution Assessment

Yes. The narrowest maintainable fix is BlueBubbles-owned canonical matching in its handle, DM peer, and conversation helpers; changing generic `resolveAgentRoute` equality would be riskier because other channels may intentionally rely on exact peer ids.
