## Summary

Keep this open: current main still allows empty Claude CLI subprocess results to become success-shaped no-payload runs, the latest shipped release matches the affected behavior, and the direct tested fix PR is closed unmerged with a compatibility/product decision still outstanding.

## What This Changes

The issue asks OpenClaw to stop treating zero-token, no-text Claude CLI subprocess completions as billing failures that can poison shared auth-profile health.

## Best Possible Solution

Reopen or replace https://github.com/openclaw/openclaw/pull/83421 after maintainer acceptance of the empty-response failover policy, with regression coverage proving blank claude-cli output cannot poison shared auth-profile health.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection shows current main can preserve blank Claude CLI output as a success-shaped no-payload run, and the issue plus the closed fix PR provide concrete runtime and after-fix evidence.

## Solution Assessment

Yes, the narrowest maintainable direction is to use the existing empty_response failure reason at the CLI runner boundary and keep that reason out of shared auth-profile health. Maintainers still need to decide whether failover is preferable to retry/no-op behavior for successful-but-empty CLI completions.
