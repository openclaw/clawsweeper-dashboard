## Summary

This issue should stay open: current main still accepts an empty Claude CLI result as a success-shaped CLI run, the latest release has the same behavior, and the only focused fix is a closed, unmerged PR with a maintainer compatibility decision still outstanding.

## What This Changes

The issue reports that empty successful Claude CLI subprocess output can poison shared auth-profile health as a billing cooldown and block later cron or Discord runs.

## Best Possible Solution

Reopen or replace the narrow empty_response classification approach from https://github.com/openclaw/openclaw/pull/83421 after maintainer acceptance of the failover policy, with regression coverage proving blank claude-cli output cannot poison shared auth-profile health.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection shows current main can preserve blank claude-cli output as a success-shaped no-payload run, while the issue and closed PR provide concrete runtime signatures and after-fix proof for the auth-state impact.

## Solution Assessment

Yes, the narrowest maintainable direction appears to classify successful-but-empty CLI output as empty_response and keep that reason out of shared auth-profile health. Maintainers still need to choose whether that should fail over immediately or use a softer retry/no-op policy.
