## What This Changes

The issue asks OpenClaw to stop treating successful-but-empty `claude-cli` subprocess responses as billing/auth health failures that can cool down a shared auth profile.

## Best Possible Solution

Land or replace the narrow empty-response handling from the closed related PR after maintainer acceptance of the failover policy, with regression coverage for CLI runner behavior and auth-profile failure accounting.

No live current-main reproduction was run. Source inspection plus the reporter's shipped-runtime auth-state and trajectory signature give a high-confidence source reproduction path for successful empty Claude CLI output reaching success/fallback policy without a non-billing bucket.

## Solution Assessment

Yes, the maintainable direction is to classify successful-but-empty Claude CLI output separately from billing/auth failures and keep it out of shared auth-profile health. The remaining question is whether that separate bucket should fail over immediately or use retry/no-op semantics.
