## Summary

This issue should stay open: current main and the latest shipped release still preserve empty Claude CLI subprocess output as a success-shaped run, and the only direct fix attempt is a closed, unmerged PR rather than an implementation on main.

## What This Changes

The issue asks OpenClaw to stop treating zero-token, no-text Claude CLI subprocess completions as billing/auth-profile cooldown failures.

## Best Possible Solution

Reopen or replace the narrow https://github.com/openclaw/openclaw/pull/83421 approach after maintainer acceptance of the empty-response failover policy, with regression coverage proving blank claude-cli output cannot poison shared auth-profile health.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection clearly shows current main can preserve an empty claude-cli result as a success-shaped no-payload run, and the issue plus the closed fix PR provide concrete runtime signatures and after-fix proof.

## Solution Assessment

Yes, the narrowest maintainable direction is to classify successful-but-empty CLI output as a non-billing empty_response and suppress that reason from shared auth-profile health. Maintainers still need to accept whether that should fail over immediately or use a softer retry/no-op policy.
