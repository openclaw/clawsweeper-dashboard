## Label Justifications

- `P1`: The report describes real cron and Discord agent runs failing quickly after auth-profile cooldown is poisoned by empty Claude CLI responses.
- `impact:session-state`: The failure path creates success-shaped no-payload runs and can leave subsequent sessions without trajectory files once cooldown blocks execution.
- `impact:auth-provider`: The core symptom is an auth-profile billing cooldown being applied to a funded claude-cli profile without a real billing failure.

## Summary

Keep open: current main still preserves empty Claude CLI subprocess output as a success-shaped no-payload run, the latest release has the same behavior, and the direct fix PR is closed unmerged rather than implemented on main.

## What This Changes

The issue requests that successful zero-token/no-text Claude CLI subprocess results not be classified as billing failures or poison auth-profile cooldown state.

## Best Possible Solution

Accept and land or replace the narrow empty_response policy fix, with regression coverage proving blank claude-cli output cannot increment billing/auth cooldown counters.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection and the linked closed PR proof show empty Claude CLI output can still become a success-shaped no-payload result on current main, but the auth-state billing write was not freshly exercised here.

## Solution Assessment

Yes, if maintainers accept the behavior change: the narrowest fix is to classify successful-but-empty CLI output as empty_response and exclude that reason from shared auth-profile health. If failover is too strict, the same CLI boundary should implement an explicit retry/no-op policy rather than billing cooldown.
