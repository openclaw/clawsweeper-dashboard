## What This Changes

The issue asks OpenClaw to stop treating empty successful claude-cli subprocess completions as billing failures that disable auth profiles.

## Best Possible Solution

Add a targeted claude-cli empty-success classification or retry path so zero-token/no-text/no-error completions do not increment billing counters or disable auth profiles, while preserving real stderr/non-zero/billing classifications.

## Reproduction Assessment

No. The issue body gives concrete live signatures and current source supports the empty-success CLI path, but this review did not run a live claude-cli current-main repro or capture the precise billing-state write.

## Solution Assessment

Yes. The narrow maintainable fix is to classify or retry successful-but-empty claude-cli results before auth-profile accounting, using the existing non-billing `format` policy or a no-op path rather than adding a user-facing setting.
