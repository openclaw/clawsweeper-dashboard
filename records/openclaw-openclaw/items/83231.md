## Summary

This issue should stay open: current main and the latest shipped release still allow a blank Claude CLI result to become a success-shaped no-payload run, and the only focused fix is a closed, unmerged PR that needs maintainer policy review before replacement work is queued.

## What This Changes

The issue reports that empty successful claude-cli subprocess completions are counted as billing failures and can disable the shared auth profile cooldown state.

## Best Possible Solution

Reopen or replace https://github.com/openclaw/openclaw/pull/83421 after maintainer acceptance of the empty-response failover policy, with regression coverage proving blank claude-cli output cannot poison shared auth-profile health.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection shows current main can preserve empty claude-cli results as success-shaped no-payload runs, and the issue plus the closed fix PR provide concrete runtime signatures and after-fix proof.

## Solution Assessment

Yes on direction, but it needs maintainer policy acceptance: the narrow fix is to classify successful-but-empty CLI output as empty_response and keep that reason out of shared auth-profile cooldown state, rather than treating it as billing.
