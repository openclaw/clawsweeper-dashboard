## Best Possible Solution

Teach direct Claude CLI empty zero-token successes to become an explicit empty-response or format fallback signal that never records `billing`, while preserving real non-zero-exit or billing-shaped stderr handling.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/83231. Current main already handles some embedded PI zero-token empty stop turns, but the direct Claude CLI subprocess path in src/agents/cli-runner.ts can build a successful result with empty text, no payloads, zero usage, and no `agentHarnessResultClassification`; `classifyEmbeddedPiRunResultForModelFallback` then returns null for non-GPT Claude models unless the result is already harness-tagged or error-shaped. Add a narrow fix so exit-0/empty-output/zero-usage Claude CLI completions are treated as an explicit empty-response or `format` fallback signal and do not increment `failureCounts.billing` or set `disabledReason: "billing"`. Preserve real billing behavior for non-zero exits or stderr/stdout containing billing-shaped provider errors. Prefer updating the direct CLI result/classification boundary over adding startup auth-state repair or new config. Add focused regression coverage for the direct CLI path and auth-state failure counts, and add a user-facing changelog entry.
