## Summary

This issue should stay open because current main and v2026.5.12 still allow empty Claude CLI stream-json results to become success-shaped no-payload CLI runs, while the only direct fix is a closed, unmerged PR with an intentional failover-policy tradeoff.

## What This Changes

The issue reports that successful-but-empty Claude CLI subprocess completions are counted as billing failures and can disable shared auth profiles.

## Best Possible Solution

Reopen or replace the narrow https://github.com/openclaw/openclaw/pull/83421 approach after maintainer acceptance of the empty-response failover policy, with regression coverage proving blank Claude CLI output cannot poison shared auth-profile health.

## Reproduction Assessment

No high-confidence live current-main reproduction was run. Source inspection shows current main still preserves blank Claude CLI output as a success-shaped no-payload run, and the issue plus the closed PR provide concrete runtime signatures for the billing cooldown path.

## Solution Assessment

Yes, the narrowest maintainable direction is to classify successful-but-empty Claude CLI output as a non-billing empty_response and keep that reason out of shared auth-profile health. Maintainers still need to decide whether failover is preferable to retry/no-op behavior for tool-only blank completions.
