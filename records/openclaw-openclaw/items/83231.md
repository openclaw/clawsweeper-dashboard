## Summary

This issue should stay open: current main still preserves empty Claude CLI results as success-shaped CLI output, the closed related fix PR was not merged, and no shipped/main fix proves the billing-cooldown symptom is resolved.

## What This Changes

The issue reports that successful-but-empty Claude CLI subprocess completions are treated as billing failures and can disable the shared auth profile.

## Best Possible Solution

Classify successful-but-empty CLI subprocess output as `empty_response` before success finalization and keep that reason out of shared auth-profile health accounting, with regression coverage for direct CLI, cron/context-engine, and auth-profile policy behavior.

## Reproduction Assessment

No live current-main reproduction was run. Source inspection does show the empty Claude CLI result is still preserved as successful output and lacks the non-billing `empty_response` handling proposed by the closed fix PR.

## Solution Assessment

Yes: using the existing `empty_response` failover reason at the CLI runner boundary and suppressing it from shared auth-profile failure state is the narrowest maintainable fix. The closed related PR already demonstrates that shape but did not land.
