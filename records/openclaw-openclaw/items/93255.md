## What This Changes

The issue reports that `sessions_send` uses a hardcoded webchat channel for explicit cross-channel session targets, which can misroute accepted messages to a sibling webchat session.

## Best Possible Solution

Review and either land https://github.com/openclaw/openclaw/pull/94409 with focused routing proof or replace it with a gateway-level fix that makes explicit session keys authoritative without regressing existing A2A delivery behavior.

## Reproduction Assessment

Partly. Current main source reproduces the hardcoded webchat channel in the `sessions_send` gateway agent call, but the exact sibling-session selection still needs a focused current-main regression or live replay.

## Solution Assessment

Unclear. Deriving the channel from the target session entry is a plausible narrow fix, but maintainers should review whether the gateway should instead treat explicit session keys as authoritative over channel hints.
