## Summary

Keep open: current main and the latest release still hardcode the `sessions_send` nested agent call as webchat, and the issue is already paired with an open closing PR that should be reviewed before issue cleanup.

## What This Changes

The issue requests that `sessions_send` stop forcing inter-session sends through the webchat channel when the resolved target session is bound to another channel.

## Best Possible Solution

Review and land or reject the linked fix PR, with regression coverage proving explicit non-webchat target sessions keep their stored channel through the `sessions_send` gateway call.

## Reproduction Assessment

Yes. Current main resolves the target session key but still sends the gateway call with `channel: INTERNAL_MESSAGE_CHANNEL`, and the latest release has the same source shape; I did not run the live Telegram/Slack-style replay in this read-only review.

## Solution Assessment

Yes in fix shape: derive the nested call channel from canonical target session metadata before the gateway call, preserving webchat only as a fallback when no stored channel is available. The exact field choice should be settled while reviewing the linked PR.
