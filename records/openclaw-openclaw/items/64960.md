## Summary

Keep open. Current main still resolves compaction to one effective provider/model and retries summarization on that same model, and the provided GitHub context shows open PR #74470 already targets this issue with closing syntax, so the issue should stay open until that implementation is reviewed and merged or closed.

## What This Changes

The issue requests ephemeral model fallback for compaction when Azure content filtering rejects the summarization prompt, without persisting fallback model selection to session state.

## Best Possible Solution

Keep this issue open while #74470 is reviewed. The desired end state is a narrow compaction fallback implementation that tries fallback candidates only for the compaction attempt, never mutates persisted session model selection, preserves explicit `compaction.model` semantics, and coordinates any hard truncation escalation with #64962.

## Reproduction Assessment

Yes. A focused regression can stub compaction summarization to throw an Azure-style filtered HTTP 400 while model fallbacks are configured; current main resolves one compaction target and retries that same `params.model`.

## Solution Assessment

Unclear until #74470 is reviewed. Reusing the session fallback chain for implicit compaction may be the narrowest maintainable repair, but explicit operator-controlled compaction fallback config and hard truncation escalation still need maintainer judgment.
