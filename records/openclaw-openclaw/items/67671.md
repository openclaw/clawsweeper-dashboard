## Summary

Keep open. Current main still lacks the requested direct single-job cron read path: CLI inspection still pages through `cron.list`, the Gateway method registry/scopes/protocol/server handlers expose no `cron.get`, and the agent cron tool has no `get` action. Open PR #75117 is already linked as the active implementation candidate, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Add an additive single-job cron inspection path by id through `openclaw cron get <id>` and Gateway RPC `cron.get`, returning the stored job definition or a clean not-found error.

## Best Possible Solution

Land PR #75117 or an equivalent narrow additive implementation that wires `cron.get` through Gateway registry, read scopes, protocol schemas/types/validators, server handling, CLI, agent tool support, docs, changelog, and focused tests while preserving existing scheduling, execution, delivery, `cron list`, and `cron show` behavior.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists on current main: `openclaw cron show` still retrieves pages via `cron.list`, and the Gateway registry/scopes/protocol/server/tool surfaces do not expose `cron.get`.

## Solution Assessment

Yes. A direct read path is the narrow maintainable fix because cron jobs are already id-addressable for mutation and an internal `CronService.getJob(id)` seam exists; the active linked PR should be reviewed rather than creating duplicate repair work.
