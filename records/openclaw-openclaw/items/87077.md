## Summary

Keep open: the patch is narrow and source-backed, and current main still routes same-origin non-asset navigations through the service worker. ## What This Changes

The PR adds an early `event.request.mode === "navigate"` return to `ui/public/sw.js`, leaving asset and API/RPC/plugin service-worker branches unchanged.

## Best Possible Solution

Land the narrow navigation bypass after the PR body proof passes the required fields and maintainers accept native HTTP auth prompts over cached app-shell navigation for top-level loads.

## Reproduction Assessment

Yes. Current source sends same-origin non-asset navigations through `event.respondWith(fetch(...))`, and the PR body provides a concrete Firefox plus Caddy reverse-proxy before/after flow; I did not rerun the live browser setup.

## Solution Assessment

Yes, with caveats. Skipping `respondWith()` before cache routing is the narrowest code path to let native 401 challenges reach the browser, but the offline app-shell trade-off needs maintainer acceptance and the proof section needs parseable fields.
