## Summary

Keep this PR open: current main still routes same-origin non-asset Control UI navigations through the service worker, and the proposed bypass is a focused fix with sufficient real browser proof. I found no blocking code-review findings, but merging still needs maintainer acceptance of the offline/PWA navigation fallback tradeoff.

## What This Changes

The PR adds an early `event.request.mode === "navigate"` return in `ui/public/sw.js`, bypassing service-worker handling for top-level Control UI navigations.

## Best Possible Solution

Maintainers should explicitly choose network-direct top-level navigations, or ask for a narrower design that preserves the offline shell while reissuing auth-challenge navigations outside the service worker.

## Reproduction Assessment

Yes. Source inspection on current main shows non-asset same-origin navigations go through `event.respondWith(fetch(...))`, and the PR includes a concrete Firefox/Caddy before-and-after proof path; I did not rerun the live proxy setup.

## Solution Assessment

Yes, if maintainers accept losing offline app-shell navigations. The early return restores the browser-native network path and preserves asset/API/RPC/plugin handling; a 401-only variant would be more complex and should be a maintainer-chosen refinement.
