## Label Justifications

- `P2`: This is a normal-priority Control UI auth/proxy bugfix with limited code surface but real deployment impact.
- `merge-risk: 🚨 compatibility`: The PR removes an existing service-worker fallback path for top-level navigations, which can change behavior for users relying on cached or offline shell loads.

## Summary

This PR is still necessary and should stay open for maintainer review: current `main` still routes top-level Control UI navigations through the service worker, the patch is narrow and matches the reported reverse-proxy auth failure, and no concrete correctness or security defect showed up in review. The remaining question is an intentional compatibility trade-off: top-level navigations will no longer use the cached app-shell fallback when offline.

## What This Changes

The PR adds an early return in `ui/public/sw.js` so same-origin GET top-level navigation requests bypass the Control UI service worker while API, RPC, plugin, asset, cache-versioning, and push-notification paths remain unchanged.

## Best Possible Solution

Land the navigation bypass if maintainers accept that native browser HTTP-auth handling should take precedence over the existing cached app-shell fallback for top-level Control UI navigations.

## Reproduction Assessment

Yes. Current source routes same-origin non-asset navigations through `event.respondWith(fetch(...))`, and the PR body provides a concrete Firefox plus Caddy reverse-proxy reproduction with before/after DevTools evidence; I did not rerun the live browser setup in this read-only review.

## Solution Assessment

Yes, with maintainer sign-off. Skipping `respondWith()` before cache routing is the narrowest way to let native `401` challenges reach the browser, but it deliberately removes the offline app-shell fallback for navigations.
