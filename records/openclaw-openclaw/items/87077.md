## Summary

Keep open: current main still service-worker-mediates same-origin top-level navigations, and this PR is a focused fix with sufficient Firefox/Caddy proof and no blocking code finding. The remaining issue is a compatibility-sensitive maintainer decision because the patch intentionally drops service-worker-served top-level navigation fallback/offline shell behavior.

## What This Changes

The PR adds an early `event.request.mode === "navigate"` return in the Control UI service worker so top-level page loads bypass `respondWith()` while asset and API/RPC/plugin routing stays unchanged.

## Best Possible Solution

Land the narrow navigation bypass after maintainer acceptance of the offline-navigation tradeoff; if offline app-shell navigation remains a product requirement, ask for a design that preserves it without returning auth-challenge responses through the service worker.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main routes same-origin non-asset navigations through `event.respondWith(fetch(...))`, and the PR body supplies a concrete Firefox/Caddy before/after flow. I did not rerun the live reverse-proxy browser setup locally.

## Solution Assessment

Yes, with maintainer acceptance. Returning before `respondWith()` is the narrow code change that lets browser-native 401 challenges run, but maintainers need to accept the shipped offline app-shell navigation tradeoff or request a narrower design.
