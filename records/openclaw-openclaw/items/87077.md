## Summary

Keep open: this is a focused, plausible fix for a source-supported Control UI service-worker/auth bug, but merge should wait for after-fix real browser/proxy proof and maintainer acceptance of the top-level navigation fallback trade-off.

## What This Changes

The PR adds an early return in the Control UI service worker fetch handler so same-origin GET top-level navigations bypass `event.respondWith()` while asset/API/plugin behavior remains on the existing paths.

## Best Possible Solution

Land the narrow navigation bypass only after redacted after-fix browser/reverse-proxy proof and explicit maintainer acceptance of the offline-shell fallback change, or revise to a narrower navigation-auth strategy.

## Reproduction Assessment

Yes, there is a high-confidence reproduction shape: the PR body gives a Caddy plus Firefox restart path, and current source shows navigations fall into the service-worker `respondWith(fetch(...))` branch. I did not run the live reverse-proxy/browser setup, so this is source-reproducible rather than directly reproduced here.

## Solution Assessment

Unclear: bypassing `navigate` before `respondWith()` is a narrow code change that matches the browser service-worker contract, but it also removes the existing navigation fallback. The safest merge path is live after-fix proof plus a maintainer decision on that fallback trade-off.
