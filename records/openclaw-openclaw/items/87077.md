## Label Justifications

- `P2`: This is a normal-priority Control UI auth bugfix for supported reverse-proxy deployments with limited surface area.
- `merge-risk: 🚨 compatibility`: The PR removes an existing service-worker fallback path for top-level navigations, which can change upgrade behavior for users relying on cached/offline shell loads.

## Summary

Keep this PR open: the one-line service-worker bypass is a plausible fix for reverse-proxy HTTP auth challenges, and the updated PR body now supplies sufficient after-fix real-environment proof. It still needs human maintainer sign-off because it intentionally removes the shipped offline app-shell fallback for all top-level navigations.

## What This Changes

The PR adds an early return for same-origin GET navigation requests in the Control UI service worker so top-level page loads bypass `event.respondWith()` while the existing asset, API, RPC, and plugin branches stay unchanged.

## Best Possible Solution

Land the navigation bypass only if maintainers accept that Control UI top-level navigations should prefer native browser auth handling over the existing offline app-shell fallback.

## Reproduction Assessment

Yes. Current `main` routes same-origin non-asset navigations through `event.respondWith(fetch(...))`, and the PR body gives a concrete Firefox plus Caddy reverse-proxy reproduction with before/after DevTools evidence; I did not rerun the live browser setup.

## Solution Assessment

Yes, with maintainer sign-off. Skipping `respondWith()` before the cache branches is the narrowest way to let native `401` challenges reach the browser, but it deliberately removes the offline app-shell fallback for navigations.
