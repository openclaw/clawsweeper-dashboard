## Label Justifications

- `P1`: The PR changes retry behavior for an active user-visible subagent completion workflow that can currently duplicate messages.
- `merge-risk: 🚨 message-delivery`: The unconditional permanent classification can suppress a needed retry and lose a completion message when prior visible delivery is not established.

## Summary

The bug is credible, but the patch suppresses retries by error text without proving prior delivery, while current main already has a safer structured partial-delivery seam.

## What This Changes

The PR classifies the embedded prompt lock-change message as a permanent subagent announcement failure and adds a test asserting the gateway is called once.

## Best Possible Solution

Propagate structured `sentBeforeError` or `visibleReplySent` evidence for the reported send-success-then-lock-error path and keep failures without delivery evidence retryable.

## Reproduction Assessment

Yes at source-and-production-log level: the linked shipped report records repeated Telegram send successes followed by the lock error, and current retry code can repeat failed announcements; this review did not independently run the concurrent race.

## Solution Assessment

No. The current design already separates retryable error classification from structured partial-delivery evidence, so the safest fix should use that state boundary rather than assume the message always means a send succeeded.
