## Summary

Keep open: the PR only shrinks the structured `details` payload, but it still returns the full config JSON as one text `content` block, which the middleware validates against a separate 100,000-character cap. The linked failure can therefore persist, and the PR also lacks after-fix real gateway proof.

## What This Changes

The PR changes the gateway tool's `config.get` branch to return the full config JSON as text while replacing structured `details` with `{ ok: true }`.

## Best Possible Solution

Fix the gateway false failure at the shared middleware content boundary, or otherwise ensure both `content` and `details` stay within validator limits, then add focused regression coverage and redacted real `gateway config.get` proof.

## Reproduction Assessment

Yes from source inspection: a large `config.get` result still produces one oversized text block after this PR, and the middleware separately rejects text over 100,000 characters. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No. The patch handles the structured `details` side but leaves the model-facing `content` over the validator cap, so it is only a partial mitigation for the reported failure.
