## Summary

Keep open. The underlying bug is still source-reproducible on current main, but this branch would drop meaningful Discord component payloads, does not reconcile existing send-validation poll behavior, and lacks required real behavior proof.

## What This Changes

The PR changes poll intent detection, Discord component parsing, poll helper tests, and the changelog to tolerate GPT-5.4/5.5 default-filled `message.send` parameters.

## Best Possible Solution

Land a consolidated normalization that treats model-filled no-op poll/component defaults as absent for plain sends while preserving intentional component metadata and existing poll-only validation; keep media-root policy separate.

## Reproduction Assessment

Yes. Current source shows `action="send"` still reaches the poll guard for nonzero default-filled poll params, and Discord empty modal defaults still reach the parser throw path; no tests were run because this review is read-only.

## Solution Assessment

No. The normalization direction is sound, but this patch is not the best fix until it preserves meaningful Discord component data, reconciles the poll send-validation contract, and includes real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Ignore empty modals without discarding components:** `extensions/discord/src/components.parse.ts:396`
  - body: When `components.modal.fields` is empty and the title is empty, this returns `null` for the entire spec. If the payload also has real `components.text`, `blocks`, or container data, the send path treats components as absent and drops the intended component rendering instead of ignoring only the empty modal.
  - confidence: 0.91
- **[P2] Update the send-validation poll cases:** `src/poll-params.ts:89-90`
  - body: The new semantic gate returns false unless a question or option is present, but `message-action-runner.send-validation.test.ts` still expects duration/public-only and negative-duration send payloads to reject. With this diff those cases resolve as normal sends, so update the runner contract or narrow the guard.
  - confidence: 0.92
- **[P2] Preserve component-only options in the no-op check:** `extensions/discord/src/components.parse.ts:378-379`
  - body: This no-op check ignores fields such as `components.container` and `components.reusable`. A send with message content plus a container accent color currently stays on the component path via content fallback; after this return it falls through to plain send and loses the component styling.
  - confidence: 0.85
