## Summary

Keep open: current main still has the reported poll false-trigger and Discord empty-modal throw paths, but this PR is not merge-ready because real behavior proof is missing and the parser can drop intentional Discord component rendering when an empty default modal accompanies real component content or container styling.

## What This Changes

The PR changes poll intent detection, Discord component parsing, poll/send validation tests, and the changelog so GPT-5.4/5.5 default-filled `message.send` poll/component parameters are treated as absent unless they show real poll or component intent.

## Best Possible Solution

Land a focused normalization that strips only model-filled no-op poll/component defaults, preserves intentional Discord text, blocks, container, reusable, and modal data, and proves a real GPT-style Discord/message send after the fix.

## Reproduction Assessment

Yes, from source inspection. Current main still rejects plain sends when broad poll creation params are detected and still throws on empty Discord modal fields before falling through to ordinary send behavior.

## Solution Assessment

No. The normalization direction is right, but the PR should strip only no-op defaults while preserving real Discord component data and should include real after-fix Discord/message-tool proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Ignore empty modals without dropping real components:** `extensions/discord/src/components.parse.ts:396`
  - body: When `modal.fields` is empty and `modal.title` is empty, this returns `null` for the entire parsed spec. If the same payload has real `components.text` or parsed `blocks`, the direct Discord send path treats components as absent instead of removing only the empty modal, dropping the intended component rendering.
  - confidence: 0.93
- **[P2] Preserve container-styled sends in the no-op check:** `extensions/discord/src/components.parse.ts:378-379`
  - body: The early no-op branch ignores `components.container`, so a caller can send normal top-level content plus a container accent/spoiler and an empty default modal/blocks object and this returns `null`. Current runtime uses top-level content as component text fallback, so this silently downgrades an intentional container-styled message to plain send.
  - confidence: 0.88
