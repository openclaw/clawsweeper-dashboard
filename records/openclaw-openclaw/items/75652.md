## Summary

Keep open: the PR is a plausible fix direction and current main still has the reported poll/component default-filled failure paths, but the current Discord parser change can discard meaningful component specs and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR changes poll intent detection, Discord component parsing, poll helper and send-validation tests, and the changelog to tolerate GPT-5.4/5.5 default-filled `message.send` poll/component parameters.

## Best Possible Solution

Land a consolidated normalization that treats model-filled no-op poll/component defaults as absent for plain sends while preserving intentional Discord component metadata and leaving workspace media access to the canonical security/policy thread.

## Reproduction Assessment

Yes. Source inspection shows current main still rejects `send` when broad poll creation params are detected and still parses object-shaped Discord components before plain sends, including the empty-modal throw path.

## Solution Assessment

No. The normalization direction is sound, but the patch should ignore only no-op defaults while preserving real Discord component text, blocks, container, reusable data, and explicit poll behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Ignore empty modals without discarding components:** `extensions/discord/src/components.parse.ts:397`
  - body: When `modal.fields` is empty and the title is empty, this returns `null` for the entire component spec. If the same payload has real `components.text`, blocks, or container settings plus a model-filled empty modal skeleton, the component send is downgraded to the plain send path and loses intended rendering.
  - confidence: 0.91
- **[P2] Preserve component-only options in the no-op check:** `extensions/discord/src/components.parse.ts:379-381`
  - body: The early no-op check ignores `components.container` and `components.reusable`, so a meaningful container or reusable setting with top-level message content can be treated as absent when blocks/modal/text are empty. That loses component metadata that current component-send tests expect to stay on the component path.
  - confidence: 0.84
