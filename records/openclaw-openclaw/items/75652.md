## Summary

Keep open: the reported message-send breakage is source-reproducible, but this PR is not merge-ready because it can discard valid Discord component intent, leaves the poll send-validation contract inconsistent, and still has no real after-fix behavior proof.

## What This Changes

The PR changes poll intent detection, Discord component parsing, poll helper tests, and the changelog to tolerate GPT-5.4/5.5 default-filled `message.send` poll/component parameters.

## Best Possible Solution

Land a consolidated normalization that treats model-filled no-op poll/component defaults as absent for plain sends while preserving intentional Discord component metadata and locking the chosen poll intent contract in focused runner tests.

## Reproduction Assessment

Yes, from source inspection. Current main still rejects `send` when broad poll creation params are detected and still parses object-shaped Discord components before plain sends, including the empty-modal throw path; PR CI also confirms the poll contract mismatch.

## Solution Assessment

No. The normalization direction is sound, but this patch is not the best fix until it preserves meaningful Discord component data, reconciles the poll send-validation contract, and includes real behavior proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Ignore empty modals without discarding components:** `extensions/discord/src/components.parse.ts:396`
  - body: When `modal.fields` is empty and `modal.title` is empty, this returns `null` for the whole component spec. If the payload also contains real `components.text` or `blocks`, the send path treats components as absent instead of ignoring only the empty modal, which can drop intended component rendering or make a component-text-only send fail content validation.
  - confidence: 0.92
- **[P2] Preserve component-only options in the no-op check:** `extensions/discord/src/components.parse.ts:378-379`
  - body: The early no-op check ignores `components.container` and `components.reusable`. A message with normal content plus a container accent/spoiler or reusable setting currently stays on the component path via content fallback, but this return makes it fall through to plain send and loses that component metadata.
  - confidence: 0.87
- **[P2] Update the send-validation poll cases:** `src/poll-params.ts:89-90`
  - body: The new semantic gate returns false unless a question or option is present, but `message-action-runner.send-validation.test.ts` still expects duration/public-only and negative-duration send payloads to reject. With this diff those cases become normal sends and CI is failing, so update the runner contract tests or narrow the guard to preserve the existing validation behavior.
  - confidence: 0.93
