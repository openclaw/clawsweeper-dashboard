## Summary

The PR addresses a real source-reproducible message-send failure, but it is not merge-ready because the Discord parser changes can drop intentional component payloads and contributor real behavior proof is still missing.

## What This Changes

The PR changes poll intent detection, Discord component parsing, focused send-validation tests, poll helper tests, and the changelog to tolerate GPT-5.4/5.5 default-filled poll and Discord component parameters on plain message sends.

## Best Possible Solution

Normalize only model-filled empty defaults while preserving intentional component payloads, then settle the poll intent contract with focused tests plus real Discord/message-tool proof.

## Reproduction Assessment

Yes, from source inspection: current main rejects send when broad poll creation params are detected and parses GPT-style object components into the empty-modal throw path. I did not run tests because this review is read-only.

## Solution Assessment

No, not yet: the poll direction is plausible, but the Discord fix should ignore only the empty default modal/skeleton instead of returning null for otherwise meaningful component specs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Ignore empty modals without discarding components:** `extensions/discord/src/components.parse.ts:396`
  - body: Returning `null` here drops the whole component spec when a real `components.text`, `blocks`, `container`, or `reusable` payload is accompanied by the same empty modal skeleton GPT emits. That makes intentional component sends fall back to the plain send path instead of preserving their Discord component rendering; skip the empty modal and continue parsing the rest of the spec.
  - confidence: 0.9
- **[P2] Preserve component-only options when detecting defaults:** `extensions/discord/src/components.parse.ts:378-380`
  - body: This early default check only considers `blocks`, `modal`, and `text`, so a caller that intentionally supplies component options such as `container` styling or `reusable` with fallback content is treated as if `components` were absent. The parser should only return `null` when every meaningful component field is empty/default, otherwise the send path can silently lose component delivery semantics.
  - confidence: 0.82
