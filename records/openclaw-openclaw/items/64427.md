## Summary

Keep open. Current main still exposes and documents Slack `dmHistoryLimit`, but the Slack monitor does not derive a DM-specific limit, does not carry one through `SlackMonitorContext`, and still gates prompt/history injection on `isRoomish`, which excludes 1:1 `im` conversations. The closed related #67020 and unmerged #67054 overlap, while open #7359 is broader cross-session Slack visibility work and should not replace this narrower DM history hydration bug.

## What This Changes

The issue asks for Slack 1:1 DM sessions to honor `channels.slack.dmHistoryLimit` by hydrating recent DM conversation history when a fresh DM session starts.

## Best Possible Solution

Implement the fix inside the Slack plugin. Resolve an effective DM history limit from per-DM overrides before the account/provider `dmHistoryLimit`, fetch recent `conversations.history` only for fresh top-level Slack `im` sessions with a positive limit, exclude the current inbound event, format prior user and bot messages through the existing Slack inbound envelope/history contract, and populate both prompt context and structured `InboundHistory`. Keep the broader cross-session visibility concerns from #7359 out of this narrow repair.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with `channels.slack.dmHistoryLimit > 0`, a fresh Slack `im` turn is classified as a direct message, fails the `isRoomish` gates, and no DM-specific limit or history fetch path exists in the Slack monitor.

## Solution Assessment

Yes. A Slack plugin-owned backfill path is the narrowest maintainable solution; the existing core `pi-embedded-runner` helper only limits already-loaded transcripts and cannot add Slack API history to a new or reset DM session.
