## Summary

Keep open. The author removed the incorrect auto-enable early return and now derives candidate IDs, but the PR still returns from any active registry before the loader compatibility and in-flight checks can prove the registry is safe to reuse.

## What This Changes

This PR changes web-provider runtime resolution to map providers from the active gateway registry after deriving candidate plugin IDs, and adds an Unreleased changelog entry for the ARM64 plugin-load performance issue.

## Best Possible Solution

Keep the ARM64 performance optimization, but implement it through the loader compatibility contract: derive the candidate IDs, reuse only a compatible and sufficient active registry, map with those candidates, and preserve fallback/in-flight behavior.

## Reproduction Assessment

Yes. A source-level reproduction is clear from the existing tests: install an empty active registry, then call resolvePluginWebSearchProviders({}) or resolvePluginWebFetchProviders({}); the PR fast path returns the empty active registry result before the loader fallback can return bundled providers.

## Solution Assessment

No. Reusing an active registry is the right performance direction, but this patch is not the best fix because it bypasses the loader’s cache-key, workspace/env, scope, and in-flight guards instead of extending or using them.
