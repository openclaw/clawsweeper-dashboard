## Summary

Keep open. The PR targets a real performance path, and the update removed the unsafe auto-enable early return, but the remaining active-registry shortcut is still too broad: it can return providers from any active registry before current main's compatibility guard and loader fallback, which can drop web providers when the active registry is empty or missing the candidate providers.

## What This Changes

The PR changes web-provider runtime resolution to derive candidate plugin IDs and map providers from the active gateway registry when present, plus an Unreleased changelog entry for the ARM64 tool-creation performance fix.

## Best Possible Solution

Keep the ARM64 performance optimization, but make the active-registry shortcut prove compatibility or provider sufficiency before returning. If that proof fails, preserve the existing in-flight and loadOpenClawPlugins fallback path, with focused regression tests for empty active registries, origin/scope filtering, and compatible active-registry reuse.

## Reproduction Assessment

Yes. A source-level reproduction is clear from the existing tests: install an empty active registry, then call resolvePluginWebSearchProviders({}) or resolvePluginWebFetchProviders({}); with the PR fast path, those calls can return the empty registry's providers before the loader fallback runs.

## Solution Assessment

No. Reusing the active gateway registry is the right performance direction, but this implementation is not yet the best fix because it treats registry presence as sufficient instead of honoring the existing compatibility contract or checking that the active registry contains the derived candidate providers.
