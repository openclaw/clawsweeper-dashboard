## Summary

Keep open. The follow-up commit addressed the earlier origin-filtering, auto-enable, and changelog concerns, but the remaining web-provider fast path still bypasses the current active-registry compatibility guard and can return an empty or incompatible active registry instead of falling back to the scoped provider load.

## What This Changes

This PR changes web-provider runtime resolution to derive candidate plugin IDs and map providers from the active plugin registry when available, and adds an Unreleased changelog entry for the ARM64 tool-creation performance fix.

## Best Possible Solution

Keep the ARM64 performance optimization, but make the active-registry shortcut prove compatibility or provider sufficiency before returning. If it cannot prove that, preserve the existing in-flight and loadOpenClawPlugins fallback path, with focused regression tests for empty active registries, origin/scope filtering, and compatible active-registry reuse.

## Reproduction Assessment

Yes. A source-level reproduction is clear: install an empty active registry as the existing web-search and web-fetch tests do, then call resolvePluginWebSearchProviders({}) or resolvePluginWebFetchProviders({}); the PR fast path returns no providers before the loader fallback can run.

## Solution Assessment

No. Reusing the active registry is the right performance direction, but this implementation is still too broad because it treats any active registry as usable instead of honoring the loader compatibility contract or checking that the active registry contains the derived candidate providers.
