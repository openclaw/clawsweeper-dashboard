## Summary

Keep open. The ARM64 performance problem appears valid and current main has not absorbed this PR, but the proposed patch bypasses web-provider candidate/origin/compatibility filtering and skips configured plugin-entry allowlist materialization in an empty-candidate case, so it needs targeted changes before merge.

## What This Changes

The PR adds an active-registry fast path for web provider resolution and an empty-candidate early return for plugin auto-enable materialization to reduce repeated plugin registry loads.

## Best Possible Solution

Keep the performance goal, but implement it through the existing plugin scope and compatibility contracts: derive candidate plugin ids first, preserve origin/workspace/config filtering and the in-flight guard, reuse the active registry only when it is compatible or safely filterable, and skip the second manifest load only when configured-entry allowlist materialization is truly unnecessary. Add focused regression tests plus an Unreleased changelog fix entry.

## Reproduction Assessment

Yes. The web-provider regression is reproducible by resolving bundled web providers while an active registry also contains an external provider; the proposed fast path can return the external provider because it only filters by params.onlyPluginIds. The auto-enable regression is reproducible with empty candidates, restrictive plugins.allow, and a known configured plugin entry without passing manifestRegistry; current main materializes the allowlist, while the PR would return early.

## Solution Assessment

No. Reusing already-loaded plugin data is the right direction for ARM64, but this implementation bypasses existing candidate and compatibility checks and treats an empty candidate list as a universal no-op. A safer fix should preserve the current selection semantics while avoiding the redundant load only after those semantics are accounted for.
