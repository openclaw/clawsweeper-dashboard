## Summary

Keep open. The ARM64 performance problem is plausible and the PR targets the right hot paths, but the proposed shortcuts bypass existing compatibility and candidate-filtering contracts and can change provider selection/config behavior before merge.

## What This Changes

The PR adds an active-registry fast path to web provider resolution and an empty-candidate early return in plugin auto-enable materialization to reduce repeated plugin registry loads.

## Best Possible Solution

Keep the performance goal, but route the optimization through the existing compatibility and candidate-resolution contracts: derive the candidate plugin ids first, respect origin and explicit scopes, reuse the active registry only when it is provably compatible or safely filterable, and preserve configured plugin-entry allowlist materialization for empty candidate sets. Add focused regression tests and an Unreleased changelog fix entry.

## Reproduction Assessment

Yes. The regression risks are reproducible by source-level paths: call web provider resolution with origin-filtered params while an active registry exists, and call auto-enable materialization with empty candidates plus restrictive plugins.allow and configured plugin entries.

## Solution Assessment

No. The proposed direction is narrower than a full reload, but the implementation is too broad; a safer fix should keep the existing load-option compatibility check or add a full-registry compatibility helper that still applies candidate filtering.

- **[P2] Preserve web provider candidate filtering:** `src/plugins/web-provider-runtime-shared.ts:155-160`
  - body: This fast path returns providers from the active registry using only `params.onlyPluginIds`.
