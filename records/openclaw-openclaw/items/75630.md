## Summary

Keep open. Current main still lets unresolved Slack thread replies proceed as ordinary messages, and this PR addresses that gap on top of the merged SDK-backed participation store without a blocking correctness or security finding from this read-only pass.

## What This Changes

The PR marks unresolved Slack thread replies as ambiguous, drops them before root-message handling, wires thread-continuation mention gating through existing participation state, and updates focused tests plus the changelog.

## Best Possible Solution

Review and land this focused Slack fix after exact-head CI or the listed targeted tests pass, keeping the ambiguity handling in Slack and the shared mention-gating change backwards-compatible.

## Reproduction Assessment

Yes. On current main, a Slack event with parent_user_id, no thread_ts, and a history lookup that returns no thread result or throws returns the original message; prepareSlackMessage then has no ambiguity drop before mention/root handling.

## Solution Assessment

Yes. The PR reuses the existing SDK-backed participation store, avoids the earlier competing JSON-store and conversations.replies fallback paths, and drops only unresolved ambiguous replies before normal message handling.
