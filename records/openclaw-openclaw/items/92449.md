## Summary

Keep open: current main and the latest release still lack WhatsApp configured ACP binding materialization, and the linked fix PR is open but not merged, so this issue remains necessary.

## What This Changes

The issue reports that WhatsApp direct messages matching `bindings[].type = "acp"` select the configured agent but execute on the gateway host because the configured ACP binding is not materialized.

## Best Possible Solution

Land or replace the linked WhatsApp-owned fix so the channel exposes configured-binding compile/match support, rewrites matching direct conversations to the ACP session before dispatch, and fails closed when readiness fails.

## Reproduction Assessment

Yes, source-level. Current main's WhatsApp handler resolves only `resolveAgentRoute` for direct inbound messages, and the plugin exposes no configured-binding provider while the registry requires `compileConfiguredBinding` and `matchInboundConversation`.

## Solution Assessment

Yes, the proposed direction is the right owner-boundary fix: add a WhatsApp-owned adapter and use the existing configured-binding readiness lifecycle. A new config surface or local fallback would be worse than honoring the existing ACP binding contract.
