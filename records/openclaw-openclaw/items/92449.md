## Summary

Keep this open: current main still lacks the WhatsApp configured-binding adapter and inbound ACP route-readiness handoff, and the direct fixing PR is open but unmerged.

## What This Changes

The issue reports that WhatsApp direct messages matching a `type: "acp"` binding select the configured agent but do not materialize the configured ACP binding, so execution continues through the gateway host route.

## Best Possible Solution

Land the focused WhatsApp fix that implements the existing configured-binding adapter contract and rewrites matching inbound turns to the configured ACP session with fail-closed readiness and direct-message regression coverage.

## Reproduction Assessment

Yes for source-level reproduction: current main's WhatsApp plugin lacks the configured-binding hooks required by the registry, and the inbound handler dispatches the ordinary WhatsApp route. I did not run a live WhatsApp-to-ACP setup in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is to use the established channel configured-binding adapter plus configured-route readiness lifecycle in the WhatsApp-owned path, not to add a new config surface or a WhatsApp-only fallback.
