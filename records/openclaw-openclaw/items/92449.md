## Summary

Keep this open: current main and v2026.6.6 still lack the WhatsApp configured-binding provider and inbound ACP readiness path, while https://github.com/openclaw/openclaw/pull/92513 is an open, unmerged fix candidate for this issue.

## What This Changes

The issue reports that WhatsApp `type: "acp"` bindings do not materialize configured ACP sessions and can silently fall back to local embedded execution.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92513, or replace it with an equivalent WhatsApp-owned configured-binding adapter plus inbound fail-closed ACP readiness path and direct-message regression coverage.

## Reproduction Assessment

Yes, source-level. Current main lacks the WhatsApp configured-binding provider and the inbound handler dispatches the ordinary `resolveAgentRoute` result, so the reported null configured-binding resolution is expected without needing a live WhatsApp account.

## Solution Assessment

Yes. The maintainable fix direction is to reuse the established channel configured-binding adapter plus `resolveConfiguredBindingRoute` and fail-closed readiness lifecycle, not to add a WhatsApp-only fallback or new config surface.
