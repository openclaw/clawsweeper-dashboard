## Summary

Current `main` still has the reported gap: WhatsApp resolves and dispatches an ordinary channel route, while configured ACP resolution requires channel hooks that WhatsApp does not provide. Sibling Telegram code both materializes the ACP binding and stops dispatch when initialization fails, so the ACP-specific problem remains distinct from the ordinary route-binding report at https://github.com/openclaw/openclaw/issues/75211 and no open or merged matching fix PR was found.

## What This Changes

Make matching WhatsApp conversations materialize and use their configured ACP binding instead of silently executing through the ordinary embedded agent route.

## Best Possible Solution

Add WhatsApp-owned direct/group conversation compilation and matching, then resolve and ensure the shared configured ACP binding immediately after route selection so every downstream consumer receives one canonical ACP-bound route and initialization failure aborts local dispatch.

## Reproduction Assessment

Yes at source level with high confidence: current main lacks WhatsApp’s mandatory configured-binding hooks and never upgrades the ordinary route before message processing. The reporter also supplies a concrete resolver probe and sanitized live observation, although this review did not repeat the live transport run.

## Solution Assessment

Yes in architecture: use the existing Plugin SDK configured-binding resolver/readiness lifecycle while keeping WhatsApp identity normalization plugin-owned. A warning-only change or adapter-only patch would not fully correct the supported ACP configuration path.
