## Summary

Keep this open: current `main` and the latest release still lack the WhatsApp configured-binding adapter and inbound ACP readiness path, and the direct fixing PR is open but unmerged.

## What This Changes

The issue reports that WhatsApp `type: "acp"` bindings are not materialized, so a configured ACP-bound WhatsApp conversation can execute through the local embedded route instead of the intended ACP backend.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92513, or replace it with an equivalent WhatsApp-owned configured-binding adapter plus inbound fail-closed ACP readiness path and direct-message regression coverage.

## Reproduction Assessment

Yes. Source inspection on current `main` shows WhatsApp inbound uses only `resolveAgentRoute` and the plugin has no configured-binding provider, so the reporter's `resolveConfiguredAcpBindingRecord(...) => null` result is expected without needing a live WhatsApp account.

## Solution Assessment

Yes. The maintainable fix direction is to reuse the established channel configured-binding adapter plus `resolveConfiguredBindingRoute` and fail-closed readiness lifecycle, not to add a WhatsApp-only fallback or new config surface.
