## Summary

Current `main` still reproduces the reported architectural gap: WhatsApp inbound handling creates only the ordinary agent route/session, and its channel plugin lacks the configured-binding adapter required to materialize an ACP record. The related route-binding report at https://github.com/openclaw/openclaw/issues/75211 remains open but covers ordinary `type: "route"` selection rather than this distinct post-selection ACP execution lifecycle, so this issue is still necessary.

## What This Changes

Make matching WhatsApp `type: "acp"` bindings materialize and dispatch through the configured ACP runtime instead of the gatewayâ€™s embedded runtime.

## Best Possible Solution

Add a WhatsApp-owned conversation normalizer and configured-binding adapter, resolve and ensure the ACP binding before reply dispatch, fail closed with a clear diagnostic when readiness fails, and add direct-message coverage proving the ACP session key replaces the embedded route.

## Reproduction Assessment

Yes. Current source deterministically follows the ordinary WhatsApp route because the channel lacks the adapter required to resolve a configured ACP record, matching the reporterâ€™s null resolver result, normal session key, and gateway-host execution evidence.

## Solution Assessment

Yes in architectural direction. Reusing the established channel adapter and fail-closed ACP readiness lifecycle is narrower and more maintainable than adding a WhatsApp-specific fallback or another configuration surface.
