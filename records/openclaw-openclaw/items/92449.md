## Summary

Keep this open: current main and the latest release still lack WhatsApp configured ACP binding materialization, and the direct fixing PR is open but unmerged.

## What This Changes

The issue reports that WhatsApp `type: "acp"` bindings can select the intended agent while failing to materialize the configured ACP session, so turns continue on the gateway host's ordinary WhatsApp route.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92513, or replace it with an equivalent WhatsApp-owned configured-binding adapter plus inbound fail-closed ACP readiness path and direct-message regression coverage.

## Reproduction Assessment

Yes. Source inspection on current main shows WhatsApp inbound uses only `resolveAgentRoute` and the plugin has no configured-binding provider, so the reporter's `resolveConfiguredAcpBindingRecord(...) => null` result is expected without a live WhatsApp account.

## Solution Assessment

Yes. The maintainable fix direction is to use the established channel configured-binding adapter plus shared route rewrite and fail-closed readiness lifecycle, without adding a WhatsApp-only fallback or new config surface.
