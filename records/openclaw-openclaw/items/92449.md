## Summary

Keep this open: current `main` and latest release `v2026.6.6` still lack WhatsApp configured ACP binding support, and the direct fixing PR is open but unmerged.

## What This Changes

The issue reports that WhatsApp direct conversations matching `type: "acp"` bindings resolve the agent but do not materialize the configured ACP session, so execution can fall back to the gateway host.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92513, or replace it with an equivalent WhatsApp-owned configured-binding adapter plus fail-closed ACP readiness before dispatch.

## Reproduction Assessment

Yes. Source inspection on current `main` shows WhatsApp inbound uses the ordinary `resolveAgentRoute` route and the plugin lacks the configured-binding adapter required for `resolveConfiguredAcpBindingRecord` to match WhatsApp conversations.

## Solution Assessment

Yes. The best fix direction is to add the WhatsApp channel configured-binding adapter and run the established configured-binding route/readiness lifecycle before processing the turn, rather than adding a new config option or local fallback.
