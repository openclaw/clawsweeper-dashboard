## Label Justifications

- `impact:security`: The report concerns a sandbox or remote-VM execution boundary being bypassed by local embedded execution.

## Summary

Keep this open: current main and the latest release still lack WhatsApp configured ACP binding materialization, and the direct fixing PR is open but unmerged.

## What This Changes

The issue requests that WhatsApp `type: "acp"` configured bindings materialize and route through the configured ACP backend, or fail closed instead of falling back to local embedded execution.

## Best Possible Solution

Review and land `https://github.com/openclaw/openclaw/pull/92513`, or replace it with an equivalent WhatsApp-owned configured-binding adapter plus inbound fail-closed ACP readiness path and focused direct-message regression coverage.

## Reproduction Assessment

Yes, at source level. Current main’s WhatsApp plugin has no configured-binding provider and its inbound handler only builds the ordinary `resolveAgentRoute` path before dispatch, so the reporter’s `resolveConfiguredAcpBindingRecord(...) => null` result is expected without a live WhatsApp account.

## Solution Assessment

Yes. The maintainable solution is to reuse the existing channel configured-binding adapter and `resolveConfiguredBindingRoute`/readiness seam, not to add a WhatsApp-only fallback or new config surface.
