## Summary

Keep open. The report matches a plausible terminal-path bug: current main has live WebChat merge coverage and adjacent session-repair protections, but I did not find a guard that recovers or flags a tool-use turn when pre-tool assistant text exists and the expected post-tool final assistant message is never written.

## What This Changes

The issue requests that final assistant text after the last tool result in a multi-tool WebChat turn be persisted, rendered, and included in next-turn context.

## Best Possible Solution

Add a regression-backed runtime guard so a tool-use turn cannot finish successfully with only pre-tool text after tool results; it should continue or retry from the latest tool result when replay-safe, or surface the existing incomplete-turn warning with replay-invalid liveness metadata.

## Reproduction Assessment

We have a source-level reproduction path but not a live current-main reproduction. The reported JSONL shape maps to current code where pre-tool assistant text creates a nonzero payload count, so incomplete-turn fallback can be skipped even if no post-tool assistant message is written.

## Solution Assessment

The best fix is not a WebChat-only refresh. The durable fix belongs in the agent terminal/transcript path so rendered UI history and next-turn context agree on the same persisted assistant result.
