## What This Changes

The branch adds shared Swift and TypeScript shell inline-command parsing, routes macOS allowlist and node-host shell-wrapper resolution through it, fails closed for startup/env-modified wrapper forms, and adds parity/regression fixtures.

## Best Possible Solution

Keep the PR open until blocked shell wrappers are denied or carried through approval/risk analysis as blocked, the changelog records the user-facing hardening, redacted macOS or Swift behavior proof is attached, and exact-head checks pass.

## Reproduction Assessment

Yes. Current main source shows the Swift wrapper parser only accepts exact second-token POSIX inline flags while the validator handles combined `c` flags, and exact-head CI reproduces the PR regression through failing approval-plan and command-analysis tests; I did not run a macOS live scenario.

## Solution Assessment

No. Sharing parser logic is a reasonable direction, but this implementation drops blocked `sh`/`dash` login-shell payloads to `null` in generic Node parsing, letting downstream approval and risk code treat them as no-payload instead of fail-closed.
