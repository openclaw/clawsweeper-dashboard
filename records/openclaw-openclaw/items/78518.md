## Summary

Keep open: current main has the reported macOS allowlist parser gap, but this PR is protected by the `maintainer` label, lacks real behavior proof, and introduces a regression for documented `bash -lc`/`zsh -lc` shell wrappers.

## What This Changes

The PR adds a Swift inline-command parser, rewires macOS shell wrapper and validator parsing for combined POSIX `c` flags, adds fail-closed startup-option handling, and expands allowlist regression tests.

## Best Possible Solution

Land a maintainer-reviewed patch that preserves documented `-lc` wrapper support, handles combined POSIX `c` flags, keeps startup-file cases fail-closed, updates the changelog, and includes macOS Swift or runtime proof.

## Reproduction Assessment

Yes, from source inspection: current main only recognizes exact POSIX inline flags in the wrapper parser, so a combined form such as `bash -xc /usr/bin/printf` falls back to the shell executable instead of the payload. I did not execute the Swift test lane.

## Solution Assessment

No, not as currently proposed: sharing the parser is a narrow maintainable direction, but the added login-startup guard blocks supported `bash -lc`/`zsh -lc` wrappers and the PR still lacks real behavior proof.
