## Summary

Keep open. Current main and the latest release still clear self-declared scopes for ordinary device-less shared-token/password operator WebSocket clients, and preserving those scopes is a security-sensitive gateway auth policy decision rather than a cleanup close.

## What This Changes

The issue asks OpenClaw to preserve operator scopes for device-less shared-token/password WebSocket API clients authenticated with the gateway shared secret.

## Best Possible Solution

Define an explicit headless API-client scope contract, preferably configured or device-token-backed, while keeping self-declared device-less WebSocket scopes fail-closed by default.

## Reproduction Assessment

Yes. Source and the checked-in compatibility baseline show token/password connects with `device: null` and requested operator scopes are accepted, then scoped RPCs fail with `missing scope`; I did not run a live gateway in this read-only review.

## Solution Assessment

No. Preserving client-declared scopes whenever shared auth succeeds is not the safest fix; a reviewed config-granted or paired-device scope model would make the trust boundary explicit.
