## Summary

Keep open. Current main and the latest release still intentionally clear self-declared scopes for ordinary device-less shared-token/password operator clients, while preserving scopes only for the narrow direct-local backend path; changing that boundary needs maintainer product/security review, especially with #52059 already open as one possible direction.

## What This Changes

The issue asks OpenClaw to preserve requested operator scopes for shared-token/password WebSocket API clients that authenticate without device identity.

## Best Possible Solution

Define an explicit headless API-client contract: either add reviewed config-granted or device-token-backed operator scopes for device-less clients, or document that external shared-token/password WebSocket clients must use device identity while the backend exception stays internal-only.

## Reproduction Assessment

Yes. Checked-in tests provide a high-confidence reproduction path: connect with valid shared token/password, `device: null`, and requested operator scopes, then observe `hello-ok` grants no scopes and scoped RPCs fail with `missing scope`.

## Solution Assessment

Unclear. The suggested direct `sharedAuthOk` preservation restores the old behavior, but it is not clearly the safest fix because it would trust self-declared operator scopes without device identity; a narrower configured-scope or device-token contract needs maintainer review.
