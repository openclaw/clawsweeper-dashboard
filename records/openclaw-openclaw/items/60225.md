## Summary

Keep open. Current main still clears self-declared operator scopes for device-less shared token/password WebSocket clients outside a narrow backend self-connect path, and the remaining decision is whether to support this headless API-client mode despite the current device-binding policy.

## What This Changes

The issue asks OpenClaw to preserve operator.read/operator.write scopes for shared-token operator WebSocket API clients that do not present device identity.

## Best Possible Solution

Define a secure, documented headless operator API auth path, likely device-paired clients/device tokens or a narrowly scoped backend exception, then align connect policy, tests, and gateway docs.

## Reproduction Assessment

Yes. Current source and tests provide a high-confidence source-level reproduction: a device-less non-backend operator connection authenticated by shared token/password is accepted with scopes cleared, so scoped calls fail with missing scope.

## Solution Assessment

No, not as a direct restore of the old sharedAuthOk guard. That would conflict with current docs and tests that intentionally clear unbound scopes, so maintainers should first choose the supported auth boundary for headless API clients.
