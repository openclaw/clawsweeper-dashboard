## Summary

Keep open. Current main and the latest release still clear self-declared scopes for ordinary device-less shared-token/password operator WebSocket clients, and changing that boundary needs explicit maintainer security/product review rather than cleanup closure.

## What This Changes

The issue asks for shared-token/password operator API clients without device identity to retain requested operator scopes instead of connecting with empty scopes.

## Best Possible Solution

Define a reviewed headless API-client contract, likely via configured or device-token-backed scopes for shared-token/password clients, while keeping self-declared device-less scopes fail-closed by default.

## Reproduction Assessment

Yes. The checked-in compatibility baseline reproduces the behavior by connecting with valid token/password auth, `device: null`, and requested operator scopes, then observing scoped RPC failure with `missing scope`.

## Solution Assessment

Unclear. The suggested `sharedAuthOk` preservation restores the old behavior but is not clearly the safest fix; a config-granted or device-token-backed scope model would preserve the security boundary more explicitly.
