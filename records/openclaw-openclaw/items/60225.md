## Summary

Close as a duplicate/satellite: the reported symptom still exists, but the remaining work is now owned by the open canonical gateway token-scope design thread and maintainer PR rather than this separate regression report.

## What This Changes

The issue asks for device-less shared-token/password Gateway API clients to retain operator scopes, or an equivalent supported path for headless API clients.

## Best Possible Solution

Keep https://github.com/openclaw/openclaw/issues/80836 and https://github.com/openclaw/openclaw/pull/81792 as the canonical path for bounded shared-token scope grants, with docs, tests, and security review aligned there.

## Reproduction Assessment

Yes, source-level. Current main accepts a device-less shared-token operator connection, clears requested scopes, and checked-in tests expect follow-up scoped RPCs to fail with `missing scope`.

## Solution Assessment

No for the direct `sharedAuthOk` preservation suggested here. The safer maintainable solution is the canonical configured token-scope contract with explicit privileged-scope bounds instead of trusting client-declared scopes by default.
