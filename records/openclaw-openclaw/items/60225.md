## Summary

Keep open. Current main still clears scopes for ordinary device-less shared-token/password operator clients, and the requested direct preservation would change a security-sensitive gateway auth boundary that already has an open config-based proposal in review.

## What This Changes

Request to preserve operator scopes for device-less shared-token/password WebSocket API clients instead of clearing them after shared auth succeeds.

## Best Possible Solution

Define an explicit headless API-client contract: reviewed config-granted token/password scopes or a device-token-backed flow, with docs and regression tests; otherwise document that external API clients need device identity.

## Reproduction Assessment

Yes. Source and checked-in tests give a high-confidence path: connect with valid shared token/password, `device: null`, and requested operator scopes, then observe empty granted scopes and `missing scope` on scoped RPCs.

## Solution Assessment

Unclear. The suggested direct `sharedAuthOk` preservation would restore the old behavior, but it is not clearly the safest solution because it would trust device-less clients' self-declared scopes instead of a reviewed scope source.
