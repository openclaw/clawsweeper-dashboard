## Summary

Keep open. Current main and v2026.4.29 still clear requested operator scopes for ordinary device-less shared-token/password WebSocket API clients, while preserving scopes only for the documented direct-loopback internal backend path; changing that behavior is an auth-boundary decision that needs maintainer product/security review.

## What This Changes

The issue requests restoring requested operator scopes for headless operator API clients that authenticate with the shared gateway token/password and omit device identity.

## Best Possible Solution

Define the supported external headless API-client contract explicitly: either add a narrow, documented token/password auth path that binds operator scopes without browser device identity and covers it with gateway protocol tests, or document that external WebSocket API clients must use device identity/device tokens while keeping the backend exception internal-only.

## Reproduction Assessment

Yes. Without running tests in this read-only review, current source and checked-in tests provide a high-confidence path: connect as an operator with valid shared token/password, request operator scopes, send `device: null`, then observe the connection succeeds but scoped RPCs fail with `missing scope`.

## Solution Assessment

Unclear. The issue's direct `sharedAuthOk` preservation would restore the old behavior, but it is not clearly the safest fix because it would let every device-less shared-secret operator client retain self-declared scopes; a narrower explicit API-client or device-token contract is safer to review.
