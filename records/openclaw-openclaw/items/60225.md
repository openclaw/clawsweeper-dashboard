## Summary

Keep open. Current main and v2026.5.7 still clear self-declared scopes for ordinary device-less shared-token/password operator WebSocket clients, while docs and tests only preserve narrower trusted paths; the remaining change needs maintainer security/product review.

## What This Changes

The issue asks OpenClaw to preserve operator scopes for device-less operator WebSocket API clients authenticated with the shared gateway token or password.

## Best Possible Solution

Define a reviewed headless API-client contract, likely through configured or device-token-backed scopes, while keeping self-declared device-less scopes fail-closed by default.

## Reproduction Assessment

Yes. Source and checked-in tests give a high-confidence reproduction path: connect as an operator with valid shared token/password auth, `device: null`, and requested operator scopes, then observe scoped RPC failure with `missing scope`.

## Solution Assessment

Unclear. The suggested `sharedAuthOk` preservation restores the old behavior but is not clearly the safest fix; a config-granted or device-token-backed scope contract would preserve the authorization boundary more explicitly.
