## Summary

Keep open: this is a focused security hardening PR for the OpenRouter video catalog path. Current main still reads that provider-controlled success body with `response.json()`, and the proposed SDK helper matches existing bounded-response helper coverage with no blocking review findings.

## What This Changes

The PR routes the OpenRouter video model catalog success JSON body through the shared bounded provider JSON reader instead of `response.json()`.

## Best Possible Solution

Land the bounded read if maintainers accept the 16 MiB provider JSON cap for OpenRouter video catalog discovery.

## Reproduction Assessment

Yes. Current main has a clear source-level reproduction path: `fetchOpenRouterVideoModels` reads the successful response with `response.json()`, and the PR body includes terminal proof showing the bounded reader stops early while the old path drains the full oversized body.

## Solution Assessment

Yes. Reusing `readProviderJsonResponse` from the provider HTTP SDK is the narrowest maintainable fix because it keeps extension code on the public plugin boundary and shares the existing cap, cancellation, and malformed JSON behavior.
