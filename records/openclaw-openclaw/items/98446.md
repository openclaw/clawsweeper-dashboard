- `P2`: This is a normal-priority user-visible OpenShell sandbox media/startup compatibility bug with a bounded but real deployment impact.
- `impact:security`: The fix must preserve sandbox and host file-read boundaries while allowing media paths from the active OpenShell workspace root.
- `impact:message-loss`: Generated or selected media from an OpenShell sandbox can fail to leave the sandbox and reach the target channel.

## Summary

Keep open: current main and the latest release still have the source-backed mismatch where OpenShell documents and validates `/sandbox` as its workspace root while shared sandbox media normalization only maps the fixed `/workspace` container root.

## What This Changes

The issue reports that OpenShell’s configured `/sandbox` workspace root is incompatible with outbound/media path handling that assumes `/workspace`.

## Best Possible Solution

Derive sandbox media path normalization from the active sandbox container workdir or backend-declared workspace roots, and use the existing sandbox filesystem bridge/read boundary so OpenShell `/sandbox/...` media is allowed without broadening host path access.

## Reproduction Assessment

Yes, at source level: OpenShell’s default and validated remote workspace is `/sandbox`, while sandbox media normalization only maps `/workspace` before enforcing sandbox-root checks. I did not run the live Kubernetes/OpenShell transfer path in this read-only review.

## Solution Assessment

Yes, the report points at the right boundary: media path mapping should follow the active sandbox backend/workdir contract instead of asking users to configure unsupported `/workspace` or adding a broad root fallback.
