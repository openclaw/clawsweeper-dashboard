## Summary

Keep open: current main still lacks Telegram Business runtime support. This remains the canonical tracker because closed duplicate reports point here, the target-repo implementation PR was closed unmerged, and an open fork PR still uses closing syntax while current code only covers update delivery, not processing or replies.

## What This Changes

Request to add Telegram Business update handling so bots can receive `business_message` and `business_connection` updates and reply from Business-connected personal chats.

## Best Possible Solution

Keep this as the canonical tracker and ship a refreshed Telegram plugin implementation with account-scoped Business connection state, Business message routing, dedupe/sequencing support, safe outbound `business_connection_id` handling, tests, docs, and live proof.

## Reproduction Assessment

Yes. Source inspection on current main shows Business update names can be delivered, but there are no `business_message` or `business_connection` handlers and no outbound `business_connection_id` mapping, so the reported Business chat flow cannot work through the current pipeline.

## Solution Assessment

Yes on the owner boundary: the Telegram plugin is the right place to solve this. A minimal `allowed_updates` change is not enough; the maintainable fix needs runtime handlers, dedupe/sequencing, durable account-scoped connection mapping, outbound payload handling, docs, tests, and live proof.
