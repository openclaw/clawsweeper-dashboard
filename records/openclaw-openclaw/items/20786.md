## Summary

Keep open: current main and the latest shipped tag still admit Telegram Business update names through grammY defaults, but the Telegram plugin does not process Business updates or attach `business_connection_id` for replies. This remains the canonical tracker because the target-repo implementation was closed with substantive review concerns and an open fork PR still uses closing syntax.

## What This Changes

Add Telegram Business support so Business-connected personal chats deliver `business_message` updates through OpenClaw and replies use the correct `business_connection_id`.

## Best Possible Solution

Ship a refreshed Telegram plugin implementation with account-scoped durable Business connection state, Business message routing through the existing DM pipeline, Business-aware dedupe/sequencing, safe outbound `business_connection_id` injection, docs, focused tests, and redacted live Telegram Business proof.

## Reproduction Assessment

Yes, at source level. Current main admits Telegram Business update names through grammY defaults but registers no Business handlers or outbound `business_connection_id` mapping, so a Telegram Business chat cannot enter the OpenClaw Telegram pipeline or receive supported replies.

## Solution Assessment

Yes on the product boundary, but not as an allowed-updates-only patch. The Telegram plugin is the right owner, and the maintainable fix must include connection state, Business-aware dedupe/sequencing, outbound reply mapping, access-control documentation, tests, and live proof.
