## Summary

Keep open because current main still lacks Telegram Business runtime handling, outbound business_connection_id support, Business update dedupe/sequencing, and docs; the target-repo PR was closed unmerged and a fork PR using closing syntax remains open.

## What This Changes

Add Telegram Business support so OpenClaw can receive business_connection/business_message updates and reply to Business-connected personal chats.

## Best Possible Solution

Ship a Telegram plugin implementation that handles Business connect/disconnect state per account, routes business_message through the existing DM pipeline, updates dedupe/sequential keys, applies business_connection_id only where appropriate, and documents the setup and access-control model.

## Reproduction Assessment

Yes. Source inspection against current main shows the Telegram plugin can allow update delivery through grammY defaults, but no Business handler routes business_message into handleInboundMessageLike and no outbound business_connection_id mapping exists.

## Solution Assessment

Yes for the requested direction: the fix belongs in the Telegram plugin, not generic core. The current open fork PR or a replacement needs maintainer review for account-scoped state, disconnect handling, outbound method support, tests, and docs.
