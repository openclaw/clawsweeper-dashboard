## Summary

Keep open because current main still lacks Telegram Business runtime handling: no `business_message`/`business_connection` handlers, no Business update dedupe/sequencing, no outbound `business_connection_id` routing, and no docs/tests covering the feature. The closed target-repo PR is unmerged, while the open fork PR still uses closing syntax and should be reviewed, imported, or replaced rather than closing the canonical tracker.

## What This Changes

Add Telegram Business Bot support so OpenClaw can receive, process, and reply to `business_message` and `business_connection` updates in the Telegram plugin.

## Best Possible Solution

Ship a Telegram plugin implementation that handles Business connection state, routes `business_message` through the existing DM pipeline, updates dedupe/sequential keys, applies `business_connection_id` only where supported, and documents setup plus access-control behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, from source inspection: current main has no Business handlers or outbound `business_connection_id` mapping, so Business updates cannot reach the existing inbound pipeline or reply path.

## Solution Assessment

Is this the best way to solve the issue? Yes for the requested direction: the fix belongs inside the Telegram plugin boundary, but the fork PR or a replacement still needs maintainer review for state persistence, access control, outbound coverage, tests, and docs.
