## Summary

Keep open. Current main still only allows Telegram Business update names; it does not register Business handlers, dedupe/sequence Business payloads, persist or apply business_connection_id for replies, or document the flow, and an open fork implementation candidate should be reviewed or replaced before this tracker closes.

## What This Changes

Support Telegram Business bots by accepting business_connection/business_message updates, routing Business messages like DMs, and preserving business_connection_id for replies.

## Best Possible Solution

Ship a Telegram plugin change that handles Business connection/message updates before normal message handling, routes Business messages through the existing authorization and inbound pipeline, persists account-scoped connection state for supported replies, and adds tests plus Telegram docs.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the gap: current main can allow Business update names, but no Business handler passes business_message into handleInboundMessageLike and no outbound business_connection_id mapping exists.

## Solution Assessment

Yes for the requested direction: this belongs in the Telegram plugin rather than core. The safe path is to review and land #1324 or replace it with an equivalent implementation that addresses state persistence, account isolation, outbound method support, tests, and docs.
