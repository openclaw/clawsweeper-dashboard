## Summary

Keep open: current main still admits Telegram Business update names through grammY defaults, but it does not process `business_message` or `business_connection`, does not add Business-aware dedupe/sequencing, and does not attach `business_connection_id` for outbound replies. The open fork PR at https://github.com/BingqingLyu/openclaw/pull/1324 still uses closing syntax for this issue, so the tracker should stay open until a corrected implementation lands or is explicitly replaced.

## What This Changes

The issue requests Telegram Business bot support so Business-connected personal chats can be received and answered through the Telegram plugin.

## Best Possible Solution

Ship a refreshed Telegram plugin implementation with account-scoped durable Business connection state, Business message routing through the existing DM pipeline, Business-aware dedupe/sequencing, safe outbound `business_connection_id` injection, docs, tests, and live Telegram Business proof.

## Reproduction Assessment

Yes, at source level. Current main allows the Business update names through grammY defaults, but has no Business handlers or outbound `business_connection_id` mapping, so the reported Business chat flow cannot pass through the existing Telegram pipeline.

## Solution Assessment

Yes on product boundary: the Telegram plugin is the right place to solve this, but adding `allowed_updates` alone is not enough. The maintainable fix needs Business connection state, inbound routing, dedupe/sequencing, outbound mapping, docs, tests, and live proof.
