## Summary

Keep open: current main still allows delivery of Telegram Business update names through grammY defaults, but the Telegram plugin does not handle `business_message` or `business_connection`, does not carry Business payloads through dedupe/sequencing, and has no outbound `business_connection_id` reply path. The issue is the canonical tracker, with a closed unmerged target PR and an open fork PR still tied to the requested work.

## What This Changes

The issue requests Telegram Business support so bots can process `business_message` and `business_connection` updates and reply in Business-connected personal chats.

## Best Possible Solution

Keep this as the canonical tracker and ship a refreshed Telegram plugin implementation with Business connection state, Business message routing, dedupe/sequencing support, safe outbound `business_connection_id` handling, tests, docs, and live proof.

## Reproduction Assessment

Yes. Source inspection on current main shows no `business_message` or `business_connection` handlers and no outbound `business_connection_id` mapping, so delivered Business updates cannot reach the inbound pipeline or reply path.

## Solution Assessment

Yes on the feature boundary: the Telegram plugin is the right place to solve this. The existing closed PR is not the best final patch as-is because prior review found dedupe, account-scoping, disconnect, and outbound mapping blockers that should be addressed in a refreshed implementation.
