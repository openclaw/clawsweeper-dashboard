Keep open: current main still lacks Telegram Business runtime handling and outbound business_connection_id support, and the issue remains the canonical tracker while the only target-repo PR is closed unmerged and the fork PR with closing syntax is still open.

## What This Changes

Request Telegram plugin support for Telegram Business updates, including business_connection/business_message handling and replies using business_connection_id.

## Best Possible Solution

Ship a maintained Telegram plugin implementation that stores Business connection state per account, routes business_message through the existing inbound and authorization pipeline, updates dedupe/sequential keys, applies business_connection_id only where supported, and documents setup plus limits.

Yes. Source inspection on current main shows no business_message/business_connection handlers and no business_connection_id outbound mapping, so Business updates cannot currently reach the inbound or reply path.

## Solution Assessment

Yes for the requested direction: this belongs in the Telegram plugin, but persistence, account scoping, access-control behavior, and live proof should be resolved through maintainer review of the open fork PR or a replacement implementation.
