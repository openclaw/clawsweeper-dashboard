## Summary

Keep open because current main still does not implement Telegram Business runtime support: Business update names exist only through grammY/default update coverage, but OpenClaw has no Business handlers, no Business dedupe/sequential modeling, no outbound business_connection_id mapping, and no focused docs/tests. The target-repo implementation PR is closed unmerged, while an open fork PR still uses closing syntax and should be reviewed, imported, or replaced before this canonical tracker closes.

## What This Changes

Request Telegram Business support in the Telegram plugin so Business-connected personal-chat messages are received and replies can be sent with the required business connection context.

## Best Possible Solution

Ship a Telegram plugin implementation that handles Business connection state, routes `business_message` through the existing inbound DM pipeline, updates dedupe/sequential keys, applies `business_connection_id` only where supported, and documents setup plus access-control behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, from source inspection: current main has no Business handlers or outbound `business_connection_id` mapping, so Business updates cannot reach the existing inbound pipeline or reply path.

## Solution Assessment

Is this the best way to solve the issue? Yes for the requested direction: the work belongs inside the Telegram plugin boundary, but the open fork PR or a replacement still needs maintainer review for state persistence, access control, outbound coverage, tests, and docs.
