## Summary

Keep open because current main and the latest release still do not implement Telegram Business runtime support: Business update types are recognized by Telegram/grammY, but OpenClaw has no Business handlers, no Business update-key/sequential-key modeling, no outbound `business_connection_id` mapping, and no focused docs/tests. The duplicate issues are already closed to this canonical tracker, the target-repo implementation PR is closed unmerged, and the remaining open fork PR should be imported, replaced, or explicitly closed by a Telegram owner before this issue closes.

## What This Changes

Add Telegram Business support so the Telegram plugin can receive `business_connection`/`business_message` updates, route Business messages through the inbound pipeline, and reply with the required Business connection context.

## Best Possible Solution

Ship a Telegram plugin implementation that handles Business connect/disconnect state per account, routes `business_message` through the existing DM authorization and inbound pipeline, updates dedupe/sequential keys for Business payloads, applies `business_connection_id` only where supported, and documents setup plus access-control behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows no Business handlers or outbound `business_connection_id` mapping, so Business updates cannot reach the existing inbound pipeline or reply path.

## Solution Assessment

Yes for the requested direction: the Telegram plugin is the right owner boundary. The best fix is not a one-line allowed-updates change; it needs handler wiring, account-scoped connection state, outbound mapping, dedupe/sequencing, docs, tests, and live proof.
