## Summary

Keep this canonical issue open: current main and v2026.6.6 allow Telegram Business update types but still do not register handlers that route those updates into OpenClaw, and the only linked implementation remains unmerged in a fork.

## What This Changes

Add Telegram Business-connected chat handling for inbound updates, connection lifecycle, and outbound replies.

## Best Possible Solution

Implement explicit Business update dispatch in the Telegram plugin, route messages through the existing inbound pipeline, extend dedupe and sequencing, and store account-scoped connection lifecycle in canonical SQLite state while reusing existing reply metadata propagation.

## Reproduction Assessment

Yes at source level: current main and v2026.6.6 allow Business updates, while the registered runtime handlers contain no Business event path. This read-only review did not exercise a live Telegram Business account.

## Solution Assessment

No, adding allowed-update names alone is not the best solution because those names are already present through grammY defaults. The narrow maintainable direction is explicit Business dispatch plus durable connection lifecycle, access-policy decisions, and focused outbound handling.
