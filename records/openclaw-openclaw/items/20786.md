## Summary

Keep open. Current main and the latest release still do not implement Telegram Business message handling: Business update names are present through grammY defaults, but OpenClaw has no business_message or business_connection runtime handlers, no Business payload update/sequential keys, no outbound connection-id state model, and no Telegram Business docs/tests; the prior target PRs are closed unmerged rather than viable fixes.

## What This Changes

Request support for Telegram Business Bot updates so business_message and business_connection events are routed through the Telegram channel and replies can include the required business_connection_id.

## Best Possible Solution

Ship a maintainer-approved Telegram plugin implementation that handles Business connect/disconnect state per account, routes business_message through the existing DM authorization and inbound pipeline, updates dedupe/sequential keys for Business payloads, applies business_connection_id only with the right account-scoped mapping, and documents/testing the Telegram Business flow.

## Reproduction Assessment

Yes from source inspection, though this review did not run a live Telegram Business account. Current main lacks Business handlers and Business-aware update keys while the locked Telegram/grammY contracts expose those update shapes.

## Solution Assessment

Unclear as an implementation decision. The narrowest maintainable path appears to be a Telegram-plugin change, but maintainers still need to settle the access-control, persistence, disconnect, and outbound business_connection_id model before another fix PR should be queued.
