## Summary

Keep open. Current main still only partially covers Telegram Business: allowed update names are present, but Business updates are not handled, deduped/sequenced, mapped for outbound business_connection_id replies, documented, or tested. The original target PR was closed unmerged and the remaining fork candidate should be reviewed or replaced rather than treating this as implemented.

## What This Changes

Request Telegram Business bot support in the Telegram plugin, including business_connection/business_message handling, inbound routing, and outbound replies with business_connection_id.

## Best Possible Solution

Ship a refreshed Telegram plugin implementation that registers account-scoped business_connection and business_message handlers, routes business_message through the existing DM authorization and inbound pipeline, persists/applies business_connection_id for supported outbound methods, updates dedupe/sequential keys for Business payloads, and documents Telegram Business setup plus access-control behavior.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: a business_message update can be allowed by polling/webhook settings, but current main has no Business handler to pass it into handleInboundMessageLike, so it is not processed. A live Telegram Business smoke would still be useful before landing a fix.

## Solution Assessment

Yes for the requested direction: this belongs in the Telegram plugin rather than core. The known PR branch is not the best implementation as-is because prior review found dedupe, account-isolation, disconnect, and outbound mapping issues that a refreshed fix should avoid.
