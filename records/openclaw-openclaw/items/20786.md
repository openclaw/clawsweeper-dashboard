## Summary

Keep open: current main and the latest release still allow Telegram Business update names but do not route `business_message` or `business_connection` through the Telegram runtime, and the only open closing PR is a conflicting fork mirror of a closed same-repo implementation with known blockers.

## What This Changes

Enable the Telegram plugin to receive, process, and reply to Telegram Business `business_message` and `business_connection` updates.

## Best Possible Solution

Keep this as the canonical tracker until a refreshed Telegram plugin implementation handles Business connect/disconnect state per account, routes `business_message` through the existing inbound pipeline, updates dedupe/sequencing, applies outbound `business_connection_id` safely, and adds focused tests, docs, and live Telegram proof.

## Reproduction Assessment

Yes. At source level, current main has no `business_message` or `business_connection` handler while allowed updates include the Business update types, so such updates can be delivered but not routed into the inbound pipeline.

## Solution Assessment

No. The allowed-updates-only proposal is too narrow; the maintainable fix needs runtime handlers, account-scoped Business connection state, outbound parameter handling, dedupe/sequencing updates, focused tests, docs, and live Telegram proof.
