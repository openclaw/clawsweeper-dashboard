## Summary

Keep open: current main and v2026.6.10 still expose Telegram Business update names through grammY defaults but do not route `business_message` or `business_connection` through the Telegram runtime, and the only open closing PR is a conflicting fork mirror of a closed implementation with known blockers.

## What This Changes

Request Telegram Business bot support so OpenClaw can receive Business-connected personal-chat updates and reply with the required Business connection context.

## Best Possible Solution

Keep this as the canonical tracker until a refreshed Telegram plugin implementation handles Business connect/disconnect state per account, routes `business_message` through the inbound pipeline, updates dedupe/sequencing, applies outbound `business_connection_id` safely, and adds focused tests, docs, and live proof.

## Reproduction Assessment

Yes at source level. Current main allows the Business update names through grammY defaults, but it has no `business_message` or `business_connection` handler and no Business payload dedupe/sequencing path, so delivered updates are not routed into OpenClaw.

## Solution Assessment

No, the minimal allowed-updates-only proposal is too narrow. The maintainable solution needs runtime handlers, account-scoped Business connection state, safe outbound `business_connection_id` handling, dedupe/sequencing updates, tests, docs, and live Telegram proof.
