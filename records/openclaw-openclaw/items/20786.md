## Summary

Keep open because current main still lacks Telegram Business runtime handling, outbound `business_connection_id` support, Business update dedupe/sequencing, and docs; an open fork PR still uses closing syntax for this issue.

## What This Changes

Add Telegram Business support so the Telegram plugin can receive `business_connection` and `business_message` updates, route Business messages through the inbound pipeline, and reply with the proper business connection context.

## Best Possible Solution

Ship a Telegram plugin implementation that handles Business connect/disconnect state per account, routes `business_message` through the existing DM pipeline, updates dedupe/sequential keys, applies `business_connection_id` only where appropriate, and documents setup plus access-control behavior.

## Reproduction Assessment

Yes, from source inspection. Current main can set allowed update names through grammY defaults, but it has no Business handler routing `business_message` into `handleInboundMessageLike()` and no outbound `business_connection_id` mapping.

## Solution Assessment

Yes for the requested direction: the fix belongs in the Telegram plugin, not generic core. The open fork PR or a replacement still needs maintainer review for account-scoped state, disconnect behavior, outbound method support, tests, and docs.
