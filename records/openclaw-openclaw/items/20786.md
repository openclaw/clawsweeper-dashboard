## Summary

Keep open: current main and v2026.5.20 still admit Telegram Business update names through grammY defaults, but the Telegram plugin does not process Business updates or attach `business_connection_id` for replies. This remains the canonical tracker because the target-repo PR closed unmerged and the fork PR using closing syntax is still open.

## What This Changes

Request Telegram Business support so OpenClaw can receive `business_connection`/`business_message` updates and reply to Business-connected personal chats.

## Best Possible Solution

Ship a refreshed Telegram plugin implementation with account-scoped durable Business connection state, Business message routing through the existing DM pipeline, Business-aware dedupe/sequencing, safe outbound `business_connection_id` injection, docs, tests, and live Telegram Business proof.

## Reproduction Assessment

Yes, at source level. Current main admits Telegram Business update names through grammY defaults but registers no Business handlers or outbound `business_connection_id` mapping, so a Telegram Business chat cannot enter the OpenClaw Telegram pipeline or receive supported replies.

## Solution Assessment

Yes on the product boundary, but not as an allowed-updates-only patch. The Telegram plugin is the right owner, and the maintainable fix must include connection state, Business-aware dedupe/sequencing, outbound reply mapping, access-control documentation, tests, and live proof.
