## Label Justifications

- `P2`: This was a normal-priority plugin SDK capability gap with bounded exec/plugin impact, not an urgent production outage.
- `impact:security`: The item concerns safe filtering of plugin-provided environment values before they reach subprocess execution.
- `impact:session-state`: The motivating use case is session-scoped identity propagation into shell subprocesses without global `process.env` mutation.

## Summary

Current main and the latest release already provide the requested plugin-facing spawn-time environment seam through the typed `resolve_exec_env` hook, with filtering and exec-path wiring, so the issue can close as implemented.

## What This Changes

Expose a plugin SDK path for plugins to contribute session-scoped environment variables to shell subprocess execution without mutating global `process.env`.

## Best Possible Solution

Keep `resolve_exec_env` as the shipped public plugin SDK seam for exec environment contribution, and leave the lower-level internal `BashSpawnHook` unexposed.

## Reproduction Assessment

Not applicable as a bug reproduction. The original missing seam was source-verifiable, and current source/docs now show plugins can use `resolve_exec_env` for filtered exec environment contribution.

## Solution Assessment

Yes. The shipped typed exec-env hook is the better public boundary than exposing the internal `BashSpawnHook`, because current agent shell execution uses `exec` and centralizes filtering, approval, and node/gateway forwarding there.
