- `impact:security`: The request is about safely filtering plugin-provided environment values before they reach subprocess execution.
- `impact:session-state`: The motivating use case is session/run-scoped identity propagation into bash subprocesses without global process.env mutation.

## Summary

Keep open: current main has an internal BashSpawnHook, but no public plugin SDK registration/export path or composed hook wiring for the agent bash tool. The request fits OpenClaw's stated direction for extending plugin APIs when a capability cannot currently be built as a plugin, but it needs maintainer API and security-boundary review before implementation.

## What This Changes

Request to expose a narrow plugin SDK seam that lets plugins contribute filtered, deterministic environment entries to bash-tool subprocess spawns.

## Best Possible Solution

Design a narrow plugin-neutral SDK/runtime hook, compose it once where agent bash tools are constructed, filter contributions through host env security helpers, and update SDK exports, docs, and focused tests together.

## Reproduction Assessment

Not applicable as a bug reproduction, but source inspection gives a high-confidence proof of the missing seam: BashSpawnHook exists internally, while the agent bash tool is built without a spawnHook and the SDK has no bash-spawn entrypoint.

## Solution Assessment

Unclear until maintainer API review: the proposed composed hook is plausible, but the permanent contract must define trust, filtering, ordering, and context fields before code is written.
