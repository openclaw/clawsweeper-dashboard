## Label Justifications

- `P2`: This was a normal-priority plugin SDK capability gap with bounded exec/plugin impact and no evidence of an urgent production outage.
- `impact:security`: The issue concerns safely filtering plugin-provided environment variables before they reach subprocess execution.
- `impact:session-state`: The motivating use case is session-scoped identity propagation into subprocesses without global `process.env` mutation.

## Summary

Close: current main and release v2026.6.8 now ship a typed `resolve_exec_env` plugin hook that lets plugins contribute filtered environment variables to `exec` subprocesses without exposing the internal `BashSpawnHook` API.

## What This Changes

Request to expose a plugin SDK path for adding filtered, session-scoped environment variables to shell/exec subprocess launches.

## Best Possible Solution

Keep `resolve_exec_env` as the canonical plugin SDK seam for subprocess env contribution, and handle richer account/channel context in separate SDK-context work if maintainers accept it.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this was a plugin SDK feature request. Source inspection verifies current main now has a typed hook, filtering, docs, and tests for the requested env contribution path.

## Solution Assessment

Yes. The shipped `resolve_exec_env` hook is a narrower maintained solution than exposing internal `BashSpawnHook`, because it targets the actual `exec` tool path and keeps filtering/audit behavior in core.
