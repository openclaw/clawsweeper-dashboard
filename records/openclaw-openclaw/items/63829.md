## Summary

Keep open: current main and latest v2026.6.10 still resolve memory-wiki from one global plugin config and one vault path, while the issue discussion shows real multi-agent isolation and security impact that no merged or viable PR now owns.

## What This Changes

The issue requests per-agent memory-wiki vault configuration so agents on one gateway can keep isolated wiki memory.

## Best Possible Solution

Define one maintainer-approved agent-scoped memory-wiki isolation contract, with schema/config precedence, vault routing, security/upgrade behavior, docs, and focused two-agent regression coverage.

## Reproduction Assessment

Yes, at source level: current main resolves memory-wiki config globally once, captures one vault path, and rejects agents.list[].plugins via the strict agent schema.

## Solution Assessment

Unclear: the requested direction is valid, but maintainers still need to choose between generic per-agent plugin config, memory-wiki-specific vault routing, templated paths, or a coordinated contract across runtime and admin surfaces.
