## Label Justifications

- `P1`: The issue blocks real multi-agent users from enabling memory-wiki safely on one gateway because knowledge and search state can cross intended agent boundaries.
- `impact:session-state`: The reported behavior mixes memory-wiki vault content, indexes, prompt context, and search results across agents that operators expect to keep isolated.

## Summary

Keep open: current main and v2026.6.8 still resolve memory-wiki from one global plugin config and one vault path, while the issue discussion shows a real multi-agent memory isolation and security gap that no merged or viable open PR now owns.

## What This Changes

Request per-agent memory-wiki vault configuration so agents on one gateway can keep isolated wiki content, indexes, and search results.

## Best Possible Solution

Define one maintainer-approved agent-scoped memory-wiki isolation contract, with schema/config precedence, vault routing, security/upgrade behavior, docs, and focused two-agent regression coverage.

## Reproduction Assessment

Yes, at source level. Current main resolves memory-wiki from global plugins.entries.memory-wiki.config once, captures one vault path, and the strict agent schema has no plugins override field; I did not run a live two-agent gateway repro in this read-only review.

## Solution Assessment

Unclear. The requested direction is valid, but maintainers still need to choose whether the best fix is generic per-agent plugin config, a memory-wiki-specific vault mode, templated paths, or a coordinated combination across tools, CLI, gateway, and background memory paths.
