## Summary

Keep open: current main and v2026.6.8 still capture one global memory-wiki vault config, while multiple users report real multi-agent knowledge-isolation pain. The remaining work needs maintainer product and security direction, not cleanup closure.

## What This Changes

Request per-agent memory-wiki vault isolation so agents on one gateway can use separate wiki roots, indexes, and search results instead of one shared vault.

## Best Possible Solution

Choose a maintainer-approved vault scoping contract, then implement it with explicit precedence, security handling, docs, and two-agent regression coverage across tools, prompt/corpus supplements, gateway methods, and CLI/admin surfaces.

## Reproduction Assessment

Yes, at source level: current main rejects `agents.list[].plugins`, memory-wiki has one strict `vault.path`, and plugin registration closes over one resolved config. I did not run a live two-agent gateway in this read-only review.

## Solution Assessment

Unclear. The need is well supported, but maintainers still need to choose whether the best fix is generic per-agent plugin config, memory-wiki-owned context-aware vault resolution, explicit vault maps, or a combination.
