## Label Justifications

- `P1`: The issue blocks real multi-agent memory-wiki use on a shared gateway because agents can share wiki state that operators expect to isolate.
- `impact:session-state`: The reported behavior can mix memory/wiki context, indices, and search results across agents with separate intended state.

## Summary

Keep open: current main and v2026.6.8 still route memory-wiki through one global vault config, while user reports and the PLUR1BUS plugin-side implementation context show this remains a real multi-agent isolation/security gap rather than conservative cleanup.

## What This Changes

Issue requests per-agent memory-wiki vault configuration so multi-agent setups can keep wiki knowledge, indices, and searches isolated instead of sharing one global vault.

## Best Possible Solution

Define a maintainer-approved memory-wiki/plugin config isolation contract, then implement vault routing with docs, doctor/upgrade behavior, and two-agent regression coverage for tools, prompts, CLI, gateway methods, and background jobs.

## Reproduction Assessment

Yes, at source level: current main resolves memory-wiki config once globally, the agent schema has no `plugins` override field, and v2026.6.8 has the same limitation; no live gateway run was attempted in this read-only pass.

## Solution Assessment

Unclear: the issue is valid, but maintainers need to choose the permanent config/plugin SDK/vault-routing contract before implementation.
