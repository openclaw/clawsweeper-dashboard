## Label Justifications

- `P1`: The issue blocks real multi-agent memory-wiki use on a shared gateway and can expose one agent's wiki context to another agent's workflow.
- `impact:session-state`: The reported behavior mixes memory/wiki state and search context across agents that operators expect to be isolated.
- `impact:security`: A shared vault can cross sensitive workspace or tenant boundaries unless per-agent wiki access is explicitly scoped.

## Summary

Keep open: current main and the latest release still resolve one global memory-wiki vault config at plugin registration, while the issue describes a real multi-agent memory isolation and sensitive-data boundary gap that needs maintainer product/security direction rather than cleanup closure.

## What This Changes

Requested per-agent memory-wiki vault configuration so agents on one gateway can keep isolated wiki roots, indexes, and search results.

## Best Possible Solution

Define one maintainer-approved agent-scoped memory-wiki/plugin configuration contract that preserves existing global behavior by default, isolates vault reads/writes/search by agent when configured, and covers schema, SecretRef, docs, and two-agent regression tests.

## Reproduction Assessment

Yes, at source level. Current main has no agent-level plugin config field, memory-wiki captures one top-level plugin config at registration, and `wiki_apply` writes through that captured vault path while search/get still reuse the same config object.

## Solution Assessment

Unclear. The requested direction is valid, but maintainers should choose whether the best fix is the broader agent-scoped plugin config tracked in https://github.com/openclaw/openclaw/issues/55401, a memory-wiki-specific vault templating surface, or a combined design that also covers prompt-supplement scoping.
