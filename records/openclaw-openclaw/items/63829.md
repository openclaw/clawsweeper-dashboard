## Summary

Keep open. Current main still captures one memory-wiki config at plugin registration, lacks an agent-level plugin config field or per-agent vault mode, and the related open PR #67584 is an implementation candidate rather than merged behavior.

## What This Changes

Add per-agent memory-wiki vault isolation so multi-agent gateways can keep wiki pages, indexes, and semantic search results separated by agent or workspace.

## Best Possible Solution

Keep this as the memory-wiki vault isolation tracker and land #67584 or an equivalent memory-wiki-owned fix that preserves the current global default, resolves vault roots from trusted invocation context or explicit mapping, updates schema/docs/changelog, and adds two-agent regression coverage for read/write paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure two agents on one gateway, observe that agent plugin config overrides are not accepted, and all memory-wiki wiki read/write paths use the single captured top-level vault config.

## Solution Assessment

No current-main solution exists yet. The narrowest maintainable direction appears to be memory-wiki-owned per-invocation vault resolution or explicit vault mapping; a generic agents.list[].plugins API is broader product/API work.
