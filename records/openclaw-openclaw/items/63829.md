## Label Justifications

- `P1`: The issue blocks safe use of memory-wiki in real multi-agent deployments by allowing wiki knowledge to commingle across agents on one gateway.
- `impact:session-state`: The reported behavior mixes durable wiki memory and search context between otherwise separate agent scopes.
- `impact:security`: Per-agent wiki vaults can contain sensitive private knowledge, and the current global vault path can expose it across agents unless operators avoid the plugin or split gateways.

## Summary

Keep open: current main and v2026.6.8 still resolve one global memory-wiki vault config per plugin registration, while the issue describes a real multi-agent memory isolation and security gap that needs maintainer product/security design.

## What This Changes

Request per-agent memory-wiki vault configuration so agents on one gateway can keep isolated wiki vaults and search results.

## Best Possible Solution

Define one maintainer-approved agent-scoped memory-wiki/plugin config contract, then cover tools, gateway, CLI/docs, SecretRef handling, and two-agent isolation tests against that contract.

## Reproduction Assessment

Yes, at source level: current main rejects agents.list[].plugins, reads plugin config only from global plugins.entries[id].config, and memory-wiki captures one vault path during plugin registration. I did not run a live two-agent gateway in this read-only pass.

## Solution Assessment

Unclear. Agent-level plugin overrides are plausible, but maintainers need to choose the permanent config/API shape and no-context behavior before an implementation should proceed.
