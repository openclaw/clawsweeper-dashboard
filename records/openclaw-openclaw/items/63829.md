## Label Justifications

- `P1`: Multiple users report real multi-agent deployments where shared memory-wiki vaults make a core agent-memory workflow unsafe or unusable today.
- `impact:session-state`: The issue is about wiki memory, search results, and prompt context crossing agent boundaries that operators expect to keep isolated.
- `impact:security`: Cross-agent vault sharing can expose sensitive knowledge from one agent or tenant to another and any fix affects plugin credential/state isolation boundaries.

## Summary

Keep open: current main and v2026.6.8 still resolve memory-wiki from one global plugin config and one vault path, while this issue has concrete multi-agent isolation and sensitive-memory impact that needs maintainer product/security direction.

## What This Changes

The issue requests per-agent memory-wiki vault configuration so multiple agents on one gateway can keep separate wiki vaults and search indexes.

## Best Possible Solution

Define a maintainer-approved agent-scoped plugin or memory-wiki vault contract, then implement it with two-agent isolation tests, docs, and explicit security/upgrade semantics.

## Reproduction Assessment

Yes, at source level. Current main has a strict agent schema with no `plugins` field, global-only plugin config resolution, and memory-wiki tools/CLI registered against one resolved vault config; I did not run a live two-agent gateway in this read-only review.

## Solution Assessment

Unclear. The requested directions are plausible, but maintainers should first choose the permanent plugin/config security contract rather than shipping an ad hoc memory-wiki-only override.
