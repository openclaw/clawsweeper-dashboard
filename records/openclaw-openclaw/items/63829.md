## Summary

Keep open. Current main still has a single memory-wiki vault config captured at plugin registration, no agent-level plugin config field, and no per-invocation wiki vault resolution; the related open PR #67584 is an implementation candidate but is not merged.

## What This Changes

The issue requests per-agent memory-wiki vault configuration or isolation so multiple agents on one gateway do not share the same wiki corpus.

## Best Possible Solution

Keep this as the memory-wiki vault isolation tracker and land #67584 or an equivalent memory-wiki-owned fix that resolves the vault root from trusted invocation context or explicit per-agent mapping, preserves the current global default, updates config schema/docs, and adds regression coverage for two agents on one gateway across wiki_search, wiki_get, wiki_apply, and gateway/CLI management surfaces.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: two agents on one gateway cannot supply separate memory-wiki plugin configs today, and all wiki read/write paths use the single captured vault path.

## Solution Assessment

No current-main solution exists yet. The narrowest maintainable direction appears to be memory-wiki-owned per-invocation vault resolution or explicit vault mapping using existing trusted tool context, with broader per-agent plugin overrides treated as a separate product/API decision if maintainers want that generalized.
