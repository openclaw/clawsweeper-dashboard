## Label Justifications

- `P1`: Multiple users report memory-wiki is unsafe or unusable in production multi-agent gateways because one shared vault can commingle isolated knowledge domains.
- `impact:security`: The issue is about cross-agent wiki visibility and potential sensitive knowledge bleed across intended isolation boundaries.
- `impact:session-state`: The affected state is agent-scoped memory/wiki context that can be shared or mis-associated across agents on one gateway.

## Summary

Keep open: current main and v2026.5.18 still resolve one memory-wiki vault from the top-level plugin config, while production commenters describe real multi-agent isolation pain. The likely implementation path is still the open memory-wiki PR plus the broader per-agent plugin-config decision, not cleanup closure.

## What This Changes

The issue requests per-agent memory-wiki vault configuration so agents on one gateway can keep isolated wiki vaults instead of sharing one global vault.

## Best Possible Solution

Keep this as the canonical memory-wiki vault isolation tracker and review, land, or replace https://github.com/openclaw/openclaw/pull/67584 with equivalent scoped vault resolution plus docs and two-agent regression coverage, while coordinating the broader API with https://github.com/openclaw/openclaw/issues/55401.

## Reproduction Assessment

Yes, at source level. Configure two agents on one gateway and current main has no valid per-agent memory-wiki plugin config while wiki read/write paths use one captured top-level vault config.

## Solution Assessment

Unclear. The open memory-wiki templating PR is a plausible narrow fix, but maintainers still need to decide how it relates to the broader per-agent plugin override contract and prompt/corpus supplement scoping.
