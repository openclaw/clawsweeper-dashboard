## Summary

Keep open: current main still has a real Feishu interactive-card gap, but this PR is not mergeable as-is because the full-card fetch parameter is not verified, `RawBody` would regress for normal messages, and the external PR still lacks real Feishu proof.

## What This Changes

The PR adds a shared Feishu interactive-card parser, wires it into inbound event and message-read paths, propagates raw card content, and adds focused Feishu tests.

## Best Possible Solution

Consolidate the Feishu-local parser/readback work into one bounded implementation that uses the verified full-card request shape, preserves `RawBody` and `CommandBody` text semantics, exposes raw card data through a separate safe field if needed, adds focused regression tests, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level. Current main routes direct inbound Feishu `interactive` events through a parser with no interactive branch, and message readback still omits full-card request parameters; I did not run a live Feishu tenant reproduction.

## Solution Assessment

No, not as currently patched. The Feishu plugin is the right boundary, but this PR should use a verified `card_msg_content_type` request path and keep `RawBody` as normalized user text rather than raw Feishu JSON.
