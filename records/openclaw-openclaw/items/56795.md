## Summary

Keep open. The Feishu inbound interactive-card gap is still source-reproducible on current main, but this PR is not mergeable as-is because it uses an unverified card-content query shape, changes `RawBody` semantics, lacks real Feishu proof, and overlaps active Feishu parser work.

## What This Changes

The PR adds a shared Feishu interactive-card parser, fetches richer inbound interactive message content, propagates optional raw card content fields, and adds focused parser/message tests.

## Best Possible Solution

Consolidate the Feishu-local parser/readback work into one implementation that uses verified full-card retrieval, preserves normalized body semantics, exposes raw card data through an explicit safe field, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes, source-level. Current main routes direct inbound Feishu events through a parser with no interactive-card branch and message readback does not request full card content, though I did not run a live Feishu tenant reproduction.

## Solution Assessment

No, not as currently patched. The Feishu plugin boundary is right, but the patch should use the verified card-content request shape and avoid changing `RawBody` from normalized user text to raw Feishu JSON.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Request the validated Feishu card-content format:** `extensions/feishu/src/send.ts:274`
  - body: The new fetch sends `params: { raw_card_content: true }`, but the SDK type for `im.message.get` only exposes `user_id_type`, and the Lark plugin uses a raw request with `card_msg_content_type=raw_card_content`. With this shape the mock can pass while Feishu still returns fallback content, so use a compatible adapter and assert the real query parameter.
  - confidence: 0.9
- **[P2] Preserve RawBody as normalized user text:** `extensions/feishu/src/bot.ts:942`
  - body: `parseFeishuMessageEvent` now stores raw Feishu JSON in `ctx.rawContent` for every message type, and this line makes `RawBody` prefer that envelope. Regular text messages would expose values like `{"text":"hello"}` instead of the legacy raw body text, breaking hooks or templates that match plain user input.
  - confidence: 0.88
