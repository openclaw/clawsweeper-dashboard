## Summary

Keep this PR open, but not merge-ready: the Feishu card parsing/readback gap is still real on current main, while this branch still needs real Feishu behavior proof and fixes for the card-content API parameter plus RawBody compatibility.

## What This Changes

The PR adds Feishu inbound interactive-card parsing, richer message fetching, raw card JSON propagation, and focused Feishu parser/bot tests.

## Best Possible Solution

Keep the fix inside the Feishu plugin read path, request the validated card-content format through a typed or raw request adapter, preserve RawBody as normalized user text, and coordinate with the canonical Feishu readback tracker before landing.

## Reproduction Assessment

Yes, source-reproducible but not live-reproduced here. Current main still reads inbound/direct card content without a full-card query or direct event hydration, so placeholder card payloads can remain unreadable.

## Solution Assessment

No, not as currently patched. The PR targets the right Feishu-owned boundary, but it should use the validated card-content query shape and keep RawBody compatible with the documented CommandBody alias.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Request the documented Feishu card-content format:** `extensions/feishu/src/send.ts:274`
  - body: The new fetch sends `params: { raw_card_content: true }`, but the current Feishu tracker and SDK contract inspection point to `card_msg_content_type` for card return format, and the SDK types do not include this boolean key. With this request shape, the follow-up fetch can still return placeholder card content while the mock passes; use a documented query via a compatible adapter and assert it in tests.
  - confidence: 0.9
- **[P2] Preserve RawBody as normalized user text:** `extensions/feishu/src/bot.ts:942`
  - body: `parseFeishuMessageEvent` now records `ctx.rawContent` for every message type, and this line makes `RawBody` prefer that raw Feishu JSON envelope. Regular text messages would expose values like `{"text":"hello"}` instead of the documented legacy `CommandBody` alias, breaking hooks or templates that match plain user text.
  - confidence: 0.88
