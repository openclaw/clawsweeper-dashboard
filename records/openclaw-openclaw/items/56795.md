## Summary

Keep open: the PR addresses a real Feishu parsing gap that current main still has, but the branch is not merge-ready because it uses the wrong Feishu card-content query shape, regresses the documented RawBody compatibility contract, is currently conflicting, and has only mocked/test proof rather than after-fix real behavior proof.

## What This Changes

The branch adds Feishu inbound interactive-card parsing, fetches richer message content for inbound interactive events, carries optional raw card JSON through Feishu context, and adds focused Feishu tests.

## Best Possible Solution

Keep the fix inside the Feishu plugin read path, request the documented card-content return format, parse returned card JSON before existing fallbacks, and preserve RawBody as the normalized command body for non-card messages.

## Reproduction Assessment

Yes. Source inspection shows current main processes direct inbound Feishu interactive events from the event payload without fetching richer card content, so placeholder-only card payloads remain unreadable in that path.

## Solution Assessment

No. The PR targets the right Feishu-owned boundary, but the merge shape should be corrected to use Feishu's documented card-content query and keep RawBody compatible with the documented CommandBody alias.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Request Feishu's documented card-content format:** `extensions/feishu/src/send.ts:274`
  - body: The new fetch sends `params: { raw_card_content: true }`, but Feishu's message get/list docs expose `card_msg_content_type` for card return format and the SDK type surface does not include this boolean key. With this request shape, the follow-up fetch can still return placeholder card content while the mock passes; use the documented query shape and assert it in the test.
  - confidence: 0.91
- **[P2] Keep RawBody on normalized message text:** `extensions/feishu/src/bot.ts:942`
  - body: `parseFeishuMessageEvent` now populates `ctx.rawContent` for every message type, and this line makes `RawBody` prefer that raw Feishu envelope. Regular text messages would expose values like `{"text":"hello"}` instead of the documented legacy `CommandBody` alias, breaking hooks or templates that match plain user text.
  - confidence: 0.88
