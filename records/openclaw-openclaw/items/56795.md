## Summary

Keep open: the Feishu gap is still source-reproducible on current main and this PR targets the right plugin surface, but the branch is conflicting, lacks real Feishu proof, and has blocking correctness issues in full-card retrieval and RawBody semantics.

## What This Changes

The PR adds a shared Feishu interactive-card parser, wires it into inbound event and message-read paths, propagates optional raw card content, and adds focused tests.

## Best Possible Solution

Consolidate the Feishu-local parser/readback work into one implementation that uses a verified full-card retrieval adapter, preserves RawBody and CommandBody text semantics, exposes raw card data through a separate safe field, adds focused regression tests, and includes redacted live Feishu proof.

## Reproduction Assessment

Yes, at source level. Current main routes direct inbound interactive events through a parser with no interactive branch and message readback omits a full-card-content request parameter, though I did not run a live Feishu tenant reproduction.

## Solution Assessment

No, not as currently patched. The Feishu plugin is the right boundary, but the patch should use a verified card-content request shape and keep RawBody as normalized user text rather than raw Feishu JSON.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Request full cards with the verified query parameter:** `extensions/feishu/src/send.ts:274`
  - body: The new fetch sends `params: { raw_card_content: true }`, but the SDK type for `im.message.get` only exposes `user_id_type`, and the Lark implementation uses a raw request with `card_msg_content_type`. With this shape, mocks can pass while Feishu still returns fallback card content, so use a verified adapter/query parameter and assert the actual request shape.
  - confidence: 0.9
- **[P2] Keep RawBody as normalized message text:** `extensions/feishu/src/bot.ts:942`
  - body: `parseFeishuMessageEvent` now stores the raw Feishu JSON envelope for every message type, and this line makes `RawBody` prefer it. Regular text messages would expose values like `{"text":"hello"}` instead of the legacy plain user text, breaking hooks or templates that consume `RawBody` directly.
  - confidence: 0.88
