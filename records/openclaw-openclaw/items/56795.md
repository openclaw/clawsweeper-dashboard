## Summary

Keep open: the PR addresses a real Feishu card-reading gap, but current head still has blocking correctness issues and only mocked verification. Current main only partially handles interactive cards, so this is not obsolete or implemented-on-main.

## What This Changes

The PR adds Feishu inbound interactive-card parsing, richer message fetching, raw card JSON propagation, shared parser tests, and optional rawContent fields on Feishu message context types.

## Best Possible Solution

Keep the fix Feishu-local: request full card JSON through a verified SDK-safe or raw request adapter, parse it into normalized user text, expose raw card data without changing RawBody alias semantics, and consolidate with the open Feishu card parser/readback threads.

## Reproduction Assessment

Yes, source-reproducible. Current main does not parse direct inbound interactive events into card text and does not request full card content in Feishu message reads, though I did not run a live Feishu tenant reproduction here.

## Solution Assessment

No, not as currently patched. The Feishu plugin boundary is right, but the patch should use the verified card-content query shape and keep RawBody as normalized user text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Request the validated Feishu card-content format:** `extensions/feishu/src/send.ts:274`
  - body: The new fetch sends `params: { raw_card_content: true }`, but the SDK type does not expose that field and the official Lark plugin uses `card_msg_content_type=raw_card_content` via a raw request. With this request shape, the follow-up fetch can still return fallback content while the mock passes; use a compatible adapter and assert the actual query parameter.
  - confidence: 0.9
- **[P2] Preserve RawBody as normalized user text:** `extensions/feishu/src/bot.ts:942`
  - body: `parseFeishuMessageEvent` now records raw Feishu JSON for every message type, and this line makes `RawBody` prefer that envelope. Regular text messages would expose values like `{"text":"hello"}` instead of the documented legacy `CommandBody` alias, breaking hooks or templates that match plain user text.
  - confidence: 0.88
