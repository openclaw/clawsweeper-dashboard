Full review comments:

- **[P2] Stop the old watchdog before replacing it:** `src/cron/service/ops.ts:198`
  - body: `CronService.start()` can delegate to `ops.start()` more than once, and today `armTimer()` stays idempotent by clearing the previous timeout. Clear the existing watchdog before starting a new one or make watchdog startup idempotent.
  - confidence: 0.86

  - confidence: 0.95
- **[P3] Exercise the actual catch re-arm path:** `src/cron/service/timer.watchdog.test.ts:38-50`
  - body: This test clears the timer and calls `armTimer()` directly, so it would still pass if the new catch-handler `armTimer(state)` call were removed. Drive a real escaping `onTimer` rejection or move this assertion to the behavior it actually covers, otherwise the regression does not prove the recovery change.
