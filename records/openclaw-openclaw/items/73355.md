Full review comments:

- **[P2] Avoid double-arming rejected timer ticks:** `src/cron/service/timer.ts:737-742`
  - body: When `onTimer` rejects from inside its `try` block, the existing `finally` already calls `armTimer(state)` before the promise rejection reaches this catch handler. Unconditionally calling `armTimer(state)` here can leave an extra untracked timer and cause spurious follow-up ticks.
  - confidence: 0.86
- **[P2] Keep the watchdog out of normal fake-timer flows:** `src/cron/service/ops.ts:198-201`
  - body: Starting the interval from every `cron.start()` means tests and code paths that drain pending timers can run the five-minute watchdog alongside the normal scheduler. CI shows an existing minute-cron test now advances `nextRunAtMs` by five minutes instead of one, so the watchdog needs isolation or safer scheduling semantics.
  - confidence: 0.88
- **[P2] Add braces to the watchdog guards:** `src/cron/service/timer.ts:779-786`
  - body: The new watchdog uses one-line `if (...) return` guards, and `check-lint` fails with `Expected { after 'if' condition`. Add braces around these guards so the PR passes the repository lint rules.
  - confidence: 0.96
- **[P2] Exercise the rejecting timer path in the test:** `src/cron/service/timer.watchdog.test.ts:40-41`
  - body: The new catch re-arm test calls `armTimer` directly with an empty store, so CI fails because no timer is armed and the `.catch` branch is never exercised. Drive a real rejection path or inject a failing dependency so the new recovery behavior is covered.
  - confidence: 0.96
- **[P2] Remove or use the onTimer import:** `src/cron/service/timer.watchdog.test.ts:3`
  - body: `onTimer` is imported in the new test file but never referenced, which is one of the current lint failures. Remove it or use it as part of the direct catch-path regression test.
  - confidence: 0.94
