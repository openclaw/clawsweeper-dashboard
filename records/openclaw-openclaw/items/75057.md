## Summary

Keep open. Current main still keys Feishu receive-time, debounce, suppressed-record, and persistent finalize dedupe on the raw `message_id`, while open PR #75078 is the active implementation candidate and should be reviewed before this issue closes.

## What This Changes

Feishu inbound audio dedupe should treat different audio attachments with the same `message_id` as distinct messages instead of dropping later payloads.

## Best Possible Solution

Implement a Feishu-local media-aware dedupe key at every receive, debounce, suppressed-record, and finalize call site, with regression coverage for equal `message_id` plus different audio `file_key` values.

## Reproduction Assessment

Yes. Two Feishu receive payloads for the same account with equal `message.message_id`, `message_type: "audio"`, and different JSON `file_key` values collide at `tryBeginFeishuMessageProcessing` before media parsing, so the second payload is dropped.

## Solution Assessment

Yes. A Feishu-owned media-aware key is the narrowest maintainable solution; changing shared core dedupe would be broader and could alter unrelated channels.
