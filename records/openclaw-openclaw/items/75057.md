## Summary

Keep open. Current main still keys Feishu receive-time claims, debounce/suppression, and persistent finalize dedupe on bare `message_id`, and open PR #75078 explicitly references this issue as the implementation candidate.

## What This Changes

The report asks Feishu inbound dedupe to treat audio messages with the same `message_id` but different attachment `file_key` values as distinct messages.

## Best Possible Solution

Land PR #75078 or an equivalent Feishu-owned patch that computes one stable media-aware dedupe key before receive claims, debounce freshness checks, suppressed-record writes, and persistent finalize decisions while preserving bare `message_id` behavior for text and true retries.

## Reproduction Assessment

Yes. Two Feishu receive payloads for the same account with equal `message.message_id`, `message_type: "audio"`, and different JSON `file_key` values collide at `tryBeginFeishuMessageProcessing` before audio parsing.

## Solution Assessment

Yes. A Feishu-local media-aware key at the existing dedupe call sites is the narrowest maintainable solution; changing shared core dedupe would risk unrelated channels.
