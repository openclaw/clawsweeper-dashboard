## Summary

Keep open. Current main still keys Feishu receive-time and persistent dedupe on the bare `message_id`, while open PR #75078 is the linked implementation candidate and should be reviewed before this issue closes.

## What This Changes

The requested fix is for Feishu inbound dedupe to process distinct audio attachments independently when Feishu reuses the same `message_id`.

## Best Possible Solution

Land PR #75078 or an equivalent Feishu-local patch that computes one stable media-aware dedupe key before receive claims and final persistent dedupe, while preserving bare `message_id` behavior for text and true identical retries.

## Reproduction Assessment

Yes. Two Feishu audio event payloads for the same account with equal `event.message.message_id` and different JSON `file_key` values collide in the receive-time claim before media parsing, so the second event can be dropped on current main.

## Solution Assessment

Yes. The linked PR’s Feishu-local media-aware dedupe-key direction matches the actual collision points; changing shared core dedupe would be broader and risk unrelated channels.
