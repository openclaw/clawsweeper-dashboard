## Summary

Keep open. Current main still keys Feishu inbound receive claims, debounce filtering, suppressed-record writes, and persistent finalize on account plus bare `message_id`, while audio `file_key` is only parsed later. The issue is also paired with open PR #75078 via closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests Feishu audio dedupe to distinguish different audio attachments when Feishu reuses the same `message_id`.

## Best Possible Solution

Land #75078 or an equivalent Feishu-owned patch that computes one stable media-aware dedupe key before any receive claim, debounce filtering, suppressed-record write, or persistent finalize decision, while preserving bare `message_id` behavior for text and true identical retries.

## Reproduction Assessment

Yes for the OpenClaw side: two Feishu receive payloads for the same account with the same `message.message_id`, `message_type: "audio"`, and different JSON `file_key` values collide at the early claim before media parsing. The live Feishu ID-reuse observation still depends on reporter or live Feishu evidence.

## Solution Assessment

Yes. A Feishu-local media-aware dedupe key applied consistently across the existing dedupe call sites is the narrowest maintainable fix; changing shared core dedupe would be broader than this channel-specific failure requires.
