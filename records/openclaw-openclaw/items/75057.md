## Summary

Keep open. Current main still appears to have the reported Feishu audio dedupe collision: the Feishu receive path, debounce path, suppressed-message records, and persistent finalize still key off the raw `message_id` before the audio `file_key` is used for media identity. Open PR #75078 is attached with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that Feishu WebSocket audio messages can reuse the same `message_id` for different voice attachments, causing OpenClaw to dedupe and drop distinct audio before it reaches the agent.

## Best Possible Solution

Land #75078 or an equivalent Feishu-owned patch that computes one stable media-aware dedupe key before every receive claim, debounce filter, suppressed-record write, and persistent finalize decision, while preserving bare `message_id` behavior for text messages and true identical retries.

## Reproduction Assessment

Yes for the OpenClaw side: two Feishu receive events for the same account with identical `message.message_id`, `message_type: "audio"`, and different JSON `file_key` values collide in the early claim and downstream dedupe paths. Live Feishu proof of reused ids still depends on reporter evidence or channel credentials.

## Solution Assessment

Yes. A Feishu-local media-aware dedupe key shared across the existing Feishu dedupe call sites is the narrowest maintainable solution; changing shared core dedupe would be broader than this channel-specific failure requires.
