## Summary

Keep open. Current main still deduplicates Feishu inbound audio on the bare `message_id` before the audio `file_key` is parsed, so the reported collision remains valid. The issue is also paired with open PR #75078 through closing syntax, so the right maintainer action is to review, land, or reject that PR before closing this issue.

## What This Changes

The issue reports that Feishu audio messages with the same `message_id` but different audio `file_key` values are incorrectly treated as duplicates and dropped.

## Best Possible Solution

Land #75078 or an equivalent Feishu-owned patch that computes a stable media-aware dedupe key before every Feishu receive claim, debounce freshness check, suppressed-record write, and persistent finalize decision, while preserving bare `message_id` behavior for text and true identical retries.

## Reproduction Assessment

Yes. Two Feishu receive payloads for the same account with equal `message.message_id`, `message_type: "audio"`, and different JSON `file_key` values collide at `tryBeginFeishuMessageProcessing` before media parsing, so the second payload is dropped.

## Solution Assessment

Yes. A Feishu-local media-aware key used consistently across the existing Feishu dedupe call sites is the narrowest maintainable fix; changing shared core dedupe would be broader and risk unrelated channels.
