## Summary

Keep open. Current main still keys Feishu receive-time, debounce, and persistent dedupe on the bare `message_id`, so the reported same-id/different-audio collision remains valid; the issue is also paired with open PR #75078, which should be reviewed, landed, or rejected before the issue closes.

## What This Changes

Report that Feishu WebSocket audio events with the same `message_id` but different audio `file_key` values are deduped as duplicates instead of both reaching the agent.

## Best Possible Solution

Land #75078 or an equivalent Feishu-owned patch that computes one stable media-aware dedupe key before every Feishu receive claim, debounce freshness check, suppressed-record write, and persistent finalize decision, while preserving bare `message_id` behavior for text messages and true identical retries.

## Reproduction Assessment

Yes. Two Feishu receive payloads for the same account with equal `message.message_id`, `message_type: "audio"`, and different JSON `file_key` values will collide at `tryBeginFeishuMessageProcessing` before audio content parsing can distinguish them.

## Solution Assessment

Yes. A Feishu-local media-aware dedupe helper used consistently across the existing Feishu dedupe call sites is the narrowest maintainable fix; changing shared core dedupe would be broader and could affect unrelated channels.
