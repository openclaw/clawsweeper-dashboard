## Summary

Keep open. Current main still keys Feishu receive, debounce, suppressed-record, and persistent dedupe paths by account plus bare `message_id`, so the reported same-ID/different-audio collision remains possible; open PR #75078 explicitly references this issue as the implementation candidate, so maintainers should review, land, replace, or close that PR before closing the issue.

## What This Changes

Feishu inbound dedupe should process distinct audio attachments independently when Feishu reuses the same `message_id`, using media identity such as `file_key` without breaking true duplicate retries.

## Best Possible Solution

Use a Feishu-owned media-aware dedupe key consistently across early receive claims, debounce coalescing, suppressed-record writes, persistent finalize, and duplicate release paths, while preserving bare `message_id` behavior for text and malformed or true retry events.

## Reproduction Assessment

Yes for the OpenClaw side: two Feishu receive payloads for the same account with the same `message.message_id`, `message_type: "audio"`, and different JSON `file_key` values collide before media parsing. The live upstream ID-reuse behavior still needs reporter or Feishu-environment confirmation because no logs were attached.

## Solution Assessment

Yes. A Feishu-local media-aware key is the narrowest maintainable fix; a shared core content-hash fallback would change broader channel semantics and is larger than this same-ID/different-audio failure requires.
