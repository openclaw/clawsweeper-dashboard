## Summary

Keep open. Current main still keys the Feishu early receive claim and final persistent duplicate check on the bare message id, so distinct audio payloads with the same Feishu id can still be dropped; open PR #75078 is the linked implementation candidate and should be reviewed before this issue closes.

## What This Changes

The issue reports that Feishu can reuse one message id for different audio attachments, causing distinct voice messages to be incorrectly deduplicated before they reach the agent.

## Best Possible Solution

Land PR #75078 or an equivalent Feishu-owned patch that computes one stable media-aware dedupe key before receive claims and final persistent processing, with any debounce/suppression paths kept consistent and bare message-id behavior preserved for text and true identical retries.

## Reproduction Assessment

Yes. Two Feishu receive payloads for the same account with equal event.message.message_id, message_type audio, and different JSON file_key values collide in the source before audio content parsing, so the second event can be dropped by current main.

## Solution Assessment

Yes. A Feishu-local media-aware key at the existing dedupe call sites is the narrowest maintainable direction; changing shared core dedupe would be broader and could affect unrelated channels.
