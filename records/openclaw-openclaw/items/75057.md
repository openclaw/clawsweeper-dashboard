## Summary

Keep open. Current `main` still keys Feishu receive, debounce, suppressed-message recording, and persistent finalize dedupe on the bare `message_id`, and open PR #75078 is the linked implementation candidate that should be reviewed before this issue closes.

## What This Changes

The issue reports that Feishu WebSocket audio events can reuse the same `message_id` for different audio attachments, causing the second attachment to be dropped by dedupe.

## Best Possible Solution

Land PR #75078 or an equivalent Feishu-owned patch that computes one stable media-aware dedupe key before every Feishu receive claim, debounce freshness check, suppressed-record write, and persistent finalize decision while preserving bare `message_id` behavior for text and true identical retries.

## Reproduction Assessment

Yes. Two Feishu receive payloads for the same account with equal `message.message_id`, `message_type: "audio"`, and different JSON `file_key` values collide at the receive-time claim before audio parsing, so the second payload is dropped.

## Solution Assessment

Yes. A Feishu-local key that incorporates audio media identity at the existing Feishu dedupe call sites is the narrowest maintainable fix; changing shared core dedupe would be broader and risk unrelated channels.
