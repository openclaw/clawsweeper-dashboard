## Summary

Keep open. Current main still dedupes Feishu inbound audio on bare `message_id` across the early processing claim, debounce filtering, suppressed-record writes, and persistent finalize path, while the audio `file_key` is parsed only later. The issue is also already paired with open PR #75078 via closing syntax, so maintainers should review, land, or reject that candidate before closing this issue.

## What This Changes

The issue reports that Feishu WebSocket audio events can reuse a `message_id` for distinct `file_key` audio attachments, causing message-id-only dedupe to drop the second audio before it reaches the agent.

## Best Possible Solution

Land #75078 or an equivalent Feishu-owned patch that computes a stable media-aware dedupe key before every Feishu receive claim, debounce freshness check, suppressed-record write, and persistent finalize decision, while preserving bare `message_id` behavior for text and true identical retries. The patch should include a regression test for same `message_id` plus different audio `file_key` values and a user-facing changelog entry.

## Reproduction Assessment

Yes. Two Feishu receive payloads for the same account with `message.message_id` equal, `message_type: "audio"`, and different JSON `file_key` values collide at `tryBeginFeishuMessageProcessing` before media parsing; the second payload is dropped as a duplicate.

## Solution Assessment

Yes. A Feishu-local media-aware key applied consistently across the existing Feishu dedupe call sites is the narrowest maintainable fix; changing shared core dedupe would be broader and would risk unrelated channels.
