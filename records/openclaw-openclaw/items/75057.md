## Summary

Keep open. Current main still deduplicates Feishu inbound audio on the bare `message_id` before the audio `file_key` is used, so same-account audio events with reused Feishu message IDs can still be dropped. The issue is also paired with open PR #75078, so maintainers should review, land, or close that implementation candidate before closing this issue.

## What This Changes

Feishu should process distinct audio attachments independently when upstream reuses the same `message_id`, by incorporating media identity into the channel dedupe key.

## Best Possible Solution

Land #75078 or an equivalent Feishu-owned patch that computes one stable media-aware dedupe key for early receive claims, debounce suppression, suppressed-record writes, and persistent finalize, using audio `file_key` while preserving current behavior for text and true identical retries.

## Reproduction Assessment

Yes for the OpenClaw side: two Feishu receive payloads for the same account with the same `message.message_id`, `message_type: "audio"`, and different JSON `file_key` values collide before dispatch. The live Feishu ID-reuse part still depends on reporter logs or live Feishu confirmation.

## Solution Assessment

Yes. A Feishu-local media-aware dedupe key applied consistently across all Feishu inbound dedupe stages is the narrow maintainable solution; changing shared core dedupe or adding broad content hashing would be larger than this channel-specific failure requires.
