## Summary

Keep open. Current main still keys the Feishu receive claim and persistent dedupe paths on the bare `message_id`, so the reported reused-id audio case can still drop the second attachment; open PR #75078 is the linked implementation candidate and should be reviewed or replaced before this issue closes.

## What This Changes

Feishu audio messages with the same upstream `message_id` but different attachment `file_key` values should be treated as distinct inbound messages.

## Best Possible Solution

Land PR #75078 or an equivalent Feishu-local patch that computes one stable media-aware dedupe key before every receive claim, debounce freshness check, suppressed-record write, and persistent finalize decision while preserving bare `message_id` behavior for text and true identical retries.

## Reproduction Assessment

Yes. Two Feishu `im.message.receive_v1` audio payloads for the same account with equal `message.message_id` and different JSON `file_key` values collide at the receive-time claim before audio parsing, so the second payload is dropped.

## Solution Assessment

Yes. A Feishu-local media-aware key at the existing Feishu dedupe call sites is the narrowest maintainable fix; changing shared core dedupe would be broader and risk unrelated channels.
