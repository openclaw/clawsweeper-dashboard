## Summary

Keep open. Current main has partial mitigations for stable or OpenClaw-prefix Node paths, but it still documents an external macOS Gateway runtime and still permits launchd service startup through shared/system Node, so the requested OpenClaw-scoped signed or dedicated TCC identity is not implemented.

## What This Changes

Request a dedicated or bundled OpenClaw-owned Node runtime so macOS Full Disk Access, Accessibility, and related TCC grants are scoped to OpenClaw rather than a shared Node executable.

## Best Possible Solution

Choose one supported signed OpenClaw-owned macOS runtime identity for TCC-sensitive Gateway work, then align launchd install/update, doctor, onboarding, and docs around that identity with live macOS proof.

## Reproduction Assessment

Yes at source level: current main still has documented external launchd Gateway paths and runtime=node service arguments that can target shared/system Node, but no live macOS TCC run was performed here.

## Solution Assessment

Unclear on exact shape: a dedicated or bundled signed runtime is the right class of fix for TCC scoping, but maintainers need to choose between app-bundled Node, signed helper/broker, installer-owned runtime, or another lifecycle model.
