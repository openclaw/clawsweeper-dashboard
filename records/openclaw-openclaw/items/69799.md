## Summary

Keep open. Current main and v2026.6.6 have a partial app-installer mitigation that can place Node under `~/.openclaw`, but the supported macOS Gateway path is still an external launchd Node runtime and shared/system Node paths remain valid, so the requested signed or dedicated OpenClaw-owned TCC identity is not fully implemented.

## What This Changes

Ship OpenClaw with a signed or dedicated macOS Node/runtime identity so TCC grants are scoped to OpenClaw instead of a shared Node executable.

## Best Possible Solution

Define one supported signed OpenClaw-owned macOS runtime identity for TCC-sensitive Gateway work, then align launchd install/update, doctor repair, onboarding, and docs around that identity instead of treating shared Node as a normal permission owner.

## Reproduction Assessment

Yes at source level: `openclaw gateway install --runtime node` can still resolve shared/system Node and write that executable into launchd ProgramArguments. I did not run a live macOS TCC smoke in this read-only review.

## Solution Assessment

Unclear. A bundled or signed runtime identity is a plausible end state, but maintainers need to choose the macOS TCC ownership model before a safe implementation can be scoped.
