## Summary

Keep open. Current main still allows the macOS managed Gateway to run through an external/shared Node executable unless an optional wrapper is configured, and the companion-app/PeekabooBridge workaround in the discussion is useful context rather than a shipped OpenClaw-owned TCC identity.

## What This Changes

Request to ship a dedicated or bundled OpenClaw Node/runtime identity so macOS TCC grants apply to OpenClaw rather than a shared Node binary.

## Best Possible Solution

Define a supported signed OpenClaw-owned macOS runtime or app-broker identity for TCC-sensitive Gateway, node-host, and channel work, then align launchd install/update, doctor repair, onboarding, migration, and docs around that model.

## Reproduction Assessment

Yes at source level: `openclaw gateway install --runtime node` without a wrapper can still resolve shared/system Node and write it into launchd `ProgramArguments`. No live macOS TCC smoke was run.

## Solution Assessment

Unclear. A bundled signed app runtime or dedicated Node identity is plausible, but maintainers need to choose the supported macOS TCC ownership model before implementation is safe.
