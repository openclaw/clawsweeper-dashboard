## Summary

Keep open. Current main still documents and implements the macOS Gateway as an external launchd service using Node by default, with shared Homebrew/system Node paths still valid; the macOS app and PeekabooBridge cover some TCC-facing actions, but they do not yet provide the dedicated OpenClaw-owned runtime identity requested here.

## What This Changes

The issue requests an OpenClaw-owned bundled or dedicated Node/runtime identity on macOS so Full Disk Access and Accessibility grants are scoped to OpenClaw rather than a shared Node binary.

## Best Possible Solution

Define a supported signed OpenClaw-owned macOS runtime or app-broker identity for TCC-sensitive Gateway, node-host, and local command work, then align launchd install/update, doctor repair, onboarding, migration, and docs around that model.

## Reproduction Assessment

Yes at source level: `openclaw gateway install --runtime node` can still resolve a shared/system Node executable and write it into launchd `ProgramArguments`. I did not run a live macOS TCC smoke in this read-only review.

## Solution Assessment

Unclear. A bundled signed runtime or dedicated Node identity would address the reported scope problem, but maintainers need to choose whether the durable fix is a bundled runtime, signed helper/app broker, Terminal-backed mode, or a combination.
