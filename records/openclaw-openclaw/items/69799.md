## Summary

Keep open. Current main still lets the macOS managed Gateway run through an external/shared Node executable unless an optional wrapper is configured, and the later companion-app/PeekabooBridge workaround comment is useful context rather than a shipped OpenClaw-owned TCC identity.

## What This Changes

The issue asks OpenClaw to ship a signed/dedicated Node or app-bundled runtime so macOS TCC grants for Full Disk Access and Accessibility attach to OpenClaw instead of a shared Node executable.

## Best Possible Solution

Define a supported signed OpenClaw-owned macOS runtime or app-broker identity for TCC-sensitive Gateway/node-host/channel work, then align launchd install/update, doctor repair, onboarding, migration, and docs around that model.

## Reproduction Assessment

Yes at source level. `openclaw gateway install --runtime node` without a wrapper, including the macOS app local-mode path, can still resolve a shared/system Node executable and write it into launchd service `ProgramArguments`; no live macOS TCC smoke was run.

## Solution Assessment

Unclear. A bundled signed app runtime or dedicated Node identity is plausible, but maintainers need to choose the supported macOS TCC model before implementation is safe.
