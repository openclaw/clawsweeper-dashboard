## Summary

Keep open. Current main still has an externally hosted Node runtime path for macOS Gateway installs: the macOS app enables the LaunchAgent through `openclaw gateway install --runtime node`, the install plan resolves the active or system Node binary, and service arguments use that Node path unless an optional wrapper is explicitly configured. App-owned IPC and PeekabooBridge paths mitigate some TCC-facing work, but they do not fully provide the requested default OpenClaw-owned TCC identity for Gateway/iMessage-style process contexts.

## What This Changes

Request a dedicated or bundled OpenClaw-owned Node/runtime identity on macOS so Full Disk Access, Accessibility, and related TCC grants are scoped to OpenClaw rather than a shared Node binary.

## Best Possible Solution

Keep this as core macOS security and packaging work. The desired end state is a stable OpenClaw-owned executable identity for TCC-sensitive Gateway and node-host operations, with installer, LaunchAgent, doctor, onboarding, and docs aligned with the related FDA and Accessibility reports.

## Reproduction Assessment

Yes. The source-level reproduction is `openclaw gateway install --runtime node` on macOS, or enabling the macOS app's local LaunchAgent path: current install planning resolves an active/system Node path and writes it into service ProgramArguments unless an optional wrapper is supplied.

## Solution Assessment

No. The current mitigations are useful but incomplete because external/global Node can still be the service executable; a maintainable fix needs a supported OpenClaw-owned runtime identity or equivalent signed app-owned boundary rather than relying on an optional wrapper workaround.
