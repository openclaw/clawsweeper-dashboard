## Summary

Keep open. Current main still lets the managed macOS Gateway run through an external/shared Node executable unless an optional wrapper is configured, so the requested OpenClaw-owned TCC identity is not implemented and remains a security/product design decision.

## What This Changes

Request a dedicated or bundled OpenClaw Node runtime on macOS so Full Disk Access, Accessibility, and related TCC grants attach to OpenClaw rather than a shared Node binary.

## Best Possible Solution

Define a supported signed OpenClaw-owned macOS runtime or app-broker identity for TCC-sensitive Gateway/node-host/channel work, then align launchd install/update, doctor repair, onboarding, migration, and docs around that model.

## Reproduction Assessment

Yes at source level, but not with live macOS TCC proof. `openclaw gateway install --runtime node` without `--wrapper`, including the macOS app local-mode path, can resolve a shared/system Node executable and write it as service `ProgramArguments`.

## Solution Assessment

Unclear. A bundled signed app runtime is plausible, but maintainers still need to choose between a bundled app runtime, a dedicated Node path, or routing more work through the signed app broker before implementation is safe.
