## Summary

Keep open. Current main still supports the macOS Gateway running through an external/shared Node executable unless an optional wrapper is configured, so the requested default OpenClaw-owned TCC boundary is not implemented.

## What This Changes

Request to ship a dedicated or bundled OpenClaw Node/runtime identity so macOS TCC grants for Full Disk Access and Accessibility are scoped to OpenClaw rather than a shared Node install.

## Best Possible Solution

Define a supported signed OpenClaw-owned macOS runtime or broker identity for TCC-sensitive Gateway/node-host/channel work, then align launchd install/update, doctor repair, onboarding, migration, and public docs around that model.

## Reproduction Assessment

Yes at source level. `openclaw gateway install --runtime node` without `--wrapper`, including the macOS app local-mode path, can resolve a shared/system Node executable and write it as the LaunchAgent executable; no live macOS TCC prompt test was run here.

## Solution Assessment

Unclear. A signed app-bundled runtime is plausible, but maintainers still need to choose between a bundled app runtime, a dedicated Node path, or routing more work through the signed app broker before implementation is safe.
