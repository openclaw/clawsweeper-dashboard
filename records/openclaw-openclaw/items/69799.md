## Label Justifications

- `P1`: This is a high-privilege macOS TCC boundary concern affecting real FDA and Accessibility grants for users running OpenClaw through shared Node.
- `impact:security`: The issue is about whether unrelated Node workloads can inherit OpenClaw-requested macOS privacy permissions.

## Summary

Keep open. Current main still supports the macOS managed Gateway through an external/shared Node executable, and the app-managed launchd path explicitly installs the Gateway with `--runtime node`; the companion-app and PeekabooBridge workaround reduces some paths but does not provide a shipped OpenClaw-owned runtime identity for this issue.

## What This Changes

The issue requests a bundled or dedicated OpenClaw-owned Node/runtime identity on macOS so TCC grants apply to OpenClaw rather than a shared `node` executable.

## Best Possible Solution

Define a supported signed OpenClaw-owned macOS runtime or app-broker identity for TCC-sensitive Gateway, node-host, and channel work, then align launchd install/update, doctor repair, onboarding, migration, and docs around that model.

## Reproduction Assessment

Yes at source level: `openclaw gateway install --runtime node` without a wrapper can still resolve a shared/system Node and write it into launchd `ProgramArguments`. No live macOS TCC smoke was run.

## Solution Assessment

Unclear. A bundled signed app runtime or dedicated Node identity is plausible, but maintainers need to choose the supported macOS TCC ownership model before implementation is safe.
