## Summary

Keep open. Current main has useful mitigations, including app-owned PeekabooBridge/system.run paths and an optional persisted service wrapper, but the supported macOS local Gateway path can still run through an external/shared Node binary by default, so the requested OpenClaw-owned TCC boundary is not implemented.

## What This Changes

The issue requests a bundled or dedicated OpenClaw-owned Node/Gateway runtime on macOS so Full Disk Access, Accessibility, and related TCC grants apply to OpenClaw instead of a shared system or version-manager Node binary.

## Best Possible Solution

Keep this as core macOS security and packaging work: choose a stable OpenClaw-owned signed runtime identity for TCC-sensitive Gateway and node-host operations, make it the supported default, and align install, update, LaunchAgent repair, doctor, onboarding, and docs around that boundary while preserving explicit wrapper escape hatches.

## Reproduction Assessment

Yes. The source-level reproduction is macOS local mode or `openclaw gateway install --runtime node` without `--wrapper`: current main resolves an active/system Node path and writes service `ProgramArguments` through that executable.

## Solution Assessment

Unclear for the exact packaging option, but the requested end state is directionally right. Optional wrappers and app-owned automation bridges reduce exposure, yet the maintainable fix needs a maintainer-chosen signed app/dedicated runtime identity rather than relying on user-supplied wrappers or docs alone.
