## Summary

Current main and the shipped v2026.5.7 release already contain the PDF.js fallback-rendering fix that matches this old-version failure: scanned/image-only PDFs now go through text extraction, then page-image rendering for sparse text, with the font-path issue fixed in the document-extract plugin.

## What This Changes

The issue reports that image-only scanned PDFs fail OpenClaw PDF analysis with a document extraction error despite being valid renderable PDFs.

## Best Possible Solution

Keep the released PDF fallback implementation and have users on 2026.4.26 update to v2026.5.7 or newer; any failure on the same file after updating should be filed as a new sample-specific report.

## Reproduction Assessment

Yes for the old release: the issue includes concrete gateway steps, model config, logs, and external validation showing a valid sparse-text scanned PDF that reaches the PDF extraction path. I did not reproduce a current-main failure because the sample PDF was not attached, and current main contains the matching released fix.

## Solution Assessment

Yes: resolving the PDF.js standard-font data path in the document-extract plugin is the narrowest maintainable fix for the reported extraction/rendering failure, while preserving the existing text-then-image fallback behavior for scanned PDFs.
