## Summary

Keep open: current main still has a plausible source-level failure path for image-only scanned PDFs in the non-native PDF extraction fallback, and the related Poppler fallback issue does not fully cover this Raspberry Pi thrown-extraction failure mode.

## What This Changes

The issue asks for the PDF tool to analyze valid image-only scanned PDFs with fallback vision models instead of failing with document extraction errors.

## Best Possible Solution

Keep this open until maintainers either consolidate it into https://github.com/openclaw/openclaw/issues/75358 with the Raspberry Pi thrown-error evidence preserved, or land a plugin-owned fix that handles PDF.js/canvas render failures with focused regression and live scanned-PDF proof.

## Reproduction Assessment

No high-confidence live reproduction was established because the sample PDF and Raspberry Pi setup are not available here. Source inspection does show the current non-native PDF path reaches the documented PDF.js/canvas renderer and wraps extractor throws in the exact `Document extraction failed for application/pdf` message.

## Solution Assessment

Unclear. The fix likely belongs in `extensions/document-extract`, but maintainers should decide whether to extend the previous Poppler fallback direction, catch/render fallback failures, or first request a shareable fixture before accepting an external-renderer path.
