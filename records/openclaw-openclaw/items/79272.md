## Summary

Keep open: the WhatsApp force-document plumbing is useful, but the new forced-document payload branch uses raw caption text and can regress native mention replacement in group captions.

## What This Changes

The PR threads `forceDocument`/`asDocument` through WhatsApp outbound media, updates docs/CLI/tool schema/snapshots/changelog, and adds WhatsApp send coverage.

## Best Possible Solution

Keep the targeted WhatsApp plumbing, change the forced-document branch to reuse the resolved caption text, and add focused group-mention coverage before landing.

## Reproduction Assessment

Yes, at source level: current main accepts `forceDocument`/`asDocument` in shared message surfaces but WhatsApp drops it before `sendMessageWhatsApp`, and the payload selector routes image MIME to `image` instead of `document`. I did not run a live WhatsApp send in this read-only review.

## Solution Assessment

No, not as submitted. The approach is narrow, but the new forced-document branch should use `resolvedPayloadText.text` so group caption mention behavior stays aligned with the existing media/document branches.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve resolved captions for forced documents:** `extensions/whatsapp/src/inbound/send-api.ts:87`
  - body: The new `asDocument` branch builds the document caption from raw `text`, while the existing image/video/document branches use `resolvedPayloadText.text` before adding mention metadata. In WhatsApp groups with LID-backed mention replacement, forced-document captions can keep the unreplaced phone token while still adding LID mentions, so native mention rendering can regress on the new path. Use `resolvedPayloadText.text` here and cover forced-document captions in the mention test.
