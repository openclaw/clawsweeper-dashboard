## Summary

Keep open: the PR is a focused WhatsApp fix, but the current patch still introduces a caption regression in the new forced-document branch by bypassing resolved WhatsApp mention text.

## What This Changes

The PR threads `forceDocument`/`asDocument` through WhatsApp outbound media, sends forced media as Baileys document payloads, preserves filenames, updates docs/CLI/schema snapshots, and adds focused tests.

## Best Possible Solution

Keep the targeted WhatsApp plumbing, change the forced-document branch to use resolved caption text, add group mention coverage for that branch, then land after targeted tests and CI are green; leave broader media-fragment or channel-default API choices to https://github.com/openclaw/openclaw/issues/41965.

## Reproduction Assessment

Yes, at source level: current main reads `forceDocument`/`asDocument` and carries it through generic outbound delivery, but WhatsApp drops it before `sendMessageWhatsApp` and then routes image MIME media to an image payload. I did not run a live WhatsApp send in this read-only review.

## Solution Assessment

No, not as submitted. The overall owner-module plumbing is the narrow maintainable fix, but the new forced-document branch should reuse `resolvedPayloadText.text` so group mention captions stay aligned with the existing media/document paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve resolved captions for forced documents:** `extensions/whatsapp/src/inbound/send-api.ts:87`
  - body: The new `asDocument` branch builds the document caption from raw `text`, while the existing image/video/document paths use `resolvedPayloadText.text` before mention metadata is added. Use the resolved caption text here and cover forced-document group captions in the mention test.
