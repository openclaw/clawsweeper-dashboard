## Label Justifications

- `P2`: This is a bounded gateway/channel routing bug that can misroute messages after binding edits but is not a core outage.
- `impact:message-loss`: Stale binding snapshots can route inbound channel messages to the wrong agent until another config reload or gateway restart refreshes the snapshot.

## Summary

Keep open: the corrected report is still valid on current main and in the latest release; binding-only edits are still classified as no-op reloads, and the external reload path returns before any runtime snapshot swap can make the new bindings visible to route resolution.

## What This Changes

The issue requests a gateway reload/runtime-snapshot fix so binding-only config edits refresh channel routing without requiring a gateway restart.

## Best Possible Solution

Adopt a core gateway reload contract that advances the runtime snapshot for binding/routing no-op edits without restarting channels, then document that channel routing reads the active snapshot per inbound.

## Reproduction Assessment

Yes, at source level: a binding-only edit maps to a no-op reload plan, and the external reload path returns before any runtime snapshot swap. I did not run a live channel repro in this read-only review.

## Solution Assessment

Unclear. The right layer is core gateway/runtime snapshot handling, but maintainers should choose whether this is an explicit snapshot-only reload kind or unconditional snapshot advancement for changed no-op paths.
