## Label Justifications

- `P2`: This is a bounded gateway/channel routing bug that can misroute messages after dynamic binding edits, with a restart workaround and limited blast radius.
- `impact:message-loss`: A stale binding snapshot can route inbound channel messages to the wrong agent until the gateway snapshot refreshes or the gateway restarts.

## Summary

Keep open: current main and v2026.6.9 still return before the hot-reload path for binding/routing-only no-op config edits, while runtime config reads stay pinned and routing caches by config object identity. The corrected issue remains source-reproducible and needs a maintainer choice on the stable gateway snapshot contract.

## What This Changes

The issue asks OpenClaw to document or centralize the channel-plugin live-routing config contract, then corrects the request to a core gateway reload fix so binding/routing-only config edits advance the runtime config snapshot without forcing a gateway restart.

## Best Possible Solution

Advance the active runtime snapshot for binding/routing-only config edits without restarting channels, add a regression proving the next route resolution sees updated bindings, then document the settled channel-plugin contract.

## Reproduction Assessment

Yes, at source level: binding/routing-only edits build a no-op reload plan, return before hot reload activation, and leave getRuntimeConfig pinned to the old snapshot that routing caches by object identity. I did not run a live gateway scenario in this read-only review.

## Solution Assessment

Unclear: the fix belongs in gateway reload/runtime snapshot handling, but maintainers still need to choose between unconditional snapshot advancement for changed no-op paths and an explicit snapshot-only reload kind.
