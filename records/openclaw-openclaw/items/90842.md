## Label Justifications

- `P2`: The gap can cause channel plugins to route inbound traffic to stale bindings, but the remaining work is a bounded docs/API design improvement rather than an urgent core outage.
- `impact:message-loss`: The reported failure mode is inbound channel messages being routed to the wrong agent when a plugin reuses a stale config reference after binding edits.

## Summary

Keep open: current main has the per-inbound config pattern in Telegram and QQBot, but the channel-plugin author docs still do not spell out the binding/routing contract and the reusable helper remains plugin-local, so the central request is not implemented yet.

## What This Changes

Request to document the channel-plugin per-inbound `getRuntimeConfig()`/`resolveAgentRoute` contract and optionally promote the QQBot `ActiveCfgProvider` pattern into the plugin SDK.

## Best Possible Solution

Document the per-inbound runtime-config routing contract in the channel plugin docs and nearby source comments; add a public SDK helper only if maintainers explicitly want that pattern stabilized and update exports, API baseline, docs, and focused tests together.

## Reproduction Assessment

Yes, source-level: `bindings`/`routing` are no-op reload sections and routing caches by config object reference, while QQBot tests show why a fresh config reference matters per inbound. I did not run the third-party Octo live reproduction.

## Solution Assessment

Unclear until maintainers choose the API direction: docs-only is the narrowest low-risk fix, while a shared helper reduces duplication but makes the pattern a public SDK contract.
