## Label Justifications

- `impact:message-loss`: A stale binding snapshot can route inbound channel messages to the wrong agent until the gateway refreshes or restarts.

## Summary

Keep open: current main still has the corrected source-level failure mode, and the closed docs PR no longer owns the remaining work. Binding-only edits are still planned as no-op reloads, so maintainers need to choose the gateway reload snapshot contract before a fix is queued.

## What This Changes

The issue requests documentation or centralization for channel plugins seeing updated routing bindings, later corrected to a core gateway snapshot-swap problem for binding-only config edits.

## Best Possible Solution

Make the gateway reload contract explicit in core, then add a focused implementation and regression coverage proving external binding edits advance the runtime config snapshot without restarting channels unnecessarily.

## Reproduction Assessment

Yes at source level: a binding-only diff reaches the reload watcher, is classified as a no-op plan, and returns before any hot-reload handler can activate a new runtime config snapshot. I did not execute a live gateway scenario in this read-only review.

## Solution Assessment

Unclear: an unconditional snapshot swap for changed no-op paths looks like the narrowest code shape, but maintainers need to choose that contract versus an explicit snapshot-swap-only rule for routing-related prefixes.
