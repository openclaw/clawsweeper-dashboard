## Label Justifications

- `P2`: This is a bounded gateway/channel routing bug that can misroute messages after binding edits, but it is not an emergency outage or security bypass.
- `impact:message-loss`: Stale binding snapshots can route inbound channel messages to the wrong agent until the gateway is restarted.

## Summary

Keep open: current main still has the corrected gateway reload failure mode, and no merged related item now owns the source fix. Binding/routing-only edits are still classified as no-op reloads, and the reload loop returns before the hot-reload path can activate a new runtime config snapshot.

## What This Changes

The issue asks to document or centralize the channel-plugin live-binding contract, then corrects the remaining work to a core gateway reload fix so binding/routing config edits refresh runtime state without restarting the gateway.

## Best Possible Solution

Make external binding/routing no-op config edits advance the process runtime snapshot without restarting channels, add focused regression coverage for that reload path, then document the channel-plugin live-binding expectation.

## Reproduction Assessment

Yes. Source inspection on current main shows a binding/routing-only edit produces a no-op reload plan and returns before the runtime snapshot activation path, which matches the reporter's corrected reproduction.

## Solution Assessment

Unclear. The fix belongs in the gateway reload/runtime snapshot layer, but maintainers still need to choose whether the permanent contract is unconditional snapshot advancement for changed no-op paths or an explicit snapshot-only reload kind.
