## Label Justifications

- `P2`: This is a bounded gateway/channel routing bug that can route messages to the wrong agent after binding edits, but it is not a confirmed emergency or broad outage.
- `impact:message-loss`: Stale bindings can misroute inbound channel messages until the gateway restarts or the runtime snapshot advances.

## Summary

Keep open: current main and v2026.6.6 still classify `bindings`/`routing` as no-op reload paths and return before the hot-reload activation path, while the public gateway docs say `bindings` changes do not need a restart. The closed docs draft no longer owns the corrected source-level fix, so this needs a maintainer decision on the gateway reload contract.

## What This Changes

The issue requests documentation or centralization of the channel routing config contract, then corrects the remaining work to a server-side reload fix so binding-only edits refresh the runtime snapshot used by routing.

## Best Possible Solution

Make binding-only config edits advance the active runtime snapshot without restarting gateway or channels, then document the channel plugin routing-config contract once the source behavior is correct.

## Reproduction Assessment

Yes, at source level: a binding-only config edit maps to a no-op reload plan, and current `applySnapshot` returns before the hot-reload path can activate a new runtime snapshot. I did not run a live channel repro in this read-only review.

## Solution Assessment

Unclear: a source fix is needed, but maintainers should choose the permanent contract between unconditional snapshot advancement for changed no-op paths and an explicit snapshot-only reload rule for `bindings`/`routing`.
