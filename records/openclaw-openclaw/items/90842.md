## Label Justifications

- `P2`: This is a bounded gateway/channel routing bug that can misroute messages after binding edits, but it is not a confirmed emergency or broad outage.
- `impact:message-loss`: Stale binding evaluation can route inbound channel messages to the wrong agent until the runtime snapshot is refreshed or the gateway is restarted.

## Summary

Keep open: current main still has the corrected source-level failure mode where binding/routing-only edits are classified as no-op reloads and return before the runtime config snapshot is advanced. The closed docs draft and the later Feishu plugin-specific fix do not resolve the core gateway reload contract.

## What This Changes

The issue asks OpenClaw to document or centralize the channel-plugin per-event config re-resolve contract, later corrected to a core reload fix for binding-only runtime snapshot refresh.

## Best Possible Solution

Update the gateway reload pipeline so binding/routing changes advance the active runtime config snapshot without a gateway or channel restart, then document the plugin-facing routing contract.

## Reproduction Assessment

Yes, at source level: binding-only external edits build a no-op plan and current applySnapshot returns before the snapshot activation path. I did not run a live third-party channel repro in this read-only review.

## Solution Assessment

Unclear: a source fix is needed, but maintainers should choose between unconditional snapshot advancement for changed no-op paths and an explicit snapshot-only reload kind. Docs or an SDK helper alone would not fix the corrected runtime failure.
