## Label Justifications

- `P2`: This is a bounded gateway/channel routing bug that can misroute messages after binding edits, with a restart workaround and limited blast radius.
- `impact:message-loss`: A stale binding snapshot can route inbound channel messages to the wrong agent until the gateway snapshot is refreshed or restarted.

## Summary

Keep open: current main and the latest release still classify binding/routing-only config edits as no-op reloads that return before any hot-reload path can advance the active runtime snapshot, so the corrected routing bug remains unsolved and no open or merged PR owns the central gateway fix.

## What This Changes

The issue asks OpenClaw to document or centralize channel-plugin live-binding config behavior, then corrects the root cause to the gateway no-op reload path not advancing the pinned runtime config snapshot for binding/routing-only edits.

## Best Possible Solution

Advance the active runtime snapshot for binding/routing-only config edits without restarting channels, add a regression proving the next route resolution sees updated bindings, then document the settled channel-plugin contract.

## Reproduction Assessment

Yes, at source level: binding/routing-only edits build a no-op reload plan, return before hot reload activation, and leave `getRuntimeConfig()` pinned to the old snapshot that routing caches by object identity.

## Solution Assessment

Unclear: the fix belongs in gateway reload/runtime snapshot handling, but maintainers still need to choose between unconditional snapshot advancement for changed no-op paths and an explicit snapshot-only reload kind.
