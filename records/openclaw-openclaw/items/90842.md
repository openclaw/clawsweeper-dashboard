## Summary

Keep open: current main has the per-inbound config pattern in Telegram and QQBot, and the QQBot regression is fixed, but the channel-author docs still do not spell out the live-bindings contract and the reusable helper remains plugin-local. The external channel report and the prior ClawSweeper review both support leaving this for maintainer direction rather than closing it as implemented.

## What This Changes

Request to document the per-inbound runtime-config re-resolve requirement for channel plugins and possibly promote the duplicated active-config helper into the public Plugin SDK.

## Best Possible Solution

Add explicit channel-author docs for live binding/routing config first, then add a narrow public SDK helper only if plugin SDK owners approve that API shape.

## Reproduction Assessment

Yes, from source: a channel that keeps passing an old cfg reference into `resolveAgentRoute` can reuse binding caches while `bindings` reload is planned as `none`. The bundled QQBot regression test demonstrates the same boundary by changing cfg between inbound events.

## Solution Assessment

Unclear pending maintainer choice: documenting the contract is the narrowest safe solution, while centralizing `createActiveCfgProvider` is plausible but would add public SDK surface. The issue should stay open until owners choose docs-only, SDK helper, or both.
