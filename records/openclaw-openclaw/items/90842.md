## Label Justifications

- `P2`: This is a bounded gateway/channel routing bug that can send messages to the wrong agent after binding edits, but it is not a confirmed emergency outage.
- `impact:message-loss`: Stale bindings can misroute inbound channel messages until the gateway or affected runtime snapshot is refreshed.

## Summary

Keep open: current main still has the corrected source-level failure mode, where binding/routing edits are classified as no-op reloads and the reload path returns before the active runtime snapshot is refreshed. The closed docs draft and adjacent channel-side fixes do not solve the core external-reload contract.

## What This Changes

Request to fix and document the gateway/channel runtime config contract so binding-only config edits reach channel routing without a gateway restart.

## Best Possible Solution

Make binding/routing no-op config changes refresh the process runtime snapshot without restarting the gateway, add focused regression coverage, then document the channel author contract.

## Reproduction Assessment

Yes, at source level: binding-only edits produce a no-op reload plan, and current `applySnapshot` returns before the runtime snapshot activation path. I did not run a live channel repro in this read-only review.

## Solution Assessment

Unclear: a source fix is needed, but maintainers should choose between an explicit snapshot-only reload kind and unconditional snapshot advancement for changed no-op paths. A channel-local `loadConfig({ pin: false })` workaround would add hot-path disk parsing and is not the durable fix.
