## Label Justifications

- `P2`: This is a bounded gateway/channel routing bug with wrong-agent delivery after binding edits, but it is not an emergency or confirmed broad outage.
- `impact:message-loss`: Stale bindings can misroute inbound channel messages to the wrong agent until a non-no-op config reload or gateway restart.

## Summary

Keep open: current main and v2026.6.6 still have the corrected failure mode where binding-only edits are classified as no-op reloads, return before runtime snapshot activation, and leave routing caches keyed to the stale config object.

## What This Changes

Requested a documented or centralized per-event runtime config re-resolve contract for channel plugins, later corrected to require a gateway reload/runtime-snapshot fix for binding-only edits.

## Best Possible Solution

Make binding-only external config edits advance the process runtime snapshot without restarting gateway/channels, then document the channel per-inbound routing-config contract.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: a binding-only diff creates a no-op plan, `applySnapshot` returns before snapshot activation, and routing caches by the unchanged config object.

## Solution Assessment

The original docs-only direction is not sufficient; a source fix is needed, but maintainers should choose between explicit snapshot-only reload classification and unconditional snapshot advancement for changed no-op paths.
