## Label Justifications

- `P2`: This is a source-backed gateway/channel routing bug with wrong-agent delivery after binding edits, but it is not a confirmed emergency outage.
- `impact:message-loss`: Stale binding evaluation can route inbound channel messages to the wrong agent until the gateway snapshot advances or the gateway restarts.

## Summary

Keep open: current main still has the corrected source-level failure mode. Binding-only reloads are planned as no-op changes, the reload loop returns before activating a new runtime config snapshot, and routing caches remain tied to the old config object.

## What This Changes

The issue asks OpenClaw to fix and document the contract that channel routing must see updated bindings after config changes without a gateway restart.

## Best Possible Solution

Make binding-only config edits advance the active runtime snapshot without restarting the gateway or channels, then document the channel inbound routing contract after the core behavior is fixed.

## Reproduction Assessment

Yes, at source level: a binding-only change is classified as a no-op reload, `applySnapshot` returns before hot reload activation, `getRuntimeConfig()` remains pinned, and routing caches by the old config object. I did not run a live channel repro in this read-only review.

## Solution Assessment

Unclear: a source fix is needed, but maintainers should choose between explicit snapshot-only reload semantics and unconditional snapshot advancement for changed no-op paths before implementation. Per-channel `pin: false` reads are a workaround, not the best core fix.
