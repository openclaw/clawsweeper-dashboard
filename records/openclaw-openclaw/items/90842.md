## Label Justifications

- `impact:message-loss`: Stale binding snapshots can route inbound channel messages to the wrong agent until the gateway is restarted.

## Summary

Keep open: current main and the latest release still have the corrected reload-snapshot failure mode, and no related merged or open PR now owns the central gateway contract. The withdrawn docs draft and merged channel-local fixes are useful context, but they do not fix binding/routing-only config edits failing to advance the active runtime config snapshot.

## What This Changes

The issue asks OpenClaw to document or centralize the channel-plugin live-binding contract, now corrected to a gateway reload/runtime-snapshot fix for binding/routing-only config edits.

## Best Possible Solution

Advance the active runtime snapshot for binding/routing-only config edits without restarting channels, add focused reload regression coverage, and document the channel-plugin expectation after the source contract is clear.

## Reproduction Assessment

Yes, at source level. A binding-only config edit builds a no-op reload plan and returns before the active runtime snapshot can advance, while routing caches by the old config identity.

## Solution Assessment

Unclear. The fix belongs in the gateway reload/runtime snapshot layer, but maintainers still need to choose whether the permanent contract is unconditional snapshot advancement for changed no-op paths or an explicit snapshot-only reload kind.
