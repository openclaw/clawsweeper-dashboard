## Summary

Keep open: current main still has the corrected source-level failure mode, where binding-only config edits are classified as no-op reloads and the reload loop returns before the runtime snapshot is swapped. The closed docs draft no longer owns the real fix, and the remaining work needs a maintainer choice on the gateway reload contract.

## What This Changes

The issue asks OpenClaw to clarify and fix how channel plugins get fresh runtime config for per-event routing after binding-only config edits.

## Best Possible Solution

Make binding-only external config edits advance the process runtime snapshot without restarting channels or the gateway, then document the per-inbound routing-config contract for channel plugins.

## Reproduction Assessment

Yes, at source level: an external binding-only config change produces a no-op reload plan, and current `applySnapshot` returns before the runtime snapshot activation path. I did not run a live channel repro in this read-only review.

## Solution Assessment

Unclear: a source fix is needed, but the best permanent shape needs maintainer direction between an explicit snapshot-only reload kind and unconditional snapshot advancement for changed no-op paths.
