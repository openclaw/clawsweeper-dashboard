## Label Justifications

- `P2`: This is a bounded gateway/channel routing bug that can misroute messages after binding edits, but it is not an emergency outage or security bypass.
- `impact:message-loss`: Stale binding snapshots can route inbound channel messages to the wrong agent until the gateway is restarted.

## Summary

Keep open: current main still lets binding/routing-only config edits produce a no-op reload that returns before the active runtime snapshot advances, so the corrected user problem remains unsolved and no related open or merged PR owns the central fix.

## What This Changes

Request to make channel-plugin live binding/routing config edits take effect without a gateway restart, with the corrected fix surface in gateway reload snapshot handling rather than docs-only guidance.

## Best Possible Solution

Advance the active runtime snapshot for binding/routing-only config edits without restarting channels, then add focused reload regression coverage and update plugin docs once the source contract is clear.

## Reproduction Assessment

Yes, at source level: binding/routing-only edits build a no-op reload plan, return before hot reload activation, and leave route caches keyed to the old pinned config reference.

## Solution Assessment

Unclear: the fix belongs in gateway reload/runtime snapshot handling, but maintainers still need to choose between unconditional snapshot advancement for changed no-op paths and an explicit snapshot-only reload kind.
