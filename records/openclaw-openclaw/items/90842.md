## Label Justifications

- `impact:message-loss`: Stale bindings can route inbound channel messages to the wrong agent until the gateway runtime snapshot advances or the gateway restarts.

## Summary

Keep open: current main and the latest release still show the corrected source-level failure mode where external binding-only config edits are accepted as no-restart changes but can fail to advance the process runtime snapshot, leaving channel routing on stale bindings. The closed docs draft at https://github.com/openclaw/openclaw/pull/90852 no longer owns the corrected runtime fix; this needs a maintainer choice on the gateway reload contract.

## What This Changes

The issue requests documenting or centralizing the channel-plugin live-routing config contract, now corrected to a core gateway reload bug where binding-only edits should refresh the active runtime config snapshot without a gateway restart.

## Best Possible Solution

Make external no-restart config edits for routing-owned surfaces advance the active runtime snapshot without restarting the gateway or channels, then add focused regression coverage and concise plugin-facing docs for the live-routing contract.

## Reproduction Assessment

Yes at source level: direct CLI binding edits write `bindings`, the gateway file-watcher path classifies those paths as no-op, returns before runtime activation, and `getRuntimeConfig()` keeps serving the pinned snapshot. I did not run a live channel repro in this read-only review.

## Solution Assessment

Unclear: a core source fix is needed, but maintainers should choose between an explicit snapshot-only reload kind and unconditional snapshot advancement for non-empty no-op diffs. Forcing channel plugins to call `loadConfig({ pin: false })` would be the wrong hot-path workaround.
