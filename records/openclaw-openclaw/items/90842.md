## Label Justifications

- `P2`: This is a bounded gateway reload/routing correctness bug that can leave channel bindings stale until restart, but it is not an emergency outage.
- `impact:message-loss`: A stale route can deliver inbound channel messages to the wrong agent or leave binding changes ignored until restart.

## Summary

Keep open: current main and v2026.6.6 still return early for no-op binding/routing reload plans before advancing the runtime snapshot, so channel plugins can continue seeing a stale config object after binding-only edits.

## What This Changes

Binding-only config edits should refresh the active runtime snapshot and channel-plugin routing contract without requiring a gateway restart.

## Best Possible Solution

Advance the runtime snapshot for binding, agent, and routing no-op config changes without restarting channels or the gateway, then document the channel routing/live-binding contract.

## Reproduction Assessment

Yes at source level: binding-only config edits are classified as no-op, `applySnapshot` returns before runtime snapshot activation, and routing caches by config object identity; no live channel repro was run because this review was read-only.

## Solution Assessment

No for the docs-only path: the corrected issue now needs a source reload fix first, with maintainer choice on the exact reload contract, and documentation should follow that behavior.
