## Label Justifications

- `P2`: This is a bounded gateway/channel routing bug with wrong-agent delivery after binding edits, but it is not an emergency or confirmed broad outage.
- `impact:message-loss`: Stale bindings can route inbound channel messages to the wrong agent until a gateway restart.

## Summary

Keep open: current main and the latest release still return before the hot-reload path for binding-only no-op reload plans, so per-inbound runtime-config consumers can keep seeing stale bindings. The related docs-only PR is closed and no open implementation PR owns the corrected core fix.

## What This Changes

Request to define or centralize the gateway/channel contract so binding-only config edits refresh routing snapshots for channel plugins without requiring a gateway restart.

## Best Possible Solution

Advance the active runtime config snapshot for binding/agent/routing-only edits without restarting gateway or channel services, then document the channel-plugin inbound config contract.

## Reproduction Assessment

Yes, at source level: a binding-only config edit maps to a no-op reload plan, and current `applySnapshot` returns before runtime snapshot activation. I did not run a live channel repro in this read-only review.

## Solution Assessment

Unclear: the corrected source fix is valid, but maintainers should choose between an explicit snapshot-only reload kind and unconditional snapshot advancement for changed no-op paths. The closed docs-only direction no longer solves the real failure mode.
