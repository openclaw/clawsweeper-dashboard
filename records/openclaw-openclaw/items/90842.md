- `impact:message-loss`: Stale route bindings can send inbound channel messages to the wrong agent until the gateway is restarted.

## Summary

Keep open: current main still leaves the corrected gateway reload contract unresolved for external binding/routing edits. The closed docs draft no longer owns the real fix, and a recent Feishu-specific merge is useful adjacent evidence but does not replace a core decision for no-op reload snapshot advancement.

## What This Changes

The issue requests a documented or centralized gateway/channel runtime-config contract so binding and routing config edits take effect without requiring a gateway restart.

## Best Possible Solution

Make binding/routing-only external config edits advance the active runtime snapshot without restarting channels or the gateway, then add focused tests and docs that lock the per-inbound dynamic routing contract.

## Reproduction Assessment

Yes, at source level: external `openclaw agents bind` writes change `bindings`, the gateway watcher classifies that as a no-op reload, and the pinned runtime config plus route-cache identity path can keep the old bindings active. I did not run a live channel repro in this read-only review.

## Solution Assessment

Unclear: the issue identifies viable core fix shapes, but the best permanent solution needs maintainer direction between unconditional runtime snapshot advancement for changed no-op paths and an explicit snapshot-only reload rule.
