## Label Justifications

- `P2`: This is a bounded gateway/channel routing bug that can route messages to the wrong agent after binding edits, but it is not a confirmed broad outage.
- `impact:message-loss`: Stale bindings can misroute inbound channel messages to the default or old agent until another reload path or restart changes the runtime config reference.

## Summary

Keep open: current main still has the corrected source-level failure mode, and the closed docs PR no longer owns the fix. Binding-only external config edits are classified as no-op reloads, the reload watcher returns before the hot-reload/snapshot activation path, and routing caches by the pinned config object reference.

## What This Changes

The issue asks OpenClaw to document or centralize the channel-plugin routing config refresh contract, then corrects that request to a gateway reload snapshot-swap fix for binding-only edits.

## Best Possible Solution

Make external binding-only edits advance the active runtime config snapshot without restarting channels or the gateway, then document the channel routing config contract in the plugin docs.

## Reproduction Assessment

Yes, at source level. An external binding-only edit diffs to a no-op reload plan, returns before hot reload/snapshot activation, and routing then keeps using the same pinned config reference.

## Solution Assessment

Unclear as a final implementation choice. A source fix is needed, but maintainers should choose between unconditional snapshot advancement for changed no-op paths and an explicit snapshot-only reload classification before a PR is queued.
