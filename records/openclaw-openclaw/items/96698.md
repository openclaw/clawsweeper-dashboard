## Summary

Keep open: current main still has a source-level path where reply-session initialization revisions compare whole JSON-stringified session entries while skill prompt blob hydration can change the in-memory `skillsSnapshot` shape/order. The open Telegram PR overlaps the generic stale-snapshot symptom but does not address this reported hydration mismatch, so this issue remains actionable.

## What This Changes

Fix the Discord second-message failure caused by reply-session initialization treating equivalent `skillsSnapshot` data as a stale snapshot after prompt blob hydration.

## Best Possible Solution

Normalize reply-session initialization revisions at the session accessor/storage boundary so persisted and hydrated `skillsSnapshot` entries compare by the same durable shape, with a focused regression test for prompt blob hydration.

## Reproduction Assessment

Yes at source level: the current revision guard JSON-stringifies the full entry, while skill snapshot creation and blob hydration can preserve the same data with different `skillsSnapshot` shape/order. I did not run the live Discord two-message flow in this read-only review.

## Solution Assessment

Yes, the reporter's suggested persisted-shape or normalized revision comparison is the narrowest maintainable fix. The optional `recordInboundSession` ordering hardening should be added only with a separate focused test if it proves necessary.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/96698: Discord/shared reply-session initialization can falsely report `stale-snapshot` after a session has a blobbed `skillsSnapshot` because the initialization revision compares whole JSON-stringified entries across persisted/hydrated shapes or key order. Keep the fix at the session accessor/session storage boundary: compare a normalized persisted/durable revision shape, or otherwise make `skillsSnapshot.prompt` versus `promptRef` and prompt/key-order differences revision-stable without weakening real concurrent-write detection. Add focused regression coverage that creates or persists a prompt-first `skillsSnapshot` large enough to be blobbed, reloads through the initialization snapshot path, and verifies `commitReplySessionInitialization` does not reject solely because hydration returns the same data in a different shape/order. Consider the optional `recordInboundSession` meta-task ordering only if a focused test proves it contributes to this failure; do not bundle unrelated channel behavior. Coordinate with https://github.com/openclaw/openclaw/pull/96550 if it is still open because it changes nearby stale-snapshot retry behavior. Do not edit CHANGELOG.md.
