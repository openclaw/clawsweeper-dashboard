## Summary

Keep open: current main still compares reply-session initialization revisions by JSON-stringifying whole session entries, while the same `skillsSnapshot` can be represented as persisted `promptRef` or hydrated `prompt`. An open closing PR exists, but it has a prior blocking review, and the newer focused candidate still needs maintainer review and direct behavior proof.

## What This Changes

The issue reports Discord second inbound messages failing because reply-session revision comparison treats persisted and hydrated `skillsSnapshot` shapes as different stale snapshots.

## Best Possible Solution

Land a focused session-accessor fix that compares a canonical persisted `skillsSnapshot` shape for revision identity while preserving hydrated runtime snapshots, then close this issue after direct Discord or Telegram behavior proof on the merged head.

## Reproduction Assessment

Yes at source level, not live: current main JSON-stringifies the full reply-init session entry while storage can persist a large `skillsSnapshot.prompt` as `promptRef` and hydrate it back to `prompt`. The reporter and a later field note provide Discord observations, but I did not run the live two-message flow.

## Solution Assessment

Yes: normalizing only the revision identity at the session accessor/storage boundary is the narrow maintainable fix. `/new` should remain a workaround, and runtime snapshot entries should stay hydrated for downstream prompt assembly.
