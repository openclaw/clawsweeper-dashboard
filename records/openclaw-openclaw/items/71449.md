## Summary

Keep open. Current main already clarifies the first startup phase as HTTP server listening, but this PR still carries useful channel-summary and startup-recovery work that is not fully redundant. It should not merge as currently diffed because it weakens Telegram webhook cleanup ordering and bypasses Feishu’s startup-specific timeout configuration.

## What This Changes

The PR changes gateway startup logging, adds a sidecar/channel health summary after post-attach startup, passes channel runtime snapshots into that summary, and adjusts Telegram webhook cleanup plus Feishu startup identity probing and related tests/scripts.

## Best Possible Solution

Land a rebased or split version that keeps current main’s HTTP-listening startup semantics, adds any channel/sidecar health summary with wording that reflects handoff state, and hardens Telegram and Feishu startup without weakening webhook confirmation, Feishu timeout configurability, or typed API boundaries.

## Reproduction Assessment

Yes. The Telegram regression is reproducible at source/test level by mocking deleteWebhook failure with getWebhookInfo returning a non-empty URL; the Feishu regression follows by setting OPENCLAW_FEISHU_STARTUP_PROBE_TIMEOUT_MS above 10000 and observing the PR’s explicit 10000ms override.

## Solution Assessment

No, not as currently diffed. The product direction is reasonable, but the narrow maintainable fix is to preserve current webhook-confirmation retry behavior and Feishu startup timeout semantics while rebasing the logging and summary work onto current main.
