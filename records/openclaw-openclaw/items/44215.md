Full review comments:

- **[P1] Keep profile-owned Windows shims append-only:** `src/infra/path-env.ts:106-116`
  - body: This adds `%LOCALAPPDATA%` pnpm, WindowsApps, and `%APPDATA%` npm to `prepend`, so they resolve before the existing PATH and trusted roots. Current main deliberately appends comparable user-writable package-manager paths to avoid binary shadowing, so merging this can change executable choice after upgrade.
  - confidence: 0.92
- **[P1] Validate Windows roots before trusting them:** `src/infra/path-env.ts:103-119`
  - body: The win32 branch reads `SystemRoot`/`WINDIR` and Program Files env values directly before prepending derived directories. Current Windows install-root helpers reject relative, UNC, and path-list values before treating roots as trusted, so this should use that contract instead of raw env strings.
  - confidence: 0.88
- **[P1] Split the simulated Windows PATH with semicolons:** `src/infra/path-env.test.ts:265`
  - body: The helper joins PATH with `;` for `platform: "win32"`, but the test splits with the host delimiter. On Linux/macOS CI, fake `C:/...` entries get split at the drive colon, so the assertion can fail for reasons unrelated to the intended behavior.
  - confidence: 0.94
- **[P2] Drop the duplicate gateway push labels:** `src/config/schema.labels.ts:83-87`
  - body: This PATH bootstrap PR also adds gateway.push labels near the auth labels, but current main already defines those keys later in `FIELD_LABELS`. The unrelated stale hunk would create duplicate object keys when rebased or merged.
