## Summary

Keep this PR open, but it is not merge-ready. Current main still lacks the requested Windows PATH bootstrap, yet the proposed patch changes command-resolution precedence in a way that conflicts with the current hardened PATH model and also carries an unrelated schema-label hunk already present on main.

## What This Changes

This PR adds a win32 PATH bootstrap branch with platform-specific PATH delimiter handling, adds a Windows PATH regression test, and adds `gateway.push.*` schema labels.

## Best Possible Solution

Add Windows bootstrap support without weakening the existing PATH hardening model: trusted runtime/system directories can be searched ahead of PATH, while user-profile and package-manager shim directories should be appended, opt-in, or handled through the documented exec configuration path. The fix should also use deterministic Windows delimiter handling in tests and leave unrelated schema labels out of the branch.

## Reproduction Assessment

Yes. Source inspection reproduces the remaining main-branch gap because there is no win32 branch in `candidateBinDirs`; the PR-specific CI failure is also source-backed because the simulated Windows PATH is joined with `;` while the test splits with the host delimiter on Linux/macOS.

## Solution Assessment

No. Windows PATH bootstrap support is a valid direction, but this patch is not the best fix because it promotes user-writable Windows shim directories ahead of trusted paths and mixes unrelated schema-label changes into the PATH patch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Avoid prepending user-writable Windows shims:** `src/infra/path-env.ts:108-117`
  - body: The win32 branch puts `%LOCALAPPDATA%\pnpm`, `%LOCALAPPDATA%\Microsoft\WindowsApps`, and `%APPDATA%\npm` in `prepend`, so OpenClaw subprocesses would resolve those user-profile shims before the existing PATH and trusted system dirs. Current main explicitly appends user/package-manager dirs to avoid PATH hijacking, and WindowsApps aliases can shadow common commands.
  - confidence: 0.92
- **[P1] Split simulated Windows PATH with semicolons:** `src/infra/path-env.test.ts:265`
  - body: The test forces `platform: "win32"`, and the helper joins PATH entries with `;`, but the assertion still splits `process.env.PATH` with host `path.delimiter`. On Linux/macOS CI that delimiter is `:`, so `C:/...` paths are split at drive letters and the test fails regardless of the intended Windows behavior.
  - confidence: 0.88
- **[P2] Drop the stale gateway.push label hunk:** `src/config/schema.labels.ts:80-84`
  - body: The `gateway.push.*` label additions are unrelated to this PATH change, and current main already has those labels. Keeping the stale hunk risks duplicate/conflicting entries during rebase and makes the PATH fix harder to review.
  - confidence: 0.81
