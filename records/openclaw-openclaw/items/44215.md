Keep open. The Windows PATH gap is still real on current main, but this PR needs rework because it prepends user-profile Windows shim directories ahead of existing/system PATH entries, has a simulated Windows delimiter test bug, carries unrelated duplicate config labels, and lacks real Windows behavior proof.

## What This Changes

## Best Possible Solution

Add Windows PATH bootstrap within the established hardening model: trusted system/runtime roots may precede PATH, profile/package-manager dirs should not shadow existing or system entries, tests should parse win32 PATH with semicolons, and unrelated schema-label churn should be dropped.
