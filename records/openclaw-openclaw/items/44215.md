## Summary

Keep this PR open. Current main still lacks Windows-specific PATH bootstrap support, so the contribution is not obsolete, but the submitted patch conflicts with the existing hardened PATH model, has a deterministic non-Windows CI test failure, and carries a stale unrelated schema-label hunk that needs maintainer-guided rework before merge.

## What This Changes

The PR adds a win32 PATH bootstrap branch and platform-specific delimiter handling in `src/infra/path-env.ts`, adds a Windows PATH regression test, and includes a `gateway.push.*` schema-label hunk.

## Best Possible Solution

Preserve the existing hardened PATH model while adding a narrow Windows bootstrap: prepend only trusted system/runtime directories, append or make opt-in any user-writable package-manager and WindowsApps locations, handle semicolon delimiters deterministically, and cover the behavior with focused tests. Keep `env.vars.PATH` governed by the existing rejected-override and `tools.exec.pathPrepend` policy.

## Reproduction Assessment

Yes. The proposed test failure is reproducible by running the new win32 simulation on a non-Windows runner, where the helper joins with `;` but the assertion splits with host `path.delimiter`; the user-writable precedence issue follows directly from `mergePath` ordering and the PR's `prepend` additions.

## Solution Assessment

No. Windows PATH bootstrap support is a reasonable direction, but this patch is not the safest implementation because it promotes user-writable shims ahead of trusted dirs and mixes unrelated schema-label changes into the branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Avoid prepending user-writable Windows shims:** `src/infra/path-env.ts:108-117`
  - body: The win32 branch puts `%LOCALAPPDATA%\pnpm`, `%LOCALAPPDATA%\Microsoft\WindowsApps`, and `%APPDATA%\npm` in `prepend`, so OpenClaw subprocesses resolve those user-writable shims before the existing PATH and system dirs. That conflicts with the current hardened model that keeps package-manager/user dirs from shadowing trusted binaries, and WindowsApps aliases can intercept common commands like `python`, `node`, or `gh`.
  - confidence: 0.92
- **[P1] Split Windows PATH assertions with semicolons:** `src/infra/path-env.test.ts:265`
  - body: The new test forces `platform: "win32"`, and the helper now joins PATH entries with `;`, but the assertion still splits `process.env.PATH` with host `path.delimiter`. On Linux/macOS CI that delimiter is `:`, so paths like `C:/...` are split at drive letters and the test fails regardless of the intended Windows behavior.
  - confidence: 0.88
- **[P2] Drop the stale gateway.push label hunk:** `src/config/schema.labels.ts:80-84`
  - body: The `gateway.push.*` label additions are unrelated to this PATH change and current main already has those schema labels. Keeping this stale hunk risks duplicate/conflicting label entries during rebase and makes the PATH fix harder to review.
  - confidence: 0.81
