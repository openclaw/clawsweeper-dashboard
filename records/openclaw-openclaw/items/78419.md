## Summary

Keep open: current main records terminal WhatsApp closes as `logged-out` or `conflict`, but the generic channel recovery and health-monitor paths still restart any non-manually stopped account with `running=false`, so the reported stopped-loop remains source-reproducible after the earlier stale-socket fixes.

## What This Changes

The issue asks OpenClaw to stop auto-restarting WhatsApp accounts after terminal Baileys disconnects, surface reauth-required state, and bound any remaining restart loops.

## Best Possible Solution

Teach the channel restart policy to honor terminal, human-action-required WhatsApp states before generic stopped-channel recovery, then decide whether the existing `healthState` surface is enough or a new reauth event/status field is warranted.

## Reproduction Assessment

Yes, by source inspection: force a WhatsApp 401/440 close, let the monitor finish with `running=false` and terminal `healthState`, then the generic recovery/health-monitor code treats the account as stopped and eligible for restart. I did not live-run the multi-tenant heap-growth scenario.

## Solution Assessment

Unclear for the full requested shape: the narrow maintainable fix is to suppress automatic restarts for explicit terminal states, while `reauthRequired`, `channel.reauth_required`, and new retry-budget knobs need maintainer API/product scoping.
