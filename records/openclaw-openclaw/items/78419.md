## Label Justifications

- `P1`: The report describes a gateway-level restart loop that breaks real multi-tenant WhatsApp channel and RPC workflows until operator intervention.
- `impact:crash-loop`: The issue centers on repeated automatic channel restarts that can exhaust memory and saturate gateway availability.
- `impact:message-loss`: The reporter observed missed replies and degraded channel/control-plane delivery while the restart loop saturated the gateway.
- `impact:session-state`: The root path loses terminal WhatsApp session state before the generic gateway restart policy decides whether to restart.

## Summary

Keep open: current main still restarts non-manual stopped WhatsApp accounts without consuming the preserved terminal `logged-out` or `conflict` state, and the open closing PR remains the active unmerged fix candidate.

## What This Changes

The issue asks OpenClaw to stop auto-restarting WhatsApp accounts after terminal disconnects and to expose enough terminal session state for operators or control planes to re-pair them.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/78511 with a focused terminal-disconnect guard for both restart paths, then track any event/config-budget expansion separately.

## Reproduction Assessment

Yes from source inspection: terminal WhatsApp closes produce preserved `logged-out` or `conflict` plugin health state, but current main still restarts any non-manual `running=false` account through generic gateway restart paths. I did not run a live WhatsApp unlink.

## Solution Assessment

Yes for the primary bug: carrying terminal disconnect state into the existing generic health/restart policy is the narrowest maintainable fix. The proposed new gateway event, top-level status field, and restart-budget config are safer as separate follow-up product decisions.
