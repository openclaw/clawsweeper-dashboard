## Summary

This should stay open: current main has useful partial mitigations for WhatsApp 440/401 terminal closes and restart caps, but the gateway health monitor still does not recognize terminal WhatsApp stopped states before attempting an automatic restart.

## What This Changes

Prevent WhatsApp accounts stopped by terminal Baileys disconnect reasons from entering health-monitor restart loops, and expose operator-visible reauth state.

## Best Possible Solution

Teach the gateway health monitor to skip automatic restarts for terminal WhatsApp states, then decide whether existing healthState is sufficient or a new reauthRequired event/status surface should be added.

## Reproduction Assessment

Yes, source-reproducible: terminal WhatsApp closes produce healthState values such as conflict or logged-out and then running=false, while the generic health policy treats that as not-running and the monitor calls startChannel(). I did not run a live 12-account WhatsApp setup.

## Solution Assessment

No, current main is only a partial solution: clearing auth and capping restarts reduce the expensive socket loop, but terminal stopped states still flow through the generic restart path. The safer fix is a terminal-state guard with focused gateway/WhatsApp regression coverage before adding any new public surface.
