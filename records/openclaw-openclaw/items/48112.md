## Summary

Keep open. Current main still uses the older text-weight fallback in the active memory-core implementation, and this PR is useful but stale against the current layout because it edits deleted `src/memory` files rather than `extensions/memory-core`. The proposed fallback also needs one remaining recall decision because it currently keeps only the top keyword score group when all strict results are below `minScore`.

## What This Changes

The PR changes the old `src/memory` manager and tests so hybrid memory search fallback derives keyword retention from post-merge keyword-backed scores after decay/MMR, with regressions for BM25 and temporal-decay cases.

## Best Possible Solution

Port the fix into `extensions/memory-core/src/memory/manager.ts`, add current-layout regressions for BM25 `textScore < 1` and temporal-decay fallback behavior, and define the fallback so keyword-backed candidates remain bounded by scoring while still honoring `maxResults` instead of collapsing to only the top score group.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with default `minScore=0.35` and `textWeight=0.3`, keyword-only merged hits can be below strict filtering, and temporal decay can lower post-merge `entry.score` below the current text-weight fallback floor.

## Solution Assessment

No. The intended fix direction is right, but this PR is not the best current patch because it edits the obsolete `src/memory` layout and the new threshold can drop all but the top keyword fallback score group; the safer path is a current memory-core port with focused tests.
