## Summary

Keep open. The underlying memory-search bug still exists on current main, but this PR is not mergeable as-is because it targets the obsolete `src/memory` layout and the revised threshold keeps only the top keyword fallback score group instead of honoring `maxResults` for fallback candidates.

## What This Changes

The PR modifies the old `src/memory` manager and tests so hybrid memory search fallback derives keyword retention from post-merge keyword-backed scores after decay/MMR, with BM25 and temporal-decay regression tests.

## Best Possible Solution

Port the fix into `extensions/memory-core/src/memory/manager.ts`, add current-layout regressions for BM25 `textScore < 1` and temporal-decay fallback behavior, and define fallback selection so keyword-backed candidates remain score-bounded while still returning up to `maxResults` instead of only the top score group.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with default `minScore=0.35` and `textWeight=0.3`, keyword-only merged hits can be below strict filtering, and temporal decay can lower post-merge `entry.score` below the current text-weight fallback floor.

## Solution Assessment

No. The intended direction is valid, but this PR is not the best current patch because it edits the old `src/memory` layout and the `Math.max(...scores)` relaxed floor collapses fallback results to only the highest score or ties.
