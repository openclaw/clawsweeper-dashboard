## Summary

Keep open: current main still documents `sessionKey="current"`, the unrestricted resolver still probes `sessions.resolve` with raw `current`, and the gateway still returns a failed `INVALID_REQUEST` response before wrapper fallback; the linked fix attempt at https://github.com/openclaw/openclaw/pull/78449 was closed unmerged because it did not remove the failed WS response and regressed literal `current` compatibility.

## What This Changes

Report that `sessionKey="current"` tool calls trigger a failed `sessions.resolve` WebSocket lookup before wrapper fallback succeeds.

## Best Possible Solution

Resolve semantic `current` before producing a failed WS resolver response, while preserving literal `current` key/sessionId compatibility and adding regression coverage that proves the failed probe is gone.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: unrestricted `sessionKey="current"` resolution can call `sessions.resolve` with `key: "current"`, and the gateway turns a missing literal key into an `INVALID_REQUEST` response. I did not run a live gateway log reproduction in this read-only review.

## Solution Assessment

Yes, the narrow maintainable direction is wrapper-side or context-aware alias resolution that avoids the failed WS response. A global gateway rejection is not sufficient because it still fails the request and conflicts with literal `current` compatibility tests.
