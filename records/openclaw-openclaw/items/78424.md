## Summary

Keep open: current main still has a source-proven log-noise bug where documented `sessionKey="current"` can trigger a failed `sessions.resolve` WS response before wrapper fallback succeeds, and the only linked fix attempt was closed unmerged because it preserved the failed response shape/regressed literal `current` compatibility.

## What This Changes

Make `sessionKey="current"` resolve consistently without emitting a failed `sessions.resolve` WS response for legitimate current-session tool calls.

## Best Possible Solution

Resolve semantic `current` before producing a failed WS resolver response, while preserving literal `current` key/sessionId compatibility and adding regression coverage that proves the failed probe is gone.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: unrestricted `sessionKey="current"` calls can send `sessions.resolve` with `key: "current"`, and the gateway returns `INVALID_REQUEST` when no literal session exists, matching the reporter's logs.

## Solution Assessment

Yes, the maintainable direction is a narrow wrapper-side or context-aware alias resolution fix. The closed gateway-only rejection approach is not sufficient because it still fails the WS request and conflicts with literal `current` compatibility.
