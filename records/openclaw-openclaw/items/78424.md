## Summary

Keep open: current main still documents `sessionKey="current"` for the caller's current session, but the gateway `sessions.resolve` RPC still receives only raw resolve params and treats `key: "current"` as a literal store key, so the failed `No session found: current` probe remains unsolved at the WS layer.

## What This Changes

The issue requests that gateway `sessions.resolve` handle the semantic `current` session alias consistently with the higher-level session tools instead of logging failed raw `current` lookups before fallback.

## Best Possible Solution

Add a narrow gateway-owned semantic-current resolution path for `sessions.resolve` that uses the calling connection's current session binding after literal `current` key/sessionId precedence is preserved, with regression tests covering direct RPC success and no failed probe.

## Reproduction Assessment

Yes at source level: current main still routes direct `sessions.resolve({ key: "current" })` through a literal key lookup with no requester session binding. I did not execute a live WebSocket reproduction in this read-only review.

## Solution Assessment

Yes: the best fix is a bounded gateway resolver/context change, not another wrapper-only fallback, because the observable failure is the gateway RPC returning and logging `INVALID_REQUEST` before wrappers recover.
