## Summary

Keep open: current main still advertises `sessionKey="current"`, the shared resolver still sends raw `current` through `sessions.resolve` on unrestricted tool calls, and the gateway still turns a missing literal key into a failed `INVALID_REQUEST` response before wrapper fallback. The related fix attempt at https://github.com/openclaw/openclaw/pull/78449 was closed unmerged because it still returned a failed response and would have regressed literal `current` compatibility.

## What This Changes

The issue requests that documented `sessionKey="current"` tool calls stop producing failed `sessions.resolve` gateway log entries.

## Best Possible Solution

Resolve semantic `current` before a failed WS resolver response while preserving literal `current` key/sessionId precedence and covering status, history, and send tool paths.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: unrestricted `sessionKey="current"` calls `sessions.resolve` with `key: "current"`, and the gateway returns a failed `INVALID_REQUEST` when no literal key exists.

## Solution Assessment

Yes. The maintainable fix is wrapper-side or context-aware alias resolution that avoids the failed WS response; a global gateway rejection is not sufficient because it still fails and conflicts with literal `current` compatibility.
