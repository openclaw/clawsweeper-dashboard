## Summary

The report is valid on current `main`: the documented `sessionKey="current"` path can still make a failing `sessions.resolve` probe before the wrapper falls back, so this should stay open for a narrow log-noise fix.

## What This Changes

The issue asks for `sessions.resolve` and the session tools to handle the documented `sessionKey="current"` alias without emitting an `INVALID_REQUEST` gateway log.

## Best Possible Solution

Keep the documented `current` alias, remove the failed resolver roundtrip/log line, and preserve existing literal `current` key/sessionId compatibility and visibility checks.

## Reproduction Assessment

Yes from source inspection: `session_status` and related wrappers can route `sessionKey="current"` through `sessions.resolve`, and the gateway returns `No session found: current` when no literal row/id exists. I did not establish a live journalctl reproduction in this read-only review.

## Solution Assessment

Yes, a narrow resolver/wrapper change is the right direction if it preserves literal `current` lookup semantics and does not broaden visibility. A simple unconditional rewrite of `current` would be less safe because current tests explicitly cover literal-key compatibility.
