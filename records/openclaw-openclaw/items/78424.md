## Summary

Keep open: current main still documents `sessionKey="current"`, the shared resolver still probes `sessions.resolve` with raw `current` on unrestricted calls, and the gateway still turns a missing literal key into a failed `INVALID_REQUEST` response before wrapper fallback. The related fix attempt at https://github.com/openclaw/openclaw/pull/78449 was closed unmerged because it still produced a failed response and would have regressed literal `current` compatibility.

## What This Changes

The issue asks to stop legitimate `sessionKey="current"` tool calls from producing failed `sessions.resolve` gateway log entries while preserving successful wrapper behavior.

## Best Possible Solution

Resolve semantic `current` before producing a failed WS resolver response, while preserving literal `current` key/sessionId compatibility and adding regression coverage that proves the failed probe is gone.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: unrestricted `sessionKey="current"` resolution calls `sessions.resolve` with `key: "current"`, and the gateway turns a missing literal key into an `INVALID_REQUEST` response before fallback.

## Solution Assessment

Yes. The narrow maintainable direction is wrapper-side or context-aware alias resolution that avoids the failed WS response; a global gateway rejection is not sufficient because it still fails the request and conflicts with literal `current` compatibility tests.
