## Summary

Keep open: current main still advertises `sessionKey="current"`, the unrestricted tool resolver still probes `sessions.resolve` with the raw `current` key, and the gateway still converts a missing literal key into a failed WS response before fallback. The related fix PR was closed unmerged because it still returned a failed response and risked breaking literal `current` session compatibility.

## What This Changes

The issue asks OpenClaw to stop logging failed `sessions.resolve` calls when agents use the documented `sessionKey="current"` alias.

## Best Possible Solution

Resolve semantic `current` before any failed WS response is produced, while preserving literal `current` key/sessionId precedence and adding focused regression coverage for the no-log-noise path.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: unrestricted `sessionKey="current"` resolution calls `sessions.resolve` with `key: "current"`, and the gateway returns a failed response when no literal key exists. I did not run a live gateway log repro in this read-only review.

## Solution Assessment

Yes, the best direction is a wrapper-side or context-aware gateway alias resolution that avoids the failed WS response. The closed PR's global rejection approach is not the best fix because it still fails the request and conflicts with literal `current` compatibility.
