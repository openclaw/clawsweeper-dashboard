## Summary

Keep open: current main still sends the documented `sessionKey="current"` alias through a failing `sessions.resolve` lookup before wrapper fallback, so the reported gateway log noise remains a valid narrow bug.

## What This Changes

The issue asks for `sessionKey="current"` to resolve without a failed `sessions.resolve` roundtrip or noisy `INVALID_REQUEST` gateway log.

## Best Possible Solution

Remove the failed resolver roundtrip for the documented alias while preserving literal `current` key/sessionId compatibility and spawned-session visibility rules.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call a session tool with `sessionKey="current"` when no literal `current` session exists, and the wrapper first sends a failing `sessions.resolve` request before falling back successfully.

## Solution Assessment

Yes, the best fix is a narrow wrapper-side or context-aware gateway resolver change that avoids the failed WS response. The closed linked PR's global rejection is not the best approach because it still fails the WS call and conflicts with literal `current` compatibility.
