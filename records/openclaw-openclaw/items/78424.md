## Summary

Keep open: current main still lets the documented `sessionKey="current"` path make a failing `sessions.resolve` probe before wrapper fallback, so the reported log noise remains a valid narrow bug.

## What This Changes

The issue asks for `sessions.resolve` and the session tool wrappers to handle `sessionKey="current"` without emitting an `INVALID_REQUEST` log line.

## Best Possible Solution

Keep the documented `current` alias, remove the failed resolver roundtrip/log line, and preserve existing literal `current` key/sessionId compatibility plus spawned visibility rules.

## Reproduction Assessment

Yes from source inspection: the wrappers can route raw `current` through `sessions.resolve`, and the gateway returns `No session found: current` when no literal row/id exists. I did not run a live journalctl reproduction in this read-only review.

## Solution Assessment

Yes, a narrow resolver or wrapper change is the best path if it avoids the failed probe while preserving literal `current` lookup semantics. An unconditional gateway alias rewrite would be less safe because compatibility tests cover literal `current` keys and sessionIds.
