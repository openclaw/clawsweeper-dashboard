## Summary

The report is valid on current main: the documented `sessionKey="current"` alias can still route through a failing `sessions.resolve` probe before fallback, producing the reported log noise.

## What This Changes

The issue requests that `sessionKey="current"` resolve cleanly for session tools or at the gateway resolver instead of emitting an `INVALID_REQUEST` probe first.

## Best Possible Solution

Keep the documented `current` alias, remove the failed resolver roundtrip/log line, and preserve literal `current` key/sessionId behavior plus spawned-session visibility checks.

## Reproduction Assessment

Yes from source inspection: `session_status` and related wrappers can route `sessionKey="current"` through `sessions.resolve`, and the gateway returns `No session found: current` when no literal row/id exists. I did not establish a live journalctl reproduction in this read-only review.

## Solution Assessment

Yes, a narrow resolver or wrapper change is the best direction if it removes the failed probe while preserving literal `current` lookup semantics and visibility checks. A broad unconditional alias rewrite would be riskier because current tests intentionally cover literal-key compatibility.
