## Summary

Keep open: current `main` still advertises `sessionKey="current"`, `session_status` can still send a raw `sessions.resolve({key:"current"})` probe before falling back, and the gateway resolver still turns a missing literal `current` key into `INVALID_REQUEST`. The merged channel-plugin fix for https://github.com/openclaw/openclaw/pull/72306 addressed tool success for one path but did not remove this gateway failure/log-noise behavior.

## What This Changes

The issue asks for `sessions.resolve` or its callers to handle the semantic `sessionKey="current"` alias without producing an `INVALID_REQUEST` failure before fallback.

## Best Possible Solution

Resolve semantic `current` before producing a failed WS resolver response, while preserving literal `current` key/sessionId compatibility and adding focused proof that the failed `sessions.resolve` log line is gone.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: call an unrestricted `session_status({sessionKey:"current"})` path that does not rewrite through `runSessionKey` first, and the wrapper can probe `sessions.resolve({key:"current"})` before falling back while the gateway returns `No session found: current`. I did not run a live gateway in this read-only review.

## Solution Assessment

Yes, the maintainable fix is still narrow: avoid the failed raw-current resolver response by resolving semantic current with requester context at the wrapper or context-aware gateway boundary, while keeping literal `current` key/sessionId precedence intact.
