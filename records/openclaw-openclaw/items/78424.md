## Summary

Keep open: current main still has the documented `sessionKey="current"` path making a failing `sessions.resolve` probe before wrapper fallback for unrestricted session tools, and the related fix PR at https://github.com/openclaw/openclaw/pull/78449 was closed unmerged after review found it would still return a failed WS response and regress literal `current` lookups.

## What This Changes

The issue reports that the documented `sessionKey="current"` alias causes a failed `sessions.resolve` gateway roundtrip and log line before tool-layer fallback succeeds.

## Best Possible Solution

Resolve semantic `current` before producing a failed WS resolver response, while preserving literal `current` key/sessionId compatibility and adding regression coverage that proves the log-producing probe is gone.

## Reproduction Assessment

Yes, from source inspection: unrestricted wrapper resolution can still call `sessions.resolve` with `key: "current"`, and the gateway responds with `INVALID_REQUEST` when no literal key exists. I did not run a live journalctl reproduction in this read-only review.

## Solution Assessment

Yes for the requested direction: a wrapper-side or context-aware resolver change is the narrowest maintainable fix. A global gateway rejection is not the best solution because it still emits a failed WS response and conflicts with existing literal `current` compatibility tests.
