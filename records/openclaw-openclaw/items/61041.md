## Summary

Keep open. Current main and the latest v2026.5.19 release still reject oversized shared gateway image attachments before any server-side resize/recompress path runs; the iOS client-side image processor is useful related work but does not cover the shared gateway parser path reported here.

## What This Changes

The issue requests automatic resize/recompression for images over the 5 MB chat attachment limit instead of rejecting them before send.

## Best Possible Solution

Add a bounded server-side shrink/recompress step in shared gateway attachment parsing before the image byte rejection, backed by parser regression tests and existing image-ops safety guards; leave configurable image-limit policy to https://github.com/openclaw/openclaw/issues/67031.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an image attachment above MAX_IMAGE_BYTES but below the general chat attachment ceiling reaches parseMessageWithAttachments and is rejected before offload or resizing.

## Solution Assessment

Yes for the focused repair direction, but not by copying the direct sharp snippet. The maintainable path is parser-side resize/recompress using the existing media image-ops safety boundary while keeping optional configurability separate.
