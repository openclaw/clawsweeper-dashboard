## Summary

Keep open. Current `main` and v2026.5.19 still reject shared gateway image attachments over `MAX_IMAGE_BYTES` before any server-side resize/recompress path can run; the merged iOS work only preprocesses iOS PhotosPicker attachments client-side.

## What This Changes

Request automatic resizing or compression of oversized image attachments instead of rejecting them at the gateway image byte limit.

## Best Possible Solution

Add a bounded server-side shrink/recompress step in shared gateway attachment parsing before the image byte rejection, backed by parser regression tests and existing image-ops safety guards.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an image attachment above `MAX_IMAGE_BYTES` but below the overall chat attachment ceiling reaches `parseMessageWithAttachments` and is rejected before offload or resizing.

## Solution Assessment

Yes for the focused repair direction. Parser-side resize/recompress is maintainable if it reuses the existing image-ops boundary; optional image-limit configurability should stay with the broader media-limit issue.
