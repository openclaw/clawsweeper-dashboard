## Summary

Keep open. Current main still rejects oversized direct chat image attachments in the shared gateway parser before any server-side resize path can run; the landed iOS work only fixes PhotosPicker client-side preprocessing, and broader media limit configurability remains tracked separately.

## What This Changes

Request auto-resizing or recompressing chat image attachments that exceed the 5 MB image cap instead of rejecting them.

## Best Possible Solution

Add a bounded server-side shrink/recompress step in shared gateway attachment parsing before the image byte rejection, with regression coverage and no new config surface; leave configurable limits to the broader media-limit issue.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an image attachment above MAX_IMAGE_BYTES but below the general chat attachment ceiling reaches parseMessageWithAttachments and is rejected before offload or resizing.

## Solution Assessment

Yes for the focused repair direction. A bounded parser-side resize/recompress step is the narrow maintainable fix for this issue, while the iOS client-side fix and broader media-limit configurability should remain separate.
