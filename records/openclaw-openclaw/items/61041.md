## Summary

Keep open. Current main and the latest release still reject oversized direct chat image attachments in the shared gateway parser before downstream sanitizer resizing can run; the related iOS PR only covers client-side composer preflight, and broader media-limit configurability remains a separate design track.

## What This Changes

Request auto-resizing or recompressing chat image attachments that exceed the gateway image byte limit instead of rejecting them.

## Best Possible Solution

Add a bounded server-side shrink/recompress step in the shared gateway attachment parser before the hard image-byte rejection, reusing existing image-ops/tool-image primitives and failing explicitly when an image still cannot fit safely.

## Reproduction Assessment

Yes. Source and test inspection give a high-confidence path: an image attachment whose estimated decoded size exceeds MAX_IMAGE_BYTES but remains under the general chat attachment ceiling reaches parseMessageWithAttachments and is rejected before offload or resizing.

## Solution Assessment

Yes for the focused repair direction. Server-side resizing in the shared gateway parser is the narrow maintainable fix for this issue; https://github.com/openclaw/openclaw/pull/73710 is useful iOS defense in depth, and https://github.com/openclaw/openclaw/issues/67031 should remain the broader configurability follow-up.
