## Summary

Keep open. Current main and the latest release still reject oversized shared gateway image attachments before any server-side resize/recompress path can run; the merged iOS work only preprocesses PhotosPicker attachments on the client side, and broader image-limit configurability remains separate.

## What This Changes

Request auto-resizing or recompressing image attachments that exceed the gateway image byte limit instead of rejecting them outright.

## Best Possible Solution

Add a bounded server-side shrink/recompress step in shared gateway attachment parsing before the image byte rejection, backed by parser regression tests and existing image-ops safety guards; leave configurable image limits to the separate media-limit issue.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an image attachment above MAX_IMAGE_BYTES but below the general chat attachment ceiling reaches parseMessageWithAttachments and is rejected before offload or resizing.

## Solution Assessment

Yes for the focused repair direction. Parser-side resize/recompress is maintainable if it reuses the existing image-ops safety boundary; optional configurability should stay with the broader media-limit issue.
