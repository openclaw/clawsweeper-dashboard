## Summary

Keep open. Current main still rejects oversized direct chat image attachments in the shared gateway parser before any resize/sanitization path can help, while the related iOS PR only covers client-side preflight and the broader media configurability issue remains a separate design track.

## What This Changes

Request automatic shrink/recompression of oversized image attachments in shared gateway handling instead of returning a size-limit error.

## Best Possible Solution

Add a bounded server-side shrink/recompress step in the shared gateway attachment parser before the hard image-byte rejection, reusing existing image-ops/tool-image primitives and preserving an explicit failure when an image still cannot fit safely.

## Reproduction Assessment

Yes. A valid image attachment whose estimated decoded size exceeds MAX_IMAGE_BYTES reaches parseMessageWithAttachments and current tests assert the rejection path; this was verified by source and test inspection rather than by running tests in the read-only review.
