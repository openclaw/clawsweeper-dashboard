## Summary

Keep open. Current main still rejects oversized direct chat image attachments in the shared gateway parser before the existing resize/recompression helpers can run, and the open iOS PR is adjacent client-side coverage rather than the shared server fix.

## What This Changes

The issue requests automatic resizing/recompression for large chat image attachments that currently hit the gateway image byte limit.

## Best Possible Solution

Add a bounded server-side shrink/recompress step in parseMessageWithAttachments for image payloads that exceed MAX_IMAGE_BYTES but are still within the configured chat attachment budget, reusing the existing image-ops/tool-image primitives and keeping explicit rejection when an image cannot be made safe.

## Reproduction Assessment

Yes. A valid base64 image attachment larger than MAX_IMAGE_BYTES reaches parseMessageWithAttachments and the current test suite asserts that it rejects before saveMediaBuffer is called.

## Solution Assessment

Yes for the repair direction. The narrow maintainable fix is shared gateway resizing before the image-byte rejection; #73710 remains client-side defense in depth and #67031 remains the broader configurability follow-up.
