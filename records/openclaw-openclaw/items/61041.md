## Summary

Keep open. Current main still rejects oversized direct gateway image attachments at the shared parser image cap before any server-side resize path can run; the landed iOS work fixes PhotosPicker client-side preprocessing only, while broader image-limit configurability remains tracked separately.

## What This Changes

Request server-side auto-resize or recompression for image attachments that exceed the current image byte limit instead of rejecting them.

## Best Possible Solution

Add a bounded server-side shrink/recompress step in shared gateway attachment parsing before the image byte rejection, backed by parser regression tests and existing image-ops safety guards; keep configurable image limits in the separate media-limit issue.

## Reproduction Assessment

Yes. Source inspection and current tests give a high-confidence path: an image attachment above MAX_IMAGE_BYTES but below the general chat attachment ceiling reaches parseMessageWithAttachments and is rejected before offload or resizing.

## Solution Assessment

Yes for the focused repair direction. Parser-side resize/recompress is maintainable if it reuses the existing image-ops safety boundary; the optional config surface should stay with the broader media-limit issue.
