## Summary

Keep open. Current main still rejects oversized direct chat image attachments in the shared gateway parser before any resize path can run; existing image resizing is limited to later provider/tool image blocks or unrelated outgoing managed images, and the open iOS PR only covers a client-side path.

## What This Changes

The issue requests automatic resizing/compression for image attachments over the 5 MB limit instead of rejecting them before send.

## Best Possible Solution

Add a server-side shrink/recompress step in the shared chat attachment parser before the hard image-byte rejection, reusing the existing image-ops/tool-image sanitization primitives and preserving explicit failure when an image still cannot fit safely. Keep the broader configurability work in #67031 and the iOS preflight/privacy work in #73710 separate.

## Reproduction Assessment

Yes. A valid image attachment whose decoded size is above MAX_IMAGE_BYTES reaches parseMessageWithAttachments and currently throws; the existing chat attachment test directly asserts that rejection path.

## Solution Assessment

Yes for the repair direction. The narrow maintainable fix is server-side resizing in the shared gateway parser before rejection; client-side resizing is useful defense in depth but does not solve all direct chat/RPC attachment inputs.
