## Summary

Keep open. Current main still rejects oversized direct chat image attachments in the shared gateway parser before the existing resize/sanitization pipeline can help, and the related open iOS PR only covers client-side preflight rather than the shared server path.

## What This Changes

The issue requests automatic resizing/compression for image attachments over the image byte limit instead of rejecting them before send.

## Best Possible Solution

Add a server-side shrink/recompress step in the shared chat attachment parser before the hard image-byte rejection, reusing existing image-ops/tool-image primitives and preserving explicit failure when an image still cannot fit safely.

## Reproduction Assessment

Yes. A valid image attachment whose decoded size exceeds MAX_IMAGE_BYTES reaches parseMessageWithAttachments and currently throws; the existing chat attachment test directly asserts that rejection path.

## Solution Assessment

Yes for the repair direction. The narrow maintainable fix is server-side resizing in the shared gateway parser; #73710 is useful client-side defense in depth, and #67031 should remain the broader configurability follow-up.
