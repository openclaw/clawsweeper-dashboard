## Summary

Current main already has the Control UI model picker calling `sessions.patch` and now guards `chat.send` behind the in-flight model switch, so the observable previous-model behavior is fixed on main. The fix is current-main-only for now because it landed after the latest v2026.5.7 release.

## What This Changes

The issue reports that changing the Control UI chat header model picker updates the dropdown but does not patch the active session, leaving the agent on the previous model.

## Best Possible Solution

Keep the current-main picker patch path and pending-send guard, then let it ship in the next release after v2026.5.7.

## Reproduction Assessment

Yes for the reported v2026.5.7 user flow: switch the Control UI model picker and immediately send a message. Against current main, source and regression tests show the picker emits `sessions.patch` and chat send waits for that patch, so the failure should no longer reproduce there.

## Solution Assessment

Yes. Solving this at the Control UI boundary is the narrow maintainable fix: emit the existing `sessions.patch` RPC, track the pending switch, and block `chat.send` until the active session override is saved.
