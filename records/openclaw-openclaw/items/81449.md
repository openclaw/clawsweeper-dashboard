## Summary

Current main already implements the documented behavior: the chat header model selector sends `sessions.patch` with the active session key and selected model, and regression coverage asserts that RPC. The proof commit is only on `main` in this checkout, with no containing release tag found, so this should close as implemented on current main rather than become a repair job.

## What This Changes

The issue reports that selecting a different model in the Control UI header updates the visible picker but does not send a `sessions.patch` RPC, leaving the session on the previous model.

## Best Possible Solution

Keep the current main implementation and close this report; if the same symptom reproduces after a release containing `b17e77a22bf43ea4b994239e91ff2589a5534140`, it should be reopened or refiled with a Gateway RPC/event trace.

## Reproduction Assessment

No; I did not establish a failing current-main reproduction. Current source and regression tests show the reported picker action now sends `sessions.patch`, while the issue evidence appears to describe the older `2026.5.7` release behavior.

## Solution Assessment

Yes; the current implementation is the narrow maintainable fix because the header select directly patches the active session, rolls back on failure, refreshes session/tool state, and waits before send when the patch is still pending.
