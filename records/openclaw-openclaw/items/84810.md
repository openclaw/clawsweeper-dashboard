The PR changes plugin public-surface dirName normalization to replace colons with dashes and adds a regression test for Telegram forum topic IDs.

## Best Possible Solution

Keep `normalizeBundledPluginDirName` strict and fix the specific caller path so scoped conversation IDs never become plugin directory selectors, with regression coverage around the active-memory/channel-contract flow and redacted live Telegram topic proof.

## Reproduction Assessment

Partially yes: source inspection confirms raw topic IDs still contain colons and the current loader rejects them if they reach dirName normalization. I did not establish a full current-main active-memory crash path because current active-memory already has a prior guard for this class of scoped channel IDs.

## Solution Assessment

No: replacing colons inside the shared loader normalizer is broader than the bug and can alias invalid IDs to real bundled plugin directories. The safer fix is at the caller/channel-contract boundary that is passing conversation IDs as plugin directory names.
