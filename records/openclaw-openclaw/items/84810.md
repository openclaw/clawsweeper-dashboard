## Summary

Keep open: the reported Active Memory/Telegram topic crash is credible, but this PR fixes it at the wrong boundary by weakening shared bundled-plugin dirName validation and creating colon-to-hyphen plugin-id aliasing. The supplied proof exercises a helper and unit test, not a real OpenClaw Telegram topic run.

## What This Changes

The PR changes `normalizeBundledPluginDirName` to replace colons with dashes and updates the public-surface runtime tests for colon-containing inputs.

## Best Possible Solution

Keep bundled public-surface dirName validation strict and fix Active Memory so embedded recall uses a runnable provider/channel id plus a non-conversation-shaped subagent session key, with a Telegram forum-topic regression test and live proof.

## Reproduction Assessment

Yes. Source inspection plus the linked issue logs show a Telegram forum-topic parent session key can be embedded into Active Memory's subagent session key and then parsed back into a colon-bearing hook `channelId`; I did not run the live Telegram path because this review is read-only.

## Solution Assessment

No. Sanitizing colons in the shared bundled plugin loader is not the best fix; the safer fix is to keep the loader strict and prevent scoped conversation ids from being used as plugin/channel dirNames in the Active Memory embedded-run path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep invalid dirNames from aliasing plugin ids:** `src/plugins/public-surface-runtime.ts:46`
  - body: Replacing `:` with `-` here makes the shared public-surface loader treat invalid ids as different plugin directories, e.g. `nextcloud:talk` can resolve as `nextcloud-talk`. This loader backs channel public artifacts, so scoped conversation ids should be rejected or handled before calling it; keep this validation strict and fix the Active Memory session/channel mapping instead.
  - confidence: 0.89
