## Summary

Keep open: current main still has the 10s gateway send timeout path and WhatsApp still lacks the existing unknown-send reconciliation hook, so the reported false-negative delivery status is not fixed or superseded.

## What This Changes

The issue asks OpenClaw to avoid permanent unknown outcomes when a WhatsApp send times out at the gateway ack layer but later succeeds after channel reconnect or start-account recovery.

## Best Possible Solution

Keep this issue as the WhatsApp send false-negative tracker and implement adapter-level unknown-send reconciliation or explicit queued/ack semantics on top of the existing durable message lifecycle, coordinated with the adjacent start-account and transport-stall work.

## Reproduction Assessment

No, not as a live end-to-end reproduction on current main. The source path is clear: message sends still use a 10s gateway deadline, server work can continue after the client times out, and WhatsApp lacks the reconciliation hook that would resolve a late successful send.

## Solution Assessment

Unclear: increasing the 10s timeout alone is not the best durable fix for multi-minute reconnects. A safer solution is to preserve idempotency and reconcile unknown-after-send WhatsApp outcomes, while separately reducing the start-account stall that exposes the timeout.
