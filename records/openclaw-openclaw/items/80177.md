## Summary

Keep open: current main still has the 10s gateway send timeout and the WhatsApp adapter does not declare or implement unknown-send reconciliation, so the reported delivered-after-timeout false-negative is not fixed or superseded.

## What This Changes

The issue reports false-negative WhatsApp dispatch status when a 10s gateway WebSocket ack timeout expires while the underlying send later succeeds.

## Best Possible Solution

Keep this as the WhatsApp dispatch false-negative tracker and implement a deliberate send-status contract, preferably WhatsApp unknown-send reconciliation or explicit queued/ack semantics coordinated with the start-account stall work.

## Reproduction Assessment

No high-confidence live reproduction was run on current main. The source path is clear: message sends still use a 10s gateway deadline, server-side send work can continue after the caller times out, and WhatsApp lacks the existing unknown-send reconciliation hook.

## Solution Assessment

Unclear: a longer timeout alone is not the best durable fix for multi-minute reconnects. The safer direction is to preserve idempotency and reconcile unknown-after-send WhatsApp outcomes, with any queued/ack status contract chosen deliberately.
