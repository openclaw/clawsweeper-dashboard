## Summary

Keep open: current main still has the 10s gateway message-send timeout path and WhatsApp still lacks the existing unknown-send reconciliation hook, so the reported false-negative dispatch status is not fixed or superseded.

## What This Changes

The issue reports a WhatsApp dispatch false-negative where a 10s gateway acknowledgement timeout returns an unknown outcome even though the WhatsApp message is later delivered.

## Best Possible Solution

Keep this as the WhatsApp dispatch false-negative tracker and implement a durable send-status contract, preferably adapter-level unknown-send reconciliation or explicit queued/ack semantics, coordinated with the start-account stall work.

## Reproduction Assessment

No, not as a high-confidence live reproduction on current main. The source path is clear: message sends still use a 10s gateway deadline, server-side send work can continue after the caller times out, and WhatsApp has no reconciliation hook for late successful sends.

## Solution Assessment

Unclear: a longer timeout alone is not the best durable fix for multi-minute reconnects. The safer direction is to preserve idempotency and reconcile unknown-after-send WhatsApp outcomes, with any queued/ack status contract chosen deliberately.
