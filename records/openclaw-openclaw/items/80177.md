## Summary

Keep open: current main still has the 10s gateway message-send timeout path, and WhatsApp still does not declare or implement unknown-send reconciliation, so the reported permanent false-negative dispatch status is not fixed or superseded.

## What This Changes

The issue asks OpenClaw to avoid permanent unknown WhatsApp dispatch outcomes when the gateway RPC times out while the underlying send later succeeds.

## Best Possible Solution

Keep this as the WhatsApp dispatch false-negative tracker and implement a durable send-status path, preferably WhatsApp unknown-send reconciliation or deliberate queued/ack semantics coordinated with the start-account stall work.

## Reproduction Assessment

No, not as a high-confidence live reproduction on current main. The source path is clear: message sends still use a 10s gateway deadline, server-side send work can complete after the caller times out, and WhatsApp lacks the reconciliation hook that would prove a late successful send.

## Solution Assessment

Unclear. A longer timeout alone is not a durable fix for multi-minute reconnects; the safer maintainable direction is an explicit send-status contract with WhatsApp unknown-send reconciliation or queued/ack semantics.
