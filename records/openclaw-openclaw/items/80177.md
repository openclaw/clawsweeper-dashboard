## Label Justifications

- `P1`: The issue describes a broken WhatsApp/gateway send workflow where users can see unknown or timed-out dispatch results despite successful platform delivery.
- `impact:message-loss`: The false-negative send status can lead operators or automation to retry already delivered WhatsApp messages or treat delivery as failed.
- `impact:session-state`: The reported send state can remain unknown even though the external WhatsApp platform has accepted and delivered the message.

## Summary

Keep open. Current main prevents blind replay of ambiguous outbound sends, but the reported false-negative still has a live source path: WhatsApp gateway sends can time out at the 10s caller boundary while the code has no queued/accepted response or WhatsApp unknown-send reconciler that can later update the caller-visible outcome.

## What This Changes

The issue reports that WhatsApp sends can return a 10s gateway timeout and permanent unknown outcome even when the platform message later delivers successfully.

## Best Possible Solution

Define one durable dispatch status contract for timed-out WhatsApp sends, preferably queued/accepted plus WhatsApp reconciliation or another explicit deferred-delivery outcome, so successful platform delivery cannot remain permanently unknown to callers.

## Reproduction Assessment

Yes, source-reproducible but not live-reproduced: current main still combines a 10s gateway caller timeout, a final-result send RPC, generic unknown-send recovery, and no WhatsApp adapter reconciler. A real-account WhatsApp stall/reconnect proof is still needed for exact timing and Baileys behavior.

## Solution Assessment

No, the current behavior is not the best way to solve the issue because it can leave callers with a permanent unknown result. The safer direction is a maintainer-chosen status/reconciliation contract rather than only raising the timeout.
