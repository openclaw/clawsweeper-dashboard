## Summary

Keep open. Current main still has a 10s default gateway-send RPC timeout and the gateway handler waits for the platform send before responding, while WhatsApp is gateway-delivered and does not declare `reconcileUnknownSend`, so the false-unknown report is source-plausible and not covered by the related open start-account issues.

## What This Changes

The issue reports that WhatsApp send-gate returns `DISPATCHED-UNKNOWN` after a 10s gateway WebSocket timeout even though the message later delivers during a WhatsApp `start-account` stall.

## Best Possible Solution

Define a gateway-delivered message-send contract that returns a safe accepted/queued outcome before client timeout or adds WhatsApp unknown-send reconciliation, while tracking the start-account stall as a separate channel reliability bug.

## Reproduction Assessment

No live reproduction was established in this read-only review. The source path is high-signal: gateway-delivered WhatsApp sends use a 10s default RPC timeout while the server waits for platform delivery before responding.

## Solution Assessment

Unclear as an implementation choice. A pure timeout knob is probably not the best fix; the safer direction is to align gateway send responses with the durable queue/unknown-send contract or implement WhatsApp reconciliation.
