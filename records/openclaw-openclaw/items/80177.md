## Summary

Keep open: current main still has a 10s gateway RPC deadline on message sends, and the WhatsApp durable-message adapter does not implement the existing unknown-send reconciliation hook, so the reported false-negative delivery status is not fixed or superseded.

## What This Changes

Request that WhatsApp outbound dispatch avoid a permanent `DISPATCHED-UNKNOWN` result when the gateway RPC times out before a late-successful WhatsApp send is acknowledged.

## Best Possible Solution

Keep this issue as the WhatsApp send false-negative tracker and implement adapter-level unknown-send reconciliation or queued/ack semantics on top of the existing durable message lifecycle, coordinated with the adjacent start-account and transport-stall work.

## Reproduction Assessment

No, not as a live end-to-end reproduction on current main. The source path is clear: message sends still use a 10s gateway deadline, server work can continue after the client times out, and WhatsApp lacks the reconciliation hook that would resolve a late successful send.

## Solution Assessment

Unclear: simply increasing the 10s timeout is not the best durable fix for multi-minute reconnects. A safer solution is to preserve idempotency and reconcile unknown-after-send WhatsApp outcomes, while separately reducing the start-account stall that exposes the timeout.
