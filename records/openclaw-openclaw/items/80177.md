## Summary

Keep open: current main still has the 10s gateway timeout path for message sends, and the WhatsApp message adapter still does not implement the existing unknown-send reconciliation hook needed to turn late successful sends into durable dispatched status. The related open performance PR may reduce the upstream stall, but it does not clearly resolve permanent false-negative dispatch outcomes.

## What This Changes

The issue asks OpenClaw to avoid permanent false-negative WhatsApp dispatch status when a gateway send ack times out after 10s but the underlying Baileys send later succeeds.

## Best Possible Solution

Keep this as the WhatsApp dispatch false-negative tracker and implement a durable send-status contract, preferably WhatsApp-backed unknown-send reconciliation or explicit queued/ack semantics coordinated with the start-account stall work.

## Reproduction Assessment

No, not as a high-confidence live reproduction on current main. Source inspection confirms the 10s gateway deadline, server-side delivery continuation, and missing WhatsApp reconciliation hook, but I did not run a real WhatsApp account through the stalled reconnect window.

## Solution Assessment

Unclear: increasing the timeout or adding a config knob alone is unlikely to be the best durable fix for multi-minute reconnects. The safer direction is adapter-level reconciliation or a deliberate queued-versus-ack status contract that preserves idempotency.
