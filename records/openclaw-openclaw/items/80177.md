## Summary

Keep open: current main still has the 10s message-action gateway timeout and the WhatsApp adapter still does not implement the existing unknown-send reconciliation hook. Related generic timeout work and the open async WhatsApp/registry PR do not cover the permanent false-negative status reported here.

## What This Changes

The issue asks OpenClaw to avoid permanent `DISPATCHED-UNKNOWN` outcomes for WhatsApp sends that complete after the 10s gateway ack timeout during reconnect or `start-account` stalls.

## Best Possible Solution

Keep this as the WhatsApp dispatch false-negative tracker and implement adapter-level unknown-send reconciliation or explicit queued/ack semantics coordinated with the start-account stall work.

## Reproduction Assessment

No, not as a high-confidence live reproduction on current main. Source inspection does show the relevant path: a 10s gateway deadline can expire while server-side WhatsApp send work continues, and the WhatsApp adapter has no reconciliation hook for late success.

## Solution Assessment

Unclear: lengthening the timeout alone would not address multi-minute reconnects or permanent unknown lock state. The safer direction is a deliberate send-status contract with idempotency and WhatsApp reconciliation through the existing durable-final capability surface.
