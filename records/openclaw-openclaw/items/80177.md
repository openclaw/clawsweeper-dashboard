## Summary

Keep open: current main still has the 10s outbound message gateway deadline and WhatsApp still lacks the reconciliation capability needed to turn an unknown-after-send outcome into a confirmed dispatched result, so the reported false-negative dispatch status is not fixed or superseded.

## What This Changes

The issue asks for WhatsApp sends that succeed after a gateway timeout to avoid permanent DISPATCHED-UNKNOWN outcomes through longer/back-pressure-aware acknowledgements, queued semantics, or post-send reconciliation.

## Best Possible Solution

Keep this as the WhatsApp dispatch false-negative tracker and choose a coherent send-status contract, preferably WhatsApp unknown-send reconciliation or explicit queued/ack semantics coordinated with the start-account stall work.

## Reproduction Assessment

No high-confidence live reproduction was run on current main. The source path is clear: outbound message sends still use a 10s gateway deadline, server-side send work can continue beyond the caller timeout, and WhatsApp lacks the reconciliation hook that would resolve unknown-after-send outcomes.

## Solution Assessment

Unclear: increasing the timeout or adding a knob alone would not handle multi-minute reconnects. The safer maintainable fix is a deliberate queued/ack/reconciliation contract for WhatsApp sends, coordinated with the start-account stall work.
