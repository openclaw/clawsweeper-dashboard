## Summary

Keep open: current main and v2026.5.12 still show the reported 10s message gateway call path and WhatsApp still does not implement the unknown-send reconciliation capability needed to resolve delivered-after-timeout outcomes.

## What This Changes

Requests fixing WhatsApp send status so delivered messages are not permanently reported as unknown after a 10s gateway dispatch timeout during reconnect or start-account stalls.

## Best Possible Solution

Keep this as the WhatsApp dispatch-status tracker and implement a deliberate send-status contract, preferably WhatsApp unknown-send reconciliation or explicit queued/ack semantics coordinated with the start-account stall work.

## Reproduction Assessment

No, not as a high-confidence live current-main reproduction. Source inspection shows the 10s gateway send deadline remains, server-side send work can continue after caller timeout, and WhatsApp lacks the reconciliation hook needed to resolve unknown outcomes.

## Solution Assessment

Unclear: simply increasing the timeout is not enough for multi-minute reconnects. The safer solution is a maintained send-status contract with idempotent WhatsApp reconciliation or explicit queued/ack semantics.
