## Summary

Keep open: current main still has the 10s gateway message-action deadline, the server-side WhatsApp send can continue after the caller times out, and the WhatsApp durable send surfaces do not expose the unknown-send reconciliation hook needed to make late success visible to callers.

## What This Changes

Prevent WhatsApp dispatch from reporting an unknown outcome after a gateway timeout when the underlying send later succeeds.

## Best Possible Solution

Keep this as the WhatsApp dispatch false-negative tracker and implement a durable send-status contract, preferably WhatsApp unknown-send reconciliation or explicit queued/ack semantics coordinated with the start-account stall work.

## Reproduction Assessment

No high-confidence live reproduction was established on current main. The source path is clear: message actions still use a 10s gateway deadline, server-side dispatch can outlive the caller, and WhatsApp lacks the durable reconciliation hook for late successful sends.

## Solution Assessment

Unclear: increasing the timeout alone is not durable for multi-minute reconnects. The safer solution is a maintained send-status contract with idempotent reconciliation or explicit queued semantics, chosen deliberately across gateway and plugin boundaries.
