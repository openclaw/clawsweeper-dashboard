## Summary

Keep open: current main and v2026.5.12 still have the 10s outbound gateway send timeout, and the WhatsApp adapter still lacks unknown-send reconciliation, so the reported delivered-after-timeout false-negative is not fixed or superseded.

## What This Changes

The issue reports false-negative WhatsApp send status when a 10s gateway RPC timeout fires while delivery later succeeds during a WhatsApp reconnect/start-account stall.

## Best Possible Solution

Keep this as the WhatsApp outbound dispatch false-negative tracker and design a deliberate send-status contract, preferably WhatsApp unknown-send reconciliation or explicit queued/ack semantics coordinated with the start-account stall work.

## Reproduction Assessment

No, not as a high-confidence live current-main reproduction. The source path is clear: message sends still use a 10s gateway deadline, server-side send work can continue after the caller times out, and WhatsApp lacks the existing unknown-send reconciliation hook.

## Solution Assessment

Unclear: a longer timeout alone is not the best durable fix for multi-minute reconnects. The safer direction is to preserve idempotency and reconcile unknown-after-send WhatsApp outcomes, with any queued/ack status contract chosen deliberately.
