## Summary

Keep open: current main still has the 10s gateway RPC timeout path, server-side send work can complete after the caller times out, and the WhatsApp adapter still does not declare or implement unknown-send reconciliation, so the false `DISPATCHED-UNKNOWN` outcome is not fixed or superseded.

## What This Changes

The issue asks OpenClaw to avoid permanent `DISPATCHED-UNKNOWN` results for WhatsApp sends that actually complete after a gateway ack timeout during reconnect or `start-account` stalls.

## Best Possible Solution

Keep this as the WhatsApp dispatch false-negative tracker and implement a durable send-status contract, likely by adding WhatsApp unknown-send reconciliation or explicit queued/ack semantics that preserve idempotency across caller timeouts.

## Reproduction Assessment

No, not as a high-confidence live current-main reproduction. The source path is clear: gateway message sends still use a 10s client deadline, server-side send work can continue after timeout, and WhatsApp lacks the existing adapter reconciliation hook for late successful sends.

## Solution Assessment

Unclear: simply raising the timeout is not a durable fix for multi-minute reconnects. The safer solution is an idempotent send-status contract with WhatsApp unknown-send reconciliation or a deliberately chosen queued/ack state.
