## Summary

Keep open: current `main` and `v2026.5.12` still preserve the reported failure shape. Message sends keep a 10s gateway RPC deadline, server-side send work waits for final delivery before replying, and WhatsApp still lacks the `reconcileUnknownSend` capability needed to turn an uncertain send into a confirmed delivered result later.

## What This Changes

The issue reports WhatsApp sends returning an unknown dispatch result after a hard 10s gateway timeout even though the message may later be delivered.

## Best Possible Solution

Keep this as the WhatsApp dispatch false-negative tracker and implement a deliberate send-status contract, preferably WhatsApp unknown-send reconciliation or explicit queued/ack semantics coordinated with https://github.com/openclaw/openclaw/issues/78165.

## Reproduction Assessment

No high-confidence live reproduction was run on current `main`. The source path is clear: message sends still use a 10s gateway deadline, server-side send work can outlive the caller timeout, and WhatsApp lacks the existing unknown-send reconciliation hook.

## Solution Assessment

Unclear: a longer timeout alone is not the best durable fix for multi-minute reconnects. The safer direction is to preserve idempotency and reconcile unknown-after-send WhatsApp outcomes, with any queued/ack status contract chosen deliberately.
