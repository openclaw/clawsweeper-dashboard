## Summary

Keep open. Current main still has a false-negative shape for this report: the legacy gateway CLI helper keeps a 10s RPC deadline, the gateway send method waits for durable/platform delivery before replying, and I did not find a current CLI-visible reconciliation path that can turn a timed-out WhatsApp dispatch from unknown into delivered after the Baileys message id appears.

## What This Changes

The issue reports a WhatsApp outbound send where the gateway dispatch RPC times out and records an unknown outcome even though the recipient later receives the message.

## Best Possible Solution

Define one outbound delivery contract that prevents a timed-out WhatsApp send from staying permanently unknown after platform delivery, while keeping the WhatsApp start-account stall tracked separately.

## Reproduction Assessment

No live current-main WhatsApp reproduction was run; source inspection gives a medium-confidence path because the legacy CLI can time out at 10s while the gateway send RPC waits for durable/Baileys delivery and lacks a caller-visible post-timeout reconciliation result.

## Solution Assessment

A simple timeout increase is not the best fix by itself; the safer direction is a durable queued/final-delivery status contract plus reconciliation, coordinated with the WhatsApp start-account stall tracked in https://github.com/openclaw/openclaw/issues/78165.
