## Summary

Keep open: current main still has the 10s gateway deadline on message sends and WhatsApp still does not declare or implement unknown-send reconciliation, so the reported false-negative delivery status is not fixed or superseded.

## What This Changes

The issue asks OpenClaw to avoid permanent `DISPATCHED-UNKNOWN` outcomes when a WhatsApp send times out at the gateway but later succeeds.

## Best Possible Solution

Keep this as the WhatsApp false-negative send-status tracker and implement adapter-level unknown-send reconciliation or explicit queued/ack semantics using the existing durable message lifecycle.

## Reproduction Assessment

No, not as a high-confidence live end-to-end reproduction on current main. The source path is clear: sends still use a 10s gateway deadline, server work can continue after the client times out, and WhatsApp lacks the reconciliation hook that would resolve a late successful send.

## Solution Assessment

Unclear for a single proposed fix direction: increasing the timeout alone would not cover multi-minute reconnects. The safer maintainable path is to preserve idempotency and reconcile unknown-after-send WhatsApp outcomes while coordinating with the related start-account and transport-stall work.
