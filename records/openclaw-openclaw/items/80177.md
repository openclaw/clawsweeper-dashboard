## Summary

Keep open: current main still has the 10s gateway send timeout path, server-side send work can finish after the caller has timed out, and WhatsApp still lacks the existing unknown-send reconciliation hook for late successful sends.

## What This Changes

The issue reports that WhatsApp sends can return `DISPATCHED-UNKNOWN` after a 10s gateway timeout even though the message later delivers successfully.

## Best Possible Solution

Keep this as the WhatsApp dispatch false-negative tracker and implement a durable send-status contract, preferably adapter-level unknown-send reconciliation or explicit queued/ack semantics coordinated with the start-account stall work.

## Reproduction Assessment

No, not as a high-confidence live reproduction on current main. The source path is clear: message sends still use a 10s gateway deadline, server-side work can continue after the caller times out, and WhatsApp has no reconciliation hook for late successful sends.

## Solution Assessment

Unclear: increasing the timeout alone is not a durable fix for multi-minute reconnects. The safer direction is a deliberate send-status contract that preserves idempotency and reconciles unknown-after-send WhatsApp outcomes.
