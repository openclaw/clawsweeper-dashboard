## Summary

Keep open: current main still has the 10s gateway send/action deadline and the WhatsApp message adapter still lacks the existing unknown-send reconciliation hook, so the reported false-negative delivery status is not fixed or superseded.

## What This Changes

The issue reports WhatsApp send-gate returning an unknown dispatch result after a 10s gateway timeout even though the message is later delivered.

## Best Possible Solution

Keep this issue as the WhatsApp send false-negative tracker and coordinate a maintainer decision around adapter-level unknown-send reconciliation or queued/ack semantics, alongside the adjacent start-account and transport-stall work.

## Reproduction Assessment

No, not as a high-confidence live end-to-end reproduction on current main. The source path is clear: send/action calls still use a 10s gateway deadline, server work can continue after the client times out, and WhatsApp lacks the reconciliation hook that would resolve a late successful send.

## Solution Assessment

Unclear: increasing the timeout alone is unlikely to handle multi-minute reconnects cleanly. The safer direction is to preserve idempotency and reconcile unknown-after-send WhatsApp outcomes, while separately reducing the start-account stall that exposes the timeout.
