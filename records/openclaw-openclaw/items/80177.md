## Summary

Keep open: current main still has the reported 10s gateway message-send deadline, and WhatsApp does not implement the existing unknown-send reconciliation hook that would turn a late successful send into a durable dispatched result.

## What This Changes

The issue reports WhatsApp sends returning an unknown dispatch result after a 10s gateway timeout even though the message is later delivered.

## Best Possible Solution

Keep this as the WhatsApp false-negative send-status tracker and implement adapter-level unknown-send reconciliation or an explicit queued/ack contract, coordinated with the open start-account and transport-stall work.

## Reproduction Assessment

No, not as a live end-to-end reproduction on current main. Source inspection does show the 10s gateway deadline, server work that can outlive the caller timeout, and no WhatsApp reconciliation hook for a late successful send.

## Solution Assessment

Unclear: raising the timeout alone is unlikely to handle multi-minute reconnects cleanly. The safer maintainable direction is to preserve idempotency and reconcile unknown-after-send WhatsApp outcomes, with a maintainer decision on whether queued/ack semantics should become a generic contract.
