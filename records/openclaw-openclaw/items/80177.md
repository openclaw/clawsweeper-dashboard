## Summary

Keep open: current main still has the 10s gateway send RPC deadline, server-side WhatsApp send work can complete after the client times out, and the WhatsApp adapter does not implement the existing unknown-send reconciliation hook.

## What This Changes

The issue requests a fix for WhatsApp dispatch calls that report an unknown timeout after the 10s gateway deadline even though the message later sends successfully.

## Best Possible Solution

Keep this as the WhatsApp false-negative send-status tracker and implement a maintained status contract, preferably WhatsApp unknown-send reconciliation or explicit queued/ack semantics, while separately reducing start-account stalls.

## Reproduction Assessment

No, not as a live end-to-end current-main reproduction. Source inspection gives a clear path: WhatsApp sends use the 10s gateway RPC, the server waits for durable delivery after the client may time out, and WhatsApp lacks an unknown-send reconciler.

## Solution Assessment

Unclear: increasing the 10s timeout alone is not enough for multi-minute reconnect windows. The narrower durable fix is to preserve idempotency and reconcile or explicitly surface queued/ack state, coordinated with the start-account stall work.
