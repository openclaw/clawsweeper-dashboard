## Summary

Keep open: current main and the latest shipped release still have the 10s message gateway deadline, and WhatsApp still does not declare or implement unknown-send reconciliation, so the reported delivered-after-timeout false-negative remains unresolved.

## What This Changes

The issue reports WhatsApp sends returning `DISPATCHED-UNKNOWN` after a 10s Gateway timeout even though the underlying send later succeeds, and asks for safer timeout/status/reconciliation semantics.

## Best Possible Solution

Keep this as the WhatsApp dispatch false-negative tracker and implement a deliberate send-status contract, preferably WhatsApp unknown-send reconciliation or explicit queued/ack semantics coordinated with the start-account stall work.

## Reproduction Assessment

No high-confidence live reproduction was run on current main. Source inspection does show the 10s gateway deadline, server-side delivery work that can continue past the caller timeout, and no WhatsApp unknown-send reconciler.

## Solution Assessment

Unclear: a longer timeout alone is not the best durable fix for reconnect stalls that last minutes. The safer solution is an explicit send outcome contract with WhatsApp reconciliation or queued/ack semantics, while the start-account stall remains tracked separately.
