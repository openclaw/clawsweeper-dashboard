## Label Justifications

- `P1`: The issue describes a broken WhatsApp/gateway send workflow where users can see unknown or timed-out dispatch results despite successful platform delivery.
- `impact:message-loss`: The false-negative send status can cause operators or automation to retry already delivered WhatsApp messages or treat delivery as failed.
- `impact:session-state`: Caller-visible send state can remain permanently unknown even though the external WhatsApp platform has accepted and delivered the message.

## Summary

Keep open. Current main still has a source-reproducible path where a WhatsApp send can outlive the caller's gateway RPC deadline, and WhatsApp still does not declare an unknown-send reconciler that could later turn the caller-visible unknown outcome into a delivered result.

## What This Changes

The issue reports that WhatsApp gateway sends can return DISPATCHED-UNKNOWN after the 10s RPC deadline even though the platform send later delivers, leaving downstream send-gate state permanently ambiguous.

## Best Possible Solution

Define and implement a maintainer-approved WhatsApp deferred-delivery contract, preferably by adding adapter-backed unknown-send reconciliation or an explicit queued/accepted outcome so callers do not retry already delivered messages.

## Reproduction Assessment

Yes for the source path: current main still combines a 10s gateway caller timeout, final-result send RPC semantics, and no WhatsApp unknown-send reconciler. A live WhatsApp stall/reconnect run is still needed to prove the exact timing against Baileys on current main.

## Solution Assessment

Unclear until maintainers choose the user-visible delivery status contract. The narrowest maintainable path is likely WhatsApp reconciliation or an explicit deferred-delivery outcome, not only increasing the gateway timeout.
