## Summary

Keep open: current main still has the reported 10s gateway send timeout path and WhatsApp does not declare or implement unknown-send reconciliation, so late successful sends can still be surfaced as unknown to callers.

## What This Changes

The issue requests fixing WhatsApp sends that hit the 10s gateway RPC deadline and leave an unknown result even though the underlying Baileys send can later deliver successfully.

## Best Possible Solution

Keep this as the WhatsApp send false-negative tracker and implement adapter-level unknown-send reconciliation or a deliberate queued/ack status contract, coordinated with separate start-account and transport-stall fixes.

## Reproduction Assessment

No, not as a live current-main reproduction in this read-only pass. The source path is clear: message sends still use a 10s gateway deadline, server-side work can continue after the caller times out, and WhatsApp lacks the reconciliation hook that would resolve a late successful send.

## Solution Assessment

Unclear: increasing the timeout alone is not a durable fix for multi-minute reconnects. The more maintainable direction is to preserve idempotency through WhatsApp unknown-send reconciliation or explicit queued/ack semantics while separately reducing the start-account stall.
