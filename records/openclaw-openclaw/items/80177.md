## Summary

Keep open: current main still has the 10s gateway send timeout path, and the WhatsApp adapter still lacks the generic `reconcileUnknownSend` hook needed to turn an unknown-after-send state into a confirmed sent result. The observable false-negative dispatch status is therefore not implemented or superseded.

## What This Changes

The issue reports WhatsApp sends returning an unknown dispatch result after a 10s gateway RPC timeout even though the message later succeeds, and asks for timeout/back-pressure, reconciliation, or queued/ack status semantics.

## Best Possible Solution

Keep this as the WhatsApp dispatch false-negative tracker and implement durable send-status semantics, preferably WhatsApp `reconcileUnknownSend` support plus a deliberate queued/ack contract for gateway timeouts.

## Reproduction Assessment

No high-confidence live reproduction was established on current main. The source path is clear: gateway sends still use a 10s deadline, server-side work can continue after the caller times out, and WhatsApp has no unknown-send reconciliation hook.

## Solution Assessment

Unclear: increasing the timeout alone is not the best durable fix for multi-minute reconnects. The safer maintainable direction is idempotent send tracking with adapter-level reconciliation and an explicitly chosen queued/ack status contract.
