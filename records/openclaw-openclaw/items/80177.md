## Summary

Keep open: current main still has the 10s gateway send/action timeout path, server-side delivery can keep running after the caller times out, and WhatsApp still lacks an adapter-level unknown-send reconciliation hook or explicit queued/ack status for this false-negative outcome.

## What This Changes

Request to fix WhatsApp dispatches that return an unknown/timeout status after 10 seconds even though the underlying WhatsApp send later succeeds.

## Best Possible Solution

Keep this as the outbound WhatsApp dispatch false-negative tracker and choose a durable send-status contract that preserves idempotency, either via WhatsApp unknown-send reconciliation or explicit queued/ack semantics coordinated with the start-account stall work.

## Reproduction Assessment

No high-confidence live reproduction was run on current main. The source path is clear: sends still use a 10s gateway deadline, server-side delivery can continue after that deadline, and WhatsApp does not currently implement the existing unknown-send reconciliation hook.

## Solution Assessment

Unclear: simply increasing the timeout is not the best durable fix for multi-minute reconnects. A maintainable solution should preserve idempotency and define how callers observe queued, sent, or unreconciled outcomes after the gateway deadline expires.
