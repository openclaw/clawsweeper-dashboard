## Label Justifications

- `P1`: The issue describes a broken active WhatsApp send workflow where delivered messages can be reported as unknown to callers.
- `impact:message-loss`: The false unknown dispatch status can cause operators or automation to retry already delivered messages or treat delivery as failed.
- `impact:session-state`: Caller-visible send state can remain unknown even though the external WhatsApp platform accepted and delivered the message.

## Summary

Keep open. Current main and the latest release still allow a WhatsApp send RPC to time out at the gateway wrapper before the server-side send finishes, and WhatsApp still has no unknown-send reconciler to convert the caller-visible unknown outcome into a delivered result later.

## What This Changes

The issue requests reliable WhatsApp dispatch status when a gateway send call times out before a later-successful platform send completes.

## Best Possible Solution

Define a maintainer-approved deferred-delivery contract for sends that outlive the caller timeout, then implement it in the WhatsApp plugin/core durable-send path through adapter-backed reconciliation or an explicit queued/accepted outcome with tests and live WhatsApp proof.

## Reproduction Assessment

Yes for the source path: current main still combines a default 10s gateway wrapper timeout, a final-result send RPC, and no WhatsApp unknown-send reconciler; live WhatsApp timing was not reproduced here.

## Solution Assessment

Unclear until maintainers choose the user-visible delivery-status contract; a WhatsApp reconciler or explicit deferred outcome is likely cleaner than only increasing the gateway timeout.
