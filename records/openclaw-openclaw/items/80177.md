## Summary

Keep open: current main still has the 10s message-action gateway deadline, server-side dispatch work can continue after the caller times out, and the WhatsApp message adapter does not implement unknown-send reconciliation, so the false-negative delivery status is not fixed or superseded.

## What This Changes

The issue requests that WhatsApp dispatch avoid permanent `unknown` outcomes when a 10s gateway acknowledgement times out but the underlying send later succeeds.

## Best Possible Solution

Keep this as the WhatsApp false-negative send-status tracker and implement adapter-level unknown-send reconciliation or explicit queued/ack semantics on top of the existing durable message lifecycle, coordinated with the adjacent start-account and transport-stall work.

## Reproduction Assessment

No, not as a live end-to-end reproduction on current main. The source path is clear: message sends still use a 10s gateway deadline, server dispatch can continue after the caller times out, and WhatsApp lacks the reconciliation hook that would resolve a late successful send.

## Solution Assessment

Unclear: raising the 10s timeout alone is not durable for multi-minute reconnects. The safer solution is to preserve idempotency and reconcile unknown-after-send WhatsApp outcomes, while separately reducing the start-account stall that exposes the timeout.
