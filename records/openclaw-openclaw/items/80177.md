## Summary

Keep open: current main still has the 10s gateway send/action timeout and WhatsApp still does not declare or implement unknown-send reconciliation, so the reported delivered-after-timeout false-negative is not fixed or superseded.

## What This Changes

Fix WhatsApp sends that return `DISPATCHED-UNKNOWN` after the gateway deadline even though the platform send later succeeds, either by reconciliation or clearer queued/ack semantics.

## Best Possible Solution

Keep this as the WhatsApp false-negative send-status tracker and implement adapter-level `reconcileUnknownSend` or explicit queued/ack semantics on the durable send lifecycle, coordinated with the start-account stall work.

## Reproduction Assessment

No, not as a high-confidence live end-to-end reproduction on current main. Source inspection does show the path: the client timeout remains 10s, gateway send work can continue past that deadline, and WhatsApp lacks the reconciliation hook that would later resolve an unknown send as delivered.

## Solution Assessment

Unclear: only increasing the timeout would not handle multi-minute reconnects reliably. The safer maintainable fix is to preserve idempotency and reconcile unknown-after-send WhatsApp outcomes, while separately reducing the start-account stall that exposes the deadline.
