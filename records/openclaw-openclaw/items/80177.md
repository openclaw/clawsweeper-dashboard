## Summary

Keep open: current main still has the 10s gateway RPC timeout path and the WhatsApp message adapter still lacks unknown-send reconciliation, so the reported delivered-after-timeout false negative is not fixed or superseded.

## What This Changes

The issue requests a fix for WhatsApp sends that return `DISPATCHED-UNKNOWN` after a 10s gateway acknowledgment timeout even though the platform send later succeeds.

## Best Possible Solution

Keep this issue as the WhatsApp send false-negative tracker and implement adapter-level unknown-send reconciliation or explicit queued/ack semantics on the existing durable message lifecycle, coordinated with the adjacent start-account and transport-stall work.

## Reproduction Assessment

No, not as a live end-to-end reproduction on current main. The source path is clear: message sends still use a 10s gateway deadline, server work can continue after the client times out, and WhatsApp lacks the reconciliation hook that would resolve a late successful send.

## Solution Assessment

Unclear: raising the 10s timeout alone is not enough for multi-minute reconnects. The safer direction is to preserve idempotency and reconcile unknown-after-send WhatsApp outcomes, while separately reducing the start-account stall that exposes the timeout.
