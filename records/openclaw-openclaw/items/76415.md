## Summary

Keep open: current main still reloads `chat.history` from the active compacted transcript and renders only a generic chat divider, while checkpoint recovery is available in the Sessions UI rather than in the chat history view. The earlier general history-hydration item #68795 is related but does not cover this compaction-specific projection gap.

## What This Changes

Request that Control UI/webchat preserve, explain, or expose recovery for pre-compaction conversation history after a chat refresh loads the compacted active transcript.

## Best Possible Solution

Choose a bounded chat-level compaction affordance that makes the boundary explicit and connects users to checkpoint context or recovery without unboundedly replaying large pre-compaction transcripts.

## Reproduction Assessment

Yes, source-reproducible: `chat.history` reloads the active transcript branch, checkpoint snapshots are stored separately, and the chat renderer currently shows only a generic compaction divider. I did not run a live long-session compaction in this read-only review.

## Solution Assessment

Unclear: current main has the storage and recovery building blocks, but the requested end state needs a maintainer decision between history reconstruction, a richer boundary, or checkpoint actions in the chat view.
