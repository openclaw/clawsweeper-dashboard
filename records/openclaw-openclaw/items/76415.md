## Summary

Keep open: current main still has the reported Control UI compaction-history gap, and an open closing PR already targets this exact issue, so the issue should remain open until that PR is reviewed and merged or closed.

## What This Changes

The issue asks for Control UI/WebChat history refresh after compaction to stop making earlier turn-by-turn messages appear silently lost, either by preserving/reconstructing history or by surfacing an explicit checkpoint recovery affordance.

## Best Possible Solution

Review and land or close PR #76437, or replace it with an equivalent chat-level compaction affordance that keeps bounded active-transcript semantics while making checkpoint recovery discoverable.

## Reproduction Assessment

Yes, source-reproducible: current main reloads chat history from the active transcript branch while storing pre-compaction snapshots separately, and the chat renderer only shows a generic compaction divider. I did not run a live long-session compaction because this review is read-only.
