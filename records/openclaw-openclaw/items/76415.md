## Summary

Keep open: current main still reloads only the active transcript and renders a generic compaction divider; checkpoint recovery exists in Sessions, but the chat view does not explain or recover the pre-compaction transcript.

## What This Changes

Request Control UI to preserve, explain, or recover pre-compaction chat history when a refreshed chat view reloads a compacted successor transcript.

## Best Possible Solution

Choose a chat-level compaction affordance that preserves bounded history semantics while making the pre-compaction checkpoint visible and recoverable from the chat experience.

## Reproduction Assessment

Yes, source-reproducible: chat.history reloads the active transcript branch while checkpoint snapshots are stored separately, and the chat renderer only shows a generic compaction divider. I did not run a live long-session auto-compaction in this read-only review.

## Solution Assessment

Unclear: current main has the storage and Sessions-page recovery building blocks, but the best chat fix needs a maintainer choice between history reconstruction, a richer boundary, or checkpoint actions in the chat view.
