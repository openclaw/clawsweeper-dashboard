## Summary

Keep open. Current main still routes the CLI web search command through raw config loading before local provider execution, while the documented SecretRef contract and existing command-secret resolver support the gateway snapshot path the reporter expected.

## What This Changes

The issue reports that `openclaw infer web search` fails on selected plugin-scoped `webSearch.apiKey` SecretRefs even though the active gateway secrets snapshot can resolve them.

## Best Possible Solution

Wire the web search and web fetch CLI commands through existing command-secret resolution for the selected provider target before local execution, preserving inactive non-selected-provider behavior.

## Reproduction Assessment

Yes. The reporter provides an exact command/config/error, and current main shows a source-reproducible path where `infer web search` reads raw config and never invokes the existing command SecretRef materialization flow before provider execution.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing `secrets.resolve`/command-secret target plumbing for web search and web fetch CLI execution rather than adding a new config surface or changing provider policy.
