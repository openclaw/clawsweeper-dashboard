## Summary

This is a valid current-main bug, but it is already tracked by an existing open fix PR that changes the same CLI web-search entrypoint and target-id plumbing, so a second open issue would duplicate that review thread.

## What This Changes

The issue reports that CLI web search reads plugin-scoped `webSearch.apiKey` SecretRefs before command-level gateway/runtime secret materialization.

## Best Possible Solution

Keep the existing CLI SecretRef materialization direction, review the linked PR for selected-provider scoping and inactive-provider diagnostics, then land or replace it with an equivalent focused fix.

## Reproduction Assessment

Yes. The reported command maps to `runWebSearchCommand`, and current source shows it passes raw runtime config into `runWebSearch` before the documented plugin-scoped Tavily SecretRef is materialized.

## Solution Assessment

Yes. The maintainable direction is to use the existing command SecretRef gateway/local resolution path for the web-search CLI before provider execution, while preserving inactive non-selected provider refs as non-fatal.
