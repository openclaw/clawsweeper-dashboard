## Summary

Keep open: current main still routes `ownsCompaction` sessions away from the normal tool-result context guard and its mid-turn precheck, while Pi auto-compaction is disabled for the same sessions. The earlier #71325 fix covers pre-prompt/windowing safeguards, but this report identifies a distinct mid-turn tool-loop gap that remains plausible from source inspection.

## What This Changes

The issue requests mid-turn context-overflow protection for `ownsCompaction`/Codex sessions during long tool loops.

## Best Possible Solution

Add a generic last-resort mid-turn pressure check for `ownsCompaction` sessions that invokes the existing recovery route or active engine compaction without bypassing context-engine ownership.

## Reproduction Assessment

Yes for the guard bypass in source, but not for the full reported 20+ tool-loop overflow in a live Codex run. Current main shows the owned-engine branch skips the normal tool-result guard and a test asserts that mid-turn precheck is not wired into that hook.

## Solution Assessment

Unclear as proposed: installing the normal guard for every session could conflict with owned context-engine semantics. The safer fix is a last-resort owned-engine mid-turn pressure signal that routes through the existing compaction/retry machinery and has regression coverage for `ownsCompaction: true`.

## Repair Work Prompt

Fix the mid-turn context-overflow guard gap for `ownsCompaction` context engines reported in https://github.com/openclaw/openclaw/issues/77561. Keep the fix generic: do not special-case Codex, lossless-claw, or any plugin id. Preserve context-engine ownership semantics, but add a last-resort mid-turn pressure check for owned engines so long tool loops can trigger the same compaction/retry recovery before provider overflow. Start in `src/agents/pi-embedded-runner/tool-result-context-guard.ts` and `src/agents/pi-embedded-runner/run/attempt.ts`; consider passing a mid-turn precheck callback or equivalent request through `installContextEngineLoopHook` rather than installing the non-owned truncation guard blindly. Update the changelog because this is a user-visible reliability fix. Do not add a new config option unless the existing guard/recovery route cannot safely express the fix.
