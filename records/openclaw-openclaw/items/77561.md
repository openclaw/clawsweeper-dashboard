## Summary

Keep open: current main still routes `ownsCompaction` sessions away from the normal tool-result guard and the opt-in mid-turn precheck, so the reported owned-engine tool-loop gap remains source-reproducible even though I did not run a live 20+ tool-call Codex session.

## What This Changes

Fix owned context-engine/Codex tool loops so `ownsCompaction` sessions get a mid-turn context-pressure safeguard instead of relying only on engine-owned compaction.

## Best Possible Solution

Add a last-resort mid-turn pressure signal for `ownsCompaction` sessions that invokes the existing recovery path or active engine compaction, without directly truncating or rewriting engine-owned context.

## Reproduction Assessment

Yes for the source-level issue: current main still skips the normal tool-result guard and mid-turn precheck for `ownsCompaction: true`, and a regression test locks in that bypass. No live Codex overflow run was performed in this read-only review.

## Solution Assessment

No to the exact proposal of installing the normal guard for every session without qualification, because that can conflict with owned context-engine semantics. The narrower maintainable fix is a last-resort owned-engine mid-turn pressure check that routes through existing compaction/retry machinery.
