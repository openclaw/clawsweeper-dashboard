## Summary

Current main and shipped releases have the central guard-bypass fix: ownsCompaction engines now compose the context-engine loop hook with the generic tool-result context guard, and the preflight path uses LLM-boundary/unwindowed pressure so tool-heavy Codex sessions compact before provider overflow.

## What This Changes

The issue reports that Codex-owned compaction paths in v2026.5.3 skipped OpenClaw's mid-turn tool-result context guard, allowing heavy tool loops to overflow context and restart sessions.

## Best Possible Solution

Keep the shipped https://github.com/openclaw/openclaw/pull/85541 implementation for this specific guard bypass; broader context-budget and compaction-observability work can continue in their separate open issues.

## Reproduction Assessment

Yes for the historical bug: the report's old v2026.5.3 guard condition matches the predecessor discussion, and https://github.com/openclaw/openclaw/pull/85541 includes saved-session proof plus regression tests for the same tool-heavy/ownsCompaction pressure path. Current main no longer reproduces the bypass because the guard is installed in both ownsCompaction and non-ownsCompaction branches.

## Solution Assessment

Yes. The current solution is the maintainable fix because it preserves context-engine compaction ownership while adding a host-side tool-result guard and provider-boundary pressure check before Codex submits oversized prompts.
