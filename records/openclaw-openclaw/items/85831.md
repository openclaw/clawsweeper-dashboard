## Summary

Keep open: this looks like a real Control UI config serialization bug on current main. Missing built-in provider baseUrl values are accepted in source config, but the Control UI can round-trip the materialized empty string back through config.set, where the core schema correctly rejects it.

## What This Changes

The issue asks for Control UI Save to preserve omitted built-in provider baseUrl defaults when saving unrelated config changes.

## Best Possible Solution

Normalize the Control UI form-submit path to omit schema-derived empty optional strings for fields absent from the source config while keeping the core provider baseUrl schema strict.

## Reproduction Assessment

Yes at source level: a built-in provider overlay may omit baseUrl, the snapshot/form path can carry a materialized empty string, and config.set then rejects that explicit empty value through the existing schema. I did not run a live Control UI browser repro in this read-only pass.

## Solution Assessment

Yes: the maintainable fix belongs in Control UI serialization/sanitization, not by accepting empty provider base URLs in core validation. The core optional-but-nonempty contract should remain strict for values users actually configure.
