## Summary

Keep open: this is a focused fix in the right xAI plugin boundary, but it is not merge-ready because the PR lacks real behavior proof and still falls back to exposing xAI-billed tools when active-model provider metadata is missing.

## What This Changes

The branch adds active-model provider checks to the xAI `code_execution` and `x_search` lazy tool factories so known non-xAI active models do not receive those tools.

## Best Possible Solution

Land a provider-policy fix that hides xAI-billed tools unless the active provider is xAI, or adds an explicit maintainer-approved opt-in for unknown/cross-provider use, with focused tests and real tool-list proof.

## Reproduction Assessment

Yes. Source inspection shows current main exposes both xAI-billed tools from xAI credential availability without an active-provider gate; I did not run a live xAI/non-xAI billing repro in this read-only review.

## Solution Assessment

No. The xAI plugin factory is the right boundary, but this patch should fail closed or use an explicit opt-in when active-model provider metadata is missing, then prove the xAI, non-xAI, and missing-metadata cases.
