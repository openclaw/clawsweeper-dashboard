## Summary

## What This Changes

The PR adds a Dreaming tab agent selector, exports existing chat agent-selection helpers for reuse, and threads optional agentId parameters through Dreaming status and diary requests.

## Best Possible Solution

Land a complete fix that uses one Dreaming selected-agent source, passes it through all read and write gateway methods, scopes status stats to the selected workspace, preserves default fallback, and includes live multi-agent UI proof.

## Reproduction Assessment

Yes. Source inspection on current main shows the Dreams tab has no agent selector props, the controller sends empty doctor.memory.* params, and the gateway handlers resolve the default agent.

## Solution Assessment

No. The approach is pointed at the right bug, but the current patch is not the best complete solution until it fixes the build errors and uses one selected-agent scope across Dreaming reads, stats, and write actions.

Full review comments:

- **[P1] Import normalizeAgentId from an exported module:** `src/gateway/server-methods/doctor.ts:4`
  - body: The PR imports `normalizeAgentId` from `../../agents/agent-scope.js`, but that barrel does not export it, and `check-prod-types` fails with TS2459. Import it from `../../routing/session-key.js` or intentionally expose it from the barrel before using it here.
  - confidence: 0.96
- **[P1] Add selectedAgentId to the shared view state:** `ui/src/ui/app-render.ts:2933`
  - body: `DreamingState` now requires `selectedAgentId` and `renderApp` writes `state.selectedAgentId`, but `AppViewState` still has no field and the existing callers/fixtures fail type checking. Add the field to the shared state contract and tests, or derive the scope without adding a new required state member.
  - confidence: 0.94
- **[P1] Derive read RPC scope from the visible selected agent:** `ui/src/ui/controllers/dreaming.ts:728`
  - body: The select value is derived from `sessionKey`, but `loadDreamingStatus` and `loadDreamDiary` only send `agentId` after `state.selectedAgentId` has been set by this selector. Opening Dreams while already in a non-default session, or switching sessions elsewhere, can show one agent selected while loading default or stale-agent data.
  - confidence: 0.89
- **[P1] Scope Dreaming actions to the selected agent:** `ui/src/ui/controllers/dreaming.ts:847`
  - body: Backfill, reset, repair, and dedupe still call their RPCs with `{}`, and the gateway action handlers still resolve the default agent. Once the PR adds a selector, those controls can mutate default-agent Dreaming artifacts while another agent is selected.
  - confidence: 0.9
- **[P2] Return selected-agent status stats:** `src/gateway/server-methods/doctor.ts:932-934`
  - body: `doctor.memory.status` now chooses a requested manager, but the Dreaming counts still come from `resolveMemoryDreamingWorkspaces` across configured agents with the default primary agent id. The selector can therefore show aggregate/default counts instead of the selected agent's status.
  - confidence: 0.84
