## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep Dreaming on one selected-agent source:** `ui/src/ui/app-render.ts:2931`
  - body: `renderDreaming` derives the selected option from `state.sessionKey`, but the RPC loaders read the new `state.selectedAgentId`, which starts as `null` and is not updated by ordinary session switches. Opening Dreaming while already on another agent, or switching sessions elsewhere, can show one agent in the selector while loading another agent's data; derive a single Dreaming agent scope and use it for both display and requests.
  - confidence: 0.92
- **[P1] Scope Dreaming actions to the selected agent:** `ui/src/ui/controllers/dreaming.ts:725`
  - body: The PR only passes `agentId` for status and diary reads; Backfill, Reset, Clear Grounded, Repair, and Dedupe still call action RPCs with `{}`, and the gateway action handlers resolve the default agent. With a non-default agent selected, those buttons can modify the default workspace while the UI says another agent is selected.
  - confidence: 0.95
- **[P2] Return per-agent status stats for the selector:** `src/gateway/server-methods/doctor.ts:900`
  - body: The status handler can now report a requested `agentId`, but it still builds `storeStats` by merging every configured Dreaming workspace via `resolveMemoryDreamingWorkspaces`. That leaves the selector able to show aggregate counts instead of the selected agent's short-term recall, grounded signals, and promotions; scope stats to the requested workspace or clearly keep this endpoint aggregate.
  - confidence: 0.87
