## Summary

Keep open: the PR fixes the main Dreaming selector gap and includes connected-browser proof for the normal two-agent path, but the latest in-flight guard is still keyed only by agent ID. An A -> B -> A switch can let the older same-agent request write state or clear tracking for the newer request.

## What This Changes

This PR adds a Dreaming tab agent selector, scopes Dreaming status/diary/actions to the selected agent, and adds UI/controller/gateway regression tests.

## Best Possible Solution

Keep the selector and gateway scoping work, but replace agent-id-only request tracking with per-request generations/tokens for both status and diary, then add an A -> B -> A deferred completion regression before merge.

## Reproduction Assessment

Yes by source inspection: use deferred doctor.memory.status and doctor.memory.dreamDiary calls, switch selectedAgentId A -> B -> A, then resolve the first A request before the second A request. The current guard compares only agent IDs, so it cannot distinguish the stale first A completion from the active second A load.

## Solution Assessment

No: the selector and RPC scoping are the right direction, but agent-id-only request tracking is not robust enough for the introduced concurrent load behavior. A per-request generation/token is the narrower maintainable fix.
