## Summary

Keep open: the PR fixes a valid linked Dreaming tab gap and includes useful connected-browser proof, but the current patch still has a session-state race where switching agents during an active Dreaming load can leave stale previous-agent data under the new selector.

## What This Changes

The PR adds an agent selector to the Control UI Dreaming tab and routes Dreaming status, diary, and diary actions through the selected agent id.

## Best Possible Solution

Keep the selector approach, but make selected-agent Dreaming loads keyed or forced so stale in-flight responses cannot populate the UI after the user switches agents.

## Reproduction Assessment

Yes, source-level. Current main has no Dreaming agent selector and sends unscoped Dreaming RPC payloads; the remaining PR race is also visible from the selector callback calling load functions that no-op while their existing loading flags are true.

## Solution Assessment

No, not quite as currently written. The selector and gateway scoping are the right direction, but the implementation should key or force selected-agent reloads so stale in-flight responses cannot leave the Dreaming tab showing the wrong agent's data.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Do not skip reloads while switching Dreaming agents:** `ui/src/ui/app-render.ts:2937-2938`
  - body: `loadDreamingStatus` and `loadDreamDiary` both return early when their loading flags are already true, so selecting a different agent during an initial refresh can make these calls no-op while the older request later fills the page with the previous agent's data under the new selector. Please make selector-driven loads keyed/forced, or ignore stale responses and then fetch the selected agent, so the tab cannot display or act on the wrong agent's Dreaming state.
