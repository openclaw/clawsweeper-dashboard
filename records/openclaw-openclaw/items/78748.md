The PR adds a Dreaming tab agent selector, exports chat session agent helpers, stores a selected Dreaming agent id, and threads optional agentId parameters into Dreaming status and diary gateway reads.

## Best Possible Solution

Use one Dreaming agent-scope source derived from the displayed selector, pass it through status, diary, and Dreaming action RPCs, scope gateway stats and mutations to that agent, update shared state contracts/tests, and add live multi-agent browser proof.

## Reproduction Assessment

Yes. Source inspection on current main shows the Dreaming tab has no selector state and sends empty doctor.memory.* params, while gateway handlers resolve the default or aggregate Dreaming scope.

## Solution Assessment

No. The fix direction is right, but this patch is incomplete because selected agent state is split from the displayed session, Dreaming action RPCs remain unscoped, and status stats still aggregate all workspaces.

  - confidence: 0.9

  - body: Even when `agentId` is supplied, the status handler still resolves all configured Dreaming workspaces using the default primary agent. That leaves the status panel mixing other agents' memory counts instead of reflecting the selected agent.
  - confidence: 0.86
