## Summary

This PR is a focused fix for the Dreaming tab's missing agent selector, and the later browser proof shows the normal two-agent switch path working. It should stay open because the selector-triggered reload still races with existing in-flight Dreaming status/diary loads, so the UI can remain on previous-agent data after the visible selector changes.

## What This Changes

The PR adds a Dreaming tab agent selector, threads an optional `agentId` through Dreaming UI controller calls and gateway doctor memory handlers, and adds focused UI/controller/gateway regression tests.

## Best Possible Solution

Keep the selector and gateway `agentId` routing, but make Dreaming status and diary loads keyed, forced, or stale-response-safe so the selected agent always owns the final rendered data.

## Reproduction Assessment

Yes, source-level. Current main has no Dreaming agent selector and sends unscoped Dreaming RPC payloads; the remaining PR race is visible because selector-triggered reloads call loaders that return while existing loads are active.

## Solution Assessment

No, not yet. The selector and `agentId` routing are the right shape, but the implementation needs keyed or stale-safe reloads before this is the best fix.
