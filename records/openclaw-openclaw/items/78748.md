## Summary

Keep open: the selector and agent-scoped RPC direction are useful, but PR head still has a source-reproducible session-state race where switching agents during an in-flight Dreaming load can leave the UI showing stale status or diary content for the previous agent.

## What This Changes

The PR adds a Dreaming tab agent selector, selected-agent UI state, agent-scoped Dreaming RPC payloads, gateway handler support for optional agentId parameters, and focused UI/gateway regression tests.

## Best Possible Solution

Keep the selector and agent-scoped gateway work, but make Dreaming loads selected-agent keyed and stale-response-safe before merge.

## Reproduction Assessment

Yes for the remaining blocker by source inspection: PR head calls the selected-agent loads from the selector while the loaders still return early on existing loading flags and do not reject stale previous-agent responses.

## Solution Assessment

No. The PR is the right broad direction, but the selected-agent implementation is not the best merge shape until it forces or keys loads on agent changes and ignores stale responses.
