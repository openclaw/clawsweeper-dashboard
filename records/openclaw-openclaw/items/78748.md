## Summary

Keep open: the PR is a focused fix for the linked Dreaming multi-agent UI gap and has useful live proof, but the current branch still has a selected-agent race where switching during an in-flight Dreaming load can leave or overwrite the view with the previous agent's data.

## What This Changes

The PR adds a Dreaming tab agent selector, selected-agent state, agentId-scoped doctor.memory gateway requests, and UI/gateway regression tests.

## Best Possible Solution

Keep the selector and agent-scoped gateway work, but make Dreaming loads selected-agent keyed and stale-response-safe before merge.

## Reproduction Assessment

Yes. Source inspection of the PR head shows selector changes call the Dreaming loaders while those loaders still return early when status or diary loading is already true, so switching during an in-flight refresh can skip the new-agent reload and allow stale writes.

## Solution Assessment

No, not as-is. The selector and agent-scoped RPC direction is maintainable, but the load path needs selected-agent request keys or stale-response guards before it is the best fix.
