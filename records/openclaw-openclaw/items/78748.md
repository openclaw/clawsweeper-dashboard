## Summary

Keep open: this is a useful Dreaming multi-agent fix with normal-path browser proof, but source inspection still shows a selected-agent in-flight race that can leave the tab showing previous-agent status or DREAMS.md after the selector and URL have moved.

## What This Changes

The PR adds a Control UI Dreaming tab agent selector, threads selected agent IDs through Dreaming UI/controller/gateway requests and actions, and adds focused regression coverage.

## Best Possible Solution

Keep the selector and selected-agent gateway scoping, but make Dreaming status and diary loads keyed, forced, cancellable, or stale-response-safe before merge.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: start a Dreaming status or diary load, switch the selector before it completes, and the new load returns early while the old response can still write state.

## Solution Assessment

No, not yet: the selector and gateway scoping are the right direction, but the best fix also needs keyed/forced reloads or stale-response guards for selected-agent switches.
