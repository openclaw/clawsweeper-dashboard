## Summary

The PR adds a Dreaming tab agent selector and threads optional selected-agent IDs through Dreaming UI, controller, and gateway memory status/diary/action calls.

## Best Possible Solution

Land this after selected-agent Dreaming reloads are made stale-response-safe or force-refreshable, with a small protocol docs update for the new optional agentId parameters.

## Reproduction Assessment

Yes. Current main source shows the Dreaming tab lacks selector props/rendering and sends empty doctor.memory params, and the PR finding is source-reproducible from the loader guards that skip selected-agent reloads while previous loads are in flight.

## Solution Assessment

No, not quite. The selected-agent plumbing is the right direction, but the implementation needs one stale-response-safe reload path before it is the narrow maintainable fix.
