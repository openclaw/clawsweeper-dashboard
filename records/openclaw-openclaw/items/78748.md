## Summary

Keep open. The PR is directionally right, but PR-head source still leaves Dreaming status and diary reads default-scoped while the UI sends a selected agent, so a non-default selection can show the wrong workspace; the supplied proof is also test-only rather than a real multi-agent Control UI run.

## What This Changes

The branch adds a Dreaming tab agent selector, exports shared chat agent-session helpers, stores selected Dreaming agent state, and sends selected-agent payloads through Dreaming UI requests and gateway actions.

## Best Possible Solution

Honor the selected agent consistently in both Dreaming reads and actions, preserve default-agent fallback when no agent is supplied, add non-default gateway regression tests, then require redacted real multi-agent Control UI proof before merge.

## Reproduction Assessment

Yes for a source-level path: PR head sends selected agent IDs from the UI controller, while gateway status and diary read handlers still resolve the default agent. I did not run a live multi-agent browser setup in this read-only review.

## Solution Assessment

No. The selector and shared session helper reuse are the right direction, but the implementation is incomplete until gateway read handlers and tests prove non-default-agent status and diary content are scoped consistently.
