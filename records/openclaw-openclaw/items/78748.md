## Summary

Keep open: the PR fixes the right Dreaming selector gap and has good normal-path proof, but it still has a source-level stale-load bug when users switch agents during an active Dreaming refresh.

## What This Changes

The PR adds a Dreaming tab agent selector, threads selected-agent context through Dreaming UI calls, scopes gateway doctor memory handlers by optional `agentId`, and adds focused regression tests.

## Best Possible Solution

Keep the selector and gateway `agentId` scoping, but make Dreaming status and diary loads keyed, forced, or stale-response-safe so the selected agent always owns the final rendered data.

## Reproduction Assessment

Yes, source-level. Current main has no Dreaming agent selector and sends unscoped Dreaming RPC payloads; the remaining PR race follows from selector-triggered loads hitting existing loading guards.

## Solution Assessment

No, not yet. The selector and gateway scoping are the right direction, but the PR needs selected-agent keyed or stale-safe loading before it is the maintainable fix.
