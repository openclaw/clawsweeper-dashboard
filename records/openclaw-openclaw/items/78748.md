## Summary

Keep open: the PR is the right narrow direction for the linked Dreaming multi-agent bug, but the latest stale-load guard is still keyed only by agent id and can accept an older response after switching A -> B -> A.

## What This Changes

The PR adds a Dreaming tab agent selector and passes the selected agent through Dreaming UI/controller loads and doctor memory RPCs.

## Best Possible Solution

Keep the selector and gateway scoping work, but key status and diary loads by a unique request token, add A -> B -> A stale-completion coverage, and capture final-head real browser or live-output proof for the in-flight switch path.

## Reproduction Assessment

Yes by source inspection. Start a status or diary load for agent A, switch to B, switch back to A before the first A request resolves, and the old A completion can pass the current agent-id guard.

## Solution Assessment

No, not yet. The implementation direction is right, but the stale-response guard needs a per-request identity rather than agent-id-only matching, plus final-head proof for the in-flight path.
