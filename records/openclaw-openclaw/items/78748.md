## Summary

Keep open: the PR fixes the normal Dreaming selector path and includes connected-browser proof, but source inspection still shows an in-flight selected-agent race where old status or diary requests can block or overwrite selected-agent data.

## What This Changes

The PR adds a Dreaming tab agent selector, threads the selected agent through UI state and doctor.memory RPCs, and adds targeted UI/gateway regression tests.

## Best Possible Solution

Keep the selector and agent-scoped gateway handlers, but make Dreaming status and diary loads keyed, forced, cancellable, or stale-response-safe before merge.

## Reproduction Assessment

Yes. Source inspection of the PR head shows onSelectAgent immediately calls loadDreamingStatus/loadDreamDiary while those loaders return early during existing loads and do not guard stale responses.

## Solution Assessment

No. The selector and selected-agent RPC scoping are the right direction, but the patch needs a stale-response-safe loading model before it is the best fix.
