## Summary

Keep open: the PR fixes the normal Dreaming agent selector flow and has usable browser proof, but source inspection still shows an in-flight selected-agent load race that can display old-agent status or diary data after switching.

## What This Changes

The PR adds a Control UI Dreaming tab agent selector and routes Dreaming status, diary, and actions through the selected agent context.

## Best Possible Solution

Keep the selector and selected-agent gateway scoping, but make Dreaming status and diary loads keyed, forced, or stale-response-safe before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start a Dreaming status or diary load, switch the Dreaming selector before it completes, and the selected-agent load can return early while the old response still writes state.

## Solution Assessment

No. The selector and agentId scoping are the right direction, but the implementation needs keyed or stale-response-safe Dreaming loads before it is the best fix.
