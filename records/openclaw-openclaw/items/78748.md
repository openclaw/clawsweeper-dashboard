## Summary

Keep open: this is a useful focused fix with supplied browser proof, but the PR still has a session-state race where a fast agent switch during active Dreaming loads can leave status or diary content from the previous agent visible after the selector and URL move to the new agent.

## What This Changes

The branch adds a Control UI Dreaming tab agent selector, threads selected agent IDs through Dreaming UI/controller and doctor memory gateway methods, and adds UI/gateway regression tests.

## Best Possible Solution

Keep the selector and optional gateway agentId scoping, but make Dreaming status and diary loads selected-agent-keyed, forced, or stale-response-safe before merge.

## Reproduction Assessment

Yes, source-level. PR head sets selectedAgentId and calls the Dreaming loaders on selector change, while those loaders still return early when status or diary loading is already true; I did not run a live browser repro in this read-only pass.

## Solution Assessment

No, not as-is. The selector and gateway agentId scoping are the right direction, but the implementation needs selected-agent-keyed or stale-response-safe loading before it is the maintainable fix.

- **[P2] Force selected-agent Dreaming reloads during active loads:** `ui/src/ui/app-render.ts:2937-2938`
