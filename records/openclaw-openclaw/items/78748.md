Keep open: the selector is useful and directly addresses the linked multi-agent Dreaming gap, but current PR head still has selected-agent scoping bugs that can show or mutate the wrong agent workspace.

## What This Changes

The PR adds a Control UI Dreaming tab agent selector, exports chat agent-session helpers, stores selected Dreaming agent state, threads some Dreaming action RPCs through `agentId`, and adds focused view coverage.

## Best Possible Solution

Use one selected-agent source for Dreaming display, status, diary, and actions; thread it through the read and write gateway handlers with default-agent fallback; add default and non-default regression coverage; then provide redacted live multi-agent UI proof.

## Reproduction Assessment

Yes. Source inspection shows current main lacks a Dreaming selector, and PR head still reloads status and diary through helpers that send `{}` to default-agent gateway read handlers.

## Solution Assessment

No. The selector is the right surface, but this PR is not the best fix until one selected-agent value is threaded through all Dreaming reads, writes, gateway handlers, and regression tests.
