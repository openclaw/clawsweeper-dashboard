## Summary

Keep open: the PR fixes the normal Dreaming selector path and has useful proof, but it still has a source-reproducible selected-agent in-flight load race that can show stale agent data after a switch.

## What This Changes

The branch adds a Dreaming tab agent selector, selected-agent UI state, agentId-aware doctor.memory handlers, and targeted UI/gateway tests.

## Best Possible Solution

Keep the selector and agent-scoped gateway handlers, but make Dreaming status and diary loads keyed, forced, cancellable, or stale-response-safe before merge.

## Reproduction Assessment

Yes. Source inspection of the PR head shows onSelectAgent calls the Dreaming reloads while those loaders return early during existing loads and do not guard stale responses.

## Solution Assessment

No. The selected-agent selector and RPC scoping are the right direction, but the implementation needs keyed or stale-safe load handling before it is the best maintainable fix.
