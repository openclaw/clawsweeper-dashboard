## Summary

Keep open: this PR addresses the linked multi-agent Dreaming bug and now has credible browser proof, but a loading-state race can still leave previous-agent status or DREAMS.md content visible after switching agents during an in-flight load.

## What This Changes

The PR adds an agent selector to the Control UI Dreaming tab and passes the selected agent through Dreaming status, diary, and action gateway calls with UI, controller, and gateway tests.

## Best Possible Solution

Land this after the selected-agent load path either ignores stale responses or queues a reload for the new agent when a prior Dreaming status/diary request is still in flight.

## Reproduction Assessment

Yes. Source inspection on current main shows Dreaming lacks an agent selector and its status/diary/action paths resolve through default-agent handling; the PR and linked issue provide a clear multi-agent scenario.

## Solution Assessment

No, not yet. The approach is the right boundary, but it needs stale-response or queued-reload handling before it is the safest maintainable fix for selected-agent Dreaming state.
