## Summary

Keep this PR open: it targets a real Dreaming UI gap, but the current patch can still show or act on the wrong agent workspace because selected-agent reads and status data are not consistently scoped, and the supplied proof is test-only.

## What This Changes

The PR adds a Dreaming tab agent selector, exports existing chat agent-session helpers, adds selected Dreaming UI state, partially threads agentId into Dreaming action RPCs, and adds focused view coverage.

## Best Possible Solution

Use one normalized Dreaming agent scope for selector display, initial load, manual refresh, status, diary, and all Dreaming actions, with default and non-default regression coverage plus redacted live multi-agent proof.

## Reproduction Assessment

Yes, source-reproducible. Current main lacks a Dreaming selector, and PR head still has source paths where non-default selection reloads status/diary without an agentId and the gateway falls back to the default agent.

## Solution Assessment

No, not yet. Adding the selector is the right direction, but the patch needs a single selected-agent contract across UI state, gateway read handlers, action handlers, and tests before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope Dreaming reads to the selected agent:** `ui/src/ui/app-render.ts:2935-2936`
  - body: `onSelectAgent` reloads status and diary after the user picks an agent, but `loadDreamingStatus` and `loadDreamDiary` still send `{}` and the gateway still resolves those reads to the default agent. In a multi-agent setup the selector can move while the visible status and DREAMS.md stay on the wrong workspace.
  - confidence: 0.91
- **[P1] Use one selected-agent source:** `ui/src/ui/app-render.ts:2898`
  - body: The selector value is derived from `state.sessionKey`, while Dreaming actions use the new `state.selectedAgentId`. If the user opens Dreaming while already on a non-default session or changes sessions elsewhere, the UI can show one agent while actions target null, stale, or default state.
  - confidence: 0.86
- **[P2] Return selected-agent status stats:** `src/gateway/server-methods/doctor.ts:926-940`
  - body: Even after the UI starts passing an agentId, `doctor.memory.status` still creates the memory manager for the default agent and merges configured workspaces. The selected-agent view needs status counts and entries for the requested agent instead of aggregate/default data.
  - confidence: 0.84
