## Summary

Keep open: the selector and agent-scoped RPC direction are useful, and the supplied browser proof covers the normal two-agent switch path, but the PR head still has a selected-agent in-flight load race that can leave Dreaming showing stale previous-agent status or diary content.

## What This Changes

## Best Possible Solution

Keep the selector and agent-scoped gateway work, but make Dreaming loads selected-agent keyed and stale-response-safe before merge.

## Reproduction Assessment

Yes. Source inspection of the PR head shows selector changes call Dreaming loaders while those loaders still return early when status or diary loading is already true, so an in-flight switch can skip the new-agent load and allow stale writes.

## Solution Assessment

No, not as-is. The selector and agentId-scoped RPC approach is the right maintainable direction, but it needs selected-agent keyed loading and stale-response protection before merge.
