## Summary

Keep open: this PR fixes the main missing Dreaming agent selector problem and includes useful two-agent browser proof, but the current diff still leaves an in-flight selected-agent reload race that can display stale Dreaming data after switching agents.

## What This Changes

The PR adds a Control UI Dreaming tab agent selector, passes the selected agent id through Dreaming status/diary/action RPCs, and adds focused UI/controller/gateway tests.

## Best Possible Solution

Keep the selector and gateway `agentId` approach, but make Dreaming status and diary loads keyed, forced, or stale-response-safe so switching during an active refresh always ends with selected-agent data.

## Reproduction Assessment

Yes, source-level. Current main has no Dreaming agent selector and sends unscoped Dreaming RPC payloads; the remaining PR race is visible because selector-triggered reloads call controller functions that return while existing loads are active.

## Solution Assessment

No, not as-is. The selector and `agentId` routing are the right direction, but the implementation needs keyed or stale-safe loads before it is the narrow maintainable fix.
