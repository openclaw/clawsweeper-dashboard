## Summary

Keep open: the PR is still not merge-ready because PR head sends selected agent IDs from the UI, but the gateway read handlers for Dreaming status and Dream Diary still resolve the default agent, so the selector can show default-agent data while actions target the selected agent. The supplied proof is test output only and still does not show a real multi-agent Control UI run.

## What This Changes

The PR adds a Dreaming tab agent selector, exports shared chat agent-session helpers, stores selected Dreaming agent state, and sends selected-agent payloads through Dreaming UI and gateway RPC paths.

## Best Possible Solution

Make the selected Dreaming agent one source of truth across selector display, status, diary, and actions, add gateway server-method coverage for non-default reads and writes, then capture redacted real multi-agent Control UI proof.

## Reproduction Assessment

Yes for source-level reproduction: current main lacks the selector, and PR head can be traced from selected-agent UI payloads to gateway read handlers that still resolve the default agent. I did not run a live browser setup in this read-only review.

## Solution Assessment

No. Adding the selector and reusing chat session selection is the right direction, but the patch must also scope the gateway read handlers and prove the real multi-agent UI path before it is the maintainable fix.
