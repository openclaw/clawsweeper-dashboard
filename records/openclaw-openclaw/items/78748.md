## Summary

Keep open: this is a valid UI/session-state fix for the linked multi-agent Dreaming gap, but the current branch still has a source-level race that can leave previous-agent Dreaming data visible after a selector change.

## What This Changes

Adds a Control UI Dreaming tab agent selector, routes Dreaming RPCs and Dreaming actions through a selected agent ID, and adds focused gateway/UI regression tests.

## Best Possible Solution

Keep the selector and gateway agentId scoping, but make Dreaming loads keyed, forced, or stale-response-safe so the selected agent owns the final rendered status and diary state.

## Reproduction Assessment

Yes, source-level. Current main has no Dreaming agent selector and sends unscoped Dreaming RPC payloads, while the remaining PR race follows from calling guarded loaders during a selector change.

## Solution Assessment

No, not as-is. The selector and agentId scoping are the right direction, but the implementation needs keyed or stale-response-safe reloads before merge.
