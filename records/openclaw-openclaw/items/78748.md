## Summary

Keep open: the normal Dreaming selector flow is well supported by the PR and proof, but the selected-agent reload path can still reuse the existing loading guards and leave stale previous-agent Dreaming data after a fast switch.

## What This Changes

The branch adds a Dreaming tab agent selector, threads selected agent IDs through Dreaming UI/controller and doctor memory gateway methods, and adds focused UI/gateway tests.

## Best Possible Solution

Keep the selector and gateway agentId scoping, but make selected-agent Dreaming status and diary loads keyed, forced, or stale-response-safe before merge.

## Reproduction Assessment

Yes, source-level. The PR selector calls loadDreamingStatus and loadDreamDiary while those loaders still return early during active loads, so an old-agent request can finish after the selected agent changes.

## Solution Assessment

No. The PR is the right direction, but the current implementation needs selected-agent-keyed or stale-safe loading before it is the narrowest maintainable fix.
