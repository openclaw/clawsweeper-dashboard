## Summary

Keep open: the patch still leaves selected-agent Dreaming reads default-scoped, so the new selector can show one agent while status/DREAMS.md come from another, and the PR has only test-output proof rather than a real multi-agent UI run.

## What This Changes

The PR adds a Dreaming tab agent selector, reuses chat agent-session helper logic, tracks a selected Dreaming agent in UI state, sends selected-agent payloads from Dreaming controller calls, and scopes several Dreaming action handlers to optional `agentId`.

## Best Possible Solution

Honor the selected agent consistently in Dreaming reads and actions, preserve default-agent fallback when no agent is supplied, add non-default gateway regression tests, then require redacted real multi-agent Control UI proof before merge.

## Reproduction Assessment

Yes for a source-level path: PR head sends selected agent IDs from the UI controller, while gateway status and diary read handlers still resolve the default agent. I did not run a live multi-agent browser setup in this read-only review.

## Solution Assessment

No. The selector and shared session helper reuse are the right direction, but the implementation is incomplete until gateway read handlers and tests prove non-default-agent status and diary content are scoped consistently.
