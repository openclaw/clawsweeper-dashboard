## Summary

Keep open: the PR is aimed at the right Dreaming multi-agent gap, but the current branch is not merge-ready because it leaves a TypeScript state-contract break and can still load or display Dreaming data for the wrong agent.

## What This Changes

The PR adds a Dreaming tab agent selector, threads a selected agent id through Dreaming status and diary requests, and lets doctor.memory status and diary handlers accept an optional agentId.

## Best Possible Solution

Use one Dreaming agent-scope source derived from the active session or selector, pass it through all status/diary/action loads and gateway handlers, scope status stats to that agent's workspace, update shared state types/tests, and provide live multi-agent browser proof.

## Reproduction Assessment

Yes. Source inspection is high-confidence: current main has no Dreaming selector contract, sends empty doctor.memory params, and resolves default or aggregate Dreaming scope in the gateway.

## Solution Assessment

No. The direction is right, but the current patch is not the best complete fix because it leaves the selected agent split across sessionKey and selectedAgentId and does not update all type/test contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Add selectedAgentId to the shared state contract:** `ui/src/ui/app-render.ts:2637`
  - body: This handler writes `state.selectedAgentId`, but `AppViewState` at the PR head still has no `selectedAgentId` member, and `DreamingState` test fixtures are also missing the new required field. That leaves the branch failing TypeScript before the UI can run.
  - confidence: 0.96
- **[P2] Keep Dreaming loads scoped to the displayed agent:** `ui/src/ui/controllers/dreaming.ts:728`
  - body: `dreamingSelectedAgentId` is derived from `state.sessionKey`, but `loadDreamingStatus` and `loadDreamDiary` use `state.selectedAgentId` and fall back to `{}`. Opening Dreaming while already on a non-default agent, or switching sessions outside this dropdown, can show one agent in the selector while loading the default agent's data.
  - confidence: 0.9
- **[P2] Scope Dreaming status stats to the requested agent:** `src/gateway/server-methods/doctor.ts:884`
  - body: Even when an `agentId` is supplied, the status handler still feeds `resolveMemoryDreamingWorkspaces` all configured agent workspaces, so the status counts remain aggregate instead of reflecting the selected agent. That leaves the selector able to change the diary while the status panel still mixes other agents' memory state.
  - confidence: 0.86
