## Summary

Keep open: the selector and agent-scoped gateway work are useful and the PR has sufficient real browser proof, but the current patch can still show stale Dreaming status or diary data when the user switches agents during an in-flight Dreaming load.

## What This Changes

The branch adds a Control UI Dreaming tab agent selector, threads selected-agent IDs through Dreaming UI/controller/gateway doctor memory methods, and adds focused regression tests.

## Best Possible Solution

Keep the selector and gateway `agentId` scoping, but make Dreaming status and diary loads keyed, forced, or stale-response-safe so the selected agent always owns the final rendered state.

## Reproduction Assessment

Yes, source-level. Current main and v2026.5.19 render the Dreaming tab without an agent selector and send unscoped Dreaming RPC payloads, and the remaining PR race follows from the unchanged loading guards.

## Solution Assessment

No, not yet. The selector and gateway scoping are the right direction, but the PR should also prevent in-flight previous-agent loads from becoming the final rendered Dreaming state.
