## Summary

Keep open: the branch addresses the main missing-agent-selector bug, but the selected-agent reload path still has a source-reproducible in-flight load race that can leave Dreaming showing the previous agent's status or diary after the selector changes.

## What This Changes

The PR adds an agent selector to the Control UI Dreaming tab and threads selected agent IDs through Dreaming UI, controller, gateway memory handlers, and focused tests.

## Best Possible Solution

Keep the selector and selected-agent gateway scoping, but make Dreaming status and diary loads keyed, forced, or stale-response-safe before merge.

## Reproduction Assessment

Yes. Source inspection shows current main sends unscoped Dreaming RPC payloads, and the PR still has a clear pending-load reproduction: start a status or diary load for one agent, switch the selector to another agent, and the selected-agent load can no-op while the old response applies.

## Solution Assessment

No, not as-is. The PR's ownership boundary is right, but the selected-agent reload path needs keyed or stale-safe request handling before it is the best fix.
