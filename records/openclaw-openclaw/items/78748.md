## Summary

Keep open: the selector and selected-agent RPC scoping are useful, and the discussion includes normal two-agent browser proof, but the PR still has a source-reproducible stale-response path when the user switches agents during an active Dreaming status or diary load.

## What This Changes

The PR adds a Control UI Dreaming tab agent selector, passes the selected `agentId` through Dreaming UI controller requests and doctor memory RPC handlers, and adds UI/gateway regression tests.

## Best Possible Solution

Keep the selector and selected-agent gateway scoping, but make Dreaming status and diary loads keyed, forced, or stale-response-safe before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start a Dreaming status or diary load, switch the Dreaming selector before it completes, and the selected-agent load can return early while the old response still writes state.

## Solution Assessment

No, not yet. The selector and backend scoping are the right direction, but the implementation needs keyed/forced loads or stale-response rejection before it is the safest fix.
