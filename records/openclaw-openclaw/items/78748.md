## Summary

Keep open: source review still finds wrong-agent Dreaming data paths in the PR, and the supplied proof is test-only rather than real multi-agent Control UI behavior.

## What This Changes

The PR adds a Control UI Dreaming tab agent selector and routes Dreaming status, diary, and actions through a selected agent id.

## Best Possible Solution

Make the selected Dreaming agent a single source of truth for tab loads, status aggregation, diary reads, and actions, then prove the flow in a redacted real multi-agent Control UI session.

## Reproduction Assessment

Yes for a source-level path: PR head uses `selectedAgentId` for Dreaming RPC payloads, but tab activation still calls the loaders before that state is seeded and status still aggregates all configured workspaces. I did not run a live multi-agent browser setup in this read-only review.

## Solution Assessment

No. The selector direction is reasonable, but the current patch leaves selected-agent initialization and status scoping incomplete; the safer fix is to derive the selected agent consistently and filter status data to that agent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Sync the selected agent before Dreaming tab loads:** `ui/src/ui/controllers/dreaming.ts:733`
  - body: `loadDreamingStatus()` now scopes requests from `state.selectedAgentId`, but the normal `dreams` tab refresh path in `app-settings.ts` still calls the Dreaming loaders before any PR-added sync runs. If a user is already in a non-default session and opens Dreaming, the selector is rendered from `state.sessionKey` while the first status/diary requests send `{}`, so the UI can show default-agent data under a non-default selection. Seed `selectedAgentId` from the current session before that refresh path, or derive the payload from the same source used by the selector.
  - confidence: 0.9
- **[P1] Filter Dreaming status to the requested agent:** `src/gateway/server-methods/doctor.ts:929-932`
  - body: Passing `agentId` as `primaryAgentId` does not make status agent-scoped: `resolveMemoryDreamingWorkspaces()` still enumerates every configured agent workspace before adding the primary workspace, and this handler merges all returned stats. A non-default selection can therefore still show all-agent counts and entries, even though the payload says `agentId` is the selected agent. Use the requested agent workspace for this selected-agent status path or filter the resolver output before merging.
  - confidence: 0.88
