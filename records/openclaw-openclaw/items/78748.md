## Summary

Keep open: the selector and agent-scoped RPC direction are useful, and the normal two-agent browser path has proof, but the PR still leaves a selected-agent session-state race when Dreaming status or diary loads are already in flight.

## What This Changes

The PR adds a Control UI Dreaming agent selector, passes selected agent IDs through Dreaming UI controllers and doctor memory RPCs, and adds targeted UI/gateway tests.

## Best Possible Solution

Keep the selector and gateway agent-scoping work, but make Dreaming status and diary loads selected-agent keyed, force the selector-triggered refresh when appropriate, and ignore stale previous-agent responses before merge.

## Reproduction Assessment

Yes for the remaining blocker by source inspection: selecting an agent calls the Dreaming loaders while their loading guards can return during an in-flight previous-agent request. The normal two-agent selector path also has connected-browser proof in the PR discussion.

## Solution Assessment

No, not yet. The implementation direction is the right narrow fix, but it needs selected-agent keyed loading and stale-response protection before it is the best mergeable solution.
