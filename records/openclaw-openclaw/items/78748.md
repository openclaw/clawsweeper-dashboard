## Summary

Keep open: the normal two-agent Dreaming selector path is useful and has supplied browser proof, but the PR still has a source-reproducible selected-agent in-flight reload race that can leave old-agent Dreaming data visible after switching agents.

## What This Changes

The branch adds a Control UI Dreaming agent selector, selected-agent state, agentId-scoped doctor.memory RPCs, and UI/gateway regression tests.

## Best Possible Solution

Keep the selector and agent-scoped RPC work, but make Dreaming status and diary loads selected-agent keyed, forced, cancellable, or stale-response-safe before merge.

## Reproduction Assessment

Yes. Source inspection of the PR head shows onSelectAgent calls the Dreaming loaders while those loaders return early when status or diary loading is already true, so switching during an in-flight refresh can skip the new-agent reload and allow stale state writes.

## Solution Assessment

No, not as-is. The selector and agent-scoped handlers are the right direction, but the implementation needs keyed or stale-safe selected-agent reloads before it is the maintainable fix.
