## Summary

Keep open: this PR addresses the main missing Dreaming agent selector and supplies connected-browser proof for the normal two-agent path, but current head still has a source-reproducible in-flight switch race that can leave previous-agent Dreaming state visible after switching agents.

## What This Changes

Adds a Control UI Dreaming tab agent selector, routes Dreaming status/diary/actions through a selected agent id, and adds UI/controller/gateway regression coverage.

## Best Possible Solution

Keep the selector and selected-agent gateway scoping, but make Dreaming status and diary loads keyed, forced, cancellable, or stale-response-safe before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: start a Dreaming status or diary load, switch the selector before it completes, and the selected-agent reload can return early while the old response still writes state.

## Solution Assessment

No, not yet. The selector and selected-agent gateway scoping are the right direction, but the UI/controller needs keyed or stale-response-safe reloads before this is the best maintainable fix.

- **[P2] Force selected-agent Dreaming reloads during active loads:** `ui/src/ui/app-render.ts:2937-2938`
  - body: When a user changes the Dreaming agent while `loadDreamingStatus` or `loadDreamDiary` is already in flight, these new calls hit the existing loading guards and return without fetching the new agent.
