## Summary

Keep open: current main still lacks configurable fallback labels, so the PR is not obsolete, but the current diff renders labels before final target resolution, applies per-agent template policy from the target instead of the requester, lacks public docs, lacks contributor-supplied runtime proof, and is currently conflicting.

## What This Changes

The PR adds optional `labelTemplate` config under global and per-agent subagent settings and uses it to synthesize a fallback `sessions_spawn` label when no explicit label is passed.

## Best Possible Solution

Land a target-aware, requester-scoped implementation that renders fallback labels after native/ACP target resolution, documents the config contract, adds focused tests, and includes redacted runtime proof.

## Reproduction Assessment

Yes for the review blockers by source inspection: configure a default template such as `{agentId}-{taskPreview}` and call `sessions_spawn` without `agentId`; the PR renders `{agentId}` before native or ACP target defaults are resolved. I did not run a live spawn.

## Solution Assessment

No, not as written. The maintainable fix should resolve the actual target first, apply per-agent template policy from the requester side, document the new public config, and add focused coverage for native and ACP paths.

Overall correctness: patch is incorrect

Overall confidence: 0.9

- **[P2] Render labels after resolving the target agent:** `src/agents/tools/sessions-spawn-tool.ts:227`
  - body: The fallback label is computed from raw `requestedAgentId` before native or ACP spawn resolves the actual target. A coordinator's configured label convention can be ignored, and a target agent can accidentally dictate the caller's label format.
  - confidence: 0.84

  - body: This adds an operator-facing config key, but the public subagent/config docs do not describe the default and per-agent paths, token behavior, or precedence. Users would not be able to discover or safely rely on the option after release.
