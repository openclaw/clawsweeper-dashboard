## Summary

Keep open: the feature is not on current main, but the PR is not merge-ready because it renders labels before target resolution, scopes the per-agent override to the target instead of the requester, lacks docs/runtime proof, and is currently conflicting.

## What This Changes

The PR adds optional `labelTemplate` config fields under global and per-agent subagent config and uses them to synthesize fallback `sessions_spawn` labels when `label` is omitted.

## Best Possible Solution

Land a target-aware, requester-scoped implementation with focused native and ACP tests, public docs for the config contract, a refreshed branch, and redacted real runtime proof showing generated labels.

## Reproduction Assessment

Yes, for the review blockers by source inspection: configure `labelTemplate` with `{agentId}` and call `sessions_spawn` without `agentId`; the PR substitutes the raw requested id before native or ACP defaults resolve the target. I did not run a live spawn.

## Solution Assessment

No, not as written. The maintainable solution should render labels after resolving the actual target, use requester-scoped per-agent policy, document the new config, and prove the runtime behavior.
