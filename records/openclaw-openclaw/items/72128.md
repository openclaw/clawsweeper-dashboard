## Summary

Keep open. Current main still lacks configurable spawned-session label templates, but this external feature PR is not ready to merge because the patch renders labels before target resolution, applies per-agent template policy to the target instead of the requester, lacks public docs, and still has no real after-fix behavior proof.

## What This Changes

Adds optional `agents.defaults.subagents.labelTemplate` and `agents.list[].subagents.labelTemplate` config fields and uses `{agentId}` / `{taskPreview}` to synthesize fallback `sessions_spawn` labels when `label` is omitted.

## Best Possible Solution

Land a target-aware implementation that renders labels after native/ACP target resolution, resolves templates from requester-scoped subagent config with defaults fallback, documents the config contract, and includes focused tests plus runtime proof.

## Reproduction Assessment

Yes, for the review blockers by source inspection. Configure `agents.defaults.subagents.labelTemplate` as `{agentId}-{taskPreview}` and call `sessions_spawn` without `agentId`; the patch substitutes an empty `{agentId}` before native or ACP target defaults run.

## Solution Assessment

No, not as written. The feature direction is plausible, but the patch should move label rendering to a target-aware point, resolve per-agent policy from the requester, and add docs before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Render labels after resolving the target agent:** `src/agents/tools/sessions-spawn-tool.ts:227`
  - body: The fallback label is computed from raw `requestedAgentId` before native or ACP spawn resolves the actual target. When `agentId` is omitted, native spawn defaults to the requester and ACP can use `acp.defaultAgent`, but `{agentId}` has already become empty, producing labels like `-Score all candidates`.
  - confidence: 0.91
- **[P2] Resolve labelTemplate from the requester agent:** `src/agents/tools/sessions-spawn-tool.ts:220-222`
  - body: This looks up `agents.list[].subagents.labelTemplate` by the requested child target, while existing subagent policy knobs such as `allowAgents` and `requireAgentId` are requester-scoped. A coordinator's per-agent label convention can be ignored, and the target can accidentally dictate the caller's label convention.
  - confidence: 0.84
- **[P3] Document the labelTemplate config:** `src/config/types.agent-defaults.ts:415-416`
  - body: This adds an operator-facing config key, but the public config docs do not describe the default or per-agent paths, token behavior, or precedence. Users would not be able to discover the option from the published docs after release.
  - confidence: 0.78
