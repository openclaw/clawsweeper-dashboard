## Summary

Keep open. The PR is a plausible fix for the linked label-template request and already addressed the duplicate-token `replaceAll` review, but the latest supplied diff still formats labels before native/ACP target resolution and leaves generated config schema, docs, changelog, and focused tests incomplete.

## What This Changes

The PR adds optional `labelTemplate` config under `agents.defaults.subagents` and `agents.list[].subagents`, then uses it to synthesize a `sessions_spawn` label when no explicit label is provided.

## Best Possible Solution

Land a narrowed version that renders the label template after resolving the effective native or ACP target agent, keeps explicit labels authoritative, documents the per-agent override semantics, regenerates config artifacts, adds a changelog entry, and covers default, explicit target, explicit label, duplicate-token, and ACP default-agent cases.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.labelTemplate` as `{agentId}-{taskPreview}` and call `sessions_spawn` from `agent:main` without `agentId`; the supplied PR diff computes the label before `spawnSubagentDirect` resolves the omitted target to `main`, so `{agentId}` becomes empty.

## Solution Assessment

No. The requested feature is reasonable, but the current patch is not the best implementation because label rendering happens too early and config artifacts/docs are incomplete; a small helper invoked after target resolution would be safer and easier to test.
