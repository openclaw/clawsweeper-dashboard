## Summary

Keep this PR open. The feature is not on current main, but the proposed implementation still has source-level correctness blockers around target resolution/requester-scoped config, lacks public docs for the new config key, and the external contributor has not supplied real runtime proof.

## What This Changes

The PR adds an optional `labelTemplate` setting under default and per-agent subagent config and uses it to generate fallback `sessions_spawn` labels when no explicit label is supplied.

## Best Possible Solution

Land a target-aware implementation that resolves the effective native or ACP target before rendering labels, applies per-agent policy from the requester, documents the config contract, adds focused coverage, and includes contributor-supplied runtime proof.

## Reproduction Assessment

Yes, for the PR review blockers by source inspection. Configure a default `{agentId}-{taskPreview}` template and call `sessions_spawn` without `agentId`; the PR renders `{agentId}` before native spawn defaults to the requester or ACP uses `acp.defaultAgent`.

## Solution Assessment

No, not as written. The feature direction is plausible, but the implementation should render after target resolution, use requester-scoped per-agent config, and document/test the public contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Render labels after resolving the target agent:** `src/agents/tools/sessions-spawn-tool.ts:227`
  - body: The fallback label is computed from raw `requestedAgentId` before native or ACP spawn resolves the actual target. When `agentId` is omitted, native spawn defaults to the requester and ACP can use `acp.defaultAgent`, but `{agentId}` has already become empty, producing labels like `-Score all candidates`.
  - confidence: 0.91
- **[P2] Resolve labelTemplate from the requester agent:** `src/agents/tools/sessions-spawn-tool.ts:220-222`
  - body: This looks up `agents.list[].subagents.labelTemplate` by the requested child target, while existing subagent policy knobs such as `allowAgents` and `requireAgentId` are requester-scoped. A coordinator's per-agent label convention can be ignored, and the target can accidentally dictate the caller's label convention.
  - confidence: 0.84
- **[P3] Document the labelTemplate config:** `src/config/types.agent-defaults.ts:415-416`
  - body: This adds an operator-facing config key, but the public subagent/config docs do not describe the default or per-agent paths, token behavior, or precedence. Users would not be able to discover the option from the published docs after release.
  - confidence: 0.78
