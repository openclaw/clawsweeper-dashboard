## What This Changes

The branch adds optional `labelTemplate` config under global and per-agent subagent config and uses it to generate a fallback `sessions_spawn` label when no explicit label is passed.

## Best Possible Solution

Land a target-aware, requester-scoped, documented `labelTemplate` implementation with focused native and ACP coverage plus redacted after-fix proof, or leave the feature out of core until that public config contract is decided.

## Reproduction Assessment

Yes by source inspection. Configure `agents.defaults.subagents.labelTemplate` as `{agentId}-{taskPreview}` and call `sessions_spawn` without `agentId`; the PR renders `{agentId}` before native requester fallback or ACP `defaultAgent` resolution, so the generated label can omit the actual target id.

## Solution Assessment

No. The current PR should resolve the final target first, read per-agent template policy from the requester agent, document the new public config contract, and prove the behavior in a real session list before merge.

  - body: The fallback label is rendered while only `requestedAgentId` is known. Native spawns without `agentId` later target the requester, and ACP spawns can later use `acp.defaultAgent`, so `{agentId}` becomes empty even though a concrete target is spawned; compute the fallback from the resolved target before passing it to native or ACP.

A coordinator's override can be ignored, and a target's template can apply to another caller; resolve the per-agent template from the requester instead.
  - confidence: 0.87
