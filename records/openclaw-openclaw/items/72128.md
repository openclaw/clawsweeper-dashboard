## Summary

Keep this PR open as the canonical implementation path for the closed label-template request, but it is not merge-ready: the fallback label is rendered before the real native/ACP target is known, per-agent config is read from the target instead of the requester, and the contributor still has no real behavior proof.

## What This Changes

The PR adds optional `labelTemplate` config under global and per-agent subagent settings and uses it to auto-generate a `sessions_spawn` label when the call omits `label`.

## Best Possible Solution

Land a target-aware, documented implementation that resolves fallback labels after the actual native or ACP target is known, defines requester-scoped per-agent precedence, and covers explicit/default/native/ACP cases with focused tests and redacted real behavior proof.

## Reproduction Assessment

Yes for the main implementation defect by source inspection: configure a template containing `{agentId}` and call `sessions_spawn` without `agentId`; the PR substitutes an empty token before native requester fallback or ACP `defaultAgent` resolution. I did not run a live spawn because this review was read-only.

## Solution Assessment

No. The requested feature is reasonable, but this implementation should move template rendering to a target-aware point in the native/ACP spawn flow and document/test the public config contract before release.
