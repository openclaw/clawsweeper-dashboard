## Summary

Keep this PR open: it is draft/protected and still has two source-reproducible correctness blockers at head 63131a40, while the proof in the body is test/static terminal output rather than representative runtime delivery through a migrated channel path.

## What This Changes

The PR adds a channel-message plugin SDK lifecycle surface, durable send-context delivery, message receipts and rendered batch plans, receive-ack primitives, plugin-loader fallback changes, docs/changelog, and migrations across bundled channels.

## Best Possible Solution

Align the durable capability map with the public lifecycle contract, pass the send-context durability policy through to outbound delivery, then merge only after maintainer SDK/channel review and representative runtime proof or an explicit proof override.

## Reproduction Assessment

Not applicable as a user bug, but yes for the review blockers: both are high-confidence source reproductions at PR head 63131a40 by reading the adapter capability type, durable capability source, send-context queue policy, and outbound queue contract.

## Solution Assessment

No. The lifecycle direction may be maintainable, but this patch should first make the public capability contracts match the advertised behavior, honor best-effort durability, and provide representative runtime proof or a maintainer override.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Expose side-effect durable capabilities on adapters:** `src/channels/plugins/outbound.types.ts:54-64`
  - body: The new durable capability source includes `reconcileUnknownSend`, `afterSendSuccess`, and `afterCommit`, and requirement checks validate requested capabilities against `outbound.deliveryCapabilities.durableFinal`. This adapter type omits those keys, so TypeScript channel adapters cannot declare support for the advertised side-effect capabilities.
  - confidence: 0.9
- **[P2] Honor best-effort send contexts:** `src/channels/message/send.ts:214`
  - body: `withDurableMessageSendContext` accepts `durability: "best_effort"` and exposes it on the context, but `ctx.send` always calls `deliverOutboundPayloads` with `queuePolicy: "required"`. A best-effort caller will still fail closed if queue persistence fails instead of falling back to direct send.
  - confidence: 0.88
