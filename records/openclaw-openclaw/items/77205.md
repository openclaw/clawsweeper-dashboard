## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Bootstrap the adapter before reconciliation:** `src/infra/outbound/delivery-queue-recovery.ts:149`
  - body: Recovery only calls `loadChannelMessageAdapter`, which reads the active registry. After a restart, a queued `unknown_after_send` entry can be processed before the bundled channel plugin is hydrated, so reconciliation is skipped and the entry is failed instead of using the channel reconciler.
  - confidence: 0.88
- **[P2] Record ack failures after sent reconciliation:** `src/infra/outbound/delivery-queue-recovery.ts:270-275`
  - body: When reconciliation reports `sent` but `ackDelivery` throws a non-`ENOENT` error, this returns `"failed"` without calling `failDelivery`. The queue entry keeps the same retry metadata, so repeated recovery can loop without backoff or max-retry cutoff.
  - confidence: 0.9
- **[P2] Preserve explicit null thread overrides:** `src/channels/turn/durable-delivery.ts:75-79`
  - body: `resolveThreadId` uses `??`, so a caller that passes `threadId: null` to intentionally clear thread targeting is forced back to `ctxPayload.MessageThreadId`. Durable delivery should preserve the explicit null the same way reply targeting does.
  - confidence: 0.92
- **[P2] Isolate send-failure lifecycle errors:** `src/infra/outbound/deliver.ts:265-271`
  - body: If the platform send fails and `afterSendFailure` also throws, the cleanup-hook error replaces the original send error. That obscures the real delivery failure and can misclassify retry/recovery diagnostics.
  - confidence: 0.87
- **[P2] Do not let fail hooks replace structured send failures:** `src/channels/message/send.ts:206-210`
  - body: `sendDurableMessageBatch` awaits `ctx.fail(result.error)` before returning the structured `{ status: "failed" }` result. If the failure hook throws, callers get the hook error instead of the delivery failure needed for retry and recovery handling.
  - confidence: 0.87
