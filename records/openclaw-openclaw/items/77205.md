## Summary

Keep open: this PR is not implemented on current main, it is a broad public SDK/channel lifecycle migration with protected human-review handling, and the latest head still has blocking lifecycle correctness issues plus insufficient real behavior proof for durable delivery.

## What This Changes

The PR adds a public `openclaw/plugin-sdk/channel-message` SDK facade and rewires durable outbound delivery, recovery, live preview finalization, receipts, docs, and bundled channel adapters onto that lifecycle.

## Best Possible Solution

Fix the lifecycle hook isolation and recovered-send commit semantics, then have a maintainer SDK/channel owner review the contract and add redacted runtime proof for at least one durable final-send or recovery path before merge.

## Reproduction Assessment

Yes for the PR findings by source inspection: a successful platform send followed by a throwing `afterSendSuccess` hook will enter the failure path, and a reconciled-sent recovery returns before commit lifecycle handling. This is not a current-main bug reproduction; the overall PR behavior still lacks live durable-delivery proof.

## Solution Assessment

No: the direction fits the plugin SDK/channel boundary, but this head is not ready because success-hook failures can cause duplicate sends and reconciled recovery skips commit lifecycle. The safer path is to isolate post-send hook failures from send outcome, run recovered commit semantics after ack, then require maintainer review plus real runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Isolate afterSendSuccess hook failures:** `src/infra/outbound/deliver.ts:254`
  - body: `afterSendSuccess` runs inside the same `try` as the platform send. If the hook throws after the message was already sent, the catch path treats the whole delivery as failed, invokes failure cleanup, and rethrows; queued or direct callers can then retry/recover an already-sent message and create duplicates. Post-send hook failures should be logged/isolated without changing the send result.
  - confidence: 0.89
- **[P2] Run commit hooks after reconciled sent recovery:** `src/infra/outbound/delivery-queue-recovery.ts:269-272`
  - body: When unknown-send reconciliation reports `sent`, this branch acks the queue entry and returns immediately. Unlike the replay branch below, it never runs the commit lifecycle, so adapters that rely on `afterCommit` cleanup after a crash can remain inconsistent even though recovery marks the delivery recovered.
  - confidence: 0.78
