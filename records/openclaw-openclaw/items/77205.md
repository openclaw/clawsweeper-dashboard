## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Bootstrap adapters before unknown-send reconciliation:** `src/infra/outbound/delivery-queue-recovery.ts:143-145`
  - body: Recovery now calls `loadChannelMessageAdapter` directly, but that loader only reads the active registry. On startup before plugin hydration, reconciliation is skipped and `unknown_after_send` entries are moved to `failed/`, which can permanently drop a delivery after a crash.
  - confidence: 0.93
- **[P2] Respect requested best-effort durability:** `src/channels/message/send.ts:132`
  - body: `withDurableMessageSendContext` accepts `durability: "best_effort"`, but `ctx.send` always passes `queuePolicy: "required"`. Best-effort callers will throw on queue-write failures instead of falling back to direct send.
  - confidence: 0.95
- **[P2] Run afterCommit hooks for non-queued sends:** `src/infra/outbound/deliver.ts:1178-1183`
  - body: When delivery proceeds without a queue entry, this path returns successful results without running `runOutboundDeliveryCommitHooks`. Adapters that use `afterCommit` for cleanup miss that hook on best-effort fallback and skip-queue sends.
  - confidence: 0.9
- **[P2] Preserve original send error when cleanup fails:** `src/infra/outbound/deliver.ts:265-269`
  - body: The lifecycle catch path awaits `afterSendFailure` directly, so a cleanup-hook error replaces the original platform send error. Durable retry and diagnostics should still report the real send failure.
  - confidence: 0.88
- **[P2] Preserve structured failed send results:** `src/channels/message/send.ts:208`
  - body: When `ctx.send` returns `{ status: "failed" }`, `sendDurableMessageBatch` awaits `ctx.fail(result.error)` and can reject with a failure-hook error instead of returning the structured failed result needed by retry and recovery callers.
  - confidence: 0.87
- **[P2] Re-export reply-prefix helpers from channel-message:** `src/plugin-sdk/channel-message.ts:21-25`
  - body: The new facade exposes `createReplyPrefixContext` but omits `createReplyPrefixOptions` and `createTypingCallbacks`. Plugin code migrating from the deprecated reply-pipeline subpath can hit missing-export errors for helpers that still exist in the reply core.
  - confidence: 0.9
