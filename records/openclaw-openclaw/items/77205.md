## What This Changes

This PR adds a `plugin-sdk/channel-message` lifecycle surface, durable send-context delivery, rendered batch plans and receipts, reply-pipeline facade migrations across bundled channels, Telegram watermark changes, plugin-loader fallback hardening, docs, changelog, and tests.

## Best Possible Solution

Align the lifecycle capability types and queue policy behavior with the advertised contract, keep focused regression coverage, then merge only after maintainer SDK/channel review and representative runtime proof or an explicit proof override.

## Reproduction Assessment

Not applicable as a user bug. For the PR blockers, yes: exact-head source inspection shows the capability map/type mismatch and the hard-coded required queue policy.

## Solution Assessment

No. The lifecycle direction looks plausible, but the patch should first expose the advertised side-effect capabilities on adapter declarations, honor best-effort durability, and provide representative runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Expose side-effect durable capabilities on adapters:** `src/channels/plugins/outbound.types.ts:54-64`
  - body: The new lifecycle surface lists and derives `reconcileUnknownSend`, `afterSendSuccess`, and `afterCommit`, and support checks validate requested capabilities against `outbound.deliveryCapabilities.durableFinal`. This adapter type omits those keys, so typed channel adapters cannot declare support for the advertised side-effect capabilities and any requirement for them will fail capability matching.
  - confidence: 0.91
- **[P2] Honor best-effort send contexts:** `src/channels/message/send.ts:214`
  - body: `withDurableMessageSendContext` accepts `durability: "best_effort"`, but `ctx.send` always forwards `queuePolicy: "required"` to `deliverOutboundPayloads`. Best-effort callers will still fail closed on queue persistence failure instead of falling back to direct send.
  - confidence: 0.92
