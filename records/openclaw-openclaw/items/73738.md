## Summary

Keep open: current main does not include the sandbox lifecycle feature, and this PR is the active path for the linked enhancement, but source review still shows media staging and package URL hardening blockers plus insufficient real behavior proof.

## What This Changes

The branch adds `workspaceLifecycle: persistent | ephemeral` sandbox config/docs/schema/runtime cleanup and tests, plus `package_url` download hardening and small gate-related fixture/Discord trust annotations.

## Best Possible Solution

Keep this PR as the implementation path after media staging uses the same run-scoped workspace, IPv4-mapped address blocking delegates to the IPv4 classifier, and maintainers accept the public sandbox config mode.

## Reproduction Assessment

Yes, source-reproducible for the blocking PR bugs: PR head resolves the actual sandbox workspace with `runId`, but attachment prestaging still resolves without one, and the new IPv6 blocker visibly omits IPv4-mapped link-local/private ranges. I did not run a live sandbox in this read-only review.

## Solution Assessment

No, not yet. The lifecycle shape is plausible, but the PR is not the narrow maintainable solution until staged media targets the run-scoped workspace and `package_url` address filtering covers IPv4-mapped blocked ranges.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Stage media in run-scoped sandbox workspaces:** `src/agents/sandbox/context.ts:429-433`
  - body: When `workspaceLifecycle` is `ephemeral`, `resolveSandboxContext()` derives the real workspace from `runId`, but `ensureSandboxWorkspaceForSession()` still has no run id and is what `chat.send` prestaging uses. Offloaded attachments are copied into the persistent per-scope workspace while the agent runs in a different run workspace, so documented `media/inbound/*` paths are missing for ephemeral runs.
  - confidence: 0.9
- **[P2] Block IPv4-mapped package URL ranges:** `scripts/resolve-openclaw-package-candidate.mjs:204-219`
  - body: `isBlockedIpv6()` only rejects a few `::ffff:` private ranges. An address like `::ffff:169.254.169.254` is treated as IPv6 by Node and passes the new DNS validation, leaving link-local metadata hosts reachable through `package_url`; map IPv4-mapped addresses through the IPv4 blocker or cover the same ranges.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR improves package URL fetching, but the new blocked-address check still misses IPv4-mapped link-local and related blocked IPv4 ranges.

Concerns:

- **[medium] IPv4-mapped link-local package URL bypass:** `scripts/resolve-openclaw-package-candidate.mjs:204`
  - body: `isBlockedIpv6()` does not pass `::ffff:` addresses through the IPv4 blocker, so a package URL host resolving to an IPv4-mapped link-local address can pass validation before tarball hash verification.
