## Summary

Keep this open: current main still drops Slack agent identity on the draft streaming path, and an open PR already uses closing syntax for this issue. The normal reply path has identity support, but the draft stream constructor/send call does not thread it through.

## What This Changes

The issue asks Slack streaming replies to preserve the configured agent username/icon identity, matching non-streaming Slack replies.

## Best Possible Solution

Land or refresh the linked Slack fix so draft preview streaming forwards `SlackSendIdentity`, with regression coverage, then decide whether native streaming should also pass identity parameters before closing the issue.

## Reproduction Assessment

Yes. Source inspection on current main shows `slackIdentity` is resolved in dispatch, but the draft stream API and initial send call have no identity field, so the reported draft streaming path is reproducible from code.

## Solution Assessment

Yes for the draft path: threading the existing `SlackSendIdentity` through `createSlackDraftStream` is the narrow maintainable fix. Native streaming should be separately verified against Slack SDK/API support before claiming the whole issue is fixed.
