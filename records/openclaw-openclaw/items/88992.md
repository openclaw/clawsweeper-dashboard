## Label Justifications

- `P1`: The linked failure is silent source-channel message loss in `message_tool_only` flows and a commenter reports the same pattern in production.
- `merge-risk: 🚨 compatibility`: The PR adds a new core `messages` config option and an opt-in behavior that changes how suppressed tool-only finals can be recovered.

- `merge-risk: 🚨 security-boundary`: The opt-in path replays private final text into a synthetic model prompt and can make it visible through a retry if the model sends it.

## Summary

Keep open for maintainer review: the patch is technically coherent and current main still only warns for the stranded-final case, but the branch adds a new opt-in message-delivery policy and replays private final text into a synthetic retry prompt, so the remaining blocker is product/privacy acceptance rather than a line-level fix.

## What This Changes

The branch adds a default-off `messages.strandedReplyRecovery` config that detects stranded `message_tool_only` finals and enqueues one non-batched follow-up retry via `message(action=send)`, plus prompt, docs, schema, tests, and qa-lab coverage.

## Best Possible Solution

If maintainers want this behavior, sponsor the default-off recovery policy explicitly and merge it with the documented privacy tradeoff; otherwise keep the current warning/default-private contract and close or narrow this PR.

## Reproduction Assessment

Yes. Current main has a checked-in qa-lab scenario and source path that reproduce no outbound plus a private-final warning when a long `message_tool_only` final omits `message(action=send)`.

## Solution Assessment

Unclear as a product decision. The implementation is a good bounded mitigation if maintainers approve opt-in recovery, but the policy and privacy boundary are not automatically settled by the code or tests.
