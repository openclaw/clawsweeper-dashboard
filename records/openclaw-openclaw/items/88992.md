## Summary

The patch has no discrete correctness defect in the reviewed diff and includes strong deterministic runtime proof, but it adds an opt-in core recovery mode that intentionally changes a privacy-sensitive message-delivery contract; the linked issue and prior review both identify this as a maintainer product decision, so the PR should remain open for explicit acceptance rather than automated cleanup.

## What This Changes

The PR adds a default-off `messages.strandedReplyRecovery` mode that detects substantive undelivered `message_tool_only` finals, schedules one isolated non-persistent follow-up asking the agent to send the original text through the message tool, strengthens prompt guidance, and adds docs, unit tests, and QA coverage.

## Best Possible Solution

If maintainers accept recovery in core, land the current default-off, one-shot design with non-persistence and collect isolation intact; otherwise retain warning-only behavior and close the linked product request rather than introducing a second delivery contract.

## Reproduction Assessment

Yes. Current main has a source-visible warning-only loss path, and the supplied deterministic QA scenario reproduces the stranded final and verifies one recovered message-tool delivery without a retry loop.

## Solution Assessment

Unclear as a product decision, but technically sound as implemented: the bounded default-off retry is safer than suppression bypass or automatic publication, while maintainers must decide whether any recovery mode belongs in the tool-only privacy contract.
