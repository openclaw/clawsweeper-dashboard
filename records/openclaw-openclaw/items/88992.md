## Label Justifications

- `P1`: The PR addresses a real user-facing silent message-loss workflow and also introduces a blocking retry-loop risk in that workflow.
- `merge-risk: 🚨 compatibility`: The diff changes behavior for existing `message_tool_only` setups that currently keep final text private unless the model calls the message tool.
- `merge-risk: 🚨 message-delivery`: The retry path can repeat indefinitely or change when a channel receives a previously suppressed final reply.
- `merge-risk: 🚨 security-boundary`: Tool-only delivery is a privacy boundary for private final text, and this PR can cause that text to be sent after a recovery prompt.

## Summary

Keep open: the PR targets a real message-loss class, but it is not merge-ready because the retry can requeue forever if the retry turn also misses the message tool, the policy change needs maintainer approval, and the proof is still mock-only.

## What This Changes

The PR changes `runReplyAgent` to skip room events, detect substantive private finals in `message_tool_only`, and enqueue a follow-up retry prompting `message(action=send)`, with focused unit coverage.

## Best Possible Solution

Land only a bounded, maintainer-approved recovery policy: make retries one-shot, keep explicit tool-only privacy cases scoped, cover the retry-failure path, and require live Telegram/Codex proof before merge.

## Reproduction Assessment

Yes, at source level: current main resolves Codex/configured tool-only source replies to suppressed final delivery while Codex exposes a final `last_agent_message`. I did not run live Telegram/Codex reproduction in this read-only review.

## Solution Assessment

No as submitted. A follow-up retry is safer than bypass delivery, but it needs a one-shot guard, scoped privacy semantics, and real transport proof before it is the best fix.
