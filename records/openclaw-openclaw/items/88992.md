## Summary

The implementation and proof are strong enough to keep this PR as the active fix for the linked message-loss issue, but merging requires explicit owner acceptance because the opt-in mode adds a core config contract and reprocesses previously private final text in an additional model turn.

## What This Changes

The PR adds a default-off `messages.strandedReplyRecovery` option that detects substantive unsent `message_tool_only` finals, enqueues one isolated non-persistent follow-up asking the model to send the captured text through `message(action=send)`, strengthens prompt guidance, and adds unit, queue, docs, and qa-lab coverage.

## Best Possible Solution

Keep the default-off, one-shot, queue-isolated retry rather than bypassing source suppression, but merge only after delivery/config owners explicitly approve and document enabled recovery as additional model processing with possible later delivery.

## Reproduction Assessment

Yes at source level: current main detects and warns about substantive unsent tool-only finals but deliberately performs no recovery, matching the reported user-visible silence. The PR also supplies a deterministic after-fix reproduction through qa-lab.

## Solution Assessment

Yes, conditionally: the default-off bounded message-tool retry is safer and more contract-preserving than direct fallback delivery, but making it a core config and replaying private text through another model turn requires explicit maintainer acceptance.
