## Label Justifications

- `merge-risk: 🚨 security-boundary`: The recovery path replays private final text into a synthetic follow-up prompt before possible source delivery, which is privacy-sensitive even with bounded default-off behavior.

## Summary

Keep open: current main only warns on substantive private finals in message-tool-only mode, while this PR adds an opt-in recovery path that is not yet implemented or superseded and needs maintainer acceptance of the privacy and delivery semantics before merge.

## What This Changes

Adds opt-in `messages.strandedReplyRecovery` handling that retries substantive stranded `message_tool_only` finals through a bounded follow-up `message(action=send)` path, with config, docs, prompt, queue, mock-provider, and QA scenario coverage.

## Best Possible Solution

Land only after maintainers explicitly accept the default-off recovery config and private-text replay semantics; otherwise keep warning-only behavior and the documented automatic-mode workaround.

## Solution Assessment

The implementation is a plausible bounded mitigation using existing queue and message-tool paths, but it is not a purely mechanical fix because maintainers must approve the new default-off config and private-text replay contract.
