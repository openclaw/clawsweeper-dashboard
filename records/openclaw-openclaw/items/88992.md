## Label Justifications

- `P1`: The PR addresses a real silent message-loss workflow reported in production and linked to the canonical open issue.
- `merge-risk: 🚨 compatibility`: The PR adds a new `messages` config option and a new opt-in behavior for existing tool-only delivery setups.

- `merge-risk: 🚨 security-boundary`: Opt-in recovery replays private final assistant text into a synthetic prompt and relies on the model to publish it only through the message tool.

## Summary

Keep open: the recovery approach is still relevant to the linked message-loss issue, but the new qa-lab recovery scenario is not self-gated to `mock-openai`, so it can skip the request-level proof that the first turn stranded and the retry used `message(action=send)`. Maintainer acceptance is also still needed for the new default-off config/privacy behavior.

## What This Changes

The PR adds opt-in `messages.strandedReplyRecovery` handling that retries stranded `message_tool_only` finals through a bounded follow-up turn, with prompt, queue, config, docs, tests, mock-openai, and qa-lab scenario updates.

## Best Possible Solution

Keep the default-off recovery shape under maintainer review, but first make the qa-lab scenario self-gate to `mock-openai` so CI proves the seeded stranded-final/retry path instead of relying on nondeterministic live-model behavior.

## Reproduction Assessment

Yes for the underlying report at source level: current main has a checked-in mock-openai scenario that forces a long `message_tool_only` final with no message tool call and verifies no outbound plus a private-final warning. I did not execute tests because this review is read-only.

## Solution Assessment

Not fully yet. The opt-in retry is a plausible bounded fix that preserves the default contract, but this PR still needs a deterministic scenario guard and maintainer acceptance for the new config/privacy semantics.
