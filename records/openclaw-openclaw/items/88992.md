## Label Justifications

- `P1`: The PR addresses a silent message-loss class in real channel workflows where an agent finishes but no user-visible reply is delivered.

- `merge-risk: 🚨 security-boundary`: The recovery path handles private final text and must preserve the no-bypass and no-persistence guarantees around `message_tool_only` output.

## Summary

Keep open: this is a coherent opt-in mitigation that is not already on current main, but it adds config-controlled recovery across a message-delivery and privacy boundary, so it belongs on the maintainer review path rather than cleanup close.

## What This Changes

Adds an opt-in `messages.strandedReplyRecovery` mode that warns on stranded `message_tool_only` finals and enqueues one follow-up retry to deliver via `message(action=send)`, with queue, persistence, docs, schema, and QA coverage.

## Best Possible Solution

Merge only after maintainers explicitly accept the opt-in recovery policy and either accept the qa-channel proof as representative or require a redacted live-transport proof for a recovered direct reply.

## Reproduction Assessment

Yes, source-reproducible: current main already has a warning-only `message_tool_only` stranded-final path, and the PR adds a deterministic qa-lab scenario for the stranded final plus one retry delivery. I did not run the scenario in this read-only review.

## Solution Assessment

Unclear. The one-shot follow-up retry is a careful mitigation that preserves the privacy contract by default, but the new config surface and default-off behavior need maintainer product approval before calling it the best fix.
