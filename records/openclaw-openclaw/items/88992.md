## Summary

Keep open. Current main still only warns and suppresses stranded `message_tool_only` finals, while this PR adds a plausible default-off recovery path with strong qa-lab proof; the remaining blocker is maintainer acceptance of the new config and private-text retry boundary.

## What This Changes

The PR adds a default-off `messages.strandedReplyRecovery` option that detects stranded `message_tool_only` finals, enqueues one non-batched follow-up retry asking the model to send via `message(action=send)`, strengthens prompt/docs/config metadata, and adds unit plus qa-lab scenario coverage.

## Best Possible Solution

Land this only after maintainers explicitly accept the default-off core config and private-text retry boundary; otherwise keep the warning-only behavior and resolve the linked issue with a different policy.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: `message_tool_only` finals without a message tool call stay private, warn only, and the existing QA scenario asserts no outbound plus the private-final warning.

## Solution Assessment

Unclear until maintainer policy acceptance. The default-off retry is safer than auto-publishing because it preserves the default contract and uses `message(action=send)`, but the new core config and private-text replay are product/security decisions.
