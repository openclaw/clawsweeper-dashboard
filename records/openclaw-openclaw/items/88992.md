## Summary

## What This Changes

The PR adds an opt-in `messages.strandedReplyRecovery` mode that detects substantive stranded `message_tool_only` final text, queues one follow-up retry to deliver via `message(action=send)`, updates prompt/config/docs, and adds unit plus qa-lab coverage.

## Best Possible Solution

Land only if maintainers explicitly accept the default-off recovery policy and private-text replay boundary; otherwise keep warning-only behavior as the supported outcome for the linked issue.

## Reproduction Assessment

Yes. Current main has a checked-in qa-lab scenario for the warning-only path, and source inspection shows substantive `message_tool_only` finals without a message send warn but stay private.

## Solution Assessment

Yes as an implementation shape if maintainers accept the opt-in policy: the branch preserves default behavior, adds a bounded retry, suppresses persistence, and still routes delivery through `message(action=send)`. The unresolved question is whether that recovery mode should exist in core.
