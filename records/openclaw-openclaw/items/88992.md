## Summary

Keep open for maintainer review: current main still only warns on stranded `message_tool_only` finals, and this PR's implementation looks mechanically sound but adds default-off recovery config plus privacy-sensitive delivery semantics that need maintainer acceptance before merge.

## What This Changes

Adds a default-off `messages.strandedReplyRecovery` option that enqueues one isolated follow-up retry for substantive stranded tool-only finals and covers the path with schema, docs, unit tests, and a qa-lab scenario.

## Best Possible Solution

Land the default-off retry only if maintainers accept the opt-in privacy and delivery semantics; otherwise keep current warning-only behavior and use the documented automatic visible-reply mode for affected channels.

## Reproduction Assessment

Yes. Current main source shows warning-only suppression with no recovery, the linked issue has production Telegram/Nextcloud evidence, and the PR adds a deterministic qa-lab scenario for the stranded-final plus one recovered retry path.

## Solution Assessment

Acceptable mitigation, not a purely mechanical bug fix. The PR avoids auto-publishing, stays default-off, bounds retries, suppresses persistence, and uses the existing follow-up queue, but the config/privacy choice remains a maintainer decision.
