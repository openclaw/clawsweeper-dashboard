## Summary

Keep open: current main and the latest release still lack a plugin-facing way for post-turn, engine-owned compactions to update the persisted status counter. The merged related Codex logging PR is adjacent but explicitly did not change compaction count behavior, and the closed duplicate points back to this issue as the canonical tracker.

## What This Changes

Request for plugin-owned context engines to report successful post-turn compactions so `/status` and `session_status` no longer show zero when compaction happened outside the built-in runtime path.

## Best Possible Solution

Add an additive, plugin-facing successful-compaction reporting mechanism that persists per-session deltas into the existing session store, then cover `/status`, `session_status`, and plugin-engine lifecycle tests without counting every `afterTurn` invocation.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: an `ownsCompaction` engine can run successful background compaction from `afterTurn`, while the host awaits a `Promise<void>` and `/status` reads only persisted `SessionEntry.compactionCount`. I did not run a live third-party engine in this read-only review.

## Solution Assessment

Unclear pending maintainer API direction: the issue should stay open, but the best fix is an explicit successful-compaction signal or idempotent stats delta rather than inferring counts from hooks or every `afterTurn` call.
