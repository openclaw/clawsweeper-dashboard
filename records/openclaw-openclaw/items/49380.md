## Label Justifications

- `P2`: The issue affects a visible session/context accounting metric for plugin-engine users, but the core agent runtime remains usable.
- `impact:session-state`: The reported mismatch is session-state drift between actual context-engine compactions and the persisted status counter.

## Summary

Keep open. Current main and the latest release still have the accounting gap: plugin `afterTurn()` can perform proactive compaction but has no count notifier/result path, while `/status` renders only persisted session `compactionCount`; the duplicate request at https://github.com/openclaw/openclaw/issues/60332 is already closed in favor of this canonical issue.

## What This Changes

The issue asks OpenClaw to let plugin-owned context engine compactions increment or otherwise appear in the `/status` and `session_status` compaction count.

## Best Possible Solution

Choose a backward-compatible context-engine reporting contract that lets plugin-owned compactions update session status, then thread it through the harness and session-store paths with focused coverage for plugin `afterTurn()` compactions and existing `/compact` behavior.

## Reproduction Assessment

Yes, by source inspection. A plugin-owned engine can perform proactive compaction in `afterTurn()`, but current main provides no count notifier/result path back to the persisted session store that `/status` reads.

## Solution Assessment

Unclear as an implementation choice. The issue correctly identifies the gap, but the best fix requires maintainers to choose the stable plugin API shape before code changes.
