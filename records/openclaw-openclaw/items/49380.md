## Summary

Keep open: current main still lacks a supported compaction-delta signal from plugin-owned `afterTurn` or maintenance work into `SessionEntry.compactionCount`, so `/status` and `session_status` can still disagree with actual plugin compactions. The report is valid, but the fix needs a maintainer API and counting-semantics decision before automation should change the public ContextEngine contract.

## What This Changes

The issue requests a ContextEngine compaction accounting path so plugin-owned compactions increment the count shown by `/status` and `session_status`.

## Best Possible Solution

Add a documented, backward-compatible ContextEngine accounting signal that persists successful plugin-owned compaction deltas into `SessionEntry.compactionCount` for both `/status` and `session_status`, with idempotent final-turn and mid-loop coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: an `ownsCompaction` engine can compact from `afterTurn` or maintenance, but current main provides no compaction-delta signal while `/status` and `session_status` read only `SessionEntry.compactionCount`.

## Solution Assessment

Unclear. The suggested callback is plausible, but the best fix needs maintainer selection of the public SDK shape and idempotent counting semantics across post-turn, mid-loop, maintenance, `/compact`, and overflow paths.
