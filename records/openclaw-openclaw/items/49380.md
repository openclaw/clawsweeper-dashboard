## Summary

Keep open: current main and the latest shipped release still have the accounting gap where plugin-owned compactions can happen from context-engine lifecycle work without any supported way to update the persisted counter used by `/status` and `session_status`.

## What This Changes

The issue requests a way for plugin-owned context-engine compactions to update the compaction count shown by `/status` and `session_status`.

## Best Possible Solution

Add a documented, backward-compatible ContextEngine accounting signal that persists successful plugin-owned compaction deltas into `SessionEntry.compactionCount` for both status surfaces, with idempotency coverage for final-turn and mid-loop paths.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: an `ownsCompaction` engine can compact from `afterTurn`, but current main gives that path no compaction-delta signal while `/status` and `session_status` read only `SessionEntry.compactionCount`.

## Solution Assessment

Unclear. The suggested callback is plausible, but the best fix needs maintainer selection of the public ContextEngine accounting shape and idempotent counting semantics rather than a one-off store update.
