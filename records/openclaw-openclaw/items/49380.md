## Label Justifications

- `P2`: The issue is a normal-priority correctness gap in operator-facing status reporting for plugin-owned compaction, with limited runtime blast radius.
- `impact:session-state`: The reported failure is drift between actual context-engine compactions and the persisted session counter shown to operators.

## Summary

Keep open. Current main still lacks a way for plugin-owned proactive compactions from `afterTurn` to update the persisted counter that `/status` and `session_status` display; the closed duplicate points here as the canonical tracker, and the related open PR is not merged and does not close this issue.

## What This Changes

Request that plugin-owned context-engine compactions update the persisted compaction counter used by `/status` and `session_status`.

## Best Possible Solution

Add a documented, backward-compatible context-engine accounting signal that records successful engine-owned compaction deltas into `SessionEntry.compactionCount` for both `/status` and `session_status`, with idempotency tests.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: an `ownsCompaction` engine can compact from `afterTurn`, but current main gives that hook no return value or callback that can update the persisted counter read by `/status` and `session_status`.

## Solution Assessment

Unclear. The suggested callback is plausible, but the best fix needs maintainer choice of public SDK shape and idempotent counting semantics rather than a one-off store update.
