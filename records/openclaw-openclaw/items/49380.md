- `impact:session-state`: The issue is about drift between actual context-engine compactions and the persisted session counter shown to operators.

## Summary

Keep open: current main still lacks a plugin-facing way for `afterTurn` or maintenance-driven compactions to update the persisted counter used by `/status` and `session_status`; https://github.com/openclaw/openclaw/issues/60332 was already closed into this canonical tracker, and the adjacent open PRs do not close this issue.

## What This Changes

The issue asks for plugin-owned context-engine compactions to increment the session compaction count shown by `/status` and `session_status`.

## Best Possible Solution

Add a documented, backward-compatible ContextEngine accounting signal that records successful plugin-owned compaction deltas into `SessionEntry.compactionCount` for both `/status` and `session_status`, with idempotency coverage for final-turn and mid-loop paths.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: an `ownsCompaction` engine can compact from `afterTurn`, but current main gives that hook no return value or callback that can update the persisted counter read by `/status` and `session_status`.

## Solution Assessment

Unclear. The suggested callback is plausible, but the best fix needs maintainer selection of the public SDK shape and idempotent counting semantics rather than a one-off store update.
