## Summary

Keep open: current main still has a source-level accounting gap where plugin-owned context-engine compactions can run after a turn without any persisted session counter update for `/status` or `session_status`.

## What This Changes

The issue asks for plugin-owned context-engine compactions to update the `/status` and `session_status` compaction counter.

## Best Possible Solution

Add a maintainer-approved plugin-facing compaction accounting contract that records successful engine-owned compaction deltas into the existing session store for both `/status` and `session_status`.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: an `ownsCompaction` engine can compact in `afterTurn`, but `afterTurn` returns `Promise<void>` and the status surfaces only read persisted `SessionEntry.compactionCount`.

## Solution Assessment

Unclear. The issue’s proposed callback is plausible, but the best fix needs a maintainer decision on the public plugin contract and idempotent counting semantics before implementation.
