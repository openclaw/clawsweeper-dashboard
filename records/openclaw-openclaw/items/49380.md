## Summary

Keep open. Current main still lacks a way for plugin-owned proactive compactions from `afterTurn` or maintenance to update the persisted counter read by `/status` and `session_status`, and the duplicate status-count request was closed to keep this issue as the canonical tracker.

## What This Changes

The issue requests an accounting path so plugin context engines can report successful compactions into the `/status` and `session_status` compaction count.

## Best Possible Solution

Add a documented, backward-compatible context-engine accounting signal that persists successful engine-owned compaction deltas into `SessionEntry.compactionCount` for both status surfaces, with clear idempotency semantics.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: an `ownsCompaction` engine can compact inside `afterTurn`, while current main gives that hook no return value or callback and both status surfaces read only persisted `SessionEntry.compactionCount`.

## Solution Assessment

Unclear. The suggested callback is plausible, but the best fix requires maintainer ownership of the public SDK shape and counting semantics rather than a one-off automated patch.
