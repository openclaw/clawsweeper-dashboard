## Summary

Keep open. Current main and the latest release still have the same accounting gap: plugin `afterTurn()` can run proactive compaction, but there is no count notifier/result path into the persisted session entry that `/status` and `session_status` render.

## What This Changes

Expose a way for plugin-owned context-engine compactions from `afterTurn()` to update the `/status` and `session_status` compaction count.

## Best Possible Solution

Choose a backward-compatible context-engine compaction reporting contract and thread successful plugin-owned compaction deltas into the existing persisted session-status counter with focused tests and docs.

## Reproduction Assessment

Yes, by source inspection. A plugin-owned engine can perform proactive compaction in `afterTurn()`, but current main provides no count notifier/result path back to the persisted session store that `/status` reads.

## Solution Assessment

Unclear as an implementation choice. The issue correctly identifies the gap, but the best fix requires maintainers to choose the stable plugin API shape before code changes.
