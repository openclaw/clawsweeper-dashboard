## Label Justifications

- `P2`: This is a normal-priority operator-facing correctness gap in status reporting for plugin-owned compaction, with limited runtime blast radius.
- `impact:session-state`: The issue is about drift between actual context-engine compactions and the persisted session counter shown to operators.

## Summary

Keep open: current main and v2026.5.22 still have no supported signal for plugin-owned afterTurn/background compactions to update the persisted counter used by `/status` and `session_status`. Existing related fixes cover streamed/embedded compaction metadata and Codex event completion, not plugin leaf compactions initiated from `afterTurn`.

## What This Changes

Plugin-owned ContextEngine compactions performed from afterTurn/background lifecycle work should be reflected in the persisted compaction count shown by `/status` and `session_status`.

## Best Possible Solution

Add a documented, backward-compatible ContextEngine compaction-delta signal that persists successful plugin-owned compaction deltas into `SessionEntry.compactionCount` for both `/status` and `session_status`, with idempotency coverage across lifecycle paths.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: an `ownsCompaction` engine can compact from `afterTurn`, but current main gives that path no compaction-delta signal while `/status` and `session_status` read only `SessionEntry.compactionCount`.

## Solution Assessment

Unclear. The callback proposal is plausible, but the best fix needs maintainer selection of the public SDK/API shape and idempotent counting semantics rather than a one-off session-store write.
