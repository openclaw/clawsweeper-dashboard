## Summary

Keep open: current main still has the source-level accounting gap where plugin-owned compactions can run after a turn without any supported way to update the persisted counter used by `/status` and `session_status`.

## What This Changes

Plugin context engines that perform their own compaction should be able to increment the compaction count shown by `/status` and `session_status`.

## Best Possible Solution

Add a documented, backward-compatible ContextEngine accounting signal that persists successful plugin-owned compaction deltas into `SessionEntry.compactionCount` for both `/status` and `session_status`, with idempotency coverage for final-turn and mid-loop paths.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: an `ownsCompaction` engine can compact from `afterTurn` or maintenance, but current main gives those paths no compaction-delta signal while `/status` and `session_status` read only `SessionEntry.compactionCount`.

## Solution Assessment

Unclear. The suggested callback is plausible, but the best fix needs maintainer selection of the public SDK shape and idempotent counting semantics rather than a one-off store update.
