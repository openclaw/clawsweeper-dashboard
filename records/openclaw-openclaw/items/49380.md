## Summary

Keep open. Current main still has no host-visible way for plugin-owned proactive compactions from `afterTurn()` or maintenance to update the session-store count that `/status` renders, and the overlapping closed items do not leave an open canonical replacement for this specific SDK/status gap.

## What This Changes

Plugin context-engine sessions should be able to surface compactions they perform during turn finalization in `/status` and `session_status`.

## Best Possible Solution

Define an additive context-engine compaction reporting contract, wire successful plugin-reported proactive compactions into `SessionEntry.compactionCount`, and cover `/status`/`session_status` plus SDK docs/tests.

## Reproduction Assessment

Yes, source-reproducible. A plugin engine can compact during `afterTurn()` or maintenance, but current main only renders the session-store count and those plugin lifecycle paths have no count-reporting channel into that store.

## Solution Assessment

Unclear. The callback direction is plausible, but because this changes a public plugin SDK contract, maintainers should choose the additive API shape before implementation.
