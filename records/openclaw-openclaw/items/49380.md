## Label Justifications

- `P2`: The issue is a normal-priority correctness gap in status reporting for plugin-owned compaction, with limited runtime blast radius.
- `impact:session-state`: The reported failure is session state drift between actual context-engine compactions and the persisted counter shown to operators.

## Summary

Keep open. Current main still has no accounting path from plugin-owned proactive compactions in `afterTurn` into the persisted session counter used by `/status` and `session_status`; the related duplicate was closed in favor of this canonical tracker.

## What This Changes

Plugin context engines that own compaction need a way to report successful proactive compactions so status surfaces do not stay at zero.

## Best Possible Solution

Add a documented, backward-compatible context-engine accounting signal that records successful engine-owned compaction deltas into `SessionEntry.compactionCount` for both `/status` and `session_status`.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: an `ownsCompaction` engine can compact from `afterTurn`, but current main gives that hook no return value or callback that can update the persisted counter read by `/status` and `session_status`.

## Solution Assessment

Unclear. The suggested callback is plausible, but the best fix requires maintainer choice of public SDK shape and idempotent counting semantics rather than an automated one-off patch.
