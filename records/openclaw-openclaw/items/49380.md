## Summary

Keep this issue open. Current main still has the reported visibility gap: status surfaces render the persisted session-store `compactionCount`, while plugin-owned `afterTurn` and maintenance paths still have no generic way to report successful leaf or proactive compactions. The closed #60332 duplicate was explicitly closed in favor of this older, fuller tracker.

## What This Changes

Request that plugin context engines with `ownsCompaction: true` can report successful leaf or proactive compactions so `/status` and `session_status` show an accurate count.

## Best Possible Solution

Add a generic, additive context-engine reporting seam, preferably a typed `notifyCompaction(count?)` or equivalent on runtime context for post-turn and maintenance work, and persist reported deltas into the existing session-store counter used by both status surfaces.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure an `ownsCompaction` engine that compacts during `afterTurn` or maintenance, then observe that the only persisted status count comes from runtime metadata or manual compact paths, not that plugin work.

## Solution Assessment

Yes. A generic plugin-facing compaction signal is the narrow maintainable fix; querying mem-engine or lossless-claw storage directly would cross the extension-agnostic core boundary.

## Repair Work Prompt

Implement a focused fix for openclaw/openclaw#49380: plugin context engines with `ownsCompaction: true` can perform leaf/proactive compactions during `afterTurn` or turn maintenance, but `/status` and `session_status` still show `Compactions: 0` because core only persists built-in/manual/runtime metadata counts.

Expected boundary:
- Add an additive, backwards-compatible plugin-facing reporting seam, preferably a typed `notifyCompaction(count?: number)` or equivalent callback on `ContextEngineRuntimeContext`, available to `afterTurn`, the owning-engine loop hook, and turn maintenance where compaction can happen.
- Persist plugin-reported successful compaction deltas into the session-store `compactionCount` consumed by `src/status/status-message.ts` and `session_status`.
- Cover foreground `afterTurn`; if background/deferred maintenance can compact after run persistence, add a store-safe update path for that case too.
- Do not query plugin-specific databases, hard-code `lossless-claw`, `mem-engine`, or other plugin ids, or move extension-owned compaction policy into core.
- Avoid double-counting built-in auto-compaction, manual `/compact`, CLI compaction, timeout recovery, and overflow recovery paths that already increment or propagate counts.
- Update context-engine/plugin docs and SDK API baseline if the public type surface changes. Add a changelog entry if the final change is user-facing; credit the human reporter/contributors per repo policy and do not credit Codex/OpenClaw/steipete.
