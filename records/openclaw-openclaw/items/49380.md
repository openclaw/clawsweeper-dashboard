## Summary

Keep open: current main and the latest release still lack a supported way for plugin-owned post-turn/background compactions to update the persisted counter used by `/status` and `session_status`. The duplicate report at https://github.com/openclaw/openclaw/issues/60332 was already closed into this issue, while https://github.com/openclaw/openclaw/pull/42629 only covered embedded-result metadata accounting, not compactions initiated from `afterTurn`.

## What This Changes

The issue asks OpenClaw to let plugin context engines report successful proactive compactions so `/status` and `session_status` do not stay at zero.

## Best Possible Solution

Add a documented, backward-compatible ContextEngine accounting signal that persists successful plugin-owned compaction deltas into `SessionEntry.compactionCount` for both `/status` and `session_status`, with idempotency coverage across all compaction paths.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: an `ownsCompaction` engine can trigger background compaction from `afterTurn`, but current main gives that path no compaction-delta signal while `/status` and `session_status` read only `SessionEntry.compactionCount`.

## Solution Assessment

Unclear. The suggested callback is plausible, but the best fix needs maintainer selection of the public SDK/API shape and idempotent counting semantics rather than a one-off session-store write.
