## Summary

Keep open: current main and the latest shipped release still have the source-level accounting gap where plugin-owned post-turn compactions can happen without a supported way to update the persisted counter used by `/status` and `session_status`. The related duplicate at https://github.com/openclaw/openclaw/issues/60332 was already closed into this issue, while the earlier merged fix at https://github.com/openclaw/openclaw/pull/42629 covered embedded-result metadata accounting rather than leaf/background compactions from `afterTurn`.

## What This Changes

Plugin-owned context-engine compactions should be able to update the persisted compaction counter shown by `/status` and `session_status`.

## Best Possible Solution

Add a documented, backward-compatible ContextEngine accounting signal that persists successful plugin-owned compaction deltas into `SessionEntry.compactionCount` for both `/status` and `session_status`, with idempotency coverage for final-turn, mid-loop, manual, and deferred-maintenance paths.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: an `ownsCompaction` engine can trigger background compaction from `afterTurn` or maintenance, but current main gives those paths no compaction-delta signal while `/status` and `session_status` read only `SessionEntry.compactionCount`.

## Solution Assessment

Unclear. The suggested callback is plausible, but the best fix needs maintainer selection of the public SDK shape and idempotent counting semantics rather than a one-off store update.
