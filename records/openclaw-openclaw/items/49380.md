## Summary

Keep open: current main and v2026.5.22 still lack a supported signal for plugin-owned `afterTurn` or background compactions to update the persisted counter used by `/status` and `session_status`. The duplicate https://github.com/openclaw/openclaw/issues/60332 was closed into this as the canonical tracker, and the adjacent https://github.com/openclaw/openclaw/issues/83932 covers Codex runtime observability rather than plugin leaf compaction accounting.

## What This Changes

Request to let plugin-owned ContextEngine `afterTurn` and background compactions update the persisted compaction counter shown by `/status` and `session_status`.

## Best Possible Solution

Add a documented, backward-compatible ContextEngine compaction-delta signal that persists successful plugin-owned compaction deltas into `SessionEntry.compactionCount` for both `/status` and `session_status`, with idempotency coverage across lifecycle paths.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: an `ownsCompaction` engine can compact from `afterTurn`, but current main gives that path no compaction-delta signal while `/status` and `session_status` read only `SessionEntry.compactionCount`.

## Solution Assessment

Unclear. The callback proposal is plausible, but the best fix needs maintainer selection of the public SDK/API shape and idempotent counting semantics rather than a one-off session-store write.
