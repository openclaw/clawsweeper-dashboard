## Summary

Keep open: the PR is aimed at the right security boundary and fixes the earlier WhatsApp predicate direction, but the MCP loopback path still collapses route provider and requester source while exposing sender identity, so cross-source MCP tool calls can be trusted as same-source requester context. The intentional fail-closed Discord compatibility change also still needs maintainer sign-off before merge.

## What This Changes

The branch carries requester identity through Discord/WhatsApp message actions, Gateway `message.action`, CLI/MCP loopback, and related tests so privileged Discord actions require trusted sender context.

## Best Possible Solution

Land a version that preserves separate route provider and requester-source identity end-to-end through CLI, MCP, Gateway, and plugin action dispatch, with tests proving cross-source privileged actions fail closed and same-source/local-owner actions still work.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: a sender-scoped MCP run with a routeable `messageChannel` and different `messageProvider` exposes `senderId` to message tools without preserving requester source. I did not execute that failing path in this read-only review.

## Solution Assessment

No; this is not yet the best fix because the direct tool path separates route and source, but the MCP path still collapses them. The maintainable fix is to bind and forward requester source separately through MCP before trusting sender identity.
