## Summary

The PostgreSQL/pgvector backend is a useful optional memory backend, but current main already has a memory-plugin slot and runtime contract for this kind of storage integration, while the built-in memory-core config remains intentionally SQLite-only. This OpenClaw core request can close in favor of a ClawHub memory plugin, with a narrower SDK/API issue only if plugin implementation exposes a concrete missing extension point.

## What This Changes

The issue requests adding PostgreSQL/pgvector as a supported memory search store driver alongside SQLite.

## Best Possible Solution

Publish PostgreSQL/pgvector as a ClawHub memory plugin that owns `plugins.slots.memory`; open a focused OpenClaw SDK/API issue only if that plugin hits a concrete missing memory-runtime capability.

## Reproduction Assessment

Not applicable; this is a feature request for a new storage backend, not a report of broken current behavior. Source inspection confirms the current built-in config is SQLite-only and the plugin memory path is available.

## Solution Assessment

Yes, closing toward ClawHub is the best fit for this request. A PostgreSQL/pgvector backend is optional deployment-specific storage, and the existing memory plugin slot/runtime contract is the maintainable extension boundary.
