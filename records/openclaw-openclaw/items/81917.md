## Summary

Keep open. Current main and v2026.5.12 still log the clean dashboard URL while sending the tokenized URL only to clipboard/browser, but changing the visible log to `#token=` conflicts with the documented no-token-in-logs security contract; the Linux/KDE browser-launch hang remains potentially actionable but is not yet reproduced.

## What This Changes

Requests changing `openclaw dashboard` fallback output so token auto-auth is usable from the printed URL and investigating a Linux/KDE browser-launch hang.

## Best Possible Solution

Keep dashboard logs free of token fragments, improve any non-secret fallback wording if maintainers want a clearer operator path, and separately reproduce the KDE launcher lifetime before changing browser-open process behavior.

## Reproduction Assessment

No, not for the full issue. Source inspection confirms the clean printed URL and manual-auth fallback path, but the Linux/KDE hang was not reproduced and the requested tokenized terminal output conflicts with current tests and docs.

## Solution Assessment

No. Logging `dashboardUrl` is not the best fix as proposed because it would put `#token=` into runtime logs; the safer route is a non-secret UX adjustment plus a focused browser-launch fix after real KDE reproduction.
