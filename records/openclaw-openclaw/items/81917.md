## Summary

Keep open. Current main intentionally keeps terminal dashboard output non-tokenized while clipboard/browser receive the tokenized URL, and the requested log-line change conflicts with the documented CVE-safe no-token-in-logs contract; the Linux/KDE browser-launch hang remains a plausible but unverified separate runtime issue.

## What This Changes

The issue requests that `openclaw dashboard` print a tokenized manual fallback URL when token auto-auth is available and stop hanging during Linux/KDE browser launch.

## Best Possible Solution

Preserve token redaction in runtime logs, then investigate the Linux/KDE launcher lifetime with focused `openUrl` coverage or real desktop proof before changing process-spawn behavior.

## Reproduction Assessment

No, not for the full issue. Source inspection gives a high-confidence path for the bare logged URL versus tokenized clipboard/browser URL, but the Linux/KDE hang needs real environment reproduction or a focused process-level test.

## Solution Assessment

No, printing the tokenized URL as requested is not the best fix because it conflicts with current security tests and docs. The safer direction is redacted recovery UX plus a separate browser-launch hang investigation.
