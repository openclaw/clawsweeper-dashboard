## Summary

Keep open: current main has gateway health checks and reusable channel probe surfaces, but classic `openclaw onboard` still configures channels and writes the config without an interactive per-channel reachability preflight. This is a new onboarding UX feature, not a broken existing contract.

## What This Changes

Add an interactive onboarding preflight that validates configured channel reachability before setup finalizes.

## Best Possible Solution

Add a generic onboarding channel preflight design that reuses plugin `probeAccount`/`auditAccount` and gateway reachability plumbing, with clear skip/opt-in behavior and no provider-specific Telegram or Discord logic in core.

## Reproduction Assessment

Not applicable; this is a feature request, not a report of a failing current contract. Source and docs show current diagnostics are available through gateway health and `channels status --probe`, but not as an interactive onboarding preflight.

## Solution Assessment

Unclear; the requested direction is reasonable, but maintainers should decide the UX/API boundary before implementation. Reusing existing probe contracts is likely the narrowest technical path if the feature belongs in onboarding.
