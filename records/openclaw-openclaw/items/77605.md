## Summary

Keep open: current main has post-onboarding channel probe/status surfaces, but `openclaw onboard` still does not offer an interactive or flag-driven per-channel reachability preflight. This is a new onboarding UX/API feature that needs maintainer direction, not a broken existing contract or a closeable duplicate.

## What This Changes

Request to add an interactive or flag-driven `openclaw onboard` connection preflight that validates channel credentials and gateway reachability before writing config.

## Best Possible Solution

Add a generic opt-in onboarding preflight that reuses plugin status probe contracts and gateway reachability checks, with clear skip behavior and credential handling decided before implementation.

## Reproduction Assessment

Not applicable; this is a feature request rather than a failing current contract. Source and docs show existing live diagnostics through `openclaw channels status --probe`, `openclaw status --deep`, and `openclaw health --verbose`, but not an onboarding preflight.

## Solution Assessment

Unclear; the direction is plausible, but maintainers should decide the onboarding UX and lifecycle boundary first. If accepted, the narrow maintainable path is to reuse generic plugin `probeAccount`/`auditAccount` and gateway reachability plumbing rather than hard-code Telegram or Discord checks in core.
