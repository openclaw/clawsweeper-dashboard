## Summary

Keep open: current main has reusable channel probe surfaces, but onboarding still only performs setup-status and gateway-health checks; the requested per-channel reachability preflight is a new onboarding UX decision.

## What This Changes

Add an interactive or flag-driven connection preflight to `openclaw onboard` that validates channel credentials, gateway port reachability, and per-channel status before finalizing setup.

## Best Possible Solution

Add a generic channel preflight design that reuses plugin `probeAccount`/`auditAccount` and gateway reachability checks, with clear skip/opt-in behavior and no provider-specific logic in core onboarding.

## Reproduction Assessment

Not applicable; this is a feature request, not a failing current contract. Source and docs show current onboarding has gateway health checks and separate live channel probes, but no integrated onboarding channel preflight.

## Solution Assessment

Unclear; the existing channel probe contracts are likely the narrowest technical surface. Maintainers should decide whether the UX belongs in onboard, `channels add`, doctor, or an explicit opt-in flag/prompt before implementation.
