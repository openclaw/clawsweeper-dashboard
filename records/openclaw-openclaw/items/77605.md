## Summary

Keep open: current main still has only gateway reachability hints plus post-setup health/doctor diagnostics, while Telegram and Discord setup status remains config-based. The feature is vision-aligned, but the maintainable path needs a product-approved channel setup/plugin contract rather than a hard-coded core handshake.

## What This Changes

The issue requests an interactive `openclaw onboard` pre-flight that validates channel credentials and reachability before finalizing the generated config.

## Best Possible Solution

Add a product-approved, bounded plugin-owned setup preflight seam that `openclaw onboard` and `openclaw channels add` can use for channel credential checks, reusing existing Telegram/Discord probes and keeping `openclaw health`/doctor as post-setup diagnostics.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request, and no live channel-failure hang was reproduced. Source inspection does show that current onboard lacks an interactive channel credential/reachability preflight.

## Solution Assessment

No, not exactly as proposed: a core `getMe` or service-specific `--test-connection` path would be the wrong long-term layer. The better solution is a generic channel setup contract that lets Telegram, Discord, and other plugins expose bounded credential probes to onboarding.
