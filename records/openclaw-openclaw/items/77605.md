## Summary

Keep open: current main has adjacent gateway/channel probe diagnostics, but not an interactive onboarding channel reachability pre-flight, and the request is a new onboarding UX/diagnostic feature rather than a broken existing contract.

## What This Changes

The issue requests a connection pre-flight during `openclaw onboard` to validate channel reachability and credential errors before finalizing configuration.

## Best Possible Solution

Design an opt-in onboarding diagnostic step that reuses plugin-owned `probeAccount`/audit contracts and existing gateway reachability checks, with clear timeout and fallback behavior documented for setup users.

## Reproduction Assessment

Not applicable: this is a feature request for an additional onboarding pre-flight, not a current-main bug with a defined failing contract. Source inspection shows the requested interactive check is not currently present in the Channels step.

## Solution Assessment

Unclear until maintainers choose the UX boundary. The narrow maintainable direction is to reuse the existing `channels.status --probe` and plugin `probeAccount` seams instead of adding provider-specific handshake policy directly to core onboarding.
