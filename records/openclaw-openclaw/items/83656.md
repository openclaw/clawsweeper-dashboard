## Label Justifications

- `P2`: The issue is source-reproducible and affects persisted pairing state, but the blast radius is limited to the logs/Gateway pairing workflow.
- `impact:data-loss`: The reported side effect rewrites `paired.json`, a persisted device-pairing state file involved in related read/write race reports.
- `impact:auth-provider`: The behavior depends on Gateway device identity, paired-device authorization, and operator scope handling.

## Summary

Keep open: current main still routes `openclaw logs --follow` through Gateway RPC as a CLI client with a device identity, and the pairing path can persist or rewrite `devices/paired.json` for that connection. The reported `operator.write` detail is not current for `logs.tail`, which is now `operator.read`, but the paired-state mutation concern remains source-reproducible.

## What This Changes

The issue requests that `openclaw logs --follow` act as a non-mutating log reader instead of registering or updating a paired CLI device.

## Best Possible Solution

Make log-only polling non-mutating for paired-device state while preserving remote `logs.tail` support, and add regression coverage proving `devices/paired.json` is not created or rewritten for a logs-only call.

## Reproduction Assessment

Yes, at source level: current main routes `openclaw logs --follow` through CLI Gateway RPC with a device identity, and the Gateway pairing path can auto-approve, persist, or metadata-update paired CLI devices. I did not run the command live because the reproduction intentionally rewrites local pairing state.

## Solution Assessment

Unclear. The requested observer mode is plausible, but maintainers should first choose the logs/auth boundary; direct local file tailing for implicit local use plus a carefully scoped non-mutating remote read path may be safer than adding a user-facing flag.
