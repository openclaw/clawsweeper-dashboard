## Label Justifications

- `P2`: The issue is a source-reproducible logs/pairing side effect with real persisted-state churn but limited blast radius.
- `impact:data-loss`: The report centers on unnecessary rewrites of `paired.json`, a persisted pairing-state file involved in related read/write race reports.
- `impact:auth-provider`: The behavior depends on Gateway device identity, operator scopes, device tokens, and paired-device authentication state.

## Summary

Keep open: current main still routes `openclaw logs --follow` through CLI Gateway RPC with a CLI device identity, and the Gateway pairing path can write `devices/paired.json` on first pairing or metadata refresh. The report’s `operator.write` detail appears outdated because `logs.tail` is now classified as `operator.read`, but the non-passive paired-state mutation remains source-reproducible.

## What This Changes

The issue requests that `openclaw logs --follow` behave as a passive log reader without registering as a paired device or rewriting `paired.json`.

## Best Possible Solution

Make `openclaw logs --follow` non-mutating for log-only reads while preserving supported remote log access, and add regression proof that `devices/paired.json` is not created or rewritten by logs polling.

## Reproduction Assessment

Yes, at source level: current main routes `openclaw logs --follow` through CLI Gateway RPC with a device identity, and the handshake/pairing path can persist or metadata-refresh paired devices. I did not live-run it because the reported reproduction intentionally rewrites local pairing state.

## Solution Assessment

Unclear. The requested observer mode is plausible, but maintainers should first choose whether local logs should use direct file tailing, a no-device backend/shared-auth call, or an explicit non-pairing observer client for remote log access.
