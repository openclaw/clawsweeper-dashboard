## Label Justifications

- `P2`: The bug has a clear source-level path and affects persisted device state, but the surface is limited to log-follow/Gateway pairing behavior rather than a core runtime outage.
- `impact:data-loss`: The reported side effect unnecessarily rewrites `paired.json`, a persisted pairing-state file involved in related read/write race and state-loss reports.
- `impact:auth-provider`: The behavior is caused by Gateway device identity, operator scopes, and paired-device authentication metadata rather than log tailing alone.

## Summary

Keep open: current `main` still routes `openclaw logs --follow` through the normal CLI Gateway RPC path with a device identity, and the paired-device success path can persist metadata back to `paired.json`; the linked broad streaming PR was closed in favor of this narrower passive-log fix rather than superseding it.

## What This Changes

The issue asks for `openclaw logs --follow` to read logs without registering or updating a Gateway paired-device record.

## Best Possible Solution

Make `openclaw logs --follow` passive for log-only reads, preserving remote read access while proving it does not create, pair, update, or rewrite `devices/paired.json`; keep any streaming protocol redesign separate unless maintainers explicitly choose it.

## Reproduction Assessment

Yes, at source level: `openclaw logs --follow` calls `logs.tail` through the CLI Gateway helper, which attaches a device identity, then the paired-device handshake path can approve or update that device and persist `paired.json`. I did not live-run the stat check because the repro intentionally rewrites local pairing state.

## Solution Assessment

Unclear. The broad `logs.subscribe` approach was explicitly closed as the wrong quick vehicle, so the best path is a narrower maintainer-approved passive logs/auth boundary with regression proof that `paired.json` is untouched.
