## Label Justifications

- `P2`: The issue affects a narrow CLI logs flow but mutates persisted Gateway pairing state and can amplify related read races.
- `impact:data-loss`: The reported side effect rewrites `devices/paired.json`, a persisted pairing-state file involved in related race and overwrite reports.
- `impact:auth-provider`: The behavior is tied to Gateway device identity, pairing approval, device tokens, and operator scope state.

## Summary

Keep open: current main and the latest release still route `openclaw logs --follow` through the normal Gateway CLI client path, which can silently pair local CLI devices or update existing paired-device metadata and persist `devices/paired.json`. The reported `operator.write` detail is no longer accurate for `logs.tail`, but the paired-state mutation concern remains source-reproducible.

## What This Changes

The issue asks for `openclaw logs --follow` to act as a passive log reader without registering as a paired Gateway device or rewriting `paired.json`.

## Best Possible Solution

Keep remote log access while giving log-only reads a non-mutating path: no pairing creation, token rotation, metadata refresh, or `paired.json` write for log tailing, with regression coverage for both implicit local and explicit remote behavior.

## Reproduction Assessment

Yes at source level: current main routes `openclaw logs --follow` through a CLI Gateway RPC with a device identity, and the Gateway pairing path can persist new pairings or metadata updates to `paired.json`. I did not live-run the reporter's stat-based repro because this review must not mutate local pairing state.

## Solution Assessment

Unclear. The requested observer mode is plausible, but maintainers should choose the auth boundary because a local direct-file tail and a remote read-only observer path solve different parts of the behavior.
