## Label Justifications

- `P2`: The issue describes a valid persisted-state side effect with limited surface but real operational impact on paired-device state and race amplification.
- `impact:data-loss`: The issue centers on unnecessary rewrites of `paired.json`, a persisted pairing-state file involved in related read/write race reports.
- `impact:auth-provider`: The behavior is tied to Gateway device identity, operator scopes, and paired-device authentication state.

## Summary

Keep open: current main still sends `openclaw logs --follow` through the normal Gateway RPC CLI client path with a local device identity, and the Gateway pairing handshake can persist or refresh paired-device state. The reporter's latest comment asks whether this is intentional, so the next step is a maintainer decision on the logs/auth boundary rather than cleanup closure.

## What This Changes

The issue asks for `openclaw logs --follow` to read logs without registering as a paired Gateway device or rewriting `paired.json`.

## Best Possible Solution

Define a log-only auth and pairing contract, then implement log following so the supported local and remote paths can be tested without creating or rewriting `devices/paired.json` for read-only log polling.

## Reproduction Assessment

Yes, at source level: `openclaw logs --follow` polls `logs.tail` through the CLI Gateway path with a local device identity, and the handshake path can persist paired-device state. I did not live-run it because the stated reproduction intentionally rewrites local pairing state.

## Solution Assessment

Unclear. The desired outcome is sound, but maintainers should choose whether the durable fix is direct local file tailing, a read-only observer Gateway mode, or an explicit no-register mode before implementation.
