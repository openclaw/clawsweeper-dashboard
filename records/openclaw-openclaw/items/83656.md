## Label Justifications

- `P2`: The issue is a valid source-backed pairing-state side effect with limited surface but real operational impact on persisted device state.
- `impact:data-loss`: The report centers on unnecessary rewrites of `paired.json`, a durable pairing-state file involved in related read/write race reports.
- `impact:auth-provider`: The behavior depends on Gateway device identity, operator scopes, and paired-device authentication state.

## Summary

Keep open: current main still sends `openclaw logs --follow` through the CLI Gateway RPC path with a device identity, and the Gateway handshake can silently pair or metadata-update that device, which writes `devices/paired.json`. The reported `operator.write` detail is not supported by current source because `logs.tail` resolves to `operator.read`, but the paired-state mutation concern remains source-reproducible.

## What This Changes

The issue asks for `openclaw logs --follow` to read logs without registering as a paired device or rewriting `devices/paired.json`.

## Best Possible Solution

Make `openclaw logs --follow` non-mutating by default while preserving remote log access, and prove that log-only polling neither creates nor rewrites `devices/paired.json`.

## Reproduction Assessment

Yes, at source level: current main routes `openclaw logs --follow` through CLI Gateway RPC with a device identity, and the Gateway pairing path can auto-approve, persist, or metadata-update paired CLI devices. I did not live-run the command because the reported reproduction intentionally rewrites local pairing state.

## Solution Assessment

Unclear. The requested observer mode is plausible, but maintainers should choose whether the best boundary is direct local file tailing, a non-pairing read-only Gateway path, or reuse of the local backend no-pairing path.
