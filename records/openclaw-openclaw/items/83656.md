## Summary

Keep open: current main still routes `openclaw logs --follow` through the CLI Gateway RPC path with a CLI device identity before falling back only on errors, and the paired-device handshake path can persist or metadata-refresh that identity into `paired.json`. The report is source-reproducible, but the fix crosses the logs/auth boundary and needs a maintainer choice before automation should change behavior.

## What This Changes

The issue asks for `openclaw logs --follow` to read logs without registering as a paired device or rewriting `paired.json`, either through a non-mutating observer mode or direct file tailing.

## Best Possible Solution

Make logs polling non-mutating by default while preserving remote log access, then add a regression that proves log-only polling does not create, pair, or rewrite `devices/paired.json`.

## Reproduction Assessment

Yes, at source level: `openclaw logs --follow` still calls `logs.tail` through CLI Gateway RPC with a device identity, and the WebSocket pairing path can persist or refresh paired-device metadata. I did not live-run it because the repro intentionally rewrites local pairing state.

## Solution Assessment

Unclear. A non-mutating logs path is the right end state, but maintainers should choose whether implicit local logs should tail the file directly, use the existing backend device-less shared-auth pattern, or introduce a narrower observer/read-only Gateway client mode.
