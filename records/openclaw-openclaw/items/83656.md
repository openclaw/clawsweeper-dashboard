## Label Justifications

- `P2`: The report describes a valid persisted-state side effect with limited surface but real operational impact on paired-device state and race amplification.
- `impact:data-loss`: The issue centers on unnecessary rewrites of `devices/paired.json`, a persisted pairing-state file involved in related read/write race reports.
- `impact:auth-provider`: The behavior is tied to Gateway device identity, operator scopes, and paired-device authentication state.

## Summary

Keep open: current `main` still routes `openclaw logs --follow` through the normal CLI Gateway RPC identity path, and the server pairing path can auto-approve local CLI devices or rewrite paired-device metadata on reconnect. The reported `operator.write` detail no longer matches current source because `logs.tail` is classified as `operator.read`, but the paired-state mutation concern remains source-reproducible.

## What This Changes

The issue asks for `openclaw logs --follow` to read logs without registering as a paired device or rewriting `devices/paired.json`.

## Best Possible Solution

Make log tailing use a non-mutating read-only path that preserves remote log access and proves `devices/paired.json` is neither created nor rewritten for log-only polling.

## Reproduction Assessment

Yes, at source level: current `main` routes `openclaw logs --follow` through CLI Gateway RPC with a device identity, and the Gateway pairing path can persist or metadata-update paired CLI devices. I did not live-run the command because the reported reproduction intentionally rewrites local pairing state.

## Solution Assessment

Unclear. The requested observer mode is plausible, but the safest fix may be direct local file tailing for implicit local use plus a carefully scoped remote read-only Gateway path; maintainers should choose the auth boundary before a repair PR.
