## Label Justifications

- `P2`: The issue is a real logs/pairing side effect with limited surface but meaningful operational impact on paired-device state and race amplification.
- `impact:data-loss`: The report concerns unnecessary rewrites of `devices/paired.json`, a persisted pairing-state file involved in related read/write race reports.
- `impact:auth-provider`: The behavior is tied to Gateway client identity, operator scopes, and paired-device authentication state.

## Summary

Keep open: current `main` still routes `openclaw logs --follow` through the normal Gateway RPC client path with a CLI device identity, and both current `main` plus the latest release still persist paired-device metadata on accepted paired-client reconnects. Related work such as https://github.com/openclaw/openclaw/issues/83657 and https://github.com/openclaw/openclaw/pull/84285 addresses the transient JSON read race, but it does not resolve whether a passive logs reader should register or rewrite `devices/paired.json`.

## What This Changes

The issue asks for `openclaw logs --follow` to stream logs without registering as a paired device or rewriting `devices/paired.json`.

## Best Possible Solution

Make the default passive local logs path non-mutating while preserving deliberate authenticated remote log access, with a regression proving `devices/paired.json` is neither created nor rewritten for log-only polling.

## Reproduction Assessment

Yes, at source level: current `main` routes `openclaw logs --follow` through Gateway RPC with a normal CLI device identity, and accepted paired reconnects persist paired metadata. I did not live-run the command because the reported reproduction intentionally mutates local pairing state and this review is read-only.

## Solution Assessment

Unclear. The non-mutating outcome is the right target, but maintainers should choose the auth boundary before implementation: local file tailing for implicit loopback, a no-device observer RPC mode, or a documented read-only remote path.
