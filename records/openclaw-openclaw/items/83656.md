## Summary

Keep open: current main still routes implicit `openclaw logs --follow` through the normal CLI Gateway path with a device identity, which can create or refresh paired-device state. The active fix path is the open draft PR at https://github.com/openclaw/openclaw/pull/85380, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue asks for `openclaw logs --follow` to read logs without registering as a paired device or rewriting `devices/paired.json`.

## Best Possible Solution

Land or replace https://github.com/openclaw/openclaw/pull/85380 with a scoped non-mutating implicit-local logs path that preserves explicit `--url` behavior and proves `paired.json` is not created or rewritten.

## Reproduction Assessment

Yes, at source level: implicit `openclaw logs --follow` uses the CLI Gateway RPC path with device identity, and the Gateway handshake can persist pairing metadata. I did not run the stat-before/stat-after repro because it intentionally mutates local pairing state.

## Solution Assessment

Unclear until the active PR is resolved. A passive backend/local log path is the right direction for the paired-device side effect, but maintainers still need to choose the safe follow-mode fallback and auth boundary before closing this issue.
