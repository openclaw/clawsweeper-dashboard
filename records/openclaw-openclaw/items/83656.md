## Label Justifications

- `P2`: The issue is source-backed and affects persisted pairing state, but the blast radius is limited to log-follow and Gateway pairing-state churn rather than a core outage.
- `impact:data-loss`: The report centers on unnecessary rewrites of `paired.json`, a durable pairing-state file involved in related read/write race and state-loss reports.
- `impact:auth-provider`: The behavior depends on Gateway device identity, operator scopes, and paired-device authentication persistence.

## Summary

Keep open: current main still routes `openclaw logs --follow` through the normal CLI Gateway client path with a device identity, and accepted local CLI handshakes can auto-pair or update paired-device metadata that persists `devices/paired.json`.

## What This Changes

The issue asks for `openclaw logs --follow` to behave as a passive log reader that does not register as a paired device or rewrite `paired.json`.

## Best Possible Solution

Define a non-mutating log-read path that preserves required remote authorization while proving `openclaw logs --follow` does not create, approve, or rewrite `devices/paired.json` for log-only observation.

## Reproduction Assessment

Yes. Source inspection shows `openclaw logs --follow` enters the CLI Gateway RPC path with a device identity, and the Gateway pairing path can auto-pair or persist metadata for that device; I did not live-run it because the reproduction rewrites local pairing state.

## Solution Assessment

Unclear. A read-only observer path or direct local file tailing is plausible, but maintainers should choose the auth and audit boundary before implementation because skipping paired-device registration changes current Gateway semantics.
