## Label Justifications

- `P2`: The issue is source-backed and affects persisted pairing state, but the blast radius is limited to log-follow client activity and Gateway pairing metadata churn.
- `impact:data-loss`: The report centers on unnecessary rewrites of `devices/paired.json`, a persisted pairing-state file already tied to related read/write race reports.
- `impact:auth-provider`: The behavior flows through Gateway device identity, operator scopes, and paired-device authentication state.

## Summary

Keep open: current main still sends `openclaw logs --follow` through the normal CLI Gateway client path with a device identity, so log-only reads can still touch device pairing state. The reported `operator.write` detail is not current-main accurate because `logs.tail` is now classified as `operator.read`, but the paired-state mutation concern remains source-reproducible.

## What This Changes

The issue requests that `openclaw logs --follow` stop using a paired Gateway client path that rewrites `devices/paired.json` for log-only reads.

## Best Possible Solution

Make log-only reads use a non-mutating read-only path that preserves remote log access while proving `devices/paired.json` is not created, paired, or rewritten by `openclaw logs --follow`.

## Reproduction Assessment

Yes, at source level: current main routes `openclaw logs --follow` through CLI Gateway RPC with a device identity, and the pairing path can persist or metadata-update that device. I did not live-run the command because the reproduction intentionally rewrites local pairing state.

## Solution Assessment

Unclear. The requested observer mode is plausible, but maintainers should choose the auth boundary between direct local file tailing, a non-pairing read-only Gateway client, or the existing persistent-websocket follow work before implementation.
