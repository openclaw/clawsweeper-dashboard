## Label Justifications

- `P2`: The issue is source-reproducible and affects persisted pairing state, but its blast radius is limited to log-follow/Gateway pairing workflows.
- `impact:data-loss`: The report concerns avoidable rewrites of `paired.json`, a persisted pairing-state file involved in related read/write race reports.
- `impact:auth-provider`: The behavior is tied to Gateway device identity, operator scopes, device tokens, and paired-device authorization state.

## Summary

Keep open: current main and the latest release still route `openclaw logs --follow` through the normal CLI Gateway RPC path with a device identity, so the paired-state mutation concern remains source-reproducible. The `operator.write` detail appears stale because `logs.tail` is now classified as `operator.read`, but the pairing and `paired.json` rewrite behavior still needs a maintainer decision.

## What This Changes

The issue asks for `openclaw logs --follow` to read logs without registering as a paired Gateway device or rewriting `paired.json`.

## Best Possible Solution

Make log following non-mutating for the supported local path while preserving a deliberate, reviewed remote log access contract, then add a regression proving log-only polling does not create or rewrite `devices/paired.json`.

## Reproduction Assessment

Yes, at source level: current main routes `openclaw logs --follow` through CLI Gateway RPC with a device identity, and the Gateway pairing path can auto-approve or metadata-update paired CLI devices. I did not live-run the command because that repro intentionally rewrites local pairing state.

## Solution Assessment

Unclear. The requested observer/no-register behavior is plausible, but maintainers should choose the auth boundary between direct local file tailing, a no-device local Gateway path, or a carefully scoped remote read-only Gateway mode before implementation.
