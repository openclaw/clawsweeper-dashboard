## Summary

Keep open: current main and v2026.5.12 still publish unquoted bracket-notation config examples, and the reported shell failure reproduces before OpenClaw receives argv. The related docs fix at https://github.com/openclaw/openclaw/pull/40642 was closed unmerged, so the narrow docs bug remains.

## What This Changes

The issue reports that documented `openclaw config set agents.list[0].tools.exec.node` examples fail with `no matches found` in shells that treat `[0]` as a glob pattern.

## Best Possible Solution

Update the English config, exec, and nodes docs to use shell-safe quoted bracket paths or dot-index paths, and add a short Paths note explaining when bracket paths need quoting.

## Reproduction Assessment

Yes. The documented unquoted path still appears on current main, and `bash -O failglob -c 'printf "%s\n" agents.list[0].tools.exec.node'` fails with the same `no match` behavior while quoted and dot-index forms pass through.

## Solution Assessment

Yes. A docs-only repair is the narrowest maintainable fix because the config parser already handles bracket segments once the shell passes the argument to OpenClaw.
