## Summary

Keep open: current main and the latest release still do not expose OpenShell sandbox-level create-time environment variables, while upstream OpenShell supports repeated `--env KEY=VALUE` on `sandbox create`. The request is valid, but the permanent shape needs maintainer product/security direction because it adds or reuses a config/env surface at the sandbox boundary.

## What This Changes

Add non-secret environment variable support for OpenShell sandbox creation from `openclaw.json`.

## Best Possible Solution

Define one canonical operator-configured sandbox env contract for OpenShell, preferably reusing existing sandbox env settings if maintainers want backend parity, then pass validated entries as `--env` during `openshell sandbox create` with manifest, docs, and focused tests updated together.

## Reproduction Assessment

Yes, source inspection gives a high-confidence absence check: current OpenShell config has no env field and the create path never emits `--env`, while upstream OpenShell supports it. I did not run a live OpenShell gateway in this read-only review.

## Solution Assessment

Unclear. The requested capability is plausible, but the best implementation should first decide whether OpenShell reuses the existing sandbox env config or gets a new plugin-owned env field, then document security and recreate semantics.
