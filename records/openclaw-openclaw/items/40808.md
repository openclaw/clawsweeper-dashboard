## Summary

Keep open: this is a real feature proposal that is not implemented on current main, but the PR needs changes before merge because it duplicates older `sessions_send` behavior and conflicts with newer lane, reply, and A2A safety contracts. It also lacks after-fix real behavior proof for the advertised concurrent session flow.

## What This Changes

The PR adds a `sessions_send_concurrent` core tool, registers it in the agent tool catalog, documents it, and changes gateway nested-lane concurrency.

## Best Possible Solution

Rework this as a smaller feature that reuses or factors current `sessions_send` start/wait/A2A helpers, preserves per-session lane and parent-child safety behavior, updates the canonical session-tool docs/navigation, and includes redacted real runtime proof with multiple target sessions.

## Reproduction Assessment

Yes for the review findings from source inspection: the PR head can be compared directly to current main's `sessions_send` and lane contracts. I did not run a live multi-session reproduction because this was a read-only review and the PR lacks real behavior proof.

## Solution Assessment

No. The maintainable path is to build on current `sessions_send` helpers and per-session lane routing rather than duplicating stale A2A logic and widening the shared nested lane.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep nested work on per-session lanes:** `src/gateway/server-lanes.ts:10`
  - body: Current main deliberately keeps the shared `nested` lane serialized and scopes agent work with `nested:<sessionKey>` lanes, so different sessions can run in parallel without interleaving unrelated shared-lane work. Raising `CommandLane.Nested` here conflicts with that contract and the current `server-lanes` test; leave the shared lane alone and enqueue each target on `resolveNestedAgentLaneForSession(resolvedKey)`.
  - confidence: 0.89
- **[P1] Honor the parent-child A2A skip:** `src/agents/tools/sessions-send-concurrent-tool.ts:404-416`
  - body: The new tool always starts `runConcurrentA2AFlowForTarget`, but current `sessions_send` skips A2A for parent-owned ACP/native child sessions because their existing completion path already reports results and the peer A2A loop can wake the parent and feed its reply back to the child. Add the same target-entry and requester-ownership guard before both fire-and-forget and waited A2A starts.
  - confidence: 0.86
- **[P2] Read the updated reply with a baseline:** `src/agents/tools/sessions-send-concurrent-tool.ts:520-528`
  - body: After `agent.wait`, this reads the latest assistant message for the whole session. If no new reply was produced, or another concurrent send to that session wins the race, the tool can return and announce a stale or wrong reply; current `sessions_send` snapshots the pre-run reply and uses the run-wait helper to read only an updated reply.
  - confidence: 0.88
- **[P2] Reject ambiguous target selectors:** `src/agents/tools/sessions-send-concurrent-tool.ts:132-134`
  - body: Targets that include both `sessionKey` and `label` silently prefer `sessionKey` and ignore `label`/`agentId`. `sessions_send` rejects that combination, and keeping the same validation here avoids generated payloads being routed to the wrong session without an error.
  - confidence: 0.82
