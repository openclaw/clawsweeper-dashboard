## Best Possible Solution

If multi-target sending belongs in core, factor it through shared `sessions_send` internals so validation, provenance, scoped lanes, run-wait, and A2A guards stay single-source, then add focused regression coverage and redacted live proof.

## Solution Assessment

- **[P1] Import tool types from current packages:** `src/agents/tools/sessions-send-concurrent-tool.ts:2-3`
  - body: Current main declares `@earendil-works/pi-agent-core` and `typebox`, but this new file imports the old `@mariozechner/pi-agent-core` and `@sinclair/typebox` packages. After rebase, the tool will not resolve until it uses the current package names.
  - confidence: 0.94

  - body: Current main keeps the exact `CommandLane.Nested` lane at default serial concurrency and uses session-scoped nested lanes for safe parallelism. Widening the shared lane lets unscoped nested work interleave and contradicts the current lane contract.
  - confidence: 0.91
- **[P2] Use scoped lanes for each target run:** `src/agents/tools/sessions-send-concurrent-tool.ts:364`
  - body: Every target dispatch uses `AGENT_LANE_NESTED`, so once the shared lane is widened, two sends to the same target can run concurrently. Current `sessions_send` dispatches through `resolveNestedAgentLaneForSession(resolvedKey)` to preserve same-session serialization.
  - confidence: 0.92
- **[P2] Annotate the primary inter-session prompt:** `src/agents/tools/sessions-send-concurrent-tool.ts:359`
  - body: The primary dispatch sends `target.message` directly. Current inter-session sends wrap the text with `annotateInterSessionPromptText`, so the target agent receives an in-prompt warning that the content came from another session/tool rather than a direct user instruction.
  - confidence: 0.89
- **[P2] Read replies through the run-wait helper:** `src/agents/tools/sessions-send-concurrent-tool.ts:520-528`
  - body: After `agent.wait`, this reads the latest assistant message from `chat.history`; if no fresh reply is produced or another message lands first, it can return a stale or wrong reply. Use the current baseline-aware `waitForAgentRunAndReadUpdatedAssistantReply` path.
  - confidence: 0.88
- **[P2] Reject ambiguous target identities:** `src/agents/tools/sessions-send-concurrent-tool.ts:131-134`
  - body: The parser stores both `sessionKey` and `label`, then later silently prefers `sessionKey`. Current `sessions_send` rejects that ambiguous input, so generated bulk payloads can send to the wrong session without a validation error.
  - confidence: 0.88
- **[P2] Reject thread-session targets before dispatch:** `src/agents/tools/sessions-send-concurrent-tool.ts:335-336`
  - body: After resolving the visible session, this proceeds to dispatch without the thread-session guard current `sessions_send` applies. Bulk inter-agent sends can therefore target a thread session that the single-send tool blocks.
  - confidence: 0.84
- **[P2] Skip duplicate A2A for parent-owned child sessions:** `src/agents/tools/sessions-send-concurrent-tool.ts:547-559`
  - body: Successful sends always start the concurrent A2A helper. Current `sessions_send` skips follow-up for parent-owned ACP/native child sessions because another result path already reports the reply; omitting that guard can duplicate output or start reply loops.
  - confidence: 0.86
- **[P2] Filter non-deliverable A2A replies:** `src/agents/tools/sessions-send-concurrent-a2a.ts:184`
  - body: The copied A2A flow only checks `REPLY_SKIP` and `ANNOUNCE_SKIP`. Current `sessions_send` also suppresses non-deliverable control replies before ping-pong and announce, so this helper can feed or post control replies that existing behavior filters out.
  - confidence: 0.83
- **[P2] Use scoped lanes inside A2A steps:** `src/agents/tools/sessions-send-concurrent-a2a.ts:176`
  - body: Ping-pong and announce steps use the shared `AGENT_LANE_NESTED` lane. Current A2A steps use `resolveNestedAgentLaneForSession(...)`, so same-session steps stay serialized while different sessions can proceed independently.
  - confidence: 0.86
