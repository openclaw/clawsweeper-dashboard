## Summary

Keep open: this is a plausible new core feature, but current main does not implement `sessions_send_concurrent` and the PR still diverges from current `sessions_send` contracts in dependency usage, nested-lane isolation, reply reads, target validation, A2A loop suppression, and non-deliverable reply filtering.

## What This Changes

Adds a new core `sessions_send_concurrent` tool with docs/tests/registration, A2A helper logic, and gateway Nested-lane concurrency changes for multi-session messaging.

## Best Possible Solution

If maintainers want this core capability, implement it as a thin multi-target wrapper over shared `sessions_send` start, wait, validation, lane, and A2A helpers, with focused bulk-send regression coverage and real behavior proof.

## Reproduction Assessment

Yes for the PR blockers: they are source-reproducible by comparing the PR head against current `sessions_send`, A2A, run-wait, dependency, and lane contracts. This is not a current-main bug reproduction because the PR proposes a new core tool.

## Solution Assessment

No. The current PR forks stale `sessions_send` behavior and widens the shared Nested lane; the narrower maintainable path is to reuse or extract current `sessions_send` internals while preserving per-session lane isolation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Import Type from the declared typebox package:** `src/agents/tools/sessions-send-concurrent-tool.ts:3`
  - body: The new file imports `Type` from `@sinclair/typebox`, but this repo declares `typebox` and current source imports from `typebox`. With pnpm package boundaries, the tool can fail module resolution before it runs.
  - confidence: 0.92
- **[P2] Use per-session nested lanes for target runs:** `src/agents/tools/sessions-send-concurrent-tool.ts:364`
  - body: Every target is enqueued on the shared `AGENT_LANE_NESTED` lane while the PR also widens that lane globally. Current `sessions_send` scopes nested work with `resolveNestedAgentLaneForSession(resolvedKey)`, so this can bypass same-session isolation and interleave work that should remain serialized per session.
  - confidence: 0.9
- **[P2] Use per-session lanes for A2A turns:** `src/agents/tools/sessions-send-concurrent-a2a.ts:176`
  - body: The A2A helper runs ping-pong and announce turns on the shared nested lane. Current `sessions_send` scopes these turns to the current or target session key, so concurrent targets can block or interleave incorrectly instead of following the existing lane contract.
  - confidence: 0.89
- **[P2] Read replies through the run-wait helper:** `src/agents/tools/sessions-send-concurrent-tool.ts:520-528`
  - body: After `agent.wait`, this reads the latest assistant message from history without a pre-run baseline or `waitForAgentRunAndReadUpdatedAssistantReply`. If an older assistant reply is still last, or the run completes without a new reply, the concurrent tool can return stale text that current `sessions_send` filters out.
  - confidence: 0.88
- **[P2] Reject targets that provide both sessionKey and label:** `src/agents/tools/sessions-send-concurrent-tool.ts:131-134`
  - body: The parser accepts a target containing both `sessionKey` and `label` and silently prefers `sessionKey`. Current `sessions_send` rejects this ambiguous input, so generated bulk payloads can accidentally send to the wrong session without an error.
  - confidence: 0.87
- **[P2] Reject thread-session targets before dispatch:** `src/agents/tools/sessions-send-concurrent-tool.ts:335-336`
  - body: Current `sessions_send` rejects direct or label-resolved thread session keys before starting an agent run. This tool resolves the target and then proceeds to dispatch, so a bulk send can target thread sessions for inter-agent coordination even though the single-target tool blocks that path.
  - confidence: 0.84
- **[P2] Mirror the parent-owned child A2A skip guard:** `src/agents/tools/sessions-send-concurrent-tool.ts:547-559`
  - body: Current `sessions_send` skips A2A follow-up when the requester is the parent of its own parent-owned ACP/native child session because another result path already reports the reply. This starts the concurrent A2A helper unconditionally after successful sends, which can create duplicate visible output or reply loops.
  - confidence: 0.85
- **[P2] Filter non-deliverable A2A replies:** `src/agents/tools/sessions-send-concurrent-a2a.ts:184`
  - body: The copied A2A flow only checks `REPLY_SKIP` and `ANNOUNCE_SKIP`. Current `sessions_send` also treats silent and heartbeat tokens as non-deliverable before ping-pong and announce delivery, so the concurrent flow can feed or post text the existing tool suppresses.
  - confidence: 0.8
