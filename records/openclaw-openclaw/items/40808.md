## Summary

Keep this PR open, but it is not merge-ready. Current main does not implement `sessions_send_concurrent`, so the work is not obsolete; however, the branch adds a new public cross-session tool and diverges from current `sessions_send` safety, lane, provenance, and A2A behavior.

## What This Changes

The PR adds a new `sessions_send_concurrent` agent tool, A2A helper, docs, tests, tool registration, and gateway nested-lane concurrency changes for sending messages to up to 20 sessions in parallel.

## Best Possible Solution

If OpenClaw wants this as a built-in tool, rebase it onto current main and build it by reusing current `sessions_send` helper seams for validation, provenance annotation, per-session lane selection, wait/reply snapshot handling, A2A skip policy, announce delivery, and security policy registration.

## Solution Assessment

No. The feature direction may be useful, but this patch is not the best current-main solution because it duplicates large parts of `sessions_send` and misses current safety and routing contracts that should be shared rather than forked.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Deny the new tool on Gateway HTTP:** `src/agents/openclaw-tools.ts:171-175`
  - body: `sessions_send_concurrent` is equivalent to `sessions_send` for cross-session injection, but the PR only registers it as a normal core tool. Because `DEFAULT_GATEWAY_HTTP_TOOL_DENY` still denies only `sessions_send`, HTTP `/tools/invoke` can expose this new injection path when the tool is otherwise allowed. Add the new id to the dangerous deny list and matching tests/docs before registering it.
  - confidence: 0.9
- **[P2] Reject targets with both sessionKey and label:** `src/agents/tools/sessions-send-concurrent-tool.ts:131-134`
  - body: `sessions_send` treats `sessionKey` plus `label` as invalid, but the new concurrent tool silently keeps `sessionKey` and ignores `label`/`agentId`. Generated payloads that include both fields can be sent to the wrong session without an error, so this should return the same validation error as `sessions_send` before resolving the target.
  - confidence: 0.9
- **[P2] Preserve inter-session prompt annotation:** `src/agents/tools/sessions-send-concurrent-tool.ts:358-376`
  - body: Current `sessions_send` wraps the primary message with `annotateInterSessionPromptText(...)` before it reaches the target model. The concurrent tool sends the raw `target.message` while only setting `inputProvenance`, which reopens the same-turn foreign-output-as-user-text problem fixed on main.
  - confidence: 0.86
- **[P2] Use per-session nested lanes for agent runs:** `src/agents/tools/sessions-send-concurrent-tool.ts:364`
  - body: The patch queues every primary and A2A run on the shared `nested` lane and then widens that global lane. Current main routes `sessions_send` work through `resolveNestedAgentLaneForSession(...)` so different target sessions can run independently without changing the shared nested-lane contract. Hard-coding `AGENT_LANE_NESTED` misses that isolation and conflicts with the current test that keeps the shared lane serial.
  - confidence: 0.83
- **[P2] Preserve the current A2A skip policy:** `src/agents/tools/sessions-send-concurrent-tool.ts:404-416`
  - body: Current `sessions_send` skips ping-pong/announce when the requester is the parent of its own background ACP child, because that child already reports through the internal task completion path. The new concurrent helper always starts A2A, so sending concurrently to that kind of child can reintroduce the parent/child echo loop fixed on main.
  - confidence: 0.78
- **[P2] Capture a baseline before reading the wait-mode reply:** `src/agents/tools/sessions-send-concurrent-tool.ts:517-528`
  - body: The new wait-mode path reads the latest assistant message from `chat.history` after `agent.wait` without capturing the pre-run snapshot. Current `sessions_send` uses `waitForAgentRunAndReadUpdatedAssistantReply` with a baseline so fast completions and existing assistant messages do not cause the caller or A2A flow to use the wrong reply.
  - confidence: 0.75
