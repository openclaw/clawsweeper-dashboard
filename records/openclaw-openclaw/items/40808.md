## Summary

Keep open: current main does not implement `sessions_send_concurrent`, but this PR is not merge-ready because it forks fast-moving `sessions_send` behavior, still diverges from current lane/provenance/run-wait/A2A contracts, and lacks contributor real behavior proof.

## What This Changes

The PR adds a core `sessions_send_concurrent` tool with docs, tests, tool registration, and gateway Nested-lane concurrency changes for concurrent inter-session sends.

## Best Possible Solution

If multi-target sending belongs in core, extract or reuse the current `sessions_send` internals so validation, provenance, scoped lanes, run-wait, A2A filtering, and parent-child suppression stay single-source, then add focused regression tests and redacted live proof.

## Reproduction Assessment

Yes for the PR blockers: source comparison against current main shows stale imports and mismatched validation, lane, provenance, run-wait, and A2A contracts. This is not a current-main bug reproduction because the PR proposes a new core tool.

## Solution Assessment

No. Forking `sessions_send` and widening the shared Nested lane is not the narrowest maintainable solution; the safer path is to reuse shared single-send internals and preserve per-session isolation.

  - confidence: 0.84

  - body: The copied A2A flow only checks `REPLY_SKIP` and `ANNOUNCE_SKIP`. Current `sessions_send` also suppresses non-deliverable control replies before ping-pong and announce, so this helper can feed or post control replies that existing behavior filters out.
  - confidence: 0.83
