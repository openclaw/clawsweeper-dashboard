## What This Changes

Adds a built-in `sessions_send_concurrent` tool with docs, tests, tool registration, A2A handling, and gateway nested-lane concurrency changes.

## Best Possible Solution

Rebase and reduce the implementation to concurrent orchestration that reuses current `sessions_send` validation, provenance, run/wait, A2A, and per-session nested-lane helpers, with focused tests and a changelog entry.

## Reproduction Assessment

Yes. Source inspection of PR head `2b2d220d5d333823ef05c439f5560d11b88d8c48` against current main shows concrete divergences in imports, prompt provenance, lane selection, target validation, thread guards, timeout handling, and reply selection.

## Solution Assessment

No. The feature direction can work, but this patch copies older `sessions_send` behavior; the safer solution is to share or wrap the current helpers instead of widening the shared nested lane and reimplementing A2A flow.

Overall confidence: 0.94

Full review comments:

- **[P1] Import Type from the current typebox package:** `src/agents/tools/sessions-send-concurrent-tool.ts:3`
  - body: Current main no longer declares `@sinclair/typebox`; new source that imports it will fail type resolution when rebased or merged. Use `import { Type } from "typebox"` like the existing tool schemas.
  - confidence: 0.96
- **[P2] Keep inter-session provenance on forwarded prompts:** `src/agents/tools/sessions-send-concurrent-tool.ts:359`
  - body: Current `sessions_send` wraps forwarded text with inter-session provenance before it reaches the target model. Sending `target.message` directly makes the receiving session see another agent's text as an ordinary user message, losing the `isUser=false` boundary.
  - confidence: 0.93
- **[P2] Do not widen the shared nested lane globally:** `src/gateway/server-lanes.ts:10`
  - body: Current main deliberately leaves the shared `nested` lane single-concurrency and routes agent work through per-session nested lanes. Raising `CommandLane.Nested` changes unrelated nested workflows and contradicts the current `server-lanes.test.ts` expectation.
  - confidence: 0.92
- **[P2] Reject targets that provide both sessionKey and label:** `src/agents/tools/sessions-send-concurrent-tool.ts:182`
  - body: The docs say `sessionKey` and `label` are mutually exclusive, and current `sessions_send` errors on that ambiguous input. This branch silently prefers `sessionKey`, so generated payloads can be sent to the wrong session while ignoring the label.
  - confidence: 0.93
- **[P2] Block thread-scoped session targets before dispatch:** `src/agents/tools/sessions-send-concurrent-tool.ts:357-364`
  - body: Current `sessions_send` rejects session keys that resolve to `:thread:` sessions so inter-agent coordination is not injected into active human-facing chat threads. The concurrent tool resolves visibility and then dispatches without that guard.
  - confidence: 0.89
- **[P2] Keep delayed replies alive after soft wait timeouts:** `src/agents/tools/sessions-send-concurrent-tool.ts:493-502`
  - body: Current `sessions_send` treats non-terminal wait timeouts as accepted and starts the background A2A flow so delayed replies can still reach the requester. This code returns `timeout` immediately and never starts the follow-up flow, dropping that current behavior.
  - confidence: 0.88
- **[P2] Read only the reply produced by this run:** `src/agents/tools/sessions-send-concurrent-tool.ts:520-528`
  - body: After `agent.wait`, this reads the latest assistant message from the whole session history. Current `sessions_send` captures a pre-run baseline and only returns a newer assistant reply, which avoids stale replies when a run produces no text or another message lands nearby.
  - confidence: 0.9
- **[P3] Add the required changelog entry:** `src/agents/tool-catalog.ts:137-144`
  - body: This registers a new user-facing `feat` tool surface, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires a changelog entry for user-facing feature changes before merge.
  - confidence: 0.86
