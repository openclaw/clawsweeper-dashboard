## Summary

Keep this PR open, but it is not ready to merge against current main. The branch is stale against recent session-tool and lane-concurrency work, and the new tool currently misses several parity guards from `sessions_send` while also introducing a build-time import mismatch.

## What This Changes

The PR adds a new `sessions_send_concurrent` tool, registers and documents it, and changes gateway nested lane concurrency for concurrent A2A session sends.

## Best Possible Solution

Rebase the feature onto current main and implement the concurrent tool by reusing the current `sessions_send` run, wait, visibility, thread-target, and per-session lane helpers instead of forking older behavior.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: the new file imports a package that current main no longer declares, and the global nested-lane change conflicts with the current `server-lanes.test.ts` expectation that the shared nested lane remains single-concurrency.

## Solution Assessment

No. The current PR direction can work, but this implementation should be narrowed to a concurrent wrapper around the current `sessions_send` primitives and should avoid changing global shared nested-lane semantics.

Full review comments:

- **[P1] Import Type from the current typebox package:** `src/agents/tools/sessions-send-concurrent-tool.ts:3`
  - body: Current main has migrated tool schemas to `typebox` and no longer declares `@sinclair/typebox` in `package.json`. Adding a new source file that imports `@sinclair/typebox` will fail type resolution after this branch is rebased or merged onto current main; use `import { Type } from "typebox"` like the existing tools.
  - confidence: 0.95
- **[P2] Do not widen the shared nested lane globally:** `src/gateway/server-lanes.ts:10`
  - body: Current main deliberately keeps the shared `CommandLane.Nested` lane at its default single concurrency and uses per-session nested lanes for agent work. Setting the shared lane to `resolveAgentMaxConcurrent` changes unrelated nested workflows and contradicts the existing `server-lanes.test.ts` coverage; the concurrent tool should route each target through `resolveNestedAgentLaneForSession` instead.
  - confidence: 0.9
- **[P2] Reject targets with both sessionKey and label:** `src/agents/tools/sessions-send-concurrent-tool.ts:182`
  - body: The docs say `sessionKey` and `label` are mutually exclusive, and current `sessions_send` returns an error for that ambiguous input. This implementation only resolves labels when `sessionKey` is absent, so generated payloads with both fields silently ignore `label` and can send to the wrong session.
  - confidence: 0.93
- **[P2] Preserve the thread-target guard from sessions_send:** `src/agents/tools/sessions-send-concurrent-tool.ts:335-336`
  - body: Current `sessions_send` rejects session keys that include thread info because thread sessions are not valid inter-agent coordination targets. The concurrent tool resolves visibility and then sends without the same `parseSessionThreadInfoFast` guard, so it reopens a target class that main now blocks.
  - confidence: 0.88
- **[P2] Read only the reply produced by this run:** `src/agents/tools/sessions-send-concurrent-tool.ts:520-528`
  - body: After `agent.wait` succeeds, this code reads the latest assistant message from the whole session history. Current `sessions_send` captures a baseline before starting the run and only returns a newer assistant reply, which avoids stale reply carryover when the run produces no assistant text or another message lands nearby.
  - confidence: 0.86
