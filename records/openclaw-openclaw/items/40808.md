If maintainers want this capability, implement it as a thin multi-target wrapper over shared current `sessions_send` validation, start, wait, lane, and A2A helpers, with focused regression coverage and redacted live proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Import Type from the current typebox package:** `src/agents/tools/sessions-send-concurrent-tool.ts:3`
  - body: Current main declares and imports `typebox`, not `@sinclair/typebox`. After merging with current main, this new direct import can fail package-boundary/module resolution and should use the same `typebox` import as the rest of the agent tools.
  - confidence: 0.9
- **[P2] Scope target runs to each session lane:** `src/agents/tools/sessions-send-concurrent-tool.ts:364`
  - body: Every target run is enqueued on the shared nested lane while current `sessions_send` uses `resolveNestedAgentLaneForSession(resolvedKey)`. This bypasses same-session serialization and can interleave work that the current lane contract keeps isolated per target session.
  - confidence: 0.91
- **[P2] Scope A2A turns to each session lane:** `src/agents/tools/sessions-send-concurrent-a2a.ts:176`
  - body: The A2A helper runs ping-pong turns on the shared nested lane, with the same pattern repeated for announce. Current `sessions_send` scopes each nested turn to the current or target session key, so concurrent targets can block or interleave incorrectly here.
  - confidence: 0.9
- **[P2] Read replies through the run-wait helper:** `src/agents/tools/sessions-send-concurrent-tool.ts:520-528`
  - body: After `agent.wait`, this reads the latest assistant history item without the pre-run snapshot and `waitForAgentRunAndReadUpdatedAssistantReply` filtering used by current `sessions_send`. A previous assistant reply can be returned as the new target result when the run produces no fresh reply.
  - confidence: 0.88
- **[P2] Reject targets that set both sessionKey and label:** `src/agents/tools/sessions-send-concurrent-tool.ts:131-134`
  - body: The parser stores both `sessionKey` and `label`, then silently prefers `sessionKey` during resolution. Current `sessions_send` rejects that ambiguous input, so generated bulk payloads can send to the wrong session without a validation error.
  - confidence: 0.87
- **[P2] Reject thread-session targets before dispatch:** `src/agents/tools/sessions-send-concurrent-tool.ts:335-336`
  - body: Current `sessions_send` rejects direct or label-resolved thread session keys before starting an agent run. This flow resolves the visible session and proceeds to dispatch without the thread-session guard, allowing bulk inter-agent sends to a target the single-send tool blocks.
  - confidence: 0.84
- **[P2] Mirror the parent-owned child A2A skip guard:** `src/agents/tools/sessions-send-concurrent-tool.ts:547-559`
  - body: Current `sessions_send` skips A2A follow-up when the requester is the parent of its own parent-owned ACP/native child session because another result path already reports the reply. This starts the concurrent A2A helper unconditionally after successful sends, which can create duplicate visible output or reply loops.
  - confidence: 0.85
- **[P2] Filter non-deliverable replies before A2A delivery:** `src/agents/tools/sessions-send-concurrent-a2a.ts:184`
  - body: The copied A2A flow only checks `REPLY_SKIP` and `ANNOUNCE_SKIP`. Current `sessions_send` also suppresses silent and heartbeat tokens before ping-pong and announce delivery, so this helper can feed or post control replies that the existing tool filters out.
  - confidence: 0.82
