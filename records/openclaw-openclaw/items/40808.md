## Best Possible Solution

If maintainers want this core capability, implement it as a thin multi-target wrapper over current `sessions_send` start, wait, lane, target-validation, and A2A helpers with focused bulk-send regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Import Type from the declared typebox package:** `src/agents/tools/sessions-send-concurrent-tool.ts:3`
  - body: The new file imports `Type` from `@sinclair/typebox`, but current package metadata declares `typebox` and existing source imports from `typebox`. On current main this module specifier is undeclared, so the new tool can fail module resolution before it runs.
  - confidence: 0.92
- **[P2] Use per-session nested lanes for target runs:** `src/agents/tools/sessions-send-concurrent-tool.ts:364`
  - body: This enqueues every target on the shared `AGENT_LANE_NESTED` lane while the PR also widens that shared lane globally. Current main keeps the shared Nested lane serial and uses `resolveNestedAgentLaneForSession(resolvedKey)` so different targets can run independently without bypassing per-session isolation.
  - confidence: 0.9
- **[P2] Use per-session lanes for A2A turns:** `src/agents/tools/sessions-send-concurrent-a2a.ts:176`
  - body: The A2A helper also runs ping-pong and announce turns on the shared nested lane. Current `sessions_send` scopes these turns with `resolveNestedAgentLaneForSession(currentSessionKey)` or the target session key, so concurrent targets can block each other or bypass the lane contract.
  - confidence: 0.89
- **[P2] Read replies through the run-wait helper:** `src/agents/tools/sessions-send-concurrent-tool.ts:520-523`
  - body: After `agent.wait`, this reads the latest assistant message directly from history without the pre-run baseline and `waitForAgentRunAndReadUpdatedAssistantReply`. If an older assistant reply is still last, or if the run does not produce a new reply, the concurrent tool can return stale text that current `sessions_send` filters out.
  - confidence: 0.88
- **[P2] Reject targets with both sessionKey and label:** `src/agents/tools/sessions-send-concurrent-tool.ts:131-134`
  - body: The parser accepts a target that supplies both `sessionKey` and `label` and then silently prefers `sessionKey`. Current `sessions_send` rejects that ambiguous input, so generated bulk payloads can accidentally send to the wrong session without an error.
  - confidence: 0.87
- **[P2] Reject thread-session targets before dispatch:** `src/agents/tools/sessions-send-concurrent-tool.ts:335-336`
  - body: Current `sessions_send` rejects direct or label-resolved thread session keys before starting an agent run. This tool resolves visibility and then proceeds to dispatch, so a bulk send can target thread sessions for inter-agent coordination even though the single-target tool blocks that path.
  - confidence: 0.84
- **[P2] Mirror the parent-owned child A2A skip guard:** `src/agents/tools/sessions-send-concurrent-tool.ts:547-559`
  - body: Current `sessions_send` skips A2A follow-up when the requester is the parent of its own parent-owned ACP/native child session, because another completion path already reports the result and A2A can create duplicate output or reply loops. This starts the concurrent A2A helper unconditionally after successful sends.
  - confidence: 0.85
- **[P2] Filter non-deliverable A2A replies:** `src/agents/tools/sessions-send-concurrent-a2a.ts:184`
  - body: The copied A2A flow only checks `REPLY_SKIP` and `ANNOUNCE_SKIP`. Current `sessions_send` also treats silent and heartbeat tokens as non-deliverable before ping-pong and announce delivery, so the concurrent flow can feed or post text that the existing tool suppresses.
  - confidence: 0.8
