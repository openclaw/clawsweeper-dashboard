## Summary

Keep open: the bug is real and the patch touches the right detector, but it should not merge yet because stripping the leading separator before the existing broad prefix check can hide genuinely different backslashed OpenClaw task names, and the external PR only has mocked test proof.

## What This Changes

This PR strips leading slash/backslash prefixes before Windows scheduled-task name matching in `findExtraGatewayServices` and adds a mocked regression for a backslashed canonical task plus a legacy task.

## Best Possible Solution

Tighten Windows task matching to exact default/profile task names, preserve detection for copy/stale OpenClaw tasks, add that regression, then provide real Windows `schtasks` or status proof before merge.

## Reproduction Assessment

Yes at source level: current main skips `OpenClaw Gateway` but not the `\OpenClaw Gateway` form shown by Windows `schtasks`, so that task can flow into the extra-service warning path. I did not run a live Windows Task Scheduler repro from this read-only Linux checkout.

## Solution Assessment

No: the direction is right, but the current patch reuses the broad `openclaw gateway` prefix check after stripping separators, which is too broad for copy or stale task names. A profile-aware exact matcher with a copy-style regression is the safer fix.
