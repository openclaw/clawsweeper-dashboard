## Summary

Keep open: current main and the latest stable release still show the OpenAI OAuth token success path parsing with raw `response.json()` and callback cleanup that only calls `server.close()`. The linked fixing PR is still open and explicitly owns this issue, so this should wait for that PR to land or be rejected.

## What This Changes

The issue asks for the OpenAI ChatGPT/Codex OAuth flow to handle compressed or non-JSON token responses without raw JSON parse errors and to exit cleanly after successful browser OAuth.

## Best Possible Solution

Review and either land or reject https://github.com/openclaw/openclaw/pull/89491; if accepted, keep the fix scoped to OpenAI OAuth token parsing and callback/prompt cleanup while leaving the broader gzip-response umbrella with https://github.com/openclaw/openclaw/issues/79752.

## Reproduction Assessment

No live current-main OAuth replay was run. Source inspection gives a focused path: a successful token response with non-JSON or binary-looking bytes reaches `response.json()`, and callback success still relies on `server.close()` only.

## Solution Assessment

Yes, the best direction is a narrow OpenAI plugin/provider OAuth repair with regression tests and live proof. The linked PR is the active implementation candidate; a broad decompression wrapper belongs with the separate umbrella issue if maintainers choose that direction.
