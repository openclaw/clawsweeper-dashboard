## Summary

Keep open: current main still routes OpenAI Codex OAuth login and refresh through the pinned pi-ai OAuth helper, with no OpenAI OAuth-specific compressed-response guard and no accepted callback-socket cleanup after success. The related compressed-response issue remains relevant, but this issue also includes a distinct localhost callback hang that is not covered by that broader report.

## What This Changes

The issue reports that OpenAI Codex OAuth can parse compressed-looking token responses as JSON and can leave the auth CLI process hanging after a successful localhost callback.

## Best Possible Solution

Add OpenAI Codex OAuth-owned hardening for token exchange/refresh decoding and callback socket cleanup, or take an upstream pi-ai fix that exposes those controls, while leaving the broader HTTP decompression work to https://github.com/openclaw/openclaw/issues/79752.

## Reproduction Assessment

No. I did not run a real OpenAI OAuth flow on macOS/Node v26, but current source and the pinned dependency make the reported failure class source-reproducible: token responses are parsed directly as JSON and accepted callback sockets are not tracked for destruction.

## Solution Assessment

Unclear. The local release-branch mitigation is plausible, but the maintainable fix should be a narrow OpenAI Codex OAuth wrapper or upstream-equivalent dependency fix rather than a broad global fetch/decompression change.
