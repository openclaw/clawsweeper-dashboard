## Summary

Keep open: this is a fresh, concrete OpenAI Codex OAuth bug report, and current main still delegates the login/refresh paths to the pinned pi-ai OAuth helper without a local decoded-response guard or visible callback-socket cleanup beyond the upstream server close.

## What This Changes

The issue reports that OpenAI Codex OAuth login can parse compressed-looking token responses as JSON bytes and can leave the CLI process alive after printing success.

## Best Possible Solution

Keep this open and fix the OpenAI Codex OAuth boundary so token login/refresh receives decoded JSON and the localhost callback flow releases accepted sockets after completion, with focused tests and redacted live OAuth proof before release.

## Reproduction Assessment

No, not as a live current-main repro. The issue has concrete release-branch output and current source plus dependency inspection shows the same token-parse and callback-cleanup path, but I did not run a macOS/Node v26 OAuth flow with live credentials.

## Solution Assessment

Unclear. The narrow maintainable direction is likely in the OpenAI Codex OAuth wrapper or upstream pi-ai helper, not a broad shared decompression fallback, but maintainers should choose that boundary before implementation.
