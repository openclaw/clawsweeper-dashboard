## Summary

Keep open: current main still delegates OpenAI Codex OAuth token exchange, refresh, and callback-server ownership to the pinned pi-ai OAuth helper, and the reviewed paths do not yet add an OAuth-specific compressed-response guard or accepted-socket cleanup.

## What This Changes

The issue reports that `openclaw models auth login --provider openai-codex` can parse compressed OAuth token responses as JSON and can leave the localhost callback socket open after a successful login.

## Best Possible Solution

Add an OpenAI Codex OAuth-owned hardening layer for token exchange/refresh decoding and callback socket cleanup, while coordinating any generic decompression work with the broader HTTP response issue.

## Reproduction Assessment

No high-confidence live reproduction was run because it requires a real OpenAI OAuth flow on macOS/Node v26. The current source and pinned dependency make the reported failure class source-reproducible: token responses are parsed directly as JSON and the local callback server cleanup does not destroy accepted sockets.

## Solution Assessment

Unclear whether the release-branch mitigation is the final best form. The maintainable fix should be a narrow OpenAI Codex OAuth wrapper or upstream-equivalent dependency fix, not a broad global fetch behavior change that duplicates the wider compressed-response investigation.
