## Summary

Keep open: current main still routes OpenAI Codex OAuth login and refresh through the pinned OAuth helper without an OpenAI-specific compressed-response guard or accepted callback-socket cleanup, so the reported auth failure and hang remain actionable.

## What This Changes

The issue reports OpenAI Codex OAuth JSON parsing failures on compressed-looking token responses and a post-success CLI hang from a lingering localhost callback socket.

## Best Possible Solution

Add OpenAI Codex OAuth-owned hardening for token exchange/refresh response decoding and callback socket cleanup, while coordinating any generic gzip handling with the broader transport issue.

## Reproduction Assessment

No high-confidence live reproduction was run because the failing path requires a real OpenAI OAuth flow on macOS/Node v26. The current source and pinned dependency make the failure class source-reproducible: token responses are parsed directly as JSON and callback cleanup only closes the server.

## Solution Assessment

Unclear whether the release-branch mitigation is the final best form. The safest direction is a narrow OpenAI Codex OAuth wrapper or upstream-equivalent helper fix, not a broad global fetch/decompression change that duplicates the wider transport investigation.
