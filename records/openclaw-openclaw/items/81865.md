## Summary

Keep open: current main and the latest release still show the OpenAI/Codex OAuth token path parsing JSON after a guarded fetch without an OAuth-specific compressed-body guard, and the localhost callback cleanup still uses server.close-only behavior rather than the upstream Codex disconnect pattern.

## What This Changes

The issue reports that OpenAI Codex OAuth can parse compressed-looking token responses as JSON and can leave the auth CLI alive after a successful browser OAuth callback.

## Best Possible Solution

Add a narrow OpenAI plugin OAuth repair that prevents compressed/binary token bodies from reaching raw JSON parsing and forces the successful localhost callback connection to close, with focused regression tests and no new config surface.

## Reproduction Assessment

Yes for a source-level reproduction: current main still has the token JSON parse and callback cleanup shapes described in the issue. No live macOS/Node 26 OAuth run was performed in this read-only review.

## Solution Assessment

Yes, the best next fix is a narrow OpenAI plugin OAuth change rather than closing this as stale or duplicating the broader HTTP decompression issue. The callback cleanup should mirror the upstream Codex close-connection behavior while the token path should add an OAuth-specific parse/decode guard or identity encoding request.
