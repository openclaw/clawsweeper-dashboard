## Summary

Keep open: current main still reaches the OpenAI Codex OAuth helper for login and refresh, and the reviewed wrapper does not add OAuth-specific compressed-response handling or accepted callback socket cleanup.

## What This Changes

The issue asks to harden OpenAI Codex OAuth login/refresh against compressed or binary-looking token responses and ensure the localhost callback socket does not keep the CLI alive after success.

## Best Possible Solution

Add OpenAI Codex OAuth-owned decoding/error hardening and callback accepted-socket cleanup, or update the pi-ai helper to provide those guarantees, with focused tests plus redacted live OAuth proof.

## Reproduction Assessment

No high-confidence live reproduction was run because it requires a real OpenAI browser OAuth flow on macOS/Node v26. Source inspection shows current main still follows the reported vulnerable paths: direct pi-ai JSON parsing for token responses and server-close-only callback cleanup.

## Solution Assessment

Unclear whether the reporter's local `Accept-Encoding: identity` plus handle cleanup mitigation is the final best form. The maintainable fix should stay narrowly in the OpenAI Codex OAuth wrapper/helper path and avoid a broad global fetch decompression fallback without the broader transport evidence.
