## Summary

Keep open: current main still routes OpenAI OAuth token success responses through raw `response.json()` and still closes the localhost callback server with `server.close()` only, while upstream Codex has explicit connection-close handling for the same callback hang class. This remains a bounded OpenAI auth reliability bug, not something current main or a canonical related item has fully resolved.

## What This Changes

The issue asks OpenClaw to handle OpenAI/Codex OAuth token responses without raw compressed/binary JSON parse failures and to exit cleanly after successful OAuth callback completion.

## Best Possible Solution

Repair the OpenAI plugin OAuth flow with structured token-response parsing for decoded JSON versus non-JSON/binary-looking bodies, explicit callback connection cleanup after success, and focused regression coverage plus live OAuth proof when credentials are available.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live current-main OAuth replay was performed, but source inspection gives a clear focused repro path for both a non-JSON token response and server-close-only callback cleanup.

## Solution Assessment

Is this the best way to solve the issue? Yes: the maintainable path is a narrow OpenAI OAuth repair plus tests, while the broader guarded-fetch compressed-response class should remain with its separate canonical issue.
