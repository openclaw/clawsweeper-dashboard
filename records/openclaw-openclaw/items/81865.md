## Summary

Keep open: current main still delegates the OpenAI Codex browser OAuth token exchange, refresh, and callback-server lifecycle to the pinned pi-ai helper, and the OpenClaw wrapper does not add an OAuth-specific decoded-body guard or accepted-socket cleanup.

## What This Changes

The issue reports that OpenAI Codex OAuth can parse compressed-looking token responses as JSON bytes and can leave the CLI process running after OAuth success.

## Best Possible Solution

Add an OpenAI Codex OAuth-owned hardening layer or upstream-equivalent dependency fix for token response decoding and callback socket cleanup, with focused regression tests and redacted live OAuth proof before landing.

## Reproduction Assessment

No high-confidence live reproduction was run because the reported path requires real OpenAI OAuth on macOS/Node v26. The current source and pinned dependency make the failure class source-reproducible: token responses are parsed directly as JSON and callback cleanup does not destroy accepted sockets.

## Solution Assessment

Unclear whether the reporter's release-branch mitigation is the final best form. The maintainable fix should be narrow to the OpenAI Codex OAuth runtime or the pi-ai helper contract, not a broad global fetch behavior change that conflicts with the wider compressed-response investigation.
