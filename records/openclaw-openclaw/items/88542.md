## Summary

Keep open: this is a useful generated-media delivery fix with convincing live proof, but the PR still loses the originating source reply mode before background completion, so the deterministic prompt can choose the wrong delivery contract in message-tool-only sessions.

## What This Changes

Threads an optional sourceReplyDeliveryMode into generated-media completion wake params and rewrites image/video/music completion prompt tests to choose deterministic message-tool versus MEDIA-line instructions.

## Best Possible Solution

Preserve the resolved sourceReplyDeliveryMode from the originating run through media tool creation, scheduling, and success/error completion wake calls, with regressions for both automatic channel and message_tool_only paths.

## Reproduction Assessment

Yes at source level: current main contains the ambiguous generated-media completion instruction, and the PR body supplies live Discord logs showing ambiguous-recipient errors before the patch and no such errors after the channel-path patch. I did not run a live Discord repro in this read-only review.

## Solution Assessment

No: deterministic prompts are the right direction, but recomputing delivery mode at completion time is a lossy fallback. The safer solution is to preserve the originating sourceReplyDeliveryMode through the media tool and scheduler path.
