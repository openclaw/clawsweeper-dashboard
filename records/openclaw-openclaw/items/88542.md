## Summary

Keep open: the PR addresses a real generated-media delivery bug and the new channel proof is useful, but the production media tool path still does not preserve the original sourceReplyDeliveryMode into the background completion prompt, so message_tool_only completions can receive the wrong MEDIA-line instruction.

## What This Changes

Threads an optional sourceReplyDeliveryMode into generated-media completion wake params and rewrites image/video/music completion prompts/tests to choose deterministic message-tool versus MEDIA-line instructions.

## Best Possible Solution

Preserve the resolved sourceReplyDeliveryMode at media tool creation/task scheduling time and use it for both success and failure completion prompts, with channel and message_tool_only regression coverage.

## Reproduction Assessment

Yes at source level: current main contains the ambiguous generated-media completion instruction, and the PR body supplies live Discord logs showing ambiguous-recipient errors before the patch and no such errors after the channel-path patch. I did not run a live Discord repro in this read-only review.

## Solution Assessment

No: deterministic prompts are the right direction, but the patch should preserve the already resolved sourceReplyDeliveryMode from the originating run instead of recomputing a lossy fallback at completion time.
