## Summary

Keep open: this PR is not merge-ready because it treats any failed recovery exception as an authoritative idle transition, even though the failure path does not prove active work was cleared or absent.

## What This Changes

The PR adds `failed` recovery outcomes to the diagnostic session-state mutation guard so failed stuck-session recovery can transition diagnostic state toward idle.

## Best Possible Solution

Keep failed outcomes non-authoritative for idle transitions, and implement ghost cleanup where the code can prove no active or queued work remains, with focused regression coverage.

## Reproduction Assessment

Yes for the review finding from source inspection: failed outcomes are created in exception paths and this added guard would route them through the idle/activity-clearing path. I did not run tests or a live recovery failure because this review was read-only.

## Solution Assessment

No. The proposed one-line fix is too broad because `failed` does not have the same recovery contract as `aborted`, `released`, or `noop/no_active_work`; a safer fix should make cleanup conditional on proof that active work is gone.

- **[P1] Do not declare failed recovery outcomes idle:** `src/logging/diagnostic-session-recovery.ts:93`
  - body: Adding `failed` here routes exception outcomes through the same authoritative cleanup path as successful recovery. Those failures can be produced before the runtime proves the embedded run was aborted, drained, released, or absent, but `applyRecoveryOutcomeToDiagnosticState` will then clear recovered-owner activity and set the session state to `idle`; for remaining queued depth it may still not satisfy the prune condition. Please keep generic failures non-mutating, or introduce a narrower outcome/path that only declares idle after recovery has proven the owner is gone.
