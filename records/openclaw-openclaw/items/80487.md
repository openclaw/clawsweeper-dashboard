## Summary

Keep open: current main still appears to drop the effective model `api` before synthetic auth hook lookup, and the linked closing PR is still open, so this issue should not be closed until that fix or an equivalent lands.

## What This Changes

The issue requests that auth resolution honor a model-level `api` override when resolving provider plugin hooks for a custom provider.

## Best Possible Solution

Thread the selected model's effective `api` through the auth resolver into provider hook refs, keep provider-level behavior as the fallback, and add a regression test for a mixed provider/model API configuration.

## Reproduction Assessment

Yes, at source level: current main accepts and preserves model-level `api`, but the auth hook path only uses provider-level `api`. I did not run the live Docker/Telegram setup.

## Solution Assessment

Yes in direction: resolving synthetic auth hook refs from the effective model API is the narrowest maintainable fix. The implementation should include production caller wiring from the resolved model, not only a direct helper parameter.
