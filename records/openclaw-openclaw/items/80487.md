## Summary

Keep open: current main still resolves provider auth hooks and configured fallback transport from provider-level API/base URL values in the reported paths, while the open closing PR has not merged.

## What This Changes

The issue reports that per-model `api` and `baseUrl` overrides in custom provider model rows are accepted by config but ignored by auth and transport resolution.

## Best Possible Solution

Carry the effective model-level `api` and `baseUrl` through auth hook discovery and model transport resolution, preferably by reviewing and landing the existing closing PR after preserving the current `models-add` metadata safeguards.

## Reproduction Assessment

Yes. Source inspection on current main shows model rows accept `api`/`baseUrl`, while the auth hook and configured fallback transport paths still read provider-level values; the reporter also supplied router logs for the mixed OpenAI/Anthropic setup.

## Solution Assessment

Yes, with one caveat. Threading effective model-level transport facts through the existing resolver chain is the narrow maintainable fix, but the implementation should preserve the existing `models-add` runtime metadata precedence.
