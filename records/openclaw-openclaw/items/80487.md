## Summary

Keep open: current main still resolves synthetic auth from provider-level `api` only, while model resolution and stream registration preserve the model-level override; the linked open PR explicitly targets this bug, so the issue should remain open until that or an equivalent fix merges.

## What This Changes

The issue reports that custom-provider models with a per-model `api` override, such as `ollama`, do not use that effective API during synthetic auth hook resolution.

## Best Possible Solution

Thread the resolved model `api` into synthetic auth hook reference resolution from the model-auth entry point, including runtime callers that use `getApiKeyForModel`, and add regression coverage for a provider-level `openai-completions` model overriding to `ollama`.

## Reproduction Assessment

Yes, at source level: current main preserves a model-level API override in model resolution and stream registration, but auth hook refs only see the provider-level API. I did not run a live Docker/Ollama repro in this read-only review.

## Solution Assessment

Yes for the direction: using the effective model API during synthetic auth hook discovery is the narrow maintainable fix. The open PR at https://github.com/openclaw/openclaw/pull/80488 should be reviewed to ensure runtime callers pass `model.api` and provide after-fix proof.
