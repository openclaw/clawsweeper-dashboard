## Summary

Keep open. Current main and the latest release still contain the source-level collect-mode behavior described here, and the broader lifecycle-notification request remains separate from this narrower batching bug.

## What This Changes

The issue reports that collect-mode queue batching is skipped when a queued batch contains an unresolved-origin item, causing compatible subagent announce messages to drain individually and potentially disappear from the requester transcript.

## Best Possible Solution

Patch the collect queue contract so unresolved-origin items can join an otherwise single resolved route batch, reset force-individual state at the correct drain scope, and cover explicit cross-channel/runtime safeguards with regression tests.

## Reproduction Assessment

Yes for source-level reproduction: current main classifies a keyed batch with one unkeyed item as cross-channel, and the force-individual collect state is shared across the drain. I did not run the full live multi-subagent scenario.

## Solution Assessment

Yes, the narrow two-part queue fix is the best fit for the existing collect-mode contract. It should not wait for the broader lifecycle notification redesign in https://github.com/openclaw/openclaw/issues/66638.
