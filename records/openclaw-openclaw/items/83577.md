## Summary

Keep open. Current main and the latest release still contain the reported collect-mode source behavior, and the broader architecture request at https://github.com/openclaw/openclaw/issues/66638 does not supersede this narrower bug-level fix.

## What This Changes

The issue reports that collect-mode subagent announce batching is skipped when any queued announce has unresolved origin routing, causing queued completion announces to drain individually and risk silent loss while the requester session is busy.

## Best Possible Solution

Patch the existing collect queue contract so unresolved-origin items can join a single resolved route batch, while explicit cross-channel and runtime-context items still drain individually with regression coverage.

## Reproduction Assessment

Yes for source-level reproduction: a keyed-plus-unkeyed queue makes `hasCrossChannelItems` return true on current main, and collect force state is shared across the drain loop. I did not run the full live multi-subagent scenario in this read-only pass.

## Solution Assessment

Yes, the issue’s narrow two-part direction fits the existing contract better than waiting for the broader redesign. The fix should include tests that preserve explicit cross-channel splitting and runtime-metadata individual draining.
