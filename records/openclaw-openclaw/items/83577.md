## Summary

The collect-mode batching bug still appears present on current main and is not superseded by the broader lifecycle-notification request or the adjacent completion timestamp fix; source inspection shows mixed keyed/unresolved-origin batches still force individual draining, and the latest shipped release contains the same helper behavior.

## What This Changes

The issue requests fixing collect-mode subagent announce batching so unresolved-origin queued items do not force otherwise compatible same-channel batches into individual draining.

## Best Possible Solution

Patch the collect queue contract so unresolved-origin items can join an otherwise single resolved route batch, reset force-individual state at the correct drain scope, and add focused regression tests for unresolved joins plus explicit cross-channel/runtime splits.

## Reproduction Assessment

Yes for source-level reproduction: current main returns cross-channel for a keyed collect batch containing one unkeyed item, and collect force state is shared across drain iterations. I did not run the full live multi-subagent scenario.

## Solution Assessment

Yes, the issue's narrow two-part fix direction matches the existing collect-mode contract better than waiting for the broader lifecycle-notification redesign. The implementation should add regression coverage that keeps real cross-channel/runtime metadata safeguards intact.
