## Summary

Keep open. Current `main` and the latest release still contain the reported collect-mode source behavior, and the broader lifecycle-notification issue does not supersede this narrower batching/message-loss bug.

## What This Changes

The issue reports that collect-mode subagent announce batching is skipped when any queued item has an unresolved origin, causing compatible announces to drain individually and risk silent transcript/message loss.

## Best Possible Solution

Patch the collect queue contract so unresolved-origin items can join an otherwise single resolved-route batch, reset force-individual state at the right drain scope, and preserve explicit cross-channel/runtime metadata safeguards with tests.

## Reproduction Assessment

Yes for source-level reproduction: current `main` marks a keyed batch containing one unkeyed item as cross-channel, and collect force state is shared across the drain loop. I did not run the full live multi-subagent scenario.

## Solution Assessment

Yes, the narrow two-part fix direction fits the existing collect-mode contract better than waiting for the broader notification redesign. The repair should add focused regressions for unresolved-origin joins and explicit cross-channel/runtime splits.
