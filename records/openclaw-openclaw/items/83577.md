## What This Changes

The issue asks OpenClaw to preserve collect-mode batching for same-channel queued subagent announces when some queued items have unresolved origin metadata.

## Best Possible Solution

Patch the collect queue contract so unresolved-origin items can join an otherwise single resolved route batch, reset force-individual state at the right drain scope, and preserve explicit cross-channel/runtime metadata safeguards with tests.

## Reproduction Assessment

Yes for source-level reproduction: current main returns cross-channel for a keyed batch containing one unkeyed item, and collect force state is shared across the drain loop. I did not run the full live multi-subagent scenario.

## Solution Assessment

Yes, the issue’s narrow two-part direction fits the existing collect-mode contract better than waiting for the broader notification redesign. The implementation should add focused regression tests for unresolved-origin joins and explicit cross-channel/runtime splits.

## Repair Work Prompt

Fix the collect-mode queued announce batching bug tracked by https://github.com/openclaw/openclaw/issues/83577. In `src/utils/queue-helpers.ts`, stop treating unkeyed-but-non-cross items as cross-channel when the rest of the batch resolves to one route; explicit `cross` and multiple distinct route keys must still force individual delivery. In `src/auto-reply/reply/queue/drain.ts`, scope `forceIndividualCollect` so one cross-channel/runtime item does not permanently poison later compatible batches in the same drain beyond the intended safety boundary. Add regression coverage for keyed-plus-unkeyed same-route collection, all-unkeyed collection, distinct-route splitting, explicit cross-channel splitting, runtime-context individual draining, and preservation of existing auth split behavior. Do not redesign subagent lifecycle delivery or add new config. Add a single `CHANGELOG.md` fixes entry if this user-visible repair lands.
