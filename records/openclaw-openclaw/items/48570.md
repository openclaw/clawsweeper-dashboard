## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the reactor key for reaction sender fields:** `extensions/whatsapp/src/inbound/monitor.ts:517-523`
  - body: Baileys emits `messages.reaction` with the outer `key` as the reacted-to message and `reaction.key` as the reaction message, but this code prefers `key.participant` and `key.fromMe`. In groups that can attribute the reaction to the target message author and run access control as the wrong sender.
  - confidence: 0.92
- **[P1] Keep reactions out of ungated reply turns:** `extensions/whatsapp/src/inbound/monitor.ts:547-550`
  - body: Every reaction is converted into a normal inbound message and enqueued. WhatsApp currently has no `reactionNotifications` or hook-only opt-in, so emoji reactions can start model turns or outbound replies instead of using a deliberate listen-only reaction surface.
  - confidence: 0.88
- **[P1] Preserve poll vote data before enqueueing updates:** `extensions/whatsapp/src/inbound/monitor.ts:472`
  - body: The loop receives each Baileys `pollUpdate`, but the synthetic message replaces it with `pollUpdateMessage: {}`. Downstream hooks and agents can only see a generic placeholder, so they still cannot read selected options from poll votes.
  - confidence: 0.93
- **[P2] Route event RSVP updates from messages.update:** `extensions/whatsapp/src/inbound/monitor.ts:444-445`
  - body: This handler exits unless `update.pollUpdates` is present, but Baileys also emits event RSVPs as `update.eventResponses`. Those RSVP updates never reach the new RSVP extractor, leaving event response support incomplete.
  - confidence: 0.86
- **[P2] Recognize the current poll creation variant:** `extensions/whatsapp/src/inbound/extract.ts:147-153`
  - body: Baileys 7.0.0-rc11 exposes `pollCreationMessageV5`, but the extractor checks only V1/V2/V3. Polls from newer WhatsApp clients can still return `undefined` here and be dropped by enrichment.
  - confidence: 0.86
- **[P2] Distinguish unknown RSVP states from maybe:** `extensions/whatsapp/src/inbound/extract.ts:187-190`
  - body: `EventResponseType.UNKNOWN` is `0` and `MAYBE` is `3`, but every non-going/non-not-going value is mapped to `maybe`. Empty or unknown RSVP updates would be reported as tentative attendees and skew downstream logic.
  - confidence: 0.9
- **[P1] Honor the WhatsApp message hook opt-in:** `extensions/whatsapp/src/auto-reply/monitor/on-message.ts:153-154`
  - body: Current main only emits WhatsApp message_received hooks after the `pluginHooks.messageReceived` opt-in, but this early-return path fires an internal hook for mention-gated group messages unconditionally. That exposes unmentioned group content despite the existing hook boundary.
  - confidence: 0.89
