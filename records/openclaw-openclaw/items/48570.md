## Summary

Keep open: current main does not implement the central WhatsApp poll/reaction/event/RSVP inbound capability, but this PR is not merge-ready because it still has blocking payload, identity, hook-privacy, proof, and rebase issues.

## What This Changes

The PR adds WhatsApp inbound extraction and listeners for polls, reactions, events, and RSVPs, fires gated group-message hooks, adds tests, and also changes gateway timeout, poll-parameter, and tsdown build-entry behavior.

## Best Possible Solution

Ship a focused WhatsApp-owned implementation that follows Baileys update/reaction contracts, preserves payloads and sender identity, respects hook opt-in privacy, keeps unrelated stale changes out, adds targeted tests/docs, and includes redacted live WhatsApp proof.

## Reproduction Assessment

Yes for source-level review: current main lacks the WhatsApp update/reaction/event ingress paths, and the PR diff plus pinned Baileys contracts expose the failing payload, RSVP, identity, and hook-privacy paths. I did not run live WhatsApp validation in this read-only review.

## Solution Assessment

No. The feature direction is useful, but this PR is too broad and should not merge until it preserves poll/RSVP data, uses the correct Baileys identity fields, respects hook privacy gating, avoids stale poll-param regression, rebases cleanly, and supplies real WhatsApp proof.

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve poll vote payloads:** `extensions/whatsapp/src/inbound/monitor.ts:472`
  - body: The messages.update handler receives each pollUpdate, but the synthetic message always uses an empty pollUpdateMessage. Agents and hooks would only see a generic poll-vote placeholder and cannot learn which option data was selected.
  - confidence: 0.94
- **[P1] Handle RSVP updates from messages.update:** `extensions/whatsapp/src/inbound/monitor.ts:444-445`
  - body: This handler skips every update that lacks pollUpdates, but Baileys emits decrypted event RSVP data as update.eventResponses. RSVP updates delivered through messages.update would still be dropped.
  - confidence: 0.9
- **[P1] Respect the WhatsApp hook opt-in gate:** `extensions/whatsapp/src/auto-reply/monitor/on-message.ts:153-154`
  - body: The gated group-message path calls triggerInternalHook directly instead of the WhatsApp hook emitter that checks pluginHooks.messageReceived. Configurations without the WhatsApp hook opt-in can still receive unmentioned group content through internal hooks.
  - confidence: 0.86
- **[P2] Derive reaction sender from the reaction event key:** `extensions/whatsapp/src/inbound/monitor.ts:517-523`
  - body: Baileys emits the top-level key as the reacted message key and puts the actual reaction message key on reaction.key. Preferring key.participant/fromMe can attribute a reaction to the original message author or mark a peer reaction as fromMe.
  - confidence: 0.88
- **[P2] Recognize pollCreationMessageV5:** `extensions/whatsapp/src/inbound/extract.ts:147-153`
  - body: The pinned Baileys proto exposes pollCreationMessageV5, but this extractor checks only V1, V2, and V3. V5 poll creation messages can still return undefined and be dropped by enrichment.
  - confidence: 0.9
- **[P2] Preserve unknown RSVP states:** `extensions/whatsapp/src/inbound/extract.ts:187-188`
  - body: EventResponseType.UNKNOWN is distinct from MAYBE, but this mapping treats every non-going and non-not-going value as maybe. Empty or unknown RSVP updates would be reported as tentative attendance.
  - confidence: 0.89
- **[P2] Preserve negative poll durations on validation path:** `src/poll-params.ts:59-60`
  - body: Current main treats non-zero numeric poll params, including invalid negative durations, as explicit poll intent so they hit poll-only validation. Changing the number branch to value > 0 lets negative numeric durations bypass that validation path.
  - confidence: 0.84
- **[P2] Include event time and location in event placeholders:** `extensions/whatsapp/src/inbound/extract.ts:174-179`
  - body: EventMessage carries startTime and location, but this placeholder only emits the name, description, and cancel flag. Downstream hooks and agents still cannot tell when or where the event is happening.
  - confidence: 0.78
