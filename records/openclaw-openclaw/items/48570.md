## Summary

Keep open: current main still lacks the WhatsApp poll/reaction/event/RSVP ingress paths, but this PR is not mergeable as-is because source and Baileys contract review show payload, RSVP, reaction-identity, and hook-privacy defects plus missing real WhatsApp proof.

## What This Changes

The PR adds WhatsApp inbound extraction/listeners for polls, reactions, events, and RSVPs, emits gated group hooks, updates tests, and also changes gateway timeout, poll-parameter, and build-entry behavior.

## Best Possible Solution

Ship a focused WhatsApp-owned design that preserves Baileys poll/RSVP/reaction payloads and identity, respects the existing privacy gate, aligns with the open reaction trackers, adds targeted tests/docs, and includes redacted live WhatsApp proof.

## Reproduction Assessment

Yes for source-level review: current main lacks these WhatsApp ingress paths, and the PR diff plus Baileys v7.0.0-rc11 contracts show the payload, RSVP, identity, and hook-privacy failures. I did not run live WhatsApp validation in this read-only review.

## Solution Assessment

No. The direction is useful, but this PR should not merge until it preserves poll/RSVP payloads, uses the correct reaction actor key, respects the WhatsApp hook opt-in boundary, drops stale side changes, and supplies real WhatsApp proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve poll vote payloads:** `extensions/whatsapp/src/inbound/monitor.ts:472`
  - body: The `messages.update` handler receives each Baileys `pollUpdate`, including the encrypted vote payload, but the synthetic message always uses an empty `pollUpdateMessage`. Hooks and agents would only see a generic poll-vote placeholder and cannot learn which option data was selected.
  - confidence: 0.94
- **[P1] Route RSVP updates from messages.update:** `extensions/whatsapp/src/inbound/monitor.ts:444-445`
  - body: This handler skips every update without `pollUpdates`, but Baileys WAMessage updates also expose `eventResponses`. RSVP updates delivered through `messages.update` would still be dropped before the new RSVP extraction can run.
  - confidence: 0.9
- **[P1] Respect the WhatsApp hook opt-in gate:** `extensions/whatsapp/src/auto-reply/monitor/on-message.ts:153-154`
  - body: The gated group-message path calls `triggerInternalHook` directly instead of the WhatsApp hook emitter that checks `pluginHooks.messageReceived`. Configurations without that opt-in can still receive unmentioned group content through hooks.
  - confidence: 0.9
- **[P2] Derive reaction sender from reaction.key:** `extensions/whatsapp/src/inbound/monitor.ts:517-523`
  - body: Baileys emits the outer `key` as the reacted-message key and puts the reaction event key on `reaction.key`. Preferring `key.participant` and `key.fromMe` can attribute a reaction to the original message author or mark a peer reaction with the wrong `fromMe` value.
  - confidence: 0.88
- **[P2] Recognize pollCreationMessageV5:** `extensions/whatsapp/src/inbound/extract.ts:147-153`
  - body: The pinned Baileys proto exposes `pollCreationMessageV5`, but the extractor checks only V1, V2, and V3. V5 poll creation messages can still return undefined and be dropped by enrichment.
  - confidence: 0.9
- **[P2] Preserve unknown RSVP states:** `extensions/whatsapp/src/inbound/extract.ts:187-188`
  - body: `EventResponseType.UNKNOWN` is distinct from `MAYBE`, but this mapping treats every non-going and non-not-going value as maybe. Empty or unknown RSVP updates would be reported as tentative attendance.
  - confidence: 0.89
- **[P2] Keep negative poll durations on the validation path:** `src/poll-params.ts:59`
  - body: Current main treats non-zero numeric poll params, including invalid negative durations, as explicit poll intent so they hit poll-only validation. Changing the number branch to `value > 0` lets negative numeric durations bypass that validation path.
  - confidence: 0.84
- **[P2] Include event time and location:** `extensions/whatsapp/src/inbound/extract.ts:176-179`
  - body: `EventMessage` carries `startTime`, `endTime`, and `location`, but this placeholder only emits the name, description, and cancel flag. Downstream hooks and agents still cannot tell when or where the event is happening.
  - confidence: 0.78
