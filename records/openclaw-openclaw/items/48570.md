## Summary

Keep open and do not merge as-is. Current main still lacks the requested WhatsApp poll/reaction/event inbound support, but this PR is stale, mixes unrelated core changes, and the proposed implementation loses important Baileys update data while bypassing the existing WhatsApp hook privacy opt-in.

## What This Changes

The PR adds WhatsApp inbound placeholder extraction and Baileys update/reaction listeners for polls, reactions, events, and RSVPs, fires message:received hooks for gated group messages, adds tests, and also changes gateway timeout, poll-parameter, and build-entry behavior.

## Best Possible Solution

Land a WhatsApp-owned implementation that preserves Baileys poll vote and RSVP payload details, handles V5 poll creation and reaction/update streams with correct sender identity, keeps WhatsApp message hooks behind the documented pluginHooks.messageReceived opt-in, and splits unrelated gateway timeout, poll-parameter, and build-entry changes into separate focused PRs if still needed.

## Reproduction Assessment

Yes for source-level review: current main and the PR diff show the missing listeners/placeholders, dropped poll payload, RSVP update gap, and hook privacy bypass without a live WhatsApp account. Live reproduction would still be useful before merge for Baileys identity and encrypted update details.

## Solution Assessment

No. The feature direction is valid, but this patch is not the best solution because it is broad and stale, loses required update data, bypasses the WhatsApp privacy gate, and mixes unrelated core/tool/build changes into the channel feature.

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve poll update payloads:** `extensions/whatsapp/src/inbound/monitor.ts:472`
  - body: `messages.update` supplies each voter's poll update, including selected option data, but the synthetic message always uses `pollUpdateMessage: {}`. Hooks and agents can only see a generic poll-vote placeholder, so the poll-response use case remains unresolved.
  - confidence: 0.94
- **[P1] Handle RSVP updates from messages.update:** `extensions/whatsapp/src/inbound/monitor.ts:444-446`
  - body: This handler exits whenever `pollUpdates` is empty, but Baileys carries event RSVP changes on `update.eventResponses`. Those RSVP updates never become inbound messages, so the advertised RSVP support misses the update-stream path.
  - confidence: 0.88
- **[P1] Respect the WhatsApp hook privacy gate:** `extensions/whatsapp/src/auto-reply/monitor/on-message.ts:154-156`
  - body: This direct `message:received` hook call bypasses the WhatsApp `pluginHooks.messageReceived` opt-in used by the normal hook emitter. A config that keeps WhatsApp message payloads out of hooks would still deliver unmentioned group content through this early return path.
  - confidence: 0.82
- **[P2] Recognize V5 poll creation messages:** `extensions/whatsapp/src/inbound/extract.ts:148-152`
  - body: The pinned Baileys proto exposes `pollCreationMessageV5`, but this extraction only checks V1, V2, and V3. V5 poll creation messages will still return `undefined` and be dropped by `enrichInboundMessage()`.
  - confidence: 0.9
- **[P2] Keep unknown RSVP states distinct:** `extensions/whatsapp/src/inbound/extract.ts:189-190`
  - body: `EventResponseType.UNKNOWN` is `0`, but this mapping treats every non-going/non-not-going value as `maybe`. Empty or unknown RSVP updates would be reported as tentative attendance and can skew automation that counts actual RSVP states.
  - confidence: 0.89
- **[P2] Keep negative poll durations on the poll path:** `src/poll-params.ts:57`
  - body: Changing numeric poll detection to `value > 0` means invalid negative durations are no longer treated as poll intent and can bypass the poll-only validation path. Current main preserves all non-zero finite values so invalid durations are still validated as poll requests.
  - confidence: 0.84
