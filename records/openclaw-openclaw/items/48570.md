## Summary

Keep open. Current main still lacks the requested WhatsApp poll/reaction/event/RSVP inbound support, and this PR is the broadest open implementation attempt, but the branch is currently dirty against main and the proposed patch still has blocking correctness and privacy issues.

## What This Changes

The PR adds WhatsApp inbound placeholders and event handlers for polls, reactions, events, RSVPs, gated group hook emission, tests, plus unrelated gateway timeout, poll-parameter, and build-entry changes.

## Best Possible Solution

Land a WhatsApp-owned implementation that preserves Baileys payload details and sender identity, handles V5 poll creation plus pollUpdates/eventResponses/reaction streams, respects WhatsApp hook privacy configuration, cleans up listeners through current lifecycle helpers, and drops unrelated core/tool/build changes from this feature PR.

## Reproduction Assessment

Yes for the source-level review: current main and the PR diff show the missing handlers, dropped poll payload, RSVP mapping issue, and gated hook bypass without needing a live WhatsApp account. Live end-to-end reproduction would still be useful before merge for Baileys identity and update-stream details.

## Solution Assessment

No. The requested direction is valid, but this patch is not the best merge shape because it is stale, broad, mixes unrelated changes, and still loses required poll/RSVP data while bypassing current WhatsApp hook privacy policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve poll update payloads:** `extensions/whatsapp/src/inbound/monitor.ts:472`
  - body: `messages.update` supplies each voter's `pollUpdate`, including selected options, but the synthetic message always uses `pollUpdateMessage: {}`. Downstream hooks and agents can only see a generic `<media:poll-vote>` and cannot tell which option was selected, leaving the poll-response use case unresolved.
  - confidence: 0.94
- **[P1] Handle RSVP updates from messages.update:** `extensions/whatsapp/src/inbound/monitor.ts:444-446`
  - body: This handler exits whenever `pollUpdates` is empty, but Baileys carries event RSVP changes on `update.eventResponses`. Those RSVP updates never become inbound messages, so the advertised RSVP support misses the update-stream path.
  - confidence: 0.88
- **[P1] Respect the WhatsApp hook privacy gate:** `extensions/whatsapp/src/auto-reply/monitor/on-message.ts:154-156`
  - body: This direct `message:received` hook call bypasses the WhatsApp `pluginHooks.messageReceived` opt-in used by the normal WhatsApp hook emitter. A config that keeps WhatsApp message payloads out of hooks would still deliver unmentioned group content through this early return path.
  - confidence: 0.82
- **[P2] Recognize V5 poll creation messages:** `extensions/whatsapp/src/inbound/extract.ts:148-152`
  - body: The pinned Baileys proto exposes `pollCreationMessageV5`, but this extraction only checks V1, V2, and V3. V5 poll creation messages will still return `undefined` and be dropped by `enrichInboundMessage()`.
  - confidence: 0.9
- **[P2] Keep unknown RSVP states distinct:** `extensions/whatsapp/src/inbound/extract.ts:187-190`
  - body: `EventResponseType.UNKNOWN` is `0`, but this mapping treats every non-going/non-not-going value as `maybe`. Empty or unknown RSVP updates would be reported as tentative attendance and can skew automation that counts actual RSVP states.
  - confidence: 0.89
- **[P2] Keep negative poll durations on the poll path:** `src/poll-params.ts:57`
  - body: Changing numeric poll detection to `value > 0` means invalid negative durations are no longer treated as poll intent and can bypass the poll-only validation path. Current main preserves all non-zero finite values so invalid durations are still validated as poll requests.
  - confidence: 0.84
