## Summary

Keep open. ## What This Changes

The PR adds WhatsApp placeholder extraction and synthetic inbound handling for polls, reactions, events/RSVPs, gated group message hooks, plus unrelated gateway timeout, poll-parameter, and build-entry changes.

## Best Possible Solution

Rebase on current main and implement a narrow WhatsApp-owned event surface that preserves Baileys payloads, gates reactions with explicit config/hook semantics, and proves the live WhatsApp flows.

## Reproduction Assessment

Yes, for the blocking PR defects we have high-confidence source-level reproduction: the diff discards `pollUpdate.vote`, ignores `eventResponses`, and routes every reaction through `enqueueInboundMessage`. I did not run a live WhatsApp account, so the original user-visible behavior is not live-reproduced here.

## Solution Assessment

No. The current PR is too broad and routes low-friction WhatsApp events as normal agent messages without preserving enough payload or defining opt-in/hook semantics; a narrower current-main implementation with live proof is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve poll vote payloads:** `extensions/whatsapp/src/inbound/monitor.ts:472`
  - body: `messages.update` gives each `pollUpdate` with the vote payload, but the synthetic message discards it and sends `pollUpdateMessage: {}`. Downstream code can only see `<media:poll-vote>`, so agents/hooks still cannot know which option was selected, which is the central poll-response use case.
  - confidence: 0.9
- **[P1] Gate reaction events before starting agent turns:** `extensions/whatsapp/src/inbound/monitor.ts:547`
  - body: Every `messages.reaction` entry is converted into a normal inbound message and enqueued, with no `reactionNotifications`-style `off | own | all` policy or hook-only path. On existing WhatsApp groups, a user emoji reaction can now trigger an agent run and possible outbound reply instead of being a passive notification.
  - confidence: 0.88
- **[P2] Handle RSVP updates from messages.update:** `extensions/whatsapp/src/inbound/monitor.ts:444-445`
  - body: The new update handler returns unless `pollUpdates` is present, but Baileys also carries event RSVP changes on `WAMessageUpdate.eventResponses`. Those RSVP updates will never synthesize an inbound message, so the new `<event-rsvp:...>` extraction is unreachable for that delivery path.
  - confidence: 0.84
- **[P2] Recognize the current poll creation variant:** `extensions/whatsapp/src/inbound/extract.ts:147-153`
  - body: The repo pins Baileys 7.0.0-rc11, whose `proto.IMessage` includes `pollCreationMessageV5`, but this branch only checks V1/V2/V3. Polls sent in the current V5 shape will still return `undefined` from `extractMediaPlaceholder()` and be dropped.
  - confidence: 0.89
- **[P2] Keep UNKNOWN RSVP distinct from maybe:** `extensions/whatsapp/src/inbound/extract.ts:187-188`
  - body: Baileys defines `UNKNOWN = 0` and `MAYBE = 3`, but this mapping treats every non-going/non-not-going value as `maybe`. Empty or unknown RSVP updates would be reported as tentative attendees, which corrupts event response state for hooks and agents.
  - confidence: 0.87
