  - body: The `messages.update` payload gives this loop each voter's `pollUpdate.vote`, but the synthetic message replaces it with an empty `pollUpdateMessage`.   - body: This handler exits unless `update.pollUpdates` is present, but Baileys update records also carry `eventResponses`. RSVPs delivered as update events never reach the new RSVP extractor, so event response support remains partial.
  - confidence: 0.84
- **[P1] Keep reaction events out of ungated reply turns:** `extensions/whatsapp/src/inbound/monitor.ts:547`
  - body: Every reaction is converted into a normal inbound message and enqueued into the auto-reply path. Existing reaction-notification patterns are config-controlled/listen-only; this can start model turns or outbound replies from emoji reactions instead of exposing a deliberate reaction event surface.
  - confidence: 0.82
- **[P2] Recognize the current poll creation variant:** `extensions/whatsapp/src/inbound/extract.ts:148-152`
  - body: Baileys 7.0.0-rc11 exposes `pollCreationMessageV5`, but the new extractor checks only V1/V2/V3. Polls from newer WhatsApp clients can still return `undefined` here and be dropped by enrichment.
  - confidence: 0.78
- **[P2] Distinguish unknown RSVP states from maybe:** `extensions/whatsapp/src/inbound/extract.ts:187-188`
  - body: `EventResponseType.UNKNOWN` is `0` and `MAYBE` is `3`, but this mapping sends every non-going/non-not-going response as `maybe`. Empty or unknown updates will be reported as tentative attendees and can corrupt downstream RSVP logic.
  - confidence: 0.9
- **[P2] Use the plugin-sdk hook runtime seam:** `extensions/whatsapp/src/auto-reply/monitor/on-message.ts:5-9`
  - body: This extension production file adds direct imports from `src/hooks/*`. Bundled plugins are supposed to use public/plugin-sdk seams, and current main already routes WhatsApp message hooks through `openclaw/plugin-sdk/hook-runtime`, so this will conflict on rebase or reintroduce a plugin boundary violation.
  - confidence: 0.76
