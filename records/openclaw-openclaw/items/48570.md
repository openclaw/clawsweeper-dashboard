## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve poll vote data before enqueueing updates:** `extensions/whatsapp/src/inbound/monitor.ts:472`
  - body: The handler iterates real `pollUpdates`, but then builds `pollUpdateMessage: {}` for every synthetic inbound message. Downstream hooks and agents only see `<media:poll-vote>` and lose the selected option data that the linked poll workflows need.
  - confidence: 0.88
- **[P1] Keep reactions out of normal reply turns:** `extensions/whatsapp/src/inbound/monitor.ts:547`
  - body: Reaction events are synthesized as regular inbound messages and enqueued through the auto-reply pipeline. That can create agent runs or hook delivery for lightweight reactions instead of the current listen-only, opt-in WhatsApp hook contract.
  - confidence: 0.82
- **[P1] Do not bypass the WhatsApp hook opt-in for gated groups:** `extensions/whatsapp/src/auto-reply/monitor/on-message.ts:145-169`
  - body: This early return path fires `message:received` internal hooks for unmentioned group messages even though current WhatsApp docs and tests require explicit hook opt-in before broadcasting private group content.
  - confidence: 0.78
- **[P2] Route event RSVP updates from messages.update:** `extensions/whatsapp/src/inbound/monitor.ts:444-445`
  - body: Baileys exposes event RSVP updates on `update.eventResponses`, but this handler immediately skips updates without `pollUpdates`. Those RSVP updates still never become inbound messages, so the advertised RSVP support is incomplete.
  - confidence: 0.85
- **[P2] Recognize Baileys V5 poll creation messages:** `extensions/whatsapp/src/inbound/extract.ts:146-151`
  - body: The pinned Baileys proto includes `pollCreationMessageV5`, but this extraction only checks V1/V2/V3. Newer WhatsApp poll creations can still return `undefined` and be dropped by inbound enrichment.
  - confidence: 0.86
- **[P2] Distinguish unknown RSVP states from maybe:** `extensions/whatsapp/src/inbound/extract.ts:183-184`
  - body: Baileys defines `UNKNOWN = 0` and `MAYBE = 3`, but the fallback maps every non-going and non-not-going response to `maybe`. Empty or unknown RSVP updates would be reported as tentative attendance.
  - confidence: 0.82
- **[P2] Preserve current poll validation for invalid non-zero durations:** `src/poll-params.ts:56`
  - body: Current main treats non-zero numeric poll duration values, including invalid negatives, as explicit poll intent so validation rejects them on the poll path. This stale `> 0` check would let negative numeric defaults bypass that validation.
  - confidence: 0.74
