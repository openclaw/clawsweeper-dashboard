## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve poll update payloads:** `extensions/whatsapp/src/inbound/monitor.ts:472`
  - body: The handler receives each Baileys poll update, including the vote payload, but the synthetic message always uses an empty `pollUpdateMessage`. Hooks and agents would only see a generic poll-vote placeholder and cannot learn which option data was selected.
  - confidence: 0.94
- **[P1] Route eventResponses from update events:** `extensions/whatsapp/src/inbound/monitor.ts:444-445`
  - body: This handler skips every update without `pollUpdates`, but Baileys emits decrypted event RSVPs as `update.eventResponses`. Those RSVP updates would still be dropped before the new RSVP extraction can run.
  - confidence: 0.92
- **[P1] Respect the WhatsApp hook opt-in gate:** `extensions/whatsapp/src/auto-reply/monitor/on-message.ts:154`
  - body: The gated group-message path calls `triggerInternalHook` directly instead of the WhatsApp hook emitter that checks `pluginHooks.messageReceived`. Configurations without that opt-in can still expose unmentioned group content through hooks.
  - confidence: 0.91
- **[P1] Use the reaction event key for sender identity:** `extensions/whatsapp/src/inbound/monitor.ts:517-523`
  - body: Baileys emits the outer `key` as the reacted-message key and puts the reaction event key on `reaction.key`. Preferring `key.participant` and `key.fromMe` can attribute a reaction to the original message author or mark a peer reaction with the wrong `fromMe` value.
  - confidence: 0.9
- **[P2] Recognize pollCreationMessageV5:** `extensions/whatsapp/src/inbound/extract.ts:147-153`
  - body: The pinned Baileys proto exposes `pollCreationMessageV5`, but the extractor checks only V1, V2, and V3. V5 poll creation messages can still return undefined and be dropped by enrichment.
  - confidence: 0.9
- **[P2] Preserve unknown RSVP states:** `extensions/whatsapp/src/inbound/extract.ts:187-188`
  - body: Baileys defines `EventResponseType.UNKNOWN = 0` and `MAYBE = 3`, but this mapping treats every non-going and non-not-going value as maybe. Empty or unknown RSVP updates would be reported as tentative attendance.
  - confidence: 0.89
- **[P2] Keep negative poll durations on the validation path:** `src/poll-params.ts:59`
  - body: Current main treats non-zero numeric poll params, including invalid negative durations, as explicit poll intent so they hit poll-only validation. Changing the number branch to `value > 0` lets negative numeric durations bypass that validation path.
  - confidence: 0.84
