## Summary

Keep open for maintainer review: current main still lacks the central WhatsApp poll/reaction/event ingress, but this external PR is not mergeable as-is because it lacks real behavior proof and has source-proven correctness and privacy blockers.

## What This Changes

The PR adds WhatsApp inbound handling for polls, poll votes, reactions, events, RSVPs, gated-group hook emission, tests, and unrelated gateway timeout, poll-parameter, and build-entry changes.

## Best Possible Solution

Ship a focused WhatsApp-owned implementation that handles Baileys poll, event, and reaction update payloads through the current inbound/listen-only hook architecture, preserves privacy opt-in and access control, adds targeted tests/docs, and includes redacted live WhatsApp proof.

## Reproduction Assessment

Yes for source-level review: current main lacks these WhatsApp update/reaction/event ingress paths, and the PR head plus Baileys contracts show the specific failing payload and privacy paths. I did not run live WhatsApp validation in this read-only review.

## Solution Assessment

No. The feature direction is useful, but this PR is too broad, loses poll and RSVP data, bypasses the WhatsApp hook privacy gate, regresses current poll-param validation behavior, and overlaps open reaction-surface decisions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve poll update payloads:** `extensions/whatsapp/src/inbound/monitor.ts:472`
  - body: The messages.update handler receives each pollUpdate, but the synthetic message always uses an empty pollUpdateMessage. Agents and hooks would only see a generic poll-vote placeholder and cannot learn which option data was selected.
  - confidence: 0.94
- **[P1] Handle RSVP updates from messages.update:** `extensions/whatsapp/src/inbound/monitor.ts:444-445`
  - body: This handler skips every update that lacks pollUpdates, so Baileys update.eventResponses payloads never become inbound messages. RSVP updates delivered through messages.update would still be dropped.
  - confidence: 0.9
- **[P1] Respect the WhatsApp hook opt-in gate:** `extensions/whatsapp/src/auto-reply/monitor/on-message.ts:153-154`
  - body: The gated group-message path calls triggerInternalHook directly instead of the WhatsApp hook emitter that checks pluginHooks.messageReceived. Configurations without the WhatsApp hook opt-in can still receive unmentioned group content through internal hooks.
  - confidence: 0.86
- **[P2] Derive reaction sender from the reaction event key:** `extensions/whatsapp/src/inbound/monitor.ts:517-523`
  - body: Baileys emits the top-level key as the reacted message key and puts the actual reaction message key on reaction.key. Preferring key.participant/fromMe can attribute a reaction to the original message author or mark a peer reaction as fromMe, breaking access-control and sender metadata.
  - confidence: 0.88
- **[P2] Recognize V5 poll creation messages:** `extensions/whatsapp/src/inbound/extract.ts:148-152`
  - body: The pinned Baileys proto exposes pollCreationMessageV5, but this extractor checks only V1, V2, and V3. V5 poll creation messages can still return undefined and be dropped by enrichment.
  - confidence: 0.9
- **[P2] Keep unknown RSVP states distinct:** `extensions/whatsapp/src/inbound/extract.ts:187-188`
  - body: EventResponseType.UNKNOWN is distinct from MAYBE, but this mapping treats every non-going and non-not-going value as maybe. Empty or unknown RSVP updates would be reported as tentative attendance.
  - confidence: 0.89
- **[P2] Preserve negative poll durations on the validation path:** `src/poll-params.ts:57-60`
  - body: Current main treats non-zero numeric poll params, including invalid negative durations, as explicit poll intent so they hit poll-only validation. Changing the number branch to value > 0 lets negative numeric durations bypass that validation path.
  - confidence: 0.84
