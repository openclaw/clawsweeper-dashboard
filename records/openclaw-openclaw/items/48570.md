## Summary

Keep open. Current main still lacks the requested WhatsApp poll/reaction/event/RSVP inbound support, but this PR is not merge-ready because the branch loses update payloads, misses Baileys message variants, bypasses the documented WhatsApp hook opt-in path, and mixes unrelated core/build changes into a stale channel feature.

## What This Changes

The PR branch adds WhatsApp extractors/listeners/tests for poll, reaction, event, and RSVP inbound messages, emits hooks for gated group messages, and changes gateway timeout, poll-parameter detection, and build entries.

## Best Possible Solution

Ship a focused WhatsApp-owned implementation that preserves poll/RSVP payload data, handles current Baileys poll/reaction/event variants, respects the documented hook opt-in, and leaves unrelated gateway/build/tool fixes to separate PRs.

## Reproduction Assessment

Yes for source-level review: current main lacks the new extraction and dedicated event-listener paths, and Baileys types show the update payload shapes the implementation must preserve. I did not run live WhatsApp validation in this read-only sweep.

## Solution Assessment

No. The feature direction is valid, but this PR is too broad and currently loses poll/RSVP data, misses V5 poll creation, bypasses the WhatsApp hook privacy gate, and includes unrelated core/build changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve poll update payloads:** `extensions/whatsapp/src/inbound/monitor.ts:472`
  - body: The messages.update handler receives each voter's poll update, but the synthetic message always uses an empty pollUpdateMessage. Hooks and agents only see a generic poll-vote placeholder and cannot learn which option data was selected.
  - confidence: 0.94
- **[P1] Handle RSVP updates from messages.update:** `extensions/whatsapp/src/inbound/monitor.ts:444-446`
  - body: This handler exits whenever pollUpdates is empty, so Baileys event response updates never become inbound messages. RSVP updates delivered through update.eventResponses will still be dropped instead of reaching hooks or agents.
  - confidence: 0.88
- **[P1] Respect the WhatsApp hook privacy gate:** `extensions/whatsapp/src/auto-reply/monitor/on-message.ts:154-156`
  - body: The gated-message path calls triggerInternalHook directly instead of the WhatsApp hook emitter that checks pluginHooks.messageReceived. Configurations that have not opted into WhatsApp message hooks can still receive unmentioned group content through this early return path.
  - confidence: 0.82
- **[P2] Recognize V5 poll creation messages:** `extensions/whatsapp/src/inbound/extract.ts:148-152`
  - body: The pinned Baileys API exposes pollCreationMessageV5, but this extraction only checks V1, V2, and V3. V5 poll creation messages can still return undefined and be dropped by enrichInboundMessage().
  - confidence: 0.9
- **[P2] Keep unknown RSVP states distinct:** `extensions/whatsapp/src/inbound/extract.ts:189-190`
  - body: EventResponseType.UNKNOWN is distinct from MAYBE, but this mapping treats every non-going/non-not-going value as maybe. Empty or unknown RSVP updates would be reported as tentative attendance.
  - confidence: 0.89
- **[P2] Keep negative poll durations on the poll path:** `src/poll-params.ts:57`
  - body: Changing numeric poll detection to value > 0 means invalid negative durations are no longer treated as poll intent and can bypass the poll-only validation path. Current main keeps all non-zero finite values on that path while still ignoring zero defaults.
  - confidence: 0.84
