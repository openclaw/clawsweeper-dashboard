## Summary

Current main still provides transcript-backed usage summaries, cache-backed `usage.cost`, provider quota snapshots, and UI aggregation, but it does not implement the requested durable per-call usage ledger with event IDs, source/run/job attribution, idempotency, or ledger-first readers. The linked CLI and aggregation work is adjacent, and maintainer discussion already treats it as non-closure for this request.

## What This Changes

Request native per-call and per-model usage logging for cost tracking across sessions, cron, heartbeat, and subagent work.

## Best Possible Solution

Add one canonical SQLite-backed usage ledger at the normalized completed model-call usage seam, then derive usage.cost, CLI/UI summaries, and backfill from that record while keeping transcripts as the conversation record and legacy fallback source.

## Reproduction Assessment

Not applicable as a failing runtime bug, but yes for the reported current behavior: source inspection confirms current main still has transcript-backed summaries and no native per-call usage ledger. The absence is visible in the usage types, gateway reader, docs, and tests.

## Solution Assessment

No to the literal `logs/usage.jsonl` shape as the best repository-native solution because root policy now makes runtime state/cache SQLite-first. The maintainable direction is one canonical ledger schema and reader path, with transcript scanning kept only as migration/backfill compatibility.
