## Summary

Keep open: current main and v2026.6.8 still provide transcript-backed usage summaries, a derived aggregate cache, diagnostics, and UI/Gateway aggregation, but not the requested durable per-call usage ledger with source/run/job attribution and ledger-first readers.

## What This Changes

Request native per-model/per-call usage logging for cost tracking, model-mix optimization, and cron/source attribution.

## Best Possible Solution

Define a canonical core completed-call usage ledger, likely SQLite-backed, written at the normalized model-call usage/cost seam with durable event ids, agent/session/run/source/job attribution, provider/model/token/cache/cost/status/latency metadata, no prompt/completion content, backfill, and ledger-first readers with transcript fallback for older data.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 expose transcript-backed summaries, derived caches, session log views, and diagnostics, but no durable per-call usage-ledger type, writer, backfill, or ledger-first reader.

## Solution Assessment

No current implementation fully solves it. The requested ledger direction is valid, but the maintainable OpenClaw shape should be a canonical core accounting contract, likely SQLite-backed under current storage policy, rather than another ad hoc runtime JSONL state file.
