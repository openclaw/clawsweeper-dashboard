## Summary

Keep this issue open: current main and v2026.6.8 still provide transcript-backed summaries, a durable transcript aggregate cache, per-session log views, diagnostics, and UI/Gateway aggregation, but not the requested durable per-call usage ledger with source/run/job attribution and ledger-first readers.

## What This Changes

The issue requests a native per-call usage ledger or dedicated usage log that records model/provider, token, cost, session/source, and background-job attribution for cost tracking.

## Best Possible Solution

Define a canonical core completed-call usage ledger, likely SQLite-backed, with durable event ids, agent/session/run/source/job attribution, provider/model/token/cache/cost/status/latency metadata, no prompt or completion content, a backfill path, and ledger-first readers with transcript fallback for older data.

## Reproduction Assessment

Yes, at source level. Current main and v2026.6.8 expose transcript-backed summaries, derived caches, per-session log views, and diagnostics, but no durable per-call usage-ledger type, writer, backfill, or ledger-first reader.

## Solution Assessment

No current implementation fully solves it. The requested ledger direction is reasonable, but the maintainable OpenClaw shape should be a canonical core accounting contract, likely SQLite-backed under current storage policy, rather than another ad hoc runtime JSONL state file.
