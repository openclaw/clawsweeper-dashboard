## Summary

Keep open: current main and v2026.6.8 still provide transcript-backed summaries, provider quota snapshots, diagnostic events, and UI/Gateway aggregation, but not the durable per-call usage ledger this request is asking for.

## What This Changes

The issue requests native per-model/per-call usage logging so users can track costs, model mix, and source/session attribution without parsing session transcripts manually.

## Best Possible Solution

Add a core usage ledger design with durable idempotent accounting events and ledger-first readers, using the repo’s canonical SQLite state model unless maintainers explicitly choose a named log/export artifact.

## Reproduction Assessment

Yes at source level: current main exposes transcript-backed summaries, derived caches, session log views, and diagnostics, but no durable per-call usage-ledger type, writer, backfill, or ledger-first reader. No live provider run is needed to prove that missing feature boundary.

## Solution Assessment

No, the current transcript-backed summary path is not the best final solution for this request because it remains a reader/aggregation layer over transcripts rather than an idempotent accounting source. The best solution needs a core ledger/storage design with privacy and reader migration decisions.
