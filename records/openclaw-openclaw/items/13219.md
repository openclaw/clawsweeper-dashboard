## Summary

Keep this issue open. Current main provides increasingly capable transcript-backed provider/model cost summaries, but it still lacks canonical per-call accounting with durable run, source, job, and idempotency attribution; the linked CLI and multi-agent aggregation PRs do not implement that capability.

## What This Changes

Add durable per-call and per-model usage accounting with session, run, and automated-job attribution for cost tracking.

## Best Possible Solution

Define one privacy-conscious SQLite accounting schema or generic observability contract at the normalized completed-model-call seam, with stable event identity and run/source attribution, then have CLI, gateway, and UI summaries consume it while treating transcripts only as historical backfill input.

## Reproduction Assessment

Yes at the source level. Current main demonstrably scans session transcripts for usage summaries and has no canonical per-call store carrying durable run, source, job, or event identity.

## Solution Assessment

No, the proposed dedicated JSONL runtime ledger is not the best current solution because repository policy requires state and accounting indexes to use SQLite; a canonical database schema or generic observability seam is safer and avoids parallel sources of truth.
