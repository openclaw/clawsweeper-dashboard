## Summary

Keep open. Current main and v2026.6.5 still provide transcript-backed usage/cost summaries, a transcript-derived cache, provider quota/status surfaces, and adjacent CLI/UI work, but not the requested durable per-call usage ledger with event IDs, source/run/job attribution, backfill, or ledger-first readers.

## What This Changes

The issue requests a native per-call/per-model usage ledger for cost tracking, source attribution, cron/background visibility, and model-mix reporting.

## Best Possible Solution

Add a core typed usage ledger at the normalized completed model-call seam, preferably in the shared state database rather than a new runtime JSONL state file, and make usage.cost, CLI/UI readers, and a backfill command prefer that ledger while preserving transcript scanning for older data.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source, docs, release-tag inspection, live issue discussion, and related PR review give a high-confidence verification path showing the requested durable per-call ledger is absent.

## Solution Assessment

Unclear for the exact public storage/API shape, but a native core ledger at the normalized model-call usage/cost seam is the maintainable direction. Transcript caches, provider quota snapshots, external real-time tools, and the closed transcript-backed CLI PR are useful adjacent surfaces but not replacements.
