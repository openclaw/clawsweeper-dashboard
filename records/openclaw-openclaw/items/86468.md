## Summary

Keep open: current `main` and the latest release still collapse DeepSeek `anyOf`/`oneOf` schemas by selecting the first non-null variant, so a const-literal union can lose every option except the first. The existing DeepSeek coverage locks the broad collapse behavior but does not cover enum-like const unions.

## What This Changes

The issue requests preserving all values in DeepSeek-normalized `anyOf` unions made from string `const` literals instead of exposing only the first literal.

## Best Possible Solution

Teach `normalizeDeepSeekSchema` to preserve non-null string `const` unions as a flat string `enum` while keeping nullable-union behavior, then add SDK and DeepSeek provider regression coverage plus a changelog entry.

## Reproduction Assessment

Yes, as a source-level reproduction: feed `normalizeDeepSeekToolSchemas` a property shaped like `anyOf: [{const:"overwrite", type:"string"}, {const:"append", type:"string"}]` and the current code selects only the first non-null variant. I did not execute a runtime test because this review was read-only.

## Solution Assessment

Yes, the narrowest maintainable fix is to special-case enum-like string const unions inside the existing DeepSeek tool-schema compatibility helper, not to add a new provider API or config option.
