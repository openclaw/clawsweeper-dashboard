## Summary

Keep open: the latest head fixes the JSON-only output and stricter direct utility failures, but it still turns update-triggered shell completion repair into a hard `openclaw update` failure. One compatibility repair is needed before merge.

## What This Changes

The PR makes `nodes list --json` emit JSON without the human summary, converts completion install failures to rejected errors, and makes the CLI startup build helper throw on spawn errors with focused tests.

## Best Possible Solution

Keep strict failures for direct completion install, but make update-triggered completion repair catch and report install failures as best-effort with a focused update regression test.

## Reproduction Assessment

Yes from source inspection: the PR head throws from `installCompletion`, and the update completion follow-up awaits it without a catch before restart follow-ups. I did not run a mutating update scenario because this review is read-only.

## Solution Assessment

No. The direct `completion --install` strictness is useful, but propagating those failures through optional update completion repair is not the narrowest maintainable fix.
