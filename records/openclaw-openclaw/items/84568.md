## Summary

Keep open: current main still has the request-time fallback from the stable channel registry to the active plugin registry on the outbound adapter path, and the new Discord beta report is a credible regression of a previously fixed message-delivery failure.

## What This Changes

The issue asks to stop outbound channel adapter resolution from falling back to the unstable active plugin registry after the pinned channel registry misses.

## Best Possible Solution

Keep outbound lookup stable against the channel-surface registry and make setup-only to full-runtime handoff explicit in the channel registry pin/repin lifecycle instead of consulting the arbitrary active registry at send time.

## Reproduction Assessment

Yes, at source level: current main has the exact active-registry fallback on the outbound adapter path, and the reporter provides a concrete Discord beta reproduction plus patched-package verification. I did not run live Discord in this read-only review.

## Solution Assessment

Yes on the direction, but the safest fix is not a blind deletion; preserve stable request-time lookup while moving the setup-only fallback into an explicit registry repin or lifecycle transition.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/84568. Current main resolves outbound adapters from `getActivePluginChannelRegistry()` and then falls back to `getActivePluginRegistry()` in `src/channels/plugins/registry-loader.ts`, which can lose Discord outbound after active registry swaps. Keep request-time outbound lookup stable against the channel-surface registry; if a pinned setup-only entry lacks send capability, promote or repin the full runtime channel registry during bootstrap/lifecycle instead of reading the arbitrary active registry at send time. Add regression coverage for a pinned runtime registry with outbound surviving an active replacement, a setup-only pinned entry being handed off to a send-capable runtime registry, and a Discord-shaped channel id not throwing `Outbound not configured`. Do not add Discord-specific core policy or new config. Add a changelog entry because this is a user-visible channel reply regression fix.
