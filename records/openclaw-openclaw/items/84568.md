## Summary

Keep open: current main still has the unstable active-registry fallback in the outbound channel registry loader, and the report matches a previously fixed failure mode that can drop Discord replies after registry replacement.

## What This Changes

The issue asks to remove or replace the outbound registry-loader fallback from the pinned channel registry to the unstable active plugin registry for Discord reply delivery.

## Best Possible Solution

Use a stable channel-surface registry for outbound sends and move any setup-only to full-runtime handoff into the channel registry pin/repin lifecycle rather than resolving outbound adapters from the arbitrary active registry at send time.

## Reproduction Assessment

Yes, at source level: current main has the exact active-registry fallback on the outbound adapter path, and the reporter provides a concrete Discord beta reproduction plus local patched-package verification. I did not run the live multi-bot Discord scenario in this read-only review.

## Solution Assessment

The requested direction is right, but a literal remove-only patch should also address the setup-only fallback tests by repinning the full runtime channel registry earlier. The maintainable fix is to keep request-time outbound lookup stable and make lifecycle transitions explicit.
