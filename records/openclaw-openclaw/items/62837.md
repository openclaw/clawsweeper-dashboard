## Summary

Keep open. Current main still does not prove the reported WhatsApp extractor-null/native tap-mention payload shape is handled; the merged adjacent fix only covers messages where `mentionedJids` are already populated.

## What This Changes

The issue asks OpenClaw to detect native WhatsApp group @-mentions when `extractMentionedJids()` returns no JIDs even though the inbound payload indicates a mention.

## Best Possible Solution

Collect a redacted current Baileys payload for this native group tap-mention, then update the WhatsApp extractor to read only current-message mention metadata and add regression tests that preserve quoted-message suppression.

## Reproduction Assessment

No, not at high confidence. Current source shows that an undefined extractor result reaches group gating and can skip mention-required messages, but this review still lacks the redacted Baileys `msg.message` payload for the reported native tap-mention shape.

## Solution Assessment

Unclear. A narrow `extractMentionedJids()` fix is likely right once the canonical payload field is known, but guessing a broad fallback could reintroduce quoted-message false positives fixed by https://github.com/openclaw/openclaw/pull/52711.
