This PR remains useful but not merge-ready: the latest head still drops the delivery failure text from persisted `deliveryError` when reclassifying delivery failure to `status=ok`, and the scheduler/backoff semantic change needs maintainer acceptance.

## What This Changes

The PR changes isolated cron finalization so nonfatal isolated-agent execution followed by delivery-dispatch failure returns `status=ok` with delivery diagnostics, plus a regression test.

## Best Possible Solution

Keep the narrow execution/delivery split, preserve delivery failure text in delivery-specific persisted state with coverage, and land only after maintainers accept the interim `ok`/`not-delivered` status model or choose the warning-status path.

## Solution Assessment
