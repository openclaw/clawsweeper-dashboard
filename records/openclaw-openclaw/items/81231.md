## Summary

Keep this PR open and do not merge yet: the stale-usage fix direction matches the linked bug, but source inspection confirms the clamp can suppress legitimate post-usage tail pressure that current tests require to trigger compaction. The external proof is also still only focused test/lint/build output, not real behavior proof from an actual session or gateway call.

## What This Changes

The PR bounds transcript-derived preflight token estimates in agent-runner memory and adds a gateway `models.list` test for `openai/gpt-5.5` and `openai/gpt-5.5-mini` allowlist output.

## Best Possible Solution

Revise the estimator to distinguish stale pre-compaction usage from legitimate latest usage plus trailing transcript pressure, preserve the conservative floor for active tail content, add a focused stale-after-compaction regression, and keep the response-level model-list assertion.

## Reproduction Assessment

Yes for the PR regression at source level: current main's combined-tail regression test constructs the legitimate high-usage-plus-tail case and expects compaction. I did not run a live Discord/Pi reproduction of the linked production bug.

## Solution Assessment

No. Bounding stale transcript usage is the right direction, but this implementation is too broad because it can discard legitimate latest-usage plus post-usage tail pressure.
