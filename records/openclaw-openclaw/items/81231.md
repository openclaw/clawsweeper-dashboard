## Summary

Keep open: this is a plausible narrow bug fix, but the latest estimator can still over-count post-usage tail after switching to the stale active-message estimate, and the external PR proof gate is still only test/CI output.

## What This Changes

The PR changes preflight transcript token estimation to bound stale usage against recent replay estimates, adds a stale-usage tail-pressure regression test, and adds a gateway models.list assertion for new OpenAI model IDs.

## Best Possible Solution

Refine the estimator so stale pre-compaction usage is capped by active replay/current prompt estimates without double-counting tail content, while preserving raw latest-usage-plus-tail pressure for non-stale usage records, then require real after-fix runtime proof before merge.

## Reproduction Assessment

No live reproduction was run, but the source-level path is clear: current main reads the raw last usage record while separately estimating active recent messages, and the PR's stale branch can double-count post-usage tail content. The linked issue and PR tests make the regression credible, but not a high-confidence real-environment reproduction.

## Solution Assessment

No. Bounding stale transcript usage is the right direction, but this implementation should distinguish stale usage from current latest usage rather than adding trailingBytesTokens to the already-active recent-message estimate after choosing the stale bound.
