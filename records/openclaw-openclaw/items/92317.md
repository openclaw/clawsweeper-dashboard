## Label Justifications

- `P2`: The defect materially limits long WebChat sessions but does not delete the underlying transcript or make the core runtime unusable.
- `impact:session-state`: Users cannot access older entries from the active session transcript once they fall outside WebChat's bounded recent-history window.

## Summary

Current main still makes older WebChat transcript entries unreachable after the fixed recent-history window, so this remains a valid canonical issue. An open pagination PR covers the same direction, but it currently lacks required real-behavior proof and has unresolved review concerns, so it is not a safe superseding target; maintainers should keep this issue open while deciding whether to repair or replace that implementation.

## What This Changes

Allow WebChat users to retrieve and review messages and tool output older than the currently loaded recent-history window.

## Best Possible Solution

Adopt one bounded backward-pagination path in the existing chat history contract, preserve the viewport anchor while deduplicating prepended entries, and prove both multi-page history and concurrent live-message behavior in a real Control UI session.

## Reproduction Assessment

Yes at source level: current main requests a fixed latest 100 entries, the gateway reads a bounded tail, and the protocol offers no selector for an earlier page. The related release report also supplies a concrete long-session browser path, although this review did not run a live browser reproduction.

## Solution Assessment

Yes in direction, but not yet in final implementation: bounded cursor-based backward loading is preferable to raising or configuring the fixed limit, while maintainers still need to settle cursor semantics and require proof that page prepends remain correct during live chat reconciliation.
