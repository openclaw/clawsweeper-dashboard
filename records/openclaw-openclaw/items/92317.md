## Summary

Current main confirms the reported limitation: WebChat requests a bounded recent history and applies a separate render budget, with no path to retrieve older pages. This is still necessary work, but it needs a product decision on pagination and resource budgets rather than a constant-only automated fix.

## What This Changes

Allow WebChat users to access older assistant replies and tool outputs beyond the current bounded active-chat history window.

## Best Possible Solution

Add an additive, bounded backward-pagination contract to gateway chat history, then have WebChat prepend older visible entries on demand while preserving scroll position, deduplicating message identities, and retaining explicit RPC and render budgets.

## Reproduction Assessment

Yes, at source level with high confidence: fixed request and render budgets plus the absence of an older-page API necessarily create a finite top of scrollback. A browser run would quantify the threshold but is not required to verify that the limitation exists.

## Solution Assessment

No concrete implementation is proposed, and merely raising the constants would not be the best solution. Bounded backward pagination across the gateway and Control UI is the narrowest durable direction, subject to maintainer choices on page size, character budgets, and tool-output treatment.
