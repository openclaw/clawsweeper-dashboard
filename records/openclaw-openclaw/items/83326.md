## Summary

Keep open: the PR targets the right cron duplicate-delivery seam, but it currently treats any resolved target as explicit and the supplied proof is unit-test-only, so it should not merge as-is.

## What This Changes

The PR passes `sourceReplyDeliveryMode: "message_tool_only"` from cron embedded agent execution when a resolved delivery target has `to`, and adds regression tests for explicit announce delivery and no-target delivery.

## Best Possible Solution

Carry delivery explicitness into the cron executor, suppress automatic source replies only for the intended explicit announce path, and cover both explicit `delivery.to` and implicit last-route cron cases before merge.

## Reproduction Assessment

Yes, at source level: the linked report gives a concrete Discord cron config, and current main reaches the embedded cron runner with delivery context but no `sourceReplyDeliveryMode` override while automatic group/channel replies can still be selected. I did not live-run the reporter's Discord guild.

## Solution Assessment

No, not as written: suppressing at this seam is the right direction, but keying off `resolvedDelivery.to` also catches implicit last/session routes. The safer fix should use the explicit delivery plan/resolution boundary or deliberately broaden the behavior with tests and docs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Gate source-reply suppression on explicit delivery:** `src/cron/isolated-agent/run-executor.ts:241`
  - body: This condition treats any resolved `to` as an explicit announce target, but implicit announce routes can also fill `to` from the last/session route while keeping delivery mode implicit. That can silently force `message_tool_only` for cron jobs without explicit `delivery.to`, overriding `messages.groupChat.visibleReplies: "automatic"`; pass the explicit delivery state into this seam and add an implicit-route regression.
