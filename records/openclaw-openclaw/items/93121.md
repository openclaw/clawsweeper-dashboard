## Summary

Keep this PR open, but it is not merge-ready: the proposed cutoff still treats unchanged daily-memory re-ingestion as fresh light input, so the linked duplicate-summary behavior can recur across dreaming days, and the supplied proof is tests/build/source-grep output rather than a real after-patch dreaming run.

## What This Changes

The branch adds a phase-signal reader and filters light-dreaming candidates when their prior `lastLightAt` is newer than their recall timestamp.

## Best Possible Solution

Keep the fix at the light selection and phase-signal boundary, but use a freshness marker that only advances for genuine new input and cover unchanged-later-day versus newly re-recalled entries before merge.

## Reproduction Assessment

Yes, source-reproducible: current main can re-ingest unchanged daily memory on a later dreaming day, and the recall writer refreshes `lastRecalledAt` even when daily dedupe suppresses count and score changes. I did not run a live overnight gateway reproduction in this read-only review.

## Solution Assessment

No. Filtering before `dedupeEntries` is the right layer, but `lastRecalledAt > lastLightAt` is not the best freshness contract because daily re-ingestion can advance `lastRecalledAt` without new memory content.
