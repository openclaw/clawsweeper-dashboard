## Summary

Keep open: the goal is still useful, but this PR is not mergeable because it starts channels before the runtime channel registry is pinned, and the author later confirmed the early channel phase is likely a no-op.

## What This Changes

This PR starts gateway channels at the beginning of `startGatewayPostAttachRuntime()` and changes the later sidecar phase to await that early channel promise.

## Best Possible Solution

Introduce a dependency-aware startup path that makes the channel runtime registry available before `startChannels()`, while keeping ACPX-dependent services on their own readiness path.

## Reproduction Assessment

Yes for source-level reproduction of the PR defect: the diff calls `startChannels()` before `loadStartupPlugins()` pins the channel registry, and channel startup enumerates that registry. I did not execute the PR branch.

## Solution Assessment

No: moving channels earlier is directionally plausible, but this implementation moves them before the channel runtime registry exists. A safer fix should pin or load channel runtime metadata first, or add an explicit dependency-ready startup seam.
