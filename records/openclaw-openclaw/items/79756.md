## Summary

Keep this PR open, but do not merge it as-is: the goal is valid, yet the branch can start channels before the plugin channel registry is pinned, then replace the later sidecar start with an await of that no-op promise. The supplied logs are real, but the after trace is not convincing because the source path can complete without starting a configured channel.

## What This Changes

The PR moves gateway channel startup earlier in `startGatewayPostAttachRuntime` and passes a wrapper into `startGatewaySidecars` so sidecars await the early channel-start promise.

## Best Possible Solution

Rework the gateway startup sequence so the channel plugin registry is loaded and pinned before any early channel start, preserve immediate channel-start error containment, and prove a configured channel starts before `plugins.runtime-post-bind` in a real run.

## Reproduction Assessment

Source-level yes: current main awaits runtime plugin loading before sidecar channel startup, and the linked report provides startup logs showing the channel delay. I did not run a live gateway/channel repro in this read-only review.

## Solution Assessment

No: moving `startChannels()` ahead of `plugins.runtime-post-bind` is the right direction, but this implementation runs before the registry pinning contract and weakens error containment. A safer fix needs an explicit registry-ready phase or split loader design plus live channel proof.
