## Summary

Keep open: the startup-latency problem is real, but this patch is not merge-ready because it starts channels before the standard Discord/Telegram channel registry is pinned, then suppresses the later normal startup path.

## What This Changes

The PR starts `params.startChannels()` at the beginning of gateway post-attach startup and passes an await-only wrapper into `startGatewaySidecars` so channel startup overlaps plugin runtime loading.

## Best Possible Solution

Rework the fix so the real channel registry is loaded and pinned before early channel startup, preserve the existing channel-start error containment or attach equivalent immediate handling, and keep the linked issue open until live channel proof shows the startup ordering works.

## Reproduction Assessment

Yes from source inspection: current main awaits `plugins.runtime-post-bind` before `startGatewaySidecars`, and `startGatewaySidecars` then calls `params.startChannels()` inside `sidecars.channel-start`. I did not run a live gateway in this read-only review.

## Solution Assessment

No: the proposed patch starts channels before the standard channel registry is ready and replaces the later normal start with an await-only wrapper. A safer solution should make registry readiness explicit before moving channel startup earlier.
