## Summary

Keep open: the linked issue still tracks a real gateway startup bug, but this PR is not safe as written because it starts channels before the loaded channel registry is pinned and can turn the later sidecar channel start into a no-op.

## What This Changes

The PR moves channel startup earlier in `startGatewayPostAttachRuntime` and makes the later sidecar phase await that early channel-start promise.

## Best Possible Solution

Replace the reorder with a dependency-aware startup seam that pins or loads the channel registry before channel startup, keeps ACPX-dependent readiness semantics explicit, and proves a configured channel connects before `plugins.runtime-post-bind` completes.

## Reproduction Assessment

Yes. Source inspection shows the PR calls `startChannels()` before the full channel registry is pinned, and `startChannels()` enumerates the active loaded channel registry.

## Solution Assessment

No. The goal is valid, but the patch moves channel startup before its registry dependency; the safer fix is a registry-ready or split channel-plugin loading seam.
