## Summary

Keep this PR open: the latency problem is real and the branch is pointed at the right gateway surface, but this patch can start channels before the plugin channel registry is populated and can move channel-start failures outside the existing contained sidecar error path.

## What This Changes

The PR moves gateway channel startup to the beginning of `startGatewayPostAttachRuntime` and passes an await-only wrapper into `startGatewaySidecars` so channel startup overlaps plugin runtime loading.

## Best Possible Solution

Keep the performance goal, but make channel startup wait for a populated channel registry, preserve immediate failure containment, add focused regression coverage for configured channels, and then provide redacted live channel proof before merge.

## Reproduction Assessment

Yes for the source-level ordering problem: current main awaits `plugins.runtime-post-bind` before `sidecars.channel-start`, and the linked report supplies matching startup logs. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No. Starting channels earlier is the right direction, but this implementation starts before the full channel registry is guaranteed populated and moves failure handling away from the existing contained sidecar path.
