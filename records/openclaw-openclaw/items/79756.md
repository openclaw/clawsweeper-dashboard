## Summary

Keep open with a blocking correctness finding: the PR’s early channel start can run before the channel registry is populated, so the later sidecar wrapper may skip the real channel start instead of decoupling it from ACPX.

## What This Changes

The PR starts gateway channels before runtime startup plugin loading and passes a wrapper to the sidecar phase that awaits the early channel promise.

## Best Possible Solution

Introduce a dependency-aware gateway startup path that makes the channel runtime registry ready before channel start while slow ACPX-dependent runtime work continues independently.

## Reproduction Assessment

Yes for source-level reproduction: the PR calls startChannels before runtime plugin loading pins the channel registry, and startChannels enumerates that registry. I did not run the live Windows ACPX cold-boot path.

## Solution Assessment

No. This one-file reorder is not the best fix because it can start channels against an empty registry and then suppress the normal later start; the safer path is an explicit channel-registry-ready or split runtime-plugin-loading seam.
