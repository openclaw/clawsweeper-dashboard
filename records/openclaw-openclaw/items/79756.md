## Summary

Keep open: the linked startup bug is still real on current main, but this PR is not a safe fix because it starts channels before the channel plugin registry is pinned and then suppresses the normal sidecar channel-start path.

## What This Changes

The PR starts `params.startChannels()` at the top of `startGatewayPostAttachRuntime()` before `plugins.runtime-post-bind` and passes a wrapper into `startGatewaySidecars()` so the later phase awaits the early promise.

## Best Possible Solution

Revise the gateway/plugin startup seam so channel registry state is loaded or pinned before any early channel start, preserve a normal-start fallback until configured channels start exactly once, and use the linked issue for the broader heartbeat and cron design.

## Reproduction Assessment

Yes. Source inspection shows the PR calls `startChannels()` before runtime plugin load pins the channel registry; `listChannelPlugins()` reads that registry, so configured channels can enumerate zero plugins, and the author's follow-up agrees the 2.9 ms trace likely proves that path.

## Solution Assessment

No. The goal is valid, but this one-file reorder is not the best fix; the safer path is a dependency-aware registry-ready or split-load design with fallback protection and focused startup coverage.
