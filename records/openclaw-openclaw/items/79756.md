## Summary

Keep open: the linked startup bug is real, but this PR is not merge-ready because its early channel start runs before the active channel registry is pinned and can suppress the normal later channel start.

## What This Changes

The branch starts channel startup at the beginning of `startGatewayPostAttachRuntime()` and passes a wrapper into `startGatewaySidecars()` so the later phase awaits that early promise.

## Best Possible Solution

Revise the gateway/plugin startup seam so channel registry state is pinned before channel start, ACPX-dependent runtime work is separated, and focused tests plus redacted live logs prove configured channels actually connect once.

## Reproduction Assessment

Yes. Source inspection shows current main still gates sidecar channel startup behind `plugins.runtime-post-bind`, and the PR's early call runs before the channel registry is pinned.

## Solution Assessment

No. The goal is valid, but this one-file reorder is the wrong layer because it can enumerate an empty channel registry and prevent the later normal startup path from running.
