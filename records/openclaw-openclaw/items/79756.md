## Summary

Keep open: the linked gateway startup bug is valid, but this PR is not ready to merge because it starts channels before the runtime channel registry is loaded and pinned, so configured channel providers can be skipped. The author’s latest follow-up independently acknowledges the same registry-order issue and asks maintainers to choose the startup design.

## What This Changes

The PR starts gateway channels at the top of `startGatewayPostAttachRuntime()` and makes the later sidecar channel phase await that early promise.

## Best Possible Solution

Implement a dependency-aware startup path that loads and pins channel plugins before channel startup, then allows ACPX-dependent runtime services to continue asynchronously with focused tests and live startup timing proof.

## Reproduction Assessment

Yes for the PR defect by source inspection: the added early `startChannels()` call runs before the runtime plugin load path pins the channel registry, and channel startup enumerates that registry. I did not execute the PR branch in this read-only review.

## Solution Assessment

No: starting channels earlier is directionally correct, but this implementation starts them before the channel registry exists. A safer fix should split channel runtime loading or add an explicit channel-registry-ready seam before provider startup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Load channel plugins before starting channels:** `src/gateway/server-startup-post-attach.ts:723-724`
  - body: This calls `params.startChannels()` before `loadStartupPlugins()` runs the runtime plugin load that pins the channel registry. On cold startup, bootstrap has installed an empty registry, so the early promise can enumerate zero channel plugins and the later sidecar wrapper skips the normal provider start.
