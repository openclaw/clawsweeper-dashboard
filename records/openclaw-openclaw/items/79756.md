## Summary

Keep open: the PR is directionally related to the linked gateway startup bug, but the patch starts channels before the runtime channel registry is loaded and pinned, so cold startup can skip configured channels instead of starting them earlier.

## What This Changes

This PR starts `params.startChannels()` at the top of `startGatewayPostAttachRuntime()` and makes later sidecar startup await that early channel-start promise.

## Best Possible Solution

Add a dependency-aware startup path, or a channel-only runtime load/pin seam, so channel plugins are available before channel start while ACPX-dependent services remain gated on ACPX readiness.

## Reproduction Assessment

Yes for the review finding: source inspection shows the early `startChannels()` call can run before `loadGatewayStartupPlugins()` pins the channel registry. I did not run the patched branch because this review is read-only.

## Solution Assessment

No: moving channel startup earlier is the right direction, but this implementation moves it before the runtime channel registry exists. A safer fix should split channel runtime availability from slow ACPX-dependent runtime work or introduce explicit startup dependencies.
