## Summary

Keep open: the PR targets the real sidecar startup latency problem in https://github.com/openclaw/openclaw/issues/79625, but the current diff starts channels before the runtime channel registry is populated, and the author’s follow-up agrees the supplied trace likely shows a no-op rather than connected channels.

## What This Changes

The branch starts channel startup at the beginning of `startGatewayPostAttachRuntime()` and makes the later sidecar channel phase await that early startup promise.

## Best Possible Solution

Introduce a dependency-aware startup path that makes the channel runtime registry available before `startChannels()` while keeping ACPX-dependent plugin services on their own readiness path.

## Reproduction Assessment

Yes, for the PR defect by source inspection: the PR head calls `startChannels()` before `loadStartupPlugins()` pins the channel registry, and channel startup enumerates plugins from that registry state. I did not execute the PR branch.

## Solution Assessment

No: moving channel startup earlier is directionally plausible, but this implementation starts before the channel registry exists. A safer solution should pin or load the channel runtime registry first, or add an explicit dependency-ready startup seam.
