## Summary

Keep this PR open, but the current patch is not merge-ready. The goal is valid, and the linked gateway startup bug remains open, but the merge result starts channels before the active channel registry is pinned, so the early startup can be a zero-channel no-op and the later sidecar path only awaits that same promise.

## What This Changes

The PR starts `params.startChannels()` early in `startGatewayPostAttachRuntime` and passes a wrapper to `startGatewaySidecars` that awaits the early promise instead of invoking channel startup there.

## Best Possible Solution

Land a revised gateway startup design that pins or loads the channel registry before starting channels, keeps ACPX/runtime-heavy plugin loading off the channel critical path, and proves a configured channel really connects early.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction of the ordering problem: normal bootstrap leaves runtime plugins unloaded, channel registry pinning happens in `plugins.runtime-post-bind`, and the PR starts channels before that point. I did not run a live gateway in this read-only review.

## Solution Assessment

No. Starting channels before ACPX/runtime-heavy plugin work is the right direction, but this patch starts them before the channel registry exists and changes failure timing instead of establishing a registry-ready boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Load the channel registry before early channel start:** `src/gateway/server-startup-post-attach.ts:1049-1054`
  - body: This starts `params.startChannels()` before `loadStartupPluginsIfNeeded()` runs `plugins.runtime-post-bind`, but channel startup enumerates `listChannelPlugins()`, which is backed by the active plugin channel registry pinned during `loadGatewayStartupPlugins`. In the normal startup path `plugins.bootstrap` used `loadRuntimePlugins: false`, so this early call can see an empty registry; the later sidecar wrapper then only awaits that no-op and never performs the real channel start.
  - confidence: 0.95
- **[P1] Attach channel-start failure handling immediately:** `src/gateway/server-startup-post-attach.ts:1049-1054`
  - body: The new promise can reject before `startGatewaySidecars` reaches its existing `sidecars.channels` try/catch, especially while plugin runtime loading is still blocked. That changes channel startup failures from a logged sidecar error into a possible unhandled startup rejection, so the early start needs immediate catch/log handling or must stay inside the sidecar lifecycle.
  - confidence: 0.86
