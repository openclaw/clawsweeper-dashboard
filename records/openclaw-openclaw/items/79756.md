- `P1`: The PR targets a real startup problem affecting live channel availability and can also break channel startup during gateway boot.

## Summary

Keep open: the linked startup bug remains valid, but this branch is not merge-ready because it starts channels before the active channel registry is pinned, so the early call can resolve without starting configured channels while the later sidecar phase only awaits that no-op promise.

## What This Changes

The PR moves gateway channel startup earlier in `startGatewayPostAttachRuntime` and makes the later sidecar phase await that early startup promise.

## Best Possible Solution

Replace the reorder with a dependency-aware startup seam that pins or loads the channel registry before channel startup, preserves ACPX-dependent readiness semantics, and proves a configured channel connects before `plugins.runtime-post-bind` completes.

## Reproduction Assessment

Yes. Source inspection shows the PR calls `startChannels()` before the full channel registry is pinned, and `startChannels()` enumerates the active loaded channel registry; I did not run a live gateway repro in this read-only review.

## Solution Assessment

No. The goal is valid, but this is not the best fix because it moves channel startup before its registry dependency; a registry-ready or split channel-plugin loading seam is safer.
