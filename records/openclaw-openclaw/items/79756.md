## Summary

Keep open: the linked bug is plausible, but this patch starts channels before the runtime channel registry is loaded and pinned, so a cold gateway can enumerate zero channel plugins and then skip the normal channel start path.

## What This Changes

The PR moves gateway channel startup into an early promise before `plugins.runtime-post-bind` and makes later sidecar startup await that promise.

## Best Possible Solution

Implement a dependency-aware startup path that makes channel runtime registry availability precede `startChannels()` while slow ACPX-dependent work continues asynchronously, with regression coverage and redacted live gateway proof.

## Reproduction Assessment

Yes for the PR defect by source inspection: the patch calls `startChannels()` before `loadStartupPlugins()` pins the channel registry, and channel startup enumerates the loaded registry. I did not run the patched branch live.

## Solution Assessment

No. Moving channels earlier is the right direction, but this implementation moves them before the registry they need exists; a safer fix should split or signal channel-runtime readiness before starting channels.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Load channel plugins before starting channels:** `src/gateway/server-startup-post-attach.ts:723-724`
  - body: This starts `params.startChannels()` before `loadStartupPlugins()` loads and pins the runtime channel registry. On a cold gateway, bootstrap has installed an empty registry, so `listChannelPlugins()` can return no channel plugins; the later wrapper then awaits that no-op promise and skips the normal channel start, leaving configured channels disconnected.
