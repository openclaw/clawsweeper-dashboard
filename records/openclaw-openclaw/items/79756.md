## Summary

Keep open: the patch is not mergeable because it can start channels before the runtime channel registry is loaded and pinned, so configured channel plugins can be skipped; the supplied proof does not show an actual channel provider starting or connecting after the change.

## What This Changes

The branch moves gateway channel startup to the beginning of `startGatewayPostAttachRuntime()` and makes the later sidecar channel phase await that early promise.

## Best Possible Solution

Implement a dependency-aware startup path or channel-only runtime load/pin seam so configured channel plugins exist before channel start while ACPX-dependent services continue asynchronously.

## Reproduction Assessment

Yes for the PR defect by source inspection: the head calls `startChannels()` before `loadStartupPlugins()` pins the runtime channel registry, and current `startChannels()` enumerates loaded channel plugins from active/pinned registry state. I did not run the patched branch in this read-only review.

## Solution Assessment

No: moving channel startup earlier is the right direction, but this implementation moves it before the channel runtime registry exists. A safer fix should split channel runtime availability from slow ACPX-dependent runtime work or add explicit startup dependencies.
