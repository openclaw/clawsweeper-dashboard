## Summary

Keep open, but this PR is not merge-ready: the linked startup bug is still valid, yet this patch starts channels before the runtime channel registry is pinned, so the early promise can enumerate zero channels and then suppress the later real channel start.

## What This Changes

The PR starts `params.startChannels()` at the top of `startGatewayPostAttachRuntime()` and passes a wrapper into sidecar startup so the later channel phase awaits that early promise.

## Best Possible Solution

Revise the gateway/plugin startup seam so channel plugin registry state is pinned before channel startup and slow ACPX-dependent runtime work runs on its own readiness path, with tests and real logs proving configured channels actually connect.

## Reproduction Assessment

Yes for source-level reproduction: current main still awaits `plugins.runtime-post-bind` before sidecars, and `startChannels()` enumerates `listChannelPlugins()` from the runtime channel registry. I did not reproduce the Windows cold-boot timing path live.

## Solution Assessment

No. The requested direction is valid, but this patch runs `startChannels()` before the active channel registry is pinned; a safer fix needs a registry-ready startup seam and separate proof for heartbeat/cron expectations.
