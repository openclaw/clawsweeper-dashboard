## Summary

Keep open, but this PR is not merge-ready: the linked startup bug is real, yet this branch starts channels before channel registry pinning and can skip configured channels while still marking sidecars ready.

## What This Changes

The PR starts `params.startChannels()` as an early promise in `startGatewayPostAttachRuntime()`, then has later sidecar startup await that promise instead of calling the normal channel start path.

## Best Possible Solution

Replace this branch with dependency-aware startup that populates channel registry state before starting channels and proves real Discord or Telegram availability before slow ACPX/plugin runtime work completes.

## Reproduction Assessment

Yes. Source inspection shows current main still gates channel startup behind `plugins.runtime-post-bind`, and the PR head starts channels before registry pinning, which can enumerate zero loaded channel plugins.

## Solution Assessment

No. The goal is valid, but this reorder is not the best fix because it runs before registry readiness; a registry-ready or split-load startup seam is the safer design.
