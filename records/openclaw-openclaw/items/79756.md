## Summary

Keep open for maintainer review: the startup problem is still real, but this branch is not safe as-is because it starts channels before the channel plugin registry is populated and then replaces the later normal channel-start call with an await of that early promise.

## What This Changes

The PR changes `src/gateway/server-startup-post-attach.ts` to start channels through an early promise before `plugins.runtime-post-bind` and have later sidecar startup await that promise.

## Best Possible Solution

Replace this reorder with a dependency-aware startup design that pins channel registry state before channel startup, keeps ACPX-dependent work on its own readiness path, and proves a real configured channel connects before slow runtime plugin work completes.

## Reproduction Assessment

Yes. Source inspection shows current main still awaits `plugins.runtime-post-bind` before channel sidecars, and the PR head calls `startChannels()` before the channel registry is pinned, which can enumerate zero loaded channel plugins.

## Solution Assessment

No. The goal is valid, but this one-file reorder is not the best fix because it runs before registry readiness and suppresses the later safe startup path; the better solution is a registry-ready or split-load startup seam.
