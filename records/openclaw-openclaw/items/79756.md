## Summary

Keep open: this draft PR targets a valid gateway startup bug, but the current patch can skip channel startup entirely because it starts channels before the runtime channel plugin registry is loaded and pinned; after-fix real behavior proof is also still missing.

## What This Changes

The PR starts `params.startChannels()` at the beginning of `startGatewayPostAttachRuntime()` and passes a wrapper into `startGatewaySidecars()` so later sidecar readiness waits for that early promise.

## Best Possible Solution

Split channel runtime availability from slow ACPX-dependent runtime work so channel plugins are loaded and pinned before channel start, then prove the dependency-driven startup path with a live ACPX-enabled gateway trace.

## Reproduction Assessment

Yes for the review finding: source inspection shows the early call reaches `startChannels()` before `loadGatewayStartupPlugins()` pins the channel registry, so a cold gateway can enumerate no channel plugins. I did not run the patched branch because this review is read-only.

## Solution Assessment

No: moving channel startup earlier is directionally aligned, but this implementation moves it before the runtime channel registry exists. A safer fix needs a channel-only runtime load/pin seam or a dependency-driven sidecar startup path that keeps ACPX-specific waits scoped to ACPX consumers.
