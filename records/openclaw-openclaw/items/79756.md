## Summary

Keep open: the linked startup-latency problem is still real, but this PR is not merge-ready because it can start channels before the channel registry is pinned and then skip the later real channel-start path.

## What This Changes

The PR moves gateway channel startup ahead of plugin runtime post-bind and makes the later sidecar startup await that early channel promise.

## Best Possible Solution

Rework or replace this PR with a dependency-aware startup design that pins or loads the channel registry before `startChannels()`, keeps ACPX-dependent work on its own readiness path, and proves a real configured channel starts while ACPX/post-bind work continues.

## Reproduction Assessment

Yes, at source level: the PR calls `startChannels()` before runtime plugin loading pins the channel registry, while channel startup enumerates loaded channel plugins from that registry. I did not run a live gateway in this read-only pass.

## Solution Assessment

No. The direction is valid, but this implementation is not the best fix because it moves the call earlier without first establishing a channel-registry-ready seam.
