## Summary

Keep open: the goal matches a real gateway startup bug, but this implementation starts channels before the runtime channel registry is pinned, and the author has acknowledged that the early channel start is likely a no-op.

## What This Changes

The PR changes gateway post-attach startup so channel startup begins before runtime plugin loading and later sidecar startup awaits that early channel promise.

## Best Possible Solution

Implement a dependency-aware startup path that makes the channel registry available before channel start while keeping slow ACPX-dependent runtime work out of the channel critical path.

## Reproduction Assessment

Yes for the PR defect by source inspection: the proposed early call runs before the runtime plugin load that pins the channel registry, so configured channels can enumerate as empty. I did not execute the branch in a live gateway.

## Solution Assessment

No. Moving channels earlier addresses the right symptom, but this PR moves them before their plugin registry dependency exists; the safer solution is a channel-registry-ready or channel-only runtime-loading seam.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Load channel plugins before starting channels:** `src/gateway/server-startup-post-attach.ts:721-722`

On a cold gateway, bootstrap currently installs an empty registry, so `startChannels()` can enumerate no channel plugins; the later wrapper then skips the normal sidecar channel start and leaves configured channels disconnected.
