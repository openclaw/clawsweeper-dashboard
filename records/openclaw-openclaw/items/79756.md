## Summary

Keep open: the linked startup-latency bug is real, but this branch is not safe because it starts channels before the loaded channel registry exists and then suppresses the later normal channel start; the contributor's follow-up comment reaches the same root-cause conclusion.

## What This Changes

This PR moves channel startup earlier in `startGatewayPostAttachRuntime` and makes the later sidecar phase await that early channel-start promise.

## Best Possible Solution

Make a channel-capable registry ready before any early channel start, or keep the existing sidecar start as the fallback until a regression test and fresh live proof show real channels start during plugin runtime loading.

## Reproduction Assessment

Yes for the PR defect from source inspection: `startChannels()` enumerates `listChannelPlugins()` before `loadGatewayStartupPlugins()` pins the loaded channel registry. I did not run a live gateway in this read-only review.

## Solution Assessment

No. The maintainable fix is to make the channel registry ready before early startup, or preserve the existing sidecar start as a fallback until that ordering is proven.
