## Summary

Keep open: the linked gateway/channel startup problem still matters, but this branch should not merge because it starts channels before the channel registry is pinned, and the contributor's follow-up correctly notes the supplied trace likely shows a zero-channel no-op.

## What This Changes

This PR starts `params.startChannels()` early in `startGatewayPostAttachRuntime` and makes the later sidecar phase await that early channel-start promise.

## Best Possible Solution

Land a design-approved gateway startup change that pins a real channel registry before starting channels, keeps ACPX/runtime-post-bind work concurrent where safe, and proves live channel availability after the corrected ordering.

## Reproduction Assessment

Yes for the PR defect from source inspection: the PR calls `startChannels()` before runtime plugins pin the channel registry, while current channel listing reads that registry. I did not run a live gateway reproduction in this read-only review.

## Solution Assessment

No, this PR is not the best fix as-is. The safer fix is to load or pin the channel registry before channel startup, then decouple only the slow ACPX/runtime work that channels do not need.
