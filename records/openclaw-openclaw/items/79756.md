## Label Justifications

- `P1`: The PR targets a startup regression affecting real channel availability, but the current patch can make channel startup fail or silently no-op.
- `merge-risk: 🚨 message-delivery`: The changed startup order can prevent configured chat channels from starting, which can drop or delay inbound/outbound channel messages.

## Summary

Keep open, but the current patch is not mergeable: it starts channels before the channel registry is populated, then prevents the normal later post-load channel start. The contributor’s later comment correctly identifies that the supplied 2.9ms trace likely shows a zero-channel no-op rather than real early channel startup.

## What This Changes

This PR moves channel startup earlier in `startGatewayPostAttachRuntime` and makes `startGatewaySidecars` await that early channel-start promise instead of calling `params.startChannels` directly.

## Best Possible Solution

Keep the decoupling goal, but establish a registry-ready channel startup phase that preserves channel error handling and proves real Discord/Telegram/heartbeat/cron startup while slow plugin runtime work continues.

## Reproduction Assessment

Yes from source inspection: the current bootstrap uses `loadRuntimePlugins: false`, the channel registry is pinned only in `loadGatewayStartupPlugins`, and `startChannels` enumerates `listChannelPlugins()`. I did not run a full gateway repro in this read-only review.

## Solution Assessment

No. The requested decoupling is directionally useful, but this implementation starts channels before the registry is ready and weakens the existing startup error-handling path.
