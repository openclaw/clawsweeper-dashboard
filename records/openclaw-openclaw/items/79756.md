## Summary

Keep open: the proposed startup order is still incorrect because it calls channel startup before the runtime channel registry is populated, and the contributor later acknowledged that the early trace likely started zero channels.

## What This Changes

The PR starts gateway channels early before runtime plugin loading and makes the later sidecar phase await that early promise.

## Best Possible Solution

Introduce a dependency-aware gateway startup path that makes channel runtime registry availability happen before channel startup while keeping ACPX-dependent runtime work on its own readiness path.

## Reproduction Assessment

Yes for the PR defect by source inspection: the patch calls `startChannels()` before runtime plugin loading pins the channel registry, and channel startup enumerates that registry. I did not execute the PR branch.

## Solution Assessment

No: moving channels earlier is directionally plausible, but this implementation moves them before the channel runtime registry exists. A safer solution should pin or load channel runtime metadata first, or add an explicit registry-ready startup seam.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Load the channel registry before starting channels:** `src/gateway/server-startup-post-attach.ts:720-722`

In the deferred startup path bootstrap can install an empty registry, so this promise can resolve after starting zero channels and the later wrapper skips the normal sidecar channel start.
