## Summary

Keep open: the branch targets a real gateway startup coupling, but this diff starts channels before the runtime channel registry is populated, so configured channels can be skipped; the author later acknowledged that ordering issue in the PR discussion.

## What This Changes

The PR starts `params.startChannels()` at the beginning of `startGatewayPostAttachRuntime()` and makes the later sidecar channel phase await that early promise.

## Best Possible Solution

Implement a dependency-aware gateway startup path that makes the channel runtime registry available before starting channels, then lets slow ACPX-dependent plugin services continue on their own readiness path.

## Reproduction Assessment

Yes: source inspection shows the PR calls `startChannels()` before `loadStartupPlugins()` pins the runtime channel registry, while channel startup enumerates that registry. I did not run the PR branch live.

## Solution Assessment

No: the one-file reorder is not the best fix because it can run before channel plugin runtime metadata exists. A safer solution should pin or load the channel registry first, or add an explicit startup dependency seam.
