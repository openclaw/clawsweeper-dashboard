## Summary

Keep this PR open, but it is not merge-ready: the goal is valid, yet the current diff starts channels before the channel plugin registry is pinned and then prevents the later real channel start.

## What This Changes

This PR moves gateway channel startup to the beginning of `startGatewayPostAttachRuntime` and makes `startGatewaySidecars` await that early channel promise instead of starting channels directly.

## Best Possible Solution

Decouple channel startup by loading and pinning the channel-capable plugin registry first, then starting channels with immediate failure handling and real transport proof.

## Reproduction Assessment

Source-reproducible, not locally run: current main waits for runtime plugin loading before channel startup, and the PR moves `startChannels()` ahead of the registry pin that `listChannelPlugins()` depends on.

## Solution Assessment

No. Starting channels before the channel registry is populated is not the right repair; the safer fix is a registry-first startup phase or an explicit registry-ready seam with failure handling preserved.
