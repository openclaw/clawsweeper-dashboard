## Summary

Keep open: the goal matches the startup-latency problem tracked at https://github.com/openclaw/openclaw/issues/79625, but this patch starts channels before runtime plugin loading pins/populates the channel registry, so the early start can be a no-op. The author later acknowledged the registry-order blocker and asked maintainers for design direction.

## What This Changes

The branch starts `params.startChannels()` at the top of `startGatewayPostAttachRuntime()` and makes the later sidecar channel phase await that early promise.

## Best Possible Solution

Introduce a dependency-aware startup path that makes channel runtime registry availability happen before `startChannels()`, while keeping ACPX-dependent services on their own readiness path.

## Reproduction Assessment

Yes for the PR defect by source inspection: the patch calls `startChannels()` before runtime plugin loading pins the channel registry, and `startChannels()` enumerates loaded channel plugins from that registry state. I did not execute the PR branch.

## Solution Assessment

No: moving channels earlier is directionally plausible, but this implementation moves them before the channel runtime registry exists. A safer solution should load or pin channel runtime metadata first, or add an explicit dependency-ready startup seam.
