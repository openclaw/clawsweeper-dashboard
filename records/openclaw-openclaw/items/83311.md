## Summary

Keep open: the PR targets a real source-supported config data-loss bug, but the widened preservation rule now copies dangerous record-valued object keys into a plain object and the supplied proof does not exercise the patched migration path.

## What This Changes

The PR changes legacy web search config migration to preserve any non-legacy, non-apiKey key under `tools.web.search` and adds regression tests for custom-provider and `openaiCodex` records.

## Best Possible Solution

Preserve unrelated `tools.web.search` records while dropping migrated legacy keys and blocked prototype keys, with regression coverage for custom records, `openaiCodex`, and prototype-key safety.

## Reproduction Assessment

Yes, source-level. A config with `tools.web.search.apiKey` or a legacy provider record plus an unrelated record-valued key reaches the current-main copy loop, where that unrelated record is not copied into `nextSearch`.

## Solution Assessment

No, not yet. Preserving non-legacy records is the right direction, but the implementation should first skip blocked object keys and be proven through the actual patched migration or doctor path.
