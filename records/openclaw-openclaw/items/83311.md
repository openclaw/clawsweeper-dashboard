## Summary

Keep open: the bug and main fix direction are valid, but the patch widens preservation without carrying over the prototype-key guard used by the sibling migration, so it is not safe to merge as written.

## What This Changes

The PR changes the legacy web search doctor migration to preserve non-legacy record-valued `tools.web.search` keys and adds regression coverage for a custom provider record plus `openaiCodex`.

## Best Possible Solution

Add a dangerous-key guard before `nextSearch[key] = value`, cover the guard plus custom-record and `openaiCodex` preservation in regression tests, then land only one corrected closing PR for the linked issue.

## Reproduction Assessment

Yes. Source inspection of current `origin/main` shows an unrelated record-valued `tools.web.search` key is not copied into `nextSearch` when the legacy migration runs, and the PR’s added tests target that path.

## Solution Assessment

No, not as written. Preserving non-legacy records is the right narrow fix, but it should also filter dangerous prototype keys like the sibling web-fetch migration before merge.
