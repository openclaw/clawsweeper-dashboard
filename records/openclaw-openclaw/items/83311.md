## What This Changes

The PR broadens legacy web search migration to preserve non-legacy record-valued `tools.web.search` keys and adds regression tests for custom-provider and `openaiCodex` records.

## Best Possible Solution

Preserve unknown web-search records while filtering blocked object keys, then land one focused linked PR with regression coverage and real output from the patched migration or doctor path.

## Reproduction Assessment

Yes, source-level. A config with `tools.web.search.apiKey` or a known legacy provider record plus an unrelated record-valued key reaches the current-main copy loop, where that unrelated record is not copied into `nextSearch`.

## Solution Assessment

No, not yet. Preserving non-legacy records is the right direction, but this implementation should first mirror the blocked-key guard used by the sibling web-fetch migration and be proven through the actual patched migration or doctor path.
