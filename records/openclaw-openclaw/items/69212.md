## Summary

Keep this PR open, but it is not merge-ready: the current patch adds a useful provider-continuation direction while introducing an ungated Responses continuation path, an encrypted metadata buffer-accounting risk, and it still lacks real runtime proof from the changed provider path.

## What This Changes

The PR adds OpenAI-compatible completions encrypted_content capture/replay, HTTP Responses previous_response_id tool continuations, and payload-shape tests.

## Best Possible Solution

Revise the PR to gate stateful Responses continuations to supported native routes, count serialized replay metadata in the buffer guard, and provide redacted runtime proof for the provider continuation path.

## Reproduction Assessment

Yes. Source inspection gives high-confidence paths: a proxy openai-responses model with trailing toolResult messages exercises the ungated previous_response_id truncation, and an encrypted-only reasoning chunk during a tool call exercises the unbudgeted replay metadata path.

## Solution Assessment

No. The direction is plausible, but the maintainable solution needs to reuse the native-route capability boundary and budget serialized encrypted replay metadata before enqueueing or merging buffered deltas.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Budget encrypted replay metadata before buffering:** `src/agents/openai-transport-stream.ts:1348-1353`
  - body: This merge preserves replayMetadata, but queuePostToolCallDelta still adds only Buffer.byteLength(next.text) to pendingPostToolCallBytes. Encrypted-only chunks can carry large encrypted_content with empty text, so a tool-call stream can bypass MAX_POST_TOOL_CALL_BUFFER_BYTES until flush; include serialized replay metadata bytes in the guard before enqueueing or merging.
  - confidence: 0.86
- **[P2] Gate previous_response_id to native Responses routes:** `src/agents/openai-transport-stream.ts:1024-1049`
  - body: resolveOpenAIResponsesContinuationPlan now runs for any same-provider/model trailing tool result, and the payload then sends previous_response_id with only the trailing function_call_output items. Proxy or custom openai-responses backends that do not support stateful OpenAI continuations lose the full context current main sends, so gate this path with the existing native route capability policy.
  - confidence: 0.88
