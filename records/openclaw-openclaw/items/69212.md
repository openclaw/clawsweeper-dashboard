## Summary

Keep open: the Volcengine replay direction is plausible, but this PR still needs real behavior proof and has source-backed blockers around unbudgeted encrypted replay metadata and ungated stateful Responses continuations for proxy/custom backends.

## What This Changes

This PR adds OpenAI-compatible completions `encrypted_content` capture/replay, Responses `previous_response_id` tool-only continuation payloads, and payload-shape tests.

## Best Possible Solution

Gate stateful Responses continuations to native route capability, count serialized replay metadata in the post-tool buffer, rebase on current main, and require redacted runtime proof for the Volcengine path.

## Reproduction Assessment

Yes by source inspection. PR head shows a proxy/custom `openai-responses` model with trailing tool results would get `previous_response_id` plus truncated input, and encrypted-only reasoning chunks can carry replay metadata while the byte guard counts only visible text.

## Solution Assessment

No. The feature direction is plausible, but the current implementation should reuse the native-route capability boundary and include serialized replay metadata in the post-tool buffer budget before merge.

## Review Findings
