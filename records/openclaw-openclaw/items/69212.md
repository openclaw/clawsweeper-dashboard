## Summary

Keep this PR open. Current main still does not implement the PR's central behaviors, but the proposed patch has blocking correctness/security concerns around ungated HTTP continuation IDs and unbudgeted encrypted replay metadata.

## What This Changes

The PR adds HTTP Responses previous_response_id tool-only continuations, OpenAI-compatible completions encrypted_content capture/replay, and payload-shape tests for those paths.

## Best Possible Solution

Land a transport change that supports native HTTP Responses/Codex Responses continuations and Volcengine encrypted reasoning replay while preserving full-context behavior for unsupported proxy routes and enforcing the existing memory caps on encrypted metadata.

## Reproduction Assessment

Yes. A context ending with an assistant responseId followed only by toolResult messages exercises the HTTP continuation builder, and a completions stream with reasoning_content plus encrypted_content exercises the replay capture path; current-main inspection shows both missing.

## Solution Assessment

No. The direction is useful, but this implementation needs route-capability gating for previous_response_id and byte accounting for encrypted replay metadata before it is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Gate continuation IDs to native Responses routes:** `src/agents/openai-transport-stream.ts:1034`
  - body: The new continuation plan runs for any Responses model with trailing tool results. Custom or proxy Responses backends that do not support `previous_response_id` would now receive only `function_call_output` items plus an unsupported parameter, where the previous full-context request could succeed. Gate this path to native OpenAI/Codex Responses-capable routes.
  - confidence: 0.87
- **[P1] Budget encrypted replay metadata in the post-tool buffer:** `src/agents/openai-transport-stream.ts:1398-1403`
  - body: This merge preserves `replayMetadata`, but `queuePostToolCallDelta` still only counts `next.text`. Encrypted-only chunks can have empty text and large `encrypted_content`, so a tool-call stream can bypass the 256KB cap until flush. Include serialized replay metadata bytes before enqueueing or merging.
  - confidence: 0.84
