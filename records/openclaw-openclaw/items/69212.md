## Summary

Kept open: current main does not include the branch's central OpenAI-compatible encrypted replay work, and the PR still has blocking correctness/security defects plus missing external real behavior proof.

## What This Changes

The branch adds OpenAI Completions `encrypted_content` capture/replay for Volcengine/Doubao reasoning, Responses `previous_response_id` tool-only continuations, and payload-shape tests.

## Best Possible Solution

Revise the PR to preserve encrypted replay support while gating stateful Responses continuations to native routes, accounting serialized replay metadata in buffer limits, rebasing over current transport changes, and adding redacted runtime proof.

## Reproduction Assessment

Yes. Source inspection of PR head gives high-confidence paths: a proxy `openai-responses` model with trailing tool results exercises the ungated `previous_response_id` truncation, and an encrypted-only reasoning chunk during a tool call exercises unbudgeted replay metadata.

## Solution Assessment

No. The direction is plausible, but the maintainable solution should reuse the existing native-route capability boundary and count serialized replay metadata before enqueueing or merging buffered deltas.
