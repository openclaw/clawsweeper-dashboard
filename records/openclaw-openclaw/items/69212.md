Keep this PR open: the direction is useful and current main still lacks the requested completions replay and Responses continuation behavior, but the PR head has a route-capability regression, a concrete encrypted-metadata buffer accounting risk, and no after-fix real behavior proof.

## What This Changes

The PR adds OpenAI-compatible completions encrypted_content capture/replay, HTTP Responses previous_response_id tool continuations, and payload-shape tests.

## Best Possible Solution

Revise the PR to keep full-context payloads for unsupported Responses routes, budget encrypted replay metadata in the existing buffer guard, and add redacted runtime proof for the changed provider continuation path.

## Reproduction Assessment

Yes. Source inspection gives high-confidence reproduction paths: a proxy openai-responses model with trailing toolResult messages exercises the ungated previous_response_id truncation, and an encrypted-only reasoning chunk during a tool call exercises the unbudgeted replay metadata path.

## Solution Assessment

No. The PR is a plausible direction, but the maintainable fix needs to reuse the native-route capability boundary and count serialized replay metadata bytes before enqueueing or merging buffered deltas.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Budget encrypted replay metadata before buffering:** `src/agents/openai-transport-stream.ts:1348-1353`
  - body: This merge preserves replayMetadata, but queuePostToolCallDelta still only adds Buffer.byteLength(next.text) to pendingPostToolCallBytes. Encrypted-only chunks can carry large encrypted_content with empty text, so a tool-call stream can bypass MAX_POST_TOOL_CALL_BUFFER_BYTES until flush; include serialized replay metadata bytes in the guard before enqueueing or merging.
  - confidence: 0.86
- **[P2] Gate previous_response_id to native Responses routes:** `src/agents/openai-transport-stream.ts:1024-1049`
  - body: resolveOpenAIResponsesContinuationPlan now runs for any same-provider/model trailing tool result, and the payload then sends previous_response_id with only the trailing function_call_output items. Proxy or custom openai-responses backends that do not support stateful OpenAI continuations lose the full context current main sends, so gate this path with the existing native route capability policy.
  - confidence: 0.88
