## Summary

Keep open: this is a plausible focused bug fix, but the current PR still has blocking correctness issues around tool-boundary output selection and hosted client-tool lifecycle handling. The supplied Telegram proof is useful context, but it should be refreshed after the code blockers are fixed.

## What This Changes

The PR changes agent/subagent lifecycle handling and subagent announcement output selection to treat unresolved tool-boundary endings as errors and avoid stale or empty completion announcements, with regression tests.

## Best Possible Solution

Keep the lifecycle hardening only for genuinely abandoned internal tool turns, preserve hosted client-tool continuations on their normal tool_calls path, make waitingForContinuation block terminal output selection, and refresh redacted current-head Telegram or log proof.

## Reproduction Assessment

Yes, source-level: a terminal assistant message with both text and a tool call leaves waitingForContinuation true while PR head still returns latestAssistantText, and hosted client tools are already represented by stopReason=tool_calls with pendingToolCalls. I did not run tests or live Telegram in this read-only review.

## Solution Assessment

No; the direction is narrow and useful, but the current implementation still returns preliminary tool-turn text and over-classifies legitimate hosted client-tool continuations as lifecycle errors. The safer fix is to distinguish external client-tool handoff from abandoned internal tool turns and treat waitingForContinuation as a hard stop until a later non-tool assistant answer exists.
