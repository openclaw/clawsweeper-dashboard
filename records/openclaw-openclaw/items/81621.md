## Summary

Keep open: this targets a real subagent completion bug family, but the central lifecycle guard is placed after the embedded lifecycle end path can already mark the run complete, so the false-success case can still survive. The supplied real-behavior proof is also from a hotpatched 2026.4.12 install, not latest main.

## What This Changes

The PR changes agent/subagent completion handling so unresolved tool-boundary endings become lifecycle errors and stale tool-boundary output is not selected as final completion text.

## Best Possible Solution

Classify unresolved tool-boundary terminal metadata at the canonical lifecycle event source, keep the output-selection hardening, then verify with targeted agent tests and redacted latest-main Telegram proof.

## Reproduction Assessment

Yes, source-level: current main can emit an embedded lifecycle end with stopReason tool_calls before agent-command's result backstop runs. I did not execute a live current-main repro in this read-only review.

## Solution Assessment

No; the direction is narrow, but the guard is placed after lifecycleEnded can already suppress it. The safer fix is to classify unresolved tool-boundary terminal events before marking lifecycle completion, or inside the embedded lifecycle terminal path itself.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Classify tool-boundary lifecycle events before ending:** `src/agents/agent-command.ts:1125-1133`
  - body: The new pending-tool guard only runs inside `if (!lifecycleEnded)`. The embedded lifecycle handler can emit `phase: "end"` with terminal `stopReason` before `runAgentAttempt` returns, and `onAgentEvent` sets `lifecycleEnded = true`, so a tool-boundary terminal event still bypasses the new error classification. Move this classification into the lifecycle event path or inspect terminal metadata before marking the lifecycle ended.
  - confidence: 0.84
