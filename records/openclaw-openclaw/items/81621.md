## Summary

Keep open: the PR is a useful bug-fix direction, but current head still has an output-selection blocker and the supplied real proof is from an older hotpatched install rather than this PR head on latest main.

## What This Changes

The PR changes agent/subagent lifecycle and announce-output handling so unresolved tool-boundary endings become lifecycle errors and stale wait/tool output is not selected as completed subagent output, with regression coverage.

## Best Possible Solution

Keep the lifecycle error classification, remove the waiting-state assistant-text escape hatch, add a text-plus-tool-call regression, and require redacted latest-main Telegram proof before merge.

## Reproduction Assessment

Yes, source-level: at PR head, an assistant turn with text plus a tool call sets waitingForContinuation but still returns latestAssistantText as completion output. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No. The lifecycle classification is narrow and appropriate, but output selection must treat waiting-for-continuation as a hard stop until a later non-tool assistant answer exists.
