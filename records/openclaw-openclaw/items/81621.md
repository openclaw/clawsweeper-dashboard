## Summary

Keep open: this is a focused and plausible bug fix with sufficient real-behavior proof, but the proposed patch still leaves two correctness paths where subagent work can be reported as successfully completed too early.

## What This Changes

The PR changes agents subagent lifecycle and announcement handling so unresolved tool-boundary endings are treated as errors and stale tool-boundary output is not selected as final completion text.

## Best Possible Solution

Keep the hosted client-tool exception, but make unresolved tool-boundary failure propagate through the returned/gateway wait status and suppress all tool-call-turn assistant text until a later assistant turn without tool calls provides the final reply.

## Reproduction Assessment

Yes, by source inspection: current main can turn terminal lifecycle `end` and raw tool-result fallback into successful subagent completion, and the PR's own regression tests model much of that path. I did not execute a runtime repro because this review is read-only.

## Solution Assessment

No. The direction is maintainable, but the patch is not yet the best fix because it emits a lifecycle error without making `agent.wait`/gateway status terminally fail, and it can still select assistant text from a tool-call turn as final output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Propagate tool-boundary failures to agent.wait:** `src/agents/agent-command.ts:1146-1157`
  - body: This branch emits a lifecycle `error`, but `agentCommand` still returns the result normally. The gateway `agent` RPC records normal returns as `status: "ok"`, and `agent.wait` races that dedupe snapshot, so a registered subagent can still be completed successfully despite the lifecycle error. Make unresolved tool-boundary completion terminal in the returned result, throw path, or gateway wait contract, and add coverage through `agent.wait`.
  - confidence: 0.86
- **[P2] Keep tool-call assistant text non-terminal:** `src/agents/subagent-announce-output.ts:304-306`
  - body: When the latest assistant message contains both text and a tool call, the new summary path leaves `waitingForContinuation` true while also keeping `latestAssistantText`; this added return would then announce that pre-tool progress text as the final result. While waiting for continuation, return no output until a later assistant turn without tool calls arrives, and add a mixed text+toolCall regression case.
  - confidence: 0.89
