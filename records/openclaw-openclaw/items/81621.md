## Summary

Keep open: the PR is useful and addresses the hosted-tool lifecycle concern, but it still has a source-level blocker where assistant text attached to a tool-call turn can be selected as terminal subagent output before the post-tool continuation arrives.

## What This Changes

The PR changes agent/subagent lifecycle handling for unresolved tool-boundary terminal states, preserves hosted client-tool handoffs, adjusts subagent completion output selection, and adds focused regression tests.

## Best Possible Solution

Keep the lifecycle-error and hosted-tool hardening, but make output selection treat every assistant turn with tool calls as non-terminal progress until a later assistant message without tool calls arrives.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a `chat.history` assistant message with text plus a non-`sessions_yield` tool call sets `waitingForContinuation` but is still returned from `latestAssistantText`.

## Solution Assessment

No. The lifecycle hardening and hosted-tool distinction are the right direction, but the PR is incomplete until the output selector stops returning text from assistant turns that are still at a tool boundary.
