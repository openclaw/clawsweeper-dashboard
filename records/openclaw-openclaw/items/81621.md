## Summary

Keep open: the PR still has a blocking output-selection path that can freeze pre-tool assistant text as completed subagent output, so the central false-completion bug is not fully fixed. The supplied Telegram proof is useful context, but it does not demonstrate the current head handling that still-broken unresolved tool-boundary case.

## What This Changes

The branch changes agent/subagent lifecycle handling and completion-announcement output selection so unresolved tool-boundary endings are treated as errors, with regression tests for pending tool calls and wait/tool output cases.

## Best Possible Solution

Keep the lifecycle error classification, but make `waitingForContinuation` block assistant text from tool-call turns until a later non-tool assistant answer arrives, add a text-plus-tool-call regression test, and refresh redacted current-head Telegram/log proof.

## Reproduction Assessment

Yes, source-level: a `chat.history` ending with an assistant message that has both text and a `toolCall` sets `waitingForContinuation` while leaving `latestAssistantText`, and PR head returns that text. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

No; the lifecycle error direction is narrow and appropriate, but the output-selection exception keeps part of the false-completion behavior alive. The safer fix is to treat waiting-for-continuation as a hard stop for terminal output selection unless a later non-tool assistant turn is present.
