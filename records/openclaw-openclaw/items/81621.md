## Summary

Keep open: this is a plausible focused bug fix, but the current head still has a blocking output-selection regression and the external proof does not show the latest-main after-fix path.

## What This Changes

The PR changes agent and subagent completion handling so unresolved tool-boundary endings become lifecycle errors and tool-boundary transcript text is not selected as final completion output.

## Best Possible Solution

Keep the lifecycle error classification, suppress all assistant text from unresolved tool-call turns until a later non-tool final reply arrives, add regression coverage for that case, and provide redacted latest-main Telegram proof.

## Reproduction Assessment

Yes, source-level: current main has a clear unresolved tool-boundary completion path via stopReason tool_calls and subagent output selection. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No, not yet: the lifecycle direction is narrow, but the PR head reintroduces premature completion text by returning latestAssistantText while waitingForContinuation is true. The safer fix is to keep waiting output empty until a later assistant message without tool calls supplies the final result.
