## Summary

Keep open: the PR is a plausible focused bug fix and the second commit addresses the prior lifecycle-placement concern, but the current diff still lets pre-tool assistant text be selected while a child is waiting for tool continuation, and the real behavior proof is not from the current PR head on latest main.

## What This Changes

The PR changes agent/subagent lifecycle handling and announce-output selection so unresolved tool-boundary endings become lifecycle errors and stale wait/tool output is not announced as completed subagent output.

## Best Possible Solution

Keep the embedded lifecycle error classification, but make waiting tool-turn output selection refuse assistant text until a later non-tool assistant answer arrives, add a text-plus-tool-call regression test, and require redacted latest-main Telegram proof before merge.

## Reproduction Assessment

Yes, source-level: current main treats tool-use terminal assistant turns as incomplete, and the PR head still has a path that selects pre-tool assistant text while `waitingForContinuation` is true. I did not run a live Telegram repro in this read-only review.

## Solution Assessment

No; the lifecycle classification direction is narrow and appropriate, but the output-selection exception keeps part of the false-completion behavior alive. The safer fix is to make waiting-for-continuation a hard stop for terminal output selection unless a later non-tool assistant turn is present.
