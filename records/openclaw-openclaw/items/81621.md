## Summary

Keep open: this is a plausible bug fix, but the current diff appears to miss the embedded lifecycle event path and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR changes agent/subagent completion handling so unresolved tool-boundary endings are treated as lifecycle errors and stale tool-boundary output is not announced as final completion.

## Best Possible Solution

Classify unresolved tool-boundary endings at the canonical lifecycle source, cover both emitted lifecycle events and the backstop path with regression tests, then require redacted after-fix live proof before merge.

## Reproduction Assessment

No high-confidence executed repro in this review; the failure path is source-reproducible from current-main lifecycle and embedded runner code. The PR body cites prior real Telegram-topic evidence but does not provide a fresh after-fix live run.

## Solution Assessment

No; the direction is narrow and relevant, but the patch should handle tool-boundary terminal events before `lifecycleEnded` suppresses the new backstop guard.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Handle tool-boundary lifecycle events before the backstop:** `src/agents/agent-command.ts:1127-1135`
  - body: The new guard only runs inside `if (!lifecycleEnded)`, but the embedded subscriber can already emit a lifecycle `end` event with terminal stop metadata. When that happens `onAgentEvent` sets `lifecycleEnded = true`, so the pending-tool-call check is skipped and the run can still be recorded as a successful end. Move this classification to the lifecycle event path or intercept the terminal event before marking it ended.
  - confidence: 0.82
