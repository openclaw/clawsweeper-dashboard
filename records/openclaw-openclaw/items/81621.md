## Summary

Keep open: this draft PR addresses the right bug family, but the proposed lifecycle hardening still misses the embedded terminal-event path, and the supplied real behavior proof is not from the exact latest-main patch path.

## What This Changes

The PR changes agent/subagent completion handling so unresolved tool-boundary endings become lifecycle errors and stale tool-boundary output is not selected as final completion output.

## Best Possible Solution

Classify unresolved tool-boundary terminal metadata at the canonical lifecycle event source, keep the output-selection hardening, then verify with targeted agent tests and redacted latest-main Telegram proof.

## Reproduction Assessment

Yes, source-level: current main can emit an embedded lifecycle `end` with `stopReason: "tool_calls"` before the `agent-command` backstop runs. I did not execute a live current-main repro in this read-only review.

## Solution Assessment

No; the direction is narrow, but the guard is placed after `lifecycleEnded` can already suppress it. The safer fix is to classify unresolved tool-boundary terminal events before marking lifecycle completion, or inside the embedded lifecycle terminal path itself.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Classify tool-boundary lifecycle events before the backstop:** `src/agents/agent-command.ts:1125-1133`
  - body: The new pending-tool guard only runs when `lifecycleEnded` is false. In the embedded path, `pi-embedded-subscribe.handlers.lifecycle.ts` can emit `phase: "end"` with `stopReason: "tool_calls"`, and `onAgentEvent` marks `lifecycleEnded = true` before this block, so the guard is skipped and the run can still be reported as successful. Move the classification into the lifecycle event path or intercept the terminal event before marking the lifecycle ended.
  - confidence: 0.83
