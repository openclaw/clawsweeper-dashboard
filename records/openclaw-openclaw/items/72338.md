## Summary

Keep open. The report is recent, well corroborated across hosts, and latest comments say v2026.4.29 still regresses; current main has several unreleased mitigations, but there is not yet affected-host retest or shipped-release proof that the broad Gateway CPU/event-loop starvation and Telegram stall are resolved.

## What This Changes

The issue asks for the Gateway to stop entering high-CPU/event-loop starvation states that make Telegram replies stall and local loopback status/deep probes time out.

## Best Possible Solution

Keep this as the canonical broad Gateway CPU/event-loop starvation tracker until maintainers retest main/dev or the next package on an affected install and can tie stable idle CPU, responsive `openclaw gateway probe` / `openclaw channels status --deep`, and a real Telegram or WebChat turn to a specific merged fix and release. Narrow follow-ups such as #73532, #75214, #68944, and #75439 can carry distinct remaining causes if retesting isolates them.

## Reproduction Assessment

Yes for the released regression: the issue thread gives high-confidence reproduction across multiple hosts and versions using CPU observation, loopback Gateway/status timeouts, and delayed or missing Telegram/WebChat replies. No current-main live reproduction or non-reproduction was available in this read-only pass.

## Solution Assessment

Unclear for closure. The current-main fixes are directionally maintainable because they remove runtime-deps repair and heavy plugin work from hot paths, but the best close path is affected-host retest plus shipped-release provenance, not closing from targeted mitigations alone.
