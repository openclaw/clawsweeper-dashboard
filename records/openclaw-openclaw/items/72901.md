## Summary

Keep open. Current main still has the Slack timestamp-only gate, and this PR is a useful narrow fix path, but it is not merge-ready because the freshness check still diverges from the session lifecycle resolver for legacy session rows and the user-facing Slack fix lacks the required changelog entry.

## What This Changes

This PR changes Slack thread-context preparation to treat stale daily/idle thread sessions as new for thread-history loading, passes config into the resolver, reexports existing session helpers through the Slack runtime barrel, and adds regression tests.

## Best Possible Solution

Land the Slack-owned stale-session fix after the thread-history gate reuses the same lifecycle timestamp resolution as session initialization and after adding the required changelog bullet; keep the active-session assistant-turn omission as a separate prompt-assembly investigation if it reproduces.

## Reproduction Assessment

Yes. The original bug is reproducible with a Slack thread session whose stored timestamp survives a daily or idle reset boundary; the remaining PR gap is reproducible with a legacy session entry whose `updatedAt` is fresh, `sessionStartedAt` is missing, and the JSONL session header predates the reset boundary.

## Solution Assessment

No, not yet. The PR direction is narrow and maintainable, but the implementation should call or expose the lifecycle resolver so Slack's prefetch gate and `initSessionState` make the same freshness decision for both current and legacy session rows.
