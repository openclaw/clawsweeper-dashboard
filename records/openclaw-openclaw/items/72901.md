## Summary

Keep open: the PR is pointed at a real Slack stale-session bug and fixes the common direct-field case, but it still diverges from the canonical lifecycle timestamp resolver for legacy session entries and lacks the required user-facing changelog entry.

## What This Changes

The PR changes Slack thread-context preparation to fetch thread history when a stored thread session is stale by reset policy, passes config into the resolver, reexports session-store helpers, and adds regression tests.

## Best Possible Solution

Land the Slack-owned stale-session fix after the gate reuses the canonical lifecycle timestamp resolution, adds legacy header fallback coverage, includes the changelog bullet, and rebases cleanly; keep the active-session assistant-turn report as a separate investigation.

## Reproduction Assessment

Yes. The original bug is reproducible on current main with a Slack thread session that crosses daily or idle reset while retaining `updatedAt`; the remaining PR gap is reproducible with a legacy row whose `updatedAt` is fresh, direct lifecycle fields are absent, and the JSONL header predates the reset boundary.

## Solution Assessment

No, not yet. The PR direction is the narrow owner-module fix, but using only direct lifecycle fields makes Slack's prefetch decision differ from `initSessionState` for legacy session rows; reusing or exposing the resolver is the safer solution.
