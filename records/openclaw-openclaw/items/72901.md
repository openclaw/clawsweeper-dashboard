## Summary

Keep open. The PR is the active narrow fix path for the stale Slack thread-history bug, but the latest head still diverges from the canonical session freshness contract for legacy session rows and is missing the required user-facing changelog entry.

## What This Changes

The PR changes Slack thread-context preparation to load thread history when a prior thread session is stale by daily/idle reset, passes config into the resolver, reexports existing session-store helpers through the Slack runtime barrel, and adds regression tests.

## Best Possible Solution

Land the Slack-owned stale-session fix after the prefetch gate reuses the same lifecycle timestamp resolution as `initSessionState`, includes a legacy `sessionFile` header regression test, and adds the required changelog bullet. Keep the active-session assistant-turn omission as a separate investigation if it still reproduces.

## Reproduction Assessment

Yes. The original bug is reproducible by a Slack thread session that crosses a daily or idle reset boundary while retaining a stored timestamp; the remaining PR gap is reproducible with a legacy session entry whose `updatedAt` is fresh, `sessionStartedAt` is absent, and the JSONL session header predates the reset boundary.

## Solution Assessment

No, not yet. The PR direction is the right narrow owner-module fix, but the implementation should call or expose the lifecycle resolver so Slack's prefetch decision and session initialization make the same freshness decision for current and legacy session rows.
