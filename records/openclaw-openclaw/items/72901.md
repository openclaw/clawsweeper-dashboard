## Summary

Keep this PR open. The Slack stale-session fix is narrow and useful, but the current head still diverges from the canonical session lifecycle freshness path for legacy session rows and is missing the required user-facing changelog entry. The within-minutes assistant-turn omission reported in the discussion appears to be a separate prompt-assembly issue, not a reason to expand this PR before the stale-session fix is made complete.

## What This Changes

The PR changes Slack thread-context preparation to load thread history for stale daily/idle thread sessions, passes config into the resolver, reexports session-store helpers, and adds targeted regression tests.

## Best Possible Solution

Land the Slack-owned stale-session fix after the prefetch gate reuses the same lifecycle timestamp resolution as `initSessionState`, adds a legacy `sessionFile` header regression test, and includes the required changelog bullet. Keep the active-session assistant-turn omission as a separate prompt/transcript investigation if it still reproduces.

## Reproduction Assessment

Yes. The original bug is reproducible by a Slack thread session that crosses a daily or idle reset boundary while retaining a stored timestamp; the remaining PR gap is reproducible with a legacy session entry whose `updatedAt` is fresh, `sessionStartedAt` is absent, and the JSONL session header predates the reset boundary.

## Solution Assessment

No, not yet. The PR direction is the right narrow owner-module fix, but the implementation should call or expose the lifecycle resolver so Slack's prefetch decision and session initialization make the same freshness decision for current and legacy session rows.
