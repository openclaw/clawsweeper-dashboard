## Summary

Keep open. The PR is the right Slack-owned direction for the stale-session thread-history bug, but the current patch still does not match the lifecycle timestamp contract used by session initialization for legacy session entries, and the user-facing Slack fix is missing an Unreleased changelog entry.

## What This Changes

The PR changes Slack thread-context preparation to load the session store, evaluate thread-session freshness with existing reset-policy helpers, pass Slack config into the resolver, re-export existing session helpers, and add regression tests for stale daily and idle thread sessions.

## Best Possible Solution

Land a narrow Slack fix after the thread-history gate either reuses an SDK-exposed lifecycle timestamp resolver or otherwise matches `initSessionState` for legacy entries, adds a regression case for that legacy shape, and adds the required Unreleased Slack fix changelog entry. Track the active-session assistant-turn omission from the PR discussion separately if it reproduces.

## Reproduction Assessment

Yes. The original bug is reproducible from current main by using a Slack thread session with a stored timestamp after a daily or idle reset boundary; the remaining PR gap is reproducible by a legacy entry with a recent `updatedAt`, missing `sessionStartedAt`, and a JSONL session header before the reset boundary.

## Solution Assessment

No, not yet. The PR direction is the narrowest maintainable Slack-owned fix, but the implementation should match the lifecycle timestamp resolver contract already used by session initialization before it is treated as complete.
