## Summary

Keep open. The PR is a narrow and useful Slack fix for stale daily/idle thread sessions, but the current head still diverges from the core session-reset path for legacy session entries that only have a JSONL session header start time, and it is missing the required user-facing changelog entry. The within-minutes assistant-turn omission reported in the discussion looks like a separate prompt-assembly issue, not a blocker for this stale-session fix.

## What This Changes

This PR changes Slack thread-context preparation to fetch thread history when a prior thread session is stale under daily/idle reset policy, passes config into the resolver, reexports existing session-store helpers through the Slack runtime barrel, and adds regression tests.

## Best Possible Solution

Land the Slack-owned stale-session fix after the Slack prefetch gate shares the same lifecycle timestamp resolution as `initSessionState`, including legacy `sessionFile` header fallback coverage, and add the Unreleased changelog bullet. Keep the active-session assistant-turn omission from the discussion as a separate prompt-assembly investigation if it reproduces.

## Reproduction Assessment

Yes. The original bug is reproducible with a Slack thread whose stored session timestamp survives a daily or idle reset boundary; the remaining PR blocker is reproducible with a legacy store entry that has fresh `updatedAt`, no `sessionStartedAt`, and a stale JSONL session header.

## Solution Assessment

No, not yet. The proposed direction is narrow and maintainable, but the implementation should call or expose the lifecycle timestamp resolver so Slack's prefetch gate and core session initialization make the same freshness decision.
