## Summary

Keep open: the reported llama.cpp/local OpenAI-compatible tool-call bug is real, but this PR's buffer replacement heuristic can break normal incremental tool-call streams and the after-fix proof is only simulated unit-test output.

## What This Changes

The PR changes the OpenAI Responses and Chat Completions stream parsers to replace tool-call argument buffers when a chunk parses as complete JSON, and adds a regression test for cumulative llama.cpp-style chunks.

## Best Possible Solution

Use a narrow cumulative-chunk detector that only replaces when the new chunk is a clear superset/prefix of the current buffer, or leave the linked bug to the cron-side recovery path in https://github.com/openclaw/openclaw/pull/88946.

## Reproduction Assessment

Yes, source-level: current main appends tool-call argument deltas, and the provided cumulative chunks explain how a server that resends complete snapshots corrupts the buffer. I did not verify the live llama.cpp setup.

## Solution Assessment

No. The PR is a plausible mitigation, but replacing on any JSON-valid chunk is too broad; the safer solution is prefix/superset cumulative detection or the active cron-side recovery PR.
