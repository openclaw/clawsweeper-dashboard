## What This Changes

The issue asks for an operator-facing warning, validation failure, or safe setup behavior when `channels.discord.enabled` is set but the external Discord plugin is not explicitly trusted via `plugins.entries.discord.enabled` or `plugins.allow`.

## Best Possible Solution

Land a focused doctor/setup diagnostic that preserves the explicit external plugin trust boundary and tells Discord operators to enable `plugins.entries.discord.enabled=true` or add `discord` to `plugins.allow`, then close this issue from the merged PR.

## Reproduction Assessment

Yes. Source inspection shows current main still skips blocker diagnostics when `plugins.entries.discord` is missing, while the external channel ownership policy treats the untrusted plugin as ineffective.

## Solution Assessment

Yes for the warning/diagnostic path, not for implicit enablement. The narrow maintainable fix is to keep explicit plugin trust and surface the missing trust as an actionable doctor/setup warning.
