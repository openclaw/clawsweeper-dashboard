## Summary

Keep open: current main and v2026.5.18 still skip the missing `plugins.entries.discord.enabled` case, and the open linked fix PR has not merged yet.

## What This Changes

The issue asks OpenClaw to warn, fail fast, or otherwise avoid silently leaving Discord disabled when `channels.discord` is enabled but the external Discord plugin lacks explicit registry enablement or trust.

## Best Possible Solution

Keep the explicit external-plugin trust boundary, then land https://github.com/openclaw/openclaw/pull/83252 or an equivalent fix that makes doctor/setup diagnostics warn about the missing Discord plugin trust entry.

## Reproduction Assessment

Yes from source, though not from a live gateway run. Current main and v2026.5.18 skip the blocker scanner when `plugins.entries.discord` is absent, while external Discord still requires explicit trust to become effective.

## Solution Assessment

Yes for an operator-facing diagnostic or setup repair; no for implicit enablement that weakens external plugin trust. The active fix path should make the production doctor/setup path emit the warning and keep the trust policy intact.
