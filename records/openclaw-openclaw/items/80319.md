Keep open: the report was narrowed to a real QA harness/mock-provider gap, and current main does not yet contain the `tool-defaults` runtime suite that would make this implemented or obsolete. ## What This Changes

The issue asks the QA runtime parity mock lane to model Codex-native workspace tools and searchable/deferred OpenClaw dynamic tools accurately.

## Best Possible Solution

Keep this as the specific harness-gap tracker and resolve it in the runtime-parity PR or a focused follow-up by separating Codex-native tools from OpenClaw dynamic tools and making mock mode model searchable/deferred loading honestly.

## Reproduction Assessment

No high-confidence current-main failing command is available because current main does not contain the `tool-defaults` runtime suite. The source path is clear, though: Codex hides native workspace tools and defaults remaining dynamic tools to searchable loading, while the mock planner only plans tools declared in `body.tools`.

## Solution Assessment

Yes, the corrected direction in the issue is the right one: fix the QA harness/mock-planner boundary instead of treating mock-provider absence as proof of broad user-facing Codex runtime tool loss. The exact implementation should be reviewed with https://github.com/openclaw/openclaw/pull/80323.
