## Summary

Current main and the latest release already preserve the configured fallback chain for auto-selected embedded fallback attempts, so the reported single-candidate exhaustion from v2026.5.22 is no longer actionable.

## What This Changes

The issue reports that embedded runtime model failover stops when an intermediate fallback candidate fails instead of continuing through the configured fallback list.

## Best Possible Solution

Keep the shipped fallback override handoff; reopen with fresh v2026.6.6-or-newer logs only if current runs still show a single-candidate chain for an auto fallback attempt.

## Reproduction Assessment

Partially; the original v2026.5.22 logs provide a concrete reproduction, and source review shows the handoff that now prevents a one-candidate embedded fallback attempt. I did not run a live provider-failure scenario in this read-only review.

## Solution Assessment

Yes; preserving the effective fallback override at the agent-command boundary and forwarding it into embedded attempts is the narrow maintainable fix for the reported standalone fallback-candidate invocation.
