## Summary

Keep open: this is a focused Discord message-loss fix attempt, but the current patch still crosses the existing `message_tool_only` privacy boundary, can bypass hook-block suppression, and does not yet include live Discord proof for the latest PR head.

## What This Changes

The PR adds Discord-specific `runReplyAgent` guards and e2e coverage that synthesize or mark source-deliverable payloads when a Discord turn lacks committed message-tool delivery evidence.

## Best Possible Solution

Preserve strict `message_tool_only` privacy by default, add a narrow generic visible guard only for approved missing-result cases, and require live Discord proof for both visible guard delivery and quiet privacy cases.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path for the review findings: the PR marks ordinary finals before hook-block metadata is applied, while current dispatch only checks the suppression-bypass marker. I did not run live Discord because this review is read-only.

## Solution Assessment

No. The narrow maintainable path is to keep normal `message_tool_only` finals private by default, skip hook-blocked payloads, and mark only the intended guard/result payload before operational prefix/trace/usage payloads are added.
