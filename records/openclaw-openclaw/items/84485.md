## Summary

Keep this PR open: the latest head still changes the documented `message_tool_only` privacy boundary, broadens Discord guard delivery beyond the stated source-reply gap, and does not yet include convincing live Discord behavior proof.

## What This Changes

The PR adds Discord-specific `runReplyAgent` guards and e2e coverage that synthesize or mark deliverable final payloads when no committed message-tool delivery evidence is seen.

## Best Possible Solution

Preserve strict `message_tool_only` privacy by default; if maintainers want a Discord fallback, make it an explicit documented policy with focused regression coverage and redacted live Discord positive and negative proof.

## Reproduction Assessment

Yes for the review findings: source inspection of the PR head shows Discord finals being marked for source-delivery bypass, while current docs and dispatch tests define missed message-tool finals as private. I did not run live Discord or local tests in this read-only review.

## Solution Assessment

No: the proposed runtime guard is not the safest default because it changes a documented privacy boundary and reaches automatic-mode Discord turns too broadly. A narrower guard/error path or a maintainer-approved opt-in fallback policy would be more maintainable.
