## Summary

Keep this PR open, but it still needs changes before merge: the latest fix only treats block-streamed replies as delivered for the `message_tool_only` guard path, while the new non-`message_tool_only` guard path can still append a Discord delivery guard after an already visible non-status block-streamed answer.

## What This Changes

The PR adds a Discord `runReplyAgent` delivery guard that returns a source-deliverable warning when a Discord turn finishes without confirmed source-visible delivery evidence.

## Best Possible Solution

Keep the guard, but make the visible-delivery predicate distinguish progress-only chatter from already delivered source-visible block replies across every branch where the guard can run.

## Reproduction Assessment

Yes. Source inspection shows a non-`message_tool_only` Discord turn with a successful non-status block-streamed answer and then an empty or `NO_REPLY` final still satisfies the new guard candidate path.

## Solution Assessment

No, not yet. The guard belongs near the `runReplyAgent` result/suppression boundary, but the predicate must count already delivered block-streamed answers as visible delivery anywhere the guard can run.
