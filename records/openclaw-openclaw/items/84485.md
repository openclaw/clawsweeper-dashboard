## Summary

Keep open: this PR targets a real Discord message-loss path, but the current head still has a blocking guard predicate bug around block-streamed progress versus a delivered answer.

## What This Changes

The PR adds a Discord-specific `runReplyAgent` delivery guard for `message_tool_only` turns and regression coverage for missing, duplicate, and current-source message-tool delivery evidence.

## Best Possible Solution

Keep the Discord guard, but make the visible-evidence predicate distinguish a substantive already-delivered answer from progress-only block output before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Source inspection gives a clear reproduction shape: Discord `message_tool_only`, no committed source `message.send`, progress/block activity, then `NO_REPLY` or empty final at `runReplyAgent`; I did not run tests because this review is read-only.

## Solution Assessment

Is this the best way to solve the issue? Not yet: the `runReplyAgent` boundary is the right layer, but the current predicate uses too broad a block-stream evidence bit and should split delivered answer evidence from progress-only activity.
