## What This Changes

The PR adds a Discord runtime guard for message-tool-only and related empty-final reply paths, plus runReplyAgent and block-streaming regression coverage.

## Best Possible Solution

Keep the generic guard path, but make delivered substantive block replies count as current-source visible evidence in every Discord mode and avoid text-prefix guessing for answer-vs-progress classification before merge.

## Reproduction Assessment

Yes, from source inspection: a non-message-tool-only Discord run with a substantive block-streamed answer and empty/NO_REPLY final reaches the new guard because `hasSubstantiveBlockReply` is ignored outside message-tool-only mode.

## Solution Assessment

No: the runtime guard is a plausible layer, but the current predicate needs to treat substantive block delivery as visible evidence across Discord modes and use a structured progress signal instead of broad text-prefix inference.

- **[P1] Count streamed answers before adding the guard:** `src/auto-reply/reply/agent-runner.ts:2121-2124`
  - body: This guard also runs when Discord is not in `message_tool_only` mode, but the visible-evidence check only honors `hasSubstantiveBlockReply` when that mode is active.   - confidence: 0.89

  - body: The new `isProgressLikeBlockReply` treats any block reply starting with common symbols like `📄`, `🔎`, or `🌐` as progress. Real assistant answers can start with those same symbols, so a delivered answer such as `📄 Summary...` will not count as substantive delivery and can still trigger a duplicate Discord guard. This needs structured status/progress metadata or a much narrower marker.
  - confidence: 0.82
