## Summary

Keep this PR open, but it should not merge as-is: the Discord delivery-loss bug is real, yet the patch still bypasses the private-final boundary for message_tool_only turns and can treat a cross-target message send as source-delivery evidence.

## What This Changes

The branch adds a Discord-specific runReplyAgent delivery guard that marks final payloads or synthesizes a guard payload when Discord turns finish without visible delivery evidence, plus e2e coverage.

## Best Possible Solution

Keep the strict message_tool_only privacy boundary, use only a generic Discord guard when current-source delivery is not verified, and require live PR-head Discord proof after the code is corrected.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: current message_tool_only suppression drops unmarked finals, and the PR's tests model the NO_REPLY, empty-final, and skipped-message cases. I did not run a live Discord reproduction in this read-only review.

## Solution Assessment

No. A generic visible guard for missing Discord delivery is a plausible fix shape, but auto-delivering ordinary private finals and accepting any target as source evidence is not the safest or narrowest implementation.
