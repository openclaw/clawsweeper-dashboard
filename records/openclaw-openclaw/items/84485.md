## Summary

Keep this PR open, but it should not merge as-is: the guard addresses a real Discord message-loss class, yet the substantive-final fallback changes the documented strict `message_tool_only` privacy/delivery contract and still lacks real Discord/Gateway proof.

## What This Changes

The PR adds a Discord-specific runtime guard that either surfaces a visible guard payload or marks one substantive final payload for delivery when a `message_tool_only` turn has no confirmed visible message-tool/media/target delivery.

## Best Possible Solution

Preserve strict `message_tool_only` privacy by default; if maintainers want recovery, use a generic visible delivery-error guard or an explicit opt-in fallback path with Discord runtime proof and upgrade coverage.

## Reproduction Assessment

Yes from source, though I did not run tests in this read-only review: current dispatch suppresses `message_tool_only` finals without bypass metadata, and the PR adds tests for the missing-message paths.

## Solution Assessment

No as written. The generic guard idea is narrow, but delivering normal final text by default conflicts with the current strict `message_tool_only` contract; an opt-in fallback or generic delivery-error notice is safer.
