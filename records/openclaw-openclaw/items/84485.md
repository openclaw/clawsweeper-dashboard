## Summary

Keep open; this is a focused Discord delivery fix with meaningful proof, but it is not merge-ready because the new guard can treat an already delivered block-streamed Discord reply as missing and append an error guard after a successful visible answer.

## What This Changes

The PR adds a Discord `runReplyAgent` guard that synthesizes or preserves a source-deliverable payload when a `message_tool_only` or related Discord turn appears to finish without a confirmed visible source reply.

## Best Possible Solution

Keep the runtime guard, but make it distinguish progress-only chatter from already delivered block-streamed replies before it can synthesize a visible error.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: an automatic Discord turn with block streaming that sends a visible non-status block reply and then ends with an empty final will satisfy `hasPotentialResponseActivity` but not the guard's visible-reply evidence check.

## Solution Assessment

No, not as written. The guard belongs near the `runReplyAgent` result/suppression boundary, but it needs to count successful block-streamed source replies as delivered or avoid the non-`message_tool_only` block-stream path.
