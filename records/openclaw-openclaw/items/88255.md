## Summary

Keep open. Current main and v2026.5.27 still call `msg.content.filter` in the export HTML template after accepting message rows without validating `message.content`, so the reported export viewer crash remains a real source bug.

## What This Changes

The issue requests an `Array.isArray` guard in the exported HTML template bundle before filtering message content for tool calls.

## Best Possible Solution

Normalize message and tool-result content before every template filter/iteration path, add a regression render test for malformed assistant content, and let the build copy step regenerate the dist bundle from source.

## Reproduction Assessment

Yes at source level: a transcript message row with an assistant role but non-array or missing content passes export parsing and reaches `msg.content.filter` in the template stats path. I did not run the browser template in this read-only review.

## Solution Assessment

Yes, the maintainable fix is to update the source export template and regression tests, not only patch a generated dist artifact. The repair should audit sibling content access sites so the malformed-message class is fixed consistently.
