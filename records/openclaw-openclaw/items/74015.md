## Summary

Keep open. Current main still exposes `llm_input` and `llm_output` as raw-conversation observation hooks, not writable Plugin SDK mutation hooks; the gap is real core SDK surface work rather than a plugin-only implementation request.

## What This Changes

Request writable Plugin SDK LLM input/output hook results so plugins can rewrite prompts, system prompts, history, and assistant output for local sanitization and resensitization.

## Best Possible Solution

Design an additive, versioned Plugin SDK mutation contract for trusted pre-provider and post-output content transforms, with explicit permissions, deterministic handler ordering, docs/API baseline updates, and coverage across embedded, CLI, and Codex harness paths.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: source and docs show observation-only LLM hooks with no writable result contract or call-site consumption.

## Solution Assessment

Unclear because there is no patch yet. The requested capability is valid, but silently making today’s fire-and-forget observation hooks mutable may be less maintainable than a deliberate permissioned mutation seam.
