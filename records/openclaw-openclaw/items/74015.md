## Summary

Keep open. Current main still implements `llm_input` and `llm_output` as observation-only Plugin SDK hooks, while the requested raw prompt/history/output mutation is a new security-sensitive SDK capability that needs maintainer API design.

## What This Changes

The issue requests writable Plugin SDK LLM input/output hooks so plugins can modify prompts, system prompts, history messages, and assistant output text.

## Best Possible Solution

Design an additive, permissioned Plugin SDK mutation seam for pre-provider input and post-output content, with docs, API baselines, and focused coverage across Pi, CLI, and Codex harnesses.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. The feature gap is high-confidence from source and docs: current LLM hooks are void-only observation hooks, and existing prompt hooks cannot replace raw prompt text, history, or assistant output.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is valid, but the maintainable solution should be an explicit permissioned mutation contract rather than silently making today’s fire-and-forget observation hooks writable.

## Security Review

Status: needs_attention

Summary: The requested SDK seam is security-sensitive because trusted plugins would be able to rewrite raw conversation content around provider calls.

Concerns:

- **[medium] Define raw-content mutation trust gates:** `src/plugins/hook-types.ts:822`
  - body: Changing `llm_input` and `llm_output` from observation to mutation would let plugins alter user prompts, history, and assistant output before or after provider boundaries, so the design should specify permissions, sequencing, auditability, and failure behavior.
  - confidence: 0.92
- **[medium] Align mutation policy with existing hook controls:** `src/config/types.plugins.ts:4`
  - body: Current config separates prompt mutation (`allowPromptInjection`) from raw conversation reads (`allowConversationAccess`); writable raw conversation hooks need explicit policy semantics instead of inheriting read-only access implicitly.
  - confidence: 0.86
