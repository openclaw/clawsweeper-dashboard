## Summary

Keep open. Current main still exposes `llm_input` and `llm_output` as observation-only Plugin SDK hooks, the runner discards handler results through the void-hook path, and embedded/CLI/harness call sites fire these hooks without awaiting or applying returned mutations. The linked draft SDK follow-up PR is useful adjacent context, but it explicitly does not implement writable prompt mutation, so this remains a real core SDK gap that needs maintainer API and security review.

## What This Changes

The issue requests making Plugin SDK LLM input/output hooks able to return modified prompt, system prompt, history, and assistant output values for privacy-oriented content transformation.

## Best Possible Solution

Design an additive, versioned Plugin SDK contract for trusted pre-provider input mutation and post-output mutation, or a deliberate evolution of the existing LLM hooks, with explicit raw-content permissions, deterministic handler ordering, documented merge/failure semantics, API baseline updates, and coverage across embedded, CLI, and Codex harness paths.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: source and docs show observation-only LLM hooks with no writable result contract or call-site consumption.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is valid, but silently making today’s fire-and-forget observation hooks mutable may be less maintainable than a designed permissioned mutation seam.
