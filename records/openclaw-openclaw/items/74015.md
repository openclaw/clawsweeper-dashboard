## Summary

Keep open. Current main still exposes `llm_input` and `llm_output` as observation-only hooks and existing prompt hooks only prepend/append context or replace system prompt text, so the requested raw prompt/history/output mutation API remains unimplemented.

## What This Changes

The issue requests writable Plugin SDK LLM hooks so trusted plugins can modify prompts, system prompts, history messages, and assistant text around model calls.

## Best Possible Solution

Design an additive trusted Plugin SDK mutation seam for pre-provider input and post-output content, with explicit gating, deterministic ordering, failure semantics, docs/API baselines, and focused PI, CLI, and Codex harness coverage.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. The feature gap is source-verifiable: current LLM hooks are void-only observations and prompt hooks do not expose raw prompt/history replacement fields.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is valid, but silently making today’s fire-and-forget observation hooks writable is riskier than a deliberate permissioned mutation contract.

## Risks / Open Questions

- The requested API is security-sensitive because trusted plugins would be able to rewrite raw user content before provider calls and alter model output after it returns.
- Ordering, merge semantics, failure policy, auditability, and interaction with existing `allowPromptInjection` and `allowConversationAccess` gates need maintainer design before implementation.
- Codex runtime docs describe `llm_input` and `llm_output` as adapter observations, so writable semantics may not be equally enforceable across all harnesses.
