## Summary

Keep open. Current main still exposes `llm_input` and `llm_output` as observation-only Plugin SDK hooks: their public handler types return `void`, the hook runner discards return values, and the embedded/CLI harness call sites fire the hooks without awaiting or applying rewritten prompt/output data. This is not ClawHub-only because a privacy/desensitization plugin is blocked by a missing core SDK mutation seam.

## What This Changes

The issue requests Plugin SDK support for plugins to rewrite the user prompt, system prompt, history messages, and assistant text around LLM calls.

## Best Possible Solution

Define an additive, permissioned Plugin SDK mutation contract for trusted pre-provider input rewriting and post-output rewriting, with deterministic handler ordering, explicit raw-content permissions, failure semantics, docs, API baseline updates, and coverage across embedded, CLI, and harness paths.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature request. The current-main verification path is high confidence: source and docs show the LLM hooks are observation-only and no call site consumes writable prompt/output results.

## Solution Assessment

Unclear because there is no patch yet. The requested capability is valid, but changing today’s fire-and-forget observation hooks directly is less maintainable than a deliberately permissioned mutation seam with ordering and audit behavior.
