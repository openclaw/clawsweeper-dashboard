## Summary

Keep open. Current main still treats `llm_input` and `llm_output` as raw-conversation observation hooks: SDK types return `void`, the hook runner uses the parallel void path, and embedded/CLI/Codex harness call sites fire the hooks without awaiting or consuming mutation results. This is a real Plugin SDK gap rather than a ClawHub-only plugin request because the needed mutable raw-prompt/output seam does not exist yet.

## What This Changes

The issue requests extending existing Plugin SDK LLM hooks so trusted plugins can rewrite the user prompt, system prompt, history, and assistant output for privacy workflows such as local PII redaction and restoration.

## Best Possible Solution

Design an additive, versioned Plugin SDK contract for trusted pre-provider and post-output mutation, or a deliberate evolution of the existing LLM hooks, with explicit raw-content permissions, deterministic handler ordering, docs, API baseline updates, and coverage across embedded, CLI, and Codex harness paths.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: source and docs show observation-only LLM hooks with no writable result contract or call-site consumption.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is valid, but silently making the current fire-and-forget observation hooks mutable may be less maintainable than a designed permissioned mutation seam.
