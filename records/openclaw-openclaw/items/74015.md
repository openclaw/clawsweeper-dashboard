## Summary

Keep open. Current main still exposes `llm_input` and `llm_output` as observation-only Plugin SDK hooks, and the existing prompt-build hooks can add context or system-prompt text but cannot rewrite the raw user prompt, history, or assistant output. This remains a core SDK design gap because an external privacy plugin cannot implement the requested desensitization/resensitization flow without a new trusted mutation seam.

## What This Changes

The issue requests writable Plugin SDK LLM input/output hooks so plugins can redact or transform prompts, system prompts, history, and assistant output around provider calls.

## Best Possible Solution

Define an additive, permissioned SDK contract for trusted pre-provider input mutation and post-output mutation, then update API baselines, config/schema/help, docs, and focused contract/runtime tests across Pi, Codex, and CLI harness paths.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. The verification path is high-confidence: current source and docs show LLM input/output hooks are observation-only and returned values are never applied.

## Solution Assessment

Unclear because there is no patch yet. The requested capability is valid, but changing today's fire-and-forget hooks in place is likely less maintainable than a deliberate sequential mutation seam with explicit trust, ordering, and failure semantics.
