## Summary

Keep open: the latest head fixes part of the bypass, but it still diverges from normal tool construction for runtime policy and group context, and it applies sandbox policy even when no sandbox is active. The broader #75125 approach is still the better implementation candidate to compare against.

## What This Changes

The PR adds manual effective tool-policy filtering to `command-dispatch: tool` skill slash commands in `src/auto-reply/reply/get-reply-inline-actions.ts`.

## Best Possible Solution

Route inline tool-dispatched skill commands through `createOpenClawCodingTools(...)` or a shared effective-tool helper with runtime policy session, target session metadata, sender/account group context, active sandbox context, provider/model context, and focused regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available: configure a restricted runtime-policy session or sender/group-scoped policy, add a user-invocable skill dispatching a denied tool such as `sessions_spawn`, and compare inline dispatch with the normal effective tool list.

## Solution Assessment

No. The intended direction is correct, but manually rebuilding the pipeline still misses canonical context and adds a non-sandbox over-filtering regression; sharing the normal tool builder or policy helper is safer.

- **[P1] Resolve policies with the runtime policy session:** `src/auto-reply/reply/get-reply-inline-actions.ts:318`
  - body: Normal turns first derive a runtime policy session key from the message context, but this branch filters with the raw `sessionKey`. Direct main aliases, command target sessions, or explicit `RuntimePolicySessionKey` contexts can still be evaluated under a different profile/group/sandbox/subagent policy than the real run, leaving the inline dispatch surface out of parity.
  - confidence: 0.9
- **[P1] Thread sender context into group policy:** `src/auto-reply/reply/get-reply-inline-actions.ts:323-327`
  - body: The PR calls `resolveGroupToolPolicy` with only `sessionKey` and `ctx.From`, while normal turns also pass message provider, account id, group channel/space, spawnedBy, and sender identity. Sender/account-scoped group policies will not be applied the same way for skill tool dispatch, so group policy can still diverge from the normal effective tool surface.
  - confidence: 0.86
- **[P2] Gate sandbox policy on an active sandbox:** `src/auto-reply/reply/get-reply-inline-actions.ts:339`
  - body: `resolveSandboxToolPolicyForAgent` returns the default sandbox deny list even when the session is not sandboxed. Because this branch always appends that policy, unsandboxed skill tool commands lose tools such as browser/cron/channel tools that normal turns keep; normal tool construction only applies `sandbox?.tools`.
  - confidence: 0.92
