## Summary

Keep this PR open, but do not merge it as-is: it moves in the right direction while still leaving provider, group/sender, and sandbox tool-policy bypasses in the skill tool-dispatch path.

## What This Changes

The PR filters skill slash commands with `command-dispatch: tool` through additional effective tool-policy checks before invoking the requested tool.

## Best Possible Solution

Route skill tool dispatch through a shared effective-tool filtering helper, or pass the same provider/model, group/account/sender, sandbox, and subagent context as normal agent turns, with targeted regression coverage for `command-dispatch: tool`.

## Reproduction Assessment

Yes. Current main reproduces from source: a user-invocable tool-dispatch skill reaches `createOpenClawTools`, applies only owner-only filtering, and can find the named tool before profile/group/subagent/sandbox policy is applied.

## Solution Assessment

No. The proposed approach is directionally right, but this implementation is not the best fix because it hand-recreates the pipeline and omits policy inputs that the established normal-turn path already uses.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Pass full provider and group context into policy resolution:** `src/auto-reply/reply/get-reply-inline-actions.ts:315-324`
  - body: The new policy pass calls `resolveEffectiveToolPolicy` without the active provider/model and `resolveGroupToolPolicy` without message provider, account, or sender fields. Normal agent turns pass those values, and the resolvers use them for `tools.byProvider`, provider profiles, account-scoped group config, and `toolsBySender`, so those policies can still be bypassed by a tool-dispatch skill.
  - confidence: 0.91
- **[P1] Include sandbox policy in skill command filtering:** `src/auto-reply/reply/get-reply-inline-actions.ts:354`
  - body: The new pipeline appends subagent filtering but never includes the sandbox tool policy. `createOpenClawCodingTools` applies `sandbox tools.allow` before subagent filtering, so sandbox-denied tools remain callable through `command-dispatch: tool`, which is one of the policy bypasses this PR is meant to close.
  - confidence: 0.84
