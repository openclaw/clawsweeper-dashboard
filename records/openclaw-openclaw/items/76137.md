## Summary

This PR should stay open because current main still has the reported bypass, but the proposed patch is not yet mergeable: it duplicates the normal tool-policy assembly and still misses sender-scoped group policy while also applying sandbox/subagent denies outside their active contexts.

## What This Changes

The PR updates `src/auto-reply/reply/get-reply-inline-actions.ts` so tool-dispatch skill slash commands resolve effective policies and filter tools through `applyToolPolicyPipeline` before execution.

## Best Possible Solution

Route inline tool-dispatched skill commands through `createOpenClawCodingTools(...)` or a shared effective-tool helper with canonical runtime policy session, target-session group metadata, sender identity, sandbox context, and regression tests.

## Reproduction Assessment

Yes. A static reproduction exists with a group `toolsBySender` policy for one sender and a tool-dispatch skill for a denied tool; the PR's new `resolveGroupToolPolicy` call has no sender/account fields to match that policy. A second static check is a non-sandbox, non-subagent session with a skill targeting `session_status` or `gateway`, which this patch filters out through default subagent/sandbox deny stages even though normal turns do not.

## Solution Assessment

No. The direction is right, but duplicating part of the policy pipeline is not the safest fix; reusing the canonical tool builder or extracting its shared policy-context construction would avoid the missing group context and active-context gating mistakes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Pass sender context into group policy:** `src/auto-reply/reply/get-reply-inline-actions.ts:323-327`
  - body: The new inline policy path calls `resolveGroupToolPolicy` with only config, session key, and a group id. Sender/account metadata is required for `toolsBySender` matching, while the normal tool builder forwards those fields, so a sender-scoped group deny can still be bypassed by a tool-dispatch skill command.
  - confidence: 0.86
- **[P2] Gate sandbox and subagent policy to active contexts:** `src/auto-reply/reply/get-reply-inline-actions.ts:337-338`
  - body: These helpers are not neutral fallbacks: sandbox policy returns default sandbox denies, and subagent policy returns main-role denies even for ordinary sessions. Calling both unconditionally makes host/non-subagent skill commands lose tools such as `session_status`, `gateway`, `cron`, and channel tools that normal agent turns would still expose.
  - confidence: 0.91
