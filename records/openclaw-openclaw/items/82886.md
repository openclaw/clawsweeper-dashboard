## Summary

Keep open: current main and the latest release still read Slack block-action thread context only from `container.thread_ts`, and Slack still does not expose heartbeat typing hooks for these system-event follow-up turns. The linked closing PR targets the right code path but is still open, draft, and conflicting, so this issue remains necessary until that PR or an equivalent fix lands.

## What This Changes

Slack interactive button/select follow-up turns in threads should preserve the originating thread context and show Slack native assistant status/typing like ordinary Slack message turns.

## Best Possible Solution

Land or replace the linked Slack PR with a focused fix that preserves message-level thread timestamps and wires Slack assistant thread status into the existing heartbeat typing hooks, backed by focused tests and real Slack proof.

## Reproduction Assessment

Yes, source-level: a Slack block action with `message.thread_ts` and no `container.thread_ts` leaves parsed `threadTs` undefined before session-key resolution, `deliveryContext.threadId`, and immediate heartbeat wake. I did not run a live Slack workspace reproduction in this read-only pass.

## Solution Assessment

Yes, the narrow fix direction is best: preserve Slack interaction thread context and implement the Slack heartbeat typing adapter through the existing channel heartbeat contract. The issue should stay open until the linked PR lands or maintainers replace it with equivalent code and proof.
