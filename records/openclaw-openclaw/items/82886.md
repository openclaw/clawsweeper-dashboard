Keep open: current main still has a source-reproducible Slack thread-context gap, and an open draft PR directly references this issue with closing syntax, so it should remain open until that PR is landed or rejected.

## What This Changes

Slack interactive reply button/select follow-up turns should show native Slack assistant status/typing in the relevant Slack thread.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/82895 once it has real Slack proof, keeping the fix in the Slack plugin by preserving `message.thread_ts` and adding heartbeat status support.

## Reproduction Assessment

Yes from source inspection, but not from a live Slack workspace. Current main reads block-action `threadTs` only from `container.thread_ts`, then uses it for session-key and `deliveryContext.threadId`, so `message.thread_ts`-only payloads lose the thread target.

## Solution Assessment

Yes: the narrow maintainable direction is to fix Slack's interaction parser and Slack heartbeat adapter inside the plugin boundary. The issue should stay open until the linked implementation PR is proven and merged or closed.
