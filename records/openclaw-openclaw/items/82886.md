## Summary

Keep open: current main still has a real Slack-owned gap for interactive reply follow-up liveness. Ordinary Slack message turns can set Slack assistant thread status, and interactive reply clicks now enqueue/wake a thread-aware heartbeat, but the Slack channel plugin does not expose the heartbeat typing hooks that heartbeat runs use for channel-native busy indicators.

## What This Changes

The issue requests Slack native assistant status/typing for button/select-triggered follow-up agent turns, matching ordinary Slack message turns.

## Best Possible Solution

Implement a Slack plugin heartbeat typing adapter that sets and clears Slack assistant thread status for heartbeat-driven Slack delivery targets, and preserve interaction threadTs from all contract-backed block-action locations.

## Reproduction Assessment

Yes, source-level: a Slack reply-button click queues a heartbeat system event for Slack delivery, but the Slack plugin has no heartbeat typing adapter, so heartbeat cannot call Slack native status while the follow-up run is active. The current parser also only reads container.thread_ts, which can drop the thread status target for payload shapes that carry the thread on the message.

## Solution Assessment

Yes, a narrow Slack plugin fix is the maintainable path: reuse ChannelHeartbeatAdapter sendTyping/clearTyping and Slack setStatus rather than adding core special cases. The thread timestamp fallback should be added only after checking the Slack/Bolt payload contract.
