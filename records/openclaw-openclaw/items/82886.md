## Summary

Keep open: current main still drops Slack block-action thread context when Slack only supplies `message.thread_ts`, and the linked draft fix PR remains open with closing syntax.

## What This Changes

Slack interactive reply follow-up turns should preserve thread context and show native assistant status/typing in threaded Slack conversations.

## Best Possible Solution

Land a Slack-plugin fix that preserves `message.thread_ts` into session and delivery context and implements heartbeat `sendTyping`/`clearTyping` through Slack assistant thread status, with focused tests and redacted live Slack proof.

## Reproduction Assessment

Yes from source inspection, not from a live Slack workspace. A block action payload with `message.thread_ts` and no `container.thread_ts` produces no parsed `threadTs`, so the queued follow-up lacks `deliveryContext.threadId`.

## Solution Assessment

Yes: the narrow maintainable fix belongs in the Slack plugin by preserving the thread fact and using the existing heartbeat adapter contract. The open linked PR follows that direction but still needs real Slack proof before landing.
