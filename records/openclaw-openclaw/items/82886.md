## Summary

Keep open: current main still drops Slack block-action thread context when Slack only provides the thread timestamp on the message payload, and the Slack plugin still lacks a heartbeat typing/status adapter. The linked draft implementation candidate at https://github.com/openclaw/openclaw/pull/82895 uses closing syntax, so this issue should stay open until that PR lands or is closed.

## What This Changes

Slack interactive button/select follow-up turns should show native Slack assistant typing/status in the relevant thread, matching ordinary Slack message turns.

## Best Possible Solution

Keep the fix in the Slack plugin boundary: preserve `message.thread_ts` for block actions, pass it through session and delivery context, and land Slack heartbeat status support after focused tests and real Slack proof.

## Reproduction Assessment

Yes from source, but not from a live Slack workspace. A block action with `message.thread_ts` and no `container.thread_ts` produces `threadTs: undefined`, and that value is what the follow-up heartbeat delivery path uses as the Slack thread target.

## Solution Assessment

Yes: the narrow maintainable solution is to preserve the Slack interaction thread fact and implement the existing heartbeat typing adapter contract inside the Slack plugin. The open draft PR at https://github.com/openclaw/openclaw/pull/82895 follows that direction, but needs real behavior proof before merge.
