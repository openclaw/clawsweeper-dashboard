## Summary

Keep open: current main and the latest release still drop Slack block-action thread context when only `message.thread_ts` is present, and Slack still has no heartbeat typing/status adapter for these system-event follow-up turns. The linked closing PR targets the right shape but is still open, draft, and conflicting, so the issue remains necessary until that PR or an equivalent fix lands.

## What This Changes

The issue requests that Slack button/select follow-up turns show native assistant status/typing in the relevant thread, matching ordinary Slack message turns.

## Best Possible Solution

Land or replace the linked Slack PR with a focused fix that preserves message-level thread timestamps and wires Slack assistant thread status into the existing heartbeat typing hooks, backed by focused tests and real Slack proof.

## Reproduction Assessment

Yes from current source inspection, not from a live Slack workspace. A Slack block action with `message.thread_ts` and no `container.thread_ts` still leaves parsed `threadTs` undefined before session-key resolution, `deliveryContext.threadId`, and the immediate heartbeat wake.

## Solution Assessment

Yes, the narrow fix direction is best: preserve Slack interaction thread context and implement Slack's existing heartbeat typing adapter contract. The issue should stay open until the linked PR lands or maintainers replace it with equivalent code and proof.
