## Summary

Keep open: current main still drops Slack block-action thread context when Slack only supplies the parent thread timestamp on `message.thread_ts`, and this issue is already paired with two open fixing PRs that use closing syntax.

## What This Changes

Slack interactive button/select follow-up turns should preserve thread context and show native Slack assistant status/typing like ordinary threaded Slack message turns.

## Best Possible Solution

Land a single Slack-plugin fix that preserves `message.thread_ts` fallback into session/delivery context and adds Slack heartbeat set/clear status support, with focused tests and live Slack proof, then close this issue.

## Reproduction Assessment

Yes from source inspection, but not from a live Slack workspace. A block action with `message.thread_ts` and no `container.thread_ts` parses with `threadTs` undefined, so the queued heartbeat follow-up loses `deliveryContext.threadId`.

## Solution Assessment

Yes: the narrow maintainable solution is to keep the thread timestamp in the Slack interaction path and implement Slack's existing heartbeat typing/status adapter surface, without changing core heartbeat policy.
