## Summary

Keep open: current main still drops Slack interaction thread context when the timestamp is only on the message payload, and Slack has no heartbeat typing/status adapter for the follow-up turn. The open fixing PR at https://github.com/openclaw/openclaw/pull/82895 references this issue with closing syntax, so the right path is to review and land or close that PR rather than close this issue early.

## What This Changes

Slack interactive button/select follow-up turns should preserve thread context and show/clear native Slack assistant status/typing.

## Best Possible Solution

Land a Slack-plugin fix that falls back to `message.thread_ts`, implements heartbeat `sendTyping`/`clearTyping` through Slack assistant thread status, and proves the visible Slack behavior before closing this issue.

## Reproduction Assessment

Yes from source inspection, but not from a live Slack workspace in this review. A block-action payload with `message.thread_ts` and no `container.thread_ts` loses `threadTs`, which then removes the delivery thread target for the heartbeat follow-up.

## Solution Assessment

Yes: the narrow maintainable fix is inside the Slack plugin, preserving the interaction thread timestamp and adding Slack heartbeat status calls through the existing channel heartbeat adapter contract. The open PR at https://github.com/openclaw/openclaw/pull/82895 is the current implementation path, pending draft/proof handling.
