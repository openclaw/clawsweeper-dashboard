## Summary

Keep open: current main and the latest release still parse Slack block-action thread context only from `container.thread_ts`, and the Slack plugin still does not expose the heartbeat typing adapter needed for interaction-triggered follow-up status. The open linked PR targets the right surfaces, but it remains draft/conflicting, so this issue is still necessary until that PR or an equivalent fix lands.

## What This Changes

The issue reports that Slack button/select-triggered follow-up turns run but do not show Slack native assistant status/typing in the originating thread.

## Best Possible Solution

Land or replace the linked Slack PR with a focused fix that preserves message-level thread timestamps and wires Slack assistant thread status into the existing heartbeat typing hooks, backed by focused tests and real Slack proof.

## Reproduction Assessment

Yes, source-level: on current main a block action with only `message.thread_ts` leaves parsed `threadTs` undefined before session-key resolution and `deliveryContext.threadId`; Slack also lacks a heartbeat typing adapter. I did not run a live Slack workspace reproduction in this read-only pass.

## Solution Assessment

Yes, the best fix shape is narrow: add the Slack payload fallback in the interaction parser and implement Slack heartbeat typing through the existing channel heartbeat contract. The issue should stay open until the linked PR lands or maintainers replace it with equivalent code and proof.
