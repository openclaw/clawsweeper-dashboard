## Summary

Keep open. Current main still registers Discord reaction add/remove listeners during monitor startup without checking whether any guild, DM, or group-DM path can emit reaction notifications; the closed unmerged #47588 confirms a narrow source-level fix direction remains available.

## What This Changes

The issue requests skipping Discord reaction add/remove listener registration when no configured guild, DM, or group-DM path can emit reaction notifications.

## Best Possible Solution

Add a Discord plugin startup predicate in registerDiscordMonitorListeners that skips reaction add/remove listeners only when configured guild reaction modes cannot emit and DM/group-DM reaction handling is disabled, while keeping the handler-level fast paths as defense in depth.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: with guildEntries present and all reactionNotifications set to off, dmEnabled false, and groupDmEnabled false, current registerDiscordMonitorListeners still reaches the add/remove listener registration calls.

## Solution Assessment

Yes. The narrow maintainable solution is a local Discord startup guard with focused provider.startup tests; the existing handler fast path is useful but does not address listener registration or event-queue overhead.

## Repair Work Prompt

Fix the Discord reaction listener startup overhead reported in https://github.com/openclaw/openclaw/issues/47516. In the Discord plugin monitor startup path, skip registering DiscordReactionListener and DiscordReactionRemoveListener only when no configured path can produce reaction notifications: all configured guild entries have reactionNotifications set to 'off', DM reactions are disabled, and group-DM reactions are disabled. Keep the existing handler-level fast paths in listeners.reactions.ts as defense in depth. Do not add core owner-specific logic or change reaction delivery semantics for guilds with 'own', 'all', or 'allowlist', for DM-only setups with no guild entries, or for enabled group-DM handling. Add focused startup tests for skip/register cases and adjust existing tests if listener registration expectations need to distinguish disabled reaction paths. Add a user-facing changelog fix entry with appropriate contributor attribution.
