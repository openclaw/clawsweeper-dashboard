## Summary

Keep open. Current `main` still registers Discord reaction add/remove listeners during monitor startup without a predicate for all-disabled reaction notification paths; the later handler fast path avoids some downstream fetch work but does not remove the listener/event-queue overhead reported here. The closed, unmerged #47588 remains useful evidence for a narrow source-level fix direction.

## What This Changes

The issue requests skipping Discord reaction add/remove listener registration when no configured guild, DM, or group-DM path can emit reaction notifications.

## Best Possible Solution

Add a Discord plugin startup predicate in `registerDiscordMonitorListeners` that skips `DiscordReactionListener` and `DiscordReactionRemoveListener` only when configured guild reaction modes cannot emit and both DM and group-DM reaction handling are disabled. Keep the existing handler-level fast paths as defense in depth, preserve registration for `own`/`all`/`allowlist` and DM-only setups, add focused startup tests, and include a user-facing changelog fix entry.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: with `guildEntries` present and all entries using `reactionNotifications: "off"`, plus `dmEnabled: false` and `groupDmEnabled: false`, current `registerDiscordMonitorListeners` still reaches both reaction listener registration calls.

## Solution Assessment

Yes. The narrow maintainable solution is a local Discord startup guard using inputs already available to `registerDiscordMonitorListeners`; the existing handler fast path is useful but does not address listener registration or event-queue overhead.
