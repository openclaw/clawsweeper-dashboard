## Summary

Keep open: the patch targets the reported session-memory symptom, but the consecutive text-equality guard is not the best fix and can miss transcript-branch duplicates while also dropping legitimate adjacent same-text assistant entries. The external PR also has only synthetic/unit-test proof, not real after-fix behavior proof from a thinking-enabled session.

## What This Changes

The PR adds a `lastAssistantText` guard in `getRecentSessionContent()` plus session-memory regression tests for collapsing consecutive identical assistant text.

## Best Possible Solution

Make session-memory read the active transcript branch or use lineage-aware filtering, then add coverage for rewritten/off-branch transcript entries and provide real thinking-enabled session proof.

## Reproduction Assessment

Yes, source-level reproduction is high-confidence: a JSONL transcript containing raw thinking content plus a stripped visible assistant entry exercises the current `getRecentSessionContent()` path. I did not run a live provider session in this read-only review.

## Solution Assessment

No. This is a plausible symptom mitigation, but the best fix should respect the transcript tree/active branch or duplicate lineage rather than deduplicating any adjacent equal assistant text.
