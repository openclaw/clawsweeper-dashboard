## Summary

The PR addresses a real session-memory duplicate problem, but the proposed text-only consecutive assistant dedupe is broader than the reported raw/cleaned thinking duplicate shape and can drop a legitimate assistant memory line when an ignored transcript row sits between identical assistant messages. It also still has only synthetic Vitest proof, while the linked bug is a real thinking-enabled `/new` or `/reset` session behavior.

## What This Changes

The PR changes session-memory transcript extraction to skip consecutive assistant messages with identical extracted text and adds regression tests for duplicate thinking-message transcripts and user-separated repeats.

## Best Possible Solution

Use a narrow parent-linked raw/cleaned thinking duplicate check, or an explicit storage-layer fix if maintainers choose that direction, with regression coverage for ignored-row boundaries and redacted real `/new` or `/reset` proof from a thinking-enabled session.

## Reproduction Assessment

Yes for the underlying bug from source: session-memory reads every user/assistant JSONL text row and the linked issue describes a raw thinking assistant entry followed by a cleaned text-only child. I did not run a live thinking-enabled provider session in this read-only review, so the PR proof remains mock-only.

## Solution Assessment

No: deduping by consecutive assistant text is broader than the reported raw/cleaned thinking duplicate shape. The safer fix is to use transcript identity, such as id/parentId plus the raw/cleaned thinking shape, or make a maintainer-approved storage-layer change.
