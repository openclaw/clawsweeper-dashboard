## Summary

Keep open: current main still has the `workspaceAccess: "none"` read-only sandbox bug, and this PR is a useful fix path, but it is not merge-ready because it misses remote bridge parity, leaves docs inconsistent, and has no real after-fix runtime proof.

## What This Changes

The PR changes sandbox Docker/browser mount handling, local FS bridge writability, config hashing, memory flush gating, docs, and tests so `workspaceAccess: "none"` is treated as writable instead of read-only.

## Best Possible Solution

Land a corrected fix or replacement that treats only `ro` as read-only for the isolated `none` workspace across Docker, browser, local FS bridge, and remote FS bridge paths, with aligned docs, regression tests, and redacted real runtime proof.

## Reproduction Assessment

Yes, at source level. Current main maps `workspaceAccess: "none"` to an isolated sandbox workspace but marks Docker, local bridge, and remote bridge write paths read-only; https://github.com/openclaw/openclaw/issues/59613 provides matching Docker logs.

## Solution Assessment

No. The direction is the right narrow fix, but the patch misses remote bridge parity, leaves public docs inconsistent, and lacks after-fix runtime proof for a sandbox permission change.

## Review Findings

Overall correctness: patch is incorrect
