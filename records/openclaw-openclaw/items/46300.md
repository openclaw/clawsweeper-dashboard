## Summary

Keep open: the underlying sandbox write bug is still present on current main and latest release, but this PR is incomplete, conflicting, and missing required real runtime proof for a sandbox write-permission change.

## What This Changes

The PR makes `workspaceAccess: "none"` writable across sandbox mounts, local FS bridge checks, memory flush gating, config hashes, docs, and related tests.

## Best Possible Solution

Land a rebased complete fix or replacement that makes only `ro` read-only across Docker/browser mounts, local and remote FS bridges, memory/docs/tests, while preserving host workspace isolation and including redacted real sandbox proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `none` selects an isolated sandbox workspace, but Docker/browser mounts plus local and remote bridge write checks still treat only `rw` as writable; https://github.com/openclaw/openclaw/issues/59613 also preserves matching Docker logs and inspect output.

## Solution Assessment

No. The direction is the right narrow fix, but this branch misses remote bridge parity, leaves public docs inconsistent, lacks required after-fix runtime proof, and needs a rebase/replacement against current mount-format code.
