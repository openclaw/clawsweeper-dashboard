## Summary

Keep open: the PR targets a real source-reproducible sandbox bug, but the current patch is still incomplete for remote sandbox writes, leaves public docs inconsistent, and lacks required after-fix real behavior proof.

## What This Changes

The PR changes sandbox mount, FS bridge, memory-flush behavior, docs, and tests so `workspaceAccess: "none"` is treated as writable instead of read-only.

## Best Possible Solution

Land a rebased complete fix or replacement that treats only `ro` as read-only across Docker/browser mounts, local and remote FS bridges, docs, and regression tests, with redacted Docker and remote sandbox proof.

## Reproduction Assessment

Yes, at source level. Current main selects an isolated sandbox workspace for `none`, but Docker/local/remote write paths still mark only `rw` writable; https://github.com/openclaw/openclaw/issues/59613 provides matching Docker logs.

## Solution Assessment

No. The direction is the right narrow fix, but this patch misses remote bridge parity, leaves public docs inconsistent, and lacks required after-fix runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

  - body: The patch makes Docker and the local FS bridge treat `none` as writable, but the SSH/OpenShell remote bridge still marks the workspace writable only for `rw` and rejects non-`rw` mutations.   - confidence: 0.91

  - confidence: 0.86
