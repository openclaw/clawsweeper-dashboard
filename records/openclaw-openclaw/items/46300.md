## Summary

Keep open. The branch addresses the earlier config-hash and memory-flush review comments, but it still leaves the SSH/OpenShell remote filesystem bridge in the old `rw`-only write path and misses required public docs/changelog coverage.

## What This Changes

The PR changes sandbox workspace mounts, local filesystem bridge writability, config hashing, memory-flush gating, docs, and tests so `workspaceAccess: "none"` can write inside the isolated sandbox workspace.

## Best Possible Solution

Land a corrected sandbox-permission patch that keeps the host agent workspace hidden for `none`, makes the isolated sandbox workspace writable across Docker, browser, local FS bridge, and SSH/OpenShell remote bridge paths, and aligns docs, changelog, and targeted regression coverage.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction: `workspaceAccess: "none"` selects the isolated sandbox workspace, but Docker mount generation, local FS metadata/write guards, remote write guards, and tests still put it in the read-only bucket; related reports #37634 and #59613 provide matching runtime evidence.

## Solution Assessment

No. The proposed direction is the right narrow fix for Docker and the local FS bridge, but this branch is not yet the best complete solution because remote bridge parity, remaining docs, changelog, and live boundary proof are still missing.
