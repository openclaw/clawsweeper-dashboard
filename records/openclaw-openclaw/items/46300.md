## Summary

Keep open: the requested sandbox bug fix is still needed, but this PR is incomplete because it does not align the SSH/OpenShell remote FS bridge, leaves public docs contradictory, and provides no after-fix real runtime proof.

## What This Changes

The PR changes sandbox mount/write checks, config hashing, memory flush gating, docs, and tests so `workspaceAccess: "none"` is treated as writable instead of read-only.

## Best Possible Solution

Land a corrected fix or replacement that treats only `ro` as read-only for the isolated `none` workspace across Docker, browser, local FS bridge, and remote FS bridge paths, with aligned docs, regression tests, and redacted real runtime proof.

## Reproduction Assessment

Yes, at source level. Current main maps `workspaceAccess: "none"` to an isolated sandbox workspace, but Docker, local bridge, remote bridge, tests, and docs still mark or describe that workspace as read-only; related Docker logs in https://github.com/openclaw/openclaw/issues/59613 match the same failure.

## Solution Assessment

No. The direction is the right narrow fix, but this patch misses remote bridge parity, leaves docs inconsistent, and lacks after-fix runtime proof for a sandbox permission change.
