## Summary

Keep open: the bug remains on current main and this PR is a plausible fix direction, but it is not merge-ready because it misses remote bridge/doc parity, lacks required real behavior proof, and is stale/conflicting against current sandbox mount code.

## What This Changes

The PR changes sandbox `workspaceAccess: "none"` to be writable for workspace mounts, FS bridge metadata, memory flush gating, config hashing, docs, and related tests.

## Best Possible Solution

Land a rebased complete fix or replacement that makes only `ro` read-only across Docker/browser mounts, local and remote FS bridges, memory/docs/tests, while preserving host workspace isolation and including redacted real sandbox proof.

## Reproduction Assessment

Yes, source-reproducible on current main: `none` selects an isolated sandbox workspace, but Docker/browser mounts plus local and remote bridge write checks still treat only `rw` as writable. https://github.com/openclaw/openclaw/issues/59613 also preserves matching Docker logs and inspect output.

## Solution Assessment

No. The direction is the right narrow fix, but this branch misses remote bridge parity, leaves public docs inconsistent, lacks required after-fix runtime proof, and needs a rebase/replacement against current mount-format code.
