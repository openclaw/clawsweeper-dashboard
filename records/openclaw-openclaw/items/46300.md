## Summary

Keep open: current main still has the reported `workspaceAccess: "none"` read-only behavior, and this PR is a plausible fix path, but the patch is not ready because it misses remote FS bridge parity, leaves docs inconsistent, and lacks required real behavior proof.

## What This Changes

The branch treats `workspaceAccess: "none"` as writable in sandbox mount, FS bridge, memory-flush, config-hash, docs, and regression-test paths.

## Best Possible Solution

Land a rebased complete fix or replacement that makes only `ro` read-only across Docker/browser mounts, local and remote FS bridges, memory/docs/tests, with redacted Docker and remote sandbox proof.

## Reproduction Assessment

Yes, source-reproducible. Current main selects an isolated sandbox workspace for `none`, but Docker/browser mounts plus local and remote bridge write checks still mark only `rw` writable; https://github.com/openclaw/openclaw/issues/59613 provides matching Docker logs.

## Solution Assessment

No. The direction is the right narrow fix, but this patch misses remote bridge parity, leaves public docs inconsistent, and lacks required after-fix runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Align the remote sandbox write bridge:** `src/agents/sandbox/fs-paths.ts:68`
  - body: The patch makes Docker and the local FS bridge treat `none` as writable, but the SSH/OpenShell bridge still marks the remote workspace writable only for `rw` and rejects non-`rw` mutations. Remote sandbox writes would still fail in the isolated `none` workspace, so update `remote-fs-bridge.ts` and its coverage with the same `ro`-only rule.
  - confidence: 0.91
- **[P3] Update the remaining sandbox permission docs:** `docs/concepts/memory.md:87`
  - body: The patch updates one memory page, but the compaction reference still says flush skips `none` as read-only and the sandboxing guide still says OpenShell `ro` and `none` restrict writes the same way. Those public docs would contradict the new runtime behavior after merge.
  - confidence: 0.86
