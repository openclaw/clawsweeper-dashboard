## Summary

Keep open: the PR fixes part of a valid current-main sandbox write bug, but it is still incomplete because the SSH/OpenShell remote filesystem bridge remains in the old read-only bucket for `workspaceAccess: "none"`, and the docs/changelog surface is not fully aligned.

## What This Changes

The PR makes `workspaceAccess: "none"` writable for Docker sandbox workspace mounts, local sandbox FS metadata/write guards, memory-flush gating, config-hash recreation, docs, and related tests.

## Best Possible Solution

Land a corrected sandbox-permission fix that keeps the host agent workspace hidden for `none`, makes only the isolated sandbox workspace writable across Docker, browser, local FS bridge, and SSH/OpenShell remote bridge paths, and aligns docs, changelog, and regression coverage.

## Reproduction Assessment

Yes. Source inspection shows current main maps `workspaceAccess: "none"` to an isolated sandbox workspace but still marks Docker, local FS bridge, and remote bridge write paths read-only; related #37634 and #59613 provide matching runtime evidence.

## Solution Assessment

No. The direction is the right narrow fix for Docker/local paths, but the proposed patch is not complete until remote bridge parity, remaining docs, changelog, and targeted validation are added.
