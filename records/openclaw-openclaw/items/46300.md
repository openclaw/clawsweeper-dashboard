## What This Changes

The PR changes sandbox `workspaceAccess: "none"` handling so the isolated sandbox workspace is writable in mounts, FS bridge metadata, memory flush gating, config hash inputs, docs, and tests.

## Best Possible Solution

Land a rebased complete fix or replacement that makes only `ro` read-only across Docker/browser mounts, local and remote FS bridges, memory/docs/tests, while preserving host isolation and including redacted sandbox proof.

## Reproduction Assessment

Yes. Source inspection on current `main` shows `none` selects an isolated sandbox workspace, but Docker/browser mounts plus local and remote bridge write checks still treat only `rw` as writable; related reports include Docker inspect/log evidence of `/workspace:ro,z`.

## Solution Assessment

No. The direction is the right narrow fix, but this branch misses remote bridge parity, leaves public docs inconsistent, is stale against current mount-format code, and lacks required after-fix real behavior proof.
