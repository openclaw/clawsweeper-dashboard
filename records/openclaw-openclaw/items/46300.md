Keep open: this PR targets a valid current-main bug, but it still misses remote OpenShell/SSH bridge parity plus public docs and changelog requirements, so it should be repaired rather than merged as-is.

## What This Changes

The PR makes `workspaceAccess: "none"` writable for isolated sandbox workspace mounts, local filesystem bridge metadata/write guards, memory-flush gating, config-hash recreation, docs, and related tests.

## Best Possible Solution

Land a corrected sandbox-permission fix that makes only the isolated `none` workspace writable across Docker, browser, local FS bridge, and SSH/OpenShell remote bridge paths while preserving host workspace isolation, `ro` read-only behavior, docs, changelog, and regression coverage.

Yes. Source inspection shows current main maps `workspaceAccess: "none"` to an isolated sandbox workspace but still marks Docker, local FS bridge, memory flush, and remote bridge write paths read-only; related #37634 and #59613 provide matching runtime evidence.

## Solution Assessment

No. The PR’s direction is the right narrow fix for Docker/local paths, but it is not the best complete solution until remote bridge parity, remaining docs, changelog, and targeted regression coverage are added.
