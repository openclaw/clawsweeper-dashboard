## Summary

Keep open: the PR addresses a real sandbox write bug and the direction is right, but the proposed patch is incomplete for SSH/OpenShell remote sandbox writes and still needs docs/changelog alignment before merge.

## What This Changes

The PR treats `workspaceAccess: "none"` as writable for sandbox workspace mounts, local FS bridge metadata/write checks, memory-flush gating, and related tests/docs.

## Best Possible Solution

Land a rebased sandbox fix that keeps the host agent workspace hidden for `none` while making only the isolated sandbox workspace writable across Docker, browser, local FS bridge, SSH/OpenShell remote bridge, docs, changelog, and regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows `workspaceAccess: "none"` resolves to an isolated sandbox workspace but Docker, local FS metadata/write guards, memory flush gating, and remote bridge paths still treat it as read-only; related #37634 and #59613 provide matching runtime evidence.

## Solution Assessment

No. The proposed direction is the narrow maintainable fix, but this PR is not complete until remote bridge parity, current-main mount hash/label preservation, docs, changelog, and targeted regression coverage are added.
