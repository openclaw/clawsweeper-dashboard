## Summary

Keep this PR open. The change is directionally useful and matches the reported `workspaceAccess: "none"` sandbox-write problem, but the proposed patch is not complete enough to merge because it leaves the SSH/OpenShell remote FS bridge in the old read-only path, misses remaining public docs, and lacks the required user-facing changelog entry.

## What This Changes

The PR changes sandbox mount generation, local FS bridge writability, sandbox config hashing, memory-flush gating, docs, and tests so `workspaceAccess: "none"` is writable inside the isolated sandbox workspace.

## Best Possible Solution

Land a corrected sandbox-permission patch that keeps the host agent workspace hidden for `none`, makes the isolated sandbox workspace writable across Docker, browser, local FS bridge, and SSH/OpenShell remote bridge paths, aligns all public docs and changelog, and proves the boundary with targeted unit coverage plus Docker and remote sandbox smokes.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction: `workspaceAccess: "none"` selects the isolated sandbox workspace, but Docker mount generation, local FS metadata/write guards, remote write guards, and tests still put it in the read-only bucket; related reports provide matching `/workspace:ro` runtime logs.

## Solution Assessment

No. The proposed direction is the right narrow fix for the local Docker/browser path, but this PR is not yet the best complete solution because the remote bridge, remaining docs, changelog, and live boundary proof are still missing.
