## Summary

Keep open: current main still has the reported `workspaceAccess: "none"` read-only behavior, but this PR is not ready to merge because it misses remote bridge parity, leaves public docs inconsistent, and has no after-fix real sandbox proof.

## What This Changes

The PR changes sandbox mount and FS bridge writability, memory flush gating, config hashing, docs, and tests so `workspaceAccess: "none"` is treated as writable instead of read-only.

## Best Possible Solution

Land a rebased complete fix or replacement that makes only `ro` read-only across Docker/browser mounts, local and remote FS bridges, memory/docs/tests, while preserving host isolation and including redacted real sandbox proof.

## Reproduction Assessment

Yes. Source inspection on current main shows `none` selects an isolated sandbox workspace, but the Docker/browser mount path and both local and remote bridge write checks still treat only `rw` as writable; https://github.com/openclaw/openclaw/issues/59613 also preserves Docker logs and inspect output.

## Solution Assessment

No. The direction is the right narrow fix, but this branch misses remote bridge parity, leaves public docs inconsistent, lacks required after-fix runtime proof, and needs a rebase against current mount-format code.
