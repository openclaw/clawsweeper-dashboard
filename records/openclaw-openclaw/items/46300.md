## Summary

Keep open: current main and the latest release still block `workspaceAccess: "none"` writes, but this branch is incomplete because it misses remote bridge parity, leaves public docs contradictory, and has no required real sandbox proof.

## What This Changes

The PR changes sandbox mount permissions, local filesystem bridge writability, config-hash inputs, memory-flush gating/docs, and focused tests so `workspaceAccess: "none"` is treated as writable instead of read-only.

## Best Possible Solution

Land a rebased complete fix or replacement that makes only `ro` read-only across Docker/browser mounts and local/remote bridges, updates sandbox/memory docs and tests, preserves host isolation, and includes redacted runtime proof.

## Reproduction Assessment

Yes, source-level. Current main maps `none` to an isolated sandbox workspace, but the Docker mount helper and local/remote write guards still allow writes only for `rw`; the canonical open issue includes user mount/log evidence.

## Solution Assessment

No, not this branch yet. The direction is the narrow maintainable fix, but the branch misses remote bridge parity, leaves docs inconsistent, and lacks required after-fix real behavior proof.
