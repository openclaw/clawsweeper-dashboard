## Summary

Keep open: the PR targets a real Feishu Bedrock schema bug, but the current patch still emits explicit `additionalProperties: true` in provider-visible object branches, which is not compatible with Bedrock strict schema validation. The open sibling https://github.com/openclaw/openclaw/pull/94990 is related but not a safe supersession target while its proof gate is failing and its ClawSweeper review did not complete.

## What This Changes

The PR replaces three Feishu bitable `Type.Any()` record value schemas with a shared union schema and adds tests for non-empty `patternProperties` values plus object-array field values.

## Best Possible Solution

Use one Feishu-local flexible JSON value schema that is non-empty, preserves object-array bitable values, avoids Bedrock-unsupported schema keywords, and has registered-tool regression coverage before landing a single candidate for https://github.com/openclaw/openclaw/issues/94547.

## Reproduction Assessment

Yes, source-level: current main and v2026.6.8 still emit the three `Type.Any()` record value schemas, and TypeBox 1.1.39 places that empty value under `patternProperties`. A live Bedrock request was not run in this read-only review.

## Solution Assessment

No. The PR fixes the empty sub-schema shape, but its explicit `additionalProperties: true` object branches conflict with Bedrock's documented strict schema subset; the safer fix is a non-empty provider-compatible JSON-value schema with proof. ([docs.aws.amazon.com](https://docs.aws.amazon.com/bedrock/latest/userguide/structured-output.html))
