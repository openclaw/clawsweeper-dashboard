## Summary

Keep open. Current main improved terminal and no-task ACP cleanup, but active ACP tasks still treat a persisted ACP session row as authoritative backing state, so the stale-row zombie class described in the discussion remains unresolved; related durable-queue PR #68718 is still open and unmerged.

## What This Changes

The issue reports that embedded-run or ACP child sessions can persist after parent termination, leaving stale task/session state for later runs.

## Best Possible Solution

Choose and land either a narrow ACP liveness repair or the broader durable queue path so a stale parent-owned ACP row cannot keep an active task alive indefinitely after parent abort, timeout, or gateway restart.

## Reproduction Assessment

Yes, from source inspection. A focused unit reproduction can seed a grace-expired running ACP task with an existing parent-owned ACP child session entry; current maintenance treats that row as backing state and skips orphan cleanup while the task is active.

## Solution Assessment

No. Current main's missing-row and terminal/orphaned-row cleanup is useful but not sufficient for the reported active stale-row leak; the maintainable fix needs an explicit ACP liveness signal such as a lease, heartbeat, runtime owner identity, or process-aware reconciliation.
