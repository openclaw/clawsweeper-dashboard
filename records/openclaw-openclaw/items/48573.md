## Summary

Keep open. Current main still only has partial cleanup: native subagent stale-run recovery is documented and recently tightened, but ACP task maintenance still treats any persisted ACP child session row as authoritative backing state, so the still-present stale-row zombie class described in the discussion remains plausible on current main.

## What This Changes

The issue requests cleanup or reconciliation for embedded-run child agents and persisted session state when a parent agent terminates unexpectedly.

## Best Possible Solution

Add a merged, process-aware or lease/heartbeat-based reconciliation path for ACP parent-owned child sessions, while keeping the existing native subagent recovery bounds. The fix should make a still-present but orphaned ACP session row stop keeping an active task alive indefinitely, close or mark the ACP task terminal after the grace period, and include regression coverage for parent abort, timeout, and gateway restart/crash cases.

## Reproduction Assessment

Yes. A focused unit reproduction can seed a stale running ACP task past the 5-minute grace period plus a still-present ACP child session entry, then call `runTaskRegistryMaintenance`; current `hasBackingSession` returns true for that entry, so the task is not marked lost.

## Solution Assessment

No. The current missing-row-only maintenance is not sufficient for the reported zombie state; the maintainable fix needs an explicit ACP liveness contract such as a lease/heartbeat, runtime owner identity, or process-aware reconciliation rather than treating persisted metadata existence as proof of life.
