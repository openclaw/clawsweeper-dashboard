## What This Changes

The issue reports that embedded-run or ACP child session state can persist after parent termination, leaving stale/zombie execution context in the session store.

## Best Possible Solution

Land a narrow ACP task-maintenance fix that uses canonical session/transcript liveness, marks truly stale child rows lost, and preserves legitimate persistent or bound ACP sessions.

## Reproduction Assessment

Yes, source-level. A grace-expired running ACP task whose childSessionKey still has any ACP session row passes hasBackingSession(), so shouldMarkLost() returns false and the stale task remains active.

## Solution Assessment

No, current main is incomplete. The best path is to finish the focused PR direction with canonical path/liveness handling and proof that stale ACP rows are reconciled without breaking live sessions.
