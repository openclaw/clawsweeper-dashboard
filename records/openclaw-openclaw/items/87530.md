## Summary

Keep this PR open, but it is not merge-ready: the account-scoped voice group direction matches the dependency contract, while the new delayed autoJoin retries can disrupt already-established Discord voice sessions and the PR lacks real Discord proof.

## What This Changes

The PR scopes Discord voice connections by account ID and adds delayed autoJoin retries after Discord READY/RESUMED and provider startup.

## Best Possible Solution

Keep the account-scoped group change, but make autoJoin retries idempotent or failed-entry scoped, lifecycle-owned, covered by focused tests, and backed by redacted real Discord proof before merge.

## Reproduction Assessment

Yes for the PR-introduced regression: with `voice.autoJoin` configured and the first join succeeding, the scheduled 8s/75s retries call `autoJoin()` again and `join()` stops the existing guild session. The original multi-bot collision has dependency-contract support, but no real Discord proof in the PR.

## Solution Assessment

No. Account-scoped `@discordjs/voice` groups are the right narrow direction, but the delayed retry path needs idempotency, lifecycle cleanup, tests, and live proof before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Make the delayed autoJoin retries idempotent:** `extensions/discord/src/monitor/provider.ts:543-544`
  - body: These two startup timers call `voiceManager.autoJoin()` even after the READY-delayed autoJoin has already succeeded. `autoJoin()` calls `join()` for each configured guild, and `join()` stops/deletes the existing session before reconnecting, so normal startup can rebuild an active voice session at 8s and again at 75s. Retry only failed/transient entries or skip when the manager is already in the configured channel.
  - confidence: 0.92
- **[P2] Tie delayed voice timers to provider shutdown:** `extensions/discord/src/monitor/provider.ts:526-541`
  - body: `scheduleVoiceAutoJoin` creates ordinary ref'd timers and never stores, clears, or unrefs them. If the Discord provider starts with voice enabled and then aborts before the 75s retry fires, the pending timer can keep the Node process or test worker alive even though lifecycle cleanup already destroyed the voice manager.
  - confidence: 0.82
