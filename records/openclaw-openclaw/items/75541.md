## Summary

Keep open. Current main still accepts a truthy context-engine assemble result with non-array `messages`, can assign `undefined` into active session state, and later reads `activeSession.messages.length` during prompt assembly.

## What This Changes

The issue requests hardening prompt assembly so enabling `@martian-engineering/lossless-claw` cannot crash model calls when its context-engine assemble result omits `messages`.

## Best Possible Solution

Add a shared runtime guard for context-engine assemble results, warn on malformed output, and fall back to sanitized pipeline messages without mutating session state.

## Reproduction Assessment

Yes. A focused fake context engine returning a truthy assemble result with `messages: undefined` follows the current code path into `activeSession.agent.state.messages = undefined` and then `activeSession.messages.length`.

## Solution Assessment

Yes. The narrowest maintainable fix is a generic context-engine result validator at the shared assemble boundary or immediate runner call site, not a lossless-claw-specific branch.

## Repair Work Prompt

Fix the context-engine assemble hardening bug reported in https://github.com/openclaw/openclaw/issues/75541. Current main lets a malformed `ContextEngine.assemble()` result with `messages: undefined` poison `activeSession.agent.state.messages`, causing a later `activeSession.messages.length` crash before provider calls. Add a generic runtime guard at the shared assemble boundary or immediate embedded-runner call site: accept only array-valued `messages`, preserve/fall back to pipeline messages on malformed output, warn diagnostically, and avoid mutating session state. Do not add a lossless-claw-specific branch, bundled plugin id, external dependency, or startup migration. Add focused regression coverage with an in-repo fake context engine; include the adjacent context-engine tests if the helper contract changes.
