## Summary

Keep open. Current main still validates the resolved context-engine object, but it does not validate the runtime `assemble()` result before assigning `assembled.messages` into the active session, so a plugin returning `messages: undefined` can still poison prompt assembly and produce the reported `.length` crash.

## What This Changes

The issue requests hardening the prompt assembly path so enabling `lossless-claw` cannot make every model call fail with `Cannot read properties of undefined (reading 'length')` when a context engine returns malformed assembly output.

## Best Possible Solution

Add a focused runtime validation boundary for context-engine assembly results, preferably in the shared harness helper or immediately at the embedded runner call site. Malformed results should be rejected with a clear warning and the runner should continue with the already sanitized pipeline messages instead of mutating `activeSession.messages`.

## Reproduction Assessment

Yes for the core boundary: a focused regression can mock a context engine whose `assemble()` returns a truthy object with `messages: undefined`, then verify current main assigns that value and later reaches `.length` reads. The exact reporter database is unavailable, so the live lossless-claw reproduction remains indirect.

## Solution Assessment

Yes. A narrow guard around `ContextEngine.assemble()` output is the best maintainable fix because OpenClaw should defend the plugin boundary even if the external engine also needs its own bug fix.
