## Summary

Keep open. Current main still trusts a truthy context-engine assemble result with non-array `messages`, so a malformed plugin result can replace session history with `undefined` and crash the later prompt diagnostics read.

## What This Changes

The issue reports prompt assembly failures when lossless-claw is enabled and returns malformed context-engine messages.

## Best Possible Solution

Validate context-engine assemble results generically before mutating session state; on malformed output, log a redacted warning and continue with the original pipeline messages, with regression coverage for the runner boundary.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a truthy assemble result with `messages: undefined` reaches `activeSession.agent.state.messages`, then prompt diagnostics read `activeSession.messages.length`.

## Solution Assessment

Yes. The best fix is a generic runtime guard at the shared assemble boundary or immediate runner call site, not a lossless-claw-specific branch in core.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75541 by adding a generic runtime guard for context-engine `assemble()` results before prompt assembly mutates active session messages. Treat a truthy result with missing or non-array `messages` as malformed: warn with redacted/diagnostic-safe context, fall back to the original pipeline messages, and do not assign malformed data into `activeSession.agent.state.messages`. Keep the fix generic to context-engine contracts; do not special-case lossless-claw, Telegram, WhatsApp, providers, or model routing. Add focused regression coverage for a fake context engine returning `messages: undefined`, preferably at the runner boundary and/or shared harness helper. Update `CHANGELOG.md` because this is a user-facing crash fix.
