## Summary

Keep open. Current main still trusts a truthy context-engine assemble result, so `messages: undefined` can overwrite the active session message array and later hit the reported `.length` read before any provider call.

## What This Changes

The issue reports a prompt-stage TypeError when `@martian-engineering/lossless-claw` is enabled because malformed context-engine assembly output can leave session messages undefined.

## Best Possible Solution

Add runtime validation at the shared context-engine assemble boundary, warn on malformed results, and fall back to the existing pipeline messages without mutating session state.

## Reproduction Assessment

Yes. A focused regression can use a fake `ContextEngine` whose `assemble()` returns a truthy result with `messages: undefined`; current main would assign it before reading `activeSession.messages.length`.

## Solution Assessment

Yes. The narrowest maintainable fix is a shared runtime guard on external plugin return shape with fake-engine tests, rather than depending on the external lossless-claw package or adding plugin-specific core policy.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75541 by hardening context-engine assemble result handling. Current main forwards raw `ContextEngine.assemble()` output and `run/attempt.ts` assigns `assembled.messages` without checking it is an array, so a plugin can return a truthy malformed result with `messages: undefined` and crash later at `activeSession.messages.length` before provider submission. Add runtime validation at the shared assemble boundary, preferably `src/agents/harness/context-engine-lifecycle.ts`, or the immediate runner call site if that keeps the fallback behavior clearer. On malformed assemble output, log/warn, keep the existing sanitized pipeline messages, and do not mutate session state. Add focused fake-engine regression coverage; do not import or depend on `@martian-engineering/lossless-claw`, and do not add plugin-specific ids/defaults to core behavior. Add a user-facing `CHANGELOG.md` Fixes entry with `Thanks @tyyim`.
