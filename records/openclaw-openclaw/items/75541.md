## Summary

Keep open. Current main still trusts the active context engine's `assemble()` output after the normal transcript sanitize/validate pass, then assigns `assembled.messages` directly into the Pi session; downstream prompt-build code still reads `.length` from that post-assembly message state. The report's lossless-claw `assemble-debug` timing and provider-independent fallback failures fit a real plugin-boundary hardening gap, not the older #66591 id-mismatch issue that was resolved by lossless-claw 0.9.

## What This Changes

Report that enabling `@martian-engineering/lossless-claw` v0.9.2 causes Telegram and WhatsApp model calls to fail before provider dispatch with a `TypeError` reading `length` during prompt assembly.

## Best Possible Solution

Harden the context-engine boundary so OpenClaw validates and sanitizes `assemble()` results before mutating session state, falls back to the pre-assembly validated messages with an actionable warning when a plugin returns malformed data, and keeps the broader prompt-authority discussion in #74233 separate.

## Reproduction Assessment

Unclear for the exact live report because the reporter's LCM database/session state is not available. A high-confidence focused reproduction is clear: mock a context engine whose `assemble()` returns a malformed message/result after the normal sanitize/validate pass and assert current main can reach downstream `.length` reads instead of falling back safely.

## Solution Assessment

Yes, the best narrow fix is a core boundary guard around context-engine `assemble()` output, plus a focused regression test. lossless-claw should also avoid producing malformed messages, but OpenClaw should not let a third-party context engine poison prompt assembly for every provider fallback.

## Repair Work Prompt

Fix the context-engine prompt assembly crash reported in https://github.com/openclaw/openclaw/issues/75541. Keep the change narrow: validate the result of active context-engine `assemble()` before assigning it into the session. If the result is missing, `messages` is not an array, or any assembled message has malformed content likely to crash downstream prompt assembly, log/warn and fall back to the already sanitized/validated pipeline messages for that attempt instead of failing the whole provider fallback chain. Prefer a small helper in the context-engine/harness boundary and reuse existing sanitizer/validator helpers where appropriate; do not redesign #74233's prompt-authority behavior and do not edit the external `@martian-engineering/lossless-claw` package. Add focused regression coverage using a fake context engine that returns malformed assembled output after `[lcm] assemble` would have completed. Add a user-facing `CHANGELOG.md` fix entry crediting `Thanks @tyyim`. Validate with targeted tests first; if the touched lanes broaden, use the repo's normal changed-gate policy.
