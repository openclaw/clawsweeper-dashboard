## Summary

Keep open. Current main still trusts a truthy context-engine assemble result with non-array `messages`, which can poison active session state and then crash when prompt diagnostics read `activeSession.messages.length`; this is a narrow core boundary bug, not a lossless-claw-specific product request.

## What This Changes

The issue reports prompt assembly failures when `@martian-engineering/lossless-claw` returns malformed context-engine messages before any provider call is made.

## Best Possible Solution

Add a generic runtime guard for context-engine assemble results, warn on malformed output, and fall back to the original pipeline messages without mutating session state.

## Reproduction Assessment

Yes. Source inspection shows a fake context engine returning a truthy assemble result with `messages: undefined` can flow into `activeSession.agent.state.messages = assembled.messages` and then hit `activeSession.messages.length`.

## Solution Assessment

Yes. The narrowest maintainable fix is a generic context-engine result validator at the shared assemble boundary or immediate runner call site, not a lossless-claw-specific branch.

## Repair Work Prompt

Fix the context-engine assemble boundary for https://github.com/openclaw/openclaw/issues/75541. Current main forwards `ContextEngine.assemble()` output through `assembleHarnessContextEngine()` and the embedded runner assigns `assembled.messages` into session state without verifying it is an array; malformed plugin output can then crash on `activeSession.messages.length`. Add a generic runtime validator/sanitizer at the shared harness boundary or immediate runner call site, warn with sanitized context, and fall back to the original pipeline messages without mutating session state when `messages` is malformed. Cover a fake context engine returning `messages: undefined` plus the well-formed path. Preserve third-party plugin compatibility and do not add lossless-claw-specific branching. Add a one-line changelog fix entry if the change is user-facing.
