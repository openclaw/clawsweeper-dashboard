## Summary

Keep open. Current main still trusts a truthy context-engine assemble result with a non-array or missing `messages` field, so the reported prompt-stage `.length` crash remains plausible before any provider call.

## What This Changes

The issue reports that enabling `@martian-engineering/lossless-claw` 0.9.2 can make all model calls fail during prompt assembly with `Cannot read properties of undefined (reading 'length')`.

## Best Possible Solution

Add runtime validation at the shared context-engine assemble boundary or immediate runner call site, warn on malformed results, and fall back to the original pipeline messages without mutating session state.

## Reproduction Assessment

Yes. A focused regression can use a fake ContextEngine whose `assemble()` returns a truthy object with `messages: undefined`; current main then assigns that value and later reads `activeSession.messages.length`.

## Solution Assessment

Yes. The narrowest maintainable fix is a generic runtime guard for plugin assemble results, with focused tests, rather than adding lossless-claw-specific behavior or changing provider fallback logic.

## Repair Work Prompt

Fix the context-engine assemble-result hardening for https://github.com/openclaw/openclaw/issues/75541. Current main trusts `ContextEngine.assemble()` output: `assembleHarnessContextEngine()` forwards the raw result, and `src/agents/pi-embedded-runner/run/attempt.ts` assigns `assembled.messages` before later reading `activeSession.messages.length`. Add a shared runtime validator at the assemble boundary or immediate runner call site that requires `messages` to be an array and handles malformed truthy results by warning and falling back to the original pipeline messages without mutating session state. Preserve `systemPromptAddition` and `promptAuthority` only from valid assemble results. Add focused regression coverage with a fake context engine returning `messages: undefined` and preferably another non-array value. Do not add core knowledge of lossless-claw, do not change provider fallback behavior, and do not relax the public ContextEngine contract. Add a user-facing `CHANGELOG.md` fix entry if the patch changes shipped behavior.
