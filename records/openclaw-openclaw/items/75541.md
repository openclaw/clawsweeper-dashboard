## Summary

Keep open. Current main still accepts an unchecked runtime `ContextEngine.assemble()` result, so a truthy assembly object with `messages: undefined` can replace session messages and later hit the reported `.length` read during prompt assembly.

## What This Changes

The issue reports a prompt assembly crash when the external `lossless-claw` context engine returns malformed assembly state with `messages` missing or undefined.

## Best Possible Solution

Add runtime validation at the shared context-engine assembly boundary before session or prompt state is mutated, falling back to sanitized pipeline messages with a clear warning when a plugin returns a malformed result.

## Reproduction Assessment

Yes for the core boundary. A focused regression can inject a fake context engine whose `assemble()` returns a truthy object with `messages: undefined`, then verify the runner does not write undefined messages or reach the heartbeat `.length` path.

## Solution Assessment

Yes. The narrowest maintainable fix is to harden `assembleHarnessContextEngine()` or its immediate embedded-runner caller against malformed third-party runtime data while leaving the external plugin responsible for its own contract violation.

## Repair Work Prompt

Fix the prompt assembly crash reported when a context engine returns a malformed `assemble()` result such as `messages: undefined`. Add runtime validation before any runner mutates session or prompt state, preferably in `src/agents/harness/context-engine-lifecycle.ts` or the immediate embedded-runner call site in `src/agents/pi-embedded-runner/run/attempt.ts`. If the result is malformed, preserve or sanitize the existing pipeline messages and emit a clear warning instead of assigning undefined. Add focused tests with a fake context engine returning a truthy malformed result; cover that session messages remain an array and prompt assembly does not hit the heartbeat `.length` path. Do not add bundled plugin-specific logic or depend on the external `lossless-claw` package. Add a user-facing `CHANGELOG.md` fix entry if runtime behavior changes.
