## Summary

Keep open. Current main still trusts a truthy context-engine assemble result and can assign `assembled.messages` into session state without checking that it is an array, so the reported `.length` crash remains a valid core hardening bug.

## What This Changes

Fix a prompt assembly crash when `lossless-claw` returns a malformed context-engine assembly result with missing messages.

## Best Possible Solution

Add generic runtime validation for `ContextEngine.assemble()` results before session mutation, warn and fall back to pipeline messages on malformed output, and keep external plugin-specific fixes outside core.

## Reproduction Assessment

Yes. A focused regression can inject a fake context engine whose `assemble()` returns a truthy object with `messages: undefined`; current main will assign that value and later hit `messages.length` during prompt assembly.

## Solution Assessment

Yes. The narrowest maintainable fix is a runtime guard at `assembleHarnessContextEngine()` or the immediate embedded-runner call site, with regression coverage proving malformed results do not poison session state.

## Repair Work Prompt

Fix openclaw/openclaw#75541. Harden the context-engine assemble boundary so a malformed third-party `ContextEngine.assemble()` result, especially a truthy result with `messages` missing or not an array, cannot replace `activeSession.agent.state.messages` and later crash prompt assembly with `Cannot read properties of undefined (reading 'length')`. Prefer a generic guard in `src/agents/harness/context-engine-lifecycle.ts` or the immediate call site in `src/agents/pi-embedded-runner/run/attempt.ts`; when the result is malformed, log/warn and fall back to the existing pipeline/preassembly messages without mutating session state. Do not add bundled `lossless-claw` dependencies, core plugin ids, or plugin-specific recovery policy. Add focused regression coverage with a fake context engine returning `messages: undefined`, update adjacent context-engine tests if the helper contract changes, and add a single-line user-facing `CHANGELOG.md` fix entry.
