## Summary

Keep open. Current main still validates and repairs the transcript before context-engine assembly, but it does not validate the `assemble()` result before mutating the active session; a malformed result can still poison `activeSession.messages` and crash later prompt-build diagnostics with `.length` reads. The existing issue comment’s analysis is consistent with current source, and the older #66591 fix covered context-engine object resolution, not this post-assembly result boundary.

## What This Changes

The issue reports a provider-independent prompt-stage crash when the `@martian-engineering/lossless-claw` context-engine plugin returns malformed assembled context.

## Best Possible Solution

Add a narrow runtime guard at the context-engine assembly boundary: validate that the result is an object with `messages` as an array, reject or ignore malformed optional fields, warn with actionable context, and keep the pre-assembly sanitized pipeline messages for the prompt attempt. Cover the embedded PI path with a focused regression test and leave broader context-engine prompt-authority behavior to #74233.

## Reproduction Assessment

Unclear for the exact live report because the reporter's lossless-claw store is not available. A high-confidence focused reproduction path exists: mock a context engine whose `assemble()` returns a truthy object with `messages: undefined`, then verify current main can assign it and later hit `.length` reads instead of falling back.

## Solution Assessment

Yes, the best narrow solution is a core boundary guard around context-engine `assemble()` output plus a regression test. Fixing lossless-claw may also be appropriate, but OpenClaw should not allow one plugin’s malformed result to break every provider fallback.

## Repair Work Prompt

Fix openclaw/openclaw#75541. Current main trusts `ContextEngine.assemble()` output after the embedded runner's normal sanitize/validate/limit pass; if a plugin returns a truthy result with `messages` missing or non-array, `activeSession.agent.state.messages` can be poisoned and downstream prompt assembly reads `.length` from undefined. Add a narrow runtime boundary guard around context-engine assembly results, preferably in the shared `src/agents/harness/context-engine-lifecycle.ts` helper or a small helper it calls, so callers only receive valid assembled messages. On malformed output, warn with actionable context and let the embedded PI path keep the pre-assembly validated pipeline messages. Do not redesign context-engine prompt authority, do not change #74233 overflow-precheck semantics, and do not edit the external lossless-claw package. Add focused regression coverage for the embedded PI context-engine path with a mock malformed `assemble()` result, plus any pure helper tests needed. Add a user-facing `CHANGELOG.md` fix entry credited to the reporter, not to bot/maintainer handles.
