## Summary

Keep open. Current main still lacks Linux desktop delivery for the reserved `system.notify` node command: the Linux policy can allow the command, but the headless node host does not advertise or handle it, and docs/CLI still describe notification delivery as macOS/node-app-only.

## What This Changes

Add Linux desktop notification support for the existing `system.notify` node command, likely via `notify-send`/libnotify with graceful headless fallback.

## Best Possible Solution

Add a focused Linux node-host `system.notify` implementation with desktop-session detection, safe argv-only backend invocation, structured unavailable handling, docs/CLI/changelog updates, and focused regression coverage while preserving existing macOS, iOS, Android, Windows, exec, and plugin behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows a Linux node host can identify as Linux while omitting `system.notify` from declared commands, and the dispatcher has no notify handler even if invocation reached it.

## Solution Assessment

Yes. A narrow core node-host change is the best direction because `system.notify` is a reserved core command; broader D-Bus or `dunstify` support can remain follow-up scope if maintainers want it.
