## Summary

Keep open. Current main still does not provide Linux desktop delivery for the reserved `system.notify` node surface: Linux policy allows the command, but the node-host neither advertises nor handles it, and docs/CLI still describe notify as macOS/node-app-only.

## What This Changes

Request Linux desktop notification support for OpenClaw nodes, preferably via `notify-send` or libnotify with graceful headless fallback.

## Best Possible Solution

Add a focused Linux node-host `system.notify` implementation with desktop-session/backend detection, structured unavailable handling, docs/CLI updates, changelog, and regression coverage while leaving macOS, iOS, Android, and exec behavior unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a Linux node-host advertises Linux metadata but does not declare or handle `system.notify`, so `openclaw nodes notify` cannot reach any desktop notification backend; I did not run a live Linux desktop node in this read-only review.

## Solution Assessment

Yes, a narrow core change is the best direction because `system.notify` is a reserved core node command and plugins cannot provide the exact missing surface. Keep the first repair small and argv-safe, with broader fallback backends only if maintainers explicitly choose that scope.
