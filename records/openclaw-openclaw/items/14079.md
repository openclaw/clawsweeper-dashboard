## Label Justifications

- `P2`: Linux desktop notifications are a meaningful but platform-scoped enhancement with limited blast radius and no core-runtime outage.
- `impact:other`: Linux desktop node users cannot receive local system notifications through the established node command, a concrete impact outside the more specific owned categories.

## Summary

Current main and v2026.6.5 still lack Linux delivery for the established `system.notify` surface; the implementation boundary is clear, but maintainers must choose the supported backend, dependency expectation, and capability-advertisement rules before work proceeds.

## What This Changes

Add Linux desktop notification delivery to the existing node `system.notify` command.

## Best Possible Solution

Choose one supported Linux desktop backend and an explicit availability contract, then implement secure core node-host advertisement and dispatch with structured unavailable errors, focused tests, and updated CLI/docs while deferring extra fallbacks until demonstrated necessary.

## Reproduction Assessment

Yes, at source level: current main can issue the reserved `system.notify` request, but the Linux headless node host neither advertises nor dispatches that command. No live Linux desktop invocation was performed during this read-only review.

## Solution Assessment

Not yet: core node-host ownership is the correct boundary, but the proposed notify-send, direct D-Bus, and daemon-specific fallback stack has not been shown to be preferable to one supported backend with explicit unavailable behavior.
