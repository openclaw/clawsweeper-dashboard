## Summary

Keep open. Current main and the latest release still do not provide Linux desktop delivery for the reserved `system.notify` node command: the Linux policy can allow it, but the headless node host neither advertises nor handles it, and the remaining backend/dependency scope needs maintainer direction.

## What This Changes

The issue requests Linux desktop notification support for OpenClaw node notifications, such as `notify-send`/libnotify on Linux desktop sessions.

## Best Possible Solution

Make an explicit maintainer decision on the Linux backend contract, then implement a focused core node-host `system.notify` path with advertisement, safe invocation, clear unavailable results, docs/CLI updates, and targeted regression tests.

## Reproduction Assessment

Yes. By source inspection, a Linux headless node host does not advertise or handle `system.notify`, and Gateway dispatch requires the node to declare the command before invocation; I did not run a live Linux desktop notification scenario.

## Solution Assessment

Unclear until maintainers choose backend scope. The owner boundary should be core node-host code because `system.notify` is a reserved core command and plugins cannot register the exact missing surface.
