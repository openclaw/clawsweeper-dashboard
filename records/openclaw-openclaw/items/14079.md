## Summary

Keep open. Current main and the latest release still do not deliver Linux desktop notifications: Linux is allowlisted for `system.notify`, but the headless node host does not advertise or handle the command, and docs/CLI still describe notification delivery as macOS-only node behavior.

## What This Changes

The issue requests Linux desktop notification support for the existing `system.notify` node surface.

## Best Possible Solution

Add focused Linux headless node-host `system.notify` support with desktop-session/backend detection, structured unavailable handling, docs/CLI/changelog updates, and regression coverage while preserving existing macOS, Android, iOS, Windows, exec, and plugin behavior.

## Reproduction Assessment

Yes, source-level rather than live. Current main shows Linux node hosts resolve as Linux but do not declare or handle `system.notify`, and the Gateway rejects undeclared commands before forwarding.

## Solution Assessment

Yes. A narrow core node-host implementation is the maintainable path because `system.notify` is core-reserved and a plugin cannot register the exact missing command surface; keep backend scope minimal unless maintainers explicitly choose broader fallbacks.

Implement Linux desktop notifications for the existing core `system.notify` node surface. Add a focused Linux headless node-host path that advertises and handles `system.notify` when a desktop session is available (`DISPLAY` or `WAYLAND_DISPLAY`), invokes a supported backend such as `notify-send` with title/body passed as argv fields rather than through a shell, and returns structured unavailable-style errors when no display or backend is present; keep D-Bus, `dunstify`, package-install, and daemon-discovery scope out unless the implementation is still small and covered.
