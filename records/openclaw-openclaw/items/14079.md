## Summary

Keep open. Current main still lacks Linux desktop delivery for the existing `system.notify` node command: Linux is allowlisted at the gateway policy layer, but the headless node host does not advertise or dispatch the command, and the public CLI/docs still describe notify as macOS-node-only.

## What This Changes

Add Linux desktop notification support for node `system.notify`, with graceful behavior on headless Linux hosts.

## Best Possible Solution

Implement a focused core Linux headless node-host `system.notify` path using safe desktop-session/backend detection, structured unavailable results for headless or missing-backend hosts, targeted tests, docs, CLI copy, and changelog coverage.

## Reproduction Assessment

Yes, source-reproducible. A Linux headless node host never declares `system.notify`, and the node-host dispatcher has no `system.notify` branch, so invoking notify on that node cannot reach desktop notification delivery on current main.

## Solution Assessment

Yes. A focused core node-host implementation is the narrow maintainable fix because the exact command is reserved by core and the gateway already has the generic node-invoke policy path.

Implement Linux desktop notification support for the existing core node `system.notify` command. Add a node-host notify helper that, on Linux desktop sessions, invokes a safe backend such as `notify-send` with title/body/priority data passed as argv only; return structured unavailable-style results when no desktop session or backend exists.
