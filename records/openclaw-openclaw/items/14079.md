## Summary

Keep open. Current main still lacks Linux desktop notification delivery for the core `system.notify` node surface: Linux is allowlisted at the Gateway policy layer, but the headless node host does not advertise or dispatch `system.notify`, and the CLI/docs still describe notification delivery as macOS node behavior. The related closed #24616 addressed the misleading rejection message, not Linux notification delivery.

## What This Changes

Request Linux desktop notification support for `system.notify`, preferably using `notify-send` or another desktop notification backend with graceful headless degradation.

## Best Possible Solution

Add focused core Linux headless node-host `system.notify` support with desktop-session/backend detection, safe helper invocation, structured unavailable results, aligned CLI/docs/changelog text, and targeted tests while leaving macOS, iOS, Android, and Windows behavior unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a Linux/headless node host never declares `system.notify`, and the Gateway requires the node to declare a command before forwarding it.

## Solution Assessment

Yes. A narrow core node-host repair is the best direction because `system.notify` is core-reserved and the current plugin command path cannot fill the exact missing command surface.
