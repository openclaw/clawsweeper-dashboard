## Summary

Keep open. Current main still lacks Linux desktop notification delivery for the existing node notification surface: Linux is allowlisted at the gateway policy layer, but the headless node host neither advertises nor handles `system.notify`.

## What This Changes

Add Linux desktop notification support for the existing `openclaw nodes notify` / `system.notify` node surface.

## Best Possible Solution

Implement a focused core Linux headless node-host `system.notify` path with desktop-session detection, a small safe backend scope, updated docs/CLI copy/changelog, and regression coverage for advertisement plus invocation behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence current-main path: Linux node hosts do not declare or dispatch `system.notify`, and gateway `node.invoke` requires the node to declare the command before forwarding.

## Solution Assessment

Yes. Because `system.notify` is a reserved core command, the maintainable fix is a narrow core node-host implementation rather than an external plugin trying to claim the same command.

## Repair Work Prompt

Fix this issue by adding Linux desktop notification support for the core headless node-host `system.notify` command. Keep the scope focused: advertise `system.notify` from Linux-capable headless node hosts only when appropriate, implement safe desktop-session detection using `DISPLAY`/`WAYLAND_DISPLAY`, prefer a small `notify-send` backend first, gracefully return a structured unavailable-style result on headless sessions or missing backend, and pass title/body as argv values without shell interpolation. Update `openclaw nodes notify` copy, nodes/CLI docs, and `CHANGELOG.md`. Add regression tests for node-host command advertisement and invoke behavior, plus any gateway/CLI coverage needed. Do not move this to a plugin, do not change macOS/iOS/Android behavior, and stop for maintainer review if the implementation needs broad D-Bus or `dunstify` fallback design.
