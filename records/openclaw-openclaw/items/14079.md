## Summary

Keep open. Current main still lacks Linux desktop notification delivery for the core `system.notify` node surface: Linux policy allowlists the command, but the headless node host neither advertises nor dispatches it, while docs and CLI still scope notify behavior to macOS/headless exclusion.

## What This Changes

Request Linux desktop notification support for the core `system.notify` / `openclaw nodes notify` path, with `notify-send` or libnotify-style delivery and graceful headless degradation.

## Best Possible Solution

Add a focused Linux headless node-host `system.notify` backend with safe argv-based delivery, desktop-session/backend detection, docs/CLI/changelog updates, and regression coverage for advertisement plus invocation behavior.

## Reproduction Assessment

Yes. Source inspection on current main is high-confidence: Linux allowlists `system.notify`, but the headless node host does not declare or dispatch it, so `node.invoke` rejects the command before a Linux notification backend can run.

## Solution Assessment

Yes. Because `system.notify` is a reserved core node-host command, the maintainable fix belongs in the core node-host path rather than an external plugin claiming the same command.
