## Summary

Keep open. Current main still lacks Linux desktop notification delivery for the core `system.notify` node surface: Linux policy allowlists the command, but the headless node host does not advertise or dispatch it, and the related closed report only improved the rejection error.

## What This Changes

Request Linux desktop notification support for the existing `openclaw nodes notify` / `system.notify` node surface, likely through `notify-send` or a Linux notification backend.

## Best Possible Solution

Add a focused Linux headless node-host `system.notify` backend with safe argv-based delivery, desktop-session/backend detection, docs/CLI/changelog updates, and regression coverage while leaving macOS, iOS, and Android behavior unchanged.

## Reproduction Assessment

Yes. Source inspection on current main is high-confidence: Linux allowlists `system.notify`, but the headless node host does not declare or dispatch it, so `node.invoke` rejects the command before a Linux notification backend can run.

## Solution Assessment

Yes. Because `system.notify` is a reserved core node-host command and already has a CLI surface, the maintainable solution belongs in the core node-host path rather than an external plugin claiming the same command.
