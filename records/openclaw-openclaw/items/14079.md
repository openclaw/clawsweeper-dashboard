## Summary

Current main still does not deliver Linux desktop notifications for the existing `system.notify` node command. The gateway policy allowlists the command on Linux, but the headless node host never advertises it and the dispatcher has no handler for it, so this remains valid rather than stale or superseded by the narrower error-message issue.

## What This Changes

Add Linux desktop notification delivery for the existing `system.notify` node command, likely using `notify-send` with graceful headless degradation.

## Best Possible Solution

Implement a focused core Linux headless node-host `system.notify` path with desktop-session/backend detection, structured unavailable or not-authorized failures, updated CLI/docs/changelog, and tests for advertisement plus invocation behavior while leaving macOS, iOS, and Android notify behavior unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: connect a Linux headless node host and invoke `openclaw nodes notify`; the gateway can allow `system.notify`, but the node does not declare or handle it, so delivery cannot occur.

## Solution Assessment

Yes. The narrowest maintainable solution is a core node-host implementation because `system.notify` is already a reserved core command and the current plugin surface cannot fill that exact command on Linux.

Add a focused headless node-host implementation that advertises `system.notify` on Linux only when appropriate, invokes a safe backend such as `notify-send` without shell interpolation, skips or returns a structured unavailable/not-authorized result when no desktop session or backend is available, and preserves current macOS/iOS/Android behavior.
