## Summary

Keep open. Current main and the latest release still have the Linux notification gap: Linux is allowlisted for `system.notify`, but the headless node host does not advertise or handle the command, while CLI/docs still describe notification delivery as macOS-node behavior. The closed related report at https://github.com/openclaw/openclaw/issues/24616 covered the rejection-message experience, not Linux notification delivery.

## What This Changes

Add Linux desktop notification support for the existing `system.notify` node command, likely through the headless node host with desktop-session detection and a Linux notification backend.

## Best Possible Solution

Add focused Linux headless node-host `system.notify` support with desktop-session/backend detection, structured unavailable handling, docs/CLI/changelog updates, and regression coverage while preserving existing macOS, Android, iOS, Windows, exec, and plugin behavior.

## Reproduction Assessment

Yes, source-level rather than live. Current main shows Linux node hosts resolve as Linux but do not declare or handle `system.notify`, and the Gateway rejects undeclared commands before forwarding.

## Solution Assessment

Yes. A narrow core node-host implementation is the maintainable direction because `system.notify` is reserved by core and a plugin cannot register the exact missing command surface.
