## Summary

Keep open. Current main still does not deliver Linux desktop notifications for the core `system.notify` node surface: Linux is allowlisted at the Gateway policy layer, but the headless node host neither advertises nor dispatches `system.notify`, and current CLI/docs still describe notification delivery as macOS node behavior. The related closed #24616 addressed the rejection message, not Linux notification delivery.

## What This Changes

Add Linux desktop notification support for OpenClaw node notifications, preferably through `notify-send` with graceful fallback or skip behavior for unsupported/headless environments.

## Best Possible Solution

Add a focused core Linux headless node-host `system.notify` implementation with desktop-session/backend detection, safe helper invocation, structured unavailable results for headless or missing-backend cases, aligned CLI/docs/changelog text, and targeted tests while leaving macOS, iOS, Android, and Windows behavior unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a Linux headless node host never declares `system.notify`, and the Gateway requires declared commands before forwarding, so `openclaw nodes notify` cannot reach a Linux notification backend.

## Solution Assessment

Yes. A narrow core node-host repair is the best direction because the exact `system.notify` command is core-reserved and current plugin policy hooks cannot bypass the missing node declaration for that command.
