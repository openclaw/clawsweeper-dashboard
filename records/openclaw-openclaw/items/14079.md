## Summary

Keep open. Current main and the latest release still do not implement Linux desktop notification delivery for the existing `system.notify` node surface; the request is valid, but backend scope and command-advertisement semantics still need maintainer direction before automation should queue a fix.

## What This Changes

Request Linux desktop notification delivery for `openclaw nodes notify` / `system.notify`, with graceful handling for headless Linux hosts and missing notification backends.

## Best Possible Solution

Choose a narrow Linux notification contract, then implement it in the core headless node host with focused tests and docs while preserving existing macOS, iOS, Android, and Windows behavior.

## Reproduction Assessment

Yes, at source level. Current main shows the Linux-capable headless node host neither advertises nor dispatches `system.notify`, and the gateway requires a live node declaration before invoking the command.

## Solution Assessment

No for the full fallback stack as written. The maintainable path is a maintainer-chosen Linux backend and explicit availability semantics, rather than adding every possible fallback by default.
