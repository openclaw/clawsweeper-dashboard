## Summary

Keep open. Current main still lacks Linux desktop notification delivery for the core `system.notify` node surface: Linux policy allowlists the command, but the headless node host neither advertises nor dispatches it, and docs/CLI still scope notify away from Linux headless nodes.

## What This Changes

The issue requests Linux desktop notification delivery for OpenClaw nodes, primarily via notify-send/libnotify with safe headless degradation.

## Best Possible Solution

Add a focused Linux headless node-host `system.notify` backend with safe argv delivery, desktop-session detection, structured unavailable behavior, docs/CLI/changelog updates, and regression coverage while preserving existing macOS, iOS, Android, and Windows behavior.

## Reproduction Assessment

Yes. Source inspection on current main is high-confidence: Linux allowlists `system.notify`, but the headless node host does not declare or dispatch it, so `node.invoke` rejects the command before any Linux notification backend can run.

## Solution Assessment

Yes, with scope control. Because `system.notify` is a reserved core node-host command with an existing CLI surface, the maintainable solution belongs in the core node-host path rather than an external plugin; a first PR should keep fallback behavior narrow and safe.
