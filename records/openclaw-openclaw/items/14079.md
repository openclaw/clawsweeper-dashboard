## Summary

Keep open. Current main and the latest release still do not deliver Linux desktop notifications for the reserved core `system.notify` node command: Linux is policy-allowlisted, but the headless node host neither advertises nor handles the command, and docs/CLI still scope notification delivery away from Linux headless hosts.

## What This Changes

The issue requests Linux desktop notification support for the existing OpenClaw node notification command surface.

## Best Possible Solution

Add focused Linux headless node-host `system.notify` support with desktop-session/backend detection, structured unavailable handling, docs/CLI/changelog updates, and regression coverage while preserving existing macOS, Android, iOS, Windows, exec, and plugin behavior.

## Reproduction Assessment

Yes, source-level rather than live. Current main shows Linux node hosts resolve as Linux but do not declare or handle `system.notify`, and the gateway rejects undeclared commands before forwarding.

## Solution Assessment

Yes. A narrow core node-host implementation is the maintainable direction because `system.notify` is reserved by core and a plugin cannot fill the exact missing command surface.

## Repair Work Prompt

Add Linux desktop notification support for the existing core node notification command. Implement a focused headless node-host `system.notify` path that advertises and handles the command for Linux desktop sessions, detects `DISPLAY`/`WAYLAND_DISPLAY`, invokes a Linux notification backend without shell interpolation, returns structured unavailable/error results for headless or missing-backend cases, and preserves current macOS/iOS/Android/Windows, `system.run`, `system.which`, exec approvals, and plugin node-host behavior. Update CLI/docs/changelog wording and add regression coverage for advertisement plus invocation behavior. Do not move this to a plugin or allow plugins to register the reserved `system.notify` command unless maintainers explicitly choose a new API direction.
