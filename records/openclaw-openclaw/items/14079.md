## Summary

Keep open. Current main still lacks Linux desktop delivery for the reserved `system.notify` node surface: the Linux policy can allow the command, but the headless node host does not advertise or handle it, and docs/CLI still present notification delivery as macOS/node-app-only. The related closed item at https://github.com/openclaw/openclaw/issues/24616 improved the rejection message; it did not add Linux notification delivery.

## What This Changes

Request Linux desktop notification delivery for the existing `system.notify` / `openclaw nodes notify` path, preferably via `notify-send` with graceful headless degradation.

## Best Possible Solution

Add a focused Linux node-host `system.notify` implementation with desktop-session/backend detection, structured unavailable handling, CLI/docs/changelog updates, and regression coverage while preserving existing macOS, iOS, Android, Windows, and exec behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows a Linux node host can identify as Linux while omitting `system.notify` from its declared commands, and the dispatcher has no notify handler even if invocation reached it.

## Solution Assessment

Yes. A narrow core node-host change is the best direction because `system.notify` is a reserved core command; broader D-Bus or `dunstify` support can be follow-up scope if maintainers want it.
