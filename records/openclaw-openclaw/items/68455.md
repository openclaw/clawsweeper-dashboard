## Summary

Keep this PR open, but do not merge it as-is: the Feishu multi-account onboarding goal is still useful, while the branch still has accountId plumbing defects, no real Feishu setup proof, and mergeability needs a refresh.

## What This Changes

The PR changes the Feishu plugin setup adapter and wizard to normalize custom account IDs, list existing Feishu accounts, and prompt for a new account ID during onboarding.

## Best Possible Solution

Keep the Feishu multi-account onboarding direction, but repair the wizard so the selected accountId is the single target for credential writes and account enablement, then prove fresh first-account, second-account, and reconfigure flows before merge.

## Reproduction Assessment

Yes for the PR defects by source inspection: the setup framework resolves accountId before finalize, but the PR branch does not pass it into runNewAppFlow/runEditFlow, and listFeishuAccountIds falls back to default for empty configs. I did not run a live Feishu onboarding flow, so this is source-reproducible rather than runtime-reproduced.

## Solution Assessment

No: prompting for custom account IDs is plausible, but this implementation is incomplete until the selected ID drives credential writes and first-run default behavior remains intact. The safer shape is to consume the setup framework's resolved accountId throughout the Feishu new/edit flows and cover that with focused tests plus real setup proof.
