## Summary

Keep open: the Feishu multi-account onboarding direction is useful, but the PR still drops the selected account before credential writes, regresses first-run/default-account behavior, and has no real Feishu onboarding proof.

## What This Changes

The PR modifies Feishu setup to prompt for custom account IDs, exports a Feishu account helper, and changes the Feishu setup adapter's account resolution.

## Best Possible Solution

Thread the resolved account ID through Feishu finalization, edit, and new-app writes; prompt only after real Feishu config exists; preserve defaultAccount resolution; add regression tests; and collect redacted real onboarding proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the resolver can return a custom accountId, but `finalize` ignores it and `runNewAppFlow` still writes through the resolved default account. I did not run live Feishu onboarding because this review is read-only and no live credentials or proof were provided.

## Solution Assessment

No. Prompting for a custom Feishu account ID fits the existing multi-account model, but this implementation is not the best fix until it carries the selected account into writes and preserves first-run/default-account semantics.
