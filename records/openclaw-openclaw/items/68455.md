## Summary

Keep open: the Feishu multi-account onboarding problem is still legitimate, but this branch is not merge-ready because the selected account ID is not propagated into the Feishu config-writing flow, fresh setup gets a misleading account prompt, the setup resolver drops configured-default behavior, and no real Feishu proof is provided.

## What This Changes

The PR changes Feishu setup helpers so onboarding can prompt for and normalize a custom account ID when configuring additional Feishu accounts.

## Best Possible Solution

Thread the resolved accountId through Feishu finalize, runEditFlow, and runNewAppFlow so config writes target the selected account; only prompt for a new ID after real Feishu configuration exists; preserve configured-default behavior; then add focused tests plus redacted real Feishu onboarding proof.

## Solution Assessment

No: the PR is in the right Feishu plugin boundary, but it is only a partial implementation until the resolved accountId is the single target for config writes and existing default-account behavior is preserved.
