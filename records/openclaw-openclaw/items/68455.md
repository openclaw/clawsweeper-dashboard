## Summary

Keep this PR open, but it is not merge-ready: the direction matches Feishu multi-account support, yet the patch drops the selected account ID before config writes, regresses fresh setup prompting, bypasses configured defaultAccount behavior, and still lacks real Feishu onboarding proof.

## What This Changes

The PR adds Feishu onboarding prompts for custom account IDs, exports an account helper, and changes the Feishu setup adapter account resolution.

## Best Possible Solution

Thread the resolved account ID through Feishu finalization, edit, and new-app writes; gate prompting on real configured Feishu accounts; preserve defaultAccount behavior; add regression tests; and collect redacted real onboarding proof.

## Reproduction Assessment

Yes for the PR defects: source inspection shows the PR resolver can return a custom accountId, while finalize ignores it and runNewAppFlow writes through resolveDefaultFeishuAccountId. I did not run live Feishu onboarding because this review is read-only and no live credentials were provided.

## Solution Assessment

No. Prompting for a custom Feishu account ID fits the documented multi-account model, but this implementation is not the best solution until it carries the selected account into writes and preserves default-account semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

  - confidence: 0.91

  - body: Replacing resolveDefaultFeishuAccountId(cfg) with normalizeAccountId(accountId) means setup surfaces that call the adapter without an explicit account no longer honor channels.feishu.defaultAccount. They resolve to the literal default account and can mark or write the wrong account.
  - confidence: 0.86
