## Summary

Keep open: the Feishu multi-account onboarding goal is valid, but the patch is not merge-ready because the selected account ID is not propagated into Feishu config writes, fresh setup can treat the fallback default as an existing account, configured defaultAccount fallback can regress, the branch conflicts with main, and no real Feishu proof is present.

## What This Changes

The PR changes Feishu setup account resolution so onboarding can prompt for a custom account ID when adding more than one Feishu account.

## Best Possible Solution

Keep this PR open, but require a rebase plus a focused repair that threads the selected accountId through Feishu finalize/edit/new-app config writes, gates the new prompt on real configured accounts, preserves defaultAccount fallback, adds regression coverage, and includes redacted real Feishu onboarding proof.

## Reproduction Assessment

Yes for source-level reproduction: the PR head resolver can return a new custom account ID, but Feishu finalize/new/edit flows still derive their write target from config, and the shared account-list helper returns the fallback default for empty config. I did not run a live Feishu onboarding flow because this read-only review has no real Feishu credentials/proof setup.

## Solution Assessment

No: the PR is in the right Feishu plugin boundary, but it is only a partial implementation until the selected accountId is the single target for credential/domain/policy writes and existing default-account behavior is preserved.

- **[P1] Propagate the selected account ID into Feishu config writes:** `extensions/feishu/src/setup-surface.ts:585-591`

  - body: `listFeishuAccountIds(cfg)` uses the shared helper that returns `default` when no accounts exist. This makes `existingAccounts.length === 0` unreachable and causes a fresh setup to ask about an existing default account that is only a fallback, not a configured Feishu bot.
  - confidence: 0.93
- **[P2] Preserve configured default accounts in the setup resolver:** `extensions/feishu/src/setup-core.ts:34-35`
  - body: Current main falls back to `resolveDefaultFeishuAccountId(cfg)` when `--account` is omitted, including `channels.feishu.defaultAccount`. Replacing that with `normalizeAccountId(accountId)` makes an omitted account target the literal `default` account instead of the configured default in existing multi-account configs.
  - confidence: 0.87
