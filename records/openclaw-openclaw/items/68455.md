## Summary

Keep open, but this PR is not merge-ready: the Feishu multi-account onboarding goal is still valid, yet the patch does not carry the selected account id into the credential-writing flows, can show a misleading first-run account prompt, is currently conflicting, and has no after-fix real behavior proof.

## What This Changes

The PR changes Feishu setup account resolution to prompt for a custom account id during onboarding and normalizes the setup adapter account resolver.

## Best Possible Solution

Thread the resolved setup accountId through Feishu finalize, runEditFlow, and runNewAppFlow, prompt only after a real Feishu configuration exists, add focused setup regression tests, and provide redacted real Feishu onboarding proof.

## Reproduction Assessment

Yes, at source level: the setup framework passes a selected accountId into finalize, but this PR's Feishu finalize/new/edit paths still re-derive the write target, and the account-list helper falls back to `default` on empty config. I did not run a live Feishu onboarding flow in this read-only review.

## Solution Assessment

No. The PR is in the right Feishu plugin boundary, but the best fix is to consume the resolved accountId throughout the existing Feishu setup flows rather than adding a prompt whose selected value is not honored by credential writes.
