## Summary

Keep open: this PR addresses a real Feishu multi-account onboarding gap, but the branch still drops the selected account ID before config writes, regresses first-run/defaultAccount behavior, and lacks real Feishu onboarding proof.

## What This Changes

The PR adds custom Feishu account ID prompting during setup, exports a Feishu account helper, and changes the Feishu setup adapter's account resolution.

## Best Possible Solution

Thread the resolved account ID through Feishu finalization, edit, and new-app writes; prompt only after real Feishu config exists; preserve defaultAccount resolution; add regression tests; and collect redacted real onboarding proof.

## Reproduction Assessment

Yes for the PR defects by source inspection: the PR resolver can return a custom accountId, but finalize ignores it and runNewAppFlow still writes through the resolved default account. I did not run live Feishu onboarding because this review is read-only and no live credentials were provided.

## Solution Assessment

No. Prompting for a custom Feishu account ID fits the existing multi-account model, but this implementation is not the best fix until it carries the selected account into writes and preserves first-run/default-account semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Propagate the selected accountId into Feishu setup writes:** `extensions/feishu/src/setup-surface.ts:643-654`
  - body: `resolveAccountIdForConfigure` can now return a custom account ID, but `finalize` ignores `accountId` and calls `runEditFlow`/`runNewAppFlow` without it. `runNewAppFlow` then resolves its target from the existing config, so choosing a new account can still write credentials and policies to the default or old account instead of the selected account.
  - confidence: 0.94
- **[P2] Gate the new-account prompt on real Feishu configuration:** `extensions/feishu/src/setup-surface.ts:556-560`
  - body: `listFeishuAccountIds(cfg)` includes the fallback `default` account when no Feishu account is configured, so `existingAccounts.length === 0` never represents fresh setup. A fresh onboarding path will ask whether to create another account even though no real Feishu account exists yet.
  - confidence: 0.91
- **[P2] Preserve configured defaultAccount in the adapter:** `extensions/feishu/src/setup-core.ts:34-36`
  - body: Replacing `resolveDefaultFeishuAccountId(cfg)` with `normalizeAccountId(accountId)` means setup surfaces that call the adapter without an explicit account no longer honor `channels.feishu.defaultAccount`. They can resolve to the literal default account and enable or write the wrong account.
  - confidence: 0.86
