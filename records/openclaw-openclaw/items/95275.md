## Summary

Keep open: this is a focused Telegram bug-fix candidate, but it is not merge-ready because the patch guards only the normal fetch classifier while the probe path can still force the misleading fallback, and the updated proof remains synthetic test output rather than real Telegram or socket-allocation proof.

## What This Changes

The PR adds numeric errno collection, an EADDRNOTAVAIL no-fallback guard in Telegram fetch classification, and classifier tests.

## Best Possible Solution

Fix both the direct fetch classifier and probe forced-fallback boundary so EADDRNOTAVAIL never promotes IP rotation while preserving valid remote/family fallback behavior, then add redacted runtime proof or an explicit maintainer proof override.

## Reproduction Assessment

Yes at source level: current main and v2026.6.8 show probeTelegram can still force-promote fallback after any getMe fetch error while the pinned-IP warning remains. I did not run a live socket-exhaustion Telegram scenario.

## Solution Assessment

No: the classifier guard is useful but incomplete because the probe forced-fallback boundary still loses the original local socket allocation error. The safer fix is to carry the original error into the fallback decision and preserve valid retry behavior for other network failures.

Full review comments:

- **[P1] Gate probe fallback on the original error:** `extensions/telegram/src/fetch.ts:477-480`
  - body: This guard only runs when the normal fetch loop classifies the error. `probeTelegram` still catches a failed `getMe`, discards the original error, and calls `forceFallback("probe timeout/network error")`, so EADDRNOTAVAIL during probe can still promote to the pinned-IP attempt and emit the misleading DNS-resolved-IP warning. Pass the caught error into that decision or skip forced fallback for local socket allocation failures.
  - confidence: 0.92
- **[P2] Preserve fallback for other numeric errnos:** `extensions/telegram/src/fetch.ts:431-434`
  - body: Adding every numeric `errno` to `ctx.codes` means a `TypeError("fetch failed")` cause with only a numeric errno no longer satisfies the existing code-less fallback branch, and the retry set contains only string codes. Either map retryable numeric errnos or special-case EADDRNOTAVAIL without turning other numeric-only network failures into fail-fast errors.
  - confidence: 0.78
