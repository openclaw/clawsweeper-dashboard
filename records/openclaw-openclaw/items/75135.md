## Summary

Keep open. Current main still has the stack-losing attachment error path, and this PR is a narrow diagnostics improvement, but one small review cleanup remains because the new chat log label names only parsing even though the same catch also covers staging failures.

## What This Changes

The PR adds stack-preserving gateway error logging and Error name/message response formatting to attachment parsing failure paths in `chat.send` and the `agent` handler, plus test mock logger methods.

## Best Possible Solution

Land the diagnostics patch after broadening the chat log label to cover attachment parse and staging failures, then keep #63432 open for the separate root-cause RangeError fix once the captured stack identifies the recursing frame.

## Reproduction Assessment

Yes for the diagnostic gap: current main directly shows `chat.send` and `agent` responding with `String(err)` from attachment failure catches without logging `err.stack`. The exact iPad/Tailscale RangeError from #63432 remains a separate reproduction target outside this PR.

## Solution Assessment

Yes, with a small refinement: logging the stack at the gateway handler boundary is the narrowest maintainable diagnostics fix while preserving protocol classification. The safer final shape is to rename the chat log label so staging failures are not misreported as parser-only failures.
