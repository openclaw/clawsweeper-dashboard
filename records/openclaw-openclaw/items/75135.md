## Summary

Keep open. Current main still has silent attachment parse/stage catch paths, and this PR is a relevant narrow diagnostics fix with no blocking correctness or security issue; the remaining cleanup is a small chat log-label adjustment because the chat catch also wraps staging.

## What This Changes

The PR adds gateway error logging for attachment parsing failures in `chat.send` and `agent`, formats `Error` responses as name/message, and extends the chat directive-tags `logGateway` test mock.

## Best Possible Solution

Land the diagnostics patch after the chat log label accurately covers parse and staging failures, then use the captured stack from #63432 to make a separate root-cause fix if the RangeError still reproduces.

## Reproduction Assessment

Yes for the diagnostic gap: current main directly shows `chat.send` and `agent` responding with `String(err)` and not logging the stack. The exact iPad/Tailscale RangeError reproduction from #63432 remains open and is outside this PR’s stated scope.

## Solution Assessment

Yes, logging the full stack at the gateway handler boundary is the narrowest maintainable diagnostics fix while preserving protocol classification. The safer refinement is to name the chat failure as parse/stage because that catch covers both operations.
