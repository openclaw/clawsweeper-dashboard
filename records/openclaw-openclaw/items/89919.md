## Summary

Keep this PR open: current main still sends cron failure notifications for this timeout class, so the central fix is useful, but the branch is not merge-ready because it also suppresses explicitly configured failure destinations and still has only synthetic after-fix proof.

## What This Changes

The PR adds an exact cron watchdog timeout classifier, suppresses matching cron failure notifications in cron state and gateway dispatch, and adds cron/gateway regression tests.

## Best Possible Solution

Keep the exact timeout classification and recorded cron error, but suppress only the unconfigured primary announce fallback while preserving explicit/global failure destinations, then add redacted real Matrix/cron after-fix proof on a refreshed branch.

## Reproduction Assessment

Source-level yes: current main can produce the exact `tool-execution-started` timeout text and gateway dispatch currently notifies on cron errors. The Matrix workflow itself was private, so the real transport path has not been live-reproduced in public proof.

## Solution Assessment

No: the exact classifier is a plausible fix, but suppressing configured failure destinations is broader than the reported noisy primary Matrix fallback. The safer fix is to keep explicit/global failure destinations active and suppress only fallback notifications when no failure destination exists.
