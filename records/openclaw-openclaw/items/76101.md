## Summary

Keep open: current main still documents the Lobster `openclaw.invoke --tool llm-task` pattern without the URL/auth setup that the embedded runner and Gateway auth path require, and the related allowlist issue tracks a different downstream failure mode.

## What This Changes

The issue asks OpenClaw to make documented Lobster-to-`llm-task` invocations work by supplying gateway invoke context, documenting the required URL/auth setup, or replacing the unsupported bridge.

## Best Possible Solution

Document the complete supported `openclaw.invoke` setup now, and only add automatic embedded-runner context later through a deliberately reviewed auth boundary.

## Reproduction Assessment

Yes. The report gives a concrete current-release path, and current main confirms the docs omit the required env/auth while the embedded runner only forwards existing `process.env`.

## Solution Assessment

A docs-first fix is the safest immediate solution; automatic credential injection is not clearly the best fix without a security design because `/tools/invoke` bearer auth is full operator access.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/76101 with a docs-only clarification. Update the Lobster and llm-task docs/READMEs so `openclaw.invoke --tool llm-task` examples state that the embedded runner does not synthesize Gateway invoke context: `OPENCLAW_URL` must point at the Gateway and `OPENCLAW_TOKEN` must be provided when Gateway token/password auth is enabled, typically by explicitly mapping from `OPENCLAW_GATEWAY_TOKEN` only in a trusted local workflow environment. Include troubleshooting for `openclaw.invoke requires --url or OPENCLAW_URL` and `401 Unauthorized`, and link to the public Tools Invoke API docs. Do not change runtime auth semantics, do not auto-inject Gateway secrets into Lobster env, and do not touch the separate #74019 allowlist guard behavior. Changelog is not expected for a docs-only clarification unless repo policy/gates require it.
