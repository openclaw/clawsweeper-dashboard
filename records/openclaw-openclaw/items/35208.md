## Summary

Keep open: current main still lacks owner/human takeover config, per-conversation pause state, and an auto-reply suppression gate; the linked implementation PR is closed unmerged and the closed duplicate points back to this issue as the canonical tracker.

## What This Changes

Request owner/human handoff for DM auto-reply by detecting manual outgoing owner messages and pausing auto-replies per conversation.

## Best Possible Solution

Choose a maintainer-approved channel/API design, likely WhatsApp-first with plugin-owned humanTakeover config, per-account/per-conversation pause state, suppression before dispatch, focused tests, and docs before expanding to other channels.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request; source inspection on current main gives a high-confidence gap because the config, schema, access-control, and dispatch paths lack human handoff state or pause/resume handling.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the public config and resume policy; the request is valid, but adding the exact proposed autoReply.ownerIntercept shape without a channel-owned contract would not be the safest maintainable path.
