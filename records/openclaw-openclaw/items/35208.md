## Label Justifications

- `impact:message-loss`: The issue is about duplicated and conflicting DM delivery when both the human owner and agent reply to the same conversation.

## Summary

Keep open: current main and v2026.6.5 still discard manual owner-sent WhatsApp DMs without recording per-conversation handoff state, so later inbound DMs continue through normal auto-reply dispatch. The linked closed, unmerged pull request is useful prior art, but no newer canonical item or merged implementation owns the remaining product decision.

## What This Changes

Add automatic per-DM human takeover that pauses channel auto-replies after owner-sent messages and resumes by cooldown or explicit action.

## Best Possible Solution

Implement a WhatsApp plugin-owned per-conversation handoff state machine with one narrow canonical config contract, persistent state only if the chosen resume semantics require it, focused regression coverage, documentation, and live Baileys proof; generalize later only through a typed channel seam when sibling transports demonstrate equivalent owner-message semantics.

## Reproduction Assessment

Yes at the source level: a manual non-self WhatsApp `fromMe` DM is dropped without recording state, and the next admitted inbound DM reaches ordinary auto-reply dispatch without a takeover guard. A live WhatsApp run was not performed in this read-only review.

## Solution Assessment

No, the proposed generic gateway and cross-channel implementation is not established as the best solution. The narrowest maintainable first step is WhatsApp plugin-owned behavior around its concrete Baileys `fromMe` signal, with the config and resume policy chosen explicitly before coding.
