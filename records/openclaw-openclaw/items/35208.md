## Label Justifications

- `P2`: Duplicate human and automated replies are a meaningful channel workflow problem, but the affected surface is bounded and the implementation needs product design first.
- `impact:message-loss`: The issue concerns duplicate and conflicting message delivery when both the human owner and agent reply in the same DM.

## Summary

Current main and v2026.6.5 still discard manual outgoing WhatsApp DMs without creating per-conversation handoff state, and the linked implementation closed unmerged; the feature remains useful but needs maintainer decisions on ownership, persistence, resume semantics, and channel scope.

## What This Changes

Add per-DM owner takeover so manual outgoing messages pause automated replies until a configured resume condition.

## Best Possible Solution

Define a WhatsApp-owned handoff contract with explicit SQLite state, owner-versus-bot echo attribution, and clear cooldown or explicit-resume semantics, then add focused inbound/auto-reply tests and WhatsApp documentation before considering a shared cross-channel API.

## Reproduction Assessment

Yes at source level: current main explicitly drops outgoing non-self WhatsApp DMs and has no per-conversation handoff guard before ordinary auto-reply dispatch. No live WhatsApp run was performed during this read-only review.

## Solution Assessment

No, the closed cross-channel in-memory cooldown PR is not the best current solution because persistence, resume semantics, and channel ownership were not settled. A focused WhatsApp-owned design should establish those contracts before broader reuse.
