- `impact:message-loss`: The affected behavior is message delivery suppression: OpenClaw should avoid duplicate/conflicting DM replies once a human owner takes over.

## Summary

Keep open: current main still has WhatsApp self-echo/access-control protections but no per-conversation owner handoff state, and the linked implementation PR is closed unmerged prior art rather than a canonical fix.

## What This Changes

The issue requests an owner takeover/human handoff mode where manual owner replies in a DM pause AI auto-replies for that thread until the chosen resume condition.

## Best Possible Solution

Implement a WhatsApp plugin-owned per-conversation handoff state machine around the Baileys `fromMe` signal, with one canonical config contract, explicit resume semantics, focused tests/docs, and live WhatsApp proof before considering a cross-channel seam.

## Reproduction Assessment

Yes, at the source level: a manual owner-sent WhatsApp DM to a non-self peer is skipped as `fromMe` without recording handoff state, and a later admitted inbound DM reaches normal auto-reply dispatch. I did not run a live WhatsApp account in this read-only review.

## Solution Assessment

No, not as a ready implementation shape: the requested behavior is useful, but the proposed generic config/resume model needs maintainer selection first. The narrowest maintainable path is WhatsApp plugin-owned behavior around its concrete Baileys `fromMe` signal.
