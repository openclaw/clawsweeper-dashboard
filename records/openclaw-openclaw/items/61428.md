## Summary

Keep open. Current main still hardcodes mapped hook agent runs to isolated sessions, and this PR is a useful narrower replacement for #58023, but the patch needs repair before merge because it rewrites the effective session after prefix-policy validation, omits generated config metadata and changelog coverage, and is stale against the current hook request-handler split.

## What This Changes

The PR adds a `sessionTarget?: "main" | "isolated"` hook-mapping option and threads it through config parsing, mapping dispatch, and gateway hook CronJob creation so mapped webhooks can run against an agent's main session.

## Best Possible Solution

Keep the mapping-level-only design, but implement it on current main by resolving the effective hook execution session before prefix validation and replay dedupe, keeping direct `/hooks/agent` isolated, updating generated config metadata/docs/changelog, and adding targeted tests for `main`, `isolated`, policy-prefix, idempotency, and non-default-agent cases.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a mapping with `sessionTarget: "main"` plus `hooks.allowedSessionKeyPrefixes`: the request handler validates the generated or configured hook key, while the PR's dispatch path later replaces it with the agent main-session key.

## Solution Assessment

No. The requested direction is plausible, but this implementation is not the safest maintainable version because the effective session key is rewritten after policy and replay-scope decisions; a shared effective-session resolver used before validation, dedupe, and dispatch would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Validate the effective main-session key before dispatch:** `src/gateway/server-http.ts:793`
  - body: When this mapped action sets `sessionTarget: "main"`, `dispatchAgentHook` replaces the already-validated hook session key with the agent main-session key. That makes `hooks.allowedSessionKeyPrefixes` validate a key that may never be used, which can both reject valid main-target mappings and allow a disallowed `agent:*` main session to run after a `hook:*` key passed validation. Resolve the effective session key before the prefix check and dispatch the same key that was validated.
  - confidence: 0.89
- **[P2] Include sessionTarget in hook replay scope:** `src/gateway/server-http.ts:793`
  - body: Forwarding `sessionTarget` changes where the hook run executes, but the idempotency replay fingerprint still omits that field. Two otherwise identical mapped deliveries with the same idempotency key can return the cached run id from an isolated run when the second request should run in the main session, or vice versa. Add `sessionTarget` to the replay scope before caching.
  - confidence: 0.82
- **[P2] Regenerate config metadata for the new mapping key:** `src/config/zod-schema.hooks.ts:47`
  - body: Adding `sessionTarget` to the strict hook mapping schema makes this a public config key, but the generated schema/help/label surfaces are not updated. Schema-driven config tooling will not expose or preserve `hooks.mappings[].sessionTarget`, creating contract drift. Regenerate the config metadata and include the generated files that change.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `src/config/types.hooks.ts:18`
  - body: This is a user-facing `feat` that adds a new hook mapping config option, but the PR does not include a `CHANGELOG.md` entry. Add a single-line entry in the active version section with contributor credit so the shipped config feature is visible in release notes.
  - confidence: 0.78
