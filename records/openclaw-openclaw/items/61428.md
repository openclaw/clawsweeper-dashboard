## Summary

Keep open. Current main still hardcodes hook-mapped agent runs to isolated sessions, and this PR is useful source work, but the patch needs changes before merge because it rewrites the effective session after policy/replay decisions and is stale against the current hook request-handler split.

## What This Changes

The PR adds a mapping-level `sessionTarget?: "main" | "isolated"` option and threads it through hook config, mapping resolution, dispatch payloads, and gateway hook CronJob creation.

## Best Possible Solution

Keep the mapping-level-only design, but resolve the effective execution session before prefix validation, replay dedupe, and dispatch; keep direct `/hooks/agent` isolated; update generated config/docs/changelog; and add focused gateway hook tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a mapping with `sessionTarget: "main"` plus `hooks.allowedSessionKeyPrefixes`: the PR validates the hook/default key in the request handler, then dispatch rewrites execution to the agent main-session key.

## Solution Assessment

No. The requested direction is maintainable, but this patch is not the safest implementation because the effective session key is chosen after policy and replay-scope decisions; a shared effective-session resolver before validation and dispatch is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Validate the effective main-session key before dispatch:** `src/gateway/server-http.ts:793`
  - body: When a mapped action sets `sessionTarget: "main"`, `dispatchAgentHook` replaces the already-validated hook session key with the agent main-session key. That makes `hooks.allowedSessionKeyPrefixes` check a key that may never be used, which can reject valid main-target mappings or allow a disallowed `agent:*` main session after a `hook:*` key passed validation. Resolve and validate the effective key before dispatch.
  - confidence: 0.9
- **[P2] Include sessionTarget in hook replay scope:** `src/gateway/server-http.ts:793`
  - body: Forwarding `sessionTarget` changes where the hook run executes, but the idempotency replay fingerprint still omits that field. Two otherwise identical mapped deliveries with the same idempotency key can return the cached run id from an isolated run when the second request should run in the main session, or vice versa.
  - confidence: 0.84
- **[P2] Regenerate config metadata for the new mapping key:** `src/config/zod-schema.hooks.ts:47`
  - body: Adding `sessionTarget` to the strict hook mapping schema makes this a public config key, but the generated schema/help/label surfaces are not updated. Schema-driven config tooling will not expose or preserve `hooks.mappings[].sessionTarget`, creating contract drift.
  - confidence: 0.87
- **[P3] Add the required changelog entry:** `src/config/types.hooks.ts:18`
  - body: This user-facing `feat` adds a new hook mapping config option, but the PR does not include a `CHANGELOG.md` entry. Add a single-line entry in the active version section with contributor credit so the shipped config feature is visible in release notes.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete policy-enforcement bug because main-target hook runs can execute on a different session key than the one checked against `hooks.allowedSessionKeyPrefixes`.

Concerns:

- **[medium] Session-prefix policy checks the wrong key:** `src/gateway/server-http.ts:793`
  - body: For mapped `sessionTarget: "main"` runs, the request handler validates the hook/default session key, but dispatch rewrites execution to the agent main-session key. On exposed hook endpoints this can bypass an operator's session-key prefix allowlist or wrongly reject allowed main-session configurations.
