## Summary

This should stay open: current main still has the runtime-only continuation prompt and a group/tool-only delivery path where a non-empty final answer can remain private without a visible send or failure. The report overlaps https://github.com/openclaw/openclaw/issues/81116, but it adds a distinct delivery-context symptom rather than being fully superseded.

## What This Changes

Ensure a WhatsApp-originated internal continuation either delivers final output to the original chat or clearly marks it non-deliverable instead of silently ending in transcript-only state.

## Best Possible Solution

Add a shared continuation-delivery invariant: channel-bound internal continuations must preserve origin delivery context, and message-tool-only runs that produce user-facing final content without a message tool send should visibly fail, warn, or recover through an approved route without violating room privacy defaults.

## Reproduction Assessment

No high-confidence live reproduction is available from this review. Source inspection shows the current-main path for the reported symptom: runtime-only continuation prompts are generic, group replies are message-tool-only by default, and follow-up final payloads are returned privately when no message tool send occurs.

## Solution Assessment

Unclear. Automatically posting normal finals would conflict with the documented message-tool-only privacy contract, so the maintainable fix likely needs a maintainer decision between preserving/forcing message-tool delivery context, surfacing an explicit non-delivery error, or sending a safe stall notice.
