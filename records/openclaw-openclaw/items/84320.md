## Summary

Keep open: the PR has strong live Telegram proof for the visible happy path, but source review still shows blocking message-delivery defects in the native draft lifecycle and the branch reverses a current Telegram maintainer note without visible owner approval.

## What This Changes

The PR adds opt-in Telegram DM answer streaming via `sendMessageDraft`, shared native draft helpers, config/schema/docs/UI hints, and focused Telegram draft tests.

## Best Possible Solution

Keep Telegram’s default edit-based streaming unless owner approval lands an opt-in native draft design that treats drafts as non-durable previews and finalizes exactly once through the answer finalization path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable `channels.telegram.streaming.nativeTransport`, stream a partial DM answer, then trigger tool progress or another non-final delivery before finalization.

## Solution Assessment

No. The opt-in direction may be viable, but the current patch reuses a durable preview contract for an ephemeral draft transport; the safer solution needs a separate non-durable draft lifecycle and owner approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep native answer drafts non-durable during rotation:** `extensions/telegram/src/bot-message-dispatch.ts:588-592`
  - body: With native answer drafts enabled, a partial answer can set `hasStreamedMessage` while still having no message id. If tool progress or a later finalized lane rotates the answer lane, the existing rotation path calls `stream.stop()`, and the new stream turns that partial preview into a persistent `sendMessage` before the real final answer.
  - confidence: 0.9
- **[P1] Return handled non-final native draft updates:** `extensions/telegram/src/native-answer-draft.ts:208`
  - body: `messageId()` stays undefined until the final `sendMessage`, so `createLaneTextDeliverer` treats non-final native draft updates as unhandled and falls back to `sendPayload`. That can make block/tool text appear both as a native draft preview and as a separate sent message.
  - confidence: 0.88
