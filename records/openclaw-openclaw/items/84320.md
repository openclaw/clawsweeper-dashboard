## Summary

Keep open. The supplied Telegram recording is useful happy-path proof, but the patch still conflicts with the current Telegram streaming maintainer note and has blocking final-delivery and generated-metadata defects that can break or lose final answers when the new opt-in path is enabled.

## What This Changes

The PR adds opt-in Telegram DM answer streaming through native sendMessageDraft, shared draft helper utilities, config/docs wiring, and unit tests for the native draft helper.

## Best Possible Solution

Keep the default edit-based Telegram answer streaming path, and only accept native answer drafts if Telegram owners approve the policy change after final materialization, failed-send fallback, metadata generation, and fallback proof are fixed.

## Reproduction Assessment

Yes for the blocking findings by source inspection: the PR-head finalization path sends raw pending text with HTML parse mode, failed final sends still satisfy sendMayHaveLanded, and generated Telegram metadata lacks streaming.nativeTransport. I did not run live Telegram fallback repros beyond inspecting the supplied recording.

## Solution Assessment

No. The opt-in concept may be acceptable, but the current PR is not the best landing shape until it preserves rendered final delivery semantics, falls back on known final-send failures, updates generated metadata, and gets explicit Telegram owner approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Send the rendered final text:** `extensions/telegram/src/native-answer-draft.ts:166`
  - body: stopAndMaterialize computes `rendered` but then sends `pendingFullText` with the rendered parse mode. With the dispatch renderer that means raw Markdown or Telegram-sensitive characters can be sent as HTML, so the final message can be rejected or visibly differ from the streamed draft; pass the rendered text to `sendMessage` instead.
  - confidence: 0.93
- **[P1] Do not mark failed final sends as delivered:** `extensions/telegram/src/native-answer-draft.ts:134`
  - body: `messageSendAttempted` is set before final `sendMessage` and is never cleared when that call throws or returns no `message_id`. The shared lane finalizer treats `sendMayHaveLanded()` as delivered when there is no message id, so a known failed final send can skip the normal fallback and leave the user without a final answer.
  - confidence: 0.9
- **[P2] Regenerate the Telegram channel metadata:** `src/config/zod-schema.providers-core.ts:116`
  - body: Adding `streaming.nativeTransport` to the runtime Zod schema is not enough because the generated bundled Telegram channel metadata still lacks that key and its UI hint. The PR body already reports this as a startup `additional properties` rejection, so the generated metadata path needs to be updated before documenting the option as usable.
  - confidence: 0.88
