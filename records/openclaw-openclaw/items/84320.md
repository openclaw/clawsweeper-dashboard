## Summary

Keep this PR open. The live Telegram recordings are useful and the earlier config metadata gap appears fixed, but the native draft failure path can still leave stale or duplicate final messages and the change replaces a current Telegram maintainer decision that needs owner approval before landing.

## What This Changes

The PR adds opt-in Telegram `streaming.nativeTransport` DM answer streaming via `sendMessageDraft`, updates config metadata/docs/UI hints, extracts native draft helpers, and adds Telegram dispatch/unit coverage.

## Best Possible Solution

Land native Telegram draft streaming only after failed-draft fallback cannot emit stale or duplicate finals and a Telegram owner approves the opt-in answer-streaming policy; keep edit-based streaming as the default path.

## Reproduction Assessment

Yes for the review finding from source inspection: a `sendMessageDraft` rejection stops the native stream, later updates are ignored by shared controls, and final materialization can send stale pending text before fallback sends the real final. I did not run a live failing Telegram repro.

## Solution Assessment

No, not yet. The opt-in design is plausible and better scoped than changing the default, but the failed-draft fallback needs a code fix and the maintainer-note policy needs owner approval before this is the best landing path.

- **[P2] Avoid materializing stale draft text after failures:** `extensions/telegram/src/native-answer-draft.ts:125-126`
  - body: When a draft update throws, this catch marks the stream stopped but leaves `pendingFullText` set to the last attempted partial. That can leave duplicate or stale Telegram messages after a transient/API draft rejection; disable native materialization after draft failure or ensure the final text replaces pending before `materialize()`.
