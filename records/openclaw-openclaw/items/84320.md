## Summary

Keep open: the happy-path proof is real, but the patch still has a blocking source-level message-delivery bug because the new native stream materializes pending answer text whenever the existing lane rotation calls `stop()`, and it also reverses the current Telegram answer-streaming maintainer policy without visible owner approval.

## What This Changes

This PR adds opt-in Telegram DM answer streaming via native `sendMessageDraft`, shared draft helpers, config schema/metadata/UI hints, docs, and tests.

## Best Possible Solution

Keep native answer streaming opt-in only if Telegram owners approve it, and split the native draft lifecycle so only true final delivery persists with `sendMessage` while non-final rotations discard or pause the ephemeral draft.

## Reproduction Assessment

Yes by source inspection. Enable `channels.telegram.streaming.nativeTransport: true` for a partial-mode DM, stream an answer partial, then trigger a tool-progress or assistant-message rotation; dispatch calls `rotateLaneForNewMessage`, which calls the new stream's `stop()` and persists pending text with `sendMessage`.

## Solution Assessment

No. The feature may be maintainable as an opt-in path, but this implementation needs separate finalization versus non-final rotation semantics plus Telegram owner approval before it is the best way to solve the issue.

## Review Findings

Overall correctness: patch is incorrect

- **[P1] Avoid final sends on non-final draft rotation:** `extensions/telegram/src/native-answer-draft.ts:166-170`
