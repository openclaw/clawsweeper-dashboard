## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Send the rendered final Telegram text:** `extensions/telegram/src/native-answer-draft.ts:166`
  - body: `renderText` converts the stream text into Telegram HTML and returns `parseMode: "HTML"`, but the final materialization throws away `rendered.text` and sends `pendingFullText` with that HTML parse mode. The draft preview can look correct while the persisted answer loses formatting or fails on raw characters that are invalid HTML entities, so the final `sendMessage` needs to use the rendered text it just computed.
  - confidence: 0.92
- **[P1] Do not mark failed final sends as delivered:** `extensions/telegram/src/native-answer-draft.ts:213`
  - body: `messageSendAttempted` is set before the final `sendMessage`, but it is never cleared when that call throws or returns no `message_id`. Lane delivery treats `sendMayHaveLanded()` as a delivered final when no message id is available, so a known final-send failure suppresses the normal fallback and can leave the user with only an expiring draft.
  - confidence: 0.9
- **[P1] Update generated Telegram metadata for the new option:** `src/config/zod-schema.providers-core.ts:116`
  - body: Adding `streaming.nativeTransport` to the Zod schema is not enough because setup/config metadata still comes from the generated bundled channel metadata and the Telegram plugin manifest. As the PR body notes, that path rejects the documented key as an additional property, so the new documented config can fail startup until the metadata generation/output and coverage are updated.
  - confidence: 0.86
