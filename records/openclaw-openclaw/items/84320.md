## Summary

Keep this PR open but not merge-ready: source review still shows native draft answer streaming can create unintended visible Telegram messages on non-final lifecycle paths, and the branch reverses a current Telegram maintainer note without visible owner approval.

## What This Changes

The branch adds opt-in Telegram DM answer streaming via `sendMessageDraft`, shared native draft helpers, config schema/metadata/UI hints, docs, maintainer-note updates, and unit tests.

## Best Possible Solution

Keep the default edit-based path, and only add opt-in native answer streaming after Telegram owners approve a lifecycle where non-final draft updates are preview-only and only an explicit finalization path calls `sendMessage`, with integration coverage for interruption, block, group, and default-off behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable `channels.telegram.streaming.nativeTransport`, stream a partial DM answer, then hit a tool-progress or non-final block path; the new native stream either materializes partial text on `stop()` or falls through because it has no message id before finalization.

## Solution Assessment

No. The underlying native draft idea is plausible, but the PR plugs it into an edit-message lifecycle whose `stop()` and `messageId()` contracts do not match ephemeral drafts; a native-specific lifecycle plus Telegram owner approval is the safer solution.

Full review comments:

- **[P1] Avoid final sends during non-final draft rotation:** `extensions/telegram/src/native-answer-draft.ts:166-170`
  - body: `stop()` is used by existing lane rotation paths before tool progress and new assistant segments. With this implementation, `stop()` calls `sendMessage` for whatever partial text is pending, so an interrupted native draft can become a persisted Telegram bubble before the real final answer. Split final materialization from non-final stop/rotation, or have rotations discard/pause native drafts.
  - confidence: 0.9
- **[P1] Handle non-final native draft updates without falling through:** `extensions/telegram/src/native-answer-draft.ts:208`
  - body: The shared lane deliverer treats a preview update as handled only after `stream.messageId()` returns a number. This native stream keeps `messageId()` undefined until final `sendMessage`, so non-final block/tool answer deliveries can update the draft and then fall through to normal `sendPayload`, creating visible messages instead of preview-only updates.
  - confidence: 0.84
