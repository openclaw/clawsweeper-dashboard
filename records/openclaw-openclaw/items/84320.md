## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Send the rendered Telegram text on final materialization:** `extensions/telegram/src/native-answer-draft.ts:140`
  - body: `materializeFinalMessage` sends the raw `text` while applying the parse mode from `renderText`; the draft preview used rendered/escaped HTML, but the final message gets unrendered model text. Raw `<` or markdown can fail Telegram HTML parsing, and because `messageSendAttempted` stays true, `sendMayHaveLanded()` lets the lane mark that failed final as delivered instead of falling back. Send the rendered text, and only treat send failures as ambiguous when they really may have landed.
  - confidence: 0.91
- **[P2] Regenerate the bundled Telegram config metadata:** `src/config/zod-schema.providers-core.ts:116`
  - body: Adding `nativeTransport` to the core Zod schema does not update `src/config/bundled-channel-config-metadata.generated.ts`; parsing the PR version shows Telegram `streaming.properties` still lacks this key and the UI hint is absent. Metadata-backed setup/config validation can therefore reject the documented `channels.telegram.streaming.nativeTransport` option with an additional-property error, matching the bug noted in the PR body.
  - confidence: 0.88
- **[P2] Fall back after sendMessageDraft runtime rejection:** `extensions/telegram/src/native-answer-draft.ts:126-129`
  - body: The edit-stream fallback in `createDraftLane` only runs when the native factory returns `undefined`. If a client exposes `sendMessageDraft` but the Bot API rejects it here, this catch only stops the native stream, so that turn loses the existing edit-based preview path despite the PR promising silent fallback. Switch the lane to `createTelegramDraftStream` on rejection or disable native before creating the lane.
  - confidence: 0.82
