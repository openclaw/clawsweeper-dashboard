## Summary

Keep this PR open, but it is not merge-ready: the native Telegram answer draft path still has blocking final-delivery defects, generated config metadata does not expose the new option, and the branch reverses a current Telegram streaming maintainer decision without explicit owner approval.

## What This Changes

The PR adds opt-in Telegram DM answer streaming via native `sendMessageDraft`, shared draft helpers, config/docs wiring, and unit tests for the new native draft path.

## Best Possible Solution

Keep edit-based Telegram answer streaming as the default, repair the native finalization and generated metadata, then let Telegram owners explicitly decide whether opt-in native answer drafts should become a supported config surface.

## Reproduction Assessment

Yes for the blocking review findings by source inspection: the PR head sends `pendingFullText` with HTML parse mode, exposes `sendMayHaveLanded` after failed final sends, and leaves generated Telegram metadata without `nativeTransport`. I did not run a live failure repro because the review was read-only.

## Solution Assessment

No, not yet. The native transport may be a reasonable opt-in direction, but the current patch should first preserve rendered final delivery, allow fallback when final materialization fails, regenerate metadata, and get explicit Telegram owner approval for changing the maintained streaming policy.
