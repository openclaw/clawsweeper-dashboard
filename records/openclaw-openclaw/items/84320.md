## Summary

Keep this PR open: the happy-path Telegram proof is useful, but the current patch still violates the Telegram draft lifecycle by letting native DM drafts behave like durable preview messages even though they have no message id until finalization.

## What This Changes

The branch adds opt-in Telegram DM answer streaming through `sendMessageDraft`, shared native draft helpers, config/schema/docs/UI hints, and Telegram draft tests.

## Best Possible Solution

Keep Telegram’s default edit-based streaming unless owner approval lands an opt-in native draft design that treats drafts as non-durable previews and finalizes exactly once through the answer finalization path.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: enable `channels.telegram.streaming.nativeTransport`, stream a partial DM answer, then hit tool progress or another non-final delivery; the native draft has no message id until finalization, so existing lane code either materializes the partial on rotation or falls through to normal send.

## Solution Assessment

No. The opt-in direction may be viable only after Telegram owner approval and a lifecycle design that separates ephemeral draft previews from durable final sends; the current patch reuses a durable preview contract that assumes a message id.
