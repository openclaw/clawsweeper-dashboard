## Summary

Keep open: current main still lacks write-time validation for ineffective `gateway.nodes.denyCommands`, and this PR is useful, but it is not merge-ready because it targets the old config-set flow and rejects pattern-looking command IDs before checking whether they are known exact IDs.

## What This Changes

This PR adds a `gateway.nodes.denyCommands` validator, wires config set/unset dry-run and write paths to run it, and adds focused CLI/helper tests.

## Best Possible Solution

Land a rebased write-time denyCommands validator in the shared config mutation path, with exact-ID-first validation and regression coverage.

## Reproduction Assessment

Yes. Source inspection of current main shows config writes reach `replaceConfigFile` without a denyCommands validator, and PR-head source shows the exact-ID regression by checking pattern syntax before `known.has`.

## Solution Assessment

No, not as currently proposed. The direction is right, but the validator must be ported to current `runConfigOperations` and must accept known exact command IDs before warning about pattern-like unknown values.

- **[P2] Move validation into runConfigOperations:** `src/cli/config-cli.ts:1037-1041`
  - body: This branch adds the deny-command check in the old `runConfigSet` mutation body, but current main routes set, patch, and delete operations through `runConfigOperations` and writes via `replaceConfigFile`.   - confidence: 0.88

Because this rejects `group:`/`*`/`?`-looking values before consulting the known set, a custom command like `foo*bar` can be allowlisted and enforced at runtime but cannot be saved in `denyCommands`.
  - confidence: 0.87
