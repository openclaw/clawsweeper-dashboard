## Summary

Keep this PR open: the underlying write-time denyCommands validation bug is still present on current main, but the branch needs repair before merge because it is stale against the current config mutation flow and rejects some exact command IDs that runtime policy would honor.

## What This Changes

The PR adds a gateway.nodes.denyCommands validator, wires it into config writes, and adds focused CLI/helper tests for invalid, parent-object, and custom-command cases.

## Best Possible Solution

Land a rebased write-time validator in the current shared config mutation paths, preserving exact-string command IDs and adding regression coverage plus a changelog entry.

## Reproduction Assessment

Yes. Source inspection of current main shows config set/patch writes reach replaceConfigFile without a denyCommands validator, while schema only enforces string-array shape and audit warns only after write.

## Solution Assessment

No, not as currently proposed. The direction is right, but the branch must be rebased into current runConfigOperations/runConfigUnset behavior and must check known exact commands before rejecting pattern-like unknown values.

- **[P2] Move validation into runConfigOperations:** `src/cli/config-cli.ts:1037-1041`
  - body: This branch adds the deny-command check around the old config set mutation body, but current main routes set, patch, and delete operations through runConfigOperations before replaceConfigFile.
