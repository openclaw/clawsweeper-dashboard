## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks write-time validation for `gateway.nodes.denyCommands`, so the linked bug remains real, but the branch still has a correctness issue around exact command IDs and needs rebasing onto the current config mutation path.

## What This Changes

This PR adds a deny-command validation helper, wires `openclaw config set` and `config unset` to reject invalid `gateway.nodes.allowCommands`/`denyCommands` mutations before writing, and adds focused CLI/helper tests.

## Best Possible Solution

Land a rebased patch in the current shared config mutation path that validates writes affecting `gateway.nodes.allowCommands` or `gateway.nodes.denyCommands`, reuses a runtime-accurate known-command catalog with the audit code, preserves dry-run JSON behavior, and only rejects entries that are truly ineffective under exact runtime matching.

## Reproduction Assessment

Yes. Current-main code inspection gives a high-confidence reproduction path for the original bug: write an unknown deny command through `openclaw config set` and the config mutation path has no pre-write deny-command validator. The PR-specific regression is also reproducible at helper level with `allowCommands: ["foo*bar"]` and `denyCommands: ["foo*bar"]`.

## Solution Assessment

No, not as currently proposed. The direction is right, but the patch should align with runtime exact-string semantics and current `runConfigOperations`; a safer version would check known exact IDs before treating unknown pattern-looking values as ineffective.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Allow exact command IDs before pattern warnings:** `src/config/validate-deny-commands.ts:134-136`
  - body: The gateway protocol accepts arbitrary non-empty command strings, and runtime policy treats `allowCommands`, declared node commands, and `denyCommands` as trimmed exact strings. Because this check rejects pattern-looking values before verifying whether they are known exact commands, a valid custom command such as `foo*bar` or `group:sync` can be allowlisted and denied at runtime but cannot be written through `config set` on this branch.
  - confidence: 0.86
