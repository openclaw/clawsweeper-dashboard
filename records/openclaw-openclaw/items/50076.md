## Summary

Keep this PR open. Current main still lacks write-time validation for ineffective `gateway.nodes.denyCommands`, but the proposed patch is not ready because it rejects valid exact custom command IDs before checking whether they are known, and the branch is now dirty against the current config mutation path.

## What This Changes

This PR adds a deny-command validation helper, wires `openclaw config set` and `config unset` to validate writes affecting `gateway.nodes.allowCommands` or `gateway.nodes.denyCommands`, and adds focused CLI/helper tests.

## Best Possible Solution

Land a rebased patch in the current shared config mutation path that validates writes affecting `gateway.nodes.allowCommands` or `gateway.nodes.denyCommands`, preserves dry-run JSON behavior, reuses or mirrors the audit known-command catalog, and accepts any known exact command ID before warning about pattern-like unknown values.

## Reproduction Assessment

Yes. Current-main code inspection gives a high-confidence reproduction path for the original bug: write an unknown deny command through `openclaw config set` and no pre-write deny-command validator runs. The PR-specific regression is also reproducible at helper level with `allowCommands: ["foo*bar"]` and `denyCommands: ["foo*bar"]`.

## Solution Assessment

No, not as currently proposed. The direction is right, but the validator should check the known exact-command set before pattern heuristics, and the integration must be rebased into the current `runConfigOperations` flow.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Allow known command IDs before pattern warnings:** `src/config/validate-deny-commands.ts:122-128`
  - body: The gateway protocol accepts arbitrary non-empty command strings, and runtime policy matches `allowCommands`, declared node commands, and `denyCommands` by trimmed exact string. Because this check rejects pattern-looking values before checking whether they are known exact commands, valid custom IDs like `foo*bar` or `group:sync` can be allowlisted and enforced at runtime but cannot be saved through `config set` on this branch.
  - confidence: 0.86
