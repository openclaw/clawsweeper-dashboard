## Summary

Keep this open. Current main has durable session files and shipped restart transcript recovery, but this Feishu/local-gateway report is about a concrete live message that never appeared in any session JSONL after a disconnect/recreated backend session. The inspected Feishu code records delivered events into the channel turn/session path, but I did not find a reconnect backfill path or a user-visible session-switch notice that would cover this exact loss mode.

## What This Changes

The issue asks OpenClaw to preserve or recover Feishu DM session history across backend session disconnect/recreation, or notify the user when context/session switches.

## Best Possible Solution

Define the supported Feishu downtime contract. If missed Feishu messages should be recoverable after gateway outage, add a Feishu reconnect/backfill or generic durable inbound queue; if provider-undelivered messages are outside OpenClaw's reach, document that boundary and add a same-channel interruption/session-switch notice for recoverable restarts.

## Reproduction Assessment

Unclear. The issue provides a concrete live timeline and message ID, but I did not reproduce a Feishu WebSocket/laptop-sleep outage on current main; source review shows delivered events are recorded before dispatch, while no OpenClaw backfill path is evident for events that never reach the process.

## Solution Assessment

Unclear. Existing session persistence and restart recovery are necessary but not sufficient for the reported loss mode; the best fix needs maintainer scoping between Feishu backfill, generic inbound durability, user notification, and docs.
