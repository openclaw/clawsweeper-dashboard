## Summary

Keep open: current main records Feishu events that reach OpenClaw and has some restart-aborted session recovery, but it still does not provide Feishu missed-message backfill, durable inbound capture during a local outage, or a proven same-channel recovery notice for the reported message that was absent from all session JSONL files.

## What This Changes

The issue requests durable handling of Feishu conversation/session data across local gateway or network interruption so messages are not silently lost from the user’s ongoing chat.

## Best Possible Solution

Define the Feishu downtime contract, then implement either provider-backed missed-message recovery/durable inbound capture or an explicit channel-visible interruption/session-switch notice, coordinated with the existing restart-recovery PRs.

## Reproduction Assessment

Unclear. The report has a concrete timeline and message ID, and source inspection shows delivered Feishu events are recorded while startup/reconnect lacks missed-event backfill; I did not reproduce the live Feishu sleep/local-gateway outage.

## Solution Assessment

Unclear. Existing persistence and restart recovery are necessary but too narrow for a Feishu message absent from all transcripts, so the maintainable fix needs maintainer scoping across Feishu backfill, generic pending inbound, and visible recovery UX.
