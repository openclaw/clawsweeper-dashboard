## Summary

Keep open: current main still lacks `--profile-id` on `models auth login`; the discussion correctly separates the earlier Codex email-detection bug from this remaining CLI feature, and the active implementation path is the open, currently conflicting PR at https://github.com/openclaw/openclaw/pull/49315.

## What This Changes

Add a `--profile-id` flag to `models auth login` so OAuth login flows can persist credentials under an operator-chosen auth profile id.

## Best Possible Solution

Review, rebase, and land https://github.com/openclaw/openclaw/pull/49315 or an equivalent narrow implementation with CLI docs, focused tests, and provider-safe profile targeting.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a request for a new CLI flag. Source inspection on current main confirms the requested flag is absent from `models auth login` while present for `paste-token`.

## Solution Assessment

Yes in direction: a narrow `models auth login --profile-id` surface is the right shape. The implementation should thread the target profile through the supported provider-auth contract or reject methods that cannot safely honor it before credentials are written.
