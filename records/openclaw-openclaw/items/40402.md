## Summary

Keep open: current main and the latest release still lack `--profile-id` on `models auth login`. The separate Codex identity-collapse problem is addressed by provider-derived email/stable IDs, but operator-controlled OAuth profile naming remains unimplemented and is already represented by https://github.com/openclaw/openclaw/pull/49315, which needs maintainer semantics review and proof before this issue can close.

## What This Changes

The issue requests a `--profile-id` CLI flag for `openclaw models auth login` so OAuth logins can be stored under operator-chosen auth profile IDs.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/49315 with a provider-auth contract change that threads a typed target before provider credential/config writes, rejects ambiguous methods, and updates docs/tests with real behavior proof.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new CLI flag. Source inspection confirms current main lacks `--profile-id` on `models auth login`, while `paste-token` has that flag.

## Solution Assessment

Yes in direction, but the feature is not implemented on main. The maintainable solution should pass an explicit target through the provider-auth contract before persistence, or reject auth methods that cannot safely honor it.
