## Summary

Keep open: current main and v2026.5.12 still lack `--profile-id` on `models auth login`, and the active implementation path remains the open, conflicting PR at https://github.com/openclaw/openclaw/pull/49315. The earlier Codex profile-collapse bug is separate and current source now derives Codex profile identity from token claims or a stable subject fallback, but operator-controlled OAuth profile naming is still not implemented.

## What This Changes

The issue requests a `--profile-id` CLI flag for `models auth login` so OAuth flows can save credentials under an operator-selected auth profile ID.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/49315 with a provider-auth contract change that adds `--profile-id`, threads a typed target before provider credential/config writes, rejects ambiguous methods, and updates docs/tests.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a CLI feature request. Source inspection confirms current main lacks `--profile-id` on `models auth login`, while `paste-token` has that flag.

## Solution Assessment

Yes in direction, but the feature is not on main. The maintainable solution should pass an explicit target through the provider-auth contract before persistence or reject auth methods that cannot safely honor it.
