## Summary

Keep open: current `main` still exposes `--profile-id` only for `models auth paste-token`, while `models auth login` has no caller-selected profile target. The remaining work is already represented by the open implementation PR at https://github.com/openclaw/openclaw/pull/49315, but that PR is still open/conflicting and needs maintainer agreement on provider-auth semantics plus real behavior proof.

## What This Changes

The issue requests a `--profile-id` flag for `openclaw models auth login` so OAuth login flows can persist credentials under an operator-selected auth profile ID.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/49315 with a provider-auth contract that passes a typed requested profile target into supported auth methods before persistence, rejects ambiguous/unsupported methods clearly, updates docs/tests, and includes redacted real behavior proof.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature request for a new CLI flag. Source inspection confirms current main lacks `--profile-id` on `models auth login`, while `paste-token` has that flag.

## Solution Assessment

Yes in direction, but the feature is not implemented on main. The maintainable solution should add an explicit provider-auth target contract before persistence, or reject auth methods that cannot safely honor a requested profile ID.
