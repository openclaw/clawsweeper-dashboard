## Summary

Keep open. Current main still does not expose an operator-selected OAuth profile alias on `models auth login`; the related OpenAI Codex email-derived profile fix covers one overwrite path, but not the broader `--profile-id` feature, and open PR #49315 is already the active implementation candidate.

## What This Changes

Add a `--profile-id <id>` option to `models auth login` so OAuth login results can be persisted under an operator-selected auth profile ID.

## Best Possible Solution

Keep this as the tracker for explicit OAuth profile aliases. Review and land PR #49315 or equivalent work that adds `--profile-id <id>` to `models auth login`, validates the requested ID against the selected provider, handles multi-profile auth results without silent ambiguity, and updates focused CLI, persistence, provider, and docs coverage.

## Reproduction Assessment

Yes. The missing feature is reproducible by inspecting current main: `models auth login` has no `--profile-id` option or typed data path, while `paste-token` does.

## Solution Assessment

Yes, the generic login-level `--profile-id` direction is the narrow maintainable solution. A provider-specific OpenAI Codex alias would not cover other OAuth providers that also need operator-selected profile names.
