## Summary

Keep open. Current main still puts the configured Gmail watcher bootstrap on the awaited Gateway sidecar startup path, and the issue discussion contains a reporter-confirmed bisection showing that Gmail watcher startup, not model-pricing, caused the multi-minute delay.

## What This Changes

Gateway startup should stop waiting on network-dependent hook/Gmail watcher initialization and allow the watcher work to finish after the Gateway becomes usable.

## Best Possible Solution

Make only the Gmail watcher bootstrap nonblocking from Gateway sidecar startup: keep the existing skip flag and error logging, add an explicit background-start log, preserve watcher shutdown/reload behavior, and add a regression test with a deliberately slow watcher proving Gateway readiness/channel startup no longer waits for it.

## Reproduction Assessment

Yes. Configure `hooks.enabled=true` with `hooks.gmail.account`, then make `gog gmail watch start` or Tailscale setup slow or mocked as unresolved; current main awaits that path before sidecars are considered ready.

## Solution Assessment

Yes for the localized bottleneck. Backgrounding `startGmailWatcherWithLogs` from `sidecars.gmail-watch` is narrower and more maintainable than adding a broad hook timeout/skip framework before multiple hook owners need one.
