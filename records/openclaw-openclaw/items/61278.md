## Summary

Keep open: current main still awaits the Gmail watcher during startup sidecars, and the reporter-confirmed bisection isolates the long Windows delay to that path rather than model-pricing.

## What This Changes

The issue asks for gateway startup to stop waiting on slow hook or sidecar initialization, especially the Gmail watcher path that delayed Windows startup.

## Best Possible Solution

Background only Gmail watcher bootstrap from the awaited startup sidecar path while preserving skip behavior, logging, and existing channel/plugin-service readiness gates.

## Reproduction Assessment

Yes. Configure `hooks.gmail.account` so Gmail watcher startup runs, then a slow `gog gmail watch start` or Tailscale setup stays on the awaited `sidecars.gmail-watch` path; the reporter also provided a successful `OPENCLAW_SKIP_GMAIL_WATCHER=1` bisection.

## Solution Assessment

Yes. The narrowest fix is to make Gmail watcher bootstrap fire-and-forget with clear logging and regression coverage; adding new timeout/skip configuration is not required for this issue.
