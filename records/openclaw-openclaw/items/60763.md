## What This Changes

The branch adds optional `expiresAtMs` and `bindSession` fields to Discord component specs, parsing/schema wiring, build output, and focused component tests.

## Best Possible Solution

Define safe Discord unbound-routing semantics and a maximum TTL, then rework the feature onto the current split component files with focused dispatch/registry tests, docs if public, and real Discord proof.

## Reproduction Assessment

Yes for the review findings: source inspection traces cleared route overrides into channel-route fallback and traces `expiresAtMs` into registry storage. I did not run a live Discord setup in this read-only review.

## Solution Assessment

No. The capability may be useful for cron-owned Discord messages, but this PR needs explicit authorization semantics, bounded expiry, current-main rework, and real behavior proof before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require a user principal for unbound components:** `extensions/discord/src/components.ts:788-790`
  - body: When `bindSession=false` clears session, agent, and account route overrides, dispatch falls back to the channel route while components without `allowedUsers` still pass. Require an origin user, allowlist, or explicit guarded shared mode before callbacks can run unbound.
  - confidence: 0.9
- **[P2] Limit caller-provided component expiry:** `extensions/discord/src/components.ts:802`
  - body: `expiresAtMs` is accepted as any finite number and copied to registry `expiresAt`, so far-future values bypass the in-memory default retention window. Clamp or reject values outside the allowed lifetime before registering entries and modals.
  - confidence: 0.86

## Security Review
