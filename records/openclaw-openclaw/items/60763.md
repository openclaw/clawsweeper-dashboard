## Summary

Keep open: the requested Discord component capability is still absent from current main, but this PR is not merge-ready because it changes callback routing and component lifetime semantics without safe authorization, bounded expiry, current-main rebase, or real Discord proof.

## What This Changes

The PR adds optional `expiresAtMs` and `bindSession` fields to Discord component specs/schema/tests so callbacks can expire at caller-specified times and route through the channel session instead of the sending session.

## Best Possible Solution

Implement a guarded Discord shared-component mode in the current split files, keeping session-bound behavior as the default and adding explicit authorization, bounded TTL handling, schema/docs/tests, and redacted live proof.

## Reproduction Assessment

Yes for the review findings: source inspection traces cleared route overrides into channel-route fallback, optional `allowedUsers`, and caller-provided expiry into stored registry expiry. I did not run a live Discord setup in this read-only review.

## Solution Assessment

No. The feature direction is useful, but the current patch is not the best fix until it defines safe shared-route authorization semantics, bounds expiry, rebases onto current main, and proves live behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Require an authorization guard for unbound routing:** `extensions/discord/src/components.ts:788-790`
  - body: With `bindSession=false`, this clears all route overrides even when no component or modal `allowedUsers` is set. Dispatch then falls back to the channel route, while current component auth allows entries with no allowlist, so a user-specific or privileged button can become triggerable by any otherwise allowed channel member. Please require an origin user, explicit allowlist, or named shared mode before unbinding.
  - confidence: 0.88
- **[P2] Bound caller-provided component expirations:** `extensions/discord/src/components.ts:442-443`
  - body: `expiresAtMs` is parsed as a plain number and stored as `entry.expiresAt` / `modal.expiresAt`, bypassing the existing 30-minute lifetime for active entries. A far-future timestamp can keep callbacks valid indefinitely; clamp or reject values outside the maintained TTL window and cover both entries and modals.
  - confidence: 0.82
