## Summary

Keep open: the requested Discord component capability is not on current main, but this PR still needs maintainer/security decisions for unbound routing and expiry bounds, plus real Discord behavior proof before it can merge.

## What This Changes

The branch adds optional `expiresAtMs` and `bindSession` fields to Discord component specs, schema, build wiring, and tests so components can use caller-chosen expiry and optionally route through the channel session.

## Best Possible Solution

Define the Discord unbound-routing authorization model and maximum TTL first, then implement the feature on the current split component files with focused dispatch/registry tests, docs if user-facing, and real Discord proof.

## Reproduction Assessment

Yes for the review findings: source inspection traces the PR's unbound route overrides into dispatch fallback and its `expiresAtMs` value into registry storage. A live Discord run was not needed to establish those code-path risks.

## Solution Assessment

No. The requested capability may be useful for cron-owned Discord component messages, but this diff needs explicit authorization semantics, bounded expiry, current-main rework, and real behavior proof before it is the maintainable solution.
