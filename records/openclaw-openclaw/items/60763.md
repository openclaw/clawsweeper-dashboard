## What This Changes

## Best Possible Solution

Rework the feature on the current split Discord component files with bounded expiry, explicit unbound-routing authorization, focused dispatch/registry tests, docs, and the required changelog entry before maintainer review.

Yes. Current-main source inspection shows the feature is absent, and the PR diff can be traced through dispatch route fallback, optional component allowlists, and registry expiry behavior without needing a live Discord run.

## Solution Assessment

No. The PR is a useful starting point, but the safest maintainable solution needs explicit authorization semantics for unbound interactions, bounded expiry policy, and a rebase onto the current split component implementation.

  - body: When `bindSession=false` clears stored route context, callbacks fall back to the channel-resolved route while missing component `allowedUsers` remains allowed. Require an origin user, allowlist, or explicit guarded shared mode before unbound components can dispatch agent callbacks.
