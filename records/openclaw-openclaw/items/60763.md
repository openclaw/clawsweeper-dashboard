## Summary

Keep open: the feature is still absent from current main, but this PR is not merge-ready because it introduces unbound Discord routing and caller-controlled component lifetimes without a defined authorization or TTL policy, and it lacks real behavior proof.

## What This Changes

The branch adds optional `expiresAtMs` and `bindSession` fields to Discord component specs/schema/building logic and tests so components can use caller-chosen expiry and optionally route through the channel session.

## Best Possible Solution

Define the Discord unbound-routing and expiry policies first, then rework the feature on the current split component files with focused dispatch/registry tests, docs, and real Discord proof.

## Reproduction Assessment

Yes for the review findings: source inspection traces the PR's unbound route overrides into dispatch fallback and its `expiresAtMs` value into registry storage. A live Discord run was not needed to establish those code-path risks.

## Solution Assessment

No. The requested capability may be useful for cron-owned component messages, but this implementation needs explicit authorization semantics, bounded expiry, and a rebase onto current main before it is the narrow maintainable solution.
