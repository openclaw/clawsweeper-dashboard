## Summary

Keep this PR open. Current main still lacks the requested Discord component `expiresAtMs` and `bindSession` fields, and the rebased-from-#60355 context shows a real cron-owned component use case, but the proposed patch is not merge-ready because it changes security-sensitive routing without requiring an authorization principal, lets caller-supplied expiries bypass the current registry TTL policy, and is stale against the current split Discord component files.

## What This Changes

The branch adds optional `expiresAtMs` and `bindSession` fields to Discord component specs, parser/building logic, the Discord message tool schema, and component tests.

## Best Possible Solution

Keep the feature direction, but update the implementation in the current split Discord component files with bounded expiry policy, explicit safe unbound-routing semantics, focused component and dispatch tests, docs, and the required user-facing changelog entry before merge.

## Reproduction Assessment

Yes. The feature gap is reproducible by current-main source inspection, and the security findings trace through the proposed cleared route overrides into dispatch fallback plus the permissive missing-allowlist path.

## Solution Assessment

No. The current patch is a plausible starting point, but the maintainable solution must define the authorization model for unbound interactions and clamp or reject caller-provided expiry before registry storage.
