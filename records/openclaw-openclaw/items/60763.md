## Summary

Keep open: this PR adds a useful Discord component capability, but it introduces unresolved authorization and lifetime-policy risks and lacks required real Discord behavior proof before merge.

## What This Changes

The branch adds optional `expiresAtMs` and `bindSession` fields to Discord component specs, tool schema exposure, component/modal build wiring, and focused component tests.

## Best Possible Solution

Define the Discord unbound-routing authorization model and maximum TTL first, then rework the feature onto the current split component files with focused dispatch/registry tests, docs if user-facing, and real Discord proof.

## Reproduction Assessment

Yes for the review findings: source inspection traces unbound route overrides into dispatch fallback and traces `expiresAtMs` into registry storage. A live Discord run was not needed to establish those code-path risks, but real Discord proof is still needed before merge.

## Solution Assessment

No. The capability may be useful for cron-owned Discord component messages, but this diff needs explicit authorization semantics, bounded expiry, current-main rework, and real behavior proof before it is the maintainable solution.
