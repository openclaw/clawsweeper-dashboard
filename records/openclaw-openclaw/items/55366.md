## Summary

Keep open. Current main still has the reported context-overflow/compaction-failure reset path: it rotates the session and returns the hard-reset message, while the only continuity mitigation is a bounded replay of recent user/assistant transcript records with no explicit reset note, task goal, or progress handoff. The related open report #63216 reinforces that hard-reset recovery is still active work rather than a shipped closed fix.

## What This Changes

The issue asks OpenClaw to inject minimal recovery context after an automatic hard context reset so an agent can resume an in-progress task instead of starting blank.

## Best Possible Solution

Keep the hard reset as a safety valve, but make the rotated session explicitly resumable by writing a small, agent-visible recovery handoff into the new session: reset reason, last user instruction, and safe recent assistant-visible progress, with tests proving it stays bounded and avoids tool/system/compaction replay.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists by making an embedded run surface a context-overflow or compaction-failure error and observing resetSessionAfterCompactionFailure rotate the session while the new transcript only receives bounded user/assistant replay, not an explicit recovery handoff.

## Solution Assessment

Yes, a narrow fix in the reset/replay helper path is the best maintainable direction. It addresses the data-loss UX without changing provider behavior, channel ownership, or the broader compaction retry policy.
