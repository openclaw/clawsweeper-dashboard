## Summary

Keep open. Current main has partial reset recovery by replaying a small user/assistant tail, but it still does not preserve or inject durable in-progress task state, tool progress, or an explicit recovery checkpoint after an unrecoverable context reset; related open items overlap but do not fully supersede this recovery-context request.

## What This Changes

Request that hard context-overflow resets seed the new session with minimal recovery context, including the task goal, last known progress, and reset reason.

## Best Possible Solution

Define a maintainer-approved reset recovery contract that persists or reconstructs a minimal checkpoint with the last user goal, reset reason, safe progress summary, and pointers to prior transcript/tool artifacts without reintroducing the overflowing context.

## Reproduction Assessment

Yes, at source level: context-overflow failures still rotate the session and only replay a bounded user/assistant tail, not durable task progress. I did not replay the original Windows Discord workload end to end.

## Solution Assessment

Unclear. Current main has a useful partial mitigation, but the requested durable recovery context needs a maintainer product decision about checkpoint shape, what state is safe to inject, and how it interacts with compaction and session snapshots.
