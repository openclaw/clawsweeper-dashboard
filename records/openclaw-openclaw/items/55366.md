## Summary

Keep open. Current main still has a hard context-overflow/compaction-failure path that rotates the session and returns the reported reset message; the only continuity mitigation is a bounded replay of recent user/assistant transcript records, without an explicit reset note, task goal, or progress handoff. The related open #63216 report confirms this reset/retry area remains active rather than fully resolved.

## What This Changes

The issue requests that hard context-limit session resets preserve minimal recovery context, including the last instruction, progress state, and reset reason, so agents can resume long-running tasks.

## Best Possible Solution

Keep the hard reset as a last-resort safety valve, but make the rotated session resumable by writing or injecting a bounded agent-visible recovery handoff that includes the reset reason, the latest user goal, and safe recent assistant-visible progress without replaying unsafe record types.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists by forcing runAgentTurnWithFallback through either the embedded context-overflow payload path or the compaction-failure catch path, then inspecting the rotated transcript produced by resetReplyRunSession and replayRecentUserAssistantMessages.

## Solution Assessment

Yes. A narrow recovery handoff in the existing reset/replay helper path is the best maintainable fix; it addresses the user-visible data-loss mode without changing provider compaction semantics or broad retry policy.
