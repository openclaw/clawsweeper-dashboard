## Summary

Keep open: this is a useful Matrix bug-fix PR with real after-fix evidence, but it is not mergeable as-is because the branch trims Matrix passwords, misses the stable `m.oauth` reset stage, and still has lint/test-type CI failures.

## What This Changes

The PR updates the Matrix plugin cross-signing bootstrap to parse UIA 401 flows, detect MAS auth metadata, surface typed MAS reset errors, add focused tests, and add a Matrix changelog entry.

## Best Possible Solution

Land this Matrix plugin fix after preserving raw credentials, supporting stable and unstable OAuth/reset stages with URL extraction, fixing CI/test typing, and documenting MAS recovery.

## Reproduction Assessment

Yes, at source level. Current main's UIA callback can be exercised with a 401 UIA body and no password, and it throws the old password fallback error without inspecting stages, reset URLs, or sessions.

## Solution Assessment

No, not as currently patched. The stage-aware UIA/MAS reset direction is the right narrow solution, but the branch must preserve raw passwords, handle stable `m.oauth`, fix CI, and update operator docs before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve the submitted Matrix password:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.ts:217-228`
  - body: The callback stores `params.password?.trim()` in `password` and then submits that trimmed value in the `m.login.password` auth dict. Matrix passwords are opaque credentials; current main only trims to test configuredness while sending the original value, so accounts with intentional leading or trailing whitespace would fail UIA after this change.
  - confidence: 0.92
- **[P2] Handle the stable MAS OAuth stage:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.ts:232-237`
  - body: The reset branch only recognizes `org.matrix.cross_signing_reset`, but the Matrix spec defines `m.oauth` for OAuth-authenticated `/keys/device_signing/upload` UIA and puts the approval URL in `params.m.oauth`. Treat `m.oauth` as the same operator reset flow and read that params URL so stable-only servers do not fall through to the unsupported-stage path.
  - confidence: 0.88
- **[P2] Remove or use the mismatch helper:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.ts:68`
  - body: `isCrossSigningKeyMismatchError` is declared but never called, and PR CI already fails lint with the unused-function error. Delete it or wire it into the intended fallback path before merge.
  - confidence: 0.99
- **[P2] Type the bootstrap mock before reading its first argument:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.test.ts:607`
  - body: This test defines `bootstrapCrossSigning` as `vi.fn(async () => {})`, so TypeScript infers a zero-argument mock and `bootstrapCrossSigning.mock.calls[0]?.[0]` fails `check-test-types` with TS2493. Give the mock an explicit bootstrap-options signature or reuse the typed mock helper.
  - confidence: 0.97
- **[P3] Document MAS reset recovery:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.ts:232-248`
  - body: The new reset-stage path changes the operator recovery workflow, but the Matrix docs still describe only the old no-auth, dummy, and password UIA sequence. Update the Matrix channel docs to explain opening the MAS reset URL, approving the reset, and rerunning bootstrap inside the approval window.
  - confidence: 0.82
