## Summary

Keep open. The PR addresses a real current-main Matrix MAS/UIA gap, but the current head still trims opaque Matrix passwords before submitting password UIA, which is a behavioral regression that should be repaired before merge.

## What This Changes

The PR changes the Matrix plugin's cross-signing bootstrap to detect MAS, route UIA by advertised stages with session threading, surface typed reset errors with URLs, add regression tests, and add a Matrix changelog entry.

## Best Possible Solution

Land the MAS/UIA implementation after preserving the raw configured password, adding a whitespace-password regression test, and updating Matrix verification docs for MAS reset recovery.

## Reproduction Assessment

Yes. The original failure is traceable from current-main UIA callback behavior and the companion MAS-fronted Synapse report; the PR regression is reproducible with a focused unit test where the configured Matrix password intentionally has leading or trailing whitespace.

## Solution Assessment

No, not as currently patched. The stage-aware UIA/session threading and MAS reset error direction is maintainable, but opaque credentials should not be normalized before submission.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the submitted Matrix password:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.ts:217-228`
  - body: The patch assigns `params.password?.trim()` to `password` and then sends that value in the `m.login.password` auth dict. Current main submits the configured password unchanged, and Matrix passwords are opaque credentials, so accounts with intentional leading or trailing whitespace would fail UIA after this change. Use trimming only to decide whether a password is configured, but submit the original `params.password`.
  - confidence: 0.9
