## Summary

Keep open. The PR addresses a source-reproducible Matrix MAS/UIA bootstrap failure, but it is not merge-ready because the patch trims an opaque Matrix password before submitting password UIA and does not document the new MAS reset recovery workflow.

## What This Changes

The PR updates Matrix cross-signing bootstrap to detect MAS, route UIA by advertised stages with session tokens, surface MAS reset URLs, add regression tests, and add a Matrix changelog entry.

## Best Possible Solution

Land the MAS/UIA implementation after preserving opaque credentials exactly, adding whitespace-password regression coverage, and documenting MAS reset recovery while keeping unrelated OTK and owner-verification issues separate.

## Reproduction Assessment

Yes. The original failure is source-reproducible from current main's UIA callback, which ignores advertised stages and sessions, and the PR regression is source-reproducible from the new password trim before m.login.password submission.

## Solution Assessment

No, not as currently patched. The stage-aware UIA and MAS reset direction is the right narrow fix, but the password must be submitted unchanged and the new operator recovery path should be documented.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the submitted Matrix password:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.ts:217-228`
  - body: The patch assigns `params.password?.trim()` to `password` and then sends that value in the `m.login.password` auth dict. Current main submits the configured password unchanged, and Matrix passwords are opaque credentials, so accounts with intentional leading or trailing whitespace would fail UIA after this change. Use trimming only to decide whether a password is configured, but submit the original `params.password`.
  - confidence: 0.91
- **[P3] Document MAS cross-signing reset recovery:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.ts:232-248`
  - body: This new reset-stage path changes the operator-facing recovery workflow, but the Matrix docs still describe only the old fixed no-auth/dummy/password UIA fallback and do not tell MAS users to open the reset URL and rerun bootstrap inside the approval window. Please update `docs/channels/matrix.md` alongside this behavior change.
  - confidence: 0.78
