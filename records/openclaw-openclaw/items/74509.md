## Summary

Keep open. The PR addresses a source-reproducible Matrix MAS cross-signing bug and is on the right path, but it still trims the password before `m.login.password` UIA and leaves the new MAS recovery workflow undocumented.

## What This Changes

The PR updates Matrix E2EE cross-signing bootstrap to detect MAS auth metadata, route UIA by advertised stages with sessions, throw typed MAS reset/unsupported-stage errors with tests, and add a Matrix changelog entry.

## Best Possible Solution

Land the MAS/UIA fix after preserving opaque password bytes, adding a whitespace-password regression test, and documenting MAS reset recovery in the Matrix channel docs while keeping unrelated OTK and owner-verification issues separate.

## Reproduction Assessment

Yes. The original failure is source-reproducible from current main's UIA callback, which ignores advertised stages and sessions; the PR regression is source-reproducible from the new trimmed-password submission path.

## Solution Assessment

No, not as currently patched. The stage-aware UIA and MAS reset direction is the narrow maintainable fix, but the password must be submitted unchanged and the new operator recovery path should be documented.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the submitted Matrix password:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.ts:217-228`
  - body: The patch assigns `params.password?.trim()` to `password` and then sends that value in the `m.login.password` auth dict. Current main submits the callback password unchanged, and Matrix passwords are opaque credentials, so callers with intentional leading or trailing whitespace would fail UIA after this change. Use trimming only to decide whether a password is configured, but submit the original `params.password`.
  - confidence: 0.88
- **[P3] Update Matrix verification docs for MAS recovery:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.ts:232-248`
  - body: This new reset-stage path changes the operator-facing recovery workflow, but the Matrix docs still describe only the old fixed no-auth/dummy/password UIA fallback and do not tell MAS users to open the reset URL and rerun bootstrap inside the approval window. Please update `docs/channels/matrix.md` alongside this behavior change.
  - confidence: 0.78
