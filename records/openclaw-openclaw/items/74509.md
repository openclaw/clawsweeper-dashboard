## What This Changes

The PR updates Matrix cross-signing bootstrap to parse UIA challenges, detect MAS auth metadata, surface a MAS reset error/URL, add regression tests, and add a Matrix changelog entry.

## Best Possible Solution

Land a stage-aware Matrix UIA/MAS reset fix after preserving raw credentials, reading stable and unstable OAuth reset URLs, fixing lint/type failures, and documenting operator recovery.

## Reproduction Assessment

Yes, at source level: current main's createSigningKeysUiAuthCallback catches failures and retries dummy/password without reading the 401 UIA body, stages, params, reset URL, or session. The companion issue supplies MAS-fronted Synapse logs; I did not run a live homeserver reproduction.

## Solution Assessment

No, not as currently patched. The stage-aware UIA/MAS reset direction is the narrow maintainable solution, but this branch must preserve raw passwords, read stable m.oauth reset URLs, fix CI/type failures, and add operator docs before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve the submitted Matrix password:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.ts:217-228`
  - body: The callback stores `params.password?.trim()` and then sends that trimmed value in the `m.login.password` auth dict. Matrix passwords are opaque credentials, so keep the original value for submission and use trimming only to decide whether a password was configured.
  - confidence: 0.92
- **[P2] Read the stable m.oauth reset URL:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.ts:62-65`
  - body: `extractMasResetUrl` only reads `params.org.matrix.cross_signing_reset.url`, but Synapse advertises the stable `m.oauth` stage and URL too. A stable-only challenge, or one whose usable URL is only under `params.m.oauth`, would lose the actionable MAS approval link.
  - confidence: 0.9
- **[P2] Remove or use the mismatch helper:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.ts:68-80`
  - body: `isCrossSigningKeyMismatchError` is declared but never called, and the current PR checks are failing on lint/type gates. Delete it or wire it into the intended fallback path before merge.
  - confidence: 0.99
- **[P2] Type the bootstrap mock before reading arguments:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.test.ts:607-611`
  - body: This test defines `bootstrapCrossSigning` as `vi.fn(async () => {})`, so TypeScript infers a zero-argument mock and `bootstrapCrossSigning.mock.calls[0]?.[0]` fails test type checking. Give the mock an explicit bootstrap-options signature or reuse the typed mock helper.
  - confidence: 0.97
- **[P3] Document the MAS reset recovery flow:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.ts:232-249`
  - body: The patch adds an operator-visible MAS reset URL and approval-window retry flow, but the Matrix channel docs still describe only no-auth, dummy, and password UIA. Update the docs so operators know to open the MAS URL and rerun bootstrap inside the approval window.
  - confidence: 0.82
