## Summary

Keep open: this PR targets a real Matrix MAS cross-signing bug, but the current branch still has blocking code and CI issues. Current `main` and v2026.5.7 still use the old UIA fallback, and the same-author companion issue remains open at https://github.com/openclaw/openclaw/issues/74504.

## What This Changes

The PR updates the Matrix plugin cross-signing bootstrap to parse UIA 401 flows, detect MAS auth metadata, surface typed MAS reset errors, add tests, and add a Matrix changelog entry.

## Best Possible Solution

Land this Matrix plugin fix after preserving raw credentials, supporting stable and unstable OAuth/reset stages with URL extraction, fixing CI/test typing, and documenting MAS recovery.

## Reproduction Assessment

Yes, at source level. Current main's `authUploadDeviceSigningKeys` callback can be exercised with a 401 UIA body and no password, and it throws the old password fallback error without inspecting stages, reset URLs, or sessions.

## Solution Assessment

No, not as currently patched. The stage-aware UIA/MAS reset direction is the right narrow fix, but the branch must preserve raw passwords, handle stable `m.oauth`, fix CI, and update operator docs before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Preserve the submitted Matrix password:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.ts:217-228`
  - body: The callback stores `params.password?.trim()` in `password` and then submits that trimmed value in the `m.login.password` auth dict. Password values should be treated as opaque credentials; use trimming only to decide whether a password is configured and pass the original string to UIA.
  - confidence: 0.88
- **[P2] Handle the stable MAS OAuth stage:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.ts:232-237`
  - body: The reset branch only explicitly recognizes `org.matrix.cross_signing_reset` and only extracts URL params from that key, but the Matrix spec defines `m.oauth` for OAuth-authenticated `/keys/device_signing/upload` UIA. Treat `m.oauth` as the same operator reset flow and read `params.m.oauth.url` so stable-only servers surface the advertised reset URL.
  - confidence: 0.9
- **[P2] Remove or use the mismatch helper:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.ts:68-80`
  - body: `isCrossSigningKeyMismatchError` is declared but never called, and the PR's lint job already fails on the unused function. Delete it or wire it into the intended fallback path before merge.
  - confidence: 0.99
- **[P2] Type the bootstrap mock before reading its first argument:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.test.ts:597`
  - body: This test defines `bootstrapCrossSigning` as `vi.fn(async () => {})`, so TypeScript infers a zero-argument mock and `bootstrapCrossSigning.mock.calls[0]?.[0]` fails `check-test-types` with TS2493. Give the mock an explicit bootstrap-options signature or reuse the typed mock helper.
  - confidence: 0.97
- **[P3] Document MAS reset recovery:** `extensions/matrix/src/matrix/sdk/crypto-bootstrap.ts:232-248`
  - body: The new reset-stage path changes the operator recovery workflow, but the Matrix docs still describe only the old no-auth, dummy, and password UIA sequence. Update the Matrix channel docs to explain opening the MAS reset URL, approving the reset, and rerunning bootstrap inside the approval window.
  - confidence: 0.82
