## Summary

Keep open: this is a new security-sensitive exec sandbox feature, not something already present on current main, and the latest PR head still has blocking correctness and supply-chain issues plus no after-fix real behavior proof.

## What This Changes

The PR adds a Linux bubblewrap namespace wrapper for safeBins-approved gateway exec commands, threads `tools.exec.nsSandbox` through exec defaults, adds bwrap tests, and includes unrelated package/UI/test edits.

## Best Possible Solution

Keep the PR open for maintainer security/product review, then require a rebased implementation that validates the config contract, preserves compatible safeBins behavior, removes unrelated package policy churn, and includes real Linux bwrap proof.

## Reproduction Assessment

Yes for the review findings: source inspection shows a normal `tools.exec.nsSandbox` config is rejected by the strict schema, and the bwrap network/env behaviors follow directly from the PR head code. I did not run a live bwrap repro.

## Solution Assessment

No: the feature direction is plausible, but this diff is not yet the narrowest maintainable solution because it ships an unreachable config key, hardcoded sandbox behavior regressions, unrelated install policy changes, and no real runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Wire nsSandbox into the strict config schema:** `src/config/types.tools.ts:253`
  - body: This adds `tools.exec.nsSandbox` to the public `ExecToolConfig`, but the strict runtime config schema still builds `ToolExecSchema` from `ToolExecBaseShape` without `nsSandbox`. A normal config using the documented key will be rejected as an unknown property before the feature can turn on.
  - confidence: 0.96
- **[P2] Preserve network for custom safeBins:** `src/infra/exec-bwrap-sandbox.ts:304-314`
  - body: `processGatewayAllowlist` always supplies `commandBins`, and `shouldShareNetwork` only keeps `--share-net` for a hardcoded set. Any operator-approved custom safeBin that needs network but is not listed here will start failing under bwrap, even though safeBins previously ran with normal host networking.
  - confidence: 0.9
- **[P2] Mount env for script safeBins:** `src/infra/exec-bwrap-sandbox.ts:52-53`
  - body: The sandbox only mounts `sh` and `bash` by default. SafeBin entries can be scripts or package-manager wrappers using `#!/usr/bin/env ...`, and those will fail inside bwrap because `/usr/bin/env` is invisible unless the user also knows to add it as a safeBin or extra bind.
  - confidence: 0.86
- **[P2] Remove unrelated lifecycle allowlist changes:** `package.json:435-439`
  - body: The PR widens pnpm's built-dependency allowlist to `@discordjs/opus` and `@tloncorp/tlon-skill` while removing `koffi`, but the bwrap exec sandbox does not use these packages. That unrelated install-script policy change broadens supply-chain exposure and can change install behavior outside the feature.
  - confidence: 0.88
