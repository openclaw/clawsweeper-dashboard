## Summary

Keep open. Current main still does not implement a `tools.exec.nsSandbox` or bubblewrap path for safeBins-approved gateway exec, and this PR is a plausible but security-sensitive implementation that is not ready to merge as-is.

## What This Changes

The PR adds an optional Linux bubblewrap namespace sandbox for safeBins-approved gateway exec commands, threads `nsSandbox` through exec defaults/runtime, adds bwrap tests, and also touches package/UI/test cleanup outside the core sandbox change.

## Best Possible Solution

Keep this open for a narrowed, rebased implementation: add the bwrap layer only with the full config contract, strict schemas, generated schema/help/docs, focused exec approval/runtime tests, Linux bwrap proof, and secops review; drop unrelated package/UI/test changes unless handled separately.

## Reproduction Assessment

Not applicable as a feature PR. The current-main gap is high-confidence from source search and config/runtime inspection: there is no accepted `nsSandbox` key and no bwrap execution wrapper today.

## Solution Assessment

No, not as currently proposed. The core direction may be reasonable, but the branch needs schema/docs alignment, absolute-shell mounting fixes, and removal of unrelated package lifecycle changes before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Mount the actual selected shell path:** `src/agents/bash-tools.exec-host-gateway.ts:205-213`
  - body: `makeBwrapParams()` passes only the shell basename into the sandbox mount list, but `runExecProcess()` later execs the absolute `getShellConfig().shell` path. When `$SHELL` or PATH resolves to `/usr/local/bin/bash`, a Nix shell, or another non-`/bin`/`/usr/bin` path, bwrap will not mount that path and every sandboxed safeBins command will fail to start.
  - confidence: 0.9
- **[P1] Add nsSandbox to the strict config schema:** `src/config/types.tools.ts:247-253`
  - body: This PR exposes `tools.exec.nsSandbox` in the TypeScript config type and runtime defaults, but the strict zod exec schema is not updated. Users setting the documented key in normal config will get an unrecognized-key validation error instead of enabling bwrap.
  - confidence: 0.93
- **[P2] Remove unrelated build-script allowlist changes:** `package.json:432-433`
  - body: The package diff enables build scripts for additional native packages and removes another entry from `onlyBuiltDependencies`, but that is unrelated to the bwrap exec feature. Mixing install/lifecycle policy changes into this security PR changes supply-chain behavior and should be split or justified separately.
  - confidence: 0.86
