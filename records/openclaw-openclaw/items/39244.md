## Summary

Keep open: current main still lacks the proposed `tools.exec.nsSandbox`/bwrap surface, and this PR is a security-sensitive host exec sandbox with unresolved correctness, supply-chain, mergeability, and real-behavior-proof blockers.

## What This Changes

The branch adds an optional Linux bubblewrap namespace sandbox for safeBins-approved exec commands via `tools.exec.nsSandbox`, threads it through exec config/runtime/tests, and includes unrelated package/UI/test cleanup.

## Best Possible Solution

Rebase or replace this with a focused security-reviewed exec sandbox change that mounts exact planned executable paths, accepts and documents the config contract, avoids unrelated package-policy edits, aligns with the broader sandbox trackers, and includes redacted Linux bwrap runtime proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Mount the exact planned executable paths:** `src/infra/exec-bwrap-sandbox.ts:232-235`
  - body: `buildBwrapArgs()` re-resolves configured safe-bin names from trusted dirs, but the enforced command executes the already-planned resolved path. On common `/bin` -> `/usr/bin` layouts this can mount `/bin/head` while the command runs `/usr/bin/head`, so sandboxed safeBins fail with command-not-found; pass the exact planned executable paths into the mount list.
  - confidence: 0.9
- **[P1] Mount the actual runtime shell path:** `src/agents/bash-tools.exec-host-gateway.ts:207-219`
  - body: `makeBwrapParams()` passes only the selected shell basename, while `runExecProcess()` executes the full shell path after `--`. Users with `/usr/local/bin/bash`, Nix shells, or any configured shell outside the trusted dirs can have every sandboxed command fail before it starts.
  - confidence: 0.91
- **[P1] Accept nsSandbox in the strict config schema:** `src/config/types.tools.ts:253-257`
  - body: This adds `tools.exec.nsSandbox` to the public exec config type and runtime defaults, but the strict zod schema still omits the key. A user setting the documented config will get an unrecognized-key validation error instead of enabling bwrap.
  - confidence: 0.93
- **[P2] Preserve network for unclassified safe-bin profiles:** `src/infra/exec-bwrap-sandbox.ts:304-313`
  - body: The gateway always supplies `commandBins`, and `shouldShareNetwork()` disables networking whenever none of those names are in the hardcoded list. Any custom profiled safeBin that legitimately needs network but is not listed will silently lose network under bwrap, contrary to the PR's shared-network behavior description.
  - confidence: 0.78
- **[P2] Split out install lifecycle policy changes:** `package.json:435-439`
  - body: The package diff changes `onlyBuiltDependencies`, which controls install-time build-script execution and is unrelated to safeBins bwrap isolation. Drop this from the branch or split it into a separately reviewed package-policy change after rebasing.
  - confidence: 0.86
