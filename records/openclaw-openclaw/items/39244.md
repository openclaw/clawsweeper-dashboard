Full review comments:

- **[P1] Add nsSandbox to the strict config schema:** `src/config/types.tools.ts:253-258`
  - body: The PR documents `tools.exec.nsSandbox` and adds it to `ExecToolConfig`, but `ToolExecBaseShape` remains strict and does not include this key. A user following the PR body's config example will get an unrecognized-key validation failure instead of enabling bwrap.
  - confidence: 0.95
- **[P1] Mount the exact shell path that runtime executes:** `src/agents/bash-tools.exec-host-gateway.ts:208-218`
  - body: This passes only the shell basename into bwrap planning, but `runExecProcess` later execs the full `getShellConfig().shell` path. If the selected shell lives outside the trusted dirs or at a different absolute path, bwrap starts and then fails to exec the shell before any safeBins command runs.
  - confidence: 0.88
- **[P1] Block broad ancestor extra binds:** `src/infra/exec-bwrap-sandbox.ts:371-375`
  - body: `assertAllowedExtraBind` rejects paths inside a few sensitive locations, but it still allows broad ancestors like `/`, `/home`, or `~`; those binds can re-expose the denied children inside the namespace and defeat the advertised filesystem boundary.
  - confidence: 0.86
- **[P2] Handle shebang interpreters for script safe bins:** `src/infra/exec-bwrap-sandbox.ts:52-53`
  - body: Only `sh` and `bash` are guaranteed in the namespace, while custom safe-bin scripts can use shebangs such as `/usr/bin/env node` or another interpreter. Mount declared interpreters safely or reject script safe bins that cannot run inside the namespace.
  - confidence: 0.8
- **[P2] Drop unrelated lifecycle allowlist changes:** `package.json:435-439`
  - body: This sandbox branch changes package install lifecycle policy for unrelated packages, while current main controls that policy through `pnpm-workspace.yaml` and docs keep `@discordjs/opus` disabled in default lanes. Keep supply-chain policy changes out of the exec sandbox PR.
  - confidence: 0.84
