## Summary

Keep open. Current main does not implement the bwrap namespace sandbox, and the PR is a meaningful security hardening direction, but the branch is not landable as-is because it has correctness bugs in the bwrap mount plan, misses the strict config schema, and mixes unrelated install policy changes into a security-sensitive exec change.

## What This Changes

This PR adds a new bwrap exec sandbox module, threads `tools.exec.nsSandbox` through safeBins gateway/runtime handling, adds bwrap-focused tests, and also changes package lifecycle, browser-test, and UI files outside the sandbox path.

## Best Possible Solution

Rebase and narrow the work into a focused, security-reviewed exec sandbox change that carries exact execution paths into the namespace, accepts and documents the full config contract, updates generated schema/help/docs/changelog, proves Linux bwrap behavior with focused tests, and splits unrelated package/UI/test cleanup into separate work.

## Reproduction Assessment

Not applicable as a feature PR. The blocking issues are reproducible by static inspection: the PR adds `tools.exec.nsSandbox` to TypeScript/runtime paths while the strict schema omits it, and the bwrap argv mounts names while runtime execution uses exact absolute paths.

## Solution Assessment

No. A bwrap layer may be a maintainable direction for safeBins defense in depth, but this branch is not the best implementation until the mount plan, schema/docs/changelog, stale diff surface, and security review are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mount the enforced safe-bin paths inside bwrap:** `src/infra/exec-bwrap-sandbox.ts:232-235`
  - body: `buildBwrapArgs()` re-resolves safe-bin names from trusted dirs, but the enforced command uses the planned `resolvedRealPath`/`resolvedPath`. On systems where PATH resolves `/usr/bin/head` while this loop binds `/bin/head`, the sandboxed command can fail with command-not-found. Pass and mount the exact execution paths used by the enforced command.
  - confidence: 0.9
- **[P1] Mount the actual runtime shell path:** `src/agents/bash-tools.exec-host-gateway.ts:207-219`
  - body: `makeBwrapParams()` only passes the shell basename into the mount list, but the runtime later execs the absolute `getShellConfig().shell` path after `--`. Users with `/usr/local/bin/bash`, Nix shells, or another shell outside the trusted dirs can have every sandboxed safeBins command fail to start.
  - confidence: 0.91
- **[P1] Add nsSandbox to the strict config schema:** `src/config/types.tools.ts:253-257`
  - body: This exposes `tools.exec.nsSandbox` in `ExecToolConfig` and threads it into runtime defaults, but `ToolExecBaseShape` in the strict zod config schema still omits the key. A user setting the documented config would get an unrecognized-key validation error instead of enabling bwrap.
  - confidence: 0.93
- **[P2] Preserve network for unclassified safe bins:** `src/infra/exec-bwrap-sandbox.ts:304-313`
  - body: The gateway always supplies `commandBins`, and `shouldShareNetwork()` disables networking whenever none of those names are in the hardcoded list. Any configured safe-bin profile that legitimately needs network but is not listed here will silently lose network under bwrap, contrary to the PR's default `--share-net` behavior.
  - confidence: 0.78
- **[P2] Split out package build-script allowlist changes:** `package.json:435-440`
  - body: The package diff changes `onlyBuiltDependencies`, which changes install-time code execution policy and is unrelated to the bwrap exec sandbox. Drop it from this security PR or split it into a separately reviewed package-policy change after rebasing.
  - confidence: 0.86
- **[P3] Add the required changelog and docs:** `src/config/types.tools.ts:253-257`
  - body: This adds a user-facing `tools.exec.nsSandbox` feature, but the branch does not update `CHANGELOG.md` or the public exec/safeBins docs. Add the release note and config/docs coverage so users can discover and safely operate the new mode.
  - confidence: 0.82
