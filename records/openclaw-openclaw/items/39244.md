## Summary

Keep open: current main does not implement bwrap namespace sandboxing, but this branch is a security-sensitive host exec change with blocking correctness and supply-chain review issues, so it needs maintainer/security review rather than cleanup closure.

## What This Changes

This PR adds a Linux bubblewrap namespace sandbox mode for safeBins exec commands, threads `tools.exec.nsSandbox` through gateway/runtime defaults, adds bwrap tests, and also changes package lifecycle allowlists plus unrelated browser/UI cleanup.

## Best Possible Solution

Ship a narrowly scoped exec sandbox implementation only after core maintainers approve the threat model, with exact path mounting, complete config/schema/docs/changelog coverage, Linux smoke proof, and no unrelated package or UI changes.

## Reproduction Assessment

Not applicable as a feature PR. The blocking review findings are source-reproducible from the PR diff: the public config key is missing from the strict schema, and the bwrap mount plan does not carry the exact runtime execution paths.

## Solution Assessment

No. A bwrap layer may be a reasonable defense-in-depth direction, but this branch is not the best implementation until the mount plan, schema/docs/changelog, stale diff surface, and security review are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mount the enforced safe-bin paths inside bwrap:** `src/infra/exec-bwrap-sandbox.ts:232-235`
  - body: `buildBwrapArgs()` re-resolves safe-bin names from trusted dirs, but the enforced command runs the resolved absolute execution paths from allowlist evaluation. On systems where the command resolves to `/usr/bin/head` while this loop binds `/bin/head`, sandboxed safeBins can fail with command-not-found; pass and mount the exact execution paths used for this run.
  - confidence: 0.9
- **[P1] Mount the actual runtime shell path:** `src/agents/bash-tools.exec-host-gateway.ts:207-219`
  - body: `makeBwrapParams()` passes only the shell basename into the mount list, while `runExecProcess()` later executes the configured shell path after `--`. Users with `/usr/local/bin/bash`, Nix shells, or another shell outside the trusted dirs can have every sandboxed safeBins command fail to start.
  - confidence: 0.91
- **[P1] Add nsSandbox to the strict config schema:** `src/config/types.tools.ts:253-257`
  - body: This exposes `tools.exec.nsSandbox` in the public config type and threads it into runtime defaults, but the strict zod config schema still omits the key. A user setting the documented config would get an unrecognized-key validation error instead of enabling bwrap.
  - confidence: 0.93
- **[P2] Preserve network for unclassified safe bins:** `src/infra/exec-bwrap-sandbox.ts:304-313`
  - body: The gateway always supplies `commandBins`, and `shouldShareNetwork()` disables networking whenever none of those names are in the hardcoded list. A configured safe-bin profile that legitimately needs network but is not listed here will silently lose network under bwrap, contrary to the PR's stated shared-network default.
  - confidence: 0.78
- **[P2] Split out package build-script allowlist changes:** `package.json:435-439`
  - body: The package diff changes `onlyBuiltDependencies`, which changes install-time code execution policy and is unrelated to the bwrap exec sandbox. Drop it from this security PR or split it into a separately reviewed package-policy change after rebasing.
  - confidence: 0.86
- **[P3] Add the required changelog and docs:** `src/config/types.tools.ts:253-257`
  - body: This adds a user-facing `tools.exec.nsSandbox` feature, but the branch does not update `CHANGELOG.md` or the public exec/safeBins docs. Add the release note and config/docs coverage so users can discover and safely operate the new mode.
  - confidence: 0.82
