## Summary

Keep open. Current main still has no `nsSandbox`/bwrap exec wrapper, so the feature is not implemented, but this security-sensitive PR is not mergeable as-is: the proposed config key is not accepted by the strict schema, the bwrap mount list can miss the actual selected shell path, and the branch mixes unrelated package lifecycle changes into host exec isolation work.

## What This Changes

This PR adds an optional Linux bubblewrap namespace sandbox for safeBins-approved gateway exec commands, threads `nsSandbox` through exec config/defaults/runtime, adds bwrap tests, and includes package/UI/test cleanup outside the core sandbox change.

## Best Possible Solution

Rebase and narrow the branch into a focused, security-reviewed core exec sandbox change: add the full config contract, strict/generated schema/help/docs/changelog, mount the exact execution paths used by the runtime, add focused approval/runtime tests plus Linux bwrap proof, and split unrelated package/UI/test cleanup into separate work.

## Reproduction Assessment

Not applicable as a feature PR. The current-main gap is high-confidence from source/schema/runtime inspection, and the blocking review findings are reproducible by reading the PR head code paths rather than needing a live bwrap run.

## Solution Assessment

No. A bwrap layer may be a reasonable direction, but this branch is not the narrowest maintainable implementation until schema/docs/changelog alignment, exact shell path mounting, rebase cleanup, and security review are complete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Mount the exact shell path used by the runtime:** `src/agents/bash-tools.exec-host-gateway.ts:207-219`
  - body: `makeBwrapParams()` only passes the selected shell basename into the mount list, but the bwrap runtime later execs the absolute `getShellConfig().shell` path. If the user's shell is `/usr/local/bin/bash`, a Nix shell, or another path outside the trusted dirs, bwrap will not mount that absolute path and sandboxed safeBins commands fail to start.
  - confidence: 0.91
- **[P1] Add nsSandbox to the strict config schema:** `src/config/types.tools.ts:253-257`
  - body: This exposes `tools.exec.nsSandbox` in the TypeScript config type and threads it into runtime defaults, but the strict zod exec schema still omits the key. Users setting the documented config will get an unrecognized-key validation error instead of enabling bwrap.
  - confidence: 0.93
- **[P2] Remove unrelated lifecycle allowlist changes:** `package.json:435-440`
  - body: The package diff changes `onlyBuiltDependencies` for native packages, which changes install-time code execution policy and is unrelated to the bwrap exec sandbox. Split this out or drop it from the security PR after rebasing against current main.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `src/config/types.tools.ts:253-257`
