## Summary

Keep open: this is a plausible but security-sensitive feature PR, not something already implemented on current main, and the branch still has source-reproducible blockers plus missing real Linux bwrap proof.

## What This Changes

The PR adds optional Linux bubblewrap namespace isolation for safeBins-approved gateway exec commands via `tools.exec.nsSandbox`, threads it through exec config/runtime, adds bwrap tests, and also touches package/UI/test cleanup.

## Best Possible Solution

Rebase or replace this with a focused security-reviewed exec sandbox change that mounts exact planned executable and shell paths, accepts and documents the config contract, avoids unrelated package-policy edits, and includes redacted Linux bwrap proof.

## Reproduction Assessment

Not applicable as a user bug reproduction path because this is a feature PR. The blocking review findings are source-reproducible from PR head plus current exec planning and strict config schema.

## Solution Assessment

No. A bwrap defense-in-depth layer may be a reasonable direction, but this branch is not the best implementation until path mounting, schema/docs coverage, network/shebang behavior, package-policy drift, security review, and real Linux proof are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Mount the exact planned executable paths:** `src/infra/exec-bwrap-sandbox.ts:232-235`
  - body: `buildBwrapArgs()` re-resolves each configured safe-bin name from trusted dirs, but enforced safeBins execution uses the resolved execution path from command analysis. On common `/bin` -> `/usr/bin` layouts this can mount `/bin/head` while the command runs `/usr/bin/head`, so sandboxed safeBins fail inside bwrap; pass the exact planned executable paths into the mount list.
  - confidence: 0.9
- **[P1] Mount the actual runtime shell path:** `src/agents/bash-tools.exec-host-gateway.ts:207-219`
  - body: `makeBwrapParams()` passes only the selected shell basename, while `runExecProcess()` executes the full shell path after `--`. Users with `/usr/local/bin/bash`, Nix shells, or any configured shell outside the trusted dirs can have every sandboxed command fail before it starts.
  - confidence: 0.91
- **[P1] Accept nsSandbox in the strict config schema:** `src/config/types.tools.ts:253-257`
  - body: This adds `tools.exec.nsSandbox` to the public exec config type and runtime defaults, but the strict zod schema still omits the key. A user setting the documented config will get an unrecognized-key validation error instead of enabling bwrap.
  - confidence: 0.93
- **[P2] Mount env for shebang-based safeBins:** `src/infra/exec-bwrap-sandbox.ts:52-53`
  - body: The sandbox’s default shell mount list omits `env`, so a safe-bin script with a `#!/usr/bin/env ...` shebang can fail inside bwrap even when the script itself is mounted. Either mount the interpreter/shebang dependencies for planned executables or reject those safeBins before sandboxing.
  - confidence: 0.82
- **[P2] Preserve network for unclassified safe-bin profiles:** `src/infra/exec-bwrap-sandbox.ts:304-313`
  - body: The gateway always supplies `commandBins`, and `shouldShareNetwork()` disables networking whenever none of those names are in the hardcoded list. Any custom profiled safeBin that legitimately needs network but is not listed will silently lose network under bwrap, contrary to the PR's shared-network behavior description.
  - confidence: 0.78
- **[P2] Split out install lifecycle policy changes:** `package.json:435-439`
  - body: The package diff changes `onlyBuiltDependencies`, which controls install-time build-script execution and is unrelated to safeBins bwrap isolation. Drop this from the branch or split it into a separately reviewed package-policy change after rebasing.
  - confidence: 0.86
