- **[P1] Add nsSandbox to the strict config schema:** `src/config/types.tools.ts:243-256`
  - body: This exposes `tools.exec.nsSandbox` in the public config type, but the strict zod config schema still omits the key. A user setting the documented config would get an unrecognized-key validation error instead of enabling bwrap.
  - confidence: 0.93
- **[P1] Mount the enforced safe-bin paths inside bwrap:** `src/infra/exec-bwrap-sandbox.ts:231-235`
  - body: `buildBwrapArgs()` re-resolves safe-bin names from trusted dirs, but the enforced command runs the resolved absolute execution paths from allowlist evaluation. On systems where those differ, sandboxed safeBins can fail with command-not-found; pass and mount the exact execution paths used for this run.
  - confidence: 0.9
- **[P1] Mount the actual runtime shell path:** `src/agents/bash-tools.exec-host-gateway.ts:207-219`
  - body: `makeBwrapParams()` passes only the shell basename into the mount list, while `runExecProcess()` later executes the configured shell path after `--`. Users with `/usr/local/bin/bash`, Nix shells, or another shell outside the trusted dirs can have sandboxed safeBins fail to start.
  - confidence: 0.91
- **[P2] Reject parent directories that expose denied binds:** `src/infra/exec-bwrap-sandbox.ts:371-375`
  - body: The denylist check only rejects paths inside a denied path, so binding a parent such as `/etc` is still allowed and exposes denied children like `/etc/shadow`. If extra binds are supported, reject ancestors of denied paths too or switch to an explicit allowlist model.
  - confidence: 0.86
- **[P2] Split out package build-script policy changes:** `package.json:435-439`
  - body: The package diff changes `onlyBuiltDependencies`, which affects install-time code execution and is unrelated to the bwrap exec sandbox. Drop it from this security PR or split/rebase it into a separately reviewed package-policy change.
  - confidence: 0.84
- **[P3] Add the required docs and changelog:** `src/config/types.tools.ts:243-256`
  - body: This adds a user-facing `tools.exec.nsSandbox` feature, but the branch does not update the public exec/safeBins docs or `CHANGELOG.md`. Add operator-facing config guidance and a release note before shipping the new mode.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Security review needs attention because the diff introduces host exec namespace isolation, configurable bind mounts, and package lifecycle policy changes.

Concerns:

- **[medium] Extra binds can expose denied sensitive children:** `src/infra/exec-bwrap-sandbox.ts:371`
  - body: The extraBind denylist checks only whether the configured path is inside a denied path; parent mounts such as `/etc` are allowed and can expose denied files. That undermines the stated sensitive-path protection for the sandbox configuration.
  - confidence: 0.86
- **[medium] Unrelated install lifecycle allowlist changes:** `package.json:435`
  - body: The PR changes `onlyBuiltDependencies`, which controls dependency build scripts. Because this affects install-time code execution and is not required for bwrap exec isolation, it should be split or separately reviewed after rebasing onto current main.
