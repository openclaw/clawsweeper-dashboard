## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Add nsSandbox to the strict config schema:** `src/config/types.tools.ts:253-258`
  - body: The PR documents `tools.exec.nsSandbox` and adds it to `ExecToolConfig`, but `ToolExecBaseShape` remains strict and does not include this key. A user following the PR body's config example will get an unrecognized-key validation failure instead of enabling bwrap, so the feature is not usable through normal config files.
  - confidence: 0.95
- **[P1] Mount the shell path that runExecProcess executes:** `src/agents/bash-tools.exec-host-gateway.ts:209-218`
  - body: This only passes the shell basename into bwrap planning, but `runExecProcess` later execs the full `getShellConfig().shell` path. If `SHELL` resolves to something like `/usr/local/bin/zsh`, bwrap mounts a different path or nothing at all, and every sandboxed safeBins command fails before running.
  - confidence: 0.88
- **[P2] Preserve networking for custom safe bins:** `src/infra/exec-bwrap-sandbox.ts:304-313`
  - body: The gateway always supplies `commandBins`, and `shouldShareNetwork()` disables networking unless the bin appears in this hardcoded list. Any configured custom safe bin that legitimately needs network access will silently lose networking under bwrap, so either keep networking as the default or make network needs part of the safe-bin profile contract.
  - confidence: 0.86
- **[P2] Handle shebang interpreters for script safe bins:** `src/infra/exec-bwrap-sandbox.ts:52-53`
  - body: Only the script path itself is mounted for safeBins, while `/usr/bin/env` and the script interpreter are not guaranteed to exist inside the namespace. Custom safe-bin scripts with common shebangs will fail inside bwrap unless the implementation rejects them up front or mounts declared interpreters safely.
  - confidence: 0.78
