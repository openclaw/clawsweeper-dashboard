## Summary

Keep open: current main does not implement this bwrap safeBins sandbox, and the PR still has source-backed correctness findings, missing real behavior proof, and security-sensitive package-policy churn.

## What This Changes

The PR adds an optional Linux bubblewrap namespace sandbox for safeBins-approved exec commands via `tools.exec.nsSandbox`, threads it through exec config/runtime, adds bwrap tests, and also touches unrelated package/UI/test cleanup.

## Best Possible Solution

Rebase or replace this with a focused security-reviewed bwrap sandbox change that mounts exact planned executable paths, accepts and documents the config contract, avoids unrelated package-policy edits, aligns with broader sandbox trackers, and includes redacted Linux bwrap runtime proof.

## Solution Assessment

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Mount exact planned executables:** `src/infra/exec-bwrap-sandbox.ts:232-235`
  - body: `buildBwrapArgs()` re-resolves each safe-bin name from trusted dirs, but the enforced command executes the resolved path from command analysis. On common `/bin` -> `/usr/bin` layouts this can mount one path while the command runs another, so sandboxed safeBins can fail with command-not-found; pass the exact planned executable paths into the mount list.
  - confidence: 0.9
- **[P1] Mount the selected shell path:** `src/agents/bash-tools.exec-host-gateway.ts:207-219`
  - body: `makeBwrapParams()` passes only the shell basename, while `runExecProcess()` executes the full shell path after `--`. Users with `/usr/local/bin/bash`, Nix shells, or any configured shell outside the trusted dirs can have sandboxed commands fail before they start.
  - confidence: 0.91
- **[P1] Wire nsSandbox into strict schema:** `src/config/types.tools.ts:253-257`
  - body: This adds `tools.exec.nsSandbox` to the public exec config type and runtime defaults, but the strict zod schema still omits the key. A user setting the documented config will get an unrecognized-key validation error instead of enabling bwrap.
  - confidence: 0.93
- **[P2] Preserve network for unknown profiled bins:** `src/infra/exec-bwrap-sandbox.ts:304-313`
  - body: The gateway supplies `commandBins`, and `shouldShareNetwork()` disables networking whenever none of those names are in the hardcoded list. Any custom profiled safeBin that legitimately needs network but is not listed will silently lose network under bwrap, contrary to the PR's shared-network behavior description.
  - confidence: 0.78
- **[P2] Remove install lifecycle policy churn:** `package.json:435-439`
  - body: The package diff changes `onlyBuiltDependencies`, which controls install-time build-script execution and is unrelated to safeBins bwrap isolation. Drop this from the branch or split it into a separately reviewed package-policy change after rebasing.
  - confidence: 0.86
