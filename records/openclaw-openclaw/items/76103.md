## Summary

Close as implemented on current main. The v2026.4.29 release regression is covered by main-only custom-provider paths: explicit `models.providers` rows are listed directly, and configured inline models with provider/model `api` resolve before depending on a bundled plugin-id catalog match.

## What This Changes

The issue asks OpenClaw to keep custom `models.providers` provider IDs that reuse bundled `api` adapters listing and resolving their configured models after the 2026.4.29 plugin activation changes.

## Best Possible Solution

Keep the current main behavior and ship it in the next release; a multi-provider Azure Foundry example can be tracked separately if maintainers want more docs coverage.

## Reproduction Assessment

Yes. The issue gives concrete release-to-release commands and config, and current main can be source-verified against the same custom `models.providers` pattern; I did not run a live Azure Foundry credentialed request.

## Solution Assessment

Yes. The maintainable fix is to make explicit provider config authoritative for listing and model resolution, while using plugin ownership only for provider-specific hooks rather than requiring custom IDs to match bundled plugin IDs.
