## Summary

Current main already addresses the reported regression: custom provider IDs can route through the bundled API owner from their configured `api`, provider-filtered model listing keeps explicit `models.providers.*.models` rows, and inline/configured API keys still count as provider auth. The proof is on `main` only; no release tag contains it yet.

## What This Changes

The issue reports that custom provider IDs using bundled API adapters such as `openai-responses`, `openai-completions`, and `anthropic-messages` return no models after upgrading to `2026.4.29`.

## Best Possible Solution

Keep the main implementation that resolves bundled adapter ownership from `models.providers.<id>.api`, preserves configured model rows, and let the next release carry it; track built-in Microsoft Foundry routing separately in #60546.

## Reproduction Assessment

Yes. The report provides a concrete config and command path; current-main verification traces the same `models list --provider <custom-id>` and runtime-provider owner resolution paths, though no live Azure call was run because no relevant env/profile credentials were present.

## Solution Assessment

Yes. The current approach is the narrow maintainable fix because it keeps custom provider IDs stable while using the configured `api` only as the bundled adapter owner hint, without hardcoding Azure Foundry provider IDs in core.
