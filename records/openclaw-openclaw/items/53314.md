## Summary

Keep open: current main still has the descendant-gated announce/retry race that this PR targets, but the proposed patch is not merge-ready because the new pending-descendant path can reintroduce retry-budget consumption for completion-message runs after five minutes and can strand cleanup if finalization rejects. The missing changelog entry is also a small policy blocker for a user-visible subagent delivery fix.

## What This Changes

The PR changes subagent announce cleanup to return a distinct descendant-blocked signal, defer cleanup without incrementing announce retry count, broaden descendant deferral, and update related announce/cleanup tests.

## Best Possible Solution

Revise the PR so descendant-blocked announces remain a distinct lifecycle outcome through the correct per-entry expiry window, completion-message runs keep their 30-minute hard-expiry semantics, finalize errors use the existing warning/reset recovery path, and regression coverage exercises both the count-flip race and stuck-descendant expiry behavior.

## Reproduction Assessment

Yes. A focused reproduction can mock the inner announce flow to see pending descendants, then make the cleanup recheck see zero descendants; current main will treat that as didAnnounce=false and increment retry accounting instead of preserving a descendant-blocked state.

## Solution Assessment

No. The sentinel direction is appropriate, but the current patch is not the best final form until it honors completion hard-expiry in the pending-descendants path, preserves normal finalize recovery, and adds the required changelog entry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor completion hard expiry for pending descendants:** `src/agents/subagent-registry-lifecycle.ts:275`
  - body: The pending-descendants branch defers only while `endedAgo <= ANNOUNCE_EXPIRY_MS`. For `expectsCompletionMessage` runs blocked by descendants after 5 minutes but before the 30-minute hard expiry, this falls back to the old two-check cleanup path; if the second descendant count has already dropped to zero, retries are consumed again and the announce can give up without a delivery attempt. Use the completion hard-expiry window for completion-message entries.
  - confidence: 0.89
- **[P2] Restore cleanup recovery for pending-descendant failures:** `src/agents/subagent-registry-lifecycle.ts:415-417`
  - body: The new `didAnnounce === -1` path catches `finalizeSubagentCleanup` with an empty handler. If persistence, attachment cleanup, or hook emission rejects after `beginSubagentCleanup` set `cleanupHandled`, the run is left handled but incomplete; route this through the same warning/reset logic as normal finalize failures.
  - confidence: 0.82
- **[P3] Add the required changelog entry:** `src/agents/subagent-announce.ts:375-379`
  - body: This changes user-visible subagent completion delivery behavior, but the diff does not update `CHANGELOG.md`. OpenClaw policy requires user-facing fixes to add an active-version changelog entry with appropriate contributor credit before merge.
  - confidence: 0.76
