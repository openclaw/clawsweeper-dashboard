## Summary

Keep open: the underlying subagent announce race still appears present on current main, but this PR is not merge-ready. The patch goes in the right direction by separating descendant-blocked cleanup from delivery failure, yet the latest diff still has lifecycle bugs around completion hard-expiry handling and finalize error recovery.

## What This Changes

The PR changes subagent announce cleanup to return a distinct descendant-blocked signal, defer cleanup without incrementing announce retry count, broaden descendant deferral, and update related announce/cleanup tests.

## Best Possible Solution

Keep the PR open, but revise the lifecycle contract so descendant-blocked announces remain distinct from delivery failures through the appropriate expiry window, completion-message runs keep their 30-minute hard-expiry semantics, finalize errors use the same recovery path as normal cleanup, and regression tests cover both the count-flip race and stuck-descendant expiry.

## Reproduction Assessment

Yes. A focused reproduction can mock countPendingDescendantRuns to return pending descendants inside runSubagentAnnounceFlow and then zero inside finalize cleanup, which drives current main from blocked announce into retry accounting; the PR body also reports manual T1 -> T2 -> T3 confirmation.

## Solution Assessment

No. The proposed direction is appropriate, but this patch is not the best final form until the pending-descendants branch uses completion hard-expiry for completion-message runs and preserves normal finalize error recovery instead of swallowing rejections.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor completion hard expiry for pending descendants:** `src/agents/subagent-registry-lifecycle.ts:275`
  - body: The pending-descendants fast path only defers while `endedAgo <= ANNOUNCE_EXPIRY_MS`. For `expectsCompletionMessage` runs that are still blocked by descendants between 5 and 30 minutes, this falls back to the old two-check retry path and can consume retries/give up without a delivery attempt. Use the completion hard-expiry window for completion-message entries.
  - confidence: 0.89
- **[P2] Restore finalize recovery for pending-descendant errors:** `src/agents/subagent-registry-lifecycle.ts:415-417`
  - body: The new `didAnnounce === -1` branch catches `finalizeSubagentCleanup` with an empty handler, unlike the normal finalize path that logs and clears `cleanupHandled` so the run can retry. If persistence or give-up cleanup rejects after `beginSubagentCleanup`, this can leave cleanup stalled instead of recoverable.
  - confidence: 0.82
