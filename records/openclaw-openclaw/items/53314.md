## Summary

Keep this PR open: it targets a real current-main nested subagent announce-loss bug, but the patch is not merge-ready because the pending-descendant path still mishandles completion hard expiry, swallows finalize recovery errors, and lacks the required changelog entry.

## What This Changes

The PR changes subagent announce cleanup to signal pending descendants separately from delivery failure, defer without consuming announce retries, broaden descendant deferral, and update focused tests.

## Best Possible Solution

Preserve a typed descendant-blocked lifecycle outcome through cleanup with the completion-message hard-expiry window, normal finalize-error recovery, focused regression coverage, and an active-version changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: make the inner announce descendant gate return blocked while a later cleanup recheck sees zero descendants, and current main can count that as an announce failure retry instead of a pending-descendant defer.

## Solution Assessment

No. The sentinel direction is reasonable, but this PR is not the best final fix until it honors the completion hard-expiry window, preserves finalize-error recovery, and adds the required changelog entry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor completion hard expiry for pending descendants:** `src/agents/subagent-registry-lifecycle.ts:275`
  - body: The pending-descendants branch defers only while `endedAgo <= ANNOUNCE_EXPIRY_MS`. For completion-message runs blocked by descendants after 5 minutes but before the 30-minute hard expiry, this falls back to the old recheck path and can consume retries without a delivery attempt. Use the completion hard-expiry window for completion-message entries.
  - confidence: 0.89

  - confidence: 0.82
