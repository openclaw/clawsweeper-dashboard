## Summary

Keep open: current main and the latest release still lack the post-turn memory-flush fallback, but this PR is not merge-ready because the fallback guard is too broad and the supplied proof does not show the patched MiniMax/WebChat memory-flush path populating totalTokens.

## What This Changes

The PR adds an estimatePromptTokensFromSessionTranscript fallback in runMemoryFlushIfNeeded when the session log usage snapshot has no prompt-token count.

## Best Possible Solution

Reuse the estimator only on the intended missing-usage transcript-read path, preserve byte-size-only forced-flush behavior, add focused regression coverage, and require redacted after-fix MiniMax/WebChat proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.8 still leave the memory-flush token path dependent on transcript usage, so a no-usage transcript leaves tokenCountForFlush undefined unless another fresh persisted total exists.

## Solution Assessment

No as-is. Reusing estimatePromptTokensFromSessionTranscript is the right direction, but the patch should guard it with shouldReadTranscript or an equivalent usage-read condition so byte-size-only forced-flush snapshots do not become token estimates.
