## Summary

Keep open: this PR targets a real memory-flush missing-usage bug, but the fallback guard is still too broad and the PR body does not yet prove the patched MiniMax/WebChat memory-flush path in a real run.

## What This Changes

The branch adds an estimatePromptTokensFromSessionTranscript fallback in runMemoryFlushIfNeeded when the session-log usage snapshot has no prompt-token count.

## Best Possible Solution

Limit the estimator fallback to the intended missing-usage transcript-read path, preserve byte-size-only forced-flush behavior, add focused regression coverage, and require redacted after-fix MiniMax/WebChat proof before merge.

## Reproduction Assessment

Yes, from source inspection. Current main and v2026.6.8 still leave the memory-flush token path dependent on transcript usage, so a no-usage transcript leaves tokenCountForFlush undefined unless another fresh persisted total exists.

## Solution Assessment

No as-is. Reusing estimatePromptTokensFromSessionTranscript is the right direction, but the patch should guard it with shouldReadTranscript or an equivalent usage-read condition so byte-size-only forced-flush snapshots do not become token estimates.
