## Summary

Keep open: the PR targets a real memory-flush missing-usage bug that current main and v2026.6.8 still have, but the current patch is over-broad and still lacks after-fix MiniMax/WebChat proof from the patched path.

## What This Changes

The PR adds a transcript-token estimator fallback in `runMemoryFlushIfNeeded` when the session log usage snapshot has no prompt-token count.

## Best Possible Solution

Keep the estimator reuse but narrow it to the intended transcript-usage path, preserve byte-size-only forced-flush behavior, add focused regression coverage, and require redacted after-fix runtime proof before merge.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.8 still leave the memory-flush token path dependent on transcript `usage`, so a no-usage transcript leaves `tokenCountForFlush` undefined unless another fresh persisted total exists.

## Solution Assessment

No. Reusing the estimator is the right direction, but the patch should only run it when transcript usage was requested, not when a byte-size-only snapshot has no usage by design.
