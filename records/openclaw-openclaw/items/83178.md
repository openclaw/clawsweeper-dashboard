## Summary

Keep open, but this PR is not merge-ready: current main and v2026.6.6 still lack the memory-flush estimator fallback, while the submitted guard also runs on byte-size-only transcript snapshots and can change persisted token state or flush decisions outside the intended stale-usage path.

## What This Changes

The PR modifies `runMemoryFlushIfNeeded` to call `estimatePromptTokensFromSessionTranscript` when the session log snapshot has no prompt-token usage and then uses that estimate for `totalTokens` and flush gating.

## Best Possible Solution

Keep the helper reuse, but narrow the fallback to the stale/unknown or near-threshold transcript-usage path, preserve byte-size-only forced-flush behavior, add focused regression coverage, and include redacted after-fix runtime proof.

## Reproduction Assessment

Yes, source-reproducible: current main and v2026.6.6 still read `sessionLogSnapshot?.usage` directly in `runMemoryFlushIfNeeded`, so a session with stale or missing `totalTokens` and no transcript usage leaves `tokenCountForFlush` undefined. I did not run a live MiniMax/WebChat reproduction in this read-only review.

## Solution Assessment

No. Reusing `estimatePromptTokensFromSessionTranscript` is the right direction, but this patch should only run that fallback when transcript usage was intentionally requested, not when a byte-size-only snapshot has no usage by design.
