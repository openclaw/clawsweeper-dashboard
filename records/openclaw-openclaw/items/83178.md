## Summary

Keep open: this PR targets a real source-reproducible memory-flush token-accounting bug that is still present on current main and v2026.6.9, but the patch is not merge-ready because the fallback can run for byte-size-only transcript reads and the PR body still lacks after-fix runtime proof.

## What This Changes

The branch adds a transcript-estimator fallback in `runMemoryFlushIfNeeded` so session `totalTokens` can be populated when transcript usage prompt tokens are absent.

## Best Possible Solution

Gate the fallback on `shouldReadTranscript` or an equivalent usage-read condition, add focused missing-usage and byte-size-only regression coverage, then require redacted after-fix runtime output before merge.

## Reproduction Assessment

Yes, by source inspection. Current main's memory-flush path reads prompt tokens only from `sessionLogSnapshot.usage`, so a no-usage transcript leaves the flush token count unavailable unless another fresh persisted total exists.

## Solution Assessment

No as-is. Reusing `estimatePromptTokensFromSessionTranscript` is the right direction, but the patch should guard it to usage-read paths so byte-size-only forced-flush snapshots do not become token estimates.
