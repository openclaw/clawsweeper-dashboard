## Summary

Keep open. The underlying MiniMax/session-state bug is source-reproducible on current main, but this PR over-matches byte-size-only snapshots and still lacks convincing after-fix real-provider proof, so it should not merge as-is.

## What This Changes

The PR adds a memory-flush fallback that estimates transcript prompt/output tokens when transcript usage data is unavailable.

## Best Possible Solution

Keep the helper reuse, guard the fallback to stale/near-threshold transcript-usage reads, add focused regression coverage for missing usage and byte-size-only snapshots, and require redacted MiniMax/WebChat after-fix output.

## Reproduction Assessment

Yes, source-reproducible: with a non-CLI memory-flush session whose persisted tokens are stale or unknown and whose transcript has no usage record, current main leaves `transcriptPromptTokens` undefined and cannot persist `totalTokens` or token-trigger the flush. I did not run a live MiniMax reproduction.

## Solution Assessment

No. Reusing `estimatePromptTokensFromSessionTranscript` is the right direction, but this PR should only invoke it when transcript usage was intentionally requested, not when usage is absent by design for a byte-size-only snapshot.
