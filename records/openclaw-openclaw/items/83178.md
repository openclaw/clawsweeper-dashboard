## Summary

Keep open: current main and v2026.6.8 still lack the memory-flush estimator fallback, but this PR is not merge-ready because its guard also fires for byte-size-only transcript snapshots and the PR still lacks after-fix real behavior proof.

## What This Changes

The PR adds a fallback call to estimate prompt/output tokens from the session transcript in runMemoryFlushIfNeeded when transcript usage data has no prompt-token count.

## Best Possible Solution

Reuse the estimator only when the memory-flush path intended to read transcript usage, preserve byte-size-only forced-flush behavior, add focused regression coverage, and require redacted after-fix runtime proof before merge.

## Reproduction Assessment

Yes by source inspection: current main’s memory-flush path reads transcript prompt tokens only from sessionLogSnapshot.usage, so a no-usage transcript leaves the flush token count unavailable unless another fresh persisted total exists. I did not run a live MiniMax/WebChat reproduction in this read-only review.

## Solution Assessment

No as-is. Reusing estimatePromptTokensFromSessionTranscript is the right direction, but the patch should guard it with shouldReadTranscript or an equivalent usage-read condition so byte-size-only forced-flush snapshots do not become token estimates.
