## Summary

Keep open. The reported bug is plausible from the current memory-flush source path, but this PR over-broadens the fallback so it can run during byte-size-only transcript checks, and the PR still needs real after-fix behavior proof before merge.

## What This Changes

Adds a memory-flush fallback in `src/auto-reply/reply/agent-runner-memory.ts` to estimate transcript prompt/output tokens when transcript usage data is unavailable.

## Best Possible Solution

Keep the helper reuse, but gate it to the transcript-usage path, add focused regression coverage for missing usage and byte-size-only snapshots, and require redacted real-provider proof before merge.

## Reproduction Assessment

Yes, from source inspection: with a non-CLI memory-flush session whose persisted tokens are stale or unknown and whose transcript has no usage record, current main leaves `transcriptPromptTokens` undefined and cannot token-trigger the flush. I did not run a live MiniMax reproduction.

## Solution Assessment

No. Reusing `estimatePromptTokensFromSessionTranscript` is the right direction, but the current PR should only invoke it when transcript usage was intentionally requested, not during byte-size-only snapshots.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate the fallback on transcript usage reads:** `src/auto-reply/reply/agent-runner-memory.ts:815-818`
  - body: This condition fires whenever `sessionLogSnapshot` exists. The memory-flush path also creates that snapshot for the default `forceFlushTranscriptBytes` byte-size check with `includeUsage: shouldReadTranscript`, so when `shouldReadTranscript` is false usage is absent by design; this patch then re-reads/estimates the transcript and can persist raw-transcript estimates or trigger a memory flush that the fresh token snapshot would have skipped. Please guard the fallback with `shouldReadTranscript` or the equivalent stale/near-threshold condition.
  - confidence: 0.9
