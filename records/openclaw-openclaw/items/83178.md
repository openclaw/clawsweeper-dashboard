## Summary

Keep this PR open: the linked memory-flush bug is still present on current main and latest release, but this patch is not merge-ready because the fallback condition is too broad and after-fix real behavior proof is still insufficient.

## What This Changes

The branch adds a session-transcript token-estimation fallback in `runMemoryFlushIfNeeded` when transcript usage lacks prompt-token data.

## Best Possible Solution

Reuse the estimator only when the memory-flush path intended to read transcript usage, preserve transcript-size-only forced-flush behavior, add focused regression coverage, and require redacted after-fix runtime proof before merge.

## Reproduction Assessment

Yes by source inspection: current main’s memory-flush path reads transcript prompt tokens only from `sessionLogSnapshot.usage`, so a no-usage transcript leaves the flush token count unavailable unless another fresh persisted total exists. I did not run a live MiniMax/WebChat reproduction in this read-only review.

## Solution Assessment

No as-is. Reusing `estimatePromptTokensFromSessionTranscript` is the right direction, but the patch should guard it with `shouldReadTranscript` or an equivalent usage-read condition so byte-size-only forced-flush snapshots do not become token estimates.
