## Summary

Keep open: the linked memory-flush bug still exists on current main and the latest release, but this PR is not merge-ready because its fallback is broader than the intended missing-usage path and the contributor proof still does not show an after-fix MiniMax/WebChat run populating totalTokens.

## What This Changes

The PR adds an estimated transcript-token fallback in `runMemoryFlushIfNeeded` when provider usage data is unavailable.

## Best Possible Solution

Keep the helper reuse, but gate it to the stale/unknown or near-threshold transcript-usage path, preserve byte-size-only forced-flush behavior, add focused regression coverage, and include redacted after-fix runtime proof.

## Reproduction Assessment

Yes. Source inspection shows current main and v2026.6.8 still read `sessionLogSnapshot?.usage` directly in `runMemoryFlushIfNeeded`, so a session with stale or missing `totalTokens` and no transcript usage leaves `tokenCountForFlush` undefined; I did not run a live MiniMax/WebChat repro.

## Solution Assessment

No. Reusing `estimatePromptTokensFromSessionTranscript` is the right direction, but the current patch should only run that fallback when transcript usage was intentionally requested, not when a byte-size-only snapshot has no usage by design.
