## Summary

## What This Changes

The PR wraps gateway `config.set`, `config.patch`, and `config.apply` writes in a serialized command lane and adds race tests for stale concurrent `config.patch` and `config.apply` writes.

## Best Possible Solution

Land a current-main fix that performs gateway config RPC read, validation, merge, base-hash checking, and commit inside one serialized mutation path while preserving current SecretRef, shared-auth, no-op, persisted-response, and restart semantics.

## Reproduction Assessment

Yes, at source level: current main still reads and computes gateway config mutations before the commit helper, and the helper uses that caller-computed config. I did not run a live gateway repro in this read-only review.

## Solution Assessment

No, not as-is. Serializing the full mutation is the right direction, but this PR needs a current-main rework and expected command-lane rejection must become a structured RPC error before merge.
