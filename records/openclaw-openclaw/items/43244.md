## Summary

Keep open: the PR still targets a real gateway RPC config-write race that is not fixed on current main, but the branch is not merge-ready because it needs a rebase, handler-level queue rejection handling, and contributor real behavior proof.

## What This Changes

The branch wraps gateway config.set, config.patch, and config.apply in a dedicated config-write command lane and adds concurrent stale-write tests for config.patch and config.apply.

## Best Possible Solution

Move config.set, config.patch, and config.apply onto the central serialized config mutation path so read, merge, base-hash validation, and write happen under one mutation lock while preserving current auth, SecretRef, no-op, and restart semantics.

## Reproduction Assessment

Yes, source-level. Current main still computes gateway config replacements from a pre-commit snapshot and writes the precomputed config without carrying the caller base hash into the locked mutation path, so overlapping callers can clobber state.

## Solution Assessment

No, not as-is. The serialization direction is right, but the branch should rebase onto the current mutation helpers and translate expected command-lane rejection into a structured RPC error before it is the narrowest maintainable fix.
