## Summary

Keep this PR open: current main still lacks gateway config-write serialization, but this branch is not mergeable as-is because it wraps older direct write bodies and would bypass newer config write safeguards.

## What This Changes

The PR serializes gateway `config.set`, `config.patch`, and `config.apply` mutations through a dedicated command lane and adds mocked race-condition tests for stale concurrent writes.

## Best Possible Solution

Rebase the serialization so a single config-write lane wraps the current `commitGatewayConfigWrite()` flow, maps lane-level rejection to an RPC error, and proves `config.set`, `config.patch`, and `config.apply` with tests plus redacted real gateway RPC race evidence.

## Reproduction Assessment

Yes, source-backed: current main has no serialized lane around the gateway config read/compute/write sequence, while docs and schemas establish the existing `baseHash` conflict contract. I did not run a live gateway race in this read-only review.

## Solution Assessment

No, not as written. A dedicated config write lane is the right direction, but the branch must wrap the current shared commit flow and return proper RPC errors when the lane rejects.
