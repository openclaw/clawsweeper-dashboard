## Summary

Keep open: this targets a real gateway config lost-update race that is not fully solved on current main, but the branch is conflicting/outdated, lacks real behavior proof, and has a blocking queue-rejection response bug.

## What This Changes

The PR wraps gateway `config.set`, `config.patch`, and `config.apply` handlers in a `config-write` command lane and adds mocked race tests for `config.patch` and `config.apply`.

## Best Possible Solution

Move gateway config RPC writes onto the current central config mutation/transform path, preserve stale `baseHash` rejection under the lock, add race coverage for all three RPCs, and prove the behavior with a real gateway/control RPC run.

## Reproduction Assessment

Yes, source-level. Current main still lets gateway `config.set`, `config.patch`, and `config.apply` compute from a pre-commit snapshot while the commit path can reuse that supplied snapshot, which is enough to show the stale-writer race without a live run.

## Solution Assessment

No. The command-lane wrapper matches the intended serialization direction, but current main now has central config mutation helpers; rebasing these handlers onto that path is safer than reviving a separate queue lane, and queue rejection handling must be fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Return an RPC error when the write lane rejects:** `src/gateway/server-methods/config.ts:314`
  - body: `enqueueCommandInLane` rejects before running the task once the gateway is draining, and `clearCommandLane` can reject queued work as well. These handlers await the enqueue without catching those cases, so a config write arriving during restart/drain can reject the handler without ever calling `respond`, leaving the RPC client waiting instead of receiving an `UNAVAILABLE` error.
  - confidence: 0.88
