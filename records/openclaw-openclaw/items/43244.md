## Summary

Keep open: the gateway RPC config-write race still exists on current main and this PR is still the canonical linked fix candidate, but the branch is stale and currently incorrect because it serializes older direct-write bodies that bypass current SecretRef and shared-auth safeguards.

## What This Changes

The PR wraps gateway `config.set`, `config.patch`, and `config.apply` in a single `config-write` command lane and adds mocked stale concurrent write tests for `config.patch` and `config.apply`.

## Best Possible Solution

Rebase the serialization around the current `commitGatewayConfigWrite` flow, preserve SecretRef/no-op/shared-auth/restart behavior, add race coverage for all three write handlers, and provide redacted real gateway proof.

## Reproduction Assessment

Yes, by source inspection: current main reads a config write snapshot and checks `baseHash` before computing each gateway RPC config mutation, but it has no single lane around the read/compute/write sequence. I did not run a live gateway race in this read-only review.

## Solution Assessment

No, not as written. A dedicated gateway config write lane is the right shape, but this branch must queue the current shared commit path rather than older direct `writeConfigFile()` bodies.
