## Summary

Keep open: this is a real gateway config lost-update bug and the linked implementation is still the canonical fix path, but the PR is not mergeable as-is because real behavior proof is missing, the branch conflicts with current main, and the write-lane rejection bug remains.

## What This Changes

The PR wraps gateway `config.set`, `config.patch`, and `config.apply` writes in a single `config-write` command lane and adds mocked race tests for `config.patch` and `config.apply`.

## Best Possible Solution

Rebase the gateway RPC writers onto the central config mutation/transform path, preserve stale `baseHash` rejection under the lock for all three write RPCs, add race coverage for `config.set`, `config.patch`, and `config.apply`, and provide redacted real gateway/control RPC proof.

## Reproduction Assessment

Yes, source-level. Current main still lets gateway config RPC handlers compute replacement configs from a pre-commit snapshot, which is enough to show the stale-writer lost-update path without a live run.

## Solution Assessment

No. The dedicated lane targets the right race, but this stale branch needs to fit the current central mutation flow and must return an RPC error when the queue rejects before the task starts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86
