## Summary

Keep this PR open. Current main still lacks serialization for gateway config writes, so the intended lost-update fix is not implemented, but the PR is stale against current gateway config/auth behavior and has unresolved review gaps before it can land.

## What This Changes

The PR wraps gateway `config.set`, `config.patch`, and `config.apply` in a dedicated `config-write` command lane and adds a race regression test for stale concurrent config writes.

## Best Possible Solution

Land a current-main implementation that serializes the full gateway RPC read, base-hash check, validation, and write sequence for `config.set`, `config.patch`, and `config.apply`, while preserving SecretRef resolution, shared-auth invalidation, and restart behavior. Add stale concurrent-write coverage for all three methods and keep broader non-gateway writer behavior tracked separately.

## Reproduction Assessment

Yes. The PR body gives a focused reproduction with two gateway config mutations from the same base hash, and current main code confirms the race window because each handler computes from a snapshot before calling an unguarded final write. I did not run tests because this review was required to stay read-only.

## Solution Assessment

No, not as-is. A serialized gateway write lane is a reasonable direction, but this PR needs to be rebased over current main, preserve newer config/auth behavior, add explicit lane-rejection handling, and cover `config.set` before it is the best landing path.
