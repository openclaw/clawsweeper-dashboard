## Summary

Keep this PR open because the underlying gateway config write race is still present on current main, but the branch is not mergeable as written: it predates later auth-sensitive config commit safeguards, lacks real behavior proof, and has review-blocking correctness issues.

## What This Changes

The PR wraps gateway `config.set`, `config.patch`, and `config.apply` in a dedicated `config-write` command lane and adds mocked concurrent-write tests for `config.patch` and `config.apply`.

## Best Possible Solution

Rebase the serialization so the lane wraps the current `commitGatewayConfigWrite()` flow, map lane-level rejection to an RPC error, and add coverage for all three write handlers plus redacted real gateway proof.

## Reproduction Assessment

Yes, source-backed: current main has no single lane around the gateway config read/compute/write sequence, so two overlapping writes with the same `baseHash` can both validate against the same snapshot before either write persists. I did not run a live gateway race in this read-only review.

## Solution Assessment

No, not as written. A dedicated config write lane is the right direction, but this branch must preserve the current shared commit path and return a proper RPC error when the lane rejects.
