## Summary

Keep open: this PR targets a real gateway config lost-update bug, but the branch is not merge-ready because the new lane path can reject without sending an RPC response and the external PR still lacks after-fix real behavior proof.

## What This Changes

The PR wraps gateway `config.set`, `config.patch`, and `config.apply` in a `config-write` command lane and adds mocked stale-write race tests for `config.patch` and `config.apply`.

## Best Possible Solution

Move gateway `config.set`, `config.patch`, and `config.apply` onto the central serialized config mutation path so read/merge/base-hash/write happen under one mutation lock, then add focused race coverage and redacted real RPC proof.

## Reproduction Assessment

Yes, source-level. Current main still computes gateway config replacements from a pre-commit snapshot before the serialized write helper, so two writes from the same base hash can still be reasoned to clobber state without running a live gateway.

## Solution Assessment

No, not as-is. The serialization direction is right, but the branch needs to rebase onto the current mutation helpers and handle command-lane rejection before it is the narrowest maintainable fix.
