## Summary

Keep open: current main still lacks a serialized gateway config write lane, but this stale PR regresses current-main SecretRef/shared-auth commit safeguards and leaves queue rejection paths without RPC responses, so it is useful source work but not mergeable as-is.

## What This Changes

The PR wraps gateway `config.set`, `config.patch`, and `config.apply` RPC handlers in a dedicated `config-write` queue and adds stale concurrent-write regression coverage for `config.patch` and `config.apply`.

## Best Possible Solution

Rebuild the serialization on current main around the existing `commitGatewayConfigWrite` flow, catch lane-level rejections, and cover stale-base races for all three config write RPCs.

## Reproduction Assessment

Yes, by source inspection. Current main checks `baseHash` against the snapshot before computing each gateway config mutation and has no config-specific serialized lane or commit-time recheck; I did not run tests because this was a read-only review.

## Solution Assessment

No, not as-is. A dedicated config write lane is the right shape, but the patch must wrap the current SecretRef, shared-auth, no-op, restart-result, and commit follow-up flow instead of restoring direct writes.
