## Summary

Keep open: the gateway RPC write race is still source-reproducible on current main, but this stale PR branch bypasses newer config/auth write safeguards and needs changes before merge.

## What This Changes

The PR wraps `config.set`, `config.patch`, and `config.apply` in a dedicated `config-write` lane and adds concurrent stale-base race tests for `config.patch` and `config.apply`.

## Best Possible Solution

Rebuild the serialization on current main around `commitGatewayConfigWrite`, preserve auth/config safeguards, catch lane-level rejections, and prove the race with focused runtime evidence plus regression tests.

## Reproduction Assessment

Yes, by source inspection: current main checks `baseHash` before computing each gateway config mutation and has no config-specific serialization, so overlapping writes from the same hash can compute from stale state. I did not run live tests in this read-only pass.

## Solution Assessment

No, not as written. A dedicated write lane is the right shape, but the PR must queue the current shared commit flow and preserve SecretRef, no-op patch, shared-auth, restart-result, and follow-up behavior.
