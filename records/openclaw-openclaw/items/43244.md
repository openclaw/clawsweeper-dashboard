## Summary

Keep this PR open, but not mergeable as written. The gateway RPC write race is still a valid current-main problem, while the PR branch is stale against the current config write flow and would bypass auth/config safeguards if carried forward directly.

## What This Changes

The branch wraps `config.set`, `config.patch`, and `config.apply` in a dedicated `config-write` lane and adds stale concurrent write tests for `config.patch` and `config.apply`.

## Best Possible Solution

Rebase the serialization around the current `commitGatewayConfigWrite` flow, preserve SecretRef/no-op/shared-auth/restart behavior, add focused race coverage for all affected write handlers, and provide redacted runtime proof of the gateway RPC race before merge.

## Reproduction Assessment

Yes, by source inspection: current main still checks `baseHash` before computing each gateway config mutation but does not serialize the read/compute/write sequence, so overlapping writes from the same hash can compute from stale state. I did not run a live gateway race in this read-only review.

## Solution Assessment

No, not as written. A dedicated write lane is the right general shape, but the branch must queue the current shared commit path rather than older direct `writeConfigFile` bodies.
