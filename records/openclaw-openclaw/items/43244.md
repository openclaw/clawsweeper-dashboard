## Summary

Keep open. The underlying gateway RPC config race is still valid on current main, but this stale PR is not mergeable as-is because it queues older handler bodies that bypass current SecretRef/shared-auth/no-op write safeguards and lacks queue-rejection response handling.

## What This Changes

The PR wraps gateway `config.set`, `config.patch`, and `config.apply` in a dedicated `config-write` command lane and adds race regression tests for stale concurrent `config.patch` and `config.apply` writes.

## Best Possible Solution

Rebuild serialization on current main by placing the full existing `config.set`/`config.patch`/`config.apply` read, base-hash check, validation, SecretRef preflight, no-op handling, `commitGatewayConfigWrite()`, restart resolution, and auth follow-up flow inside one config write lane with expected lane rejection handling and race coverage for all three methods.

## Reproduction Assessment

Yes. The focused path is two gateway config writes started from the same `config.get` hash; current main checks the hash before computing each mutation and has no config-specific lane or commit-time recheck. I did not run tests because this review was required to stay read-only.

## Solution Assessment

No, not as-is. A serialized config write lane is the right shape, but this PR should be rebuilt over current main so it preserves newer config/auth safeguards and maps expected queue rejections to normal RPC failures.
