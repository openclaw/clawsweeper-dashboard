## Summary

Keep open: the gateway RPC config race still appears valid on current main, but this PR is not merge-ready because it wraps older direct-write bodies, bypasses current config commit safeguards, lacks queue-rejection RPC handling, and only provides mocked/unit proof.

## What This Changes

This PR serializes gateway `config.set`, `config.patch`, and `config.apply` through a dedicated config-write lane and adds mocked stale-write race tests for `config.patch` and `config.apply`.

## Best Possible Solution

Wrap the current `commitGatewayConfigWrite()` flow in one config-write lane, map lane-level rejection to an RPC error, and prove `config.set`, `config.patch`, and `config.apply` with focused tests plus redacted real gateway RPC race evidence.

## Reproduction Assessment

Yes, source-backed: current main still reads and validates a config snapshot before committing each gateway config write without serializing the full read/compute/write sequence. I did not run a live gateway race in this read-only review.

## Solution Assessment

No, not as written. A dedicated config-write lane is the right direction, but this branch must be rebased around the current shared commit flow and must return proper RPC errors when the lane rejects.
