## Summary

Keep open: the gateway config write race is real and the write-lane direction is useful, but this PR is stale against current main and would bypass current SecretRef/shared-auth commit safeguards if merged as-is. It also lacks after-fix real behavior proof beyond tests.

## What This Changes

The PR wraps `config.set`, `config.patch`, and `config.apply` in a dedicated `config-write` lane and adds race tests for `config.patch` and `config.apply`.

## Best Possible Solution

Rebuild the serialization on current main around the existing `commitGatewayConfigWrite` flow, catch lane-level rejections, and cover stale-base races without weakening auth/config semantics.

## Reproduction Assessment

Yes, by source inspection. Current main checks `baseHash` before computing each gateway config mutation and has no config-specific serialization, so overlapping writes from the same hash can compute from stale state; I did not run live tests in this read-only pass.

## Solution Assessment

No, not as written. A dedicated config write lane is the right shape, but it needs to wrap the current shared commit flow and preserve SecretRef, shared-auth, no-op patch, restart-result, and follow-up behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Handle config lane rejection before returning:** `src/gateway/server-methods/config.ts:314`
  - body: The handlers await `enqueueCommandInLane()` directly. If the lane rejects before the queued body runs, such as during gateway drain, no code calls `respond`, so callers can wait indefinitely; catch expected queue errors and return an unavailable response for all three write handlers.
  - confidence: 0.84
- **[P1] Preserve config.set commit safeguards:** `src/gateway/server-methods/config.ts:326`
  - body: The queued `config.set` body writes with `writeConfigFile()` directly. Current main preflights submitted SecretRefs and commits through `commitGatewayConfigWrite()` so auth generation and follow-up behavior stay consistent; queue that current flow instead.
  - confidence: 0.87
- **[P1] Keep patch and apply on the shared commit flow:** `src/gateway/server-methods/config.ts:418`
  - body: `config.patch` and `config.apply` also write directly from the queued body. That drops current behavior for no-op patches, SecretRef preflight, shared-auth disconnect calculation, and centralized restart write-result handling; wrap the current commit path in the lane.
  - confidence: 0.86
