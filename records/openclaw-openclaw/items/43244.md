## Summary

Keep open. The underlying gateway RPC lost-update bug is still source-reproducible on current main, and this PR targets the right area, but the branch is conflicting, lacks real gateway/control RPC proof, and needs a current-main rework before it is merge-ready.

## What This Changes

The PR wraps gateway `config.set`, `config.patch`, and `config.apply` in a `config-write` command lane and adds mocked race tests for stale concurrent `config.patch` and `config.apply` writes.

## Best Possible Solution

Rework the fix on current main by moving `config.set`, `config.patch`, and `config.apply` onto a serialized path that re-reads inside the mutation section while preserving current SecretRef, auth-disconnect, restart, noop, and response semantics.

## Reproduction Assessment

Yes at source level: current main still computes gateway `config.set`, `config.patch`, and `config.apply` mutations from a pre-commit snapshot, and the commit helper writes that caller-computed config under the mutation lock. I did not run a live gateway RPC race in this read-only review.

## Solution Assessment

No, not as submitted. Serializing the full mutation is the right direction, but this branch must be rebased onto current main, preserve newer config/auth behavior, handle lane rejections, and include real gateway/control RPC proof.
