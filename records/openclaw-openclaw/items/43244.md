## Summary

Keep this PR open, but it is not merge-ready: the gateway config race remains source-backed on current main, while the branch is stale against newer auth-sensitive config write safeguards and still lacks real behavior proof.

## What This Changes

The PR serializes gateway `config.set`, `config.patch`, and `config.apply` through a dedicated config-write lane and adds mocked race tests for `config.patch` and `config.apply`.

## Best Possible Solution

Wrap the current `commitGatewayConfigWrite()` flow in a config-write lane, map lane-level rejection to an RPC error, and prove `config.set`, `config.patch`, and `config.apply` with focused tests plus redacted real gateway RPC race evidence.

## Reproduction Assessment

Yes, source-backed: current main still reads and validates a config snapshot before committing each gateway config write without serializing the full read/compute/write sequence, and the linked issue includes a concrete lost-update repro. I did not run a live gateway race in this read-only review.

## Solution Assessment

No, not as written. The dedicated lane is the right direction, but the branch must wrap the current shared commit flow and return proper RPC errors when the lane rejects.
