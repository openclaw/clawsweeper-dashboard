## Summary

Keep open. The lost-update bug is valid and current main still does not serialize the gateway config RPC write sequence, but this PR is stale against current main and is not safe to merge as-is because it queues older handler bodies that miss newer config/auth safeguards and leaves queue-level rejection handling unresolved.

## What This Changes

The PR wraps gateway `config.set`, `config.patch`, and `config.apply` handlers in a dedicated `config-write` command lane and adds race regression tests for stale concurrent `config.patch` and `config.apply` writes.

## Best Possible Solution

Implement serialization on top of the current main handlers: run the full read, base-hash check, validation, SecretRef preflight, no-op handling, `commitGatewayConfigWrite()`, restart-result resolution, and follow-up queueing inside a single config write lane. Add handler-boundary catch logic for expected lane rejections and regression coverage for stale concurrent `config.set`, `config.patch`, and `config.apply`.

## Reproduction Assessment

Yes. The PR body gives a focused two-write same-base-hash reproduction, current main still reads and validates each gateway config write outside any config-specific lane, and the docs promise base-hash protection against concurrent clobbering; I did not run tests because this review was required to stay read-only.

## Solution Assessment

No, not as-is. A serialized config write lane is the right direction, but the PR should be rebuilt over current main so it preserves newer config/auth safeguards and maps expected queue rejection errors to normal RPC failures.
