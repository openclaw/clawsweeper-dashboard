## Summary

Keep open: the gateway lost-update bug is still source-reproducible on current main, but this stale PR should not merge as-is because it wraps older direct config writes that bypass current SecretRef/shared-auth commit safeguards and it still lacks real behavior proof.

## What This Changes

The PR wraps gateway `config.set`, `config.patch`, and `config.apply` in a `config-write` command lane and adds mocked stale concurrent-write tests for `config.patch` and `config.apply`.

## Best Possible Solution

Rebase the serialization around the current shared config commit path, preserving SecretRef, no-op, shared-auth, restart, and runtime follow-up behavior, then add race coverage for all three handlers with redacted real gateway proof.

## Reproduction Assessment

Yes, by source inspection: current main reads and validates a config snapshot before computing each gateway RPC mutation, and the write path uses that prepared snapshot without a single lane around the read/compute/write sequence. I did not run a live gateway race because this review was read-only.

## Solution Assessment

No, not as written. A dedicated gateway config write lane is a plausible fix shape, but this branch must wrap the current shared commit flow instead of older direct `writeConfigFile()` bodies.
