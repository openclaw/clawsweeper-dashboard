## Summary

This PR still targets a real gateway config race, but it should not merge as-is because the branch is stale against current main, lacks required real behavior proof, and introduces an RPC hang path when the command lane rejects before calling `respond`.

## What This Changes

The PR wraps gateway `config.set`, `config.patch`, and `config.apply` writes in a `config-write` command lane and adds mocked race coverage for `config.patch` and `config.apply`.

## Best Possible Solution

Move the gateway config RPC writes onto the central config mutation/transform path, preserve stale `baseHash` rejection under the lock, add race coverage for all three RPCs, and prove the behavior with a real gateway/control RPC run.

## Reproduction Assessment

Yes, source-level. Current main still lets gateway config RPC handlers compute a replacement from a pre-commit snapshot before the central write path runs, which is enough to show how a stale writer can revive older config state.

## Solution Assessment

No. The serialization direction is useful, but the branch should be rebased onto the current central config mutation helpers and must handle queue or mutation rejection with an RPC error response.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Return an RPC error when the write lane rejects:** `src/gateway/server-methods/config.ts:314`
  - body: `enqueueCommandInLane` can reject before the task runs when the gateway is draining, and queued work can also be rejected when a lane is cleared. These handlers await the enqueue without catching those cases, so a config RPC can fail before `respond` is called and leave the client waiting instead of receiving an `UNAVAILABLE` error.
