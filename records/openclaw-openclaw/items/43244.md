## Summary

Keep open: the underlying race is valid and current main still lacks config-write serialization, but this PR is stale and would regress newer SecretRef, shared-auth, no-op, and restart-handling safeguards while leaving queue rejection unhandled.

## What This Changes

The PR wraps gateway `config.set`, `config.patch`, and `config.apply` in a `config-write` command lane and adds race regression coverage for stale concurrent `config.patch` and `config.apply` writes.

## Best Possible Solution

Rebuild the serialization on current main around the full existing `config.set`/`config.patch`/`config.apply` commit flow, add expected lane-rejection responses, and cover all three methods with stale-base race tests.

## Reproduction Assessment

Yes. Source inspection shows current main checks `baseHash` against a snapshot before computing each gateway config mutation and has no config-specific serialized lane or commit-time recheck; I did not run tests because this review was read-only.

## Solution Assessment

No, not as-is. A dedicated config write lane is the right shape, but the patch needs to be rebuilt over current main so the lane wraps the existing SecretRef, shared-auth, no-op, restart-result, and commit follow-up flow.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle config lane rejection before returning:** `src/gateway/server-methods/config.ts:314`
  - body: The handlers await `enqueueCommandInLane()` without catching queue-level failures. If the gateway is draining or the lane is cleared before the queued body runs, no code calls `respond`, so the RPC caller can wait indefinitely; catch expected lane errors around each enqueue and send an unavailable response.
  - confidence: 0.84
- **[P1] Preserve config.set commit safeguards:** `src/gateway/server-methods/config.ts:326`
  - body: The queued `config.set` body writes the parsed config directly with `writeConfigFile()`. Current main preflights submitted SecretRefs and commits through `commitGatewayConfigWrite()` so auth generation and follow-up behavior stay consistent; queue that current flow instead of restoring direct writes.
  - confidence: 0.86
- **[P1] Keep patch and apply on the shared commit flow:** `src/gateway/server-methods/config.ts:418`
  - body: `config.patch` and `config.apply` also write from the queued body instead of preserving current-main behavior such as no-op patch handling, SecretRef preflight, shared-auth disconnect calculation, and centralized restart write-result handling. Wrap the current shared commit flow in the lane.
  - confidence: 0.83
- **[P3] Cover config.set in the race suite:** `src/gateway/server.config-write-race.test.ts:120`
  - body: The patch queues `config.set` along with `config.patch` and `config.apply`, but the new race suite only exercises patch and apply. Add the same stale-base concurrent-write regression test for `config.set` so the third protected handler cannot regress silently.
  - confidence: 0.78
