## Summary

Keep this PR open: it targets a real gateway config lost-update bug that is still source-reproducible on current main, but the branch is conflicting/stale, lacks after-fix real gateway proof, and has a blocking queue-rejection response bug.

## What This Changes

The PR wraps gateway `config.set`, `config.patch`, and `config.apply` in a `config-write` command lane and adds mocked stale-write race tests for `config.patch` and `config.apply`.

## Best Possible Solution

Rework the gateway write handlers onto the current central config mutation path so the read/merge/base-hash check/write sequence is serialized for all three RPCs, then add focused race coverage and real gateway RPC proof.

## Reproduction Assessment

Yes, source-level. Current main still lets gateway write handlers compute replacement configs from a pre-commit snapshot, and the commit helper does not carry the caller base hash into the locked write path, so overlapping writes from the same base can still clobber state without a live run.

## Solution Assessment

No, not as-is. The serialization direction is right, but this stale branch needs to be rebased onto the current mutation helpers and must handle command-lane rejection before it is a safe fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Return an RPC error when the write lane rejects:** `src/gateway/server-methods/config.ts:314`
  - body: `enqueueCommandInLane` can reject before the task runs when the gateway is draining, and `clearCommandLane` can reject queued work. Since these handlers await the enqueue without catching those cases, a config write arriving during restart/drain can reject the handler without ever calling `respond`, leaving the RPC client waiting instead of receiving an `UNAVAILABLE` error.
  - confidence: 0.88
