## Summary

Keep open: the gateway RPC config-write race is still source-backed on current main, but this PR is stale and incorrect because it queues older direct write bodies that bypass current config commit safeguards and it lacks real behavior proof.

## What This Changes

The PR serializes gateway `config.set`, `config.patch`, and `config.apply` through a `config-write` command lane and adds mocked concurrent-write tests for `config.patch` and `config.apply`.

## Best Possible Solution

Rebase the serialization around the current shared config commit path, preserving SecretRef, no-op, shared-auth, restart, and persisted-result behavior, then add all three race cases plus redacted real gateway proof.

## Reproduction Assessment

Yes, source-backed: current main has no single lane around the gateway config read/compute/write sequence, so concurrent writes using the same `baseHash` can pass validation before either write completes. I did not run a live gateway race in this read-only review.

## Solution Assessment

No, not as written. A dedicated config write lane is a plausible fix, but it must wrap the current `commitGatewayConfigWrite()` flow and handle lane-level rejections instead of restoring older direct `writeConfigFile()` bodies.
