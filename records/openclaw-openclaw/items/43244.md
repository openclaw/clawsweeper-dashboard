## Summary

Keep open: the gateway config lost-update race still appears source-reproducible on current main, but this PR is not mergeable as written because it predates current auth-sensitive config commit flow and lacks real behavior proof.

## What This Changes

The PR wraps gateway `config.set`, `config.patch`, and `config.apply` in a `config-write` command lane and adds mocked stale concurrent-write tests for `config.patch` and `config.apply`.

## Best Possible Solution

Rebase the serialization around the current shared config commit path, preserving SecretRef, no-op, shared-auth, restart, and persisted-result behavior, then add race coverage for all three handlers plus redacted real gateway proof.

## Reproduction Assessment

Yes, source-backed: current main still reads and validates a config snapshot before computing each gateway RPC config mutation, then writes without a single lane around the read/compute/write sequence. I did not run a live gateway race in this read-only review.

## Solution Assessment

No, not as written. A dedicated gateway config write lane is a plausible fix shape, but this branch must wrap the current `commitGatewayConfigWrite()` flow instead of older direct `writeConfigFile()` bodies.
