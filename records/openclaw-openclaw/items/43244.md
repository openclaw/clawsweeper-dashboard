## Summary

Keep open: the gateway config race is still source-reproducible on current main and this PR is the linked implementation path, but the branch cannot merge as-is because it is conflicting, lacks required real behavior proof, and drops RPC responses if the write lane rejects.

## What This Changes

The branch wraps gateway `config.set`, `config.patch`, and `config.apply` in a dedicated `config-write` command lane and adds stale concurrent write tests for `config.patch` and `config.apply`.

## Best Possible Solution

Move gateway `config.set`, `config.patch`, and `config.apply` onto the central serialized config mutation path so read/merge/base-hash/write happen under one mutation lock, then add focused race coverage and redacted real RPC proof.

## Reproduction Assessment

Yes, source-level. Current main still computes gateway config replacements from a pre-commit snapshot and then writes the precomputed config without carrying the caller base hash into the locked mutation path, so two writes from the same base hash can still clobber state.

## Solution Assessment

No, not as-is. The serialization direction is right, but the branch needs to rebase onto the current mutation helpers and handle command-lane rejection before it is the narrowest maintainable fix.
