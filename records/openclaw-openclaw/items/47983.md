## Summary

Keep open. Current main still has the visible Control UI/WebChat attach button wired through an unscoped `document.querySelector('.agent-chat__file-input')?.click()` path, while existing tests only exercise direct hidden-input change events and do not cover the reported visible button-to-picker activation.

## What This Changes

The issue reports that the WebChat/Control UI upload button in OpenClaw 2026.3.13 does nothing instead of opening the file picker and allowing attachments to be sent.

## Best Possible Solution

Keep the existing attachment serialization and Gateway media path intact, but make the chat composer Attach file control target its own file input through a component-scoped ref or label-backed input, add `type="button"`, and add a regression test that clicks the visible control and verifies the hidden input activation path.

## Reproduction Assessment

Unclear. The issue gives concrete Chrome/Windows steps and current main still exposes the same visible-button path, but this read-only review did not run a live browser/native file-picker smoke; a focused view test can cover the visible button invoking its own input.

## Solution Assessment

Yes. A narrow Control UI fix to the button-to-input activation path plus a view-level regression test is the most maintainable solution; downstream attachment serialization and Gateway media handling should stay unchanged for this bug.
