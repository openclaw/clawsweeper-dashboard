## Summary

Keep open: the trusted-local HOME override is a plausible direction, but the current branch still rejects the advertised config key through the strict runtime schema, regresses explicit-HOME mkdir behavior, lacks public docs for the isolation boundary, and has only test-output proof.

## What This Changes

The PR adds a Codex app-server `homeDir` config/env override, passes it to stdio launches as `HOME` while keeping `CODEX_HOME` isolated, updates manifest metadata, and adds auth-bridge coverage.

## Best Possible Solution

Land the trusted-local override only after schema, mkdir selection, docs, focused tests, and redacted live app-server proof all align around the intended isolation boundary.

## Reproduction Assessment

Yes for the blocking patch bug: source inspection shows the PR reads `config.homeDir` while the strict `appServer` schema omits `homeDir`, so the advertised config key is rejected before resolution. I did not run tests because this is a read-only review.

## Solution Assessment

No as written. The explicit override is a plausible narrow shape, but it needs schema support, explicit-HOME mkdir preservation, public docs, focused regression coverage, and live proof before it is the maintainable fix.
