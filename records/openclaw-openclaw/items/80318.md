## Summary

Keep open: the PR is a plausible fix for the linked Codex app-server HOME override request, but the current patch still rejects the advertised config key through the strict runtime schema, regresses explicit HOME mkdir behavior, lacks public docs, and has only test-output proof rather than live app-server proof.

## What This Changes

The PR adds a Codex app-server `homeDir` config/env override so spawned stdio processes can use a real OS `HOME` while keeping `CODEX_HOME` isolated.

## Best Possible Solution

Land a documented trusted-local stdio HOME override only after the runtime schema, explicit-HOME mkdir behavior, tests, public docs, and live redacted app-server proof all match the intended isolation boundary.

## Reproduction Assessment

Yes for the PR-introduced blockers: source inspection of the PR head shows `homeDir` is read by the resolver while the strict app-server zod schema omits it, and the mkdir guard is keyed to `homeDir` instead of the selected HOME source. I did not run tests because this was a read-only review.

## Solution Assessment

No as written. The explicit override is a plausible narrow direction, but it needs schema acceptance, explicit-HOME mkdir preservation, public docs, focused tests, and live proof before it is the maintainable fix.

That regresses the existing explicit-HOME path setup; skip mkdir only when the selected HOME actually came from the trusted `homeDir` override.
  - confidence: 0.84

  - body: The manifest adds a public config/UI field and env fallback, but `docs/plugins/codex-harness.md` still omits `homeDir` and says `HOME` is reserved for per-agent isolation. Users need the trusted-local boundary and fallback documented with the new surface.
  - confidence: 0.79
