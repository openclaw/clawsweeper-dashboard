## Summary

Keep open: the PR targets a real Codex app-server configuration gap, but the current head still rejects the advertised config key in the strict runtime schema, regresses explicit HOME directory creation when `homeDir` is also set, omits the new isolation boundary from docs, and only supplies targeted test output rather than live app-server proof.

## What This Changes

Adds a Codex app-server `appServer.homeDir` config/env override, manifest UI hint, and auth-bridge coverage so stdio launches can set `HOME` separately from isolated `CODEX_HOME`.

## Best Possible Solution

Land a documented trusted-local HOME override only after runtime schema, mkdir behavior, focused tests, and redacted live app-server proof all align while default HOME isolation stays unchanged.

## Reproduction Assessment

Yes for the PR-head blocker: source inspection shows the strict app-server zod schema omits `homeDir` while the resolver reads it, so the advertised config path is rejected. I did not run a live `gh auth status` reproduction in this read-only review.

## Solution Assessment

No as written. The explicit override is a plausible narrow direction, but the schema, explicit-HOME mkdir behavior, public docs, and live proof need to be fixed before this is the maintainable solution.
