## What This Changes

## Best Possible Solution

Land a documented explicit stdio HOME override only after runtime schema, manifest, docs, regression tests, and redacted live app-server/local-auth proof agree on the isolation boundary.

Yes for the patch defect: PR head source shows the resolver reads `homeDir` while the strict `appServer` schema omits it, so the config key is rejected before resolution. ## Solution Assessment

No as written. The explicit override is a plausible narrow direction, but it must accept the config key in the runtime schema, preserve explicit HOME behavior, update public docs, and include live proof before merge.

Full review comments:

- **[P2] Accept homeDir in the runtime schema:** `extensions/codex/src/app-server/config.ts:338-340`
  - body: Adding `homeDir` to the type, manifest, and resolver does not make `plugins.entries.codex.config.appServer.homeDir` work because `readCodexPluginConfig` validates `appServer` with a strict zod object that still lacks this key.   - confidence: 0.94

  - body: When both `startOptions.env.HOME` and `homeDir` are present, explicit `env.HOME` wins, but this condition skips creating that explicit HOME path solely because `homeDir` exists. That regresses the existing explicit-HOME behavior; track whether `nativeHome` actually came from `homeDir` before skipping `mkdir`.
  - confidence: 0.82
- **[P3] Document the app-server HOME override:** `extensions/codex/openclaw.plugin.json:332-335`
  - body: This adds a public `appServer.homeDir` field and env fallback, but the Codex harness docs still omit both and still say `HOME` is reserved for per-agent isolation. Please update the supported `appServer` fields and environment override list so users see the trusted-local boundary and the new knob in the public docs.
