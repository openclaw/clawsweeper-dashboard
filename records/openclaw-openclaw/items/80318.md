## Summary

Keep open: the proposed override is a plausible fix for the linked Codex app-server HOME problem, but the branch still fails its advertised config path because the strict runtime schema rejects `homeDir`, and the PR lacks live real-behavior proof.

## What This Changes

The PR adds a Codex app-server `homeDir` config/env override, manifest UI hint, and auth-bridge test to set spawned `HOME` while keeping `CODEX_HOME` isolated.

## Best Possible Solution

Land a documented opt-in trusted-local HOME override only after the schema accepts it, explicit `HOME` mkdir behavior is preserved, tests cover config and bridge paths, and redacted live app-server proof shows local CLI auth works.

## Reproduction Assessment

Yes for the blocking PR bug: source inspection shows the PR resolver reads `config.homeDir` while the strict app-server schema still omits `homeDir`, so the advertised plugin config key is rejected. I did not run tests because this review is read-only.

## Solution Assessment

No as written. The explicit override is a plausible narrow direction, but it needs schema, mkdir semantics, docs, tests, and live proof aligned before it is the maintainable solution.

  - confidence: 0.82

  - body: This adds a public `appServer.homeDir` field and env fallback, but the Codex harness docs still omit both and still say `HOME` is reserved for per-agent isolation. Users need the trusted-local boundary and env fallback documented on the public Codex harness page.
  - confidence: 0.78
