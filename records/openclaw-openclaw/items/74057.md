Keep this PR open: the direction is still useful for the linked llama.cpp report, but the PR head parses the wrong `/props` response shape and is missing the required changelog entry for a user-facing provider fix.

## Best Possible Solution

Keep the provider-discovery fix boundary, parse documented nested `/props.default_generation_settings.n_ctx` with a compatibility fallback, preserve explicit configured caps, add the changelog entry, and merge only after focused validation and CI are green.

## Reproduction Assessment

Yes. A static reproduction is to feed the documented llama.cpp `/props` payload with `default_generation_settings.n_ctx` into the PR parser; the PR reads only `data.n_ctx`, so `contextTokens` remains unset.

No. The approach is the right maintainable boundary, but this implementation should parse the documented nested response shape, keep a top-level fallback, and include the required changelog entry before merge.
