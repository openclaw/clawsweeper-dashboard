## Summary

Keep open. Current main still lacks the llama.cpp `/props` runtime-context discovery this PR is trying to add, and the PR head is not ready because it reads the wrong `/props` response shape and omits the required user-facing changelog entry.

## What This Changes

The PR adds opportunistic `/props` discovery to self-hosted OpenAI-compatible provider setup, maps `/v1/models` `meta.n_ctx_train` to native `contextWindow`, maps runtime context into `contextTokens`, and extends focused provider setup tests.

## Best Possible Solution

Revise the PR so `/props` parsing prefers `default_generation_settings.n_ctx` while optionally retaining a top-level fallback, update the focused tests to use the documented llama.cpp shape, preserve explicit configured caps ahead of live discovery, and add a single-line Unreleased Fixes changelog entry crediting the reporter/contributor.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available: feed the PR parser a documented llama.cpp `/props` response with `default_generation_settings.n_ctx`, and `runtimeContextTokens` remains undefined because the code reads only `data.n_ctx`.

## Solution Assessment

No. The approach is narrow and compatible with OpenClaw's `contextTokens` contract, but this implementation must parse the documented nested `/props` field and record the user-facing fix in the changelog before it is the best solution.
