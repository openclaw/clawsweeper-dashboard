## Summary

Keep this PR open; it addresses the prior nested `/props` parsing, changelog, and real-proof blockers, but the current diff still applies one `/props` runtime cap to every discovered model, which is unsafe for llama.cpp router or multi-model servers.

## What This Changes

The branch adds self-hosted OpenAI-compatible llama.cpp `/props` discovery, maps nested `/props.default_generation_settings.n_ctx` to `contextTokens`, preserves `/v1/models.meta.n_ctx_train` as `contextWindow`, updates focused provider tests, and adds a changelog entry.

## Best Possible Solution

Keep the provider-discovery approach, but associate llama.cpp runtime caps with the correct discovered model or only apply a shared `/props` cap when discovery has a single valid model.

## Reproduction Assessment

Yes. Source inspection plus upstream llama.cpp docs show the PR fetches `/props` without a `model` query and then applies that one cap to every model, while router mode uses `?model=` to select the model-specific props response.

## Solution Assessment

No. The direction is narrow, but this implementation should avoid stamping the default model's runtime `n_ctx` onto unrelated models before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Scope the props cap to the selected model:** `src/plugins/provider-self-hosted-setup.ts:192-193`
  - body: The helper fetches `/props` once without a `model` query, then writes that runtime cap to every discovered model. llama.cpp router docs say GET endpoints select a model via `?model=...`; with multiple `/v1/models` rows this can record the default model's `n_ctx` for unrelated models, so session budgeting can still show the wrong denominator. Query `/props?model=<id>` per discovered model or only apply the shared cap when there is one valid model.
  - confidence: 0.86
