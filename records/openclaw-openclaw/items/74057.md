## Summary

Keep open: the PR fixes the right self-hosted llama.cpp context-budget path and now has focused tests, proof, and green checks, but the latest model-scoped `/props?model=...` probe can autoload unloaded llama.cpp router models by default before merge.

## What This Changes

The PR adds llama.cpp `/props` runtime context discovery for self-hosted OpenAI-compatible providers, maps `/v1/models.meta.n_ctx_train` to `contextWindow`, maps `/props` `n_ctx` to `contextTokens`, and updates tests plus changelog.

## Best Possible Solution

Keep the runtime-cap discovery, but make model-scoped llama.cpp `/props` lookups pass `autoload=false` or otherwise skip unloaded router models while preserving explicit configured `contextWindow` precedence.

## Reproduction Assessment

Yes. Source inspection plus the llama.cpp router docs give a high-confidence path: a router `/v1/models` response with multiple models makes the PR request `/props?model=<id>` for each model, and upstream defaults those requests to autoload unless `autoload=false` is present.

## Solution Assessment

No. The direction is the narrow maintainable fix for https://github.com/openclaw/openclaw/issues/73664, but the implementation needs a router-safe props query before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Avoid autoloading router models during props discovery:** `src/plugins/provider-self-hosted-setup.ts:91`
  - body: llama.cpp router mode autoloads a model by default when a GET endpoint is scoped with `?model=...`; the docs say callers must pass `autoload=false` to avoid that side effect. This helper now probes `/props?model=...` for every discovered model, so a read-only model discovery can start loading every router model. Add `autoload=false` to model-scoped props probes or only probe models known to be loaded.
  - confidence: 0.87
