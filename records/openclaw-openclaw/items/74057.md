## Summary

Keep open: the branch fixes the single-model llama.cpp path, but it still applies one unscoped `/props` runtime cap to every discovered model, which is wrong for llama.cpp router mode.

## What This Changes

The PR updates self-hosted provider discovery, focused tests, and changelog to read llama.cpp runtime `n_ctx` from `/props` into `contextTokens` while preserving `n_ctx_train` as native `contextWindow`.

## Best Possible Solution

Keep the runtime-cap discovery, but scope `/props` by model in llama.cpp router mode or apply an unscoped cap only when exactly one valid model is discovered, while preserving explicit configured `contextWindow` values and the SSRF guard.

## Reproduction Assessment

Yes. Source inspection plus upstream llama.cpp router docs give a concrete reproduction: return two `/v1/models` rows with different `/props?model=...` caps, and this PR will copy one unscoped `/props` cap to both rows.

## Solution Assessment

No. The direction is the narrow right fix for the linked bug, but the implementation should make the `/props` lookup model-scoped in router mode or single-model-only before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope the props cap to each selected model:** `src/plugins/provider-self-hosted-setup.ts:192-193`
  - body: The helper fetches `/props` once and writes that runtime cap to every discovered model. llama.cpp router mode routes GET endpoints like `/props` by the URL-encoded `model` query parameter, so a multi-model server can report the default model's `n_ctx` for unrelated models and leave session budgeting wrong. Fetch `/props?model=<id>` per model or only apply the shared cap when there is one valid model.
  - confidence: 0.88
