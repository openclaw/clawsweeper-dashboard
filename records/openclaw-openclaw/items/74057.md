## Summary

Keep open: the PR is useful and includes tests, changelog, and real behavior proof, but the latest head still fetches one unscoped llama.cpp `/props` value and applies it to every discovered model, which can keep session budgeting wrong for multi-model router setups.

## What This Changes

The PR adds llama.cpp `/props` runtime-context discovery to self-hosted OpenAI-compatible model discovery, maps `/v1/models.meta.n_ctx_train` to `contextWindow` and `/props.default_generation_settings.n_ctx` to `contextTokens`, updates focused tests, and adds a changelog entry.

## Best Possible Solution

Keep the runtime-cap discovery, but make the `/props` cap model-scoped in llama.cpp router mode or apply a shared cap only when there is exactly one valid discovered model.

## Reproduction Assessment

Yes. Source inspection of the PR head plus upstream llama.cpp router docs gives a high-confidence reproduction path: return multiple `/v1/models` rows with different `/props?model=...` caps and the PR will still stamp the one unscoped `/props` cap onto every model.

## Solution Assessment

No. The overall direction is the narrow maintainable fix, but this implementation should scope runtime caps to the selected model, or limit unscoped `/props` use to single-model discovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope the props cap to the selected model:** `src/plugins/provider-self-hosted-setup.ts:192-193`
  - body: The helper fetches `/props` once without a `model` query, then writes that runtime cap to every discovered model. llama.cpp router mode routes GET endpoints such as `/props` using the URL-encoded `model` query parameter, so a multi-model server can record the default model's `n_ctx` for unrelated models and leave session budgeting wrong. Query `/props?model=<id>` per discovered model or only apply the shared cap when there is one valid model.
  - confidence: 0.88
