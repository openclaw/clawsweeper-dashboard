## Summary

Keep open. The PR targets a real self-hosted provider/session-budgeting bug and current main still lacks llama.cpp `/props` discovery, but the proposed parser does not match the current upstream `/props` response shape and the user-facing fix is missing a changelog entry.

## What This Changes

The PR adds llama.cpp `/props` discovery to self-hosted OpenAI-compatible model discovery, maps `/v1/models` `meta.n_ctx_train` to `contextWindow`, and stores the runtime cap as `contextTokens` when `/props` reports it.

## Best Possible Solution

Keep the PR direction, but update the implementation to read `default_generation_settings.n_ctx` from llama.cpp `/props` while optionally retaining a top-level fallback for older/reporter-observed servers, keep explicit configured caps authoritative, update tests to cover the upstream shape, add the required changelog line, then rerun the focused provider setup test plus core type/lint checks.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to feed the documented llama.cpp `/props` payload with `default_generation_settings.n_ctx` into the PR parser; the current PR reads only `data.n_ctx`, so `contextTokens` remains unset.

## Solution Assessment

No. The architectural direction is narrow and appropriate, but the current proposed parser should follow the upstream nested response shape before merge; a nested-first parser with an optional top-level compatibility fallback is the safer fix.
