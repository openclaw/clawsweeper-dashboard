## Summary

Keep open. The intended provider fix is valid and current main still lacks llama.cpp runtime-context discovery, but this PR head does not solve the linked bug against the current llama.cpp `/props` contract because it reads a top-level `n_ctx` instead of `default_generation_settings.n_ctx`; it also lacks the required user-facing changelog entry.

## What This Changes

The branch adds guarded self-hosted OpenAI-compatible `/props` discovery and maps llama.cpp model metadata into discovered model `contextWindow` and `contextTokens` fields.

## Best Possible Solution

Revise the PR so llama.cpp runtime discovery reads `default_generation_settings.n_ctx` from `/props` while preserving the existing `contextWindow` and `contextTokens` split, keeps explicit configured caps authoritative, updates tests to use the upstream response shape, and records the provider fix in the changelog.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available: pass the documented llama.cpp `/props` payload with `default_generation_settings.n_ctx` to the PR parser, and `runtimeContextTokens` remains undefined because the code reads only `data.n_ctx`.

## Solution Assessment

No. The direction is narrow and maintainable, but the current implementation should parse the documented nested response shape; keeping a top-level fallback would be acceptable as compatibility, but the upstream-shaped test must be primary.
