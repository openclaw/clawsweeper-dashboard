## Summary

Keep open. The PR targets the right self-hosted provider surface and preserves the explicit configured cap, but current llama.cpp `/props` exposes the runtime cap under `default_generation_settings.n_ctx`, while the PR reads only a top-level `n_ctx`, so the linked session-denominator bug can still reproduce on current servers; the user-facing fix also needs a changelog entry.

## What This Changes

The PR adds llama.cpp `/props` discovery to self-hosted OpenAI-compatible model discovery, maps `/v1/models` `meta.n_ctx_train` to `contextWindow`, and records a discovered runtime cap as `contextTokens`.

## Best Possible Solution

Keep the PR direction, but parse llama.cpp `/props` nested-first from `default_generation_settings.n_ctx`, retain an optional top-level fallback for compatibility, preserve explicit configured caps, and document the user-facing fix in the active changelog.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to feed a documented llama.cpp `/props` payload with `default_generation_settings.n_ctx` into the PR parser; because it reads only `data.n_ctx`, `contextTokens` remains unset.

## Solution Assessment

No. The approach is narrow and maintainable, but the parser and test fixture should match the upstream response shape before merge, and the user-facing provider/session-budgeting fix needs the required changelog line.
