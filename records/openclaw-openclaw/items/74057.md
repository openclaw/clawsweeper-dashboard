## Summary

Keep this PR open but not merge-ready: the remaining model-scoped llama.cpp `/props` probe can autoload unloaded router models during discovery, which is a read-only path.

## What This Changes

The PR updates self-hosted provider discovery, focused tests, and the changelog to read llama.cpp runtime context from `/props` into `contextTokens` while preserving native `contextWindow` metadata.

## Best Possible Solution

Keep the runtime-cap discovery, but make model-scoped llama.cpp `/props` probes router-safe with `autoload=false` and add a focused regression assertion.

## Reproduction Assessment

Yes. Source inspection plus upstream llama.cpp docs give a high-confidence path: a router `/v1/models` response with multiple models makes this PR request `/props?model=<id>` for each model, and upstream defaults those requests to autoload unless `autoload=false` is present.

## Solution Assessment

No. The PR targets the right provider-discovery surface for https://github.com/openclaw/openclaw/issues/73664, but the model-scoped router probe needs the per-request autoload guard before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Avoid autoloading router models during props discovery:** `src/plugins/provider-self-hosted-setup.ts:91`
  - body: llama.cpp router mode autoloads a model by default when a GET endpoint is scoped with `model`, and the docs say callers can pass `autoload=false` per request. This helper probes `/props?model=...` for every discovered model, so a read-only discovery can start loading unloaded router models. Add `autoload=false` to model-scoped props probes or otherwise avoid probing unloaded router models.
  - confidence: 0.87
