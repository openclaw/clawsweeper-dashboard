## Summary

Keep open. The PR is a useful, narrow fix for the open linked bug, but the current head still parses the llama.cpp `/props` response from the wrong JSON path, so it would not populate `contextTokens` for current llama.cpp servers; it also still lacks the required user-facing changelog entry.

## What This Changes

The PR adds opportunistic llama.cpp `/props` discovery to self-hosted OpenAI-compatible model discovery, maps `/v1/models` `meta.n_ctx_train` to `contextWindow`, maps runtime context to `contextTokens`, and adds provider setup tests.

## Best Possible Solution

Revise the PR so the parser reads `default_generation_settings.n_ctx` from current llama.cpp `/props` responses, preserves the existing `contextWindow` and `contextTokens` split, keeps explicit configured caps authoritative, updates the focused tests to cover the upstream shape, and adds a single-line changelog fix entry.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available: the PR head reads `data.n_ctx`, while current llama.cpp documents the runtime context at `default_generation_settings.n_ctx`; with that real response shape, the new `contextTokens` value remains undefined.

## Solution Assessment

No. The product direction is narrow and appropriate, but the current implementation should parse the documented nested `/props` response shape, with a test using that shape; keeping a top-level fallback would be acceptable only as backward compatibility.
