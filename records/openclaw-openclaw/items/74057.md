## Summary

Keep this PR open: it targets a real llama.cpp runtime-context bug and includes useful proof, but the current diff still fetches one unscoped `/props` value and applies that cap to every discovered model.

## What This Changes

The PR adds llama.cpp `/props` discovery to self-hosted OpenAI-compatible provider setup, maps `/v1/models.meta.n_ctx_train` to `contextWindow`, stores `/props.default_generation_settings.n_ctx` as `contextTokens`, updates tests, and adds a changelog entry.

## Best Possible Solution

Keep the runtime-cap discovery, but scope `/props` results to each selected model or apply a shared cap only when exactly one valid model is discovered.

## Reproduction Assessment

Yes. Source inspection of the PR head plus upstream llama.cpp router docs gives a high-confidence path: a multi-model server can expose different `/props?model=...` caps while the PR applies one unscoped `/props` cap to all discovered rows.

## Solution Assessment

No. The direction is the right narrow fix, but the implementation should model-scope runtime caps or restrict unscoped `/props` use to single-model discovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope the props cap to the selected model:** `src/plugins/provider-self-hosted-setup.ts:192-193`
  - body: The helper fetches `/props` once without a `model` query, then writes that runtime cap to every discovered model. llama.cpp router mode routes GET endpoints such as `/props` through the URL-encoded `model` query parameter, so a multi-model server can stamp one model's `n_ctx` onto unrelated models and make session budgeting wrong. Query `/props?model=<id>` per discovered model or only apply the shared cap when there is exactly one valid model.
  - confidence: 0.88
