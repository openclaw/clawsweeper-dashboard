## Summary

Keep open: the PR is a useful bug fix with sufficient real behavior proof, but the current diff still assigns one unscoped llama.cpp `/props` cap to every discovered model, which is wrong for router-mode multi-model servers.

## What This Changes

This PR adds llama.cpp `/props` runtime context discovery to self-hosted OpenAI-compatible model discovery, maps `/v1/models.meta.n_ctx_train` to `contextWindow`, maps `/props.default_generation_settings.n_ctx` to `contextTokens`, and updates tests plus changelog.

## Best Possible Solution

Keep the runtime-cap discovery, but make `/props` model-scoped in router mode or apply a shared unscoped cap only for single-model discovery while preserving explicit configured `contextWindow` values.

## Reproduction Assessment

Yes. Source inspection of the PR head plus llama.cpp router docs gives a high-confidence path: return two `/v1/models` rows with different `/props?model=...` caps and the PR will copy one unscoped `/props` cap to both rows.

## Solution Assessment

No. The direction is the right narrow fix, but this implementation needs model-scoped `/props` discovery or a single-model-only guard before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope the props cap to the selected model:** `src/plugins/provider-self-hosted-setup.ts:192-193`
  - body: The helper fetches `/props` once without a `model` query and then writes that runtime cap to every discovered model. llama.cpp router mode routes GET endpoints such as `/props` through the URL-encoded `model` query parameter, so a multi-model server can report the default model's `n_ctx` for unrelated models and leave session budgeting wrong. Query `/props?model=<id>` per model or only apply the shared cap when there is one valid model.
  - confidence: 0.88
