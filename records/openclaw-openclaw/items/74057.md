## Summary

Keep this PR open. The latest branch fixes the earlier shared-cap problem and supplies real behavior proof, but the model-scoped llama.cpp router probes still omit `autoload=false`, so discovery can load every advertised router model while trying to read `/props`.

## What This Changes

The PR adds llama.cpp `/props` runtime-context discovery for self-hosted OpenAI-compatible models, maps `/v1/models.meta.n_ctx_train` to `contextWindow`, writes `/props.default_generation_settings.n_ctx` to `contextTokens`, adds tests, and updates the changelog.

## Best Possible Solution

Keep the runtime-cap discovery, but make model-scoped router `/props` probes non-autoloading and gracefully omit `contextTokens` when an unloaded model cannot report props.

## Reproduction Assessment

Yes. Source inspection plus llama.cpp router docs give a high-confidence path: a multi-model `/v1/models` response makes the PR issue `/props?model=<id>` for every id, and upstream autoloads unloaded models by default unless `autoload=false` is supplied.

## Solution Assessment

No. The fix direction is right, but the model-scoped router probe should avoid autoloading models; adding `autoload=false` to model-scoped `/props` requests is the narrower maintainable correction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Disable router autoload for props probes:** `src/plugins/provider-self-hosted-setup.ts:91`
  - body: llama.cpp router mode autoloads the queried model by default for GET endpoints unless `autoload=false` is set. This helper now probes `/props?model=<id>` for every discovered model, so a normal discovery or models-list refresh can load all advertised models into memory/VRAM just to read context caps. Add `autoload=false` to model-scoped props URLs, and tolerate missing caps for unloaded models.
