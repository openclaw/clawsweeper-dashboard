## Summary

Keep this open: current main and v2026.6.8 still lack a Discord reconnect-triggered delivery drain, and the linked PR is open/unmerged candidate work rather than implementation provenance.

## What This Changes

The issue requests Discord outbound delivery queuing and retry after Discord gateway reconnects so messages attempted during short reconnect windows are not lost.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/94320 with a bounded Discord-owned reconnect drain that reuses the shared delivery queue helper and avoids adding a new config surface.

## Reproduction Assessment

Yes, from source inspection. Discord durable sends write to the shared queue, but current Discord lifecycle reconnect handling never drains Discord-matched pending entries, unlike Telegram and WhatsApp.

## Solution Assessment

Yes. The narrow maintainable solution is a Discord-owned reconnect drain using the existing shared delivery queue runtime; adding `channels.discord.delivery.queue` or another new config key is not the best path.
