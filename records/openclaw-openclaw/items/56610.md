## Summary

Current main still lacks a Discord-owned reconnect drain for failed outbound delivery queue entries. The generic queue and WhatsApp reconnect parity path exist, but Discord’s READY/reconnect lifecycle only updates gateway status and waits for stop/error events, so the reported reliability gap remains actionable.

## What This Changes

Add Discord outbound delivery queue retry on gateway WebSocket reconnect, matching the existing WhatsApp reconnect drain behavior.

## Best Possible Solution

Add a Discord plugin lifecycle drain after gateway READY/reconnect, scoped to channel and account, reusing the existing delivery queue runtime without adding a new user-facing config knob unless implementation proves one is necessary.

## Reproduction Assessment

Yes for a source-level reproduction: create or fail a delivery queue entry with channel="discord", drive the Discord gateway lifecycle through READY/reconnect, and observe that current main never calls drainPendingDeliveries. A live disconnect reproduction was not run.

## Solution Assessment

Yes, a narrow Discord-owned reconnect drain using the existing queue runtime is the best fit. A broad core retry policy or startup-only recovery change would not specifically cover the reconnect-window parity gap.
