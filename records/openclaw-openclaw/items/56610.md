## Summary

Keep this open: current main has the shared write-ahead delivery queue, Discord final-reply failure surfacing, and startup recovery, but I did not find the Discord-specific reconnect drain that would retry queued outbound Discord messages when the gateway becomes ready again.

## What This Changes

The issue requests durable Discord outbound queueing and retry across gateway WebSocket reconnect windows, matching the existing WhatsApp-style reconnect recovery behavior.

## Best Possible Solution

Wire Discord gateway READY/reconnect recovery into the existing delivery-queue-runtime drain path, with account-scoped selection and tests, without adding a new Discord config surface.

## Reproduction Assessment

Yes, source-level: Discord currently uses the shared durable send path and startup recovery, while only WhatsApp/Telegram wire drainPendingDeliveries to reconnect or healthy-poll events. I did not run a live Discord outage reproduction.

## Solution Assessment

Yes, the best fix is to reuse the existing delivery queue drain helper at the Discord gateway lifecycle boundary. Adding a new channels.discord.delivery.queue config key is not the best path because the repository already has a shared durability contract and a high bar for new config.
