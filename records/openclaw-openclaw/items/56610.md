## Summary

Keep open: current main has the generic outbound delivery queue and reconnect drains in other channels, but Discord still does not drain queued outbound deliveries after gateway readiness, so the reported message-loss gap remains actionable.

## What This Changes

The issue requests Discord retry or queueing for outbound messages sent during gateway WebSocket reconnect windows.

## Best Possible Solution

Add a Discord-owned reconnect drain after gateway READY/RESUMED that reuses the existing delivery queue runtime, scopes by channel/account, and avoids a new config knob unless maintainers explicitly choose one.

## Reproduction Assessment

Yes at source level: seed or fail a queued delivery with channel="discord", drive the Discord gateway lifecycle through READY/reconnect, and current main has no drainPendingDeliveries call to recover it. I did not run a live Discord disconnect.

## Solution Assessment

Yes: a Discord-owned reconnect drain that reuses the existing delivery queue runtime is the narrowest maintainable fix. A broad core retry subsystem or new channels.discord delivery config is not needed for this reliability gap.
