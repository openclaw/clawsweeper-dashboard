## Summary

Keep open: current main and v2026.5.18 still have the generic delivery queue and WhatsApp reconnect drain, but Discord has no channel-owned drain after gateway READY/reconnect, so the reported outbound message-loss gap remains.

## What This Changes

Request Discord outbound deliveries that fail during gateway WebSocket reconnect to be queued and retried after reconnect, matching the existing WhatsApp delivery-queue behavior.

## Best Possible Solution

Keep this open until Discord drains queued outbound deliveries on READY/reconnect via the existing queue runtime, scoped by channel and account without a new config knob unless maintainers request one.

## Reproduction Assessment

Yes for source-level reproduction: seed or fail a queued delivery with channel="discord", drive the Discord gateway lifecycle through READY/reconnect, and current main has no drainPendingDeliveries call to pick it up. I did not run a live Discord disconnect.

## Solution Assessment

Yes: a Discord-owned reconnect drain reusing the existing queue runtime is the narrowest maintainable fix. A broad core retry policy or a new user-facing config option is not needed for the core reliability gap.
