## Summary

Keep open: current main still has the generic outbound delivery queue and WhatsApp/Telegram reconnect drains, but Discord has no READY/reconnect hook that drains queued outbound deliveries, so the reported Discord message-loss gap remains actionable.

## What This Changes

The issue requests Discord outbound delivery queue recovery on gateway reconnect, matching the existing WhatsApp reconnect drain behavior.

## Best Possible Solution

Add a Discord-owned reconnect drain after gateway readiness that reuses the existing delivery queue runtime, scopes by channel/account, and avoids adding a new config knob unless maintainers explicitly want one.

## Reproduction Assessment

Yes at source level: seed or fail a queued delivery with channel="discord", drive the Discord gateway lifecycle through READY/reconnect, and current main has no drainPendingDeliveries call to pick it up. I did not run a live Discord disconnect.

## Solution Assessment

Yes: a Discord-owned reconnect drain reusing the existing queue runtime is the narrowest maintainable fix. A broad core retry subsystem or new channels.discord delivery config is not needed for this reliability gap.

Add a Discord-plugin-owned reconnect drain after gateway readiness/reconnect that calls openclaw/plugin-sdk/delivery-queue-runtime drainPendingDeliveries, scopes by normalized Discord account id and channel, and reuses existing deliverOutboundPayloadsInternal through the SDK runtime. Preserve existing Discord REST retry behavior; only use bypassBackoff for the verified reconnect/no-active-listener failure class. Add focused coverage for the Discord lifecycle/startup drain hook, account-scoped queue selection, backoff behavior, and no duplicate replay while a live delivery is claimed. Keep the scope outbound-only and do not mix in the inbound reconnect-backfill work from https://github.com/openclaw/openclaw/pull/81226. Add a changelog entry because this is user-visible message delivery behavior.
