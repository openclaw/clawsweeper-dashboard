## Summary

Keep this open: current main has shared durable outbound queue plumbing, Discord final-reply failure surfacing, and startup recovery, but Discord still lacks the reconnect/READY drain that would retry account-scoped queued Discord deliveries after the gateway becomes ready again.

## What This Changes

The issue requests Discord outbound delivery queue/retry behavior for gateway reconnect windows so messages are not silently dropped.

## Best Possible Solution

Wire Discord gateway READY/reconnect recovery into the existing delivery-queue-runtime drain path with account-scoped selection and tests, without adding a new Discord config surface.

## Reproduction Assessment

Yes, source-level: Discord durable sends can leave failed queue entries and the Discord lifecycle observes reconnect/READY state, but current source does not connect those events to a Discord-specific pending-delivery drain. I did not run a live Discord outage reproduction.

## Solution Assessment

Yes. The narrow maintainable fix is to reuse the existing delivery queue drain helper at the Discord gateway lifecycle boundary; adding a new channels.discord.delivery.queue config key is not the best first step because the shared durability contract already exists.
