## Summary

Keep open: current main still has the shared durable outbound queue and Discord gateway readiness handling, but Discord does not drain pending outbound deliveries on READY/RESUMED or reconnect, so queued Discord messages can remain stranded until a later gateway startup recovery.

## What This Changes

Request Discord outbound messages that fail during gateway WebSocket reconnect windows be queued and retried when Discord reconnects, matching the existing WhatsApp-style reconnect drain.

## Best Possible Solution

Add a Discord-plugin-owned reconnect drain on Gateway READY/RESUMED that reuses openclaw/plugin-sdk/delivery-queue-runtime, scopes entries by Discord channel/account, and avoids a new config knob unless maintainers explicitly want one.

## Reproduction Assessment

Yes at source level: seed or leave a queued Discord delivery and drive the Discord Gateway lifecycle through READY/RESUMED; current main has no Discord drainPendingDeliveries call to recover it. I did not run a live Discord disconnect.

## Solution Assessment

Yes: reusing the existing durable delivery queue from the Discord plugin is the narrowest maintainable fix. A broad core retry subsystem or new channels.discord.delivery config surface is not necessary for this gap.
