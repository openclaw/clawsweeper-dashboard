## Summary

Keep open: current main still has no Discord-owned reconnect drain for failed outbound delivery queue entries, while WhatsApp has that parity path and the generic queue API already exists.

## What This Changes

Request Discord outbound messages that fail during gateway WebSocket reconnect windows be queued and retried after reconnect, matching WhatsApp delivery-queue behavior.

## Best Possible Solution

Keep the generic delivery queue in core and add a Discord plugin lifecycle drain after gateway READY/reconnect, scoped to channel and account, reusing the plugin-sdk delivery-queue runtime rather than adding a broad core migration or new config surface unless implementation proves a missing SDK seam.

## Reproduction Assessment

Yes for a focused source-level reproduction: enqueue or fail a delivery entry with channel="discord", drive the Discord gateway lifecycle through READY/reconnect, and assert current main never calls drainPendingDeliveries. A live Discord disconnect reproduction was not run.

## Solution Assessment

Yes, the best fix is a narrow Discord-owned reconnect drain using the existing delivery queue runtime. A broad core retry policy or startup-only recovery change would not specifically close the reconnect-window parity gap.
