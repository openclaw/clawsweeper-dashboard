## Summary

Keep this open: current main and latest v2026.6.6 have the shared durable outbound queue and Discord final-reply failure surfacing, but Discord still does not drain queued outbound deliveries when its gateway reaches READY after reconnect.

## What This Changes

The issue asks OpenClaw to retry queued outbound Discord messages after Discord gateway WebSocket reconnects, matching the existing WhatsApp reconnect-drain behavior.

## Best Possible Solution

Wire Discord gateway READY/reconnect recovery into the existing delivery-queue-runtime drain path with account-scoped selection and regression coverage, without adding a new Discord config surface.

## Reproduction Assessment

Yes, at source level: Discord durable final delivery can leave pending failed entries, and current Discord gateway lifecycle has no reconnect drain while WhatsApp and Telegram do. I did not run a live Discord outage reproduction.

## Solution Assessment

Yes, the narrow maintainable fix is to reuse the existing shared delivery-queue drain at Discord's gateway READY/reconnect boundary. The proposed new `channels.discord.delivery.queue` config is not the best default path because the repo already has a shared durability contract and a high bar for new config.
