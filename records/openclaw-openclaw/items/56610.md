## Summary

Keep this open: current main and v2026.6.8 have the shared outbound queue, Discord durable final delivery, and startup recovery, but Discord still does not wire a reconnect-ready drain like WhatsApp and Telegram do.

## What This Changes

The issue requests Discord outbound message queueing and retry when messages fail during Discord WebSocket reconnect or API-degradation windows.

## Best Possible Solution

Wire Discord gateway READY/reconnect recovery into the existing delivery-queue-runtime drain path with account-scoped selection and tests, without adding a new Discord config key.

## Reproduction Assessment

Yes, source-level: current main writes Discord final replies through durable outbound delivery and has a shared reconnect drain helper, while only WhatsApp and Telegram wire that helper to reconnect/healthy-poll events. I did not establish a live Discord outage reproduction.

## Solution Assessment

Yes, the narrow maintainable fix is to reuse the existing delivery queue drain helper at the Discord gateway lifecycle boundary. Adding `channels.discord.delivery.queue` is not the best path because the shared queue contract already exists and repo policy sets a high bar for new config.
