## Summary

Keep this open: current `main` and `v2026.6.8` still lack a Discord reconnect-triggered delivery drain, and https://github.com/openclaw/openclaw/pull/94320 is an open, unmerged candidate fix rather than implementation provenance.

## What This Changes

The issue requests Discord outbound delivery queuing and retry on gateway reconnect so messages sent during reconnect windows are not silently dropped.

## Best Possible Solution

Review and land or replace https://github.com/openclaw/openclaw/pull/94320 with a bounded Discord-owned reconnect drain that reuses the shared delivery queue helper and does not add a new config surface.

## Reproduction Assessment

Yes, from source: Discord durable sends write to the shared queue, but current Discord lifecycle reconnect handling never drains Discord-matched pending entries, unlike WhatsApp and Telegram. I did not run a live Discord reconnect reproduction in this read-only review.

## Solution Assessment

Yes, the best solution is a narrow Discord-owned reconnect drain using the existing shared delivery queue runtime. Adding `channels.discord.delivery.queue` or another new config key is not the best path because the queue contract already exists and repo policy sets a high bar for new config.
