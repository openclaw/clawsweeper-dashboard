## Summary

Keep open: current main still has the shared durable delivery queue and Discord durable outbound sends, but Discord has no reconnect/READY drain for queued outbound deliveries, so the reported message-loss gap remains actionable.

## What This Changes

Requested Discord outbound delivery queue and retry behavior across gateway reconnect windows, matching WhatsApp delivery recovery.

## Best Possible Solution

Add a Discord-plugin-owned reconnect drain after gateway READY/RESUMED that reuses `openclaw/plugin-sdk/delivery-queue-runtime`, scopes entries by channel and normalized account, and avoids a new config knob unless maintainers explicitly choose one.

## Reproduction Assessment

Yes at source level: seed a queued delivery with `channel: "discord"`, drive the Discord gateway lifecycle through READY/reconnect, and current main has no `drainPendingDeliveries` call to recover it. I did not run a live Discord disconnect.

## Solution Assessment

Yes: a Discord-owned reconnect drain using the existing durable delivery queue is the narrowest maintainable fix. A broad core retry subsystem or new `channels.discord.delivery` config surface is not needed for this gap.
