## Summary

Keep open: current main has a shared write-ahead delivery queue and Discord durable outbound sends, but Discord still lacks a reconnect/READY drain that replays queued Discord outbound deliveries after the gateway reconnects. The open related PR is nearby but scoped to inbound missed-message backfill, not this outbound queue gap.

## What This Changes

The issue requests Discord outbound delivery queue retry after WebSocket reconnects so cron reports, sub-agent completions, and other messages are not lost during reconnect windows.

## Best Possible Solution

Add a Discord-plugin-owned reconnect drain after gateway READY/RESUMED that reuses the existing delivery-queue runtime, scopes entries by Discord channel and normalized account, and avoids adding a new config knob unless maintainers explicitly choose one.

## Reproduction Assessment

Yes, at source level: seed a queued delivery with `channel: "discord"`, drive the Discord gateway lifecycle through READY/reconnect, and current main has no `drainPendingDeliveries` hook to recover it. I did not run a live Discord disconnect.

## Solution Assessment

Yes: a Discord-owned reconnect drain using the existing durable delivery queue is the narrowest maintainable fix. A broad core retry subsystem or new `channels.discord.delivery` config surface is not needed for this gap.
