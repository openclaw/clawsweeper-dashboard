## Summary

Keep open: current main has durable outbound queue plumbing, Discord final-reply failure surfacing, and startup recovery, but it still lacks a Discord gateway reconnect/READY drain for queued outbound Discord messages. The related production report was already folded into this issue, and no merged PR now owns the remaining outbound reconnect retry work.

## What This Changes

Request Discord outbound delivery retry by draining queued messages after gateway WebSocket reconnects, matching the existing WhatsApp reconnect recovery pattern.

## Best Possible Solution

Wire Discord gateway READY/RESUMED recovery into the existing delivery-queue-runtime drain path with account-scoped selection and regression tests, without adding a new config surface.

## Reproduction Assessment

Yes, at source level: current main shows durable queue/failure handling and a WhatsApp reconnect drain, while Discord lifecycle code has no corresponding drainPendingDeliveries hook. I did not run a live Discord outage reproduction.

## Solution Assessment

Yes, the best fix is to reuse the existing delivery queue drain helper at the Discord gateway lifecycle boundary. Adding a new channels.discord.delivery.queue option is not the best first path because repository policy keeps config surface small and the shared durability contract already exists.
