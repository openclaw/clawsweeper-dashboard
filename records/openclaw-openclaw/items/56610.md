## Summary

Keep open: current main still has durable outbound queue recovery and WhatsApp reconnect drains, but Discord reconnect readiness does not drain queued outbound deliveries, so the reported message-loss gap remains actionable.

## What This Changes

The issue requests Discord outbound delivery queueing and retry after gateway WebSocket reconnects, matching WhatsApp reconnect recovery.

## Best Possible Solution

Add a Discord-owned reconnect drain that reuses the existing delivery queue runtime on READY/reconnect, scoped by channel and account, with regression coverage and no new user-facing config unless maintainers explicitly want a policy knob.

Yes for source-level reproduction: seed or fail a queued delivery with channel="discord", drive the Discord gateway lifecycle through READY/reconnect, and current main has no drainPendingDeliveries call to pick it up. I did not run a live Discord disconnect.

## Solution Assessment

Yes: a Discord-owned reconnect drain reusing the existing queue runtime is the narrowest maintainable path. A broad core retry subsystem or new config surface is not needed for the reliability gap described here.
