Keep open: current main and the latest release still lack a Discord-specific reconnect delivery drain, and the open linked PR is an active fix candidate but has not merged or cleared proof/review yet.

## What This Changes

Discord outbound messages attempted during gateway reconnect should be queued and retried after reconnect, matching existing durable delivery recovery behavior.

## Best Possible Solution

Keep this issue as the canonical tracker until https://github.com/openclaw/openclaw/pull/94320 is repaired, proven in a real Discord setup, and either landed or deliberately closed.

## Reproduction Assessment

Yes, source-level: current main has generic durable outbound queue plumbing and sibling reconnect-drain patterns, while Discord lacks reconnect draining in its gateway lifecycle. I did not run a live Discord disconnect reproduction in this read-only review.

## Solution Assessment

Yes in direction: using the shared delivery queue from the Discord gateway lifecycle is the narrow maintainable fix, and no new config surface appears necessary. The active PR still needs its retry lookup repaired and real Discord proof before it is the best mergeable solution.
