- `impact:message-loss`: The reported failure mode is lost outbound Discord cron reports, sub-agent completions, and other messages during reconnect windows.

## Summary

Keep open: current main still has Discord durable outbound sends and the shared pending-delivery drain API, but Discord does not call the drain on Gateway READY/RESUMED or reconnect, so queued outbound messages can remain stranded after the reported reconnect window. The related open PR addresses inbound missed-message backfill and reply hydration, not this outbound delivery-queue drain.

## What This Changes

Request Discord to queue and retry outbound messages that fail during Gateway WebSocket reconnect windows, matching WhatsApp-style pending-delivery recovery.

## Best Possible Solution

Add a Discord-plugin-owned reconnect drain after Gateway READY/RESUMED that reuses `openclaw/plugin-sdk/delivery-queue-runtime`, scopes by Discord channel/account, and avoids a new config knob unless maintainers explicitly request one.

## Reproduction Assessment

Yes at source level: seed or leave a queued Discord delivery and drive the Discord Gateway lifecycle through READY/RESUMED; current main has no Discord `drainPendingDeliveries` call to recover it. I did not run a live Discord disconnect.

## Solution Assessment

Yes: reusing the existing durable delivery queue from the Discord plugin is the narrowest maintainable fix. A broad core retry subsystem or new `channels.discord.delivery` config surface is not necessary for this gap.
