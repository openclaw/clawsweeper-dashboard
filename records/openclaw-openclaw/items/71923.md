## Summary

Keep open. The PR targets a real doctor-auth labeling bug from the linked report, but the current head still disables the existing interactive OAuth refresh for the default/runtime auth store when multi-agent diagnostics are enabled.

## What This Changes

The PR enumerates per-agent doctor auth stores, loads them read-only for diagnostics, labels multi-agent auth notes with agent ids, exports a direct auth-store source helper, and adds tests plus a changelog entry for the linked doctor-auth issue.

## Best Possible Solution

Keep the read-only labeled diagnostics for non-default direct agent stores, but distinguish the default/runtime target so it remains eligible for the existing interactive OAuth refresh path, with regression coverage for both behaviors.

## Reproduction Assessment

Yes. Source inspection at PR head is enough: a direct non-default auth store causes the default/runtime target to be labeled, and the `params.agentId` gate then skips `confirmAutoFix` for that default pass.

## Solution Assessment

No. The labeling direction is right, but refresh eligibility should be based on the target kind or `agentDir === undefined`, not on whether the diagnostic title has an agent id suffix.
