## Summary

Keep open. The PR moves in the right direction for per-agent doctor auth diagnostics, but its current refresh gate regresses the default/runtime auth store when any non-default direct agent store exists: the default pass is labeled with an agent id, and then all labeled passes skip the existing interactive OAuth refresh path.

## What This Changes

The PR changes doctor auth health reporting to enumerate direct per-agent auth stores, load them in read-only mode, suffix multi-agent auth diagnostics with agent ids, export a direct auth-store source helper, add regression tests, and add a changelog entry for #71867.

## Best Possible Solution

Keep the per-agent read-only diagnostics and direct-store filtering, but decouple display labeling from refresh eligibility: the default/runtime `agentDir: undefined` pass should retain the existing interactive OAuth refresh prompt, while non-default direct agent-store passes remain diagnostics-only.

## Reproduction Assessment

Yes. A static reproduction is enough: with `main` and `coder` direct auth stores, the PR includes the default store as `{ agentId: main, agentDir: undefined }`, then line 307 forces `shouldRefresh` to `false` because an agent label is present.

## Solution Assessment

No. The requested direction is right, but the current patch is not the best fix because it uses the display agent label as the refresh gate; the safer boundary is refresh eligibility based on the default/runtime target, not whether a note title has a suffix.
