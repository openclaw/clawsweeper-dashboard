## Summary

Keep open: the PR targets retired `src/web` WhatsApp files, while current main still has the bare `file` fallback in the active WhatsApp plugin. The intended behavior is useful, but it needs to be ported to `extensions/whatsapp` before it can fix shipped behavior.

## What This Changes

The PR appends MIME-derived extensions to WhatsApp document filename fallbacks in old `src/web` auto-reply and send-API paths and updates old-layout send-API tests.

## Best Possible Solution

Land a small WhatsApp plugin change that resolves document filenames in one shared place: preserve names that already have extensions, append `extensionForMime()` when the chosen document name lacks one, leave unknown MIME types unchanged, cover both send API and auto-reply document paths, and add focused tests plus a changelog entry.

Yes. Static reproduction on current main is high-confidence: `createWebSendApi(...).sendMessage(..., Buffer.from("pdf"), "application/pdf")` without `sendOptions.fileName` reaches `extensions/whatsapp/src/inbound/send-api.ts:67`, and the active test asserts `fileName: "file"`.

## Solution Assessment

No. The intended MIME-derived extension behavior is the right narrow fix, but this PR edits stale `src/web` files; the maintainable solution is to port it into the active WhatsApp plugin using `openclaw/plugin-sdk/media-mime`.
