## Summary

Keep open: the WhatsApp document filename bug still exists on current main, but this PR cannot merge as-is because it edits the removed old `src/web` WhatsApp layout while the active implementation lives in `extensions/whatsapp`. The change is still useful source work and should be ported or replaced in the plugin.

## What This Changes

The PR appends MIME-derived file extensions to WhatsApp document filename fallbacks in the old `src/web` auto-reply and send API paths and updates old-layout send API tests for PDF and CSV fallbacks.

## Best Possible Solution

Land a small WhatsApp plugin change: add a shared document filename resolver that preserves names with extensions, appends `extensionForMime()` when the chosen document name lacks one, leaves unknown MIME types unchanged, applies it to the plugin send API and auto-reply normalization path, and updates focused tests plus the changelog.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `createWebSendApi(...).sendMessage(..., Buffer, "application/pdf")` without `sendOptions.fileName` reaches the active fallback at `extensions/whatsapp/src/inbound/send-api.ts:67`, and the current test asserts `fileName: "file"`.

## Solution Assessment

No, not as currently written. The intended MIME-extension behavior is the right narrow fix, but this PR edits stale `src/web` files; the maintainable solution is to port it into `extensions/whatsapp` using the plugin SDK MIME helper.
