## Summary

Keep open: the filename-extension bug appears source-reproducible, but this PR currently edits legacy `src/web` WhatsApp files that are absent from current main while the live WhatsApp plugin still falls back to `file` for document sends without a filename.

## What This Changes

The branch appends MIME-derived extensions to legacy `src/web` WhatsApp document filename fallbacks and updates the legacy send API tests.

## Best Possible Solution

Retarget the MIME-derived fallback to the active WhatsApp plugin send and outbound media normalization paths, compose it with filename sanitization, and prove it with focused plugin tests plus redacted live WhatsApp delivery evidence.

## Reproduction Assessment

Yes, by source inspection: current main sets document `fileName` to `"file"` when `sendOptions.fileName` is absent, and the active PDF fallback test asserts that behavior. I did not run a live WhatsApp delivery reproduction in this read-only review.

## Solution Assessment

No. The intended fix is narrow, but this PR applies it to removed legacy `src/web` files instead of the active `extensions/whatsapp` plugin paths; the safer solution is to port the resolver and tests to the plugin implementation.
