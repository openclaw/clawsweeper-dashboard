## Summary

Keep open: current main and v2026.6.10 still lack single-enabled-account inference for omitted message-tool account IDs, and the active fix path is the open but not yet merged https://github.com/openclaw/openclaw/pull/85671.

## What This Changes

The issue asks the outbound message tool to infer the only enabled account for a selected channel when accountId is omitted instead of falling through to a disabled default account.

## Best Possible Solution

Review, refresh, and land or replace the shared outbound account resolver so explicit/default/bound account IDs keep precedence, exactly one enabled configured account is inferred, and ambiguous or unavailable account states fail clearly with docs and tests.

## Reproduction Assessment

Yes at source level, but not by a live WhatsApp run in this review. Current main leaves accountId unset unless it is explicit/default/bound, and WhatsApp omitted-account sends ask the default helper, so a disabled default plus one enabled named account still selects default.

## Solution Assessment

Yes for the owner boundary: the maintainable direction is a shared resolver after channel selection using plugin config adapter hooks. It still needs maintainer acceptance of the compatibility-sensitive fallback policy and review of the open fix PR or a replacement.
