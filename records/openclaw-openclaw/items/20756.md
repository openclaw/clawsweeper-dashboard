## Summary

Keep open. Current main and the latest release still leave omitted message-tool `accountId` unresolved until WhatsApp falls back to its default-account helper, and the linked implementation PR is open but not merged.

## What This Changes

The issue asks outbound message actions without `accountId` to choose the only enabled configured account instead of falling back to a disabled default account.

## Best Possible Solution

Refresh and land the linked shared-resolver PR, or an equivalent fix, that preserves explicit/default/bound account precedence, auto-selects exactly one enabled configured account, and documents/tests clear zero-or-multiple-account errors.

## Reproduction Assessment

Yes, at source level. With `accountId` omitted, current main leaves the runner account unset unless explicit/default/binding sources provide one, and WhatsApp then resolves through the default-account helper.

## Solution Assessment

Yes, the best fix shape is a generic selected-channel account resolver using plugin adapter hooks. A WhatsApp-only fallback would duplicate outbound routing policy and miss the same class in other multi-account channels.
