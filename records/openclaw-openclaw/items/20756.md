## Summary

Keep open. Current main still leaves omitted `accountId` unresolved in the shared message runner, then WhatsApp falls back to the default-account helper, so a disabled `default` plus one enabled named account can still target `default`; the prior implementation PR is closed unmerged.

## What This Changes

The issue requests that the message tool choose the only enabled account for a channel when `accountId` is omitted instead of falling back to a disabled `default` account.

## Best Possible Solution

Centralize omitted-account resolution in the shared outbound path using the selected channel plugin adapter, then document and test enabled-default, single-enabled, zero-enabled, and multiple-enabled cases.

## Reproduction Assessment

Yes, at source level. With `accountId` omitted, current main leaves the runner account unset and WhatsApp resolves through the default-account helper, so a disabled `default` plus one enabled named account still targets `default`.

## Solution Assessment

Yes. A generic selected-channel account resolver is the narrow maintainable fix; a WhatsApp-only fallback would duplicate routing policy and miss the same issue class in other plugin channels.
