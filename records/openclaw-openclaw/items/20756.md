## Summary

Keep open. Current main still lacks single-enabled-account inference for omitted outbound message accountId, and the open linked fix PR explicitly targets this issue but is unmerged and currently conflicting.

## What This Changes

The issue requests that outbound message sending infer the only enabled account for a selected channel when accountId is omitted instead of falling back to a disabled default account.

## Best Possible Solution

Review, refresh, and land or replace the linked shared-resolver PR, preserving explicit/default/bound account precedence and documenting the accepted omitted-account fallback policy.

## Reproduction Assessment

Yes, at source level: current main leaves omitted accountId unresolved after explicit/default/bound checks, and WhatsApp then falls back to the default account. I did not run live WhatsApp delivery because this was a read-only review.

## Solution Assessment

Yes for the owner boundary: a shared resolver after channel selection is the maintainable shape, not a WhatsApp-only fallback. The current open PR is the right direction but needs rebase, proof, and maintainer acceptance of the fallback policy.
