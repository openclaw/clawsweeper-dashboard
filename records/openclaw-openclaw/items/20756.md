## Summary

Keep open. Current main still has the reported account-selection gap: omitted message-tool account IDs only come from explicit/default/bound sources, WhatsApp omitted sends still fall back to the default-account helper, and the related implementation PR was closed unmerged.

## What This Changes

The issue requests that the message tool auto-select the only enabled account for a selected channel when `accountId` is omitted instead of falling back to a disabled `default` account.

## Best Possible Solution

Add a shared outbound account resolver after channel selection that uses the selected channel plugin config adapter and emits clear zero-enabled or multiple-enabled errors.

## Reproduction Assessment

Yes, at source level. With `accountId` omitted, current main leaves the shared runner without an account and WhatsApp then resolves through the default helper, so a disabled `default` plus one enabled named account still targets `default`.

## Solution Assessment

Yes. A generic resolver through the selected channel plugin config adapter is the narrowest maintainable fix; a WhatsApp-only fallback would duplicate shared routing policy and miss other plugin channels.
