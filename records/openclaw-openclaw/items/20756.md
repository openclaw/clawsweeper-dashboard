## Summary

Keep open: current main and v2026.6.8 still omit single-enabled-account inference for message sends, and https://github.com/openclaw/openclaw/pull/85671 is the active open fix candidate.

## What This Changes

The issue requests that omitted message-tool account selection prefer the only enabled configured account instead of falling through to a disabled default account.

## Best Possible Solution

Land or replace the open shared outbound account resolver so message sends preserve explicit/default/bound precedence, infer exactly one enabled configured account, and document/test ambiguous or missing account failures.

## Reproduction Assessment

Yes at source level, but not by a live WhatsApp run in this review. Current main leaves accountId unset unless it is explicit/default/bound, and WhatsApp omitted-account sends ask the default helper, so a disabled default plus one enabled named account still selects default.

## Solution Assessment

Yes, the best direction is a shared resolver after channel selection that uses plugin config adapter hooks. A WhatsApp-only fallback would be the wrong owner boundary and would miss sibling channel behavior.
