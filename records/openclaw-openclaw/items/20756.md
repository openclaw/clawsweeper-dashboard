## Summary

Keep open: current main and the latest release still do not infer the only enabled configured account when the message tool omits `accountId`, and https://github.com/openclaw/openclaw/pull/85671 is the active open fix candidate with closing syntax for this issue.

## What This Changes

The issue requests outbound message-account inference when a selected channel has exactly one enabled configured account and no `accountId` is supplied.

## Best Possible Solution

Land or replace the open shared outbound account resolver so explicit/default/bound precedence is preserved, exactly one enabled configured account is inferred, ambiguous or missing account cases fail clearly, and docs/tests cover the behavior.

## Reproduction Assessment

Yes at source level. Current main leaves `accountId` unset unless it is explicit/default/bound, and WhatsApp omitted-account sends still fall through to the default-account helper.

## Solution Assessment

Yes, the best owner boundary is a shared resolver after channel selection that uses plugin config adapter hooks. A WhatsApp-only fallback would be too narrow, and the issue should stay open until the linked PR lands or is replaced.
