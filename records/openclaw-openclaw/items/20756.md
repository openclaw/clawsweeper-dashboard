## Summary

Keep open. Current `main` still resolves the message-tool account only from an explicit `accountId`, a supplied default, or an agent binding; when that remains unset, the WhatsApp path falls back to the default-account helper without checking enabled/configured account state. The related implementation PR #20801 is closed unmerged, so there is no merged or shipped fix to close against.

## What This Changes

The issue requests that outbound `message` tool sends auto-select the only enabled account for the selected channel when `accountId` is omitted, instead of falling back to a disabled `default` account.

## Best Possible Solution

Add a shared outbound account resolver immediately after channel selection. It should preserve explicit/default/bound account IDs, then use the selected channel plugin config adapter to prefer an enabled configured default account, auto-select exactly one enabled/configured account, and fail clearly when zero or multiple eligible accounts remain.

## Reproduction Assessment

Yes. With the issue's WhatsApp config shape (`default` disabled and one named account enabled), current `main` leaves `accountId` unset in the shared runner and the WhatsApp send path resolves `default`, which leads to the reported `No active WhatsApp Web listener (account: default)` path.

## Solution Assessment

Yes. The maintainable fix is the issue's proposed behavior implemented generically through the channel plugin config adapter after channel selection, rather than a WhatsApp-only special case in the extension send helper.
