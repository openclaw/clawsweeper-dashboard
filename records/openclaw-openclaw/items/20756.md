## Summary

Keep open. Current main still leaves omitted message-tool account IDs unresolved unless they come from explicit input, a default override, or an agent binding, so WhatsApp can still fall back to a disabled `default` account when one enabled named account exists.

## What This Changes

The issue requests automatic message-tool account selection when a channel has exactly one enabled account and `accountId` is omitted.

## Best Possible Solution

Add a shared outbound account resolver after channel selection that uses the selected channel plugin config adapter, preserves existing precedence, auto-selects exactly one enabled/configured account, and reports clear ambiguity or no-account errors.

## Reproduction Assessment

Yes, at source level. On current main, omitting `accountId` leaves the shared runner without an account, and WhatsApp then resolves through the default helper, so a disabled `default` plus one enabled named account still targets `default`.

## Solution Assessment

Yes. A generic resolver through the selected channel plugin config adapter is the narrowest maintainable direction; a WhatsApp-only fallback would duplicate shared routing policy and miss other plugin channels.
