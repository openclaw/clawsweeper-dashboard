## Summary

Keep this issue open: current main and the latest release still omit single-enabled-account inference, and the open linked implementation PR has not merged yet.

## What This Changes

The issue asks outbound message sending to infer the selected channel's only enabled account when `accountId` is omitted instead of falling back to a disabled `default` account.

## Best Possible Solution

Land or replace the linked shared outbound-account resolver so explicit/default/bound account choices keep priority, exactly one enabled configured account can be inferred, and zero-or-multiple-account cases fail with clear user-facing errors.

## Reproduction Assessment

Yes at source level: with `default` disabled and one enabled named account, an omitted-account message send still leaves account inference to the default-account path. I did not run a live WhatsApp send in this read-only review.

## Solution Assessment

Yes for the owner boundary: shared account resolution after channel selection is the narrow maintainable fix because the plugin config adapter already exposes the needed account facts. The exact fail-closed compatibility policy still needs maintainer acceptance on the linked PR.
