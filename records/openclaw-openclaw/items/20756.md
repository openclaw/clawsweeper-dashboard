## Summary

Keep open: current main and the latest release still do not infer the only enabled account for omitted-account message sends, and the open fixing PR has not merged yet.

## What This Changes

The issue requests that the message tool choose the only enabled channel account when `accountId` is omitted instead of falling back to a disabled `default` account.

## Best Possible Solution

Land a refreshed shared outbound account resolver after channel selection, or replace the stale/conflicting fixing PR, with docs and tests covering default, single-enabled, zero-enabled, and multiple-enabled account cases.

## Reproduction Assessment

Yes at source level: current main leaves omitted `accountId` unresolved after explicit/default/bound checks, and WhatsApp then falls back to its default-account helper. I did not run a live WhatsApp send in this read-only sweep.

## Solution Assessment

Yes for the likely architecture: the maintainable fix belongs in a shared resolver after channel selection, using plugin config adapter hooks rather than a WhatsApp-only special case. It still needs maintainer acceptance for the changed fallback semantics and a mergeable PR.
