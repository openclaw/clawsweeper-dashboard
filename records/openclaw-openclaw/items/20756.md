## Summary

Keep open. Current main still lacks enabled-account fallback for omitted message-tool `accountId`, the direct implementation PR is closed unmerged, and the existing channel config adapter is enough for a focused shared resolver.

## What This Changes

The issue asks for the message tool to choose the only enabled account when `accountId` is omitted instead of falling back to a disabled `default` account.

## Best Possible Solution

Add a shared outbound account resolver after channel selection that preserves explicit/default/bound accounts, then chooses an enabled configured default or exactly one enabled configured account with clear zero-or-multiple-account errors.

## Reproduction Assessment

Yes, for the reported current behavior. With `channels.whatsapp.accounts.default.enabled=false` and one named account enabled, source inspection shows the runner leaves `accountId` unset and WhatsApp resolves the omitted account to `default`.

## Solution Assessment

Yes. The best repair is a generic resolver through the existing channel config adapter after channel selection; a WhatsApp-only fallback would duplicate policy and miss the shared outbound path.

## Repair Work Prompt

Fix the message tool omitted-account selection for multi-account channels. After channel selection in the shared outbound path, preserve explicit `accountId`, supplied `defaultAccountId`, and target-bound agent account IDs first; when still omitted, use the selected channel plugin config adapter to prefer an enabled configured default account, then auto-select exactly one enabled configured account, and throw clear errors for zero or multiple enabled/configured accounts. Keep the resolver generic in `src/infra/outbound` and do not hard-code WhatsApp-only behavior. Add regression coverage for default disabled plus one enabled account, default enabled, zero enabled, and multiple enabled accounts. Update WhatsApp send coverage, channel routing/message docs, and add a user-facing `CHANGELOG.md` entry. Related refs: https://github.com/openclaw/openclaw/issues/20756 and closed unmerged PR https://github.com/openclaw/openclaw/pull/20801.
