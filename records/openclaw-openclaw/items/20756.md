## Summary

Keep open. Current main and the latest release still use the omitted-account default fallback, while the active implementation candidate is open, conflicting, and unmerged.

## What This Changes

The issue requests that the message tool infer the only enabled account for a selected channel when `accountId` is omitted instead of falling back to a disabled `default` account.

## Best Possible Solution

Land or replace the linked shared outbound account resolver so explicit/default/bound account precedence is preserved, single enabled configured accounts are inferred, ambiguous cases fail clearly, and routing/message docs match the behavior.

## Reproduction Assessment

Yes, at source level: current main leaves omitted `accountId` unresolved after explicit/default/bound checks, then WhatsApp falls back to the default-account helper. I did not run a live WhatsApp send in this read-only review.

## Solution Assessment

Yes for the architecture: the maintainable fix belongs in a shared outbound account resolver after channel selection, not in a WhatsApp-only fallback. The linked PR still needs refresh and maintainer acceptance of the fallback-policy change.
