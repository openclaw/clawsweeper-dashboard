## Label Justifications

- `impact:message-loss`: The current omitted-account fallback can target a disabled default account and prevent the intended proactive message from being delivered.

## Summary

Current main still leaves omitted message-tool account selection to explicit, runner-default, or bound-account inputs. The open closing PR implements the requested shared resolver, but it is conflicting, changes documented multi-account fallback behavior, and still lacks live WhatsApp proof, so this issue remains necessary until that PR is safely landed or replaced.

## What This Changes

Automatically select the sole enabled and configured channel account when the message tool omits `accountId`, while preserving explicit and default account choices.

## Best Possible Solution

Use one shared post-channel-selection resolver that preserves explicit, runner-default, and binding choices, prefers a valid configured default, infers only an unambiguous enabled and configured account, and returns clear zero-account or ambiguous-account errors with focused tests, updated docs, and live WhatsApp proof.

## Reproduction Assessment

Yes at source level, though not through a live run: when `accountId`, the runner default, and a matching binding are absent, the shared runner leaves the account unresolved rather than selecting the sole enabled account.

## Solution Assessment

Yes at the architectural level: a shared resolver after channel selection is narrower and more consistent than a WhatsApp-only fallback, but maintainers must confirm the default-versus-ambiguity policy before landing it.
