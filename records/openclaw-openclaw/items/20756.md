## Summary

Keep open. Current main and v2026.5.22 still leave omitted outbound message `accountId` unresolved until channel-specific fallback, and an open linked PR already owns the shared resolver fix.

## What This Changes

The issue requests that outbound message actions auto-select the only enabled configured account when `accountId` is omitted instead of falling back to a disabled `default` account.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/85671 if it passes proof and maintainer review, then close this issue from the merged shared resolver implementation.

## Reproduction Assessment

Yes, at source level. Current main leaves `accountId` unset unless explicit/default/binding sources provide one, and WhatsApp then falls through to default-account resolution; no live WhatsApp send was run in this read-only review.

## Solution Assessment

Yes. The maintainable direction is a generic selected-channel account resolver using plugin adapter hooks; a WhatsApp-only fallback would duplicate routing policy and miss the same class for other multi-account channels.
