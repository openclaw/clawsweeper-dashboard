## Summary

Keep open. Current main and the latest shipped tag still leave omitted message-tool `accountId` unresolved in the shared runner, after which WhatsApp falls back to the default-account helper that does not check enabled state; the related implementation PR https://github.com/openclaw/openclaw/pull/20801 is closed unmerged.

## What This Changes

The issue asks the message tool to auto-select the only enabled account for a selected channel when `accountId` is omitted instead of falling back to a disabled `default` account.

## Best Possible Solution

Add a shared outbound account resolver after channel selection that uses plugin config adapter hooks to choose an enabled configured default or exactly one enabled configured account, with clear zero/multiple-account errors.

## Reproduction Assessment

Yes, at source level. With `accountId` omitted, current main leaves the shared runner account unset and WhatsApp resolves through `resolveDefaultWhatsAppAccountId`, so a disabled `default` plus one enabled named account still targets `default`.

## Solution Assessment

Yes. A generic resolver on the selected channel plugin config adapter is the narrow maintainable fix; a WhatsApp-only fallback would duplicate routing policy and miss the same class of bug in other plugin channels.
