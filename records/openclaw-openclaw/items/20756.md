## Summary

Keep open. Current main still has the reported account-selection gap: omitted message-tool account IDs come only from explicit/default/bound sources, and WhatsApp omitted sends still fall through to the default-account helper even when that account is disabled.

## What This Changes

Request auto-selecting the only enabled outbound message account when `accountId` is omitted and the configured `default` account is disabled.

## Best Possible Solution

Add a shared outbound account resolver after channel selection that uses the selected channel plugin config adapter, preserves existing explicit/default/binding precedence, and gives clear zero-or-multiple-enabled-account errors.

## Reproduction Assessment

Yes, at source level. With `accountId` omitted, current main leaves the shared runner without an account and WhatsApp then resolves through the default helper, so a disabled `default` plus one enabled named account still targets `default`.

## Solution Assessment

Yes. A generic resolver through the selected channel plugin config adapter is the narrowest maintainable direction; a WhatsApp-only fallback would duplicate shared outbound routing policy.
