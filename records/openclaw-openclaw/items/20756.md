## Summary

Keep open. Current `main` and the latest release still leave omitted message-tool `accountId` unresolved in the shared runner, then WhatsApp falls through to default-account resolution, so a disabled `default` plus one enabled named account can still target the disabled account.

## What This Changes

The issue requests that outbound message sends without `accountId` automatically choose the single enabled account for the selected channel when the configured default account is disabled.

## Best Possible Solution

Centralize omitted-account resolution after channel selection using the selected channel plugin adapter, preserving explicit/default/bound accounts first and covering enabled default, single enabled, zero enabled, and multiple enabled account cases in tests and docs.

## Reproduction Assessment

Yes, at source level. With `accountId` omitted, current `main` leaves the runner account unset and WhatsApp resolves through the default-account helper, so a disabled `default` plus one enabled named account still targets `default`.

## Solution Assessment

Yes. The maintainable fix is a generic selected-channel account resolver using plugin adapter hooks; a WhatsApp-only fallback would duplicate outbound routing policy and miss the same class in other multi-account channels.
