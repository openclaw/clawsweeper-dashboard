## Summary

Keep open. Current main and the latest release still do not infer the only enabled configured account for omitted message-tool `accountId`, and the linked implementation PR is still open, conflicting, and not merged.

## What This Changes

The issue requests message-tool account inference that preserves explicit/default/bound account choices, then auto-selects exactly one enabled configured account when `accountId` is omitted.

## Best Possible Solution

Land a shared selected-channel account resolver that uses plugin config adapter hooks, auto-selects exactly one enabled configured account, and documents/tests clear zero-enabled and multiple-enabled errors.

## Reproduction Assessment

Yes, at source level. With `accountId` omitted, current main leaves the runner account unset unless explicit/default/binding sources provide one, and WhatsApp then resolves through the default-account helper.

## Solution Assessment

Yes, the best fix shape is a generic selected-channel account resolver using plugin adapter hooks. A WhatsApp-only fallback would duplicate outbound routing policy and miss the same class in other multi-account channels.
