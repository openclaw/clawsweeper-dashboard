- `impact:message-loss`: The report describes omitted-account sends targeting a disabled default account, causing messages not to reach the intended WhatsApp account.

## Summary

Keep open: current main and the latest release still omit single-enabled-account inference for message sends, and the open linked pull request is the active fix path, so this issue should not close before that work lands or is replaced.

## What This Changes

Request auto-selecting the single enabled configured account for message-tool sends when `accountId` is omitted instead of falling back to a disabled default account.

## Best Possible Solution

Land a refreshed shared outbound account resolver after channel selection, with focused tests/docs and maintainer-accepted fallback semantics.

## Reproduction Assessment

Yes at source level, but not from a live WhatsApp run in this review. Current main chooses `accountId` only from explicit/default/bound sources and WhatsApp omitted-account sends call the default helper, so a disabled default plus one enabled named account still targets default.

## Solution Assessment

Yes, the best direction is a shared resolver after channel selection that preserves explicit/default/bound precedence before inferring one enabled configured account. A WhatsApp-only fallback would be the wrong owner boundary and would miss sibling channel behavior.
