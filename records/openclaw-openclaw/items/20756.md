## Summary

Keep open: current main and the latest release still leave omitted message-tool account selection to explicit/default/bound-account paths, so a disabled `default` WhatsApp account can still be selected. The remaining work is already represented by the open linked implementation PR https://github.com/openclaw/openclaw/pull/85671, which should be reviewed rather than queueing another fix.

## What This Changes

The issue requests that the outbound message tool infer the only enabled configured account when `accountId` is omitted instead of falling back to a disabled `default` account.

## Best Possible Solution

Review, refresh, and either land or reject https://github.com/openclaw/openclaw/pull/85671, preserving explicit/default/bound-account precedence before adding a generic enabled-account resolver.

## Reproduction Assessment

Yes at source level, but not by executing a live send. Current main leaves omitted `accountId` unresolved after explicit/default/bound checks, and WhatsApp then falls back to its default-account helper.

## Solution Assessment

Yes, the best direction is a shared resolver after channel selection, not a WhatsApp-only fallback. The issue should stay open while the linked PR is reviewed because the compatibility policy and proof still need maintainer acceptance.
