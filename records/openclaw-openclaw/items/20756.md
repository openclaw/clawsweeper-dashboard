## Summary

Keep open. Current main and the latest release still use the documented default-account fallback when the message tool omits accountId, and the active fix candidate at https://github.com/openclaw/openclaw/pull/85671 is still open, conflicting, and awaiting follow-up rather than merged.

## What This Changes

The issue requests auto-selecting the only enabled account for outbound message sends when the selected channel has no explicit accountId.

## Best Possible Solution

Land a refreshed shared resolver after channel selection that preserves explicit/default/bound account precedence, uses channel plugin adapter hooks for enabled/configured checks, and updates focused tests plus routing/message docs.

## Reproduction Assessment

Yes at source level: current main leaves omitted accountId unresolved after explicit/default/bound checks, then WhatsApp falls back to the default-account helper. I did not run a live WhatsApp send in this read-only sweep.

## Solution Assessment

Yes for the architecture: a shared outbound account resolver after channel selection is the right owner boundary, not a WhatsApp-only fallback. The linked PR still needs refresh/proof and maintainer acceptance of the fallback-policy change before it can be the final solution.
