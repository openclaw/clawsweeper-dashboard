## Summary

Keep open. Current main and the latest shipped tag still preserve only explicit, input-default, or target-bound account IDs in the message runner, so omitted WhatsApp `accountId` can still fall back to a disabled `default` account; the related implementation attempt at https://github.com/openclaw/openclaw/pull/20801 was closed unmerged and no canonical replacement is open.

## What This Changes

Requests message-tool account selection to choose the only enabled account when `accountId` is omitted instead of falling back to a disabled `default` account.

## Best Possible Solution

Keep this issue open and add a shared outbound account resolver after channel selection that selects an enabled configured default or exactly one enabled configured account through plugin config adapter hooks.

## Reproduction Assessment

Yes, at source level. With `accountId` omitted, current main leaves `params.accountId` unset in the shared runner and WhatsApp then resolves through `resolveDefaultWhatsAppAccountId`, so a disabled `default` plus one enabled named account still targets `default`.

## Solution Assessment

Yes. A generic resolver on the selected channel plugin config adapter is the narrow maintainable fix; a WhatsApp-only fallback would duplicate routing policy and miss the same class of bug in other plugin channels.
