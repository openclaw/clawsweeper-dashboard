## Summary

Keep open. Current main and the latest release still use explicit/default/bound account IDs only, while the open linked pull request is the active implementation candidate and should be reviewed or replaced before this issue closes.

## What This Changes

The issue requests that outbound message actions infer the only enabled account when accountId is omitted instead of falling through to a disabled default account.

## Best Possible Solution

Review and land a refreshed shared outbound account resolver that preserves explicit/default/bound account precedence, uses plugin config adapter hooks, and documents accepted zero-or-multiple-account behavior.

## Reproduction Assessment

Yes at source level, not with a live run in this read-only review. Current main leaves omitted accountId unset unless explicit/default/bound data exists, and WhatsApp omitted sends still resolve through a default-account helper that ignores enabled state.

## Solution Assessment

Yes for the owner boundary, but not yet merged. A shared resolver after channel selection is the maintainable path because it can use existing plugin adapter hooks instead of adding WhatsApp-only fallback policy.
