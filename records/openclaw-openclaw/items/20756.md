## Summary

Keep open. Current main still has the reported routing gap: omitted message-tool account IDs are preserved only from explicit/default/bound sources, then WhatsApp falls back to the default-account helper even when that account is disabled.

## What This Changes

The issue requests that outbound message actions without `accountId` choose an enabled configured default account or the sole enabled account instead of falling back to a disabled `default` account.

## Best Possible Solution

Add a shared outbound account resolver after channel selection that uses the selected channel plugin config adapter and emits clear zero-enabled or multiple-enabled errors.

## Reproduction Assessment

Yes, at source level. On current main, omitting `accountId` leaves the shared runner without an account, and WhatsApp then resolves through the default helper, so a disabled `default` plus one enabled named account still targets `default`.

## Solution Assessment

Yes. A generic resolver through the selected channel plugin config adapter is the narrowest maintainable fix; a WhatsApp-only fallback would duplicate shared routing policy and miss other plugin channels.
