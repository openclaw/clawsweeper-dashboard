## Summary

Current main and v2026.6.5 still route an omitted message-tool account through explicit/default/bound selection and WhatsApp’s default-account fallback. The issue must remain open because https://github.com/openclaw/openclaw/pull/85671 is an unmerged closing PR and is still missing accepted live behavior proof.

## What This Changes

Auto-select the sole enabled configured channel account when the message tool omits `accountId`, while preserving explicit, default, and bound account choices.

## Best Possible Solution

Keep this issue canonical until the linked PR proves a real WhatsApp send with disabled `default` and one enabled named account, then land the shared adapter-backed resolver with approved ambiguity semantics and aligned routing docs.

## Reproduction Assessment

Yes, from source with high confidence: omit `accountId` with disabled `default` and one enabled named WhatsApp account; current main leaves the account unset until WhatsApp resolves its default account, reproducing the reported routing failure.

## Solution Assessment

Yes on architecture: resolving an unambiguous enabled account through the selected channel’s config adapter is narrower and more maintainable than a WhatsApp-only workaround. The zero/multiple-account behavior still requires explicit maintainer approval before merge.
