## Summary

Keep open. Current main and the latest release still leave omitted message-tool `accountId` unresolved in the shared runner, then WhatsApp falls through to default-account resolution, so a disabled `default` plus one enabled named account can still target the disabled account; the related implementation at https://github.com/openclaw/openclaw/pull/20801 is closed unmerged.

## What This Changes

When the message tool omits `accountId`, it should use an enabled configured default or exactly one enabled account instead of falling back to a disabled `default` account.

## Best Possible Solution

Centralize omitted-account selection after channel selection using the selected channel plugin adapter, then document and test enabled default, single enabled, zero enabled, and multiple enabled account cases.

## Reproduction Assessment

Yes, at source level. With `accountId` omitted, current main leaves the runner account unset and WhatsApp resolves through the default-account helper, so a disabled `default` plus one enabled named account still targets `default`.

## Solution Assessment

Yes. The maintainable fix is a generic selected-channel account resolver using plugin adapter hooks; a WhatsApp-only fallback would duplicate routing policy and miss the same class in other multi-account channels.
