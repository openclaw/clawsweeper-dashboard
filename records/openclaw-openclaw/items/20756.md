## Summary

Keep open because current main and the latest release still lack single-enabled-account inference, and the live closing PR for the same behavior is open, unmerged, and conflicting.

## What This Changes

The issue requests omitted `accountId` handling in the outbound message tool to choose the single enabled channel account instead of falling back to a disabled `default` account.

## Best Possible Solution

Land a refreshed shared resolver after channel selection that preserves explicit/default/bound account precedence, selects exactly one enabled configured account, and documents the zero-or-multiple-account behavior.

## Reproduction Assessment

Yes at source level: omitted account selection after channel resolution has no single-enabled-account resolver, and WhatsApp omitted sends still fall back to the default-account helper. I did not run a live WhatsApp send in this read-only review.

## Solution Assessment

Mostly yes: a shared resolver after channel selection is the right owner boundary and avoids a WhatsApp-only fallback. The exact fail-closed policy for zero or multiple enabled accounts still needs maintainer acceptance because it changes omitted-account fallback semantics.
