## Label Justifications

- `P2`: This is a bounded outbound routing improvement, but it affects real proactive message delivery for multi-account channel setups.
- `impact:message-loss`: The reported behavior can route omitted-account sends to a disabled WhatsApp account, causing proactive messages not to reach the user.

## Summary

Keep open. Current main and the latest release still do not infer the only enabled account after channel selection, and https://github.com/openclaw/openclaw/pull/85671 is an open, unmerged implementation candidate rather than a merged fix.

## What This Changes

The issue asks the message tool to infer the single enabled account for the selected channel when `accountId` is omitted instead of falling back to a disabled default account.

## Best Possible Solution

Review, refresh, and land a shared post-channel-selection account resolver that preserves explicit/default/bound account ids first, selects exactly one enabled configured account, and documents clear zero-or-multiple-account errors.

## Reproduction Assessment

Yes at source level: with `default` disabled and one named WhatsApp account enabled, current main leaves `accountId` unset in the shared runner and WhatsApp send resolves the omitted account back to the default helper. I did not run a live WhatsApp send in this read-only review.

## Solution Assessment

Yes directionally: a shared resolver after channel selection is the right boundary because the adapter already exposes account listing, enabled, and configured hooks. The linked PR or a replacement still needs maintainer acceptance of the fail-closed policy and real behavior proof before this issue can close.
