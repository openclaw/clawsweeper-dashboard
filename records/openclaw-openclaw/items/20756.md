## Summary

Keep open. Current main and v2026.5.22 still leave omitted message-tool `accountId` unresolved until WhatsApp falls back to its default-account helper, and an open closing PR already owns the fix path.

## What This Changes

The issue requests that outbound message sending infer the only enabled account when `accountId` is omitted instead of falling back to a disabled `default` account.

## Best Possible Solution

Review and land the linked shared-resolver PR if it passes code review and proof, then close this issue from the merged implementation.

## Reproduction Assessment

Yes, at source level. With `accountId` omitted, current main leaves the runner account unset unless explicit/default/binding sources provide one, and WhatsApp then resolves through the default-account helper.

## Solution Assessment

Yes. The maintainable fix is a generic selected-channel account resolver using plugin adapter hooks; a WhatsApp-only fallback would duplicate outbound routing policy and miss the same class in other multi-account channels.
