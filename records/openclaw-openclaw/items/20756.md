## Summary

Keep open. Current main and v2026.6.5 still resolve omitted message-tool accounts through explicit/default/bound selection and WhatsApp’s default-account fallback, while https://github.com/openclaw/openclaw/pull/85671 is an unmerged, conflicting closing PR that still needs maintainer policy review and real behavior proof.

## What This Changes

The issue requests message-tool account inference when an omitted `accountId` leaves exactly one enabled account for the selected channel.

## Best Possible Solution

Keep this issue open as the canonical tracker until https://github.com/openclaw/openclaw/pull/85671, or a narrow replacement, lands a shared adapter-backed resolver with approved ambiguity semantics, docs, regression tests, and real delivery proof.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: omit `accountId` with disabled `default` and one enabled named WhatsApp account, and current main leaves account selection to WhatsApp’s default fallback. I did not run live WhatsApp delivery in this read-only review.

## Solution Assessment

Yes on the architectural direction: a shared resolver after channel selection using plugin config adapters is narrower than a WhatsApp-only workaround. Maintainers still need to approve the zero/multiple-account policy and land the linked PR or replacement.
