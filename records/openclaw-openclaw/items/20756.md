## Summary

Keep open: current main and v2026.6.10 still do not infer the only enabled account for omitted message-tool accountId, and https://github.com/openclaw/openclaw/pull/85671 is the active open implementation candidate. The earlier same-purpose PR was closed unmerged, so it does not resolve the issue.

## What This Changes

The issue asks the message tool to auto-select the only enabled configured channel account when accountId is omitted instead of falling back to a disabled default account.

## Best Possible Solution

Land or replace the open shared outbound account resolver so message sends preserve explicit/default/bound precedence, infer exactly one enabled configured account, and document/test ambiguous or missing account failures.

## Reproduction Assessment

Yes at source level: current main leaves accountId unset unless it is explicit/default/bound, and WhatsApp omitted-account sends then ask the default helper. I did not run a live WhatsApp send in this read-only review.

## Solution Assessment

Yes for the owner boundary: a shared resolver after channel selection uses the existing plugin config adapter seam and is better than a WhatsApp-only fallback. It still needs maintainer acceptance of the compatibility-sensitive ambiguous-account behavior before the issue can close.
