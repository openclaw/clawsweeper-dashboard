## Summary

Keep open. Current main still resolves omitted message-tool accounts only from an explicit account, a supplied default, or an agent binding, while the WhatsApp send path falls back to the default-account helper without checking enabled/configured account state; the direct implementation PR is closed unmerged.

## What This Changes

When a message send omits `accountId`, auto-select the selected channel's only enabled account instead of falling back to a disabled default account.

## Best Possible Solution

Add a shared outbound account resolver immediately after channel selection that preserves explicit/default/bound accounts, then chooses an enabled configured default or exactly one enabled configured account, with clear zero-or-multiple-account errors.

## Reproduction Assessment

Yes. With `channels.whatsapp.accounts.default.enabled=false` and one named WhatsApp account enabled, source inspection shows the shared runner leaves `accountId` unset and the WhatsApp send path resolves `default`, producing the reported default-listener error path.

## Solution Assessment

Yes. The narrow maintainable fix is a generic account resolver through the existing channel config adapter after channel selection, not a WhatsApp-only fallback in the extension send helper.
