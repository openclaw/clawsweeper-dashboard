## Summary

Keep open. Current main still resolves omitted message-tool accounts only from explicit input, the caller default, or agent bindings, while WhatsApp omitted sends still fall back to its default-account helper; https://github.com/openclaw/openclaw/pull/85671 is the active implementation candidate but remains open and unmerged.

## What This Changes

The issue requests omitted `message` tool account selection to prefer an enabled default account or exactly one enabled configured account instead of falling through to a disabled `default` account.

## Best Possible Solution

Land or replace the linked implementation with a shared outbound account resolver after channel selection that preserves explicit/default/bound accounts first, then infers exactly one enabled configured account with clear zero-account and multiple-account errors.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence reproduction path: configure WhatsApp with `default` disabled and one other enabled account, omit `accountId` in a proactive `message` send, and current code falls through to the WhatsApp default-account helper rather than selecting the enabled account.

## Solution Assessment

Yes, directionally. A shared resolver after channel selection is the maintainable fix because it can use the channel config adapter hooks and preserve explicit/default/bound account precedence; a WhatsApp-only fallback would be too narrow.
