## Summary

Keep open. Current `origin/main` still leaves omitted message-tool account IDs unresolved unless they come from explicit params, a runner default, or a target-bound binding; WhatsApp then falls through to its default-account helper, and the only related implementation PR is closed unmerged.

## What This Changes

The issue requests that the outbound message tool choose the only enabled account for a selected channel when `accountId` is omitted, instead of falling back to a disabled `default` account.

## Best Possible Solution

Add a shared outbound account resolver after channel selection that uses the selected channel plugin config adapter, prefers an enabled configured default, auto-selects exactly one enabled/configured account, and emits clear zero-or-multiple-account errors.

## Reproduction Assessment

Yes, at source level. With `accountId` omitted, current main leaves the shared runner without an account and WhatsApp then resolves through the default helper, so a disabled `default` plus one enabled named account still targets `default`.

## Solution Assessment

Yes. A generic resolver through the selected channel plugin config adapter is the narrowest maintainable fix; a WhatsApp-only fallback would duplicate shared routing policy and miss other plugin channels.
