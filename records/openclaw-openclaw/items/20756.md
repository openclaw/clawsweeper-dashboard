## Summary

Keep open. Current main still routes omitted `accountId` sends through explicit/default/bound account sources only, and WhatsApp still falls back to the default-account helper; the related implementation PR was closed unmerged and no canonical replacement is open.

## What This Changes

Request that `openclaw message` auto-select the sole enabled account for a selected channel when `accountId` is omitted instead of falling back to a disabled `default` account.

## Best Possible Solution

Add a shared outbound account resolver after channel selection that uses plugin config adapter hooks to choose an enabled configured default or exactly one enabled configured account, with focused tests and docs updates.

## Reproduction Assessment

Yes, at source level. With `accountId` omitted, current main does not set `params.accountId` in the shared runner and WhatsApp falls back through `resolveDefaultWhatsAppAccountId`, so a disabled `default` plus one enabled named account still targets `default`.

## Solution Assessment

Yes. A generic resolver through the selected channel plugin config adapter is the narrow maintainable path; a WhatsApp-only fallback would duplicate routing policy and miss other plugin channels.
