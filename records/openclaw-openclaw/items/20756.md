## Summary

Keep open. Current main still does not auto-select the only enabled account when the message tool omits `accountId`; the shared runner preserves only explicit/default/bound IDs, and WhatsApp still resolves omitted sends through the default-account helper. The closed related implementation PR is useful context, but it did not merge and no narrower canonical item has replaced this issue.

## What This Changes

The issue asks the message tool to choose the single enabled account for a selected channel when `accountId` is omitted and `default` is disabled.

## Best Possible Solution

Add a shared outbound account resolver after channel selection that preserves explicit/default/bound precedence, prefers an enabled configured default, selects exactly one enabled/configured account, and raises clear zero-or-multiple-account errors.

## Reproduction Assessment

Yes, at source level. With `accountId` omitted, current main leaves the shared runner without an account and WhatsApp resolves through the default-account helper, so a disabled `default` plus one enabled named account still targets `default`.

## Solution Assessment

Yes. A generic resolver through the selected channel plugin config adapter is the narrow maintainable path; a WhatsApp-only fallback would duplicate routing policy and miss other plugin channels.
