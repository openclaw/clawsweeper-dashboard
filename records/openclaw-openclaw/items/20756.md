## Summary

Keep open. Current main still leaves omitted message-tool `accountId` unresolved unless it comes from an explicit parameter, runner default, or agent binding; WhatsApp then falls back to the default-account helper, and the direct implementation PR #20801 is closed unmerged.

## What This Changes

Request auto-selecting the only enabled account for a selected message channel when `accountId` is omitted instead of falling back to a disabled `default` account.

## Best Possible Solution

Add a shared outbound account resolver after channel selection that preserves explicit/default/bound selections, then chooses an enabled configured default or exactly one enabled configured account with clear zero-or-multiple-enabled-account errors.

## Reproduction Assessment

Yes, source inspection gives a high-confidence path: with WhatsApp `default` disabled and one named account enabled, the runner leaves `accountId` unset and WhatsApp resolves the omitted account through the default helper. I did not run tests because this review is read-only and the code path is clear from current main.

## Solution Assessment

Yes. The maintainable fix is a generic resolver through `ChannelConfigAdapter` after channel selection; a WhatsApp-only fallback would duplicate routing policy and miss other plugin channels.
