## Label Justifications

- `P2`: The requested account-selection behavior is bounded to outbound message routing but affects real proactive message delivery for multi-account channel users.
- `impact:message-loss`: The report describes omitted-account WhatsApp sends targeting a disabled default account, so messages may not reach the only enabled account.

## Summary

Keep open: current main and v2026.6.10 still omit single-enabled-account inference, and https://github.com/openclaw/openclaw/pull/85671 is the active open fix candidate rather than a merged implementation.

## What This Changes

The issue asks the outbound message tool to select the only enabled account when `accountId` is omitted instead of falling back to a disabled `default` account.

## Best Possible Solution

Land or replace a shared outbound account resolver that preserves explicit/default/bound precedence, infers exactly one enabled configured account, and documents/tests ambiguous or missing-account failures.

## Reproduction Assessment

Yes at source level: current main leaves `accountId` unset unless it is explicit, defaulted by the caller, or target-bound, and the WhatsApp omitted-account path still falls through to the default helper. I did not run a live WhatsApp reproduction in this read-only review.

## Solution Assessment

Yes in direction: a shared resolver after channel selection is the right owner boundary because the plugin config adapter already exposes enabled/configured account hooks. It is not ready to close until the linked PR or a replacement lands and maintainers accept the compatibility-sensitive fallback policy.
