## Summary

Keep open: current main and the latest release still leave omitted `accountId` routed through default-account fallback, and the open implementation candidate at https://github.com/openclaw/openclaw/pull/85671 has not merged yet.

## What This Changes

The issue asks the outbound message tool to auto-select the only enabled configured account when `accountId` is omitted instead of falling back to a disabled `default` account.

## Best Possible Solution

Land or replace the linked shared resolver PR: preserve explicit, runner-default, and bound-account precedence; infer exactly one enabled configured account through channel adapter hooks; and document clear zero-or-multiple-account errors with focused tests and real behavior proof.

## Reproduction Assessment

Yes at source level, but not through a live run in this read-only review. Current main leaves omitted `accountId` unresolved after explicit/default/bound checks, and WhatsApp then falls back to the default-account helper that ignores enabled state.

## Solution Assessment

Yes for the owner boundary: a shared resolver after channel selection is better than a WhatsApp-only fallback. The implementation path still needs maintainer acceptance of fail-closed omitted-account semantics plus proof on the linked PR.
