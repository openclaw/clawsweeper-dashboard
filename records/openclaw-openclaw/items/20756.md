## Summary

Keep open. Current main still leaves omitted `accountId` unresolved after channel selection, so WhatsApp proactive sends can still fall through to default-account behavior even when a different named account is the only enabled one. The linked implementation PR is closed unmerged, and the latest maintainer discussion already points to a shared resolver as the remaining work.

## What This Changes

The issue requests that the outbound `message` tool automatically choose the only enabled account for the selected channel when `accountId` is omitted.

## Best Possible Solution

Add a shared outbound account resolver after channel selection that preserves explicit, tool-default, and bound account IDs first, then uses the selected plugin config adapter to prefer an enabled configured default account, choose exactly one enabled configured account, and fail clearly when zero or multiple accounts can send.
