## Summary

Keep open. Current main still has only built-in gateway roles plus scoped operator tokens as a partial workaround; custom named roles, a `gateway.roles` config surface, role assignment during pairing, and an `openclaw roles create` flow remain new security-sensitive product work.

## What This Changes

Request to add custom named gateway roles with configurable operator scope bundles, assignable during device pairing and manageable through config and CLI.

## Best Possible Solution

Decide whether custom gateway RBAC roles are desired; if yes, add a reviewed versioned role schema, role-to-scope resolver, pairing assignment semantics, CLI/docs/protocol updates, and regression tests for no privilege widening.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection is high confidence because custom role names are rejected today and no `gateway.roles` or `openclaw roles` surface exists.

## Solution Assessment

Unclear. Scoped operator tokens and separate Gateway guidance cover part of the least-privilege need, but custom named roles require a maintainer-owned authorization design rather than a narrow mechanical patch.
