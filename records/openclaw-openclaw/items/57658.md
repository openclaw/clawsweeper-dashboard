## Summary

Keep open. Current main still supports only the built-in gateway roles plus scoped operator tokens; the requested custom named roles, `gateway.roles` config, pairing-time role assignment, and `openclaw roles create` flow remain security-sensitive product work.

## What This Changes

Request custom gateway roles with user-defined scope sets, assignable during device pairing and manageable through config and CLI.

## Best Possible Solution

Keep this as maintainer-owned gateway RBAC design work; if accepted, add a versioned `gateway.roles` schema, fail-closed role-to-scope resolver, pairing assignment semantics, CLI/docs/protocol updates, and regression tests for no privilege widening.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives high confidence that custom role names are rejected today and no `gateway.roles` or `openclaw roles` surface exists.

## Solution Assessment

Unclear. The requested config and CLI shape is plausible, but custom named roles change gateway authorization semantics and need maintainer product/security direction rather than a narrow mechanical patch.
