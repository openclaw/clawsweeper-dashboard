## Summary

Keep open. Current main still has only built-in gateway roles, plus scoped operator tokens as a partial workaround; it does not provide custom named roles, a `gateway.roles` config surface, role assignment during pairing, or an `openclaw roles create` flow.

## What This Changes

Add custom gateway roles with configurable scope bundles, config/CLI surfaces, and pairing-time assignment for limited device access.

## Best Possible Solution

Decide whether scoped operator tokens are the intended RBAC model; if custom roles are desired, implement a versioned gateway role schema, role-to-scope resolver, pairing assignment, CLI management, protocol/docs updates, and regression tests for no silent privilege widening.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source-level verification is high confidence because custom role strings currently fail role parsing and targeted searches found no `gateway.roles` or roles CLI surface.

## Solution Assessment

Unclear. Scoped operator tokens already cover read-only/read-write access in part, but custom named roles require a maintainer-owned core auth design rather than a narrow mechanical patch.
