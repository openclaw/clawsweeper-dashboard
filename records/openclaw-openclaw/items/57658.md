## Summary

Keep open. Current main partially covers the limited-access use case with scoped operator device tokens and no-silent-upgrade protections, but it still lacks named custom roles, a `gateway.roles` config surface, pairing-time custom role assignment, and an `openclaw roles create` CLI flow. The remaining work is core gateway authorization and device-pairing design, so it needs maintainer product/security review rather than cleanup closure.

## What This Changes

The issue requests configurable custom gateway roles with user-defined scope bundles, pairing-time role assignment, and CLI/config management for those roles.

## Best Possible Solution

Make an explicit maintainer decision on the RBAC model. If named roles are wanted, add a versioned `gateway.roles` role-to-scope config schema, resolve custom roles into the existing operator-scope authorization path, support pairing-time role assignment and CLI management/listing, update protocol/docs, and add regression tests proving limited roles cannot gain admin, approval, pairing, or token-management privileges. If scoped operator tokens are the intended model, document that workflow and keep custom named roles out of core.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The capability gap is still high-confidence from static verification: custom role strings fail role parsing, current config types do not expose `gateway.roles`, and the CLI only supports device token rotation within already approved roles.

## Solution Assessment

Unclear until maintainers choose the product model. A thin named-role-to-scope layer over the existing operator scope system would be the narrowest maintainable implementation if custom roles are desired; otherwise the safer solution is to document and refine scoped operator device tokens instead of adding another RBAC surface.
