## Summary

Keep open. Current main still only accepts built-in Gateway roles and scoped operator tokens; configurable named roles, a `gateway.roles` config surface, pairing-time role assignment, and an `openclaw roles create` CLI remain unimplemented security-sensitive feature work.

## What This Changes

The issue requests custom Gateway roles with user-defined scope combinations in config, pairing, token rotation, and CLI workflows.

## Best Possible Solution

Define a maintainer-reviewed Gateway RBAC design first; if accepted, implement fail-closed role-to-scope config, pairing/rotation semantics, CLI/docs/protocol updates, and regression coverage.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source, docs, and tests give a high-confidence current-main check that custom role names and `gateway.roles` do not exist while scoped operator tokens do.

## Solution Assessment

Unclear until maintainers choose the authorization model. Existing scoped operator tokens cover part of the least-privilege use case, but configurable named roles affect Gateway auth, pairing, token rotation, and the documented trust boundary.
