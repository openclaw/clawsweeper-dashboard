## Summary

Keep open: the PR is a focused usage-accounting patch, but it does not fix the linked custom OpenAI-compatible provider path because it only falls back after nonzero attempt usage already exists. Current main still gates streaming usage requests for generic custom providers, and the PR's new tests mock `attemptUsage` directly instead of proving that the reported provider stream reaches embedded-runner accounting.

## What This Changes

The PR changes embedded-runner usage aggregation so zeroed final assistant usage snapshots no longer override nonzero attempt-level usage, and adds focused unit coverage for that synthetic state.

## Best Possible Solution

Fix the OpenAI-compatible streaming/request boundary that supplies attempt usage, or keep the linked issue open for the maintainer compatibility decision instead of merging a post-hoc zero-snapshot fallback as the complete fix.

## Reproduction Assessment

Yes at source level, but not through a live custom MaaS run in this read-only review. The reported config lacks streaming-usage compat opt-in, current main omits the usage request for that generic custom path, and the PR tests bypass that path by injecting `attemptUsage`.

## Solution Assessment

No. This is a plausible mitigation for a narrower synthetic state, but the best fix for the linked issue needs to make nonzero OpenAI-completions usage reach `attemptUsage` in the first place or explicitly preserve the current compat boundary.
