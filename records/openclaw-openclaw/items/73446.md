## Summary

Keep open. Current main still has no durable path-preserving Mintlify locale switcher, the docs publish pipeline still generates locale navigation from `GENERATED_LOCALES`, live docs still do not serve standalone docs-root assets, and the related fix PRs were closed unmerged because the custom-script hook and locale source of truth were not proven.

## What This Changes

The issue requests preserving the current docs page path when switching Mintlify locales instead of navigating to the target locale homepage.

## Best Possible Solution

Keep this as the canonical tracker until maintainers either get Mintlify to preserve language-switch paths upstream or wire a supported docs publish/config hook through the sync pipeline that derives locale mappings from `GENERATED_LOCALES` and is verified in a Mintlify preview or live deployment.

## Reproduction Assessment

Yes. The issue gives precise live-docs steps and DOM inspection, #34021 reports the same user-visible behavior, and live checks confirm the English and Chinese target pages exist while the proposed standalone asset path is not served.

## Solution Assessment

Unclear. The desired behavior is valid, but the standalone docs-root JavaScript approach rejected in #73415 and #73436 is not the best maintainable fix until the Mintlify deployment hook is proven; the safer path is upstream Mintlify behavior or a publish-pipeline integration using the generated locale source of truth.
