## Summary

Keep open. Current main and the live docs deployment still do not show a durable path-preserving Mintlify locale switcher, and the related standalone-script PRs were closed rather than merged because the deployment hook and locale source of truth were not proven.

## What This Changes

The issue asks for the Mintlify docs language switcher to preserve the current page path when switching between English and localized docs pages.

## Best Possible Solution

Keep this as the canonical tracker until maintainers either get Mintlify to preserve language-switch paths upstream, or wire a supported publish/config hook through the docs sync pipeline that derives locale mappings from `GENERATED_LOCALES` and is verified in a Mintlify preview or live docs deployment.

## Reproduction Assessment

Yes. The issue gives precise live-docs steps and a Puppeteer DOM inspection path, #34021 reports the same behavior, and current source plus live asset checks show no deployed path-preserving handler; final fix validation should click from `/install/docker` to Simplified Chinese and confirm `/zh-CN/install/docker`.

## Solution Assessment

Unclear. The rejected standalone docs-root JavaScript approach is not the best maintainable fix until the Mintlify deployment hook is verified; the safer direction is an upstream Mintlify fix or a publish-pipeline integration that uses the generated locale source of truth.
