## Summary

Keep open. Current main still has no durable path-preserving Mintlify locale switcher, the live docs still do not serve standalone docs-root JS/CSS assets, and the related standalone-script PRs were closed unmerged because the deploy hook and generated locale source of truth were not proven.

## What This Changes

The issue requests preserving the current docs page path when users switch locales through the Mintlify docs language dropdown.

## Best Possible Solution

Keep this as the canonical tracker until maintainers either get Mintlify to preserve locale-switch paths upstream or wire a supported docs publish/config hook through the sync pipeline, deriving locale mappings from `GENERATED_LOCALES` and validating in a Mintlify preview or live deployment.

## Reproduction Assessment

Yes. The issue gives exact live-docs steps and DOM details, the older #34021 report describes the same behavior, and live checks confirm the English and Chinese target pages exist while proposed standalone asset paths are not served.

## Solution Assessment

Unclear. The desired behavior is valid, but the rejected standalone docs-root JavaScript approach is not a maintainable fix unless the Mintlify deployment hook is proven; the safer path is an upstream Mintlify fix or publish-pipeline integration.
