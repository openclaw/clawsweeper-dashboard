## Summary

Keep open. Current main still lacks a durable path-preserving Mintlify locale switcher, the live docs deployment still does not serve docs-root JavaScript/CSS assets directly, and the two attempted standalone-script PRs were closed rather than merged because the deployment hook and generated locale source of truth were not proven.

## What This Changes

The issue asks for the Mintlify docs language switcher to keep the current page path when switching between English and localized docs pages.

## Best Possible Solution

Keep this as the canonical tracker until maintainers either get Mintlify to preserve language-switch paths upstream or wire a supported publish/config hook through the docs sync pipeline that derives locale mappings from `GENERATED_LOCALES` and is verified in a Mintlify preview or live deployment.

## Reproduction Assessment

Yes. The issue gives precise live-docs steps and a Puppeteer DOM inspection path, #34021 reports the same behavior, and live checks confirm the relevant English and Chinese pages exist while standalone docs-root assets remain 404.

## Solution Assessment

Unclear. The rejected standalone docs-root JavaScript approach is not the best maintainable fix until the Mintlify deployment hook is verified; the safer direction is an upstream Mintlify fix or a publish-pipeline integration that uses the generated locale source of truth.
