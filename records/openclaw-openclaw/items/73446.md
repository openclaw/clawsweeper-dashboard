## Summary

Keep open. Live docs still reproduce the path-loss behavior, current main has no durable path-preserving locale switcher, and the prior standalone-script fixes were closed because the supported Mintlify/publish integration point is unresolved.

## What This Changes

The issue asks OpenClaw docs to preserve the current page path when the Mintlify language dropdown switches locales instead of navigating to the target locale homepage.

## Best Possible Solution

Keep this as the canonical tracker until maintainers either get Mintlify to preserve locale-switch paths upstream or wire a supported publish/config hook that derives locale mappings from `GENERATED_LOCALES` and is verified in a Mintlify preview or live deployment.

## Reproduction Assessment

Yes. A live Chromium replay on May 13, 2026 opened /install/docker, switched to 简体中文, and landed on /zh-CN even though /zh-CN/install/docker returns 200.

## Solution Assessment

Unclear. The requested behavior is correct, but the earlier standalone docs-root JavaScript approach is not the best fix unless OpenClaw proves a supported Mintlify asset hook; upstream Mintlify behavior or publish-pipeline integration is safer.
