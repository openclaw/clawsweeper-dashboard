## Summary

Keep open. Current main still lacks a durable path-preserving Mintlify locale switcher, the live docs deployment still does not serve standalone docs-root JavaScript/CSS assets, and the prior fix PRs were closed rather than merged because the deployment hook and generated locale source of truth were not proven.

## What This Changes

The issue asks for the Mintlify docs language switcher to preserve the current page path when changing locales instead of navigating to the target locale homepage.

## Best Possible Solution

Keep this as the canonical tracker until maintainers either get Mintlify to preserve language-switch paths upstream or wire a supported publish/config hook through the docs sync pipeline that derives locale mappings from `GENERATED_LOCALES` and is verified in a Mintlify preview or live deployment.

## Reproduction Assessment

Yes. The issue gives precise live-docs steps and a Puppeteer DOM inspection path, #34021 reports the same behavior, and live checks confirm the relevant English and Chinese pages exist while the proposed standalone script path is not served.

## Solution Assessment

Unclear. The rejected standalone docs-root JavaScript approach is not the best maintainable fix until the Mintlify deployment hook is verified; the safer direction is an upstream Mintlify fix or a publish-pipeline integration that uses the generated locale source of truth.
