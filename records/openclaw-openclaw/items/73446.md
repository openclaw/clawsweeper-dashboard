## Summary

Keep open. Current main still relies on Mintlify's built-in locale picker, and a live Chromium replay still lands on the Chinese homepage instead of the equivalent localized Docker page.

## What This Changes

The issue asks for the docs language switcher to preserve the current page path when changing locales.

## Best Possible Solution

Keep this as the canonical tracker until maintainers either get Mintlify to preserve locale-switch paths upstream or wire a supported publish/config hook that derives locale mappings from `GENERATED_LOCALES` and is verified in preview or live docs.

## Reproduction Assessment

Yes. A live Chromium replay on current docs opened `/install/docker`, selected Simplified Chinese in the Mintlify dropdown, and landed on `/zh-CN` while `/zh-CN/install/docker` returned HTTP 200.

## Solution Assessment

Unclear. The desired behavior is right, but the prior standalone docs-root JavaScript approach is not the safest fix unless OpenClaw first proves how Mintlify serves and configures that hook; an upstream Mintlify fix or supported publish-pipeline integration is safer.
