## Summary

Keep open. Current main still relies on Mintlify's built-in locale picker and docs publish config without a durable path-preserving hook, and the prior standalone-script approach remains unproven because docs-root JS/CSS is not served on the live site.

## What This Changes

The issue asks for the Mintlify docs language switcher to preserve the current page path when switching locales instead of landing on the target locale homepage.

## Best Possible Solution

Keep this as the canonical tracker until Mintlify preserves locale-switch paths upstream or OpenClaw wires a supported publish/config hook derived from GENERATED_LOCALES and validates it in preview or live docs.

## Reproduction Assessment

Yes. The issue body and prior ClawSweeper live Chromium replay provide a high-confidence path, and current live checks confirm both the English and localized Docker pages exist while the standalone docs-root script path is still not served.

## Solution Assessment

Unclear. The desired behavior is right, but the standalone JS-file approach from the closed PRs is not the best fix unless the Mintlify serving/config hook is proven; upstream Mintlify behavior or publish-pipeline integration is safer.
