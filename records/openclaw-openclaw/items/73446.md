## Summary

Keep open. Current live docs still reproduce the path-loss behavior, and current main only generates locale navigation for Mintlify; it has no durable path-preserving language-switch hook or served custom-script integration.

## What This Changes

The issue asks the docs language switcher to preserve the current page path when changing locales, such as `/install/docker` to `/zh-CN/install/docker`.

## Best Possible Solution

Keep this as the canonical tracker until Mintlify preserves locale-switch paths upstream or OpenClaw wires a supported publish-pipeline hook that derives locale mappings from `GENERATED_LOCALES` and is verified in a Mintlify preview or live deployment.

## Reproduction Assessment

Yes. A live Chromium replay against `docs.openclaw.ai` reproduces the issue by switching from `/install/docker` to Simplified Chinese and landing on `/zh-CN` instead of `/zh-CN/install/docker`.

## Solution Assessment

Unclear. The requested behavior is right, but the earlier standalone docs-root JavaScript approach is not the best fix unless OpenClaw proves a supported Mintlify serving/configuration path; an upstream Mintlify fix or publish-pipeline integration is safer.
