## Summary

Keep open: current main and the latest release still leave the raw `openclaw nodes invoke --command browser.proxy` path on generic invoke/proxy timeout budgets, while the Browser tool path already has a longer Chrome MCP existing-session management budget. The FSB extension-bridge comments are useful comparison context, but they are external and do not replace the supported OpenClaw node browser proxy behavior.

## What This Changes

The issue asks OpenClaw to make direct node `browser.proxy` calls to a Chrome MCP `user` existing-session profile wait long enough for the consent flow instead of failing before the dialog appears.

## Best Possible Solution

Align the raw node `browser.proxy` path with the existing Browser tool Chrome MCP management timeout behavior, including gateway/CLI wait budget and node-host browser proxy params, without adding a new config option or changing managed-browser defaults.

## Reproduction Assessment

Yes from source inspection: the issue's raw `nodes invoke` path can put only a 15s top-level node timeout on the request while the node-host browser plugin receives no `browser.proxy` timeout and falls back to 20s. I did not run a live Chrome MCP approval dialog repro in this read-only review.

## Solution Assessment

Yes: the best fix is a narrow timeout propagation/default alignment in the browser plugin and direct node-invoke boundary. Replacing the flow with an extension bridge would be a different product path and is not needed to fix this supported route.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/56118. Current main gives Browser-tool Chrome MCP existing-session management actions a 45s browser proxy timeout plus 5s gateway slack, but direct `openclaw nodes invoke --command browser.proxy --params '{"path":"/tabs","query":{"profile":"user"},"method":"GET"}'` defaults to a 15s top-level CLI/node invoke timeout and the node-host `browser.proxy` handler falls back to 20s because it receives no params timeout. Make raw direct node `browser.proxy` calls that target the `user`/Chrome MCP existing-session path use a consent-safe timeout across both the gateway/CLI wait and the node-host browser proxy dispatch. Keep the fix inside the browser plugin/direct node proxy boundary; do not add a new config option, do not change managed `openclaw` profile defaults, do not restore the Chrome extension relay, and do not broaden unrelated node APIs. Add focused regression coverage for the direct node-host command and direct `nodes invoke`/gateway propagation shape. Do not edit CHANGELOG.md.
