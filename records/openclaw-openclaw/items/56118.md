## Summary

Keep open: current main still leaves the reporter's direct `openclaw nodes invoke --command browser.proxy` path with a shorter CLI/proxy timeout than the Chrome MCP existing-session attach path, even though the normal browser tool has since gained a longer existing-session node-proxy budget.

## What This Changes

The issue asks OpenClaw to let remote node `browser.proxy` requests to the Chrome MCP `user` existing-session profile wait long enough for the Chrome consent flow instead of failing with `Connection closed`.

## Best Possible Solution

Align direct `node.invoke` browser-proxy timeout and abort plumbing with the existing-session attach budget, preserve the newer browser-tool 45s manage behavior, and add focused regression coverage for `profile=user` tab/manage routes through a node proxy.

## Reproduction Assessment

Yes, source-reproducible with high confidence. The issue gives concrete remote gateway/node steps, and current main still has 15s/20s direct invoke and node-proxy budgets around a 30s Chrome MCP attach path, although I did not run the live remote-node consent flow.

## Solution Assessment

Yes. The narrow maintainable fix is to align timeout and abort propagation at the browser plugin/node-proxy boundary; MCP session reuse or pre-authorization would be larger product work.
