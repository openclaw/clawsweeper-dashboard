## Summary

Keep open: current main still leaves the reported direct `openclaw nodes invoke --command browser.proxy` route on shorter CLI, node-invoke, and browser-proxy timeout budgets than the browser tool's existing-session node route. The merged browser-node routing work fixed an earlier hard block, but it does not close this direct proxy consent-timeout path.

## What This Changes

The issue asks OpenClaw to make direct node browser proxy access to a Chrome MCP `user`/existing-session profile wait long enough for Chrome's consent dialog and attach handshake.

## Best Possible Solution

Implement browser-owned timeout normalization for direct `browser.proxy` node invokes so command params, node invoke, and CLI transport share a consent-safe budget for Chrome MCP existing-session attach; leave MCP session reuse, pre-authorization, and extension-bridge designs as separate product work.

## Reproduction Assessment

Yes, source-reproducible with high confidence. The reporter provides a concrete direct node invoke command, and current main still routes that path through 15000 ms node, 30000 ms transport, and 20000 ms proxy budgets while the browser tool's existing-session node path uses 45000/50000 ms.

## Solution Assessment

Yes. The narrow maintainable fix is timeout alignment at the direct browser-proxy node boundary; MCP session reuse, pre-authorization, or the FSB extension bridge are broader product alternatives.
