## Summary

Keep open: current main and the latest release still leave the direct `openclaw nodes invoke --command browser.proxy` path on shorter generic node/proxy timeout budgets, while the Browser tool’s node route already gives Chrome MCP existing-session management calls a longer consent-safe budget. The FSB extension-bridge comments are useful comparison context, but they are external and do not replace the supported OpenClaw Browser plugin/node proxy fix.

## What This Changes

The issue reports that direct remote `browser.proxy` node invocation against `profile="user"` closes before the Chrome MCP consent dialog appears, even though local gateway access works.

## Best Possible Solution

Make direct `browser.proxy` node invokes for existing-session profiles use the same consent-safe request and transport budgets as the Browser tool, with diagnostics and tests, without increasing unrelated node invoke defaults.

## Reproduction Assessment

Yes, at source level. The reporter gives a concrete direct `nodes invoke` command, and current main still routes that path through shorter CLI, gateway, and node-host browser proxy timeout budgets than the Browser tool's existing-session node route.

## Solution Assessment

Yes. Aligning timeout propagation at the direct browser-proxy node boundary is narrower and more maintainable than MCP session reuse, pre-authorization, or replacing the path with an extension bridge.
