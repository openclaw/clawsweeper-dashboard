## Summary

Keep this open: current main still has the reported direct `openclaw nodes invoke --command browser.proxy` timeout mismatch for Chrome MCP existing-session profiles, and no merged canonical fix was found in the provided discussion or local/live checks.

## What This Changes

The issue reports that direct node browser proxy access to a Chrome MCP `user` profile fails with `Connection closed` before Chrome can show the consent dialog, while local gateway access and managed profiles work.

## Best Possible Solution

Fix the bundled browser plugin/node-proxy boundary so direct `browser.proxy` manage and tab routes for existing-session profiles get aligned consent-aware timeouts and abort propagation.

## Reproduction Assessment

Yes. The issue gives concrete remote gateway/node `browser.proxy` steps, and current source still shows a 15s CLI invoke default and 20s node-proxy default around a 30s Chrome MCP attach flow.

## Solution Assessment

Yes. The narrowest maintainable fix is to align browser plugin/node-proxy timeout policy and pass timeout/abort context into the direct Chrome MCP tab/manage path; local MCP session reuse or pre-authorization would be larger product changes.
