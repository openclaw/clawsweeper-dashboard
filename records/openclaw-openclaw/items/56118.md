## Summary

Keep open: current `main` and v2026.5.19 still leave the reporter's direct `openclaw nodes invoke --command browser.proxy` path on shorter node/proxy budgets than the browser tool's Chrome MCP existing-session node path. The merged routing work in https://github.com/openclaw/openclaw/pull/68891 fixed a related hard-block, and the FSB comment is a useful comparison point, but neither resolves this direct-invoke timeout mismatch.

## What This Changes

The issue asks OpenClaw to make direct `openclaw nodes invoke --command browser.proxy` work for node-hosted Chrome MCP `profile=user` sessions without the consent attach flow failing before the dialog appears.

## Best Possible Solution

Normalize the browser-owned timeout path for direct `browser.proxy` node invokes so command params, node invoke, and CLI transport share a Chrome MCP existing-session attach budget, while leaving broader node timeout defaults and alternate extension-bridge designs separate.

## Reproduction Assessment

Yes, source-reproducible with high confidence. The reporter gives concrete remote gateway/node steps, and current main still routes direct `browser.proxy` invokes through 15000 ms node, 30000 ms transport, and 20000 ms browser-proxy budgets while the browser tool uses 45000/50000 ms for existing-session node attach.

## Solution Assessment

Yes. The narrow maintainable fix is timeout alignment at the direct browser-proxy node boundary; MCP session reuse, pre-authorization, or an extension bridge like FSB are larger product/architecture directions rather than the first bug fix.
