## Summary

Keep this open. Current main still has code evidence that the direct remote-node `browser.proxy` `/tabs?profile=user` path can time out before the Chrome MCP existing-session consent flow finishes; recent improvements cover higher-level browser tool/manage calls and Chrome MCP diagnostics, but they do not prove the exact direct `openclaw nodes invoke` path is fixed.

## What This Changes

The issue requests that node-hosted `browser.proxy` calls to Chrome MCP `profile=user` wait long enough for an existing Chrome session consent dialog when invoked through a remote node.

## Best Possible Solution

Fix this in the bundled browser plugin/node-proxy boundary: give direct `browser.proxy` existing-session manage routes such as `GET /tabs` a consent-aware budget, align the gateway `node.invoke` timeout with the inner proxy timeout, and propagate timeout/abort options through Chrome MCP reachability and tab listing.

## Reproduction Assessment

Yes. The issue gives a concrete remote gateway/node `openclaw nodes invoke --command browser.proxy` reproduction, and current source shows the direct path still has 15s/20s outer boundaries around a 30s Chrome MCP attach flow.

## Solution Assessment

Yes, the narrow maintainable direction is to fix the browser plugin’s node-proxy timeout policy and Chrome MCP list/reachability plumbing. Reusing a local gateway MCP session would be a larger behavior change and is not necessary to make direct remote-node proxy calls wait correctly.
