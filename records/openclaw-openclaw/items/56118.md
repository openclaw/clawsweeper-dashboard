## Best Possible Solution

Normalize direct `browser.proxy` node-invoke timeout propagation for existing-session browser profiles so CLI transport, gateway node invoke, and node-host proxy dispatch share the consent-safe browser budget without changing unrelated node commands.

The observable bug is that `openclaw nodes invoke --command browser.proxy --params '{"path":"/tabs","query":{"profile":"user"},"method":"GET"}'` uses the generic node invoke/proxy budgets instead of the consent-safe existing-session browser budget, so Chrome MCP autoConnect can fail before the user can approve the prompt. Keep the fix narrow: propagate or derive the browser existing-session manage timeout for direct `browser.proxy` requests targeting `profile="user"` or other existing-session profiles, include gateway/CLI/node-host slack as needed, and do not raise timeouts for unrelated node commands. Add focused tests for direct nodes invoke/browser.proxy timeout propagation and the browser tool route, update docs/changelog if the user-visible timeout behavior changes, and stop if the implementation turns into a new config option or browser backend policy decision.
