## Summary

Keep open: current main and the latest release still leave the reporter's direct `openclaw nodes invoke --command browser.proxy` path with 15s CLI and 20s node-host proxy budgets around a 30s Chrome MCP existing-session attach. The normal browser tool has a longer `profile="user"` node-proxy budget, but that does not cover the direct node invocation described here.

## What This Changes

The issue asks OpenClaw to make direct node-host `browser.proxy` calls to Chrome MCP `profile="user"` wait long enough for the existing-session consent flow to appear and complete.

## Best Possible Solution

Align direct `node.invoke` browser-proxy timeout and abort plumbing with the existing-session attach budget, preserve the browser tool's newer 45s manage behavior, and add focused regression coverage for `profile="user"` tabs through a node proxy.

## Reproduction Assessment

Yes, source-reproducible with high confidence. The issue provides concrete remote gateway/node steps, and current main still maps that direct command to 15s/20s proxy budgets around a 30s Chrome MCP attach path, although I did not run the live remote-node consent flow.

## Solution Assessment

Yes. The narrow maintainable fix is to align timeout and abort propagation at the direct node proxy boundary; MCP session reuse or pre-authorization would be larger product work outside this bug fix.
