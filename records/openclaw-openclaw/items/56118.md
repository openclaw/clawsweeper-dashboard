## Summary

Keep open: current main still leaves the reporter's direct `openclaw nodes invoke --command browser.proxy` path on shorter node/proxy budgets than the browser tool's Chrome MCP existing-session node route, so the `profile=user` consent-flow failure remains source-reproducible. The merged browser-node routing work in https://github.com/openclaw/openclaw/pull/68891 fixed the earlier hard block, and the FSB Chrome-extension/MCP-bridge comment is useful comparison context, but neither closes this direct node-proxy timeout bug.

## What This Changes

The issue asks for Chrome MCP existing-session `profile=user` to work through direct `openclaw nodes invoke --command browser.proxy` on a remote browser node instead of closing before Chrome consent appears.

## Best Possible Solution

Implement browser-owned timeout normalization for direct `browser.proxy` node invokes so command params, node invoke, and CLI transport share a consent-safe budget for Chrome MCP existing-session attach, while leaving broader MCP session reuse or extension-bridge designs as separate product work.

## Reproduction Assessment

Yes, source-reproducible with high confidence. The reporter provides a concrete remote gateway/node command, and current main still routes direct `browser.proxy` invokes through 15000 ms node, 30000 ms transport, and 20000 ms proxy budgets while the browser tool's existing-session node path uses 45000/50000 ms.

## Solution Assessment

Yes. The narrow maintainable fix is timeout alignment at the direct browser-proxy node boundary; MCP session reuse, pre-authorization, the FSB extension bridge, or broader node timeout config are larger separate work.
