## Summary

Keep open: current main and v2026.5.19 still leave the reporter's direct `openclaw nodes invoke --command browser.proxy` path on shorter budgets than the browser tool's Chrome MCP existing-session node path, so the documented node-hosted `user` profile attach flow remains source-reproducible as a direct-invoke timeout mismatch.

## What This Changes

The issue asks for direct node `browser.proxy` calls to wait long enough for Chrome MCP existing-session consent when using the `user` profile through a remote node.

## Best Possible Solution

Implement a browser-owned timeout normalization for direct `browser.proxy` node invokes so command params, node invoke, and CLI transport share a budget that covers Chrome MCP existing-session attach; keep broader node timeout config and multi-step recovery separate.

## Reproduction Assessment

Yes, source-reproducible with high confidence. The reporter provides concrete remote gateway/node steps, and current main still routes the direct `browser.proxy` invoke through 15000 ms node and 30000 ms transport budgets while the existing-session browser tool path uses 45000/50000 ms.

## Solution Assessment

Yes. The narrow maintainable fix is timeout alignment at the direct browser-proxy node boundary; MCP session reuse, pre-authorization, broad node timeout config, or multi-step existing-session recovery are larger separate work.
