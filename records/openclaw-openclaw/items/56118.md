## Summary

Keep open: current main still has the direct `openclaw nodes invoke --command browser.proxy` timeout mismatch for Chrome MCP existing-session profiles, and no merged canonical fix was found for this remote node proxy path.

## What This Changes

The issue requests fixing node-host `browser.proxy` so Chrome MCP existing-session `user` profiles can attach through a remote browser node without the consent flow being cut off by shorter proxy/CLI timeouts.

## Best Possible Solution

Align the direct node browser proxy and browser tool timeout/abort plumbing with the Chrome MCP attach budget, then add focused regression coverage for existing-session `user` profile `browser.proxy` tab/manage routes.
