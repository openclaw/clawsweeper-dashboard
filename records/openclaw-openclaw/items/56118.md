## Summary

Keep open: current main and the latest release still leave the reporter's direct `openclaw nodes invoke --command browser.proxy` path with shorter CLI/node-proxy budgets than the Chrome MCP existing-session attach path. The higher-level browser tool has a longer existing-session timeout, but that does not cover the direct node invocation reported here.

## What This Changes

The issue reports that direct node browser-proxy invocation for Chrome MCP existing-session user profiles fails before Chrome shows the remote-debugging consent dialog.

## Best Possible Solution

The product should let the direct browser-proxy node path wait through the Chrome MCP attach and consent budget, while preserving the existing browser tool behavior and clear timeout diagnostics.

## Reproduction Assessment

Yes, source-reproducible with high confidence. The reporter provides concrete remote gateway/node steps, and current main still routes direct `openclaw nodes invoke --command browser.proxy` through 15s/20s budgets around a 30s Chrome MCP attach path, although I did not run the live remote-node consent flow.

## Solution Assessment

Yes. The narrow maintainable fix is to align timeout and abort propagation at the direct node browser-proxy boundary; MCP session reuse or pre-authorization would be larger product work outside this bug fix.
