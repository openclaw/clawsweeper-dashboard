## Summary

Keep open: current main still supports the reported node-host browser proxy path, but direct `openclaw nodes invoke --command browser.proxy` still bypasses the longer existing-session timeout budget that the browser tool uses for `profile="user"` through a browser node. The FSB comments are useful comparison context for a different extension/MCP bridge approach, but they do not provide an OpenClaw fix for this direct proxy path.

## What This Changes

The issue requests that direct node-host `browser.proxy` calls to the Chrome MCP `user` profile allow enough time for the existing-session consent flow instead of failing before the Chrome dialog appears.

## Best Possible Solution

Normalize the direct `browser.proxy` node-invoke timeout path for existing-session browser profiles so CLI transport, gateway node invoke, and node-host proxy dispatch share a consent-safe budget without changing unrelated node commands.

## Reproduction Assessment

Yes, source-reproducible with high confidence. The reporter gives a concrete direct `nodes invoke` command, and current main still routes that path through shorter CLI, gateway, and node-host proxy timeout budgets than the browser tool's existing-session node route.

## Solution Assessment

Yes. The narrow maintainable fix is timeout alignment at the direct browser-proxy node boundary; MCP session reuse, pre-authorization, or an extension bridge are broader product alternatives.
