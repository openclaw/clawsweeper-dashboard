## Summary

Keep this PR open: it targets a real ACPX diagnostic bug that current main still has, but the proposed path guard has a Windows escape hole and the PR still lacks after-fix real ACP run proof.

## What This Changes

The PR adds an ACPX `runTurn` wrapper that reads `eventLog.active_path`, validates it under a trusted sessions base, creates the parent directory, appends projected NDJSON frames per yielded event, and adds focused tests.

## Best Possible Solution

Land an ACPX event-log writer with platform-safe path containment, a maintainer-accepted stream format, focused regression coverage, and redacted real ACP run proof.

## Reproduction Assessment

Yes at source level: current main advertises `eventLog.active_path` through the pinned `acpx` package while OpenClaw's ACPX wrapper delegates `runTurn` without writing that path. I did not run a live ACP session in this read-only review.

## Solution Assessment

No, not as currently patched. The OpenClaw-side hook is plausible under the current `acpx` public runtime contract, but the path containment check needs a platform-safe fix and maintainers still need to accept the projected stream format.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Block Windows absolute paths outside the event-log base:** `extensions/acpx/src/runtime.ts:902`
  - body: The trusted-base check only rejects `relative()` results starting with `..` or `/`. On Windows, a different-drive path like `D:\outside\file` remains absolute after `path.relative(...)` and passes this check, so a corrupted session record can still redirect writes outside `~/.acpx/sessions`. Use a platform-aware absolute-path check such as `path.isAbsolute(rel)` and add a win32 regression before accepting `active_path`.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff adds a new file-write sink and its containment check misses Windows cross-drive absolute paths.

Concerns:

- **[medium] Windows cross-drive active_path bypass:** `extensions/acpx/src/runtime.ts:902`
  - body: `path.relative(trustedBase, resolved)` can return an absolute drive path on Windows when `resolved` is on another drive; the current `startsWith("..") || startsWith("/")` check does not reject that value before `fs.appendFile`.
  - confidence: 0.91
