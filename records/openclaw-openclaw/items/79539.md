## Summary

Keep open. The PR targets a real acpx diagnostics gap, but the patch does not yet reliably create the advertised stream, does not emit valid JSON-RPC session/update frames, trusts a persisted path, and lacks required after-fix real behavior proof.

## What This Changes

The PR adds an acpx runtime wrapper that reads `eventLog.active_path` from the loaded session record, appends a JSONL frame for each yielded ACP runtime event, and adds unit tests for that stream.

## Best Possible Solution

Use an acpx-compatible writer boundary that derives a safe per-session path, creates the parent directory, emits valid JSON-RPC session/update lines, and covers both real filesystem behavior and the accepted lossy/raw semantics.

## Reproduction Assessment

Yes, at source level: current main advertises acpx session event-log metadata through the dependency but `AcpxRuntime.runTurn` only delegates the event stream and never writes the advertised log. I did not run a live ACP session in this read-only review.

## Solution Assessment

No. The PR is aimed at the right extension boundary, but the direct append is incomplete because it misses directory creation, emits a non-JSON-RPC frame shape, trusts persisted path metadata, and still needs real after-fix proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Create the event stream directory before appending:** `extensions/acpx/src/runtime.ts:893`
  - body: The PR's own live repro starts with `/home/codeclaw/.acpx/sessions/` missing, but this code calls `appendFile` on the advertised file directly. `appendFile` will fail with `ENOENT` when the parent directory does not exist, and the catch only logs the error, so the advertised stream can remain missing while the mocked unit test passes. Create the parent directory or delegate to the acpx event writer before appending.
  - confidence: 0.93
- **[P2] Emit valid JSON-RPC session/update frames:** `extensions/acpx/src/runtime.ts:890-892`
  - body: The advertised stream is an ACP event log, and acpx's own writer validates JSON-RPC messages. This frame only writes `{ params: { update: ... } }`, omitting `jsonrpc: "2.0"` and `method: "session/update"`, so consumers that read the stream as JSON-RPC notifications will not recognize these lines even though the new test only checks `params.update.sessionUpdate`.
  - confidence: 0.9
- **[P2] Derive or validate the event log path before writing:** `extensions/acpx/src/runtime.ts:881-893`
  - body: This appends to whatever `eventLog.active_path` is stored in the session record. A stale or edited record can redirect writes outside the acpx session directory; the dependency's own writer derives the active path from the record id before writing. Derive the path from the trusted session id or validate it stays under the expected acpx session directory before honoring it.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds no dependency or workflow risk, but it introduces a file-write sink from persisted ACP session metadata that should be path-derived or validated.

Concerns:

- **[medium] Unvalidated event log path write:** `extensions/acpx/src/runtime.ts:893`
  - body: `eventLog.active_path` is read from the session record and used directly as an append target. If the record is stale, corrupted, or writable by an unexpected actor, this can append event data to an unintended path as the gateway user; derive the path from the session id or constrain it to the expected acpx session directory.
  - confidence: 0.78
