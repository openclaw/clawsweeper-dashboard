## Summary

Keep open: the reported ACPX event-log bug is real and current main still lacks a writer, but this PR has a blocking cross-platform path-guard issue, lacks after-fix real ACP run proof, and still needs maintainer acceptance of projected rather than raw event-log frames.

## What This Changes

The PR adds an ACPX `runTurn` JSONL writer for the session record's advertised `eventLog.active_path`, constrains the path to a trusted sessions directory, and adds focused runtime tests.

## Best Possible Solution

Implement the event-log writer with a cross-platform base-path guard and maintainer-approved stream semantics, then land it with redacted real ACP run proof showing the file and JSONL contents.

## Reproduction Assessment

Yes at source level: current main advertises `eventLog.active_path` through the pinned ACPX package, while OpenClaw's `AcpxRuntime.runTurn` delegates without writing that file. I did not run a live ACP session in this read-only review.

## Solution Assessment

No, not yet. The extension-level hook is a plausible narrow fix, but the path guard needs a drive-aware check, the projected frame format needs maintainer acceptance or documentation, and the PR still needs real after-fix ACP proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Block cross-drive event log paths:** `extensions/acpx/src/runtime.ts:901-902`
  - body: The containment check treats any `path.relative()` result that does not start with `..` or `/` as inside the trusted base. On Windows, a different-drive absolute path such as `D:\\temp\\x.stream.ndjson` returns `D:\\temp\\x.stream.ndjson`, so a manipulated session record can still steer writes outside `~/.acpx/sessions`; use a drive-aware `path.isAbsolute(rel)`/`rel === ".." || rel.startsWith(".." + sep)` check and add a win32 regression.
  - confidence: 0.9

## Security Review
