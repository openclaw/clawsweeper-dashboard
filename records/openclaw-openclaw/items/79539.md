## Summary

Keep open: the reported ACPX diagnostic gap is still real, but this PR is not merge-ready because it writes only the legacy `runTurn` path while current OpenClaw manager traffic prefers `startTurn`, and it records projected OpenClaw events under an ACPX-advertised stream path instead of real ACP `session/update` frames.

## What This Changes

The PR updates the ACPX runtime wrapper to append projected per-event NDJSON frames to the session record's advertised `eventLog.active_path` and adds tests for that stream-writing behavior.

## Best Possible Solution

Implement event-log persistence at the ACPX event source, or in one shared OpenClaw wrapper helper covering both `startTurn.events` and `runTurn`, and emit real `session/update` frames unless maintainers explicitly approve a documented projected-stream contract.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: current main advertises ACPX `eventLog.active_path` through `acpx@0.10.0`, while neither the current wrapper nor the latest release writes the stream. I did not run a live ACP container repro in this read-only review.

## Solution Assessment

No, this PR is not the best fix as written. The maintainable fix should cover the current `startTurn` path and preferably live in `openclaw/acpx` where the raw `session/update` notification exists, rather than writing projected events only from the wrapper's legacy `runTurn`.
