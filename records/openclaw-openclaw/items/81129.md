## Summary

Keep open: current main still has a source-level path to the reported pnpm 11 failure through the WhatsApp plugin's Baileys/libsignal dependency graph, and the fix requires a maintainer decision about pnpm supply-chain policy rather than conservative cleanup.

## What This Changes

The issue reports that adding a new workspace package causes `pnpm install` or a build-triggered install to fail with `ERR_PNPM_EXOTIC_SUBDEP` for Baileys' transitive `libsignal` dependency.

## Best Possible Solution

Keep pnpm 11 source checkouts working by removing the transitive exotic dependency blocker or by adding a narrowly justified pnpm workspace policy with regression coverage for the WhatsApp Baileys path.

## Reproduction Assessment

Yes, at source level: current main pins pnpm 11.1.0, includes `packages/*` plus `extensions/*` in the workspace, and still resolves Baileys' libsignal dependency through a GitHub tarball. I did not run the write-heavy install flow in this read-only review.

## Solution Assessment

Unclear: the reported behavior is valid, but maintainers should choose whether to remove or replace the exotic dependency path or explicitly relax pnpm policy for this known case.
