## Summary

Keep open: current main still has a source-level path to the reported pnpm 11 failure through the WhatsApp plugin's Baileys/libsignal dependency graph, and there is no current workspace policy that clearly permits that exotic transitive dependency during a re-resolution.

## What This Changes

Adding a new workspace package should not make pnpm install/build fail with ERR_PNPM_EXOTIC_SUBDEP from WhatsApp's Baileys/libsignal dependency graph.

## Best Possible Solution

Keep pnpm 11 source checkouts working by removing the transitive git/tarball dependency edge or adding a narrowly justified pnpm policy with regression coverage for the WhatsApp dependency path.

## Reproduction Assessment

Yes, at source level: current main pins pnpm 11.1.0, pnpm 11 defaults block-exotic-subdeps to true, and the WhatsApp plugin brings Baileys' libsignal git tarball as a transitive dependency. I did not run the write-heavy Windows add-a-package flow in this read-only review.

## Solution Assessment

Unclear: the report is valid, but maintainers should choose whether to remove or replace the exotic dependency, pin/patch it differently, or explicitly relax pnpm policy for this known case.
