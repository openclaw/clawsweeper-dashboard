## Summary

Keep open: the feature slice is useful, but the latest PR still puts Telegram-specific topic parsing and ACP binding lookup in shared status code, and it lacks the required changelog entry for user-visible `/status` output.

## What This Changes

The PR passes Telegram message context into `/status`, adds a `channelStatusLines` output slot, and renders Telegram topic configured/live ACP binding state with focused tests.

## Best Possible Solution

Keep the visibility feature, but expose a generic channel/plugin-owned status-line hook and implement Telegram configured/live/drift ACP topic rendering inside the Telegram plugin, with a single-line Unreleased changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main shows `/status` lacks message context/command target inputs and has no status-message slot for channel-owned topic routing lines; the PR adds unit coverage for the intended rendering paths, but I did not run a live Telegram bot.

## Solution Assessment

No. The visibility slice is narrow and useful, but the current diff hard-codes Telegram parsing and binding lookup in shared status code instead of routing it through a generic channel/plugin seam.
