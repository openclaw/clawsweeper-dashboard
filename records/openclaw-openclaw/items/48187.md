## Summary

Keep open: this is an active feature PR, but it currently routes Telegram-specific status logic through core status code and the supplied proof is a synthetic helper invocation rather than a Telegram `/status` run.

## What This Changes

The PR adds `/status` plumbing and tests to render Telegram topic id, delivery target, configured/live ACP binding, drift, and ACP runtime lines.

## Best Possible Solution

Expose this through a generic channel/plugin-owned status-line hook, with Telegram preparing the topic routing snapshot at its existing routing boundary and proving it with a live Telegram `/status` run.

## Reproduction Assessment

Not applicable: this is a feature PR rather than a bug report. Source inspection shows current main lacks a channel-owned `/status` line slot, while the PR proof only exercises a helper with constructed Telegram topic metadata.

## Solution Assessment

No. The user-visible output is useful, but hardcoding Telegram topic parsing and binding lookup in core status is not the narrowest maintainable solution; a plugin/channel status seam would preserve the owner boundary.
