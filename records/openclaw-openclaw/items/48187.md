## Summary

Keep open. The visibility feature is useful and the discussion identifies a real operator debugging gap, but the current patch still wires Telegram-specific topic parsing and binding lookup into shared status rendering and does not include live Telegram `/status` proof.

## What This Changes

The branch adds Telegram topic routing metadata to `/status` by passing message context into status rendering, adding channel status lines, and adding a core Telegram topic status helper with tests.

## Best Possible Solution

Expose a generic channel/plugin-owned status-line hook, have Telegram prepare the topic routing snapshot at its existing routing boundary, and prove the result with a live Telegram `/status` run.

## Reproduction Assessment

Not applicable: this is a feature PR rather than a bug report. Source inspection and the PR diff show current main lacks a channel-owned `/status` line slot, while the supplied proof is helper-level rendering rather than live Telegram behavior.

## Solution Assessment

No. The visible output is useful, but hardcoding Telegram topic parsing and binding lookup in shared core status rendering is not the narrowest maintainable solution; a generic channel/plugin status seam would preserve the owner boundary.
