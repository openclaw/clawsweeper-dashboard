## Summary

Keep open. The PR addresses a real Telegram topic visibility gap, including the configured/live/drift state called out in discussion, but the proposed implementation still puts Telegram-specific topic parsing and ACP binding lookup directly in shared status rendering instead of behind a channel/plugin-owned seam.

## What This Changes

The PR passes Telegram message context into `/status`, adds a channel status-line slot to status output, and renders Telegram topic configured/live ACP binding state with focused tests.

## Best Possible Solution

Keep the visibility feature, but have core accept and render generic channel-owned status lines while Telegram implements configured/live/drift ACP topic rendering through a typed plugin or SDK seam that reuses its existing conversation and binding helpers.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the gap: `/status` has no message context or command target context and no output slot for channel-owned topic routing state; the PR’s tests cover configured/live/drift rendering paths, but this review did not run a live Telegram bot.

## Solution Assessment

No, not as written. The product slice is narrow and useful, but the maintainable solution is a generic channel/plugin status-line seam with Telegram-specific parsing and binding lookup kept in the Telegram plugin.
