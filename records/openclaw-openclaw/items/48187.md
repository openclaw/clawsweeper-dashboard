## Summary

Keep this PR open: the visibility slice is useful and still not implemented on current main, but the latest patch remains blocked because shared status code computes Telegram-specific routing state directly and the user-facing `/status` change lacks a changelog entry.

## What This Changes

The branch threads message context into `/status`, adds a `channelStatusLines` output slot, and renders Telegram topic configured/live ACP binding details with focused tests.

## Best Possible Solution

Keep the visibility feature, but have core render generic channel-owned status lines while Telegram computes configured/live/drift ACP topic details through a typed plugin or SDK seam, then add the required changelog entry.

## Reproduction Assessment

Yes. Source inspection of current main shows `/status` has no message context, command target, or channel-owned output slot, and the PR diff shows the new visibility is currently computed by a core Telegram helper rather than a plugin seam.

## Solution Assessment

No. The current patch adds a useful render slot, but it is not the best maintainable solution because Telegram-specific target parsing and ACP binding lookup still live in shared status code.
