## Summary

Keep open. Current main still lacks the requested Telegram topic routing visibility in `/status`, and the PR head is a useful implementation slice with green public checks, but it needs maintainer follow-up before merge because the patch puts Telegram-specific rendering and target parsing directly in shared core status code.

## What This Changes

The PR passes Telegram message context into `/status`, adds channel status lines below the session line, and introduces Telegram topic status rendering/tests for configured ACP bindings, live bindings, drift warnings, and ACP runtime metadata.

## Best Possible Solution

Keep the feature focused, but route the Telegram-specific status rendering through a generic channel/plugin-owned status-line seam instead of hard-coding Telegram parsing in core. The landed version should preserve the message-context plumbing, show configured versus live binding drift, cover non-topic no-op behavior, and avoid unrelated infra changes.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the visibility gap: `/status` has no message context, no command target context, and no status-line slot for channel-owned topic routing state; the PR’s unit tests cover the intended configured/live/drift cases, though I did not run a live Telegram bot.

## Solution Assessment

No, not as written. The visibility slice is the right narrow product direction, but the current patch duplicates Telegram target grammar in shared core status code; the safer maintainable fix is a generic status-line extension point with Telegram-owned rendering behind the existing plugin/binding APIs.
