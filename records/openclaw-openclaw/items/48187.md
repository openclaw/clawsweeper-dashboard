## Summary

Keep open: current main still lacks the requested Telegram topic visibility in `/status`, but the PR should not merge as-is because it hard-codes Telegram routing and binding lookup in shared status code and still needs the required changelog entry.

## What This Changes

The PR passes message context into `/status`, adds a status-message slot for channel-owned lines, and renders Telegram topic configured/live ACP binding state with unit coverage.

## Best Possible Solution

Keep the visibility feature, but expose a generic channel/plugin-owned status-line hook and implement Telegram configured/live/drift ACP topic rendering inside the Telegram plugin, with a single-line Unreleased changelog entry.

## Reproduction Assessment

Yes for the feature gap by source inspection: current main's `/status` path lacks message context/command target inputs and `buildStatusMessage()` has no channel-owned status-line slot. I did not run a live Telegram topic repro in this read-only review.

## Solution Assessment

No. The display slot is useful, but the PR puts Telegram target parsing and ACP binding lookup in shared `src/status`; the safer direction is a generic channel/plugin status-line seam that reuses Telegram-owned routing helpers.
