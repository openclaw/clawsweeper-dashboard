## Summary

Keep open: the feature gap is real and the PR is useful, but the current patch still puts Telegram-specific topic parsing and ACP binding display in shared status code instead of a channel/plugin-owned status seam; it also lacks the required user-facing changelog entry.

## What This Changes

The PR passes Telegram command context into `/status`, adds channel-owned status-line plumbing to the status message renderer, and renders Telegram forum-topic configured/live ACP binding state with tests.

## Best Possible Solution

Keep the visibility feature, but route topic/binding status through an additive channel/plugin-owned `/status` line hook implemented by the Telegram plugin, with a single-line Unreleased changelog entry.

## Reproduction Assessment

Yes for the feature gap: current main lacks message context/command target inputs and has no channel-owned status-line slot, so Telegram topic routing state cannot be rendered by `/status` today. This was verified by source inspection rather than a live Telegram run.

## Solution Assessment

No. The display slot is a useful direction, but the current implementation hard-codes Telegram topic grammar and binding lookup in shared status code instead of exposing a generic channel/plugin seam and letting Telegram render its own lines.
