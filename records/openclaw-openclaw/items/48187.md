## Summary

Keep open. The visibility goal is useful, but the latest head still has blocking issues: native slash `/status` can miss the new topic context, Telegram-specific topic parsing remains in shared status code, and the supplied proof is helper output rather than live Telegram topic proof.

## What This Changes

The PR adds Telegram forum-topic routing, binding, drift, and ACP metadata lines to `/status`, threads message context through some status call sites, adds status tests, and includes unrelated test type/lint cleanups.

## Best Possible Solution

Add a plugin-owned `/status` line hook that receives command/message context, implement the Telegram configured/live/drift topic lines in the Telegram plugin using existing binding helpers, pass context through every `/status` path including native slash, and verify with redacted live Telegram topic output.

## Reproduction Assessment

Yes for source-level review: current main lacks the exact topic status-line implementation, and the PR path shows native slash `/status` can still omit `ctx`. I did not run a live Telegram forum-topic bot in this read-only review.

## Solution Assessment

No. The visibility direction is useful, but the current patch is not the best maintainable solution because it misses the native slash context path and keeps Telegram topic grammar in shared core status code instead of a channel/plugin-owned hook.
