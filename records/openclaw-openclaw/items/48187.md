## Summary

Keep open. Current main still lacks the requested Telegram topic-routing visibility in `/status`, and this PR adds useful operator diagnostics, but the current patch hard-codes Telegram-specific routing and ACP binding parsing into shared status code instead of routing it through a channel/plugin-owned seam.

## What This Changes

The PR passes Telegram message context into `/status`, adds a channel status-line slot to status output, and renders Telegram topic configured/live ACP binding state with focused tests.

## Best Possible Solution

Keep the visibility feature, but make core accept and render generic channel-owned status lines while Telegram implements the configured/live/drift ACP topic rendering through a typed plugin or SDK seam using its existing binding helpers.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the gap: `/status` has no message context or command target context and no output slot for channel-owned topic routing state; the PR’s tests cover the intended configured/live/drift rendering paths, but this read-only review did not run a live Telegram bot.

## Solution Assessment

No, not as written. The product slice is narrow and useful, but the maintainable solution is a generic channel/plugin status-line seam with Telegram-specific rendering kept in the Telegram plugin rather than shared core status code.
