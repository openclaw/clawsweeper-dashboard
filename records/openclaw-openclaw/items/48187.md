## Summary

Keep open. Current main still lacks this Telegram topic `/status` visibility feature, but this PR is not merge-ready because it misses the native slash fast path, puts Telegram routing grammar in shared status code, and still lacks real Telegram proof.

## What This Changes

The PR threads message context into several `/status` reply paths and adds status rendering/tests for Telegram topic routing, configured/live ACP bindings, drift warnings, and ACP runtime metadata.

## Best Possible Solution

Add a generic channel/plugin-owned `/status` line hook, implement Telegram configured/live/drift topic state inside the Telegram plugin, thread topic context through every `/status` path including native slash, and verify with redacted live Telegram topic proof.

## Reproduction Assessment

Yes by source inspection for the feature gap: current main has no message-context or channel-status-line seam in shared `/status`, and native fast `/status` still calls the status reply builder without context. I did not run a live Telegram forum-topic reproduction in this read-only review.

## Solution Assessment

No. The visibility goal is useful, but this patch misses native slash `/status` context and should move Telegram-specific routing grammar behind a channel/plugin-owned hook instead of shared status code.
