## Summary

Keep open: current main still has the downstream failure mode described here. Restart recovery marks a non-resumable interrupted main session failed and logs it, but there is no visible channel recovery notice or clean next-turn reset for the affected Telegram chat.

## What This Changes

The issue requests bounded Telegram-visible recovery or clean session reset when a Codex app-server startup failure leaves a main-session transcript unrecoverable after gateway restart.

## Best Possible Solution

Keep the existing resumable restart recovery, but add a channel-agnostic abandoned-session path that safely notifies the stored chat route or makes the next user turn start from a coherent fresh state when the transcript tail cannot be resumed.

## Reproduction Assessment

Yes. Source inspection plus the existing unit test provide a high-confidence downstream reproduction path: an aborted running main session with a non-resumable assistant tail is marked failed with no gateway resume or visible delivery; a full live repro would additionally force the Codex startup failure and restart sequence.

## Solution Assessment

Yes. The best fix is not to fold this into the Codex startup retry issue alone, but to add bounded recovery behavior in main-session restart recovery while keeping Telegram-specific transport code out of core.
