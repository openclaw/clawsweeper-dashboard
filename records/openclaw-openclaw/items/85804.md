## Label Justifications

- `P1`: A current-release Telegram main-session workflow can stall or lose coherent replies after a Codex startup failure and gateway restart.
- `impact:message-loss`: The user-visible failure is a missing or out-of-context Telegram reply with no bounded recovery notice.
- `impact:session-state`: The failure depends on interrupted main-session transcript state, restart recovery, and whether the active session can be safely resumed or reset.

## Summary

Keep open: current main and the latest release still have a source-proven path where an interrupted main session with a non-resumable transcript tail is marked failed with only logs/session-state changes, while the affected Telegram chat gets no bounded recovery notice or clean reset guarantee.

## What This Changes

Add recovery behavior for Telegram-backed main sessions that become unrecoverable after Codex app-server startup failure and gateway restart.

## Best Possible Solution

Define one channel-agnostic restart-recovery policy for externally routable main sessions: when resume is unsafe, clear or rotate the binding safely and send an idempotent sanitized notice to the persisted route, while keeping the Codex startup retry fix separate.

## Reproduction Assessment

Yes, at source level: a running main session with `abortedLastRun: true` and an assistant-tail transcript follows the existing test path to `recovered=0 failed=1`, no Gateway resume call, and a failed session row. I did not run a live Telegram/Codex restart replay.

## Solution Assessment

Unclear until maintainers choose the external-channel recovery policy. The narrowest maintainable direction is a shared route-aware recovery notice or clean session reset, coordinated with the open broader channel recovery work rather than a Telegram-only branch.
