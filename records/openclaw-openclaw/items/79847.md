## Summary

Keep this open: current main still passes qmd-scoped XDG variables into mcporter, and mcporter 0.10.x uses those variables for config discovery when no explicit config is supplied. The active fix path is https://github.com/openclaw/openclaw/pull/79745, which is still open and needs normal maintainer handling before this issue can close.

## What This Changes

The issue requests isolating qmd-manager subprocess environments so qmd keeps per-agent XDG state while mcporter can still find its configured server state.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/79745 or an equivalent fix that gives mcporter a clean process env and explicit per-agent config while preserving configured qmd server definitions.

## Reproduction Assessment

Yes. Current main source shows qmd-scoped XDG variables flowing into mcporter child spawns, and mcporter v0.10.2 source shows `XDG_CONFIG_HOME` controls config discovery when no explicit config path is supplied.

## Solution Assessment

Yes. The narrow maintainable fix is to separate qmd-only environment from mcporter process state, either by the reported env split or the active PR's explicit per-agent mcporter config plus clean mcporter env approach.
