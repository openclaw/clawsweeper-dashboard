## Summary

Keep open: current main and the latest release still pass QMD-scoped XDG variables into mcporter, while mcporter 0.10.2 uses `XDG_CONFIG_HOME` for config discovery. The adjacent fix PR was closed unmerged, so there is no safe implemented or duplicate close path.

## What This Changes

The issue reports qmd-manager leaking QMD-scoped `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` into mcporter child processes, breaking mcporter 0.10+ QMD memory searches.

## Best Possible Solution

Land a focused memory-core fix that keeps qmd CLI spawns on the per-agent QMD XDG env while preventing mcporter config discovery from being redirected into qmd state, with regression tests for both spawn paths.

## Reproduction Assessment

Yes, from source inspection: current main passes QMD-scoped XDG env into mcporter, and mcporter 0.10.2 consults `XDG_CONFIG_HOME` for config discovery. I did not execute the live daemon scenario in this read-only review.

## Solution Assessment

Yes, a per-target environment split or equivalent explicit mcporter config path is the narrow maintainable direction. The earlier generated-config PR was plausible, but it is closed unmerged and no current-main implementation replaces it.
