## Summary

This should stay open because current main still passes qmd-scoped XDG variables into mcporter, and an open closing PR is already targeting this exact bug. The issue is source-reproducible against the checked-out code and mcporter 0.10.2 path-discovery source, so the right cleanup is to review and land a focused fix rather than close the issue early.

## What This Changes

Stop qmd-manager from leaking qmd-scoped XDG config/cache environment variables into mcporter child processes so mcporter 0.10+ can resolve its own config.

## Best Possible Solution

Keep this issue open until https://github.com/openclaw/openclaw/pull/79981 or a reconciled qmd-manager fix lands and proves mcporter 0.10+ no longer resolves config through qmd-only XDG state.

## Reproduction Assessment

Yes. Current main constructs qmd-scoped XDG variables and passes them to `runMcporter`, while mcporter 0.10.2 uses `XDG_CONFIG_HOME` for config discovery; I did not execute the live daemon flow.

## Solution Assessment

Yes, the per-target environment split in the open closing PR is the narrowest direct fix for this report. The overlapping per-agent `--config` approach may also solve the immediate config lookup, but maintainers should reconcile the open PRs before landing one path.
