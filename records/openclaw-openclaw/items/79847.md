## Summary

Keep this open: current main still passes the qmd-scoped XDG environment to mcporter, and the only directly related OpenClaw fix is still an open PR that needs review and real behavior proof.

## What This Changes

The issue asks OpenClaw to stop leaking qmd-only `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` into mcporter child processes so mcporter 0.10.x can find its normal config.

## Best Possible Solution

Land a reviewed fix that gives mcporter its own config-discovery context, preferably with real QMD/mcporter 0.10.x proof and a regression test that prevents qmd-only XDG state from breaking mcporter config lookup again.

## Reproduction Assessment

Yes. Current main passes qmd-scoped XDG variables into `runMcporter`, and mcporter v0.10.2 uses `XDG_CONFIG_HOME` for config lookup when no explicit config path is provided; I did not execute the live daemon scenario in this read-only pass.

## Solution Assessment

Yes, the reported direction is narrow: split the per-target environment or pass an explicit agent-local mcporter config so mcporter no longer depends on leaked qmd XDG state for config discovery. The linked PR appears to target the same failure but should remain under review until proven and merged.
