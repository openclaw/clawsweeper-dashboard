## Summary

This should stay open: current main still passes qmd-scoped XDG variables into mcporter, and mcporter 0.10.x now uses those variables for its own config lookup. There is already an open adjacent fix PR, but it still needs review/proof and may need to preserve custom mcporter setups from this report.

## What This Changes

The issue asks OpenClaw to stop leaking qmd-specific `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` values into mcporter child processes while preserving qmd's per-agent state isolation.

## Best Possible Solution

Keep this issue open until the qmd-manager mcporter path either splits per-target environments or lands an equivalent reviewed fix that proves mcporter 0.10.x can still find the intended qmd server config without leaking qmd-only XDG state.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: current main passes qmd-scoped XDG variables into `runMcporter`, and mcporter 0.10.x uses `XDG_CONFIG_HOME` for config discovery; I did not execute the live daemon scenario in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? The requested per-target environment split is a narrow maintainable fix, while the active PR's generated-config approach needs maintainer review to ensure custom mcporter server setups keep working.
