## Summary

This should stay open: current main still passes qmd-scoped XDG variables into mcporter, and mcporter 0.10.2 honors `XDG_CONFIG_HOME` for config discovery. The related open PR may solve the observed failure through explicit per-agent mcporter config, but it is not merged and still needs maintainer review/proof.

## What This Changes

The issue reports that qmd-manager reuses qmd-specific `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` for mcporter child processes, causing mcporter 0.10.x to miss the user's `~/.mcporter` config.

## Best Possible Solution

Keep this issue open until qmd-manager either splits qmd and mcporter environments or lands an equivalent reviewed fix that proves mcporter 0.10.x can find the intended qmd server config without inheriting qmd-only XDG state.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection and the mcporter 0.10.2 contract: current main passes qmd-scoped `XDG_CONFIG_HOME` into `runMcporter`, and mcporter uses that env var for config discovery when no explicit config is supplied.

## Solution Assessment

Is this the best way to solve the issue? The requested per-target env split is a narrow maintainable fix; the open per-agent mcporter config PR may also solve the reported failure, but it still needs review and real behavior proof before this issue can close.
