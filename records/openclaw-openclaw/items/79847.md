## Summary

Keep open: current main and the latest release still pass qmd-scoped XDG variables into mcporter, and mcporter 0.10.x uses those variables for config discovery. The active fix path is https://github.com/openclaw/openclaw/pull/79745, which is open and now includes real behavior proof, so this issue should close only after that or an equivalent fix lands.

## What This Changes

The issue reports that qmd-manager reuses its qmd-scoped environment for mcporter child processes, causing mcporter 0.10+ to miss the user's mcporter config under memory.backend=qmd.

## Best Possible Solution

Land or supersede https://github.com/openclaw/openclaw/pull/79745 so mcporter gets an explicit agent config and a clean process env, then close this issue after the fix is on main and shipped.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: current main passes qmd-scoped XDG variables into mcporter, and mcporter v0.10.2 uses XDG_CONFIG_HOME for home config discovery when no explicit config is supplied.

## Solution Assessment

Is this the best way to solve the issue? Yes: the open per-agent --config plus clean mcporter process-env approach is the narrowest durable fix, while the simple env split from the issue would also address the immediate XDG leak.
