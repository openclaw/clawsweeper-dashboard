## Summary

Keep open: current main still passes qmd-scoped XDG variables into mcporter, and upstream mcporter 0.10.x uses those variables for config discovery. The active fix path is https://github.com/openclaw/openclaw/pull/79745, but it is still open and unmerged.

## What This Changes

The issue asks OpenClaw to stop leaking qmd-scoped XDG_CONFIG_HOME and XDG_CACHE_HOME into mcporter child processes so mcporter 0.10.x can find the configured qmd server.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79745 or an equivalent qmd-manager fix that keeps qmd-only XDG state out of mcporter while preserving custom mcporter server setups.

## Reproduction Assessment

Yes, from source inspection and dependency source: current main passes qmd-scoped XDG variables to mcporter, and mcporter 0.10.2 uses XDG_CONFIG_HOME for config discovery when no explicit config path is supplied. I did not execute the live daemon scenario.

## Solution Assessment

Yes, the narrow maintainable solution is to isolate mcporter's process environment and/or pass an explicit agent-scoped mcporter config while leaving qmd's own XDG state scoped per agent. The open isolation PR appears to target that end state but still needs normal maintainer review and landing.
