## Summary

Keep open: current main still routes qmd-scoped XDG variables into mcporter, matching the reported mcporter 0.10+ failure path, and the active overlapping PR is unmerged and still needs maintainer review/proof.

## What This Changes

The issue asks qmd-manager to stop passing qmd-scoped XDG_CONFIG_HOME and XDG_CACHE_HOME to mcporter child processes because mcporter 0.10+ uses them for its own config discovery.

## Best Possible Solution

Land a focused memory-core fix that keeps qmd-only XDG scoping out of mcporter spawns, or an equivalent explicit-config approach that preserves configured mcporter server definitions, with regression coverage and redacted live mcporter 0.10.x proof.

## Reproduction Assessment

Yes, from source inspection and dependency contract: current main passes qmd-scoped XDG variables into runMcporter, and mcporter v0.10.2 uses XDG_CONFIG_HOME for config discovery. I did not execute the full live gateway plus qmd server scenario in this read-only review.

## Solution Assessment

Yes, the issue's per-target environment split is the narrowest maintainable fix for the reported leak. The active generated-config PR may also address the immediate failure, but it should preserve existing custom mcporter server definitions before landing.
