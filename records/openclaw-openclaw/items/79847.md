## Summary

Keep open: current main and the latest release still pass qmd-scoped XDG environment variables into mcporter, and the concrete fix is only present in an open unmerged PR. The closed env-split PR did not merge, while https://github.com/openclaw/openclaw/pull/79745 is the active fix path with real behavior proof but still needs maintainer review and merge handling.

## What This Changes

The issue reports that QMD memory search spawns mcporter with qmd-scoped XDG environment variables, causing mcporter 0.10.x to resolve its config from the wrong directory.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79745 or an equivalent focused fix that keeps qmd-only XDG state out of mcporter spawns while preserving configured mcporter server definitions.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: current main passes qmd-scoped XDG variables into `runMcporter`, and mcporter v0.10.2 uses `XDG_CONFIG_HOME` for config discovery when no explicit config is supplied; I did not run the live daemon scenario.

## Solution Assessment

Is this the best way to solve the issue? Yes, the issue's per-target environment split is a narrow fix, and the active PR's explicit per-agent mcporter config plus clean process env is also a maintainable solution if it passes normal review and CI.
