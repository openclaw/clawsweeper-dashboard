## Label Justifications

- `P1`: QMD memory search through mcporter is a broken agent workflow for users on mcporter 0.10 or newer.
- `impact:session-state`: The failure affects memory backend state/config routing and prevents QMD-backed memory search from working through mcporter.

## Summary

Keep open: current main and the latest release still pass QMD-scoped XDG environment variables into mcporter, while the active replacement fix is still open and unmerged.

## What This Changes

Prevent qmd-manager's QMD-specific XDG_CONFIG_HOME/XDG_CACHE_HOME environment from leaking into mcporter child spawns so mcporter 0.10+ can find its configured qmd server.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92499, or an equivalent qmd-manager fix that keeps QMD-scoped state on qmd while preserving user and workspace mcporter config discovery.

## Reproduction Assessment

Yes, from source inspection: current main passes QMD-scoped XDG variables into runMcporter, and mcporter v0.10.2 uses XDG_CONFIG_HOME for config discovery when no explicit config is supplied. I did not execute the live daemon path in this read-only review.

## Solution Assessment

Yes on direction: the maintainable fix is to isolate mcporter config/env discovery from QMD's per-agent XDG state while keeping QMD's own scoped env. The active replacement PR is the path to review before this issue can close.
