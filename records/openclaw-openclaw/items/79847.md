## Summary

Keep this open: current main still passes qmd-scoped XDG variables into mcporter, and the linked fix PR is still open rather than merged.

## What This Changes

The issue reports that qmd-manager passes qmd-scoped `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` into mcporter, causing mcporter 0.10.x to miss the user's configured qmd server.

## Best Possible Solution

Land a reviewed qmd-manager fix that gives mcporter an explicit config or a clean per-target environment, with regression coverage and redacted real mcporter/QMD behavior proof before closing this issue.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: current main passes qmd-scoped XDG variables to mcporter, and mcporter 0.10.2 uses `XDG_CONFIG_HOME` for config discovery; I did not run the live daemon scenario.

## Solution Assessment

Is this the best way to solve the issue? The issue's per-target env split is a narrow maintainable fix, and the open PR's explicit `--config` approach is also plausible, but current main is not fixed until one reviewed path lands with proof.
