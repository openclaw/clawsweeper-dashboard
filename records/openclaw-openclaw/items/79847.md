## Summary

Keep open: current main and the latest shipped release still pass qmd-scoped XDG variables into mcporter, and the active fix path is an open PR rather than landed code.

## What This Changes

The issue asks OpenClaw to stop leaking qmd-scoped XDG_CONFIG_HOME and XDG_CACHE_HOME into mcporter child processes so mcporter 0.10+ can resolve its configured qmd server.

## Best Possible Solution

Land or replace the active per-agent mcporter config plus clean-env fix, with regression coverage that preserves custom and remote qmd server definitions.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection and the reporter's concrete steps: current main sends qmd-scoped XDG variables into mcporter, and mcporter 0.10.2 uses XDG_CONFIG_HOME for config discovery when no explicit config is supplied.

## Solution Assessment

Is this the best way to solve the issue? Yes: keep qmd-scoped XDG only on qmd-owned execution and give mcporter either a clean process env plus explicit per-agent config, as the active PR proposes.
