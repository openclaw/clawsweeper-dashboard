## Summary

This should stay open: current main still passes qmd-scoped XDG variables into mcporter spawns, and mcporter 0.10.x uses those variables for config discovery when no explicit config path is supplied. The open overlapping fix PR may solve the concrete failure, but it is not merged and still needs review/proof rather than closing this report now.

## What This Changes

The issue reports that qmd-manager leaks qmd-scoped XDG environment variables into mcporter child processes, breaking mcporter 0.10.x config discovery for QMD memory search.

## Best Possible Solution

Land a reviewed qmd-manager fix that gives mcporter an explicit agent-scoped config and/or strips qmd-only XDG variables from mcporter spawns, with redacted live proof against mcporter 0.10.x and QMD memory search.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: current main passes qmd-scoped XDG variables into runMcporter, and mcporter v0.10.2 uses XDG_CONFIG_HOME for home config candidates when no explicit config path is supplied; I did not execute the live daemon scenario.

## Solution Assessment

Is this the best way to solve the issue? Yes, the requested per-target environment split is a narrow maintainable fix; the open PR's explicit config-path isolation also appears viable for the reported failure, but should be reviewed for defense-in-depth env cleanup and real behavior proof.
