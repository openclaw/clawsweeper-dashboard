## Summary

Keep open: current main still passes qmd-scoped XDG variables into mcporter, and mcporter 0.10.x source confirms those variables affect config discovery unless an explicit config path is supplied. The adjacent fix PR is open and appears directionally relevant, but it is not merged and still lacks real behavior proof.

## What This Changes

The issue reports that qmd-manager reuses qmd-scoped XDG env for mcporter child processes, causing mcporter 0.10.x to search the wrong config directory and fail QMD memory searches.

## Best Possible Solution

Land a reviewed qmd-manager fix that gives mcporter an explicit per-agent config or a mcporter-specific environment while preserving qmd's per-agent state, then prove the gateway path against mcporter 0.10.x.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source and dependency inspection: current main sends qmd-scoped XDG variables into mcporter, and mcporter 0.10.x uses XDG_CONFIG_HOME for config discovery when no explicit config is supplied.

## Solution Assessment

Is this the best way to solve the issue? The requested per-target env split is a narrow maintainable fix; the open PR's explicit --config approach also addresses the concrete mcporter discovery failure but still needs real behavior proof and possibly an XDG-strip follow-up.
