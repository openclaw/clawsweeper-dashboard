## Summary

This should stay open: current main still sends qmd-scoped XDG variables into mcporter, mcporter 0.10.x intentionally uses XDG_CONFIG_HOME for config discovery, and https://github.com/openclaw/openclaw/pull/79745 is related but still open and unproven.

## What This Changes

The issue asks OpenClaw to stop leaking qmd-only XDG_CONFIG_HOME and XDG_CACHE_HOME values into mcporter child processes so mcporter 0.10.x can find the intended QMD server config.

## Best Possible Solution

Land a reviewed fix that gives mcporter an explicit per-agent config or a clean mcporter-specific environment, prove it against mcporter 0.10.x, then close this issue after merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: current main passes qmd-scoped XDG_CONFIG_HOME and XDG_CACHE_HOME into runMcporter, and mcporter 0.10.2 uses XDG_CONFIG_HOME for config discovery unless --config or MCPORTER_CONFIG is explicit.

## Solution Assessment

Is this the best way to solve the issue? Yes, per-target mcporter config/env isolation is the narrow maintainable fix; the open related PR is a plausible implementation but should be reviewed, proven, and merged before this issue closes.
