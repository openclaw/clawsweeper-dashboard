## Summary

Current main still has the reported env leak and the focused fix path is an open PR, so this issue should stay open until that PR lands or is replaced. Closing now would lose the canonical bug thread for the mcporter 0.10.x breakage.

## What This Changes

The issue reports that QMD manager leaks qmd-scoped XDG env vars into mcporter spawns, causing mcporter 0.10+ to miss the user's configured qmd server.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79745 or an equivalent focused fix that keeps qmd-only XDG state out of mcporter and proves mcporter 0.10.x memory search works.

## Reproduction Assessment

Yes, from source inspection and the mcporter 0.10.2 contract: current main passes qmd-scoped XDG variables into mcporter, and mcporter uses `XDG_CONFIG_HOME` for home config discovery when no explicit config is provided. I did not execute the live gateway scenario.

## Solution Assessment

Yes, the active PR's per-agent mcporter config plus clean mcporter process env is the best current route because it fixes the config-discovery failure and adds defense in depth without a new user config knob. Closing before that lands would be premature.
