## Summary

Keep this open: current main and the latest release still pass qmd-scoped XDG variables into mcporter, so the reported mcporter 0.10.x failure is still source-reproducible. https://github.com/openclaw/openclaw/pull/79745 is the active canonical fix path and now has sufficient real-behavior proof, but it has not merged yet.

## What This Changes

The issue reports that qmd-manager leaks qmd-scoped XDG_CONFIG_HOME and XDG_CACHE_HOME into mcporter subprocesses, causing mcporter 0.10.x to miss the configured qmd server.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/79745 or an equivalent focused change that gives mcporter an explicit per-agent config and strips qmd-only XDG state from mcporter spawns, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes. Current main and v2026.5.12 build one qmd-scoped env and pass it to mcporter, while mcporter v0.10.2 honors XDG_CONFIG_HOME for config discovery; I did not run the live daemon scenario.

## Solution Assessment

Yes, the active PR direction is the right maintainable fix: explicit per-agent mcporter config plus a clean mcporter process env solves the regression without adding a new user config knob. The issue should stay open until that fix is merged.
