## Summary

Keep this open: current main still passes qmd-scoped XDG variables into mcporter, and mcporter 0.10.2 uses `XDG_CONFIG_HOME` for config lookup. The right fix path is active in https://github.com/openclaw/openclaw/pull/79745, but that PR is still open and should land or be closed before this issue is cleaned up.

## What This Changes

The issue reports that QMD memory mcporter spawns inherit qmd-scoped XDG config/cache environment variables, causing mcporter 0.10.x to miss the user's mcporter config.

## Best Possible Solution

Land a reviewed fix that stops qmd-only XDG state from influencing mcporter config discovery while preserving configured qmd keep-alive servers.

## Reproduction Assessment

Yes. Current main creates a qmd-scoped shared env and passes it to `runMcporter`, while mcporter v0.10.2 uses `XDG_CONFIG_HOME` to build config candidates; I did not run the live daemon flow.

## Solution Assessment

Yes, the requested per-target env split or the active per-agent `--config` approach with a cleaned mcporter env is the narrow maintainable fix. The open PR should address its remaining lifecycle-preservation finding before this issue closes.
