## Summary

Keep open: current main still passes the qmd-scoped XDG environment into mcporter, and mcporter 0.10.x uses XDG_CONFIG_HOME for config discovery. The adjacent fix PR is open but not merged or proven in a real setup yet.

## What This Changes

The issue requests stopping QMD manager's qmd-scoped XDG_CONFIG_HOME and XDG_CACHE_HOME from leaking into mcporter child processes so mcporter 0.10.x can find its configured qmd server.

## Best Possible Solution

Land a reviewed QMD manager fix that scopes mcporter separately from qmd, either by splitting child environments or by passing an explicit per-agent mcporter config with real mcporter/QMD proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: current main passes qmd-scoped XDG variables to runMcporter, and mcporter 0.10.2 uses XDG_CONFIG_HOME for config lookup. I did not execute the live daemon scenario in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Yes: scoping the environment per child process is the narrowest direct fix for the leak. The open explicit-config PR may solve the immediate failure too, but it should land only after real behavior proof and review of custom mcporter setup compatibility.
