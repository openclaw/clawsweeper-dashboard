## Summary

Keep this open: current main still sends qmd-scoped XDG variables to mcporter, mcporter 0.10.x uses those variables for config discovery, and the overlapping fix PR is still open and blocked on real behavior proof.

## What This Changes

The issue requests that qmd-manager stop leaking qmd-only XDG_CONFIG_HOME and XDG_CACHE_HOME values into mcporter subprocesses so mcporter 0.10.x can find the intended user config.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/79745 or an equivalent qmd-manager change that gives mcporter an explicit agent-scoped config or a non-qmd XDG environment, with real mcporter 0.10.x behavior proof before closing this issue.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main passes qmd-scoped XDG_CONFIG_HOME and XDG_CACHE_HOME into runMcporter, and mcporter v0.10.2 uses XDG_CONFIG_HOME when no explicit config path is provided.

## Solution Assessment

Yes, the narrow maintainable fix is to separate mcporter config/env handling from qmd's XDG-scoped process state. The open PR's --config approach likely fixes the reported failure, while stripping qmd-only XDG vars would add defense in depth.
