## Summary

Keep open: current main still passes qmd-scoped XDG variables into mcporter, and mcporter 0.10.2 uses XDG_CONFIG_HOME for its own config discovery. The related fix path is still open at https://github.com/openclaw/openclaw/pull/79745, so this issue should wait for a reviewed/proven QMD mcporter fix rather than be closed now.

## What This Changes

The issue asks OpenClaw to stop leaking qmd-scoped XDG_CONFIG_HOME and XDG_CACHE_HOME into mcporter child processes so mcporter 0.10+ can find the intended configured server.

## Best Possible Solution

Keep this issue open until qmd-manager either splits qmd and mcporter spawn environments or lands an equivalent reviewed fix that preserves the intended user-configured mcporter server path.

## Reproduction Assessment

Yes, source-reproducible: current main passes qmd-scoped XDG variables into runMcporter, and mcporter 0.10.2 uses XDG_CONFIG_HOME to discover its own config. I did not execute the full live gateway daemon scenario in this read-only pass.

## Solution Assessment

Yes, the requested per-target env split is the narrowest maintainable fix for the reported leak. An explicit mcporter config approach can also work, but it must preserve or intentionally replace existing custom mcporter server definitions and needs real behavior proof.
