## Summary

Keep open: current main and the latest v2026.6.6 release still pass QMD-scoped XDG variables into mcporter, while mcporter v0.10.2 uses XDG_CONFIG_HOME for config discovery. The active replacement PR is the right ownership path but is not merged yet, so the issue remains necessary.

## What This Changes

Stop QmdMemoryManager from passing QMD-scoped XDG_CONFIG_HOME and XDG_CACHE_HOME to mcporter so mcporter 0.10+ can find the configured qmd server.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/92499, or an equivalent qmd-manager fix that isolates QMD env from mcporter while preserving user and workspace mcporter configs.

## Reproduction Assessment

Yes, from source inspection: current main passes QMD-scoped XDG variables into runMcporter, and mcporter v0.10.2 uses XDG_CONFIG_HOME for config discovery. I did not execute the live daemon scenario in this read-only review.

## Solution Assessment

Yes on direction: the fix belongs in qmd-manager's mcporter spawn/config routing, not in mcporter. The open replacement PR is the current best landing path if review confirms upgrade compatibility for existing mcporter configs.
