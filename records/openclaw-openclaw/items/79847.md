## Summary

Keep this issue open: current main still passes qmd-scoped XDG variables into mcporter, and the directly linked fixing PR is open, not merged. The issue should close only after https://github.com/openclaw/openclaw/pull/79981 lands, or after maintainers choose and land the overlapping https://github.com/openclaw/openclaw/pull/79745 path.

## What This Changes

The issue asks for qmd-manager to stop leaking qmd-scoped XDG_CONFIG_HOME and XDG_CACHE_HOME into mcporter subprocesses so mcporter 0.10.x can find its normal config.

## Best Possible Solution

Land one reviewed qmd-manager fix that preserves qmd's agent-scoped env for qmd spawns while preventing mcporter config discovery from seeing those qmd-only XDG variables, then close this issue from the merged PR.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source and dependency inspection: current main passes qmd-scoped XDG variables to runMcporter, and mcporter v0.10.2 uses XDG_CONFIG_HOME for config discovery when no explicit config path is supplied.

## Solution Assessment

Is this the best way to solve the issue? Yes, the per-target env split is the narrowest maintainable fix for the reported leak; the open explicit-config PR may also solve current config discovery, but it leaves a defense-in-depth XDG leak unless combined with env isolation.
