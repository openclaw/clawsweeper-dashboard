## Label Justifications

- `P1`: The issue describes a deterministic QMD memory search failure for users on `memory.backend: qmd` with mcporter 0.10+.
- `impact:session-state`: The affected path is memory retrieval through the QMD memory backend, which feeds agent context and session recall.

## Summary

Keep open: current main and v2026.6.6 still pass QMD-scoped XDG variables into mcporter, and mcporter 0.10.2 uses `XDG_CONFIG_HOME` for config discovery. A replacement fix PR is open but unmerged, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue reports that QMD memory uses one per-agent spawn environment for both qmd and mcporter, causing mcporter 0.10+ to miss the user's qmd server config.

## Best Possible Solution

Review and land `https://github.com/openclaw/openclaw/pull/92499` or a narrower equivalent that keeps QMD env scoped to qmd/generated config while preserving external mcporter config sources, then close this issue after merge verification.

## Reproduction Assessment

Yes. Source inspection shows current main passes QMD-scoped XDG env into mcporter, and mcporter v0.10.2 uses `XDG_CONFIG_HOME` for config discovery; I did not execute the live daemon scenario in this read-only review.
