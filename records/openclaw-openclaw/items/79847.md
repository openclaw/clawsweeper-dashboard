## Summary

Keep open: current main still passes qmd-scoped XDG variables into mcporter, and mcporter 0.10.x uses those variables for config discovery when no explicit config path is supplied. The active overlapping PR may fix the concrete failure, but it is still open and needs real behavior proof plus review of custom mcporter setups.

## What This Changes

The issue asks OpenClaw to stop leaking qmd-scoped `XDG_CONFIG_HOME` and `XDG_CACHE_HOME` into mcporter child processes so mcporter 0.10+ can find the intended qmd server config.

## Best Possible Solution

Land a reviewed qmd-manager fix that isolates mcporter from qmd-only XDG state, either by using a dedicated mcporter environment or by passing an explicit per-agent config path while preserving supported custom setups, with regression coverage and real mcporter/QMD proof.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes from source inspection: current main passes qmd-scoped XDG variables into `runMcporter`, and mcporter v0.10.2 consults `XDG_CONFIG_HOME` for config discovery when no explicit config is supplied; I did not run the live daemon scenario.

## Solution Assessment

Is this the best way to solve the issue? The requested per-target environment split is a narrow maintainable fix; the active PR's explicit `--config` approach also addresses the reported config-discovery failure, but it still needs proof and custom-setup review before it can close this issue.
