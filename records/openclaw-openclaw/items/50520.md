## Summary

Keep open: the PR targets the right outbound boundary and current main still has the reported leak, but the latest PR head is not merge-ready because it strips only fixed sentinel labels, misses generated structured metadata labels, and is stale against current main’s reply-target label.

## What This Changes

The PR adds a leading inbound-metadata sanitizer call to outbound payload planning, adds Location to the sanitizer sentinel list, adds outbound regression tests, and adds a changelog entry for #39847.

## Best Possible Solution

Refresh the PR onto current main and make the leading outbound sanitizer match every metadata block shape generated by `buildInboundUserContextPrefix`, while preserving timestamp-only replies and mid-message examples.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: on current main, a payload whose text starts with `Conversation info` metadata followed by visible reply text is parsed and delivered without metadata stripping; on the PR head, a leading `Conversation info` block followed by `context 1 (untrusted metadata):` would leave the generated structured block in the returned text.

## Solution Assessment

No, not yet. Sanitizing at the outbound plan boundary is the narrow maintainable fix, but the helper must cover generated structured labels and the branch must preserve current main’s reply-target sentinel before this is complete.
