## Summary

Keep open: the PR uses the right shared outbound delivery boundary and adds location/changelog coverage, but it still misses generated structured untrusted-context blocks that current channel context plumbing can emit and users can see if echoed.

## What This Changes

The PR calls `stripLeadingInboundMetadata()` before outbound reply-directive parsing, adds Location metadata sentinel coverage and regression tests, and adds a changelog entry for the outbound metadata leak.

## Best Possible Solution

Extend the leading outbound sanitizer to cover every metadata block shape generated by `buildInboundUserContextPrefix()`, including dynamic structured labels, while preserving timestamp-only replies and mid-message examples.

## Reproduction Assessment

Yes. On the PR head, pass `normalizeReplyPayloadsForDelivery()` a leading `Conversation info` block followed by `context 1 (untrusted metadata):` fenced JSON and visible reply text; the first block is stripped but the generated structured block remains deliverable.

## Solution Assessment

No, not yet. The outbound plan boundary is the narrow maintainable place to fix this, but the helper it calls must recognize generated structured metadata labels before this is complete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Strip generated structured metadata blocks:** `src/infra/outbound/payloads.ts:141`
  - body: The sanitizer call here still relies on `stripLeadingInboundMetadata()`, whose sentinel list contains only fixed labels plus Location. `buildInboundUserContextPrefix()` can also emit generated `context N (untrusted metadata):` or custom structured blocks from channel `untrustedContext`; after the first recognized block is removed, those blocks remain in delivered text. Please cover generated structured metadata and add a regression payload with one of those blocks.
  - confidence: 0.89
