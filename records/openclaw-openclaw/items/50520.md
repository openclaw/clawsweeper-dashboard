## Summary

Keep open: current main still has the reported outbound leak, and this PR is the canonical fix path, but the PR head still misses a generated reply-chain metadata header and has no real channel behavior proof.

## What This Changes

The branch strips leading inbound metadata before outbound reply directive parsing, extends metadata sentinel coverage for location and dynamic labels, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land this PR or a replacement after adding reply-chain sentinel coverage, focused tests, and redacted real Discord or Telegram delivery proof.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to pass `normalizeReplyPayloadsForDelivery` a payload beginning with generated Conversation/Sender/Reply-chain metadata; current main parses `payload.text` before any inbound metadata stripping.

## Solution Assessment

No, not as currently proposed. The outbound plan boundary is the narrow maintainable fix point, but the sanitizer must also recognize the generated reply-chain header and the PR needs real channel delivery proof before merge.
