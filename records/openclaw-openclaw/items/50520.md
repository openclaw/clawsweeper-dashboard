## Summary

Keep open: the PR fixes the right outbound planning boundary, but the proposed sanitizer still misses a generated reply-chain metadata header, so internal context can still leak into delivered channel text. It also lacks after-fix real Discord or Telegram delivery proof.

## What This Changes

The PR strips leading inbound metadata before outbound reply directive parsing and adds sanitizer/outbound regression coverage plus a changelog entry.

## Best Possible Solution

Update the sanitizer to cover every generated inbound metadata header, especially reply-chain blocks, keep the outbound-plan call site, and require redacted real channel delivery proof before landing.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to pass `normalizeReplyPayloadsForDelivery` a payload beginning with generated Conversation/Sender or reply-chain metadata; current main sends raw `payload.text` into directive parsing before any metadata strip.

## Solution Assessment

No, not as currently proposed. Sanitizing at `createOutboundPayloadPlanEntry` is the narrow maintainable fix point, but the PR must also recognize generated reply-chain headers and provide real channel delivery proof.
