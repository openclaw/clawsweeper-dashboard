## Summary

Keep open: the bug is valid and the shared outbound boundary is the right fix direction, but this PR still misses one generated reply-chain metadata header and lacks required real channel proof for an external PR.

## What This Changes

The PR calls `stripLeadingInboundMetadata` before outbound reply directive parsing, expands metadata sentinel coverage for location and dynamic structured labels, adds regression tests, and updates the changelog.

## Best Possible Solution

Land a rebased narrow fix that sanitizes leading inbound metadata at `createOutboundPayloadPlanEntry`, covers every generated metadata header including reply-chain blocks, preserves timestamp-only and mid-message examples, and includes real channel delivery proof.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: pass `normalizeReplyPayloadsForDelivery` a payload beginning with generated Conversation/Sender/Reply-chain metadata; current main parses raw `payload.text` before any inbound-metadata stripping.

## Solution Assessment

No, not as currently proposed. The outbound plan boundary is the narrow maintainable fix, but the sanitizer must also recognize the generated reply-chain header and the PR needs real channel proof before merge.
