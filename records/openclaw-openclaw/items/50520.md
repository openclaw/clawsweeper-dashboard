## Summary

Keep open: this PR targets a real outbound privacy bug, but the current head still misses one generated reply-chain metadata header and only has test/CI proof rather than real channel delivery proof.

## What This Changes

The PR strips leading inbound metadata before outbound reply directive parsing, expands metadata sentinel coverage for location and dynamic structured labels, adds regression tests, and updates the changelog.

## Best Possible Solution

Land a narrow fix that sanitizes at `createOutboundPayloadPlanEntry`, covers every generated inbound metadata header including reply-chain, location, and dynamic labels, preserves clean timestamp and mid-message examples, and includes redacted real channel delivery proof.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to pass `normalizeReplyPayloadsForDelivery` a payload beginning with generated Conversation/Sender/Reply-chain metadata; current main parses raw `payload.text` before any inbound metadata stripping.

## Solution Assessment

No, not as currently proposed. The outbound plan boundary is the right fix point, but the sanitizer must also recognize the generated reply-chain header and the PR needs real channel delivery proof before merge.
