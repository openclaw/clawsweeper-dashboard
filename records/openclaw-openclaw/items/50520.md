## Summary

Keep open. Current main still builds outbound delivery plans from raw reply text, so the privacy-facing metadata echo leak remains, but this PR is dirty against main and its current diff introduces sanitizer-scope regressions that should block merge until maintainers choose the canonical branch and narrower helper behavior.

## What This Changes

The PR imports `stripInboundMetadata` into outbound payload normalization, strips payload text before reply directive parsing, adds an outbound echo regression test, and changes shared timestamp-only `stripInboundMetadata` expectations.

## Best Possible Solution

Choose one canonical implementation for #39847, rebase it onto `createOutboundPayloadPlanEntry()`, strip only echoed leading inbound metadata before directive parsing with an outbound-safe helper, keep shared `stripInboundMetadata` timestamp behavior intact, and add coverage for stripped leading echoes, preserved timestamp-only outbound replies, and intentionally quoted metadata examples.

## Reproduction Assessment

Yes. On current main, a `ReplyPayload` whose text starts with `Conversation info (untrusted metadata):` followed by user-visible response text enters `parseReplyDirectives(payload.text ?? "", ...)` raw, then flows through the outbound delivery plan without metadata stripping.

## Solution Assessment

No, not as-is. The PR targets the right delivery gap, but the broad shared sanitizer call and shared timestamp semantic change are not the narrowest maintainable fix; the safer path is a local outbound-leading sanitizer in the current plan-entry flow.
