## Summary

Keep open: this is a real privacy-sensitive outbound delivery bug and this PR is the canonical fix path, but the current head still misses a generated reply-chain metadata header and only reports tests/CI rather than real channel delivery proof.

## What This Changes

The PR strips leading inbound metadata before outbound reply directive parsing, expands sanitizer coverage for location and dynamic metadata labels, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Keep the shared outbound-plan fix point, add reply-chain metadata coverage to the sanitizer and tests, then require redacted real channel delivery proof before landing.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to pass `normalizeReplyPayloadsForDelivery` a payload beginning with generated Conversation/Sender or reply-chain metadata; current main sends raw text into directive parsing without stripping.

## Solution Assessment

No, not as currently proposed. Sanitizing at `createOutboundPayloadPlanEntry` is the narrow maintainable approach, but the sanitizer must also recognize the generated reply-chain header and the PR needs real channel proof.
