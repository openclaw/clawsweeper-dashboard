## Summary

Keep open. The refreshed PR now uses the right outbound boundary and avoids the earlier broad-stripping regression, but it still leaves one generated inbound metadata block deliverable and lacks the required changelog entry for a user-facing fix.

## What This Changes

The PR calls `stripLeadingInboundMetadata()` before outbound reply directive parsing and adds delivery normalization regression tests for leading metadata echoes, directives, timestamp prefixes, and metadata-looking examples.

## Best Possible Solution

Use the leading-only outbound sanitizer before directive parsing, keep timestamp-only and mid-message examples intact, extend generated sentinel coverage to include location metadata, and add the required changelog entry.

## Reproduction Assessment

Yes. Current main can be reproduced with a `ReplyPayload` whose text starts with `Conversation info (untrusted metadata):`; for the remaining PR gap, add a generated `Location (untrusted metadata):` block after a stripped leading block and the helper leaves it in delivered text.

## Solution Assessment

No, not yet. The outbound call site is the right narrow fix, but the sanitizer contract must cover the static metadata labels emitted by `buildInboundUserContextPrefix()` before this is the best implementation.
