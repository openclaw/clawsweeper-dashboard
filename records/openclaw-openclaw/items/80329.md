## Summary

Keep open. Current main has the OpenAI-compatible HTTP operator surface and sandbox env composition seams, but it does not provide hidden per-call sandbox env ingress; adjacent open items cover static sandbox SecretRefs or runtime metadata, not this specific gateway feature. Because the request adds a new header/config surface for secret-bearing data, it needs maintainer product and security design rather than conservative cleanup.

## What This Changes

The issue requests an opt-in OpenAI-compatible gateway mechanism for trusted callers to inject per-call sandbox environment variables into tool execution without exposing values to the model.

## Best Possible Solution

If accepted, add a documented, off-by-default gateway option and command-context carrier that injects validated request-scoped env only at sandbox exec spawn, with tests proving values bypass prompts/transcripts/logs and dangerous env names are rejected.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Source inspection gives high confidence that current main has no hidden per-call sandbox env path from /v1/chat/completions to sandbox exec.

## Solution Assessment

Unclear until maintainers choose the product and security contract. A header-based opt-in fits the existing x-openclaw convention, but the exact header shape, allowed names, redaction, lifetime, and interaction with existing sandbox env config need design review first.

## Security Review

Status: needs_attention

Summary: The request is security-sensitive because it would move operator-supplied secret values into untrusted tool execution environments.

- **[medium] Define secret redaction and non-persistence before implementation:**
