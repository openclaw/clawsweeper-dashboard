## Summary

Keep open. Current main still has no request-scoped, model-hidden sandbox env path from the OpenAI-compatible HTTP endpoint into sandbox exec, and the requested trusted header/config surface is security-sensitive product work rather than cleanup closure.

## What This Changes

The issue requests an off-by-default way for operator-trusted OpenAI HTTP callers to inject per-call sandbox environment variables that reach tool exec without entering model context or persisted session state.

## Best Possible Solution

If accepted, add a documented off-by-default gateway option that validates request env names, carries values through command context without prompt/session persistence, merges them only at sandbox process spawn, and proves redaction and non-leakage with focused tests.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Source inspection gives high confidence that current main has no hidden per-call sandbox env path from `/v1/chat/completions` to sandbox exec.

## Solution Assessment

Unclear until maintainers define the product and security contract. A header-based opt-in matches nearby `x-openclaw-*` conventions, but the header format, validation, redaction, lifetime, and relation to configured sandbox env need explicit design approval.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would pass operator-supplied secret values into executable sandbox environments.

  - confidence: 0.9
