## Summary

Keep open. Current main still lacks a request-scoped, model-hidden path from the OpenAI-compatible HTTP endpoint into sandbox exec env, and accepting this would add a new secret-bearing gateway/config surface that needs maintainer product and security design.

## What This Changes

Request to let operator-trusted OpenAI-compatible HTTP callers inject per-call sandbox environment variables without exposing the values to the model prompt.

## Best Possible Solution

If accepted, add a documented, off-by-default gateway option and command-context carrier that injects validated request-scoped env only at sandbox exec spawn, with tests proving values bypass prompts, transcripts, persisted state, and logs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable: this is a feature request, not a broken existing behavior report. Source inspection gives high confidence that current main has no hidden per-call sandbox env path from `/v1/chat/completions` to sandbox exec.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the product and security contract. Header-based opt-in fits nearby `x-openclaw-*` conventions, but the exact wire shape, validation, redaction, lifetime, and relationship to broader credential-isolation work need design review first.
