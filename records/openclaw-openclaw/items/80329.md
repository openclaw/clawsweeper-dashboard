## Summary

Keep open. Current main still has no request-scoped, model-hidden sandbox env path from the OpenAI-compatible HTTP endpoint into sandbox exec, and this would add a new secret-bearing gateway/config contract that needs maintainer security and product design.

## What This Changes

Request to let trusted OpenAI-compatible HTTP callers inject per-call environment variables into sandboxed tool execution without exposing the values to the model or persisted context.

## Best Possible Solution

If accepted, add a documented off-by-default gateway option and command-context carrier that injects validated request-scoped env only at sandbox exec spawn, with tests proving values bypass prompts, transcripts, persisted state, and logs.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a new capability request. Source inspection gives high confidence that current main has no supported hidden per-call sandbox env path from `/v1/chat/completions` to sandbox exec.

## Solution Assessment

Unclear until maintainers choose the product and security contract. Header-based opt-in fits nearby `x-openclaw-*` conventions, but the exact header shape, validation, redaction, lifetime, and interaction with configured sandbox env need design review first.
