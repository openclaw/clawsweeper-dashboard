## Summary

Keep open: current main still has no per-call OpenAI-compatible header or request-context path that injects trusted caller secrets into sandbox exec env, and the earlier related custom-header request was closed specifically in favor of this issue as the canonical design thread. The request is plausible but security- and product-sensitive because it adds a new operator credential path into executable environments.

## What This Changes

Requests an opt-in OpenAI-compatible HTTP header mechanism for trusted callers to pass per-request sandbox environment variables into exec without exposing values to the model.

## Best Possible Solution

Design an explicit operator-only, opt-in gateway-to-exec env contract with strict name validation, value redaction, no prompt or persistence path, and focused tests for both rejected dangerous env names and successful sandbox-only injection.

## Reproduction Assessment

Not applicable as a bug reproduction. Source inspection gives high confidence that current main lacks the requested per-call sandbox-env header/config path.

## Solution Assessment

Unclear. The requested direction is plausible, but maintainers need to choose the permanent API and security boundary before implementation; a narrow patch that only forwards headers would be too risky.
