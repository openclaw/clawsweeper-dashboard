## Summary

Keep open. Current main still has no request-scoped, model-hidden sandbox env carrier from the OpenAI-compatible HTTP endpoint into sandbox tool execution, and the requested trusted header/config surface needs maintainer product and security design before implementation.

## What This Changes

The issue requests an off-by-default trusted HTTP mechanism for injecting per-call sandbox environment variables into tool execution without exposing the values to the model.

## Best Possible Solution

If accepted, add a documented off-by-default gateway option and command-context carrier that injects validated request-scoped env only at sandbox spawn, with tests proving values bypass prompts, transcripts, persisted state, and logs.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Source inspection gives high confidence that current main has no hidden per-call sandbox env path from /v1/chat/completions to sandbox exec.

## Solution Assessment

Unclear until maintainers choose the product and security contract. A header-based opt-in fits nearby x-openclaw-* conventions, but the wire shape, validation, redaction, lifetime, and precedence rules need design review first.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would move operator-supplied secret values into executable sandbox environments.

Concerns:

- **[medium] Define redaction and non-persistence before implementation:** `src/gateway/openai-http.ts:957`
  - body: Request-scoped secret values must not enter prompts, transcripts, config snapshots, logs, or persisted session state; tests should prove that only names or redacted values can surface outside the spawned process env.
  - confidence: 0.88
- **[medium] Validate injected env names before spawn:** `src/infra/host-env-security.ts:95`
  - body: Request-supplied env can alter process behavior if PATH, loader hooks, shell startup vars, proxy/TLS controls, or other dangerous names are accepted without the existing host env security policy.
  - confidence: 0.86
- **[medium] Document the operator and tool-output boundary:** `docs/gateway/openai-http-api.md:47`
  - body: The OpenAI-compatible endpoint is documented as full operator access, but env values handed to sandboxed tools can still be printed by skill code, so the accepted contract should make that exposure explicit.
  - confidence: 0.82
