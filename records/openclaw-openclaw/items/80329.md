## Summary

Keep open: current main and the latest release still have no request-scoped OpenAI-compatible header/config path that carries trusted caller secrets into sandbox exec env, and the earlier custom-header report was closed in favor of this issue as the canonical design thread. The request is plausible, but it adds a new credential-to-exec-env surface and needs maintainer product/security direction before implementation.

## What This Changes

The issue requests an off-by-default operator-trusted per-call mechanism to inject sandbox environment variables into tool execution without exposing those values to the model prompt.

## Best Possible Solution

Design an explicit operator-only, opt-in gateway-to-sandbox-env contract with strict env-name validation, value redaction, no prompt or persistence path, and focused tests for accepted sandbox-only injection plus rejected dangerous variables.

## Reproduction Assessment

Not applicable as a bug reproduction, but the absence is source-reproducible: current main has no per-call sandbox-env carrier in OpenAI HTTP request context, agent command ingress, or sandbox env composition. The latest release has the same relevant shape.

## Solution Assessment

Unclear. The requested direction is plausible, but a narrow header-forwarding patch would not be the best solution without a maintainer-approved security/API boundary for auth, validation, redaction, and lifetime.
