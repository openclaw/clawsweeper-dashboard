## Summary

Keep open: current main and v2026.6.10 still do not provide a request-scoped OpenAI-compatible header or config path that carries trusted caller secrets into sandbox exec env, and the requested credential-to-exec surface needs maintainer product/security direction before implementation.

## What This Changes

The issue requests an opt-in OpenAI-compatible gateway mechanism for trusted callers to inject per-request sandbox environment variables into tool execution without exposing the values to the model.

## Best Possible Solution

Define an explicit operator-only, opt-in gateway-to-sandbox-env contract with strict env-name validation, value redaction, no prompt or persistence path, and focused tests before implementing the header path.

## Reproduction Assessment

Not applicable as a bug reproduction, but yes for source-level absence: current main has no per-call sandbox-env carrier in OpenAI HTTP request parsing, agent command ingress, gateway config, or sandbox env composition.

## Solution Assessment

Unclear. The requested header direction is plausible, but a narrow header-forwarding patch would not be the best solution without a maintainer-approved security/API boundary for auth, validation, redaction, and lifetime.
