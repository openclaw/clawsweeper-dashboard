## Summary

Keep open. Current main has the OpenAI-compatible operator HTTP surface and sandbox env composition seams, but no hidden per-call sandbox env ingress; because the request adds a new secret-bearing header/config contract, it needs maintainer product and security design instead of conservative cleanup.

## What This Changes

Add an opt-in way for trusted OpenAI-compatible HTTP callers to inject request-scoped environment variables into sandboxed tool execution without exposing those values to the model.

## Best Possible Solution

If accepted, add a documented off-by-default gateway option and command-context carrier that injects validated request-scoped env only at sandbox exec spawn, with tests proving values bypass prompts, transcripts, persisted config, and logs.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Source inspection gives high confidence that current main has no hidden per-call sandbox env path from /v1/chat/completions to sandbox exec.

## Solution Assessment

Unclear until maintainers choose the product and security contract. A header-based opt-in fits the existing x-openclaw convention, but the header shape, config name, redaction, lifetime, and interaction with configured sandbox env need design review first.
