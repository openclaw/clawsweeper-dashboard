## Summary

Keep open. Current main still has no request-scoped, model-hidden sandbox environment channel from the OpenAI-compatible HTTP request into sandboxed tool execution, and the requested trusted-header/config/security contract needs maintainer product and security review rather than stale cleanup.

## What This Changes

The issue requests an opt-in per-call HTTP header mechanism for operator-trusted callers to inject short-lived environment variables into sandboxed tool execution without exposing the values to the model context.

## Best Possible Solution

Design this as an explicit opt-in gateway security feature with a typed request-scoped env map, sanitizer/redaction reuse, tests for non-persistence and prompt exclusion, and docs that define the operator-trusted boundary.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Source inspection confirms the described existing mechanisms are static config, routing/context headers, or prompt-visible files rather than the requested per-call sandbox env path.

## Solution Assessment

Unclear until maintainers choose the security contract. The proposed opt-in header shape is plausible, but the best solution needs explicit agreement on trusted callers, env-name validation, redaction, non-persistence, and whether this belongs in the OpenAI HTTP endpoint or a more general gateway/plugin API.
