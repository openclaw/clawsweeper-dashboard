## Summary

Keep open. Current main still lacks a request-scoped, model-hidden sandbox env path from the OpenAI-compatible HTTP endpoint into sandbox exec, and the requested header/config surface is a security-sensitive feature that needs maintainer product review.

## What This Changes

The issue requests an off-by-default way for trusted OpenAI-compatible HTTP callers to inject per-call sandbox environment variables without putting values into model context.

## Best Possible Solution

If accepted, add a documented, off-by-default gateway option and command-context carrier that injects validated request-scoped env only at sandbox exec spawn, with tests proving values bypass prompts, transcripts, persisted state, and logs.

## Reproduction Assessment

Not applicable: this is a feature request, not a broken existing behavior report. Source inspection gives high confidence that current main has no hidden per-call sandbox env path from `/v1/chat/completions` to sandbox exec.

## Solution Assessment

Unclear until maintainers choose the product and security contract. Header-based opt-in fits nearby `x-openclaw-*` conventions, but the exact header shape, validation, redaction, lifetime, and interaction with configured sandbox env need design review first.

## Security Review

Status: needs_attention

Summary: The request is security-sensitive because it would move operator-supplied secret values into executable sandbox environments.

Concerns:

- **[medium] Define redaction and non-persistence before implementation:** `src/gateway/openai-http.ts:597`
  - body: Request-scoped secret values must not enter prompts, transcripts, config snapshots, logs, or persisted session state; tests should prove that only names or redacted values can surface outside the spawned process env.
  - confidence: 0.88
- **[medium] Validate injected env names before spawn:** `src/infra/host-env-security.ts:95`
  - body: Request-supplied env can alter process behavior if names such as PATH, loader hooks, shell startup vars, proxy controls, or TLS controls are accepted without the existing host env security policy.
  - confidence: 0.86
- **[medium] Document the operator and tool-output boundary:** `docs/gateway/openai-http-api.md:47`
  - body: The OpenAI-compatible endpoint is documented as full operator access, but env values handed to sandboxed tools can still be printed by skill code, so the accepted contract should make that exposure explicit.
  - confidence: 0.82
