## Label Justifications

- `P1`: A supported core `sessions_spawn` workflow can leave spawned subagents accepted but inert, blocking multi-agent workflows for affected users.
- `impact:session-state`: The reported failure leaves child sessions in a misleading running state with no messages, transcript, system prompt marker, or token accounting.
- `impact:crash-loop`: The issue is about a hang or availability failure where spawned runs remain stuck until timeout instead of starting or failing cleanly.

## Summary

Keep open: current main and the latest release still route generated `sessions_spawn` child sessions through the supported `non-main` sandbox path after the gateway has already returned `accepted`, and the related Docker-diagnostics PR was closed unmerged as the wrong fix shape for this specific failure.

## What This Changes

The issue reports that `agents.defaults.sandbox.mode: "non-main"` causes `sessions_spawn` child sessions to be accepted but never start executing.

## Best Possible Solution

Keep `non-main` sandboxing supported and make Docker-backed native child runs either reach prompt/transcript startup or transition to a terminal failed state with actionable sandbox startup diagnostics and regression coverage.

## Reproduction Assessment

Yes at source level, but not as a live Docker-backed current-main repro in this review. Current main still creates generated child subagent keys, classifies them as sandboxed in `non-main` mode, and resolves Docker sandbox context after the accepted gateway run path.

## Solution Assessment

Not applicable as a patch review. The maintainable fix should stay in the native subagent startup/lifecycle path and should not disable `non-main` sandboxing or rely only on generic Docker diagnostics.
