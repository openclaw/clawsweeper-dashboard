## Summary

Keep this issue open. Current main still routes native `sessions_spawn` children through a non-main sandboxed child session and still acknowledges the child `agent` run before sandbox/container initialization is proven. There is generic async lifecycle error plumbing now, but I did not find a focused Docker-backed regression or live proof that the reported zero-message/zero-token sandboxed child either starts cleanly or transitions to an actionable error state.

## What This Changes

The issue requests that `sessions_spawn` subagents work, or fail visibly, when `agents.defaults.sandbox.mode` is set to `"non-main"`.

## Best Possible Solution

Add a focused regression or live Docker proof for native `sessions_spawn` under `agents.defaults.sandbox.mode: "non-main"`. The end state should prove that the child Docker-backed run records transcript/session progress when startup succeeds, and that sandbox/container startup failures move the child run and background task to a visible error or timeout state with actionable logs/status instead of leaving a running zero-token child.
