## Summary

Keep open: current main still supports `non-main` sandboxed subagent sessions and still routes generated child keys through sandbox resolution after the non-blocking accepted acknowledgement; no merged PR or release evidence shows this accepted-but-inert failure is fixed.

## What This Changes

The issue reports that `agents.defaults.sandbox.mode: "non-main"` causes `sessions_spawn` child runs to be accepted but never initialize, emit messages, write transcripts, or log errors.

## Best Possible Solution

Keep `non-main` sandboxing supported and make Docker-backed child subagent runs either start normally or reach a terminal failed state with actionable sandbox startup diagnostics and regression coverage.

## Reproduction Assessment

No high-confidence current-main live reproduction was established. Source inspection still shows the reported path: generated non-main child keys are accepted first, then sandbox context is resolved during embedded-runner startup.

## Solution Assessment

Not applicable as a patch review. The maintainable fix should stay in the native subagent startup/lifecycle path and should not disable `non-main` sandboxing or rely only on generic Docker diagnostics.
