## Summary

Keep open. Current main still documents `non-main` as a supported sandbox mode, native `sessions_spawn` still creates generated child session keys that are classified as sandboxed, and the gateway still acknowledges the child run before the async startup path executes; no merged PR or current-main proof shows that Docker-backed child startup now succeeds or fails terminally with diagnostics.

## What This Changes

The issue reports that `agents.defaults.sandbox.mode: "non-main"` causes native `sessions_spawn` subagents to be accepted but never initialize, leaving empty history, no transcript, no token accounting, and no gateway error.

## Best Possible Solution

Keep `non-main` sandboxing supported and make Docker-backed native child runs either reach prompt/transcript startup or transition to a terminal failed state with actionable sandbox startup diagnostics and regression coverage.

## Reproduction Assessment

Yes at source level, but not as a live Docker repro in this review. Current main still creates generated child session keys, classifies them as sandboxed in `non-main` mode, and acknowledges the run before child sandbox startup executes.

## Solution Assessment

Not applicable as a patch review. The maintainable fix should stay in the native subagent startup/lifecycle path and should not disable `non-main` sandboxing or rely only on generic Docker diagnostics.
