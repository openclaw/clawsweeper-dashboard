## Summary

Keep open. Current main still supports `non-main` sandboxing but the native `sessions_spawn` path can acknowledge a child before that child reaches Docker sandbox setup, and the only related diagnostics PR was closed unmerged as the wrong fix shape for this sessions_spawn bug.

## What This Changes

Fix `sessions_spawn` so subagents inheriting `agents.defaults.sandbox.mode: "non-main"` initialize or fail visibly instead of stalling after an accepted status.

## Best Possible Solution

Keep `non-main` sandboxing supported and make native sandboxed child runs either reach prompt/transcript startup or transition to a terminal failed state with actionable Docker/startup diagnostics.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes, source inspection shows the current child key, `non-main` sandbox classification, accepted-before-dispatch gateway path, and sandbox setup before prompt progress; a live Docker run was not performed in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Not applicable as a patch review; the maintainable fix belongs in the native subagent startup/lifecycle path, not by disabling `non-main` sandboxing or merging Docker-only diagnostics that leave sessions_spawn state handling unchanged.
