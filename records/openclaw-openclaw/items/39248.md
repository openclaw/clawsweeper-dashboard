## Summary

Keep open. Current main still supports and routes generated subagent sessions through the `non-main` sandbox path, while the only linked Docker diagnostics PR was closed unmerged and recent registry durability fixes only cover adjacent missing-run-row symptoms, not the reported Docker-backed child startup stall.

## What This Changes

The issue reports that `agents.defaults.sandbox.mode: "non-main"` causes `sessions_spawn` subagents to be accepted but never initialize, with no messages, tokens, transcript, or useful error.

## Best Possible Solution

Keep `non-main` sandboxing supported and make native sandboxed child runs either reach prompt/transcript startup or transition to a terminal failed state with actionable Docker/startup diagnostics and regression coverage.

## Reproduction Assessment

Yes at source level: current main still creates a non-main child key, classifies it as sandboxed, and acknowledges `sessions_spawn` before the child sandbox startup path runs. I did not establish a live Docker-backed failing run against current main, so this is not a live reproduction.

## Solution Assessment

Not applicable as a patch review. The maintainable fix should stay in the native subagent startup/lifecycle path and should not disable `non-main` sandboxing or rely only on generic Docker diagnostics.
