## Summary

Keep open: current main still routes generated subagent sessions through the supported `non-main` sandbox path only after returning an accepted run, and neither source history nor the related closed diagnostics PR proves that sandboxed children now initialize successfully. The issue also carries a protected maintainer-review label and needs live current-main Docker proof before repair routing.

## What This Changes

Fix `sessions_spawn` children that are accepted but never initialize when `agents.defaults.sandbox.mode` is `non-main`.

## Best Possible Solution

Keep the non-blocking spawn contract, but make Docker sandbox preparation guarantee one of two observable outcomes: the child begins execution, or the run and child session transition promptly to a terminal error with sanitized, actionable gateway diagnostics; validate both successful and forced-failure paths in a real Docker setup.

## Reproduction Assessment

No, not yet at high confidence on current main. The report provides a concrete historical procedure and source confirms the accepted-before-sandbox-start lifecycle, but modern gateway error handling makes a real Docker-backed success and forced-failure run necessary to prove the current symptom.

## Solution Assessment

Yes, conditionally: repairing terminal-state and diagnostic propagation at the post-accept sandbox boundary is the narrowest maintainable direction, provided live proof also confirms successful `non-main` child initialization and the work does not expand into unrelated sandbox inheritance semantics.
