## Summary

Keep open. Current main still supports `agents.defaults.sandbox.mode: "non-main"`, still classifies generated native subagent sessions as sandboxed, and still acknowledges `sessions_spawn` before async child dispatch; the related Docker diagnostics PR was closed unmerged after being narrowed away from this lifecycle path.

## What This Changes

Fix the `sessions_spawn` path so subagents created under `agents.defaults.sandbox.mode: "non-main"` actually initialize or fail visibly instead of staying accepted/running with no messages or tokens.

## Best Possible Solution

Keep `non-main` sandboxing supported and make native sandboxed child runs either reach prompt/transcript startup or transition to a terminal failed state with actionable Docker/startup diagnostics and regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main creates a non-main subagent key, classifies it as sandboxed, and acknowledges before async child dispatch; a live Docker run was not performed in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Not applicable as a patch review; the maintainable fix belongs in the native subagent startup/lifecycle path, not by disabling `non-main` sandboxing or merging Docker-only diagnostics that leave child session state handling unchanged.
