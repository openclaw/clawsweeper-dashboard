## Summary

Keep open. Current main still has the supported `non-main` sandbox classification and asynchronous `sessions_spawn` startup path described by the report, while the related Docker diagnostics PR was closed unmerged and the Docker-specific duplicate was closed in favor of this issue as the canonical tracker.

## What This Changes

The issue reports that `agents.defaults.sandbox.mode: "non-main"` causes native `sessions_spawn` child sessions to be accepted but never initialize, emit messages, write transcripts, or surface startup errors.

## Best Possible Solution

Keep `non-main` sandboxing supported and make native sandboxed child runs either reach prompt/transcript startup or transition to a terminal failed state with actionable Docker/startup diagnostics and regression plus live Docker proof.

## Reproduction Assessment

Yes at source level, but not as a live Docker repro here. Current main still creates a generated child session key, classifies it as sandboxed under `non-main`, and acknowledges the run before child sandbox startup executes.

## Solution Assessment

Not applicable as a patch review. The maintainable fix should stay in the native subagent startup/lifecycle path and should not disable `non-main` sandboxing or rely only on generic Docker diagnostics.
