## Summary

Keep open. Current main still treats spawned native subagents as non-main sandboxed sessions, resolves the Docker sandbox before normal run progress, and does not contain the related startup lifecycle signals; the linked lifecycle/sandbox work remains open.

## What This Changes

The issue reports that `agents.defaults.sandbox.mode: "non-main"` lets `sessions_spawn` return an accepted child run that never initializes, logs, records transcript output, or consumes tokens.

## Best Possible Solution

Land or adapt focused startup error handling with Docker-backed regression coverage so non-main sandboxed `sessions_spawn` children either make first progress or move to a visible terminal error with actionable logs.

## Reproduction Assessment

Yes, source-reproducible with high confidence. The reporter provided concrete config and commands, and current main still sends native child session keys through the non-main Docker sandbox startup path before any transcript/token progress can occur.

## Solution Assessment

No, closing is not the best way to solve it. The narrow maintainable path is to review the open startup-lifecycle work and add sandbox-specific proof for the non-main native subagent case.
