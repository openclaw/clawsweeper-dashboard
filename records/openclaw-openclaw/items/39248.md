## Summary

Keep open: current main still routes `sessions_spawn` children into the supported non-main sandbox path after returning an accepted run, and there is no merged fix or live current-main proof that the accepted-then-stalled child behavior is resolved.

## What This Changes

Reporter asks that `sessions_spawn` subagents initialize or fail visibly when `agents.defaults.sandbox.mode` is `non-main` instead of returning accepted and stalling.

## Best Possible Solution

Make sandboxed native `sessions_spawn` either start the child run or surface Docker/sandbox startup failure into logs and child session/subagent state, with live Docker proof; leave target-sandbox/env inheritance semantics to the related open items.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not yet from a live current-main run; the user supplied concrete steps and source still shows the accepted-before-sandbox-start path, but this needs a forced Docker startup failure or real non-main sandboxed spawn run for high-confidence proof.

## Solution Assessment

Is this the best way to solve the issue? Unclear until live proof pins the current failure point; the likely best fix is a focused lifecycle/logging repair for sandbox startup failures, not a broad sandbox inheritance redesign.
