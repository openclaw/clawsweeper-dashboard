## Summary

Keep open. Current main still documents `non-main` sandboxing and still routes generated subagent session keys into the sandboxed child-run path after returning an accepted response; no merged PR fixes the reported startup/terminal-state gap, and the only related Docker diagnostics PR was closed unmerged after being scoped away from `sessions_spawn`.

## What This Changes

The issue asks for `sessions_spawn` subagents to initialize or fail clearly when `agents.defaults.sandbox.mode` is `"non-main"` instead of stalling silently after an accepted response.

## Best Possible Solution

Keep `non-main` sandboxing supported and make native sandboxed child runs either reach prompt/transcript startup or transition to a terminal failed state with actionable Docker/startup diagnostics and regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at source level: current main creates a non-main subagent key, acknowledges before dispatch, and resolves Docker sandbox context before prompt progress; a live Docker run was not performed in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Not applicable as a patch review; the maintainable fix belongs in the native subagent startup/lifecycle path, not by disabling `non-main` sandboxing or merging Docker-only diagnostics that leave child session state handling unchanged.
