## Summary

Keep open. Current main still supports `non-main` sandboxing for generated subagent sessions, and the only linked Docker-diagnostics PR was closed unmerged after being narrowed away from this `sessions_spawn` lifecycle failure.

## What This Changes

The issue reports that `agents.defaults.sandbox.mode: "non-main"` causes native `sessions_spawn` child sessions to be accepted but never initialize or record messages/tokens.

## Best Possible Solution

Keep `non-main` sandboxing supported and make native sandboxed child runs either reach prompt/transcript startup or transition to a terminal failed state with actionable Docker/startup diagnostics and regression coverage.

## Reproduction Assessment

Yes at source level: current main still creates a non-main subagent key, classifies it as sandboxed, and acknowledges `sessions_spawn` before async child dispatch. I did not perform a live Docker reproduction in this read-only review.

## Solution Assessment

Not applicable as a patch review; the maintainable fix belongs in the native subagent startup/lifecycle path. Disabling `non-main` sandboxing or merging Docker-only diagnostics would leave the child session state problem unresolved.
