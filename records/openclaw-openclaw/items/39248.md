## Summary

Keep open: current main still treats generated subagent session keys as sandboxed in `non-main` mode and still acknowledges the child run before the sandbox-aware runner path starts. No merged PR or canonical issue fully owns this exact accepted-but-inert `sessions_spawn` failure yet.

## What This Changes

The issue reports that `agents.defaults.sandbox.mode: "non-main"` makes `sessions_spawn` child sessions return accepted but never initialize, emit tokens, write transcripts, or log actionable errors.

## Best Possible Solution

Keep `non-main` sandboxing supported and make Docker-backed child subagent runs either start normally or reach a terminal failed state with actionable sandbox startup diagnostics and regression coverage.

## Reproduction Assessment

No live high-confidence current-main reproduction was established in this read-only review. Source inspection still shows the relevant path: a generated non-main child key is accepted first, then routed into sandbox-aware embedded-runner startup.

## Solution Assessment

Not applicable as a patch review. The maintainable fix should stay in the native subagent startup/lifecycle path and should not disable `non-main` sandboxing or rely only on generic Docker diagnostics.
