## Summary

Keep open: current main still routes generated non-main subagent sessions into sandbox startup after `sessions_spawn` has acknowledged the run, and the related Docker diagnostics PR closed unmerged after removing the `sessions_spawn` scope.

## What This Changes

The issue reports that `agents.defaults.sandbox.mode: "non-main"` leaves `sessions_spawn` child sessions accepted but inert instead of starting or failing cleanly.

## Best Possible Solution

Keep `non-main` sandboxing supported and make Docker-backed native child runs either reach prompt/transcript startup or transition to a terminal failed state with actionable sandbox startup diagnostics and regression coverage.

## Reproduction Assessment

Yes at source level, but not as a live Docker-backed current-main repro in this review. Current main still creates generated child subagent keys, classifies them as sandboxed in `non-main` mode, and starts sandbox resolution after the accepted response path.

## Solution Assessment

Not applicable as a patch review. The maintainable fix should stay in the native subagent startup/lifecycle path and should not disable `non-main` sandboxing or rely only on generic Docker diagnostics.
