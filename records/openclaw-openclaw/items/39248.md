## Label Justifications

- `P1`: A supported core `sessions_spawn` workflow can leave spawned subagents accepted but inert, blocking multi-agent workflows for affected users.
- `impact:session-state`: The reported failure leaves child sessions in a misleading running state with no messages, transcript, system prompt marker, or token accounting.

## Summary

Keep open. Current `main` still documents `non-main` as supported, generated native subagent session keys still enter the sandboxed path, and the gateway still acknowledges the child run before async startup; no merged fix or live current-main proof shows Docker-backed native child startup now starts or fails cleanly.

## What This Changes

The issue reports that `agents.defaults.sandbox.mode: "non-main"` makes `sessions_spawn` accept native subagent runs that never initialize, emit messages, write transcripts, or surface errors.

## Best Possible Solution

Keep `non-main` sandboxing supported and make Docker-backed native child runs either reach prompt/transcript startup or transition to a terminal failed state with actionable sandbox startup diagnostics and regression coverage.

## Reproduction Assessment

Yes at source level, but not as a live Docker-backed current-main repro in this review. Current `main` still creates generated child session keys, classifies them as sandboxed in `non-main` mode, and acknowledges the run before child sandbox startup executes.

## Solution Assessment

Not applicable as a patch review. The maintainable fix should stay in the native subagent startup/lifecycle path and should not disable `non-main` sandboxing or rely only on generic Docker diagnostics.
