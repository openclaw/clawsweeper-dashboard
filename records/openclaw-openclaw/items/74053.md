## Summary

Keep open. Current main still builds the heartbeat runtime context with `To: sender`, and the sender resolver can still fall back to the literal `heartbeat` sentinel when no delivery target resolves. Main already has a broader system-event persistence guard that mitigates the original persisted `lastTo` corruption, but the in-turn `ctx.To` behavior remains wrong and the PR’s one-line runtime change is narrowly aligned.

## What This Changes

The PR changes heartbeat runtime context construction so `ctx.To` comes from `delivery.to`, adds a target-none regression test, and adds an Unreleased changelog entry.

## Best Possible Solution

Land the narrow heartbeat context correction on current main, keep the existing system-event session persistence guard, tighten the new regression assertion to the exact target-none value, and leave the distinct global `heartbeat.to` inheritance problem tracked by #73900 out of this PR.

## Reproduction Assessment

Yes for the narrowed runtime-context issue: with `heartbeat.target: "none"`, current main resolves no delivery recipient, can derive `sender` as `heartbeat`, and then passes that sentinel as `ctx.To`. The full persisted `lastTo` corruption path is not a high-confidence current-main repro because the system-event persistence guard now preserves existing routes.

## Solution Assessment

Yes. Deriving `ctx.To` from `delivery.to` while leaving `From: sender` alone is the narrowest maintainable fix for this field; the safer test assertion is to require `capturedTo` to be `undefined` in the target-none case.
