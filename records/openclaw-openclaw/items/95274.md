## Summary

Keep open for maintainer review: current main and the latest release still have the Windows QMD command parsing bug, while this branch is a strong proof-positive candidate fix. The remaining decision is maintainer selection of one canonical same-root PR and acceptance of the existing-config parsing boundary.

## What This Changes

This PR adds QMD-specific Windows absolute command extraction in `resolveMemoryBackendConfig` and regression tests for drive-letter, wrapper-with-args, and UNC `memory.qmd.command` values.

## Best Possible Solution

Land one canonical backend-config command-tokenization fix, with this PR as a strong candidate after exact-head validation, then close the sibling PRs and the canonical issue as appropriate.

## Reproduction Assessment

Yes. Current main still routes unquoted `memory.qmd.command` through `splitShellArgs`, whose unquoted-backslash handling strips Windows separators before QMD spawn resolution consumes the command.

## Solution Assessment

Yes. Preserving Windows drive-letter and UNC executable tokens in backend config fixes the point where data is lost; changing the shared parser globally or trying to recover in the spawn layer would be broader and riskier.
