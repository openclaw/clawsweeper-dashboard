## Summary

Keep this PR open: current main and v2026.6.9 still have the Windows QMD command parsing bug, and this branch is a focused, proof-positive candidate fix rather than a stale or safely superseded PR.

## What This Changes

The PR adds QMD-specific Windows absolute command extraction for `memory.qmd.command` and regression tests for drive-letter, wrapper, and UNC command values.

## Best Possible Solution

Land exactly one canonical backend-config command-tokenization fix, with this PR as a strong candidate, then close the same-root candidate PRs after merge.

## Reproduction Assessment

Yes. Current main still routes unquoted `memory.qmd.command` through `splitShellArgs`, whose unquoted-backslash handling strips Windows separators before the QMD spawn path consumes the command.

## Solution Assessment

Yes. Preserving Windows drive-letter and UNC executable tokens in backend config is a narrow fix at the point where data is lost; changing the shared shell parser globally or trying to recover later in the spawn layer would be broader and riskier.
