## Summary

Keep this PR open: current main and v2026.6.9 still have the QMD command backslash-stripping path, and this branch is a focused, proof-positive candidate fix. It should not be cleanup-closed before maintainers choose the canonical same-root PR and accept the existing-config parsing boundary.

## What This Changes

The branch adds a QMD-specific Windows absolute-command resolver for `memory.qmd.command` plus regression tests for drive-letter, wrapper-with-args, and UNC command paths.

## Best Possible Solution

Land exactly one canonical backend-config command-tokenization fix, with this PR as a strong candidate, then close the same-root candidate PRs after merge.

## Reproduction Assessment

Yes, source-level. Current main still routes unquoted `memory.qmd.command` through `splitShellArgs`, whose unquoted-backslash handling strips Windows separators before the QMD spawn path consumes the command.

## Solution Assessment

Yes. Preserving Windows drive-letter and UNC executable tokens in backend config is the narrowest fix at the point where data is lost; changing the shared shell parser globally or recovering later in the spawn layer would be broader and riskier.
