## Summary

Keep this PR open for maintainer selection: the patch is a focused, proof-backed candidate for the Windows QMD command parsing bug, but it changes parsing for an existing config key and several same-root PRs are open.

## What This Changes

The branch adds a QMD-specific resolver that preserves Windows absolute `memory.qmd.command` paths before shell-style parsing, plus regression tests for drive-letter, wrapper-with-args, and UNC command values.

## Best Possible Solution

Land exactly one canonical backend-config command-tokenization fix, with this PR as a strong candidate, then close the sibling PRs and the linked issue after merge.

## Reproduction Assessment

Yes, source-level. Current main still routes unquoted `memory.qmd.command` through `splitShellArgs`, whose unquoted-backslash handling strips Windows separators before QMD spawn resolution.

## Solution Assessment

Yes. Preserving Windows drive-letter and UNC executable tokens in backend config is the narrowest fix at the point where data is lost; changing the shared shell parser globally or recovering later in the spawn layer would be broader and riskier.
