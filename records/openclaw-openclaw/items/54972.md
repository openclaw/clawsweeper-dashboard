## Summary

Keep open. Current main still sends Feishu streaming replies as CardKit `card_id` interactive messages, but quoted-message and thread-history reads still parse only inline card content and never request raw CardKit content before falling back to `[Interactive Card]`.

## What This Changes

The issue asks for Feishu streaming CardKit replies to be readable by other agents when fetched as quoted messages or thread history.

## Best Possible Solution

Land one Feishu-owned read-path fix that requests validated raw CardKit content for both `getMessageFeishu()` and `listFeishuThreadMessages()`, parses returned card summary/body text before existing fallbacks, and adds focused quoted-message and thread-history regression tests.

## Reproduction Assessment

Yes. Source inspection on current main shows a streaming interactive body shaped as `{ "type": "card", "data": { "card_id": "..." } }` has no inline elements, so the parser falls through to `[Interactive Card]` and both read APIs omit raw-card retrieval params.

## Solution Assessment

Unclear for the exact Feishu API contract, but the issue direction is valid. The maintainable fix belongs in the Feishu plugin read path, coordinated with #56795 and the prior #55466 approach so only one raw-card parsing path lands.
