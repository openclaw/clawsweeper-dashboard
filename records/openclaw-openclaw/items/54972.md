## Summary

Keep open. Current main still sends Feishu streaming replies as CardKit `card_id` interactive messages, while the quoted-message and thread-history read paths parse only inline card content and do not request raw CardKit content; the overlapping open PR at https://github.com/openclaw/openclaw/pull/56795 remains unmerged and conflicting.

## What This Changes

Feishu streaming CardKit replies should be readable by other agents instead of resolving to `[Interactive Card]` when quoted or loaded from thread history.

## Best Possible Solution

Land one Feishu-owned read-path fix that requests validated raw CardKit content for both `getMessageFeishu()` and `listFeishuThreadMessages()`, parses returned summary/body text before existing fallbacks, and adds focused quoted-message and thread-history regression tests.

## Reproduction Assessment

Yes, source-reproducible. On current main, a streaming reply body shaped as `{ type: "card", data: { card_id } }` has no inline elements, so `parseInteractiveCardContent()` falls through to `[Interactive Card]` and both read APIs omit a raw-card content request.

## Solution Assessment

Unclear for the exact Feishu API contract, but the direction is valid. The narrow maintainable fix belongs in the Feishu plugin read path, coordinated with https://github.com/openclaw/openclaw/pull/56795 and the prior https://github.com/openclaw/openclaw/pull/55466 approach so only one raw-card parser lands.
