## Summary

Keep open. Current main still has the reported Feishu readback gap: streaming replies are sent as CardKit `card_id` interactive messages, but the quoted-message and thread-history read paths still parse only inline card content and never request raw CardKit content before falling back to `[Interactive Card]`. The closed #55466 branch is unmerged, and open #56795 is overlapping follow-up work rather than a shipped fix for this issue.

## What This Changes

Feishu streaming CardKit card references should be readable by other agents in quoted-message and thread-history contexts instead of surfacing `[Interactive Card]`.

## Best Possible Solution

Land one Feishu-owned read-path fix that requests validated raw CardKit content for both `getMessageFeishu()` and `listFeishuThreadMessages()`, parses returned card summary/body text such as `json_card` before the existing inline-card/post fallbacks, and adds focused regression tests for quoted-message and thread-history reads of streaming CardKit cards. Coordinate #56795 and the prior #55466 approach so maintainers choose one implementation path while keeping the separate Feishu search-indexing issue #74767 distinct.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: an interactive message body shaped as `{"type":"card","data":{"card_id":"..."}}` has no inline elements, so the current parser falls through to `[Interactive Card]`, and both read APIs omit raw-card params.

## Solution Assessment

Unclear for the exact API contract, but the issue direction is valid. The narrow maintainable fix belongs in the Feishu plugin read path, with maintainer validation of the raw-card IM API shape before merging or replacing #56795.
