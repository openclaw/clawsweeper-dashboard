## Summary

Keep open: current `main` still has a source-visible `node.list` path that can throw the reported `.trim()` TypeError, although the broader claim that this stalls every Feishu-dispatched agent session still needs live confirmation.

## What This Changes

The issue reports recurring `node.list` `TypeError: Cannot read properties of undefined (reading 'trim')` errors followed by Feishu-dispatched agent sessions receiving no replies.

## Best Possible Solution

Harden the gateway node metadata boundaries so malformed cap/command/role entries cannot make `node.list` fail, add regression tests around the malformed metadata case, and then treat any remaining Feishu no-reply behavior as a separate follow-up if it still reproduces.

## Reproduction Assessment

No, not for the full Feishu hang; I did not establish a live current-main run that stalls all sessions. Source inspection does show a credible current-main `node.list` TypeError path when malformed node metadata reaches `uniqueSortedStrings`.

## Solution Assessment

Yes, the narrowest maintainable fix is defensive normalization at the node metadata boundaries and in the catalog helper, without adding Feishu-specific logic or a new config option.
