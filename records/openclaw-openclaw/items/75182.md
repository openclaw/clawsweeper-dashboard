## Summary

Keep open. Current `main` still has the shared MEDIA parser behavior described in the report: a no-whitespace JSON suffix after a slash-prefixed local path is captured as part of the same candidate and can flow into outbound local-file loading. A focused closing PR, #75202, is already open and unmerged, so this issue should wait for that PR to land or be superseded rather than being closed or queued for a duplicate repair.

## What This Changes

The issue requests hardening shared MEDIA directive parsing so a local media path followed by serialized JSON is not passed to channel delivery as one polluted file path.

## Best Possible Solution

Land or revise #75202, or an equivalent shared-parser fix, with focused regression coverage in `src/media/parse.test.ts`; channel adapters can add defense-in-depth, but the shared parser should stop emitting polluted local paths first.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `MEDIA:/path/to/image.png"}],"details":...` is captured as one payload, split only on whitespace, accepted as a slash-prefixed local path, and then passed into local media loading. I did not run tests because this sweep was read-only.

## Solution Assessment

Yes. A shared parser-level fix with a regression test is the narrowest maintainable solution; a channel-only workaround would leave polluted `mediaUrls` available to every outbound adapter.
