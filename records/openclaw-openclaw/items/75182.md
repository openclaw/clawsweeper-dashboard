## Summary

Keep open. Current `main` still has the shared MEDIA parser behavior that can accept a slash-prefixed local path plus same-line serialized JSON as one media source, and an open closing PR (#75202) already targets this exact issue, so the issue should wait for that PR to land or be superseded.

## What This Changes

The issue reports that a malformed `MEDIA:` directive can append serialized JSON to a local media path, causing downstream file reads to target a polluted path.

## Best Possible Solution

Land or revise the linked shared-parser fix so local MEDIA paths stop at a valid media/document boundary before serialized JSON, add focused parser regression coverage, and let the linked PR close this issue after merge.

## Reproduction Assessment

Yes. The current source gives a high-confidence reproduction path: a one-line local `MEDIA:` directive with no whitespace before the JSON suffix is captured as one payload, accepted as a local path, and later read as that exact path. I did not run tests or a live channel because this review was read-only.

## Solution Assessment

Yes. A shared parser-level fix with regression coverage is the narrowest maintainable solution; a channel-only workaround would leave the same parsed polluted source available to other outbound media paths.
