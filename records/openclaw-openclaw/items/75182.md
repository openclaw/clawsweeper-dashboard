## Summary

Keep open. Current main still has the shared MEDIA parser path that can accept a slash-prefixed local path with appended no-whitespace JSON as one media source, and the linked closing PR remains open and unmerged, so this issue should wait for that PR to land or be superseded.

## What This Changes

The issue requests hardening shared MEDIA directive parsing so a local media path followed by serialized tool-result JSON is trimmed or rejected before channel delivery attempts local file I/O.

## Best Possible Solution

Land or revise the linked shared-parser fix with focused regression coverage in `src/media/parse.test.ts`, then let GitHub close this issue after the PR merges.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a one-line `MEDIA:/path/to/image.png"}],"details":...` token is captured as one payload, not split without whitespace, accepted as a local path, and later read as that exact path. I did not run tests because this sweep was read-only.

## Solution Assessment

Yes. A shared parser-level fix with parser regression coverage is the narrowest maintainable solution; channel adapters can add defense-in-depth, but they should not be the primary fix.
