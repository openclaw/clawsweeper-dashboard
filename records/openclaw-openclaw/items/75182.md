## Summary

Keep open. Current main still has the shared MEDIA parser path that can treat a slash-prefixed local path plus trailing serialized JSON as one valid media source, and an open fix PR (#75202) is already linked to close this issue after merge.

## What This Changes

The issue asks OpenClaw to harden shared MEDIA directive parsing so local media paths do not absorb trailing serialized tool-result JSON before channel delivery.

## Best Possible Solution

Land or revise the linked shared-parser fix so MEDIA local paths stop at a valid media/document boundary before serialized JSON, add focused parser regression coverage, and let the linked PR close this issue after merge.

## Reproduction Assessment

Yes. The current source path is enough to reproduce with high confidence: a one-line local MEDIA directive with no whitespace before the JSON suffix is captured as one payload, accepted as a local path, and later read as that exact path. I did not run tests or a live channel because this review was read-only.

## Solution Assessment

Yes. A shared parser-level fix with regression coverage is the narrowest maintainable solution; #75202 is the right implementation lane to review or supersede before closing this issue.
