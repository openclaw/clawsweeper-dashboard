## Summary

Keep open. Current main and v2026.6.10 still preserve the documented no-override boundary, but same-name workspace hooks can still be dropped in favor of eventless managed hooks while status/check readiness does not flag that no handler will register.

## What This Changes

The issue asks OpenClaw to surface or prevent workspace hook collisions where a managed hook wins but has no metadata events, silently disabling the user's workspace hook logic.

## Best Possible Solution

Keep the no-override safety boundary, but add a maintainer-approved diagnostic path that flags same-name collisions and selected hooks with no metadata events before users rely on them.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: configure a managed hook and an explicitly enabled workspace hook with the same name, with the selected managed hook carrying no metadata events. The policy keeps the managed entry, the loader skips registration for zero events, and hooks status/check can still derive readiness from enablement and requirements.

## Solution Assessment

Unclear until maintainers choose the user-facing diagnostic contract. Allowing workspace shadowing would contradict current docs, so the safer direction appears to be hook status/check, doctor, dashboard, or loader-failure surfacing rather than changing precedence by default.
