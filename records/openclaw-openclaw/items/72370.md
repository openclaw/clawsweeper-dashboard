## Summary

Keep open. Current main still preserves managed-hook precedence, still skips eventless winners at runtime, and the hook status/CLI path still treats an enabled eventless hook as loadable without surfacing the ignored workspace collision, so the prior cleanup review remains valid.

## What This Changes

The issue reports same-name workspace hooks being rejected behind managed hooks, with eventless managed hooks then winning resolution and making the user’s workspace hook behavior silently ineffective.

## Best Possible Solution

Preserve the current managed-hook trust boundary, but make this state actionable: record ignored same-name workspace collisions, mark enabled hooks with no event metadata as not loadable, expose clear reasons in `openclaw hooks list`, `info`, and `check` text/JSON output, and document rename/remove guidance for workspace hooks that collide with managed names.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to define a managed hook and workspace hook with the same name, leave the managed hook with no metadata events, and observe that current resolution keeps the managed entry while runtime loading skips it and status still considers it loadable.

## Solution Assessment

Yes. A diagnostics-focused repair is the narrowest maintainable solution; changing precedence so workspace code can override managed hooks would be a separate trust-policy decision.
