## Summary

Keep open. Current main still preserves the managed-hook precedence that blocks same-name workspace hooks, and the remaining bug is that this failure is not actionable in status/CLI output when the winning managed hook has empty event metadata.

## What This Changes

The issue reports workspace hooks with the same names as OpenClaw-managed hooks being rejected, while empty managed hooks win resolution and leave the workspace hook behavior inactive.

## Best Possible Solution

Preserve the current trust boundary, but make the failure actionable: track ignored workspace-hook collisions, mark enabled hooks with empty event metadata as not loadable, expose clear text and JSON reasons in `openclaw hooks list/check/info`, and document rename/remove guidance for same-name workspace hooks.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to define a managed hook and a workspace hook with the same name, leave the managed hook with no metadata events, and observe that policy keeps the managed entry, the loader skips it, and status readiness ignores the empty event list.

## Solution Assessment

Yes. A diagnostics-focused repair is the narrowest maintainable solution; allowing workspace hooks to override managed hooks would be a broader trust-policy change and is not needed to make the reported failure visible and fixable.
