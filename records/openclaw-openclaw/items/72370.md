## Summary

Keep open. Current main and v2026.6.10 still preserve the documented no-override boundary, but same-name workspace hooks can still be dropped in favor of eventless managed hooks with only startup log warnings and no durable hooks check/status diagnostic.

## What This Changes

The issue asks OpenClaw to make managed/workspace hook-name collisions and selected hooks with no metadata events visible or fail loudly instead of silently disabling workspace hook logic.

## Best Possible Solution

Keep the no-override safety boundary, but add a maintainer-approved diagnostic path that flags same-name collisions and enabled selected hooks with no metadata events before users rely on them.

## Reproduction Assessment

Yes. From source inspection, create a managed hook and an explicitly enabled workspace hook with the same name; policy keeps the managed entry, and if that kept hook has no metadata events the loader warns and registers no handler while hooks status/check still derive readiness from enablement and requirements.

## Solution Assessment

Unclear until maintainers choose the user-facing diagnostic contract. Allowing workspace shadowing would contradict current docs, so the safer direction appears to be hook status/check, doctor, dashboard, or loader-failure surfacing rather than changing precedence by default.
