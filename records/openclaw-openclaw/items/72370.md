## Summary

Keep open. Current main still enforces managed-hook precedence, only logs same-name workspace collisions during discovery, and can report an enabled managed hook with empty metadata events as loadable even though the loader registers no handler.

## What This Changes

The issue reports that workspace hooks sharing names with OpenClaw-managed hooks are rejected at gateway startup and effectively replaced by managed hooks with no registered events.

## Best Possible Solution

Keep managed-hook precedence, but carry ignored-collision and empty-events diagnostics into hook status/check output, CLI JSON/text, docs, and focused tests so users can identify the winning hook and rename or remove the workspace hook.

## Reproduction Assessment

Yes. A source-level repro is a managed hook and a workspace hook with the same name where the managed hook is enabled but has no metadata events; current main keeps the managed entry, skips handler registration, and still has status data that can mark it loadable.

## Solution Assessment

Yes. A diagnostics-focused fix is the narrowest maintainable path because it addresses the silent-disable failure without weakening the documented managed-hook trust boundary.
