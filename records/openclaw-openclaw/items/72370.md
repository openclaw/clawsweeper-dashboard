## Summary

Keep open. Current main still preserves the documented no-override rule for workspace hooks, but an eventless managed hook can still win the same-name collision, register no handlers, and be reported as loadable because status only checks config and requirements.

## What This Changes

The issue reports that same-name workspace hooks are rejected behind managed hooks and can be replaced by managed hook entries that register no events, leaving workspace hook logic disabled without actionable status output.

## Best Possible Solution

Preserve managed-hook precedence, but mark eventless winners non-loadable and surface same-name collision/blocking reasons in hook status, CLI text, and JSON output.

## Reproduction Assessment

Yes, at source level. A managed and workspace hook with the same name resolves to the managed hook, and if that winner has no metadata events, the loader registers nothing while status can still mark it loadable.

## Solution Assessment

Yes. A diagnostics-focused fix is the narrowest maintainable path because the no-override precedence is documented; changing precedence would be a broader behavior decision.
