## Summary

Keep open. Current main and v2026.6.8 still prevent workspace hooks from overriding managed hooks, and the unresolved part is the diagnostic/status gap when the surviving managed hook has no events; that needs maintainer product choice rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to stop silently disabling explicitly configured workspace hooks that collide with managed hook names, either by surfacing or failing the collision or by flagging eventless winners in hook status.

## Best Possible Solution

Keep the documented no-override safety boundary, but add a maintainer-approved diagnostic surface that makes same-name collisions and enabled eventless hooks visible before users rely on them.

## Reproduction Assessment

Yes, from source inspection: create a managed hook and an explicitly enabled workspace hook with the same name, and policy keeps the managed entry; if that kept hook has no metadata events, the loader warns and registers no handler while status remains based on enablement and requirements. I did not run a live gateway in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract details. Allowing workspace shadowing would contradict the documented no-override rule, so the narrowest maintainable direction appears to be diagnostics/status surfacing rather than changing precedence.
