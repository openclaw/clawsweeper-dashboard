## Summary

Keep open. Current main and the latest release still preserve managed-hook-over-workspace collision behavior and still treat an eventless winning hook as only a loader warning, so the remaining diagnostic/product decision is not obsolete.

## What This Changes

The issue asks OpenClaw to make workspace hook name collisions and eventless managed-hook winners actionable instead of leaving user hook code disabled with only startup log warnings.

## Best Possible Solution

Keep the documented no-override safety boundary, but add a maintainer-approved diagnostic surface for same-name collisions and eventless enabled hooks if that is the desired product behavior.

## Reproduction Assessment

Yes for the current-main source behavior: create a managed hook and an explicitly enabled workspace hook with the same name, and policy keeps the managed entry; if the kept hook has no metadata events, the loader only warns and registers no handler. A live gateway repro was not run in this read-only review.

## Solution Assessment

Unclear until maintainers choose the contract details. The current docs already reject workspace overrides, so the narrowest maintainable direction appears to be better diagnostics/status surfacing rather than allowing arbitrary workspace shadowing.
