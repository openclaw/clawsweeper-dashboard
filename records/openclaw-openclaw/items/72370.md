## Summary

Keep open. Current main still preserves the documented no-override policy for workspace hooks, but the actionable bug remains: a managed hook with no metadata events can win a same-name collision, register no handlers, and still be reported as ready by hook status/check surfaces.

## What This Changes

The issue asks OpenClaw to stop silently replacing same-named workspace hooks with eventless managed hook winners, or at least surface that collision as an actionable hook readiness problem.

## Best Possible Solution

Preserve managed-hook precedence, but carry enough collision and eventless-hook state into hook list/info/check/status so an enabled no-op winner is not reported ready and users get a clear rename or configuration action.

## Reproduction Assessment

Yes, by source inspection. A same-name workspace/managed collision resolves to the managed hook and drops the workspace entry, an eventless winner registers no handlers, and status still marks enabled hooks loadable when requirements pass.

## Solution Assessment

Yes. A diagnostics-focused fix is the narrowest maintainable solution because the no-override precedence is documented and tested; changing override policy would be a separate product decision.
