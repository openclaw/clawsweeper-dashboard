## Summary

Keep open. Current main still preserves the intended managed-over-workspace hook precedence, but the reported bad outcome remains actionable: an enabled managed hook with empty event metadata can win a same-name workspace collision, get skipped by the loader, and still look ready/loadable in hook status surfaces.

## What This Changes

The issue asks OpenClaw to make same-name workspace hook collisions with managed hooks actionable when the kept managed hook has no events, rather than silently leaving the user’s workspace hook disabled and the managed hook ineffective.

## Best Possible Solution

Preserve managed-over-workspace precedence, but make the failed state visible: track ignored hook collisions, mark enabled hooks with empty event metadata as not ready, expose clear reasons in hooks list/info/check JSON and text output, and document rename/remove guidance for same-name workspace hooks.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to define a managed hook and workspace hook with the same name, leave the managed hook with no metadata events, and observe that current main keeps the managed entry, skips it at runtime, and still computes status loadability without considering the empty event list.

## Solution Assessment

Yes. A diagnostics-focused repair is the narrowest maintainable solution; changing workspace hooks to override managed hooks would alter the trust policy and is not required to make the reported failure actionable.
