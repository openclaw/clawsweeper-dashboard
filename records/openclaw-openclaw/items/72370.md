## Summary

Keep open. Current main and v2026.6.8 still keep managed hooks ahead of same-name workspace hooks, and the remaining user problem is that collisions plus eventless managed hooks are only visible in logs rather than a durable status/diagnostic surface.

## What This Changes

The issue asks OpenClaw to stop silently no-oping same-name workspace hooks by surfacing or failing hook collisions and enabled eventless hooks, or by clarifying the override contract.

## Best Possible Solution

Keep the documented no-override boundary, but add a maintainer-approved diagnostic path that makes same-name hook collisions and enabled eventless hooks visible before users rely on them.

## Reproduction Assessment

Yes. From source inspection, create a managed hook and an explicitly enabled workspace hook with the same name; policy keeps the managed entry, and if that kept hook has no metadata events the loader warns and registers no handler while status remains based on enablement and requirements.

## Solution Assessment

Unclear until maintainers choose the diagnostic contract. Allowing workspace shadowing would contradict the documented no-override rule, so the narrow maintainable direction appears to be status/diagnostic surfacing rather than changing precedence by default.
