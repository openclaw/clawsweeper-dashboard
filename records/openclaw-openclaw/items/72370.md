## Summary

Keep open. Current main still preserves managed-hook precedence, which is documented and security-relevant, but it does not make the ignored workspace collision or an eventless winning hook actionable through hook status/check output, so the diagnostics bug remains valid and narrow.

## What This Changes

The issue reports that workspace hooks sharing names with managed hooks are ignored, while eventless managed winners can leave the user's custom hook logic inactive with only startup log warnings.

## Best Possible Solution

Keep the managed-over-workspace trust boundary, but carry collision and metadata state into hook diagnostics: show the ignored workspace hook name/source/path, the winning hook source/path, mark enabled hooks with no events as not loadable, and give clear CLI/docs guidance to rename or remove the colliding workspace hook.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a managed hook and workspace hook with the same name where the managed hook is enabled but has no metadata events; current main keeps the managed entry, skips runtime registration, and does not expose the collision or empty-events blocker in status/check output.

## Solution Assessment

Yes. A diagnostics-focused repair is the narrowest maintainable solution; allowing workspace hooks to override managed hooks would change the documented trust policy and should remain a separate maintainer product/security decision.
