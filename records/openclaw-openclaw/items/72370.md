## Summary

Keep open. Current main still intentionally preserves managed-hook precedence, but it does not retain ignored workspace-collision details for diagnostics and still treats an enabled hook with empty metadata events as loadable, so the reported status/check observability bug remains valid.

## What This Changes

The issue reports that workspace hooks sharing names with managed hooks are rejected at Gateway startup, leaving same-named managed hooks to win even when they have no metadata events and no actionable status/check diagnostic.

## Best Possible Solution

Keep the existing trust policy, but carry collision and metadata state into hook diagnostics: show the ignored workspace hook name/source/path, the winning hook source/path, mark enabled hooks with no metadata events as not loadable, and document the rename/remove path for colliding workspace hooks.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a managed hook and workspace hook with the same name where the managed hook is enabled but has no metadata events; current main keeps the managed entry, skips handler registration, and exposes no collision or empty-events blocker in hook status/check data.

## Solution Assessment

Yes. A diagnostics-focused repair is the narrowest maintainable solution because it addresses the silent failure without weakening the documented managed-hook precedence policy.
