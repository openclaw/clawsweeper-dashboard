## Summary

Keep open. Current main still preserves managed-hook precedence, logs same-name workspace collisions only during discovery, and treats enabled eventless winners as loadable in status/CLI, so the silent-disable diagnostics problem remains valid.

## What This Changes

Report requests actionable diagnostics when workspace hooks collide with managed hook names and the selected managed hook has no metadata events, instead of a warning-only no-op.

## Best Possible Solution

Keep the documented managed-hook precedence policy, but carry ignored-collision and empty-events state into hook status/check/list JSON and text output with focused docs and regression coverage.

## Reproduction Assessment

Yes, source-level. A managed hook and workspace hook with the same name, where the managed hook is enabled but has no metadata events, follows a clear current-main path: the workspace hook is ignored, the managed hook registers no handler, and status still marks it loadable.

## Solution Assessment

Yes. A diagnostics-focused repair is the narrowest maintainable fix because it preserves the documented precedence rule while making the no-op state visible to users and automation.
