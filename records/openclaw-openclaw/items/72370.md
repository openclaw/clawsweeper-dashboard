## Summary

Keep open. Current `main` still enforces the documented no-override policy, but an enabled managed hook with no metadata events is skipped by the loader while hook status/check can still mark it ready, so the diagnostics bug remains actionable.

## What This Changes

The issue requests clearer handling when same-name workspace hooks are rejected in favor of eventless managed hooks that effectively disable the workspace logic.

## Best Possible Solution

Keep managed-hook precedence intact, but carry collision and eventless-readiness state into hook status/check/list/info so an enabled no-op winner is reported as not ready with an actionable rename or metadata fix.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an enabled managed hook and workspace hook with the same name where the managed hook has no metadata events; policy keeps the managed entry, loader registers no handler, and status still marks it loadable.

## Solution Assessment

Yes. A diagnostics-focused repair is the narrowest maintainable solution because docs already define the no-override policy, while changing hook precedence would be a broader policy change.
