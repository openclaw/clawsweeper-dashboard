## Summary

Keep open. Current main still keeps managed-hook precedence, logs same-name workspace collisions only at load time, and treats enabled hooks with empty metadata events as loadable in status/CLI diagnostics, so the silent-disable problem remains valid.

## What This Changes

The issue reports workspace hooks being ignored when they share names with OpenClaw-managed hooks, with eventless managed hooks winning and preventing custom workspace logic from running.

## Best Possible Solution

Keep managed-hook precedence, but carry ignored-collision and empty-events state into hook status/check output, CLI JSON/text, docs, and focused tests so users can see the winning hook and the rename or remove path.

## Reproduction Assessment

Yes. A source-level repro is a managed hook and workspace hook with the same name where the managed hook is enabled but has no metadata events; current main keeps the managed entry, skips handler registration, and status still marks it loadable.

## Solution Assessment

Yes. A diagnostics-focused fix is the narrowest maintainable solution because it addresses the silent failure without weakening the managed-hook precedence policy.
