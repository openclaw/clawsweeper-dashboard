## Summary

Keep open. The prior ClawSweeper review correctly identified a remaining diagnostics bug, and current `main` still has the same behavior: same-name workspace hooks lose to managed hooks, eventless managed hooks are skipped by the loader, and hook status/check output can still mark the winning eventless hook as loadable without surfacing the ignored workspace collision.

## What This Changes

The issue reports that workspace hooks with names matching managed hooks are ignored, causing managed hook entries with no event metadata to win and behave as no-ops without clear status diagnostics.

## Best Possible Solution

Preserve the current trust boundary, but make this failure actionable: track ignored workspace hook collisions, mark enabled hooks with empty event metadata as not loadable, expose clear reasons in `openclaw hooks list/check/info` text and JSON output, and document rename/remove guidance for same-name workspace hooks.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to define a managed hook and a workspace hook with the same name, leave the managed hook with no metadata events, and observe that current policy keeps the managed entry, the loader skips it, and status readiness ignores the empty event list.

## Solution Assessment

Yes. A diagnostics-focused repair is the narrowest maintainable solution; allowing workspace hooks to override managed hooks would be a broader trust-policy change and is not needed to make the reported failure visible and fixable.
