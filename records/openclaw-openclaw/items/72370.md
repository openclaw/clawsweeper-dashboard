## Summary

Keep open. Current main still preserves the documented no-override policy for workspace hooks, but an enabled managed hook with empty metadata events can win the collision, register no handlers, and still appear ready in hook status/check surfaces.

## What This Changes

The issue asks OpenClaw to make same-name workspace hook rejections and eventless managed hook replacements visible and actionable instead of silently leaving custom workspace logic disabled.

## Best Possible Solution

Preserve managed-hook precedence, but carry collision and eventless-hook state into hook status/check/list/info so an enabled no-op winner is reported as not ready with an actionable reason.

## Reproduction Assessment

Yes, by source inspection. A managed hook and workspace hook with the same name resolves to the managed hook, and if that managed hook has no metadata events, the loader registers nothing while status still treats it as loadable.

## Solution Assessment

Yes. A diagnostics-focused fix is the narrowest maintainable path because the no-override precedence is documented and already covered by tests, while the readiness/status contract is where the observable bug remains.
