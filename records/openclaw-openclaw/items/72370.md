## Summary

Keep open. Current main still rejects same-name workspace hooks by policy, only logs the ignored workspace collision, skips eventless managed hooks at runtime, and reports readiness from enablement plus requirements without treating empty event metadata or ignored collisions as actionable status failures.

## What This Changes

The issue asks OpenClaw to make same-name workspace hook rejection and eventless managed hook replacement visible and actionable instead of silently disabling workspace hook logic.

## Best Possible Solution

Preserve the managed-hook trust boundary, but carry ignored collision data into hook status and CLI JSON/text output, mark enabled hooks with no events as not loadable with a clear blocked reason, and document rename/remove guidance for workspace hooks that collide with managed names.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to define a managed hook and a workspace hook with the same name, leave the managed hook with empty metadata events, and observe that resolution keeps the managed entry, the loader skips it, and status readiness does not flag the empty events.

## Solution Assessment

Yes. A diagnostics-focused repair is the narrowest maintainable solution; allowing workspace hooks to override managed hooks would be a broader trust-policy change and is not needed to make the reported failure visible and fixable.
