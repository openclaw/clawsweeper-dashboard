## Summary

Keep open. Current main still preserves the managed-hook-over-workspace collision behavior and still treats eventless hooks as a loader warning rather than an actionable status failure, so the remaining diagnostic/product decision is not obsolete.

## What This Changes

The issue reports that workspace hooks whose names collide with managed hooks are rejected at gateway startup, leaving custom hook logic inactive without status-level diagnostics.

## Best Possible Solution

Keep the documented no-override safety boundary, but make a configured same-name workspace collision and any eventless winning hook actionable in hook status/check output with focused regression tests.

## Reproduction Assessment

Yes, source-reproducible: create a managed hook and an explicitly enabled workspace hook with the same name, then the policy keeps the managed entry and logs the cannot-override warning. If the kept managed hook has no metadata events, the loader warns and registers no handler while status still uses loadable eligibility rather than registered events.

## Solution Assessment

Unclear until maintainers choose the contract details. The current docs already reject workspace overrides, so the narrowest maintainable direction appears to be better diagnostics/status surfacing rather than allowing arbitrary workspace shadowing.
