## Summary

Keep open. Current main still preserves the no-override policy for same-name workspace hooks, but the remaining diagnostics gap is real: the policy/loader can drop the workspace hook, let an eventless managed hook win, and the status/check path can still treat that hook as loadable instead of surfacing an actionable no-op condition.

## What This Changes

The issue asks OpenClaw to stop silently disabling same-name workspace hooks when an eventless managed hook wins the collision, or at least expose the condition through status/check diagnostics.

## Best Possible Solution

Keep managed-hook precedence, but carry blocked same-name workspace sources and eventless winner state into hooks list/info/check and status so users see an actionable disabled/no-handler reason.

## Reproduction Assessment

Yes, by source inspection. A same-name workspace hook is rejected by policy, an eventless managed hook can then register no handlers, and current status readiness does not treat that empty-events winner as a problem.

## Solution Assessment

Yes, a diagnostics-focused fix is the best path. Allowing workspace hooks to override managed hooks would contradict the documented policy, while surfacing the blocked/no-op condition preserves the contract and fixes the misleading status behavior.
