## Summary

Keep open. Current main still documents and enforces managed-hook precedence, but ignored same-name workspace collisions are warning-only and enabled hooks with no metadata events still appear loadable in the status data that backs `openclaw hooks list/check`, so the diagnostics bug remains valid.

## What This Changes

The issue reports workspace hooks with names matching managed hooks being ignored at startup and then effectively replaced by managed hooks with no metadata events.

## Best Possible Solution

Keep the documented no-override policy, but make ignored workspace collisions and enabled hooks with empty metadata events visible as not-ready/actionable in hook status, hook check/list/info JSON and text, docs, and regression tests.

## Reproduction Assessment

Yes, source-level: define a managed hook and a workspace hook with the same name, with the managed hook enabled but no metadata events. Current main keeps the managed entry, skips registration because the events list is empty, and still computes it as loadable for status/check output.

## Solution Assessment

Yes. A diagnostics-focused repair is the narrowest maintainable fix because it preserves the documented managed-hook precedence while making the effective no-op state visible and actionable.
