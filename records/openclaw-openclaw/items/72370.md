## Summary

Keep open. Current main still preserves the documented no-override policy for workspace hooks, but an eventless managed hook can still win a same-name collision, register no handlers, and appear ready because hook status does not treat an empty event list as non-loadable.

## What This Changes

The issue asks OpenClaw to make same-name workspace-hook rejection and eventless managed-hook winners actionable instead of silently disabling workspace hook logic.

## Best Possible Solution

Preserve managed-hook precedence, but carry collision and eventless-hook state through loader/status/CLI output so hooks with no event registrations are not reported ready and users see an actionable blocked reason.

## Reproduction Assessment

Yes, at source level. A managed and workspace hook with the same name resolves to the managed hook, and if that winner has no metadata events, the loader registers nothing while status can still mark it loadable.

## Solution Assessment

Yes. A diagnostics-focused fix is the narrowest maintainable path because the no-override precedence is documented; changing precedence would be a broader product behavior change.
