## Summary

Keep this open: current main still enforces the managed-over-workspace same-name policy and still has no operator-visible diagnostic for the ignored workspace entry beyond the startup warning, so the central report is not implemented on main.

## What This Changes

The issue requests that colliding workspace hooks either load safely or produce an actionable status/dashboard diagnostic instead of being silently replaced by same-name managed hooks.

## Best Possible Solution

Keep the documented non-override rule unless maintainers intentionally change it, but surface ignored workspace collisions and eventless winning hooks through an operator-visible diagnostic such as `openclaw hooks check`.

## Reproduction Assessment

Yes from source inspection: a same-name workspace entry is filtered out before status reporting, and an eventless winning managed hook only emits a loader warning. I did not run a gateway repro because this review was required to keep the checkout read-only.

## Solution Assessment

No final fix shape is confirmed. A diagnostic-only fix looks safest, while allowing eventless managed hooks to be shadowed would change the documented trust and precedence policy.
