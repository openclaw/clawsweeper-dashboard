## Summary

Keep open. The current main implementation still blocks workspace hooks from shadowing OpenClaw-managed hook code, and the same-name collision path can still leave the workspace's hook logic inactive while status only reports the managed hook config as loadable; the issue remains a valid narrow repair candidate rather than a stale cleanup close.

## What This Changes

The issue reports that workspace-defined hooks with the same names as managed hooks are rejected at gateway startup, causing custom workspace hook logic to be replaced by eventless managed hooks without actionable status diagnostics.

## Best Possible Solution

Preserve the managed-code safety boundary, but make same-name workspace collisions with missing managed events actionable through hook loading/status diagnostics and focused regression tests.

## Reproduction Assessment

Yes from source, but not by a live local run in this read-only review. A workspace hook with the same package id as an openclaw-managed hook still takes the rejection branch in src/hooks/workspace-hooks.ts, while the managed hook loader can still accept a module with no metadata events.

## Solution Assessment

Unclear until maintainers choose the policy outcome. The safest repair is not to silently allow arbitrary workspace overrides, but to make the collision/no-event state visible and actionable in loader/status diagnostics with tests.
