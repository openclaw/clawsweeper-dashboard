## Summary

Keep open. Current main still enforces managed-hook precedence and logs same-name workspace collisions only at startup, while enabled managed hooks with empty event metadata are still reported as loadable by status/CLI diagnostics.

## What This Changes

The issue asks for same-name workspace hook collisions with eventless managed hooks to be surfaced or fixed instead of silently disabling the workspace hook logic.

## Best Possible Solution

Keep managed-hook precedence, but carry ignored-collision and empty-event state into hook status/check/list/info output with focused regression tests, docs, and changelog coverage.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: create enabled managed and workspace hooks with the same name, leave the managed hook without metadata events, and current main keeps the managed entry, skips handler registration, and still reports it as loadable.

## Solution Assessment

Yes. A diagnostics-focused repair is the narrowest maintainable solution because it fixes the silent-disable behavior without weakening the documented managed-hook precedence policy.

## Repair Work Prompt

Fix the hook diagnostics bug reported in https://github.com/openclaw/openclaw/issues/72370. Preserve the documented rule that workspace hooks cannot override managed, bundled, or plugin hooks. Add a narrow status/diagnostic path so ignored same-name workspace collisions are retained and surfaced by `openclaw hooks list`, `openclaw hooks info`, and `openclaw hooks check` JSON/text output. Treat enabled hooks with no `metadata.openclaw.events` as not ready or otherwise actionable in status/check output, matching the loader behavior that skips handler registration. Add focused regression tests around managed-vs-workspace collision plus an eventless managed winner, update docs for the diagnostic behavior, and add a user-facing changelog entry. Do not add a config option and do not allow workspace shadowing of managed code.
