## Summary

Keep open. Current main still preserves managed-hook precedence but does not make the losing workspace collision or the winning hook's empty event metadata actionable in hook status/check output, so the reported diagnostics gap remains valid.

## What This Changes

The issue reports that workspace hooks with names matching OpenClaw-managed hooks are ignored at gateway startup, while eventless managed hooks win resolution and leave the user without visible hook-status diagnostics.

## Best Possible Solution

Preserve the managed-over-workspace trust boundary, but retain enough collision and metadata state to show the user what happened: ignored workspace hook name, winning source/path, eventless winner marked not loadable, clear CLI JSON/text reasons, and docs guidance to rename or remove the colliding workspace hook.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a managed hook and workspace hook with the same name, with the managed hook enabled but no metadata events: current policy keeps the managed entry, the loader skips it, and status/check does not flag empty events as a blocker.

## Solution Assessment

Yes. A diagnostics-focused repair is the narrowest maintainable fix; allowing workspace hooks to override managed hooks would solve the reporter's specific case by weakening the current trust policy, which is a broader product/security decision.
