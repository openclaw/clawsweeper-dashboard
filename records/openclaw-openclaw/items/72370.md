## Summary

Keep open: current main and the latest release still preserve the reported managed-over-workspace hook collision behavior, while status/check output does not appear to surface the eventless kept hook as not loadable. The user’s rename workaround is valid, but the operator-facing diagnostic gap remains unresolved.

## What This Changes

The issue reports that workspace hook packages whose ids collide with OpenClaw-managed hooks are ignored at gateway startup, leaving users with eventless managed hooks and no status/dashboard warning.

## Best Possible Solution

Keep managed-hook precedence unless maintainers explicitly change that contract, but make collisions and eventless enabled hooks visible through hook status/check output or another operator-facing diagnostic.

## Reproduction Assessment

Yes at source level: current main still resolves colliding managed/workspace hook names to the managed entry, logs the cannot-override warning, and skips hooks with empty metadata events. I did not run a live gateway repro because this review is read-only.

## Solution Assessment

Unclear as an implementation choice: changing override precedence would be compatibility-sensitive, while adding operator-facing diagnostics for ignored collisions or eventless enabled hooks is the safer maintainable direction.
