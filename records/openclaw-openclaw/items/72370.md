## Summary

Keep open. Current main still preserves managed-hook precedence over workspace hooks, but the ignored workspace collision is only logged and the winning hook can still appear loadable even when it has no event metadata, so the reported diagnostics gap remains valid.

## What This Changes

The issue requests actionable hook status/check diagnostics when workspace hooks collide with managed hooks and the winning managed hook has no events, instead of leaving the workspace hook inactive with only startup log warnings.

## Best Possible Solution

Preserve the managed-over-workspace trust boundary, but carry collision metadata into hook status so list/info/check text and JSON show the ignored workspace hook, winning source/path, and a clear rename/remove action. Also mark enabled hooks with no event metadata as not loadable with a blocked reason, and document the behavior.

## Reproduction Assessment

Yes. A code-level reproduction is a managed hook and a workspace hook with the same name where the managed hook has no metadata events: current policy keeps the managed entry, the loader skips registration, and status/check has no empty-events or ignored-collision blocker.

## Solution Assessment

Yes. A diagnostics-focused repair is the narrowest maintainable solution; allowing workspace hooks to override managed hooks would solve the reporter's instance by weakening a documented precedence and trust boundary.
