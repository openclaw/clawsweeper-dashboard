## What This Changes

The issue asks for QMD-backed memory search to avoid silently falling back to FTS-only on the first `memory_search` after a gateway restart when mcporter startup output contaminates JSON parsing.

## Best Possible Solution

Keep this issue open until a live qmd/mcporter first-search repro or dependency-level stdout proof identifies the real boundary, then fix stdout purity or parsing with focused qmd-manager regression coverage.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows malformed mcporter call stdout would still throw and trigger fallback, but daemon-start contamination is unproven because daemon start and call use separate subprocess captures.

## Solution Assessment

No, the proposed literal `[mcporter]` line filter is not the best fix. It targets the later call stdout without proving that boundary and misses unprefixed daemon output; the better path is live proof followed by a stdout-purity or structured-parser fix.
