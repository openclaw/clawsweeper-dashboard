## Summary

Keep open. Current main still raw-parses the full mcporter call stdout and the start-daemon tests only cover clean output, so the reported first-query fallback remains a valid, narrow memory-core hardening bug rather than cleanup-close material.

## What This Changes

The issue requests hardening QMD mcporter search so the first `memory_search` after a gateway restart does not fall back to FTS because noisy mcporter stdout breaks JSON parsing.

## Best Possible Solution

Suppress or discard daemon-start stdout where possible, robustly extract the mcporter call JSON payload before mapping results, and add a first-search-after-restart noisy-stdout regression test.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction: configure `memory.qmd.mcporter.startDaemon=true`, mock the first mcporter call stdout to include a `[mcporter]` or daemon status line before valid JSON, and the current `JSON.parse(result.stdout)` path fails into builtin fallback. I did not perform a live qmd/mcporter gateway run in this read-only sweep.

## Solution Assessment

No, the proposed prefix-only filter is not the best durable fix. The narrower maintainable path is to discard daemon-start stdout and parse the mcporter call output with balanced JSON object/array extraction before applying the existing `structuredContent` and `results` mapping.
