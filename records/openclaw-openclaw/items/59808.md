## Label Justifications

- `P1`: The issue describes a real QMD memory-search workflow silently degrading immediately after gateway restart for mcporter-backed users.
- `impact:session-state`: The affected surface is memory retrieval and context recall, where semantic QMD results can be bypassed and reported as fallback.

## Summary

Keep open. Current main and the latest stable release still have the source-level failure path: QMD mcporter daemon startup runs through the stdout-capturing wrapper, the mcporter call output is raw-parsed as JSON, and the related fix path remains open in https://github.com/openclaw/openclaw/pull/79745.

## What This Changes

Fix QMD memory search so the first mcporter-backed query after a gateway restart does not degrade to FTS-only because daemon-start stdout contaminates JSON parsing.

## Best Possible Solution

Review and land or split the QMD mcporter daemon/stdout fix from https://github.com/openclaw/openclaw/pull/79745 with focused first-call regression coverage, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, at source level. With mcporter-backed QMD and `startDaemon=true`, the first search awaits `mcporter daemon start`, current main still raw-parses mcporter call stdout, and mcporter v0.8.1 emits daemon-start status text on stdout.

## Solution Assessment

No, not the issue body's prefix-only filter by itself. The maintainable fix should discard or isolate daemon-start stdout and parse only the structured mcporter call response, with first-call regression coverage.
