## Summary

Keep open. Current main still parses the full mcporter JSON stdout directly and the mcporter wrapper still cannot discard daemon-start stdout, so the reported first-query fallback remains plausible and narrowly actionable. The earlier ClawSweeper comment remains accurate after checking current main, and there is no open PR with closing syntax for this issue.

## What This Changes

Fix the mcporter-backed QMD memory search path so the first search after a gateway restart does not fall back to builtin/FTS when mcporter startup log lines appear on stdout.

## Best Possible Solution

Keep this open until memory-core handles mcporter startup noise locally: discard daemon-start stdout via the existing runner option, defensively parse the structured JSON payload from mcporter call output, and lock the first-search-after-restart path with a focused regression test.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is available by enabling QMD mcporter with startDaemon=true, mocking the first mcporter search output to include a [mcporter] startup/log line before a valid JSON result, and observing the current JSON.parse(result.stdout) path throw before the fallback manager switches to builtin search.

## Solution Assessment

Yes. The narrow repository-side fix is to harden the mcporter wrapper and parser in memory-core; upstream mcporter should keep logs on stderr, but OpenClaw still needs to tolerate supported mcporter versions that write startup noise to stdout.
