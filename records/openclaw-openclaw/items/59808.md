## Label Justifications

- `P1`: The issue describes a real QMD memory-search workflow silently degrading immediately after gateway restart for mcporter-backed users.
- `impact:session-state`: The affected surface is memory retrieval/context state, where semantic QMD recall can be bypassed and reported as fallback.

## Summary

Keep open. Current main and the latest stable release still have the source-level failure path: QMD mcporter daemon startup is run through the stdout-capturing wrapper and the subsequent mcporter call output is parsed as raw JSON; the related fix PR remains open.

## What This Changes

The issue reports that the first QMD-backed memory search after a gateway restart falls back to FTS because mcporter daemon startup text can pollute stdout before JSON parsing.

## Best Possible Solution

Review and land or split the QMD mcporter daemon/stdout fix from https://github.com/openclaw/openclaw/pull/79745 with focused first-call regression coverage, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, at source level. With mcporter-backed QMD and `startDaemon=true`, the first search awaits `mcporter daemon start`, then raw-parses mcporter call stdout; mcporter v0.8.1 emits daemon-start status text on stdout, which matches the reported JSON parse failure.

## Solution Assessment

No, not the issue body's prefix-only filter by itself. The maintainable fix should discard or isolate daemon-start stdout and parse only the structured mcporter call response, with a first-call regression test; the linked QMD mcporter PR appears to be the current implementation path once its review feedback is resolved.
