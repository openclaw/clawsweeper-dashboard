## Summary

Keep open. Current main still starts mcporter without discarding stdout and then raw-parses the mcporter call stdout, while the existing daemon-start coverage only exercises clean output; this remains a narrow, source-backed memory-core bug with no duplicate fix PR found.

## What This Changes

The issue reports that the first qmd-backed memory search after a gateway restart can parse noisy mcporter stdout as JSON and fall back to FTS-only mode.

## Best Possible Solution

Discard daemon-start stdout where possible, robustly extract or validate the mcporter call JSON payload before mapping results, and add a first-search-after-restart noisy-stdout regression test.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction: enable memory.qmd.mcporter.startDaemon, make the first mcporter stdout contain a daemon/status line before valid JSON, and the current JSON.parse(result.stdout) path throws before falling back. I did not run a live qmd/mcporter gateway in this read-only sweep.

## Solution Assessment

No, the issue's prefix-only filter is not the best durable fix. The maintainable fix is to suppress daemon-start stdout and parse/extract the mcporter call JSON with the same tolerance expected for noisy CLI output.
