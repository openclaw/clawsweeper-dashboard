## Summary

Keep open. Current main still starts mcporter through the same stdout-capturing runner and still raw-parses mcporter call stdout, so the first post-restart QMD search can still fall back when mcporter emits startup/status text; a related open PR appears to address the parser path but has not merged.

## What This Changes

Request to keep mcporter-backed QMD vector search from falling back to FTS on the first `memory_search` after a gateway restart when mcporter startup/status text appears on stdout.

## Best Possible Solution

Land or split the QMD mcporter parser/daemon-stdout fix from https://github.com/openclaw/openclaw/pull/79745 with focused regression coverage, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, at source level: with mcporter-backed QMD and `startDaemon=true`, the first search awaits daemon start and then raw-parses mcporter call stdout, so a leading daemon/status line produces the reported fallback. I did not run a live qmd/mcporter gateway in this read-only review.

## Solution Assessment

No for the issue's prefix-only filter; the better fix is to suppress or discard daemon-start stdout where only the exit code matters and parse the mcporter call's structured JSON defensively, with a first-call regression test.
