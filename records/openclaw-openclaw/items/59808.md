## Label Justifications

- `P1`: A mcporter-backed QMD memory search can silently degrade immediately after gateway restart, affecting an active agent memory workflow.
- `impact:session-state`: The affected surface is memory retrieval/context recall, where semantic QMD results can be bypassed and replaced by fallback search state.

## Summary

Keep open. Current main and the latest stable release still run mcporter daemon startup through the stdout-capturing wrapper and raw-parse mcporter call stdout; the likely fix path is the open superseding PR https://github.com/openclaw/openclaw/pull/92499, not closing this issue yet.

## What This Changes

The issue reports that the first mcporter-backed QMD memory search after gateway restart can see daemon stdout before JSON and silently fall back from vector QMD results to the builtin/FTS path.

## Best Possible Solution

Land a reviewed fix, preferably by finishing or splitting https://github.com/openclaw/openclaw/pull/92499, that isolates daemon-start stdout and parses only the final structured mcporter JSON response with first-call regression coverage.

## Reproduction Assessment

Yes, at source level. With mcporter-backed QMD and `startDaemon=true`, the first search awaits `mcporter daemon start`, current main raw-parses the following mcporter stdout, and mcporter daemon-start status messages are written to stdout.

## Solution Assessment

No, not the issue body's prefix-only filter by itself. The maintainable fix should discard or isolate daemon-start stdout and parse the final structured mcporter response robustly, with the open superseding PR or a narrower split carrying the regression tests.
