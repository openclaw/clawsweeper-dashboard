Keep open: current main and v2026.6.10 still parse mcporter call stdout directly, while the reported daemon-start contamination path remains unproven because OpenClaw captures daemon start and call output in separate subprocess invocations. The two direct filter PRs were closed unmerged, and the open QMD/mcporter sidecar PR is only partially overlapping rather than a merged fix for this issue.

## What This Changes

The issue reports the first qmd-backed `memory_search` after a gateway restart falling back to FTS-only when mcporter daemon startup output contaminates JSON parsing.

## Best Possible Solution

Confirm the actual first-query stdout contamination with live qmd/mcporter or dependency-level proof, then fix stdout purity or structured JSON parsing at the mcporter call boundary with focused qmd-manager regression coverage.

## Reproduction Assessment

No high-confidence live reproduction was established. Source inspection shows non-JSON mcporter call stdout would still throw and trigger fallback, but the daemon-start contamination path is not proven because daemon start and call use separate subprocess captures.

## Solution Assessment

No. The literal `[mcporter]` line filter proposed in the issue is not the best fix because it targets the later call stdout without proving that boundary and misses unprefixed daemon output; the better path is live proof followed by a boundary-level stdout or parser fix.
