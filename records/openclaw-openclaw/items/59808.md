## Label Justifications

- `P1`: The issue describes a real QMD memory-search workflow silently degrading immediately after gateway restart for mcporter-backed users.
- `impact:session-state`: The affected surface is memory retrieval and context recall, where semantic QMD results can be bypassed and reported as fallback.

## Summary

Keep open. Current main and the latest stable release still have the source-level failure path: mcporter daemon startup runs through the stdout-capturing wrapper, mcporter call output is raw-parsed as JSON, and the related fix path is still open in https://github.com/openclaw/openclaw/pull/79745.

## What This Changes

The issue reports that the first mcporter-backed QMD memory search after a gateway restart parses daemon startup stdout as JSON and silently falls back to FTS-only mode.

## Best Possible Solution

Review and land or split the QMD mcporter daemon/stdout fix from https://github.com/openclaw/openclaw/pull/79745 with focused first-call regression coverage, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes, at source level. With mcporter-backed QMD and `startDaemon=true`, the first search awaits `mcporter daemon start`, current main still raw-parses mcporter call stdout, and mcporter daemon-start status text is written to stdout.

## Solution Assessment

No, not the issue body's prefix-only filter by itself. The maintainable fix should discard or isolate daemon-start stdout and parse only the structured mcporter call response, with first-call regression coverage.
