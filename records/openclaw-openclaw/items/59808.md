## Summary

Keep open. Current main still assumes mcporter call stdout is pure JSON, the daemon-start wrapper still cannot discard stdout, and the existing mcporter tests only cover clean stdout, so the reported first-query fallback remains actionable.

## What This Changes

The issue reports that mcporter-backed QMD memory search falls back to FTS on the first query after gateway restart because noisy mcporter stdout is parsed as JSON.

## Best Possible Solution

Fix memory-core to discard mcporter daemon-start stdout where possible, tolerate noisy mcporter call output before parsing JSON, and add a first-search-after-restart regression test.

## Reproduction Assessment

Yes, source-reproducible with high confidence. A focused unit test can enable mcporter startDaemon, mock the first mcporter path to include a [mcporter] line before valid JSON, and observe the current JSON.parse(result.stdout) failure feeding the FTS fallback.

## Solution Assessment

Yes for a narrow repository-side hardening fix, but the proposed prefix-only line filter is weaker than the best fix. Prefer exposing discardStdout for daemon startup and extracting the structured JSON payload robustly before parsing call output.
