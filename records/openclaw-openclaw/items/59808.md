## Summary

Keep open. Current main and the latest release still capture mcporter daemon stdout and raw-parse the mcporter call stdout, while the daemon-start tests only cover clean output; no paired fixing PR was found.

## What This Changes

The issue requests fixing mcporter-backed QMD search so the first `memory_search` after a gateway restart uses the QMD/vector path instead of falling back to FTS when daemon startup text appears on stdout.

## Best Possible Solution

Keep the QMD mcporter path using vector search on the first post-restart query by suppressing daemon-start stdout and parsing the structured call payload defensively, with focused regression coverage.

## Reproduction Assessment

Yes, at source level: enable `memory.qmd.mcporter.startDaemon`, return daemon/status text before a valid mcporter JSON call payload on the first search, and the current `JSON.parse(result.stdout)` path throws before falling back. I did not run a live qmd/mcporter gateway in this read-only review.

## Solution Assessment

No, the proposed prefix-only stdout filter is not the best durable fix. The maintainable path is to discard daemon-start stdout where only the exit code matters and parse or extract the mcporter call JSON object without accepting arbitrary log text as the payload.
