## Summary

Keep open. Current main still parses raw mcporter call stdout as JSON, the mcporter wrapper still has no way to discard startup stdout, existing daemon tests only cover clean stdout, and no open PR references this issue with closing syntax.

## What This Changes

The issue requests hardening QMD mcporter search so the first `memory_search` after a gateway restart does not fall back to FTS because of noisy mcporter stdout.

## Best Possible Solution

Keep this open until memory-core discards mcporter daemon-start stdout where possible, tolerates noisy mcporter JSON output locally, and adds a first-search-after-restart regression test.

## Reproduction Assessment

Yes. A focused unit reproduction can configure QMD mcporter with `startDaemon=true`, mock the first mcporter call stdout to include a `[mcporter]` startup/log line before valid JSON, and observe the current `JSON.parse(result.stdout)` path fail before fallback handling takes over.

## Solution Assessment

Yes for a narrow repository-side hardening fix. Prefer exposing the existing `discardStdout` seam for daemon startup plus robust extraction of the structured mcporter JSON payload, rather than relying only on an upstream mcporter stderr change or a prefix-specific line filter.
