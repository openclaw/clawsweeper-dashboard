## Summary

Keep open. Current main still raw-parses mcporter call stdout and the mcporter daemon tests only cover clean output; mcporter 0.8.1 also writes daemon status lines to stdout, and a public PR search found no open fix for this issue. The exact live stdout-leak mechanism should be confirmed in the fix, but the repository-side parser remains brittle in the reported path.

## What This Changes

The issue asks OpenClaw to stop qmd-backed memory search from falling back to builtin FTS when the first mcporter-backed query after gateway restart receives noisy startup output before JSON.

## Best Possible Solution

Keep this issue open until memory-core tolerates noisy mcporter JSON output, suppresses daemon-start stdout where possible, and adds a first-search-after-restart regression test.

## Reproduction Assessment

Yes. A high-confidence unit reproduction can configure `memory.qmd.mcporter.startDaemon=true`, mock the first mcporter call stdout to include a `[mcporter]` or daemon status line before valid JSON, and observe the current `JSON.parse(result.stdout)` path fail into builtin fallback.

## Solution Assessment

No, not the exact proposed prefix filter as the best fix. A more maintainable fix should expose `discardStdout` for daemon startup and parse the mcporter call payload via balanced JSON object/array extraction before applying the existing `structuredContent` and `results` mapping.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/59808. Current main still parses mcporter qmd call stdout directly in `extensions/memory-core/src/memory/qmd-manager.ts`, so noisy stdout on the first search after daemon restart can throw and force builtin fallback. Keep the fix inside memory-core/host-process code; do not change QMD collection semantics, provider behavior, or core/plugin boundaries. Prefer exposing `discardStdout` through `runMcporter()` and using it for `mcporter daemon start`, then harden `runQmdSearchViaMcporter()` so it extracts the structured JSON object or array from noisy `mcporter call --output json` stdout before applying the current `structuredContent`/`results` mapping. Avoid a prefix-only `[mcporter]` line filter. Add a regression test with `startDaemon=true` where the first mcporter call stdout has a mcporter/daemon log line before valid JSON and search still returns QMD results. Add a user-facing `CHANGELOG.md` fix entry; do not credit bot/maintainer handles if no human contributor attribution is appropriate.
