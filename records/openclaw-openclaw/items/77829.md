## Summary

Keep open. Current main has adjacent voice surfaces and Discord-specific voice-channel support, but not a generic trusted handoff from the current Telegram/Discord/Slack/etc. thread into a live voice room that preserves the same OpenClaw session and original IM transcript.

## What This Changes

Add a first-class session-bound voice handoff that lets an active IM thread open a live voice room or call while preserving the same OpenClaw session and canonical written transcript.

## Best Possible Solution

Design a generic Gateway/SDK voice-handoff contract with short-lived trusted tokens, actual sessionId binding, a default web voice room, and optional channel-native adapters that preserve original delivery and transcript semantics.

## Reproduction Assessment

Not applicable: this is a feature request, not a report of broken current behavior. Source and docs review shows the adjacent current surfaces but no generic session-bound IM voice handoff to reproduce.

## Solution Assessment

No, treating this as already solved or as a simple cleanup item would miss the central product/API gap. The maintainable path is an explicit handoff design that reuses Talk/VoiceClaw/voice-call/channel routing pieces without forging session keys from user text.
