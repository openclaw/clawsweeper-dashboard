## Summary

Keep open: the security goal is valid, but the branch appears to break configured plaintext provider keys because the strip pass replaces/removes `apiKey` from the prompt-facing catalog without moving config-sourced keys into runtime auth. The supplied proof covers catalog generation, legacy migration, and discovery, but not request-time auth after stripping.

## What This Changes

The PR strips plaintext provider `apiKey` values from generated and legacy `models.json`, migrates eligible legacy catalog keys to `auth-profiles.json`, preserves non-secret markers, and adds agent model config regression tests.

## Best Possible Solution

Keep the prompt-facing catalog stripped, but preserve every credential source by moving config-sourced plaintext keys into durable auth profiles or threading them into pi runtime auth before replacing catalog keys, then prove both catalog cleanliness and request auth.

## Reproduction Assessment

Yes, at source level: a custom provider with a plaintext `models.providers.<id>.apiKey` is stripped to `secretref-managed`, migration only reads previous `models.json` keys, and runtime discovery calls `discoverAuthStorage(agentDir)` without the config. I did not run a live repro because this review was read-only.

## Solution Assessment

No, not yet: stripping the prompt-facing catalog is the right direction, but the current implementation must preserve config-sourced credentials in runtime auth before replacing the catalog key. A safer fix is to add a durable auth-profile or discovery handoff plus a regression around request auth, not only model discovery.
