## Summary

Keep this PR open: the latest head moves in the right direction and covers the earlier plaintext reintroduction gap, but the new migration can still drop the only persisted provider credential if the auth-profile write fails, and the supplied real-run proof predates the migration commit.

## What This Changes

The PR strips plaintext provider `apiKey` values from generated `models.json` entries and adds a migration from existing `models.json`-only plaintext keys into auth profiles.

## Best Possible Solution

Make the credential move atomic: write a prompt-safe `models.json` only after the existing plaintext key is durably migrated or an explicit recovery path preserves provider auth, and keep the broader security roadmap open unless remaining layers are separately tracked.

## Reproduction Assessment

Yes at source level: `upsertAuthProfileWithLock` can return `null` on a lock/save failure, and the new migration ignores that result before `ensureOpenClawModelsJson` writes the stripped catalog. I did not run a filesystem failure repro because this review is read-only.

## Solution Assessment

No. The direction is right, but the migration must make credential preservation durable before redaction and the PR should avoid auto-closing the broader security roadmap as a side effect.
