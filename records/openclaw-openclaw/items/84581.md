## Summary

Keep open: the PR addresses a real provider-key exposure path, but the patch currently strips the field that pi-coding-agent still requires to load non-built-in custom providers with model rows, so existing custom-provider flows can regress.

## What This Changes

The PR strips plaintext provider `apiKey` values from generated and merged `models.json`, migrates legacy catalog-only keys to `auth-profiles.json`, and adds regression coverage for stripping, migration, and fail-closed persistence.

## Best Possible Solution

Keep the prompt-facing catalog secret-safe while preserving pi registry compatibility, then add a regression that a non-built-in custom provider remains discoverable and usable after stripping plaintext secrets.

## Reproduction Assessment

Yes, at source level. Current main can preserve plaintext `apiKey` values into `models.json`, and the PR's stripped output conflicts with pi-coding-agent's custom-provider `apiKey` validation path.

## Solution Assessment

No, not as-is. Final stripping plus migration is the right security direction, but the implementation needs a compatible non-secret catalog contract or consumer refactor before merge.
