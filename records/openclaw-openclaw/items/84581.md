## Summary

Keep open. The PR has strong proof for the intended stripping/migration paths, but the latest diff still has a P1 auth-boundary regression: a target agent can skip migrating its local legacy catalog key because an inherited profile exists, then write the stripped catalog and silently switch credentials.

## What This Changes

The PR strips plaintext provider `apiKey` values from generated and legacy agent `models.json`, migrates eligible legacy keys into auth profiles, preserves non-secret markers, and adds regression coverage.

## Best Possible Solution

Strip prompt-facing catalog secrets, but migrate legacy local credentials before the catalog write using target-local persisted profile checks and keep regressions for inherited-profile, catalog-only, env-name, discovery, and fail-closed paths.

## Reproduction Assessment

Yes, by source inspection. A non-main agent with a local legacy `models.json` plaintext key and an inherited same-provider profile reaches the PR's merged-store profile check, skips migration, and then writes a stripped catalog.

## Solution Assessment

No. The security direction is right, but the migration guard must distinguish target-local persisted profiles from inherited overlays before the PR is safe to merge.
