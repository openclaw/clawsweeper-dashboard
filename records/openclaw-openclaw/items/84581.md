## Label Justifications

- `P1`: The PR addresses a provider-key prompt leak and changes credential migration behavior that can break real auth flows during upgrade.
- `merge-risk: 🚨 compatibility`: Merging changes the persisted `models.json` credential contract for existing users and can remove a legacy credential path.
- `merge-risk: 🚨 auth-provider`: The diff moves provider credentials from `models.json` into auth profiles and can change which provider credential wins.
- `merge-risk: 🚨 security-boundary`: The patch is explicitly enforcing a secret boundary between prompt-facing catalog data and runtime credentials, and the migration guard can cross per-agent credential boundaries.

## Summary

Keep open: the security direction is useful and the supplied proof is strong, but the latest patch still has a blocking auth-boundary regression in the legacy key migration path.

## What This Changes

The PR strips plaintext provider `apiKey` values from generated and legacy `models.json`, migrates eligible legacy catalog credentials into `auth-profiles.json`, preserves non-secret provider markers, and adds regression coverage.

## Best Possible Solution

Keep the prompt-safe catalog change, but decide migration using only target-local persisted auth-profile state before stripping local legacy keys, and prove inherited or external profiles do not suppress local migration.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create a non-main agent with a local legacy plaintext `models.json` key plus an inherited or external same-provider profile, then the guard at `src/agents/models-config.ts:213` skips migration before the stripped catalog is written.

## Solution Assessment

No. Stripping prompt-facing secrets is the right direction, but the current implementation should use target-local persisted profile state for migration decisions instead of the merged runtime auth-store view.
