## Summary

The security direction is good, and the supplied proof covers the intended stripping and migration paths, but the latest PR head still has a P1 auth-boundary regression: a local legacy catalog key can be stripped without being migrated when an inherited or external same-provider profile is visible.

## What This Changes

The PR strips plaintext provider `apiKey` values from generated and legacy agent `models.json`, migrates eligible legacy keys into `auth-profiles.json`, preserves non-secret markers, and adds focused regression tests.

## Best Possible Solution

Keep the plaintext-stripping direction, but make the migration guard check only profiles persisted in the target agent store and add a non-main inherited/external-profile upgrade regression before merge.

## Reproduction Assessment

Yes, by source inspection. Construct a non-main agent with a local legacy `models.json` plaintext provider key and a same-provider inherited or external profile; the PR head sees that profile through `ensureAuthProfileStore()`, skips local migration, and then writes the stripped catalog.

## Solution Assessment

No, not yet. The stripping and migration approach is maintainable, but the profile-existence guard must be target-local so cleanup does not silently switch or lose agent-specific credentials.
