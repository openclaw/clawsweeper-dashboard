## Summary

Keep open: the intended redaction and happy-path migration are now well supported, but one upgrade-safety bug remains because the PR ignores a failed auth-profile upsert before writing the stripped catalog.

## What This Changes

The PR strips plaintext provider apiKey values from generated and merged models.json output, migrates existing models.json-only keys into auth profiles, and adds regression coverage.

## Best Possible Solution

Make the migration fail closed before removing old plaintext keys, and keep the broader security roadmap open unless maintainers split out a narrower Layer 1 issue.

## Reproduction Assessment

Yes at source level. Current main serializes merged provider apiKey values into models.json, and the remaining PR blocker is also source-reproducible because upsertAuthProfileWithLock can return null while the caller still writes the stripped catalog.

## Solution Assessment

No, not yet. Final stripping plus migration is the right direction, but the migration must verify persistence before removing the old key and the broad roadmap should not be auto-closed by this partial layer.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Require migration to persist before stripping keys:** `src/agents/models-config.ts:186-190`
  - body: `upsertAuthProfileWithLock()` returns `AuthProfileStore | null` when the auth-store update fails, but this code ignores that result and then writes the stripped `models.json`. Please fail the models write, retry, or otherwise keep the old key until the migration is durable.
