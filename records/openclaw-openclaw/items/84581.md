## Summary

Keep open: the underlying plaintext model-key leak is valid and the PR has useful stripping coverage, but the branch still moves legacy credential repair into the `ensureOpenClawModelsJson` startup/load path and its real-behavior proof is stale for the current head.

## What This Changes

The PR strips provider `apiKey` values from generated and legacy `models.json`, migrates legacy `models.json`-only keys into agent auth profiles, adjusts plugin metadata snapshot resolution, and adds focused model-config tests.

## Best Possible Solution

Keep the prompt-facing `models.json` stripping path, preserve runtime auth through existing config/profile resolution, and move any one-time legacy `models.json` credential migration into an explicit doctor or secrets repair flow with focused upgrade tests.

## Reproduction Assessment

Yes. Current main's model-config plan writes provider data directly to `models.json`, and the docs describe `models.json` `apiKey` precedence, so a plaintext configured or legacy provider key follows a clear source-level leak path.

## Solution Assessment

No. The stripping direction is the right narrow fix, but the PR's startup/load-time credential migration is not the best boundary; legacy repair should be explicit through doctor or a secrets repair flow.
