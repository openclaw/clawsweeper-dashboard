## Label Justifications

- `P1`: This is an auth/security fix for prompt-facing provider key exposure, but the current patch can regress real upgraded agent credentials.

## Summary

Keep this PR open: the strip-and-migrate direction is useful and the supplied proof covers the main happy paths, but the current migration guard can skip a target-agent legacy key when an inherited profile exists, creating a P1 auth upgrade regression before merge.

## What This Changes

The PR strips plaintext provider `apiKey` values from generated and legacy agent `models.json`, migrates eligible legacy keys into `auth-profiles.json`, preserves non-secret markers, and adds model-config regression coverage.

## Best Possible Solution

Land the strip-and-migrate approach only after the migration guard checks target-local persisted profiles before stripping the catalog, with an upgrade regression for inherited profile collisions and an explicit maintainer decision on the compatibility/fail-closed tradeoff.

## Reproduction Assessment

Yes via source inspection: create a non-main agent with a local legacy `models.json` plaintext key and a main/external auth profile for the same provider. The PR loads a merged auth store, skips migration, then writes the stripped catalog.

## Solution Assessment

No, not yet: stripping prompt-facing keys and migrating credentials is the right direction, but the migration must check target-local persisted profiles before stripping. The remaining compatibility and fail-closed behavior also needs maintainer acceptance.
