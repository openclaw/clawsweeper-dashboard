## Summary

Keep open: the PR is security-positive and has strong proof for legacy/custom catalog cleanup, but it still appears to regress model availability for built-in providers whose only credential is `cfg.models.providers.<id>.apiKey` because that prompt-facing key is stripped without a replacement in auth storage or request config.

## What This Changes

The PR strips plaintext provider `apiKey` values from generated and legacy `models.json`, migrates eligible legacy catalog-only keys into `auth-profiles.json`, preserves non-secret markers, and adds agents regression coverage.

## Best Possible Solution

Land a prompt-safe credential boundary that keeps `models.json` free of plaintext keys while preserving built-in and custom provider availability through auth profiles or in-memory discovery auth, with release notes for the intentional storage break.
