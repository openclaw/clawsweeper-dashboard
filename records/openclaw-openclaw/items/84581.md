## Summary

Keep open: current main still has the prompt-facing `models.json` provider-key leak, and this PR has useful stripping and regression coverage, but it is not merge-ready because it performs legacy credential migration during normal runtime catalog writes, carries auth/security upgrade risk, and the proof is stale to the current head.

## What This Changes

The PR strips plaintext provider `apiKey` values from generated and legacy agent `models.json`, migrates legacy catalog keys into local auth profiles, preserves non-secret markers, adjusts plugin metadata snapshot resolution, and adds regression tests.

## Best Possible Solution

Land a narrowed security fix that strips the prompt-facing `models.json` catalog while moving legacy credential adoption to `openclaw doctor --fix` or another maintainer-approved explicit repair path, then refresh current-head proof.

## Reproduction Assessment

Yes, source-level: current main writes `finalProviders` directly into `models.json`, so a provider `apiKey` can persist into the prompt-facing catalog. I did not run a live repro because this review was read-only.

## Solution Assessment

No as currently shaped: stripping the prompt-facing catalog is the right direction, but normal runtime writes should not also perform legacy credential migration without an explicit maintainer decision. The safer path is prompt stripping plus an explicit doctor/repair migration.
