The PR strips provider `apiKey` values from generated and legacy agent `models.json`, migrates old catalog-only keys into `auth-profiles.json`, preserves non-secret/custom-provider markers, and adds regression tests.

## Best Possible Solution

Strip prompt-facing plaintext keys, but migrate legacy env-backed catalog values into env-backed auth profiles and explicitly validate the intended direct-plaintext catalog deprecation path before merge.

No. The strip-and-migrate approach is the right direction, but the migration must preserve env-backed semantics, for example by writing a `keyRef` for env-shaped legacy values instead of copying them into `key`.
