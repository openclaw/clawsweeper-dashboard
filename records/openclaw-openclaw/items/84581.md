## Summary

Keep open: this PR is aimed at a real provider-key exposure path and has strong after-fix proof, but the branch can still skip migration for a non-main agent when an inherited or external same-provider auth profile exists, then write a stripped models.json that loses the local legacy credential.

## What This Changes

The PR strips plaintext provider apiKey values from generated and legacy models.json, migrates eligible legacy catalog keys into auth-profiles.json, preserves non-secret discovery markers, and adds focused regression tests.

## Best Possible Solution

Keep the strip-and-migrate direction, but make the migration guard check only profiles persisted in the target agent store before stripping, with upgrade coverage for inherited and external profile collisions.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: create a non-main agent with a local legacy plaintext models.json key plus a same-provider inherited or external profile, then the branch guard at src/agents/models-config.ts:213 skips migration before writing the stripped catalog.

## Solution Assessment

No. Stripping plaintext keys and migrating legacy credentials is the right direction, but the branch should use target-agent persisted profile state for migration decisions instead of the merged runtime auth store.
