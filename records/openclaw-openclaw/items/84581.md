## Summary

Keep this PR open: the security direction is useful and the latest head fixes the earlier env-backed migration blocker, but the branch still has a P1 upgrade regression where agent-local catalog credentials can be stripped without being migrated when a main or external auth profile exists for the same provider.

## What This Changes

The PR strips plaintext provider `apiKey` values from generated and legacy `models.json`, migrates legacy catalog-only credentials into auth profiles, preserves env/keyRef/custom-provider markers, and adds agents regression coverage.

## Best Possible Solution

Land the strip and migration after ensuring agent-local legacy catalog keys are migrated into agent-local auth profiles even when main or external profiles exist, with focused upgrade coverage.

## Reproduction Assessment

Yes, from source inspection: create a non-main agent with a local `models.json` plaintext apiKey and a main auth profile for the same provider; the PR's overlaid-store check skips migration, then writes the stripped catalog. I did not run the repro because this review is read-only.

## Solution Assessment

No. The strip/migration is the right direction, but the migration guard should use target-agent persisted auth state, not the overlaid runtime auth store, before removing the catalog key.
