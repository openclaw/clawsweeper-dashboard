## Summary

Keep open: the PR targets the right security boundary and the supplied proof covers the main catalog-stripping and migration paths, but the latest diff still has a compatibility bug for built-in-provider inline models whose only auth is a config plaintext key.

## What This Changes

The PR strips plaintext model provider apiKey values from generated and legacy models.json, migrates eligible legacy catalog-only keys into auth-profiles.json, preserves non-secret markers, and adds agents regression coverage.

## Best Possible Solution

Land the secret-stripping and migration approach only after config-backed discovery/availability remains intact for both custom and built-in inline providers, with release guidance for removing plaintext models.json keys.

## Reproduction Assessment

Yes, at source level. Current main serializes final model providers into models.json, and PR head strips apiKey values; the remaining compatibility bug follows from pi-coding-agent ModelRegistry using models.json provider apiKey for availability when authStorage lacks env/profile credentials.

## Solution Assessment

No, not yet. The security direction is right, but the implementation should preserve registry availability/request-auth semantics for config-backed built-in inline providers before merge.
