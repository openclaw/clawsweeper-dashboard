## Summary

The latest head addresses the earlier local-vs-inherited migration gap and includes strong after-fix proof, but the patch still has a P1 auth-boundary regression: it mutates the possibly inherited auth store returned by `ensureAuthProfileStore()` while migrating a target agent's legacy catalog key.

## What This Changes

The PR strips plaintext provider `apiKey` values from generated and legacy `models.json`, migrates eligible legacy keys into local auth profiles, preserves non-secret markers, and adds focused regression coverage.

## Best Possible Solution

Keep the catalog-stripping and local migration design, but make migration side effects target-local only and prove runtime snapshots cannot cross agent credential boundaries.

## Reproduction Assessment

Yes, by source inspection. A non-main agent with only a main runtime auth snapshot can receive `mainStore` from `ensureAuthProfileStore()`, and the PR then writes the worker credential into that returned store.

## Solution Assessment

No. The stripping and local migration direction is the right maintainable shape, but the implementation must avoid mutating merged or inherited auth-store objects.
