## Summary

Keep this PR open: the security direction is right and the supplied live-output proof is useful, but the latest patch still appears to break PI registry availability for built-in providers whose only credential is a literal `models.providers.<id>.apiKey` in config.

## What This Changes

The branch strips plaintext provider `apiKey` values from generated and legacy `models.json`, migrates eligible legacy catalog keys into `auth-profiles.json`, preserves non-secret markers, and adds agents regression coverage.

## Best Possible Solution

Keep `models.json` prompt-safe, but bridge configured and migrated credentials into PI discovery through a non-prompt-facing auth source so built-in and custom provider availability stays aligned with runtime auth.

## Reproduction Assessment

Yes, at source level. The PR removes built-in provider `apiKey` values from `models.json`, while current runtime registry discovery commonly calls `discoverAuthStorage(agentDir)` without config, so a config-only built-in provider can lose PI availability.

## Solution Assessment

No, not yet. The prompt-safe catalog boundary is the right direction, but the patch needs a non-prompt auth bridge or migration for built-in config-only credentials before it is the safest fix.
