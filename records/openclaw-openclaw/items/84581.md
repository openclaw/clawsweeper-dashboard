## Label Justifications

- `merge-risk: 🚨 compatibility`: The generated `models.json` contract changes, and repeated regeneration can make existing custom provider model rows invalid for pi.

## Summary

Keep this PR open: the security direction is right, but the latest marker-compatibility change can still drop the non-secret marker on a later regeneration and make migrated custom providers unloadable.

## What This Changes

The PR strips plaintext model provider `apiKey` values from generated and merged agent `models.json`, migrates legacy catalog-only keys into `auth-profiles.json`, and adds regression coverage for stripping, migration, and fail-closed persistence.

## Best Possible Solution

Keep `models.json` prompt-safe while making the non-secret marker durable across first and subsequent regenerations for migrated custom providers, with regression coverage for the repeated-write path.

## Reproduction Assessment

Yes, at source level. Start from an existing custom provider whose `models.json` already contains `apiKey: "secretref-managed"` after migration, regenerate from config with model rows and no plaintext `apiKey`, and the PR can emit a provider with models but no marker, which pi 0.75.3 rejects for non-built-in providers.

## Solution Assessment

No, not as-is. Stripping and migration are the right direction, but the patch needs durable marker preservation plus updated real behavior proof for the latest head before it is the narrow maintainable fix.
