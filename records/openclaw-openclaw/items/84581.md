## Summary

Keep this PR open: it addresses the right security bug and has strong live-output proof, but the latest head still has a P1 auth-boundary regression where a non-main agent can lose its local legacy catalog key if any inherited same-provider profile exists.

## What This Changes

This PR strips provider apiKey values from agent models.json, migrates eligible legacy catalog-only keys into auth-profiles.json, preserves marker compatibility, and adds models-config regression tests.

## Best Possible Solution

Strip prompt-facing catalog secrets while migrating legacy keys into a target-agent-local persisted auth profile before any catalog write, with regression coverage for local, inherited-main, external-overlay, env-backed, and persistence-failure upgrade paths.

## Reproduction Assessment

Yes, by source inspection. A non-main agent with a local legacy models.json plaintext key and an inherited same-provider profile reaches the PR's merged-store profile check, skips migration, and then writes the stripped catalog.

## Solution Assessment

No, not yet. The direction is right, but the implementation should distinguish target-agent persisted profiles from inherited or external overlays before deciding that a local legacy catalog key is already covered.
