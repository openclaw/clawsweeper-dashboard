## Summary

Keep open for maintainer review: the latest PR appears to implement the intended prompt-catalog secret stripping with useful live-output proof and no blocking code findings, but the credential migration and fail-closed behavior are upgrade-sensitive and need explicit maintainer acceptance before merge.

## What This Changes

The PR strips plaintext model provider `apiKey` values from generated and legacy `models.json`, migrates eligible legacy keys to `auth-profiles.json`, preserves non-secret markers for custom-provider discovery, and adds regression coverage.

## Best Possible Solution

Land a prompt-safe catalog implementation that strips plaintext keys, preserves runtime auth through config/env/auth profiles, migrates legacy catalog-only keys durably, and documents the upgrade behavior at release time.

## Reproduction Assessment

Yes at source level. Current main serializes provider entries into `models.json` without a final plaintext strip pass, and upstream pi stores provider `apiKey` from that file for request auth.

## Solution Assessment

Yes, assuming maintainers accept the upgrade policy. The PR uses the existing auth-profile boundary instead of inventing a new secret store, strips the prompt-facing catalog, and fails before destructive cleanup if migration cannot persist.
