## Summary

Keep this PR open: the security direction and proof are useful, but the latest diff still has a P1 auth-boundary regression where a legacy per-agent catalog key can be stripped without being migrated when an inherited or external same-provider profile exists.

## What This Changes

The branch strips plaintext provider `apiKey` values from generated and legacy `models.json`, migrates eligible legacy catalog keys into `auth-profiles.json`, preserves non-secret markers, and adds focused regression coverage.

## Best Possible Solution

Keep the prompt-facing catalog stripped, but decide migration from the target agent's persisted auth store only and prove the inherited/external profile case before merge.

## Reproduction Assessment

Yes, by source inspection. The PR calls `ensureAuthProfileStore()` and then `listProfilesForProvider()`, while current main merges main-agent profiles into non-main stores before returning that store.

## Solution Assessment

No, not yet. The stripped catalog is the right security direction, but the migration guard should check target-local persisted profiles rather than the merged runtime-visible store.
