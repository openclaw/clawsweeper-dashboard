## Summary

Keep open: the security direction is useful and the supplied proof covers the main happy paths, but the latest head still has a P1 auth-boundary regression where an inherited or external same-provider profile can suppress migration of a local legacy catalog key before the catalog is stripped.

## What This Changes

The PR strips plaintext provider `apiKey` values from generated and legacy `models.json`, migrates eligible legacy keys into `auth-profiles.json`, preserves non-secret discovery markers, and adds regression tests.

## Best Possible Solution

Keep the plaintext-stripping direction, but make migration decisions from the target agent's persisted local auth store and require upgrade proof for both normal migration and fail-closed persistence behavior.

## Reproduction Assessment

Yes, from source inspection. Construct a non-main agent with a local legacy `models.json` apiKey and a same-provider inherited or external profile; the PR head checks the effective merged store, skips migration, and then writes the stripped catalog.

## Solution Assessment

No, not yet. The stripping and migration approach is the right shape, but the migration guard must distinguish target-local persisted profiles from inherited or external runtime profiles before this is safe to merge.
