## Summary

Keep open: the current PR head still checks the merged auth-profile view before migrating legacy catalog keys, so a non-main agent can strip its local `models.json` key while falling back to a main or external profile for the same provider.

## What This Changes

The PR strips plaintext provider `apiKey` values from generated and legacy agent `models.json`, migrates eligible legacy catalog keys into `auth-profiles.json`, preserves non-secret markers, and adds agents regression coverage.

## Best Possible Solution

Land the strip and migration only after checking target-agent persisted profiles separately from merged fallback/external profiles, with focused upgrade coverage for the non-main-agent case.

## Reproduction Assessment

Yes, from source inspection: create a non-main agent with a local legacy `models.json` plaintext key and a main/external auth profile for the same provider; the merged-store check can skip migration before the stripped catalog is written. I did not execute it because this review is read-only.

## Solution Assessment

No, not as written. The stripping/migration direction is appropriate, but the migration guard should distinguish target-agent persisted profiles from inherited or external overlays before this is safe to merge.
