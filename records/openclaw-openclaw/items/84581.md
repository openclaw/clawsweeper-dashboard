## Summary

Keep open: the earlier null-upsert blocker is addressed and the proof is strong, but the migration guard still treats blank configured provider keys as real credentials, so an old models.json-only key can be stripped without being migrated.

## What This Changes

The branch strips plaintext provider apiKey values from generated models.json output, migrates old models.json-only keys into auth-profiles.json, and adds planner/write regression tests.

## Best Possible Solution

Normalize configured apiKey values before the migration skip check, treat blank or whitespace as missing while still honoring non-empty strings and SecretRefs, and add a focused upgrade regression test.

## Reproduction Assessment

Yes at source level. Current main preserves and serializes provider apiKey values in merge mode, and the PR's blank configured-key path can be traced through hasConfiguredProviderApiKey without running the read-only checkout.

## Solution Assessment

No, not yet. Stripping plus migration is the right direction, but the migration guard needs to use the same non-empty/SecretRef semantics as runtime auth before this is safe to merge.
