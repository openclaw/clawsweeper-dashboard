## Summary

Keep open: the security direction and proof are strong, but the migration guard still treats inherited or external same-provider auth profiles as if they were target-local, so a non-main agent can lose its local legacy catalog credential when the stripped catalog is written.

## What This Changes

This PR strips plaintext provider apiKey values from generated and legacy models.json, migrates eligible legacy catalog keys into auth-profiles.json, preserves non-secret markers for pi discovery, and adds focused models-config regression tests.

## Best Possible Solution

Strip prompt-facing catalog secrets while migrating legacy keys based on the target agent's persisted/local auth store only, with regression coverage for main-store and external profile overlays.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path against the PR head: create a non-main agent with a local legacy plaintext models.json key plus a same-provider inherited or external profile, then the skip at src/agents/models-config.ts:213 prevents migration before the catalog is stripped.

## Solution Assessment

No. The overall approach is maintainable, but the migration should check only target-agent persisted profiles before deciding that a legacy local catalog key is safe to drop.
