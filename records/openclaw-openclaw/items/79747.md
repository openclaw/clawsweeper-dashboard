## Summary

Keep open: the updated browser/gateway proof is now sufficient, but the PR still introduces two source-confirmed pinned-chat rename bugs that should block merge.

## What This Changes

This PR adds a Control UI pinned-chats sidebar with local and `sessions.patch`-backed preferences, chat-history snapshot caching, protocol/session-store fields, Swift mirrors, docs, styling, and tests.

## Best Possible Solution

Keep the feature direction, but fix the pinned rename data model so stale/missing pins are unpin-only and rename drafts use canonical session labels before maintainer UX/changelog acceptance.

## Reproduction Assessment

Not applicable for the feature request: this PR adds new Control UI functionality. The blocking review findings are source-reproducible in the PR head by rendering a stale pinned key or two duplicate-label pinned sessions and using Rename.

## Solution Assessment

No: the overall approach is plausible and the proof is now acceptable, but the rename path needs a separate raw label from the display label and should not allow missing pins to send `sessions.patch` label updates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Disable rename for missing pinned chats:** `ui/src/ui/app-render.helpers.ts:674-681`
  - body: Pinned rows with stale keys are marked missing, but the row still renders Rename. Saving sends `sessions.patch`, whose gateway path creates a new entry when none exists, so renaming a deleted pin can resurrect a ghost session instead of leaving the row unpin-only.
  - confidence: 0.86
- **[P2] Seed rename drafts from raw labels:** `ui/src/ui/app-render.helpers.ts:681`
  - body: Duplicate pinned labels are disambiguated by appending ` · <scope>` for display, then Rename seeds the draft from that rendered label. Saving or blurring without edits can persist the UI-only suffix; keep a raw label for editing.
  - confidence: 0.84
