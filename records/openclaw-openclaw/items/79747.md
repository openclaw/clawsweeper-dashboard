Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Move pinned session params into the package protocol schema:** `src/gateway/protocol/schema/sessions.ts:172-176`
  - body: Current main validates `sessions.patch` from `packages/gateway-protocol/src/schema/sessions.ts` with `additionalProperties: false`. Adding the pinned fields only under the old `src/gateway/protocol/schema` path leaves the Control UI payloads rejected or missing after the required rebase, so the fields and generated clients need to move to the package protocol surface.
  - confidence: 0.92
- **[P1] Use scoped agent params for pinned parallel chats:** `ui/src/ui/app-render.helpers.ts:392-396`
  - body: When the active chat is the `global` alias for a selected non-default agent, `parseAgentSessionKey(state.sessionKey)` returns null and this sends `agentId: "main"`. Current main uses selected-agent helpers for this boundary; pinned parallel creation needs the same scoping or it can create the new chat under the wrong agent.
  - confidence: 0.9
- **[P1] Route pinned renames through the session patch helper:** `ui/src/ui/app-render.helpers.ts:468`
  - body: This direct `sessions.patch` call only sends `{ key, label }`, so renaming a pinned `global` entry can patch the default agent instead of the currently selected global agent. Reuse or extend the existing `patchSession`/selected-session request helper so rename has the same `agentId` behavior as the rest of Control UI session patching.
  - confidence: 0.86
