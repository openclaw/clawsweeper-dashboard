## Summary

Keep open. Current main still hardcodes memory-core short-term promotion writes to `MEMORY.md`, exposes no destination override in the apply API, manual CLI, managed dreaming config, plugin schema, or docs, and the related open PRs address adjacent bloat/contamination problems rather than configurable promotion target files.

## What This Changes

Add a configurable memory-core promotion destination so short-term promotions can write to a target other than `MEMORY.md` while remaining searchable.

## Best Possible Solution

Implement a memory-core-owned promotion-target resolver shared by manual promotion and managed dreaming, defaulting to `MEMORY.md`, supporting documented date tokens for constrained workspace-relative Markdown paths, creating missing files with the existing header behavior, and ensuring the resolved target is indexed/searchable.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: any `applyShortTermPromotions` caller writes to `workspaceDir/MEMORY.md`, and both manual CLI and managed dreaming reach that function without a destination override.

## Solution Assessment

Yes. A focused memory-core configuration and resolver is the narrowest maintainable solution for this issue; adjacent fixes that cap or clean `MEMORY.md` growth do not provide separate promotion destinations.
