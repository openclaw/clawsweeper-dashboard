## Summary

Keep open. Current main still writes short-term promotion output to a single hardcoded `MEMORY.md` path, and the adjacent open PRs address bloat/contamination symptoms rather than adding a configurable promotion destination.

## What This Changes

Add a memory-core configuration option for where short-term promotions are written, while keeping `MEMORY.md` as the default.

## Best Possible Solution

Implement a memory-core-owned promotion target resolver shared by manual promotion and managed dreaming, defaulting to `MEMORY.md`, with safe workspace-relative Markdown paths, documented date tokens, missing-file creation, search indexing coverage, tests, and docs.

## Repair Work Prompt

Implement configurable memory-core short-term promotion targets for issue #71330. Add a memory-core-owned resolver used by both `openclaw memory promote --apply` and managed dreaming promotion. Default to `MEMORY.md`; support a constrained workspace-relative Markdown target under `plugins.entries.memory-core.config.dreaming` or a clearly named promotion config key; support documented date tokens such as `YYYY`, `MM`, and `DD`; reject absolute paths, traversal, non-Markdown targets, and symlink escapes; create missing target files with the existing long-term-memory header behavior. Ensure promoted content remains searchable by either constraining targets to indexed locations or extending the default QMD collection behavior safely. Update manifest schema/UI hints, config resolver types, CLI JSON/text reporting, docs, changelog, and focused tests. Do not change unrelated dreaming phase storage semantics, provider behavior, or bootstrap limits.
