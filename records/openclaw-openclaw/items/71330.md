## Summary

Keep open. Current main still writes short-term memory promotions to `MEMORY.md` unconditionally, exposes no manual or managed promotion target override, and the memory-core config schema/docs do not contain a `promotionTarget`-style option. The related open PR #74088 is useful adjacent size-budget work, but by its own scope it does not make the promotion destination configurable.

## What This Changes

Request to add a memory-core configuration option for the file path used by short-term promotion writes, defaulting to `MEMORY.md` while allowing separate or date-templated Markdown targets that remain searchable.

## Best Possible Solution

Implement the feature in the bundled memory-core plugin with one shared promotion target resolver used by both manual `memory promote --apply` and managed dreaming. The end state should keep `MEMORY.md` as the default, validate workspace-relative Markdown targets, support any accepted date-token format deterministically, create missing target files with the existing header behavior, and include the resolved target in memory indexing without requiring users to duplicate `memorySearch.extraPaths` manually.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `applyShortTermPromotions()` always resolves `workspaceDir/MEMORY.md`, and both manual CLI and managed dreaming call it without a target override. I did not run a live write because this review was required to stay read-only.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix if implemented inside memory-core. `memorySearch.extraPaths` can make a manually separated file searchable, but it cannot change where promotions are written, so a writer-owned config option is still needed.
