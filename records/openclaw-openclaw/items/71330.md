## Summary

Keep open. Current main still hardcodes short-term promotion writes to `MEMORY.md` and exposes no promotion target through apply options, CLI, managed dreaming config, plugin schema, or docs. The adjacent open PRs address `MEMORY.md` growth or contamination, but not the requested destination-file configuration.

## What This Changes

Request to make memory-core's short-term and dreaming promotion destination configurable instead of always writing promoted entries to `MEMORY.md`.

## Best Possible Solution

Add a memory-core-owned promotion target resolver that defaults to `MEMORY.md`, expands a small documented date-token set deterministically, validates the resolved Markdown path, creates missing files with the existing long-term-memory header behavior, and is shared by manual `memory promote --apply` and managed dreaming. The selected target should be included in memory search indexing without requiring users to duplicate config.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: promotion apply resolves `workspaceDir/MEMORY.md`, and both manual CLI and managed dreaming call the writer without a destination override. I kept the review read-only and did not run a live write path.

## Solution Assessment

Yes. A memory-core config path shared by the promotion writer, CLI, dreaming config, docs, and indexing is the narrowest maintainable solution for the requested destination-file behavior; the adjacent open PRs do not solve the target-file requirement.
