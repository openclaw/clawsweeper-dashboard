## Summary

Keep open. Current main still hardcodes short-term/deep promotion output to `MEMORY.md`, exposes no promotion-target setting in the memory-core CLI, dreaming config, plugin schema, or docs, and the related open PRs address adjacent MEMORY.md bloat/contamination symptoms rather than configurable destination files.

## What This Changes

Request to add a memory-core configuration option for where promoted short-term memories are written, defaulting to `MEMORY.md` while supporting date-tokenized Markdown targets that remain searchable.

## Best Possible Solution

Implement a memory-core-owned promotion target resolver shared by manual promotion and managed dreaming. It should default to `MEMORY.md`, accept a constrained workspace-relative Markdown target with documented date tokens, create missing files with the existing long-term memory header behavior, ensure the resolved target is indexed for `memory_search`, and keep the behavior inside the plugin rather than adding core-specific special cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: `applyShortTermPromotions` always resolves `workspaceDir/MEMORY.md`, and both manual CLI and managed dreaming call it without any destination override; I kept the review read-only and did not perform a live write.

## Solution Assessment

Yes. A memory-core config option plus one shared resolver is the narrowest maintainable solution; the adjacent open PRs are useful for MEMORY.md bloat/contamination but do not solve destination-file configurability.
