## Summary

Keep open. Current main still hardcodes short-term promotion writes to `MEMORY.md` and exposes no configurable promotion target; https://github.com/openclaw/openclaw/pull/78764 is an active archive-first implementation candidate, but it is not merged and the broader config/API decision remains open.

## What This Changes

Request a memory-core config option for short-term promotions to write to a configurable, searchable target file instead of always appending detailed promotion sections to `MEMORY.md`.

## Best Possible Solution

Decide whether the archive-first flow in https://github.com/openclaw/openclaw/pull/78764 satisfies this issue; if configurable targets are still desired, add a memory-core-owned safe target resolver shared by CLI and dreaming with schema, docs, indexing, and regression coverage.

## Reproduction Assessment

Not applicable as a strict bug reproduction, but yes for current behavior: source inspection confirms promotion apply paths still target `workspaceDir/MEMORY.md` with no destination override.

## Solution Assessment

Unclear until maintainers choose the product contract. A configurable resolver is maintainable if flexibility is required, while the active archive-first PR may solve the bootstrap-bloat problem with a narrower storage policy.
