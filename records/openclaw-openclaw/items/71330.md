## Summary

Keep open. Current main still hardcodes short-term promotion writes to `MEMORY.md`, and the related open PRs address contamination or size symptoms without adding a configurable promotion destination.

## What This Changes

The issue requests a memory-core config option for choosing where short-term promotion blocks are written instead of always appending them to `MEMORY.md`.

## Best Possible Solution

Add a memory-core-owned promotion target resolver shared by manual promotion and managed dreaming, with a backward-compatible `MEMORY.md` default, constrained workspace-relative Markdown paths, documented date tokens, safe file creation, and explicit indexing behavior for searchable targets.

## Reproduction Assessment

Yes. Source inspection on current main shows both manual and managed promotion paths call `applyShortTermPromotions`, which resolves and writes only `workspaceDir/MEMORY.md` with no destination override.

## Solution Assessment

Yes. A shared memory-core resolver is the narrow maintainable direction; the adjacent size-cap and contamination PRs reduce symptoms but do not separate promotion storage from curated bootstrap memory.
