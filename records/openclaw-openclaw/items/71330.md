## Summary

Keep open. Current main and v2026.6.6 still write short-term promotions to `workspaceDir/MEMORY.md`; the merged budget/contamination fixes reduce growth and noise, but they do not add a configurable promotion target or define alternate-file indexing/search semantics.

## What This Changes

The issue requests a configurable memory-core promotion target file, including date-templated paths and memory-search visibility for promoted content outside `MEMORY.md`.

## Best Possible Solution

Decide the product/API shape first; if accepted, implement one shared promotion-target resolver with safe path validation, default `MEMORY.md`, docs/schema alignment, indexing semantics, and regression coverage for CLI plus managed dreaming.

## Reproduction Assessment

Yes for current behavior, but not as a strict bug reproduction: source inspection confirms manual and managed short-term promotion still write only to `workspaceDir/MEMORY.md` with no destination override.

## Solution Assessment

Unclear. A configurable target is directionally useful, but the best maintainable solution needs a shared resolver and search/indexing contract rather than a one-off path string in one caller.
