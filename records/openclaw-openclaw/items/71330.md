## Summary

Keep open. Current main still has a single hardcoded promotion destination, with manual and managed dreaming paths both writing to `MEMORY.md`; the adjacent open PRs address bloat or contamination but do not add a configurable target file.

## What This Changes

Add a memory-core configuration option for where short-term promotion blocks are written, defaulting to `MEMORY.md` and supporting alternate searchable Markdown targets.

## Best Possible Solution

Implement a memory-core-owned promotion-target resolver shared by manual promotion and managed dreaming, defaulting to `MEMORY.md`, supporting constrained workspace-relative Markdown paths with documented date tokens, creating missing files with the existing header behavior, and ensuring the resolved target is indexed or explicitly constrained to indexed paths.

## Reproduction Assessment

Yes. Source inspection on current main shows every apply path reaches `applyShortTermPromotions`, which always resolves and writes `workspaceDir/MEMORY.md` with no destination override.

## Solution Assessment

Yes. The requested memory-core config and shared resolver are the narrowest maintainable solution; the adjacent cap and contamination PRs reduce symptoms but do not separate promotion storage from curated bootstrap memory.
