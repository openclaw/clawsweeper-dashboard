## Summary

Keep open. Current main and the latest release still write short-term promotions to `MEMORY.md`; the merged budget and contamination fixes reduce growth/noise but do not add configurable promotion targets or settle alternate-file search/indexing semantics.

## What This Changes

The issue requests configurable memory-core promotion target paths instead of always appending promoted short-term memories to `MEMORY.md`.

## Best Possible Solution

Decide the product/API shape first; if accepted, implement one shared promotion-target resolver with safe path validation, default `MEMORY.md`, docs/schema alignment, indexing semantics, and regression coverage for CLI plus managed dreaming.

## Reproduction Assessment

Yes for current behavior at source level: managed dreaming and manual promotion both call `applyShortTermPromotions`, which still resolves `workspaceDir/MEMORY.md` with no destination override. I did not run a live promotion cycle in this read-only review.

## Solution Assessment

Unclear. A configurable target is directionally useful, but the best maintainable solution needs a shared resolver plus a search/indexing contract rather than a one-off path string in one caller.
