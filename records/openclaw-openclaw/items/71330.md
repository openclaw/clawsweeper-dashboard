## Summary

Keep open. Current main still hardcodes short-term promotion writes to `MEMORY.md`, exposes no promotion target in the manual or managed promotion paths, and the memory-core config/docs do not describe a target-file option. The adjacent open PR #74088 addresses a related size-budget problem, but it does not make the promotion destination configurable.

## What This Changes

The issue requests a configurable memory-core promotion target file so short-term promotions can be written somewhere other than `MEMORY.md` while remaining searchable.

## Best Possible Solution

Implement a memory-core-owned promotion target resolver shared by manual `memory promote --apply` and managed dreaming. It should default to `MEMORY.md`, validate constrained Markdown targets, optionally expand deterministic date tokens if accepted, create missing target files with the existing header behavior, and make the resolved target searchable without requiring users to manually duplicate indexing config.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the writer always resolves `workspaceDir/MEMORY.md`, and both manual CLI and managed dreaming call it without a target override. I did not run a live write because this review was required to stay read-only.

## Solution Assessment

Yes. A memory-core config path is the narrowest maintainable direction because the write destination lives in the promotion writer; existing indexing extras can index other files but cannot change where promotions are written. #74088 is useful adjacent size-budget work, but it does not solve the target-file requirement.
