## Summary

Keep open. Current main still writes short-term promotions only to `MEMORY.md`, exposes no configurable promotion target in the apply options, manual CLI, managed dreaming path, plugin schema, or docs, and the adjacent #74088 work is a size-budget fix rather than a destination-file feature.

## What This Changes

Add a memory-core configuration option that lets short-term memory promotions write to a configurable Markdown target file instead of always appending to `MEMORY.md`.

## Best Possible Solution

Implement a memory-core-owned promotion target resolver used by both manual `memory promote --apply` and managed dreaming. It should default to `MEMORY.md`, validate a constrained Markdown target, expand documented date tokens deterministically, create missing parent files with the existing header behavior, and ensure the resolved target is included in memory search indexing without user-side duplication.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main always resolves `workspaceDir/MEMORY.md`, and both manual CLI and managed dreaming call the writer without a destination override; I kept the review read-only and did not run a live write.

## Solution Assessment

Yes. A memory-core config path shared by the promotion writer, CLI, dreaming, docs, and indexing is the narrowest maintainable solution; #74088 is useful adjacent size-budget work but does not solve the target-file requirement.
