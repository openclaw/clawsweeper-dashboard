## Summary

Keep open. Current main still hardcodes memory-core short-term promotion output to `MEMORY.md`, with no destination override in the apply API, manual CLI path, managed dreaming path, plugin schema, or docs. The related open PRs address adjacent `MEMORY.md` contamination or size-budget symptoms, but they do not implement configurable promotion target files.

## What This Changes

Request configurable memory-core promotion destination files so short-term/deep promotions can be written somewhere other than `MEMORY.md` while remaining searchable.

## Best Possible Solution

Implement the feature inside `memory-core` with one shared promotion-target resolver used by both manual promotion and managed dreaming. It should default to `MEMORY.md`, support documented date tokens, create missing Markdown files with the existing header behavior, ensure the resolved target is indexed/searchable, and update schema, docs, changelog, and focused tests without moving plugin-owned behavior into core.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: any qualifying `applyShortTermPromotions` call resolves and writes `workspaceDir/MEMORY.md`, and both manual CLI and managed dreaming reach that function without a target override. I kept this review read-only and did not run a live write.

## Solution Assessment

Yes. A memory-core-owned target configuration is the narrowest maintainable solution for this request; the adjacent contamination and budget PRs may reduce bloat but do not separate algorithmic promotion output from the bootstrap file or provide configurable destination/indexing behavior.
