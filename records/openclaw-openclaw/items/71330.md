## Summary

The current source still confirms the prior review: this remains a valid memory-core feature request, not a close candidate. The hardcoded MEMORY.md write target is still present, no target option is exposed through the apply API, manual CLI, managed dreaming config, plugin schema, or docs, and the two related open PRs address adjacent MEMORY.md bloat/contamination bugs rather than configurable destination files.

## What This Changes

Add a configurable memory-core short-term promotion target file so promoted memories can be written somewhere other than MEMORY.md while remaining searchable.

## Best Possible Solution

Implement this in memory-core as a single promotion-target resolver shared by manual promote and managed dreaming. It should default to MEMORY.md, accept a constrained Markdown workspace-relative target with documented date tokens, create missing files with the existing header behavior, include the resolved target in memory indexing/search when needed, and keep core generic rather than adding owner-specific behavior outside the plugin.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: the promotion writer always resolves workspaceDir/MEMORY.md, and both manual CLI and managed dreaming call it without any destination override; I kept the review read-only and did not run a live write.

## Solution Assessment

Yes. A memory-core-owned config path shared by the writer, CLI, dreaming scheduler, docs, schema, and indexing is the narrowest maintainable solution; the related open PRs are useful adjacent fixes but do not solve destination-file configurability.
