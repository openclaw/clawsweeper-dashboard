## Summary

Keep open. Current main and v2026.6.8 still write short-term promotions to `MEMORY.md`; merged compaction and contamination fixes reduce the worst symptoms but do not add a configurable target file or define alternate-file search semantics.

## What This Changes

Request to add a memory-core configuration option that routes short-term promotion output to a configurable, optionally date-templated file instead of always appending to `MEMORY.md`.

## Best Possible Solution

Choose the promotion-destination contract first; if accepted, implement one shared resolver with safe workspace-relative validation, default `MEMORY.md`, approved date-token behavior, schema/docs alignment, and memory-search indexing coverage.

## Reproduction Assessment

Yes, at source level: managed dreaming and manual promotion still call a writer that resolves `workspaceDir/MEMORY.md` with no destination override. I did not run a live promotion cycle in this read-only review.

## Solution Assessment

Unclear. A configurable target is directionally useful, but the best maintainable solution needs a shared path resolver plus search/indexing semantics rather than a one-off path string in one caller.
