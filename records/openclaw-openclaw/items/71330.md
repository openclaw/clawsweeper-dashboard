## Summary

Keep open. Current main still hardcodes memory-core short-term promotion writes to `MEMORY.md`, exposes no destination override in the apply API, manual CLI, managed dreaming config, plugin schema, or docs, and the two related open PRs address adjacent bloat/contamination problems rather than configurable promotion target files.

## What This Changes

Request to add a memory-core configuration option that lets short-term promotions write to a configurable Markdown target file, defaulting to `MEMORY.md` and supporting date-token paths while keeping promoted content searchable.

## Best Possible Solution

Implement this inside memory-core with one shared promotion-target resolver used by manual promotion and managed dreaming. The resolver should default to `MEMORY.md`, accept constrained workspace-relative Markdown targets with documented date tokens, create missing files with the existing header behavior, ensure the resolved target is indexed/searchable, and update schema, docs, changelog, and focused tests without moving owner-specific behavior into core.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: any `applyShortTermPromotions` caller writes to `workspaceDir/MEMORY.md`, and both manual CLI and managed dreaming reach that function without a destination override. I kept the review read-only and did not run a live write.

## Solution Assessment

Yes, the requested memory-core-owned target configuration is the narrowest maintainable solution for this specific problem. The adjacent open PRs may reduce `MEMORY.md` growth or contamination, but they do not separate algorithmic promotion output from the bootstrap file or provide configurable destination/indexing behavior.
