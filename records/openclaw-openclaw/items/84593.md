## Summary

Current main now has a dedicated CLI compaction lifecycle that clears external resume state after successful compaction, and explicit or configured session resets rotate CLI bindings. However, it also intentionally documents and tests that provider-owned CLI sessions survive the implicit daily default, so the remaining dispute is a compatibility-sensitive product policy rather than a clearly safe autonomous bug fix.

## What This Changes

Define when long-lived Claude CLI resume bindings expire across OpenClaw session rollovers, ensure compaction cannot leave an overgrown resumed backend conversation, and clean up stale live-model-switch state.

## Best Possible Solution

Keep explicit and configured resets deterministic, preserve successful CLI compaction’s binding invalidation, and have maintainers decide whether the implicit daily exemption remains the default before adding any narrowly tested lifecycle change; avoid a new TTL option unless a concrete state cannot be handled by existing reset and compaction boundaries.

## Reproduction Assessment

Yes for the narrow source behavior: current main intentionally preserves provider-owned CLI bindings across the implicit daily default. No high-confidence current-main reproduction establishes that this behavior alone still causes the reported token overrun or bypasses the newer CLI compaction lifecycle.

## Solution Assessment

No, the suggested combined change is not yet the best solution because it mixes a compatibility policy change, optional TTL configuration, compaction behavior, legacy cleanup, and live-switch finalization. The narrowest maintainable path is to decide the implicit-reset contract first and separately reproduce any remaining compaction failure.
