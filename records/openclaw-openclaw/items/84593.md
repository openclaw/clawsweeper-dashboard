## Summary

Keep open: current main still preserves provider-owned claude-cli resume bindings across the implicit lifecycle path and defers OpenClaw compaction when claude-cli declares native ownership, while the latest issue comment adds live evidence that a manual compaction summary can remain inert behind the same resumed Claude CLI session. The related open work at https://github.com/openclaw/openclaw/issues/80933 and https://github.com/openclaw/openclaw/pull/93198 covers context-budget persistence, not the broader resume-binding invalidation policy here.

## What This Changes

The issue asks for a lifecycle invariant so long-lived claude-cli resume bindings and stale liveModelSwitchPending state cannot survive resets or compaction in ways that bypass OpenClaw context management.

## Best Possible Solution

Maintainers should decide the provider-owned claude-cli continuity contract, then implement a bounded invalidation path that makes OpenClaw compaction or explicit lifecycle expiry clear stale CLI bindings with focused session, compaction, liveModelSwitchPending, and docs coverage.

## Reproduction Assessment

Partially yes: current source and tests clearly reproduce the preserved provider-owned CLI binding and claude-cli no-op compaction surfaces. The exact full live over-budget current-main scenario was not rerun, but the 2026.6.1 comment supplies redacted live evidence.

## Solution Assessment

Unclear: clearing all claude-cli bindings by default is a plausible fix but may be the wrong compatibility tradeoff. The best fix needs maintainer policy on when provider-owned continuity must yield to OpenClaw compaction or lifecycle expiry.
