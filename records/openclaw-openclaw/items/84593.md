## Summary

Keep open: current `main` partially covers explicit reset and CLI compaction paths, but the reported long-lived/default lifecycle gap is not resolved because docs and tests intentionally preserve provider-owned Claude CLI bindings across the implicit daily default and there is no TTL-style invalidation policy.

## What This Changes

The issue asks OpenClaw to clear or invalidate stale Claude CLI resume bindings across new/reset/long-lived session lifecycle boundaries so CLI-backed sessions cannot silently outlive OpenClaw context lifecycle expectations.

## Best Possible Solution

Keep the explicit reset and CLI compaction protections, then make a maintainer decision on provider-owned CLI continuity for implicit daily/long-lived sessions and encode it as a tested lifecycle policy with matching docs and doctor repair behavior.

## Reproduction Assessment

Partially yes: current source and tests show provider-owned Claude CLI bindings are preserved across implicit daily reset boundaries, which is the disputed lifecycle path. No: this read-only pass did not reproduce the exact over-budget live session with compactionCount remaining zero.

## Solution Assessment

Unclear: clearing bindings on explicit `/new`, `/reset`, and explicit reset policies is already the right boundary and is partly implemented. The requested default/TTL invalidation conflicts with current docs and tests, so the best fix needs maintainer policy first rather than an automatic code sweep.
