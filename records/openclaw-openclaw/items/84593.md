## Summary

Keep open: current main and the latest release still preserve provider-owned claude-cli bindings across the implicit long-lived session path, while claude-cli is also marked as native-compaction-owned so OpenClaw can defer compaction instead of clearing the stale resume binding. Explicit reset cleanup is partly implemented, but the broader invalidation policy reported here remains unresolved and is not superseded by the narrower contextTokens work.

## What This Changes

The issue asks OpenClaw to prevent stale claude-cli resume bindings and stuck live model-switch state from surviving session reset, rollover, and compaction boundaries unless backend continuity is explicitly preserved.

## Best Possible Solution

Maintainers should decide the provider-owned claude-cli continuity contract, then implement a bounded invalidation path that clears stale CLI bindings when OpenClaw compaction or lifecycle expiry requires a fresh backend session, with targeted session, compaction, liveModelSwitchPending, and docs coverage.

## Reproduction Assessment

Yes at source level: current main explicitly preserves provider-owned claude-cli bindings across implicit stale boundaries and tests that behavior, while claude-cli compaction can still no-op through ownsNativeCompaction. I did not rerun the full live OAuth over-budget scenario, but the June 10, 2026 issue comment provides redacted live evidence on 2026.6.1.

## Solution Assessment

Unclear: clearing claude-cli bindings everywhere would address the stale resume symptom but may be too broad for users relying on backend continuity. The best fix needs maintainer policy on when provider-owned continuity yields to OpenClaw compaction or lifecycle expiry.
