## Summary

Keep open: current main and the latest release still defer ordinary claude-cli auto-compaction to backend-owned native compaction, and manual `/compact` still records compaction without clearing stored CLI resume bindings; explicit reset cleanup only covers part of the reported lifecycle gap.

## What This Changes

The issue asks OpenClaw to invalidate stale claude-cli resume bindings across reset, compaction, and long-lived session lifecycle boundaries so new OpenClaw context does not resume an old Claude CLI conversation.

## Best Possible Solution

Keep the documented provider-owned CLI continuity only where intentional, and add a maintainer-approved invalidation path that clears claude-cli resume bindings after OpenClaw-managed compaction or explicit lifecycle reset so the next turn uses the current OpenClaw transcript.

## Reproduction Assessment

Yes at source level: current main still has the backend-owned claude-cli auto-compaction no-op and manual compaction does not clear persisted CLI bindings. The issue discussion also supplies redacted live 2026.6.1 evidence for the same stale resume path.

## Solution Assessment

Unclear: the requested invariant is directionally right for compaction/reset boundaries, but a blanket TTL or implicit daily reset change would conflict with the current documented provider-owned CLI continuity behavior. The best fix needs a maintainer policy decision before changing defaults.
