## Summary

Keep open: current main has explicit reset and CLI compaction protections, but the reported long-lived/default lifecycle question remains because implicit daily defaults intentionally preserve provider-owned CLI bindings and there is no TTL-style invalidation policy.

## What This Changes

The issue requests a lifecycle policy so long-lived Claude CLI-backed sessions do not carry stale CLI resume bindings or live model switch state across new/reset/expiry boundaries unless explicitly preserved.

## Best Possible Solution

Keep the explicit reset and compaction clearing behavior, then define a maintainer-approved provider-owned CLI binding expiration policy for implicit long-lived sessions with matching docs, doctor repair behavior, tests, and liveModelSwitchPending cleanup semantics.

## Reproduction Assessment

Partially: current source and tests show provider-owned Claude CLI bindings are preserved across implicit daily default boundaries. No high-confidence reproduction of the exact over-budget live session with compactionCount remaining zero was run in this read-only pass.

## Solution Assessment

Unclear: clearing bindings on explicit /new, /reset, and successful CLI compaction is already implemented. The remaining default/TTL invalidation behavior conflicts with current docs and tests, so maintainers should choose the lifecycle policy before implementation.
