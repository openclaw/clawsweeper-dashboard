## Summary

Keep open: current main and v2026.5.22 still make duplicate suppression visible only after a successful message-tool completion, so the reported failed-send then retry-before-success order can still emit the automatic final text as a duplicate.

## What This Changes

The issue asks OpenClaw to make message-tool duplicate suppression retry-aware when a send fails and is retried in the same turn.

## Best Possible Solution

Track same-route retry evidence through the message-tool lifecycle so duplicate final text can see an in-flight or later-successful retry, while preserving fallback delivery when every send attempt fails.

## Reproduction Assessment

Yes, source inspection provides a high-confidence reproduction path: fail a message-tool send, start a retry in the same turn, then emit message_end before the retry commits successful evidence. I did not run a live Telegram/proxy reproduction.

## Solution Assessment

Yes, the best fix shape is narrow lifecycle handling rather than a blanket suppression toggle. The implementation should expose retry-aware same-route evidence to duplicate suppression without marking all failed sends as delivered.
