## Summary

Keep open: current main and latest release still have the reported fail-then-retry dedupe gap, and the adjacent open PRs are related delivery work rather than merged implementation for this exact source path.

## What This Changes

The issue reports duplicate messaging-channel delivery when a failed `message` tool call is retried in the same agent turn and the final assistant text is also delivered.

## Best Possible Solution

Track same-route retry evidence through the message-tool lifecycle so final duplicate suppression can see a retry that is in flight or has succeeded, while preserving fallback delivery when all send attempts fail.

## Reproduction Assessment

Yes, source-reproducible: failed message-tool completion deletes pending evidence, retry start records only pending state, and `message_end` checks committed successful-send text before emitting. I did not run a live Telegram/proxy reproduction.

## Solution Assessment

Yes, the right fix shape is narrow lifecycle handling, not a broad suppression toggle: make retry-aware evidence visible to duplicate suppression while keeping the all-failed-attempt fallback reply.
