## Summary

Keep open: current main still has the reported source-level gap, because message-tool pending evidence is tracked but final reply suppression only uses committed successful delivery evidence.

## What This Changes

The issue reports duplicate outbound messaging when a failed message tool call is retried in the same agent turn and the assistant text reply is also delivered.

## Best Possible Solution

Implement pending-send-aware final reply handling that holds matching same-source final replies while a send is in flight, suppresses them after confirmed delivery, and releases them on failure or non-delivery.

## Reproduction Assessment

Yes, at source level: retry `tool_execution_start` records pending evidence, but block and final reply suppression only consult committed successful evidence. I did not run a live Telegram retry replay.

## Solution Assessment

No to the raw proposed fix of recording pending sends as already sent. The safer solution needs a deferred pending-reply decision so successful retries suppress duplicates without losing fallback replies when the retry fails.
