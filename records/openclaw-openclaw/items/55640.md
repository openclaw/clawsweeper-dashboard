## Summary

Keep open. Current main still has the failed-send/retry gap: failed `message` tool sends discard pending evidence, in-flight retry evidence is not checked by `message_end` or final payload dedupe, and related items cover different delivery or final-commentary paths.

## What This Changes

The issue reports duplicate messaging-channel delivery when a failed `message` tool send is retried in the same turn and the assistant text reply is delivered too.

## Best Possible Solution

Fix the shared embedded messaging-tool lifecycle so same-route, same-content in-flight retries suppress duplicate `message_end` and final payload delivery while preserving one fallback reply if all send attempts fail.

## Reproduction Assessment

Yes, with a high-confidence source reproduction path: a failed `tool_execution_end` deletes pending evidence, a retry starts with the same target/text, then `message_end` before retry success emits the text because only committed successful sends are checked. I did not run a live Telegram retry in this read-only review.

## Solution Assessment

Yes. The right fix is a narrow lifecycle change that tracks in-flight same-route/same-content retry evidence for suppression without permanently treating failed sends as delivered.
