## Summary

Keep open: current main and the latest release still use committed-success-only message-tool evidence, so the reported retry timing gap remains source-reproducible. The original pending-as-sent fix would risk dropping the only user-visible fallback reply, so this needs maintainer choice on hold/release semantics before a repair PR.

## What This Changes

The issue reports duplicate messaging-channel delivery when a failed `message` tool call is retried in the same turn and a matching text reply flushes before the retry commits successful delivery evidence.

## Best Possible Solution

Implement a hold-and-release reply path for same-route in-flight message sends only after maintainers decide the turn-end release default and route match key.

## Reproduction Assessment

Yes by source inspection: a failed first send clears pending evidence without committing sent evidence, while the matching text reply suppression paths only consult committed successful sends. I did not run a live Telegram retry replay.

## Solution Assessment

No for the issue's original proposed fix: recording pending sends as already sent would preserve duplicate suppression but can drop the user's only reply when the send never succeeds. The safer direction is deferred hold-and-release for matching in-flight sends.
