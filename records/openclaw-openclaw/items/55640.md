## Summary

Keep open. Current main still has the failed-send/retry suppression gap: message-tool sends are pending until success, failed sends delete that pending evidence, and both message_end plus final payload dedupe only see committed successful send state.

## What This Changes

Fix duplicate messaging-channel delivery when a failed `message` tool send is retried in the same assistant turn and the final text reply is delivered too.

## Best Possible Solution

Track enough in-flight same-route, same-content message-tool retry evidence to suppress duplicate message_end/final delivery, while preserving one fallback reply if all send attempts fail.

## Reproduction Assessment

Yes, at source level. A failed tool_execution_end deletes pending send evidence, a retry start records only pending state, and message_end/final dedupe cannot see that in-flight retry because they check only committed successful sends.

## Solution Assessment

Yes, a narrow lifecycle fix is the best way to solve this. The repair should extend the existing embedded messaging-tool dedupe state rather than changing channel policy or disabling duplicate suppression.
