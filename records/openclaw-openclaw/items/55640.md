## Summary

Keep open: current main still has a source-level gap where retry-start evidence is tracked, but block-reply and final-payload suppression only consult committed successful message-tool delivery evidence.

## What This Changes

The issue asks OpenClaw to avoid duplicate visible delivery when a failed message-tool send is retried in the same turn and the retry overlaps an automatic final reply.

## Best Possible Solution

Model failed, pending-retry, and committed message-tool delivery separately, then suppress same-route duplicate final text only when a retry is in flight or committed while preserving fallback replies for failed-only sends.

## Reproduction Assessment

Yes, by source inspection rather than a fresh live run. If a retry is pending or the assistant final text is processed before retry delivery commits, current suppression only sees empty committed message-tool evidence and can deliver the automatic reply.

## Solution Assessment

Unclear as proposed. Recording every pending send would fix the duplicate path, but the current code documents a real fallback-loss risk after failed delivery, so the best fix should distinguish pending retry from failed-only sends.
