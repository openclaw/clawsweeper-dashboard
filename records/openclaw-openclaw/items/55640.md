## Summary

Current main still keeps pending message-tool sends separate from committed delivery evidence, discards failed sends, and suppresses both streaming and final replies only from committed successful sends. The reported failed-send then in-flight retry window remains source-reproducible and should stay open for a focused fix.

## What This Changes

Fix duplicate outbound delivery when a failed `message` tool send is retried in the same turn before the assistant text reply is delivered.

## Best Possible Solution

Add a shared messaging-tool lifecycle state that suppresses same-route, same-content in-flight retry duplicates while still delivering a fallback if all send attempts fail.

## Reproduction Assessment

Yes, by source inspection: after a failed message tool end clears pending state, a same-target retry can be pending when `message_end` runs, and current main emits the assistant text because suppression only sees committed successful sends. I did not run a live Telegram turn in this read-only review.

## Solution Assessment

Yes, a narrow lifecycle fix in embedded message-tool bookkeeping and reply payload dedupe is the right direction. It should avoid treating every pending send as delivered forever, because a final fallback still needs to go out when every attempt fails.
