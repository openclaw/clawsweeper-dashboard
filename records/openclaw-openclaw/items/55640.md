## Summary

Keep open: current main still has the failed-send/retry suppression gap because failed message-tool sends are discarded, in-flight retry state is not consulted by duplicate suppression, and no merged fix or closing PR covers this exact path.

## What This Changes

The issue reports duplicate messaging delivery when a failed `message` tool send is retried in the same assistant turn and the final text is also auto-delivered.

## Best Possible Solution

Track same-route, same-content in-flight retry evidence for duplicate suppression while preserving one fallback reply when all message-tool attempts fail.

## Reproduction Assessment

Yes, source-reproducible: failed `tool_execution_end` removes pending message-tool evidence, and the `message_end` plus final payload dedupe paths only inspect committed successful sends. I did not run a live Telegram/proxy reproduction in this read-only review.

## Solution Assessment

Yes, a narrow lifecycle fix is the right direction. The repair should extend existing messaging-tool dedupe state for in-flight retries rather than changing channel policy or globally suppressing failed-send fallbacks.
