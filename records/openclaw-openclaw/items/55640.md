## Summary

Keep open: current main and the latest release still have the pending-vs-committed timing gap, and the safe repair needs a maintainer decision on hold-and-release fallback semantics before implementation.

## What This Changes

The issue reports duplicate Telegram delivery when a failed `message` tool call is retried in the same agent turn before suppression has committed successful-send evidence.

## Best Possible Solution

Implement same-route hold-and-release for in-flight message-tool sends after maintainers choose release-by-default behavior and the exact route/text match key.

## Reproduction Assessment

Yes by source inspection: a failed message-tool send clears pending evidence without committing sent evidence, while retry-time block/final reply suppression checks only successful committed sends. I did not run a live Telegram retry replay.

## Solution Assessment

No for the original proposed pending-as-sent fix: current tests intentionally preserve the fallback reply after message-tool errors. The safer design is deferred hold-and-release for matching in-flight sends.
