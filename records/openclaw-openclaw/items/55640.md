## Summary

Keep open: current main still has the failed-send/retry suppression gap, and the related duplicate-delivery PR is open rather than merged. The source path still discards failed message-tool evidence and only dedupes final replies against committed successful sends.

## What This Changes

The issue asks OpenClaw to prevent duplicate channel delivery when a failed `message` tool send is retried in the same turn and the final assistant text is emitted before the retry finishes.

## Best Possible Solution

Track same-route pending retry delivery evidence through `message_end` and final payload dedupe, preserve automatic fallback when all message-tool sends fail, and land or supersede the related open PR with focused regression coverage.

## Reproduction Assessment

Yes. The current-main source path shows failed `tool_execution_end` deletes pending message-tool evidence, while `message_end` and final payload dedupe only inspect committed successful sends; I did not run a live Telegram/proxy reproduction.

## Solution Assessment

Yes. A narrow lifecycle fix is the maintainable path: keep same-route pending retry evidence available for duplicate suppression without globally suppressing the fallback reply when every send attempt fails.
