## Summary

Keep open. Current main still has the same failed-send/retry gap: pending message-tool sends are recorded, failed sends are discarded, and both message_end block replies and final payload dedupe consult only committed successful sends, so the same-turn retry duplicate remains source-reproducible. The adjacent open PR #61183 is about rewritten tool arguments for #61157, and the closed #76915 report describes the opposite dropped-final-reply behavior rather than a fix for this issue.

## What This Changes

Suppress duplicate messaging-channel auto replies when a message tool send fails and is retried in the same agent turn.

## Best Possible Solution

Fix the shared embedded messaging-tool lifecycle so same-target same-content in-flight retries suppress duplicate message_end/final delivery only while preserving fallback delivery when no retry succeeds.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is: failed message tool end clears pending state, a retry starts with the same target/text, message_end fires before the retry succeeds, and current main emits the text because suppression only sees committed successful sends.

## Solution Assessment

Yes, a narrow shared lifecycle fix is the best direction, but it should not simply treat all pending sends as delivered. The safer fix is to track in-flight retry evidence for same-route/same-content suppression and release a fallback if every send attempt fails.

## Repair Work Prompt

Fix openclaw/openclaw#55640. Current main records message sends in pendingMessagingTargets/pendingMessagingTexts at tool_execution_start, discards them on failed tool_execution_end, and message_end/final reply dedupe only considers committed successful sends. Implement a narrow shared agent/reply runtime fix so a failed message tool send followed by an in-flight retry with the same route and visible content does not also auto-deliver the assistant text in the same turn. Preserve fallback behavior: if no retry ultimately succeeds, the user must still receive one fallback reply. Do not make this Telegram-specific, do not globally suppress unrelated post-tool commentary, and do not change provider/channel policy. Add focused regression coverage for failed first send + retry start + message_end before retry success, failed first send + retry failure fallback, successful single send duplicate suppression, and single failed send fallback. Add a user-facing CHANGELOG.md Fixes entry for the messaging duplicate fix. Validate with the targeted agent/reply tests first, then run the changed gate in Testbox before handoff.
