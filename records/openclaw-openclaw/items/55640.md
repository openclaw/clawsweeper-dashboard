## Summary

Keep this issue open. Current main still has the failed-send/retry gap: message tool sends are tracked as pending at tool start, failed sends are removed without becoming suppression state, and both message-end block replies and final payload suppression consult only committed successful sends. The prior ClawSweeper comment correctly identified the same in-flight retry window, and the adjacent open PR #61183 is about rewritten tool args rather than this retry suppression bug.

## What This Changes

The issue reports duplicate visible delivery when a failed messaging tool send is retried in the same assistant turn and the automatic text reply is emitted before the retry result commits.

## Best Possible Solution

Keep the issue open and fix the embedded messaging-tool lifecycle: track same-turn in-flight retry sends separately from committed successful deliveries, make both message_end block replies and final payload suppression consider same-target same-text in-flight retries, and preserve fallback delivery when no send attempt ultimately succeeds.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to emit a failed message tool result, then a retry tool_execution_start with the same target/text, then message_end before the retry tool_execution_end; current main still emits the block/final text because suppression sees only committed successful sends.

## Solution Assessment

Yes, a narrow embedded lifecycle fix is the best direction. This should not be Telegram-specific; it belongs in the shared message-tool bookkeeping and reply suppression boundary with focused tests for failed first send, in-flight retry, message_end, and retry success/failure outcomes.

## Repair Work Prompt

Fix the failed messaging-tool send retry duplicate-delivery bug described in https://github.com/openclaw/openclaw/issues/55640. Work in the shared embedded message-tool lifecycle, not Telegram-specific code. Current main records pending message sends in `src/agents/pi-embedded-subscribe.handlers.tools.ts`, deletes pending entries on failed tool results, and only exposes committed successful sends to message_end/final-payload suppression. Add a retry-aware/in-flight suppression state that lets same-turn, same-target, same-text retries suppress automatic block replies and final payloads while preserving fallback delivery if no send ever succeeds. Keep provider/account/thread target semantics intact and do not suppress cross-target sends. Coordinate mentally with adjacent PR #61183, which touches adjusted message-tool bookkeeping but does not solve this issue. Add focused regressions for: first `message` send fails, retry `tool_execution_start` records the same target/text, `message_end` arrives before retry result, block reply is suppressed, retry success commits normally; also cover retry failure/no-success fallback so the only visible reply is not lost. Changelog is expected because this is user-facing duplicate delivery behavior. Do not broaden text-leak policy from #25592 beyond this narrow retry lifecycle.
