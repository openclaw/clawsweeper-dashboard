## Summary

Keep open: current main still has a source-reproducible in-flight retry gap. The subscription records message-tool sends as pending at tool start, but duplicate suppression only consults committed successful sends, so a retry `message(action=send)` followed by assistant text before the retry result commits can still deliver both.

## What This Changes

The issue requests preventing duplicate channel delivery when a failed message tool send is retried and final assistant text is emitted in the same turn.

## Best Possible Solution

Add in-flight message-tool reply handling that holds matching same-source final text until the pending send resolves, suppresses it on success, and releases it on failure, dry-run, or explicit-route sends.

## Reproduction Assessment

Yes by source inspection, not by a live Telegram run. The path is `tool_execution_start` for a retry send, assistant `message_end` with matching text while the send is still pending, then a successful `tool_execution_end`; current suppression ignores that pending state.

## Solution Assessment

No to the raw proposed fix of treating pending sends as already sent. The safer fix is to defer matching final replies while a message send is pending, then decide after the tool result so failed retries still produce a visible fallback.
