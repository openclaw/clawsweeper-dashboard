## Summary

Keep open: current main still has the source-level in-flight retry gap. The handler records pending message-tool sends at start, but duplicate suppression and route matching only consult committed successful sends, so a retry send followed by assistant final text before the retry result commits can still deliver both.

## What This Changes

The issue asks OpenClaw to prevent duplicate channel delivery when a failed `message` tool call is retried in the same turn and the assistant final text is emitted before the retry result commits.

## Best Possible Solution

Add in-flight message-tool reply handling that holds matching same-source final text while a send is pending, suppresses it after confirmed delivery, and releases it on failure, dry-run, or explicit-route sends.

## Reproduction Assessment

Yes, by source inspection rather than a live Telegram run. The reproducible path is retry `tool_execution_start`, assistant `message_end` with matching text while the retry is pending, then successful `tool_execution_end`; current suppression ignores pending state.

## Solution Assessment

No to the raw proposed fix of recording pending sends as already sent. The safer solution is a deferred pending-reply decision so successful retries suppress duplicates without losing fallback replies when the retry fails.
