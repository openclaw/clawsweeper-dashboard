## Summary

Keep open. Current main still has the failed-send/retry suppression gap: failed message-tool sends remove pending delivery evidence, in-flight retry evidence is not consulted by message_end or final payload dedupe, and the related closed item covers a different successful-send commentary false positive.

## What This Changes

The issue reports duplicate messaging-channel delivery when a failed message tool send is retried in the same turn and assistant text is also emitted.

## Best Possible Solution

Fix the shared embedded messaging-tool lifecycle so same-route, same-content in-flight retries suppress duplicate message_end/final delivery while preserving one fallback delivery if all send attempts fail.

## Reproduction Assessment

Yes. The source reproduction is: failed tool_execution_end deletes pending send evidence, a retry starts with the same target/text, then message_end before retry success emits the text because only committed successful sends are checked.

## Solution Assessment

Yes, a narrow lifecycle fix is the right direction, but it should track in-flight same-route/same-content retry evidence rather than treating every pending send as delivered forever.
