## Summary

Keep open: current main and the latest release still have a source-level pending-send gap where retry-start evidence is tracked, but final reply suppression and route dedupe only consult committed successful message-tool evidence.

## What This Changes

The issue reports duplicate messaging-channel delivery when a failed message tool call is retried and followed by assistant text in the same turn.

## Best Possible Solution

Implement pending-send aware final-reply handling that holds matching same-source text while a send is in flight, suppresses it after confirmed delivery, and releases it on failure or non-delivery.

## Reproduction Assessment

Yes, at source level. A retry `tool_execution_start` records pending evidence, but `message_end` and block-reply suppression only consult committed successful evidence until `tool_execution_end`.

## Solution Assessment

No to the raw proposed fix of recording pending sends as already sent. The safer solution is a deferred pending-reply decision so successful retries suppress duplicates without losing fallback replies when the retry fails.
