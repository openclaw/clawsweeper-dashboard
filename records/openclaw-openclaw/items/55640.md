## Summary

Keep open: current main and the latest release still have the retry/final-reply dedupe gap. The related duplicate-delivery PR is still open, so it is a candidate path rather than implemented evidence.

## What This Changes

The issue reports duplicate visible message delivery when a failed `message` tool send is retried in the same turn and the final assistant text is also delivered.

## Best Possible Solution

Track same-route pending retry evidence through `message_end` and final payload dedupe while preserving fallback delivery when every send attempt fails, then land or supersede the related open PR with focused regression coverage.

## Reproduction Assessment

Yes, source-reproducible: a failed message-tool end deletes pending evidence, a retry start records only pending state, and `message_end` checks only committed successful-send text before emitting. I did not run a live Telegram/proxy reproduction.

## Solution Assessment

Yes. A narrow lifecycle fix is the maintainable path: keep same-route pending retry evidence available for duplicate suppression without globally suppressing fallback replies when every send attempt fails.
