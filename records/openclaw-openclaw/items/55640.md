## Summary

Keep open: current main and the latest release still have the committed-only suppression timing gap, and the safe repair needs a maintainer decision on hold-and-release fallback semantics before implementation.

## What This Changes

The issue reports duplicate channel delivery when a failed `message` tool send is retried in the same turn and the assistant text reply is also delivered.

## Best Possible Solution

Implement same-route deferred hold-and-release for in-flight message-tool sends after maintainers choose release-by-default behavior and the route/text match key.

## Reproduction Assessment

Yes, by source inspection: failed message-tool sends clear pending evidence without committing sent evidence, while retry-time block/final suppression checks only committed successful sends. I did not run a live Telegram retry replay.

## Solution Assessment

No for the original proposed pending-as-sent fix: current tests intentionally preserve fallback replies after message-tool errors. The maintainable direction is deferred hold-and-release for matching in-flight sends, with explicit release behavior if delivery never succeeds.
