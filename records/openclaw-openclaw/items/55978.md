## Summary

Keep open: current main still has the hardcoded Cron workspace form offset, and this PR is a narrow plausible fix. It should not be closed as implemented because the fix is not on main, the branch is dirty, and https://github.com/openclaw/openclaw/pull/47582 proposes a competing normal-flow UX for the same panel.

## What This Changes

The PR changes the Cron workspace form from a hardcoded `74px` sticky offset to `top: 0` with `--shell-topbar-height` in CSS, plus a stylesheet assertion and changelog entry.

## Best Possible Solution

Choose the intended desktop Cron editor behavior; land this variable-backed sticky fix after rebase if the editor should stay pinned, or prefer the normal-flow alternative if sticky behavior should go away.

## Reproduction Assessment

Yes, at source-and-proof level: current main still has the `74px` sticky offset while the shell uses `--shell-topbar-height: 52px`, and the PR body includes before/after recordings. I did not run a fresh browser reproduction in this read-only review.

## Solution Assessment

Yes for the reported offset mismatch: `top: 0` relative to the content scroller plus `--shell-topbar-height` in `max-height` is the narrow sticky-editor fix. The broader UX choice is unresolved because another open PR removes sticky behavior instead.
