## Summary

Keep open: current main still has the hardcoded Cron form sticky offset, and this PR is a focused, plausible bug fix, but it overlaps an open normal-flow layout alternative that needs maintainer choice before merge.

## What This Changes

The PR changes the Control UI Cron workspace form sticky CSS from a hardcoded 74px offset to top: 0 with a max-height based on --shell-topbar-height, and adds a CSS regression test plus changelog entry.

## Best Possible Solution

Define the desktop Cron form layout contract, then either land this variable-backed sticky offset fix after rebase or close it in favor of the normal-flow layout PR.

## Reproduction Assessment

Yes, at source level: current main renders the Cron form with .cron-workspace-form, and that rule still hardcodes top: 74px while the shell exposes --shell-topbar-height. I did not run a live browser reproduction, but the PR body includes before/after recordings of the visible behavior.

## Solution Assessment

Unclear as a product direction until maintainers choose the desktop layout contract. If the intended UX is to keep the Cron editor sticky, this PR is the narrowest maintainable fix for the offset mismatch.
