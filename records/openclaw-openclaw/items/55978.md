## Summary

Keep open: current main still has the hardcoded Cron form offset, and this PR is a narrow plausible fix with sufficient after-fix recording proof. The remaining blocker is maintainer UX judgment because an open normal-flow alternative changes the same desktop panel behavior in a different direction.

## What This Changes

The PR changes the Cron workspace form CSS from a hardcoded sticky offset to `top: 0` with a `--shell-topbar-height` based max-height, adds a CSS regression assertion, and adds a changelog entry.

## Best Possible Solution

Choose one intended desktop Cron editor behavior: land the variable-backed sticky-offset fix if the editor should stay pinned, or close this path in favor of the normal-flow alternative if the inner scroller should be removed.

## Reproduction Assessment

Yes, at source-and-proof level: current main still applies the `74px` sticky offset while the shell topbar variable is `52px`, and the PR provides before/after `/cron` recordings. I did not run a fresh live browser reproduction in this read-only review.

## Solution Assessment

Yes for the reported offset mismatch: `top: 0` relative to the content scroller plus `--shell-topbar-height` in the height calculation is the narrowest sticky-editor fix. The broader sticky versus normal-flow UX choice remains unclear because https://github.com/openclaw/openclaw/pull/47582 targets the same panel differently.
