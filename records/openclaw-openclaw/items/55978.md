## Summary

Keep open: current main still has the stale Cron form sticky offset, and this PR is a narrow implementation with after-fix recording proof. It should not be closed as cleanup; the remaining decision is whether maintainers want to preserve sticky behavior with this offset fix or choose the open normal-flow alternative.

## What This Changes

The PR replaces the Cron workspace form's hardcoded sticky offset with `top: 0` and a `--shell-topbar-height` max-height calculation, adds a CSS assertion, and adds a changelog entry.

## Best Possible Solution

Define the desktop Cron editor layout contract, then land this PR if the editor should remain sticky, or close it in favor of the normal-flow alternative if that is the intended UX.

## Reproduction Assessment

Yes, at source level: current main renders `.cron-workspace-form` with a hardcoded `74px` sticky offset while the shell topbar variable is `52px`, and the PR body provides concrete `/cron` scroll steps plus before/after recordings. I did not run a live browser reproduction in this read-only pass.

## Solution Assessment

Yes if maintainers want to preserve sticky behavior: using `--shell-topbar-height` removes the duplicated stale layout constant with a small CSS change. It remains a product/UX choice because https://github.com/openclaw/openclaw/pull/47582 proposes normal document flow instead.
