## Summary

This PR is a plausible narrow bug fix, and current main still has the hardcoded Cron form sticky offset. I do not see a blocking correctness issue in the patch, but it should stay open for maintainer layout judgment because https://github.com/openclaw/openclaw/pull/47582 proposes the opposite normal-flow UX on the same panel, and the branch needs current-main integration before landing.

## What This Changes

This PR changes the Control UI Cron form sticky CSS to use `top: 0` plus `var(--shell-topbar-height)` in `max-height`, adds a CSS regression test, and adds a changelog entry.

## Best Possible Solution

If maintainers want to keep the desktop Cron editor sticky, rebase this PR onto current main and land the variable-backed offset fix; otherwise choose the normal-flow direction in https://github.com/openclaw/openclaw/pull/47582 and close this PR.

## Reproduction Assessment

Yes, at source level: current main renders the Cron form with `.cron-workspace-form`, and that rule still hardcodes `top: 74px` while the shell exposes `--shell-topbar-height: 52px`. I did not run a live browser repro, but the PR body also includes before/after recordings for the visual behavior.

## Solution Assessment

Yes, if the intended desktop UX is to keep the Cron editor pinned: using the existing shell CSS variable is the narrowest maintainable fix. If maintainers prefer the panel to scroll in normal document flow, https://github.com/openclaw/openclaw/pull/47582 is the competing solution to choose instead.
