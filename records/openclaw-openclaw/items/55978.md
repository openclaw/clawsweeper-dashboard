Keep open: current main still has the stale hardcoded Cron form sticky offset, and this PR is a narrow patch with no blocking correctness finding. The remaining blocker is maintainer UX direction because the related normal-flow PR changes the same Cron form behavior in a different direction.

## What This Changes

The PR updates the Cron workspace form sticky CSS from hardcoded `74px` values to `top: 0` plus `--shell-topbar-height`, adds a CSS assertion test, and adds a changelog entry.

## Best Possible Solution

Keep one intended desktop Cron editor behavior: land the variable-backed sticky-offset fix with focused browser verification if the editor should stay pinned, or close this path in favor of the normal-flow alternative if the inner scroller should be removed.

## Reproduction Assessment

Yes, at source-and-proof level: current main still applies the `74px` sticky offset while the shell topbar variable is `52px`, and the PR provides before/after `/cron` recordings. I did not run a fresh live browser reproduction.

Yes for the reported offset mismatch: `top: 0` relative to the content scroller plus `--shell-topbar-height` in the height calculation is the narrowest fix. The broader normal-flow versus sticky UX choice remains unclear because https://github.com/openclaw/openclaw/pull/47582 targets the same panel differently.
