## Summary

Keep open: current main still has the stale Cron form sticky offset and this PR is a plausible narrow fix, but it needs maintainer UX choice because an open related PR proposes the opposite normal-flow layout, and the branch is currently conflicting with main.

## What This Changes

This PR replaces the Cron workspace form’s hardcoded desktop sticky offset with `top: 0` plus a shell topbar CSS-variable max-height, adds a CSS assertion, and adds a changelog bullet.

## Best Possible Solution

Define the intended desktop Cron editor layout, then land this PR if the editor should remain pinned, or close it in favor of the normal-flow alternative if that is the preferred UX.

## Reproduction Assessment

Yes, at source level: current main renders `.cron-workspace-form` with a hardcoded `74px` sticky offset while the shell topbar variable is `52px`, and the PR body provides concrete `/cron` scroll steps plus before/after recordings. I did not run a live browser reproduction in this read-only pass.

## Solution Assessment

Yes if maintainers want to preserve sticky behavior: using `--shell-topbar-height` removes the duplicated stale layout constant with a small CSS change. The final path is still a maintainer UX choice because https://github.com/openclaw/openclaw/pull/47582 proposes normal document flow instead.
