## Best Possible Solution

Land the audit-boundary alignment after correcting the release note to point users to `openclaw security audit`, while preserving the source PR context and @menhguin credit.

## Reproduction Assessment

Yes, source-reproducible. A `workspace/skills/linked/SKILL.md` symlink whose target is inside the workspace but outside `workspace/skills` passes current main's audit root check while the loader rejects it against the skills-root boundary.

## Solution Assessment

Mostly yes. Matching the audit containment root to the loader's resolved `workspace/skills` root is the narrow maintainable fix; the remaining safer adjustment is to correct the changelog command wording.
