The PR changes the workspace skill symlink audit to check `SKILL.md` realpaths against the workspace `skills` directory, updates audit/docs/changelog copy, and adds regression coverage for inside-workspace/outside-skills symlinks.

## Best Possible Solution

Yes. A high-confidence reproduction is a workspace `skills/linked/SKILL.md` symlink whose target is inside the workspace but outside `workspace/skills`; current main’s audit accepts it while the loader rejects that same shape.

## Solution Assessment

Mostly yes. Matching the audit containment root to the loader’s resolved `workspace/skills` root is the narrow maintainable fix; the safer adjustment is to correct the changelog command from `openclaw doctor` to `openclaw security audit`.
