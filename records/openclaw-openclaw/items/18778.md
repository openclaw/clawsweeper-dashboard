## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Rebase Activity config onto the Canvas plugin:** `src/config/types.gateway.ts:39-46`
  - body: Current main resolves Canvas host settings from `plugins.entries.canvas.config.host` and only treats root `canvasHost` as legacy migration input. Adding `canvasHost.activity` here creates a second config source that will drift on upgrades; put the Activity settings under the Canvas plugin owner instead.
  - confidence: 0.94
- **[P1] Require a token before bypassing Gateway auth:** `src/gateway/a2ui-activity.ts:34-35`
  - body: When Activity mode is enabled and no token is configured, this returns `true`, and `server-http.ts` accepts the A2UI request before normal Gateway auth. That makes public A2UI access fail open for any deployment that enables the bridge without a secret.
  - confidence: 0.96
- **[P1] Scope A2UI pushes to one Activity session:** `src/gateway/a2ui-activity.ts:73-75`
  - body: The hub stores a single `lastJsonl` payload and broadcasts every push/reset to every connected Activity client. A second Activity can receive another session's canvas state, so the bridge needs per-session or per-launch scoping before merge.
  - confidence: 0.9
- **[P2] Keep Activity tokens out of WebSocket URLs:** `src/canvas-host/a2ui/index.html:325-326`
  - body: The browser bridge copies `activityToken` into the WebSocket query string and falls back to `ws://` on HTTP pages. That leaks a bearer-style secret through URLs and possibly cleartext transport; use a safer server-issued cookie or exchange and require TLS for tokenized Activity access.
  - confidence: 0.86
