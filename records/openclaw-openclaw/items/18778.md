  - body: When `activity.enabled` is true and `activity.token` is unset or blank, this returns `true`; `authorizeCanvasRequest` then accepts A2UI HTTP/WS before bearer or node-capability auth. That makes public A2UI access unauthenticated for a one-line config mistake.

  - body: The hub stores a single `lastJsonl` and broadcasts each `canvas.a2ui.pushJSONL` to all connected Activity clients. Without a node/session key in the hub, one conversation's canvas stream can be delivered to unrelated Discord Activity clients.
  - confidence: 0.91
- **[P2] Stop sending Activity tokens in WebSocket URLs:** `src/canvas-host/a2ui/index.html:321-325`
  - body: The browser bridge copies `activityToken` into the WebSocket query string and uses `ws://` on HTTP pages. That leaks the token through URL logging/history and can send it in cleartext on misconfigured deployments.
  - confidence: 0.9
- **[P1] Rebase launch handling into the Discord plugin:** `src/discord/components.ts:35-60`

## Security Review

Status: needs_attention

Summary: The diff introduces concrete security-boundary concerns around tokenless A2UI access, URL token transport, and unbounded Activity fanout.

Concerns:

- **[high] Tokenless Activity mode bypasses Gateway auth:** `src/gateway/a2ui-activity.ts:35`
  - body: With `activity.enabled=true` and no token, `isA2uiActivityAccessAllowed` returns true and Gateway auth accepts the request before normal bearer or node-capability checks.
