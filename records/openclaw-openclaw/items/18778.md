## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Require an activity secret before bypassing auth:** `src/gateway/a2ui-activity.ts:35-36`
  - body: `isA2uiActivityAccessAllowed` returns true when Activity mode is enabled without a token, and `authorizeCanvasRequest` accepts that before normal Gateway auth for A2UI. A public gateway with this option enabled would expose A2UI state, so this needs to fail closed or validate a signed Discord assertion.
  - confidence: 0.95
- **[P1] Rebase Canvas hosting onto the Canvas plugin:** `src/gateway/server-runtime-state.ts:3-4`
  - body: This branch still imports `../canvas-host` and reads root `canvasHost` config, but current main moved Canvas/A2UI hosting behind `extensions/canvas` plugin routes and plugin-owned config. Reintroducing this core path would create a competing Canvas owner and bypass the current migration boundary.
  - confidence: 0.93
- **[P2] Bound Activity payloads and slow clients:** `src/gateway/a2ui-activity.ts:79-82`
  - body: `broadcastPush` stores arbitrary JSONL and sends it to every connected WebSocket without a byte cap or `bufferedAmount` handling. A large A2UI payload or slow clients can grow memory and stall the gateway, so add payload limits and slow-client drop/termination behavior.
  - confidence: 0.9
- **[P2] Avoid sending Activity tokens over insecure WebSockets:** `src/canvas-host/a2ui/index.html:321-325`
  - body: The browser bridge copies `activityToken` from the page URL into the WebSocket URL and uses `ws://` when the page is loaded over HTTP. Activity auth should require WSS or a protected cookie/session exchange rather than leaking secrets through URLs and cleartext transport.
  - confidence: 0.86
