## Summary

This PR is still necessary and appears technically correct: current `main` intentionally intersects requested exec policy with stricter host approvals, while the linked report shows that users can mistake the resulting clamp for lost configuration. The proposed read-only startup warning uses the canonical policy resolver, avoids node/sandbox-owned scopes, and does not weaken enforcement; however, the external contributor’s supplied live proof exercises `exec-policy show`, not the new gateway-startup warning, so real after-fix behavior proof remains a merge gate.

## What This Changes

Adds a one-time gateway startup warning when local host approvals tighten the requested global exec security policy, shares the clamp predicate with exec-policy reporting, adds focused tests, and updates troubleshooting remediation.

## Best Possible Solution

Keep the canonical fail-closed host security floor and add this single read-only gateway diagnostic, with redacted terminal proof showing the patched gateway emits the warning once for a real local clamp before merge.

## Reproduction Assessment

Yes at the source level: current `main` demonstrably intersects requested exec security with stricter host approvals, reproducing the linked report’s effective `allowlist` outcome after restart. This review did not run the patched gateway, so the new warning itself is not independently reproduced.

## Solution Assessment

Yes. Reusing the canonical effective-policy snapshot for a one-time, read-only startup warning is narrower and safer than weakening host approvals, adding compatibility fallback behavior, or creating another policy source of truth.
