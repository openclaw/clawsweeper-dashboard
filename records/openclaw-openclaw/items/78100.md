## Summary

Keep open: current main still has a source-visible synchronous plugin metadata path from channel status into registry discovery, and an open linked PR already uses closing syntax for this issue, so the issue should remain paired until that PR is merged or closed.

## What This Changes

The issue reports that slow synchronous plugin registry discovery during gateway channel status and restart flows can block the event loop and trigger WhatsApp Web disconnect loops on slow filesystems.

## Best Possible Solution

Review and either land or reject PR #78101 as the near-term mitigation, then close this issue only after a merged fix reaches main; track deeper async discovery work separately if the remaining latency still matters.

## Reproduction Assessment

No live high-confidence reproduction was run in this read-only review. Source inspection does verify that configured `channels.status` can synchronously load plugin metadata through `applyPluginAutoEnable` and `loadPluginRegistrySnapshotWithMetadata`, matching the reported hot path.

## Solution Assessment

Unclear whether the linked 2-second TTL is the best long-term solution because plugin metadata freshness is an explicit boundary rule. It is a plausible narrow mitigation, while async discovery or caller-owned snapshots may be the more durable direction.
