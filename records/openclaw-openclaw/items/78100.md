## Summary

Keep open: current main still has a source-backed path from `channels.status` into synchronous plugin metadata discovery, and an open linked PR already carries the proposed fix, so the issue should remain open until that PR lands or is replaced.

## What This Changes

Reduce repeated synchronous plugin registry discovery during gateway channel status and channel restart flows, especially on slow filesystems where it can block the event loop and trigger WhatsApp reconnect loops.

## Best Possible Solution

Land an owner-approved fix that keeps status/restart flows on an explicit current metadata snapshot or a carefully bounded cache/async path, with regression coverage and slow-filesystem/WhatsApp proof.

## Reproduction Assessment

Yes from source, but not from a live run in this review: `channels.status` calls auto-enable, which can rebuild plugin metadata and synchronously validate persisted registry files. The reporter and linked PR provide detailed WSL2/WhatsApp timing logs for the real latency symptom.

## Solution Assessment

Unclear: the linked 2-second coalescing PR targets the right hot path, but it intersects with the documented no-hidden-wall-clock-cache boundary. A maintainer should decide whether that PR is acceptable or whether to pass an explicit metadata snapshot/lookup table or async-ify discovery instead.
