## Summary

Keep open: current main still splits Telegram media captions on raw text before rendering in both direct and reply media paths, but the requested serialized-HTML length guard needs live or contract-level proof because Telegram documents caption limits after entity parsing rather than by raw HTML payload length.

## What This Changes

The issue asks OpenClaw to decide Telegram media captions after Markdown/HTML rendering so oversized rendered captions are sent as follow-up text instead of inline media captions.

## Best Possible Solution

Verify the actual Telegram failure first; if captions really fail, add a Telegram-owned helper that measures the effective caption according to Telegram parsing semantics and reuse it in both direct send and reply media delivery.

## Reproduction Assessment

No high-confidence failing reproduction yet. Source inspection confirms raw-before-render planning, but Telegram's documented limit is after entity parsing, so the examples need a live Telegram failure or a targeted regression test that matches Bot API semantics.

## Solution Assessment

No: the proposed rendered-HTML string length check is not yet proven to be the best fix. A safer solution would first prove the real rejection case, then compare the effective parsed caption length rather than blindly using serialized HTML length.
