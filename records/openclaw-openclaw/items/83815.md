## Summary

Keep open: current main and the latest release still choose Telegram media captions before rendering, but the requested rendered-HTML length guard is not safe to apply without live Telegram proof because the Bot API documents caption limits after entity parsing. The linked fix PR is closed unmerged, so there is no implemented fix to close against.

## What This Changes

The issue asks OpenClaw to decide Telegram media caption splitting after HTML rendering so oversized rendered captions become follow-up text instead of inline media captions.

## Best Possible Solution

First prove the real Telegram rejection; if valid, add a Telegram-owned helper that measures the effective post-entity caption length and reuse it in direct and reply media paths with focused regression coverage.

## Reproduction Assessment

No high-confidence failing reproduction yet. Source inspection confirms raw-before-render caption planning and HTML expansion, but the Telegram contract says caption length is after entity parsing, so the cited examples need live Telegram failure proof or a dependency-faithful failing test.

## Solution Assessment

No: a serialized rendered-HTML length check is not proven to be the best fix. The safer path is to verify the rejection first and then enforce Telegram's effective parsed-caption semantics rather than raw escaped payload length.
