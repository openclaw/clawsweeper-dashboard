## Summary

Keep open: current main and the latest release still plan Telegram media captions from raw text before rendering, but the remaining fix is not safe to automate because Telegram documents caption limits after entity parsing and there is no live Telegram rejection proof for the escaped-HTML examples. ([core.telegram.org](https://core.telegram.org/bots/api%20?utm_source=openai))

## What This Changes

The issue reports that Telegram media captions can be accepted under OpenClaw's raw 1024-character check and then exceed the limit after Telegram HTML rendering.

## Best Possible Solution

First prove the real Telegram rejection case; if it exists, implement a Telegram-owned caption decision helper that measures the effective entity-parsed caption length and reuse it in both direct media sends and reply media delivery.

## Reproduction Assessment

No high-confidence failing reproduction yet. Source inspection confirms raw-before-render caption planning on current main, but Telegram's documented after-entities limit means the examples need live Bot API rejection proof or a dependency-faithful failing test.

## Solution Assessment

No: the proposed serialized-HTML length rule is not yet proven to match Telegram's contract. The safer solution is an entity-aware caption-length decision after live or contract-level proof of the failure.
