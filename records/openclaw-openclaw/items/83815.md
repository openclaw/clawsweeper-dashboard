## Summary

Keep open: current main and the latest release still plan Telegram media captions from raw text before HTML rendering, but the actual Telegram failure and the proposed serialized-HTML length rule remain unproven against the Bot API's after-entities caption contract. The linked fix PR is closed unmerged, and no canonical merged fix now owns the remaining work.

## What This Changes

The issue asks OpenClaw to decide Telegram media caption eligibility after rendering to HTML and send oversized rendered captions as follow-up text.

## Best Possible Solution

Verify the real Telegram failure first; if it reproduces, add a Telegram-owned helper that measures effective post-entity caption length and reuse it in both direct media send and reply media delivery.

## Reproduction Assessment

No high-confidence failing reproduction yet. Source inspection confirms raw-before-render planning and serialized HTML expansion, but the Bot API contract says caption limits are after entities parsing and no live Telegram failure proof is attached.

## Solution Assessment

Unclear: the requested rendered-HTML string limit is not proven to match Telegram semantics. The safer path is live proof first, then an entity-aware caption-length guard if the server actually rejects the current payload.
