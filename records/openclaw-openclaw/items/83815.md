## Summary

Keep open: current main and the latest release still split Telegram media captions on raw text before HTML rendering, but the reported failure needs live Telegram proof because the Bot API caption contract is after entities parsing and the linked fix PR was closed unmerged.

## What This Changes

The issue requests that Telegram media captions be evaluated after rendering so oversized outgoing HTML captions are sent as follow-up text instead of inline media captions.

## Best Possible Solution

Keep the issue open until a live Telegram probe identifies an actual Bot API rejection, then implement an entity-aware Telegram caption decision reused by direct sends and reply media delivery.

## Reproduction Assessment

No high-confidence failing reproduction yet. Source inspection confirms raw-before-render caption planning and HTML escaping expansion, but Telegram documents caption limits after entities parsing, so the examples need live Bot API proof.

## Solution Assessment

No, the rendered-HTML string-length guard proposed in https://github.com/openclaw/openclaw/pull/83816 is not proven to be the best fix. A safer fix would first prove the real rejection case, then compare the effective parsed caption length or otherwise match Telegram Bot API semantics.
