## Summary

Keep this PR open, but do not merge the current branch as-is. Current main still lacks both requested fixes, so the contribution is useful, but the PR introduces two concrete regressions: an over-broad tool mismatch classifier and nondeterministic locale precedence during startup. The branch also needs a rebase onto current main's split sanitizer module before it can be safely landed.

## What This Changes

The PR adds awaitable Control UI locale startup readiness, gates initial app rendering on locale loading, and rewrites tool_use/tool_result mismatch errors to suggest `/new` with focused tests.

## Best Possible Solution

Land a focused rebased update or replacement that makes startup locale loading awaitable, applies persisted or navigator locale first, then applies `settings.locale` as the authoritative override before first render. Keep the tool mismatch fallback narrow to real tool_result/tool_use pairing rejections, wire the sanitizer-side behavior into the current split sanitizer module, and add false-positive coverage.

## Reproduction Assessment

Yes. The underlying current-main gaps are visible from source: startup locale loading is fire-and-forget before immediate render, and tool_result/tool_use mismatch strings do not reach a `/new` recovery classifier. The PR-introduced defects are also reproducible by source inspection: a generic `tool_use_id is required` string matches the new regex, and mixed persisted/settings locales can resolve in either order under `Promise.all`.

## Solution Assessment

No. The proposed direction is maintainable, but this branch is not the best implementation until locale precedence is serialized, the mismatch classifier requires pairing context, and the sanitizer change is rebased onto the current module split.
