## Summary

Keep open: current main does not implement the PR's first-render i18n readiness contract, and the live PR is not merge-ready because it lacks real behavior proof and introduces source-level defects; the tool-mismatch half is already better handled by main's transcript repair.

## What This Changes

The PR combines a Control UI i18n startup render gate with agent error formatting for tool_use/tool_result mismatch messages and related tests.

## Best Possible Solution

Rework or replace the branch with a narrow i18n startup fix that serializes locale selection, preserves topbar lifecycle setup, and leaves the tool-mismatch recovery to current transcript repair unless a narrowly scoped copy change is still needed.

## Reproduction Assessment

Yes, at source level for the PR defects: the diff shows the broad matcher, concurrent locale setters, and delayed empty first render while current lifecycle observes .topbar only once. I did not establish live browser or provider proof in this read-only review.

## Solution Assessment

No. The i18n direction is useful, but this patch can misclassify provider errors, race locale sources, and skip topbar setup after the delayed first render; the safer path is a narrowed i18n-only repair plus existing transcript repair for the tool_result loop.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Narrow the tool_use_id mismatch matcher:** `src/agents/pi-embedded-helpers/errors.ts:835-836`
  - body: The added regex treats any `tool_use_id` substring as transcript corruption, so unrelated provider validation errors such as missing or invalid fields would be rewritten to `/new` guidance. Require actual tool_result/corresponding tool_use mismatch context before rewriting.
  - confidence: 0.91
- **[P2] Apply the settings locale after startup hydration:** `ui/src/ui/app.ts:119-126`
  - body: The constructor starts `i18n.ready` and `i18n.setLocale(this.settings.locale)` in the same `Promise.all`. If the stored or navigator locale and settings locale differ, whichever async load settles last wins, so startup can render in the wrong locale.
  - confidence: 0.86
- **[P2] Retry topbar observation after the delayed render:** `ui/src/ui/app.ts:726-728`
  - body: Returning `nothing` before `localeReady` lets Lit run `firstUpdated()` while `.topbar` is absent. Because current lifecycle calls `observeTopbar()` once, the ResizeObserver and `--topbar-height` setup can be skipped for the real UI.
  - confidence: 0.88
