## Summary

Keep open. The PR targets plausible bug fixes that are not fully implemented on current main, but this branch is not merge-ready because it lacks real behavior proof and introduces source-review P2 correctness issues in both touched paths.

## What This Changes

The PR adds an i18n startup readiness/render gate and rewrites tool_use/tool_result mismatch provider errors to tell users to start a fresh session.

## Best Possible Solution

Replace or update the branch with a narrow fix that serializes startup/settings locale selection, preserves topbar lifecycle setup after delayed rendering, uses a specific orphaned-tool-result classifier if fallback copy is still needed, and includes real runtime proof.

## Reproduction Assessment

Yes, at source level for the PR defects: the diff shows the broad regex, concurrent locale setters, and delayed empty first render while current lifecycle observes .topbar only from firstUpdated. I did not run browser or provider tests because this review is read-only.

## Solution Assessment

No. The direction is useful, but this implementation is not the safest fix because it can misclassify unrelated errors, race locale sources, and skip topbar setup after the delayed first render.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Narrow the tool_use_id mismatch matcher:** `src/agents/pi-embedded-helpers/errors.ts:838-839`
  - body: The new regex treats any error containing `tool_use_id` as conversation-history corruption. Generic validation errors that merely mention the field would tell users to abandon the session with `/new`; require actual `tool_result`/corresponding `tool_use` mismatch context before rewriting.
  - confidence: 0.91
- **[P2] Apply the settings locale after startup hydration:** `ui/src/ui/app.ts:119-126`
  - body: The constructor starts `i18n.ready` and `i18n.setLocale(this.settings.locale)` concurrently. If the stored or navigator locale and settings locale differ, the slower async load can overwrite the settings value, so the initial render can use the wrong language.
  - confidence: 0.86
- **[P2] Retry topbar observation after the delayed render:** `ui/src/ui/app.ts:726-728`
  - body: Returning `nothing` for the first update lets `firstUpdated()` run before `.topbar` exists. Current lifecycle code calls `observeTopbar()` only once, so the ResizeObserver and `--topbar-height` setup can be skipped after `localeReady` renders the real UI.
  - confidence: 0.88
