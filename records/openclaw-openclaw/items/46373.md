Keep open: the PR still targets a plausible Control UI i18n startup bug, but the current patch has source-level correctness blockers and no real behavior proof. The tool_use/tool_result part is largely superseded by current transcript-repair paths, so it should not be merged in this broad regex form.

## What This Changes

The PR combines a Control UI i18n startup readiness/render gate with agent error formatting for tool_use/tool_result mismatch messages and related tests.

## Best Possible Solution

Rework or replace the branch with a narrow i18n startup fix that serializes locale selection, preserves topbar lifecycle setup, and omits or tightly scopes any remaining tool-mismatch copy.

## Reproduction Assessment

Yes, at source level for the PR defects: the diff shows the broad matcher, concurrent locale setters, and delayed empty first render while current lifecycle observes .topbar only from firstUpdated. I did not establish live browser or provider proof in this read-only review.

## Solution Assessment

No. The direction is useful for the i18n startup bug, but this patch can misclassify provider errors, race locale sources, and skip topbar setup after the delayed first render.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Narrow the tool_use_id mismatch matcher:** `src/agents/pi-embedded-helpers/errors.ts:835-836`
  - body: The regex's first alternative is only `tool_use_id`, so unrelated provider validation messages that mention the field are rewritten as transcript corruption and tell users to start `/new`. Require actual tool_result/corresponding tool_use mismatch context before rewriting.
  - confidence: 0.91
- **[P2] Apply the settings locale after startup hydration:** `ui/src/ui/app.ts:119-126`
  - body: The constructor starts `i18n.ready` and `i18n.setLocale(this.settings.locale)` concurrently. If the persisted or navigator locale and settings locale differ, the slower async load can overwrite the intended settings locale before the first render.
  - confidence: 0.86
- **[P2] Retry topbar observation after the delayed render:** `ui/src/ui/app.ts:726-728`
  - body: Returning `nothing` for the first update lets `firstUpdated()` run before `.topbar` exists. Since current lifecycle code calls `observeTopbar()` only once, the ResizeObserver and `--topbar-height` setup can be skipped after `localeReady` renders the real UI.
  - confidence: 0.88
