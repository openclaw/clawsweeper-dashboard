## Summary

Keep open: current main still does not implement the PR's i18n before-first-render gate, while the tool-mismatch half overlaps with the narrower open follow-up at https://github.com/openclaw/openclaw/pull/45393. This PR should not merge as-is because it lacks required real behavior proof and has source-level correctness blockers.

## What This Changes

The PR adds a Control UI locale readiness gate before rendering and rewrites tool_use/tool_result mismatch errors to suggest starting a fresh session.

## Best Possible Solution

Split or rebase this into a narrow i18n startup fix that serializes locale readiness and preserves topbar setup; handle the tool-mismatch class through the narrower transcript-repair follow-up already tracking that behavior.

## Reproduction Assessment

Yes, at source level for the PR defects: the diff shows the broad matcher, concurrent locale setters, and delayed empty render while current lifecycle observes .topbar only once. The original browser/provider flows still need real behavior proof before merge.

## Solution Assessment

No. The i18n direction is useful, but this patch can misclassify provider errors, race locale sources, and skip topbar setup after delayed first render; the safer path is a narrowed i18n repair plus the canonical tool-mismatch repair work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Narrow the tool_use_id mismatch matcher:** `src/agents/pi-embedded-helpers/errors.ts:835-836`
  - body: The added regex treats any `tool_use_id` substring as transcript corruption, so unrelated provider validation errors such as missing or invalid fields would be rewritten to `/new` guidance. Require actual `tool_result`/corresponding `tool_use` mismatch context before rewriting.
  - confidence: 0.91
- **[P2] Apply the settings locale after startup hydration:** `ui/src/ui/app.ts:119-126`
  - body: The constructor starts `i18n.ready` and `i18n.setLocale(this.settings.locale)` in the same `Promise.all`. If the stored or navigator locale and settings locale differ, whichever async load settles last wins, so startup can render in the wrong locale.
  - confidence: 0.86
- **[P2] Retry topbar observation after the delayed render:** `ui/src/ui/app.ts:726-728`
  - body: Returning `nothing` before `localeReady` lets Lit run `firstUpdated()` while `.topbar` is absent. Because current lifecycle calls `observeTopbar()` once, the ResizeObserver and `--topbar-height` setup can be skipped for the real UI.
  - confidence: 0.88
