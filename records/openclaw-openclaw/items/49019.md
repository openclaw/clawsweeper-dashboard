## Summary

Keep open: current main still implements `POST /hooks/agent` as an async hook that returns only `{ ok, runId }`, and the requested result-return/announcement-control mode is absent from source and docs. PR #67433 is an active implementation candidate, so this issue should stay open until that PR is reviewed and landed or closed.

## What This Changes

Request an opt-in wait/result mode for `POST /hooks/agent`, plus optional suppression of main-session announcements for machine-callback workflows.

## Best Possible Solution

Land a reviewed Gateway hook implementation with regression tests, docs, and changelog, then close this issue once the shipped endpoint supports the new opt-in mode.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection confirms current main lacks `waitForResult`, `announceToMain`, and a result-bearing `/hooks/agent` response path.

## Solution Assessment

Unclear pending maintainer review. A narrow opt-in `/hooks/agent` extension looks like the right surface because it preserves the current `sessions_send` HTTP deny posture, but the exact response/error/timeout contract should be settled in PR #67433.
