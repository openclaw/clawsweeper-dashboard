## Summary

Keep open. Current main still implements POST /hooks/agent as an async runId-only ingress, and open PR #67433 directly targets the requested result-return mode, so closing before that PR or an equivalent implementation lands would be premature.

## What This Changes

Add an opt-in POST /hooks/agent mode that waits for the isolated agent turn, returns a deterministic result, and can suppress main-session announcement for machine callbacks.

## Best Possible Solution

Review and land PR #67433, or an equivalent maintainer-approved change, with focused gateway hook tests and docs that preserve the default async behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows /hooks/agent normalizes only the existing async fields, dispatches in a void async path, and returns only { ok, runId }; I did not run a live server because this review is read-only.

## Solution Assessment

Unclear whether the exact proposed field names are final, but an opt-in core /hooks/agent extension is the right implementation boundary. The safest path is maintainer review of PR #67433 or an equivalent narrow patch that keeps current async callers compatible.
