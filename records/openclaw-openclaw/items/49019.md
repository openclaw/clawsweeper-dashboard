## Summary

Keep open. Current main still exposes POST /hooks/agent as async-only hook ingress, and the requested opt-in wait/result-return behavior is not present in the payload type, normalizer, HTTP response path, dispatcher, docs, or focused hook tests. The provided GitHub context also shows open PR #67433 as a direct implementation candidate, so the right cleanup outcome is maintainer review of that PR or an equivalent implementation, not closure.

## What This Changes

The issue requests an opt-in result-return mode for POST /hooks/agent so external automation can wait for a deterministic agent result instead of polling sessions_history.

## Best Possible Solution

Keep this issue open and review PR #67433, or land an equivalent maintainer-approved change. The end state should keep the default async runId-only behavior unchanged while adding an explicitly opted-in blocking/result response, a narrow deterministic status/result shape, documented timeout/error/idempotency semantics, delivery suppression behavior, and focused gateway hook tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: send or inspect a POST /hooks/agent request with wait/result fields and current main will normalize only the existing async fields, dispatch in a void async path, and return only { ok, runId }. I did not run a live server because this review was required to remain read-only, but the implementation, docs, and tests all agree on the current behavior.

## Solution Assessment

Unclear whether the exact proposed field names are the best final API, but the requested direction is a valid core gateway feature rather than already implemented behavior. The safest solution is a maintainer-reviewed opt-in extension to /hooks/agent that preserves the current security model and default async response.
