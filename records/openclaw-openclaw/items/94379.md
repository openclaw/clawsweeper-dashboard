## Summary

Keep open: current main and v2026.6.8 still use the hostless port preflight, so the reported Chrome CDP false negative remains valid. Three open closing PRs already target this issue, so cleanup should wait for one narrow fix to land.

## What This Changes

Make `ensurePortAvailable()` detect IPv4-only loopback occupants before managed Chrome launches, so users get a clear port-in-use error instead of a misleading CDP HTTP 401 failure.

## Best Possible Solution

Land one narrow PR that makes `ensurePortAvailable()` probe the loopback/address-family behavior Chrome actually uses, add IPv4-only regression coverage, then close this issue and duplicate fix PRs.

## Reproduction Assessment

Yes, source-reproducible: current main calls `tryListenOnPort({ port })` without `host`, while managed Chrome launches and reports CDP on `127.0.0.1:<port>`. I did not run a socket repro because this review is read-only, but the issue and linked PRs include live Node proof for the address-family mismatch.

## Solution Assessment

No merged fix exists yet. The maintainable solution is a small owner-local change in the shared port preflight, preferably reusing the existing multi-family inspection behavior or explicitly matching Chrome's loopback bind with regression coverage.
