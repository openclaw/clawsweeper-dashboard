## Summary

Keep open: current main and the latest release still use the hostless port preflight, and two open PRs already reference this issue as their closing target. This should close only after one narrow fix PR lands.

## What This Changes

The issue reports that Chrome CDP port preflight uses a hostless listener, can miss an IPv4-only loopback occupant, and then surfaces a misleading Chrome CDP HTTP 401 startup failure.

## Best Possible Solution

Land one narrow fix PR that makes the shared port preflight probe the Chrome CDP loopback family, or delegates to the existing multi-family port inspection helper, with focused regression coverage.

## Reproduction Assessment

Yes by source inspection: current main still calls tryListenOnPort with only the port before launching Chrome with the same CDP port, matching the reported false-negative preflight condition. I did not run a live macOS dual-stack reproduction because this review is read-only.

## Solution Assessment

Yes: the best path is a narrow fix in the shared port helper, either probing 127.0.0.1 for this guard or delegating to the existing multi-family port inspection logic, plus a focused regression test. The issue should stay open until one linked fix PR lands.
