## Summary

Keep open: current main and the latest release still use a hostless port probe before managed Chrome launch, while the browser path binds Chrome by CDP port and can still fall through to the misleading CDP startup error. No merged or open canonical item found covers the IPv4-only loopback pre-flight mismatch.

## What This Changes

The issue asks OpenClaw to make the managed Chrome CDP port pre-flight detect IPv4-only loopback occupants instead of surfacing a misleading Chrome CDP HTTP 401 startup failure.

## Best Possible Solution

Update the shared port availability check so managed Chrome pre-flight probes the loopback family Chrome actually binds, reuses existing multi-family diagnostics where appropriate, and adds focused regression coverage without adding config surface.

## Reproduction Assessment

No high-confidence local current-main reproduction was established in this read-only pass. The source path is clear and the reporter supplied a macOS IPv4-only listener proof, but the local Linux Node probe returned EADDRINUSE for hostless-after-IPv4.

## Solution Assessment

Yes, the best fix is a narrow shared port-probe repair rather than a browser-specific workaround. The browser launch path already delegates to `ensurePortAvailable`, and the sibling port inspection helper already has the multi-family probing shape needed for consistent diagnostics.
