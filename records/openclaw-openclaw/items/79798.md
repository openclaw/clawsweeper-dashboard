## Summary

Keep open: this is a recent, plausible performance bug that overlaps current Codex app-server notification, Telegram progress, and active-memory hot paths, but the exact config delta in the report conflicts with current main because whole-agent runtime pins are legacy and ignored.

## What This Changes

The issue asks OpenClaw to prevent significant Telegram chat latency and intermittent gateway probe timeouts when main interactive sessions run through the Codex runtime.

## Best Possible Solution

Reproduce the slowdown on current main with the real provider/model runtime policy, Telegram direct chat, and active-memory toggled on and off, then fix the specific bottleneck in Codex notification handling, Telegram progress/status delivery, active-memory pre-prompt execution, or gateway queue/liveness accounting.

## Reproduction Assessment

No. The field steps are plausible, but current main ignores the stated `agents.defaults.agentRuntime.id` delta and I did not establish a live current-main macOS Telegram timing reproduction.

## Solution Assessment

Unclear. Closing or auto-fixing would be premature; the narrow maintainable path is to first prove whether the slowdown comes from Codex app-server notifications/tools, hook relay pressure, active-memory blocking, or gateway queue/liveness state.
