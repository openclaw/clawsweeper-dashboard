## Summary

Keep open: current main still runs the Local CLI TTS command inside the Gateway process and the documented macOS `say` example only writes the provider output file; I found no launchd-aware speaker playback path or docs that solve the daemon/system-user case.

## What This Changes

Request support for macOS Local CLI TTS playback to speakers when OpenClaw runs as a launchctl-installed daemon/system user.

## Best Possible Solution

Decide the product contract first, then either document a supported macOS wrapper using the existing Local CLI config or add a plugin-owned macOS playback helper that avoids hard-coded UIDs and preserves provider file output.

## Reproduction Assessment

Unclear for a full live repro: the report gives concrete macOS launchd steps and source shows Local CLI runs from the Gateway process, but I did not run a macOS launchd/system-user speaker test. The missing core path is source-reproducible because no `launchctl asuser` or equivalent GUI-session bridge exists in current main.

## Solution Assessment

Unclear: the reporter's wrapper is plausible, but hard-coding `launchctl asuser 501` is not the best maintainable product solution. The safer path is a maintainer decision between documented user wrapper configuration and a plugin-owned macOS helper that resolves the active GUI user/session correctly.
