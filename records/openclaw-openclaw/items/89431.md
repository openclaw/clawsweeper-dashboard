## Summary

Keep open: current main and the latest release still run the Local CLI TTS command inside the Gateway process and document `say` only as an output-file command, with no launchd/GUI-session bridge for audible daemon playback.

## What This Changes

The issue reports that documented macOS Local CLI TTS using `say` does not play through speakers when OpenClaw runs as a launchd daemon/system user.

## Best Possible Solution

Make an explicit maintainer decision between documenting a supported wrapper for daemon speaker playback and adding a plugin-owned macOS playback helper that resolves the active GUI user/session without hard-coded UIDs.

## Reproduction Assessment

Medium-confidence source reproduction: current main runs the configured Local CLI command in the Gateway process context, but I did not run a live macOS launchd/system-user speaker test.

## Solution Assessment

Unclear: the reporter's wrapper is a plausible workaround, but hard-coding `launchctl asuser 501` is not the best maintainable product solution. The safer path is a maintainer decision between documented wrapper configuration and a plugin-owned macOS helper that resolves the active GUI session correctly.
