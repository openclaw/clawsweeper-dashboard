## Summary

Keep open: current main and the latest release still run the Local CLI TTS command from the Gateway process and the docs still show the macOS `say -o` file-output form, so there is no launchd-aware speaker playback path for the reported daemon case. The only linked fix PR was closed unmerged because it touched unrelated skill scripts.

## What This Changes

The issue asks OpenClaw to make macOS Local CLI TTS speaker playback work when the Gateway daemon is installed under launchctl.

## Best Possible Solution

Decide the product contract first, then either document a supported macOS wrapper using the existing Local CLI config or add a plugin-owned macOS playback helper that resolves the active GUI user/session without hard-coded UIDs.

## Reproduction Assessment

Source-reproducible but not live-reproduced: the report gives concrete macOS launchd steps, and current code executes the configured Local CLI command inside the Gateway process. I did not run a macOS launchd/system-user speaker test, so the audio-session failure is not fully reproduced here.

## Solution Assessment

Unclear: the reporter's wrapper is a useful workaround, but hard-coded `launchctl asuser 501` is not the best maintainable product solution. The safer path is maintainer choice between documented wrapper configuration and a plugin-owned macOS helper that resolves the active GUI user/session correctly.
