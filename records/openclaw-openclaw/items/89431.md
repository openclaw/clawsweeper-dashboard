## Summary

Keep open: current main and v2026.6.9 still run Local CLI TTS by spawning the configured command from the Gateway process, the docs still show macOS `say -o` file synthesis, and no built-in `launchctl asuser` or equivalent GUI-session speaker bridge exists. The remaining decision is the supported macOS daemon-to-speaker contract, not a cleanup close.

## What This Changes

The issue reports that macOS local TTS using the documented `say` command does not produce audible speaker output when OpenClaw runs as a launchd daemon/system user.

## Best Possible Solution

A maintainer should choose the supported macOS daemon-to-speaker contract: document a wrapper using existing Local CLI config, or add a plugin-owned macOS helper that discovers the active GUI user/session without hard-coded UIDs.

## Reproduction Assessment

No high-confidence live reproduction yet. The issue gives concrete macOS launchd steps, and source review points to Gateway-process CLI spawning with no GUI-session handoff, but audible speaker behavior still needs real macOS daemon proof.

## Solution Assessment

Unclear. The reporter's wrapper is useful evidence, but a hard-coded `launchctl asuser 501` wrapper is not the best maintainable product fix; maintainers should choose between documented wrapper configuration and a plugin-owned macOS helper.
