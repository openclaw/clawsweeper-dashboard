## Summary

Keep open: current main and the latest release still run the Local CLI TTS command directly from the Gateway process, and the docs still show the macOS `say -o` file-output configuration without a launchd-aware speaker playback path.

## What This Changes

The issue reports that macOS TTS using the documented local `say` command does not play through speakers when OpenClaw runs as a launchd daemon/system user.

## Best Possible Solution

A maintainer should decide the product contract, then either document a supported macOS wrapper using existing Local CLI config or add a plugin-owned macOS helper that resolves the active GUI user/session without hard-coded UIDs.

## Reproduction Assessment

Source-reproducible but not live-reproduced: current code executes the configured Local CLI command inside the Gateway process and the docs still show the macOS `say -o` output-file form. I did not run a macOS launchd/system-user speaker test, so the audio-session failure is not fully reproduced here.

## Solution Assessment

Unclear: the reporter's wrapper is a useful workaround, but hard-coded `launchctl asuser 501` is not the best maintainable product solution. The safer path is maintainer choice between documented wrapper configuration and a plugin-owned macOS helper that resolves the active GUI user/session correctly.
