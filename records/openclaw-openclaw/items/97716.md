## Summary

Keep open: focused availability hardening is needed, but this patch applies the cap after `postTokenForm` has already buffered the token response, so it does not fix the OOM path and still needs changed-path proof.

## What This Changes

The PR adds shared bounded response readers to OpenAI ChatGPT/Codex OAuth token exchange and refresh response parsing.

## Best Possible Solution

Apply the 16 MiB cap at the first body read inside `postTokenForm`, keep exchange and refresh behavior stable, and add focused oversized token-response tests plus redacted changed-path proof.

## Reproduction Assessment

Source-level yes, live replay no. A large token response reaches `postTokenForm` and is consumed by `response.arrayBuffer()` before any new bounded reader in this PR runs.

## Solution Assessment

No. The cap belongs at the first body read in `postTokenForm`; downstream bounded parsing of a rewrapped response is too late to prevent OOM.
