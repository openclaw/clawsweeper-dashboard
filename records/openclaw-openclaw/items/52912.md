## Summary

Keep open: this is a plausible bug-fix PR, but the current patch broadens credential propagation by syncing every OAuth credential to sibling agent stores, conflicts with the documented OAuth portability boundary, is stale/conflicting against current main, and lacks real after-fix behavior proof.

## What This Changes

The branch routes plugin-provider auth profile writes through a shared sibling-sync helper, syncs Anthropic token and OAuth credentials to sibling agent dirs, adds auth-choice coverage, and imports DeepSeek into the provider contract registry.

## Best Possible Solution

Replace or rework this as a narrow maintainer-approved Anthropic token persistence and guidance fix that preserves OAuth copy opt-in/read-through semantics and drops the unrelated registry change.

## Reproduction Assessment

Yes for the blocking PR regression from source inspection: the PR predicate makes every OAuth credential sibling-syncable, and the generic plugin-auth write path fans that into sibling agent dirs. I did not live-run the original Anthropic setup-token gateway symptom, which current main may partially mask through read-through inheritance.

## Solution Assessment

No. A shared persistence helper could be useful, but this implementation broadens OAuth credential copying and only addresses one slice of the linked issue; a token-focused fix with explicit policy boundaries is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restrict OAuth auth-profile sync to intended contexts:** `src/plugins/provider-auth-helpers.ts:222-223`
  - body: The new sync policy returns true for every OAuth credential, so any plugin auth method that returns OAuth credentials will write refresh material into all sibling agent stores.
