## Summary

Keep this PR open, but do not merge it as-is. The branch targets a real Anthropic token propagation problem from #51911, but its implementation broadens OAuth refresh-token copying across sibling agents and only changes the plugin auth-choice path, leaving the reported models auth setup-token and paste-token surfaces untouched.

## What This Changes

The PR routes plugin provider auth profile writes through a new sibling-sync helper, syncs Anthropic token and OAuth credentials to sibling agent dirs, adds Anthropic token sync coverage, and adds DeepSeek to the bundled provider contract registry.

## Best Possible Solution

Land a narrower maintainer-approved auth persistence change that covers the intended Anthropic setup-token/manual-token flows across interactive plugin auth, models setup-token, paste-token, and non-interactive setup where needed; keep OAuth refresh-token copying explicitly opt-in; and split unrelated provider contract-registry cleanup out of the auth fix.

## Reproduction Assessment

Yes. The PR diff itself reproduces the main regression risk because any OAuth-returning plugin auth method would now call the shared writer with sibling sync enabled, while current docs and portability code keep OAuth non-portable by default; the linked #51911 command surfaces remain traceable in current `models/auth.ts` and the Anthropic non-interactive setup-token method.

## Solution Assessment

No. This is not the best way to solve the issue because it broadens OAuth credential propagation, misses relevant token entrypoints, and mixes unrelated registry work into a credential-handling PR; the safer direction is an Anthropic-token-scoped persistence/read-through fix with explicit tests for each affected command path.
