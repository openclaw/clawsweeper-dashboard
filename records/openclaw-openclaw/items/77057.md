## Summary

Keep open: the chat refresh cap is a plausible narrow performance fix, but the process-wide `isCliProvider` cache can return stale results after plugin registry/setup metadata changes, and the PR still needs the required changelog entry before merge.

## What This Changes

The PR memoizes `isCliProvider`, caps Control UI final-chat session refreshes to five rows, and adds a setup lookup cache regression test.

## Best Possible Solution

Keep the chat-specific refresh cap, but make CLI-provider memoization invalidation-aware or scoped to one session-list build, add registry/setup change coverage, and add an Unreleased changelog entry.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection shows current main still has no chat-specific refresh limit, while current main now has a server-side default bound of 100 and the linked report supplies the large-store timing data.

## Solution Assessment

No, not as written. The chat refresh bound is narrow, but the global cache must account for plugin registry/setup metadata state or be scoped so it cannot outlive the state it observed.
