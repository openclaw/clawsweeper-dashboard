## Summary

Keep open: this remains a plausible optional performance follow-up, but the patch should not merge as written because the new process-wide CLI-provider cache can go stale across plugin registry/setup metadata changes and the PR is missing the required changelog entry.

## What This Changes

The PR adds process-wide `isCliProvider` memoization, caps chat-final session refreshes to five rows, and adds a setup-backend cache regression test.

## Best Possible Solution

Keep the chat-refresh limit if maintainers still want the extra cleanup, but make CLI-provider memoization registry/setup-state-aware or scope it to one session-list build, then add regression coverage and an Unreleased changelog entry.

## Reproduction Assessment

Yes for the patch defect by source inspection: the cache key omits runtime registry and setup metadata state that the current lookup reads. The original large-store performance symptom was not live-reproduced here, but current main and the linked issue discussion show the relevant `sessions.list` path and separate main fix.

## Solution Assessment

No, not as written. The chat refresh limit is narrow, but the process-wide cache needs invalidation-aware keying or narrower scope before this is a maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Key the CLI-provider cache on registry state:** `src/agents/model-selection-cli.ts:14-19`
  - body: The new cache key only includes the normalized provider and configured `cliBackends` keys, but `isCliProvider` also reads `resolveRuntimeCliBackends()` and `resolvePluginSetupCliBackendRuntime()`. Those results can change when the active plugin registry or setup metadata changes in the same process, so a cached true or false result can survive a plugin reload until restart. Include registry/setup metadata state in the key, invalidate on registry changes, or keep this memoization scoped to one sessions-list build.
