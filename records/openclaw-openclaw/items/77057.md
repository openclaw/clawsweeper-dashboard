## Summary

Keep open: the performance direction is useful for #77056 and related #76562, but the proposed process-global CLI-provider cache can return stale results across plugin metadata/config changes, and the PR is missing the required changelog entry.

## What This Changes

This PR caches CLI provider classification, caps Control UI chat-final session refreshes to five recent sessions, and adds a regression test for cached setup CLI backend lookup.

## Best Possible Solution

Land a narrowed version that keeps the chat-final refresh cap while caching CLI-provider classification only within a valid plugin metadata/config fingerprint, with regression coverage and a changelog entry.

## Reproduction Assessment

Yes for the review blocker: source inspection shows the new global cache key omits plugin metadata/control-plane inputs that the cached setup lookup already treats as variable. The original large-store latency report is source-plausible from the current sessions refresh path, but I did not run a large-store benchmark.

## Solution Assessment

No. Capping the chat-final refresh is a narrow maintainable UI fix, but the CLI-provider cache needs the same invalidation boundary as plugin metadata/setup lookup before the PR should land.
