## Summary

Keep open. Current main still lacks imsg RPC tapback filtering, but this PR needs one correctness fix before merge because upstream imsg RPC reaction metadata uses `is_reaction`/`reaction_type` fields rather than the new `is_tapback`/`associated_message_type` fields, leaving the metadata-only empty-text path able to reach pairing before the empty-body drop.

## What This Changes

The PR adds iMessage imsg RPC tapback detection, parser/type fields, debounce filtering, monitor tests, and a changelog entry for dropping tapback reactions before agent dispatch.

## Best Possible Solution

Keep the owner-module fix in the iMessage plugin, but update it to parse and detect the actual imsg RPC reaction metadata (`is_reaction`, `reaction_type`, `reaction_emoji`, `is_reaction_add`, `reacted_to_guid`) before access control, keep the quoted text heuristic and debounce filtering, and leave full reaction-event surfacing to #39031.

## Reproduction Assessment

Yes. The linked bug gives a concrete imsg RPC tapback scenario, current main source shows a non-empty `Loved "..."` body can proceed through dispatch, and the PR finding is reproducible by using upstream-shaped `{ is_reaction: true, text: "" }` under `dmPolicy: "pairing"`.

## Solution Assessment

No, not as currently proposed. The narrow plugin-local strategy is right, but the metadata checks should match the upstream imsg RPC contract; otherwise only the text heuristic reliably fixes the common path.
