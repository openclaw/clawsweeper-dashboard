## Summary

Keep this PR open, but do not merge as-is. The branch targets a real current-main iMessage gap and already repaired the earlier debounce/access-order review defects, but the metadata path is still aligned to `is_tapback` / `associated_message_type` fields that imsg v0.5.0 RPC does not emit for reaction events; the actual upstream RPC payload exposes `is_reaction` and related reaction fields, so metadata-only or nonstandard-text reactions can still bypass the early drop.

## What This Changes

The PR adds iMessage imsg RPC tapback detection/filtering, debounced-burst tapback stripping, parser/type fields, tests, and a changelog entry.

## Best Possible Solution

Keep the fix in the iMessage plugin, but align the payload contract with imsg RPC by recognizing `is_reaction` and related reaction fields, keep the quoted-text fallback for older text-shaped tapbacks, preserve the debounce handling, and leave full reaction-event surfacing to #39031.

## Reproduction Assessment

Yes. The linked bug gives a concrete imsg RPC scenario, and current main source inspection shows `Loved "..."` text has no tapback-specific drop before normal dispatch; a focused unit check around `resolveIMessageInboundDecision` is enough to reproduce the main gap.

## Solution Assessment

No, not as-is. The owner-module approach is the right shape, but the metadata signals need to match imsg's actual RPC contract before this is the narrowest maintainable fix.
