## Summary

Keep open. Current main and the latest release still use the dropdown-first Agents page workflow, and the Default button path still stages a config edit rather than persisting the new default immediately; the related reports are useful cluster context, but they do not fully supersede this issue because the default-persistence complaint is explicit here.

## What This Changes

The issue requests that the Control UI Agents page show agents as a visible list/sidebar instead of hiding them in a dropdown, and also reports that changing the default agent does not persist.

## Best Possible Solution

Repair the Control UI Agents page with an accessible visible agent list/sidebar on desktop and a compact responsive fallback, showing agent name/id/default state and useful identity/metadata. The Default action should either persist through the existing config save path immediately or make the required Save step clear enough that refresh cannot silently discard the user's expectation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: renderAgents() still uses .agents-select for all agents, and the Default handler only stages config form changes until a separate Save action is invoked.

## Solution Assessment

Yes. A focused Control UI repair is the narrowest maintainable path: update the Agents view, its config/default-agent interaction, and targeted UI/controller tests without adding core API or plugin-surface changes.
