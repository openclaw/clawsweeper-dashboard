## Summary

Keep open: current main still has the Discord component transcript gap, but this PR needs author rework before merge because it changes shared captioned-media transcript semantics, introduces an unresolved adapter metadata trust/size boundary, and lacks real Discord after-fix proof.

## What This Changes

The PR adds Discord component transcript summarization, returns it as `meta.transcriptText` from the Discord outbound adapter, and changes shared outbound delivery mirroring to prefer that enriched text.

## Best Possible Solution

Land a rebased implementation that mirrors bounded, labeled Discord component text through an explicit enriched-transcript path, preserves ordinary media-only transcript semantics, and keeps one idempotent transcript write for component sends.

## Reproduction Assessment

Yes, source-reproducible: current main sends and registers Discord component messages while shared transcript mirroring uses only plain mirror text/media and ignores component-label context. I did not run a live Discord reproduction in this read-only review.

## Solution Assessment

No. The direction is useful, but the current patch changes shared captioned-media transcript behavior and persists generic adapter metadata as assistant session memory without size or trust boundaries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate media transcript merging to enriched sends:** `src/config/sessions/transcript-mirror.ts:44-49`
  - body: The new text+media branch in `resolveMirroredTranscriptText` runs for every caller, not only Discord component summaries. Current main expects ordinary captioned media to mirror only `report.pdf`; this patch would change those transcripts to include the caption plus filename and add duplicate/noisy assistant context. Keep text+filename merging behind an explicit enriched-adapter signal.
