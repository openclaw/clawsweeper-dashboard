## Summary

Keep open: current main still lacks Discord component transcript mirroring, but this PR has a blocking shared media-mirroring regression, unresolved transcript metadata trust/size concerns, and no real Discord after-fix proof.

## What This Changes

The PR adds Discord component transcript text extraction, returns that text as delivery-result metadata, updates shared transcript mirroring to prefer it, changes text-plus-media mirror handling, and adds regression tests.

## Best Possible Solution

Land a narrow Discord component transcript mirror through one idempotent path, gated to explicit enriched adapter summaries, with preserved media mirror behavior and bounded/labeled UI-derived transcript text.

## Reproduction Assessment

Yes, source-reproducible: current main sends and registers Discord component messages but returns no enriched transcript metadata, and the shared mirror path ignores delivery metadata while the linked bug gives concrete button-click context-loss steps. I did not run a live Discord repro in this read-only pass.

## Solution Assessment

No: the direction is useful, but this patch broadens shared captioned-media transcript behavior and promotes generic adapter metadata into assistant transcript memory without explicit enriched-text trust and size boundaries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate media transcript merging to enriched sends:** `src/config/sessions/transcript-mirror.ts:44`
  - body: The new text+media branch in `resolveMirroredTranscriptText` runs for every caller, not only Discord component results. Current main expects ordinary captioned media to mirror only `report.pdf`; this patch would persist `caption\nreport.pdf` across channels and duplicate assistant context. Keep the old media-only behavior unless the caller explicitly passes enriched adapter transcript text.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The diff promotes adapter-provided transcript metadata into assistant session memory without explicit trust and size constraints.

Concerns:

- **[medium] Constrain adapter transcript metadata before persistence:** `src/infra/outbound/deliver.ts:808`
  - body: `meta.transcriptText` comes from channel delivery results and is promoted into persisted assistant transcript memory. Without an internal-only contract, allowlist, labeling, or escaping boundary, an adapter can write assistant-visible text unrelated to the actual sent message.
  - confidence: 0.82

  - body: The component summary concatenates component text, labels, and select options, and the shared deliver path joins adapter text before writing the session transcript. Add hard character or line bounds before persistence to avoid unbounded transcript growth and downstream context cost.
  - confidence: 0.78
