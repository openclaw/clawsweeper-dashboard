## Summary

Keep open: the PR targets a real Discord Components v2 transcript gap that still exists on current main, but the proposed shared transcript changes introduce merge/duplication and trust-boundary problems that should be fixed before merge.

## What This Changes

The PR adds Discord component transcript summarization, returns that text as outbound result metadata, and changes shared outbound delivery mirroring to persist enriched adapter text.

## Best Possible Solution

Land a narrow enriched-transcript implementation that mirrors Discord component UI text through one idempotent path, preserves ordinary media-mirror semantics, dedupes mixed batches, and bounds or labels UI-derived transcript text before persistence.

## Reproduction Assessment

Yes. Source inspection on current main shows Discord component sends register interaction entries and return ids, while shared mirroring only uses params.mirror.text/mediaUrls and never sees component labels.

## Solution Assessment

No. The direction is right, but this diff broadens normal media transcript behavior and promotes unbounded adapter metadata into assistant transcript history; an explicit enriched-text signal with bounds is safer.

Full review comments:

- **[P2] Gate media transcript merging to enriched sends:** `src/config/sessions/transcript-mirror.ts:44-48`
  - body: The new text+media branch runs for every caller, not only adapter-enriched component sends. Current coverage expects a captioned media send to persist `report.pdf`, so keep the old media-only behavior unless the caller explicitly passes enriched adapter transcript text.

  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The patch mirrors adapter-provided metadata into assistant session transcripts without explicit size or trust bounds.

Concerns:

- **[medium] Bound adapter transcript metadata before persistence:** `src/infra/outbound/deliver.ts:808`
  - body: `deliverOutboundPayloadsCore` would join `meta.transcriptText` from delivery results and persist it as assistant transcript text without a character, line, or result cap. Channel adapters/plugins can populate result metadata, so this should be bounded before it can inflate session files or downstream model context.
  - confidence: 0.78
- **[medium] Treat UI-derived transcript text as lower-trust context:** `src/infra/outbound/deliver.ts:816`
  - body: The PR stores component labels/options and adapter-provided transcript metadata as an assistant message. That may be acceptable for context recovery, but the persisted text should be explicitly labeled or constrained so UI-derived content is not silently promoted to high-authority assistant instructions.
  - confidence: 0.72
