## Summary

Keep open: this targets a real Discord Components v2 transcript gap that current main still does not implement, but the proposed patch should not merge as-is because it changes shared captioned-media transcript semantics, lacks real behavior proof, and leaves the new adapter metadata trust/size boundary unresolved.

## What This Changes

The PR adds Discord component transcript summarization, returns it as delivery-result metadata, changes outbound transcript mirroring to prefer that metadata, and adds Discord/outbound regression coverage.

## Best Possible Solution

Land a narrow active-plugin implementation that mirrors Discord component prompt text and interactive labels through one idempotent transcript path, gates text-plus-media merging to explicit enriched adapter summaries, preserves existing captioned-media behavior, and bounds or labels UI-derived transcript text before persistence.

## Reproduction Assessment

Yes, source-reproducible. Current main sends and registers Discord component messages without adding component prompt or label context to the mirrored assistant transcript, and the linked bug report gives concrete live steps for a button-click context loss.

## Solution Assessment

No. The proposed direction is useful, but this patch broadens shared media transcript behavior for all callers and routes generic metadata into assistant memory without an explicit enriched-text trust and size boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate media transcript merging to enriched sends:** `src/config/sessions/transcript-mirror.ts:44`
  - body: The new text+media branch in `resolveMirroredTranscriptText` runs for every caller, not only Discord component results. Current main's delivery mirror coverage expects a captioned media send to persist only `report.pdf`; this patch would change ordinary captioned media transcripts to `caption\nreport.pdf` across channels, adding duplicate assistant text. Gate the merge on an explicit enriched-adapter signal or keep the existing media-only behavior for ordinary sends.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The diff adds a new generic metadata-to-assistant-transcript path that needs explicit trust, size, and labeling constraints before merge.

  - body: `meta.transcriptText` comes from channel delivery results and is promoted into persisted assistant transcript memory. Without an explicit internal contract or allowlist, a plugin/adapter can write arbitrary assistant-visible text unrelated to the actual sent message.
