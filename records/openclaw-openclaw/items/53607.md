## Summary

Keep open: the underlying Discord Components v2 transcript gap still exists on current main, but this PR should not merge as-is because it changes shared media transcript behavior and promotes adapter metadata into assistant session history without a settled size/trust boundary.

## What This Changes

The PR adds Discord component transcript summarization, attaches that summary as outbound result metadata, teaches shared outbound delivery to mirror that metadata into session transcripts, adjusts transcript text/media resolution, and adds Discord/outbound regression tests.

## Best Possible Solution

Land a refreshed implementation that mirrors Discord component text and interactive labels through one idempotent transcript write path, keeps ordinary captioned-media filename semantics unchanged, and bounds or labels UI-derived enriched transcript text before storing it as assistant session history.

## Reproduction Assessment

Yes. The current-main code path is enough to reproduce the bug with high confidence: Discord component sends register component entries and return ids, while shared mirroring only uses params.mirror.text/mediaUrls and ignores result metadata; the docs confirm component clicks return as normal inbound messages.

## Solution Assessment

No, not as-is. The direction is appropriate, but the PR's shared helper change is broader than the Discord component case and the metadata persistence boundary needs a narrower enriched-text signal plus size/trust handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate media transcript merging to enriched sends:** `src/config/sessions/transcript-mirror.ts:44`
  - body: The new text+media branch in resolveMirroredTranscriptText runs for every caller, not only Discord component results. Current main has coverage expecting a captioned media send to persist the filename (`report.pdf`) rather than `caption\nreport.pdf`; keep the old media-only behavior unless the caller explicitly passes enriched adapter transcript text.
  - confidence: 0.92

  - body: In mixed result batches, prepending `params.mirror.text` to `adapterParts` can duplicate content when the enriched component transcript already includes the same base message, such as `Pick a color` plus `Pick a color\n[Red]`. Merge or dedupe the base text before appending adapter summaries.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff has no dependency or workflow supply-chain change, but it promotes adapter/plugin result metadata into persisted assistant transcript text without a clear size or trust boundary.

Concerns:

- **[medium] Bound adapter transcript metadata before persistence:** `src/infra/outbound/deliver.ts:808`
  - body: deliverOutboundPayloadsCore would join `meta.transcriptText` from delivery results and persist it as assistant transcript text without a character, line, or result cap. Channel adapters/plugins can populate result metadata, so this should be bounded before it can inflate session files or downstream model context.
  - confidence: 0.78
- **[medium] Treat UI-derived transcript text as lower-trust context:** `src/infra/outbound/deliver.ts:816`
  - body: The PR stores component labels/options and adapter-provided transcript metadata as an assistant message. That may be acceptable for context recovery, but the persisted text should be explicitly labeled or constrained so UI-derived content is not silently promoted to high-authority assistant instructions.
  - confidence: 0.72
