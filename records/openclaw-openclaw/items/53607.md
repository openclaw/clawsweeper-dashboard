## Summary

Keep open: this PR targets a real Discord Components v2 transcript gap, but the current patch is not merge-ready because it broadens shared media transcript behavior and promotes adapter metadata into assistant transcript history without a clear size/trust boundary.

## What This Changes

The PR adds Discord component transcript summarization, returns it as outbound result metadata, teaches shared outbound delivery to mirror that metadata, and adds Discord/outbound regression tests.

## Best Possible Solution

Land a refreshed implementation that mirrors Discord component text and interactive labels through one idempotent transcript write path, preserves ordinary captioned-media filename semantics, and bounds or labels UI-derived enriched transcript text before storing it as assistant history.

## Reproduction Assessment

Yes. Current main provides a high-confidence source-level reproduction: component sends register interaction entries and return ids, while shared mirroring only uses params.mirror.text/mediaUrls and never sees component labels.

## Solution Assessment

No. The approach is directionally right, but this patch is too broad in the shared transcript helper and needs an explicit enriched-text signal plus size/trust handling before merge.

- **[P2] Gate media transcript merging to enriched sends:** `src/config/sessions/transcript-mirror.ts:44`
  - body: The new text+media branch in `resolveMirroredTranscriptText` runs for every caller, not only adapter-enriched component sends.   - body: In mixed result batches, prepending `params.mirror.text` to `adapterParts` can duplicate content when the enriched component transcript already includes the same base message, such as `Pick a color` plus `Pick a color\n[Red]`. Merge or dedupe the base text before appending adapter summaries.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: Security review needs attention because the patch promotes adapter/plugin result metadata into persisted assistant transcript text without an explicit size or trust boundary.

Concerns:

- **[medium] Bound adapter transcript metadata before persistence:** `src/infra/outbound/deliver.ts:808`
  - body: `deliverOutboundPayloadsCore` would join `meta.transcriptText` from delivery results and persist it as assistant transcript text without a character, line, or result cap. Channel adapters/plugins can populate result metadata, so this should be bounded before it can inflate session files or downstream model context.
  - confidence: 0.78
- **[medium] Treat UI-derived transcript text as lower-trust context:** `src/infra/outbound/deliver.ts:816`
  - body: The PR stores component labels/options and adapter-provided transcript metadata as an assistant message. That may be acceptable for context recovery, but the persisted text should be explicitly labeled or constrained so UI-derived content is not silently promoted to high-authority assistant instructions.
  - confidence: 0.72
