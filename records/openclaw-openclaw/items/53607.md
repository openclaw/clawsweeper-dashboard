## Summary

Keep open. The PR addresses a real Discord Components v2 transcript gap that is still present on current main, but the branch is not merge-ready because it regresses ordinary captioned-media transcript mirroring, lacks real Discord behavior proof, and needs a clearer trust/size boundary for adapter-provided transcript metadata.

## What This Changes

The PR adds Discord component transcript summarization, propagates it as `meta.transcriptText` through outbound delivery mirroring, and adds Discord/outbound regression tests.

## Best Possible Solution

Land a rebased narrow fix that records a bounded, labeled Discord component UI summary through one idempotent mirror path without changing ordinary media transcript semantics or treating generic plugin metadata as trusted assistant memory.

## Reproduction Assessment

Yes, at source level. Current main sends and registers Discord component messages, but the component payload result carries no transcript summary and the shared mirror path only persists `params.mirror.text` or media filenames; I did not run a live Discord reproduction.

## Solution Assessment

No. The central fix direction is right, but this branch broadens captioned-media transcript behavior and routes generic adapter metadata into assistant memory without an explicit bounded/trusted contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate text-plus-media mirrors to enriched sends:** `src/config/sessions/transcript-mirror.ts:44`
  - body: The new branch in `resolveMirroredTranscriptText` runs whenever any caller passes both text and media URLs. Current main expects ordinary captioned media sends to mirror only the filename (`report.pdf`), so this changes non-enriched channel sends into `caption\nreport.pdf` and can duplicate assistant text in session history. Keep the old media-only behavior unless the caller explicitly passes enriched adapter transcript text.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The diff introduces a generic adapter-metadata-to-assistant-transcript path that needs an explicit trust and size boundary before merge.

Concerns:

- **[medium] Treat transcriptText as an explicit trusted contract:** `src/infra/outbound/deliver.ts:810`
  - body: `deliverOutboundPayloadsCore` reads `r.meta?.transcriptText` from the generic `OutboundDeliveryResult.meta` bag and persists it as assistant transcript memory. That metadata is channel/plugin-owned, so this should be an explicit internal enriched-transcript contract or otherwise restricted/labeled before it is fed back to the model as assistant context.
  - confidence: 0.86

  - body: The PR joins adapter transcript parts and component labels before persisting them, but the changed path does not cap transcript length or line count. Add a shared bound/truncation point before writing mirrored adapter text into session history.
