- `impact:message-loss`: The issue is about misrouting a channel reply away from the originating group conversation.

## Summary

Keep open. Current main still has a source-reproducible iMessage routing path where an anchorless `chat_id: 0`, `is_group: false` watch payload is accepted, classified as direct, and can build a sender-DM reply target; the companion `openclaw/imsg` issue tracks the bad upstream payload but does not remove OpenClaw's need to fail closed before routing.

## What This Changes

The issue requests an iMessage monitor guard that repairs malformed group link-preview watch payloads from stable message history or drops them instead of routing replies to the sender DM.

## Best Possible Solution

Land a focused iMessage plugin guard before inbound routing that recovers anchorless watch payloads by GUID through existing `chats.list` / `messages.history`, then drops with a clear warning when recovery fails.

## Reproduction Assessment

Yes, from source inspection: the reported payload parses, is not classified as a group, resolves the sender as the direct peer, and builds `To: imessage:<sender>`. I did not run tests because this review is read-only and repository tests can write artifacts.

## Solution Assessment

Yes, the best fix shape is a pre-routing iMessage monitor guard that either repairs the conversation from the message GUID or fails closed. That keeps the safety invariant local to the channel adapter instead of relying on downstream session routing to infer missing group metadata.
