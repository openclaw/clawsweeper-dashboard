## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Propagate busy fields before formatting them:** `src/commands/channels/status.ts:110-124`
  - body: The new formatter reads `account.busy`, `account.activeRuns`, and `account.lastRunActivityAt`, but computed account snapshots drop those runtime fields before `openclaw channels status --deep` sees them. Discord and Telegram-style computed adapters can therefore still omit the busy/stale bits in production output even while a channel run is active; forward these fields through the shared runtime snapshot projection or plugin extras before formatting them.
  - confidence: 0.9
- **[P1] Recompute lane size after clearing interrupt queues:** `src/auto-reply/reply/get-reply-run.ts:901-905`
  - body: When an interrupt/reset clears the session lane, this resolver still uses the `laneSize` captured before `clearCommandLane()`. `resolvePreparedReplyQueueState()` can then treat the cleared lane as active and return the shutdown/queued path instead of starting the interrupted request; re-read the queue size inside the resolver or update it after clearing.
  - confidence: 0.91
- **[P2] Mark busy-turn acknowledgements deliverable:** `src/auto-reply/reply/agent-runner.ts:122-127`
  - body: In `message_tool_only` mode, dispatch suppresses automatic source replies unless payload metadata sets `deliverDespiteSourceReplySuppression`. These new queued/steered acknowledgements are plain payloads returned before any message-tool run, so Discord/Telegram can still appear silent; mark them with the existing suppression-bypass metadata while preserving send-policy denial.
  - confidence: 0.88
- **[P2] Preserve inbound activity for accepted Discord actions:** `extensions/discord/src/monitor/provider.ts:119`
  - body: Making raw gateway events transport-only is reasonable for messages, but interactions and reactions do not currently record accepted human inbound activity after their filters. Accepted slash commands or reaction events can therefore stop advancing Discord's inbound timestamp; add accepted-action inbound updates for those paths.
  - confidence: 0.83
