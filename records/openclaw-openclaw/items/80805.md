## Summary

Keep open: this is plausible bug-fix work, but the PR is draft, dirty against current main, lacks real Discord/Telegram behavior proof, and has a blocking active-turn concurrency issue.

## What This Changes

The branch adds active/stale channel-turn status fields, wraps Discord/Telegram message processing in turn tracking, adjusts activity recording, and returns visible Discord/Telegram acknowledgements for queued or steered turns.

## Best Possible Solution

Rebase onto the current channel-turn/plugin architecture, preserve concurrent active-turn state per account, add focused regression coverage, and provide redacted live Discord/Telegram proof for the busy-turn acknowledgement and status behavior.

## Reproduction Assessment

Yes for the review finding: source inspection shows a single per-account activeTurnId is overwritten by overlapping Discord/Telegram turns, then cleared when the newer turn ends. I did not run a live transport reproduction for the reported responsiveness fix.

## Solution Assessment

No: the direction is useful, but this branch is stale against current plugin paths and the active-turn store needs concurrent-turn semantics. The narrower maintainable solution belongs in the current src/channels/turn and plugin SDK flow plus extension-specific tests/proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Track concurrent turns per account:** `src/infra/channel-activity.ts:114-119`
  - body: startChannelTurn stores only one activeTurnId for a channel/account. The new Discord and Telegram wrappers use account-level keys for every message, so two simultaneous chats on the same account can overwrite each other; when the shorter turn ends, channel status reports idle while the longer turn is still running. Keep a per-account map/count of active tokens, or only clear the account status when no active turn remains.
  - confidence: 0.84
