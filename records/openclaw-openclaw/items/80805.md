- **[P1] Keep reaction accepted callback in the typed params:** `extensions/discord/src/monitor/listeners.reactions.ts:119`
  - body: The PR passes `onAcceptedEvent` into `handleDiscordReactionEvent` and later reads `params.onAcceptedEvent`, but that handler is typed as its local event object intersected with `DiscordReactionRoutingParams`, which does not include the callback. CI is already failing TS2353/TS2339 on this path, so add the optional callback to the handler/routing params or pass it through a typed local before merge.
  - confidence: 0.97
- **[P2] Refresh status for accepted Discord messages:** `extensions/discord/src/monitor/provider.ts:560-564`
  - body: After splitting raw gateway events from accepted inbound status, accepted plain messages still only call `recordChannelActivity`; `channels.status` uses that fallback only when the runtime snapshot has no `lastInboundAt`. A later human message can therefore leave an older slash/reaction timestamp visible, so wire accepted message preflight into the new status callback or report the newest timestamp.
  - confidence: 0.84
- **[P2] Record accepted interactions before long handlers finish:** `extensions/discord/src/monitor/listeners.ts:69-70`
  - body: The interaction listener marks accepted activity only after `client.handleInteraction` resolves, but command/component handlers can defer and then wait on session locks or compaction. During that wait, status still has only the raw `lastEventAt`; report acceptance when dispatch matches/acknowledges the interaction rather than after the full handler completes.
  - confidence: 0.78
