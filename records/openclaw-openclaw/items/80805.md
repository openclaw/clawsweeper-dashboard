## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Record accepted interactions before long handlers:** `extensions/discord/src/monitor/listeners.ts:73-75`
  - body: This callback runs only after `client.handleInteraction(...)` resolves, but dispatch awaits the command/component/modal handler before returning `true`. Slash/component handlers can wait on locks or compaction, and if they throw after defer/ack then `lastInboundAt` is never refreshed, so channel status still looks idle while accepted human input is being processed. Report acceptance from the dispatch path as soon as the interaction is matched/deferred/acknowledged, before awaiting the long handler.
  - confidence: 0.87
- **[P2] Only acknowledge follow-ups that were queued:** `src/auto-reply/reply/agent-runner.ts:1327`
  - body: The PR head ignores `enqueueFollowupRun(...)`'s boolean result and always returns the visible queued acknowledgement. Current main checks that return value because enqueue can reject aborted, duplicate, already queued, or drop-policy rejected follow-ups; without preserving that guard, users can be told a message was queued when no queue item exists.
  - confidence: 0.9
