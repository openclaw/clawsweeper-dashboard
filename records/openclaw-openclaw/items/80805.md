## Summary

This PR addresses the earlier review blockers, but one Discord status path still needs a narrow fix: after raw gateway events stop updating `lastInboundAt`, accepted plain Discord messages can fail to refresh runtime inbound activity when an older accepted interaction or reaction already set that field. Keep it open for that focused repair and normal CI/merge handling.

## What This Changes

The PR adds Discord/Telegram busy-run status reporting, active-turn acknowledgement payloads, Discord accepted-action activity tracking, and focused regression tests.

## Best Possible Solution

Keep the PR's busy-state and acknowledgement approach, with a small Discord status fix so every accepted human input path refreshes inbound activity without counting bot/self/filtered events.

## Reproduction Assessment

Yes by source inspection: the PR routes raw Discord events through `lastEventAt` only, accepted plain messages update global activity, and channel status ignores that fallback once runtime `lastInboundAt` is already non-null. I did not run a live mixed slash-command/message sequence.

## Solution Assessment

No, not quite yet; the broad approach is maintainable, but the Discord inbound timestamp fix should also cover accepted plain messages or make status choose the newest runtime/global inbound timestamp.
