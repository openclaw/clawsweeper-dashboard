## Summary

Keep open: this is still a useful downstream recovery fix for the linked stuck opencode-go run bug, but the latest head still classifies stale post-model recovery from an ever-started model flag rather than active model-call state, so it can incorrectly enter idle-timeout fallback after the model call is no longer in flight.

## What This Changes

The PR preserves diagnostic `stuck_recovery` abort reasons through embedded-run abort/drain handling, classifies them before failover, tracks active potential-side-effect tool executions, and adds focused regression tests.

## Best Possible Solution

Land this downstream recovery fix only after the classifier uses active model-call state and maintainer review settles how it should pair with the provider-boundary opencode-go fix.

## Reproduction Assessment

Yes. Source inspection shows a stale recovery after model-call completion can still see `modelCallStartedForStuckRecovery=true` even though current diagnostic activity removes the active model call on completion/error.

## Solution Assessment

No. The PR is a plausible fix, but the best fix must classify `model_idle_timeout` only while a model call is active or reset that state when the model call ends.
