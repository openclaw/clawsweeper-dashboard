## Summary

Keep this PR open. Current main still lacks adaptive daily-plus-idle reset semantics, but the PR is not merge-ready because the proposed hook reason logic can disagree with the lifecycle-based reset decision, the help text misstates the defaulted `idleMinutes` behavior, the changelog is missing, and #59411 proposes an overlapping public config shape.

## What This Changes

The PR adds `session.reset.mode: "adaptive"` across session reset types, schema/help/docs, freshness evaluation, session-end reason typing, and focused tests so daily and idle expiry use AND semantics for that mode.

## Best Possible Solution

Land one coherent daily-plus-idle AND reset contract after maintainers choose the public API shape. If this PR is selected, update it to derive adaptive hook reasons from the same lifecycle stale state used for rollover, describe `idleMinutes` as defaulted, add the changelog entry, keep existing `daily` and `idle` behavior unchanged, and validate the focused session/config/hook lanes.

## Reproduction Assessment

Yes. Current main can be checked by source inspection: adaptive is absent from the reset type/schema and freshness remains OR-based; the PR-specific hook bug follows from a rollover where lifecycle timestamps are stale but `updatedAt` was refreshed after the daily boundary.

## Solution Assessment

No, not as-is. A focused `adaptive` mode is a plausible implementation, but the current patch should first fix the lifecycle reason mismatch, correct the optional/defaulted `idleMinutes` docs, add changelog coverage, and be reconciled with #59411's broader API proposal.
