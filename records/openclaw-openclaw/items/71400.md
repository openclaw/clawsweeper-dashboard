## What This Changes

The PR adds `session.reset.mode: "adaptive"` across session reset config types, schema/help/docs, hook reason typing, runtime freshness logic, and focused tests so daily and idle rollover can use AND semantics.

## Best Possible Solution

Land one coherent daily-plus-idle AND contract after maintainers choose between this `adaptive` mode and #59411's broader API. If this PR is selected, rebase it onto current lifecycle timestamp freshness, make adaptive use `sessionStartedAt` and `lastInteractionAt`, fix hook reason reporting and config help text, add changelog coverage, preserve existing `daily` and `idle` semantics, and validate the focused session/config/hook lanes.

Yes. Current main can be verified by source inspection: adaptive mode is absent from the reset type/schema and docs still describe OR semantics. ## Solution Assessment

No, not as-is. A narrow `adaptive` mode is a plausible implementation, but this branch needs rebase and contract cleanup, and maintainers should first resolve the overlap with #59411's `idleGuard`/timezone API proposal.
