## Summary

Keep open: current main still lacks adaptive session reset semantics, and this PR remains a plausible implementation path. It is not merge-ready because it changes the public reset config surface while leaving the generated config-doc baseline unchanged, and the API direction overlaps with https://github.com/openclaw/openclaw/pull/59411.

## What This Changes

The branch adds `session.reset.mode: "adaptive"` with daily-plus-idle AND reset semantics across config types, Zod schema/help, docs, runtime freshness/session-end reason wiring, and focused tests.

## Best Possible Solution

Choose one public reset contract, then land a coherent change with runtime semantics, exported types, Zod schema/help, generated config baseline, docs, hook/harness reason typing, and focused tests aligned.

## Reproduction Assessment

Yes. Source inspection on current main shows only `daily` and `idle` modes, OR freshness semantics, and docs that say whichever daily/idle condition expires first wins; the PR body supplies terminal proof for the proposed after-fix runtime path.

## Solution Assessment

No, not as-is. The adaptive implementation direction is plausible, but the generated config baseline must be refreshed and maintainers should first resolve whether this API or the overlapping `idleGuard`/timezone PR is the contract to keep.
