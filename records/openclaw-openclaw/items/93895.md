## Summary

Current main and the shipped v2026.6.8 release have moved cron persistence to SQLite-backed rows with transactional runtime-state updates and quarantine handling, so the reported split JSON migration race is no longer present.

## What This Changes

The issue reports silent loss of persisted cron jobs during hot reload when the legacy split JSON migration leaves `jobs.json` and `-state.json` out of sync.

## Best Possible Solution

Keep the shipped SQLite cron store, doctor migration, and quarantine behavior as the canonical fix for this historical JSON split-migration race.

## Reproduction Assessment

Yes for the affected release: the report includes production backups, config-write timing, and concrete legacy split-store code snippets that explain the data loss. I did not run v2026.5.18 locally, and current main should not reproduce because that split JSON persistence path is gone.

## Solution Assessment

Yes. The shipped database-first cron store is the maintainable fix because it removes the split JSON persistence window and adds doctor/quarantine handling rather than layering another runtime fallback onto the old file format.
