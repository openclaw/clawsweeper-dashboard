## Summary

Keep open. Current main and v2026.6.8 still use a hardcoded two-second macOS Voice Wake HUD capture window, and the issue is already paired with an open PR that uses closing syntax, so it should remain open until that PR lands or is replaced.

## What This Changes

Request to make the macOS wake-word HUD post-trigger capture silence window configurable instead of fixed at two seconds.

## Best Possible Solution

Review and land a safe version of https://github.com/openclaw/openclaw/pull/93632, preferably reusing `talk.silenceTimeoutMs` so the HUD gains configurability without adding a new config key.

## Reproduction Assessment

Yes, from source inspection rather than a live macOS run. Current main hardcodes the Voice Wake HUD post-trigger silence window at 2.0 seconds while the existing `talk.silenceTimeoutMs` config is consumed by Talk mode only.

## Solution Assessment

Yes, the best direction appears to be reusing the existing `talk.silenceTimeoutMs` contract in the Voice Wake startup/refresh path instead of adding `talk.wakeCaptureSilenceMs`. The linked PR is the active candidate, so the issue should remain open until that PR lands or is replaced.
