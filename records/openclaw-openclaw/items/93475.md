## Summary

Keep open: current main and the latest release still hardcode the macOS Voice Wake HUD post-trigger silence window at 2.0 seconds, and the open linked PR is the active fix candidate but still has product/proof review blockers.

## What This Changes

The issue requests making the macOS HUD wake-word capture silence window configurable so natural pauses do not prematurely finalize and submit speech.

## Best Possible Solution

Review and resolve the linked PR as the implementation candidate, with a compatibility-safe config contract, preserved 2000 ms HUD fallback when unset, docs for the final semantics, and native macOS HUD proof before merge.

## Reproduction Assessment

Yes, by source inspection: current main hardcodes the Voice Wake post-trigger silence window to 2.0 seconds and `monitorCapture` finalizes when that threshold elapses. I did not run a live macOS HUD capture in this read-only review.

## Solution Assessment

Unclear at product level: reusing `talk.silenceTimeoutMs` may be the leaner path than adding `talk.wakeCaptureSilenceMs`, but maintainers still need to accept the shared-key semantics and proof requirements before this is solved.
