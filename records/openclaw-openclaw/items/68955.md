## Summary

Keep open. Current main and v2026.6.10 still route Telegram native and plugin slash commands through separate `bot.command` handlers without the regular message-path ack reaction and typing dependencies, while the only direct implementation is an open, conflicting external fork PR with broad unrelated churn rather than a safe upstream landing path.

## What This Changes

The issue requests Telegram slash commands such as `/new` and `/reset` to emit the same ack reaction and typing feedback as regular Telegram messages.

## Best Possible Solution

Land a narrow Telegram-owned feedback path after command auth and before slow native or plugin command work, preserving native command fast-pathing while adding focused tests and live Telegram proof.

## Reproduction Assessment

Yes, at source level: current main and v2026.6.10 wire ack reactions and early typing through the regular Telegram message context while native and plugin slash commands run through separate handlers without those dependencies. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Unclear pending maintainer direction. A small helper invoked after command auth appears safer than routing commands through the full message pipeline, but maintainers still need to confirm the desired command-feedback lifecycle.
