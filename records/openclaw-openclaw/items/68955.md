## Summary

Keep open. Current main still wires Telegram ack reactions and typing only through the regular inbound-message path, while built-in and plugin native slash commands use separate `bot.command` handlers without those feedback dependencies or calls; the linked external fork PR is open and unmerged, so the requested behavior remains outstanding.

## What This Changes

The issue requests Telegram slash commands such as `/new` and `/reset` to emit the same ack reaction and typing indicator used for regular inbound Telegram messages.

## Best Possible Solution

Add a Telegram-owned post-auth native-command feedback helper that reuses the existing ack gate and shared chat-action handler while preserving the native command fast path.

## Reproduction Assessment

Yes, at source level: regular Telegram messages call the existing typing and ack paths, while native slash commands enter separate `bot.command` handlers without those dependencies or calls. I did not run a live Telegram E2E reproduction in this read-only pass.

## Solution Assessment

Yes, if maintainers accept the UX change: explicit post-auth feedback in the Telegram native-command path is narrower and safer than forcing slash commands through the full regular-message pipeline. The linked fork can inform the fix, but a narrow upstream patch or replacement is likely cleaner.
