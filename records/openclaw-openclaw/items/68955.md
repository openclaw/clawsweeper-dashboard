## Summary

Keep open. Current main still sends Telegram ack reactions and typing from the regular message path, while native slash commands remain on separate `bot.command` handlers with no feedback plumbing; the only linked implementation is an open, conflicting fork PR rather than an upstream fix.

## What This Changes

Request to emit Telegram ack reaction and typing feedback for native slash commands such as `/new` and `/reset` under the existing feedback config.

## Best Possible Solution

Add a Telegram-owned post-auth native-command feedback helper that reuses the existing ack gate and shared chat-action handler, with focused tests and live Telegram proof while preserving the native command fast path.

## Reproduction Assessment

Yes, at source level: regular Telegram messages build a feedback-capable context and call typing/ack logic, while native slash commands enter separate `bot.command` handlers without those dependencies or calls. I did not run live Telegram in this read-only review.

## Solution Assessment

Unclear until maintainers choose the native-command feedback contract, but the narrow post-auth helper direction is the best-looking implementation shape. Rerouting native commands through the full regular-message pipeline would conflict with the maintained fast-path boundary.
