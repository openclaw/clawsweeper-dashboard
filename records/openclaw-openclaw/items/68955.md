## Summary

Keep open. Current main and v2026.6.8 still keep Telegram slash commands on separate `bot.command` handlers without the regular message-path ack reaction and typing dependencies, and the only linked implementation is an open conflicting fork PR rather than an upstream fix.

## What This Changes

Request that Telegram slash commands such as `/new` and `/reset` emit the same ack reaction and typing feedback as regular inbound messages.

## Best Possible Solution

Land a narrow Telegram-owned command feedback path after command auth and before slow command work, preserving fast-path command routing while adding focused tests and real Telegram proof.

## Reproduction Assessment

Yes, at source level. Current main shows regular Telegram messages build feedback state in `buildTelegramMessageContext`, while slash commands enter separate `bot.command` handlers without the ack or typing dependencies; no live Telegram run was performed in this read-only review.

## Solution Assessment

Unclear pending maintainer direction. A small helper invoked after command auth looks like the narrow owner-boundary fix, while routing commands through the full message pipeline would conflict with the fast-path command design.
