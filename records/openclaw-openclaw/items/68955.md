## Summary

Keep open. Current main still wires Telegram ack reactions and typing through the regular message path, while native slash commands remain separate `bot.command` handlers without those feedback dependencies or calls; the only linked implementation is an open external fork PR, not upstream code.

## What This Changes

Request that Telegram slash commands such as `/new` emit the same ack reaction and typing feedback as regular inbound messages.

## Best Possible Solution

Add a Telegram-owned post-auth native-command feedback helper that reuses the existing ack gate and shared chat-action handler, with focused unit coverage and live Telegram proof while preserving the native command fast path.

## Reproduction Assessment

Yes at source level: regular Telegram messages build a feedback-capable context and call typing/ack logic, while native slash commands enter separate `bot.command` handlers without those dependencies or calls. I did not run live Telegram in this read-only review.

## Solution Assessment

Unclear until maintainers choose the final UX contract, but a narrow post-auth helper in the native-command path is the best-looking direction. Rerouting slash commands through the full regular-message pipeline would conflict with the maintained fast-path boundary.
