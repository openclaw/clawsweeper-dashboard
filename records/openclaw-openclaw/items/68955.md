## Summary

Keep open. Current main and v2026.6.10 still keep Telegram slash commands on separate native/plugin `bot.command` handlers without the regular message-path ack reaction and typing dependencies, and the only direct implementation is an open external fork PR that is dirty and not an upstream landing path.

## What This Changes

The issue requests Telegram slash commands such as `/new` to emit the same ack reaction and typing feedback as regular inbound messages.

## Best Possible Solution

Land a narrow Telegram-owned command feedback path after command auth and before slow native/plugin command work, preserving native command routing while adding focused tests and real Telegram proof.

## Reproduction Assessment

Yes, at source level: current main wires typing and ack reactions through the regular Telegram message context, while native and plugin slash commands run through separate `bot.command` handlers without those feedback dependencies. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Unclear pending maintainer direction. A small helper invoked after command auth appears narrower than routing commands through the full message pipeline, but the open fork PR is too dirty to treat as the direct upstream fix path.
