## Summary

Keep open. Current main still has Telegram ack reaction and typing wiring only on the regular inbound-message path, while built-in and plugin native slash-command handlers remain separate and lack those feedback dependencies or calls; the linked fork PR is open and unmerged, so the requested behavior is still outstanding.

## What This Changes

Request Telegram native slash commands such as `/new` to emit the existing ack reaction and typing feedback under the current reaction/typing configuration.

## Best Possible Solution

Add a Telegram-owned post-auth native-command feedback helper that reuses the existing ack gate and shared chat-action handler without routing slash commands through the full regular-message pipeline.

## Reproduction Assessment

Yes, at source level: regular Telegram messages call the existing typing and ack paths, while native slash commands enter separate `bot.command` handlers without those dependencies or calls. I did not run a live Telegram E2E reproduction in this read-only review.

## Solution Assessment

Yes, if maintainers accept the UX change: explicit post-auth feedback in the Telegram native-command path is narrower and safer than forcing slash commands through the full regular-message pipeline. The linked fork can inform the fix, but a narrow upstream patch is likely cleaner.
