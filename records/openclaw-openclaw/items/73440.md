## Summary

Keep open. The proposed memo helps the first normalization pass but is not yet a complete fix for the linked large-ownerAllowFrom slowdown because the authorization path still rebuilds and scans owner state per message; the helper and changelog blockers are also unresolved.

## What This Changes

The PR adds WeakMap memoization and a colocated test for `resolveOwnerAllowFromList` to avoid re-filtering `commands.ownerAllowFrom` arrays for each authorization pass.

## Best Possible Solution

The desired end state is a per-config-array compiled owner authorization cache that preserves provider-prefix filtering and config reload semantics while eliminating per-message linear owner-list derivation and membership scans.

## Reproduction Assessment

Yes, from source inspection. Current main enters `resolveCommandAuthorization`, resolves owner allowlists twice, rebuilds owner state, and uses linear membership checks, so a large `commands.ownerAllowFrom` list exercises the reported path even though I did not run a live load reproduction in this read-only sweep.

## Solution Assessment

No. The WeakMap direction is maintainable, but caching only the normalized `string[]` is not the best endpoint for this performance bug because downstream wildcard stripping, owner-list construction, and membership lookup remain O(n).
