## Summary

Keep open: the PR is pointed at the right command-auth performance bug, but it still leaves the large-list authorization path O(n) per message after the formatted list is cached, and the reset helper/changelog blockers remain unresolved.

## What This Changes

The PR adds a WeakMap cache around `resolveOwnerAllowFromList` plus a colocated memoization test for large `commands.ownerAllowFrom` arrays.

## Best Possible Solution

Keep the WeakMap-per-raw-array direction, but cache compiled owner authorization state and membership structures while preserving current owner-prefix semantics, test isolation, and changelog policy.

## Reproduction Assessment

Yes, by source inspection. Configure a large `commands.ownerAllowFrom` list and send an inbound message; current main enters `resolveCommandAuthorization`, resolves owner lists, rebuilds owner state, and performs linear membership checks on that path.

## Solution Assessment

No. Memoizing only the formatted `string[]` is a useful partial optimization, but the narrow maintainable endpoint is to cache compiled owner state or membership sets for the same raw-array/provider/account/plugin key.
