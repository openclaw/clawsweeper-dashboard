## What This Changes

The PR adds a WeakMap-backed memoization layer and colocated tests for ownerAllowFrom list resolution in command authorization.

## Best Possible Solution

Land a narrow command-auth fix that caches the normalized owner list plus O(1) membership data per stable raw-array/key, removes the no-op test reset helper, and includes real gateway/log proof for the large-list path.

## Reproduction Assessment

Yes. Source inspection shows current main still walks commands.ownerAllowFrom and then scans owner membership with Array.includes during command authorization; I did not run a live 9,000-entry Gateway benchmark in this read-only review.

## Solution Assessment

No. Memoizing normalized arrays is a useful first layer, but it is not the best complete fix while sender matching still does O(n) includes against those arrays on every message.

Full review comments:

- **[P2] Cache owner membership as a Set:** `src/auto-reply/command-auth.ts:285`
  - body: The memoized value is still a `string[]`, so after the first normalization hit every authorization still runs `ownerState.ownerList.includes(candidate)` and `ownerCandidatesForCommands.includes(candidate)`. For the reported large-list non-owner path, that leaves per-message O(n) membership work and makes the PR's "2 x Map lookups" claim false; cache a membership structure or otherwise make sender matching O(1) while preserving the list shape callers need.
  - confidence: 0.9
- **[P3] Remove the no-op reset helper:** `src/auto-reply/command-auth.ts:352-357`
  - body: `resetOwnerAllowFromMemoForTest` is exported as a defensive isolation hook but its body does not reset anything. Future tests that call it would still share memoized entries, so either delete the export or make the cache resettable.
  - confidence: 0.82
