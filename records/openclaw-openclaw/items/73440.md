## Summary

Keep open: current main still has the large `ownerAllowFrom` command-auth performance problem, and this PR is a relevant narrow fix attempt, but the patch still leaves per-message O(n) owner-list work and lacks real Gateway/performance proof.

## What This Changes

The PR adds a WeakMap cache around `resolveOwnerAllowFromList` plus a colocated memoization test for large `commands.ownerAllowFrom` lists.

## Best Possible Solution

Compile or memoize the owner authorization state or membership sets per raw allowlist/provider/account/plugin so repeated inbound messages avoid both normalization and downstream wildcard/dedupe/includes scans, then prove the large-list Gateway path.

## Reproduction Assessment

Yes at source level: current main walks the raw `ownerAllowFrom` array and then checks resolved owner arrays with linear scans in the inbound command authorization path. I did not run a live 9,000-entry Gateway benchmark in this read-only review.

## Solution Assessment

No: this PR is a plausible partial mitigation, but it caches only the formatted list while the caller still performs large-list wildcard, dedupe, and membership scans on every message. The better fix should cache or compile the owner authorization lookup shape itself.
