## Summary

Keep open. Current main still has the large `commands.ownerAllowFrom` authorization hot path, and this PR is relevant, but the head only caches the normalized list and still leaves downstream owner-state derivation and membership checks linear, with a no-op test helper and missing changelog entry.

## What This Changes

The PR adds a WeakMap memo around `resolveOwnerAllowFromList` and a colocated memoization test for large `commands.ownerAllowFrom` lists.

## Best Possible Solution

Keep the narrow command-auth performance direction, but cache a compiled owner authorization entry for the raw-array/provider/account/plugin key, including wildcard state and membership sets, while preserving provider-prefix filtering and config reload semantics.

## Reproduction Assessment

Yes. Configure a large `commands.ownerAllowFrom` list and send an inbound message; current main enters `resolveCommandAuthorization`, resolves owner allowlists, derives owner state, and performs linear membership checks on that path.

## Solution Assessment

No. The WeakMap direction is maintainable, but caching only the normalized `string[]` is not the best endpoint for the linked performance bug because the downstream owner-state derivation and lookup remain O(n).
