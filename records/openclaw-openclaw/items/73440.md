## Summary

Keep open: this is a useful partial mitigation, but it is not merge-ready because it caches only the formatted owner allowlist while the hot authorization path still rebuilds and scans large arrays, and the external-contributor real behavior proof is missing.

## What This Changes

The branch adds a WeakMap cache and colocated tests around `resolveOwnerAllowFromList` in command authorization.

## Best Possible Solution

Cache a membership-ready owner authorization shape per stable raw list and provider/account/plugin key, preserve existing owner semantics, add focused regression coverage, and require redacted large-list Gateway benchmark or log proof before the linked issue closes.

## Reproduction Assessment

Yes, source-reproducible: current main still walks large `commands.ownerAllowFrom` arrays and checks owner membership with linear `includes` calls in the command authorization path. I did not run a live 9,000-entry Gateway benchmark in this read-only review.

## Solution Assessment

No: caching only the formatted string array is a plausible mitigation, but it is not the best fix for the linked bug because the remaining owner-state and membership checks still scale with the list size. The safer fix is a cached owner-auth/membership shape plus real large-list proof.
