## Summary

Keep open: current main still has the source-level command-authorization performance bug, and this PR is a plausible fix path, but the branch is not merge-ready because it only caches normalized arrays, leaves downstream linear owner-state and membership work, and lacks external-contributor real behavior proof.

## What This Changes

The PR adds a WeakMap memo around `resolveOwnerAllowFromList` plus a colocated memoization test for large `commands.ownerAllowFrom` lists.

## Best Possible Solution

Cache a compiled owner authorization entry keyed by the relevant raw arrays, provider, account, and plugin, including wildcard flags and membership sets while preserving provider-prefix filtering and config reload semantics.

## Reproduction Assessment

Yes, source-level rather than live-run. Configure a large `commands.ownerAllowFrom` list and send an inbound message; current main builds command context, resolves owner allowlists, derives owner state, and performs linear membership checks on that path.

## Solution Assessment

No. The WeakMap direction is reasonable, but caching only the formatted `string[]` is not the best endpoint because downstream owner-state derivation and membership lookups remain O(n).
