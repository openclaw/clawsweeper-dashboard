## Summary

Keep open. Current main still has the large owner allowlist hot path, and this PR is useful but not merge-ready because it caches only the normalized list while leaving downstream per-message owner-state and membership scans in place, plus it has a no-op test reset helper and no changelog entry.

## What This Changes

This PR adds a WeakMap cache around `resolveOwnerAllowFromList` keyed by the raw allowlist array and channel/account/plugin tuple, plus a colocated memoization test file.

## Best Possible Solution

Keep the narrow command-auth performance direction, but cache a compiled owner authorization entry for the same raw-array/provider/account/plugin key, including wildcard state and membership sets, while preserving channel prefix filtering and config reload semantics.

## Reproduction Assessment

Yes. Configure a large `commands.ownerAllowFrom` list and send an inbound message; current main enters `resolveCommandAuthorization`, resolves owner allowlists, derives owner state, and performs linear membership checks on each pass.

## Solution Assessment

No. The WeakMap direction is maintainable, but caching only `string[]` is not the best endpoint for #50289 because the downstream owner-state derivation and lookup remain O(n).
