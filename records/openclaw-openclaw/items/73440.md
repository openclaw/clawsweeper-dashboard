## Summary

Keep open. The PR targets a real owner authorization performance bug, but the current diff only memoizes the normalized string list while the hot path still rebuilds owner state and performs linear membership checks per message; it also leaves a no-op test helper and omits the promised changelog entry.

## What This Changes

The PR adds a WeakMap memoization layer around `resolveOwnerAllowFromList` plus a colocated test for `commands.ownerAllowFrom` cache behavior.

## Best Possible Solution

Keep the WeakMap-by-raw-array direction, but cache compiled owner authorization state and membership sets for the provider/account/plugin key, preserve config reload semantics, remove or implement the reset helper, and add focused regression coverage plus the user-facing changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: configure a large `commands.ownerAllowFrom` list, send an inbound message, and current main enters `resolveCommandAuthorization`, resolves owner allowlists, rebuilds owner state, and performs linear membership checks.

## Solution Assessment

No. The WeakMap approach is a good foundation, but caching only the normalized `string[]` is not the narrow best fix for the reported hot path because downstream owner-state derivation and lookup remain linear.
