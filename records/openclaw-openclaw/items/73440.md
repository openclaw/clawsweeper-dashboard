## Summary

Keep open. The linked performance bug is valid and this PR targets the right command-auth hot path, but the current patch caches only the normalized string array while the merged result still rebuilds owner state and performs linear membership checks for every inbound message; it also leaves a no-op test reset helper and omits the required changelog entry.

## What This Changes

This PR adds a WeakMap cache around command owner allowlist normalization keyed by the raw allowlist array and channel/account/plugin tuple, plus a colocated Vitest regression file.

## Best Possible Solution

Keep the narrow command-auth fix direction, but cache or compile the owner authorization data needed by the downstream hot path, including wildcard state and membership lookups, so large `commands.ownerAllowFrom` lists avoid repeated per-message normalization and membership scans while preserving channel-prefix, account/plugin, context override, and config reload semantics.

## Reproduction Assessment

Yes. Configure a large `commands.ownerAllowFrom` list and send an inbound message; current main and the PR merge path enter command authorization, and the PR merge still scans the resolved owner list with wildcard filtering, Set construction, and `includes` membership checks on every pass.

## Solution Assessment

No. The WeakMap direction is maintainable, but caching only `string[]` is not the best fix for the linked bug because the downstream owner-state derivation remains O(n); a compiled cache entry with membership data is the safer endpoint.
