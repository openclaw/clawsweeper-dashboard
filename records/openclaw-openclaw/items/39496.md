- **[P1] Avoid timestamp-only stale-message rejection:** `extensions/feishu/src/dedup.ts:226-227`
  - body: The new watermark rejects `createTimeMs <= lastTime` for each chat/sender. Because this code explicitly normalizes second-level timestamps, two distinct messages sent by the same user in the same second can share the same value and the later one is skipped before handling.
  - confidence: 0.9
- **[P2] Bound create_time dedupe state:** `extensions/feishu/src/dedup.ts:208`
  - body: `lastProcessedCreateTime` is process-wide state with no TTL or size cap. Long-running gateways that see many chat/sender pairs will grow this map monotonically, unlike the existing bounded persistent dedupe cache.
  - confidence: 0.84
- **[P1] Scope contact cache rows by Feishu account:** `extensions/feishu/src/contacts.ts:23`
  - body: The contacts table uses `open_id` as a global primary key while the APIs accept `accountId`, and local search returns rows before account-specific sync. Multi-account installs can return, prune, or use another tenant's contact for send/calendar actions.
  - confidence: 0.88
- **[P1] Scope group cache rows by Feishu account:** `extensions/feishu/src/groups.ts:56`
  - body: The groups table uses `chat_id` as a global primary key and local lookups run before account-specific sync. One tenant's group can satisfy another tenant's lookup or be deleted by another tenant's sync.
  - confidence: 0.86
- **[P1] Preserve the held claim for direct abort messages:** `extensions/feishu/src/monitor.account.ts:373-381`
  - body: `dispatchFeishuMessageDirect` calls `handleFeishuMessage` without `processingClaimHeld: true`, but receive handling has already claimed the same message ID. Normal stop messages can be treated as duplicates and fail to abort the in-flight run.
  - confidence: 0.84
- **[P2] Gate proactive local file sends explicitly:** `extensions/feishu/src/feishu-ops-tools.ts:64-65`
  - body: The new file action accepts a local `file_path` and uploads it from the host during agent tool calls. Bundling this needs explicit opt-in, path constraints, or equivalent policy before merge.
