## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid timestamp-only stale-message rejection:** `extensions/feishu/src/dedup.ts:226-227`
  - body: The create_time watermark rejects createTimeMs <= lastTime for each chat/sender. Same-second messages or reordered webhook delivery can be dropped before handling, so this needs a retry-specific tie-breaker or a bounded dedupe design.
  - confidence: 0.9
- **[P2] Bound the create-time dedupe map:** `extensions/feishu/src/dedup.ts:208`
  - body: lastProcessedCreateTime is process-wide state with no TTL or size cap. Long-running gateways that see many chat/sender pairs can grow this map forever, unlike the existing bounded persistent dedupe cache.
  - confidence: 0.82
- **[P1] Scope contact cache rows by Feishu account:** `extensions/feishu/src/contacts.ts:22-23`
  - body: The contacts table uses open_id as a global primary key and local searches return before account-specific sync. Multi-account installs can return, prune, or use another tenant's contact for send/calendar actions.
  - confidence: 0.89
- **[P1] Scope group cache rows by Feishu account:** `extensions/feishu/src/groups.ts:55-56`
  - body: The groups table uses chat_id as a global primary key and local lookups run before account-specific sync. One tenant's groups can satisfy or be deleted by another tenant's sync.
  - confidence: 0.87
- **[P1] Preserve the held claim for direct abort messages:** `extensions/feishu/src/monitor.account.ts:373-381`
  - body: dispatchFeishuMessageDirect calls handleFeishuMessage without processingClaimHeld: true, but receive handling already claimed the same message ID. Normal stop messages can be treated as duplicates and fail to abort the in-flight run.
  - confidence: 0.84
- **[P2] Skip recalled entries without dropping the whole batch:** `extensions/feishu/src/monitor.account.ts:341-349`
  - body: The queued dispatch guard returns when any sourceMessageIds entry was recalled, which drops unrecalled messages merged into the same debounce batch. Recall handling should filter the recalled IDs rather than abort the entire queued dispatch.
  - confidence: 0.78
- **[P2] Gate proactive file sends explicitly:** `extensions/feishu/src/feishu-ops-tools.ts:64-65`
  - body: The new file action accepts a local file_path and uploads it from the host during agent tool calls. Bundling this needs explicit opt-in, path constraints, or equivalent policy before merge.
  - confidence: 0.8
- **[P2] Route ops tools to an enabled account:** `extensions/feishu/src/feishu-ops-tools.ts:162-166`
  - body: The ops tools register after finding enabled accounts but execute without passing accountId, so multi-account setups can fall back to a disabled or wrong default account while another account made the tools available.
  - confidence: 0.77
