## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve the claimed message when bypassing the queue:** `extensions/feishu/src/monitor.account.ts:373-381`
  - body: The receive path claims the message before debounce, but the fast-abort bypass calls `handleFeishuMessage` without `processingClaimHeld: true`. That lets the finalizer treat the stop message as already in flight, so the new fast abort can silently skip instead of cancelling the active run.
  - confidence: 0.9
- **[P1] Do not use create_time as a chat-wide watermark:** `extensions/feishu/src/dedup.ts:225-227`
  - body: Rejecting `createTimeMs <= lastTime` for a chat/sender pair drops legitimate messages that share a second-level timestamp or arrive out of order. Current main solves the retry case with a stable per-message dedupe key, so this watermark adds visible message-loss risk.
  - confidence: 0.92
- **[P1] Scope Feishu ops cache rows by account:** `extensions/feishu/src/contacts.ts:427-430`
  - body: `searchContactsOrSync` returns local cache hits before syncing the requested account, while the table is keyed only by `open_id`. In multi-account setups, a contact from one Feishu tenant can satisfy another account's lookup and feed the wrong recipient into send/calendar actions.
  - confidence: 0.87
- **[P1] Route file uploads through the media sandbox:** `extensions/feishu/src/proactive-send.ts:87-93`
  - body: The new file send action accepts a model-provided `file_path` and opens it directly with `fs.createReadStream`, bypassing the existing `loadWebMedia`/`localRoots` guard used by Feishu media upload paths. This can upload arbitrary host-readable files unless it is routed through the sandbox or an explicit confirmation policy.
  - confidence: 0.9
- **[P1] Filter recalled entries instead of dropping the merged dispatch:** `extensions/feishu/src/monitor.account.ts:345-349`
  - body: The queued-dispatch guard returns when any source ID was recalled, which also drops unrecalled messages that were merged into the same debounced dispatch. Recall handling should remove the recalled entry without suppressing later valid user text in the batch.
  - confidence: 0.84
