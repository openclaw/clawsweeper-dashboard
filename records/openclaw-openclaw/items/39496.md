## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate file uploads behind media roots:** `extensions/feishu/src/proactive-send.ts:87-93`
  - body: `filePath` comes from the tool call and is opened directly with `fs.createReadStream` before uploading to Feishu. Existing Feishu media sending requires local roots for local path reads, so this new path can exfiltrate arbitrary host files; reuse the media sandbox or remove file-path sending until there is an explicit safe boundary.
  - confidence: 0.9
- **[P1] Pass an explicit Feishu account to ops tools:** `extensions/feishu/src/feishu-ops-tools.ts:128-132`
  - body: The tools register after finding enabled accounts, but execution calls `searchContactsOrSync` and the other ops helpers without an `accountId`. In multi-account configs this falls back to the default account while the new global caches can return IDs from another tenant, so sends and calendar operations can target the wrong Feishu account.
  - confidence: 0.86
- **[P1] Do not use create_time as a monotonic watermark:** `extensions/feishu/src/dedup.ts:225-227`
  - body: Rejecting every message with `createTimeMs <= lastTime` treats Feishu timestamps as strictly increasing per sender. Same-second timestamps or out-of-order delivery will make legitimate messages look stale and silently drop them, so this dedupe needs an exact retry key or bounded cache that cannot suppress new content.
  - confidence: 0.88
- **[P2] Store ops cache in account-scoped plugin state:** `extensions/feishu/src/contacts.ts:14-15`
  - body: The new contacts cache hard-codes `~/.openclaw/data/feishu-contacts.db` and shells out to `sqlite3`, bypassing the plugin state store and `OPENCLAW_STATE_DIR` ownership used by current Feishu dedupe state. This makes state placement, account isolation, and upgrade behavior inconsistent with the current runtime contract.
  - confidence: 0.82
