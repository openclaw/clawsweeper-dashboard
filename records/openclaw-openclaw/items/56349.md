## Summary

Keep open. Current main has partial `message_sending` coverage in the shared outbound dispatcher and newer inbound dispatcher wrappers, but WebChat final/delta delivery and some plugin-owned reply dispatchers still bypass a guaranteed pre-send boundary. The discussion also includes recent live evidence and related unmerged PRs, so this is active platform work rather than cleanup-close material.

## What This Changes

The issue requests a guaranteed pre-send outbound policy boundary so every user-visible message can be validated, modified, or blocked before channel delivery.

## Best Possible Solution

Define one documented outbound policy contract and route WebChat, `chat.send`/`chat.inject`, explicit message sends, auto-replies, streaming previews/finals, and bundled/external channel-plugin dispatchers through a shared pre-send helper. The contract should state cancel, rewrite, hook-failure, streaming, and observability semantics, then add contract tests that fail when a normal user-visible delivery path bypasses it.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to register a `message_sending` hook and trigger a Gateway WebChat final reply; current `server-chat.ts` constructs and broadcasts the final `chat` payload without consulting the hook runner.

## Solution Assessment

Yes, the issue's single-boundary direction is the maintainable way to solve the platform invariant. The current partial implementation and isolated per-channel wiring are not enough, and the safer path is a shared core/SDK helper plus coverage tests rather than more ad hoc channel fixes.
