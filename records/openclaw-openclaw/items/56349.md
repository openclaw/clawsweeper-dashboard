## Summary

Keep open. Current main has meaningful hook-aware outbound infrastructure, but source inspection still shows WebChat final delivery and sampled channel-plugin reply paths that can reach user-visible sends without one guaranteed `message_sending` pre-send boundary.

## What This Changes

Request a single unbypassable outbound policy boundary that can validate, rewrite, or block every outbound message before channel delivery.

## Best Possible Solution

Define and document one outbound lifecycle contract, then route WebChat final delivery, default reply dispatch, explicit sends, and bundled channel-plugin dispatch through shared pre-send policy helpers with regression tests for each user-visible path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to register a `message_sending` hook and deliver a WebChat final assistant reply: `emitChatFinal` currently broadcasts the final payload directly without consulting the hook runner.

## Solution Assessment

Unclear at full scope. The requested single boundary is the right end state, but the implementation needs maintainer-approved lifecycle semantics across WebChat, direct sends, reply dispatch, and channel-plugin runtime paths.
