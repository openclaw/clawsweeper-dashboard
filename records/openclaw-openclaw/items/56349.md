## Summary

Keep open. Current main has a durable outbound path where `message_sending` can cancel or modify delivery, and `message(action=send)` now uses it, but Slack inbound reply dispatch still falls back to a direct `deliverReplies` path with only post-send hooks, so the requested unbypassable pre-send guarantee is not implemented across all outbound paths.

## What This Changes

The issue requests a single enforceable pre-send outbound policy boundary that all agent replies, tool sends, and auto-replies must pass through before any channel adapter sends content.

## Best Possible Solution

Define one versioned core outbound message lifecycle that every visible channel send must use, migrate remaining direct reply paths such as Slack onto it, and explicitly document any non-user-content native/status sends that are outside the guarantee.

## Reproduction Assessment

Yes, at source level: Slack inbound replies can still route through `deliverReplies` without durable delivery options, and that path sends with `sendMessageSlack` while only emitting post-send hooks. I did not run a live Slack reproduction in this read-only review.

## Solution Assessment

Unclear pending maintainer design: a unified pre-send boundary is the right end state, but maintainers need to decide whether `message_sending` is the universal policy hook or whether a new versioned outbound policy API should own fail-closed semantics and audit behavior.
