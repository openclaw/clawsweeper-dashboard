## Summary

Keep open. Current main has meaningful partial `message_sending` coverage through the durable outbound substrate, but WebChat final broadcasts and sampled bundled channel-plugin send paths still bypass one guaranteed pre-send boundary; narrower open issues cover slices of this, not the full invariant requested here.

## What This Changes

The issue requests a single unbypassable pre-send policy boundary that can validate, modify, or block every outbound message path before channel delivery.

## Best Possible Solution

Define one documented outbound lifecycle contract and route WebChat, explicit sends, auto-replies, and bundled/third-party channel dispatch through a shared pre-send helper, with tests that fail when normal user-visible paths bypass it.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to register a `message_sending` hook and deliver a WebChat final assistant reply: current `emitChatFinal` directly broadcasts the final payload without consulting the hook runner.

## Solution Assessment

Unclear at full scope. The requested single boundary is the right end state, but current piecemeal hook wiring and legacy channel-plugin paths need maintainer-approved lifecycle semantics before an automated fix should be attempted.
