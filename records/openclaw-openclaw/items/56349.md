## Summary

Keep open. Current main has useful hook-aware outbound and durable message seams, but sampled WebChat, gateway chat.send, Slack, Mattermost, and Signal reply paths still show direct final/broadcast/send delivery without a guaranteed `message_sending` pre-send boundary.

## What This Changes

The issue requests a single unbypassable outbound pre-send policy boundary that can validate, modify, or block every outbound message path.

## Best Possible Solution

Keep this umbrella open until maintainers define one outbound lifecycle contract and migrate remaining WebChat, gateway, tool, auto-reply, and channel-plugin sends through a shared hook-aware send context with regression coverage.

## Reproduction Assessment

Yes, at source level. Register a `message_sending` hook and send a WebChat final assistant reply or sampled Slack/Mattermost/Signal channel reply; current main has direct broadcast/send paths there with no `runMessageSending` or `beforeDeliver` seam.

## Solution Assessment

No, current main is only a partial solution. The shared outbound and durable send paths are the right substrate, but the best fix needs a maintainer-approved universal lifecycle boundary rather than more isolated channel wiring.
