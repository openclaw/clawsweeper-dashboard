## Summary

Keep open. Current main has meaningful partial `message_sending` coverage, but WebChat/control UI delivery and sampled bundled channel-plugin send paths still do not pass through one guaranteed pre-send boundary, matching the recent live Telegram/WebChat evidence in the discussion.

## What This Changes

The issue requests a single unbypassable outbound policy boundary that can validate, rewrite, or block every user-visible outbound message before channel delivery.

## Best Possible Solution

Define one documented outbound policy contract and route WebChat, explicit sends, auto-replies, and channel-plugin dispatch through a shared pre-send helper, with contract tests that fail when any normal user-visible path bypasses it.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to register a `message_sending` hook and deliver an assistant reply through WebChat/control UI: current `server-chat.ts` directly broadcasts final chat payloads without consulting the hook runner.

## Solution Assessment

Unclear at full scope. The requested invariant is the right direction, but the current piecemeal hook wiring is not the best final shape; maintainers should approve a single contract covering streaming/final delivery and plugin-owned send adapters.
