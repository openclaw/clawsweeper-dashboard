## Summary

Current main already addresses the reported Telegram/json5 startup failure at the package/runtime-deps layer: `json5` is declared as a root runtime dependency, publish verification now catches undeclared root dist bare imports, and current docs say startup no longer mutates or regenerates plugin dependency trees. The fix is present on main but is not in `v2026.4.29`.

## What This Changes

The issue reports that OpenClaw 2026.4.29 Telegram startup fails because bundled ESM output imports `json5` as a bare package while plugin runtime-deps staging removes local package patches on restart.

## Best Possible Solution

Keep the package-level runtime dependency and publish-time bare-import guard, then ship them in the next release so affected `v2026.4.29` installations can update instead of patching staged `node_modules`.

## Reproduction Assessment

Yes. The report gives a concrete `v2026.4.29` packaged Telegram startup path and `ERR_MODULE_NOT_FOUND` signature; this read-only review verified current-main source and provenance rather than running a gateway startup.

## Solution Assessment

Yes. The current main approach fixes the observable missing-package failure by packaging/mirroring the dependency and guarding future undeclared root dist imports, which is more maintainable than rewriting emitted imports or patching staged package metadata.
