The PR redesigns the Control overview dashboard into a two-column layout with a sticky status rail and clickable summary metrics for gateway, channels, sessions, usage, skills, cron, events, and attention state.

## Best Possible Solution

Keep the redesign direction, but make the rail use skeleton or `n/a` placeholders until the relevant snapshots load, add the required changelog entry and focused render coverage, then attach real Control UI behavior proof before merge.

## Reproduction Assessment

Yes, from source inspection: nullable overview snapshots are rendered by the new rail before they are known, and unknown values are displayed as zero-like counts. I did not run the browser UI, so this is source-reproducible rather than live-reproduced.

## Solution Assessment

No; the dashboard redesign is plausible, but the current implementation should first preserve the established loading/N/A semantics, add the required changelog entry, and provide real behavior proof.
