## What This Changes

The issue reports that `openclaw memory index --force` can send oversized embedding batches to an OpenAI-compatible Qianfan endpoint and abort the whole index instead of chunking or skipping only irreducible failures.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80241 or an equivalent memory-core fix that adaptively splits concrete low-limit embedding batches, preserves provider/quota failures, and adds regression coverage plus real low-limit endpoint proof.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: configure an OpenAI-compatible embedding endpoint whose unknown model allows about 2000 tokens per batch, then index enough chunks for OpenClaw to form one batch under its 8000 budget but over the provider limit.

## Solution Assessment

Yes. The narrow maintainable fix belongs in memory embedding batching/provider-limit adaptation rather than Qianfan-specific provider code, with the linked PR as the active implementation path once its review blockers are resolved.
