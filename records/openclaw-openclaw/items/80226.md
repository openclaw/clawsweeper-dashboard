## What This Changes

The issue reports `openclaw memory index --force` aborting when a Qianfan/OpenAI-compatible embedding endpoint rejects combined batches over its 2000 limit.

## Best Possible Solution

Land https://github.com/openclaw/openclaw/pull/80241 or a focused replacement that adaptively splits rejected embedding batches, preserves real provider failures, and adds regression coverage before closing this issue.

## Reproduction Assessment

Yes. Source inspection shows current main can send an unknown OpenAI-compatible embedding provider a combined batch under OpenClaw's 8000 budget but over the reporter's 2000 provider limit, which matches the logged `openclaw memory index --force` failure.

## Solution Assessment

Yes, the best fix belongs in memory-core embedding batching and retry classification rather than in Qianfan-specific chat-provider code. Because an open PR already references this issue with closing syntax, the issue should stay open until that PR or a replacement is merged.
