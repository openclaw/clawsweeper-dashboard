## Summary

Keep this open: current main has some chunk and batch guards, but the source still allows an unknown OpenAI-compatible embedding model to be batched up to the generic 8k budget, which matches the reported 2k provider-limit failure path.

## What This Changes

The issue asks for memory indexing to pre-chunk content before embedding and to skip single-file embedding failures instead of failing the whole index.

## Best Possible Solution

Implement provider-agnostic embedding limit handling in the memory indexing pipeline so inline batches respect actual provider limits and unrecoverable per-file embedding failures preserve the rest of the index with a warning.

## Reproduction Assessment

Yes, source-level reproduction is high confidence: configure an OpenAI-compatible embedding endpoint whose model allows only 2000 tokens per batch, then index enough memory chunks for OpenClaw to assemble a batch below its 8000 cap but above the provider cap. I did not run the live Qianfan endpoint in this read-only pass.

## Solution Assessment

Yes, the narrowest maintainable fix belongs in memory embedding batching and provider-limit adaptation, not in Qianfan chat-provider code. A safer implementation should split or retry oversized inline batches and only skip irreducible file failures after recording a clear warning.
