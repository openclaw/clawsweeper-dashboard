## Summary

Keep open: current main has a recent generic chunk-limit pass, but it still does not fully cover the reported OpenAI-compatible/Qianfan case where the endpoint enforces a 2,000-token combined embedding batch limit for an unknown model.

## What This Changes

The issue requests memory indexing to pre-chunk and batch embedding inputs within the configured model/provider limit so `openclaw memory index --force` does not fail the whole run on oversized embedding requests.

## Best Possible Solution

Add a generic provider-aware or configurable embedding batch cap for memory indexing, with regression coverage for an OpenAI-compatible endpoint that rejects combined batches above 2,000 tokens.

## Reproduction Assessment

Yes, source inspection gives a high-confidence reproduction path: use an OpenAI-compatible memory embedding model not in `OPENAI_MAX_INPUT_TOKENS`, index enough chunks to exceed a 2,000-token combined request, and have the endpoint return the reported 400 error. I did not run the live Qianfan endpoint in this read-only review.

## Solution Assessment

No, current main is only a partial solution because it caps known/provider-declared chunk sizes but still batches unknown OpenAI-compatible models up to 8,000 estimated tokens. The maintainable fix should make the effective inline batch cap reflect the actual provider/model limit instead of relying on the OpenAI defaults.
