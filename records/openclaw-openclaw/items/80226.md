## Summary

Keep this open: current main and the latest release still use the generic 8000/8192 embedding budget for unknown OpenAI-compatible embedding models, and the issue is already paired with an open fix PR using closing syntax.

## What This Changes

Memory indexing should split or adapt embedding batches for providers such as Qianfan bge-large-zh that reject batches around a 2000-token or batch-size limit.

## Best Possible Solution

Review and either repair/land https://github.com/openclaw/openclaw/pull/80241 or replace it with a narrow memory-core fix that adaptively splits concrete provider size failures, preserves provider/quota failures as command failures, and includes real low-limit endpoint proof.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: configure an OpenAI-compatible embedding endpoint whose model rejects combined batches around 2000, then index enough chunks for current main to build a batch below 8000 but above that provider limit.

## Solution Assessment

Yes, keeping this issue open for the linked implementation PR is the right path. The maintainable fix belongs in memory-core embedding batching and error classification rather than in Qianfan-specific provider code.
