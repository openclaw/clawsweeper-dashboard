## Summary

Keep this open: current main and the latest release still let unknown OpenAI-compatible embedding models use the generic 8k input budget and then send inline aggregate batches up to 8k, which matches the reported 2k-provider-limit failure path.

## What This Changes

The issue requests chunking and failure isolation for `openclaw memory index --force` when an OpenAI-compatible embedding provider rejects oversized embedding batches.

## Best Possible Solution

Fix memory embedding batching so OpenAI-compatible providers with lower aggregate input budgets split or retry inline batches safely, preserve successful files, and warn only on irreducible per-file size errors.

Yes. Source-level reproduction is high confidence: configure an OpenAI-compatible memory embedding endpoint with a 2000-token aggregate batch limit and index enough uncached text chunks for OpenClaw to build a batch under its 8000 cap but over the provider limit; I did not run the live Qianfan endpoint.

## Solution Assessment

Yes. The narrow fix belongs in memory embedding batch sizing and adaptive recovery, not in Qianfan-specific chat/provider routing; it should avoid a new user-facing config knob if provider metadata or retry splitting can cover the case.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/80226. Current memory indexing lets unknown OpenAI-compatible embedding models fall back to an 8192 per-input budget and then sends inline aggregate batches up to `EMBEDDING_BATCH_MAX_TOKENS` (8000), which fails for providers such as Qianfan bge-large-zh that reject aggregate embedding batches over 2000 tokens. Implement provider-agnostic handling in the memory embedding pipeline: respect declared provider limits when present, adaptively split/retry oversized inline batches when the provider reports an input/batch-size error, and preserve already-successful files while warning/skipping only irreducible per-file size failures. Do not hardcode Qianfan, do not add a new config option unless source review proves it is unavoidable, and keep core plugin-agnostic. Add focused regression coverage for unknown OpenAI-compatible lower aggregate limits and command/indexing failure isolation. Add a single CHANGELOG fix entry if the change is user-facing.
