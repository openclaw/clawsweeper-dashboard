## Summary

Keep this open: current main and the latest release still batch unknown OpenAI-compatible memory embeddings against the generic 8k budget, so the reported 2k provider-limit failure remains source-reproducible; an open linked PR already owns the fix path.

## What This Changes

The issue asks for memory indexing to chunk embedding inputs to provider limits and warn/skip irreducible per-file failures instead of aborting the whole index.

## Best Possible Solution

Land a focused memory-core fix that adaptively splits concrete provider batch-size failures, preserves provider/quota failures as command failures, and records clear warnings only for irreducible per-file inputs.

## Reproduction Assessment

Yes. Configure an OpenAI-compatible embedding provider without maxInputTokens and a roughly 2000-token batch limit, then index missing chunks totaling more than that but under OpenClaw's 8000 inline cap; current main will send them as one batch.

## Solution Assessment

Yes, the right fix belongs in memory embedding batching and provider-limit adaptation rather than a Qianfan-only hardcoded cap. The open linked PR is the current implementation path, but it still needs review and proof before this issue can close.
