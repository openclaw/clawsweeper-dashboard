## Summary

Keep open: current main and the latest release still batch unknown OpenAI-compatible embeddings up to the generic 8k budget and do not adaptively split the reported 2k provider-size error. The open linked PR is the current implementation path, so this issue should stay open until that fix or a replacement lands.

## What This Changes

Requests memory indexing to pre-chunk or adaptively split embedding batches for low-limit OpenAI-compatible providers and skip irreducible per-file failures instead of aborting the whole index.

## Best Possible Solution

Land a narrow memory-core fix through https://github.com/openclaw/openclaw/pull/80241 or a replacement once it has corrected review blockers and real low-limit endpoint proof, then close this issue after merge.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: configure an OpenAI-compatible embedding provider without maxInputTokens and a combined batch cap around 2000, then index enough memory chunks for current main to build a batch below 8000 but above that provider cap; I did not run Qianfan live.

## Solution Assessment

Yes. The maintainable direction is provider-agnostic adaptive splitting and per-file isolation in memory-core, not hardcoded Qianfan behavior; the existing linked PR is the right work queue but still needs review/proof before landing.
