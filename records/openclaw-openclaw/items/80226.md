## Summary

Keep this open: current main still batches inline memory embedding chunks against a generic 8000-byte/token budget for unknown OpenAI-compatible models, which leaves the reported low-limit Qianfan failure path intact. The issue is already paired with an open closing PR, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that `openclaw memory index --force` can fail against a low-limit OpenAI-compatible embedding endpoint because memory indexing does not split provider-rejected embedding batches before sending them.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/80241 after its classifier and real-behavior proof blockers are resolved, or replace it with an equivalent narrow memory-core fix.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: configure an OpenAI-compatible embedding provider with a combined batch limit below OpenClaw's generic 8000 cap, then index enough uncached memory chunks to produce a provider-rejected batch.

## Solution Assessment

Yes, the best fix belongs in provider-agnostic memory-core batch splitting and error classification rather than in Qianfan-specific defaults. A hardcoded item-count cap is less robust than adaptive split-and-retry for concrete provider size failures.
