## Summary

Keep open: the iMessage staging direction is useful and the PR has real proof, but the latest diff still leaves HEIC conversion unbounded and also mixes in an unrelated core target-prefix change that regresses the existing plugin-prefix routing contract.

## What This Changes

The PR stages local native iMessage inbound attachments into managed inbound media, converts HEIC/HEIF images to JPEG before agent dispatch, preserves remote raw-path handling, updates tests, and adds a changelog entry.

## Best Possible Solution

Land the plugin-owned staging path after using a bounded conversion helper or bounded runExec call, and keep service/chat target parsing in the iMessage plugin or SDK contract rather than hardcoding bundled target grammar in core.

## Reproduction Assessment

Yes. Current main is source-reproducible: iMessage forwards allowed original_path values into media fields while the media tools do not add arbitrary source parents to local roots; the PR body adds real macOS terminal proof for the staged HEIC-to-JPEG path.

## Solution Assessment

No, not as-is. Staging attachments inside the iMessage plugin is the right boundary, but the converter needs bounded process execution and the unrelated core target-prefix change should be removed or handled through plugin-owned parsing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Bound the HEIC conversion process:** `extensions/imessage/src/monitor/media-staging.ts:79-91`
  - body: `execFileAsync("sips", ...)` is still awaited without timeout or maxBuffer while staging inbound attachments. Node's execFile default timeout is `0`, so a stalled conversion can hold the monitor before dispatch; use the existing bounded `runExec` pattern or a bounded SDK conversion helper so failure falls back cleanly.
  - confidence: 0.88
- **[P2] Keep iMessage target grammar out of core routing:** `src/infra/outbound/channel-target-prefix.ts:18-21`
  - body: Adding `imessage`, `sms`, and chat-id forms to the generic core prefix block makes `resolveTargetPrefixedChannel("imessage:+1555")` ignore a plugin that advertises that prefix, reversing the existing cron delivery contract. This attachment PR should leave provider-prefix routing plugin-driven, or move any iMessage service-prefix handling into the iMessage parser.
  - confidence: 0.8
