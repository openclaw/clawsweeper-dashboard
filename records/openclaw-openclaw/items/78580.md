## Summary

Keep this PR open: the user-facing iMessage media fix is valid and has real terminal proof, but the latest head still stages local attachment bytes after only a string root match and needs canonical root validation before merge.

## What This Changes

The PR stages native local iMessage inbound attachments into managed inbound media, converts HEIC/HEIF images to JPEG, preserves remote raw-path handling, updates tests, adjusts target-prefix handling, and adds a changelog entry.

## Best Possible Solution

Keep the plugin-owned staging approach, but require realpath-based root validation and symlink-parent regression coverage for local iMessage staging before merge.

## Reproduction Assessment

Yes, source reproducible: current main forwards allowed native iMessage attachments as raw `original_path` values, while image-tool local roots do not include the Messages attachment tree. I did not run a live macOS iMessage Gateway in this read-only review.

## Solution Assessment

No, not yet. Owner-boundary staging is the right maintainable direction, but the patch should canonicalize and revalidate the resolved local attachment path before conversion, read, or managed-media save.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Canonicalize attachments before staging:** `extensions/imessage/src/monitor/media-staging.ts:71`
  - body: The helper receives paths that were only string-matched against `attachmentRoots`, then `lstat`s the final path and later reads or converts the same path. A symlinked parent directory under an allowed root will still be followed, so an allowed-looking Messages path can stage content outside the intended iMessage roots. Please resolve the real path and re-check it against the effective local attachment roots before conversion/read/save.
  - confidence: 0.87

## Security Review

Status: needs_attention

Concerns:

- **[medium] Revalidate canonical attachment paths:** `extensions/imessage/src/monitor/media-staging.ts:71`

Because this PR copies the file into OpenClaw-managed media before downstream readers apply their own local-root checks, a locally altered allowed-looking path could stage content outside the intended iMessage attachment roots.
