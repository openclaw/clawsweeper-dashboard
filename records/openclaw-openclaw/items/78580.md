## Summary

Keep open: the PR fixes a source-reproducible iMessage media bug and includes real terminal proof, but the latest head still stages files after only string-root filtering and final-path `lstat`, leaving a canonical-path bypass risk.

## What This Changes

The PR stages local native iMessage inbound attachments into managed inbound media, converts HEIC/HEIF via `sips`, preserves remote raw paths, adjusts target-prefix handling, and adds tests plus a changelog entry.

## Best Possible Solution

Keep the plugin-owned staging and HEIC conversion approach, but require canonical realpath validation against the effective local iMessage attachment roots before any conversion, read, or managed-media save.

## Reproduction Assessment

Yes, source reproducible: current main forwards allowed native iMessage attachments as raw `original_path` values, while media tools intentionally do not trust arbitrary source paths as local roots. I did not run a live macOS Gateway in this read-only review.

## Solution Assessment

No, not yet. The staging direction is the right owner-boundary fix, but the helper should resolve the canonical file path and re-check it against the effective local attachment roots before reading or converting.

## Review Findings

Overall correctness: patch is incorrect

  - body: `stageIMessageAttachments` receives paths admitted by a string root check, then `lstat`/`readFile`/`sips` the original path. `lstat` rejects only the final path if it is a symlink; a symlinked parent under an allowed Messages root is still followed and copied into managed inbound media.
