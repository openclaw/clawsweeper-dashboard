## Summary

Keep this PR open: it addresses a valid source-reproducible iMessage attachment bug and supplies sufficient real terminal proof, but the latest diff still has two narrow blockers before merge: unbounded HEIC conversion and an unrelated core target-prefix routing change.

## What This Changes

The PR stages native iMessage inbound attachments into managed media, converts HEIC/HEIF images to JPEG before dispatch, preserves remote raw-path handling, and adjusts related tests/changelog.

## Best Possible Solution

Land the plugin-owned staging path after bounding HEIC conversion and removing or separately designing the unrelated core target-prefix change.

## Reproduction Assessment

Yes. Current main source inspection shows iMessage forwards allowed `original_path` attachment values while media tools do not trust arbitrary source parents, and the PR body supplies real macOS terminal proof for the fixed staging path.

## Solution Assessment

No, not as-is. Staging native iMessage media inside the plugin is the right boundary, but the converter needs bounded process execution and the target-prefix routing change should be removed or handled as a separate plugin/core contract decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Bound the HEIC conversion process:** `extensions/imessage/src/monitor/media-staging.ts:79-91`
  - body: `execFileAsync("sips", ...)` is awaited inline without a timeout, and Node's default timeout is `0`. Because this runs before inbound dispatch, a stalled HEIC conversion can block the native iMessage monitor instead of falling back; pass bounded exec options or use an existing bounded conversion helper.
  - confidence: 0.88
- **[P2] Keep service prefixes out of the core blocklist:** `src/infra/outbound/channel-target-prefix.ts:19-21`
  - body: Adding `imessage`/`sms` service prefixes to the generic blocklist makes core ignore a plugin that advertises those target prefixes, reversing the existing cron/session delivery contract for `channel: last`. This attachment PR should drop the core routing change or move service-specific parsing into the iMessage-owned target normalization path.
  - confidence: 0.8
