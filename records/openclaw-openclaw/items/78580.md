## Summary

Keep open: the PR fixes a real iMessage attachment-delivery bug, but the latest head still awaits an unbounded `sips` conversion while staging inbound media, so one concrete P2/security blocker remains before merge.

## What This Changes

The PR stages native iMessage inbound attachments into managed inbound media, converts HEIC/HEIF images to JPEG, preserves remote-host raw paths, adds staging/policy tests, and updates target-prefix behavior plus the changelog.

## Best Possible Solution

Land the plugin-owned staging path after the HEIC converter uses bounded execution and has focused regression coverage, while keeping canonical root checks and remote-host raw-path behavior intact.

## Reproduction Assessment

Yes. Current main is source-reproducible because it forwards valid iMessage `original_path` values while media-tool roots intentionally do not trust arbitrary source parents, and the PR supplies real macOS terminal proof for the fixed staging path.

## Solution Assessment

No, not as-is. The plugin-owned staging direction is appropriate, but the direct `sips` call needs a timeout and output bound, or reuse of the existing bounded image conversion path, so conversion failure falls back instead of hanging the monitor.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Bound the HEIC conversion process:** `extensions/imessage/src/monitor/media-staging.ts:79-91`
  - body: `execFileAsync("sips", ...)` is awaited while staging inbound attachments, but no timeout is passed and Node's default `execFile` timeout is `0`. Because the monitor awaits staging before dispatch, a stuck conversion can block iMessage delivery; use a bounded exec path or pass timeout/maxBuffer options and cover that behavior in the staging tests.
  - confidence: 0.87
