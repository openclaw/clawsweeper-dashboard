## Summary

Keep open: the fix is useful and has real proof, but the latest head still awaits an unbounded HEIC conversion process before iMessage dispatch, so the PR should not merge as-is.

## What This Changes

The PR stages native iMessage inbound attachments into OpenClaw-managed media, converts HEIC/HEIF photos to JPEG, preserves remote-host raw-path handling, adjusts target-prefix behavior, and adds iMessage media staging tests.

## Best Possible Solution

Land the plugin-owned staging path after bounding HEIC conversion with a timeout/maxBuffer or shared converter, while preserving canonical root checks, fallback behavior, and the remote raw-path contract.

## Reproduction Assessment

Yes. Current main is source-reproducible because it forwards allowed iMessage `original_path` media values while media-tool roots intentionally do not trust arbitrary source parents, and the PR body supplies real macOS terminal proof for the fixed staging path.

## Solution Assessment

No, not as-is. The plugin-boundary staging direction is narrow and maintainable, but the direct `sips` child process must be bounded so conversion failure falls back instead of hanging the monitor.
