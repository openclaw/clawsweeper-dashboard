## Summary

Keep open: the PR fixes a real iMessage attachment delivery problem and has sufficient real behavior proof, but the latest head still introduces an unbounded `sips` conversion process on the inbound dispatch path.

## What This Changes

The PR stages local native iMessage inbound attachments into managed inbound media, converts HEIC/HEIF to JPEG, preserves remote raw-path handling, updates target-prefix behavior, adds tests, and adds a changelog entry.

## Best Possible Solution

Land the plugin-owned staging path after bounding HEIC conversion with a timeout/maxBuffer or reusing the SDK HEIC conversion helper, while preserving canonical root checks and remote raw-path behavior.

## Reproduction Assessment

Yes. Current main is source-reproducible: it forwards allowed iMessage `original_path` values while media-tool roots do not trust arbitrary Messages attachment parents, and the PR body supplies real terminal proof for the fixed staging path.

## Solution Assessment

No, not as-is. The owner-boundary staging approach is the right direction, but the direct `sips` call needs bounded execution or the existing SDK image conversion path so conversion failure can fall back instead of hanging dispatch.

## Review Findings
