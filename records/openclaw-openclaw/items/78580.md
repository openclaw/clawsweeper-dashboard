## Summary

Keep open. The staging direction is correct and the supplied real behavior proof is sufficient, but the latest diff runs `sips` without a timeout on the inbound iMessage path, so this needs a narrow repair before merge.

## What This Changes

This PR stages local native iMessage inbound attachments into managed inbound media, converts HEIC/HEIF attachments to JPEG, preserves remote raw-path handling, adjusts target-prefix behavior, adds tests, and updates the changelog.

## Best Possible Solution

Land the plugin-owned staging path after bounding HEIC conversion with a timeout/maxBuffer or reusing the SDK image conversion helper, while preserving canonical root checks and remote raw-path behavior.

## Reproduction Assessment

Yes. Current main is source-reproducible: it forwards allowed iMessage `original_path` media values while media-tool roots do not trust arbitrary source parents, and the PR body supplies real macOS terminal proof for the fixed staging path.

## Solution Assessment

No, not as-is. The owner-boundary fix is right, but the direct `sips` call needs the same bounded execution behavior as existing image conversion code so conversion failure falls back instead of hanging the monitor.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Bound the HEIC conversion process:** `extensions/imessage/src/monitor/media-staging.ts:79-91`
  - body: `execFileAsync("sips", ...)` is awaited inline while staging inbound attachments, but no timeout is passed, and Node defaults `execFile` to `timeout: 0`. Existing OpenClaw `sips` image helpers use bounded `runExec` calls; without a bound here, a stuck HEIC conversion can keep the iMessage monitor from dispatching the message or later queued work. Please use a bounded exec path or the SDK HEIC converter so this fails and falls back instead of hanging.
  - confidence: 0.86
