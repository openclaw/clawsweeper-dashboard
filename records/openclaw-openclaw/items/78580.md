## Summary

Keep open: the PR addresses a valid iMessage inbound media bug and supplies real terminal proof, but the new local staging path needs a small security/correctness fix before merge because it reads the accepted path without canonical root revalidation.

## What This Changes

The PR stages native iMessage inbound attachments into OpenClaw-managed inbound media, converts local HEIC/HEIF files to JPEG before dispatch, preserves remote-host raw paths, adds iMessage tests, and updates the changelog.

## Best Possible Solution

Keep the PR open, add canonical root validation to local attachment staging, preserve the post-policy staging and remote raw-path behavior, then land after the focused iMessage tests and exact-head CI pass.

## Reproduction Assessment

Yes, source reproducible: current main forwards allowed native iMessage attachments as raw `original_path` values, while the image tool's local roots do not include the Messages attachment tree. I did not run a live macOS iMessage gateway in this read-only review.

## Solution Assessment

No, not yet. Staging local attachments at the iMessage owner boundary is the right maintainable direction, but the patch should canonicalize and revalidate the attachment path before `fs.readFile`/`saveMediaBuffer`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Canonicalize attachments before staging:** `extensions/imessage/src/monitor/media-staging.ts:71`
  - body: The monitor validates the original string path against `attachmentRoots`, but this new helper then only `lstat`s the final path component before `fs.readFile`. If any parent directory under an allowed attachment root is a symlink, the read follows it and copies a file outside the allowed roots into managed media. Please re-check the canonical `realpath` against the effective iMessage attachment roots, or use an equivalent safe-read helper, before saving the buffer.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff avoids new dependencies and shell execution, but local staging needs canonical path validation before reading attachment bytes.

Concerns:

- **[medium] Revalidate canonical attachment paths:** `extensions/imessage/src/monitor/media-staging.ts:71`
  - body: `fs.lstat` on the final path does not catch symlinked parent directories. Since the PR copies the file into OpenClaw-managed media before downstream readers can canonicalize it, malformed or locally altered attachment paths under an allowed root could stage content outside the intended iMessage attachment roots.
  - confidence: 0.84
