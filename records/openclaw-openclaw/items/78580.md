## Summary

Keep open. The PR addresses a valid current-main iMessage media bug and includes sufficient terminal proof, but it still has two merge-blocking issues: HEIC conversion uses an unbounded child process, and the branch changes generic core target-prefix routing outside the attachment fix.

## What This Changes

The PR stages native iMessage inbound attachments into managed inbound media, converts HEIC/HEIF photos to JPEG before dispatch, preserves remote-host raw paths, updates tests, changes target-prefix routing behavior, and adds a changelog entry.

## Best Possible Solution

Land the plugin-owned staging and HEIC-to-JPEG path after bounding conversion execution and removing the unrelated core target-prefix behavior change.

## Reproduction Assessment

Yes. Current main is source-reproducible: it forwards valid iMessage original_path values into MediaUrl/MediaPaths while the image tool roots stay limited to default/workspace roots, and the PR discussion adds real v2026.5.12-beta.3 HEIC evidence.

## Solution Assessment

No, not as-is. Staging attachments in the iMessage plugin is the right boundary, but the converter needs bounded process execution and the target-prefix routing change should be removed or handled through plugin-owned parsing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Bound the HEIC conversion process:** `extensions/imessage/src/monitor/media-staging.ts:79-91`
  - body: `execFileAsync("sips", ...)` is awaited with no timeout or maxBuffer while staging inbound media. Node's execFile timeout default is 0, so a stuck conversion can hold the native iMessage monitor before dispatch; use a bounded process helper such as `openclaw/plugin-sdk/process-runtime` or pass explicit limits.
  - confidence: 0.88
- **[P2] Keep service prefixes out of core routing:** `src/infra/outbound/channel-target-prefix.ts:6-23`
  - body: Adding iMessage service/chat prefixes to the generic target-kind block makes `resolveTargetPrefixedChannel("imessage:+1555")` ignore a plugin that advertises `targetPrefixes: ["imessage"]`, reversing the existing cron delivery contract. This attachment fix should leave provider-prefix routing unchanged or keep service grammar inside the iMessage plugin parser.
  - confidence: 0.8
