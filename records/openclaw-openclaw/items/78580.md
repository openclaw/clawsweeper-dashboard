## Summary

Keep open: the staging direction is useful and the supplied proof is sufficient, but the latest head still awaits an unbounded `sips` conversion before dispatching inbound iMessage media, which should block merge until bounded.

## What This Changes

The PR stages native iMessage inbound attachments into managed media, converts local HEIC/HEIF attachments to JPEG, preserves remote-host raw paths, adjusts target-prefix handling, adds tests, and updates the changelog.

## Best Possible Solution

Land the plugin-owned staging path after bounding HEIC conversion with a timeout and output cap or by reusing the existing SDK media conversion helper, while preserving canonical root checks and remote raw-path behavior.

## Reproduction Assessment

Yes. Current main is source-reproducible because it forwards allowed iMessage `original_path` values while media-tool roots do not trust arbitrary source parents, and the PR body supplies terminal proof for the fixed staging path.

## Solution Assessment

No, not as-is. The owner-boundary fix is the right shape, but the direct `sips` call needs bounded execution so conversion failure falls back instead of hanging inbound dispatch.
