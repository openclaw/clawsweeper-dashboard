## Summary

Keep open: the PR addresses a source-reproducible iMessage attachment bug and has real terminal proof, but the current patch still awaits an unbounded `sips` conversion on the inbound monitor path.

## What This Changes

The PR stages native iMessage inbound attachments into managed media, converts HEIC/HEIF images to JPEG, preserves remote raw-path handling, adds tests, and updates target-prefix/changelog coverage.

## Best Possible Solution

Land the plugin-owned staging path after bounding or reusing the existing SDK HEIC conversion helper, while preserving canonical root checks and the remote raw-path contract.

## Reproduction Assessment

Yes, at source level. Current main forwards allowed native iMessage `original_path` media values while image-tool roots stay bounded to default/workspace roots, so raw Messages attachment paths can reach tools outside trusted roots.

## Solution Assessment

No, not as-is. The staging boundary is appropriate, but the converter needs a timeout/maxBuffer or should reuse the existing SDK HEIC conversion path so failure falls back instead of hanging dispatch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Bound the HEIC conversion process:** `extensions/imessage/src/monitor/media-staging.ts:79-91`
  - body: `execFileAsync("sips", ...)` is awaited while staging inbound iMessage attachments, but the call passes no timeout and Node defaults `execFile` to `timeout: 0`. Existing OpenClaw `sips` helpers use bounded `runExec`; a stalled conversion here would hold the monitor before dispatching the message, so please use bounded exec or the SDK HEIC converter.
  - confidence: 0.86
