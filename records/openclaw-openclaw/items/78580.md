## Summary

Keep open: the iMessage staging direction is useful and current main still forwards raw native attachment paths, but the PR still awaits an unbounded HEIC `sips` conversion before dispatch, so it needs a narrow repair before merge.

## What This Changes

The branch stages local native iMessage inbound attachments into managed inbound media, converts HEIC/HEIF images to JPEG, preserves remote raw-path handling, extends target-prefix exclusions, adds regression tests, and updates the changelog.

## Best Possible Solution

Land the plugin-owned staging path after bounding `sips` with timeout/maxBuffer or a shared SDK conversion helper, while preserving canonical root checks and remote raw-path behavior.

## Reproduction Assessment

Yes. Current main is source-reproducible: allowed native iMessage attachments are forwarded from `original_path`, while media-tool local roots do not widen to arbitrary Messages attachment parents.

## Solution Assessment

No, not as-is. The staging location and plugin ownership boundary are right, but the HEIC converter must use bounded process execution so conversion failure falls back instead of hanging inbound dispatch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Bound the HEIC conversion process:** `extensions/imessage/src/monitor/media-staging.ts:79-91`
  - body: `execFileAsync("sips", ...)` is awaited inline while staging inbound attachments, but no timeout is passed and Node defaults `execFile` to `timeout: 0`. A stuck conversion can keep the iMessage monitor from dispatching this message or later queued work, so use a bounded exec path or shared converter and preserve the existing fallback behavior.
  - confidence: 0.87
