## Summary

Keep open. Current main has a reusable CLI progress primitive and startup/RPC coverage, but the requested UX improvement is still only partially implemented because slow Skills ClawHub and Plugins install/update paths still run through direct helpers or loggers instead of a consistent progress wrapper.

## What This Changes

The issue requests loading/progress indicators for slow OpenClaw CLI commands so users can tell whether work is still progressing.

## Best Possible Solution

Scope progress around the existing createCliProgress/withProgress contract for selected slow CLI boundaries, starting with Skills ClawHub search/install/update and Plugins install/update, while preserving TTY, non-TTY, JSON, and plain output semantics.

## Reproduction Assessment

No high-confidence runtime reproduction is available because the issue does not name exact commands or timings. Source inspection does confirm plausible current-main coverage gaps in Skills and Plugins CLI flows.

## Solution Assessment

Unclear. Adding loaders is directionally right, but the maintainable solution is a scoped rollout at command boundaries with output-mode tests rather than blanket progress output across the CLI.
