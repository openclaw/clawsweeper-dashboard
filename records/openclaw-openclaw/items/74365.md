## Summary

Keep open. Current main has a shared CLI progress helper and several slow command families already use it, but the broad loading-state request is not fully covered: Skills ClawHub search/install/update and plugin install/update paths still call slow helpers directly or rely on logger/lifecycle messages rather than the shared progress wrapper.

## What This Changes

The issue requests broader CLI loading/progress indicators so slow commands do not appear blank or stuck.

## Best Possible Solution

Use the existing shared progress helper as the CLI UI contract, then scope the rollout to selected slow command families such as Skills ClawHub search/install/update and plugin install/update with tests for TTY, non-TTY, JSON/plain output, and install/update diagnostics.

## Reproduction Assessment

No high-confidence runtime reproduction is available because the report lacks exact commands and timings. Source inspection does verify remaining progress coverage gaps in current main for Skills ClawHub and plugin install/update paths.

## Solution Assessment

Unclear. Adding loaders is directionally right, but the maintainable approach is a scoped rollout of withProgress or progress callbacks at selected command boundaries rather than blanket loaders across every CLI action.
