## Summary

Keep open. Current main has a shared CLI progress primitive and some startup/RPC coverage, but the request is still only partially satisfied because several ClawHub skill/plugin network and install/update command paths still run without the shared progress wrapper.

## What This Changes

The issue requests visible loading/progress states for slow OpenClaw CLI commands.

## Best Possible Solution

Use the existing ProgressReporter/withProgress contract at scoped slow command boundaries, especially ClawHub skill/plugin search/install/update paths, with tests for interactive and parseable output modes.

## Reproduction Assessment

No. The issue lacks exact commands and timings, so there is not a high-confidence runtime reproduction path; source inspection does confirm plausible current-main gaps in slow ClawHub skill/plugin command paths.

## Solution Assessment

Unclear. Adding loaders is directionally right, but the maintainable solution is a scoped rollout through the existing progress abstraction rather than blanket CLI output changes.
