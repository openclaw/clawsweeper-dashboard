## Summary

Keep open. Current main has a shared CLI progress helper and several adopted call sites, but the requested broad loading-state coverage is still only partial: Skills ClawHub search/install/update and Plugins install/update paths still go through direct helper calls, ordinary loggers, or trace-only timing rather than a consistent progress wrapper.

## What This Changes

Add loading/progress indicators to slow OpenClaw CLI commands so users can distinguish slow work from a hang.

## Best Possible Solution

Define the shared CLI progress helper as the UI contract, then roll it out to selected remaining slow command families with progress callbacks at command boundaries, preserving parseable output modes and updating docs/changelog only where user-visible behavior changes.

## Reproduction Assessment

No high-confidence runtime reproduction is available because the issue lacks exact commands and timing data. Source inspection does provide a high-confidence static reproduction of partial coverage: several slow Skills and Plugins paths on current main still do not use withProgress.

## Solution Assessment

Unclear. Adding loaders is directionally right, but the maintainable fix is not blanket progress across every command; it should be a scoped rollout through withProgress or progress-reporter callbacks where the command can safely preserve TTY, non-TTY, JSON, and plain output behavior.
