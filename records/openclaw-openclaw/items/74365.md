## Summary

Keep open. Current main has a shared CLI progress primitive and some command coverage, but the broad loading-state request is still incomplete: several slow skills and plugin install/update paths still do network or install work without command-boundary progress wrappers.

## What This Changes

The issue requests loading or progress indicators for slow OpenClaw CLI commands so users can tell commands are progressing instead of stalled.

## Best Possible Solution

Use the existing shared progress helper as the standard UI contract, choose a small set of slow command families first, and add mode-aware tests/docs so TTY progress improves without changing machine-readable output.

## Reproduction Assessment

No high-confidence runtime reproduction exists for the full issue because the report lacks command names and timings. Source inspection does provide high-confidence evidence of remaining coverage gaps in skills and plugin install paths on current main.

## Solution Assessment

Unclear. Adding loaders is directionally right, but the maintainable path is a scoped rollout of the existing progress helper at selected slow command boundaries rather than blanket loaders across every CLI action.
