## Summary

Keep open. Current main has a shared CLI progress helper and several slow command paths already use it, but the broad loading-state request is not fully satisfied: Skills ClawHub search/install/update and plugin install/update flows still rely on direct network/install helpers, logger messages, or lifecycle tracing rather than a consistent progress wrapper.

## What This Changes

Add visible loading/progress indicators to slow OpenClaw CLI commands so users can tell work is still progressing.

## Best Possible Solution

Use the existing shared progress helper as the CLI UI contract, then scope the rollout to selected slow command families such as Skills ClawHub search/install/update and plugin install/update with tests for TTY, non-TTY, JSON/plain output, and docs/changelog updates where behavior changes.

## Reproduction Assessment

No high-confidence runtime reproduction is available because the report lacks exact commands and timings. Source inspection does verify remaining progress coverage gaps in current main for Skills and Plugins paths while other slow paths already use the shared helper.

## Solution Assessment

Unclear. Adding loaders is directionally right, but the maintainable solution is a scoped rollout of withProgress/progress callbacks at selected command boundaries rather than blanket loaders across every CLI action.
