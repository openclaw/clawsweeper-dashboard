## Summary

Keep open. Current main has a shared CLI progress helper and several slow paths already use it, but the broad loading-state request is only partially covered: Skills ClawHub search/install/update and plugin install/update still rely on direct helpers, logger messages, or lifecycle tracing rather than a consistent progress wrapper.

## What This Changes

Request loading/progress indicators for slow OpenClaw CLI command paths so users can tell commands are still progressing.

## Best Possible Solution

Use the existing shared progress helper as the CLI UI contract, then scope rollout to selected remaining slow command families such as Skills ClawHub search/install/update and plugin install/update, with focused tests for TTY, non-TTY, JSON/plain output, and docs/changelog updates where behavior changes.

## Reproduction Assessment

No high-confidence runtime reproduction is available because the report lacks exact commands and timings. Source inspection does verify plausible current-main coverage gaps in Skills and Plugins paths while other slow paths already use the shared helper.

## Solution Assessment

Unclear. Adding loaders is directionally right, but the maintainable solution is a scoped rollout of withProgress/progress callbacks at selected command boundaries rather than blanket loaders across every CLI action.
