## Summary

Keep open. Current main has a shared CLI progress helper and several slow command families already use it, but the broad loading-state request is still only partially satisfied: Skills ClawHub search/install/update and plugin install/update paths still use direct helper calls, plain logger callbacks, or lifecycle tracing rather than consistent progress wrappers.

## What This Changes

The issue asks for user-visible loading/progress indicators around slow OpenClaw CLI commands so blank pauses are distinguishable from hangs.

## Best Possible Solution

Use the existing shared progress helper as the CLI UI contract, then scope the rollout to selected slow command families such as Skills ClawHub search/install/update and plugin install/update with explicit handling for TTY, non-TTY, JSON/plain output, existing logger messages, and lifecycle tracing.

## Reproduction Assessment

No high-confidence runtime reproduction is available because the report lacks exact commands and timings. Source inspection does verify remaining progress coverage gaps in current main while other slow paths already use the shared helper.

## Solution Assessment

Unclear. Adding loaders is directionally right, but the maintainable solution is a scoped rollout of withProgress/progress callbacks at selected command boundaries rather than blanket loaders across every CLI action.
