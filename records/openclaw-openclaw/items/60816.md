## Summary

Keep open: current main still serializes same-file upstream edit execution, but OpenClaw does not expose the requested third result state. The current wrapper only converts likely post-write failures to normal success or rethrows the original mismatch/error, and checked-in regression coverage still expects the already-at-target case to reject rather than return a noop.

## What This Changes

The issue requests a distinct non-error noop result when an edit request finds the requested final content already present, especially for concurrent identical edit calls.

## Best Possible Solution

Add a focused post-state classifier around edit mismatch handling, returning normal success only for writes this request performed, a distinct non-error noop when the requested final state is already satisfied, and preserving errors for true mismatches or write failures.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: run two identical `path` + `edits[]` edit calls against the same file so the queued follower sees `oldText` gone, or call edit when the target content is already present; current code rethrows or reports success, not noop.

## Solution Assessment

Yes. A narrow classifier in OpenClaw's recovery wrapper, or the same change upstream in Pi, is the best fit; broad global serialization or suppressing all mismatch errors would risk hiding real edit failures.
