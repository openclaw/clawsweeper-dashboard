## Summary

Keep open: current main still serializes same-file edits through upstream Pi, but it does not provide the requested third result state. A queued identical follower still hits the upstream missing-oldText path, and OpenClaw's recovery wrapper either converts a changed post-state to normal success or preserves the error when the file was already at the target state.

## What This Changes

The issue requests distinct non-error noop semantics for identical concurrent edit-tool calls when the desired final file content is already present.

## Best Possible Solution

Add a focused post-state classifier around edit mismatch handling: return normal success only when this request actually wrote, return a distinct non-error noop when the requested final state is already satisfied, and preserve errors for true mismatches or write failures. Cover canonical edits[] first, with flat input only if maintainers confirm that remains an OpenClaw contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: run two identical edits against the same file so the queued follower sees oldText gone, and run an already-up-to-date edit where the target content is present before the call; current recovery/test behavior shows those states are success/error rather than noop.

## Solution Assessment

Yes, a narrow classifier in OpenClaw's edit recovery wrapper or the same change upstream in Pi is the best fit. Broad global serialization or suppressing all mismatch errors would be less maintainable and could hide genuine failures.
