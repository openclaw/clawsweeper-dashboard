## Summary

Keep open: current main still lacks the requested noop result state, and the issue is already paired with the open implementation PR https://github.com/openclaw/openclaw/pull/77638 using closing syntax, so this should stay open until that PR is reviewed and merged or replaced.

## What This Changes

The issue asks for the edit recovery path to return a distinct non-error noop when an identical concurrent edit has already left the file in the requested final state.

## Best Possible Solution

Land or replace the linked focused fix so the edit wrapper reports success only for writes this request performed, reports a distinct noop when the target state is already satisfied, and preserves errors for true mismatches or write failures.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: an identical queued `path` plus `edits[]` follower sees `oldText` gone after the leader, and current main still rethrows or hints the mismatch instead of returning a noop.

## Solution Assessment

Yes. A narrow classifier in the edit recovery wrapper, or the same fix upstream in Pi, is the maintainable solution; broad global serialization or suppressing all mismatches would hide real edit failures.
