## Summary

Current main still lacks a distinct noop branch for already-at-target edit followers, and an open linked PR explicitly closes this issue but has not merged, so the issue should stay open while that PR is reviewed or repaired.

## What This Changes

The issue requests edit-tool result semantics that return success for the edit that writes, noop for an identical concurrent follower that finds the desired final state, and errors for true mismatches or write failures.

## Best Possible Solution

Land a focused fix in the current built-in edit tool that returns a separate noop result only when readback proves the requested target state is already satisfied, with regression coverage for single-edit, edits[], substring replacements, true mismatch, and non-identical concurrent edits.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: same-file edits serialize, the follower reads the already-updated file, applyEditsToNormalizedContent throws the exact-text mismatch, and edit.ts decorates that mismatch instead of returning noop.

## Solution Assessment

Yes in direction, but not yet merged. The narrow maintainable fix belongs in src/agents/sessions/tools/edit.ts as post-state classification, not as a broad dedupe layer or new global serialization behavior.
