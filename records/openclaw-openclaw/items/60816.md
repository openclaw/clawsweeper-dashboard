## Summary

Keep open: current main still has only success/error edit recovery classification, and the live closing PR for the requested noop semantics is still open rather than merged.

## What This Changes

The issue asks the edit tool to return a distinct noop result when an identical concurrent edit finds the file already in the requested target state, for both single-replace and `edits[]` inputs.

## Best Possible Solution

Review and land https://github.com/openclaw/openclaw/pull/77638 after its review finding is resolved, or replace it with an equivalent narrow wrapper/test fix, then close this issue after the fix reaches main.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: same-file edit calls are queued upstream, and a follower whose target is already present reaches OpenClaw’s recovery wrapper with no distinct noop branch on current main.

## Solution Assessment

Yes. The narrow maintainable solution is a post-state classifier in the edit recovery wrapper that returns noop only when the requested final state is satisfied while preserving real mismatch and write failures.
