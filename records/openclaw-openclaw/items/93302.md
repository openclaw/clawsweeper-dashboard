## Summary

Keep open: the PR fixes the reported stale-root shape only by adding a broad unit-missing fallback, so any sudo-to-root machine-scope unit-not-found result can now fall through to root's user manager. That is compatibility-sensitive, and the external proof is still a mocked Vitest harness rather than live Linux systemd behavior.

## What This Changes

The PR changes the daemon systemd helper to retry bare `--user` when a sudo-to-root machine-scoped `systemctl` call reports the unit missing, with a regression test for restart.

## Best Possible Solution

Keep the machine-scope default, add a root-owned stale-sudo fallback only when the current/root unit is clearly the intended owner, and prove both root-owned and sudo-caller flows on live Linux.

## Reproduction Assessment

Yes for source reproduction: current main prefers non-root `SUDO_USER` under effective root, and PR head falls back to bare `--user` on unit-missing. I did not run a live Linux systemd reproduction in this read-only review.

## Solution Assessment

No. The fallback is a plausible mitigation for the root-owned stale-sudo report, but the best fix should preserve sudo caller scope unless root's own service is clearly the intended target.
