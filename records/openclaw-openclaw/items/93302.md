## Summary

Keep open: the linked bug is valid, but this patch still retargets every sudo-to-root unit-missing machine-scope result to root's user manager and only supplies mocked proof, so it remains compatibility- and availability-risky before merge.

## What This Changes

The branch changes `execSystemctlUser` to retry bare `systemctl --user` after a sudo-to-root machine-scoped unit-missing result and adds a mocked restart regression test.

## Best Possible Solution

Preserve machine-scope sudo behavior by default, only fall back to root's `--user` when the root-owned unit is clearly the target, and prove both root-owned stale-sudo and sudo-caller cases on live Linux.

## Reproduction Assessment

Yes for source reproduction: current main and v2026.6.6 prefer non-root `SUDO_USER` under effective root, while the linked issue supplies live before/after stale-sudo evidence. I did not run a fresh live Linux systemd reproduction in this read-only review.

## Solution Assessment

No. The fallback is a plausible mitigation, but the best fix should preserve sudo-caller scope unless root's own service is clearly the intended target.
