## Summary

Keep open: the PR targets a real gateway/systemd bug, but the current diff is not merge-ready because the new unit-file guard can suppress the established sudo-to-root machine scope for sudo-driven installs and lifecycle commands, and the supplied proof is mock-only.

## What This Changes

The PR changes systemd user-scope resolution to use bare `--user` when the resolved gateway unit path exists under the current environment's `HOME`, and adds a Vitest case for stale `SUDO_USER` with `/root`.

## Best Possible Solution

Keep the root-owned stale-`SUDO_USER` fix, but narrow the guard so sudo-driven non-root user unit installs and lifecycle commands still prefer the sudo caller's machine scope, with regression tests and live proof.

## Reproduction Assessment

Source-level yes: the PR-introduced risk follows from `installSystemdService` writing a unit under `env.HOME` before activation and the new guard treating any resolved unit as current-root ownership. I did not run live systemd commands or tests because this review is read-only.

## Solution Assessment

No. The requested stale-root-shell behavior is valid, but unit-file existence alone is too broad; the fix should distinguish root's own unit from a sudo caller's user unit or pass the already-detected installed scope into the resolver.
