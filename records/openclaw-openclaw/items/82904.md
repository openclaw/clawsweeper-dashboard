## Summary

Keep open. Current main still uses the weaker HTTP reachability probe to decide managed Chrome launch failure, then calls the stronger CDP diagnostic only for the error text, so the reported self-contradicting failure remains plausible and source-backed.

## What This Changes

The issue requests that managed Chrome launch readiness use the same CDP diagnostic result for both the success/failure decision and the error message.

## Best Possible Solution

Make the final managed-launch readiness decision from one captured `diagnoseChromeCdp` result, preserve existing singleton-lock and stderr handling, and add a regression test for a failed final HTTP probe with a ready diagnostic.

## Reproduction Assessment

Yes, from source: current main can throw after a final `isChromeReachable` false while embedding a later successful `diagnoseChromeCdp` result in the error. I did not run the intermittent macOS cold-start path live.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable fix: keep or replace the lightweight polling as needed, but make the final decision and message use the same diagnostic result.
