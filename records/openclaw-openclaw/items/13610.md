## Summary

Keep open. Current main and the latest release still expose the shipped SecretRef baseline for env, file, and exec providers only, while native AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, and Google Cloud Secret Manager support remains unresolved security-sensitive provider architecture work.

## What This Changes

The issue requests native integration with external secrets managers such as AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, and Google Cloud Secret Manager.

## Best Possible Solution

Keep this as the native secrets-provider umbrella until maintainers either land a reviewed plugin SDK seam for external providers or explicitly decide to own AWS/Vault/Azure/GCP providers in core with aligned schemas, CLI, docs, tests, and live-provider proof.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is current-main and latest-release source/docs inspection, which shows supported SecretRef/provider sources remain env, file, and exec only.

## Solution Assessment

Unclear pending maintainer and secops API direction. Native provider support is a reasonable goal, but the maintainable path should be decided between the open plugin-provider seam and explicit core-owned vendor providers before implementation.
