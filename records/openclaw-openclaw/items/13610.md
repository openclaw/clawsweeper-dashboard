## Summary

Keep open: current main and the latest shipped baseline support env/file/exec SecretRefs, but native AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, and Google Cloud Secret Manager support remains unresolved and security-sensitive.

## What This Changes

The issue requests native external secrets-manager integrations so OpenClaw config can reference managed secret stores instead of plaintext credential values.

## Best Possible Solution

Keep this issue as the native secrets-manager umbrella until maintainers choose a security-reviewed provider/plugin contract, then split AWS, Vault, Azure, GCP, and other backends into focused provider follow-ups with redacted live backend proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source inspection gives a high-confidence current-main check that native cloud/Vault SecretRef sources are absent and the supported contract remains env/file/exec.

## Solution Assessment

Unclear pending maintainer direction. A plugin or ClawHub provider seam fits OpenClaw's architecture better than bundling every vendor directly in core, but the exact public contract and security boundaries still need approval.

## Risks / Open Questions

- Native provider support changes credential lookup, external dependency, caching, audit, redaction, activation, and failure semantics, so it needs secops review before implementation.
- There are at least two open implementation directions, pluggable SecretRef sources and manifest-level exec provider integrations, so maintainers need to choose the public contract before this umbrella can close.
- Exec providers can wrap Vault, 1Password, cloud CLIs, or custom resolvers today, but that workaround does not provide native source types, provider-specific schema, configure/audit UX, or provider-owned tests for AWS/Vault/Azure/GCP.
