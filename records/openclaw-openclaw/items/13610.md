## Summary

Keep open: current and latest shipped code implement the external-secrets baseline with env/file/exec only, while native cloud/Vault providers and a provider/plugin contract remain unresolved security-sensitive product work.

## What This Changes

The issue requests native external secret-manager integration for AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, Google Cloud Secret Manager, and formalized environment-variable secret references.

## Best Possible Solution

Keep this as the native secrets-provider umbrella until maintainers approve a security-reviewed SecretRef provider/plugin contract, then split AWS, Vault, Azure, GCP, and similar backends into focused provider/plugin follow-ups with live backend proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source, docs, tests, release inspection, and issue context give high confidence that native AWS/Vault/Azure/GCP SecretRef sources are absent and the supported contract remains env/file/exec.

## Solution Assessment

Unclear until maintainers choose the security and product contract. A plugin or ClawHub provider seam fits OpenClaw's lean-core direction, but credential-provider execution is security-sensitive and needs explicit secops/product approval before implementation.
