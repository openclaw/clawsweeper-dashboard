## Summary

Keep open: current main and the latest shipped tag have the env/file/exec SecretRef baseline plus documented exec-wrapper paths, but native AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, and Google Cloud Secret Manager support still needs a security-reviewed provider/API decision.

## What This Changes

Request native external secrets-manager integrations for AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, Google Cloud Secret Manager, and related SecretRef resolution, caching, fallback, and audit behavior.

## Best Possible Solution

Keep this as the native secrets-manager provider umbrella until maintainers approve a security-reviewed SecretRef/provider contract, then split AWS, Vault, Azure, GCP, and similar backends into focused provider or plugin follow-ups with live backend proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source, docs, release inspection, and live issue/PR context give high confidence that native AWS/Vault/Azure/GCP SecretRef sources are absent and the supported contract remains env/file/exec.

## Solution Assessment

Unclear until maintainers choose the security and product contract. A plugin or ClawHub provider seam fits OpenClaw's lean-core direction, but credential-provider execution is security-sensitive and needs explicit secops/product approval before implementation.
