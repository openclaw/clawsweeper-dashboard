## Summary

Keep open. Current main ships the SecretRef baseline for env, file, and exec sources, but native AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, and Google Cloud Secret Manager provider sources are still not implemented, and recent maintainer discussion keeps this as the umbrella.

## What This Changes

Request native external secrets-manager provider integrations with lazy loading, caching, fallback errors, audit logging, and config syntax for cloud/Vault/env-backed secrets.

## Best Possible Solution

Keep this as the native secrets-provider umbrella until maintainers choose either a reviewed plugin SDK provider seam or explicit core-owned AWS/Vault/Azure/GCP providers with aligned contracts.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is current-main source and docs inspection, which shows SecretRef/provider sources remain env, file, and exec only.

## Solution Assessment

Unclear pending maintainer and secops direction. Exec wrappers are a useful bridge, but native provider semantics need either a reviewed plugin-provider seam or an explicit core decision to own AWS, Vault, Azure, and GCP integrations.
