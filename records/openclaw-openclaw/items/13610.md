## Summary

Keep open. Current main implements the shipped SecretRef baseline for env/file/exec providers, but native AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, and Google Cloud Secret Manager support is still absent and the plugin-provider seam remains an open maintainer/security decision.

## What This Changes

The issue requests first-class external secrets-manager integrations so OpenClaw config can reference cloud/Vault-backed secrets instead of storing plaintext credentials.

## Best Possible Solution

Keep this as the native secrets-provider umbrella until maintainers choose either a reviewed plugin SDK provider seam with provider implementations outside core or explicit core-owned AWS/Vault/Azure/GCP providers with aligned contracts, docs, and tests.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection on current main, which shows supported SecretRef/provider sources remain env, file, and exec only.

## Solution Assessment

Unclear until maintainer and secops direction is settled. The likely maintainable path is to review a generic plugin-provider seam such as https://github.com/openclaw/openclaw/pull/72548 and then decide which vendor providers belong in plugins/ClawHub versus core.
