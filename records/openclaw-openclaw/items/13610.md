## Summary

Current main ships the generic SecretRef baseline and a manifest-backed plugin integration for exec resolvers, but it still does not provide native AWS, Vault, Azure, or GCP source types or a settled in-process provider SDK. Explicit repository discussion keeps this issue as the umbrella while the broader plugin-source proposal remains open, so the requested provider work is still necessary but should not be hardcoded into core.

## What This Changes

Add first-class external secrets-manager integrations for AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, and Google Cloud Secret Manager with lazy resolution, caching, graceful failures, audit logging, and rotation support.

## Best Possible Solution

Approve a narrow, security-reviewed generic SecretRef provider contract, keep vendor credentials and dependencies in independently maintained plugins, and use this umbrella to track representative AWS/Vault/Azure/GCP implementations until the extension path covers their lifecycle and audit needs.

## Reproduction Assessment

Not applicable: this is a request for additional provider capabilities, not a failure of the documented env/file/exec behavior.

## Solution Assessment

No, adding four vendor implementations directly to core is not the narrowest maintainable solution; a generic plugin contract plus externally maintained provider plugins better matches current architecture and dependency ownership.
