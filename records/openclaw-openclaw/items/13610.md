## Summary

Keep open. Current main ships the SecretRef baseline for env, file, and exec sources, but it still does not provide native AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, or Google Cloud Secret Manager provider sources, and the discussion already identifies this as the remaining native-provider umbrella.

## What This Changes

The issue requests native external secrets-manager integrations for OpenClaw config values, including AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, Google Cloud Secret Manager, and formal env support.

## Best Possible Solution

Keep this as the native secrets-manager provider umbrella and have maintainers decide whether to land a generic secret-provider plugin SDK seam like #72548 or add first-class core-owned AWS/Vault/Azure/GCP providers with aligned schemas, resolver dispatch, gateway protocol, CLI, docs, tests, and secops review.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is current-main source and docs inspection, which shows SecretRef/provider sources are env, file, and exec only.

## Solution Assessment

Unclear pending maintainer direction. Exec providers are a workable bridge, but native provider semantics likely need either a reviewed plugin-provider seam or an explicit core decision to own AWS, Vault, Azure, and GCP integrations.
