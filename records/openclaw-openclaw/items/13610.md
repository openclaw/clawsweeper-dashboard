## Summary

Keep open. Current main and the latest release only ship the env/file/exec SecretRef baseline; native AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, and Google Cloud Secret Manager providers remain unresolved security-sensitive provider architecture work.

## What This Changes

Request native external secret-manager sources with config references, caching/reload behavior, clear errors, audit logging, CLI/docs support, and provider-specific implementations.

## Best Possible Solution

Keep this as the native secrets-provider umbrella until maintainers choose a reviewed plugin SDK seam for external providers or explicitly decide to own AWS/Vault/Azure/GCP providers in core with aligned schemas, CLI, docs, and tests.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is current-main and latest-release source/docs inspection, which shows supported SecretRef/provider sources remain env, file, and exec only.

## Solution Assessment

Unclear pending maintainer and secops direction. Native provider support is a reasonable goal, but the maintainable path should be decided between a plugin-provider seam and explicit core-owned providers before implementation.
