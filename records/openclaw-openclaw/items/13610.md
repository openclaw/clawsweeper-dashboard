## Summary

Keep open. Current main and the latest release still provide the shipped SecretRef baseline for env, file, and exec providers, while the requested native AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, and Google Cloud Secret Manager integrations remain unresolved security-sensitive provider architecture work.

## What This Changes

Request native external secrets-manager integrations for OpenClaw config secrets, including cloud and Vault providers, lazy loading, caching, fallback errors, audit logging, and formalized env references.

## Best Possible Solution

Keep this as the native secrets-provider umbrella until maintainers choose a reviewed plugin SDK seam for external providers or explicitly decide to own AWS/Vault/Azure/GCP providers in core with aligned schemas, CLI, docs, and tests.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is current-main and release source/docs inspection, which shows supported SecretRef/provider sources remain env, file, and exec only.

## Solution Assessment

Unclear pending maintainer and secops direction. Native provider support is a reasonable goal, but the maintainable path should be decided between a plugin-provider seam and explicit core-owned providers before implementation.
