## Summary

Keep open. Current main still ships only the env/file/exec SecretRef baseline, the issue discussion explicitly keeps this as the umbrella for native AWS/Vault/Azure/GCP provider work, and the active plugin-seam path is still an open PR rather than merged core behavior.

## What This Changes

The issue requests native external secrets-manager integration for providers such as AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, Google Cloud Secret Manager, and formalized environment-variable references.

## Best Possible Solution

Keep this as the native secrets-manager provider umbrella until maintainers choose a reviewed plugin SDK seam for external providers or explicitly decide to own AWS/Vault/Azure/GCP providers in core with aligned schemas, resolver, CLI, docs, and tests.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is current-main source/docs inspection showing SecretRef sources, provider config, schema validation, resolver dispatch, and gateway wire schema still limited to env/file/exec.

## Solution Assessment

Unclear pending maintainer and secops direction. Native provider support is reasonable, but the maintainable path should be decided between a plugin-provider seam like https://github.com/openclaw/openclaw/pull/72548 and explicit core-owned providers before implementation.
