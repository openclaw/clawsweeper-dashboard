## Summary

Keep open. Current main has the shipped SecretRef baseline for env, file, and exec sources, but it still does not implement native AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, or Google Cloud Secret Manager sources, and recent maintainer discussion keeps this as the native provider umbrella.

## What This Changes

Request native external secret-provider integrations for AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, Google Cloud Secret Manager, and formalized environment-variable secret references.

## Best Possible Solution

Keep this as the native secrets-manager provider umbrella: either land a reviewed generic secret-provider plugin SDK seam and move AWS/Vault/Azure/GCP into plugins, or explicitly add first-class native providers across config types, schemas, resolver dispatch, gateway protocol, CLI configure/audit/apply flows, docs, and tests.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection on current main, which shows accepted SecretRef/provider sources are env, file, and exec only.

## Solution Assessment

Unclear pending maintainer direction. Exec wrappers are a useful workaround, but native provider semantics should probably go through a reviewed plugin-provider seam unless maintainers explicitly choose core-owned AWS/Vault/Azure/GCP providers.
