## Summary

Current main ships the SecretRef baseline for env, file, and exec providers, but the requested native AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, and Google Cloud Secret Manager integrations are still not implemented. Recent issue discussion and the open plugin-seam PRs confirm this remains an active umbrella rather than a cleanup candidate.

## What This Changes

Request native external secrets-manager integrations and first-class configuration/runtime support for cloud and Vault providers instead of storing credentials directly in OpenClaw config files.

## Best Possible Solution

Keep this issue as the native secrets-provider umbrella until maintainers decide and land either a generic SecretRef provider plugin/manifest seam with vendor plugins, or explicit first-class cloud/Vault providers across config schemas, resolver dispatch, CLI/docs, audit behavior, and tests.

## Reproduction Assessment

Not applicable. This is a feature request, and source inspection confirms the current supported SecretRef sources are env, file, and exec rather than native AWS/Vault/Azure/GCP providers.

## Solution Assessment

Unclear as a single implementation path. The maintainable direction likely starts with a reviewed plugin or manifest provider seam before adding vendor providers, because the request crosses config, SDK, resolver, CLI, docs, and security ownership boundaries.
