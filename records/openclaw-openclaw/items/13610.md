## Summary

Keep open: current main and the latest release still expose SecretRef as env/file/exec only, while native AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, and Google Cloud Secret Manager support remains an unresolved security-sensitive provider/API decision.

## What This Changes

The issue requests native external secret-manager support for providers such as AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, Google Cloud Secret Manager, and formalized environment-variable references.

## Best Possible Solution

Keep this as the native secrets-manager umbrella until maintainers choose and land a security-reviewed provider/plugin contract, then split AWS, Vault, Azure, and GCP support into focused plugins or explicit first-class provider follow-ups.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source inspection gives a high-confidence current-main check that native cloud/Vault SecretRef sources are absent and the supported contract remains env/file/exec.

## Solution Assessment

Unclear pending maintainer direction. A plugin or ClawHub provider seam fits OpenClaw architecture better than adding every vendor directly to core, but the exact contract and security boundaries still need approval.
