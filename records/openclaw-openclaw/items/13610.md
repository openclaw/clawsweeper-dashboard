## Summary

Keep open: current main and the latest release still provide the shipped SecretRef baseline for env/file/exec only, while native cloud/Vault providers and the plugin/provider contract remain unresolved security-sensitive product work.

## What This Changes

The issue requests native external secret-manager integrations such as AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, and Google Cloud Secret Manager, plus related config, caching, fallback, audit, and rotation behavior.

## Best Possible Solution

Keep this as the native secrets-provider umbrella until maintainers approve a security-reviewed provider/plugin contract, then split AWS, Vault, Azure, GCP, and similar backends into focused plugin or ClawHub follow-ups with live backend proof.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source, docs, tests, release inspection, and issue context give high confidence that native AWS/Vault/Azure/GCP SecretRef sources are absent and the supported contract remains env/file/exec.

## Solution Assessment

Unclear until maintainers choose the security and product contract. A plugin or ClawHub provider seam fits OpenClaw's lean-core direction, but credential-provider execution is security-sensitive and needs explicit secops/product approval before implementation.
