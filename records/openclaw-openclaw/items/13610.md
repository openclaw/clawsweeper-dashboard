## Summary

Keep open: current main still implements only env/file/exec SecretRef providers, while native AWS/Vault/Azure/GCP sources or a pluggable provider contract remain unmerged and security/product-sensitive. The discussion and open PRs show active implementation paths, so this issue is still useful as the umbrella.

## What This Changes

The issue requests native integration with external secrets managers such as AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, and Google Cloud Secret Manager.

## Best Possible Solution

Keep this as the native secrets-manager umbrella while maintainers review a security-audited provider contract; move vendor implementations to plugins/ClawHub unless maintainers explicitly choose first-class core providers.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source inspection on current main gives a high-confidence check that provider sources remain env/file/exec only.

## Solution Assessment

Unclear until maintainers choose the public contract. A plugin or manifest provider seam fits the repo architecture better than hard-coding each vendor in core, but the secret-resolution boundary needs explicit security review.
