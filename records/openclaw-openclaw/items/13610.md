## Summary

Keep open. Current main has the shipped SecretRef baseline for env, file, and exec sources, but it still lacks native AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, and Google Cloud Secret Manager source types, and recent maintainer/context comments already treat this as the remaining native-provider umbrella.

## What This Changes

Request native integration with AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, Google Cloud Secret Manager, and formalized env-backed secrets with caching, rotation, fallback, and audit behavior.

## Best Possible Solution

Keep this as the native secrets-manager provider umbrella and have maintainers choose between a reviewed generic secret-provider plugin SDK seam like #72548 or core-owned AWS/Vault/Azure/GCP providers with aligned types, schemas, resolver dispatch, gateway protocol, CLI, docs, tests, and secops review.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is current-main source and docs inspection, which shows the supported SecretRef/provider sources remain env, file, and exec only.

## Solution Assessment

Unclear pending maintainer direction. Exec providers are a workable bridge, but native provider semantics should land either through a reviewed plugin-provider seam or an explicit decision that core owns AWS, Vault, Azure, and GCP integrations.
