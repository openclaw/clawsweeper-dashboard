## Summary

Keep open. Current main has the shipped SecretRef baseline for env, file, and exec sources, but it still does not implement native AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, or Google Cloud Secret Manager sources; the latest maintainer discussion also explicitly keeps this as the native provider umbrella.

## What This Changes

Request native external secrets-manager integrations so OpenClaw credentials can reference AWS Secrets Manager, Vault, Azure Key Vault, Google Cloud Secret Manager, and formal env providers instead of plaintext config values.

## Best Possible Solution

Keep this as the native secrets-manager provider umbrella. The maintainable end state is either a reviewed generic secret-provider plugin SDK seam followed by AWS/Vault/Azure/GCP provider plugins, or an explicit maintainer decision to add first-class native providers with matching types, schemas, resolver dispatch, gateway protocol, CLI configure/audit/apply flows, docs, and tests.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection on current main: accepted SecretRef/provider sources are env, file, and exec only, with Vault shown through exec rather than as a native source.

## Solution Assessment

Unclear pending maintainer direction. Exec wrappers are a useful workaround, but they do not provide native provider source types or provider-specific schema/CLI/audit semantics; a generic plugin-provider seam appears safer unless maintainers explicitly want these providers in core.
