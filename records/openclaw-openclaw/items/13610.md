## Summary

Keep open. Current main ships the SecretRef baseline for env, file, and exec sources, and the docs show exec-wrapper examples for Vault and 1Password, but native AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, and Google Cloud Secret Manager provider sources are still not implemented.

## What This Changes

The issue requests native external secrets-manager integrations for OpenClaw configuration and credential resolution.

## Best Possible Solution

Keep this as the native secrets-manager provider umbrella and have maintainers decide whether to land a generic secret-provider plugin SDK seam like #72548 or add first-class core-owned AWS/Vault/Azure/GCP providers with aligned schemas, resolver dispatch, gateway protocol, CLI, docs, tests, and secops review.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is current-main source and docs inspection, which shows SecretRef/provider sources remain env, file, and exec only.

## Solution Assessment

Unclear pending maintainer direction. Exec providers are a workable bridge, but native provider semantics need either a reviewed plugin-provider seam or an explicit core decision to own AWS, Vault, Azure, and GCP integrations.
