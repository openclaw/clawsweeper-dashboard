## Summary

Keep open: current main has the shipped env/file/exec SecretRef baseline, but it still does not provide native AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, or Google Cloud Secret Manager sources, and the remaining path needs maintainer product and security review.

## What This Changes

Request to add native external secrets-manager integrations and config syntax for providers such as AWS Secrets Manager, Vault, Azure Key Vault, and Google Cloud Secret Manager.

## Best Possible Solution

Keep this issue as the native secrets-manager umbrella until maintainers choose and land a security-reviewed provider/plugin contract, then move individual AWS/Vault/Azure/GCP integrations into focused plugins or explicit first-class provider follow-ups.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Source inspection gives a high-confidence current-main check that native cloud/Vault SecretRef sources are absent and the supported contract remains env/file/exec.

## Solution Assessment

Unclear pending maintainer direction. A plugin/ClawHub provider seam fits OpenClaw architecture better than adding every vendor directly to core, but the exact contract and security boundaries still need approval.

- Exec wrappers can integrate Vault, 1Password, and custom resolvers today, but they do not provide native provider source types, vendor schemas, provider-specific CLI UX, or first-class AWS/Vault/Azure/GCP test coverage.
