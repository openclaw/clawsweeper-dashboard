## Label Justifications

- `P3`: The underlying secure-secret capability and plugin seam are shipped, leaving optional vendor integrations as ecosystem expansion rather than an urgent core defect.
- `impact:security`: The request concerns keeping credentials out of plaintext configuration and preserving secret-access boundaries.
- `impact:auth-provider`: The requested integrations supply credentials used by provider and channel authentication paths.

## Summary

OpenClaw now ships the generic SecretRef runtime and a manifest-first plugin integration contract, so AWS, Vault, Azure, and GCP integrations can be delivered as installable plugins without additional vendor-specific core code; the active `claw-vault` implementation already demonstrates that path.

## What This Changes

Add first-class AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, and Google Cloud Secret Manager credential sources with caching, reload, audit, and failure handling.

## Best Possible Solution

Keep OpenClaw core’s generic SecretRef and manifest contract stable, and publish separately reviewed AWS, Azure, GCP, and additional Vault integrations as plugins through ClawHub, following the existing `claw-vault` pattern.

## Reproduction Assessment

Not applicable; this is a feature request, and current-main source plus the working external Vault plugin establish the supported implementation path.

## Solution Assessment

No for vendor-specific core integrations; the shipped manifest-first plugin contract is the narrower maintainable solution and avoids adding cloud SDKs, credentials policy, and vendor lifecycle behavior to core.
